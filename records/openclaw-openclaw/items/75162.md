## Summary

Keep open. Current main still lacks these assembly hooks, and the linked AICM plugin is useful evidence of plugin-author demand, but the proposed public transcript mutation/observation API has blocking policy, correctness, and release-process issues before it can merge.

## What This Changes

The PR adds `before_assemble` and `after_assemble` plugin hook types, HookRunner methods, embedded-runner call sites around prompt assembly, and a pnpm lockfile importer for `extensions/aicm`.

## Best Possible Solution

Settle the core API shape first, then implement one guarded extension point with explicit policy classification, accumulated-event semantics for sequential transforms, an honest final model-input payload, SDK/docs/contract tests, a changelog entry, and no unrelated lockfile drift.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. Static review is sufficient here: current main lacks these hook names, and the PR diff shows raw-message hooks added outside the existing policy classifications with non-chained transform behavior and an incomplete final-context payload.

## Solution Assessment

Is this the best way to solve the issue? No. A guarded assembly or LLM-interception seam may be valid, but this patch is not the narrowest maintainable version because it bypasses existing safety gates and does not implement the stated sequential or final-context semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Gate assembly hooks through hook policies:** `src/plugins/hook-types.ts:106-107`
  - body: The new hooks expose or mutate full conversation messages, but they are only added to the generic hook-name list. Because they are not classified with prompt-injection and conversation-access hooks, a non-bundled plugin can register raw transcript hooks without the existing `allowPromptInjection` and `allowConversationAccess` gates.
  - confidence: 0.93
- **[P2] Pass accumulated messages to each assembly handler:** `src/plugins/hooks.ts:1318-1325`
  - body: `runBeforeAssemble` promises chained transformations, but `runModifyingHook` calls every handler with the original event. A later plugin therefore sees the unmodified message list and can undo an earlier plugin's deletion or reorder when it returns its own `messages`.
  - confidence: 0.9
- **[P2] Expose the actual model input to after_assemble:** `src/agents/pi-embedded-runner/run/attempt.ts:2879`
  - body: The `after_assemble` payload passes only `activeSession.messages`, while the request sent to the model also includes prompt text, system/runtime context, and optional images. Hook users will not observe the final context the PR says is about to be sent to the LLM.
  - confidence: 0.84
- **[P3] Remove the stale AICM lockfile importer:** `pnpm-lock.yaml:234`
  - body: The PR no longer contains an `extensions/aicm` package, but the lockfile still adds `extensions/aicm: {}` as a workspace importer. That leaves package metadata inconsistent with the workspace package set.
  - confidence: 0.8
- **[P3] Add the required changelog entry:** `src/plugins/hook-types.ts:106-107`
  - body: This is a user-facing plugin API feature, but the PR does not add a `CHANGELOG.md` entry. Repo policy requires user-facing `feat` changes to include one before merge.
  - confidence: 0.86
