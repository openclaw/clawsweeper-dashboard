## Summary

Keep this PR open. Current main does not implement the requested assembly hooks, and the companion AICM plugin shows real downstream demand, but the current diff has merge blockers around raw transcript safety policy, sequential semantics, final-payload accuracy, and repository hygiene.

## What This Changes

The PR adds typed `before_assemble` and `after_assemble` plugin hooks, HookRunner methods, embedded-runner call sites around context assembly, and an extra `extensions/aicm` lockfile importer.

## Best Possible Solution

Define a guarded core plugin API only after the safety and product boundary is explicit: decide whether this belongs in existing prompt/LLM/context-engine seams, classify the hooks under the right safety gates, make sequential transforms feed accumulated state forward, expose an honest final model-submission payload, add SDK/docs/contract tests and a changelog entry, and remove unrelated lockfile drift.

## Reproduction Assessment

Yes. Static review is sufficient: current main lacks the hook names, and the PR diff shows the raw-message hooks added outside existing policy classifications with non-chained modifying input and incomplete final-context observation.

## Solution Assessment

No. A guarded assembly extension point may be a valid direction, but this patch is not the best implementation because it bypasses existing safety gates and does not accurately implement the stated sequential or final-context semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Gate assembly hooks through hook policies:** `src/plugins/hook-types.ts:106-107`
  - body: The new hooks expose or replace full session messages, but the hook names are only added to the generic hook list. Add them to the appropriate prompt-injection and conversation-access classifications, or an external plugin can register raw transcript hooks without the existing safety gates.
  - confidence: 0.93
- **[P2] Feed accumulated messages to each assembly handler:** `src/plugins/hooks.ts:1318-1325`
  - body: `runBeforeAssemble` claims plugins can chain transformations, but `runModifyingHook` calls each handler with the original event. Later plugins therefore see the unmodified message list and can unintentionally undo earlier deletions or rewrites.
  - confidence: 0.9
- **[P2] Refresh pre-prompt count after before_assemble:** `src/agents/pi-embedded-runner/run/attempt.ts:2595-2620`
  - body: The hook runs after `prePromptMessageCount` is captured. If a plugin adds or removes messages, the after-turn assistant lookup and context-engine finalization still use the old boundary and can misidentify the current attempt's assistant message.
  - confidence: 0.86
- **[P2] Expose the actual model input to after_assemble:** `src/agents/pi-embedded-runner/run/attempt.ts:2881`
  - body: The `after_assemble` payload passes only `activeSession.messages`, but the model call also includes prompt text, system/runtime context, and optional images. Hook users will not observe the final context the PR says is about to be sent to the LLM.
  - confidence: 0.84
- **[P3] Remove the stale AICM lockfile importer:** `pnpm-lock.yaml:234`
  - body: The PR no longer contains an `extensions/aicm` package, but the lockfile still adds `extensions/aicm: {}` as a workspace importer. That leaves package metadata inconsistent with the workspace package set.
  - confidence: 0.8
- **[P3] Add the required changelog entry:** `src/plugins/hook-types.ts:106-107`
  - body: This is a user-facing plugin API feature, but the PR does not add a `CHANGELOG.md` entry. Repo policy requires user-facing `feat` changes to include one before merge.
  - confidence: 0.86
