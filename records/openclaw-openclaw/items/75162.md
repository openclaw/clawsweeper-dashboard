## Summary

Keep open. The PR adds a plausible core Plugin SDK extension point, but the latest head still has blocking policy and correctness issues, lacks real behavior proof, and overlaps unresolved raw-context mutation design work.

## What This Changes

The PR adds `before_assemble` and `after_assemble` plugin hooks, hook runner methods, embedded-runner call sites, and a changelog entry for context assembly interception.

## Best Possible Solution

Define one guarded context-assembly mutation contract with explicit conversation and prompt-injection policy, cloned/validated message payloads, final model-input semantics, docs/API coverage, and regression tests across the embedded runner and related hook surfaces.

## Reproduction Assessment

Yes. Source review of the PR head is enough to reproduce the review findings: `before_assemble` can rewrite messages without the prompt-injection gate, `after_assemble` observes a stale system prompt, and live session arrays are passed directly to plugin code.

## Solution Assessment

No. A guarded assembly hook may be useful, but this patch is not the best mergeable shape until it matches current-main prompt assembly, enforces the existing trust policies, documents the public contract, and includes real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Gate before_assemble with prompt-injection policy:** `src/plugins/hook-types.ts:177`
  - body: `before_assemble` can replace the model-visible message list, but this addition only classifies it as a conversation hook. With `allowConversationAccess=true` and `allowPromptInjection=false`, a non-bundled plugin can still rewrite or inject context, bypassing the policy that blocks prompt-mutating hooks.
  - confidence: 0.92
- **[P2] Pass cloned messages into assembly hooks:** `src/agents/pi-embedded-runner/run/attempt.ts:2605`
  - body: The hook payload hands `activeSession.messages` to plugin code directly. A handler that mutates the array and then returns void or throws still changes session state despite the result contract and catch path; use cloned payloads and only apply validated returned messages.
  - confidence: 0.86
- **[P2] Report the runtime system prompt to after_assemble:** `src/agents/pi-embedded-runner/run/attempt.ts:2886`
  - body: The event says it observes the final model input, but it sends `systemPromptText` before `runtimeContext` is composed into the system prompt below. Observability plugins will miss context that is actually visible to the model on non-runtime-only submissions.
  - confidence: 0.9
