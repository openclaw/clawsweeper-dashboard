## Summary

Keep this PR open for maintainer review. Current main does not implement the requested assembly hooks, and the linked AICM plugin shows real downstream demand, but the patch adds raw transcript mutation and observation hooks without the existing plugin safety policy gates and does not fully implement the stated sequential/final-context semantics.

## What This Changes

This PR adds typed `before_assemble` and `after_assemble` plugin hooks, wires them around the embedded runner context assembly path, re-exports their event/result types, and adds an `extensions/aicm` lockfile importer.

## Best Possible Solution

Define the assembly-extension contract as a narrow, guarded plugin API: classify transcript mutation/observation under explicit safety gates, decide how it coexists with the context-engine slot, chain sequential message transforms correctly, expose the real model-submission payload or rename the observation hook, add SDK/docs/contract tests plus the required changelog entry, and remove unrelated lockfile drift before merge.

## Reproduction Assessment

Yes. Static review is enough here: current main lacks the hook names, and the PR diff shows the hooks added outside existing policy classifications, with non-chained modifying input and an observation payload that omits prompt/system/images from the actual model call.

## Solution Assessment

No. A guarded assembly extension point may be the right core direction, but this patch is not the best implementation because it bypasses existing safety gates and does not accurately implement the stated sequential or final-context semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Gate assembly hooks through hook policies:** `src/plugins/hook-types.ts:106-107`
  - body: The new hooks expose or mutate full conversation messages, but they are only added to the generic hook-name list. Because they are not classified with prompt-injection and conversation-access hooks, a non-bundled plugin can register raw transcript hooks without the existing `allowPromptInjection` and `allowConversationAccess` gates.
  - confidence: 0.93
- **[P2] Pass accumulated messages to each assembly handler:** `src/plugins/hooks.ts:1318-1325`
  - body: `runBeforeAssemble` promises chained transformations, but `runModifyingHook` calls every handler with the original event. A later plugin therefore sees the unmodified message list and can undo an earlier plugin's deletion or reorder when it returns its own `messages`.
  - confidence: 0.9
- **[P2] Expose the actual model input to after_assemble:** `src/agents/pi-embedded-runner/run/attempt.ts:2878`
  - body: The `after_assemble` payload passes only `activeSession.messages`, while the request sent to the model also includes prompt text, system/runtime context, and optional images. Hook users will not observe the final context the PR says is about to be sent to the LLM.
  - confidence: 0.84
- **[P3] Remove the stale AICM lockfile importer:** `pnpm-lock.yaml:234`
  - body: The PR no longer contains an `extensions/aicm` package, but the lockfile still adds `extensions/aicm: {}` as a workspace importer. That leaves package metadata inconsistent with the workspace package set.
  - confidence: 0.8
- **[P3] Add the required changelog entry:** `src/plugins/hook-types.ts:106-107`
  - body: This is a user-facing plugin API feature, but the PR does not add a `CHANGELOG.md` entry. Repo policy requires user-facing `feat` changes to include one before merge.
  - confidence: 0.86
