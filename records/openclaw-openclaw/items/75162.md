## Summary

Keep this PR open, but do not merge it as-is. The AICM companion plugin shows a real downstream use case, but the proposed core hook API exposes raw transcript mutation and observation before the safety policy, final-context contract, chaining semantics, docs/tests, and lockfile hygiene are settled.

## What This Changes

The PR adds typed `before_assemble` and `after_assemble` plugin hooks, wires them into the Pi embedded runner around context assembly, re-exports their event/result types, and adds an `extensions/aicm` lockfile importer.

## Best Possible Solution

Define this as a narrow, guarded assembly-extension contract in core: classify mutation and observation under explicit hook policy gates, decide whether the supported surface is message-only or full model-input observation, document ordering and payload authority, add SDK/docs/contract tests, preserve real chained state between sequential handlers, and remove unrelated lockfile drift. The AICM plugin is useful as a prototype/demand signal, but the core API should be frozen only after that policy review.

## Reproduction Assessment

Yes. Static review is enough: current main lacks the new hook names, and the PR patch shows the hooks added outside existing safety classifiers, with non-chained modifying-event input and message-only observation despite the stated final-context semantics.

## Solution Assessment

No. A guarded assembly hook may be a valid core direction, but this implementation is not the best fix because it bypasses current plugin safety policy and does not accurately implement the sequential or final-context contract it advertises.

## Review Findings

Full review comments:

- **[P1] Gate assembly hooks through hook policy:** `src/plugins/hook-types.ts:106-107`
  - body: The new hooks expose or mutate full conversation messages, but the diff only adds them to the generic hook-name list. Because they are not classified with the prompt-injection and conversation-access hooks, a non-bundled plugin can register raw transcript hooks without the current `allowPromptInjection` and `allowConversationAccess` gates.
  - confidence: 0.93
- **[P2] Thread modified messages into later handlers:** `src/plugins/hooks.ts:1321-1324`
  - body: `runBeforeAssemble` promises chained transformations, but `runModifyingHook` calls every handler with the original event. A later plugin therefore sees the unmodified message list and can undo an earlier plugin's deletion or reorder when it returns its own `messages`.
  - confidence: 0.9
- **[P2] Report the actual model input to after_assemble:** `src/agents/pi-embedded-runner/run/attempt.ts:2914`
  - body: The `after_assemble` payload passes only `activeSession.messages`, while the request sent to the model also includes prompt text, system/runtime context, and optional images. Hook users will not observe the final context the PR says is about to be sent to the LLM.
  - confidence: 0.84
- **[P3] Remove the stale AICM lockfile importer:** `pnpm-lock.yaml:234`
  - body: The force-pushed PR no longer contains an `extensions/aicm` package, but the lockfile still adds `extensions/aicm: {}` as a workspace importer. That leaves package metadata inconsistent with the workspace package set and should be removed from this PR.
  - confidence: 0.8
