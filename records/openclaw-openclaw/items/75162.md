## Summary

Keep this PR open. Current main does not implement these assembly hooks, and the linked AICM plugin is useful demand signal, but the proposed public transcript mutation/observation API has blocking policy and correctness issues before it should merge.

## What This Changes

The PR adds `before_assemble` and `after_assemble` typed plugin hooks, HookRunner methods, embedded-runner call sites around prompt assembly, and an extra `extensions/aicm` pnpm lockfile importer.

## Best Possible Solution

Define one guarded core interception contract after maintainer product/security review, preferably aligned with existing `llm_input` and context-engine seams; then enforce policy gates, make transform ordering explicit and tested, expose the actual model-submission payload, document the public plugin API, add the required changelog entry, and remove unrelated lockfile drift.

## Reproduction Assessment

Yes. Static review is enough: current main lacks the hook names, and the PR diff shows raw message mutation/observation hooks added without the existing safety classifications, with non-chained transform input and an incomplete final-context payload.

## Solution Assessment

No. A guarded assembly extension point may be a reasonable direction, but this patch is not the best implementation because it bypasses current hook safety policy and does not implement the stated sequential or final-context semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Gate assembly hooks through hook policies:** `src/plugins/hook-types.ts:106-107`
  - body: The new hooks expose or mutate full conversation messages, but they are only added to the generic hook-name list. Because they are not classified with prompt-injection and conversation-access hooks, a non-bundled plugin can register raw transcript hooks without the existing safety gates.
  - confidence: 0.93
- **[P2] Pass accumulated messages to each assembly handler:** `src/plugins/hooks.ts:1318-1325`
  - body: `runBeforeAssemble` promises chained transformations, but `runModifyingHook` calls every handler with the original event. A later plugin therefore sees the unmodified message list and can undo an earlier plugin's deletion or reorder when it returns its own `messages`.
  - confidence: 0.9
- **[P2] Expose the actual model input to after_assemble:** `src/agents/pi-embedded-runner/run/attempt.ts:2878`
  - body: The `after_assemble` payload passes only `activeSession.messages`, while the request sent to the model also includes prompt text, system/runtime context, and optional images. Hook users will not observe the final context the PR says is about to be sent to the LLM.
  - confidence: 0.84
- **[P3] Remove the stale AICM lockfile importer:** `pnpm-lock.yaml:234`
  - body: The PR no longer contains an `extensions/aicm` package, but the lockfile still adds `extensions/aicm: {}` as a workspace importer. That leaves package metadata inconsistent with the workspace package set.
  - confidence: 0.8
- **[P3] Add the required changelog entry:** `src/plugins/hook-types.ts:106-107`
  - body: This is a user-facing plugin API feature, but the PR does not add a `CHANGELOG.md` entry. Repo policy requires user-facing `feat` changes to include one before merge.
  - confidence: 0.86
