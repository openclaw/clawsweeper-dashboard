## Summary

Keep open, but the current PR is not mergeable as-is. The proposed hook surface is useful core/plugin-API work, yet the diff introduces raw transcript mutation and observation hooks outside the existing safety gates, does not implement the promised chained before_assemble semantics, exposes only messages for a hook described as final assembled context, and carries lockfile/changelog/docs gaps.

## What This Changes

This PR adds typed `before_assemble` and `after_assemble` plugin hooks, wires them around the Pi embedded-runner context assembly path, re-exports their event/result types, and adds an `extensions/aicm` lockfile importer.

## Best Possible Solution

Define a narrow, guarded assembly-extension contract before merge: classify mutation and observation under the existing hook safety gates, decide whether this belongs beside `before_prompt_build`/`llm_input` or a context-engine seam, implement true chained message transforms, expose or rename the final-context payload honestly, add SDK/docs/contract tests and changelog coverage, and remove unrelated lockfile drift.

## Reproduction Assessment

Yes. Static review gives a high-confidence reproduction path: register two `before_assemble` handlers and the second will receive the original event because `runModifyingHook` never updates the event between handlers; inspect the embedded runner and the proposed `after_assemble` payload to see that it omits prompt/system/image inputs sent to the model.

## Solution Assessment

No. A guarded assembly hook may be a valid core feature, but this patch is not the best implementation because it bypasses existing hook policy gates and does not accurately implement the claimed sequential or final-context semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Gate assembly hooks through hook policies:** `src/plugins/hook-types.ts:106-107`
  - body: The new hooks expose or mutate full conversation messages, but they are only added to the generic hook-name list. Because they are not classified with prompt-injection and conversation-access hooks, a non-bundled plugin can register raw transcript hooks without the existing `allowPromptInjection` and `allowConversationAccess` gates.
  - confidence: 0.93
- **[P2] Pass accumulated messages to each assembly handler:** `src/plugins/hooks.ts:1318-1325`
  - body: `runBeforeAssemble` promises chained transformations, but `runModifyingHook` calls every handler with the original event. A later plugin therefore sees the unmodified message list and can undo an earlier plugin's deletion or reorder when it returns its own `messages`.
  - confidence: 0.9
- **[P2] Expose the actual model input to after_assemble:** `src/agents/pi-embedded-runner/run/attempt.ts:2878-2881`
  - body: The `after_assemble` payload passes only `activeSession.messages`, while the request sent to the model also includes prompt text, system/runtime context, and optional images. Hook users will not observe the final context the PR says is about to be sent to the LLM.
  - confidence: 0.84
- **[P3] Remove the stale AICM lockfile importer:** `pnpm-lock.yaml:234`
  - body: The PR no longer contains an `extensions/aicm` package, but the lockfile still adds `extensions/aicm: {}` as a workspace importer. That leaves package metadata inconsistent with the workspace package set.
  - confidence: 0.8
- **[P3] Add the required changelog entry:** `src/plugins/hook-types.ts:106-107`
  - body: This is a user-facing plugin API feature, but the PR does not add a `CHANGELOG.md` entry. Repo policy requires user-facing `feat` changes to include one before merge.
  - confidence: 0.86
