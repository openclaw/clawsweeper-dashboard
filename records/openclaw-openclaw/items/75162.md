## Summary

Keep open. The proposed core extension point is plausible and the companion AICM plugin idea shows real demand, but the latest PR head still exposes raw transcript mutation/observation without the existing hook safety gates and has correctness issues around chaining and the advertised final-context payload.

## What This Changes

The PR adds typed `before_assemble` and `after_assemble` plugin hooks, HookRunner methods, embedded-runner call sites around context assembly, and a stray `extensions/aicm` pnpm lockfile importer.

## Best Possible Solution

Define a guarded core plugin API only after the policy boundary is explicit: classify the hooks under the right safety gates, decide whether this should extend existing `llm_input`/context-engine seams instead, make sequential transforms feed the accumulated event state to later handlers, expose an honest model-submission payload, add SDK/docs/contract tests and a changelog entry, and remove unrelated lockfile drift.

## Reproduction Assessment

Yes. Static review is enough: current main lacks the hook names, and the latest PR diff shows the new raw-message hooks added outside existing policy classifications with non-chained modifying input and an incomplete final-context observation payload.

## Solution Assessment

No. A guarded assembly extension point may be a reasonable core direction, but this patch is not the best implementation because it bypasses existing safety gates and does not accurately implement the stated sequential or final-context semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Gate assembly hooks through hook policies:** `src/plugins/hook-types.ts:106-107`
  - body: The new hooks expose or mutate full conversation messages, but they are only added to the generic hook-name list. Because they are not classified with prompt-injection and conversation-access hooks, a non-bundled plugin can register raw transcript hooks without the existing `allowPromptInjection` and `allowConversationAccess` gates.
  - confidence: 0.93
- **[P2] Pass accumulated messages to each assembly handler:** `src/plugins/hooks.ts:1318-1325`
  - body: `runBeforeAssemble` promises chained transformations, but `runModifyingHook` calls every handler with the original event. A later plugin therefore sees the unmodified message list and can undo an earlier plugin's deletion or reorder when it returns its own `messages`.
  - confidence: 0.9
- **[P2] Expose the actual model input to after_assemble:** `src/agents/pi-embedded-runner/run/attempt.ts:2878`
  - body: The `after_assemble` payload passes only `activeSession.messages`, while the request sent to the model also includes prompt text, system/runtime context, and optional images. Hook users will not observe the final context the PR says is about to be sent to the LLM.
  - confidence: 0.84
- **[P3] Remove the stale AICM lockfile importer:** `pnpm-lock.yaml:234`
  - body: The PR no longer contains an `extensions/aicm` package, but the lockfile still adds `extensions/aicm: {}` as a workspace importer. That leaves package metadata inconsistent with the workspace package set.
  - confidence: 0.8
- **[P3] Add the required changelog entry:** `src/plugins/hook-types.ts:106-107`
  - body: This is a user-facing plugin API feature, but the PR does not add a `CHANGELOG.md` entry. Repo policy requires user-facing `feat` changes to include one before merge.
  - confidence: 0.86
