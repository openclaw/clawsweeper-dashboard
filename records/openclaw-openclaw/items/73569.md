## Summary

Keep open. Current main still collapses the no-provider media-understanding path into an undefined result and the CLI still emits the generic path-based empty transcript/description errors; linked fix PRs are open but unmerged.

## What This Changes

The issue requests clear missing-provider configuration errors for `infer audio transcribe` and `infer image describe` instead of path-blaming empty-result messages.

## Best Possible Solution

Land one canonical fix that preserves the existing empty-result errors when a provider actually runs, but carries missing-provider/no-attempt state far enough for `infer audio transcribe`, `infer image describe`, and `infer image describe-many` to emit actionable configuration errors with regression tests and a changelog entry.

## Reproduction Assessment

Yes. The source path is deterministic on current main: no resolved media entries yields a skipped no-output decision, the runtime returns undefined text, and the CLI throws the generic path-based error.

## Solution Assessment

Yes. The narrow maintainable solution is to distinguish missing-provider/no-attempt from provider-ran-but-empty-output at the media runtime or CLI boundary; maintainer review should choose which open PR supplies that implementation.
