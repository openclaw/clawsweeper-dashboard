## Summary

Keep open. Current main still converts the no-provider/no-attempt media-understanding path into generic file-path errors, and three open PRs already reference this issue with closing syntax, so the issue should remain open until one canonical fix is reviewed and landed.

## What This Changes

The issue asks for `infer audio transcribe` and `infer image describe` to report missing provider configuration clearly instead of emitting path-blaming empty transcript/description errors.

## Best Possible Solution

Land one canonical fix that preserves the existing empty-result errors for providers that actually ran, but carries missing-provider/no-attempt state far enough for `infer audio transcribe`, `infer image describe`, and `infer image describe-many` to emit actionable configuration errors with focused regression tests and a user-facing changelog entry.

## Reproduction Assessment

Yes. With no configured media-understanding providers, the current source path is deterministic: no entries resolve, the runner returns a skipped decision with no outputs, the runtime returns undefined text, and the CLI throws the generic path-based error.

## Solution Assessment

Yes. The narrow maintainable solution is to distinguish missing-provider/no-attempt from provider-ran-but-empty-output at the media runtime or CLI boundary; choosing which linked PR should supply that implementation is maintainer review work.
