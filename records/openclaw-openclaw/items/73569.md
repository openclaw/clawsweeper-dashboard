## Summary

Keep open. Current main still collapses the no-provider/no-attempt media-understanding path into undefined text and the infer CLI then emits the generic path-based empty-result errors; three open PRs already reference this issue with closing syntax, so the issue should remain open until one canonical fix lands.

## What This Changes

The issue asks `infer audio transcribe`, `infer image describe`, and the related image-description path to report missing media provider configuration instead of reusing empty transcript/description errors that blame the input path.

## Best Possible Solution

Land one canonical fix that preserves the existing empty-result errors when a provider actually runs, but carries missing-provider/no-attempt state far enough for `infer audio transcribe`, `infer image describe`, and `infer image describe-many` to emit actionable configuration errors with focused regression tests and a user-facing changelog entry.

## Reproduction Assessment

Yes. With no configured media-understanding provider, the current source path is deterministic: provider resolution returns no entries, the runner returns a skipped decision with no outputs, the runtime returns undefined text, and the CLI throws the generic path-based error.

## Solution Assessment

Yes. The narrow maintainable solution is to distinguish missing-provider/no-attempt from provider-ran-but-empty-output at the media runtime or CLI boundary; choosing which linked PR should supply that implementation is maintainer review work.
