## Summary

Keep open. Current main still collapses the no-provider media-understanding path into `undefined` text and the infer CLI still throws the generic path-based empty transcript/description errors; three open PRs already reference this issue with closing syntax, so it should remain open until a canonical fix lands.

## What This Changes

The issue asks `infer audio transcribe` and `infer image describe` to report missing media provider configuration instead of path-blaming empty transcript/description errors.

## Best Possible Solution

Land one canonical fix that preserves existing empty-result errors after a provider runs, but carries missing-provider/no-attempt state far enough for `infer audio transcribe`, `infer image describe`, and likely `infer image describe-many` to emit actionable configuration errors with regression tests and a user-facing changelog entry.

## Reproduction Assessment

Yes. The issue body gives deterministic no-provider CLI steps, and current source independently verifies the path: no resolved media entries produce a skipped/no-output decision, the runtime returns undefined text, and the CLI throws the generic path-based error.

## Solution Assessment

Yes, the requested direction is the narrow maintainable fix. The best implementation is to distinguish missing-provider/no-attempt from provider-ran-but-empty-output at the media runtime or CLI boundary, then choose one of the linked PRs as the canonical patch.
