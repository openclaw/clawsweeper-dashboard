## Summary

Keep open. Current `main` still maps the no-provider/no-attempt media-understanding path to the same generic file-path no-output errors, and open PRs already reference this issue with closing syntax, so it should remain open until one canonical fix lands.

## What This Changes

The issue requests clearer `infer audio transcribe` and `infer image describe` errors when no relevant provider is configured, while preserving the existing no-output errors for configured providers that return empty text.

## Best Possible Solution

Land one canonical fix that preserves the existing empty-result errors for providers that actually ran, but carries missing-provider/no-attempt state far enough for `infer audio transcribe`, `infer image describe`, and likely `infer image describe-many` to emit actionable provider-configuration errors with focused regression tests and a user-facing changelog entry.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. Source inspection on current `main` shows the deterministic path: no resolved media provider entries produce a skipped/no-output decision, the runtime returns `text: undefined`, and the CLI throws the generic path-blaming error.

## Solution Assessment

Is this the best way to solve the issue? Yes. The narrow maintainable solution is to distinguish missing-provider/no-attempt from provider-ran-but-empty at the media runtime or CLI boundary; choosing which existing PR should land is a maintainer review decision.
