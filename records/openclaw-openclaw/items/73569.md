## Summary

Close as implemented on current main: the media-understanding runtime now preserves skipped/no-attempt decisions and the infer CLI emits explicit missing-provider configuration errors for image understanding and audio transcription instead of path-blaming empty-result errors.

## What This Changes

The issue asks the infer media CLI to distinguish missing audio/image provider configuration from a provider that ran but returned empty output.

## Best Possible Solution

Keep the current main implementation and treat the remaining open fix PRs as superseded cleanup rather than new work for this issue.

## Reproduction Assessment

Yes. The original no-provider path is deterministic, and current tests now model the skipped/no-attempt result for image and audio infer commands; current main no longer routes that state to the misleading path-based error.

## Solution Assessment

Yes. The fix is narrow: it preserves existing provider-empty errors and changes wording only when the runtime reports a skipped decision with no provider attempts.
