## Summary

Keep open. Current main still collapses the no-provider media-understanding path into the generic path-based transcript/description errors, and three open unmerged PRs already reference this issue with closing syntax, so the issue should remain open until one canonical fix lands.

## What This Changes

The issue requests actionable missing-provider configuration errors for `infer audio transcribe` and `infer image describe` when no relevant media provider is configured.

## Best Possible Solution

Land one canonical fix that preserves provider-ran-but-empty-output errors, but carries missing-provider/no-attempt state far enough for `infer audio transcribe`, `infer image describe`, and likely `infer image describe-many` to emit actionable configuration guidance with focused regression tests and a user-facing changelog entry.

## Reproduction Assessment

Yes. Without configured media-understanding providers, the current source path is deterministic: no entries resolve, `runCapability` returns a skipped decision with no outputs, the runtime returns undefined text, and the CLI throws the generic path-based error.

## Solution Assessment

Yes. The narrow maintainable direction is to distinguish skipped/no-provider/no-attempt results from provider-ran-empty results at the media runtime or CLI boundary; the remaining maintainer task is choosing which linked PR supplies that implementation.
