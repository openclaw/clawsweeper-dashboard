## Summary

Keep open. Current main still collapses the no-provider media-understanding path into `undefined` text and the infer CLI still throws the generic path-based empty transcript/description errors. Three open PRs already reference this issue with closing syntax, so the issue should remain open until a canonical fix lands.

## What This Changes

The issue requests clearer missing-provider configuration errors for `infer audio transcribe` and `infer image describe` instead of path-blaming empty transcript/description messages.

## Best Possible Solution

Land one canonical fix that preserves the existing empty-result errors when a provider actually runs, but carries missing-provider/no-attempt state far enough for `infer audio transcribe`, `infer image describe`, and likely `infer image describe-many` to emit actionable configuration errors with regression tests and a user-facing changelog entry.

## Reproduction Assessment

Yes. The issue body gives deterministic CLI steps with no configured media provider, and current source independently verifies the path: no resolved provider entries return a skipped/no-output decision, the runtime returns undefined text, and the CLI throws the generic path-based error.

## Solution Assessment

Yes. The narrow maintainable solution is to distinguish missing-provider/no-attempt from provider-ran-but-empty-output at the media runtime or CLI boundary; choosing among the linked PR implementations is maintainer review work.
