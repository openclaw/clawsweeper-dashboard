## Summary

Keep open. Current main still collapses the no-provider/no-attempt media-understanding path into undefined text and the infer CLI still throws the generic path-based empty transcript/description errors; three open PRs already reference this issue with closing syntax, so the issue should remain open until a canonical fix lands.

## What This Changes

The issue asks for clearer `infer audio transcribe` and `infer image describe` missing-provider errors instead of generic path-based empty-result messages.

## Best Possible Solution

Land one canonical fix that preserves the existing empty-result errors when a provider actually runs, but carries missing-provider/no-attempt state far enough for `infer audio transcribe`, `infer image describe`, and likely `infer image describe-many` to emit actionable configuration errors with focused regression tests and a user-facing changelog entry.

## Reproduction Assessment

Yes. The issue body gives deterministic CLI steps with no configured media provider, and current source independently shows the same path: no resolved entries returns a skipped/no-output decision, the runtime returns undefined text, and the CLI throws the generic path-based error.

## Solution Assessment

Yes. The narrow maintainable direction is to distinguish missing-provider/no-attempt from provider-ran-but-empty-output at the media runtime or CLI boundary; choosing which linked PR should supply that implementation is maintainer review work.
