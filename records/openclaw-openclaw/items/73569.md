## Summary

Keep open. Current main still collapses the no-provider/no-attempt media-understanding path into the generic image/audio empty-result errors, and three open PRs already reference this issue with closing syntax, so the issue should remain open until maintainers choose and land one canonical fix.

## What This Changes

The issue asks for `infer audio transcribe` and `infer image describe` to report missing media provider configuration instead of generic file-path empty-result errors when no provider is configured.

## Best Possible Solution

Land one canonical fix that preserves the existing empty-result errors when a provider actually ran, but carries missing-provider/no-attempt state far enough for `infer audio transcribe`, `infer image describe`, and `infer image describe-many` to emit actionable configuration errors with focused regression tests and a user-facing changelog entry.

## Reproduction Assessment

Yes. The source-level path is deterministic on current main: no resolved media entries produce a skipped decision with no outputs, the runtime returns undefined text, and the CLI throws the generic path-based error. I did not run the CLI or tests because this review was explicitly read-only.

## Solution Assessment

Yes. The narrow maintainable fix is to distinguish missing-provider/no-attempt results from provider-ran-but-empty-output at the media runtime or CLI boundary; the maintainer decision is which linked PR should provide that implementation.
