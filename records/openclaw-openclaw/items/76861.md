Current main still has the reported direct Anthropic request-shape bug, and the linked implementation PR was closed without merging, so this issue should stay open and is a good narrow repair candidate.

## What This Changes

The issue requests omitting unsupported sampling parameters, especially `temperature`, when gateway payloads target Anthropic reasoning/adaptive Claude models, with OpenRouter-routed Anthropic refs called out for verification.

## Best Possible Solution

Add capability-aware payload filtering for direct Anthropic adaptive Claude models, cover it with regression tests, and either include a verified OpenRouter Anthropic fix or split that route into a clearly linked follow-up.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: configure `params.temperature`, route a non-thinking turn to `anthropic/claude-opus-4-7` or `anthropic/claude-sonnet-4-6`, and current main builds a payload containing `temperature`. I did not run a live Anthropic request because no relevant API key is configured in the environment/profile and `pnpm` is unavailable in this checkout.

## Solution Assessment

Mostly yes: gating direct Anthropic temperature forwarding on the existing adaptive-thinking capability is the narrow maintainable fix. The safer complete repair should explicitly verify whether OpenRouter-routed Anthropic refs need the same filtering in the OpenAI-compatible path.
