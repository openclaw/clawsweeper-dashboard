## Summary

Keep open: the bug is real, but the branch is not merge-ready because the new immediate lifecycle-error rendering is not idempotent with the existing delayed chat-error synthesis, and the supplied proof is handler-level only.

## What This Changes

The PR changes TUI event handling and tests so lifecycle errors with `endedAt` render a visible run error, clear the active run, and leave retryable lifecycle errors active.

## Best Possible Solution

Make terminal lifecycle-error rendering idempotent with the gateway and embedded backend delayed chat-error paths, then prove it with focused regression coverage plus a real TUI/backend run.

## Reproduction Assessment

Yes, by source inspection: current main sets only the footer activity status on lifecycle `phase: "error"`, while visible error rendering and active-run cleanup are handled by a separate chat-error path. I did not run tests because this review is read-only.

## Solution Assessment

No, not yet: the direction is right, but the current patch should also dedupe or finalize the run against the later chat-error path before it is the safest maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Deduplicate delayed chat errors after lifecycle rendering:** `src/tui/tui-event-handlers.ts:675-676`
  - body: After this new lifecycle branch renders `run error:` and clears the active run, the existing gateway and embedded-backend paths can still synthesize a later chat `state: "error"` for the same lifecycle failure. Because this branch only calls `terminateRun()` and never records the run as finalized or otherwise suppresses that later chat error, the TUI can add the same run error a second time and reprocess the completed run. Please make the lifecycle-handled run idempotent with the delayed chat-error path and cover that interaction.
