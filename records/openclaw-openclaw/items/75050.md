## Summary

Keep open. The docs gap is real, but the PR is not merge-ready because its PowerShell examples imply that transient proxy variables followed by `openclaw gateway restart` will affect an installed gateway service, while current main deliberately does not persist `HTTP_PROXY`, `HTTPS_PROXY`, or `NO_PROXY` into managed service environments.

## What This Changes

The PR adds Feishu/Lark troubleshooting documentation for mixed proxy setups using `NO_PROXY` alongside `HTTP_PROXY` and `HTTPS_PROXY`.

## Best Possible Solution

Keep the useful mixed-proxy troubleshooting section, but revise it to separate foreground `openclaw gateway run` environment examples from installed service behavior, and leave #72595 as the place for any future channel-level direct/ambient proxy policy.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the review finding: a static check of the changed snippet against current `buildServiceEnvironment` tests and the `gateway restart` service path shows transient PowerShell proxy env does not become the installed service environment.

## Solution Assessment

Is this the best way to solve the issue? No as written; documenting `NO_PROXY` is a good narrow docs improvement, but the examples need to describe how the gateway process actually receives those variables for manual launches versus managed services.
