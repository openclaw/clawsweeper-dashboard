## Summary

Keep this open: current main still has a source-level path where Dreaming narrative sessions are created with run-specific keys and rely on the same process reaching `deleteSession`; the focused cleanup PR is open but unmerged and does not close this issue yet.

## What This Changes

The issue requests durable cleanup for Dreaming child sessions that remain visible after a Gateway restart interrupts the in-process cleanup path.

## Best Possible Solution

Land or update the focused memory-core cleanup so Dreaming uses bounded transient session keys and durable cleanup for restart-left completed sessions, with regression coverage that preserves the generic plugin-owned deletion contract.

## Reproduction Assessment

Yes, at source level. Current main creates timestamped Dreaming narrative session keys, deletes them only from the in-process `finally` path, and the scrubber keeps live transcript-backed rows that would still appear in the session store after a restart skips deletion.

## Solution Assessment

No complete fix is on main yet. The narrow maintainable direction is the memory-core stable-key/pre-cleanup work from https://github.com/openclaw/openclaw/pull/70464 plus restart-orphan coverage, rather than a broad Gateway startup scanner.
