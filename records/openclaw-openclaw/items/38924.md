## Summary

Keep open: current main does not already implement AAMA Phase 1, and this external PR still has blocking correctness/security issues, failing protocol/secret checks, and no after-fix real behavior proof.

## What This Changes

The branch adds an AAMA Phase 1 approval-token and memory policy gate across gateway send/poll, outbound delivery/message actions, session-memory writes, protocol params, and tests.

## Best Possible Solution

Rework AAMA behind an explicit maintainer-approved design, preserve gateway/protocol contracts, update generated clients/docs, fix token propagation and suspension semantics, then demonstrate real outbound and memory-gate behavior.

## Reproduction Assessment

Yes for the PR findings from source inspection: with Phase 1 enabled, gateway send/poll paths enforce locally and then call gateway RPCs without the same approval context, while plugin actions are all mapped through the external-send gate. I did not execute the branch.

## Solution Assessment

No. Centralizing an outbound guard may be a reasonable direction, but this branch applies it inconsistently across gateway boundaries and too broadly across plugin actions; the safer path is a design-backed, narrower implementation that keeps protocol artifacts and runtime semantics aligned.

  - body: `sendMessage` now checks AAMA locally before gateway delivery, but the gateway `send` request still omits `approvalToken`/`approvalNonce` from `aamaPolicy.actionParams`.   - confidence: 0.88

  - body: `handlePluginAction` now guards every plugin action, while `buildActionType` maps every unknown action to `external_followup_send`. Read-style actions such as `search` or `list-pins` are valid channel actions on main, but this makes them require external-send approval tokens and can trigger the missing-token suspension path.
  - confidence: 0.86

  - confidence: 0.84
