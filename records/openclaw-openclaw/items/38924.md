## Summary

Keep open. Current main still has no AAMA approval-token surface, and this PR remains unique but blocked by security-sensitive behavior, gateway protocol propagation gaps, failing protocol validation, and missing real behavior proof.

## What This Changes

Adds AAMA Phase 1 HMAC/SQLite approval-token enforcement across outbound send, poll, plugin-action, gateway send/poll, and session-memory append-only paths with tests and fixtures.

## Best Possible Solution

Land AAMA only after maintainer-approved security/protocol scope is documented, missing approval becomes a non-destructive block, approval context is propagated end to end, generated protocol artifacts are synced, and redacted real setup proof is attached.

## Reproduction Assessment

Yes for the blocking PR findings by source inspection: the PR head persists suspension on missing approval tokens and omits approval context from gateway send/poll calls after local enforcement. I did not run branch tests because this review is read-only.

## Solution Assessment

No. The current branch is not the best mergeable solution until the security design, gateway protocol propagation, generated artifacts, CI failures, and real behavior proof are resolved.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Avoid suspending autonomy for a missing approval token:** `src/infra/aama-spine-controls.ts:682-690`
  - body: When enforcement is enabled, an omitted token or nonce calls `suspendAutonomy` and persists global suspension before throwing. A missing parameter should block that action without permanently disabling later approved outbound traffic.
  - confidence: 0.94
- **[P1] Forward approval context on gateway sends:** `src/infra/outbound/message.ts:197-203`
  - body: `sendMessage` consumes approval context before the gateway path, but the gateway `send` request does not include the token or nonce. Gateway-side enforcement can reject the already-approved send and burn the one-time token.
  - confidence: 0.9
- **[P1] Forward poll actor and approval context to the gateway:** `src/infra/outbound/message.ts:330-337`
  - body: The poll path checks AAMA locally, then sends a gateway `poll` request without `agentId`, `approvalToken`, or `approvalNonce`. Gateway enforcement can default the actor to `system` or fail with a missing-token error.
  - confidence: 0.9
- **[P2] Gate only externally sending plugin actions:** `src/infra/outbound/message-action-runner.ts:728-738`
  - body: `handlePluginAction` now enforces AAMA for every non-send/poll/broadcast plugin action, while unknown actions map to `external_followup_send`. Read-oriented actions can unexpectedly require send approval.
  - confidence: 0.86
- **[P2] Regenerate the gateway protocol artifacts:** `src/gateway/protocol/schema/agent.ts:48-59`
  - body: The PR adds send/poll protocol fields, but generated gateway client models are not updated and the protocol check is failing. Schema changes need generated/client outputs kept in sync before merge.
  - confidence: 0.78
- **[P3] Honor actionBlocked before writing session memory:** `src/hooks/bundled/session-memory/handler.ts:332-357`
  - body: The new memory gate can return `actionBlocked`, but the handler only handles quarantine and then writes. If a future record carries unresolved contradiction age, a blocked decision would still persist output.
  - confidence: 0.75

## Security Review

Status: needs_attention

Summary: The diff introduces security-sensitive approval enforcement with concrete availability and token-propagation risks.

Concerns:

- **[high] Missing approval can persistently suspend all autonomy:** `src/infra/aama-spine-controls.ts:682`
  - body: A missing token or nonce is treated like an approval-token bypass and persists global suspension state, creating a denial-of-service path from ordinary misconfiguration or outdated callers.
  - confidence: 0.94
- **[medium] Gateway enforcement can reject consumed one-time tokens:** `src/infra/outbound/message.ts:197`
  - body: Local send/poll enforcement can consume an approval token before calling a gateway request that lacks the same approval context, causing remote enforcement to fail after the token has been burned.
  - confidence: 0.9
