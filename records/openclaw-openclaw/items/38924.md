## Summary

Keep open: current main does not already implement AAMA Phase 1, and this external PR still has blocking correctness/security issues, failing protocol/secret checks, and no after-fix real behavior proof.

## What This Changes

The branch adds an AAMA Phase 1 approval-token and memory policy gate across gateway send/poll, outbound delivery/message actions, session-memory writes, protocol params, and tests.

## Best Possible Solution

Rework AAMA behind an explicit maintainer-approved design, preserve gateway/protocol contracts, update generated clients/docs, fix token propagation and suspension semantics, then demonstrate real outbound and memory-gate behavior.

## Reproduction Assessment

Yes for the PR findings from source inspection: with Phase 1 enabled, gateway send/poll paths enforce locally and then call gateway RPCs without the same approval context, while plugin actions are all mapped through the external-send gate. I did not execute the branch.

## Solution Assessment

No. Centralizing an outbound guard may be a reasonable direction, but this branch applies it inconsistently across gateway boundaries and too broadly across plugin actions; the safer path is a design-backed, narrower implementation that keeps protocol artifacts and runtime semantics aligned.

  - body: `sendMessage` now checks AAMA locally before gateway delivery, but the gateway `send` request still omits `approvalToken`/`approvalNonce` from `aamaPolicy.actionParams`.   - body: `sendPoll` enforces the AAMA policy with the supplied actor/action params, but the gateway `poll` request does not include `agentId`, `approvalToken`, or `approvalNonce`. The gateway then re-enforces as `system` or with no token, so valid approved polls fail under Phase 1 enforcement.
  - confidence: 0.88
- **[P2] Narrow AAMA enforcement to outbound mutating actions:** `src/infra/aama-spine-controls.ts:417-424`
  - body: `handlePluginAction` now guards every plugin action, while `buildActionType` maps every unknown action to `external_followup_send`. Read-style actions such as `search` or `list-pins` are valid channel actions on main, but this makes them require external-send approval tokens and can trigger the missing-token suspension path.
  - confidence: 0.86

  - body: This PR adds new gateway send/poll schema fields, but the generated protocol artifacts were not committed; CI `pnpm protocol:check` prints the missing Swift model diff. Protocol schema changes need the generated client surfaces updated with the schema change.
  - confidence: 0.84

  - body: The added AAMA fixtures use `phase1-secret`/approval-secret test values that detect-secrets flags in CI. Rename the fixtures to a clearly non-secret placeholder or add the repository’s expected inline allowlist comment so the `secrets` job can pass.

## Security Review

Status: needs_attention

Summary: The patch touches security-sensitive policy enforcement and currently introduces denial-of-service and secret-scanner concerns that need attention before merge.

- **[medium] Missing tokens permanently suspend autonomy:** `src/infra/aama-spine-controls.ts:682`
  - body: When a required approval token or nonce is absent, the PR calls `suspendAutonomy`, persists `autonomy_suspended`, and has no visible unsuspend path. A configuration gap or older caller can therefore create a durable system-wide outbound denial of service instead of only blocking that one action.
