## Summary

Keep open. The release reports provide concrete large-session-store latency evidence, and current main still turns every Control UI `sessions.changed` notification into a full `sessions.list` reload even though fixed-interval polling was replaced by subscriptions.

## What This Changes

Reduce Control UI and Gateway session-list pressure so idle Control UI/webchat tabs do not repeatedly call slow `sessions.list` on large session stores.

## Best Possible Solution

Keep this open until Control UI applies ordinary `sessions.changed` row payloads incrementally and reserves full `sessions.list` reloads for initial load, explicit refresh, filter changes, event gaps, or payloads that cannot be reconciled locally.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to deliver a `sessions.changed` frame to Control UI: current tests expect `loadSessions`, and `loadSessions` calls `sessions.list`; the issue and follow-up comment provide the large-store release logs showing the user-visible impact.

## Solution Assessment

Yes. The narrow maintainable fix is to stop treating every push event as a full-list invalidation, then benchmark whether the server list path still needs cache/lightweight-row work for large stores.

## Repair Work Prompt

Fix the Control UI session refresh pressure reported in openclaw/openclaw#76166. Current main subscribes to `sessions.changed` but still calls `loadSessions` for every event, which causes full `sessions.list` reloads on large session stores. Implement a narrow repair so ordinary create/send/patch/compact/update events with enough row payload are applied locally via `applySessionsChangedEvent` without immediately reissuing `sessions.list`; keep full reloads for initial load, manual refresh, filter changes, sequence gaps, cleanup/delete or missing-row cases that cannot be reconciled safely. If server-side changes are needed, keep them generic and bounded: avoid owner/plugin-specific policy, preserve protocol compatibility, and consider a lightweight/cached row path rather than broad migrations. Update focused UI/Gateway tests to prove a `sessions.changed` event with payload no longer triggers `sessions.list`, while unreconcilable events still do. Add a single-line user-facing changelog entry because this is a Control UI/Gateway performance fix. Do not broaden into the #76188 provider/defaults/low-power-host program.
