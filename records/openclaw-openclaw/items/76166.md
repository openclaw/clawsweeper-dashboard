## Summary

Keep open. Current main still applies each `sessions.changed` payload and then immediately reloads the full session list, and the issue is already being tracked by open PR #76277 rather than a shipped fix.

## What This Changes

The issue requests eliminating repeated Control UI `sessions.list` churn on large session stores in v2026.4.29 and bounding the Gateway session-list hot path.

## Best Possible Solution

Control UI should apply ordinary `sessions.changed` payloads locally and reserve full `sessions.list` reloads for initial load, manual refresh, filter changes, or events that cannot be patched safely, while keeping Gateway row enrichment bounded for large stores.

## Reproduction Assessment

Yes. The issue has concrete v2026.4.29 timing/log evidence from large stores, and current-main source plus tests show a direct `sessions.changed` to `sessions.list` reload path still exists.

## Solution Assessment

Yes. The narrow maintainable direction is to land PR #76277 or an equivalent change that removes redundant UI full-list invalidations and avoids extra Gateway lookup work, then validate with the targeted Control UI and Gateway session tests.
