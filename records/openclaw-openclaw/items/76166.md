## Summary

Keep open. The fixed 8-second polling shape appears gone, but current main still handles every `sessions.changed` event by issuing a full `sessions.list`, and the list path still does per-row work that can be expensive on large stores.

## What This Changes

The issue asks OpenClaw to stop Control UI from repeatedly invoking expensive `sessions.list` calls on large session stores in v2026.4.29 and current builds.

## Best Possible Solution

Control UI should apply ordinary `sessions.changed` row payloads incrementally and reserve `sessions.list` for initial load, manual refresh, filter changes, reconnect/gap recovery, or events that cannot be reconciled locally.

## Reproduction Assessment

Yes. The released symptom has concrete v2026.4.29 logs and a second matching user report, and current-main source inspection shows the remaining event-triggered full reload path even though live reproduction was not run.

## Solution Assessment

Yes. The narrow maintainable fix is to stop treating every pushed session notification as a full-list invalidation, then keep server-side `sessions.list` bounded or cached if large stores still make manual/full refresh expensive.

## Repair Work Prompt

Fix the Control UI/Gateway session refresh regression tracked by https://github.com/openclaw/openclaw/issues/76166. Current main subscribes to `sessions.changed`, but `ui/src/ui/app-gateway.ts` still calls `loadSessions` for every event, which turns pushed updates into repeated full `sessions.list` calls on large stores. Update the UI/session controller so ordinary `sessions.changed` events with a session row payload update `sessionsResult` incrementally without immediately reissuing `sessions.list`; keep full list refresh for initial load, manual refresh, filter/limit changes, reconnect or event-gap recovery, deletes/cleanup, and events that cannot be applied locally. Preserve existing session row fields, checkpoint-cache invalidation behavior, active-session/chat history behavior, and third-party protocol compatibility. Add focused tests in the Control UI event/controller tests proving a normal `sessions.changed` event no longer calls `loadSessions`, while unreconcilable or explicitly full-refresh cases still do. If inspection shows the server still needs a small bounded-list/cache tweak, keep it limited to `src/gateway/server-methods/sessions.ts` or `src/gateway/session-utils.ts`; do not add broad migrations or plugin-specific behavior. Add a user-facing `CHANGELOG.md` fix entry unless the final patch is test-only/internal.
