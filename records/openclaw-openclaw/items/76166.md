## Summary

Keep open. The old fixed-interval polling report was closed after Control UI moved to `sessions.subscribe`, but current main still reloads the full session list on every `sessions.changed` event, and the server-side list path can still perform per-row transcript-derived work on large stores.

## What This Changes

The issue reports repeated expensive Control UI `sessions.list` calls on v2026.4.29 against large session stores, causing Gateway latency, GC pressure, and RPC timeouts.

## Best Possible Solution

Control UI should use `sessions.changed` row payloads for ordinary create/send/patch/compact updates without immediately reissuing `sessions.list`, while `sessions.list` should keep transcript-derived work bounded, cached, or explicitly requested for large stores.

## Reproduction Assessment

Yes for the released symptom: the issue and follow-up comment provide concrete v2026.4.29 logs, large session-store sizes, timing samples, and a browser-tab mitigation. Current-main live reproduction was not run, but source inspection confirms the remaining event-triggered full reload path.

## Solution Assessment

Yes, the maintainable fix is to stop treating every push notification as a full-list invalidation and reserve `sessions.list` for initial load, manual refresh, filters, and events that cannot be applied locally. If that still leaves slow calls, the safer follow-up is a server-side cached/lightweight list mode rather than longer polling intervals.
