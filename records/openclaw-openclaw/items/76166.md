## Summary

Keep open. Current main still has the event-triggered full reload path, and the only specifically linked fix PR is closed unmerged; the report has concrete release logs and a narrow replacement-fix path.

## What This Changes

The issue asks Control UI/Gateway to stop repeated expensive `sessions.list` calls on large session stores, especially after `sessions.changed`-based refreshes in v2026.4.29.

## Best Possible Solution

Land a focused UI/Gateway fix that uses pushed session row payloads for ordinary message-phase updates and reserves full `sessions.list` reloads for initial load, manual refresh, filters, and events that cannot be applied locally.

## Reproduction Assessment

Yes. The release report and follow-up comment give concrete v2026.4.29 logs and mitigation evidence, and current main source inspection shows the same `sessions.changed` to `loadSessions` to `sessions.list` path still exists.

## Solution Assessment

Yes, a narrow reload-avoidance fix is the best first step. If slow initial/manual lists remain after that, a separate server-side cache or lightweight list mode would be the safer follow-up.
