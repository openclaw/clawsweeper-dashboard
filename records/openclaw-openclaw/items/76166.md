## Summary

Keep open. Current main subscribes to `sessions.changed`, but the Control UI still turns every pushed `sessions.changed` event into a full `sessions.list` refresh even though the pushed payload already carries session-row metadata.

## What This Changes

Stop Control UI from repeatedly issuing expensive `sessions.list` calls on large session stores when pushed `sessions.changed` updates should suffice.

## Best Possible Solution

Make Control UI rely on rich `sessions.changed` payloads for row updates and reserve full `sessions.list` calls for initial load, filter changes, manual refresh, and explicit fallback paths.

## Reproduction Assessment

Yes. The current-main source path is deterministic: a `sessions.changed` event reaches `handleGatewayEvent`, which applies the payload and then calls `loadSessions`, and `loadSessionsOnce` sends `sessions.list`.

## Solution Assessment

No. The current implementation combines push events with a full-list reload, so the narrow maintainable fix is to make the pushed row update the normal path and fall back to `sessions.list` only when the event cannot update the displayed state safely.
