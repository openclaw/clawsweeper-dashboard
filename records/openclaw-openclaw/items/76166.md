## Summary

Keep open. Current main has post-release session-list performance improvements, but the Control UI still turns every `sessions.changed` event into a `sessions.list` reload, so the reported pressure pattern is not fully resolved.

## What This Changes

The issue reports Control UI repeatedly issuing expensive `sessions.list` calls on v2026.4.29 despite the prior subscribe-based fix.

## Best Possible Solution

Control UI should treat row-carrying `sessions.changed` events as authoritative for ordinary session updates and reserve `sessions.list` for initial load, filter changes, manual refresh, or non-row invalidations while keeping Gateway list construction bounded.

## Reproduction Assessment

Yes for the released symptom: the report and follow-up comment provide concrete v2026.4.29 logs, session-store sizes, timing samples, and browser-tab mitigation. Source inspection on current main confirms a deterministic remaining path from `sessions.changed` to `loadSessions` to `sessions.list`.

## Solution Assessment

Yes, the narrow maintainable fix is to stop treating every pushed session update as a full-list invalidation. If slow calls remain after that, the safer follow-up is server-side cached/lightweight list behavior rather than longer polling intervals.
