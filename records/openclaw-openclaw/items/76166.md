## Summary

Keep open. Current main still reloads the full session list after every `sessions.changed` event, so the released v2026.4.29 reports are not obsolete until the linked WebUI latency PR or an equivalent fix lands.

## What This Changes

The issue reports repeated expensive Control UI `sessions.list` calls on large v2026.4.29 session stores and asks for push/event handling and Gateway list cost to be bounded.

## Best Possible Solution

Land or revise #76277 so ordinary hot `sessions.changed` payloads update Control UI state without a full `sessions.list`, while keeping server-side list work bounded for large stores.

## Reproduction Assessment

Yes for the released symptom: the issue body and corroborating comment include concrete v2026.4.29 logs, large session-store sizes, repeated `sessions.list` timings, and browser-tab mitigation. Current-main source inspection also confirms the remaining event-triggered full reload path.

## Solution Assessment

Yes, the linked PR direction is the narrow maintainable fix: use pushed row payloads for ordinary message/update events and reserve full `sessions.list` for initial load, filters, manual refresh, or incomplete events. Server-side caching/lightweight rows remain a useful secondary guard if slow calls persist.
