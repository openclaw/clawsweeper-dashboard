## Summary

Keep open. The released reports are concrete, and current main still converts every `sessions.changed` push into a full `sessions.list` refresh, so the earlier subscription work does not yet remove the repeated expensive list path.

## What This Changes

The issue asks OpenClaw to stop Control UI/WebChat from repeatedly issuing slow `sessions.list` calls on large session stores and to bound the Gateway work behind that RPC.

## Best Possible Solution

Use `sessions.changed` row payloads as the normal Sessions table update path and reserve `sessions.list` for initial load, explicit refresh/filter changes, reconnect reconciliation, or events that cannot be applied locally; then keep server-side list work bounded and cacheable for large stores.

## Reproduction Assessment

Yes. The issue and corroborating comment provide a high-confidence release reproduction, and current-main source inspection confirms the same event-triggered full-list reload path still exists; I did not run a live large-store repro in this read-only review.

## Solution Assessment

Yes. The narrow maintainable direction is to stop treating every push event as a full-list invalidation before increasing intervals or adding broader Gateway policy; server-side lightweight/cached list work is the safer follow-up if slow calls remain.
