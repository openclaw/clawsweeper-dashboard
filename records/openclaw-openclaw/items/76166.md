## Summary

Keep open. Current main has the subscription path from the earlier polling fix, but every `sessions.changed` notification still triggers a full `sessions.list` reload, matching the v2026.4.29 large-store latency reports.

## What This Changes

Control UI should stop repeatedly issuing expensive `sessions.list` calls on large session stores and keep Gateway list work bounded.

## Best Possible Solution

Control UI should apply ordinary `sessions.changed` row payloads locally and reserve `sessions.list` for initial load, manual refresh, filter changes, and events that cannot be represented incrementally; server list work should remain bounded or cached for large stores.

## Reproduction Assessment

Yes. The released symptom has concrete v2026.4.29 logs, large-store sizes, timing samples, and a browser-tab mitigation, and current-main source inspection confirms the remaining event-triggered full reload path.

## Solution Assessment

Yes. The narrowest maintainable fix is to stop using every push notification as a full-list invalidation, with server-side caching or lightweight list mode as a follow-up only if explicit list calls remain too expensive.
