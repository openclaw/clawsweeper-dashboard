## Summary

Keep open. Current main and the latest release still expose only the existing Xiaomi MiMo V2 Flash/Pro/Omni catalog entries and there is no `multimodalFallback` config key; the request is useful feature work, but it combines provider metadata with a new compatibility-sensitive routing contract that needs maintainer design first.

## What This Changes

Request to add `xiaomi/mimo-v2.5` to the bundled Xiaomi provider catalog and introduce automatic attachment-capability routing from text-only primary models to a multimodal fallback.

## Best Possible Solution

Split the work into a verified Xiaomi plugin catalog/docs/onboarding update and a separately designed generic attachment-capability routing policy that reuses the existing `imageModel` path where possible and adds new config only with maintainer approval.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a feature request; source inspection confirms the requested catalog entry and `multimodalFallback` key are absent, but I did not live-reproduce the reported attachment error/stripping path.

## Solution Assessment

Is this the best way to solve the issue? Unclear; adding `multimodalFallback` exactly as proposed is not obviously the best path because OpenClaw already has `imageModel` fallback semantics for images, while video/audio routing needs a broader capability contract.
