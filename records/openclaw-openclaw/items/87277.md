## Summary

Keep open. Current main and the latest release have partially added Xiaomi Token Plan MiMo-V2.5 support, but the pay-as-you-go Xiaomi catalog still lacks `xiaomi/mimo-v2.5`, the shipped metadata does not match the requested video/audio-capable row, and there is no `multimodalFallback` config or generic pre-dispatch multimodal rerouting contract.

## What This Changes

Requested adding Xiaomi MiMo-V2.5 catalog/onboarding/docs metadata and automatic per-turn routing to a multimodal fallback when a text-only primary model receives image, video, or audio attachments.

## Best Possible Solution

Split the already-shipped Token Plan support from the remaining work: verify Xiaomi V2.5 metadata in the Xiaomi plugin/docs, then design a generic attachment-capability routing policy that extends existing `imageModel`/model-fallback semantics only if maintainers approve the new config contract.

## Solution Assessment

Is this the best way to solve the issue? Not as a one-shot implementation yet; the catalog update is partly shipped through Token Plan, while generic multimodal fallback routing needs maintainer API design before adding a new config key.
