## Summary

Keep open. Current main still lacks the requested generic RPM/RPD-aware fallback cooldown contract: provider config/schema do not accept quota fields, auth-profile usage state has no minute/day quota counters, and fallback preflight only reacts to existing cooldown state or prior failures. Recent usage-window infrastructure is for status/reporting and the related usage-aware routing issue remains open, not a merged canonical replacement.

## What This Changes

The issue requests per-provider RPM/RPD quota configuration that computes exact cooldownUntil windows and skips fallback candidates before sending requests that would exceed the configured quota.

## Best Possible Solution

Define an opt-in generic quota contract, persist exact minute/day request state with reset-time calculation, integrate pre-request fallback skipping with cooldownUntil under the auth-profile store lock, and update config schema/help/docs plus focused auth-profile and model-fallback tests while coordinating scope with #38775.

## Reproduction Assessment

Not applicable as a runtime bug reproduction because this is a feature request. Source inspection gives a high-confidence verification path that the requested contract is absent across config/schema, auth-profile state, docs, and fallback preflight.

## Solution Assessment

Unclear until maintainer review. Quota-aware cooldowns are plausible, but the narrowest maintainable solution needs an API boundary decision because the request crosses public config, persisted auth state, provider/plugin semantics, and concurrency behavior.
