## Summary

Keep open. Current main still lacks the requested generic RPM/RPD-aware fallback cooldown contract: provider config has no quota fields, auth-profile usage state has no minute/day counters, and fallback preflight only reacts to existing cooldowns; the OpenAI Codex WHAM path is provider-specific rather than a generic quota API.

## What This Changes

Request generic per-provider RPM/RPD-aware cooldowns so model fallback can skip quota-exhausted providers until the relevant reset window.

## Best Possible Solution

Define an opt-in generic quota contract, persist exact minute/day quota state with reset-time calculation, integrate pre-request fallback skipping with cooldownUntil under the auth-profile store lock, and update config schema/help/docs plus focused tests.

## Reproduction Assessment

Not applicable as a runtime bug reproduction because this is a feature request. Source inspection gives a high-confidence verification path that the requested contract is absent across config/schema, auth-profile state, fallback preflight, docs, and tests.

## Solution Assessment

Unclear until maintainer review. Quota-aware cooldowns are plausible, but the narrowest maintainable solution needs an API boundary decision because the request crosses public config, persisted auth state, provider/plugin semantics, and concurrent writes.
