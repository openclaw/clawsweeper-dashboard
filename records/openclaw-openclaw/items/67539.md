## Summary

Keep open. Current main still only builds generic TTS prompt guidance and the speech-provider contract has no provider-owned prompt-hint hook, while the Google Gemini provider and docs confirm expressive tags are supported only once the model already emits them.

## What This Changes

The issue requests provider-owned, model-aware TTS prompt hints so runtime system prompts can advertise only the active speech provider’s expressive controls.

## Best Possible Solution

Add a backwards-compatible provider prompt-hint hook to the speech-provider contract, have speech-core append only the selected provider's short hint when TTS is enabled, let Google own Gemini audio-tag wording, and cover active-provider-only plus policy-gated behavior in tests and docs.

## Reproduction Assessment

Yes. A high-confidence reproduction is source inspection on current main: enable TTS and build the system prompt, and it can only include generic TTS guidance because no provider hook or provider context is present.

## Solution Assessment

Yes. The provider-owned hook is the narrowest maintainable direction because it keeps vendor/model syntax in the owning provider instead of hard-coding Gemini guidance in speech-core or relying on optional skills/docs.

Add a backwards-compatible optional prompt-hint hook to the speech-provider contract, pass enough context for the active provider to decide whether to emit a short hint (resolved model, voice/provider config, modelOverrides, and target where feasible), and append only the selected provider's hint from speech-core when TTS is enabled.
