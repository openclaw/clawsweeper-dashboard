## Summary

Keep open: current main still lacks a stale replay fix, but this PR is not merge-ready because the `app-chat` guard can drop legitimate same-text draft commits and the supplied proof is still source/unit-test only despite later comments saying CI is green.

## What This Changes

The PR adds a one-shot `pendingDraftClear` marker in `ui/src/ui/app-chat.ts` plus app-chat tests to ignore the first post-send draft change that matches the submitted text.

## Best Possible Solution

Land one canonical chat-view composer-mirror stale-replay guard that preserves same-text re-entry, likely by rebasing/adapting https://github.com/openclaw/openclaw/pull/89714 or reworking this PR, with real Control UI browser proof.

## Reproduction Assessment

Yes, source-reproducible. The linked issue gives browser steps, and current main can clear the local mirror after send then later commit a replayed textarea value through `syncComposerValue` while the send is active.

## Solution Assessment

No. This is a plausible mitigation, but `app-chat` only sees the next draft string and cannot distinguish stale replay from intentional same-text full-value input; the chat-view input boundary is the safer fix location.
