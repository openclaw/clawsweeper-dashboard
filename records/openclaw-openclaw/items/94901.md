## Summary

Keep open. This is a focused Control UI bug-fix PR, but it is not merge-ready because the app-level stale-replay guard can also discard a legitimate full same-text re-entry after send, and the supplied proof is source/test-only rather than real browser behavior.

## What This Changes

The PR adds an app-chat `pendingDraftClear` marker that drops the first same-session draft change matching the just-submitted text after send, plus app-chat tests for stale replay and different new input.

## Best Possible Solution

Put the stale-replay discriminator at the Control UI composer mirror boundary, preserve intentional same-text re-entry with focused tests, and require real browser proof before merge.

## Reproduction Assessment

Yes, at source level. Current main clears the host/local draft after send, but the chat view can later commit a replayed textarea value through `onDraftChange` while the send is active; the linked issue provides browser steps for the visible symptom.

## Solution Assessment

No. The patch is a plausible mitigation, but the app-chat string-only guard is too coarse; the maintainable fix should distinguish stale replay from real input in the composer/input path and prove the visible browser flow.
