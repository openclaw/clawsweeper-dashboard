## What This Changes

The PR adds a one-shot `pendingDraftClear` marker to `app-chat` and tests that a replayed submitted draft is ignored while a different follow-up draft is accepted.

## Best Possible Solution

Move the stale-replay discriminator to the composer mirror/input boundary, preserve intentional same-text re-entry with focused coverage, and add real browser proof; alternatively land a rebased equivalent canonical PR.

## Reproduction Assessment

Yes, at source level. Current main clears the host draft after send, but the chat view can later commit a replayed textarea value through `syncComposerValue` while the send is active; the linked issue provides browser steps for the visible symptom.

## Solution Assessment

No. This is a plausible mitigation, but the app-chat string-only guard is not the best boundary because it cannot tell stale replay from intentional same-text full-value input.
