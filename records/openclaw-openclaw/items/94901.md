## What This Changes

Adds a one-shot `pendingDraftClear` marker to `ChatHost`, wraps `handleChatDraftChange`, and adds app-chat tests for stale replay after send clears the composer.

## Best Possible Solution

Move the stale-replay discriminator to the chat-view composer/input mirror, preserve intentional same-text re-entry with focused tests, and land one canonical fix with real Control UI browser proof.

## Reproduction Assessment

Yes, source-reproducible. The issue gives browser steps, and current main can clear the host draft after send while a later textarea input path still commits `target.value` through `syncComposerValue`.

## Solution Assessment

No. This is a plausible mitigation, but `app-chat` only receives draft strings and cannot distinguish stale replay from intentional same-text full-value input; the safer fix belongs at the composer/input boundary.
