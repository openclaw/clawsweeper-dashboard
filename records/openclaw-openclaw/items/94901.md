## Summary

Keep open: this is a focused Control UI bug-fix PR, but the app-chat guard is too coarse and can drop legitimate same-text input after send, and the contributor proof is still source/unit-test only despite the follow-up comments.

## What This Changes

The branch adds a `pendingDraftClear` marker in `app-chat` plus app-chat tests to ignore a same-session submitted-draft replay after send clears the composer.

## Best Possible Solution

Move the stale-replay discriminator to the chat-view composer/input mirror or land the proof-positive candidate after rebase, preserving intentional same-text re-entry with real Control UI browser proof.

## Reproduction Assessment

Yes, source-reproducible. The linked issue gives browser steps, and current main can clear the local mirror after send then later commit a replayed textarea value through `syncComposerValue` while the send is active.

## Solution Assessment

No. This is a plausible mitigation, but `app-chat` only receives the next draft string and cannot distinguish stale replay from intentional same-text full-value input; the chat-view input boundary is the safer fix location.
