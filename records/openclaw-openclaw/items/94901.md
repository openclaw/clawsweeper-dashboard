## Summary

Keep open; the linked bug is real, but this PR is not merge-ready because the app-chat string-only guard can drop legitimate same-text input after send and the contributor proof is still source/unit-test only.

## What This Changes

The PR adds a `pendingDraftClear` marker to `ui/src/ui/app-chat.ts` and app-chat tests to ignore a replayed submitted draft after send clears the composer.

## Best Possible Solution

Move the stale-replay discriminator to the chat-view composer/input mirror, preserve intentional same-text re-entry with focused tests, and land one canonical fix with real Control UI browser proof.

## Reproduction Assessment

Yes, source-reproducible. The issue gives browser steps, and current main can clear the local mirror after send then later commit a replayed textarea value through `syncComposerValue` while the send is active.

## Solution Assessment

No. This is a plausible mitigation, but `app-chat` only sees the next draft string and cannot distinguish stale replay from intentional same-text full-value input; the chat-view input boundary is the safer fix location.
