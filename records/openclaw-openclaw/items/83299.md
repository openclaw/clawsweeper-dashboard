## Summary

Keep open: the benchmark direction fits the landed Gateway restart instrumentation, but the diff has a source-reproducible probe-loop race that can mark successful restarts as timeouts, and the external-PR proof does not show a full restart benchmark run on this branch.

## What This Changes

This PR adds a `pnpm test:restart:gateway` package script, a new repeated Gateway restart benchmark CLI, and focused tests for its parsing, fixture, trace, and probe helpers.

## Best Possible Solution

Keep the benchmark as developer tooling, but make the HTTP recovery probe authoritative after an unavailable window, add regression coverage for the ready-trace race, and attach redacted terminal/log proof from a real built Gateway benchmark run.

## Reproduction Assessment

Yes for the review finding: source inspection shows the probe loop can stop on `hasRestartReadySignal` before taking a final HTTP recovery sample. I did not execute the benchmark because this was a read-only checkout review.

## Solution Assessment

No, not yet. The new benchmark belongs near the existing startup benchmark, but the probe exit condition should not treat the ready trace as a substitute for confirming `/healthz` and `/readyz` recovery after an unavailable window.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Keep probing until HTTP recovers:** `scripts/bench-gateway-restart.ts:617-618`
  - body: `isDone` becomes true as soon as the restart ready trace is parsed, and this loop checks it before issuing another request. If the endpoint already had a non-200/error sample, the loop can break before sampling the recovered 200 and returns `ms: null`, which is later reported as `next_*_timeout` even though the Gateway recovered. Keep polling until the endpoint returns 200 after the unavailable window, or take one final probe when the ready trace arrives.
  - confidence: 0.88
