## Summary

Keep open. Current main and the latest release still parse exponent-form values such as `1e5` as strings on the shared config-value path, and the PR is a focused parser/test change with no functional or security regression found. The remaining blockers are a required user-facing changelog entry and maintainer agreement that unquoted exponent-looking values should now coerce to numbers.

## What This Changes

The PR extends `parseConfigValue` numeric detection to accept exponent notation and adds a colocated Vitest file covering existing value parsing plus scientific notation.

## Best Possible Solution

Add the required Unreleased Fixes changelog entry, then land or adapt the parser/test change if maintainers agree that unquoted exponent notation should be numeric on `/config`, `/mcp`, `/debug`, and `openclaw mcp set`; otherwise document the string-preserving policy and close the PR.

## Reproduction Assessment

Yes. The current source and a read-only regex check reproduce the issue directly: the existing regex rejects exponent notation and returns those values as strings through the shared parser path.

## Solution Assessment

Yes, with a policy caveat. The proposed regex is a narrow maintainable change that preserves the existing integer, decimal, boolean, null, JSON, and quoted-string branches, but maintainers still need to accept the value-coercion behavior and the PR needs a changelog entry.
