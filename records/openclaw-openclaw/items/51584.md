## Summary

Keep open: the static built-in channel fallback is a useful part of the fix, but this PR does not normalize the reported bare `telegram:slash:<id>` session key before channel inference, so the stated path can still resolve as unknown.

## What This Changes

The PR imports `normalizeChatChannelId` and uses it as a fallback after `normalizeAnyChannelId` in `sandbox explain` channel inference.

## Best Possible Solution

Land one reconciled fix that normalizes bare channel session keys, falls back to bundled channel ids when the plugin registry is empty, and covers `allowedByConfig` with regression coverage and redacted real CLI proof.

## Reproduction Assessment

Yes, by source inspection. The reported `telegram:slash:<id>` input remains non-agent-scoped, fails agent-session parsing, and therefore never reaches this PR's static channel fallback.

## Solution Assessment

No. The static fallback is a necessary no-registry piece, but the reported bare session key also needs normalization before this fallback is reachable.
