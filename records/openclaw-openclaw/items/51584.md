## Summary

Keep open: the PR is a useful half-fix, but it does not fix the PR body's bare `telegram:slash:<id>` path because that input fails agent-scoped parsing before the new fallback can run. The linked open PR https://github.com/openclaw/openclaw/pull/51286 covers the complementary key-normalization half, so maintainers should reconcile these into one focused fix rather than merge this branch as-is.

## What This Changes

This PR imports `normalizeChatChannelId` and uses it as a fallback after runtime channel-registry normalization in `sandbox explain` channel inference.

## Best Possible Solution

Land one consolidated sandbox-explain fix that wraps non-agent-scoped colon session keys, uses registry-independent bundled-channel fallback for diagnostics, and covers bare plus agent-scoped Telegram slash sessions with focused tests.

## Reproduction Assessment

Yes. Source inspection on current main shows `--session telegram:slash:<id>` remains raw, fails `parseAgentSessionKey`, leaves `channel` undefined, and therefore cannot make `allowedByConfig` true.

## Solution Assessment

No. The static fallback is necessary for the no-plugin-registry CLI path, but this PR does not normalize or infer bare colon session keys before relying on that fallback.
