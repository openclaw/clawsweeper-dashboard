## Summary

Keep this PR open, but it is not merge-ready: the static channel fallback is useful, yet the reported bare `telegram:slash:<id>` path still exits before that fallback is reachable. The complementary open PR at https://github.com/openclaw/openclaw/pull/51286 covers key normalization, and this PR still lacks after-fix real behavior proof.

## What This Changes

The PR imports `normalizeChatChannelId` and uses it as a static fallback after `normalizeAnyChannelId` in `sandbox explain` channel inference.

## Best Possible Solution

Land one reconciled fix that normalizes non-agent-scoped channel session keys, falls back to bundled channel ids when plugins are unloaded, and covers the elevated `sandbox explain --session telegram:slash:<id>` output with regression coverage and redacted real CLI proof.

## Reproduction Assessment

Yes, by source inspection. The reported `telegram:slash:<id>` input remains unscoped on current main, fails agent-session parsing, leaves `channel` undefined, and keeps `allowedByConfig` false.

## Solution Assessment

No. The static fallback is a necessary piece for no-registry CLI inference, but the reported bare session key also needs normalization before this fallback is reachable.
