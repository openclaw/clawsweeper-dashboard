## Summary

Keep open: this PR covers the static-registry fallback half of the sandbox explain bug, but the stated `telegram:slash:<id>` CLI input is still returned as a bare key before channel inference, so the new fallback is unreachable for the reported command. The branch also needs real behavior proof and reconciliation with the complementary open fix at https://github.com/openclaw/openclaw/pull/51286.

## What This Changes

The PR imports `normalizeChatChannelId` and uses it as a fallback after `normalizeAnyChannelId` when `sandbox explain` infers a channel from a session key.

## Best Possible Solution

Land one reconciled fix that normalizes non-agent-scoped channel session keys, resolves bundled channels when plugins are unloaded, and adds regression coverage plus redacted real CLI proof.

## Reproduction Assessment

Yes, by source inspection. The reported `telegram:slash:<id>` input remains non-agent-scoped, `parseAgentSessionKey` rejects it, and this PR's fallback is therefore unreachable for the stated command.

## Solution Assessment

No. The static fallback is a necessary no-registry piece, but the reported path also needs bare session-key normalization before that fallback can run.
