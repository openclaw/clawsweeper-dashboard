## Summary

Keep open: the static channel fallback is useful, but this PR still misses the reported bare `telegram:slash:<id>` path because that key is rejected before the new fallback can run. It also lacks after-fix real CLI proof and overlaps the complementary open fix at https://github.com/openclaw/openclaw/pull/51286.

## What This Changes

The PR imports `normalizeChatChannelId` and falls back to it after `normalizeAnyChannelId` when `sandbox explain` infers a channel from an agent-scoped session key.

## Best Possible Solution

Land one reconciled fix that wraps non-agent-scoped channel session keys, falls back to bundled channel ids when plugins are unloaded, adds focused regression coverage, and includes redacted real CLI proof.

## Reproduction Assessment

Yes, by source inspection. On current main, `telegram:slash:<id>` remains unscoped, fails agent-session parsing, leaves `channel` undefined, and therefore keeps `allowedByConfig` false.

## Solution Assessment

No. The static fallback is necessary, but this PR is not sufficient without the non-agent-scoped session-key normalization covered by https://github.com/openclaw/openclaw/pull/51286.
