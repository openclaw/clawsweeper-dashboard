## Summary

Keep open: the static bundled-channel fallback is a useful half of the fix, but this PR does not address the raw `telegram:slash:<id>` path because that key is returned unchanged and exits channel inference before the new fallback can run.

## What This Changes

The PR imports `normalizeChatChannelId` and uses it as a fallback after `normalizeAnyChannelId` when inferring a `sandbox explain` channel from an agent-scoped session key.

## Best Possible Solution

Land one reconciled fix that normalizes or infers bare channel-prefixed keys, falls back to bundled channel ids when plugins are unloaded, adds regression coverage for `telegram:slash:<id>` elevated diagnostics, and includes redacted CLI proof.

## Reproduction Assessment

Yes, by source inspection on current main: a raw `telegram:slash:<id>` key is returned unchanged, parsing requires an `agent:` key, and unresolved channel identity keeps `allowedByConfig` false. I did not run live Telegram or CLI proof in this read-only review.

## Solution Assessment

No. The static fallback is necessary for agent-scoped keys when plugins are unloaded, but this PR alone is not sufficient because the exact reported bare session key never reaches that fallback.
