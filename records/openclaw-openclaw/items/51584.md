## Summary

Keep this PR open: it addresses the static channel-registry fallback half of the Telegram slash-session bug, but it still leaves the reported bare `telegram:slash:<id>` input unscoped before inference, so the fallback is not reached for the stated CLI path. It also still lacks required after-fix real behavior proof from the contributor.

## What This Changes

The PR imports `normalizeChatChannelId` and uses it as a static fallback after `normalizeAnyChannelId` in `sandbox explain` channel inference.

## Best Possible Solution

Land one consolidated fix that normalizes non-agent-scoped channel session keys, falls back to bundled channel ids when plugins are unloaded, and covers `sandbox explain --session telegram:slash:<id>` elevated output with regression coverage and redacted real CLI proof.

## Reproduction Assessment

Yes, by source inspection. The reported `telegram:slash:<id>` input remains unscoped, fails `parseAgentSessionKey`, leaves `channel` undefined, and therefore keeps `allowedByConfig` false.

## Solution Assessment

No. The static fallback is a necessary piece for no-registry CLI inference, but the reported bare session key also needs normalization before this fallback is reachable.
