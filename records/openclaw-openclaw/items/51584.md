## Summary

Keep this PR open, but do not merge it as-is. The static channel fallback is a useful part of the fix, yet it only runs after agent-scoped parsing succeeds, so it does not fix the documented `openclaw sandbox explain --session telegram:slash:<id>` path from the PR body and linked bug.

## What This Changes

This PR imports `normalizeChatChannelId` into `src/commands/sandbox-explain.ts` and uses it as a fallback after runtime channel-registry normalization when inferring a channel from an agent-scoped session key.

## Best Possible Solution

Land one consolidated sandbox-explain fix that normalizes non-agent-scoped colon session keys before store lookup and inference, applies bundled-channel fallback consistently for inferred and stored channel metadata, and adds regression coverage for bare and agent-scoped Telegram slash sessions with an empty runtime channel registry.

## Reproduction Assessment

Yes. The failure path is source-reproducible on current main: `telegram:slash:<id>` is preserved as a raw colon key, `parseAgentSessionKey` rejects it, and the proposed fallback is never reached.

## Solution Assessment

No. The fallback is a useful ingredient, but the patch is too narrow because it only handles agent-scoped inference and misses the documented bare session-key form that triggered the bug.

## Review Findings
