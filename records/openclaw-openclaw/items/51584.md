## Summary

Keep open: this is a useful part of the sandbox-explain fix, but it is not mergeable alone because the reported bare `telegram:slash:<id>` input never reaches the new fallback. The PR also lacks after-fix real behavior proof and overlaps with the complementary open fix at https://github.com/openclaw/openclaw/pull/51286.

## What This Changes

This PR imports `normalizeChatChannelId` and uses it as a fallback after runtime channel-registry normalization in `sandbox explain` channel inference.

## Best Possible Solution

Land one consolidated sandbox-explain fix that wraps non-agent-scoped colon session keys, uses registry-independent bundled-channel fallback, and covers bare plus agent-scoped Telegram slash elevated diagnostics with focused tests.

## Reproduction Assessment

Yes. Source inspection shows `telegram:slash:<id>` remains raw, fails agent-session parsing, leaves `channel` undefined, and the PR's fallback is unreachable for that reported input.

## Solution Assessment

No. The static fallback is necessary for the no-plugin-registry path, but this PR does not normalize or infer bare colon session keys before relying on that fallback.
