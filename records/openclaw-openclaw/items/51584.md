## Summary

Keep open and do not merge as-is. The static fallback addresses one real root cause, but the proposed patch still leaves the documented bare `telegram:slash:<id>` CLI path unresolved because the fallback only runs after agent-scoped session parsing succeeds.

## What This Changes

The PR imports `normalizeChatChannelId` in `src/commands/sandbox-explain.ts` and uses it after `normalizeAnyChannelId` when inferring a channel from an agent-scoped session key.

## Best Possible Solution

Land one consolidated sandbox-explain fix that normalizes non-agent-scoped colon session keys before store lookup and inference, applies bundled-channel fallback consistently for inferred and stored channel metadata, and adds regression coverage for bare and agent-scoped Telegram slash sessions with an empty runtime channel registry.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: pass `--session telegram:slash:<id>`, observe that it remains raw, fails `parseAgentSessionKey`, never reaches the fallback, and leaves elevated channel resolution undefined.

## Solution Assessment

No. The fallback is a useful ingredient, but it is not the complete fix because the reported bare session form still bypasses the patched code path; combine it with the normalization approach from #51286 or equivalent bare-key inference.
