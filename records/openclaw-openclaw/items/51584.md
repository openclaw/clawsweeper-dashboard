## Summary

Keep open: the static-registry fallback is useful for agent-scoped session keys, but this PR does not fix the reported `telegram:slash:<id>` CLI path because non-agent colon keys are returned unchanged and rejected before the new fallback runs.

## What This Changes

The PR imports `normalizeChatChannelId` and falls back to it when `inferProviderFromSessionKey` cannot resolve a channel candidate through the runtime registry.

## Best Possible Solution

Land one canonical fix that normalizes non-agent-scoped channel session keys before inference, keeps the bundled static channel fallback for unloaded registries, and adds focused regression coverage for the reported Telegram slash/elevated allowFrom path.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: `openclaw sandbox explain --session telegram:slash:<id>` remains raw at `normalizeExplainSessionKey`, `parseAgentSessionKey` rejects it, and the changed fallback is never reached.

## Solution Assessment

No. The fallback is a useful part of the fix, but the narrow maintainable solution must also normalize or handle non-agent-scoped channel session keys and prove it with a regression test.
