## Summary

Keep open: the static-channel fallback is useful, but this PR alone still does not handle the reported bare `telegram:slash:<id>` input and it overlaps the complementary open fix at https://github.com/openclaw/openclaw/pull/51286.

## What This Changes

The PR imports `normalizeChatChannelId` and uses it as a fallback after `normalizeAnyChannelId` when `sandbox explain` infers a channel from an agent-scoped session key.

## Best Possible Solution

Land one reconciled fix that normalizes raw channel-prefixed session keys, falls back to bundled channel IDs when plugins are unloaded, adds regression coverage for the elevated diagnostic path, and includes redacted real behavior proof.

## Reproduction Assessment

Yes. Source inspection on current main shows `telegram:slash:<id>` remains unscoped, `parseAgentSessionKey` rejects it, and unresolved channel identity keeps `allowedByConfig` false.

## Solution Assessment

No. The static fallback is necessary for agent-scoped keys when plugins are unloaded, but this PR alone is not sufficient because the exact reported bare session key never reaches that fallback.
