## Summary

This PR should stay open but not merge as-is: the static channel fallback is useful, yet the exact reported bare `telegram:slash:<id>` input still fails before the new fallback runs, and the external PR only has unit-test proof.

## What This Changes

The PR imports `normalizeChatChannelId` and uses it as a fallback after `normalizeAnyChannelId` while inferring the `sandbox explain` channel from an agent-scoped session key.

## Best Possible Solution

Land one reconciled fix that wraps non-agent-scoped channel session keys, falls back to bundled channel ids when plugins are unloaded, adds focused regression coverage, and includes redacted real CLI proof.

## Reproduction Assessment

Yes, by source inspection. On current main, `telegram:slash:<id>` stays unscoped, fails agent-session parsing, leaves `channel` undefined, and therefore keeps `allowedByConfig` false.

## Solution Assessment

No. The static fallback is necessary, but this PR is not sufficient without non-agent-scoped session-key normalization like the complementary open PR covers.
