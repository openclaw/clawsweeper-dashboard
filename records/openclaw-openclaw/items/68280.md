## Summary

Keep open: the PR targets a real gateway auth diagnostic problem, but latest head is not merge-ready because explicit URL/RPC override paths are still coupled to local auth credentials and the supplied live proof does not show the unresolved-SecretRef failure path.

## What This Changes

The PR threads gateway probe auth `failureReason` through status, gateway-status, and daemon status paths so unresolved probe credentials can short-circuit before a WebSocket connect, with focused regression tests.

## Best Possible Solution

Keep the local auth fail-fast diagnostic, but scope it to effective local shared-secret auth and make all explicit URL overrides use only explicit credentials or no credentials per the documented `--url` contract.

## Reproduction Assessment

Yes. Source inspection of PR head shows explicit targets still resolve local auth and daemon RPC URL overrides still forward `daemonProbeAuth`, while current docs and auth code define the expected explicit-credential and default-token contracts.

## Solution Assessment

No. The PR addresses the right failure class, but the maintainable fix needs one credential contract that separates local targets from explicit overrides and uses effective auth semantics instead of raw mode checks.

  - confidence: 0.9

  - body: This guard treats an unset `gateway.auth.mode` as open, but current auth resolution defaults unset mode to token and docs say gateway auth is required by default. A config with unset mode and an unresolved token SecretRef can still probe without surfacing the auth failure.
  - confidence: 0.84
