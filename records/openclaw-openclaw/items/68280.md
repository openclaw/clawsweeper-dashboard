## Summary

Keep this PR open: it is a plausible gateway bug fix, but the latest head still has source-reproducible fail-fast gaps for inferred local auth and lacks required real behavior proof. The diff also still needs explicit URL credential-forwarding review before merge.

## What This Changes

The PR threads gateway probe auth `failureReason` through daemon, status, and gateway-status paths so unresolved required local auth can fail before anonymous loopback probes, with focused regression tests.

## Best Possible Solution

Derive fail-fast from the effective credential plan or interactive auth result, keep explicit URL overrides from inheriting implicit local credentials, then add redacted real CLI/gateway proof.

## Reproduction Assessment

Yes, source-reproducible: with unset `gateway.auth.mode` and a configured but unresolved local `gateway.auth.token` or password SecretRef, the PR head skips `failureReason` and the anonymous probe path can still run. I did not run a live command because this review is read-only.

## Solution Assessment

No, not yet. The maintainable fix should use effective credential resolution for local fail-fast and apply the documented no-implicit-credentials rule to explicit URL overrides.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Use effective auth before treating unset mode as open:** `src/gateway/probe-auth.ts:183-185`
  - body: When `gateway.auth.mode` is unset, local credentials can still be active if `gateway.auth.token` or password is configured. This guard returns before checking the interactive/effective auth result, so an unresolved configured SecretRef produces only a warning and the anonymous loopback probe can still run, which is the behavior this PR is meant to prevent.
  - confidence: 0.87
- **[P2] Preserve inferred local auth in gateway status:** `src/commands/gateway-status/helpers.ts:189-198`
  - body: `authModeRequiresCredentials` is false for unset `auth.mode`, but `resolveGatewayProbeSurfaceAuth` still treats configured local token/password inputs as candidates in that case. A localLoopback target with an unresolved configured SecretRef therefore gets diagnostics without `failureReason`, and `runGatewayStatusProbePass` proceeds to probe without auth.
  - confidence: 0.85

## Security Review

Status: needs_attention

Summary: No dependency or workflow supply-chain issue was found, but explicit URL override credential forwarding remains unresolved in the touched auth paths.

Concerns:

- **[low] Do not forward local daemon credentials to RPC URL overrides:** `src/cli/daemon-cli/status.gather.ts:572`
  - body: When `probeUrlOverride` is set, the PR bypasses local fail-fast but still passes `daemonProbeAuth` into `probeGatewayStatus`; unless these were explicit CLI credentials, a user-supplied RPC URL can receive local configured credentials despite the no-implicit-credentials URL override contract.
  - confidence: 0.8

  - body: `resolveAuthForTarget` still maps `kind: "explicit"` to the local auth surface and returns config/env local credentials for that URL. That can send local gateway credentials to an arbitrary `--url` target; use explicit auth only or a host-aware policy matching the documented override contract.
  - confidence: 0.82
