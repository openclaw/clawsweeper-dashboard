## Summary

Keep open: the latest PR still uses raw `gateway.auth.mode` instead of effective local credential state, explicit URL paths still consult or forward implicit local credentials, and real behavior proof remains test-only.

## What This Changes

The PR threads gateway probe auth `failureReason` through daemon, status, and gateway-status probe paths so unresolved required local auth can skip anonymous WebSocket probes, with focused regression tests.

## Best Possible Solution

Keep the PR open until fail-fast uses the effective credential plan, explicit URL overrides require explicit credentials without consulting local config/env auth, and the contributor adds redacted live CLI/gateway proof.

## Reproduction Assessment

Yes. Source inspection of the PR head shows unset `gateway.auth.mode` returns before effective local SecretRef auth is considered, and explicit URL targets still resolve the local auth surface before probing.

## Solution Assessment

No, not yet. The narrow maintainable fix should derive failure from effective credential resolution and apply the documented explicit URL no-implicit-credentials rule before probing.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Use effective local auth before skipping unset mode:** `src/gateway/probe-auth.ts:183-185`
  - body: When `gateway.auth.mode` is unset but `gateway.auth.token` or password is configured through an unresolved SecretRef, local auth is still effectively active. This early return skips `failureReason`, so the anonymous loopback probe can still run instead of failing with the resolver diagnostic.
  - confidence: 0.88
- **[P2] Preserve unset-mode local fail-fast in gateway status:** `src/commands/gateway-status/helpers.ts:189-198`
  - body: `authModeRequiresCredentials` is false for unset `auth.mode`, but configured local SecretRef credentials can still be the active credential source. A localLoopback target with an unresolved configured token/password therefore gets diagnostics without `failureReason`, and `runGatewayStatusProbePass` proceeds to probe without auth.
  - confidence: 0.86
- **[P2] Stop resolving local auth for explicit URLs:** `src/commands/gateway-status/helpers.ts:176-180`
  - body: The docs say CLI `--url` overrides never reuse implicit config/env credentials, but this still maps `kind: "explicit"` to the local auth surface. A resolvable local token/password can therefore be attached to an arbitrary explicit URL; explicit targets should use only explicit CLI credentials or fail with the documented missing-credentials error.
  - confidence: 0.84
