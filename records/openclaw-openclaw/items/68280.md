## Summary

This PR should stay open. The local-loopback fail-fast behavior is useful and now has real CLI proof, but the latest diff still violates the documented explicit-URL credential boundary by allowing stored local gateway credentials to flow to user-supplied override targets.

## What This Changes

The PR adds fail-fast handling for local gateway probe/status paths when required probe auth cannot be resolved, plus regression coverage for SecretRef and auth-mode cases.

## Best Possible Solution

Keep the local loopback unresolved-auth fail-fast, but isolate explicit URL overrides so they use only explicitly supplied CLI credentials or no credentials, with tests covering resolved local SecretRefs/env credentials, unresolved SecretRefs, and local-loopback fail-fast.

## Reproduction Assessment

Yes. Source inspection of PR head shows explicit targets still resolve through the local auth surface and then pass token/password into `probeGateway`, which is enough to reproduce the credential-boundary problem without running the CLI.

## Solution Assessment

No. The local fail-fast behavior is a good direction, but the PR is not the best complete fix until explicit URL overrides are kept off implicit local credentials and covered by regression tests.
