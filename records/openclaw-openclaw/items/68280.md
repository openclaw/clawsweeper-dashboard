## Summary

Keep open: the latest PR head still has source-reproducible fail-fast gaps, missing real behavior proof, and unresolved explicit-URL credential forwarding in the touched auth paths.

## What This Changes

The PR threads local gateway probe auth failureReason through status, daemon, and gateway-status paths so unresolved required local auth can return a probe failure before anonymous loopback WebSocket connects.

## Best Possible Solution

Use effective credential resolution for local fail-fast, honor the no-implicit-credentials rule for explicit URL overrides, and add redacted real CLI/gateway proof before merge.

## Reproduction Assessment

Yes, source-reproducible: on the PR head, configure gateway.auth.token or password as an unresolved SecretRef with gateway.auth.mode unset and the new raw authMode guard skips failureReason, allowing the anonymous local probe path to run. I did not run live commands because this sweep is read-only.

## Solution Assessment

No, not yet. The patch should derive fail-fast from the effective credential surface or interactive auth result and should keep explicit URL overrides from inheriting implicit local credentials.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Use effective auth before treating unset mode as open:** `src/gateway/probe-auth.ts:183-185`
  - body: When gateway.auth.mode is unset, current main can still activate local token/password inputs from configured credentials. This early return skips the new failureReason for an unresolved configured SecretRef, so the anonymous loopback probe can still run instead of failing fast.
  - confidence: 0.87
- **[P2] Preserve inferred local auth in gateway status:** `src/commands/gateway-status/helpers.ts:189-198`
  - body: authModeRequiresCredentials is false for unset auth.mode, but resolveGatewayProbeSurfaceAuth still resolves configured local token/password inputs in that case. A localLoopback target with an unresolved configured SecretRef therefore returns diagnostics without failureReason and proceeds to probe without auth.
  - confidence: 0.85

## Security Review

Status: needs_attention

Summary: No dependency or workflow supply-chain issue was found, but explicit URL override credential forwarding remains unresolved in the touched gateway auth paths.

Concerns:

- **[low] Do not forward local daemon credentials to RPC URL overrides:** `src/cli/daemon-cli/status.gather.ts:572`
  - body: When probeUrlOverride is set, the PR bypasses local fail-fast but still passes daemonProbeAuth into probeGatewayStatus. Unless those credentials came from explicit CLI auth, a user-supplied RPC URL can receive local configured credentials despite the documented override-safety contract.
  - confidence: 0.8
- **[low] Do not resolve local auth for explicit status URLs:** `src/commands/gateway-status/helpers.ts:176`
  - body: resolveAuthForTarget still maps kind: "explicit" to the local auth surface and can return config/env local credentials for an arbitrary --url target. That can send local gateway credentials to a user-supplied endpoint; explicit targets should use explicit auth only or a host-aware policy matching the documented contract.
  - confidence: 0.82
