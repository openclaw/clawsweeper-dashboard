## Summary

Keep open. Current main still lacks the requested local auth fail-fast path, but the latest PR head is not merge-ready because it still skips inferred local auth when `gateway.auth.mode` is unset, lacks the required user-facing changelog entry, and needs maintainer/security review for explicit URL credential forwarding.

## What This Changes

The PR adds gateway probe auth `failureReason` propagation, short-circuits local status/probe paths when required local auth is unresolved, and adds focused regression coverage for those probe/status paths.

## Best Possible Solution

Update the PR so fail-fast follows the effective credential plan for inferred auth across status/probe paths, settle or tighten explicit URL credential forwarding, add the changelog entry, and validate the exact head.

## Reproduction Assessment

Yes. Source inspection shows current main has no `failureReason` path, and the PR defect is source-reproducible because its raw `authMode` guards skip fail-fast for unset-mode inferred local credentials documented as active.

## Solution Assessment

No. The fail-fast direction is narrow and useful, but the implementation should use effective credential planning rather than raw `gateway.auth.mode`, and explicit URL credential forwarding needs an approved policy.

  - body: When `gateway.auth.mode` is unset, a configured `gateway.auth.token` or `gateway.auth.password` can still be the effective local auth source.   - confidence: 0.85
