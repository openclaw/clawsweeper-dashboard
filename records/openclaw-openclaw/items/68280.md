## Summary

Keep this PR open: it contains a meaningful gateway/status auth diagnostic fix, but the current head still lets `openclaw daemon status` fall through to restart-health and open a secondary unauthenticated loopback probe after the new fail-fast result.

## What This Changes

The PR adds `failureReason`-based fail-fast handling for missing required local gateway auth across gateway status/probe/status scan paths, with explicit URL carve-outs, cached device-token guards, and focused tests.

## Best Possible Solution

Carry the local auth fail-fast through every daemon status probe path, including restart-health, then have gateway/auth owners explicitly accept the fail-closed upgrade semantics.

## Reproduction Assessment

Yes, source-reproducible. On PR head, daemon status synthesizes the auth failure, then still calls restart-health when the service is loaded and `rpc.ok !== true`, and restart-health can call `probeGateway` without auth.

## Solution Assessment

No. The main fail-fast approach is plausible, but it is not the best fix until the sibling daemon restart-health path cannot perform the anonymous loopback probe the PR is meant to avoid.
