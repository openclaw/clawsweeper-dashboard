## Summary

Keep open: the local loopback fail-fast direction is useful and the PR now has real terminal proof, but PR head still forwards daemon-resolved local credentials to explicit `daemon status --rpc-url` probe targets.

## What This Changes

This PR adds `failureReason` plumbing to gateway probe/status auth resolution, skips local probe attempts when required auth is unresolved, and adds regression tests for SecretRef, auth-mode, and explicit URL cases.

## Best Possible Solution

Keep the local loopback unresolved-auth fail-fast, but make every explicit override target use only explicit CLI credentials or no auth, with regression tests for resolved local credentials, unresolved SecretRefs, and override probes.

## Reproduction Assessment

Yes. Source inspection of PR head shows `--rpc-url` becomes `gateway.probeUrl` while locally resolved `daemonProbeAuth` is still forwarded into the probe call.

## Solution Assessment

No. The local-loopback fail-fast is a good direction, but the implementation must also isolate daemon explicit URL overrides from stored local credentials before merge.
