## Summary

Keep this PR open: the fail-fast direction is useful, but the current head still conflicts with the documented explicit URL credential contract and misses effective-auth cases where configured local credentials make auth required even with unset raw mode.

## What This Changes

The PR threads gateway probe auth `failureReason` through daemon status, top-level status, and gateway-status paths so unresolved required credentials can fail before an anonymous WebSocket probe.

## Best Possible Solution

Keep the fail-fast behavior, but derive it from the effective credential plan and make explicit `--url`/`--rpc-url` paths use only explicit credentials or a documented missing-explicit-credentials failure.

## Reproduction Assessment

Yes. Source inspection of PR head f18cb65 shows explicit URL targets still resolve local auth and unset raw auth mode still bypasses fail-fast before effective configured auth is considered.

## Solution Assessment

No. The patch is close in shape, but the maintainable fix needs one credential-contract path that distinguishes local targets from explicit overrides and bases fail-fast on effective auth, not only raw `gateway.auth.mode`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep explicit gateway-status URLs off local auth:** `src/commands/gateway-status/helpers.ts:176-181`
  - body: `--url` is documented as requiring explicit credentials and never reusing config/env auth, but this still resolves `kind: "explicit"` through the local surface. When `gateway.auth.token`, `gateway.auth.password`, or the matching env var resolves, `probe-run.ts` will attach those credentials to the user-supplied URL; explicit targets should use only `--token`/`--password` or return the documented missing-explicit-credentials error.
  - confidence: 0.9
- **[P1] Do not forward daemon auth to RPC URL overrides:** `src/cli/daemon-cli/status.gather.ts:599-605`
  - body: The new override branch skips the local failure reason, but the probe call still forwards `daemonProbeAuth` to `gateway.probeUrl`. If `--rpc-url` points outside the local daemon and local config/env auth resolves, the daemon status probe can send those implicit credentials to the override target, which conflicts with the same explicit URL safety contract.
  - confidence: 0.84
- **[P2] Derive local fail-fast from effective auth:** `src/gateway/probe-auth.ts:183-185`
  - body: Returning whenever raw `gateway.auth.mode` is unset misses configs where `gateway.auth.token`, `gateway.auth.password`, or env credentials make shared-secret auth effective. With an unresolved configured SecretRef and no explicit mode, this path still returns no `failureReason`, so the anonymous local probe can continue instead of surfacing the resolver diagnostic.
  - confidence: 0.88
- **[P2] Apply the effective-auth check in gateway status too:** `src/commands/gateway-status/helpers.ts:189-199`
  - body: `authModeRequiresCredentials` has the same raw-mode assumption in `resolveAuthForTarget`. A local loopback target with configured but unresolved auth and unset `gateway.auth.mode` gets diagnostics without `failureReason`, so `runGatewayStatusProbePass` can still attempt the unauthenticated probe.
  - confidence: 0.86
