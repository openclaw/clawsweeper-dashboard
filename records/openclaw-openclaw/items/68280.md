## Summary

Keep open: this is a useful gateway auth diagnostic fix, but the PR is not merge-ready because `openclaw daemon status` can still fall through to the restart-health loopback probe after the new auth failure, and the supplied terminal proof does not cover that changed daemon-status path.

## What This Changes

The PR adds `failureReason` plumbing so missing required local gateway auth returns structured auth failures across gateway status/probe surfaces instead of attempting anonymous local WebSocket probes.

## Best Possible Solution

Carry the local auth fail-fast through every daemon status probe path, including restart-health, then have gateway/auth owners explicitly accept the fail-closed upgrade semantics.

## Reproduction Assessment

Yes, source-reproducible. On PR head, daemon status synthesizes the auth failure, then still calls restart-health when the service is loaded and `rpc.ok !== true`, and restart-health can call `probeGateway` without auth.

## Solution Assessment

No. The main fail-fast approach is plausible, but it is not the best fix until the sibling daemon restart-health path cannot perform the anonymous loopback probe the PR is meant to avoid.
