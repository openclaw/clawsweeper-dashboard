## Summary

Keep this PR open: it is a focused gateway auth/probe bug fix, but the current head still has blocking auth-contract and credential-forwarding problems, and the external contributor has not provided real after-fix CLI/gateway proof.

## What This Changes

The PR threads gateway probe auth `failureReason` through daemon, status, and gateway-status probe paths so unresolved required auth can skip anonymous WebSocket probes, with focused regression tests.

## Best Possible Solution

Derive fail-fast from the effective credential plan, honor the documented explicit-URL explicit-credentials-only contract, and require redacted live CLI/gateway proof before merge.

## Reproduction Assessment

Yes. Source inspection of the PR head shows unset raw `gateway.auth.mode` can suppress fail-fast even when configured local SecretRefs make auth active, and explicit URL/RPC override paths still consult or forward local credentials.

## Solution Assessment

No. The patch should use effective credential resolution instead of raw mode checks and should treat explicit URL/RPC overrides as explicit-credential-only before probing.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Use effective auth before skipping unset local mode:** `src/gateway/probe-auth.ts:183-185`
  - body: When `gateway.auth.mode` is unset but `gateway.auth.token` or password is configured through an unresolved SecretRef, the credential planner treats that local auth surface as active. This early return suppresses `failureReason`, so the local probe can still proceed anonymously instead of surfacing the resolver diagnostic.
  - confidence: 0.88
- **[P2] Keep local-loopback fail-fast when mode is inferred:** `src/commands/gateway-status/helpers.ts:189-199`
  - body: `authModeRequiresCredentials` is false for unset `auth.mode`, but configured local SecretRef credentials can still be the effective credential source. A localLoopback target with an unresolved configured token/password therefore gets diagnostics without `failureReason`, and `runGatewayStatusProbePass` still probes without auth.
  - confidence: 0.86
- **[P2] Stop resolving local auth for explicit URLs:** `src/commands/gateway-status/helpers.ts:176-180`
  - body: The docs say CLI `--url` overrides do not fall back to config or environment credentials, but this still maps `kind: "explicit"` to the local auth surface. A resolvable local token/password can be attached to an arbitrary explicit URL, and unresolved local config can block an unrelated explicit target.
  - confidence: 0.87
- **[P2] Do not forward daemon auth to RPC URL overrides:** `src/cli/daemon-cli/status.gather.ts:567-573`
  - body: When `probeUrlOverride` is set, the PR skips the local fail-fast check but still passes `daemonProbeAuth` into `probeGatewayStatus`. That can send locally resolved daemon gateway credentials to a user-supplied RPC URL instead of requiring credentials explicitly provided for that override.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff still allows implicit local gateway credentials to reach explicit URL override targets.

Concerns:

- **[low] Explicit gateway-status URLs can inherit local credentials:** `src/commands/gateway-status/helpers.ts:176`
  - body: `resolveAuthForTarget` maps explicit targets to the local auth surface, so a user-supplied `--url` can receive locally configured or environment gateway credentials despite the documented explicit-credentials-only contract.
  - confidence: 0.87

  - body: The `probeUrlOverride` path suppresses local fail-fast but still forwards `daemonProbeAuth` into `probeGatewayStatus`, so local gateway credentials can be sent to the override target unless they were explicitly supplied for that URL.
  - confidence: 0.82
