## Summary

Keep this PR open: the local auth fail-fast direction is useful, but the daemon-status path still has a secondary restart-health probe path that can bypass the new failure reason, and the current real behavior proof does not exercise `openclaw daemon status`. The auth/explicit-URL behavior also remains compatibility and credential-boundary sensitive enough to need maintainer acceptance before merge.

## What This Changes

Adds gateway status/probe auth `failureReason` plumbing that short-circuits local loopback probes when configured local auth is required but unresolved, with tests for local, explicit URL, paired-device-token, and auth-mode guard paths.

## Best Possible Solution

Carry one local probe-auth decision through gateway status, top-level status, daemon status, and restart health, then land only with daemon-status proof plus maintainer-accepted upgrade and explicit-URL credential behavior.

## Reproduction Assessment

Source-reproducible: the PR creates a synthetic `rpcFailureReason` result in daemon status, and source inspection shows the subsequent restart-health block can still call `probeGateway()` without that failure context. I did not run the live daemon path because this review is read-only.

## Solution Assessment

No. The fail-fast shape is plausible, but it is not the best merge-ready fix until the daemon-status secondary probe shares the same auth decision and maintainers accept the upgrade/security boundary.
