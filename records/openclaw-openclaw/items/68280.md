## Summary

Keep open. Current main still lacks the PR's fail-fast local probe auth behavior, and the latest PR head appears to preserve the recent timeout/fallback fixes that earlier reviews flagged. The remaining blockers are a low-severity maintainer/security decision about explicit URL credential forwarding and a required changelog entry, not grounds to close the PR.

## What This Changes

The PR adds a `failureReason` result to local gateway probe auth resolution, short-circuits local loopback status/probe paths when required credentials are unresolved, and adds focused regression coverage for gateway, top-level status, and daemon status paths.

## Best Possible Solution

Merge the fail-fast path only after maintainer/security sign-off decides whether URL override probes may receive implicit stored credentials, then add a release-note entry and run the focused gateway/status tests plus the changed gate.

## Reproduction Assessment

Yes. The source path is deterministic: current main resolves auth warnings but still calls `probeGateway`, while the PR head introduces `failureReason` and synthetic failed probe results for the local unresolved-auth path.

## Solution Assessment

Unclear as final merge shape. The `failureReason` approach is a narrow maintainable fix for local loopback auth, but the explicit URL credential behavior should be settled before this becomes the final solution.
