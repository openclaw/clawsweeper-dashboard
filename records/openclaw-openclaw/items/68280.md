## Summary

Keep open. Current main still lacks the proposed local auth fail-fast result, and the latest PR head contains a plausible implementation with the earlier authMode and explicit-URL false positives addressed, but the branch is stale/merge-conflicted against current main and needs maintainer resolution around current probe timeout/fallback behavior and the explicit-URL credential-forwarding question.

## What This Changes

The PR adds `failureReason`-based fail-fast handling for local gateway probe/status auth resolution, skips anonymous loopback probing when required local auth is unresolved, and adds focused regression coverage for the affected gateway/status paths.

## Best Possible Solution

Rebase the fail-fast implementation onto current main, layer `failureReason` handling around the existing handshake-timeout and local RPC fallback logic, settle whether explicit URL overrides may receive stored local credentials, update the public status contract/docs if output changes, and then run focused gateway/status tests plus the changed gate.

Yes. For the requested behavior, current main can be inspected deterministically: unresolved local auth currently produces warnings/empty auth and the gateway/status paths still call `probeGateway` rather than returning a local failed result.
