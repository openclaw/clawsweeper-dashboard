## Summary

Keep open. The fail-fast direction is relevant and current main does not yet implement the PR’s `failureReason` short-circuit, but the branch still needs a small test-isolation fix, a required changelog entry, and maintainer/security sign-off on explicit URL credential forwarding before merge.

## What This Changes

The PR adds local gateway/status probe auth `failureReason` plumbing so required unresolved local credentials return a synthetic failed probe instead of attempting an anonymous loopback WebSocket connection.

## Best Possible Solution

Keep the `failureReason` fail-fast shape for known local loopback probes, isolate the new tests from ambient env, add the active-release changelog entry with contributor credit, and have maintainers decide whether explicit URL override probes must stop reusing implicit local credentials or document/approve the exception.

## Reproduction Assessment

Yes. The product behavior is visible from the deterministic source path: current main resolves auth warnings and still probes, while the PR head short-circuits on `failureReason`; the test flake is reproducible by running the new token-mode helper test with `OPENCLAW_GATEWAY_TOKEN` set.

## Solution Assessment

Unclear as the final merge shape. The local `failureReason` approach is narrow and maintainable, but the branch should not merge until the URL override credential policy is settled and the mechanical test/changelog issues are fixed.
