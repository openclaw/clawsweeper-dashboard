## Summary

Keep open. The PR targets a real gateway/status auth bug, but the latest head still skips fail-fast for inferred local auth modes, is missing the required changelog entry, and leaves explicit URL credential-forwarding policy unresolved.

## What This Changes

The PR adds gateway probe auth `failureReason` propagation, short-circuits local status/probe paths when required local credentials are unresolved, and adds focused regression tests.

## Best Possible Solution

Merge only after fail-fast uses effective local auth requirements rather than raw `gateway.auth.mode`, explicit URL credential policy is approved or tightened, the changelog is updated, and the exact PR head validates cleanly.

## Reproduction Assessment

Yes. Source inspection shows current main still has no `failureReason` path, while the PR adds one but leaves the unset-mode SecretRef case source-reproducible through the raw `authMode` guard.

## Solution Assessment

No as a merge-ready fix. The local-loopback fail-fast direction is narrow, but it should be based on effective credential planning and should not leave the documented explicit URL credential policy ambiguous.

  - body: When `gateway.auth.mode` is unset, local auth can still be required if `gateway.auth.token` or `gateway.auth.password` is configured and can win. This guard treats every unset mode as open, so an unresolved inferred SecretRef keeps returning only a warning and the status/probe path still attempts an anonymous loopback connection.
  - confidence: 0.86

Use the effective credential plan here as well before deciding that no credentials are required.
  - confidence: 0.84

  - body: This changes user-facing CLI status/probe behavior by replacing anonymous local probes with an auth failure result when local credentials are required but unavailable. Repo policy requires a `CHANGELOG.md` entry for user-facing fixes, but this PR does not touch the changelog.
