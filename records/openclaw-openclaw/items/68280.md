## Summary

Keep open: the local-loopback auth fail-fast is useful, but the PR head still lets daemon status fall through into restart-health probing after the synthetic auth failure, so the anonymous loopback probe can still happen on that sibling path.

## What This Changes

The PR adds gateway status/probe failureReason plumbing so unresolved required local loopback auth fails fast while explicit URL overrides and cached paired device-token installs continue to probe.

## Best Possible Solution

Keep the local-loopback fail-fast direction, but make every status/probe surface share the same no-anonymous-probe invariant, add daemon-status regression and terminal proof, and have maintainers explicitly accept the fail-closed upgrade behavior.

## Solution Assessment

No, not yet. The local-loopback fail-fast is a reasonable fix shape, but the daemon status sibling still needs the same no-probe guard or equivalent state propagation before this is the best fix.
