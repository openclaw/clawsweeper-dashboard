## Summary

Keep open. Current `main` and the latest release still source-reproduce the Control UI path where `/model status` and `/model list` are preserved as non-empty args and sent to `sessions.patch`; the linked fix PRs are open/conflicting or closed unmerged, not shipped fixes.

## What This Changes

The issue requests that Control UI handle `/model status` and `/model list` as informational `/model` subcommands instead of sending `status` or `list` as model IDs to `sessions.patch`.

## Best Possible Solution

Land or replace one canonical Control UI fix that routes `/model list` and `/model status` to the documented/shared informational flows before model mutation, preserves `/model <provider/model>` selection compatibility, and adds focused executor regression coverage.

## Reproduction Assessment

Yes, at source level. `/model status` parses to `args: "status"`, Control UI preserves that argument, and `executeModel` sends every non-empty `/model` arg to `sessions.patch`; I did not run a live UI session in this read-only review.

## Solution Assessment

Yes. The maintainable fix is to classify documented informational subcommands before model mutation while preserving normal model selection, and to land that through one canonical PR rather than parallel conflicting branches.
