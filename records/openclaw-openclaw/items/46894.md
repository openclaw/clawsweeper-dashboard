## Summary

Keep open: current main still routes Control UI `/model status` and `/model list` through the model-mutation branch, while docs and the shared directive handler define those as informational commands. The issue is already paired with the open, conflicting fix PR at https://github.com/openclaw/openclaw/pull/46895, so this should stay open until that PR is reviewed, refreshed, replaced, or closed.

## What This Changes

The issue reports that Control UI local `/model status` and `/model list` are treated as model-setting requests instead of informational slash-command subcommands.

## Best Possible Solution

Land or replace one canonical Control UI fix that recognizes the documented `/model list` and `/model status` modes before model mutation, preserves normal model selection, and adds focused executor regression coverage.

## Reproduction Assessment

Yes. Source inspection shows `/model status` and `/model list` parse into non-empty args, then current Control UI `executeModel` sends those args as `model` in `sessions.patch`, matching the reporter's logs.

## Solution Assessment

Yes, the best fix is a narrow Control UI command-routing repair before the model patch branch, but it must align with the documented shared `list` and `status` behaviors rather than collapsing them into the bare summary path.
