## Summary

Keep open. Current main at 3d96111a5afe377b529c9bb5a9db510d74607344 and the latest release still source-reproduce the Control UI path where `/model status` and `/model list` are preserved as non-empty args and sent to `sessions.patch`; the linked fix PRs remain open and unmerged.

## What This Changes

The issue asks the Control UI to treat `/model status` and `/model list` as informational `/model` subcommands instead of model ids passed to `sessions.patch`.

## Best Possible Solution

Land or replace one canonical Control UI fix that routes `/model list` and `/model status` to their documented informational flows before model mutation while preserving `/model <provider/model>` selection.

## Reproduction Assessment

Yes. Source inspection shows `/model status` parses to `args: "status"`, dispatch preserves it, and `executeModel` sends every non-empty `/model` arg to `sessions.patch`; I did not run a live Control UI session in this read-only review.

## Solution Assessment

Yes, the requested direction is the right fix boundary because docs and the shared directive path already define `list` and `status` as informational modes. The maintainer task is to choose, revise, or replace one linked PR rather than create a parallel fix.
