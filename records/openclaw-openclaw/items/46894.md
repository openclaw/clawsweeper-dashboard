The issue asks Control UI to treat `/model status` and `/model list` as informational slash-command modes instead of model selection values.

## Best Possible Solution

Land or replace one canonical Control UI fix that routes `/model list` and `/model status` to the documented/shared informational flows before model mutation, preserves normal model selection compatibility, and adds focused executor regression coverage.

Yes, at source level. `/model status` parses to `args: "status"`, Control UI preserves that argument, and `executeModel` sends every non-empty `/model` arg to `sessions.patch`; I did not run a live UI session in this read-only review.

## Solution Assessment

Yes. The maintainable fix is to classify documented informational subcommands before model mutation while preserving normal model selection, and to land that through one canonical PR rather than parallel conflicting branches.
