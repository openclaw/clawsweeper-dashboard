## Summary

Keep open. Current main and the latest release still source-reproduce the Control UI path where `/model status` and `/model list` are preserved as non-empty arguments and sent to `sessions.patch`; linked PRs are implementation candidates, not shipped fixes.

## What This Changes

The issue requests that Control UI treat `/model status` and `/model list` as informational slash-command modes instead of model names passed to `sessions.patch`.

## Best Possible Solution

Land or replace one canonical Control UI fix that routes `/model list` and `/model status` to their documented flows before model mutation, preserves normal `/model <provider/model>` selection compatibility, and adds focused regression coverage.

## Reproduction Assessment

Yes, at source level. `/model status` parses to `args: "status"`, Control UI passes that through, and `executeModel` sends every non-empty `/model` arg to `sessions.patch`; I did not run a live UI session in this read-only review.

## Solution Assessment

Yes. The maintainable fix is to align the Control UI executor with the existing docs and shared directive behavior by recognizing informational model subcommands before model mutation, preferably by landing or replacing one linked PR instead of creating another parallel branch.
