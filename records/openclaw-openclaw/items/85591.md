## Summary

Keep this PR open: the Docker mount hardening is focused and backed by terminal proof, but it is not merge-ready because the new read-only skill contract is only applied to Docker bind mounts while sandbox file-bridge write paths still resolve the same `rw` workspace skill paths as writable.

## What This Changes

The PR adds read-only Docker overlays for workspace skill roots in `workspaceAccess: "rw"`, includes those mounts in sandbox hashes, bumps the mount format version, and adds mount/hash tests.

## Best Possible Solution

Apply the read-only skill-path contract consistently across Docker mounts and sandbox file bridges, then land only after maintainers accept the compatibility change for `rw` sandbox users.

## Reproduction Assessment

Yes, at source level and with contributor Docker proof. The current and PR-head source show Docker `rw` workspace skill roots were writable through the workspace mount, while the PR proof shows direct Docker writes blocked but source inspection still shows the fs bridge write path writable.

## Solution Assessment

No, not yet. Read-only Docker overlays are the right kernel boundary for exec paths, but the fix is incomplete until the sandbox file bridges enforce the same read-only skill paths and maintainers accept the compatibility tradeoff.
