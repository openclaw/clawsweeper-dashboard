## Summary

Keep open: the rw Docker hardening is useful and the branch removed the earlier chmod approach, but the new non-rw `/workspace/skills` overlay duplicates an existing read-only workspace mount and can leave reused containers pointed at stale synced skills after refresh.

## What This Changes

The branch adds Docker read-only bind overlays and hash state for workspace skill roots across sandbox modes, wires them into container and browser creation, and adds focused mount/hash tests.

## Best Possible Solution

Keep the rw skill-root overlay hardening, but remove the redundant non-rw skills overlay or make skill sync preserve the mounted source inode before asking maintainers to accept the rw compatibility break.

## Reproduction Assessment

Yes, at source level. A focused repro would start a ro or none Docker sandbox with synced skills, refresh those skills while the container remains running, and inspect whether the reused `/workspace/skills` bind still sees the old source after `syncSkillsToWorkspace()` removed and recreated it.

## Solution Assessment

No, not as submitted. A Docker read-only overlay is the right boundary for rw workspace skill roots, but adding the same overlay in non-rw modes duplicates the existing `/workspace:ro` protection and creates a refresh hazard.
