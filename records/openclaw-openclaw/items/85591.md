## Summary

Keep open: this is useful partial sandbox hardening, but it should not merge as the fix for https://github.com/openclaw/openclaw/issues/17931 yet because non-rw sandbox modes still copy writable skills, existing containers can miss overlays when skill directories appear later, and the PR only provides tests/CI rather than real Docker behavior proof.

## What This Changes

The PR adds Docker read-only bind overlays for existing workspace `skills/` and `.agents/skills/` directories when `workspaceAccess` is `rw`, bumps the sandbox mount format version, and adds mount-argument tests.

## Best Possible Solution

Derive protected skill mounts from the resolved skill set for every sandbox workspace mode, make the generated mount state part of container recreation/hash behavior, document or explicitly accept the compatibility change, and prove the write-denial path in a real Docker sandbox.

## Reproduction Assessment

Yes, at source level: current main mounts the whole workspace writable for `rw`, and for `ro`/`none` it calls `syncSkillsToWorkspace`, which copies skills into the writable sandbox workspace. I did not run a live Docker container in this read-only review.

## Solution Assessment

No. The rw overlay is a useful partial fix, but the best solution needs to cover copied skills in non-rw modes and container recreation when the required overlay set changes before this can close the linked security issue.
