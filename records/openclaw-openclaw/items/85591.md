## Summary

Keep open for maintainer merge review: current head has sufficient terminal proof and I found no blocking code-review finding, but the PR intentionally changes sandbox security and compatibility behavior.

## What This Changes

The PR mounts existing workspace `skills` and `.agents/skills` roots read-only in `rw` Docker sandboxes, updates sandbox hash state, and teaches generic and remote fs bridges to classify those roots as read-only.

## Best Possible Solution

Land this after maintainer acceptance of the hardened default, keeping absent skill roots writable until they exist and preserving the existing whole-workspace read-only behavior for `ro` and `none`.

## Reproduction Assessment

Yes, at source level: current main makes the primary workspace mount writable in `rw` mode and the fs bridges only know about workspace, agent, and bind mounts. I did not execute Docker in this read-only review.

## Solution Assessment

Yes, with maintainer acceptance: read-only nested bind overlays plus matching bridge policy and hash state are the narrow maintainable fix for existing skill roots. The safer alternative, if compatibility wins, is an explicit editable-skills mode rather than keeping writable skill roots as the default.
