## Risks / Open Questions

- Merging intentionally makes existing workspace `skills` and `.agents/skills` roots read-only inside `workspaceAccess: "rw"` Docker sandboxes, so workflows that edit those paths from inside the sandbox will start failing with read-only errors.
- Recently used hot containers can keep the old mount layout until operators recreate them, so the mount-hash upgrade behavior should stay visible in merge verification.
- Custom Docker binds are appended after managed workspace mounts, so an operator-owned bind can still shadow the protected skill path; maintainers should accept that as an escape hatch before treating this as complete hardening.
