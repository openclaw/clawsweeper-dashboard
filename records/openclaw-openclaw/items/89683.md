## Summary

Keep open. Current main and the latest release still promote failed tool cards to visible Tool error UI without a turn-success guard, while the direct candidate fix at https://github.com/openclaw/openclaw/pull/90122 remains open and unmerged.

## What This Changes

The issue reports that Control UI/WebChat renders failed internal assistant tool calls as prominent red chat error banners even when the assistant turn also has a clean final reply.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/90122 if maintainers accept its projection-boundary fix, or replace it with an equivalent UI render/projection change that collapses non-terminal internal tool failures while preserving terminal failure visibility.

## Reproduction Assessment

Yes, at source level: current main maps failed tool cards directly to Tool error UI and has no turn-success projection guard. I did not run a fresh live browser reproduction in this read-only review.

## Solution Assessment

Yes. The Control UI chat projection/render boundary is the narrow owner for deciding whether a failed internal tool is terminal to the visible turn; changing Codex exec failure semantics or direct-chat dispatch would be broader or the wrong surface.
