## Summary

Keep open. Current main and the latest release still promote failed tool cards to visible `Tool error` UI without a turn-success guard, and the direct closing PR is open but unmerged.

## What This Changes

The issue asks Control UI/WebChat to keep non-terminal hidden/internal tool failures out of the normal visible chat timeline when a clean assistant reply exists.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/90122 if maintainers accept its projection-boundary fix, or replace it with an equivalent UI render/projection change that collapses non-terminal internal tool failures while preserving terminal failure visibility.

## Reproduction Assessment

Yes, source-reproducible: current main maps failed tool cards directly to `Tool error` UI without checking whether the turn later produced a normal assistant reply. I did not run a fresh live browser reproduction in this read-only pass.

## Solution Assessment

Yes. The Control UI chat projection/render boundary is the narrow owner for deciding whether a failed internal tool is terminal to the visible turn; changing Codex exec failure semantics or direct-chat dispatch would be broader or the wrong surface.
