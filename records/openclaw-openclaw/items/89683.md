## Summary

Keep open. Current main has partial default hiding for normalized tool-result history entries, but source, tests, and the Codex exec-event contract still support the reported failure mode: failed internal tool activity can carry command/status details and the Control UI still has visible failed-tool rendering paths instead of a proven projection boundary for completed assistant turns.

## What This Changes

The issue reports that Control UI/WebChat renders failed hidden assistant tool calls as visible red error banners even when the turn has a successful final reply.

## Best Possible Solution

Keep this open and fix the Control UI projection so default chat rendering suppresses internal tool-call/tool-result failures after a successful assistant outcome, while retaining explicit debug/activity access for tool details.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source inspection gives a plausible path through Codex failed exec events and OpenClaw tool-card rendering, but I did not establish a live current-main reproduction without the reporter's screenshot or stored transcript artifact.

## Solution Assessment

Is this the best way to solve the issue? Yes: the narrow maintainable fix is to normalize the chat projection/rendering boundary so hidden internal tool activity is not treated as primary user-facing error content, rather than only restyling failed tool cards.
