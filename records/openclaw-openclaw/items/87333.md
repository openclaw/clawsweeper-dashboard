## Summary

Keep open: current main still has the unguarded path that can pin a tool-discovery registry as the channel surface, and the PR’s guard plus regression test look targeted. The remaining blocker is that the external PR only shows unit-test output, not real post-restart/frontend/outbound behavior proof.

## What This Changes

This PR skips channel-surface pinning when a standalone tool-discovery load returns an empty channel registry, and adds a regression test for preserving the pinned startup channel registry.

## Best Possible Solution

Merge the narrow guard with regression coverage after the contributor adds redacted real behavior proof showing tools.catalog after gateway restart no longer breaks outbound delivery.

## Reproduction Assessment

Yes, source-level. Current main routes tools.catalog through toolDiscovery=true and surface="channel", then pins the loaded registry without checking whether it has any channel registrations.

## Solution Assessment

Yes, likely. The guard is narrow because it preserves the active tool-discovery registry while only skipping channel pinning when toolDiscovery returns an empty channel registry, but it still needs real behavior proof before merge.
