## Label Justifications

- `P1`: The PR targets a channel workflow regression where opening the frontend can wipe the pinned channel registry and break outbound operations.
- `merge-risk: 🚨 message-delivery`: Merging changes which registry remains selected for channel delivery after tool-discovery loads.
- `merge-risk: 🚨 availability`: The touched runtime registry state can affect whether gateway channel operations remain usable after plugin discovery.

## Summary

Keep open for maintainer review: current main still unconditionally pins a channel-surface standalone registry after a tool-discovery load, while this PR adds a narrow channel-empty guard with source-level regression coverage and sufficient real terminal proof.

## What This Changes

The PR prevents empty channel registries produced by standalone plugin tool discovery from replacing the startup-pinned channel registry, and adds regression tests for channel and http-route pinning behavior.

## Best Possible Solution

Land this PR or an equivalent small guard after maintainer review, preserving active registry updates for tool discovery while preventing empty channel registries from replacing the startup-pinned channel registry.

## Reproduction Assessment

Yes. Source inspection traces tools.catalog through ensureStandalonePluginToolRegistryLoaded into a channel-surface toolDiscovery standalone load, and current main then unconditionally pins that loaded registry; I did not rerun the live gateway flow in this read-only review.

## Solution Assessment

Yes. The proposed fix is the narrowest maintainable guard I found: it keeps the active registry update for tool discovery, skips only empty channel-surface pin replacement, and leaves http-route pinning covered by a sibling regression test.
