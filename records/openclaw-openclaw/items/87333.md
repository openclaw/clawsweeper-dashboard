## Label Justifications

- `P1`: The PR targets a channel workflow regression where opening the frontend can wipe the pinned channel registry and break outbound operations.
- `merge-risk: 🚨 message-delivery`: Merging changes which registry remains selected for channel delivery after standalone tool-discovery loads.
- `merge-risk: 🚨 availability`: The touched runtime registry state can affect whether gateway channel operations remain usable after plugin discovery.

## Summary

Keep open: current main still has the empty-registry overwrite path, and this PR is a focused fix with sufficient terminal proof and no blocking findings. It still needs a maintainer merge decision because the touched path is mutable global plugin registry state used by channel delivery.

## What This Changes

The PR adds a channel-surface toolDiscovery guard in the standalone runtime registry loader plus regression tests so empty tool-discovery registries do not replace the pinned channel registry while http-route pinning still works.

## Best Possible Solution

Land this PR or an equivalent small guard after maintainer review, preserving active registry updates for tool discovery while skipping only empty channel-surface pin replacement.

## Reproduction Assessment

Yes, source-reproducible. Current main traces tools.catalog into a channel-surface toolDiscovery standalone load, and current installStandaloneRegistry can still pin that empty channel registry over the startup-pinned registry; I did not run a live gateway reproduction in this read-only review.

## Solution Assessment

Yes. Guarding installStandaloneRegistry is narrower than changing the tools.catalog surface or adding outbound fallback behavior, because it preserves active registry updates for tool discovery and changes only empty channel pin replacement.
