## Label Justifications

- `P1`: The PR addresses a broken channel workflow where opening the frontend can clear the pinned channel registry and break outbound delivery until restart.
- `merge-risk: 🚨 message-delivery`: The patch changes channel registry pinning, so an incorrect guard could leave outbound channel delivery pointed at stale or missing registry state.

## Summary

Keep this PR open: current main still unconditionally repins the channel registry from standalone tool-discovery loads, while the PR adds a narrow guard plus regression coverage and real terminal proof for the reported channel-registry wipe.

## What This Changes

The PR adds a guard so channel-surface tool-discovery loads with no channels update the active plugin registry without overwriting the pinned channel registry, plus regression tests for channel and http-route pinning.

## Best Possible Solution

Land the narrow guard with its regression test once maintainers accept the channel-registry pinning risk and required checks remain green.

## Reproduction Assessment

Yes. Source inspection shows tools.catalog reaches a channel-surface, activate=false, toolDiscovery standalone registry load, and the PR body includes before/after terminal output where channels.status is cleared before the fix and preserved after it.

## Solution Assessment

Yes. The guard sits at the registry install boundary after active tool-discovery state is updated but before channel pin replacement; changing tools.catalog to use only the active surface would bypass existing tool-registry selection semantics more broadly.
