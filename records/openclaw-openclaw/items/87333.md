## Summary

Keep open. ## What This Changes

The PR adds a guard so empty tool-discovery registry loads do not replace the pinned channel registry, plus a regression test for that state.

## Best Possible Solution

Scope the empty-channel early return to channel-surface installs, keep the regression test, and require redacted real gateway/UI/outbound proof before merge.

## Reproduction Assessment

Yes. Source inspection shows tools.catalog can run a channel-surface tool-discovery load with activate=false, and current main pins the returned registry even though tool-discovery suppresses runtime channels; I did not run a live repro in this read-only review.

## Solution Assessment

No, not quite. The channel-preservation idea is narrow and appropriate, but the early return should be scoped to the channel surface so http-route installs are not skipped, and real behavior proof is still missing.
