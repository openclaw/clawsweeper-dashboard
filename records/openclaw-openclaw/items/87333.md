## Summary

Keep this PR open: it targets a real channel-registry wipe with convincing terminal proof, but the current guard is too broad and can skip HTTP-route registry pinning when a tool-discovery registry has routes but no channels.

## What This Changes

The PR adds an empty-channel tool-discovery guard in the standalone runtime registry installer and a regression test for preserving a pinned channel registry.

## Best Possible Solution

Scope the early return to `surface === "channel"` and add coverage showing HTTP-route installs with no channels still pin the route registry.

## Reproduction Assessment

Yes. Current source shows `tools.catalog` can request a channel-surface tool-discovery load, and the PR body supplies real terminal before/after output showing the channel registry wipe and preservation path.

## Solution Assessment

No. The fix is in the right owner area, but the early return should be scoped to `surface === "channel"` because tool-discovery suppresses runtime channel hydration while still allowing capability handlers such as HTTP routes.
