- `P1`: The PR addresses a reported regression where opening the frontend after gateway restart can wipe configured channel state and break outbound delivery.

## Summary

Keep open: current main still has the channel-registry overwrite path, and this PR is a focused fix with source-backed root cause, supplied terminal proof, passing relevant checks, and no blocking review findings. Because it changes process-global channel registry pinning, it still deserves normal maintainer review before merge.

## What This Changes

The PR adds a channel-surface guard so empty tool-discovery registries update the active tool registry without repinning the channel registry, plus regression tests for channel and http-route surfaces.

## Best Possible Solution

Merge the narrow guard and regression test after plugin-runtime maintainers confirm empty tool-discovery channel registries must never replace the startup-pinned channel registry.

## Reproduction Assessment

Yes from source inspection, though I did not run the gateway locally. Current main routes tools.catalog through a channel-surface toolDiscovery load that can pin an empty channel registry, and the PR body supplies real before/after terminal output for the failing gateway path.

## Solution Assessment

Yes. The patch fixes the implicated install point while preserving active tool-discovery registry updates and leaves full channel/http-route registry installs unchanged; changing outbound lookup or adding a runtime fallback would be broader and less direct.
