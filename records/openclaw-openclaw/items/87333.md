## Label Justifications

## Summary

Keep open: current main and v2026.6.6 still have the unguarded channel pin path, while this PR is a narrow registry fix with regression coverage and sufficient real terminal proof. I found no blocking patch findings.

## What This Changes

The PR adds a tool-discovery/channel guard in standalone runtime registry installation plus regression tests so a channel-less tool registry does not replace the startup-pinned channel registry.

## Best Possible Solution

Land the narrow guard with the regression test once maintainers accept that channel-less tool-discovery loads should not clear the startup channel pin.

## Reproduction Assessment

Yes, from source inspection: tools.catalog calls ensureStandalonePluginToolRegistryLoaded, which uses a channel-surface toolDiscovery load, and current main then pins the loaded registry even when channels is empty. I did not run the live gateway path myself.

## Solution Assessment

Yes. The guard sits at the narrow install/pin boundary, preserves the active tool-discovery registry, and leaves normal channel and http-route pinning behavior unchanged; changing the tools.catalog surface would be a broader behavior change.
