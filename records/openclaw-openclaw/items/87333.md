## Summary

Keep this PR open for maintainer review: current main and the latest release still have the unconditional channel-surface pin, while the PR adds a narrow guard and regression coverage for the reported registry wipe. ## What This Changes

The PR adds a channel-surface guard so empty tool-discovery registries do not replace an already pinned channel registry, and adds regression tests for channel preservation plus HTTP-route pinning.

## Best Possible Solution

Land the narrow channel-surface guard with the added regression tests after current-head validation confirms the pinned startup channel registry survives `tools.catalog` while normal full channel and HTTP-route registry updates still work.

## Solution Assessment

Yes: the patch fixes the owner boundary where the empty discovery registry overwrites the channel pin, instead of adding outbound fallback behavior or changing tool catalog loading. The main alternative, changing `ensureStandalonePluginToolRegistryLoaded` to avoid the channel surface entirely, is broader and would need separate proof for tool/runtime reuse semantics.
