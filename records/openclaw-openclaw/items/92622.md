## Summary

No blocking findings from this read-only review. ## What This Changes

The PR adds a WebChat follow-lock state so streaming updates do not auto-scroll after the user scrolls up, wires that state into the app, and adds scroll regression tests.

## Best Possible Solution

Land one narrow helper-level follow-lock fix for https://github.com/openclaw/openclaw/issues/92386, preserve the existing auto-scroll settings, and retire the overlapping repair path after maintainers choose the landing branch.

## Reproduction Assessment

Yes from source inspection and the linked issue: streaming calls scheduleChatScroll while handleChatScroll can leave near-bottom follow state set during the programmatic guard. I did not execute the live UI in this read-only review.

## Solution Assessment

Yes, the helper-level follow-lock is the right owner boundary for this bug because the race lives in shared WebChat scroll state, not in a message-rendering caller. The older broader repair is an alternative, but this branch is narrower and based on current main.
