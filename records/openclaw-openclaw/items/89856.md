## Summary

Keep open. Current `main` and v2026.6.6 still skip the normal cache-TTL pruning factory and marker path for Codex app-server turns, while Codex owns native thread history outside the OpenClaw replay context that cache-TTL pruning trims.

## What This Changes

Configured `agents.defaults.contextPruning.mode = "cache-ttl"` should arm cache-TTL pruning and record cache markers for Codex app-server sessions.

## Best Possible Solution

Add Codex-harness-aware pruning or an explicit unsupported-path diagnostic at the app-server boundary while keeping canonical `openai/*` routing; do not revive `openai-codex` or append cache markers without real pruning.

## Reproduction Assessment

Yes from source inspection: the same eligibility gate controls both pruning registration and `openclaw.cache-ttl` marker append, and Codex app-server turns keep native history outside the normal OpenClaw replay surface. A live OAuth canary was not run.

## Solution Assessment

No; a simple provider-name allowlist is not the best fix because Codex app-server history is not the normal OpenClaw message replay surface. The best path is a Codex-aware pruning/history API integration or a clear unsupported-path diagnostic.
