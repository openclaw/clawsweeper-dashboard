## Summary

Keep open. Current main and the latest release still do not make configured cache-TTL pruning active for Codex app-server sessions: the normal OpenClaw pruning path is gated by provider eligibility, while the Codex harness keeps native thread continuity outside the normal message-context extension path.

## What This Changes

The issue asks for `agents.defaults.contextPruning.mode = "cache-ttl"` to be active and observable for Codex runtime sessions using the legacy `openai-codex`/Codex path.

## Best Possible Solution

Add a Codex-harness-aware pruning or diagnostic path at the Codex app-server boundary while preserving the canonical `openai` provider route; do not just re-enable the legacy `openai-codex` provider name or append a cache marker without real pruning behavior.

## Reproduction Assessment

Yes from source inspection for the reported missing `openclaw.cache-ttl` marker and inactive pruning path: current main only appends the marker when provider eligibility passes, and Codex app-server turns do not use the normal pruned message-context path. A live Codex OAuth run was not attempted.

## Solution Assessment

Unclear as an implementation path. The issue is valid, but the best fix likely belongs at the Codex app-server context/history boundary or as an explicit unsupported-path diagnostic, not as a simple provider allowlist change.
