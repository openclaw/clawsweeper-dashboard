## Summary

Keep open. Current main and the latest v2026.6.8 release still do not provide real cache-TTL pruning for Codex app-server turns: the normal pruning/marker path is provider-gated, and the Codex harness keeps native thread history outside OpenClaw's pruned replay messages.

## What This Changes

The issue reports that configured cache-TTL context pruning does not arm for Codex app-server sessions, leaving no `openclaw.cache-ttl` marker and no pruning of retained tool results.

## Best Possible Solution

Add a Codex-harness-aware pruning/history integration or an explicit unsupported-path diagnostic at the app-server boundary; do not revive `openai-codex` routing or append cache markers without real native-history pruning.

## Reproduction Assessment

Yes, from source inspection: current main gates both cache-TTL pruning registration and marker append behind provider eligibility, while the Codex app-server path keeps native history outside OpenClaw's pruned replay messages. I did not run the live OAuth canary.

## Solution Assessment

No, a simple provider-name allowlist is not the best fix because Codex app-server history is not the normal OpenClaw message replay surface. The maintainable fix is either a Codex-aware native-history pruning API/integration or a clear unsupported-path diagnostic.
