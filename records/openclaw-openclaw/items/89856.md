## Summary

Keep open. Current main and v2026.6.8 still gate cache-TTL pruning and `openclaw.cache-ttl` marker emission behind provider eligibility that excludes OpenAI/Codex, while Codex app-server sessions keep native thread history outside the normal OpenClaw pruned-message replay path.

## What This Changes

The issue reports that configured cache-TTL context pruning is silently inactive for OpenAI/Codex app-server sessions, leaving no `openclaw.cache-ttl` marker and retaining old tool results.

## Best Possible Solution

Add Codex-harness-aware native-history pruning, or an explicit unsupported diagnostic at the Codex app-server boundary, without reviving legacy `openai-codex` routing or marker-only behavior.

## Reproduction Assessment

Yes, at source level: current main gates both cache-TTL pruning registration and marker append behind provider eligibility, while Codex app-server turns keep native thread continuity outside the normal pruned-message replay path. I did not run the live OAuth canary.

## Solution Assessment

No, a simple provider-name allowlist is not the best fix because it could emit markers without pruning native Codex thread history. The maintainable solution is Codex-aware native-history pruning or a clear unsupported-path diagnostic.
