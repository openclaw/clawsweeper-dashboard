## Summary

Keep open. Current main and the latest release still gate cache-TTL pruning and `openclaw.cache-ttl` marker emission behind provider eligibility that excludes OpenAI/Codex, while Codex app-server sessions keep native thread continuity outside the embedded OpenClaw pruned-message replay path.

## What This Changes

The issue asks for configured cache-TTL context pruning to become active for OpenAI Codex/Codex app-server runtime sessions.

## Best Possible Solution

Define the Codex-owned pruning contract: either implement cache-TTL for Codex app-server with native-thread-safe pruning/rotation and marker proof, or explicitly document/exclude Codex and route long-session pruning to the rolling-pruning work.

## Reproduction Assessment

Yes. Source inspection shows the configured cache-TTL path is skipped for OpenAI/Codex eligibility and Codex app-server turns do not replay OpenClaw-pruned history; I did not run the live canary.

## Solution Assessment

Unclear. A provider allowlist is not clearly the best fix because Codex app-server owns native thread continuity; the safer solution needs a Codex-aware pruning/rotation contract or an explicit unsupported-path diagnostic.
