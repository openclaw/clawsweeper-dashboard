## Summary

Keep open. Current main and v2026.6.8 still gate cache-TTL pruning and `openclaw.cache-ttl` markers behind provider eligibility that excludes OpenAI/Codex, while Codex app-server sessions continue through native Codex thread history rather than OpenClaw's pruned message replay path.

## What This Changes

The issue asks for configured cache-TTL context pruning and `openclaw.cache-ttl` transcript markers to be active for OpenAI/Codex runtime sessions instead of silently skipped.

## Best Possible Solution

Define and test the Codex-native behavior: either make configured pruning affect the app-server model-visible history, or document and diagnose that `cache-ttl` is unsupported for native Codex thread continuity.

## Reproduction Assessment

Yes. Source inspection reproduces the configuration gap: OpenAI/Codex is not cache-TTL eligible, so both the pruning extension and `openclaw.cache-ttl` marker path skip the configured mode; a live canary was not run in this read-only review.

## Solution Assessment

No. Adding OpenAI/Codex to the current eligibility check would only arm markers and the generic pruning extension; the maintainable fix must account for Codex native thread history or make unsupported behavior explicit.
