## What This Changes

The PR changes Control UI session creation to derive `parentSessionKey` from the current selected session instead of the possibly stale `sessionsResult`, and adds a stale-session-list regression test.

## Best Possible Solution

Land the narrow Control UI parent-lineage fix after a real dashboard/browser proof shows a stale-list new chat remains parent-linked and displays assistant replies.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: if the selected session key is not present in `state.sessionsResult.sessions`, current main passes no `parentSessionKey`. I did not run a live browser repro in this read-only review.

## Solution Assessment

Yes for the code shape: the Control UI should pass the selected session as the parent and let the gateway's existing parent validation reject genuinely unknown parents. The solution is not merge-ready until real dashboard behavior proof is supplied.
