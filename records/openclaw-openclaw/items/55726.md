## Summary

Keep open. The PR addresses a real current-main security-audit gap, but the current diff is not mergeable as-is because the new legacy `dm.policy` scan can override a safer modern `dmPolicy`, producing false CRITICAL audit findings for closed DMs.

## What This Changes

The PR changes the security exposure matrix collector so open `dmPolicy` and legacy `dm.policy` paths are merged with open `groupPolicy` for the existing elevated/runtime-fs audit findings while preserving the existing `open_groups` check IDs.

## Best Possible Solution

Update the DM-policy collector to mirror effective policy resolution: modern `dmPolicy` wins, legacy `dm.policy` is only a fallback when the modern key is absent, and account-level policy follows the same precedence/inheritance rules. Add focused tests for modern open DM policy, legacy-only fallback, account-level policy, and modern-overrides-legacy negative cases, then review and land this PR to resolve #55612.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: current main misses `dmPolicy="open"` in `collectExposureMatrixFindings()`, and the PR bug is reproducible with a Discord/Slack config that has `dmPolicy: "allowlist"` plus stale `dm: { policy: "open" }`.

## Solution Assessment

No. The direction is correct, but the implementation should resolve effective DM policy consistently with the config schemas instead of treating modern and legacy keys as independently open.
