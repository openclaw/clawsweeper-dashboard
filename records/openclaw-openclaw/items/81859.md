## Summary

Keep open. Current main still lacks a source-backed way to detect launchd submitted updater jobs, while the related closed PR discussion shows the stale job producer was not proven to be OpenClaw and the service-env wrapper is intentional secret-safe behavior, so this needs maintainer scoping rather than cleanup closure.

## What This Changes

The issue asks OpenClaw to prevent or diagnose macOS update failures where a stale launchd submitted update job and the generated service-env wrapper leave the gateway down after a beta update.

## Best Possible Solution

Keep this open while maintainers decide the OpenClaw-owned scope: source-proven update/service diagnostics and wrapper failure reporting should be handled separately from cleanup of externally produced submitted launchd jobs.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No; the incident logs are detailed and source inspection proves current status/doctor do not inspect submitted launchd jobs, but I did not reproduce the launchd loop on current main and the prior investigation says the submitted-job producer was not OpenClaw.

## Solution Assessment

Is this the best way to solve the issue? Unclear; adding blanket cleanup for submitted `*.openclaw.update.*` jobs is not proven safe, while changing the service-env wrapper needs a maintainer security/product decision.
