## Label Justifications

- `P2`: The bug blocks checkpoint visibility and restore/branch access for historical sessions, but it has a bounded gateway/session surface and an open targeted fix path.
- `impact:session-state`: The affected data is session checkpoint state: valid transcript checkpoints exist on disk but are not surfaced through session rows or compaction APIs.

## Summary

Keep open: current main still makes checkpoint rows and compaction APIs depend on stored `compactionCheckpoints` metadata, while the exact disk-discovery fix is tracked by an open closing PR.

## What This Changes

The issue asks for the Web UI and session APIs to show existing disk checkpoint files for historical sessions whose checkpoint metadata is missing.

## Best Possible Solution

Review and land or replace https://github.com/openclaw/openclaw/pull/81108 so gateway session rows and `sessions.compaction.*` consistently discover valid disk checkpoint transcripts when stored metadata is missing.

## Reproduction Assessment

Yes. A current-main session entry with no `compactionCheckpoints` but valid `{session}.checkpoint.{uuid}.jsonl` files will return an empty `sessions.compaction.list` result because the gateway and row projection consume only stored metadata.

## Solution Assessment

Yes. Gateway-side disk discovery behind the existing session APIs is the narrow maintainable fix; a UI-only workaround would still leave get, branch, and restore inconsistent.
