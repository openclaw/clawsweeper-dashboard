## Summary

Keep open: current main has an OpenClaw-aware safe restart path, but the reported raw systemd/SIGTERM restart from inside a Telegram or Discord turn can still bypass that path and drop the visible reply. A production confirmation and related open PRs show this is active restart work, not conservative cleanup material.

## What This Changes

The issue requests a drain-aware guard, refusal, or deferred restart path so chat-initiated gateway restarts do not drop in-flight Telegram or Discord replies.

## Best Possible Solution

Route channel-turn gateway restarts through one drain-aware path, and explicitly guard or document raw service-manager restarts that cannot preserve final delivery.

## Reproduction Assessment

Yes, at source level: safe restarts count pending replies, but a raw SIGTERM without a restart intent still enters the stop path. The issue also has an independent production confirmation for a Telegram-originated turn.

## Solution Assessment

No, current main is not a complete solution for this report. The safe restart coordinator is the right foundation, but raw systemd restarts from a chat turn still need a hard handoff, refusal, or documented unsupported boundary.
