## Summary

Keep open: current main has a drain-aware safe restart path, but the reported raw Linux systemd/SIGTERM path can still bypass it and drop the in-flight Telegram or Discord reply. The fresh production confirmation and overlapping open restart PRs make this maintainer coordination work, not a cleanup close.

## What This Changes

The issue asks OpenClaw to prevent or safely defer Gateway self-restarts initiated from Telegram/Discord chat turns so the current visible reply is not silently lost.

## Best Possible Solution

Route channel-originated Gateway restart requests through one drain-aware path, then add a hard guard or documented refusal for raw service-manager restarts that cannot preserve the final channel reply.

## Reproduction Assessment

Yes, at source level: safe restarts count pending replies, but a raw service-manager SIGTERM without restart intent still enters the normal stop path and can kill the in-flight channel delivery chain before finalization.

## Solution Assessment

No, current main is not complete for this report. The safe restart coordinator is the right base, but prompt guidance and explicit `--safe` do not protect raw service-manager restarts launched from a chat turn.
