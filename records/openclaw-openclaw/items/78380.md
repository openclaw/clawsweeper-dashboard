## Summary

Keep open: current main has an OpenClaw-aware safe restart path, but raw systemd/SIGTERM restarts from inside a channel turn can still bypass it and drop in-flight replies. The production confirmation plus related open PRs make this active restart reliability work, not conservative cleanup material.

## What This Changes

The issue asks OpenClaw to prevent or defer gateway self-restarts invoked from channel turns so Telegram/Discord replies are delivered instead of silently dropped.

## Best Possible Solution

Route channel-turn gateway restarts through one drain-aware safe restart path, with a hardened detached systemd handoff or an explicit refusal/documented unsupported boundary for raw service-manager restarts that cannot preserve delivery.

## Reproduction Assessment

Yes, at source level: current main treats SIGTERM without a restart intent as stop, while the systemd adapter still performs direct `systemctl --user restart` under a control-group unit. The issue also has an independent production confirmation for a Telegram-originated turn.

## Solution Assessment

No, current main is not a complete solution for this report. The safe restart coordinator is the right foundation, but raw service-manager restarts from a chat turn still need a hardened handoff, refusal, or documented unsupported boundary.
