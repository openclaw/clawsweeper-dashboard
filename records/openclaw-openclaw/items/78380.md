## Summary

Keep open: current main has safe restart primitives, but unsafe plain/service-manager restart paths still exist and the requested automatic block/defer behavior for channel-originated exec self-restarts is not implemented.

## What This Changes

The issue requests that gateway restarts initiated from a Telegram/Discord chat turn be blocked or deferred so the current channel reply is delivered before the gateway exits.

## Best Possible Solution

Keep the existing safe restart primitive, then add a maintainer-approved guard or conversion path so channel-originated maintenance cannot use direct service-manager restarts before the visible reply has drained.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: a direct systemctl-triggered SIGTERM without OpenClaw restart intent is handled as stop, while the documented drain behavior only applies to the --safe/gateway-tool path. I did not live-run Telegram or Discord in this read-only review.

## Solution Assessment

Unclear. The existing safe restart path is the right primitive, but the best user-facing fix needs a maintainer decision between blocking unsafe exec commands, defaulting more restart paths to safe behavior, or converting channel-originated restart requests into deferred gateway-tool restarts.
