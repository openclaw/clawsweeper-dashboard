## Label Justifications

- `P1`: A real channel-originated maintenance turn can silently lose the user-visible Telegram/Discord reply while the gateway appears healthy after restart.
- `impact:message-loss`: The reported failure is a dropped in-flight channel reply after a mid-turn gateway self-restart.

## Summary

Keep open: current main and the latest release still have the reported raw service-manager self-restart hazard. The safe restart coordinator and gateway tool paths drain or coordinate replies, but an in-turn raw `systemctl --user restart openclaw-gateway.service` reaches the gateway as SIGTERM without restart intent and is handled as a stop, so the current Telegram/Discord reply can still be lost.

## What This Changes

The issue requests preventing chat-originated gateway self-restarts from dropping the current Telegram/Discord reply by blocking, deferring, or safely coordinating inline restart attempts.

## Best Possible Solution

Make the gateway tool or `openclaw gateway restart --safe` the supported chat-initiated maintenance path, and have raw in-turn service-manager restart attempts refuse or route to that coordinator with a visible explanation.

## Reproduction Assessment

Yes, source-level: raw SIGTERM without an OpenClaw restart intent is handled as `stop`, while pending-reply drain is wired only for expected restart closes. The issue also has a production confirmation comment for the Telegram-originated failure mode.

## Solution Assessment

No, current main is not the best end state yet. The safe restart coordinator is the right foundation, but raw service-manager restart attempts still need a maintainer decision on block versus rewrite versus explicitly unsupported behavior.
