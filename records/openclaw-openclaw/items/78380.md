## Summary

Keep open: current main and v2026.6.10 still send raw SIGTERM without an OpenClaw restart intent down the stop path, so a systemd/launchd self-restart from an exec turn bypasses the safe restart coordinator and restart-only reply drain.

## What This Changes

The issue reports that channel-originated gateway self-restarts can kill the current Telegram or Discord reply before visible delivery.

## Best Possible Solution

Keep this open until maintainers define and implement a restart-safety policy that routes channel-initiated maintenance through the safe coordinator or refuses unsafe inline service-manager restarts.

## Reproduction Assessment

Yes, at source level: current main sends SIGTERM without an OpenClaw restart intent down the stop path, while restart drain and pending-reply drain are only wired for restart paths. I did not run a live destructive channel restart in this read-only review.

## Solution Assessment

Unclear as an implementation path: the safe restart coordinator is the right foundation, but maintainers need to decide whether raw service-manager restarts from exec/tool turns should be blocked, converted, or documented as unsupported.
