## Label Justifications

- `P1`: A real channel-originated maintenance turn can silently lose the user-visible Telegram/Discord reply while the gateway appears healthy after restart.
- `impact:message-loss`: The reported failure is a dropped in-flight channel reply after a mid-turn gateway self-restart.

## Summary

Keep open: current main still has the reported raw service-manager self-restart hazard. OpenClaw now has coordinated safe restart paths that drain replies, but a raw in-turn `systemctl --user restart openclaw-gateway.service` still reaches the Gateway as SIGTERM without a restart intent and is handled as a stop, so the current channel reply can be lost.

## What This Changes

The issue asks OpenClaw to prevent or safely defer gateway self-restarts initiated during channel-originated chat turns so Telegram/Discord replies are not lost.

## Best Possible Solution

Make the gateway tool or `openclaw gateway restart --safe` the only supported chat-initiated maintenance path, and have raw in-turn service-manager restart attempts refuse or route to that coordinator with a visible explanation.

## Reproduction Assessment

Yes, source-level. Raw SIGTERM without an OpenClaw restart intent is handled as `stop`, while pending-reply drain is only wired for `isRestart`; the issue also has a production confirmation comment.

## Solution Assessment

No, this is not solved on current main. The safe restart coordinator is the right foundation, but the remaining best fix needs a maintainer decision on how raw exec/service-manager restart attempts should be blocked or redirected.
