## Label Justifications

- `P1`: A channel-originated maintenance turn can silently drop the user-visible Telegram or Discord reply while the gateway later comes back healthy.
- `impact:message-loss`: The reported failure is lost in-flight channel replies during a mid-turn gateway self-restart.

## Summary

Keep open: current main and v2026.6.10 still handle a raw service-manager SIGTERM without an OpenClaw restart intent as a stop, so the restart-only pending-reply drain is skipped. The newer open restart-drain issues and PRs cover adjacent forced or skip-deferral paths, but they do not settle the raw in-turn service-manager restart policy reported here.

## What This Changes

Prevent chat-initiated gateway self-restarts from killing in-flight Telegram or Discord replies before the visible response is delivered.

## Best Possible Solution

Keep this as the canonical tracker for raw in-turn service-manager self-restart policy, and make those attempts either refuse with visible guidance or route through the existing safe restart coordinator while preserving the narrower linked restart-drain issues.

## Reproduction Assessment

Yes, at source level: raw SIGTERM without an OpenClaw restart intent maps to the stop path, and pending-reply drain runs only for expected restart closes. The issue also has reporter logs and an independent production confirmation for a Telegram-originated turn.

## Solution Assessment

Unclear until maintainers choose the policy shape. The safe restart coordinator is the right foundation, but raw in-turn service-manager restarts still need a deliberate block, rewrite, or unsupported-behavior decision.
