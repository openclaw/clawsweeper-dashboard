## Summary

Keep open. The reporter's canon-guardian plugin and #75750 are useful reference implementations for the fallback-context slice, and #70990 appears to track part of the failover-hook work, but current main still leaves meaningful native/core requests unresolved across fallback visibility, config health rollback, filesystem/tool policy, budget/health guardrails, and security hardening. This bundle is too broad and security/product-sensitive for automatic close or a single repair PR.

## What This Changes

This issue requests a consolidated set of platform features around model fallback visibility/context, config and crash recovery, prompt-injection and filesystem/tool safety, subagent/cost/model-health guardrails, doctor/update/write hooks, and per-binary trust.

## Best Possible Solution

Keep this as a maintainer-triage umbrella only long enough to split the remaining work into focused canonical issues: fallback visibility/context, config-write recovery, crash/subagent/cost guardrails, model health/preflight, filesystem/tool policy, doctor/update/write hooks, and exec trust. Plugin-suitable slices should use ClawHub or documented plugin APIs; core trust-boundary slices need explicit security review.

## Reproduction Assessment

Partly yes. Current main has a source- and test-backed reproduction for non-verbose fallback notices being omitted while lifecycle events fire; the remaining entries are feature/security requests with production workaround evidence and need per-slice acceptance criteria rather than one shared repro.

## Solution Assessment

No, not as one bundled implementation. The maintainable solution is to review or land focused follow-ups like #70990 where appropriate, preserve the canon-guardian plugin from #75750 as reference input, and split the remaining core/security items into owner-reviewed changes.
