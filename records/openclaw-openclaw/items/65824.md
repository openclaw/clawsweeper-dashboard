## Summary

Keep open. Current main has important adjacent coverage and the latest `canon-guardian` plugin sketch is useful evidence that one fallback-context workaround may belong in user land, but the umbrella is not implemented: fallback notices remain verbose-gated, `llm_input` is documented and typed as observation-only while `before_prompt_build` is the prompt-mutation hook, and the remaining config safety, model preflight, budget, path-policy, exec-trust, and subagent circuit-breaker slices still need maintainer triage.

## What This Changes

The issue requests a consolidated set of platform hardening features around model fallback visibility/context, config recovery, crash recovery, prompt/tool safety, subagent/cost/model-health guardrails, doctor previews, path-level tool policy, update hooks, file-write hooks, and exec trust.

## Best Possible Solution

Keep this as a maintainer-triage umbrella or split it into focused canonical issues, route plugin-suitable slices through documented plugin hooks or ClawHub guidance, and implement core safety slices only with explicit config contracts, tests, docs, and security/product review.

## Reproduction Assessment

Partly yes. The fallback visibility gap is source- and test-backed on current main, and the reporter provided a concrete 2026.4.27 model-compatibility failure; several other entries are feature requests with production workarounds and need slice-specific reproduction or design checks.

## Solution Assessment

Not applicable as one patch. The narrowest maintainable path is separate owner-reviewed changes: model-failover hook work can proceed independently, while fallback UX/context, config health rollback, cost/model health gates, path policy, and subagent respawn limits need distinct designs.
