## Summary

Keep open. The latest reporter-provided plugin workaround is useful context for the fallback-context slice, but current main still leaves meaningful parts of the bundle unresolved, including non-verbose fallback visibility, config write health rollback, path-level filesystem policy, pre-call budget enforcement, scheduled model health routing, and several security-sensitive hardening requests. This is too broad and security/product-heavy for an automatic close or one repair PR.

## What This Changes

The issue requests a consolidated set of reliability, safety, model fallback, config recovery, tool policy, hook, budget, and health-check improvements for intensive OpenClaw operation.

## Best Possible Solution

Keep this as a maintainer-triage umbrella only if maintainers want one, otherwise split the remaining work into canonical focused issues: fallback visibility/context, config-write recovery, subagent/cost guardrails, model health/preflight, filesystem/tool policy, and plugin-hook ergonomics. Plugin-suitable slices should move toward ClawHub or documented plugin API improvements; core trust-boundary slices need explicit maintainer/security design.

## Reproduction Assessment

Partly yes. Current main has a source- and test-backed reproduction for non-verbose fallback notices being omitted while lifecycle events still fire; the other entries are feature/security requests with production workarounds and need focused per-slice reproduction or design acceptance.

## Solution Assessment

No, not as one bundled implementation. The maintainable path is to review the open failover hook subset separately, preserve the reporter’s canon-guardian plugin as useful design input, and split the remaining core/security work into narrow owner-reviewed changes.
