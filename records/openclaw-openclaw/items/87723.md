## Summary

Keep open: current main and v2026.5.27 still have a real false-positive path where source-authored custom env SecretRef markers in generated `models.json` can be reported as plaintext, even though docs and model generation treat those markers as non-secret.

## What This Changes

The issue requests that `openclaw secrets audit --check` distinguish environment variable references from plaintext API keys so env-var names in `models.json` do not produce plaintext-secret warnings.

## Best Possible Solution

Teach secrets audit to suppress only source-authoritative env SecretRef markers for generated `models.json` provider keys, while continuing to warn on arbitrary all-caps strings and real plaintext keys.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: generate or write a `models.json` provider `apiKey` marker from a custom env SecretRef such as `CUSTOM_PROVIDER_API_KEY`, then the current audit path treats it as plaintext unless it is in the known marker list. I did not run a live command because this review is read-only.

## Solution Assessment

Yes, the best fix is a narrow source-authoritative marker check, not the issue body's broader keyword heuristic. That preserves true plaintext detection while removing false positives for markers OpenClaw itself writes.
