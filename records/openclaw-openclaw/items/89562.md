## Summary

Keep open: current main still has a plausible voice-call/session lifecycle gap, and I did not find a merged fix or a canonical issue that owns this exact hangup-aborts-`sessions_spawn` behavior. The source shows voice-call embedded runs still run on the `voice` lane and `sessions_spawn` only records an accepted child after the child agent call/register path, but the exact live hangup-to-abort race still needs maintainer lifecycle confirmation or a focused repro.

## What This Changes

The issue asks for voice-call hangups not to abort independent `sessions_spawn` child work once spawn execution has begun.

## Best Possible Solution

Define the hangup-versus-stop lifecycle contract, then add a bounded fix so an accepted `sessions_spawn` child can survive voice-call teardown while deliberate stop/reset paths still cascade, with focused regression coverage for the voice-call hangup race.

## Reproduction Assessment

No high-confidence runnable reproduction yet. Source inspection shows the plausible lane/abort and child-spawn acceptance paths, but a live voice-call hangup or focused failing test is still missing for current main.

## Solution Assessment

Unclear until maintainers choose the lifecycle boundary. A gateway/tool-level accepted-spawn decoupling looks cleaner than a plugin-local grace timer if it preserves the documented non-blocking `sessions_spawn` contract without weakening deliberate stop/reset semantics.
