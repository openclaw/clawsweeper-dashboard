## Summary

Keep open: current main still has an unresolved Voice Call/session lifecycle question. Source shows voice consults run embedded agent work on the voice lane and `sessions_spawn` registers accepted ACP children only after spawn startup returns, but there is no current fix or live proof establishing the desired hangup behavior.

## What This Changes

Hanging up a Voice Call consult should not abort an in-flight `sessions_spawn` child session before the child has initialized.

## Best Possible Solution

Define and test a Voice Call consult lifecycle contract where hangup ends call audio while either preserving accepted independent child sessions or explicitly canceling all call-owned work, then implement that policy at the plugin/gateway boundary with live or focused regression proof.

## Reproduction Assessment

No: there is not yet a high-confidence live reproduction against current main. Source shows a plausible voice consult to embedded runner to `sessions_spawn` path, but it does not fully prove the specific hangup-to-abort signal without a real call or focused failing test.

## Solution Assessment

Unclear: a gateway-level `sessions_spawn` carve-out and a plugin-level grace drain are both plausible. The best fix depends on the maintainer decision for whether hangup cancels all call-owned work or allows independent child sessions to continue.
