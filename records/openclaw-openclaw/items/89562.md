## Label Justifications

- `P2`: This is a plausible shipped workflow bug with limited blast radius around voice-call initiated background session work, but it is not a core outage or security emergency.
- `impact:session-state`: The report is about spawned child session/run state being aborted or never registered when the parent voice-call lifecycle ends.

## Summary

Keep open: current main and the latest release still route voice-call embedded runs through the `voice` lane, and the source path plausibly lets parent-run aborts cancel `sessions_spawn` before the child ACP session is fully accepted and tracked.

## What This Changes

The issue reports that ending a voice call aborts in-flight `sessions_spawn` work before spawned child ACP sessions initialize.

## Best Possible Solution

Define a bounded accepted-spawn lifecycle boundary so child work that `sessions_spawn` has accepted and dispatched is registered and no longer depends on the parent voice-call attempt signal, while pre-accept failures still clean up and report normally.

## Reproduction Assessment

No high-confidence end-to-end repro is available from this review, but the source path is reproducible with medium confidence: voice runs use the `voice` lane, embedded attempts relay parent abort signals, tool execution receives that signal, and ACP spawn registration happens only after dispatch returns.

## Solution Assessment

Unclear whether the suggested gateway exemption or plugin grace wait is the best fix. A config knob is not the right shape; maintainers should choose the lifecycle boundary that preserves normal cancellation while protecting accepted `sessions_spawn` child work.
