## Label Justifications

- `P2`: The report describes a bounded session-lifecycle bug affecting voice-call background work, not an emergency outage.
- `impact:session-state`: The central failure is abort propagation from the call session into spawned child session state.

## Summary

Keep open: current main and the latest release still route voice-call responses through the `voice` lane, abort embedded attempts through the parent run signal, and only record accepted `sessions_spawn` work after child dispatch returns, so the reported lifecycle gap is not implemented away.

## What This Changes

The issue asks for voice-call hangup not to abort in-flight `sessions_spawn` work once the spawned child session is logically independent of the call.

## Best Possible Solution

Define and test a narrow accepted-spawn lifecycle boundary so child sessions accepted by `sessions_spawn` survive parent voice-call teardown while pre-accept failures and explicit cancellations still clean up normally.

## Reproduction Assessment

No high-confidence live reproduction has been run. Source inspection shows the plausible path: voice responses use the `voice` lane, embedded attempts propagate abort into tool execution, and `sessions_spawn` accepted-run tracking happens only after child dispatch returns.

## Solution Assessment

Unclear. A narrow accepted-spawn lifecycle boundary is likely more maintainable than a broad grace timeout or new config knob, but maintainers need to define how hangup, timeout, and explicit cancellation interact.
