## Summary

Keep open: the report is a credible P1 Telegram delivery regression, current main still uses the raw rich-message path for default text and draft previews, and the linked repair PR is open but not merged or proof-complete.

## What This Changes

The issue reports that upgrading to 2026.6.8-beta.1 made every Telegram-bound agent reply render as an unsupported message across multiple Telegram clients, with downgrade to 2026.6.7-beta.1 restoring delivery.

## Best Possible Solution

Land a focused Telegram fix that restores readable default text, edit, and draft delivery through standard sendMessage/editMessageText with HTML/plain fallback, then prove it in real Telegram clients before closing.

## Reproduction Assessment

Yes, source and report evidence give a high-confidence reproduction path: run 2026.6.8-beta.1 or current main with a Telegram-bound agent and trigger a normal reply or /model response, where current code still uses raw rich-message sends. I did not perform a live Telegram repro in this read-only review.

## Solution Assessment

Unclear until https://github.com/openclaw/openclaw/pull/93279 has live client proof, but restoring readable sendMessage/editMessageText defaults is the safest emergency direction. Rich Telegram transport can be revisited after default delivery is proven stable.
