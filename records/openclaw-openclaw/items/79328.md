## Summary

Keep open: the plugin-owned direction has useful maintainer/domain-owner support and terminal proof, but the current head still has merge-blocking command availability, docs, dependency metadata, and generated-doc issues under a protected maintainer review label.

## What This Changes

The PR moves the `openclaw path` surface from core into a bundled `oc-path` plugin with local parser dependencies, docs, labeler, tests, and lockfile changes.

## Best Possible Solution

Land the plugin-owned `oc-path` direction only after default CLI availability, docs, generated plugin inventory, lockfile metadata, CI, and maintainer direction are all clean.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: the PR removes the core `path` registrar, the bundled plugin manifest lacks `enabledByDefault`, and current plugin loading disables and skips such bundled plugins before CLI registration; the docs and lockfile mismatches are directly visible on the PR head.

## Solution Assessment

No. The core-to-plugin move is plausible and has domain-owner support, but the current diff is not the best mergeable shape until it preserves fresh default `openclaw path` availability and aligns docs, generated inventory, changelog, and dependency metadata with the trimmed implementation.
