## What This Changes

The PR removes the core `src/oc-path` implementation and core `openclaw path` registration, reintroduces the same CLI as a bundled `extensions/oc-path` plugin with plugin-local dependencies, and updates CLI docs and labeler metadata.

## Best Possible Solution

## Reproduction Assessment

Yes. From source, invoking `openclaw path` without a subcommand on the PR branch would use a recreated parent command without the current `applyParentDefaultHelpAction(path)` behavior, while current main has that helper wired.

## Solution Assessment

No. The plugin move may be a reasonable direction, but this draft still needs maintainer product approval, restored parent-command help behavior, generated plugin docs, and real behavior proof before merge.

- **[P2] Restore parent help behavior for `openclaw path`:** `extensions/oc-path/src/cli.ts:549-553`
  - body: The core command previously ended with `applyParentDefaultHelpAction(path)`, which preserves the behavior from https://github.com/openclaw/openclaw/issues/73077: `openclaw path` with no subcommand prints help and exits 0. That leaves public plugin docs inconsistent with the new shipped plugin surface; please rerun the plugin inventory generator and include the generated reference/index updates.
  - confidence: 0.78
