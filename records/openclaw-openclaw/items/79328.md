## Summary

Keep open: the bundled-plugin direction is consistent with OpenClaw's plugin boundary, but the PR still has a source-reproducible JSONC insertion bug that should be fixed before merge.

## What This Changes

This PR replaces the prior core oc-path implementation with an optional bundled `oc-path` plugin that owns `openclaw path`, docs, tests, package metadata, and label routing.

## Best Possible Solution

Land the plugin-owned CLI after normalizing JSONC insertion parents through the same concrete path resolution used by lookup, with a regression test for positional parent insertion.

## Reproduction Assessment

Yes. On PR head 4e5db9d83b27837f0e677990da202f0d24c0aa95, source inspection shows `resolveJsoncInsertion` accepts positional parents while `mutateAt` later treats the raw `$last` segment as `Number("$last")`, so the set path can fail after reporting a valid insertion point.

## Solution Assessment

No. The bundled-plugin ownership is the right direction, but the implementation is not the best mergeable form until JSONC insertion mutation consumes a resolved concrete parent path and the missing regression coverage is added.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Normalize JSONC insertion parent segments:** `extensions/oc-path/src/oc-path/universal.ts:607-609`
  - body: `resolveJsoncInsertion` accepts insertion parents reached through positional segments such as `items.$last.tags`, but `mutateJsoncContainer` replays the raw parent path and array descent uses `Number(seg)`. That means a path can report a valid insertion point and then fail as unresolved during `set`; normalize positional segments during mutation or pass the concrete resolved parent path through.
  - confidence: 0.91
- **[P3] Clarify validate parse-error exit codes:** `docs/cli/path.md:354`
  - body: The validate section and CLI test pin malformed `validate` input to exit 1, but the global exit-code table still says parse errors are 2. Scripts using the table will handle invalid paths incorrectly; either make `validate` return 2 or carve out its documented invalid-path behavior in the table.
  - confidence: 0.79
