## Summary

Keep open: the plugin-owned direction has supportive discussion and real terminal proof, but the branch is not merge-ready because the new CLI docs do not match the opt-in activation or trimmed grammar, and the user-facing change is missing a changelog entry.

## What This Changes

The PR adds a bundled `oc-path` plugin with plugin-local parser dependencies, plugin CLI registration for `openclaw path`, docs and labeler updates, tests, fixtures, and lockfile entries.

## Best Possible Solution

Land the plugin-owned oc-path direction after the docs match the chosen opt-in/default policy and actual md/jsonc/jsonl grammar, with release notes for the new CLI surface.

## Reproduction Assessment

Yes for the review findings: source inspection of the PR manifest/docs plus current plugin CLI policy shows the availability docs mismatch, and the implementation contradicts the grammar docs. I did not run tests because this is a read-only review.

## Solution Assessment

No. The plugin boundary is maintainable and supported by the related discussion, but this PR is not the best mergeable form until docs and changelog line up with the manifest activation and trimmed grammar; if maintainers want core-like availability, `enabledByDefault` is the safer alternative to documenting opt-in.
