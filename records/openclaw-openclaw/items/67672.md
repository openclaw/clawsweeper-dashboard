## Summary

Keep open. Current main still lacks an explicit awaited Slack SocketModeClient disconnect in the stop path, and the PR is a plausible narrow fix, but it needs to be adapted to the current provider-support helper and release hygiene before merge.

## What This Changes

The PR adds a Slack SocketModeClient disconnect helper, calls it before Bolt app stop during Slack cleanup, and removes a redundant SlackProbe cast.

## Best Possible Solution

Land a Slack-owned cleanup in provider-support's gracefulStopSlackApp that awaits the underlying SocketModeClient.disconnect() before app.stop(), with focused reconnect cleanup coverage and a changelog entry.

## Reproduction Assessment

No high-confidence live reproduction was run for the full production OOM symptom. There is a high-confidence static path: health-monitor restarts stop a running Slack account, current cleanup only awaits Bolt app.stop(), and Bolt's receiver stop does not await the underlying client disconnect.

## Solution Assessment

Yes directionally, but not as currently rebased. The narrow maintainable solution is to put the awaited disconnect in the shared Slack cleanup helper on current main, keep the stopAccount hook out, and cover the cleanup behavior with a focused test.
