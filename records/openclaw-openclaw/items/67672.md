## Summary

Keep open: current main still lacks an explicit awaited Slack SocketModeClient disconnect, and the related Feishu fix is only precedent, not a Slack superseder. The PR remains useful but is not merge-ready because its disconnect hunk targets an older provider-local helper instead of the active provider-support.ts cleanup path, and the external PR still lacks after-fix real Slack proof.

## What This Changes

The PR adds a Slack SocketModeClient accessor/disconnect helper, awaits it before Bolt app stop during provider cleanup, and removes a redundant SlackProbe cast.

## Best Possible Solution

Land a narrow Slack-owned cleanup on current main that awaits the underlying SocketModeClient.disconnect() before Bolt app.stop() in the shared Slack cleanup helper, keeps stopAccount out, preserves the redundant-cast cleanup, and adds focused disconnect call-order/awaiting coverage.

## Reproduction Assessment

No high-confidence live reproduction was run for the production OOM. Source inspection confirms the health-monitor stop/start path and current cleanup gap, and dependency source confirms SocketModeClient.disconnect() is the awaitable teardown primitive.

## Solution Assessment

No, not as currently proposed. The direction is narrow and maintainable, but the disconnect must move into provider-support.ts on current main with regression coverage and after-fix runtime proof.
