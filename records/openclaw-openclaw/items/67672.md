## Summary

Keep open. Current main still lacks the Slack SocketModeClient awaited-disconnect cleanup, and the PR addresses a real gap supported by the Slack SDK behavior and the reported stale-socket restart path. It should not be closed as implemented, but the branch is currently dirty against main and needs adaptation to the current provider-support helper plus a changelog entry and focused regression test before merge.

## What This Changes

The PR branch adds a Slack SocketModeClient disconnect helper, calls it before Bolt app stop during Slack cleanup, and removes a redundant SlackProbe cast.

## Best Possible Solution

Land a narrow Slack-owned cleanup on current main that awaits the underlying SocketModeClient.disconnect() in the shared Slack shutdown helper before Bolt app.stop(), keeps the removed stopAccount hook out, adds a focused reconnect-helper regression test for disconnect ordering/awaiting, and records the user-facing fix in the changelog.

## Reproduction Assessment

No high-confidence live reproduction was run for the full production OOM symptom. There is a high-confidence static reproduction path for the cleanup gap: stale health restarts call stopChannel, Slack cleanup reaches gracefulStopSlackApp, and current main only awaits Bolt app.stop() while Bolt's SocketModeReceiver.stop() does not await the SocketModeClient disconnect promise.

## Solution Assessment

Yes, directionally: explicitly awaiting SocketModeClient.disconnect() in Slack-owned cleanup is the narrow maintainable fix. The best landing shape is to adapt it to the current provider-support helper, keep the stopAccount hook out, and add regression coverage plus changelog credit.
