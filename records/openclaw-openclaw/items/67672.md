## Summary

Keep open. The Slack stale-socket leak fix is still plausible and not implemented on current main, but this branch is conflicting, places the disconnect in an inactive provider-local helper instead of the current shared helper, and lacks after-fix real Slack runtime proof.

## What This Changes

The PR adds a Slack SocketModeClient accessor/disconnect helper, awaits it before provider stop, and removes a redundant Slack probe cast.

## Best Possible Solution

Land a current-main Slack cleanup that awaits SocketModeClient.disconnect() in provider-support gracefulStopSlackApp before app.stop(), keeps stopAccount out, and adds focused call-order/awaiting coverage.

## Reproduction Assessment

No high-confidence live reproduction was run for the production OOM. Source inspection confirms the health monitor stop/start path and current cleanup gap, and dependency source shows SocketModeClient.disconnect() is the awaitable teardown primitive.

## Solution Assessment

No, not as currently proposed. The direction is narrow and maintainable, but the disconnect needs to move into the active shared provider-support.ts helper with regression coverage and after-fix runtime proof.
