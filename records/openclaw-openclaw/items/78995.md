## Summary

Current main already provides a supported managed-service wrapper path that can inject Node heap flags without weakening the intentional NODE_OPTIONS block, and that support shipped in v2026.4.26.

## What This Changes

The issue requests a supported way to configure Node.js heap flags for the managed gateway service because plist edits are regenerated and NODE_OPTIONS is filtered.

## Best Possible Solution

Keep the shipped wrapper-based service customization path and document heap-wrapper examples if maintainers want more operator guidance, while preserving the NODE_OPTIONS security block.

## Reproduction Assessment

Not applicable: this is a feature/support-path request rather than a broken existing behavior report. Source and docs inspection show the supported wrapper path exists on current main and is covered by tests.

## Solution Assessment

Yes: the existing wrapper contract is the narrowest maintainable way to pass service-specific Node flags while keeping launchd/systemd generation and host-env filtering under OpenClaw control. A dedicated gateway.nodeHeapSizeMb config key would be a separate product choice, not necessary for the operational path requested here.
