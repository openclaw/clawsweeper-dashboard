## Summary

Keep open: current main still lacks `MEDIA:/path#doc` handling, and the WhatsApp outbound path does not thread `forceDocument` into media loading or Baileys payload selection. This is useful, but it is a new channel/API behavior because the existing public flag is documented as Telegram-only.

## What This Changes

Request a way, such as `MEDIA:/path#doc`, to force media delivery as a document attachment so WhatsApp does not compress quality-sensitive images.

## Best Possible Solution

Choose a canonical public document-delivery control, document supported channels, and thread it through WhatsApp outbound media preparation so normal image sends stay unchanged unless the user explicitly requests document mode.

## Reproduction Assessment

Not a bug-style reproduction, but the missing capability is high-confidence from source: `loadWebMediaInternal` has no `#doc` parsing and the WhatsApp adapter does not pass `forceDocument` into `sendMessageWhatsApp`. I did not run live WhatsApp delivery.

## Solution Assessment

Unclear. The requested `#doc` fragment is plausible, but current main already has `forceDocument`/`asDocument` with Telegram-only docs, so maintainers should choose the public surface before implementation.
