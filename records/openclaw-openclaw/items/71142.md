## Summary

Keep open. Current main and v2026.5.18 removed the old blanket 5 MB parser cap, but the Control UI upload-policy request is still unresolved because image attachments remain capped by the fixed 6 MiB hydration guard and the browser still has no advertised upload limit for prevalidation.

## What This Changes

Request to make Control UI chat upload limits configurable and visible to the browser so larger media can be accepted and prevalidated consistently with model and channel limits.

## Best Possible Solution

Define one effective Control UI upload policy, then thread the resolved limit through Gateway image acceptance, bootstrap/client prevalidation, tests, and docs while respecting the WebSocket payload ceiling and the broader image-limit contract.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure agents.defaults.mediaMaxMb above 6 MiB and upload an image larger than MAX_IMAGE_BYTES; chat.send uses the shared max, but parseMessageWithAttachments still rejects the fixed image cap, and the browser lacks prevalidation before FileReader/WebSocket work.

## Solution Assessment

Unclear. Reusing agents.defaults.mediaMaxMb avoids another competing config source, but maintainers need to decide whether Control UI needs an independent upload-limit contract and how it should coordinate with the broader image-limit policy.
