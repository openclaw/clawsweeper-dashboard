## Summary

Keep open. Current main only partly satisfies the request: `chat.send` already uses the shared configurable attachment cap, but images above the separate 6 MiB image cap still fail, the Control UI bootstrap contract does not expose an effective upload limit, and the browser paste/file/drop paths still read data URLs before any size prevalidation. The recent issue discussion by mayank6136 and hclsys correctly narrowed the remaining work to those gaps rather than the older hardcoded 5 MB path.

## What This Changes

The issue requests a configurable Control UI upload size limit so larger images and other media can be uploaded consistently with OpenClaw media limits and channel behavior.

## Best Possible Solution

Define one effective Control UI upload policy, expose the resolved byte limit through the bootstrap contract, reject oversized paste/file/drop inputs before reading data URLs, keep server-side parsing and MIME checks authoritative, align image hydration with the accepted image limit, and document the configured limit plus the practical WebSocket/base64 ceiling.

## Reproduction Assessment

Yes. Configure `agents.defaults.mediaMaxMb` above 6 MiB and upload an image larger than `MAX_IMAGE_BYTES` through Control UI; source and tests show `chat.send` reaches the shared cap path but `parseMessageWithAttachments` rejects the image at the fixed 6 MiB guard.

## Solution Assessment

Unclear because there is no final patch and maintainers have not chosen the public config shape. The narrowest maintainable direction appears to be reusing the existing shared media cap unless maintainers explicitly want an independent Control UI override.
