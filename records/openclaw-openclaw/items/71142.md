## Summary

Keep open. Current main and the latest v2026.6.6 release still lack an effective Control UI upload policy: the shared server cap is configurable, but images remain capped by `MAX_IMAGE_BYTES`, the browser does not receive an upload limit for prevalidation, and the open stack-safe large-attachment work is adjacent rather than a replacement.

## What This Changes

Request a configurable, consistent Control UI upload size limit so web chat can accept larger media within gateway, media-processing, and transport constraints.

## Best Possible Solution

Define one effective Control UI upload policy, then align server image acceptance, bootstrap data, browser prevalidation, docs, and focused tests with that policy and the WebSocket/base64 ceiling.

## Reproduction Assessment

Yes, by source inspection: configure `agents.defaults.mediaMaxMb` above 6 MiB and send an image larger than `MAX_IMAGE_BYTES` through Control UI; the server still rejects it after the shared cap. The browser path also reads selected files with `FileReader.readAsDataURL` before any size-aware prevalidation.

## Solution Assessment

Unclear. Reusing existing `agents.defaults.mediaMaxMb` appears to be the leaner config direction, but maintainers need to decide whether Control UI needs an independent knob and how it should interact with the stack-safe large-attachment work.
