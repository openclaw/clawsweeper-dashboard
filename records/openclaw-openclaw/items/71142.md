## Summary

Keep open. Current main and v2026.6.10 already route the main server parser limit through `agents.defaults.mediaMaxMb`, but the remaining Control UI upload-size problem is not solved: image uploads still hit the shared 6 MiB image hydration cap, the bootstrap payload does not expose an effective upload limit, and the browser file paths read full files before size-aware validation.

## What This Changes

Request to make Control UI chat uploads configurable and consistent with larger media limits for images and other attachments.

## Best Possible Solution

Reuse `agents.defaults.mediaMaxMb` unless maintainers explicitly approve a separate Control UI knob, expose the resolved effective cap to the browser, prevalidate before `FileReader`, handle the 6 MiB image hydration boundary deliberately, and document the WebSocket/base64 ceiling with focused tests.

## Reproduction Assessment

Yes from source for the remaining current-main behavior: a large image can pass the shared `mediaMaxMb` parser cap and still hit `MAX_IMAGE_BYTES`, while the browser file paths read attachments before any server-provided limit. I did not run a live browser upload in this read-only review.

## Solution Assessment

Unclear until maintainers choose the policy owner. The narrowest maintainable path appears to reuse `agents.defaults.mediaMaxMb`; a separate `gateway.controlUi` knob should only land with explicit product approval and upgrade/docs coverage.
