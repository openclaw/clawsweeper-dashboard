## Summary

Keep open. Current main only partially addresses the request: the old blanket 5 MB parser cap has been replaced by the shared media limit path, but images still hit a fixed 6 MiB guard, Control UI bootstrap exposes no upload policy, and browser upload paths still read full data URLs before any size prevalidation.

## What This Changes

The issue requests a configurable Control UI upload limit so larger images and media can be uploaded consistently with OpenClaw media and channel limits.

## Best Possible Solution

Define one effective Control UI upload policy, expose the resolved byte limit through the bootstrap contract, reject oversized paste/file/drop inputs before reading data URLs, keep server-side parsing and MIME checks authoritative, align image hydration with the accepted image limit, and document the configured limit plus WebSocket/base64 ceiling.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: configure `agents.defaults.mediaMaxMb` above 6 MiB and upload an image larger than `MAX_IMAGE_BYTES` through Control UI; `chat.send` reaches the shared cap path, then `parseMessageWithAttachments` rejects the fixed image cap.

## Solution Assessment

Unclear because there is no final patch and maintainers have not chosen the public config shape. The narrowest maintainable direction appears to be reusing the existing shared media cap unless maintainers explicitly want an independent Control UI override.
