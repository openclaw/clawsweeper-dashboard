## Summary

Keep open: current main and v2026.6.10 no longer have the original fixed 5 MB chat attachment parser cap, but the requested Control UI upload-limit behavior is still incomplete because images remain capped by the 6 MiB media-core image limit and the browser still has no server-provided upload ceiling for prevalidation.

## What This Changes

Requests a configurable Control UI upload-size policy so larger media can be accepted and prevalidated consistently with gateway and media limits.

## Best Possible Solution

Choose one maintainer-approved upload policy, expose the resolved effective limit to the Control UI, prevalidate before file reads and sends, document the WebSocket/base64 ceiling, and keep server enforcement aligned with model/media runtime caps.

## Reproduction Assessment

Yes at source level: current main still rejects image attachments above the 6 MiB media-core cap and Control UI file paths do not prevalidate size before `FileReader.readAsDataURL`. I did not run a live browser upload in this read-only review.

## Solution Assessment

Unclear until maintainers choose policy: reusing `agents.defaults.mediaMaxMb` avoids a new config key, while a Control UI-specific knob gives independent tuning but adds config surface.
