## Summary

Keep open. Current main only partially satisfies the request: `chat.send` now uses the shared configurable attachment cap, but images still hit a fixed 6 MiB parser guard, the Control UI bootstrap contract does not expose an effective upload policy, and browser paste/file/drop paths still read files before any size prevalidation.

## What This Changes

Request to make Control UI chat upload size limits configurable so larger images and files can be accepted consistently with media and channel limits.

## Best Possible Solution

Define one effective Control UI upload policy, thread the resolved byte cap through Gateway bootstrap, reject oversized paste/file/drop inputs before reading data URLs, keep server-side MIME and byte checks authoritative, align the image hydration cap with the accepted upload cap, and document the configured limit plus the practical WebSocket/base64 ceiling.

## Reproduction Assessment

Yes. Configure `agents.defaults.mediaMaxMb` above 6 MiB and upload an image larger than `MAX_IMAGE_BYTES` through Control UI or the `chat.send` attachment path; current source and tests show it rejects at the fixed image guard after passing the shared cap.

## Solution Assessment

Unclear because there is no patch yet and maintainers have not chosen the public config shape. The narrowest maintainable direction appears to be reusing the existing shared media cap unless maintainers explicitly want a Control UI-specific override.
