## Summary

Keep open. Current main and v2026.6.8 still only partially address the request: `chat.send` has a configurable shared server cap, but images still hit the fixed `MAX_IMAGE_BYTES` limit, the Control UI does not receive an effective upload limit, and picker/paste/drop paths still read files before size-aware prevalidation.

## What This Changes

The issue requests configurable Control UI upload limits so web chat uploads can align with OpenClaw media limits and avoid rejecting common larger images.

## Best Possible Solution

Define one effective Control UI upload policy, then align server image acceptance, bootstrap data, browser prevalidation, docs, and focused tests with that policy and the WebSocket/base64 ceiling.

## Reproduction Assessment

Yes, from source inspection. Configure `agents.defaults.mediaMaxMb` above 6 MiB and send an image larger than `MAX_IMAGE_BYTES` through Control UI; the shared max check can pass, but the fixed image cap still rejects it, and the browser reads selected files before size-aware prevalidation.

## Solution Assessment

Unclear until maintainers choose the policy owner. Reusing `agents.defaults.mediaMaxMb` appears leaner than adding a new Control UI config key, but either path needs coordinated server, bootstrap, browser, docs, and transport-ceiling behavior.
