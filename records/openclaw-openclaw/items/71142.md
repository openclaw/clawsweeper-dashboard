## Summary

Keep open. Current main and v2026.6.8 have already moved beyond the original blanket 5 MiB server parser cap, but the effective Control UI upload policy is still incomplete: images above `MAX_IMAGE_BYTES` are still rejected, the bootstrap contract does not expose an upload limit, and file/paste/drop paths still read files before size-aware browser validation.

## What This Changes

The issue requests a configurable effective upload-size policy for Control UI chat attachments so larger media can be accepted and rejected early with clear browser feedback.

## Best Possible Solution

Define one effective Control UI upload policy, then align server image acceptance, bootstrap payload, browser prevalidation, docs, and focused tests with that policy and the WebSocket/base64 ceiling.

## Reproduction Assessment

Yes, source-level. Configure `agents.defaults.mediaMaxMb` above 6 MiB and send an image larger than `MAX_IMAGE_BYTES` through Control UI; the shared max check can pass, but the fixed image cap still rejects it, and the browser reads files before size-aware validation.

## Solution Assessment

Unclear until maintainers choose the policy owner. Reusing `agents.defaults.mediaMaxMb` appears leaner than adding a parallel Control UI config key, but either path needs coordinated server, bootstrap, browser, docs, and transport-ceiling behavior.
