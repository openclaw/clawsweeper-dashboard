## Summary

Keep open. Current main only partially satisfies the request: `chat.send` uses the shared configurable attachment cap, but image uploads are still rejected above the separate 6 MiB image cap, the Control UI bootstrap contract does not expose an effective upload policy, and the browser file/paste/drop paths still read attachments before any upload-size prevalidation.

## What This Changes

Request to make Control UI chat upload size configurable so larger supported media, especially modern phone photos, can be uploaded without hitting a hardcoded small limit.

## Best Possible Solution

Define one effective Control UI upload policy, thread the resolved byte cap through Gateway bootstrap, reject oversized paste/file/drop inputs before reading data URLs, keep server-side MIME and byte checks authoritative, align the image hydration cap with the accepted upload cap, and document the configured limit plus the practical WebSocket/base64 ceiling.

## Reproduction Assessment

Yes. Configure `agents.defaults.mediaMaxMb` above 6 MiB and upload an image larger than `MAX_IMAGE_BYTES` through Control UI; source and tests show `chat.send` reaches the shared cap path but `parseMessageWithAttachments` rejects the image at the fixed 6 MiB guard.

## Solution Assessment

Unclear because there is no patch yet and maintainers have not chosen the public config shape. The narrowest maintainable direction appears to be reusing the existing shared media cap unless maintainers explicitly want a Control UI-specific override.
