## Summary

Keep open. Current main and v2026.6.8 already replaced the original fixed 5 MB parser cap with `agents.defaults.mediaMaxMb`, but the Control UI still lacks an exposed upload limit/prevalidation path and image attachments above `MAX_IMAGE_BYTES` are still rejected before reaching the agent.

## What This Changes

The issue requests a configurable Control UI upload-size limit so larger media files can be uploaded through web chat without hard caps and late server-side rejection.

## Best Possible Solution

Use the existing `agents.defaults.mediaMaxMb` policy if maintainers accept that owner, expose the resolved effective limit to Control UI, add browser prevalidation before `FileReader`, and handle the image hydration cap explicitly rather than adding a competing config knob without product approval.

## Reproduction Assessment

Yes from source for the remaining gap: current main rejects images above `MAX_IMAGE_BYTES` and the Control UI reads files before size-aware validation. I did not run a live browser upload in this read-only review.

## Solution Assessment

Unclear as a product choice. Reusing `agents.defaults.mediaMaxMb` is the narrowest current config path, but maintainers still need to decide whether Control UI should share that limit or own a separate upload policy.
