## Summary

Keep open. Current main partially addresses the older hardcoded parser cap by routing `chat.send` through the shared `agents.defaults.mediaMaxMb` limit, but the Control UI upload problem remains: image attachments are still capped by a separate 6 MiB image guard, the bootstrap contract does not expose an effective upload limit to the browser, and paste/file/drop paths still read full data URLs before any size prevalidation. The issue discussion by mayank6136 and hclsys usefully narrowed the work to those remaining gaps.

## What This Changes

Request to make Control UI browser uploads use a configurable/effective size limit so larger user media, especially high-resolution phone photos, can be uploaded consistently with OpenClaw media handling.

## Best Possible Solution

Define one effective Control UI upload policy, expose the resolved byte limit and practical transport ceiling through the bootstrap contract, reject oversized paste/file/drop inputs before `FileReader.readAsDataURL`, align image parsing with the accepted image policy, and document the limit with focused Gateway and UI tests.

## Reproduction Assessment

Yes. Configure `agents.defaults.mediaMaxMb` above 6 MiB and upload an image larger than `MAX_IMAGE_BYTES` through the Control UI; current source and tests show `parseMessageWithAttachments` rejects it at the fixed image guard after the shared maxBytes check.

## Solution Assessment

Unclear because there is no patch and the public config shape still needs maintainer choice. The narrowest maintainable path appears to reuse the existing shared media cap unless maintainers explicitly want an independent `gateway.controlUi` override.
