## Label Justifications

- `P1`: Pending exec approvals can leave active agent/channel workflows continuing before the operator decision is known.
- `impact:security`: The issue is about the host exec approval boundary and how guarded commands are scheduled around operator approval.
- `impact:session-state`: The reported failure mode is session/turn state drift while an approval is pending.

## Summary

Keep this open. Current main still treats foreground exec approvals as an immediate `approval-pending` tool result with detached follow-up execution, while the related reports at https://github.com/openclaw/openclaw/issues/69386 and https://github.com/openclaw/openclaw/issues/69570 show the same approval-scheduling family is still active and needs maintainer policy.

## What This Changes

The issue asks for foreground exec approvals to pause the agent turn until the approval decision is known instead of returning a model-visible pending result.

## Best Possible Solution

Define one foreground exec-approval turn-control contract for gateway and node hosts, covering approval, denial, timeout, heartbeat runs, native approval UI, follow-up delivery, and audit logs, then update docs and regression coverage together.

## Reproduction Assessment

Yes, source-level. Configure gateway or node exec to require approval and call a foreground command; current main returns `approval-pending` and resolves the final decision in a detached continuation.

## Solution Assessment

Unclear. Awaiting approval may be right, but current docs and tests intentionally encode the async pending/follow-up model, so maintainers should choose the turn-control contract before implementation.
