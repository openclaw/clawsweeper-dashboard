## Summary

Keep this open. Current main still surfaces foreground gateway and node exec approvals as model-visible `approval-pending` tool results while detached continuations wait for approve, deny, or timeout and emit follow-up events later; the docs and tests also encode that async flow.

## What This Changes

Exec approvals should pause or hard-stop foreground agent turns until the approval is approved, denied, or times out instead of letting the turn continue after an `approval-pending` result.

## Best Possible Solution

Define one core exec-approval turn-control contract for foreground interactive approvals, keep background and headless automation behavior separate, then update gateway/node/shared approval code, docs, and regression tests for denial, timeout, heartbeat, and follow-up delivery paths.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. A source-level reproduction is enough: run a foreground gateway or node exec that requires approval and current main returns `details.status = "approval-pending"` while a detached continuation waits for the final decision.

## Solution Assessment

Is this the best way to solve the issue? Unclear. Awaiting the final decision matches the request, but current main intentionally documents and tests an async pending/follow-up model, so maintainers should choose whether blocking or turn suspension/resume is the safer contract.
