## Label Justifications

- `P1`: Pending exec approvals can let active agent workflows continue with dependent work or more approval requests before the operator decision is known.
- `impact:security`: The issue is about the host exec approval boundary and how unresolved approval decisions gate command execution.
- `impact:session-state`: Returning `approval-pending` as a tool result can let the agent transcript and later actions drift before the command result exists.

## Summary

Keep this open. Current main still returns a model-visible `approval-pending` result while a detached continuation waits for the approval decision, and the open WebChat-only PR does not settle the broader gateway/node/heartbeat turn-control contract.

## What This Changes

Request that foreground exec approval requests pause the agent turn until approval is approved, denied, or timed out.

## Best Possible Solution

Define one foreground exec-approval turn-control contract across gateway and node hosts, then update docs and regression coverage for approval, denial, timeout, heartbeat automation, native approval UI, and follow-up audit behavior.

## Reproduction Assessment

Yes, source-level. Configure gateway or node exec to require approval and call a foreground command; current main registers the approval, returns `details.status = "approval-pending"`, and resolves the approval in a detached continuation.

## Solution Assessment

Unclear. Awaiting approval may be the right UX for foreground turns, but current docs and tests intentionally encode async pending/follow-up behavior, so maintainers should choose the cross-host contract before a fix PR.
