## Summary

Keep this open. Current main still implements foreground exec approval as an immediate model-visible `approval-pending` result plus detached approval-resolution follow-up work, and changing that contract affects security-sensitive host exec scheduling, timeouts, native approval UI, and audit behavior.

## What This Changes

The issue requests that foreground exec approval requests pause the agent turn until the user approves, denies, or the request times out.

## Best Possible Solution

Define a single foreground exec-approval turn-control contract that covers gateway and node hosts, denial, timeout, heartbeat, native approval cards, follow-up delivery, and auditability, then update docs and regression coverage to match.

## Reproduction Assessment

Yes, source-level. Configure gateway or node exec to require approval and call a foreground command; current main registers the approval, returns `details.status = "approval-pending"`, and handles the final decision in a detached continuation.

## Solution Assessment

Unclear. Awaiting approval may address the reported flow, but current docs and tests intentionally encode the async pending/follow-up model, so maintainers should choose whether the new contract blocks the tool call, hard-stops and resumes the turn, or keeps async delivery with scheduler safeguards.
