## Label Justifications

- `P1`: The current approval-pending flow can break active agent/channel workflows by letting turns continue or spawn follow-up work before the approval outcome is known.
- `impact:session-state`: The issue is about agent turn state and transcript/tool-result ordering drifting while an exec approval remains unresolved.
- `impact:security`: Exec approval is a host command-execution boundary, and changing pending/deny/timeout semantics affects that security-sensitive workflow.

## Summary

Keep open. Current main still implements foreground exec approval as an immediate model-visible `approval-pending` result with detached decision follow-up work, and the open WebChat-only inline-output PR does not cover the broader gateway/node, Telegram, heartbeat, denial, timeout, and audit contract requested here.

## What This Changes

The issue requests that foreground exec approval requests hard-pause the current agent turn until allow, deny, or timeout instead of returning `approval-pending` and letting the turn continue.

## Best Possible Solution

Define one foreground exec-approval turn-control contract across gateway and node hosts, WebChat and chat channels, heartbeat/direct modes, denial, timeout, native approval UI, and audit delivery; then land the WebChat inline-output slice only if it matches that policy.

## Reproduction Assessment

Yes, at source level. Configure gateway or node exec to require approval and call a foreground command; current main returns `details.status = "approval-pending"` while a detached continuation waits for the approval decision.

## Solution Assessment

Unclear. Awaiting approval inline may be the right direction, but current docs/tests intentionally encode the async pending/follow-up contract and open PR https://github.com/openclaw/openclaw/pull/82019 covers only the WebChat gateway slice.
