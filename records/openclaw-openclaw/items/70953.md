## Label Justifications

- `P1`: The issue affects real agent/channel exec approval workflows where the agent can continue without the user's command decision or output.
- `impact:session-state`: The report is about agent-turn state drifting when `approval-pending` is treated as a normal tool result.
- `impact:security`: The behavior sits on the exec approval boundary and affects how user decisions gate command execution.

## Summary

Keep open. Current main has a useful partial fix for WebChat and native chat gateway approvals, but source, tests, and docs still preserve immediate `approval-pending` behavior for explicit follow-up modes, non-native/headless gateway paths, and node-host approvals, so the central approval-scheduling problem remains.

## What This Changes

The issue requests that foreground exec approvals block the active agent turn until approve, deny, or timeout instead of returning `approval-pending` and allowing the turn to continue.

## Best Possible Solution

Keep this open as the canonical approval-scheduling item and design a security-reviewed fix that blocks foreground gateway and node approvals consistently while preserving explicit async, background, headless, unavailable-route, and timeout semantics.

## Reproduction Assessment

Yes from source inspection for the remaining paths: current code and tests still return `approval-pending` for node-host approvals, non-native gateway channels, and explicit follow-up modes. I did not run a live E2E repro in this read-only review.

## Solution Assessment

No, current main is only a partial solution. The best fix is not a blanket wait everywhere; maintainers need to define the safe foreground, async, headless, and timeout boundaries before changing shared approval scheduling.
