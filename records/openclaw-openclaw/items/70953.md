## Label Justifications

- `P1`: Foreground exec approval scheduling can leave real agent/channel workflows proceeding without the command decision or output.
- `impact:session-state`: The issue is about agent-turn state drifting when an approval-pending tool result is treated as a normal result.
- `impact:security`: The behavior sits on the exec approval boundary and affects how user decisions gate command execution.

## Summary

Keep open. Current main only waits inline for a narrow internal/WebChat gateway route; non-inline gateway approval flows and node-host approval flows still return `approval-pending` and resolve later through detached follow-up work, so the central user problem remains.

## What This Changes

The issue requests that foreground exec approval requests block the agent turn until the user approves, denies, or times out the approval.

## Best Possible Solution

Keep this issue open as the canonical maintainer-owned approval scheduling item, and design a bounded change that preserves headless/background behavior while blocking foreground interactive exec approvals consistently.

## Reproduction Assessment

Yes. Current source and tests still show non-inline gateway and node approval paths returning `approval-pending` while resolving the decision asynchronously.

## Solution Assessment

No. Current main contains a useful WebChat/internal inline mitigation, but it is not the full requested foreground approval pause semantics across gateway and node approval routes.
