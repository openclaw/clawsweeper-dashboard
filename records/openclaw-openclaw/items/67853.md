## Summary

Keep this PR open. The underlying memory-wiki fallback work is still useful because current main still returns no supplement results when the wiki corpus registration is missing, but the PR is not merge-ready: its tool-registry change can temporarily drop all plugin tools during an equivalent in-flight load, and the branch is currently dirty against main.

## What This Changes

The PR adds direct memory-wiki fallback paths for memory-core tools, changes plugin tool registry loading, exports a memory-wiki API subpath, preserves memory capability during snapshot loads, and adds memory/plugin regression coverage plus a local memory check script.

## Best Possible Solution

Rebase the PR onto current main, keep the memory-wiki fallback and package-boundary approach, and adjust the in-flight plugin-tool path so a compatible active registry remains available while a fresh non-activating snapshot load is already running.

## Reproduction Assessment

Yes. The reproduction path is a focused code/test check: PR head returns undefined from resolvePluginToolRegistry when isPluginRegistryLoadInFlight is true, and its test expects resolvePluginTools to return an empty list in that case.

## Solution Assessment

No. Fresh non-activating snapshot loads are a reasonable direction for stale tool closures, but dropping compatible active plugin tools during concurrent equivalent loads is not the narrowest maintainable solution; reuse the compatible active registry before degrading.
