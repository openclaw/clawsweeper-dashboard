## Summary

Keep open: the useful wiki fallback is still not present on current main, but this PR is not merge-ready because PR head can drop plugin tools during in-flight registry loads and still lacks required real behavior proof.

## What This Changes

The PR adds memory-wiki fallback paths for shared memory tools, changes plugin tool registry loading, exports a memory-wiki API subpath, and adds memory/plugin regression checks.

## Best Possible Solution

Rebase onto current main, preserve the channel/active/standalone registry fallback order, keep the wiki fallback behind the public memory-wiki API, and add redacted real runtime proof.

## Reproduction Assessment

Yes, at source level. Current main still has a supplement-only shared wiki path, and PR head returns no plugin registry during an in-flight load before trying compatible active registry reuse; I did not run a live runtime repro in this read-only review.

## Solution Assessment

No. The direct wiki fallback direction is useful, but the resolver change should reuse a compatible loaded registry before degrading on in-flight loads, and the new local gate should use the repo test runner.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Reuse active tools before in-flight fallback:** `src/plugins/tools.ts:66-67`
  - body: When an equivalent plugin load is already in flight, this branch returns `undefined` before consulting any loaded compatible registry. `resolvePluginTools` then returns no plugin tools for that request, so concurrent gateway/tool-server traffic can temporarily lose plugin tools even when a safe active registry exists; check compatible loaded registries first, then degrade only if none exists.
  - confidence: 0.92
- **[P3] Use the repo test runner in the local gate:** `scripts/pre-pr-memory-check.sh:31`
  - body: The new local gate calls `pnpm vitest run`, bypassing the repo's `pnpm test` / `scripts/run-vitest.mjs` routing that applies OpenClaw's Vitest config and cache behavior. Use `pnpm test <targets>` or the runner wrapper so this script proves the same lanes maintainers rely on.
  - confidence: 0.82
