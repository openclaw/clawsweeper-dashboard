## Summary

Keep this PR open. Current main has not implemented the memory-wiki fallback work, and the branch contains useful follow-up from review discussion, but the PR head still introduces a blocking plugin-tool availability regression during in-flight snapshot loads.

## What This Changes

The branch adds direct memory-wiki fallback paths for memory-core tools, changes plugin tool resolution to fresh non-activating snapshot loads, exposes `openclaw/extensions/memory-wiki/api`, preserves memory capability state during snapshot loads, and adds QMD, memory-wiki, plugin alias, and local pre-PR regression coverage.

## Best Possible Solution

Land a rebased version that keeps the useful memory-wiki fallback and contract coverage, preserves current main's live memory config behavior, and changes the in-flight plugin-tool path to fall back to a compatible active registry before returning no tools.

## Reproduction Assessment

Yes. The PR head's mocked in-flight test exercises the relevant branch and expects an empty tool list, while current main shows the compatible active-registry check happens before the in-flight guard.

## Solution Assessment

No. Fresh non-activating snapshot loads are a reasonable direction for stale tool closures, but dropping all plugin tools during concurrent equivalent loads is not the narrowest maintainable behavior; reuse a compatible active registry when one exists.
