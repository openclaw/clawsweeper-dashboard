## Summary

Keep open: this is a useful Linux/systemd gateway restart fix, but the current PR is not merge-ready because final-head real systemd proof is missing, the branch is conflicting, and the patch still uses shared-temp marker IPC while dropping current restart-intent payload semantics.

## What This Changes

This PR changes Linux systemd gateway self-restarts to schedule detached `systemd-run --user` handoffs, classify expected SIGTERM as a supervised restart, and adds daemon/run-loop tests plus a changelog entry.

## Best Possible Solution

Rebase onto current main, reuse or extend the protected state-dir restart-intent contract for the systemd expectation, preserve payload-based SIGTERM restart semantics, and attach redacted live user-systemd proof before merge.

## Reproduction Assessment

Yes, at source level. Current main synchronously calls `systemctl --user restart` while generated units use `KillMode=control-group`, and the local systemd man pages confirm restart stop/start plus control-group killing semantics; I did not run a live user-systemd repro.

## Solution Assessment

No, not yet. The detached handoff direction is appropriate, but the PR should use the protected restart-intent/state-dir contract, preserve current SIGTERM payload options, and provide final-head live systemd proof.

## Review Findings

Full review comments:

- **[P2] Store restart markers outside shared temp:** `src/daemon/systemd.ts:84-88`
  - body: The marker path is derived from the unit name under `TMPDIR`/`os.tmpdir()`, and the consumer trusts a plain readable file after a best-effort unlink. A local process can pre-create or spoof this path so a later unrelated SIGTERM is treated as a restart; use the OpenClaw state dir with atomic `0600` writes and file integrity checks, or reuse the restart-intent contract.
  - confidence: 0.86

When rebasing this branch, keep the payload semantics so update and forced restarts do not silently lose their configured drain behavior.
  - confidence: 0.84
