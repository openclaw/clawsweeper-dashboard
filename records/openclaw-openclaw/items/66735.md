## Summary

Keep open: this PR targets a real current-main Linux/systemd restart bug, but it is not merge-ready because the restart marker uses a predictable shared-tmp path, the branch conflicts with current main, and final-head real systemd proof is missing.

## What This Changes

The branch adds Linux/systemd self-restart detection, a detached `systemd-run --user` handoff, SIGTERM restart-marker handling, regression tests, and a changelog entry.

## Best Possible Solution

Rebase onto current main, move the marker/intent contract to a user-owned OpenClaw state location or nonce-protected atomic `0600` file, preserve current restart-intent drain controls, and land with focused tests plus live systemd proof for external and in-unit restart paths.

## Reproduction Assessment

Yes, at source level. Current main restarts the same `KillMode=control-group` systemd unit synchronously, so a caller inside that unit can be stopped with the service cgroup during restart.

## Solution Assessment

No, not yet. The detached handoff is the right implementation direction, but the marker IPC needs hardening and the stale branch needs rebase plus final-head live proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Store restart markers outside shared tmp:** `src/daemon/systemd.ts:84-88`
  - body: The expectation marker is a predictable unit-derived filename under `TMPDIR`/`os.tmpdir()`, while the handoff ignores marker write failures and the SIGTERM path trusts any readable matching file. A stale or pre-created tmp file can block the real marker or make an unrelated SIGTERM take restart semantics, so store it in a user-owned OpenClaw state location with atomic `0600` writes and ownership/link checks, or use an equivalent nonce-protected contract.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The patch introduces command/restart marker IPC that currently trusts a predictable shared-tmp file.

Concerns:

- **[low] Predictable tmp restart marker can be spoofed:** `src/daemon/systemd.ts:84`
  - body: The marker name is unit-derived under shared tmp, and the consumer does not check owner, mode, or link count before honoring it. A local process could create or block the marker and alter how the gateway handles a later SIGTERM.
  - confidence: 0.82
