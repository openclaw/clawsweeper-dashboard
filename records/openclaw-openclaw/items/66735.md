## Summary

Keep open: the underlying Linux/systemd bug is real and current main still lacks a self-restart handoff, but this PR is not merge-ready because the restart marker IPC is spoofable under shared tmp, the branch conflicts with newer restart lifecycle code, and final-head real systemd proof is missing.

## What This Changes

The PR adds Linux/systemd self-restart detection, a detached `systemd-run --user` handoff, SIGTERM restart-marker handling, regression tests, and a changelog entry.

## Best Possible Solution

Move the restart expectation contract to a user-owned OpenClaw state location or nonce-protected atomic `0600` file, rebase onto current restart lifecycle code, and land with focused tests plus final-head systemd proof.

## Reproduction Assessment

Yes, at source level: current main restarts the same `KillMode=control-group` unit synchronously, and systemd's contract kills all remaining processes in that unit cgroup on stop. I did not run a live systemd reproduction in this read-only review.

## Solution Assessment

No, not yet. The detached handoff direction is appropriate, but the marker IPC needs hardening and the branch must be rebased onto current restart-intent behavior before it is the maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Store restart markers outside shared tmp:** `src/daemon/systemd.ts:84-88`
  - body: The marker path is a predictable unit-derived filename under `TMPDIR`/`os.tmpdir()`, and the handoff writes it with shell redirection while the consumer trusts any readable matching file without ownership, mode, link-count, or nonce checks. A local process can pre-create, block, or spoof the marker so an unrelated `SIGTERM` is handled as a restart; use the OpenClaw state dir with atomic `0600` writes and link checks, or an equivalent nonce-backed intent contract.
  - confidence: 0.86

## Security Review

Status: needs_attention
