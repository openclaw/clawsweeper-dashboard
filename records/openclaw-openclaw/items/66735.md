- `P2`: This is a focused Linux/systemd gateway restart fix with real operator impact, but it is not an emergency and still needs hardening/proof.
- `merge-risk: 🚨 security-boundary`: The PR introduces a predictable shared-temp marker that can alter how the gateway interprets SIGTERM.

## Summary

Keep open: current main still lacks the Linux/systemd self-restart handoff, but this branch is not merge-ready because the marker IPC is spoofable, the stale SIGTERM path must preserve current restart-intent payload handling after rebase, and final-head real systemd proof is still missing.

## What This Changes

The PR changes Linux systemd gateway restarts so in-unit restarts schedule a detached `systemd-run --user` handoff, treat the resulting SIGTERM as a supervised restart, and add daemon/run-loop tests plus a changelog entry.

## Best Possible Solution

Rebase onto current main, reuse or extend the protected state-dir restart-intent contract for the systemd expectation marker, preserve current restart-intent payload semantics, and attach final-head Linux user-systemd smoke proof before merge.

## Reproduction Assessment

Yes, at source level. Current main synchronously runs `systemctl --user restart` for a unit using `KillMode=control-group`, and local systemd man pages confirm restart stop/start plus control-group killing semantics; I did not run a live user-systemd repro.

## Solution Assessment

No, not yet. The detached handoff direction is appropriate, but the current branch should use the protected restart-intent/state-dir contract, preserve current SIGTERM payload options, and provide final-head live systemd proof.
