## Summary

Keep open: current main still lacks the Linux/systemd self-restart handoff, and this PR targets a valid gateway restart bug. The implementation direction looks sound, but the branch needs maintainer rebase/review against newer restart-lifecycle code and final-head live systemd proof before merge.

## What This Changes

The PR adds Linux/systemd in-unit restart detection, a detached `systemd-run --user` restart handoff, systemd restart expectation markers consumed by the gateway SIGTERM path, regression tests, and a changelog entry.

## Best Possible Solution

Rebase this implementation onto current main, preserve the split between synchronous external restarts and detached in-unit systemd restarts, keep marker-based SIGTERM handling compatible with current restart-intent controls, then land with targeted tests, changed-gate validation, and disposable user-systemd smoke for both paths.

## Reproduction Assessment

Yes, at source level. Current main restarts the same `KillMode=control-group` systemd service synchronously from the caller, so invoking the restart from inside that service can put the caller in the cgroup that systemd stops.

## Solution Assessment

Yes as a direction, pending rebase and live proof. A transient `systemd-run --user` handoff is narrower than changing unit kill semantics, but it must be reconciled with current restart-intent and safe-restart code before merge.
