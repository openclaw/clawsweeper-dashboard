## Summary

Keep open: current main still has the Linux/systemd self-restart failure mode and this PR targets the right area, but the branch is not mergeable as-is because the new restart marker is spoofable under shared tmp, final-head real systemd proof is missing, and the PR predates newer restart-intent handling on main.

## What This Changes

The PR adds Linux/systemd in-unit restart detection, a detached `systemd-run --user` restart handoff, SIGTERM marker handling, regression tests, and a changelog entry.

## Best Possible Solution

Land a rebased version that uses the existing user-owned restart-intent style contract, or an equivalent nonce-protected atomic `0600` marker, while preserving current restart drain payload semantics and proving the final head on real user systemd.

## Reproduction Assessment

Yes, source-reproducible: current main calls `systemctl --user restart` synchronously while generated units use `KillMode=control-group`, whose systemd contract kills remaining processes in the unit cgroup on stop. I did not run a live systemd reproduction in this read-only review.

## Solution Assessment

No, not yet. The detached handoff direction is appropriate, but the marker IPC needs hardening and the branch needs a rebase that preserves current restart-intent behavior before it is the narrow maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Store restart markers outside shared tmp:** `src/daemon/systemd.ts:84-88`
  - body: The new marker path is a predictable unit-derived filename under `TMPDIR`/`os.tmpdir()`, and later code writes and trusts it without ownership, mode, link-count, or nonce checks. A local process can pre-create, block, or spoof the marker so an unrelated SIGTERM is treated as a restart; use the existing state-dir restart-intent style contract with atomic `0600` writes and link checks, or an equivalent nonce-backed marker.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The patch introduces a concrete local IPC concern by trusting a predictable shared-tmp restart marker.

Concerns:

- **[low] Predictable tmp restart marker can be spoofed:** `src/daemon/systemd.ts:84`
  - body: The marker name is derived from the systemd unit under shared tmp, and the SIGTERM consumer trusts any readable matching file after parsing only unit and timestamp. Another local process could pre-create or replace that marker and alter stop-versus-restart handling.
  - confidence: 0.84
