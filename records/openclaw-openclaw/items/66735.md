## Summary

Keep open: current main still has the Linux/systemd self-restart hazard this PR targets, and the detached handoff direction is useful. The patch is not merge-ready because the new restart marker is a predictable shared-tmp file and current-head real user-systemd proof is still missing.

## What This Changes

The PR adds Linux/systemd in-unit restart detection, a detached `systemd-run --user` restart handoff, systemd restart expectation markers consumed by the gateway SIGTERM path, regression tests, and a changelog entry.

## Best Possible Solution

Rebase the handoff onto current main, store the restart expectation marker in a user-owned OpenClaw state location with ownership/size checks or an equivalent nonce-protected contract, then land with targeted tests and live user-systemd smoke for both external and in-unit restart paths.

## Reproduction Assessment

Yes, at source level. Current main restarts the same `KillMode=control-group` systemd service synchronously from the caller, so invoking restart from inside that service can place the caller in the cgroup that systemd stops.

## Solution Assessment

No, not yet. The detached handoff is the right direction, but the marker IPC should not trust a predictable shared-tmp file, and the current PR head still needs live proof after the later marker/SIGTERM repairs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Store restart markers outside shared tmp:** `src/daemon/systemd.ts:84-88`
  - body: The expectation marker falls back to a predictable filename in `os.tmpdir()`, while the handoff ignores marker write failures and the SIGTERM path trusts any readable matching file. A stale or pre-created tmp file can block the real marker or make an unrelated SIGTERM take restart semantics, so use the user-owned OpenClaw state dir with atomic `0600` writes/ownership checks or an equivalent nonce-protected marker.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The patch introduces a local command/restart marker path that currently trusts a predictable shared-tmp file.

Concerns:

- **[low] Predictable tmp restart marker can be spoofed:** `src/daemon/systemd.ts:84`
  - body: The marker name is unit-derived under `os.tmpdir()` and the consumer does not check ownership, link count, or mode before honoring it. This can let a local process create or block the marker and alter how the gateway handles a later SIGTERM.
  - confidence: 0.82
