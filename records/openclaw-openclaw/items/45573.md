## Summary

Keep open. Current main still leaves ordinary non-forum Telegram group messages without a last-route update, and the core session recorder still returns without awaiting the inbound metadata write on that no-route path. The linked repair PR #45610 is closed unmerged, and the only relevant post-review change found was a changelog-only edit, not a source or regression-test fix.

## What This Changes

The issue reports Telegram group history collapsing to one short stored session and asks for durable persistence of group conversation sessions across multi-day use and gateway restarts.

## Best Possible Solution

Add focused Telegram/core regression coverage and make ordinary non-forum group inbound session metadata durable before dispatch continues, while preserving one session per regular group, separate forum-topic sessions, and the no-last-route behavior for bound ACP topic routes.

## Reproduction Assessment

Unclear for the full live restart sequence because this read-only pass did not run Telegram or gateway restart tests. The source-level reproduction path is clear: build a regular non-forum Telegram group context, observe updateLastRoute is undefined, then recordInboundSession returns before awaiting metaTask when no route update exists.

## Solution Assessment

Unclear whether the closed PR's updateLastRoute-only approach is the best fix. The safer direction is to add regression tests and either provide a safe regular-group route update or await the no-route metadata path without changing documented group/topic session keys.
