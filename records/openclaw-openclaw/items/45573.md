## Summary

Keep open. Current main still leaves ordinary non-forum Telegram group messages without an `updateLastRoute`, and the core recorder can return on that path before the inbound metadata write is awaited; the linked repair PR is closed unmerged.

## What This Changes

The issue reports that Telegram group chat history is not durably persisted across normal group traffic and gateway restarts.

## Best Possible Solution

Add focused regression coverage and make regular Telegram group inbound session metadata durable while preserving one flat session per regular group and topic isolation only for forum topics.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the source-level path: build a regular non-forum Telegram group context, observe `updateLastRoute` is omitted, and follow `recordInboundSession` returning before `metaTask` is awaited; the full live restart data-loss sequence was not run.

## Solution Assessment

Is this the best way to solve the issue? Yes: the narrow maintainable fix is to make regular-group inbound metadata durable with regression tests, without reusing the closed PR blindly or changing the documented regular-group session-key contract.

## Repair Work Prompt

Fix #45573 as a narrow Telegram/core session persistence bug. Ordinary non-forum Telegram groups must keep the documented flat session key (`agent:<agentId>:telegram:group:<chatId>`) while ensuring inbound session metadata is durably recorded before dispatch/restart-sensitive work can proceed. Do not split regular group reply threads into `:topic:` sessions, and preserve forum topic behavior, configured ACP bindings, DM topic behavior, and `mainDmOwnerPin` semantics. Use #45610 only as source context, not as a patch to apply blindly.
