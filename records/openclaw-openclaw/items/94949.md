## Summary

Keep open: this is a focused candidate for a valid CLI port-preflight bug, but it still needs real behavior proof and should not collapse inconclusive probe results into “not busy” before merge.

## What This Changes

The PR exports `checkPortInUse`, routes `isPortBusy` through that multi-endpoint probe, and adjusts one `gateway --force` helper test mock for repeated host probes.

## Best Possible Solution

Reuse the multi-family probe for `isPortBusy`, but preserve a closed status mapping where `busy` is true, `free` is false, and `unknown` surfaces as an error or diagnostic instead of a false not-busy result.

## Reproduction Assessment

No high-confidence live repro was established in this review. The current-main source path is clear because `isPortBusy` still performs a hostless bind and `forceFreePortAndWait` exits when that helper reports false.

## Solution Assessment

No, not as-is. Reusing `checkPortInUse` is the right owner-boundary direction, but the implementation should preserve non-busy probe failures instead of collapsing `unknown` into false.
