## Summary

Keep open: the PR targets a real current-main `gateway --force` port-detection bug, but it is not merge-ready because it collapses an inconclusive multi-host probe into a false not-busy result and the supplied proof is only unit-test output.

## What This Changes

The PR exports `checkPortInUse`, routes `isPortBusy` through it instead of a hostless `tryListenOnPort`, and updates one gateway force helper test mock for repeated probes.

## Best Possible Solution

Reuse `checkPortInUse` for multi-family detection, but map `busy`, `free`, and `unknown` explicitly and add redacted real-output proof from a live IPv4-only listener.

## Reproduction Assessment

No high-confidence live repro was established in this read-only pass. The source path is clear: current main uses a hostless bind in `isPortBusy`, and `forceFreePortAndWait` exits early when that helper reports false.

## Solution Assessment

No, not as-is. Reusing the existing multi-family checker is the right direction, but the implementation should preserve `unknown` instead of collapsing it into a false not-busy result.
