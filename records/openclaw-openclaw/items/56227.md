## Summary

Keep open: current main and v2026.5.22 still close or hand off the serving Gateway before a replacement is proven healthy, so the requested previous-instance-preserving restart remains unresolved despite several useful safe-restart improvements.

## What This Changes

The issue requests an atomic Gateway restart flow that starts and health-checks a replacement before stopping the existing reachable instance.

## Best Possible Solution

Design a Gateway lifecycle handoff that keeps the old Gateway reachable until a replacement is listening and healthy, with explicit contracts for supervised, unmanaged, update, and in-process restart paths.

## Reproduction Assessment

Yes for a high-confidence source-level reproduction: current main closes the server before replacement handling and only the update path has a narrow child health fallback. I did not run a live failing managed-service restart.

## Solution Assessment

Unclear: the atomic-swap direction fits the user problem, but maintainers need to choose the supervisor, lock, port, update-restart, and default-behavior contract before implementation.
