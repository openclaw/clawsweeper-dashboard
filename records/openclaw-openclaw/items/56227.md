## Summary

Keep open. Current main still closes the active Gateway before replacement startup is proven, and daemon/service restart paths perform preflight or post-restart diagnostics rather than preserving or rolling back to the old listener. The existing stale label and prior ClawSweeper comment do not change the remaining product gap.

## What This Changes

Request that Gateway restarts behave transactionally by keeping the previous instance running until a replacement is healthy and reporting restart failure through an active channel when possible.

## Best Possible Solution

Design a transactional, supervisor-aware Gateway restart model: validate changes before mutation where possible, stage or probe a replacement Gateway, prove it is reachable, transfer endpoint/channel ownership, and stop the old instance only after the replacement is healthy. If startup fails, leave the old Gateway reachable and surface diagnostics through existing CLI/status surfaces and any still-active channel delivery path.

## Reproduction Assessment

Yes. A high-confidence source reproduction is a SIGUSR1 or config-triggered restart where `params.start` fails after `server.close`; current main has no path that restores the prior listener afterward.

## Solution Assessment

Unclear. The no-disconnect product target is the right direction, but the literal start-new-first approach needs maintainer design for same-port binding, process supervisors, persisted config timing, and channel ownership before it is clearly the narrowest maintainable fix.
