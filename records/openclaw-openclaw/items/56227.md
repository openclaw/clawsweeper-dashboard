## Summary

Keep open: current main has safer restart-drain and update-restart handling, but ordinary Gateway restarts still close the serving instance before any replacement health is proven, so the requested old-instance-preserving handoff remains a real lifecycle gap.

## What This Changes

The issue asks Gateway restarts to keep the previous Gateway instance reachable until a replacement instance is started and verified healthy.

## Best Possible Solution

Design a Gateway lifecycle handoff that validates replacement readiness while the current Gateway remains reachable, with explicit behavior for supervised and unmanaged installs plus focused tests and docs.

## Reproduction Assessment

Yes for a source-level reproduction: ordinary restarts close the current server before replacement handling, and the non-update spawned/supervised path exits without replacement health proof. I did not run a live managed-service failure scenario.

## Solution Assessment

Unclear. The requested atomic-swap direction is reasonable, but maintainers need to choose the port, lock, supervisor, update-restart, and default-behavior contract before a safe implementation should be queued.
