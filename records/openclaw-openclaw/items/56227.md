## Summary

Keep open: current main and the latest release still close or hand off the serving Gateway before ordinary restart replacement health is proven, so the requested old-instance-preserving restart remains unresolved and needs lifecycle/product review.

## What This Changes

The issue asks for Gateway restarts to keep the current instance serving until a replacement process is started and health-checked.

## Best Possible Solution

Design a Gateway lifecycle handoff that validates replacement readiness while the current Gateway remains reachable, with explicit behavior for supervised and unmanaged installs plus focused tests and docs.

## Reproduction Assessment

Yes for a source-level reproduction: current main closes the current server before replacement handling, and ordinary full-process restart exits without replacement health proof. I did not run a live managed-service failure scenario.

## Solution Assessment

Unclear. The atomic-swap direction is reasonable, but maintainers need to choose the lock, port, supervisor, update-restart, and default-behavior contract before implementation.
