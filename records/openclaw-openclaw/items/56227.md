## Label Justifications

- `P2`: This is a meaningful gateway availability and setup-safety improvement, but it is not an active emergency outage and needs design before implementation.
- `impact:crash-loop`: The reported failure mode can leave the gateway unavailable after a failed restart, cutting off API and channel access.

## Summary

Keep open: current main and the latest release still provide restart deferral, duplicate-start protection, and post-restart health checks, but they do not implement the requested old-instance-preserving atomic handoff for replacement startup failure.

## What This Changes

Request that gateway restarts start and verify a replacement instance before stopping the current gateway, keeping the current instance available if replacement startup fails.

## Best Possible Solution

Define an explicit gateway supervisor handoff and rollback contract, then implement restart continuity so the current listener stays available until a replacement is healthy or the restart is safely rejected.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: current main closes or restarts the active gateway before replacement health is proven, and existing tests cover failed restart startup after the old server has already been closed; I did not run a live supervisor failure repro.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers choose the lifecycle contract; the atomic-swap goal is reasonable, but the implementation needs owner design across managed and unmanaged restarts rather than a narrow code patch.
