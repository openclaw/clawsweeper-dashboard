## Summary

Keep open. Current main still closes the active Gateway before replacement startup is proven, while managed service restart paths provide post-restart diagnostics rather than preserving the old instance; the requested no-disconnect behavior remains unimplemented and needs maintainer design.

## What This Changes

Request transactional Gateway restarts that start and verify a replacement before stopping the existing instance, preserving contact if the replacement fails.

## Best Possible Solution

Design a supervisor-aware transactional restart model: validate changes before committing where possible, stage or probe a replacement Gateway, prove readiness, transfer endpoint/channel ownership, then stop the old instance only after the replacement is healthy. If replacement startup fails, keep the old Gateway reachable and surface diagnostics through existing CLI/status surfaces and any still-active channel path that can safely deliver them.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a SIGUSR1/config/service restart where the next Gateway startup fails after `server.close`; current main has no path that reattaches or restores the previous listener afterward.

## Solution Assessment

Unclear. The requested no-disconnect outcome is the right product target, but the literal start-new-first approach needs maintainer design for same-port handoff, service supervisors, persisted config timing, and channel ownership before it is clearly the narrowest maintainable fix.
