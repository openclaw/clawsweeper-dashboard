## Summary

Keep open: current main and v2026.6.8 still stop or exit the active gateway before replacement health is proven, so existing mitigations do not satisfy the requested old-instance-preserving handoff.

## What This Changes

The issue asks for gateway restarts to start and health-check a replacement before stopping the previous instance, preserving channel and API reachability when restart startup fails.

## Best Possible Solution

Define a maintainer-approved gateway handoff contract, then implement restart continuity so the current listener stays available until replacement health succeeds or the restart is safely rejected.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: current main closes or exits the active gateway before replacement health is proven, while tests only cover retry after the old server has already been closed.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers choose the lifecycle contract; the atomic handoff goal is reasonable, but the implementation should be designed across managed and unmanaged restart modes rather than patched locally.
