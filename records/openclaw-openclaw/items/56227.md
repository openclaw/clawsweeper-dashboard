## Summary

Keep open. Current main still has active-work restart preflight and update-respawn health checks, but the normal Gateway restart path closes the active server before a replacement is proven healthy and has no rollback to the old listener after a replacement startup failure.

## What This Changes

The issue requests a transactional Gateway restart that keeps the previous instance available until the replacement is healthy.

## Best Possible Solution

Design a supervisor-aware transactional restart model that validates or stages the replacement, proves readiness, transfers endpoint and channel ownership, and rolls back cleanly without dropping the old listener.

## Reproduction Assessment

Yes, at source level. A SIGUSR1/config restart reaches `server.close` before replacement readiness is proven, and a later `params.start` failure leaves `server` null rather than restoring the prior listener.

## Solution Assessment

Unclear. The no-disconnect target is sound, but the literal start-new-first sequence is not obviously the narrowest maintainable fix because same-port binding, lock ownership, service supervisors, and channel handoff all need design.
