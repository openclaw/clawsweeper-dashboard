## Summary

Keep open: current main and v2026.5.12 still close the serving gateway before ordinary restart replacement health is proven. Existing safe-restart work defers active work, and update restarts have a specialized child-health fallback, but the requested old-instance-preserving handoff remains unimplemented and needs lifecycle design.

## What This Changes

The issue asks for gateway restarts to start and verify a replacement before stopping the existing gateway, keeping the old instance alive when replacement startup fails.

## Best Possible Solution

Design a gateway handoff that preserves active-work drain, managed/unmanaged service semantics, update restart freshness, and health checks, then drops the old serving path only after replacement verification or failure notification is possible.

## Reproduction Assessment

No high-confidence live reproduction was run. Source and current regression tests show the ordinary restart path closes or restarts the serving process before replacement health is established, but full proof needs a live managed-service failing-replacement scenario.

## Solution Assessment

Unclear. The atomic-swap direction is reasonable, but the maintainable fix needs lifecycle ownership decisions around ports, locks, supervisors, update restarts, and operator notification before implementation is safe.
