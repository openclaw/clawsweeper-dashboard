## Summary

Keep open: current main and v2026.6.8 still close or exit the active gateway before replacement health is proven, and related rollback/restart work only covers narrower pieces of the requested old-instance-preserving handoff.

## What This Changes

Request that gateway restarts use an atomic handoff: start and health-check the replacement before stopping the current gateway, keeping channels and API access reachable if startup fails.

## Best Possible Solution

Define a maintainer-approved gateway handoff contract, then implement restart continuity so the current listener remains available until replacement health succeeds or the restart is safely rejected.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: current main and v2026.6.8 close or exit the active gateway before replacement health is proven, though no live supervisor repro was run.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers choose the lifecycle contract; the atomic handoff goal is reasonable, but the implementation should be designed across managed, unmanaged, config, and update restart modes.
