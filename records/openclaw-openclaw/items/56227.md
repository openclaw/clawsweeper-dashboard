## Summary

Keep open: current main and v2026.6.6 still use SIGUSR1 or service-manager restart handoff plus post-restart health checks, not a previous-instance-preserving atomic swap. The request is valid reliability work, but it needs maintainer design across gateway supervisor semantics before implementation.

## What This Changes

The issue requests a gateway restart mode that keeps the existing gateway reachable until a replacement instance starts and passes health checks.

## Best Possible Solution

Define and implement an upgrade-safe gateway restart contract that either atomically swaps to a healthy replacement or documents a compatible safer fallback across supervised and unmanaged modes.

## Reproduction Assessment

No live high-confidence reproduction was run. Source and tests show current restart paths schedule SIGUSR1 or service-manager restarts with deferral and post-restart health checks rather than preserving the previous listener until a replacement is healthy.

## Solution Assessment

Unclear until maintainers choose the restart contract. The atomic-swap direction targets the right availability problem, but the best fix needs a design decision across supervisor ownership, port binding, locks, health checks, update handoff, and fallback notification.
