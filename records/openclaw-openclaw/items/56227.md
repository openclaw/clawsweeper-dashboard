## Summary

Keep open: current main still shuts down the serving Gateway before normal restart replacement health is proven. The merged safe-restart work defers active OpenClaw work and update restarts have a narrow child-health fallback, but the requested old-instance-preserving handoff is still a lifecycle/product gap.

## What This Changes

The issue requests an atomic Gateway restart handoff that starts and verifies the replacement before stopping the existing reachable instance.

## Best Possible Solution

Design a supervised and unmanaged Gateway handoff that validates replacement readiness while keeping the current Gateway reachable, and make the default or opt-in behavior explicit in docs and tests.

## Reproduction Assessment

Yes for a source-level reproduction: ordinary restarts close the current server and then spawn or hand off without replacement health proof. I did not run a live managed-service scenario with a deliberately failing replacement process.

## Solution Assessment

Unclear. The atomic-swap direction is reasonable, but the maintainable solution needs lifecycle decisions around ports, locks, supervisors, update restarts, and user notification before code should be queued.
