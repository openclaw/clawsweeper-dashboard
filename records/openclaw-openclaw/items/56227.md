## Summary

Keep open: current main and v2026.5.18 still tear down the serving gateway before ordinary restart replacement health is proven. The merged safe-restart work and update-specific health fallback are adjacent improvements, but they do not implement the requested old-instance-preserving handoff.

## What This Changes

The issue requests a gateway restart handoff that starts and verifies a replacement gateway before stopping the existing serving instance.

## Best Possible Solution

Design a restart handoff that keeps active-work drain, managed/unmanaged service semantics, supervisor behavior, update freshness, and health checks intact, then drops the old serving path only after replacement verification or a recoverable failure notification path exists.

## Reproduction Assessment

No high-confidence live reproduction was run. Source and tests make the current behavior source-reproducible: ordinary restart closes or exits the serving gateway before replacement health is established, but a full repro still needs a live managed-service failing-replacement scenario.

## Solution Assessment

Unclear. The atomic-swap direction matches the user-visible goal, but the maintainable solution needs lifecycle design around ports, locks, supervisors, update restarts, and notification rather than a narrow mechanical patch.
