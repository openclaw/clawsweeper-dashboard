## Summary

Keep open: current main and v2026.6.1 still close or hand off the active Gateway before a replacement is proven healthy for normal restart paths, so the previous-instance-preserving restart request remains unresolved. Recent safe-restart and update-recovery PRs are useful partial mitigations, but they do not provide the requested atomic swap contract.

## What This Changes

The issue asks for gateway restart semantics that start and health-check a replacement before stopping the currently serving gateway, so failed restarts do not disconnect all channels or API access.

## Best Possible Solution

Define a maintainer-approved restart contract that keeps the current Gateway serving until a replacement has passed readiness, with explicit exceptions or fallback behavior for supervisors and update paths that cannot support true atomic handoff.

## Reproduction Assessment

Yes, at source level: trigger a SIGUSR1 or safe restart and make the next Gateway start fail; current code closes the active server before the failing replacement start and then keeps only the process alive. I did not run a live repro in this read-only review.

## Solution Assessment

Unclear for the requested start-new-first design: it is directionally the right availability goal, but the maintainable implementation needs a cross-supervisor port/lock/update design decision. Current main is only a partial mitigation, not the best end state for this issue.
