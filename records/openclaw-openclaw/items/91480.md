Keep open for normal PR review and landing: current main still lacks the user-intent guard, the patch is narrowly aimed at the linked WebChat session-state bug, and I found no blocking correctness findings.

## What This Changes

This PR makes WebChat's shared new-session helper no-op unless called with an explicit user intent token, updates the sidebar/chat/slash `/new` callers, adds regression coverage, and restores real timers in one task-maintenance test.

## Best Possible Solution

Land the narrow helper-boundary fix after routine CI/maintainer review, keep explicit New session and `/new` behavior intact, then close the linked issue after merge.

## Reproduction Assessment

Source-reproducible, not live-reproduced: current main's helper can call `sessions.create` with command hooks, and the gateway maps that to a WebChat `command:new` reset path. The exact reconnect caller behind the reporter's flow was not isolated in this review.

## Solution Assessment

Yes. Gating the shared WebChat helper at explicit user-action boundaries is a narrow maintainable fix; blocking `sessions.create` at the gateway would be too broad because other session creation paths are valid.
