## Summary

Keep open for normal maintainer review; current main still allows the shared WebChat session helper to create a new session without an explicit intent argument, while this PR guards that helper and opts in only the visible New Chat and typed `/new` user-action paths. I found no blocking patch defect or security concern; the only remaining uncertainty is that the supplied proof is a terminal source-level probe rather than a live browser reconnect smoke.

## What This Changes

The branch adds an explicit `{ source: "user" }` intent gate to WebChat session creation, updates New Chat and `/new` callers to pass it, and adds helper regression coverage with minor gateway formatting churn.

## Best Possible Solution

Land this helper-boundary fix after maintainer and CI review, keeping `sessions.create` as the gateway primitive while making WebChat session creation opt-in only at explicit user actions.

## Reproduction Assessment

Yes, at source level: current main permits `createChatSession(state)` to reach `sessions.create` without an intent argument, and the linked issue supplies concrete reconnect logs for the user-visible symptom. I did not run the full live browser reconnect flow in this read-only review.

## Solution Assessment

Yes, this is the best small fix I found: it makes the shared helper fail closed unless a UI action explicitly passes user intent, while preserving the sidebar, chat-view, and typed `/new` entry points. A gateway-level block or config option would be broader and would not address accidental helper reuse as directly.
