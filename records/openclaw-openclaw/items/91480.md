- `P1`: The linked report describes WebChat archiving or switching away from completed conversations without user action, which is an urgent session-state workflow regression for affected users.

## Summary

Keep open: this PR is the active fix path for a valid WebChat session-state bug, and current main does not yet have the user-intent guard. After tracing the UI action boundaries, startup refresh path, session creation helper, and gateway reset-hook behavior, I found no blocking patch finding; the remaining uncertainty is live browser reconnect proof rather than a code defect.

## What This Changes

The PR gates WebChat's shared new-session helper behind an explicit user-intent argument, updates the sidebar/chat-view/slash-command callers, adds helper regression coverage, and includes small formatting-only gateway churn.

## Best Possible Solution

Land a narrow UI-boundary fix that makes automatic WebChat startup/reconnect paths unable to create a new session, while keeping explicit New Chat and typed /new behavior intact through the existing sessions.create contract.

## Reproduction Assessment

Yes for the helper boundary: current main source shows createChatSession can call sessions.create without an intent marker, and the PR proof/test exercise the no-intent and explicit-user paths. The exact full browser reconnect sequence from https://github.com/openclaw/openclaw/issues/89760 was not live-reproduced in this review.

## Solution Assessment

Yes, this is the best narrow fix for the reviewed code path: guard the shared WebChat helper by default and mark only explicit user action boundaries. A broader gateway sessions.create protocol restriction would risk breaking legitimate non-UI session creation paths without evidence that the lower-level RPC is the bug owner.
