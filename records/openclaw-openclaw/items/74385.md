## Summary

Keep open: current main still shows the Terminal Hatch TUI as ready/idle before awaited post-connect initialization finishes, and PR #75120 is open and unmerged with closing syntax for this issue.

## What This Changes

The reporter asks for Terminal Hatch to avoid appearing stuck or ultra slow and to show a loader while startup work is still running.

## Best Possible Solution

The TUI should enter a busy initialization activity after connect and return to idle only after agent/session/history initialization completes, with regression coverage for the hatch auto-message path.

## Reproduction Assessment

Yes for the visible UI-state bug: follow `openclaw onboard` to `Hatch in Terminal`, where current main sets the local TUI to ready before awaited startup work completes. The exact reported duration still needs timing data from the reporter's environment.

## Solution Assessment

Yes, adding an initialization busy state around post-connect `refreshAgents()` and `loadHistory()` is the narrow maintainable fix for the confirmed status problem. If startup remains slow after that, the load duration should be profiled separately.
