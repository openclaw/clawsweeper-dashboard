## What This Changes

The issue reports that Terminal Hatch appears stuck or laggy during onboarding because the TUI shows a ready-looking idle status while post-connect startup work is still running.

## Best Possible Solution

Review and either land or close https://github.com/openclaw/openclaw/pull/93999; once a startup-loader fix merges, this issue can close as implemented.

## Reproduction Assessment

Yes, by source inspection: current `main` sets a ready connection status before awaited post-connect work and the renderer only shows a loader for statuses in `TUI_BUSY_ACTIVITY_STATUSES`, which does not include startup initialization. I did not run an interactive macOS Hatch session in this read-only review.

## Solution Assessment

Yes, the narrow fix direction is to mark the existing `client.onConnected` post-connect initialization window as a busy TUI activity while preserving final ready and startup-failure rendering. The open PR is the current candidate for that path and should be reviewed directly before this issue is closed.
