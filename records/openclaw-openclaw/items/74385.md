## What This Changes

The issue asks for Terminal Hatch to avoid appearing stuck or laggy during post-connect startup, preferably by showing a loader or status while initialization work runs.

## Best Possible Solution

The TUI should expose a busy initialization status around post-connect agent/session/history loading, with regression coverage that preserves existing send, wait, stream, and idle transitions.

## Reproduction Assessment

Yes for the visible UI-state bug: follow `openclaw onboard` to `Hatch in Terminal`, where current main enters the local TUI and marks the connection ready before awaited startup work completes. The precise latency still needs timing data from the reporter's environment.

## Solution Assessment

Yes, the linked PR direction is the narrow maintainable fix for the confirmed status problem. If startup remains slow after the loader fix, the `listAgents()` and `loadHistory()` duration should be profiled separately.
