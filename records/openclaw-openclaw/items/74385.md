## Summary

Keep this open: current main still has the reported Terminal Hatch status gap, and open PR #75120 is already linked with closing syntax for the issue. The issue should stay open until that PR or an equivalent fix lands.

## What This Changes

The issue reports that onboarding Terminal Hatch appears stuck or very slow because the TUI shows `local ready | idle` while startup initialization continues before the first auto-message becomes responsive.

## Best Possible Solution

Land PR #75120 if review confirms it preserves reconnect, send, wait, stream, and idle transitions; otherwise land an equivalent narrow TUI change that marks post-connect initialization as busy and adds regression coverage. Treat deeper history or agent-list latency as a separate follow-up only if users still see excessive startup time after the status fix.

## Reproduction Assessment

Yes for the visible UI-state bug: `openclaw onboard` -> `Hatch in Terminal` launches the local TUI, and current main sets `local ready` before awaited agent refresh/history loading while the busy-state set lacks an initialization state. The exact reported duration still needs reporter-specific timing or profiling data.
