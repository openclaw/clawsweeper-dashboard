## Summary

Keep open: current main still reports `local ready | idle` before awaited TUI initialization work completes, and linked PR #75120 remains open and unmerged with closing syntax for this issue.

## What This Changes

The issue reports that Terminal Hatch during onboarding can feel stuck or very slow because the terminal TUI does not show a loader during startup initialization.

## Best Possible Solution

The TUI should expose a busy initialization status around post-connect agent, session, and history loading, with regression coverage that preserves existing send, wait, stream, and idle transitions.

## Reproduction Assessment

Yes for the visible UI-state bug: follow `openclaw onboard` to `Hatch in Terminal`, where current main enters the local TUI and can show ready/idle before awaited initialization finishes. The precise latency still needs timing data from the reporter's environment.

## Solution Assessment

Yes; reviewing and landing PR #75120, or an equivalent narrow TUI startup busy-state fix, is the maintainable path. If launch remains slow afterward, `listAgents()` and `loadHistory()` should be profiled separately.
