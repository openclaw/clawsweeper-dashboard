## Repair Work Prompt

Fix the Terminal Hatch TUI startup feedback bug from this issue. Current `main` sets the connection to `local ready`/`connected` before awaited post-connect initialization (`refreshAgents()`, remembered-session restore, autocomplete refresh, `loadHistory()`, and optional bootstrap auto-message), while `TUI_BUSY_ACTIVITY_STATUSES` has no initialization status, so users can see `local ready | idle` during startup. Add a narrow initialization busy state through the existing status-loader path, set it only when post-connect initialization is actually taking over an idle/non-busy status, preserve reconnect streaming and existing startup-failure rendering, and avoid changing config, provider routing, or the underlying performance work. Use the closed PR `https://github.com/openclaw/openclaw/pull/75120` only as background; do not copy its startup-error regression. Add focused regression coverage for the busy status and startup handoff, and include release-note context in the PR body or commit message; do not edit `CHANGELOG.md`.
