## Summary

Keep open: current main still shows Terminal Hatch as ready/idle before awaited TUI initialization work finishes, and open PR #75120 is already linked as the implementation candidate.

## What This Changes

The issue reports Terminal Hatch appearing stuck or ultra slow after onboarding and asks for a loader or clearer responsive status during startup.

## Best Possible Solution

Add a startup activity state around post-connect agent/session/history initialization, with TUI status regression coverage; review and land #75120 if it preserves existing send, wait, stream, and idle transitions.

## Reproduction Assessment

Yes for the visible UI-state bug: follow `openclaw onboard` to `Hatch in Terminal`, where current main enters the local TUI and marks the connection ready before awaited startup work completes. The precise latency still needs timing data from the reporter's environment.

## Solution Assessment

Yes, an `initializing` busy state is the narrow maintainable fix for the confirmed status problem. If startup remains slow after that lands, `listAgents()` and `loadHistory()` duration should be profiled separately.
