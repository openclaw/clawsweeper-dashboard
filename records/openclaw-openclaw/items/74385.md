## Summary

Keep open: current main still lacks a Terminal Hatch initialization busy state, and PR #75120 is open, unmerged, and explicitly linked as the implementation candidate.

## What This Changes

The issue asks for Terminal Hatch to avoid apparent hangs or lag by showing a loader/busy state during onboarding-to-TUI startup.

## Best Possible Solution

Land or replace PR #75120 so the TUI shows an initialization busy state during post-connect agent/session/history loading, then profile `listAgents()` or `loadHistory()` separately if startup remains slow.

## Reproduction Assessment

Yes for the visible UI-state bug: follow `openclaw onboard` to `Hatch in Terminal`, where current main marks the local TUI ready before awaited startup work completes. The precise latency still depends on the reporter's environment and history/provider state.

## Solution Assessment

Yes, the linked PR direction is the narrow maintainable fix for the confirmed status problem. It should not try to solve all startup performance in the same change unless profiling identifies a separate bottleneck.
