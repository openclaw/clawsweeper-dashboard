## Summary

Keep open: current main has the bounded Terminal Hatch timeout from #76241, but it still reports `local ready | idle` before awaited TUI initialization finishes, and #75120 remains open as the active fix candidate.

## What This Changes

Terminal Hatch startup should avoid visible lag by showing a loader or busy state while post-connect initialization runs.

## Best Possible Solution

Land #75120 or an equivalent narrow TUI startup busy-state patch, then profile `refreshAgents()` and `loadHistory()` only if Terminal Hatch remains slow afterward.

## Reproduction Assessment

Yes for the visible stale-status path from source: run `openclaw onboard`, choose Hatch in Terminal, and current main sets `local ready` before `refreshAgents()` and `loadHistory()` finish. The precise reported latency still needs timing data from the reporter's environment.

## Solution Assessment

Yes; the #75120 direction is the narrowest maintainable fix for the missing loader because it reuses existing busy-state rendering. The merged timeout bound only mitigates stuck provider runs and does not solve the pre-auto-message idle display.
