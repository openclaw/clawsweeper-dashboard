## What This Changes

The issue reports that Terminal Hatch appears stuck or ultra slow after onboarding prompts because the TUI shows `local ready | idle` instead of a loader while post-connect initialization runs.

## Best Possible Solution

Review and either land or close https://github.com/openclaw/openclaw/pull/93999; once a startup-loader fix merges, close this issue as implemented.

## Reproduction Assessment

Yes, by source inspection: current main marks the TUI connection ready before awaited post-connect work, and the renderer only shows a loader for statuses in the busy-status set, which does not include startup initialization. I did not run an interactive macOS Hatch session in this read-only review.

## Solution Assessment

Yes, the best path is to reuse the existing TUI busy-status renderer for the post-connect initialization window while preserving active-run and failure statuses. The active linked PR is the current candidate for that fix, so opening another fix lane would duplicate the work.
