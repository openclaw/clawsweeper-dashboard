## Summary

Keep open: current main still exposes the reported Hatch terminal path with a misleading ready/idle status while post-connect initialization awaits, and the provided GitHub context shows an open, unmerged linked PR already tracking the narrow fix.

## What This Changes

The issue reports that `openclaw onboard` followed by `Hatch in Terminal` can appear stuck or very slow before the embedded TUI resumes, while the status line shows ready/idle instead of a loader.

## Best Possible Solution

Land PR #75120 if review confirms it preserves existing send, wait, stream, reconnect, and idle transitions; otherwise merge an equivalent narrow TUI change that marks post-connect initialization as busy and adds regression coverage around the Hatch startup path.

## Reproduction Assessment

Yes for the visible UI-state bug: `openclaw onboard` -> `Hatch in Terminal` reaches code that sets `local ready` before awaiting `refreshAgents()` and `loadHistory()`, while the busy loader excludes initialization. The exact duration is environment-dependent and still needs timing data if maintainers want to address underlying slowness beyond the missing loader.

## Solution Assessment

Yes: the linked fix direction is the narrowest maintainable solution because it reuses the existing TUI busy-loader path for post-connect initialization without changing provider, model, gateway, or onboarding behavior. This issue should stay open until that PR or an equivalent patch merges.
