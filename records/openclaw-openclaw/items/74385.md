## Summary

Current main still has the misleading Terminal Hatch startup state: it enters local TUI mode, shows `local ready | idle`, then awaits agent/session/history initialization without a busy status; the open closing PR remains the right path to resolve it.

## What This Changes

The issue reports Terminal Hatch appearing stuck or very slow after onboarding, with no loader while startup initialization runs.

## Best Possible Solution

Review and land PR #75120, or an equivalent narrow TUI startup busy-state patch, while leaving deeper `refreshAgents()` or `loadHistory()` performance work to a separate profiled issue if needed.

## Reproduction Assessment

Yes for the visible UI-state bug: follow `openclaw onboard` to `Hatch in Terminal`, where current main can render `local ready | idle` before awaited post-connect initialization completes. The exact latency still depends on the reporter's local agent/session/history state.

## Solution Assessment

Yes; adding an initialization busy state around the existing post-connect startup awaits is the narrowest maintainable fix for the misleading status, and it does not require changing onboarding or the embedded runtime boundary.
