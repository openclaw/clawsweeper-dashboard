## Summary

Keep open: current main has same-provider auth-profile rotation before model fallback, but it intentionally defaults to one rate-limit profile rotation rather than exhausting all remaining profiles, so the remaining request is a maintainer policy/compatibility decision rather than a cleanup close.

## What This Changes

The issue requests that rate-limit or auth-profile-level failures retry remaining auth profiles for the same provider/model before falling through to the next model provider.

## Best Possible Solution

Keep this as the canonical maintainer decision: either retain the documented one-profile default and close with the existing tuning path, or intentionally change the default/provider policy with focused upgrade, docs, and E2E coverage.

## Reproduction Assessment

Yes for the remaining default-policy behavior: current source and E2E expectations show only one same-provider rate-limit rotation by default before cross-provider fallback. I did not run a live multi-account Codex reproduction.

## Solution Assessment

Unclear. Current main has a deliberate, documented cap plus an override; exhausting all profiles by default may be better for this pool scenario but needs a maintainer compatibility decision.
