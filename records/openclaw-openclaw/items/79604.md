## Summary

Keep open for maintainer policy triage: current main intentionally allows one same-provider rate-limit profile rotation by default and exposes a config knob for more, while this issue asks to exhaust the provider’s profile pool before cross-provider fallback.

## What This Changes

The issue requests changing failover so a rate-limited auth profile rotates through the remaining profiles for the same provider before moving to the next fallback provider.

## Best Possible Solution

Decide whether `usage_limit_reached` should follow the existing configurable cap or whether the default rate-limit policy should change to exhaust the provider pool, then update code, tests, and docs consistently.

## Reproduction Assessment

No high-confidence bug reproduction against the current documented contract. Source and tests do show the requested exhaust-all-profiles behavior is not the default because `rateLimitedProfileRotations` defaults to 1.

## Solution Assessment

Unclear as submitted. A generic profile sub-loop would bypass the existing cap and cooldown policy; the safer path is a maintainer decision on the default/cap or a focused regression if `usage_limit_reached` skips the one allowed rotation.
