## Summary

Keep open: current main and v2026.6.10 implement bounded same-provider auth-profile rotation, but the default still rotates only once before cross-provider fallback, so the requested exhaust-current-provider behavior remains a compatibility-sensitive fallback policy decision.

## What This Changes

The issue asks OpenClaw to keep rotating through remaining auth profiles for the same provider/model on profile-level failures before moving to the next fallback provider.

## Best Possible Solution

Maintainers should choose an explicit same-provider fallback policy, then implement it through the existing rotation-cap path with focused runtime tests and docs that preserve upgrade expectations.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.10 have a three-profile E2E expectation that defaults to two OpenAI attempts before Groq fallback. I did not run a live multi-profile Codex workload in this read-only review.

## Solution Assessment

Unclear until maintainers choose the policy. The suggested sub-loop is plausible, but the current implementation intentionally uses a configurable cap, so the safest solution is a maintainer-approved default or opt-in mode rather than an automatic exhaust-all change.
