## Summary

Keep open for maintainer policy triage: current main already supports same-provider auth-profile rotation and a config knob to exhaust more profiles, but shipped docs and tests intentionally default rate-limit failures to one same-provider retry before model fallback. The remaining question is whether that default should change for Codex/openai-codex, or whether the reporter can prove a narrower `usage_limit_reached` path that skips even the one allowed rotation.

## What This Changes

Request same-provider auth-profile rotation during a failover sequence before moving from the preferred provider to the next fallback provider.

## Best Possible Solution

Decide whether Codex/openai-codex rate-limit failover should keep the configurable capped default or exhaust the provider profile pool by default; if the policy changes, update the prompt-side and assistant-side failover paths, docs, and regression tests together.

## Reproduction Assessment

No high-confidence bug reproduction against the current documented contract. Source and E2E tests do show the reported default behavior: one same-provider rate-limit retry is allowed, then the configured fallback model is used unless `rateLimitedProfileRotations` is raised.

## Solution Assessment

Unclear as submitted. Adding a generic sub-loop in `model-fallback.ts` would bypass the embedded runner's existing cap and cooldown policy; the safer path is a maintainer decision on the default cap, or a focused bug report if `usage_limit_reached` skips the already-allowed rotation.
