## Summary

Keep open for maintainer policy triage: current main supports same-provider auth-profile rotation but intentionally caps rate-limit rotation at one retry by default, while this issue asks to exhaust the provider profile pool before cross-provider fallback.

## What This Changes

Request same-provider auth-profile rotation to continue through available profiles during failover before moving to the next provider/model fallback.

## Best Possible Solution

Decide whether `openai-codex` usage-limit failures should keep the documented configurable cap or default to exhausting the provider profile pool, then align docs and regression tests.

## Reproduction Assessment

No high-confidence reproduction for the exact production claim that only one profile is ever attempted on current main. Source and tests do show the remaining policy gap: the default cap does not exhaust a three-profile provider pool before model fallback.

## Solution Assessment

Unclear as submitted: a generic profile sub-loop would bypass the existing cap semantics. The safer path is a maintainer decision on the default/cap behavior, or a focused regression if `usage_limit_reached` skips the one allowed rotation.
