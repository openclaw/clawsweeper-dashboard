## Summary

Keep open for maintainer decision: current main already supports same-provider auth-profile rotation, but docs and tests intentionally cap rate-limit rotations at one by default, while the issue asks to exhaust the provider pool before cross-provider fallback.

## What This Changes

The issue requests failover to try remaining auth profiles for the same provider/model before falling through to the next provider candidate.

## Best Possible Solution

Decide whether `usage_limit_reached` should follow the existing configurable cap or whether the default rate-limit policy should change to exhaust the provider pool, then update the embedded failover tests and docs accordingly.

## Reproduction Assessment

No high-confidence bug reproduction under the current documented contract. Source and tests do show the requested pool-exhaustion behavior is not the default because `rateLimitedProfileRotations` defaults to 1.

## Solution Assessment

Unclear as submitted: adding a generic profile sub-loop would bypass the existing embedded-runner cap and cooldown policy. A safer path is a maintainer decision on the cap/default, or a focused regression if `usage_limit_reached` skips the one allowed auto rotation.
