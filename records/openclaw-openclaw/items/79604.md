## Summary

Keep open for maintainer policy triage. Current main already supports configurable same-provider auth-profile rotation, but the documented and tested default intentionally caps rate-limit rotation at one retry, while this issue asks whether OpenClaw should exhaust the provider pool before cross-provider fallback.

## What This Changes

The issue requests trying remaining auth profiles for the same provider during one failover sequence before falling through to the next provider.

## Best Possible Solution

Decide whether rate-limit and usage-limit failures should keep the current configurable one-rotation default or auto-exhaust the provider pool, then update failover policy, regression tests, docs, and UI discoverability accordingly.

## Reproduction Assessment

No high-confidence bug reproduction under the current documented contract. Source and tests do show the requested pool-exhaustion behavior is not the default because rateLimitedProfileRotations defaults to 1.

## Solution Assessment

Unclear as submitted: adding a generic profile sub-loop would bypass the existing configurable cap and current docs. The safer path is a maintainer decision on the cap/default, or a narrower regression if usage_limit_reached is skipping even the one allowed rotation.
