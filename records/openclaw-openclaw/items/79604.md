## Summary

Keep open for maintainer policy triage: current main now has same-provider Codex/OpenAI auth-profile rotation and a configurable cap, but the documented default still allows one same-provider retry before model fallback while this issue asks to exhaust the provider pool before falling through.

## What This Changes

Request same-provider auth-profile rotation within a failover candidate before moving to the next provider/model.

## Best Possible Solution

Decide whether Codex usage-limit and rate-limit failures should exhaust ordered profiles by default or continue using auth.cooldowns.rateLimitedProfileRotations, then align the embedded runner, Codex harness tests, and public docs with that contract.

## Reproduction Assessment

No high-confidence bug reproduction against current documented behavior. Source and tests show the default cap leaves a three-profile pool unexhausted, while current docs and Codex harness coverage show at least one same-run auth-profile rotation is expected.

## Solution Assessment

Unclear as submitted: adding a generic sub-loop would bypass the existing cap and current Codex harness auth-forwarding policy. A safer solution is a maintainer decision on the default cap or a focused special case for usage_limit_reached if the one allowed rotation is still skipped.
