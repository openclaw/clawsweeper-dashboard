## Summary

Keep open for maintainer policy triage: current main already rotates same-provider auth profiles, but the shipped and documented default intentionally allows only one rate-limit rotation before model fallback, while this issue asks to exhaust the provider pool first.

## What This Changes

The issue requests changing failover so profile-level failures try remaining auth profiles for the same provider/model before falling through to the next fallback provider.

## Best Possible Solution

Decide whether rate-limit failover should keep the shipped capped default, recommend configuring `auth.cooldowns.rateLimitedProfileRotations` for pool exhaustion, or change the default/provider-specific policy with matching docs and tests.

## Reproduction Assessment

No high-confidence bug reproduction against the current documented contract. Source and regression tests do show the reported default behavior: OpenClaw tries one same-provider rate-limit rotation, then falls back unless `rateLimitedProfileRotations` is raised.

## Solution Assessment

Unclear as submitted. Adding a generic outer `model-fallback.ts` sub-loop would bypass the embedded runner's existing cap/cooldown policy; the safer path is a maintainer decision on the default or a focused bug fix if `usage_limit_reached` skips the one allowed rotation.
