## Summary

Keep open for maintainer policy triage: current main already has same-provider auth-profile rotation, but the documented and tested default intentionally caps rate-limit rotation at one retry, while this issue asks to exhaust more of the provider pool before cross-provider fallback.

## What This Changes

The issue requests changing failover so profile-level failures try additional auth profiles for the same provider/model before moving to the next fallback provider.

## Best Possible Solution

Decide whether usage_limit_reached should keep the configurable cap or whether the default should exhaust the provider pool, then update runtime tests and public docs to match that policy.

## Reproduction Assessment

No high-confidence reproduction of the reported zero-rotation bug is established. Source and tests do show the requested exhaust-all-profiles behavior is not the default because rateLimitedProfileRotations defaults to 1.

## Solution Assessment

Unclear as submitted: adding a generic sub-loop would bypass the existing cap and cooldown policy. A safer path is a maintainer decision on the default/cap semantics, or a targeted bug fix if usage_limit_reached skips the one allowed rotation.
