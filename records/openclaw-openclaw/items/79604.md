## Summary

Keep open for maintainer policy triage: current main supports same-provider auth-profile rotation but intentionally caps rate-limit rotations at one by default, while this issue asks to exhaust the provider pool before cross-provider fallback or prove a narrower Codex `usage_limit_reached` bypass.

## What This Changes

The issue asks OpenClaw to retry remaining auth profiles for the same model-fallback candidate before moving to the next provider.

## Best Possible Solution

Have maintainers decide whether `usage_limit_reached` should keep the existing configurable cap or change the default to exhaust the provider pool, then align failover tests and docs with that policy.

## Reproduction Assessment

Yes for the current default behavior: source, docs, and E2E expectations show rate-limit profile rotation is capped at one retry before model fallback. No for a current-contract bug, because that cap is documented and configurable.

## Solution Assessment

Unclear as submitted: a generic profile sub-loop would bypass the existing cap and cooldown policy. The safer path is a maintainer decision on the cap/default, plus a focused regression test if `usage_limit_reached` skips the one allowed rotation.
