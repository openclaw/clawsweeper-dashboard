## Summary

Keep open for maintainer failover-policy triage: current main rotates same-provider auth profiles, but intentionally caps rate-limit rotations at one by default, while this request asks to exhaust or otherwise prefer the OpenAI Codex profile pool before cross-provider fallback.

## What This Changes

The issue requests that same-provider auth-profile failover continue through remaining healthy profiles for a candidate before switching to the next fallback provider.

## Best Possible Solution

Have maintainers decide whether usage_limit_reached should keep the configurable cap, raise the default, or get provider-specific pool exhaustion, then align failover tests and docs with that policy.

## Reproduction Assessment

Yes for source-level current-main behavior: the default cap of one means a three-profile pool will not be exhausted before fallback, as shown by source, docs, and E2E expectations. The exact zero-rotation OpenAI Codex production path was not live-reproduced.

## Solution Assessment

Unclear as submitted: a generic exhaust-the-pool sub-loop would bypass the current documented cap. The safer path is maintainer policy judgment on the cap/default or a narrower Codex usage_limit_reached rule.
