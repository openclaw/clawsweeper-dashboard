## Summary

Keep open for maintainer policy triage: current main already supports same-provider auth-profile rotation, but intentionally caps rate-limit rotations at `auth.cooldowns.rateLimitedProfileRotations` with default `1`, while this issue asks to exhaust the provider pool before cross-provider fallback.

## What This Changes

Request that profile-level failures such as rate limits or `usage_limit_reached` try remaining auth profiles for the same provider/model before moving to the next fallback provider.

## Best Possible Solution

Decide whether `usage_limit_reached` and rate-limit failures should keep the shipped configurable cap or change the default to exhaust the available profile pool, then align docs and regression tests with that policy.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No as a bug against the current documented contract; source and E2E tests do show the requested exhaust-all-profiles behavior is not the default because `rateLimitedProfileRotations` defaults to `1`.

## Solution Assessment

Is this the best way to solve the issue? Unclear as submitted: a generic profile sub-loop would bypass the existing cap and cooldown policy, so the safer path is a maintainer decision on the cap/default or a focused regression if `usage_limit_reached` skips even the one allowed rotation.
