## Summary

Keep open: current main and v2026.6.8 implement bounded same-provider auth-profile rotation plus `auth.cooldowns.rateLimitedProfileRotations`, but they intentionally do not exhaust every profile by default, so the remaining request is a maintainer fallback-policy decision rather than a cleanup close.

## What This Changes

The issue requests rotating through auth profiles within one fallback candidate before moving to the next provider/model candidate.

## Best Possible Solution

Use this issue as the canonical maintainer decision point: either keep the documented one-profile default and improve discovery of the tuning knob, or intentionally change the default/provider policy with focused docs, release-note context, and E2E coverage.

## Reproduction Assessment

Yes, for the remaining default-policy behavior: current source, docs, latest release, and E2E coverage show only one same-provider rate-limit rotation by default before cross-provider fallback. I did not run a live multi-account Codex reproduction.

## Solution Assessment

Unclear: current main solves the original zero-rotation symptom and exposes a tuning knob, but the requested exhaust-all default is not obviously the best compatibility tradeoff. Maintainers should decide whether default latency should favor same-provider pool utilization or faster cross-provider fallback.
