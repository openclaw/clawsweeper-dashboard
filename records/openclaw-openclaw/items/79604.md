## Summary

Keep open for maintainer product decision: current main already supports same-provider auth-profile rotation, but intentionally caps rate-limit rotations at one by default and documents/tests that policy, while this issue asks to exhaust the provider pool before cross-provider fallback. The exact production zero-rotation `usage_limit_reached` path was not live-reproduced in this read-only pass, so this should not be closed as implemented or stale.

## What This Changes

The issue asks OpenClaw failover to try additional auth profiles for the same provider/model within one request before falling through to the next configured provider.

## Best Possible Solution

Decide the intended default policy for rate-limit and Codex usage-limit failures, then either keep the documented one-rotation default with clearer operator guidance or change the default/cap semantics with focused embedded-run tests for multi-profile Codex pools.

## Reproduction Assessment

No high-confidence live reproduction of the exact production `usage_limit_reached` zero-rotation path was established. Source and regression tests do give a clear current-main reproduction of the requested default-policy gap: with three profiles, default rate-limit handling tries the initial profile plus one rotation, then model fallback, not the whole pool.

## Solution Assessment

Unclear as submitted. A generic profile sub-loop is not the safest fix because current main already has a documented cap and cooldown contract; the narrow path is a maintainer policy decision plus targeted tests for the exact Codex usage-limit classification and desired default.
