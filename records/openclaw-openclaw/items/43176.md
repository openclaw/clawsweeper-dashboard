## Summary

Keep open: current main still has no broad-find timeout guard, but this PR is not merge-ready because PR head misses common broad-scan forms and the external contributor has not supplied real behavior proof.

## What This Changes

The PR adds a 20s timeout cap for broad home/root `find` exec commands, tests for the guard, and system-prompt guidance to prefer scoped workspace searches.

## Best Possible Solution

Keep the narrow guard direction if maintainers accept the policy, but fix the parser edge cases, add coverage for them, and require redacted real terminal or log proof from a capped exec run.

## Reproduction Assessment

Yes at source level: current main lacks a `find`-specific timeout cap, and PR head can be traced to miss `find .` from home, multi-root `find`, and `${HOME}` forms. I did not run a live broad home/root scan in this read-only review.

## Solution Assessment

No. A targeted broad-find cap is plausible, but this implementation is not yet the narrow maintainable solution because common shell forms bypass it and the runtime policy still needs maintainer acceptance plus real proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Cap home-rooted relative finds before workspace exemption:** `src/agents/bash-tools.exec.ts:355-359`
  - body: When `workdir` is the user's home directory, `find . ...` resolves to the workdir and returns uncapped before `isBroadRoot` is checked. That misses the broad home scan this guard is meant to limit; classify home/root before exempting workspace-scoped paths, or avoid exempting home.
  - confidence: 0.91
- **[P2] Inspect every find start path:** `src/agents/bash-tools.exec.ts:344-345`
  - body: `find` accepts multiple start paths, but this parser keeps only the first path-like token. From a workspace, `find . / -type f` records `.` and returns uncapped as workspace-scoped while still traversing `/`; collect all start paths and cap when any resolved root is broad.
  - confidence: 0.9
- **[P2] Resolve ${HOME} as a home root:** `src/agents/bash-tools.exec.ts:296-304`
  - body: The resolver handles `$HOME` and `$HOME/...` but not `${HOME}` or `${HOME}/...`, which the shell expands to the same directory before `find` runs. As written, `find ${HOME} ...` is treated as a literal relative path and bypasses the intended cap.
  - confidence: 0.86
