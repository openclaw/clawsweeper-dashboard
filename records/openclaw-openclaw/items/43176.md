## Summary

Keep open: current main still has no broad-find timeout guard, and this PR remains a plausible but incomplete exec policy change with concrete parser bugs and no contributor real behavior proof.

## What This Changes

The PR adds a 20s exec timeout cap for broad home/root `find` scans without explicit timeout, parser coverage, and system-prompt guidance for scoped discovery.

## Best Possible Solution

If maintainers want this policy, land a focused guard that evaluates every `find` start path, handles common shell home forms, exempts only truly scoped searches, documents the behavior, and includes redacted real exec proof.

## Reproduction Assessment

Yes at source level: current main uses `explicitTimeoutSec ?? defaultTimeoutSec` for exec and passes that into `runExecProcess`, so broad `find $HOME` without `params.timeout` inherits the normal run budget. I did not run a live broad home/root scan in this read-only review.

## Solution Assessment

No, not as currently proposed. A targeted broad-find cap is plausible, but this implementation needs parser fixes, real after-fix proof, and maintainer acceptance of the hardcoded timeout policy.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Inspect every find start path:** `src/agents/bash-tools.exec.ts:344-345`
  - body: `find` accepts multiple start paths, but this parser records only the first path-like token. From a workspace, `find . / -type f` records `.` and returns uncapped as workspace-scoped while still traversing `/`; collect all start paths and cap when any resolved root is broad.
  - confidence: 0.9
- **[P2] Resolve braced HOME roots:** `src/agents/bash-tools.exec.ts:283-289`
  - body: The root checks handle `$HOME` and `$HOME/...` but not `${HOME}` or `${HOME}/...`, which the shell expands to the same directory before `find` runs. As written, `find ${HOME} ...` is treated as a relative path and bypasses the intended cap.
  - confidence: 0.86
- **[P2] Classify home-rooted relative finds before workspace exemption:** `src/agents/bash-tools.exec.ts:355-359`
  - body: When `workdir` is the user's home directory, `find . ...` resolves to the workdir and returns uncapped before the code checks whether the resolved root is home or filesystem root. That misses the home scan this guard is meant to limit.
  - confidence: 0.91
