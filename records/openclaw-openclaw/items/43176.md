## Summary

Keep open: the broad-find guard is still useful and current main does not implement it, but this PR is not mergeable as-is because the parser still misses common broad scan forms and the external contributor has not supplied real behavior proof.

## What This Changes

The PR adds a `find`-specific exec timeout cap, new guard tests, and one system-prompt hint encouraging scoped workspace searches.

## Best Possible Solution

Land a corrected, policy-approved broad-find exec guard that analyzes shell segments, evaluates every start path, handles common home forms, preserves explicit operator controls, and includes focused tests plus redacted real exec proof.

## Reproduction Assessment

Yes at source level: current main routes broad home/root `find` commands through the normal exec timeout path because no command-class cap exists. I did not run a live broad home/root scan in this read-only review.

## Solution Assessment

No, not as currently proposed. A targeted cap is plausible, but this implementation needs parser fixes, real after-fix proof, and maintainer agreement on the timeout policy.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Handle shell-prefixed find commands:** `src/agents/bash-tools.exec.ts:262-264`
  - body: `extractFindTraversalTokens` returns no traversal tokens unless the first shell token is `find`, so common exec commands like `cd repo && find $HOME ...` or `FOO=1 find / ...` bypass the cap and keep the full default timeout. Please inspect shell command segments or otherwise detect `find` after safe prefixes before returning.
  - confidence: 0.84
- **[P2] Inspect every find start path:** `src/agents/bash-tools.exec.ts:344-345`
  - body: `find` accepts multiple start paths, but this parser records only the first path-looking token. From a workspace, `find . / -type f` records `.` and can return uncapped through the workspace exemption while still traversing `/`; collect all start paths and cap when any resolved root is broad.
  - confidence: 0.9
- **[P2] Resolve braced HOME roots:** `src/agents/bash-tools.exec.ts:283-289`
  - body: The resolver handles `$HOME` and `$HOME/...` but not `${HOME}` or `${HOME}/...`, which the shell expands to the same directory before `find` runs. As written, `find ${HOME} ...` is treated as a relative literal and bypasses the intended cap.
  - confidence: 0.86
- **[P2] Classify home-relative roots before workspace exemption:** `src/agents/bash-tools.exec.ts:355-359`
  - body: When `workdir` is the user's home directory, `find . ...` resolves to home and returns uncapped through the workspace exemption before the broad-root check. That misses the home scan this guard is meant to limit.
  - confidence: 0.91
