## Summary

Keep open: current main still has no broad-find timeout cap, and this PR addresses a real source-level timeout gap, but the proposed guard still misses common shell/find forms and the external PR lacks after-fix real behavior proof.

## What This Changes

The PR adds an exec helper to cap broad home/root `find` scans at 20 seconds, focused guard tests, and system-prompt guidance to prefer scoped workspace searches.

## Best Possible Solution

Keep the narrow guard direction only if the parser handles all `find` start paths, home-shell forms, and home-workdir cases, with regression tests and redacted real runtime proof of a capped exec run.

## Reproduction Assessment

Yes at source level. Current main has no `find`-specific timeout cap, and PR head can be inspected to miss `find . /`, `${HOME}`, and home-workdir cases; I did not run a live broad home/root scan in this read-only review.

## Solution Assessment

No. The cap is plausible, but this implementation is not the best mergeable form because common command forms bypass it and the runtime policy plus real proof are still unresolved.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Cap home-rooted relative finds before workspace exemption:** `src/agents/bash-tools.exec.ts:355-359`
  - body: When `workdir` is the user's home directory, `find . ...` resolves to the workdir and returns uncapped before `isBroadRoot` is checked. That misses the broad home scan this guard is intended to limit; classify home/root before exempting workspace-scoped paths, or avoid exempting home itself.
  - confidence: 0.91
- **[P2] Inspect every find start path:** `src/agents/bash-tools.exec.ts:344-345`
  - body: `find` accepts multiple start paths, but this parser records only the first path-like token. From a workspace, `find . / -type f` records `.` and returns uncapped as workspace-scoped while still traversing `/`; collect all start paths and cap when any resolved root is broad.
  - confidence: 0.9
- **[P2] Resolve ${HOME} as a home root:** `src/agents/bash-tools.exec.ts:283-289`
  - body: The resolver handles `$HOME` and `$HOME/...` but not `${HOME}` or `${HOME}/...`, which the shell expands to the same directory before `find` runs. As written, `find ${HOME} ...` is treated as a literal relative path and bypasses the intended cap.
  - confidence: 0.86
