## Summary

Keep open. The change targets a real exec reliability gap and current main has no broad-`find` cap, but the latest patch still misses common broad home/root forms and lacks after-fix real behavior proof, so it should not merge as-is.

## What This Changes

The PR adds a `maybeCapBroadFindTimeoutSec` guard, focused tests, and system-prompt guidance to cap broad home/root `find` scans at 20 seconds when no explicit timeout or traversal escape hatch is present.

## Best Possible Solution

Keep the narrow guard direction, but parse all `find` start paths, treat home/root as broad before workspace exemptions, support `${HOME}` forms, cover those cases in tests, and add redacted terminal or log proof showing the capped exec behavior.

## Reproduction Assessment

Yes at source level: current main applies only explicit/default exec timeouts and has no `find`-specific cap. I did not run a live broad home scan in this read-only review.

## Solution Assessment

No, not yet. The guard is a reasonable direction, but the current parser leaves broad home/root traversals uncapped in common shell forms, so the patch needs fixes before it is the narrow maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Cap relative finds when workdir is home:** `src/agents/bash-tools.exec.ts:355-359`
  - body: The workspace exemption returns before the broad-root check, so `find . ...` or `find $HOME ...` with `workdir` equal to the user's home directory is treated as safe and inherits the normal timeout. Since `find` defaults to the current directory, this misses the broad home scan the PR is meant to cap; check home/root before exempting `resolvedRoot === resolvedWorkdir`, or avoid exempting home.
  - confidence: 0.91
- **[P2] Inspect every find start path:** `src/agents/bash-tools.exec.ts:344-345`
  - body: `find` accepts multiple start paths, but the parser records only the first path-like token. A command such as `find . / -type f` from a workspace sees `.` first, returns uncapped as workspace-scoped, and still traverses `/`; collect all start paths and cap when any resolved root is broad.
  - confidence: 0.88
- **[P2] Resolve ${HOME} as the home directory:** `src/agents/bash-tools.exec.ts:296-304`
  - body: The resolver handles `$HOME` and `$HOME/...` but not `${HOME}` or `${HOME}/...`, which the shell expands to the same home directory before running `find`. As written, `find ${HOME} ...` is resolved as a literal relative path and bypasses the cap.
  - confidence: 0.86
