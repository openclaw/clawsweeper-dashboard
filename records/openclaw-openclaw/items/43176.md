## Summary

Keep open: current main still has no broad-find timeout cap and the idea is useful, but this PR is not merge-ready because it lacks real behavior proof, is currently conflicting, adds parser false negatives, and changes default un-timed exec behavior without a settled compatibility policy.

## What This Changes

The PR adds an agent exec guard that caps broad `$HOME` or `/` `find` scans at 20 seconds when no explicit timeout is set, plus system-prompt guidance and tests.

## Best Possible Solution

A rebased, maintainer-approved exec guard should reuse existing command analysis, evaluate every `find` start root and home-variable form, preserve or explicitly document compatibility behavior, and include real runtime proof.

## Reproduction Assessment

Yes at source level: current main routes local exec through the normal explicit/default timeout path and has no broad-find cap before `runExecProcess`. I did not run a live broad `$HOME` or `/` scan in this read-only review.

## Solution Assessment

No, not as currently proposed. A targeted guard is plausible, but this branch needs a rebase, real behavior proof, parser fixes, and maintainer agreement on the default 20-second compatibility policy.
