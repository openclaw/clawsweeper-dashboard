## Summary

Keep open: current main still lacks a broad-find exec timeout guard, and the related heavy-exec work is complementary, but this PR still has parser false negatives and lacks after-fix real behavior proof.

## What This Changes

The PR adds a 20s timeout cap for broad home/root `find` exec commands without an explicit timeout, tests the guard, and adds system-prompt guidance to prefer scoped workspace searches.

## Best Possible Solution

Keep the narrow guard direction only if maintainers accept the policy, then fix the parser edge cases, add targeted coverage, and require redacted real exec proof from a capped broad scan.

## Reproduction Assessment

Yes at source level: current main lacks a `find`-specific cap, and PR head can be traced to miss `find .` from home, multi-root `find`, and `${HOME}` roots. I did not run a live broad home/root scan in this read-only review.

## Solution Assessment

No, not yet. A targeted broad-find cap is plausible, but this implementation needs parser fixes, coverage for the missed shell forms, maintainer acceptance of the runtime policy, and real after-fix proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Cap home-rooted relative finds before workspace exemption:** `src/agents/bash-tools.exec.ts:355-359`
  - body: When `workdir` is the user's home directory, `find . ...` resolves to the workdir and returns uncapped before broad-root classification. That misses the home scan this guard is meant to limit; classify home/root before the workspace-scoped exemption, or avoid exempting home.
  - confidence: 0.91
- **[P2] Inspect every find start path:** `src/agents/bash-tools.exec.ts:344-345`
  - body: `find` accepts multiple start paths, but this parser keeps only the first path-like token. From a workspace, `find . / -type f` records `.` and returns uncapped as workspace-scoped while still traversing `/`; collect all start paths and cap when any resolved root is broad.
  - confidence: 0.9
- **[P2] Resolve ${HOME} as a home root:** `src/agents/bash-tools.exec.ts:296-304`
  - body: The root checks handle `$HOME` and `$HOME/...` but not `${HOME}` or `${HOME}/...`, which the shell expands to the same directory before `find` runs. As written, `find ${HOME} ...` is treated as a literal relative path and bypasses the intended cap.
  - confidence: 0.86
