## What This Changes

The branch adds a 20-second cap for broad `find` scans over home/root paths without an explicit timeout, parser tests, and system-prompt guidance to prefer scoped workspace discovery.

## Best Possible Solution

Land a rebased guard that evaluates every `find` start path, recognizes common home expansions, classifies home/root before workspace exemptions, preserves explicit operator controls, and includes redacted real exec proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Inspect every find start path:** `src/agents/bash-tools.exec.ts:344-345`
  - body: `find` accepts multiple starting paths, but this parser records only the first path-like token. From a workspace, `find . / -type f` can be exempted through `.` while still traversing `/`; collect all start paths and cap when any resolved root is broad.
  - confidence: 0.9
- **[P2] Handle braced HOME roots:** `src/agents/bash-tools.exec.ts:276-289`
  - body: The resolver handles `$HOME` and `$HOME/...` but not `${HOME}` or `${HOME}/...`, which common shells expand to the same home directory before `find` runs. As written, `find ${HOME} ...` is treated as a relative literal and bypasses the intended cap.
  - confidence: 0.86
- **[P2] Classify broad roots before workspace exemptions:** `src/agents/bash-tools.exec.ts:355-359`
  - body: When `workdir` is the user's home directory, `find . ...` resolves to home and returns uncapped through the workspace exemption before broad-root classification. That misses the home scan this guard is meant to limit.
  - confidence: 0.91
