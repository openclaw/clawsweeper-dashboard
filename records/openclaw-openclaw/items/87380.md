## Summary

Keep this PR open for maintainer review: the implementation is small and appears correct, with sufficient terminal proof, but it adds a public gateway config key that intentionally changes the managed gateway process trust boundary.

## What This Changes

The PR adds `gateway.tls.extraCaCerts`, documents it, and feeds the trimmed config value into `NODE_EXTRA_CA_CERTS` during managed gateway service install-plan generation.

## Best Possible Solution

Land this only after maintainer acceptance that `gateway.tls.extraCaCerts` is the intended durable CA-trust contract for managed gateway installs, with the process-wide trust caveat kept visible in docs.

## Reproduction Assessment

Yes from source, but not from a live before/after run here. Current main regenerates the managed service env from install-plan defaults, so a hand-edited `NODE_EXTRA_CA_CERTS` value is not a durable contract across forced reinstalls.

## Solution Assessment

Yes as an implementation shape if maintainers accept the new public config contract. It is narrower than preserving arbitrary generated service-env edits and reuses the existing startup TLS env path, but the config/security decision is still maintainer-owned.
