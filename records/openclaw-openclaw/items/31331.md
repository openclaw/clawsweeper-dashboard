## Summary

Keep open. Current main still resolves managed sandbox and sandbox-browser workspace mounts in the gateway process namespace and passes those paths unchanged to Docker, while the docs only describe the required DooD host-path parity and the direct fix PRs remain closed unmerged.

## What This Changes

The issue requests that Docker-hosted gateways using Docker-outside-of-Docker sandboxing create `workspaceAccess` bind mounts from host-visible workspace paths instead of gateway-container internal paths.

## Best Possible Solution

Adopt one supported DooD contract end to end: safely remap managed sandbox/browser bind sources to host paths before validation, hashing, and Docker create, or make Docker setup enforce host-path parity with clear diagnostics and regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: run the gateway in Docker with the standard `/home/node/.openclaw` bind mount, enable Docker sandboxing, and managed `workspaceAccess` mounts are resolved inside the gateway container but passed unchanged to the host Docker daemon.

## Solution Assessment

Unclear. Runtime remapping fixes the observed failure, but setup-time host-path parity enforcement is the safer docs-aligned alternative; maintainers should choose the supported contract before implementation.
