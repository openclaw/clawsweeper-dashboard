## Summary

Keep open. Current main and the latest release still do not implement automatic Docker-outside-of-Docker bind-source remapping for sandbox workspace mounts; current docs instead require host-path parity, while this issue is already the canonical report for users whose Docker install cannot make `workspaceAccess` work without dangerous overrides.

## What This Changes

The issue requests fixing Docker-installed Gateway sandbox `workspaceAccess` so sandbox containers mount host-resolvable workspace paths instead of Gateway-container internal paths.

## Best Possible Solution

Maintainers should choose the supported Docker-out-of-Docker workspace contract, then either implement secure host-source remapping/setup migration with regression and live Docker proof, or make the host-path parity contract explicit across Docker install docs and setup so `workspaceAccess` works without dangerous bind overrides.

## Reproduction Assessment

Yes at source level. Current main resolves the workspace path in the Gateway namespace and passes it directly into Docker bind arguments; I did not run a live Docker-out-of-Docker container repro in this read-only review.

## Solution Assessment

Unclear as an implementation choice. Automatic remapping is plausible and has prior PRs, but current docs codify host-path parity, so maintainers should first choose the supported Docker setup/security contract.
