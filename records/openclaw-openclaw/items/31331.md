## Summary

Keep open: current main and the latest release still build automatic Docker sandbox bind sources from the resolved workspace paths without the unmerged host-source remapping attempted in related PRs, and the remaining decision touches sandbox security and the documented Docker-outside-of-Docker path contract.

## What This Changes

The issue reports Docker-install gateway sandboxes using container-internal workspace paths for Docker bind sources, preventing Docker-outside-of-Docker `workspaceAccess` mounts from resolving on the host.

## Best Possible Solution

Maintainers should choose the Docker-install sandbox contract, then either revive/adapt the host-source remap with sandbox-security tests and real DooD proof, or deliberately keep the documented host-path/identical-volume requirement and update setup/docs to make that path reliable.

## Reproduction Assessment

Yes, source-level. Current main still feeds resolved workspace paths directly into automatic Docker bind specs, and this matches the reported Docker-outside-of-Docker namespace mismatch; no live Docker repro was run in this read-only review.

## Solution Assessment

Unclear until maintainer review. A host-source remap is a plausible narrow fix, but the best permanent solution must reconcile sandbox security validation with the current documented DooD host-path and identical-volume-map contract.
