## Summary

Keep open. Current main and v2026.5.22 still build Docker sandbox binds from Gateway-namespace workspace paths, while this remains the canonical Docker install/DooD workspaceAccess failure with multiple user confirmations and closed unmerged remap attempts.

## What This Changes

The issue requests fixing Docker Gateway/DooD sandbox workspaceAccess so sandbox containers bind the host workspace path instead of the Gateway container-internal path.

## Best Possible Solution

Adopt one supported Docker Gateway sandbox contract end to end: safe remap of managed workspace/browser bind sources to host paths, or enforced host-path parity with diagnostics, docs, tests, and real Docker proof.

## Reproduction Assessment

Yes, at source level. Current main resolves workspace paths in the Gateway namespace, passes them into Docker sandbox creation, and formats them directly as Docker bind sources; I did not run a live DooD daemon scenario in this read-only review.

## Solution Assessment

Unclear. Runtime remapping matches the user reports and prior closed fix attempts, but current docs encode host-path parity, so maintainers should choose the intended Docker install contract before implementation.
