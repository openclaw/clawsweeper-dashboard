## Summary

Keep open. Current main still feeds gateway-visible sandbox workspace paths directly into managed Docker bind mounts for both normal sandbox and sandbox-browser containers, while the Docker install maps host storage into `/home/node/.openclaw/...` inside the gateway container. The docs now describe the DooD host-path parity requirement, but the bundled Docker install/setup path does not enforce that contract, and the direct fix PRs are closed unmerged.

## What This Changes

The issue requests fixing Docker-install/Docker-outside-of-Docker sandbox `workspaceAccess` so sandbox containers bind host-resolvable workspace paths instead of gateway-container internal paths.

## Best Possible Solution

Keep this as the canonical Docker-install/DooD sandbox workspace bind bug. The end state should be one explicit supported contract: either remap managed sandbox and sandbox-browser bind sources from gateway-container paths to host paths before validation, hashing, and `docker create`, or make the Docker install/setup flow configure and validate host-path parity up front with clear failure messages. Add regression coverage for `workspaceAccess` `rw`, `ro`, and `none`, plus sandbox-browser creation.
