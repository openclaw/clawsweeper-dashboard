## Label Justifications

- `P1`: Docker-installed gateways with sandboxing can have unusable sandbox workspace access across real user setups and released versions.
- `impact:session-state`: The affected path decides which workspace state is mounted into sandboxed agent sessions, causing the sandbox to see the wrong or empty workspace.
- `impact:security`: The issue and its durable fix touch Docker sandbox bind-source boundaries, and the reported workaround requires dangerous bind override flags.

## Summary

Keep open. Current main and v2026.6.10 still pass Gateway-visible sandbox workspace paths directly into Docker bind creation, while the docs confirm DooD bind sources are evaluated in the host namespace; the direct remapping PRs are closed unmerged and the open related PR only covers a narrower materialized-skills variant with blockers.

## What This Changes

The issue asks OpenClaw to make Docker-installed Gateway sandbox workspaceAccess mounts use host-resolvable workspace paths instead of container-internal paths.

## Best Possible Solution

Keep this canonical issue open until maintainers land or replace a security-reviewed Docker Gateway/DooD bind-source contract shared by workspace, agent, skill, browser, and custom mounts, with docs/setup alignment, focused tests, and live proof for workspaceAccess none, ro, and rw.

## Reproduction Assessment

Yes, at source level. Current main resolves Gateway-visible workspace paths and formats them directly into Docker bind specs, while the docs confirm host Docker evaluates DooD bind sources in the host namespace; I did not run a live Docker/DooD repro.

## Solution Assessment

Unclear. Automatic host-source remapping has plausible prior art in the closed PRs, but the best fix needs a maintainer-approved operator contract and sandbox validation before it is safe to land.
