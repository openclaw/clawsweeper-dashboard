## Summary

Keep open. Current main and the latest release still build managed Docker sandbox workspace binds from Gateway-resolved paths, while the report is the canonical Docker Gateway/Docker-outside-of-Docker `workspaceAccess` bind-mount failure and the earlier remap PRs were closed unmerged.

## What This Changes

The issue reports that Docker-installed Gateway sandboxing with Docker-outside-of-Docker mounts Gateway-internal workspace paths instead of host paths, making `workspaceAccess` unusable.

## Best Possible Solution

Adopt one supported Docker Gateway sandbox contract end to end: either safely remap managed workspace and browser bind sources to host paths, or enforce and document host-path parity with clear diagnostics and regression coverage.

## Reproduction Assessment

Yes, from source inspection: current main resolves sandbox workspace paths in the Gateway namespace and formats those paths directly into Docker `-v` bind sources. I did not run a live Docker daemon scenario in this read-only review.

## Solution Assessment

Unclear. Runtime remapping directly addresses the reports and the closed fix PRs, but current docs now codify host-path parity, so maintainers should choose the intended contract before implementation.
