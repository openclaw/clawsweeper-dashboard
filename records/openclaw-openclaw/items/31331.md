## Summary

Keep open. Current main and the latest release still pass gateway-container workspace paths directly into Docker sandbox bind creation, while the direct host-source remapping PRs were closed unmerged; this remains the canonical Docker Gateway/DooD workspaceAccess bug.

## What This Changes

The issue reports that Docker-installed Gateway sandboxes use container-internal workspace paths for Docker bind mounts, making workspaceAccess unusable with Docker-outside-of-Docker.

## Best Possible Solution

Keep this canonical issue open until maintainers implement a durable Docker Gateway sandbox bind-source contract, with security-reviewed host-source remapping or an equally explicit setup/doctor path.

## Reproduction Assessment

Yes, at source level. Current main resolves gateway-visible workspace paths and feeds them directly into Docker bind creation; I did not run a live Docker Gateway/DooD repro in this read-only review.

## Solution Assessment

Unclear. Automatic host-source remapping was prototyped in closed PRs, but the permanent fix must preserve sandbox security and Docker operator contracts.
