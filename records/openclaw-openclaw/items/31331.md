## Summary

Keep open. Current main and v2026.5.12 still pass Gateway-namespace sandbox workspace paths directly into Docker bind sources, and this remains the canonical Docker Gateway/DooD workspaceAccess bind-mount bug that needs a maintainer contract decision.

## What This Changes

The issue asks OpenClaw Docker Gateway sandboxing to bind the host workspace path into sandbox containers instead of using the Gateway container's internal path.

## Best Possible Solution

Adopt one supported Docker Gateway sandbox contract end to end: either safely remap managed workspace and browser bind sources to host paths, or enforce host-path parity through setup, diagnostics, docs, tests, and real Docker proof.

## Reproduction Assessment

Yes, at source level. Current main resolves workspace paths in the Gateway namespace, passes them into Docker sandbox creation, and formats them directly as Docker bind sources; I did not run a live DooD daemon scenario.

## Solution Assessment

Unclear. Runtime remapping matches the user reports and the closed fix PRs, but current docs now encode host-path parity, so maintainers should choose the intended contract before implementation.
