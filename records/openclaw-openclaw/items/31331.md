- `P1`: Docker-installed gateways with sandboxing can have unusable sandbox workspace access across real user setups and released versions.

## Summary

Keep open. Current main and v2026.6.8 still feed gateway-visible workspace paths directly into Docker sandbox bind creation, while the direct host-source remapping PRs are closed unmerged; this remains the canonical Docker Gateway/DooD workspaceAccess bug.

## What This Changes

The issue reports that Docker-installed Gateways using Docker-outside-of-Docker create sandbox workspaceAccess bind mounts from container-internal paths instead of host-resolvable paths.

## Best Possible Solution

Keep this canonical issue open until maintainers land a security-reviewed Docker Gateway/DooD sandbox bind-source contract, with Docker install/setup docs and live Docker proof for workspaceAccess none, ro, and rw.

## Reproduction Assessment

Yes, at source level. Current main resolves gateway-visible workspace paths and feeds them directly into Docker bind creation; I did not run a live Docker Gateway/DooD repro in this read-only review.

## Solution Assessment

Unclear. Automatic host-source remapping has plausible prior art in closed PRs, but the best fix must preserve sandbox security and define the Docker operator contract deliberately.
