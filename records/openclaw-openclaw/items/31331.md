## Summary

Keep open. Current main still builds Docker sandbox bind sources from Gateway-resolved filesystem paths and passes them to Docker without a host-namespace remap, while the discussion already marks this as the canonical Docker-install/DooD `workspaceAccess` bind-mount failure cluster.

## What This Changes

The issue requests Docker-install sandbox workspace mounts to use host-resolvable bind sources instead of Gateway-container-internal paths when sandbox workspace access is enabled.

## Best Possible Solution

Adopt one supported Docker-out-of-Docker sandbox contract end to end: safely remap managed workspace/browser bind sources to host paths, or enforce host-path parity in Docker setup with clear diagnostics and regression coverage.

## Reproduction Assessment

Yes, from source inspection. Current main resolves workspace paths in the Gateway namespace and uses them directly as Docker bind sources; I did not run a live Docker daemon scenario.

## Solution Assessment

Unclear. Runtime remapping matches the report and prior closed fix attempts, while current docs require host-path parity, so maintainers should choose and enforce one contract before implementation.
