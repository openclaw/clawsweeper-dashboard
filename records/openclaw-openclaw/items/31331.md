## Summary

Keep open. Current main and v2026.5.12 still pass Gateway-resolved sandbox workspace paths directly into Docker bind mounts, while the issue is the canonical Docker Gateway/DooD workspaceAccess bind-mount failure and now needs a maintainer contract decision rather than another speculative fix PR.

## What This Changes

The issue requests Docker Gateway/Docker-outside-of-Docker sandbox workspaceAccess use host-resolvable bind sources instead of gateway-internal paths.

## Best Possible Solution

Adopt one supported Docker Gateway sandbox contract end to end: either safely remap managed workspace and browser bind sources to host paths, or enforce host-path parity through setup, diagnostics, docs, tests, and real Docker proof.

## Reproduction Assessment

Yes, source-level: current main resolves workspace paths in the Gateway namespace, passes them into Docker sandbox creation, and formats them directly as Docker bind sources. I did not run a live DooD daemon scenario in this read-only review.

## Solution Assessment

Unclear. Runtime remapping matches the user reports and the closed fix PRs, but current docs now encode host-path parity, so maintainers should choose the intended contract before implementation.
