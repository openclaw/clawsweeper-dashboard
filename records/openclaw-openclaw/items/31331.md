## Summary

Keep open. Current main still passes Gateway-resolved sandbox workspace paths directly into Docker bind sources, while this is the canonical Docker gateway/DooD workspace-bind failure and the direct remap PRs were closed unmerged.

## What This Changes

The issue reports that Docker-installed Gateway sandboxing uses container-internal workspace paths as host Docker bind sources, breaking Docker-outside-of-Docker workspace access.

## Best Possible Solution

Adopt one supported Docker-out-of-Docker sandbox contract end to end: safely remap managed workspace/browser bind sources to host paths, or enforce host-path parity in Docker setup with clear diagnostics and regression coverage.

## Reproduction Assessment

Yes, from source inspection: current main resolves sandbox workspace paths in the Gateway namespace and formats those paths directly into Docker `-v` bind sources. I did not run a live Docker daemon scenario in this read-only review.

## Solution Assessment

Unclear. Runtime remapping matches the reported failure and the closed fix PRs, but current docs codify host-path parity, so the maintainable fix should follow a maintainer decision on that contract.
