## Summary

Keep open. Current main still has the Docker sandbox create path pass the resolved gateway-visible workspace path straight into Docker bind arguments, while the Docker install path pins the gateway workspace inside the container; the related remap PRs were closed unmerged and this issue remains the canonical cluster parent.

## What This Changes

The issue reports that Docker-hosted Gateway plus Docker-outside-of-Docker sandboxing mounts a gateway-container internal workspace path instead of the host workspace path, making sandbox workspace access unusable.

## Best Possible Solution

Keep this canonical issue open until maintainers choose the Docker gateway sandbox path contract, then implement it across Docker create args, bind-source validation, setup/Compose docs, and Docker install smoke coverage.

## Reproduction Assessment

Yes at source level. Current main resolves workspace paths in the gateway namespace and passes them directly into Docker bind arguments; I did not run a live Docker-out-of-Docker reproduction in this read-only review.

## Solution Assessment

Unclear as an implementation choice. Automatic remapping is a plausible fix, but current docs codify host-path parity and prior remap PRs closed unmerged, so the best fix needs maintainer approval of the path contract and security model first.
