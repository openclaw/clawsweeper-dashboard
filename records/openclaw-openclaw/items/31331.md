## Label Justifications

- `P1`: The issue describes a broken Docker install sandbox workflow with multiple user confirmations across later releases.
- `impact:session-state`: Incorrect bind sources can attach a sandbox to an empty or wrong workspace, causing agent workspace state drift.
- `impact:security`: The affected behavior is Docker sandbox bind-source handling, where any fix must preserve host-path and sandbox boundary validation.

## Summary

Keep open. Current main still formats Docker sandbox workspace mounts from Gateway-namespace paths, and this issue remains the canonical Docker-install/DooD workspaceAccess bind-mount cluster with multiple user confirmations and closed unmerged remap attempts.

## What This Changes

The issue asks OpenClaw to make Docker-install/Docker-outside-of-Docker sandbox workspaceAccess mount the host workspace path instead of a Gateway-container-internal path.

## Best Possible Solution

Adopt one supported Docker Gateway sandbox contract end to end: either safely remap managed workspace and browser bind sources to host paths, or enforce host-path parity through setup, diagnostics, docs, tests, and real Docker proof.

## Reproduction Assessment

Yes, at source level. Current main resolves workspace paths in the Gateway namespace, passes them into Docker sandbox creation, and formats them directly as Docker bind sources; I did not run a live DooD daemon scenario.

## Solution Assessment

Unclear. Runtime remapping matches the user reports and prior closed fix attempts, but current docs encode host-path parity, so maintainers should choose the intended contract before implementation.
