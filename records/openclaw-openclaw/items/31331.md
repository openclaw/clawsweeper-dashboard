## Summary

Keep open. Current main still builds Docker sandbox mounts from Gateway-resolved paths and passes those strings to Docker as bind sources, while the live discussion marks this as the canonical Docker-install/DooD `workspaceAccess` bind-mount cluster and both direct remap PRs are closed unmerged.

## What This Changes

The issue reports that Docker-installed Gateway plus Docker-outside-of-Docker sandboxing uses container-internal workspace paths as Docker bind sources instead of host-resolvable paths.

## Best Possible Solution

Adopt one supported Docker-out-of-Docker sandbox contract end to end: either safely remap managed workspace/browser bind sources to host paths, or enforce host-path parity in Docker setup with clear diagnostics and regression coverage.

## Reproduction Assessment

Yes, from source inspection. Current main resolves workspace paths in the Gateway namespace and formats them directly into Docker `-v` bind sources; I did not run a live Docker daemon scenario.

## Solution Assessment

Unclear. Runtime remapping matches the reported failure and prior closed PRs, but current docs require host-path parity, so maintainers should choose and enforce one contract before implementation.
