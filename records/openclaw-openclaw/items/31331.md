## Summary

Keep open. Current main still resolves managed sandbox workspace mounts in the gateway container namespace and passes those paths directly to Docker, while the current docs describe a host-path parity constraint rather than implementing or enforcing it for the Docker install path.

## What This Changes

The issue asks OpenClaw to make Docker-install sandbox workspaceAccess use host-resolvable bind sources instead of gateway-container paths in Docker-outside-of-Docker setups.

## Best Possible Solution

Adopt one supported DooD contract end to end: either safely remap managed sandbox and browser bind sources to host paths before validation and Docker create, or make Docker setup enforce host-path parity with clear diagnostics and regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence current-main path: Docker setup maps state/workspace into `/home/node`, sandbox context resolves those container paths, and Docker create receives them unchanged as bind sources.

## Solution Assessment

Unclear. Runtime remapping would fix the observed failure, but docs now describe host-path parity as the supported DooD contract; maintainers should choose which contract OpenClaw will enforce before a fix lands.
