## Summary

Keep open. Current main still resolves Docker sandbox workspace paths inside the Gateway process and passes those paths directly to Docker bind sources, while the issue remains the canonical parent for the Docker-install/DooD namespace mismatch cluster and the direct remap PRs were closed unmerged.

## What This Changes

Docker-installed Gateway sandboxes should bind `/workspace` from host-resolvable paths instead of Gateway-container-internal paths when using Docker-outside-of-Docker sandboxing.

## Best Possible Solution

Adopt one supported Docker-out-of-Docker sandbox contract end to end: either safely remap managed workspace/browser bind sources to host paths, or enforce host-path parity in Docker setup with clear diagnostics and regression coverage.

## Reproduction Assessment

Yes from source inspection. Current main resolves sandbox workspace paths in the Gateway process and formats them directly into Docker `-v` bind sources; I did not run a live Docker daemon scenario in this read-only review.

## Solution Assessment

Unclear. Runtime remapping matches the user report and the closed fix attempts, while current docs now describe host-path parity as the supported DooD contract, so maintainers should choose and enforce one path before implementation.
