## Summary

Keep open. Current main and v2026.6.5 still mount the resolved Gateway/container workspace paths directly into Docker sandbox containers, and this issue is the canonical open item for the Docker-outside-of-Docker bind-source mismatch.

## What This Changes

The issue asks Docker Gateway + Docker-outside-of-Docker sandboxing to bind host-resolvable workspace paths for `workspaceAccess` instead of Gateway-container-internal paths.

## Best Possible Solution

Keep this as the canonical issue until maintainers choose and implement a Docker sandbox contract: either enforce/document host-path parity through setup and doctor checks, or add a safe tested host-source remapping path for Docker Gateway deployments.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: Docker sandbox creation uses the resolved Gateway workspace paths as bind sources, and Docker setup/onboarding can persist container-internal workspace paths. I did not run the live Docker Compose repro in this read-only review.

## Solution Assessment

Unclear. The prior remapping PRs show a plausible direction, but the best fix requires a maintainer decision on whether OpenClaw should auto-translate Gateway container mounts or instead harden setup/doctor around explicit host-path parity.
