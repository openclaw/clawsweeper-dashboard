## Summary

Keep this open. Current main still builds managed Docker sandbox and browser bind mounts from gateway-visible paths, while the Docker install maps host storage into `/home/node/.openclaw` inside the gateway container. The docs now describe the DooD host-path parity requirement, but code/setup still neither remaps managed bind sources nor enforces parity, and the direct fix PRs are closed unmerged.

## What This Changes

The issue requests a fix for Docker-install/Docker-outside-of-Docker sandboxes where `workspaceAccess` managed bind sources use container-internal gateway paths instead of host-resolvable paths.

## Best Possible Solution

Adopt one supported DooD contract end to end. Either remap managed sandbox and sandbox-browser bind sources from gateway-container paths to host Docker paths before validation, hashing, and `docker create`, or make the Docker install/setup path enforce host-path parity by default and fail clearly when parity is absent. Add regression coverage for `workspaceAccess` `rw`, `ro`, and `none`, plus browser sandbox creation.

## Reproduction Assessment

Yes. The reproduction path is high-confidence from source inspection: a Docker gateway with config/workspace mounted into `/home/node/.openclaw` will resolve default sandbox paths in the gateway namespace, then pass those paths directly as Docker bind sources for sibling sandbox containers.

## Solution Assessment

Unclear. The current docs-only parity contract is not enough by itself because setup/runtime do not enforce it, while remapping bind sources is also security-sensitive; maintainers should choose the supported contract before implementation.
