## Label Justifications

- `P1`: Multiple released Docker installs are reported unable to provide the configured workspace to sandboxed agents.
- `impact:session-state`: The sandbox receives an empty or incorrect workspace instead of the agent's actual files and state.
- `impact:security`: The current workaround and eventual repair affect which host filesystem sources may cross the sandbox bind boundary.

## Summary

Current main and v2026.6.6 still pass Gateway-visible workspace paths directly to the host Docker daemon, so the canonical Docker-outside-of-Docker sandbox failure remains necessary; the repair needs maintainer agreement on safe self-mount discovery and fail-closed behavior.

## What This Changes

Make Docker-outside-of-Docker sandbox workspace mounts resolve to host-visible source paths instead of Gateway-container paths.

## Best Possible Solution

Add one canonical managed-mount translation from Gateway-container paths to host Docker sources, validate translated roots under the existing sandbox policy, and emit a clear actionable error rather than silently using an unresolved container path.

## Reproduction Assessment

Yes, at source level with high confidence: current main and v2026.6.6 pass Gateway-container workspace paths unchanged to the host Docker daemon, whose bind-source namespace is the host. The issue also includes concrete `docker inspect` output and several independent released-version confirmations.

## Solution Assessment

The prior translation approach targets the correct owner boundary, but it is not merge-ready as-is because robust self-container discovery, root-mount handling, strict source validation, and explicit failure behavior must be settled together.
