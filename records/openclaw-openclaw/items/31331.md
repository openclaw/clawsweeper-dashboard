## Summary

Keep open. Current main still formats gateway-namespace sandbox workspace paths directly as Docker bind sources, while the live issue remains the canonical Docker-install/DooD workspaceAccess cluster with multiple user confirmations and only closed, unmerged remap attempts.

## What This Changes

The issue reports Docker-install/Docker-outside-of-Docker sandbox workspaceAccess using gateway-container paths as Docker bind sources instead of host-resolvable workspace paths.

## Best Possible Solution

Adopt one Docker Gateway sandbox contract end to end: either safely remap managed workspace/browser bind sources to host paths, or enforce host-path parity through setup, diagnostics, docs, tests, and real Docker proof.

## Reproduction Assessment

Yes, at source level. Current main resolves workspace paths in the gateway namespace, passes them into Docker sandbox creation, and formats them directly as Docker bind sources; I did not run a live DooD daemon scenario.

## Solution Assessment

Unclear. Runtime remapping matches the reports and closed fix attempts, but current docs now encode host-path parity, so maintainers should choose the intended contract before code changes.
