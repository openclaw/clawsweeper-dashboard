## Label Justifications

- `P1`: Docker-installed gateways with sandboxing can have unusable sandbox workspace access across real user setups and released versions.
- `impact:session-state`: The affected path decides which workspace state is mounted into sandboxed agent sessions, causing the sandbox to see the wrong or empty workspace.
- `impact:security`: The issue and any durable fix touch Docker sandbox bind-source boundaries, and the reported workaround requires dangerous bind override flags.

## Summary

Keep open. Current main and the latest release still pass Gateway-visible sandbox workspace paths directly into Docker bind creation, while the direct host-source remapping PRs are closed unmerged and the open related PR only targets the narrower materialized-skills variant.

## What This Changes

The issue reports Docker Gateway/DooD sandbox workspaceAccess bind mounts using container-internal paths instead of host-resolvable paths.

## Best Possible Solution

Keep this canonical issue open until maintainers land or replace a security-reviewed Docker Gateway/DooD bind-source contract with docs, setup alignment, focused tests, and live Docker proof for workspaceAccess none, ro, and rw.

## Reproduction Assessment

Yes, at source level. Current main resolves Gateway-visible workspace paths and passes them directly into Docker bind creation, and the docs confirm the host Docker daemon evaluates DooD bind sources in the host namespace; I did not run a live Docker/DooD repro in this read-only review.

## Solution Assessment

Unclear. Automatic host-source remapping has plausible prior art in closed PRs, but the best fix must preserve sandbox security and define the Docker operator contract deliberately before it lands.
