## Summary

Keep open. Current main and v2026.6.1 still pass Gateway-resolved workspace paths directly to Docker `-v`; docs now require host-path parity rather than automatic remapping, and the related remap PRs were closed unmerged, so the canonical Docker sandbox contract remains unresolved.

## What This Changes

The issue reports that Docker-installed Gateway sandboxing with Docker-outside-of-Docker creates workspace bind mounts from container-internal paths instead of host paths, making `workspaceAccess` unusable without dangerous custom bind overrides.

## Best Possible Solution

Keep a canonical maintainer-owned item open until the Docker sandbox contract is settled: either make setup/docs enforce host-path parity end to end, or implement a narrow host-source remapping design with security validation and live Docker proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path for the reported configuration: a Docker Gateway with an internal workspace path passes that internal path into Docker `-v` as the host source for `/workspace`.

## Solution Assessment

Unclear as a product decision. Automatic remapping is plausible, but current docs instead define a host-path parity contract for DooD, so maintainers need to choose the permanent contract before an implementation should land.
