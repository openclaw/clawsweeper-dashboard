## Summary

Keep open. Current main still passes Gateway-resolved Docker sandbox workspace paths directly into managed bind mounts, and this issue remains the canonical cluster for a confirmed Docker-install/DooD namespace mismatch with the two direct fix attempts closed unmerged.

## What This Changes

The issue reports Docker-install sandbox `workspaceAccess` mounts using Gateway-container paths instead of host-resolvable Docker bind sources.

## Best Possible Solution

Adopt one supported Docker-out-of-Docker sandbox contract end to end: either safely revive host-source remapping for managed workspace/browser binds with security validation, or enforce host-path parity during Docker setup with clear diagnostics and regression coverage.

## Reproduction Assessment

Yes from source inspection: current main resolves sandbox workspace paths in the Gateway process and formats them directly into Docker bind sources. I did not run a live Docker daemon scenario in this read-only review.

## Solution Assessment

Unclear. Runtime remapping matches the user report and the closed fix attempts, while current docs describe host-path parity as the supported DooD contract; maintainers should choose and enforce one path.
