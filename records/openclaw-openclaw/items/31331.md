## Label Justifications

- `P1`: The documented Docker install plus sandbox workflow remains unusable for multiple users unless they restructure mounts or enable dangerous overrides.
- `impact:session-state`: The sandbox can receive an empty or incorrect workspace instead of the agent's configured files and working state.
- `impact:security`: The workaround and any automatic fix both cross the Docker host bind-source validation boundary.

## Summary

Current main and v2026.6.5 still rely on identical host/container workspace paths for Docker-outside-of-Docker deployments rather than translating gateway-container paths to Docker-host bind sources. Multiple released-version confirmations and two closed-unmerged fix attempts show the core failure remains, but resolving it requires an explicit decision about the sandbox trust boundary and failure behavior.

## What This Changes

Fix Docker-outside-of-Docker sandbox workspace mounts so `workspaceAccess` uses host-resolvable sources rather than gateway-container-only paths.

## Best Possible Solution

Define and enforce one DooD namespace contract end to end: either securely translate automatic workspace sources through verified gateway mount metadata, or detect non-parity during setup/startup and provide an actionable repair before creating a sandbox.

## Reproduction Assessment

Yes, at source level with high confidence: when the gateway-visible workspace path differs from its host source path, current main passes that path unchanged to the host Docker daemon, which resolves it in the host namespace. A live current-main Docker Compose run was not performed during this read-only review.

## Solution Assessment

No final best implementation is approved yet. Mount-table remapping addresses the observable failure, but maintainers must decide metadata trust, unmapped-path behavior, startup diagnostics, and allowed-root validation first.

## Risks / Open Questions

- Automatic translation would trust Docker mount metadata to authorize host bind roots; malformed, overly broad, or unexpected mappings could weaken sandbox filesystem isolation.
- Keeping path parity as the only supported model leaves Docker deployment easy to misconfigure and can silently create an empty host directory instead of exposing the intended workspace.
- Fallback behavior requires a product decision: retaining an unmapped path recreates the failure, while failing closed could stop previously functioning setups at startup.
