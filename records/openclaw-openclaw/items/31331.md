## Summary

Keep open. Current main still creates managed sandbox mounts from paths resolved in the gateway/container namespace and passes them unchanged to Docker; the issue is canonical with multiple confirmations, while docs now describe host-path parity rather than enforcing or remapping it.

## What This Changes

Docker-install sandbox workspace mounts should use host-resolvable bind sources for `/workspace` instead of gateway-container internal paths.

## Best Possible Solution

Adopt one supported DooD contract end to end: either safely remap managed sandbox and browser bind sources to host paths before validation and Docker create, or make Docker setup enforce host-path parity with clear diagnostics and regression coverage.

## Reproduction Assessment

Yes. Source inspection on current main shows Docker setup/runtime paths resolve inside `/home/node`, sandbox context passes those paths through, and Docker create receives them unchanged as bind sources; no live Docker run was performed in this read-only review.

## Solution Assessment

Unclear. Runtime remapping matches the user report and the two closed fix attempts, while current docs describe host-path parity as the supported DooD contract; maintainers should choose the contract before implementation.
