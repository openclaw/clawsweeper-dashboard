## Label Justifications

- `P1`: The issue describes a broken Docker install sandbox workflow with multiple independent user confirmations across later releases.
- `impact:session-state`: Incorrect bind sources can attach a sandbox to an empty or wrong workspace, causing agent workspace state drift.

## Summary

Keep open. Current main and the latest shipped release still pass Gateway-namespace workspace paths directly into Docker sandbox bind mounts, while the discussion and related closed PRs show this is the canonical Docker-install/DooD workspaceAccess cluster that still needs a maintainer contract decision.

## What This Changes

The issue requests Docker-install/DooD sandbox workspaceAccess mounts to use host-resolvable workspace bind sources instead of Gateway-container paths.

## Best Possible Solution

Adopt one supported Docker Gateway sandbox contract end to end: either safely remap managed workspace and browser bind sources to host paths, or enforce host-path parity through setup, doctor diagnostics, docs, focused tests, and real Docker proof.

## Reproduction Assessment

Yes, at source level. Current main resolves workspace paths in the Gateway namespace, passes them into Docker sandbox creation, and formats them directly as Docker bind sources; I did not run a live DooD daemon scenario.

## Solution Assessment

Unclear. Runtime remapping matches the user reports and prior closed fix attempts, but current docs encode host-path parity, so maintainers should choose the intended contract before implementation.
