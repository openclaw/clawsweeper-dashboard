## Summary

Keep open. Current main and v2026.6.10 still construct Docker sandbox bind sources from Gateway-visible paths, the docs only document a host-path parity workaround, and the related implementation work is either closed unmerged or narrower and still open.

## What This Changes

The issue reports Docker-installed gateways passing container-internal workspace and sandbox paths to the host Docker daemon, making sandbox workspaceAccess mounts unusable in DooD setups.

## Best Possible Solution

Keep this canonical issue open until maintainers land a security-reviewed Docker Gateway/DooD bind-source contract covering workspaceAccess modes, Docker install docs/setup, and live DooD proof.

## Reproduction Assessment

Yes, at source level: current main resolves workspace and sandbox paths inside the Gateway process and passes them directly into Docker bind specs. I did not run a live Docker Gateway/DooD container in this read-only review.

## Solution Assessment

Unclear. Host-source remapping is plausible, but the best solution must preserve sandbox bind validation and settle the Docker operator contract before landing.
