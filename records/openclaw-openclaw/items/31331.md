- `impact:session-state`: Incorrect bind sources can attach the sandbox to an empty or wrong workspace, causing agent workspace state drift.

## Summary

Keep open. Current main and the latest shipped tag still pass Gateway-namespace workspace paths directly into Docker sandbox bind sources, while the issue remains the canonical Docker install/DooD workspaceAccess failure with multiple user confirmations and closed unmerged remap attempts.

## What This Changes

The issue reports that Docker-installed Gateways using Docker-outside-of-Docker sandboxing mount `/workspace` from container-internal paths instead of host-resolvable workspace paths.

## Best Possible Solution

Adopt one supported Docker Gateway sandbox contract end to end: either safely remap managed workspace and browser bind sources to host paths, or enforce host-path parity through setup, diagnostics, docs, tests, and real Docker proof.

## Reproduction Assessment

Yes, at source level. Current main resolves workspace paths in the Gateway namespace, passes them into Docker sandbox creation, and formats them directly as Docker bind sources; I did not run a live Docker-outside-of-Docker daemon scenario in this read-only review.

## Solution Assessment

Unclear. Runtime remapping matches the user reports and prior closed fix attempts, but current docs encode host-path parity, so maintainers should choose the intended Docker install contract before implementation.
