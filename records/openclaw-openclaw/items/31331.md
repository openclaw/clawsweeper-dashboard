## Summary

Keep open. Current main still builds managed Docker sandbox workspace binds from Gateway-resolved paths, while the issue is the canonical Docker gateway/DooD `workspaceAccess` bind failure and the remap fix PRs were closed unmerged.

## What This Changes

The issue reports that Docker-hosted Gateways using Docker-outside-of-Docker sandboxing create workspace bind mounts with container-internal paths instead of host-resolvable paths.

## Best Possible Solution

Adopt one supported Docker gateway sandbox contract end to end: either safely remap managed workspace/browser bind sources to host paths, or enforce/document host-path parity with clear diagnostics and regression coverage.

## Reproduction Assessment

Yes, from source inspection: current main resolves sandbox workspace paths in the Gateway namespace and formats those paths directly into Docker `-v` bind sources. I did not run a live Docker daemon scenario in this read-only review.

## Solution Assessment

Unclear. Runtime remapping matches the reported failure and the closed fix PRs, but current docs codify host-path parity, so the maintainable solution should follow a maintainer decision on that contract.
