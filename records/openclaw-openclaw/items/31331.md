## Summary

Keep open. Current main still carries Gateway-namespace workspace paths into Docker sandbox bind sources, and this issue remains the canonical cluster with multiple later confirmations and unmerged remap attempts.

## What This Changes

The issue reports that Docker Gateway plus Docker-outside-of-Docker sandboxing mounts `/workspace` from container-internal paths instead of host-resolvable workspace paths.

## Best Possible Solution

Adopt one supported Docker Gateway sandbox contract end to end: either safely remap managed workspace and browser bind sources to host paths, or enforce host-path parity through setup, diagnostics, docs, tests, and real Docker proof.

## Reproduction Assessment

Yes, at source level. Current main resolves workspace paths in the Gateway namespace, passes them into Docker sandbox creation, and formats them directly as Docker bind sources; I did not run a live DooD daemon scenario.

## Solution Assessment

Unclear. Runtime remapping matches the user reports and closed fix attempts, but current docs now encode host-path parity, so maintainers should choose the intended contract before implementation.
