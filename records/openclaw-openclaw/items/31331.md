## Summary

Keep this open. Current main still passes gateway-container workspace paths directly to the host Docker daemon for managed sandbox and browser mounts; the docs now describe the DooD host-path parity requirement, but the Docker setup/runtime path does not enforce or remap it, and the direct fix PRs remain closed unmerged.

## What This Changes

The issue asks OpenClaw Docker installs using Docker-outside-of-Docker sandboxing to bind the real host workspace path into `/workspace` instead of the gateway container's internal path.

## Best Possible Solution

Adopt one supported DooD contract end to end: either safely remap managed sandbox/browser bind sources from gateway-container paths to host Docker paths before validation, hashing, and `docker create`, or make Docker setup enforce host-path parity and fail clearly when parity is absent, with regression coverage for `workspaceAccess` `rw`, `ro`, `none`, and browser containers.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: run the gateway in Docker with the standard `/home/node/.openclaw` bind mount, enable Docker sandboxing, and the managed sandbox mount source is resolved in the gateway namespace but passed unchanged to the host Docker daemon.

## Solution Assessment

Unclear. A runtime remap can address the observable failure, but it changes the sandbox bind-source security boundary; setup-time host-path parity enforcement is the safer competing contract and maintainers should choose before implementation.
