## Label Justifications

- `P1`: The issue blocks a real Docker-installed Gateway sandbox workflow for multiple users and affects agent workspace access, but it is not an emergency data-loss or crash-loop report.
- `impact:session-state`: The broken bind source prevents the sandboxed session from seeing or writing the intended agent workspace state.
- `impact:security`: The remaining fix path touches Docker bind mounts, allowed source roots, reserved targets, and user workarounds involving dangerous sandbox flags.

## Summary

Keep open. Current main and the latest release still use the Gateway-resolved workspace path directly as the Docker bind source, while the current docs require host-path parity rather than implementing automatic DooD bind-source remapping; the related remap PRs were closed unmerged and the issue has multiple confirming reports.

## What This Changes

The issue requests that Docker-installed Gateways using Docker-outside-of-Docker sandboxing map `workspaceAccess` mounts to host-resolvable paths instead of container-internal Gateway paths.

## Best Possible Solution

Make the Docker Gateway sandbox contract explicit and safe: either implement tested host-source remapping with strict allowed-root validation, or keep host-path parity as the supported contract and fail early with a clear diagnostic when the Gateway config uses container-internal paths.

## Reproduction Assessment

Yes, by source inspection: Docker sandbox create still uses the Gateway-resolved `workspaceDir` as the Docker bind source, and Docker-installed Gateways pin that path to `/home/node/.openclaw/workspace`. I did not run a live Docker DooD reproduction in this read-only review.

## Solution Assessment

Unclear; automatic host-mount remapping is a plausible fix, but it changes the Docker sandbox security and deployment contract. The best path needs maintainer product/security judgment plus tests and live Docker proof for both fresh installs and upgrades.
