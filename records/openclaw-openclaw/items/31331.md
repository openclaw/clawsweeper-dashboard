## Summary

Keep open. Current main and the latest release still build managed Docker sandbox workspace binds from Gateway-resolved paths, while this is the canonical Docker Gateway/Docker-outside-of-Docker `workspaceAccess` bind-mount failure and the earlier remap PRs were closed unmerged.

## What This Changes

The issue requests that Docker-installed Gateway sandbox `workspaceAccess` mounts use host-resolvable workspace paths instead of Gateway-container internal paths under Docker-outside-of-Docker.

## Best Possible Solution

Adopt one supported Docker Gateway sandbox contract end to end: either safely remap managed workspace and browser bind sources to host paths, or enforce host-path parity through setup, diagnostics, docs, and regression coverage.

## Reproduction Assessment

Yes, from source inspection: current main resolves workspace paths in the Gateway namespace and formats them directly into Docker `-v` bind sources. I did not run a live Docker daemon scenario in this read-only review.

## Solution Assessment

Unclear. Runtime remapping matches the user reports and the closed fix PRs, but current docs and setup now encode host-path parity, so maintainers should choose the intended contract before implementation.
