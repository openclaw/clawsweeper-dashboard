## Summary

Keep open. Current main and v2026.5.20 still pass Gateway-namespace workspace paths directly into Docker sandbox bind sources, while this remains the canonical Docker install/DooD workspaceAccess failure with multiple user confirmations and closed unmerged remap attempts.

## What This Changes

The issue reports Docker install sandbox workspaceAccess binding Gateway-container paths instead of host-resolvable workspace paths in Docker-outside-of-Docker deployments.

## Best Possible Solution

Pick one supported Docker Gateway sandbox contract and make runtime code, setup scripts, docs, diagnostics, tests, and real Docker proof consistent with that contract.

## Reproduction Assessment

Yes, at source level. Current main resolves workspace paths in the Gateway namespace, passes them into Docker sandbox creation, and formats them directly as Docker bind sources; live Docker-outside-of-Docker proof is still needed before landing a fix.

## Solution Assessment

Unclear. Runtime remapping matches the user reports and prior closed fix attempts, but current docs encode host-path parity, so maintainers should choose the intended Docker install contract before implementation.
