## Summary

Keep open. Current `main` and the latest shipped tag still format Gateway-resolved sandbox workspace paths directly into Docker bind sources, while this issue is the canonical Docker gateway/DooD workspace-bind cluster and the direct remap PRs were closed unmerged.

## What This Changes

The issue asks for Docker-install sandbox `workspaceAccess` binds to use host-resolvable workspace paths instead of Gateway-container-internal paths.

## Best Possible Solution

Adopt one supported Docker-out-of-Docker sandbox contract end to end: safely remap managed workspace/browser bind sources to host paths, or enforce host-path parity in Docker setup with clear diagnostics and regression coverage.

## Reproduction Assessment

Yes, from source inspection. Current `main` resolves workspace paths in the Gateway namespace and formats them directly into Docker `-v` bind sources; I did not run a live Docker daemon scenario.

## Solution Assessment

Unclear. Runtime remapping matches the reported failure and prior closed PRs, but current docs require host-path parity, so maintainers should choose and enforce one contract before implementation.
