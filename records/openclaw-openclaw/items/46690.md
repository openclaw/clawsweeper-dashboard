## Summary

Keep this issue open: current main documents and loads only configured skill roots, and it has some adjacent install/audit hardening, but it still does not warn about arbitrary orphan `SKILL.md` directories outside those roots.

## What This Changes

The issue requests canonical-path guardrails or diagnostics for user-created skill directories outside OpenClaw's loaded skill roots.

## Best Possible Solution

Add a bounded, explicit diagnostic or route creation through the existing workspace install/workshop path, while avoiding an automatic broad startup scan unless maintainers accept that UX and privacy tradeoff.

## Reproduction Assessment

Yes, source-reproducible: a valid `SKILL.md` directory outside the documented roots is not loaded by `loadWorkspaceSkillEntries`, and `skills check` only reports discovered entries. I did not run the write path because this is a read-only review.
