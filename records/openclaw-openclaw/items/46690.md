The issue requests guardrails and diagnostics for skill directories with `SKILL.md` files created outside OpenClaw's configured skill roots.

## Best Possible Solution

Define a bounded orphan-skill diagnostic policy, then implement it in `openclaw skills check` or doctor with matching docs and skill-creator guidance while leaving the normal loader rooted in configured skill locations.

Yes. Source inspection shows a valid `SKILL.md` under an unconfigured directory is outside the documented roots and is not reached by the loader, skills status report, doctor status, or workspace skill audit.

Unclear. A broad home-depth scan would catch the reported example, but the maintainable fix needs maintainer policy for scan bounds, privacy redaction, performance, duplicate handling, and scaffold override behavior.
