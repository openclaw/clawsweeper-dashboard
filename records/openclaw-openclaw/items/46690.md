## Summary

Keep open. Current main and the latest release still only load and report configured skill roots, so a valid `SKILL.md` directory elsewhere remains invisible; the remaining decision is the privacy-safe scan scope and diagnostic surface, not whether the gap exists.

## What This Changes

The issue requests warnings or diagnostics for skill-like directories created outside canonical skill roots so they are not silently orphaned.

## Best Possible Solution

Define a bounded orphan-skill diagnostic in `openclaw skills check` or doctor, keep runtime loading limited to supported roots, and steer creation/install flows toward workspace or managed skill directories.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a valid `SKILL.md` directory outside every supported skill root is not loaded or reported by `skills list`, `skills check`, status, or the current workspace audit.

## Solution Assessment

Unclear. The requested warning is useful, but maintainers should first choose the privacy-safe scan roots, depth limits, redaction, duplicate policy, and whether this belongs in `skills check`, doctor, or skill-creation tooling.
