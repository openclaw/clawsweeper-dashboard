## Summary

Keep open. Current main and the latest release still only load and report configured skill roots, so orphan `SKILL.md` directories outside those roots remain invisible; the unresolved work is choosing a privacy-safe diagnostic and creation-path policy.

## What This Changes

The issue asks OpenClaw to guard against valid skill repositories being created or left outside canonical loaded skill paths.

## Best Possible Solution

Define a bounded orphan-skill diagnostic in `openclaw skills check` or doctor, keep runtime loading limited to supported roots, and align skill creation/install guidance with the active workspace or managed skill directories.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a valid `SKILL.md` directory outside every supported skill root is not loaded or reported by `skills list`, `skills check`, status, or the current workspace audit.

## Solution Assessment

Unclear. The requested warning is useful, but maintainers should first choose the privacy-safe scan roots, depth limits, redaction, duplicate policy, and diagnostic surface.
