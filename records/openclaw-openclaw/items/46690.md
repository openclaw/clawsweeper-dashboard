## Summary

Keep open. Current main still has no orphan-skill discovery outside documented/configured skill roots, and the bundled skill creator still accepts arbitrary output paths, so the requested diagnostic remains unimplemented.

## What This Changes

The issue asks OpenClaw to warn about valid `SKILL.md` directories created outside canonical skill roots and to steer scaffolding toward workspace skills.

## Best Possible Solution

Add a bounded orphan-skill diagnostic, likely in `skills check` and/or doctor, that compares approved candidate `SKILL.md` scan roots against documented load roots; keep implicit startup scanning opt-in unless maintainers approve it.

## Reproduction Assessment

Yes. A source-level reproduction is clear: put a valid `SKILL.md` under a non-root path such as `$HOME/my-new-skill` outside `<workspace>/skills`, `.agents/skills`, `~/.openclaw/skills`, and `skills.load.extraDirs`; current loading, status, CLI check, and audit paths have no enumerator that can report it.

## Solution Assessment

Unclear. The requested broad home/startup scan would catch the example, but a bounded explicit diagnostic is the narrower maintainable direction unless maintainers choose a privacy-safe implicit scan policy.
