## Summary

Keep open. Current main still loads and reports skills only from documented/configured roots; `allowSymlinkTargets` helps intentional sibling-repo layouts, but it does not detect unrelated orphan `SKILL.md` directories or warn when skill-creator scaffolds outside a load root.

## What This Changes

The issue requests safeguards or diagnostics for skill repositories created outside canonical skill paths so they do not become undiscovered orphan directories.

## Best Possible Solution

Add a bounded orphan-skill diagnostic, likely in `openclaw skills check` or doctor, and align skill-creator defaults or override warnings after maintainers choose scan scope, privacy behavior, and trust policy.

## Reproduction Assessment

Yes, source-reproducible. A valid `SKILL.md` outside the documented/configured roots is not enumerated by the loader, skills list/check, doctor readiness, or status paths unless it is moved into a known root or explicitly configured.

## Solution Assessment

Unclear. The suggested home-directory/startup scan would catch the example, but the maintainable fix needs a bounded scope, redaction/privacy policy, and scaffold override policy first.
