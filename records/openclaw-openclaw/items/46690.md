## Summary

Keep open. Current main still documents and loads bounded skill roots only; `skills check`, doctor workspace status, and the workspace skill audit do not inventory arbitrary orphan `SKILL.md` directories outside those roots, and the existing review/labels correctly route this to maintainer product policy first.

## What This Changes

The issue asks for diagnostics or scaffolding guardrails that warn when valid skill directories are created outside canonical OpenClaw skill roots.

## Best Possible Solution

Add a bounded orphan-skill diagnostic in `openclaw skills check` or doctor, then align skill-creator warnings once maintainers define scan scope, redaction, duplicate handling, and override policy.

## Reproduction Assessment

Yes, source-reproducible. A valid `SKILL.md` under a directory such as `~/my-new-skill` is outside the documented/configured roots and is not reached by the loader, `skills check`, doctor workspace status, or the workspace-only skill audit.

## Solution Assessment

Unclear. A broad home-depth scan would catch the reported example, but the maintainable solution needs explicit maintainer policy for scan bounds, redaction, performance, duplicate handling, and scaffold override behavior.
