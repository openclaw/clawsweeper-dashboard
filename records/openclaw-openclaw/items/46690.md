## Summary

Keep open. Current main still lacks orphan-skill diagnostics: documented skill discovery, `skills check`, status, and the security audit all operate from known/configured roots, while the bundled skill initializer still accepts any requested output directory. The report is valid, but the exact scan surface needs maintainer judgment.

## What This Changes

Request a guard or diagnostic that warns when skill-like `SKILL.md` directories are created outside documented OpenClaw skill roots, plus safer defaults for skill scaffolding.

## Best Possible Solution

Add a bounded explicit diagnostic, likely under `openclaw skills check` and/or doctor, that compares candidate `SKILL.md` folders against documented load roots and reports folders that will not be loaded. Keep startup behavior conservative unless maintainers explicitly approve background home-directory scanning, and update docs plus focused tests around canonical-root behavior.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: place a valid `$HOME/my-new-skill/SKILL.md` outside `<workspace>/skills`, `.agents/skills`, `~/.openclaw/skills`, and configured `skills.load.extraDirs`; current `loadWorkspaceSkillEntries`, `skills check`, status, and the workspace skill audit have no path that enumerates it.

## Solution Assessment

Unclear. The requested broad home-directory or startup scan would catch the example, but the safer maintainable direction is a bounded explicit diagnostic unless maintainers choose a privacy-safe implicit scan policy.
