## Summary

Keep open. Current main and the latest release still enumerate configured skill roots only, while `skills list` and `skills check` do not run an orphan-skill inventory; solving this needs maintainer policy for scan scope, privacy/redaction, duplicate rules, and scaffold behavior.

## What This Changes

The issue requests warnings or diagnostics for `SKILL.md` directories created outside supported OpenClaw skill roots, plus safer skill scaffolding defaults.

## Best Possible Solution

Define a bounded orphan-skill diagnostic for `openclaw skills check` or doctor, keep normal loading limited to configured roots, and document or scaffold users toward `openclaw skills install` and workspace skill directories.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: create a valid `SKILL.md` under an unconfigured directory such as `~/my-skill`, and current loader/status/check paths will not load or report it; I did not create files because this review is read-only.

## Solution Assessment

Unclear. The requested direction is useful, but maintainers should first choose scan roots, depth limits, redaction, duplicate handling, CLI surface, and scaffold override semantics.
