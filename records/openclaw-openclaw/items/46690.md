## Summary

Keep open. Current main still loads and reports only supported/configured skill roots; recent main added Git/local skill installs that reduce one source of confusion, but there is still no orphan-skill diagnostic for skill-like directories outside those roots, and the safe scan scope needs maintainer policy.

## What This Changes

Request an OpenClaw skills diagnostic or scaffold guard that warns about valid `SKILL.md` directories created outside supported skill roots.

## Best Possible Solution

Define a bounded orphan-skill diagnostic in `openclaw skills check` or doctor, keep normal loading limited to supported roots, and steer creation/install flows toward existing workspace or managed skill directories.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a valid `SKILL.md` directory outside every supported/configured skill root will not be loaded or reported by current `skills list`, `skills check`, status, or workspace audit paths.

## Solution Assessment

Unclear. The requested warning is useful, but maintainers should first choose the privacy-safe scan roots, depth limits, redaction, duplicate semantics, and whether this belongs in `skills check`, doctor, or a scaffold/install workflow.
