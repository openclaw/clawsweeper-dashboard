## Summary

Keep open. Current main still only loads and reports configured skill roots; it has symlink-aware containment for managed/user roots, but no diagnostic that searches for arbitrary orphan `SKILL.md` directories outside the canonical skill locations, so the reported confusion remains real and unresolved.

## What This Changes

The issue asks for diagnostics and scaffold-path guardrails so skill-like directories created outside canonical OpenClaw skill roots are warned about instead of silently ignored.

## Best Possible Solution

Add a bounded orphan-skill diagnostic in `openclaw skills check` or doctor, then align skill-creator warnings once maintainers decide scan bounds, redaction, performance limits, and override policy.

## Reproduction Assessment

Yes, source-reproducible. A valid `SKILL.md` under an arbitrary directory such as `~/my-skill` is outside the documented roots and is not reached by the loader, skills list/check, doctor status, or the current workspace-only audit paths.

## Solution Assessment

Unclear. The suggested home-depth scan would catch the reported example, but the maintainable fix needs explicit decisions on scan bounds, privacy/redaction, performance, duplicate handling, and scaffold override behavior.
