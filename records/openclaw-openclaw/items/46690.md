## Summary

Keep open. Current main still has bounded skill loading/status/audit paths and the bundled skill-creator still accepts arbitrary output directories; the newer symlink-target config helps intentional sibling-repo layouts but does not detect orphan SKILL.md directories created elsewhere.

## What This Changes

The issue requests warnings or diagnostics for skill directories created outside OpenClaw's documented skill roots.

## Best Possible Solution

Add a bounded orphan-skill diagnostic, likely in openclaw skills check or doctor, and align skill-creator defaults or override warnings after maintainers choose the scan scope and privacy policy.

## Reproduction Assessment

Yes. Source inspection shows a valid SKILL.md outside the documented roots will not be enumerated by loading, skills list/check, status, doctor readiness, or the workspace symlink audit unless it is added through a configured root.

## Solution Assessment

Unclear. The issue's broad home-directory or startup scan would catch the example, but the maintainable fix needs a maintainer-approved privacy, performance, and override policy before implementation.
