## Summary

Keep open: current main still has no orphan-skill diagnostic, and the bundled skill-creator initializer can write a skill under any requested path, but adding home-directory scanning or changing creation defaults needs maintainer product judgment rather than cleanup closure.

## What This Changes

Request to warn about or detect SKILL.md skill directories created outside canonical/configured skill roots and make skill scaffolding default to the workspace skills directory.

## Best Possible Solution

Define a narrow diagnostic contract: keep loading limited to documented roots, point OpenClaw-owned creation flows at <workspace>/skills, and put any orphan detection behind a bounded skills check or doctor-style diagnostic rather than an unconditional startup scan.

## Reproduction Assessment

Yes, source-reproducible for the current behavior. The CLI status path only reports loadWorkspaceSkillEntries results from documented/configured roots, while init_skill.py can create a SKILL.md directory under any requested --path.

## Solution Assessment

Unclear. A broad home-directory/startup scan is likely not the narrowest maintainable default; a bounded diagnostic plus clearer OpenClaw-owned scaffolder behavior would be safer if maintainers want this feature.
