## Summary

Keep open. Current main still only loads, lists, checks, and audits documented/configured skill roots, while the bundled skill initializer still accepts an arbitrary output path, so the orphan-skill diagnostic requested here remains unimplemented.

## What This Changes

Request a guard or diagnostic for skill-like directories created outside documented OpenClaw skill roots.

## Best Possible Solution

Add an explicit bounded orphan-skill diagnostic, likely in skills check and/or doctor, that compares candidate SKILL.md folders against documented load roots and reports folders that will not be loaded; keep startup scanning opt-in unless maintainers approve it.

## Reproduction Assessment

Yes. A source-level reproduction is clear: place a valid SKILL.md under a non-root path such as $HOME/my-new-skill and outside <workspace>/skills, .agents/skills, ~/.openclaw/skills, and skills.load.extraDirs; current loading, skills check, status, and audit paths have no enumerator that can find it.

## Solution Assessment

Unclear. The broad home-directory or startup scan requested would catch the example, but the safer maintainable direction is a bounded explicit diagnostic unless maintainers choose a privacy-safe implicit scan policy.
