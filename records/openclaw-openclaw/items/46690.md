## Summary

Keep open. Current main still only loads and reports configured skill roots, and the active discussion/labels point to a real unresolved diagnostic gap that needs maintainer product policy before implementation.

## What This Changes

The issue asks OpenClaw to warn about valid `SKILL.md` directories created outside canonical skill roots and to steer scaffolding toward workspace-owned skill paths.

## Best Possible Solution

Add a bounded orphan-skill diagnostic in `openclaw skills check` or doctor, then align skill-creator warnings once maintainers decide scan scope and override policy.

## Reproduction Assessment

Yes, source-reproducible. A valid `SKILL.md` placed under an arbitrary directory such as `~/my-skill` is outside the documented/configured roots and is not reached by the loader, skills CLI status/check flow, doctor workspace status, or workspace-only skill audit.

## Solution Assessment

Unclear. The requested home-depth scan would catch the reported example, but the maintainable solution needs explicit policy for scan scope, privacy/redaction, performance, duplicate handling, and scaffold override behavior.
