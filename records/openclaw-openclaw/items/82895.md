## Summary

Keep open: this is a recent draft PR for a source-reproducible Slack bug, and current main still drops the interaction thread when Slack only supplies it on the message object. The code direction looks correct and I found no blocking implementation defect, but the external-PR real-behavior proof is only local mocked test output, not a real Slack run.

## What This Changes

The PR falls back from Slack block-action `container.thread_ts` to `message.thread_ts`, adds Slack heartbeat typing/clear status calls through `assistant.threads.setStatus`, and adds focused Slack tests.

## Best Possible Solution

The Slack plugin should preserve interaction thread timestamps into session and delivery context, then use the existing heartbeat adapter contract to set and clear native Slack assistant status for threaded follow-up turns.

## Reproduction Assessment

Yes from source inspection, not from a live Slack workspace. Current main reads block-action `threadTs` only from `container.thread_ts` and then uses that value for the session key and `deliveryContext.threadId`, so payloads with only `message.thread_ts` lose thread context.

## Solution Assessment

Yes for the code direction: the fallback preserves the existing thread-aware route, and Slack's documented `assistant.threads.setStatus` API supports setting and clearing status by `channel_id` and `thread_ts`. The remaining blocker is real behavior proof, not a different implementation shape.
