## Summary

Keep open: the PR is still necessary, appears correct, and has sufficient real behavior proof; current main and the latest release still have uncapped ClawHub JSON/error body reads, while the PR head caps the shared and install-resolution paths. The remaining maintainer decision is whether to accept the documented 16 MiB JSON cap and 8 KiB diagnostic cap once exact-head checks finish.

## What This Changes

The branch routes ClawHub JSON and error response body reads through bounded response helpers and adds regression tests for oversized success, error, and install-resolution streams.

## Best Possible Solution

Land the bounded ClawHub response-reader hardening after exact-head checks complete and maintainers accept the documented caps; keep artifact-download sizing as separate follow-up work.

## Reproduction Assessment

Yes, source-reproducible: current main and v2026.6.8 call response.text() and response.json() on untrusted ClawHub response bodies without a byte cap. I did not run the full test suite in this read-only review, but the PR body includes after-fix oversized-stream cancellation proof and I live-checked that a normal ClawHub search response is small and parseable.

## Solution Assessment

Yes, this is the best fix: centralizing ClawHub JSON reads through the existing readResponseWithLimit helper covers shared metadata and install resolution without adding config or a parallel API. readResponseTextSnippet is the existing narrow helper for bounded diagnostics, and artifact downloads remain outside this focused in-memory JSON/error-body change.
