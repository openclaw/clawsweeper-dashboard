## Summary

Keep this PR open for maintainer review: the patch appears correct, the central hardening is not on current main or v2026.6.8, and the remaining question is whether maintainers accept the proposed ClawHub response-size caps.

## What This Changes

The PR routes ClawHub JSON and error response body reads through bounded response helpers and adds regression tests for oversized success, error, and install-resolution streams.

## Best Possible Solution

Land the bounded ClawHub response-reader hardening once maintainers accept the caps and exact-head checks finish; keep artifact-download sizing as separate follow-up work.

## Reproduction Assessment

Yes, source-reproducible: current main and v2026.6.8 call response.text() and response.json() on untrusted ClawHub response bodies without a byte cap. The PR body also includes after-fix oversized-stream cancellation proof and a live normal-path ClawHub search proof.

## Solution Assessment

Yes, this is the best fix: centralizing ClawHub JSON reads through the existing readResponseWithLimit helper covers shared metadata and install resolution without adding config or a parallel API. readResponseTextSnippet is the existing narrow helper for bounded diagnostics, and artifact downloads remain outside this focused in-memory JSON/error-body change.
