## Summary

Keep open: current main and the latest release still use unbounded ClawHub JSON/error response readers, while this PR caps the shared and install-resolution JSON paths with sufficient real behavior proof. The remaining blocker is not a code finding, but the branch is currently conflicting with current main and maintainers still need to accept or tune the new response-size caps.

## What This Changes

This PR routes ClawHub JSON and error body reads through bounded response helpers and adds regression tests for oversized success, error, and install-resolution responses.

## Best Possible Solution

Rebase onto current main preserving ClawHub timeout clamping, then land the bounded ClawHub JSON/error readers once maintainers accept or tune the caps; keep artifact-download sizing as a separate contract decision.

## Reproduction Assessment

Yes. Source inspection shows current main and v2026.6.9 still use direct response.text() and response.json() on ClawHub metadata/error paths, and the PR body includes after-fix live normal-path output plus oversized-stream cancellation proof.

## Solution Assessment

Yes, after rebase. Centralizing ClawHub JSON reads through readResponseWithLimit and diagnostics through readResponseTextSnippet is the narrow owner-boundary fix; per-caller caps would duplicate policy, and artifact downloads are a separate binary-download contract.
