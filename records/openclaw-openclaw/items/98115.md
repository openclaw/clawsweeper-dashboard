## What This Changes

The PR adds best-effort `node.list` lookups to `openclaw devices list` and `openclaw devices approve` so matching paired node devices can print the pending `openclaw nodes approve <requestId>` command, with regression tests.

## Best Possible Solution

Land the additive devices CLI guidance after real CLI/Gateway proof, and keep or narrow the linked issue if Android or gateway diagnostic surfaces still need follow-up.

## Reproduction Assessment

Yes, at source level: current main shows `devices list` and explicit `devices approve` do not bridge from visible paired device/IP context to pending node approval commands. I did not run a live Android or Gateway repro in this read-only review.

## Solution Assessment

Mostly yes: reusing `node.list` diagnostics on device-oriented CLI surfaces is the right owner boundary and avoids changing approval semantics. The unresolved question is whether this PR's CLI-only scope is enough to close the broader linked report.
