## Summary

Keep this PR open: it targets a real Feishu streaming bug path, but the suffix-only change currently has a correctness issue around failed CardKit updates and still needs real Feishu/Lark behavior proof before merge.

## What This Changes

The PR changes Feishu CardKit streaming to send suffix-only text updates, starts the streaming markdown element empty, and updates focused tests to expect delta payloads.

## Best Possible Solution

Keep the fix Feishu-local, track the append base from successfully sent/displayed content instead of optimistic logical text, and require redacted live Feishu/Lark proof before merge.

## Reproduction Assessment

Yes for the source-level precondition: current main sends full accumulated text to the CardKit content endpoint and the focused tests assert that payload. I did not live-reproduce the Feishu client duplication, so the external rendering symptom still needs real Feishu/Lark proof.

## Solution Assessment

No as currently written: suffix-only content addresses the reported duplication direction, but it should not base future deltas on text that may not have been successfully sent. Live Feishu/Lark proof is also needed because the public generated API schema describes full-content updates.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve unsent text after failed delta updates:** `extensions/feishu/src/streaming-card.ts:401-405`

If one CardKit update fails, the next update computes its suffix from text Feishu never received, so the visible card permanently drops the missed segment; keep a separate last-sent append base or only advance it after a successful update.
