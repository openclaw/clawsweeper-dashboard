## Best Possible Solution

Use a trusted control-plane prompt, mechanically enforce the repair diff before commit, send an exact ClawSweeper CI-failure payload, and require maintainer security approval plus real workflow-run proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Enforce the promised patch boundary before committing:** `.github/workflows/ci-fix-agent.yml:303-319`
  - body: The prompt says the agent must not touch unrelated files, workflows, release configs, or security policy, but this step only checks whether any diff exists before the next step commits `git add -A`. A Codex mistake can therefore push generated files or forbidden workflow/security changes to a PR branch with the workflow's write token; add a mechanical diff allowlist/blocklist here and fail before commit when it is violated.
  - confidence: 0.9
- **[P2] Resolve PR numbers with an available token:** `.github/workflows/clawsweeper-dispatch.yml:310-312`
  - body: This step shells out with `GH_TOKEN="${GITHUB_TOKEN}"`, but the step only exports `GH_TOKEN` as the ClawSweeper app token and does not export `GITHUB_TOKEN`. With an empty token, the fallback returns `0`, so CI-failure dispatches can reach ClawSweeper without the exact PR number needed to target the failed pull request.
  - confidence: 0.85
- **[P3] Scope the retry cadence to the target PR:** `.github/workflows/ci-fix-agent.yml:89-109`
  - body: The step is labeled as a per-PR cadence gate, but it counts all recent `ci-fix-agent.yml` runs in the repository. Three unrelated CI-fix attempts in six hours would suppress repairs for every other PR, so filter the run history by the resolved PR or head SHA.
  - confidence: 0.8
