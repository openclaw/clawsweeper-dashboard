## Summary

## What This Changes

The PR adds a CI-failure workflow that can run a Codex repair agent, a repair prompt, and a ClawSweeper workflow_run dispatch for failed CI metadata.

## Best Possible Solution

Keep CI failure notification metadata-only by default, and only add branch-writing repair after maintainer policy approval, trusted-main control inputs, strict diff-scope enforcement, per-PR cadence, correct dispatch tokens, and redacted workflow-run proof.

## Reproduction Assessment

Not applicable as a current-main product bug; this is new CI automation. The blocking defects are source-reproducible from the proposed workflow lines and the live CodeQL status on the exact head.

## Solution Assessment

No; automatic branch-writing repair is not safe in this shape. The safer path is metadata-only dispatch or a trusted-main, tightly bounded repair workflow with explicit maintainer approval and real workflow-run proof.

  - body: After checking out the PR branch, codex-action reads `.github/codex/prompts/ci-fix-agent.md` from that branch while using a secret and a later write-token push. The prompt and control-plane policy should come from trusted main, not from the branch being repaired.

  - body: The enforcement step only checks whether any diff exists, then `git add -A` stages everything Codex produced. This can commit unrelated files or forbidden CI/security changes despite the prompt rules, so the workflow needs a mechanical path/status allowlist before staging.
  - confidence: 0.93
- **[P2] Resolve failed-CI PRs with an exported target token:** `.github/workflows/clawsweeper-dispatch.yml:310-312`
  - body: This command runs `gh` with `GH_TOKEN="${GITHUB_TOKEN}"`, but this job does not export `GITHUB_TOKEN`; lookup failures fall back to `0`, so ClawSweeper can receive an unusable PR number for failed CI events.
  - confidence: 0.89
- **[P2] Scope the retry budget to the target PR:** `.github/workflows/ci-fix-agent.yml:98-108`
  - body: The gate counts all recent `ci-fix-agent.yml` runs, not runs for `PR_NUMBER` or the head SHA. Three unrelated failures in six hours would disable repair for other PRs, which makes the automation unreliable under normal CI churn.
  - confidence: 0.86
