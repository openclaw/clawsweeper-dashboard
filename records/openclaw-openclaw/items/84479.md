  - body: The enforcement step only records that some change exists, then the commit step stages everything with `git add -A`. - **[P2] Resolve failed PRs with the app token:** `.github/workflows/clawsweeper-dispatch.yml:317-319`
  - body: This step exports the ClawSweeper app token as `GH_TOKEN`, but the PR lookup overrides it with `${GITHUB_TOKEN}`, which is not mapped in the step. That can make the lookup unauthenticated or use the wrong credential, so the dispatched payload can carry `pr_number: 0`.
  - confidence: 0.88
- **[P2] Send real failed-job logs to Codex:** `.github/workflows/ci-fix-agent.yml:154-160`
  - body: The triage workflow builds `log_excerpt` from check-run annotations only, not from failed job logs. Many GitHub Actions failures do not emit annotations, so the repair workflow can invoke Codex with no useful failure context and push a speculative fix.
  - confidence: 0.82
- **[P2] Create the ClawSweeper app token before dispatching:** `.github/workflows/ci-fix-agent.yml:218-220`
  - body: This step checks for `CLAWSWEEPER_APP_PRIVATE_KEY` and sets the app client id, but it never creates or exports the app token before calling `repos/openclaw/clawsweeper/dispatches`. The repository token is scoped to this repo, so the cross-repo dispatch can silently fail even when the app key exists.
  - confidence: 0.84
