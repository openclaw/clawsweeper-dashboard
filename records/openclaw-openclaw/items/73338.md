Yes, with a compatibility caveat: recording the runtime port in the existing Gateway lock and consulting it only for default local TUI resolution is the narrow owner-boundary fix. The safer alternative, if maintainers require env-port precedence to remain public API, is to make the active lock fallback only apply when no env port source is present.
