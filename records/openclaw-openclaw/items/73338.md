## Summary

## What This Changes

The branch records the active Gateway runtime port in lock metadata, makes default local `openclaw tui` resolution use that verified lock port, recognizes titled Gateway processes, and updates focused tests and docs.

## Best Possible Solution

Land this branch or an equivalent lock-backed fix after maintainers accept active-lock precedence and rerun current-base validation, while preserving explicit `--url`, `OPENCLAW_GATEWAY_URL`, remote config, and auth precedence.

## Reproduction Assessment

Yes. Source inspection on current main and v2026.6.9 shows TUI default URL resolution still comes from config/env/default port data while the Gateway lock payload does not persist the active runtime `--port`; the PR body adds after-fix live Gateway logs.

## Solution Assessment

Yes, with a compatibility caveat. Recording the runtime port in the existing Gateway lock and consulting it only for default local TUI resolution is the narrow owner-boundary fix, but maintainers must accept whether that active lock should outrank `OPENCLAW_GATEWAY_PORT` for TUI.
