## Summary

Keep open: current main and the latest release still lack the active-lock port behavior, and this PR is the viable linked fix candidate. ## What This Changes

The PR records the Gateway runtime port in the active lock, teaches default local `openclaw tui` resolution to use that verified lock port, updates process-title detection, tests, and docs.

## Best Possible Solution

Land this branch or an equivalent lock-backed fix once maintainers accept the TUI port-precedence behavior, preserving explicit `--url`, `OPENCLAW_GATEWAY_URL`, and remote config precedence with fresh current-base validation.

## Reproduction Assessment

Yes. Source inspection shows current main and `v2026.6.8` still build the default TUI URL from config/env/default port data while the Gateway lock payload does not persist the active runtime `--port`; I did not run the live two-process repro in this read-only review.

## Solution Assessment

Yes, with a compatibility caveat. Recording the runtime port in the existing Gateway lock and consulting it only for default local TUI resolution is the narrow owner-boundary fix; if maintainers treat `OPENCLAW_GATEWAY_PORT` as a public TUI override, the safer alternative is to make the active lock a fallback below that env source.
