## Summary

Keep open. Current main still has a source-provable gap where an explicit `sessions_spawn.model` is resolved and reported as applied before the child run has any validation that the selected provider has a runnable CLI backend; the open related PRs are complementary, not a complete fix for the latest reproduced failure mode.

## What This Changes

The issue asks for `sessions_spawn(model=...)` to actually run the sub-agent on the requested model or fail visibly instead of silently falling back to the orchestrator default.

## Best Possible Solution

Add a narrow spawn-time validation for explicit resolved model providers so missing or unrunnable CLI backends fail with a clear error before `modelApplied` or an accepted child run is returned.

## Reproduction Assessment

Yes at source level. Configure a spawn with an explicit `provider/model` whose provider has no configured/registered CLI backend, and current main can persist the requested model and return `modelApplied` without preflighting the backend contract that the child runtime later depends on.

## Solution Assessment

Yes. The narrow maintainable fix is to validate the resolved provider with the same CLI backend resolver before accepting the spawn, while leaving the broader metadata/default-precedence work in the related PRs separate.
