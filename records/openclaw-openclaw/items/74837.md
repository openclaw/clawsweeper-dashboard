## Summary

Keep open. This is a real `sessions_spawn.model` correctness report with fresh current-main discussion, and the linked open PR at https://github.com/openclaw/openclaw/pull/74892 only addresses part of the response-flag semantics, not the remaining runtime handoff/fallback concern.

## What This Changes

The issue asks for `sessions_spawn(model=...)` to make the child session run with the requested model instead of silently using the orchestrator default while reporting `modelApplied: true`.

## Best Possible Solution

Keep the prior override-persistence behavior, then add a narrow guarantee that an explicit spawn model is either passed through to the child runtime as a prepared fact or rejected before `modelApplied` can report success.

## Reproduction Assessment

Yes, with source-backed confidence from the May 8 repro comments: spawn with an explicit model whose provider/runtime is not honored, then compare `modelApplied: true` with the child transcript/logged runtime model. I did not run the live provider scenario here, so this is source-reproducible rather than locally reproduced.

## Solution Assessment

Unclear whether the best narrow patch is fail-fast backend validation or passing a prepared provider/model runtime fact into the child `agent` request. The current linked PR is not sufficient for the remaining behavior because it changes response semantics without closing the runtime handoff gap.
