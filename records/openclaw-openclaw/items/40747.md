## Summary

Keep open: current main still lacks root-index Feishu doc insertion, but this PR needs branch fixes and inspectable live Feishu proof before merge.

## What This Changes

The PR makes Feishu `feishu_doc(action: "insert")` accept optional `after_block_id`, adds optional root-child `index`, and passes that position into Docx insertion.

## Best Possible Solution

Land a rebased Feishu plugin change that exposes root-child index insertion, validates `index` as a non-negative integer, handles empty anchors explicitly, preserves current block-tree normalization, adds focused regression coverage, and includes redacted live Feishu proof.

## Reproduction Assessment

Yes by source inspection for the reported limitation: current main requires `after_block_id` in the insert schema and the runtime only dispatches anchored insert. I did not run live Feishu calls in this read-only review.

## Solution Assessment

No as-is. Exposing the Feishu Descendant API `index` is the right narrow direction, but this branch should tighten validation, preserve or explicitly reject empty anchors, rebase over current ordering logic, add regression coverage, and provide inspectable live proof.

- **[P2] Constrain insert index validation:** `extensions/feishu/src/doc-schema.ts:43-48`
  - body: The new root-child `index` is documented as a 0-based position, but `Type.Number()` lets negative and fractional values pass OpenClaw validation and fail later in Feishu. Use `Type.Integer({ minimum: 0, ... })`, matching the existing positional image index schema.

  - confidence: 0.84
