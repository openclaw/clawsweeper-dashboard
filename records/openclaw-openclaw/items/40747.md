## Summary

Keep open: the user problem is real and the PR’s runtime direction matches the existing Descendant API path, but the branch is dirty against current main and the new schema accepts invalid positional indexes before reaching Feishu.

## What This Changes

The branch makes Feishu `feishu_doc` insert accept an optional `index`, makes `after_block_id` optional, and passes the index into the insert path when no anchor block is supplied.

## Best Possible Solution

Land a rebased or replacement Feishu plugin change that exposes root-position inserts with integer validation, regression coverage, docs/skill guidance, and a changelog entry while preserving `after_block_id` semantics.

## Reproduction Assessment

Yes from source inspection: current main requires `after_block_id` in the schema and `insertDoc` always resolves that anchor before inserting. I did not run live Feishu calls in this read-only review.

## Solution Assessment

No as-is. The Descendant API index path is the right narrow direction, but the PR should constrain `index` to a non-negative integer, rebase or be replaced on current main, and add tests/docs/changelog before merge.
