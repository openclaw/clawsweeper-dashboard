## Summary

Keep open: current main still lacks no-anchor/root-position Feishu Docx inserts, and this PR is a plausible narrow plugin feature, but the patch needs stricter index validation and contributor-supplied real behavior proof before merge.

## What This Changes

The PR makes Feishu `feishu_doc` insert accept optional `after_block_id` plus a root-child `index`, then passes that index into Docx insertion control flow.

## Best Possible Solution

Land a rebased Feishu plugin change that exposes no-anchor root-position inserts through the Descendant index, validates a non-negative integer index, preserves anchored behavior, and covers the schema/docs/test surface.

## Reproduction Assessment

Yes for the current limitation: source inspection shows current main requires `after_block_id` in the insert schema and resolves that anchor before insertion, so a root-position insert cannot be expressed. I did not run live Feishu calls in this read-only pass.

## Solution Assessment

No as-is. The Descendant API index path is the narrow maintainable direction, but the PR should use non-negative integer validation and include real behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Constrain insert index validation:** `extensions/feishu/src/doc-schema.ts:44-48`
  - body: The new root-child `index` is a 0-based position, but `Type.Number()` lets negative and fractional values pass tool validation and fail later as Feishu API errors. Use `Type.Integer({ minimum: 0, ... })`, matching the existing `upload_image` index schema.
