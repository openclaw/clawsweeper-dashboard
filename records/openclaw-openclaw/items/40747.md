## Summary

Keep open: current main still only supports anchor-based Feishu doc inserts, and this PR is a focused plugin-level feature direction, but it should not merge as-is because the new index is under-validated and contributor-attached real Feishu proof is still missing.

## What This Changes

The PR makes Feishu `feishu_doc` insert accept optional `after_block_id`, adds a root-child `index`, and passes that index into Docx Descendant insertion.

## Best Possible Solution

Expose no-anchor root-position inserts in the Feishu plugin through Descendant `index`, validate it as a non-negative integer, preserve anchored behavior, and add proof plus focused coverage where practical.

## Reproduction Assessment

Yes for the current limitation by source inspection: current main requires `after_block_id` in the insert schema and `insertDoc` always resolves that anchor before computing the Descendant insertion index. I did not run live Feishu calls in this read-only pass.

## Solution Assessment

No as-is. The Descendant `index` path is the narrow maintainable direction, but the PR should constrain `index` with `Type.Integer({ minimum: 0 })` and include real behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9
