## Summary

Keep open: current main still requires an `after_block_id` anchor for Feishu doc inserts, so the PR addresses a real limitation, but the branch is not merge-ready because `index` is under-validated and contributor-visible live proof is still missing.

## What This Changes

The branch makes Feishu `feishu_doc` insert accept optional `after_block_id`, adds an optional root-child `index`, and forwards that index into Docx descendant insertion.

## Best Possible Solution

Land a rebased Feishu plugin change that exposes no-anchor root-position inserts through the Descendant API, validates `index` as a non-negative integer, preserves anchored behavior, and includes redacted live Feishu proof.

## Reproduction Assessment

Yes for the current limitation: source inspection shows current main requires `after_block_id` in the schema and computes insert position only from that anchor. I did not run live Feishu calls in this read-only review.

## Solution Assessment

No as-is. Exposing the Descendant API index is the narrow maintainable direction, but the PR should constrain `index` to a non-negative integer and attach real after-fix proof before merge.
