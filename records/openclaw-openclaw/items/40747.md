## Summary

Keep this PR open, but it still needs changes before merge. Current main still lacks a no-anchor Feishu document insert path, and the PR addresses the right plugin-owned surface, but both Greptile and the existing ClawSweeper review correctly identify a blocking schema bug: the new root-child `index` is modeled as `Type.Number()` instead of a non-negative integer.

## What This Changes

The PR makes Feishu `feishu_doc` insert accept an optional `after_block_id`, adds a root-level `index` parameter, and forwards that index to Descendant insertion when no anchor block is provided.

## Best Possible Solution

Land the Feishu plugin feature after constraining `index` to `Type.Integer({ minimum: 0 })`, preserving anchored insert behavior, adding focused schema/runtime coverage for root-position inserts, updating the bundled Feishu document skill guidance, and adding a changelog entry that credits the contributor.

## Reproduction Assessment

Yes. Static current-main verification is high confidence: a no-anchor insert cannot satisfy the current schema, and the runtime always dereferences an anchor block before computing the insertion position. The PR-side bug is also directly reproducible from the diff because `Type.Number()` accepts fractional values for a positional index.

## Solution Assessment

No as proposed. Exposing Feishu Descendant root insertion through the Feishu plugin is the right boundary, but the patch should reject negative and fractional indexes before API execution and should land with focused tests and docs/changelog updates.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Constrain insert index to non-negative integers:** `extensions/feishu/src/doc-schema.ts:43-48`

Those invalid values would reach Feishu's API even though the existing `upload_image` positional schema uses `Type.Integer({ minimum: 0 })`, so users get later API errors instead of schema validation.
