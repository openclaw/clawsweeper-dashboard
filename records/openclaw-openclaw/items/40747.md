## Summary

Keep open: this PR addresses a real Feishu doc insertion gap and the runtime direction is reasonable, but the patch should not merge as-is because the new `index` parameter is too loosely validated and the user-facing change lacks the required changelog entry.

## What This Changes

The PR makes Feishu `feishu_doc` insert accept optional `after_block_id` plus an optional root-child `index`, then passes that index through `insertDoc` to Descendant insertion.

## Best Possible Solution

Land a narrow Feishu plugin change that preserves `after_block_id` insertion, exposes a non-negative integer root `index` path, and adds focused regression coverage plus a changelog entry.

## Reproduction Assessment

Yes, from current-main source: the insert schema requires `after_block_id`, and the handler passes that required value into `insertDoc`. A tool call using only `index` cannot be accepted on current main without this feature.

## Solution Assessment

No, not as written. Exposing Feishu's existing positional insertion support is the right narrow direction, but the new schema should use `Type.Integer({ minimum: 0 })` and the PR should include the required changelog and focused regression coverage.
