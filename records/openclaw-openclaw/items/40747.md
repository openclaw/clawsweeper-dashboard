## Summary

Keep this PR open: it targets a real missing Feishu plugin capability, but the current patch still has two correctness issues and lacks attached real behavior proof after the fix.

## What This Changes

The PR makes Feishu `feishu_doc(action: "insert")` accept an optional `after_block_id`, adds an optional root-child `index`, and forwards that index into Docx descendant insertion.

## Best Possible Solution

Land a rebased Feishu plugin change that exposes root-position inserts through the Descendant API index, validates positional inputs, preserves anchored behavior, adds focused regression coverage, and includes redacted live Feishu proof.

## Reproduction Assessment

Yes for the current limitation: source inspection shows current main requires `after_block_id` in the insert schema and passes only that anchor into `insertDoc`. I did not run live Feishu calls.

## Solution Assessment

No as-is. The Descendant API index path is the right narrow direction, but the patch should constrain `index`, preserve provided-anchor semantics, and attach real after-fix proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Constrain insert index validation:** `extensions/feishu/src/doc-schema.ts:43-48`
  - body: The new root-child `index` is documented as a 0-based position, but `Type.Number()` lets negative and fractional values pass OpenClaw validation and fail later in Feishu. Use `Type.Integer({ minimum: 0, ... })`, matching the existing positional image index schema.
  - confidence: 0.92
- **[P2] Treat only undefined as no-anchor mode:** `extensions/feishu/src/docx.ts:886`
  - body: `after_block_id` remains a plain optional string, so callers can provide an empty string. With `if (afterBlockId)`, that provided anchor is treated as omitted and inserts at root index 0 instead of following the old anchored error path; check for `undefined` explicitly or reject empty anchors in the schema.
  - confidence: 0.82
