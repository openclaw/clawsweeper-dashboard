## Summary

Keep this PR open: the Feishu root-index insert capability is still absent on current main and the proposed direction is useful, but the branch has correctness issues and still lacks inspectable real Feishu behavior proof before merge.

## What This Changes

The PR makes Feishu `feishu_doc(action: "insert")` accept optional `after_block_id`, adds an optional root-child `index`, and forwards that position into Docx descendant insertion.

## Best Possible Solution

Land a rebased Feishu plugin change that exposes root-child index insertion, validates `index` as a non-negative integer, handles empty anchors explicitly, preserves current block-tree normalization, adds focused regression coverage, and includes redacted live Feishu proof.

## Reproduction Assessment

Yes for the reported limitation, by source inspection: current main requires `after_block_id` in the insert schema and only dispatches anchored insert, so root-index insertion cannot be expressed. I did not run live Feishu calls in this read-only review.

## Solution Assessment

No as-is. Exposing the Descendant API `index` is the right narrow direction, but the patch should tighten validation, preserve or explicitly reject empty anchors, rebase over current ordering logic, add regression coverage, and provide inspectable live proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Constrain insert index validation:** `extensions/feishu/src/doc-schema.ts:43-48`
  - body: The new root-child `index` is documented as a 0-based position, but `Type.Number()` lets negative and fractional values pass OpenClaw validation and fail later in Feishu. Use `Type.Integer({ minimum: 0, ... })`, matching the existing positional image index schema.
  - confidence: 0.92
- **[P2] Preserve provided empty anchors:** `extensions/feishu/src/docx.ts:886`
  - body: With `if (afterBlockId)`, a caller that provides `after_block_id: ""` no longer follows the anchored path and will insert at root index 0. Check for `undefined` explicitly or reject empty strings in the schema so provided anchors keep the old failure behavior instead of silently changing target.
  - confidence: 0.82
