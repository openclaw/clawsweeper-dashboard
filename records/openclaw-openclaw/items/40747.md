## Summary

Keep open: the Feishu insert-at-index capability is still absent from current main and the approach is plausible, but this PR is not merge-ready because it has concrete schema/compatibility defects, no inspectable real behavior proof, and the branch is currently conflicting.

## What This Changes

The PR makes `feishu_doc` insert accept an optional `after_block_id` plus a new optional `index` parameter and routes omitted-anchor inserts through the existing Feishu Descendant API path.

## Best Possible Solution

Keep the Feishu plugin change, but land it only after rebasing, constraining `index` as a non-negative integer, preserving provided-anchor semantics, adding focused regression coverage, and attaching redacted live Feishu proof.

## Reproduction Assessment

Yes. Current main's schema requires `after_block_id` for `feishu_doc` insert and the runtime accepts `afterBlockId: string`, so an omitted-anchor insert is source-reproducibly unsupported without needing a live Feishu document.

## Solution Assessment

No, not as written. Exposing the existing Descendant API index is a maintainable direction, but the PR should first preserve provided-anchor behavior, validate `index` as a non-negative integer, and provide real Feishu proof.
