## Summary

Keep open: the Feishu insert-index feature is still useful and not implemented on current main, but this PR is not merge-ready because the new index parameter is unconstrained and contributor-supplied real behavior proof is still absent.

## What This Changes

The PR makes Feishu `feishu_doc` insert accept optional `after_block_id`, adds a root-child `index` parameter, and passes that index into Docx descendant insertion.

## Best Possible Solution

Land a rebased Feishu plugin change that exposes no-anchor root-position inserts through the Descendant index, validates `index` as a non-negative integer, preserves anchored behavior, and includes redacted live Feishu proof.

## Reproduction Assessment

Yes for the current limitation: current main requires `after_block_id` in the schema and routes insert through an anchor-only code path. I did not run live Feishu calls in this read-only review.

## Solution Assessment

No as-is. The Descendant API index path is the narrow maintainable direction, but the PR should use non-negative integer validation and include real behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Constrain insert index validation:** `extensions/feishu/src/doc-schema.ts:44-48`
  - body: The new root-child `index` is a 0-based position, but `Type.Number()` accepts negative and fractional values, so OpenClaw validation would pass inputs that Feishu rejects later. Use `Type.Integer({ minimum: 0, ... })`, matching the existing `upload_image` index schema.
