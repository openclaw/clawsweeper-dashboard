## Summary

Keep open. Current main does route Telegram network config into the grammY/undici transport path, but it still does not apply the requested explicit IPv4-only behavior to the first Bot API dispatcher: `family: 4` is only added for the later sticky fallback attempt. The report is valid, narrow, and suitable for a focused Telegram transport repair.

## What This Changes

The issue requests that Telegram `network.autoSelectFamily` and `network.dnsResultOrder` config be honored by the grammY/undici Bot API client so IPv6-broken hosts can force IPv4 without timing out.

## Best Possible Solution

Honor explicit Telegram IPv4-only operator intent on the initial Bot API dispatcher, likely by treating `autoSelectFamily:false` plus `dnsResultOrder:'ipv4first'` as an IPv4-pinned direct connect policy, while leaving default Node 22 behavior, proxy paths, sticky fallback, per-account transport isolation, and explicit `autoSelectFamily:true` or `verbatim` configurations intact.

## Reproduction Assessment

Yes for the source-level bug: construct `resolveTelegramTransport(undefined, { network: { autoSelectFamily:false, dnsResultOrder:'ipv4first' } })` and inspect the first dispatcher policy; it lacks `family:4` while the fallback policy includes it. The full live timeout still needs an IPv6-broken host to reproduce end to end.

## Solution Assessment

Yes, the best fix is a narrow Telegram transport-policy change in `extensions/telegram/src/fetch.ts` with regression coverage. A global Node DNS or undici dispatcher change would have wider blast radius and would risk the inverse network case.
