## Summary

Keep open. Current main does route Telegram network config into the Bot API transport, but explicit IPv4-only operator intent still does not pin the first dispatcher to IPv4; `family: 4` is only applied after the sticky fallback path is reached.

## What This Changes

The issue asks Telegram Bot API requests to honor `channels.telegram.network.autoSelectFamily:false` plus `dnsResultOrder:'ipv4first'` on IPv6-broken hosts without timing out on IPv6.

## Best Possible Solution

Pin the initial Telegram dispatcher to IPv4 only when explicit config clearly requests IPv4-only behavior, while preserving proxy modes, per-account isolation, sticky fallback, and IPv6-friendly settings.

## Reproduction Assessment

Yes for the source-level issue: inspect `resolveTelegramTransport` with `autoSelectFamily:false` and `dnsResultOrder:'ipv4first'`; the first dispatcher lacks `family:4` while the fallback policy sets it. Full live reproduction still requires a host where IPv6 to `api.telegram.org` is unreachable and IPv4 works.

## Solution Assessment

Yes. The narrowest maintainable fix belongs in `extensions/telegram/src/fetch.ts` with focused transport-policy coverage rather than changing process-global DNS or undici dispatcher behavior.
