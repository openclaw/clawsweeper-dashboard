## Summary

Keep open. Current main still wires Telegram network config into the grammY/undici transport, but the initial dispatcher is not IPv4-pinned for the explicit IPv4-only configuration in this report; `family:4` is only applied after the sticky fallback path is promoted. No open PR specifically references this issue with closing syntax, and the adjacent generic networking PR does not resolve this Telegram-specific first-dispatcher behavior.

## What This Changes

The issue asks OpenClaw to make Telegram Bot API requests honor `channels.telegram.network.autoSelectFamily` and `dnsResultOrder` so IPv6-broken hosts can force IPv4 without an `/etc/hosts` workaround.

## Best Possible Solution

Keep this issue open and land a narrow Telegram plugin fix that recognizes explicit IPv4-only operator intent on the initial direct Bot API dispatcher, with regression tests for first-attempt dispatch policy and safeguards for proxy and IPv6-friendly configurations.

## Reproduction Assessment

Unclear for the full live timeout because this review did not have an IPv6-broken host. The narrower source-level reproduction is high-confidence: construct `resolveTelegramTransport` with `autoSelectFamily:false` and `dnsResultOrder:'ipv4first'` and the first dispatcher policy lacks `family:4`.

## Solution Assessment

Yes, a narrow Telegram transport change is the best way to solve this. It should live in `extensions/telegram/src/fetch.ts` and focused Telegram tests, not in global Node DNS defaults or core config migration.
