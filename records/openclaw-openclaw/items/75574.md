## Summary

Keep open. Current main still threads Telegram network config into the grammY fetch path, but the first direct undici dispatcher is not IPv4-pinned for explicit IPv4-first/no-auto settings; IPv4-only behavior is only reached after fallback, so the reported timeout shape remains a valid narrow bug.

## What This Changes

The issue requests that Telegram Bot API transport apply `channels.telegram.network` IPv4-first/no-auto settings to grammY/undici so requests do not time out on IPv6-broken hosts.

## Best Possible Solution

Pin the initial direct Telegram dispatcher to IPv4 only for explicit IPv4-first/no-auto configuration, while preserving proxy modes, sticky fallback, per-account isolation, and IPv6-friendly configs.

## Reproduction Assessment

Yes for a source-level reproduction: resolve Telegram transport with `network.autoSelectFamily=false` and `network.dnsResultOrder="ipv4first"`, then inspect that the first dispatcher is created without `family: 4`. A full live repro needs a host where IPv6 to `api.telegram.org` is unreachable and IPv4 works.

## Solution Assessment

Yes. The narrowest maintainable fix is local to the Telegram transport policy in `extensions/telegram/src/fetch.ts` with focused dispatcher-policy regression tests, not a global Node DNS or undici dispatcher change.
