## Summary

Keep open. Current main still reads and passes Telegram network config into the grammY Bot API fetch path, but the initial undici dispatcher is created with `forceIpv4: false`; IPv4 pinning is only applied after fallback, so the reported IPv6-broken/IPv4-working timeout remains a valid narrow bug.

## What This Changes

Telegram should apply `channels.telegram.network.autoSelectFamily=false` plus `dnsResultOrder:'ipv4first'` to the initial grammY/undici Bot API dispatcher so IPv6-broken hosts use IPv4 immediately.

## Best Possible Solution

Pin the initial Telegram dispatcher to IPv4 only for explicit IPv4-only config while preserving proxy modes, per-account isolation, sticky fallback, and IPv6-friendly configs.

## Reproduction Assessment

Yes for a source-level reproduction: resolve Telegram transport with `network.autoSelectFamily=false` and `network.dnsResultOrder='ipv4first'`, then inspect the first dispatcher policy/options; current main lacks `family: 4` on that first attempt. A full live reproduction still requires a host where IPv6 to `api.telegram.org` is unreachable and IPv4 works.

## Solution Assessment

Yes. The narrowest maintainable fix is local to `extensions/telegram/src/fetch.ts` with focused dispatcher-policy tests; changing global Node DNS settings or undici's global dispatcher would be broader and riskier.

Fix the Telegram transport bug reported in https://github.com/openclaw/openclaw/issues/75574. When `channels.telegram.network.autoSelectFamily` is explicitly `false` and `channels.telegram.network.dnsResultOrder` is `ipv4first`, the initial Bot API undici dispatcher should be IPv4-pinned instead of waiting for sticky fallback after a failed first request. Keep the fix inside the Telegram plugin transport, primarily `extensions/telegram/src/fetch.ts`, and add focused regression coverage in `extensions/telegram/src/fetch.test.ts` that inspects the first dispatcher policy/options. Do not change global Node DNS settings, undici global dispatcher behavior, startup migrations, or broad config schema unless a test proves it is necessary. Be careful not to regress #41671: IPv6-friendly explicit configs such as `autoSelectFamily:true` or `dnsResultOrder:'verbatim'` must not be forced to IPv4.
