## Summary

Keep open. Current main routes Telegram network config into the Bot API transport, but the initial grammY/undici dispatcher still uses `forceIpv4:false`; `family:4` is only applied on the sticky fallback path, so the reported IPv4-only operator intent is not implemented for the first request.

## What This Changes

The issue asks Telegram Bot API requests to honor `channels.telegram.network.autoSelectFamily:false` plus `dnsResultOrder:'ipv4first'` on hosts where IPv6 egress is broken.

## Best Possible Solution

Make Telegram's initial dispatcher honor explicit IPv4-only operator intent while preserving proxy handling, default Node happy-eyeballs behavior, per-account transport isolation, and IPv6-friendly explicit configs.

## Reproduction Assessment

Unclear for the full live timeout because this review did not run on an IPv6-broken host. The source-level reproduction is high-confidence: create a Telegram transport with `autoSelectFamily:false` and `dnsResultOrder:'ipv4first'`, then inspect that the first dispatcher policy lacks `family:4`.

## Solution Assessment

Yes, a narrow Telegram transport policy change is the best path. The repair should live in `extensions/telegram/src/fetch.ts` with focused regression tests, not in global Node DNS state or core config migration.
