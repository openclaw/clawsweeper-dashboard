## Summary

Keep open. Current main reads Telegram network config and routes it into the Bot API transport, but the first grammY/undici dispatcher still does not pin IPv4 for the reported `autoSelectFamily:false` plus `dnsResultOrder:ipv4first` configuration; IPv4 pinning is only present on the fallback dispatcher.

## What This Changes

The issue asks for Telegram network config to affect grammy/undici HTTP requests so IPv4-first settings prevent timeouts on IPv6-broken hosts.

## Best Possible Solution

Make Telegram's initial dispatcher honor explicit IPv4-only operator intent while preserving proxy handling, fallback behavior, and IPv6-friendly explicit configs.

## Reproduction Assessment

Yes for the source-level bug: construct `resolveTelegramTransport(undefined, { network: { autoSelectFamily: false, dnsResultOrder: "ipv4first" } })` and inspect that the first dispatcher lacks `family:4` while the fallback has it. The full network timeout still needs an IPv6-broken host to reproduce live.

## Solution Assessment

Yes, the narrow maintainable fix belongs in Telegram transport policy and focused regression tests. It should not use process-global DNS state or a core config migration.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/75574 in the Telegram plugin only. The observable bug is that `channels.telegram.network.autoSelectFamily:false` with an effective `dnsResultOrder:"ipv4first"` reaches the first Bot API dispatcher without `connect.family = 4`, so IPv6-broken hosts can still time out before fallback. Update `extensions/telegram/src/fetch.ts` so explicit IPv4-only operator intent is honored on the initial dispatcher for direct, env-proxy, and explicit-proxy transport policy where applicable, while preserving existing sticky fallback, pinned-IP fallback, proxy handling, and SSRF protections. Do not force IPv4 when the effective DNS order is `verbatim` or `autoSelectFamily:true`, because #41671 and #55099 cover IPv6-working configurations. Add focused regression tests in `extensions/telegram/src/fetch.test.ts` for the first dispatcher and keep existing fallback tests passing. Add a single-line `CHANGELOG.md` fix entry because this is user-facing. Do not touch core config migrations or global DNS defaults unless the Telegram transport tests prove it is necessary.
