## Summary

Keep open. Current main does pass Telegram network config into the grammY/undici transport, but the reported IPv4-only behavior is still not implemented on the first Bot API dispatcher: `autoSelectFamily:false` plus `dnsResultOrder:ipv4first` produces an ordered lookup with auto family selection disabled, while `family:4` is only used after the sticky fallback path promotes to the IPv4 dispatcher.

## What This Changes

The issue requests that Telegram `channels.telegram.network` settings, especially `dnsResultOrder: "ipv4first"` and `autoSelectFamily: false`, affect actual grammY/undici requests so IPv6-broken hosts use IPv4 without timing out.

## Best Possible Solution

Make Telegram's initial Bot API dispatcher honor explicit IPv4-first operator intent in `extensions/telegram/src/fetch.ts`, while preserving default Node 22 behavior, proxy handling, per-account transport isolation, sticky fallback behavior, and IPv6-friendly configurations. Add focused regression tests that prove the first dispatcher is IPv4-pinned only for the intended config shape.

## Reproduction Assessment

Yes for the narrow source-level bug: constructing `resolveTelegramTransport` with `network: { autoSelectFamily:false, dnsResultOrder:"ipv4first" }` currently yields a first dispatcher without `family:4`. Unclear for the full live timeout because this review did not run on the reporter's IPv6-broken Oracle Cloud host.

## Solution Assessment

Yes, a narrow Telegram transport-policy repair is the best path. The fix belongs in `extensions/telegram/src/fetch.ts` policy selection rather than in core DNS defaults, global undici state, or process-wide Node options.
