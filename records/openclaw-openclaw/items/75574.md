## Summary

Keep open. Current main routes Telegram network config into the Bot API transport, but IPv4-only intent is still not applied to the first dispatcher; `family:4` is only set on the sticky fallback path after an initial network failure.

## What This Changes

The issue requests that Telegram Bot API requests honor `channels.telegram.network.autoSelectFamily` and `dnsResultOrder` so IPv6-broken hosts can use IPv4 without an `/etc/hosts` workaround.

## Best Possible Solution

Implement a narrow Telegram transport policy change so resolved IPv4-first plus disabled auto-family selection pins the initial dispatcher to IPv4, while preserving proxies, per-account isolation, default happy-eyeballs behavior, and IPv6-friendly explicit configs.

## Reproduction Assessment

Yes for the source-level issue: create a Telegram transport with `autoSelectFamily:false` and `dnsResultOrder:'ipv4first'`, then inspect `dispatcherAttempts[0].dispatcherPolicy.connect` and the first Agent options; `family:4` is absent today. The full network timeout still requires an IPv6-broken host to reproduce live.

## Solution Assessment

Yes, a narrow fix in `extensions/telegram/src/fetch.ts` with focused transport-policy tests is the best path. A global Node DNS/default-dispatcher change would be broader and riskier, especially with the related IPv6-working case.
