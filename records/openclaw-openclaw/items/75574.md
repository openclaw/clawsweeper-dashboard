## Summary

Keep open. Current main does thread Telegram network config into the grammY client and undici dispatcher, so the report’s suspected “not passed at all” root cause is not fully accurate, but the reported v2026.4.29 failure is fresh and the first dispatcher still does not force `family: 4` for the explicit IPv4-first config; IPv4-only behavior is currently reached only through fallback/timeouts.

## What This Changes

Request that Telegram `network.autoSelectFamily` and `network.dnsResultOrder` reliably affect grammY/undici Bot API calls on IPv6-broken hosts where IPv4 works.

## Best Possible Solution

Make Telegram’s first Bot API dispatcher match explicit IPv4-first operator intent instead of relying on timeout-driven fallback, while preserving per-account isolation, proxy handling, and the IPv6-friendly paths tracked by #41671.

## Reproduction Assessment

Unclear for the full live network failure because this checkout does not provide an IPv6-broken Oracle Cloud host. The narrower source-level mismatch is reproducible by a focused transport unit test: explicit `autoSelectFamily:false` plus `dnsResultOrder:ipv4first` currently creates a first dispatcher without `family:4`.

## Solution Assessment

Unclear, but the safest likely fix is narrow: adjust Telegram transport policy so explicit IPv4-first config selects an IPv4-only first dispatcher, then add regression tests around the grammY client/send/probe paths. A broad core DNS or global undici change would not be the best solution.
