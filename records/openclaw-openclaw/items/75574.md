## Summary

Keep open. Current main passes Telegram network config into the Telegram transport, but the first undici dispatcher still does not pin IPv4 when the operator sets `autoSelectFamily:false` with `dnsResultOrder:'ipv4first'`; IPv4 pinning only happens after an initial fallback-triggering network failure.

## What This Changes

Fix Telegram network transport so configured IPv4-first/auto-family settings affect grammY/undici requests before the first attempt on IPv6-broken hosts.

## Best Possible Solution

Update the Telegram-owned transport policy so explicit IPv4-first intent pins the initial dispatcher to IPv4 without changing global Node DNS/default-dispatcher state or weakening proxy and IPv6-friendly configurations.

## Reproduction Assessment

Yes. Source-level reproduction is to create a Telegram transport with `network:{autoSelectFamily:false,dnsResultOrder:'ipv4first'}` and inspect `dispatcherAttempts[0].dispatcherPolicy.connect`; current main lacks `family:4` on that first attempt.

## Solution Assessment

Yes. A narrow fix in `extensions/telegram/src/fetch.ts` with focused transport-policy tests is the maintainable path; a process-global DNS or default-dispatcher change would be broader and risk the mirror IPv6-working case.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/75574. The observable bug is that Telegram `channels.telegram.network` with `autoSelectFamily:false` and `dnsResultOrder:'ipv4first'` is read and passed into `resolveTelegramTransport`, but the first undici dispatcher is created with `forceIpv4:false`, so `connect.family` is absent until a fallback network error occurs. Implement a narrow Telegram-owned fix in `extensions/telegram/src/fetch.ts` so explicit IPv4-only/IPv4-first intent pins the initial dispatcher to IPv4, while preserving default happy-eyeballs behavior, explicit `verbatim`/IPv6-friendly configs, per-account isolation, proxy/env-proxy paths, fallback dispatchers, and #41671's mirror case. Add focused tests that assert `dispatcherAttempts[0].dispatcherPolicy.connect.family === 4` only for the intended explicit IPv4-only policy and that existing proxy/fallback behavior remains intact. Add a one-line `CHANGELOG.md` fix entry because this is user-facing. Do not change core global DNS defaults, process-global undici dispatchers, dependency versions, or unrelated Telegram behavior.
