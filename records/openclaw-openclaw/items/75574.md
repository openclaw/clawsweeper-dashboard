## Summary

Keep open. Current main still reads and passes Telegram network config into the Bot API transport, but the first undici dispatcher is not IPv4-pinned when the user explicitly sets `autoSelectFamily:false` with `dnsResultOrder:'ipv4first'`; IPv4-only pinning is only used after the sticky fallback path is reached.

## What This Changes

Telegram should apply `channels.telegram.network.autoSelectFamily=false` and `dnsResultOrder:'ipv4first'` to grammY/undici Bot API requests so IPv6-broken hosts use IPv4 on the initial request.

## Best Possible Solution

Update the Telegram transport to pin the initial dispatcher to IPv4 only for explicit IPv4-only config while preserving proxy modes, per-account isolation, sticky fallback, and IPv6-friendly configs.

## Reproduction Assessment

Yes for the source-level issue: create a Telegram transport with `network.autoSelectFamily=false` and `network.dnsResultOrder='ipv4first'`, then inspect the first `dispatcherAttempts` policy or Agent connect options; current main lacks `family: 4` on that first attempt. A full live reproduction still requires a host where IPv6 to `api.telegram.org` is unreachable and IPv4 works.

## Solution Assessment

Yes. The narrowest maintainable fix belongs in `extensions/telegram/src/fetch.ts` with focused transport-policy tests; changing global Node DNS settings or undici's global dispatcher would be broader and riskier.
