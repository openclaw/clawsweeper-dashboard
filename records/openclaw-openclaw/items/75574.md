## Summary

Keep open. Current main still accepts and routes Telegram network config into the grammY/undici fetch path, but the initial dispatcher is not IPv4-pinned for explicit IPv4-first config, so the reported IPv6-broken/IPv4-working timeout remains actionable.

## What This Changes

Telegram should honor explicit `channels.telegram.network.autoSelectFamily=false` plus `dnsResultOrder=ipv4first` for Bot API requests on IPv6-broken hosts.

## Best Possible Solution

Pin the initial Telegram dispatcher to IPv4 only for explicit IPv4-first operator config while preserving proxy handling, per-account isolation, sticky fallback, and IPv6-friendly configurations.

## Reproduction Assessment

Yes for a source-level reproduction: resolve Telegram transport with `network.autoSelectFamily=false` and `network.dnsResultOrder='ipv4first'`, then inspect the first dispatcher policy/options. A full live reproduction still requires a host where IPv6 to `api.telegram.org` is unreachable and IPv4 works.

## Solution Assessment

Yes. The narrowest maintainable fix is local to the Telegram transport resolver and its dispatcher-policy tests; changing global Node DNS settings or the undici global dispatcher would be broader and riskier.
