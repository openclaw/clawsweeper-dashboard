## Summary

Keep open. Current main does pass Telegram network config into the grammY/undici transport path, so the issue's suspected root cause is only partly right, but the requested IPv4-first behavior is still not implemented on the first dispatcher: explicit `autoSelectFamily:false` plus `dnsResultOrder:ipv4first` only disables auto family selection and installs an ordered lookup, while `family:4` is reserved for the later sticky fallback path.

## What This Changes

The issue requests that Telegram `channels.telegram.network` settings make grammY/undici Bot API requests use the intended IPv4-first transport instead of timing out on hosts with broken IPv6 egress.

## Best Possible Solution

Make Telegram's initial Bot API dispatcher honor explicit IPv4-only operator intent, while leaving default Node 22 behavior, proxy handling, per-account transport isolation, and IPv6-friendly configurations intact. Add focused regression coverage that distinguishes first-attempt IPv4 pinning from the existing sticky fallback path.

## Reproduction Assessment

Unclear for the full live timeout because this review did not have an IPv6-broken Oracle Cloud host. The narrower source-level bug is reproducible with a focused transport assertion: `autoSelectFamily:false` plus `dnsResultOrder:'ipv4first'` currently creates the first dispatcher without `family:4`.

## Solution Assessment

Yes for a narrow Telegram transport repair. The maintainable fix should live in `extensions/telegram/src/fetch.ts` policy selection rather than changing core DNS defaults or global undici/Node process behavior.
