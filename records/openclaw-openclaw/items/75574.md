## Summary

Keep open. Current main routes Telegram network config into the grammY/undici fetch path, but the first direct dispatcher still uses `forceIpv4: false`; explicit IPv4-only config is not pinned to `family: 4` until fallback, so the reported IPv6-broken/IPv4-working timeout remains a valid narrow Telegram plugin bug.

## What This Changes

The issue asks for Telegram network config to force IPv4 on the actual grammY/undici Bot API transport so IPv6-broken hosts do not time out.

## Best Possible Solution

Pin the initial Telegram dispatcher to IPv4 only for explicit IPv4-only configuration while preserving proxy modes, per-account isolation, sticky fallback, and IPv6-friendly configs.

## Reproduction Assessment

Yes for a source-level reproduction: resolve Telegram transport with `network.autoSelectFamily=false` and `network.dnsResultOrder='ipv4first'`, then inspect the first direct dispatcher policy; current main lacks `connect.family = 4` there. A full live reproduction still needs a host where IPv6 to `api.telegram.org` is unreachable and IPv4 works.

## Solution Assessment

Yes. The narrow maintainable fix is local to `extensions/telegram/src/fetch.ts` and focused dispatcher-policy tests; changing global Node DNS settings or adding another config switch would be broader than this bug requires.
