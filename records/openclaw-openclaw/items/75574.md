## Summary

Keep open. Current main still routes Telegram config into the Bot API fetch path, but the initial undici dispatcher for explicit IPv4-first/no-auto config is not pinned to IPv4; IPv4-only behavior is only available after fallback, so the reported timeout shape remains valid.

## What This Changes

The issue requests that Telegram Bot API transport apply `channels.telegram.network` IPv4-first/no-auto settings to grammY/undici so requests avoid unreachable IPv6 paths.

## Best Possible Solution

Pin the initial direct Telegram dispatcher to IPv4 only for explicit IPv4-first/no-auto configuration, while preserving proxy modes, sticky fallback, per-account isolation, and IPv6-friendly configs.

## Reproduction Assessment

Yes for a source-level reproduction: resolve Telegram transport with `network.autoSelectFamily=false` and `network.dnsResultOrder="ipv4first"`, then inspect that the first dispatcher lacks `family: 4`. A full live repro needs an IPv6-broken/IPv4-working Telegram host.

## Solution Assessment

Yes. The narrowest maintainable fix is local to `extensions/telegram/src/fetch.ts` with focused dispatcher-policy tests; changing global Node DNS or undici dispatcher state would be broader and riskier.
