## Summary

Keep open. Current main passes Telegram network config into the Bot API transport, but the first grammY/undici dispatcher is still built with `forceIpv4:false`; `family:4` is only applied after the sticky fallback path is promoted, so the reported IPv4-only operator intent is not implemented for the initial request.

## What This Changes

Request that Telegram Bot API requests honor `channels.telegram.network.autoSelectFamily:false` and `dnsResultOrder:'ipv4first'` on IPv6-broken hosts instead of timing out through grammY/undici.

## Best Possible Solution

Make Telegram's initial dispatcher honor explicit IPv4-only operator intent while preserving proxy handling, per-account transport isolation, no process-global DNS mutation, and IPv6-friendly explicit configs.

## Reproduction Assessment

Yes for the source-level behavior: create a Telegram transport with `autoSelectFamily:false` and `dnsResultOrder:'ipv4first'`, then inspect `dispatcherAttempts[0].dispatcherPolicy.connect` and see that current main lacks `family:4`. The full live network timeout remains environment-dependent because this review did not run on an IPv6-broken host.

## Solution Assessment

Yes, a narrow Telegram transport policy change is the best path. The fix should live in `extensions/telegram/src/fetch.ts` with focused regression tests rather than changing global Node DNS state or core config migration.
