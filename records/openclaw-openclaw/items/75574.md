## Summary

Keep open. Current main threads Telegram network config into the grammY/undici transport, but the first Bot API dispatcher still does not pin IPv4 when the operator explicitly configures `autoSelectFamily:false` with `dnsResultOrder:ipv4first`; IPv4 pinning is only available after the sticky fallback path is promoted.

## What This Changes

The issue requests that Telegram Bot API requests honor `channels.telegram.network.dnsResultOrder` and `autoSelectFamily` on hosts where IPv6 resolution is unusable but IPv4 reaches `api.telegram.org`.

## Best Possible Solution

Make Telegram's initial Bot API dispatcher honor explicit IPv4-only operator intent while preserving default Node 22 behavior, proxy handling, per-account transport isolation, sticky fallback behavior, and IPv6-friendly configs.

## Reproduction Assessment

Unclear for a full live reproduction because this review did not have an IPv6-broken host. The narrower source-level reproduction is high-confidence: with `autoSelectFamily:false` and `dnsResultOrder:'ipv4first'`, current main builds the first dispatcher without `family:4`.

## Solution Assessment

Yes, the best fix is a narrow Telegram transport policy change plus focused regression tests. It should live in `extensions/telegram/src/fetch.ts` and tests, not in global Node DNS defaults or core config migration.
