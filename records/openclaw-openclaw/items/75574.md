## Summary

Keep open. Current main does thread Telegram network config into the grammY client and undici dispatcher, so the report's suspected root cause is only partly accurate, but explicit IPv4-first operator intent still is not honored on the first dispatcher: `family: 4` is only applied after fallback or forced fallback. The report is fresh, detailed, and narrow enough for a focused Telegram transport repair.

## What This Changes

The issue requests that Telegram `network.autoSelectFamily` and `network.dnsResultOrder` settings affect actual grammY/undici Telegram API requests so IPv6-broken hosts can use IPv4 without `/etc/hosts` pinning.

## Best Possible Solution

Make Telegram's first Bot API dispatcher honor explicit IPv4-first configuration without relying on timeout-driven fallback, while preserving per-account isolation, explicit proxy handling, env-proxy behavior, pinned fallback, and the IPv6-friendly paths tracked by #41671.

## Reproduction Assessment

Unclear for the full live network failure because this review did not have an IPv6-broken Oracle Cloud host. The narrower source-level mismatch is reproducible with a focused transport unit test that creates Telegram transport with `autoSelectFamily:false` plus `dnsResultOrder:ipv4first` and asserts the first dispatcher does not currently include `family:4`.

## Solution Assessment

Unclear for the exact network environment, but the safest likely solution is a narrow Telegram transport policy change plus regression tests. A broad core DNS, global undici dispatcher, or process-wide Node DNS mutation would not be the best fix for this issue.
