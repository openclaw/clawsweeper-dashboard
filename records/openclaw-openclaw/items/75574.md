## Summary

Keep open. Current main still routes Telegram network config into the grammY/undici path, but the primary dispatcher is not IPv4-pinned for the explicit IPv4-first configuration in this report, so the timeout remains a narrow actionable Telegram transport bug.

## What This Changes

The issue asks for Telegram `network.autoSelectFamily=false` plus `dnsResultOrder=ipv4first` to reliably force IPv4 for Bot API requests on IPv6-broken hosts.

## Best Possible Solution

Pin the initial Telegram dispatcher to IPv4 for explicit IPv4 operator intent while preserving proxies, per-account isolation, sticky fallback, and IPv6-friendly configurations.

## Reproduction Assessment

Yes, source-level: resolve Telegram transport with `network.autoSelectFamily=false` and `network.dnsResultOrder='ipv4first'`, then inspect the primary dispatcher policy and see it lacks `family: 4`. Full live proof still needs a host where IPv6 to `api.telegram.org` fails and IPv4 works.

## Solution Assessment

Yes. The narrowest maintainable fix is local to Telegram transport policy and regression tests; changing global Node DNS or undici defaults would be broader and riskier.
