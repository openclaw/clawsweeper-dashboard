## Summary

Keep open. Current main routes Telegram network config into the Bot API transport, probes, and lookup paths, but `autoSelectFamily:false` plus `dnsResultOrder:'ipv4first'` still does not pin the first undici dispatcher to IPv4; `family:4` is only applied after a fallback-triggering network failure.

## What This Changes

The issue asks for Telegram `channels.telegram.network` settings to be applied to grammY/undici requests so IPv6-broken hosts can force IPv4 behavior without `/etc/hosts` workarounds.

## Best Possible Solution

Implement a narrow Telegram transport policy change so explicit IPv4-first, auto-family-disabled config pins the initial direct/env-proxy dispatcher to IPv4 while preserving proxy behavior, per-account isolation, default happy-eyeballs behavior, and IPv6-friendly explicit configs.

## Reproduction Assessment

Yes for the source-level issue: create a Telegram transport with `autoSelectFamily:false` and `dnsResultOrder:'ipv4first'`, then inspect `dispatcherAttempts[0].dispatcherPolicy.connect`; `family:4` is absent on current main. The full timeout needs an IPv6-broken host for live reproduction.

## Solution Assessment

Yes, the best fix is local to `extensions/telegram/src/fetch.ts` plus focused transport-policy tests. A global Node DNS/default-dispatcher change would be broader and would risk regressing the IPv6-working mirror case in #41671.
