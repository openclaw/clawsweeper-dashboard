## Summary

Keep open. Current main routes Telegram network config into the Bot API transport, but the first undici dispatcher still does not pin IPv4 for the reported IPv4-only operator intent; `family: 4` is only applied on the sticky fallback dispatcher.

## What This Changes

The issue requests that Telegram `network.dnsResultOrder` and `network.autoSelectFamily` settings be honored by the grammY/undici transport so IPv6-broken hosts can force IPv4.

## Best Possible Solution

Update the Telegram transport policy so explicit IPv4-only config affects the initial dispatcher, with regression tests that preserve proxy handling, sticky fallback behavior, and IPv6-friendly explicit configs.

## Reproduction Assessment

Yes for the source-level bug: construct `resolveTelegramTransport` with `autoSelectFamily: false` and `dnsResultOrder: "ipv4first"`, then inspect that the first dispatcher policy lacks `family: 4`. The full live network timeout still needs an IPv6-broken host.

## Solution Assessment

Yes. A narrow change in `extensions/telegram/src/fetch.ts` with focused transport tests is the maintainable path; global Node DNS state or core config migration would be broader and riskier.
