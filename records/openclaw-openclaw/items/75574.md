## Summary

Keep open. Current main routes Telegram network config into the Bot API transport, but the first undici dispatcher still does not pin IPv4 when `autoSelectFamily:false` and `dnsResultOrder:'ipv4first'` express IPv4-only operator intent; `family:4` is only used after the fallback path is promoted.

## What This Changes

The issue asks for Telegram `channels.telegram.network` settings to be applied to grammY/undici API requests immediately so IPv6-broken hosts can use IPv4 without `/etc/hosts` workarounds.

## Best Possible Solution

Pin the initial Telegram dispatcher to IPv4 when explicit config requests IPv4-first with auto-family disabled, while preserving proxies, per-account transport isolation, default Happy Eyeballs behavior, and the inverse IPv6-working case tracked by #41671.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: construct a Telegram transport with `autoSelectFamily:false` and `dnsResultOrder:'ipv4first'`, inspect the first dispatcher policy/options, and observe that `connect.family` is absent before fallback promotion.

## Solution Assessment

Yes. A narrow change in `extensions/telegram/src/fetch.ts` plus focused transport-policy tests is the maintainable path; changing global Node DNS/default dispatchers would be broader and would risk unrelated channels or accounts.
