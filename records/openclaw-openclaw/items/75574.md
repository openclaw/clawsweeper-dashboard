## Summary

Keep open. Current main still recognizes and threads Telegram network config into the grammY/undici transport, but explicit IPv4-first operator intent does not pin the primary dispatcher to IPv4, so the reported timeout remains a narrow actionable Telegram transport bug.

## What This Changes

The issue requests that Telegram `network.autoSelectFamily=false` plus `network.dnsResultOrder=ipv4first` actually force IPv4 for grammY/undici Bot API requests on IPv6-broken hosts.

## Best Possible Solution

Pin the initial Telegram dispatcher to IPv4 for explicit IPv4 operator intent, preserve proxy and sticky-fallback behavior, and add focused regression coverage for the primary dispatcher policy.

## Reproduction Assessment

Yes, source-level: resolve Telegram transport with `network.autoSelectFamily=false` and `network.dnsResultOrder='ipv4first'`, then inspect the primary dispatcher policy and see it lacks `family: 4`. Full live proof still needs a host where IPv6 to `api.telegram.org` fails and IPv4 works.

## Solution Assessment

Yes. The narrowest maintainable fix belongs in Telegram transport policy and regression tests; changing global Node DNS, global undici defaults, or disabling fallback broadly would be riskier.
