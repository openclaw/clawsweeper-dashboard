## Summary

Keep open. Current main routes Telegram network config into the grammY/undici transport, but the initial dispatcher still does not pin IPv4; `family: 4` is only applied after the sticky fallback path is promoted.

## What This Changes

The issue requests that Telegram Bot API requests honor `channels.telegram.network.dnsResultOrder` and `autoSelectFamily` on IPv6-broken hosts by applying them to the real grammY/undici HTTP dispatcher.

## Best Possible Solution

Implement a narrow Telegram transport policy fix that pins the initial dispatcher to IPv4 for explicit IPv4-only intent while preserving proxies, per-account isolation, default happy-eyeballs behavior, and IPv6-friendly configs.

## Reproduction Assessment

Yes for the source-level issue: construct a Telegram transport with `autoSelectFamily:false` and `dnsResultOrder:'ipv4first'`, then inspect `dispatcherAttempts[0].dispatcherPolicy.connect` or the first Agent options; `family:4` is absent on current main. The live timeout path still needs an IPv6-broken host.

## Solution Assessment

Yes. The narrowest maintainable fix is in `extensions/telegram/src/fetch.ts` with focused transport-policy tests; a global Node DNS or dispatcher change would be broader and riskier.
