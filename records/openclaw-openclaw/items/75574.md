## Summary

Keep open. Current main does pass Telegram network config into the grammY/undici transport path, but the specific IPv4-only behavior requested here is still missing on the first dispatcher: explicit `autoSelectFamily:false` plus `dnsResultOrder:ipv4first` disables auto family selection and installs an ordered lookup, while `family:4` is still reserved for the later sticky fallback attempt.

## What This Changes

The issue requests that Telegram `channels.telegram.network.dnsResultOrder` and `autoSelectFamily` settings force Telegram Bot API requests onto IPv4-first or IPv4-only behavior on hosts with broken IPv6 egress.

## Best Possible Solution

Make Telegram's initial Bot API dispatcher honor explicit IPv4-only operator intent, probably when `autoSelectFamily:false` is paired with `dnsResultOrder:'ipv4first'`, while leaving default Node 22 behavior, proxy handling, per-account transport isolation, and explicit IPv6-friendly configurations intact.

## Reproduction Assessment

Unclear for the full live timeout because this review did not have an IPv6-broken host. The narrower source-level issue is high-confidence: current main builds the first dispatcher for `autoSelectFamily:false` plus `dnsResultOrder:'ipv4first'` without `family:4`.

## Solution Assessment

Yes, a narrow Telegram transport fix is the best path. The change should live in `extensions/telegram/src/fetch.ts` policy selection and focused regression tests, not in global Node DNS/undici defaults or core config migration.
