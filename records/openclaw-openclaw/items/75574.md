## Summary

Keep open. Current main passes Telegram network config into the Bot API transport, but the first undici dispatcher still only receives `autoSelectFamily:false` plus ordered lookup; `family:4` is reserved for fallback attempts, so the reported IPv4-only operator intent is not fully implemented on the initial request path.

## What This Changes

The issue asks Telegram `channels.telegram.network` settings to make grammY/undici Bot API requests use the configured IPv4-first behavior on hosts where IPv6 attempts time out.

## Best Possible Solution

Make Telegram's initial dispatcher honor explicit IPv4-only operator intent while preserving proxy handling, per-account transport isolation, sticky fallback behavior, and explicit IPv6-friendly configs.

## Reproduction Assessment

Yes for the source-level failure: create a Telegram transport with `autoSelectFamily:false` and `dnsResultOrder:'ipv4first'`, then inspect that the first dispatcher lacks `family:4`. The full live timeout remains unverified because this review did not run on an IPv6-broken host.

## Solution Assessment

Yes, a narrow Telegram transport policy change with focused regression tests is the best path. The fix should stay in the Telegram plugin transport rather than changing global Node DNS state or core config migration.
