## Summary

Keep open. Current main wires Telegram network config into the grammY/undici transport, but the first Bot API dispatcher still does not pin IPv4 for the reported IPv4-only operator intent; IPv4 pinning is only present on the later sticky fallback path.

## What This Changes

The issue requests that Telegram Bot API requests honor `channels.telegram.network.autoSelectFamily` and `dnsResultOrder` on IPv6-broken hosts so requests use IPv4 instead of timing out.

## Best Possible Solution

Update the Telegram transport policy so the initial Bot API dispatcher honors explicit IPv4-only operator intent, with regression coverage for the first dispatcher and guardrails that preserve proxy handling, per-account isolation, sticky fallback behavior, and IPv6-friendly configs.

## Reproduction Assessment

Yes for the narrow source-level behavior: construct the Telegram fetch transport with `autoSelectFamily:false` and `dnsResultOrder:'ipv4first'`, then inspect the first dispatcher policy and it lacks `family:4`. The full timeout requires an IPv6-broken host like the reporter's Oracle Cloud setup.

## Solution Assessment

Yes. The narrowest maintainable solution belongs in `extensions/telegram/src/fetch.ts` and its focused tests, not in global Node DNS defaults or core config migration.
