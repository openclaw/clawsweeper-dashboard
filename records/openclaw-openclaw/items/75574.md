## Summary

Keep open. Current main still routes Telegram network config into the Bot API transport, but the first undici dispatcher only disables `autoSelectFamily`; it does not pin IPv4 unless the sticky fallback path is reached after an initial network failure.

## What This Changes

Fix Telegram Bot API transport so `channels.telegram.network` IPv4-first settings are applied to grammY/undici requests on IPv6-broken hosts.

## Best Possible Solution

Implement a narrow Telegram transport policy change that pins the initial dispatcher to IPv4 only when the resolved config clearly requests IPv4-only behavior, while keeping proxies, per-account isolation, fallback behavior, and IPv6-friendly configs intact.

## Reproduction Assessment

Yes for the source-level issue: create a Telegram transport with `autoSelectFamily:false` and `dnsResultOrder:'ipv4first'`, then inspect the first dispatcher policy or Agent connect options; current main lacks `family:4` on that first attempt. A full live reproduction still requires a host where IPv6 to `api.telegram.org` is unreachable and IPv4 works.

## Solution Assessment

Yes. The narrowest maintainable fix belongs in `extensions/telegram/src/fetch.ts` with focused transport-policy tests; changing global Node DNS settings or undici's global dispatcher would be broader and riskier.

## Repair Work Prompt

Fix the Telegram network config bug reported in openclaw/openclaw#75574. Current main passes `channels.telegram.network` into `resolveTelegramTransport`, but the initial undici dispatcher uses `forceIpv4:false`, so `autoSelectFamily:false` plus `dnsResultOrder:'ipv4first'` does not set `connect.family = 4` until after a sticky fallback. Update `extensions/telegram/src/fetch.ts` so resolved IPv4-only operator intent pins the first dispatcher to IPv4, while preserving explicit proxies, env proxies, per-account transport isolation, default Node 22 happy-eyeballs behavior, sticky fallback, pinned fallback IP behavior, and IPv6-friendly configs such as `autoSelectFamily:true` or `dnsResultOrder:'verbatim'`. Add focused regression coverage in `extensions/telegram/src/fetch.test.ts`; consider `extensions/telegram/src/network-config.test.ts` only if resolver semantics change. Include a user-facing `CHANGELOG.md` fix entry. Do not change global Node DNS/default dispatcher behavior, unrelated Telegram config schema, or plugin/core boundaries. Pay attention to related mirror case #41671 so the fix does not globally force IPv4 after generic network errors.
