## Summary

Keep open. Current main still routes Telegram network config into the grammY/undici fetch path, but explicit IPv4-first operator config is not used to pin the initial dispatcher to IPv4; the related open PRs cover probe reuse or a new env-var workaround rather than this existing-config contract.

## What This Changes

The issue requests that Telegram `network.autoSelectFamily` and `network.dnsResultOrder` be applied to the actual grammY/undici HTTP transport so broken-IPv6 hosts use IPv4 without `/etc/hosts` pinning.

## Best Possible Solution

Pin the initial Telegram dispatcher to IPv4 only for explicit IPv4-first operator config while preserving proxy behavior, per-account isolation, sticky fallback recovery, and IPv6-friendly configurations.

## Reproduction Assessment

Yes, source-level: resolve Telegram transport with `network.autoSelectFamily=false` and `network.dnsResultOrder='ipv4first'`, then inspect the initial dispatcher policy and connect options. A full live reproduction still needs a host where IPv6 to `api.telegram.org` is unreachable and IPv4 works.

## Solution Assessment

Yes. The narrow maintainable fix is local to the Telegram transport resolver and dispatcher-policy tests; adding a new global dispatcher policy or a new env var is broader than honoring the existing channel config.
