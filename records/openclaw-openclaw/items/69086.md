## Label Justifications

- `P2`: This is a normal-priority fallback/session correctness issue with source-level evidence and limited blast radius.
- `impact:session-state`: The guard misclassifies transcript state by treating an old assistant response as an answer to the latest user turn.
- `impact:message-loss`: A misclassified fallback retry can make the agent continue from stale context instead of answering the fresh user turn.

## Summary

Keep open: current main and the latest release still use the broad first-assistant session-history guard, and the retry-prompt hook remains a maintainer product/API decision. The open linked PR is useful but only covers the guard half, so it is not a complete replacement for this issue.

## What This Changes

Request to narrow attempt-execution session-history detection to assistant replies after the latest user turn and add a supported fallback retry prompt customization hook.

## Best Possible Solution

Land or replace the bounded assistant-after-latest-user guard fix, then make an explicit maintainer decision on whether retry-prompt tuning belongs in a plugin hook, lifecycle event, config override, or no new core surface.

## Reproduction Assessment

Yes, source-level: a user-assistant-fresh-user JSONL transcript still returns true because current main stops at the first assistant record. I did not execute a failing test because this review kept the checkout read-only.

## Solution Assessment

Unclear as a full solution. The assistant-after-latest-user guard is the narrow bug fix, but exposing a retry-prompt hook or override needs maintainer product/API direction before implementation.
