Keep this open: current main and v2026.5.7 still have the broad session-history guard, there is no retry-prompt override surface, and the open linked PR only covers the guard portion.

## What This Changes

The issue requests scoping attempt-execution session-history detection to assistant messages after the latest user turn and adding a supported retry-continuation prompt override hook.

## Best Possible Solution

Land or replace a tail-aware latest-user guard with regression coverage, then split or explicitly decide the retry-prompt override as a default-preserving config/plugin/event surface.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence fixture: a JSONL transcript with user -> assistant -> fresh unanswered user still returns true at the earlier assistant record before the latest user turn can reset state.

## Solution Assessment

Unclear for the full issue. The guard-fix direction is narrow and source-backed, but the retry-prompt hook is a new extension/config surface and should be split or approved before the broader issue is closed.
