## Summary

Keep this issue open. Current main still has both reported gaps: the session-history scan returns on the first assistant record anywhere in the transcript, and the fallback retry marker is still hard-coded with no supported operator override; the related implementation PR was closed unmerged.

## What This Changes

The issue requests changing attempt-execution fallback retry handling so session history means assistant-after-latest-user, and adding a supported override hook for the retry continuation prompt.

## Best Possible Solution

Land a focused core fix that changes the transcript scan to track assistant-after-latest-user, adds the missing regression tests, and exposes a small documented operator override for the fallback retry marker while preserving the current default prepend behavior.

## Reproduction Assessment

Yes. A JSONL transcript shaped user -> assistant -> fresh user reproduces the guard bug by returning `true` at the earlier assistant record before the later user turn is considered; current tests do not cover that shape.

## Solution Assessment

Yes. The narrow maintainable solution is to keep the existing bounded JSONL parser but track/reset state by message role, and to add a config-backed retry marker override rather than a broader plugin/event hook unless maintainers want a larger API.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/69086. Current main still treats any assistant record in an attempt-execution JSONL session file as session history for fallback retries, and the fallback retry marker is hard-coded. Update `src/agents/command/attempt-execution.helpers.ts` so the bounded JSONL scan tracks whether an assistant message appears after the latest user message, resetting the flag on each user message and returning that final state. Apply the same semantics to Claude CLI transcript history because it delegates to the same helper. Add regression coverage for user -> assistant -> fresh user returning false and user -> assistant -> user -> assistant returning true. Expose a small supported operator override for the fallback retry marker, preferably config-backed under `agents.defaults` with type/schema/help/docs wiring, while preserving the current default marker and #65760 behavior of prepending retry context to the original body rather than replacing the body. Do not patch built `dist/` assets, do not add plugin-specific behavior, and avoid broad fallback-runner refactors. Add or update user-facing docs under model failover/config docs and include a `CHANGELOG.md` fix entry crediting the reporter (`Thanks @kenners22`) and prior-art PR if appropriate.
