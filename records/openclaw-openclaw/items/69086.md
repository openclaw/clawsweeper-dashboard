## Summary

Current main still has the broad session-history guard and no supported retry-prompt override; the issue should stay open because the linked open PR only addresses the guard portion and the hook/API surface still needs maintainer judgment.

## What This Changes

The issue asks to make fallback retry history detection depend on an assistant message after the latest user message and to expose a supported retry continuation prompt override.

## Best Possible Solution

Land or replace the guard fix with tail-aware regression coverage, then decide whether the retry prompt should become a config setting, plugin hook, or runtime callback and split that surface if needed.

## Reproduction Assessment

Yes for the guard bug by source inspection: a user -> assistant -> fresh unanswered user transcript will still return true at the earlier assistant record. The retry-prompt override is a feature/API request rather than a reproduction question.

## Solution Assessment

Unclear for the full issue. The guard-fix direction is narrow, but the open PR is partial and the retry-prompt override needs a supported extension-surface decision before this can close.
