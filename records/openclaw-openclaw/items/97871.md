## Summary

Keep open: maintainer follow-up narrowed this from a blanket Ollama/LM Studio failure to a real OpenClaw-side local OpenAI-compatible stream case where a 200 SSE response opens but no first event arrives; current main and the latest release still do not show a bounded first-event diagnostic for the completions local-agent path.

## What This Changes

The issue reports that `openclaw agent --local` hangs for local providers even though the lean local model probe succeeds with the same provider setup.

## Best Possible Solution

Add a bounded first-event or no-first-token diagnostic for local OpenAI-compatible agent streams while preserving long inter-token silence for slow local prompt evaluation, then validate it against both a synthetic stalled SSE server and real Ollama/LM Studio setups.

## Reproduction Assessment

Partially: we have a high-confidence source and maintainer-described reproduction for the narrowed no-first-SSE-event mechanism, but not for the reporter's exact real Ollama and LM Studio setups. The key path is local-provider idle timeout resolving to disabled while the completions stream lacks a first-event guard.

## Solution Assessment

Unclear until maintainers choose the timeout policy. A narrow first-event timeout/diagnostic looks better than re-enabling a general local idle watchdog, because slow local providers can legitimately stay silent during prompt evaluation.
