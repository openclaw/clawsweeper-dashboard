## Summary

Keep this issue open: current main and the latest release still use presence-only abort-marker checks, and the open linked fix PR already owns the focused repair. Closing before that PR merges would race the normal issue-to-PR lifecycle.

## What This Changes

The issue asks Gateway chat projection to ignore stale retained abort markers when a fresh run reuses the same client run/idempotency key.

## Best Possible Solution

Review and land or replace the linked Gateway PR so abort suppression uses ordered marker freshness while preserving current active-abort behavior.

## Reproduction Assessment

Yes. Source inspection shows a high-confidence path: seed `chatRunState.abortedRuns`, register a same-key fresh run, then deliver assistant/lifecycle events; current main checks only marker presence before suppressing projection.

## Solution Assessment

Yes. The linked PR's direction is the narrow maintainable fix: compare abort marker freshness against the active chat run registration in Gateway state/projection instead of clearing retained markers early.
