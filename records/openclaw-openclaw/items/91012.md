## Summary

Keep this issue open: current `main` and the latest release still use presence-only abort-marker suppression, while the focused linked fix PR is open and referenced with closing syntax.

## What This Changes

The issue reports that retained Gateway chat abort markers keyed by a reused run/idempotency id can suppress fresh chat delta/final events.

## Best Possible Solution

Review and land or replace the focused linked PR so Gateway chat projection ignores stale abort markers while preserving current-run abort suppression.

## Reproduction Assessment

Yes, source-reproducible: seed `chatRunState.abortedRuns`, register a same-key fresh run, then deliver assistant delta and lifecycle end events. Current main's presence-only `Map.has` check suppresses projection before any freshness comparison; I did not execute tests because this review is read-only.

## Solution Assessment

Yes, the linked PR direction targets the right owner boundary by adding freshness ordering to Gateway chat run/abort state instead of clearing markers early. The issue should remain open until that PR is reviewed and merged or replaced.
