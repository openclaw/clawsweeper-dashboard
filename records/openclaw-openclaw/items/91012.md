- `impact:session-state`: The bug comes from retained Gateway abort state being interpreted as current state for a later same-key run.

## Summary

Keep this open: current main and the latest release still use presence-only Gateway chat abort-marker checks, and https://github.com/openclaw/openclaw/pull/91013 is an open closing PR rather than a merged fix.

## What This Changes

The issue reports that retained Gateway chat abort markers can suppress delta/final events for a fresh chat run that reuses the same client run or idempotency key.

## Best Possible Solution

Review and land or replace the linked PR so Gateway abort suppression compares marker freshness against the active chat run registration while preserving current-run abort behavior.

## Reproduction Assessment

Yes. From source inspection, seeding `chatRunState.abortedRuns`, registering a same-key fresh run, and sending assistant/lifecycle events reaches a presence-only `Map.has` abort check with no freshness comparison; I did not run tests because this review is read-only.

## Solution Assessment

Yes. The best path is the linked PR's owner-boundary fix: keep abort markers but order them against chat run registration freshness instead of clearing retained markers early or adding a second runtime fallback.
