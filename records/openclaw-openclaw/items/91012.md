## Summary

Keep this issue open because current main and the latest release still use presence-only abort-marker suppression, while the focused fix at https://github.com/openclaw/openclaw/pull/91013 is open and unmerged.

## What This Changes

The issue reports that a retained Gateway chat abort marker for a reused client run/idempotency key can suppress delta/final projection for a later fresh chat run.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/91013 if its ordered abort-marker freshness contract is accepted, or close it and replace it with an equivalent Gateway-owned freshness check plus regression coverage.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: seed an older abortedRuns marker, register a same-key fresh run, then deliver assistant and lifecycle events; current main's Map.has checks suppress projection before any freshness comparison. I did not run tests because this review is read-only.

## Solution Assessment

Yes, the linked fix direction is the right owner boundary: Gateway chat state should compare abort marker freshness against the active run registration while preserving current-run abort suppression. The issue should stay open until that PR or an equivalent fix is merged.
