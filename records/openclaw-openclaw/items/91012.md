## Summary

Keep this issue open: current main and the latest release still use presence-only abort-marker checks, and the open linked PR is the active fix path that closes this issue once merged.

## What This Changes

The issue asks for stale retained chat abort markers to stop suppressing fresh same-key chat delta/final events while preserving active-run abort suppression.

## Best Possible Solution

Review and land or replace the focused linked PR so Gateway compares abort markers against active chat-run registration freshness while keeping current-run abort suppression intact.

## Reproduction Assessment

Yes, at source level: seed `chatRunState.abortedRuns`, register a fresh same-key run, then deliver assistant and lifecycle events; current main's presence-only `Map.has` checks suppress projection before any freshness comparison. I did not execute tests because this review is read-only.

## Solution Assessment

Yes, the active fix direction is the maintainable one: keep the Gateway abort marker contract but make projection suppression depend on marker freshness relative to the active chat run. The issue should remain open until the linked PR is merged or superseded by an equivalent fix.
