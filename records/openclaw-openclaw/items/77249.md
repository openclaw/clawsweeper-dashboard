## Summary

Keep open: current main and v2026.6.10 improved ordinary Slack reconnects, but they still do not add an independent no-event liveness path for the reported zombie WSS state.

## What This Changes

The issue requests Slack Socket Mode recovery and observable logging when the WebSocket becomes stale without emitting disconnect/error events.

## Best Possible Solution

Keep this open until Slack publishes a real transport-liveness signal or maintainer-approved watchdog that can restart the channel without false positives on quiet workspaces.

## Reproduction Assessment

No deterministic live reproduction is available, but the source-level blind spot is clear: if Slack emits no watched socket event and Slack does not publish transport activity, the monitor can remain healthy while waiting indefinitely.

## Solution Assessment

Unclear as a product choice: the narrow code area is Slack transport health, but maintainers need to choose between SDK lifecycle instrumentation, a message-flow watchdog, or an API probe based on live proof and false-positive tolerance.
