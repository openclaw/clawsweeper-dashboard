## Label Justifications

- `P1`: The issue describes a broken Slack channel workflow affecting real users, with the process appearing healthy while Slack messages receive no response until manual restart.
- `impact:message-loss`: The reported failure mode silently suppresses inbound Slack messages because the Socket Mode channel can stay wedged without delivery or logs.

## Summary

Keep open. Current main and v2026.6.6 still let the Slack Socket Mode supervisor wait indefinitely on the disconnect waiter when no SDK disconnect/error/start-failure event arrives; native reconnect and the open reconnect-cap PR improve adjacent paths but do not prove recovery from the no-event zombie WSS case. Prior Slack health work also means the fix should use transport/SDK lifecycle evidence, not a naive inbound-message inactivity timer.

## What This Changes

The issue requests a Slack Socket Mode liveness watchdog or equivalent reconnect path so a zombie WebSocket cannot leave the Slack channel silently wedged.

## Best Possible Solution

Add Slack-owned transport liveness/recovery based on SDK lifecycle or keepalive evidence, preserve the quiet-workspace health semantics from https://github.com/openclaw/openclaw/pull/69833, and coordinate with https://github.com/openclaw/openclaw/pull/73162 rather than using app message volume as transport proof.

## Reproduction Assessment

Yes from source inspection, but not from a deterministic live repro. Current main still has an event-only disconnect waiter that the provider awaits without an idle deadline, matching the reported no-event hang if the Slack SDK never emits a disconnect/start-failure/error event.

## Solution Assessment

Unclear for the proposed message-flow watchdog. The issue is valid, but the best fix should be transport-lifecycle based because using inbound app-event silence as socket liveness was previously reverted for quiet Slack workspaces.
