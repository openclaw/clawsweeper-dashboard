## Summary

Keep open: current main still has an event-only Slack Socket Mode reconnect waiter, and the existing channel health monitor cannot recover this no-event path because Slack does not publish transport activity into the runtime snapshot.

## What This Changes

Request a Slack Socket Mode liveness fallback so a no-event zombie WSS cannot leave the OpenClaw reconnect supervisor waiting indefinitely and silently.

## Best Possible Solution

Add a Slack-owned transport-liveness fallback or bounded health probe that feeds the existing stale-socket health policy and forces reconnect, with focused reconnect tests and a changelog entry.

## Reproduction Assessment

No deterministic live reproduction is available. Source inspection gives a high-confidence conditional path: if the Slack SDK emits none of the watched disconnect/error/start-failure events after start, the supervisor awaits forever.

## Solution Assessment

Unclear: the reported message-flow watchdog would close the hang, but current docs and health policy deliberately avoid treating app events as transport liveness. A Slack transport signal or cheap probe tied into `lastTransportActivityAt` is the safer maintainable direction.
