## Summary

Keep open: current main still has an event-only Slack Socket Mode disconnect waiter with Bolt auto-reconnect disabled, so the reported no-event zombie WSS path can leave the Slack monitor awaiting forever. Related Slack ping/pong tuning and event-loop reports provide useful context, but they do not supersede this specific unbounded waiter gap.

## What This Changes

The issue asks OpenClaw to add a bounded Slack Socket Mode liveness fallback so zombie WSS sessions reconnect or log instead of waiting forever.

## Best Possible Solution

Add a Slack-owned bounded transport-health fallback, preferably SDK/socket-activity or cheap-probe backed rather than plain app-event silence, with regression coverage, docs, and changelog updates.

## Reproduction Assessment

No deterministic current-main reproduction is available. Source inspection gives a high-confidence conditional path: if the Slack SDK emits none of `disconnected`, `unable_to_socket_mode_start`, `error`, or abort after start, the supervisor awaits forever.

## Solution Assessment

Unclear: the requested message-flow watchdog would close the gap, but current docs explicitly avoid treating app events as transport liveness. A transport- or probe-backed fallback with an owner-chosen timeout/config is safer.
