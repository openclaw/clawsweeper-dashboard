## Summary

Keep open: current main improved Slack recovery by re-enabling native Socket Mode reconnect, but the no-event zombie path is still not bounded by OpenClaw; the waiter remains event-only and Slack does not publish transport activity/probe status for health recovery.

## What This Changes

Add Slack Socket Mode liveness recovery so a zombie WSS with no SDK disconnect/error event logs and reconnects instead of requiring a process restart.

## Best Possible Solution

Add a Slack-owned bounded transport-health fallback, preferably from SDK lifecycle/heartbeat evidence or a cheap probe, that logs and reconnects when Socket Mode makes no progress without reclassifying quiet app-event traffic as transport failure by default.

## Reproduction Assessment

Yes for a source-level conditional path, but not for deterministic live reproduction. Current main awaits an event-only `disconnectWaiter.promise` with no bound, so if Slack emits none of the watched events and native reconnect does not recover, the supervisor remains pending.

## Solution Assessment

Unclear. The proposed message-flow watchdog would close the gap, but it conflicts with the current Slack health model; a transport- or probe-backed fallback with an owner-chosen timeout/config surface is safer.
