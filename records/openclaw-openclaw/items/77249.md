## Label Justifications

- `P1`: The issue describes a broken Slack channel workflow where the agent becomes unreachable until operator restart.
- `impact:crash-loop`: The Slack provider can hang in a running gateway and require a manual process restart to recover availability.
- `impact:message-loss`: Slack mentions and inbound messages can be silently ignored while the channel appears healthy.

## Summary

Keep open: current main and the latest release still park the Slack Socket Mode supervisor on SDK disconnect/error events only, while Slack does not publish a separate OpenClaw transport-liveness timestamp that gateway health can use for this no-event zombie path.

## What This Changes

The issue reports Slack Socket Mode reconnect supervision hanging indefinitely when the websocket becomes stale without emitting disconnect/error events.

## Best Possible Solution

Keep this as the canonical Slack Socket Mode liveness bug and add Slack-owned stale-transport detection that marks the channel unhealthy and forces a bounded reconnect when the SDK never emits a disconnect/error event.

## Reproduction Assessment

No for a high-confidence live repro in this review; yes at source level for the missing guard. Current main still has an event-only disconnect waiter, and the reported zombie WSS condition is specifically the case where none of those events arrive.

## Solution Assessment

The issue direction is valid, but the exact proposed message-flow watchdog is not clearly the best fix by itself because quiet workspaces are normal. A transport activity signal, SDK heartbeat observation, or explicit health probe should be chosen before implementation.
