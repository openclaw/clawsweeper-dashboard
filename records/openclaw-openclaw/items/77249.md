## Label Justifications

- `P1`: The issue describes a real Slack channel outage where inbound messages stop while the gateway appears healthy and requires operator restart.
- `impact:message-loss`: The reported failure suppresses Slack replies and mentions until restart, so user messages can be silently missed.
- `impact:crash-loop`: The affected class is process availability and hang recovery: the service remains active but the Slack channel is wedged until manual restart.

## Summary

Keep open: current main still waits for Slack Socket Mode SDK events or abort, and the gateway health policy still will not mark a connected Slack account stale without explicit transport activity, so the reported no-event zombie WSS gap remains credible and unresolved.

## What This Changes

The issue requests Slack Socket Mode liveness recovery when the reconnect supervisor hangs indefinitely on a stalled WebSocket that emits no SDK disconnect/error event.

## Best Possible Solution

Add a Slack-owned transport-liveness signal or probe-backed reconnect fallback that feeds the existing health monitor, with live current-main proof and without treating quiet app traffic alone as socket failure.

## Reproduction Assessment

No. The source path is clear and the production observations are credible, but I did not establish a deterministic live current-main Slack reproduction for a no-event zombie WSS state.

## Solution Assessment

Unclear. A message-flow watchdog is plausible, but current health policy deliberately avoids treating app-event silence as transport failure, so maintainers should choose between transport activity, probe-backed recovery, or another Slack-owned liveness signal.
