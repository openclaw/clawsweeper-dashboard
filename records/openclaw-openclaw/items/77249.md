## Summary

Keep open: current main still has a Slack Socket Mode reconnect supervisor that can wait forever if the Slack socket client emits no disconnect/error/start-failure event, and the existing health monitor intentionally does not treat Slack app-event silence as transport liveness.

## What This Changes

The issue asks OpenClaw to recover Slack Socket Mode when a zombie WSS leaves the reconnect waiter pending with no emitted event or log.

## Best Possible Solution

Add a Slack-owned bounded transport-health fallback, preferably by publishing real Slack SDK/socket transport activity or using a cheap probe, with regression coverage for the no-event waiter path and docs/changelog updates.

## Reproduction Assessment

No deterministic live reproduction is available. Source inspection gives a high-confidence conditional path: if the Slack SDK emits none of `disconnected`, `unable_to_socket_mode_start`, or `error` after startup, the supervisor awaits forever.

## Solution Assessment

Unclear: the requested message-flow watchdog would close the gap, but it conflicts with the current app-event versus transport-liveness policy and may false-positive. A transport-backed timestamp or bounded probe is the safer maintainable direction.
