## Summary

Keep open: current main still has an event-only Slack Socket Mode disconnect waiter, and the Slack SDK pong-timeout warning is filtered before OpenClaw emits its own reconnect signal.

## What This Changes

The issue asks OpenClaw’s Slack Socket Mode supervisor to detect and recover from a stalled WSS connection that never emits the watched disconnect/error events.

## Best Possible Solution

Add Slack-owned transport recovery that turns a stalled SDK ping/pong close path into a bounded reconnect with one diagnostic log, without treating ordinary quiet app-event periods as transport failure.

## Reproduction Assessment

Yes from source inspection: if the Slack socket client emits none of the watched disconnect/error events, the current waiter never resolves and the reconnect loop cannot log or recover. I did not live-reproduce the AWS/NLB zombie condition.

## Solution Assessment

Yes, a narrow Slack plugin fix around the SDK ping/pong warning or close-handshake timeout is the best direction; a blanket message-flow watchdog is less safe because current docs explicitly say app events are not transport liveness.

## Repair Work Prompt

Fix the Slack Socket Mode reconnect hang reported in openclaw/openclaw#77249. Keep the change inside the Slack plugin unless a small generic helper reuse is clearly needed. Do not add a default message-flow idle watchdog that reconnects quiet but healthy workspaces; current docs say app events are not transport liveness. Instead, make the existing SDK ping/pong timeout or close-handshake path bounded: when the Slack SDK reports a pong/server-ping timeout or the socket enters a close path without a later disconnected/error event, emit one OpenClaw diagnostic and force the existing reconnect loop to proceed through gracefulStop/backoff. Add focused regression coverage for an eventless waiter/close-handshake stall, update Slack docs only if a new user-visible config or behavior note is needed, and add a CHANGELOG.md fix entry because this is user-facing Slack reliability work. Do not touch unrelated channel/plugin behavior or dependency versions.
