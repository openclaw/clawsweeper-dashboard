## Label Justifications

- `P1`: The issue describes a real Slack channel outage where inbound messages stop while the gateway appears healthy and requires operator restart.
- `impact:message-loss`: The reported failure suppresses Slack replies and mentions until restart, so user messages can be silently missed.
- `impact:crash-loop`: The affected class is process availability and hang recovery: the service remains active but the Slack channel is wedged until a manual restart.

## Summary

Keep open: current main and v2026.6.8 still do not publish Slack transport activity or add an independent no-event watchdog, so native SDK reconnects cover emitted ping/pong failures but not the reported zombie WSS path where no SDK event wakes OpenClaw.

## What This Changes

The issue asks OpenClaw to recover and log when Slack Socket Mode becomes silently stale without emitting disconnect or error events, instead of requiring a manual process restart.

## Best Possible Solution

Add a Slack-owned transport-liveness signal or probe-backed reconnect fallback that feeds the existing health monitor, with live current-main proof and without treating quiet app traffic alone as socket failure.

## Reproduction Assessment

No. The issue has detailed production observations and a source-reproducible no-event waiter path, but I did not establish a deterministic live current-main Slack reproduction.

## Solution Assessment

Unclear. The preferred message-flow watchdog is not clearly the best fix because current health policy deliberately avoids treating app-event silence as transport failure; a transport-activity signal or probe-backed fallback needs maintainer signoff.
