## Summary

Keep open: current main improves ordinary Slack Socket Mode reconnects, but it still has no independent no-event liveness path for the reported zombie WSS state, and v2026.6.8 has the same blind spot.

## What This Changes

The issue reports a Slack Socket Mode reconnect supervisor hang where a stale WebSocket emits no disconnect/error event, leaving Slack unresponsive until the OpenClaw process is manually restarted.

## Best Possible Solution

Add a Slack-owned transport-liveness signal or probe-backed reconnect fallback that feeds the existing health monitor, with live current-main proof and without treating quiet app traffic alone as socket failure.

## Reproduction Assessment

No. The issue has detailed production observations and a source-reproducible no-event waiter path, but I did not establish a deterministic live current-main Slack reproduction.

## Solution Assessment

Unclear. The preferred message-flow watchdog is not clearly the best fix because current health policy deliberately avoids treating app-event silence as transport failure; a transport-activity signal or probe-backed fallback needs maintainer signoff.
