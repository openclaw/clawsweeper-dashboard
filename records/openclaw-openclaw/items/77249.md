## Summary

Keep open: current main and v2026.6.6 still lack an independent Slack Socket Mode transport-liveness signal for the no-event zombie websocket path; related work covers ping timeouts or reconnect caps but not this central failure.

## What This Changes

The issue asks for Slack Socket Mode to detect and recover from a zombie websocket/no-event state instead of hanging silently until manual restart.

## Best Possible Solution

Keep this as the canonical Slack Socket Mode no-event liveness bug and add a Slack-owned transport-liveness or probe-backed fallback that marks the channel unhealthy and forces a bounded reconnect without treating normal workspace quietness as failure.

## Reproduction Assessment

No for a high-confidence live reproduction in this review; yes at source level for the missing guard. Current main still has an event-only disconnect waiter, and the reported zombie WSS condition is specifically the case where none of those events arrive.

## Solution Assessment

Unclear. The issue direction is valid, but the exact message-flow watchdog proposal is not clearly the best fix because quiet Slack workspaces are normal; maintainers should choose a transport activity signal, SDK heartbeat observation, or explicit health probe before implementation.
