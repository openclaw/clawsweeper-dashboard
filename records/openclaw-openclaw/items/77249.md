## Summary

Keep open: current main still has a source-visible no-event Slack Socket Mode waiter path, and the newer macOS report says the same silent wedge still occurs on v2026.5.7. Adjacent timeout and reconnect-cap work does not close the specific no-emitter-event gap.

## What This Changes

The issue asks OpenClaw to recover Slack Socket Mode when the WebSocket/reconnect supervisor goes silent without a disconnect/error event and otherwise requires a manual gateway restart.

## Best Possible Solution

Add a Slack-owned bounded transport-health fallback, preferably provider-proven socket activity or a cheap probe, with synthetic no-event waiter coverage and matching docs/changelog updates.

## Reproduction Assessment

No deterministic live reproduction is available. Source inspection gives a high-confidence conditional path: if Slack/Bolt emits none of the watched disconnect/start-failure/error events after start, the supervisor awaits indefinitely while status can remain healthy.

## Solution Assessment

Unclear. The proposed message-flow watchdog would recover from the reported silence, but current Slack docs intentionally avoid treating app events as transport liveness; a transport- or probe-backed fallback with an owner-chosen timeout is safer.
