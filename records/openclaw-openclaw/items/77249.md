## Summary

Keep open: current main and v2026.6.10 still lack an independent Slack Socket Mode no-event liveness path. The reconnect loop still waits on SDK events, while the gateway stale-socket monitor requires `lastTransportActivityAt` and Slack does not currently publish that transport signal.

## What This Changes

The issue asks OpenClaw to detect and recover from Slack Socket Mode websocket stalls that emit no SDK event or log instead of wedging until manual restart.

## Best Possible Solution

Add a Slack-owned transport liveness signal or probe-backed reconnect fallback that feeds the existing health monitor, with live proof for the zombie WSS case and an explicit false-positive policy.

## Reproduction Assessment

No high-confidence deterministic live reproduction is available. The issue has two production observations and a source-inspectable waiter path that can hang if Slack emits no disconnect/error/start-failure event.

## Solution Assessment

Unclear. The requested watchdog targets the right owner area, but maintainers should choose between transport activity, event-silence, or probe-backed recovery because each has different false-positive and API-traffic behavior.
