## Summary

Keep open: current main still has the event-only Slack Socket Mode waiter and no Slack-published transport-liveness timestamp, while the issue now has two real-environment reports of the same silent wedge after reconnect work. The remaining fix needs Slack owner judgment on the liveness signal rather than cleanup closure.

## What This Changes

The issue asks OpenClaw to recover automatically when Slack Socket Mode enters a silent no-event zombie state instead of waiting indefinitely for SDK disconnect/error events.

## Best Possible Solution

Add a Slack-owned bounded transport-health fallback, preferably by feeding real Socket Mode heartbeat/ping/pong activity into `lastTransportActivityAt` or using a cheap probe, with synthetic no-event waiter coverage and updated Slack docs/changelog.

## Reproduction Assessment

No deterministic live reproduction is available from this review. Source inspection gives a high-confidence conditional path: if Slack's SDK never emits `disconnected`, `unable_to_socket_mode_start`, or `error` after startup, OpenClaw awaits the disconnect waiter indefinitely.

## Solution Assessment

Unclear. The requested message-flow watchdog would wake the supervisor, but the current docs and health policy deliberately avoid treating app-event silence as transport failure; a transport-activity or probe-backed fallback is the narrower maintainable direction unless owners choose a policy change.
