## Summary

Keep open: current main and the latest release still have a Slack Socket Mode waiter that can await forever if the Slack SDK emits no disconnect/error/start-failure event, and a later report was closed back to this issue as the canonical liveness problem.

## What This Changes

The issue asks OpenClaw to detect and recover from Slack Socket Mode zombie WebSocket states that emit no SDK event or log and otherwise require manual restart.

## Best Possible Solution

Add a Slack-owned bounded transport-liveness fallback, preferably by publishing real Socket Mode heartbeat/ping/pong activity or a cheap probe into channel health, then restart/log on stale transport with focused synthetic coverage.

## Reproduction Assessment

No deterministic live reproduction is available. Source inspection gives a high-confidence conditional path: if Slack's SDK never emits the watched events and no abort fires, OpenClaw awaits the disconnect waiter indefinitely.

## Solution Assessment

Unclear: the requested message-flow watchdog would wake the supervisor, but it treats quiet app traffic as socket failure. A transport-activity or probe-backed fallback is the narrower maintainable direction unless owners intentionally change that policy.
