## Summary

Keep open: current main still has an event-only Slack Socket Mode disconnect waiter and an unbounded await, so a no-event zombie WSS path can still wedge the Slack channel without reaching the reconnect logging path.

## What This Changes

The issue requests Slack Socket Mode recovery and logging when the reconnect supervisor sees a zombie WSS with no SDK disconnect/error/start-failure event.

## Best Possible Solution

Add a Slack-owned bounded transport-health fallback, preferably from SDK/socket activity or a cheap probe, with regression coverage and docs/changelog updates.

## Reproduction Assessment

No deterministic live reproduction is available. Source inspection gives a conditional repro: after `app.start()`, if the Slack SDK emits none of `disconnected`, `unable_to_socket_mode_start`, or `error`, the supervisor awaits forever.

## Solution Assessment

Unclear: the requested message-flow watchdog would close the observed gap, but it conflicts with the current policy that app events are not transport liveness. A transport- or probe-backed fallback with an owner-chosen timeout is safer.
