## Summary

Keep open: current main still has an event-only Slack Socket Mode reconnect waiter with no bounded no-event fallback, and the generic gateway stale-socket monitor cannot help because Slack does not publish provider-proven transport activity.

## What This Changes

The issue requests a Slack Socket Mode liveness fallback so a zombie WSS with no SDK disconnect/error event is logged and reconnected without a manual service restart.

## Best Possible Solution

Add a Slack-owned bounded transport-health fallback that feeds provider-proven transport activity or performs a cheap probe, with regression coverage, docs, and changelog updates.

## Reproduction Assessment

No deterministic live reproduction is available. Source inspection gives a high-confidence conditional path: if the Slack SDK emits none of `disconnected`, `unable_to_socket_mode_start`, or `error` after start, the supervisor awaits forever.

## Solution Assessment

Unclear: the requested message-flow watchdog would close the gap, but it conflicts with the current transport-liveness policy for quiet workspaces. A transport- or probe-backed fallback with an owner-chosen timeout is the safer direction.
