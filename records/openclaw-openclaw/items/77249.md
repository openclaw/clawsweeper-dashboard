## Summary

Keep open: current main still has an event-only Slack Socket Mode disconnect waiter with no bounded fallback for the reported no-event zombie path, and a newer 2026.5.7 report confirms the silent wedge after recent reconnect changes.

## What This Changes

The issue requests a Slack Socket Mode liveness fallback so the reconnect supervisor recovers when the underlying WSS stops producing disconnect/error events.

## Best Possible Solution

Add a Slack-owned bounded transport-health fallback, preferably by publishing real Socket Mode heartbeat/ping/pong activity into the existing health monitor or by using a cheap probe, with synthetic no-event waiter coverage and docs/changelog updates.

## Reproduction Assessment

No deterministic live reproduction is available. Source inspection gives a high-confidence conditional path: if Slack's SDK never emits the watched disconnect/error/start-failure events after startup, OpenClaw awaits the disconnect waiter indefinitely.

## Solution Assessment

Unclear: the requested message-flow watchdog would wake the supervisor, but it treats quiet app traffic as socket failure. A transport-activity or probe-backed fallback is the narrower maintainable direction unless owners intentionally change that policy.
