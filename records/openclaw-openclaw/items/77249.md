## Summary

Keep open: current main still relies on Slack SDK disconnect/error/start-failure events and has no bounded no-event recovery path, while the production zombie-WSS trigger remains observational rather than deterministic.

## What This Changes

Add Slack Socket Mode liveness/reconnect handling so a zombie WSS that emits no SDK disconnect/error does not leave the Slack channel silent until manual restart.

## Best Possible Solution

Implement a Slack-plugin fallback that resolves the supervisor when transport health is stale without treating normal quiet channels as dead; cover it with fake-emitter/timer tests and update Slack Socket Mode docs/changelog.

## Reproduction Assessment

No high-confidence deterministic reproduction path was established. Source inspection does show that if the Slack SDK never emits disconnect/error/start-failure, current main waits forever on the disconnect waiter.

## Solution Assessment

Unclear: the requested message-flow watchdog is not obviously the best fix because current Slack docs deliberately avoid using app events as transport liveness. A transport- or probe-backed bounded fallback with an owner-chosen timeout would be safer.
