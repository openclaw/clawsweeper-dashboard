## Summary

Keep open: current main and v2026.6.8 still have no independent Slack Socket Mode transport-liveness signal for the reported no-event zombie websocket path, even though newer native reconnect handling covers SDK-emitted ping/pong failures.

## What This Changes

The issue reports Slack Socket Mode reconnect supervision wedging silently when a websocket stops producing disconnect/error events, leaving Slack messages unanswered until an operator restarts OpenClaw.

## Best Possible Solution

Keep this as the canonical Slack Socket Mode no-event liveness bug and add a Slack-owned transport-liveness or probe-backed reconnect path that marks the channel unhealthy without treating normal workspace quietness as failure.

## Reproduction Assessment

No; there is not a high-confidence live reproduction path in this review. The source path is reproducible by inspection: after startup the Slack supervisor waits only on SDK disconnect/error/start-failure events, and the reported failure is exactly the case where none arrive.

## Solution Assessment

Unclear; the issue identifies the right missing guard, but the message-flow watchdog proposal is only one possible fix. The best maintainable solution should use Slack transport heartbeat activity, SDK-level observation, or a bounded explicit probe so quiet but healthy Slack workspaces are not restarted.
