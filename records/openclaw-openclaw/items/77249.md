## Summary

Keep open: current main still has an event-only Slack Socket Mode waiter with no bounded fallback for the no-event zombie path, and a newer 2026.5.7 macOS report says the same silent wedge still happens after recent reconnect work.

## What This Changes

The issue asks OpenClaw to recover or emit diagnostics when Slack Socket Mode stops delivering events without emitting the SDK disconnect/error events that wake the reconnect supervisor.

## Best Possible Solution

Add a Slack-owned bounded transport-health fallback, preferably by publishing real Socket Mode heartbeat/ping/pong activity into the existing health monitor or by a cheap probe, with synthetic no-event waiter coverage plus docs/changelog updates.

## Reproduction Assessment

No deterministic live reproduction is available. Source inspection gives a high-confidence conditional path: if Slack's SDK never emits `disconnected`, `unable_to_socket_mode_start`, or `error` after startup, OpenClaw awaits the disconnect waiter indefinitely.

## Solution Assessment

Unclear: the requested message-flow watchdog would wake the supervisor, but it treats quiet app traffic as socket failure. A transport-activity or probe-backed fallback is the narrower maintainable direction unless owners intentionally change that policy.
