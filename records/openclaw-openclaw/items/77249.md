## Summary

Keep open: current main still has an event-only Slack Socket Mode reconnect waiter with an unbounded await, so the reported no-event zombie WSS path remains possible and needs an owner-chosen liveness policy.

## What This Changes

The issue asks OpenClaw's Slack Socket Mode supervisor to detect and recover from a zombie WebSocket that emits no disconnect, error, or start-failure event.

## Best Possible Solution

Add a Slack-owned bounded transport-health fallback, preferably based on Slack SDK/socket activity or a cheap probe rather than plain app-event silence, with synthetic timer/emitter regression coverage and docs/changelog updates.

## Reproduction Assessment

No deterministic live reproduction is available. Source inspection gives a high-confidence conditional path: if the Slack SDK emits none of `disconnected`, `unable_to_socket_mode_start`, or `error` after start, the supervisor awaits forever.

## Solution Assessment

Unclear: the requested message-flow watchdog would close the gap, but it may false-positive on quiet workspaces because current docs avoid treating app events as transport liveness. A transport- or probe-backed fallback with an owner-chosen timeout is safer.
