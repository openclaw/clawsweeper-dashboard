## Summary

Keep open: current main still leaves Slack Socket Mode recovery dependent on SDK emitter events, while Slack does not publish transport activity into the generic health monitor, so the reported no-event zombie WSS path remains an unresolved reliability gap.

## What This Changes

Request recovery and logging for Slack Socket Mode when a zombie WebSocket produces no disconnect/error event, avoiding manual process restarts.

## Best Possible Solution

Add a Slack-owned bounded transport-health fallback, preferably based on SDK/socket activity or a cheap probe, with regression tests, operator-visible logging, docs, and a changelog entry.

## Reproduction Assessment

No deterministic live reproduction is available. Source inspection gives a high-confidence conditional path: if the Slack SDK emits none of `disconnected`, `unable_to_socket_mode_start`, or `error` after start, the supervisor awaits forever.

## Solution Assessment

Unclear. The requested message-flow watchdog would close the hang, but current policy treats app events as application state rather than transport proof; a transport- or probe-backed fallback is the safer maintainable direction.
