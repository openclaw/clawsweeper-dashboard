## Summary

Keep open: current main still has an event-only Slack Socket Mode disconnect waiter with an unbounded await, so the reported no-event zombie WSS path remains possible and needs a maintainer-chosen liveness policy.

## What This Changes

The issue asks OpenClaw’s Slack Socket Mode supervisor to detect and recover when a WebSocket stalls without emitting Slack SDK disconnect/error events.

## Best Possible Solution

Add a Slack-owned bounded transport-health fallback, preferably based on SDK/socket activity or a cheap probe, with reconnect logging/status updates, regression coverage, docs, and a changelog entry.

## Reproduction Assessment

No deterministic live reproduction is available. Source inspection gives a high-confidence conditional path: if the Slack SDK emits none of the watched disconnect/error events after startup, the supervisor awaits forever.

## Solution Assessment

Unclear: the requested message-flow watchdog would close the gap, but current code and docs deliberately avoid treating app events as transport liveness. A transport- or probe-backed bounded fallback is the safer direction.
