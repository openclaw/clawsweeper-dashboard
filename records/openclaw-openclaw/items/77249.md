## Summary

Keep open: current main still has an event-only Slack Socket Mode reconnect supervisor, so the source path can hang indefinitely if the Slack SDK never emits disconnect/error/start-failure; the real AWS/NLB zombie-WSS trigger remains observational and needs maintainer choice of the safest liveness signal.

## What This Changes

Request a bounded Slack Socket Mode recovery path so a no-event zombie websocket emits diagnostics and reconnects instead of requiring a process restart.

## Best Possible Solution

Add a Slack-plugin transport/probe-backed fallback that resolves the supervisor on stale Socket Mode health without using ordinary app silence alone, with fake-emitter/timer coverage and Slack docs/changelog updates.

## Reproduction Assessment

No, there is not a high-confidence live reproduction path for the reported AWS/NLB zombie WSS. Source inspection does give a conditional reproduction: a started Socket Mode client that never emits `disconnected`, `unable_to_socket_mode_start`, or `error` leaves the reconnect waiter pending forever.

## Solution Assessment

Unclear: the requested message-flow watchdog is not clearly the best fix because current Slack code and docs deliberately avoid treating app events as transport liveness. A bounded SDK transport-state check or cheap probe chosen by the Slack owner would be safer.
