## Summary

Keep open: current main and v2026.6.10 still have an event-only Slack disconnect waiter, and the generic stale-socket health monitor requires `lastTransportActivityAt`, which Slack does not publish today.

## What This Changes

Issue reports that Slack Socket Mode reconnect supervision can hang indefinitely when the WebSocket stops producing disconnect/error events, leaving Slack unresponsive until process restart.

## Best Possible Solution

Add Slack-owned transport liveness tracking or a bounded stale-socket recovery path that updates `lastTransportActivityAt` from real socket/heartbeat signals and has live Slack proof against false positives.

## Reproduction Assessment

No high-confidence live reproduction is available. The production trigger is intermittent, but current source still has an event-only Slack disconnect waiter and no Slack transport-activity timestamp for the generic stale-socket monitor.

## Solution Assessment

No, not as currently implemented. Closing as implemented would be premature because the current generic monitor intentionally ignores app events without `lastTransportActivityAt`, and Slack does not publish that field today.
