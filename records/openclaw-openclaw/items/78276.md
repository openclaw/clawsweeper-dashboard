## Summary

Keep open: current `main` and the latest release still have the source-proven bootstrap-token lost-update shape, and the open closing PR remains the active security-sensitive fix path.

## What This Changes

The issue reports that concurrent setup-code issuance and bootstrap-token verification can race across processes and revive a consumed bootstrap token.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/78277 or an equivalent security-reviewed fix that serializes every bootstrap-token state read-modify-write path across processes and proves the race with non-reentrant coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: separate processes can each load `devices/bootstrap.json` and later replace it with stale whole-file state because `device-bootstrap` uses a module-local async lock instead of a cross-process file lock.

## Solution Assessment

Yes on direction, but the linked implementation is not complete until the review findings and proof gap are resolved. The narrow maintainable fix is one cross-process serialization boundary around all bootstrap-token state mutations, plus regression proof that cannot re-enter the same process lock.
