## Summary

Current main and the latest release still have the source-proven lost-update shape for bootstrap-token state, and an open closing PR already owns the security-sensitive fix path.

## What This Changes

The issue reports that concurrent setup-code issuance and Gateway bootstrap-token verification can stale-write `devices/bootstrap.json` and make a consumed setup code valid again.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/78277 or an equivalent security-reviewed fix that serializes every bootstrap-token state read-modify-write path across processes and adds non-reentrant regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence interleaving: separate OpenClaw processes can each load `devices/bootstrap.json` and then replace it with stale whole-file state because current bootstrap locking is process-local.

## Solution Assessment

Yes. The narrow maintainable fix is cross-process serialization around bootstrap-token issuance, verification, redemption, revocation, clear, and restore paths, with a true cross-process or stale-snapshot regression test.
