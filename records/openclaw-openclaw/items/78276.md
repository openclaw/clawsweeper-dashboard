## Summary

Current main still has the reported lost-update shape in bootstrap token state, and open PR #78277 already tracks this issue with closing syntax, so this should stay open for maintainer/security review until that fix or an equivalent one lands.

## What This Changes

The issue reports that concurrent setup-code issuance and bootstrap verification can restore a consumed bootstrap token in `devices/bootstrap.json`.

## Best Possible Solution

Land #78277 or an equivalent maintainer-reviewed fix that serializes all bootstrap-token state read-modify-write paths across processes and covers stale snapshot writeback with regression tests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence interleaving: separate OpenClaw processes can each load `devices/bootstrap.json` and later replace it with stale whole-file state because the current lock is process-local.

## Solution Assessment

Yes. Cross-process serialization around issuance, verification, redemption, revocation, clear, and restore paths is the narrow maintainable solution; the existing file-lock pattern fits this better than a broader pairing redesign.
