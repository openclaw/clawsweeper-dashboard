## Summary

Keep open: current main and v2026.5.12 still have the source-proven bootstrap-token lost-update shape, and the open closing PR still owns the security-sensitive fix path before this issue can close.

## What This Changes

The issue reports that concurrent setup-code issuance and bootstrap-token verification can resurrect consumed bootstrap tokens in the shared bootstrap state file.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/78277 or an equivalent security-reviewed fix that serializes every bootstrap-token read-modify-write path across processes and adds non-reentrant regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: separate processes can each load devices/bootstrap.json and later replace it with stale whole-file state because device-bootstrap uses a module-local async lock instead of a cross-process file lock.

## Solution Assessment

Yes on direction, but not complete until the linked PR is fixed and proven. The narrow maintainable solution is cross-process serialization around every mutating bootstrap-token state path, plus regression proof that cannot re-enter the same process lock.
