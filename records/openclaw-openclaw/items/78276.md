## Label Justifications

- `P1`: The issue affects a security-sensitive Gateway setup-code auth path and can allow stale bootstrap credentials to be accepted again.
- `impact:security`: The reported race crosses the bootstrap-token authentication boundary and can revive a consumed pairing credential.

## Summary

Keep open: current main and the latest release still keep bootstrap-token state in whole-file JSON writes guarded only by a process-local lock, and the linked fix PR is open and unmerged with prior proof/fix gaps.

## What This Changes

The issue reports that concurrent setup-code issuance and bootstrap-token verification can rewrite `devices/bootstrap.json` from stale whole-file snapshots, reviving consumed setup codes.

## Best Possible Solution

Serialize every mutating `devices/bootstrap.json` read-modify-write across processes, add a non-reentrant multi-process or stale-snapshot regression test, and keep this issue open until the linked PR or a replacement lands.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path where concurrent setup-code issuance and bootstrap verification/revocation load stale JSON snapshots and write them back under only process-local serialization; I did not run a live two-process Gateway repro in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is to put every bootstrap-token state read-modify-write behind a cross-process lock or equivalent Gateway-owned transaction, with regression proof that cannot re-enter the same process lock.
