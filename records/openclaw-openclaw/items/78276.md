## Summary

Current main still has the source-proven lost-update shape for bootstrap-token state, and the issue is already paired with an open closing PR that has unresolved maintainer review feedback. This should stay open until that PR or an equivalent security-reviewed fix lands.

## What This Changes

Report that concurrent setup-code issuance and bootstrap verification can resurrect a consumed bootstrap token by writing stale whole-file `devices/bootstrap.json` state.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/78277 or an equivalent maintainer-reviewed fix that serializes every bootstrap-token state read-modify-write path across processes and includes a non-reentrant regression test.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence interleaving: separate processes can each load `devices/bootstrap.json` and then replace it with stale whole-file state because current bootstrap locking is process-local.

## Solution Assessment

Yes, the narrow maintainable fix is cross-process serialization around bootstrap-token issuance, verification, redemption, revocation, clear, and restore paths. The active PR is pointed in that direction, but its test/proof concerns need resolution before merge.
