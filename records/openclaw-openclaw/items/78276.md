## Summary

Current main still has the reported cross-process lost-update shape in bootstrap token state, and an open linked fix PR explicitly tracks this issue, so conservative cleanup should keep it open until that PR or an equivalent fix lands.

## What This Changes

The issue reports that concurrent setup-code issuance and bootstrap-token verification can lose updates in `devices/bootstrap.json`, reviving consumed tokens or deleting replacement codes.

## Best Possible Solution

Land #78277 or an equivalent maintainer-reviewed fix that wraps all bootstrap-token read-modify-write paths in cross-process locking and adds regression coverage, then close this issue from the merge.

## Reproduction Assessment

Yes, source inspection gives a high-confidence interleaving: separate processes can each load `devices/bootstrap.json` and later replace it with stale state because `createAsyncLock` is process-local. I did not run the multi-process Gateway race in this read-only sweep.

## Solution Assessment

Yes, cross-process serialization around issuance, verification, redemption, revocation, and clear/restore paths is the narrow maintainable fix. The existing `withFileLock` sidecar pattern is a better fit for this report than a broader pairing redesign.
