## Summary

Keep open: current main and the latest release still show process-local whole-file bootstrap-token state writes, and the same-author linked PR is open but not merged and still has proof/review blockers.

## What This Changes

The issue reports that concurrent setup-code issuance and bootstrap verification can overwrite `devices/bootstrap.json` with stale state, reviving consumed bootstrap tokens.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/78277 or an equivalent security-reviewed fix that serializes every bootstrap-token state mutation across processes and proves the cross-process lost-update case.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence interleaving: separate Gateway and setup-code issuer processes can each load `devices/bootstrap.json` and then replace it with stale whole-file state because current main uses only a module-local lock.

## Solution Assessment

Yes. The narrow maintainable solution is cross-process serialization around bootstrap-token issuance, verification, redemption, revocation, clear, and restore paths, with real behavior proof and a non-reentrant regression test before merge.
