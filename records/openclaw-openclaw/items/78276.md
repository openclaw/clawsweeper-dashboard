## Summary

Keep open: current main and v2026.5.12 still have process-local serialization around bootstrap-token state writes, and the linked closing PR remains open with security/test/proof follow-up still unresolved.

## What This Changes

The issue reports that cross-process setup-code issuance and bootstrap verification can overwrite stale `devices/bootstrap.json` state and revive a consumed bootstrap token.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/78277 or an equivalent security-reviewed fix that serializes every bootstrap-token state read-modify-write path across processes and adds non-reentrant regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: separate processes can each load `devices/bootstrap.json` and later replace it with stale whole-file state because `device-bootstrap` uses a module-local async lock instead of a cross-process file lock.

## Solution Assessment

Yes on direction, but not complete yet. The narrow maintainable fix is one cross-process lock or transactional API for all bootstrap-token state mutations, with a child-process or stale-snapshot regression test.
