## Summary

Keep this PR open: it targets a plausible WebChat reconnect bug, but the patch removes the intentional token-only launch reset and would let a fresh tokenized launch reuse stale persisted session selection for the same gateway scope.

## What This Changes

The branch deletes Control UI token-only session reset logic, updates URL/session tests to preserve persisted session selection, and adds a WebChat reconnect changelog entry.

## Best Possible Solution

Keep the fresh tokenized-launch reset contract, then add a narrower reconnect restoration path that preserves a known existing session and covers both launch and reconnect cases.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection and the focused current-main test show the token-only URL reset path, but the full Windows WebChat Gateway crash/restart flow is not proven here.

## Solution Assessment

No. Deleting the reset solves the focused preservation test but also removes the prior stale-session launch guardrail; the safer fix should distinguish reconnect restoration from fresh tokenized launch.
