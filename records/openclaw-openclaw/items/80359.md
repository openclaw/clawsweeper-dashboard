## Summary

Keep open. The PR has focused tests and real browser proof for URL hydration, but the current diff still uses token equality as the reconnect signal, and source inspection shows ordinary token-only dashboard launches can reuse that same stored token and skip the reset that current main intentionally performs.

## What This Changes

The branch changes Control UI URL hydration so token-only WebChat URLs preserve the selected session when the URL token matches the already-loaded token, and adds focused app-settings tests.

## Best Possible Solution

Preserve reconnect sessions through an explicit `session=` URL or a separate reconnect marker while keeping ordinary token-only dashboard launches covered as fresh opens to `main`.

## Reproduction Assessment

Yes for the blocking review finding: source inspection shows dashboard token-only URLs, restored same-gateway tokens, and persisted non-main sessions combine with the PR guard to skip the reset. The full Gateway crash/restart scenario was not independently reproduced.

## Solution Assessment

No. Token equality alone is not the narrowest maintainable reconnect signal because normal dashboard launches can carry the same token; an explicit session or separate reconnect marker is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep token-only dashboard launches on main:** `ui/src/ui/app-settings.ts:214-216`
  - body: The new guard treats any token-only URL whose token equals `host.settings.token` as a reconnect. `openclaw dashboard` also emits token-only URLs from the configured gateway token, and settings load can restore that same token from `sessionStorage` along with a persisted non-main session, so an ordinary dashboard launch can skip the reset that https://github.com/openclaw/openclaw/pull/47453 intentionally requires. Please require an explicit session or separate reconnect signal instead of token equality alone.
