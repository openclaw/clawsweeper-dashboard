## Summary

Keep open: the patch still uses token equality as the reconnect signal, which can make a same-token fresh dashboard launch preserve a stale selected session contrary to the current token-only launch contract; the contributor has already acknowledged that this needs maintainer direction.

## What This Changes

This PR changes Control UI token-only WebChat URL handling so a matching stored token preserves the selected session, with focused settings tests and a changelog entry.

## Best Possible Solution

Use an unambiguous reconnect signal, such as an explicit `session=` URL or a distinct reconnect marker, while preserving token-only fresh launches to `main`.

## Reproduction Assessment

Yes for the review finding: source inspection shows `openclaw dashboard` can create a same-token token-only URL, Control UI restores that token and prior session selection, and the PR guard skips the reset when they match. No full live Windows Gateway crash/restart replay was established.

## Solution Assessment

No. Token equality alone is ambiguous between reconnect and ordinary dashboard launch; explicit `session=` or a separate reconnect/session-known signal is the safer contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep same-token fresh launches on main:** `ui/src/ui/app-settings.ts:214-216`
  - body: The new guard treats a token-only URL with `token === host.settings.token` as a reconnect, but `openclaw dashboard` emits tokenized URLs from the configured gateway token and Control UI restores that same token from sessionStorage. A normal fresh dashboard launch after selecting an old session can therefore preserve the stale session instead of honoring the token-only launch contract from https://github.com/openclaw/openclaw/pull/47453. Please use an explicit `session=` URL or a distinct reconnect/session-known marker, and keep same-token fresh launches covered by a reset test.
  - confidence: 0.87
