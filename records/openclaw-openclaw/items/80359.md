## Summary

Keep this PR open: the central bug is plausible, but the current patch still uses token equality as the reconnect signal, and the author has acknowledged that this is ambiguous for normal dashboard launches that reuse the same gateway token.

## What This Changes

The PR changes Control UI URL hydration so matching-token token-only WebChat URLs preserve the selected session, adds focused settings tests, and adds a changelog entry for the linked recovery issue.

## Best Possible Solution

Keep token-only fresh launches on main unless the URL carries an explicit session or another unambiguous reconnect marker, then cover same-token fresh launch, matching reconnect, different-token launch, and explicit session URLs.

## Reproduction Assessment

Yes for the source-level review finding: the PR diff, current main reset path, dashboard tokenized URL construction, and persisted token/session storage show how a same-token fresh launch can be misclassified. No full live Windows Gateway crash/restart replay was established.

## Solution Assessment

No. Token equality alone is not the best fix because ordinary dashboard launches can reuse that token; a distinct reconnect/session-known signal or explicit session URL is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep same-token fresh launches on main:** `ui/src/ui/app-settings.ts:214-216`
  - body: The new guard treats a token-only URL with `token === host.settings.token` as a reconnect, but `openclaw dashboard` opens and copies tokenized URLs using the configured gateway token. A fresh dashboard launch after selecting an old session can therefore preserve that stale session instead of honoring the token-only launch contract from https://github.com/openclaw/openclaw/pull/47453. Please use an explicit `session=` URL or a distinct reconnect/session-known marker, and keep same-token fresh launches covered by a reset test.
  - confidence: 0.87
