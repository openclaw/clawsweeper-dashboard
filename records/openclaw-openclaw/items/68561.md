## Summary

Keep open. Current main partially moved the original invalid cron shape forward by requiring main-session cron jobs to use `systemEvent` and rejecting non-webhook channel delivery, but it still does not warn or validate when main-session system-event text asks the agent to emit `ANNOUNCE_SKIP`, while the gateway still suppresses that exact reply from chat/history.

## What This Changes

The issue asks for clearer documentation and diagnostics when a main-session cron payload contains `ANNOUNCE_SKIP`, plus possible dry-run/config validation improvements.

## Best Possible Solution

Keep the global exact-control-token suppression for subagents, but add cron-local ergonomics: document the hazard, warn or fail closed when main-session system-event text contains the exact control token, and add focused regression coverage for create/update/runtime diagnostics.

## Reproduction Assessment

Yes. A source-backed reproduction is clear: create a main-session cron job with `payload.kind="systemEvent"` whose text asks for an exact `ANNOUNCE_SKIP` reply; cron enqueues that text, and gateway chat projection/history suppress exact `ANNOUNCE_SKIP` assistant replies.

## Solution Assessment

Yes. The narrowest maintainable fix is cron-specific docs plus validation or warning, not changing the shared control-reply suppressor that subagent announce flows intentionally rely on.
