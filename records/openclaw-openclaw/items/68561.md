## Summary

Keep open. Current main still accepts and enqueues main-session cron system-event text without an `ANNOUNCE_SKIP` diagnostic, while exact `ANNOUNCE_SKIP` assistant replies remain globally suppressed from chat/history.

## What This Changes

Add cron-local warning/validation/docs for main-session cron payloads that can trigger the `ANNOUNCE_SKIP` control-token suppressor.

## Best Possible Solution

Keep global exact-control-token suppression, but add cron-local diagnostics and docs for main-session `systemEvent` text that asks the agent to emit `ANNOUNCE_SKIP`.

## Reproduction Assessment

Yes. A source-backed reproduction is clear: create a `sessionTarget: "main"` cron job whose system-event text instructs the agent to reply exactly `ANNOUNCE_SKIP`; cron enqueues that text and the gateway suppresses that exact assistant reply.

## Solution Assessment

Yes. The narrowest maintainable fix is warning/validation plus cron documentation and focused regression tests; changing the shared suppressor would risk breaking documented subagent announce semantics.
