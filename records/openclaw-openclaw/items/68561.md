## Summary

Keep open: current main still has the cron ergonomics gap. Main-session cron payload text is enqueued unchanged, exact `ANNOUNCE_SKIP` assistant replies are suppressed from chat/history, and the cron docs/CLI validation still do not warn about this control-token hazard.

## What This Changes

Request warning, validation, or documentation so main-session cron payloads containing `ANNOUNCE_SKIP` do not appear to skip work silently.

## Best Possible Solution

Preserve global exact-control-token suppression, add cron-local diagnostics and docs for main-session `systemEvent` text that can prompt an exact `ANNOUNCE_SKIP` reply, and treat cron dry-run as a separate feature request.

## Reproduction Assessment

Yes, at source level: a main-session cron job can enqueue payload text that asks for an exact `ANNOUNCE_SKIP` reply, and current gateway code suppresses that exact assistant reply from visible chat/history. I did not run a live Feishu cron job.

## Solution Assessment

Yes for the narrow repair: cron-specific warning or validation, documentation, and regression coverage are the maintainable fix. Changing global `ANNOUNCE_SKIP` suppression would risk breaking the established control-reply behavior.
