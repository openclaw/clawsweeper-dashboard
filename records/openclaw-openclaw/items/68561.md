## Summary

Keep open: current main still accepts main-session cron `systemEvent` text containing `ANNOUNCE_SKIP` without a cron-local warning, and the gateway still intentionally suppresses exact `ANNOUNCE_SKIP` assistant replies, so the confusing silent-output path remains source-reproducible.

## What This Changes

The issue asks for clearer cron behavior when a main-session payload mentions `ANNOUNCE_SKIP`, including docs, warnings or validation, and a way to test jobs before waiting for the schedule.

## Best Possible Solution

Preserve the global exact-control-token suppression contract, add cron-local diagnostics and docs for main-session `systemEvent` text that can prompt an exact `ANNOUNCE_SKIP` reply, and point dry-run needs at the existing `openclaw cron run --wait` flow unless maintainers want a separate feature.

## Reproduction Assessment

Yes, at source level: a main-session cron job can enqueue payload text that asks for an exact `ANNOUNCE_SKIP` reply, and current gateway code suppresses that exact assistant reply from visible chat/history. I did not run a live Feishu cron job.

## Solution Assessment

Yes for the narrow repair: cron-specific warning or validation, documentation, and regression coverage are the maintainable fix. Changing global `ANNOUNCE_SKIP` suppression would risk breaking the established control-reply behavior.
