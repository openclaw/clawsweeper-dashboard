## Summary

Keep open: current main still has the cron ergonomics gap. Main-session cron payload text is enqueued unchanged, exact `ANNOUNCE_SKIP` assistant replies are globally suppressed from chat/history, and the cron docs still do not warn about this main-session control-token hazard.

## What This Changes

The issue requests clearer cron behavior when a main-session `systemEvent` payload can lead the agent to emit `ANNOUNCE_SKIP`, including documentation, warning or validation, and a possible dry-run workflow.

## Best Possible Solution

Preserve global exact-control-token suppression, but add cron-local diagnostics and docs for main-session `systemEvent` text that can prompt an exact `ANNOUNCE_SKIP` reply; treat a full dry-run mode as a separate feature.

## Reproduction Assessment

Yes, at source level: a main-session cron job can enqueue payload text that asks for an exact `ANNOUNCE_SKIP` reply, and current gateway code suppresses that exact assistant reply from visible chat/history. I did not run a live Feishu cron job.

## Solution Assessment

Yes for the narrow repair: cron-specific warning or validation, documentation, and regression coverage are the maintainable fix. Changing global `ANNOUNCE_SKIP` suppression would risk breaking the documented subagent control-reply behavior.
