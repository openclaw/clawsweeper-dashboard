## Summary

Keep open. Current main still leaves the reported cron ergonomics bug in place: main-session cron system events are accepted and enqueued without any `ANNOUNCE_SKIP` diagnostic, while exact `ANNOUNCE_SKIP` assistant replies are still globally hidden from chat and history.

## What This Changes

Request clearer documentation and diagnostics for main-session cron jobs whose system-event text can prompt the agent to reply with the hidden `ANNOUNCE_SKIP` control token.

## Best Possible Solution

Keep the global exact-control-token suppression for chat/subagent behavior, but add cron-local ergonomics: document that main-session jobs enqueue system events for heartbeat, add a clear warning or validation diagnostic when main-session system-event text contains the exact `ANNOUNCE_SKIP` control token, and cover the behavior with focused cron and gateway tests.

## Reproduction Assessment

Yes. A source-backed reproduction is clear: create a main-session cron job whose `systemEvent` asks the agent to reply exactly `ANNOUNCE_SKIP`; the cron service enqueues that text, and gateway projection/history suppress exact `ANNOUNCE_SKIP` replies.

## Solution Assessment

Yes. The narrowest maintainable fix is cron-specific docs plus warning/validation and regression tests; changing the global `ANNOUNCE_SKIP` suppressor would risk breaking documented subagent announce behavior.
