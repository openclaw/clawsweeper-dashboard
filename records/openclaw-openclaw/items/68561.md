## Summary

Keep open: current main still has the cron ergonomics gap. Main-session cron jobs enqueue system-event text unchanged, gateway chat/history still suppress exact `ANNOUNCE_SKIP` assistant replies, and cron docs/CLI validation do not warn about this control-token hazard.

## What This Changes

The issue requests clearer cron behavior, warnings or validation, docs, and dry-run support for main-session payloads that mention `ANNOUNCE_SKIP`.

## Best Possible Solution

Preserve global exact-control-token suppression, add cron-local diagnostics and docs for main-session `systemEvent` text that can prompt an exact `ANNOUNCE_SKIP` reply, and leave cron dry-run as a separate feature discussion.

## Reproduction Assessment

Yes, at source level: a main-session cron job can enqueue payload text that asks for an exact `ANNOUNCE_SKIP` reply, and current gateway code suppresses that exact assistant reply from visible chat/history. I did not run a live Feishu cron job.

## Solution Assessment

Yes for the narrow repair: cron-specific warning or validation, documentation, and regression coverage are the maintainable fix. Changing global `ANNOUNCE_SKIP` suppression would risk breaking the established control-reply behavior.
