## Summary

Keep open: current main still has the reported cron ergonomics gap. Cron does not treat `ANNOUNCE_SKIP` in the payload as a skip signal, but main-session cron can prompt the agent to answer the exact internal control token, and that reply is still hidden from chat/history without a cron-local warning or docs.

## What This Changes

The issue asks for clearer cron behavior when a main-session cron payload contains `ANNOUNCE_SKIP`, including documentation, diagnostics, dry-run support, or validation.

## Best Possible Solution

Keep the global exact-control-token suppression, but add cron-local warning or validation diagnostics and docs for main-session system-event text that asks for exact `ANNOUNCE_SKIP`, with focused regression coverage.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: a main-session cron `systemEvent` can ask the agent to reply exactly `ANNOUNCE_SKIP`, cron enqueues it without a diagnostic, and gateway chat projection suppresses that exact assistant reply. I did not run a live cron/channel reproduction because this review is read-only.

## Solution Assessment

Yes, the best narrow fix is cron-specific diagnostics plus docs and tests; changing the global `ANNOUNCE_SKIP` suppressor would risk breaking the documented subagent announce behavior. Treat a full `cron dry-run` command as a separate feature follow-up unless a maintainer explicitly includes it.
