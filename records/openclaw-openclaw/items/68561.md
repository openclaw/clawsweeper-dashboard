## Summary

Keep open: current main still has the cron ergonomics gap described in the prior review. Main-session cron enqueues the systemEvent text without a cron-specific warning, exact `ANNOUNCE_SKIP` assistant replies are still suppressed from chat/history, and the cron docs still only document silent-token suppression for isolated `NO_REPLY`, not this main-session hazard.

## What This Changes

The issue asks for clearer cron main-vs-isolated payload behavior plus warning, validation, or dry-run support when main-session cron payloads mention `ANNOUNCE_SKIP`.

## Best Possible Solution

Keep global exact-control-token suppression, but add cron-local warning or validation diagnostics and docs for main-session systemEvent text that can prompt an exact `ANNOUNCE_SKIP` reply; leave full dry-run mode for a separate product decision.

## Reproduction Assessment

Yes from source inspection: a main-session cron job can enqueue systemEvent text that asks for an exact `ANNOUNCE_SKIP` reply, and the gateway then suppresses that exact assistant reply from chat/history. I did not establish a live Feishu reproduction in this read-only review.

## Solution Assessment

Yes, the narrow maintainable fix is cron-specific diagnostics plus docs and regression coverage. Changing the global `ANNOUNCE_SKIP` suppressor would risk breaking the documented subagent announce behavior.
