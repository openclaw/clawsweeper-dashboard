## Summary

Keep open: current main still has the reported cron diagnostics/docs gap. Main-session cron jobs enqueue system-event text without checking for `ANNOUNCE_SKIP`, while exact `ANNOUNCE_SKIP` assistant replies are still globally hidden from chat and history, so the confusing silent outcome remains source-reproducible.

## What This Changes

The issue asks for clearer cron behavior around `ANNOUNCE_SKIP` in `sessionTarget: "main"` payloads, including documentation, warning or validation, and a broader dry-run suggestion.

## Best Possible Solution

Add cron-local warning or validation diagnostics and docs for main-session system-event text that can prompt an exact `ANNOUNCE_SKIP` reply, while leaving the global exact-control-token suppressor intact and treating dry-run mode separately.

## Reproduction Assessment

Yes from source inspection: a main-session cron job can enqueue systemEvent text that asks for an exact `ANNOUNCE_SKIP` reply, and the gateway suppresses that exact assistant reply from chat/history. I did not establish a live Feishu reproduction in this read-only review.

## Solution Assessment

Yes for the narrow repair: cron-specific diagnostics, documentation, and regression coverage are the most maintainable fix. Changing the global `ANNOUNCE_SKIP` suppressor would risk breaking the existing subagent/control-reply behavior.
