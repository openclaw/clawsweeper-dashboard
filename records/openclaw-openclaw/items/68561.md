## Summary

Keep open: current main still has the cron ergonomics gap. Main-session cron payload text is enqueued without any `ANNOUNCE_SKIP` diagnostic, exact `ANNOUNCE_SKIP` assistant replies are still globally suppressed from chat/history, and the cron docs do not document this main-session hazard.

## What This Changes

The issue requests cron diagnostics and documentation for `ANNOUNCE_SKIP` in main-session cron payloads, plus possible dry-run and validation improvements.

## Best Possible Solution

Keep global exact-control-token suppression, but add cron-local warning or validation diagnostics plus cron docs and regression coverage for main-session systemEvent text that can prompt an exact `ANNOUNCE_SKIP` reply; treat full dry-run mode as a separate product decision.

## Reproduction Assessment

Yes from source inspection: a main-session cron job can enqueue systemEvent text that asks for an exact `ANNOUNCE_SKIP` reply, and the gateway suppresses that exact assistant reply from chat/history. I did not establish a live Feishu reproduction in this read-only review.

## Solution Assessment

Yes for the narrow repair: cron-specific diagnostics, documentation, and regression coverage are the most maintainable fix. Changing the global `ANNOUNCE_SKIP` suppressor would risk breaking the documented subagent control-reply behavior.
