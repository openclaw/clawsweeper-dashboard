## Summary

Keep open: current main still has the reported cron/gateway ergonomics gap. Main-session cron accepts and enqueues the payload text without an ANNOUNCE_SKIP diagnostic, exact ANNOUNCE_SKIP assistant replies are still suppressed from chat/history, and the cron docs do not document this main-session hazard. The related open PR at https://github.com/openclaw/openclaw/pull/48883 is broader and not a merged closure path for this issue.

## What This Changes

The issue requests clearer cron handling for main-session payloads that mention ANNOUNCE_SKIP, including docs, warning or validation, and a possible dry-run command.

## Best Possible Solution

Keep global exact-control-token suppression, but add cron-local diagnostics or validation plus cron docs and regression coverage for main-session systemEvent text that can prompt an exact ANNOUNCE_SKIP reply; treat full dry-run mode as a separate feature decision.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: a main-session cron job can enqueue systemEvent text asking for an exact ANNOUNCE_SKIP reply, and the gateway suppresses that exact assistant reply from chat-visible output. I did not establish a live Feishu reproduction in this read-only review.

## Solution Assessment

Yes for the narrow repair: cron-specific diagnostics, documentation, and regression coverage are the maintainable fix. Changing the global ANNOUNCE_SKIP suppressor would risk breaking the intentionally covered control-reply behavior.
