## Summary

Keep open: current main still has the cron ergonomics gap. Main-session cron text is enqueued verbatim, exact `ANNOUNCE_SKIP` assistant replies are still globally suppressed, and the cron docs/CLI do not warn about that interaction.

## What This Changes

The issue requests warning, validation, and documentation for main-session cron jobs whose system-event payload can lead the agent to reply with the hidden `ANNOUNCE_SKIP` control token.

## Best Possible Solution

Add cron-local diagnostics and docs for the `ANNOUNCE_SKIP` hazard while preserving the existing global control-token suppression used by subagent announce/reply flows.

## Reproduction Assessment

Yes. A source-backed reproduction is to create a `sessionTarget: "main"` cron job whose system event tells the agent to reply exactly `ANNOUNCE_SKIP`; cron enqueues that text, and the gateway hides exact `ANNOUNCE_SKIP` assistant output.

## Solution Assessment

Yes. The narrow maintainable solution is warning/validation plus cron documentation and regression tests; changing the global suppressor would risk breaking documented subagent behavior.
