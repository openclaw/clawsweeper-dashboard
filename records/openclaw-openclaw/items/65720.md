## Summary

Keep open for maintainer product and security architecture review: current main and v2026.6.8 still serve Control UI from the Gateway HTTP process and do not provide a crash-survivor same-port static fallback or config contract.

## What This Changes

Request an opt-in Gateway mode that keeps a static WebUI or maintenance page accessible on the normal Gateway port after the Gateway process crashes.

## Best Possible Solution

Define the supported Gateway crash-resilience architecture first; if core should own WebUI fallback, implement a supervised rescue/static sidecar with explicit config schema, auth/origin semantics, docs, and tests for port handoff and recovery.

## Reproduction Assessment

Not applicable: this is an enhancement request, not broken documented behavior. Source inspection gives high confidence that current main lacks the requested crash-survivor WebUI mode.

## Solution Assessment

No, not as proposed: a simple in-process config toggle cannot serve a page after the Gateway process exits. A safer solution needs a supervised sidecar or rescue-Gateway design with explicit security and upgrade semantics.
