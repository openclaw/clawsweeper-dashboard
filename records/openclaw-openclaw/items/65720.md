## Summary

Keep open. Current main still serves the Control UI from the Gateway HTTP process on the Gateway port, and neither current main nor the latest release has a crash-survivor static server, same-port takeover path, or config contract for the requested fallback.

## What This Changes

Request to keep the WebUI reachable after a Gateway crash by serving an opt-in static fallback page on the same port.

## Best Possible Solution

Define the supported resilience architecture first; if core should own it, implement a supervised sidecar or rescue-Gateway capability with typed config, schema/help, docs, and tests for port handoff, recovery, and Control UI security boundaries.

## Reproduction Assessment

Not applicable as a bug reproduction because this is an enhancement request. Source inspection gives a high-confidence current-main gap: Control UI serving is in the Gateway HTTP process and no crash-survivor same-port fallback exists.

## Solution Assessment

No, not as proposed. An in-process `gateway.serveStaticOnCrash` toggle cannot respond after process exit; a supervised sidecar or rescue-Gateway design with explicit auth and port-handoff semantics would be safer.

## Risks / Open Questions

- Same-port fallback requires explicit process supervision and port handoff because an exited Gateway process cannot serve a page itself.
- A static fallback for the Control UI must define auth, origin, TLS, device-pairing, and information-disclosure boundaries before implementation.
- The existing service-worker cache can make previously visited UI routes partially load offline, so maintainers should decide whether that is a sufficient subset or separate behavior.
