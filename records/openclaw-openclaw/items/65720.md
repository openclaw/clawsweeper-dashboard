## Summary

Keep open for maintainer product and security architecture review: current main and v2026.6.8 still serve Control UI from the Gateway HTTP process and do not provide a crash-survivor same-port static fallback or config contract.

## What This Changes

The issue requests an opt-in Gateway resilience feature that would keep a static WebUI or maintenance page reachable after the Gateway process crashes.

## Best Possible Solution

Define the supported resilience architecture first; if core should own it, implement a supervised rescue/static sidecar with explicit config schema, security semantics, docs, and tests for port handoff and recovery.

## Reproduction Assessment

Not applicable: this is an enhancement request, and source inspection shows the requested crash-survivor mode is absent rather than a broken documented behavior.

## Solution Assessment

No, not as proposed: an in-process toggle cannot serve pages after that process exits, so the safer solution needs a supervised sidecar or rescue-Gateway design with explicit security semantics.
