## Summary

Keep open for maintainer product/security architecture review: current main and v2026.6.6 still serve the Control UI from the Gateway process and do not provide a crash-survivor same-port static fallback or config contract.

## What This Changes

The issue requests an opt-in Gateway crash fallback that keeps a static WebUI available on the same port after the Gateway process exits.

## Best Possible Solution

Define the supported resilience architecture first; if core should own this, implement a supervised rescue/static sidecar with explicit config schema, security boundaries, docs, and tests for port handoff and recovery.

## Reproduction Assessment

Not applicable: this is an enhancement request, and source inspection shows the requested crash-survivor mode is absent rather than a broken documented behavior.

## Solution Assessment

No, not as proposed: an in-process `gateway.serveStaticOnCrash` toggle cannot serve pages after that process exits; the safer solution needs a supervised sidecar/rescue-Gateway design with explicit security semantics.

## Risks / Open Questions

- Same-port WebUI availability after a process crash needs a supervisor, sidecar, or port-handoff architecture, not an in-process option.
- A static fallback page intersects Control UI auth, origin, TLS, device-pairing, and information-disclosure boundaries.
- Existing service-worker/cache behavior may partially cover already-open or cached UI cases, so maintainers need to decide whether that is separate from the desired server-availability guarantee.
