## Summary

Keep open. Current main still serves the Control UI from the Gateway HTTP process on the Gateway port, and the only static fallback found is the normal SPA route fallback while that process is alive; there is no crash-survivor static server or `gateway.controlUi`/`gateway` config for same-port maintenance content after a hard Gateway crash.

## What This Changes

Request an opt-in Gateway/Control UI resilience mode that keeps a static maintenance page reachable on the Gateway port after the Gateway process crashes.

## Best Possible Solution

Decide the supported resilience architecture first. If this belongs in core, implement it as a Gateway/service-manager capability with typed config, schema/help, docs, and tests for port handoff, custom maintenance content, restart recovery, auth/origin/TLS behavior, and safe disclosure; otherwise document supervised restart or a rescue-gateway deployment as the supported pattern.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. The current-main gap is high-confidence: source and docs show the Control UI is served by the Gateway process on the Gateway port, and no crash-survivor fallback config or server exists.

## Solution Assessment

Unclear. The requested simple `gateway.serveStaticOnCrash` toggle is probably not the best complete solution because a crashed process cannot serve the page and the admin WebUI security contract must be preserved; a designed supervisor/service-manager fallback or explicit supervised-restart guidance is safer.

## Risks / Open Questions

- A same-port fallback after a hard process crash likely requires a supervisor, companion process, or service-manager handoff; the crashed Gateway process cannot serve its own fallback page.
- Any fallback page for an admin Control UI surface needs explicit auth, TLS, origin, device identity, and information-disclosure decisions before implementation.
- The product direction is not yet settled between a core static-maintenance server, stronger supervised restart behavior, a rescue gateway pattern, or documentation that external supervision is the supported resilience layer.
