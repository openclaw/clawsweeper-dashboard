## Summary

Keep open. Current main still serves the Control UI from the Gateway HTTP process on the Gateway port, and the only static fallback found is the normal SPA route fallback while that process is alive; there is no crash-survivor static server or supported same-port maintenance page option.

## What This Changes

The issue requests an opt-in Gateway crash fallback that keeps the WebUI reachable with a static maintenance page on the Gateway port.

## Best Possible Solution

Decide the supported resilience architecture first; if it belongs in core, implement it as a Gateway/service-manager capability with typed config, schema/help, docs, and tests for port handoff, custom maintenance content, restart recovery, auth, origin, TLS, and safe disclosure.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. The current-main gap is high-confidence from source, docs, config schema, and focused search: the Control UI depends on the live Gateway HTTP process and has no crash-survivor fallback server.

## Solution Assessment

Unclear. The requested `gateway.serveStaticOnCrash`-style toggle is likely too narrow by itself because a crashed process cannot serve the page; a designed supervisor/sidecar/rescue-Gateway pattern with explicit security semantics would be safer.
