## Summary

Keep open. Current main still serves the Control UI from the Gateway HTTP process on the Gateway port; the existing SPA, mount, and service-worker fallbacks do not provide a supported same-port static maintenance page after the Gateway process exits.

## What This Changes

The issue requests an opt-in Gateway resilience mode where the WebUI remains reachable after a Gateway crash, potentially through a static fallback or maintenance page.

## Best Possible Solution

Define the supported resilience architecture first; if it belongs in core, implement a supervisor, sidecar, or rescue-Gateway capability with typed config, schema/help, docs, and tests for port handoff, recovery, and Control UI security boundaries.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. The current-main gap is high confidence from source, docs, config schema, and the absence of any crash-survivor server path.

## Solution Assessment

Unclear. A simple in-process gateway.serveStaticOnCrash toggle is not enough because a crashed process cannot serve the page; a designed supervisor or sidecar/rescue Gateway with explicit security semantics is safer.
