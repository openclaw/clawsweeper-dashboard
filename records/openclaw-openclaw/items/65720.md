## Summary

Keep open for maintainer product and security architecture review; current main and the latest release still serve the Control UI from the Gateway HTTP listener and do not provide a crash-survivor same-port static fallback or config contract.

## What This Changes

Request an opt-in Gateway fallback mode that keeps a static WebUI reachable after the Gateway process crashes, with a custom maintenance message/config option.

## Best Possible Solution

Define the supported resilience architecture first; if core should own it, implement a supervised sidecar or rescue-Gateway capability with typed config, schema/help, docs, and tests for port handoff, recovery, and Control UI security boundaries.

## Reproduction Assessment

Not applicable as a bug reproduction because this is an enhancement request. Source inspection gives a high-confidence current-main gap: Control UI serving is in the Gateway process and no crash-survivor same-port fallback exists.

## Solution Assessment

No, not as proposed. An in-process `gateway.serveStaticOnCrash` toggle cannot serve after process exit; a supervised sidecar or rescue-Gateway design with explicit auth and port-handoff semantics would be safer.
