## Summary

Keep open for maintainer product and security architecture review: current main and v2026.6.10 still serve Control UI from the in-process Gateway HTTP listener and have no crash-survivor same-port static fallback or config contract.

## What This Changes

Request an opt-in Gateway/Control UI crash-resilience mode that keeps a same-port static WebUI available after the Gateway process exits.

## Best Possible Solution

Have maintainers define the supported crash-resilience architecture first; if core owns it, implement a supervised rescue/static sidecar with explicit port handoff, auth/origin/TLS semantics, config schema, docs, and upgrade tests.

## Reproduction Assessment

Not applicable: this is an enhancement request, not broken documented behavior. Source inspection gives high confidence that current main lacks the requested crash-survivor WebUI mode.

## Solution Assessment

No, not as a simple in-process toggle: once the Gateway process exits, it cannot serve a static page. The maintainable path needs a supervisor or rescue-sidecar design with explicit security and upgrade semantics.
