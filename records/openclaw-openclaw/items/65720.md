## Summary

Keep open: current main and v2026.6.6 still serve the Control UI from the live Gateway HTTP process, and the requested same-port crash-survivor static fallback remains a product/security architecture decision rather than an implemented feature or a narrow bug fix.

## What This Changes

Request an opt-in Gateway/Control UI mode where a lightweight static maintenance page remains reachable after the Gateway process crashes.

## Best Possible Solution

Define the supported crash-survivor Gateway/Control UI resilience architecture first; if core owns it, implement a supervised rescue/static responder with explicit config schema, auth/origin semantics, docs, and cross-platform restart proof.

## Reproduction Assessment

Not applicable: this is an enhancement request, and source inspection shows the requested crash-survivor mode is absent rather than a broken documented behavior.

## Solution Assessment

No, not as proposed: an in-process gateway.serveStaticOnCrash-style option cannot serve pages after the Gateway process exits. The maintainable solution needs a supervised sidecar/rescue-Gateway design with explicit security and port-handoff semantics.
