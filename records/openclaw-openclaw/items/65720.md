## Summary

Keep open. Current main still serves the Control UI from the Gateway HTTP process on the Gateway port; the existing fallbacks are SPA routing, browser mount recovery, and service-worker caching for prior visits, not a crash-survivor same-port static server or supported maintenance-page mode.

## What This Changes

Request a Gateway/Control UI resilience feature that keeps a static WebUI or maintenance page reachable after the Gateway process crashes.

## Best Possible Solution

Define the supported resilience architecture first; if it belongs in core, implement a supervisor/sidecar/rescue-Gateway capability with typed config, schema/help, docs, and tests for port handoff, recovery, and the Control UI security boundary.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. The current-main gap is high-confidence from source, docs, config schema, and the absence of any crash-survivor server path.

## Solution Assessment

Unclear. A simple in-process `gateway.serveStaticOnCrash` toggle is not enough because a crashed process cannot serve the page; a designed supervisor or sidecar/rescue Gateway with explicit security semantics is safer.
