## Summary

Keep open for maintainer product and security architecture review; current main and the latest release still serve the Control UI from the Gateway process and do not provide a crash-surviving same-port static fallback or config contract.

## What This Changes

The issue requests an opt-in Gateway crash fallback that keeps a static WebUI or maintenance page reachable on the normal Gateway port after the main Gateway process exits.

## Best Possible Solution

Define the supported resilience architecture first; if core should own it, implement a supervised rescue or sidecar capability with typed config, schema/help, docs, and tests for port handoff, recovery, and Control UI security boundaries.

## Reproduction Assessment

Not applicable as a bug reproduction because the issue asks for a new resilience mode. Source inspection gives high confidence that current main has no crash-surviving same-port Control UI fallback.

## Solution Assessment

No, not as proposed. An in-process `gateway.serveStaticOnCrash` style toggle cannot serve after process exit; a supervised sidecar or rescue-Gateway design with explicit auth and port-handoff semantics would be safer.

## Risks / Open Questions

- A same-port fallback cannot be implemented as an in-process toggle after the Gateway exits; it needs a supervisor, sidecar, parent process, or service-manager handoff design.
- Keeping any Control UI surface reachable during Gateway failure raises auth, stale-state, TLS/origin, and information-disclosure questions that need maintainer product and security review.
- The existing service-worker cache may partially load a previously visited UI shell offline, so maintainers should decide whether that is separate behavior or part of the desired fallback story.
