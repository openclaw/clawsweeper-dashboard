## Summary

Keep open. Current main still serves the Control UI from the Gateway HTTP process on the Gateway port; there is no crash-survivor static server, same-port takeover mode, or canonical duplicate/fix that resolves the request.

## What This Changes

Request that the WebUI stay reachable after a Gateway crash by serving a static maintenance or fallback page, optionally controlled by Gateway config.

## Best Possible Solution

Define the supported resilience architecture first; if it belongs in core, implement a supervised sidecar or rescue-Gateway capability with typed config, schema/help, docs, and tests for port handoff, recovery, and Control UI security boundaries.

## Reproduction Assessment

Not applicable as a bug reproduction because this is an enhancement request. Source and docs give a high-confidence current-main gap: Control UI serving is in the Gateway HTTP process and no crash-survivor same-port fallback mode exists.

## Solution Assessment

No, not as proposed. An in-process `gateway.serveStaticOnCrash` toggle cannot respond after the process exits; a supervised sidecar or rescue-Gateway design with explicit auth and port-handoff semantics is safer.
