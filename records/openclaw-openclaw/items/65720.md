## Summary

Keep open: current main and v2026.6.10 still serve Control UI from the in-process Gateway HTTP listener with no crash-survivor static fallback or public config contract, so this needs maintainer product and security architecture review.

## What This Changes

The issue requests an opt-in Gateway crash-survivor static WebUI fallback so the UI remains reachable when the Gateway process exits.

## Best Possible Solution

Have maintainers define the supported crash-resilience architecture first; if core owns it, implement a supervised rescue/static sidecar or documented supervisor contract with explicit port handoff, auth/origin/TLS semantics, config schema, docs, and upgrade tests.

## Reproduction Assessment

Not applicable: this is an enhancement request, not broken documented behavior. Source inspection gives high confidence that current main lacks the requested crash-survivor WebUI mode.

## Solution Assessment

No, not as a simple in-process toggle: once the Gateway process exits, it cannot serve a static page. The maintainable path needs a supervisor or rescue-sidecar design with explicit security and upgrade semantics.
