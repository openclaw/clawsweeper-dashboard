## Summary

Keep open. Current main and the latest release still manage macOS gateway install/status through the user LaunchAgent and gui launchd domain, while two open pull requests already target the duplicate system LaunchDaemon conflict, so this issue should stay open until one candidate fix lands.

## What This Changes

The issue asks OpenClaw to detect an existing same-label macOS system LaunchDaemon before `openclaw gateway install` writes or bootstraps a user LaunchAgent that can restart-loop the gateway.

## Best Possible Solution

Land one narrow duplicate-manager guard that blocks user LaunchAgent mutation when a same-label system LaunchDaemon is present, then leave broader managed LaunchDaemon lifecycle work to the existing companion issues.

## Reproduction Assessment

Yes, at source level with live reporter evidence. Current main still checks and installs only the gui-domain LaunchAgent, while the report gives a concrete headless macOS system LaunchDaemon precondition; I did not run a destructive duplicate-manager repro on macOS.

## Solution Assessment

Yes for the narrow bug: a pre-write/pre-bootstrap system-domain conflict guard is the best small fix. Full managed LaunchDaemon support is a separate product/security scope and should stay with the broader linked issues.
