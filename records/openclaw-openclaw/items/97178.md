## Summary

Keep open. Current `main` still has the reported LaunchAgent-only install/status path, and two open PRs now claim this issue with closing syntax, so the issue should stay open until one fix lands.

## What This Changes

The issue reports that `openclaw gateway install` can create a gui-domain LaunchAgent over an existing same-label system LaunchDaemon, causing launchd restart loops and client disconnects.

## Best Possible Solution

Land one focused launchd lifecycle fix that refuses or clearly reports a same-label system LaunchDaemon before creating, repairing, or restarting a user LaunchAgent, and leave first-class `--system` LaunchDaemon install support to the broader product issue.

## Reproduction Assessment

Yes at source level, with strong reporter live evidence from v2026.6.10. Current `main` still writes and bootstraps only a gui LaunchAgent while system LaunchDaemon discovery is limited to deep diagnostics.

## Solution Assessment

Yes for the narrow bug. The best fix is a pre-create/pre-repair conflict guard for same-label system LaunchDaemons; first-class managed system LaunchDaemon support is broader and already overlaps https://github.com/openclaw/openclaw/issues/51860.
