## Summary

Keep open. Current main and the latest v2026.6.10 release still install the macOS gateway service as a gui-domain LaunchAgent, while system LaunchDaemon discovery exists only in status/doctor-style inspection paths, so the reported duplicate-manager failure remains a valid narrow bug.

## What This Changes

The issue requests that `openclaw gateway install` detect a same-label system-domain LaunchDaemon before creating a conflicting gui-domain LaunchAgent.

## Best Possible Solution

Add a focused pre-write or pre-bootstrap guard in the macOS gateway installer that detects a same-label system LaunchDaemon and refuses with clear recovery or replacement guidance, without adding full LaunchDaemon lifecycle support in this issue.

## Reproduction Assessment

Yes at source level, but not through a live macOS run in this read-only review. Current main and v2026.6.10 still write and bootstrap the managed Darwin service only as a gui-domain LaunchAgent, while system LaunchDaemon discovery is separate.

## Solution Assessment

Yes for the narrow reported failure. A pre-install conflict guard is the smallest maintainable fix; a full managed LaunchDaemon mode should remain with the broader companion issues.
