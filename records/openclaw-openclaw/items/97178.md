## Summary

Keep open. Current main and v2026.6.10 still manage macOS gateway service install/status through the user LaunchAgent path, and two open PRs now explicitly own the narrow duplicate-manager guard but have not merged.

## What This Changes

The issue reports that macOS gateway install/status can miss an existing system-domain LaunchDaemon and add a conflicting user-domain LaunchAgent, causing a launchd restart storm and dropped clients.

## Best Possible Solution

Land one narrow duplicate-manager guard that refuses user LaunchAgent install/restart before mutation when a same-label system LaunchDaemon is loaded or present, while keeping broader managed LaunchDaemon mode in https://github.com/openclaw/openclaw/issues/51860.

## Reproduction Assessment

Yes, at source level with reporter live evidence. Current main still probes and installs the gui-domain LaunchAgent path without a same-label system-domain guard, matching the reported duplicate-manager trigger; I did not run a live macOS LaunchDaemon mutation in this read-only review.

## Solution Assessment

Yes for this narrow issue. A pre-write and pre-bootstrap conflict guard is the smallest maintainable fix for the duplicate-manager failure; full system LaunchDaemon lifecycle support belongs in the broader companion issue.
