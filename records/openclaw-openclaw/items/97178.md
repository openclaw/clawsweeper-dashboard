## Summary

Keep open. Current main still writes and bootstraps only a gui-domain macOS LaunchAgent, while the reporter supplied live v2026.6.10 evidence of a system LaunchDaemon false-negative leading to a duplicate-manager restart storm; two open PRs now specifically claim this issue and should be reviewed rather than closing it early.

## What This Changes

The issue asks `openclaw gateway install` and related status/doctor paths to detect an existing same-label system LaunchDaemon before creating a duplicate user LaunchAgent.

## Best Possible Solution

Review and land one narrow same-label system LaunchDaemon conflict guard for install/restart-style user LaunchAgent mutation, while leaving full managed LaunchDaemon lifecycle design to the broader companion issues.

## Reproduction Assessment

Yes, at source level with reporter live evidence. Current main and v2026.6.10 still route macOS install/status through the user LaunchAgent path and gui launchd domain, while the issue shows a real system LaunchDaemon host that this misses.

## Solution Assessment

Yes for the narrow reported bug. A pre-write/pre-bootstrap same-label system LaunchDaemon guard is the smallest maintainable fix; full `--system` LaunchDaemon lifecycle support should stay in the companion issues.
