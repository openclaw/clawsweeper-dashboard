## Summary

Keep open. Current main and v2026.6.10 still run macOS `gateway install` through the user LaunchAgent service path, so the reported same-label system LaunchDaemon conflict is not fixed.

## What This Changes

The issue reports that `openclaw gateway install` can create a user LaunchAgent while an existing system LaunchDaemon already supervises the same gateway label and port.

## Best Possible Solution

Add a narrow macOS duplicate-supervisor guard shared by install/status/doctor so a system LaunchDaemon for the same OpenClaw gateway label or port blocks user LaunchAgent installation with clear recovery guidance, without adding full LaunchDaemon lifecycle support in this issue.

## Reproduction Assessment

Yes at source level and from reporter live evidence, but not from an independent live run here. Current main still checks only the user LaunchAgent service before install, while the reporter captured a real system LaunchDaemon setup where status reports the user service missing and points operators at install.

## Solution Assessment

No, current behavior is not the best solution because install lacks the cross-domain guard. The narrow maintainable fix is to detect and refuse the duplicate supervisor before writing or bootstrapping the user LaunchAgent; full `--system` LaunchDaemon support should stay out of this repair unless maintainers separately sponsor it.
