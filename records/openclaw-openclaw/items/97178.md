## Summary

Keep open. Current `main` and the latest tagged release still use the macOS user LaunchAgent install/status path without a same-label system LaunchDaemon guard, and two linked closing PRs remain open rather than merged.

## What This Changes

The issue reports that `openclaw gateway install` can add a gui-domain LaunchAgent over an existing system-domain LaunchDaemon, causing a duplicate-manager restart storm and client disconnects.

## Best Possible Solution

Land one narrow conflict guard that blocks same-label system LaunchDaemons before gui LaunchAgent install/restart/status repair, and keep full managed `--system` LaunchDaemon support with the broader linked product issue.

## Reproduction Assessment

Yes at source level: current `main` writes and bootstraps the user LaunchAgent without checking for a same-label system LaunchDaemon, and the reporter supplied live headless macOS evidence on v2026.6.10. I did not personally mutate a live macOS LaunchDaemon during this read-only review.

## Solution Assessment

Yes for the narrow bug: a pre-write/pre-restart same-label system-domain guard is the maintainable fix. Full managed system LaunchDaemon install/stop/uninstall support is a broader product/security decision tracked separately by https://github.com/openclaw/openclaw/issues/51860.
