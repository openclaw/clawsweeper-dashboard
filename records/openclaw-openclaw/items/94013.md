## Summary

Keep open. The patch targets a real Feishu monitor startup bug: current main can select a context-only `channelRuntime` and then dereference `debounce`, and the proposed guard mirrors the already-merged Feishu bot-path guard. I found no blocking code defect, but the supplied proof is simulation/unit-level rather than real Feishu or gateway behavior, so it still needs real behavior proof before merge.

## What This Changes

The PR changes Feishu monitor startup to fall back to the registered full runtime when an injected `channelRuntime` lacks `inbound`, and adds `monitorSingleAccount` guard tests.

## Best Possible Solution

Land the narrow monitor runtime guard after contributor adds redacted real Feishu or gateway startup proof showing the partial-runtime path no longer prevents incoming message processing.

## Reproduction Assessment

Yes from source inspection: current main selects any truthy `params.channelRuntime`, while the receive handler immediately reads `channelRuntime.debounce` and the public surface only guarantees `runtimeContexts`. I did not run a live Feishu repro in this read-only review.

## Solution Assessment

Yes for the known partial-runtime failure: checking `channelRuntime?.inbound` at the monitor consumption boundary matches the already-merged Feishu bot-path guard and avoids a broader SDK contract change for this narrow bug. Merge should still wait for real behavior proof.
