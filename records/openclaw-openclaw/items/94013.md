## What This Changes

The PR changes Feishu monitor startup to fall back to the registered full channel runtime when an injected runtime lacks `inbound`, and adds guard tests for partial, full, and undefined `channelRuntime` inputs.

## Best Possible Solution

Land the narrow monitor runtime guard after redacted Feishu/gateway startup proof shows inbound processing succeeds, or after a maintainer explicitly accepts the proof override for this internal runtime path.

## Reproduction Assessment

Yes from source inspection: current main can select a context-only Feishu `channelRuntime`, and the receive handler immediately needs full debounce and command runtime APIs. I did not run a live Feishu account or gateway startup reproduction in this read-only review.

## Solution Assessment

Yes for the known partial-runtime failure: guarding at the monitor consumption boundary matches the already-merged Feishu bot-path fix and avoids a broader SDK contract change. Merge readiness still depends on real behavior proof or maintainer proof override.
