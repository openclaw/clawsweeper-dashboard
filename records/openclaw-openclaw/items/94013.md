## Summary

Keep open: current main and the latest release still use the unguarded Feishu monitor runtime selection, and this PR is the open candidate fix. The patch looks narrowly correct, but the external PR still lacks after-fix real Feishu or gateway startup proof unless a maintainer grants the requested proof override.

## What This Changes

The PR changes Feishu monitor startup to fall back to the registered full runtime when an injected `channelRuntime` lacks `inbound`, and adds guard tests for partial, full, and absent runtime inputs.

## Best Possible Solution

Land the narrow monitor runtime guard after redacted real Feishu or gateway startup proof, or after an explicit maintainer proof override, then keep the linked user report open until the merged fix is validated against inbound replies.

## Reproduction Assessment

Yes from source inspection: current main can select a partial Feishu `channelRuntime`, while the receive handler immediately needs full debounce and command runtime APIs. I did not establish a live Feishu account reproduction in this read-only review.

## Solution Assessment

Yes for the known partial-runtime failure: checking `channelRuntime?.inbound` at monitor startup matches the already-merged Feishu bot-path guard and keeps the repair inside the Feishu plugin boundary. A broader SDK/runtime contract change is not needed for this narrow fix, and a handler-level optional debounce guard would hide the wrong runtime selection rather than fixing it.
