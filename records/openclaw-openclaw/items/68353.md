## Summary

Keep open. Current main still lacks the requested first-class `chat.send.author` protocol field, durable transcript author metadata, and `chat.history` / `session.message` projection; the related open `systemInputProvenance` PR is narrower and should be coordinated rather than treated as a superseding fix.

## What This Changes

Requests an optional `author` object on `chat.send`, persisted and projected as `senderLabel`, `authorId`, and `authorName` for multi-user shared sessions.

## Best Possible Solution

Define one additive gateway author-attribution contract covering schema, validation, transcript storage, `chat.history` and `session.message` projection, UI/client typing, docs, tests, and explicit trust semantics coordinated with the open provenance and token-identity work.

## Reproduction Assessment

Yes, as a source-level capability check: current main rejects `chat.send.author`, stores no author fields on chat-send transcript rows, and has no history/event projection path for `authorId` or `authorName`. This is a missing public protocol capability rather than a broken existing bug contract.

## Solution Assessment

Unclear. An optional `author` object is plausible, but maintainers should first decide whether it is client-declared display metadata, operator-trusted provenance, or attested room identity before making it authoritative in visible history.
