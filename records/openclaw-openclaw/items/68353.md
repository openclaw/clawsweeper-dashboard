## Summary

Keep open. Current main still lacks the requested first-class `chat.send.author` contract, durable `authorId`/`authorName` storage, and history/event projection; the related open provenance PR covers only an operator-admin subset and does not supersede the broader shared-session attribution request.

## What This Changes

The issue requests an optional validated per-message `author` object for `chat.send`, persisted and surfaced through `chat.history` and `session.message` as authoritative attribution for multi-user shared sessions.

## Best Possible Solution

Define an additive gateway protocol contract for per-message authorship that covers validation, storage, `chat.history` and `session.message` projection, UI/client typing, docs, tests, and explicit display-only versus attested identity semantics.

## Reproduction Assessment

Yes, by source inspection rather than a failing runtime bug. Current main accepts no `chat.send.author` field and has no durable projection path for `authorId` or `authorName`, so the requested behavior cannot be exercised through the present protocol.

## Solution Assessment

Unclear. An additive `author` object is plausible, but the maintainable solution depends on whether maintainers want client-declared display metadata, operator-admin provenance, or attested room identity to own this attribution path.
