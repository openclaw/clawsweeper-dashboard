## Summary

Keep open. Current main still lacks the requested first-class `author` contract on `chat.send`; attribution remains derived from the existing client identity or inbound metadata, and the request changes a public protocol plus an identity trust model that needs maintainer/security/product direction before implementation.

## What This Changes

The issue requests an optional validated `author` object on `chat.send`, persistence and projection of author metadata into `chat.history` and `session.message`, optional attestation, and related sanitizer hardening.

## Best Possible Solution

Define an additive gateway protocol contract for per-message authorship that covers validation, transcript storage, `chat.history` and `session.message` projection, UI/client typing, docs, tests, and explicit trust semantics. Split the timestamp-prefixed metadata sanitizer hardening if maintainers want that smaller safety improvement first.

## Reproduction Assessment

Yes. A code-level reproduction is high-confidence: a `chat.send` payload containing `author` is rejected by the current schema because the schema has no such field and disallows additional properties; the sanitizer path also still uses exact sentinel matching for timestamp-prefixed metadata.

## Solution Assessment

Unclear. The proposed additive field is plausible, but presenting client-declared authors as authoritative is not the best solution until maintainers decide whether the field is display-only, roster/device-attested, or scoped by session kind.
