## Summary

Keep open. Current main and v2026.6.10 still lack a first-class `chat.send.author` payload, durable author fields on user turns, and `authorId`/`authorName` history/event projection; the open provenance PR only partially overlaps and is not a viable replacement while the identity trust contract remains unsettled.

## What This Changes

The issue requests an optional per-message `author` object on `chat.send` that persists and surfaces authoritative user attribution for multi-user shared sessions.

## Best Possible Solution

Define one additive Gateway authorship contract covering schema, validation, trust semantics, transcript storage, `chat.history` and `session.message` projection, client/UI typing, docs, and focused tests before implementation.

## Reproduction Assessment

Yes, as a source-level capability check. Current main rejects unknown `chat.send.author` through the closed schema, and the runtime, storage, projection, and UI send paths inspected have no first-class author metadata.

## Solution Assessment

Unclear. An optional `author` object is plausible, but maintainers should decide whether it is display metadata, operator-trusted provenance, or attested room identity before making it authoritative in history and events.
