## Summary

Keep open. Current main still lacks the requested first-class `author` wire contract, persistence, and `chat.history`/`session.message` projection path; the adjacent Control UI browser-local identity work does not solve gateway-level shared-session attribution.

## What This Changes

The issue requests an optional per-message `author` object on `chat.send`, persisted and surfaced as `senderLabel`, `authorId`, and `authorName` for shared multi-user sessions.

## Best Possible Solution

Define an additive gateway protocol contract for authorship that covers validation, storage, `chat.history`/`session.message` projection, client/UI typing, docs, tests, and explicit display-only versus attested trust semantics.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: a `chat.send` payload with `author` is outside the current `additionalProperties: false` schema, and the accepted runtime/transcript/projection path has no field that can round-trip `authorId` or `authorName`.

## Solution Assessment

Unclear. An optional additive field is plausible, but allowing it to override `senderLabel` is not the best solution until maintainers decide whether it is client-declared display metadata, session-kind-attested identity, or bound to a gateway/device roster.
