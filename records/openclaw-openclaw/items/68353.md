## Summary

Keep open. Current main still lacks the requested first-class `chat.send.author` protocol field, durable author metadata on chat-send transcript rows, and `chat.history`/`session.message` author projection; the related open provenance PR is narrower and does not supersede this shared-session author contract.

## What This Changes

The issue requests an optional validated `chat.send` author object that is persisted and surfaced as `senderLabel`, `authorId`, and `authorName` for multi-user shared sessions.

## Best Possible Solution

Define one additive gateway author-attribution contract covering schema, validation, storage, history/events, UI/client typing, docs, tests, and explicit display-only versus attested identity semantics.

## Reproduction Assessment

Yes, as a source-level capability check: current main accepts no `chat.send.author` field and has no durable projection path for `authorId` or `authorName`. This is a missing protocol capability rather than a failing existing bug contract.

## Solution Assessment

Unclear. An optional `author` object is plausible, but maintainers should first choose whether the value is client-declared display metadata, operator-trusted provenance, or attested room identity.
