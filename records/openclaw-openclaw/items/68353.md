## Summary

Keep open. Current main still lacks the requested first-class `chat.send.author` field, durable author metadata on chat-send transcript rows, and `chat.history` / `session.message` projection for `authorId` or `authorName`; the related open provenance PR is narrower and does not supersede this shared-session history attribution contract.

## What This Changes

The issue requests an additive per-message `author` object for `chat.send` that can drive `senderLabel`, `authorId`, and `authorName` in chat history and live session events for shared multi-user sessions.

## Best Possible Solution

Define one additive gateway author-attribution contract covering schema, validation, storage, history/events, UI/client typing, docs, tests, and explicit display-only versus attested identity semantics.

## Reproduction Assessment

Yes, as a source-level capability check: current main has no accepted `chat.send.author` field, no durable author fields on chat-send transcript rows, and no projection path for `authorId` or `authorName`. This is a missing public protocol capability rather than a broken existing bug contract.

## Solution Assessment

Unclear. An optional `author` object is plausible, but maintainers should first decide whether it is client-declared display metadata, operator-trusted provenance, or attested room identity before making it override visible attribution.
