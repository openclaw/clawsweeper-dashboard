## Summary

Keep open. Current main still lacks the requested first-class `author` wire contract, transcript persistence, and `chat.history`/`session.message` projection path; the related multi-token issue #43903 is adjacent identity/isolation work, not a substitute for per-message authorship. Because this changes a public Gateway protocol and identity-attribution trust model, it needs maintainer/security/product direction before implementation.

## What This Changes

Add an optional validated `author` object to `chat.send`, persist it with user transcript rows, and surface `senderLabel`, `authorId`, and `authorName` for multi-user shared sessions.

## Best Possible Solution

Define an additive Gateway authorship contract covering validation, storage on user transcript rows, display projection, SDK/client/UI typing, docs, tests, and explicit display-only versus attested identity semantics.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: `author` is rejected by the current `additionalProperties: false` schema, and the accepted runtime/transcript/event path has no field that could round-trip `authorId` or `authorName`.

## Solution Assessment

Unclear. An optional additive field is plausible, but making it override `senderLabel` is not the best maintainable solution until maintainers decide whether the value is client-declared display metadata, session-kind-attested identity, or bound to a gateway/device roster.
