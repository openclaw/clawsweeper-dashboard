## Summary

Keep open. Current main still lacks a first-class `chat.send.author` field, durable chat-send author metadata, and `chat.history` / `session.message` projection for `authorId` or `authorName`; the related token/provenance items remain adjacent open work rather than a merged replacement for this contract.

## What This Changes

The issue requests additive per-message author attribution on `chat.send`, with validated author fields stored and surfaced through chat history and session events for multi-user shared sessions.

## Best Possible Solution

Define one additive gateway authorship contract covering schema, validation, transcript storage, `chat.history` and `session.message` projection, UI/client typing, docs, tests, and explicit trust semantics for client-declared versus attested identity.

## Reproduction Assessment

Yes, as a source-level capability check: current main has no accepted `chat.send.author` field, no stored author fields on chat-send transcript rows, and no `authorId`/`authorName` history or event projection. This is a missing public protocol capability rather than a failing existing bug contract.

## Solution Assessment

Unclear. An optional `author` object is a plausible shape, but maintainers should first decide whether that data is display metadata, operator-trusted provenance, or attested room identity before making it authoritative in visible history.
