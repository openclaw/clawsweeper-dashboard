## Summary

Keep open. Current main still lacks a first-class `author` field on `chat.send`, has no persistence path for `authorId`/`authorName`, and still projects display attribution from `senderLabel` or inbound metadata; the related multi-token work in #43903 is adjacent but does not supersede this protocol request.

## What This Changes

Add an optional validated `author` object to `chat.send` and round-trip it through `chat.history` and `session.message` as authoritative per-message attribution for shared sessions.

## Best Possible Solution

Define an additive gateway authorship contract covering validation, transcript storage, history/event projection, UI/client typing, docs, tests, and explicit display-only versus attested trust semantics.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: `author` is rejected by the current `chat.send` schema and there is no storage or projection path that could round-trip `authorId` or `authorName`.

## Solution Assessment

Unclear. An optional field is plausible, but allowing it to override `senderLabel` should wait for maintainer/security direction on whether the values are client-declared display metadata or attested identity.
