## Summary

Keep open. Current main still lacks the requested first-class `chat.send.author` field, durable `authorId`/`authorName` storage, and `chat.history`/`session.message` projection; the related open provenance PR only covers an operator-admin subset and does not supersede the broader shared-session attribution request.

## What This Changes

The issue requests an optional `chat.send.author` object that round-trips as authoritative per-message author attribution in chat history and session message events.

## Best Possible Solution

Define an additive gateway protocol contract for per-message authorship that covers validation, storage, history/events, client/UI typing, docs, tests, and explicit display-only versus attested identity semantics.

## Reproduction Assessment

Yes, by source inspection rather than a failing runtime bug: current main accepts no `chat.send.author` field and has no durable projection path for `authorId` or `authorName`. The requested behavior is a new protocol capability, not a broken existing contract.

## Solution Assessment

Unclear. An additive `author` object is plausible, but the best solution needs maintainer/security/product direction on whether the values are client-declared display metadata, operator-admin provenance, or attested room identity.
