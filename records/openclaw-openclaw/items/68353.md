## Summary

Keep open. Current main and the latest shipped release still lack a first-class `author` field on `chat.send`, do not persist per-message author metadata, and cannot project `authorId`/`authorName` through `chat.history` or `session.message`; the related open provenance PR is a narrower alternative that needs maintainer protocol/security direction rather than a duplicate close.

## What This Changes

Request adding optional per-message author attribution to `chat.send` and surfacing it through stored chat history and session message events for shared multi-user sessions.

## Best Possible Solution

Define an additive gateway protocol contract for per-message authorship that covers validation, storage, `chat.history` and `session.message` projection, client/UI typing, docs, tests, and explicit display-only versus attested identity semantics.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: a `chat.send` payload with `author` is outside the current closed schema, and the accepted runtime/transcript/projection path has no field that can round-trip `authorId` or `authorName`. I did not run a live gateway because the protocol and persistence gap is directly visible in current source and shipped release source.

## Solution Assessment

Unclear. An optional additive author object is plausible, but letting it override `senderLabel` is not the best solution until maintainers decide whether the values are client-declared display metadata, operator-provenance metadata, or roster/device-attested identity.
