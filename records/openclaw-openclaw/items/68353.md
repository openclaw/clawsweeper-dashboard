## Summary

Keep open. Current main and the latest release still lack a first-class `chat.send.author` field, durable author metadata on chat-send user turns, and `authorId`/`authorName` projection; the related open provenance PR overlaps but is not a merged replacement and remains blocked on the same trust-contract question.

## What This Changes

Request an additive `chat.send.author` protocol field plus persisted and projected per-message author metadata for multi-user shared sessions.

## Best Possible Solution

Define one additive gateway authorship contract covering schema, validation, trust semantics, transcript storage, history and `session.message` projection, client/UI typing, docs, and focused tests before implementation.

## Reproduction Assessment

Yes, as a source-level capability check. Current main rejects unknown `chat.send.author` because the schema is closed, and the inspected runtime/storage/projection paths have no first-class author metadata.

## Solution Assessment

Unclear. An optional `author` object is plausible, but maintainers should decide whether it is display metadata, operator-trusted provenance, or attested room identity before making it authoritative in history and events.
