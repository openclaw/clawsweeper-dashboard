## Summary

Keep open. Current main still lacks the requested first-class `author` wire contract, persistence, and `chat.history`/`session.message` projection path; the newer browser-local Control UI identity work is adjacent local rendering, not the shared gateway attribution model. Because this changes a public protocol and identity trust semantics, it needs maintainer/security/product direction before implementation.

## What This Changes

Add optional per-message author metadata to `chat.send`, persist it, and expose it through chat history/events as `senderLabel`, `authorId`, and `authorName`, with optional attestation and sanitizer hardening follow-ups.

## Best Possible Solution

Define an additive gateway protocol contract for per-message authorship that covers validation, storage on user transcript rows, projection into `chat.history` and `session.message`, SDK/client/UI typing, docs, tests, and explicit trust semantics for display-only versus attested identity. Keep the sanitizer timestamp hardening as a separate narrow change if it can land independently.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: adding `author` to `chat.send` is outside the current `additionalProperties: false` schema, and the accepted runtime/transcript path has no field that could round-trip `authorId` or `authorName`.

## Solution Assessment

Unclear. An optional additive field is plausible, but making it override `senderLabel` is not the best solution until maintainers decide whether the value is client-declared display metadata, session-kind-attested identity, or bound to a gateway/device roster.
