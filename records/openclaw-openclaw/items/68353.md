## Summary

Keep open. Current main and v2026.6.10 still lack a first-class `chat.send.author` payload, durable author fields on chat-send user turns, and `authorId`/`authorName` projection; the related open provenance PR is conflicting and only partially overlaps rather than replacing this gateway authorship contract.

## What This Changes

Requested first-class per-message author attribution for `chat.send` so multi-user shared sessions can surface actual user identity in history instead of install-class client identity.

## Best Possible Solution

Define one additive gateway authorship contract covering schema, validation, trust semantics, transcript storage, `chat.history` and `session.message` projection, client/UI typing, docs, and focused tests before implementation.

## Reproduction Assessment

Yes, as a source-level capability check. Current main rejects unknown `chat.send.author` because the schema is closed, and the inspected runtime, storage, projection, and UI send paths have no first-class author metadata.

## Solution Assessment

Unclear. An optional `author` object is plausible, but maintainers should decide whether it is display metadata, operator-trusted provenance, or attested room identity before making it authoritative in history and events.
