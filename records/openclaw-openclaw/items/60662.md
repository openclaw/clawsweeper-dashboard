## Summary

Keep open. Current main has a low-level macOS `screen.snapshot` command, but the agent-facing screenshot/media workflow and failure-response behavior still do not cover the reported responsibility-shift case well enough to close it.

## What This Changes

The issue reports that when an agent-requested screenshot fails, the assistant can shift the same requested operation back to the user instead of providing an OpenClaw-owned failure path or alternative.

## Best Possible Solution

OpenClaw should expose an agent-facing OS-level `screen_snapshot` media path backed by `screen.snapshot`, return stable permission/capture failures, and cover the response behavior so the assistant explains OpenClaw-owned next steps rather than delegating the same failed action back to the user.

## Reproduction Assessment

No, not with high confidence. The macOS/TCC scenario is plausible and current source inspection confirms remaining gaps, but this review did not have a live denied-screen-recording run or transcript evidence for the exact assistant response.

## Solution Assessment

No current-main solution is sufficient. The low-level `screen.snapshot` primitive is useful, but the maintainable fix needs an agent-visible media action and failure contract rather than relying on generic `nodes.invoke` or model behavior.
