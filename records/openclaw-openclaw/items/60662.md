## Summary

Keep open. Current main has a low-level macOS `screen.snapshot` node command, but the agent-facing `nodes` media surface still lacks a dedicated screenshot action and generic invoke still returns raw JSON, so the reported assistant-facing failure UX is not resolved.

## What This Changes

The issue asks OpenClaw to avoid shifting an agent-owned action back to the user after a failed operation, using macOS screenshot capture as the concrete example.

## Best Possible Solution

OpenClaw should expose a dedicated agent-facing `screen_snapshot` media action backed by `screen.snapshot`, return stable permission/capture failures, and guide the assistant to explain OpenClaw-owned limitations or useful alternatives without delegating the same failed capture back to the user.

## Reproduction Assessment

No, not with high confidence for the exact assistant transcript. Source inspection shows the agent-facing screenshot path is still missing, but this review did not run a live macOS denied-screen-recording scenario or capture the current assistant wording.

## Solution Assessment

No current-main solution is sufficient. The low-level snapshot primitive is useful, but generic `nodes.invoke` is not the best user-facing screenshot flow because it lacks image/media result handling and stable failure UX.
