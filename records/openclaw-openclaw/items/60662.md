## Summary

Keep open. Current main has the low-level macOS `screen.snapshot` node command, but the agent-facing `nodes` tool still has no dedicated screen-snapshot media action and generic invoke still returns raw JSON, so the reported assistant-facing screenshot failure UX is not resolved.

## What This Changes

The issue requests better handling when an agent cannot perform a requested action, using macOS screenshot capture failure and user handoff as the example.

## Best Possible Solution

Add an agent-facing `screen_snapshot` media action backed by `screen.snapshot`, return stable permission/capture failures, and guide the assistant to explain OpenClaw-owned limitations or alternatives without delegating the same failed capture back to the user.

## Reproduction Assessment

No, not with high confidence. Source inspection shows the current agent-facing screenshot gap, but this review did not run a live macOS denied-screen-recording scenario or capture the exact assistant transcript on current main.

## Solution Assessment

No current-main solution is sufficient. The low-level `screen.snapshot` primitive is useful, but generic `nodes.invoke` is not the best user-facing screenshot flow because it lacks dedicated image/media handling and stable failure UX.
