## Summary

Keep open. Current main still has the hardened low-level macOS `screen.snapshot` command, but the agent-facing `nodes` tool still exposes only `screen_record` plus generic invoke, so the reported screenshot request and failure UX are not resolved.

## What This Changes

The issue asks OpenClaw to avoid assistant responses that hand a failed requested operation back to the user, with the screenshot/TCC-denied flow as the motivating case.

## Best Possible Solution

Add a dedicated agent-facing `screen_snapshot` media action backed by the existing `screen.snapshot` primitive, with tested result handling and failure guidance that offers OpenClaw-owned alternatives instead of redirecting the same operation back to the user.

## Reproduction Assessment

No high-confidence live reproduction is available for the exact assistant transcript on current main. Source inspection still shows the missing agent-facing screenshot action and generic invoke path, which supports the remaining gap but not the precise model wording.

## Solution Assessment

No, the current low-level primitive alone is not the best complete solution for this issue. The maintainable fix should be an explicit agent-facing media action and failure contract rather than relying on generic `nodes.invoke` plus model discretion.
