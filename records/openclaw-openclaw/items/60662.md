## Summary

Keep open. Current main has a low-level macOS `screen.snapshot` node command, but the agent-facing `nodes` media surface still has no dedicated screenshot action and generic invoke still returns raw JSON, so the reported assistant-facing failure UX is not resolved.

## What This Changes

The issue asks OpenClaw to preserve agent-owned failure handling when a requested action such as taking a screenshot cannot be completed, instead of asking the user to perform the same action manually.

## Best Possible Solution

OpenClaw should provide a dedicated agent-facing `screen_snapshot` media action backed by `screen.snapshot`, return stable permission/capture failures, and guide the assistant to explain OpenClaw-owned limitations or useful alternatives without delegating the same failed capture back to the user.

## Reproduction Assessment

No, not with high confidence. Source inspection shows the agent-facing screenshot path is still missing, but this review did not run a live macOS denied-screen-recording scenario or capture the exact assistant transcript on current main.

## Solution Assessment

No current-main solution is sufficient. The low-level snapshot primitive is useful, but generic `nodes.invoke` is not the best user-facing screenshot flow because it lacks image/media result handling and stable failure UX.
