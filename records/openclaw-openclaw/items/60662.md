## Summary

Keep open. Current main and the latest release expose the low-level macOS `screen.snapshot` command, but the agent-facing `nodes` media surface still lacks a dedicated screenshot action and the generic invoke path only returns raw JSON, so the reported assistant-facing failure UX is not resolved.

## What This Changes

The issue asks OpenClaw to avoid telling users to perform the same action themselves when an agent cannot complete a requested screenshot-like operation.

## Best Possible Solution

OpenClaw should provide an agent-facing `screen_snapshot` media action backed by `screen.snapshot`, return stable permission/capture failures, and guide the assistant to explain OpenClaw-owned limitations or alternatives without delegating the same failed capture back to the user.

## Reproduction Assessment

No, not with high confidence. Source inspection confirms a plausible gap in the agent-facing screenshot path, but this review did not run a live macOS denied-screen-recording scenario or capture the exact assistant transcript on current main.

## Solution Assessment

No current-main solution is sufficient. The low-level snapshot primitive is useful, but generic `nodes.invoke` is not the best user-facing screenshot flow because it lacks image/media handling and stable failure UX.
