## Summary

Keep open. Current main has the low-level macOS `screen.snapshot` node command, but the agent-facing `nodes` media surface still has no dedicated screenshot action and generic invoke still returns raw JSON, so the reported assistant-facing failure UX is not resolved or superseded.

## What This Changes

The issue asks OpenClaw to handle screenshot/action failures without delegating the same requested operation back to the user.

## Best Possible Solution

Add a dedicated agent-facing screenshot media action backed by `screen.snapshot`, return stable permission/capture failures, and guide assistant responses to explain OpenClaw-owned limitations or useful alternatives without delegating the same capture back to the user.

## Reproduction Assessment

No, not with high confidence for the exact reported transcript. Source inspection shows the agent-facing screenshot path is still missing, but this review did not run a live macOS denied-screen-recording scenario or reproduce the DeepSeek response on current main.

## Solution Assessment

No, the broad intent-classifier/state-machine proposal is not yet the narrowest maintainable fix. A better first step is a dedicated `screen_snapshot` agent tool plus stable macOS snapshot error contracts and failure-response guidance.
