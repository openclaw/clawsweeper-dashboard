## Summary

Current main no longer uses the affected Baileys rc.9 WhatsApp QR path; the WhatsApp plugin now depends on Baileys 7.0.0-rc10, whose upstream source includes the unified-session QR auth hooks missing from rc.9.

## What This Changes

The issue reports WhatsApp QR pairing timing out silently after scan on OpenClaw 2026.3.13 with Baileys 7.0.0-rc.9.

## Best Possible Solution

Keep the rc10 dependency path that carries upstream unified-session QR auth behavior, and handle the now-overlapping open patch PR separately if maintainers still want any unique follow-up.

## Reproduction Assessment

Yes for the affected 2026.3.x path: the report gives concrete Docker/dashboard QR steps plus same-phone proof that WhatsApp Web worked while OpenClaw stalled, and upstream rc.9 lacks the unified-session hooks. I did not run a live physical WhatsApp QR reproduction against current main.

## Solution Assessment

Yes. Upgrading the WhatsApp Baileys dependency to rc10 is a narrower maintainable solution than carrying the open local rc.9 backport because rc10 already includes the upstream unified-session implementation.
