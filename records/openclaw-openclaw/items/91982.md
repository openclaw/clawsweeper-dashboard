## Summary

Keep open: current main and the latest release still have a source-backed path where anthropic-vertex re-applies Anthropic cache policy to an already shaped payload, so this is not conservative cleanup-closeable.

## What This Changes

The issue reports that anthropic-vertex-provider can add an extra Anthropic cache_control marker when active-memory injects recalled context, causing Vertex to reject requests with too many cache breakpoints.

## Best Possible Solution

Keep the issue open and make Anthropic-family payload policy idempotent for already shaped Vertex payloads, with regression coverage that proves the total cache_control markers stay within the provider limit.

## Reproduction Assessment

Yes for a source-level reproduction, not a live one: current code shows active-memory only injects `prependContext` on recall hits, native Anthropic shaping can pre-mark system/tools/messages, and the Vertex wrapper can add another system marker on a second policy pass.

## Solution Assessment

Yes: the narrow maintainable fix is to make `src/agents/anthropic-payload-policy.ts` idempotent and limit-aware for already shaped payloads, rather than disabling active-memory or changing memory trust semantics.
