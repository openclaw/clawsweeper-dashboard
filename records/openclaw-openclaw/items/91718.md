## Label Justifications

- `P2`: The issue describes bounded but real resource leakage in the `memory_search` workflow when embedding providers stall.
- `impact:other`: The impact is backend resource retention after a timed-out tool call, which is maintainer-visible but outside the more specific owned impact labels.

## Summary

Keep open: current main and v2026.6.1 still race `memory_search` against a timer without a tool-owned `AbortSignal`, while the embedding provider contract already supports abortable calls lower in the stack.

## What This Changes

The issue requests abort propagation from the `memory_search` tool-level deadline into the memory search embedding call chain so timed-out embedding work is cancelled.

## Best Possible Solution

Thread a tool-owned abort signal through the memory-core search stack and keep the change additive and optional at any public manager contract boundary.

## Reproduction Assessment

Yes from source: a focused fake-timer path can make `manager.search` or `provider.embedQuery` never settle, advance the 15s tool deadline, and observe that no tool-owned signal exists to abort the underlying work. I did not run a live stalled Ollama/backend reproduction.

## Solution Assessment

Yes. The narrow maintainable fix is to propagate an optional deadline signal through `memory_search` -> `MemorySearchManager.search` -> `embedQueryWithRetry` and combine it with the existing embedding operation timeout signal, without adding config or changing result shape.
