## Summary

Keep open. Current main still has a source-proven path where fatal isolated cron outcomes become `isError` delivery payloads, are passed into normal announce dispatch, and can be sent through outbound channel delivery; the separate failure-notification path does not suppress that completion-announce path.

## What This Changes

The issue asks OpenClaw to prevent failed cron runs using announce delivery from posting raw provider error payloads to Discord, Telegram, or other user-facing channels.

## Best Possible Solution

Add a structured fatal-run skip before normal completion announce dispatch, while preserving run history, diagnostics, and the separate failure-notification delivery path.

## Reproduction Assessment

Yes, at source level. Fatal cron outcomes become `isError` delivery payloads, current main passes them into `dispatchCronDelivery`, and dispatch sends normal announce payloads without a fatal-status guard.

## Solution Assessment

Yes. The narrow maintainable fix is to use the existing structured fatal/error state to skip normal completion announce delivery for failed runs; a raw-text regex guard like the closed contributor PR is less robust.
