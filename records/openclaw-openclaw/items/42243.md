## Summary

Keep this open. Current main still turns fatal cron run failures into `isError` delivery payloads and sends them through the normal announce completion path, while the only directly linked fix PR is closed unmerged.

## What This Changes

The issue requests that failed cron runs using announce delivery avoid posting raw provider error payloads to user-facing channels.

## Best Possible Solution

Add a structured fatal-run skip before normal completion announce dispatch, keeping run history, diagnostics, and failure notifications intact.

## Reproduction Assessment

Yes, source-reproducible: a provider or run-level failure becomes an `isError` delivery payload, current main passes it into `dispatchCronDelivery`, and dispatch sends normal announce payloads without a fatal-status guard. I did not run a live channel delivery in this read-only pass.

## Solution Assessment

Yes, the maintainable fix is to use the existing structured fatal/error state to skip normal completion announce delivery for failed runs. A raw-text regex guard like the closed contributor PR is less robust and should not replace structured outcome handling.
