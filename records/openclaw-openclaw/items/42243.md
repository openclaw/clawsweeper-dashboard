## Summary

Keep this open. Current main still synthesizes fatal cron run failures into `isError` delivery payloads and passes them into the normal announce dispatch path, while the linked fix PR was closed unmerged and the merged provider-error sanitization work does not remove this cron delivery route.

## What This Changes

Failed cron runs using announce delivery should avoid sending raw provider error payloads to Discord/Telegram while retaining run history and failure notification behavior.

## Best Possible Solution

Use the structured fatal/error outcome to skip normal announce completion delivery for failed cron runs, keep error status/run history/failure notifications intact, and add focused regression coverage.

## Reproduction Assessment

Yes, source-reproducible: an announce cron run with a provider or run-level error becomes a fatal `isError` delivery payload and current main calls `dispatchCronDelivery` without a fatal-status skip. I did not run a live channel delivery in this read-only pass.

## Solution Assessment

Yes for the requested outcome, but the best implementation should use the existing structured fatal/error state instead of reviving PR #46236's raw-text regex guard. The failure-notification path should remain unchanged.
