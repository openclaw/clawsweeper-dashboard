## Label Justifications

- `P1`: The issue describes a real cron delivery workflow leaking raw provider error text into user-facing external channels.
- `impact:security`: Raw provider and operational error JSON can expose internal details to Discord, Telegram, or other channel recipients.

## Summary

Keep open: current main and the latest release still allow fatal cron `isError` payloads to flow into normal announce delivery, and the open linked fix PR should be reviewed or landed before this issue closes.

## What This Changes

Prevent failed cron runs using announce delivery from posting raw provider error JSON into Discord, Telegram, or other user-facing channels.

## Best Possible Solution

Land a focused fix like the linked PR after proof: suppress normal completion announce for fatal structured cron errors while preserving run diagnostics and failure notifications.

## Reproduction Assessment

Yes, at source level. Current main builds fatal `isError` cron delivery payloads, passes them to `dispatchCronDelivery`, and outbound delivery still sends text-only error payloads normally.

## Solution Assessment

Yes. The maintainable fix is to gate fatal structured cron errors before normal announce completion delivery, because generic outbound delivery intentionally still supports text-only `isError` payloads for other callers.
