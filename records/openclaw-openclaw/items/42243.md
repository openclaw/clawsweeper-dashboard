- `P1`: The issue describes a current external-channel disclosure path affecting real cron delivery workflows.
- `impact:security`: Raw provider and operational error text can be posted into Discord/Telegram channels where it may expose sensitive implementation details.

## Summary

Keep open: current main still synthesizes fatal cron/provider failures into `isError` delivery payloads and passes them through normal announce delivery, while the related contributor PR is closed unmerged and the fixed sandbox/subagent paths cover different leaks.

## What This Changes

The issue requests that failed cron runs in `delivery.mode: "announce"` record/log provider errors without posting raw error text to Discord, Telegram, or other user-facing channels.

## Best Possible Solution

Use structured fatal cron state to suppress normal completion announce delivery for failed isolated runs, while preserving run history, diagnostics, and the separate failure-notification path.

## Reproduction Assessment

Yes, at source level. Current main builds fatal `isError` cron delivery payloads, passes them to `dispatchCronDelivery`, and outbound delivery still sends text-only error payloads normally.

## Solution Assessment

Yes. The narrow maintainable fix is to suppress normal completion announce when the existing fatal/error state is set; the closed raw-text guard PR was weaker because it missed structured `deliveryPayloads`.
