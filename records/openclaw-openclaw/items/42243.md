## Summary

Keep this open. Current main sanitizes some raw provider JSON before user-facing text, but source inspection still shows fatal cron error payloads can flow through the normal announce delivery path, and the linked fix PR was closed without merging.

## What This Changes

The issue asks for failed isolated cron runs with announce delivery to avoid posting raw provider error output to Discord, Telegram, or other channels.

## Best Possible Solution

Suppress normal announce completion delivery for fatal cron run outcomes using the structured fatal/error-payload state, while keeping run history, error status, and explicit failure-notification routing intact.

## Reproduction Assessment

Yes, from source inspection: provider/run-level failures become `isError` delivery payloads, then the announce dispatch and outbound layers can still send them. I did not execute a live channel reproduction in this read-only pass.

## Solution Assessment

The closed raw-text regex guard from PR #46236 is not the best long-term shape; the safer fix is to use the existing structured fatal/error-payload signal in the cron announce path and add regression coverage around run-level provider errors.
