## Summary

Keep open. Current main still converts fatal isolated-cron/provider failures into error delivery payloads and passes them through normal announce delivery, while the focused related PR is closed and unmerged.

## What This Changes

Cron announce delivery should suppress failed/error runs instead of sending raw or synthesized error text to configured Discord, Telegram, or other channels.

## Best Possible Solution

Add a structured fatal-run guard at isolated cron finalization or dispatch so normal `delivery.mode: "announce"` fallback delivery is skipped whenever the run outcome is fatal or the selected delivery payload is an error, while preserving `lastRunStatus: "error"`, cron run history, and configured failure notifications.

## Reproduction Assessment

Yes. The static path is high-confidence: provider/run-level errors synthesize `isError: true` payloads, `run.ts` still dispatches them, and `delivery-dispatch.ts` sends non-empty payloads without an error-status guard.

## Solution Assessment

Yes, the requested direction is the narrowest maintainable fix if implemented from structured run outcome/error-payload state. A raw-text regex guard like the closed PR proposed would be weaker because the code already has `hasFatalErrorPayload` and `isError` signals available.
