## Summary

Keep this PR open: the reported cron timeout/ledger problem is plausible and supported by production evidence plus related user reports, but the branch is draft/conflicting and currently regresses or misses important current-main cron timeout contracts before it can merge.

## What This Changes

The PR adds a cron `agentTurn` `runTimeoutSeconds` wall-clock timeout, proactive task-ledger finalization on timeout, and focused timer regression tests.

## Best Possible Solution

Rebase onto current main, layer the wall-clock ceiling into the existing execution-start timeout and cleanup flow, then wire `runTimeoutSeconds` through schema, update merging, docs, and focused cron/task tests.

## Reproduction Assessment

Yes, source-reproducible: current main has no `runTimeoutSeconds` contract, and a fake isolated runner that ignores abort would exercise the missing wall-clock ceiling. The PR body also reports production stale/lost task counts and after-hot-patch improvement.

## Solution Assessment

No, not as currently proposed. The direction is useful, but the patch must preserve current timeout cleanup semantics and expose the new option through all public cron contracts before merge.

  - body: This wrapper is based on older timer code: it races `executeJobCore` directly and would bypass current main's execution-start timeout deferral and `cleanupTimedOutCronAgentRun` path.   - confidence: 0.86
