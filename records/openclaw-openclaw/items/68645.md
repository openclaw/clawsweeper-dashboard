## Summary

Keep open: this targets a real cron lifecycle gap and current main still lacks a cron `runTimeoutSeconds` wall-clock contract, but the draft/conflicting branch is not mergeable as-is because it is based on an older timeout flow and misses the public contract wiring.

## What This Changes

The PR adds a cron `agentTurn` `runTimeoutSeconds` wall-clock timeout, proactively closes task-ledger rows on timeout errors, and adds timer regression tests.

## Best Possible Solution

Rebase onto current main, layer the wall-clock ceiling into the existing phased timeout cleanup flow, and wire the new cron contract through schemas, tools, docs, update merging, and focused regression tests.

## Reproduction Assessment

Yes, source-reproducible: current main has no cron `runTimeoutSeconds` path, and the PR’s fake isolated runner that ignores abort demonstrates the missing wall-clock ceiling. The PR body also reports production stale/lost task counts improving after the hot patch.

## Solution Assessment

No, not as currently proposed. The direction is useful, but the branch must preserve current main’s timeout cleanup semantics and wire the new public cron contract before merge.
