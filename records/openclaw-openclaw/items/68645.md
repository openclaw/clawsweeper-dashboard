## Summary

Keep open: the production problem and the direction are credible, and related comments connect it to the isolated-cron timeout/lifecycle family in https://github.com/openclaw/openclaw/issues/68156 and https://github.com/openclaw/openclaw/issues/68157. ## What This Changes

The branch adds cron `agentTurn` `runTimeoutSeconds` handling, proactive task-ledger closing on timer errors, and focused timeout regression tests.

## Best Possible Solution

Rebase onto current main, layer the wall-clock ceiling into the existing timeout cleanup flow, wire the new field through schemas/tooling/docs/update paths, and get owner agreement on the default policy before merge.

## Reproduction Assessment

Yes, source-reproducible: current main has no cron `runTimeoutSeconds` contract, and the PR's test simulates an isolated runner that ignores the inner abort. The PR body also reports production stale task counts improving after the hot patch.

## Solution Assessment

No, not as currently proposed. The direction is useful, but the branch should be rebased onto the current timeout cleanup implementation and the new public cron contract must be fully wired before merge.

  - body: This wrapper is based on older timer code: it calls `executeJobCore` directly and races a new timer, bypassing current main's execution-start deferral, phase tracking, and `cleanupTimedOutCronAgentRun` path.
