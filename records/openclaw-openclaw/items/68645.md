## Summary

Keep open. The PR is a plausible fix for real cron timeout/task-ledger failures, but the draft branch is now conflicting and its timer wrapper is based on older main semantics, so it would regress current isolated-run timeout cleanup and still misses protocol/update plumbing for the new field.

## What This Changes

The PR adds a cron agentTurn `runTimeoutSeconds` wall-clock timeout, proactively finalizes timed-out task-ledger rows, and adds focused timer regression coverage.

## Best Possible Solution

Rebase onto current main, implement the wall-clock ceiling as an additive timeout contract that preserves existing cleanup semantics, then wire the field through schema, update merging, docs, and focused cron/task tests.

## Reproduction Assessment

Yes, source-reproducible: current main has no `runTimeoutSeconds` path, and an isolated-runner fake-timer test that ignores abort would exercise the missing wall-clock ceiling. The PR body also reports production stale/lost task counts and post-hot-patch improvement.

## Solution Assessment

No, not as currently proposed. The direction is useful, but the branch must be rebased onto current timeout cleanup semantics and the new field must be wired through the public cron contract before merge.

## Review Findings

- **[P2] Preserve current timeout cleanup when rebasing:** `src/cron/service/timer.ts:117`
  - body: This wrapper is based on older timer code: it calls `executeJobCore` directly and rejects on timeout without preserving current main's execution-start deferral and `cleanupTimedOutCronAgentRun` path.   - confidence: 0.88

  - body: Adding `runTimeoutSeconds` to the internal payload type is not enough. The cron RPC payload schemas are strict and still omit this field, and update merging in `jobs.ts` does not carry it, so API clients cannot create or update jobs with the new option reliably.
  - confidence: 0.9
- **[P2] Strip legacy root runTimeoutSeconds after copying it:** `src/cron/normalize.ts:370-372`
  - body: The top-level passthrough copies `runTimeoutSeconds` into `payload`, but `stripLegacyTopLevelFields` is not updated to delete the root field. Legacy-style inputs would retain an out-of-schema top-level key instead of mirroring the existing `timeoutSeconds` cleanup path.
  - confidence: 0.86
