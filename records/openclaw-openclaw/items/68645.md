## Summary

Keep this PR open: it addresses a real cron timeout/bookkeeping problem, but the branch is draft, merge-conflicting, and would regress newer current-main timeout cleanup unless it is rebased and completed through the cron API contract.

## What This Changes

The PR adds a cron agentTurn `runTimeoutSeconds` wall-clock timeout, proactive task-ledger finalization on timeout errors, and focused timer regression tests.

## Best Possible Solution

Rebase onto current main, layer the wall-clock ceiling into the existing timeout cleanup flow, and wire the new field through schema, update merging, normalization, docs, and focused cron/task tests.

## Reproduction Assessment

Yes, source-reproducible: current main has no cron `runTimeoutSeconds` path, and the proposed fake-runner test exercises a run that ignores the inner abort. The PR body also reports production hot-patch metrics with `stale_running` dropping from 29 to 0.

## Solution Assessment

No, not as currently proposed. The direction is useful, but the branch must be rebased onto current timeout cleanup semantics and the new payload field must be wired through the public cron contracts before merge.

  - body: This wrapper is based on older timer code: it calls `executeJobCore` directly and rejects on timeout without preserving current main's execution-start deferral and `cleanupTimedOutCronAgentRun` path.   - confidence: 0.88

  - confidence: 0.9

  - confidence: 0.86
