## Summary

Keep this PR open, but it is not merge-ready: the bug is real and the direction is useful, yet the branch is draft/conflicting and its timeout wrapper is based on older timer code that would bypass current cleanup semantics while the new public field is not wired through all cron contracts.

## What This Changes

The PR adds a cron `agentTurn` wall-clock `runTimeoutSeconds` field, closes task-ledger rows from the timer error path, and adds focused timer regression tests.

## Best Possible Solution

Rebase onto current main, layer the wall-clock ceiling into the existing timeout cleanup flow, then wire the new field through gateway schema, cron tool/docs, update merging, and focused regression tests.

## Reproduction Assessment

Yes, source-reproducible: current main has no cron `runTimeoutSeconds` path, and the PR's fake isolated runner that ignores abort demonstrates the missing wall-clock ceiling. The PR body also reports production stale/lost task counts improving after the hot patch.

## Solution Assessment

No, not as currently proposed. The direction is useful, but the branch must preserve current main's timeout cleanup semantics and wire the new public cron contract before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve current timeout cleanup when rebasing:** `src/cron/service/timer.ts:117`
  - body: This wrapper is based on older timer code: it calls `executeJobCore` directly and races a new timer, bypassing current main's execution-start deferral and `cleanupTimedOutCronAgentRun` path. Applying this shape as-is would regress stuck isolated-run cleanup, so the wall-clock timer needs to be layered into the current implementation.
  - confidence: 0.88
- **[P2] Wire runTimeoutSeconds through cron contracts:** `src/cron/types.ts:104`
  - body: Adding `runTimeoutSeconds` only to the internal payload type leaves the strict gateway cron schemas, cron tool schema/help, docs, and update merge/build paths unaware of the field. Clients and model tools would not be able to create or update jobs with this option reliably.
  - confidence: 0.9
- **[P2] Strip legacy root runTimeoutSeconds after copying it:** `src/cron/normalize.ts:373-374`
  - body: The flat-input recovery copies top-level `runTimeoutSeconds` into `payload`, but `stripLegacyTopLevelFields` does not delete the root key. Legacy-style inputs would retain an out-of-schema top-level field instead of matching the existing `timeoutSeconds` cleanup path.
  - confidence: 0.86
