## Summary

Keep open. The production report and follow-up comments point to a real cron/task-ledger failure mode, but this draft PR is stale against current cron timeout cleanup and is incomplete for the public cron contract.

## What This Changes

The PR adds a cron `agentTurn` `runTimeoutSeconds` wall-clock timeout, proactively closes timed-out cron task-ledger rows, and adds regression tests.

## Best Possible Solution

Rebase the patch onto current main, preserve the existing timeout-start and cleanup behavior, add a hard wall-clock ceiling only with the approved default policy, and wire the field through protocol schemas, update merging, docs, and focused cron/task tests.

## Reproduction Assessment

Yes. The PR gives production stale/lost task counts and current main has fake-timer timeout cleanup tests; source inspection also confirms there is still no separate `runTimeoutSeconds` path.

## Solution Assessment

No, not as currently proposed. The direction is plausible, but the branch is incomplete for the cron RPC/update contract and must be rebased so it does not regress current timeout cleanup behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Preserve current timeout cleanup on rebase:** `src/cron/service/timer.ts:117`
  - body: This timeout wrapper is based on older timer code: it calls `executeJobCore` directly and rejects on timeout without preserving current main's execution-start deferral and `cleanupTimedOutAgentRun` path. Rebasing this as-is would regress stuck isolated-run cleanup that current tests cover, so the wall-clock timer needs to be layered onto the current implementation.
  - confidence: 0.88
- **[P2] Wire runTimeoutSeconds through cron contracts:** `src/cron/types.ts:104`
  - body: Adding `runTimeoutSeconds` to the internal payload type is not enough. The cron RPC payload schemas are strict and still omit this field, and update merging in `jobs.ts` does not carry it, so API clients cannot create or update jobs with the new option reliably.
  - confidence: 0.9
- **[P2] Strip the copied root runTimeoutSeconds field:** `src/cron/normalize.ts:373-374`
  - body: The top-level passthrough copies `runTimeoutSeconds` into `payload`, but `stripLegacyTopLevelFields` is not updated to delete the root field. Legacy-style inputs would retain an out-of-schema top-level key instead of mirroring the existing `timeoutSeconds` cleanup path.
  - confidence: 0.86
