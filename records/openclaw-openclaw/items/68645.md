## Summary

Keep open. The production-backed cron/task-ledger problem is credible, but current main only partially overlaps the proposed fix and the branch is incomplete against the public cron contract, so it should not be closed or merged as-is.

## What This Changes

The PR adds a cron agent-turn `runTimeoutSeconds` wall-clock timeout, proactively finalizes timed-out cron task-ledger rows, and adds fake-timer regression coverage.

## Best Possible Solution

Either keep `timeoutSeconds` as the single documented outer cron deadline, or land an additive `runTimeoutSeconds` design that is wired through cron types, Gateway schemas, normalization, service update merging, docs/CLI surfaces as needed, and regression tests while preserving current timeout cleanup and task reconciliation.

## Reproduction Assessment

Yes. The PR gives production counts and focused fake-timer tests, and current-main inspection confirms the remaining API gap: there is timeout cleanup but no cron `runTimeoutSeconds` contract.

## Solution Assessment

No, not as currently proposed. The direction may be useful, but the patch is incomplete against strict RPC schemas and legacy normalization, and it needs a rebase over current timeout cleanup plus a maintainer decision on timeout policy.
