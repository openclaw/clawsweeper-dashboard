## Summary

Keep open: the reported cron timeout/task-ledger problem is plausible and current main still lacks a cron `runTimeoutSeconds` path, but this draft PR is stale against current timeout cleanup behavior and is incomplete for the public cron contracts.

## What This Changes

The PR adds a cron `agentTurn` `runTimeoutSeconds` wall-clock timeout, proactively finishes timed-out task-ledger rows, and adds timer regression tests.

## Best Possible Solution

Rebase onto current main, layer the wall-clock ceiling into the existing timeout cleanup flow, settle the default policy, and wire the field through schema, update merging, docs, and focused cron/task tests.

## Reproduction Assessment

Yes, source-reproducible: current main has no cron `runTimeoutSeconds` path, and a fake-timer isolated-runner test that ignores abort would exercise the missing wall-clock kill. The PR body also gives production stale/lost task counts and post-hot-patch improvement.

## Solution Assessment

No, not as currently proposed. The direction is plausible, but the branch must be rebased onto current timeout cleanup semantics and the new field must be wired through the full cron contract before merge.
