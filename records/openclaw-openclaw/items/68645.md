## Summary

## What This Changes

The PR adds cron agentTurn wall-clock `runTimeoutSeconds` handling, proactive task-ledger finalization on timer errors, normalization/type support, and timer regression tests.

## Best Possible Solution

Rebase or replace the branch on current main, integrate the wall-clock ceiling with the existing watchdog/cleanup flow, and wire the new cron field through schemas, tools, docs, update merging, and focused regression tests.

## Reproduction Assessment

Yes, at source level. Current main has no cron `runTimeoutSeconds` wall-clock ceiling, and the PR’s unresponsive-runner test models the missing guard when an isolated run ignores abort; the PR body also reports production stale-running rows clearing after the hot patch.

## Solution Assessment

No, not as currently proposed. The direction is useful, but the patch must preserve current main’s phase-aware timeout cleanup and wire the new public cron field end to end before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve the current timeout cleanup flow:** `src/cron/service/timer.ts:117`
  - body: This wraps `executeJobCore` directly in a new `Promise.race`. Current main now defers isolated agentTurn timeout arming until execution starts, tracks phase milestones, and calls `cleanupTimedOutCronAgentRun` before returning the timeout result; rebasing this shape would drop that cleanup and can reintroduce stuck isolated-session state.
  - confidence: 0.88

  - confidence: 0.9

  - body: The flat-input recovery copies top-level `runTimeoutSeconds` into `payload`, but `stripLegacyTopLevelFields` does not delete the root key. Legacy-style inputs would retain an out-of-schema top-level field instead of matching the existing `timeoutSeconds` cleanup path.
  - confidence: 0.86
