## Summary

Keep open. Current main still appears to leave shared-session heartbeat runtime model/context metadata behind after a heartbeat with `heartbeat.model`; the existing code scopes the model choice for the heartbeat run, but post-run session persistence and `session_status` still make the heartbeat model look like the active session model. The existing docs even describe heartbeat model bleed as a context-overflow scenario, so this is not safely closeable as implemented.

## What This Changes

The issue asks for `agents.defaults.heartbeat.model` to be scoped to the heartbeat turn so the main session returns to its configured primary model afterward.

## Best Possible Solution

Keep the heartbeat model override as an explicit per-run selection, but prevent heartbeat-only runtime model/context metadata from becoming the selected shared-session state after the run. Add a regression test that runs a shared-session heartbeat with a smaller `heartbeat.model`, then verifies `session_status` and the next normal turn resolve to the configured primary model/window; preserve `isolatedSession` behavior and existing heartbeat.model precedence.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to configure shared-session `heartbeat.model` differently from the primary, run `runHeartbeatOnce`, and inspect the session store/status path: the heartbeat model is persisted as runtime identity and then displayed as the active status model.

## Solution Assessment

No. The current implementation solves the earlier bug where heartbeat.model was ignored, but it does not complete the scoped override contract because shared-session runtime model/context state is still overwritten by the heartbeat run.
