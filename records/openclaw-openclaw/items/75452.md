## Summary

Keep open. Current main still applies `heartbeat.model` for the heartbeat run, then writes that run's provider/model/context metadata back to the shared session store, and the issue is already paired with open PR #75567 using closing syntax.

## What This Changes

The issue requests that `agents.defaults.heartbeat.model` apply only to heartbeat turns and not persist as the shared session runtime model afterward.

## Best Possible Solution

Land a narrow core fix that preserves or restores the pre-heartbeat shared-session runtime model and context metadata after heartbeat runs while keeping `heartbeat.model` as the per-run selection. The fix should include regression coverage around `runHeartbeatOnce`, `updateSessionStoreAfterAgentRun`, and `session_status`, and should update the heartbeat docs/changelog if the documented workaround becomes obsolete.

## Reproduction Assessment

Yes. Configure a shared-session heartbeat with a different `heartbeat.model`, run a heartbeat, and inspect the session store/status path: current main selects the heartbeat model for the run and then persists that runtime identity for later status and turns.

## Solution Assessment

Yes for the requested direction: scoping `heartbeat.model` to the heartbeat run and preserving the prior shared runtime model is the narrow maintainable contract. Current main does not yet implement that contract, and open PR #75567 should be reviewed or replaced with an equivalent fix.
