## Summary

Keep open. Current main still applies `heartbeat.model` only during heartbeat model selection, then persists that run's runtime model and context metadata back into the shared session; open PRs #75557 and #75567 reference this with closing syntax, so the issue should remain open until one lands or is replaced.

## What This Changes

A shared-session heartbeat using `agents.defaults.heartbeat.model` should not leave subsequent user turns pinned to the heartbeat model after the heartbeat finishes.

## Best Possible Solution

Land #75567 or an equivalent narrow core fix that preserves or restores the previous shared-session runtime model and context metadata after heartbeat runs, with regression coverage for `updateSessionStoreAfterAgentRun` and `session_status`.

## Reproduction Assessment

Yes. Configure a shared-session heartbeat with a different `heartbeat.model`, let the heartbeat run, then inspect `session_status` or the session store; current main selects the heartbeat model and then writes that runtime identity back to the shared session.

## Solution Assessment

Yes directionally. A heartbeat-specific preserve/restore path in session persistence is narrower than changing model selection semantics or requiring isolated sessions for correctness, and #75567 is the active implementation candidate.
