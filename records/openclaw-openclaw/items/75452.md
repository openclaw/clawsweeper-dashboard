## Summary

Keep open. Current main still scopes `heartbeat.model` for selection but then persists the heartbeat run's runtime model/context metadata onto the shared session, and the status/recovery surfaces still treat that as heartbeat model bleed. The related fix PR #75557 was closed without merge, so there is no implemented fix to close against.

## What This Changes

The issue asks for `agents.defaults.heartbeat.model` to behave as a single-turn heartbeat override and stop leaving the shared session on the heartbeat model after the heartbeat completes.

## Best Possible Solution

Keep `heartbeat.model` as an explicit per-run model selection, but prevent heartbeat-only runtime model/context metadata from replacing the shared session's selected/runtime state after the run. Add regression coverage for a shared-session heartbeat using a smaller heartbeat model, then verify `session_status` and the next normal turn resolve to the configured primary model/window while `isolatedSession` behavior remains intact.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to run a shared-session heartbeat with `heartbeat.model` different from the primary, have the run report the heartbeat provider/model/context, and inspect the session store/status path: current main persists and displays that heartbeat runtime identity.

## Solution Assessment

Yes. The narrow maintainable fix is to make heartbeat runtime identity persistence transient for shared-session heartbeats, not to rely on docs or force `isolatedSession`; model-selection precedence for normal user turns should remain unchanged.
