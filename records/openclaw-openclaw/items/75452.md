## Summary

Keep open. Current main still has the reported source-level behavior: heartbeat runs can select `heartbeat.model` for that run, but the shared session persistence path still writes the run result's runtime provider/model/context metadata back to the same session with no heartbeat-scoped preserve/restore flag. The provided GitHub context also shows open PR #75567 uses closing syntax for this issue, so the issue should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue reports that `agents.defaults.heartbeat.model` is intended as a single heartbeat-turn override but persists into the shared session after the heartbeat completes, causing later user turns to use the heartbeat model.

## Best Possible Solution

Review and land PR #75567 or an equivalent narrow core fix that preserves or restores the prior shared-session runtime provider/model/context fields after heartbeat runs, with regression coverage around heartbeat persistence and `session_status`.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is available: configure a shared-session heartbeat with a different `heartbeat.model`, let the heartbeat run, then inspect the shared session store or `session_status`; current main writes the heartbeat run's runtime identity back into the shared session.

## Solution Assessment

Yes directionally. A heartbeat-scoped preserve/restore path in session persistence is the narrowest maintainable fix; changing global model-selection semantics or requiring `isolatedSession` would not directly address the shared-session persistence bug.
