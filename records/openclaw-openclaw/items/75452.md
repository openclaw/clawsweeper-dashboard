## Summary

Keep open. Current main still scopes `heartbeat.model` for selecting the heartbeat run, but the completed run's provider/model/context metadata is then written back to the shared session store and `session_status` displays that runtime identity. Open PR #75567 references this issue with closing syntax, so the issue should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue reports that `agents.defaults.heartbeat.model` is intended to override only the heartbeat turn but remains active for later shared-session user turns.

## Best Possible Solution

Land a narrow core fix that keeps `heartbeat.model` as a per-run selection while preserving or restoring the prior shared-session runtime model and context metadata after heartbeat runs, with regression coverage for `runHeartbeatOnce`, `updateSessionStoreAfterAgentRun`, and `session_status`. Preserve isolated heartbeat behavior and the existing precedence where `heartbeat.model` wins only for the heartbeat run.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to run a shared-session heartbeat with a different `heartbeat.model`, then inspect the session store/status path: current main selects the heartbeat model for the run and then persists that runtime identity for later status and turns.

## Solution Assessment

No for current main. The existing implementation fixes the earlier bug where `heartbeat.model` was ignored, but it does not complete the scoped override contract; PR #75567 appears to be the active narrow fix candidate and should be reviewed or replaced with equivalent coverage.
