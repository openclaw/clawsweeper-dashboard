## Summary

Keep open: current main still has browser-control backend primitives but no Control UI browser preview tab, route, renderer, or live screenshot/log consumer. The related visible-browser/login-handoff tracker at https://github.com/openclaw/openclaw/issues/63926 is still open and narrower, and the FSB reference comment usefully reinforces separating read-only live view from explicit write actions.

## What This Changes

The issue requests a live Browser Preview Panel in the Dashboard for remote/container users, showing screenshots, URL/title, console errors, and optional manual control of the built-in browser.

## Best Possible Solution

Ship a security-reviewed, admin-scoped Browser area in Control UI that reuses browser.request or additive typed gateway methods for live read-only state, with explicit operator steps for navigation, clicks, credentials, and submit actions.

## Reproduction Assessment

Not applicable as a runtime bug because this is a feature request. Source inspection gives high confidence that current main has browser-control APIs but no Control UI browser tab, route, render branch, or live preview consumer.

## Solution Assessment

Unclear until maintainers scope the UX and security model. The requested direction is useful, but the maintainable path should coordinate with https://github.com/openclaw/openclaw/issues/63926 and the plugin UI descriptor surface instead of a narrow automated patch.
