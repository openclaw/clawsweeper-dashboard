## Summary

Keep open: current main and the latest release still expose browser-control primitives without a live Control UI browser preview panel, and the related login-handoff and plugin-UI RFC threads remain open rather than replacing this broader request.

## What This Changes

The issue requests a live browser preview panel in the Dashboard for remote/container users, including screenshots, URL/title, console errors, and optional manual browser interaction.

## Best Possible Solution

Design an admin-scoped, read-only-first Control UI browser surface that reuses browser.request and browser-control where safe, adds explicit handoff gates for write actions, and coordinates with the open login-handoff and plugin-UI API threads.

## Reproduction Assessment

Yes at the source level: current main and v2026.6.6 have no Control UI browser route, renderer, or documented live preview panel, while backend browser-control primitives exist separately. This is a missing feature rather than a live runtime failure that needs a failing test.

## Solution Assessment

No; the issue's panel sketch is useful but not yet the full best solution. The safer maintainable path is an admin-scoped, read-only-default UI with explicit write-action handoff and a deliberate decision about whether to extend the generic plugin UI contribution surface.
