## Summary

Current main and v2026.6.6 still expose browser-control primitives without a Control UI browser route, renderer, or live-preview consumer. The narrower login-handoff tracker and generic plugin-UI RFC remain open rather than replacing this broader request, so the item is still necessary but requires product and security decisions before implementation.

## What This Changes

Add a live Control UI browser panel for remote users to view the managed browser, inspect URL, title, and errors, and optionally perform guarded interactions.

## Best Possible Solution

Design an admin-scoped, host-rendered browser surface that is read-only by default, reuses browser.request and browser-control primitives, separates sensitive write actions behind explicit user handoff, and deliberately establishes or avoids a reusable safe plugin descriptor contract.

## Reproduction Assessment

Yes at the source level: current main and v2026.6.6 have no Control UI browser route or renderer, while the browser plugin exposes separate request and control primitives. This is a missing capability rather than a runtime failure requiring a live failing test.

## Solution Assessment

No, the issue's direct panel sketch is not yet a complete best-fix design. The maintainable path is a host-owned, admin-scoped, read-only-first UI with explicit write-action handoff and a deliberate decision about the generic plugin contribution RFC.

## Risks / Open Questions

- A live view can expose authenticated pages, tokens, personal data, console output, and screenshots to any operator allowed to open the panel, so authorization and redaction boundaries must be designed first.
- Combining passive observation with clicks, credential entry, navigation, or final-submit actions would expand the operator control boundary; read and write capabilities should not be implicitly granted together.
- A browser-specific hardcoded UI path could conflict with the open generic Control UI contribution design and create a second long-term plugin/UI ownership model.
