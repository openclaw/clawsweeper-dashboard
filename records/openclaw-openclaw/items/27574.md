## Summary

Keep open. Current main still has browser backend/control primitives, but it does not provide the requested live browser preview panel in the Control UI; the existing related #63926 is narrower and still open rather than a replacement for this broader remote preview/dashboard request.

## What This Changes

The issue requests a live browser preview panel in the OpenClaw dashboard for remote/container users, including screenshots, URL/title visibility, console errors, and optional interaction.

## Best Possible Solution

Keep this issue open as the broad remote browser-preview/dashboard tracker. The desired end state is a maintainer-designed, admin-only Control UI browser surface that reuses browser.request/browser-control primitives where safe, coordinates with #63926 for visible browser management and login handoff, and ships with targeted UI/backend coverage plus docs if implemented.

## Risks / Open Questions

- A live remote browser panel is security-sensitive because it may expose page contents, logged-in sessions, console/network data, screenshots, and possibly click/navigation control through the Control UI.
- The broader preview/debugging request overlaps with open #63926, so maintainers should coordinate scope instead of letting two browser UI designs diverge.
- It is still undecided whether this belongs as a core Control UI view, a richer browser plugin contribution surface, or a combination of generic SDK additions plus browser-owned UI metadata.
