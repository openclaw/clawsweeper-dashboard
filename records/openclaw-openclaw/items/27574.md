## Summary

Keep open: current main and the latest release still lack a live Dashboard/Control UI browser preview panel, while the existing browser-control and plugin descriptor plumbing are only primitives for a future design.

## What This Changes

The issue requests a live Dashboard/Control UI panel for watching the OpenClaw-managed browser, including screenshots, URL/title, console errors, and optional manual interaction.

## Best Possible Solution

Design an admin-scoped, read-only-first browser preview that reuses browser.request/browser-control where safe, explicitly gates write actions, and coordinates with the open browser-handoff and plugin UI contribution work.

## Reproduction Assessment

Yes, at the source level: current main has browser-control primitives but no Control UI browser route, tab, renderer, live preview consumer, or docs for such a panel. This is a missing feature rather than a failing current behavior path.

## Solution Assessment

No, not as sketched alone. The panel idea is useful, but the best maintainable solution needs an admin-scoped read-only default, explicit write-action gates, and a maintainer decision on whether the browser UI should use generic plugin contribution slots.
