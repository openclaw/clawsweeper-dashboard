## Label Justifications

- `P2`: The panel is a meaningful roadmap-aligned remote-browser usability enhancement, but it is not an urgent regression in defined existing behavior.
- `impact:security`: The proposed panel would reveal sensitive authenticated browser state and may enable privileged remote interaction.

## Summary

Current main and v2026.6.5 still expose browser-control primitives without a browser tab, route, render branch, or live preview consumer. The request remains distinct from the narrower login-handoff tracker and generic plugin UI RFC, so it should stay open for coordinated product and security design.

## What This Changes

Add a live Control UI browser panel showing remote browser screenshots, URL/title, console errors, and optional interaction for remote deployments.

## Best Possible Solution

Define an admin-only, host-rendered browser surface with read-only observation by default, explicit guarded write actions, bounded update and retention behavior, and a reusable native-panel plugin contract if the current descriptor API is insufficient.

## Reproduction Assessment

Yes at source level: current main and v2026.6.5 provide browser backend operations but no Control UI browser tab, route, render branch, or live consumer. This is a verified capability gap rather than a runtime bug reproduction.

## Solution Assessment

The broad tracker is appropriate, but the suggested 1–2 FPS panel plus optional unrestricted interaction is not yet a complete best solution. The safer maintainable direction separates read-only visibility from explicit write actions and resolves the host/plugin native-panel contract first.
