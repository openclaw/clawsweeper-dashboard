## Summary

Keep open: current main still has browser-control backend primitives but no Control UI/Dashboard browser preview panel, and the related login-handoff tracker remains open and narrower. The feature is security-sensitive because it would expose authenticated browser pixels, URLs, logs, and possible control actions in an operator UI.

## What This Changes

The issue requests a live Browser Preview Panel in the Dashboard/Control UI for remote browser visibility and troubleshooting.

## Best Possible Solution

Ship a maintainer-scoped, admin-only Browser area in Control UI that reuses browser.request or additive typed gateway methods for live read-only state, with explicit operator steps for navigation, clicks, credentials, and final submit actions.

## Reproduction Assessment

Not applicable as a runtime bug because this is a feature request. Source inspection gives high confidence that current main has browser-control APIs but no Control UI browser tab, route, render branch, or live preview consumer.

Unclear until maintainers scope the UX and security model. The requested direction is useful, but the maintainable path should coordinate with https://github.com/openclaw/openclaw/issues/63926 and define a secured Control UI browser surface rather than a narrow automated patch.
