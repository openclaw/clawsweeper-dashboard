## Summary

Keep open. Current main still runs live memory-search query embeddings through a single-shot provider call, while the existing retry/backoff helper is only used by batch embedding paths and the retry classifier does not include the reported `fetch failed`, socket close, ECONNRESET/EPIPE, or TLS-disconnect failures. Recent issue comments also confirm the problem after 2026.4.26, and the related open PR is batch-indexing scoped rather than a complete live-query fix.

## What This Changes

The issue asks OpenClaw to make live memory-search single-query embedding requests resilient to transient provider-agnostic socket/TLS failures in the SSRF-guarded remote embedding path.

## Best Possible Solution

Add a narrow bounded retry/backoff path for live query embeddings using the existing memory embedding retry helper and an expanded transport-error classifier. Keep auth/config/provider response failures fail-fast, and defer global Undici dispatcher tuning or new user-facing config until the scoped retry is proven insufficient.

## Reproduction Assessment

Yes. The issue provides concrete live loops and recent provider-diverse confirmations, and current main gives a deterministic unit seam: make `provider.embedQuery()` fail once with a transient socket-style error and then succeed, then assert `MemoryManager.search()` retries and returns results.

## Solution Assessment

Yes for the scoped repair direction. Reusing the existing memory embedding retry policy for live query embeddings is narrower and safer than first changing global SSRF fetch dispatchers, provider configuration, or keep-alive pooling behavior.
