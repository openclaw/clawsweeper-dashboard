## Summary

Keep open. Current main still has a single-shot live memory-search query embedding path, the existing retry loop is only used by batch embeddings, and the retry classifier still excludes the reported socket/TLS transport failures. The related open PR is useful but batch-scoped, so it does not close this live-query report.

## What This Changes

The issue requests reliable handling for intermittent remote embedding transport failures in live `openclaw memory search` query embeddings.

## Best Possible Solution

Add a bounded retry/backoff path for live query embeddings, reusing or lightly generalizing the existing memory embedding retry helper and extending the classifier to narrow transient undici/TLS/socket failures. Keep auth/config/provider response errors fail-fast, and defer global dispatcher tuning or new config knobs until the scoped retry has evidence it is insufficient.

## Reproduction Assessment

Yes. The issue gives concrete provider-diverse live `openclaw memory search` loops, and current main has a deterministic unit seam: make `provider.embedQuery()` fail once with `fetch failed`/socket wording and then succeed, then assert live search retries instead of surfacing `Memory search failed`.

## Solution Assessment

Yes, the narrow retry direction is the best first fix. It targets the failing live-query surface without changing global guarded-fetch behavior, provider selection, config schema, or non-retryable error handling.

## Repair Work Prompt

Fix the live memory-search remote embedding transport retry gap for this issue. Current main retries batch embeddings but `embedQueryWithTimeout()` calls `provider.embedQuery(text)` once, so transient undici/TLS/socket failures surface as `Memory search failed: fetch failed | ...`.

Expected boundary:
- In `extensions/memory-core`, make live query embeddings use a bounded retry/backoff path equivalent to the existing batch embedding retry loop.
- Extend the retry classifier only for narrow transient transport/socket failures reported here, such as `fetch failed`, `other side closed`, `ECONNRESET`, `EPIPE`, `UND_ERR_SOCKET`, socket hangup/closed, and `Client network socket disconnected before secure TLS connection was established`.
- Keep auth/config/schema errors, 4xx provider responses other than already-retryable quota/rate-limit behavior, and oversized input errors fail-fast.
- Do not add provider-specific branches, global dispatcher keep-alive changes, new config knobs, dependency changes, or broad fetch-guard behavior changes unless the narrow query retry cannot be implemented safely.
- Preserve existing batch behavior and coordinate with related #44166/#44167 so the live-query fix does not conflict with batch split/retry work.
- Add a user-facing `CHANGELOG.md` fix entry under the active version with appropriate reporter credits such as `Thanks @kevinheinrichs, @heithmurray, and @kaufmanchang`; do not credit bots.

Suggested tests:
- Add/extend a deterministic memory-core test where `provider.embedQuery()` fails once with a retryable transport error and then succeeds, proving `manager.search()` retries and returns results.
- Add classifier tests for the reported transport strings/codes and negative tests for non-retryable provider/auth errors.
- Keep provider tests focused; only touch memory-host SDK or provider files if the chosen implementation requires it.
