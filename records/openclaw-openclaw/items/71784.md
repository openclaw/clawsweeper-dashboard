## Summary

Keep open. Current main still has no retry for live query embedding transport failures: `MemoryManager.search()` performs a single `embedQueryWithTimeout()` call, the retry loop/classifier is batch/service-error oriented, and related PR #44167 remains open and scoped to batch indexing rather than this live search path.

## What This Changes

Make live builtin memory search resilient to transient remote embedding fetch, TLS, and socket failures across providers.

## Best Possible Solution

Add bounded retry/backoff for live query embeddings using the shared transport-error classifier, keep auth/config/non-retryable HTTP failures fail-fast, and leave dispatcher tuning as a follow-up only if retries are insufficient.

## Reproduction Assessment

Yes. The issue includes concrete live repro loops across OpenAI, Gemini, and OpenRouter-style providers, and current main has a deterministic unit seam: make `provider.embedQuery()` fail once with a socket-style transport error during `MemoryManager.search()` and assert the search retries and returns results.

## Solution Assessment

Yes. A scoped live-query retry is the narrowest maintainable fix for the observed user failure; broader undici pool changes or a config knob are riskier and can remain follow-up work after a retry fix is covered by tests.
