## Summary

Keep open. Current main and v2026.5.7 classify the reported socket-style embedding errors as retryable, but live memory search still sends the per-query embedding through `embedQueryWithTimeout()` without the retry loop; reporter follow-ups also show the same failure class after 2026.4.26.

## What This Changes

Request is to make live memory search retry transient remote embedding transport failures such as `fetch failed`, `other side closed`, `ECONNRESET`, and `EPIPE`.

## Best Possible Solution

Add a bounded retry/backoff around live query embeddings using the existing memory embedding retry classifier, while keeping auth/configuration and non-retryable provider errors fail-fast.

## Reproduction Assessment

Yes, at source level. Mock `provider.embedQuery()` to fail once with `TypeError("fetch failed | other side closed")` and current `MemoryManager.search()` will abort through `embedQueryWithTimeout()` because that path is not wrapped in `runMemoryEmbeddingRetryLoop()`.

## Solution Assessment

Yes. A scoped live-query retry using the existing classifier is the narrowest maintainable fix; dispatcher tuning or a new config knob should remain follow-up work unless retry evidence proves it is insufficient.
