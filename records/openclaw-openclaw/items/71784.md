## Summary

Keep open. Current main and v2026.5.18 still route live builtin memory-search query embeddings through a timeout-only helper, while the existing retry classifier already covers the reported socket failures; related open PRs cover batch indexing or opt-in query controls, not a shipped default fix for this issue.

## What This Changes

The issue requests reliable live memory search by handling transient guarded remote embedding socket failures during per-query embedding calls.

## Best Possible Solution

Apply the existing bounded memory embedding retry loop to live query embeddings, preserve immediate failure for auth/config/non-retryable errors, and add focused regression coverage for one transient `embedQuery()` failure followed by success.

## Reproduction Assessment

Yes, at source level. Mock `provider.embedQuery()` to throw `TypeError("fetch failed | other side closed")` once and current main will abort through `embedQueryWithTimeout()` because the live query path is not wrapped in `runMemoryEmbeddingRetryLoop()`.

## Solution Assessment

Yes. Reusing the existing retry classifier and bounded retry loop for live query embeddings is the narrowest maintainable fix; dispatcher tuning or new config controls can remain follow-up work if default retry proves insufficient.
