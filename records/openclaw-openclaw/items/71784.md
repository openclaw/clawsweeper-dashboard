## Summary

Keep open. Current main has retry classification for the reported socket-style embedding errors, but live memory search still calls the query embedding provider once; recent issue comments also report the same failure class after later releases, and the related open PR is batch-indexing scoped rather than a live-query fix.

## What This Changes

The issue asks OpenClaw to make live memory-search query embeddings resilient to intermittent SSRF-guarded remote fetch socket failures across providers.

## Best Possible Solution

Add a bounded retry/backoff around live query embeddings using the existing memory embedding retry classifier, while keeping auth/configuration and non-retryable provider errors fail-fast.

## Reproduction Assessment

Yes, at source level. Mock `provider.embedQuery()` to fail once with `TypeError("fetch failed | other side closed")` and current `MemoryManager.search()` will abort through `embedQueryWithTimeout()` because the query path is not wrapped in `runMemoryEmbeddingRetryLoop()`.

## Solution Assessment

Yes. A scoped live-query retry using the existing classifier is the narrowest maintainable fix; dispatcher tuning or a new config knob should remain follow-up work unless retry evidence proves it is insufficient.
