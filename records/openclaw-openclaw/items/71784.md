## Summary

Keep open. Current main has retry classification for the reported transport errors, but live memory-search query embeddings still bypass the retry loop and call the provider once; the related open PR is batch-indexing scoped and does not close this live query gap.

## What This Changes

The issue reports intermittent live memory-search embedding failures on remote providers when guarded fetch hits transient socket/TLS errors such as `fetch failed`, `other side closed`, `ECONNRESET`, and `EPIPE`.

## Best Possible Solution

Add a scoped retry/backoff around live query embeddings using the existing memory embedding retry policy, while keeping auth/configuration and non-retryable provider errors fail-fast.

## Reproduction Assessment

Yes. A deterministic current-main repro path is to mock `provider.embedQuery()` to fail once with a reported socket-style error, then assert `MemoryManager.search()` retries and returns results; current source shows it would still abort before vector search.

## Solution Assessment

Yes. A bounded live-query retry using the existing classifier is the narrowest maintainable fix; dispatcher tuning or a new config knob should remain follow-up work unless retry evidence proves it is insufficient.
