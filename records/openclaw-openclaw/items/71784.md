## Summary

Keep open. Current main still retries batch embeddings but live memory search query embeddings remain single-shot, and later issue comments plus an open targeted PR show the same socket-error class is still a real remaining memory-core bug.

## What This Changes

The issue asks OpenClaw to make live builtin memory search resilient to transient remote embedding transport failures such as `fetch failed`, `other side closed`, `ECONNRESET`, and TLS socket disconnects.

## Best Possible Solution

Land a scoped live-query embedding retry around the existing retry classifier/backoff path, with regression coverage for one transient `embedQuery()` failure followed by success; keep dispatcher tuning, new config knobs, QMD lock contention, and batch reindex work separate unless evidence shows they are required.

## Reproduction Assessment

Yes, at source level. Mock `provider.embedQuery()` to throw `TypeError("fetch failed | other side closed")` once and current `MemoryManager.search()` will abort because `embedQueryWithTimeout()` is timeout-only and not wrapped in `runMemoryEmbeddingRetryLoop()`.

## Solution Assessment

Yes. Reusing the existing bounded memory embedding retry classifier for live query embeddings is the narrowest maintainable fix; dispatcher tuning or a user config knob should be follow-up only if retry coverage is insufficient.
