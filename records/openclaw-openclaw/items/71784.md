## Summary

Keep open. Current main still has a single-shot live query embedding path for memory search, while batch embeddings are the only path using the existing retry loop; the recent provider-diverse comments and related open PR do not show this live-query surface fixed.

## What This Changes

The issue requests reliable live memory-search embedding queries when remote providers hit transient guarded-fetch/Undici socket failures.

## Best Possible Solution

Live query embeddings should share a bounded retry/backoff policy with batch embeddings for narrow transient transport/socket failures, while auth/config/schema errors and non-retryable provider responses continue to fail fast. Dispatcher keep-alive tuning or user-facing retry config should be deferred unless the scoped retry does not resolve the observed failures.

## Reproduction Assessment

Yes. The issue provides concrete OpenAI, Gemini, and OpenRouter live search loops with matching socket errors, and current source gives a deterministic unit seam by making provider.embedQuery() fail once with a reported transient transport error and then succeed.

## Solution Assessment

Yes for the scoped repair direction. Reusing or lightly generalizing the existing memory embedding retry helper for live query embeddings is narrower and safer than first changing global guarded-fetch dispatcher behavior or adding config knobs.

## Repair Work Prompt

Fix the live memory-search embedding reliability bug reported in https://github.com/openclaw/openclaw/issues/71784. Current main retries batch embeddings but `MemoryManagerEmbeddingOps.embedQueryWithTimeout()` directly wraps `provider.embedQuery(text)` with a timeout, so `openclaw memory search` can fail on transient guarded-fetch/Undici socket errors such as `fetch failed`, `other side closed`, `ECONNRESET`, `EPIPE`, and `Client network socket disconnected before secure TLS connection was established`. Keep the fix scoped to memory live-query embeddings: reuse or lightly generalize the existing retry helper in `extensions/memory-core/src/memory/manager-embedding-policy.ts` and `manager-embedding-ops.ts`, classify only narrow transient transport/socket failures as retryable, and do not change provider selection, auth/config/schema, timeout constants, cache format, SSRF policy, global Undici dispatcher behavior, or QMD SQLite lock handling. Add focused unit coverage that a query embed failing once with a transport error retries and succeeds, while non-retryable auth/config/provider HTTP errors still fail fast. Coordinate with related batch retry work in #44167/#44166 if it lands first, but this fix must cover the live query path. Add a user-facing `CHANGELOG.md` fix entry crediting the reporter and confirming commenters where appropriate, avoiding bot/maintainer handles. Validate with the focused memory tests and then `pnpm check:changed` via Blacksmith Testbox before handoff.
