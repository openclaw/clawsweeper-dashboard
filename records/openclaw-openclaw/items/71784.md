## Summary

Keep open. Current main still leaves live memory-search query embeddings on a single-shot remote provider call, while the existing retry path only covers batch indexing and still does not classify the reported socket/TLS transport failures as retryable.

## What This Changes

The issue requests reliable live memory search by retrying transient remote embedding transport failures such as `fetch failed`, `other side closed`, `ECONNRESET`, `EPIPE`, and TLS disconnects.

## Best Possible Solution

Add a narrow bounded retry/backoff path for live query embeddings using the existing memory embedding retry helper, and expand the classifier to cover common transient transport/socket errors while keeping auth, config, and provider HTTP status failures fail-fast. Leave broader Undici dispatcher tuning or new config knobs for a follow-up only if the scoped retry is insufficient.

## Reproduction Assessment

Yes. The issue provides concrete live repro loops and provider-diverse confirmations, and current main gives a focused unit seam: make `provider.embedQuery()` fail once with a transient socket-style error and then succeed, then assert `MemoryManager.search()` retries and returns results.

## Solution Assessment

Yes for the proposed scoped repair direction. Reusing the existing memory embedding retry policy for live query embeddings is narrower and more maintainable than first changing global guarded-fetch dispatchers, provider config, or keep-alive pooling behavior.

Fix openclaw/openclaw#71784 with a narrow memory-core reliability PR.
