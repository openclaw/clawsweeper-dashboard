## Summary

Keep open. Current main still leaves live memory-search query embeddings on a single-shot remote fetch path, while the existing retry/backoff helper is wired to batch embedding paths only; the issue comments add recent provider-diverse live confirmations, and the related open PR is batch-indexing scoped rather than a fix for this live-query surface.

## What This Changes

The issue requests reliable live memory-search embedding queries across remote providers by handling transient undici/TLS socket failures such as `fetch failed`, `other side closed`, `ECONNRESET`, and `EPIPE`.

## Best Possible Solution

Live query embeddings should share a bounded retry/backoff policy for narrow transient transport failures with batch embeddings, while auth/config/schema errors and non-retryable provider responses continue to fail fast. Dispatcher keep-alive tuning or a user-facing config knob should wait until the scoped retry path is proven insufficient.

## Reproduction Assessment

Yes. The issue and comments provide concrete live loops across OpenAI, Gemini, and OpenRouter-like providers, and current source gives a deterministic unit seam: make `provider.embedQuery()` fail once with a reported transient socket error and then succeed, then assert `manager.search()` retries and returns results.

## Solution Assessment

Yes for the scoped repair direction. Applying the retry classifier to live query embeddings is narrower and safer than first changing global SSRF-guarded fetch dispatchers or adding new configuration.
