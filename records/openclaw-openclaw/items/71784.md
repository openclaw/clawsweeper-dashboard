## Summary

Keep open. Current main and the latest release still run live memory-search query embeddings through a timeout-only path, while the existing retry classifier already recognizes the reported transport failures and the open related PR is batch/reindex-scoped rather than a fix for live queries.

## What This Changes

The issue requests reliable live memory-search query embeddings when remote providers intermittently raise socket/TLS transport errors such as `fetch failed` or `other side closed`.

## Best Possible Solution

Wrap live query embeddings in the existing bounded memory embedding retry loop, preserve fast failure for auth/config/non-retryable errors, and add focused regression coverage for a one-time transient `embedQuery()` failure.

## Reproduction Assessment

Yes, at source level. Mock `provider.embedQuery()` to throw `TypeError("fetch failed | other side closed")` once and current main will abort through `embedQueryWithTimeout()` because the live query path is not wrapped in `runMemoryEmbeddingRetryLoop()`.

## Solution Assessment

Yes. Reusing the existing retry classifier and bounded retry loop for live query embeddings is the narrowest maintainable fix; dispatcher tuning or new config knobs should be follow-up only if retry evidence proves insufficient.
