## Summary

Keep open. Current main still sends live memory-search query embeddings through a single provider call with no retry for the reported socket/TLS transport failures, and the related open PR is scoped to batch indexing rather than this live query path.

## What This Changes

The issue asks for live memory-search query embeddings to survive transient remote embedding transport/socket failures instead of intermittently failing or hanging.

## Best Possible Solution

Add a bounded retry/backoff path for live query embeddings using a shared narrow transport-error classifier, while keeping authentication, configuration, and non-retryable HTTP failures fail-fast.

## Reproduction Assessment

Yes. The issue includes concrete live repro loops across OpenAI, Gemini, and OpenRouter, and current main has a deterministic unit seam: make `provider.embedQuery()` fail once with `fetch failed | other side closed`, then assert `MemoryManager.search()` retries and returns results.

## Solution Assessment

Yes. A scoped live-query retry is the narrowest maintainable first fix for the user-visible failure; dispatcher keep-alive tuning can remain follow-up work if live validation shows retries are insufficient.

## Repair Work Prompt

Fix the live memory-search query embedding transport failure reported in https://github.com/openclaw/openclaw/issues/71784. Current main retries batch embeddings for selected service errors but `MemoryManager.search()` calls `embedQueryWithTimeout()` once; add a bounded retry/backoff path for live query embeddings that treats narrow transient transport failures (`fetch failed`, `other side closed`, `ECONNRESET`, `EPIPE`, `UND_ERR_SOCKET`, TLS socket disconnect/terminated/socket hang up) as retryable while preserving fail-fast behavior for auth/config/client HTTP errors. Reuse or extend the existing memory embedding retry policy instead of adding provider-specific branches. Add deterministic tests for a query embed failing once then succeeding and for non-retryable failures still surfacing. Review PR #44167 for compatible classifier wording but do not broaden this fix to batch splitting unless needed. Add a user-facing CHANGELOG.md fix entry with appropriate reporter credit; do not change provider config/schema, cache format, or broad dispatcher defaults unless the query retry cannot be implemented safely.
