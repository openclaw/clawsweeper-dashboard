## Summary

Keep open. Current main still has a single-shot live query embedding path that can propagate the reported transport errors through `openclaw memory search`, and the related open retry PR is batch-indexing scoped rather than a fix for live query recall.

## What This Changes

The issue asks OpenClaw to make live memory-search query embeddings resilient to intermittent remote-provider fetch, TLS, and socket reset failures.

## Best Possible Solution

Add a focused bounded retry/backoff for live query embeddings, extend retry classification only for narrow transient socket/transport failures, and keep auth/config/schema and persistent HTTP errors fail-fast. Broader Undici pool tuning or user-facing retry knobs should wait for evidence after the scoped live-query repair.

## Reproduction Assessment

Yes. The issue provides concrete live loops across OpenAI, Gemini, and OpenRouter, and current source gives a deterministic unit seam: make `provider.embedQuery()` fail once with `fetch failed | other side closed` and then succeed, and `MemoryManager.search()` currently rejects instead of retrying.

## Solution Assessment

Yes for a scoped live-query retry. It is the narrowest maintainable fix for the observable failure; dispatcher policy tuning or new config would be a riskier first step and should be reserved for follow-up if retries do not resolve the user-visible failures.
