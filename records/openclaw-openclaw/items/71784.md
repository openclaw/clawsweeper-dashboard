## Summary

Keep open. Current main and v2026.5.7 still send live memory-search query embeddings through a single-shot `embedQueryWithTimeout()` path instead of the existing memory embedding retry loop, and reporter follow-ups show the same transport error class after 2026.4.26.

## What This Changes

The issue requests reliable live memory search by retrying transient remote embedding socket/TLS failures in the per-query embedding path.

## Best Possible Solution

Wrap live query embeddings in the existing bounded memory embedding retry loop while preserving fast failure for auth, configuration, and non-retryable provider errors; leave dispatcher tuning or config knobs for follow-up evidence if retry is insufficient.

## Reproduction Assessment

Yes, at source level. Mock `provider.embedQuery()` to fail once with `TypeError("fetch failed | other side closed")`; current `MemoryManager.search()` will abort through `embedQueryWithTimeout()` because that path is not wrapped in `runMemoryEmbeddingRetryLoop()`.

## Solution Assessment

Yes. A scoped live-query retry using the existing classifier is the narrowest maintainable fix; provider-specific branching, dispatcher tuning, or a new config knob should stay follow-up unless retry evidence proves insufficient.
