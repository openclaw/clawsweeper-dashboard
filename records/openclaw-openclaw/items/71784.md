## Summary

Keep open. The report is recent, provider-diverse, and still matches current main: builtin live memory search awaits a single remote query embedding before vector lookup, while only batch embedding uses the retry helper. The open related PR is scoped to batch indexing for #44166 and does not cover this live-query failure surface.

## What This Changes

The issue asks OpenClaw to make live builtin memory search resilient to transient remote embedding transport failures such as `fetch failed`, `other side closed`, `ECONNRESET`, `EPIPE`, and TLS socket disconnects.

## Best Possible Solution

Add a focused bounded retry/backoff path for live query embeddings, sharing the existing memory embedding retry policy where possible and narrowly classifying transient transport/socket failures while preserving immediate failures for auth, config, schema, and non-retryable HTTP errors. Broader Undici dispatcher tuning can remain a follow-up if the scoped retry does not address the user-visible failure.

## Reproduction Assessment

Yes. The issue provides a concrete `openclaw memory search` loop with multiple remote providers, and current-main source inspection shows the live builtin search path still performs a single un-retried `provider.embedQuery()` call whose failure reaches the CLI.

## Solution Assessment

Yes. A targeted live-query embedding retry is the narrowest maintainable first fix; adding a config knob or broad SSRF/Undici pool changes should wait until there is evidence the scoped retry is insufficient.
