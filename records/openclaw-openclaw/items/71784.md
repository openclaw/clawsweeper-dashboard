## Summary

Keep open. The report is recent, has provider-diverse follow-up confirmations, and current main still has a single-shot live query embedding path that can surface the reported `fetch failed`/socket reset errors through `openclaw memory search`; the related open retry work is scoped to batch indexing and does not close this live-query surface.

## What This Changes

The issue requests a reliability fix for live builtin memory search so remote embedding transport/socket failures are retried or otherwise handled instead of aborting semantic recall.

## Best Possible Solution

Add a bounded retry/backoff for live query embeddings, reuse or lightly generalize the existing memory embedding retry helper, expand retry classification only for narrow transient transport/socket failures, and keep auth/config/schema/non-retryable HTTP failures fail-fast. Broader Undici pool tuning or new config knobs should wait for evidence after the scoped live-query fix.

## Reproduction Assessment

Yes. The issue provides a concrete `openclaw memory search` loop with OpenAI and Gemini plus later OpenRouter confirmation, and current source gives a deterministic unit seam by making `provider.embedQuery()` fail once with a reported transport error and then succeed.

## Solution Assessment

Yes for the likely repair direction. A focused live-query retry is the narrowest maintainable fix for the observable failure; broad dispatcher policy changes or user-facing retry config would be riskier first steps.
