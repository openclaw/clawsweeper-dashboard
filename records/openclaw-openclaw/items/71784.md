## Summary

Keep open. Current main still has a single-shot live memory-search query embedding path, the shared retry classifier does not cover the reported transport/socket failures, and the related open retry PR is scoped to batch indexing rather than this live-query surface.

## What This Changes

The issue asks OpenClaw to make live memory search query embeddings resilient to intermittent provider-agnostic Undici/TLS socket failures.

## Best Possible Solution

Add a bounded retry/backoff for live query embeddings using the existing memory embedding retry helper, expand retry classification only for narrow transient transport/socket failures, and keep auth/config/schema/non-retryable HTTP failures fail-fast. Broader Undici pool tuning or user-facing retry knobs should wait until the scoped live-query fix is in place and measured.

## Reproduction Assessment

Yes. The issue provides a concrete `openclaw memory search` loop across OpenAI, Gemini, and later OpenRouter, and current main gives a deterministic test seam by mocking `provider.embedQuery()` to fail once with `fetch failed | other side closed` before succeeding.

## Solution Assessment

Yes, the narrow live-query retry is the best first repair. The existing batch retry direction is useful but does not solve this issue by itself, and changing dispatcher pooling globally would have a larger blast radius.

## Repair Work Prompt

Fix live memory search query embedding retries for https://github.com/openclaw/openclaw/issues/71784. Current main retries batch embeddings but `embedQueryWithTimeout()` directly calls `provider.embedQuery(text)`, so transient transport failures such as `fetch failed`, `other side closed`, `ECONNRESET`, `EPIPE`, Undici socket/connect errors, and TLS disconnect wording fail `openclaw memory search` immediately. Add a bounded retry/backoff for the live query embedding path, preferably reusing or lightly generalizing the existing memory embedding retry helper. Expand retry classification only for narrow transient transport/socket failures and keep auth/config/schema, 4xx, payload-size, and other non-retryable provider errors fail-fast. Coordinate with #44167/#44166 if that batch retry work has landed or is still open; do not introduce provider-specific branching, config/schema changes, broad Undici dispatcher policy changes, or remote HTTP security/SSRF behavior changes unless strictly required. Add focused unit coverage for query retry success-after-one-failure and fail-fast non-retryable behavior, plus classifier coverage for the reported socket strings. Add a user-facing CHANGELOG.md fix entry with proper reporter credit.
