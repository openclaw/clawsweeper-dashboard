## Summary

Keep this PR open: the Matrix non-raw JSON path is still unbounded on current main and latest release, and the branch is a focused owner-local hardening, but maintainers still need to accept or tune the new fail-closed 8 MiB cap before merge.

## What This Changes

The branch adds an 8 MiB default byte cap and shared bounded-reader handling for Matrix `performMatrixRequest` non-raw JSON responses, plus regression tests for content-length rejection, streamed overflow, idle timeout, and under-cap success.

## Best Possible Solution

Land the bounded `performMatrixRequest` JSON read after maintainers accept or tune the 8 MiB cap, update or waive the JSON idle-timeout diagnostic, and track matrix-js-sdk response buffering separately if broader Matrix hardening is desired.

## Reproduction Assessment

Yes. Current main and v2026.6.8 still use `await response.text()` for non-raw Matrix responses, and the PR body reports a localhost negative control where the old path kept buffering instead of rejecting.

## Solution Assessment

Yes for the narrow `performMatrixRequest` bug: reusing the existing bounded reader and content-length parser is the clean owner-local fix. The exact 8 MiB default remains a maintainer compatibility decision, and the matrix-js-sdk fetch wrapper should be separate follow-up work if broader hardening is required.
