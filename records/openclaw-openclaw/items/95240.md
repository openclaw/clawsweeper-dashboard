## Summary

Keep open for normal PR review: the central Matrix transport hardening is useful and source-backed, but maintainers still need to accept the new 8 MiB fail-closed cap and there is a small JSON-path diagnostic cleanup to make before or after merge.

## What This Changes

The PR adds an 8 MiB default cap and bounded reader for Matrix `performMatrixRequest` non-raw responses, plus tests for content-length rejection, streamed overflow, idle timeout, and under-cap success.

## Best Possible Solution

Land the bounded `performMatrixRequest` JSON read once maintainers accept or tune the 8 MiB compatibility tradeoff, fix or waive the JSON idle-timeout diagnostic, and track matrix-js-sdk `/sync` buffering separately if broader Matrix hardening is desired.

## Reproduction Assessment

Yes by source inspection and PR proof: current main still uses `await response.text()` for non-raw Matrix responses, and the PR body reports a localhost HTTP harness negative control showing the stream kept buffering. I did not rerun the harness because this review is read-only.

## Solution Assessment

Yes for the `performMatrixRequest` bug: reusing the existing bounded reader and content-length parser is the narrow owner-local fix. It is not a complete Matrix transport-wide solution because the matrix-js-sdk fetch wrapper still buffers responses, so that should be a separate maintainer-tracked follow-up or an explicit non-goal.
