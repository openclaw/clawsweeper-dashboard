## Summary

Close as ClawHub/plugin work: DingTalk support is useful, and the official external connector exists, but current OpenClaw policy and documented channel-plugin APIs route this kind of optional service-specific channel through external plugin distribution and ClawHub rather than bundling another large core plugin.

## What This Changes

The PR adds a bundled DingTalk channel under `extensions/dingtalk` with registration, config schema, runtime store, Stream-mode transport, text/media messaging, and new runtime dependencies.

## Best Possible Solution

Keep DingTalk as the official external connector and publish or promote it through ClawHub; open narrow SDK/API issues only for specific external-plugin gaps that the current contracts cannot support.

Not applicable: this is a new bundled channel feature, not a current-main bug report. The review path is source/docs/policy inspection plus PR diff review, not a failing reproduction.

No: adding a 9,937-line service-specific channel to core is not the narrowest maintainable path while the official external plugin and documented channel-plugin setup/runtime surfaces exist. The safer path is ClawHub publication plus focused SDK follow-ups if a concrete missing seam is proven.
