## Summary

Keep open. Current main still has no supported path to pass arbitrary inbound OpenAI-compatible HTTP headers into an agent run or Python skill environment, and the requested credential passthrough needs an explicit security/product contract.

## What This Changes

Request to pass a caller-supplied custom HTTP header through the OpenAI-compatible Gateway request into skill code.

## Best Possible Solution

Define an allowlisted per-request credential mapping with redaction, audit behavior, sandbox rules, docs, and gateway/skill tests; otherwise document static `skills.entries.<skill>.env` or `apiKey` as the supported credential path.

## Reproduction Assessment

Yes for the missing capability: source inspection shows `/v1/chat/completions` and `/v1/responses` never copy arbitrary request headers into `AgentCommandIngressOpts` or skill env injection. I did not run a live gateway because this review was kept read-only.

## Solution Assessment

Unclear. Passing arbitrary headers directly is not the best maintainable solution; a safer path is an explicit allowlisted mapping contract, or clear docs that only static config-backed skill credentials are supported.
