## Summary

Keep open. Current main still has no supported path to pass arbitrary inbound OpenAI-compatible HTTP headers into an agent run or Python skill environment, and the related provider-header issue/PR is adjacent rather than a duplicate because it targets outbound LLM provider request headers.

## What This Changes

Request to pass a custom HTTP header sent to the OpenAI-compatible Gateway into a Python skill for authentication.

## Best Possible Solution

Define an explicit allowlisted per-request credential mapping with redaction, audit behavior, sandbox rules, docs, and gateway/skill tests, or document static skills.entries.<skill>.env/apiKey as the supported credential path.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection gives a high-confidence verification path: current handlers only read fixed x-openclaw headers and do not carry arbitrary request headers into AgentCommandIngressOpts or skill env injection.

## Solution Assessment

Unclear pending maintainer design. Passing arbitrary headers directly is not the best maintainable solution; a safer path is an explicit allowlisted mapping contract, or clear docs that only static config-backed skill credentials are supported.
