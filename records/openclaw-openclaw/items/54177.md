## Summary

Keep open. Current main still has no supported arbitrary inbound header passthrough from OpenAI-compatible HTTP requests into agent runs or skill environments; related provider/MCP header items are adjacent and do not cover this credential flow.

## What This Changes

Request to pass a caller-provided custom HTTP header through the OpenAI-compatible HTTP endpoint and read it from a Python skill script.

## Best Possible Solution

Define an explicit per-request credential/header mapping contract with allowlists, redaction, audit behavior, scope/session lifetime, sandbox behavior, docs, and gateway/skill tests, or document static skills.entries env/apiKey as the only supported credential path.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection gives a high-confidence verification path: current main only retains fixed x-openclaw routing/session/channel headers and static skill env config, with no arbitrary inbound header map passed into agent or skill execution.

## Solution Assessment

Unclear pending maintainer design. Direct arbitrary header passthrough is not the best maintainable shape; an explicit allowlisted credential mapping or documented static skills.entries env/apiKey path is safer.
