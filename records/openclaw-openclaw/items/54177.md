## Summary

Keep open. Current main still lacks a supported arbitrary inbound HTTP header passthrough from OpenAI-compatible requests into agent runs or skill environments, and the linked provider-header work is adjacent outbound-provider configuration rather than this caller-to-skill credential flow.

## What This Changes

The issue requests a way to pass a custom HTTP header through the OpenAI-compatible completions gateway and read it from a Python skill.

## Best Possible Solution

Define an explicit per-request credential/header mapping contract with allowlists, redaction, scope/session lifetime, sandbox behavior, docs, and Gateway/skill tests, or document static `skills.entries` env/apiKey as the supported credential path.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection gives a high-confidence verification path: current main only carries fixed routing/session/channel fields and static skill env config, with no arbitrary inbound header map passed to skills.

## Solution Assessment

Unclear pending maintainer design. Direct arbitrary header passthrough is not the best maintainable shape; an explicit allowlisted credential mapping or documented static skill credential path is safer.
