## Summary

Keep open. Current main still has no supported path to copy an arbitrary inbound OpenAI-compatible HTTP header into the agent command, run context, or Python skill environment; the only supported headers are fixed OpenClaw routing/control headers, while skill credentials remain static config-backed `env`/`apiKey` values.

## What This Changes

Request to pass a custom inbound HTTP header such as `custom-token` through the OpenAI-compatible completions flow so a Python skill can read it for per-request authentication.

## Best Possible Solution

Add an explicit, allowlisted per-request credential mapping if maintainers want this capability: selected inbound headers should become a narrowly scoped per-run context or per-skill environment value with redaction, audit behavior, sandbox rules, docs, and gateway/skill tests. If dynamic credentials are intentionally unsupported, document the static `skills.entries.<skill>.env` or `apiKey` path as the supported alternative.

## Reproduction Assessment

Yes for the missing capability: a source-level trace through `/v1/chat/completions` or `/v1/responses` shows custom request headers are never copied into `AgentCommandIngressOpts` or skill environment. I did not run a live gateway because this review was required to keep the checkout read-only.

## Solution Assessment

Unclear. Passing arbitrary headers directly is not the best maintainable solution; the safer path is a maintainer-designed allowlist/mapping contract, or explicit documentation that only static config-backed skill credentials are supported.

## Risks / Open Questions

- Forwarding arbitrary inbound headers into skill execution could leak gateway auth, cookies, proxy identity headers, or other caller secrets unless the feature is explicitly allowlisted, scoped, redacted, and audited.
- The report names `/v1/completions`, while current main documents and implements `/v1/chat/completions` and `/v1/responses`; maintainers should clarify whether legacy completions compatibility is part of the desired scope.
- Sandboxed skill behavior is unresolved: current docs say `skills.entries.*.env/apiKey` apply to host runs only, so dynamic per-run credentials would need explicit sandbox semantics.
