## Summary

Keep open. Current main still has no supported path to pass an arbitrary inbound OpenAI-compatible HTTP header into an agent command, run context, or Python skill environment; the supported request headers are fixed OpenClaw routing/control headers, and skill credentials are static config-backed env/apiKey values. The request is valid but security-sensitive, so it needs maintainer design before implementation.

## What This Changes

The issue requests a supported way to pass a custom inbound HTTP header through an OpenAI-compatible completion request and read it from a Python skill.

## Best Possible Solution

Add an explicit allowlisted per-request credential mapping only if maintainers want this capability: selected inbound headers should become a narrowly scoped per-run or per-skill value with redaction, audit behavior, sandbox rules, docs, and gateway/skill tests. If dynamic credentials are intentionally unsupported, document skills.entries.<skill>.env/apiKey as the supported static credential path.

## Reproduction Assessment

Yes for the missing capability: a source-level trace through /v1/chat/completions and /v1/responses shows arbitrary custom headers are not copied into AgentCommandIngressOpts, AgentRunContext, or skill env injection. I did not run a live gateway because this review was required to keep the checkout read-only.

## Solution Assessment

Unclear. Passing arbitrary headers directly into Python skill environment variables is not the safest maintainable fix; an allowlisted, redacted, per-request mapping contract or explicit documentation of the static config alternative would be safer.
