## Summary

Keep open. Current main still lacks a supported inbound HTTP header passthrough from the OpenAI-compatible Gateway request into agent runs or skill environments, and the related provider-header work is an adjacent outbound-provider configuration path rather than this caller-to-skill credential flow.

## What This Changes

Request support for passing a custom HTTP header from an OpenAI-compatible Gateway request into a Python skill for downstream authentication.

## Best Possible Solution

Define an explicit allowlisted per-request credential/header mapping contract into skill runtime, with redaction, scope, session lifetime, sandbox behavior, docs, and Gateway/skill tests; otherwise document static `skills.entries` env/apiKey as the supported credential path.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection gives a high-confidence verification path: current main only carries fixed routing/session/channel fields and static skill env config, with no arbitrary inbound header map passed to skills.

## Solution Assessment

Unclear pending maintainer design. Direct arbitrary header passthrough is not the best maintainable shape; an explicit allowlisted credential mapping or documented static skill credential path is safer.
