## Summary

Keep this issue open. Current main still has no supported path to carry an arbitrary inbound OpenAI-compatible HTTP header such as `custom-token` into the agent run or expose it to a Python skill; the implemented surface only preserves fixed OpenClaw routing/control headers and config-backed skill env/API-key injection. The requested behavior is also security-sensitive enough to need maintainer product/security design before implementation.

## What This Changes

The issue requests a way to pass a custom OpenAI-compatible HTTP header through the completions flow and read it from a Python skill.

## Best Possible Solution

Decide the product/security contract explicitly. If OpenClaw should support dynamic caller credentials for skills, add an allowlisted mapping from selected OpenAI-compatible inbound headers into per-run context or per-skill environment, with redaction, per-skill opt-in, sandbox semantics, docs, and gateway/skill tests. If unsupported, document the supported alternative of fixed credentials through `skills.entries.<skill>.env` or `skills.entries.<skill>.apiKey`.

## Risks / Open Questions

- Forwarding arbitrary inbound HTTP headers into skill execution could leak caller credentials, gateway bearer tokens, cookies, trusted-proxy identity headers, or other secrets without an explicit allowlist, redaction, audit, and per-skill/run scope model.
- The report names `/v1/completions`, while current main documents and implements `/v1/chat/completions` and `/v1/responses`; maintainers should clarify the intended compatibility endpoint before designing this feature.
- Current skill env injection applies to host runs only, and sandboxed skills do not inherit host `process.env` by default, so any dynamic header-to-skill design needs explicit sandbox behavior.
