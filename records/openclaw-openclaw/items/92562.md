## Label Justifications

- `P2`: This is a normal-priority session-state bug with silent behavior override loss after routine rollover, but the affected path and fix are narrow.
- `impact:session-state`: The issue is about per-session behavior state drifting from the user's persisted choice during session rollover.
- `impact:data-loss`: The rollover can silently drop user-set session override values such as `thinkingLevel` from the next persisted session row.

## Summary

Keep open: current main and v2026.6.6 still copy behavior overrides only when `resetTriggered` is true, so an implicit daily or idle rollover can drop `/think` and sibling session settings. The active fix path is the open, clean related PR at https://github.com/openclaw/openclaw/pull/92575.

## What This Changes

The issue reports that session-level behavior overrides such as `/think medium` survive explicit `/new` but revert to default after implicit daily or idle session rollover.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/92575 or an equivalent narrow patch that carries existing behavior overrides across implicit stale rollovers while keeping CLI/spawn lineage explicit-reset-only.

## Reproduction Assessment

Yes, source-level: seed a stale existing session with `thinkingLevel: "medium"`, send a normal non-reset message, and current main enters the new-session branch with `resetTriggered === false`, skipping the only behavior override copy path. I did not run a live wall-clock or channel repro because this review is read-only.

## Solution Assessment

Yes, the narrow maintainable fix is to copy existing behavior override fields for any rollover from an existing entry, just as current main already does for preserved model/auth selection. The related PR uses that shape and adds focused regression coverage.
