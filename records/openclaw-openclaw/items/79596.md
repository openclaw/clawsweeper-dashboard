## Summary

Keep open: current main still allows Gateway readiness to follow ACPX lazy backend facade registration, while the requested stricter “real ACPX is fully initialized or degraded” behavior conflicts with the documented lazy ACPX startup contract and needs maintainer judgment.

## What This Changes

Requests that Gateway readiness wait for the real ACPX runtime backend to finish registering, or report ACPX as degraded after a timeout, instead of treating the lazy ACPX facade as ready.

## Best Possible Solution

Decide whether ACPX lazy facade registration should count as Gateway-ready; if not, add an explicit ACPX readiness/degraded signal using the existing startup probe and timeout concepts, with docs and regression coverage.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection does show that current main can mark Gateway sidecars ready after the ACPX lazy facade registers, while the real ACPX runtime starts or probes later.

## Solution Assessment

Unclear. Waiting for the real ACPX runtime or reporting it as degraded could satisfy the request, but it changes the documented default that ACPX does not spawn or probe an ACP agent during Gateway startup unless explicitly opted in.
