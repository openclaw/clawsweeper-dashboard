## Summary

Keep open: current main shows a real timing split between gateway readiness and the full ACPX runtime service, but it also appears intentional because ACPX registers a lazy backend and documents startup probing as opt-in. This needs maintainer judgment on readiness semantics rather than conservative cleanup closure.

## What This Changes

The issue asks for gateway ready to wait until the acpx plugin has completed initialization, or to report it as degraded after a timeout.

## Best Possible Solution

Decide whether lazy ACPX registration should count as Gateway-ready; if not, make readiness wait for an ACPX health probe with timeout/degraded reporting and add regression plus cold-start validation.

## Reproduction Assessment

No high-confidence live reproduction path was established. Source inspection does show that `gateway ready` can follow lazy ACPX facade registration while the real ACPX service/probe may happen later, but the issue lacks logs and I did not run a cold Windows gateway.

## Solution Assessment

Unclear. Waiting on the real ACPX service or probe would match the requested behavior, but it conflicts with the documented default that ACPX does not spawn/probe during Gateway startup unless explicitly opted in.
