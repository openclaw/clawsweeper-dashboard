## Summary

Keep open: current main still has the ACPX startup probe opt-in, so the lazy ACPX facade can satisfy Gateway sidecar startup before the real runtime probe/health signal exists. An open closing PR, https://github.com/openclaw/openclaw/pull/79624, already targets this issue and should be reviewed or revised before this issue is closed.

## What This Changes

Request that Gateway readiness distinguish ACPX lazy facade registration from the real ACPX runtime being initialized and usable.

## Best Possible Solution

Review and either land https://github.com/openclaw/openclaw/pull/79624 or replace it with an equivalent maintainer-approved readiness fix that preserves a documented opt-out for lazy ACPX startup.

## Reproduction Assessment

Yes at source level, but not by a live current-main run in this review. With OPENCLAW_ACPX_RUNTIME_STARTUP_PROBE unset, current main registers a deferred ACPX backend and docs/tests still encode lazy startup, while the reporter added Windows logs showing the ready-to-backend gap.

## Solution Assessment

Likely yes, pending maintainer approval of the default change. The open PR is a narrow candidate for this issue's readiness semantics, while the broader sidecar parallelism work belongs in https://github.com/openclaw/openclaw/issues/79625.
