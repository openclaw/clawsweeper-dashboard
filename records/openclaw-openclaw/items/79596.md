## Summary

Keep open: current main still keeps ACPX startup probing opt-in and can report Gateway readiness after ACPX facade/backend registration rather than confirmed ACPX runtime readiness; the closed related PR was not merged and was itself reviewed as incomplete.

## What This Changes

The issue asks Gateway readiness and status surfaces to distinguish ACPX lazy facade/backend registration from a fully usable or degraded ACPX runtime.

## Best Possible Solution

Make the ACPX readiness contract explicit in the ACPX-owned startup path: either keep lazy registration but expose a clear not-ready/degraded state, or intentionally await a bounded ACPX probe before Gateway readiness and update docs/tests to match.

## Reproduction Assessment

Yes from source inspection and reporter logs, though not from a fresh live run here. Current main still defaults to lazy ACPX registration and `/readyz` does not distinguish facade/backend registration from confirmed ACPX runtime readiness.

## Solution Assessment

Unclear. A narrow code change is possible, but the previous default-flip PR was incomplete because the real probe stayed detached from startup, and maintainers still need to choose the intended lazy-vs-ready contract.
