## Summary

Keep open. Current main still lacks a configurable Brave Search baseUrl: the bundled Brave runtime builds both web and LLM-context requests from fixed Brave API constants, the plugin manifest/schema/docs expose only apiKey and mode, and the related implementation PRs found in the discussion/search are closed unmerged.

## What This Changes

The issue requests a Brave Search baseUrl configuration option so operators can route Brave API calls through a proxy while preserving the default Brave endpoint when omitted.

## Best Possible Solution

Land a provider-owned Brave plugin change that adds `plugins.entries.brave.config.webSearch.baseUrl`, normalizes and validates it, derives both `/res/v1/web/search` and `/res/v1/llm/context` from the resolved base while preserving the default Brave endpoint, includes the resolved base in cache identity, and updates manifest UI hints, schema/generated docs, Brave docs, focused tests, and changelog together.

## Reproduction Assessment

Yes. A high-confidence read-only reproduction is to inspect current main: `webSearch.baseUrl` is absent from the Brave manifest/config/docs, and both Brave request paths are constructed from hardcoded constants.

## Solution Assessment

Yes, the requested capability is a maintainable fix when implemented in the bundled Brave plugin surface. The safer shape is the current plugin-owned path, not reviving old core-only `tools.web.search.*` wiring except for deliberate compatibility if maintainers choose it.
