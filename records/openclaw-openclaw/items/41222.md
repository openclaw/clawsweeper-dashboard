## Summary

Keep this PR open. Current main has overlapping directive-tag indentation fixes, but it still has the media retained-body whitespace collapse this PR targets, and the PR patch itself has concrete multiline directive/media cleanup regressions that should be resolved before merge.

## What This Changes

The PR changes inline directive and media parsing so multiline reply text keeps indentation after reply/audio/media tag stripping, with focused parser and outbound JSON regression tests.

## Best Possible Solution

Refresh or selectively rework the PR against current main: keep the shipped code-block-aware directive behavior, preserve meaningful multiline indentation after valid `MEDIA:` and audio tag stripping, avoid adding indentation or blank paragraphs from removed directive syntax, and add focused helper plus outbound JSON regressions.

## Reproduction Assessment

Yes. A high-confidence helper-level reproduction is `splitMediaFromOutput("MEDIA:https://x.test/a.png\n  hello\n    world")` on current main, which follows the retained-body cleanup path that still collapses spaces and trims the result.

## Solution Assessment

No. The direction is right, but this exact patch is not the best merge candidate because it drops normalization in places where removed directive or media syntax can introduce visible indentation or blank paragraphs; a narrower current-main refresh is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Trim removed MEDIA prefix padding from captions:** `src/media/parse.ts:218`
  - body: When a line starts with spaces before `MEDIA:`, `pieces` still contains those prefix spaces after the token is removed. Returning `pieces.join("")` makes `  MEDIA:https://x.test/a.png caption` produce an artificially indented `  caption`, which can change Markdown rendering.
  - confidence: 0.9
- **[P2] Normalize tag-stripped multiline directive boundaries:** `src/utils/directive-tags.ts:36`
  - body: For multiline text, this cleanup only removes spaces before newlines. A directive on its own line or before a separator space can therefore leave `line1\n\nline2` or `\n world`, adding a paragraph break or indentation that came from the directive syntax rather than the message body.
  - confidence: 0.86
- **[P2] Collapse gaps after audio tag removal:** `src/media/parse.ts:233`
  - body: Changing audio cleanup to `audioTagResult.text.trim()` preserves the newline pair left when `[[audio_as_voice]]` is on a line by itself, so outbound text gains an extra blank paragraph after the directive is stripped.
  - confidence: 0.88
