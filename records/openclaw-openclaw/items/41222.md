Refresh or replace this branch with a current-main patch that preserves real body indentation after reply/audio/media tag stripping, keeps the shipped code-block-aware directive behavior, trims only whitespace introduced by removed syntax, and adds helper plus outbound JSON regressions for media-retained text and directive-only line stripping.

No. The direction is right, but this exact diff is not the best merge candidate because it removes normalization around stripped directive/media syntax too broadly and predates current main’s parser changes. A narrower current-main repair should preserve authored indentation without adding indentation or blank paragraphs from removed tags.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Trim media-token prefix padding from captions:** `src/media/parse.ts:218`
  - body: When a line starts with spaces before `MEDIA:`, `pieces` still contains those prefix spaces after the token is removed. Returning raw `pieces.join("")` makes `  MEDIA:https://x.test/a.png caption` produce `  caption`, adding indentation that came from directive syntax rather than body text.
  - confidence: 0.9
- **[P2] Normalize gaps after stripped multiline directives:** `src/utils/directive-tags.ts:36`
  - body: For multiline input, this cleanup only removes spaces before newlines. A directive on its own line, or a directive followed by a separator space, can leave `line1\n\nline2` or `\n world`, adding visible formatting that was introduced by the stripped tag.
  - confidence: 0.86
- **[P2] Collapse gaps after audio tag removal:** `src/media/parse.ts:233`
  - body: Changing audio cleanup to `audioTagResult.text.trim()` preserves the newline pair left when `[[audio_as_voice]]` is on a line by itself, so outbound text gains an extra blank paragraph after the directive is stripped.
  - confidence: 0.88
