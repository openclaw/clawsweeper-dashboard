## Summary

This PR should stay open, but the current head is not merge-ready: current main has already shipped the reply-directive indentation subset, while the media-extraction path still has a real indentation bug and this patch introduces narrow cleanup regressions around stripped media/audio/directive syntax.

## What This Changes

The PR changes inline directive and media parsing so multiline reply text keeps indentation after reply, audio, and media tag stripping, with parser and outbound regression tests.

## Best Possible Solution

Refresh or replace the branch with a current-main patch that preserves authored body indentation across reply/audio/media extraction while trimming only whitespace introduced by removed syntax, backed by parser and outbound JSON regressions.

## Reproduction Assessment

Yes. Source-level reproduction is clear: retained text after splitMediaFromOutput("MEDIA:https://x.test/a.png\n  hello\n    world") reaches cleanLineText and final whitespace folding on current main, which collapses indentation after media extraction; this read-only review did not execute tests.

## Solution Assessment

No. The direction is right, but this exact PR head is not the best fix because it removes cleanup around stripped directive/media syntax too broadly and leaves artificial spaces or blank lines in several tag-stripping cases.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Trim media-token prefix padding from captions:** `src/media/parse.ts:218`
  - body: When a line starts with spaces before `MEDIA:`, `pieces` still contains those prefix spaces after the token is removed. Returning raw `pieces.join("")` makes `  MEDIA:https://x.test/a.png caption` produce `  caption`, adding indentation that came from directive syntax rather than body text.
  - confidence: 0.9
- **[P2] Normalize gaps after stripped multiline directives:** `src/utils/directive-tags.ts:36`
  - body: For multiline input, this cleanup only removes spaces before newlines. A directive on its own line, or a directive followed by a separator space, can leave an extra blank line or leading space that was introduced by the stripped tag.
  - confidence: 0.86
- **[P2] Collapse gaps after audio tag removal:** `src/media/parse.ts:233`
  - body: Changing audio cleanup to `audioTagResult.text.trim()` preserves the newline pair left when `[[audio_as_voice]]` is on a line by itself, so outbound text gains an extra blank paragraph after the directive is stripped.
  - confidence: 0.88
