## Summary

Keep this PR open: the underlying outbound indentation bug is real and only partly addressed on current main, but this branch still has parser regressions and lacks inspectable real behavior proof before merge.

## What This Changes

The PR changes inline directive and media parsing plus focused tests so outbound reply text keeps multiline indentation after reply/audio/media tag stripping.

## Best Possible Solution

Refresh or replace the branch with a current-main patch that preserves authored body indentation across reply/audio/media extraction while trimming only whitespace introduced by stripped directive syntax.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: on current main, retained text after splitMediaFromOutput("MEDIA:https://x.test/a.png\n  hello\n    world") reaches cleanLineText and the final whitespace-folding chain. I did not execute tests because this was a read-only review.

## Solution Assessment

No. The direction is right, but the current PR head removes cleanup too broadly and leaves artificial caption indentation plus blank-line/leading-space artifacts after directive and audio tag stripping.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Trim media-token prefix padding from captions:** `src/media/parse.ts:221`
  - body: When a line starts with spaces before MEDIA:, returning raw pieces.join("") preserves prefix spaces that belonged to the removed directive. For input like "  MEDIA:https://x.test/a.png caption", the remaining text becomes artificially indented and can change Markdown rendering.
  - confidence: 0.9
- **[P2] Normalize gaps after stripped multiline directives:** `src/utils/directive-tags.ts:36`
  - body: For multiline input, this cleanup only removes spaces before newlines. A directive on its own line or followed by a separator space can leave an extra blank line or leading space introduced by the stripped tag.
  - confidence: 0.86
- **[P2] Collapse gaps after audio tag removal:** `src/media/parse.ts:233`
  - body: Changing audio cleanup to audioTagResult.text.trim() preserves the newline pair left when [[audio_as_voice]] is on a line by itself, so outbound text gains an extra blank paragraph after the directive is stripped.
  - confidence: 0.88
