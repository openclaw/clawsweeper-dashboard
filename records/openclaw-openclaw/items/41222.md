## Summary

Keep open. Current main and https://github.com/openclaw/openclaw/pull/55960 cover the reply-directive indentation subset, but current media extraction still folds retained indentation, and this PR has concrete parser cleanup regressions plus no inspectable real behavior proof.

## What This Changes

The PR changes inline directive and media parsing, with parser and outbound tests, so multiline outbound reply text keeps indentation after reply/audio/media tag stripping.

## Best Possible Solution

Refresh or replace the branch with a current-main patch that preserves authored body indentation across reply/audio/media extraction while trimming only whitespace introduced by stripped directive syntax, coordinated with https://github.com/openclaw/openclaw/pull/69312.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: on current main, retained text after splitMediaFromOutput("MEDIA:https://x.test/a.png\n  hello\n    world") reaches cleanLineText and final whitespace folding. I did not execute tests because this was a read-only review.

## Solution Assessment

No. The direction is right, but the current PR head removes cleanup too broadly and leaves artificial caption indentation plus blank-line/leading-space artifacts after directive and audio tag stripping.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Trim media-token prefix padding from captions:** `src/media/parse.ts:218`
  - body: When a line starts with spaces before `MEDIA:`, `pieces` still contains those prefix spaces after the token is removed. Returning raw `pieces.join("")` makes `  MEDIA:https://x.test/a.png caption` produce `  caption`, adding indentation from directive syntax rather than body text.
  - confidence: 0.9
- **[P2] Normalize gaps after stripped multiline directives:** `src/utils/directive-tags.ts:36`
  - body: For multiline input, this cleanup only removes spaces before newlines. A directive on its own line, or a directive followed by a separator space, can leave an extra blank line or leading space that was introduced by the stripped tag.
  - confidence: 0.86
- **[P2] Collapse gaps after audio tag removal:** `src/media/parse.ts:233`
  - body: Changing audio cleanup to `audioTagResult.text.trim()` preserves the newline pair left when `[[audio_as_voice]]` is on a line by itself, so outbound text gains an extra blank paragraph after the directive is stripped.
  - confidence: 0.88
