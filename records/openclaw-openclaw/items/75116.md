## Summary

Keep open. Current main still lacks the requested per-server `injectCallerContext` MCP config contract and injection behavior, and the issue is paired with open closing PR #75118, which should be reviewed or closed before the issue is resolved. The proposed behavior also crosses a caller/session identity disclosure boundary, so maintainer security/privacy review is part of the remaining decision.

## What This Changes

The issue requests a per-server `mcp.servers.<name>.injectCallerContext` option that forwards OpenClaw caller identity headers to opted-in remote bundle MCP servers without affecting existing servers.

## Best Possible Solution

Review PR #75118 for the trust boundary and implementation details, then land a shared bundle-MCP/config-layer change if accepted: remote-only, default-off, non-overwriting caller-header merge, stripped downstream control flag, placeholder env population when any merged server opts in, and matching schema/help/docs/tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure a remote `mcp.servers.<name>.url` entry with `injectCallerContext: true` and inspect the generated bundle MCP config; current main has no typed/schema support, no injection branch in the bundle merge path, and loopback-only placeholder env population.

## Solution Assessment

Yes, pending maintainer approval of the trust boundary. A per-server default-off flag is the narrow maintainable direction; a global flag would create broader accidental identity leakage risk, and PR #75118 is the active implementation candidate.
