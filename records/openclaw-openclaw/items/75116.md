## Summary

Keep open: current main still lacks the requested per-server caller-context opt-in, and https://github.com/openclaw/openclaw/pull/75118 is an open closing PR for the same work. Closing this issue before that PR is reviewed, fixed, and merged would drop the canonical tracking thread for a security-sensitive config feature.

## What This Changes

Add an opt-in `mcp.servers.<name>.injectCallerContext` flag that forwards OpenClaw caller headers to trusted remote bundle MCP servers without changing existing defaults.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/75118, or an equivalent narrow implementation with schema/types, URL-only default-off injection, non-overwriting header merge, flag stripping, tests, docs/changelog, and trusted-endpoint wording.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a feature request, but the absence is source-verifiable on current main: the typed config, schema, bundle merge path, and CLI env preparation do not implement the requested opt-in behavior.

## Solution Assessment

Is this the best way to solve the issue? Yes as a direction, pending maintainer/security approval. A per-server, default-off, URL-only opt-in is the narrow maintainable shape, but the linked PR still needs review fixes and real behavior proof before merge.
