## Summary

Keep open: current main still lacks the requested per-server caller-context injection contract, and #75118 is an open, unmerged closing PR for the same work. The issue should stay open until that PR, or an equivalent maintainer-approved security-conscious implementation, lands or is closed.

## What This Changes

Add an opt-in per-server `mcp.servers.<name>.injectCallerContext` flag for remote bundle MCP servers so OpenClaw can merge caller/session headers without overwriting user headers and without emitting the control flag downstream.

## Best Possible Solution

Review #75118 for the trust boundary and implementation details, then land that PR or an equivalent narrow change if accepted: typed config/schema/docs, default-off remote-only injection, user-header precedence, downstream control-flag stripping, placeholder env population for opted-in merged layers, and focused tests.

## Reproduction Assessment

Yes. A high-confidence static reproduction is to configure a remote `mcp.servers.<name>.url` with `injectCallerContext: true` and inspect current main's generated bundle MCP path; there is no typed/schema contract or injection branch, and `OPENCLAW_MCP_*` env values are only populated for the loopback runtime.

## Solution Assessment

Yes, pending maintainer security approval. The proposed per-server default-off flag is the narrow maintainable shape; a global flag or implicit forwarding would create broader accidental caller-identity disclosure risk.

## Security Review

Status: needs_attention

Summary: The requested feature is security-sensitive because it would forward caller/session identity headers to operator-selected remote MCP endpoints.

Concerns:

- **[medium] Review remote caller-context disclosure:**
