## Summary

Keep open. Current main still lacks the requested explicit `injectCallerContext` contract and caller-header injection behavior, and the issue is paired with open, unmerged closing PR #75118, so the terminal outcome should wait for that PR or an equivalent maintainer-approved change.

## What This Changes

The issue requests a per-server `mcp.servers.<name>.injectCallerContext` option that lets trusted remote bundle MCP servers receive OpenClaw caller/session headers without changing existing configs or overwriting user headers.

## Best Possible Solution

Review #75118 for the trust boundary and implementation details, then land that PR or an equivalent narrow change if accepted: typed schema/config docs, default-off remote-only header injection, no overwrite of configured headers, downstream flag stripping, placeholder env availability when any merged layer opts in, and focused bundle MCP tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure a remote `mcp.servers.<name>.url` entry with `injectCallerContext: true` and inspect the generated bundle MCP config; current main has no injection branch and does not define the requested typed config field.

## Solution Assessment

Yes, pending maintainer approval of the security boundary. A per-server opt-in flag is the narrow maintainable shape; a global flag would increase accidental caller-identity disclosure risk, and #75118 is already the active implementation candidate.
