## Summary

Keep open: current main still has the false empty-reply logging path for deliberate silent completions, so this PR is not obsolete. The diff looks narrow and has no blocking review findings, but it likely needs maintainer review or a rebase against the newer runner metadata before merge.

## What This Changes

The PR adds a `silentCompletion` marker for exact `NO_REPLY` embedded runs, uses it to suppress empty-result logging in local delivery and gateway CLI paths, and adds focused regression tests.

## Best Possible Solution

Land a rebased fix that preserves a typed silent-completion signal for deliberate exact `NO_REPLY` completions, gates both local delivery and gateway CLI empty-result logging on that signal, and keeps focused regression tests for the runner helper plus both output surfaces.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: an embedded result whose exact `NO_REPLY` assistant text is filtered to an empty payload list will still reach the local or gateway empty-result logging branches on current main.

## Solution Assessment

Mostly yes. The proposed direction targets the right boundary, but the current-main version should use one shared helper or canonical metadata shape and consciously decide whether the linked `ANNOUNCE_SKIP` case is in scope.
