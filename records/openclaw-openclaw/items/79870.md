## Summary

Keep open. Current main still has a source-reproducible stale phase attribution: the channel manager measures the long-lived Slack monitor as `channels.slack.start-account`, so later liveness warnings can keep pointing at Slack startup after readiness even though that does not prove Slack is the actual event-loop blocker.

## What This Changes

The issue asks maintainers to investigate Slack Socket Mode reconnects and event-loop stalls reported under a long-running `channels.slack.start-account` diagnostic phase after gateway readiness.

## Best Possible Solution

Narrow diagnostic/startup phase measurement to the actual channel startup handoff or initial socket start, then use the corrected liveness output to profile any remaining event-loop blocker.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Partially: source inspection gives high confidence for the stale `phase=channels.slack.start-account` attribution, but the exact 15-44 second event-loop stalls and reconnects were not live-reproduced on current main.

## Solution Assessment

Is this the best way to solve the issue? The report's Slack-specific conclusion is not yet proven; the narrow maintainable fix is to stop treating the long-lived Slack monitor as an active startup phase, then profile the real blocker if disconnects remain.
