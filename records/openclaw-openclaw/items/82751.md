## Summary

Current main already fixes the reported Telegram isolated-ingress `Bot not initialized` loop by initializing the grammy bot before any spooled update is dispatched; the fix is merged on main but not yet in the latest shipped release.

## What This Changes

The issue reports that isolated Telegram polling can repeatedly fail inbound spooled updates after a transient startup `getMe` probe failure leaves the bot without `botInfo`.

## Best Possible Solution

Keep the merged init-before-drain fix on main and ship it in the next release; any remaining desire for louder startup-probe diagnostics should be handled as a separate logging enhancement.

## Reproduction Assessment

No current-main failure path remains: source inspection shows isolated ingress now initializes the bot before spooled `handleUpdate`, and the regression test asserts that order. The original v2026.5.12 failure mode was credible, but it is already fixed on main.

## Solution Assessment

Yes. Initializing the bot at the start of the isolated ingress cycle is the narrowest maintainable fix because it matches grammy's `handleUpdate` contract and reuses the existing recoverable setup retry path.
