## Summary

The issue should stay open: current main still constructs the Telegram bot without `botInfo` when the startup probe fails, and the isolated polling path dispatches spooled updates through `handleUpdate()` without an explicit `init()`, matching the reported permanent inbound failure mode.

## What This Changes

The issue asks OpenClaw to recover from transient Telegram startup `getMe` probe failures when isolated polling ingress is enabled, either by initializing the bot before isolated dispatch or by making the probe failure visible.

## Best Possible Solution

Patch the Telegram plugin so isolated polling ingress guarantees a grammy bot is initialized before handling spooled updates, and add focused regression coverage for a thrown startup probe followed by isolated inbound dispatch.

## Reproduction Assessment

Yes from source inspection: the current Telegram startup path can return a bot without `botInfo` after a thrown probe, and the isolated ingress path can then call `handleUpdate()` without `bot.init()`. I did not execute a live network-failure reproduction in this read-only review.

## Solution Assessment

Yes, initializing before isolated dispatch is the narrowest maintainable fix; surfacing thrown probe failures at default warning level is a useful companion diagnostic but should not be the only fix.
