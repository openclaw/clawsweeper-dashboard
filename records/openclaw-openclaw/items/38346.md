## Summary

Keep open. Current main still cannot express the requested Telegram split: allowlist policy rejects non-allowlisted group senders before passive ingest or hooks can observe them, while open policy admits those senders into ordinary group handling.

## What This Changes

Request a Telegram group mode that passively observes all group messages for sender identification while only allowlisted users can trigger bot responses.

## Best Possible Solution

Define an explicit opt-in Telegram passive-observation contract, preferably metadata-minimized and documented, while keeping reply authorization fail-closed and separate from groupPolicy "open".

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives a high-confidence current-main behavior trace: non-allowlisted Telegram group messages are rejected before passive ingest, while open group policy admits those senders for normal handling.

## Solution Assessment

No. The proposed groupPolicy "open" plus groupAllowFrom shape conflates passive visibility with trigger authorization; a separate opt-in passive-observation contract would be narrower and safer.
