## Summary

Keep open. Current main still cannot express the requested Telegram split: allowlist policy rejects non-allowlisted group senders before passive ingest or hooks can observe them, while open policy admits those senders into normal group handling. This is a valid feature request that needs privacy, retention, trigger-authorization, and config-shape decisions before implementation.

## What This Changes

The issue requests a Telegram mode that observes all group messages and records sender IDs while only allowlisted senders can trigger bot responses.

## Best Possible Solution

Define an explicit opt-in Telegram passive-observation contract, preferably metadata-minimized and documented, while keeping reply authorization fail-closed and separate from groupPolicy "open".

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives a high-confidence current-main behavior trace: allowlist-denied Telegram group messages return before passive ingest or processing, while groupPolicy "open" admits those senders into normal handling.

## Solution Assessment

No. The proposed groupPolicy "open" plus groupAllowFrom shape conflates passive visibility with trigger authorization; a separate opt-in passive-observation contract would be narrower and safer.
