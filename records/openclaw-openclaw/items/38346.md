## Summary

Keep open. Current `main` still cannot express the requested Telegram split: `groupPolicy: "open"` admits non-allowlisted senders for ordinary group turns, while `groupPolicy: "allowlist"` drops non-allowlisted senders before passive history or `message:received` ingest can record them.

## What This Changes

The issue requests a Telegram mode where all group messages can be observed for user identification while only allowlisted users can trigger bot replies.

## Best Possible Solution

Define an explicit opt-in Telegram passive-observation or raw-inbound observation contract that can record minimal sender metadata while preserving fail-closed trigger authorization and documenting privacy-mode prerequisites.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives a high-confidence current-main behavior trace: allowlist-rejected Telegram group senders are dropped before passive ingest, while `groupPolicy: "open"` admits ordinary non-allowlisted group turns.

## Solution Assessment

No. The proposed `groupPolicy: "open"` config is not the best solution because current contracts define it as open sender admission; the maintainable path is a separate opt-in passive observation mode or pre-filter hook with fail-closed reply authorization.
