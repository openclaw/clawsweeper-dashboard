## Summary

Keep open: current main and v2026.5.12 still reject non-allowlisted Telegram group senders before passive ingest or room-event handling, and the requested observe-all/respond-few mode needs product and security decisions rather than cleanup closure.

## What This Changes

The issue requests a Telegram group mode that passively observes all messages for future identification/reference while only allowlisted users can trigger bot responses.

## Best Possible Solution

Define an explicit opt-in Telegram passive-observation contract with payload minimization, retention rules, diagnostics, and trigger authorization kept separate from group admission.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection gives a high-confidence current-main and v2026.5.12 trace: allowlist-denied Telegram group messages return before passive ingest and normal inbound processing.

## Solution Assessment

No. The proposed groupPolicy open plus groupAllowFrom shape conflates group admission with trigger authorization; a separate opt-in passive-observation or audit hook is the narrower maintainable direction.
