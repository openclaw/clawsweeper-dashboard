## Summary

Keep open. Current main still lacks a Telegram mode that observes or ingests all visible group messages while separately allowing only selected senders to trigger agent turns, commands, tools, or visible replies; the closest existing ingest hook only runs after group authorization and mention-gating. The related open report about unauthorized auto-replies is adjacent ID-discovery work, not a duplicate of this passive observation request.

## What This Changes

The issue asks for Telegram group messages to be observable for user identification while only allowlisted users can trigger bot responses.

## Best Possible Solution

If maintainers approve the product/security tradeoff, implement an explicit opt-in Telegram group/topic observe-before-trigger mode in the bundled Telegram plugin, keep trigger authorization separate for agent turns, text/native commands, tool execution, and visible replies, and update config schema, docs, security-audit guidance, and regression tests around unauthorized sender handling.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: with Telegram groupPolicy "allowlist", a non-allowlisted sender is returned from shouldSkipGroupMessage before processInboundMessage; with groupPolicy "open", the current policy evaluator treats the sender as allowed rather than applying groupAllowFrom as a trigger-only list.

## Solution Assessment

No for the issue's proposed config as written: groupPolicy "open" currently means open triggering for normal messages, not observe-only admission with allowlisted triggers. The maintainable direction is a new explicit opt-in observe/ingest mode with separate authorization gates, after product/security review.
