## Summary

Keep open. Current main still cannot express the requested Telegram split: `groupPolicy: "open"` admits ordinary non-allowlisted group senders, while `groupPolicy: "allowlist"` rejects non-allowlisted senders before passive history or message hooks can observe them.

## What This Changes

Request Telegram group behavior that passively observes all group messages for sender identification while only allowlisted users can trigger bot replies.

## Best Possible Solution

Define an explicit opt-in Telegram passive-observation or generic raw-inbound contract that records minimal sender metadata under documented privacy controls while keeping reply authorization fail-closed.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives a high-confidence current-main behavior trace: allowlist-rejected Telegram group messages return before passive ingest, while `groupPolicy: "open"` admits ordinary non-allowlisted group turns.

## Solution Assessment

No. Reusing `groupPolicy: "open"` is not the best solution because that mode intentionally admits all group senders; passive observation needs a separate opt-in contract from trigger authorization.
