## Summary

Keep open. Current main has opt-in Telegram/group quiet room events and mention-skip ingest, but it still cannot observe allowlist-denied group senders while limiting response triggers to allowlisted users; that split needs explicit Telegram product, privacy, retention, and config decisions.

## What This Changes

The issue requests a Telegram group mode where all messages can be observed for future sender identification while only allowlisted users can trigger bot responses.

## Best Possible Solution

Define a Telegram-owned passive-observation contract that is opt-in, metadata-minimized, documented, and separate from reply/trigger authorization, while preserving current fail-closed allowlist behavior by default.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives a high-confidence current-main behavior trace: allowlist-denied group messages return before passive ingest or processing, while open policy admits all senders rather than enforcing a trigger-only allowlist.

## Solution Assessment

No. The proposed groupPolicy open plus groupAllowFrom shape conflates admission with trigger authorization; a separate opt-in passive-observation setting would be narrower and safer.

## Risks / Open Questions

- Telegram Bot API privacy mode may prevent full group visibility unless operators disable it through BotFather, so docs and diagnostics need to be part of any implementation.
