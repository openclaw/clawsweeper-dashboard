## Summary

Keep open. Current main has ambient room-event and passive-hook paths for allowed Telegram group messages, but it still does not support observing allowlist-denied group senders while limiting response triggers to allowlisted users. The requested split needs explicit Telegram product/security decisions because it expands collection of non-allowlisted group metadata/content and cannot safely be modeled as `groupPolicy: "open"` plus `groupAllowFrom`.

## What This Changes

Request a Telegram mode that observes all group messages/user IDs but only lets allowlisted senders trigger bot responses.

## Best Possible Solution

Define a Telegram-owned passive-observation contract that is opt-in, metadata-minimized, documented, and separate from reply/trigger authorization while preserving fail-closed allowlist behavior by default.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives a high-confidence current-main behavior trace: allowlist-denied group messages return at `shouldSkipGroupMessage` before passive ingest or room-event handling, while `groupPolicy: "open"` admits all senders.

## Solution Assessment

No. The proposed `groupPolicy: "open"` plus `groupAllowFrom` shape conflates group admission with trigger authorization; a separate opt-in passive-observation setting would be narrower and safer.
