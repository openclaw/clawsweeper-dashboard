## Summary

Keep open. The PR addresses a real ACPX/Codex compatibility gap, but the proposed guard still treats explicit old partial or range-style Codex ACP adapter specs as supported, so it does not fully prevent the opaque `session/new` failure it is meant to replace.

## What This Changes

The PR adds an ACPX runtime guard, focused tests, docs, and a changelog entry for `openai-codex/gpt-5.5` startup when explicit `@zed-industries/codex-acp` adapter versions are older than `0.12.0`.

## Best Possible Solution

Keep the guard at the ACPX wrapper boundary, but make explicit npm version/range specs conservative: any explicit `@zed-industries/codex-acp` spec that can resolve below `0.12.0` should fail early for `gpt-5.5`, while unpinned commands, bundled wrapper/local binary paths, and `0.12.0+` specs continue to pass.

## Reproduction Assessment

Yes. An explicit command like `npx @zed-industries/codex-acp@0.11` has no patch segment, so the PR's `parseSemver` returns `undefined`, `supportsCodexAcpGpt55` returns true, and the stale adapter can still receive the `gpt-5.5` startup override.

## Solution Assessment

No as written. The wrapper-boundary guard and error shape are the right direction, but the version check needs npm-range-aware or conservatively rejecting behavior for explicit specs that can resolve below `0.12.0`.
