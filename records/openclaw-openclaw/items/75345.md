## Summary

Keep open. The PR is pointed at the right ACPX boundary, but its version check still treats explicit npm partial/range specs that can resolve below Codex ACP 0.12.0 as supported, preserving the stale-adapter failure it is trying to prevent.

## What This Changes

The PR adds an ACPX runtime guard, focused tests, docs, and a changelog entry for `openai-codex/gpt-5.5` when explicit Codex ACP adapter versions are older than `0.12.0`.

## Best Possible Solution

Keep the guard at the ACPX wrapper boundary, but make explicit npm version/range specs conservative: any explicit `@zed-industries/codex-acp` spec that can resolve below `0.12.0` should fail early for `gpt-5.5`, while unpinned commands, bundled wrapper/local binary paths, and `0.12.0+` specs continue to pass.

## Reproduction Assessment

Yes. A configured command like `npx @zed-industries/codex-acp@0.11` is a valid npm partial range below `0.12.0`, but the PR's parser cannot extract a full patch version and therefore lets the stale adapter receive `gpt-5.5`.

## Solution Assessment

No as written. The wrapper-boundary guard and error shape are the right direction, but the version check needs npm-range-aware parsing or conservative rejection for explicit specs that may resolve below `0.12.0`.
