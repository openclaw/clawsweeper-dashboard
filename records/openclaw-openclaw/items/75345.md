## Summary

Keep open. Current main still lacks the requested stale-adapter guard, and the PR is directionally correct but not mergeable as-is because explicit old partial/range specs can still bypass the new check and reach Codex ACP startup with `gpt-5.5`.

## What This Changes

The PR adds ACPX runtime validation, focused tests, ACP docs, and a changelog entry for rejecting `openai-codex/gpt-5.5` when an explicit `@zed-industries/codex-acp` adapter command is pinned below `0.12.0`.

## Best Possible Solution

Keep the guard in the ACPX runtime wrapper, but make explicit `@zed-industries/codex-acp` specs npm-range-aware or conservative: any explicit spec that can resolve below `0.12.0` should fail early for `gpt-5.5`, while unpinned commands, bundled wrapper/local binary paths, and specs provably at `0.12.0+` continue to work.

## Reproduction Assessment

Yes. Applying the PR logic to `npx @zed-industries/codex-acp@0.11` or `^0.11` with `openai-codex/gpt-5.5` leaves `parseSemver` undefined, so `supportsCodexAcpGpt55` returns true and the stale adapter still reaches startup.

## Solution Assessment

No as written. The wrapper-boundary guard and user-facing error are the right shape, but the version check must parse npm ranges or reject explicit specs that cannot prove compatibility.
