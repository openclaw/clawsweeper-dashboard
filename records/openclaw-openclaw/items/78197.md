## Summary

Keep open: current main still has the relayable exec-completion special case that can revive a stripped `HEARTBEAT_OK` and deliver it outward, while the docs say pure heartbeat ack replies should be dropped.

## What This Changes

The issue asks OpenClaw to suppress pure `HEARTBEAT_OK` replies from relayable exec-completion heartbeat runs instead of delivering them as chat alerts.

## Best Possible Solution

Suppress exact no-op/ack replies before exec-completion delivery while preserving non-empty exec summaries and structured `heartbeat_respond` notifications, with focused regression coverage and a changelog entry.

## Reproduction Assessment

Yes, source-level. Current main strips pure `HEARTBEAT_OK` to `shouldSkip`, then the relayable exec-completion branch restores the original reply text and sends it as an outbound payload.

## Solution Assessment

Yes, the narrow maintainable fix is to stop overriding a normalized skip for pure no-op tokens while keeping meaningful exec-completion text deliverable. PR #73785 follows that direction but still needs current-main integration review.
