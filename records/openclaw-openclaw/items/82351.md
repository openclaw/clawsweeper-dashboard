## Summary

Keep open. This is a protected `maintainer`-labeled feature PR and the patch still needs human review: I found a blocking retry-path bug for context engines that rotate transcripts, plus a security-sensitive raw tool payload preservation change.

## What This Changes

The PR adds a context-engine projection SDK contract plus Codex app-server docs, tests, and lifecycle logic to bootstrap context-engine context once per epoch and rotate Codex threads when projection metadata changes.

## Best Possible Solution

Land a refined version of the SDK seam that adopts rotated compaction transcript handles during Codex retry paths and either redacts/gates bootstrap tool payload preservation or documents an approved trust boundary.

## Reproduction Assessment

Yes for the review findings by source inspection: the forced-compaction retry path ignores `compactResult.result.sessionFile`, and the new preserve mode serializes raw tool payloads. I did not run tests because this was a read-only review.

## Solution Assessment

No. The thread-bootstrap direction is plausible, but the patch should first adopt rotated transcript handles and resolve the raw tool-payload trust boundary before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Adopt rotated compaction transcripts before retrying:** `extensions/codex/src/app-server/run-attempt.ts:1622-1623`
  - body: When forced context-engine compaction succeeds, this retry path rebuilds the prompt from `params.sessionFile` and later finalizes the turn against that same original file. Context-engine compaction can return a successor `result.sessionFile`, and the main embedded runner already adopts it before continuing; without that, a rotating compaction engine will restart Codex from stale pre-compaction history and write after-turn state to the old transcript.
  - confidence: 0.88
- **[P1] Sanitize preserved bootstrap tool payloads:** `extensions/codex/src/app-server/context-engine-projection.ts:199-211`
  - body: `toolPayloadMode: "preserve"` now serializes full tool call and tool result objects into the Codex prompt, and the added test asserts secret-shaped values like `sk-secret` and `API_KEY=...` survive. That weakens the existing elision guard for any context engine that requests thread-bootstrap projection, so this should preserve only redacted/safe payloads or be gated by an explicit approved policy.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The diff introduces a concrete secret-exposure risk by preserving raw historical tool call/result payloads in Codex bootstrap prompts.

Concerns:

- **[medium] Raw tool payloads can expose secrets:** `extensions/codex/src/app-server/context-engine-projection.ts:199`

That can send private command arguments or tool output to the Codex app-server and prompt/trajectory surfaces unless the payloads are redacted or explicitly policy-gated.
  - confidence: 0.84
