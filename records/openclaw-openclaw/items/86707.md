## Summary

Close as implemented: current main and the latest release already register the macOS Canvas command family through the enabled-by-default Canvas plugin, and the gateway allowlist consumes plugin node-invoke policies even though `PLATFORM_DEFAULTS.macos` itself stays Canvas-free.

## What This Changes

The issue asks for macOS `canvas.*` node commands declared by the macOS app to pass gateway command policy, or for clearer guidance when they are intentionally opt-in.

## Best Possible Solution

Keep Canvas command policy plugin-owned; if a live install still rejects these commands, investigate why the bundled Canvas plugin is not active rather than adding hardcoded core defaults.

## Reproduction Assessment

No high-confidence current-main reproduction is established. Source and release inspection show the default Canvas plugin already contributes these commands to the gateway allowlist; I did not run a live macOS node.

## Solution Assessment

Yes. The best fix shape is the current plugin-owned policy, which matches the Canvas refactor direction and avoids reintroducing Canvas-specific command defaults into core gateway lists.
