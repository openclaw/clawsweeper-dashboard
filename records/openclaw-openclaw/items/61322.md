## Summary

Keep this PR open. Current main still preserves same-agent stale absolute sessionFile paths, so the bug is real and the PR direction is useful, but the proposed patch is not merge-ready because its remap bypasses the active sessionsDir used for custom session.store paths.

## What This Changes

The PR adds a same-agent stale absolute sessionFile remap in session path resolution plus a unit regression test for remapping into the current state directory.

## Best Possible Solution

Revise the PR so same-agent stale-root remaps validate the embedded agent and single transcript filename, then construct the destination under the already-resolved active sessionsDir/realBase while preserving existing cross-agent structural fallback behavior. Add regression coverage for both the default state-root case and a custom session.store path.

## Reproduction Assessment

Yes. On current main, a same-agent old-root absolute sessionFile reaches the structural fallback and remains pointed at the stale path. The PR-specific blocker is reproducible by passing opts.sessionsDir from a custom store and observing that the new helper returns resolveAgentSessionsDir(...) instead of the active sessionsDir.

## Solution Assessment

No. The fix direction is right, but this implementation is not the best way to solve it because it bypasses the active store boundary. The narrow maintainable fix is to remap only the validated filename under the resolved sessionsDir/realBase already being used by resolvePathWithinSessionsDir().

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Honor the active sessionsDir when remapping stale paths:** `src/config/sessions/paths.ts:188`
  - body: The new remap returns `resolveAgentSessionsDir(...)`, which ignores `opts.sessionsDir` when callers resolved a custom `session.store` or store path. In that setup a stale same-agent `sessionFile` is redirected into the default state tree instead of the active store directory, splitting transcript writes away from the configured store. Build the remap under the resolved `sessionsDir`/`realBase` instead.
  - confidence: 0.91
