## Summary

Keep open: current main still has the reported session path bug, and this PR is a plausible fix path, but the patch can redirect custom `session.store` users back into the default state tree.

## What This Changes

The PR adds same-agent old-root `sessionFile` remapping in session path resolution plus a focused regression test.

## Best Possible Solution

Retarget the same-agent remap to the active resolved sessions directory, preserve existing cross-agent fallback behavior, and cover both default and custom-store cases.

## Reproduction Assessment

Yes. The paired issue gives concrete stale metadata and CLI steps, and current-main source inspection shows the resolver can still return the old absolute path before the write lock creates that path's directory.

## Solution Assessment

No as written. Session path resolution is the right boundary, but the remap must rebuild under the active `sessionsDir` or `realBase`, not the default state-derived agent sessions directory.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Remap stale paths under the active sessionsDir:** `src/config/sessions/paths.ts:188`
  - body: When `resolveSessionFilePath` is called with a custom `sessionsDir` from `session.store`, this helper rebuilds the same-agent remap via `resolveAgentSessionsDir(...)`, ignoring the active store. That can split transcript writes outside the configured store; pass the active resolved sessions directory into the remap and add custom-store coverage.
  - confidence: 0.92
