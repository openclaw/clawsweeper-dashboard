## Summary

Keep open: the bug is real and the PR is useful, but its current remap target can bypass custom session-store roots, so it should not merge as-is.

## What This Changes

The PR adds same-agent stale absolute sessionFile remapping in the session path resolver and a focused regression test.

## Best Possible Solution

Retarget the same-agent stale-root remap to the active resolved sessions directory, preserve cross-agent fallback behavior, and cover both default-store and custom-store cases.

## Reproduction Assessment

Yes. The paired issue gives concrete stale metadata and CLI steps, and current-main source shows the resolver can still return the old absolute structural path before the write lock creates that path’s directory.

## Solution Assessment

No as written. Session path resolution is the right boundary, but the helper must rebuild under the active sessionsDir/realBase rather than the default state-derived sessions directory.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Remap stale paths under the active sessionsDir:** `src/config/sessions/paths.ts:188`
  - body: `resolveSessionFilePathOptions` can pass a custom `sessionsDir` from `session.store`, but this helper rebuilds same-agent stale paths with `resolveAgentSessionsDir(...)`. That sends resumed transcript writes to the default state tree instead of the active store; pass the active base or `sessionsDir` into the remap and cover the custom-store case.
  - confidence: 0.93
