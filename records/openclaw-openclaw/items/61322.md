## Label Justifications

- `P1`: The underlying bug and the introduced PR risk both affect resumed or targeted agent session writes for real migrated/custom-store setups.
- `merge-risk: 🚨 compatibility`: The patch changes how persisted absolute `sessionFile` metadata is interpreted during upgrade and can break existing custom `session.store` layouts.
- `merge-risk: 🚨 session-state`: A wrong remap target can split or mis-associate transcript state between the active store and the default state tree.

## Summary

Keep this PR open because it targets a real session-state bug that current main still does not fix, but the current patch is not merge-ready: the new remap path can ignore an active custom session store and redirect transcript writes into the default state tree.

## What This Changes

The PR adds same-agent cross-root `sessionFile` remapping in `src/config/sessions/paths.ts` plus a focused unit regression in `src/config/sessions.test.ts`.

## Best Possible Solution

Keep the narrow resolver fix, but remap same-agent stale paths relative to the active `sessionsDir`/`realBase`, preserve sibling-agent fallback behavior, and add a custom `session.store` regression before merge.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: call `resolveSessionFilePath` with an explicit `agentId`, a custom `sessionsDir` derived from `storePath`, and a stale same-agent absolute `sessionFile`; the PR helper returns the default agent sessions dir instead of the active store dir. I did not run mutating tests in this read-only review.

## Solution Assessment

No. The narrow resolver location is appropriate, but the implementation should remap into the active `sessionsDir` rather than unconditionally using `resolveAgentSessionsDir`.
