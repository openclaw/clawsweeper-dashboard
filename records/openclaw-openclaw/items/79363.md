## Summary

Keep open. The PR addresses a real allowlist gap, but the new evaluator trusts any configured builtin name, so code-evaluating builtins can be marked safe despite the PR's security claim that they are excluded.

## What This Changes

The PR adds an opt-in `tools.exec.safeBuiltins` config path, threads it through gateway/node exec allowlist evaluation, documents it, updates generated baselines, and adds safe-builtin tests.

## Best Possible Solution

Merge only after the safe-builtin evaluator fails closed for code-evaluating builtins, the docs match the actual default, and maintainers accept the new exec-approval policy knob.

## Reproduction Assessment

Yes. Source inspection on current main shows pathless builtin segments cannot satisfy allowlist, safeBins, or skill branches, and the PR body includes before/after evaluator output for the linked `cd && git` case.

## Solution Assessment

No. The opt-in config direction is plausible, but the implementation should first hard-deny code-evaluating builtins and align the user-facing default guidance with the code.

## Review Findings

Full review comments:

- **[P2] Reject unsafe builtin names before trusting safeBuiltins:** `src/infra/exec-safe-builtins.ts:63`
  - body: This accepts any configured string as a safe builtin. If `eval`, `source`, or `.` is present in `tools.exec.safeBuiltins`, the pathless code-evaluating builtin is marked `safeBuiltins` and bypasses allowlist approval, contradicting the PR's security section. Please filter to a closed safe set or hard-deny those names before returning true.
  - confidence: 0.88

That will steer users toward a riskier list while thinking it is the default; please make the docs/example match the stateless default or clarify the opt-in trade-off.
  - confidence: 0.82
