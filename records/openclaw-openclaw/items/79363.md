## Summary

Keep open: the revised PR removes the old config surface and is much narrower, but it is not merge-ready because `cd` is treated as safe while chain analysis still resolves later segments against the original cwd. That can make exec allowlist approval diverge from what the shell actually runs, especially on the node path; the PR also still lacks real configured OpenClaw exec-approval proof.

## What This Changes

The PR adds an internal safe-builtin classifier, wires shell allowlist evaluation to auto-satisfy `:`, `cd`, `false`, `pwd`, and `true`, and adds focused Vitest coverage.

## Best Possible Solution

Make `cd` safe only after cwd-aware chain analysis and node/gateway regression coverage exist, or keep `cd` approval-gated while allowing only truly stateless builtins.

## Reproduction Assessment

Yes from source, but not from a live configured exec flow in this read-only review. Current main has no safe-builtin satisfaction path for `cd`, and the PR source shows the new `cd` path while chain parts still use the original cwd.

## Solution Assessment

No. Removing the config surface is an improvement over the earlier shape, but default-allowing `cd` before cwd-aware chain evaluation is not the narrowest safe solution.
