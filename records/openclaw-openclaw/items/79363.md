## Summary

Keep open. The PR addresses a real pathless-builtin gap and fixed the earlier arbitrary-builtin filtering issue, but it still has blocking exec-policy issues around state-mutating builtins, host=node precheck propagation, config metadata, and configured-flow proof.

## What This Changes

The PR adds an opt-in `tools.exec.safeBuiltins` config with allowlist/runtime support, tests, docs, generated config hashes, and a changelog entry for selected POSIX shell builtins.

## Best Possible Solution

Keep the PR open until the policy is made state-safe or stateless-only, the same resolved builtins reach every approval decision path, config metadata is aligned, and the contributor supplies redacted configured exec-flow proof.

## Reproduction Assessment

Yes, source-reproducible. Current main has no builtin satisfaction branch, and pathless executable resolutions cannot match ordinary allowlist entries, so `cd /tmp && git status` cannot satisfy allowlist for the `cd` segment without new behavior.

## Solution Assessment

No, not yet. The opt-in direction is plausible, but this branch supports cwd/env-mutating builtins without state-aware allowlist resolution and leaves host=node precheck inconsistent; a safer implementation would be stateless-only or carry cwd/env mutations forward before approving later segments.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Do not allow stateful builtins without state tracking:** `src/infra/exec-safe-builtins.ts:28-33`
  - body: Adding `cd`, `export`, and `unset` to the supported set lets a configured builtin mutate cwd or PATH before later segments run, while command analysis resolves every segment with the original cwd/env. A chain can therefore satisfy allowlist against one executable and run another; either keep this stateless-only or carry state mutations into subsequent resolution before approving.
  - confidence: 0.84
- **[P2] Thread safeBuiltins through node precheck:** `src/agents/bash-tools.exec.ts:1238`
  - body: `createExecTool` resolves the new set, but host=node requests still have no `safeBuiltins` field and `analyzeNodeApprovalRequirement` evaluates durable node allowlists with `safeBins: new Set()` only. With `host=node`, `safeBuiltins: ["cd"]`, and a node allowlist for git, `cd /tmp && git status` can still register an approval before the node's final policy check can use the option.
  - confidence: 0.86
- **[P3] Add safeBuiltins config help metadata:** `src/config/zod-schema.agent-runtime.ts:491`
  - body: The PR adds `safeBuiltins` to the public runtime Zod shape, but `FIELD_HELP` and `FIELD_LABELS` still have no `tools.exec.safeBuiltins` entry. Generated schema/control UI metadata will expose the option without the title and description provided for nearby exec config keys.
  - confidence: 0.78
