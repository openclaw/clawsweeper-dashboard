## Summary

Keep open. The linked bug is still present on current main, but this PR has a blocking security issue in the new allowlist bypass path plus two docs/API contract mismatches before it is mergeable.

## What This Changes

The PR adds optional `tools.exec.safeBuiltins` config, runtime allowlist plumbing, docs, changelog text, and regression tests for configured POSIX shell builtins.

## Best Possible Solution

Land a maintainer-approved opt-in `safeBuiltins` implementation that validates names against a documented supported set and keeps config, docs, generated baselines, and regression coverage aligned.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: pathless builtin segments cannot match normal allowlist entries and current main has no builtin satisfaction branch. I did not run a live host exec flow in this read-only review.

## Solution Assessment

No, not yet. The opt-in config direction is plausible, but the current patch should fail closed to a supported builtin set and align docs/API comments before maintainers decide whether to accept the policy knob.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Reject unsupported safeBuiltin names before allowing them:** `src/infra/exec-safe-builtins.ts:62`
  - body: `isSafeBuiltinSegment` trusts any configured pathless token, so `tools.exec.safeBuiltins: ["eval"]`, `source`, or `.` can mark code-executing shell builtins as approved even though the PR says they are excluded. Validate against a closed supported set before returning `safeBuiltins`.
  - confidence: 0.9
- **[P3] Document the actual builtin default set:** `docs/tools/exec-approvals-advanced.md:153-156`
  - body: The docs describe `cd`, `export`, and `unset` as part of the conservative default-style set, but the code intentionally excludes those state-mutating builtins from `DEFAULT_SAFE_BUILTINS`. Users copying this example would opt into riskier behavior than the docs imply.
  - confidence: 0.86
- **[P3] Remove the nonexistent SDK constant reference:** `src/config/types.tools.ts:286`
  - body: The config type comment tells users to use `DEFAULT_SAFE_BUILTINS` from the SDK, but this PR does not add a public SDK export for that constant. Either export a real public helper through the SDK contract or document the literal supported values.
  - confidence: 0.78
