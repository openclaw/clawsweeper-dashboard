## Summary

Keep open. The opt-in builtin allowlist approach is plausible, but the patch leaves the `host=node` pre-approval path without the new `safeBuiltins` set, so configured node exec users can still be prompted or denied before node-side policy sees the setting.

## What This Changes

The PR adds opt-in `tools.exec.safeBuiltins` config, allowlist evaluator support, docs/changelog, and regression tests for pathless POSIX shell builtins in exec allowlist mode.

## Best Possible Solution

Carry the normalized builtin set through every exec approval precheck and host path that can decide allowlist satisfaction, then add focused regression coverage for gateway and host=node while keeping the closed-set, opt-in behavior documented.

## Reproduction Assessment

Source-reproducible. Current main lacks a builtin satisfaction branch, and the proposed head has a clear host=node path where `safeBuiltins` is not passed into pre-approval analysis; I did not run tests in the read-only checkout.

## Solution Assessment

No, not yet. The closed-set opt-in design may be the right direction, but the implementation needs to cover node pre-approval and show end-to-end configured exec behavior before merge.
