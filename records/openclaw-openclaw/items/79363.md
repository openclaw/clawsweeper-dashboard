## What This Changes

The PR adds optional `tools.exec.safeBuiltins` config, threads it through exec allowlist evaluation for node/gateway paths, documents it, updates generated config baselines/changelog, and adds focused tests.

## Best Possible Solution

Merge only after the evaluator filters to a closed supported builtin set or hard-denies code-evaluating builtins, the docs match the actual off-by-default/stateless behavior, and maintainers accept the new exec approval policy knob.

Yes. Current-main source shows pathless builtin segments cannot satisfy allowlist, safeBins, or skills, and the linked report plus PR body provide concrete `cd ~/` and `cd /tmp && git status` examples.

## Solution Assessment

No. The opt-in config direction is plausible, but this implementation should fail closed for unsupported/code-evaluating builtins and align the docs before merge.

- **[P2] Reject unsafe builtin names before trusting safeBuiltins:** `src/infra/exec-safe-builtins.ts:62`
  - body: `isSafeBuiltinSegment` trusts any configured builtin token once it has no resolved filesystem path.
