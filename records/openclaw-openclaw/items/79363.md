## Summary

Keep open. The linked allowlist gap is valid, but this PR is not merge-ready because the new runtime auto-allow path broadens exec approval semantics for cwd/env-mutating builtins and the documented config key is not threaded through the normal agent exec configuration path.

## What This Changes

Adds an opt-in `tools.exec.safeBuiltins` config/docs/schema path plus allowlist evaluator tests to auto-satisfy selected POSIX shell builtins in exec allowlist mode.

## Best Possible Solution

Either land a narrow docs clarification for the existing builtin limitation, or redesign runtime support so only stateless builtins are auto-allowed unless the evaluator accurately carries cwd/env mutations into later segment resolution and all config surfaces are wired and tested.

## Reproduction Assessment

Yes. Source inspection of current main gives a high-confidence reproduction path: pathless builtins cannot match ordinary allowlist entries, and segment satisfaction has no builtin branch; the linked issue provides concrete `cd` and `cd && git` examples.

## Solution Assessment

No. The PR's current direction is not the safest maintainable fix because it trusts state-mutating builtins without modeling their effect on later command resolution and misses normal agent config threading.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Do not auto-allow state-changing builtins without state tracking:** `src/infra/exec-safe-builtins.ts:15-20`
  - body: Adding `cd`, `export`, and `unset` to the safe builtin set lets a chain change cwd or PATH/env before later segments run, but the allowlist evaluator resolves every segment against the original cwd/env. That can make approval match one executable while the shell runs another, for example after `cd` with a relative executable or after `export PATH=...`. Please either restrict this to stateless builtins or carry/reject cwd and env mutations before trusting later segments.
  - confidence: 0.88
- **[P2] Thread safeBuiltins through agent exec config:** `src/config/zod-schema.agent-runtime.ts:471`
  - body: The schema/docs now accept `tools.exec.safeBuiltins`, but the normal agent exec setup still resolves only `safeBins`, `strictInlineEval`, and the safe-bin profile fields from `src/agents/pi-tools.ts`. A user setting this in global or per-agent config will not get the documented behavior in the regular exec tool path until that resolver and its coverage are updated.
  - confidence: 0.84

## Security Review

Status: needs_attention
