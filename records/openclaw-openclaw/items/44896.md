## Summary

Keep open. Current main still has the Moonshot/Kimi payload inconsistency: enabled thinking with a pinned tool choice is downgraded to `thinking.type: "disabled"`, while the OpenAI-compatible transport can independently emit `reasoning_effort`; the repo docs say incompatible Moonshot `tool_choice` values should normalize to `auto`.

## What This Changes

The issue asks OpenClaw to stop sending the invalid Moonshot/Kimi combination of non-off `reasoning_effort` with `thinking.type: "disabled"` when pinned tool choice is present.

## Best Possible Solution

Normalize pinned or otherwise incompatible Moonshot `tool_choice` values to `auto` whenever effective thinking is enabled, while preserving explicit disabled thinking and K2.6 `thinking.keep` gating.

## Reproduction Assessment

Yes. A high-confidence source-level path exists on current main: non-off Moonshot thinking plus pinned `tool_choice` becomes `thinking.type: "disabled"`, while the transport can still send `reasoning_effort`.

## Solution Assessment

No for the literal suggested `"disabled"` to `"enabled"` replacement, because that can leave a Moonshot-incompatible pinned tool choice in place. The narrower maintainable fix is the docs-aligned `tool_choice: "auto"` fallback when thinking is enabled.
