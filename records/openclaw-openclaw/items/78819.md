## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Gate proof logging behind operator-owned opt-in:** `extensions/telegram/src/runtime-proof.ts:88-94`
  - body: `resolveTelegramRuntimeProofRunId` accepts run IDs from inbound text, and dispatch passes raw Telegram message fields into that resolver. A Telegram user can include a `run_id` marker and trigger raw proof log emission plus JSONL writes for the turn, so proof capture needs an operator or harness-owned opt-in instead of prompt text.
  - confidence: 0.9
- **[P1] Use non-linkable proof identifiers:** `extensions/telegram/src/runtime-proof.ts:113`
  - body: `hashTelegramRuntimeProofId` emits a deterministic unsalted SHA-256 prefix for session and message identifiers. Telegram IDs and session keys remain linkable across proof runs and may be guessable for low-entropy values, so use a run-scoped salt, HMAC, or opaque per-run nonce.
  - confidence: 0.88
- **[P2] Use the shared diagnostics surface for proof records:** `extensions/telegram/src/runtime-proof.ts:177`
  - body: The branch appends a new Telegram-specific JSONL proof file even though current main already has sanitized diagnostics for message and delivery metadata. Extend that shared diagnostics path or prove why it is insufficient before adding a second retention and access contract.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff does not change dependencies or workflows, but the sanitized-proof design adds shareable proof logs with user-triggered activation, stable linkable identifier hashes, and a new sidecar artifact.

Concerns:

- **[medium] Inbound text can trigger proof logging:** `extensions/telegram/src/runtime-proof.ts:88`
  - body: Because the run ID can come from message text, a Telegram user can cause proof events and proof-file writes without an operator-owned proof-mode gate.
  - confidence: 0.9

  - confidence: 0.88
