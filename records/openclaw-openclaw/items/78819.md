## Summary

This PR should stay open, but it is not merge-ready: the useful Telegram proof-instrumentation idea still lacks real Telegram or Windows E2E proof, and the current design creates proof logs from user/config inputs with stable linkable identifiers and a new sidecar proof file.

## What This Changes

The PR adds Telegram runtime-proof JSON events, dispatch instrumentation, a `telegram-runtime-proof.jsonl` proof artifact, and tests for sanitized proof logging.

## Best Possible Solution

Keep Telegram proof instrumentation behind a maintainer-approved operator opt-in, remove inbound text activation, use non-linkable per-run identifiers, reuse or extend shared diagnostics where possible, rebase, and require redacted live Telegram proof.

## Reproduction Assessment

Yes for the review blockers: source inspection of the PR head shows env/message/config proof triggers, deterministic identifier hashing, JSONL writes, and a dirty branch state. No live Telegram reproduction was run, and the requested runtime proof behavior still lacks real behavior proof.

## Solution Assessment

No. The Telegram plugin is the right owner boundary, but the current implementation should be replaced with an operator-owned proof path that avoids inbound text activation, avoids stable cross-run identifiers, and preferably extends the existing diagnostics surface instead of adding a separate sidecar.
