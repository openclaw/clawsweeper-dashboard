## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P1] Gate proof logging behind an operator-owned opt-in:** `extensions/telegram/src/runtime-proof.ts:88-90`
  - body: The helper accepts `run_id=...` from inbound Telegram text, so any message can turn on proof log emission and JSONL writes for that turn. Proof capture should be controlled by an OpenClaw-owned operator/harness opt-in, not user-controlled prompt text.
  - confidence: 0.86
- **[P1] Use non-linkable proof identifiers:** `extensions/telegram/src/runtime-proof.ts:113`
  - body: A deterministic unsalted SHA-256 prefix over Telegram/session/message identifiers remains stable across proof runs and can be brute-forced for low-entropy IDs. Use a run-scoped salt, HMAC, or opaque per-run nonce so proof can correlate events without exposing stable identifiers.
  - confidence: 0.84
- **[P2] Use an OpenClaw-owned proof trigger:** `extensions/telegram/src/runtime-proof.ts:84`
  - body: `STOMME_E2E_RUN_ID` is a project-specific harness name wired into production Telegram runtime code. This should either stay in that harness or become a reviewed OpenClaw-owned opt-in before it becomes a supported environment surface.
  - confidence: 0.9
