## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Gate proof logging behind operator-owned opt-in:** `extensions/telegram/src/runtime-proof.ts:88-94`
  - body: `resolveTelegramRuntimeProofRunId` accepts run ids from inbound text, and dispatch passes RawBody/BodyForAgent/Body/CommandBody into that resolver. A Telegram user can therefore include `run_id=...` and trigger raw proof log emission plus JSONL writes for the turn; proof capture should be controlled by an OpenClaw-owned operator or harness opt-in, not prompt text.
  - confidence: 0.9
- **[P1] Use non-linkable proof identifiers:** `extensions/telegram/src/runtime-proof.ts:113`
  - body: `hashTelegramRuntimeProofId` emits a deterministic unsalted SHA-256 prefix for session and message identifiers. Telegram IDs and session keys remain linkable across proof runs and may be guessable for low-entropy values, so use a run-scoped salt, HMAC, or opaque per-run nonce instead.
  - confidence: 0.88
- **[P2] Replace the project-specific proof surface:** `extensions/telegram/src/runtime-proof.ts:84`
  - body: `STOMME_E2E_RUN_ID` wires a project-specific harness name into production Telegram runtime code and, together with the `e2e-` config-slug fallback, creates a new proof activation contract. Keep that in the external harness or replace it with a reviewed OpenClaw-owned opt-in before merge.
  - confidence: 0.86
