## Summary

Keep open. Current main still rotates the native hook relay generation on each registration and requires an exact generation match for bridge and gateway invocations; adjacent tests assert that cached commands from the prior generation fail, and no relay-surface change exists between the latest release and current main.

## What This Changes

The issue reports an OpenClaw 2026.5.26 regression where Codex native hook relay commands can keep a stale generation UUID after relay re-registration and fail with `Native hook relay unavailable`.

## Best Possible Solution

Keep this open until the Codex native hook relay preserves tool-call availability across same-session relay re-registration, with regression coverage that old cached commands after a re-register no longer strand native tools while genuinely stale bridges still fail closed.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: register a relay, build a command with its generation, re-register the same relay id, then invoke with the old generation and current main rejects it as stale. I did not run tests or live repro because this review was read-only.

## Solution Assessment

Unclear which suggested repair is best without a maintainer decision on the generation fencing contract. The narrow end state should keep stale bridge protection but avoid breaking cached Codex hook definitions for the active session.
