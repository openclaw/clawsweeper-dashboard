## Label Justifications

- `P2`: This is a real but narrow `/steer` routing bug with limited blast radius and an open paired fix path.
- `impact:message-loss`: The reported control message can be routed to an idle slash lane instead of the active run, suppressing the intended steering delivery.
- `impact:session-state`: The bug is about selecting the wrong session lane for an active run in split slash/direct or slash/DM session state.

## Summary

Keep open: current main still resolves `/steer` to a single target key and checks only that lane for an active embedded run, while an open paired fix PR already references this issue with closing syntax.

## What This Changes

The issue asks for `/steer` on an idle slash-command lane to target an active same-peer direct/DM lane before falling back as a normal prompt.

## Best Possible Solution

Review and land the paired fix PR, or replace it with the same narrow sibling-lane fallback plus focused regression coverage and real channel proof or maintainer override.

## Reproduction Assessment

Yes, by source inspection. A focused harness can call `handleSteerCommand` with `sessionKey` on a `:slash:` lane, no active run for that key, and an active embedded run registered for the same-peer `:direct:` or `:dm:` key; current main checks only the slash key.

## Solution Assessment

Yes. The narrowest maintainable fix is in `/steer` target/session-id resolution: keep an active run on the requested key, then only for idle `:slash:` lanes probe same-peer `:direct:` and `:dm:` siblings before preserving the existing fallback behavior.
