## Label Justifications

- `P2`: This is a normal-priority user-visible `/steer` bug with a narrow command/session-routing surface.
- `impact:message-loss`: The steering text can be routed to an idle slash lane instead of the active conversation lane, so the intended active run does not receive the command.
- `impact:session-state`: The bug is about resolving the wrong session lane when active run state exists on a sibling direct/DM key.

## Summary

Keep open: current main still resolves `/steer` to a single target lane and an open paired PR already references this issue with closing syntax, so this should close only after that implementation is reviewed and merged or replaced.

## What This Changes

The issue asks `/steer` to fall back from an idle `:slash:` command lane to the same-peer `:direct:` or `:dm:` lane when that sibling lane has the active embedded run.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/81613, or replace it with the same narrowly scoped fallback plus regression coverage for slash-lane to same-peer direct/DM active-run selection.

## Reproduction Assessment

Yes, at source level. A focused harness can call `handleSteerCommand` with a `:slash:` session key that has no active run and a same-peer `:direct:` or `:dm:` key with an active run; current main checks only the slash key.

## Solution Assessment

Yes. The narrowest maintainable fix is in `/steer` target/session-id resolution: preserve the requested key when active, then only for idle `:slash:` lanes probe same-peer `:direct:` and `:dm:` siblings before preserving existing fallback behavior.
