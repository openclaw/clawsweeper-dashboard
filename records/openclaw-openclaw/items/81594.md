## Label Justifications

- `P2`: The bug affects `/steer` routing for a specific slash-lane/direct-lane workflow with limited blast radius.
- `impact:message-loss`: The issue is about a management command being routed to the wrong lane and failing to affect the active run.
- `impact:session-state`: The failure depends on active embedded-run state being associated with a sibling session lane rather than the command lane.

## Summary

Keep open: current main and the latest release still resolve `/steer` against one target session key, and an open paired PR already references this issue with closing syntax, so the issue should stay open until that PR is reviewed and resolved.

## What This Changes

The issue reports that text `/steer` can resolve to a slash-command lane even when the active embedded run is on a sibling direct or DM lane.

## Best Possible Solution

Review and land the open paired fix PR, or replace it with the same narrow slash-lane fallback plus focused regression coverage and real channel proof.

## Reproduction Assessment

Yes. A focused harness can call `handleSteerCommand` with a `:slash:` target/session key that has no active run and a same-peer `:direct:` or `:dm:` key with an active run; current main checks only the slash key.

## Solution Assessment

Yes. The narrowest maintainable fix is in `/steer` session-id resolution: preserve the requested key when active, then only for idle `:slash:` targets probe same-peer `:direct:` and `:dm:` siblings before falling back.
