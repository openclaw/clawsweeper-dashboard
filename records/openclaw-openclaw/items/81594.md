## Label Justifications

- `P2`: This is a normal-priority command targeting bug with a narrow affected surface and an open fix path.
- `impact:message-loss`: The `/steer` command can be routed to the wrong lane and fail to reach the active run it is meant to steer.
- `impact:session-state`: The root problem is session-lane state selection between slash and direct/DM keys for the same peer.

## Summary

Keep open: current main still resolves `/steer` against one target key, and the open paired fix PR references this issue with closing syntax, so the issue should remain open until that implementation or an equivalent fix lands.

## What This Changes

The issue asks for text `/steer` on an idle slash-command lane to fall back to an active same-peer direct/DM lane instead of reporting no active run.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/81613, or replace it with the same tightly scoped same-peer slash-to-direct/DM fallback plus regression coverage.

## Reproduction Assessment

Yes. A focused harness can call `handleSteerCommand` with a `:slash:` session key that has no active run and a same-peer `:direct:` or `:dm:` key with an active run; current main checks only the slash key.

## Solution Assessment

Yes. The narrowest maintainable fix belongs in `/steer` session-id resolution: preserve the requested key when active, then probe only same-peer direct/DM siblings for idle slash lanes.
