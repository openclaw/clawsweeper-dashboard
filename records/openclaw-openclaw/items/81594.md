## Summary

Keep open: current main still has a source-reproducible gap in `/steer` target selection. The handler resolves one target key and checks only that key for an active embedded run, while the command/session infrastructure can keep slash-command and direct target lanes separate; the linked Telegram draft at https://github.com/openclaw/openclaw/pull/81592 covers sequentialization, not this sibling-lane target fallback.

## What This Changes

The issue requests that text `/steer` fall back from an idle slash-command lane to an active sibling direct/DM lane for the same peer.

## Best Possible Solution

Codify the `/steer` session-key contract with a focused fallback: prefer an active run on the requested raw key, then probe same-peer direct/DM siblings only for idle slash lanes, with regression coverage for the relevant channel command paths.

## Reproduction Assessment

Yes. A focused current-main check can construct `handleSteerCommand` params with `sessionKey` on `agent:main:telegram:slash:<peer>`, an active embedded run registered for `agent:main:telegram:direct:<peer>`, and observe that current code checks only the slash key.

## Solution Assessment

Yes. The narrowest maintainable fix is in `/steer` target/session-id resolution, preserving native `CommandTargetSessionKey` precedence and the existing normal-prompt fallback when no slash/direct/DM target has an active run.
