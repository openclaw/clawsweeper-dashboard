## Label Justifications

- `P2`: The report describes a bounded `/steer` routing bug with a misleading no-active-run result, not a core outage.
- `impact:message-loss`: The affected `/steer` command can fail to reach the active run when it resolves to the wrong lane.
- `impact:session-state`: The bug is about selecting the wrong session lane for an active embedded run.

## Summary

Keep open: current main and the latest release still lack the slash-to-direct/DM active-run fallback, and the linked closing PR is still open, conflicting, and awaiting proof rather than merged.

## What This Changes

The issue reports that text `/steer` can resolve to a slash-command session lane instead of an active sibling direct/DM lane.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/81613 with a rebased fix that falls back from inactive slash lanes to active sibling direct/DM lanes and adds focused regression plus real behavior proof.

## Reproduction Assessment

Yes from source: a non-native/text command turn with `sessionKey` on a `:slash:` lane and an active sibling direct/DM run will resolve only the slash key and then continue as a normal prompt. I did not run a live channel repro.

## Solution Assessment

Yes, the best fix direction is a narrow resolver fallback from an inactive slash lane to active sibling direct/DM lanes. The open PR is the right kind of fix but still needs rebase, regression coverage, and real behavior proof before landing.
