- `impact:message-loss`: The reported behavior can cause a steering message to be treated as having no active run instead of reaching the active run.
- `impact:session-state`: The failure is caused by resolving the wrong sibling session lane for an active embedded run.

## Summary

Keep open: current main and the latest release still lack the slash-to-direct/DM fallback, and the linked closing PR is open rather than merged, so this issue remains the canonical user-facing bug until that PR lands or is replaced.

## What This Changes

The issue asks `/steer` to target an active sibling direct/DM lane when the slash-command lane has no active embedded run.

## Best Possible Solution

Land or replace the linked PR with a current-main fallback that tries active direct/DM sibling lanes only after a slash-lane `/steer` target has no active run, with focused regression coverage and real behavior proof.

## Reproduction Assessment

Yes for a source-level reproduction: use a text/non-native `/steer` turn whose `sessionKey` is a `:slash:` lane while the active run exists only on the sibling `:direct:` or `:dm:` key. Current main checks the slash key and stored entry only, then falls through as a normal prompt.

## Solution Assessment

Yes, the best fix shape is a narrow fallback in `resolveSteerSessionId()` plus regression tests for the lane contract. The current linked PR is the right implementation candidate, but it needs refresh/proof before the issue can be closed.
