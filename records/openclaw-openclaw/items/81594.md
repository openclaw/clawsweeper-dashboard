## Summary

Keep open: current main still resolves `/steer` to one target key and checks only that key for an active embedded run, while an open closing PR already tracks the same fix and should be reviewed before this issue closes.

## What This Changes

Report requests `/steer` to fall back from an idle slash-command lane to the same-peer direct/DM lane when that sibling lane has the active embedded run.

## Best Possible Solution

Review and land or reject https://github.com/openclaw/openclaw/pull/81613 after adding focused regression coverage and acceptable real behavior proof for the cross-lane `/steer` path.

## Reproduction Assessment

Yes, source inspection gives a high-confidence harness path: call `handleSteerCommand` with a text `/steer` session key on `agent:main:<channel>:slash:<peer>`, no active run on that key, and an active run on the matching `:direct:` or `:dm:` key. Current main checks only the slash key and falls back as if there is no active run.

## Solution Assessment

Yes, the narrowest maintainable fix is in `/steer` target/session-id resolution: keep the requested key when active, then probe same-peer direct/DM siblings only when an idle slash lane was selected. The existing open PR is the implementation candidate, but the issue should remain open until that PR merges or is replaced.
