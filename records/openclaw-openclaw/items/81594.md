## Label Justifications

- `P2`: This is a concrete channel command bug with a narrow affected surface and an open focused fix path.
- `impact:message-loss`: `/steer` can route to the wrong lane and fail to deliver steering guidance to the active run.
- `impact:session-state`: The failure depends on active-run state living on a sibling session lane while the command resolves the slash lane.

## Summary

Keep open: current main still checks only the resolved `/steer` target lane for an active embedded run, and https://github.com/openclaw/openclaw/pull/81613 remains open with closing syntax for this issue.

## What This Changes

Text `/steer` should fall back from an idle slash-command lane to the same-peer direct/DM lane when that sibling has the active embedded run.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/81613, or replace it with the same narrow same-peer slash-to-direct/DM fallback plus regression coverage.

## Reproduction Assessment

Yes. A focused current-main harness can call `handleSteerCommand` with a `:slash:` session key that has no active run and a same-peer `:direct:` or `:dm:` key with an active run; source inspection shows current code checks only the slash key.

## Solution Assessment

Yes. The narrowest maintainable fix belongs in `/steer` target/session-id resolution: preserve an active run on the requested key, then only for idle slash lanes probe same-peer direct/DM siblings before falling back.
