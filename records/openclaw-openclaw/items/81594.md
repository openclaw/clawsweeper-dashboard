## Summary

Keep open: current main still checks only the resolved `/steer` target lane for an active embedded run, and an open fix PR already references this issue with closing syntax. The issue should stay open until that paired implementation is reviewed and resolved.

## What This Changes

The issue reports that text `/steer` can resolve to a slash-command lane and miss an active embedded run on the sibling direct/DM lane for the same peer.

## Best Possible Solution

Review and land the open paired fix PR, or replace it with the same narrowly scoped fallback plus regression coverage for slash-lane to same-peer direct/DM active-run selection.

## Reproduction Assessment

Yes. A focused current-main harness can call `handleSteerCommand` with `sessionKey` on a `:slash:` lane, no active run for that key, and an active embedded run registered for the same-peer `:direct:` or `:dm:` key; current code checks only the slash key.

## Solution Assessment

Yes. The narrowest maintainable fix is in `/steer` target/session-id resolution: preserve an active run on the requested key, then only for idle `:slash:` lanes probe same-peer `:direct:` and `:dm:` siblings before using the existing fallback behavior.
