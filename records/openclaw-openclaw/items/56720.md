## Summary

Keep open: current main still lacks Gmail excluded-label support and this PR is the canonical fix path, but the latest diff breaks the documented empty-list opt-out by omitting the flag entirely.

## What This Changes

The PR adds Gmail excluded-label config, schema, docs, generated metadata, and tests, then appends `--exclude-labels` to `gog gmail watch serve` when labels are configured.

## Best Possible Solution

Land a corrected version that restores default `SPAM,TRASH,DRAFT,SENT` exclusions, passes an explicit empty `--exclude-labels` value for `excludeLabels: []`, and keeps schema, docs, generated metadata, and tests aligned.

## Reproduction Assessment

Yes. Source inspection of current main shows no `excludeLabels` in the Gmail runtime config/schema and `buildGogWatchServeArgs` returns without `--exclude-labels`; the gateway watcher calls that builder.

## Solution Assessment

No, not as-is. The typed config/runtime/docs approach is maintainable, but the builder must emit `--exclude-labels` even when the list is empty to honor the PR docs and upstream `gogcli` empty-string contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P2] Preserve empty exclude-label overrides:** `src/hooks/gmail.ts:255-257`
  - body: The new help text says `hooks.gmail.excludeLabels: []` lets every label trigger, but this guard drops `--exclude-labels` for an empty array. Upstream `gog gmail watch serve` has its own `SPAM,TRASH` default when the flag is omitted, so an explicit empty override would still filter spam/trash instead of disabling exclusions. Always pass `--exclude-labels` with `cfg.excludeLabels.join(",")`, including the empty string.
  - confidence: 0.94
