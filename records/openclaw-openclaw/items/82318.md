## Label Justifications

- `P1`: The report describes a real recovery split-brain that can leave agents stuck during an active channel/session incident.
- `impact:session-state`: The affected behavior can create divergent session stores and stale running-state metadata for the same agent.

## Summary

Keep open: current main still resolves session CLI targets from the process-derived home/state directory and lacks a Codex shadow session-store guard or doctor detector. Related fixes cover app-server HOME inheritance and the separate reset/orphan issue, but they do not close this recovery split-brain report.

## What This Changes

The issue asks OpenClaw to prevent CLI/session recovery from silently reading or mutating Codex-isolated HOME shadow session stores, and to add recovery diagnostics or explicit store targeting.

## Best Possible Solution

Add a shared shadow-state guard and doctor detector: online recovery should prefer the running gateway/canonical state, while offline writes should require an explicit canonical env or store target and print the exact path before mutation.

## Reproduction Assessment

Yes, from source inspection for the core path: with `HOME` pointed at `agent/*/codex-home/home` and without canonical `OPENCLAW_STATE_DIR`/`OPENCLAW_CONFIG_PATH`, the session-store target resolves under that shadow home. I did not run a mutating live recovery repro in this read-only review.

## Solution Assessment

Unclear until maintainers choose the recovery UX. The narrow maintainable direction is a shared Codex-shadow detector plus gateway-first online recovery and explicit offline store targeting, rather than changing global HOME semantics.
