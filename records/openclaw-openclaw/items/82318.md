## Summary

Keep open: current main partly mitigates the default Codex app-server HOME rewrite, but the CLI/session store path still derives from the process HOME or OPENCLAW_HOME when OPENCLAW_STATE_DIR/OPENCLAW_CONFIG_PATH are absent, and doctor does not specifically detect nested codex-home shadow session stores.

## What This Changes

The issue asks OpenClaw to prevent session recovery commands from silently reading or mutating shadow state under isolated Codex/ACP HOME directories.

## Best Possible Solution

Define one canonical recovery policy for shadow Codex/ACP homes: refuse or loudly warn unless an explicit state/config/store target is supplied, print the exact store before mutation, add doctor detection, and document the supported recovery env.

## Reproduction Assessment

Yes. Source inspection shows that with HOME set to an agents/*/agent/codex-home/home path and no explicit OPENCLAW_STATE_DIR/OPENCLAW_CONFIG_PATH/--store, the sessions CLI resolves the store under that HOME-derived .openclaw tree.

## Solution Assessment

Unclear. Current main only solves the default Codex app-server HOME rewrite; the best fix needs a maintainer decision on whether session recovery should refuse, warn, or discover the gateway's canonical state path.
