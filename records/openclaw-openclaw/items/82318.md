## Label Justifications

- `impact:session-state`: The reported failure creates divergent session stores and stale running-session state, which is directly session-state drift.

## Summary

Keep open: current main reduces the default Codex app-server trigger by preserving inherited HOME, but the CLI/session maintenance path still derives OpenClaw state from the process HOME when canonical env is absent, and doctor does not specifically detect nested Codex shadow session stores. The reported split-brain recovery bug remains source-reproducible.

## What This Changes

The issue asks OpenClaw to prevent session recovery, list, and cleanup commands from reading or mutating a Codex-isolated HOME shadow state tree instead of the canonical Gateway session store.

## Best Possible Solution

Add one shared shadow-state guard and doctor detector: online recovery should use the running Gateway's canonical state, while offline writes should require explicit canonical env or --store and print the exact target path before mutation.

## Reproduction Assessment

Yes. From source inspection, running OpenClaw session commands with HOME pointed at a Codex native home and without canonical OPENCLAW_STATE_DIR/OPENCLAW_CONFIG_PATH resolves the default store under that shadow HOME.

## Solution Assessment

No. Current main is only a partial mitigation because default Codex app-server launches preserve HOME, but the OpenClaw CLI and doctor still lack a shared guard or canonical Gateway-state derivation for shadow HOME recovery paths.
