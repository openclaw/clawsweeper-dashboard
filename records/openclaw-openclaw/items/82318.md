## Summary

Keep open: current main still lets session-state resolution follow the process HOME when no canonical OpenClaw state/config env is set, so a CLI launched from an isolated Codex/ACP home can inspect or mutate a different session tree from the running gateway. The merged fix for the related recovery issue addressed reset transcript rotation and ACPX orphan reaping, but it does not add a Codex-HOME shadow guard, gateway-derived recovery target, or doctor detector for nested shadow session stores.

## What This Changes

The issue asks OpenClaw to prevent session recovery commands from silently using a Codex/ACP-isolated HOME as a shadow OpenClaw state tree instead of the gateway's canonical per-agent session store.

## Best Possible Solution

Add a shared Codex-shadow state guard and doctor detector, make online recovery prefer the running gateway's canonical session store, and require explicit canonical env or `--store` for offline writes while printing the exact path before mutation.

## Reproduction Assessment

Yes. Source inspection shows that without canonical `OPENCLAW_STATE_DIR` or config env, the session-store path follows the process HOME and CLI session paths can read or write that derived tree; I did not run a live mutating repro because this review is read-only.

## Solution Assessment

Unclear as a product decision. The narrow maintainable direction is a shared shadow-state detector plus gateway-first recovery for online writes, but maintainers should choose whether offline commands warn, refuse, or require explicit store/state targeting.
