## Summary

Keep this open. Current main has normal-path gateway cleanup and qa-channel readiness probing, but the host QA suite still lacks signal-owned shutdown for interrupted runs, restart handling still includes fixed sleeps before readiness checks, and qa-channel config is still emitted before the run’s transport readiness is confirmed.

## What This Changes

The issue requests that qa-lab reliably reap temporary gateway processes/directories, replace restart sleeps with readiness handshakes, gate qa-channel runtime config on a healthy QA bus, and avoid orphan QA configs polluting live incident signals.

## Best Possible Solution

Land a focused qa-lab fix that gives the host suite explicit SIGINT/SIGTERM ownership of spawned gateway process groups and temp roots, makes restart/config mutation paths wait on concrete gateway and transport readiness rather than fixed delays, and prevents qa-channel activation from proceeding when the run’s QA bus is unhealthy.

## Reproduction Assessment

Yes. A high-confidence reproduction path is to start the host qa-channel suite, interrupt it after the detached gateway child starts, then inspect the openclaw-qa-suite temp root and child process group; source inspection shows the suite command has no matching signal cleanup path.

## Solution Assessment

Yes. The narrow maintainable solution belongs in extensions/qa-lab because the failure is QA harness ownership/readiness behavior, not a core gateway policy change; the closed broad PR #65511 is no longer the right vehicle.
