## Summary

Keep this open. Current main still has normal-path QA gateway cleanup and a qa-channel readiness probe, but the requested interruption ownership, sleep-free restart readiness, and qa-channel config gating are not fully implemented; the related implementation PR #65511 is closed unmerged.

## What This Changes

Request hardening the QA harness so interrupted QA suite runs reap temporary gateways and runtime directories, restart handling uses readiness checks instead of fixed sleeps, and qa-channel config is emitted only for healthy QA bus runs.

## Best Possible Solution

Land a focused extensions/qa-lab repair that gives the suite host explicit ownership of spawned gateway process groups and temp roots on SIGINT/SIGTERM/exit, replaces restart-settling sleeps with concrete gateway/RPC/transport readiness checks, and prevents qa-channel activation from proceeding when the QA bus for that run is unhealthy.

## Reproduction Assessment

Yes. Start the qa-channel lane with `pnpm openclaw qa suite`, interrupt or detach it after the detached gateway child starts, then inspect the `openclaw-qa-suite-*` temp root and child process group; source inspection shows suite cleanup depends on the async finally path rather than signal-owned shutdown.

## Solution Assessment

Yes, the requested direction is the narrowest maintainable fix: keep the behavior in extensions/qa-lab, add suite-level ownership and readiness handshakes, and avoid moving QA-specific recovery policy into the core gateway.
