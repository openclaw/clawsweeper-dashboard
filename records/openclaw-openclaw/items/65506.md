## Summary

Keep this open. Current main still relies on normal async suite teardown for the QA gateway child, still has fixed restart-settle sleeps, and still enables qa-channel config before the run-specific transport readiness check; the related implementation PR was closed unmerged.

## What This Changes

The issue requests QA harness changes to prevent orphan temporary gateways, replace restart sleeps with readiness handshakes, gate qa-channel config on a healthy QA bus, and keep orphan QA configs out of live incidents.

## Best Possible Solution

Land a focused qa-lab repair that gives suite runs explicit signal-owned process/temp-root cleanup, replaces restart settling sleeps with concrete gateway/RPC/transport readiness, and prevents qa-channel activation when that run's QA bus is unhealthy.

## Reproduction Assessment

Yes from source inspection: run `pnpm openclaw qa suite`, interrupt after the detached gateway child starts, and inspect the child process group and `openclaw-qa-suite-*` temp root. I did not execute the live interruption path in this read-only pass, but current main shows the suite lacks signal-owned cleanup.

## Solution Assessment

Yes, the requested direction is the narrowest maintainable fix. The QA docs place suite startup/teardown in qa-lab and transport readiness in the adapter, so this should stay in `extensions/qa-lab` rather than becoming core gateway recovery policy.
