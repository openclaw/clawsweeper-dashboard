## Summary

This PR should stay open for maintainer/contributor follow-up rather than cleanup-close: current main does not implement the worker-state sidecar or doctor JSON option, but the live head still contains blocking doctor health and --json regressions and has no real behavior proof.

## What This Changes

The branch adds a task-registry observer that writes `worker-state.json`, adds `openclaw doctor --json`, and includes focused tests for both surfaces.

## Best Possible Solution

Repair or split the doctor JSON work so live Gateway validation and JSON-mode exit semantics are preserved, then decide whether worker-state belongs as a stable sidecar or a database/API surface aligned with the runtime-state direction.

## Reproduction Assessment

Yes for the PR defects: source inspection of PR head `77645ab64c1cb4c9eb34da0a7c1c2a27fccb61ca` shows the Gateway health check replaced with a local snapshot and the JSON writer bypassed by an early return. I did not run tests because this was a read-only review.

## Solution Assessment

No. The task-registry observer seam is plausible, but this branch is not the best merge path while it weakens doctor health validation, adds a partial JSON CLI contract, and needs maintainer agreement on the new state-file surface.
