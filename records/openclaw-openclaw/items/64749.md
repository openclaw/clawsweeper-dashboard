## Summary

Keep open for maintainer direction, but this PR is not merge-ready: the head commit is unchanged after the prior review, the diff still weakens doctor Gateway validation, the new `doctor --json` path can exit without JSON, and current main’s merged database-first runtime-state direction conflicts with adding an always-written `worker-state.json` sidecar.

## What This Changes

The PR adds a default task-registry observer that writes `worker-state.json`, tests that snapshot, and also adds partial `openclaw doctor --json` plumbing.

## Best Possible Solution

Expose observable worker/task state through the current SQLite-backed task/runtime-state contract or an explicit debug/export/API surface, and split any complete `doctor --json` contract into a separate PR that preserves live Gateway validation.

## Reproduction Assessment

Yes for the PR defects: the diff itself shows the Gateway health RPC replaced by a local snapshot and the handled update path returning before JSON output. I did not run tests because this review is read-only.

## Solution Assessment

No. The observer idea may be useful, but this branch is not the best merge path while it adds an always-written sidecar file after the database-first refactor and mixes in partial doctor JSON behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Restore live Gateway validation in doctor:** `src/commands/doctor-gateway-health.ts:27`
  - body: Replacing `healthCommand(...)` with `getHealthSnapshot(...)` stops doctor from calling the configured Gateway `status` path. A local snapshot can succeed when the Gateway is stopped or auth is invalid, so `healthOk` can become true in the failure cases doctor is supposed to surface.
  - confidence: 0.92
- **[P2] Avoid adding a normal-runtime sidecar state store:** `src/tasks/task-registry.store.ts:66-67`
  - body: Current main has moved runtime/task state toward SQLite-backed stores, with sidecar files reserved for migration/debug/export boundaries. Installing this observer by default writes `worker-state.json` on normal task events, creating a second runtime state surface that conflicts with that boundary.
  - confidence: 0.86
- **[P2] Emit JSON before handled --json exits:** `src/flows/doctor-health.ts:81-82`
  - body: When `--json` is used, the command can still return early via `updateResult.handled` before reaching the JSON write block. Interactive runs where the update preflight handles the command can therefore produce no machine-readable payload despite opting into JSON output.
  - confidence: 0.84
