## Summary

Keep open: current main does not implement the proposed snapshot or doctor JSON behavior, and the branch still has blocking review issues plus missing real behavior proof.

## What This Changes

The PR adds an always-on task-registry observer that writes `worker-state.json`, tests that snapshot, and also adds partial `openclaw doctor --json` plumbing.

## Best Possible Solution

Expose worker/task state through the SQLite-backed task/runtime-state contract or an explicit debug/export/API surface, and split any complete doctor JSON contract into a separate PR that preserves live Gateway validation.

## Reproduction Assessment

Yes for the PR defects: source inspection of the PR head shows the Gateway health RPC replaced by a local snapshot and a handled update path returning before JSON output. I did not run tests because this is a read-only review.

## Solution Assessment

No. The observability goal may be useful, but this branch is not the best merge path while it adds a normal-runtime sidecar file after the database-first refactor and mixes in partial doctor JSON behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Restore live Gateway validation in doctor:** `src/commands/doctor-gateway-health.ts:27`
  - body: Replacing the live Gateway status path with `getHealthSnapshot(...)` stops doctor from proving configured Gateway reachability and token/auth correctness before setting `healthOk` true. A local snapshot can succeed when the Gateway is stopped or rejecting auth, which masks the failures this check is meant to surface.
  - confidence: 0.92
- **[P2] Avoid adding a normal-runtime sidecar state store:** `src/tasks/task-registry.store.ts:69-70`
  - body: Installing this observer by default writes `worker-state.json` on normal task registry events, creating a second runtime state surface after current main moved task/runtime state to SQLite. This should be an explicit debug/export/API surface or use the existing SQLite-backed task state contract.
  - confidence: 0.87
- **[P2] Emit JSON before handled --json exits:** `src/flows/doctor-health.ts:81-82`
  - body: When `--json` is used, `updateResult.handled` can return before the JSON summary write runs, so an interactive handled update flow can produce no machine-readable payload despite opting into JSON output.
  - confidence: 0.84
