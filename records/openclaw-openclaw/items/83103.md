## Summary

Keep open. The issue is recent and current main has no Feishu startup/probe fix after v2026.5.12; source confirms Feishu still does a startup bot-identity probe with only short in-memory probe caching, while the reported per-message `start-account` loop still needs live/log proof before choosing the exact fix.

## What This Changes

The issue reports Feishu responses taking 4-7 minutes after v2026.5.12 because `channels.feishu.start-account` appears to re-run on every incoming message.

## Best Possible Solution

Confirm the Feishu account restart loop with verbose logs or a live repro, then fix the task exit/restart trigger and add a Feishu persistent startup bot-identity cache only if probe latency remains on the critical path.

## Reproduction Assessment

No high-confidence end-to-end reproduction yet. Source inspection confirms the startup-probe/cache path, but I did not reproduce the 4-7 minute Feishu delay or prove why `channels.feishu.start-account` repeats on every message.

## Solution Assessment

Unclear. Matching Telegram's persistent startup identity cache is plausible for the probe-delay part, but the maintainable fix should also prove and address the apparent account restart loop instead of only masking it.
