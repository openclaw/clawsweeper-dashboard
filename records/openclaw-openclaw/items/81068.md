## Summary

Keep open: this is a focused CLI bug fix with useful supplied proof, but the shared routing condition is too indirect for merge because it keys off the human command label instead of the actual JSON output mode.

## What This Changes

The PR routes secret-resolution diagnostics to stderr when `commandName` contains `--json` and adds unit plus CLI e2e coverage for `status --json` with an unresolved Discord SecretRef.

## Best Possible Solution

Route secret diagnostics from an explicit command output mode or diagnostic stream, preserve non-JSON stdout behavior, and cover `status --json` plus at least one shared-helper JSON caller whose command label does not include `--json`.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: current main logs `[secrets]` diagnostics through `runtime.log` before `status --json` writes JSON. The PR body also supplies terminal proof that the patched status path parses as JSON, though I did not run it locally.

## Solution Assessment

No. Routing based on `commandName` text fixes the reported status label but is not the best shared solution because other JSON callers use the same helper with labels that omit `--json`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Pass JSON mode explicitly for secret diagnostics:** `src/cli/command-config-resolution.ts:9`
  - body: The helper now decides stderr routing by searching the display `commandName` for `--json`, but not every JSON caller includes that flag in the label. For example, `modelsStatusCommand` resolves secrets with `commandName: "models status"` before `opts.json` writes the payload, so an unresolved model SecretRef would still prefix stdout with `[secrets]`. Pass an explicit JSON/diagnostic-stream option from the command instead of inferring it from the label.
  - confidence: 0.86
