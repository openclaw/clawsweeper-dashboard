## Summary

Keep open: the linked status JSON bug is valid and the PR includes convincing terminal proof, but the implementation routes diagnostics by parsing the display command name, leaving shared JSON callers that use labels without `--json` exposed to the same stdout pollution.

## What This Changes

The PR routes command SecretRef diagnostics to stderr when the command label contains `--json` and adds unit plus CLI e2e coverage for `status --json` with an unresolved Discord SecretRef.

## Best Possible Solution

Pass an explicit JSON output mode or diagnostic-stream option into `resolveCommandConfigWithSecrets`, preserve non-JSON stdout behavior, and cover both `status --json` and a shared-helper JSON caller whose label omits `--json`.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: `status --json` resolves command secrets before `writeRuntimeJson`, and current main emits those diagnostics through `runtime.log`; the PR body also supplies after-fix terminal proof for the reported status path.

## Solution Assessment

No. The PR fixes the reported label but the command-name heuristic is not the narrow maintainable contract because other JSON callers use the same helper with labels that do not include `--json`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Pass JSON mode explicitly for secret diagnostics:** `src/cli/command-config-resolution.ts:9-10`
  - body: The helper chooses stderr by parsing the display `commandName`, but JSON callers can invoke it with labels that omit `--json`. For example, `modelsStatusCommand({ json: true })` calls `loadModelsConfig({ commandName: "models status", runtime })` before `writeRuntimeJson`, so unresolved model SecretRefs would still prefix stdout with `[secrets]`. Pass an explicit output mode or diagnostic stream option instead.
  - confidence: 0.86
