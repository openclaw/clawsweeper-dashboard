## Summary

Keep open: current main still has the bounded WebChat history behavior this request describes, and the existing cursor-capable session history endpoint is not wired into the WebChat scrollback UI. The related issues cover adjacent tool-noise, refresh-state, transcript-browser, and search work, but none is a canonical duplicate for paginated in-thread scrollback.

## What This Changes

Request adds WebChat paging so users can load older messages beyond the current bounded recent-history window.

## Best Possible Solution

Add WebChat paging state and a visible load-older control backed by the existing cursor history endpoint, with dedupe, scroll-anchor preservation, and focused tests for long tool-heavy transcripts while keeping live WS events unchanged.

## Reproduction Assessment

Yes for the current limitation: source and tests show WebChat fetches and renders a bounded recent window with a passive hidden-count notice. I did not run a live browser repro in this read-only review.

## Solution Assessment

Yes, the recommended direction is the maintainable one: use the existing cursor history surface for older pages instead of increasing default payload size. Maintainers still need to choose the exact affordance and filtering/overscan behavior.
