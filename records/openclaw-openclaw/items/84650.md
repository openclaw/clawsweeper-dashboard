## Summary

Keep open: current main still has the bounded WebChat history behavior described here, and no merged/current implementation gives WebChat an actionable load-older path. The open pagination PR at https://github.com/openclaw/openclaw/pull/72178 is a relevant implementation candidate, but it has unresolved review/proof blockers and does not close this issue; https://github.com/openclaw/openclaw/pull/83992 covers a related refresh-visible-history slice, not paginated in-thread scrollback.

## What This Changes

The issue requests paginated WebChat scrollback so users can load messages older than the current bounded recent history window.

## Best Possible Solution

Add WebChat paging state and a visible load-older affordance backed by the existing cursor history contract or an additive chat.history pagination contract, with dedupe by stable transcript metadata, scroll-anchor preservation, and focused long tool-heavy transcript tests.

## Reproduction Assessment

Yes at source level: current main requests a fixed 100-message WebChat history window, renderer tests expect only the last 100 messages plus a passive hidden notice, and no UI path loads older pages. I did not run a live browser repro in this read-only review.

## Solution Assessment

Yes, the requested direction is the maintainable one: use cursor pagination for older pages instead of raising the default history payload. The exact WebChat affordance and WebSocket-versus-HTTP integration still need maintainer product/API direction.
