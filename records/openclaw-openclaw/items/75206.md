## Summary

Keep open. Current main still leaves the central stale-chunk failure mode in place: Gateway `update.run` can run the global package updater from the live Gateway process, the package tree can be swapped before the old process exits, and release validation still treats the resulting self-swapped `ERR_MODULE_NOT_FOUND` as acceptable after a nominally successful update. The follow-up comment usefully confirms the same root cause on a CLI-wrapper path, while related PR #67702 is narrower and still open.

## What This Changes

Fix package and auto-update so global installs cannot leave a live Gateway or CLI process holding stale hashed dist chunk imports after replacing the package tree.

## Best Possible Solution

Add a generic package-update safety boundary: Gateway or auto-update initiated package-manager updates should delegate replacement to a fresh external updater or fail before mutating the active package tree. Release validation should stop accepting stale self-swapped missing chunk exits unless an intentional recovery path proves the old process exits cleanly.

## Reproduction Assessment

Yes. The issue and follow-up comment provide concrete versions, install method, missing hashed chunk names, timestamps, and restart recovery, and current source shows the matching live-process package-swap path.

## Solution Assessment

No. Telegram-specific restart or alert handling would not cover the CLI-wrapper repro; the maintainable fix is a shared package-update handoff or fail-closed boundary before package mutation.

## Repair Work Prompt

Implement a narrow fix for https://github.com/openclaw/openclaw/issues/75206. The observable bug is that package/auto-update can replace the active global package tree while the live Gateway or invoking CLI process still holds ESM module records for old hashed dist chunks, causing `ERR_MODULE_NOT_FOUND` until restart. Fix the generic update boundary, not Telegram-specific recovery: package-manager updates initiated by Gateway `update.run` or auto-update should delegate package replacement to a fresh external updater before mutation, or fail closed before mutation with a clear status when safe handoff is unavailable. Tighten cross-OS release validation so a successful JSON update followed by stale self-swapped `ERR_MODULE_NOT_FOUND` is no longer accepted unless the code also proves an intentional recovery path exits the old process cleanly. Preserve existing CLI hardening for shell-driven package updates, keep core extension-agnostic, do not change dependency patches or package-manager versions, and add a user-facing `CHANGELOG.md` fix entry with appropriate non-bot credit. Check related PR https://github.com/openclaw/openclaw/pull/67702 for context but do not make this a git-mode-only fix.
