## Summary

Keep open: current main still source-reproduces the stale current-conversation binding path, while the stuck-session half is only partly addressed by current diagnostic recovery code. Open fix PRs already cover the remaining binding hygiene work, so this issue should stay open until one lands or is explicitly replaced.

## What This Changes

The issue reports gateway session resume losing continuity when stale LCM/current-conversation bindings point at missing session files, plus stuck processing sessions that only logged diagnostics before recovery.

## Best Possible Solution

Land or replace the generic stale-binding reconciliation that validates runtime targets before route rewrite and unbinds keys removed by cleanup, while preserving the current cron-run guard and existing stuck-session recovery coverage.

## Reproduction Assessment

Yes, source-reproducible: current main can resolve and touch a persisted current-conversation binding whose target session/transcript is missing, and cleanup can prune missing-transcript store rows without clearing matching bindings. I did not run the live Discord/LCM setup.

## Solution Assessment

Yes: generic binding target validation plus cleanup-time unbinding is the narrowest maintainable fix for the remaining binding hygiene bug. Restoring LCM history into a new session would be more channel/plugin-specific and is not required to stop routing to orphaned targets.
