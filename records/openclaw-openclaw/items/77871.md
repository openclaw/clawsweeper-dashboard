## Summary

This should stay open: current main appears to have stuck-session recovery, but the stale runtime binding and cleanup invalidation paths still trust orphaned session keys, so the reported continuity-loss path remains plausible from source inspection.

## What This Changes

The issue requests gateway session-resume recovery for orphaned LCM/current-conversation bindings, cleanup-time binding invalidation, and stuck-session escalation.

## Best Possible Solution

Validate runtime current-conversation bindings against the session store/transcript before route rewrite, unbind stale targets during resolution and cleanup, and keep the existing stuck-session recovery with focused regression coverage.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source inspection gives a credible reproduction path for the stale-binding half: create a current-conversation binding to a session key, remove its store/transcript, then resolve the same conversation and observe the route still rewrite to the missing target. I did not live-run the gateway/Discord path, so this is source-reproducible rather than fully reproduced.

## Solution Assessment

Is this the best way to solve the issue? Yes for the stale-binding half: a generic binding validation and cleanup invalidation seam is narrower than channel-specific recovery and uses existing binding-service unbind support. The stuck-session half should first be verified against current diagnostics recovery before adding more policy.
