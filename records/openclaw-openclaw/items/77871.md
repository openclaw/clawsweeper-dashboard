## Summary

Keep open: current main has actionable stuck-session recovery, but runtime conversation bindings still resolve and refresh stale targets without validating that the target session entry/transcript exists, and cleanup paths can prune session rows without unbinding matching conversation bindings.

## What This Changes

Request gateway session-resume recovery for stale LCM/current-conversation bindings, missing session files, and stuck session queues.

## Best Possible Solution

Add generic session-binding hygiene: validate a runtime binding’s target session before route rewrite, unbind stale targets with a warning, and have cleanup pruning clear matching bindings through the same lifecycle path without adding LCM-specific core logic.

## Reproduction Assessment

Yes, source-reproducible: a persisted current-conversation binding can point at a missing target session/transcript, and current main still resolves and touches that binding before rewriting the route. I did not run a live Discord/LCM reproduction because this review was read-only.

## Solution Assessment

Yes, the narrowest maintainable fix is generic binding reconciliation and cleanup unbinding. Restoring LCM history into a new session would be plugin-specific and should not be required for core to stop routing to orphaned session targets.

## Repair Work Prompt

Fix the stale runtime conversation binding regression reported in https://github.com/openclaw/openclaw/issues/77871. Add generic core hygiene so current-conversation/runtime bindings do not route to missing OpenClaw session state: when resolving a non-plugin-owned runtime binding, validate that the target session key still has a usable session row and transcript; if stale, unbind that binding, log/diagnose once, and fall through to the normal fresh persisted session route. Also make session cleanup paths that prune missing/stale/capped/budget-evicted session entries clear matching session bindings through the existing session binding service. Do not add Lossless Claw/LCM-specific ids, migrations, provider policy, or history-restore behavior in core. Add regression tests for stale binding resolution and cleanup-driven unbinding, update docs only if the public behavior changes, and add a user-facing CHANGELOG.md fix entry.
