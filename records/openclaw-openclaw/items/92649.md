## Summary

Keep open: current main does not have Kimi Code usage reporting, and this PR is a useful path, but it should not merge while Kimi usage auth can reuse Moonshot credentials and send them to the Kimi Code usage endpoint.

## What This Changes

Adds a Kimi Code usage fetcher, wires Kimi usage auth/snapshot hooks, maps Kimi usage aliases, exports the helper through the plugin SDK, and updates focused tests plus the API baseline.

## Best Possible Solution

Restrict Kimi usage auth to Kimi Code credential sources unless maintainers intentionally update docs and add live proof that Moonshot/Open Platform keys are supported for this endpoint.

## Reproduction Assessment

Yes, from source inspection: loadProviderUsageSummary asks each usage provider for auth, and the new Kimi hook accepts moonshot config/env credentials before calling the Kimi Code usage endpoint. I did not run a live Moonshot-key reproduction because no credential is available, but repo docs and upstream Kimi docs make the mismatch high-confidence.

## Solution Assessment

No. The provider-owned usage hook is the right layer, but Moonshot credential reuse is not the safest implementation; keep Kimi Code and Moonshot usage auth separate unless maintainers deliberately change that contract with proof.
