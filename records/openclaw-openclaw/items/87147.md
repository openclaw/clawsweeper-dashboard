## What This Changes

This PR changes `sessions.usage` omitted-agent handling to search all configured agents, adds `usageAgentId` request state plus a Usage-page agent selector, and updates focused tests/i18n.

## Best Possible Solution

Keep `sessions.usage` scoped by default, make the Control UI use explicit agent scopes or a maintainer-approved explicit all-agents contract, then land with type fixes and real multi-agent proof.

## Reproduction Assessment

Yes. Current source shows the Web UI `loadUsage` request omits `agentId` while current main's backend defaults omitted `agentId` to the default agent, so non-main rows are source-reproducible without running a live setup.

## Solution Assessment

No. The frontend state direction is plausible, but making omitted backend `agentId` mean all agents is not the safest fix for a scoped session API; use explicit per-agent requests or an approved explicit all-agents contract.

- **[P1] Preserve the scoped sessions.usage default:** `src/gateway/server-methods/usage.ts:880`
  - body: This makes an omitted `agentId` load the combined store without a scope and pass `undefined` into discovery, so every configured agent is returned. Current main and the v2026.5.17 hardening PR https://github.com/openclaw/openclaw/pull/81386 scope omitted `agentId` to the default agent; widening the default reintroduces cross-agent session visibility for any Gateway caller that omits `agentId`. Keep the backend default scoped and make the UI use explicit per-agent requests, or add an explicit maintainer-approved all-agents contract with docs/tests.
  - confidence: 0.9

  - body: All names imported from `usageTypes.ts` here are types only, but the PR turns the `import type` into a runtime import. Current main enables `verbatimModuleSyntax`, and `usageTypes.ts` exports no values for these names, so the type/build lane will fail. Keep this as `import type`.
  - confidence: 0.86
- **[P2] Update UsageProps fixtures for the new required fields:** `ui/src/ui/views/usageTypes.ts:41`
  - body: Adding required `usageAgentId` and `onAgentChange` fields leaves the existing `createUsageProps` fixture in `ui/src/ui/views/usage.test.ts` missing required `UsageProps` members. Add defaults to every fixture, or make the fields optional intentionally, so the UI typecheck/test lanes do not fail.
  - confidence: 0.8
