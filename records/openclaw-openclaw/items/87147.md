## Summary

Keep open: the PR is a plausible direction, but the new selector is populated from already-returned usage sessions, so a fresh default-scoped Usage page still cannot discover or select non-default agents.

## What This Changes

The PR adds `usageAgentId` state, passes it to `sessions.usage`, and renders a Control UI Usage-page agent selector with updated English UI copy and fixture types.

## Best Possible Solution

Populate the usage server-scope selector from configured agents, pass the selected `agentId`, keep omitted-agent backend behavior scoped to the default agent, and prove the flow with a real multi-agent Control UI run.

## Reproduction Assessment

Yes. Source inspection shows current `sessions.usage` defaults omitted `agentId` to the default agent while the current UI omits `agentId`, and the PR's selector still derives choices from those already-scoped results.

## Solution Assessment

No. Passing `agentId` is the right direction, but the selector needs an independent configured-agent source such as `agents.list`; deriving options from `sessions.usage` results keeps the non-default agents unreachable.
