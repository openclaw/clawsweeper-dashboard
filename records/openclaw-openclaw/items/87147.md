Full review comments:

- **[P1] Add usageAgentId to AppViewState:** `ui/src/ui/app-render-usage-tab.ts:66`
  - body: This line reads `state.usageAgentId`, but the PR does not add that property to `AppViewState` in `ui/src/ui/app-view-state.ts`, whose usage section currently stops at `usageScope` before the selection fields. The Control UI TypeScript build will reject the new `renderUsageTab` access until the shared app view-state type is updated.
  - confidence: 0.94
- **[P2] Handle agentId on older usage gateways:** `ui/src/ui/controllers/usage.ts:228`
  - body: The usage controller already retries when older gateways reject newer `sessions.usage` fields, but the new `agentId` field has no matching unsupported-param fallback. A user connected to a pre-`agentId` gateway can pick an agent from session-derived options and turn a previously working usage page into an invalid-request error.
  - confidence: 0.78
- **[P2] Do not label the default-scoped request as all agents:** `ui/src/ui/views/usage.ts:606`
  - body: The empty option is labeled `All agents`, but clearing `usageAgentId` omits the field and current `sessions.usage` resolves that to the configured default agent, not every agent. That makes the initial selector state misreport the data set; use default-agent wording or implement a true all-agent request contract.
  - confidence: 0.88
- **[P2] Regenerate generated locale bundles:** `ui/src/i18n/locales/en.ts:782`
  - body: Adding an English i18n key without the generated locale and `.i18n` updates leaves `pnpm ui:i18n:check` dirty; `test:ui` runs that check before UI tests. Run the Control UI i18n sync and include the generated outputs, or avoid introducing a new key.
  - confidence: 0.86
