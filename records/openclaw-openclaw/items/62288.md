## Summary

Keep open. Current main has improved the managed default, pageReady status, and DevToolsActivePort messaging, but the canonical existing-session attach diagnostics work is still incomplete because profile/doctor output still loses attach failure reasons and Chrome MCP tab selection can still treat browser-internal pages as normal pages. The FSB link in the latest comment is useful prior art for the desired managed/user/owned-tab diagnostic split, but it does not make the OpenClaw browser plugin bug obsolete.

## What This Changes

Improve browser existing-session attach reliability and diagnostics while keeping the managed openclaw profile as the default lane.

## Best Possible Solution

Finish a focused browser-plugin repair that preserves Chrome MCP attach failure reason codes through status, profiles, doctor, and CLI output, filters browser-internal Chrome MCP pages before default target selection, and leaves larger extension-bridge ideas as separate product work.

## Reproduction Assessment

Yes, at source level. Mocking Chrome MCP transport/list failures and a list_pages response where an internal chrome:// page precedes a real tab should reproduce the remaining diagnostics and default-selection gaps without a live Chrome setup.

## Solution Assessment

No, current main is only a partial solution. The narrow maintainable fix belongs in the browser plugin's Chrome MCP status/profile/doctor and tab-listing paths, while the FSB-style extension bridge should stay as prior art unless maintainers choose a larger product direction.
