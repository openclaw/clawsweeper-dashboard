## Summary

Keep open. Current main covers the managed `openclaw` default, better DevToolsActivePort messaging, `pageReady` status, and CDP internal-target filtering, but the canonical acceptance bar is still not met because profile/doctor diagnostics drop attach reason codes and Chrome MCP tab selection can still surface internal pages.

## What This Changes

The issue asks OpenClaw to keep the managed `openclaw` browser lane as the reliable default while hardening `profile="user"` existing-session attach, fallback behavior, and diagnostics.

## Best Possible Solution

Finish this as a narrow browser-plugin fix: preserve attach failure reason codes through status/profiles/doctor, keep managed `openclaw` as the default fallback guidance, and filter or de-prioritize Chrome MCP browser-internal pages before default target selection.

## Reproduction Assessment

Yes. Source-level reproduction is high-confidence: mock Chrome MCP transport/list failures and a `list_pages` response with `chrome://inspect` before a real page; current main drops attach reasons from `/profiles`/doctor and does not filter Chrome MCP internal pages before selection.

## Solution Assessment

No. Current main is only a partial solution; the maintainable fix belongs in the browser plugin status/profile/doctor and Chrome MCP tab-listing paths, with additive diagnostics and focused regression coverage rather than core fallback policy changes.
