## Summary

Keep open. Current main has the managed `openclaw` default and `pageReady` status improvements, but the canonical existing-session diagnostics problem is still present because profile listing still drops Chrome MCP attach failure details and Chrome MCP tab listing still maps every `list_pages` result into a selectable page.

## What This Changes

The issue asks OpenClaw’s browser plugin to prefer the managed `openclaw` lane by default, make `profile="user"` attach failures less brittle around DevToolsActivePort, and report managed-vs-existing-session health separately.

## Best Possible Solution

Finish this in the browser plugin by carrying structured Chrome MCP attach/page failure reasons through status, profiles, and doctor output, and by filtering or deprioritizing browser-internal Chrome MCP pages before default target selection.

## Reproduction Assessment

Yes, at source level. Mocking Chrome MCP transport/list failures during profile listing and a `list_pages` response with an internal `chrome://` page before a real page should reproduce the remaining current-main gaps without a live Chrome setup.

## Solution Assessment

No, current main is only a partial solution. The narrow maintainable fix is to preserve attach failure reasons through the browser plugin’s status/profile/doctor paths and to filter Chrome MCP target selection, while leaving any Chrome-extension bridge redesign for separate product discussion.
