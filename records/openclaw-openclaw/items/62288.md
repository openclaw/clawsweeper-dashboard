## Summary

Keep open. Current main has several partial fixes for this browser lane, including managed `openclaw` as the default and better DevToolsActivePort fallback text, but it still does not satisfy the canonical acceptance bar from the discussion: per-profile diagnostics do not preserve existing-session failure reasons, and target selection can still prefer browser-internal Chrome MCP pages over real user tabs.

## What This Changes

The issue requests hardening the browser plugin's `existing-session` Chrome MCP attach path, preferring the managed `openclaw` profile by default, and improving fallback/status diagnostics when `profile="user"` fails.

## Best Possible Solution

Keep this issue as the canonical browser plugin follow-up and finish the extension-owned behavior: explicit `profile="user"` should fail with reason-coded existing-session errors, status/doctor should show managed and existing-session health independently, and default target selection should avoid Chrome/DevTools/internal pages unless explicitly targeted.

## Reproduction Assessment

Yes. The live report has two independent user confirmations, and a focused source-level repro is clear: mock Chrome MCP `list_pages` to return `chrome://inspect` before a real tab and verify current default selection still picks the first page; mock DevToolsActivePort failures to verify the diagnostics path.

## Solution Assessment

No. Current main is only a partial solution; the maintainable path is a narrow browser plugin fix with regression tests around Chrome MCP target filtering, reason-coded attach failures, and per-profile doctor/profile output rather than core migrations or a broader browser redesign.
