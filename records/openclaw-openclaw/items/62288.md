- `impact:session-state`: The failure can attach to or fall back to the wrong browser session/profile, which mis-associates browser state for agent work.

## Summary

Keep open. Current main already prefers the managed `openclaw` browser lane and has partial Chrome MCP diagnostics, but the canonical existing-session repair is still incomplete because attach failure reasons are not carried through profiles/doctor output and Chrome MCP page selection can still treat browser-internal pages as normal tabs.

## What This Changes

The issue asks OpenClaw to make the managed browser lane the reliable default, harden `profile="user"` Chrome MCP attach, and improve fallback/status diagnostics when existing-session attach fails.

## Best Possible Solution

Finish a focused browser-plugin repair that preserves Chrome MCP attach failure reason codes through status, profiles, doctor, and CLI output, filters browser-internal Chrome MCP pages before default target selection, keeps `openclaw` as the default lane, and leaves extension-bridge architecture for separate product work.

## Reproduction Assessment

Yes, at source level. A mocked Chrome MCP list containing an internal `chrome://` page before a real page exercises the default-selection gap, and mocked Chrome MCP attach failures show profile/doctor output losing the underlying reason.

## Solution Assessment

No, current main is only a partial solution. The narrow maintainable fix is to carry structured attach diagnostics and reuse the existing internal-target filtering pattern for Chrome MCP pages, not to replace the browser architecture with an extension bridge in this issue.
