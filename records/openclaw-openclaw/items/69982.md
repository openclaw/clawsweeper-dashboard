## Summary

Keep this PR open. The reported leak is real on current main and the patch is directionally useful, but it is incomplete because the same-button toggle dismissal still removes the existing popover without unregistering the document capture listener; the same author also has a newer open related PR, so maintainers should choose one canonical branch rather than closing this as implemented.

## What This Changes

The PR changes `ui/src/ui/chat/grouped-render.ts` to add a shared delete-confirm popover dismissal helper so Cancel and Delete remove the document click listener before removing the popover.

## Best Possible Solution

Land one complete, narrow UI patch that routes every delete-confirm dismissal path through one cleanup routine or tracked active cleanup reference: Cancel, Delete, outside click, and clicking the same delete button again while the popover is open. Add a focused regression test that proves the document listener is removed; then close or supersede the duplicate branch once the canonical fix merges.

## Reproduction Assessment

Yes. Open a delete confirmation, let the `requestAnimationFrame` callback register the document capture listener, then click the same delete button again; the current PR’s still-present `existing.remove(); return;` path removes the popover without `removeEventListener`. Current main also reproduces the original Cancel/Delete leak directly from the separate `popover.remove()` calls.

## Solution Assessment

No, not as written. A shared cleanup helper is the right narrow solution, but this patch only covers part of the dismissal surface and needs the toggle-close path covered before merge.
