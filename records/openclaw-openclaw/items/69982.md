## Summary

Keep this PR open: it targets a real current-main listener leak in the Control UI delete-confirm popover, but the proposed patch is incomplete because the same-button toggle close path still removes the popover without unregistering the captured document click listener.

## What This Changes

The PR modifies `ui/src/ui/chat/grouped-render.ts` to add a shared delete-confirm popover dismissal helper for Cancel, Delete, and outside-click cleanup of the document click listener.

## Best Possible Solution

Update this PR or land an equivalent narrow patch that routes every delete-confirm dismissal path through one cleanup routine, including Cancel, Delete, outside click, and clicking the delete button again while the popover is open. Add a focused regression test that observes `document.addEventListener` and `document.removeEventListener`; no docs or changelog entry is needed for this internal UI leak fix.

## Reproduction Assessment

Yes. The code-backed reproduction is to open the delete confirmation, wait for the `requestAnimationFrame` document listener registration, then dismiss with Cancel/Delete on current main or click the same delete button again on the PR branch; in those paths the popover is removed without listener cleanup.

## Solution Assessment

No, not as written. The shared helper is the right maintainable direction, but it must also cover the existing-popover toggle branch and should be locked with a focused regression test.
