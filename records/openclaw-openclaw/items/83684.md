## Summary

Keep open: this is a maintainer-authored, protected-label PR that addresses a plausible QR/mobile onboarding regression, but it changes an auth security boundary and the patch misses a restored-bootstrap retry path where the operator handoff can still be lost.

## What This Changes

The PR restores setup-code QR bootstrap to return a node token plus a bounded operator handoff token, updates native client scope handling, documents the contract, and adds focused gateway/bootstrap tests.

## Best Possible Solution

Keep the bounded native QR handoff only if maintainers explicitly accept the setup-code trust model, then fix the retry gap and verify the native onboarding path end to end.

## Reproduction Assessment

Yes. Current main is source-reproducible as node-only setup-code bootstrap with no operator deviceTokens, and the PR body plus related merged commits trace how that regressed the native handoff contract.

## Solution Assessment

Not yet. Restoring the bounded handoff in the gateway profile is a plausible fix, but the implementation should preserve the operator handoff through restored-bootstrap retries and needs maintainer acceptance of the security boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.78

Full review comments:

- **[P2] Preserve operator handoff on restored bootstrap retries:** `src/gateway/server/ws-connection/message-handler.ts:1408`
  - body: The handoff list is only populated when this connection just auto-approves the bootstrap pairing. If `sendFrame` throws, the existing recovery restores the bootstrap token but leaves the paired device and tokens in place; the next bootstrap retry skips `requirePairing`, leaves `handoffBootstrapProfile` null, and returns only the node token. Derive the handoff from the restored setup profile or paired operator token on that retry, or roll back the paired-device write on hello-send failure.
  - confidence: 0.78
