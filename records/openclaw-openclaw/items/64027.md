## Summary

Keep this PR open. Current main still lacks the requested Feishu in-place card update behavior, and the PR is a useful implementation candidate, but it is not ready because the new update helper only patches callbacks with `message_id` while existing accepted callback shapes can carry only `open_message_id` or `context.open_message_id`; the PR also includes unrelated Feishu reply-dispatcher and monitor-comment test churn.

## What This Changes

The PR adds an optional Feishu card-action update payload and handler-side card patching, parses `message_id` from card callbacks, and also changes Feishu reply-dispatcher session-token footer plumbing while deleting monitor comment handler tests.

## Best Possible Solution

Revise and rebase this PR into a focused Feishu-owned change that keeps the update-card envelope and handler wiring, normalizes the target card id across `message_id`, `open_message_id`, and `context.open_message_id`, adds targeted tests for cancel/confirm/quick/malformed/missing-id cases, and moves unrelated footer/session-store or monitor-comment test changes to a separate review.

## Reproduction Assessment

Yes. Current main reproduces the missing feature by inspection because cards cannot carry `uc` and the handler never calls `updateCardFeishu`; the PR-specific regression reproduces with the existing `open_message_id`-only callback shapes covered by lifecycle tests, where the new helper returns before patching.

## Solution Assessment

No. The implementation direction is reasonable, but the narrow maintainable fix should normalize the existing callback target id fallback before patching and keep unrelated Feishu reply-dispatcher/test churn out of this feature PR.
