## Summary

Keep open. Current main still lacks in-place Feishu card updates, and the PR is useful, but the proposed helper only patches when a new `message_id` field is present even though current accepted callback shapes already use `open_message_id` and `context.open_message_id`; those callbacks would still dispatch while leaving the clicked card stale.

## What This Changes

The PR adds a Feishu card-action `uc` update-card envelope field, parses a callback `message_id`, calls `updateCardFeishu` for cancel/confirm/quick actions, and also changes Feishu reply footer session-store behavior while deleting monitor comment handler tests.

## Best Possible Solution

Revise the PR into a focused Feishu card-update change: normalize the patch target from `message_id ?? open_message_id ?? context.open_message_id`, keep card update failures non-fatal to command dispatch, cover cancel/confirm/quick/malformed/missing-id/open-message-id cases, restore or split unrelated reply-dispatcher and comment-test changes, add the required changelog entry, then run targeted Feishu tests plus a live callback check.

## Reproduction Assessment

Yes. Current main's missing behavior is reproducible by inspection of the card-action envelope and handler paths, and the PR-specific regression is reproducible with the existing lifecycle callback shapes that carry `open_message_id` but no top-level `message_id`.

## Solution Assessment

No. The implementation direction is maintainable, but this patch is not the best final form because it keys the update on only one callback id shape and bundles unrelated Feishu reply footer/test churn into the card update feature.
