## Summary

Keep this PR open, but it is not merge-ready: current main still lacks the requested Feishu in-place card update feature, while this branch has source-level blockers, stale/conflicting Feishu callback code, and no real Feishu behavior proof.

## What This Changes

The branch adds a Feishu `uc`/`updateCard` card-action payload, parses `message_id`, patches cards on cancel/confirm/quick actions, and also changes reply-dispatcher footer/session handling while deleting Feishu comment monitor tests.

## Best Possible Solution

Rebase into a focused Feishu plugin change that patches by the normalized callback card id, uses current narrow SDK subpaths, keeps unrelated churn out, adds targeted tests, and includes redacted live Feishu proof.

## Reproduction Assessment

Yes, source-level. PR head's `maybeUpdateCard` only accepts `event.message_id`, while current main accepts callback shapes that provide `open_message_id` via the top-level payload or context.

## Solution Assessment

No. The direction is plausible, but this branch should preserve current callback id normalization, avoid deprecated broad SDK imports, and be narrowed before it is a maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

  - confidence: 0.9

Import `loadSessionStore` from the narrow session-store runtime subpath, or keep this unrelated footer/session change out of the card-update PR.
  - confidence: 0.86
