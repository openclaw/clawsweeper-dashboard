## Best Possible Solution

Land after provider auth listing and resolution share one explicit non-interactive contract, and after realtime audio upload uses bounded off-main buffering while keeping MainActor only for UI state.

## Solution Assessment

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Use one auth store for listing and resolving profiles:** `src/plugin-sdk/provider-auth.ts:305-308`
  - body: `listUsableProviderAuthProfileIds` can now return IDs from the promptable fallback store, but `resolveProviderAuthProfileApiKey` then reloads `loadAuthProfileStoreForSecretsRuntime`. Profiles only visible in the fallback pass are advertised as configured and still fail to resolve, so OpenAI realtime OAuth can fail at session start with missing auth.
  - confidence: 0.9
- **[P2] Move realtime audio upload off MainActor:** `apps/ios/Sources/Voice/RealtimeTalkRelaySession.swift:21-23`
  - body: The audio tap creates a MainActor task for every buffer, and that path base64-encodes JSON and awaits `talk.session.appendAudio`. When gateway upload is slower than capture, those tasks can pile up on the UI actor and make Talk Mode stutter; use a bounded background actor or queue for audio upload and hop to MainActor only for status updates.
  - confidence: 0.84
