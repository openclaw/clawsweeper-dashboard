## Summary

Keep this PR open: it targets a valid memory-core bug, but the current patch is not merge-ready because the new settlement fanout can wait indefinitely on a hung sibling supplement, and the posted proof is still test-output-only rather than observed live tool output.

## What This Changes

The PR changes memory-core supplement search to use settled-result aggregation, warn with the rejecting plugin id, adds memory-core regression coverage, and adds a changelog fix entry.

## Best Possible Solution

Land a bounded memory-core partial-failure implementation that preserves fulfilled supplement results, warns for failed or timed-out supplements by plugin id, and is proven with redacted real memory_search corpus=all output.

## Reproduction Assessment

Yes, source-reproducible: register one resolving and one rejecting memory corpus supplement, then call memory_search with corpus=all; current main awaits Promise.all in searchMemoryCorpusSupplements, so the rejection aborts sibling results. I did not run tests because this review is read-only.

## Solution Assessment

Mostly yes on the root fix direction: handling partial supplement failure inside memory-core is the narrow owner-module solution. The current implementation still needs a bounded wait/abort path for hung supplements and real behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Bound supplement searches before waiting for all settlements:** `extensions/memory-core/src/tools.shared.ts:169`
  - body: `Promise.allSettled` waits for every supplement promise to settle. If one supplement hangs while another rejects, `memory_search corpus=all` now never returns, so the partial-failure path can stall the hot memory tool instead of degrading; add a bounded per-supplement wait/abort and preserve the fulfilled results.
  - confidence: 0.82
