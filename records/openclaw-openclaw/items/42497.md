## Summary

Keep open: this PR targets a real gateway WS self-contention bug that is not implemented on current main, but the branch is not merge-ready because the proposed local dispatch path bypasses current startup, authorization, timeout, and request-frame contracts and still lacks real behavior proof.

## What This Changes

The PR adds a gateway-local dispatch registry, registers it from server plugin fallback setup, and makes `callGatewayTool` try in-process dispatch for embedded calls before the WebSocket path.

## Best Possible Solution

Rework the patch as an adapter over the current in-process gateway dispatch contract, preserving resolver startup, credentials, caller scopes, timeout, expectFinal, and JSON-frame semantics, then prove a real embedded cron or gateway-tool run no longer times out.

## Reproduction Assessment

Yes for the PR regressions: source inspection shows the new local branch runs before current option, scope, timeout, and frame-normalization handling. The original active-session cron timeout is supported by linked issue logs, but was not live-run in this read-only review.

## Solution Assessment

No. In-process dispatch is a plausible fix direction, but this implementation should be rebased onto the current dispatch contract and preserve existing gateway authorization and transport semantics before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Register local dispatch from the active resolver path:** `src/gateway/server-plugins.ts:44-47`
  - body: Current gateway startup installs fallback context with `setFallbackGatewayContextResolver`, but this PR registers the dispatcher only from `setFallbackGatewayContext`. After rebasing, production startup would not register the local dispatcher, so embedded calls would keep using WebSocket.
  - confidence: 0.9
- **[P1] Preserve caller scopes before local dispatch:** `src/agents/tools/gateway.ts:154-160`
  - body: The local branch returns before `extra.scopes` or `resolveLeastPrivilegeOperatorScopesForMethod(method, params)` can run. Embedded calls that rely on approval, pairing, admin, or dynamic plugin scopes can execute under fallback/synthetic credentials instead of the caller-specific authorization contract.
  - confidence: 0.9
- **[P1] Honor token and timeout options on the local path:** `src/agents/tools/gateway.ts:154-160`
  - body: Calls with `gatewayToken` or `timeoutMs` but no `gatewayUrl` still take the local path, so explicit credential intent is ignored and stuck in-process handlers lose the bounded timeout behavior provided by the WebSocket path.
  - confidence: 0.88
- **[P2] Normalize params through the gateway frame contract:** `src/agents/tools/gateway.ts:156-158`
  - body: The local path forwards raw params directly instead of the JSON request frame used by WebSocket. Optional fields set to `undefined` can reach validators as present invalid values even though the same call over WebSocket would omit them.
  - confidence: 0.83

## Security Review

  - body: The early local return can execute a gateway method before resolving explicit tokens, explicit scope overrides, or least-privilege scopes, so embedded calls may run under ambient/fallback credentials rather than the caller's intended authorization contract.
