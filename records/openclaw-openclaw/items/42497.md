## Summary

Keep open: the PR targets a real gateway self-contention bug and current main still routes agent gateway tools through WebSocket, but this branch is conflicting, lacks real behavior proof, and introduces auth, timeout, startup, and frame-contract regressions.

## What This Changes

The PR adds a local in-process gateway dispatch registry and makes agent gateway tool calls try it before falling back to WebSocket.

## Best Possible Solution

Rebase and rework the in-process adapter on the current dispatch contract while preserving option resolution, scopes, timeout, expectFinal, and JSON-frame semantics, then prove a real embedded cron or gateway tool call no longer times out.

## Reproduction Assessment

Yes for the PR regressions: source inspection shows the new local branch runs before current option, scope, timeout, and frame-normalization handling. The original embedded cron WebSocket timeout was supported by related issue logs but was not live-run in this read-only review.

## Solution Assessment

No. In-process dispatch is a plausible fix direction, but this implementation is not the best mergeable version because it bypasses established gateway transport and authorization contracts and is wired to an outdated startup path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Register local dispatch on the resolver startup path:** `src/gateway/server-plugins.ts:44-47`
  - body: Current main installs fallback gateway context through setFallbackGatewayContextResolver, but this PR registers the local dispatcher only from setFallbackGatewayContext. After a rebase, production startup would not register the dispatcher, so the intended embedded-tool fix would not run.
  - confidence: 0.91
- **[P1] Preserve caller scopes before local dispatch:** `src/agents/tools/gateway.ts:154-160`
  - body: The local branch returns before extra.scopes or resolveLeastPrivilegeOperatorScopesForMethod(method, params) are applied. Embedded calls that rely on admin, pairing, approvals, or dynamic plugin scopes can run with ambient/fallback scopes instead of the caller-specific authorization contract.
  - confidence: 0.9
- **[P1] Honor token and timeout options on the local path:** `src/agents/tools/gateway.ts:154-160`
  - body: Calls with gatewayToken or timeoutMs but no gatewayUrl still take the local path, so explicit credential intent is ignored and stuck in-process handlers lose the bounded timeout behavior provided by the WebSocket path.
  - confidence: 0.88
- **[P2] Normalize params through the gateway frame contract:** `src/agents/tools/gateway.ts:156-158`
  - body: The local path forwards raw params directly instead of the JSON request frame used by WebSocket. Optional properties set to undefined can reach validators as present invalid values even though the same call over WebSocket would omit them.
  - confidence: 0.83

## Security Review

Status: needs_attention

Summary: The diff introduces a local gateway control-plane path before preserving the current credential, scope, timeout, and frame-normalization contracts.

Concerns:

- **[high] Local dispatch bypasses caller authorization intent:** `src/agents/tools/gateway.ts:154`
  - body: The early local return can execute a gateway method before resolving explicit tokens, explicit scope overrides, or least-privilege scopes, so embedded calls may run under ambient/fallback credentials rather than the caller's intended authorization contract.
  - confidence: 0.88
- **[medium] Local dispatch skips bounded request timeout:** `src/agents/tools/gateway.ts:154`
  - body: Because timeoutMs is not applied on the local path, a stuck in-process handler can block an active agent turn instead of failing under the timeout behavior used by the WebSocket request path.
  - confidence: 0.84
