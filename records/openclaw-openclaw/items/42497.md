## Summary

Keep open: the linked gateway self-contention bug appears real and this PR is still the canonical open fix path, but the patch is not mergeable as-is because it bypasses gateway auth/scope/timeout/request-frame semantics, registers on an obsolete startup path, lacks real behavior proof, and must be reconciled with current main.

## What This Changes

The PR adds a local gateway dispatch registry, registers it from gateway fallback setup, and makes agent `callGatewayTool` prefer in-process dispatch before falling back to WebSocket.

## Best Possible Solution

Rebase and rework the local dispatch adapter over the current in-process gateway contract: register from the active resolver path, resolve options and scopes first, pass timeout/expectFinal/synthetic scopes, JSON-normalize params, and prove an embedded cron run no longer times out.

## Reproduction Assessment

Yes for the PR regressions by source inspection: the local branch runs before current option/scope/timeout handling and registers on a stale startup path. The original cron timeout has concrete linked reports and logs, but I did not live-run it in this read-only review.

## Solution Assessment

No. In-process dispatch is a plausible fix direction, but this implementation needs to reuse the current gateway dispatch contract instead of bypassing caller authorization and transport semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Register local dispatch from the active resolver path:** `src/gateway/server-plugins.ts:49`
  - body: Current gateway startup installs fallback context with `setFallbackGatewayContextResolver`, but this patch registers the dispatcher only from `setFallbackGatewayContext`. After rebasing, production startup would not install the local dispatcher, so embedded calls would keep using WebSocket.
  - confidence: 0.9
- **[P1] Preserve gateway scopes on local dispatch:** `src/agents/tools/gateway.ts:154-160`
  - body: The local branch returns before explicit `extra.scopes` or least-privilege scopes are resolved. In-process dispatch can then run under ambient or synthetic credentials rather than the same authorization contract as the WebSocket path.
  - confidence: 0.91
- **[P1] Honor token and timeout options locally:** `src/agents/tools/gateway.ts:154-160`
  - body: Calls with `gatewayToken` or `timeoutMs` but no `gatewayUrl` still take the local path. That silently ignores explicit credential intent and removes the bounded timeout behavior that protects gateway tool calls on the WebSocket path.
  - confidence: 0.88
- **[P2] Normalize params through the request frame contract:** `src/agents/tools/gateway.ts:157`
  - body: The local path forwards raw params directly instead of the JSON request frame used by WebSocket. Optional fields set to `undefined` can reach validators as present invalid values even though the same call over WebSocket would omit them.
  - confidence: 0.83

## Security Review

Status: needs_attention

Summary: The diff introduces a local gateway control-plane path before preserving current caller authorization, credential, scope, and timeout semantics.

Concerns:

- **[high] Local dispatch bypasses caller authorization intent:** `src/agents/tools/gateway.ts:154`
  - body: The early local return can execute a gateway method before resolving explicit tokens, explicit scope overrides, or least-privilege scopes, so embedded calls may run under ambient or synthetic credentials rather than the caller's intended authorization contract.
  - confidence: 0.89
- **[medium] Local dispatch skips bounded request timeout:** `src/agents/tools/gateway.ts:154`
  - body: Because `timeoutMs` is not applied on the local path, a stuck in-process handler can block an active agent turn instead of failing under the timeout behavior used by the WebSocket request path.
  - confidence: 0.84
