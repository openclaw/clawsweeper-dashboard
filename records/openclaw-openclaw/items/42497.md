## Summary

Keep open: current main still uses the WebSocket-backed gateway tool path and the linked duplicate tracker was closed in favor of this PR, but the branch cannot merge as-is because the local dispatch path is stale against current gateway startup and bypasses important auth, timeout, scope, and JSON-frame contracts.

## What This Changes

The PR adds an in-process gateway dispatch registry, registers `dispatchGatewayMethod` during fallback context setup, and makes `callGatewayTool` prefer local dispatch for embedded gateway-tool calls with focused unit coverage.

## Best Possible Solution

Rebase and rework the local-dispatch adapter so it registers on the current resolver startup path while preserving effective scopes, explicit credential intent, bounded timeouts, JSON-frame normalization, and final-response behavior, then prove the embedded cron/gateway flow in a real run.

## Reproduction Assessment

Yes for the PR regressions by source inspection: the PR head calls local dispatch before the current option, scope, timeout, and serialization path, and current main starts through the resolver hook the PR does not register on. I did not live-run the original embedded cron timeout scenario.

## Solution Assessment

No. In-process dispatch is a plausible fix direction for the linked bug, but this implementation needs to preserve the established gateway transport and authorization contract before it is mergeable.

## Security Review
