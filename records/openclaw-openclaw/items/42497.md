## Summary

Keep open. Current main still lacks the requested local dispatch path, so this remains the canonical implementation candidate, but the branch is stale against newer gateway dispatch/auth contracts, has blocking correctness and security findings, and still lacks contributor real behavior proof.

## What This Changes

The PR adds a local gateway-dispatch registry, registers it during gateway fallback context setup, and makes agent gateway tool calls prefer that in-process dispatcher for embedded calls.

## Best Possible Solution

Rebase and rework the local dispatch adapter over the current in-process gateway contract, preserving auth, token, timeout, expectFinal, scopes, and request-frame normalization, then prove an embedded cron or gateway-tool run no longer times out.

## Reproduction Assessment

Yes for the PR regressions by source inspection: the local branch runs before current auth/scope/timeout handling and registers on an older startup path. I did not live-run the original embedded cron timeout in this read-only review.

## Solution Assessment

No. In-process dispatch is a plausible fix direction, but this implementation needs to reuse the current gateway dispatch contract instead of bypassing caller authorization and transport semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Register local dispatch from the active resolver path:** `src/gateway/server-plugins.ts:49`
  - body: Current gateway startup installs fallback context with `setFallbackGatewayContextResolver`, but this patch registers the dispatcher only from `setFallbackGatewayContext`. After rebasing, production startup would not install the local dispatcher, so embedded calls would keep using WebSocket.
  - confidence: 0.9
- **[P1] Preserve caller scopes before local dispatch:** `src/agents/tools/gateway.ts:154-160`
  - body: The local branch returns before current `extra.scopes` and least-privilege scope resolution. In-process dispatch can therefore run with ambient or synthetic client scopes instead of the same authorization contract as the WebSocket path.
  - confidence: 0.91
- **[P1] Honor token and timeout options locally:** `src/agents/tools/gateway.ts:154-160`
  - body: Calls with `gatewayToken` or `timeoutMs` but no `gatewayUrl` still take the local path. That silently ignores explicit credential intent and removes the bounded timeout behavior that currently protects gateway tool calls.
  - confidence: 0.88
- **[P2] Normalize params through the request frame contract:** `src/agents/tools/gateway.ts:157`
  - body: The local path forwards raw params directly instead of the JSON request frame used by WebSocket. Optional properties set to `undefined` can reach validators as present invalid values even though the same call over WebSocket would omit them.
  - confidence: 0.83

## Security Review

Status: needs_attention

Summary: The diff introduces a local gateway control-plane path before preserving current caller authorization, credential, scope, and timeout semantics.

Concerns:

- **[high] Local dispatch bypasses caller authorization intent:** `src/agents/tools/gateway.ts:154`
  - body: The early local return can execute a gateway method before resolving explicit tokens, explicit scope overrides, or least-privilege scopes, so embedded calls may run under ambient or synthetic credentials rather than the caller’s intended authorization contract.
  - confidence: 0.89
- **[medium] Local dispatch skips bounded request timeout:** `src/agents/tools/gateway.ts:154`
  - body: Because `timeoutMs` is not applied on the local path, a stuck in-process handler can block an active agent turn instead of failing under the timeout behavior used by the WebSocket request path.
  - confidence: 0.84
