## Summary

Keep open: this targets a real embedded gateway-tool timeout, but the branch is conflicting, lacks real behavior proof, and current main's gateway auth/context contracts make the added local branch unsafe as written.

## What This Changes

The PR adds a process-global local gateway dispatcher, registers it from server-plugin startup context, and makes agent gateway-tool calls use that in-process path before falling back to WebSocket.

## Best Possible Solution

Rebase and replace the registry shortcut with a current-main in-process dispatch path that preserves URL/token overrides, approval-runtime token handling, least-privilege or explicit scopes, expectFinal, timeoutMs, request normalization, and resolver-based startup registration.

## Reproduction Assessment

Yes for the PR regressions by source inspection: the added local branch runs before current gateway option/scope/timeout handling, and current startup uses setFallbackGatewayContextResolver. I did not live-run the original embedded cron timeout in this read-only review.

## Solution Assessment

No. The maintainable fix should reuse the current dispatchGatewayMethodInProcess option surface and current resolver startup path instead of adding a method/params-only process-global shortcut.

After rebasing, production startup can miss the registration entirely, leaving the embedded cron/tool calls on the same timeout-prone WS path.
  - confidence: 0.88

  - body: The WS path validates and sends a JSON request frame, which drops object properties whose value is `undefined`; this local branch forwards the original params object directly. Optional fields that succeed over WS can fail schema validation or behave differently in embedded local dispatch.
  - confidence: 0.78
