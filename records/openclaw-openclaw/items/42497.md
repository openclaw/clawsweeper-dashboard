## Summary

Keep open: the fix direction is relevant, and the linked issue was already closed in favor of this canonical PR, but the branch is not mergeable as-is because it misses current gateway startup wiring and changes credential, scope, timeout, and frame-normalization semantics.

## What This Changes

The PR adds a local in-process gateway dispatch registry and makes embedded agent gateway tool calls prefer it before falling back to WebSocket.

## Best Possible Solution

Rebase and rework the local-dispatch adapter on the current gateway startup path while preserving token, scope, timeout, expectFinal, and JSON-frame semantics, then prove a real embedded cron or gateway tool call no longer times out.

## Reproduction Assessment

Yes for the PR regressions by source inspection: the diff enters local dispatch before the current option, scope, timeout, and serialization path, and current main starts through a resolver hook the PR does not register on. I did not live-run the original embedded cron timeout scenario.

## Solution Assessment

No. In-process dispatch is a plausible fix direction, but this implementation is not the narrowest maintainable version because it does not preserve the established gateway transport and authorization contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Register local dispatch from the resolver startup path:** `src/gateway/server-plugins.ts:44-49`
  - body: Current main starts the gateway by installing fallback context through setFallbackGatewayContextResolver, but this PR registers local dispatch only in setFallbackGatewayContext. After rebase, the production startup path would not register the dispatcher, so the central fix would not run.
  - confidence: 0.91
- **[P1] Preserve caller scopes on the local path:** `src/agents/tools/gateway.ts:154-158`
  - body: The local branch returns before extra.scopes or resolveLeastPrivilegeOperatorScopesForMethod(method, params) are applied. Dynamic plugin scopes and explicit approval/pairing scope overrides can therefore authorize differently than the WebSocket path.
  - confidence: 0.9
- **[P1] Honor token and timeout options before local dispatch:** `src/agents/tools/gateway.ts:154-160`
  - body: The guard skips local dispatch for gatewayUrl and expectFinal, but calls with gatewayToken or timeoutMs still take the local path. That silently ignores explicit credential intent and removes the bounded timeout behavior callers rely on for stuck gateway methods.
  - confidence: 0.88
- **[P2] Normalize params through the gateway frame contract:** `src/agents/tools/gateway.ts:155-158`
  - body: The local path forwards params directly instead of the JSON request frame used by the WebSocket path. Optional properties set to undefined can therefore reach validators as present invalid values even though the same call over WebSocket would omit them.
  - confidence: 0.83

## Security Review

Status: needs_attention

Summary: The diff introduces an in-process gateway control-plane path before the current credential, scope, timeout, and frame-normalization contract is preserved.

Concerns:

- **[high] Local dispatch bypasses caller authorization intent:** `src/agents/tools/gateway.ts:154`
  - body: The early local return can execute a gateway method before resolving explicit tokens, explicit scope overrides, or least-privilege scopes, so embedded calls may run under ambient or synthetic credentials rather than the caller's intended authorization contract.
  - confidence: 0.88
- **[medium] Local dispatch skips bounded request timeout:** `src/agents/tools/gateway.ts:154`
  - body: Because timeoutMs is not applied on the local path, a stuck in-process handler can block an active agent turn instead of failing under the timeout behavior used by the WebSocket request path.
  - confidence: 0.84
