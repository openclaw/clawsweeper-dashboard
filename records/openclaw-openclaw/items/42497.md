## What This Changes

The branch adds a local gateway dispatch registry, registers it from gateway startup context, and routes `callGatewayTool` through that in-process path for embedded tool calls with new tests.

Rebase the fix onto current main and implement the embedded gateway fast path through the existing in-process dispatch option surface while preserving URL/token overrides, timeouts, scopes, approval-runtime behavior, and JSON-equivalent parameter normalization.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve timeout and token handling before local dispatch:** `src/agents/tools/gateway.ts:154-160`
  - body: This branch returns from the local path before `resolveGatewayOptions`, but the guard only checks `gatewayUrl` and `expectFinal`. Calls that pass `timeoutMs` or a token-only `gatewayToken` now skip the normal timeout/auth handling, so embedded tools can hang indefinitely or succeed under ambient credentials instead of the supplied token.
  - confidence: 0.9
- **[P2] Normalize params before bypassing WebSocket serialization:** `src/agents/tools/gateway.ts:155-158`
  - body: The local path forwards `params` directly, while the WS path serializes the request and drops `undefined` fields. Existing tools build objects with optional fields set to `undefined`; sending those directly can make TypeBox validators reject calls that currently succeed over WebSocket.
  - confidence: 0.82
- **[P2] Carry in-process dispatch options through the registry:** `src/gateway/local-dispatch.ts:15`
  - body: The new registry type accepts only `method` and `params`, but current main's in-process dispatcher carries options for timeouts, final responses, scoped clients, and synthetic scopes. Keeping this narrow registry forces call sites to drop or bypass those contracts instead of preserving existing gateway semantics.
  - confidence: 0.84
