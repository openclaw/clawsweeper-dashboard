## Summary

Keep open: the PR targets a real gateway WebSocket self-contention bug, but the current branch is not mergeable because its local dispatch path bypasses gateway auth, scope, timeout, and serialization semantics. Current main still routes `callGatewayTool` through the WebSocket client, so the underlying work remains valid but needs redesign.

## What This Changes

The PR adds a gateway local-dispatch registry, registers `dispatchGatewayMethod` during gateway startup, and makes `callGatewayTool` prefer in-process dispatch for embedded non-`expectFinal` calls, with focused tests.

## Best Possible Solution

Rework the fix into an explicit in-process gateway request adapter that first resolves normal gateway options and effective scopes, constructs an equivalent authorized client, applies bounded timeout behavior, preserves JSON-frame parameter normalization, and deliberately handles final-response semantics, with regression tests for token, timeout, scopes, params, cron, node, and session calls.

## Reproduction Assessment

Yes. A focused unit test can register a local dispatcher, call `callGatewayTool` with `gatewayToken`, `timeoutMs`, explicit scopes, or optional `undefined` params, and verify the PR path returns locally before the current WebSocket contract is applied.

## Solution Assessment

No. In-process dispatch is a plausible direction for the self-contention bug, but this implementation is not the best fix because it early-returns around the existing transport/auth contract instead of adapting the local path to match it.

  - body: The local branch returns before `extra.scopes` or `resolveLeastPrivilegeOperatorScopesForMethod` are applied, and the registered dispatcher only receives method/params.   - confidence: 0.9

  - body: The guard only excludes explicit `gatewayUrl` and `expectFinal`, so calls with `gatewayToken` or `timeoutMs` still take the in-process path. That ignores explicit credential intent and removes the bounded request timeout used by existing gateway tool callers.
  - confidence: 0.88
- **[P2] Normalize params before in-process dispatch:** `src/agents/tools/gateway.ts:154-155`
  - body: The local path forwards the live params object, while the WebSocket path sends a JSON frame. Optional properties set to `undefined` can therefore reach gateway validators locally even though they are omitted over WebSocket, making embedded calls fail that currently succeed.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff introduces a new in-process gateway control-plane path without preserving the existing authorization, credential, and timeout contract.

Concerns:

- **[high] Local dispatch bypasses caller authorization intent:** `src/agents/tools/gateway.ts:152`
  - body: The early local return can execute a gateway method before resolving explicit tokens, least-privilege scopes, or explicit scope overrides, so embedded calls may run under ambient or synthetic credentials rather than the caller's intended authorization contract.
  - confidence: 0.88

  - body: Because `timeoutMs` is not applied on the local path, a stuck in-process handler can block an active agent turn instead of failing under the timeout behavior used by the WebSocket request path.
