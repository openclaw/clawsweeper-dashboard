## Summary

Keep open because the underlying active-session gateway timeout still appears unsolved on current main, but this PR is not mergeable as written: it is dirty against main, lacks real behavior proof, and its local shortcut bypasses current gateway auth, scope, timeout, and request-shape semantics.

## What This Changes

Adds a process-global local gateway dispatch registry, registers it during gateway startup, and makes `callGatewayTool` prefer that in-process path for embedded calls with added tests.

## Best Possible Solution

Rebase or replace this branch with a narrow implementation that routes same-process embedded gateway tool calls through current `dispatchGatewayMethodInProcessRaw` while preserving explicit gateway URL/token handling, least-privilege scopes, approval runtime, timeout/`expectFinal`, and JSON-shaped params.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source inspection gives a high-confidence path for the PR regressions and a clear current-main path for cron tools still using `callGateway`, but I did not live-run the original active-session cron timeout in this read-only review.

## Solution Assessment

Is this the best way to solve the issue? No; the fix should use current `dispatchGatewayMethodInProcessRaw` options and preserve the existing WebSocket-path auth, scope, timeout, final-response, and request-normalization semantics.
