## Summary

Keep this PR open: it targets a real gateway/cron timeout bug and current main does not route agent gateway tools through the proposed local path, but the branch is not mergeable as-is because it lacks real behavior proof and introduces gateway auth/transport regressions.

## What This Changes

The PR adds a local in-process gateway dispatch registry and makes agent gateway tool calls try that path before falling back to WebSocket.

## Best Possible Solution

Rebase and rework local dispatch as an adapter over the current in-process dispatch contract, preserving resolver startup, credentials, scopes, timeout, expectFinal, and JSON-frame semantics, then prove a real embedded cron or gateway tool run no longer times out.

## Reproduction Assessment

Yes for the PR regressions: source inspection shows the new local branch runs before current option, scope, timeout, and frame-normalization handling. The original embedded cron timeout is supported by linked issue logs, but was not live-run in this read-only review.

## Solution Assessment

No. In-process dispatch is a plausible fix direction, but this implementation is not the best mergeable version because it bypasses current gateway startup, authorization, timeout, and transport contracts.

Overall confidence: 0.93

Full review comments:

- **[P1] Register local dispatch from the active resolver path:** `src/gateway/server-plugins.ts:44-47`
  - body: Current gateway startup installs fallback context with setFallbackGatewayContextResolver, but this PR registers the dispatcher only from setFallbackGatewayContext. After rebasing, production startup would not register the local dispatcher, so embedded calls would keep using WebSocket.
  - confidence: 0.9
- **[P1] Preserve caller scopes before local dispatch:** `src/agents/tools/gateway.ts:154-160`
  - body: The local branch returns before extra.scopes or resolveLeastPrivilegeOperatorScopesForMethod(method, params) can run. Embedded calls that rely on approval, pairing, admin, or dynamic plugin scopes can execute under the synthetic fallback client instead of the caller-specific authorization contract.
  - confidence: 0.9
- **[P1] Honor token and timeout options on the local path:** `src/agents/tools/gateway.ts:154-160`
  - body: Calls with gatewayToken or timeoutMs but no gatewayUrl still take the local path, so explicit credential intent is ignored and stuck in-process handlers lose the bounded timeout behavior provided by the WebSocket path.
  - confidence: 0.88
- **[P2] Normalize params through the gateway frame contract:** `src/agents/tools/gateway.ts:156-158`
  - body: The local path forwards raw params directly instead of the JSON request frame used by WebSocket. Optional properties set to undefined can reach method validators as present invalid values even though the same call over WebSocket would omit them.
  - confidence: 0.83

## Security Review

Status: needs_attention

Summary: The diff introduces a local gateway control-plane path before preserving current credential, scope, timeout, and request-frame contracts.

Concerns:

- **[high] Local dispatch bypasses caller authorization intent:** `src/agents/tools/gateway.ts:154`
  - body: The early local return can execute a gateway method before resolving explicit tokens, explicit scope overrides, or least-privilege scopes, so embedded calls may run under ambient/fallback credentials rather than the caller's intended authorization contract.
  - confidence: 0.88

  - body: Because timeoutMs is not applied on the local path, a stuck in-process handler can block an active agent turn instead of failing under the timeout behavior used by the WebSocket request path.
