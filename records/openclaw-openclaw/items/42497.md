## Summary

Keep open. The underlying WS self-contention bug is valid and in-process dispatch is plausible, but this PR still introduces a gateway control-plane path that bypasses current auth, scope, timeout, and serialization semantics.

## What This Changes

The PR adds a local gateway-dispatch registry, registers it during gateway startup, and makes embedded agent gateway tool calls prefer in-process dispatch for non-final calls.

## Best Possible Solution

Rework the local path as an explicit in-process gateway request adapter that preserves option resolution, token intent, effective scopes, bounded timeout behavior, JSON-frame parameter normalization, and final-response semantics, with focused regression tests.

## Reproduction Assessment

Yes. A focused unit test can register a local dispatcher and call `callGatewayTool` with `gatewayToken`, `timeoutMs`, explicit `scopes`, or params containing optional `undefined` fields to show the local branch returns before the current WebSocket contract is applied.

## Solution Assessment

No. In-process dispatch is a reasonable direction for #40237, but this implementation is not the best fix because it early-returns around the existing gateway transport and authorization contract instead of adapting the local path to match it.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Pass effective scopes into local dispatch:** `src/agents/tools/gateway.ts:152-158`
  - body: The local branch runs before `extra.scopes` or `resolveLeastPrivilegeOperatorScopesForMethod` are applied. When `dispatchGatewayMethod` falls back to a synthetic client, gateway handlers can see ambient or synthetic scopes instead of the same explicit or least-privilege scopes as the WebSocket path, changing authorization for scoped calls.
  - confidence: 0.9
- **[P1] Honor explicit auth and timeouts locally:** `src/agents/tools/gateway.ts:152-158`
  - body: The guard only excludes explicit `gatewayUrl` and `expectFinal`, so calls that pass `gatewayToken` or `timeoutMs` still use the in-process path. That ignores explicit credential intent and removes the bounded timeout behavior existing callers rely on for stuck gateway methods.
  - confidence: 0.88
- **[P2] Normalize params before local dispatch:** `src/agents/tools/gateway.ts:154-155`
  - body: The local path forwards the live params object instead of the JSON request frame used by the WebSocket path. Optional properties set to `undefined` can therefore reach validators that accept omitted fields but reject present invalid values, making embedded calls fail when the WebSocket call would succeed.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff adds an in-process gateway control-plane route without preserving the existing authorization, credential, and timeout contract.

Concerns:

- **[high] Local dispatch bypasses caller authorization intent:** `src/agents/tools/gateway.ts:152`
  - body: The early local return can execute a gateway method before resolving explicit tokens, explicit scope overrides, or least-privilege scopes, so embedded calls may run under ambient or synthetic credentials rather than the caller's intended authorization contract.
  - confidence: 0.88
- **[medium] Local dispatch skips bounded request timeout:** `src/agents/tools/gateway.ts:152`
  - body: Because `timeoutMs` is not applied on the local path, a stuck in-process handler can block an active agent turn instead of failing under the timeout behavior used by the WebSocket request path.
  - confidence: 0.84
