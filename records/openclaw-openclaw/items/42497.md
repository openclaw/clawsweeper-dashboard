## Summary

Keep open; source review still finds blocking correctness and security regressions, and current main has not implemented the local dispatch fix. The approach can be salvaged only after preserving the gateway auth, scope, timeout, and serialization contracts.

## What This Changes

The PR adds a local gateway dispatch registry, registers it from fallback gateway setup, and makes embedded agent gateway tool calls prefer in-process dispatch when no explicit gatewayUrl or expectFinal is set.

## Best Possible Solution

Rebase and rework the local dispatch adapter so it registers on the current resolver startup path and preserves effective scopes, explicit credential intent, bounded timeouts, JSON-frame normalization, and final-response behavior with focused regression tests.

## Reproduction Assessment

Yes for the PR regressions by source inspection: current main starts with setFallbackGatewayContextResolver, and callGatewayTool currently resolves options, scopes, timeout, and WebSocket serialization before dispatch. I did not live-run the original cron self-contention timeout.

## Solution Assessment

No. In-process dispatch is a reasonable direction for the linked bug, but this implementation misses the production registration path and returns before preserving the established gateway authorization, timeout, and serialization contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Register local dispatch from resolver startup:** `src/gateway/server-plugins.ts:44-47`
  - body: Current main starts the gateway by calling setFallbackGatewayContextResolver, not setFallbackGatewayContext. Registering the dispatcher only from setFallbackGatewayContext means production startup will not enable the local path after rebase, so the linked self-contention fix will not run.
  - confidence: 0.91
- **[P1] Pass effective scopes into local dispatch:** `src/agents/tools/gateway.ts:152-158`
  - body: The local branch returns before extra.scopes or resolveLeastPrivilegeOperatorScopesForMethod(method) are applied. dispatchGatewayMethod then runs with an ambient client or a synthetic write-scoped client, so embedded calls can execute with different authorization than the WebSocket path or fail for admin methods such as cron.add and cron.run.
  - confidence: 0.9
- **[P1] Honor explicit auth and timeout options locally:** `src/agents/tools/gateway.ts:152-158`
  - body: The guard skips local dispatch for gatewayUrl and expectFinal, but calls with gatewayToken or timeoutMs still take the local path. That ignores explicit credential intent and removes the bounded timeout behavior callers rely on for stuck gateway methods.
  - confidence: 0.88
- **[P2] Normalize params before in-process dispatch:** `src/agents/tools/gateway.ts:154-155`
  - body: The local path forwards params directly instead of the JSON request frame used by the WebSocket path. Optional properties set to undefined can therefore reach validators as present invalid values even though the same call over WebSocket would omit them.
  - confidence: 0.83

## Security Review

Status: needs_attention

Summary: The diff introduces an in-process gateway control-plane route that does not preserve the existing authorization, credential, and timeout contract.

Concerns:

- **[high] Local dispatch bypasses caller authorization intent:** `src/agents/tools/gateway.ts:152`
  - body: The early local return can execute a gateway method before resolving explicit tokens, explicit scope overrides, or least-privilege scopes, so embedded calls may run under ambient or synthetic credentials rather than the caller's intended authorization contract.
  - confidence: 0.88
- **[medium] Local dispatch skips bounded request timeout:** `src/agents/tools/gateway.ts:152`
  - body: Because timeoutMs is not applied on the local path, a stuck in-process handler can block an active agent turn instead of failing under the timeout behavior used by the WebSocket request path.
  - confidence: 0.84
