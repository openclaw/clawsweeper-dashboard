## Summary

Keep this PR open as the canonical implementation attempt for the gateway WebSocket self-contention bug, but do not merge it as-is: the branch is stale/conflicting, lacks real behavior proof, and its local dispatch path still bypasses current gateway auth, scope, timeout, startup, and frame-normalization contracts.

## What This Changes

The PR adds a local in-process gateway dispatch registry, registers it from server plugin context setup, and makes agent gateway tool calls try that path before falling back to WebSocket.

## Best Possible Solution

Rebase and rework the in-process dispatch adapter on the current gateway startup path while preserving token, scope, timeout, expectFinal, and JSON-frame semantics, then prove a real embedded cron or gateway tool call no longer times out.

## Reproduction Assessment

For this PR review, yes: the regressions are source-reproducible because the diff enters local dispatch before current option, scope, timeout, and serialization handling, and current main starts through a resolver hook the PR does not register on. The original embedded cron timeout was not live-run in this read-only pass.

## Solution Assessment

No. In-process dispatch is a plausible fix direction, but this implementation is not the best mergeable version because it bypasses established gateway transport and authorization contracts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Register dispatch on the resolver startup path:** `src/gateway/server-plugins.ts:44-47`
  - body: Current main installs the gateway fallback context with `setFallbackGatewayContextResolver`, but this PR registers local dispatch only in `setFallbackGatewayContext`. After rebase, production startup would not register the dispatcher, so the central embedded-tool fix would not run.
  - confidence: 0.91
- **[P1] Preserve caller scopes before local dispatch:** `src/agents/tools/gateway.ts:152-160`
  - body: The local branch returns before `extra.scopes` or `resolveLeastPrivilegeOperatorScopesForMethod(method, params)` are applied. That changes auth-sensitive behavior for embedded calls that rely on admin, pairing, approvals, or dynamic plugin scopes.
  - confidence: 0.9
- **[P1] Honor token and timeout options on the local path:** `src/agents/tools/gateway.ts:152-160`
  - body: Calls with `gatewayToken` or `timeoutMs` but no `gatewayUrl` still take the local path, so explicit credential intent is ignored and stuck in-process handlers lose the bounded timeout behavior of the WebSocket path.
  - confidence: 0.88
- **[P2] Normalize params through the gateway frame contract:** `src/agents/tools/gateway.ts:153-157`
  - body: The local path forwards `params` directly instead of the JSON request frame used by WebSocket. Optional keys set to `undefined` can therefore reach validators as present invalid values even though the same call over WebSocket would omit them.
  - confidence: 0.83

## Security Review

Status: needs_attention

Summary: The diff introduces an in-process gateway control-plane path before the current credential, scope, timeout, and frame-normalization contract is preserved.

Concerns:

- **[high] Local dispatch bypasses caller authorization intent:** `src/agents/tools/gateway.ts:152`
  - body: The early local return can execute a gateway method before resolving explicit tokens, explicit scope overrides, or least-privilege scopes, so embedded calls may run under ambient or synthetic credentials rather than the caller's intended authorization contract.
  - confidence: 0.88
- **[medium] Local dispatch skips bounded request timeout:** `src/agents/tools/gateway.ts:152`
  - body: Because `timeoutMs` is not applied on the local path, a stuck in-process handler can block an active agent turn instead of failing under the timeout behavior used by the WebSocket request path.
  - confidence: 0.84
