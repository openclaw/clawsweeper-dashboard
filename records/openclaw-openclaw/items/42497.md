## Summary

Keep open: this PR is the canonical follow-up for the gateway WebSocket self-contention bug, but the current branch is not mergeable as-is because its local-dispatch path is wired to an outdated startup hook and bypasses current gateway authorization, timeout, and request-frame contracts. The prior discussion is useful, including the linked duplicate issue cleanup and the author's follow-up commits, but real behavior proof is still missing.

## What This Changes

The PR adds a local gateway dispatch registry, registers it from gateway startup, and makes agent gateway tool calls prefer in-process dispatch for embedded calls, with new unit coverage.

## Best Possible Solution

Rebase and rework local dispatch as an adapter over the current in-process dispatch contract, preserving resolver startup, credentials, scopes, timeout, expectFinal, and JSON-frame semantics, then prove a real embedded cron or gateway tool run no longer times out.

## Reproduction Assessment

Yes for the PR regressions: source inspection shows the new local branch runs before current option, scope, timeout, credential, and frame-normalization handling. The original embedded cron WebSocket timeout is supported by linked issue logs but was not live-run in this read-only review.

## Solution Assessment

No. In-process dispatch is a plausible fix direction, but this implementation is not the best mergeable version because it bypasses established gateway transport and authorization contracts and is wired to an outdated startup path.

- **[P1] Register local dispatch from the resolver path:** `src/gateway/server-plugins.ts:44-47`
  - body: Current gateway startup installs fallback context through `setFallbackGatewayContextResolver`, but this PR registers the local dispatcher only from `setFallbackGatewayContext`. After rebasing, production startup would not register the dispatcher, so embedded tool calls would keep using the WebSocket path.
  - confidence: 0.91
- **[P1] Preserve caller scopes before local dispatch:** `src/agents/tools/gateway.ts:154-160`
  - body: The local branch returns before current `extra.scopes` and `resolveLeastPrivilegeOperatorScopesForMethod(method, params)` handling can run. Embedded calls that rely on approvals, pairing, admin, or dynamic plugin scopes can execute under fallback/synthetic credentials rather than the caller-specific authorization contract.
  - confidence: 0.9
- **[P1] Honor token and timeout options on the local path:** `src/agents/tools/gateway.ts:154-160`
  - body: Calls with `gatewayToken` or `timeoutMs` but no `gatewayUrl` still take the local path, so explicit credential intent is ignored and stuck in-process handlers lose the bounded timeout behavior provided by the WebSocket path.
  - confidence: 0.88
- **[P2] Normalize params through the gateway frame contract:** `src/agents/tools/gateway.ts:156-158`
  - body: The local path forwards raw params directly instead of the JSON request frame used by WebSocket. Optional properties set to `undefined` can reach method validators as present invalid values even though the same call over WebSocket would omit them.
  - confidence: 0.83

## Security Review

Status: needs_attention

Summary: Needs attention: the diff introduces a local gateway control-plane path before preserving the current credential, scope, timeout, and frame-normalization contracts.

  - body: The early local return can execute a gateway method before resolving explicit tokens, explicit scope overrides, or least-privilege scopes, so embedded calls may run under ambient/fallback credentials rather than the caller's intended authorization contract.
