## Summary

Keep this PR open because the underlying gateway self-contention bug is still relevant and current main has not implemented the requested in-process agent-tool dispatch, but the branch is not mergeable as-is due to stale gateway integration, auth/scope/timeout contract regressions, and missing real behavior proof.

## What This Changes

The PR adds a local in-process gateway dispatch registry and makes agent gateway tool calls try it before falling back to WebSocket.

## Best Possible Solution

Rebase and rework local dispatch as an adapter over the current in-process dispatch contract, preserving resolver startup, credentials, scopes, timeout, expectFinal, and JSON-frame semantics, then prove a real embedded cron or gateway tool run no longer times out.

## Reproduction Assessment

Yes for the PR regressions: source inspection shows the new local branch runs before current option, explicit-scope, timeout, and frame-normalization handling. The original embedded cron WebSocket timeout is supported by linked issue logs but was not live-run in this read-only review.

## Solution Assessment

No. In-process dispatch is a plausible fix direction, but this implementation is not the best mergeable version because it bypasses established gateway transport and authorization contracts and is wired to an outdated startup path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Register local dispatch from the resolver path:** `src/gateway/server-plugins.ts:44-47`
  - body: Current gateway startup installs fallback context through `setFallbackGatewayContextResolver`, but this PR registers the local dispatcher only from `setFallbackGatewayContext`. After rebasing, production startup would not register the dispatcher, so embedded tool calls would keep using the WebSocket path.
  - confidence: 0.91
- **[P1] Preserve caller scopes before local dispatch:** `src/agents/tools/gateway.ts:154-160`
  - body: The local branch returns before current `extra.scopes` or `resolveLeastPrivilegeOperatorScopesForMethod(method, params)` handling can run. Embedded calls that rely on approvals, pairing, admin, or dynamic plugin scopes can execute under the fallback/synthetic client rather than the caller-specific authorization contract.
  - confidence: 0.9
- **[P1] Honor token and timeout options on the local path:** `src/agents/tools/gateway.ts:154-160`
  - body: Calls with `gatewayToken` or `timeoutMs` but no `gatewayUrl` still take the local path, so explicit credential intent is ignored and stuck in-process handlers lose the bounded timeout behavior provided by the WebSocket path.
  - confidence: 0.88
- **[P2] Normalize params through the gateway frame contract:** `src/agents/tools/gateway.ts:156-158`
  - body: The local path forwards raw params directly instead of the JSON request frame used by WebSocket. Optional properties set to `undefined` can reach method validators as present invalid values even though the same call over WebSocket would omit them.
  - confidence: 0.83

## Security Review

Status: needs_attention

Summary: The diff introduces a local gateway control-plane path before preserving the current credential, scope, timeout, and frame-normalization contracts.

Concerns:

- **[high] Local dispatch bypasses caller authorization intent:** `src/agents/tools/gateway.ts:154`
  - body: The early local return can execute a gateway method before resolving explicit tokens, explicit scope overrides, or least-privilege scopes, so embedded calls may run under ambient/fallback credentials rather than the caller's intended authorization contract.
  - confidence: 0.88
- **[medium] Local dispatch skips bounded request timeout:** `src/agents/tools/gateway.ts:154`
  - body: Because `timeoutMs` is not applied on the local path, a stuck in-process handler can block an active agent turn instead of failing under the timeout behavior used by the WebSocket request path.
  - confidence: 0.84
