## Summary

Keep this PR open, but do not merge it as-is. The in-process dispatch direction matches the reported WS self-contention bug, but the current diff is stale against main's gateway startup path and bypasses existing gateway authorization, timeout, scope, and request-frame semantics.

## What This Changes

The PR adds a local gateway dispatch registry, registers it from gateway fallback setup, and makes embedded agent gateway tool calls prefer in-process dispatch for non-final calls.

## Best Possible Solution

Rebase and rework the local path as a gateway request adapter registered on the current startup resolver path, preserving effective scopes, explicit credential intent, bounded timeouts, JSON-frame normalization, and final-response behavior with focused regression tests.

## Reproduction Assessment

Yes for the PR regressions by source inspection: current startup uses `setFallbackGatewayContextResolver`, and focused tests can register a local dispatcher to show token, timeout, scope, and params behavior diverge from the WebSocket path. I did not live-run the original WS self-contention scenario in this read-only review.

## Solution Assessment

No. In-process dispatch is a reasonable fix direction for #40237, #5703, and #6508, but this implementation misses the current startup hook and returns before the established gateway transport and authorization contract is applied.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Register local dispatch on the resolver startup path:** `src/gateway/server-plugins.ts:44-47`
  - body: Current main starts the gateway by calling `setFallbackGatewayContextResolver(() => gatewayRequestContext)`, not `setFallbackGatewayContext`. Registering the dispatcher only in `setFallbackGatewayContext` means the new in-process path will not be enabled by the production startup path after this PR is rebased, so the central fix will not run.
  - confidence: 0.91
- **[P1] Pass effective scopes into local dispatch:** `src/agents/tools/gateway.ts:152-158`
  - body: The local branch returns before `extra.scopes` or `resolveLeastPrivilegeOperatorScopesForMethod(method)` are applied. `dispatchGatewayMethod` then uses the ambient request client or a synthetic write-scoped client, so calls can run with different authorization than the WebSocket path or fail for admin methods such as `cron.add`/`cron.run`.
  - confidence: 0.9
- **[P1] Honor explicit auth and timeout options locally:** `src/agents/tools/gateway.ts:152-158`
  - body: The guard skips local dispatch for explicit `gatewayUrl` and `expectFinal`, but calls with `gatewayToken` or `timeoutMs` still take the local path. That ignores explicit credential intent and removes the bounded timeout behavior callers rely on for stuck gateway methods.
  - confidence: 0.88
- **[P2] Normalize params before local dispatch:** `src/agents/tools/gateway.ts:154-155`
  - body: The local path forwards `params` directly instead of the JSON request frame used by the WebSocket path. Optional properties set to `undefined` can therefore reach validators as present invalid values even though the same call over WebSocket would omit them.
  - confidence: 0.83

## Security Review

Status: needs_attention

Summary: The diff introduces an in-process gateway control-plane route that does not preserve the existing authorization, credential, and timeout contract.

Concerns:

- **[high] Local dispatch bypasses caller authorization intent:** `src/agents/tools/gateway.ts:152`
  - body: The early local return can execute a gateway method before resolving explicit tokens, explicit scope overrides, or least-privilege scopes, so embedded calls may run under ambient or synthetic credentials rather than the caller's intended authorization contract.
  - confidence: 0.88
- **[medium] Local dispatch skips bounded request timeout:** `src/agents/tools/gateway.ts:152`
  - body: Because `timeoutMs` is not applied on the local path, a stuck in-process handler can block an active agent turn instead of failing under the timeout behavior used by the WebSocket request path.
  - confidence: 0.84
