## Summary

Keep open, but not merge-ready: the underlying active-session gateway timeout still is not fixed on current main, while this stale branch is conflicting, lacks real behavior proof, and introduces auth, scope, timeout, and request-shape regressions.

## What This Changes

The PR adds a gateway-local dispatch registry, registers it during gateway startup, and makes agent gateway tools prefer in-process dispatch for embedded calls with new unit coverage.

## Best Possible Solution

Rebase or replace this with a current-main implementation that uses the existing in-process dispatch contract while preserving auth, scopes, timeout, `expectFinal`, and request-normalization semantics, then prove the active-session cron/tool flow in a real setup.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the PR regressions at source level: the added local branch returns before current auth, scope, timeout, and request-shape handling. I did not live-run the original active-session cron timeout in this read-only review.

## Solution Assessment

Is this the best way to solve the issue? No. The maintainable fix should integrate with current `dispatchGatewayMethodInProcessRaw` semantics instead of adding a parallel registry that skips current gateway contracts.
