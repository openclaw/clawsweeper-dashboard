## Summary

Keep open: the linked gateway self-contention bug remains plausible and this PR is a relevant fix attempt, but the current patch is not mergeable because it misses the current gateway startup path and bypasses auth, scope, timeout, and request-frame semantics. It also lacks after-fix real behavior proof from a real embedded cron or gateway-tool run.

## What This Changes

Adds a local gateway dispatch registry, registers it from gateway plugin context setup, and makes agent gateway tool calls try in-process dispatch before WebSocket for embedded calls.

## Best Possible Solution

Rework the fix as an adapter over the current in-process dispatch contract: resolve options and scopes first, register through the resolver startup path, forward timeout/expectFinal/scope intent, normalize params like the request frame, and prove a real embedded cron run no longer times out.

## Reproduction Assessment

Yes for the PR regressions: source inspection shows the local branch would bypass current option, scope, timeout, and request-frame handling, and registration targets the stale startup path. The original cron timeout has concrete linked issue logs, but I did not live-run it in this read-only review.

## Solution Assessment

No. In-process dispatch is a plausible direction, but this implementation needs to reuse the current gateway dispatch contract instead of bypassing caller authorization and transport semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Register local dispatch from the resolver path:** `src/gateway/server-plugins.ts:44-47`
  - body: Current gateway startup installs fallback context with `setFallbackGatewayContextResolver`, but this registration is added only in `setFallbackGatewayContext`. After rebasing, production startup would not install the local dispatcher, so embedded calls would keep using WebSocket.
  - confidence: 0.9
- **[P1] Pass resolved scopes to in-process dispatch:** `src/agents/tools/gateway.ts:154-160`
  - body: The local branch returns before `extra.scopes` or least-privilege scopes are resolved. `dispatchGatewayMethod` then falls back to its synthetic client scopes, so admin cron actions like `cron.add`/`cron.run` and explicit scoped calls no longer preserve the WebSocket authorization contract.
  - confidence: 0.91
- **[P1] Honor token and timeout options on the local path:** `src/agents/tools/gateway.ts:154-160`
  - body: Calls with `gatewayToken` or `timeoutMs` but no `gatewayUrl` still take the local path. That silently ignores explicit credential intent and removes the bounded timeout behavior that protects gateway tool calls on the WebSocket path.
  - confidence: 0.88
- **[P2] Normalize params through the request-frame contract:** `src/agents/tools/gateway.ts:156-158`
  - body: The local path forwards raw params directly instead of the JSON request frame used by WebSocket. Optional fields set to `undefined` can reach validators as present invalid values even though the same call over WebSocket would omit them.
  - confidence: 0.83

## Security Review

Status: needs_attention

Summary: The diff introduces a local gateway control-plane path before preserving current caller authorization, credential, scope, and timeout semantics.

Concerns:

- **[high] Local dispatch drops caller authorization context:** `src/agents/tools/gateway.ts:154`
  - body: The early local return can execute a gateway method before resolving explicit tokens, explicit scope overrides, or least-privilege scopes, so embedded calls may run under synthetic or ambient credentials rather than the caller's intended authorization contract.
  - confidence: 0.89
- **[medium] Local dispatch skips bounded request timeout:** `src/agents/tools/gateway.ts:154`
  - body: Because `timeoutMs` is not applied on the local path, a stuck in-process handler can block an active agent turn instead of failing under the timeout behavior used by the WebSocket request path.
  - confidence: 0.84
