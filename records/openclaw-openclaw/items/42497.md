## Summary

Keep open: the PR targets a real embedded gateway-tool timeout bug, but the branch is stale against current gateway dispatch/auth contracts, has source-visible P1 regressions around auth, scopes, timeouts, and startup registration, and still lacks real behavior proof.

## What This Changes

The PR adds a process-global local gateway dispatch registry, registers it from fallback gateway setup, and makes `callGatewayTool` prefer in-process dispatch for embedded calls unless `gatewayUrl` or `expectFinal` is set.

## Best Possible Solution

Rebase onto current main and implement a scoped in-process gateway-tool path that preserves gateway options, tokens, scopes, approval runtime identity, timeout, `expectFinal`, and request-frame semantics, then prove embedded cron no longer times out.

## Reproduction Assessment

Yes for the PR regressions by source inspection: the added local branch runs before current auth/scope/timeout handling and registers on an older startup path. I did not live-run the original embedded cron timeout in this read-only review.

## Solution Assessment

No. In-process dispatch is a plausible fix direction, but this implementation should be rebuilt on the current scoped gateway dispatch contract instead of adding a process-global shortcut.
