## Summary

Keep open: current main and the latest release still publish the usage-cost cache lock with a hard link and only handle `EEXIST`, while the open closing PR remains unmerged and conflicting.

## What This Changes

The issue requests a portable usage-cost cache lock fallback when `fs.link` fails on filesystems without hard-link support.

## Best Possible Solution

Land or replace the open linked PR with one canonical fallback that preserves `EEXIST` contention and stale-lock semantics, plus focused regression coverage.

## Reproduction Assessment

Yes at source level: force `fs.promises.link` to reject with `ENOTSUP` and current main rethrows it because only `EEXIST` is handled. I did not run a live SMB/NFS/virtiofs mount repro in this read-only review.

## Solution Assessment

Yes: the narrow maintainable fix is at the usage-cost lock writer, either replacing the hard-link publish with exclusive create or adding a known-errno exclusive-create fallback. The existing linked PR is the current candidate, so this issue should stay open until that path lands or is replaced.
