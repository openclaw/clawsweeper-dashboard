## Summary

Keep open: current main and v2026.6.8 still publish the usage-cost cache refresh lock with a hard link and rethrow unsupported-link errors, while the open closing PR remains unmerged.

## What This Changes

The issue reports that usage-cost cache refresh lock creation uses `fs.promises.link` and fails on SMB/NFS/virtiofs-style filesystems that reject hard links.

## Best Possible Solution

Keep this issue open until https://github.com/openclaw/openclaw/pull/81388 or a narrow replacement lands with the same lock semantics and unsupported-hardlink fallback.

## Reproduction Assessment

Yes. Source inspection shows current main still calls `fs.promises.link` to publish the lock and rethrows non-`EEXIST` errors, so an ENOTSUP from `link` reproduces the reported failure path.

## Solution Assessment

Yes. The best fix is narrow and centralized at `writeUsageCostCacheLockAtomically`, preserving existing `EEXIST` and stale-lock handling while adding a portable unsupported-hardlink fallback.
