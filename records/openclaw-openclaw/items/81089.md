## Summary

Keep open: current main and the latest release still publish the usage-cost cache lock with fs.promises.link and rethrow non-EEXIST errors, while the open linked fix PR remains unmerged and conflicting.

## What This Changes

The issue reports that the session usage-cost cache refresh lock uses a hard link, causing ENOTSUP failures on SMB/NFS/virtiofs-style filesystems without hard-link support.

## Best Possible Solution

Refresh or replace the linked PR, then land one narrow unsupported-hardlink fallback in writeUsageCostCacheLockAtomically that preserves existing EEXIST and stale-lock semantics.

## Reproduction Assessment

Yes at source level. Current main still calls fs.promises.link to publish the lock and rethrows non-EEXIST errors, so an ENOTSUP from link follows the reported failure path; I did not run a live SMB/NFS/virtiofs mount repro.

## Solution Assessment

Yes. The best fix is the narrow helper-level fallback in writeUsageCostCacheLockAtomically, preserving current contention and stale-lock behavior rather than changing the broader cache refresh flow.
