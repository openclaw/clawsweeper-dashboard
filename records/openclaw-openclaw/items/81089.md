## Summary

Keep open: current main and the latest tagged release still publish the usage-cost cache lock with a hard link and rethrow unsupported-link errors, while the live closing PR is still open and should remain the implementation path until it merges or is replaced.

## What This Changes

The issue reports that session usage-cost cache lock acquisition depends on `fs.promises.link`, which fails with `ENOTSUP` on filesystems without hard-link support.

## Best Possible Solution

Review, land, or replace the linked fallback PR so unsupported-hardlink errors use an exclusive-create lock path while existing contention and cleanup behavior remain covered by tests.

## Reproduction Assessment

Yes, source-level: current main calls `fs.promises.link` during usage-cost refresh and only handles `EEXIST`, so an `ENOTSUP` from an unsupported-hardlink mount escapes the refresh path. I did not mount a live SMB/NFS/virtiofs filesystem in this read-only review.

## Solution Assessment

Yes, the narrow maintainable fix is to preserve the existing lock semantics and fall back to exclusive create only for known unsupported-hardlink errors. The existing open PR is the right implementation path to review before closing this issue.
