## Summary

Keep open: current main still publishes the usage-cost cache lock via a hard link and rethrows unsupported-link errors, while the open linked fix PR is unmerged and dirty. This issue should remain the tracker until that PR lands or maintainers replace it with an equivalent narrow fix.

## What This Changes

The issue reports that session usage-cost cache refresh can crash on filesystems where hard links are unsupported because the lock writer uses `fs.promises.link`.

## Best Possible Solution

Review and land or replace the existing linked fix PR so the lock writer uses a portable exclusive-create fallback while preserving the current contention and stale-lock semantics.

## Reproduction Assessment

Yes at source level: force `fs.promises.link` to reject with `ENOTSUP` and current main rethrows it because only `EEXIST` is handled. I did not run a live SMB/NFS/virtiofs mount repro in this read-only review.

## Solution Assessment

Yes, the narrow best fix is to preserve the lock semantics while adding a portable exclusive-create fallback or replacement at the lock writer. The open linked PR is the current implementation candidate, so this issue should stay open until that PR lands or is replaced.
