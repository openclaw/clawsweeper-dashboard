## Summary

Keep open: current main and the latest release still publish the usage-cost cache lock with `fs.promises.link` and rethrow non-`EEXIST` errors, while the linked closing PR remains open and conflicting.

## What This Changes

The issue asks OpenClaw to make session usage-cost cache lock acquisition work on filesystems where hard links are unsupported.

## Best Possible Solution

Review and land or replace https://github.com/openclaw/openclaw/pull/81388 with a narrow lock fallback that preserves the existing contention and stale-lock semantics.

## Reproduction Assessment

Yes, at source level: force `fs.promises.link` in `writeUsageCostCacheLockAtomically` to reject with `ENOTSUP`, and current main rethrows because only `EEXIST` is handled. I did not run a live unsupported-hardlink filesystem repro in this read-only review.

## Solution Assessment

Yes for the current routing: the narrow maintainable solution is to add or replace with an exclusive-create fallback at the lock writer while preserving existing stale-lock behavior. The issue should stay open until the linked PR lands or maintainers choose an equivalent replacement.
