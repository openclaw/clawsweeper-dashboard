## Summary

Keep open: current main still publishes the usage-cost cache refresh lock with `fs.promises.link` and rethrows unsupported-hardlink errors, while the active closing fix remains an open, conflicting PR.

## What This Changes

The issue asks OpenClaw to avoid hard-link-only session usage-cost cache lock creation on filesystems that report `ENOTSUP`.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/81388 with a narrow exclusive-create fallback and regression coverage, then close this issue from the merged fix.

## Reproduction Assessment

Yes, source-level: current main calls `fs.promises.link` during usage-cost cache refresh and only handles `EEXIST`, so `ENOTSUP` from an unsupported-hardlink filesystem escapes. I did not mount an affected filesystem during this read-only review.

## Solution Assessment

Yes, the narrow maintainable fix is to preserve the existing lock behavior while falling back to exclusive create for known unsupported-hardlink errors; the open closing PR is the right path to review before closing this issue.
