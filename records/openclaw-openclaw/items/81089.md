## Summary

Keep open: current main and the latest release still publish the usage-cost cache lock with `fs.promises.link` and rethrow non-`EEXIST` errors, while the closing implementation PR is still open and conflicting.

## What This Changes

The issue requests a portable session usage-cost cache lock when hard links fail with `ENOTSUP` or related errors on SMB/NFS/virtiofs-style filesystems.

## Best Possible Solution

Land or replace the linked fix with a narrow portable lock writer fallback that preserves the existing `EEXIST` contention and stale-lock semantics, with focused regression coverage and acceptable unsupported-filesystem proof.

## Reproduction Assessment

Yes at source level: force `fs.promises.link` to reject with `ENOTSUP` and current main rethrows it because only `EEXIST` is handled. I did not run a live SMB/NFS/virtiofs mount reproduction in this read-only review.

## Solution Assessment

Yes, the narrow maintainable fix is at the lock writer: keep or replace the hard-link path with an exclusive-create fallback while preserving existing lock semantics. The issue should stay open until the linked PR lands or is replaced.
