## Summary

Keep open: current main still publishes the usage-cost cache lock with a hard link and rethrows non-EEXIST errors, so ENOTSUP on filesystems without hard-link support remains a source-proven crash path.

## What This Changes

The issue reports that usage-cost cache lock acquisition depends on hard links and fails with ENOTSUP on SMB/NFS/virtiofs-style filesystems.

## Best Possible Solution

Move usage-cost refresh lock acquisition to a portable exclusive-create path or add an unsupported-hard-link fallback while preserving existing busy, stale-lock, malformed-lock, and token-release semantics.

## Reproduction Assessment

No live unsupported filesystem was mounted, so there is not a high-confidence runtime reproduction from this checkout. The source path is high-confidence: ENOTSUP from fs.promises.link is not handled and will escape the usage-cost cache refresh lock acquisition.

## Solution Assessment

Yes. Replacing the hard-link publish step or adding a fallback for unsupported-link errors is the narrowest maintainable fix; documenting a local-filesystem workaround would leave the crash path in normal usage refresh flows.
