## Summary

Keep open: current main still publishes the usage-cost cache lock with `fs.promises.link` and rethrows non-`EEXIST` errors, while https://github.com/openclaw/openclaw/pull/81388 remains the open closing fix path for this issue.

## What This Changes

The issue requests a portable usage-cost cache lock path when hard-link creation returns `ENOTSUP` on SMB/NFS/virtiofs-style filesystems.

## Best Possible Solution

Land or replace the narrow exclusive-create fallback in `src/infra/session-cost-usage.ts` with regression coverage and real affected-filesystem proof, then let the merged PR close this issue.

## Reproduction Assessment

Yes, source-level: current main calls `fs.promises.link` during usage-cost cache refresh and only handles `EEXIST`, so an unsupported-hardlink `ENOTSUP` escapes the lock acquisition path. I did not mount a live SMB/NFS/virtiofs filesystem in this read-only review.

## Solution Assessment

Yes: preserving the existing lock semantics while falling back to exclusive create for known unsupported-hardlink errors is the narrow maintainable fix. The open linked PR is the right implementation path to review, rebase, land, or replace before closing this issue.
