## Summary

Keep open: current main still publishes the usage-cost cache refresh lock with `fs.promises.link()` and only treats `EEXIST` as contention, so ENOTSUP on filesystems without hard-link support remains a source-reproducible crash path. The issue already has an open linked implementation PR, so cleanup should wait for that PR to be reviewed, landed, or closed.

## What This Changes

The issue asks OpenClaw to stop relying on hard links for the session usage-cost cache refresh lock because `fs.promises.link()` fails with ENOTSUP on SMB/NFS/virtiofs-style filesystems.

## Best Possible Solution

Land a narrow fallback to portable exclusive-create lock publication when hard links are unsupported, with focused regression coverage for the ENOTSUP path and unchanged stale-lock behavior.

## Reproduction Assessment

No live unsupported filesystem was mounted, so there is not a high-confidence runtime reproduction from this checkout. The source path is high-confidence: ENOTSUP from `fs.promises.link` is not handled and will escape lock acquisition.

## Solution Assessment

Yes. Replacing the hard-link publish step or adding an unsupported-link fallback is the narrowest maintainable fix; the open linked PR is the appropriate implementation candidate to review before closing this issue.
