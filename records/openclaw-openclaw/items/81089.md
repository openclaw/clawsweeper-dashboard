## Label Justifications

- `P2`: The issue is a bounded but real filesystem compatibility crash in the session usage-cost cache refresh path.
- `impact:crash-loop`: The reported failure can abort the refresh path with an `ENOTSUP` lock acquisition error.

## Summary

Keep open: current main and the latest release still publish the usage-cost cache lock with `fs.promises.link`, and the open closing PR has not merged yet.

## What This Changes

The issue reports that session usage-cost cache lock acquisition crashes on filesystems that do not support hard links because the lock publish path uses `fs.promises.link`.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/81388 with a narrow exclusive-create fallback and regression coverage, then close this issue from the merged fix.

## Reproduction Assessment

Yes, source-level: current main calls `fs.promises.link` during usage-cost cache refresh and only handles `EEXIST`, so `ENOTSUP` from an unsupported-hardlink mount escapes. I did not run a live SMB/NFS/virtiofs mount in this read-only review.

## Solution Assessment

Yes: preserving the existing lock semantics while falling back to exclusive create for known unsupported-hardlink errors is the narrow maintainable fix. The existing open PR is the right implementation path to review before closing this issue.
