## Summary

Keep open: current main and the latest release still publish the usage-cost cache refresh lock with `fs.promises.link()` and rethrow unsupported-link errors, while the linked closing implementation remains open and unmerged.

## What This Changes

The issue asks OpenClaw to avoid crashing when usage-cost cache lock acquisition hits `ENOTSUP` from hard-link creation on SMB/NFS/virtiofs-style filesystems.

## Best Possible Solution

The usage-cost cache lock writer should preserve current `EEXIST` and stale-lock semantics while using a portable exclusive-create fallback when hard links are unsupported.

## Reproduction Assessment

Yes at source level: current main calls `fs.promises.link` to publish the lock and rethrows any non-`EEXIST` error, so an `ENOTSUP` link rejection reproduces the reported failure. I did not run a live SMB/NFS/virtiofs mount in this read-only pass.

## Solution Assessment

Yes: the narrow maintainable fix belongs at `writeUsageCostCacheLockAtomically`, changing only unsupported hard-link publish behavior while preserving the existing lock contract. The linked PR follows that shape but remains unmerged and conflicting, so this issue should stay open until that path lands or is replaced.
