## Summary

Keep open: current main still hard-links the usage-cost cache lock and throws any non-EEXIST error, so ENOTSUP on filesystems without hard-link support remains a real source-proven crash path.

## What This Changes

The issue requests replacing or falling back from hard-link-based usage-cost cache lock creation so session usage/cost refreshes do not crash on filesystems without hard-link support.

## Best Possible Solution

Move the usage-cost refresh lock to a portable exclusive-create path or unsupported-hard-link fallback while preserving the existing lock payload, stale-lock cleanup, busy result, and release-token checks.

## Reproduction Assessment

No live unsupported filesystem was mounted, so there is not a high-confidence runtime reproduction from this checkout. The source path is high-confidence: ENOTSUP from `fs.promises.link` is not handled and will escape the usage-cost cache refresh lock acquisition.

## Solution Assessment

Yes, replacing the hard-link publish step or adding a fallback on unsupported-link errors is the narrowest maintainable fix. A docs-only workaround would leave a source-proven crash path in a normal usage/cost refresh flow.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/81089. The observable bug is that usage-cost cache refresh lock acquisition publishes `.usage-cost-cache.json.lock` with `fs.promises.link`, so filesystems without hard-link support throw ENOTSUP/EOPNOTSUPP/EPERM instead of producing a lock or a busy result. Keep the scope to the usage-cost cache lock path: preserve the existing lock JSON payload, stale/malformed lock cleanup, EEXIST contention behavior, and release-token safety. Replace `writeUsageCostCacheLockAtomically` with a portable exclusive-create lock or add an unsupported-hard-link fallback; consider the existing fs-safe lock manager only if it preserves the cache-specific semantics. Do not add a config option or make this docs-only. Add focused regression coverage that simulates unsupported hard links and proves `refreshCostUsageCache` or `loadCostUsageSummaryFromCache` no longer throws, while existing busy/stale lock tests still pass. Add a single `CHANGELOG.md` fix entry because this is user-visible crash behavior.
