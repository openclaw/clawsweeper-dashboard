## Summary

Keep open: current `main` and the latest release still publish the usage-cost cache lock with `fs.promises.link`, and the open closing PR has not merged yet.

## What This Changes

The issue reports `ENOTSUP` crashes when the session usage-cost cache lock uses `fs.promises.link` on filesystems without hard-link support.

## Best Possible Solution

Refresh or replace the linked closing PR with a narrow fallback from unsupported hard-link errors to exclusive lock-file creation, preserving the existing contention and stale-lock semantics.

## Reproduction Assessment

Yes, at source level: make `fs.promises.link` reject with `ENOTSUP` and current `main` rethrows it because only `EEXIST` is handled. I did not run a live SMB/NFS/virtiofs mount repro in this read-only review.

## Solution Assessment

Yes, the narrow maintainable fix is at the lock writer: preserve lock semantics and add or replace with a portable exclusive-create path for unsupported hard-link errors. The issue should stay open while https://github.com/openclaw/openclaw/pull/81388 is still open and unmerged.
