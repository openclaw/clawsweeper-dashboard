## Summary

Keep open: current main still publishes the usage-cost cache lock with `fs.promises.link` and rethrows non-`EEXIST` errors, while https://github.com/openclaw/openclaw/pull/81388 is an open closing PR for the same fallback fix. This issue should remain the tracker until that PR or a replacement merges.

## What This Changes

The issue reports that usage-cost cache lock acquisition crashes with `ENOTSUP` on filesystems where hard links are unsupported.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/81388, or a maintainer replacement, with an exclusive-create fallback and regression coverage; then close this issue after the fix is merged.

## Reproduction Assessment

Yes, source-level: current main calls `fs.promises.link` during usage-cost refresh and only handles `EEXIST`, so `ENOTSUP` from an unsupported-hardlink mount escapes the refresh path. I did not mount an affected filesystem in this read-only review.

## Solution Assessment

Yes: preserving the current lock semantics while falling back to exclusive create for known unsupported-hardlink errors is the narrow maintainable fix. The open closing PR is the right implementation path to review before closing this issue.
