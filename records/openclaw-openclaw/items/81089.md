- `impact:crash-loop`: The report is about a lock-acquisition error that can make usage-cost refresh paths fail on SMB/NFS/virtiofs-style mounts.

## Summary

Keep open: current main and v2026.5.18 still publish the usage-cost cache lock via `fs.promises.link` and rethrow unsupported-link errors, while the linked fix PR is open and unmerged.

## What This Changes

The issue requests replacing or falling back from the session usage-cost cache hard-link lock to a portable exclusive-create lock on filesystems without hard-link support.

## Best Possible Solution

Review, update, and land https://github.com/openclaw/openclaw/pull/81388 or an equivalent narrow fallback with regression coverage, then close this issue from the merged fix.

## Reproduction Assessment

Yes, source-level: current main calls `fs.promises.link` during usage-cost cache refresh and only handles `EEXIST`, so an `ENOTSUP` from an unsupported-hardlink mount escapes. I did not mount a live SMB/NFS/virtiofs filesystem in this read-only review.

## Solution Assessment

Yes: preserving the existing lock semantics while falling back to exclusive create for known unsupported-hardlink errors is the narrow maintainable fix. The existing open PR is the right implementation path to review before closing this issue.
