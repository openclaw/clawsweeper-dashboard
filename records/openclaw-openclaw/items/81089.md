## Summary

Keep open: current main and the latest release still publish the usage-cost cache lock with `fs.promises.link` and rethrow unsupported-link errors, while the open closing PR is unmerged. The issue should remain the tracker until that PR lands or maintainers replace it with an equivalent narrow fix.

## What This Changes

The issue asks OpenClaw to avoid crashing when the session usage-cost cache lock uses hard-link publication on filesystems where `link(2)` returns unsupported errors.

## Best Possible Solution

Review and land or replace the existing linked fix PR with one canonical exclusive-create fallback that preserves the current contention and stale-lock semantics.

## Reproduction Assessment

Yes at source level: current main still publishes the lock with `fs.promises.link` and rethrows any non-`EEXIST` error, so an `ENOTSUP` from that call escapes lock acquisition. I did not run a live SMB/NFS/virtiofs mount reproduction in this read-only review.

## Solution Assessment

Yes: the narrow repair is to preserve the current lock contention and stale-lock semantics while adding an exclusive-create fallback or replacement at the lock writer. The open linked PR is the current implementation candidate, so this issue should stay open until that PR lands or is replaced.
