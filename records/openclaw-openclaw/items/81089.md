## Summary

Keep open: current main still publishes the usage-cost cache lock with `fs.promises.link` and only treats `EEXIST` as lock contention, so unsupported-hardlink errors remain a valid source-level bug. The open closing PR is the active implementation path and this issue should stay open until that PR lands or is replaced.

## What This Changes

The issue asks OpenClaw to avoid ENOTSUP failures by not relying solely on hard links for usage-cost cache lock publication on filesystems without hard-link support.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/81388 with a narrow unsupported-hardlink fallback and regression coverage, then close this issue after the fix reaches main.

## Reproduction Assessment

Yes, source-level: current main writes the lock with `fs.promises.link` and only handles `EEXIST`, so `ENOTSUP` from an unsupported-hardlink mount is not converted into normal lock contention. I did not mount a live SMB/NFS/virtiofs filesystem in this read-only review.

## Solution Assessment

Yes: preserving the current lock contract while falling back to exclusive create for known unsupported-hardlink errors is the narrow maintainable fix. The linked open PR is the right implementation path to review, land, or replace.
