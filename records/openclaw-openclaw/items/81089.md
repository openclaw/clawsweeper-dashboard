## What This Changes

The issue reports that the session usage-cost cache refresh lock fails with ENOTSUP on filesystems where hard links are unsupported.

## Best Possible Solution

Land or replace the existing linked fix with a narrow lock-helper fallback that preserves current EEXIST contention and stale-lock cleanup semantics, then close this issue from the merged fix.

## Reproduction Assessment

Yes at source level: make fs.promises.link reject with ENOTSUP in writeUsageCostCacheLockAtomically and current main rethrows because only EEXIST is handled. I did not perform a live unsupported-filesystem mount reproduction in this read-only review.

## Solution Assessment

Yes, the best fix is a narrow fallback or replacement in the existing lock writer, with regression coverage for unsupported-link errno handling and preserved contention behavior. The current issue should stay open until the open linked PR lands or maintainers replace it.
