## Summary

Keep open: current main still hard-links the usage-cost cache lock and rethrows non-EEXIST errors, so ENOTSUP remains a source-proven crash path. Two open PRs already reference this issue with closing syntax, so the issue should stay open until one fix lands or maintainers close the candidate PRs.

## What This Changes

The issue requests portable usage-cost cache refresh locking when hard links are unsupported on SMB/NFS/virtiofs-style filesystems.

## Best Possible Solution

Land one narrow fix that falls back to exclusive create for unsupported hard-link errors while preserving stale-lock cleanup, token-checked release, and EEXIST busy semantics.

## Reproduction Assessment

No high-confidence live reproduction was established because this read-only review did not mount an unsupported filesystem. The source path is high-confidence: `refreshCostUsageCache` reaches a hard-link lock publish, and non-EEXIST errors are rethrown.

## Solution Assessment

Yes. A narrow unsupported-hard-link fallback to exclusive create is the maintainable fix; a docs-only workaround would leave a real crash path in the usage/cost refresh flow.
