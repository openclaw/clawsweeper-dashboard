## Summary

Keep open: current main still publishes the usage-cost cache lock with a hard link and only treats EEXIST as lock contention, so ENOTSUP from filesystems without hard-link support remains a real source-proven crash path.

## What This Changes

The issue asks OpenClaw to make the usage-cost cache refresh lock portable on filesystems where hard links are unsupported.

## Best Possible Solution

Replace the hard-link publish step with portable exclusive-create behavior or an unsupported-hard-link fallback while preserving the existing lock payload, stale-lock cleanup, busy result, and release-token checks.

## Reproduction Assessment

No live unsupported filesystem was mounted in this read-only review, so there is not a high-confidence runtime reproduction from the checkout. The source path is high-confidence: ENOTSUP from fs.promises.link is not handled and will escape usage-cost cache refresh lock acquisition.

## Solution Assessment

Yes. A portable exclusive-create lock path or fallback on unsupported hard-link errors is the narrow maintainable fix; a docs-only workaround would leave a source-proven crash path in normal usage/cost refresh flows.
