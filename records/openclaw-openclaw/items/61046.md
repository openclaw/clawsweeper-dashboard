## Summary

Keep open: the bug is still source-reproducible on current main and the PR is aimed at the right path, but it should not merge as submitted because alias rows can still stay keyed under the default provider and no after-fix real CLI proof is present.

## What This Changes

The PR changes configured model-list resolution for bare model keys and fallback refs to infer a unique provider from configured providers, and adds focused unit tests.

## Best Possible Solution

Land a narrow models-list fix that uses one inferred-provider contract for configured rows, fallback rows, and alias indexing, while preserving display-normalization guards and adding aliased bare-key regression coverage.

## Reproduction Assessment

Yes, by source inspection. Current main sends bare fallback and configured model-list refs through `DEFAULT_PROVIDER`, contrary to the documented alias/unique-provider/default resolution order; I did not run a live command in this read-only review.

## Solution Assessment

No, not as submitted. Inferring the provider is the right direction, but the alias index and configured-row alias lookup need to resolve bare keys under the same effective provider before this is the narrow maintainable fix.
