## Summary

Keep this PR open: current main still has the reported models-list provider drift, but the submitted patch is not merge-ready because it resolves bare configured rows under the inferred provider while alias indexes remain keyed through the default provider.

## What This Changes

The PR updates `resolveConfiguredEntries` to infer a unique configured provider for bare model keys and fallback refs, and adds unit tests for bare keys, bare fallbacks, ambiguous IDs, and explicit provider prefixes.

## Best Possible Solution

Update the models-list configured view so bare-ref provider inference is shared consistently across configured rows, fallback rows, alias indexing, and alias fallback resolution, then add regression coverage for bare configured aliases and alias fallback values before landing this PR or an equivalent maintainer patch.

## Reproduction Assessment

Yes. A config with `models.providers.google.models: [{ id: "gemini-2.0-flash-001" }]` plus a bare configured model key or fallback demonstrates the current-main wrong-provider path by inspection; adding an `alias` to that bare key reproduces the PR-specific alias loss/wrong-provider fallback.

## Solution Assessment

No, not as submitted. Inferring the provider is the right direction, but the patch needs to keep `aliasIndex` and `aliasesByKey` aligned with the same inferred provider rather than only changing row parsing.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Keep aliases keyed to the inferred provider:** `src/commands/models/list.configured.ts:77-80`
  - body: When a bare `agents.defaults.models` key has an `alias`, this loop now parses the configured row with the inferred provider, but `aliasesByKey` was populated from an alias index still built with `DEFAULT_PROVIDER`. The row can show as `google/<model>` while losing its alias, and alias fallbacks can still resolve to `openai/<model>`, preserving the wrong-provider display for alias users.
