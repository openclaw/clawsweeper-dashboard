## Summary

Keep open: the PR targets a real `openclaw models list` providerless-resolution gap, but the submitted diff is conflicting, lacks required real behavior proof, and still leaves alias indexing inconsistent with the inferred provider.

## What This Changes

The PR changes configured model-list resolution so providerless configured model keys and fallback entries infer a provider from configured models before falling back to the default provider.

## Best Possible Solution

Land a rebased narrow fix that applies one providerless-resolution contract to configured rows, fallback rows, and alias indexing while preserving display-normalization guards and adding aliased bare-key regression coverage.

## Reproduction Assessment

Yes, by source inspection. Current main sends configured-list fallback refs and bare configured model keys through `DEFAULT_PROVIDER`, which conflicts with the documented providerless resolution order; I did not run a live command in this read-only review.

## Solution Assessment

No, not as submitted. Inferring the provider is the right direction, but alias indexing and alias-based fallback resolution need the same effective-provider rule before this is the narrow maintainable fix.
