## Summary

Keep open: current main still has the bare-key and bare-fallback default-provider path, so this is a plausible bug fix, but the submitted patch is incomplete because alias indexing remains on the old default-provider resolution and external real behavior proof is missing. The linked issue was closed for a separate runtime-normalization display fix, which does not make this bare-key PR obsolete.

## What This Changes

The PR changes `openclaw models list` configured-entry resolution so bare configured model keys and fallback refs infer a unique provider from `models.providers`, and adds focused unit tests.

## Best Possible Solution

Land a narrow models-list fix that uses one inferred-provider contract for configured rows, fallback rows, and alias indexing, while preserving the display-normalization guard and covering aliased bare keys in regression tests.

## Reproduction Assessment

Yes, by source inspection. Current main passes `DEFAULT_PROVIDER` into bare fallback and configured-key parsing, and the PR-specific alias regression follows from `aliasesByKey` still being copied from an alias index built through `DEFAULT_PROVIDER`.

## Solution Assessment

No, not as submitted. Inferring the provider is the right direction, but the alias index and configured-row alias lookup need to resolve bare keys under the same effective provider before the patch is safe to merge.
