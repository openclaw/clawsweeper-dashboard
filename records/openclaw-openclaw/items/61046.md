## Summary

Keep open: this PR targets a real `openclaw models list` bug that still exists on current `main` and in the latest release, but the submitted patch is incomplete because alias indexing still uses the default provider and the external PR lacks after-fix real behavior proof.

## What This Changes

The PR adds configured-provider inference for bare model keys and fallback entries in `openclaw models list`, plus unit tests for the inferred-provider cases.

## Best Possible Solution

Land a rebased narrow fix that applies providerless resolution consistently to configured rows, fallback rows, and alias indexing, with alias regression coverage and current display-normalization guards preserved.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes, by source inspection: a config with `models.providers.google` containing `gemini-2.0-flash-001` and a bare `agents.defaults.models` key still flows through `DEFAULT_PROVIDER` in current `main`, despite the documented providerless resolution order.

## Solution Assessment

Is this the best way to solve the issue? No, not as submitted. Inferring the provider is the right direction, but the fix also needs to align alias indexing and alias-based fallback resolution with the inferred provider before it is maintainable.
