## Summary

Keep open: current main still has a providerless configured-list gap, but this branch should not merge as-is because alias metadata and alias-based fallback refs remain keyed through the default provider, and the PR still lacks real after-fix CLI proof.

## What This Changes

The PR changes `openclaw models list` configured-entry resolution so bare configured model keys and fallback entries infer a unique provider from configured models before falling back to the default provider.

## Best Possible Solution

Land a narrow models-list fix that applies one provider-inference contract to configured rows, fallback rows, and alias indexing while preserving display-normalization guards and adding aliased bare-key regression coverage.

## Reproduction Assessment

Yes, by source inspection. Current main sends fallback and configured model-list refs through `DEFAULT_PROVIDER`, which conflicts with the documented providerless resolution order; I did not run a live command in this read-only review.

## Solution Assessment

No, not as submitted. Inferring the provider is the right direction, but alias indexing and alias-based fallback resolution need to use the same effective provider before this is the narrow maintainable fix.
