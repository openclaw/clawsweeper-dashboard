## Summary

Keep this PR open: current main still has the models-list provider display bug, and the PR targets the right surface, but the submitted patch has a concrete alias-key regression and still needs the required changelog entry before it should merge.

## What This Changes

The PR changes `resolveConfiguredEntries` so bare configured model keys and fallback refs infer a unique provider from `models.providers`, and adds focused tests for bare keys, bare fallbacks, ambiguity, and explicit provider prefixes.

## Best Possible Solution

Land a narrow models-list fix that uses the same bare-ref provider inference contract for configured rows and fallback rows while keeping alias index keys aligned with the resolved provider. The regression coverage should include bare configured keys with aliases and alias fallback values, and the fix should include a `CHANGELOG.md` entry under the active release fixes section.

## Reproduction Assessment

Yes. By inspection, a config with `models.providers.google.models: [{ id: "gemini-2.0-flash-001" }]` plus a bare `agents.defaults.models` key or fallback still resolves through `DEFAULT_PROVIDER` on current main; the PR-specific regression reproduces when that bare key also has an `alias`.

## Solution Assessment

No, not as submitted. Inferring the provider is the right direction, but the alias index and configured-row alias lookup need to resolve bare keys under the same effective provider, preferably through a small shared helper, and the PR needs changelog coverage.
