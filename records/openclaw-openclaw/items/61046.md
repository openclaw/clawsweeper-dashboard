## Summary

Keep open: this is a plausible bug-fix PR for a source-reproducible models-list display path, but the patch is not ready because alias-backed configured models can still resolve through the default provider and after-fix real behavior proof is missing.

## What This Changes

The PR adds provider inference for bare configured model keys and fallback entries in `openclaw models list`, plus unit tests for provider inference cases.

## Best Possible Solution

Land a rebased narrow fix that applies providerless resolution consistently to configured rows, fallback rows, and alias indexing, with alias regression coverage and current display-normalization guards preserved.

## Reproduction Assessment

Yes, by source inspection. Current main still sends configured-list fallback refs and bare configured-model keys through `DEFAULT_PROVIDER`, while the docs define alias, unique configured-provider, then default-provider resolution order.

## Solution Assessment

No, not as submitted. Inferring the provider is the right direction, but alias indexing and alias-based fallback resolution need the same effective-provider rule before this is the narrow maintainable fix.
