## Summary

Keep open: the linked Gemini display issue was closed after a shipped display-normalization fix, but this PR still targets a distinct providerless model-list resolution gap that current main does not fully cover, and the branch is not merge-ready because alias handling, proof, and merge conflicts remain unresolved.

## What This Changes

The PR changes `openclaw models list` configured-entry resolution so bare model keys and fallback values infer a unique configured provider before falling back to the default provider.

## Best Possible Solution

Replace or refresh this PR with a current-main fix that centralizes providerless configured-model resolution through the shared model-selection helper, keeps alias indexing and fallback resolution aligned, and includes redacted real `openclaw models list` output plus focused regression tests.

## Reproduction Assessment

Yes, source-level: configure a bare `agents.defaults.models` key or fallback under a unique `models.providers.google` model and current main either stays on the default-provider fallback path or lacks alias alignment. I did not run the CLI because this was a read-only review.

## Solution Assessment

No. The direction is useful, but the patch is too narrow because it infers the provider at two model-list call sites while leaving alias indexing and alias fallback resolution on the old default-provider path.
