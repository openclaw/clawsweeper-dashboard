## Summary

Keep open: the PR targets a real current-main models-list bug, but the submitted patch leaves alias indexing on the default-provider path and therefore can still mislabel aliased bare configured keys or alias fallbacks under the wrong provider.

## What This Changes

The branch updates `resolveConfiguredEntries` to infer a unique configured provider for bare model refs in `openclaw models list` and adds focused unit coverage for bare keys, fallbacks, ambiguous matches, and explicit prefixes.

## Best Possible Solution

Use one shared provider-inference and alias-resolution contract for configured rows, fallback rows, and alias indices, with regression coverage for aliased bare keys and alias fallback values.

## Reproduction Assessment

Yes, by source inspection. Current main still parses bare configured keys and fallback values through `DEFAULT_PROVIDER`, while docs require alias lookup and unique configured-provider inference before default fallback.

## Solution Assessment

No, not as submitted. Inferring the provider is the right direction, but the alias index must be built or consulted through the same effective-provider contract so row keys, aliases, and fallback aliases agree.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

- **[P2] Align alias indexing with inferred providers:** `src/commands/models/list.configured.ts:77-80`
  - body: When a bare `agents.defaults.models` key has an `alias`, this loop now parses the configured row with an inferred provider, but `buildModelAliasIndex` was still built with `DEFAULT_PROVIDER`.
