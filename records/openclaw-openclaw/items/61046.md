## Summary

Keep open, but this PR is not merge-ready: it targets a real providerless `openclaw models list` display gap, but the patch leaves alias lookup keyed through the default provider, lacks real CLI proof, and currently conflicts with `main`.

## What This Changes

The PR changes `openclaw models list` configured-entry resolution so bare configured model keys and fallback strings infer a unique provider from `models.providers`, with Vitest coverage for those cases.

## Best Possible Solution

Use one alias-aware configured-model resolver for list entries so bare keys, aliases, configured rows, and fallback entries share the same canonical provider/model key; then rebase and show redacted `openclaw models list` output from a real config.

## Reproduction Assessment

Yes, source-level: configure a bare `agents.defaults.models` key under a unique configured provider with an `alias`, then use that alias as a fallback; the alias index can still resolve through `DEFAULT_PROVIDER` before the PR's inferred default is used. I did not run the CLI because this was a read-only review.

## Solution Assessment

No. The direction is useful, but the patch is not the best fix because it adds provider inference at two call sites while leaving alias indexing and alias fallback resolution on the old provider path.

Overall correctness: patch is incorrect

Full review comments:

- **[P2] Keep alias resolution on the inferred provider:** `src/commands/models/list.configured.ts:49`
  - body: The new `effectiveDefault` moves bare configured rows and fallback strings to an inferred provider, but `aliasIndex` is still built with `DEFAULT_PROVIDER`, and `resolveModelRefFromString()` consumes alias matches before parsing. A bare `agents.defaults.models` key with `alias` can therefore lose its alias on the inferred-provider row, and fallback values that use that alias can still resolve to the default-provider key. Please keep aliases and bare configured rows on the same inferred key.
