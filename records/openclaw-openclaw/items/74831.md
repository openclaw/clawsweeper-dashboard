## Summary

Keep open. Current main does not contain the sharded registry change, but this PR should not merge as-is because the patch removes existing registry ordering guarantees and introduces a concrete path-safety issue in the new storage layout.

## What This Changes

This PR shards sandbox container and browser registries into per-entry JSON files, migrates legacy monolithic files, adds `readRegistryEntry`, and updates Docker/registry tests.

## Best Possible Solution

Keep the current serialized registry behavior until a sharded design preserves path safety, one-shot migration coordination, same-entry mutation ordering, targeted error handling, and regression/performance evidence.

## Reproduction Assessment

Yes. Current main has focused tests for remove/update resurrection, and PR-head inspection shows a legacy `containerName` containing path separators reaches `entryFilePath` unchanged while same-entry update and remove run independently.

## Solution Assessment

No. Per-entry sharding is a plausible performance direction, but this implementation drops existing safety invariants instead of replacing them with encoded filenames plus per-entry or migration coordination.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Encode registry entry filenames:** `src/agents/sandbox/registry.ts:83`
  - body: `containerName` is parsed as a string and then used directly in `path.join`. A malformed legacy entry such as `../x` can make migration, write, or remove target files outside the sharded registry directory, so encode/hash the filename or reject non-basename names before using it as a path.
  - confidence: 0.84
- **[P2] Make legacy migration single-writer:** `src/agents/sandbox/registry.ts:169-174`
  - body: Every public call can run migration. If two callers read the legacy file before either deletes it, the later migration can rewrite stale legacy entries after another caller already updated sharded state; claim or lock the legacy file before fanning it out.
  - confidence: 0.88
- **[P2] Preserve same-entry remove/update ordering:** `src/agents/sandbox/registry.ts:219`
  - body: `updateRegistry` reads an existing per-entry file and later writes it without coordination, while `removeRegistryEntry` independently deletes the same path. A delayed update can resurrect an entry after removal, so keep this guarantee with a per-entry lock, tombstone, compare-and-swap, or equivalent protocol.
  - confidence: 0.9
- **[P3] Propagate unexpected registry removal errors:** `src/agents/sandbox/registry.ts:112-116`
  - body: `force: true` already ignores missing files, so this catch also hides permission and path-type failures that can leave stale registry files while removal reports success. Only suppress the expected missing-file or concurrent-remove case.
  - confidence: 0.83

## Security Review

Status: needs_attention

Summary: Needs attention: the diff introduces a local storage path-safety issue by using raw registry names as shard filenames.

Concerns:

- **[medium] Raw registry names can escape the shard directory:** `src/agents/sandbox/registry.ts:83`
  - body: `entryFilePath` appends an unvalidated registry `containerName` to the shard directory. Legacy registry entries only need `containerName: string` to pass schema validation, so malformed state can traverse out of the registry directory during migration, write, or removal.
  - confidence: 0.84
