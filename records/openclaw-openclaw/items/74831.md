## Summary

Keep open. The branch is useful in direction, but PR head introduces blocking path-safety and concurrency regressions relative to current main's locked registry invariants, so it should not be merged as-is.

## What This Changes

The PR shards sandbox container and browser registries into per-entry JSON files, adds `readRegistryEntry`, migrates legacy monolithic files, and updates Docker lookup/tests.

## Best Possible Solution

Keep the current locked registry until a revised sharded design preserves path safety, one-shot migration ordering, same-entry mutation ordering, narrow delete-error handling, and focused regression tests.

## Reproduction Assessment

Yes. Current main has focused remove/update race tests, and PR head can be inspected directly: `containerName` reaches `path.join` as an unconstrained string, and delayed per-entry update can write after a remove.

## Solution Assessment

No. Per-entry sharding is a reasonable performance direction, but this implementation removes safety invariants instead of replacing them with encoded filenames and a coordinated mutation protocol.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Encode registry entry filenames:** `src/agents/sandbox/registry.ts:83`
  - body: `containerName` is schema-validated only as a string and then used directly in `path.join`. A malformed legacy entry such as `../x` can make migration, write, or remove target files outside the sharded registry directory; encode/hash the filename or reject non-basename names before path use.
  - confidence: 0.84
- **[P2] Make legacy migration single-writer:** `src/agents/sandbox/registry.ts:168-174`
  - body: Every public registry call can run migration. If two callers read the legacy file before either deletes it, the later migration can rewrite stale legacy entries over sharded state that another caller already updated or removed; claim/rename/lock the legacy file before fan-out.
  - confidence: 0.88
- **[P2] Preserve same-entry remove/update ordering:** `src/agents/sandbox/registry.ts:219`
  - body: `updateRegistry` reads an existing entry and later writes it without coordination, while `removeRegistryEntry` independently deletes the same file. A delayed update can resurrect an entry after removal, regressing the current locked behavior and test coverage.
  - confidence: 0.9
- **[P3] Propagate unexpected registry removal errors:** `src/agents/sandbox/registry.ts:111-116`
  - body: `force: true` already ignores missing files, so this catch also hides permission, path-type, or filesystem failures that can leave stale registry files while removal reports success. Only suppress the expected missing/concurrent-remove case.
  - confidence: 0.83
