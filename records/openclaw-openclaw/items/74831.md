## Summary

Keep open. The sharded-registry direction may be useful, but this PR is not mergeable because it removes current serialized mutation guarantees and introduces a path-safety issue in local registry storage.

## What This Changes

This PR replaces monolithic sandbox container/browser registry JSON files with per-entry sharded files, adds `readRegistryEntry`, migrates legacy registry files, and updates the Docker ensure path plus registry tests.

## Best Possible Solution

Keep the current serialized registry behavior until a sharded design preserves the same safety invariants with encoded or hashed filenames, coordinated one-shot migration, per-entry mutation ordering, narrow delete-error handling, regression tests, and concrete performance evidence.

## Reproduction Assessment

Yes. Current main has focused tests for remove/update resurrection, and the PR-head path escape is directly visible by passing a malformed legacy `containerName` containing `..` into `entryFilePath`.

## Solution Assessment

No. Per-entry sharding is a reasonable direction, but this implementation is not the best fix because it drops existing safety guarantees instead of replacing them with a path-safe filename scheme and coordinated mutation protocol.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Encode registry entry filenames:** `src/agents/sandbox/registry.ts:83`
  - body: `containerName` is parsed as a string and then used directly in `path.join`. A malformed legacy entry such as `../x` can make migration, write, or remove target files outside the sharded registry directory. Encode or hash the filename, or reject non-basename names before using it as a path.
  - confidence: 0.84
- **[P2] Make legacy migration single-writer:** `src/agents/sandbox/registry.ts:169-174`
  - body: Every public call can run migration. If two callers read the legacy file before either deletes it, the later migration can rewrite stale legacy entries after another caller already updated sharded state. Claim or rename the legacy file, or otherwise coordinate migration before writing entries.
  - confidence: 0.88
- **[P2] Preserve same-entry remove/update ordering:** `src/agents/sandbox/registry.ts:219`
  - body: `updateRegistry` reads an existing per-entry file and later writes it without coordination, while `removeRegistryEntry` independently deletes the same file. A delayed update can resurrect an entry after removal; keep the current guarantee with a per-entry lock, tombstone, compare-and-swap, or equivalent protocol.
  - confidence: 0.9
- **[P3] Propagate unexpected registry removal errors:** `src/agents/sandbox/registry.ts:112-116`
  - body: `force: true` already ignores missing files, so this catch also hides permission and path-type failures that can leave stale registry files while removal reports success. Only suppress the expected missing-file or concurrent-remove case.
  - confidence: 0.83
