## Summary

Keep open. The sharded registry direction targets a real sandbox hot path, but PR head still has correctness and storage-safety blockers: raw registry names become filesystem paths, legacy migration is multi-writer, and current main's same-entry remove/update ordering guarantee is lost.

## What This Changes

The branch replaces monolithic sandbox container/browser registry JSON files with per-entry files, adds `readRegistryEntry` for Docker container lookup, migrates legacy registry files, and rewrites sandbox registry/config-hash tests.

## Best Possible Solution

Keep the sharded-registry performance direction only after the storage layer uses encoded or hashed filenames, makes legacy migration single-writer or idempotent under concurrent first use, preserves same-entry mutation ordering with a per-entry lock/tombstone/CAS-style protocol, narrows delete-error suppression, and restores focused regression coverage.

## Reproduction Assessment

Yes. Current main has focused tests for concurrent remove/update ordering in `src/agents/sandbox/registry.test.ts`, and PR-head source inspection shows the new per-entry read/merge/write path has no equivalent coordination while the test coverage was rewritten away.

## Solution Assessment

No. Per-entry sharding may be the right performance shape, but this implementation is not the narrowest safe solution until it preserves path safety, migration ordering, and same-entry mutation semantics from current main.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Encode registry entry filenames:** `src/agents/sandbox/registry.ts:83`
  - body: `containerName` is parsed as any string and then passed through `path.join` as a filename. A malformed legacy entry such as `../../openclaw` can make migration, write, or remove operate outside the sharded registry directory, so encode/hash the filename or reject non-basename names before using it as a path.
  - confidence: 0.84
- **[P2] Make legacy migration single-writer:** `src/agents/sandbox/registry.ts:169-174`
  - body: Every public call can run this migration. If two callers read the legacy file before either deletes it, a later migration can rewrite stale legacy entries after another caller already updated a per-entry file, so claim/rename the legacy file or make migration writes single-writer/idempotent before writing shards.
  - confidence: 0.88
- **[P2] Preserve same-entry remove/update ordering:** `src/agents/sandbox/registry.ts:219`
  - body: `updateRegistry` can read an existing per-entry file and later write it after `removeRegistryEntry` deletes that same file, resurrecting stale sandbox state. Current main has tests for this ordering; keep an equivalent guarantee with a per-entry lock, tombstone, compare-and-swap, or similar protocol.
  - confidence: 0.9
- **[P3] Propagate unexpected registry removal errors:** `src/agents/sandbox/registry.ts:112-116`
  - body: This catch hides every `fs.rm` failure. Since `force: true` already ignores missing files, permission and path-type errors can leave stale registry files while callers report success; only suppress the expected missing-file or concurrent-remove case.
  - confidence: 0.83
