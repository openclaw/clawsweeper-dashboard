## Summary

Keep open. The PR is a plausible performance direction, but current head removes main’s same-entry mutation ordering guarantee, suppresses unexpected delete failures, and introduces unvalidated registry-name file paths, so it should not merge as-is.

## What This Changes

The PR shards sandbox container and browser registries from monolithic JSON files into per-entry JSON files, adds single-entry container lookup for Docker ensure, migrates legacy registries, and updates sandbox registry tests.

## Best Possible Solution

Keep the sharded-registry direction only after preserving same-name remove/update ordering with a per-entry lock, tombstone, compare-and-swap, or equivalent protocol; validate or encode entry filenames so names cannot escape the shard directories; propagate non-ENOENT deletion failures; restore focused race coverage; and add the required changelog entry if this remains a user-facing perf change.

## Reproduction Assessment

Yes for the blocking PR regression. Current main has focused tests that delay a registry update, race a removal, and assert the entry stays deleted; the PR removes that coverage and the new write path can recreate the file after removal.

## Solution Assessment

No. Per-entry sharding may be the right performance approach, but this implementation is not the safest maintainable fix until it preserves current ordering, filesystem safety, and error-propagation contracts.
