## Summary

Keep open. Current main still uses the locked monolithic sandbox registry, so the sharded-registry direction is potentially useful, but this PR should not merge as-is because it removes tested same-entry mutation ordering, makes first-run migration racy, suppresses unexpected delete failures, and introduces a filename safety issue in the new per-entry storage.

## What This Changes

The PR replaces monolithic sandbox container/browser registry JSON files with per-entry registry files, adds `readRegistryEntry` for Docker ensure, migrates legacy registry files, and rewrites sandbox registry/config-hash tests.

## Best Possible Solution

Keep the sharded-registry performance direction only after the storage layer uses safe encoded or hashed filenames, makes legacy migration single-writer or otherwise idempotent under concurrent first use, preserves same-entry update/remove ordering with a per-entry lock, tombstone, compare-and-swap, or equivalent protocol, and narrows delete-error suppression to expected missing-file races.

## Reproduction Assessment

Yes for the blocking regressions. Current main has focused registry race tests proving remove/update ordering, and source inspection of PR head shows those tests were removed while the replacement read/merge/write path has no equivalent same-entry coordination.

## Solution Assessment

No. Per-entry sharding may be a maintainable performance fix, but this implementation is not the safest way to solve it until filename safety, migration concurrency, same-entry mutation ordering, and deletion error propagation are preserved.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Encode registry entry filenames:** `src/agents/sandbox/registry.ts:83`
  - body: `containerName` is parsed as any string and then used directly in `path.join`. A malformed legacy entry such as `../../openclaw` can make migration/write/remove target files outside the sharded registry directory. Encode or hash the filename, or reject non-basename names before using it as a path.
  - confidence: 0.84
- **[P2] Make legacy migration single-writer:** `src/agents/sandbox/registry.ts:169-174`
  - body: Every public call can run this migration. If two callers read the legacy file before either deletes it, a later migration can rewrite stale legacy entries after another caller already updated a different per-entry file. Claim/rename the legacy file or make per-entry migration writes single-writer/idempotent before writing sharded entries.
  - confidence: 0.88
- **[P2] Preserve same-entry remove/update ordering:** `src/agents/sandbox/registry.ts:219`
  - body: `updateRegistry` can read an existing per-entry file and later write it after `removeRegistryEntry` deletes that same file, resurrecting stale container or browser state. Current main has tests that lock this behavior down; keep an equivalent guarantee with a per-entry lock, tombstone, compare-and-swap, or similar protocol.
  - confidence: 0.9
- **[P3] Propagate unexpected registry removal errors:** `src/agents/sandbox/registry.ts:112-116`
  - body: This catch hides every `fs.rm` failure. Since `force: true` already ignores missing files, permission and path-type errors can leave stale registry files while recreate/prune reports success. Only suppress the expected missing-file or concurrent-remove case.
  - confidence: 0.83

## Security Review

Status: needs_attention

Summary: The diff introduces a local storage path-safety issue by using raw registry `containerName` values as filesystem path components for sharded entry files.

Concerns:

- **[medium] Raw registry names can escape the shard directory:** `src/agents/sandbox/registry.ts:83`
  - body: `entryFilePath` appends an unvalidated registry `containerName` to the shard directory. Legacy registry entries only need `containerName: string` to pass schema validation, so malformed data can traverse out of the registry directory during migration, write, or removal. Use an encoded/hash filename or strict basename validation.
  - confidence: 0.84
