## Summary

Keep open. The sharded-registry direction is plausible, but the current PR head has merge-blocking storage safety and ordering regressions: raw registry names become filesystem path components, legacy migration can run as multiple writers, and same-entry update/remove ordering is no longer protected.

## What This Changes

The PR shards sandbox container/browser registry storage from monolithic JSON files into per-entry files, adds `readRegistryEntry`, migrates legacy registries, and updates Docker registry lookup/tests.

## Best Possible Solution

Keep the per-entry registry performance direction only after filenames are encoded or hashed, legacy migration is made single-writer or idempotent under concurrency, same-entry update/remove ordering remains protected, and unexpected delete failures are surfaced.

## Reproduction Assessment

Yes. Existing current-main tests reproduce the same-entry remove/update race guarantee, and PR-head source inspection shows that guarantee was removed without an equivalent per-entry coordination mechanism.

## Solution Assessment

No. Sharding may be the right direction, but this implementation is not the narrowest maintainable solution until the storage safety, migration concurrency, and same-entry ordering contracts are preserved.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Encode registry entry filenames:** `src/agents/sandbox/registry.ts:82-83`
  - body: `containerName` is parsed as any string and then used directly in `path.join`. A malformed legacy entry such as `../../openclaw` can make migration/write/remove target files outside the sharded registry directory. Encode or hash filenames, or reject non-basename names before using them as paths.
  - confidence: 0.84
- **[P2] Make legacy migration single-writer:** `src/agents/sandbox/registry.ts:169-174`
  - body: Every public call can run this migration. If two callers read the legacy file before either deletes it, a later migration can rewrite stale legacy entries after another caller already updated a per-entry file. Claim/rename the legacy file or make migration writes single-writer/idempotent before writing shards.
  - confidence: 0.88
- **[P2] Preserve same-entry remove/update ordering:** `src/agents/sandbox/registry.ts:219`
  - body: `updateRegistry` can read an existing per-entry file and later write it after `removeRegistryEntry` deletes that same file, resurrecting stale sandbox state. Current main has tests for this behavior; keep an equivalent guarantee with a per-entry lock, tombstone, compare-and-swap, or similar protocol.
  - confidence: 0.9
- **[P3] Propagate unexpected registry removal errors:** `src/agents/sandbox/registry.ts:110-116`
  - body: This catch hides every `fs.rm` failure. Since `force: true` already ignores missing files, permission and path-type errors can leave stale registry files while recreate/prune reports success. Only suppress the expected missing-file or concurrent-remove case.
  - confidence: 0.83

## Security Review

Status: needs_attention
