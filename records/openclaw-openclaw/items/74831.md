## Summary

Keep open and do not merge as-is. The sharded-registry direction may be useful, but the PR currently removes storage safety guarantees from current main: raw registry names become path components, migration is not single-writer, and same-entry update/remove ordering can resurrect deleted sandbox state.

## What This Changes

The PR shards sandbox container and browser registry state into per-entry JSON files, migrates legacy monolithic registry files, adds `readRegistryEntry`, and updates Docker ensure-path tests.

## Best Possible Solution

Keep the current serialized registry behavior until a sharded design preserves the same safety invariants: encoded or hashed filenames, coordinated one-shot migration, per-entry mutation ordering, narrow delete-error handling, regression tests for those cases, and performance evidence for the lock-contention improvement.

## Reproduction Assessment

Yes. Current main has focused registry tests for remove/update resurrection, and a minimal PR-head test can delay `updateRegistry` while `removeRegistryEntry` deletes the same entry; a malformed legacy entry with `..` in `containerName` exercises the filename escape path by inspection of `entryFilePath`.

## Solution Assessment

No. Per-entry sharding is a plausible performance solution, but this implementation is not the safest form because it drops existing registry mutation guarantees instead of replacing them with an encoded filename scheme plus per-entry or migration-specific coordination.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Encode registry entry filenames:** `src/agents/sandbox/registry.ts:83`
  - body: `containerName` is parsed as a string and then used directly in `path.join`. A malformed legacy entry such as `../x` can make migration, write, or remove target files outside the sharded registry directory. Encode or hash the filename, or reject non-basename names before using it as a path.
  - confidence: 0.84
- **[P2] Make legacy migration single-writer:** `src/agents/sandbox/registry.ts:169-174`
  - body: Every public call can run migration. If two callers read the legacy file before either deletes it, the later migration can rewrite stale legacy entries after another caller already updated sharded state. Claim or rename the legacy file, or otherwise coordinate migration before writing entries.
  - confidence: 0.88
- **[P2] Preserve same-entry remove/update ordering:** `src/agents/sandbox/registry.ts:219`
  - body: `updateRegistry` reads an existing per-entry file and later writes it without coordination, while `removeRegistryEntry` independently deletes the same file. A delayed update can resurrect an entry after removal; keep the current guarantee with a per-entry lock, tombstone, compare-and-swap, or equivalent protocol.
  - confidence: 0.9
- **[P3] Propagate unexpected registry removal errors:** `src/agents/sandbox/registry.ts:112-116`
  - body: `force: true` already ignores missing files, so this catch also hides permission and path-type failures that can leave stale registry files while removal reports success. Only suppress the expected missing-file or concurrent-remove case.
  - confidence: 0.83
