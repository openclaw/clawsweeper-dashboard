## Summary

Keep open. The PR still introduces path-safety and registry ordering regressions while replacing the current lock-backed registry mutation model.

## What This Changes

The PR shards sandbox container and browser registries into per-entry JSON files, adds direct container entry lookup, migrates legacy monolithic files, and updates Docker lookup/tests.

## Best Possible Solution

Keep the current locked registry until a revised sharded design preserves path safety, one-shot migration ordering, same-entry mutation ordering, narrow delete-error handling, and focused regression tests.

## Reproduction Assessment

Yes. The diff gives source-level reproduction paths: migrate or write an entry with a `../` name, and interleave `updateRegistry` after it has read an entry with `removeRegistryEntry` for the same name.

## Solution Assessment

No. Per-entry sharding is a reasonable performance direction, but this patch removes existing safety invariants without replacing them with encoded filenames and coordinated per-entry mutation semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Encode registry entry filenames:** `src/agents/sandbox/registry.ts:83`
  - body: `containerName` is schema-validated only as a string and then passed directly into `path.join`. A malformed legacy entry such as `../x` can make migration, write, or remove target files outside the sharded registry directory; encode/hash the filename or reject non-basename names before path use.
  - confidence: 0.84
- **[P2] Make legacy migration single-writer:** `src/agents/sandbox/registry.ts:168-174`
  - body: Every public registry call can run migration. If two callers read the legacy file before either deletes it, the later migration can rewrite stale legacy entries over sharded state that another caller already updated or removed; claim/rename/lock the legacy file before fan-out.
  - confidence: 0.88
- **[P2] Preserve same-entry remove/update ordering:** `src/agents/sandbox/registry.ts:219`
  - body: `updateRegistry` reads an existing entry and later writes it without coordination, while `removeRegistryEntry` independently deletes the same file. A delayed update can resurrect an entry after removal, regressing the current locked behavior and test coverage.
  - confidence: 0.9
- **[P3] Propagate unexpected registry removal errors:** `src/agents/sandbox/registry.ts:111-116`
  - body: `force: true` already ignores missing files, so this catch also hides permission, path-type, or filesystem failures that can leave stale registry files while removal reports success. Only suppress the expected missing/concurrent-remove case.
  - confidence: 0.83

## Security Review

Status: needs_attention

Summary: The PR introduces a local path-safety concern by using raw registry names as shard filenames.

Concerns:

- **[medium] Raw registry names can escape the shard directory:** `src/agents/sandbox/registry.ts:83`
