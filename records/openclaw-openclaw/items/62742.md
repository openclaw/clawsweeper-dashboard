## Summary

Keep open. Current main still has the unsafe Microsoft Foundry subscription lookup, and the PR’s provider-local guard is functionally correct, but the PR is still an open implementation change and needs the required changelog entry plus normal maintainer CI/merge judgment.

## What This Changes

The PR updates `extensions/microsoft-foundry/auth.ts` to replace the selected subscription `subs.find(...)!` assertion with a checked local match and a descriptive error when no matching subscription is found.

## Best Possible Solution

Add a single Unreleased `### Fixes` changelog bullet crediting the contributor, keep the provider-local guard, and land after targeted Microsoft Foundry tests plus changed-surface CI are green.

## Reproduction Assessment

Yes. A focused unit harness can mock multiple enabled Azure subscriptions and make `ctx.prompter.select` return an id not present in that list; current main then dereferences `undefined.tenantId`. The live “subscription deleted between list and select” explanation is less direct because the displayed options are derived from the same in-memory list.

## Solution Assessment

Yes. The guard is the narrowest maintainable fix for the unsafe boundary and is safer than silently falling back to another subscription, which could select the wrong Azure account context.
