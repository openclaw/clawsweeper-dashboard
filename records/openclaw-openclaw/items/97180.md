## Summary

Keep open: current main and v2026.6.10 still auto-enable and allowlist `codex` during doctor repair for OpenAI routes that resolve to the Codex runtime, while the open migration-plan PR does not cover this doctor-specific plugin mutation boundary.

## What This Changes

The issue asks for `openclaw doctor --fix` to preserve explicit plugin/operator intent instead of silently adding and enabling the `codex` plugin when `openai` is enabled.

## Best Possible Solution

Keep this doctor-specific tracker open and make doctor repair preserve explicit plugin state/allowlists unless maintainers approve a documented, tested OpenAI-through-Codex migration contract.

## Reproduction Assessment

Yes, at source level. Current main has a doctor repair test matching the reported shape and expecting `plugins.entries.codex.enabled = true` plus `plugins.allow += "codex"`; I did not run `doctor --fix` because this review is read-only.

No merged solution is the clear best fix yet. The source path is clear, but the best repair needs maintainer confirmation on whether doctor should auto-enable Codex for official OpenAI routes or warn and preserve explicit plugin state.
