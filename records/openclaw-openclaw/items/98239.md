## Summary

Keep open: the updated field evidence and current source support a real Control UI availability and Gateway exposure-boundary problem, and current main still does not prove the full `/pair qr` trigger is fixed.

## What This Changes

The issue reports that `/pair qr` from Control UI webchat can change `gateway.bind` to `tailnet`, restart Gateway, and break an existing Tailscale Serve webchat route.

## Best Possible Solution

Keep pairing URL generation non-mutating, isolate any path that persists `gateway.bind=tailnet`, and require explicit operator confirmation or a supported managed Serve/public URL path before any disruptive Gateway bind change.

## Reproduction Assessment

No. There is no high-confidence full current-main reproduction for the `/pair qr` trigger yet, but the broken route state is source-reproducible once `gateway.bind=tailnet` is persisted because loopback is no longer served.

## Solution Assessment

Unclear. Preventing silent bind mutation is the right invariant, but maintainers still need to choose the supported behavior for externally managed Tailscale Serve before a fix path is safe.
