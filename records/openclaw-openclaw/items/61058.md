## Summary

Keep open. Current main still lacks a safe env-reference classification path for secrets audit: config audit works from env-substituted values, `.env` audit warns on any non-empty known secret key, and auth-profile refs can still fall through to `PLAINTEXT_FOUND`; related fixes remain open or closed-unmerged.

## What This Changes

Requests that `openclaw secrets audit --check` classify `${VAR_NAME}` and `$VAR_NAME` environment references as safe env references instead of `PLAINTEXT_FOUND`.

## Best Possible Solution

Land one canonical, source-aware secrets-audit fix that compares authored values where needed, distinguishes env refs from plaintext across `openclaw.json`, auth profiles, and `.env`, and documents the exact `$VAR`/`${VAR}` semantics.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: `${OPENAI_API_KEY}` in `openclaw.json` can be resolved before audit uses `snapshot.config`, while bare `$OPENAI_API_KEY` is not a recognized SecretRef template and can still reach plaintext classification.

## Solution Assessment

No. Current main is not the best solution because it still lacks an env-reference classification path; the maintainable path is to consolidate or land the related fixes with focused regression coverage.
