## Summary

Keep open. Current main and the latest release still have the reported audit-classification gap: the secrets audit contract has no safe env-reference finding code, `${VAR}` config values can be substituted before classification, bare `$VAR` is not parsed as a SecretRef/env template, and related fixes are still open rather than shipped.

## What This Changes

The issue requests that `openclaw secrets audit --check` classify `$VAR` and `${VAR}` references in secret-bearing config and env surfaces as safe env references instead of `PLAINTEXT_FOUND`.

## Best Possible Solution

Land one canonical, source-aware secrets-audit fix that distinguishes exact authored env references from real plaintext across config, auth-profile, and `.env` audit paths; document the supported syntax and add focused regression coverage for exact env refs, mixed strings, real plaintext, and unresolved refs.

## Reproduction Assessment

Yes. The source path is high-confidence: with an authored `${OPENAI_API_KEY}` on an audited config target and that env var set, config loading substitutes the value before audit classification; with `$OPENAI_API_KEY`, current parsing leaves it as an ordinary string that can still reach `PLAINTEXT_FOUND`.

## Solution Assessment

No. Current main is not the best solution because the observable false-positive class remains and the audit contract lacks a safe env-reference classification; the safer path is a reviewed consolidation of #64344, #54141, or a focused successor.
