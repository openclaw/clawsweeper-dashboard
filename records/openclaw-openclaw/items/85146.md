## Summary

Keep open: source inspection still supports the reported plugin-install and ACPX wrapper-regeneration path, and current main does not clearly guarantee that an unrelated plugin install preserves a working older Claude ACP adapter or surfaces useful Claude wrapper diagnostics. This needs ACPX/package-policy owner review rather than cleanup closure.

## What This Changes

The issue reports that installing an unrelated plugin can silently rewrite the Claude ACP wrapper from a working `@zed-industries` ACP adapter to the newer `@agentclientprotocol` adapter, causing ACP session spawn failures.

## Best Possible Solution

Define an upgrade-safe ACPX policy where ordinary plugin installs preserve existing ACP adapter/runtime choices unless an explicit update or doctor path changes them, and extend Claude ACP wrapper diagnostics to expose redacted failure details instead of a bare internal error.

## Reproduction Assessment

No high-confidence live reproduction was established because the report depends on Android 10 Termux plus a prior `@zed-industries/claude-agent-acp` setup. Source inspection supports the core path: plugin install runs managed npm convergence and ACPX rewrites the generated Claude wrapper from the currently resolved installed adapter.

## Solution Assessment

Unclear: current main pins the newer Claude ACP package and documents command overrides, but that does not settle whether ordinary plugin installs should preserve an older working adapter. The maintainable fix likely needs an ACPX/package policy decision plus a narrow diagnostics improvement for Claude failures.
