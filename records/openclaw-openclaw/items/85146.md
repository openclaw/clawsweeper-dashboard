## Label Justifications

- `P1`: The report describes a fresh user-facing regression where installing an unrelated plugin can break ACP sessions_spawn for an existing Claude ACP setup.
- `impact:session-state`: The observed failure leaves ACP session creation unusable for the configured Claude agent.
- `impact:auth-provider`: The affected path changes ACP runtime/provider adapter resolution and the generated Claude ACP wrapper target.

## Summary

Keep open: current main still has an ACPX/plugin-install compatibility question here, and the source path supports the reported mechanism even though the Android Termux failure was not live-reproduced.

## What This Changes

The issue asks OpenClaw to stop ordinary plugin installs from silently replacing an existing Claude ACP adapter/wrapper and to surface useful diagnostics when ACP spawn fails.

## Best Possible Solution

Define an upgrade-safe ACPX/plugin-install policy that preserves explicit or existing working ACP adapter choices during ordinary plugin installs, reserves adapter migration for an explicit update/doctor path, and adds redacted Claude ACP failure diagnostics.

## Reproduction Assessment

No high-confidence live reproduction was established for the Android Termux setup, but source inspection supports the core mechanism: plugin install can reconverge managed npm dependencies and ACPX startup rewrites the generated Claude wrapper from the resolved adapter binary.

## Solution Assessment

Unclear: current main pins the newer Claude ACP package and supports explicit per-agent command overrides, but that does not settle whether ordinary plugin installs should preserve older working adapters by default. The maintainable fix needs an ACPX/package policy decision plus a narrow diagnostics improvement.
