## Label Justifications

- `P1`: The original regression made an established ACP spawn workflow unusable after routine plugin activity.
- `impact:session-state`: The reported failure blocked creation and continuation of ACP-backed sessions.
- `impact:auth-provider`: The failure involved ACP adapter dependency selection and runtime routing.

## Summary

The reported failure mode is addressed in the latest stable release: ACPX now owns an exact Claude adapter dependency, resolves its plugin-local executable, preserves explicit adapter commands, and captures redacted launch diagnostics instead of relying on a floating router-skill default.

## What This Changes

Prevent unrelated plugin installation from silently replacing a working Claude ACP adapter and make ACP spawn failures diagnosable.

## Best Possible Solution

Keep the shipped ACPX-owned exact adapter pin, plugin-local executable resolution, configured override support, and redacted wrapper diagnostics as the canonical behavior.

## Reproduction Assessment

Yes at source level: the historical floating adapter selection and generated-wrapper behavior are identifiable, while the released implementation now uses an exact ACPX-owned dependency and plugin-local executable. No live Android/Termux run was needed to establish that the reported core behavior has been replaced.

## Solution Assessment

Yes. Exact dependency ownership and plugin-local execution fix the ownership boundary more reliably than a floating `npx` command; the configuration override shared in the discussion remains a useful workaround for intentionally selecting an older adapter.
