## Summary

The report’s attribution to `openclaw plugins install` is not supported by current source, but the underlying compatibility concern remains valid: ACPX preparation rewrites the generated Claude wrapper to the bundled adapter path, and the available workaround does not establish that existing implicit adapter choices or launch diagnostics are safe. Keep this open for a maintainer decision on adapter ownership and a focused current-main reproduction of the failure output.

## What This Changes

Preserve a working Claude ACP adapter across unrelated plugin operations and report the underlying adapter launch failure instead of a generic internal error.

## Best Possible Solution

Define one canonical ownership rule: preserve explicit user-configured Claude adapter commands, update only the absence/default case to the bundled pinned adapter, and propagate the child process exit cause and stderr through ACP session-spawn diagnostics.

## Reproduction Assessment

Partly yes: current source provides a high-confidence reproduction of wrapper regeneration and command normalization, but not of the reported Android package crash or generic error on current main. A live Termux or equivalent adapter-launch check is still needed for the full issue.

## Solution Assessment

No, the workaround alone is not the best complete solution. Explicit configuration is the right supported control, but OpenClaw should also define which commands are user-owned and preserve actionable process errors when the managed adapter fails.
