## Label Justifications

- `P1`: The report describes a fresh user-facing regression where installing an unrelated plugin can break ACP `sessions_spawn` for an existing setup.
- `impact:session-state`: The failure is observed through `sessions_spawn(runtime="acp")`, leaving ACP session creation unusable for the configured Claude agent.
- `impact:auth-provider`: The affected path changes ACP runtime/provider adapter resolution and the generated Claude ACP wrapper package target.

## Summary

Keep open: the report describes a fresh ACPX/plugin-install regression path with plausible source support, and current main does not clearly guarantee that ordinary plugin installs preserve an existing Claude ACP adapter choice or give useful Claude spawn diagnostics.

## What This Changes

Prevent `openclaw plugins install` from silently changing an existing ACP Claude adapter setup and surface useful ACP spawn diagnostics when it fails.

## Best Possible Solution

Define an upgrade-safe ACPX policy where ordinary plugin installs preserve existing ACP adapter/runtime choices unless an explicit update/doctor path changes them, and extend wrapper diagnostics so Claude ACP failures expose redacted stderr instead of a bare internal error.

## Reproduction Assessment

No high-confidence current-main live reproduction was established because the report depends on Android Termux plus a prior `@zed-industries/claude-agent-acp` setup. Source inspection does support the core mechanism: plugin install runs managed npm convergence and ACPX startup rewrites the generated Claude wrapper from the currently resolved installed adapter.

## Solution Assessment

Unclear: current main pins the newer Claude ACP package exactly and documents command overrides, but that does not settle whether ordinary plugin installs should preserve an older working adapter. The maintainable fix likely needs an ACPX/package policy decision plus a narrow diagnostics improvement for Claude failures.
