## Label Justifications

- `P2`: The report affected plugin install side effects on ACP runtime setup, but the blast radius is limited to managed plugin dependency convergence rather than a core outage.
- `impact:auth-provider`: The issue concerns ACP runtime adapter package selection and whether sessions_spawn can route to the configured ACP backend after plugin installation.

## Summary

Current main and the shipped 2026.5.x line already fix the managed npm-root behavior behind this report: plugin installs now keep existing managed dependency pins/peers stable, and ACPX uses manifest-pinned local adapter packages instead of letting later plugin installs silently float the ACP adapter stack.

## What This Changes

The issue reports that installing an unrelated plugin rewrites the ACP wrapper to a newer ACP package and breaks an existing working ACP runtime setup.

## Best Possible Solution

Keep the shipped managed npm-root pinning and ACPX manifest-pinned adapter behavior; any remaining Termux-only Claude ACP incompatibility should be tracked as a new current-version ACP runtime issue.

## Reproduction Assessment

Yes. The original failure path is source-reproducible as a managed npm-root dependency convergence problem during `openclaw plugins install`; I did not rerun the old 2026.5.7 Termux setup live.

## Solution Assessment

Yes. Fixing the managed npm install boundary is the maintainable solution because it prevents unrelated plugin installs from floating or pruning existing shared-root dependencies instead of adding ACP-specific wrapper exceptions.
