## Summary

Keep open: the PR is useful and implements most of the requested session-envelope surface, but it currently has a concrete enforcement gap for global-scoped sessions because the envelope reader can look in the wrong agent store and silently skip a persisted envelope.

## What This Changes

The PR adds optional session runtime envelopes to Gateway session create/patch schemas, persists them on session entries, enforces tool/path/command constraints in before-tool-call handling, and updates docs, changelog, and focused tests.

## Best Possible Solution

Fix the envelope reader to use the same canonical session-store target semantics as Gateway session create/patch, then add a regression covering global scope with a non-main default agent and templated stores.

## Reproduction Assessment

Yes. A focused reproduction is a config with `session.scope: "global"`, default agent `ops`, and a `{agentId}` session store template: persist an envelope on `global`, then call the new envelope reader or before-tool-call path with `sessionKey: "global"` and observe it reads the `main` store instead of the `ops` store.

## Solution Assessment

No, not as currently patched. The declarative envelope plus before-tool-call enforcement is the right narrow direction, but the store lookup must share Gateway canonical targeting before this can be considered a safe implementation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Use canonical session targeting for envelope reads:** `src/agents/session-runtime-envelope.ts:177-178`
  - body: `sessions.create` and `sessions.patch` persist a `global` session envelope in the configured default agent's store, but this reader derives the store with `resolveAgentIdFromSessionKey("global")`, which falls back to `main`. With `session.scope: "global"`, a default agent like `ops`, and a templated session store, tool execution reads the wrong store, sees no envelope, and allows calls that should be blocked. Resolve the envelope through the same canonical Gateway/session-store targeting logic used by session mutations and add a regression for that config.
  - confidence: 0.88
