## Summary

Keep open: current main still lacks the session envelope surface, and this PR should not merge until the path canonicalization bypass is fixed and reviewed as a new authorization boundary.

## What This Changes

The PR adds an optional persisted `envelope` to Gateway session create/patch RPCs, enforces tool/path/command constraints in the before-tool-call path, updates docs/protocol Swift models/tests, and carries small config/script guardrail changes.

## Best Possible Solution

Make envelope path checks share the same workspace, home, and container path resolution semantics as the executing tools, keep network scope deferred to sandbox/proxy enforcement, then land after focused tests and maintainer/security review.

## Reproduction Assessment

Yes. Current main source shows the Gateway session schemas reject `envelope`, and PR head source shows the added persistence/enforcement paths plus a source-reproducible path-matching bypass.

## Solution Assessment

No, not yet. Persisting and rechecking the envelope through `before_tool_call` is a reasonable core seam, but the matcher must canonicalize paths exactly like the executing tools before it is a safe authorization boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Resolve paths with tool execution semantics before matching envelopes:** `src/agents/session-runtime-envelope.ts:40-45`
  - body: The matcher only applies `path.resolve` to absolute inputs and `path.normalize` to relative inputs, but host file/edit execution expands `~` and resolves relative paths against the workspace root. That lets an equivalent spelling bypass `deniedPaths` or `allowedPaths`, so the new envelope is not a reliable authorization boundary until it uses the same path resolver as the tools that execute the call.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The PR introduces a new tool authorization boundary, and the current path matcher can be bypassed by alternate path spellings.

Concerns:

- **[high] Path deny rules can be bypassed:** `src/agents/session-runtime-envelope.ts:45`
  - body: Envelope path checks do not expand `~` or resolve relative paths with the workspace/root semantics used later by host file/edit execution, so denied paths can still be reached through equivalent input forms.
  - confidence: 0.84
