## Summary

Keep open: current main still lacks the requested session envelope surface, and the PR head appears to implement it without a blocking review finding; the remaining gate is owner review for the new persisted authorization semantics.

## What This Changes

The branch adds an optional Gateway session runtime envelope, stores or clears it on session entries, enforces tool/path/command constraints before tool execution, and updates docs, tests, changelog, scripts, and generated Swift protocol models.

## Best Possible Solution

Land the PR after Gateway/session/tool-authorization owner approval, keeping the declarative envelope additive and leaving network policy to sandbox/proxy follow-up from #74632.

## Reproduction Assessment

Yes. On current main, sending `envelope` to `sessions.create` or `sessions.patch` targets closed schemas that do not define the field; the PR adds focused create, patch, lookup, matcher, and before-tool-call regression coverage.

## Solution Assessment

Yes, with owner-review caveats. Persisting the declarative policy on `SessionEntry` and enforcing it through the existing before-tool-call path is the narrow maintainable core path, while deferring network constraints avoids a misleading hook-level boundary.
