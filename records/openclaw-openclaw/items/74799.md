## Summary

Keep open: the PR is a useful implementation path for the linked session-envelope request, but the current head introduces a concrete path-policy bypass in the new authorization boundary and needs maintainer/security review before merge.

## What This Changes

The PR adds optional per-session runtime envelopes to Gateway session create/patch, persists them on session entries, enforces tool/path/command constraints before tool execution, updates protocol docs/generated Swift models/tests, and includes small config/script guardrail fixes.

## Best Possible Solution

Fix envelope path canonicalization to use the same path forms as the executing tools, keep network constraints deferred to sandbox/proxy enforcement, then land after owner/security review and green focused plus changed-gate validation.

## Reproduction Assessment

Yes. Current main source shows the closed Gateway session schemas reject `envelope`, and PR head source shows the new create/patch/persistence/enforcement paths. The review finding is also source-reproducible from the mismatch between PR path normalization and current file-tool path resolution.

## Solution Assessment

No, not yet. Persisting the envelope and enforcing it through before-tool-call is the right core seam, but the matcher must canonicalize paths exactly like the file tools before it is a safe authorization boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Resolve tool paths before envelope matching:** `src/agents/session-runtime-envelope.ts:40-45`
  - body: The new matcher evaluates the raw path string with `path.resolve`/`path.normalize`, but the host file tools later expand `~` and resolve relative paths against the workspace root. That lets a session with a denied absolute path such as a home `secrets/**` directory still reach it via `~/...`, and lets absolute workspace denies be reached with relative paths. Please canonicalize with the same path semantics as the executing tools before applying allow/deny globs.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The patch adds an authorization boundary, and the current matcher can be bypassed for denied path rules by alternate path spellings.

Concerns:

- **[high] Path deny rules can be bypassed:** `src/agents/session-runtime-envelope.ts:45`
  - body: Envelope path checks do not expand `~` or resolve relative paths the way host Read/Write/Edit execution does, so denied paths can be accessed through equivalent forms that the matcher does not compare against the deny glob.
  - confidence: 0.84
