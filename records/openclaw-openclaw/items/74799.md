## Summary

Keep open. Current main still lacks the requested Gateway session envelope field, and the PR is the active implementation for linked issue #74632, but the latest head still has a blocking security-sensitive path-enforcement gap on hook entry points that do not pass workspace context.

## What This Changes

The PR adds optional per-session runtime envelopes to Gateway session create/patch, persists them on session entries, enforces tool/path/command constraints before tool execution, and updates protocol docs, generated Swift models, changelog, and tests.

## Best Possible Solution

Keep this PR open as the implementation candidate for #74632, fix the remaining workspace-context enforcement gap, then have Gateway/session/tool-authorization owners review the final policy semantics before merge.

## Reproduction Assessment

Yes, by source inspection. Relative path enforcement depends on workspace context, while the native pre_tool_use relay and HTTP tools.invoke path still call the hook without that context.

## Solution Assessment

No. The feature direction is plausible, but the latest PR head is not the best mergeable form while provider-native and HTTP tool calls can evaluate relative paths without the active workspace.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Pass workspace context into envelope checks:** `src/agents/pi-tools.before-tool-call.ts:429-434`
  - body: `evaluateSessionRuntimeEnvelope` resolves relative paths only when `ctx.workspaceDir` is present, but the native `pre_tool_use` relay and HTTP `tools.invoke` paths still call `runBeforeToolCallHook` without that context. A session envelope such as `deniedPaths: ["/repo/secrets/**"]` can therefore miss a relative tool input like `secrets/token.txt` on those surfaces. Please resolve and pass the active agent/session workspace for every hook entry point that can enforce the session envelope.
  - confidence: 0.9
