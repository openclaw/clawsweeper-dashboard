## Summary

Keep open: current main still lacks the requested session envelope surface, and this active PR is the implementation candidate. I found no discrete blocking review finding in the diff, but the change adds a Gateway/session authorization boundary, so the remaining step is maintainer and security-owner review plus changed-gate proof before merge.

## What This Changes

The branch adds an optional per-session runtime envelope to `sessions.create` and `sessions.patch`, persists it on `SessionEntry`, enforces tool/path/command constraints before tool execution, and updates protocol docs, changelog, and focused tests.

## Best Possible Solution

Land the additive session-envelope implementation after Gateway/tool-authorization owner and security review, with focused tests plus changed-gate proof. Keep network constraints and external authorization callbacks as separate follow-up design work rather than adding hook-level URL scanning here.

## Reproduction Assessment

Yes. Current-main schema inspection reproduces the gap: create/patch reject unknown `envelope` fields, `SessionEntry` has no envelope storage, and before-tool-call enforcement has no session-envelope lookup path.

## Solution Assessment

Yes, with merge prerequisites. Persisting a declarative session field and enforcing it through the existing before-tool-call path is a narrow maintainable approach for tool/path/command constraints; deferring network controls to sandbox/proxy enforcement is safer than trying to infer network access from hook parameters.
