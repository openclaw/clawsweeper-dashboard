## Summary

Keep open: the PR is a useful implementation of the per-session envelope request, but it should not merge as-is. The latest head fixed the earlier changelog gap and wires schema, persistence, docs, and before-tool-call checks, but the enforcement helper reads the session store through a narrower agent-id/path lookup than the Gateway session APIs use, which can miss persisted envelopes for global or aliased sessions and leave tool calls unenforced.

## What This Changes

The PR adds an optional `envelope` to `sessions.create` and `sessions.patch`, persists it on session entries, enforces tool/path/command constraints before tool execution, updates protocol exports/docs, adds focused tests, and adds an Unreleased changelog entry.

## Best Possible Solution

Land the additive session-envelope implementation after fixing enforcement lookup to use the same canonical session-store resolution as Gateway create/patch/read paths, then rerun the focused agent/gateway tests and changed gate. Keep network constraints and external authorization callbacks as explicit follow-up work tied to the broader #74632 discussion rather than adding hook-level URL scanning here.

## Reproduction Assessment

Yes. A high-confidence static reproduction is to persist an envelope on a global or alias-resolved session through the Gateway, then run a tool in that canonical session with a per-agent/default-agent store: the PR read helper looks in the store derived from the raw session key instead of the Gateway-selected store, so the envelope can be absent at enforcement time.

## Solution Assessment

No, not quite yet. The overall design of a persisted declarative envelope enforced through before-tool-call is the right narrow core seam, but the current implementation must share Gateway canonical session lookup before it can be relied on as an enforcement boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Resolve envelopes through the canonical session store:** `src/agents/session-runtime-envelope.ts:176-186`
  - body: `readSessionRuntimeEnvelope` derives the store from `resolveAgentIdFromSessionKey(sessionKey)` and then looks up the raw key. That is narrower than the Gateway resolver used by `sessions.create`, `sessions.patch`, and `loadSessionEntry`; for global/default-agent stores or alias-resolved sessions, the envelope can be persisted in the Gateway-selected store while this check reads the main-agent/raw-key location and treats the session as having no envelope. Reuse the canonical Gateway session target/entry resolution or pass the already loaded canonical entry into the hook path so declared envelopes cannot be skipped.
  - confidence: 0.82

## Security Review
