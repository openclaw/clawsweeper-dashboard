## Summary

Keep this PR open: the implementation is directionally plausible, but the current diff leaves a concrete path-policy bypass for tools that default to the workspace when their optional path parameter is omitted.

## What This Changes

The PR adds an optional Gateway session runtime envelope to create/patch RPCs, persists it on session entries, enforces tool/path/command constraints before tool execution, updates generated protocol models/docs/changelog, and adds focused tests.

## Best Possible Solution

Fix the implicit workspace/default-path enforcement gap, then land only if Gateway/session/tool-authorization owners accept the additive envelope semantics and the intentionally deferred network boundary.

## Reproduction Assessment

Yes, by source inspection. Current main rejects envelope fields, and the PR evaluator allows allowedPaths to pass when a path-scoped tool omits its optional path even though upstream grep/find/ls default omitted paths to the workspace/current directory.

## Solution Assessment

No, not yet. The protocol and persistence shape is a reasonable additive approach, but the enforcement layer must evaluate implicit default paths before this is a dependable session policy boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Enforce envelopes on implicit workspace paths:** `src/agents/session-runtime-envelope.ts:305-309`
  - body: When `allowedPaths` is set, this only checks explicit path-like parameters. Tools such as upstream `grep`, `find`, and `ls` make `path` optional and default it to the current workspace, so a session limited to `/repo/src/**` can call `grep` with no `path` and search `/repo`. Treat the implicit workspace/current directory as a checked path for path-scoped tools before allowing the call.
  - confidence: 0.88
