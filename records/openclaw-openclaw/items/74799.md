## What This Changes

The PR adds a per-session runtime envelope to `sessions.create` and `sessions.patch`, persists it on session entries, enforces tool/path/command constraints in the before-tool-call path before execution and after parameter rewrites, and updates protocol docs, changelog, and focused tests.

## Best Possible Solution

Land this additive session-envelope implementation if Gateway/tool-authorization owners agree with the persisted-envelope semantics, while keeping network constraints and any out-of-process authorization callback as explicit follow-up work rather than adding unreliable hook-level URL scanning here.

## Reproduction Assessment

Yes. Current main’s `sessions.create` and `sessions.patch` schemas omit `envelope` under `additionalProperties: false`, and `SessionEntry` has no persisted envelope field, so the feature gap is directly reproducible by sending an `envelope` field to either RPC.

## Solution Assessment

Yes, pending maintainer review. A persisted declarative envelope enforced through the existing before-tool-call path is the narrow core seam for tool/path/command limits, and deferring network policy to sandbox/proxy enforcement is safer than trying to infer network use from hook parameters.
