## Summary

Keep open: current main still lacks the `sessions.create` / `sessions.patch` envelope contract, and this PR appears to implement the requested core path without blocking findings. Because it creates a persisted tool-authorization boundary, Gateway/session/tool-policy owner review remains the right next step before merge.

## What This Changes

The PR adds optional session runtime envelope schemas, persistence, before-tool-call enforcement, docs/changelog coverage, generated Swift protocol models, focused tests, and small validation/bundle guardrail fixes needed by the branch.

## Best Possible Solution

Land this PR after owner approval, keeping the declarative envelope additive and leaving network policy to a sandbox/proxy follow-up tracked from #74632.

## Reproduction Assessment

Yes. On current main, sending `envelope` to `sessions.create` or `sessions.patch` targets closed schemas that do not define the field; the PR adds focused create, patch, lookup, matcher, and before-tool-call regression coverage.

## Solution Assessment

Yes, with owner-review caveats. Persisting the declarative policy on `SessionEntry` and enforcing it through the existing before-tool-call path is the narrow maintainable core path, while deferring network constraints avoids a misleading hook-level boundary.
