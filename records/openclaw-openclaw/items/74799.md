## Summary

Keep open for maintainer review: the PR addresses the current main gap with additive schema, persistence, enforcement, docs, generated models, tests, changelog, and green exact-head checks, but it creates a new tool-authorization boundary that should get Gateway/session/tool-policy owner approval before merge.

## What This Changes

The PR adds an optional Gateway session runtime envelope for `sessions.create` and `sessions.patch`, persists it on session entries, and enforces tool/path/command constraints before tool execution.

## Best Possible Solution

Land this PR after owner approval, keeping the declarative envelope additive and leaving network/callback policy to the existing follow-up discussion in #74632.

## Reproduction Assessment

Yes. On current main, sending `envelope` to `sessions.create` or `sessions.patch` hits closed schemas that do not define the field; the PR adds focused create, patch, lookup, matcher, and before-tool-call regression coverage.

## Solution Assessment

Yes, with owner-review caveats. Persisting the declarative policy on `SessionEntry` and enforcing it through the existing before-tool-call path is the narrow maintainable core path, while deferring network constraints avoids a misleading hook-level boundary.
