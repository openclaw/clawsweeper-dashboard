## Summary

Keep open. Current main partially loads date-prefixed session-memory artifacts for first-turn startup context, but memory-core rem-backfill, live daily ingestion, historical seeding, REM harness, and Gateway Dream Diary backfill still use exact `YYYY-MM-DD.md` matchers, so the reported slugged-file visibility bug remains.

## What This Changes

The issue asks OpenClaw to align session-memory hook filenames with memory-core dreaming/backfill discovery so hook-created `memory/YYYY-MM-DD-*.md` files are not ignored.

## Best Possible Solution

Add one shared dated-memory filename parser that accepts canonical `YYYY-MM-DD.md` and bounded `YYYY-MM-DD-*.md` session-memory artifacts, then reuse it across memory-core discovery, live ingestion, historical seeding, REM harness, and Gateway backfill.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a hook-created file like `memory/2026-04-21-ai-assistant-config.md` matches the writer’s date-plus-slug format but fails every exact `YYYY-MM-DD.md` matcher used by rem-backfill, live daily ingestion, historical seeding, REM harness, and Gateway backfill.

## Solution Assessment

Yes. A reader-side parser fix is the narrowest maintainable solution because it preserves existing hook output and makes already-created slugged artifacts visible; changing only the hook writer or adding a config option would leave historical files or product policy unresolved.
