## Summary

Keep open. Current main still has the reported cross-system mismatch: the bundled session-memory hook writes dated slug files, while memory-core backfill, live ingestion, historical seeding, rem-harness grounded discovery, and Gateway Dream Diary backfill still accept only exact date-only daily files. The shipped startup-context fix helps first-turn prompt loading, but it does not make slugged artifacts participate in dreaming promotion or grounded backfill.

## What This Changes

The issue requests that session-memory hook artifacts and memory-core dreaming/backfill discovery use a compatible daily memory filename convention so `YYYY-MM-DD-slug.md` files are not skipped.

## Best Possible Solution

Introduce a shared dated-memory filename matcher/parser that accepts both canonical `YYYY-MM-DD.md` files and slugged `YYYY-MM-DD-*.md` session artifacts, then reuse it across memory-core CLI backfill/rem-harness discovery, live daily ingestion, historical seeding, and Gateway Dream Diary backfill. Keep the existing writer behavior unless maintainers choose a broader migration, because changing only future writes would still leave existing slugged artifacts invisible.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: `memory/2026-04-21-ai-assistant-config.md` matches the session-memory writer shape but is rejected by the exact date-only regexes used by rem-backfill, rem-harness grounded discovery, daily ingestion, historical seeding, and Gateway Dream Diary backfill.

## Solution Assessment

Yes, a narrow reader-side discovery fix is the best current direction. It is backward-compatible for existing files and avoids turning this bug into a broader memory naming migration or config-design decision.
