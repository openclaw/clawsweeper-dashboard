## Summary

Keep open: the idea is useful, but the current patch wires the rewrite through `afterToolCall`, while the missing-tool path it is trying to improve is still emitted as an immediate result that bypasses that hook in the pinned/current agent-core dependency.

## What This Changes

The PR adds a Pi embedded runner hook and helper that rewrite unknown-tool error results into SKILL.md guidance when the missing tool name matches a visible skill.

## Best Possible Solution

Route the skill hint through the actual missing-tool result construction path, or upgrade to an agent-core version whose documented hook covers immediate missing-tool outcomes, then add focused regression coverage for skill and non-skill unknown names.

## Reproduction Assessment

Yes for the blocking review finding: source inspection shows a nonexistent tool name is handled as an immediate `Tool X not found` result before `afterToolCall` runs. I did not run a live model session.

## Solution Assessment

No. The current hook is not the best way to solve this until immediate missing-tool results pass through the rewrite path or the dependency contract changes to guarantee that behavior.
