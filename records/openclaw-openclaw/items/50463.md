## Summary

Keep open. The UX direction is useful and not implemented on current main, but the branch is stale against the current Pi dependency surface and still has correctness/security blockers around where missing-tool results are rewritten and which skills are visible to the run.

## What This Changes

The branch extracts unknown-tool parsing and adds a Pi embedded runner `afterToolCall` rewrite that turns missing-tool results matching registered skills into `SKILL.md` guidance.

## Best Possible Solution

Implement the hint at the current missing-tool result boundary, preserve the existing public hook behavior, match only run-visible skills, and add focused regression coverage for visible, hidden, non-skill, punctuation, and collision cases.

## Reproduction Assessment

Yes, source-based. Current main has no skill-specific `SKILL.md` hint, and the pinned agent loop creates a generic `Tool X not found` result for missing tool calls.

## Solution Assessment

No. The proposed UX is reasonable, but this branch should move the rewrite to the current missing-tool boundary and use the run-visible skill set instead of a private old hook surface and full persisted snapshot.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Route skill hints through the missing-tool boundary:** `src/agents/pi-embedded-runner/run/attempt.ts:1925-1938`
  - body: The pinned `@earendil-works/pi-agent-core@0.74.0` returns missing tools as immediate results from `prepareToolCall`, and `afterToolCall` only runs inside `finalizeExecutedToolCall` for prepared executions. Because this branch only installs an `afterToolCall` rewrite, the common `Tool X not found` path on current main keeps the generic error.
  - confidence: 0.9
- **[P2] Compose with the current public hook surface:** `src/agents/pi-embedded-runner/run/attempt.ts:1935-1938`
  - body: Current `Agent` exposes `afterToolCall` as a public mutable property that `AgentSession` already uses for extension `tool_result` dispatch; the current dependency does not expose this private `_afterToolCall`/`setAfterToolCall` API. After rebase, this needs to compose through the public property or it will fail to build or bypass the existing hook.
  - confidence: 0.86
- **[P2] Use the run-visible skills snapshot:** `src/agents/pi-embedded-runner/run/attempt.ts:1954-1955`
  - body: Sandboxed non-rw runs clear `skillsSnapshotForRun` before exposing skills to the model, but this hook checks `params.skillsSnapshot`. A guessed hidden skill name can therefore be confirmed by the new hint even when that skill was not visible in the run.
  - confidence: 0.88
- **[P3] Parse the full prompt-facing skill name:** `src/agents/pi-embedded-helpers/errors.ts:555-559`
  - body: The helper only captures `[a-z0-9_-]+`, while OpenClaw renders raw skill names in `<available_skills>`. Valid prompt-facing names with a leading `$`, spaces, or punctuation can be truncated or missed, so they will not get the intended hint.
  - confidence: 0.8
