## Summary

Keep open: current main still lacks the requested skill-aware unknown-tool hint, and this external PR is focused, but the branch is not mergeable as written because it targets an older Pi API and misses the current missing-tool result path.

## What This Changes

The PR adds an agent error rewrite so unknown tool calls that match registered skills tell the model to read the skill's SKILL.md.

## Best Possible Solution

Implement the hint on the current Pi missing-tool/immediate result handling path or another supported current API, while preserving existing tool-result hooks and matching against the run's hydrated visible skills.

## Reproduction Assessment

Yes. Source inspection against current main and `@earendil-works/pi-agent-core@0.74.0` shows current missing-tool results stay generic and current main has no SKILL.md hint rewrite; the PR body screenshot shows the intended after-fix behavior in a real session.

## Solution Assessment

No. The intent is narrow and useful, but the current patch uses a stale private setter and a hook boundary that current missing-tool immediate results do not cross; it should target the current dependency contract and hydrated skill data.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Route missing-tool results through the skill hint:** `src/agents/pi-embedded-runner/run/attempt.ts:1940`
  - body: Current `@earendil-works/pi-agent-core@0.74.0` creates missing-tool failures as `kind: "immediate"`; both execution paths emit those results directly, and `afterToolCall` only runs from `finalizeExecutedToolCall` for prepared tools. Installing this rewrite in `setAfterToolCall` means the `Tool X not found` case this PR targets still returns the generic message on current main.
  - confidence: 0.94
- **[P2] Rebase onto the current Pi package API:** `src/agents/pi-embedded-runner/run/attempt.ts:3-8`
  - body: The PR imports `AfterToolCall*` from `@mariozechner/pi-agent-core` and calls `activeSession.agent.setAfterToolCall`, but current main depends on `@earendil-works/pi-agent-core@0.74.0`; its `Agent` exposes `afterToolCall` as an option/property and no `setAfterToolCall` method. A rebase onto current main will fail typechecking until the hook wiring uses the current API.
  - confidence: 0.93
- **[P2] Match restored skills without resolvedSkills:** `src/agents/pi-embedded-helpers/errors.ts:573-575`
  - body: This matcher returns false whenever `skillsSnapshot.resolvedSkills` is absent, but session loading strips that runtime-only field while preserving the prompt/catalog, and the runner reloads skill entries for that case. Restored sessions can still show the skill in `<available_skills>` yet skip the new hint; pass hydrated run skill data or fall back to `skillsSnapshot.skills`.
  - confidence: 0.86
