## Summary

Keep open. The feature is still not implemented on current main, but this branch should not merge as-is because it calls unsupported pi-agent APIs and hooks a path that misses the pinned dependency's missing-tool result.

## What This Changes

The branch extracts unknown-tool parsing and adds a skill-aware `afterToolCall` rewrite that returns a `SKILL.md` hint for missing-tool errors matching registered skills.

## Best Possible Solution

Keep the feature direction, but implement it through a supported immediate missing-tool result path, preserve the existing `AgentSession` tool-result hook, define visible-skill and tool-collision semantics, and add focused helper/runner regression coverage plus a user-facing changelog entry before merge.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: current main has no skill-aware unknown-tool rewrite, and the pinned pi-agent source shows `Tool X not found` is created as an immediate prepare-time result that does not run `afterToolCall`.

## Solution Assessment

No. This PR is not the best way to solve the issue because it calls a nonexistent setter and relies on a hook path that misses the main missing-tool result; a supported immediate-result seam or upstream hook contract is safer.
