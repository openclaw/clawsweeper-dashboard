## Summary

Keep open: the user-facing hint is useful, but the branch is stale against the current Pi Agent API, misses the current missing-tool path, and uses the wrong skill visibility boundary.

## What This Changes

The PR extracts unknown-tool parsing and adds an embedded-runner afterToolCall rewrite that turns unknown-tool errors matching registered skills into SKILL.md guidance.

## Best Possible Solution

Implement the hint at the current OpenClaw-owned missing-tool/result boundary using run-visible skills, full prompt-facing names, collision preservation, and focused regression coverage.

## Reproduction Assessment

Yes, source-based. Current main still has generic unknown-tool handling, the PR screenshot shows the intended after-fix text, and the pinned Pi source shows this hook wiring misses immediate missing-tool results.

## Solution Assessment

No. The desired behavior should be implemented against the current @earendil Pi API and OpenClaw's run-visible skill/result boundary, not through stale private hook state.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Use the current Pi Agent API:** `src/agents/pi-embedded-runner/run/attempt.ts:4-10`
  - body: Current main pins @earendil-works/pi-agent-core@0.74.0, whose Agent exposes afterToolCall as an option/property and has no setAfterToolCall method. Importing @mariozechner types and calling the setter will not compile or compose after rebasing.
  - confidence: 0.95
- **[P2] Route immediate missing-tool results through the hint:** `src/agents/pi-embedded-runner/run/attempt.ts:1937`
  - body: In the pinned Pi loop, an unknown tool returns an immediate Tool <name> not found result from prepareToolCall, and afterToolCall only runs for prepared/executed tools. Installing this rewrite as an afterToolCall hook leaves the main skill-called-as-tool case unchanged.
  - confidence: 0.94
- **[P2] Match only skills visible in this run:** `src/agents/pi-embedded-runner/run/attempt.ts:1953`
  - body: Current main clears skillsSnapshotForRun for sandboxed non-rw runs before building the prompt/runtime state, but this rewrite checks params.skillsSnapshot. A guessed hidden skill name could receive confirmation even though it was not visible to the model in that run.
  - confidence: 0.89
- **[P2] Handle snapshots without resolvedSkills:** `src/agents/pi-embedded-helpers/errors.ts:575-578`
  - body: hasVisibleSkillMatch returns false when resolvedSkills is absent, but current session persistence deliberately strips resolvedSkills and the runner reloads entries when that happens. The hint can silently disappear for prompt/skills-only snapshots unless the rewrite uses the run-visible skill state.
  - confidence: 0.86
- **[P3] Parse full prompt-facing skill names:** `src/agents/pi-embedded-helpers/errors.ts:557-558`
  - body: resolveUnknownToolName captures only [a-z0-9_-]+, while the skills prompt renders raw skill.name and supported names can contain spaces or punctuation. Valid visible skills such as GitHub Actions would be truncated and miss the hint.
  - confidence: 0.82
