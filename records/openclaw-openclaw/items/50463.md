## Summary

Keep open: the idea is focused and useful, but this branch is incorrect against current main because it targets a stale Pi API and wires the hint to a hook path that current missing-tool results bypass; restored sessions can also omit the field this matcher requires.

## What This Changes

The PR rewrites unknown-tool tool-result text into a SKILL.md hint when the missing name matches a registered skill, and wires that rewrite through an embedded-runner afterToolCall hook.

## Best Possible Solution

Implement the hint on the supported current Pi/OpenClaw missing-tool path, preserve existing hook behavior, and match visible skills even when persisted snapshots omit resolvedSkills.

## Reproduction Assessment

Yes, source-reproducible. Current main returns generic Tool X not found text for Pi missing-tool results, and the PR screenshot shows the intended after-fix behavior in a real session.

## Solution Assessment

No. The requested behavior is narrow and maintainable, but this branch uses a stale Pi package API and a hook boundary that current prepare-time missing-tool results do not reach.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Route missing-tool immediate results through the hint:** `src/agents/pi-embedded-runner/run/attempt.ts:1940`
  - body: Current @earendil-works/pi-agent-core 0.74.0 returns missing tools as kind: "immediate" from prepareToolCall, and both sequential and parallel paths finalize those results without invoking afterToolCall. Installing the rewrite only with setAfterToolCall means the common skill-as-tool failure still returns the generic Tool not found text after rebasing.
  - confidence: 0.94
- **[P2] Rebase the hook onto the current Pi API:** `src/agents/pi-embedded-runner/run/attempt.ts:3-8`
  - body: The branch imports @mariozechner/pi-agent-core types and calls activeSession.agent.setAfterToolCall, but current main uses @earendil-works/pi-agent-core 0.74.0 where Agent exposes afterToolCall as an option/property and no setter. This needs a current-API implementation before the branch can build or merge.
  - confidence: 0.93
- **[P2] Match restored skills when resolvedSkills is absent:** `src/agents/pi-embedded-helpers/errors.ts:573-575`
  - body: hasVisibleSkillMatch returns false whenever skillsSnapshot.resolvedSkills is missing, but restored sessions deliberately strip that runtime-only cache while keeping the lightweight skills list and reloading entries for the run. Those visible restored skills would not receive the new hint.
  - confidence: 0.87
