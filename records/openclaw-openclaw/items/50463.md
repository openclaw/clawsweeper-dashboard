## Summary

Keep open: current main does not have the skill-specific unknown-tool hint, but this PR still has blocking correctness and low-severity security issues in the agent tool-call boundary.

## What This Changes

The PR extracts unknown-tool parsing and adds a Pi embedded runner hook that rewrites missing-tool results matching registered skills into `SKILL.md` guidance.

## Best Possible Solution

Implement the hint at the current missing-tool result boundary, preserve extension `tool_result` behavior, match only run-visible skills, and cover hidden, non-skill, punctuation, and collision cases.

## Reproduction Assessment

Yes, source-based: current main returns generic missing-tool text and has no skill-specific `SKILL.md` hint path, while the PR body screenshot shows the desired after-fix behavior.

## Solution Assessment

No. The UX direction is useful, but this patch should move to the current missing-tool boundary, use the public hook surface only where needed, and match against `skillsSnapshotForRun`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Route skill hints through missing-tool preparation:** `src/agents/pi-embedded-runner/run/attempt.ts:1937`
  - body: The current pinned `@earendil-works/pi-agent-core@0.74.0` returns unknown tools as `kind: "immediate"` and emits them without calling `afterToolCall`. Because this patch only rewrites from `setAfterToolCall`, the main `Tool X not found` path still returns the generic error.
  - confidence: 0.91
- **[P2] Compose with the current public hook surface:** `src/agents/pi-embedded-runner/run/attempt.ts:1937`
  - body: Current `Agent` exposes `afterToolCall` as a public property and `AgentSession` already assigns it for extension `tool_result` dispatch; there is no public `setAfterToolCall` method in the pinned dependency. This needs to preserve the existing hook instead of depending on a private/stale API.
  - confidence: 0.88
- **[P2] Use the run-visible skills snapshot:** `src/agents/pi-embedded-runner/run/attempt.ts:1955`
  - body: Sandboxed non-rw runs clear `skillsSnapshotForRun` before exposing skills to the model, but this patch checks `params.skillsSnapshot`. If the rewrite runs, a guessed hidden skill name can be confirmed even though that skill was not visible in the run.
  - confidence: 0.86
- **[P3] Parse complete prompt-facing skill names:** `src/agents/pi-embedded-helpers/errors.ts:555-559`
  - body: `resolveUnknownToolName()` captures only `[a-z0-9_-]+`, while OpenClaw renders raw skill names in `<available_skills>`. Visible names with a leading `$`, spaces, or punctuation are truncated or missed, so the new hint fails for valid names the model can see.
  - confidence: 0.8
