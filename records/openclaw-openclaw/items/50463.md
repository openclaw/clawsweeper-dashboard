## Summary

Keep open: the feature direction is plausible, but the current PR is not mergeable because it wires the rewrite through an unsupported/pinned dependency path and misses the main missing-tool result path it is meant to improve.

## What This Changes

The PR extracts unknown-tool parsing, adds a skill-aware tool-result rewrite helper, and wires an embedded-runner afterToolCall hook to add a SKILL.md hint when a missing tool name matches a registered skill.

## Best Possible Solution

Keep the feature direction, but implement it through a supported missing-tool result seam that composes with AgentSession.afterToolCall, handles full skill-name matching semantics, and includes focused regression coverage plus a changelog entry.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: current main has no skill-aware rewrite, and a model calling a registered skill as an unavailable tool still gets the generic missing-tool result in the pinned dependency path. I did not execute a live agent run because this review was read-only and pnpm is unavailable in the checkout.

## Solution Assessment

No. The current PR is not the best fix because it calls an unsupported setter/private hook and relies on afterToolCall for a missing-tool path that the pinned dependency emits as an immediate result before afterToolCall runs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Use the supported afterToolCall property:** `src/agents/pi-embedded-runner/run/attempt.ts:1937`
  - body: @mariozechner/pi-agent-core@0.71.1 exposes `agent.afterToolCall` as a public property; it does not expose `setAfterToolCall` or `_afterToolCall`. This call will fail against the current type/runtime contract and also fails to compose with AgentSession's existing handler.
  - confidence: 0.97
- **[P1] Route missing-tool results through a supported seam:** `src/agents/pi-embedded-runner/run/attempt.ts:1937`
  - body: The target `Tool X not found` result is produced by `prepareToolCall()` as an immediate result, and the sequential/parallel paths emit immediate results without calling `finalizeExecutedToolCall()`, where `afterToolCall` runs. Even with the setter fixed, this hook misses the main case the PR is trying to improve.
  - confidence: 0.94
- **[P3] Match complete skill names before adding the hint:** `src/agents/pi-embedded-helpers/errors.ts:557-558`
  - body: `resolveUnknownToolName()` captures only `[a-z0-9_-]+`, but visible skill names come from raw skill metadata and may contain spaces or punctuation. A valid skill such as `GitHub Actions` would be extracted as `GitHub`, so the new hint silently fails for those skills.
  - confidence: 0.82
- **[P3] Add the required changelog entry:** `src/agents/pi-embedded-helpers/errors.ts:610`
  - body: This adds new user-visible guidance, but the branch does not update `CHANGELOG.md`. OpenClaw requires user-facing fix/feature changes to include an active-version changelog entry before merge.
  - confidence: 0.86
