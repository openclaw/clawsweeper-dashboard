## Summary

This PR is a plausible focused agent UX improvement, but the current patch does not reach the missing-tool path on current main’s pinned Pi agent package, so it should stay open for revision rather than close as stale or implemented.

## What This Changes

The PR adds a helper to rewrite unknown-tool error text into a skill-specific SKILL.md hint and installs it from the embedded attempt runner’s after-tool-call hook.

## Best Possible Solution

Implement the hint at the current missing-tool result path for the pinned `@earendil-works/pi-agent-core` behavior, use the same run-visible skill snapshot that built the prompt, preserve existing hook composition, and add focused regression coverage for skill match, non-skill unknown tool, sandbox-hidden skills, and tool-name collisions.

## Reproduction Assessment

Yes, source-based. Current main’s pinned Pi agent loop creates a generic immediate `Tool X not found` result for missing tools, and current OpenClaw main has no skill-specific rewrite in that path.

## Solution Assessment

No. Installing the rewrite only as an `afterToolCall` hook is not the narrowest correct solution on current main because the dependency does not call that hook for immediate missing-tool results; the fix should process the immediate missing-tool result path directly or through a current OpenClaw wrapper.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Route immediate missing-tool results through the hint:** `src/agents/pi-embedded-runner/run/attempt.ts:1937`
  - body: Current main pins `@earendil-works/pi-agent-core@0.74.0`, whose `prepareToolCall()` returns a `Tool X not found` result with `kind: "immediate"`. In the published loop, `afterToolCall` only runs from `finalizeExecutedToolCall()` for prepared/executed tools, so this hook does not see the missing-tool case the PR is trying to rewrite. Wire the helper into the immediate missing-tool path or an OpenClaw wrapper around those results.
  - confidence: 0.91
- **[P2] Use the run-visible skill snapshot for hints:** `src/agents/pi-embedded-runner/run/attempt.ts:1954`
  - body: The runner clears `skillsSnapshotForRun` when a sandboxed run should not expose workspace skills, and that value is what drives the skill prompt/env. Passing `params.skillsSnapshot` here can still advertise a persisted skill as available even when this run hid skills, producing a misleading recovery hint. Use the same run-visible snapshot that built the prompt.
  - confidence: 0.82
- **[P3] Match the full registered skill name:** `src/agents/pi-embedded-helpers/errors.ts:556-557`
  - body: `resolveUnknownToolName()` only captures `[a-z0-9_-]+`, but skill names rendered into the prompt can include spaces or symbols and users/models may spell skill invocations with a leading `$`. Those valid skill-looking calls would fail to get the new hint, so parse quoted tool names more broadly before comparing against the snapshot.
  - confidence: 0.74
