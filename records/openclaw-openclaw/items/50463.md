## Summary

Keep open: the skill-hint behavior is still not implemented on current main, but this branch is not merge-ready because it targets retired `pi-embedded-*` files and the hook-based design misses the active runtime's immediate missing-tool path.

## What This Changes

The PR rewrites unknown-tool tool results that match registered skill names by adding a `Read its SKILL.md first` hint through an `afterToolCall` hook in legacy pi-embedded agent files.

## Best Possible Solution

If maintainers want this feature, rework it against the active `embedded-agent-*`/agent-core boundary, preferably before generic missing-tool results are emitted, with focused regression coverage for skill matches, non-skill unknown tools, tool/skill collisions, and immediate results.

## Reproduction Assessment

Yes by source inspection: current `prepareToolCall` emits `Tool X not found` as an immediate result when no tool is resolved, and `afterToolCall` only runs later for prepared/executed tools. The contributor screenshot also shows the intended after-fix output on the PR branch.

## Solution Assessment

No; the branch's hook is only a plausible old-runtime approach and is the wrong layer for current main. A current-runtime missing-tool/deferred-tool boundary with regression tests is the safer solution.
