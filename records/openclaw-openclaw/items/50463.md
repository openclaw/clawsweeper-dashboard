## Summary

Keep open. The requested hint is still useful and not implemented on current main, but this branch needs changes before merge because it is based on stale Pi package APIs, wires the hint through a path current Pi does not use for missing tools, and can match skills outside the run-visible skill set.

## What This Changes

The PR adds a helper and embedded-runner hook to rewrite unknown-tool errors into a SKILL.md hint when the missing tool name matches a registered skill.

## Best Possible Solution

Implement the hint at the current OpenClaw-owned missing-tool result boundary using @earendil-works contracts, run-visible skills, collision preservation, robust quoted-name parsing, and focused regression coverage.

## Reproduction Assessment

Yes, source-based. Current main still has generic missing-tool handling with no skill hint, and the current Pi dependency shows the common missing-tool result is an immediate prepare result.

## Solution Assessment

No. The proposed direction is reasonable, but the current branch is not the best implementation because it targets stale Pi APIs and should be moved to the current missing-tool/run-visible-skill boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Use the current Pi package API:** `src/agents/pi-embedded-runner/run/attempt.ts:3-8`
  - body: Current main uses `@earendil-works/pi-agent-core` 0.74.0 and the current Agent type exposes `afterToolCall` as an option/property, not the old `@mariozechner` imports plus `setAfterToolCall`. After rebasing, this branch will not compile or compose against the dependency currently pinned by OpenClaw.
  - confidence: 0.96
- **[P2] Route missing-tool results through the hint:** `src/agents/pi-embedded-runner/run/attempt.ts:1940`
  - body: The current Pi loop returns missing tools as immediate prepare results and only applies `afterToolCall` from `finalizeExecutedToolCall` for prepared/executed tools. Because this patch only installs an after-tool-call hook, the normal `Tool X not found` path this feature targets will still return the generic error on current main.
  - confidence: 0.94
- **[P2] Match only skills visible in this run:** `src/agents/pi-embedded-runner/run/attempt.ts:1954-1957`
  - body: The runner can clear skills for sandboxed non-read-write sessions before building the run state, but the patch checks `params.skillsSnapshot`. A model that guesses a hidden skill name could get a SKILL.md hint even when that skill was removed from this run's prompt/runtime state.
  - confidence: 0.9
- **[P2] Handle snapshots without resolved skills:** `src/agents/pi-embedded-helpers/errors.ts:573-575`
  - body: `resolvedSkills` is optional while the lightweight `skills` list is required, and the runner reloads skill entries whenever `resolvedSkills` is absent. Returning false here can suppress the hint for sessions that still advertise the skill through the snapshot prompt/catalog.
  - confidence: 0.84
- **[P3] Parse quoted skill names completely:** `src/agents/pi-embedded-helpers/errors.ts:558-562`
  - body: The parser only captures `[a-z0-9_-]+`, but prompt-facing skill names are raw `skill.name` values and can include punctuation such as nested skill names. Valid errors like `Tool "foo/bar" not found` would be truncated and fail the skill match.
  - confidence: 0.8
