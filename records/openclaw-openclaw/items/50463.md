## Summary

Keep open: the feature is plausible and the PR body includes real after-fix proof, but the patch is not correct against the dependency contract or current main. The proposed `afterToolCall` hook does not see the missing-tool path this feature targets, and the branch is stale against the current Pi package namespace/API.

## What This Changes

The PR adds a helper and embedded-runner hook to rewrite unknown-tool errors into a SKILL.md hint when the missing tool name matches a registered skill.

## Best Possible Solution

Implement the hint at the immediate missing-tool result boundary, or use a dependency API that explicitly runs for immediate results, then add focused tests on current `@earendil-works` Pi packages.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a missing tool is returned as an immediate `Tool X not found` result before `afterToolCall`, so the proposed hook will not rewrite the central case without moving the rewrite boundary.

## Solution Assessment

No. The intent is sound, but the current implementation hooks the wrong lifecycle point and uses a stale Pi API; the maintainable fix should operate where immediate missing-tool results are created or emitted.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Route missing-tool results through the skill hint:** `src/agents/pi-embedded-runner/run/attempt.ts:1940`
  - body: The new hook is installed via `afterToolCall`, but the targeted `Tool X not found` case is produced as an immediate result by `prepareToolCall` in both the PR branch's Pi core and current main's Pi core. Immediate results are emitted directly and never enter `finalizeExecutedToolCall`, so this feature keeps returning the generic missing-tool error for the main scenario it is meant to fix. Move the rewrite to the immediate-result path or update the dependency/API so immediate results invoke the hook.
  - confidence: 0.96

Rebasing this diff onto current main needs to use the current package namespace/API or it will not typecheck.
  - confidence: 0.9
