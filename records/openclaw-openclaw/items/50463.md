## Summary

Keep open: the requested hint is useful and not implemented on current main, but this PR still needs code changes before merge because it targets a stale Pi API and wires the rewrite through a hook path that current missing-tool results do not hit.

## What This Changes

The PR adds a helper that rewrites unknown-tool errors into a SKILL.md hint when the unknown name matches a registered skill, and wires that helper into the embedded Pi agent run attempt path.

## Best Possible Solution

Implement the hint on the current Pi missing-tool/immediate result path or another supported current API, while preserving existing hook behavior and matching against hydrated visible skills or the persisted skill-name snapshot.

## Reproduction Assessment

Yes. Source inspection against current main and `@earendil-works/pi-agent-core@0.74.0` shows missing tools still return generic immediate results, and the PR screenshot shows the intended after-fix output in a live session.

## Solution Assessment

No. The intended behavior is narrow, but the current patch uses a stale package/API and an `afterToolCall` hook path that current missing-tool immediate results bypass.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Route missing-tool results through the skill hint:** `src/agents/pi-embedded-runner/run/attempt.ts:1925`
  - body: Current `@earendil-works/pi-agent-core@0.74.0` creates missing-tool failures as `kind: "immediate"`, and both execution paths emit those results without calling `finalizeExecutedToolCall` or `afterToolCall`. Installing the rewrite in this hook means the `Tool X not found` case this PR targets still returns the generic message on current main.
  - confidence: 0.94
- **[P2] Rebase onto the current Pi package API:** `src/agents/pi-embedded-runner/run/attempt.ts:4-8`
  - body: The PR imports `AfterToolCall*` from `@mariozechner/pi-agent-core` and calls `activeSession.agent.setAfterToolCall`, but current main depends on `@earendil-works/pi-agent-core@0.74.0`; its `Agent` exposes `afterToolCall` as an option/property and no `setAfterToolCall` method. This will not typecheck against current main until the hook wiring is updated.
  - confidence: 0.93
- **[P2] Match restored skills without resolvedSkills:** `src/agents/pi-embedded-helpers/errors.ts:573-575`
  - body: This matcher returns false whenever `skillsSnapshot.resolvedSkills` is absent, but that field is optional and session loading strips it while preserving `skillsSnapshot.skills`. Restored sessions can still show the skill in the available-skills prompt yet skip the new hint; pass hydrated run skill data or fall back to the persisted skill names.
  - confidence: 0.86
