## Summary

Keep open: current main still lacks the requested SKILL.md hint, but this branch is not merge-ready against the current agent loop because the added hook is not on the missing-tool result boundary and it can expose skills hidden from the run.

## What This Changes

The branch extracts unknown-tool parsing and adds an embedded-runner tool-result hook that rewrites missing-tool errors matching registered skills with a SKILL.md hint.

## Best Possible Solution

Implement the hint at the current missing-tool result boundary, preserve existing extension tool-result hooks, use only the run-visible skill set, and cover skill match, non-skill, hidden skill, punctuation, and tool-collision cases.

## Reproduction Assessment

Yes, source-based. Current main has no SKILL.md hint, and the pinned agent-loop source creates a generic `Tool X not found` result for missing tools.

## Solution Assessment

No. The UX direction is reasonable, but this PR wires the rewrite through the wrong current-main boundary and uses the persisted skill snapshot instead of the run-visible one.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Route skill hints through the missing-tool path:** `src/agents/pi-embedded-runner/run/attempt.ts:1925-1938`
  - body: The current pinned agent loop returns missing tools as immediate `Tool X not found` results from `prepareToolCall`, and those immediate results do not run `afterToolCall`. Since this PR only installs an `afterToolCall` rewrite, the model path it is meant to fix still receives the generic error on current main.
  - confidence: 0.9
- **[P2] Compose with the public afterToolCall hook:** `src/agents/pi-embedded-runner/run/attempt.ts:1936-1938`
  - body: Current `Agent` exposes `afterToolCall` as a mutable public property and `pi-coding-agent` already installs extension `tool_result` dispatch there. Reading a private `_afterToolCall` field and calling `setAfterToolCall` does not match the current dependency surface and can drop or bypass the existing handler after rebase.
  - confidence: 0.86
- **[P2] Use the run-visible skills snapshot:** `src/agents/pi-embedded-runner/run/attempt.ts:1954`
  - body: Sandboxed non-rw runs intentionally clear `skillsSnapshotForRun` before exposing skills to the model, but this hook checks `params.skillsSnapshot`. A guessed hidden skill name can therefore be confirmed by the new hint even when that skill was not visible in the run.
  - confidence: 0.88
- **[P3] Parse the full prompt-facing skill name:** `src/agents/pi-embedded-helpers/errors.ts:555-559`
  - body: The new helper only captures `[a-z0-9_-]+`, while OpenClaw renders raw skill names in the available-skills prompt and has separate sanitization because names may contain spaces or symbols. Valid prompt-facing skills with punctuation or spaces will miss the hint or be truncated before matching.
  - confidence: 0.8
