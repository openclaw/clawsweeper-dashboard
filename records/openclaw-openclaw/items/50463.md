## Summary

Keep open. The UX direction is useful and current main still lacks a skill-specific unknown-tool hint, but this PR is not correct against the current runner/dependency contract and needs focused repairs before merge.

## What This Changes

The PR extracts unknown-tool parsing and adds a Pi embedded runner hook that rewrites missing-tool results matching registered skills into `SKILL.md` guidance.

## Best Possible Solution

Implement the hint at the current missing-tool result boundary, preserve extension `tool_result` behavior, match only run-visible skills, parse full prompt-facing names, and cover hidden, non-skill, punctuation, and collision cases.

## Reproduction Assessment

Yes, source-based. Current main returns generic missing-tool text with no skill-specific `SKILL.md` hint path, and the PR body screenshot shows the desired after-fix behavior in a real session.

## Solution Assessment

No. The direction is useful, but this patch should move to the current missing-tool boundary, compose with the public hook surface, and match against `skillsSnapshotForRun`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Route skill hints through missing-tool preparation:** `src/agents/pi-embedded-runner/run/attempt.ts:1937`
  - body: The current pinned `@earendil-works/pi-agent-core@0.74.0` returns unknown tools as `kind: "immediate"` and emits those results without calling `afterToolCall`. Because this patch rewrites only from the after-tool hook, the main `Tool X not found` path still returns the generic error.
  - confidence: 0.91
- **[P2] Compose with the public afterToolCall property:** `src/agents/pi-embedded-runner/run/attempt.ts:1937`
  - body: Current `Agent` exposes `afterToolCall` as a public property and has no public `setAfterToolCall` method, while `AgentSession` already assigns that property for extension `tool_result` dispatch. This needs to preserve the existing handler through the current public surface instead of depending on a stale/private API.
  - confidence: 0.9
- **[P2] Use the run-visible skills snapshot:** `src/agents/pi-embedded-runner/run/attempt.ts:1955`
  - body: Sandboxed non-rw runs clear `skillsSnapshotForRun` before exposing skills to the model, but this patch checks `params.skillsSnapshot`. If the rewrite runs, a guessed hidden skill name can be confirmed even though that skill was not visible in the run.
  - confidence: 0.87
- **[P3] Parse complete prompt-facing skill names:** `src/agents/pi-embedded-helpers/errors.ts:555-559`
  - body: `resolveUnknownToolName()` captures only `[a-z0-9_-]+`, while OpenClaw renders raw skill names in `<available_skills>`. Visible names with a leading `$`, spaces, or punctuation are truncated or missed, so the new hint fails for valid names the model can see.
  - confidence: 0.81
