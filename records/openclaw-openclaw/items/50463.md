## Summary

Keep open: the requested hint is useful and current main still lacks it, but the branch needs changes before merge because the hook-only implementation misses the current dependency's immediate missing-tool path and uses the wrong skill visibility boundary.

## What This Changes

The branch extracts unknown-tool parsing and adds an embedded-runner hook that rewrites matching missing-tool results with a SKILL.md hint when the name matches a registered skill.

## Best Possible Solution

Implement the hint at the actual missing-tool result boundary, use the same run-visible skill snapshot that built the prompt, broaden parsing to the full skill-name surface, and add focused regression tests.

## Reproduction Assessment

Yes, source-based. Current main lacks a skill-specific unknown-tool hint, the dependency path for missing tools is inspectable, and the PR screenshot demonstrates the intended branch behavior in a live session.

## Solution Assessment

No. The desired behavior is reasonable, but the current hook-only patch is not the best implementation because it misses immediate missing-tool results and should use the run-visible skill snapshot.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Route immediate missing-tool results through the hint:** `src/agents/pi-embedded-runner/run/attempt.ts:1925-1937`
  - body: The currently pinned pi-agent-core returns `Tool X not found` as an immediate result from `prepareToolCall`, and those immediate results are emitted without `finalizeExecutedToolCall` or `afterToolCall`. Because this rewrite is only installed as an `afterToolCall` hook, the main missing-tool path this feature targets still keeps the generic error.
  - confidence: 0.9
- **[P2] Use the run-visible skill snapshot:** `src/agents/pi-embedded-runner/run/attempt.ts:1954`
  - body: The runner clears `skillsSnapshotForRun` when sandboxing hides workspace skills and uses that value to build the prompt, but this hook passes the persisted `params.skillsSnapshot`. That can tell the model a hidden skill exists in runs where the skill was not exposed.
  - confidence: 0.86
- **[P3] Parse the full skill-name surface:** `src/agents/pi-embedded-helpers/errors.ts:555-559`
  - body: Skill names are rendered from raw `skill.name` and can contain spaces or symbols before command sanitization, but the new parser captures only `[a-z0-9_-]+`. Valid names such as `$openclaw-pr-maintainer` or `GitHub Actions` will not match and will keep the generic error.
  - confidence: 0.8
