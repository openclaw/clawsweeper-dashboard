## Summary

Keep open. The proposed UX is useful, and the PR includes real-session screenshot proof, but the patch is not correct against current main: it wires the hint through a hook path that misses current missing-tool results, uses stale/private Pi APIs, and can disclose hidden skill names in sandboxed runs.

## What This Changes

The branch extracts unknown-tool parsing and adds an agent-runner hook that rewrites missing-tool errors matching registered skills into `SKILL.md` guidance.

## Best Possible Solution

Implement the hint at the current missing-tool result boundary, compose through the current public Pi hook surface, match only run-visible skills, and add focused regression coverage for visible, hidden, non-skill, punctuation, and collision cases.

## Reproduction Assessment

Yes, source-based. Current main has no skill-specific `SKILL.md` hint, and the pinned Pi agent loop creates generic `Tool X not found` results for missing tool calls.

## Solution Assessment

No. The UX direction is reasonable, but this branch should move the rewrite to the missing-tool boundary and use the run-visible skill set instead of a stale/private hook surface and full persisted snapshot.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Route the hint through the missing-tool result path:** `src/agents/pi-embedded-runner/run/attempt.ts:1925-1938`
  - body: Current `@earendil-works/pi-agent-core@0.74.0` creates missing-tool results as `kind: "immediate"` in `prepareToolCall`; `afterToolCall` is invoked only from `finalizeExecutedToolCall` for prepared executions. Because this branch installs only an `afterToolCall` rewrite, a model calling a skill name as an unknown tool still gets the generic error.
  - confidence: 0.9
- **[P2] Compose with the current Agent hook API:** `src/agents/pi-embedded-runner/run/attempt.ts:1935-1938`
  - body: Current main uses `@earendil-works/pi-agent-core`, where `Agent` exposes a public `afterToolCall` property and has no `setAfterToolCall` or private `_afterToolCall` API. After rebase this will not build and can bypass existing tool-result dispatch unless it composes through the public property.
  - confidence: 0.88
- **[P2] Match only skills visible in this run:** `src/agents/pi-embedded-runner/run/attempt.ts:1954-1955`
  - body: Sandboxed non-rw runs clear `skillsSnapshotForRun` before prompt exposure, but this passes the full `params.skillsSnapshot` into the matcher. A guessed hidden skill name can therefore be confirmed by the new hint even when that skill was not visible to the model in the run.
  - confidence: 0.9
- **[P3] Parse the complete prompt-facing skill name:** `src/agents/pi-embedded-helpers/errors.ts:555-559`
  - body: The extractor captures only `[a-z0-9_-]+`, while `<available_skills>` renders raw `skill.name` and command-name sanitization exists separately. Visible skills such as `$openclaw-pr-maintainer` or `GitHub Actions` will be truncated or missed, so the new hint will not fire for valid skill names.
  - confidence: 0.83
