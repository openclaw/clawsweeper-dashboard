## Summary

Keep this PR open: the requested skill-specific hint is useful and not already on current main, and the contributor supplied real-session screenshot proof, but the patch is not correct against the current runner and Pi dependency contracts.

## What This Changes

The PR extracts unknown-tool parsing helpers and adds an embedded-runner hook that rewrites unknown-tool error tool results when the missing name matches a registered skill.

## Best Possible Solution

Implement the hint at the actual missing-tool result boundary, compose with the current `afterToolCall` property only where that hook really applies, use `skillsSnapshotForRun`, and cover skill-match, non-skill, hidden-skill, collision, and broad-name cases with focused tests.

## Reproduction Assessment

Yes, source-based. Current main lacks the skill hint, and the pinned `@earendil-works/pi-agent-core@0.74.0` source shows missing tools are immediate results that bypass the proposed `afterToolCall` hook.

## Solution Assessment

No. The UX direction is reasonable, but this PR should be moved to the active missing-tool boundary and use the same visible skill set that was exposed to the run.

- **[P2] Preserve the current afterToolCall handler:** `src/agents/pi-embedded-runner/run/attempt.ts:1937`
  - body: Current `pi-coding-agent` installs extension `tool_result` dispatch on `agent.afterToolCall`, while this code reads `_afterToolCall` and calls `setAfterToolCall`. Against the current dependency surface that drops the existing handler or fails entirely, so compose through the current property/API instead of the private field/setter assumption.
  - confidence: 0.86
- **[P2] Route missing-tool results through the actual boundary:** `src/agents/pi-embedded-runner/run/attempt.ts:1925-1937`
  - body: The pinned `pi-agent-core` returns missing tools as immediate results from `prepareToolCall`, and those immediate results are emitted without running `afterToolCall`. A model calling a skill name as a nonexistent tool will still see the generic message unless the rewrite is applied where immediate missing-tool results are created or emitted.
  - confidence: 0.91
- **[P2] Use the run-visible skill snapshot:** `src/agents/pi-embedded-runner/run/attempt.ts:1954`
  - body: The runner clears `skillsSnapshotForRun` for sandboxed non-rw runs before exposing skills in the prompt, but this hook uses the persisted `params.skillsSnapshot`. That can hint at a skill name that was intentionally hidden from this run, so the rewrite needs the same visible snapshot used to build `skillsPrompt`.
  - confidence: 0.86
- **[P3] Match valid skill names before comparing:** `src/agents/pi-embedded-helpers/errors.ts:555-559`
  - body: The extractor only captures `[a-z0-9_-]+`, but OpenClaw renders raw skill names and separately sanitizes command names because names may contain spaces or symbols. Valid skills such as names with spaces, punctuation, or `$` spelling will miss the hint or be truncated before matching.
  - confidence: 0.8
