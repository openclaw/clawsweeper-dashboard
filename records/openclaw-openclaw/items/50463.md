## Summary

This PR addresses a real agent usability gap, but the branch is not correct against current main: it uses stale Pi package/API assumptions, misses the current missing-tool boundary, and can match hidden skills rather than the run-visible skill set.

## What This Changes

The PR rewrites unknown-tool error tool results that match registered skill names into a `Read its SKILL.md first` hint, with helper extraction in `errors.ts` and an `afterToolCall` hook in `attempt.ts`.

## Best Possible Solution

Implement the hint at the current OpenClaw-owned missing-tool result boundary using current `@earendil-works` contracts, run-visible skills, collision preservation, prompt-facing skill-name parsing, and focused regression coverage.

## Reproduction Assessment

Yes, source-based. Current main still has generic missing-tool behavior, and the pinned Pi package has an immediate `Tool X not found` path that can be targeted by a regression test.

## Solution Assessment

No. The desired behavior is useful, but this branch uses a stale hook/API path and raw snapshot matching; the narrower maintainable fix should operate where current main handles missing-tool results for the visible run state.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Use the current Pi package API:** `src/agents/pi-embedded-runner/run/attempt.ts:4-10`
  - body: Current main imports and pins `@earendil-works/pi-agent-core`, whose Agent contract exposes `afterToolCall` as an option/property and has no `setAfterToolCall` method. This branch imports from the old `@mariozechner` namespace and calls the stale setter, so it will not compile or compose correctly after rebasing.
  - confidence: 0.96
- **[P2] Route missing-tool results through the hint:** `src/agents/pi-embedded-runner/run/attempt.ts:1925-1941`
  - body: The current Pi loop creates missing-tool failures as immediate `Tool X not found` results from `prepareToolCall`, before normal tool execution. Installing the rewrite only as an `afterToolCall` hook leaves the main skill-called-as-tool case with the generic error.
  - confidence: 0.94
- **[P2] Match only skills visible in this run:** `src/agents/pi-embedded-runner/run/attempt.ts:1956-1957`
  - body: Current main clears the skill snapshot for sandboxed non-read-write runs before building runtime state, but this rewrite checks `params.skillsSnapshot`. A guessed hidden skill name could receive a confirmation hint even though the skill was not visible to the model in that run.
  - confidence: 0.9
- **[P2] Handle snapshots without resolvedSkills:** `src/agents/pi-embedded-helpers/errors.ts:572-578`
  - body: `resolvedSkills` is optional while lightweight `skills` is required, and the runner has reload logic for missing resolved entries. Returning false when `resolvedSkills` is absent can suppress the hint for sessions that still know about prompt-visible skills.
  - confidence: 0.86
- **[P3] Parse prompt-facing skill names:** `src/agents/pi-embedded-helpers/errors.ts:557-558`
  - body: The parser only captures `[a-z0-9_-]+`, but prompt-facing skill names may include spaces, punctuation, or a leading `$`. Those valid visible skill names will miss the new hint even when the tool error names them.
  - confidence: 0.8
