## Summary

Keep open: the requested skill hint is useful and not on current main, but this PR is not mergeable as-is because it targets a stale Pi API, misses the current missing-tool result path, and can confirm hidden skill names.

## What This Changes

The branch modifies agent error formatting and embedded-runner tool-call handling to rewrite unknown-tool errors that match registered skills into SKILL.md guidance.

## Best Possible Solution

Implement the hint at OpenClaw's current missing-tool/result boundary using run-visible skills, full prompt-facing names, collision preservation, and focused regression coverage.

## Reproduction Assessment

Yes, source-based. Current main still has generic unknown-tool handling, and the current Pi dependency returns missing-tool results before afterToolCall finalization, matching the path this PR is trying to improve.

## Solution Assessment

No. The behavior is useful, but this implementation should be moved from a stale private afterToolCall setter to the current missing-tool/result boundary and should use only skills visible in the run.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Use the current Pi package API:** `src/agents/pi-embedded-runner/run/attempt.ts:4-10`
  - body: Current main pins @earendil-works/pi-agent-core@0.74.0, whose Agent exposes `afterToolCall` as an option/property and has no `setAfterToolCall` method. Importing `@mariozechner/pi-agent-core` types and calling the setter will fail after rebasing onto main.
  - confidence: 0.96
- **[P2] Route missing-tool results through the hint:** `src/agents/pi-embedded-runner/run/attempt.ts:1937`
  - body: In the current Pi loop, an unknown tool returns an immediate `Tool <name> not found` result from `prepareToolCall`, and that path is emitted without `finalizeExecutedToolCall` or `afterToolCall`. Installing the rewrite only as an afterToolCall hook leaves the main skill-called-as-tool case unchanged.
  - confidence: 0.94
- **[P2] Match only skills visible in this run:** `src/agents/pi-embedded-runner/run/attempt.ts:1953-1954`
  - body: Current main clears `skillsSnapshotForRun` for sandboxed non-rw runs before building the prompt/runtime state, but this rewrite checks `params.skillsSnapshot`. A guessed hidden skill name could receive a confirmation hint even though it was not visible to the model in that run.
  - confidence: 0.89
- **[P2] Handle snapshots without resolvedSkills:** `src/agents/pi-embedded-helpers/errors.ts:575-578`
  - body: `resolvedSkills` is runtime-only and may be absent from persisted snapshots while the runner still reloads skill entries for the run. Returning false when `resolvedSkills` is missing can suppress the hint for sessions that still have prompt-visible skills.
  - confidence: 0.86
- **[P3] Parse full prompt-facing skill names:** `src/agents/pi-embedded-helpers/errors.ts:557-558`
  - body: `resolveUnknownToolName()` only captures `[a-z0-9_-]+`, but skill names rendered to the prompt can contain spaces or punctuation. Valid skills such as `GitHub Actions` would be truncated and fail to receive the hint.
  - confidence: 0.82
