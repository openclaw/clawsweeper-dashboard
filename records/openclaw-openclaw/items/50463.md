## Summary

Keep open: current main still returns the generic missing-tool message, and this PR is a useful direction, but the proposed hook does not fit the current Pi agent-core contract and can cross the run-visible skill boundary.

## What This Changes

The branch extracts unknown-tool parsing and adds a Pi embedded runner `afterToolCall` rewrite that turns registered-skill missing-tool errors into `SKILL.md` guidance.

## Best Possible Solution

Port the hint to the current missing-tool result boundary, compose with the supported hook surface, match only run-visible skills, and add focused coverage for hidden, non-skill, punctuation, persisted-snapshot, and tool-collision cases.

## Reproduction Assessment

Yes, source-based. Current main has a generic unknown-tool path with no skill hint, and the PR body screenshot demonstrates the intended after-fix behavior for skill and non-skill cases.

## Solution Assessment

No. The UX direction is useful, but the patch should be moved to the current missing-tool preparation boundary and should use the run-visible skills set plus the supported `afterToolCall` API shape.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Route skill hints through missing-tool preparation:** `src/agents/pi-embedded-runner/run/attempt.ts:1940`
  - body: The pinned `@earendil-works/pi-agent-core@0.74.0` returns unknown tools as `kind: "immediate"` and emits those results without calling `finalizeExecutedToolCall`, where `afterToolCall` runs. A hook installed here will miss the primary `Tool X not found` path this feature is meant to improve.
  - confidence: 0.92
- **[P2] Use the current afterToolCall API:** `src/agents/pi-embedded-runner/run/attempt.ts:1940`
  - body: Current main’s `Agent` exposes `afterToolCall` as a public property and has no `setAfterToolCall` method. After this branch is ported to current main, this call will fail type checking or runtime unless the existing public hook is composed through the supported property.
  - confidence: 0.9
- **[P2] Match only skills visible in this run:** `src/agents/pi-embedded-runner/run/attempt.ts:1956`
  - body: The runner clears `skillsSnapshotForRun` for sandboxed non-rw runs before exposing skills, but this passes the full `params.skillsSnapshot`. A guessed hidden skill name could be confirmed, and persisted snapshots without `resolvedSkills` can also miss skills that were loaded for the prompt.
  - confidence: 0.88
- **[P3] Parse the full prompt-facing skill name:** `src/agents/pi-embedded-helpers/errors.ts:560-561`
  - body: `resolveUnknownToolName()` captures only `[a-z0-9_-]+`, while OpenClaw renders raw skill names in `<available_skills>` and separately sanitizes slash-command names. Valid skills such as `GitHub Actions` or `$openclaw-pr-maintainer` can be truncated or missed, so the new hint silently fails for names the model can see.
  - confidence: 0.82
