## Summary

Keep open. The requested skill-specific unknown-tool hint is not on current main, but this PR needs changes because it wires the rewrite through the wrong current runner boundary and can expose hidden skills in sandboxed runs.

## What This Changes

The branch extracts unknown-tool parsing and adds a Pi embedded runner `afterToolCall` rewrite that turns missing-tool results matching registered skills into `SKILL.md` guidance.

## Best Possible Solution

Implement the hint at the current missing-tool result boundary, preserve existing hook behavior, use only run-visible skills, avoid tool-name collision rewrites, and add focused regression coverage for visible, hidden, non-skill, punctuation, and collision cases.

## Reproduction Assessment

Yes, source-based. Current main has no skill-specific `SKILL.md` hint, and the pinned agent-loop source creates a generic `Tool X not found` result for unknown tool calls.

## Solution Assessment

No. The UX direction is reasonable, but the PR should be moved to the current missing-tool boundary and should use the run-visible skill set instead of a full persisted snapshot.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Route skill hints through the missing-tool boundary:** `src/agents/pi-embedded-runner/run/attempt.ts:1925-1938`
  - body: The pinned agent loop returns unknown tools as immediate `Tool X not found` results from `prepareToolCall`, and those results do not enter `finalizeExecutedToolCall`, where `afterToolCall` runs. Because this PR only installs an `afterToolCall` rewrite, the common path it targets can still receive the generic missing-tool result.
  - confidence: 0.9
- **[P2] Compose with the current public hook surface:** `src/agents/pi-embedded-runner/run/attempt.ts:1935-1938`
  - body: Current `Agent` exposes `afterToolCall` as a public mutable property that is passed into the loop config; the dependency source does not expose this private `_afterToolCall`/`setAfterToolCall` surface. After rebase, this should compose through the current public hook path instead of relying on a private field.
  - confidence: 0.86
- **[P2] Use the run-visible skills snapshot:** `src/agents/pi-embedded-runner/run/attempt.ts:1954-1955`
  - body: Sandboxed non-rw runs clear `skillsSnapshotForRun` before exposing skills to the model, but this hook checks `params.skillsSnapshot`. A guessed hidden skill name can therefore be confirmed by the new hint even when that skill was not visible in the run.
  - confidence: 0.88
- **[P3] Parse the full prompt-facing skill name:** `src/agents/pi-embedded-helpers/errors.ts:555-559`
  - body: The helper only captures `[a-z0-9_-]+`, while OpenClaw renders raw skill names in the available-skills prompt. Valid skill names containing spaces or punctuation can be truncated or missed before matching, so they will not get the intended hint.
  - confidence: 0.8
