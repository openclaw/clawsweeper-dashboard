## Summary

Keep open: the requested behavior is useful and has real-session proof, but the current patch is incorrect against current main because it hooks a path that the pinned agent-core does not use for missing-tool results and it matches against the full persisted skill snapshot rather than the run-visible skill set.

## What This Changes

The PR extracts unknown-tool parsing and adds an embedded-runner hook that rewrites missing-tool results matching registered skills into `SKILL.md` guidance.

## Best Possible Solution

Implement the hint at an OpenClaw-owned missing-tool result boundary, using the same run-visible skill set that builds the prompt, full prompt-facing skill-name parsing, and focused regression coverage.

## Reproduction Assessment

Yes, source-based. Current main has no `SKILL.md first` rewrite, and the pinned agent-core returns missing tools as immediate prepare results that bypass `afterToolCall`; the PR screenshot shows the intended after-fix behavior on the branch.

## Solution Assessment

No. The direction is good, but this patch should move off the stale/private hook boundary and match only skills visible to the current run.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Route skill hints through missing-tool results:** `src/agents/pi-embedded-runner/run/attempt.ts:1941`
  - body: The current pinned `@earendil-works/pi-agent-core@0.74.0` returns `Tool X not found` from `prepareToolCall()` as an immediate result and emits it without calling `finalizeExecutedToolCall()` or `afterToolCall`. Because this patch rewrites only from `afterToolCall`, the main skill-as-unknown-tool case still returns the generic error on current main.
  - confidence: 0.92
- **[P2] Use the current agent-core hook surface:** `src/agents/pi-embedded-runner/run/attempt.ts:1937-1941`
  - body: Current main depends on `@earendil-works/pi-agent-core@0.74.0`, whose `Agent` exposes a public `afterToolCall` property and no public `setAfterToolCall` or private `_afterToolCall` contract. Rebase this onto the current API or avoid the hook entirely so the patch compiles and preserves existing runtime behavior.
  - confidence: 0.9
- **[P2] Match only run-visible skills:** `src/agents/pi-embedded-runner/run/attempt.ts:1955`
  - body: The run clears `skillsSnapshotForRun` for non-rw sandbox sessions before exposing skills, but this hook checks the full `params.skillsSnapshot`. A guessed hidden skill name can therefore be confirmed by the hint even when that skill was not visible in this run.
  - confidence: 0.88
- **[P3] Parse complete prompt-facing skill names:** `src/agents/pi-embedded-helpers/errors.ts:561-562`
  - body: `resolveUnknownToolName()` captures only `[a-z0-9_-]+`, while OpenClaw renders raw skill names and supports names with spaces or punctuation. A valid skill such as `GitHub Actions` is parsed as `GitHub`, so the new hint silently misses those skills.
  - confidence: 0.82
