## Summary

Keep open: the requested behavior is useful and the PR has visible proof, but the patch is not correct against current main because it wires the hint through a stale Pi API and a lifecycle hook that current missing-tool results bypass.

## What This Changes

The PR adds a helper and embedded-runner hook to rewrite unknown-tool tool results into a SKILL.md hint when the missing tool name matches a registered skill.

## Best Possible Solution

Implement the hint on the current Pi immediate missing-tool result path while preserving existing tool-result/plugin hooks and add focused regression coverage.

## Reproduction Assessment

Yes. Source inspection against current main and the pinned Pi package shows a registered-skill-as-tool call still reaches the immediate `Tool X not found` path, and this patch's `afterToolCall` hook does not run for that path.

## Solution Assessment

No. The intent is sound, but the proposed hook uses a stale package API and sits after the current immediate-result boundary; the maintainable fix should rewrite the generated missing-tool result where it is created, emitted, or otherwise guaranteed to pass through a supported current API.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Route missing-tool results through the skill hint:** `src/agents/pi-embedded-runner/run/attempt.ts:1940`
  - body: The new hook is installed through `afterToolCall`, but current `@earendil-works/pi-agent-core@0.74.0` returns missing tools as `kind: "immediate"` results and emits them without calling `finalizeExecutedToolCall`, which is where `afterToolCall` runs. A model calling a skill name as a nonexistent tool will still get the generic `Tool X not found` result unless the rewrite moves to the immediate-result path or a dependency API that explicitly covers immediate results.
  - confidence: 0.94
- **[P2] Rebase the hook onto the current Pi API:** `src/agents/pi-embedded-runner/run/attempt.ts:3-8`
  - body: This branch imports `AfterToolCall*` from `@mariozechner/pi-agent-core` and calls `activeSession.agent.setAfterToolCall`, but current main depends on `@earendil-works/pi-agent-core@0.74.0`, whose `Agent` type exposes `afterToolCall` but no `setAfterToolCall` method. As written, the rebased patch will not typecheck against current main and should use the current package namespace/API.
  - confidence: 0.9
