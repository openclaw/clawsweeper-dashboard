## Summary

Keep open: the idea is useful and the PR has real-session proof, but the branch is stale against current main and the implementation misses the current missing-tool path while also risking hidden-skill disclosure.

## What This Changes

The PR extracts unknown-tool parsing and adds an embedded-runner `afterToolCall` rewrite that turns missing-tool errors matching registered skills into SKILL.md guidance.

## Best Possible Solution

Move the hint to the current missing-tool result boundary or an approved Pi API seam, compose with existing tool-result middleware, match only run-visible skills by full prompt-facing name, and add focused helper/runner coverage.

## Reproduction Assessment

Yes, source-based. Current main has no `SKILL.md first` rewrite, and the inspected PR screenshot shows the branch can produce the intended skill-match and non-skill control outputs.

## Solution Assessment

No. The user-facing direction is good, but this hook is the wrong boundary for current main and should be reworked around the current missing-tool contract and run-visible skill snapshot.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Route skill hints through missing-tool preparation:** `src/agents/pi-embedded-runner/run/attempt.ts:1937`
  - body: Current `@earendil-works/pi-agent-core@0.74.0` returns a missing tool as an immediate preparation result, and that path is emitted without `finalizeExecutedToolCall`, where `afterToolCall` runs. Since this hook only rewrites from `afterToolCall`, the normal skill-called-as-tool case still returns the generic error on current main.
  - confidence: 0.91
- **[P2] Use the current afterToolCall API:** `src/agents/pi-embedded-runner/run/attempt.ts:1935-1938`
  - body: Current `Agent` exposes `afterToolCall` as a public property and has no public `setAfterToolCall` or `_afterToolCall` composition contract. This patch reads an old private field and calls a method absent from the pinned dependency, so it will not compose with the current session/tool-result middleware after rebase.
  - confidence: 0.9
- **[P2] Match only skills visible in this run:** `src/agents/pi-embedded-runner/run/attempt.ts:1955`
  - body: Sandboxed non-rw runs clear `skillsSnapshotForRun` before exposing skills, but the new rewrite checks the full `params.skillsSnapshot`. A guessed hidden skill name can then be confirmed by the hint even though that skill was not visible to the model in this run.
  - confidence: 0.88
- **[P3] Parse complete prompt-facing skill names:** `src/agents/pi-embedded-helpers/errors.ts:556-559`
  - body: The helper only captures `[a-z0-9_-]+`, while the prompt renders raw skill names and OpenClaw supports names with `$`, spaces, `/`, and other punctuation. Valid names like `$openclaw-pr-maintainer` or `GitHub Actions` are truncated or missed, so the hint fails for real prompt-facing skills.
  - confidence: 0.82
