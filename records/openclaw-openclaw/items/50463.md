## Summary

Keep this PR open: the feature is useful and has real after-fix proof, but the current branch is not correct against current main because it wires the hint through a stale/private Pi hook path, misses the dependency-owned missing-tool path, and can disclose skills that were hidden for the run.

## What This Changes

The PR extracts unknown-tool parsing and adds an embedded-runner hook that rewrites missing-tool errors matching registered skills into SKILL.md guidance.

## Best Possible Solution

Implement the hint at an approved current missing-tool/result boundary using only run-visible skills, with focused coverage for hidden skills, tool/skill collisions, and punctuation-bearing skill names.

## Reproduction Assessment

Yes, source-based. Current main lacks the SKILL.md hint, the PR screenshot shows after-fix CLI-visible text plus a non-skill control, and the blocking regressions are visible from current main and the pinned dependency source.

## Solution Assessment

No. The direction is useful, but this patch is not the best implementation because it relies on a stale/private Pi hook surface, misses the current dependency's prepare-time missing-tool path, and uses the full skill snapshot instead of the run-visible skill set.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Route skill hints through the missing-tool result path:** `src/agents/pi-embedded-runner/run/attempt.ts:1940`
  - body: Current @earendil-works/pi-agent-core 0.74.0 creates `Tool X not found` as a `prepareToolCall` immediate result and only invokes `afterToolCall` from `finalizeExecutedToolCall` for prepared tools. Wiring this only through `setAfterToolCall` means the skill-called-as-tool case still returns the generic missing-tool text on current main.
  - confidence: 0.92
- **[P2] Use the current Pi hook API:** `src/agents/pi-embedded-runner/run/attempt.ts:1937-1940`
  - body: The branch reads `_afterToolCall` and calls `activeSession.agent.setAfterToolCall`, but current main's pinned Pi Agent exposes a public `afterToolCall` property and no `setAfterToolCall` method. This will not typecheck or compose correctly after rebase.
  - confidence: 0.9
- **[P2] Match only run-visible skills:** `src/agents/pi-embedded-runner/run/attempt.ts:1954-1957`
  - body: The rewrite checks `params.skillsSnapshot` even though current main deliberately clears `skillsSnapshotForRun` for sandboxed non-rw runs before exposing skills. A guessed hidden skill name can therefore be confirmed by the hint even when that skill was not visible to the model for this run.
  - confidence: 0.88
- **[P3] Parse complete prompt-facing skill names:** `src/agents/pi-embedded-helpers/errors.ts:558-562`
  - body: The helper captures only `[a-z0-9_-]+`, while OpenClaw renders raw skill names in the available_skills prompt and sanitizes only separate slash-command names. Valid visible skills containing `$`, spaces, or other punctuation are truncated or missed, so the new hint silently fails for those names.
  - confidence: 0.82
