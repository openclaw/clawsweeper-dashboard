## Summary

Keep open: the UX direction is useful, but the PR head is stale against current main's Pi dependency and wires the rewrite through a hook path that misses current immediate missing-tool results; it also checks the full skill snapshot instead of the run-visible snapshot.

## What This Changes

The branch refactors unknown-tool parsing and adds an embedded-runner afterToolCall rewrite that turns unknown-tool errors matching registered skills into SKILL.md guidance.

## Best Possible Solution

Implement the hint at the current OpenClaw-owned missing-tool/result boundary using only run-visible skills, preserving tool-name collisions and full prompt-facing skill names, with focused regression coverage.

## Reproduction Assessment

Yes, source-based. Current main still has only generic unknown-tool handling in the agent path, the PR screenshot shows the desired after-fix text, and the current Pi dependency source shows why this branch's hook wiring misses immediate missing-tool results.

## Solution Assessment

No. The product direction is narrow and useful, but this patch uses a stale/private Pi hook surface, misses the current missing-tool path, and should instead use the current result boundary with run-visible skill data.

Full review comments:

- **[P1] Use the current Pi package and hook API:** `src/agents/pi-embedded-runner/run/attempt.ts:4-10`
  - body: Current main pins `@earendil-works/pi-agent-core@0.74.0`, whose `Agent` exposes `afterToolCall` as an option/property and has no `setAfterToolCall` method. This branch imports the old `@mariozechner` types and calls `setAfterToolCall`, so it will not typecheck or compose after rebasing.
  - confidence: 0.94
- **[P2] Route immediate missing-tool results through the hint:** `src/agents/pi-embedded-runner/run/attempt.ts:1940`
  - body: In the pinned Pi loop, an unknown tool returns an immediate `Tool ... not found` result from `prepareToolCall`; `afterToolCall` only runs for prepared/executed tools. Installing this rewrite only as an `afterToolCall` hook leaves the main skill-called-as-tool case unchanged.
  - confidence: 0.92
- **[P2] Match only run-visible skills:** `src/agents/pi-embedded-runner/run/attempt.ts:1956`
  - body: Current main clears `skillsSnapshotForRun` for sandboxed non-rw runs before building the prompt/runtime state, but this rewrite checks `params.skillsSnapshot`. A model that guesses a hidden skill name could get confirmation that it exists even though it was not visible in the run.
  - confidence: 0.88
- **[P3] Parse complete prompt-facing skill names:** `src/agents/pi-embedded-helpers/errors.ts:558-561`
  - body: The new helper captures only `[a-z0-9_-]+`, while prompt-facing skill names come from raw `skill.name` and may include spaces, slashes, a leading `$`, or other punctuation. Those valid visible skills would be truncated or missed, so the hint silently fails for part of the supported skill surface.
  - confidence: 0.8
