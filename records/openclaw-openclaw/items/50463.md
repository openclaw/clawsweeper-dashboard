## Summary

Keep open. The requested recovery hint is still a plausible agent UX improvement, but this branch is not mergeable against current main because it uses an unsupported pi-agent hook API and the chosen `afterToolCall` path does not see the pinned dependency's immediate missing-tool result.

## What This Changes

The PR extracts unknown-tool parsing, adds a skill-aware tool-result rewrite helper, and attempts to install an embedded-runner `afterToolCall` hook that replaces matching `Tool not found` messages with a `SKILL.md` hint.

## Best Possible Solution

Keep the feature direction, but implement it through a supported missing-tool/immediate-result path for the pinned pi-agent packages, compose with the existing `AgentSession` tool-result hook, define visible-skill and tool-collision semantics explicitly, and add focused helper/runner regression coverage plus a user-facing changelog entry before merge.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: current main has no skill-aware unknown-tool rewrite, and the pinned pi-agent source shows `Tool X not found` is created as an immediate prepare-time result that does not run `afterToolCall`.

## Solution Assessment

No. This PR is not the best way to solve the issue because it calls a nonexistent setter and relies on a hook path that misses the main missing-tool result; a supported immediate-result seam or upstream hook contract is safer.
