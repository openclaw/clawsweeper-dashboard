## Summary

Keep open: the feature direction is useful, but this PR is not correct against current main because it hooks a stale/private `afterToolCall` boundary, misses the current missing-tool path, and can consult skills that were intentionally hidden from the run.

## What This Changes

The PR extracts unknown-tool parsing and adds an embedded-runner `afterToolCall` rewrite that turns missing-tool errors matching registered skills into SKILL.md guidance.

## Best Possible Solution

Implement the hint at the current missing-tool result boundary or an approved Pi seam, using only run-visible skills and full prompt-facing skill names, with focused helper and runner coverage.

## Reproduction Assessment

Yes, source-based. Current main has no `SKILL.md first` rewrite, the PR screenshot shows the intended branch behavior, and the pinned dependency source shows the common missing-tool path does not pass through this hook.

## Solution Assessment

No. The user-facing direction is reasonable, but the current patch uses the wrong runtime boundary and the wrong skill visibility source; the safer path is a missing-tool boundary or approved dependency seam.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Route missing-tool results through the rewrite:** `src/agents/pi-embedded-runner/run/attempt.ts:1940`
  - body: The pinned `@earendil-works/pi-agent-core@0.74.0` returns `Tool X not found` as an immediate prepare result, and those immediate branches are emitted without calling `finalizeExecutedToolCall`, where `afterToolCall` runs. Since this patch only rewrites from `afterToolCall`, the normal skill-called-as-tool case still returns the generic error on current main.
  - confidence: 0.92
- **[P2] Use the current Agent hook API:** `src/agents/pi-embedded-runner/run/attempt.ts:1937-1940`
  - body: Current main pins a dependency whose `Agent` exposes `afterToolCall` as a public property and has no public `setAfterToolCall` or `_afterToolCall` composition contract. After rebase, this call will not compile or compose with the current session middleware.
  - confidence: 0.9
- **[P2] Match only skills visible in this run:** `src/agents/pi-embedded-runner/run/attempt.ts:1954-1957`
  - body: Sandboxed non-rw runs clear the run-visible skill snapshot before prompt exposure, but the rewrite checks `params.skillsSnapshot`. A guessed hidden skill name can therefore be confirmed by the hint even though the model was not shown that skill in this run.
  - confidence: 0.88
- **[P3] Parse complete prompt-facing skill names:** `src/agents/pi-embedded-helpers/errors.ts:558-562`
  - body: The helper only captures `[a-z0-9_-]+`, while OpenClaw renders raw `skill.name` values in the prompt and supports names with punctuation or spaces. Valid visible names such as `$openclaw-pr-maintainer` or `GitHub Actions` will be truncated or missed, so the hint silently fails for supported skills.
  - confidence: 0.82
