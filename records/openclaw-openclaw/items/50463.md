## Summary

Keep open: the idea is useful and the contributor provided screenshot proof, but the current patch is not correct against the pinned agent-core contract because the hook is installed on `afterToolCall` while missing-tool results are returned as immediate prepare-time errors that bypass that hook.

## What This Changes

The PR adds an agent hook and helper to rewrite unknown-tool tool-result errors into SKILL.md hints when the missing tool name matches a registered skill, while skipping names that collide with allowed tools.

## Best Possible Solution

Route the rewrite through the actual missing-tool immediate-result path, or update to a dependency contract where `afterToolCall` covers immediate missing-tool results, and base the match on the effective run-visible skill set with focused regression coverage.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main plus `@earendil-works/pi-agent-core@0.74.0` returns generic missing-tool results from `prepareToolCall`, and the PR hook is not on that path. I did not run a live model session.

## Solution Assessment

No. Installing only an `afterToolCall` rewrite is not the best fix for the stated behavior while the pinned dependency bypasses that hook for missing tools; the rewrite needs to happen where the missing-tool result is created or after a verified dependency update.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Route missing-tool results through a path that actually runs:** `src/agents/pi-embedded-runner/run/attempt.ts:1936`
  - body: The hook is installed through `afterToolCall`, but the pinned `@earendil-works/pi-agent-core@0.74.0` returns missing tools as immediate `prepareToolCall` results and only invokes `afterToolCall` from `finalizeExecutedToolCall` for prepared calls. That means the target case, a model calling a skill name as an unavailable tool, still gets the generic `Tool X not found` result. Please move the rewrite onto the immediate missing-tool path or land it with a dependency version whose contract explicitly runs `afterToolCall` for those results.
  - confidence: 0.93
- **[P2] Use the effective visible skill set for hints:** `src/agents/pi-embedded-runner/run/attempt.ts:1955`
  - body: Earlier in the runner, `skillsSnapshotForRun` can be set to `undefined` when sandbox settings intentionally hide workspace skills, and the skills prompt is built from that effective value. The hook passes `params.skillsSnapshot` instead, so once the hook runs it can tell the model that a hidden skill is available. Use the same effective run-visible skill source that built the prompt, and cover the sandbox/no-`resolvedSkills` cases in tests.
  - confidence: 0.86
