## Summary

Keep open: the feature direction is useful, but this branch is stale against current `main` and introduces runtime/security regressions by wiring the hint through a private/stale Pi hook and matching against the full skill snapshot.

## What This Changes

The branch extracts unknown-tool parsing and adds an embedded-runner `afterToolCall` rewrite that turns missing-tool errors matching registered skills into SKILL.md guidance.

## Best Possible Solution

Implement the hint at the current missing-tool/result boundary using only run-visible skills, while preserving tool collisions and covering punctuation-bearing skill names.

## Reproduction Assessment

Yes, source-based. Current main still has only generic unknown-tool handling, and the PR screenshot demonstrates the intended after-fix text plus a non-skill control.

## Solution Assessment

No. The direction is useful, but this patch is not the best implementation because it depends on a stale/private Pi hook surface and checks the full skill snapshot instead of the run-visible skill set.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Use the current Pi hook API:** `src/agents/pi-embedded-runner/run/attempt.ts:1937-1940`
  - body: Current main pins `@earendil-works/pi-agent-core@0.74.0`, whose `Agent` exposes an `afterToolCall` property and no `setAfterToolCall` method. This branch reads `_afterToolCall` and calls `setAfterToolCall`, so it will not typecheck or compose after rebasing onto the current dependency.
  - confidence: 0.92
- **[P2] Route the hint through the missing-tool path:** `src/agents/pi-embedded-runner/run/attempt.ts:1940`
  - body: In the pinned Pi loop, unknown tools return an immediate `Tool ... not found` result from `prepareToolCall`, and `afterToolCall` only runs for prepared/executed tools. Because the rewrite is only installed through `afterToolCall`, the main skill-called-as-tool case still gets the generic missing-tool result on current main.
  - confidence: 0.91
- **[P2] Match only run-visible skills:** `src/agents/pi-embedded-runner/run/attempt.ts:1956`
  - body: Current main clears the run-visible skill snapshot for sandboxed non-rw runs, but this rewrite checks `params.skillsSnapshot`. A model that guesses a hidden skill name could receive confirmation that it exists even though it was not visible in the run.
  - confidence: 0.88
- **[P3] Parse complete prompt-facing skill names:** `src/agents/pi-embedded-helpers/errors.ts:558-561`
  - body: Skill names are rendered from `skill.name`, including nested names like `name/subskill`, but the helper only captures `[a-z0-9_-]+`. Valid visible skills with punctuation or spaces can be truncated or missed, so the hint silently fails for those names.
  - confidence: 0.82
