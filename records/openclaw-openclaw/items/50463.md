## Summary

Keep this PR open: the user-facing hint is useful and the screenshot proof is credible, but the proposed hook path does not match current main's pinned agent-core contract and still misses the target missing-tool result path.

## What This Changes

The PR adds agent error rewriting so unknown-tool results that match registered skills tell the model to read the skill's SKILL.md.

## Best Possible Solution

Rework the PR against current @earendil agent-core so the hint is applied on the immediate missing-tool result path and matched against the same prompt-visible skill set used for the run.

## Reproduction Assessment

Yes. Source inspection shows current main returns generic Tool X not found for missing tools through @earendil-works/pi-agent-core 0.74.0, and current OpenClaw has no SKILL.md hint path for that result.

## Solution Assessment

No. A hook-only implementation is not the best fix on current main because the pinned dependency skips afterToolCall for immediate missing-tool results; the rewrite should target that supported path or land with an updated dependency contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Route missing-tool hints through the current result path:** `src/agents/pi-embedded-runner/run/attempt.ts:1937`
  - body: Current main pins @earendil-works/pi-agent-core@0.74.0, where missing tools are immediate prepare results and do not pass through afterToolCall; the Agent API also exposes afterToolCall as a property, not setAfterToolCall/_afterToolCall. Once the missing-tool path is wired correctly, this can suppress hints for skills the model actually saw, so pass or derive the effective prompt-visible skill set instead.
  - confidence: 0.84
