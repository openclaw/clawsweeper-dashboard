## Summary

Keep this PR open because the user-facing hint is useful and the PR includes real behavior proof, but the branch is not correct against current main: it targets stale Pi APIs, misses the current missing-tool path, and can match skills outside the active run boundary.

## What This Changes

The PR adds a skill-aware rewrite for unknown-tool errors so a tool call matching a registered skill tells the model to read that skill's SKILL.md.

## Best Possible Solution

Implement the hint at the current OpenClaw-owned missing-tool result boundary, using active-run visible skills, robust quoted-name parsing, current @earendil contracts, and focused regression coverage.

## Reproduction Assessment

Yes, source-based. Current main still has generic unknown-tool handling with no SKILL.md hint, and the PR screenshot shows the intended after-fix skill-match and non-skill control behavior.

## Solution Assessment

No. The feature direction is good, but this branch hooks stale/private Pi APIs and misses the current immediate missing-tool path; the safer fix is a focused current-main repair at the missing-tool boundary.

- **[P1] Use the current Pi agent API:** `src/agents/pi-embedded-runner/run/attempt.ts:4-10`
  - body: Current main uses @earendil-works/pi-agent-core and no longer depends on @mariozechner/pi-agent-core. This branch imports @mariozechner hook types and calls setAfterToolCall/_afterToolCall, so it will not typecheck or compose correctly after rebase.

- **[P2] Route immediate missing-tool results through the rewrite:** `src/agents/pi-embedded-runner/run/attempt.ts:1925-1941`
  - body: In the current @earendil-works pi-agent-core, missing tools return a kind:"immediate" error from prepareToolCall and do not run afterToolCall. Wiring the hint only through afterToolCall leaves the main Tool X not found case generic.
  - confidence: 0.94
- **[P2] Match only skills visible in the active run:** `src/agents/pi-embedded-runner/run/attempt.ts:1955-1957`
  - body: The runner clears or narrows skillsSnapshotForRun before building the active prompt, but this hook checks params.skillsSnapshot. A guessed skill name can receive a SKILL.md hint even when that skill was hidden from the active run.
  - confidence: 0.9
- **[P2] Handle snapshots without resolved runtime skills:** `src/agents/pi-embedded-helpers/errors.ts:573-577`
  - body: resolvedSkills is optional and current runner reloads workspace skill entries when it is absent. Requiring resolvedSkills here suppresses the hint for persisted snapshots that still advertise skills through the lightweight skills array.
  - confidence: 0.84
- **[P3] Parse the full quoted skill name:** `src/agents/pi-embedded-helpers/errors.ts:557-560`
  - body: The parser captures only [a-z0-9_-]+, but prompt-facing skill names can contain spaces, punctuation, or a leading $. Those valid names are truncated or ignored, so the hint will not fire for common skill spellings.
  - confidence: 0.8
