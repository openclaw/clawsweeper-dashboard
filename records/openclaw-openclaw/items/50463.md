## Summary

Keep this PR open for maintainer follow-up. Current main still lacks the requested skill-aware recovery, but the proposed branch is not correct against current main's pinned pi-agent packages and needs request-changes rather than cleanup closure.

## What This Changes

The PR adds an unknown-tool error rewrite helper and attempts to install an embedded-runner `afterToolCall` hook that tells the model when a missing tool name matches a visible skill.

## Best Possible Solution

Keep the feature direction, but implement it through a supported missing-tool result path for the pinned pi-agent packages, compose with existing plugin tool-result hooks, define visible-skill and collision semantics explicitly, and add targeted regression tests before merge.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: current main has no skill-aware unknown-tool rewrite, and the pinned dependency source shows the target `Tool X not found` path is an immediate prepare-time result.

## Solution Assessment

No. The current PR is not the best way to solve it because it calls a nonexistent Agent setter and depends on an `afterToolCall` path that the pinned dependency does not run for missing tools; a supported immediate-result seam is safer.
