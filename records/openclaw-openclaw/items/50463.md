## Summary

Keep open: the requested hint is still useful and not present on current main, but this branch is not mergeable as-is because it targets stale Pi APIs, hooks the wrong missing-tool boundary for current dependencies, and can match skills outside the active run.

## What This Changes

The PR adds a skill-aware rewrite for unknown-tool errors so calls to registered skill names receive a SKILL.md hint instead of the generic tool-not-found text.

## Best Possible Solution

Implement the hint on current main at the OpenClaw-owned missing-tool result boundary using @earendil contracts, run-visible skills, robust quoted-name parsing, collision preservation, and focused regression tests.

## Reproduction Assessment

Yes, source-based. Current main still returns generic unknown-tool handling with no SKILL.md hint, and the PR screenshot shows the intended after-fix behavior in a live agent session.

## Solution Assessment

No. The user-facing direction is useful, but this branch is not the best implementation because it targets stale Pi APIs and wires the rewrite through a path that misses current missing-tool results and run-visible skill filtering.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Use the current Pi agent API:** `src/agents/pi-embedded-runner/run/attempt.ts:4-8`
  - body: Current main uses @earendil-works/pi-agent-core 0.74.0, where Agent exposes afterToolCall as an option/property. This branch imports @mariozechner types and calls setAfterToolCall plus private _afterToolCall, so it will not compile or compose after rebase.
  - confidence: 0.96
- **[P2] Apply the hint where missing tools are produced:** `src/agents/pi-embedded-runner/run/attempt.ts:1940`
  - body: In the current @earendil Pi loop, unknown tools are returned as immediate prepare results and afterToolCall only runs for prepared/executed tools. Installing only this hook leaves the normal Tool X not found path generic.
  - confidence: 0.94
- **[P2] Match only skills visible in this run:** `src/agents/pi-embedded-runner/run/attempt.ts:1956-1957`
  - body: The runner can clear or filter the skills snapshot before building the run prompt/runtime state, but the patch checks params.skillsSnapshot. A guessed hidden skill name could still get a SKILL.md hint even when it was removed from the actual run context.
  - confidence: 0.9
- **[P2] Handle snapshots without runtime-resolved skills:** `src/agents/pi-embedded-helpers/errors.ts:573-577`
  - body: resolvedSkills is optional and persisted snapshots can omit it, while the lightweight skills list is required and runtime entries may be rebuilt. Requiring resolvedSkills here can suppress the hint for sessions that still advertise the skill after reload.
  - confidence: 0.84
- **[P3] Parse the full quoted tool name:** `src/agents/pi-embedded-helpers/errors.ts:560-561`
  - body: The parser captures only [a-z0-9_-]+, but prompt-facing skill names can include $ prefixes, spaces, punctuation, or path-like separators. Those valid names are truncated or ignored, so the new hint will not fire for common skill spellings.
  - confidence: 0.8
