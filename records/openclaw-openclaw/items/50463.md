## Summary

Keep open: the requested skill hint is useful and current main still lacks it, but this branch is not merge-ready because it hooks the wrong tool-result boundary for the current dependency contract and can expose skills hidden from the run.

## What This Changes

The PR extracts unknown-tool parsing helpers and adds an embedded-runner hook that rewrites missing-tool error results when the missing name matches a registered skill.

## Best Possible Solution

Implement the hint at the current missing-tool result boundary, preserve existing extension tool-result hooks, use the run-visible skill set, and cover skill match, non-skill, hidden skill, punctuation, and tool-collision cases.

## Reproduction Assessment

Yes, source-based. Current main lacks the SKILL.md hint, and the pinned `pi-agent-core` source shows generic missing-tool results are created as immediate results before `afterToolCall`.

## Solution Assessment

No. The UX direction is reasonable, but this patch wires it through an `afterToolCall` path that current `pi-agent-core` does not run for missing tools and uses the wrong skill snapshot for sandbox-hidden skills.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Route missing-tool results through the immediate path:** `src/agents/pi-embedded-runner/run/attempt.ts:1925-1937`
  - body: The pinned `pi-agent-core` returns `Tool ${toolCall.name} not found` from `prepareToolCall` as an immediate result, and `afterToolCall` only runs from `finalizeExecutedToolCall` for prepared/executed tools. Because this hook is only installed on `afterToolCall`, the skill-as-tool miss this feature targets still keeps the generic error on current main.
  - confidence: 0.91
- **[P2] Compose with the current afterToolCall property:** `src/agents/pi-embedded-runner/run/attempt.ts:1938-1942`
  - body: Current `Agent` exposes `afterToolCall` as a mutable property, and `pi-coding-agent` already installs extension `tool_result` dispatch there. Reading a private `_afterToolCall` field and calling `setAfterToolCall` does not match the current dependency surface and risks dropping the existing extension handler; capture and assign the public property instead where that hook is actually used.
  - confidence: 0.88
- **[P2] Use the run-visible skill snapshot:** `src/agents/pi-embedded-runner/run/attempt.ts:1954`
  - body: The runner clears `skillsSnapshotForRun` before exposing skills in sandboxed non-rw runs, but this hook checks `params.skillsSnapshot`. That can reveal a persisted skill name that was intentionally hidden from this run, so the rewrite needs the same visible skill set used to build `skillsPrompt`.
  - confidence: 0.87
- **[P3] Parse the same skill names the prompt can expose:** `src/agents/pi-embedded-helpers/errors.ts:555-559`
  - body: `resolveUnknownToolName()` only captures `[a-z0-9_-]+`, while OpenClaw renders raw skill names and separately sanitizes slash-command names because they may contain spaces or symbols. Valid skills such as names with spaces, punctuation, or `$` spelling will miss the hint or be truncated before matching.
  - confidence: 0.8
