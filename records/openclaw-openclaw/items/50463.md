## Summary

Keep open: the requested hint is useful and not present on current main, but this patch still targets a hook path that the pinned agent-core dependency bypasses for immediate missing-tool results and it uses the persisted skill snapshot instead of the run-visible skill set.

## What This Changes

The PR adds an embedded-runner unknown-tool error rewrite that tells the model when the unknown tool name is actually a registered skill and to read its SKILL.md.

## Best Possible Solution

Implement the hint at the actual missing-tool result boundary for the current agent-core contract, derive matches from the run-visible skill set, and cover skill match, non-skill, hidden-skill, collision, and broad-name cases with focused regression tests.

## Reproduction Assessment

Yes, source-based. Current main has no skill-specific unknown-tool hint, and the PR screenshot shows the intended after-fix behavior for skill and non-skill calls.

## Solution Assessment

No. The user-visible behavior is good, but this patch should be moved to the actual missing-tool boundary and tied to the run-visible skill set before it is a maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Route immediate missing tools through the hint:** `src/agents/pi-embedded-runner/run/attempt.ts:1940`
  - body: The rewrite is installed only through `setAfterToolCall`, but the pinned `@earendil-works/pi-agent-core@0.74.0` returns missing tools as immediate results and emits those without calling `finalizeExecutedToolCall` or `afterToolCall`. That leaves the main skill-as-tool case on the generic `Tool X not found` message, so the hint needs to run at the immediate missing-tool boundary or after the dependency exposes that hook.
  - confidence: 0.9
- **[P2] Use the run-visible skill set:** `src/agents/pi-embedded-runner/run/attempt.ts:1956`
  - body: This passes `params.skillsSnapshot` into the hint rewrite, but current main clears that snapshot for sandboxed non-rw runs before constructing the prompt-visible skills. Using the persisted snapshot here can advertise a skill name that was intentionally hidden from the current run, so the rewrite should use the same run-visible snapshot or loaded entries used for the prompt.
  - confidence: 0.86
- **[P3] Match the prompted skill-name surface:** `src/agents/pi-embedded-helpers/errors.ts:560-561`
  - body: `resolveUnknownToolName()` only captures `[a-z0-9_-]+`, while OpenClaw renders the raw `skill.name` in the skills prompt and separately sanitizes command names because skill names can contain spaces or symbols. A valid skill such as `GitHub Actions` would be truncated to `GitHub` and never get the hint.
  - confidence: 0.78
