## Summary

Keep open: the requested hint is useful and the screenshot proof is sufficient, but the current patch is not correct against current main because it uses a stale/private Pi hook seam and the full skill snapshot instead of the run-visible skill boundary.

## What This Changes

The PR extracts unknown-tool parsing and adds an embedded-runner `afterToolCall` rewrite that turns missing-tool errors matching registered skills into SKILL.md guidance.

## Best Possible Solution

Implement the hint at an approved missing-tool/immediate-result boundary using only run-visible skills, with focused coverage for hidden skills, tool/skill collisions, and punctuation-bearing skill names.

## Reproduction Assessment

Yes, source-based. Current main has no skill-specific unknown-tool hint, and the pinned Pi dependency source shows the generic `Tool X not found` path; I did not run live behavior because this review is read-only.

## Solution Assessment

No. The direction is useful, but this patch is not the best implementation because it depends on a stale/private hook seam and bypasses the current run-visible skill boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Route missing-tool results before `afterToolCall`:** `src/agents/pi-embedded-runner/run/attempt.ts:1937-1940`
  - body: Current `@earendil-works/pi-agent-core@0.74.0` returns unknown tools as `prepareToolCall` immediate results, and `afterToolCall` only runs from `finalizeExecutedToolCall` for prepared tools. Wiring the rewrite only here means the model-calls-skill-as-tool case still returns the generic missing-tool text after rebase.
  - confidence: 0.92
- **[P2] Use the current Pi hook contract:** `src/agents/pi-embedded-runner/run/attempt.ts:1932-1937`
  - body: Current main's pinned `Agent` exposes a public `afterToolCall` property and no public `setAfterToolCall` method or `_afterToolCall` contract. After rebase this private setter/read will not compose or typecheck against the dependency surface.
  - confidence: 0.9
- **[P2] Match only skills visible in this run:** `src/agents/pi-embedded-runner/run/attempt.ts:1954-1957`
  - body: Sandboxed non-rw runs intentionally clear the run-visible skill snapshot before building the prompt/runtime skill state, but the rewrite checks `params.skillsSnapshot`. A guessed hidden skill name could therefore be confirmed by the hint even though that skill was not visible to the model for this run.
  - confidence: 0.88
- **[P3] Parse complete prompt-facing skill names:** `src/agents/pi-embedded-helpers/errors.ts:558-562`
  - body: The helper captures only `[a-z0-9_-]+`, while OpenClaw renders raw skill names in the prompt and separately sanitizes command names because skills may contain spaces or symbols. Valid visible names such as `$openclaw-pr-maintainer` or `GitHub Actions` are truncated or missed, so the new hint fails for those skills.
  - confidence: 0.82
