## Summary

Keep open. The requested hint is useful and not implemented on current main, but this branch is not ready because it targets the old Pi package/API, installs the rewrite on a hook that current Pi does not call for missing-tool results, and can confirm skills that were hidden from the run.

## What This Changes

The PR rewrites unknown-tool errors matching registered skill names into guidance telling the model to read the skill's SKILL.md.

## Best Possible Solution

Implement the hint at the current OpenClaw-owned missing-tool result boundary using current @earendil-works contracts, run-visible skills, collision preservation, robust skill-name parsing, and focused regression coverage.

## Reproduction Assessment

Yes, source-based. Current main still has generic unknown-tool handling without the SKILL.md hint, and @earendil-works/pi-agent-core@0.74.0 produces missing-tool results through an immediate prepareToolCall path that can be targeted in a regression test.

## Solution Assessment

No. The branch uses a stale private setter path and raw params.skillsSnapshot; the maintainable fix should target current-main missing-tool handling with the run-visible skill state.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Use the current Pi package API:** `src/agents/pi-embedded-runner/run/attempt.ts:4-10`
  - body: Current main pins `@earendil-works/pi-agent-core@0.74.0`, whose `Agent` exposes `afterToolCall` as an option/property and has no `setAfterToolCall` method. This import and setter path are from the old `@mariozechner` packages, so the branch will not compile or compose correctly after rebasing.
  - confidence: 0.96
- **[P2] Route missing-tool results through the hint:** `src/agents/pi-embedded-runner/run/attempt.ts:1939-1940`
  - body: In the current pinned Pi loop, a missing tool returns an immediate `Tool X not found` result from `prepareToolCall`, and `afterToolCall` only runs for prepared/executed tools. Installing the rewrite only through this hook leaves the main skill-called-as-tool case unchanged on current main.
  - confidence: 0.94
- **[P2] Match only skills visible in this run:** `src/agents/pi-embedded-runner/run/attempt.ts:1956-1957`
  - body: Current main clears the skill snapshot for sandboxed non-rw runs before building prompt/runtime state, but this rewrite checks `params.skillsSnapshot`. A guessed hidden skill name could receive a confirmation hint even though it was not visible to the model in that run.
  - confidence: 0.9
- **[P2] Handle snapshots without resolvedSkills:** `src/agents/pi-embedded-helpers/errors.ts:572-577`
  - body: `resolvedSkills` is optional, and the embedded runner reloads skill entries when it is absent. Returning false when this field is missing can suppress the hint for sessions that still have prompt-visible skills from the lightweight snapshot or reloaded entries.
  - confidence: 0.86
- **[P3] Parse prompt-facing skill names:** `src/agents/pi-embedded-helpers/errors.ts:560-561`
  - body: The unknown-tool parser captures only `[a-z0-9_-]+`, while skills can be rendered with raw names containing spaces or punctuation and users may spell skills with a leading `$`. Those visible skills will miss the new hint even when the tool error names them.
  - confidence: 0.82
