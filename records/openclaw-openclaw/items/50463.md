## Summary

Keep open. The UX improvement is useful, and the screenshot proof is adequate for the visible text change, but the current patch is not correct against current main: the pinned agent-core dependency bypasses `afterToolCall` for immediate missing-tool results, and the hint lookup does not use the same effective skill visibility that builds the run prompt.

## What This Changes

The PR adds a helper and `afterToolCall` hook to rewrite unknown-tool errors into SKILL.md hints when the missing tool name matches a registered skill.

## Best Possible Solution

Implement the hint at the actual missing-tool result path, or land it with a verified dependency update that covers immediate missing-tool results, and match against the effective run-visible skill set with focused regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: current main pins `@earendil-works/pi-agent-core@0.74.0`, where missing tools become immediate results that bypass `afterToolCall`, so the PR's hook does not reach the target generic missing-tool message.

## Solution Assessment

No. A hook-only rewrite is not the best fix while the pinned dependency bypasses that hook for missing tools; the safer direction is a supported interception point for immediate missing-tool results plus run-visible skill matching.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Route missing-tool errors through the immediate path:** `src/agents/pi-embedded-runner/run/attempt.ts:1936`
  - body: The hook is installed through `afterToolCall`, but the pinned `@earendil-works/pi-agent-core@0.74.0` returns unknown tools as immediate `prepareToolCall` results and bypasses `finalizeExecutedToolCall`, which is where `afterToolCall` runs. A model that calls a skill name as an unavailable tool will still get the generic `Tool X not found` message unless this rewrite runs on the immediate missing-tool path or lands with a dependency contract that explicitly invokes the hook there.
  - confidence: 0.93
- **[P2] Match against the run-visible skill set:** `src/agents/pi-embedded-runner/run/attempt.ts:1955`
  - body: The hook passes raw `params.skillsSnapshot`, but the runner builds the prompt from `skillsSnapshotForRun` and reloads skill entries when `resolvedSkills` is absent. This can both hint hidden host skills in non-rw sandbox runs and miss visible skills from persisted snapshots, so the rewrite needs to use the same effective skill source that produced the prompt.
  - confidence: 0.86
