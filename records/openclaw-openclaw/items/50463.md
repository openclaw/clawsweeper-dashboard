## Summary

Keep open. The feature direction is useful and the PR has real behavior proof, but the current patch is stale against the Pi dependency surface and can disclose hidden skill names, so it should not merge as-is.

## What This Changes

The branch extracts unknown-tool parsing and adds an embedded-runner `afterToolCall` rewrite that turns missing-tool errors matching registered skills into SKILL.md guidance.

## Best Possible Solution

Implement the hint at an approved current missing-tool/result boundary using only run-visible skills, with focused coverage for hidden skills, tool/skill collisions, and punctuation-bearing skill names.

## Reproduction Assessment

Yes, source-based. Current main lacks the `SKILL.md first` hint, the PR screenshot demonstrates the intended after-fix text plus a non-skill control, and the blocking regressions are visible from current main and the pinned dependency source.

## Solution Assessment

No. The direction is useful, but this patch is not the best implementation because it depends on a stale/private Pi hook surface and uses the full skill snapshot instead of the run-visible skill set.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Use the current Pi hook API:** `src/agents/pi-embedded-runner/run/attempt.ts:1937-1940`
  - body: Current main pins `@earendil-works/pi-agent-core@0.74.0`, whose `Agent` exposes an `afterToolCall` property and no `setAfterToolCall` method, while this branch imports `@mariozechner/pi-agent-core` and reads `_afterToolCall`. After rebase, this hook wiring will not typecheck or compose with the current dependency.
  - confidence: 0.91
- **[P2] Route the hint through missing-tool results:** `src/agents/pi-embedded-runner/run/attempt.ts:1940`
  - body: In the pinned Pi agent loop, unknown tools return an immediate `Tool ... not found` result from `prepareToolCall`, and `afterToolCall` is only invoked for prepared/executed tools. Because this rewrite is only installed through `afterToolCall`, the main skill-called-as-tool case still gets the generic missing-tool result on current main.
  - confidence: 0.9
- **[P2] Match against the run-visible skill snapshot:** `src/agents/pi-embedded-runner/run/attempt.ts:1956-1957`
  - body: Current main clears `skillsSnapshotForRun` for sandboxed non-rw runs before exposing skills, but the rewrite checks `params.skillsSnapshot`. A model that guesses a hidden skill name can receive confirmation that it exists even though that skill was not visible in the run.
  - confidence: 0.88
- **[P3] Parse complete prompt-facing skill names:** `src/agents/pi-embedded-helpers/errors.ts:558-562`
  - body: Skill names are rendered from `skill.name`, including nested names such as `name/subskill` and names with spaces or symbols, but the helper only captures `[a-z0-9_-]+`. Valid visible skills can be truncated or missed, so the hint silently fails for those names.
  - confidence: 0.82
