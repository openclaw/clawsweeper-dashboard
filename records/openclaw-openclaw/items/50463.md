## Summary

Keep open: the goal is useful and the PR includes real after-fix proof, but the patch is not correct for current main because the hook point misses current immediate missing-tool results and still has snapshot/name-matching issues.

## What This Changes

The PR adds a helper that rewrites unknown-tool tool-result errors into a skill-specific SKILL.md hint and installs it from the embedded runner after-tool-call hook.

## Best Possible Solution

Implement the hint at a current-main hook or wrapper that actually sees missing-tool results, use the run-visible skill snapshot, preserve tool-collision behavior, and cover skill match, non-match, sandbox-hidden skills, and broader skill names with focused tests.

## Reproduction Assessment

Yes, source-based. Current main’s pinned Pi loop emits generic immediate Tool X not found results for missing tools, and current OpenClaw source has no skill-specific rewrite on that path.

## Solution Assessment

No. The afterToolCall-only approach is not the narrowest correct fix for current main because immediate missing-tool results bypass that hook; the fix should use a path that sees those results or land the dependency hook first.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Route immediate missing-tool results through the hint:** `src/agents/pi-embedded-runner/run/attempt.ts:1937`
  - body: Current main pins @earendil-works/pi-agent-core@0.74.0, where prepareToolCall returns `Tool X not found` as an immediate result and afterToolCall only runs from finalizeExecutedToolCall for prepared tools. This hook will not see the missing-tool case after rebase, so wire the rewrite into a path that actually receives immediate results or land the dependency hook first.
  - confidence: 0.91
- **[P2] Use the run-visible skill snapshot for hints:** `src/agents/pi-embedded-runner/run/attempt.ts:1954`
  - body: The runner clears `skillsSnapshotForRun` for sandboxed runs that should not expose workspace skills, and that value builds the prompt/env. Passing `params.skillsSnapshot` here can advertise a persisted skill that this run hid, producing a misleading recovery hint.
  - confidence: 0.82
- **[P3] Parse the full registered skill name before matching:** `src/agents/pi-embedded-helpers/errors.ts:556-557`
  - body: `resolveUnknownToolName()` only captures `[a-z0-9_-]+`, but prompt-visible skill names can include spaces or symbols and users may spell them with a leading `$`. Parse quoted tool names more broadly before comparing against visible skills so valid skill-looking calls do not silently miss the hint.
  - confidence: 0.74
