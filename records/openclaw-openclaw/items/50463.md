## Summary

Keep open: the skill-hint behavior is still useful and is not on current main, but this branch needs changes before merge because it relies on the wrong missing-tool boundary, uses the full persisted skill snapshot, and needs current hook/parser alignment.

## What This Changes

The branch extracts unknown-tool parsing and adds a Pi embedded runner `afterToolCall` rewrite that turns missing-tool results matching registered skills into `SKILL.md` guidance.

## Best Possible Solution

Implement the hint at the current missing-tool result boundary, preserve extension `tool_result` hook behavior, match only run-visible skills, and cover visible, hidden, non-skill, punctuation, and collision cases.

## Reproduction Assessment

Yes, source-based. Current main has no skill-specific `SKILL.md` hint, and the pinned agent loop creates generic immediate `Tool X not found` results for missing tool calls.

## Solution Assessment

No. The intended UX is reasonable, but the branch should move the rewrite to the current missing-tool boundary, compose through the public hook surface, use `skillsSnapshotForRun`, and parse the full prompt-facing skill name.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Route skill hints through missing-tool preparation:** `src/agents/pi-embedded-runner/run/attempt.ts:1925-1938`
  - body: The pinned `@earendil-works/pi-agent-core@0.74.0` creates unknown-tool failures as `kind: "immediate"` results in `prepareToolCall`, and those immediate results bypass `finalizeExecutedToolCall` where `afterToolCall` runs. Because this PR only installs an `afterToolCall` rewrite, the main `Tool X not found` path still returns the generic error this feature is meant to improve.
  - confidence: 0.91
- **[P2] Compose with the current public hook surface:** `src/agents/pi-embedded-runner/run/attempt.ts:1935-1938`
  - body: Current `Agent` exposes `afterToolCall` as a public property, and `AgentSession` already assigns it for extension `tool_result` dispatch. The private `_afterToolCall`/`setAfterToolCall` path used here is not present in the current pinned dependency, so after rebase this either fails to build or bypasses the existing extension hook.
  - confidence: 0.88
- **[P2] Use the run-visible skills snapshot:** `src/agents/pi-embedded-runner/run/attempt.ts:1954-1955`
  - body: Sandboxed non-rw runs clear `skillsSnapshotForRun` before exposing skills to the model, but this hook checks `params.skillsSnapshot`. If the rewrite path runs, a guessed hidden skill name can be confirmed by the new hint even when that skill was not visible in the run.
  - confidence: 0.86
- **[P3] Parse complete prompt-facing skill names:** `src/agents/pi-embedded-helpers/errors.ts:555-559`
  - body: `resolveUnknownToolName()` only captures `[a-z0-9_-]+`, while OpenClaw renders raw skill names in `<available_skills>`. Valid visible names with a leading `$`, spaces, or punctuation are truncated or missed, so the new hint silently fails for names the model can actually see.
  - confidence: 0.8
