## What This Changes

The PR adds an unknown-tool error rewrite so registered skills called as tools get a SKILL.md hint, touching embedded runner hook wiring and assistant-error parsing helpers.

## Best Possible Solution

Implement the SKILL.md hint at a current OpenClaw-owned missing-tool/tool-result boundary using current @earendil-works contracts, run-visible skills only, collision preservation, and robust quoted-name parsing.

## Reproduction Assessment

Yes, source-based. Current main still lacks the skill hint, and @earendil-works/pi-agent-core 0.74.0 shows the PR's afterToolCall hook would miss immediate missing-tool results.

## Solution Assessment

No. The direction is reasonable, but the branch should be reworked away from stale @mariozechner/setAfterToolCall wiring and toward a current boundary that can actually rewrite the missing-tool result seen by the model.

  - confidence: 0.84

  - body: The parser captures only [a-z0-9_-]+, but prompt-facing skill names can contain $, spaces, punctuation, or path separators. Those valid names will be truncated or ignored, so the hint will not fire for common skill spellings.
  - confidence: 0.8
