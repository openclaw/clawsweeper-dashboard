## Summary

Keep open: the requested hint is still useful and not implemented on current main, but this branch is stale against the current Pi package boundary and would miss or leak important cases if merged as-is.

## What This Changes

The branch adds a skill-aware rewrite for unknown-tool errors and wires it into embedded agent runs via an afterToolCall hook.

## Best Possible Solution

Implement the hint at the current OpenClaw-owned missing-tool/result boundary using only run-visible skills, preserving tool-name collisions and full prompt-facing skill names, with focused regression coverage.

## Reproduction Assessment

Yes, source-based. Current main still only has generic unknown-tool handling for the agent path, and the pinned Pi package returns missing tools as immediate results that this branch's hook would not see.

## Solution Assessment

No. The product direction is narrow and useful, but this patch uses stale/private hook wiring and should instead target the current missing-tool/result boundary with run-visible skill data.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Use the current Pi hook API:** `src/agents/pi-embedded-runner/run/attempt.ts:4-10`
  - body: Current main pins @earendil-works/pi-agent-core@0.74.0, whose Agent exposes afterToolCall as an option/property and has no setAfterToolCall method. This branch imports the old @mariozechner types and calls the old setter, so it will not compose after rebasing.
  - confidence: 0.94
- **[P2] Route immediate missing-tool results through the hint:** `src/agents/pi-embedded-runner/run/attempt.ts:1936`
  - body: In the pinned Pi loop, an unknown tool returns an immediate `Tool ... not found` result from prepareToolCall, and afterToolCall only runs for prepared/executed tools. Installing the rewrite only as an afterToolCall hook leaves the main skill-called-as-tool case unchanged.
  - confidence: 0.92
- **[P2] Match only run-visible skills:** `src/agents/pi-embedded-runner/run/attempt.ts:1956-1957`
  - body: Current main clears skillsSnapshotForRun for sandboxed non-rw runs before building the prompt/runtime state, but this rewrite checks params.skillsSnapshot. A model that guesses a hidden skill name could get confirmation that it exists even though it was not visible in the run.
  - confidence: 0.88
- **[P3] Parse complete prompt-facing skill names:** `src/agents/pi-embedded-helpers/errors.ts:556-561`
  - body: The helper captures only `[a-z0-9_-]+`, while the skills prompt renders raw skill names that can contain spaces, slashes, a leading `$`, or other punctuation. Those valid visible skills would be truncated or missed, so the hint silently fails for part of the supported surface.
  - confidence: 0.8
