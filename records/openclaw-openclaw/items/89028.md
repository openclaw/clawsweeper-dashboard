## Summary

Keep this PR open, but it is not merge-ready: the runtime-level title-only Readability check is the right layer, yet the new raw body extraction can select a fake `<body>` token inside script/comment text before the real document body. The intentional provider-fallback behavior also remains a compatibility/cost decision for maintainers.

## What This Changes

The PR changes `web_fetch` so title-only Readability output is treated as failed extraction, tries the configured provider fallback, then falls back to body-only basic HTML cleanup with regression coverage.

## Best Possible Solution

Make the body fallback locate the parsed or sanitized document body safely, keep the documented provider-before-basic-cleanup order, then merge once maintainers accept the provider/cost compatibility change.

## Reproduction Assessment

Yes, from source: current main returns any non-empty Readability text, so a title-only extractor result skips provider/raw fallback. The new review finding is also source-reproducible because the PR searches raw HTML for `<body>` before removing scripts or comments.

## Solution Assessment

No, not yet. Runtime-level title-only detection is the best layer and the provider order matches docs, but the raw regex body locator should be replaced with parser-safe or sanitized body selection before merge.
