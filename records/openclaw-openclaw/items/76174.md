## Summary

Keep this issue open. The report has enough concrete evidence to treat the direct OpenAI API-key embedded-run timeout as a real provider-runtime problem, and current main still routes native `openai/*` Responses calls through WebSocket auto mode unless `transport: "sse"` is explicitly set.

## What This Changes

Investigate and fix direct OpenAI API-key embedded runs timing out with zero tokens while direct OpenAI HTTPS requests from the same host succeed.

## Best Possible Solution

Keep the OpenAI direct API-key route provider-owned, but add a bounded no-output diagnostic/fallback path for WebSocket auto mode and document `transport: "sse"` as the immediate diagnostic workaround.
