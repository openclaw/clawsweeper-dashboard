## Summary

Keep this open. The report is recent, detailed, and maps to the current direct OpenAI Responses path; current main has diagnostics and SSE escape hatches, but I do not see high-confidence evidence that the zero-token OpenAI API-key hang is fixed.

## What This Changes

The issue reports that direct `openai/*` API-key embedded runs on `v2026.4.29` hang with zero tokens until timeout while direct OpenAI `curl` calls from the same host succeed quickly.

## Best Possible Solution

Keep the issue open until maintainers validate direct OpenAI API-key runs on current main, compare `transport: "auto"` versus `transport: "sse"`, and either fix the websocket no-event path or document the supported diagnostic/workaround path.

## Reproduction Assessment

Yes. The issue has a high-confidence reproduction path: run an embedded PI turn with direct `OPENAI_API_KEY` and `openai/gpt-5.5` or `openai/gpt-5.4`, then compare default auto/websocket transport with explicit SSE while collecting model-call diagnostics.

## Solution Assessment

Unclear. There is no proposed patch; the safest fix direction is likely a narrow OpenAI Responses transport change, but choosing between defaulting to SSE, adding a websocket pre-output watchdog/fallback, or improving diagnostics needs maintainer validation against live OpenAI behavior.
