## Summary

Keep open: current main and v2026.6.6 still have daily/idle timestamp-based reset policy, heartbeat isolation, and thread-binding max-age, but no general normal-session max-age or max-turn rotation. This remains a valid core session lifecycle feature that needs maintainer product decisions before implementation.

## What This Changes

The issue requests configurable maximum session age and maximum turn limits that rotate long-running agent sessions before context bloat degrades reliability.

## Best Possible Solution

Extend the existing core session reset lifecycle with explicit max-age and max-turn semantics, including state-preservation behavior and coverage across chat, CLI, gateway, cron, and heartbeat entry points.

## Reproduction Assessment

Yes, at source level: current reset config only exposes daily/idle timestamp policy, and the normal reuse gates evaluate freshness without max-age or max-turn inputs. This is a missing feature rather than a broken existing contract.

## Solution Assessment

Unclear until maintainers choose the product policy; the best implementation should extend the existing reset lifecycle rather than add a separate cleanup loop that could diverge from chat, CLI, gateway, cron, and heartbeat behavior.
