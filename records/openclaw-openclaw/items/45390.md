## Summary

Keep open: current main and latest release still lack a normal-session max-age or max-turn rotation policy. Daily/idle reset, heartbeat-aware freshness, compaction memory flush, and thread-binding max-age cover adjacent lifecycle cases but not the requested automatic rotation contract.

## What This Changes

Requested configurable session max-age and max-turn rotation for long-running agent sessions, with safe state persistence before opening a fresh session.

## Best Possible Solution

Define one session lifecycle contract for max-age and max-turn rotation, including memory/continuity semantics, then update config types, schema, reset-path logic, docs, and focused regression coverage together.

## Reproduction Assessment

Yes, at source level: current main has daily and idle freshness checks, but no normal-session max-age or max-turn policy to enforce the requested rotation. A live multi-day repro is unnecessary for this triage because the missing config/runtime branch is directly visible in source and docs.

## Solution Assessment

Unclear as an implementation path: TTL/max-turn rotation is plausible, but the public config shape and pre-rotation memory behavior need maintainer product review. A narrow fix PR should wait until maintainers decide how this coordinates with reset hooks, memory flush, provider-owned CLI sessions, and context-window auto-split work.
