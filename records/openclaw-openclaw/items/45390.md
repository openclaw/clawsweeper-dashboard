## Summary

Keep open: current main and v2026.6.6 support daily/idle timestamp reset and thread-binding max-age, but not configurable normal-session max-age or max-turn automatic rotation.

## What This Changes

The issue requests configurable maximum session age and turn-count limits that automatically rotate long-lived agent sessions after preserving state.

## Best Possible Solution

Define one opt-in core reset lifecycle extension with max-age and max-turn semantics, coordinated with the reset-memory design, and cover chat plus Gateway agent entry points.

## Reproduction Assessment

Yes. Source inspection on current main shows the strict reset schema only accepts daily/idle fields, freshness evaluation is timestamp-only, and chat/Gateway reuse paths have no max-lifetime or turn-count input.

## Solution Assessment

Unclear until maintainers choose the lifecycle policy. The maintainable direction is to extend the existing reset lifecycle rather than add a separate cleanup loop that can diverge across chat, Gateway, cron, and heartbeat behavior.
