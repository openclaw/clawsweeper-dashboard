## Summary

Keep open. Current main has daily/idle/manual reset, heartbeat freshness fixes, focused thread max-age/unfocus controls, and compaction memory flush support, but it still does not provide the requested normal-session configurable max-age and max-turn automatic rotation with safe pre-rotation persistence.

## What This Changes

Request a configurable session TTL and turn-count cap that saves memory, starts a fresh session, and logs automatic rotation when exceeded.

## Best Possible Solution

Define a generic normal-session rollover contract under the existing session reset lifecycle that supports max-age and max-turn limits, persists required counters, emits typed rollover reasons, excludes maintenance traffic intentionally, and connects safe pre-rotation persistence through the hook/lifecycle system.

## Reproduction Assessment

Not applicable as a feature request. Static verification is high-confidence: current schema, types, runtime reuse logic, docs, and bundled hook metadata expose daily/idle/manual reset and focused thread max-age, but no normal-session max-age or max-turn rotation policy.

## Solution Assessment

Unclear until maintainer/API review. The requested direction is valid, but the maintainable fix should extend the existing session.reset/session lifecycle contract rather than adding standalone keys that bypass resetByType, resetByChannel, and hook semantics.
