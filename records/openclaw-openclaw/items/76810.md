## Summary

Keep open: current main still suppresses fallback attribution in normal non-verbose chat replies, even though it tracks the selected and active fallback model internally. Making that model identity visible by default needs maintainer UX/policy direction because it changes normal channel output and may need an opt-out/config shape.

## What This Changes

Expose the active fallback candidate model in normal chat replies so users can distinguish a timed-out primary model from a fallback response.

## Best Possible Solution

Add a maintainer-approved fallback identity surface, preferably structured channel metadata with a minimal inline fallback for channels that cannot render metadata, and document the default and opt-out behavior.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: fallback transitions emit lifecycle metadata regardless of verbosity, but `buildFallbackNotice` is only called under `verboseEnabled`, and the e2e test asserts verbose-off replies omit `Model Fallback:`.

## Solution Assessment

No. The current verbose-only notice is useful diagnostic plumbing, but it does not solve normal chat identity confusion; the narrow maintainable fix needs a product decision on default visibility, opt-out/config, and channel rendering.
