## Summary

Keep open: current main records fallback model attribution internally and exposes it through verbose/status paths, but default chat replies still omit a fallback marker; changing that is a user-facing UX policy feature rather than a narrow bug fix.

## What This Changes

The issue requests user-visible fallback model attribution when a primary model times out and a different fallback candidate answers in the same chat identity.

## Best Possible Solution

Add a maintainer-approved fallback identity surface, preferably structured channel metadata with a minimal inline fallback for channels that cannot render metadata, plus docs for default visibility and opt-out behavior.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: with `resolvedVerboseLevel: "off"`, fallback transitions emit lifecycle state but the normal reply does not get `Model Fallback:` text, and the e2e test asserts that behavior.

## Solution Assessment

No, a quick default-on inline prefix is not clearly the best fix because it changes normal chat disclosure. The maintainable solution needs product direction on default visibility, channel rendering, and opt-out/config shape.
