## Summary

Keep open: current main has fallback attribution internally and a verbose-only visible notice, but normal chat replies still suppress the model-fallback marker when verbose is off, which is the confusing default path described in the issue.

## What This Changes

The issue requests default-visible tagging or metadata for replies produced by a fallback candidate model.

## Best Possible Solution

Add a maintainer-approved fallback identity surface, preferably structured channel metadata with a minimal inline fallback for channels that cannot render metadata, and document the default/opt-out policy.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: a fallback response with `resolvedVerboseLevel: "off"` reaches final delivery without `Model Fallback:` text, and the existing e2e test asserts that behavior. I did not run tests because this review was required to keep the checkout read-only.

## Solution Assessment

No, the current verbose-only notice is useful diagnostic plumbing but does not solve the normal chat identity confusion described here. The best fix needs a product decision on default visibility, opt-out/config shape, and channel-specific rendering.
