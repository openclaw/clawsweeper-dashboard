## Summary

Keep open: the PR has a useful focused fix and sufficient persistence proof, but it is not merge-ready because it only latches the followup-runner fallback path while the main reply fallback loop can still persist repeated assistant error stubs.

## What This Changes

The PR adds a `suppressAssistantErrorPersistence` guard option, plumbs it through the embedded runner, and latches followup fallback candidates so queued user messages and assistant error stubs are only persisted once.

## Best Possible Solution

Apply the same per-fallback persistence latch to both `createFollowupRunner` and the main `agent-runner-execution` embedded fallback callback, preserving the first terminal assistant error and adding regression coverage for followup plus steer/active cascades.

## Reproduction Assessment

Yes, at source level: `runWithModelFallback` invokes the run callback once per candidate, and current main has two embedded fallback callers. I did not run a live OpenRouter/Anthropic reproduction in this read-only review.

## Solution Assessment

No. The approach is right for queued followups, but the fix must also cover `agent-runner-execution` or prove that steer/active runs cannot persist assistant error stubs through that path.
