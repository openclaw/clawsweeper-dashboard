## Summary

Keep open: the all-embedded persistence fix and proof are useful, but the patch still suppresses assistant error persistence by fallback callback index, so mixed CLI/embedded or early-failure cascades can drop the first embedded error separator.

## What This Changes

The PR adds fallback-cascade latches and regression tests so queued user messages and assistant error stubs are not repeatedly persisted across model fallback candidates.

## Best Possible Solution

Keep the bounded persistence latch, but key assistant-error suppression to an actual embedded assistant-error persistence or consume event, with mixed CLI-to-embedded and early-failure coverage in both fallback callers before merge.

## Reproduction Assessment

Yes, at source and proof level: current fallback callers re-enter a run callback per candidate, and the PR body supplies real SessionManager output for the all-embedded duplication case. I did not run a live OpenRouter/channel reproduction in this read-only review.

## Solution Assessment

No, not as-is: the latch direction is right, but assistant-error suppression must be keyed to actual embedded persistence rather than raw fallback candidate index. The safer fix is a small embedded-write latch plus mixed-runtime regression coverage.

- **[P1] Latch assistant suppression to embedded writes:** `src/auto-reply/reply/followup-runner.ts:560`
  - body: This suppresses assistant error persistence for every callback after the first, but a prior callback can take the CLI branch or fail before the embedded SessionManager appends an assistant error. The next embedded failure then runs with suppression already enabled, dropping the only assistant error separator and potentially leaving the session ending in the queued user message; flip this latch only after an embedded error was actually persisted or intentionally consumed.
  - confidence: 0.9
- **[P1] Keep main reply suppression scoped to embedded attempts:** `src/auto-reply/reply/agent-runner-execution.ts:1576`
  - body: The main reply path has the same raw candidate-index latch before runtime selection. If the first candidate does not write an embedded assistant error, the first embedded fallback can still be suppressed, so the active-run/steer path can lose its only failure separator; track embedded assistant-error persistence instead of callback count.
  - confidence: 0.88
