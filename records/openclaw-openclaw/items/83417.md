## Summary

Keep open: the PR is a useful fix for queued followup retries, but it leaves the normal reply fallback loop uncovered, so the linked steer/active-run assistant-error-stub corruption can still happen after merge.

## What This Changes

The PR adds fallback-candidate latches and a new `suppressAssistantErrorPersistence` guard option so followup fallback retries persist a queued user message and assistant error stub only once, with regression tests.

## Best Possible Solution

Apply the same per-fallback persistence latch to both `createFollowupRunner` and the main `agent-runner-execution` embedded fallback callback, then prove followup and steer/active runs persist at most one queued user turn and one terminal assistant error across a multi-candidate cascade.

## Reproduction Assessment

Yes at source level: current main has two embedded `runWithModelFallback` caller paths, and the PR only latches persistence in the followup one. I did not run a live OpenRouter/Anthropic channel reproduction in this read-only review.

## Solution Assessment

No. The direction is right for queued followups, but the best fix must cover the main reply fallback loop as well, or prove that steer/active runs cannot traverse it when assistant error stubs are persisted.
