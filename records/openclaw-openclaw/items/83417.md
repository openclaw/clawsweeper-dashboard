## Label Justifications

- `P1`: The PR addresses an active agent fallback/session corruption path that can leave affected sessions unable to process later messages.
- `impact:session-state`: The diff changes how queued user and assistant error entries are persisted to session JSONL during fallback retries.
- `impact:message-loss`: The reported corruption can cause provider rejection loops that prevent queued followup messages from being delivered normally.
- `merge-risk: 🚨 session-state`: Merging this changes which fallback-candidate messages are retained or suppressed in the persisted transcript.
- `merge-risk: 🚨 message-delivery`: If suppression fires before any embedded assistant separator is persisted, later turns can still hit invalid transcript role shape and fail delivery.

## Summary

The PR should stay open because it fixes the all-embedded fallback duplication path, but the latest diff advances assistant-error suppression per fallback callback rather than per embedded session write, which can drop the first embedded error separator after a CLI or otherwise non-persisting candidate.

## What This Changes

The PR adds fallback-cascade latches so embedded followup and main reply retries only persist the queued user message and assistant error stub once, with regression tests for the guard and both fallback callers.

## Best Possible Solution

Keep the bounded persistence latch, but make assistant-error suppression depend on the embedded session path actually having written or intentionally consumed the first assistant error separator, with mixed CLI/embedded and early-failure regression coverage before merge.

## Reproduction Assessment

Yes, at source and proof level: current fallback callers run one callback per candidate, and the PR body shows a real SessionManager cascade reproducing and fixing the all-embedded duplication shape. I did not run a live provider/channel reproduction in this read-only review.

## Solution Assessment

No, not quite: the all-embedded latch is the right direction, but using a raw candidate-attempt counter for assistant-error suppression is too broad because CLI or early-failing candidates may not have written the first embedded assistant error stub.

Overall confidence: 0.88

Full review comments:

- **[P1] Latch assistant suppression after an embedded write:** `src/auto-reply/reply/followup-runner.ts:560`
  - body: This counter advances for every fallback callback before the runtime branch is known. Current tests already cover a CLI-backed candidate failing over to an embedded candidate; with this patch that embedded candidate receives `suppressAssistantErrorPersistence: true` even though the CLI path did not write a SessionManager assistant error stub. If that embedded candidate fails, the queued user entry can be left without the separator this PR relies on, so a later turn can still form consecutive user messages. Gate this on an actual embedded persistence event or an embedded-only latch and add the mixed-runtime regression.
  - confidence: 0.88
- **[P1] Keep main reply suppression scoped to embedded attempts:** `src/auto-reply/reply/agent-runner-execution.ts:1569`
  - body: The main reply path has the same pre-branch attempt counter, so a CLI candidate or another non-persisting candidate can make the first embedded fallback suppress its assistant error stub. That can reintroduce the bad transcript shape on active runs even though the all-embedded tests pass. Track whether the embedded SessionManager path has actually written or consumed the first error separator before suppressing later ones.
  - confidence: 0.86
