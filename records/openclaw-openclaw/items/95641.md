## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Avoid retrying after visible stream events:** `extensions/amazon-bedrock/stream.runtime.ts:334-336`
  - body: The retry loop can continue after `messageStart` or content deltas have already been pushed. `agentLoop` appends a new assistant message on every start event, so a mid-stream Bedrock drop followed by retry can leave the first attempt's assistant turn or streamed preview in context before the retried turn succeeds. Retry only before visible output, or buffer and roll back the failed attempt with regression coverage.
  - confidence: 0.91
- **[P2] Classify named Bedrock errors outside instanceof:** `extensions/amazon-bedrock/stream.runtime.ts:367-380`
  - body: The new retry fixtures throw `Error` objects with Bedrock `name` values, but this branch only checks those names when the value is a `BedrockRuntimeServiceException`. Those named `InternalServerException`, `ServiceUnavailableException`, and `ModelStreamErrorException` values fall through as non-transient, so the advertised retry path and tests do not match.
  - confidence: 0.88
- **[P2] Align the pre-aborted signal test with runtime:** `extensions/amazon-bedrock/stream.runtime.test.ts:604`
  - body: This test spies on `BedrockRuntimeClient.prototype.send` without adding a pre-send guard in production. The runtime still builds the command and calls `send` with the already-aborted signal, so the zero-call assertion is wrong unless the production code explicitly checks `options.signal.aborted` before calling the SDK.
  - confidence: 0.84
