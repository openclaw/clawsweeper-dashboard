## Summary

Keep open. Current main does not implement exec `tools.exec.allowedHosts`, and the PR head still has blocking command-execution policy regressions: sandbox wildcard allowlists skip the approval/obfuscation path, and node-host map buckets are not visible to the legacy node-host evaluator.

## What This Changes

The PR adds `tools.exec.allowedHosts`, per-host `exec-approvals.json` allowlist maps, doctor migration/docs, and sandbox/gateway/node approval plumbing.

## Best Possible Solution

If maintainers want this feature, rebase it onto current exec approval internals and preserve existing ask, obfuscation, durable approval, and node-host allowlist semantics with regression coverage for sandbox wildcard approvals and node-host map buckets.

## Reproduction Assessment

Yes, for the PR regression. Source inspection of the PR head shows sandbox `"*"` allowlists return before `requiresExecApproval`, and node-host checks still consume the flat `approvals.allowlist` while map `node` buckets are omitted from that fallback.

## Solution Assessment

No. The feature direction may be useful, but the current patch is not the narrowest maintainable solution because it bypasses approval policy for one sandbox wildcard case and does not fully adapt node-host allowlist resolution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve approval checks for sandbox wildcards:** `src/agents/bash-tools.exec-host-gateway.ts:95-99`
  - body: This early return exits before obfuscation detection and `requiresExecApproval`, so a sandbox request with a `"*"` allowlist runs immediately even when `ask: "always"` or an obfuscated command should produce an approval prompt. Keep the host-side path-resolution workaround, but still run the approval decision path before allowing execution.
  - confidence: 0.94
- **[P2] Include node buckets in the flat allowlist fallback:** `src/infra/exec-approvals.ts:589-598`
  - body: Node-host execution still evaluates `approvals.allowlist`, but map-form allowlists only contribute the `default` bucket to that flat fallback. A config such as `{ "node": [{ "pattern": "..." }] }` with no default will therefore deny node commands that the new per-host format says are allowed.
  - confidence: 0.88
