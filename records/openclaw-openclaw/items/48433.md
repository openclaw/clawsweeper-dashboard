## Summary

Keep open. The PR proposes a real command-exec feature that is not on current main, but the current head is not safe to merge as reviewed because it introduces approval-policy regressions and lacks inspectable real behavior proof.

## What This Changes

Adds optional `tools.exec.allowedHosts`, per-host `exec-approvals.json` allowlist maps, doctor migration, docs, and approval plumbing for gateway, sandbox, and node exec paths.

## Best Possible Solution

Rework the feature against current exec target routing so allowed host selection, per-host allowlists, approval prompts, obfuscation checks, sandbox execution, and node-host compatibility preserve existing security boundaries, then land with focused tests and redacted real-environment proof.

## Reproduction Assessment

Yes for the PR-introduced defects via source inspection: the sandbox wildcard return precedes approval checks, and node-host checks still read a flat allowlist that omits host-specific `node` buckets. The original request is a feature, not a current-main bug.

## Solution Assessment

No. The direction may be useful, but this patch is not the best current solution until it preserves approval-policy boundaries across sandbox, gateway, and node hosts and includes real proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve approval checks for sandbox wildcards:** `src/agents/bash-tools.exec-host-gateway.ts:95-99`
  - body: This early return bypasses obfuscation detection and `requiresExecApproval`, so a sandbox request with `"*"` in its allowlist runs immediately even when `ask: "always"` or an obfuscated command should prompt. Keep the sandbox wildcard workaround, but still run the approval decision path before execution.
  - confidence: 0.94

  - confidence: 0.88
