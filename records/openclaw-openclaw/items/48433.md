## Summary

Keep open. The requested feature is not implemented on current main, and although several earlier review defects were addressed, the current PR head still has blocking command-execution security/correctness issues and lacks inspectable real behavior proof.

## What This Changes

This PR adds optional `tools.exec.allowedHosts`, per-host `exec-approvals.json` allowlist maps, doctor migration, docs, and tests for gateway/sandbox/node exec routing.

## Best Possible Solution

Rebase and rework the feature against current `ExecTarget` routing so allowed host selection, per-host allowlists, sandbox wildcard handling, node-host compatibility, and approval prompts preserve existing security boundaries, then add redacted real Telegram or terminal proof.

## Reproduction Assessment

Yes for the PR-introduced defects by source inspection: the sandbox wildcard return precedes approval checks, node-host paths consume the flat allowlist, and the branch schema omits current `auto` host support. The requested `allowedHosts` behavior itself is a new feature, not a current-main bug.

## Solution Assessment

No. The direction may be useful, but this patch is not the best current solution until it preserves approval-policy boundaries across sandbox, gateway, and node hosts, reconciles current `auto` routing, and includes real proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve approval checks for sandbox wildcards:** `src/agents/bash-tools.exec-host-gateway.ts:95-99`
  - body: This early return bypasses shell analysis, obfuscation detection, and `requiresExecApproval`, so a sandbox request with `"*"` in its allowlist runs immediately even when `ask: "always"` or an obfuscated command should prompt. Keep the sandbox wildcard support, but still route through the approval decision path before execution.
  - confidence: 0.94
- **[P2] Include node buckets in the flat allowlist fallback:** `src/infra/exec-approvals.ts:589-598`
  - body: Map-form allowlists populate the legacy flat `resolved.allowlist` only from `default`, but node-host evaluators still read that flat list. A config such as `{ "node": [...] }` with no default bucket will miss and deny node commands that the per-host format should allow.
  - confidence: 0.88
- **[P2] Preserve current auto exec host support:** `src/config/zod-schema.agent-runtime.ts:425-426`
  - body: Current main accepts `tools.exec.host: "auto"` and routes it through `ExecTarget`, but this branch's schema and types only enumerate sandbox/gateway/node. Reconcile `allowedHosts` with the current target model so valid current configs are not rejected after rebase.
  - confidence: 0.82
