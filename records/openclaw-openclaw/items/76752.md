## Summary

Keep open: current main still treats `fsync` `EPERM` as fatal in shared media buffer writes, and the PR is a narrow functional fix for the linked Windows Telegram media failure; the remaining cleanup is changelog credit and preferably targeted regression coverage.

## What This Changes

The PR wraps `writeSavedMediaBuffer`'s `handle.sync()` call to swallow only `EPERM` and adds a changelog fix entry for Windows Telegram media downloads.

## Best Possible Solution

Keep the narrow `EPERM`-only fsync handling in the media store while preserving non-EPERM failures, with contributor credit and focused regression coverage before merge.

## Reproduction Assessment

Yes for source-level reproduction: current main will reject if `handle.sync()` throws `EPERM` during `saveMediaBuffer`, and Telegram media routes call that function. No live Windows/antivirus run was performed.

## Solution Assessment

Yes, functionally. Swallowing only `EPERM` at `handle.sync()` keeps the existing temp-write and rename path intact while preserving other sync failures; the PR should still add changelog credit and ideally a mock regression test.
