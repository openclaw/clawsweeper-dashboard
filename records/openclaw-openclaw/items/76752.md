## Summary

Keep open: current main still propagates `handle.sync()` failures in the shared media store, and Telegram media downloads use that path. The PR is a narrow bug fix, but it should get focused regression coverage before merge.

## What This Changes

The PR catches `EPERM` from `writeSavedMediaBuffer`'s post-write `handle.sync()` and adds a changelog fix entry for Windows Telegram media downloads.

## Best Possible Solution

Land a narrow media-store sync guard with regression tests that swallow only `EPERM` from post-write sync and preserve other failure behavior.

## Reproduction Assessment

Yes for source-level reproduction: current main awaits `handle.sync()` without a guard, and Telegram media downloads call `saveMediaBuffer`. A focused test can stub `fs.open(...).sync()` to reject with `EPERM`; I did not establish a live Windows antivirus/filesystem reproduction.

## Solution Assessment

Yes: treating only post-write `EPERM` as best-effort is the narrow maintainable fix and matches existing best-effort sync precedent. The safer merge shape is to add focused unit coverage around `EPERM` versus non-`EPERM` sync failures.
