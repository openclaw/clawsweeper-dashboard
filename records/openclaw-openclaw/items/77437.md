## Summary

Keep open: the linked bug remains source-reproducible on current main, and this PR is aimed at the right gateway MCP path, but it still lacks real MCP/claude-cli proof and the proposed yield callback appears to pause child run records rather than the active yielded run.

## What This Changes

The PR threads `sessionId` and an MCP-specific `onYield` callback through gateway scoped tool resolution and the MCP loopback cache/server for `sessions_yield`.

## Best Possible Solution

Bind MCP `sessions_yield` to the active run/session being yielded, keep the existing tool-factory context seam, add focused MCP loopback regression coverage, and require redacted real claude-cli proof before merge.

## Reproduction Assessment

Yes. Source inspection shows current main resolves MCP loopback tools without `sessionId` or `onYield`, and `sessions_yield` deterministically returns `No session context` when `sessionId` is absent; I did not run a live claude-cli loopback reproduction.

## Solution Assessment

No for the patch as written. Threading the existing parameters is the right direction, but the proposed callback should update the active yielded run rather than child runs selected by `requesterSessionKey`, and the PR still needs real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Pause the yielding run, not its children:** `src/gateway/mcp-http.ts:119`
  - body: `requesterSessionKey` identifies runs spawned by the current session; the tracked run itself is stored under `childSessionKey`, and existing yielded-run handling calls `markSubagentRunPausedAfterYield` for that specific run. This callback therefore pauses descendants of a claude-cli subagent instead of the subagent run that yielded, and a root-session yield can stamp active child runs as ended before they finish. Resolve the active run for `requestContext.sessionKey` or pass a run-bound id before marking the run paused.
  - confidence: 0.84
