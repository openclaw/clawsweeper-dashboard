## Summary

Keep open: this PR targets a real gateway MCP `sessions_yield` regression, but the current callback pauses runs spawned by the yielding session instead of the active yielding run, and real claude-cli MCP after-fix proof is still missing.

## What This Changes

The PR threads optional `sessionId` and `onYield` through gateway MCP loopback tool resolution and adds an MCP `onYield` callback that marks matching subagent registry entries paused.

## Best Possible Solution

Bind MCP `sessions_yield` to the active run by `childSessionKey` or a run-bound id, keep the gateway tool-factory plumbing, add focused loopback regression coverage, and require redacted real claude-cli MCP proof before merge.

## Reproduction Assessment

Yes. Source inspection on current main shows MCP loopback tools are resolved without `sessionId` or `onYield`, and `sessions_yield` deterministically returns `No session context` when `sessionId` is absent; I did not run a live claude-cli loopback setup.

## Solution Assessment

No for the patch as written. Passing the existing context through the loopback path is the right direction, but the callback should pause the active run identified by `childSessionKey` or an equivalent run-bound id, not requester children.

## Review Findings

Overall correctness: patch is incorrect
