## Summary

Keep open: current main still has the reported MCP loopback missing-context path, but this PR’s new yield callback marks requester children instead of the active yielding run and still lacks contributor-supplied real MCP/claude-cli proof.

## What This Changes

The PR adds optional `sessionId`/`onYield` plumbing through gateway scoped tool resolution and the MCP loopback cache/server for `sessions_yield`.

## Best Possible Solution

Keep the gateway context seam, bind MCP yield to the active subagent run, add focused loopback regression coverage, and require redacted real claude-cli MCP output before merge.

## Reproduction Assessment

Yes. Source inspection shows current main builds MCP loopback tools without `sessionId` or `onYield`, and `sessions_yield` deterministically returns `No session context` when `sessionId` is missing; I did not run a live claude-cli setup.

## Solution Assessment

No for the patch as written. Passing the existing context through the gateway path is the right direction, but the callback must pause the active run identified by `childSessionKey` or an equivalent run-bound id rather than requester children.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Pause the active run, not requester children:** `src/gateway/mcp-http.ts:119`
  - body: `requesterSessionKey` identifies runs spawned by the current session; the tracked run itself is stored under `childSessionKey`. This callback therefore pauses descendants of a claude-cli subagent instead of the subagent run that yielded, and a root-session yield can stamp active child runs as ended before they finish. Resolve the active run for `requestContext.sessionKey` or pass a run-bound id before marking the run paused.
  - confidence: 0.84
