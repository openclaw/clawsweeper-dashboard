## Summary

Keep open: the PR addresses a real current-main gap, but its MCP yield callback appears to mark the requester’s child runs as paused instead of marking the run/session that actually yielded, which can make active children look ended or paused before they finish.

## What This Changes

The PR threads `sessionId` and `onYield` through gateway MCP tool resolution and adds an MCP loopback yield callback that marks matching subagent runs paused.

## Best Possible Solution

Land a covered MCP-loopback fix that passes existing yield context through the gateway path without marking active child runs as yielded before they actually yield or complete.

## Reproduction Assessment

Yes, from source inspection: current main builds MCP loopback tools without `sessionId` or `onYield`, and `sessions_yield` returns `No session context` when `sessionId` is absent. I did not run a live claude-cli loopback scenario in this read-only review.

## Solution Assessment

No, not as written: plumbing the existing parameters is the right direction, but the proposed callback should not mark every child run whose requester is the yielding session as `sessions_yield`. A narrower registry helper should mark the yielding run when one exists, or otherwise leave child run state untouched.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Mark the yielding run, not its children:** `src/gateway/mcp-http.ts:122`
  - body: This loop selects entries whose `requesterSessionKey` is the current MCP session, which are the child runs spawned by the yielding session. `markSubagentRunPausedAfterYield` sets `endedAt` and `pauseReason` on the passed run record, so a parent calling `sessions_yield` can make still-running children appear ended/paused before they finish; the callback should instead target the run represented by the current session, such as a matching `childSessionKey`, or avoid mutating child records.
  - confidence: 0.87
