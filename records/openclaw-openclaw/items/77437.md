## Summary

Keep open: the linked bug is real on current main, but this PR still has a blocking callback bug and lacks contributor-supplied real MCP/claude-cli proof.

## What This Changes

The PR threads sessionId and onYield through gateway MCP loopback tool resolution so sessions_yield can run on the claude-cli MCP path.

## Best Possible Solution

Thread run-bound yield context through MCP loopback, pause the active subagent run by childSessionKey or an equivalent active-run lookup, add focused regression coverage, and require redacted real claude-cli MCP proof before merge.

## Reproduction Assessment

Yes. Source inspection of current main shows the MCP loopback path constructs tools without sessionId/onYield, and createSessionsYieldTool deterministically returns No session context when sessionId is absent; the PR discussion also reports a synthetic reproduction.

## Solution Assessment

No for this PR as written. The parameter-plumbing direction is right, but the new callback matches requesterSessionKey and therefore pauses children of the active session instead of the active yielded run.

## Review Findings
