## Summary

Keep open: this PR targets a real source-reproducible gateway/MCP `sessions_yield` bug, but the current callback pauses requester child runs instead of the active yielding run, and the external PR still lacks real claude-cli MCP proof.

## What This Changes

The PR threads `sessionId` and `onYield` through gateway MCP loopback tool resolution into `createOpenClawTools`, then marks matching subagent runs paused when `sessions_yield` is called.

## Best Possible Solution

Bind MCP `sessions_yield` to the active run, preserve the gateway tool-factory plumbing seam, add focused loopback regression coverage, and require redacted real claude-cli MCP proof before merge.

## Reproduction Assessment

Yes, source-reproducible. Current main builds MCP loopback tools without `sessionId` or `onYield`, and `sessions_yield` deterministically returns `No session context` when `sessionId` is absent; I did not run a live claude-cli loopback setup.

## Solution Assessment

No for the patch as written. Threading the existing parameters is the right direction, but the callback should pause the active run identified by `childSessionKey` or an equivalent run-bound id, not requester children.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Pause the active run instead of requester children:** `src/gateway/mcp-http.ts:122`
  - body: The MCP callback filters on `entry.requesterSessionKey === requestContext.sessionKey`, but registry records use `requesterSessionKey` for runs spawned by that session and `childSessionKey` for the run whose MCP tool call is yielding. As written, a yielding claude-cli subagent leaves its own run unpaused and can mark an active descendant as yielded; select the active run by `childSessionKey` or pass a run-bound id before calling `markSubagentRunPausedAfterYield`.
  - confidence: 0.86
