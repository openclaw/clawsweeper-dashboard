## What This Changes

The issue requests that `openclaw doctor --fix` stop leaving legacy cron `notify:true` unresolved while still claiming the cron store was normalized when `cron.webhook` is unset.

## Best Possible Solution

Land or intentionally revise the focused doctor migration in https://github.com/openclaw/openclaw/pull/89396, preserving existing delivery objects and invalid configured-webhook warnings, then close this issue as implemented.

## Reproduction Assessment

Yes, at source level. A cron job with `notify:true`, no valid `cron.webhook`, and `delivery.mode` none/absent/webhook reaches the warning-only branch and then doctor re-counts the same raw `notify:true` marker on the next run.

## Solution Assessment

Yes, the active fix direction is the narrow owner-boundary solution if maintainers accept the persisted-marker cleanup. Doctor owns legacy config repair, and dropping inert `notify` while preserving delivery breaks the loop without adding runtime fallback behavior.
