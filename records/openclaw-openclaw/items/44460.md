## What This Changes

The issue asks `openclaw doctor --fix` to either resolve legacy cron `notify: true` metadata when `cron.webhook` is unset or avoid claiming normalization while unresolved legacy fields remain.

## Best Possible Solution

Land or replace the linked PR with a focused cron-doctor fix and regression tests that either remove inert no-webhook `notify: true` metadata or make the success output conditional on all legacy fields being resolved.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a cron job with top-level `notify: true`, `delivery.mode: "none"`, and no `cron.webhook` reaches the warning-only branch and keeps `raw.notify`.

## Solution Assessment

Yes for the narrow repair direction, but it is not solved on main. The maintainable fix is to remove inert no-webhook legacy notify metadata or avoid printing the normalization success message while that legacy field remains.
