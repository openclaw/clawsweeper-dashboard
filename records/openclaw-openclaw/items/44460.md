## Summary

Keep open: current main and the latest release still leave `notify: true` unresolved when `cron.webhook` is unset, and the linked implementation PR remains open and conflicting rather than merged.

## What This Changes

The issue asks `openclaw doctor --fix` to fully resolve or clearly report legacy cron jobs that retain top-level `notify: true` when `cron.webhook` is unset.

## Best Possible Solution

Land or replace the linked PR with a focused cron-doctor fix and regression tests that either remove inert no-webhook `notify: true` metadata or avoid claiming normalization while it remains unresolved.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: a `notify: true` cron job with no `cron.webhook` reaches the warning-only branch and keeps `raw.notify`. If another migration also changes the store, current main can still print `Cron store normalized` before the unresolved warning.

## Solution Assessment

Yes for the narrow cron-doctor direction, but not yet solved on main. The maintainable fix is to remove inert no-webhook legacy notify metadata or make the success output conditional on all legacy fields being resolved.
