## Summary

Keep this PR open, but it is not ready to merge: the linked gateway probe bug is real on current main and the latest release, while the patch broadens the full timeout to inactive configured remote targets and still has only mocked Vitest proof rather than a live remote probe run.

## What This Changes

The PR makes configured remote and explicit gateway probe targets use the caller timeout budget and adds helper and command regression tests.

## Best Possible Solution

Land a narrower fix that gives the full budget to active configured remote and explicit targets while preserving a short cap for inactive configured remotes, with a regression test and redacted live remote proof.

## Reproduction Assessment

Yes. The current main and v2026.5.27 source still cap non-loopback configured remote probes at 1500ms while the linked report gives concrete remote-mode steps with a larger timeout; I did not run a live remote gateway in this read-only review.

## Solution Assessment

No. The patch fixes the active configured-remote budget path, but the unconditional configRemote early return also changes inactive configured remote behavior and needs a narrower condition plus live proof.

Full review comments:

- **[P2] Preserve the inactive configured-remote cap:** `src/commands/gateway-status/helpers.ts:136-137`
  - body: `resolveTargets` adds a `configRemote` target whenever `gateway.remote.url` exists, even when `gateway.mode` is local, and the probe pass still probes every base target. With this unconditional early return, a local-mode config that still has a stale remote URL now spends the full caller or handshake timeout on that inactive target instead of the existing 1500ms cap, so `gateway probe --timeout 30000` can stall on a target the config says is inactive. Please keep the full budget to active configured remotes and explicit URLs, and add a local-mode stale remote regression test.
