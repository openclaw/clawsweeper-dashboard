## Summary

Keep open. Current main still has explicit Telegram topic `agentId` routing, top-level ACP bindings, runtime conversation bindings, and manual `/acp spawn` flows, but it does not expose a `defaultAcp` config surface or an inbound routing branch that auto-spawns ACP for an unbound new Telegram topic.

## What This Changes

The issue requests an opt-in Telegram per-DM/per-chat default ACP binding that auto-spawns and binds an ACP session for new topics without requiring `/acp spawn` in each topic.

## Best Possible Solution

Design a first-class, opt-in Telegram default ACP binding surface with explicit precedence, authorization, allowed-agent/cwd policy, and per-topic session isolation; then implement it through the existing binding/spawn seams with strict config schema, docs, and focused Telegram/ACP tests.

## Reproduction Assessment

Not applicable as a feature request. The absence is high-confidence from current-main source inspection: strict config schemas reject unknown default fields, and inbound Telegram routing has no default ACP materialization path.

## Solution Assessment

Unclear until maintainer product/security review. A `defaultAcp`-style config is plausible, but the maintainable solution needs explicit operator opt-in, policy gates, precedence rules, and coordination with the open ACP thread-isolation work before implementation.
