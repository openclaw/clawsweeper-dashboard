## Summary

Keep open: current main has partial watchdog fixes, but the TUI still clears the active run, switches to idle, and emits the reported prompt after a quiet watchdog window even when later events for the same run can still arrive.

## What This Changes

This issue asks for quiet but still-active TUI chat runs to remain visibly working until authoritative run/session state says they are terminal.

## Best Possible Solution

Before the watchdog clears a run or prompts the user, have the TUI refresh authoritative session/run state and keep or rearm streaming when the current session still has an active run.

## Reproduction Assessment

Yes from source inspection and the existing unit harness: a chat delta arms the watchdog, timer expiry clears `activeChatRunId`, sets idle, and logs the reported prompt. I did not execute the test or a live provider run because this review is read-only.

## Solution Assessment

No. The current implementation is not the best fix because it treats total event silence as stale without consulting the gateway's active-run state, even though `sessions.list` already exposes `hasActiveRun`.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/78360. Current main still lets `src/tui/tui-event-handlers.ts` clear `activeChatRunId`, set idle, and emit the timeout prompt after a quiet watchdog window. Use existing authoritative state before clearing: `sessions.list` already returns `hasActiveRun`, but the TUI types/session info do not expose or consume it. Extend the TUI session-list/session-info path as needed so the watchdog can refresh/check the current session; if the current session still has an active run, keep the status streaming/running, rearm the watchdog, and do not show the timeout prompt. Only clear the run and reload history when authoritative state says no active run remains or the recovery path genuinely cannot confirm activity. Preserve existing tool-event, lifecycle-event, reconnect, lost-final recovery, and abort behavior. Do not add a config option or change gateway protocol incompatibly. Add regression coverage for a quiet active run with no deltas but `hasActiveRun: true`, and for the stale/lost-final path with `hasActiveRun: false`. Add a one-line user-facing `CHANGELOG.md` fix entry.
