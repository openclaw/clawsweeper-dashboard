## Summary

Keep open: current main and the latest release still allow the TUI watchdog to clear an active run and show the reported timeout prompt based on stream silence alone, while the gateway already exposes active-run state that the TUI does not currently carry into its session info path.

## What This Changes

The issue asks for the TUI watchdog to keep quiet but still-active chat runs visibly working instead of switching to idle or prompting the user to send another message solely because stream deltas paused.

## Best Possible Solution

Before the watchdog clears a TUI run or prompts the user, refresh or consume authoritative session/run state such as `sessions.list.hasActiveRun`; keep or rearm streaming while the current session still has an active run, and only fall back to idle after terminal state or no active run.

## Reproduction Assessment

Yes. Source inspection and the existing fake-timer test show a current-main path where a delta arms the watchdog and timer expiry clears the active run, sets idle, and logs the reported prompt without consulting active-run state.

## Solution Assessment

No. The current implementation treats stream silence as stale enough to reset the UI; the narrower maintainable fix is to check gateway/session active-run state before clearing TUI run state.
