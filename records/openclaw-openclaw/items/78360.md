## Summary

Keep open: current main and v2026.5.12 still let the TUI watchdog clear an active run to idle and emit the reported prompt based on stream silence, while the TUI does not carry the gateway's existing active-run signal. The related timeout/config and stale-notice PRs do not supersede this active-run correctness bug.

## What This Changes

The issue asks for quiet but still-active TUI runs to remain visibly working instead of being marked idle or prompting the user to send another message based only on missing stream deltas.

## Best Possible Solution

Before the watchdog clears a TUI run or prompts the user, refresh or consume authoritative session/run state such as `sessions.list.hasActiveRun`; keep or rearm the working state while the current session still has an active run, and only fall back to idle after terminal state or no active run.

## Reproduction Assessment

Yes, at source level. A chat delta arms the watchdog, timer expiry clears the active run to idle and logs the reported prompt, and current tests assert that behavior without consulting gateway active-run state.

## Solution Assessment

No, the current implementation is not the best solution because it treats stream silence as enough to reset the UI. The narrower maintainable fix is to check authoritative active-run state before clearing TUI run state, leaving configurable timeout policy to the separate config thread.

In the TUI streaming watchdog path, do not clear `activeChatRunId`, set idle, or append `This response is taking longer than expected. Send another message to continue.` solely because no stream delta arrived if the current session still has an authoritative active run. Thread the existing gateway `sessions.list.hasActiveRun` signal through the TUI session-list/session-info path or add a narrow helper, refresh/check it on watchdog expiry, and rearm/keep `streaming` while the backend reports the run active.
