## Summary

Keep open: current main still clears the active TUI run and emits the reported watchdog prompt after a quiet watchdog window without checking authoritative active-run state. The earlier related fix covers tool/lifecycle/reconnect proof-of-life, but this issue describes a remaining total-silence active-run case and no paired open PR was found.

## What This Changes

The issue asks for quiet but still-active TUI chat runs to remain visibly working until session/run state is terminal, instead of being marked idle from text-delta silence alone.

## Best Possible Solution

Have the TUI watchdog refresh authoritative session/run state before clearing a quiet run, keep or rearm the active run when the current session still has active work, and preserve timeout recovery for truly stale runs.

## Reproduction Assessment

Yes. Source inspection and the existing unit harness give a high-confidence path: a chat delta arms the watchdog, timer expiry clears `activeChatRunId`, sets idle, and logs the reported prompt without consulting active session state.

## Solution Assessment

No. The current implementation treats event silence as terminal enough to clear TUI state; the narrower maintainable fix is to consult active-run/session state before prompting or switching to idle.
