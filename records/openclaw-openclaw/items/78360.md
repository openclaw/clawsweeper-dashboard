## Summary

Keep open: current main and the latest release tag still clear the active TUI run and show the reported watchdog prompt from stream silence alone, while the authoritative active-run signal already exists but is not consumed by the TUI watchdog path.

## What This Changes

The issue requests that the TUI streaming watchdog keep quiet but still-active chat runs visibly working instead of switching to idle or prompting solely because no recent stream delta arrived.

## Best Possible Solution

Before the watchdog clears a run or prompts the user, refresh or consume authoritative session/run state such as `sessions.list.hasActiveRun`; keep streaming/rearm when the current session still has an active run and only fall back to idle when no active run remains or a terminal event is known.

## Reproduction Assessment

Yes. Source inspection and the existing fake-timer test show a current-main path where a chat delta arms the watchdog, timer expiry clears `activeChatRunId`, sets idle, and logs the reported prompt without consulting active-run state.

## Solution Assessment

No. The current implementation treats event silence as stale state; the narrower maintainable fix is to check the gateway/session active-run signal before clearing TUI run state.
