## Summary

Keep open: current main still clears the active TUI run, switches to idle, and logs the reported watchdog prompt solely from a quiet watchdog window. Related work fixed reconnect/tool-event proof-of-life and there are open config/default PRs, but the quiet-active-run bug is still present and has a clear source-level repair path.

## What This Changes

The issue asks the TUI streaming watchdog to keep a quiet but still-active run visibly working until authoritative session or run state says it is terminal.

## Best Possible Solution

Before the watchdog clears a run or prompts the user, refresh or consume authoritative session/run state such as `sessions.list.hasActiveRun`; rearm or keep streaming when the current session still has an active run, and only fall back to idle when no active run remains or a terminal event is known.

## Reproduction Assessment

Yes. Source inspection and the existing fake-timer tests show a current-main path where a chat delta arms the watchdog, timer expiry clears `activeChatRunId`, sets idle, and logs the reported prompt without consulting active-run state.

## Solution Assessment

No. The current implementation is not the best solution because it treats event silence as stale state; the narrower maintainable fix is to check the gateway/session active-run signal before clearing TUI run state.

Send another message to continue.` when the watchdog expires after stream silence, even if the gateway/session still has an active run. Before clearing the active run or logging the prompt, have the TUI refresh or consume authoritative session state and keep/rearm streaming when the current session still reports an active run, using the existing `sessions.list` `hasActiveRun` signal or an equivalent current contract.
