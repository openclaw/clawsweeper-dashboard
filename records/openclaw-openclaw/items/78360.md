## Summary

Keep open: current main still lets the TUI streaming watchdog clear the active run, switch to idle, and show the reported prompt from silence alone. The gateway already exposes an authoritative active-run signal, but the TUI watchdog path still does not consume it; adjacent open PRs cover timeout configuration or notice cleanup, not this active-run check.

## What This Changes

This issue reports that the TUI watchdog treats quiet but still-active chat runs as idle and prompts the user to send another message.

## Best Possible Solution

Before the watchdog clears a run or prompts, refresh or consume `sessions.list.hasActiveRun` for the current session and rearm/keep streaming when the backend still reports an active run; keep timeout configurability and notice dismissal as separate follow-ups.

## Reproduction Assessment

Yes, at source level. The current fake-timer test and handler code show a delta arming the watchdog, timer expiry clearing `activeChatRunId`, switching to idle, and logging the reported prompt without checking active-run state.

## Solution Assessment

Yes, the issue's suggested active-run refresh direction is the narrow maintainable fix. Raising the timeout or only dismissing the notice mitigates symptoms but still allows the TUI to lose authoritative active-run state.
