## Summary

Keep open: current main and v2026.6.6 still expose ACPX permissionMode as plugin-global config, while the requested binding and agent-runtime ACP scopes remain limited to agent/backend/mode/cwd/label-style defaults. The request is valid and security-relevant, but it needs a maintainer security/config contract decision before implementation.

## What This Changes

Request per-binding and per-agent ACP permission defaults so different ACP sessions can use different permissionMode and nonInteractivePermissions policies instead of one global ACPX setting.

## Best Possible Solution

Design a generic, documented ACP permission-default surface for binding and agent runtime config, map it cleanly to ACPX and backend-advertised permission controls, and cover the precedence chain with focused security/config tests.

## Reproduction Assessment

Yes. Source inspection shows strict current config schemas reject permissionMode at the requested binding and agent-runtime scopes, while ACPX consumes permissionMode only from global plugin config; I did not run a live ACP harness repro in this read-only review.

## Solution Assessment

Unclear. The proposed precedence is plausible, but the best fix should settle a generic ACP permission-default contract instead of adding only ACPX-specific keys that could conflict with permissionProfile.
