## Summary

Keep open: current main and the latest release still expose ACPX permissionMode only as global plugin config, while binding and agent-runtime ACP config remain limited to agent/backend/mode/cwd/label style fields. The remaining work is useful but security-sensitive config design, not a stale or already-implemented cleanup item.

## What This Changes

The issue requests ACPX permissionMode overrides at binding and agent-runtime scope, with global plugin config only as fallback.

## Best Possible Solution

Design a generic, documented ACP permission-default surface for binding and agent runtime config, map it cleanly to ACPX and supported backend controls, and add focused validation/security tests for the precedence chain.

## Reproduction Assessment

Yes, from source inspection: the strict current config schemas reject permissionMode at the requested binding and agent-runtime scopes, and ACPX consumes permissionMode only from global plugin config. I did not run a live ACP harness repro in this read-only review.

## Solution Assessment

Unclear: the requested precedence is reasonable, but the best fix should probably use a generic ACP runtime permission-default contract rather than only adding ACPX-specific keys. Maintainers should decide how permissionMode, permissionProfile, and nonInteractivePermissions should compose before implementation.
