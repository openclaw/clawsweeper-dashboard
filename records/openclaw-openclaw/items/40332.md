## Summary

Keep open. Current main still has plugin-global ACPX startup permission policy only; binding config, per-agent ACP runtime defaults, configured binding persistence, and ACP runtime startup do not carry scoped permissionMode or nonInteractivePermissions.

## What This Changes

Request per-binding and per-agent ACPX permissionMode and nonInteractivePermissions overrides with plugin-global config as the fallback.

## Best Possible Solution

Define a typed scoped ACP permission contract, compute effective policy as binding > agent runtime > plugin global, persist and rehydrate it for configured bindings, pass it through ACP runtime startup or an explicit runtime-control seam, and document/test the security semantics.

## Reproduction Assessment

Yes. Source inspection on current main is enough: strict schemas reject the proposed keys, and the binding/runtime paths carry mode/cwd/backend/label but not scoped permission policy.

## Solution Assessment

Yes in direction, but not as an automatic narrow fix. The requested precedence matches the existing ACP override shape, while permission and non-interactive prompt semantics need maintainer security/product review before implementation.

Summary: The issue is security-sensitive because ACPX permission policy is currently plugin-global and can over-permit lower-trust ACP sessions.
