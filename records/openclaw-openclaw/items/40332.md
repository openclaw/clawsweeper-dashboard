## Summary

Keep open. Current main has global ACPX permission settings and session-level runtime permissionProfile controls, but it still lacks configured per-binding or per-agent ACP permission defaults, so the May 12 Gemini ACP over-permission use case remains valid.

## What This Changes

Request scoped ACP permission policy resolution at binding and agent-runtime levels before falling back to the global ACPX plugin config.

## Best Possible Solution

Define a typed scoped ACP permission contract resolved as binding > agent runtime > plugin global, pass the effective policy through the ACP session/runtime backend seam, and document/test the security behavior.

## Reproduction Assessment

Yes. Source inspection on current main reproduces the gap: strict binding and per-agent runtime schemas omit scoped permission fields, and configured ACP binding creation does not carry permission policy into ACPX/runtime setup.

## Solution Assessment

Yes directionally, but not as a narrow automatic fix. The requested precedence matches existing ACP override layers, while the exact permissionProfile versus ACPX permissionMode contract needs maintainer security/product judgment.
