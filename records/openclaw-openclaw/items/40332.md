## Summary

Keep open: current main and v2026.6.10 still lack per-binding or per-agent ACP permission defaults, so the central security/product request remains valid and not superseded.

## What This Changes

The issue requests scoped ACPX permissionMode overrides at the binding and agent-runtime levels instead of one gateway-wide plugin default.

## Best Possible Solution

Define a maintainer-approved scoped ACP permission contract, preferably reusing a generic ACP permission profile where possible, then implement schema, binding materialization, runtime option propagation, docs, and security-focused tests together.

## Reproduction Assessment

Yes, as a source-level feature gap: current strict schemas reject the requested scoped keys, and the runtime path consumes only plugin-global ACPX permission config. I did not run a live ACP session because the read-only source check proves the config limitation.

## Solution Assessment

Unclear. The requested precedence is plausible, but the best fix should settle the generic `permissionProfile` versus ACPX `permissionMode` contract before adding new public config keys.
