## Summary

Keep open. Current main and v2026.5.22 still have plugin-level ACPX permission policy only; binding and agent-runtime ACP config accept mode/cwd/backend/label-style fields, while scoped permission policy remains a security-sensitive config/API design decision.

## What This Changes

The issue requests per-binding and per-agent ACP permissionMode overrides with plugin-level ACPX config as the fallback.

## Best Possible Solution

Add a typed, security-reviewed scoped ACP permission contract, preferably aligned with canonical permissionProfile/runtime-option semantics, with binding > agent runtime > plugin fallback precedence, backend capability checks or explicit rejection, docs, and regression coverage.

## Reproduction Assessment

Yes, source-level. Current main rejects the requested binding and per-agent permission fields through strict schemas, and configured binding startup has no path to pass a scoped permission policy into ACP session initialization.

## Solution Assessment

Unclear pending maintainer product/security decision. The requested precedence matches existing ACP override patterns, but the durable fix should probably align permissionMode/nonInteractivePermissions with the newer generic permissionProfile runtime-option contract instead of adding a one-off parallel surface.
