## Summary

Keep open. Current main still exposes ACPX permission policy as plugin-global config only; strict agent and binding schemas, configured binding resolution, persisted binding metadata, ACP runtime inputs, runtime startup, and docs do not carry per-binding or per-agent permission policy.

## What This Changes

Add scoped ACP permission policy overrides for per-binding and per-agent runtime ACP sessions, falling back to global ACPX plugin config.

## Best Possible Solution

Define a typed scoped ACP permission contract for agent runtime and binding config, compute effective policy as binding > agent runtime > plugin global, persist/rehydrate it, pass it through ACP runtime startup, and document/test the precedence.

## Reproduction Assessment

Yes. Source inspection on current main is enough to reproduce the configuration gap: strict schemas reject the proposed keys, and the binding/runtime paths do not carry a scoped permission policy.

## Solution Assessment

Yes in direction, but not as an automatic narrow fix. The proposed precedence matches existing ACP override shape, while permission and nonInteractivePermissions semantics are security-sensitive config/runtime contract decisions.

## Security Review

Status: needs_attention

Summary: The issue is security-sensitive because the current global ACPX permission policy can over-permit lower-trust sessions in mixed-agent deployments.

Concerns:

- **[medium] Global ACPX policy can over-permit mixed-trust sessions:** `extensions/acpx/src/service.ts:68`
  - body: createLazyDefaultRuntime passes one plugin-level permissionMode and nonInteractivePermissions value into the ACPX runtime. If one ACP agent requires approve-all, lower-trust ACP bindings using the same backend inherit that broader policy because no scoped override is validated, persisted, or passed at session startup.
