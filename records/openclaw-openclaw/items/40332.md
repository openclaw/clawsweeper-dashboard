## Summary

Keep open. Current main still only supports ACPX `permissionMode` and `nonInteractivePermissions` as plugin-global config, while the requested per-binding and per-agent override surface remains unimplemented and security-sensitive.

## What This Changes

Request to add scoped ACPX permission policy overrides on `bindings[].acp` and `agents.list[].runtime.acp`, falling back to `plugins.entries.acpx.config`.

## Best Possible Solution

Define a typed scoped ACP permission contract, resolve policy as binding > agent runtime > plugin global, pass it through an explicit ACP runtime/backend seam, and document and test the security behavior.

## Reproduction Assessment

Yes. Source inspection on current main is enough: strict binding and per-agent runtime schemas omit the requested keys, and the ACP runtime ensure contract only carries session identity, mode, model/thinking, cwd, and env while ACPX receives one plugin-global permission policy.

## Solution Assessment

Yes directionally, but not as an automatic narrow bug fix. The requested precedence matches existing ACP override patterns, while permission and non-interactive prompt semantics need maintainer security/product review before becoming public config.

## Security Review

Status: needs_attention

Summary: The issue is security-sensitive because the current ACPX permission policy is gateway-global for sessions using that backend.

- **[medium] Global ACPX policy can over-permit mixed-trust sessions:** `extensions/acpx/src/service.ts:85`
  - body: `createLazyDefaultRuntime` passes one plugin-level `permissionMode` and `nonInteractivePermissions` value into the ACPX runtime.
