## Summary

Keep open. Current main still models ACPX `permissionMode` and `nonInteractivePermissions` as plugin-global settings, while the binding, per-agent runtime, persistent binding, and ACP runtime-backend contracts do not carry a scoped permission policy; the latest Gemini comment confirms this remains an operational security problem, not only a hypothetical feature request.

## What This Changes

Request scoped ACPX permission policy overrides at `bindings[].acp` and `agents.list[].runtime.acp`, falling back to the existing plugin-global ACPX config.

## Best Possible Solution

Define a typed scoped ACP permission contract, resolve it as binding > agent runtime > plugin global, pass it through the ACP runtime backend seam, and document/test the security behavior including any bounded-write profile decision.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence reproduction path: the strict binding and per-agent runtime schemas omit scoped permission keys, and ACPX still receives one plugin-global permission policy.

## Solution Assessment

Yes directionally, but not as an automatic narrow bug fix. The requested precedence matches existing ACP override layers, while public permission and non-interactive prompt semantics need maintainer security/product review before becoming a config contract.
