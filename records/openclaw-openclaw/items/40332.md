## Summary

Keep open. Current main and v2026.5.22 still only support global ACPX permissionMode/nonInteractivePermissions; binding and agent-runtime ACP config only carry mode/cwd/backend/label, so the remaining work is a security-sensitive public config/API design decision.

## What This Changes

Request scoped ACP permission policy so ACP sessions can use per-binding or per-agent permissionMode instead of one global ACPX plugin setting.

## Best Possible Solution

Add a typed, security-reviewed scoped ACP permission contract, ideally aligned with permissionProfile/runtime-option semantics, with binding > agent runtime > plugin fallback precedence, backend capability handling, docs, and regression coverage.

## Reproduction Assessment

Yes, source-level. Current main rejects the requested binding and per-agent permission fields through strict schemas, and configured binding startup has no path to pass scoped permission policy into ACP session initialization.

## Solution Assessment

Unclear pending maintainer product/security decision. The requested precedence matches existing ACP override patterns, but the durable solution should probably align permissionMode/nonInteractivePermissions with permissionProfile rather than adding a one-off parallel surface.
