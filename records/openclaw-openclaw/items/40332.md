## Summary

Keep open. Current main still does not expose scoped ACPX `permissionMode` or `nonInteractivePermissions` on bindings or per-agent ACP runtime config, and the requested precedence is security-sensitive feature work rather than a stale cleanup close.

## What This Changes

The issue requests per-binding and per-agent ACP permission policy overrides that fall back to the current global ACPX plugin configuration.

## Best Possible Solution

Define a typed scoped ACP permission contract, resolve effective policy as binding > agent runtime > plugin global, pass it through an explicit ACP runtime/backend seam, and document and test the security behavior.

## Reproduction Assessment

Yes. Source inspection on current main is enough: strict binding and per-agent ACP runtime schemas omit the proposed keys, and the runtime startup path only carries plugin-global ACPX permission policy.

## Solution Assessment

Yes in direction, but not as an automatic narrow bug fix. The requested precedence matches existing ACP override patterns, while permission and non-interactive prompt semantics need maintainer security/product review before becoming public config.

## Security Review

Status: needs_attention

Summary: The issue is security-sensitive because ACPX permission policy is currently gateway-global for all sessions using that backend.

Concerns:

- **[medium] Global ACPX policy can over-permit mixed-trust sessions:** `extensions/acpx/src/service.ts:85`
  - body: `createLazyDefaultRuntime` passes one plugin-level `permissionMode` and `nonInteractivePermissions` value into the ACPX runtime. If one ACP agent requires `approve-all`, lower-trust ACP bindings on the same backend inherit that broader startup policy because no scoped override is validated, persisted, or passed at session creation.
