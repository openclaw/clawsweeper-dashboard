## Summary

Keep open. Current main and v2026.5.7 still expose ACPX `permissionMode` and `nonInteractivePermissions` as plugin-global configuration only; the strict binding schemas, per-agent ACP runtime schemas, configured-binding materialization, and runtime startup path do not carry scoped permission policy. This is valid feature work, but it is security-sensitive public config/product design rather than stale cleanup.

## What This Changes

Request per-binding and per-agent ACPX permission policy overrides so mixed-trust ACP sessions do not all inherit the same plugin-global permission mode.

## Best Possible Solution

Define a typed scoped ACP permission contract, resolve effective policy as binding > agent runtime > plugin global, persist it for configured bindings, pass it through an explicit ACP runtime seam, and document/test reconfiguration semantics.

## Reproduction Assessment

Yes. Source inspection on current main is a high-confidence reproduction: the proposed binding and per-agent keys are absent from strict schemas, binding materialization does not compute a policy, and runtime session creation has no permission-policy input.

## Solution Assessment

Yes in direction, but not as an automatic narrow fix. The requested precedence matches the existing ACP override model, while permission and non-interactive prompt semantics need maintainer security/product review before adding public config.
