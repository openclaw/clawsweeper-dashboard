## Summary

Keep open. Current main still has only plugin-global ACPX permission config plus per-session runtime `permissionProfile` controls; it does not accept or materialize scoped per-binding or per-agent ACP permission defaults, and the recent Gemini comment adds a concrete over-permission use case.

## What This Changes

Request scoped ACP permission policy so configured ACP bindings and per-agent ACP runtime defaults can override the global ACPX `permissionMode` fallback.

## Best Possible Solution

Define one typed scoped ACP permission contract with binding > agent runtime > backend/plugin fallback precedence, apply it through ACP session initialization/runtime controls, and document/test the security behavior.

## Reproduction Assessment

Yes, source-level. Current main's strict binding and per-agent ACP runtime schemas omit scoped permission fields, and configured binding initialization does not carry any permission policy into ACP session startup.

## Solution Assessment

Unclear pending maintainer product/security choice. The requested precedence is directionally sound, but a maintainable fix should reuse the generic ACP runtime `permissionProfile` path where possible rather than hard-coding only ACPX-specific names.
