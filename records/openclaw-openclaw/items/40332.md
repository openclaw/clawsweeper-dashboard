## Summary

Keep open. Current main and the latest shipped release still expose ACPX permission policy as plugin-global config plus per-session runtime controls; scoped per-binding or per-agent ACP permission defaults remain absent and security-sensitive.

## What This Changes

Add per-binding and per-agent ACP permission defaults with binding > agent runtime > ACPX plugin fallback precedence.

## Best Possible Solution

Define one typed scoped ACP permission contract with binding > agent runtime > backend/plugin fallback precedence, apply it during ACP session initialization/runtime controls, and document/test the security behavior.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes, source-level: current main's strict binding and per-agent ACP runtime schemas omit scoped permission fields, and configured binding initialization does not carry any permission policy into ACP session startup.

## Solution Assessment

Is this the best way to solve the issue? Unclear pending maintainer product/security choice; the requested precedence is directionally sound, but the implementation should reuse the generic ACP runtime `permissionProfile` path where possible instead of hard-coding only ACPX-specific names.
