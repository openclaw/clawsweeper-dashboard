## Label Justifications

- `P2`: The global-only policy prevents practical least-privilege multi-agent ACP setups, but users retain a documented global workaround.
- `impact:security`: The issue concerns unintended sharing of automatic file-write and shell-execution authority across ACP sessions.

## Summary

Current main still has one gateway-wide ACPX permission policy, while agent and configured-binding ACP overlays only carry agent/backend/mode/cwd-style runtime choices. The concrete Gemini report confirms the least-privilege gap is operational, so this remains useful work; however, automatic write and shell authorization is a security-sensitive product contract, and the proposed ACPX-specific fields should not be added to generic core config without maintainer agreement on a backend-neutral policy model.

## What This Changes

Add per-binding and per-agent ACP permission overrides, with the existing global ACPX plugin configuration as fallback.

## Best Possible Solution

Define a backend-neutral ACP authorization profile in the session launch/runtime contract, resolve binding over agent over the existing safe global default exactly once, and let ACPX validate and translate that profile to its harness-specific permission flags with focused fresh-install, upgrade, and least-privilege tests.

## Reproduction Assessment

Yes, at source-reproduction confidence rather than a locally executed failure: current main passes one plugin-global policy into ACPX, and the detailed Gemini ACP example demonstrates the resulting non-interactive write failure and over-broad workaround.

## Solution Assessment

No, not exactly as proposed. Scoped precedence is the right product direction, but a backend-neutral ACP authorization contract is safer and more maintainable than adding ACPX-specific permissionMode fields directly to generic core agent and binding configuration.

## Risks / Open Questions

- Binding-scoped elevation can grant write and shell authority to every message matching a channel binding, so identity matching, precedence, audit visibility, and who may configure elevated profiles need an explicit security decision.
- Using the literal ACPX permissionMode enum in generic agent and binding schemas would couple core configuration to one backend and create a second policy source unless the runtime contract is designed first.
- Fresh-install and upgrade behavior must keep the current safe global default when no scoped profile exists; ambiguous fallback or partial propagation could silently over-permit sessions.
