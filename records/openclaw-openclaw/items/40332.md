## Summary

Keep open. Current main and v2026.5.22 still only expose ACPX permission policy as plugin-level config, while binding and per-agent ACP schemas plus binding startup carry only agent/backend/mode/cwd/label; the later Gemini use case confirms this is a real over-permission tradeoff that needs security/product design rather than cleanup closure.

## What This Changes

The issue requests binding-scoped and per-agent ACP permission overrides so different ACP sessions can use different ACPX permission policies instead of sharing one global plugin setting.

## Best Possible Solution

Add a typed, security-reviewed scoped ACP permission contract with binding > agent runtime > plugin fallback precedence, backend capability checks or explicit rejection, documentation, and regression coverage for fresh and upgraded configurations.

## Reproduction Assessment

Yes, source-level. Current main rejects the requested binding and per-agent permission fields through strict schemas and has no binding startup path that can pass a scoped permission policy into ACP session initialization.

## Solution Assessment

Unclear pending maintainer product/security decision. The requested precedence matches existing ACP defaults, but the maintainable fix likely needs a generic permissionProfile or ACPX per-session contract instead of only adding one plugin-specific field.
