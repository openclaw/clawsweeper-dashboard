## Summary

Keep open. Current `main` and the latest release still lack binding- or agent-scoped ACP permission defaults, and the issue has concrete discussion showing the global ACPX workaround can over-permit unrelated sessions.

## What This Changes

The issue requests binding-level and per-agent ACP permission overrides so different ACP sessions can use different permission policy instead of sharing one global ACPX setting.

## Best Possible Solution

Define one typed scoped ACP permission contract with binding > per-agent runtime > plugin fallback precedence, align it with the existing canonical `permissionProfile` control path where possible, and document/test backend support plus rejection behavior.

## Reproduction Assessment

Yes, source-level. Current `main` still omits scoped permission fields from the strict per-agent and binding ACP schemas, and configured binding startup does not pass a scoped permission option into ACP session initialization.

## Solution Assessment

Unclear pending maintainer product/security decision. The requested precedence is directionally sound, but the maintainable shape should likely align ACPX `permissionMode` and `nonInteractivePermissions` with the generic `permissionProfile` runtime-control path instead of adding only one plugin-specific field.
