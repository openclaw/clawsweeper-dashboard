## Summary

Keep open. Current main has session-level ACP runtime permission controls and plugin-global ACPX permission config, but the requested per-binding and per-agent default permission policy is still not accepted by the strict config schemas or carried through configured ACP binding initialization.

## What This Changes

Request adding ACP permission policy overrides at binding and per-agent runtime config levels, falling back to the existing ACPX plugin-global permission config.

## Best Possible Solution

Define one typed scoped ACP permission contract with binding > agent runtime > backend/plugin fallback precedence, apply it through ACP session initialization/runtime controls, and document/test the security behavior.

## Reproduction Assessment

Yes, source-level. Current main's strict binding and per-agent ACP runtime schemas omit scoped permission fields, and configured binding initialization does not carry any permission policy into ACP session startup.

## Solution Assessment

The requested precedence is directionally sound, but the exact solution needs maintainer product/security choice. A maintainable fix should reuse the generic ACP runtime permissionProfile path where possible rather than hard-coding only ACPX-specific names.
