## Summary

Keep open. Current main still only supports ACPX permission policy as plugin-global config, while the requested scoped policy is absent from strict config schemas, configured binding materialization, persisted binding metadata, ACP runtime startup, ACPX service construction, and user docs. The request remains valid and security-sensitive because mixed-trust ACP sessions can be over-permitted when one session requires broader global permissions.

## What This Changes

Request per-binding and per-agent ACPX `permissionMode` overrides, with `bindings[].acp` taking precedence over `agents.list[].runtime.acp` and global `plugins.entries.acpx.config`, plus similar consideration for `nonInteractivePermissions`.

## Best Possible Solution

Define a typed scoped ACP permission contract for both agent runtime config and ACP binding config, validate and persist those fields, compute effective policy as binding > agent runtime > plugin global, pass that policy through ACP runtime startup without silently reusing a broader global policy, and document/test the precedence.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: adding `bindings[].acp.permissionMode` or `agents.list[].runtime.acp.permissionMode` is rejected by strict schemas, and any bypassed values would be dropped before binding persistence and ACP runtime startup.

## Solution Assessment

Yes in direction, but it needs a cross-surface core/runtime contract rather than a plugin-only patch. The proposed precedence matches the existing mode/cwd/backend override model, while the security-sensitive details should be approved by a maintainer before implementation.
