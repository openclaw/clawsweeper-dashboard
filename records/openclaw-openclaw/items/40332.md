## Summary

Keep open. Current main still lacks per-binding or per-agent ACP permission defaults, and the live discussion includes a concrete Gemini ACP case showing the current global ACPX workaround can either block needed writes or over-permit unrelated sessions.

## What This Changes

The issue requests scoped ACP permission controls per binding and per agent instead of relying only on the global ACPX plugin permission mode.

## Best Possible Solution

Define one typed scoped ACP permission contract with binding > agent runtime > backend/plugin fallback precedence, align it with the generic permissionProfile path where possible, and document/test backend support and rejection behavior.

## Reproduction Assessment

Yes, source-level. Current main and v2026.5.19 omit scoped permission fields from the strict binding and per-agent ACP schemas, and configured binding initialization does not pass any scoped permission option into ACP session startup.

## Solution Assessment

Unclear pending maintainer product/security decision. The requested precedence is directionally sound, but the maintainable shape should likely align ACPX permissionMode/nonInteractivePermissions with the generic permissionProfile runtime-control path instead of adding only one plugin-specific field.
