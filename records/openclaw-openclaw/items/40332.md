## Summary

Keep open. Current main still exposes ACPX permission policy as plugin-global config only; scoped agent or binding fields are absent from strict config, binding materialization, persisted metadata, ACP runtime startup, and docs, so the mixed-trust over-permission concern remains.

## What This Changes

Request per-binding and per-agent ACPX `permissionMode` overrides, with binding config taking precedence over agent runtime config and the plugin-global fallback.

## Best Possible Solution

Define a typed scoped ACP permission contract for agent runtime and binding config, compute effective policy as binding > agent runtime > plugin global, persist and rehydrate it, pass it through ACP runtime startup, and document/test the precedence.

## Reproduction Assessment

Yes. Source inspection on current main is enough: the proposed keys are rejected by strict schemas, and no later binding, metadata, or runtime startup path has fields to carry them.

## Solution Assessment

Yes in direction, but this should be a cross-surface core/runtime contract rather than an ACPX-only patch. The proposed precedence matches existing ACP mode/cwd/backend override shape, while the security-sensitive details need maintainer approval.

## Security Review

Status: needs_attention

Summary: The issue is security-sensitive because today one plugin-global ACPX permission policy can over-permit lower-trust sessions in mixed-agent deployments.

Concerns:

- **[medium] Global ACPX policy can over-permit mixed-trust sessions:** `extensions/acpx/src/service.ts:68`
  - body: `createLazyDefaultRuntime` passes one plugin-level `permissionMode` and `nonInteractivePermissions` into the ACPX runtime. If one ACP agent requires `approve-all`, other lower-trust ACP bindings using the same backend inherit that broader policy because no scoped override is validated, persisted, or passed at session startup.
  - confidence: 0.9
