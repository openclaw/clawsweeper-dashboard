## Summary

Keep open. Current main still lacks the requested per-binding and per-agent ACP permission policy surface; ACPX permissions remain plugin-global, while the newer `permissionProfile` runtime-option mapping is session/runtime-control plumbing rather than a configurable binding or agent override.

## What This Changes

The issue requests binding-level and per-agent ACP permission policy overrides, falling back to the current ACPX plugin-global configuration.

## Best Possible Solution

Define a typed scoped ACP permission contract, resolve policy as binding > agent runtime > plugin global, pass it through an explicit ACP runtime/backend seam, and document and test the security behavior.

## Reproduction Assessment

Yes. Source inspection on current main is enough: the strict binding and per-agent runtime schemas omit the requested keys, and ACPX still receives one plugin-global permission policy.

## Solution Assessment

Yes directionally, but not as an automatic narrow bug fix. The requested precedence matches existing ACP override patterns, while public permission and non-interactive prompt semantics need maintainer security/product review before becoming config contract.
