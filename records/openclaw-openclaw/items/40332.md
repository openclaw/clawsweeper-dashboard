Keep open: current main and the latest release still lack per-binding or per-agent ACP permission defaults, so the central security/product request remains valid and not superseded.

## What This Changes

The issue requests ACP `permissionMode` overrides at binding and agent-runtime scope instead of only a global ACPX plugin setting.

## Best Possible Solution

Define one maintainer-approved scoped ACP permission-default contract, generic enough to reuse `permissionProfile` while mapping cleanly to ACPX, then document precedence and cover schema, binding materialization, runtime options, and security tests.

## Reproduction Assessment

Yes, as a source-level feature gap. Current strict binding and agent runtime schemas reject the requested scoped keys, while ACPX still consumes `permissionMode` only from global plugin config.

## Solution Assessment

Unclear. The requested precedence is plausible, but the best fix should settle a generic scoped ACP permission contract instead of adding ACPX-specific keys without a security/product decision.
