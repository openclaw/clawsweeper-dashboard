## Summary

Keep open. Current main still exposes ACPX permissions only as plugin-global config, while the strict binding/per-agent ACP schemas and persistent binding metadata do not carry scoped permission policy; the latest Gemini ACP comment adds a concrete 2026.5.7 over-permission use case.

## What This Changes

Request scoped ACPX permissionMode and nonInteractivePermissions overrides at binding and per-agent runtime levels instead of only plugin-global config.

## Best Possible Solution

Adopt a typed scoped ACP permission contract resolved as binding > agent runtime > plugin global, pass the effective policy through the ACP runtime backend seam, and document/test the security behavior.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence reproduction path: strict binding and per-agent runtime schemas omit scoped permission keys, and ACPX still receives one plugin-global permission policy.

## Solution Assessment

Yes directionally, but not as an automatic narrow fix. The requested precedence matches existing ACP override layers, while the exact permission and non-interactive semantics need maintainer security/product review.
