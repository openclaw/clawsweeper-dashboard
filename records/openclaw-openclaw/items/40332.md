## Summary

Keep open. Current main still leaves ACPX permissions as plugin-global config plus session-level runtime controls, but the strict per-binding and per-agent ACP config paths do not accept or propagate scoped permission defaults; the May 12 Gemini comment adds a concrete over-permission use case.

## What This Changes

Requested per-binding and per-agent ACP permissionMode overrides so mixed-trust ACP sessions do not share one plugin-global ACPX permission policy.

## Best Possible Solution

Define a typed scoped ACP permission contract resolved as binding > agent runtime > backend/plugin global, pass the effective policy through ACP session initialization/runtime controls, and document/test the security behavior.

## Reproduction Assessment

Yes. Source inspection on current main shows strict binding and per-agent ACP runtime schemas omit scoped permission fields, configured binding initialization passes only agent/mode/cwd/backend, and ACPX permissionMode remains plugin-global.

## Solution Assessment

Yes directionally, but not as a narrow automatic fix. The requested precedence matches existing ACP override layers, while the exact permissionProfile versus ACPX permissionMode/nonInteractivePermissions public contract needs maintainer security/product judgment.
