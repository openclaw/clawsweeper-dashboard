## Summary

Keep open. Current main and the latest release still only support global ACPX permissionMode/nonInteractivePermissions, while the requested binding and per-agent config scopes remain absent from strict schemas and binding startup.

## What This Changes

The issue requests ACP permission policy overrides at binding and per-agent runtime scope, falling back to the global ACPX plugin config.

## Best Possible Solution

Define one typed scoped ACP permission contract with binding > per-agent runtime > plugin fallback precedence, reuse the existing permissionProfile runtime-control path where possible, and document/test unsupported-backend behavior.

## Reproduction Assessment

Yes, source-level. The strict config schemas and configured-binding startup path still omit scoped ACP permission fields, so the example config would be rejected or have no effect on current main.

## Solution Assessment

Unclear pending maintainer product/security decision. The requested precedence is directionally sound, but the maintainable shape should align with the existing canonical permissionProfile runtime-control path instead of adding only a plugin-specific one-off field.
