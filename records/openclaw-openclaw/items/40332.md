## Summary

Keep open. Current main still lacks the requested per-binding and per-agent ACP permission policy surface; ACPX permissions are still resolved from plugin-global config, and the latest Gemini ACP comment adds a concrete current-release over-permission use case.

## What This Changes

Add configurable per-binding and per-agent ACP permission overrides, with plugin-global ACPX permission config as the fallback.

## Best Possible Solution

Define a typed scoped ACP permission contract, resolve it as binding > agent runtime > plugin global, pass it through an explicit ACP backend seam, and document/test the resulting security behavior including bounded artifact-write cases.

## Reproduction Assessment

Yes. Source inspection on current main `b499d8f60c1f6825ac1a56e3f3b5a64f7e226fc2` is enough: the strict binding and per-agent ACP runtime schemas omit the requested keys, and ACPX still consumes one plugin-global permission policy.

## Solution Assessment

Yes directionally, but not as an automatic narrow fix. The requested precedence matches existing ACP override patterns, while permission and non-interactive prompt semantics are public security behavior that need maintainer/product review before becoming config contract.
