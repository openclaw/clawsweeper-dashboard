## Summary

Keep open. Current main and the latest shipped v2026.5.7 release still expose ACPX permission policy only as plugin-global config, and the new Gemini use case confirms the same over-permission tradeoff is still real rather than hypothetical.

## What This Changes

The issue requests per-binding and per-agent ACPX permission policy overrides, with plugin-global permissionMode only as the fallback.

## Best Possible Solution

Define a typed scoped ACP permission contract, resolve it as binding > agent runtime > plugin global, pass it through an explicit ACP runtime/backend seam, and document/test the security behavior.

## Reproduction Assessment

Yes. Source inspection on current main is enough: the strict binding and per-agent runtime schemas omit the requested keys, and ACPX still receives one plugin-global permission policy; I did not need a live ACP run to prove the config contract gap.

## Solution Assessment

Yes directionally, but not as an automatic narrow bug fix. The requested precedence matches existing ACP override layers, while public permission and non-interactive prompt semantics need maintainer security/product review before becoming config contract.

## Security Review

Status: needs_attention

Summary: The issue is security-sensitive because one plugin-global ACPX permission policy can over-permit lower-trust sessions that share the same backend.

- **[medium] Global ACPX policy can over-permit mixed-trust sessions:** `extensions/acpx/src/service.ts:85`
