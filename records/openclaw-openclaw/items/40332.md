## Summary

Keep open. Current main and the latest release still lack per-binding or per-agent ACP permission defaults; only plugin-global ACPX permission config and per-session runtime controls exist, so this remains a security-sensitive feature/config decision.

## What This Changes

The issue requests binding-level and agent-runtime-level ACP permission overrides ahead of the existing global ACPX permissionMode fallback.

## Best Possible Solution

Define one typed scoped ACP permission contract with binding > agent runtime > plugin/backend fallback precedence, preferably reusing the generic permissionProfile runtime option where it fits, and add docs plus security-focused tests for unsupported-backend behavior.

## Reproduction Assessment

Yes, source-level. Current main and v2026.5.19 omit scoped permission fields from strict binding/per-agent schemas, and configured binding initialization does not pass any scoped permission option into ACP session startup.

## Solution Assessment

Unclear pending maintainer product/security decision. The requested precedence is directionally sound, but the maintainable shape should likely align ACPX permissionMode/nonInteractivePermissions with the generic permissionProfile runtime-control path instead of adding only one plugin-specific field.
