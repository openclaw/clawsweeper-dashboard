## Summary

Keep this PR open, but the patch is not merge-ready as-is: it fixes a real multi-bot Slack attribution problem by removing a default identity overlay, yet it also breaks the documented active-agent identity behavior for existing Slack setups without adding an opt-in/configured replacement or real behavior proof.

## What This Changes

This PR changes the Slack outbound adapter so adapter-routed sends never forward `OutboundIdentity` into `sendMessageSlack` as per-message `username`, `icon_url`, or `icon_emoji`.

## Best Possible Solution

Keep the multi-bot attribution fix, but land it behind a documented, tested Slack account/default policy that preserves existing active-agent identity behavior unless maintainers explicitly choose a breaking default flip.

## Reproduction Assessment

Yes from source, but not with a live Slack workspace: current main maps outbound agent identity into Slack `username`/icon fields and sends those to `chat.postMessage`. The visual multi-bot symptom is source-backed and consistent with the PR report, but I did not reproduce it against real Slack.

## Solution Assessment

No as-is. Dropping identity unconditionally is too broad; the maintainable fix should preserve or explicitly migrate the documented active-agent identity behavior while giving multi-bot deployments a clear default or opt-out path.
