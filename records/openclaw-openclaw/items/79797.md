## Summary

Keep open. Current main still has the reported diagnostic gap: the active `status --json` route disables channel summaries before the Telegram resolver can run, and the lower summary resolver still excludes setup-fallback plugins. Two related fix PRs were closed unmerged, and recent performance work deliberately keeps the lean JSON path off manifest-backed discovery, so this needs maintainer contract direction before a repair lane should touch it.

## What This Changes

The issue requests that `openclaw status --json` include a Telegram entry in `channelSummary` when Telegram is configured, enabled, and working.

## Best Possible Solution

Define the `status --json` contract, then either restore a lightweight configured-channel `channelSummary` path that includes setup-backed Telegram without plugin runtime or manifest-rescan regressions, or document that default JSON intentionally omits this field and reserve full channel summaries for a deeper status mode.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: `statusCommand({ json })` reaches `scanStatusJsonFast`, which passes `includeChannelSummary: false`, so `getStatusSummary` returns an empty array before Telegram setup-fallback resolution can run.

## Solution Assessment

Unclear. Flipping `includeSetupFallbackPlugins` is necessary for callers that reach `buildChannelSummary`, but it is not sufficient for the default CLI JSON path and may conflict with the lean status performance contract from https://github.com/openclaw/openclaw/issues/79129.
