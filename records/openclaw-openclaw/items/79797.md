## Summary

Keep open: current main and the latest release still intentionally keep default `status --json` off channel-summary plugin loading, while the setup-fallback resolver gap remains when channel summaries are reached.

## What This Changes

The issue asks for `openclaw status --json` to include a Telegram entry in `channelSummary` when Telegram is configured, enabled, and actively working.

## Best Possible Solution

Define the `status --json` channel-summary contract, then implement a lightweight configured-channel summary path or explicitly document that configured channel summaries belong only to a richer status mode.

## Reproduction Assessment

Yes, source-reproducible: current main's default `scanStatusJsonFast()` passes `includeChannelSummary: false`, and a current test asserts the fast JSON path stays off channel plugin summary loading. I did not run a live Telegram/WSL2 setup.

## Solution Assessment

Unclear: changing only plugin auto-enable is not the best fix because current `status --json` bypasses channel summary by policy. Maintainers should decide the JSON contract, then choose a lightweight resolver or a richer-mode-only behavior.
