## Summary

Keep open. Current `main` still has the reported diagnostic gap: `status --json` suppresses `channelSummary`, and the lower-level summary resolver still excludes setup-fallback channel plugins such as Telegram; related fix PRs were closed unmerged and an open lean-JSON PR makes this a contract decision.

## What This Changes

The issue requests that `openclaw status --json` include a Telegram entry in `channelSummary` when Telegram is configured, enabled, and working.

## Best Possible Solution

Define the `status --json` contract, then either populate `channelSummary` through lightweight read-only setup-fallback metadata without starting channel runtimes or document/remove the empty field from the lean JSON payload.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: `status --json` reaches `scanStatusJsonFast`, which passes `includeChannelSummary: false`, so the JSON payload can keep returning `channelSummary: []` for configured Telegram. I did not run a live Telegram setup.

## Solution Assessment

Unclear. Enabling setup fallback in `buildChannelSummary` is necessary for setup-backed channels, but it is not sufficient for the reported CLI path while fast JSON disables channel summaries; maintainers should settle that contract first.
