## Summary

Keep open. Current main still has the reported diagnostic gap: the CLI JSON path suppresses channel summaries before Telegram resolution can run, and the lower-level channel-summary resolver still disables setup-fallback plugins; the two linked fix PRs were closed unmerged.

## What This Changes

The issue asks for `openclaw status --json` to include a Telegram `channelSummary` entry when Telegram is configured, enabled, and delivering successfully.

## Best Possible Solution

Confirm the `status --json` contract, then either populate `channelSummary` from read-only setup-fallback metadata without starting channel runtimes or document/remove the empty field from the lean JSON path.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: `statusCommand({ json: true })` uses `scanStatusJsonFast`, which passes `includeChannelSummary: false`, so `getStatusSummary` returns `channelSummary: []` before Telegram summary resolution can run. I did not run a live Telegram setup in this read-only review.

## Solution Assessment

Unclear. Enabling setup fallback in `buildChannelSummary` is necessary for summary callers that reach it, but the reported CLI also needs the fast JSON contract changed or clarified.
