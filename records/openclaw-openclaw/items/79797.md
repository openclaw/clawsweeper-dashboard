## Summary

Keep open. Current main still has a source-reproducible status JSON diagnostic gap, and an open closing PR is the active implementation candidate; this issue should not be closed before that PR is reviewed, landed, or replaced.

## What This Changes

The issue asks for a configured active Telegram channel to appear in `openclaw status --json` `channelSummary` instead of returning an empty array.

## Best Possible Solution

Update the JSON status path and setup-safe resolver so configured setup-backed channels appear in `channelSummary` without starting channel runtimes, with regression tests and redacted real CLI output from a configured channel.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: `statusCommand({ json })` reaches `scanStatusJsonFast`, which passes `includeChannelSummary: false`, so `getStatusSummary` can return `channelSummary: []` without loading Telegram summary metadata.

## Solution Assessment

Unclear. Enabling setup fallback in `buildChannelSummary` is relevant, but the reported CLI command also needs the fast JSON path to request a channel summary or the contract must be documented differently.
