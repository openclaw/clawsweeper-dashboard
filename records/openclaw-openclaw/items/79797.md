## Summary

Keep open: current main and v2026.6.6 still keep default `status --json` off channel-summary plugin loading, and the linked fix PR was closed unmerged. The remaining work is a maintainer decision on how to restore or redefine `channelSummary` without undoing the lean JSON status path.

## What This Changes

The issue asks for `openclaw status --json` to include a Telegram entry in `channelSummary` when Telegram is configured, enabled, and delivering messages.

## Best Possible Solution

Define the `status --json` channel-summary contract, then implement a lightweight configured-channel summary path or explicit `--all` behavior with regression coverage for Telegram setup fallback and JSON fast-path performance.

## Reproduction Assessment

Yes for source reproduction: current main's default `scanStatusJsonFast()` passes `includeChannelSummary: false`, and current tests assert the fast JSON path stays off channel plugin summary loading. I did not run a live Telegram/WSL2 setup.

## Solution Assessment

Unclear. Changing only plugin auto-enable is not the best fix because current `status --json` bypasses channel summary by policy; maintainers should decide the JSON contract, then choose a lightweight resolver or `--all` behavior.
