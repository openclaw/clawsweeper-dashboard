## Summary

Keep this open. Current main has model allowlists, fallback chains, and configure-time multi-model selection, but the first-run onboarding path still prompts for one default model and applies only that primary model, so the requested onboarding behavior remains missing and needs maintainer UX direction.

## What This Changes

Request to let onboarding register multiple LLM models or endpoints while preserving one default model for cost/performance tradeoffs.

## Best Possible Solution

Add a first-run onboarding step after provider auth/default selection that preserves the chosen primary model and lets users select additional provider-scoped models into `agents.defaults.models` and `agents.defaults.model.fallbacks` using the existing model-picker helpers. Keep automatic routing based on task complexity out of this issue unless maintainers explicitly expand the scope.

## Reproduction Assessment

Not applicable as a runtime bug report; this is a feature gap. There is a high-confidence source-level behavior check: current onboarding reaches `promptDefaultModel`/`applyPrimaryModel`, while the multi-select allowlist/fallback helpers are only wired through configure and model-picker tests.

## Solution Assessment

Unclear until maintainer product direction is set. The narrow maintainable path appears to be reusing `promptModelAllowlist`, `applyModelAllowlist`, and `applyModelFallbacksFromSelection` in `src/wizard/setup.ts`, not designing automatic model routing here.
