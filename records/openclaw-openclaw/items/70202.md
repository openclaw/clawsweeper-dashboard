## Summary

Keep open. Current main has global model allowlists, fallback chains, and configure-time multi-select, but first-run onboarding still prompts for and applies a single default model, and no linked PR or canonical issue supersedes the onboarding-specific gap.

## What This Changes

Request first-run onboarding to let users select multiple usable models while choosing one default, so cost and capability tradeoffs do not require post-onboarding reconfiguration.

## Best Possible Solution

Add a first-run onboarding step that writes the existing `agents.defaults.models` allowlist and fallback chain around the chosen primary, leaving automatic routing to a separate product decision.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. The high-confidence source check is that first-run onboarding still calls `promptDefaultModel`/`applyPrimaryModel`, while the multi-select allowlist path is confined to configure.

## Solution Assessment

Unclear until maintainers choose the onboarding UX. The narrow maintainable direction is to reuse `promptModelAllowlist`, `applyModelAllowlist`, and `applyModelFallbacksFromSelection` rather than adding new config or task-complexity routing here.
