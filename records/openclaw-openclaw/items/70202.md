## Summary

Keep open. Current main has model allowlists, fallback chains, and session/UI switching, but first-run onboarding still applies a single default-model selection and the onboarding-specific multi-model setup gap has not been superseded.

## What This Changes

Request to let first-run onboarding configure multiple available models while choosing one default model to reduce cost and switching friction.

## Best Possible Solution

Add a first-run onboarding step that reuses the existing model allowlist and fallback helpers around one chosen primary model, leaving automatic task-complexity routing for a separate decision.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. The high-confidence source check is that first-run onboarding still calls `promptDefaultModel`/`applyPrimaryModel`, while the multi-select allowlist path is confined to configure.

## Solution Assessment

Unclear until maintainers choose the onboarding UX. The narrow maintainable direction is to reuse `promptModelAllowlist`, `applyModelAllowlist`, and `applyModelFallbacksFromSelection` rather than adding new config or automatic routing here.
