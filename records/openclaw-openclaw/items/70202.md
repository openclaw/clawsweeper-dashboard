## Summary

Keep open. Current main has model allowlists, fallback chains, and configure-time multi-model selection, but first-run onboarding still prompts for and applies one default model only, so the requested onboarding behavior remains missing.

## What This Changes

Request to let onboarding select multiple LLM models while keeping one default so users can balance cost, latency, and capability.

## Best Possible Solution

Add an onboarding step that preserves the chosen primary model and uses the existing allowlist/fallback helpers to populate `agents.defaults.models` and `agents.defaults.model.fallbacks`; keep automatic routing as separate future work.

## Reproduction Assessment

Not applicable as a runtime bug report; this is a feature gap. The high-confidence source check is that `runSetupWizard` still reaches `promptDefaultModel`/`applyPrimaryModel`, while multi-model selection is only wired through configure and model-picker helpers.

## Solution Assessment

Unclear until maintainers choose the onboarding UX. The narrow maintainable direction appears to be reusing `promptModelAllowlist`, `applyModelAllowlist`, and `applyModelFallbacksFromSelection` in `src/wizard/setup.ts` rather than designing task-complexity routing here.
