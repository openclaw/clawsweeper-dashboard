## Summary

Keep open. Current main has multi-model config, fallbacks, and configure-time multi-select, but first-run onboarding still prompts for one default model and the onboarding UX/policy needs maintainer judgment.

## What This Changes

Allow onboarding to configure multiple models while still choosing one default model to reduce cost and friction.

## Best Possible Solution

Add a first-run onboarding step that reuses the existing allowlist and fallback helpers to seed agents.defaults.models and agents.defaults.model.fallbacks around the chosen primary; keep automatic routing separate.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. The source check is high-confidence: runSetupWizard calls promptDefaultModel/applyPrimaryModel and has no promptModelAllowlist path.

## Solution Assessment

Unclear until maintainers choose the onboarding UX. The narrow maintainable path is to reuse promptModelAllowlist, applyModelAllowlist, and applyModelFallbacksFromSelection instead of adding new config or automatic routing here.
