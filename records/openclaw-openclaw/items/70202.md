## Summary

Keep open. Current main has configure-time multi-model allowlist/fallback support, but the classic first-run onboarding path still prompts for one default model, and adding multi-selection there needs maintainer UX and default-policy judgment.

## What This Changes

Request first-run onboarding support for selecting multiple usable models while choosing one default model to reduce cost and switching friction.

## Best Possible Solution

Add an optional first-run onboarding step that seeds `agents.defaults.models` and `agents.defaults.model.fallbacks` around the chosen primary model, while splitting UI switching and automatic routing into separate follow-ups.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. Source inspection gives a high-confidence current-main gap: setup uses `promptDefaultModel`/`applyPrimaryModel`, while the multi-select allowlist path lives in configure/model-picker code.

## Solution Assessment

Unclear until maintainers choose the onboarding UX. If accepted, the narrow maintainable path is to reuse `promptModelAllowlist`, `applyModelAllowlist`, and `applyModelFallbacksFromSelection` rather than add a new config contract or automatic routing here.
