## Summary

Keep open. Current main and v2026.6.8 still have single-pass provider/model onboarding; the reusable multi-model allowlist and fallback helpers exist in configure/model-picker paths, but the first-run onboarding UX and default/fallback policy remain unimplemented and need maintainer scoping.

## What This Changes

The issue requests first-run onboarding support for configuring multiple models and selecting one default model to reduce cost and switching friction.

## Best Possible Solution

Maintainers should choose the canonical onboarding scope, then implement a narrow first-run repeat-provider/default/fallback flow using the existing model-picker allowlist and fallback helpers while splitting UI switching and automatic routing into separate work.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request. Source inspection confirms current main and v2026.6.8 still use a single-pass onboarding provider/model setup while multi-model controls live in configure/model flows after onboarding.

## Solution Assessment

Unclear until maintainers choose the product scope and canonical issue. The best implementation direction is to reuse `promptModelAllowlist`, `applyModelAllowlist`, and `applyModelFallbacksFromSelection`, while preserving scripted and explicit auth-choice behavior.
