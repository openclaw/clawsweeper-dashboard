## Summary

Keep open. The cancellation problem is real and narrow, but this PR should not merge as-is because it targets stale package-level provider files, current main now runs the batch providers from extensions, production batch options still cannot carry an AbortSignal, and the proposed polling sleeps leave listener cleanup bugs.

## What This Changes

The PR adds optional AbortSignal parameters to the old memory-host OpenAI, Voyage, and Gemini batch helper paths and makes several polling sleeps and fetch calls partially abort-aware.

## Best Possible Solution

Retarget this as a narrow replacement fix on current main: add an optional cancellation contract through the memory batch runtime, thread it through the OpenAI/Gemini/Voyage provider adapters and batch helpers, use one cleanup-safe cancellable sleep pattern, pass signals into relevant HTTP calls, and cover the behavior with focused provider batch tests.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists: current main has no signal in MemoryEmbeddingBatchOptions or the memory-core batchEmbed call, and provider polling still uses bare setTimeout sleeps. The PR diff also shows listener registration without normal-path cleanup.

## Solution Assessment

No. The proposed branch is useful prior art, but it is not the best fix because it edits stale provider paths and only partially implements cancellation; a current-main replacement in the extension provider files and shared batch contract is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Thread the signal from the production batch API:** `packages/memory-host-sdk/src/host/batch-runner.ts:10`
  - body: Adding `signal` only to `EmbeddingBatchExecutionParams` does not make disconnect cancellation reachable. The production `MemoryEmbeddingBatchOptions` contract and memory-core `batchEmbed` call still have no signal field, so provider adapters will keep passing `undefined` in normal runs.
  - confidence: 0.91
- **[P2] Clean up abort listeners after each polling sleep:** `packages/memory-host-sdk/src/host/batch-openai.ts:192-201`
  - body: Each polling sleep registers an abort listener but never removes it when the timer resolves normally, and already-aborted signals are not rejected before sleeping. Long polls retain one stale listener per interval; the same pattern appears in the Gemini and Voyage additions.
  - confidence: 0.9
- **[P2] Pass the signal into Gemini status polling:** `packages/memory-host-sdk/src/host/batch-gemini.ts:160`
  - body: `fetchGeminiBatchStatus` gains a `signal` parameter, but the patch does not pass it from `waitForGeminiBatch` or attach it to the status request. Aborts during an in-flight Gemini status request will still wait on the network call.
  - confidence: 0.84
