## Summary

Keep open. The cancellation goal is valid, but this PR is stale against current main's provider-plugin batch files and still does not make disconnect cancellation reachable through the production memory batch API.

## What This Changes

The PR adds optional AbortSignal plumbing to memory-host batch execution and OpenAI, Gemini, and Voyage batch polling/fetch helpers.

## Best Possible Solution

Retarget the fix to current main by adding an optional cancellation contract through the memory batch runtime and provider-plugin helpers, with cleanup-safe cancellable sleeps and focused provider tests.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists: current main has no signal field in the production memory batch options/call path, and OpenAI/Gemini/Voyage polling waits on uncancelable timers or sleeps.

## Solution Assessment

No. The intended direction is right, but this branch is not the best implementation because it changes stale package paths and leaves production cancellation only partially wired.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Thread the signal from the production batch API:** `packages/memory-host-sdk/src/host/batch-runner.ts:10`
  - body: Adding signal to this low-level execution type does not make disconnect cancellation reachable. The production MemoryEmbeddingBatchOptions contract and memory runtime still do not accept or forward a signal, so normal OpenAI/Gemini/Voyage batch runs continue with params.signal undefined.
  - confidence: 0.91
- **[P2] Remove abort listeners after polling sleeps:** `packages/memory-host-sdk/src/host/batch-openai.ts:192-201`
  - body: Each polling sleep registers an abort listener, but the listener is not removed when the timer resolves normally and already-aborted signals are not rejected before sleeping. Long polls retain stale listeners and can still miss cancellation; the same pattern appears in the Gemini and Voyage additions.
  - confidence: 0.9
- **[P2] Pass the signal into Gemini status polling:** `packages/memory-host-sdk/src/host/batch-gemini.ts:160`
  - body: fetchGeminiBatchStatus gains a signal parameter, but the patch does not consistently attach it to the status request from waitForGeminiBatch. Aborts during an in-flight Gemini status poll can still wait for the network call instead of stopping promptly.
  - confidence: 0.84
