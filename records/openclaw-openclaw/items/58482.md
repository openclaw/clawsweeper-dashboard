## Summary

Keep this PR open because the cancellation gap still exists on current main, but the branch is not mergeable as-is: it targets stale/deleted memory-host provider files, does not expose a production AbortSignal contract, and lacks required real behavior proof.

## What This Changes

The PR adds optional AbortSignal parameters to memory-host batch-runner and OpenAI/Gemini/Voyage batch helper code for polling and HTTP cancellation.

## Best Possible Solution

Define an additive AbortSignal contract on the current memory batch API, thread it through memory-core and provider adapters into extensions/openai, extensions/google, and extensions/voyage, use abortable sleeps with listener cleanup plus HTTP signal forwarding, and add focused cancellation tests.

## Reproduction Assessment

Yes. By source inspection, current main has no signal field in MemoryEmbeddingBatchOptions, memory-core does not forward a signal, and the OpenAI/Gemini/Voyage provider batch loops wait on uncancelable timers.

## Solution Assessment

No. The direction is useful, but this PR is not the best merge path because it targets stale/deleted package provider files and only partially wires cancellation; a current-main retarget is safer.
