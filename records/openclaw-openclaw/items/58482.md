## Summary

Keep open: the underlying cancellation problem is still source-reproducible on current main, but this PR is not merge-ready because it targets stale legacy helper paths, is conflicting, and lacks required after-fix real behavior proof.

## What This Changes

The PR adds optional AbortSignal plumbing to memory-host batch polling helpers for OpenAI, Voyage, and Gemini.

## Best Possible Solution

Retarget cancellation to the current provider-plugin path: add a signal-aware memory batch contract, forward it from memory-core into provider adapters, cancel sleeps and in-flight HTTP requests cleanly, and add focused regression coverage plus redacted real behavior proof.

## Reproduction Assessment

Yes. Source inspection on current main shows the memory batch contract has no signal field, memory-core cannot forward one, and the live OpenAI/Gemini/Voyage loops wait on uncancelable timers.

## Solution Assessment

No. The direction is useful, but this branch is not the best merge path because it targets stale package helper files and still leaves incomplete cancellation semantics; the safer fix belongs in the current provider-plugin contract and implementations.

  - body: Adding `signal` to the legacy batch-runner type does not make disconnect cancellation work on current main: production OpenAI, Gemini, and Voyage batch code now lives under `extensions/*/embedding-batch.ts`, and `MemoryEmbeddingBatchOptions` still has no signal to carry through memory-core into provider runtimes.

  - confidence: 0.9

  - body: `waitForOpenAiBatch` accepts a signal, but the status fetch inside the wait loop still calls `fetchOpenAiBatchStatus` without passing it, so an in-flight status request remains uncancelable. This leaves disconnect cancellation incomplete even in the legacy files touched by the PR.
  - confidence: 0.87
