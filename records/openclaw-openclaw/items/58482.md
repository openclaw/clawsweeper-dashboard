## Summary

Keep open. The cancellation problem is credible, but this PR is not mergeable as-is because it targets stale memory-host provider helper files, misses the current provider-plugin contract, and still has no after-fix real behavior proof.

## What This Changes

The branch adds optional AbortSignal plumbing to memory-host batch-runner and legacy OpenAI/Gemini/Voyage batch helper files.

## Best Possible Solution

Retarget cancellation to the current MemoryEmbeddingBatchOptions path, memory-core call site, provider adapters, and extensions/openai, extensions/google, and extensions/voyage batch implementations, with listener-clean abortable sleeps, HTTP signal forwarding, focused tests, and redacted real behavior proof.

## Reproduction Assessment

Yes. By source inspection, current main has no signal field in MemoryEmbeddingBatchOptions, memory-core cannot forward one, and the live OpenAI/Gemini/Voyage batch loops wait on uncancelable timers.

## Solution Assessment

No. The direction is useful, but the proposed patch is not the best merge path because it targets stale package files and only partially solves cancellation semantics; a current provider-plugin retarget is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Retarget cancellation to the live provider batch path:** `packages/memory-host-sdk/src/host/batch-openai.ts:67`
  - body: The branch adds signal support under packages/memory-host-sdk/src/host/batch-*.ts, but those provider batch files no longer exist on current main. Production OpenAI, Gemini, and Voyage batch work now lives under extensions/*/embedding-batch.ts, and MemoryEmbeddingBatchOptions still has no signal, so the change cannot deliver disconnect cancellation as proposed.
  - confidence: 0.94
- **[P2] Clean up polling abort listeners:** `packages/memory-host-sdk/src/host/batch-openai.ts:191-200`
  - body: Each polling sleep registers an abort listener but never removes it when the timer resolves normally, and it does not check an already-aborted signal before sleeping. Long polls can retain stale listeners, and pre-aborted requests can still wait a full interval; the same pattern appears in the Gemini and Voyage additions.
  - confidence: 0.9
- **[P2] Forward the signal into Gemini status fetches:** `packages/memory-host-sdk/src/host/batch-gemini.ts:160`
  - body: fetchGeminiBatchStatus gains a signal parameter, but the status HTTP request still does not attach it to withRemoteHttpResponse, and the wait path does not pass it through. Aborts during an in-flight Gemini status poll can continue waiting on the network call instead of stopping promptly.
  - confidence: 0.84
