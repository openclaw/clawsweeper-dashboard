## Summary

Keep this PR open, but it is not merge-ready: the cancellation problem is real, while the branch is dirty against current main, targets legacy provider batch files instead of the live plugin implementations, still has cancellation gaps, and lacks real behavior proof.

## What This Changes

The PR adds optional AbortSignal plumbing to memory-host batch-runner and legacy OpenAI/Gemini/Voyage batch helper files for canceling batch polling.

## Best Possible Solution

Retarget cancellation to MemoryEmbeddingBatchOptions, the memory-core batch call site, provider adapters, and the current extensions/openai, extensions/google, and extensions/voyage batch implementations with listener-clean abortable sleeps, HTTP signal forwarding, focused tests, and redacted real behavior proof.

## Reproduction Assessment

Yes. Source inspection on current main shows the memory batch contract has no signal field, memory-core cannot forward one, and the live OpenAI/Gemini/Voyage polling loops wait on uncancelable timers.

## Solution Assessment

No. The direction is useful, but this branch is not the best merge path because it targets stale package provider files and still leaves incomplete cancellation semantics; the safer fix belongs in the current provider-plugin contract and implementations.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Retarget cancellation to the live provider path:** `packages/memory-host-sdk/src/host/batch-runner.ts:10`
  - body: Adding `signal` here changes the legacy batch helper contract, but current main's production OpenAI, Gemini, and Voyage batch code lives under `extensions/*/embedding-batch.ts`, and `MemoryEmbeddingBatchOptions` still has no signal. As written, disconnect cancellation cannot reach the live `batchEmbed` path.
  - confidence: 0.94
- **[P2] Clean up abort listeners after polling sleeps:** `packages/memory-host-sdk/src/host/batch-openai.ts:191-200`
  - body: The new polling sleep registers an abort listener but never removes it when the timer resolves normally, and it does not handle an already-aborted signal before starting the timer. Long polls retain one listener per interval and pre-aborted requests can keep waiting; the same pattern appears in the Gemini and Voyage additions.
  - confidence: 0.9
- **[P2] Forward signals into status HTTP polls:** `packages/memory-host-sdk/src/host/batch-openai.ts:156`
  - body: `waitForOpenAiBatch` accepts a signal, but its status fetch still does not pass that signal into `fetchOpenAiBatchStatus`, so an in-flight status request remains uncancelable; the Gemini status path has the same gap. This leaves disconnect cancellation incomplete even on the legacy files touched by the PR.
  - confidence: 0.87
