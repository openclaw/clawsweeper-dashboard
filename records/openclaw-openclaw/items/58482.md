## Summary

Keep open: the cancellation goal is useful and current main still has uncancelable memory batch polling, but this branch is not merge-ready because it targets stale package provider files, leaves the live provider-plugin path without a signal contract, has polling listener cleanup gaps, and lacks required real behavior proof.

## What This Changes

The PR adds optional AbortSignal plumbing to memory-host batch-runner and OpenAI/Gemini/Voyage batch polling and fetch helpers.

## Best Possible Solution

Define an additive AbortSignal field on the current memory batch API, propagate it through memory-core and OpenAI/Gemini/Voyage adapters into extension batch HTTP calls and abortable sleeps with cleanup, then add focused cancellation tests.

## Reproduction Assessment

Yes. By source inspection, current main has no signal field in MemoryEmbeddingBatchOptions, memory-core does not forward a signal, and the OpenAI/Gemini/Voyage polling loops wait on uncancelable timers.

## Solution Assessment

No. The direction is sound, but this PR should be retargeted to the current extension-owned provider batch files and current memory batch API instead of patching stale package provider files.

## Review Findings

Overall confidence: 0.9

Full review comments:

- **[P1] Retarget cancellation to the live provider path:** `packages/memory-host-sdk/src/host/batch-runner.ts:7-10`
  - body: The diff adds signal support under packages/memory-host-sdk/src/host/batch-*.ts, but current main's production OpenAI, Gemini, and Voyage batch code lives in extensions/*/embedding-batch.ts and MemoryEmbeddingBatchOptions still has no signal. After resolving this branch as-is, disconnect cancellation would remain unreachable in the live memory provider path.
  - confidence: 0.94
- **[P2] Clean up polling abort listeners:** `packages/memory-host-sdk/src/host/batch-openai.ts:191-200`
  - body: Each new polling sleep registers an abort listener but never removes it when the timer resolves normally, and it misses signals that were already aborted before the listener is attached. Long polls can retain stale listeners, and pre-aborted requests still wait a full interval; the same pattern appears in the Gemini and Voyage additions.
  - confidence: 0.9
- **[P2] Forward the signal into Gemini status fetches:** `packages/memory-host-sdk/src/host/batch-gemini.ts:160`
  - body: fetchGeminiBatchStatus gains a signal parameter, but the status HTTP request does not attach it to withRemoteHttpResponse. Aborts during an in-flight Gemini status poll can continue waiting on the network call instead of stopping promptly.
  - confidence: 0.84
