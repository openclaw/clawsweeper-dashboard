## Summary

Keep open: the cancellation problem still appears present on current main, but this PR is not correct to merge because it targets stale memory-host provider files while the production batch helpers now live in provider plugins, and the signal is still not wired from the production memory batch API.

## What This Changes

The PR adds optional AbortSignal plumbing to old memory-host OpenAI, Gemini, and Voyage batch polling/fetch helpers.

## Best Possible Solution

Add an optional AbortSignal contract through MemoryEmbeddingBatchOptions, pass it from memory-core into the provider runtimes, update the current extension batch helpers to cancel sleeps and HTTP requests with listener cleanup, and cover OpenAI, Gemini, and Voyage with focused tests.

## Reproduction Assessment

Yes. Source inspection of current main shows the production memory batch contract has no signal field, memory-core does not forward one, and the OpenAI/Gemini/Voyage polling loops wait on uncancelable timers.

## Solution Assessment

No. The intended cancellation direction is right, but this branch is stale and incomplete; the maintainable fix needs to target the current provider-plugin batch files and wire the signal from the production memory runtime.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Thread the signal from the production batch API:** `packages/memory-host-sdk/src/host/batch-runner.ts:10`
  - body: Adding signal only to EmbeddingBatchExecutionParams does not make disconnect cancellation reachable. Current main's MemoryEmbeddingBatchOptions and memory-core batchEmbed call path still have no signal to provide, so normal OpenAI/Gemini/Voyage batch runs continue with params.signal undefined.
  - confidence: 0.91
- **[P2] Clean up abort listeners after polling sleeps:** `packages/memory-host-sdk/src/host/batch-openai.ts:192-201`
  - body: Each polling sleep registers an abort listener but never removes it when the timer resolves normally, and already-aborted signals are not rejected before sleeping. Long polls can retain stale listeners and still miss cancellation; the same pattern appears in the Gemini and Voyage additions.
  - confidence: 0.9
- **[P2] Pass the signal into Gemini status polling:** `packages/memory-host-sdk/src/host/batch-gemini.ts:160`
  - body: fetchGeminiBatchStatus gains a signal parameter, but waitForGeminiBatch still does not consistently attach it to the status request. Aborts during an in-flight Gemini status poll can continue waiting on the network call instead of stopping promptly.
  - confidence: 0.84
