## Summary

Keep this PR open: the cancellation problem is still valid, but the branch is not mergeable without real behavior proof and a retarget to the current provider-plugin batch path.

## What This Changes

The PR threads optional AbortSignal parameters through the old memory-host batch runner and OpenAI/Gemini/Voyage package batch helpers.

## Best Possible Solution

Add an AbortSignal to the current memory batch contract, thread it from memory-core through the provider adapters into extensions/openai, extensions/google, and extensions/voyage, use abortable sleeps with listener cleanup plus HTTP signal forwarding, and add focused cancellation tests.

## Reproduction Assessment

Yes. By source inspection, current main has no signal field in MemoryEmbeddingBatchOptions, memory-core does not forward a signal, and the provider batch loops wait on uncancelable timers.

## Solution Assessment

No. The direction is useful, but this PR is not the best merge path because it targets stale package batch files and only partially implements cancellation; a current-main retarget is safer.
