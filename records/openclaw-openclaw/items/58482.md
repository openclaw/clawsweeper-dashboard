## Summary

The PR should stay open because the cancellation problem still exists on current main, but the branch is conflicting and targets provider batch files that were moved into provider plugins after the PR opened; the current patch needs a rebase/retarget plus real behavior proof before merge.

## What This Changes

The branch adds optional AbortSignal parameters to memory-host batch runner and OpenAI, Voyage, and Gemini batch polling helpers, forwarding the signal into some sleeps and HTTP calls.

## Best Possible Solution

Retarget cancellation into MemoryEmbeddingBatchOptions, memory-core batch invocation, provider adapters, and the live OpenAI/Gemini/Voyage plugin batch files using a shared abort-safe wait/request pattern, with focused regression coverage and redacted real-run proof.

## Reproduction Assessment

Yes, at source level. Current main still has uncancelable polling sleeps in the live OpenAI/Gemini/Voyage batch implementations and no signal in the MemoryEmbeddingBatchOptions path from memory-core to provider batch runtimes.

## Solution Assessment

No, not as currently proposed. The right fix is to rebase onto the provider-plugin move and thread a signal through the current memory batch contract and all provider polling/upload/status/download calls with listener cleanup and already-aborted handling.
