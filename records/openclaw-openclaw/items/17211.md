## Summary

Keep open. Current main has compaction checkpoint snapshot recovery and UI branch/restore controls, but it still does not implement the requested native checkpoint engine, deterministic replay packets, task-boundary contract fields, diffing, or validation hooks; the existing maintainer discussion also explicitly kept this umbrella open.

## What This Changes

Request a native checkpoint engine with create/list/resume/diff APIs, structured task-boundary state, deterministic replay packets, and validation hooks for reliable long-running handoffs.

## Best Possible Solution

Keep this as the native checkpoint/replay umbrella and design a versioned Gateway/session persistence plus runtime/context-engine contract, coordinating with the related explicit session snapshot work instead of treating current compaction snapshots as complete.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. The current-main gap is high-confidence from registered Gateway methods, protocol schemas, server behavior, tests, docs, and the absence of exact checkpoint/replay method names.

## Solution Assessment

No current implementation solves the issue. The existing compaction snapshot branch/restore path is useful but not the best or complete solution for deterministic checkpoint/replay/diff semantics; the maintainable path is a versioned protocol and runtime contract design.
