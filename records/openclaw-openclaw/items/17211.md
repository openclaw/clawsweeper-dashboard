## Summary

Keep open. Current main still only provides compaction-scoped checkpoint inspection and snapshot branch/restore; it does not implement the requested native checkpoint engine with create/list/resume/diff APIs, deterministic replay packets, task-boundary contracts, diffing, or validation hooks. The live discussion and related work also distinguish the merged compaction-checkpoint recovery surface from this broader checkpoint/replay umbrella.

## What This Changes

Requests a native checkpoint engine with checkpoint.create/list/resume/diff APIs, deterministic replay packets, structured task-boundary state, diffing, and validation hooks.

## Best Possible Solution

Design a versioned native checkpoint/replay contract across Gateway/session protocol, persistence, context-engine/runtime, UI, docs, validation policy, and tests while preserving current compaction branch/restore as an adjacent recovery primitive.

## Reproduction Assessment

Not applicable: this is a feature request rather than a broken current-main behavior report. The remaining gap is source-verifiable from the Gateway method registry, protocol schema, server handlers, context-engine interface, tests, docs, and existing discussion.

## Solution Assessment

No current implementation solves the issue. Treating existing compaction branch/restore as deterministic resume would conflate snapshot recovery with the requested native replay/diff engine; the maintainable path is a maintainer-led API and replay-safety design.
