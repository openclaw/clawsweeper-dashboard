## Summary

Keep open. Current main still builds Feishu card headers by concatenating raw `identity.emoji` with the identity name in reply, streaming, presentation, and direct outbound card paths; the card builders then serialize that title verbatim. The adjacent card-header toggle PR is related UX work, but it does not sanitize descriptive non-emoji identity text.

## What This Changes

The issue requests that Feishu card message headers stop displaying descriptive non-emoji text from `identity.emoji` and instead use the agent name or an emoji-only prefix.

## Best Possible Solution

Add a Feishu-local shared card-header title resolver used by reply, streaming, presentation, and direct outbound card paths. It should preserve explicit presentation titles, keep valid identity names, include `identity.emoji` only when it is an emoji-only sequence under the chosen policy, drop descriptive prose from identity-derived headers, and add focused Feishu regression tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: set `identity.emoji` to descriptive text plus emojis, send a long or card-forced Feishu message, and the current card header construction passes that raw string into the Feishu card title.

## Solution Assessment

Yes, a narrow Feishu-local sanitizer is the best maintainable direction. Changing the global identity parser would risk breaking other identity uses, while the leak is caused by Feishu card-header rendering.
