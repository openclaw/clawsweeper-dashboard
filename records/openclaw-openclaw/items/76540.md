## Summary

Keep open. The PR targets a valid startup-planning regression for configured speech providers, but the new collector does not follow the documented active-persona TTS provider path, so persona-selected speech plugins can still be omitted from Gateway startup.

## What This Changes

The PR adds Gateway startup-planner logic, tests, and a changelog entry so configured `messages.tts` speech-provider plugins start with the Gateway.

## Best Possible Solution

Align startup speech-provider discovery with effective TTS provider resolution, including active personas and disablement semantics, then land with focused regression coverage.

## Reproduction Assessment

Yes, at source level: current main's startup planner has channel/root/harness/memory activation paths but no speech-provider activation path, matching the linked v2026.5.2 report. On the PR branch, the remaining persona gap is also source-reproducible because TTS resolution selects persona providers while the new collector skips `personas`.

## Solution Assessment

No. The approach is the right narrow layer, but it should reuse or mirror effective TTS provider selection closely enough that documented persona providers and disabled provider blocks do not drift from startup activation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Include persona-selected speech providers in startup:** `src/plugins/gateway-startup-plugin-ids.ts:223-225`
  - body: The collector reserves `personas` and never walks the active persona, but TTS selection can choose `messages.tts.personas.<id>.provider` before `messages.tts.provider`. A config with `messages.tts.persona: "narrator"` and a persona provider such as `microsoft` will still omit that speech plugin from the startup set, leaving the documented persona TTS path broken.
  - confidence: 0.88
- **[P3] Honor disabled blocks for direct TTS selections:** `src/plugins/gateway-startup-plugin-ids.ts:194-197`
  - body: `addConfiguredTtsProviderIds` adds `value.provider` before checking the matching provider block, so `messages.tts.provider: "microsoft"` plus `messages.tts.providers.microsoft.enabled: false` still starts Microsoft. The PR says disabled provider config is respected, and the Microsoft provider's `enabled` field is the documented usage gate.
  - confidence: 0.77
