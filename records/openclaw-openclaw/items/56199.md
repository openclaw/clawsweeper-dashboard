## Summary

Keep open. Current main has a real Volcengine Coding Plan provider path, but the reported Lite/Coding Plan plus DeepSeek V3.2 failure is not proven fixed: DeepSeek V3.2 is still cataloged under the general `/api/v3` provider while the `/api/coding/v3` plan provider does not list a DeepSeek row, and no live Volcengine key was available to verify the runtime 401 path.

## What This Changes

The issue reports that a Volcano Engine Lite/Coding Plan API key passes OpenClaw setup/status checks but fails at runtime with `401 The API key format is incorrect` when using a Volcengine DeepSeek V3.2 route.

## Best Possible Solution

Confirm the Volcengine Coding Plan contract with a real Lite/Plan key, then align the bundled plugin and docs so plan keys only expose/use models valid for `/api/coding/v3`, with DeepSeek V3.2 handled through the vendor-supported model id or `ark-code-latest` workflow and covered by targeted catalog/probe tests.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No live reproduction yet: the issue gives a plausible key/provider/model path and current source shows the likely boundary mismatch, but this environment has no Volcengine key for `openclaw models status --probe --probe-provider volcengine-plan` or a real chat completion.

## Solution Assessment

Is this the best way to solve the issue? Unclear. The existing `volcengine-plan` split is the right direction, but the safest fix needs provider-owner confirmation of the valid Coding Plan DeepSeek model path before changing catalog rows or auth gating.
