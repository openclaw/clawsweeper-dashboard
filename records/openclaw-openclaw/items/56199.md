## Summary

Keep open. Current main still has the Volcengine Coding Plan split, but the bundled plan catalog does not include the vendor-documented `deepseek-v3.2` Coding Plan model while the general `/api/v3` catalog still contains the online-inference DeepSeek ID that the vendor guide warns not to use for Coding Plan. ([developer.volcengine.com](https://developer.volcengine.com/articles/7615528054736945158))

## What This Changes

The issue asks OpenClaw to support a Volcano Engine Lite/Coding Plan API key with DeepSeek V3.2 instead of passing setup checks and then failing at runtime with `401 The API key format is incorrect`.

## Best Possible Solution

Align the bundled Volcengine Coding Plan catalog, provider docs, and concept docs with the vendor-supported `/api/coding/v3` model names, including `volcengine-plan/deepseek-v3.2`, while keeping online-inference IDs under the general provider. ([developer.volcengine.com](https://developer.volcengine.com/articles/7615528054736945158))

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source-reproducible but not live-reproduced: current source shows the catalog/base-URL mismatch against vendor guidance, but no Volcengine Lite/Coding Plan credential was present for a real `models status --probe` or chat completion.

## Solution Assessment

Is this the best way to solve the issue? Yes for a fix PR: update the Volcengine plugin-owned manifest/docs/tests rather than adding core key-format validation or startup migrations.
