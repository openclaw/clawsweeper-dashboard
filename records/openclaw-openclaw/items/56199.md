## Summary

Keep open: current main and v2026.5.18 still expose the general Volcengine DeepSeek row on `/api/v3` from the same auth choice that also registers the Coding Plan `/api/coding/v3` provider, so a Lite/Coding Plan key can still be guided toward a model/endpoint family that returns the reported 401.

## What This Changes

The issue reports a Volcengine Lite/Coding Plan UUID-style API key that OpenClaw accepts during setup but fails at runtime with `401 The API key format is incorrect` when a DeepSeek Volcengine route is used.

## Best Possible Solution

Align the Volcengine plugin setup, picker, catalog, and docs so Lite/Coding Plan keys route users to the Coding Plan `/api/coding/v3` model surface, while preserving the general `/api/v3` catalog for standard Volcengine keys.

## Reproduction Assessment

No high-confidence live reproduction exists because this review did not have a Volcengine Lite/Coding Plan credential. Source inspection gives a medium-confidence path: current main can present the general `/api/v3` DeepSeek row from the same auth choice whose Coding Plan key should target `/api/coding/v3`.

## Solution Assessment

Yes, the best fix is provider-owned catalog, picker, and docs alignment rather than core key-format validation or a new config option. The repair should verify the vendor-supported Coding Plan model IDs before adding any new model rows.
