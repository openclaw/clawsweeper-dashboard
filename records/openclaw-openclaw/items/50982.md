## Label Justifications

- `P2`: The reported gap affected a specific Telegram cron workflow with a limited blast radius.

## Summary

Current main and the shipped v2026.5.19 release already support cron `--thread-id` for Telegram topic delivery, with CLI, gateway schema, delivery-planning, and regression-test coverage in place.

## What This Changes

The issue requests `--thread-id` support for `openclaw cron add/create` and `openclaw cron edit` so Telegram cron announcements can target forum topics.

## Best Possible Solution

Keep the shipped cron `--thread-id` support; any remaining list-display or failure-alert-loop concerns from the comments should be handled as narrower follow-up reports.

## Reproduction Assessment

No current-main failure reproduces from source inspection: the cron add/create and edit commands now expose `--thread-id`, and tests cover the generated delivery payloads. The original gap is still well described by the issue discussion and carried-forward PR history.

## Solution Assessment

Yes. The current implementation adds the narrow CLI flag, keeps the value in the existing `delivery.threadId` contract, and relies on the existing threaded delivery path instead of adding a Telegram-only cron storage model.
