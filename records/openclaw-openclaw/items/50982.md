## Summary

Close as implemented. Current main and the shipped v2026.4.29 tag now expose `--thread-id` on both cron add/create and cron edit, pass the parsed value through `delivery.threadId`, and have focused CLI/protocol tests plus user-facing cron docs for Telegram forum-topic delivery.

## What This Changes

Add `--thread-id` support to `openclaw cron add/create` and `openclaw cron edit` so Telegram cron announcements can target forum topics.

## Best Possible Solution

Keep the shipped cron CLI implementation and close this tracker; any remaining malformed Telegram target validation or failure-alert routing cascade should be filed as a narrower follow-up with redacted job/queue examples.

## Reproduction Assessment

Yes. The original gap is reproducible by comparing `message send --thread-id` with the former cron add/edit surfaces; current main now has direct source and regression-test proof that cron add/edit accept and transmit `delivery.threadId`.

## Solution Assessment

Yes. Passing a validated cron CLI option into the existing `delivery.threadId` contract is the narrow maintainable fix because the Gateway schema, delivery plan, and Telegram topic dispatch path already support threaded delivery.
