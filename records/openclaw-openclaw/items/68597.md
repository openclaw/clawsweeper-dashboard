## Summary

Keep open: the underlying memory symlink boundary bug still appears real for workspace and QMD reads on current main, but this PR is not merge-ready because it patches the removed `src/memory-host-sdk` runtime and still authorizes workspace memory reads against too broad a root.

## What This Changes

The PR adds realpath containment checks and symlink regression tests for `memory_get` workspace, extraPath, and QMD reads.

## Best Possible Solution

Port the containment guard to `packages/memory-host-sdk`, authorize default memory reads against canonical memory roots instead of the whole workspace, apply canonical collection-root containment for QMD reads, preserve the merged extraPath hardening, and add focused regression tests plus real after-fix proof.

## Reproduction Assessment

Yes. Current main has a source-reproducible path: `memory_get` can accept lexically valid `memory/...` or QMD paths and then stat/read the resolved file without canonical parent containment against the intended memory or collection root.

## Solution Assessment

No, not as currently shaped. Realpath or parent-symlink containment is the right direction, but this branch must be moved to the package-owned runtime and use memory/QMD roots rather than the obsolete path and broad workspace root.
