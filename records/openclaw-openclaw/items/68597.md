## Summary

Keep open: the underlying symlink-boundary bug is valid and source-reproducible, but this branch is merge-conflicting, targets the older `src/memory-host-sdk` runtime instead of the current package-owned runtime, uses the workspace root too broadly for workspace memory reads, and still lacks external real behavior proof.

## What This Changes

The PR adds realpath containment checks and symlink regression tests for `memory_get` workspace, extraPath, and QMD reads.

## Best Possible Solution

Port a canonical containment helper into `packages/memory-host-sdk`, export it through the current plugin SDK storage surface, constrain workspace reads to the memory root rather than the whole workspace, apply the same guard to QMD collection reads, and keep focused regression tests for workspace, extraPath, and QMD symlinked-directory escapes.

## Reproduction Assessment

Yes. Current main has a source-reproducible path: `memory_get` can accept a lexically valid `memory/...` or QMD path and then stat/read the resolved path without canonical containment against the intended memory or collection root.

## Solution Assessment

No, not as currently shaped. Realpath containment is the right direction, but this branch must be moved to the package-owned runtime and use the correct memory/QMD roots instead of the obsolete path and broad workspace root.
