## Label Justifications

- `impact:message-loss`: The image attachment is rejected before the configured CLI backend can receive and process it.

## Summary

Current main still rejects stale or missing `google-gemini-cli` image capability metadata before CLI prestaging, while the bundled backend is configured for workspace image references. Two focused fixes were proposed in https://github.com/openclaw/openclaw/pull/91790 and https://github.com/openclaw/openclaw/pull/91849, but both closed unmerged, so no canonical implementation currently owns the remaining work.

## What This Changes

Fix the Gateway capability gate so image-capable Gemini CLI models accept attachments before CLI image prestaging.

## Best Possible Solution

Make the Gateway’s early attachment decision reflect the bundled Google CLI backend’s image capability, including supported Gemini aliases and stale or missing catalog rows, while retaining explicit negative coverage for genuinely text-only models.

## Reproduction Assessment

Yes, at source level with high confidence: the Gateway runs `resolveGatewayModelSupportsImages` before attachment parsing, and its current fallthrough rejects stale or missing Gemini CLI capability metadata despite the backend’s configured image path. The issue also provides a real Gateway reproduction, but this read-only review did not rerun it.

## Solution Assessment

The focused shim from the closed PR is a plausible repair, but the best implementation should first confirm whether provider-owned capability metadata can supply the same fact without adding another drifting Gateway source of truth. Either shape must cover stale and missing rows, supported aliases, and a text-only negative case.
