## Summary

Keep open: current main and the latest release still lack a `google-gemini-cli` image-capability fallback, so image attachments can be rejected before the Gemini CLI prestage path runs. Several linked PRs now attempt this same repair, but none is merged, so this issue should remain the canonical tracker until one lands.

## What This Changes

The issue reports that image prompts routed through `google-gemini-cli` fail at the Gateway attachment gate before the CLI backend can use its `@<workspace-image-path>` image prestage flow.

## Best Possible Solution

Land one narrow linked PR or equivalent main change that lets image-capable Gemini CLI models pass the Gateway attachment gate while preserving text-only rejection.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main can return false for `google-gemini-cli` before `parseMessageWithAttachments` throws the reported `UnsupportedAttachmentError`. I did not rerun the live OAuth-backed Gateway flow in this read-only review.

## Solution Assessment

The reporter’s local shim proves the failure shape, but the best maintainable fix is a narrow provider/capability update with stale/missing catalog regression tests and preserved text-only rejection. Existing linked PRs are plausible implementation candidates, but this issue should stay open until one merges.
