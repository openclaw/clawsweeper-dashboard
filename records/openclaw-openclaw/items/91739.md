## Summary

Keep open: current main and the latest release still have a pre-CLI image capability gate that can reject `google-gemini-cli` when the catalog row is stale or missing, even though the bundled Gemini CLI backend and CLI runner image prestage path are configured for `@<workspace-image-path>` image prompts. No merged fix, duplicate, or canonical replacement was found in the local history/search pass.

## What This Changes

The issue reports that Gateway rejects image attachments for `google/gemini-3.5-flash` routed through the `google-gemini-cli` backend before CLI image prestage can run.

## Best Possible Solution

Keep the issue open and fix the provider capability contract so Google Gemini CLI image-capable model families pass the Gateway attachment gate while text-only models still reject early.

## Reproduction Assessment

Yes. The source path shows `resolveGatewayModelSupportsImages` can return false for `google-gemini-cli` before `parseMessageWithAttachments` accepts the image, and the issue includes real Gateway commands that hit the matching `UnsupportedAttachmentError`; I did not rerun the live Gemini CLI flow in this read-only review.

## Solution Assessment

No, the reporter's local hard-coded shim is only a proof of the failure shape. The best fix should centralize Google CLI image capability through provider/catalog capability metadata or a narrow shared CLI capability helper, with regression tests for stale/missing catalog rows and text-only rejection.
