## Summary

Keep open for maintainer review. Current main and the latest release still hard-filter all Gemini freshness shortcuts, and this PR is a focused, proof-backed fix for the `day`/`pd` failure, but it intentionally changes those callers from hard 24-hour filtering to best-effort recency guidance.

## What This Changes

The PR changes Gemini `web_search` so `day`/`pd` use a query recency hint, preserves hard `timeRangeFilter` use for wider freshness and explicit dates, updates docs/tests/cache partitioning, and carries two lint cleanups.

## Best Possible Solution

Land the day-only Gemini fallback after maintainer acceptance of the compatibility tradeoff and normal validation, while keeping wider freshness and explicit date ranges hard-filtered.

## Reproduction Assessment

Yes. Current main and `v2026.6.9` source build a hard 24-hour `timeRangeFilter` for `day`/`pd`, and the PR body provides redacted live Gemini endpoint output showing exact 24-hour hard windows fail while wider ranges succeed; I did not rerun the credentialed live API in this read-only review.

## Solution Assessment

Yes, with a maintainer compatibility choice. The patch keeps the fix in the Google plugin request-construction boundary, softens only the endpoint-rejected 24-hour shortcut, preserves hard week/month/year/date ranges, and partitions cache keys.
