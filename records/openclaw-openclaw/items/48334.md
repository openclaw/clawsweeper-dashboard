## Summary

Keep open: current main still has a source-reproducible cron context-window fallback, but this PR is not merge-ready because the status-side repair violates the read-only status contract and the external PR still lacks inspectable real behavior proof.

## What This Changes

The PR adds an async context-window resolver for cron session persistence, changes status summaries to repair stale 200k session context windows, adds per-agent contextTokens typing, and adds regression tests.

## Best Possible Solution

Land a rebased fix that blocks registry readiness only for cron persistence, keeps status summaries on a read-only cached path, validates positive stored values and explicit caps, and includes focused regression coverage plus redacted real behavior proof.

## Reproduction Assessment

Yes, by source inspection rather than an executed live repro: current main can persist DEFAULT_CONTEXT_TOKENS when finalizeCronRun calls lookupContextTokens with allowAsyncLoad false before the model registry cache is warm.

## Solution Assessment

No. The cron-side blocking resolver direction is reasonable, but the status-side repair should stay read-only/cached and preserve positive-value plus explicit-cap guards before this PR is mergeable.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Keep status summaries on the read-only resolver:** `src/commands/status.summary.ts:181-187`
  - body: `getStatusSummary` now awaits `resolveContextTokensForModelAsync`, whose loader can write/prepare models.json and import provider discovery. Plain `status` currently passes `allowAsyncLoad: false` and has coverage for avoiding async context warmup, so this repair needs a cached read-only path instead.
  - confidence: 0.9
- **[P2] Ignore non-positive stored context-token values:** `src/commands/status.summary.ts:229-230`
  - body: The new fallback treats any numeric `entry.contextTokens` as authoritative. That drops the existing `> 0` guard, so `0`, negative, or `NaN` session values can produce invalid remaining/context output instead of falling back to the resolved model window.
  - confidence: 0.9
- **[P2] Preserve explicit per-agent caps in recent rows:** `src/commands/status.summary.ts:292`
  - body: `sessions.recent` rebuilds rows from raw store paths without passing the store's agent id. For legacy keys that do not parse to an agent id, a legitimate per-agent `200000` cap can be mistaken for a stale fallback and upgraded to the registry window.
  - confidence: 0.84
