## Summary

Keep open. Current main still has the reported cron/status context-window split, but this PR is not merge-ready because the status-side repair triggers async model registry work from plain status, accepts invalid stored context-token values, and can repair legitimate capped agent rows in the combined recent list.

## What This Changes

The PR adds an async context-window resolver, uses it for isolated cron session persistence and status summary display repair, adds per-agent contextTokens type plumbing, and adds regression tests.

## Best Possible Solution

Port the narrow cron persistence fix onto current main, keep plain status on a non-writing read-only resolver or cached snapshot, repair only known stale DEFAULT_CONTEXT_TOKENS display values when no explicit default or per-agent cap applies, validate stored context-token values before display, and add focused regression coverage plus a changelog entry.

## Reproduction Assessment

Yes. A high-confidence code-path reproduction exists: with a cold context cache, finalizeCronRun can fall through to DEFAULT_CONTEXT_TOKENS, and getStatusSummary then treats that stored value as authoritative. No live profile gateway run was performed because this review is read-only.

## Solution Assessment

No, not as-is. The cron-side blocking resolver direction is right, but the status-side implementation should avoid model discovery or models.json writes, preserve explicit caps, and keep the previous positive-number guard for stored values.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Keep status context repair read-only:** `src/commands/status.summary.ts:181-187`
  - body: `getStatusSummary` now awaits `resolveContextTokensForModelAsync`, whose loader can call `ensureOpenClawModelsJson` and model discovery. Plain `openclaw status` is documented and tested as a fast read-only path, so status repair should use a non-writing resolver or snapshot instead of triggering the async loader.
  - confidence: 0.87
- **[P2] Ignore nonpositive stored context tokens:** `src/commands/status.summary.ts:214-215`
  - body: `storedContextTokens` accepts any numeric `entry.contextTokens` and then wins over the resolved model window. The previous resolver only accepted positive overrides, so `0`, negative numbers, or `NaN` can now produce invalid context, remaining-token, and percent output instead of falling back.
  - confidence: 0.9
- **[P2] Preserve per-agent caps in recent rows:** `src/commands/status.summary.ts:292`
  - body: `sessions.recent` rebuilds rows from raw store paths without passing the store's agent id, so legacy keys that do not parse to an agent id make `hasExplicitContextTokensCap` return false. A real per-agent 200000 cap can then be upgraded to the registry window in the combined recent list.
  - confidence: 0.82
