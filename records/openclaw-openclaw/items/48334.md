## Summary

Keep open: current main still has the source-reproducible cron context-window persistence bug, but this PR is not merge-ready because the status-side repair conflicts with the current read-only status contract and after-fix real behavior proof is still missing.

## What This Changes

The PR adds an async model-context resolver, uses it when cron isolated-agent sessions persist contextTokens, adds status-side display repair for stale 200k values, and updates related typing/tests.

## Best Possible Solution

Land a rebased cron persistence fix that waits for provider-aware registry readiness when storing cron session context, keeps status cached/read-only with positive-value and explicit-cap guards, and includes focused tests plus redacted profile-gateway proof.

## Reproduction Assessment

Yes. Source inspection on current main shows finalizeCronRun can persist DEFAULT_CONTEXT_TOKENS when lookupContextTokens runs with allowAsyncLoad: false before the registry cache is warm and no positive previous value exists.

## Solution Assessment

No. The cron-side provider-aware blocking lookup is the right direction, but the branch's status-side async repair breaks the current read-only status boundary and needs stronger stored-value and explicit-cap guards.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Keep status summaries on cached read-only resolution:** `src/commands/status.summary.ts:181-187`
  - body: getStatusSummary now awaits resolveContextTokensForModelAsync, whose loader can call models.json setup and provider discovery. Current status/status --json intentionally uses allowAsyncLoad: false, so this reintroduces startup side effects into a read-only status command.
  - confidence: 0.9
- **[P2] Validate stored context-token values:** `src/commands/status.summary.ts:214-230`
  - body: The new fallback treats any numeric entry.contextTokens as authoritative. That drops the existing positive-value guard, so 0, negative, or NaN session-store values can leak into status output instead of falling back to the resolved model window.
  - confidence: 0.9
- **[P2] Preserve explicit caps when the agent id is unknown:** `src/commands/status.summary.ts:49-50`
  - body: hasExplicitContextTokensCap returns false when no agent id is parsed, but sessions.recent is rebuilt from raw store keys where legacy keys may not parse. A legitimate stored 200000 cap can then be silently upgraded to the registry window.
  - confidence: 0.84
