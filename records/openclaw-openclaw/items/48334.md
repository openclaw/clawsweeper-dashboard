## Summary

Keep open. Current main still has the reported cron/status context-window split, and this PR is a focused implementation candidate, but the branch is not merge-ready as-is because the status repair path introduces read-only side effects and needs tighter guards for explicit caps and invalid stored values.

## What This Changes

The PR adds an async context-window resolver, uses it for isolated cron session persistence and status summaries, adds guarded display repair for stale 200k context windows, and adds related agent config/test coverage.

## Best Possible Solution

Port the narrow behavior onto current main: use a blocking provider/model context-window resolver for isolated cron persistence, keep plain status read-only, repair only known stale DEFAULT_CONTEXT_TOKENS display values when no explicit cap applies, validate stored values before using them, and cover cron persistence plus status repair edge cases.

## Reproduction Assessment

Yes. A high-confidence code-path reproduction exists on current main: cron finalization can persist DEFAULT_CONTEXT_TOKENS through lookupContextTokens(...allowAsyncLoad:false), and status then treats that stored positive value as authoritative; no live profile gateway run was performed because this review is read-only.

## Solution Assessment

No, not as-is. The intended cron-side fix is the right direction, but the status-side implementation should avoid writes/provider discovery and must preserve explicit caps and finite positive stored-value guards before this is the best maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Keep status context repair read-only:** `src/commands/status.summary.ts:181-187`
  - body: `getStatusSummary` now awaits `resolveContextTokensForModelAsync`, whose cache loader calls `ensureOpenClawModelsJson` and model discovery. Plain `openclaw status` is documented and currently implemented as a fast read-only path, so this can unexpectedly create/update `models.json` or load discovery during status; use a non-writing resolver/snapshot for display repair instead.
  - confidence: 0.87
- **[P2] Ignore nonpositive stored context tokens:** `src/commands/status.summary.ts:214-215`
  - body: This accepts any numeric `entry.contextTokens` and then returns it before the resolved model window. The previous resolver ignored nonpositive values, so a stale or manually edited session with `0`, a negative number, or `NaN` can now produce invalid context/remaining/% output instead of falling back.
  - confidence: 0.9
- **[P2] Preserve per-agent caps in recent rows:** `src/commands/status.summary.ts:292`
  - body: `sessions.recent` rebuilds per-agent stores without passing the store's agent id, so legacy keys such as `main` leave `agentId` undefined and `hasExplicitContextTokensCap` returns false. A legitimate per-agent 200000 cap can then be upgraded to the registry window in the combined recent list.
  - confidence: 0.82
