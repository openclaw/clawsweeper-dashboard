## Summary

Keep open: current main still has the cron cold-cache context-token fallback, but this PR is not mergeable as written because the status-side repair can make `openclaw status` perform model discovery/models.json writes, accepts invalid stored token caps, and can override real per-agent caps in recent rows.

## What This Changes

The PR adds an async model-context resolver, applies it to cron session persistence and status summary display repair, adds per-agent context-token type plumbing, and updates regression tests.

## Best Possible Solution

Port the cron persistence fix onto current main, add a read-only status display repair for known stale `DEFAULT_CONTEXT_TOKENS` values, preserve explicit caps and positive-number guards, and add the required changelog entry.

## Reproduction Assessment

Yes, by source inspection on current main. `finalizeCronRun` still uses `lookupContextTokens(modelUsed, { allowAsyncLoad: false })` before falling back to the stored session value or `DEFAULT_CONTEXT_TOKENS`, so a cold registry can still persist the hard fallback.

## Solution Assessment

No. The cron-side blocking resolver direction is right, but the status-side implementation should not trigger model discovery or models.json writes and must keep explicit caps plus positive stored-value validation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Keep status context repair read-only:** `src/commands/status.summary.ts:181-187`
  - body: `getStatusSummary` now awaits `resolveContextTokensForModelAsync`, whose cache loader can call `ensureOpenClawModelsJson` and provider discovery. Plain `openclaw status` is documented and tested as a fast read-only path, so the display repair needs a non-writing resolver or cached snapshot instead of the async loader.
  - confidence: 0.88
- **[P2] Validate stored context token values before using them:** `src/commands/status.summary.ts:214-215`
  - body: The new fallback treats any numeric `entry.contextTokens` as authoritative. That drops the previous positive-number guard, so `0`, negative, or `NaN` session values can produce invalid remaining-token and percent output instead of falling back to the resolved model window.
  - confidence: 0.9
- **[P2] Preserve per-agent caps in recent rows:** `src/commands/status.summary.ts:292`
  - body: `sessions.recent` rebuilds rows from raw store paths without passing the store's agent id. For legacy keys that do not parse to an agent id, a real per-agent `200000` cap can be mistaken for the stale fallback and upgraded to the registry window.
  - confidence: 0.82
- **[P3] Add the required changelog entry:** `src/cron/isolated-agent/run.ts:739-746`
  - body: This is a user-facing `fix` that changes persisted cron session context windows and status output, but the PR does not touch `CHANGELOG.md`. OpenClaw policy requires a changelog entry before merge.
  - confidence: 0.86
