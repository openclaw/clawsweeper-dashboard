## Summary

Keep open: the reported bug remains source-reproducible on current main, but this branch still introduces status read-only/cap handling regressions and lacks inspectable real behavior proof.

## What This Changes

The branch adds a blocking context-token resolver for cron session persistence, status display repair for stale 200k context windows, per-agent context-token typing, and regression tests.

## Best Possible Solution

Land a current-main fix that blocks registry readiness only for cron persistence, keeps status display read-only, validates positive stored context-token values, preserves explicit caps, and includes focused regression coverage plus redacted real behavior proof.

## Reproduction Assessment

Yes. Source inspection shows current main can persist DEFAULT_CONTEXT_TOKENS when finalizeCronRun uses lookupContextTokens with allowAsyncLoad false before the model registry cache is warm.

## Solution Assessment

No. The cron-side blocking resolver direction is reasonable, but the status-side repair should use a read-only/cached path and retain positive-value and explicit-cap guards before this is mergeable.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Keep status repair read-only:** `src/commands/status.summary.ts:181-187`
  - body: getStatusSummary now awaits resolveContextTokensForModelAsync, whose loader can call models.json setup and provider discovery. Plain status/status --json currently passes allowAsyncLoad: false and has test coverage for staying read-only, so this display repair needs a non-writing cached path instead.
  - confidence: 0.88
- **[P2] Validate stored context token values:** `src/commands/status.summary.ts:214-230`
  - body: The new fallback treats any numeric entry.contextTokens as authoritative. That drops the current >0 guard, so 0, negative, or NaN stored values can produce invalid remaining/context output instead of falling back to the resolved model window.
  - confidence: 0.9
- **[P2] Preserve per-agent caps in recent rows:** `src/commands/status.summary.ts:292`
  - body: sessions.recent rebuilds rows from raw store paths without passing the store's agent id. For legacy keys that do not parse to an agent id, a legitimate per-agent 200000 cap can be mistaken for the stale fallback and upgraded to the registry window.
  - confidence: 0.82
