## Summary

Keep this PR open: the reported cron context-window fallback is still source-reproducible on current main, but the proposed branch is conflicting, lacks inspectable real behavior proof, and still has status-path correctness regressions that should block merge.

## What This Changes

The PR adds async model-registry context resolution for cron isolated-agent session persistence, status-side stale context display repair, per-agent context-token typing, and regression tests.

## Best Possible Solution

Land a rebased cron-side fix that awaits registry readiness only when persisting cron session context, keeps status on cached/read-only resolution, preserves positive-value and explicit-cap guards, and includes focused regression tests plus redacted real behavior proof.

## Reproduction Assessment

Yes, by source inspection: current main can persist DEFAULT_CONTEXT_TOKENS when finalizeCronRun calls lookupContextTokens with allowAsyncLoad false before the model registry cache is warm and no previous positive session value exists.

## Solution Assessment

No. The cron-side blocking resolver is the right direction, but the status-side async display repair violates the read-only status contract and drops important stored-value/cap guards.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Keep status summaries on cached read-only resolution:** `src/commands/status.summary.ts:181-187`
  - body: getStatusSummary now awaits resolveContextTokensForModelAsync, which can warm the model registry and import provider discovery. Status/status --json currently stay on a cached read-only path, so this reintroduces startup side effects in a status command.
  - confidence: 0.9
- **[P2] Validate stored context-token values:** `src/commands/status.summary.ts:230`
  - body: The new fallback treats any numeric entry.contextTokens as authoritative. That drops the existing positive-value guard, so 0, negative, or NaN session-store values can leak into status output instead of falling back to the resolved model window.
  - confidence: 0.9
- **[P2] Preserve explicit caps when the agent id is unknown:** `src/commands/status.summary.ts:50`
  - body: hasExplicitContextTokensCap returns false without an agent id, but sessions.recent is also rebuilt from raw store paths where legacy keys may not parse. A legitimate stored 200000 cap can be silently upgraded to the registry window.
  - confidence: 0.84
