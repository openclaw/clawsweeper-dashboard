## Summary

Keep open: current main still has the cron cold-cache context fallback, but this PR is not mergeable as-is because the status-side changes cross the read-only status boundary, drop context-token guards, and still lack inspectable real behavior proof.

## What This Changes

The PR adds an async context-window resolver, uses it from cron session persistence and status summaries, adds per-agent context-token typing, and adds regression tests.

## Best Possible Solution

Land a rebased cron-side fix that waits for registry readiness when persisting cron session context, keeps status cached/read-only with positive-value and explicit-cap guards, and includes focused tests plus redacted real proof.

## Reproduction Assessment

Yes. Source inspection shows current main can persist DEFAULT_CONTEXT_TOKENS when finalizeCronRun uses lookupContextTokens with allowAsyncLoad false before the model registry cache is warm and no previous positive session value exists.

## Solution Assessment

No. The cron-side blocking resolver is the right direction, but the status-side async repair violates the read-only status contract and drops stored-value and explicit-cap guards.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Keep status summaries on cached read-only resolution:** `src/commands/status.summary.ts:178-186`
  - body: getStatusSummary now awaits resolveContextTokensForModelAsync, whose loader can call models.json setup and provider discovery. Current status/status --json intentionally uses allowAsyncLoad: false, so this reintroduces startup side effects into a read-only status command.
  - confidence: 0.9
- **[P2] Validate stored context-token values:** `src/commands/status.summary.ts:211-223`
  - body: The new fallback treats any numeric entry.contextTokens as authoritative. That drops the existing positive-value guard, so 0, negative, or NaN session-store values can leak into status output instead of falling back to the resolved model window.
  - confidence: 0.9
- **[P2] Preserve explicit caps when the agent id is unknown:** `src/commands/status.summary.ts:49-52`
  - body: hasExplicitContextTokensCap returns false when no agent id is parsed, but sessions.recent is rebuilt from raw store keys where legacy keys may not parse. A legitimate stored 200000 cap can then be silently upgraded to the registry window.
  - confidence: 0.84
