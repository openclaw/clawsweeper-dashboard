## Summary

Keep open. The underlying cron context-window bug is source-reproducible and the cron-side direction is useful, but this PR is not merge-ready because it conflicts with current main, lacks inspectable real behavior proof, and the status-side async repair conflicts with the current read-only status boundary while risking incorrect context accounting for existing session stores.

## What This Changes

The branch adds an async context-window resolver, switches cron isolated-agent session persistence to it, adds status display repair for stale 200k entries, and updates config typing plus regression tests.

## Best Possible Solution

Land a rebased cron persistence fix that resolves context tokens through provider/config-aware runtime before storing session state, keeps status read-only, and includes guards/tests for explicit caps plus redacted profile-gateway proof.

## Reproduction Assessment

Yes. Source inspection on current main shows the cron finalization path can still persist `DEFAULT_CONTEXT_TOKENS` when `lookupContextTokens(..., { allowAsyncLoad: false })` misses a cold cache and there is no prior positive session value.

## Solution Assessment

No. The cron-side provider-aware resolution is the right direction, but the status-side async discovery is not the best path because current main has a dedicated read-only status resolver and the repair needs stored-value and cap guards.

- **[P2] Keep status summaries read-only:** `src/commands/status.summary.ts:181-187`
  - body: `getStatusSummary` now awaits `resolveContextTokensForModelAsync`, whose loader can run models.json setup and provider discovery. Current main keeps status/status --json on cached read-only resolution, so this reintroduces startup side effects into a read-only status command.
  - confidence: 0.9
- **[P2] Validate stored context-token values:** `src/commands/status.summary.ts:229-230`
  - body: The new fallback treats any numeric `entry.contextTokens` as authoritative, dropping the prior positive-value guard from `resolveContextTokensForModel`. A stored `0`, negative value, or `NaN` can now produce invalid remaining/% status output instead of falling back to the resolved model window.
  - confidence: 0.87
- **[P2] Preserve caps when the agent id is unknown:** `src/commands/status.summary.ts:49-50`
  - body: When the session key cannot be parsed to an agent id, the repair path assumes there is no explicit cap. Legacy keys can therefore have a legitimate stored 200k cap displayed as a larger registry value, which misreports context accounting for capped agents.
  - confidence: 0.82
