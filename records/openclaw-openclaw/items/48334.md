## Summary

Keep open. The underlying cron/profile context-window bug is still present on current main, but this PR should not merge as-is because the status-side repair can violate the documented read-only status path and mishandle stored or capped context-token values.

## What This Changes

The PR adds an async context-token resolver, uses it in isolated cron session persistence and status summary display repair, adds per-agent contextTokens type plumbing, and adds regression coverage.

## Best Possible Solution

Port the cron persistence fix onto current main, keep status summary on a non-writing read-only resolver or cache snapshot, repair only known stale DEFAULT_CONTEXT_TOKENS display values when no explicit default or per-agent cap applies, and preserve positive-number validation for stored values.

## Reproduction Assessment

Yes. Source inspection on current main shows finalizeCronRun still uses lookupContextTokens(modelUsed, { allowAsyncLoad: false }) before falling back to DEFAULT_CONTEXT_TOKENS, so a cold cache can still persist the hard fallback.

## Solution Assessment

No. The cron-side blocking resolver direction is right, but the status-side implementation should avoid model discovery or models.json writes, preserve explicit caps, and keep the previous positive-number guard for stored values.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Keep status context repair read-only:** `src/commands/status.summary.ts:181-187`
  - body: getStatusSummary now awaits resolveContextTokensForModelAsync, whose loader can call ensureOpenClawModelsJson and model discovery. Plain openclaw status is documented and tested as a fast read-only path, so the display repair should use a non-writing resolver or cached snapshot instead of triggering the async loader.
  - confidence: 0.87
- **[P2] Ignore nonpositive stored context tokens:** `src/commands/status.summary.ts:214-215`
  - body: storedContextTokens accepts any numeric entry.contextTokens and then wins over the resolved model window. The previous resolver only accepted positive overrides, so 0, negative numbers, or NaN can now produce invalid context, remaining-token, and percent output instead of falling back.
  - confidence: 0.9
- **[P2] Preserve per-agent caps in recent rows:** `src/commands/status.summary.ts:292`
  - body: sessions.recent rebuilds rows from raw store paths without passing the store's agent id, so legacy keys that do not parse to an agent id make hasExplicitContextTokensCap return false. A real per-agent 200000 cap can then be upgraded to the registry window in the combined recent list.
  - confidence: 0.82
