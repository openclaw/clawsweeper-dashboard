## Summary

Keep open: this PR targets a valid current-main cron/session context-window bug, but the proposed status-summary repair still violates the read-only status contract, drops stored-value validation, can override real per-agent caps, and lacks required real behavior proof.

## What This Changes

The branch adds an async model-context resolver for cron persistence, changes status summaries to repair stale 200k context values, adds per-agent context-token typing, and updates regression tests.

## Best Possible Solution

Land a current-main fix that blocks only the cron persistence lookup on registry readiness, keeps status display read-only, validates positive stored context-token values, preserves explicit caps, and includes focused regression coverage plus redacted real behavior proof.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: current main can persist DEFAULT_CONTEXT_TOKENS when finalizeCronRun hits lookupContextTokens with allowAsyncLoad false before the model registry cache is warm. I did not run a live profile gateway, so this is source-reproducible rather than live-reproduced.

## Solution Assessment

No. The cron-side blocking resolver direction is reasonable, but the status-side repair must remain read-only and retain positive-number and explicit-cap guards before this is the narrow maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Keep status repair read-only:** `src/commands/status.summary.ts:181-187`
  - body: getStatusSummary now awaits resolveContextTokensForModelAsync, whose cache loader can call models.json setup and provider discovery. Plain status/status --json is intentionally read-only, so the display repair needs a non-writing resolver or cached snapshot instead.
  - confidence: 0.88
- **[P2] Validate stored context token values:** `src/commands/status.summary.ts:214-215`
  - body: The new fallback treats any numeric entry.contextTokens as authoritative. That drops the current positive-number guard, so 0, negative, or NaN stored values can produce invalid context, remaining-token, or percent output instead of falling back.
  - confidence: 0.9
- **[P2] Preserve per-agent caps in recent rows:** `src/commands/status.summary.ts:292`
  - body: sessions.recent rebuilds rows from raw store paths without passing the store's agent id. For legacy keys that do not parse to an agent id, a real per-agent 200000 cap can be mistaken for the stale fallback and upgraded to the registry window.
  - confidence: 0.82
