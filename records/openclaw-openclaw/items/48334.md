## Summary

Keep open. Current main still has a source-reproducible cron session context-window fallback, but this PR is not mergeable as-is because it lacks inspectable real behavior proof and introduces status-summary correctness regressions.

## What This Changes

The PR adds an async model-context resolver, uses it when cron isolated-agent runs persist session context, changes status summary display repair, adds per-agent contextTokens typing, and adds regression tests.

## Best Possible Solution

Land a rebased cron persistence fix that waits for registry-ready context resolution when writing cron session state while keeping status summaries cached/read-only with positive stored-value and explicit-cap guards.

## Reproduction Assessment

Yes, by source inspection. Current main can persist DEFAULT_CONTEXT_TOKENS when cron finalize uses lookupContextTokens with allowAsyncLoad: false before the model registry cache is warm and no previous positive session value exists.

## Solution Assessment

No. The cron-side blocking resolver is the right direction, but the status-side async repair changes the read-only status contract and weakens stored-value and explicit-cap handling.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Keep status summaries on cached read-only resolution:** `src/commands/status.summary.ts:178-186`
  - body: getStatusSummary now awaits resolveContextTokensForModelAsync, whose loader can call models.json setup and provider discovery. Current status/status --json intentionally resolves context with allowAsyncLoad: false, so this reintroduces startup side effects into a read-only status command.
  - confidence: 0.9
- **[P2] Validate stored context-token values:** `src/commands/status.summary.ts:211-223`
  - body: The new fallback treats any numeric entry.contextTokens as authoritative. That drops the existing positive-value guard, so 0, negative, or NaN session-store values can leak into status output instead of falling back to the resolved model window.
  - confidence: 0.9
- **[P2] Preserve explicit caps when the agent id is unknown:** `src/commands/status.summary.ts:49-52`
  - body: hasExplicitContextTokensCap returns false when no agent id is parsed, but sessions.recent is rebuilt from raw store keys where legacy keys may not parse. A legitimate stored 200000 cap can then be silently upgraded to the registry window.
  - confidence: 0.84
