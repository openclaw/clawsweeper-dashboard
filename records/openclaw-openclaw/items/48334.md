## Summary

Keep open: current main still has the cold-cache cron context-window fallback, but this branch should not merge as-is because the status repair path breaks the read-only status contract and mishandles stored or capped context token values. External real behavior proof is also missing.

## What This Changes

The PR adds an async context-token resolver for cron session persistence, changes status summaries to repair stale 200k session values, adds per-agent context-token type plumbing, and updates regression tests.

## Best Possible Solution

Port the cron persistence fix onto current main, add a read-only display repair for known stale `DEFAULT_CONTEXT_TOKENS` values, and preserve explicit caps plus positive-number validation.

## Reproduction Assessment

Yes. Source inspection on current main shows `finalizeCronRun` still performs a non-async `lookupContextTokens(..., { allowAsyncLoad: false })` before falling back to the previous session value or `DEFAULT_CONTEXT_TOKENS`, so a cold registry can persist 200000.

## Solution Assessment

No. The cron-side blocking resolver direction is right, but the status-side repair must remain read-only and must retain positive-value and explicit-cap guards.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Keep status repair read-only:** `src/commands/status.summary.ts:181-187`
  - body: `getStatusSummary` now awaits `resolveContextTokensForModelAsync`, whose cache loader can call `ensureOpenClawModelsJson` and provider discovery. Plain `openclaw status` is intentionally read-only, so this display repair needs a non-writing resolver or cached snapshot instead.
  - confidence: 0.88
- **[P2] Validate stored context token values:** `src/commands/status.summary.ts:214-215`
  - body: The new fallback treats any numeric `entry.contextTokens` as authoritative. That drops the current positive-number guard, so `0`, negative, or `NaN` stored values can produce invalid context, remaining-token, or percent output instead of falling back.
  - confidence: 0.9
- **[P2] Preserve per-agent caps in recent rows:** `src/commands/status.summary.ts:292`
  - body: `sessions.recent` rebuilds rows from raw store paths without passing the store's agent id. For legacy keys that do not parse to an agent id, a real per-agent `200000` cap can be mistaken for the stale fallback and upgraded to the registry window.
  - confidence: 0.82
