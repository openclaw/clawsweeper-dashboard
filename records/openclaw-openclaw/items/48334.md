## Summary

Keep open: current main still has a source-reproducible cron context-window fallback, but this external PR is not mergeable as-is because it lacks inspectable real behavior proof and its status-side repair conflicts with the current read-only status boundary.

## What This Changes

The PR adds async context-token resolution for cron session persistence, status-side stale 200k display repair, per-agent contextTokens typing, and regression tests.

## Best Possible Solution

Land a rebased fix that blocks registry readiness only for cron persistence, keeps status on a cached/read-only path, preserves positive-value and explicit-cap guards, and includes focused regression coverage plus redacted real behavior proof.

## Reproduction Assessment

Yes, by source inspection: current main can persist DEFAULT_CONTEXT_TOKENS when finalizeCronRun calls lookupContextTokens with allowAsyncLoad false before the model registry cache is warm and no previous positive session value exists.

## Solution Assessment

No. The cron-side blocking resolver direction is reasonable, but status-side repair should stay read-only/cached and preserve stored-value and explicit-cap guards before this PR is mergeable.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Keep status summaries read-only:** `src/commands/status.summary.ts:181-187`
  - body: `getStatusSummary` now awaits `resolveContextTokensForModelAsync`, which can warm the model registry and import provider discovery. Current main keeps status/status --json on a read-only cached path, so this would reintroduce startup side effects in a status command.
  - confidence: 0.9
- **[P2] Validate stored context-token values:** `src/commands/status.summary.ts:230`
  - body: The new fallback treats any numeric `entry.contextTokens` as authoritative. That drops the existing positive-value guard, so 0, negative, or NaN session-store values can leak into status output instead of falling back to the resolved model window.
  - confidence: 0.9
- **[P2] Preserve caps when agent id is unknown:** `src/commands/status.summary.ts:50`
  - body: `hasExplicitContextTokensCap` returns false without an agent id, but `allSessions` rebuilds rows from raw store paths where legacy keys may not parse. A configured 200000 cap can be silently upgraded to the registry window.
  - confidence: 0.84
