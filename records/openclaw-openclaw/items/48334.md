## Summary

Keep open: the cron context-window persistence bug is still source-reproducible on current main, but this PR is not merge-ready because the status-side changes break the read-only status boundary, can report invalid stored token values, can override explicit caps, and still lacks inspectable real behavior proof.

## What This Changes

The PR adds an async model context resolver for cron session persistence, status-side stale-token display repair, per-agent contextTokens typing, and regression tests.

## Best Possible Solution

Rebase and land a cron-side persistence fix that waits for registry readiness without making status startup non-read-only, preserves positive stored values and explicit caps, and includes focused tests plus redacted profile-gateway proof.

## Reproduction Assessment

Yes. Source inspection shows current main can persist DEFAULT_CONTEXT_TOKENS when finalizeCronRun uses lookupContextTokens with allowAsyncLoad false before the model registry cache is warm and no previous positive session value exists.

## Solution Assessment

No. The cron-side blocking resolver is the right direction, but the PR's status-side async repair violates the read-only status contract and drops important stored-value and explicit-cap guards.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Keep status summaries on cached read-only resolution:** `src/commands/status.summary.ts:178-186`
  - body: getStatusSummary now awaits resolveContextTokensForModelAsync, whose loader can call models.json setup and provider discovery. Current status/status --json intentionally uses allowAsyncLoad: false, so this reintroduces startup side effects into a read-only status command.
  - confidence: 0.9
- **[P2] Validate stored context-token values:** `src/commands/status.summary.ts:230`
  - body: The new fallback treats any numeric entry.contextTokens as authoritative. That drops the existing positive-value guard, so 0, negative, or NaN session-store values can leak into status output instead of falling back to the resolved model window.
  - confidence: 0.9
- **[P2] Preserve explicit caps when the agent id is unknown:** `src/commands/status.summary.ts:50-52`
  - body: hasExplicitContextTokensCap returns false when no agent id is parsed, but sessions.recent is rebuilt from raw store keys where legacy keys may not parse. A legitimate stored 200000 cap can then be silently upgraded to the registry window.
  - confidence: 0.84
