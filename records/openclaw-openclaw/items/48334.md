## Summary

Keep open: current main still has a source-reproducible cron context-window fallback, but this branch is not merge-ready because it regresses the read-only status path, can misreport capped or invalid stored context values, is dirty against main, and lacks real behavior proof.

## What This Changes

The branch adds async model context-token resolution, uses it from cron and status summaries, adds per-agent contextTokens typing, and adds regression tests for cron persistence and status display repair.

## Best Possible Solution

Land a rebased cron persistence fix that resolves the provider-aware model context window during cron finalization while keeping status on its current read-only resolver; handle stale displayed values only through a read-only, cap-preserving path or a doctor migration with focused tests.

## Reproduction Assessment

Yes, by source inspection rather than a live run. Current main can still fall through to DEFAULT_CONTEXT_TOKENS for a first cron run when lookupContextTokens misses with allowAsyncLoad: false and no prior positive session value exists.

## Solution Assessment

No. The cron-side provider-aware resolution is the right direction, but the PR's status-side async discovery and repair logic are not the narrowest maintainable shape because they violate current read-only status guards and cap/value validation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Keep status summaries on the read-only resolver:** `src/commands/status.summary.ts:178-184`
  - body: This switches status to `resolveContextTokensForModelAsync`, which awaits registry loading. Current main explicitly keeps `status` and `status --json` read-only with `allowAsyncLoad: false` so they do not start model discovery or plugin scans; merging this would make a status command block on that runtime work and reintroduce side effects in a diagnostic path.
  - confidence: 0.9
- **[P2] Preserve agent caps when repairing fallback tokens:** `src/commands/status.summary.ts:46-47`
  - body: `hasExplicitContextTokensCap` returns false whenever no agent id is available, but the all-sessions pass builds rows without an agent override and legacy keys may not parse to an agent id. A legitimate per-agent 200k cap can then be displayed as a larger registry window, so remaining/% usage is wrong for capped agents.
  - confidence: 0.82
- **[P2] Validate stored context token values before reuse:** `src/commands/status.summary.ts:214-215`
  - body: The new `storedContextTokens` path accepts any numeric `entry.contextTokens`, including 0, negative values, or NaN. The current resolver only treats positive numbers as overrides, so malformed or older session stores would now emit invalid context/remaining-token values instead of falling back to the resolved model window.
  - confidence: 0.78
