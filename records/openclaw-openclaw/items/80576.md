## Summary

Keep open: current main still has a source-visible path where a subagent spawn can trigger session-store maintenance without preserving the requester/main session row, so this is not safe to close as fixed or duplicate.

## What This Changes

The issue reports that in OpenClaw 2026.5.7, spawning a sub-agent from Control UI makes the `main` agent/session disappear.

## Best Possible Solution

Preserve the requester/main session row during subagent child-session writes and add a focused regression proving session maintenance cannot evict `agent:main:main` while spawning a subagent.

## Reproduction Assessment

Yes, at source level: with default/enforced maintenance and a store over the cap high-water, a `sessions_spawn` child-row write can run capping without preserving `agent:main:main`. I did not run the full WSL2 Control UI flow.

## Solution Assessment

No close path is justified yet. The narrow maintainable fix is to preserve lifecycle-critical requester/main keys during subagent session-store writes while keeping the intentional default-enforce maintenance policy.
