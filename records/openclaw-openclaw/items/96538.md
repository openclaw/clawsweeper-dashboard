## Summary

Keep open: current main still has a source-reproducible path where media-store IDs containing `---<uuid>` become `WebMediaResult.fileName`, and Telegram passes that value directly to `grammy` `InputFile` for outbound delivery.

## What This Changes

The issue requests stripping OpenClaw media-store `---<uuid>` suffixes from filenames sent through Telegram outbound attachments.

## Best Possible Solution

Normalize media-store-backed `WebMediaResult.fileName` to the caller-facing original filename in the shared media loader, then cover both `media://inbound` and Telegram outbound send behavior.

## Reproduction Assessment

Yes. Source inspection shows stored media filenames can include `---<uuid>`, `loadWebMedia` returns that physical basename, and Telegram passes it to `InputFile`; I did not run a live Telegram send.

## Solution Assessment

Yes. The best fix is shared media-loader filename normalization using the existing store helper; stripping only in Telegram would duplicate media-store internals inside the plugin boundary.

## Repair Work Prompt

Fix the Telegram outbound filename suffix leak reported at https://github.com/openclaw/openclaw/issues/96538. Current main stores media IDs as `name---<uuid>.<ext>` and `loadWebMedia` returns that physical basename for `media://inbound` or managed inbound paths, which Telegram then passes into `new InputFile(..., fileName)`. Keep the fix in the shared media filename normalization layer, preferably reusing `extractOriginalFilename`, so all channel callers get the caller-facing filename without duplicating media-store internals in the Telegram plugin. Add focused coverage for `media://inbound/report---<uuid>.pdf` and for Telegram outbound send preserving `report.pdf` in the `InputFile` filename. Do not add config, migrations, or CHANGELOG.md edits.
