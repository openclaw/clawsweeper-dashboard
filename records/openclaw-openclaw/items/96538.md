## Summary

Keep this issue open: current main still appears to derive Telegram outbound attachment filenames from the staged media path, so a media-store ID like `photo---<uuid>.jpg` can plausibly reach the Telegram `InputFile` name. The report is narrow, source-reproducible, and fits a small bug-fix path rather than cleanup closure.

## What This Changes

The issue asks OpenClaw to strip internal media-cache UUID suffixes from filenames before sending outbound files through Telegram.

## Best Possible Solution

Normalize the Telegram-facing attachment filename at the outbound media boundary using the existing media-store original-name semantics, without changing stored media IDs or adding configuration.

## Reproduction Assessment

Yes, from source inspection: Telegram uses the loaded media `fileName`, and local staged media currently derives that name from the path basename. I did not run a live Telegram send, so this is source-reproducible rather than directly reproduced.

## Solution Assessment

Yes, a narrow fix at the outbound filename inference boundary looks like the best path. The existing `extractOriginalFilename` behavior should be reused or mirrored rather than adding a Telegram-specific config option or changing media storage IDs.
