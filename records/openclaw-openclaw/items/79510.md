## What This Changes

The PR makes gateway config hot reloads cancellation-aware during shutdown/restart, drains active reload work on reloader stop, moves reloader stop before channel teardown, and adds focused regression coverage plus test-only expectation updates.

## Best Possible Solution

Land the cancellation-aware gateway reload lifecycle fix after normal maintainer merge checks; this is narrower than adding process-wide channel single-flight and directly stops stale reload work.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: defer a channel hot reload behind active work, trigger SIGUSR1/gateway-tool restart, and current main can still run the old deferred channel start after the new gateway lifecycle starts. I did not live-bind a Telegram webhook in this read-only review.

## Solution Assessment

Yes, the proposed cancellation and drain path is the narrow maintainable fix for this race. It fences the config reloader at shutdown and makes deferred channel reload work observe cancellation instead of masking stale work with broader channel start single-flight.
