No blocking findings. This should stay open as an active fix PR: current main still has the source-reproducible deferred channel reload/restart race, and the PR adds the expected cancellation and drain path with focused regression coverage and supplied terminal proof.

## What This Changes

The PR modifies gateway config reload, shutdown, and reload-handler code to abort deferred channel reloads during shutdown/restart, with focused tests, a changelog entry, and test-only expectation/type alignments.

## Best Possible Solution

Merge a focused gateway lifecycle cancellation fix after normal maintainer review, keeping the behavior generic to config reload/restart rather than adding Telegram-specific policy.

## Reproduction Assessment

Yes. The linked report plus current-main source inspection gives a high-confidence path: defer a channel hot reload behind active work, trigger gateway-tool/SIGUSR1 restart before the deferral drains, and the old reload can still call channel start while the restarted gateway starts channels.

## Solution Assessment

Yes. Cancelling and draining config reload work at the gateway close/restart boundary is the narrow maintainable fix; process-wide channel single-flight would be broader and would mask stale lifecycle work instead of stopping it.
