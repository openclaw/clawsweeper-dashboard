## Summary

Keep open: this is an active bug-fix PR for a source-reproducible gateway restart race, and current main still lacks the cancellation/drain path that the branch adds. I found no blocking correctness findings in the proposed gateway lifecycle fix, and the supplied terminal proof is sufficient for this non-visual runtime path.

## What This Changes

The PR passes gateway config-reload cancellation context into hot reloads, drains active reloads during shutdown, skips cancelled channel restarts, adds focused regression coverage, and updates related tests and changelog text.

## Best Possible Solution

Land a gateway-level cancellation fence for deferred channel reloads so shutdown/restart owns pending reload work before channel lifecycle teardown.

## Reproduction Assessment

Yes. Current main has a clear source path: deferred channel reload waits without cancellation, shutdown currently stops channels before the reloader, and the linked issue supplies a concrete Telegram EADDRINUSE timeline for the race.

## Solution Assessment

Yes. Passing cancellation state from the config reloader and stopping it before channel teardown is a narrow lifecycle fix; a process-wide channel single-flight would be broader and would mask stale reload work instead of cancelling it.
