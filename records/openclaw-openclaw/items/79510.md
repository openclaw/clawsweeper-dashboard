Keep this PR open for normal PR handling. Current main still has the reported deferred channel-reload/restart race, and the patch addresses that path with cancellation/drain coverage and supplied real behavior proof, but GitHub currently reports the branch as merge-conflicting.

## What This Changes

The branch adds cancellation context and drain behavior to gateway config hot reloads, stops the reloader before channel teardown, skips deferred channel restart work after shutdown/restart cancellation, updates gateway regression tests and changelog, and adjusts current test expectations in command and Microsoft Teams test files.

## Best Possible Solution

Rebase or otherwise resolve the current conflicts, rerun the normal gateway/check gates, and land this cancellation-based fix so the linked bug closes on merge.

## Reproduction Assessment

Yes. Source inspection on current main plus the linked issue logs give a high-confidence path: defer a Telegram channel hot reload behind active work, request a SIGUSR1/gateway-tool restart during the deferral, and the old reload path can start the channel while the restarted gateway starts it too.

## Solution Assessment

Yes. Fencing the config reloader and making deferred channel reload work cancellation-aware is narrower than adding process-wide channel single-flight and directly targets the stale lifecycle work.
