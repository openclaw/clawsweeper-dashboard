## Summary

Keep open: current main still has the deferred channel-reload race shape, and this PR adds a focused cancellation path with regression coverage and sufficient after-fix proof. I found no blocking correctness finding in the proposed gateway changes.

## What This Changes

The PR passes a cancellation context through gateway config hot reloads, aborts and drains active reloads during close before channel teardown, skips deferred channel restarts after shutdown/restart cancellation, and updates focused tests and the changelog.

## Best Possible Solution

Land the focused gateway-owned cancellation/drain fix after normal PR refresh and merge gates, keeping the broader targeted channel-reload command tracked separately.

## Reproduction Assessment

Yes. The linked report's race maps directly to current main: channel reload defers on active work, then unconditionally stops/starts channels, while shutdown stops channels before stopping the config reloader and has no cancellation context.

## Solution Assessment

Yes. Cancelling and draining the config reloader during shutdown/restart is the narrow maintainable fix for stale deferred reload work; a per-channel single-flight start would be broader than needed for this race.
