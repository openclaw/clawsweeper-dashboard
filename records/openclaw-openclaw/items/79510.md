## Summary

Keep open for normal maintainer review/landing: current main still lacks the gateway reload cancellation fence, and this PR appears to implement the narrow fix with focused coverage, sufficient real behavior proof, and green exact-head checks.

## What This Changes

This PR adds cancellation context to gateway config hot reloads, drains active reloads during stop, stops the config reloader before channel teardown, and adds gateway regression coverage plus test-only rebase adjustments and a changelog entry.

## Best Possible Solution

Land the lifecycle cancellation fence on main after normal maintainer review, then close the linked bug once the PR merges.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: defer a channel reload behind active work, trigger gateway restart before the deferral drains, and current main can still call startChannel from the old reload path. I did not live-bind a Telegram webhook in this read-only review.

## Solution Assessment

Yes, the PR's abort/drain context and earlier config-reloader shutdown are the narrow maintainable fix for stale deferred reload work. A process-wide channel single-flight would be broader and would mask old lifecycle work rather than canceling it.
