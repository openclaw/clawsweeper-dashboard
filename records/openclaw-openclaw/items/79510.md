## Summary

Keep this PR open: the cancellation direction is right and the supplied proof/CI are useful, but the patch still leaves a shutdown-start window where a deferred channel reload can resume before the reloader is actually stopped.

## What This Changes

The branch adds gateway config-reload cancellation context, drains active reloads on reloader stop, moves reloader shutdown before channel teardown, and updates gateway plus related test expectations and changelog text.

## Best Possible Solution

Abort or fence config reload work as soon as gateway close/restart begins, then keep the PR's cancellation-aware reload handler so deferred channel starts cannot run during any awaited shutdown step.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: a deferred channel reload waits for active work while gateway close/restart can enter awaited shutdown steps, and current main has no shutdown-start cancellation fence for that reload.

## Solution Assessment

No. The cancellation context is the right fix direction, but placing configReloader.stop() after shutdown hooks/session drain is not early enough to close the reported race.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Stop the reloader before shutdown waits can drain:** `src/gateway/server-close.ts:263`
  - body: The new shutdown order aborts the config reloader only after gateway shutdown/pre-restart hooks, the session-end drain, and bonjour/tailscale cleanup. While any of those awaited steps run, active work can finish and the deferred channel reload can pass its wait and stop/start Telegram before the abort signal is sent. Move the reloader stop/fence to the start of close handling, or otherwise mark reload cancellation as soon as shutdown begins, and add a regression where the drain releases active work before channel teardown.
  - confidence: 0.86
