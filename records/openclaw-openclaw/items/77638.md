## Summary

Keep open: the PR targets the right edit-recovery bug, but the new noop branch is too broad and can hide non-mismatch edit failures as successful noops.

## What This Changes

The PR adds an edit-tool noop classifier, regression tests, and a changelog entry for already-at-target concurrent edit followers.

## Best Possible Solution

Keep the PR open and narrow the noop recovery to exact edit-mismatch followers while preserving non-mismatch error propagation.

## Reproduction Assessment

Yes. By source inspection, a file already at the requested target state plus a non-mismatch `base.execute` throw now reaches the new branch and returns `Edit noop` before the original error can be rethrown.

## Solution Assessment

No. The classifier belongs in this wrapper, but it should only convert the upstream exact-text mismatch follower case to noop and should leave unrelated errors on the existing throw path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Gate noop recovery on mismatch errors:** `src/agents/pi-tools.host-edit.ts:254-261`
  - body: The new branch runs for every caught error once the file looks already up to date. If `base.execute` throws an AbortError, permission failure, diff-generation error, or any other non-exact-match error while the file already contains the target text, this now returns `isError: false` and hides the failure. The requested recovery is for exact-text mismatch followers, so guard this branch on the mismatch error before returning noop.
