## Summary

Keep open: the patch is the right wrapper-level direction, but it still misses the upstream multi-edit mismatch error shape, so the linked issue's `edits[]` case would remain broken.

## What This Changes

The PR adds an edit-tool recovery branch that returns a distinct noop result when a mismatched edit finds the file already at the requested target state, with regression tests and a changelog entry.

## Best Possible Solution

Keep the post-state classifier, but extend mismatch recognition to the pinned upstream single-edit and multi-edit not-found messages and pin regression coverage to the real multi-edit error shape.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: upstream emits `Could not find edits[0] in ...` for a queued multi-edit follower, while this PR only treats `Could not find the exact text in ...` as recoverable.

## Solution Assessment

No. The classifier is the right narrow fix shape, but the predicate and multi-edit test need to match the upstream dependency contract before this solves the linked bug.
