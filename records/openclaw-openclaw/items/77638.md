## Summary

Keep open: the PR is a good narrow fix shape, but it still misses the upstream multi-edit not-found error string, so real `edits[]` followers can keep throwing instead of returning the new noop result.

## What This Changes

The PR adds an edit-tool recovery branch, tests, and a changelog entry for returning `Edit noop: content already up to date in <path>.` when an edit mismatch finds the file already at the requested target state.

## Best Possible Solution

Keep the wrapper-level post-state classifier, but extend mismatch recognition to the pinned upstream single-edit and multi-edit not-found messages and pin regression coverage to the real upstream shapes.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a queued `edits[]` follower whose target is already present receives upstream’s `Could not find edits[0] in ...` error, while this PR’s gate only recognizes `Could not find the exact text in`.

## Solution Assessment

No. The state classifier is the right narrow direction, but the predicate and multi-edit regression need to match the pinned upstream multi-edit error contract before merge.
