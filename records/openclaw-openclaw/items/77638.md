## Summary

Keep open: the PR targets the right recovery wrapper and regression tests, but the new noop branch can mask non-mismatch edit-tool failures because it runs before the mismatch predicate; the branch is also currently merge-conflicting.

## What This Changes

The PR adds an already-at-target noop classifier/result to `wrapEditToolWithRecovery`, regression tests for the linked edit concurrency issue, and an Unreleased changelog fix entry.

## Best Possible Solution

Keep the recovery-wrapper approach, but gate already-target noop recovery on the exact edit mismatch class, preserve all other errors, add a regression for unrelated thrown errors, and resolve the branch conflict before merge.

## Reproduction Assessment

Yes. Source inspection shows current main’s wrapper refuses success when before/after snapshots match and then rethrows/enhances the upstream exact-match mismatch, which is the linked follower path after the leader has already replaced `oldText`.

## Solution Assessment

No. The recovery-wrapper location is right, but the proposed branch is too broad because it returns noop for any thrown edit error when the file is already target; it should apply only to the exact-match follower mismatch.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Restrict noop recovery to mismatch errors:** `src/agents/pi-tools.host-edit.ts:254-261`
