## Summary

Keep open: this maintainer-labeled Mattermost feature PR is directionally useful, but the current diff misses the default draft-preview finalization path, so visible threaded replies can still fail to mark the thread as participated and follow-ups can still be dropped.

## What This Changes

The PR adds a Mattermost thread-participation cache backed by plugin state, threads it into the Mattermost mention gate, records participation after visible thread replies, and adds gate/cache tests.

## Best Possible Solution

Keep the Mattermost-local cache/gate shape, but record participation from a delivery result that covers both normal sends and preview-finalized visible replies, then document and prove the default thread-follow-up behavior.

## Reproduction Assessment

Yes, by source inspection: default Mattermost streaming creates a draft-preview path, and successful same-thread finalization edits that preview without calling the deliverPayload callback where this PR records participation. I did not run a live Mattermost instance in this read-only review.

## Solution Assessment

No: the Mattermost-local cache and gate boundary is a reasonable fix shape, but recording must cover all visible final delivery outcomes, including preview-finalized replies. The safer repair is to centralize the participation mark after visible final delivery and add a regression test for default streaming.
