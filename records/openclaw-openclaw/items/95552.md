## Summary

Keep open: this maintainer-labeled PR is useful Mattermost parity work, but it still has a blocking delivery-path bug where default draft-preview finalization can visibly answer without recording thread participation, so the follow-up this PR is meant to allow can still be dropped.

## What This Changes

The PR adds a Mattermost thread-participation cache backed by plugin state, feeds that state into the Mattermost mention gate, records visible thread replies, and adds gate/cache tests.

## Best Possible Solution

Keep the Mattermost-local cache/gate approach, but record participation after every visible final delivery path, document the thread exception, and then prove the default restart follow-up behavior.

## Reproduction Assessment

Yes, by source inspection: default Mattermost streaming creates a draft-preview path, and successful same-thread finalization edits that preview without calling the deliverPayload callback where this PR records participation. I did not run a live Mattermost instance in this read-only review.

## Solution Assessment

No. The Mattermost-local cache and gate boundary is reasonable, but the implementation must record all visible final delivery outcomes, including preview-finalized replies, and document the new default behavior before it is the best fix.
