## Label Justifications

- `P1`: A real Feishu channel workflow can silently lose intended @mention rendering or notification behavior in default card/streaming delivery.
- `impact:message-loss`: The issue is about channel mentions being rendered or delivered with the wrong Feishu message-format syntax.

## Summary

Keep open: current main and the latest release still do not normalize agent-authored Feishu `<at user_id="...">name</at>` tags at the post, card, proactive outbound, or streaming-card boundaries, and the linked open PR only addresses part of the post-message structured-mention problem.

## What This Changes

The issue requests Feishu-owned outbound normalization so agent-authored mention tags are converted to the correct Feishu syntax for post, text, and interactive card delivery.

## Best Possible Solution

Add one Feishu-owned outbound normalization helper and apply it at the post, card, proactive outbound, and streaming-card boundaries while preserving the current no-automatic-mention-cascade behavior.

## Reproduction Assessment

Yes, at source level. A Feishu reply or proactive send with card render mode or streaming enabled and agent text containing `<at user_id="ou_xxx">Name</at>` still flows into card/post delivery without format-specific conversion on current main.

## Solution Assessment

Yes. A localized Feishu plugin normalizer with focused regression tests is the narrow maintainable fix; the separate push-notification limitation should stay tracked in https://github.com/openclaw/openclaw/issues/73469.
