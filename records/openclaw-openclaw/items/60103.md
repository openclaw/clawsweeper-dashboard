## Label Justifications

- `P1`: A real Feishu channel workflow can silently lose intended @mention behavior for agent replies in the default card/streaming path.
- `impact:message-loss`: The issue is about channel mentions rendering or notifying incorrectly when Feishu outbound message formats receive the wrong mention syntax.

## Summary

Keep open. Current main and the latest release still pass Feishu agent-authored mention tags through outbound post, card, streaming-card, and proactive-send paths without converting them to the format-specific Feishu syntax; the related open PR only covers part of the post-message structured-mention path, and the streaming notification issue remains separate.

## What This Changes

Request automatic Feishu outbound mention-tag normalization so agent-authored `<at ...>` tags are converted for card, text/post, and streaming delivery formats.

## Best Possible Solution

Add a Feishu-owned outbound normalization helper at the post/text, card, proactive outbound, and streaming-card boundaries, while preserving the current no-automatic-mention-cascade behavior and coordinating with the partial post-message work in https://github.com/openclaw/openclaw/pull/49983.

## Reproduction Assessment

Yes, at source level. A Feishu reply or proactive send with card render mode or streaming enabled and agent text containing `<at user_id="ou_xxx">Name</at>` still flows into card/post delivery without conversion on current main.

## Solution Assessment

Yes. A localized Feishu plugin normalizer plus focused regression tests is the narrow maintainable fix; the streaming push-notification limitation should remain a separate Feishu API strategy discussion.
