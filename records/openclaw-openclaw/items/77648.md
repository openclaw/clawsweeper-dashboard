## Summary

Keep open: the report matches current Slack source behavior, and the protected `maintainer` label requires explicit maintainer handling rather than automated cleanup.

## What This Changes

Ensure every successful visible threaded Slack send records bot thread participation so later untagged replies bypass mention gating unless explicit mentions are required.

## Best Possible Solution

Add post-success participation recording inside `sendMessageSlack`, keep duplicate action-runtime recording unless follow-up cleanup proves it unnecessary, and cover direct send, outbound adapter, and inbound mention-gate behavior with focused tests.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: a direct `sendMessageSlack(..., { threadTs })` call can visibly post into a thread without seeding the participation cache, so a later untagged reply depends only on `parent_user_id` and can be dropped as `no-mention`. I did not run a live Slack workspace scenario.

## Solution Assessment

Yes, centralizing recording in `sendMessageSlack` after successful threaded delivery is the narrowest maintainable fix because direct replies, Block Kit/outbound delivery, media, and approval sends converge there. Existing action-runtime recording can remain as harmless idempotent coverage unless tests support removing it.
