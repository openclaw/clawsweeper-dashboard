## Label Justifications

- `impact:session-state`: The report concerns an unexpectedly large cached prefix and continued context growth immediately after creating a fresh session.

## Summary

Keep this issue open with the linked implementation PR. The reported two-account discrepancy is credible and current/shipped OpenClaw does share one agent-owned Codex home across auth profiles, but neither the issue nor the PR yet proves that this shared local state causes the provider-reported cached prefix; whole-home account isolation also changes ownership of Codex configuration, skills, sessions, and other persistent state.

## What This Changes

Investigate why two OpenAI OAuth accounts using the same native Codex setup begin fresh sessions with substantially different cached context sizes.

## Best Possible Solution

Use the linked PR as the sole implementation vehicle, capture first-turn payload sizing and token usage for both accounts under controlled shared and clean homes, then either isolate only proven account-owned state or add per-profile homes with explicit migration and fresh-install/upgrade coverage.

## Reproduction Assessment

No, not yet with high confidence. The reporter provides a repeatable two-account `/new` flow, but there is no controlled account/home swap, captured first-turn payload, or live after-fix run establishing whether local Codex state or upstream account state causes the discrepancy.

## Solution Assessment

No, the proposed whole-home isolation is not yet established as the best solution. It is plausible, but upstream CODEX_HOME owns several persistent surfaces beyond authentication, so causal live proof and an upgrade-state decision are required first.
