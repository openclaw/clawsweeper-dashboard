## Summary

Current main still launches every native Codex process for an agent with the same agent-owned `CODEX_HOME`, while upstream Codex roots authentication, native thread state, configuration, and diagnostics there. The issue remains necessary because the account-specific token inflation has not been independently reproduced or causally tied to shared local state, and the paired closing PR remains open without real two-account proof.

## What This Changes

Investigate why one of two OpenAI OAuth accounts starts fresh native Codex sessions with a much larger cached context prefix on the same machine.

## Best Possible Solution

Run a controlled two-account matrix that holds the OAuth profile and `CODEX_HOME` independently constant, then implement account isolation only if the inflated prefix follows local state, with an explicit upgrade plan for existing native Codex state.

## Reproduction Assessment

No. The reporter gives a concrete repeatable comparison and source inspection confirms a shared local state root, but no independent current-main run shows whether the inflation follows the OAuth account, the shared home contents, or remote Codex behavior.

## Solution Assessment

Unclear. Per-profile home isolation is plausible, but the best fix cannot be selected until the causal matrix is run and maintainers decide how existing persisted Codex state should behave during upgrades.
