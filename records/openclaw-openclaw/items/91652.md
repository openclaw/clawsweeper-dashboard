## Summary

Keep open: current main partially addresses related Telegram fallback behavior, but the central reported problem still has a plausible source path where a transcript recovery or queued followup turn can generate assistant content without delivering it back to Telegram.

## What This Changes

The issue reports that Telegram async voice transcript recovery can generate an assistant response that is then dropped by the Telegram delivery mirror as `No response generated`.

## Best Possible Solution

Preserve the original Telegram delivery target for transcript recovery turns and deliver generated assistant content exactly once, while keeping message-tool-only privacy for ambient room events that intentionally produce no source reply.

## Reproduction Assessment

No high-confidence live repro was established in this read-only review. Source inspection gives a medium-confidence path: generated recovery/followup content can miss Telegram delivery when no sendable final is routed and message-tool-only or fallback suppression decisions own the visible path.

## Solution Assessment

Unclear at the exact code-boundary level. The best repair should be a focused Telegram/followup routing fix with a regression test, not a new config option or broad fallback that would weaken message-tool-only privacy.
