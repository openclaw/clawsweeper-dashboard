## Summary

Keep open: current main still rejects shared-secret Bearer session-history reads, and this PR is a focused, apparently correct fix with REST/SSE coverage and supplied real behavior proof. The remaining decision is whether maintainers accept `/sessions/:key/history` as another shared-secret trusted-operator HTTP surface, plus landing coordination with the open internal-session hardening PR.

## What This Changes

The PR switches `/sessions/:key/history` initial auth and SSE reauth to the shared-secret compatible HTTP scope resolver and adds REST/SSE regression tests.

## Best Possible Solution

Land this narrow resolver/test change if maintainers accept session-history HTTP as a shared-secret trusted-operator surface, then preserve compatibility with the internal-session guard during landing.

## Reproduction Assessment

Yes. Source inspection shows current main maps shared-secret Bearer requests to no trusted declared scopes on this endpoint, then `chat.history` authorization requires `operator.read`, producing the reported 403 and SSE reauth closure path.

## Solution Assessment

Yes, assuming the trust-boundary decision is accepted. Reusing `resolveOpenAiCompatibleHttpOperatorScopes` for both initial auth and SSE reauth is the narrowest implementation and leaves stricter endpoints like session kill/control UI on the trusted resolver.
