## What This Changes

The PR changes session-history HTTP and SSE authorization to use the OpenAI-compatible resolver and adds bearer HTTP/SSE regression coverage plus a revocation mock update.

## Best Possible Solution

Land the narrow resolver/test change only after redacted real gateway GET/SSE proof and maintainer acceptance of the session-history bearer boundary, ideally coordinated with the internal-session HTTP guard.

## Reproduction Assessment

Yes. Current main clearly routes /sessions/:key/history and its SSE reauth through resolveTrustedHttpOperatorScopes, so shared-secret bearer requests without trusted declared scopes fail chat.history authorization before reading history.

## Solution Assessment

Yes, with a boundary caveat. Reusing the existing OpenAI-compatible resolver at the initial request and SSE reauth points is the narrow maintainable fix if maintainers agree this route belongs in that bearer trust bucket.
