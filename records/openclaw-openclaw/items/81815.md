## Summary

Keep open for maintainer handling. Current main still denies shared-secret bearer callers on the session-history REST and SSE authorization paths, while this PR’s narrow resolver/test change appears correct and has supplied real terminal proof; the remaining question is maintainer acceptance of adding this route to the shared-secret default-scope HTTP bucket.

## What This Changes

The PR switches `/sessions/:key/history` REST and SSE authorization to the OpenAI-compatible shared-secret default-scope resolver and adds bearer-auth REST/SSE regression coverage.

## Best Possible Solution

Land this narrow resolver/test change with trust-model documentation alignment if maintainers accept session history as a shared-secret default-scope HTTP surface.

## Reproduction Assessment

Yes. Current main can be traced from both session-history authorization points through `resolveTrustedHttpOperatorScopes`, which returns no scopes for shared-secret bearer auth and therefore fails `chat.history` read-scope authorization; I did not run tests because this was a read-only review.

## Solution Assessment

Yes, if maintainers accept the trust-boundary policy. Reusing `resolveOpenAiCompatibleHttpOperatorScopes` at both REST and SSE call sites is the narrowest maintainable implementation for that policy, with focused regression coverage.
