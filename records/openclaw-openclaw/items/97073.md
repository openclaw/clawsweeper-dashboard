## Label Justifications

- `P2`: This is a normal-priority Slack session-context bug with concrete user impact but no core outage, security emergency, or release-blocker signal.
- `impact:session-state`: A stale Slack thread session row can cause the next reset turn to lose expected thread context while creating a new OpenClaw session.
- `impact:message-loss`: Prior Slack thread messages can be omitted from the prompt even though the configured thread history limit should allow bounded context seeding.

## Summary

Keep open. Current main still has the timestamp-only Slack thread-history gate while reply session initialization uses reset-policy freshness, and this issue has both a protected `maintainer` label and an open linked fix PR, so cleanup closure would be premature.

## What This Changes

The issue asks for Slack replies in an existing thread after a daily or idle session reset to seed bounded Slack thread history into the new prompt.

## Best Possible Solution

Keep this issue as the canonical tracker until https://github.com/openclaw/openclaw/pull/97100 or an equivalent reset-aware Slack fix lands while preserving the fresh-session bloat guard, `initialHistoryLimit: 0`, `historyScope`, and `inheritParent` semantics.

## Reproduction Assessment

Yes at source level: create an existing Slack thread session row that is stale under the reset policy but still has `updatedAt`, then process a thread reply; Slack prepare skips history while session initialization treats the row as stale and starts a new session. I did not run a live Slack reproduction in this read-only review.

## Solution Assessment

Yes, the best fix direction is to align Slack preparation with the same lifecycle freshness contract used by session initialization through a narrow Slack/plugin SDK seam. Replaying archived OpenClaw transcripts or treating `inheritParent` as the fix would solve the wrong layer.
