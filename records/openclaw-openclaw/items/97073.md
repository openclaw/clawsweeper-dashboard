## Label Justifications

- `P2`: This is a normal-priority Slack/session-context bug with concrete user impact but no core outage, security bypass, or release-blocker signal.
- `impact:session-state`: A stale Slack thread session row can cause a reset turn to lose expected thread context when session freshness and Slack history seeding disagree.
- `impact:message-loss`: Prior Slack thread messages that should fit within `initialHistoryLimit` can be omitted from the prompt after rollover.

## Summary

Keep open. Current main and the latest release still have the Slack timestamp-only thread-history gate, the issue has the protected `maintainer` label, and the linked implementation candidate at https://github.com/openclaw/openclaw/pull/97100 is open rather than merged.

## What This Changes

Slack thread replies after a daily reset should seed bounded prior Slack thread history instead of starting the reset turn without thread context.

## Best Possible Solution

Land a focused reset-aware Slack thread-history fix through the linked PR or a narrow replacement while preserving the fresh-row bloat guard, `initialHistoryLimit: 0`, `historyScope`, and `inheritParent` semantics.

## Reproduction Assessment

Yes at source level: create an existing Slack thread session row stale under the daily reset policy, then process a reply in that thread; Slack prepare sees a timestamp and skips history while session initialization evaluates the row as stale and starts a new session. I did not run a live Slack reproduction in this read-only pass.

## Solution Assessment

Yes, the narrow fix direction is to align Slack's initial thread-history gate with the same reset freshness contract used by session initialization. Replaying archived OpenClaw transcripts or redefining `inheritParent` would be broader and riskier than bounded Slack API history seeding.
