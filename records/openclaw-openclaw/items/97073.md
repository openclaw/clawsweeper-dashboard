## Label Justifications

- `P2`: This is a normal-priority Slack session-context bug with concrete user impact but no core outage or release-blocker signal.
- `impact:session-state`: A stale Slack thread session row can cause the next agent turn to lose expected context after reset rollover.
- `impact:message-loss`: Prior Slack thread messages can be omitted from prompt context even when the configured initial history limit would allow them.

## Summary

Keep open. Current main and the latest release still have the reported Slack/session mismatch, and the protected `maintainer` label requires explicit maintainer handling rather than cleanup closure.

## What This Changes

Slack thread replies after a reset-policy rollover should seed bounded Slack thread history instead of starting the next prompt without prior thread context.

## Best Possible Solution

Implement a narrow reset-aware Slack thread context decision inside the Slack plugin boundary, preserving fresh-session bloat protection, `initialHistoryLimit: 0`, `historyScope`, and `inheritParent` semantics.

## Reproduction Assessment

Yes at source level: create an existing Slack thread session row stale under the daily reset policy, then process a reply in that thread; Slack prepare sees a timestamp and skips history while reply session initialization can roll the row into a new session. I did not run a live Slack reproduction in this read-only review.

## Solution Assessment

Yes, the proposed direction is the right shape: align Slack's initial thread-history gate with the same reset freshness contract used by session initialization. The fix should not replay archived OpenClaw transcripts or add a new config option.
