## Label Justifications

- `P2`: This is a normal-priority Slack/session-context bug with concrete user impact but no core outage or release-blocker signal.
- `impact:session-state`: The issue is about a stale Slack thread session row causing the next agent turn to lose expected thread context after rollover.
- `impact:message-loss`: Prior Slack thread messages can be skipped from the prompt context even though the configured history limit would otherwise allow them.

## Summary

Keep open. Current main and the latest release still have a reset-policy mismatch: Slack thread-history seeding treats any stored thread session timestamp as an existing session, while reply session initialization can immediately roll that same stale row into a fresh OpenClaw session; the protected `maintainer` label also blocks conservative cleanup closure.

## What This Changes

The issue asks for Slack thread-history seeding to become reset-policy-aware so the first Slack thread reply after daily session rollover can receive bounded prior thread context.

## Best Possible Solution

Implement a narrow reset-aware Slack thread context decision using the plugin SDK session/freshness helpers, preserving fresh existing-session skip behavior, `initialHistoryLimit: 0`, `historyScope`, and `inheritParent` semantics.

## Reproduction Assessment

Yes at source level: create an existing Slack thread session row stale under the daily reset policy, then process a reply in that thread; Slack prepare sees a timestamp and skips history, while `initSessionState` evaluates the row as stale and starts a new session. I did not run a live Slack reproduction in this read-only pass.

## Solution Assessment

Yes, the proposed direction is the narrowest maintainable fix: align Slack's initial thread-history gate with the same reset freshness contract used by session initialization. The implementation should stay inside the Slack plugin/SDK boundary rather than replaying archived OpenClaw transcripts or adding a new config option.
