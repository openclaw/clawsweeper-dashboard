## Label Justifications

- `P2`: This is a normal-priority Slack/session-context bug with concrete user impact but no core outage, security emergency, or release-blocker signal.
- `impact:session-state`: A stale Slack thread session row can cause the next agent turn to lose expected thread context after reset rollover.
- `impact:message-loss`: Prior Slack thread messages can be omitted from prompt context even though the configured history limit would otherwise allow them.

## Summary

Keep open. Current main and v2026.6.10 still use a timestamp-only Slack thread-history gate while reply session initialization uses reset-policy freshness, so a stale thread row can skip bounded Slack history and then roll into a fresh OpenClaw session; the protected `maintainer` label also blocks conservative cleanup closure.

## What This Changes

The issue requests reset-aware Slack thread-history seeding so an existing Slack thread that rolls over after the daily reset can still fetch bounded prior Slack messages.

## Best Possible Solution

Align Slack's initial thread-history decision with the same reset freshness contract used by session initialization, while preserving fresh existing-session skip behavior, `initialHistoryLimit: 0`, `historyScope`, and `inheritParent` semantics.

## Reproduction Assessment

Yes, at source level: seed an existing Slack thread session row that is stale by daily reset policy, then process a reply in that thread; Slack prepare sees a stored timestamp and skips history, while `initSessionState` evaluates the same row as stale and starts a new session. I did not run a live Slack reproduction in this read-only pass.

## Solution Assessment

Yes, the requested direction is the narrow maintainable fix: make Slack's initial thread-history gate reset-aware instead of replaying archived OpenClaw transcripts or adding a new config option. The implementation should use an approved Slack/plugin SDK boundary rather than importing core internals from the plugin.
