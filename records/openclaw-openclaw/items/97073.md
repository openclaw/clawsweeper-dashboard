## Label Justifications

- `P2`: This is a normal-priority Slack/session-context bug with concrete prompt-context impact but no core outage or release-blocker signal.
- `impact:session-state`: The issue centers on stale Slack thread session rows causing the next agent turn to lose expected thread context after rollover.
- `impact:message-loss`: Prior Slack thread messages can be skipped from the prompt context even though the configured history limit would otherwise allow them.

## Summary

Keep open. Current main and v2026.6.10 still use the timestamp-only Slack thread-history gate, while session initialization can roll the same stale row to a new session; the linked fix PR is still open, draft/conflicting, and the protected `maintainer` label blocks cleanup closure.

## What This Changes

Requests Slack thread replies after daily session rollover to seed bounded Slack API thread history when the existing thread session row is reset-stale.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/97100 or an equivalent focused fix that makes Slack initial thread-history seeding reset-aware while preserving fresh-session, `initialHistoryLimit`, `historyScope`, and `inheritParent` semantics.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: seed an existing Slack thread session row stale under the daily reset policy with a current `updatedAt`, then process a thread reply; Slack prepare skips `conversations.replies` while session initialization will roll the row to a new session.

## Solution Assessment

Is this the best way to solve the issue? Yes: the narrow maintainable fix is to align Slack's initial thread-history gate with the same reset freshness contract used by session initialization, not to replay archived OpenClaw transcripts or add a new config option.
