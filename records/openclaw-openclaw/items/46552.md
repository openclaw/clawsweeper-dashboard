## Summary

Keep open: the partial/block streaming clarification is still useful and is not fully present on current main, but this PR still over-promises `interrupt` as a way to stop in-flight tools immediately. Current docs and source support an abort-and-wait contract, with actual tool cancellation depending on runtime/tool abort handling.

## What This Changes

The PR adds `docs/concepts/queue.md` guidance about steer with partial/block streaming, non-streaming fallback, tool-boundary timing, and session queue troubleshooting.

## Best Possible Solution

Land a small docs-only update against current main that keeps the partial/block streaming gotcha and aligns `interrupt` wording with the abort-and-wait/tool-boundary contract.

## Reproduction Assessment

Not applicable; this is a docs-only PR, not a failing runtime bug report. The claims are source-checkable against current queue docs, Discord streaming docs, and the runtime queue helper.

## Solution Assessment

No; as written it is not the best solution because it tells users `interrupt` can stop an in-flight tool immediately. The safer fix is to describe `interrupt` as aborting/replacing the active run, while noting cancellation depends on the runtime/tool honoring abort.
