## Summary

Keep this PR open: the docs direction is useful and the Greptile partial-vs-block concern was addressed, but the current head still overpromises that `interrupt` stops an in-flight tool immediately. Current docs and source describe abort/replace semantics that wait for the active run to end, with actual tool cancellation dependent on the backend/tool honoring abort.

## What This Changes

The PR adds queue documentation for steer behavior with partial/block streaming, non-streaming fallback, tool-boundary timing, and troubleshooting for session overrides and streaming-off fallback.

## Best Possible Solution

Land a concise docs update that keeps the partial/block streaming gotcha and troubleshooting guidance while describing `interrupt` as aborting/replacing the active run, with cancellation depending on the runtime and tool honoring abort.

## Reproduction Assessment

Not applicable; this is a docs-only PR, not a failing runtime bug report. The behavior claims are source-checkable against the queue, steering, streaming docs, and runtime queue implementation.

## Solution Assessment

No, not as currently worded. The docs direction is narrow and useful, but the `interrupt` sentence should be softened so it does not promise immediate in-flight tool termination.
