## Summary

Keep this maintainer-labeled PR open; the gateway-side scope restoration matches the stated direction, but the patch does not update the native bootstrap handoff filters that still drop `operator.talk.secrets`, so the advertised native Talk config path remains incomplete.

## What This Changes

This PR adds `operator.talk.secrets` to the setup-code/bootstrap operator handoff scope and updates gateway/docs expectations for the expanded handoff.

## Best Possible Solution

Align the gateway allowlist, Swift/Kotlin bootstrap persistence filters, native tests, and protocol docs around the same bounded scope set, then prove a trusted setup-code bootstrap can read Talk config secrets without granting admin or pairing.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: current gateway bootstrap excludes `operator.talk.secrets`, while native clients still filter the handed-off operator token to a smaller scope set. I did not run a live mobile pairing flow in this read-only review.

## Solution Assessment

No, not as submitted: adding the server allowlist entry is necessary but incomplete without the native persistence allowlists and tests. The safer fix is one aligned gateway plus native contract with real setup-code/Talk readback proof.
