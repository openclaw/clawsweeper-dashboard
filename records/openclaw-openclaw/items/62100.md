## Summary

Keep open. The requested model-resolution behavior is not fully implemented on current main, but this PR is not safe to merge as written because it changes provider-qualified parsing and widens model allowlist matching.

## What This Changes

The PR changes model-selection parsing and allowlist resolution to treat selected qwen/minimax slash-style refs as OpenRouter-native and adds regression tests for allowlisted OpenRouter native slash IDs.

## Best Possible Solution

Rework the fix at the allowed-model resolver layer: when raw <vendor>/<model> is supplied, map it to OpenRouter only if openrouter/<vendor>/<model> is explicitly configured or uniquely allowlisted, preserve provider-qualified refs otherwise, keep allowedKeys canonical, and add negative tests for real qwen/minimax providers plus positive tests for several OpenRouter vendor prefixes.

## Reproduction Assessment

Yes. A focused unit test can configure openrouter/qwen/qwen3.6-plus:free and a real qwen provider/model, then verify qwen/qwen3.6-plus:free is not accepted through the OpenRouter allowlist and is not rewritten away from the qwen provider.

## Solution Assessment

No. The PR addresses a real gap, but solving it inside parseModelRef and by adding bare allowed keys is too broad; resolver-level matching against explicit OpenRouter allowlist/config entries is the narrower maintainable fix.
