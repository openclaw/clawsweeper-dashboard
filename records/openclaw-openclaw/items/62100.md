## Summary

Keep open: current main does not implement the requested slash-style OpenRouter shorthand, but this PR is not mergeable as-is because it can misroute provider-qualified refs, widens model allowlist scope, covers only two vendor prefixes, and lacks after-fix real behavior proof.

## What This Changes

The PR changes model-selection parsing, configured-provider inference, allowlist construction, and tests so selected `minimax/...` and `qwen/...` refs can resolve through allowlisted OpenRouter entries.

## Best Possible Solution

If maintainers want this shorthand, implement resolver-level exact matching only for configured OpenRouter nested model IDs, keep allowlist keys provider-scoped, add regression coverage, and document the accepted shorthand behavior.

## Reproduction Assessment

Yes, by source inspection: the PR diff rewrites explicit `qwen/...` and `minimax/...` refs to OpenRouter and adds nested OpenRouter model IDs as bare allowed keys. Current main also shows the underlying requested shorthand is not implemented because provider inference is only applied to non-slash selections.

## Solution Assessment

No. Parse-time vendor rewrites and bare allowlist keys are not the safest solution; exact configured OpenRouter matching in the resolver would preserve provider intent and allowlist scope.
