## Summary

Keep open: the PR has a plausible compatibility goal, but it is not merge-ready because the patch can misroute provider-qualified refs, widen model allowlist scope, and lacks after-fix real behavior proof.

## What This Changes

The branch adds OpenRouter slash-ref inference in model selection and tests so raw `minimax/...` and `qwen/...` inputs can resolve to configured `openrouter/...` allowlist entries.

## Best Possible Solution

If maintainers want this shorthand, implement resolver-level exact matching only for configured OpenRouter nested model ids while preserving explicit provider prefixes and provider-scoped allowlist keys, then align docs/tests and proof.

## Reproduction Assessment

Yes for the patch defects, by source inspection: current main treats slash-containing refs as provider-qualified, while the PR rewrites selected prefixes and adds a bare allowlist key. I did not run a live OpenRouter setup.

## Solution Assessment

No. Parse-time vendor rewrites plus bare allowlist keys are not the narrowest maintainable solution; exact configured matching in the resolver would preserve provider intent and allowlist scope.
