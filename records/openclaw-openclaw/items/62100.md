## Summary

Keep open. Current main still has the requested OpenRouter native slash-id gap in the allowed-model resolver, and the PR is not mergeable because it changes parser/provider scoping in ways that can misroute provider-qualified refs and weaken allowlist enforcement.

## What This Changes

The PR modifies model-selection parsing, allowlist resolution, and regression tests so selected qwen/minimax slash-style model refs can resolve through allowlisted OpenRouter entries.

## Best Possible Solution

Rework the resolver to map raw <vendor>/<model> to OpenRouter only when an explicit openrouter/<vendor>/<model> entry is configured or uniquely allowlisted, while preserving explicit provider refs and canonical allowedKeys.

## Reproduction Assessment

Yes. Configure an allowlist with openrouter/qwen/qwen3.6-plus:free plus a real qwen provider/model, then resolve qwen/qwen3.6-plus:free through the allowed-model path and verify it neither bypasses provider scoping nor gets rewritten away from the qwen provider.

## Solution Assessment

No. The PR targets a real gap, but parse-time vendor rewrites and bare allowed keys are broader than necessary; resolver-level matching against configured OpenRouter entries is the narrower maintainable approach.

  - body: This branch rewrites qwen/<model> and minimax/<model> to provider openrouter whenever defaultProvider is openrouter.   - confidence: 0.89

  - body: Adding parsed.model to allowedKeys means an allowlist entry for openrouter/qwen/qwen3.6-plus:free can also admit qwen/qwen3.6-plus:free as a separate provider key. The gateway checks provider/model keys, so this widens the configured provider scope.
  - confidence: 0.93
- **[P2] Resolve all configured OpenRouter slash IDs:** `src/agents/model-selection.ts:149-155`
  - body: The new parseModelRef guard only recognizes minimax/ and qwen/, so the same OpenRouter-native pattern still fails for deepseek/, meta-llama/, mistralai/, google/, and other configured vendors. Resolve against configured openrouter/<raw> entries instead of maintaining a partial vendor list.
  - confidence: 0.84
