## Summary

Keep open: the bug is valid on current main, but this PR still misses the embedded payload error contract, so a short error payload can still become a memory filename slug. It also still needs contributor-supplied after-fix real behavior proof before merge.

## What This Changes

The PR extracts `slugifyLLMResponse`, adds response-shape checks and post-truncation dash stripping for LLM session-memory slugs, and adds focused slug-generator tests.

## Best Possible Solution

Land a narrow repair that rejects `payload.isError === true` before slugification, keeps the post-truncation dash strip, and adds `generateSlugViaLLM` boundary coverage.

## Reproduction Assessment

Yes, at source level. Current main can be reproduced by mocking `runEmbeddedPiAgent` to return `{ payloads: [{ text: "quota exceeded", isError: true }] }`; `generateSlugViaLLM` reads only `text`, and session-memory will use any non-null slug when `llmSlug` is enabled.

## Solution Assessment

No. The helper extraction and dash-order fix are useful, but text-shape heuristics alone are not the best fix because the embedded runner already exposes `payload.isError` for this boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Reject error payloads before slugifying:** `src/hooks/llm-slug-generator.ts:134`
  - body: The embedded runner exposes `payload.isError`, but this code drops the flag and passes only `payload.text` into `slugifyLLMResponse`. A short error such as `quota exceeded` stays under the helper thresholds and returns `quota-exceeded`, so session-memory can still write an error-derived filename; reject `payload.isError === true` before slugifying and cover that at the `generateSlugViaLLM` boundary.
  - confidence: 0.9
