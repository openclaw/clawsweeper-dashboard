## Summary

Keep open: the reported bug is valid and the dash-order fix is useful, but the PR still slugifies short error payloads because it drops `payload.isError`, and it still lacks after-fix real OpenClaw proof.

## What This Changes

The branch extracts a `slugifyLLMResponse` helper, adds slug shape checks and post-truncation dash stripping, and adds focused slug-generator unit coverage.

## Best Possible Solution

Land a narrow repair that rejects `payload.isError === true` before slugification, preserves the post-truncation dash strip, adds `generateSlugViaLLM` boundary coverage, and includes redacted after-fix real behavior proof.

## Reproduction Assessment

Yes, at source level. Current main can be reproduced by mocking `runEmbeddedPiAgent` to return an error payload with short text, because `generateSlugViaLLM` reads only `text` and session memory persists any non-null slug.

## Solution Assessment

No. The helper extraction and dash-order fix are useful, but the patch should reject `payload.isError === true` before calling `slugifyLLMResponse` rather than relying only on text-shape heuristics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Reject error payloads before slugifying:** `src/hooks/llm-slug-generator.ts:134`
