## Summary

Keep open. Current main still has the reported slug path, and the PR is useful but not merge-ready because it still accepts short `payload.isError` responses and lacks the required changelog entry for a user-visible filename fix.

## What This Changes

The PR extracts a `slugifyLLMResponse` helper, filters malformed raw LLM slug responses, trims dashes after truncation, and expands slug-generator tests.

## Best Possible Solution

Land a narrow version that rejects `payload.isError === true` before slugification, keeps the helper and post-truncation dash strip, adds boundary coverage at `generateSlugViaLLM`, and records the fix in the changelog.

## Reproduction Assessment

Yes. A mocked embedded run returning `{ payloads: [{ text: "quota exceeded", isError: true }] }` is enough to show the current and proposed text-only slug path can produce `quota-exceeded` instead of falling back.

## Solution Assessment

No. The helper extraction and dash-order fix are good, but the best fix must honor the embedded runner's existing `payload.isError` contract instead of relying only on text-shape heuristics.
