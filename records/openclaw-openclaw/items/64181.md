## Summary

Keep open: this is a plausible focused bug fix, but the PR head still ignores the embedded-runner error-payload flag, so short error texts can still become memory filename slugs; it also lacks required after-fix real behavior proof.

## What This Changes

The branch extracts LLM slug response slugification into a helper, rejects several non-slug response shapes, strips dashes after truncation, and adds colocated slug-generator tests.

## Best Possible Solution

Reject `payload.isError === true` before slugifying, keep the post-truncation dash fix and focused tests, add generator-level regression coverage, then land with redacted after-fix memory-file proof and coordination with https://github.com/openclaw/openclaw/pull/65134.

## Reproduction Assessment

Yes, at source level. The embedded result type and payload builder preserve `isError`, while the PR head reads only `.text`, so a mocked `{ text: "quota exceeded", isError: true }` payload would still produce a non-null slug.

## Solution Assessment

No. The helper extraction and dash-order fix are useful, but the reliable narrow fix must honor the embedded-runner `isError` payload contract before text slugification.
