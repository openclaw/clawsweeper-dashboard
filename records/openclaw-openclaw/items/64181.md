## Summary

Keep open. The reported bug is valid against current main and the PR contains useful source-local work, but the branch is not merge-ready because it still slugifies short payloads marked `isError: true`; it also needs the required user-facing changelog entry before landing.

## What This Changes

The PR extracts a pure `slugifyLLMResponse` helper, adds response-shape checks and post-truncation dash stripping for LLM slug output, and expands colocated slug-generator tests.

## Best Possible Solution

Land a narrow version of this PR that rejects `payload.isError === true` before slugification, keeps the pure helper and post-truncation dash strip, adds boundary coverage for `generateSlugViaLLM`, and records the user-facing fix in `CHANGELOG.md`.

## Reproduction Assessment

Yes. Current main has a high-confidence code-backed path: `generateSlugViaLLM` slugifies raw payload text and session-memory writes any non-null slug into the filename; a short error payload such as `quota exceeded` still reproduces against the PR helper if the runner marks it as `isError`.

## Solution Assessment

No. The helper extraction and dash-order change are good, but the best fix must use the embedded runner's existing `payload.isError` contract instead of relying only on text-shape heuristics.
