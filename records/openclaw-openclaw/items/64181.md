## Summary

Keep open: the PR has useful slug cleanup and trailing-dash coverage, but it still does not reject embedded-runner payloads marked `isError`, so short provider failures can still become memory filename slugs. It also lacks contributor-supplied after-fix real behavior proof.

## What This Changes

The PR extracts slug text normalization into a helper, adds slug-generator tests, rejects several non-slug response shapes, and strips dashes after truncation.

## Best Possible Solution

Amend this PR to reject `payload.isError === true` before text slugification, keep the post-truncation dash fix and focused tests, coordinate with https://github.com/openclaw/openclaw/pull/65134, then land after redacted real memory-file proof.

## Reproduction Assessment

Yes, at source level. A mocked embedded-runner result with `[{ text: "quota exceeded", isError: true }]` still flows through the PR merge result into `slugifyLLMResponse`, and session memory uses any non-null slug in filenames.

## Solution Assessment

No. The helper extraction and dash-order fix are useful, but the reliable fix must consume the embedded runner’s `isError` payload contract before slugifying text.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Reject error payloads before slugifying:** `src/hooks/llm-slug-generator.ts:132-134`
  - body: The embedded runner already marks failed output with `isError`, but this path reads only `text` and passes it to `slugifyLLMResponse`. Short failures like `quota exceeded` satisfy the new shape checks and still become `quota-exceeded`, so reject `payload.isError === true` before slugification and cover that boundary in `generateSlugViaLLM` tests.
