## Summary

Keep this PR open: the reported slug bug is still present on current main and the PR contains useful narrow work, but the current branch still accepts short error payloads because it ignores the embedded runner's `payload.isError` contract.

## What This Changes

The PR extracts `slugifyLLMResponse`, adds response-shape checks and post-truncation dash stripping for LLM session-memory slug output, and expands colocated slug-generator tests.

## Best Possible Solution

Land a narrow repair that rejects `payload.isError === true` before slugification, keeps the post-truncation dash fix and focused tests, and leaves broader slug-runner rewrites to their own PRs.

## Reproduction Assessment

Yes. Source inspection shows current main slugifies the first embedded payload text and session-memory writes any non-null slug; a short error payload marked with `isError: true` remains a high-confidence reproduction path for the PR defect.

## Solution Assessment

No. The helper extraction and dash-order fix are good, but the branch should use the existing `payload.isError` contract before any text-shape heuristic and cover that at the `generateSlugViaLLM` boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Reject error payloads before slugifying:** `src/hooks/llm-slug-generator.ts:134`
  - body: The embedded runner exposes `payload.isError`, but this branch drops that flag and passes only `payload.text` into `slugifyLLMResponse`. A short error such as `quota exceeded` stays under the helper thresholds and returns `quota-exceeded`, so session-memory can still write an error-derived filename. Check `payload.isError === true` before slugifying and add boundary coverage for `generateSlugViaLLM`.
  - confidence: 0.9
