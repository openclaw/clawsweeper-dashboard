## Summary

Keep open. Current main still has the malformed session-memory filename behavior, and this PR is useful, but the branch does not fully fix the reported error-payload path because it ignores the embedded runner's `payload.isError` flag; a short provider failure such as `quota exceeded` can still become a filename slug. The user-facing fix also needs the required changelog entry before merge.

## What This Changes

The PR extracts a `slugifyLLMResponse` helper, routes LLM slug responses through response-shape checks and post-truncation dash stripping, and adds colocated tests for slug normalization, rejection, and trailing-dash behavior.

## Best Possible Solution

Land this PR or an equivalent narrow patch after rejecting `payload.isError === true` before slugification, adding a regression test at the `generateSlugViaLLM` boundary, and recording the user-facing fix in `CHANGELOG.md`. Keep the pure helper and post-truncation dash strip because those address real parts of the reported failure.

## Reproduction Assessment

Yes. The issue has a high-confidence code-backed reproduction: current main slugifies raw payload text into filenames, and the PR still accepts a short `isError: true` payload because it only checks the text shape.

## Solution Assessment

No. The proposed helper and dash-order change are good, but the best fix must use the embedded runner's existing error-payload contract instead of relying only on heuristics over provider text.
