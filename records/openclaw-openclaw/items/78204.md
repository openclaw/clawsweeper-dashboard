## Summary

The reported Feishu group symptom matches the documented group visible-reply policy rather than a proven Feishu send failure: group turns can show typing while normal final assistant text is intentionally not posted unless the model calls the message tool or group visible replies are set to automatic.

## What This Changes

The issue asks OpenClaw to reliably send visible replies for @mentioned Feishu group-chat messages that currently show typing and then no response.

## Best Possible Solution

Keep the existing documented group visible-reply policy, use `messages.groupChat.visibleReplies: "automatic"` when visible final replies should not depend on model tool calls, and require logs/config for any new source-level Feishu failure report.

## Reproduction Assessment

Source-reproducible for the documented behavior, not live-reproduced as a Feishu transport bug: current code and tests show default group turns suppress automatic visible final replies while typing remains enabled. The issue lacks logs proving a Feishu send failure or showing behavior after `messages.groupChat.visibleReplies: "automatic"`.

## Solution Assessment

Yes for cleanup: closing as support/config is the narrowest maintainable outcome because current shipped docs and source already describe this behavior. A code change to make all mentioned group replies visible by default would be a product/security policy decision, not a conservative bug fix.
