## Summary

Keep this issue open. Current main still lacks llama.cpp `/props.n_ctx` discovery in the self-hosted OpenAI-compatible provider path, and the provided discussion shows #74057 is an open implementation candidate with closing syntax for this issue.

## What This Changes

The issue requests that llama.cpp/self-hosted provider session context denominators use the actual loaded runtime context (`n_ctx`) or configured cap instead of the model training context (`n_ctx_train`).

## Best Possible Solution

Land or revise #74057, or an equivalent narrow provider-discovery change, so explicit configured caps remain authoritative while llama.cpp-compatible servers can report their smaller loaded runtime context through `/props.n_ctx` as the effective `contextTokens` cap.

## Reproduction Assessment

Yes. The source-level path is high confidence: current main still queries only `/models` during self-hosted OpenAI-compatible discovery, and no current provider setup code or tests read llama.cpp `/props.n_ctx`.
