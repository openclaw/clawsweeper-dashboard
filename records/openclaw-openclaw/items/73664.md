## Summary

Keep this issue open because current main still does not discover llama.cpp `/props.n_ctx`, and the linked closing PR is open rather than merged.

## What This Changes

Report asks the self-hosted llama.cpp provider/session path to show the loaded runtime context window instead of the model training context window.

## Best Possible Solution

Land #74057 or an equivalent narrow provider-discovery fix that stores llama.cpp runtime `n_ctx` as effective `contextTokens` while preserving native `contextWindow` metadata.

## Reproduction Assessment

Yes. A llama.cpp OpenAI-compatible server whose `/v1/models` reports `meta.n_ctx_train` and whose `/props` reports a smaller `n_ctx` should reproduce the wrong denominator on current main because the provider discovery path never reads `/props`.

## Solution Assessment

Yes. Capturing the runtime cap during self-hosted provider discovery is the narrow maintainable direction; sessions should consume stored model/session metadata rather than performing provider-specific HTTP probing during display.
