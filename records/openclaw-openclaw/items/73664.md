## Summary

Keep this issue open: current main still discovers self-hosted OpenAI-compatible models from `/models` only, does not read llama.cpp `/props.n_ctx`, and the closing PR #74057 is open and unmerged.

## What This Changes

Use the llama.cpp runtime context window as the effective session denominator instead of the larger training context reported by OpenAI-compatible model metadata.

## Best Possible Solution

Review and land #74057, or an equivalent narrow discovery fix that records llama.cpp `/props.n_ctx` as effective `contextTokens` while preserving native `contextWindow` metadata.

## Reproduction Assessment

Yes. A llama.cpp OpenAI-compatible server whose `/v1/models` reports a larger training window while `/props` reports a smaller loaded `n_ctx` follows the current source path because main only reads `/models` during self-hosted discovery.

## Solution Assessment

Yes. Capturing the runtime cap during provider discovery is the narrow maintainable fix; the sessions table should keep consuming stored session/model metadata rather than doing live provider-specific HTTP calls.
