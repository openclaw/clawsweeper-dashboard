## Summary

Current main now solves the reported user path by resolving the effective runtime context from configured model metadata before falling back to discovered model metadata, so a configured `contextWindow: 65536` wins over a larger live `n_ctx_train` value.

## What This Changes

The issue requests that llama.cpp self-hosted sessions use the actual runtime context cap, preferably the configured `models.json` value or `/props.n_ctx`, instead of the advertised training context.

## Best Possible Solution

Keep the current generic config-first runtime context resolution, and handle any additional llama.cpp `/props` auto-discovery as separate PR review work rather than keeping this report open.

## Reproduction Assessment

Yes. The old report is reproducible with a llama.cpp OpenAI-compatible server whose live metadata advertises `n_ctx_train: 262144` while `models.json` config sets `contextWindow: 65536`; current main no longer follows the discovered value for that configured model path.

## Solution Assessment

Yes. The implemented direction matches the simpler suggested fix by trusting configured model context caps before provider-discovered metadata, avoiding llama.cpp-specific probing in the sessions renderer.
