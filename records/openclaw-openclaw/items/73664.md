## Summary

Keep this issue open: current main still does not discover llama.cpp `/props.n_ctx` for the self-hosted OpenAI-compatible provider path, and #74057 is an open, unmerged closing PR rather than a shipped fix.

## What This Changes

Report that llama.cpp/self-hosted OpenAI-compatible sessions should show the runtime context cap (`n_ctx`) instead of the model training context (`n_ctx_train`).

## Best Possible Solution

Land #74057 or an equivalent narrow discovery fix that stores llama.cpp `/props.n_ctx` as effective `contextTokens` while preserving `contextWindow` metadata.

## Reproduction Assessment

Yes. A llama.cpp OpenAI-compatible server whose `/v1/models` exposes only the larger training context while `/props` exposes a smaller loaded `n_ctx` should reproduce the misleading denominator because current main never reads `/props` in this path.

## Solution Assessment

Yes. Capturing the runtime cap during provider discovery/storage is the narrowest maintainable direction; the sessions renderer should consume stored model/session metadata rather than adding provider-specific HTTP probing.
