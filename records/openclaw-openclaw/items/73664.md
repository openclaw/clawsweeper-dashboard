## Summary

Keep this issue open: current main still does not read llama.cpp `/props.n_ctx`, and existing generated `models.json` model limits are not preserved when regenerated; the open PR #74057 is the active implementation candidate and should land or be replaced before closing.

## What This Changes

The issue asks for llama.cpp self-hosted OpenAI-compatible sessions to use the server's loaded runtime context window instead of the model training context window for token utilization.

## Best Possible Solution

Land #74057 or an equivalent narrow discovery fix that records llama.cpp `/props.n_ctx` as the effective `contextTokens` while keeping `contextWindow` as native metadata.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a llama.cpp OpenAI-compatible server whose `/v1/models` exposes the training window while `/props.n_ctx` is the smaller loaded runtime cap will not be corrected because current discovery never calls `/props`.

## Solution Assessment

Yes. Capturing the runtime cap during self-hosted provider discovery as `contextTokens` is the narrow maintainable fix; the sessions table should keep consuming stored session/model metadata rather than doing live provider-specific probes.
