## Summary

Keep this issue open because current main still lacks llama.cpp `/props.n_ctx` runtime-context discovery in the self-hosted OpenAI-compatible provider path, and #74057 is an open closing PR rather than a merged fix.

## What This Changes

Report that llama.cpp-backed OpenAI-compatible models show the training context window in `openclaw sessions` instead of the loaded runtime context window.

## Best Possible Solution

Review and land #74057, or an equivalent narrow discovery fix, so llama.cpp `/props.n_ctx` is stored as effective `contextTokens` while native `contextWindow` metadata remains intact.

## Reproduction Assessment

Yes. A llama.cpp OpenAI-compatible server whose `/v1/models` exposes the larger training context while `/props` exposes a smaller loaded `n_ctx` should reproduce the wrong denominator because current main never reads `/props` for that provider path.

## Solution Assessment

Yes. Capturing the runtime cap during self-hosted provider discovery is the narrowest maintainable direction; the sessions table should consume stored model/session metadata rather than doing provider-specific HTTP probing.
