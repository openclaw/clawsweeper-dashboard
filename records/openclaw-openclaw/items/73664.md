## Summary

Keep this issue open: current main still has the reported self-hosted OpenAI-compatible discovery gap, and the linked fix PR is open rather than merged.

## What This Changes

The issue asks OpenClaw to show llama.cpp's loaded runtime context window, not the larger training context, in session token usage.

## Best Possible Solution

Land #74057 or an equivalent narrow discovery fix that records llama.cpp `/props.n_ctx` as effective `contextTokens` while preserving native `contextWindow` metadata.

## Reproduction Assessment

Yes. A llama.cpp OpenAI-compatible server whose `/v1/models` reports a larger training context while `/props` reports a smaller loaded `n_ctx` should reproduce the wrong denominator because current main never reads `/props` for this path.

## Solution Assessment

Yes. Capturing the runtime cap during provider discovery is the narrowest maintainable direction; the sessions table should continue consuming stored session/model metadata rather than doing live provider-specific HTTP calls.
