## Summary

Keep this issue open: current main still does not read llama.cpp `/props.n_ctx` during self-hosted OpenAI-compatible discovery, and PR #74057 is the open, unmerged implementation candidate that uses closing syntax for this issue.

## What This Changes

The issue asks OpenClaw to use llama.cpp's runtime context cap as the session denominator instead of the model's trained context metadata.

## Best Possible Solution

Land PR #74057 or an equivalent narrow discovery fix that records llama.cpp `/props.n_ctx` as the effective runtime `contextTokens` while preserving configured caps.

## Reproduction Assessment

Yes, as a source-level path. A llama.cpp OpenAI-compatible server whose `/v1/models` advertises a larger trained context while `/props` reports a smaller loaded `n_ctx` still goes through current discovery code that never reads `/props`.

## Solution Assessment

Yes. Capturing the runtime cap during provider discovery/config resolution is the narrow maintainable fix; the sessions table should continue consuming stored/resolved model metadata rather than doing live provider-specific HTTP calls.
