## Summary

Keep this issue open because current main still has no llama.cpp `/props.n_ctx` discovery path, and #74057 is an open, unmerged closing PR for the same behavior.

## What This Changes

The report asks the llamacpp/OpenAI-compatible provider path to show the loaded runtime context window in Sessions instead of training context metadata.

## Best Possible Solution

Record llama.cpp's loaded runtime `n_ctx` as `contextTokens` during provider discovery while keeping `contextWindow` as native/training metadata and preserving explicit configured caps.

## Reproduction Assessment

Yes, by source inspection: current main never reads `/props.n_ctx`, and the sessions table uses stored/configured context tokens for its denominator. I did not run a live llama.cpp server in this read-only review.

## Solution Assessment

Yes. Capturing the runtime cap during self-hosted provider discovery is the narrow maintainable fix; probing llama.cpp from the sessions display path would spread provider-specific behavior into a command surface.
