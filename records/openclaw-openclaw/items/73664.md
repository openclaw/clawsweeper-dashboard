## Summary

Keep this issue open because current main still lacks llama.cpp runtime `n_ctx` discovery for the affected self-hosted provider path, and #74057 is an open, unmerged closing PR for the report.

## What This Changes

The issue asks for llama.cpp self-hosted provider session context usage to use the loaded runtime context window rather than the model training context ceiling.

## Best Possible Solution

Review and land #74057, or an equivalent narrow provider-discovery fix that stores llama.cpp `/props.n_ctx` as `contextTokens` while preserving native `contextWindow` metadata.

## Reproduction Assessment

Yes, by source inspection: current main still discovers self-hosted OpenAI-compatible models from `/models` only, and the sessions table denominator comes from the resolved `contextTokens` value. I did not run a live llama.cpp server in this read-only review.

## Solution Assessment

Yes. Capturing the loaded runtime cap during self-hosted provider discovery is the narrow maintainable direction; adding llama.cpp HTTP probing to the sessions display path would spread provider-specific behavior into a command surface.
