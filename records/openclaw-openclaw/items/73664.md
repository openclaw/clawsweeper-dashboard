## Summary

Keep this issue open. Current main still has self-hosted OpenAI-compatible discovery reading only the `/models` endpoint, while the linked implementation candidate #74057 remains open and unmerged for this issue.

## What This Changes

The issue requests that llama.cpp self-hosted provider session usage use the loaded runtime context window instead of the model training context reported by OpenAI-compatible metadata.

## Best Possible Solution

Revise and land #74057, or an equivalent narrow self-hosted provider discovery change, so llama.cpp-compatible servers store the loaded `/props` runtime context as `contextTokens` while retaining `n_ctx_train` or configured values as native model metadata.

## Reproduction Assessment

Yes. A llama.cpp OpenAI-compatible server whose `/v1/models` reports training metadata but whose `/props` runtime context differs should reproduce the wrong denominator because current main never queries `/props` and session listing consumes stored/configured/model context tokens only.

## Solution Assessment

Yes, the best boundary is self-hosted provider discovery rather than a sessions-table special case, so all context-token consumers get the same effective runtime cap. The linked PR may still need normal review or revision before it is the mergeable implementation.
