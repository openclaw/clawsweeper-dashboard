## Summary

Keep this issue open. Current main still has the reported product gap: self-hosted OpenAI-compatible discovery does not read llama.cpp `/props.n_ctx`, while session display uses stored/configured/model-catalog context tokens for its denominator. The already-linked #74057 is an open implementation candidate with closing syntax, so this should wait for that PR or an equivalent fix to land.

## What This Changes

The issue requests that llama.cpp self-hosted provider sessions use the loaded runtime context window (`n_ctx`) instead of the training context ceiling (`n_ctx_train`) for usage percentages.

## Best Possible Solution

Land or revise #74057, or an equivalent narrow provider-discovery change, so configured caps stay authoritative while llama.cpp-compatible servers can report their smaller loaded runtime context through `/props.n_ctx` as the effective session context cap.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: current main has no `/props.n_ctx` lookup in the self-hosted OpenAI-compatible discovery path, and the session denominator is derived from stored/configured/catalog context tokens rather than a llama.cpp runtime-context probe.

## Solution Assessment

Yes. Keeping this issue open while maintainers review #74057 is the best path; a duplicate ClawSweeper repair branch would add churn before the existing closing PR is resolved.
