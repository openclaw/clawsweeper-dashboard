## Summary

Keep this issue open because current main still lacks llama.cpp runtime `n_ctx` discovery for the affected self-hosted OpenAI-compatible path, and #74057 is an open, unmerged closing PR for the same report.

## What This Changes

The report asks the llamacpp provider/session display to use the loaded runtime context window (`n_ctx`, e.g. 65k) instead of the model training context (`n_ctx_train`, e.g. 262k).

## Best Possible Solution

Review and land #74057, or an equivalent narrow provider-discovery fix that stores llama.cpp `/props.n_ctx` as `contextTokens` while preserving native `contextWindow` metadata.

## Reproduction Assessment

Yes, by source inspection: current main’s self-hosted OpenAI-compatible discovery only requests `/models`, never `/props`, and the sessions display formats its denominator from stored/configured context tokens. I did not run a live llama.cpp server in this read-only review.

## Solution Assessment

Yes. Capturing the runtime cap during self-hosted provider discovery is the narrow maintainable direction; putting llama.cpp HTTP probing in the sessions display path would spread provider-specific behavior into a command surface.
