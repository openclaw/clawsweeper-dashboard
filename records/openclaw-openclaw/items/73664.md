## Summary

Keep this issue open: current main still discovers self-hosted OpenAI-compatible models from `/models` only, does not read llama.cpp `/props.n_ctx`, and PR #74057 is the open, unmerged closing implementation candidate.

## What This Changes

The issue asks for llama.cpp self-hosted model context display to use the runtime `n_ctx` or effective configured cap rather than `n_ctx_train`.

## Best Possible Solution

Land PR #74057 or an equivalent narrow discovery fix that records llama.cpp `/props.n_ctx` as `contextTokens` while preserving native `contextWindow` metadata.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a llama.cpp server whose `/v1/models` exposes only the larger training window and whose `/props` exposes a smaller `n_ctx` will still be discovered from `/models` only on current main.

## Solution Assessment

Yes. Capturing the runtime cap during self-hosted provider discovery is the narrow maintainable solution; the sessions table should keep consuming stored/model metadata instead of making display-time provider-specific HTTP calls.
