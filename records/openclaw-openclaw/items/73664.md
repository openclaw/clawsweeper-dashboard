## Summary

Current main implements the simpler requested fix path: configured per-model context caps are now preferred over runtime/discovered model metadata before session `contextTokens` are recorded, so the reported `models.json` `contextWindow: 65536` case no longer gets overridden by a larger llama.cpp training window.

## What This Changes

The issue asks for llama.cpp-backed self-hosted OpenAI-compatible sessions to show the actual runtime context cap instead of the model training context window.

## Best Possible Solution

Keep the current config-first runtime context behavior; treat #74057 as a possible separate `/props` discovery enhancement or close it as superseded if maintainers do not want that extra provider-specific probe.

## Reproduction Assessment

No, not against current main: the source path now selects the configured model `contextTokens`/`contextWindow` before runtime model metadata, so the reporter's configured 65536 cap would feed the session context budget. I did not run a live llama.cpp server in this read-only review.

## Solution Assessment

Yes. Preferring explicit configured model context caps is the narrowest maintainable fix for the reported case and avoids adding display-time provider-specific HTTP probes; `/props` discovery can remain an enhancement if desired.
