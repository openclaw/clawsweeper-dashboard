## Summary

Keep open: the PR targets a plausible Gateway startup concurrency problem, but the latest patch still has a blocking fake-timer test regression and the real-behavior proof is not strong enough for the claimed polling-stall fix.

## What This Changes

The PR serializes Gateway channel/account startup with a 3-second handoff stagger, moves primary model prewarm after channel startup, and updates fanout tests plus the changelog.

## Best Possible Solution

Make the stagger test-safe, attach isolated Windows multi-Telegram proof for this patch’s intended behavior, then have Gateway owners decide whether the mitigation should be global or narrower.

## Reproduction Assessment

Unclear for the full Windows polling-stall symptom; current main source does show provider startup is handed off without awaiting provider lifetime, so concurrent startup is source-reproducible but I did not live-reproduce the Windows multi-account stall.

## Solution Assessment

No, not yet. The approach is plausible, but it still needs timer-safe tests, stronger isolated behavior proof, and maintainer acceptance of the global 3-second startup policy.
