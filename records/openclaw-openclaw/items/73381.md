## Summary

Keep open. Current main still enforces `sendPolicy=deny` in the Gateway `agent` handler before checking whether the request asked for outbound delivery, and open PR #73500 is already the narrow implementation candidate for this issue.

## What This Changes

A non-delivery Gateway agent invocation should not be rejected solely because the session outbound `sendPolicy` resolves to `deny`.

## Best Possible Solution

Land PR #73500 or an equivalent patch that enforces `sendPolicy=deny` only for `deliver: true`, preserves explicit delivery blocking, and adds deny/non-delivery regression coverage.

## Reproduction Assessment

Yes. A Gateway `agent` request with effective `sendPolicy=deny` and `deliver` omitted or false follows the current source path that rejects at `src/gateway/server-methods/agent.ts:1042` before delivery intent is evaluated.

## Solution Assessment

Yes. Gating the Gateway deny check on `request.deliver === true` is the narrow maintainable fix because it matches the embedded agent path and keeps outbound-delivery protection intact.
