## Summary

Keep open. Current main still enforces `session.sendPolicy=deny` in the Gateway `agent` handler before the handler computes whether the request is asking for external delivery, so the reported non-delivery quick-agent path remains a valid bug. The issue is also paired with open, unmerged PR #73500 using closing syntax, so the right cleanup path is to review and land that PR or an equivalent focused fix rather than closing this issue now.

## What This Changes

Gateway `agent` requests without external delivery should not be blocked by `session.sendPolicy=deny`, while explicit outbound delivery should remain blocked.

## Best Possible Solution

Land PR #73500 or an equivalent focused change that gates the Gateway `agent` send-policy rejection on `request.deliver === true`, with regression coverage for both non-delivery allow under deny and explicit-delivery denial.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: call the Gateway `agent` method with an effective session `sendPolicy` of `deny` and `deliver` omitted or false; current main returns `INVALID_REQUEST` before dispatching the agent.

## Solution Assessment

Yes. The narrowest maintainable fix is to align the Gateway handler with the embedded agent path by enforcing send policy only for `request.deliver === true`, while preserving the denial path for actual outbound delivery.
