## Summary

Keep open. Current main still rejects Gateway `agent` requests when `resolveSendPolicy` returns `deny` before the handler computes whether the request wants external delivery, and ordinary CLI agent calls send `deliver: false` when `--deliver` is omitted. The issue is also paired with open, unmerged PR #73500 using closing syntax, so the right cleanup path is PR review or an equivalent fix rather than closing this issue now.

## What This Changes

The issue requests that non-delivery Gateway agent invocations bypass outbound `session.sendPolicy=deny` while explicit external delivery remains blocked.

## Best Possible Solution

Land PR #73500 or an equivalent focused change that gates the Gateway `agent` send-policy rejection on `request.deliver === true`, with regression coverage for both non-delivery allow under deny and explicit-delivery denial.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: invoke the Gateway `agent` method with an effective session `sendPolicy` of `deny` and `deliver` omitted or false; current main returns `INVALID_REQUEST` before dispatching the agent.

## Solution Assessment

Yes. The linked fix direction is the narrowest maintainable solution because it mirrors the embedded agent path while keeping actual outbound delivery blocked when `deliver: true`.
