## Summary

Keep open. Current main still enforces the Gateway `agent` send-policy denial before computing whether the request wants external delivery, so a non-delivery quick-agent call can still be rejected. The issue is also paired with open, unmerged PR #73500 using closing syntax, so the right path is to review and land that PR or an equivalent focused fix.

## What This Changes

The issue requests that Gateway `agent` invocations without external delivery bypass `session.sendPolicy=deny` while preserving denial for explicit outbound delivery.

## Best Possible Solution

Land PR #73500 or an equivalent focused change that gates the Gateway `agent` send-policy rejection on explicit delivery, with regression coverage for both non-delivery allow-under-deny and explicit-delivery rejection.

## Reproduction Assessment

Yes. Invoke the Gateway `agent` method with an effective session `sendPolicy` of `deny` and `deliver` omitted or false; current source returns `INVALID_REQUEST` before dispatching the agent or computing delivery intent.

## Solution Assessment

Yes. Gating the Gateway send-policy check on `request.deliver === true` is the narrowest maintainable fix because it mirrors the embedded agent path and keeps actual outbound delivery blocked.
