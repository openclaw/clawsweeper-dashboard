## Summary

This PR should stay open, but it is not merge-ready. The schema-hardening direction is useful, yet the branch adds several maxima that are stricter than current runtime or docs contracts and it has no after-fix real behavior proof beyond build/test claims.

## What This Changes

The branch adds TypeBox descriptions plus `maxItems` and numeric maximum bounds across agent tool schemas for browser, gateway, canvas, memory, message, node, session, subagent, and web-fetch tools.

## Best Possible Solution

Keep the schema-hardening direction, but derive each bound from the current executor/API contract or change the runtime and docs together where a tighter limit is intentionally desired.

## Reproduction Assessment

Yes for the blocking regressions by source inspection: current docs and executors allow requests that this PR's new TypeBox maxima would reject at tool-schema validation. I did not run live gateway/model calls in this read-only review, and the PR body also says that path was not verified.

## Solution Assessment

No. Schema hardening is a reasonable direction, but this branch is not the best current form because several proposed maxima are tighter than existing runtime and documented contracts; the safer fix is to align schemas, runtime clamps, docs, and focused tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Allow node timeout to cover media recordings:** `src/agents/tools/nodes-tool.ts:111-115`
  - body: `durationMs` remains valid up to 300000 ms for node media actions, but `timeoutMs` is now capped at 120000 ms. That prevents callers from setting an RPC timeout long enough for valid long `screen_record` or `camera_clip` requests.
  - confidence: 0.93
- **[P2] Keep browser snapshot limit at the backend cap:** `src/agents/tools/browser-tool.schema.ts:99`
  - body: The snapshot action still forwards `limit` to backends that accept up to 2000 nodes, but the schema now caps `limit` at 1000. Large-page snapshot requests between 1001 and 2000 nodes will fail validation even though the implementation can serve them.
  - confidence: 0.9
- **[P2] Preserve 200-row sessions list callers:** `src/agents/tools/sessions-list-tool.ts:36-41`
  - body: The schema now caps `sessions_list.limit` at 50, while internal session-resolution code still requests 200 rows and the gateway protocol accepts any positive integer. Larger installs can no longer enumerate enough sessions to resolve older or less-active targets.
  - confidence: 0.92
- **[P2] Do not cap long-running spawned sessions at one hour:** `src/agents/tools/sessions-spawn-tool.ts:35-48`
  - body: `runTimeoutSeconds` and its compatibility alias are currently documented and implemented as non-negative values, with `0` meaning no timeout. Capping them at 3600 rejects legitimate long-running subagent jobs that the runtime still supports.
  - confidence: 0.9
- **[P2] Permit Discord media channel types:** `src/agents/tools/message-tool.ts:328-330`
  - body: Capping channel-management `type` at 15 excludes `ChannelType.GuildMedia`, which current Discord send/thread code already treats as a supported forum-like channel. This blocks legitimate Discord channel create/edit tool calls for media channels.
  - confidence: 0.86
