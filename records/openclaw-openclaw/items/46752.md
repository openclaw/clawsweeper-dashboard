## Summary

Keep this PR open, but it is not merge-ready: the hardening direction is useful, yet several proposed schema bounds reject calls that current runtime code still accepts, and the external PR lacks after-fix real behavior proof.

## What This Changes

The PR adds TypeBox descriptions, numeric maximums, and array maxItems constraints across agent tool schemas for browser, message, nodes, sessions, subagents, gateway, canvas, memory, and web fetch tools.

## Best Possible Solution

Rebase on current main, move stale browser schema work to the browser plugin path, keep only bounds that mirror current runtime/backend/provider contracts, and add real tool-schema or gateway behavior proof.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: the PR diff adds schema maxima that conflict with current-main callers and runtimes such as sessions_list limit 200, sessions_spawn timeouts beyond one hour, node 300000 ms recordings, Discord GuildMedia, and browser 2000-node snapshots. I did not run live validation because this review was read-only.

## Solution Assessment

No. Schema hardening is a reasonable direction, but this patch applies some caps below existing contracts and needs to be rebased so the schema follows current runtime facts instead of inventing new limits.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Let node media timeouts cover valid recordings:** `src/agents/tools/nodes-tool.ts:114`
  - body: The schema caps nodes `timeoutMs` at 120000 ms while the same tool still allows media actions with `durationMs` up to 300000 ms. A valid 5-minute `screen_record` or `camera_clip` call can no longer set a long enough gateway timeout, so it will fail schema validation or time out before the supported action completes.
  - confidence: 0.94
- **[P2] Keep sessions_list at the 200-row contract:** `src/agents/tools/sessions-list-tool.ts:38`
  - body: This maximum rejects `sessions_list({ limit: 200 })`, but current code still uses a 200-row `sessions.list` lookup for announce-target recovery. Larger installs need that range to find older sessions, so lowering the schema cap to 50 breaks an existing contract.
  - confidence: 0.92
- **[P2] Keep spawned runs unbounded when requested:** `src/agents/tools/sessions-spawn-tool.ts:45`
  - body: Adding `maximum: 3600` to `runTimeoutSeconds` changes the sessions_spawn contract: current runtime treats any non-negative value as valid and uses 0 for no timeout. Long-running jobs that intentionally run past one hour would now be rejected by the tool schema before reaching the existing runtime path.
  - confidence: 0.9
- **[P2] Allow Discord media channel types:** `src/agents/tools/message-tool.ts:329`
  - body: Capping channel management `type` at 15 blocks Discord `GuildMedia`, which the Discord plugin already handles as a supported forum-like channel type. This turns valid channel-create requests for newer Discord channel types into schema failures.
  - confidence: 0.88
- **[P2] Keep browser snapshot limit at backend cap:** `src/agents/tools/browser-tool.schema.ts:99`
  - body: The schema lowers browser snapshot `limit` to 1000, but the browser snapshot backends still accept up to 2000 nodes. Agents inspecting large pages would lose access to the supported full snapshot range.
  - confidence: 0.86
