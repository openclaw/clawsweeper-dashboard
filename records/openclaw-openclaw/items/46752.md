## Summary

Keep open. The schema-hardening direction is useful, but current main confirms several proposed bounds are below supported runtime contracts, and the browser change targets a stale core path instead of the active browser plugin schema.

## What This Changes

The PR adds TypeBox maxItems, numeric bounds, and descriptions across built-in agent tool schemas for browser, canvas, gateway, memory, message, nodes, session, subagent, and web_fetch tools.

## Best Possible Solution

Land a rebased schema-hardening patch that encodes only contract-backed limits, updates the active browser plugin schema, preserves existing long-running/session/media workflows, and adds focused schema contract tests.

## Reproduction Assessment

Yes. Static review against current main reproduces the regressions by comparing the PR's schema maxima with current runtime code, dependency types, docs, and tests for node recordings, sessions, browser snapshots, subagent timeouts, and Discord media channels.

## Solution Assessment

No. Schema-only hardening is a maintainable direction, but this branch is not the best implementation because several limits are too tight and one browser change targets a stale path instead of the active plugin-owned schema.

  - body: The runtime and docs allow finite non-negative runTimeoutSeconds, with 0 meaning no timeout. Adding maximum: 3600 rejects legitimate subagent or ACP jobs intended to run past an hour.
  - confidence: 0.91

  - body: Current main registers the browser tool from extensions/browser, so editing the old core browser schema leaves the active surface unchanged. The proposed 1000 snapshot cap is also below the backend cap of 2000.
  - confidence: 0.84

  - body: The type maximum blocks Discord GuildMedia, which the current Discord runtime treats as forum-like and tests cover for thread creation. Valid media-channel thread behavior would fail schema validation.
