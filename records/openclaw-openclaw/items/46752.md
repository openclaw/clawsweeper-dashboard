## Summary

Keep this PR open, but it is not merge-ready: the hardening direction is useful, yet several proposed schema caps reject values current main still accepts under existing runtime or upstream contracts, and the external PR still lacks real behavior proof.

## What This Changes

The PR adds TypeBox maxItems, minimum/maximum bounds, and descriptions across browser, gateway, message, node, session, memory, canvas, subagent, and web fetch tool schemas.

## Best Possible Solution

Keep schema hardening, but rebase it onto current main and only add bounds that match existing runtime clamps or verified upstream limits, with regression coverage for the edge values that must remain accepted.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: applying the PR diff would reject current-main-valid values such as five-minute node recordings with matching timeouts, browser snapshot limit 2000, sessions_list limit 200, sessions_spawn timeouts above one hour, and Discord GuildMedia channel type 16.

## Solution Assessment

No. Schema hardening is the right direction, but this implementation is not the narrow maintainable fix until each cap is aligned with the runtime or upstream contract and proven through the real tool-schema path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Allow node timeout to cover recording duration:** `src/agents/tools/nodes-tool.ts:114`
  - body: The schema accepts recording durations up to 300000 ms via durationMs but caps timeoutMs at 120000 ms, so a valid five-minute camera_clip or screen_record request cannot set a long enough gateway timeout and can time out before completion.
  - confidence: 0.93
- **[P2] Permit Discord media channel type:** `src/agents/tools/message-tool.ts:329`
  - body: Capping type at 15 rejects Discord GuildMedia channels, whose enum value follows GuildForum in discord-api-types and which the Discord runtime already treats as forum-like, so legitimate channel-create calls can fail schema validation.
  - confidence: 0.91
- **[P2] Keep browser snapshot limit at the backend cap:** `src/agents/tools/browser-tool.schema.ts:99`
  - body: The browser snapshot path forwards limit to backends that clamp ARIA snapshots at 2000 nodes, but this lowers the tool schema maximum to 1000, rejecting valid large-page snapshot requests before the backend can serve them.
  - confidence: 0.9
- **[P2] Keep sessions_list compatible with 200-row callers:** `src/agents/tools/sessions-list-tool.ts:38`
  - body: The new maximum of 50 rejects existing sessions_list({ limit: 200 }) use, including internal announce-target lookup, so larger installs may fail to find older or less-active sessions that current main still allows.
  - confidence: 0.92
- **[P2] Do not cap sessions_spawn runs at one hour:** `src/agents/tools/sessions-spawn-tool.ts:45`
  - body: Current runtime accepts any non-negative runTimeoutSeconds or timeoutSeconds and passes the resolved value through, with 0 preserving no-timeout behavior; adding maximum: 3600 turns legitimate long-running spawned sessions into validation failures.
  - confidence: 0.9
