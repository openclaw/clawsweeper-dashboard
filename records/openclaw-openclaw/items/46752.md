## Summary

This PR should stay open at https://github.com/openclaw/openclaw/pull/46752. The schema-hardening direction is useful, but the branch is conflicted against current main and several proposed caps would reject values that current runtime, protocol, docs, or dependency contracts still allow.

## What This Changes

The PR adds TypeBox `maxItems`, numeric `maximum`/`minimum`, and parameter descriptions across agent, browser, message, node, session, and web tool schemas.

## Best Possible Solution

Rebase or replace the stale branch and land only schema bounds that match current runtime, protocol, dependency, and docs limits, with regression coverage for the tightened contracts.

## Reproduction Assessment

Yes. Source inspection of the PR diff against current main gives a high-confidence path: schema validation would reject valid values such as node recordings needing 300000 ms, browser snapshots up to 2000 nodes, sessions_list calls with limit 200, long-running spawned sessions, and Discord GuildMedia channel type 16.

## Solution Assessment

No, not as-is. Schema hardening is the right direction, but the current PR must be rebased and its caps derived from active runtime and dependency contracts instead of narrowing them accidentally.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Allow node timeouts to cover max recordings:** `src/agents/tools/nodes-tool.ts:114`
  - body: This cap makes valid long node media actions fail before completion: the same schema allows recording `durationMs` up to 300000 ms, but `timeoutMs` is capped at 120000 ms, so callers cannot set a sufficient RPC timeout for five-minute `camera_clip` or `screen_record` requests.
  - confidence: 0.91
- **[P2] Restore browser snapshot limit to the backend cap:** `src/agents/tools/browser-tool.schema.ts:98`
  - body: The browser snapshot implementations still clamp to 2000 nodes, but the schema now rejects `limit` values above 1000. Large-page snapshot requests that the backend can serve would fail validation before reaching the browser tool.
  - confidence: 0.9
- **[P2] Keep sessions_list limit aligned with 200-row callers:** `src/agents/tools/sessions-list-tool.ts:34`
  - body: The new maximum of 50 rejects existing `sessions_list` calls with `limit: 200`, including current internal session-resolution code. Larger installs can no longer enumerate enough sessions to find older targets.
  - confidence: 0.89
- **[P2] Do not cap spawned sessions at one hour:** `src/agents/tools/sessions-spawn-tool.ts:41`
  - body: The runtime and docs accept any non-negative `runTimeoutSeconds`, with `0` meaning no timeout. Adding `maximum: 3600` turns legitimate long-running spawned sessions into schema-validation failures.
  - confidence: 0.88
- **[P2] Permit Discord media channel types:** `src/agents/tools/message-tool.ts:329`
  - body: Discord runtime code already handles `GuildMedia` as a supported forum-like channel, but this maximum excludes that enum value. Channel-management calls for supported media channels would be rejected at schema validation.
  - confidence: 0.86
