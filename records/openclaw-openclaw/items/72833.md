## Summary

Keep open. Current main still exposes fal media generation through curated static model arrays, while the requested full FAL catalog discovery is valid but broad enough to need maintainer product/API scoping.

## What This Changes

Request dynamic discovery and exposure of FAL.ai's full image/video model catalog instead of the current curated fal provider model lists.

## Best Possible Solution

Define a fal-owned, schema-aware catalog strategy for supported media modes, then add a backwards-compatible media-catalog SDK seam only if static provider metadata cannot express it.

## Reproduction Assessment

Not applicable as a bug reproduction. There is a high-confidence current-state check: inspect the fal provider registrations and `image_generate`/`video_generate` list paths, which are still backed by static `provider.models` arrays on current main.

## Solution Assessment

Unclear. Exposing every FAL endpoint as a built-in selectable media model is not yet proven to be the narrowest maintainable solution; a filtered, schema-aware catalog with explicit capability mapping is safer.
