## Summary

Keep open. Current main still exposes fal image and video generation through static provider-owned model arrays, while this asks for dynamic full FAL.ai catalog discovery across many mixed media modalities; that is real provider/API/product work, not conservative cleanup.

## What This Changes

Request dynamic discovery and exposure of the full FAL.ai media model catalog in the fal image/video generation provider surfaces instead of the current curated model lists.

## Best Possible Solution

Define a scoped, provider-owned FAL media catalog strategy first: filter by OpenClaw-supported media capability, map per-model schemas and limits inside the fal plugin, and add an additive generic discovered-media-catalog SDK seam only if the existing image/video contracts are insufficient.

## Reproduction Assessment

Not applicable as a bug reproduction. There is a high-confidence current-state check: inspect current main or call the image_generate/video_generate list surfaces, which are bounded by the static models arrays registered by the fal providers.

## Solution Assessment

Unclear. The feature direction is valid, but exposing all FAL endpoints as selectable built-in media models is not yet proven to be the narrowest maintainable solution; a filtered, capability-mapped catalog with provider-owned schema handling is safer.
