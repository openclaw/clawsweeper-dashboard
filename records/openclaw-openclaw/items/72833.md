## Summary

Keep open. Current main still exposes FAL image and video generation through static or curated provider model arrays, and the runtime list surfaces serialize those registered arrays rather than discovering the FAL.ai catalog dynamically. The request remains valid provider/API/product work, not conservative cleanup.

## What This Changes

The issue requests dynamic discovery and exposure of the full FAL.ai model catalog as selectable OpenClaw media-generation models/providers.

## Best Possible Solution

Define a scoped, provider-owned FAL media catalog strategy first: filter by supported image/video capabilities, map per-model request schemas and limits inside the FAL plugin, cache/fail gracefully when discovery is unavailable, and add an additive generic discovered-media-catalog SDK seam only if the existing image/video provider contracts are insufficient.

## Reproduction Assessment

Not applicable as a bug reproduction. There is a high-confidence current-state check: inspect the FAL provider registrations and the `image_generate`/`video_generate` list surfaces on current main, which show static or curated registered model arrays rather than dynamic FAL catalog discovery.

## Solution Assessment

Unclear. The requested direction is valid, but exposing every FAL endpoint directly as a built-in selectable model is not yet proven to be the narrowest maintainable solution; a filtered, capability-mapped catalog with provider-owned schema handling is safer.
