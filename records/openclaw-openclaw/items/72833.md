## Summary

Keep open. Current main still exposes fal image/video generation through curated provider-owned model arrays, while the request asks for dynamic full FAL.ai catalog discovery across many modalities and model-specific schemas; that is real provider/API/product work, not conservative cleanup.

## What This Changes

Request dynamic discovery and selectable exposure of the full FAL.ai model catalog for the bundled fal image/video generation provider.

## Best Possible Solution

Define a scoped fal-owned media catalog strategy: discover or generate a filtered catalog for OpenClaw-supported image/video modes, map per-model schemas and limits inside the fal plugin, and add an additive media-catalog SDK seam only if the current static models contract is insufficient.

## Reproduction Assessment

Not applicable as a bug reproduction. There is a high-confidence current-state check: inspect the fal provider registrations or the image_generate/video_generate list paths, which are bounded by static provider.models arrays on current main.

## Solution Assessment

Unclear. The feature direction is valid, but exposing all FAL endpoints as built-in selectable media models is not yet proven to be the narrowest maintainable solution; a filtered, capability-mapped catalog with provider-owned schema handling is safer.
