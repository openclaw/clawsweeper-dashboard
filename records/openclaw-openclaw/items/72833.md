## Summary

Keep open. Current main still exposes fal media generation through curated static model arrays and static media-provider contracts, while fal’s own docs describe a 1,000+ model surface plus a model-search API with optional OpenAPI schema expansion; this needs maintainer product/API scoping rather than cleanup closure. citeturn1view0turn1view1

## What This Changes

The issue requests dynamic FAL.ai catalog discovery so the fal media provider can expose the full model catalog rather than curated static entries.

## Best Possible Solution

Define a fal-owned, schema-aware catalog strategy for OpenClaw-supported media modes, and add a backwards-compatible media-catalog SDK seam only if plugin-local metadata cannot support selectable model discovery.

## Reproduction Assessment

Not applicable as a bug reproduction. There is a high-confidence current-state check: inspect the fal provider registrations and `image_generate`/`video_generate` list paths on current main, which expose only static `provider.models` arrays.

## Solution Assessment

Unclear. The full-catalog direction is useful, but exposing every fal endpoint as a selectable built-in media model is not yet proven to be the narrowest maintainable solution; a filtered, schema-aware fal catalog is safer.

## Risks / Open Questions

- The fal catalog spans heterogeneous image, video, audio, 3D, realtime, and workflow endpoints, so exposing every endpoint without schema-aware capability filtering could make model selection noisy or unsafe.
- Current media-generation provider contracts do not have an async catalog/discovery seam comparable to text model provider catalogs.
- The fal provider docs mention `openclaw models list --provider fal`, but current fal provider registration has no text-model catalog hook, so docs and product scope may need clarification while designing this.
