## Summary

Keep open. Current main still exposes fal media generation through curated static model arrays, and the list surfaces render those arrays directly, so the requested broad FAL catalog discovery remains unsolved and needs maintainer scoping.

## What This Changes

Request dynamic discovery and exposure of the full FAL.ai image/video model catalog in the fal media generation provider instead of the current curated model list.

## Best Possible Solution

Define a fal-owned catalog strategy that maps supported models to OpenClaw media modes/options, and add a backward-compatible media discovery seam only if static metadata cannot represent it safely.

## Reproduction Assessment

Not applicable as a bug reproduction. There is a high-confidence current-state check: inspect the fal image/video provider registrations and list actions on current main, which are bounded by provider.models arrays.

## Solution Assessment

Unclear. Exposing all FAL endpoints as built-in selectable media models is not yet proven to be the narrowest maintainable solution; a filtered, schema-aware catalog is safer.

## Risks / Open Questions

- FAL endpoints are heterogeneous across image, video, edit, reference, audio, and provider-specific option schemas, so exposing every endpoint without model-specific capability mapping could create broken selectable models.
- The current media generation provider API has only static model arrays, so true dynamic discovery may require an additive typed seam rather than a fal-only implementation change.
- The docs and UI/list surfaces may need coordinated updates once maintainers decide whether this should be a full catalog, filtered catalog, or external plugin path.
