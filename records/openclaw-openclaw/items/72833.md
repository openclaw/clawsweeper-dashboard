## Summary

Keep open. Current main still exposes fal media models through static registered arrays, and the list surfaces serialize those arrays; dynamic full-catalog discovery across mixed FAL media modalities remains valid but needs maintainer design rather than conservative cleanup.

## What This Changes

Request to expand the bundled fal image/video generation provider from curated static model lists to dynamic discovery of the full FAL.ai catalog as selectable media models.

## Best Possible Solution

Define a scoped, provider-owned FAL media catalog plan: filter by OpenClaw-supported media modes, map model-specific inputs and limits inside the fal plugin, and add a generic discovered-media-catalog SDK hook only if the existing image/video provider contracts cannot represent the selected catalog safely.

## Reproduction Assessment

Not applicable as a bug reproduction. There is a high-confidence current-state check: inspect the fal provider builders or the image_generate/video_generate list paths on current main and they return static registered model arrays.

## Solution Assessment

Unclear. The requested direction is useful, but exposing all FAL endpoints directly is not yet proven to be the narrowest maintainable solution; a filtered, capability-aware fal catalog with provider-owned schema handling is safer.
