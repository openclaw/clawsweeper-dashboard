## Summary

Keep open. Current main still exposes fal media generation through curated static model arrays, and the runtime list surfaces return those arrays directly; the requested full dynamic FAL.ai catalog is real fal/provider API work, not conservative cleanup.

## What This Changes

The issue requests expanding the bundled fal image and video generation provider to dynamically discover and expose FAL.ai's full model catalog as selectable media models.

## Best Possible Solution

Define a fal-owned media catalog strategy that filters FAL models to OpenClaw-supported image/video modes, maps per-model schemas and limits inside the fal plugin, and adds a backward-compatible media catalog SDK seam only if the static provider.models contract is insufficient.

## Reproduction Assessment

Not applicable as a bug reproduction. The current-state check is high confidence: inspect fal provider registrations or run the image_generate/video_generate list paths, which are bounded by static provider.models arrays on current main.

## Solution Assessment

Unclear. Exposing every FAL endpoint as a built-in selectable model is not yet proven to be the narrowest maintainable solution; a filtered, capability-mapped fal catalog with provider-owned schema handling is safer.
