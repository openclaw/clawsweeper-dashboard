## Summary

Keep open. Current main still exposes fal image and video generation through provider-owned static model arrays, while this asks for broad dynamic FAL.ai catalog discovery across mixed media modalities; that is real provider/API/product work rather than conservative cleanup.

## What This Changes

The issue requests dynamically exposing the full FAL.ai model catalog through the OpenClaw fal media provider instead of a small hardcoded model list.

## Best Possible Solution

Define a scoped, provider-owned FAL media catalog strategy: discover or cache only OpenClaw-supported media models, filter by image/video mode capability, map per-model schema and limits inside the fal plugin, and add an additive generic discovered-media-catalog SDK seam only if the existing static provider contracts are insufficient.

## Reproduction Assessment

Not applicable as a bug reproduction. There is a high-confidence current-state check: inspect the fal image/video providers and action=list surfaces on current main, which show static provider.models arrays rather than dynamic catalog discovery.

## Solution Assessment

Unclear. The feature direction is valid, but exposing all FAL endpoints as selectable built-in media models is not proven to be the narrowest maintainable solution; a filtered capability-mapped catalog with provider-owned schema handling is safer.
