## Summary

Keep open: current main has related image-history and replay hygiene, but it does not yet add a final Responses `image_url` sanitizer, sanitize raw image-like text in the Codex app-server mirrored-history path, or quarantine poisoned Codex thread bindings.

## What This Changes

The issue reports that raw or malformed image payloads in replayed history can become invalid OpenAI Responses `image_url` entries and repeatedly wedge a session.

## Best Possible Solution

Add a shared Responses image-payload sanitizer at the transport boundary and in the Codex mirrored-history/bootstrap path, then clear or quarantine an existing Codex app-server binding only when definite malformed image poison is detected.

## Reproduction Assessment

No high-confidence end-to-end reproduction was run because the original raw payload is intentionally absent. Source inspection gives a medium-confidence reproduction path: malformed text or malformed image records can survive replay/mirrored-history paths, and current Responses sends have no final `image_url` validator.

## Solution Assessment

Yes: the narrow maintainable fix is layered defense, with a final Responses payload sanitizer plus Codex mirrored-history sanitization and binding quarantine on definite poison detection. The existing image-block sanitizer alone is too narrow because raw base64 text and already-shaped Responses payloads can bypass it.
