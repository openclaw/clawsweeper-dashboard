## Summary

Keep open. Current main still lacks a final OpenAI/Azure Responses `image_url` sanitizer and Codex mirrored-history/binding poison handling; existing image cleanup validates structured OpenClaw image blocks but does not cover malformed Responses-shaped payloads or raw image-looking text in mirrored history.

## What This Changes

Requests hardening so raw or invalid historical image payloads cannot be replayed as OpenAI Responses `image_url` parts or keep a Codex app-server thread permanently wedged.

## Best Possible Solution

Add a shared Responses-image sanitizer at the final OpenAI/Azure send boundary, reuse it for provider wrapper payload hooks, sanitize Codex mirrored history before projection, and quarantine the Codex app-server binding when poisoned image history is detected.

## Reproduction Assessment

No live high-confidence reproduction was run in this read-only review. Source inspection gives a clear reproduction path: a malformed Responses-shaped `function_call_output.output[*].image_url` can survive final request construction, while Codex mirrored history is read without invalid-image quarantine.

## Solution Assessment

Yes. The proposed direction is the maintainable fix: validate/degrade malformed image payloads at the final Responses boundary and in Codex mirrored-history recovery rather than relying on earlier structured image-block sanitizers.
