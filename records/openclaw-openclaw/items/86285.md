## Summary

Keep open: the current head is a focused attempt at the right bug, but the auto-response suppression is still too late for pre-ready caller audio, so the double-greeting race can remain. The protected maintainer label and missing live Twilio/OpenAI proof also make this a human-review item before merge.

## What This Changes

The PR keeps voice-call queued initial greetings on the explicit ready trigger and adds OpenAI provider logic/tests that temporarily disable Realtime server-VAD auto responses around manual `response.create` turns.

## Best Possible Solution

Suppress OpenAI auto-response before any queued caller audio can reach Server VAD for an explicit initial greeting, preserve non-OpenAI explicit greetings, add a pre-ready-audio regression, and require redacted live outbound Twilio/OpenAI proof.

## Reproduction Assessment

Yes, source-reproducible: current main queues caller audio before OpenAI session readiness, flushes it before the ready greeting, and OpenAI Server VAD can auto-create a response when user speech stops. I did not run the live Twilio/OpenAI call.

## Solution Assessment

No. The PR moves in the right direction, but suppressing auto-response inside `requestResponseCreate()` does not cover pre-ready caller audio that is flushed before the explicit greeting path runs.

- **[P1] Suppress auto responses before queued audio reaches VAD:** `extensions/openai/realtime-voice-provider.ts:1278-1279`
  - body: The new suppression is sent from `requestResponseCreate()`, but `session.updated` drains `pendingAudio` before `onReady` fires the initial greeting.
