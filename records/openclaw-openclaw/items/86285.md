## Summary

Keep this PR open for review: the latest patch is still source-reproducibly vulnerable to the startup audio ordering that can produce two OpenAI responses, and the contributor explicitly has not supplied live Twilio/OpenAI proof.

## What This Changes

The PR changes the voice-call/OpenAI realtime path to keep queued outbound initial greetings on the explicit ready trigger while temporarily disabling and restoring OpenAI server-VAD auto responses around manual `response.create` turns.

## Best Possible Solution

Keep the explicit greeting path, but enter a single response-source mode before any queued caller audio is flushed, then restore auto-response after the manual turn and prove it with a pre-ready-audio regression plus redacted live Twilio/OpenAI evidence.

## Reproduction Assessment

Yes, by source inspection: Twilio media frames can reach `bridge.sendAudio()` before OpenAI readiness, then `session.updated` drains queued audio before the manual greeting suppression runs. I did not run a live Twilio/OpenAI call in this read-only review.

## Solution Assessment

No. Suppressing auto-response inside `requestResponseCreate()` is too late for audio buffered before `session.updated`; the fix should suppress before that drain for queued initial greetings and prove the real call ordering.
