## What This Changes

The PR changes the voice-call/OpenAI realtime path so queued outbound initial greetings stay on the explicit ready-trigger path while OpenAI server-VAD auto responses are temporarily disabled and restored around manual `response.create` calls, with focused regressions.

## Best Possible Solution

Make OpenAI initial-turn suppression active before any queued caller audio is drained for that first manual greeting, add a regression for media-before-`session.updated`, and then provide redacted live Twilio/OpenAI outbound-call proof for the exact head.

## Reproduction Assessment

Yes, by source inspection: send caller media before OpenAI emits `session.updated`, then the bridge drains that queued audio before `onReady` triggers greeting and suppression. I did not run a live Twilio/OpenAI call in this read-only review.

## Solution Assessment

No; the current patch suppresses auto-response too late for queued pre-ready audio. The safer fix is to cover the initial greeting's pre-ready audio window while preserving the explicit OpenAI greeting trigger and non-OpenAI behavior.
