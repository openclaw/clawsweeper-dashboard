## Summary

Keep open: the patch still leaves a fast-caller-audio ordering hole where OpenAI server VAD can auto-create a response before the explicit initial greeting suppression takes effect, and the signed head still lacks live Twilio/OpenAI proof.

## What This Changes

The PR changes voice-call/OpenAI realtime initial greeting handling and OpenAI manual response creation to suppress server-VAD auto responses around explicit `response.create`, with focused regression tests.

## Best Possible Solution

Suppress OpenAI auto responses before any pre-ready caller audio is drained when a queued initial greeting exists, keep the explicit greeting as the single first response source, add a regression for that ordering, and verify with redacted live Twilio/OpenAI proof.

## Reproduction Assessment

Yes from source, but not from a live run in this review. Current code drains pre-ready audio after `session.updated` while OpenAI VAD defaults to auto response, then fires the explicit ready greeting afterward; the linked report supplies the live Twilio/OpenAI symptom.

## Solution Assessment

No. The explicit greeting plus temporary suppression direction is plausible, but the current patch suppresses after queued caller audio can already reach OpenAI, so it needs an earlier suppression point and live proof.
