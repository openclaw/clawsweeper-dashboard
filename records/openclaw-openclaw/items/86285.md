## Summary

Keep this PR open: it is a plausible fix direction, but the current head still leaves a response-ordering race for queued caller audio and the contributor-side proof is still test-only.

## What This Changes

The PR keeps voice-call OpenAI outbound initial greetings on the explicit ready-trigger path and suppresses OpenAI realtime auto responses around manual response creation to avoid double greetings.

## Best Possible Solution

Move the OpenAI auto-response suppression earlier for queued initial greetings so it is active before any pre-ready caller audio is drained, add a focused regression for that ordering, and then attach redacted live Twilio/OpenAI outbound-call proof for the exact signed head.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: queue caller audio before OpenAI `session.updated`, then `session.updated` drains that audio before `onReady` triggers the manual greeting suppression. The linked issue also reports repeated live Twilio/OpenAI double greetings, but I did not run the live call path here.

## Solution Assessment

No, not as-is. Temporary suppression around manual `response.create` is a reasonable direction, but it must cover the pre-ready queued-audio window before the manual greeting is requested.
