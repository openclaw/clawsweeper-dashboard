- `merge-risk: 🚨 message-delivery`: The diff can suppress the only current path that delivers the outbound initial message to OpenAI realtime calls.
- `merge-risk: 🚨 compatibility`: Existing users who pass `--message` or `initialMessage` on OpenAI realtime outbound calls could see that configured greeting stop being honored.

## Summary

Keep open: the bug is real, but this patch avoids the duplicate OpenAI response by suppressing the only current path that carries the configured outbound initial message, so it can turn a double greeting into a missing configured greeting.

## What This Changes

The PR changes the voice-call realtime handler to skip `triggerGreetingOnReady` for OpenAI providers while preserving explicit initial greetings for other realtime providers, and adds focused handler tests.

## Best Possible Solution

Keep one OpenAI initial response while preserving the configured greeting content, either by feeding that content into the OpenAI auto-response path or by narrowly suppressing OpenAI auto-response only around the explicit first greeting.

## Reproduction Assessment

Yes from source: current `main` routes `initialMessage` into `initialGreetingInstructions`, and the PR disables the only `onReady` path that consumes those instructions for OpenAI. I did not run a live Twilio/OpenAI call in this read-only review.

## Solution Assessment

No. The proposed path removes the duplicate response but does not preserve the configured greeting content, so the safer repair is to keep a single response source that still receives the initial message.
