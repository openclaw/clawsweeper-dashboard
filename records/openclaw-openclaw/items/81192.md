## Summary

Keep open. This active protected-label PR changes real onboarding behavior, but the current patch still has blockers: it reuses the broad `installed` signal for migration offers and runs full Codex migration detection during onboarding, which can re-prompt existing installs and launch a 60-second Codex app-server probe before migration opt-in.

## What This Changes

The branch adds a post-install migration helper, adds `pluginId` to Codex runtime install results, and calls the helper from interactive and non-interactive provider auth-choice onboarding flows.

## Best Possible Solution

Keep the post-install migration offer, but gate it on a real fresh harness install, use a light or short Codex source probe before prompting, and document the new onboarding prompt behavior.

## Reproduction Assessment

Yes by source inspection: a Codex-routed auth choice calls the new helper when `installed` is true, and the helper immediately runs provider `detect()`. I did not run the wizard in this read-only checkout.

## Solution Assessment

No. The direction is reasonable, but the implementation should not treat an already-present/repair result as a fresh install and should not run full Codex app-server detection just to decide whether to show an optional migration offer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Honor the new-install contract for migration offers:** `src/plugins/provider-auth-choice.ts:167-180`
  - body: `ensureCodexRuntimePluginForModelSelection` returns `installed: true` when the Codex plugin is already on disk and only repaired/enabled, so using `installed` here routes existing harnesses through a helper whose contract says the ids were just installed. That can re-surface the migration prompt on later auth-choice/setup passes instead of anchoring it to the install event; carry a separate fresh-install signal or update this call path before prompting.
  - confidence: 0.86
- **[P2] Avoid full Codex detection before migration opt-in:** `src/wizard/setup.post-install-migration.ts:67-71`
  - body: The new helper calls the provider's full `detect()` before the user accepts migration. For Codex, `detect()` reaches `discoverCodexSource()`, which always asks `codex app-server` for `plugin/list` with the default 60s request timeout, so onboarding can spawn or wait on Codex just to decide whether to show a prompt. Use a light filesystem-only probe, a short explicit timeout, or defer app-server inventory until the user chooses the migration flow.
  - confidence: 0.82

## Security Review
