## Summary

Keep open for review, but this patch is not merge-ready: the detected legacy encoding names are passed directly to `Buffer#toString`, which throws for the key encodings this PR is meant to support, and the proof exercises a helper rather than the read-tool call site.

## What This Changes

The branch adds a new encoding-detection utility and wires the read tool text path to decode files using detected BOM/content heuristics.

## Best Possible Solution

Use a supported decoder at the read-tool call site, preferably the existing strict-UTF-8 plus Windows-codepage fallback for the linked GBK bug, or deliberately implement broader TextDecoder-based auto-detection with focused tests and real read-tool proof.

## Reproduction Assessment

Yes at source level: the diff routes detected `shift-jis`, `gbk`, and `utf-16be` values into `Buffer#toString`, and Node rejects those Buffer encodings. I did not run the full PR branch because this was a read-only review.

## Solution Assessment

No as written: auto-detection is a plausible direction, but the call site must use a real decoder and the linked GBK bug likely has a narrower in-tree fix through the existing Windows codepage helper.
