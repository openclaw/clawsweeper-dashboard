## Summary

Keep open: this PR is a focused attempt at the linked GBK read-tool bug, but the current patch would route detected legacy encodings into unsupported `Buffer.toString()` names and its own proof shows the GBK case is still misclassified.

## What This Changes

The PR adds a new encoding-detection utility and changes the session read tool to decode text files with the detected encoding instead of always using UTF-8.

## Best Possible Solution

Use a supported decoder path, preferably reusing the strict-UTF-8 then Windows-codepage/TextDecoder precedent, and add focused read-tool coverage plus real GBK proof before merge.

## Reproduction Assessment

Yes at source level: the PR passes detected legacy names to `Buffer.toString()`, and Node rejects the advertised GBK, Shift-JIS, and UTF-16BE names with `ERR_UNKNOWN_ENCODING`. The PR body also reproduces the central GBK sample as misclassified rather than fixed.

## Solution Assessment

No. Automatic detection is a plausible direction, but this implementation is not the best fix because it bypasses the existing supported TextDecoder/codepage path and does not correctly handle the linked GBK case.
