## Label Justifications

- `P2`: This is a normal-priority security/message-delivery hardening issue with strong source evidence but a policy-sensitive, limited blast radius.
- `impact:security`: The report concerns internal model summaries or reasoning-like text becoming visible when silence was intended.
- `impact:message-loss`: The requested behavior changes whether channel replies are delivered or suppressed when `NO_REPLY` appears with other text.

## Summary

Keep open: current main only partially covers this class of `NO_REPLY` leaks. Reasoning-prefixed and cron-specific cases have targeted fixes, but the shared reply normalizer and outbound delivery tests still preserve ordinary substantive text followed by `NO_REPLY`, and the linked fork PR that would change that policy is open, conflicting, and unmerged.

## What This Changes

Request to suppress the entire model reply when `NO_REPLY` appears with preceding text instead of stripping only the token.

## Best Possible Solution

Define a maintainer-approved shared silent-token classifier that suppresses internal or explicit silent-intent payloads while preserving legitimate substantive replies, then apply it consistently across shared reply normalization and affected delivery paths.

## Reproduction Assessment

Yes, as a source-reproducible issue. Current main strips a generic trailing `NO_REPLY` and returns the non-empty prefix, and adjacent tests assert that behavior; I did not run a live channel delivery in this read-only review.

## Solution Assessment

No, not as a blanket suppress-if-contains rule. The safer path is a maintainer-approved classifier that distinguishes internal or silent-intent payloads from legitimate substantive replies before changing shared delivery semantics.
