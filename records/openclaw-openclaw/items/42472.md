## Summary

Keep open: current main still preserves preamble text in the main auto-reply normalizer when a response ends with `NO_REPLY`, and open PRs #42873, #46626, and #69394 already track overlapping fixes that need maintainer selection before closure.

## What This Changes

The issue requests post-processing model replies so a response containing or trailing `NO_REPLY` suppresses the whole user-visible message instead of only stripping the token.

## Best Possible Solution

Pick a narrow policy for trailing or own-line silent tokens, apply it consistently across final normalization and delivery guards, preserve ordinary inline mentions, and cover streaming separately if maintainers want that behavior.

## Reproduction Assessment

Yes from source inspection: `normalizeReplyPayload({ text: "summary\n\nNO_REPLY" })` strips the trailing token and keeps the remaining summary because it still has content. The existing test suite also asserts that behavior today.

## Solution Assessment

No for the broad suggested `/NO_REPLY/` form, because current docs/tests preserve some mixed-content cases. A narrower trailing or own-line silent-token helper used consistently across delivery paths is the maintainable direction.

## Risks / Open Questions

- Current main has conflicting boundaries: auto-reply normalization preserves trailing-token preambles while cron delivery suppresses them.
- A broad `/NO_REPLY/` rule could suppress legitimate user-facing text that mentions the token; a trailing or own-line rule is safer.
- Streaming/block delivery may need separate handling because already-sent partial chunks cannot always be retroactively suppressed.
