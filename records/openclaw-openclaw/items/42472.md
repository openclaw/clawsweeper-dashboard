## Summary

Keep open: current main still preserves and delivers preamble text when shared reply normalization sees mixed content ending in `NO_REPLY`, and open linked PRs are already tracking alternative fixes that need maintainer policy/security review.

## What This Changes

Request suppression of the entire visible reply when model output contains or ends with `NO_REPLY`, instead of only stripping the token.

## Best Possible Solution

Define one documented silent-reply classifier and apply it consistently across normalization, directive parsing, outbound planning, cron/subagent handling, gateway projection, and runner display paths with false-positive, media, and real-delivery proof.

## Reproduction Assessment

Yes, source-level: current main strips a trailing mixed `NO_REPLY` and returns the non-empty preamble in `normalizeReplyPayload`, and the adjacent test asserts that behavior. I did not run a live channel flow because this review is read-only.

## Solution Assessment

No, not as a broad contains-based post-process. The safer maintainable path is a shared, documented classifier applied consistently across delivery and display paths while preserving legitimate token mentions, media, and channel metadata.
