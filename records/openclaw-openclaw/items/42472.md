## Summary

Keep open: current main and the latest release still keep the preamble in the shared normalizer and exact-token directive paths, while the open related PR is an implementation candidate that still needs maintainer policy/security review and proof before this issue can close.

## What This Changes

Request to suppress an entire assistant reply when `NO_REPLY` appears after a preamble, instead of stripping only the token and delivering the preceding text.

## Best Possible Solution

Define one documented silent-reply classifier for own-line or trailing silent-token intent, then apply it consistently across normalization, directive parsing, outbound planning, cron/subagent delivery, gateway projection, and display paths with false-positive and media-payload coverage.

## Reproduction Assessment

Yes, source-level: current main strips a trailing mixed `NO_REPLY` and returns the non-empty preamble in `normalizeReplyPayload`, and the adjacent test asserts that behavior. I did not run live channel proof because this review is read-only.

## Solution Assessment

No, not as a raw contains-based post-process. The safer path is a shared, documented classifier that preserves legitimate inline mentions while suppressing own-line or trailing silent-token intent across all delivery and display paths.
