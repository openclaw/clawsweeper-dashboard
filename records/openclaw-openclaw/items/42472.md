## Summary

Keep open: current main and v2026.6.6 only partially address the reported NO_REPLY leak. Reasoning-prefixed leaks and cron trailing-token delivery are now covered, but the broad requested own-line/contains-token suppression still conflicts with current tests and docs that preserve substantive mixed NO_REPLY text.

## What This Changes

The issue requests suppressing the entire model reply when `NO_REPLY` appears after visible text so heartbeat or background summaries do not leak to users.

## Best Possible Solution

Choose one shared mixed-NO_REPLY suppression policy, then apply it consistently across the reply normalizer, outbound payload planning, cron/gateway/channel delivery, and focused regression tests.

## Reproduction Assessment

Yes, at source level. Current main still strips NO_REPLY from generic mixed text and delivers the remaining content in normalizeReplyPayload, while tests preserve substantive trailing-token replies.

## Solution Assessment

No for an automatic blind fix. A shared guard is plausible, but the broad suppress-if-present direction conflicts with current docs/tests, so maintainers need to choose the policy boundary first.
