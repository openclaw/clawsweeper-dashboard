## Summary

Keep open: current main still strips a trailing `NO_REPLY` token while preserving the preceding text in the main reply normalizer, and open PRs #42873, #46626, and #69394 already cover overlapping fixes that need maintainer selection before closure.

## What This Changes

Suppress the entire reply when a model output includes a silent `NO_REPLY` signal instead of stripping only the token and delivering preceding text.

## Best Possible Solution

Choose one narrow trailing or own-line silent-token policy, apply it consistently across normalization and delivery guards, and update tests/docs while preserving legitimate inline mentions.

## Reproduction Assessment

Yes. Source inspection and the existing test show `normalizeReplyPayload({ text: "summary\n\nNO_REPLY" })` strips the token and keeps the preceding text as a visible payload.

## Solution Assessment

No for the broad contains-based rule as written; a narrower own-line or trailing-token helper shared across delivery paths is the safer maintainable direction.
