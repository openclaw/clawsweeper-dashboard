- `P2`: The issue affects outbound silent-token delivery and internal-text leakage, but the remaining work is a bounded normal-priority policy and hardening decision.
- `impact:security`: Malformed silent-control output can expose internal summaries or reasoning-like text into external chats when a heartbeat or background task should stay private.
- `impact:message-loss`: The central behavior is deciding when channel messages should be delivered, stripped, or fully suppressed.

## Summary

Keep open: current main and v2026.6.6 have partial NO_REPLY hardening, but they still deliberately preserve some mixed-content trailing-token replies, and this issue is paired with an open conflicting linked PR, so the remaining suppression policy needs maintainer product/security review.

## What This Changes

Request suppressing an entire generated reply when the model appends or includes NO_REPLY so internal heartbeat or cron summaries do not reach users.

## Best Possible Solution

Define one shared mixed-NO_REPLY silent-token policy, then apply it consistently across reply normalization, outbound payload planning, cron/gateway/channel delivery, and tests while preserving the approved media and substantive-reply behavior.

## Reproduction Assessment

Yes, at source level. Current normalizeReplyPayload strips trailing NO_REPLY and returns the remaining text, and adjacent tests assert that substantive text followed by NO_REPLY is delivered; no tests were run in this read-only review.

## Solution Assessment

Unclear. A gateway or shared-pipeline guard is plausible, but current docs and tests preserve substantive mixed NO_REPLY replies, so maintainers should choose the shared policy before implementation.
