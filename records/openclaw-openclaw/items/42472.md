## Summary

Keep open: current main and the latest release still preserve text before a trailing or mixed `NO_REPLY` in shared reply normalization and outbound delivery, while linked implementation PRs are still open or unresolved.

## What This Changes

Request suppressing the entire reply when a model emits preamble text followed by a `NO_REPLY` silent marker.

## Best Possible Solution

Define and document one shared own-line or trailing silent-token classifier, then apply it consistently across reply normalization, outbound payload planning, cron/subagent delivery, gateway display, and tests while preserving media and structured payload behavior.

## Reproduction Assessment

Yes, source-level. Current main strips a trailing mixed `NO_REPLY` and returns the non-empty preamble in `normalizeReplyPayload`, with adjacent tests asserting that behavior; no live channel run was performed in this read-only review.

## Solution Assessment

No, not as a raw contains-based post-process. The maintainable path is a shared own-line or trailing-token classifier with explicit compatibility semantics, payload-preservation tests, and consistent use across delivery paths.
