## Summary

Keep open: current main and v2026.6.6 still preserve some substantive text before a trailing NO_REPLY, so the broad suppress-if-present request is not implemented and needs maintainer product/security judgment.

## What This Changes

The issue requests suppressing the entire reply when model output contains or ends with NO_REPLY, instead of stripping only the token and delivering preceding text.

## Best Possible Solution

Maintainers should choose one explicit NO_REPLY contract, then implement it in the shared classifier/normalizer and affected delivery paths with tests that protect both leak prevention and false-positive delivery cases.

## Reproduction Assessment

Yes, at source level. Current main strips trailing NO_REPLY from generic mixed-content replies and delivers the remaining text, with tests explicitly preserving that behavior.

## Solution Assessment

No for an automatic fix. A stronger suppression guard is plausible, but it changes an encoded mixed-token contract, so maintainers need to choose the security/message-delivery boundary first.
