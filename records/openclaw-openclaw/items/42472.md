## Summary

Keep open: current main and v2026.6.6 still preserve the generic strip-and-deliver behavior for substantive text ending in NO_REPLY, while only narrower reasoning-prefixed and cron paths are partially hardened; the open closing fork PR and conflicting tests make this a maintainer policy/security decision, not a cleanup close.

## What This Changes

The issue asks OpenClaw to suppress the entire visible reply when model output contains or ends with a NO_REPLY silence token after preamble text.

## Best Possible Solution

Define one maintainer-approved silent-token classifier for shared reply delivery, preserving legitimate substantive replies while suppressing internal/reasoning or explicit silent-intent payloads, then land it with focused coverage across the shared normalizer and affected delivery paths.

## Reproduction Assessment

Yes, source-level: current main still strips a generic trailing NO_REPLY and returns the non-empty preamble, and the adjacent test asserts that behavior. I did not run live channel proof in this read-only review.

## Solution Assessment

No, not as a blanket `/NO_REPLY/` post-process rule. The maintainable fix needs a maintainer-approved classifier because broad suppression conflicts with current tests and can silently drop legitimate visible replies.
