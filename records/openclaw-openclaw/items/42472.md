## Summary

Keep open: current main and v2026.6.8 still preserve generic strip-and-deliver behavior for substantive text ending in NO_REPLY, while reasoning-prefixed and cron-specific paths are only partial fixes. The remaining behavior is a security/message-delivery policy decision, especially with an open conflicting fork PR still linked as a closing candidate.

## What This Changes

The issue requests suppressing the entire reply when model output contains or ends with NO_REPLY instead of only stripping the token.

## Best Possible Solution

Define one maintainer-approved shared silent-token classifier that suppresses internal or explicit silent-intent payloads while preserving legitimate substantive replies, then land it with coverage across the shared normalizer and affected delivery paths.

## Reproduction Assessment

Yes. Source-level inspection shows current main strips a generic trailing NO_REPLY and returns the non-empty preamble, and the adjacent test asserts that behavior; no live channel run was performed in this read-only review.

## Solution Assessment

No, not as a blanket /NO_REPLY/ post-process rule. The safer direction is a maintainer-approved classifier that distinguishes internal/silent-intent payloads from legitimate substantive replies before changing shared delivery semantics.
