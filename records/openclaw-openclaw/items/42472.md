- `P2`: The issue affects outbound silent-token delivery and possible internal-text leakage, but the remaining work is a bounded normal-priority policy and hardening decision.
- `impact:security`: Malformed silent-control output can expose internal summaries or reasoning-like text into external chats when a heartbeat or background task should stay private.
- `impact:message-loss`: The central behavior is deciding when channel messages should be delivered, stripped, or fully suppressed.

## Summary

Keep open: current main and v2026.6.6 only partially harden NO_REPLY handling. The requested broad own-line or contains-token suppression still conflicts with current tests that preserve substantive trailing-token replies, and an open linked PR is an implementation candidate rather than a reason to close this issue.

## What This Changes

The issue requests suppressing an entire auto-reply when model output includes trailing or own-line NO_REPLY so pre-token text is not delivered.

## Best Possible Solution

Define one shared mixed-NO_REPLY suppression policy, then apply it consistently across reply normalization, outbound payload planning, cron/gateway/channel delivery, and regression tests while preserving the approved media and substantive-reply behavior.

## Reproduction Assessment

Yes, at source level. A payload like `All tasks completed.\nNO_REPLY` is not classified as silent by current main, then `normalizeReplyPayload` strips only the token and keeps the preceding text.

## Solution Assessment

No for an automatic code change. A shared guard is plausible, but the broad suppress-if-present direction conflicts with current tests preserving substantive trailing-token replies, so maintainers should choose the policy first.
