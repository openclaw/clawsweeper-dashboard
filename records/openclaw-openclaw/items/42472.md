## Summary

Keep open: current main and the latest release still preserve preamble text before a trailing `NO_REPLY` in the shared reply normalizer, and open linked PRs are already carrying competing implementation paths that need maintainer policy review.

## What This Changes

The issue asks for mixed replies containing or ending with `NO_REPLY` to suppress the whole outgoing message instead of stripping only the token and delivering earlier text.

## Best Possible Solution

Choose and document one shared silent-token classifier, then land or replace the open linked implementation PRs with regression coverage across normalization, directive/outbound planning, cron/subagent paths, and dispatcher delivery.

## Reproduction Assessment

Yes. Source inspection shows `normalizeReplyPayload` still returns the preamble after stripping a trailing `NO_REPLY`, and current tests assert that strip-and-deliver behavior; no runtime command was run because this review is read-only.

## Solution Assessment

Unclear. The requested contains-based post-process would stop the reported leak, but it can also suppress legitimate messages that mention the token, so the safer path is a maintainer-chosen shared classifier boundary before implementation.
