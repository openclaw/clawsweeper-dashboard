## Summary

Keep open. Current main still strips a trailing `NO_REPLY` token from mixed-content text and returns the non-empty preamble as deliverable content, while open PRs already target overlapping fixes and policy choices.

## What This Changes

The issue requests suppressing the entire reply when model output contains or ends with `NO_REPLY`, instead of stripping only the token and delivering preceding text.

## Best Possible Solution

Define one shared boundary-aware silent-intent helper for exact tokens, JSON envelopes, own-line or trailing control tokens, and ordinary inline mentions, then reuse it consistently across reply normalization, outbound planning, cron, and gateway projection.

## Reproduction Assessment

Yes. Source inspection and the current #30916 test provide a high-confidence path: `normalizeReplyPayload({ text: "File's there. Not urgent.\n\nNO_REPLY" })` remains non-null and keeps the preamble after stripping the token.

## Solution Assessment

Unclear as originally suggested. A raw `/NO_REPLY/` suppression rule is too broad for current inline-mention compatibility, so the maintainable fix should be a shared boundary-aware predicate with focused regressions.
