## Summary

Keep open: current main and v2026.5.22 still strip mixed trailing NO_REPLY into deliverable preamble text in shared reply normalization and outbound delivery, while cron has only scoped hardening. The remaining work is security-sensitive and needs maintainer policy review because broader suppression can conflict with existing mixed-token behavior and overlaps open implementation PRs.

## What This Changes

The issue requests a post-processing rule that suppresses an entire assistant reply when a model appends or includes NO_REPLY after internal preamble text.

## Best Possible Solution

Define one documented silent-reply classifier for the maintainer-approved boundary, apply it consistently across shared delivery surfaces, preserve media and structured payloads, and require real delivery proof before closing this issue.

## Reproduction Assessment

Yes, at source level. Current main strips a trailing mixed NO_REPLY and returns the non-empty preamble in normalizeReplyPayload, and adjacent tests assert that behavior; no live channel run was performed in this read-only review.

## Solution Assessment

No, not as a raw contains-based post-process. The maintainable path is a shared classifier with explicit compatibility semantics, payload-preservation tests, docs, and consistent use across delivery paths.
