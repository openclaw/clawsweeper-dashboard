## Summary

Keep open: current main still strips a trailing `NO_REPLY` token while preserving the preceding text in the shared reply normalizer, and live GitHub still has open, competing implementation PRs rather than a merged fix.

## What This Changes

The issue asks OpenClaw to suppress an entire reply when a model emits `NO_REPLY` after other text, instead of delivering the preamble with only the token removed.

## Best Possible Solution

Choose one documented shared silent-token classifier, apply it consistently across normalization and final delivery paths, update regression tests/docs, then close this issue after the selected PR merges.

## Reproduction Assessment

Yes. Source inspection shows `normalizeReplyPayload` currently turns `summary\n\nNO_REPLY` into deliverable preamble text, and the current test suite asserts that behavior; no test command was run because this review is read-only.

## Solution Assessment

Unclear as proposed. A broad contains-based rule can suppress legitimate text that discusses the token, so maintainers should choose a narrower shared own-line, trailing-token, or reasoning-prefixed policy before landing one implementation.
