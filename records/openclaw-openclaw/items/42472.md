## Summary

Keep open. Current main still strips a trailing NO_REPLY token and returns the non-empty preamble from the shared reply normalizer, and the colocated regression test still expects that preamble to be delivered; cron has a narrower trailing-token suppression path, but the shared auto-reply/directive/outbound paths are not fully fixed. Open PRs #42873, #46626, and #69394 already target overlapping fixes, so this should stay open until maintainers choose and land one canonical policy.

## What This Changes

The issue requests a gateway/reply-pipeline guard that suppresses an entire model response when NO_REPLY appears after internal preamble text instead of delivering the preamble with the token stripped.

## Best Possible Solution

Define one shared silent-intent helper with explicit rules for exact tokens, JSON envelopes, glued leading tokens, own-line or trailing control tokens, and ordinary inline mentions, then reuse it across reply normalization, directive parsing, outbound delivery projection, gateway live/history projection, cron, and subagent follow-up paths with regression tests for each delivery surface.

## Reproduction Assessment

Yes. Current source and tests give a high-confidence reproduction path: the shared normalizer strips NO_REPLY from `"File's there. Not urgent.\n\nNO_REPLY"` and the test at `src/auto-reply/reply/reply-utils.test.ts:122` still expects the preamble to be returned.

## Solution Assessment

Unclear as originally suggested. A raw `/NO_REPLY/` match is too broad for legitimate inline mentions; the maintainable fix is an explicit shared predicate for trailing or own-line control-token intent, applied consistently across all delivery paths.
