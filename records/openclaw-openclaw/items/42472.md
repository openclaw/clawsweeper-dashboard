## Summary

Keep open: current main still strips a trailing NO_REPLY token while preserving the preceding text in the main reply normalizer, and live GitHub shows open implementation PRs with competing suppression policies rather than a merged fix.

## What This Changes

The issue requests suppressing the entire outgoing reply when a model emits a NO_REPLY suppression token after other text.

## Best Possible Solution

Review and land one canonical shared silent-token classifier that covers the chosen policy across normalization and final delivery, then update regression tests and docs before closing the related issues.

## Reproduction Assessment

Yes. Source inspection shows normalizeReplyPayload currently turns "summary\n\nNO_REPLY" into a deliverable preamble, and the existing test suite asserts that behavior; no test command was run because this review is read-only.

## Solution Assessment

No for the broad contains-based rule as written. A narrower shared own-line or trailing-token classifier is safer, but maintainers need to choose the exact policy before one PR lands.
