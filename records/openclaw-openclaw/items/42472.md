## Summary

Keep open: current main and latest `v2026.5.19` still strip a trailing `NO_REPLY` and deliver the non-empty preamble, while https://github.com/openclaw/openclaw/pull/46626 already tracks a broader fix that needs maintainer policy, security, and proof review.

## What This Changes

Request broadening `NO_REPLY` handling so a model response with text before a trailing or own-line `NO_REPLY` is suppressed instead of delivered after stripping the token.

## Best Possible Solution

Define one documented silent-reply classifier and apply it consistently across normalization, directive parsing, outbound planning, cron/subagent delivery, gateway projection, and display paths with regression coverage for false positives and media-bearing payloads.

## Reproduction Assessment

Yes, source-level: current main strips a trailing mixed `NO_REPLY` and returns the non-empty preamble in `normalizeReplyPayload`, and the adjacent test asserts that behavior. I did not run live channel proof because this review is read-only.

## Solution Assessment

No, not as a raw contains-based post-process. The safer path is a shared, documented classifier that preserves legitimate inline mentions while suppressing own-line or trailing silent-token intent across all delivery/display paths.
