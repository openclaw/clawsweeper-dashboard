## Summary

Keep open: current main still strips a trailing `NO_REPLY` token while preserving the preceding text in the main reply normalizer, and several open PRs cover competing fixes that need maintainer selection.

## What This Changes

Request to suppress an entire reply when model output includes or ends with `NO_REPLY`, instead of stripping only the token and delivering the preamble.

## Best Possible Solution

Choose one canonical case-insensitive trailing or own-line silent-token policy, share it across normalization and final delivery guards, update tests/docs, and preserve legitimate inline mentions.

## Reproduction Assessment

Yes. Source inspection and the current regression test show `normalizeReplyPayload({ text: "summary\n\nNO_REPLY" })` strips the token but keeps the preamble as a deliverable payload.

## Solution Assessment

No for the broad contains-based rule as written. A narrower own-line or trailing-token helper applied consistently across delivery paths is safer, but maintainers need to choose the exact policy before one PR lands.
