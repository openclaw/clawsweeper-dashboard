## Summary

Keep open. Current main still strip-delivers a non-empty preamble before trailing `NO_REPLY` in the shared reply normalizer, and open PR #42873 is already attached as the closing fix while #46626 and #69394 cover overlapping policy variants.

## What This Changes

Request to suppress an entire assistant reply when raw output includes or ends with `NO_REPLY`, instead of stripping only the token and delivering preceding text.

## Best Possible Solution

Choose one boundary-aware silent-intent rule and share it across reply normalization, directive parsing, outbound delivery, cron, live chat, and history projection with focused regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: `normalizeReplyPayload({ text: "File's there. Not urgent.\n\nNO_REPLY" })` still returns the preamble, and the current test suite asserts that behavior.

## Solution Assessment

Unclear as originally suggested. Matching any `/NO_REPLY/` occurrence is too broad; the safer maintainable solution is a shared boundary-aware silent-intent helper with explicit tests for own-line/trailing tokens versus ordinary inline mentions.
