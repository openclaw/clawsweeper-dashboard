## Summary

Keep open: current main and v2026.5.12 still strip a trailing `NO_REPLY` and deliver the preceding text, and live linked implementation PRs are still open rather than merged. The remaining decision is a shared silent-token policy, not cleanup closure.

## What This Changes

The issue requests reply post-processing that suppresses the entire model reply when `NO_REPLY` appears after visible preamble text.

## Best Possible Solution

Define and document one shared silent-token classifier, then land or replace the linked implementation PRs with coverage across normalization, directive/outbound planning, cron/subagent paths, gateway projection/history, and real delivery proof.

## Reproduction Assessment

Yes, source-level. Current `normalizeReplyPayload` strips trailing `NO_REPLY` and returns the non-empty preamble, and the current normalizer test asserts that strip-and-deliver behavior; no runtime test was run because this review is read-only.

## Solution Assessment

Unclear as requested. A broad contains-based post-process would stop this leak but can suppress legitimate messages that mention the token, so the safer solution is a maintainer-selected shared classifier and documented policy.
