## Summary

Keep open: current main and v2026.6.6 only partially handle NO_REPLY leakage; broad own-line or suppress-if-present behavior remains a product/security delivery-policy decision, and the linked fork PR that would close this issue is still open and conflicting.

## What This Changes

Request suppressing an entire reply when model output contains or ends with NO_REPLY instead of stripping only the token and delivering preceding text.

## Best Possible Solution

Define one shared silent-token policy for mixed NO_REPLY payloads, then apply it consistently across auto-reply normalization, cron delivery, gateway display/history, streaming, and channel delivery with regression coverage for the approved behavior.

## Reproduction Assessment

Yes, at source level. Current normalizeReplyPayload strips trailing NO_REPLY and returns the remaining text, and the adjacent test asserts text followed by NO_REPLY is delivered; no tests were run because this was a read-only review.

## Solution Assessment

Unclear. A gateway-level guard is plausible for protecting silent heartbeats, but current docs and tests preserve substantive mixed NO_REPLY replies, so maintainers should choose the shared policy before implementation.
