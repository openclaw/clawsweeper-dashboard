## Summary

Current main partially hardens NO_REPLY handling for recognized reasoning and explicit silent-intent payloads, but it intentionally continues delivering ordinary status or substantive text before a trailing token. The reported heartbeat-summary case therefore remains unresolved and needs a product decision that balances privacy against false suppression.

## What This Changes

Suppress the complete response when visible text is followed by a standalone NO_REPLY token, rather than delivering the preceding text.

## Best Possible Solution

Define an explicit turn-level silent-intent signal or a narrowly approved heartbeat-summary classification in the shared payload classifier, enforce it before streamed blocks and final delivery, and preserve ordinary substantive trailing-token replies.

## Reproduction Assessment

Yes. Current-main source and tests show that a generic substantive or status preamble followed by standalone NO_REPLY is stripped to the preamble and remains deliverable unless it matches the newer reasoning or silent-intent classifier.

## Solution Assessment

No, the proposed match-anywhere or any-trailing-token rule is not the best fix because it conflicts with an existing false-suppression regression contract. The safer direction is a shared semantic or explicit turn-level signal applied before every delivery path.
