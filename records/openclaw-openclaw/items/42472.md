## Summary

Keep open: current main and the latest release still preserve text before a trailing or mixed `NO_REPLY` in shared reply normalization and outbound delivery, while only cron has a scoped partial hardening. The remaining work is actionable, but it changes security-sensitive silent-reply semantics, overlaps open implementation PRs, and needs maintainer policy review before a fix lands.

## What This Changes

The issue requests suppressing the entire model reply when `NO_REPLY` appears in or at the end of the output, instead of stripping only the token and delivering any preceding text.

## Best Possible Solution

Define and document one shared silent-reply classifier for own-line or trailing silent-token intent, then apply it consistently across reply normalization, outbound payload planning, cron/subagent delivery, gateway display, and tests while preserving non-text payloads.

## Reproduction Assessment

Yes, source-level. Current main strips a trailing mixed `NO_REPLY` and returns the non-empty preamble in `normalizeReplyPayload`, and adjacent tests assert that behavior; no live channel run was performed in this read-only review.

## Solution Assessment

No, not as a raw contains-based post-process. The maintainable path is a shared own-line or trailing-token classifier with explicit compatibility semantics, payload-preservation tests, and consistent use across delivery paths.
