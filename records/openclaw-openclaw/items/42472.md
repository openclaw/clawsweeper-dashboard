## Summary

Keep open. Current main still strip-delivers a non-empty preamble before trailing `NO_REPLY` in the shared reply normalizer, and open PRs already target overlapping fixes, so this should wait for a canonical maintainer policy rather than be closed or split into another repair lane.

## What This Changes

Suppress an entire model reply when `NO_REPLY` signals silent intent so preamble text is not delivered to users.

## Best Possible Solution

Define one shared silent-intent helper with explicit rules for exact tokens, JSON envelopes, glued leading tokens, own-line or trailing control tokens, and ordinary inline mentions, then reuse it across reply normalization, directive parsing, outbound planning, gateway live/history projection, cron, and subagent follow-up paths with focused regressions.

## Reproduction Assessment

Yes. Source and tests provide a high-confidence reproduction path: `normalizeReplyPayload({ text: "File's there. Not urgent.\n\nNO_REPLY" })` is still expected to return a non-null payload containing the preamble.

## Solution Assessment

Unclear as originally suggested. Treating any `/NO_REPLY/` occurrence as silent is too broad; the maintainable fix is a shared, boundary-aware silent-intent predicate applied consistently across delivery surfaces.
