## Summary

Keep open: current main still preserves and delivers preamble text in the core reply normalization path when a response ends with `NO_REPLY`, and the issue is already paired with open implementation PRs that use or discuss closing this work. The remaining decision is which suppression semantics to adopt without regressing the existing mixed-text contract.

## What This Changes

Request to suppress an entire reply when the model emits internal text followed by `NO_REPLY` instead of delivering the preamble.

## Best Possible Solution

Adopt one central silent-token helper and apply it consistently across reply normalization, outbound delivery, cron, and gateway history/streaming paths, with tests that preserve intentional inline mentions while suppressing selected trailing or own-line silent tokens.

## Reproduction Assessment

Yes. Source inspection and existing tests show `normalizeReplyPayload` strips a trailing `NO_REPLY` token but returns the remaining preamble for delivery; no live run is needed to see the current-main path.

## Solution Assessment

No, not as a broad `/NO_REPLY/` contains rule. The maintainable fix should use a shared trailing or own-line detector chosen by maintainers, then cover all delivery paths without suppressing legitimate inline mentions.
