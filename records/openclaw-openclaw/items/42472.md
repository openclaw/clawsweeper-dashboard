## Summary

Keep open: current main and v2026.5.20 still preserve mixed trailing `NO_REPLY` preamble text in shared reply paths, and the remaining fix needs maintainer policy/security review before the issue can close.

## What This Changes

Request suppressing an entire model reply when it contains or ends with `NO_REPLY` so preamble text is not delivered to users.

## Best Possible Solution

Define one documented silent-reply classifier for own-line or trailing silent-token intent, then apply it consistently across normalization, directive parsing, outbound planning, cron/subagent delivery, gateway projection, and display paths with false-positive and non-text payload coverage.

## Reproduction Assessment

Yes, source-level. Current main strips a trailing mixed `NO_REPLY` and returns the non-empty preamble in `normalizeReplyPayload`, and the adjacent test asserts that behavior; no live channel run was performed in this read-only review.

## Solution Assessment

No, not as a raw contains-based post-process. The maintainable path is a shared classifier with explicit own-line/trailing-token semantics, compatibility tests, and payload preservation across delivery paths.
