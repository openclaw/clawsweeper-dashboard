## Summary

Keep open: current main has partial fixes for reasoning-prefixed and cron-specific `NO_REPLY` leaks, but shared reply normalization still intentionally strips and delivers generic substantive text followed by `NO_REPLY`. Broad suppression remains a maintainer security/product decision, and https://github.com/BingqingLyu/openclaw/pull/429 is open but conflicting and unmerged.

## What This Changes

Requested post-processing so an auto-reply containing or ending with `NO_REPLY` suppresses the entire reply instead of delivering the text before the token.

## Best Possible Solution

Define a maintainer-approved shared silent-token classifier that suppresses internal or explicit silent-intent payloads while preserving legitimate substantive replies, then apply it consistently across shared reply normalization and affected delivery paths.

## Reproduction Assessment

Yes. Source inspection shows current main strips a generic trailing `NO_REPLY` and returns the non-empty prefix, and adjacent tests assert that behavior; I did not run a live channel delivery in this read-only review.

## Solution Assessment

No, not as a blanket suppress-if-contains rule. The safer direction is a maintainer-approved classifier that distinguishes internal or silent-intent payloads from legitimate substantive replies before changing shared delivery semantics.
