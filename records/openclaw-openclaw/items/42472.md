## Summary

Keep open: current main still strips a trailing `NO_REPLY` and delivers the non-empty preamble in the shared reply normalizer, while linked implementation PRs are open and the remaining behavior requires a maintainer silent-token policy decision.

## What This Changes

Request that any model reply containing, or at least ending with, `NO_REPLY` suppress the entire visible reply instead of delivering text before the token.

## Best Possible Solution

Define one shared, documented silent-reply classifier and land or replace the linked PRs with coverage across normalization, directive/outbound planning, cron/subagent handling, gateway projection/history, and real delivery proof.

## Reproduction Assessment

Yes, source-level. Current main strips a trailing `NO_REPLY` and returns the non-empty preamble in `normalizeReplyPayload`, and the colocated test asserts that strip-and-deliver behavior; no runtime flow was run in this read-only review.

## Solution Assessment

No as a broad contains-based post-process. The safer maintainable solution is a shared, case-insensitive classifier and documented policy applied consistently across all delivery and display paths while preserving legitimate token mentions, media, and channelData semantics.
