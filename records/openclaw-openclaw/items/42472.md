## Summary

Keep open: current main and the latest release still preserve preamble text before a trailing `NO_REPLY` in the shared reply normalizer, while open implementation PRs have not merged and disagree on the right suppression boundary.

## What This Changes

Request to suppress the entire reply when model output contains or ends with `NO_REPLY`, instead of delivering text that precedes the token.

## Best Possible Solution

Land or replace with one shared, documented silent-token classifier that suppresses internal preambles before delivery while preserving legitimate token mentions and media semantics, with regression coverage across normalizer, outbound, cron, and dispatcher paths.

## Reproduction Assessment

Yes. Source inspection shows `normalizeReplyPayload` turns `summary\n\nNO_REPLY` into a deliverable preamble, and current tests assert that behavior; no runtime test was run because this review is read-only.

## Solution Assessment

Unclear as proposed. A broad contains-based rule can suppress legitimate messages that discuss the token, so maintainers should choose a narrower shared own-line, trailing-token, or reasoning-prefixed policy before landing one implementation.
