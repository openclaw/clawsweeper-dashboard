## Summary

Keep open: current main and the latest tagged release still preserve preamble text before a trailing `NO_REPLY` in shared normalization, while the existing docs/tests show this is a compatibility/security policy boundary rather than a settled narrow bug fix.

## What This Changes

The issue requests suppressing an entire assistant reply when a model emits preamble text plus `NO_REPLY`, instead of stripping only the token and delivering the preamble.

## Best Possible Solution

Define one documented silent-reply classifier for own-line or trailing silent-token intent, then apply it consistently across normalization, directive parsing, outbound planning, cron/subagent delivery, gateway projection, and UI display paths.

## Reproduction Assessment

Yes, source-level. Current main strips a trailing mixed `NO_REPLY` and returns the non-empty preamble in `normalizeReplyPayload`, and the adjacent test asserts that behavior; no live channel run was performed in this read-only review.

## Solution Assessment

No, not as a raw contains-based post-process. The maintainable path is a shared classifier with explicit own-line/trailing-token semantics, compatibility tests, and payload preservation across delivery paths.
