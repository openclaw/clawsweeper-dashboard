## Summary

Keep open: current main and the latest release still preserve text before a trailing `NO_REPLY` in the shared normal reply path, while newer cron-specific hardening only partially covers this leak class. The remaining work is a compatibility/security policy decision with overlapping open implementation work, so it should not be closed as implemented or stale.

## What This Changes

The issue asks OpenClaw to suppress an entire assistant reply when a model includes or appends `NO_REPLY`, instead of stripping the token and delivering preceding text.

## Best Possible Solution

Define one documented silent-reply classifier for own-line or trailing silent-token intent, then apply it consistently across normalization, directive parsing, outbound planning, cron/subagent delivery, gateway projection, and UI display paths.

## Reproduction Assessment

Yes, source-level. Current main strips a trailing mixed `NO_REPLY` and returns the non-empty preamble in `normalizeReplyPayload`, and the adjacent test asserts that behavior; no live channel run was performed in this read-only review.

## Solution Assessment

No, not as a raw contains-based post-process. The maintainable path is a shared classifier with explicit own-line or trailing-token semantics, compatibility tests, and payload preservation across delivery paths.
