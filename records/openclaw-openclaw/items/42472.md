## Summary

Keep open: current main and the latest release still preserve text before a trailing `NO_REPLY` in the shared reply normalization and outbound delivery paths, while cron has only a scoped partial hardening. The remaining work is actionable, but it changes silent-token security and delivery semantics and overlaps open implementation work, so it needs maintainer review rather than cleanup closure.

## What This Changes

Request to suppress an entire assistant response when `NO_REPLY` appears as a trailing or own-line silent marker instead of delivering preceding text.

## Best Possible Solution

Define one documented silent-reply classifier for own-line or trailing silent-token intent, then apply it consistently across the shared reply, outbound, cron/subagent, gateway, and display paths with compatibility tests.

## Reproduction Assessment

Yes, source-level. Current main strips a trailing mixed `NO_REPLY` and returns the non-empty preamble in `normalizeReplyPayload`, and the adjacent test asserts that behavior; no live channel run was performed in this read-only review.

## Solution Assessment

No, not as a raw contains-based post-process. The maintainable path is a shared classifier with explicit own-line or trailing-token semantics, compatibility tests, and consistent use across all delivery paths.
