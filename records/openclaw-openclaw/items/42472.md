## Summary

Keep open: current main and v2026.5.12 still strip a trailing mixed-content `NO_REPLY` and deliver the remaining preamble in shared reply normalization, while broader suppression is already covered by open linked PRs and needs a maintainer policy/security decision.

## What This Changes

Request a post-processing policy that suppresses an entire model reply when `NO_REPLY` appears in or at the end of the output, to avoid partial internal-summary leaks.

## Best Possible Solution

Define one documented, case-insensitive silent-reply classifier and land or replace the linked PRs with coverage across normalization, directive/outbound planning, cron/subagent handling, gateway projection/history, runner handling, and real delivery proof.

## Reproduction Assessment

Yes, source-level: current main strips a trailing mixed `NO_REPLY` and returns the non-empty preamble in `normalizeReplyPayload`, and the adjacent test asserts that strip-and-deliver behavior. I did not run a live channel flow because this review is read-only.

## Solution Assessment

No, not as a broad contains-based post-process. The safer maintainable solution is a shared, documented classifier applied consistently across delivery and display paths while preserving legitimate token mentions, media, and channelData semantics.
