## Summary

Current main has meaningful adjacent hardening—boundary-safe prompt-file reads, a code-safety scanner, governed Skill Workshop writes, and ClawHub security verdict metadata—but it still lacks one shared Markdown-content trust or validation contract across ordinary bootstrap and local skill prompt ingestion. The issue should remain open because the central gap is real, while the proposed rules and enforcement require explicit security and product decisions.

## What This Changes

Add security validation and policy controls for Markdown that becomes model-visible prompt context or is distributed as a skill.

## Best Possible Solution

Define a threat-model-backed shared trust and diagnostic contract for model-visible files, integrate it with existing ClawHub verdict and Skill Workshop surfaces, begin with explicit diagnostics or distribution decisions, and exclude nested-Git caching and broad workspace-write blocking.

## Reproduction Assessment

Yes at source level: current bootstrap and local skill read paths can be traced and do not share the requested Markdown-content policy. This is not a strict bug reproduction because current docs and tests do not promise a forbidden-pattern validator.

## Solution Assessment

No, not as proposed wholesale. A centralized trust and diagnostic contract is the maintainable direction, while blanket HTML-comment rejection, automatic wrapping, nested Git state, and general read/write customization require separate decisions or should be rejected.
