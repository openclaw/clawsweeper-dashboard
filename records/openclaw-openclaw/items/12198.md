## Summary

Keep open: current main and the latest release have adjacent hardening, including boundary-safe reads, Skill Workshop scanning, and ClawHub verification metadata, but they still do not provide one shared Markdown validation contract for ordinary bootstrap files and local model-visible `SKILL.md` ingestion.

## What This Changes

The issue requests Markdown validation or linting for prompt-bearing files such as skill definitions and bootstrap Markdown to reduce prompt-injection risk.

## Best Possible Solution

Define a threat-model-backed shared trust and diagnostic contract for model-visible Markdown, then apply it consistently to prompt ingestion, skill loading, Skill Workshop, ClawHub install/update, and publish-side verdicts while rejecting broad workspace-write blocking and nested Git state unless maintainers explicitly choose them.

## Reproduction Assessment

Yes at source level: current bootstrap and local skill paths can be traced and do not share the requested Markdown-content policy. This is not a strict bug reproduction because current docs and tests do not promise a forbidden-pattern validator.

## Solution Assessment

No, not as proposed wholesale. A centralized trust and diagnostics contract is the maintainable path, while blanket HTML-comment rejection, automatic wrapping, nested Git tracking, and general workspace-write interception need separate decisions or rejection.
