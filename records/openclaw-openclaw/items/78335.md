## Summary

Keep open: the PR targets a plausible plugin-contract feature, but the current diff is not merge-ready because concrete `tools.allow` entries do not select wildcard contract entries, wildcard ownership limits are undefined, and the supplied proof is only test output.

## What This Changes

The PR adds glob matching for plugin `contracts.tools` entries in runtime enforcement and cold-load selection, updates manifest/docs text, and adds a helper-level contract test.

## Best Possible Solution

Specify bounded wildcard contract semantics and implement them end-to-end across cold-load selection, allow/deny filtering, availability metadata, undeclared-tool diagnostics, docs, changelog, tests, and live proof.

## Reproduction Assessment

Yes for the review finding: source inspection of the patched selection path shows `contracts.tools: ["xxx-*"]` with `tools.allow: ["xxx-test"]` leaves no selected tool name. This is a PR-path reproduction, not a current-main bug reproduction.

## Solution Assessment

No: direct glob matching in the existing string list is incomplete because it does not carry pattern semantics through concrete allowlists, denylists, metadata availability, and diagnostics. A safer solution is to define the contract first, then add integration coverage around a real dynamic plugin path.
