## Summary

Keep open: the latest PR still leaves a concrete allowlist/runtime-selection gap, the external proof is test-only, and wildcard ownership semantics need maintainer policy before merge.

## What This Changes

The PR adds glob matching for plugin `contracts.tools`, applies it to contract enforcement and plugin runtime selection, documents the manifest behavior, and adds a contract test around the shared glob helper.

## Best Possible Solution

Define bounded wildcard contract semantics and implement them consistently across contract enforcement, cold-load selection, allow/deny filtering, availability metadata, diagnostics, docs, changelog, and live dynamic-plugin proof.

## Reproduction Assessment

Yes for the PR-path finding: source inspection of latest PR head `e383224e157980cb6b902f8fc6b18da6eff25334` shows `contracts.tools: ["xxx-*"]` with `tools.allow: ["xxx-test"]` still leaves no selected manifest tool. Current main has no bug reproduction because the linked report is a feature request for new manifest semantics.

## Solution Assessment

No: the patch adds helper matching but does not make concrete allowlists select pattern-owned plugin runtimes, and it leaves broad wildcard policy, changelog, and live proof unresolved. A safer solution should define explicit semantics and cover the real runtime path.
