## Summary

Keep open: the feature is plausible, but this patch is not mergeable because pattern-owned plugins still do not load under concrete restrictive tool allowlists, the broad wildcard ownership semantics need maintainer review, and the supplied proof is test output rather than a live dynamic-plugin run.

## What This Changes

The branch updates plugin manifest/tool contract matching to treat `contracts.tools` entries as glob patterns, with docs and a helper-level test.

## Best Possible Solution

Define additive wildcard contract semantics that preserve manifest-first discovery, restrictive allowlists, denylists, tool metadata availability, undeclared-tool diagnostics, and docs/tests for dynamic plugin tools.

## Reproduction Assessment

Yes for the review finding: source inspection shows `contracts.tools: ["xxx-*"]` with `tools.allow: ["xxx-test"]` leaves no selected manifest tool name, so the plugin is skipped before it can register the concrete tool. This is a patch-path reproduction, not a current-main bug reproduction.

## Solution Assessment

No: directly adding glob matching to the existing string list is incomplete because it does not carry pattern semantics through cold-load selection, optional-tool filtering, metadata availability, and security policy. A safer solution is to specify the contract behavior first, then add integration tests that exercise a real plugin manifest plus runtime registration path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Match concrete allowlists against contract patterns:** `src/plugins/tools.ts:372-374`
  - body: With `contracts.tools: ["xxx-*"]` and a restrictive `tools.allow: ["xxx-test"]`, this code iterates over the manifest entry `xxx-*` and checks it against the allowlist patterns, so no selected tool name survives and the owning plugin never cold-loads. That breaks the concrete allowlist path documented for plugin-owned tools, which is especially important for the dynamic tools this PR is enabling.
  - confidence: 0.87

## Security Review

Status: needs_attention

Summary: The patch changes a tool-ownership enforcement boundary and documents all-match wildcard contracts without defining limits.

Concerns:

- **[medium] Define limits for all-match tool contracts:** `src/plugins/tool-contracts.ts:31`
  - body: `findUndeclaredPluginToolNames` now accepts any runtime tool name when a manifest declares `contracts.tools: ["*"]`, which removes the current exact static ownership check for that plugin. Maintainers should decide whether all-match contracts are allowed, or require bounded patterns and explicit diagnostics before merging.
  - confidence: 0.78
