## Summary

Keep open: the feature is plausible, but the branch is incomplete because restrictive `tools.allow` still will not cold-load a concrete dynamic tool, the required changelog entry is missing, and the external proof is test output rather than live plugin behavior. Wildcard ownership semantics also need maintainer/security review before merge.

## What This Changes

The PR adds glob matching for plugin `contracts.tools` declarations in contract enforcement and runtime plugin selection, updates plugin manifest docs/types, and adds a contract test around glob helper behavior.

## Best Possible Solution

Define bounded wildcard contract semantics and implement them end-to-end across contract validation, cold-load selection, allow/deny filtering, cached descriptor availability, diagnostics, docs, changelog, and live dynamic-plugin proof.

## Reproduction Assessment

Yes for the PR-path finding: source inspection shows `contracts.tools: ["xxx-*"]` with `tools.allow: ["xxx-test"]` does not select the plugin. No current-main bug reproduction applies because the linked report asks for a new manifest capability.

## Solution Assessment

No: matching helper patterns is not enough because the concrete allowlist and cold-load path still operate on manifest pattern strings. A safer solution should define the contract semantics first and cover a real dynamic plugin under both default and restrictive tool policy.
