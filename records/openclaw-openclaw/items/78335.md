## Summary

Keep open: the latest PR head still does not implement wildcard `contracts.tools` end-to-end under restrictive `tools.allow`, lacks live runtime proof, and broad wildcard ownership semantics need maintainer review.

## What This Changes

The PR adds glob matching for plugin `contracts.tools`, changes contract enforcement and runtime plugin selection, updates manifest docs/types, and adds a contract test around the glob helper.

## Best Possible Solution

Define bounded wildcard contract semantics and implement them across contract validation, cold-load selection, allow/deny filtering, cached descriptor availability, diagnostics, docs, changelog, and live dynamic-plugin proof.

## Reproduction Assessment

Yes for the PR-path finding: source inspection of PR head `8434678f7e9cdfb9866dd39e78c65425dcdc8bd9` shows `contracts.tools: ["xxx-*"]` with `tools.allow: ["xxx-test"]` still does not select the plugin. No current-main bug reproduction applies because the linked report asks for a new manifest capability.

## Solution Assessment

No: matching helper patterns is not enough because restrictive allowlists and cold-load selection still operate on manifest pattern strings. A safer solution should define bounded contract semantics first and cover a real dynamic plugin under both default and restrictive tool policy.
