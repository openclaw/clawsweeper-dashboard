- `P2`: Long-running bundled MCP tools can fail after the SDK default timeout, but the blast radius is limited to configured MCP tool execution and active fix PRs already exist.

## Summary

Keep open. Current main and v2026.5.18 still call bundled MCP tools without MCP `RequestOptions.timeout`, and an open linked implementation PR references this issue with closing syntax, so the issue should remain open until the timeout contract is reviewed and landed or the PR is closed.

## What This Changes

Allow bundled MCP tool calls configured through `mcp.servers` to use a configurable per-request timeout instead of always inheriting the MCP SDK 60-second default.

## Best Possible Solution

Review and land or replace the active timeout PRs with one focused patch that documents and tests bundled-MCP request-timeout behavior across config, schema, docs, and runtime, with real proof from a long-running MCP tool.

## Reproduction Assessment

Yes, at source level. Configure a bundled MCP tool that runs longer than 60 seconds and current main reaches `Client.callTool` without `RequestOptions.timeout`, so SDK 1.29.0 applies its 60-second default.

## Solution Assessment

Unclear on the exact public contract. Passing `RequestOptions.timeout` into `Client.callTool` is the narrow runtime path, but the field name, default, and progress-reset semantics need maintainer choice across the open PRs.
