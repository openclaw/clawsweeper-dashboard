## Summary

Keep open. Current main and the latest release still omit MCP per-request timeout options for bundled MCP tool calls, and open PR #73536 is already the linked implementation candidate with closing syntax.

## What This Changes

Bundled MCP tool calls should honor a configurable per-server request timeout instead of always falling back to the MCP SDK's 60-second default.

## Best Possible Solution

Review and land or revise #73536 so bundled MCP resolves a per-server request timeout, passes it as Client.callTool RequestOptions.timeout, and updates tests/docs/changelog for the chosen config semantics.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: configure a bundled MCP server with a timeout above 60 seconds and call a tool that runs longer than 60 seconds; current main forwards no RequestOptions.timeout, so the SDK default still applies.

## Solution Assessment

Unclear on the exact config naming, but the narrow maintainable fix is clear: propagate a validated per-server request timeout into the third Client.callTool argument and cover that behavior with focused tests/docs.
