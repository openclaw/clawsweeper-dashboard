## Summary

Keep open. Current main and the latest tagged release still omit MCP per-request timeout options for bundled MCP tool calls, and open PR #73536 is already the linked implementation candidate with closing syntax.

## What This Changes

The issue requests configurable bundled MCP tool-call timeouts so slow MCP tools are not capped by the MCP SDK's 60-second default.

## Best Possible Solution

Land a focused fix that resolves a per-server MCP request timeout, passes it as the third `Client.callTool` argument, and updates tests, docs, and changelog for the chosen config semantics.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: configure a bundled MCP server with a timeout above 60 seconds and call a tool that runs longer than 60 seconds; current main forwards no MCP request timeout, so the SDK default still applies.

## Solution Assessment

No for current main. The narrow maintainable path is the linked PR's direction, with maintainer review on whether connection and request timeout semantics should share one config knob or split into separate fields.
