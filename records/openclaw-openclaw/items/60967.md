## Summary

Keep open. Current main and the latest release still omit MCP per-request timeout options for bundled MCP tool execution, and an open linked implementation PR already targets this issue with closing syntax.

## What This Changes

The issue requests configurable bundled MCP tool-call request timeouts so long-running tools are not capped by the MCP SDK's 60-second default.

## Best Possible Solution

Review and land or replace https://github.com/openclaw/openclaw/pull/73536 with a focused patch that plumbs an explicit bundle MCP request-timeout policy through config, runtime, tests, docs, and changelog.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: configure a bundled MCP server whose tool runs longer than 60 seconds, then execute it through the materialized bundle MCP tool path; current main reaches Client.callTool without RequestOptions.timeout, so SDK 1.29.0 applies its 60-second default.

## Solution Assessment

Unclear on the exact public API. The narrow implementation direction is clear, but maintainers should choose whether the request timeout is a new requestTimeoutMs field or documented inheritance from connectionTimeoutMs, and whether progress notifications should reset the timer.
