## Summary

Keep open. Current main and the latest release still omit MCP per-request timeout options for bundled MCP tool execution, and an open linked implementation PR already targets this issue with closing syntax.

## What This Changes

Bundle MCP tool calls should be configurable past the MCP SDK's 60-second default instead of timing out during long-running `mcp.servers` tool execution.

## Best Possible Solution

Review and land or replace the active timeout PRs with one focused patch that documents and tests bundle MCP request-timeout behavior across config/schema/docs/runtime, with real proof from a long-running MCP tool.

## Reproduction Assessment

Yes. A high-confidence source-level path exists: configure a bundled MCP server with a tool that runs longer than 60 seconds, execute it through materialized bundle MCP tools, and current main reaches `Client.callTool` without `RequestOptions.timeout`, so SDK 1.29.0 applies its 60-second default.

## Solution Assessment

Unclear on the exact public API. Passing `RequestOptions.timeout` into `Client.callTool` is the narrow technical fix, but the config field, default, and progress-reset semantics need maintainer choice before this should close.
