## Summary

Keep open. Current main and the latest release still call bundled MCP tools without forwarding MCP request timeout options, while the pinned MCP SDK defaults requests to 60,000 ms. Open PR #73536 is an unmerged implementation candidate with closing syntax, so this issue should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue requests configurable bundled MCP tool-call request timeouts so long-running MCP tools are not capped by the MCP SDK default.

## Best Possible Solution

Keep the issue open and review PR #73536 as the canonical implementation candidate. The desired end state is for bundle MCP sessions to resolve a per-server request timeout, pass it to `Client.callTool(..., undefined, { timeout })`, and add focused tests/docs for the chosen config semantics.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction path exists: configure a bundled MCP server with a timeout above 60 seconds and call a tool that runs longer than 60 seconds; current main forwards no MCP `RequestOptions.timeout`, so the SDK default still applies.

## Solution Assessment

No for current main. The narrow maintainable fix is to pass a resolved per-server timeout into the MCP SDK call path, with maintainer review on whether that timeout should reuse `connectionTimeoutMs` or be a separate public config field.
