## Summary

Keep open. Current main and the latest release still call bundled MCP tools without MCP `RequestOptions.timeout`, and open PR https://github.com/openclaw/openclaw/pull/73536 references this issue with closing syntax, so this issue should stay open until the timeout contract is reconciled and landed or the PR is closed.

## What This Changes

Request configurable bundled-MCP tool-call timeouts so long-running MCP tools are not capped by the MCP SDK's 60-second default.

## Best Possible Solution

Review and land one reconciled patch that schema-documents bundled-MCP request-timeout behavior, passes correct SDK request options, adds focused regression coverage, and includes real proof from a long-running MCP tool.

## Reproduction Assessment

Yes, at source level. Current main materializes bundled MCP tools through `Client.callTool` without `RequestOptions.timeout`, and SDK v1.29.0 defaults omitted request timeouts to 60000 ms.

## Solution Assessment

Unclear on the exact public contract. Passing `RequestOptions.timeout` into `Client.callTool` is the narrow runtime path, but the field name, default, and progress-reset semantics need maintainer choice across the open PRs.
