## Summary

Keep open. Current main and the latest release still call bundled MCP tools without MCP `RequestOptions.timeout`, and open implementation PRs already target this problem but need timeout-contract review before the issue can close.

## What This Changes

The issue requests configurable bundled-MCP tool-call timeouts so long-running `mcp.servers` tools are not capped by the MCP SDK's 60-second default.

## Best Possible Solution

Land one reconciled patch that documents a per-server bundled-MCP request-timeout contract, passes correct SDK request options, adds focused regression coverage, and includes redacted proof from a long-running MCP tool.

## Reproduction Assessment

Yes, at source level. Configure a bundled MCP tool that runs longer than 60 seconds and current main reaches `Client.callTool` without `RequestOptions.timeout`, so the pinned MCP SDK applies its default request timeout.

## Solution Assessment

Unclear on the exact public contract. Passing MCP `RequestOptions.timeout` is the narrow runtime path, but the field name, default, and progress-reset semantics need maintainer choice across the open PRs.
