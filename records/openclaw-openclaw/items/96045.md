## Summary

Keep open. The revised PR fixes the earlier legacy-recovery blocker and now has sufficient redacted Gateway proof, but the new service validator would reject the existing Control UI default cron payload because the general form defaults to a recurring schedule with `deleteAfterRun: true` and always submits that field.

## What This Changes

The PR adds cron service validation and tests to reject recurring cron jobs with `deleteAfterRun: true` while allowing legacy invalid jobs to be renamed, disabled, or cleared.

## Best Possible Solution

Decide the one-shot-only cron contract, then align the Control UI and user-facing help/schema expectations so recurring jobs cannot submit `deleteAfterRun: true` by default while preserving legacy recovery paths.

## Reproduction Assessment

No, not yet. Service-level rejection is a plausible owner-boundary fix, but it is not the best merge shape until caller surfaces are aligned and maintainers choose it over the competing recurring-cleanup/default direction.
