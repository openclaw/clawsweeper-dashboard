## Summary

Keep open. The strict-validation direction is plausible, but the patch currently makes unrelated updates to pre-existing recurring jobs with `deleteAfterRun: true` fail unless the update also clears the flag, and the external PR still has only test/CI evidence rather than real behavior proof.

The PR adds cron service validation that rejects `deleteAfterRun: true` unless the schedule is `kind: "at"`, with gateway cron tests for create and update rejection.

## Best Possible Solution

If maintainers choose the one-shot-only contract, keep service-level rejection but preserve safe recovery paths for already-persisted invalid jobs and require real create/update proof before merge.
