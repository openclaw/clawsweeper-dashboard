## Label Justifications

- `P1`: The underlying cron behavior can leave intended one-shot delete-after-run jobs running repeatedly, and this PR changes that user-facing scheduling contract.
- `merge-risk: 🚨 compatibility`: The PR rejects a previously accepted recurring cron shape and changes the Gateway/API behavior clients see during creates and updates.

## Summary

This PR should stay open: it now has live Gateway proof and compatibility-aware legacy recovery coverage, but the new `cron.add` service validation error is not classified as a Gateway invalid request, so invalid recurring creates can surface as a handler failure instead of the promised validation response.

## What This Changes

The PR adds cron service validation and tests to reject `deleteAfterRun: true` on recurring schedules while allowing already-persisted invalid jobs to be renamed, disabled, cleared, or re-enabled after they become valid.

## Best Possible Solution

Land the one-shot-only contract only after `cron.add` maps this validation to INVALID_REQUEST, tests assert that response code, and maintainers accept rejecting recurring `deleteAfterRun` instead of honoring it.

## Reproduction Assessment

Yes, source-level. Current main accepts explicit `deleteAfterRun` on recurring schedules while the timer only deletes `at` jobs after success; I did not run a live repro because this was a read-only review.

## Solution Assessment

No, not yet. Service-level rejection is a maintainable solution if maintainers choose the one-shot-only contract, but the Gateway create error must be returned as INVALID_REQUEST before the patch is merge-ready.
