## Summary

Keep open. The reporter supplied concrete 2026.5.7 timing/profiling plus local before/after proof, and current main still has the same source-backed status-summary paths that can load plugin metadata during shallow model/runtime labeling.

## What This Changes

The issue requests keeping default non-deep `openclaw status` on a bounded read-only path instead of repeatedly loading plugin metadata while rendering session summaries.

## Best Possible Solution

Patch the status summary path to do manifest-free persisted model parsing, cache or inject CLI-provider labeling per render, and reuse already-built session rows while leaving broader installed-index caching to the related plugin-index issue.

## Repair Work Prompt

Fix the default/non-deep `openclaw status` session-summary path so it does not repeatedly load plugin metadata while labeling session model/runtime. Scope: thread optional `allowManifestNormalization?: boolean` through `resolvePersistedOverrideModelRef`, `resolvePersistedModelRef`, and `resolvePersistedSelectedModelRef`; pass `allowManifestNormalization: false` from `src/commands/status.summary.runtime.ts`; ensure status-only raw model parsing also disables manifest normalization; avoid repeated `isCliProvider` metadata discovery during one status summary render by injecting or caching a status-scope CLI-provider resolver; reuse the per-agent `buildSessionRows` results for `sessions.recent`/`allSessions` rather than rebuilding every store. Add regression coverage that fails if status summary/model parsing calls `loadPluginMetadataSnapshot` for shallow session labeling and that covers the row-reuse behavior. Keep https://github.com/openclaw/openclaw/issues/80345 broader installed-index caching out of scope and do not add a wall-clock TTL metadata cache. Because this is a user-facing performance fix, include a `CHANGELOG.md` entry when shipping.
