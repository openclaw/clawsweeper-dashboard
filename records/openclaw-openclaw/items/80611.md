## Summary

Keep open: current main still leaves plain text `openclaw status` on a session-summary path that can re-enter manifest-backed model normalization and per-row CLI-provider discovery, while the docs promise plain status stays fast and read-only. The closed related report only covered the lean `status --json` path, so this default text-path bug remains actionable.

## What This Changes

Plain text `openclaw status` should avoid repeated plugin metadata/model normalization while rendering session summaries, keeping expensive probes behind `--deep` or `--all`.

## Best Possible Solution

Land a narrow status-summary performance fix that keeps default text status on a shallow read-only path without changing `--deep`, `--all`, or the broader installed-index architecture.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not from this checkout alone: the issue supplies concrete timing commands and profiling, but I did not run them against a matching v2026.5.7 npm/WSL2 install. Source inspection does confirm the reported shallow-status call path is still present on current main.

## Solution Assessment

Is this the best way to solve the issue? Yes, the narrowest maintainable direction is to keep status summary parsing/runtime labeling on a cheap path and reuse already-built rows; broad installed-index caching belongs to the separate performance thread unless needed independently.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/80611 by keeping plain/default text `openclaw status` session summary rendering on a shallow read-only path. Scope the fix to status summary/model parsing/runtime labeling: thread an optional `allowManifestNormalization` through `resolvePersistedOverrideModelRef`, `resolvePersistedModelRef`, and `resolvePersistedSelectedModelRef`; pass `allowManifestNormalization: false` from `status.summary.runtime.ts`; avoid repeated `isCliProvider` discovery during one status summary render, preferably via a status-scoped cache or injected resolver; and reuse the per-agent `buildSessionRows` results instead of rebuilding every store for `allSessions`. Do not solve this by adding broad process-global installed-index caching, which belongs to https://github.com/openclaw/openclaw/issues/80345, and do not change `--deep` or `--all` semantics. Add focused regression tests proving shallow status does not trigger manifest metadata loading through persisted model parsing and does not rebuild rows unnecessarily. Add a `CHANGELOG.md` fix entry because this is user-visible CLI performance. Validation should include the targeted status tests plus a before/after timing command on a source or packaged setup close to the reported environment, with private paths/secrets redacted.
