## Summary

Keep open: the report is a real, source-backed CLI performance bug and is paired with open implementation PRs, including https://github.com/openclaw/openclaw/pull/80614 with closing syntax. Current main still shows the default text status summary can enter manifest-backed model normalization and per-row CLI-provider discovery, so this should stay open until the linked fix path lands or maintainers choose a different canonical repair.

## What This Changes

Plain non-deep `openclaw status` is reported to spend about 20 seconds repeatedly loading plugin/model metadata while rendering the session summary.

## Best Possible Solution

Review and land a narrow status-summary fix that disables manifest normalization for status-only persisted model parsing, memoizes CLI-provider facts within one status scan, and preserves correct global session aggregation; keep the broader gateway metadata work on https://github.com/openclaw/openclaw/issues/78461.

## Reproduction Assessment

Unclear for a current-main timing reproduction: I did not run `/usr/bin/time openclaw status` in the reported 2026.5.7/WSL2 setup, and current main has later metadata memoization. Source inspection still confirms the status-summary model/runtime paths described in the report, and linked PRs provide before/after proof on affected installs.

## Solution Assessment

Yes on direction, but not as a close decision: the narrow status-summary changes in the linked implementation path are the maintainable fix. Closing this issue before the open linked PRs merge would lose the issue/PR pairing and the reporter's default text-status repro.
