## Summary

Keep open: this is a valid, source-backed CLI performance bug and it is still paired with open implementation PRs, including a draft PR with closing syntax. Current main still has the status-summary paths that can enter manifest-backed model normalization and per-row CLI-provider discovery, so closing before the linked fixes land would lose the issue/PR pairing.

## What This Changes

The issue reports that plain non-deep `openclaw status` on OpenClaw 2026.5.7 spends about 20 seconds repeatedly loading plugin/model metadata while rendering the session summary.

## Best Possible Solution

Land a narrow status-summary fix that disables manifest normalization for status-only persisted model parsing, precomputes CLI-provider facts within one status scan, and preserves correct global session aggregation; keep the broader daemon/plugin metadata family on https://github.com/openclaw/openclaw/issues/78461.

## Reproduction Assessment

No, not as a live current-main timing reproduction. The 2026.5.7 report and linked PRs provide real before/after proof, and source inspection on current main still confirms the relevant status-summary model/runtime paths.

## Solution Assessment

Yes on direction, but not as a close decision. The maintainable fix is the narrow status-summary work already represented by the linked PRs, with the shared-store aggregation regression in the draft PR repaired before landing.
