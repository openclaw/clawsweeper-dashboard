## Summary

Keep open: current main still returns count-only primary MCP `content` for the reported tools, and an open same-repo PR already uses closing syntax for this issue, so the issue should stay open until that fix is reviewed and merged or closed.

## What This Changes

The issue asks for MCP `messages_read` and `conversations_list` to expose usable message/conversation data in primary tool `content`, not only in `structuredContent`.

## Best Possible Solution

Review and land one focused MCP formatting fix, preferably the same-repo PR #77041 or a narrower replacement, with regression coverage for primary `content` on `conversations_list` and `messages_read`.

## Reproduction Assessment

Yes, at source level: any successful `conversations_list` or `messages_read` call with data returns primary text generated by `summarizeResult`, which only contains a count. I did not mark this fully reproduced because the live Claude.ai/supergateway path was not run in this read-only checkout.

## Solution Assessment

Yes, the narrowest maintainable solution is to change MCP channel result formatting so the primary `content` includes serialized usable data while preserving `structuredContent` for clients that consume it. Gateway session lookup and transcript storage do not need to change.
