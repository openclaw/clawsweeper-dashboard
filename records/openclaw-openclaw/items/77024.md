## Summary

Keep open: current main still returns count-only primary `content` for the reported MCP tools while placing the usable payload under `structuredContent`, which matches the report and conflicts with the documented generic MCP use case.

## What This Changes

The issue requests that `messages_read` and `conversations_list` expose full conversation/message data in primary MCP tool content instead of only in `structuredContent`.

## Best Possible Solution

Keep the issue open and fix the MCP channel tool result formatting so primary `content` carries a serialized usable payload while preserving `structuredContent` for clients that consume it.

## Reproduction Assessment

Yes, at source level: any successful `messages_read` or `conversations_list` call with data currently returns primary text from `summarizeResult`, which is only a count. I did not mark this fully reproduced because I did not run a live MCP client path in this read-only checkout.

## Solution Assessment

Yes, the narrowest maintainable fix is in the MCP channel tool response formatting, ideally with a shared helper and focused regression coverage. Gateway session listing/history behavior should not need to change.
