## Summary

Keep open. Current main still exposes only the narrower `tools.fs.workspaceOnly` boolean, while the requested allow/deny path policy and exec filesystem boundary remain unimplemented; the active PathGuard PR is related but open, conflicting, and explicitly leaves arbitrary shell execution out of scope.

## What This Changes

The issue requests a configurable filesystem access policy with sandbox/full mode plus allowed and denied paths for read, write, exec, and related tools.

## Best Possible Solution

Define a reviewed `tools.fs` or successor ACL contract with deny-wins path rules, canonical path handling, media/apply_patch/subagent coverage, and an explicit exec strategy, then land a fixed PathGuard-style implementation or split exec into a tracked follow-up.

## Reproduction Assessment

Yes for the missing capability. Source inspection shows current main only accepts and propagates `tools.fs.workspaceOnly`, while shell execution is built without a filesystem policy input.

## Solution Assessment

Unclear for the exact `tools.fileAccess` proposal. A deny-wins allow/deny policy is directionally appropriate, but the maintainable solution likely belongs under `tools.fs` or a successor contract after maintainers settle exec semantics.
