## Summary

Keep open. Current main still exposes only the binary `tools.fs.workspaceOnly` filesystem guard, while the request is for a configurable allow/deny path policy with explicit exec semantics; the related PathGuard PR is open, partial, and does not close this issue.

## What This Changes

The issue requests filesystem access configuration with sandbox/full mode, `allowedPaths`, `denyPaths`, denial errors, and coverage for read, write, and exec tools.

## Best Possible Solution

Define a versioned filesystem ACL under `tools.fs` or a successor contract, enforce deny-wins path rules across file/media/patch surfaces, and explicitly settle exec as sandboxed, out of scope, or separately tracked before landing or replacing the PathGuard work.

## Reproduction Assessment

Yes. Source inspection on current main shows strict config, public types, runtime policy, tests, and docs only model `tools.fs.workspaceOnly`; shell execution is constructed without filesystem policy input.

## Solution Assessment

Unclear for the exact `tools.fileAccess` shape. A deny-wins allow/deny policy is directionally appropriate, but the maintainable solution likely belongs under `tools.fs` or a successor contract after exec semantics are decided.
