## Summary

Current main and the latest release still provide only the coarse `tools.fs.workspaceOnly` boundary, while the requested deny-wins allow/deny policy and host-shell coverage remain unimplemented. The linked broad implementation PRs are still open, conflicting, propose different contracts, and explicitly leave arbitrary `exec` filesystem access out of scope, so this issue remains the appropriate canonical product/security discussion.

## What This Changes

Add configurable, per-path filesystem allow/deny enforcement for agent file operations and define corresponding protection for shell execution.

## Best Possible Solution

Select one canonical `tools.fs` policy contract with deny-wins rules, per-agent narrowing, robust cross-platform canonicalization, and an explicitly documented shell threat model; then land one focused implementation with bypass tests, upgrade/default compatibility proof, and real host-environment validation.

## Reproduction Assessment

Yes at the source-contract level: current main’s strict schema rejects the requested path-policy keys, and its runtime policy carries only `workspaceOnly`. That is sufficient to reproduce the reported configuration gap without executing an agent against sensitive paths.

## Solution Assessment

No, the original `tools.fileAccess` sketch is not yet established as the best permanent solution. The maintainable path is a single canonical `tools.fs` contract coupled to an honest decision about whether shell access is isolated by containers/OS controls or remains outside the guarantee.
