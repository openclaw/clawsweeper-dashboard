## Summary

Keep open. Current main still only implements the binary `tools.fs.workspaceOnly` guard, while this request asks for a new allow/deny filesystem policy plus explicit exec semantics; the active PathGuard work at https://github.com/openclaw/openclaw/pull/60981 is relevant but unmerged and explicitly leaves arbitrary shell execution out of scope.

## What This Changes

The issue requests a configurable filesystem access policy with sandbox/full mode, allowed paths, deny paths, and coverage across file and exec tools.

## Best Possible Solution

Define a versioned filesystem ACL under `tools.fs` or a successor surface, enforce deny-wins path rules across file/media/patch surfaces, and explicitly decide whether exec is sandboxed, out of scope, or tracked separately.

## Reproduction Assessment

Yes, by source inspection for the missing capability. Current main’s strict schema, public type, runtime policy, and tests only model `tools.fs.workspaceOnly`, and shell execution is constructed without filesystem policy input.

## Solution Assessment

Unclear for the exact `tools.fileAccess` shape. A deny-wins allow/deny policy is directionally appropriate, but the maintainable solution likely belongs under `tools.fs` or a successor contract after exec semantics are settled.
