## Summary

Keep open. Current main still has only the narrower `tools.fs.workspaceOnly` boolean and `apply_patch` workspace guard; allow/deny path config and filesystem policy for shell execution remain absent, while related implementation PRs are still open and incomplete for the full request.

## What This Changes

Add configurable filesystem access restrictions with allowed/deny paths and a sandbox/full mode enforced across filesystem and exec tools.

## Best Possible Solution

Define a versioned `tools.fs` or successor policy with deny-wins path rules for file/media/patch surfaces and an explicit exec strategy, then land a reviewed implementation or split exec into a tracked follow-up.

## Reproduction Assessment

Yes for the missing capability. Source inspection shows the strict config schema and public type only accept `tools.fs.workspaceOnly`, while shell execution is constructed without a filesystem policy input.

## Solution Assessment

Unclear for the exact `tools.fileAccess` shape. A deny-wins allow/deny policy is directionally appropriate, but the maintainable contract likely belongs under `tools.fs` or a successor surface after exec semantics are settled.
