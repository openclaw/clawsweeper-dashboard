## Summary

Keep open. Current main and v2026.6.8 still expose only the narrower `tools.fs.workspaceOnly` boolean, while the requested allow/deny path policy and exec coverage remain unmerged security/product work.

## What This Changes

The issue requests a configurable filesystem sandbox policy with mode, allowed paths, denied paths, and enforcement across file and exec tools.

## Best Possible Solution

Define one canonical `tools.fs` policy contract with path canonicalization, deny precedence, file/media/apply_patch coverage, and an explicit exec strategy, then land one reviewed implementation path or split exec into a tracked follow-up.

## Reproduction Assessment

Yes, by source inspection: the strict config schema accepts only `tools.fs.workspaceOnly`, and host file tools still allow outside-workspace access when that boolean is unset or false. I did not run a live agent attempt because this was a read-only review.

## Solution Assessment

No, not as a direct `tools.fileAccess` key addition. The best solution needs maintainer-approved policy semantics for canonicalization, inheritance, media/apply_patch coverage, and whether exec can be enforced or must be explicitly out of scope.
