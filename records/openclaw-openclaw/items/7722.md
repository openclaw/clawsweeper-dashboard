## Summary

Keep open. Current main and the latest release still only expose the narrower `tools.fs.workspaceOnly` guard, while the requested configurable allow/deny filesystem policy and exec strategy remain unsettled; the open related PRs are useful implementation candidates but are not merged or canonical yet.

## What This Changes

The issue requests a first-class filesystem access policy config for agents, including allow paths, deny paths, sandbox/full modes, and enforcement across filesystem and exec-style tools.

## Best Possible Solution

Keep this as the canonical filesystem ACL tracker until maintainers choose one config contract and land enforcement for read, write, edit, apply_patch, local media, and an explicit exec strategy or tracked exec follow-up.

## Reproduction Assessment

Yes by source inspection. The current strict config schema accepts only `tools.fs.workspaceOnly`, and the runtime policy only carries that boolean, so the reported `tools.fileAccess` and requested path rules remain absent on current main.

## Solution Assessment

Unclear for the exact `tools.fileAccess` shape. A maintainer-approved canonical policy under `tools.fs` or a successor config contract is the better end state than adding a parallel one-off config key.
