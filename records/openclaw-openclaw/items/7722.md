## Summary

Keep open. Current main and the latest release still expose only the narrower `tools.fs.workspaceOnly` boolean, while the requested allow/deny filesystem policy and exec-wide boundary remain unimplemented and are still under active, security-sensitive PR review.

## What This Changes

The issue requests a configurable filesystem access policy with allow/deny paths, or an equivalent `tools.fs` successor, enforced across file and exec tools.

## Best Possible Solution

Keep this as the canonical filesystem ACL tracker until an owner-reviewed, deny-wins path policy lands across filesystem tools and local-file surfaces, with exec either covered by a real boundary or explicitly split into a tracked follow-up.

## Reproduction Assessment

Yes. Source inspection shows the strict schema, public type, runtime policy, docs, and tests only support `tools.fs.workspaceOnly`, so the requested path ACL cannot currently be expressed or enforced.

## Solution Assessment

No for closing or queueing a quick fix. The maintainable path is a security/product-reviewed filesystem ACL contract, likely by narrowing or replacing the active PathGuard/filesystem-roots work and explicitly deciding the exec boundary.
