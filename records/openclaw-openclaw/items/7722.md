## Summary

Keep open. Current main still exposes only `tools.fs.workspaceOnly`, rejects the requested allow/deny path policy surface, and does not apply a filesystem ACL to arbitrary `exec`; recent comments add concrete operator context and the open PathGuard PR is relevant but unmerged and explicitly partial.

## What This Changes

Request a configurable filesystem sandbox policy with mode, allowed paths, and deny paths enforced across file operations and exec-like access.

## Best Possible Solution

Keep this as the canonical filesystem ACL tracker until OpenClaw lands a reviewed `tools.fs` policy with deny-wins allow/deny rules, covered file/media/apply_patch behavior, and either a designed exec boundary or a linked exec follow-up.

## Reproduction Assessment

Yes. Source inspection shows the strict schema, public type, runtime policy, and tests only support `tools.fs.workspaceOnly`, so the requested path policy cannot currently be expressed or enforced.

## Solution Assessment

No for closing or queueing a quick fix. The maintainable path is maintainer security/product review of a versioned filesystem ACL design, likely by reviewing or replacing the open PathGuard PR and explicitly resolving the exec gap.
