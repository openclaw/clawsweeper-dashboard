## Summary

Keep open. Current main still has only the narrower `tools.fs.workspaceOnly` boolean guard, while the requested allow/deny filesystem policy and exec coverage remain unimplemented.

## What This Changes

Request adding configurable filesystem sandboxing with mode, allowed paths, denied paths, and enforcement across read, write, edit, and exec tools.

## Best Possible Solution

Keep this tracked until OpenClaw has a versioned filesystem policy contract with deny-wins path rules for filesystem and local media tools, plus a maintainer-approved exec strategy or a deliberate linked follow-up for exec.

## Reproduction Assessment

Yes. Source inspection is a high-confidence reproduction path: the strict schema rejects the requested keys, the runtime policy only models `workspaceOnly`, and exec construction receives no filesystem allow/deny policy.

## Solution Assessment

No. The current `workspaceOnly` guard is useful but narrower than the requested policy; the best solution should settle a versioned `tools.fs` or successor contract and an explicit exec strategy rather than only adding the literal `tools.fileAccess` shape.
