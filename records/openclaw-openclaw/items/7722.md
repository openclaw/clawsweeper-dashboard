## Summary

Keep open. Current main still exposes only the narrower `tools.fs.workspaceOnly` boolean, does not accept the requested `tools.fileAccess` or allow/deny path rules, and does not apply filesystem ACL policy to arbitrary `exec` commands; the active PathGuard PR is related but unmerged and explicitly leaves shell execution out of scope.

## What This Changes

The issue requests a configurable filesystem access policy with allow/deny path rules for filesystem tools and exec behavior.

## Best Possible Solution

Keep this as the canonical filesystem ACL tracker until OpenClaw lands a maintainer-reviewed, versioned `tools.fs` policy with deny-wins allow/deny rules, covered file/media/apply_patch behavior, and an explicit exec boundary or linked exec follow-up.

## Reproduction Assessment

Yes. Source inspection shows the strict schema, public config type, runtime policy, and tests only support `tools.fs.workspaceOnly`, so the requested `tools.fileAccess`, `allowedPaths`, and `denyPaths` cannot be expressed or enforced on current main.

## Solution Assessment

No for closing or queueing a quick automated fix. The maintainable path is security/product review of a versioned filesystem ACL design, likely by reviewing or replacing the open PathGuard PR and explicitly resolving the exec gap.
