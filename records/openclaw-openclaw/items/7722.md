## Summary

Keep open. Current `main` and the latest release still only expose the narrower `tools.fs.workspaceOnly` guard; the requested allow/deny filesystem ACL and exec-wide path policy remain unimplemented and need security/product review.

## What This Changes

Add configuration-driven filesystem sandboxing with allow/deny path rules for file tools and exec-related filesystem access.

## Best Possible Solution

Keep this as the canonical filesystem ACL tracker until a reviewed policy lands with deny-wins path rules and the exec boundary is either included or explicitly tracked separately.

## Reproduction Assessment

Yes, source-reproducible. Current schema, public types, runtime policy, docs, and tests only support `tools.fs.workspaceOnly`, so the requested path ACL cannot currently be expressed or enforced.

## Solution Assessment

No for closing or queueing a quick fix. The maintainable path is security/product review of a versioned filesystem ACL design, likely by narrowing or replacing https://github.com/openclaw/openclaw/pull/60981 and explicitly deciding the exec boundary.
