## Summary

Keep open. Current main and the latest release still expose only the narrower `tools.fs.workspaceOnly` boolean, while the requested allow/deny filesystem policy and exec-boundary decision remain unresolved; the related open PRs are useful candidates but are not merged and do not settle the full behavior.

## What This Changes

Request a filesystem sandboxing config surface with sandbox/full mode, allowed paths, deny paths, deny-wins enforcement for file operations, and a strategy for exec coverage.

## Best Possible Solution

Keep this as the canonical filesystem access policy issue until maintainers choose one versioned `tools.fs` contract, enforce it consistently across local file/media paths, decide or split the exec strategy, and land it with schema, docs, tests, and upgrade guidance.

## Reproduction Assessment

Yes at source level. The strict config schema and public types only accept `tools.fs.workspaceOnly`, and current tests preserve unrestricted host filesystem access when that flag is false or unset; I did not run tests because this review is read-only.

## Solution Assessment

No, closing is not the right solution. The current implementation is a useful workspace-only guard, but the maintainable fix needs an owner-reviewed filesystem policy contract plus an explicit decision for exec rather than treating existing behavior as complete.
