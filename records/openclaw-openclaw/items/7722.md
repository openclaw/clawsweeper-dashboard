## Summary

Keep open: current main and v2026.6.6 still expose only the narrower `tools.fs.workspaceOnly` boolean, while the requested allow/deny path policy and exec coverage remain absent. The open PathGuard and `tools.fs.roots` PRs are useful but partial or competing implementation candidates that need maintainer security/product direction before this can close.

## What This Changes

The issue requests a config-driven filesystem sandbox policy with allowed paths, denied paths, deny-wins precedence, and enforcement across read, write, and exec tools.

## Best Possible Solution

Define one maintainer-reviewed filesystem policy contract, likely under `tools.fs` or a successor schema, with canonical path handling, deny precedence, file/media/apply_patch coverage, subagent inheritance, docs/doctor decisions, and an explicit exec strategy.

## Reproduction Assessment

Yes by source inspection: current schema and runtime policy accept only `tools.fs.workspaceOnly`, so the requested `tools.fileAccess`/allow/deny shape remains unsupported. I did not run the invalid config because this review is read-only.

## Solution Assessment

Unclear as a final implementation shape: the security direction is aligned, but the permanent config key, compatibility plan, deny/allow semantics, and exec enforcement strategy need maintainer design before code is safe to land.
