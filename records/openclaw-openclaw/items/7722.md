## Summary

Current main still provides only the narrower `tools.fs.workspaceOnly` boundary. Two open, conflicting PRs explore path-based policies, but neither is merged and the stronger host-root proposal explicitly leaves shell execution unchanged, so the central security and product decision remains necessary.

## What This Changes

Introduce configurable filesystem path access controls beyond a binary workspace boundary, with defined enforcement across file tools and shell execution.

## Best Possible Solution

Choose one canonical `tools.fs` policy contract for host file and media operations, preserve a clear compatibility default, and separately define enforceable exec isolation rather than presenting best-effort command parsing as a complete boundary.

## Reproduction Assessment

Yes. Current main’s strict schema, public type, policy resolver, and tests establish that only the binary `workspaceOnly` behavior exists and the requested path-rule keys are rejected.

## Solution Assessment

No, not exactly as proposed. Allow/deny rules are appropriate for owned file and media tools, but exec needs a separately enforceable isolation design rather than best-effort extraction of paths from commands.
