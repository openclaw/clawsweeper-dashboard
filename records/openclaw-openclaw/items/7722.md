## Summary

Keep open. Current main still exposes only the narrower `tools.fs.workspaceOnly` boolean; the requested allow/deny filesystem policy is absent from the strict config contract and runtime policy shape, and the active PathGuard PR remains open with arbitrary shell execution explicitly out of scope.

## What This Changes

Add a configurable filesystem access policy, such as `tools.fileAccess` or `tools.fs.allowedPaths`/`denyPaths`, to constrain agent file access beyond the current workspace-only flag.

## Best Possible Solution

Keep this as the canonical tracker until OpenClaw lands a versioned `tools.fs` ACL with deny-wins allow/deny semantics across file tools, local media, apply_patch, spawned subagents, and either a clearly designed exec boundary or a linked exec follow-up.

## Reproduction Assessment

Yes, source-level. The strict schema, public config type, and runtime policy only expose `workspaceOnly`, so the requested `tools.fileAccess` or `allowedPaths`/`denyPaths` config cannot be expressed on current main.

## Solution Assessment

No for closing or queueing an automated fix now. The maintainable path needs maintainer security/product review of the policy shape and the open PathGuard approach, especially around exec semantics.
