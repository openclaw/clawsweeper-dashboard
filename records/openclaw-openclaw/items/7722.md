## Summary

Keep open: current main and the latest release still only expose the narrower `tools.fs.workspaceOnly` boolean, not the requested deny-wins filesystem allow/deny policy or a reliable exec strategy. The open PathGuard and roots PRs are useful implementation candidates, but they are unmerged, partly competing, and do not by themselves make this issue cleanup-ready.

## What This Changes

Request a configurable filesystem access policy, originally proposed as `tools.fileAccess`, with allowed paths, denied paths, sandbox/full modes, and enforcement across read, write, and exec tools.

## Best Possible Solution

Define one maintainer-approved `tools.fs` or successor filesystem policy contract, then implement deny-wins allow/deny semantics across file tools, local media, `apply_patch`, spawned sessions, and a clearly scoped exec strategy with docs and upgrade guidance.
