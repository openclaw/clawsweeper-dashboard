## Summary

Keep open. Current main still implements only the narrower `tools.fs.workspaceOnly` boolean policy; the requested `tools.fileAccess`/`mode`/`allowedPaths`/`denyPaths` contract is not in the strict config schema, and shell exec is still constructed without a filesystem allow/deny policy.

## What This Changes

The issue requests configurable filesystem sandboxing with sandbox/full mode, allowed paths, deny paths, deny-wins enforcement, and coverage across file and exec tools.

## Best Possible Solution

Keep this tracked until OpenClaw has a versioned `tools.fs` or successor path-policy contract with deny-wins path rules for filesystem/local media tools, plus a maintainer-approved exec strategy or an explicit linked follow-up for exec.

## Reproduction Assessment

Yes. Source inspection of current main is a high-confidence reproduction path: the strict schema rejects the requested keys, the runtime policy only models `workspaceOnly`, and exec construction receives no filesystem path policy.

## Solution Assessment

No. The current `workspaceOnly` guard is useful but narrower than the requested security model; the maintainable solution should settle the config contract and exec semantics, likely coordinated with PR #60981 rather than treating the boolean guard as complete.
