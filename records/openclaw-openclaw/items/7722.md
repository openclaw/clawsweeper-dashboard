## Summary

Keep open. Current main still implements only the narrower `tools.fs.workspaceOnly` boolean guard, the requested allow/deny path policy is absent from the strict config contract, and the related PathGuard PR remains open and explicitly leaves arbitrary shell execution out of scope.

## What This Changes

The issue requests a configurable filesystem sandbox policy with allow and deny path rules enforced across filesystem and exec tools.

## Best Possible Solution

Keep this issue as the canonical tracker until OpenClaw lands a versioned filesystem policy, likely under `tools.fs`, with deny-wins allow/deny semantics for file tools, local media, and apply_patch plus an explicit exec boundary or linked follow-up.

## Reproduction Assessment

Yes for the reported config gap at source level: the strict `tools.fs` schema and public type only include `workspaceOnly`, and no `tools.fileAccess` or allow/deny path fields exist on current main. I did not run a live config load because this review is read-only.

## Solution Assessment

No for closing or queueing a new automated fix now. The maintainable path needs maintainer security/product review of the policy shape and the open PathGuard PR, especially for exec semantics.
