## Summary

Keep open. Current main and the latest release still only expose the binary `tools.fs.workspaceOnly` guard, while this request asks for a new allow/deny filesystem policy plus explicit exec semantics; the active PathGuard PR is relevant but unmerged and explicitly leaves arbitrary shell execution out of scope.

## What This Changes

Request configurable filesystem sandboxing with sandbox/full mode, allowed paths, deny paths, and enforcement across read, write, and exec tools.

## Best Possible Solution

Define a versioned filesystem ACL under `tools.fs` or a successor surface, enforce deny-wins path rules across file/media/patch surfaces, and explicitly decide whether exec is sandboxed, out of scope, or tracked separately.

## Reproduction Assessment

Yes for the missing capability by source inspection. Current main’s strict config schema, public type, and runtime policy only model `tools.fs.workspaceOnly`, and shell execution is constructed without filesystem policy input.

## Solution Assessment

Unclear for the exact `tools.fileAccess` shape. A deny-wins allow/deny policy is directionally appropriate, but the maintainable solution likely belongs under `tools.fs` or a successor contract after exec semantics are settled.
