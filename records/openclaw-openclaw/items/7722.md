## Summary

Keep open. Current main and v2026.5.12 still expose only the narrower `tools.fs.workspaceOnly` guard, while the requested allow/deny filesystem policy and exec coverage remain unresolved; recent operator comments and the active PathGuard PR confirm this is still a live security/product design item.

## What This Changes

The issue requests a configurable filesystem sandbox policy for tools, including mode, allowed paths, denied paths, and enforcement across file and exec tools.

## Best Possible Solution

Keep this issue as the canonical filesystem ACL tracker until a reviewed `tools.fs` or successor policy lands with deny-wins `allowedPaths`/`denyPaths` semantics and the shell-exec boundary is either designed into the policy or explicitly split to https://github.com/openclaw/openclaw/issues/74379.

## Reproduction Assessment

Yes. Source inspection on current main and v2026.5.12 shows the strict schema, public type, runtime policy, docs, and tests only support `tools.fs.workspaceOnly`, so the requested path policy cannot currently be expressed or enforced.

## Solution Assessment

No for closing or queueing a quick fix. The maintainable path is maintainer security/product review of a versioned filesystem ACL design, likely by repairing or replacing https://github.com/openclaw/openclaw/pull/60981 and explicitly deciding the exec boundary.
