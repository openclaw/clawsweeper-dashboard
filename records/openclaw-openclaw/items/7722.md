## What This Changes

Add configurable allow/deny filesystem path policy for agent file operations, with a defined strategy for shell execution, without requiring full container sandboxing.

## Best Possible Solution

Define one canonical policy under `tools.fs` with deny-wins, operation-aware roots enforced across every host filesystem and local-media path, preserve documented upgrade behavior, and use real sandbox or OS isolation for unrestricted `exec` instead of presenting shell-command parsing as a security boundary.

## Reproduction Assessment

Yes, at source level. Current main and `v2026.6.5` expose only `tools.fs.workspaceOnly`, so the proposed path-policy keys are rejected and no granular enforcement path exists.

## Solution Assessment

No, not exactly as proposed. Granular policy is appropriate for filesystem and media tools, but maintainers must define one compatible contract and treat `exec` as a separate isolation problem because shell access cannot be secured by path-argument matching alone.
