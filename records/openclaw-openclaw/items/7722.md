## Summary

Keep open. Current main and the latest release still expose only the narrower `tools.fs.workspaceOnly` boolean, while the requested configurable allow/deny filesystem policy and the exec coverage remain unresolved security/product work with competing open PRs.

## What This Changes

Request to add configurable filesystem sandboxing with allowed/denied paths so file and exec tool access can be restricted by policy.

## Best Possible Solution

Define one canonical `tools.fs` or successor policy contract with normalization, deny precedence, file/media/apply_patch coverage, and explicit exec semantics, then land or replace the current open implementation candidates under that contract.

## Reproduction Assessment

Yes, by source inspection: current main and v2026.6.10 accept only `tools.fs.workspaceOnly`, while tests preserve outside-workspace access when that boolean is false or unset. I did not run a live agent attempt because this was a read-only review.

## Solution Assessment

No, not as a direct `tools.fileAccess` key addition. The maintainable fix needs a maintainer-approved `tools.fs` or successor contract with canonicalization, inheritance, media/apply_patch coverage, and explicit exec in-scope or out-of-scope behavior.
