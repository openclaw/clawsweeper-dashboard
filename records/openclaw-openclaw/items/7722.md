## Summary

Keep open: current main and v2026.6.10 still expose only the narrower `tools.fs.workspaceOnly` / `tools.exec.applyPatch.workspaceOnly` controls, while the requested allow/deny path policy and explicit exec strategy remain unmerged security/product work.

## What This Changes

The issue requests a new filesystem access policy config, originally sketched as `tools.fileAccess`, with sandbox/full mode plus allowed and denied path lists enforced across file and exec tools.

## Best Possible Solution

Define a maintainer-approved `tools.fs` or successor filesystem policy with normalized deny-wins allow/deny rules enforced across file tools, local media, and `apply_patch`, then either cover exec in the same boundary or document and track exec as a separate explicit policy surface.

## Reproduction Assessment

Yes, at source level: the current strict config schema only accepts `tools.fs.workspaceOnly`, so the reported `tools.fileAccess` / allow-deny config remains invalid without running a parser test.

## Solution Assessment

No, the current `workspaceOnly` boolean is only a partial workaround; the best fix needs a deliberate security/config contract with path normalization, deny precedence, and an explicit exec scope decision.
