## Summary

Keep open. Current main and the latest release still provide only the narrower `tools.fs.workspaceOnly` boolean, while the requested allow/deny filesystem policy, `tools.fileAccess`/mode surface, and exec coverage remain unimplemented; the two related open PRs are useful implementation candidates but are not merged and still need maintainer security/product direction.

## What This Changes

Request a configurable filesystem access policy with mode, allowed paths, and deny paths that can constrain file operations and exec beyond OS user permissions.

## Best Possible Solution

Define one canonical `tools.fs` filesystem policy contract with path canonicalization, deny precedence, coverage across file/media/apply_patch, and an explicit exec strategy before landing one implementation path or splitting exec into a tracked follow-up.

## Reproduction Assessment

Yes, by source inspection. The strict config schema only accepts `tools.fs.workspaceOnly`, so the requested `tools.fileAccess`, `allowedPaths`, and `denyPaths` config remains rejected, and current host filesystem tools still allow outside-workspace access when the boolean is unset or false.

## Solution Assessment

No as a simple config-key addition. The maintainable path is a maintainer-reviewed filesystem policy contract that defines canonicalization, inheritance, media/apply_patch coverage, and whether exec is enforceable or explicitly separate.
