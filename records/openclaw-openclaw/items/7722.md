## Summary

Keep open. Current main and the latest release still expose only the narrower `tools.fs.workspaceOnly` boolean, while this request asks for a new rule-based filesystem ACL plus exec semantics that need maintainer security/product design.

## What This Changes

Add configurable filesystem access restrictions with mode, allowed paths, deny paths, and enforcement across file and exec tools.

## Best Possible Solution

Define a versioned filesystem ACL under `tools.fs` or a successor surface, enforce deny-wins path rules across file/media/patch surfaces, and explicitly decide whether exec is sandboxed, out of scope, or tracked separately.

## Reproduction Assessment

Yes for the missing capability. Source inspection shows the strict config schema and public type only accept `tools.fs.workspaceOnly`, while shell execution is constructed without a filesystem policy input.

## Solution Assessment

Unclear for the exact `tools.fileAccess` shape. A deny-wins allow/deny policy is directionally appropriate, but the maintainable solution likely belongs under `tools.fs` or a successor contract after exec semantics are settled.
