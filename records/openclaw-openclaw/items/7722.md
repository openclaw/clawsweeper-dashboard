## Summary

Keep open. Current main and latest release still expose only the narrower `tools.fs.workspaceOnly` boolean, while the requested allow/deny filesystem policy and exec coverage remain unresolved; the two open PRs are useful candidates but are not merged and represent competing or partial policy shapes.

## What This Changes

Request to add configurable filesystem allow/deny path policy, including sandbox/full modes and coverage for read, write, edit, and exec tools.

## Best Possible Solution

Land one maintainer-approved filesystem policy surface under `tools.fs` or a successor contract, with canonical path evaluation, deny-wins semantics, documented compatibility behavior, full local-file surface coverage, and an explicit exec strategy.

## Reproduction Assessment

Yes by source inspection: the current schema and public type only accept `tools.fs.workspaceOnly`, and tests preserve unrestricted host filesystem access when that flag is false or unset. I did not run a live config parse because this was a read-only review, but the strict Zod schema makes the reported rejection source-reproducible.

## Solution Assessment

No, the current implementation is not the full solution because it only provides a workspace-only boolean and does not cover configurable allow/deny rules or exec. The best fix needs maintainer direction on the canonical policy model before choosing between the open candidate PRs or a narrower design.
