## Summary

Keep open. Current main and the latest release still expose only the boolean `tools.fs.workspaceOnly` guard; the requested allow/deny path policy and exec coverage remain unimplemented, and recent discussion treats this as the canonical tracker rather than a duplicate.

## What This Changes

The issue requests a configurable filesystem sandbox policy with `mode`, `allowedPaths`, and `denyPaths` enforced across file operations and exec.

## Best Possible Solution

Keep this as the canonical tracker, choose one versioned `tools.fs` ACL contract, rework or replace the active implementation PR, and either include exec policy semantics or explicitly split exec to https://github.com/openclaw/openclaw/issues/74379 before calling the issue complete.

## Reproduction Assessment

Yes. Source inspection on current main and `v2026.5.12` shows the strict schema, public type, runtime policy, docs, and tests only support `tools.fs.workspaceOnly`, so the requested path policy cannot currently be expressed or enforced.

## Solution Assessment

No for closing or queueing a quick fix. The maintainable path is maintainer security/product review of a versioned filesystem ACL design, likely by repairing or replacing https://github.com/openclaw/openclaw/pull/60981 and explicitly deciding the exec boundary.
