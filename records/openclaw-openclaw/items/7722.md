## Summary

Keep open. Current main still exposes only the binary `tools.fs.workspaceOnly` guard, while this issue asks for configurable filesystem ACLs with allow/deny paths and explicit exec semantics; the duplicate cleanup around https://github.com/openclaw/openclaw/issues/79023 and https://github.com/openclaw/openclaw/issues/79167 also treats this as the canonical unresolved tracker.

## What This Changes

The issue requests config-driven filesystem sandboxing with `allowedPaths`/`denyPaths`, a sandbox/full mode, and enforcement across read, write, and exec tools.

## Best Possible Solution

Define a versioned filesystem ACL under `tools.fs` or a successor contract, enforce deny-wins path rules across file/media/patch surfaces, and explicitly decide whether exec is sandboxed, out of scope, or tracked as a separate boundary.

## Reproduction Assessment

Yes, by source inspection for the missing capability. Current main’s strict schema, public type, runtime policy, tests, and docs only model `tools.fs.workspaceOnly`, and shell execution is built without filesystem ACL input.

## Solution Assessment

Unclear for the exact `tools.fileAccess` shape. A deny-wins allow/deny policy is directionally appropriate, but the maintainable solution likely belongs under `tools.fs` or a successor contract after exec semantics are settled.
