## Summary

Keep open. Current main still only implements the narrower `tools.fs.workspaceOnly` boolean; it does not accept the requested `tools.fileAccess` or allow/deny path-rule policy, and shell `exec` is not governed by filesystem path rules.

## What This Changes

Request a configurable filesystem sandbox policy with allow/deny path rules, preferably covering read, write, edit, apply_patch, local media, and exec behavior.

## Best Possible Solution

Keep this tracked until OpenClaw has a versioned filesystem policy contract with deny-wins path rules for filesystem tools and local media, plus an explicit maintainer-approved exec strategy or a linked follow-up that owns the exec gap.

## Reproduction Assessment

Yes. Source inspection is a high-confidence reproduction path: the strict schema rejects the requested keys, runtime policy only models `workspaceOnly`, and exec construction receives no filesystem allow/deny policy.

## Solution Assessment

No. The current `workspaceOnly` boolean is a useful narrower guard, but it is not the requested allow/deny sandbox policy; the open PathGuard PR is partial until merged and the exec gap is resolved or split deliberately.
