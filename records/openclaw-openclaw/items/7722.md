## Summary

Keep open. Current main and the latest release still expose only the narrower `tools.fs.workspaceOnly` boolean guard; the requested allow/deny filesystem ACL, `tools.fileAccess` surface, and exec-wide path policy remain unimplemented, while related PRs are still open and need owner/security review.

## What This Changes

The issue requests a config-driven filesystem sandboxing policy with allowed and denied paths enforced across file and exec tools.

## Best Possible Solution

Keep this as the canonical filesystem ACL tracker until an owner-reviewed, versioned policy lands across file tools, apply_patch, local media, prompt media, and subagent propagation, with exec either covered by a designed boundary or explicitly tracked separately.

## Reproduction Assessment

Yes. Source inspection shows the strict schema, public type, runtime policy, docs, and tests only support `tools.fs.workspaceOnly`, so the requested path ACL cannot currently be expressed or enforced.

## Solution Assessment

No for closing or queueing a quick fix. The maintainable path is security/product review of a versioned filesystem ACL design, likely by narrowing or replacing the open PathGuard/roots work and explicitly deciding the exec boundary.
