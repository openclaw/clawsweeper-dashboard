## Summary

Keep open. Current main still only exposes the narrower `tools.fs.workspaceOnly` boolean and sandbox modes, while the requested allow/deny filesystem policy plus exec-boundary behavior is not implemented and remains security/product-sensitive.

## What This Changes

Request filesystem access configuration, such as `tools.fileAccess` or a successor `tools.fs` policy, with allowed and denied paths enforced across file operations and exec behavior.

## Best Possible Solution

Keep this as the canonical filesystem ACL tracker until an owner-reviewed `tools.fs` policy surface lands with deny-wins path rules across filesystem tools and local-file surfaces, and with exec either covered by a real boundary or explicitly split into a linked follow-up.

## Reproduction Assessment

Yes, source-reproducible. The strict schema, public types, runtime policy, docs, and tests only support `tools.fs.workspaceOnly`, so the requested allow/deny ACL cannot currently be configured or enforced.

## Solution Assessment

No for closing or queueing a quick fix. The maintainable solution is a security/product-reviewed filesystem policy contract, likely under `tools.fs`, with an explicit exec boundary decision.
