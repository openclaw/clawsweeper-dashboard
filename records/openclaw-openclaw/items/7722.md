## Summary

Keep open. Current main and v2026.6.8 still expose only the narrower `tools.fs.workspaceOnly` boolean, while the requested allow/deny filesystem policy and the requested exec coverage remain unmerged security/product work; the open implementation PRs are useful context but are partial or blocked rather than completed replacements.

## What This Changes

Requests a configurable filesystem access policy with allow/deny paths and sandbox/full mode enforced across read, write, edit, and exec-style access.

## Best Possible Solution

Define one canonical `tools.fs` policy contract with path normalization, deny precedence, file/media/apply_patch coverage, and an explicit exec strategy, then land or replace the current open implementation candidates under that contract.

## Reproduction Assessment

Yes, by source inspection: the strict config schema accepts only `tools.fs.workspaceOnly`, and current tests preserve outside-workspace access when that boolean is unset or false. I did not run a live agent attempt because this was a read-only review.

## Solution Assessment

No, not as a direct `tools.fileAccess` key addition. The maintainable fix needs a maintainer-approved `tools.fs` or successor contract covering canonicalization, inheritance, media/apply_patch coverage, and explicit exec semantics.
