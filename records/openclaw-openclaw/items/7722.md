## Summary

Keep open. Current main and the latest release still expose only the narrower `tools.fs.workspaceOnly` boolean, reject the requested allow/deny path policy surface, and leave arbitrary `exec` outside a filesystem ACL; the open PathGuard PR is relevant but unmerged and explicitly partial.

## What This Changes

Request to add configurable filesystem access policy, such as `tools.fileAccess` or `tools.fs.allowedPaths`/`denyPaths`, with deny-wins enforcement across file tools and exec.

## Best Possible Solution

Keep this as the canonical filesystem ACL tracker until OpenClaw lands a maintainer-reviewed `tools.fs` policy with deny-wins allow/deny rules, covered file/media/apply_patch behavior, and either an explicit exec boundary or a linked exec follow-up.

## Reproduction Assessment

Yes. Source inspection shows the strict schema, public config type, runtime policy, and tests only support `tools.fs.workspaceOnly`, so the requested `tools.fileAccess`, `allowedPaths`, and `denyPaths` cannot be expressed or enforced on current main.

## Solution Assessment

No for closing or queueing a quick automated fix. The maintainable path is security/product review of a versioned filesystem ACL design, likely by reviewing or replacing the open PathGuard PR and explicitly resolving the exec gap.
