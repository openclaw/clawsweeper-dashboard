## Summary

Keep open. Current main and v2026.5.7 still expose only the binary `tools.fs.workspaceOnly` guard, while this request asks for a new allow/deny filesystem policy plus exec semantics that need security/product design.

## What This Changes

Request a configurable filesystem sandboxing policy with allowed paths, denied paths, and enforcement across read/write/edit/exec tools.

## Best Possible Solution

Define a versioned filesystem ACL under `tools.fs` or a successor surface, enforce deny-wins path rules across file/media/patch surfaces, and explicitly decide whether exec is sandboxed, out of scope, or tracked separately.

## Reproduction Assessment

Yes for the missing capability. Source inspection shows the strict config schema and public type only accept `tools.fs.workspaceOnly`, and the shell tool is constructed without any filesystem policy input.

## Solution Assessment

Unclear for the exact `tools.fileAccess` shape. A deny-wins allow/deny policy is directionally appropriate, but the maintainable solution likely belongs under `tools.fs` or a successor contract after exec semantics are settled.
