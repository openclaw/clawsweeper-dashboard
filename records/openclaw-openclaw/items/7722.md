## Summary

Keep open. Current main still implements only the narrower `tools.fs.workspaceOnly` boolean; the requested path allow/deny policy, `tools.fileAccess` shape, and shell-exec filesystem strategy are not present, and the related PathGuard PR remains open and partial.

## What This Changes

Add a filesystem access configuration that can sandbox file operations with allow/deny path rules and cover read, write, edit, and exec tools.

## Best Possible Solution

Define a versioned `tools.fs` or successor policy with canonical deny-wins allow/deny rules across file, local-media, and apply_patch surfaces, plus a maintainer-approved exec strategy or explicit split follow-up.

## Reproduction Assessment

Yes for the missing capability. Source inspection shows the strict config schema and public type only accept `tools.fs.workspaceOnly`, while exec construction has no filesystem policy input.

## Solution Assessment

Unclear for the exact `tools.fileAccess` shape. A deny-wins allow/deny policy is a reasonable direction, but the maintainable path likely belongs under `tools.fs` or a successor contract after exec semantics are settled.
