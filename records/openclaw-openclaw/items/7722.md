## Summary

Keep open: current main and the latest v2026.6.6 release still only expose the boolean `tools.fs.workspaceOnly` filesystem guard, while this issue asks for a rule-based filesystem policy with allow/deny paths and an explicit exec story. The related PathGuard and roots PRs are useful implementation candidates, but both are still open and neither cleanly completes the full requested security boundary.

## What This Changes

The issue requests a configurable filesystem sandbox policy, originally sketched as `tools.fileAccess`, with allow paths, deny paths, sandbox/full mode, and enforcement across file and exec tools.

## Best Possible Solution

Keep this issue as the canonical filesystem access-policy tracker until maintainers choose and land one versioned `tools.fs` contract with explicit file-tool, media, compatibility, and exec semantics.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: adding the requested `tools.fileAccess` or `tools.fs.allowedPaths`/`denyPaths` keys would hit the strict runtime schema, which currently accepts only `tools.fs.workspaceOnly`.

## Solution Assessment

Unclear. A built-in tool-layer filesystem policy fits the security goal, but the best shape still needs maintainer decisions around `tools.fs` naming, default-deny behavior, compatibility/migration, and the exec boundary.
