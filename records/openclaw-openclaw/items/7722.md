## Summary

Keep open: current main and v2026.6.10 still expose only the narrower `tools.fs.workspaceOnly` boolean, while the requested allow/deny filesystem policy and exec coverage remain unmerged security/product work.

## What This Changes

The issue requests a configurable filesystem sandboxing policy with allow paths, deny paths, deny precedence, and coverage for read, write, and exec tools.

## Best Possible Solution

Define one canonical `tools.fs` or successor policy contract with normalization, deny precedence, file/media/apply_patch coverage, and an explicit exec strategy, then land or replace the current open implementation candidates under that contract.

## Reproduction Assessment

Yes at source level: the strict config schema accepts only `tools.fs.workspaceOnly`, and current tests preserve outside-workspace access when that boolean is unset or false. I did not run a live agent attempt because this review is read-only.

## Solution Assessment

No as a direct `tools.fileAccess` key addition. The maintainable fix needs a maintainer-approved `tools.fs` or successor contract covering canonicalization, inheritance, file/media/apply_patch coverage, and explicit exec semantics.
