## Summary

Keep open. Current main still only exposes the narrower `tools.fs.workspaceOnly` boolean, while this request asks for a rule-based filesystem ACL plus exec semantics that need security/product design.

## What This Changes

Request to add filesystem sandboxing configuration with allow/deny path rules, a sandbox/full mode, and enforcement across read, write, edit, and exec tools.

## Best Possible Solution

Define a versioned filesystem ACL under `tools.fs` or a successor contract, enforce deny-wins path rules across file/media/patch surfaces, and explicitly decide whether exec is sandboxed, out of scope, or tracked separately.

## Reproduction Assessment

Yes for the missing capability. Source inspection shows the strict config schema and public types only accept `tools.fs.workspaceOnly`, while shell execution is constructed without any filesystem allow/deny policy input.

## Solution Assessment

Unclear for the exact `tools.fileAccess` shape. The allow/deny policy direction is reasonable, but the maintainable solution likely belongs under `tools.fs` or a successor contract after exec semantics are settled.
