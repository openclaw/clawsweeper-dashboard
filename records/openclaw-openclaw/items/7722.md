## Summary

Keep open. Current main still implements only the narrower `tools.fs.workspaceOnly` boolean, while the requested allow/deny path policy, `tools.fileAccess` shape, and shell-exec filesystem strategy are not present; the related PathGuard PR is open, partial context rather than a shipped fix.

## What This Changes

Request adds configurable filesystem sandboxing with mode, allowed paths, and deny paths enforced across file and exec tools.

## Best Possible Solution

Define a versioned `tools.fs` or successor policy with deny-wins allow/deny rules across file, local-media, and apply_patch surfaces, plus a maintainer-approved exec strategy or explicit follow-up issue for exec.

## Reproduction Assessment

Yes for the missing capability. Source inspection shows the strict config schema and public type only accept `tools.fs.workspaceOnly`, while exec construction has no filesystem policy input.

## Solution Assessment

Unclear for the exact `tools.fileAccess` shape. A deny-wins allow/deny policy is a reasonable direction, but the maintainable path likely belongs under `tools.fs` or a successor contract after exec semantics are settled.
