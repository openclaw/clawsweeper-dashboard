## Summary

Keep open. Current main still implements only the narrower `tools.fs.workspaceOnly` boolean guard; the requested allow/deny filesystem policy and `tools.fileAccess`-style config are not in the strict config contract, and exec remains outside filesystem path-policy enforcement.

## What This Changes

Request to add configuration-enforced filesystem sandboxing with mode, allowed paths, and deny paths across file operations and exec.

## Best Possible Solution

Define and ship a versioned filesystem policy under `tools.fs` or a successor contract, with deny-wins path rules across file/local-media/apply_patch surfaces and a maintainer-approved exec strategy or linked follow-up.

## Reproduction Assessment

Yes for the missing capability. Source inspection of current main shows the strict schema accepts only `tools.fs.workspaceOnly`, the runtime policy only models that boolean, tests preserve unrestricted default access, and exec receives no filesystem path policy.

## Solution Assessment

No for treating current main as complete. The existing `workspaceOnly` guard is useful but materially narrower than the requested allow/deny security model, and the maintainable path needs a settled config contract plus explicit exec semantics.
