## Summary

Keep open. Current main still exposes only the binary `tools.fs.workspaceOnly` filesystem guard, while this issue asks for configurable allow/deny path rules plus explicit exec coverage; the active PathGuard PR is useful context but remains open and explicitly leaves arbitrary shell execution out of scope.

## What This Changes

The issue requests a filesystem policy config with sandbox/full mode and allowedPaths/denyPaths enforcement for read, write, and exec tools.

## Best Possible Solution

Define a versioned filesystem ACL under `tools.fs` or a successor contract, enforce deny-wins path rules across filesystem/media/patch surfaces, and explicitly decide whether exec is sandboxed, out of scope, or tracked separately.

## Reproduction Assessment

Yes, by source inspection for the missing capability. Current main’s strict schema, public type, runtime policy, tests, and docs only model `tools.fs.workspaceOnly`, and shell execution is constructed without filesystem policy input.

## Solution Assessment

Unclear for the exact `tools.fileAccess` shape. A deny-wins allow/deny policy is directionally appropriate, but the maintainable solution likely belongs under `tools.fs` or a successor contract after exec semantics are settled.
