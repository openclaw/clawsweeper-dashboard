## Summary

Keep open: current main and v2026.6.6 still expose only the narrower `tools.fs.workspaceOnly` boolean, while this issue asks for a configurable allow/deny filesystem policy with an explicit exec strategy. The related open PRs are useful implementation tracks, but they are unmerged and do not by themselves resolve the full cross-tool security request.

## What This Changes

The issue requests a new filesystem access policy config that supports sandbox/full mode plus allowed and denied paths across file and exec tools.

## Best Possible Solution

Keep this issue open until maintainers choose and land a canonical `tools.fs` or successor filesystem policy contract, with deny-wins path rules and an explicit decision for exec coverage or a linked follow-up that owns exec.

## Reproduction Assessment

Yes at source level: the strict config schema only accepts `tools.fs.workspaceOnly`, and current runtime policy carries only that boolean. I did not run a config parser because this review is read-only, but the schema path explains the reported unrecognized-key failure.

## Solution Assessment

No current implementation is the complete solution. The best path is a maintainer-approved security/config design that covers file tools, apply_patch/local media, and either a reliable exec strategy or a separate canonical exec-policy issue.
