## Summary

Keep open. Current main and the latest release still expose only `tools.fs.workspaceOnly`, the active PathGuard PR at https://github.com/openclaw/openclaw/pull/60981 is open/conflicting and explicitly does not sandbox arbitrary shell exec, and recent operator comments add concrete production/security context.

## What This Changes

The issue requests a new configurable filesystem access policy with allow/deny paths that restricts file tools and exec beyond the current `workspaceOnly` boolean.

## Best Possible Solution

Keep this as the canonical filesystem ACL tracker until a reviewed `tools.fs` allow/deny policy lands and the shell-exec gap is either designed into that policy or explicitly split to https://github.com/openclaw/openclaw/issues/74379.

## Reproduction Assessment

Yes. Source inspection on current main and `v2026.5.12` shows the strict schema, public type, runtime policy, docs, and tests only support `tools.fs.workspaceOnly`, so the requested path policy cannot currently be expressed or enforced.

## Solution Assessment

No for closing or queueing a quick fix. The maintainable path is maintainer security/product review of a versioned filesystem ACL design, likely by repairing or replacing https://github.com/openclaw/openclaw/pull/60981 and explicitly deciding the exec boundary.
