## Summary

Keep open. Current main still implements only the binary `tools.fs.workspaceOnly` guard, while this request asks for a configurable allow/deny filesystem policy and explicit exec coverage; the active PathGuard work is relevant but remains open and says arbitrary shell execution is out of scope.

## What This Changes

Request configurable filesystem sandboxing via `tools.fileAccess` or an equivalent policy with mode, allowed paths, denied paths, and enforcement across file and exec tools.

## Best Possible Solution

Define an owner-reviewed filesystem ACL contract under `tools.fs` or a successor surface, enforce deny-wins path rules across file/media/patch surfaces, and explicitly decide whether exec is sandboxed, out of scope, or tracked as a separate security item.

## Reproduction Assessment

Yes, by source inspection for the missing capability. Current main’s strict schema, public type, runtime policy, tests, and docs only model `tools.fs.workspaceOnly`, and shell execution is constructed without filesystem policy input.

## Solution Assessment

Unclear for the exact `tools.fileAccess` shape. A deny-wins allow/deny policy is directionally appropriate, but the maintainable solution likely belongs under `tools.fs` or a successor contract after exec semantics are settled.
