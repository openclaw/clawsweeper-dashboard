## Summary

Keep open. Current main still exposes only the narrower `tools.fs.workspaceOnly` boolean, rejects the requested `tools.fileAccess`/allow-deny path surface, and does not apply filesystem ACL policy to arbitrary `exec`; the open PathGuard PR is relevant but still unmerged and explicitly leaves shell execution out of scope.

## What This Changes

The issue requests a configurable filesystem ACL surface with sandbox/full mode, `allowedPaths`, and `denyPaths` enforced across file and exec tools.

## Best Possible Solution

Keep this as the canonical filesystem ACL tracker until OpenClaw lands a maintainer-reviewed `tools.fs` policy with deny-wins allow/deny rules, covered file/media/apply_patch behavior, and an explicit exec boundary or linked exec follow-up.

## Reproduction Assessment

Yes. Source inspection shows the strict schema, public config type, runtime policy, and tests only support `tools.fs.workspaceOnly`, so the requested `tools.fileAccess`, `allowedPaths`, and `denyPaths` cannot be expressed or enforced on current main.

## Solution Assessment

No for closing or queueing a quick automated fix. The maintainable path is security/product review of a versioned filesystem ACL design, likely by reviewing or replacing the open PathGuard PR and explicitly resolving the exec gap.
