## Summary

Keep open: current main and the latest release still only expose the narrower `tools.fs.workspaceOnly` filesystem guard, while this issue asks for a configurable allow/deny filesystem policy and an explicit exec strategy.

## What This Changes

The issue requests configurable filesystem sandboxing with allow and deny path rules enforced across filesystem, media, apply_patch, and exec-related access.

## Best Possible Solution

Keep this issue open as the canonical tracker until maintainers choose one `tools.fs` or successor filesystem policy model, define the exec boundary explicitly, and land source, docs, schema/help, migration/doctor, and proof for all local-path ingestion surfaces.

## Reproduction Assessment

Yes, from source inspection: the strict `tools.fs` schema accepts only `workspaceOnly`, the public type has no allowed/denied path fields, and exec defaults receive no filesystem policy. I did not run a live config command because this review is read-only.

## Solution Assessment

Partly: the need is valid and aligned with OpenClaw security goals, but the exact `tools.fileAccess` shape should not be assumed best because current policy lives under `tools.fs` and there are competing open models. The best fix needs maintainer product/security direction before implementation.
