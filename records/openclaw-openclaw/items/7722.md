## Summary

Keep open. Current `main` still only exposes the narrower `tools.fs.workspaceOnly` guard; there is no `tools.fileAccess` or allow/deny path policy, and shell exec is still configured separately from filesystem policy. Recent operator comments and the open PathGuard work show this remains an active security/product design item.

## What This Changes

The issue requests a configurable filesystem sandbox policy with sandbox/full mode plus allowed and denied paths enforced across filesystem tools and exec.

## Best Possible Solution

Keep this issue as the canonical filesystem ACL tracker until a reviewed policy lands with deny-wins `allowedPaths`/`denyPaths` semantics and the shell-exec boundary is either designed into that policy or explicitly split to https://github.com/openclaw/openclaw/issues/74379.

## Reproduction Assessment

Yes. Source inspection on current `main` and `v2026.5.12` shows the strict schema, public type, runtime policy, docs, and tests only support `tools.fs.workspaceOnly`, so the requested path ACL cannot currently be expressed or enforced.

## Solution Assessment

No for closing or queueing a quick fix. The maintainable path is maintainer security/product review of a versioned filesystem ACL design, likely by repairing or replacing https://github.com/openclaw/openclaw/pull/60981 and explicitly deciding the exec boundary.
