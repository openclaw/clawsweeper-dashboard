## Summary

Keep open. Current main still exposes only the binary `tools.fs.workspaceOnly` filesystem guard, while this asks for a configurable filesystem ACL with allow/deny paths and explicit exec semantics; live discussion and duplicate closures confirm this remains the canonical unresolved tracker.

## What This Changes

Add a configurable filesystem sandbox policy, originally proposed as `tools.fileAccess`, with allowed paths, denied paths, deny-wins behavior, and coverage for file and exec tools.

## Best Possible Solution

Define a versioned filesystem ACL under `tools.fs` or a successor surface, enforce deny-wins path rules across file/media/patch surfaces, and explicitly decide whether exec is sandboxed, out of scope, or tracked as a separate boundary.

## Reproduction Assessment

Yes, by source inspection for the missing capability. Current main’s strict schema, public type, runtime policy, tests, and docs only model `tools.fs.workspaceOnly`, and shell execution is constructed without filesystem policy input.

## Solution Assessment

Unclear for the exact `tools.fileAccess` shape. A deny-wins allow/deny policy is directionally appropriate, but the maintainable solution likely belongs under `tools.fs` or a successor contract after exec semantics are settled.
