## Summary

Keep open. Current main still provides only the narrower `tools.fs.workspaceOnly` boolean plus separate exec approval/sandbox controls; it does not accept or enforce the requested allow/deny filesystem policy, and the related PathGuard PR is still open and explicitly partial for exec.

## What This Changes

The issue requests a configurable filesystem access policy with sandbox/full mode, allowed paths, denied paths, and enforcement across filesystem and exec tools.

## Best Possible Solution

Keep this tracked until OpenClaw has a versioned filesystem policy contract, likely under `tools.fs` or a successor schema, with deny-wins path rules enforced for filesystem tools, local media, apply_patch, subagent/session policy inheritance, and an explicit maintainer-approved strategy for exec.

## Reproduction Assessment

Yes. Source inspection reproduces the config rejection and behavior gap: strict schema only accepts `tools.fs.workspaceOnly`, current policy types carry no path rules, and exec wiring does not receive filesystem policy data.

## Solution Assessment

No. The current `workspaceOnly` boolean is a narrower guardrail, not the requested allow/deny sandbox policy; #60981 is a plausible partial direction but should not be treated as the best complete solution until merged and the exec gap is resolved or intentionally split.
