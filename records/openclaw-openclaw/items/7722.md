## Summary

Keep open. Current main and the latest release still expose only the narrower `tools.fs.workspaceOnly` filesystem guard, while this issue requests a configurable allow/deny filesystem policy and an explicit exec strategy; the active implementation PRs are useful context but remain open and partial.

## What This Changes

Request a configurable filesystem access policy with sandbox/full mode, allowed paths, deny paths, and enforcement across read, write, edit, apply_patch, and exec-like access.

## Best Possible Solution

Keep this issue as the canonical tracker until maintainers choose a `tools.fs` or successor filesystem policy contract, land a reviewed implementation for filesystem/media/apply_patch coverage, and explicitly decide whether exec is in scope or a documented separate follow-up.

## Reproduction Assessment

Yes, source-level. The strict current schema and public type only accept `tools.fs.workspaceOnly`, so the reported `tools.fileAccess`, `allowedPaths`, and `denyPaths` configuration remains outside the contract; I did not run a live config-load command because this review is read-only.

## Solution Assessment

No current implementation is complete. `tools.fs.workspaceOnly` is useful partial hardening, while the requested allow/deny policy needs a maintainer-approved config shape and a deliberate exec strategy; the open related PRs are candidates, not completion.
