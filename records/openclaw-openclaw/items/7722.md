## Summary

Keep open. Current main and the latest shipped filesystem-policy code still expose only the narrower `tools.fs.workspaceOnly` boolean, while this request asks for a new allow/deny path policy and an explicit exec strategy.

## What This Changes

Request to add configurable filesystem sandboxing with mode, allowed paths, denied paths, and enforcement across read, write, and exec tools.

## Best Possible Solution

Keep this as the canonical filesystem access-policy tracker until maintainers settle the `tools.fs` schema, land or replace the PathGuard work, and explicitly define the exec boundary.

## Reproduction Assessment

Yes, at source level. The strict config schema and public types only accept `tools.fs.workspaceOnly`, so the requested `tools.fileAccess`, `allowedPaths`, and `denyPaths` surface is rejected or unavailable on current main.

## Solution Assessment

No, not as a direct `tools.fileAccess` drop-in yet. The narrow maintainable path is likely a versioned `tools.fs` or successor policy with deny-wins path rules, media/apply_patch coverage, and an explicit exec strategy.
