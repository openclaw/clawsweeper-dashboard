## Summary

Current main still implements only a coarse workspace boundary, while granular path policy remains unresolved across two competing open PRs and the shell-execution boundary. This issue remains necessary as the canonical place for the security and product decision.

## What This Changes

Add configurable allow/deny filesystem path enforcement for file operations and command execution, with global and per-agent policy support.

## Best Possible Solution

Adopt one canonical, versioned tools.fs policy with deny-wins canonical path rules, explicit per-agent merge semantics, coverage for every local-path tool, and a documented decision for exec; then land one linked implementation and close or supersede the competing PR.

## Reproduction Assessment

Yes. Current main's strict ToolFsSchema and boolean ToolFsPolicy provide a high-confidence source-level reproduction: granular allow/deny keys are rejected and no rule evaluator exists.

## Solution Assessment

Unclear. Granular path enforcement is directionally appropriate, but the proposed shape is not yet the best proven solution because maintainers must select the public contract and settle cross-tool and exec semantics.
