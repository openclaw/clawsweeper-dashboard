## Summary

Keep open. Current main still implements only the narrower `tools.fs.workspaceOnly` boolean; the requested `tools.fileAccess`/mode/allowedPaths/denyPaths contract and shell `exec` filesystem policy are not present, and open PR #60981 is partial/unmerged context rather than completion.

## What This Changes

Add a configurable filesystem sandbox policy for agents, with allow/deny paths and mode selection enforced across file and exec tools.

## Best Possible Solution

Define a versioned `tools.fs` or successor policy with deny-wins allow/deny rules across core file, local-media, and apply_patch surfaces, plus a maintainer-approved exec strategy or separate tracked exec follow-up.

## Reproduction Assessment

Yes for the missing capability. Source inspection shows the strict config contract accepts only `tools.fs.workspaceOnly`, the runtime policy only models that boolean, tests preserve unrestricted default access, and exec receives no filesystem path policy.

## Solution Assessment

Unclear for the exact requested `tools.fileAccess` shape. A deny-wins allow/deny policy is a reasonable direction, but the maintainable path likely belongs under `tools.fs` or a successor contract after maintainers settle exec semantics.
