## What This Changes

Request a configurable filesystem sandboxing policy with allow/deny paths that can constrain filesystem and exec-style tool access beyond the current workspace-only guard.

## Best Possible Solution

Define and implement a maintainer-approved filesystem policy contract, likely under `tools.fs` or a successor, with canonical deny-wins path checks across filesystem/media/apply_patch surfaces and an explicit exec strategy or tracked exec follow-up.

## Reproduction Assessment

Yes, at source level: current main's strict `ToolFsSchema` and `FsToolsConfig` only accept `workspaceOnly`, so the requested `tools.fileAccess` or allow/deny path keys would be rejected or ignored. I did not run config validation because this review is read-only.

## Solution Assessment

No, not as the exact `tools.fileAccess` shape by itself. The maintainable solution is a canonical `tools.fs` or successor policy contract with security-reviewed path canonicalization, deny precedence, media/apply_patch coverage, and a deliberately designed exec boundary.
