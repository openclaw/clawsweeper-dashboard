## Summary

Keep open. Current main and the latest release still expose only the narrower `tools.fs.workspaceOnly` boolean guard; the requested allow/deny filesystem ACL and exec-wide path policy remain unimplemented, while related implementation candidates are still open and need security/product review.

## What This Changes

The issue requests a configurable filesystem access policy with sandbox/full modes, allow/deny path rules, and enforcement across file and exec tools.

## Best Possible Solution

Keep this as the canonical filesystem ACL tracker until a reviewed deny-wins path policy lands across filesystem tools and related local-file surfaces, with the exec boundary either included or explicitly tracked separately.

## Reproduction Assessment

Yes. Source inspection shows the strict schema, public type, runtime policy, docs, and tests only support `tools.fs.workspaceOnly`, so the requested path ACL cannot currently be expressed or enforced.

## Solution Assessment

No for closing or queueing a quick fix. The maintainable path is security/product review of a versioned filesystem ACL design, likely by narrowing or replacing https://github.com/openclaw/openclaw/pull/60981 and explicitly deciding the exec boundary.
