## Summary

Keep open. Current main still implements only the narrower `tools.fs.workspaceOnly` boolean guard; the requested allow/deny path policy and `tools.fileAccess`-style config are not in the strict config contract, and shell exec is still outside filesystem policy enforcement.

## What This Changes

Request adds configurable filesystem allow/deny sandboxing via path rules enforced across filesystem and exec tools.

## Best Possible Solution

Define and ship a versioned filesystem policy under `tools.fs` or a successor contract, with deny-wins path rules across file/local-media tools and a maintainer-approved exec strategy or linked follow-up.

## Reproduction Assessment

Yes for the missing capability. Source inspection of current main shows the strict schema rejects the requested path-rule keys, the runtime policy only models `workspaceOnly`, and exec receives no filesystem path policy.

## Solution Assessment

No. The current `workspaceOnly` guard is useful but narrower than the requested security model; the maintainable path needs a settled config contract and exec semantics, likely coordinated with #60981 rather than treating the boolean guard as complete.
