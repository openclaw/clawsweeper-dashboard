## Summary

Keep open. Current main still exposes only the narrower `tools.fs.workspaceOnly` boolean, the requested allow/deny filesystem policy and `tools.fileAccess` config surface are absent, exec is not covered by a filesystem ACL, and the related PathGuard PR remains open rather than merged.

## What This Changes

Request a filesystem sandboxing config with mode, allowed paths, and deny paths enforced across read, write, edit, and exec tools.

## Best Possible Solution

Keep this as the canonical filesystem ACL tracker until OpenClaw lands a maintainer-reviewed, versioned `tools.fs` policy with deny-wins allow/deny rules, covered file/media/apply_patch behavior, and an explicit exec boundary or linked exec follow-up.

## Reproduction Assessment

Yes. Source inspection shows the strict schema, public config type, and runtime policy only support `tools.fs.workspaceOnly`, so the requested `tools.fileAccess`, `allowedPaths`, and `denyPaths` cannot be expressed on current main.

## Solution Assessment

No for closing or queueing a new automated fix now. The narrow maintainable path is security/product review of a versioned `tools.fs` ACL design, likely by reviewing or replacing the open PathGuard PR rather than adding a quick `tools.fileAccess` shim.
