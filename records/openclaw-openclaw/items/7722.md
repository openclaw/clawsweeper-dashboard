## Summary

Keep open. Current main and the latest release still expose only the narrower `tools.fs.workspaceOnly` boolean; the requested allow/deny filesystem ACL and exec-wide enforcement boundary remain unimplemented and security-sensitive.

## What This Changes

Requests a config-level filesystem policy, originally `tools.fileAccess`, with allowed/denied paths and sandbox/full modes enforced across filesystem tools and exec.

## Best Possible Solution

Keep this as the canonical filesystem ACL tracker until an owner-reviewed `tools.fs` policy surface lands with deny-wins path rules, canonical path semantics, upgrade behavior, and an explicit exec decision.

## Reproduction Assessment

Yes. Source inspection of current main and `v2026.5.20` shows only `tools.fs.workspaceOnly` is accepted and enforced, while tests preserve unrestricted host access when it is false or unset.

## Solution Assessment

No for closing or a quick automated fix. The maintainable solution is a security/product-reviewed filesystem policy contract, likely under `tools.fs`, with explicit semantics for deny precedence, canonicalization, upgrades, subagents, media/apply_patch coverage, and exec scope.
