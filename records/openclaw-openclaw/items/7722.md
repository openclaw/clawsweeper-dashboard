## Summary

Keep open. Current main still implements only the narrower `tools.fs.workspaceOnly` boolean; it does not accept `tools.fileAccess`, `mode`, `allowedPaths`, or `denyPaths`, and shell `exec` is not governed by a filesystem allow/deny policy. Open PR #60981 is relevant but unmerged and explicitly leaves arbitrary shell execution out of scope.

## What This Changes

Request a filesystem sandboxing config with mode, allowed paths, and deny paths enforced across file operations and exec.

## Best Possible Solution

Define and ship a versioned filesystem policy under `tools.fs` or a successor config contract, with deny-wins path rules across file/local-media/apply_patch surfaces and a maintainer-approved exec strategy or linked follow-up.

## Reproduction Assessment

Yes for the missing capability. Source inspection shows the strict config contract accepts only `tools.fs.workspaceOnly`, the runtime policy only models that boolean, tests preserve unrestricted default access, and exec receives no filesystem path policy.

## Solution Assessment

Unclear for the exact requested shape. A deny-wins allow/deny policy is a reasonable direction, but maintainers still need to settle the public config surface and explicit shell-exec semantics before this can be treated as a narrow implementation task.
