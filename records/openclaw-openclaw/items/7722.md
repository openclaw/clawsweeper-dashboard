## Summary

Keep open. Current main and the latest shipped release still expose only the narrower `tools.fs.workspaceOnly` boolean, reject the requested allow/deny path policy surface, and do not apply a filesystem ACL to arbitrary shell `exec`; the active PathGuard PR is relevant but remains open, partial, and security-reviewed as not merge-ready.

## What This Changes

The issue requests a configurable filesystem access policy such as `tools.fileAccess` or a `tools.fs` successor with allow/deny path rules enforced across file and exec tools.

## Best Possible Solution

Keep this as the canonical filesystem ACL tracker until a reviewed `tools.fs` policy lands with deny-wins allow/deny rules, covered file/media/apply_patch behavior, and either a designed exec boundary or a linked exec follow-up.

## Reproduction Assessment

Yes. Source inspection shows the strict schema, public type, runtime policy, docs, and tests only support `tools.fs.workspaceOnly`, so the requested path policy cannot currently be expressed or enforced.

## Solution Assessment

No for closing or queueing a quick fix. The maintainable path is maintainer security/product review of a versioned filesystem ACL design, likely by reviewing or replacing the active PathGuard PR and explicitly resolving the exec gap.
