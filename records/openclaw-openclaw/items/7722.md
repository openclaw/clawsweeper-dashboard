## Summary

Keep open. Current main still exposes only the binary `tools.fs.workspaceOnly` guard, while this requests a configurable filesystem ACL with allow/deny paths plus explicit exec semantics; the active PathGuard PR is related but open and explicitly leaves arbitrary shell execution out of scope.

## What This Changes

Request filesystem sandboxing configuration with mode, allowed paths, and denied paths for read, write, edit, apply_patch, and exec surfaces.

## Best Possible Solution

Define a versioned filesystem ACL under `tools.fs` or a successor contract, enforce deny-wins path rules across file/media/patch surfaces, and explicitly decide whether exec is sandboxed, out of scope, or tracked separately before landing https://github.com/openclaw/openclaw/pull/60981 or a narrower replacement.

## Reproduction Assessment

Yes, by source inspection for the missing capability. Current main’s strict schema, public type, runtime policy, tests, and docs only model `tools.fs.workspaceOnly`, and shell execution is constructed without filesystem policy input.

## Solution Assessment

Unclear for the exact `tools.fileAccess` shape. A deny-wins allow/deny policy is directionally appropriate, but the maintainable solution likely belongs under `tools.fs` or a successor contract after exec semantics are settled.
