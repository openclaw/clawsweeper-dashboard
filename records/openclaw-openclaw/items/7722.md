## Summary

Keep open. Current main still has only the narrower `tools.fs.workspaceOnly` boolean and `apply_patch` workspace guard; rule-based allow/deny path config and any filesystem policy for arbitrary shell execution remain absent, while the active PathGuard PR is partial and unmerged.

## What This Changes

The issue requests a configurable filesystem access policy, `tools.fileAccess` or an equivalent surface, with allow/deny path rules enforced across read, write, edit, and exec tools.

## Best Possible Solution

Define a versioned `tools.fs` or successor ACL with deny-wins path rules for file/media/patch surfaces, plus an explicit exec strategy; then land, narrow, or replace the active PathGuard implementation with that contract.

## Reproduction Assessment

Yes for the missing capability. Source inspection shows the strict config schema and public type only accept `tools.fs.workspaceOnly`, while shell execution is constructed without a filesystem policy input.

## Solution Assessment

Unclear for the exact `tools.fileAccess` shape. A deny-wins allow/deny policy is directionally appropriate, but the maintainable solution likely belongs under `tools.fs` or a successor surface after exec semantics are settled.
