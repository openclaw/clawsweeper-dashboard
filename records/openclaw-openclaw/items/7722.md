## Summary

Keep open. Current main still has only the narrower `tools.fs.workspaceOnly` boolean and filesystem safety primitives; it does not expose the requested `tools.fileAccess`/allow-deny config contract or apply a filesystem path policy to shell execution.

## What This Changes

The issue requests configurable filesystem sandboxing with allow/deny paths, sandbox/full mode, and enforcement across file and exec tools.

## Best Possible Solution

Define a versioned `tools.fs` or successor policy with deny-wins allow/deny rules across file, local-media, and patch surfaces, then either design a defensible exec enforcement model or split exec into a tracked follow-up.

## Reproduction Assessment

Yes for the missing capability. Source inspection shows the strict config schema and public type only accept `tools.fs.workspaceOnly`, while exec construction still has no filesystem policy input.

## Solution Assessment

Unclear for the exact `tools.fileAccess` shape. A deny-wins allow/deny policy is directionally appropriate, but the maintainable contract likely belongs under `tools.fs` or a successor surface after exec semantics are settled.
