## Summary

Current main still provides only the coarse `tools.fs.workspaceOnly` boundary; granular allow/deny policy and shell-execution coverage remain unresolved across competing open proposals. This issue remains useful as the canonical security and product-contract discussion.

## What This Changes

Add configurable filesystem allow/deny boundaries for agent file operations and define how those restrictions apply to shell execution.

## Best Possible Solution

Define one canonical `tools.fs` policy with deny-wins path rules, per-agent inheritance, canonical-path and symlink semantics, explicit defaults and upgrade behavior, and either enforceable execution isolation or a clearly separate shell security boundary; then converge or close competing proposals around that contract.

## Reproduction Assessment

Yes. Current-main source gives a high-confidence reproduction: the strict schema and public config type accept only `tools.fs.workspaceOnly`, so the proposed granular keys are rejected and no equivalent path policy exists.

## Solution Assessment

No single active proposal is yet established as the best solution. The maintainable path is to select one security-reviewed policy contract and consolidate the PathGuard, roots, and alias work rather than landing parallel filesystem authorization models.
