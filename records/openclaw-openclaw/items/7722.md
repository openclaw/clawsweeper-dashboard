## Summary

Keep this issue open: current main and v2026.6.5 still expose only the binary `tools.fs.workspaceOnly` boundary, while the requested path rules and shell-execution coverage remain unresolved; two conflicting open PRs propose materially different partial designs and require a maintainer security/product decision.

## What This Changes

Add configurable filesystem allow/deny rules for agent file and shell tools without requiring container sandboxing.

## Best Possible Solution

Adopt one canonical, versioned `tools.fs` policy with deny-wins canonical path enforcement across all file-bearing tool and media paths, preserve or explicitly migrate the shipped `workspaceOnly` contract, and document exec as either isolation-backed or deliberately outside the file-policy guarantee.

## Reproduction Assessment

Yes, at high confidence from source: current main's strict schema accepts only `tools.fs.workspaceOnly`, and tool construction has no per-path policy or corresponding exec integration.

## Solution Assessment

No, the issue's exact `tools.fileAccess` shape should not be adopted as-is; the best fix belongs under the existing `tools.fs` contract and requires a deliberate, cross-tool security model rather than config-only path matching.
