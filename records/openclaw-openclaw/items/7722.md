## Summary

Keep open: current main and v2026.6.10 still expose only the narrower workspace-only filesystem guard, while the requested allow/deny path policy and exec strategy remain unmerged security/product work.

## What This Changes

Request a configurable filesystem sandboxing policy with allowed and denied paths that applies to file tools and exec.

## Best Possible Solution

Define a maintainer-approved `tools.fs` or successor filesystem policy with normalized deny-wins allow/deny rules across file tools, local media, and `apply_patch`, then either cover exec in the same boundary or explicitly split and track exec policy.

## Reproduction Assessment

Yes, at source level: the strict config schema only accepts `tools.fs.workspaceOnly`, so the reported `tools.fileAccess` and allow/deny keys remain invalid on current main.

## Solution Assessment

No for the current implementation: `workspaceOnly` is only a partial workaround, and the exact permanent policy shape needs maintainer security/product review before it becomes the best fix.
