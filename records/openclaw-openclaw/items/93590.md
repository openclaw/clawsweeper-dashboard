## Summary

Keep this PR open, but it is not merge-ready: the keyless Firecrawl endpoint works, yet the new `web_fetch` provider would become an implicit fallback for existing no-key setups, contradicting the opt-in claim and the current key-free provider policy.

## What This Changes

Adds a credential-free `firecrawl-free` web_search and web_fetch provider to the bundled Firecrawl plugin, makes Firecrawl client requests omit Authorization when keyless, and updates provider registration and contract tests.

## Best Possible Solution

Keep keyless Firecrawl explicit-only, and either add a tested web_fetch opt-in seam that mirrors the key-free web_search policy or omit the web_fetch provider until maintainers approve that behavior.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes from source: register a `requiresCredential: false` web_fetch provider with no keys, and current `resolveWebFetchProviderId` returns that provider during auto-detect; this review did not run a merged PR branch because the checkout is read-only.

## Solution Assessment

Is this the best way to solve the issue? No: the keyless request path is plausible, but registering keyless web_fetch without an explicit-only gate is the wrong default boundary for existing users.
