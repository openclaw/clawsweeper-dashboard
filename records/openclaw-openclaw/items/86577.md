## Summary

Keep this PR open: the file-backed per-account replay store is a plausible direction, but the patch still drops an in-flight duplicate before it can recover from a retryable release, and the supplied proof is test-only rather than real Discord behavior proof.

## What This Changes

The PR makes Discord inbound replay dedupe file-backed per account, passes the OpenClaw state dir into the Discord monitor, adds structured inbound outcome logging, and extends focused Discord monitor tests.

## Best Possible Solution

Keep the per-account persistent replay namespace, but make the Discord claim wrapper distinguish duplicate from inflight and retry once after an inflight release before dropping the delivery.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: ClaimableDedupe returns an inflight pending promise that release rejects, but Discord still maps inflight to false and drops the duplicate before queueing. I did not execute tests because this review was required to keep the checkout read-only.

## Solution Assessment

No: persisting committed replay entries is useful, but it is not enough for the stated retryable recovery path while inflight duplicates are still treated as terminal duplicates.

Full review comments:

- **[P1] Retry inflight duplicate claims after release:** `extensions/discord/src/monitor/inbound-dedupe.ts:82`
  - body: `createClaimableDedupe.claim` can return `inflight` with a pending promise, and `release` rejects that promise when a retryable worker failure releases the original claim. Handle `inflight` by waiting for one pending release and re-claiming before treating it as a duplicate.
