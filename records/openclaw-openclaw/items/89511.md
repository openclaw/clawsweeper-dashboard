## Label Justifications

- `P1`: The issue reports a Gateway crash during active user work that can leave a real channel session without a final reply.
- `impact:crash-loop`: The central failure includes an uncaught `RangeError` and Gateway restart/drain behavior.

- `impact:session-state`: Affected sessions can remain `running` without terminal model or session-ended events after the failure.

## Summary

Keep open: current main and the latest release still have the central failure shape. Pending final-delivery text can still be persisted without a size cap, session-store persistence still serializes that state through `JSON.stringify`, restart recovery remains marker-gated, and `RangeError: Invalid string length` is still treated as a fatal uncaught exception rather than a recoverable bounded failure.

## What This Changes

The issue reports long-running Gateway turns that can crash with `RangeError: Invalid string length` and leave Slack-facing sessions running without a final assistant reply.

## Best Possible Solution

Land a coherent core fix that caps persisted and replayed pending-final or diagnostic text before JSON serialization, handles oversized serialization failures as recoverable where possible, and terminalizes or notifies affected channel sessions after crash or restart.

## Reproduction Assessment

No live high-confidence reproduction was run. The current-main source path is reproducible by inspection: uncapped pending-final text can reach JSON session-store serialization, while restart recovery still depends on durable marking before recovery runs.

## Solution Assessment

No current implementation is the best complete fix. Related PRs address pieces of restart marking or pending-final cleanup, but the maintainable solution needs one boundary for text caps, fatal serialization handling, durable recovery state, and user-facing notification.
