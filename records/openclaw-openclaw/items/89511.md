## Summary

Keep open: current main and v2026.6.9 still retain the central crash shape because pending-final text can be persisted without a size cap and serialized through JSON.stringify, while RangeError: Invalid string length remains a fatal uncaught exception. Recent restart and pending-final fixes address important adjacent recovery races, but they do not prove the Gateway avoids the serialization crash or always delivers a final reply/retry notice after this failure.

## What This Changes

The issue asks OpenClaw to prevent Gateway RangeError crashes during long-running Slack/Gateway turns and to avoid leaving accepted sessions running without a final user-visible reply.

## Best Possible Solution

Land a maintainer-owned Gateway/session reliability fix that caps persisted pending-final or diagnostic payloads before JSON serialization, handles serialization RangeError as recoverable where possible, and terminalizes or notifies affected channel sessions after crash or restart.

## Reproduction Assessment

No live high-confidence reproduction was run. The current-main source path is reproducible by inspection: uncapped pending-final text can reach JSON session-store serialization, and the reported RangeError remains fatal if it escapes as an uncaught exception.

## Solution Assessment

No current implementation is the best complete fix. The landed restart and pending-final repairs are useful partial fixes, but the maintainable solution needs one agreed boundary for payload caps, serialization failure handling, durable recovery, and user-visible channel notification.
