## Summary

Keep open: current main still has the reported failure shape. Pending final-delivery text is persisted without a size cap, session-store persistence still serializes that data through JSON, restart recovery still only scans running sessions already marked `abortedLastRun`, and `RangeError: Invalid string length` is not handled as a benign uncaught exception.

## What This Changes

The issue reports a Gateway `RangeError: Invalid string length` during long-running Slack-originated turns that can leave sessions running without a final user-visible reply.

## Best Possible Solution

Land a bounded core fix that caps persisted and replayed delivery text, treats oversized serialization failures as recoverable where possible, and terminalizes or notifies affected channel sessions after crash or restart.

## Reproduction Assessment

No live high-confidence reproduction was run. The current-main source path is reproducible by inspection: uncapped pending final-delivery text can reach session-store JSON serialization, while restart recovery still depends on `abortedLastRun` being written before the crash.

## Solution Assessment

No current implementation is the best complete fix. Related PRs address pieces of pending-final replay or restart recovery, but the maintainable solution needs one coherent boundary for text caps, fatal serialization handling, durable recovery state, and user-facing notification.
