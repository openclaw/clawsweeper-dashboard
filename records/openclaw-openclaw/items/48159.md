## Summary

Keep open. Current main still does not have a hard per-session ephemeral/no-persistence mode; the current session config, persisted session row, Gateway session RPC schemas, transcript write paths, memory flush path, and context-engine/plugin surfaces do not carry a durable privacy flag or shared no-write gate. Related open issues such as #62436, #58713, #60745, and #48509 cover narrower passthrough, agent-to-agent, cleanup, or persistence-hook slices, but none supersede this cross-channel secure chat request.

## What This Changes

The issue requests a per-session secure/ephemeral mode that prevents transcripts, session metadata, compaction summaries, memory flushes, and plugin or context-engine indexes from persisting conversation content.

## Best Possible Solution

Keep this as a core product/security design item for an additive ephemeral session contract across config, Gateway protocol, SessionEntry, session creation/patch flows, transcript/session-store persistence, memory flush, compaction/context-engine lifecycle, plugin SDK visibility, channel defaults, commands, UI/status verification, and tests proving that no durable transcript, index, memory file, compaction summary, or surviving content metadata is created for ephemeral sessions.

## Reproduction Assessment

Not applicable as a feature request, but current-main verification is high-confidence. Source and docs checks show the requested flag cannot be configured or passed through the session RPC surface, while normal session creation, transcript append, compaction, and memory flush paths still target durable storage.

## Solution Assessment

Unclear for the exact implementation. The requested hard-gate direction is appropriate, but the best solution needs maintainer product/security design because it spans core session contracts, Gateway protocol, plugin APIs, memory/compaction behavior, channel defaults, and user-visible verification.
