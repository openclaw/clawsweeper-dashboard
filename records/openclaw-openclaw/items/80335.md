## Summary

Keep open. The throttling direction is plausible, but the PR currently drops buffered assistant/thinking deltas at a throttle-window boundary, leaves the direct abort cleanup path unwired, and only provides test/check output rather than real behavior proof from a running setup.

## What This Changes

The branch adds Gateway agent-event text throttling, cumulative buffering state, cleanup hooks, and focused Gateway tests across chat, abort, maintenance, and startup paths.

## Best Possible Solution

Keep the Gateway-owned throttle only after it preserves ordered deltas across throttle-window transitions, wires cleanup through every abort path, and includes after-fix proof from a real Gateway/run setup.

## Reproduction Assessment

Yes for the blocking PR regressions: the diff shows a buffered event being deleted at the next send after the throttle window, and the real abort caller still omits the new maps. I did not run tests because this is a read-only review.

## Solution Assessment

No. Gateway-side throttling can be the right owner boundary, but this implementation needs to flush or merge buffered deltas before fresh sends and make the cleanup state required across all abort callers.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Flush pending agent deltas before sending a fresh chunk:** `src/gateway/server-chat.ts:696-697`
  - body: When the throttle window has elapsed, this branch deletes the buffered event and sends only the current payload. A burst like `Hel` at 0 ms, `lo` at 50 ms, and `!` at 200 ms emits `Hel` and `!` but loses `lo` for SDK/raw agent-stream consumers; flush or merge the buffered payload before clearing it.
  - confidence: 0.95
- **[P2] Require abort cleanup maps in chat abort ops:** `src/gateway/chat-abort.ts:116-117`
  - body: Because the new maps are optional, `createChatAbortOps()` still does not pass them for direct chat abort/stop requests. Aborting a run with a buffered assistant/thinking event can leave stale state behind and the abort lifecycle can flush that text after cancellation.
  - confidence: 0.93
