## Summary

Keep open: current main and the latest release still leave the reported `telegram:slash:<id>` sandbox-explain path source-reproducible, because bare colon session keys are returned unscoped and channel inference only handles agent-scoped keys. The issue is the canonical tracker for the remaining unscoped Telegram slash-session gap; related target PRs were closed as incomplete, while live closing references still point at open cross-repo PR attempts.

## What This Changes

The issue reports Telegram slash sessions resolving to `channel: (unknown)` in `openclaw sandbox explain`, which blocks configured `tools.elevated.allowFrom.telegram` authorization for elevated host commands.

## Best Possible Solution

Land a narrow fix that scopes bare channel-prefixed session keys before inference and uses static bundled-channel normalization for built-in channels, with a regression test asserting `channel: telegram` and `allowedByConfig: true` for `telegram:slash:<id>`.

## Reproduction Assessment

Yes, by source inspection: `openclaw sandbox explain --session telegram:slash:<id>` stays unscoped, fails `parseAgentSessionKey`, leaves `channel` undefined, and therefore keeps `allowedByConfig` false. I did not run a live Telegram or local CLI reproduction in this read-only review.

## Solution Assessment

No current implementation does not fully solve it. The best fix is the combined narrow path from the closed PR reviews: normalize bare channel-prefixed keys to agent-scoped form, then resolve built-in channels through static metadata when the runtime registry is unavailable.
