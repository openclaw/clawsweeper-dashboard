## What This Changes

The issue requests that Telegram native slash sessions resolve to channel `telegram` so elevated `allowFrom` authorization and `sandbox explain` diagnostics can succeed.

## Best Possible Solution

Land one focused fix that makes `sandbox explain --session telegram:slash:<id>` resolve `telegram`, keeps agent-scoped Telegram native slash metadata behavior intact, and adds regression coverage for both channel resolution and elevated diagnostics.

## Reproduction Assessment

Yes, by source inspection. Passing `telegram:slash:<id>` to `sandbox explain` remains unscoped, fails `parseAgentSessionKey`, leaves `channel` undefined, and therefore keeps `allowedByConfig` false.

## Solution Assessment

No, current main is not the best solution because the raw documented Telegram slash key path still fails diagnostics. The safest path is to reconcile the open PRs into one narrow normalization or static-channel fallback fix with regression tests.
