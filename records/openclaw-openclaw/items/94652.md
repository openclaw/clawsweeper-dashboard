## Summary

Keep open. The PR targets a valid open Feishu message-loss bug and current main still has the `[interactive]` fallback, but the branch is not merge-ready because it imports `send.ts` into `bot-content.ts` while the existing bot test suite hoists a `./send.js` mock that does not provide the newly imported export.

## What This Changes

The PR exports the existing Feishu interactive-card parser from `send.ts` and calls it when formatting `interactive` sub-messages inside Feishu `merge_forward` content.

## Best Possible Solution

Extract or move the interactive-card parser into a neutral Feishu content helper used by both `send.ts` and `bot-content.ts`, add a merge_forward interactive-card regression test, and require redacted real Feishu proof before merge.

## Reproduction Assessment

Yes, source-level. Current main sends Feishu `merge_forward` sub-messages with `msg_type: "interactive"` through a formatter that falls through to `[interactive]`, and the PR's new test-mock conflict is visible from the existing static import and hoisted `./send.js` mock.

## Solution Assessment

No. Reusing the existing parser is the right direction, but importing `send.ts` into `bot-content.ts` crosses the mocked test boundary; a neutral shared helper plus regression proof is the narrower maintainable fix.
