## What This Changes

The PR exports the existing Feishu interactive-card parser from `send.ts` and calls it when formatting `interactive` sub-messages inside Feishu `merge_forward` content.

## Best Possible Solution

Move the shared interactive-card parser into a neutral Feishu content helper used by both paths, add merge_forward interactive-card regression coverage, and require redacted real Feishu proof before merge.

## Reproduction Assessment

Yes, source-level. Current main sends Feishu `merge_forward` sub-messages with `msg_type: "interactive"` through `formatSubMessageContent`, which has no `interactive` case and falls back to `[interactive]`.

## Solution Assessment

No. Reusing the existing parser is the right behavior, but importing `send.ts` from `bot-content.ts` crosses an existing hoisted mock boundary; a neutral helper plus regression test is the narrower maintainable fix.
