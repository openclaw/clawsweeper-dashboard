## Summary

Keep open: current main has partial stuck-session warning backoff, lane recovery, and trajectory flush bounding, but the diagnostic state can still remain `processing` after recovery finds no active work, and stalled `model_call` activity still classifies with `recovery=none`.

## What This Changes

The issue reports Gateway diagnostic sessions stuck in `processing` after CLI timeout or model backend failure, causing event-loop degradation, high CPU, and manual recovery.

## Best Possible Solution

Add diagnostics-owned recovery/reconciliation that clears stale `processing` entries after active-work checks pass and gives stalled model-call activity a bounded recovery path consistent with existing embedded-run safeguards.

## Reproduction Assessment

Yes, source-reproducible: create a diagnostic session state in `processing` without active embedded/reply/lane work and let the heartbeat exceed the stuck threshold, or emit `model.call.started` without a terminal event to produce `active_work_without_progress` with `recovery=none`. I did not run the long live Gateway scenario.

## Solution Assessment

No, current main is only a partial solution: warning backoff, stale-lane release, and trajectory write bounding reduce symptoms but do not clear the stale diagnostic processing entry or recover stalled model-call state. The narrow maintainable fix belongs in the diagnostic recovery/session-state path with regression tests.

## Repair Work Prompt

Fix the stale Gateway diagnostic session ghost from https://github.com/openclaw/openclaw/issues/77115. Stay inside the diagnostics/session recovery boundary: do not add a config option, do not change provider defaults, and do not broaden trajectory capture policy. Current main can leave `diagnosticSessionStates` in `processing` after `recoverStuckDiagnosticSession` finds no active embedded/reply/lane work, and `activeWorkKind=model_call` stalls classify as `recovery=none`. Add a narrow recovery/reconciliation path that marks or deletes stale processing diagnostic entries once active-work checks prove there is no real work left, and extend the existing bounded stalled-run recovery only where it is safe for model-call stalls associated with an embedded run. Preserve safeguards for active embedded runs, active reply work, and active lane tasks. Add regression coverage in `src/logging/diagnostic.test.ts`, `src/logging/diagnostic-stuck-session-recovery.runtime.test.ts`, and/or `src/logging/diagnostic-stuck-session-recovery.integration.test.ts` showing the ghost stops counting as active and that active work is not cleared prematurely. Add a changelog fix entry because this is user-facing Gateway recovery behavior. Validate with targeted diagnostic tests and then `pnpm check:changed` in Testbox per repo policy.
