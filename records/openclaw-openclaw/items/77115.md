## Summary

Keep open: current main has partial stuck-session backoff and lane recovery, but the in-memory diagnostic entry can still remain in `processing` with no active work because pruning only removes idle states and recovery does not mark no-active-work cases idle.

## What This Changes

The issue reports a ghost in-memory gateway session that stays stuck in `processing` after CLI timeout/SIGKILL, degrading event-loop health and blocking CLI commands.

## Best Possible Solution

Clear or idle stale diagnostic processing entries once recovery proves there is no active embedded/reply/lane work, with regression coverage that the heartbeat stops counting the ghost as active while preserving active-work safeguards.

## Reproduction Assessment

Yes, source-reproducible: create a diagnostic session state in `processing` without active embedded/reply work and let the heartbeat exceed the stuck threshold; current recovery can release the lane but leaves the diagnostic entry processing. I did not run the long live gateway stress scenario.

## Solution Assessment

No, current main is only a partial solution: warning backoff and lane recovery help, but they do not remove the ghost from the diagnostic state tracker. The safer repair is a narrow recovery-side state cleanup after active-work checks pass.
