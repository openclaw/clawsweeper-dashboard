## Summary

Keep open: current main has stuck-session warning backoff and lane recovery, but it still does not mark or remove the stale diagnostic `processing` entry after no-active-work recovery, so the reported in-memory ghost can remain visible to the heartbeat.

## What This Changes

Request automatic cleanup/recovery for a stuck in-memory Gateway session ghost that is absent from the session store and degrades CLI and event-loop health.

## Best Possible Solution

Teach stale-session recovery to mark or delete the diagnostic session state once active-work checks pass, with regression coverage for ghost removal and active-work preservation.

## Reproduction Assessment

Yes, source-reproducible: create a diagnostic session state in `processing` with no active embedded/reply/lane work, advance past `diagnostics.stuckSessionWarnMs`, and observe recovery without diagnostic-state cleanup. I did not run the long live Gateway stress reproduction.

## Solution Assessment

Yes, a narrow recovery-side diagnostic-state cleanup after active-work checks is the best next fix. The open trajectory-flush PR addresses related event-loop saturation, but not this stale in-memory diagnostic entry.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/77115 on current main. The bug is that a diagnostic session can remain in `processing` after stuck-session recovery proves there is no active embedded run, reply work, or lane task, so the heartbeat keeps treating the ghost as active/stuck. Implement the fix in the diagnostic recovery/state boundary: mark the stale diagnostic session idle or delete it only after the existing active-work safeguards pass. Do not clear real active embedded runs, active reply operations, active command-lane work, queued user work, or add a new config option. Add focused regression coverage in the diagnostic heartbeat/recovery tests proving the ghost stops being counted after no-active-work recovery and that active-work cases remain protected. Add a single-line `CHANGELOG.md` fix entry because this is user-facing Gateway stability. Related context: #77124 and #77133 cover trajectory flush event-loop saturation and should not be treated as the full fix for this issue.
