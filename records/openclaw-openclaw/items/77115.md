## Summary

Keep open: current main has partial liveness backoff and lane recovery, but source inspection still shows processing diagnostic ghosts can remain tracked and a pending model-call path can stay stalled with `recovery=none`.

## What This Changes

The report asks Gateway diagnostics to recover stuck in-memory processing sessions that are absent from session storage and causing CLI timeouts, high CPU, and repeated stuck-session heartbeats.

## Best Possible Solution

Clear or idle stale diagnostic processing entries after recovery proves there is no active work, and make the existing model idle-timeout/terminal diagnostic flow cover pre-stream model calls so active model-call bookkeeping cannot persist indefinitely.

## Reproduction Assessment

Yes, source-reproducible: put a diagnostic session in `processing` with no active lane/reply/embedded work, or emit a model-call start without a terminal event, then advance the heartbeat past the stuck threshold. I did not run the full 11.5-hour live Gateway scenario.

## Solution Assessment

No, current main is only a partial solution: backoff and lane recovery reduce noise, but they do not clear stale processing tracker entries and model-call acquisition can still sit outside idle-timeout recovery. The narrow fix is to repair those existing recovery/timeout contracts without adding a new policy knob.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/77115 on current main. Source inspection shows two narrow stale-session gaps: (1) `recoverStuckDiagnosticSession` can release or no-op a stale lane but leaves the diagnostic session state in `processing`, and `pruneDiagnosticSessionStates` only TTL-prunes idle states; (2) `streamWithIdleTimeout` does not guard a provider stream promise that never resolves before the async iterable exists, so `model.call.started` can leave `activeWorkKind=model_call` stuck with `recovery=none`. Implement the smallest existing-contract fix: after recovery proves no active embedded run/reply work/lane task remains, mark or remove the matching diagnostic session state so the heartbeat stops counting the ghost; make pre-stream model acquisition honor the existing idle timeout and emit terminal diagnostic model-call activity on timeout/error so run activity clears. Add regression coverage for both paths. Do not add a new config option, do not change trajectory caps/timeouts, and do not clear real active embedded/reply/lane work. Add a user-facing `CHANGELOG.md` fix entry.
