## Summary

Keep open: current main has trajectory-flush and embedded-run recovery improvements, but it still leaves source-backed paths where stale diagnostic processing/activity state can keep reporting a ghost session instead of reconciling to idle or clearing itself.

## What This Changes

The issue reports Gateway ghost sessions stuck in diagnostic processing state after CLI/provider failures, causing repeated liveness warnings, CLI timeouts, high CPU, and requiring manual cleanup or restart.

## Best Possible Solution

Reconcile stale diagnostic processing/activity state to idle or remove it after recovery proves no active work remains, and add regression coverage for no-active-work and stale model_call paths while preserving safeguards for live embedded runs, reply work, and active lane tasks.

## Reproduction Assessment

Yes, source-reproducible: create a processing diagnostic session older than the stuck threshold with no active embedded/reply/lane work and recovery can no-op without clearing the diagnostic entry. A model_call activity that never receives a terminal model/run event is also classified as stalled with recovery=none.

## Solution Assessment

No, current main is not the complete fix: it adds partial backoff/embedded-run and trajectory protections, but it does not reconcile the stale diagnostic bookkeeping that keeps the ghost visible. The narrow maintainable repair is lifecycle/recovery cleanup with focused regression tests, not a broad new timeout policy.
