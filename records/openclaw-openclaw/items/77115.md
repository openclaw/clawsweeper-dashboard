## Summary

Keep open: current main still has source-backed stale-session paths. Processing diagnostic entries are not pruned, recovery can release or no-op without clearing the diagnostic state, and a stalled `model_call` is classified as active work with `recovery=none`, matching the later report.

## What This Changes

The issue requests automatic Gateway recovery for ghost `processing` sessions that survive outside the session store and can degrade the event loop and CLI responsiveness.

## Best Possible Solution

Gateway diagnostics should clear stale `processing` bookkeeping after verified no-active-work recovery and use the existing extended abort-drain recovery for stalled model calls that belong to an active embedded run.

## Reproduction Assessment

Yes, source-reproducible: a `processing` diagnostic state is not TTL-pruned, and a session with `model.call.started` but no terminal model event becomes `session.stalled` with `activeWorkKind=model_call` while recovery is never invoked. I did not run the long live Gateway scenario.

## Solution Assessment

No, current main is only a partial solution. The narrow maintainable fix is recovery-side state reconciliation plus extending the existing no-progress abort path to stalled model calls, not a new config knob or a trajectory-only fix.
