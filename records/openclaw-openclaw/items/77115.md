## Summary

Keep open: current main has partial mitigation for related trajectory flush stalls and stuck-session warnings, but source inspection still shows stale diagnostic `processing` entries can survive recovery and continue counting as active work; the newer `model_call` data point also maps to a non-recoverable stalled-active-work path.

## What This Changes

The issue asks OpenClaw to recover stale in-memory gateway session bookkeeping so ghost processing sessions do not keep the gateway busy or block CLI/session work.

## Best Possible Solution

Clear or idle stale diagnostic processing entries after recovery proves no active work remains, reconcile terminal run/model activity, and preserve safeguards for real active embedded, reply, lane, tool, and model work.

## Reproduction Assessment

Yes, source-reproducible: create a diagnostic session state in `processing` with no active embedded/reply/lane work and let the heartbeat exceed the stuck threshold; recovery can release/no-op the lane while the diagnostic entry remains `processing`. The separate `model_call` path is also source-backed as stalled active work with no recovery eligibility.

## Solution Assessment

No, current main is only a partial solution: trajectory flush bounding and warning backoff help related symptoms, but they do not retire stale diagnostic processing state or recover the reported model-call stall class. The narrow maintainable fix is cleanup/reconciliation in the diagnostic recovery/activity layer, not a broad gateway restart workaround.
