## Summary

Keep open: current main still leaves a stale `processing` diagnostic entry possible after recovery finds no active work, so the reported ghost-session path remains source-reproducible.

## What This Changes

Report requests automatic cleanup for a Gateway diagnostic session stuck in `processing` after CLI timeouts so it cannot keep diagnostics and queue liveness degraded.

## Best Possible Solution

Clear or mark idle the stale diagnostic session state after recovery proves there is no active embedded, reply, or lane work, with regression coverage for both cleanup and active-work safeguards.

## Reproduction Assessment

Yes, source-reproducible: seed a `diagnosticSessionStates` entry as old `processing` with no active embedded/reply/lane work, then the heartbeat classifies it as stuck and recovery returns without clearing that diagnostic state. I did not run the reported 11.5-hour live Gateway stress case.

## Solution Assessment

No, current main is only a partial solution: it can release or no-op the lane but leaves the diagnostic entry `processing`. The narrow maintainable fix is recovery-side diagnostic state cleanup after active-work checks pass.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/77115. When stuck-session recovery proves there is no active embedded run, active reply work, or active lane task left, clear the corresponding diagnostic session state or mark it idle so the heartbeat no longer counts it as a `processing` ghost. Preserve active-work safeguards: do not clear active embedded runs unless the existing `allowActiveAbort` path has safely handled them, and do not clear `active_reply_work` or `active_lane_task` cases. Add regression coverage showing a stale `processing` diagnostic state with no active work is no longer reported/counts as active after recovery, and that active reply/lane work remains protected. Keep this focused on diagnostic recovery/state cleanup; do not change trajectory flush behavior or add a new config option. Add a user-facing changelog entry if the PR changes shipped behavior.
