## Summary

Keep open: current main has stuck-session warning backoff and lane recovery, but the diagnostic session entry can still remain in `processing` after recovery finds no active work, so the reported in-memory ghost is not fully resolved. The linked trajectory-flush issue and PR cover a related event-loop saturation symptom, not this diagnostic-state cleanup gap.

## What This Changes

The issue requests automatic cleanup/recovery for Gateway diagnostic session ghosts that stay `processing` after CLI timeouts and make the Gateway/CLI unresponsive.

## Best Possible Solution

Clear or mark idle stale diagnostic `processing` entries once recovery proves there is no active embedded/reply/lane work, with regression coverage that the heartbeat stops counting the ghost while preserving active-work safeguards.

## Reproduction Assessment

Yes, source-reproducible: create a diagnostic session state in `processing` with no active embedded/reply/lane work, advance past the stuck threshold, and recovery can no-op or release the lane while leaving the diagnostic entry processing. I did not run the long live Gateway stress scenario.

## Solution Assessment

No, current main is only a partial solution because it backs off warnings and can release lanes, but it does not clean the diagnostic state tracker. The narrow maintainable fix is recovery-side state cleanup after active-work checks pass.

## Repair Work Prompt

Fix the Gateway stuck-session ghost reported at https://github.com/openclaw/openclaw/issues/77115. Current main can classify a stale diagnostic session as `session.stuck` and run recovery, but `recoverStuckDiagnosticSession` only releases/no-ops the lane and never clears or idles the `diagnosticSessionStates` entry, so the heartbeat can keep treating the ghost as processing. Add a narrow recovery-side cleanup path after active embedded run, active reply work, and active lane task checks prove there is no active work; preserve safeguards for real active work and queued lane work. Add regression coverage showing a stale processing diagnostic entry stops counting as active/stuck after successful no-active-work recovery, plus existing lane recovery behavior remains intact. Do not change trajectory writer caps/timeouts or queued-file-writer behavior; related trajectory flush work is tracked separately by #77124/#77133. Add a user-facing changelog fix entry if runtime behavior changes.
