## Summary

Keep open: current main has stuck-session warning backoff and some stale lane recovery, but it still does not clear stale diagnostic `processing` state after no-active-work recovery and it leaves stalled `model_call` activity with `recovery=none`.

## What This Changes

The issue reports a gateway diagnostic/session bookkeeping ghost that remains `processing`, degrades the event loop, and blocks CLI/gateway work until manual intervention.

## Best Possible Solution

Make stuck-session recovery reconcile diagnostic state and activity to idle only after active embedded, reply, lane, tool, and model work checks prove there is no live work, then add regression coverage for stale no-active-work and stale model-call ghosts.

## Reproduction Assessment

Yes, source-reproducible: create a diagnostic session state in `processing` with no active embedded/reply/lane work, or a stale activity snapshot with `activeWorkKind=model_call`, and advance the heartbeat past the stuck threshold. Current main keeps the processing/activity bookkeeping alive instead of clearing it.

## Solution Assessment

No, current main is only a partial solution: backoff, liveness log filtering, and lane recovery help, but stale diagnostic state can still remain active. The narrow maintainable fix is recovery-side reconciliation with active-work safeguards and focused tests.
