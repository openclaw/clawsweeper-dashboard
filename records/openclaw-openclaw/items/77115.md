## Summary

Keep open: current main has partial stuck-session classification, backoff, and lane recovery, but source inspection still shows stale `processing` diagnostic state can survive no-active-work recovery and stalled `model_call` activity is classified with `recovery=none`.

## What This Changes

The issue asks for Gateway recovery when a ghost diagnostic session remains `processing`, blocks session work, degrades the event loop, and survives normal session-store cleanup.

## Best Possible Solution

Add narrow diagnostic recovery that idles or removes stale `processing` entries after no-active-work recovery, and reconcile terminal run/model activity so a crashed provider cannot leave a session permanently stalled with `recovery=none`.

## Reproduction Assessment

Yes, source-reproducible: create a diagnostic session in `processing` with no active work and let the heartbeat exceed the stuck threshold, or leave `model_call` activity active past the threshold. Current source shows the first path can no-op without clearing state and the second reports `recovery=none`.

## Solution Assessment

No, current main is not the complete solution: it classifies and backs off repeated diagnostics, but it does not clear the stale processing entry and does not recover or reconcile stalled `model_call` ghosts. The safer fix is a focused cleanup/reconciliation change in diagnostics rather than a new user-facing option.

## Repair Work Prompt

Fix the stuck diagnostic session ghost tracked by this issue. On current main, a diagnostic session can remain `processing` after stuck-session recovery finds no active work or after model-call/tool-call activity outlives terminal run/session events. Keep the change in diagnostics/session recovery/run-activity code; do not add a config option or broad provider policy. Add a helper or recovery-side path that marks/removes stale diagnostic state once recovery proves there is no active embedded run, reply work, or active lane task. Reconcile terminal `run.completed`, embedded-run end, model-call error/completion, and session idle signals so active model/tool bookkeeping cannot leave `recovery=none` forever after a crashed backend. Preserve safeguards for real active embedded runs, reply operations, and queued lane work. Add regression coverage for no-active-work cleanup and the stalled `model_call` ghost path; add a changelog entry if the fix is user-facing.
