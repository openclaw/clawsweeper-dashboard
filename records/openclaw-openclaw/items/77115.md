## Summary

Keep open: current main has partial trajectory/backoff/recovery improvements, but the diagnostic state tracker can still retain stale `processing` entries, and stalled `model_call` activity is explicitly classified with `recovery=none`.

## What This Changes

Fix Gateway diagnostic/session lifecycle recovery so in-memory processing ghosts from CLI timeout or backend failure do not keep the Gateway busy or block CLI calls.

## Best Possible Solution

Reconcile stale diagnostic processing state with actual embedded-run, model-call, tool-call, reply, and lane activity, then clear or idle only entries proven to have no live work while preserving active-work safeguards.

## Repair Work Prompt

Fix the remaining stuck-session ghost bug from https://github.com/openclaw/openclaw/issues/77115. Current main partially handles trajectory flush and embedded-run recovery, but stale diagnostic `processing` entries can survive after recovery finds no active work, and stalled `model_call` activity is classified with `recovery=none`. Implement a narrow recovery/reconciliation path in the logging diagnostics area so entries proven to have no active embedded run, reply work, command lane task, model call, or tool call are marked idle or removed, and ensure terminal run/model/tool events clear activity aliases correctly. Preserve safeguards for real active work and queued lane work; do not add a new config option, do not broaden trajectory policy, and do not change unrelated channel/plugin behavior. Add regression coverage for no-active-work recovery clearing the diagnostic state and for the stale model-call path from the additional issue comment. Add a user-facing changelog entry because this fixes Gateway/CLI degradation.
