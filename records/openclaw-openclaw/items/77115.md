## Summary

Keep open: current main has partial stuck-session backoff and embedded-run abort recovery, but it still leaves stale `processing` diagnostic entries alive and treats stalled `model_call` activity as non-recoverable, matching both the original ghost-session report and the later llama.cpp data point.

## What This Changes

The issue asks OpenClaw to recover in-memory processing session ghosts that outlive active work/session-store state and degrade Gateway responsiveness.

## Best Possible Solution

Add targeted diagnostic-state reconciliation that marks or deletes stale processing entries after safe no-active-work recovery, and clear stale model-call activity when the owning run has completed or is safely abort-drained.

## Reproduction Assessment

Yes, source-reproducible: a diagnostic session state left in `processing` with no active work remains outside normal pruning, and a stale `model_call` activity is classified as stalled with no recovery path. I did not run the long live Gateway stress scenario.

## Solution Assessment

No, current main is not yet the full fix: backoff and embedded-run abort recovery help, but they do not reconcile the diagnostic state tracker or stale model-call activity. The narrow maintainable repair is to clean those bookkeeping records only after the existing active-work safeguards pass.
