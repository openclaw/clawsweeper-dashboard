## Summary

Keep open: current main has partial mitigations for diagnostic warning backoff, embedded-run abort recovery, and trajectory flush pressure, but source inspection still shows stale `processing` diagnostic entries can remain counted as active and stalled `model_call` activity still has `recovery=none`.

## What This Changes

The issue requests automatic cleanup and recovery for Gateway sessions that remain stuck in `processing` after CLI timeout or backend failure, causing high CPU, event-loop delay, and CLI timeouts.

## Best Possible Solution

Clear stale diagnostic session state once recovery proves there is no active work, and allow the existing extended stalled-run recovery to handle model-call stalls that mask an abortable embedded run, with focused regression coverage.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes, source-reproducible: a `processing` diagnostic state with no active embedded/reply/lane work remains in the map after recovery, and a stale `model_call` classifies as stalled with `recovery=none`.

## Solution Assessment

Is this the best way to solve the issue? No current complete fix exists on main; the narrow maintainable fix is to update the diagnostic recovery path and tests rather than add a new timeout knob or rely on gateway restarts.
