## Summary

Keep open: current main still has a source-backed stale diagnostic session path where `processing` entries can remain active after recovery finds no work, and the newer comment's stalled `model_call` path is classified with `recovery=none`.

## What This Changes

The issue requests automatic recovery for stale gateway session state that can leave CLI requests timing out and the gateway burning CPU until manual cleanup or restart.

## Best Possible Solution

Add bounded diagnostic recovery that reconciles stale `processing` state after active-work checks pass, handles lost terminal model-call activity, and preserves real in-flight work with regression coverage.

## Reproduction Assessment

Yes, source-level: create a diagnostic session in `processing`, leave it past `stuckSessionWarnMs`, and either provide no active work or emit `model.call.started` without a terminal event. Current main either logs no-active-work without clearing the state or classifies the model call as stalled with no recovery.

## Solution Assessment

No, current main is only a partial solution: warning backoff and lane reset exist, but they do not clear the stale diagnostic entry or recover the stalled `model_call` path. The narrow fix should reconcile diagnostic state after safeguards prove there is no active work, rather than adding a broad timeout knob.
