## Summary

Keep open: current main has newer stuck-session classification, backoff, and lane-recovery code, but source inspection does not prove it fully handles the reported ghost because processing diagnostic states are not evicted and recovery intentionally preserves lanes with active command work.

## What This Changes

The issue reports that a gateway operator session can remain in memory after CLI timeouts, causing repeated stuck-session diagnostics, event-loop delay, CPU load, and CLI hangs.

## Best Possible Solution

Add a focused regression fix so a stale processing diagnostic session with no authoritative active run/store backing is marked idle or evicted after recovery, and queued work is released without cutting off confirmed active work.

## Reproduction Assessment

No high-confidence live reproduction is available from this review. Source inspection does show a plausible current-main path where a stale `processing` diagnostic entry can remain in memory, but it does not prove the full CPU and CLI timeout cascade.

## Solution Assessment

Unclear. The current recovery/backoff direction is the right base, but the best fix should close the remaining ghost-state cleanup gap with targeted tests rather than broad cron or gateway redesign.

## Repair Work Prompt

Fix the gateway stuck-session ghost reported at https://github.com/openclaw/openclaw/issues/77115. Stay within the existing diagnostic/session recovery design: add regression coverage showing that a stale diagnostic session in `processing` with no authoritative active run/store backing does not remain as an in-memory ghost indefinitely and does not keep queued work blocked. Prefer marking/evicting the diagnostic session state after successful or no-active-work recovery, and preserve the current protection for confirmed active embedded runs or legitimate active command-lane work. Stop and ask for maintainer review before force-clearing active command-lane tasks without a reliable stale/zombie signal. Update the changelog because this is user-facing gateway reliability behavior.
