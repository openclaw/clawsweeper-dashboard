## Summary

Close as duplicate/superseded: the remaining stuck-session recovery scope was handled through the canonical recovery thread at https://github.com/openclaw/openclaw/issues/71127, and current main now covers the no-active-work ghost, stale model/tool active work, idle queued embedded-run stalls, and terminal-progress queued stalls. The newest Windows `memory.sync` data is useful, but its root memory-core rename failure is already tracked in https://github.com/openclaw/openclaw/issues/78640.

## What This Changes

The issue asks for automatic cleanup and recovery of Gateway sessions stuck in `processing` after CLI, model, or memory failures so they do not block channels, CLI calls, or the event loop indefinitely.

## Best Possible Solution

Keep the current-main diagnostic recovery implementation, use https://github.com/openclaw/openclaw/issues/78640 for the Windows memory-core rename failure, and ask for a fresh report only if a build containing current main still produces a stuck session with redacted logs.
