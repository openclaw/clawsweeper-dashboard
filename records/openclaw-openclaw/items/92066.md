## Summary

Keep this PR open, but the current patch is not safe to merge: it fixes the Workboard CLI symptom by letting the generic CLI path request the full default operator scope set, including admin, rather than preserving the documented `operator.write` boundary for dispatch.

## What This Changes

This PR removes the explicit Workboard dispatch CLI `operator.write`/`operator.read` scope override and adds a unit test asserting that dispatch delegates scope selection to the gateway runtime.

## Best Possible Solution

Fix the gateway/plugin descriptor classification so `workboard.cards.dispatch` is recognized as `operator.write`, or otherwise keep the Workboard dispatch CLI on a narrow write/read scope path while adding real CLI/Gateway proof.

## Reproduction Assessment

Yes. Source inspection shows current Workboard dispatch asks for only read/write scopes, while the server-side unclassified-method fallback requires `operator.admin`; the linked issue provides the matching released-user steps, though I did not run a live Gateway repro in this read-only review.

## Solution Assessment

No. The PR is a plausible symptom workaround, but the best fix is to make the gateway classify the Workboard method's declared `operator.write` scope or otherwise keep dispatch least-privilege rather than silently requesting admin-capable defaults.
