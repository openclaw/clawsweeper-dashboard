## Summary

Keep open: the PR adds a reasonable Codex plugin goal surface that current main lacks, but the latest head still introduces a merge-blocking latency regression by awaiting best-effort goal polling before returning a bound reply.

## What This Changes

The branch adds Codex `/goal` and `/codex goal` controls, goal RPC capability constants, terminal-goal system-event/heartbeat bridging, tests, slash-command docs, and a changelog entry.

## Best Possible Solution

Keep the plugin-local goal controls and bridge, but decouple terminal-goal observation from bound reply delivery while preserving untrusted system-event enqueueing.

## Reproduction Assessment

Yes for the review finding by source inspection: latest head awaits `notifyTerminalCodexGoalBestEffort` after collecting `replyText`, and that helper awaits `thread/goal/get` before the reply is returned. The PR feature itself is not a bug report.

## Solution Assessment

No; the Codex plugin boundary and command surface are appropriate, but an observational completion bridge should not sit on the user-visible reply path. A fire-and-forget or short-capped goal check is the narrower maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Do not block bound replies on goal polling:** `extensions/codex/src/conversation-binding.ts:417-427`
  - body: After the Codex turn has already completed, this awaits the goal bridge before returning the reply. The helper then calls `thread/goal/get` with the full app-server request timeout, which defaults to 60 seconds, so a slow or wedged goal RPC can delay every bound conversation reply even though the bridge is only observational. Return the reply first, or make the goal poll fire-and-forget with a short cap.
  - confidence: 0.86
