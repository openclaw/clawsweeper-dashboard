## Summary

Keep open: the PR still proposes Codex goal functionality that is absent from current main, but the latest head has a source-proven reply-latency regression and insufficient real behavior proof, so it should not merge as-is.

## What This Changes

The PR adds `/goal` and `/codex goal` Codex controls, bridges terminal Codex goal states into system events, updates docs/changelog, and adds focused Codex tests.

## Best Possible Solution

Keep the plugin-local goal command direction, but decouple or tightly cap terminal-goal polling from reply delivery and require real bound `/goal` or terminal-bridge proof before merge.

## Reproduction Assessment

Yes for the review finding: PR head awaits `notifyTerminalCodexGoalBestEffort` before returning the reply, and that helper calls `thread/goal/get` with the normal app-server request timeout. The PR feature itself is not a bug report.

## Solution Assessment

No; the command surface is plausible, but observational goal polling should not block user-visible reply delivery. A fire-and-forget bridge or short timeout would be the safer implementation.
