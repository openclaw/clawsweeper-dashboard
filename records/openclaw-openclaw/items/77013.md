## Summary

Keep open; the feature is useful but the current PR is not merge-ready because it promotes dynamic Codex/user-derived goal text into trusted system context, omits required user-facing release/docs updates, and lacks visible protocol support or graceful gating for the new goal RPCs.

## What This Changes

The PR adds `/goal` and `/codex goal ...` controls for bound Codex app-server conversations and sends a heartbeat-backed completion event when a bound Codex goal reaches a terminal state.

## Best Possible Solution

Keep the feature in the Codex plugin after terminal goal events are queued as untrusted, the new command is documented and changelogged, regression coverage includes the trust boundary, and goal RPC support is either proved or safely gated.

## Reproduction Assessment

Not applicable; this is a feature PR rather than a bug report. Source inspection shows current main only registers the existing `/codex` surface while the PR adds `/goal`, `/codex goal`, and the completion bridge.

## Solution Assessment

No; the plugin boundary is appropriate, but this is not yet the safest maintainable implementation because dynamic completion text is trusted and the new user-facing/protocol surfaces are not fully covered.
