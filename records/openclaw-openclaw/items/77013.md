## Summary

Keep this PR open. Current main still lacks the Codex `/goal` command and goal RPC surface, but latest PR head still has a PR-introduced latency blocker: bound replies wait for best-effort goal polling before returning to the user.

## What This Changes

The PR adds Codex plugin `/goal` and `/codex goal` controls, app-server goal RPC methods, terminal-goal system-event and heartbeat bridging, tests, docs, changelog text, and a Google Meet test typing adjustment.

## Best Possible Solution

Keep the plugin-local goal controls and bridge, but decouple terminal goal observation from user-visible reply delivery with a fire-and-forget or tightly capped poll plus regression coverage.

## Reproduction Assessment

Yes for the review finding by source inspection: PR head awaits `notifyTerminalCodexGoalBestEffort` after the turn collector completes, and that helper awaits `thread/goal/get` with the app-server request timeout. The PR itself is a feature, not a user bug report.

## Solution Assessment

No; the Codex plugin command surface is a reasonable place for the feature, but an observational completion bridge should not block the bound reply path. A non-blocking or short-capped goal check is the narrower maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Do not block bound replies on goal polling:** `extensions/codex/src/conversation-binding.ts:417-427`
  - body: After the Codex turn has completed, this awaits the goal bridge before returning the reply. The helper then calls `thread/goal/get` with the full app-server request timeout, which defaults to 60 seconds, so a slow or wedged goal RPC can delay every bound conversation reply even though the bridge is only observational. Return the reply first, or make the goal poll fire-and-forget with a short cap.
  - confidence: 0.86
