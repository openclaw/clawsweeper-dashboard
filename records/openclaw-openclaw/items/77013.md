## Summary

Keep open: current main does not have the native Codex goal bridge, and this PR is useful source work, but it should not merge as-is because it treats Codex/user-derived goal text as trusted system context and lacks the required changelog entry.

## What This Changes

The PR adds Codex `/goal` and `/codex goal` controls, goal get/set/clear app-server methods, terminal goal heartbeat events for bound Codex conversations, Codex tests, and a Google Meet test mock typing fix.

## Best Possible Solution

Keep the feature in the Codex plugin, but enqueue or wrap dynamic goal objective/reply content as untrusted, add regression coverage for the trust boundary, and add one Unreleased changelog entry before merge.

## Reproduction Assessment

Not applicable; this is a feature PR rather than a bug report. Source inspection shows current main only exposes `/codex`, while the PR adds the new `/goal` command and terminal-goal bridge.

## Solution Assessment

No; the Codex plugin is the right place for the bridge, but the current implementation crosses the system-event trust boundary and omits the required changelog. The safer solution is to keep static wake instructions trusted, if needed, while marking Codex/user-derived fields untrusted or wrapping them as data.

## Review Findings

Overall correctness: patch is incorrect

Full review comments:

- **[P1] Do not trust bridged Codex goal text:** `extensions/codex/index.ts:53`
  - body: The event body includes goal `objective` and `replyText`, both derived from the Codex thread/user content, but enqueues the result as `trusted: true`. System-event formatting turns trusted entries into `System:` lines, while comparable exec/ACP outputs use `trusted: false`, so this can promote prior model/user text into trusted instructions on the heartbeat turn. Mark the dynamic event untrusted or wrap/sanitize the dynamic fields before enqueueing.
