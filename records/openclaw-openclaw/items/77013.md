## Summary

Keep this PR open: the feature is not implemented on current main, and the PR is useful, but the latest diff introduces one concrete reply-latency bug that should be fixed before merge.

## What This Changes

The PR adds `/goal` and `/codex goal` controls for Codex app-server thread goals, documents/registers the command, and queues terminal-goal system events with heartbeat wake for bound Codex conversations.

## Best Possible Solution

Keep the plugin-local goal controls and terminal-goal bridge, but make terminal-goal polling non-blocking or tightly capped before merge.

## Reproduction Assessment

The feature request itself is not a bug report. For the PR finding, yes by source inspection: after turn completion, `runBoundTurn` awaits `notifyTerminalCodexGoalBestEffort`, which awaits `thread/goal/get` before returning the already-collected reply.

## Solution Assessment

No; the Codex plugin boundary and command surface are appropriate, but the completion bridge should not block user-visible replies on observational goal polling. Return the reply first, or fire-and-forget/cap the goal check with errors swallowed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Do not block bound replies on goal polling:** `extensions/codex/src/conversation-binding.ts:417-427`
  - body: After the Codex turn has already completed, this awaits the goal bridge before returning the reply. The helper calls `thread/goal/get` with the full app-server request timeout, which defaults to 60 seconds, so a slow or wedged goal RPC can delay every bound conversation reply even though the bridge is only observational. Return the reply before this check, or make the goal poll fire-and-forget with a short cap.
  - confidence: 0.84
