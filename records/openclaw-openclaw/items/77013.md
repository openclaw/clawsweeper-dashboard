## Summary

Keep open: the PR is useful but still not merge-ready because latest head keeps observational Codex goal polling on the bound-reply critical path, so a slow goal RPC can delay an already-collected user-visible reply.

## What This Changes

The PR adds Codex plugin `/goal` and `/codex goal` controls, app-server goal RPC methods, terminal-goal system event and heartbeat bridging, tests, docs, changelog text, and a Google Meet test typing adjustment.

## Best Possible Solution

Keep the plugin-local goal controls and bridge, but return bound replies independently of goal polling by making the terminal-goal check fire-and-forget or tightly capped, with regression coverage for a slow goal RPC.

## Reproduction Assessment

Yes for the review finding by source inspection: latest head awaits `notifyTerminalCodexGoalBestEffort` after collecting `replyText`, and that helper awaits `thread/goal/get` before the reply is returned. The PR feature itself is not a bug report.

## Solution Assessment

No; the Codex plugin boundary and command surface are appropriate, but an observational completion bridge should not sit on the user-visible reply path. A non-blocking or short-capped goal check is the narrower maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Do not block bound replies on goal polling:** `extensions/codex/src/conversation-binding.ts:417-427`
  - body: After the Codex turn has already completed, this awaits the goal bridge before returning the reply. The helper then calls `thread/goal/get` with the full app-server request timeout, which defaults to 60 seconds, so a slow or wedged goal RPC can delay every bound conversation reply even though the bridge is only observational. Return the reply first, or make the goal poll fire-and-forget with a short cap.
  - confidence: 0.86
