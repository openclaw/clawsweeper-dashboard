## Summary

Keep open: current main does not include this feature, and the PR is useful, but latest head still puts best-effort goal polling on the bound-reply return path.

## What This Changes

The PR adds Codex plugin `/goal` and `/codex goal` controls, goal RPC method names, terminal-goal event/heartbeat bridging, and docs/tests/changelog coverage.

## Best Possible Solution

Keep the plugin-local goal controls and bridge, but return bound replies independently of goal polling with a fire-and-forget or short-capped terminal-goal check and regression coverage.

## Reproduction Assessment

Yes for the review finding: source inspection of PR head shows the reply awaits `notifyTerminalCodexGoalBestEffort`, which awaits `thread/goal/get`. The PR itself is a feature, not a bug report.

## Solution Assessment

No; the Codex plugin boundary and command surface are plausible, but an observational completion bridge should not sit on the user-visible reply path. A non-blocking or tightly capped goal check is the narrower maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Do not block bound replies on goal polling:** `extensions/codex/src/conversation-binding.ts:417-427`
  - body: After the Codex turn has already completed, this awaits the goal bridge before returning the reply. The helper then calls `thread/goal/get` with the full app-server request timeout, which defaults to 60 seconds, so a slow or wedged goal RPC can delay every bound conversation reply even though the bridge is only observational. Return the reply first, or make the goal poll fire-and-forget with a short cap.
  - confidence: 0.86
