## Summary

Keep open: current main still lacks the Codex goal command/RPC surface, and the PR adds the intended plugin-local feature, but latest head still has a blocking latency regression because bound replies await observational goal polling before returning.

## What This Changes

The PR adds `/goal` and `/codex goal` controls for bound Codex thread goals, documents/registers the slash command, queues untrusted terminal-goal system events with heartbeat wake, and includes a Google Meet test-type fix.

## Best Possible Solution

Keep the plugin-local goal controls and untrusted terminal-goal bridge, but make terminal-goal polling non-blocking or tightly capped before merge.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? The feature itself is not a bug report; for the PR finding, yes by source inspection: `runBoundTurn` awaits `notifyTerminalCodexGoalBestEffort`, which awaits `thread/goal/get` after the reply text is already collected.

## Solution Assessment

Is this the best way to solve the issue? No; the Codex plugin boundary and command surface are appropriate, but the completion bridge should not block user-visible replies on observational goal polling.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Do not block bound replies on goal polling:** `extensions/codex/src/conversation-binding.ts:417-427`
  - body: After the Codex turn has already completed, this awaits the goal bridge before returning the reply. The helper calls `thread/goal/get` with the full app-server request timeout, which defaults to 60 seconds, so a slow or wedged goal RPC can delay every bound conversation reply even though the bridge is only observational. Return the reply before this check, or make the goal poll fire-and-forget with a short cap.
  - confidence: 0.84
