## Summary

Keep open: current main does not have the requested Codex goal command surface, but the latest PR head still puts best-effort goal polling on the bound-reply path and lacks required real behavior proof.

## What This Changes

The PR adds `/goal` and `/codex goal` controls, Codex app-server goal RPC wiring, terminal-goal system-event/heartbeat bridging, docs/changelog updates, command and binding tests, and a Google Meet test typing adjustment.

## Best Possible Solution

Keep the plugin-local goal controls and bridge, but move terminal-goal polling off the user-visible reply path or short-cap it, add regression coverage for a slow goal RPC, and require real setup proof after the fix.

## Reproduction Assessment

Yes for the review finding by source inspection: latest head awaits `notifyTerminalCodexGoalBestEffort` after `replyText`, and that helper awaits `thread/goal/get` using the runtime request timeout. The PR feature itself is not a bug report.

## Solution Assessment

No; the Codex plugin boundary and command surface are reasonable, but an observational completion bridge should not block the reply path. A fire-and-forget or tightly capped goal check is the narrower maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Do not await goal polling before returning bound replies:** `extensions/codex/src/conversation-binding.ts:449-459`
  - body: After the Codex turn has already produced `replyText`, this awaits the best-effort goal bridge before returning the reply. The helper calls `thread/goal/get` with the full app-server request timeout, which defaults to 60 seconds, so a slow or wedged goal RPC can delay every bound conversation reply even though the bridge is only observational. Return the reply independently, or make the goal poll fire-and-forget with a short cap.
