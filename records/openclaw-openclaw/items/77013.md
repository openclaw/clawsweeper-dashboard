The PR adds Codex app-server goal RPC controls, a top-level `/goal` alias, terminal-goal wakeups for bound Codex conversations, docs/tests, and a changelog entry.

## Best Possible Solution

Keep the existing core `/goal` surface as the session-goal owner, expose Codex-native goal controls under `/codex goal`, and bridge completion through current untrusted/session-safe event APIs with real bound-session proof.

Yes. Source inspection is enough to reproduce the main blocker: register the PR's Codex `/goal` plugin command, then send `/goal status`; `handlePluginCommand` runs before `handleGoalCommand`, so the core session-goal command is bypassed.

No. `/codex goal` is a plausible owner-local control path, but taking over top-level `/goal` is the wrong layer while current main already reserves that behavior for durable OpenClaw session goals.
