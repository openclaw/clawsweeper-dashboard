## Summary

Keep open. This is a fresh report against v2026.4.29, and current main still has the same relevant lock behavior: embedded runs can hold the session write lock for the configured run timeout window, the default run timeout is very long, and startup/doctor cleanup only removes locks judged stale rather than live locks owned by the current gateway process. The exact 50% CPU loop is not proven from source alone, but the reported lock payload and workaround are plausible enough that this should not be closed as already fixed.

## What This Changes

The issue asks OpenClaw to prevent gateway session `.jsonl.lock` files from remaining stuck after a hung task and restart, because the stuck lock causes high CPU until manually removed.

## Best Possible Solution

Keep this open and fix the session-lock lifecycle as a coordinated lock-loss and hung-run recovery path: active attempts should be abortable or notified when their lock is force-released, transcript writes after lock loss should be guarded, and gateway restart recovery should not resume or leave a session in a state that repeatedly contends on a live lock. The fix should preserve existing startup/doctor cleanup for genuinely stale locks and avoid blindly deleting current-process locks.

## Reproduction Assessment

Unclear. The report has concrete operational steps, environment, lock payload, and a manual workaround, but the checkout review did not reproduce the 50% CPU loop; a focused regression should simulate a live gateway-owned session lock plus repeated contention or restart recovery around a hung attempt.

## Solution Assessment

Unclear. The requested direction is right at a product level, but the safe implementation should not simply delete live locks or shorten stale cleanup; it should coordinate lock-loss notification, abort/cleanup, and transcript write guarding, likely with #44510 and #63819 considered together.
