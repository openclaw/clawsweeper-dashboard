## Summary

Keep open. Current main still does not reclaim the reporter's current-PID lock shape when the lock payload has a valid `starttime`, and an open PR explicitly linked with closing syntax is already proposing the narrow lock fix but has not merged.

## What This Changes

The issue asks OpenClaw to release or recover leaked gateway session `.jsonl.lock` files after shutdown or restart so the gateway does not keep contending and consuming high CPU until manual lock deletion.

## Best Possible Solution

Review and land #75822 or an equivalent narrow fix that classifies untracked current-process locks with matching `starttime` as reclaimable in both acquisition and startup/doctor cleanup, while preserving active `HELD_LOCKS` entries and coordinating any broader hung-run recovery with #44510 and #63819.

## Reproduction Assessment

Unclear. The issue provides concrete operational steps, a lock payload, and a manual workaround, and current source explains why a valid-starttime current-PID lock is not reclaimed; I did not live-reproduce the exact high-CPU loop.

## Solution Assessment

Yes for the current routing: keeping this issue open while #75822 is reviewed is the safest path. The narrow implementation should share the self-owned lock classification across acquisition and cleanup rather than closing the issue before the linked PR merges.
