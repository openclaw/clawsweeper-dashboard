## Summary

Keep this issue open: current main has several session-lock safeguards, but the reported v2026.4.29 symptom is not proven obsolete. The reported lock payload includes `pid`, `createdAt`, and `starttime`, while current orphan-self recovery explicitly refuses to reclaim current-PID locks that have a valid `starttime`, so a restart-created lock can still plausibly remain contended until manual deletion or stale-age cleanup.

## What This Changes

The issue requests that gateway restart and session-lock handling recover from stuck `.jsonl.lock` files without a high-CPU lock acquisition loop or manual lock deletion.

## Best Possible Solution

Add a focused core fix in the session write-lock layer that can safely reclaim a current-PID sidecar when the process-local held-lock map does not own that normalized session file, while preserving protection for genuinely active in-process locks. Cover it with targeted lock tests and keep broader write-guard/data-corruption work coordinated with #44510 rather than mixing large lifecycle changes into this repair.

## Reproduction Assessment

Yes for the lock-state bug: create a `.jsonl.lock` with the current PID, a valid current `starttime`, and no corresponding `HELD_LOCKS` entry, then attempt `acquireSessionWriteLock`; current main will not classify it as orphan-self. The exact 50% CPU symptom was not live-reproduced in this read-only review.

## Solution Assessment

Yes, the narrowest maintainable fix is in `src/agents/session-write-lock.ts` plus focused regression coverage. A broad force-cleanup flag or unconditional startup deletion would be riskier because valid active writers must remain protected.
