## Summary

Keep open. Current main still leaves the reported Google AI Studio Gemma shorthand unresolved: the community comment correctly points to `gemma-4-26b-a4b-it`, while OpenClaw still accepts `gemma-4-26b` as a forward-compatible Gemma id and would send it to Google as `models/gemma-4-26b`.

## What This Changes

The issue asks OpenClaw to make Google AI Studio Gemma 4 model selection work for the reported `gemma-4-26b` setup instead of returning Google’s `models/gemma-4-26b` 404.

## Best Possible Solution

Add a Google-owned exact normalization for `gemma-4-26b` to `gemma-4-26b-a4b-it`, or reject that shorthand before dispatch with a clear supported-model hint, without moving provider-specific behavior into Telegram or core routing.

Yes, for a source-level current-main path: configuring `google/gemma-4-26b` leaves the model id unchanged, resolves it as a forward-compatible Gemma model, and the native Google transport builds `models/gemma-4-26b`. I did not run a live Google API call.

## Solution Assessment

Yes. The narrowest maintainable fix is in the Google plugin's model-id normalization or validation surface; changing Telegram or generic core model routing would miss the provider-owned id contract.
