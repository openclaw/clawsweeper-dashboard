## Summary

Keep open. Current main still leaves the reported Google Gemma shorthand unresolved: `gemma-4-26b` is not a documented Google AI Studio model id, but OpenClaw accepts `gemma-*` dynamically and would send the bare shorthand to the native Google endpoint, matching the reporter's 404. Related PR #68996 covers a native-routing problem, not this alias/validation path.

## What This Changes

The issue asks OpenClaw to run a Google AI Studio Gemma 4 26B model selected as `gemma-4-26b` instead of returning a Google NOT_FOUND error.

## Best Possible Solution

Fix this inside the Google plugin/model setup surface by preventing unsupported Gemma shorthand from reaching dispatch, either by normalizing the exact known shorthand to `gemma-4-26b-a4b-it` or rejecting it early with a clear hint.

## Reproduction Assessment

Yes for the source-level runtime path: configuring `google/gemma-4-26b` passes through Google model normalization unchanged, resolves as a Gemma forward-compatible model, and reaches the native transport as `models/gemma-4-26b`. I did not run a live Google API call.

## Solution Assessment

Yes. A provider-owned Google model-id normalization or validation fix is the narrowest maintainable solution; changing Telegram or broad core routing would miss the setup/provider boundary.
