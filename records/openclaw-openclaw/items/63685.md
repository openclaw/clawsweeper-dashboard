## Summary

Keep open. Current main still lets the unsupported Google AI Studio shorthand `gemma-4-26b` pass through the Google provider without aliasing or early validation, so the reported `models/gemma-4-26b` 404 remains a real model-id handling gap with a known workaround (`gemma-4-26b-a4b-it`).

## What This Changes

The issue asks OpenClaw to make Google AI Studio Gemma 4 26B work when configured through the terminal setup instead of failing with Google `NOT_FOUND` for `models/gemma-4-26b`.

## Best Possible Solution

Keep the fix in the Google plugin/model-id boundary by mapping the known `gemma-4-26b` shorthand to `gemma-4-26b-a4b-it` or rejecting it before dispatch with a clear supported-id hint.

## Reproduction Assessment

Yes, source-level: configure `google/gemma-4-26b`, let provider model normalization and dynamic Gemma resolution run, and the native Google transport will request `models/gemma-4-26b`. I did not make a live Google API call.

## Solution Assessment

Yes. A provider-owned Google model-id normalization or validation fix is the narrowest maintainable solution; Telegram and core routing are downstream of the bad model id.
