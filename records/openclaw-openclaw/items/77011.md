## Summary

Keep open. Current main appears to have fixed the staged update, Claude CLI stale-resume, and cron pre-transcript rows, but the latest canary comment exposes a remaining cleanup failure: an invalid persisted sessionId such as `agent:main:main` still reaches `resolveSessionFilePath()` and can abort `sessions cleanup --fix-missing` before missing transcript rows are reported or pruned.

## What This Changes

The issue requests hardening around gateway self-updates, Claude CLI missing-transcript recovery, and cron/session cleanup rows that leave live chats stuck on stale transcripts.

## Best Possible Solution

Ship a narrow sessions cleanup/doctor fix that treats invalid persisted session IDs as corrupt cleanup candidates under `--fix-missing`, while keeping the existing update, Claude CLI, and cron hardening in place.

## Reproduction Assessment

Yes, for the remaining cleanup failure there is a high-confidence source reproduction: put an invalid stored `sessionId` like `agent:main:main` in a session store and run `sessions cleanup --fix-missing`. Current code will pass that value to `resolveSessionFilePath()`, whose validator rejects it before cleanup can complete.

## Solution Assessment

No. Closing as implemented is not the best terminal state while cleanup can still abort on corrupt persisted session IDs; the maintainable fix is a focused corruption-tolerance path with regression coverage.
