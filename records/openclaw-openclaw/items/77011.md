## Summary

Keep open. Current main has already hardened the package-update and stale Claude CLI resume paths, but the latest canary comment plus current source show a remaining sessions cleanup/doctor robustness problem around corrupt or invalid session-store rows, and the cron missing-transcript row behavior is not conclusively fixed.

## What This Changes

The issue reports that a failed gateway-triggered package update can leave stale session-store state, causing missing transcript resume loops and cron/session cleanup problems.

## Best Possible Solution

Keep the shipped update and Claude CLI hardening, add a focused cleanup/doctor robustness fix for invalid or missing transcript rows, then verify or split the cron/session-store row behavior if it remains.

## Reproduction Assessment

Partially yes: a corrupt sessions.json row with sessionId set to agent:main:main is enough to hit validateSessionId through --fix-missing cleanup and abort before missing rows are pruned. The original package-update failure and cron row creation were not live-reproduced in this read-only review.

## Solution Assessment

No, closing as implemented is not the best solution. Current main solves two important subpaths, but the remaining cleanup blocker should be fixed narrowly before this issue is considered resolved.

## Repair Work Prompt

Fix the remaining cleanup blocker from https://github.com/openclaw/openclaw/issues/77011. `openclaw sessions cleanup --all-agents --dry-run --fix-missing --json` must not abort when a sessions.json entry has an invalid/corrupt `sessionId` such as `agent:main:main` or a bad `sessionFile`; it should classify the row as missing/corrupt for --fix-missing preview/apply and continue reporting/pruning other missing transcript rows. Keep the existing update.run staging and Claude CLI missing-transcript recovery behavior unchanged, and do not add a config option. Add regression coverage for cleanup preview/apply and doctor state integrity so doctor warns or skips corrupt rows instead of crashing. If cron/session-store semantics for shell-only or delivery-none jobs require a broader behavior choice, stop at the cleanup robustness fix and call that out. Add a CHANGELOG.md Fixes entry because this changes user-facing CLI repair behavior.
