## Summary

Keep this PR open. It is a focused fix for the linked heartbeat active-hours scheduling bug, but the latest head is still incomplete because reload reuse only compares the raw activeHours object while the active-hours predicate also depends on the effective user timezone; the PR is also draft, maintainer-labeled, and not maintainer-modifiable.

## What This Changes

The PR adds an active-hours-aware phase-slot seek helper, wires heartbeat runner advancement and config reload scheduling through it, adds scheduler/e2e coverage, and updates the changelog.

## Best Possible Solution

Land the active-hours-aware phase seeking approach after schedule reuse invalidates on all effective active-hours inputs, including agents.defaults.userTimezone when the active-hours timezone is omitted, user, or invalid, while keeping runHeartbeatOnce as the final runtime guard.

## Reproduction Assessment

Yes. A fake-timer runner test can start with activeHours.timezone omitted or set to user, push nextDueMs far ahead under one agents.defaults.userTimezone, then hot-reload only userTimezone and observe that this PR preserves the stale nextDueMs.

## Solution Assessment

No. Seeking forward to the first in-window phase slot is the right narrow fix, but this PR is not yet the best complete solution because reload invalidation misses effective timezone changes and the new test file has unused locals to clean up.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Invalidate schedules when user timezone changes:** `src/infra/heartbeat-runner.ts:1743-1745`
  - body: This reuse key only compares the raw activeHours object, but isWithinActiveHours resolves omitted, user, and invalid activeHours.timezone values through agents.defaults.userTimezone. Hot-reloading userTimezone with the same activeHours can keep a far-future nextDueMs from the old timezone, so the timer does not move into the newly active window. Include the effective timezone inputs in the reuse check, or recompute when userTimezone changes.
  - confidence: 0.9

  - confidence: 0.78
