## Summary

Keep this PR open. The PR is draft and protected by the `maintainer` label, and the latest head still has a bounded reload correctness bug: schedules are invalidated when the raw `activeHours` object changes, but not when the effective timezone changes through `agents.defaults.userTimezone`.

## What This Changes

This PR adds active-hours-aware heartbeat phase seeking, wires it into runner advancement and config reload scheduling, adds scheduler coverage, and updates the changelog.

## Best Possible Solution

Land the active-hours-aware phase-seeking approach after making reload scheduling invalidate preserved due times for all effective active-hours inputs, including `userTimezone` when the active-hours timezone is omitted, `user`, or invalid, while keeping `runHeartbeatOnce` as the final guard.

## Reproduction Assessment

Yes. A fake-timer runner test can start with `activeHours.timezone: "user"` and a `userTimezone` that pushes `nextDueMs` far ahead, then hot-reload only `agents.defaults.userTimezone`; the PR keeps the old `prevState.nextDueMs` because the raw activeHours fields did not change.

## Solution Assessment

No. Seeking forward to the first in-window phase slot is the right narrow fix, but the proposed reload invalidation is incomplete because the active-hours predicate depends on more than the raw activeHours object.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Invalidate schedules when user timezone changes:** `src/infra/heartbeat-runner.ts:1743-1745`
  - body: This reuse key only compares the raw `activeHours` object, but `isWithinActiveHours` resolves omitted, `user`, and invalid `activeHours.timezone` values through `agents.defaults.userTimezone`. Hot-reloading `userTimezone` with the same activeHours object can keep a far-future `prevState.nextDueMs` from the old timezone, so the timer does not move into the newly active window. Include the effective timezone inputs in the reuse check, or recompute when userTimezone changes.
  - confidence: 0.88
