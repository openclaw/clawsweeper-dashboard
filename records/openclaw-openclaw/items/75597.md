## Summary

Keep open. The linked heartbeat active-hours bug is real and this PR takes the right general direction, but the current head is not ready to merge: schedule reuse still misses effective timezone changes from `agents.defaults.userTimezone`, and CI lint is failing on newly added code/tests.

## What This Changes

The PR adds an active-hours-aware heartbeat phase-slot seek helper, wires runner advancement and config reload scheduling through it, adds scheduler/e2e coverage, and updates the changelog.

## Best Possible Solution

Land the active-hours-aware phase seeking approach after schedule reuse invalidates on every effective active-hours input, including default user timezone changes, while preserving `runHeartbeatOnce` as the final runtime guard and keeping new tests/lint clean.

## Reproduction Assessment

Yes. Current main can be reproduced with fake timers using `every: "4h"`, an `Asia/Shanghai` active-hours window, and a quiet-hours restart; the PR-specific bug can be reproduced by hot-reloading only `agents.defaults.userTimezone` while activeHours remains omitted, `user`, or invalid.

## Solution Assessment

No. Seeking forward to the first in-window phase slot is the narrow maintainable fix, but this PR is incomplete until it recomputes on effective timezone changes and passes the repo lint gate.
