## Summary

Keep open. The PR targets a valid heartbeat scheduler bug from the linked issue, but it is draft, has the protected `maintainer` label, and the proposed reload path still preserves a stale future `nextDueMs` when `activeHours` changes while interval and phase stay the same.

## What This Changes

This PR adds active-hours-aware heartbeat phase seeking, wires it into heartbeat runner interval advancement and config reload scheduling, adds focused scheduler coverage, and updates the changelog.

## Best Possible Solution

Land the active-hours-aware scheduler direction, but make reload scheduling invalidate preserved due times when `activeHours` or other heartbeat scheduling inputs change, while keeping `runHeartbeatOnce` as the final execution guard and avoiding new public phase semantics in this narrow fix.

## Reproduction Assessment

Yes. A fake-timer runner test can start with an active-hours window that pushes `nextDueMs` far into the future, then call `runner.updateConfig` before that due time with activeHours removed or widened; the PR reuses the old `prevState.nextDueMs` before applying the new predicate.

## Solution Assessment

No. Seeking forward to the first in-window phase slot is the right core approach, but the proposed reload path is incomplete because operator changes to active hours do not take effect until the stale preserved due time expires.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Recompute due times when active hours change:** `src/infra/heartbeat-runner.ts:1729`
  - body: The reload path still calls `resolveNextDue(..., prevState)` before applying the active-hours seek, so it reuses `prevState.nextDueMs` whenever interval and phase match. If the old active-hours window already pushed `nextDueMs` far ahead, removing or widening activeHours through hot reload leaves the timer armed for that stale future slot instead of scheduling from the new config. Recompute from `now` when activeHours or other scheduling inputs change, or include those fields in the prev-state reuse key.
  - confidence: 0.88
