## Summary

Keep open. Current main still has the active-hours scheduling gap from the linked issue, and this draft PR is a focused fix candidate, but the proposed reload path can preserve a stale active-hours-adjusted due time after heartbeat config changes. The `maintainer` label and draft state also make this inappropriate for cleanup close.

## What This Changes

This draft PR adds active-hours-aware heartbeat phase seeking, wires it into heartbeat runner scheduling and config reload, adds scheduler/unit coverage, and updates the changelog.

## Best Possible Solution

Land an active-hours-aware scheduler fix that seeks phase slots for startup and interval advancement while also invalidating preserved `nextDueMs` when heartbeat scheduling inputs, especially `activeHours`, change during hot reload.

## Reproduction Assessment

Yes. Start the runner with an active-hours window that pushes `nextDueMs` far into the future, then call `runner.updateConfig` before that due time with activeHours removed or widened while keeping the same interval and phase; the proposed code reuses the old `prevState.nextDueMs` before applying the new predicate.

## Solution Assessment

No. Seeking to an in-window slot is the right direction, but the reload path should recompute from `now` or include active-hours scheduling fields in the previous-state reuse key so operator config changes take effect immediately.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Recompute due times when active hours change:** `src/infra/heartbeat-runner.ts:1729`
  - body: The new reload path still calls `resolveNextDue(..., prevState)` before applying the active-hours seek, so it reuses `prevState.nextDueMs` whenever interval and phase match. If the old active-hours window already pushed `nextDueMs` far ahead, removing or widening activeHours through hot reload leaves the timer armed for that stale future slot instead of scheduling from the new config, so heartbeat can stay dormant after the operator fixes the quiet-hours window. Recompute from `now` when activeHours/scheduling config changes, or include those fields in the prev-state reuse key.
  - confidence: 0.87
