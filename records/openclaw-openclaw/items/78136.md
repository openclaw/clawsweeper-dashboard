## Summary

Keep this open: current main still has a source-visible readiness gap while the command queue is marked as gateway-draining, and the reported duplicate in-process restart sequence is not clearly fixed by the existing reset path.

## What This Changes

The issue asks OpenClaw to prevent Docker in-process gateway restarts from leaving command work rejected while `/healthz` and `/readyz` still report healthy or ready.

## Best Possible Solution

Make restart drain state part of readiness and make duplicate in-process restart signals coalesce or no-op across shutdown/startup so a ready gateway always accepts new model/tool work.

## Reproduction Assessment

Unclear for the full 20-minute stuck-after-ready race because I did not run a live Docker/Raspberry Pi scenario. Source inspection does show a high-confidence current-main path where `gatewayDraining` rejects new work while readiness ignores that state.

## Solution Assessment

Yes, the best fix is a narrow lifecycle/readiness change rather than a broad Docker policy change: `/readyz` should go red while the queue is draining, and duplicate SIGUSR1 restarts should not be able to leave the queue marked draining after startup completes.
