## Summary

Keep open: current main now clears stale command-queue drain state on in-process restart, but gateway readiness still does not consult the command-queue drain flag, so the `/readyz` mismatch remains. The related readiness PR was closed unmerged and is not a safe superseding path.

## What This Changes

The issue asks OpenClaw to prevent Docker in-process restarts from leaving command work rejected while `/healthz` and `/readyz` still report healthy or ready.

## Best Possible Solution

Wire readiness to the command-queue restart-drain gate with a transient failing reason, keep `/healthz` as shallow liveness, and preserve the existing restart-iteration reset that clears stale drain state.

## Reproduction Assessment

Yes from source inspection: `markGatewayDraining()` rejects command work, while the readiness checker does not read `isGatewayDraining()` or any equivalent drain-state input. Full Docker reproduction was not run in this read-only review.

## Solution Assessment

No. Current main partially fixes stale drain recovery after in-process restart, but the narrow remaining fix is to make readiness reflect the active drain gate and avoid caching that transient state.
