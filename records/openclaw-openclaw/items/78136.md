## Summary

Keep open: current main still has the source-visible readiness mismatch. New command-queue work is rejected while the gateway drain flag is set, but `/readyz` is still based on startup/channel/event-loop readiness and the related fix PR was closed unmerged.

## What This Changes

The issue asks for Docker/in-process gateway restarts to clear command-queue drain state reliably and for `/readyz` to report not-ready while new model/tool work is rejected during drain.

## Best Possible Solution

Expose command-queue drain state to readiness as an uncached `gateway-draining` failure while keeping `/healthz` shallow, and cover both the drain accessor/readiness behavior and restart-loop reset/coalescing path with focused tests.

## Reproduction Assessment

Yes for the readiness mismatch: source inspection shows `markGatewayDraining()` makes new enqueues fail, while the readiness checker has no drain-state dependency. The full Docker duplicate-restart stuck state has credible reporter logs but was not live-reproduced here.

## Solution Assessment

Yes: the narrow maintainable fix is a command-queue drain accessor checked by readiness before cached channel readiness, plus regression coverage for restart-loop drain reset. A broad Docker restart-policy change is not needed for the first fix.
