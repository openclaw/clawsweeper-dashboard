## Summary

Keep open: current main still has the source-visible readiness mismatch. The command queue can reject new work while the gateway drain flag is set, but `/readyz` is still wired only to startup/channel/event-loop readiness; the narrow linked fix PR remains closed and unmerged.

## What This Changes

The issue asks that in-process gateway restarts clear or coalesce drain state correctly and that `/readyz` report not-ready whenever model/tool work is being rejected for restart draining.

## Best Possible Solution

Expose command-queue drain state to readiness as an uncached `gateway-draining` failure while keeping `/healthz` shallow, and add focused coverage for the accessor/readiness behavior plus restart-loop drain reset/coalescing.

## Solution Assessment
