## Summary

Keep open: current main still has a source-visible readiness mismatch. The command queue can reject new work while `gatewayDraining` is true, but `createReadinessChecker()` and the `server.impl.ts` wiring still have no drain-state input, and the only directly linked fix PR was closed unmerged.

## What This Changes

The issue requests that Docker/in-process gateway restarts clear command-queue drain state reliably and that `/readyz` report not-ready while the gateway is rejecting work due to restart drain.

## Best Possible Solution

Expose the command-queue drain state to readiness as an uncached `gateway-draining` failure while keeping `/healthz` shallow, then add focused regression coverage for drain readiness and restart reset behavior.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the readiness mismatch from source: set `gatewayDraining`, observe `enqueueCommandInLane()` reject, and inspect that current `/readyz` has no drain-state input. I did not run a live Docker reproduction of the full stuck-drain incident.

## Solution Assessment

Is this the best way to solve the issue? Yes for the readiness portion: adding a drain-state accessor and checking it before cached channel readiness is the narrowest maintainable fix. Duplicate restart/coalescing should be covered with a focused regression, without changing broader replay semantics unless maintainers choose that separately.
