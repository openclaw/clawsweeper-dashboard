## Summary

Keep open: the report is recent, detailed, and current main still has a source-visible readiness gap where command enqueue can reject because the gateway is draining while `/readyz` only checks startup/channel/event-loop readiness.

## What This Changes

The issue asks for Docker in-process gateway restarts to clear the command-queue drain state reliably and for readiness to fail while new model/tool work is rejected for restart draining.

## Best Possible Solution

Teach readiness about the command-queue draining state and harden duplicate in-process restart handling so a restarted gateway cannot report ready while model/tool enqueue is still blocked.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No live Docker reproduction was run, but source inspection shows the `/readyz` path can return ready without checking `gatewayDraining`, while enqueue rejects whenever that flag is true.

## Solution Assessment

Is this the best way to solve the issue? Yes: the narrow maintainable fix is to expose drain state to readiness and add focused regression coverage for duplicate in-process restarts, rather than changing Docker healthcheck semantics or adding a new config knob.
