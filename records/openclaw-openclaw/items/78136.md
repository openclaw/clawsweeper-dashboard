## Summary

Keep open: current main still has the readiness mismatch described here. The command queue can reject new work while `gatewayDraining` is true, but `/readyz` is still driven only by startup, channel, skip, cache, and event-loop checks; the only linked fix PR was closed unmerged.

## What This Changes

The issue asks for Docker in-process gateway restarts to clear drain state reliably and for `/readyz` to report not-ready while model/tool work is rejected for restart draining.

## Best Possible Solution

Expose the command-queue drain state to the readiness checker as an uncached `gateway-draining` failure while keeping `/healthz` shallow, and add focused regression coverage around restart drain and readiness behavior.

## Reproduction Assessment

No live Docker reproduction was run for the full stuck-drain incident. The readiness mismatch is high-confidence source-reproducible: enqueue rejects while `gatewayDraining` is true, and current `/readyz` has no drain-state input.

## Solution Assessment

Yes for the readiness portion: the narrow maintainable fix is to add an `isGatewayDraining()`-style accessor and check it before cached channel readiness. A separate duplicate-restart regression should confirm the drain flag is cleared after in-process restart iterations.
