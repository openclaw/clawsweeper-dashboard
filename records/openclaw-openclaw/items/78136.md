## Summary

Keep open: current main still has the source-visible readiness mismatch. The command queue can reject new work while `gatewayDraining` is true, but `/readyz` is still wired only to startup/channel/event-loop readiness and the linked fix PR remains closed unmerged.

## What This Changes

The issue reports that Docker in-process gateway restarts can leave the command queue draining while `/healthz` and `/readyz` continue to report healthy/ready.

## Best Possible Solution

Expose command-queue drain state to readiness as an uncached `gateway-draining` failure while keeping `/healthz` shallow, and cover both the accessor/readiness behavior and restart-loop drain reset with focused tests.

## Reproduction Assessment

Yes for the readiness mismatch: current source shows enqueues reject while `gatewayDraining` is true, and `/readyz` has no drain-state input. The full Docker duplicate-restart stuck state has strong reporter logs but was not live-reproduced in this read-only pass.

## Solution Assessment

Yes for the readiness portion: the narrow maintainable fix is a command-queue drain accessor checked before cached channel readiness. The duplicate-restart side should be handled with focused reset/coalescing regression coverage, not a broad Docker restart-policy change.
