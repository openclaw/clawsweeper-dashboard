## Summary

Keep open: current main still has a source-visible readiness mismatch. New work can be rejected while the command queue is in gateway-draining mode, but `/readyz` still has no input from that drain flag, and the only linked fix PR was closed unmerged.

## What This Changes

The issue asks OpenClaw to clear Docker/in-process restart drain state reliably and make `/readyz` fail whenever the gateway is rejecting new model/tool work due to restart drain.

## Best Possible Solution

Expose command-queue drain state to readiness as an uncached `gateway-draining` failure while keeping `/healthz` shallow, and add focused regression coverage for drain reset and `/readyz` behavior.

## Reproduction Assessment

Partially. There is no live current-main Docker reproduction of the full stuck-drain incident, but the readiness mismatch is high-confidence source-reproducible because enqueue rejects on `gatewayDraining` while `/readyz` has no drain-state input.

## Solution Assessment

Yes for the readiness portion: the narrow maintainable fix is an `isGatewayDraining()`-style accessor checked before cached channel readiness. The restart-loop side should be covered by a focused duplicate-restart/reset regression rather than a broad Docker restart policy change.
