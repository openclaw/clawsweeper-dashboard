## Summary

Keep open: this is a useful partial Feishu account-routing fix, but the patch only consults runtime session-binding records and still misses the configured agent/channel binding path described by the linked issue; real behavior proof is also absent.

## What This Changes

The PR threads `ctx.sessionKey` into Feishu resource tools and adds a resolver step that selects a unique runtime Feishu session binding before `channels.feishu.defaultAccount`.

## Best Possible Solution

Resolve a unique configured or runtime Feishu account before global `defaultAccount` with maintainer-approved precedence, keep explicit and contextual accounts first, and add focused configured-binding coverage plus redacted runtime proof.

## Reproduction Assessment

Yes, at source level: current main can route a Feishu tool call with no valid `ctx.agentAccountId` to `channels.feishu.defaultAccount` before any binding-aware account source. I did not run a live multi-tenant Feishu cron, relay, or heartbeat repro in this read-only review.

## Solution Assessment

No: the PR is a plausible partial fix, but using only runtime session-binding records misses the configured agent binding source described by the linked bug. The safer fix is to resolve a unique configured or runtime Feishu account through an SDK-safe seam before falling back to global `defaultAccount`.
