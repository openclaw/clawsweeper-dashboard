## Summary

Keep open: the reported fork-context loss is real, but this PR is not merge-ready because the newly written header+user fork file is cleared by the runner's no-assistant preparation path before the child context is built. The supplied proof shows the intermediate fork file exists, not that a forked child run retains the inherited turn.

## What This Changes

The PR removes the assistant-message gate from `writeBranchedSession` and adds a regression test for writing a forked transcript when the parent branch has only a user turn.

## Best Possible Solution

Make the fork writer and child-run preparation share one invariant: a returned fork file must exist and retain inherited no-assistant branch entries through the child run, with caller-level regression coverage.

## Reproduction Assessment

Yes, from source inspection: a parent transcript with a header plus user message and no assistant hits the current `writeBranchedSession` no-write gate, and the PR branch still needs full child-run coverage because preparation clears that same no-assistant shape. ## Solution Assessment

No. Writing the fork file is a necessary intermediate step, but the best fix must also preserve that inherited branch through `prepareSessionManagerForRun` or an equivalent runner boundary.
