Full review comments:

- **[P1] Scope task-event subscription before publishing it:** `src/plugin-sdk/task-events.ts:14-17`
  - body: This public SDK subpath exports raw task registry events and `TaskRecord` types to plugins. `TaskRecord` includes task text, errors, agent IDs, owner/session metadata, labels, and summaries, so shipping this without a scoped payload and privacy/stability contract creates an overly broad third-party API.
  - confidence: 0.9
- **[P1] Use the approved memory event or storage path:** `extensions/memory-core/src/task-outcome-recorder.ts:101-103`
  - body: The recorder writes task outcomes to `memory/.dreams/task-outcomes.jsonl` with plain `fs.appendFile`. Existing memory host events use a typed event schema and `appendRegularFile` with symlink-parent rejection, so this adds divergent state and bypasses the established filesystem safety boundary.
  - confidence: 0.88
- **[P2] Register cleanup for the global task listener:** `extensions/memory-core/index.ts:34`
  - body: `registerTaskOutcomeRecorder(api)` returns an unsubscribe function, but registration discards it. Because the listener lives in a process-global set, reload, disable, or reset paths can leave stale listeners active and duplicate outcome writes unless cleanup is wired through plugin lifecycle.
  - confidence: 0.85
