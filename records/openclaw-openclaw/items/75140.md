## Summary

Keep this PR open. Current main does not implement the proposed task-event SDK seam or memory-core task-outcome log, but the PR branch has merge-blocking lifecycle and config-snapshot bugs plus an unresolved retention/privacy concern for the new local outcome log.

## What This Changes

This PR adds a plugin SDK task-registry event subpath, fans registry observer events out to additional listeners, and wires memory-core to append terminal task outcome JSONL records for dreaming feedback.

## Best Possible Solution

Revise the PR so memory-core owns the recorder through runtime lifecycle cleanup, resolves current config inside the event listener, and ships the outcome log only with an explicit retention or configuration story approved for memory-core.

## Reproduction Assessment

Yes. On the PR branch, register memory-core twice or rebuild the plugin host after the first registration, then transition one task to a terminal state; the stale listener can append duplicate outcome rows. The stale-config bug is reproducible by changing the default agent or workspace after registration and emitting a later terminal task event.

## Solution Assessment

No, not as-is. The additive SDK seam is a plausible direction, but the maintainable implementation should register the unsubscribe with api.registerRuntimeLifecycle, use api.runtime.config.current() inside the listener, and settle retention or gating before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Register the recorder cleanup:** `extensions/memory-core/index.ts:34`
  - body: `registerTaskOutcomeRecorder(api)` installs a process-wide task-registry listener and returns an unsubscribe, but this call discards it. When the plugin registry is rebuilt or memory-core is disabled and enabled again, the old listener remains active, so terminal task outcomes can be appended multiple times using stale config/logger state.
  - confidence: 0.91
- **[P2] Resolve config at event time:** `extensions/memory-core/src/task-outcome-recorder.ts:123-129`
  - body: The task-registry listener is long-lived, but it resolves the default agent and workspace from registration-time `api.config`. After a config reload changes the default agent or workspace, later task outcomes can be written to the wrong workspace or skipped; use `api.runtime.config.current()` inside the listener before resolving paths.
  - confidence: 0.84
