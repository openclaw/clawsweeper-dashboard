## Summary

Keep this PR open. Current main does not already implement the task-outcome recorder, but the proposed patch is not merge-ready because the memory-core listener is process-wide and not cleaned up, it captures registration-time config in a long-lived handler, and the new local outcome log needs a retention/privacy decision.

## What This Changes

This PR adds a plugin SDK task-registry event subpath, fans task registry observer events to additional listeners, and wires memory-core to write terminal task outcome JSONL records for later dreaming feedback.

## Best Possible Solution

Keep the additive SDK seam only if memory-core manages the recorder as plugin-owned lifecycle state, resolves current config at event time, and either documents or gates the new task-outcome retention behavior before merge.

## Reproduction Assessment

Yes. Register memory-core twice in one process or replace the plugin host registry, then transition one task to a terminal state; because the first listener is never unsubscribed, both listeners can append outcome rows for the same event.

## Solution Assessment

No for merge as-is. The narrow maintainable fix is to register the unsubscribe with api.registerRuntimeLifecycle, use api.runtime.config.current() inside the event listener, and get maintainer sign-off on retention or configuration gating.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Register the recorder cleanup:** `extensions/memory-core/index.ts:34`
  - body: registerTaskOutcomeRecorder installs a process-wide task-registry listener, but the plugin entry discards the returned unsubscribe. When the plugin registry is rebuilt or memory-core is disabled and enabled again, the old listener remains active, so terminal tasks can be appended multiple times and the stale listener keeps using old config/logger state.
  - confidence: 0.91
- **[P2] Resolve config at event time:** `extensions/memory-core/src/task-outcome-recorder.ts:121`
  - body: The task-registry listener is a long-lived handler, but it resolves the default agent and workspace from registration-time api.config. If the runtime config changes after registration, later task outcomes can be written to the wrong workspace or skipped; use api.runtime.config.current() inside the listener before resolving the agent/workspace.
  - confidence: 0.82
