## Summary

Keep open: current main does not implement this feature, and the PR is not merge-ready because the diff leaves lifecycle cleanup, stale config resolution, SDK/docs/changelog, and retention/privacy blockers.

## What This Changes

The PR adds a plugin SDK task-registry event subscription subpath and wires memory-core to append terminal task outcome JSONL records for dreaming feedback.

## Best Possible Solution

Revise the PR so memory-core registers the task outcome listener through runtime lifecycle cleanup, resolves live config per event, documents the SDK/log behavior, adds the required changelog entry, and ships only after maintainers approve retention or redaction policy.

## Reproduction Assessment

Yes. On the PR branch, load or register memory-core twice and transition one task from running to a terminal state; the first listener remains subscribed and can append duplicate rows. The stale-config path is reproducible by changing the default agent or workspace after registration and then emitting a later terminal task event.

## Solution Assessment

No, not as-is. The additive SDK seam is plausible, but the implementation should bind the long-lived listener to existing host lifecycle cleanup, resolve live config inside the callback, and settle data retention before merge.
