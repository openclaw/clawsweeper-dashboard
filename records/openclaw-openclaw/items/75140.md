## Summary

Keep this PR open. Current main does not already provide the proposed task-events SDK seam or memory-core task-outcome recorder, but the branch is not merge-ready because the recorder listener is process-global without plugin lifecycle cleanup, resolves stale registration-time config, and introduces a retention-sensitive outcome log that needs maintainer approval.

## What This Changes

This PR adds a task-events plugin SDK subpath, fans task-registry observer events out to plugin listeners, and wires memory-core to append terminal task outcome JSONL records for later dreaming feedback.

## Best Possible Solution

Keep the additive SDK seam only if memory-core owns the recorder as lifecycle-managed plugin state, resolves current runtime config for each event, and either documents, expires, or gates the new outcome log before release.

## Reproduction Assessment

Yes. Register memory-core twice in one process or replace the plugin host registry, then transition a task into a terminal state; because the first listener is never unsubscribed, both listeners can append outcome rows for the same event.

## Solution Assessment

No for merge as-is. The narrow maintainable fix is to register the unsubscribe with api.registerRuntimeLifecycle, read api.runtime.config.current() inside the listener, and get maintainer sign-off on the new retention behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Register the recorder cleanup:** `extensions/memory-core/index.ts:34`
  - body: `registerTaskOutcomeRecorder` installs a process-wide task-registry listener, but the plugin entry discards the unsubscribe. When the plugin registry is rebuilt or memory-core is disabled and enabled again, the old listener remains active, so later terminal tasks can be appended multiple times with stale logger/config state.
  - confidence: 0.91
- **[P2] Resolve config at event time:** `extensions/memory-core/src/task-outcome-recorder.ts:123-129`
  - body: The task-registry listener is long-lived, but it resolves the fallback agent and workspace from registration-time `api.config`. If runtime config changes after registration, later task outcomes can be written to the wrong workspace or skipped; use `api.runtime.config.current()` inside the listener before resolving the agent/workspace.
  - confidence: 0.82
