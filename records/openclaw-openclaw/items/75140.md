## Summary

Keep this PR open. The task-event SDK seam is plausible, but the branch is not merge-ready because the memory-core listener is process-global without plugin lifecycle cleanup, captures registration-time config for later task events, and adds a default local outcome log that needs an explicit retention/privacy decision.

## What This Changes

Adds a plugin SDK task-registry event subscription subpath and wires memory-core to append terminal task outcome JSONL records for future dreaming feedback.

## Best Possible Solution

Revise this PR so the recorder is lifecycle-managed, resolves live config per event, documents the SDK/log contract, adds the required changelog entry, and lands only after maintainers approve the retention model.

## Reproduction Assessment

Yes. On the PR branch, register or reload memory-core twice and transition one task into a terminal state; the first process-global listener remains subscribed and can write duplicate or stale outcome rows. The stale-config path is reproducible by changing the default agent or workspace after registration and then emitting a later terminal task event.

## Solution Assessment

No, not as-is. A narrow additive task-event seam can be a maintainable solution, but the memory-core consumer should use the existing plugin lifecycle cleanup and live runtime config before merge, with docs/changelog and retention handled in the same change.
