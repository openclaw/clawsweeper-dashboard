## Summary

Keep this PR open. Current main does not implement the proposed task-events SDK seam or memory-core task-outcome log, but the PR is not merge-ready because the new process-wide listener is not lifecycle-managed, the long-lived handler uses registration-time config, and the new retained outcome log needs maintainer retention review.

## What This Changes

This PR adds a plugin SDK task-registry event subpath, fans task registry observer events out to additional listeners, and wires memory-core to append terminal task outcome JSONL records for dreaming feedback.

## Best Possible Solution

Revise the existing PR so memory-core owns the recorder through runtime lifecycle cleanup, resolves current config inside the event listener, and only ships the new outcome log with an explicit maintainer-approved retention or configuration story plus focused tests.

## Reproduction Assessment

Yes. On the PR branch, register memory-core twice or rebuild the plugin host after the first registration, then transition one task from running to a terminal state; the stale listener remains subscribed and can append duplicate outcome rows. The config bug is also reproducible by changing the default agent/workspace after registration and emitting a later terminal task event.

## Solution Assessment

No, not as-is. The additive SDK seam is a plausible direction, but the maintainable implementation should register the unsubscribe with api.registerRuntimeLifecycle, use api.runtime.config.current() inside the long-lived listener, and settle retention/gating before merge.
