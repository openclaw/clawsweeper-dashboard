## Summary

Keep this PR open. Current main does not already provide the proposed task-events SDK seam or memory-core task outcome recorder, but the PR branch introduces two concrete lifecycle/config bugs and a retention decision that should be resolved before merge.

## What This Changes

This PR adds a plugin SDK task-registry event subpath, fans task registry events out to additional listeners, and wires memory-core to append terminal task outcome JSONL records for dreaming feedback.

## Best Possible Solution

Revise the existing PR so memory-core owns the recorder through runtime lifecycle cleanup, resolves live config inside the listener, and ships the new outcome log only with an explicit retention or configuration story plus focused regression tests.

## Reproduction Assessment

Yes. On the PR branch, registering memory-core twice or replacing the plugin host leaves the first task-registry listener subscribed, then a later running-to-terminal task transition can append duplicate outcome rows. The stale-config bug is reproducible by changing the default agent/workspace after registration and emitting a later terminal task event.

## Solution Assessment

No, not as-is. The additive task-events SDK seam is a plausible direction, but the maintainable implementation should register the returned unsubscribe via api.registerRuntimeLifecycle, read api.runtime.config.current() inside the long-lived listener, and settle retention behavior before merge.

  - body: `registerTaskOutcomeRecorder(api)` returns an unsubscribe for a process-wide task-registry listener, but this call drops it.   - confidence: 0.91
- **[P2] Read live config inside the listener:** `extensions/memory-core/src/task-outcome-recorder.ts:123-129`
  - body: The task-registry listener is long-lived, but it resolves the fallback agent and workspace from registration-time `api.config`. After a config reload changes the default agent or workspace, later task outcomes can be written to the wrong workspace or skipped; resolve from `api.runtime.config.current()` inside the listener.
  - confidence: 0.84
