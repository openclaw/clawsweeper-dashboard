## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Scope task-event subscription before publishing it:** `src/plugin-sdk/task-events.ts:14-17`
  - body: This public SDK subpath exposes `addTaskRegistryEventListener` over raw task registry events. Any loaded plugin could observe task text, labels, errors, agent IDs, runtimes, and status transitions without a scoped capability, documented stability contract, or security model.
  - confidence: 0.9
- **[P1] Avoid direct workspace appends for outcome state:** `extensions/memory-core/src/task-outcome-recorder.ts:101-103`
  - body: The recorder writes persistent task outcomes to `memory/.dreams/task-outcomes.jsonl` with plain `fs.appendFile`. Existing memory events use `appendRegularFile` with symlink-parent rejection and current dreaming state is SQLite-backed plugin state, so this creates an unsafe and divergent persistence path.
  - confidence: 0.91
- **[P2] Register listener cleanup with the plugin lifecycle:** `extensions/memory-core/index.ts:31`
  - body: `registerTaskOutcomeRecorder(api)` returns an unsubscribe function, but the plugin registration discards it. Since this installs a process-global listener, plugin reload or disable paths can leave stale listeners active and duplicate or stale-write task outcomes; register the cleanup through `api.lifecycle.registerRuntimeLifecycle(...)` or keep the listener lifecycle-owned by core.
  - confidence: 0.84
