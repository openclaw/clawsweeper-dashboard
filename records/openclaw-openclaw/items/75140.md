## Summary

Keep this PR open. Current main does not already implement task-outcome recording, and the PR is useful, but it is not merge-ready because memory-core installs a process-wide task-registry listener without registering the returned unsubscribe for plugin cleanup; the new local outcome log also needs a retention/gating decision.

## What This Changes

This PR adds a plugin SDK task-events subpath, fans task-registry observer events to additional listeners, and wires memory-core to write terminal task outcome JSONL records for dreaming feedback.

## Best Possible Solution

Keep the additive SDK seam only if the memory-core recorder is lifecycle-managed, covered by a duplicate-listener or plugin-cleanup regression test, and paired with an explicit retention/gating decision for task labels and error text. The public SDK subpath should also be documented before merge.

## Reproduction Assessment

Yes. Load memory-core, rebuild or replace the plugin registry without a process restart, then terminalize a task; because the old listener is never unsubscribed, multiple listeners can append duplicate outcome rows and retain stale config/logger references.

## Solution Assessment

No, not as-is. The narrow maintainable fix is to register the recorder unsubscribe with api.registerRuntimeLifecycle cleanup, then add a focused regression test and resolve whether recording should follow dreaming.enabled or a separate retention policy.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Unsubscribe the recorder on plugin cleanup:** `extensions/memory-core/index.ts:34`
  - body: `registerTaskOutcomeRecorder` installs a process-wide task-registry listener, but this registration discards the returned unsubscribe instead of registering it with the plugin runtime lifecycle. When the plugin registry is rebuilt or memory-core is disabled and enabled again, the old listener remains active, so each terminal task can be appended multiple times and the stale listener keeps using the old config/logger. Register a lifecycle cleanup that calls the unsubscribe.
  - confidence: 0.91
