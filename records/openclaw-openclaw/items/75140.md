## Summary

Keep this PR open. Current main does not implement the proposed task-events SDK seam or memory-core task outcome recorder, and the PR still needs revisions for listener lifecycle cleanup, live config resolution, public SDK documentation, changelog coverage, and outcome-log retention review.

## What This Changes

The PR adds a plugin SDK task-registry event subscription subpath and wires memory-core to append terminal task outcome records for dreaming feedback.

## Best Possible Solution

Revise this PR so memory-core registers recorder cleanup through runtime lifecycle, resolves current config inside the listener, documents the new SDK/log behavior, adds the required changelog entry, and ships only after maintainers approve the retention or gating model.

## Reproduction Assessment

Yes. On the PR branch, load or register memory-core twice and then transition one task from running to a terminal state; the first listener remains subscribed and can append duplicate outcome rows with stale state. The config issue is reproducible by changing the default agent or workspace after registration and emitting a later terminal task event.

## Solution Assessment

No, not as-is. The additive SDK seam is a plausible direction, but the implementation should use existing lifecycle cleanup and live config seams before merge, with docs/changelog and retention handled in the same PR.

  - body: registerTaskOutcomeRecorder(api) returns an unsubscribe for the process-wide task-registry listener, but this call discards it.   - body: This listener is long-lived, but it resolves the default agent and workspace from registration-time api.config. After a config reload changes the default agent or workspace, later task outcomes can be written to the wrong workspace or skipped; read api.runtime.config.current() inside the callback before resolving paths.
