## Summary

Keep this PR open. Current main still lacks the proposed task-event SDK surface and memory-core task-outcome recorder, but the patch has blocking listener lifecycle/config bugs plus an unresolved retention/privacy decision for the new outcome log.

## What This Changes

This PR adds a plugin SDK task-event subscription subpath, fans task-registry events out to additional listeners, and wires memory-core to append terminal task outcome JSONL records for dreaming feedback.

## Best Possible Solution

Revise the existing PR so memory-core registers the recorder only for full runtime activation, owns the unsubscribe through runtime lifecycle cleanup, resolves current config inside the long-lived event listener, and ships the outcome log only with an explicit retention or configuration story plus focused tests.

## Reproduction Assessment

Yes. On the PR branch, run a non-activating plugin discovery or register memory-core twice, then transition one task to a terminal state; the stale/global listener can remain subscribed and duplicate writes. The config bug is reproducible by changing the default agent/workspace after registration and emitting a later terminal task event.

## Solution Assessment

No. The additive SDK seam is plausible, but this implementation is not yet the narrowest maintainable solution because it bypasses plugin activation/lifecycle cleanup and closes over stale registration-time config.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Gate and clean up the recorder listener:** `extensions/memory-core/index.ts:34`
  - body: `registerTaskOutcomeRecorder(api)` subscribes to the process-global task-registry listener during every `definePluginEntry` registration mode and discards the unsubscribe. Discovery/CLI metadata loads are supposed to be non-activating, and full reloads or disable/enable cycles can leave stale listeners that append duplicate rows; gate this to `api.registrationMode === "full"` and register the unsubscribe with `api.registerRuntimeLifecycle`.
  - confidence: 0.93
- **[P2] Resolve config inside the event listener:** `extensions/memory-core/src/task-outcome-recorder.ts:121-129`
  - body: The task-registry listener is long-lived, but it resolves the fallback agent and workspace from registration-time `api.config`. After config reload changes the default agent or workspace, later terminal task outcomes can be written to the wrong workspace or skipped; use `api.runtime.config.current()` inside the listener before resolving paths.
  - confidence: 0.84

## Security Review

Status: needs_attention
