## Summary

Keep open: the PR is a plausible feature direction, but the current patch still has lifecycle and live-config correctness issues, lacks after-fix real behavior proof, and needs a retention/privacy decision for the new outcome log.

## What This Changes

The PR adds a public task-events plugin SDK subpath and wires the memory-core plugin to write terminal task outcome JSONL records for later dreaming feedback.

## Best Possible Solution

Merge only after the recorder is lifecycle-cleaned, reads live config, documents the new SDK/log behavior, provides real behavior proof, and has an approved retention model.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: register or reload memory-core twice, then emit a running-to-terminal task transition; the old listener remains active, and config reloads can route later writes through stale api.config.

## Solution Assessment

No, not as-is. The additive SDK seam is reasonable, but the implementation needs lifecycle cleanup, live config inside the callback, public docs for the new contract, and a settled retention policy.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Register the recorder cleanup:** `extensions/memory-core/index.ts:34`
  - body: registerTaskOutcomeRecorder(api) returns an unsubscribe for the process-wide task-registry listener, but this call discards it. When the plugin host is rebuilt or memory-core is disabled and re-enabled, the old listener remains active and can append duplicate outcome rows using stale logger/config state.
  - confidence: 0.91
- **[P2] Read live config inside the listener:** `extensions/memory-core/src/task-outcome-recorder.ts:123-129`
  - body: This listener is long-lived, but it resolves the default agent and workspace from registration-time api.config. After a config reload changes the default agent or workspace, later task outcomes can be written to the wrong workspace or skipped; read api.runtime.config.current() inside the callback before resolving paths.
  - confidence: 0.86
- **[P3] Document the task-events SDK subpath:** `src/plugin-sdk/task-events.ts:1`
  - body: This adds a public openclaw/plugin-sdk/task-events subpath, but the PR only updates exports, the entrypoint list, and generated hashes. The plugin SDK boundary requires docs/plugins coverage for new public subpaths, so third-party plugin authors would not have the supported contract or lifecycle expectations documented.
  - confidence: 0.82
