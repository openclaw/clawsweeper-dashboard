## Summary

Keep open. The feature direction is plausible, but the PR should not merge as-is because it leaks a process-wide listener, reads stale registration-time config in a long-lived callback, lacks docs for the new public SDK/log behavior, has only test/CI proof, and still needs a retention/privacy decision for the new outcome log.

## What This Changes

The PR adds a plugin SDK task-registry event listener subpath and wires memory-core to write terminal task outcome JSONL records under each workspace for later dreaming feedback.

## Best Possible Solution

Merge only after the recorder is lifecycle-cleaned, reads live config, documents the new public SDK/log behavior, and maintainers approve or gate retention for the outcome log.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: register or reload memory-core twice, then emit one running-to-terminal task transition; the old global listener remains subscribed and can duplicate writes, while config reloads before later events can route writes with stale api.config.

## Solution Assessment

No, not as-is. The additive SDK seam is a reasonable shape, but the implementation should use lifecycle cleanup, resolve live config inside the callback, document the contract, and settle retention before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Register the recorder cleanup:** `extensions/memory-core/index.ts:34`
  - body: registerTaskOutcomeRecorder(api) returns an unsubscribe for the process-wide task-registry listener, but this call discards it. When the plugin host is rebuilt or memory-core is disabled and re-enabled, the old listener remains active and can append duplicate outcome rows using stale logger/config state.
  - confidence: 0.91

  - body: This listener is long-lived, but it resolves the default agent and workspace from registration-time api.config. After a config reload changes the default agent or workspace, later task outcomes can be written to the wrong workspace or skipped; read api.runtime.config.current() inside the callback before resolving paths.
  - confidence: 0.86
- **[P3] Document the task-events SDK subpath:** `src/plugin-sdk/task-events.ts:1`
  - body: This adds a public openclaw/plugin-sdk/task-events subpath, but the PR only updates exports, the entrypoint list, and generated hashes. The plugin SDK boundary requires docs/plugins coverage for new public subpaths, so third-party plugin authors would not have the supported contract or lifecycle expectations documented.
  - confidence: 0.82
