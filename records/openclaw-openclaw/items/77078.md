## Summary

Keep open. The PR addresses a plausible Talk relay/default problem, but the current patch lets any relay RPC rebind an active relay session, so a stale previous browser connection can regain control and close or act on the refreshed session; the new default/config also needs maintainer product review and user-facing docs/changelog coverage.

## What This Changes

The PR adds `talk.realtimeTransport`, defaults Google browser Talk to the Gateway relay in auto mode, allows relay sessions to rebind to a refreshed browser connection, adjusts Google PCM silence handling, updates tests, and refreshes generated config schema.

## Best Possible Solution

Land a version that makes the Google relay default an intentional documented behavior, transfers relay ownership only through a guarded handoff, and adds regression coverage for stale old-connection stop/control packets.

## Reproduction Assessment

Yes, from source inspection. In the PR head, `relayAudio` from `conn-2` rebinds the session, then a delayed `relayStop` from `conn-1` calls the same helper and closes that session; I did not execute tests in this read-only pass.

## Solution Assessment

No. Defaulting Google to the relay may be the right direction, but the implementation should preserve relay ownership or use an explicit handoff and should update the user-facing docs/changelog for the new config/default.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Restrict relay ownership transfer:** `src/gateway/talk-realtime-relay.ts:238-239`
  - body: This rebinds the session for every relay RPC, not only the refreshed browser's audio/control handoff. After `conn-2` takes over, any delayed `relayStop`, `relayMark`, or tool-result request from `conn-1` will call this same helper, move ownership back, and in the stop case close the live session. Keep the transfer guarded so stale connections cannot regain control.
  - confidence: 0.87
- **[P3] Document the new realtime transport default:** `src/config/types.gateway.ts:76-77`
  - body: The patch adds `talk.realtimeTransport` and changes omitted/auto Google behavior to `gateway-relay`, but the docs still describe Google Live browser Talk as receiving a browser WebSocket token and the PR has no changelog entry. Update the Talk docs/config help and changelog before merge so users can configure and diagnose the new default.
  - confidence: 0.8

## Security Review

Status: needs_attention

Summary: The diff broadens relay-session control across WebSocket connections and needs an ownership check before merge.

Concerns:

- **[medium] Relay sessions can be taken over by stale or unrelated connections:** `src/gateway/talk-realtime-relay.ts:238`
  - body: Changing `getRelaySession` to rebind on any mismatched `connId` lets any write-scoped connection that knows the relay UUID take over the session through audio, mark, tool-result, or stop RPCs. The intended reconnect handoff should be scoped so old or unrelated connections cannot regain control or close the session.
