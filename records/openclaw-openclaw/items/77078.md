## Summary

Keep open: the PR addresses a real Talk relay gap and useful Google routing direction, but it is not merge-ready because relay rebinds become bearer-token-only, the new public config option is not documented across config surfaces, and the required changelog entry is missing.

## What This Changes

The PR makes browser Talk default Google realtime sessions to the Gateway relay, adds a `talk.realtimeTransport` escape hatch, rebinds relay control across browser reconnects, and tweaks Google PCM16 silence/audio-end handling.

## Best Possible Solution

Keep the relay-default direction only if maintainers approve the default change, then land it with owner-bound relay reconnects, aligned config docs/help/schema surfaces, changelog coverage, and targeted Talk/Google relay tests.

## Reproduction Assessment

Yes, by source inspection. Current main still asks Google for a browser session before relay fallback, and the actual relay broadcaster drops `talk.realtime.relay` events because the event has no scope guard.

## Solution Assessment

No, not yet. The intended route can be maintainable, but the relay reconnect should be bound to a stable client identity and the new config/default behavior must be documented and changeloged before merge.

## Security Review

Status: needs_attention

Summary: Relay reconnect support currently broadens control from the original WebSocket connection to any write-scoped connection holding the relay id.

Concerns:

- **[medium] Relay session can be hijacked with a leaked id:** `src/gateway/talk-realtime-relay.ts:237`
  - body: The PR removes the `connId` ownership check and retargets relay events to the latest connection that sends the same `relaySessionId`; because relay events include conversation audio/transcripts and tool calls, reconnect should be limited to the same stable client/device/auth identity.
