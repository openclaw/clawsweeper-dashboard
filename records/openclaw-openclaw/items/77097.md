## Summary

Keep open: current main and the latest release still generate and accept bearer-equivalent Control UI media credentials in `?token=` query strings, so this is a live security-sensitive auth design issue rather than cleanup.

## What This Changes

The issue asks OpenClaw to stop placing reusable Control UI gateway/device credentials in `/__openclaw__/assistant-media` query strings and replace that with browser-compatible media auth.

## Best Possible Solution

Serve assistant media without putting reusable gateway or device credentials in request URLs, using a short-lived scoped mechanism that works with native browser media elements.

## Reproduction Assessment

Yes. Source inspection on current main shows the UI builds `/__openclaw__/assistant-media?...&token=<auth>` and the gateway accepts that query token; current tests also assert the same path, although I did not run tests because this review is read-only.

## Solution Assessment

Yes for the remediation direction: replace bearer-equivalent query parameters with a browser-compatible media auth mechanism, then remove query-token parsing. The exact signed-URL versus same-origin session/cookie design needs maintainer security review.
