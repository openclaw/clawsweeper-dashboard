## Summary

Keep open: current main still builds Control UI assistant-media URLs with a bearer-equivalent `token` query parameter and the gateway accepts that query value for the same media route, so this is a live security-sensitive auth design issue.

## What This Changes

Rework Control UI assistant media auth so reusable gateway, password, or device credentials are not exposed in URL query strings.

## Best Possible Solution

Serve assistant media through a short-lived, scoped browser-compatible auth mechanism, then remove reusable credential query parsing and URL generation.

## Reproduction Assessment

Yes. Source inspection on current main shows the UI builds `/__openclaw__/assistant-media?...&token=<auth>` and the gateway accepts that query token; current tests assert the same behavior, though I did not run them because this review is read-only.

## Solution Assessment

Yes for the remediation direction: replace reusable query credentials with scoped browser-compatible media auth, then remove query-token parsing. The exact signed-URL versus same-origin session/cookie design needs maintainer security review.
