## Summary

Keep open: the PR targets a real security-sensitive prompt-size gap, but the branch is incomplete against current main because it drops public prompt mutation fields, leaves resolver and CLI prompt-visible paths uncapped, and lacks after-fix real behavior proof.

## What This Changes

The branch adds a 50,000-character clamp and warning for selected plugin hook prompt fields, plus phase and legacy hook-runner truncation tests.

## Best Possible Solution

Implement one shared post-aggregation prompt-context budget at the resolver or prompt-preparation boundary for embedded and CLI paths, preserving all public mutation fields and covering queued, turn-prepare, heartbeat, phase, and legacy contributions.

## Reproduction Assessment

Yes, by source inspection: current main concatenates plugin-controlled prompt-visible text from multiple hook and queued-contribution paths without a shared cap. I did not run a live failing scenario in this read-only review.

## Solution Assessment

No. The direction is useful, but this PR clamps too early and too narrowly; the maintainable fix is a shared post-aggregation cap that preserves the complete prompt mutation contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Preserve all prompt mutation fields:** `src/plugins/hooks.ts:276-279`
  - body: After clamping, `runBeforePromptBuild` returns only `systemPrompt` and `prependContext`. Current main exposes `appendContext`, `prependSystemContext`, and `appendSystemContext`, so plugins using those public fields would silently lose prompt mutations before the resolver can apply them.
  - confidence: 0.93
- **[P2] Clamp the full resolver output:** `src/agents/pi-embedded-runner/run/attempt.ts:906-915`

Oversized prompt text can still bypass the mitigation unless the cap is applied after full resolver aggregation.
  - confidence: 0.9
