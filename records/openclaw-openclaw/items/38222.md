## Summary

Keep open: this is a plausible security/session-state hardening PR, but the branch is not mergeable as-is because it drops current prompt mutation fields, leaves uncapped prompt-visible bypass paths, and lacks after-fix real behavior proof.

## What This Changes

The PR adds a 50,000-character clamp for plugin hook `systemPrompt` and `prependContext` in hook runners plus one embedded-runner concatenation point, with truncation tests.

## Best Possible Solution

Implement one shared post-aggregation prompt-context budget in prompt preparation that preserves every public prompt mutation field across embedded and CLI paths, then cover it with focused tests and redacted real-run proof.

## Reproduction Assessment

Yes. Source inspection shows current main joins plugin-provided prompt-visible text from multiple hook and queued-injection sources without a shared cap; I did not run a live oversized-plugin flow in this read-only review.

## Solution Assessment

No. The PR is a useful hardening direction, but it clamps too early and too narrowly while dropping current prompt mutation fields; the safer solution is a shared post-aggregation budget helper.
