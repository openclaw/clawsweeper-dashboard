## Summary

Keep open. The branch addresses a real prompt-budget hardening gap, but it is not merge-ready against current main: the current plugin hook contract has more prompt-visible fields and prompt sources than the PR clamps, and the proposed `runBeforePromptBuild` return path would drop existing mutation fields. Current main still lacks a complete shared prompt-context clamp, so the work remains useful but needs a refreshed or replacement implementation.

## What This Changes

The PR adds a `MAX_HOOK_CONTEXT_LENGTH` clamp for selected plugin hook prompt strings, applies an additional embedded-runner `prependContext` clamp, and adds phase and legacy hook truncation tests.

## Best Possible Solution

Refresh this PR or land a successor that clamps every prompt-visible plugin contribution after shared aggregation, preserves all public mutation fields, covers embedded and CLI prompt preparation, emits actionable warnings, adds focused tests for phase and legacy hook paths, and includes the required changelog entry.

## Reproduction Assessment

Yes. Static inspection of current main is enough: a plugin can return oversized text through prompt-visible hook fields or queued/turn/heartbeat contribution paths, and the resolver/runners concatenate that text into prompts without a cap.

## Solution Assessment

No. The proposed patch is a useful start, but it is not the best fix because it clamps only selected fields and wires one combined clamp in the embedded runner instead of applying a shared post-aggregation limit across all prompt-visible hook outputs and runner paths.
