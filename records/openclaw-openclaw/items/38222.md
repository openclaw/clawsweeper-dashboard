## Summary

Keep open: the hardening target is valid, but the branch is conflicting and still caps only selected prompt fields and sources while current main accepts more prompt mutation paths; it also lacks required after-fix real behavior proof.

## What This Changes

The PR adds a 50,000-character clamp for plugin hook `systemPrompt` and `prependContext`, applies it to hook runners plus one embedded legacy/phase prepend concatenation, and adds hook truncation tests.

## Best Possible Solution

Implement a shared post-aggregation prompt-context budget in prompt preparation that preserves all public prompt mutation fields across embedded, CLI, queued injection, turn-prepare, heartbeat, `before_prompt_build`, and legacy `before_agent_start` sources.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main joins plugin-provided prompt-visible text from multiple hook and queued-injection sources without a shared cap; I did not run a live oversized-plugin flow in this read-only review.

## Solution Assessment

No. The PR is a useful hardening direction, but it clamps too early and too narrowly; the safer solution is one shared prompt-preparation budget that covers every prompt mutation field and source.
