## Summary

Keep open: the underlying prompt-injection size hardening is still not implemented on current main, but this PR is incomplete, merge-conflicting, security-sensitive, and lacks after-fix real behavior proof.

## What This Changes

The PR adds a hook context length constant/helper, truncation tests, and a combined embedded-runner `prependContext` clamp.

## Best Possible Solution

Implement one shared post-aggregation prompt-context limit over the full resolver output for embedded and CLI paths, preserving every public prompt mutation field and covering queued, turn-prepare, heartbeat, phase, and legacy contributions.

## Reproduction Assessment

Yes, by source inspection: current main concatenates plugin-controlled prompt-visible text from several hook and queued-contribution paths without a shared cap. I did not run a live oversized-plugin scenario in this read-only review.

## Solution Assessment

No. The direction is useful, but this PR clamps too early and too narrowly; the safer fix is a shared resolver or prompt-preparation boundary cap that preserves the full hook result shape.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Preserve all prompt mutation fields:** `src/plugins/hooks.ts:276-279`
  - body: After clamping, `runBeforePromptBuild` returns only `systemPrompt` and `prependContext`. Current main exposes `appendContext`, `prependSystemContext`, and `appendSystemContext`, so plugins using those public fields would silently lose prompt mutations before the resolver can apply them.
  - confidence: 0.93
- **[P2] Clamp the full resolver output:** `src/agents/pi-embedded-runner/run/attempt.ts:911-917`
  - body: The added final clamp only combines embedded `prependContext`. Current main also accepts `appendContext`, system-context fields, queued injections, `agent_turn_prepare`, heartbeat contributions, and the CLI resolver path, so oversized prompt text can still bypass this mitigation.
  - confidence: 0.9
