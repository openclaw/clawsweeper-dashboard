## Summary

Keep open. Current main still has unbounded prompt-visible plugin context, but this PR is not mergeable as-is because the partial clamp would drop existing hook fields and leave current prompt-injection surfaces bypassable.

## What This Changes

The PR adds `MAX_HOOK_CONTEXT_LENGTH`/`clampHookString`, clamps selected hook prompt fields plus embedded combined `prependContext`, and adds phase and legacy hook truncation tests.

## Best Possible Solution

Land a refreshed patch or successor that applies a shared post-aggregation cap to every prompt-visible plugin contribution, preserves all public hook fields, covers embedded and CLI paths, emits actionable warnings, tests the affected hook sources, and adds the required changelog entry.

## Reproduction Assessment

Yes. Source inspection is enough: current main concatenates oversized plugin-controlled prompt text through phase, legacy, queued, turn-prepare, and heartbeat paths before embedded and CLI prompt assembly without a cap.

## Solution Assessment

No. The proposed direction is useful, but this implementation is not the best fix because it clamps only selected fields and does not enforce one shared post-aggregation limit across all prompt-visible plugin outputs.

## Review Findings
