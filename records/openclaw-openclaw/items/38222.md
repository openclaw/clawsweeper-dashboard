## Summary

This is a valid security-hardening direction, but the PR is not merge-ready against current main: it drops current public prompt mutation fields, clamps only selected early paths instead of the shared resolved prompt context, and still lacks after-fix real behavior proof.

## What This Changes

The branch adds a 50,000-character clamp for plugin hook `systemPrompt` and `prependContext`, warns on truncation, clamps one embedded concatenation path, and adds hook truncation tests.

## Best Possible Solution

Implement one shared post-aggregation prompt-context budget in prompt preparation that preserves every public prompt mutation field across embedded, CLI, queued injection, turn-prepare, heartbeat, `before_prompt_build`, and legacy `before_agent_start` sources.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main joins plugin-provided prompt-visible text from multiple hook and queued-injection sources without a shared cap; I did not run a live oversized-plugin flow in this read-only review.

## Solution Assessment

No. The PR is useful hardening work, but this diff clamps too early and too narrowly; the maintainable fix should enforce a shared prompt-preparation budget after all prompt-visible sources are aggregated.
