## Summary

Keep this PR open, but it should not merge as-is: the hardening direction is useful, yet the current branch drops public prompt mutation fields and still leaves uncapped prompt-visible plugin paths after aggregation.

## What This Changes

The PR adds a 50,000-character clamp and warning for selected plugin hook prompt fields, plus tests for before_prompt_build and before_agent_start truncation.

## Best Possible Solution

Implement a shared post-aggregation prompt-context budget at the resolver or prompt-preparation boundary that preserves every public prompt mutation field and covers embedded plus CLI consumers.

## Reproduction Assessment

Yes, source-reproducible: current main aggregates queued, turn-prepare, heartbeat, before_prompt_build, and legacy before_agent_start prompt-visible strings without a shared cap, then embedded and CLI paths apply them. I did not run a live failing setup in this read-only review.

## Solution Assessment

No. The PR is a useful hardening direction, but it clamps too early and too narrowly, and one changed return path drops public prompt mutation fields.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Preserve all prompt mutation fields:** `src/plugins/hooks.ts:276-279`
  - body: After clamping, runBeforePromptBuild returns only systemPrompt and prependContext. Current main exposes appendContext, prependSystemContext, and appendSystemContext, so plugins using those public fields would silently lose prompt mutations before the resolver can apply them.
  - confidence: 0.94
- **[P2] Clamp after full prompt aggregation:** `src/agents/pi-embedded-runner/run/attempt.ts:911-917`

Oversized prompt-visible text can still bypass the mitigation unless the budget is enforced on the full resolved result.
  - confidence: 0.9
