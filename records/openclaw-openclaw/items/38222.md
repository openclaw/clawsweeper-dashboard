## Summary

Keep open: the PR is a plausible security-hardening direction, but current main has broader prompt mutation paths than this branch handles, and the external PR still lacks after-fix real behavior proof.

## What This Changes

The PR adds a 50,000-character clamp for plugin hook `systemPrompt` and `prependContext`, applies it in hook runners and an embedded-runner join, and adds hook truncation tests.

## Best Possible Solution

Implement a shared post-aggregation prompt-context budget in prompt preparation that covers embedded and CLI consumers and all prompt mutation sources while preserving current public hook fields.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main joins plugin-provided prompt-visible fields from multiple hook sources without a shared cap, though I did not run a live oversized-plugin flow in this read-only review.

## Solution Assessment

No. The PR is useful hardening, but it clamps too early and too narrowly; the safer solution is a shared prompt-preparation budget that preserves all current prompt mutation fields.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Preserve every prompt mutation field:** `src/plugins/hooks.ts:276-279`
  - body: After clamping, `runBeforePromptBuild` returns only `systemPrompt` and `prependContext`. Current main exposes `appendContext`, `prependSystemContext`, and `appendSystemContext`, so rebasing this branch as-is would silently drop those public prompt mutations before the resolver can apply them.
  - confidence: 0.94
- **[P2] Clamp the resolved prompt mutation result:** `src/agents/pi-embedded-runner/run/attempt.ts:911-917`

Oversized prompt-visible text can still bypass the mitigation unless the budget is enforced on the full resolved result.
  - confidence: 0.9
