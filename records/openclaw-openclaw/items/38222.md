## Summary

Keep open: the PR is a useful security-hardening direction, but the current branch is incomplete because it drops public prompt mutation fields, does not enforce a shared post-aggregation budget across embedded and CLI prompt paths, and lacks real behavior proof.

## What This Changes

The branch adds a 50,000-character clamp and warning for selected plugin prompt hook fields, related hook tests, and an embedded prepend-context join clamp.

## Best Possible Solution

Implement one shared post-aggregation prompt-context budget at the resolver or prompt-preparation boundary, preserving all public prompt mutation fields and covering embedded and CLI consumers.

## Reproduction Assessment

Yes, source-reproducible: current main joins hook, queued, turn-prepare, and heartbeat prompt-visible strings without a shared cap, then embedded and CLI paths apply them. I did not run a live failing setup in this read-only review.

## Solution Assessment

No. The direction is useful, but this branch clamps too early and too narrowly; the maintainable fix is a shared resolver/preparation budget that preserves the full prompt mutation contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Preserve all prompt mutation fields:** `src/plugins/hooks.ts:276-279`
  - body: After clamping, `runBeforePromptBuild` returns only `systemPrompt` and `prependContext`. Current main exposes `appendContext`, `prependSystemContext`, and `appendSystemContext`, so plugins using those public fields would silently lose prompt mutations before the resolver can apply them.
  - confidence: 0.94

Oversized prompt-visible text can still bypass the mitigation unless the budget is enforced on the full resolved result.
  - confidence: 0.9
