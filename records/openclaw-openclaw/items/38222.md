## Summary

Keep open. Current main still lacks any hook-injected prompt-context length enforcement, so the underlying security/robustness problem is real, but this PR is stale against the current prompt-hook pipeline and has a blocking regression because the new post-processing drops existing prompt mutation fields.

## What This Changes

The PR adds a 50,000-character clamp and warning for selected plugin hook prompt strings, applies an extra embedded-runner combined `prependContext` clamp, and adds phase/legacy hook truncation tests.

## Best Possible Solution

Refresh this work against current main or land a successor that clamps every prompt-visible plugin injection field after shared aggregation, covers embedded and CLI prompt preparation, preserves existing hook priority and mutation-field semantics, emits actionable warnings, and includes focused tests plus a changelog entry if shipped.

## Reproduction Assessment

Yes. The current-main static path is enough to reproduce: a plugin can return oversized text through `appendContext`, system-context fields, queued next-turn injection, `agent_turn_prepare`, heartbeat contribution, `before_prompt_build`, or legacy `before_agent_start`, and the resolver/runners concatenate that text without a cap.

## Solution Assessment

No. The current PR is a useful start, but it is not the best fix because it clamps only part of the public prompt mutation contract and drops fields from `runBeforePromptBuild`; the safer fix is a shared post-aggregation clamp used by both runner paths.
