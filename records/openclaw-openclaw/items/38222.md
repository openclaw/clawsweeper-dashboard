## Summary

Keep open: the prompt-inflation risk is real on current main, but this PR is incomplete because it drops newer public prompt mutation fields and clamps only selected hook paths instead of the full resolved prompt-visible context.

## What This Changes

The PR adds a 50,000-character truncation helper for selected before_prompt_build and before_agent_start prompt strings, applies it to one embedded-runner join, and adds hook truncation tests.

## Best Possible Solution

Implement a shared post-aggregation prompt-context budget at the prompt resolver or preparation boundary that preserves every public prompt mutation field and covers embedded plus CLI consumers.

## Reproduction Assessment

Yes, source-reproducible: current main aggregates queued injections, agent_turn_prepare, heartbeat, before_prompt_build, and legacy before_agent_start prompt-visible strings without a shared cap, then embedded and CLI paths apply them. I did not run a live failing setup in this read-only review.

## Solution Assessment

No. The PR is a useful hardening direction, but it clamps too early and too narrowly; the safer solution is a shared budget after all prompt-visible hook/context aggregation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

  - body: After clamping, runBeforePromptBuild returns only systemPrompt and prependContext. Current main exposes appendContext, prependSystemContext, and appendSystemContext, so rebasing this branch as-is would silently drop those public prompt mutations before the resolver can apply them.
  - confidence: 0.94

Oversized prompt-visible text can still bypass the mitigation unless the budget is enforced on the full resolved result.
  - confidence: 0.9
