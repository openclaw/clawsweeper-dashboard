## Summary

Keep open for maintainer/security review: the underlying unbounded prompt-context problem is still present on current main, but this PR is not the right mergeable fix because it drops public prompt mutation fields, caps only selected paths, and lacks contributor real-behavior proof.

## What This Changes

The branch adds a 50,000-character clamp and warning for selected plugin hook prompt fields, plus phase/legacy hook truncation tests and one embedded prepend-context join clamp.

## Best Possible Solution

Implement one shared post-aggregation prompt-context budget at the resolver or prompt-preparation boundary, preserving the full public mutation contract across embedded, CLI, and harness paths.

## Reproduction Assessment

Yes, by source inspection: current main accepts prompt-visible text from plugin hooks, queued injections, turn-prepare, and heartbeat contribution paths, then feeds it into model prompts without a shared length cap. I did not run a live failing setup in this read-only review.

## Solution Assessment

No. The direction is useful, but the PR clamps too early and too narrowly; the safer fix is a shared post-aggregation budget that preserves all prompt mutation fields and covers every runtime consumer.
