## Summary

Keep open: the PR targets a real prompt-size hardening gap, but the current branch is dirty against main and the patch is incomplete because it drops public hook mutation fields and leaves other prompt-visible hook paths uncapped.

## What This Changes

The PR adds a 50,000-character clamp helper for selected plugin prompt hook strings, phase/legacy hook truncation tests, and one embedded combined prependContext clamp.

## Best Possible Solution

Land a refreshed patch or replacement that preserves the full hook result contract, applies one shared post-aggregation cap to every prompt-visible plugin contribution across embedded, CLI, and harness paths, adds focused regression coverage, and includes a changelog entry.

## Reproduction Assessment

Yes. Source inspection on current main is enough: plugin prompt hook results are concatenated through the shared resolver and embedded/CLI runners without a cap.

## Solution Assessment

No. The direction is right, but the current patch is not the best solution because it clamps only selected fields and one embedded join instead of enforcing the limit over the complete prompt-visible hook surface.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Preserve all prompt-build mutation fields:** `src/plugins/hooks.ts:276-279`
  - body: The new post-processing return in runBeforePromptBuild only includes systemPrompt and prependContext. Current main exposes appendContext, prependSystemContext, and appendSystemContext, so plugins using those public fields would lose their mutations when this branch is merged.
  - confidence: 0.94
- **[P2] Clamp the full prompt-visible hook surface:** `src/plugins/hooks.ts:277-278`
  - body: The cap is applied only to systemPrompt, prependContext, and one embedded combined prependContext path. Current main also feeds appendContext, system-context fields, queued injections, agent_turn_prepare, heartbeat_prompt_contribution, and the CLI resolver into prompts, so oversized hook text can still bypass the mitigation.
  - confidence: 0.9
- **[P3] Add the required changelog entry:** `src/plugins/hooks.ts:105`
  - body: This changes user-facing plugin hook behavior, but the PR changes only code and tests. OpenClaw policy requires a CHANGELOG.md entry for user-facing fixes before merge.
  - confidence: 0.86
