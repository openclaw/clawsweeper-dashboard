## Summary

Keep open: this is a plausible Control UI bug fix, but the PR is not merge-ready because after-fix real behavior proof is missing and the patch still leaves timed-out `chat.history` RPCs pending underneath the UI timeout.

## What This Changes

This PR changes Control UI `loadChatHistory` to race `chat.history` against a 10s timeout and apply late successful history responses with session/freshness guards.

## Best Possible Solution

Rebase onto current main and implement bounded `chat.history` recovery at the Gateway client/history boundary with request cleanup, focused regression coverage, and redacted real behavior proof.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection shows a never-settling `GatewayBrowserClient.request('chat.history')` would keep `loadChatHistory` pending, but current main already includes startup-specific mitigations that need to be accounted for.

## Solution Assessment

No. A fixed 10s `Promise.race` is a plausible UX mitigation, but it is not the best merge shape while it leaves the underlying RPC pending and conflicts with current main's newer retry/reconciliation path.
