## Summary

Keep open: current main improved startup responsiveness, but `loadChatHistory` still has a source-reproducible never-settling request path, and this PR still needs real behavior proof plus a cancellation/pending-cleanup revision before merge.

## What This Changes

The PR modifies Control UI `loadChatHistory` to race `chat.history` against a 10s timeout, clear the timer, and conditionally apply late history results.

## Best Possible Solution

Port a bounded history-timeout recovery onto current main with request cancellation or pending-entry cleanup, preserving current history limits, startup retry behavior, session-id handling, optimistic-tail preservation, and focused regression coverage.

## Reproduction Assessment

Yes, source-reproducible: current `loadChatHistory` awaits `state.client.request("chat.history")`, and the `awaitHistory` refresh test expects the call to remain pending until history resolves. I did not run a live browser/Gateway reproduction in this read-only review.

## Solution Assessment

No. The user-facing direction is valid, but this PR's UI-level `Promise.race` leaves the underlying Gateway pending request alive; the safer fix is cancellation or explicit pending cleanup on the timeout path.
