## Summary

Keep open. Current main still has a source-reproducible indefinite wait when `chat.history` never settles, but this PR is not merge-ready because the timeout path leaves the underlying Gateway request pending and the contributor has not provided after-fix real Control UI/Gateway proof.

## What This Changes

The PR changes Control UI `loadChatHistory` to race `chat.history` against a 10s timeout, clear the timer, apply late successful history responses behind guards, and invalidate late history on send.

## Best Possible Solution

Port a bounded history-timeout recovery onto current main with request cancellation or pending-entry cleanup, preserving current history limits, startup retry behavior, session-id handling, optimistic-tail preservation, and focused regression coverage.

## Reproduction Assessment

Yes, source-reproducible: a never-settling `chat.history` promise keeps `loadChatHistory` awaiting, and current tests expect an `awaitHistory` refresh to remain pending until history resolves. I did not run a live browser/Gateway reproduction in this read-only pass.

## Solution Assessment

No. A UI-level `Promise.race` mitigates the spinner, but the mergeable fix needs cancellation or pending cleanup of the underlying Gateway request and a careful port over current main's newer history guards and limits.
