## Summary

Keep open. The bug target is plausible and current main has related startup/history mitigations, but this PR still lacks real behavior proof and the timeout path leaves the underlying RPC pending after the UI gives up.

## What This Changes

The PR changes Web UI chat history loading to time out `chat.history` after 10 seconds and conditionally apply late successful results.

## Best Possible Solution

Move the timeout into a cancellable/client-owned request path or explicitly cancel the pending RPC when the UI times out, then rebase onto current startup/history handling and add focused controller or client coverage plus real behavior proof.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection shows the reported failure mode is plausible because `loadChatHistory` only clears `chatLoading` in `finally`, while `GatewayBrowserClient.request` can remain pending until a response or socket close.

## Solution Assessment

No. The current PR’s UI-level race dismisses the loader, but it does not clean up or cancel the underlying gateway request; a safer fix should pair the timeout with client-side cancellation or pending-entry cleanup and preserve current main’s startup/history contracts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.78

Full review comments:

- **[P2] Cancel timed-out chat.history RPCs:** `ui/src/ui/controllers/chat.ts:121`
