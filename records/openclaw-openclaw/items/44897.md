## What This Changes

The PR changes Control UI `loadChatHistory` to race `chat.history` against a 10s timeout, clear the timer, and optionally apply late history responses behind freshness guards.

## Best Possible Solution

Port bounded history-timeout recovery onto current main with request cancellation or pending-entry cleanup, preserving history limits, startup retry, session-id handling, optimistic-tail preservation, and focused regression coverage.

## Reproduction Assessment

Yes, source-reproducible: a never-settling `chat.history` promise keeps `loadChatHistory` awaiting, and current tests expect an `awaitHistory` refresh to remain pending until history resolves. I did not run a live browser/Gateway reproduction in this read-only pass.

## Solution Assessment

No. A UI-level `Promise.race` mitigates the spinner, but the mergeable fix needs cancellation or pending cleanup of the underlying Gateway request and a port over current main's newer history guards and limits.
