## Summary

Keep open. The bug still exists on current main, but the PR’s caller-side timeout leaves the underlying browser RPC pending and the external contributor has not provided after-fix real behavior proof.

## What This Changes

The PR wraps Control UI `loadChatHistory`'s `chat.history` request in a 10-second timeout, clears the timer, and applies guarded late successful history responses.

## Best Possible Solution

Add request-level timeout or cancellation for stalled browser RPCs, or a targeted `chat.history` wrapper that rejects and removes the pending request while preserving current main’s retry and history reconciliation behavior.

## Reproduction Assessment

Yes for a focused source-level path: a `loadChatHistory` call whose `state.client.request("chat.history")` promise never settles keeps `chatLoading` true because `finally` is not reached. I did not establish a live real-gateway reproduction in this read-only review.

## Solution Assessment

No. A caller-side `Promise.race` improves the spinner UX, but the underlying `GatewayBrowserClient` pending RPC remains live; the narrower maintainable fix should retire or cancel that pending request when the timeout wins.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.78

Full review comments:

- **[P2] Cancel timed-out chat.history RPCs:** `ui/src/ui/controllers/chat.ts:121`
  - body: The timeout only races the caller promise. `GatewayBrowserClient` removes entries from `pending` only when a response frame arrives or the socket closes, so retrying after a timeout can stack unresolved `chat.history` requests in the stalled-gateway case this PR targets. Add a request-level timeout/cancel path, or otherwise reject and remove the pending RPC when this timeout wins.
