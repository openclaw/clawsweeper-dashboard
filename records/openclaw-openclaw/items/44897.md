## Summary

Keep open. This PR targets a real Control UI hang class, but current main still does not provide a generic timeout/cancellation path for never-settling `chat.history` RPCs, and the PR's UI-level timeout leaves the underlying Gateway request pending while also missing contributor real behavior proof.

## What This Changes

The PR changes Control UI `loadChatHistory` to race `chat.history` against a 10s timeout and apply late successful history responses with session/freshness guards.

## Best Possible Solution

Implement bounded `chat.history` recovery at the Gateway client/history boundary with request cleanup or cancellation, preserve current main's retry/session/history-limit behavior, and add focused regression coverage plus redacted real behavior proof.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection shows a never-settling `GatewayBrowserClient.request('chat.history')` would keep `loadChatHistory` pending, and this PR's timeout path would leave the underlying pending RPC alive.

## Solution Assessment

No. A UI-level `Promise.race` is a plausible spinner mitigation, but the mergeable fix should clean up or cancel the underlying Gateway request while preserving current main's startup retry and history reconciliation path.
