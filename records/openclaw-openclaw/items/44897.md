## Summary

Keep open, but not merge-ready: current main still lacks a generic client-side timeout for a never-settling `chat.history` RPC, so the reported UI hang is plausible, but this branch needs real behavior proof and should not land with the unresolved pending-RPC cleanup issue.

## What This Changes

The PR wraps Control UI `loadChatHistory` in a 10s `Promise.race` timeout, late-applies successful history results under freshness guards, and invalidates late history on chat sends.

## Best Possible Solution

Preserve current main's bounded history and startup-unavailable handling, then add a timeout/recovery path where the Gateway browser client or chat-history helper can also clean up or cancel the pending RPC.

## Reproduction Assessment

No live high-confidence reproduction was established in this read-only review. Source inspection shows the current UI still awaits a `chat.history` request with no generic timeout, so a never-settling RPC can keep the loading state pending even though current main has server-side mitigations for common slow-history causes.

## Solution Assessment

No. The PR addresses the visible spinner, but the timeout is implemented at the caller while the underlying WebSocket RPC remains pending; a cleaner fix should put timeout/cancel semantics where the pending request can be removed and should preserve current bounded-history behavior.
