## Summary

Keep open: the WebUI timeout problem is still source-reproducible on current main, but this PR should not merge yet because it lacks real behavior proof and its timeout race leaves the underlying Gateway RPC pending when the timeout wins.

## What This Changes

The PR modifies Control UI `loadChatHistory` to race `chat.history` against a 10s timeout, clear the timer, and conditionally apply late history responses.

## Best Possible Solution

Port a bounded history-timeout recovery onto current main with request cancellation or explicit pending-entry cleanup, preserving current history limits, startup retry behavior, session-id handling, optimistic-tail preservation, and focused regression coverage.

## Reproduction Assessment

Yes, source-reproducible: current `loadChatHistory` awaits `state.client.request("chat.history")`, and the `awaitHistory` refresh test expects the call to remain pending until history resolves. I did not run a live browser/Gateway reproduction in this read-only review.

## Solution Assessment

No. The user-facing direction is valid, but this PR's UI-level `Promise.race` leaves the underlying Gateway pending request alive; the safer fix is cancellation or explicit pending cleanup on the timeout path, rebased onto current main.

## Review Findings
