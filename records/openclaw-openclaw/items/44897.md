## Summary

## What This Changes

The PR changes Control UI `loadChatHistory` to race `chat.history` against a 10s timeout, clear the timer, and conditionally apply late history responses.

## Best Possible Solution

Rebase onto current main and implement bounded history-timeout recovery with request cancellation or explicit pending-entry cleanup, preserving the newer history guards and adding focused regression coverage plus real Control UI proof.

## Reproduction Assessment

Yes, source-reproducible for the narrow bug: current `loadChatHistory` awaits `chat.history`, and the await-history refresh test expects the call to stay pending until history resolves. I did not run a live browser/Gateway reproduction in this read-only review.

## Solution Assessment

No. The user-facing timeout direction is valid, but this PR's UI-level race does not clean up the underlying pending Gateway request and must be rebased onto the newer current-main history logic.
