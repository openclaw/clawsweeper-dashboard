## Summary

Keep open. Current main still lacks a browser-side timeout for an unresolved Control UI `chat.history` RPC, but this PR should not land as-is because it races the UI promise without canceling or clearing the underlying pending gateway request and is now dirty against newer chat-history state guards.

## What This Changes

The PR adds a 10-second timeout around Control UI `loadChatHistory`, clears the timeout timer, applies late successful history responses under freshness guards, and bumps freshness when sending chat messages.

## Best Possible Solution

Add a request-level timeout or cancellation path for Control UI browser RPCs and use it for `chat.history`, so the visible loading state and the `GatewayBrowserClient` pending map are both cleaned up while preserving current main's history guards.

## Reproduction Assessment

Yes, at source level. A never-settling `GatewayBrowserClient.request("chat.history")` keeps `loadChatHistory` awaiting with `chatLoading` true because the browser client stores the pending request without any timeout path.

## Solution Assessment

No. The local `Promise.race` fixes the visible spinner but leaves the underlying gateway request pending; a request-level timeout/cancel path integrated with the current main `loadChatHistory` flow is the narrower maintainable solution.
