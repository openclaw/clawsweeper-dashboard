## Summary

Keep open. Current main still has the observable hang path, and this PR is a useful but not-yet-mergeable fix because its timeout path can leave browser RPCs pending and its delayed-success path bypasses current-main history reconciliation.

## What This Changes

The PR changes `ui/src/ui/controllers/chat.ts` to race the initial `chat.history` request against a 10-second timeout and conditionally apply delayed history responses after a timeout.

## Best Possible Solution

Land a current-main-native timeout path that exits the empty chat loading state with a targeted error, cancels or removes the timed-out browser RPC, and reuses the existing history filtering and optimistic-tail reconciliation for any safe delayed success while preserving startup retry, session/version guards, missing-scope handling, and active-stream behavior.

## Reproduction Assessment

Yes. Mock `state.client.request("chat.history")` with a never-settling promise; current main keeps `loadChatHistory` pending with `chatLoading` true, and the empty chat renderer shows the loading skeleton for that state.

## Solution Assessment

No. A bounded UI timeout is the right direction, but this PR is not the best current-main fix because it leaves the browser RPC pending and applies late successful history through a narrower path than normal history reconciliation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Cancel timed-out history RPCs:** `ui/src/ui/controllers/chat.ts:121-123`
  - body: When the timeout branch wins, the original `state.client.request("chat.history")` is still registered in `GatewayBrowserClient.pending`; that client only removes entries on a response frame or socket close/stop. Retrying after a stalled gateway can accumulate unresolved pending RPCs, so the timeout path needs an explicit cancel/remove seam.
  - confidence: 0.86
- **[P2] Reuse history reconciliation for late applies:** `ui/src/ui/controllers/chat.ts:108-110`
  - body: The late success handler writes history after only filtering silent assistant replies. Current main also hides heartbeat, synthetic, and internal history artifacts and preserves optimistic tail messages, so delayed history can reintroduce hidden entries or drop local optimistic content unless it calls the same reconciliation path as normal history loads.
  - confidence: 0.82
