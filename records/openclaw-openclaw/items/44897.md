## Summary

Keep open. Current main still has the user-visible hang path, but this PR should not merge as-is because the timeout leaves the browser RPC pending and the late-success branch bypasses current history reconciliation safeguards.

## What This Changes

The PR adds a 10-second `chat.history` race in Control UI `loadChatHistory`, tracks a global freshness token, and conditionally applies late history responses after a timeout.

## Best Possible Solution

Land a current-main-native timeout path that exits empty chat loading, cancels or removes the timed-out browser RPC, and reuses the shared history apply/reconciliation logic for any safe late success.

## Reproduction Assessment

Yes. Mock `state.client.request("chat.history")` with a never-settling Promise; current main keeps `loadChatHistory` pending with `chatLoading` true, and the empty chat renderer shows the loading skeleton for that state.

## Solution Assessment

No. A bounded UI timeout is the right direction, but this PR is not the best fix because it leaves `GatewayBrowserClient.pending` entries alive and applies late history through a narrower path than current main uses.

## Review Findings
