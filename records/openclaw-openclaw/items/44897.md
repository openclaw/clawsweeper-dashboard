## Summary

Keep open: current main still has the indefinite `chat.history` await, but this PR needs changes because the timeout path leaves underlying browser RPCs pending, bypasses current history reconciliation on late success, and lacks the required user-facing changelog entry.

## What This Changes

The PR modifies Control UI `loadChatHistory` to race the initial `chat.history` request against a 10-second timeout, clear the timer, and conditionally apply late successful responses.

## Best Possible Solution

Land a current-main-native fix that exits empty-chat loading on a bounded `chat.history` stall, forgets or cancels timed-out browser RPCs, reuses the shared history filtering/reconciliation path for safe late success, and adds regression tests plus a changelog entry.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: mock `state.client.request("chat.history")` with a never-settling Promise and `loadChatHistory` never reaches the `finally` that clears `chatLoading`. With no chat items, the current view renders the loading skeleton for that state.

## Solution Assessment

No. A bounded UI timeout is the right direction, but this PR is not the best fix until the browser RPC pending entry is cancellable or forgotten and late successes go through the same history reconciliation as normal loads.
