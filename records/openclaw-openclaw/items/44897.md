## Summary

Keep open. The underlying spinner hang is still source-reproducible on current main, but this PR is not mergeable as-is because it times out the UI without cleaning up the underlying Gateway RPC and still lacks contributor-provided real behavior proof.

## What This Changes

The PR changes Control UI `loadChatHistory` to race `chat.history` against a 10s timeout, apply late successful responses under freshness guards, and invalidate late history on chat send.

## Best Possible Solution

Implement bounded `chat.history` recovery with request cleanup or cancellation at the Gateway client/history boundary while preserving current main's retry, session, history-limit, and optimistic-tail behavior, then add focused regression coverage and redacted real behavior proof.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection shows a never-settling `GatewayBrowserClient.request('chat.history')` would keep current-main `loadChatHistory` pending and `chatLoading` true.

## Solution Assessment

No. A UI-level `Promise.race` is a plausible spinner mitigation, but the mergeable fix should clean up or cancel the underlying Gateway request while preserving current main's newer startup retry and history reconciliation behavior.
