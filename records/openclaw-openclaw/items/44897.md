## Summary

Keep open. The PR targets a real WebUI hang that current main only partly mitigates through startup-unavailable retries and bounded history payloads, but the patch still needs real behavior proof and should retire the underlying pending RPC when the timeout wins.

## What This Changes

The PR wraps `loadChatHistory`'s `chat.history` request in a 10-second timeout, clears the timer, and conditionally applies late successful history responses.

## Best Possible Solution

Add timeout or cancellation at the browser RPC layer, or a targeted `chat.history` wrapper that rejects and retires the pending request, while preserving current main's startup retry and bounded-history behavior with focused regression coverage and redacted real UI proof.

## Reproduction Assessment

Yes for a focused source-level path: a `loadChatHistory` call whose `state.client.request("chat.history")` promise never settles keeps `chatLoading` true because `finally` is not reached. I did not establish a live real-gateway reproduction in this read-only review.

## Solution Assessment

No. A caller-side `Promise.race` improves the spinner UX, but the underlying `GatewayBrowserClient` pending RPC remains live; the narrower maintainable fix should retire or cancel that pending request when the timeout wins.

## Review Findings

Overall correctness: not a patch

Overall confidence: 0

Full review comments:

- **[P2] Cancel the timed-out chat.history RPC:** `ui/src/ui/controllers/chat.ts:121`
  - body: The timeout only races the caller's promise. `GatewayBrowserClient` keeps the request in its pending map until a response frame arrives or the socket closes, so after the 10-second timeout users can retry and stack orphaned `chat.history` entries during the same stalled-gateway condition this PR targets. Add a request-level timeout/cancel path, or otherwise reject and remove the pending RPC when this timeout wins.
  - confidence: 0.82
