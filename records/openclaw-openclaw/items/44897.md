## Summary

Keep open. The bug is still plausible on current main, but this PR is not merge-ready because it lacks real browser/Gateway proof, conflicts with current main, and the timeout path leaves the underlying Gateway RPC pending.

## What This Changes

The PR updates Control UI chat history loading to race `chat.history` against a 10-second timeout and conditionally apply late successful history responses.

## Best Possible Solution

Implement a current-main chat-history timeout that cancels or removes the pending Gateway RPC, preserves the existing history/session guards, and proves the Control UI exits loading without losing live chat state.

## Reproduction Assessment

Yes at source level: a never-resolving `chat.history` request leaves `chatLoading` true on current main, and the chat view renders that state as a loading skeleton. I did not run a live browser/Gateway reproduction in this read-only review.

## Solution Assessment

No, not as proposed. The fix should be rebased onto current main and pair the UI timeout with cancellation or explicit pending cleanup instead of only racing the request promise.
