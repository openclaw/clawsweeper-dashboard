## Summary

Keep open. Current main still has the reported hang path, but this PR should not merge as-is because it times out only the awaiting UI code while the browser RPC remains pending, and its late-success branch bypasses current main's history filtering and optimistic-tail reconciliation.

## What This Changes

The PR changes Control UI chat history loading to race `chat.history` against a 10-second timeout, clean up the timeout timer, apply some late successful responses, and bump a freshness token on send.

## Best Possible Solution

Land a current-main-native timeout path that exits empty chat loading, cancels or removes timed-out browser RPCs, and reuses the shared history apply/reconciliation logic for any safe late success.

## Reproduction Assessment

Yes. Mock `state.client.request("chat.history")` with a never-settling Promise; current main keeps `loadChatHistory` pending with `chatLoading` true, and the empty chat renderer shows the loading skeleton for that state.

## Solution Assessment

No. A bounded UI timeout is the right direction, but this PR is not the best fix because it leaves `GatewayBrowserClient.pending` entries alive and applies late history through a narrower path than current main uses.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

  - body: When the timeout wins, the original `chat.history` request remains registered in `GatewayBrowserClient.pending` until a response or socket close. Repeated retries against a stalled gateway can accumulate unresolved browser RPCs, so the timeout path needs an explicit cancel/remove seam.
  - confidence: 0.86

  - confidence: 0.84
