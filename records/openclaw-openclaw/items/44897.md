## Summary

Keep open. Current main has partial responsiveness and server-side mitigations, but it does not implement this PR's bounded `chat.history` recovery; the PR is also conflicting, lacks real behavior proof, and leaves timed-out Gateway RPCs pending, so it should not be merged as-is or closed as implemented.

## What This Changes

The branch changes `ui/src/ui/controllers/chat.ts` to race `chat.history` against a 10s timeout, clear the timer, and apply late successful history responses behind freshness guards.

## Best Possible Solution

Port bounded history recovery onto current main with request cancellation or explicit pending-entry cleanup, preserving the current history guards, limits, startup retry, optimistic-tail behavior, and focused regression coverage.

## Reproduction Assessment

Yes, from source for the narrow pending-request path: a never-settling `chat.history` promise keeps `loadChatHistory` awaiting, and current tests deliberately expect an `awaitHistory` refresh to remain pending. No live browser/Gateway repro was run.

## Solution Assessment

No. A UI-level `Promise.race` mitigates the spinner, but the mergeable fix needs cancellation or cleanup of the underlying Gateway request and must be ported onto current main's newer history behavior.
