## Summary

Keep open. The underlying WebUI hang is still source-plausible, but this PR is not merge-ready because it lacks real behavior proof, is conflicting with current main, and the timeout path can leave the underlying Gateway RPC pending.

## What This Changes

The PR adds a 10-second Control UI timeout around `chat.history`, applies late successful history responses under guards, and invalidates late history on chat send.

## Best Possible Solution

Rebase onto current main and implement bounded history-timeout recovery through a cancellable or explicit-cleanup Gateway request path while preserving current history guards and adding focused proof.

## Reproduction Assessment

Yes, source-reproducible for the narrow bug: current `loadChatHistory` awaits `chat.history`, and an existing await-history refresh test expects the call to stay pending until history resolves. I did not run a live browser/Gateway reproduction in this read-only review.

## Solution Assessment

No. A UI timeout is a reasonable direction, but this patch needs request cancellation or pending cleanup and a rebase that preserves current-main history safeguards before it is the maintainable fix.
