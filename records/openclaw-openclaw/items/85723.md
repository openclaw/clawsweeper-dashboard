## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Keep continuation replies in the originating chat:** `src/plugins/host-hook-scheduled-turns.ts:384`
  - body: `/goal` asks for `deliveryMode: "announce"`, but this scheduler only derives an explicit target for Discord-shaped session keys and otherwise stores `{ channel: "last" }`; the new test even locks Telegram/WhatsApp-shaped sessions to that fallback. A goal started from a non-Discord chat can therefore continue later in whatever “last” resolves to instead of the trusted originating chat, which breaks the same-session guarantee. Carry the command/tool delivery context, or resolve all supported session keys, before scheduling the lease.
  - confidence: 0.87
- **[P2] Check lease cleanup before confirming stopped state:** `extensions/goal/src/workflow.ts:38-41`
  - body: For `paused`, `done`, `blocked`, and `waiting_approval`, this call ignores `clearSessionContinuationLease()`'s `{ removed, failed }` result and immediately writes the stopped state. If cron listing/removal fails, the old lease can still fire after the user was told the goal stopped; handle `failed > 0` before updating the state or reporting success.
  - confidence: 0.85
- **[P2] Check lease cleanup before deleting goal state:** `extensions/goal/src/command.ts:155-158`
  - body: `/goal clear` deletes the visible goal state even when `clearSessionContinuationLease()` fails to remove the queued lease. That leaves a scheduled continuation with no matching visible goal, so the user can see a follow-up run after clearing; only delete or confirm clear once cleanup succeeds, and surface the cleanup failure otherwise.
  - confidence: 0.84
