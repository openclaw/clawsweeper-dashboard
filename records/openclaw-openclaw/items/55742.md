## Summary

Keep open, but this PR is not merge-ready: it is conflicting and stale against current main, lacks real behavior proof, misses the current descriptor-backed plugin execution path, and carries an obsolete compaction auth hunk that would regress header-only auth.

## What This Changes

The branch wraps resolved plugin tools to append a narrow identity/session context to `execute`, adds resolver tests, and updates compaction safeguard auth mocks/API usage.

## Best Possible Solution

Rebase onto current main, implement execute-context injection once through the shared plugin execution contract so both cached descriptor and live registry paths behave identically, export/document the SDK type if maintainers accept the API, preserve current compaction auth behavior, and require redacted real gateway/plugin proof.

## Reproduction Assessment

Yes, by source inspection: current main's dispatcher and plugin resolver pass `signal` and `onUpdate` to tool execution, while the plugin factory receives identity at creation time. I did not run tests because this was a read-only review.

## Solution Assessment

No, not as currently proposed. The right shape needs to be rebased onto current main, cover descriptor-backed and live plugin execution, and expose the contract through the supported plugin SDK if maintainers accept it.

  - body: This stores only `auth.apiKey` after calling `getApiKeyAndHeaders`, but the resolved auth can be header-only. In that case the later compaction request will look unauthenticated even though auth resolution succeeded; preserve `auth.headers` as current main's `resolveModelAuth` path does.
  - confidence: 0.88

  - confidence: 0.86

If this is meant to be public plugin API, define and re-export it through the SDK barrels and add contract tests.
  - confidence: 0.82
