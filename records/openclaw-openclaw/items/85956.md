## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Publish the early prompt to selected-session subscribers:** `extensions/codex/src/app-server/run-attempt.ts:5725-5737`
  - body: The new turn-start mirror writes the user prompt, but `mirrorCodexAppServerTranscript` only emits a transcript update with `sessionFile`/`sessionKey`. `createTranscriptUpdateBroadcastHandler` returns when `update.message` is missing, so selected WebChat subscribers still do not get a `session.message` for this early prompt. Emit the appended prompt message, id, and sequence so WebChat can show it before turn completion.
  - confidence: 0.92
- **[P2] Guard selected-session subscription sync against stale completions:** `ui/src/ui/controllers/sessions.ts:504-514`
  - body: `switchChatSession` and reconnect paths fire `syncSelectedSessionMessageSubscription` without awaiting it, but the helper mutates `chatSessionMessageSubscriptionKey` after awaited unsubscribe/subscribe calls. A slower earlier sync can finish after a later selected session and leave the UI subscribed to the wrong stream. Add a generation/client/session guard or serialize/cancel these syncs before mutating state.
  - confidence: 0.88
- **[P2] Adopt run ids across canonical session-key aliases:** `ui/src/ui/controllers/chat.ts:673-675`
  - body: Passive adoption still requires exact `payload.sessionKey === state.sessionKey`. Existing tests already cover the gateway emitting `agent:main:main` while UI state holds `main`; with `chatRunId === null`, those deltas return before the new adoption block and the observer never adopts the run. Normalize or compare supported alias/canonical pairs before gating adoption.
  - confidence: 0.9
- **[P2] Track requested and canonical subscription keys separately:** `ui/src/ui/controllers/sessions.ts:512-513`
  - body: The helper stores the canonical key returned by `sessions.messages.subscribe`, then compares it against the raw selected key on the next sync. If the selected key is `main` and the server returns `agent:main:main`, every sync looks like a session switch and unsubscribes/resubscribes unnecessarily, creating avoidable gaps in the live stream. Keep the requested key for change detection and the canonical key for unsubscribe.
  - confidence: 0.86
