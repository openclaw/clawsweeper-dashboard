## Summary

Keep open. Current main records interrupted cron runs and has memory diagnostics, but it does not prove or fix the reported V8 heap OOM root cause, and it does not add an explicit safe replay or needs-replay recovery workflow for interrupted cron deliveries.

## What This Changes

The issue reports a Gateway V8 heap OOM that interrupted active cron jobs and requests clearer memory diagnostics plus cron recovery after restart.

## Best Possible Solution

Keep this open as gateway/cron resilience work: use existing memory diagnostics to narrow the heap-growth source, then define and implement explicit interrupted-run recovery semantics such as a needsReplay state, safe retry policy, or documented replay command with regression coverage.

## Solution Assessment
