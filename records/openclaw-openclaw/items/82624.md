## Best Possible Solution

Keep the transcript-state validation hardening, but make rejected-id tracking occurrence-safe and add duplicate-id regression coverage before merge.

Yes. Source inspection of the current PR head gives a high-confidence path: a valid row id, a later malformed row reusing that id, and then a valid child with `parentId` set to that id is treated as having a rejected parent.

## Solution Assessment

No. `TranscriptFileState` is the right layer for this hardening, but the patch should not let a malformed duplicate occurrence poison an id that was already accepted.
