## Summary

Keep open: this maintainer-labeled PR is directionally useful, but current head still has a correctness bug where a malformed duplicate row can poison the accepted entry id used by later children.

## What This Changes

The PR adds validation/filtering for Pi transcript JSONL entries before TranscriptFileState builds the active leaf, plus regression tests and a changelog line.

## Best Possible Solution

Keep the malformed-entry filtering, but only mark a malformed id as rejected when that id has not already been accepted, and add duplicate-id recovery coverage.

## Reproduction Assessment

Yes. Source inspection of PR head gives a high-confidence path: write a valid row id, then a malformed duplicate of that id, then a valid child whose parentId is that id; the child is relinked away from the accepted original row.

## Solution Assessment

No. Validating persisted transcript rows in TranscriptFileState is the right layer, but the rejected-id bookkeeping needs a narrow fix so skipped duplicate rows cannot invalidate an accepted entry identity.

## Review Findings
