## Summary

Keep open: this PR addresses a real transcript-state hardening gap and includes focused proof, but it is not merge-ready because the new validator drops valid empty-summary compaction entries that current manual compaction behavior relies on.

## What This Changes

This PR adds validation and filtering for persisted Pi transcript JSONL entries before TranscriptFileState indexes them, adds focused regression coverage, and updates the changelog.

## Best Possible Solution

Keep the malformed-entry filtering, but align entry-specific validation with Pi's persisted `SessionEntry` contract and add regression coverage for valid empty compaction summaries remaining readable as the active leaf.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: a transcript with `appendCompaction("", keepId, 200)` is valid on current main, but the PR's `isString(candidate.summary)` check would drop that compaction row before `TranscriptFileState` chooses the leaf.

## Solution Assessment

No; the hardening direction is right, but the current implementation is too strict for valid persisted Pi entries. The safer solution is to reject missing or wrong-typed payloads without treating allowed empty strings as malformed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Accept valid empty compaction summaries:** `src/agents/pi-embedded-runner/transcript-file-state.ts:81`
  - body: This guard now rejects compaction entries whose `summary` is an empty string because `isString()` trims and requires non-empty text. Pi can persist empty manual compaction summaries, and current `manual-compaction-boundary` coverage relies on reading that compaction as the leaf; dropping it makes `readTranscriptFileState()` fall back to the previous entry and can send compaction/rewrite/truncation helpers down the wrong path. Please allow valid empty summary strings and cover that case in this reader.
  - confidence: 0.91
