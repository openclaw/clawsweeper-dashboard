## Summary

Keep this issue open: current main still depends on `node:sqlite` for task and Task Flow persistence, so runtimes without that built-in can still fail restore/upsert durability even though scheduled sweep rejection handling is now partially guarded.

## What This Changes

The issue requests a runtime-safe persistence path for Gateway task registry maintenance when `node:sqlite` is unavailable in Homebrew Node.js.

## Best Possible Solution

Add a maintainer-approved storage selection layer for task and Task Flow registries that probes `node:sqlite`, keeps SQLite where available, provides a durable supported fallback where unavailable, and documents/tests the contract.

## Reproduction Assessment

Yes for the remaining storage failure: source inspection shows restore, upsert, and maintenance paths still call `requireNodeSqlite()` through the default SQLite stores when `node:sqlite` is missing. I did not run the exact Homebrew Node runtime locally, so this is source-reproducible rather than locally reproduced.

## Solution Assessment

Unclear. The suggested `better-sqlite3` replacement or JSON fallback could address the runtime gap, but maintainers should choose the backend and durability/security contract before implementation.
