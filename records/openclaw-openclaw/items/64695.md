## Summary

Keep this issue open: current main and the latest release still route task and Task Flow persistence through `node:sqlite`, so runtimes missing that builtin can still fail durable restore/upsert paths. The related JSON-fallback PR was closed unmerged, and its review discussion reinforces that the fallback choice needs maintainer design input.

## What This Changes

The issue requests a supported gateway persistence path for Homebrew Node.js and other runtimes where the experimental `node:sqlite` builtin is unavailable.

## Best Possible Solution

Add a maintainer-approved storage selection layer for task and Task Flow registries that probes `node:sqlite`, keeps SQLite where available, provides a durable supported fallback where unavailable, and documents the runtime contract.

## Reproduction Assessment

Yes for source reproduction: current main still routes restore, upsert, and maintenance persistence through `requireNodeSqlite()` when using the default stores. I did not execute the exact Homebrew Node runtime from the report, so this is source-reproducible rather than locally reproduced.

## Solution Assessment

Unclear. The requested runtime-safe persistence direction is right, but the proposed `better-sqlite3` replacement or JSON fallback needs maintainer choice on backend, packaging, locking, and durability semantics.
