## Summary

Keep this issue open: current main still depends on `node:sqlite` for task and Task Flow persistence, so Homebrew Node runtimes without that built-in module can still lose restore/upsert durability even though the scheduled sweep rejection path is now caught.

## What This Changes

Request a runtime-safe task registry and Task Flow persistence path when `node:sqlite` is missing in Homebrew Node.js.

## Best Possible Solution

Add a maintainer-approved storage selection layer for both registries: probe `node:sqlite`, keep SQLite where available, provide a durable supported fallback where unavailable, and update docs/tests for restore, upsert, maintenance, and Task Flow persistence.

## Reproduction Assessment

Yes for the remaining storage failure. On a runtime where `require('node:sqlite')` throws, current main still reaches `requireNodeSqlite()` from task restore/upsert and Task Flow restore/upsert paths; the exact scheduled-sweep process crash appears partially mitigated by the current rejection handling.

## Solution Assessment

Unclear. The issue's suggested `better-sqlite3` replacement or JSON fallback could address the runtime gap, but maintainers should choose the backend and durability/security contract before implementation.
