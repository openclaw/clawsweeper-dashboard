## Label Justifications

- `P1`: The issue affects task durability and reported gateway availability on a real supported Node runtime path, but it is runtime-specific rather than a global outage.
- `impact:data-loss`: When `node:sqlite` is unavailable, task and Task Flow state cannot reliably restore or persist durable records.
- `impact:crash-loop`: The report is about repeated gateway crashes from task maintenance, and current source still contains the throwing storage boundary even though scheduled rejection handling improved.

## Summary

Keep this issue open: current `main` and the latest release still make task and Task Flow persistence depend on `node:sqlite` with no runtime availability probe or fallback. The scheduled sweep rejection path is partly hardened now, but unsupported Node builds can still fail restore/write durability paths instead of preserving task state.

## What This Changes

The issue reports gateway/task persistence crashes on Homebrew Node builds where `node:sqlite` is unavailable and asks for a runtime-safe task storage backend or fallback.

## Best Possible Solution

Add a maintainer-approved runtime-safe storage strategy for both task registries: probe `node:sqlite`, keep SQLite where available, choose a durable fallback or supported dependency where unavailable, and document diagnostics and upgrade behavior.

## Reproduction Assessment

Yes, source-level: in any runtime where `require('node:sqlite')` fails, the default task and Task Flow stores still call `requireNodeSqlite()` on restore and write paths. I did not run the exact Homebrew Node bottle from the report.

## Solution Assessment

Unclear. A fallback or replacement backend is the right class of fix, but the closed JSON fallback PR shows the final approach needs maintainer decisions on locking, filesystem safety, durability, and packaging.
