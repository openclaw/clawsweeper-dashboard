## Summary

Keep this issue open: current main still uses `node:sqlite` directly for task and Task Flow persistence, so a runtime without that built-in can still fail restore/upsert durability even though scheduled sweep rejection handling is guarded.

## What This Changes

Fix the gateway task persistence crash on Node runtimes where `require("node:sqlite")` is unavailable.

## Best Possible Solution

Add a maintainer-approved runtime-safe storage layer for both registries that probes `node:sqlite`, preserves SQLite where supported, and provides a durable documented fallback or supported dependency where it is not.

## Reproduction Assessment

Yes for a source repro: use a Node runtime where `require("node:sqlite")` throws, then hit task restore/upsert or the maintenance sweep path; current main still routes those operations through `requireNodeSqlite()`. The local Node v24.14.1 runtime has `node:sqlite`, so I did not reproduce the Homebrew-specific failure locally.

## Solution Assessment

Unclear. The suggested `better-sqlite3` replacement or JSON fallback could address the missing built-in, but maintainers should first choose the backend, packaging, durability, and security contract, and the final fix must cover both task registry and Task Flow registry persistence.
