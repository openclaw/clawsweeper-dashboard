## Summary

Keep open. Current main still makes the task registry and Task Flow registry depend on `node:sqlite` with no runtime availability probe or fallback store, so the reported Homebrew runtime remains unable to restore or persist durable task state even though the scheduled sweep now consumes its rejected promise.

## What This Changes

The issue requests a fix for gateway task persistence on Homebrew Node.js runtimes where the `node:sqlite` built-in module is unavailable.

## Best Possible Solution

Add a maintainer-approved runtime-safe storage selection layer for both registries: probe `node:sqlite` before default use, keep SQLite where available, provide a durable fallback or supported dependency where unavailable, and update tests and docs for restore, upsert/update, stale-task maintenance, and Task Flow persistence.

## Reproduction Assessment

Yes. On a Node runtime where `require("node:sqlite")` throws, current main still routes task restore/upsert and maintenance lost-marking through `requireNodeSqlite()`; I verified the source path, though I did not have the exact Homebrew Node v23.3.0 bottle in this runner.

## Solution Assessment

Unclear. The issue's suggested `better-sqlite3` dependency or JSON fallback could address availability, but the best fix needs maintainer selection of storage backend and durability/security semantics before implementation.
