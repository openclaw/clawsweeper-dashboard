## Label Justifications

- `P1`: The issue affects gateway task durability and reported process availability on a real supported-runtime path, but it is runtime-specific rather than a global outage.
- `impact:data-loss`: When `node:sqlite` is unavailable, task and Task Flow state cannot reliably restore or persist durable records.
- `impact:crash-loop`: The report is about repeated gateway crashes from the task maintenance path, and current source still contains the throwing storage boundary even though scheduled rejection handling improved.

## Summary

Keep this issue open: current `main` and the latest release still require `node:sqlite` for task and Task Flow persistence with no availability probe or fallback, while the linked fallback PR closed unmerged after build, durability, and security concerns. The crash symptom is partly mitigated for scheduled sweeps, but the unsupported runtime still cannot reliably restore or persist task state.

## What This Changes

The issue reports gateway crashes and task persistence failures when Homebrew Node.js lacks the `node:sqlite` builtin used by task registry maintenance.

## Best Possible Solution

Add a maintainer-approved runtime-safe storage strategy for both task registries: probe `node:sqlite`, keep SQLite where available, choose a durable fallback or supported dependency where unavailable, and document the resulting behavior.

## Reproduction Assessment

Yes, source-level: in any runtime where `require('node:sqlite')` fails, current default task and Task Flow stores still call `requireNodeSqlite()` on restore/upsert paths. I did not run the exact Homebrew Node bottle from the report.

## Solution Assessment

Unclear. A fallback or replacement backend is the right class of fix, but the closed JSON fallback PR shows the final approach needs maintainer decisions on locking, filesystem safety, durability, and packaging before implementation.
