## Summary

Keep this issue open: current main still uses `node:sqlite` as the only default persistence backend for the task and Task Flow registries, so the reported runtime class can still lose restore/upsert durability even though the scheduled sweep promise path is now guarded.

## What This Changes

The issue reports gateway task persistence crashes on Node runtimes where `node:sqlite` is unavailable.

## Best Possible Solution

Add a maintainer-approved runtime-safe storage strategy for both registries: probe `node:sqlite`, keep SQLite where supported, provide a durable hardened fallback or dependency where unavailable, and document the resulting runtime contract.

## Reproduction Assessment

Yes for the remaining storage failure: source inspection shows restore, upsert, and maintenance paths still call `requireNodeSqlite()` through default SQLite stores when `node:sqlite` is missing. I did not locally run the exact Homebrew Node runtime from the report.

## Solution Assessment

Unclear. The suggested `better-sqlite3` replacement or JSON fallback could fix the runtime gap, but maintainers should choose the supported backend and durability/security contract before implementation.
