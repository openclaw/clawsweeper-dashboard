## Summary

Keep this issue open: current main still routes task registry and Task Flow registry persistence through `node:sqlite`, so a runtime without that built-in still loses restore/upsert durability. The scheduled sweep path now avoids an unhandled rejection, but the underlying supported-runtime storage failure remains.

## What This Changes

The issue asks OpenClaw to stop crashing or losing task durability on runtimes where task persistence cannot load `node:sqlite`.

## Best Possible Solution

Add a maintainer-approved runtime-safe storage selection layer for task and Task Flow registries that keeps SQLite where available, provides a durable hardened fallback or supported dependency where unavailable, and updates tests/docs for the contract.

## Reproduction Assessment

Yes for the remaining storage failure: current main still enters `requireNodeSqlite()` through default task and Task Flow persistence, and `node --no-experimental-sqlite` reproduces the same missing built-in error family. I did not run the exact Homebrew Node v23.3.0 bottle from the report.

## Solution Assessment

Unclear. The suggested `better-sqlite3` replacement or JSON fallback could solve the runtime gap, but maintainers should choose the backend and durability/security contract before implementation.
