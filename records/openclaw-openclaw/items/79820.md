## Summary

Keep open: current main still exposes `chat.inject` as an assistant-only, closed-schema transcript append, while the requested user-role replay and stable caller-ID semantics are a new Gateway protocol capability that needs maintainer design.

## What This Changes

Request adding user-role synthetic transcript replay and stable caller/durable ID semantics to `chat.inject` or a companion Gateway API.

## Best Possible Solution

Design an additive Gateway replay API or carefully extended `chat.inject` contract that appends synthetic user and assistant turns through the transcript manager, returns durable IDs, and documents parent-chain semantics.

## Reproduction Assessment

Yes for the current limitation: source inspection shows the schema is closed to `sessionKey`, `message`, and `label`, and the implementation always writes an assistant transcript message. This is not a broken documented behavior; it is a missing replay capability.

## Solution Assessment

Unclear as an implementation until maintainers choose the API shape; the safest direction is an additive protocol design or companion replay method rather than ad hoc extra fields on the current assistant-note shortcut.
