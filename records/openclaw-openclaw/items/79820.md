## Summary

Keep open: current `main` still exposes `chat.inject` as a closed-schema, assistant-only transcript append, and the requested user-role replay plus durable caller-ID semantics are a new Gateway protocol capability that needs maintainer design.

## What This Changes

The issue requests extending `chat.inject` or adding a companion replay API for synthetic user turns, stable caller IDs, and documented transcript parent-chain behavior.

## Best Possible Solution

Design an additive Gateway replay API or carefully versioned `chat.inject` extension that appends synthetic user and assistant turns through the transcript manager, returns durable cross-reference IDs, and documents parent-chain semantics with protocol and Gateway tests.

## Reproduction Assessment

Yes for the reported current behavior: source inspection shows a closed `chat.inject` schema and an assistant-only append path that returns only the Gateway-generated message ID. This is not a broken documented behavior; it is a missing replay capability.

## Solution Assessment

Not applicable as a patch yet: the requested behavior changes the public Gateway protocol and transcript semantics. The safer direction is an additive, documented contract rather than ad hoc extra fields on the current closed schema.
