## Summary

Keep open: current main still exposes `chat.inject` as an assistant-only, closed-schema transcript append, while the issue asks for new user replay and durable caller/idempotency semantics that are not implemented and need Gateway protocol design.

## What This Changes

The issue requests Gateway transcript injection support for synthetic user-turn replay plus stable client/caller IDs or durable returned IDs.

## Best Possible Solution

Design an additive Gateway replay API or carefully extended `chat.inject` contract that appends synthetic user and assistant turns through the transcript manager, returns durable IDs, and documents parent-chain semantics.

## Reproduction Assessment

Yes for the reported current behavior: source inspection shows the schema is closed to `sessionKey`, `message`, and `label`, and the implementation always writes an assistant transcript message. This is a missing replay capability rather than broken documented behavior.

## Solution Assessment

Not applicable as a patch yet: the issue requests a new public Gateway contract. The safer direction is an additive protocol design with explicit role, ID, idempotency, auth, and parent-chain semantics rather than ad hoc extra fields on the current closed schema.
