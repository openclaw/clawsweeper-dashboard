## Summary

Keep open: current main still exposes `chat.inject` as a closed, assistant-only transcript injection path, and the matching user-role/idempotency implementation remains in an open unmerged PR.

## What This Changes

The issue requests an additive `chat.inject` contract for synthetic transcript replay with user-role turns, durable caller/idempotency IDs, and documented parent-chain behavior.

## Best Possible Solution

Review and land an additive Gateway protocol change like https://github.com/openclaw/openclaw/pull/81513, with generated clients, public docs, parent-chain tests, and coordination with the related attribution work if both advance.

## Reproduction Assessment

Yes for the current limitation: source inspection shows the closed `chat.inject` schema and assistant-only handler path on current main. A live Gateway probe is not required to confirm that `role` and caller id fields are outside the accepted wire contract.

## Solution Assessment

Yes: an additive Gateway protocol update is the right maintainable direction, and the open implementation PR is the place to finish review, docs, and validation. Closing the issue before that PR lands would lose the tracking item for a still-missing current-main capability.
