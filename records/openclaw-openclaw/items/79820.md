## Summary

Keep open: current main still exposes `chat.inject` as a closed, assistant-only Gateway transcript append, while the requested user-role replay and idempotency contract is not merged. The linked implementation PR is open and unmerged, so it is the right place to continue review rather than closing this issue now.

## What This Changes

Request an additive Gateway `chat.inject` contract for synthetic user turns, stable caller-supplied replay IDs, and documented transcript parent-chain behavior.

## Best Possible Solution

Land a reviewed additive Gateway protocol update, likely through the linked PR or a replacement, that supports user-role transcript replay, durable idempotent IDs, and docs for parent-chain and broadcast semantics.

## Reproduction Assessment

Yes for the reported current behavior: source inspection shows current main rejects extra `chat.inject` fields and always writes assistant transcript messages. This is a missing protocol capability rather than a broken documented behavior.

## Solution Assessment
