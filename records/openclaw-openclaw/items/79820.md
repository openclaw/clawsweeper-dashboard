## Summary

Keep open: current `main` and latest release still expose `chat.inject` as an assistant-only, closed-schema transcript append, while the requested user-role replay and stable caller ID semantics are new Gateway protocol behavior that needs maintainer design.

## What This Changes

Request asks `chat.inject` or a companion method to replay synthetic user turns with durable caller IDs and documented parent-chain semantics.

## Best Possible Solution

Design an additive Gateway replay API or carefully extended `chat.inject` contract that appends synthetic user and assistant turns through the transcript manager, returns durable IDs, and documents parent-chain semantics.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path for the reported current behavior: a `chat.inject` request with `role` or caller ID fields is rejected by the closed schema, and accepted calls always append an assistant message. I did not run a live WebSocket probe because the read-only review already proves the contract path.

## Solution Assessment

Unclear as an implementation because no patch is proposed yet. The best direction is an additive Gateway protocol design, or a tightly scoped extension of `chat.inject`, rather than ad hoc acceptance of extra fields in the existing assistant-note method.
