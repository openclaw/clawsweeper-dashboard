## Summary

Keep open. Current main and v2026.5.12 still expose an assistant-only `chat.inject` contract, while https://github.com/openclaw/openclaw/pull/81513 is the active implementation path for user-role and idempotent replay support.

## What This Changes

The issue requests expanding `chat.inject` so synthetic turns can append user-role transcript messages, carry stable idempotency/caller IDs, and document parent-chain behavior.

## Best Possible Solution

Land one coordinated additive Gateway contract that covers user-role replay, idempotent/canonical message IDs, accepted metadata or attribution fields, parent-chain documentation, and focused regression coverage.

## Reproduction Assessment

Yes. A source-level reproduction is to call `chat.inject` with `role` or `idempotencyKey`; current main and v2026.5.12 reject those fields via the closed `ChatInjectParamsSchema`.

## Solution Assessment

Yes in direction, not complete in repository state. An additive Gateway protocol update with docs and transcript tests is the maintainable fix; the linked implementation PR targets that seam but still needs review/landing before this issue is resolved.
