## Summary

Keep open: current main still exposes `chat.inject` as an assistant-only, closed-schema transcript append, while the issue asks for new replay and idempotency semantics that are not implemented and need Gateway protocol design.

## What This Changes

The issue requests extending `chat.inject` or adding a companion replay method so synthetic turns can include user-role messages, durable caller IDs, and documented parent-chain behavior.

## Best Possible Solution

Design an additive Gateway replay API or carefully extended `chat.inject` contract that appends user and assistant synthetic turns through the transcript manager, returns durable IDs, and documents parent-chain semantics.

## Reproduction Assessment

Yes for the reported current behavior: source inspection shows the schema is closed to assistant-injection fields and the implementation always writes an assistant transcript message. This is not a broken documented behavior; it is a missing replay capability.

## Solution Assessment

Unclear until protocol design: a companion replay method or a carefully extended `chat.inject` contract could work, but loose extra fields would be risky without transcript trust and parent-chain rules.
