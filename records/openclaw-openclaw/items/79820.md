## Summary

Keep open: current main still lacks user-role replay and stable caller/idempotency fields for `chat.inject`, and the directly related implementation is still an open PR rather than merged or shipped. The request is concrete, but it changes the public Gateway protocol and transcript trust model, so it should be handled through maintainer review of the linked PR or an equivalent additive API design.

## What This Changes

The issue requests extending `chat.inject` or adding a companion replay API so synthetic turns can include user-role messages, durable caller-supplied IDs, and documented parent-chain behavior.

## Best Possible Solution

Review and land, or revise, the open Gateway implementation so `chat.inject` gains an additive replay contract for user-role turns and idempotent caller references, with tests and docs for transcript parent-chain behavior.

## Reproduction Assessment

Yes for the current limitation: source inspection on current main shows the public schema rejects the requested fields and the handler still writes assistant-only transcript entries. This is a missing replay capability, not a broken documented bug.

## Solution Assessment

Unclear until maintainer review of the open PR finishes. An additive Gateway contract is the right direction, but the exact `chat.inject` shape, dedupe behavior, authorization/trust boundary, and documentation need acceptance before this is the best maintainable solution.
