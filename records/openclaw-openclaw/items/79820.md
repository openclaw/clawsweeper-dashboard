## Summary

Keep open: current main still exposes `chat.inject` as an assistant-only closed-schema transcript append, while the request asks for new user replay, idempotency, and parent-chain semantics that require Gateway protocol design.

## What This Changes

The issue requests extending `chat.inject` or adding a companion replay method for synthetic user turns, stable caller IDs, returned durable IDs, and documented transcript parent-chain behavior.

## Best Possible Solution

Design an additive Gateway replay API or carefully extended `chat.inject` contract that appends user and assistant synthetic turns through the transcript manager, returns durable IDs, and documents parent-chain semantics.

## Reproduction Assessment

Yes for the reported current behavior: source inspection shows the schema is closed to `sessionKey`, `message`, and `label`, and the implementation always writes an assistant transcript message. This is not broken documented behavior; it is a missing replay capability.

## Solution Assessment

Unclear as an implementation plan until maintainers choose the public API shape. The narrowest maintainable direction is additive protocol semantics with explicit role, caller-ID, idempotency, and transcript-chain constraints.
