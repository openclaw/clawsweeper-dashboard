## Summary

Keep open. Current main still exposes `chat.inject` as an assistant-only transcript append with a closed parameter schema, while the requested user-role replay, stable caller/idempotency IDs, metadata/source contract, and parent-chain documentation are additive Gateway protocol work rather than an already-fixed bug.

## What This Changes

Request an additive `chat.inject` or companion synthetic-turn replay contract for user-role turns, stable caller IDs, metadata/source fields, and documented transcript parent-chain behavior.

## Best Possible Solution

Define an additive replay contract that preserves existing assistant-only `chat.inject` behavior while adding an approved user-turn path, stable idempotency/client references, bounded provenance metadata, returned stored IDs, and parent-chain docs/tests.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives a high-confidence current-main capability gap: the closed `chat.inject` schema and handler accept only the existing assistant-note contract.

## Solution Assessment

Unclear until maintainers approve the API and trust model. The earlier role-only PR shape is too narrow; the maintainable solution should cover replay role, stable cross-reference IDs, provenance bounds, returned IDs, and documented parent-chain semantics together.
