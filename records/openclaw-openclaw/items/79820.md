## Summary

Keep open: current main still exposes `chat.inject` as assistant-only and does not accept user-role or caller idempotency fields. The matching implementation is in an open, unmerged PR, so the issue should remain the tracking item until that work lands or is rejected.

## What This Changes

Request to extend `chat.inject` or add a companion replay API for synthetic user turns, stable caller IDs, and documented transcript parent-chain behavior.

## Best Possible Solution

Review and land an additive Gateway protocol change like https://github.com/openclaw/openclaw/pull/81513, with generated clients, docs, parent-chain tests, and clear idempotent replay semantics.

Yes for the current limitation: source inspection shows the closed `chat.inject` schema and assistant-only append path on current main. I did not need a live Gateway probe because the accepted wire contract and handler path are explicit in source.

## Solution Assessment
