## Summary

Keep open: current main still exposes `chat.inject` as an assistant-only, closed-schema append, while the request asks for new replay and caller-ID semantics that need Gateway protocol design.

## What This Changes

Request extending `chat.inject` or adding a companion API for synthetic user-turn replay, stable caller IDs, and documented transcript parent-chain behavior.

## Best Possible Solution

Design an additive Gateway replay contract that writes synthetic user and assistant turns through the transcript manager, returns durable IDs, and documents parent-chain semantics.

## Reproduction Assessment

Yes for the reported current behavior: source inspection shows the schema is closed to `sessionKey`, `message`, and `label`, and the implementation always writes an assistant transcript message. This is not a broken documented behavior; it is a missing replay capability.

## Solution Assessment

Not applicable as a patch yet: the request changes a public Gateway contract and transcript trust semantics. The maintainable direction is an additive API design or carefully versioned extension, not an ad hoc field pass-through.
