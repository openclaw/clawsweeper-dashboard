## Summary

Keep open. Current `main` and the latest release still lack WebChat/Control UI inline-button rendering, a WebChat callback RPC, and a real `gateway.webchat` capability/config path; the duplicate WebChat issue was already closed in favor of this one, while the Telegram webhook variant was closed as plugin-scoped.

## What This Changes

The issue requests WebChat/Control UI support for inline message buttons, including capability config, WebSocket payload fields, callback routing, and Control UI rendering.

## Best Possible Solution

Implement an additive WebChat interactive-control path based on the shared `presentation` contract: preserve sanitized final-only button blocks through chat events/history, render them in Control UI, and route clicks through a scoped `chat.callback` or equivalent session-local path before considering webhook routing.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source and release inspection give high confidence that the requested WebChat button rendering, callback RPC, and config path are absent on current `main` and `v2026.5.12`.

## Solution Assessment

No patch is proposed, so there is no implementation to approve as-is. The narrow maintainable direction is a protocol/config/UI design around shared `presentation`, with webhook routing scoped separately until its security model is explicit.
