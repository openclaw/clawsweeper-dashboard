## Summary

Keep open. Current main still lacks the requested WebChat/Control UI inline-button rendering, callback RPC, and capability/config path; the closed related items do not supersede it, and the remaining work needs maintainer design across public Gateway protocol, config, UI, and callback security semantics.

## What This Changes

Add WebChat/Control UI inline button rendering, a callback RPC, and capability/config support for interactive buttons.

## Best Possible Solution

Implement an additive WebChat interactive-control path around the shared `presentation` contract, preserve sanitized final-only controls through chat events/history, render them in Control UI, and add a scoped in-session `chat.callback`; treat webhook egress as a separate hardened follow-up unless maintainers explicitly include it.

## Reproduction Assessment

Yes, at source level. Current main shows the missing schema, method registration, method scope, config path, final-event projection, and Control UI render/click paths without needing a live Gateway run.

## Solution Assessment

No for the literal proposed shape. The user outcome is valid, but the safer maintainable path is to implement WebChat support for the shared `presentation` contract plus a scoped callback RPC, with webhook egress designed separately.

## Security Review

Status: needs_attention

Summary: The issue is not a patch, but the requested callback and webhook path crosses browser-to-Gateway and outbound-network trust boundaries.

Concerns:

- **[medium] Define callback RPC authorization and replay rules:** `src/gateway/method-scopes.ts:152`
  - body: A `chat.callback` method would let an authenticated browser turn UI clicks into session input or actions. It needs method scoping, session/message ownership checks, callback value validation, replay or disabled-state behavior, and audit semantics before shipping.
  - confidence: 0.88
- **[medium] Harden webhook egress separately:**
  - body: The proposed `webchat.callbackWebhookUrl` would let browser clicks trigger Gateway-originated HTTP requests. It needs explicit URL policy, SSRF protection, authentication/signing, timeout/retry behavior, and observability rather than being bundled as incidental UI-button plumbing.
  - confidence: 0.82
