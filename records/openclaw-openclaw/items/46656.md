## Summary

Keep open. Current main still lacks Webchat/Control UI inline-button support: the chat protocol has no button/presentation event or callback RPC, Webchat/Control UI config has no inline-button capability path, and the Control UI message model/renderer has no button action branch. Closed related issues #54720 and #92533 correctly consolidate into this canonical tracker rather than superseding it.

## What This Changes

The issue requests Webchat/Control UI inline-button support, including capability/config shape, final chat payload/history support, authenticated callback handling, and Control UI rendering.

## Best Possible Solution

Add an additive Webchat interactive-control path based on shared MessagePresentation: final-only sanitized button blocks through chat events/history, Control UI rendering with clicked/disabled state, and an authenticated scoped chat.callback RPC back into the bound session.

## Reproduction Assessment

Yes at source level: portable presentation exists and sibling channels render it, but the current Webchat protocol, config, server projection, and Control UI paths lack controls and callbacks.

## Solution Assessment

The feature direction is valid, but the best fix should use existing portable MessagePresentation typed actions and an authenticated gateway RPC, not legacy shared buttons fields or a parallel webhook-first HTTP surface.
