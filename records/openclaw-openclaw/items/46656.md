## Summary

Keep open. Current main and the latest release still lack WebChat/Control UI inline-button delivery, rendering, and callback RPC support, while the related duplicate proposals were closed into this tracker rather than replacing it.

## What This Changes

The issue requests WebChat/Control UI inline-button support using Gateway chat payloads, a callback RPC, and a capability/config contract.

## Best Possible Solution

Design a v1 around the existing `MessagePresentation` typed-action contract, authenticated Gateway RPC, sanitized final/history projection, and Control UI rendering, while deferring webhook routing and avoiding a retired WebChat config surface unless maintainers explicitly approve it.

## Reproduction Assessment

Yes, as a source-level capability gap: current schemas, method catalogs, server chat events, docs, and UI render paths lack WebChat inline-button and callback support. I did not run a live browser repro because this was a read-only source review.

## Solution Assessment

No, not exactly as originally written. The best path is the later refined shape: typed `MessagePresentation` actions over an authenticated Gateway RPC, not legacy opaque `buttons` fields or a parallel webhook endpoint for v1.
