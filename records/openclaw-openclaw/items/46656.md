## Summary

Keep open. Current main still lacks the requested WebChat/Control UI inline-button path: no chat protocol button payload, no `chat.callback` method, no Control UI capability/config field, and no UI content model or renderer for message buttons. The closed duplicate at https://github.com/openclaw/openclaw/issues/54720 correctly points back here as the canonical tracker, while https://github.com/openclaw/openclaw/issues/46841 was Telegram-specific webhook routing rather than WebChat support.

## What This Changes

Requested WebChat/Control UI support for inline buttons, including a capability/config path, WebSocket button delivery, callback RPC, and Control UI rendering.

## Best Possible Solution

Implement an additive WebChat interactive-control path based on the shared `presentation` contract, preserve sanitized final-only controls through chat events/history, render them in Control UI, and add a scoped callback path that initially routes selected values back into the same session; treat webhook routing as separate hardening work.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection gives high confidence that current main still has no WebChat button/callback/config/rendering path.

## Solution Assessment

No concrete patch is proposed, so there is no implementation to approve as-is. The maintainable direction is contract-first WebChat presentation/callback work, not adding Telegram-native `buttons` fields directly to the shared protocol.
