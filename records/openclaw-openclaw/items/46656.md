## Summary

Keep open. Current main still lacks the requested WebChat/Control UI inline-button path across the Gateway chat schema, callback method registration, WebChat config, final chat projection, and Control UI rendering; the broader shipped presentation work does not provide a WebChat click/callback route.

## What This Changes

Add WebChat/Control UI inline button support, including a config capability, button payload delivery, callback RPC, and built-in Control UI rendering.

## Best Possible Solution

Implement an additive WebChat interactive-presentation path that preserves sanitized final-only button blocks through live events and history, renders them in Control UI, and adds a scoped callback route back into the same session while deferring webhook routing to a separate security design.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source, docs, and related-thread inspection give high confidence that current main still lacks the requested WebChat button/callback capability.

## Solution Assessment

No concrete patch is proposed, so there is no implementation to approve as-is. The maintainable direction is to build on the shared `presentation` contract and add an explicit WebChat callback contract, rather than bolting Telegram-shaped `buttons` straight into the WebSocket payload without product/security scoping.

## Security Review

Status: needs_attention

Summary: The issue would add a new browser-triggered callback and optional webhook delivery path, so authorization and outbound-delivery constraints need explicit design before implementation.

Concerns:

- **[medium] Scope WebChat callback authorization:** `src/gateway/method-scopes.ts:164`
  - body: A `chat.callback` method would let browser clients inject action values into a session, so it must preserve the same session/device/operator authorization boundaries as `chat.send` and avoid cross-session callback reuse.
  - confidence: 0.86
- **[medium] Harden optional callback webhooks:**
  - body: The proposed webhook route would POST user-triggered callback data to configured URLs, which needs SSRF-safe URL policy, timeout/retry limits, idempotency, and clear failure semantics before it belongs in core.
  - confidence: 0.82
