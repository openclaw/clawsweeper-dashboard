## Summary

Current main still lacks end-to-end Webchat rendering and activation of portable message actions, so the central request remains useful. The original Telegram-shaped capability config, raw callback payload, and generic webhook proposal should not be implemented as written because OpenClaw now has typed presentation actions, has retired channel-style Webchat config, and requires an explicit authorization boundary for browser-triggered actions.

## What This Changes

Requests first-class inline action buttons in Webchat and Control UI, including rendering, callback transport, capability policy, and optional webhook routing.

## Best Possible Solution

Extend the existing `MessagePresentation` contract through persisted/final Gateway chat messages and Control UI rendering, then add a narrowly typed, session-bound activation request for approved command, callback, and URL action classes; omit a Webchat capability knob by default and track generic webhook routing separately behind a security-reviewed plugin or hook boundary.

## Reproduction Assessment

Yes at the source level: current main defines portable presentation buttons, while the Gateway chat contract and Control UI event model have no end-to-end rendering or activation path. This is a clearly verifiable feature gap rather than a failing established Webchat contract.

## Solution Assessment

No, the full proposed design is not the best solution. Reusing typed portable presentation actions is narrower and consistent with current architecture; a raw callback RPC, new capability config, and generic webhook bypass should be omitted or decided separately.
