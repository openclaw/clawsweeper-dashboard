## Summary

Keep open. Current main still lacks WebChat/Control UI inline buttons: the chat protocol has no button/presentation event or callback RPC, `gateway.controlUi` has no inline-button capability schema, and the Control UI normalizes/renders only text/media/canvas/tool content. Related discussion correctly consolidates this as the canonical tracker, but the implementation still needs product and security decisions around typed presentation actions, scoped `chat.callback`, replay/idempotency, clicked state, and webhook scope.

## What This Changes

Request full WebChat/Control UI inline button support, including capability/config shape, final chat event/history payloads, authenticated callback RPC, and Control UI rendering.

## Best Possible Solution

Add an additive WebChat interactive-control path based on shared `MessagePresentation`: final-only sanitized button blocks through chat events/history, Control UI rendering with clicked/disabled state, and an authenticated scoped `chat.callback` RPC that routes typed actions back into the bound session; leave webhook routing to separate hardened plugin/design work.

## Reproduction Assessment

Yes at source level: portable presentation exists and sibling channels render it, but the current WebChat protocol, config, server, and Control UI paths lack controls and callbacks.

## Solution Assessment

The feature direction is valid, but the best fix is not legacy `buttons` fields or a parallel HTTP webhook endpoint; it should use typed `presentation` actions and an authenticated gateway RPC.
