## Summary

Keep open. Current main still lacks the requested WebChat/Control UI inline-button path, and the related closed issues consolidate into this tracker rather than superseding it.

## What This Changes

Request WebChat/Control UI support for portable inline buttons, including a capability/config path, chat event/history payloads, a callback RPC, and Control UI rendering.

## Best Possible Solution

Implement an additive WebChat interactive-control path over shared MessagePresentation typed actions, preserving sanitized final-only controls through chat events/history and adding an authenticated scoped chat.callback RPC; leave webhook routing for a separate hardened design.

## Reproduction Assessment

Yes, at source level. Portable presentation and Telegram rendering exist, but the current WebChat protocol, final projection, config schema, and Control UI renderer do not carry or display button controls and no callback RPC is registered.

## Solution Assessment

Partly. WebChat button support is the right product direction, but the best implementation should use shared MessagePresentation typed actions and authenticated Gateway RPC instead of a legacy buttons field or webhook-first HTTP surface.
