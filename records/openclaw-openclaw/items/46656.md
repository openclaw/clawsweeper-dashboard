## Summary

Keep open. Current main still lacks the requested WebChat/Control UI inline-button and callback path, and the remaining work crosses public Gateway protocol, strict config schema, Control UI rendering/history, and callback/webhook security design. The closed related WebChat duplicate does not supersede this canonical tracker.

## What This Changes

The issue requests WebChat/Control UI inline-button support, including a config capability, chat event button payloads, callback RPC routing, and Control UI rendering.

## Best Possible Solution

Implement an additive WebChat interactive-control path using the shared `presentation` contract: preserve sanitized final-only button/select blocks through chat events and history, render them in Control UI with clicked/disabled state, and add a scoped in-session `chat.callback` RPC. Treat webhook routing as a separate hardened follow-up unless maintainers explicitly include it in the first design.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on current main: the config schema rejects the requested capability path, the Gateway chat protocol/method registry has no callback or interactive fields, and the Control UI normalizer/renderer has no button item or click path.

## Solution Assessment

No for the literal proposed shape. The user problem is valid, but a Telegram-style `buttons` field plus immediate webhook routing is not the safest first contract; shared `presentation` plus scoped in-session callback routing is the narrower maintainable path.

## Security Review

Status: needs_attention

Summary: The issue itself requests callback and optional webhook behavior that affects Gateway trust boundaries and egress, so security design is required before implementation.

Concerns:

- **[medium] Define callback RPC trust boundaries:** `src/gateway/method-scopes.ts:147`
  - body: A WebChat callback RPC would let authenticated browser clicks become session input. It needs method scoping, session/message ownership checks, callback value validation, replay or disabled-state behavior, and audit semantics before shipping.
  - confidence: 0.88
- **[medium] Harden webhook egress separately:**
  - body: The proposed callback webhook would let UI clicks trigger Gateway-originated HTTP requests. That needs explicit SSRF and egress policy, authentication, retry behavior, and observability rather than being bundled into the first UI-button change by default.
  - confidence: 0.82
