## Summary

Keep open. Current main still lacks the requested WebChat/Control UI inline-button path: no button or presentation payload in the chat WS contract, no `chat.callback` method, no WebChat inline-button config, and no Control UI button content/rendering path. The closed duplicate at https://github.com/openclaw/openclaw/issues/54720 points back here as the canonical tracker, so this remains useful.

## What This Changes

The issue requests WebChat/Control UI inline button support, including capability configuration, WebSocket button delivery, callback handling, and Control UI rendering.

## Best Possible Solution

Implement an additive WebChat interactive-control path based on the shared `presentation` contract: preserve sanitized final-only button blocks through chat events and history, render them in Control UI with clicked/disabled state, and add a scoped callback path that initially routes selected values back into the same session; handle webhook routing separately.

## Solution Assessment

No concrete patch is proposed, so there is no implementation to approve as-is. The maintainable direction is a scoped, additive Gateway protocol/config/UI design using the shared presentation contract, with webhook routing separated for security review.
