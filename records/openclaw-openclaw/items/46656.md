## Summary

Keep open: current main and v2026.6.10 still lack WebChat/Control UI inline-button delivery, a `chat.callback` RPC, a WebChat capability/config contract, and Control UI rendering/click handling. ## What This Changes

Request full WebChat/Control UI inline-button support across Gateway chat payloads, callback RPC handling, capability/config shape, Control UI rendering, tests, and docs.

## Best Possible Solution

Keep this as the canonical tracker, choose a typed shared-presentation plus authenticated Gateway `chat.callback` contract, defer webhook routing, then implement final-only projection, Control UI rendering/click state, tests, and docs together.

## Reproduction Assessment

Yes at source level. Current main has no WebChat button/presentation payload in the chat schemas, no `chat.callback` method registration, no WebChat inline-button config path, and no Control UI interactive message item/render branch.

## Solution Assessment

No, not as originally proposed with legacy `buttons` and optional webhook routing. The maintainable path is shared typed `presentation` actions plus an authenticated Gateway RPC, with webhook routing deferred or separately hardened.
