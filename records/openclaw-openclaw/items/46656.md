## Summary

Keep open. Current main still lacks the requested WebChat/Control UI inline-button path: no chat event presentation/buttons payload, no `chat.callback` RPC or scope, no WebChat inline-button config, and no Control UI message item/render path for interactive controls; the remaining work is valid but crosses public protocol, config, UI, callback authorization, and optional webhook semantics.

## What This Changes

Add WebChat/Control UI inline-button support through config, Gateway WebSocket chat payloads and callback RPC, callback routing, and Control UI rendering.

## Best Possible Solution

Implement an additive WebChat interactive-control path using the shared `presentation` contract: preserve sanitized final-only button blocks through chat events/history, render them in Control UI with clicked/disabled state, and add a scoped callback RPC that initially routes selected values back into the same session; keep webhook routing as a separately hardened design.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection gives a high-confidence current-behavior check that the WebChat protocol, config, callback RPC, and Control UI renderer still lack the requested interactive-control path.

## Solution Assessment

Unclear as an implementation because no patch is proposed. The requested direction is useful, but the narrow maintainable solution should follow the shared `presentation` contract and separate the higher-risk webhook routing decision from the initial WebChat button callback path.
