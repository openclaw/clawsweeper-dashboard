## Summary

Keep open. Current main and the latest release still lack the requested WebChat/Control UI inline-button path, while the closed related WebChat button reports correctly consolidate into this issue rather than superseding it.

## What This Changes

Request to add WebChat/Control UI inline button rendering, a scoped callback RPC, and capability/config support for interactive message actions.

## Best Possible Solution

Add an additive WebChat path based on shared MessagePresentation typed actions: sanitized final-only event/history blocks, Control UI rendering with clicked or disabled state, and an authenticated scoped chat.callback RPC into the bound session; keep webhook routing as a separate hardened design.

## Reproduction Assessment

Yes, at source level: shared presentation and Telegram rendering exist, while the current WebChat protocol, config, projection, and Control UI message model have no button or callback path.

## Solution Assessment

Not exactly as originally proposed: the feature direction is valid, but the maintainable shape should use shared MessagePresentation typed actions and an authenticated gateway RPC rather than legacy shared buttons fields or a webhook-first HTTP surface.
