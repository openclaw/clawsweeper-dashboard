## Summary

Keep open. Current main still lacks the requested WebChat/Control UI inline-button path: no WebSocket button or presentation payload in the chat schema, no `chat.callback` RPC, no WebChat inline-button config, no final-chat projection for interactive metadata, and no Control UI message item/render branch. The related WebChat item at https://github.com/openclaw/openclaw/issues/54720 is closed as the narrower duplicate, while https://github.com/openclaw/openclaw/issues/46841 is Telegram-specific webhook routing, so this issue remains the canonical WebChat tracker.

## What This Changes

Request full inline button support for WebChat/Control UI, including capability config, final chat payloads, callback RPC handling, and Control UI rendering.

## Best Possible Solution

Implement an additive WebChat interactive-control path on the shared `presentation` contract: preserve sanitized final-only controls through chat events/history, render them in Control UI, and add a scoped callback RPC that initially routes selected values back into the same session; treat webhook bypass as a separate hardened design decision.

## Reproduction Assessment

Not applicable as a bug reproduction. Source inspection against current main gives high-confidence proof that the requested WebChat capability is still absent.

## Solution Assessment

No. Mirroring Telegram’s legacy `buttons` field directly is not the best long-term shape because current docs make `presentation` the portable rich-message contract; the maintainable solution should extend WebChat around that contract and define callback authorization deliberately.
