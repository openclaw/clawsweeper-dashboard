## Summary

Keep open. Current main and the latest release still lack the requested WebChat/Control UI inline-button path, while the related duplicate was closed in favor of this issue and the Telegram webhook variant was closed as a separate plugin-scoped item.

## What This Changes

The issue requests inline button support for WebChat/Control UI, including button payload delivery, callback RPC handling, and capability/config support.

## Best Possible Solution

Define an additive WebChat interactive-control contract based on shared presentation semantics, preserve sanitized final-only controls through chat events/history, render them in Control UI, and route clicks through a scoped callback path; defer webhook routing until its security model is explicit.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source and release inspection give high confidence that the requested WebChat button rendering, callback RPC, and config path are absent on current main and v2026.5.12.

## Solution Assessment

No patch is proposed, so there is no implementation to approve as-is. The maintainable direction is a protocol/config/UI design around the shared presentation contract, with webhook routing considered separately rather than copied into the first pass.
