## Summary

Keep open. Current main still lacks the requested WebChat/Control UI inline-button and callback path, and the related closed items do not supersede it; the remaining work crosses public Gateway protocol, config schema, Control UI rendering/history, and callback/webhook security design.

## What This Changes

The issue requests WebChat/Control UI inline button rendering, a WebChat capability/config path, WebSocket delivery of interactive payloads, and a callback RPC or webhook route for button clicks.

## Best Possible Solution

Implement an additive WebChat interactive-control path around the shared `presentation` contract: preserve sanitized final-only controls through chat events and history, render them in Control UI, and add a scoped in-session callback RPC; treat webhook routing as a separate hardened follow-up unless maintainers explicitly include it.

## Reproduction Assessment

Yes, at source level. Current main shows the missing schema, method registration, config, final-event projection, and Control UI render/click paths without needing a live Gateway run.

## Solution Assessment

No for the literal proposed shape. The outcome is useful, but the safer maintainable first step is a WebChat implementation of the shared `presentation` contract plus a scoped `chat.callback`, with webhook egress designed separately.

## Security Review

Status: needs_attention

Summary: The issue requests browser callback and optional webhook behavior that crosses Gateway trust boundaries and needs security design before implementation.

Concerns:

- **[medium] Define callback RPC authorization and replay semantics:** `src/gateway/method-scopes.ts:152`
  - body: A WebChat callback RPC would turn authenticated browser clicks into session input. It needs method scoping, session/message ownership checks, callback value validation, replay or disabled-state behavior, and audit semantics before shipping.
  - confidence: 0.88
- **[medium] Harden webhook egress separately:**
  - body: The proposed callback webhook would let browser UI clicks trigger Gateway-originated HTTP requests. That needs explicit SSRF/egress policy, authentication, retries, and observability rather than being bundled as an incidental UI-button feature.
  - confidence: 0.82
