## Summary

Keep open. Current main and the latest release still lack the requested WebChat/Control UI inline-control path: no button/presentation payload in the WebSocket chat contract, no scoped callback RPC, no WebChat capability/config surface, and no Control UI renderer for clickable controls. The narrower duplicate at https://github.com/openclaw/openclaw/issues/54720 is closed in favor of this canonical tracker.

## What This Changes

Requests WebChat/Control UI support for inline button presentation, callback reporting, and capability/config controls.

## Best Possible Solution

Implement an additive WebChat interactive-control path using the shared presentation contract, preserve sanitized final-only control blocks through chat events and history, render them in Control UI, and route clicks through a scoped same-session callback; split webhook routing into a separate security-reviewed follow-up.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source and docs inspection give high confidence that current main lacks the WebChat event payload, callback RPC, config path, and Control UI renderer surfaces requested here.

## Solution Assessment

No concrete patch is proposed, so there is no implementation to approve. The requested direction is valid, but the maintainable path should use shared presentation and a scoped callback design rather than copying Telegram-shaped buttons into WebChat alone.
