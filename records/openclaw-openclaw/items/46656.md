## Summary

Keep open. Current main still lacks WebChat/Control UI inline-button rendering, event/history projection, and callback RPC support, while the latest related discussion consolidates the design around this issue instead of superseding it.

## What This Changes

Request to add inline button rendering and callbacks to WebChat/Control UI, including capability configuration, final chat event/history payloads, an authenticated callback RPC, and Control UI rendering.

## Best Possible Solution

Use the existing MessagePresentation typed-action contract for WebChat controls, preserve sanitized final-only controls through chat events/history, add scoped authenticated chat.callback, and defer webhook routing to a separate security design.

## Reproduction Assessment

Yes, at source level: presentation buttons exist and channel renderers can consume them, but current WebChat protocol, config, final projection, callback registration, and Control UI rendering do not carry or display controls.

## Solution Assessment

Partly: WebChat button support is the right direction, but the best implementation should use typed MessagePresentation actions and authenticated Gateway RPC rather than a legacy buttons field or webhook-first surface.
