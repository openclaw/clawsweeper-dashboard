## Summary

Keep open. Current main still has the prompt/runtime mismatch: internal exec-completion prompts ask for `HEARTBEAT_OK` without `<final>` or `NO_REPLY` guidance, while strict final-tag extraction can erase a bare `NO_REPLY` before incomplete-turn handling sees it. Open PR #73785 is adjacent post-payload suppression for #73149, not a fix for this zero-payload prompt path.

## What This Changes

Fix delivery-disabled async exec-completion runs so they produce an intentional silent final reply instead of empty-response retries and incomplete-turn errors.

## Best Possible Solution

Update the internal/no-output exec completion contract to request exactly `<final>NO_REPLY</final>` for intentional non-delivery, with regression coverage for strict-final unwrapped silent replies and without relaxing final-tag enforcement broadly.

## Reproduction Assessment

Yes. A high-confidence source reproduction is an exec completion with `canRelayToUser=false`, strict final-tag enforcement, and a bare `NO_REPLY`: current main can strip it to empty assistant text, then take the empty-response retry and incomplete-turn path.

## Solution Assessment

Yes for a narrow prompt/runtime contract fix. Prefer explicit `<final>NO_REPLY</final>` guidance or a tightly scoped raw-silent guard for exec-completion events; broad changes to final-tag extraction would risk leaking reasoning text.
