## Summary

Keep open. Current main still has the reported internal exec-completion prompt wording, and the tagged-provider final-tag enforcement plus empty-response retry path makes the reported bare, non-final silent reply failure plausible and narrowly actionable.

## What This Changes

The issue requests changing non-deliverable async exec-completion handling so the model produces a runtime-accepted silent reply instead of triggering empty-response retries and incomplete-turn errors.

## Best Possible Solution

Make the no-output and `deliverToUser=false` exec-completion path settle as an intentional silent/no-op result that the embedded runner accepts without retrying, while preserving normal user relay for relayable exec output. Add focused coverage for the prompt and the runtime no-delivery path, and include a user-facing changelog fix entry.

## Reproduction Assessment

Yes. The static reproduction path is high confidence: call `buildExecEventPrompt` with `deliverToUser: false`, then follow the current tagged-provider final-tag enforcement and empty-response retry logic; the existing tests already cover the empty-turn retry/error side when silence is not allowed.

## Solution Assessment

Unclear as proposed. The reporter's final-wrapped silent-token direction addresses the final-tag race, but the implementation should choose the token and runtime normalization together so the heartbeat runner neither retries nor leaks `NO_REPLY` as visible heartbeat output.
