## Summary

Current main still has the reported mismatch: hidden exec-completion prompts ask for `HEARTBEAT_OK`, while the runtime has a separate `NO_REPLY` silent-reply contract and an empty-response retry path that can surface incomplete-turn errors when no final payload is produced.

## What This Changes

The issue requests changing internal async exec-completion prompts so disabled user delivery produces an explicit `<final>NO_REPLY</final>` silent reply instead of a bare heartbeat acknowledgment.

## Best Possible Solution

Keep the heartbeat default ack contract unchanged, but make the exec-completion no-delivery and no-output prompts use the shared silent reply token with explicit final-answer wrapping, then cover that with prompt-level and heartbeat-runner regression tests plus a user-facing changelog fix entry.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: `buildExecEventPrompt(..., { deliverToUser: false })` currently returns the `HEARTBEAT_OK only` branch, and existing embedded-runner tests show zero-payload terminal turns retry and can surface incomplete-turn errors.

## Solution Assessment

Yes, the requested prompt-level fix is the narrowest maintainable direction. It aligns this specific hidden-delivery prompt with the existing `NO_REPLY` contract instead of weakening the runtime’s general empty-turn detection.
