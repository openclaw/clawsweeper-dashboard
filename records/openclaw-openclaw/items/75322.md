## Summary

Keep open. Current main still has the prompt/runtime mismatch described in the report, and the adjacent no-op suppression PR only handles emitted sentinel payloads after a visible payload exists, not the zero-payload strict-final path reported here.

## What This Changes

The issue requests changing async exec-completion handling so internal-only command completions produce or accept an intentional silent terminal result instead of triggering empty-response retries and incomplete-turn delivery.

## Best Possible Solution

Make the no-delivery exec-completion path terminate as intentional silence without entering visible-answer retry logic, while preserving normal relay behavior for output-bearing exec completions. The narrowest maintainable path is to align the prompt and runner contract around an explicit final-wrapped silent reply, with tests covering strict final-tag zero-payload behavior.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists: route an exec completion through `canRelayToUser=false`, observe the `HEARTBEAT_OK` prompt, then apply strict final-tag extraction to a bare `NO_REPLY` and follow the empty-response retry path to the incomplete-turn error.

## Solution Assessment

Yes for a focused fix, but not for closing this issue. The suggested direction is appropriate if implemented with coverage for the no-delivery exec branch and without relying solely on post-emission sentinel filtering from #73785.

## Repair Work Prompt

Fix the async exec-completion no-delivery prompt/runtime conflict from https://github.com/openclaw/openclaw/issues/75322. Current main asks `buildExecEventPrompt(..., { deliverToUser: false })` to reply `HEARTBEAT_OK only`, but strict reasoning-tag runs require `<final>...</final>` and can turn bare `NO_REPLY` into zero payloads, causing empty-response retries and incomplete-turn delivery. Keep the fix scoped to exec completion no-delivery and empty-output behavior; preserve normal delivery for relayable/output-bearing exec completions. Coordinate with #73785/#73149 as adjacent post-emission sentinel suppression, but do not depend on that PR as the whole fix. Add focused tests in `src/infra/heartbeat-events-filter.test.ts` and embedded-runner coverage for the strict zero-payload/silent terminal path. Add a single-line `CHANGELOG.md` fix entry crediting the reporter/context; do not add broad provider, channel, or plugin changes.
