## Summary

Keep open. Current main still has the reported prompt/runtime mismatch: internal exec-completion prompts ask for bare `HEARTBEAT_OK`, while the tagged-provider path can discard unwrapped assistant text and send a zero-payload turn through retry and incomplete-turn handling.

## What This Changes

The issue requests that async exec-completion events with no command output or user delivery disabled settle as accepted silent/no-op replies instead of prompting models toward a response shape that can trigger empty-turn retries.

## Best Possible Solution

Make the no-output and `deliverToUser=false` exec-completion path resolve as an intentional silent/no-op result that the embedded runner accepts without retrying, while preserving normal user relay for relayable exec output and keeping heartbeat acknowledgement behavior separate from silent-delivery behavior.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: call `buildExecEventPrompt` with `deliverToUser: false`, then follow the current tagged-provider final-tag extraction and empty-response retry path; the existing tests already cover the retry/error behavior when silence is not allowed.

## Solution Assessment

Unclear as proposed. The reporter's final-wrapped silent-token direction addresses the final-tag race, but the final fix should choose the token and runtime normalization together so the runner neither retries nor leaks `NO_REPLY` or `HEARTBEAT_OK` as user-visible output.
