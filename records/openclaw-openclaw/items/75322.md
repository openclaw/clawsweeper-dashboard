## Summary

Keep open. Current main and the latest release still have the legacy internal exec-completion prompt asking for `HEARTBEAT_OK only` without `NO_REPLY` or `<final>` guidance, while strict final-tag handling can erase a bare `NO_REPLY` before the incomplete-turn guard runs.

## What This Changes

The issue asks OpenClaw to align the internal/no-output async exec-completion prompt with the runtime silent-reply contract so disabled user delivery does not produce empty-turn errors.

## Best Possible Solution

Update the legacy internal/no-output exec-completion prompt to express the existing silent terminal contract unambiguously, with focused regression coverage for strict final-tag sessions.

## Reproduction Assessment

Yes, source-reproducible with high confidence: `deliverToUser=false`, non-response-tool exec completion, strict final-tag enforcement, and a bare `NO_REPLY` can produce zero payloads and enter the empty-response retry path. I did not run a live model in this read-only review.

## Solution Assessment

Yes. The narrowest maintainable fix is to align the internal/no-output exec-completion prompt with the existing silent-reply final contract; https://github.com/openclaw/openclaw/pull/73785 is adjacent emitted-payload filtering, not a complete fix for this zero-payload prompt path.
