## Summary

Keep open. Current main still has the reported prompt/runtime conflict: the heartbeat exec-completion no-delivery branch asks for `HEARTBEAT_OK` without a final-wrapper contract, while strict final-tag handling can turn a bare silent reply into zero visible payloads and drive the empty-response retry path.

## What This Changes

The issue requests fixing the internal-only heartbeat exec-completion path so silent async command completions do not surface incomplete-turn errors.

## Best Possible Solution

Make the no-delivery exec-completion path produce or accept an intentional silent terminal result without invoking visible-answer retries, while preserving normal user relay for output-bearing exec completions. Add focused prompt/runtime regression coverage and a user-facing changelog fix entry crediting the reporter.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists: send an exec completion through `buildExecEventPrompt` with `deliverToUser: false`, have a strict final-tag model emit bare `NO_REPLY`, then follow the current zero-payload empty-response retry and incomplete-turn path.

## Solution Assessment

Unclear for the reporter's exact diff, but the direction is right. The narrowest maintainable fix is to align the internal-only exec prompt with the runtime silent-reply contract, likely by explicitly requiring `<final>NO_REPLY</final>` and covering the strict-mode zero-payload path rather than broadly accepting silent tokens everywhere.
