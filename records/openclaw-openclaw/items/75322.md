## Summary

Keep open. Current main and latest release still keep the legacy disabled-delivery exec-completion prompt that asks for `HEARTBEAT_OK only` without `NO_REPLY` or `<final>` guidance, and the strict final-tag path can still turn an unwrapped silent reply into zero payloads and an incomplete-turn error.

## What This Changes

The issue asks OpenClaw to align disabled-delivery async exec-completion prompts with the runtime silent-reply/final-tag contract so background completions do not surface incomplete-turn noise.

## Best Possible Solution

Update the legacy internal/no-output exec-completion prompt to express the existing silent terminal contract unambiguously, with focused regression coverage for strict final-tag sessions and no behavior change to structured `heartbeat_respond` mode.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: `deliverToUser=false`, non-response-tool exec completion, strict final-tag enforcement, and a bare `NO_REPLY` can produce zero payloads and enter the empty-response retry/incomplete-turn path.

## Solution Assessment

Yes. The narrowest maintainable fix is to align the internal/no-output exec-completion prompt with the existing silent-reply final contract; the adjacent sentinel-output PR is useful but not sufficient for this prompt-layer zero-payload path.
