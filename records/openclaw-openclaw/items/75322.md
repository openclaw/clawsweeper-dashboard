## Summary

Keep open. Current main still has the reported internal exec-completion prompt wording, the focused tests still expect `HEARTBEAT_OK only`, and the final-tag extraction plus empty-response retry path makes the provided bare `NO_REPLY` trace a plausible current bug rather than a stale report.

## What This Changes

The issue asks OpenClaw to stop non-deliverable async exec-completion prompts from causing bare silent replies to be classified as empty turns and surfaced as incomplete-turn errors.

## Best Possible Solution

Make non-relayable and no-output exec-completion events settle as an intentional internal no-op that the embedded runner accepts without retrying, while preserving normal relay behavior for command output and failed metadata-only completions.

## Reproduction Assessment

Yes. The high-confidence static path is to call `buildExecEventPrompt` with `deliverToUser: false`, then follow the current reasoning-provider final-tag extraction and empty-response retry/error logic when a bare `NO_REPLY` produces zero payloads.

## Solution Assessment

Unclear for the reporter's exact prompt-only diff. The safer narrow fix is to align the generated prompt, silent token, and runner normalization so the heartbeat run neither retries nor delivers the silent token as visible output.
