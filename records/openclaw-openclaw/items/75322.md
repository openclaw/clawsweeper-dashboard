## Summary

Keep open. Current main still leaves the internal-only exec-completion prompt asking for `HEARTBEAT_OK` without final-tag or silent-token guidance, and strict final filtering can still erase a bare `NO_REPLY` before the newer exact-silent `assistantTexts` guard can see it.

## What This Changes

The issue requests that non-user-delivered async exec-completion prompts avoid ambiguous bare `HEARTBEAT_OK` replies that race strict final-tag and silent-reply handling.

## Best Possible Solution

Update the internal/no-output exec completion contract so non-delivered successful exec completions produce an intentional silent terminal result, with regression coverage for strict-final unwrapped `NO_REPLY`.

## Reproduction Assessment

Yes. A high-confidence source reproduction is an exec completion with `canRelayToUser=false`, strict final-tag enforcement, and a bare `NO_REPLY`: current main can strip it to empty visible text before `assistantTexts`, then run the empty-response retry/incomplete-turn path.

## Solution Assessment

Yes for a narrow prompt/runtime contract fix. Prefer explicit `<final>NO_REPLY</final>` guidance or a scoped raw silent-token guard for exec-completion system events, not a broad change to final-tag extraction.
