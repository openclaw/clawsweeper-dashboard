## Summary

Keep open. Current main still has the reported internal exec-completion prompt wording, and the strict final-tag plus empty-response retry path still makes the reported bare silent reply failure plausible and narrowly actionable.

## What This Changes

The issue requests changing internal async exec-completion handling so user-delivery-disabled or no-output branches do not trigger incomplete-turn errors when the model emits a silent reply.

## Best Possible Solution

Make the no-output and `deliverToUser=false` exec-completion path settle as an intentional silent/no-op result accepted by the embedded runner, while preserving normal relay for real exec output and actionable failures. Add focused prompt and runtime coverage plus a user-facing changelog fix entry.

## Reproduction Assessment

Yes. The static reproduction path is high confidence: call `buildExecEventPrompt` with `deliverToUser: false`, then follow reasoning-tag final extraction and the empty-response retry path when a bare `NO_REPLY` produces zero payloads.

## Solution Assessment

Unclear as proposed. The reporter's `<final>NO_REPLY</final>` prompt direction addresses the final-tag race, but the implementation should verify the token and runtime handling together so OpenClaw neither retries nor leaks a silent sentinel as visible output.
