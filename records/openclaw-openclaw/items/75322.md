## Summary

Keep open. Current main still has the reported internal-only exec-completion prompt conflict: the no-delivery branch asks for a `HEARTBEAT_OK` text reply while strict final-tag mode can discard a bare `NO_REPLY` before heartbeat/no-op output filtering sees a payload. The related #73785 discussion is useful context, but it targets post-payload sentinel suppression and does not cover this zero-payload strict-final path.

## What This Changes

The issue requests fixing internal-only async exec completion prompts so suppressed delivery does not cause bare silent replies to become empty-response retries and user-visible incomplete-turn errors.

## Best Possible Solution

Make the internal-only exec-completion path produce or accept an intentional silent terminal outcome without entering visible-answer retry, while preserving normal user relay behavior for output-bearing exec completions and keeping the newer `heartbeat_respond` tool path compatible.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main by resolving an exec completion with `canRelayToUser` false, following the `HEARTBEAT_OK` prompt into strict `<final>` extraction, and observing that a bare non-final `NO_REPLY` yields `payloadCount=0` and the empty-response retry path.

## Solution Assessment

Unclear for the reporter's exact diff, but the direction is right. The safest fix is a narrow prompt/runtime contract change for no-delivery exec completions, with tests proving that the silent path terminates cleanly and relayable exec output remains deliverable.
