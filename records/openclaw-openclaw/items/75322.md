## Summary

Current main still contains the conflicting internal exec-completion prompt described in this issue, and related discussion correctly distinguishes it from the adjacent sentinel-suppression PR; the issue should remain open for a narrow runtime prompt/retry fix.

## What This Changes

The issue requests that internal-only async exec completion prompts avoid `HEARTBEAT_OK`/silent-reply ambiguity and require a runtime-acceptable final-wrapped silent response.

## Best Possible Solution

Make internal-only and metadata-only successful exec completions produce an intentional final-wrapped silent terminal result, while preserving relayable exec output and failed no-output command reporting.

## Reproduction Assessment

Yes. The high-confidence path is an exec completion with `canRelayToUser=false`, strict final-tag enforcement, and a bare `NO_REPLY`; current source shows that path yields no final payload and can enter empty-response retry/incomplete-turn handling.

## Solution Assessment

Yes, a narrow prompt-layer fix with regression coverage is the best path. Prefer explicit `<final>NO_REPLY</final>` wording for no-delivery exec completions rather than relaxing strict final extraction globally.
