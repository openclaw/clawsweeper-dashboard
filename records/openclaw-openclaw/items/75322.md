## Summary

Keep open. Current main still has the internal-only exec-completion prompt that asks for `HEARTBEAT_OK` without a `<final>...</final>` example, and the strict final-tag plus empty-response retry paths make the reporter's zero-payload `NO_REPLY` failure mode credible. The related open PR #73785 is adjacent mitigation, but it does not cover the prompt branch or the strict zero-payload path described here.

## What This Changes

Fix the heartbeat exec-completion prompt/runtime path so internal-only async command completions do not trigger visible empty-response retries and incomplete-turn errors when the model returns a silent reply.

## Best Possible Solution

Make the no-delivery exec-completion path produce or accept an intentional silent terminal result without invoking visible-answer retries, while preserving normal relay behavior for output-bearing exec completions. Add focused prompt and embedded-runner coverage, plus a user-facing changelog fix entry crediting the reporter/context.

Yes. Unclear for the exact suggested diff, but the direction is sound. The safest fix is either explicit `<final>NO_REPLY</final>` prompting for internal exec completions or a narrowly scoped runtime allowance for intentional silent internal completions, without changing delivery-capable exec summaries.
