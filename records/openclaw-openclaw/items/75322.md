## Summary

Keep open. Current main still has the reported HEARTBEAT_OK-based internal exec-completion prompt, while reasoning-tag final extraction can turn a bare NO_REPLY into zero payloads and the embedded runner still retries/errors on zero-payload turns unless the run is explicitly allowed to be silent.

## What This Changes

The issue requests making async exec-completion events with disabled user delivery settle as intentional silent/no-op results instead of causing empty-response retries and incomplete-turn errors.

## Best Possible Solution

Add a focused core runtime fix that aligns the no-output and `deliverToUser=false` exec-completion prompt with the runtime's silent-reply contract, or otherwise marks that run path as intentionally silent, so the embedded runner accepts it without visible-answer retries while relayable exec output still reaches the user.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: call `buildExecEventPrompt` with `deliverToUser: false`, then follow reasoning-tag final extraction and the current empty-response retry path when the model emits bare `NO_REPLY`.

## Solution Assessment

Unclear as the exact suggested diff. Final-wrapped `NO_REPLY` is a plausible part of the fix, but the best solution should also test the runtime outcome so the no-delivery branch cannot retry, leak a control token, or change normal heartbeat ack behavior.
