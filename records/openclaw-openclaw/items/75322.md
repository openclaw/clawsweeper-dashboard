## Summary

Keep open. Current main still has the reported internal-only exec-completion prompt, and the surrounding final-tag and empty-response retry paths make the zero-payload `NO_REPLY` failure mode plausible and narrowly repairable.

## What This Changes

The issue requests that the heartbeat async exec-completion no-delivery path stop prompting for bare `HEARTBEAT_OK` in a way that can race strict final-tag silent-reply handling.

## Best Possible Solution

Make internal-only exec-completion handling terminate as intentional silence without entering visible-answer retries, while preserving user-relay summaries for real exec output and failed completions. The narrowest maintainable path is to align the no-delivery prompt and tests around an explicit final-wrapped silent result, or to set an equally narrow runtime silence allowance for this specific internal heartbeat path.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists: resolve an exec-completion heartbeat with `canRelayToUser` false, observe the current bare `HEARTBEAT_OK` prompt, then follow strict `<final>` extraction and empty-response retry behavior for a bare non-final `NO_REPLY`.

## Solution Assessment

Unclear for the exact suggested diff, because the no-output branch and relayable failure cases need careful separation. The best fix is a narrow prompt/runtime contract change for internal-only exec completions, with regression coverage for the zero-payload strict-mode path.
