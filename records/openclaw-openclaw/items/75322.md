## Summary

Keep open. Current main still has the internal-only exec-completion prompt conflict: the no-delivery path asks for `HEARTBEAT_OK` without final-tag guidance, while strict final extraction can turn an unwrapped silent reply into zero payloads and the empty-response retry path can surface an incomplete-turn error.

## What This Changes

The issue requests making `buildExecEventPrompt` handle internal-only async exec completions as intentional silent replies instead of ambiguous heartbeat acknowledgments.

## Best Possible Solution

Make internal-only and metadata-only successful exec completions produce an intentional silent terminal result, while preserving relayable exec output and failed no-output command reporting.

## Reproduction Assessment

Yes. The high-confidence path is an exec completion with `canRelayToUser=false`, strict final-tag enforcement, and a bare silent reply: current source shows this becomes zero visible payloads and enters the empty-response retry/incomplete-turn path.

## Solution Assessment

Yes for a narrow prompt-layer fix with regression coverage. Prefer explicit `<final>NO_REPLY</final>` or a scoped heartbeat response tool outcome for no-delivery exec completions, not a broad relaxation of strict final-tag extraction.
