## Summary

Keep open. Current main still contains the reported prompt/retry mismatch: the internal-only exec-completion prompt asks for `HEARTBEAT_OK` without a `<final>` example, while strict final-tag extraction can turn a bare `NO_REPLY` into zero payloads and the embedded runner then retries/surfaces an incomplete-turn error. The discussion also correctly notes that open PR #73785 is adjacent mitigation, not a full fix for this zero-payload path.

## What This Changes

The issue requests changing internal-only async exec-completion handling so disabled user delivery terminates silently instead of prompting models into bare `NO_REPLY` responses that produce incomplete-turn errors.

## Best Possible Solution

Make disabled-delivery exec completions terminate as an intentional silent result without entering visible-answer retries. The narrowest maintainable path is to update the no-delivery prompt and add embedded-runner coverage for the strict final-tag zero-payload case, while preserving normal output relay and handling the no-output branch deliberately rather than broadening `NO_REPLY` behavior everywhere.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists on current main: an exec completion with `canRelayToUser` false receives the `HEARTBEAT_OK` internal-only prompt, a strict final-tag model can emit bare `NO_REPLY`, final extraction yields zero payloads, and the embedded runner takes the empty-response retry/error path.

## Solution Assessment

Yes for a focused runtime/prompt fix, but the reporter's exact diff should be refined. The fix should either prompt for `<final>NO_REPLY</final>` only on intentional internal no-delivery paths or add a narrow runtime silent terminal path for those exec completions, without changing user-deliverable exec output behavior.
