## Summary

Keep open. Current main still has the reported internal exec-completion prompt wording, and the existing final-tag enforcement plus empty-response retry path makes the reporter's bare `NO_REPLY` failure mode plausible and narrowly actionable.

## What This Changes

The issue asks OpenClaw to stop internal async exec-completion events with disabled user delivery from triggering empty-response retries and user-visible incomplete-turn errors when the model replies with the silent token without final-answer wrapping.

## Best Possible Solution

Make internal/no-output exec-completion events settle as intentional silence accepted by the embedded runner, while preserving normal relay for user-deliverable exec output. Add focused prompt and runner coverage, plus a user-facing changelog fix entry.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: call `buildExecEventPrompt` with `deliverToUser: false`, then follow reasoning-provider final-tag enforcement to `payloads=0` and the current empty-response retry/error behavior when silence is not explicitly allowed.

## Solution Assessment

Unclear as proposed. Final-wrapped `NO_REPLY` is a plausible narrow fix, but the maintainable solution should choose the prompt text and runtime silent/no-delivery handling together so the runner neither retries nor leaks a control token.
