## Summary

Keep open. Current main still has the no-delivery exec-completion prompt conflict, and the related #73785 output-filter work is adjacent rather than a fix for the zero-payload strict-final path.

## What This Changes

The issue asks OpenClaw to handle async exec-completion events with delivery disabled without triggering empty-response retries or incomplete-turn errors.

## Best Possible Solution

Make the no-delivery exec-completion path produce or accept an intentional silent terminal result without invoking visible-answer retries, while preserving normal relay behavior for output-bearing exec completions.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists by resolving an exec completion with `canRelayToUser` false, following the `HEARTBEAT_OK` prompt into strict `<final>` extraction, and observing that unwrapped silent text becomes zero payloads and reaches the empty-response retry path.

## Solution Assessment

Unclear for the reporter's exact diff. A final-wrapped silent result for the no-delivery branch is the right direction, but tests should protect normal relayable exec output and no-output failure behavior from accidental suppression.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/75322. Current main's `buildExecEventPrompt` tells no-output exec events and `deliverToUser=false` exec completions to reply `HEARTBEAT_OK only` without `<final>` wrapping; in strict-final sessions a model can emit bare `NO_REPLY`, leaving `payloads=0` and triggering empty-response retry/incomplete-turn noise. Implement a narrow fix in the heartbeat exec-completion prompt and/or runner classification so the no-delivery path yields an intentional silent terminal result. Do not change normal relayable exec output behavior, broad silent-reply policy, or unrelated cron prompt behavior. Add focused regression coverage for the strict-final zero-payload/no-delivery case and update `CHANGELOG.md` if the fix is user-facing under repo policy.
