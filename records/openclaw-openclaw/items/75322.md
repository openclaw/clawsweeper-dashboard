## What This Changes

The issue asks OpenClaw to make async exec-completion events with user delivery disabled settle silently instead of triggering empty-response retries and incomplete-turn errors.

## Best Possible Solution

Make the no-delivery exec-completion path produce or accept an intentional silent terminal result without invoking visible-answer retries, while preserving normal relay behavior for output-bearing exec completions.

## Repair Work Prompt

Fix the async exec-completion no-delivery bug described in https://github.com/openclaw/openclaw/issues/75322. Current `buildExecEventPrompt(..., { deliverToUser: false })` asks for `HEARTBEAT_OK only` without `<final>` wrapping; on strict final-tag providers a model can emit bare `NO_REPLY`, leaving zero payloads and causing the embedded runner to retry and surface an incomplete-turn error. Keep the fix narrow: make the no-delivery exec-completion path become an intentional silent/no-op terminal result accepted by the runner, while preserving normal relay behavior for output-bearing exec completions. Treat https://github.com/openclaw/openclaw/pull/73785 and https://github.com/openclaw/openclaw/issues/73149 as adjacent output-filtering work, not a substitute for this prompt/runtime fix. Be careful with the no-output branch when a real delivery target exists, because blindly switching it to `NO_REPLY` could leak or rewrite the silent token through outbound delivery. Add or update focused tests in `src/infra/heartbeat-events-filter.test.ts` and the closest embedded-runner/heartbeat-runner tests to prove the no-delivery path does not retry or surface an error and relayable exec output still delivers. Add a single-line `CHANGELOG.md` fix entry thanking `@mbrya859-byte`, `@martingarramon`, and `@arcabotai`. Do not change generic direct/group silent-reply policy or unrelated heartbeat token semantics.
