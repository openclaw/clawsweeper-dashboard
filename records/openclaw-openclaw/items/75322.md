## Summary

Keep open. Current main still has the internal-only exec-completion prompt conflict: the heartbeat path asks for `HEARTBEAT_OK` while strict final-tag extraction discards unwrapped text and can still enter the empty-response retry path; related #73785 is adjacent output filtering, not a fix for the zero-payload prompt-layer case.

## What This Changes

The issue asks for the internal-only async exec completion prompt to stop steering models toward an unwrapped heartbeat or silent response that triggers empty-response retries.

## Best Possible Solution

Align the internal-only exec-completion path with the strict final-tag and silent-reply contract, or explicitly mark that path as intentional silence before empty-response retries, while preserving normal relay and failed-exec behavior.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: route an exec completion through `canRelayToUser=false`, get the current `HEARTBEAT_OK` internal prompt, emit bare `NO_REPLY` or any unwrapped response under strict final-tag extraction, and observe zero payloads entering the empty-response retry path.

## Solution Assessment

Unclear for the reporter's exact diff. Prompting for exactly `<final>NO_REPLY</final>` is the narrowest likely fix, but the implementation should include regression tests that prove relayable output and metadata-only failures still behave correctly.

## Repair Work Prompt

Fix the internal-only async exec completion silence bug from https://github.com/openclaw/openclaw/issues/75322. Current main asks `buildExecEventPrompt(..., { deliverToUser: false })` to reply `HEARTBEAT_OK only`, while strict final-tag runs discard bare `NO_REPLY`/unwrapped text and enter empty-response retries. Make the no-delivery exec-completion path produce or accept an intentional silent terminal result without visible-answer retries. Prefer changing the no-output and no-delivery exec prompt wording to require exactly `<final>NO_REPLY</final>` unless code inspection shows a narrower runtime-silence hook is safer. Preserve normal relay behavior for output-bearing exec completions and the current metadata-only failed-exec message. Add/update focused tests in `src/infra/heartbeat-events-filter.test.ts` and, if needed, `src/agents/pi-embedded-runner/run.incomplete-turn.test.ts` for the zero-payload strict-final path. Check overlap with #73785 but do not rely on it as the fix. Add a single-line `CHANGELOG.md` fix entry because this is user-visible channel noise. Do not broaden heartbeat/cron suppression semantics beyond this exec-completion path.
