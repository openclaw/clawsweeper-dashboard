## Summary

Keep open. Current main still has the reported no-delivery exec-completion prompt conflict: the runtime asks for `HEARTBEAT_OK` without final wrapping while strict final-tag handling discards bare `NO_REPLY`, so the zero-payload retry path can still surface an incomplete-turn error. The linked #73785 work is adjacent output filtering after a sentinel payload exists, not a fix for this prompt-layer zero-payload path.

## What This Changes

The issue requests changing internal/no-delivery async exec-completion prompting so it uses the accepted silent-reply contract instead of an ambiguous unwrapped `HEARTBEAT_OK` instruction.

## Best Possible Solution

Make the no-delivery exec-completion path produce an intentional accepted silent terminal result, preferably by aligning the generated prompt with the strict final-tag and `NO_REPLY` contract, while preserving normal user relay behavior for output-bearing and failed exec completions.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists by resolving an exec completion with `canRelayToUser=false`, following the current `HEARTBEAT_OK` prompt into strict `<final>` extraction, and observing that a bare non-final `NO_REPLY` becomes zero payloads and enters the empty-response retry path.

## Solution Assessment

Yes for the no-delivery exec-completion branch: changing the prompt to an explicit final-wrapped silent reply is the narrowest maintainable direction. The empty-output delivery-capable branch should be covered carefully so a silent sentinel is not surfaced where a visible failure/status update is still expected.
