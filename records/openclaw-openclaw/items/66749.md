## Summary

Keep open. The branch addresses a real bug, but the current patch is not ready: it globally suppresses cron-run exec wakes despite current main’s cron routing and delivery contracts, its new test assertion omits `trusted: false`, and the posted environment note is not after-fix proof.

## What This Changes

The PR adds subagent and cron-run guards around exec-event heartbeat requests in the bash exec runtime, adds a subagent regression test, and adds a changelog entry.

## Best Possible Solution

Land a narrowed producer-side subagent guard, keep cron-run behavior tied to the existing scoped wake and delivery-mode policy or a separate cron design, fix the regression assertion, and require redacted patched-setup proof.

## Reproduction Assessment

Yes, at source level. Current main unconditionally requests exec-event heartbeats from both exec completion call sites, and `resolveHeartbeatSession` routes forced subagent keys to the main session; I did not run a live WebChat/subagent reproduction.

## Solution Assessment

No for the current patch. The subagent producer-side guard is the narrow maintainable direction, but the cron-run guard conflicts with current main’s explicit cron wake routing and delivery-mode tests, and the new assertion is incomplete.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Keep cron-run wakes on the existing policy:** `src/agents/bash-tools.exec-runtime.ts:351`
  - body: This condition suppresses every cron-run exec heartbeat. Current main deliberately remaps cron-run exec events and wakes to the owning main/global queue, and cron delivery tests only disable automatic exec completion notifications for `delivery.mode` `none`; announce and webhook modes still keep them. Restrict this guard to subagents or make cron suppression follow the existing delivery policy.
  - confidence: 0.92
- **[P2] Match the queued system event payload:** `src/agents/bash-tools.exec-runtime.test.ts:524-527`
  - body: `emitExecSystemEvent` enqueues `trusted: false` unconditionally, so this exact `toHaveBeenCalledWith` object omits a field present on the actual payload and will fail. Add `trusted: false` or use a partial matcher.
  - confidence: 0.98
