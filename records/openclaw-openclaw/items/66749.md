## Summary

Keep open: the subagent wake suppression is a plausible narrow bug fix, but the current PR still has blocking test/type failures, lacks required real behavior proof, and now suppresses cron-run wakes even though current heartbeat resolution treats forced cron-run keys as their own target rather than a subagent-style fallback.

## What This Changes

The PR guards bash exec completion heartbeat requests for subagent and cron-run session keys, adds one emitExecSystemEvent regression test, and records the fix in the changelog.

## Best Possible Solution

Land a corrected producer-side subagent guard with passing regression coverage, and either remove cron-run suppression or replace it with a separately proven cron-specific routing fix plus real setup proof.

## Reproduction Assessment

Yes from source inspection. Current main unconditionally requests exec-event heartbeats for child session keys, and forced subagent keys are resolved back to the main session; I did not run a live WebChat/subagent scenario.

## Solution Assessment

No for the current patch. A producer-side subagent guard is the narrow maintainable direction, but the submitted cron-run suppression is not supported by current resolver behavior and the added test does not compile.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.95

Full review comments:

- **[P1] Keep cron-run exec wakes targetable:** `src/agents/bash-tools.exec-runtime.ts:351`
  - body: The guard also suppresses cron-run session keys, but current `resolveHeartbeatSession` only remaps subagent keys; non-subagent forced keys are canonicalized and returned as their own session. Dropping this wake can leave cron-run exec completion events queued instead of waking the owning run, so scope this PR to subagent keys or add a cron-specific reroute with coverage.
  - confidence: 0.88
- **[P1] Use the current heartbeat mock:** `src/agents/bash-tools.exec-runtime.test.ts:494`
  - body: The new test calls `requestHeartbeatNowMock`, but this file now mocks `requestHeartbeat` as `requestHeartbeatMock`. CI already fails test typechecking on this stale identifier before the regression test can exercise the guard.
  - confidence: 0.99
- **[P2] Match the queued system-event payload:** `src/agents/bash-tools.exec-runtime.test.ts:490-493`
  - body: `emitExecSystemEvent` always passes `trusted: false` to `enqueueSystemEvent`, but the new expectation omits that field. Once the mock name is fixed, this assertion should match the current payload shape or use an object-containing matcher.
  - confidence: 0.97
