## Summary

Keep open. Current main still has the unguarded exec-event heartbeat wake in both exec completion paths, and the heartbeat resolver still treats forced subagent session keys as main-session wakes. The PR remains a plausible implementation candidate for the same-author open issue #66748; the newer cron-run comment is a maintainer scope question, not cleanup-close evidence.

## What This Changes

The PR imports `isSubagentSessionKey`, skips `requestHeartbeatNow` for subagent session keys in both exec-event wake call sites, and adds an `emitExecSystemEvent` regression test.

## Best Possible Solution

Keep the implementation path open and land this PR or an equivalent narrow patch after maintainer review confirms scope. The final patch should preserve system-event enqueueing, suppress redundant wakes for non-user-facing child sessions that should not wake the parent, and include regression coverage for both explicit exec system events and notify-on-exit if that path remains guarded.

## Reproduction Assessment

Yes. A source-level reproduction is clear: on current main, call `emitExecSystemEvent` with `sessionKey: "agent:main:subagent:abc"` or let a background subagent exec exit with `notifyOnExit`; both paths reach `requestHeartbeatNow` with an `exec-event` wake, and the heartbeat resolver maps forced subagent keys to the main session.

## Solution Assessment

Unclear. For the original subagent symptom, guarding the wake at the two exec-event call sites is the narrowest maintainable shape; the newer cron-run report may make a shared child-run-session guard the better final solution before merge.
