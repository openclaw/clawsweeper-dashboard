## Summary

Keep open: the subagent heartbeat bug is source-reproducible, but this PR still has a blocking cron-run behavior regression, a failing regression-test assertion, and no after-fix real behavior proof for the external contributor gate.

## What This Changes

The PR guards both exec-event heartbeat call sites for subagent and cron-run session keys, adds a subagent regression test, and adds a changelog entry.

## Best Possible Solution

Land a corrected producer-side subagent guard, keep cron behavior scoped to the existing delivery-mode policy or a separate cron-specific design, fix the regression assertion, and require redacted real setup proof before merge.

## Reproduction Assessment

Yes, at source level. Current main unconditionally requests exec-event heartbeats after exec system events, and forced subagent keys resolve back to the main session; I did not run a live WebChat/subagent scenario in this read-only review.

## Solution Assessment

No for the current patch. The subagent guard is the narrow maintainable direction, but the cron-run guard conflicts with existing cron delivery-mode behavior and the new test assertion does not match the current queued payload shape.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Keep cron-run exec wakes tied to delivery mode:** `src/agents/bash-tools.exec-runtime.ts:351`
  - body: The new guard suppresses every cron-run exec wake, but current cron code only disables exec notify-on-exit when `delivery.mode` is `none`, and tests assert announce and webhook cron runs keep automatic exec completion notifications. Keep this guard scoped to subagents or handle cron through an explicit cron delivery/remap path so legitimate cron completion events are not stranded.
  - confidence: 0.9
- **[P2] Match the queued system-event payload:** `src/agents/bash-tools.exec-runtime.test.ts:490-493`
  - body: `emitExecSystemEvent` always enqueues `trusted: false`, but the added exact `toHaveBeenCalledWith` expectation omits that field. Vitest compares the full object here, so the new regression test fails unless it includes `trusted: false` or uses a partial matcher.
  - confidence: 0.97
