## Summary

Keep open. The reported subagent heartbeat bug is source-reproducible on current main, but the current PR branch still has blocking test defects, over-suppresses cron-run exec wakes, and lacks after-fix real behavior proof.

## What This Changes

The PR guards exec-event heartbeat requests for subagent and cron-run session keys in the bash exec runtime, adds an emitExecSystemEvent regression test, and adds a changelog entry.

## Best Possible Solution

Land a corrected producer-side subagent guard, keep cron behavior scoped to the existing cron delivery policy or a cron-specific remap, add passing regression coverage, and require redacted real behavior proof before merge.

## Reproduction Assessment

Yes, at source level. Current main unconditionally requests an exec-event heartbeat after enqueueing exec system events, and forced subagent keys resolve back to the main session; I did not run a live WebChat/subagent scenario in this read-only review.

## Solution Assessment

No for the current patch. The subagent guard is the narrow maintainable direction, but the cron-run guard conflicts with current delivery-mode-specific cron behavior and the added test still fails against the current mock and payload shape.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Keep cron-run exec wakes intact:** `src/agents/bash-tools.exec-runtime.ts:351`
  - body: The new guard also suppresses cron-run session keys, but current cron code only disables exec notify-on-exit when `delivery.mode` is `none` and explicitly keeps automatic exec completion notifications for announce/webhook delivery. Dropping every cron-run wake here can strand legitimate cron completion events, so keep this guard scoped to subagents or handle cron in a cron-specific delivery/remap path.
  - confidence: 0.9
- **[P1] Use the current heartbeat mock:** `src/agents/bash-tools.exec-runtime.test.ts:494`

  - confidence: 0.99

  - body: `emitExecSystemEvent` always passes `trusted: false` to `enqueueSystemEvent`, but the new exact expectation omits that field. Once the mock name is fixed, this assertion still fails unless it matches the current payload or uses a partial matcher.
  - confidence: 0.97
