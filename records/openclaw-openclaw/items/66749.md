## Summary

Keep open. The subagent wake bug is source-reproducible and the producer-side guard is plausible, but the latest PR still suppresses cron-run wakes that current main intentionally remaps and tests, its new exact test assertion omits `trusted: false`, and it lacks contributor-run real behavior proof.

## What This Changes

The PR updates the bash exec runtime to suppress exec-event heartbeat requests for subagent and cron-run session keys, adds a subagent regression test, and adds a changelog entry.

## Best Possible Solution

Land a narrowed producer-side subagent guard, keep cron-run behavior tied to the existing delivery-mode policy or a separate cron design, fix the regression assertion, and require redacted real setup proof before merge.

## Reproduction Assessment

Yes, at source level. Current main requests exec-event heartbeats for forced subagent session keys, and heartbeat resolution routes those keys back to the main session; I did not run a live WebChat/subagent setup in this read-only review.

## Solution Assessment

No for the current patch. The subagent producer-side guard is the narrow maintainable direction, but the cron-run guard conflicts with current main’s explicit cron wake routing and delivery-mode tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Keep cron-run wakes tied to delivery mode:** `src/agents/bash-tools.exec-runtime.ts:350`
  - body: This guard drops every cron-run exec wake, but current main deliberately remaps cron-run events to the owning main/global queue and only suppresses automatic exec notifications when `delivery.mode` is `none`; announce and webhook modes still keep them. Narrow this to subagents or route cron through the existing delivery-mode policy so legitimate cron completion events are not stranded.
  - confidence: 0.92
- **[P2] Match the queued system-event payload:** `src/agents/bash-tools.exec-runtime.test.ts:524-527`
  - body: `emitExecSystemEvent` always enqueues `trusted: false`, but this exact `toHaveBeenCalledWith` expectation omits that field. The new regression test will fail unless it includes `trusted: false` or uses a partial matcher.
  - confidence: 0.98
