## Summary

Keep open: the subagent-side direction is still valid, but this patch also suppresses cron-run exec wakes against the current cron delivery-mode contract, and the new exact mock assertion omits the queued `trusted: false` payload field. The external PR also still lacks after-fix proof from a real setup.

## What This Changes

This PR guards exec-event heartbeat requests for subagent and cron-run session keys, adds a subagent regression test, and adds a changelog entry.

## Best Possible Solution

Land a corrected producer-side subagent guard with a matching payload assertion, keep cron behavior tied to the existing delivery-mode policy or a separate cron-specific design, and require redacted real setup proof before merge.

## Reproduction Assessment

Yes, at source level. Current main requests exec-event heartbeats after queueing exec system events, and forced subagent keys resolve back to the main session; I did not run a live WebChat or subagent scenario.

## Solution Assessment

No for this patch. The subagent guard is the narrow maintainable direction, but the cron-run guard conflicts with existing cron delivery-mode behavior and the new test assertion does not match the queued payload shape.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

- **[P1] Keep cron-run wakes scoped to delivery policy:** `src/agents/bash-tools.exec-runtime.ts:351`
  - body: This guard suppresses every cron-run exec wake, but current cron setup only disables automatic exec notifications when `delivery.mode` is `none`, and tests assert announce/webhook cron runs keep those notifications enabled.
