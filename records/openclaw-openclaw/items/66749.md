## Summary

Keep this PR open: the subagent wake suppression is a valid source-level bug fix direction, but the current branch also suppresses cron-run exec wakes in a way that conflicts with the existing cron delivery-mode contract, and the added regression test does not match the queued event payload.

## What This Changes

The PR guards exec-event heartbeat requests for subagent and cron-run session keys in the bash exec runtime, adds a subagent unit test, and records a changelog fix entry.

## Best Possible Solution

Land a narrowed producer-side subagent guard, keep cron-run behavior tied to the existing delivery-mode policy or a separate cron design, fix the regression assertion, and require redacted real setup proof before merge.

## Reproduction Assessment

Yes, at source level. Current main unconditionally wakes heartbeat after exec system events, and forced subagent session keys resolve back to the main session; I did not run a live WebChat/subagent setup in this read-only review.

## Solution Assessment

No for the current patch. The subagent guard is the narrow maintainable direction, but the cron-run guard conflicts with existing cron delivery behavior and the added test assertion does not match the payload contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

- **[P1] Keep cron-run exec wakes tied to delivery mode:** `src/agents/bash-tools.exec-runtime.ts:351`
  - body: This guard suppresses every cron-run exec wake, but the current cron path only disables exec notify-on-exit for `delivery.mode` `none`, and tests assert announce and webhook runs keep automatic exec completion notifications.
