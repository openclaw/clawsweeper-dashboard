## Summary

Keep open: current main and the latest release still leave `${XDG_CONFIG_HOME}` literal in the shared path resolver used by configured workspaces and ClawHub skill installs, while the linked fix PR is open cross-repository and not landed.

## What This Changes

The issue reports that skill installation writes under a literal `${XDG_CONFIG_HOME}` directory instead of expanding the environment variable into the intended workspace path.

## Best Possible Solution

Keep this issue as the canonical tracker until maintainers land one focused shared path resolver change with regression coverage for `${XDG_CONFIG_HOME}/workspace`, or explicitly document that only absolute workspace paths are supported.

## Reproduction Assessment

Yes, at source level. Current main sends configured workspace strings through `resolveUserPath`, while `resolveHomeRelativePath` only expands `~` before `path.resolve`, so `${XDG_CONFIG_HOME}/workspace` remains literal without needing a Docker repro in this read-only review.

## Solution Assessment

Unclear. The likely maintainable fix is shared env-placeholder expansion in the path resolver, but maintainers need to confirm the cross-surface behavior for unknown, empty, and sensitive env vars before a broad helper change lands.
