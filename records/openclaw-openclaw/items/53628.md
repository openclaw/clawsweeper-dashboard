## Summary

Keep open: current main and v2026.6.10 still leave `$VAR` and `${VAR}` placeholders literal in the shared user/workspace path flow used before ClawHub skill install targets are built, and the linked fix PR is still open and unmerged.

## What This Changes

The issue reports that a Docker skill install treats `${XDG_CONFIG_HOME}` as a literal directory name instead of expanding the configured environment variable.

## Best Possible Solution

Keep this issue as the canonical tracker until maintainers land one focused path-resolution fix with regression coverage, or explicitly document that env placeholders are unsupported in workspace/install paths.

## Reproduction Assessment

Yes, at source level. Current main sends `${XDG_CONFIG_HOME}/workspace` through resolver paths that only expand `~`, then skill install appends `skills/<slug>` under that unresolved path.

## Solution Assessment

Unclear. Shared env-placeholder expansion is likely the right boundary, but maintainers should choose the cross-surface semantics for unknown, empty, and sensitive variables before landing it broadly.
