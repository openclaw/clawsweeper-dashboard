## Summary

Keep this open: current `main` and the latest release still allow an environment-provided workspace path containing `${XDG_CONFIG_HOME}` to resolve as a literal directory, and the only active fix is an open cross-repository PR that needs maintainer direction.

## What This Changes

The issue reports that ClawHub skill installation treats `${XDG_CONFIG_HOME}` as a literal directory segment instead of expanding it from the Docker environment.

## Best Possible Solution

Adopt a narrow, tested path-resolution fix for environment-provided workspace/config path values without duplicating existing config-file substitution semantics or breaking literal-dollar paths.

## Reproduction Assessment

Yes from source inspection: an env-provided `OPENCLAW_WORKSPACE_DIR=${XDG_CONFIG_HOME}/workspace` reaches `path.resolve` unchanged and then skill installation appends `skills/<slug>`. I did not run a Docker repro in this read-only review, but the reporter supplied terminal evidence showing the literal directory was created.

## Solution Assessment

Unclear until maintainers decide the exact contract for env-provided path values. A shared resolver change is plausible, but it should be shaped around existing config env substitution rather than adding a second broad expansion path accidentally.
