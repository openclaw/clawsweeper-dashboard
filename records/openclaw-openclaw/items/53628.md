## Summary

Keep open: current main and the latest release still pass `${XDG_CONFIG_HOME}/workspace` through path resolution as a literal path, then install ClawHub skills under that unresolved workspace. The same-repository fix attempts are closed unmerged, and the open fork PR is not a landed OpenClaw fix.

## What This Changes

The issue requests that ClawHub/OpenClaw skill installs expand `${XDG_CONFIG_HOME}` in workspace paths instead of creating a literal `${XDG_CONFIG_HOME}` directory.

## Best Possible Solution

Land one maintainer-approved shared path-expansion contract for configured/user paths, cover both resolver helpers and ClawHub skill install targeting, then close this issue after the fix is on `openclaw/openclaw` main.

## Reproduction Assessment

Yes, at source level. Current main sends `OPENCLAW_WORKSPACE_DIR=${XDG_CONFIG_HOME}/workspace` through `path.resolve()` unchanged and then appends `skills/<slug>` for ClawHub installs.

## Solution Assessment

Unclear until maintainers choose the config expansion contract. The best technical layer is shared path resolution plus default workspace handling, not a ClawHub-only display or install patch.
