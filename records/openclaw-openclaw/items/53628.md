## Summary

Keep open. Current main still lets `${XDG_CONFIG_HOME}` remain literal in the shared user-path resolver before ClawHub skill install paths are built, and the remaining work is already covered by the open fix PR https://github.com/openclaw/openclaw/pull/53718.

## What This Changes

The issue reports that `${XDG_CONFIG_HOME}` in the configured skill workspace path is treated as a literal directory during ClawHub skill installation.

## Best Possible Solution

Land one shared resolver fix that expands supported `$VAR` and `${VAR}` placeholders before path resolution while preserving existing tilde expansion and unknown-placeholder behavior, then verify the external ClawHub CLI path if it is separate.

## Reproduction Assessment

Yes, by source inspection. A configured workspace string like `${XDG_CONFIG_HOME}/workspace` reaches `resolveUserPath` and then `path.resolve(trimmed)` without env substitution, and a direct Node `path.resolve` check preserved the literal placeholder.

## Solution Assessment

Yes. Fixing the shared path resolver is the narrowest maintainable solution because workspace resolution and ClawHub skill install targets already use that path helper; a one-off install-output patch would leave other path callers inconsistent.
