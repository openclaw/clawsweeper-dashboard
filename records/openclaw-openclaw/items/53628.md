## Label Justifications

- `P2`: This is a normal-priority path-resolution bug with limited blast radius but a clear affected Docker skill-install workflow.
- `impact:data-loss`: The bug can put user-installed skill files in a literal placeholder directory, making the intended workspace copy appear missing.

## Summary

Keep open: current main and the latest release still pass `${XDG_CONFIG_HOME}` through the shared path resolver as a literal path segment, so skill installs can still land under a literal placeholder directory. The remaining work is already represented by the open fork-side implementation at https://github.com/BingqingLyu/openclaw/pull/1726, while the target-repo fix attempts were closed unmerged.

## What This Changes

The issue reports that skill installation treats `${XDG_CONFIG_HOME}` literally in Docker instead of expanding it to the configured directory.

## Best Possible Solution

Land one focused shared resolver fix that expands environment placeholders before filesystem resolution, preserves deliberate passthrough for unresolved placeholders, covers both resolver helpers and the ClawHub skill-install path, then close this issue.

## Reproduction Assessment

Yes, at source level: current main sends `${XDG_CONFIG_HOME}/workspace` through the non-tilde `path.resolve(trimmed)` branch and then appends `skills/<slug>` under that unresolved workspace. I did not run the Docker install path because this review was read-only.

## Solution Assessment

Yes, the maintainable fix belongs in the shared path resolver used by workspace and skill-install paths, with tests for known, unknown, and empty env placeholders. Maintainers still need to pick or replace the open linked implementation path.
