## Summary

Keep open: current main and the latest release still do not expand `$VAR` or `${VAR}` placeholders in the shared user/workspace path resolver, so a workspace path containing `${XDG_CONFIG_HOME}` can still flow into the skill install target as a literal segment. There is already an open linked fork PR for this exact path-expansion fix, so this should not queue a duplicate repair until that PR is reviewed, replaced, or closed.

## What This Changes

The issue requests that skill installation expand `${XDG_CONFIG_HOME}` from the environment instead of creating a literal `${XDG_CONFIG_HOME}` directory.

## Best Possible Solution

Land or replace the linked path-expansion fix so canonical user/workspace path resolution expands `$VAR` and `${VAR}` before skill install computes `skills/<slug>`, with focused regression coverage for Docker-style workspace env overrides.

## Reproduction Assessment

Yes, at source level: set the workspace path to `${XDG_CONFIG_HOME}/workspace` while `XDG_CONFIG_HOME` is set, and current main resolves that string with `path.resolve` rather than env substitution before appending `skills/<slug>`. I did not run the live Docker ClawHub install during this read-only review.

## Solution Assessment

Yes: the maintainable fix belongs in shared user/workspace path resolution, not in ClawHub output formatting or a one-off skill installer branch. The linked PR's direction matches that, but it still needs target-repo review and coverage before the issue can close.
