## Summary

Keep open: current main and the latest release still preserve `${XDG_CONFIG_HOME}` as a literal path segment on the skill-install workspace path, so the central user problem is not implemented on main. The target-repo fix attempts are closed unmerged, while an open fork-side PR still references this issue and should be reviewed, replaced, or closed before queuing duplicate repair work.

## What This Changes

The issue reports that skill installs write into a literal `${XDG_CONFIG_HOME}` directory instead of expanding the environment variable to the configured Docker path.

## Best Possible Solution

Land a focused target-repo fix that expands `$VAR` and `${VAR}` in canonical user/workspace path resolution before skill installs compute their target directory, with Docker-style regression coverage for the workspace env override.

## Reproduction Assessment

Yes, at source level: set the workspace path to `${XDG_CONFIG_HOME}/workspace` while `XDG_CONFIG_HOME` is set, and current main resolves the workspace with `path.resolve` rather than env substitution before appending `skills/<slug>`. I did not run the Docker ClawHub install in this read-only review.

## Solution Assessment

Yes, the narrowest maintainable fix is to expand env placeholders in the shared path/workspace resolution path, including the direct `OPENCLAW_WORKSPACE_DIR` override. A partial resolver-only patch should be checked against the reported Docker env path before landing.
