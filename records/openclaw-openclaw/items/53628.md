## Summary

Keep open: current main still resolves configured `${XDG_CONFIG_HOME}` workspace paths as literal path segments before ClawHub builds the skill install target, and the remaining fix is already represented by open PR https://github.com/openclaw/openclaw/pull/53718.

## What This Changes

Expand environment placeholders in user path resolution so ClawHub skill installs do not write into a literal `${XDG_CONFIG_HOME}` directory.

## Best Possible Solution

Land PR https://github.com/openclaw/openclaw/pull/53718 or a maintainer-picked equivalent that expands supported `$VAR` and `${VAR}` placeholders in the shared resolver, preserves unresolved placeholders safely, verifies the real ClawHub install path, then close this issue.

## Reproduction Assessment

Yes. Source inspection shows `${XDG_CONFIG_HOME}/workspace` reaches `resolveHomeRelativePath` and then `path.resolve(trimmed)` without env substitution, and a direct Node probe preserved the literal placeholder; I did not run the full Docker install in this read-only review.

## Solution Assessment

Yes. Fixing the shared resolver is the narrowest maintainable path because workspace and ClawHub install paths already use it; a one-off ClawHub output or install patch would leave other configured paths inconsistent.
