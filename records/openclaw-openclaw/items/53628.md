## Summary

Keep open: current main and the latest release still resolve `OPENCLAW_WORKSPACE_DIR=${XDG_CONFIG_HOME}/workspace` as a literal relative path, which matches the reported ClawHub skill install target under `/app/${XDG_CONFIG_HOME}/workspace/skills/...`. The fix shape is clear, but expanding arbitrary `$VAR` placeholders across user path resolution is a config-contract decision, and the same-repo fix attempts are closed unmerged.

## What This Changes

The issue requests that skill installation expand `${XDG_CONFIG_HOME}` instead of treating it as a literal directory segment when resolving the workspace skills path.

## Best Possible Solution

Adopt one canonical path-expansion contract for configured/user paths, apply it to `OPENCLAW_WORKSPACE_DIR` and shared user-path helpers, and add focused regression coverage proving ClawHub installs target `${XDG_CONFIG_HOME}/workspace/skills/<slug>` after expansion.

## Reproduction Assessment

Yes. Source inspection shows `OPENCLAW_WORKSPACE_DIR=${XDG_CONFIG_HOME}/workspace` reaches `path.resolve()` unchanged and ClawHub install then appends `skills/<slug>`; the reporter’s terminal evidence matches that literal directory path.

## Solution Assessment

Unclear until maintainers choose the expansion contract. The best technical layer is shared path resolution plus the default workspace env override, not a ClawHub-only display or install patch.
