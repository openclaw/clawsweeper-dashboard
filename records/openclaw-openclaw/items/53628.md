## Label Justifications

- `P2`: This is a valid configured-path bug that can install skills into the wrong directory, but it has limited blast radius and is not an outage or security incident.
- `impact:data-loss`: The issue can write installed skill files into an unexpected literal directory, making the skill unavailable from the configured workspace.

## Summary

Keep open: current main still does not expand `${XDG_CONFIG_HOME}`-style placeholders in the shared path flow that can feed ClawHub skill installs, and the exact remaining fix is already represented by the open implementation PR https://github.com/openclaw/openclaw/pull/53718.

## What This Changes

The issue reports that ClawHub skill installation treats `${XDG_CONFIG_HOME}` as a literal path segment instead of expanding it from the Docker/runtime environment.

## Best Possible Solution

Land one refreshed shared resolver fix that expands supported env placeholders before filesystem resolution, preserves deliberate passthrough for unresolved placeholders, proves the ClawHub install path in Docker or an equivalent real setup, then close this issue.

## Reproduction Assessment

Yes, at source level. Current main sends `${XDG_CONFIG_HOME}/workspace` through the resolver to `path.resolve(trimmed)` without env substitution, and ClawHub appends `skills/<slug>` under that unresolved workspace path.

## Solution Assessment

Yes. Fixing the shared path resolver is the narrow maintainable solution because workspace resolution and ClawHub installs already flow through it; a ClawHub-only patch would leave other configured path callers inconsistent.
