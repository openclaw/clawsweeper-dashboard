## Summary

Keep open: current main still preserves `${XDG_CONFIG_HOME}` as a literal path segment in the shared user-path resolver, and the exact remaining fix is already represented by open PR https://github.com/openclaw/openclaw/pull/53718.

## What This Changes

The issue reports that ClawHub skill installation treats `${XDG_CONFIG_HOME}` as a literal directory segment instead of expanding the environment variable.

## Best Possible Solution

Land one maintainer-approved shared resolver fix that expands supported env placeholders before filesystem resolution, preserves safe passthrough for unresolved placeholders, covers the ClawHub install path, and then close this issue after the PR merges.

## Reproduction Assessment

Yes, at source level. Current main passes `${XDG_CONFIG_HOME}/workspace` through `resolveHomeRelativePath` to `path.resolve(trimmed)` without substitution, then builds the ClawHub skill target from that workspace path; I did not run the full Docker install flow in this read-only review.

## Solution Assessment

Yes. Fixing the shared resolver is the narrowest maintainable solution because workspace and ClawHub install paths already use it; a ClawHub-only display or install patch would leave other configured paths inconsistent.
