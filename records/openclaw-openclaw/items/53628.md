## Summary

Keep open: current main and latest v2026.5.18 still leave `${XDG_CONFIG_HOME}` as a literal path segment in user/workspace path resolution, so the reported ClawHub install bug remains valid. Open PR https://github.com/openclaw/openclaw/pull/53718 is the canonical fix path and still needs refresh/proof before this issue can close.

## What This Changes

Expand `${XDG_CONFIG_HOME}` and similar env placeholders in ClawHub skill install workspace paths instead of treating them as literal directories.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/53718 or a maintainer-picked equivalent that expands supported env placeholders in the shared resolver, preserves unresolved-placeholder safety, proves a real ClawHub/Docker install path, then closes this issue.

## Reproduction Assessment

Yes, source-reproducible with high confidence. Current main sends `${XDG_CONFIG_HOME}/workspace` through `resolveHomeRelativePath` to `path.resolve(trimmed)` without substitution, and the reporter confirmed a literal `${XDG_CONFIG_HOME}` directory was created.

## Solution Assessment

Yes. Fixing the shared resolver is the narrowest maintainable solution because workspace, config, and ClawHub install paths already flow through it; a one-off ClawHub display or install patch would leave other configured paths inconsistent.
