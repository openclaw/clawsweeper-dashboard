## Summary

Keep open: current main and the latest release still resolve `${XDG_CONFIG_HOME}/workspace` as a literal user path, so the reported skill-install destination bug remains. The same-repository fix attempts are closed unmerged, and the open cross-repository closing PR is still not a landed OpenClaw fix.

## What This Changes

The issue asks OpenClaw skill installs to expand `${XDG_CONFIG_HOME}` in configured workspace paths instead of creating a literal `${XDG_CONFIG_HOME}` directory.

## Best Possible Solution

Land a maintainer-approved shared path resolver change with regression coverage for `${XDG_CONFIG_HOME}/workspace`; if env placeholders are not supported, document the absolute-path or `OPENCLAW_WORKSPACE_DIR` requirement instead.

## Reproduction Assessment

Yes. Source inspection shows configured workspaces flow through `resolveUserPath`, while `resolveHomeRelativePath` only expands `~` and then calls `path.resolve` on the raw `${XDG_CONFIG_HOME}` string; I did not run the full Docker install path.

## Solution Assessment

Unclear. The likely repair is shared env-placeholder expansion in user path resolution, but maintainers need to accept the compatibility and security semantics because the helper is used beyond skill installs.
