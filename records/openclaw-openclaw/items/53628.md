## Summary

Keep open: current main and the latest release still resolve `${XDG_CONFIG_HOME}/workspace` as a literal path, and ClawHub skill installs then write under that unresolved workspace. Same-repository fix attempts are closed unmerged, while the open fork PR is not a landed OpenClaw fix.

## What This Changes

The issue requests environment-placeholder expansion for skill/workspace path resolution so a ClawHub skill install uses the real `XDG_CONFIG_HOME` directory instead of creating a literal `${XDG_CONFIG_HOME}` folder.

## Best Possible Solution

Land one maintainer-approved shared path-expansion contract for configured/user paths, cover both resolver helpers and ClawHub skill install targeting, then close this issue after the fix reaches `openclaw/openclaw` main.

## Reproduction Assessment

Yes, at source level. Current main sends `${XDG_CONFIG_HOME}/workspace` through `path.resolve()` unchanged and then appends `skills/<slug>` for ClawHub installs; I did not run a live Docker install in this read-only review.

## Solution Assessment

Unclear until maintainers choose the config expansion contract. The best technical layer is shared path resolution plus default workspace handling, not a ClawHub-only display or install patch.
