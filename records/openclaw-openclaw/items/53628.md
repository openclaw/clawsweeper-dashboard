Keep open: current main and the latest release still do not expand `$VAR` or `${VAR}` in the shared user-path/default-workspace path resolution used before skill install targets are built, and the linked closing PR remains open/unmerged.

## What This Changes

The issue asks OpenClaw/ClawHub skill installation to expand `${XDG_CONFIG_HOME}` instead of creating a literal `${XDG_CONFIG_HOME}` workspace directory.

## Best Possible Solution

Keep this issue as the canonical tracker until maintainers land one focused path-resolution fix with regression coverage, or explicitly document that env placeholders are unsupported in workspace/install paths.

## Reproduction Assessment

Yes, at source level. Current main keeps `$VAR` and `${VAR}` literal in the shared path/default-workspace resolver paths before skill install targets are built; I did not run a Docker write-path repro because this review is read-only.

Unclear. Shared env-placeholder expansion is the likely maintainable fix, but maintainers should choose the cross-surface behavior for missing, empty, and sensitive variables before landing it broadly.
