## Summary

Keep open. Current main still lacks `announceTimeoutMs` in the sub-agent docs, but this PR’s wording is too broad because the configured value is applied to each gateway announce call while transient failures can retry before fallback or failure.

## What This Changes

This PR adds `agents.defaults.subagents.announceTimeoutMs` to `docs/tools/subagents.md` in the config reference and nested sub-agent example.

## Best Possible Solution

Keep the docs-only scope, but describe `announceTimeoutMs` as the per-call gateway announce timeout and mention that transient retries can extend total wall-clock time before fallback or failure.

## Reproduction Assessment

Yes. Compare current `docs/tools/subagents.md` with the schema/runtime, then inspect the existing timeout test that expects four direct `agent` calls after repeated gateway timeout errors.

## Solution Assessment

No, not as written. A docs-only update is the right solution, but the wording should avoid implying that one configured value bounds the entire announce step.
