## Summary

Keep open. Current `main` still has a real sub-agent docs gap: `announceTimeoutMs` is accepted by the schema and used by announce-delivery runtime/tests, but it is absent from `docs/tools/subagents.md`. The PR is correctly scoped to docs, but its new sentence is inaccurate because the value is applied to each direct gateway announce call inside retry handling, not as a total wall-clock limit for the whole announce step.

## What This Changes

This PR adds `agents.defaults.subagents.announceTimeoutMs` to the sub-agents documentation config reference and nested sub-agent configuration example in `docs/tools/subagents.md`.

## Best Possible Solution

Keep the narrow documentation addition, but word `announceTimeoutMs` as the per-call gateway timeout for direct child-session announce delivery attempts and mention that transient retries can extend total wall-clock time before fallback or final failure.

## Reproduction Assessment

Yes. The docs drift is reproducible by comparing `docs/tools/subagents.md` with the schema/runtime option, and the wording bug is reproducible by inspecting the existing timeout test that expects four direct announce attempts after repeated gateway timeouts.

## Solution Assessment

No, not as written. A docs-only change in `docs/tools/subagents.md` is the right scope, but the new wording should describe per-attempt gateway timeout semantics rather than a maximum total time for the whole announce step.
