## Summary

Current main already implements the intended `web_fetch` charset fix and does so more completely than this PR branch: it decodes raw bytes, honors declared HTTP/XML/HTML meta charsets, keeps bounded reads, and includes regressions for legacy charset decoding plus the unrelated-meta false-positive case. The implementation is present in the shipped `v2026.4.29` tag, so this PR is now obsolete rather than the canonical change to merge.

## What This Changes

The PR branch changes `web_fetch` response reading to decode raw bytes with charset detection from `Content-Type` or a bounded HTML meta scan, and adds legacy charset regression tests.

## Best Possible Solution

Keep the shipped `v2026.4.29` implementation as the canonical fix and close this PR as already implemented on main; future charset changes should build on the current attribute parser and existing regression tests.

## Reproduction Assessment

Yes. The linked bug has a high-confidence mocked reproduction path in `src/agents/tools/web-tools.fetch.test.ts` for legacy byte sequences that previously produced replacement characters, including HTTP charset and HTML meta charset cases.

## Solution Assessment

Yes for current main, no for reviving this PR branch. The shipped implementation is the narrower maintainable end state because it centralizes byte decoding in `readResponseText`, preserves bounded reads, and avoids the PR branch's heuristic meta false positive.

## Review Findings

Full review comments:

- **[P2] Parse meta attributes before trusting charset text:** `src/agents/tools/web-shared.ts:118`
  - body: The new regex accepts any `charset=` token inside a `<meta>` tag, so a UTF-8 page with `<meta name="description" content="charset=Shift_JIS">` can be decoded as Shift_JIS and return mojibake. Current main avoids this by reading the actual `charset` attribute and only honoring `content` when `http-equiv=content-type`.
