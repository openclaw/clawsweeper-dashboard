## Label Justifications

## Summary

This PR remains necessary: current main still treats socket-open as the WhatsApp login success boundary, while the submitted patch correctly adds the missing queued-persistence and stable disk-auth check. The implementation is focused, existing CLI and QR callers clean up failed login sockets, and no actionable correctness or security defect was found; the linked issue should remain open until this PR lands.

## What This Changes

The PR requires WhatsApp credentials to be stably linked on disk after socket connection before reporting login success and adds regressions for delayed, unstable, or missing persistence.

## Best Possible Solution

Land the canonical post-connection durability gate using the existing queue-backed auth decision, preserving retryable failures and existing socket cleanup without adding storage formats, migrations, configuration, or a second persistence path.

## Reproduction Assessment

Yes, from source with high confidence. Current main reports success after socket-open even though credential saving is separately queued and bounded, which establishes the reported false-success race; I did not run a live WhatsApp QR pairing.

## Solution Assessment

Yes. Reusing the existing queue-backed stable disk-auth decision at the missing success boundary is narrower and more maintainable than adding another flush mechanism, and both callers already clean up failure sockets.
