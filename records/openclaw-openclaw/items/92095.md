## Label Justifications

- `P1`: The regression can make a successfully linked WhatsApp channel lose authentication and message delivery after a Docker rebuild.
- `merge-risk: 🚨 availability`: The patch intentionally converts credential persistence exceeding the bounded wait into a retryable login failure.

## Summary

This is a focused, architecturally correct fix for the linked P1 regression. Baileys emits the relevant credential update synchronously before connection-open, OpenClaw immediately enqueues that write, and the new shared login gate drains the queue and confirms linked auth on disk before either CLI or QR login reports success.

## What This Changes

The PR makes WhatsApp CLI and QR login wait for stable, persisted linked credentials after socket connection and adds regression coverage for delayed, missing, and unstable auth persistence.

## Best Possible Solution

Keep the shared controller-level durability gate, using the existing bounded credential queue and disk-backed auth decision so all WhatsApp login surfaces report success only for restart-reusable credentials.

## Reproduction Assessment

Yes. Current main resolves the shared login waiter on socket-open alone, while credential persistence is a separately queued event and the documented successful state is stored on disk.

## Solution Assessment

Yes. The controller is the narrow shared boundary for CLI and QR login, and reusing the existing auth decision avoids introducing a duplicate persistence mechanism or new configuration surface.
