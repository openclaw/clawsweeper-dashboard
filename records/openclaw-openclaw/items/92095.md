## Summary

The patch is a focused and source-backed fix with no actionable correctness finding, but this external auth-flow PR should remain open until it has after-fix proof from a real WhatsApp pairing and restart or container-rebuild path; the supplied local runtime output simulates socket-open and does not satisfy OpenClaw's live external-API proof requirement.

## What This Changes

The PR makes WhatsApp login verify stable linked credentials through the existing queue-backed disk-auth decision before reporting success, with regression coverage for delayed, unstable, and missing persistence.

## Best Possible Solution

Keep the queue-backed durable-auth gate at the shared login-success boundary, then verify a real QR pairing remains authenticated after restarting the process or rebuilding a container with the same mounted state directory.

## Reproduction Assessment

Yes at source level: current main resolves login success from socket-open while the credentials are persisted asynchronously through a separate queue. This review did not perform a live WhatsApp QR pairing and Docker-rebuild reproduction.

## Solution Assessment

Yes; placing the existing queue-backed disk-auth decision at the shared login-success boundary is narrower and more maintainable than adding another persistence mechanism or changing credential storage. The pinned Baileys source confirms that the relevant credential update is emitted before connection-open.
