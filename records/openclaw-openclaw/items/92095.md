## Summary

This PR is still necessary: current main and the latest stable release still report WhatsApp login success at socket-open, while this branch moves the shared CLI/QR success boundary to the existing durable auth decision helper and adds focused regression coverage. It should stay open for maintainer review because there is an intentional retry/availability tradeoff and another open fix PR for the same linked bug.

## What This Changes

The branch makes WhatsApp login wait for persisted auth state after socket-open and adds CLI, QR, and delayed credential-save regression tests.

## Best Possible Solution

Land one canonical durable-auth fix for the linked WhatsApp issue, preferably this branch if maintainers accept retrying on unstable auth, then close or supersede the duplicate PR path.

## Solution Assessment

Yes, this is the best observed fix shape. It puts the durability check in the shared CLI/QR login waiter and reuses the existing auth decision helper instead of adding a parallel persistence reader.
