## Summary

Keep open. The PR targets a useful exec-approval UX improvement, but the current head hard-codes an `allow-always` fallback row that conflicts with the current `ask=always` policy, and it still lacks contributor-provided real chat proof.

## What This Changes

The PR changes exec approval forwarded chat fallback text to show pre-filled `/approve <id> <decision>` rows and adds tests for Markdown versus non-Markdown channel formatting.

## Best Possible Solution

Generate fallback command rows from the shared allowed-decision/action descriptor source, keep channel-aware formatting, and merge only after redacted real chat proof demonstrates the improved approval flow.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not as a current-main bug; this is a UX feature request, but the branch defect is source-reproducible from the hard-coded fallback rows versus the current `ask=always` allowed-decision contract.

## Solution Assessment

Is this the best way to solve the issue? No, not as currently patched; the visible rows should be generated from `resolveExecApprovalRequestAllowedDecisions` or `buildExecApprovalActionDescriptors` so notification copy cannot drift from gateway policy.
