## Summary

Keep open for maintainer PR review: the UX goal is plausible, but the current branch is not merge-ready because the fallback command rows are hard-coded instead of policy-derived, real chat proof is missing, and the live PR is merge-conflicting.

## What This Changes

The PR changes exec approval chat notifications to show copyable decision-specific `/approve` command rows and adds focused forwarder tests.

## Best Possible Solution

Generate visible fallback command rows from the shared allowed-decision descriptors, preserve channel-aware formatting, update focused tests, and merge only after redacted real chat proof shows the improved approval flow.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not as a current-main bug because this is a UX enhancement, but the PR defect is source-reproducible: the added fallback rows are hard-coded while `ask=always` only allows `allow-once` and `deny`.

## Solution Assessment

Is this the best way to solve the issue? No; the branch should derive fallback rows from `resolveExecApprovalRequestAllowedDecisions` or `buildExecApprovalActionDescriptors` so notification copy cannot drift from gateway policy.
