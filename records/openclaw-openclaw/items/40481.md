## Summary

Keep open. This is a useful UX enhancement, but the current PR still offers a disallowed `allow-always` fallback command for `ask=always` approvals and lacks contributor-provided real chat proof for the claimed Telegram/mobile flow.

## What This Changes

This PR changes exec approval forwarded chat notifications to show three copyable `/approve` command rows, with Markdown wrapping only for Markdown-capable channels, and updates forwarder tests.

## Best Possible Solution

Generate visible fallback command rows from the shared allowed-decision descriptors, preserve channel-aware formatting, update focused tests, and merge only after redacted real chat proof shows the improved approval flow.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not as a current-main bug because this is a UX enhancement, but the PR defect is source-reproducible: the added fallback rows are hard-coded while `ask=always` only allows `allow-once` and `deny`.

## Solution Assessment

Is this the best way to solve the issue? No; the fallback rows should be derived from `resolveExecApprovalRequestAllowedDecisions` or `buildExecApprovalActionDescriptors` so notification copy cannot drift from gateway policy.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Derive fallback commands from allowed decisions:** `src/infra/exec-approval-forwarder.ts:255-257`
  - body: The fallback text always includes `allow-always`, but `ask=always` requests resolve to only `allow-once` and `deny`, and the gateway rejects decisions outside that allowed set. Build these rows from `resolveExecApprovalRequestAllowedDecisions` or `buildExecApprovalActionDescriptors` so forwarded notifications cannot offer a failing command.
  - confidence: 0.92
