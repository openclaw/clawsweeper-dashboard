## Summary

Keep open: current main still uses policy-aware generic fallback text, while this PR changes the fallback toward the requested UX but currently advertises a disallowed `allow-always` command for `ask=always` and lacks real chat-flow proof.

## What This Changes

The PR changes exec approval forwarded chat notifications to show three pre-filled `/approve` command rows with channel-aware Markdown wrapping and updates forwarder tests.

## Best Possible Solution

Generate visible fallback command rows from shared allowed-decision action descriptors, keep channel-aware formatting, and merge only after redacted real chat proof shows the improved approval flow.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not as a current-main bug; this is a UX feature request, but the PR defect is source-reproducible because the added rows are hard-coded while `ask=always` only allows `allow-once` and `deny`.

## Solution Assessment

Is this the best way to solve the issue? No, not as patched; the maintainable fix is to derive the visible command rows from `resolveExecApprovalRequestAllowedDecisions` or `buildExecApprovalActionDescriptors` so notification copy cannot drift from gateway policy.
