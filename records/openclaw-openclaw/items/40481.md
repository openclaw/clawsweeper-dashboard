## Summary

Keep open: the PR still targets a real generic approval-notification UX gap on current main, but the branch is not mergeable as-is because it hard-codes an unavailable `allow-always` command for `ask=always` requests and lacks contributor-supplied real behavior proof.

## What This Changes

The PR changes exec approval forwarding fallback text to show ready-to-copy `/approve <id> ...` command lines and adds markdown-aware fallback rendering coverage.

## Best Possible Solution

Generate fallback command lines from the shared approval action descriptors or resolved allowed decisions, preserve non-markdown copy behavior, cover markdown/plain-channel and `ask=always` cases, then attach redacted real Telegram proof.

## Reproduction Assessment

Not applicable as a UX feature PR rather than a bug report. Source inspection confirms current main still renders the generic fallback hint, while the PR branch changes that text.

## Solution Assessment

No, not as-is. The maintainable approach is to derive ready-to-copy commands from the same allowed-decision contract used by interactive approval actions instead of hard-coding every decision.
