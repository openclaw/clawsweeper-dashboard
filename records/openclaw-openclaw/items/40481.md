## Summary

Keep open: the UX change is not on current main, but this PR cannot merge as-is because it hard-codes an unavailable `allow-always` command for `ask=always`, lacks inspectable real chat proof, and the live branch is merge-conflicting.

## What This Changes

The PR replaces the generic exec-approval reply hint with copyable `/approve <id> <decision>` command rows in forwarded chat notifications.

## Best Possible Solution

Generate fallback command rows from the shared allowed-decision descriptors, keep Markdown wrapping channel-aware, update the `ask=always` regression coverage, and provide redacted real Telegram or equivalent chat proof.

## Reproduction Assessment

Not applicable as a current-main bug reproduction; this is a UX feature PR. The PR defect is source-reproducible because the branch hard-codes `allow-always` while current main excludes that decision for `ask=always`.

## Solution Assessment

No, not as-is. The maintainable solution is to derive the displayed command rows from `resolveExecApprovalRequestAllowedDecisions` or `buildExecApprovalActionDescriptors` instead of hard-coding all decisions.
