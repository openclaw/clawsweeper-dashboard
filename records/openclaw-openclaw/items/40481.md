## Summary

Keep open: the UX request is still valid, but this PR’s current fallback text hard-codes an `allow-always` command that current policy can reject, and the external PR still lacks real after-fix chat proof.

## What This Changes

The branch changes exec approval forwarded chat notifications from a generic `/approve` hint to three pre-filled command rows, with Markdown wrapping only on Markdown-capable channels and updated forwarder tests.

## Best Possible Solution

Generate visible fallback command rows from the shared allowed-decision descriptors, preserve channel-aware formatting, and merge only after redacted real chat proof shows the improved approval flow.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not as a current-main bug; this is a UX enhancement, but the PR defect is source-reproducible because the added rows are hard-coded while `ask=always` only allows `allow-once` and `deny`.

## Solution Assessment

Is this the best way to solve the issue? No; the fallback rows should be derived from `resolveExecApprovalRequestAllowedDecisions` or `buildExecApprovalActionDescriptors` so notification copy cannot drift from gateway policy.

## Review Findings
