## Summary

Keep open: the requested UX change is still not fully implemented in the generic forwarded fallback on current main, but this branch cannot merge as-is because it hard-codes `allow-always` where the approval policy can remove that decision and it still lacks real behavior proof.

## What This Changes

The PR changes exec approval forwarded notification fallback text to show full `/approve` commands for each decision, with markdown-aware formatting and focused tests.

## Best Possible Solution

Generate ready-to-copy fallback command lines from the shared approval action descriptors or resolved allowed decisions, adapt the change to current `buildExecApprovalRequestMessage`, and add redacted real Telegram/chat proof for the visible notification flow.

## Reproduction Assessment

Not applicable as a UX feature PR. Source inspection confirms current main still renders generic fallback text, while the branch changes that fallback text.

## Solution Assessment

No, not as-is. The maintainable fix is to derive the command lines from the same allowed-decision contract used by interactive approval buttons so `ask=always` does not offer `allow-always`.
