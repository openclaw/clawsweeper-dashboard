## Summary

This PR should stay open because the requested copyable `/approve` notification UX is not yet on current main and the current branch still needs a policy-aware revision plus contributor real behavior proof before merge. The prior ClawSweeper review finding remains valid: the branch offers `allow-always` even for `ask=always` requests where that decision is not allowed.

## What This Changes

The PR changes exec approval forwarded chat notifications to show pre-filled `/approve <id> <decision>` command lines, with Markdown wrapping only on Markdown-capable channels, and adds focused forwarder tests.

## Best Possible Solution

Generate visible fallback command rows from the shared allowed-decision/action descriptor source, preserve Markdown-aware formatting per channel, and merge only after redacted real chat proof shows the improved approval flow.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not as a current-main bug; this is a UX feature request, but the branch defect is source-reproducible because hard-coded fallback rows conflict with the current `ask=always` allowed-decision contract.

## Solution Assessment

Is this the best way to solve the issue? No, not as currently patched; the rows should be derived from `resolveExecApprovalRequestAllowedDecisions` or `buildExecApprovalActionDescriptors` so notification copy cannot drift from gateway policy.
