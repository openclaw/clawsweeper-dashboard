## Summary

Keep open: the PR is aimed at a real UX problem and already addresses the earlier non-markdown backtick concern, but the current branch still advertises `allow-always` unconditionally even when current policy and resolver code reject it for `ask=always`; it also lacks the required changelog entry.

## What This Changes

The PR changes exec-approval forwarded chat notification text and tests to show ready-to-copy `/approve <id> <decision>` commands, with backtick wrapping skipped for non-markdown channels.

## Best Possible Solution

Update this PR or a narrow replacement branch so fallback approval command lines are generated from request-specific allowed-decision descriptors, preserving markdown-aware wrapping and coverage for default, non-markdown, and `ask=always` cases.

## Reproduction Assessment

Yes. Static inspection provides a high-confidence reproduction path: an `ask: "always"` forwarded approval must omit `allow-always`, and current tests plus resolver code show that advertising it would produce a rejected command.

## Solution Assessment

No. Ready-to-copy commands are the right UX direction, but hard-coding all three decisions is not the narrow maintainable solution; the text should be derived from the same allowed-decision model used by interactive actions and resolver validation.
