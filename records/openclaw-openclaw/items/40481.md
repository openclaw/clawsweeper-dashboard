## Summary

Keep this PR open: the UX goal is still plausible, but the current branch advertises an invalid `allow-always` command for `ask=always` requests and still lacks contributor-provided real chat proof.

## What This Changes

The PR changes generic exec approval forwarding fallback text/tests to emit full `/approve <id> <decision>` rows with Markdown wrapping only on Markdown-capable channels.

## Best Possible Solution

Generate fallback command rows from the shared allowed-decision descriptors, keep Markdown wrapping channel-aware, and verify the copy/paste approval flow with redacted real chat proof.

## Reproduction Assessment

Not applicable as a current-main bug reproduction; this is a UX feature PR. Yes for the PR defect: source inspection shows the branch hard-codes `allow-always` even though `ask=always` allows only `allow-once` and `deny`.

## Solution Assessment

No, not as-is. The maintainable path is to derive command rows from `resolveExecApprovalRequestAllowedDecisions` or `buildExecApprovalActionDescriptors` and then require real chat proof before merge.
