## Summary

Keep open: this is a focused UX improvement, but the PR is not correct as-is because it hard-codes an unavailable decision in some current approval-policy cases and still lacks contributor-supplied real behavior proof for the chat approval flow.

## What This Changes

The PR changes exec approval forwarding fallback text and tests to show full `/approve <id> ...` commands, with Markdown backticks only on Markdown-capable channels.

## Best Possible Solution

Generate the fallback command lines from the shared action descriptors or resolved allowed-decision set, preserve current interactive payload behavior, and attach redacted real Telegram or chat proof for the approval flow.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a UX feature PR. Source comparison gives a high-confidence baseline: current main still has generic fallback text, while current approval contracts already make decisions policy-aware.

## Solution Assessment

No, not as-is. The direction is narrow and useful, but the implementation should derive visible command text from the shared allowed-decision/action contract so it never advertises an unavailable decision.
