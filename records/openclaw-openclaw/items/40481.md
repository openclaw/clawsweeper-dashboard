## Summary

Keep this PR open: the UX direction is useful, but the current patch is not mergeable because it advertises an `allow-always` command in fallback text even when the effective exec approval policy disallows that decision.

## What This Changes

The PR changes exec-approval forwarded chat notification text and tests to show ready-to-copy `/approve <id> <decision>` commands, with backtick wrapping skipped for non-markdown channels.

## Best Possible Solution

Land a narrow version that renders fallback command suggestions from the same allowed-decision model used by interactive approvals, preserves channel-appropriate wrapping, and records the user-facing change in the changelog.

## Reproduction Assessment

Yes. Static inspection gives a high-confidence path: an `ask: "always"` forwarded approval would advertise `allow-always` in the PR text, while current resolver policy rejects that decision.

## Solution Assessment

No. Ready-to-copy commands are the right UX direction, but hard-coding all three decisions is not the maintainable fix; the text should be generated from the existing allowed-decision descriptors.
