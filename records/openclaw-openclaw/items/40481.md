## Summary

Keep open. The PR addresses a real UX gap that current main still has, but the branch is not merge-ready because its fallback text unconditionally advertises `allow-always` even for requests where current policy filters that decision out and the resolver rejects it.

## What This Changes

The PR changes exec-approval forwarded chat notification text and tests to show ready-to-copy `/approve <id> <decision>` commands, with backtick wrapping skipped for non-markdown channels.

## Best Possible Solution

Update this PR or a narrow replacement branch so fallback approval command lines are generated from the request-specific allowed-decision descriptors, preserving markdown-aware wrapping, the `ask=always` unavailable explanation, and focused coverage for default, non-markdown, and `ask=always` cases.

## Reproduction Assessment

Yes. Static inspection and existing tests reproduce the defect: an `ask=always` request must omit `allow-always`, while the PR diff advertises it unconditionally and current resolver code rejects it.

## Solution Assessment

No. Pre-filled commands are the right UX direction, but hard-coding all three decisions is not the narrow maintainable fix; the command list should be derived from `resolveExecApprovalRequestAllowedDecisions` or `buildExecApprovalActionDescriptors`, with a changelog entry for the user-facing copy change.
