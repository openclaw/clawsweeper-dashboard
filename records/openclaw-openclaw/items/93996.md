## Summary

Keep this PR open: it targets the real WhatsApp group-directory bug tracked by https://github.com/openclaw/openclaw/issues/76722, but the current patch still depends on an in-process WhatsApp controller that the standalone directory CLI does not create, and the PR supplies only mocked test output rather than live linked-account proof.

## What This Changes

The branch registers a WhatsApp `listGroupsLive` handler, implements it via Baileys `groupFetchAllParticipating()` from the registered socket with config fallback, adds mocked WhatsApp directory tests, and adjusts one voice-call test mock type.

## Best Possible Solution

Implement the WhatsApp live group directory in a plugin-owned path that works for the standalone directory CLI against persisted linked auth, keep a safe config fallback when live access is unavailable, and require redacted live proof before merge.

## Reproduction Assessment

Yes, source-level. Current main only exposes config-backed WhatsApp group listing, and PR head's live method falls back to that same path whenever the standalone CLI process has no registered WhatsApp controller.

## Solution Assessment

No. Calling Baileys from the WhatsApp plugin is the right owner boundary, but the current process-local registry lookup is not the best fix for the standalone directory CLI path.
