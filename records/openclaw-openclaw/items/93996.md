## Summary

Keep open: this PR targets a real WhatsApp group-discovery bug, but it is not merge-ready because the new live lookup uses a process-local registered controller that the reported standalone directory CLI path is not shown to have, so it can still fall back to the same empty config-backed result. The PR also still lacks redacted live linked-account proof; the supplied evidence is mocked unit/type/lint output only.

## What This Changes

The branch registers a WhatsApp `listGroupsLive` directory handler, implements it via Baileys `groupFetchAllParticipating()` from the registered WhatsApp connection controller with config fallback, adds mocked WhatsApp directory tests, and adjusts one voice-call test mock type.

## Best Possible Solution

Implement WhatsApp live group listing in a plugin-owned path that works from the standalone directory CLI against persisted linked auth or another maintainer-approved live runtime, keep a safe config fallback, and require redacted live proof before merge.

## Reproduction Assessment

Yes, source-level. Current main only exposes config-backed WhatsApp group listing, and PR head's live method falls back to that same path whenever the standalone CLI process has no registered WhatsApp controller.

## Solution Assessment

No. Calling Baileys from the WhatsApp plugin is the right owner boundary, but the current process-local registry lookup is not the best fix for the standalone directory CLI path.
