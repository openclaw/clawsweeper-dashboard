## Summary

Keep open: current main still allows a create proposal whose target skill was created outside the workshop to remain pending, and the Control UI can keep showing it as waiting after a failed lifecycle call instead of refreshing status.

## What This Changes

Reconcile Skill Workshop proposals with already-active target skills so the UI does not keep offering a pending apply path for a skill that is already installed.

## Best Possible Solution

Add service-level stale reconciliation for pending create proposals whose target skill or support files now exist, and refresh the Control UI after lifecycle failures that may have changed backend state.

## Reproduction Assessment

Yes, source-reproducible: create a pending Skill Workshop create proposal, create the target `SKILL.md` manually, then `listSkillProposals` still reports `pending` and `applySkillProposal` throws before marking it stale. I did not run a live temp repro because this was a read-only review.

## Solution Assessment

Yes, the best fix is a narrow service/UI repair rather than a new config option: make target-created create proposals resolve to `stale` consistently and make the UI refresh after failed lifecycle calls.

## Repair Work Prompt

Fix the Skill Workshop stale-proposal state drift reported at https://github.com/openclaw/openclaw/issues/90388. In current main, a pending create proposal whose target `skills/<key>/SKILL.md` is created manually can remain `pending`; `applySkillProposal` then throws `Target skill already exists` without marking the proposal `stale`, and the Control UI only refreshes after successful lifecycle RPCs. Implement a bounded repair in the Skill Workshop service so pending create proposals are marked `stale` with a clear status reason when the target skill or create-target support file already exists before apply/revise/list/inspect as appropriate, while preserving no-clobber behavior. Update the Control UI lifecycle action path to force a status refresh after failed apply/reject calls when the backend may have changed proposal state. Add focused regression coverage for create proposal target-created-after-proposal, support-file collision if applicable, and UI refresh-after-error behavior. Do not add new config options, do not change shared-skill targeting, and do not edit CHANGELOG.md.
