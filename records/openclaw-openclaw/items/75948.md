## Summary

Keep this PR open for maintainer review. The continuity bug is credible and the PR targets the right UI/TUI surfaces, but the diff sends a new `sessionId` field through `chat.send` without updating Gateway's strict protocol contract or handler, so the follow-up sends it is meant to preserve would fail validation.

## What This Changes

The PR adds Control UI and TUI session-id continuity across reconnects/relaunches, plus focused tests, docs, and a changelog entry.

## Best Possible Solution

Align the Gateway protocol and handler with the new continuity contract, then keep the UI/TUI state changes only if a schema/handler regression test proves post-history sends continue the intended transcript.

## Reproduction Assessment

Yes. A static reproduction path is clear: after `chat.history` returns a `sessionId`, the patched clients include `sessionId` on `chat.send`, and current Gateway validation rejects that extra property before dispatch.

## Solution Assessment

No. Client-side persistence alone is not the best fix because the server-side `chat.send` contract must accept and honor the backing session id; otherwise the new field either fails validation or would be ignored.
