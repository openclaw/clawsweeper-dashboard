## Summary

Close as implemented on main. The PR's exact Feishu dispatcher parameter is not present on current main, but the observable Feishu reply-media failure is already solved by the shared reply-media normalizer: local reply media is validated with agent-scoped access, staged into managed media storage, and only then handed to the Feishu dispatcher. That safe current design shipped in v2026.4.15 and is contained in the latest release.

## What This Changes

The PR imports `getAgentScopedMediaLocalRoots`, derives roots at both Feishu bot dispatcher construction paths, adds `mediaLocalRoots` to `CreateFeishuReplyDispatcherParams`, and forwards it into `sendMediaFeishu`.

## Best Possible Solution

Keep the shipped shared reply-media staging design and close this redundant Feishu-specific PR. Related open PRs such as #51375 and #58591 should be reviewed against the same current-main behavior rather than adding channel-local root plumbing unless someone provides a fresh production path that bypasses `dispatchReplyFromConfig`.

## Reproduction Assessment

Yes. The original failure path is an agent reply carrying a local media path; current main routes Feishu agent replies through `dispatchReplyFromConfig`, which normalizes/stages media before the dispatcher sees it, and tests cover workspace-relative staging plus Feishu delivery normalization.

## Solution Assessment

Yes. The current shared normalizer is the more maintainable fix because it centralizes local-file validation and staging for reply delivery across channels; the PR's Feishu-only `mediaLocalRoots` threading is reasonable but redundant for the production reply path now on main.
