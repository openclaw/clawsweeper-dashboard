## Summary

Keep open. The report is fresh, detailed, and still maps to a missing Feishu-owned doctor repair/quarantine path on current main. The paired open PR #74397 uses closing syntax for the same issue, so this issue should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue requests that stale or corrupt Feishu local runtime state and Feishu-scoped direct session artifacts stop being able to drive sustained gateway CPU hot loops, ideally through scoped validation, quarantine, logging, and doctor repair.

## Best Possible Solution

Land #74397 or an equivalent Feishu-owned doctor repair that detects corrupt Feishu-local JSON and Feishu direct session artifacts, archives/rebuilds only Feishu-scoped state, preserves Feishu App ID/secret config plus non-Feishu and ACP binding sessions, and locks the behavior with focused Feishu doctor tests.

## Reproduction Assessment

Unclear. The issue includes strong field evidence, CPU samples, logs, ruled-out causes, and a successful manual workaround, but the private stale/corrupt state files are not available for deterministic reproduction against current main.

## Solution Assessment

Yes. A Feishu-owned doctor repair through the existing channel doctor adapter is the narrowest maintainable path because the affected state is Feishu-local runtime/dedup data plus Feishu-scoped direct session metadata, not a generic core startup migration.
