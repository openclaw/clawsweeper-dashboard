## Label Justifications

- `P1`: Accepted user messages can wait behind a wedged maintenance embedded run for minutes until manual stop or delayed session recovery.
- `impact:session-state`: The report centers on session-scoped embedded-run and deferred context-maintenance state remaining active without useful progress.
- `impact:message-loss`: Inbound messages are accepted into the queue but not processed while the session is blocked behind maintenance.
- `impact:crash-loop`: The user-visible failure mode is a session hang requiring manual intervention or a much later recovery path.

## Summary

Keep open: current main and the latest release still enqueue deferred context-engine turn maintenance without the command-lane task timeout options that already exist, and the linked/related work does not own this exact maintenance-lock/no-timeout failure.

## What This Changes

The issue requests bounded turn-maintenance execution, burst control for same-instant maintenance enqueue storms, and interactive recovery behavior so queued user messages are not blocked behind a wedged embedded maintenance run.

## Best Possible Solution

Add bounded deferred maintenance execution using the existing command-lane timeout/abort machinery, plus a maintainer-approved burst-control policy and any interactive recovery threshold change.

## Reproduction Assessment

Yes at source level: current main enqueues deferred turn maintenance without taskTimeoutMs even though command lanes support task timeouts. I did not run a live lossless-claw lock-contention scenario, so the exact production stall remains log-backed rather than locally reproduced.

## Solution Assessment

Unclear as a final policy: the existing command-lane timeout machinery is the narrow implementation seam, but maintainers still need to choose the timeout budget, cancellation behavior, and burst-control shape.
