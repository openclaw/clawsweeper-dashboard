## Label Justifications

- `P1`: The report describes an interactive session where accepted user messages can wait behind wedged maintenance for minutes until manual stop or delayed recovery.
- `impact:session-state`: The affected path is session-scoped embedded-run and context-maintenance state staying active without useful progress.
- `impact:message-loss`: Queued inbound messages are accepted but not processed while maintenance blocks the session from reaching the next turn.
- `impact:crash-loop`: The user-visible failure is a session hang requiring manual intervention or a much later recovery path.

## Summary

Keep open: current main and v2026.6.10 still enqueue deferred context-engine turn maintenance without the command-lane task timeout options that already exist, while the linked repairs cover adjacent lane and recovery failures rather than this maintenance-lock/no-timeout gap.

## What This Changes

The issue requests bounded and coalesced deferred context-engine turn maintenance so a wedged maintenance run cannot stall an interactive session until stuck-session abort recovery.

## Best Possible Solution

Add bounded maintenance execution using the existing command-lane timeout/abort hooks, coalesce or stagger bursty background maintenance, and make any shorter interactive recovery policy a maintainer-approved design.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level, because current main and v2026.6.10 enqueue deferred turn maintenance without taskTimeoutMs while the command queue supports it; no live lossless-claw lock-contention run was executed, so the end-to-end production stall is not fully reproduced here.

## Solution Assessment

Is this the best way to solve the issue? Unclear as a final fix: using the existing command-lane timeout machinery is the narrowest likely mechanism, but the timeout budget, burst coalescing shape, and interactive abort policy need maintainer direction.
