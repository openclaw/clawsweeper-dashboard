## Summary

Keep open: current main and v2026.6.10 still enqueue deferred turn maintenance without a per-task timeout, and the open linked PR is the active fix candidate rather than merged implementation.

## What This Changes

The issue requests a bounded timeout for deferred turn-maintenance work, burst coalescing or staggering for same-instant maintenance batches, and reconsideration of the interactive stuck-run abort threshold.

## Best Possible Solution

Review and land or deliberately revise the linked timeout-and-fence PR, then track any remaining burst coalescing or interactive recovery policy as narrower follow-up work if needed.

## Reproduction Assessment

Yes at source level: current main and v2026.6.10 enqueue deferred turn maintenance without taskTimeoutMs even though the command queue supports task timeouts. I did not run the live lock-contention scenario, so the production stall remains log-backed rather than locally reproduced.

## Solution Assessment

Mostly yes: using the existing command-lane timeout seam plus a late-side-effect fence is the narrow repair for the central hang. The timeout budget/default, burst coalescing, and any shorter interactive abort policy still need maintainer judgment.
