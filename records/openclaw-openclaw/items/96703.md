## Summary

Keep open: current main and v2026.6.10 still enqueue deferred context-engine turn maintenance without the command-lane task timeout options that already exist, and no linked PR currently owns this exact maintenance-lock/no-timeout failure. The older related items cover adjacent scheduler and recovery seams, but they do not close this issue's central lock-contention timeout gap.

## What This Changes

The issue reports that post-turn context-engine maintenance can wedge behind a maintenance lock with no per-task timeout, leaving queued user messages blocked until the broader session abort or manual stop.

## Best Possible Solution

Add a bounded maintenance execution path using the existing command-lane timeout/abort machinery, coalesce or stagger bursty cross-session turn maintenance, and keep any shorter interactive recovery policy behind an explicit maintainer-approved design.

## Reproduction Assessment

Yes at source level: current main and v2026.6.10 enqueue deferred turn maintenance without taskTimeoutMs even though the command queue supports task timeouts. I did not run a live lossless-claw lock-contention reproduction, so the exact production stall remains log-backed rather than locally reproduced.

## Solution Assessment

Unclear as a final fix: using the existing command-lane timeout machinery is the right narrow direction, but the timeout budget, burst coalescing shape, and interactive abort-threshold policy need maintainer choice. The earlier https://github.com/openclaw/openclaw/pull/93727 fix removed the old lane-idle livelock but does not bound plugin maintenance execution.
