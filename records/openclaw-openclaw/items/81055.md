## Summary

Keep open: current main still has a source-proven path where `status --json` secret diagnostics can be written through `runtime.log` before the JSON payload, and an open unmerged PR already targets this issue with closing syntax.

## What This Changes

The issue asks for `openclaw status --json` to keep `[secrets]` diagnostics off stdout so `jq` and `JSON.parse` consumers receive a clean JSON document.

## Best Possible Solution

Land a corrected version of https://github.com/openclaw/openclaw/pull/81068 that routes command secret diagnostics away from stdout for JSON commands with explicit mode/stream handling and focused regression coverage.

## Reproduction Assessment

Yes. Source inspection shows `status --json` can resolve secrets, call `runtime.log` for `[secrets]` diagnostics, and only later write the JSON payload; a live proof should use a temp config with an unresolved SecretRef and parse stdout with `jq` or `JSON.parse`.

## Solution Assessment

Yes for the direction, but not yet for the current open PR as reviewed. The maintainable fix should pass explicit JSON/diagnostic-stream intent into shared secret resolution rather than inferring output mode from the display command name.
