## Summary

Keep open: current `origin/main` still has the source-proven path where route-first `openclaw status --json` can emit `[secrets]` diagnostics through `runtime.log` before the JSON document, so the closed sibling https://github.com/openclaw/openclaw/issues/72962 does not cover this status path.

## What This Changes

Ensure `openclaw status --json` keeps stdout as parseable JSON when SecretRef resolution produces diagnostics.

## Best Possible Solution

Preserve the route-first JSON contract by routing these secret diagnostics to stderr before any JSON payload is written, or by keeping them only in `secretDiagnostics`, with focused regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: route-first `status --json` can resolve command secrets, emit `[secrets]` through `runtime.log`, and only later write the JSON payload.

## Solution Assessment

Yes. The narrow maintainable fix is to keep stdout reserved for the JSON document while preserving diagnostics on stderr or inside the existing `secretDiagnostics` field, not to add a new user option or change SecretRef semantics.
