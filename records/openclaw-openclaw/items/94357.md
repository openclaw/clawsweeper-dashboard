## Summary

Keep open: the WhatsApp hook-only direction is useful, but the patch is not merge-ready because observe-only messages can still be marked read and the no-session behavior needs explicit maintainer confirmation.

## What This Changes

The PR changes WhatsApp access control so disabled DM/group messages with `pluginHooks.messageReceived` become observe-only, adds admission tests, and short-circuits `processMessage` after hook emission.

## Best Possible Solution

Keep the PR open, thread `shouldMarkRead` through the WhatsApp monitor read-receipt path, confirm the no-session hook-only contract, and then prove a real disabled-policy WhatsApp inbound run.

## Reproduction Assessment

Yes. Source inspection shows the PR creates accepted observe-only messages with `shouldMarkRead=false`, while the current monitor queues and finalizes read receipts without checking that flag; I did not run live WhatsApp in this read-only review.

## Solution Assessment

No, not yet. The direction is plausible, but the implementation must preserve the no-read-receipt invariant and settle whether WhatsApp should bypass or reuse shared observe-only turn semantics.
