## Summary

Keep open. The PR targets a real source-reproducible stale-lock bug, but the current patch is not merge-ready because it changes generic plugin SDK lock semantics in a way that can reclaim live long-running locks, and the proof is test-only.

## What This Changes

The PR makes `shouldRemoveDeadOwnerOrExpiredLock` treat expired or unparseable `createdAt` as authoritative before pid liveness and updates stale-lock tests for pid reuse.

## Best Possible Solution

Fix the pid-reuse stale-lock bug with a lock-owner strategy that preserves generic SDK mutual exclusion for live long-running locks, or explicitly approve and document the SDK semantic change with sibling-surface tests.

## Reproduction Assessment

Yes, from source inspection: current main returns on pid liveness before checking `createdAt`, and the existing test encoded that stale pid-owned locks stay held. I did not run tests because this review is read-only.

## Solution Assessment

No, not as written. The central bug is real, but applying createdAt-first reclamation to every SDK file lock needs a narrower owner-identity fix or explicit maintainer approval with sibling proof.
