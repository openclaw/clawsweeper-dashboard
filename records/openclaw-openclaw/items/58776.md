## Label Justifications

- `P1`: Silent sub-agent startup aborts can block real multi-agent workflows by leaving child runs looking healthy until timeout.
- `impact:session-state`: The issue is about child run lifecycle/session state being stuck or misclassified during startup aborts.
- `impact:crash-loop`: The user-visible symptom is a hang-like availability failure where aborted startup waits look like long-running sessions.

## Summary

Keep open. Current main and the latest shipped tag still lack the requested `first-progress`/`startup-failed` lifecycle phases; the relevant consumers handle only `start`/`end`/`error`, and an aborted `end` without a stop reason still resolves as `timeout`. The open implementation path is https://github.com/openclaw/openclaw/pull/77415, which remains unmerged.

## What This Changes

Add explicit sub-agent `first-progress` and typed `startup-failed` lifecycle reporting so startup aborts are not indistinguishable from slow or hung child runs.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/77415 or a replacement that carries additive startup progress and startup failure phases through embedded emission, registry completion, gateway wait/session state, SDK normalization, docs, and focused regression coverage.

## Reproduction Assessment

Yes, at source level. Current main still handles the affected lifecycle as `start`/`end`/`error` and maps aborted `end` without a stop reason to timeout, though I did not replay the reporter's live corpus fixture.

## Solution Assessment

Yes. The additive `first-progress`/`startup-failed` direction is the narrow maintainable fix, and this issue should stay open until the linked PR or a replacement implementation merges.
