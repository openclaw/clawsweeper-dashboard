## Summary

Keep open. Current main still lacks the requested lifecycle distinction: embedded runs only emit start/end/error lifecycle phases, sub-agent and gateway wait consumers ignore additive phases, and aborted end events still become timeout outcomes after the retry grace window.

## What This Changes

The issue requests additive sub-agent lifecycle phases for first progress and typed startup failures so parent sessions can distinguish silent startup aborts from slow or hung child runs.

## Best Possible Solution

Add a backwards-compatible core lifecycle seam: emit first progress once from the first qualifying assistant text or tool call, emit typed startup failure for silent startup abort/timeout/empty-output/auth-failure cases, and preserve existing start/end/error behavior for current consumers.

## Reproduction Assessment

Yes at the code-path level: emitting lifecycle `end` with `aborted:true` and no stop reason reproduces the timeout mapping in the current sub-agent registry, gateway wait, and session lifecycle state tests. The full live transcript fixture is unavailable in this checkout.

## Solution Assessment

Yes, the requested additive lifecycle events are the right maintainable direction if implemented with explicit consumer handling and compatibility tests. The safer repair is to add the new phases without removing retry-grace behavior for transient provider retries or changing existing terminal events.
