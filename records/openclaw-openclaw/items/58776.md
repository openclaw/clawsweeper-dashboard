## Label Justifications

- `P1`: Sub-agent startup aborts can remain indistinguishable from healthy slow starts or long timeouts, blocking real multi-agent workflows until lifecycle reporting is fixed.
- `impact:session-state`: The issue is about child run lifecycle/session state being stuck or misclassified during startup aborts.
- `impact:crash-loop`: The user-visible symptom is a hang-like availability failure where aborted startup waits look like long-running sessions.

## Summary

Keep open. Current main and the latest shipped tag still expose the reported source-level lifecycle gap: embedded subagent runs emit/consume start/end/error only, aborted end events collapse to timeout, and the open successor PR remains unmerged.

## What This Changes

The issue requests additive sub-agent lifecycle signals so startup aborts and empty-output startup failures surface separately from ordinary timeouts or slow starts.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/77415 or a replacement that carries first-progress/startup-failed through embedded lifecycle emission, registry completion, gateway wait snapshots, session lifecycle state, SDK normalization, docs, and focused regression coverage.

## Reproduction Assessment

Yes, at source level. Current main still handles only start/end/error in the affected lifecycle consumers and maps end+aborted to timeout, though I did not replay the reporter's live corpus fixture.

## Solution Assessment

Yes. The additive first-progress/startup-failed direction is the narrow maintainable fix, and this issue should stay open until the open successor PR or a replacement implementation merges.
