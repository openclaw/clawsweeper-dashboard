## Label Justifications

- `impact:crash-loop`: The user-visible failure is a hang-like run that waits until timeout instead of surfacing a terminal startup failure.

## Summary

Keep open: current main and the latest release still only expose lifecycle start/end/error handling for this path, so the requested first-progress/startup-failed behavior is not implemented. The open implementation candidate at https://github.com/openclaw/openclaw/pull/77415 should be reviewed or replaced, but it is not merged and is not enough to close this issue yet.

## What This Changes

Add sub-agent first-progress and startup-failed lifecycle signals so parent runs can distinguish slow startup from aborted startup.

## Best Possible Solution

Land or replace the open candidate with a current-main implementation that carries first-progress/startup-failed through embedded lifecycle emission, subagent registry, agent.wait/session projection, SDK normalization, docs, focused tests, and redacted real behavior proof.

## Reproduction Assessment

Yes at source level: current main only emits/projects lifecycle start/end/error and maps aborted end events to timeout, while first-progress and startup-failed are absent. I did not run a live Gateway/subagent startup-failure scenario in this read-only pass.

## Solution Assessment

No, not solved on current main. The additive lifecycle direction is plausible, but the best fix needs a current-main port with SDK/docs/tests and maintainer approval for the lifecycle contract before this issue closes.
