## Summary

Keep open: current main and v2026.6.8 still drop the requested provider-returned image metadata on the checked OpenAI-compatible, direct OpenAI, Google, and fal image-generation paths, and https://github.com/openclaw/openclaw/pull/85466 is the active implementation candidate.

## What This Changes

Capture provider-returned usage/cost metadata for image generation where upstream responses include it, without synthesizing fake usage for providers that do not return it.

## Best Possible Solution

Review and land or replace the provider-local metadata passthrough in https://github.com/openclaw/openclaw/pull/85466, preserving only upstream-returned metadata in the existing `ImageGenerationResult.metadata` bag.

## Reproduction Assessment

Yes, at source level: current main forwards `ImageGenerationResult.metadata`, but the checked provider paths do not populate response-level OpenAI-compatible usage, Google `usageMetadata`, or the named fal response fields. I did not run live generations in this read-only review.

## Solution Assessment

Yes, with maintainer acceptance of the additive SDK payload shape: provider-local passthrough into the existing metadata bag is the narrowest maintainable path and avoids synthesized token counts. The linked open PR is the active candidate, so this issue should stay open until that PR or an equivalent change lands.
