## Summary

Keep open: the fix direction is useful, but the PR still has a stale hook message-id bug after filtering identity-less payload results, and the supplied proof is a synthetic test-helper smoke rather than a real cron/channel run.

## What This Changes

The PR filters identity-less outbound delivery results in shared delivery code and adds a Telegram cron announce regression test for empty receipts.

## Best Possible Solution

Keep the identity filtering, derive hook ids only from the current payload's deliveredResults, cover multi-payload suppression, and require redacted live or equivalent cron/channel proof before merge.

## Reproduction Assessment

Yes, at source level: current main pushes identity-less text/formatted/media send results into the results array, and sendDurableMessageBatch treats a non-empty results array as sent. I did not run the live WeChat/Feishu cron flow in this read-only review.

## Solution Assessment

No, not yet: filtering identity-less results is the right narrow fix, but hook ids should be derived per payload and the affected cron/channel behavior still needs real proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Use per-payload message ids for suppressed sends:** `src/infra/outbound/deliver.ts:1630`
  - body: After this filters identity-less results, a later payload can add no new results while messageId still comes from results.at(-1), which can point at a previous payload. Use the current payload's deliveredResults, or omit the id when none, in the text and text-fallback branches so suppressed sends do not reuse another send's platform id.
