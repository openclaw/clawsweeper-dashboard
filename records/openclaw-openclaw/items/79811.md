## Summary

Keep open: this is a plausible focused bug fix for the linked cron delivery bug, but the head still has a stale `message_sent` id regression after filtering and the posted proof uses a synthetic local adapter rather than the reported cron/channel path.

## What This Changes

The branch filters identity-less outbound delivery results from text, formatted-text, and media paths, and adds a Telegram cron announce regression test for empty receipts.

## Best Possible Solution

Keep the identity filtering direction, derive hook ids from the current payload's `deliveredResults`, add a multi-payload regression test, and require redacted real cron/channel proof before landing.

## Reproduction Assessment

Yes, at source level: current main appends identity-less text/formatted/media send results, and the durable send wrapper treats any non-empty result array as sent. I did not run the live WeChat/Feishu cron flow in this read-only review.

## Solution Assessment

No, not yet: filtering identity-less results is the right narrow fix, but the PR should avoid stale hook message ids and prove the affected cron/channel behavior after the fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Use per-payload results for hook message ids:** `src/infra/outbound/deliver.ts:1624-1630`

Derive the hook id from `deliveredResults` in this text branch and the text fallback branch so suppressed payloads do not point at another send.
