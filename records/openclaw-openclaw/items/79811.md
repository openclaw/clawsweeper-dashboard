## Summary

Keep open: the fix direction is valid, but the current PR still has a correctness bug in `message_sent` hook metadata after filtering no-identity results, and the after-fix proof is synthetic rather than a real cron/channel run.

## What This Changes

The branch filters identity-less outbound delivery results in the shared delivery runtime and adds a Telegram cron announce regression test for empty receipts.

## Best Possible Solution

Keep the identity filtering direction, derive hook ids from each payload's `deliveredResults`, add a multi-payload regression, and require redacted live cron/channel proof before landing.

## Reproduction Assessment

Yes, at source level: current main appends identity-less outbound adapter results and durable sending treats any non-empty result array as sent. The linked issue also supplies a live WeChat/Feishu symptom report, but I did not run that live flow in this read-only review.

## Solution Assessment

No, not yet: filtering identity-less results is the right narrow fix, but the PR should avoid stale hook message ids and prove the affected cron/channel behavior after the fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Use per-payload results for hook message ids:** `src/infra/outbound/deliver.ts:1624-1630`
  - body: After this filters identity-less results, a later payload can append no new results while `messageId` still comes from `results.at(-1)`, which can point to a prior payload. Derive hook ids from `deliveredResults` in the text and text-fallback paths so suppressed payloads do not emit another send's id.
