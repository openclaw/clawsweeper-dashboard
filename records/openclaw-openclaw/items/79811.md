## Summary

Keep open: the fix direction matches the linked cron delivery bug, but this PR still has a correctness issue where `message_sent` hook metadata can reuse a stale platform message id after an identity-less send, and the external real-behavior proof gate is not satisfied.

## What This Changes

The PR filters outbound adapter results without a platform delivery identity from shared delivery results and adds a cron announce regression test for an empty Telegram receipt.

## Best Possible Solution

Keep the identity filtering, derive hook message ids only from the current payload's delivered results, add a multi-payload regression test, and verify a real cron announce path before landing.

## Reproduction Assessment

Yes, at source level: current main appends identity-less outbound results and the linked issue gives concrete WeChat/Feishu cron announce steps. I did not execute the live channel flow because this review was read-only.

## Solution Assessment

No, not yet: filtering identity-less results is the right narrow direction, but the PR should avoid stale `message_sent` ids and prove the real cron announce behavior after the fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Use per-payload results for hook message ids:** `src/infra/outbound/deliver.ts:1665-1667`
  - body: After identity-less sends are filtered, a later payload can add zero `results` entries while `messageId` still comes from `results.at(-1)`, which may belong to an earlier payload. That makes the suppressed payload's `message_sent` hook point at the wrong platform message; use the current `deliveredResults` for this id, and apply the same pattern to the text fallback branch below.
  - confidence: 0.88
