## Summary

Keep open: the PR targets a real source-level cron delivery false-positive, but the head still has a blocking hook metadata bug and the proof is still synthetic rather than an after-fix real cron/channel run.

## What This Changes

The PR filters identity-less outbound delivery results from shared delivery handling and adds a cron announce regression test for empty Telegram receipts.

## Best Possible Solution

Keep the identity filtering, derive hook message ids only from the current payload's delivered results, add multi-payload regression coverage, and require redacted after-fix cron/channel proof before merge.

## Reproduction Assessment

Yes, at source level: current main appends identity-less outbound results, and the linked issue gives concrete WeChat/Feishu cron announce steps. I did not execute the live credentialed channel flow in this read-only review.

## Solution Assessment

No, not yet: filtering identity-less results is the right narrow direction, but the PR should avoid stale message_sent ids and prove the real cron announce behavior after the fix.
