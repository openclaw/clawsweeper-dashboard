## Summary

Keep open: the PR addresses a valid cron delivery false-positive path, but the current patch still has a per-payload hook metadata bug and lacks after-fix real behavior proof.

## What This Changes

The PR filters identity-less outbound delivery results from shared delivery accounting and adds a cron Telegram announce regression test for empty platform receipts.

## Best Possible Solution

Keep the identity filtering, derive hook ids from the current payload's deliveredResults, add multi-payload regression coverage, and require redacted real cron/channel proof before merge.

## Reproduction Assessment

Yes, at source level: current main appends identity-less outbound results and the linked issue provides concrete WeChat/Feishu cron announce steps. I did not execute the live channel flow in this read-only review.

## Solution Assessment

No, not yet: filtering identity-less results is the right narrow direction, but the PR should avoid stale message_sent ids and prove the real cron announce behavior after the fix.
