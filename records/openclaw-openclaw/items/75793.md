## Summary

Keep open. Current main still rejects non-TTY `models auth login` before provider selection, and the command does not expose a credential flag even though API-key provider methods already have non-interactive plumbing in the onboarding path.

## What This Changes

The issue requests a scriptable `openclaw models auth login` path, such as `--api-key <KEY>`, so operators can provision provider credentials without an interactive TTY or hand-editing `auth-profiles.json`.

## Best Possible Solution

Add a narrow non-interactive API-key login path that reuses the existing provider auth/profile persistence contract, updates CLI help and docs, and leaves prompt-based OAuth/device/custom methods interactive until each provider exposes an explicit headless method.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: any non-TTY invocation enters `modelsAuthLoginCommand` and hits the unconditional `process.stdin.isTTY` guard before provider auth can run.

## Solution Assessment

Yes, the maintainable fix is to relax the guard only for explicit provider/method plus credential input and reuse the provider API-key auth helpers rather than documenting manual `auth-profiles.json` edits or adding a parallel store writer.
