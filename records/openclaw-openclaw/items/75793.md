## Summary

Keep open. Current main still rejects non-TTY `models auth login` before provider or method handling, and the CLI registration exposes no credential input flag, while provider API-key methods already have a non-interactive persistence path through onboarding.

## What This Changes

The issue requests scriptable credential provisioning for `openclaw models auth login`, starting with API-key input, so automation does not need an interactive TTY or manual `auth-profiles.json` edits.

## Best Possible Solution

Add a narrow non-interactive API-key login path that reuses provider auth metadata and auth-profile persistence, updates CLI help and user docs, and leaves broader OAuth/token redesign to provider-owned methods with explicit headless support.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: with `process.stdin.isTTY` false, `modelsAuthLoginCommand` throws at the top of the command before any supplied provider or method can be used.

## Solution Assessment

Yes for the API-key subset: the maintainable path is to route explicit credential input through the existing provider non-interactive API-key contract. A blanket non-interactive OAuth/token flag would be underspecified and should not be part of the first fix.

## Repair Work Prompt

Fix the scriptability gap for `openclaw models auth login` reported at https://github.com/openclaw/openclaw/issues/75793. Add a narrow non-interactive API-key path to the direct login command by reusing existing provider auth metadata and `runNonInteractive`/auth-profile persistence contracts rather than hand-writing a parallel `auth-profiles.json` schema. Expected boundary: support explicit provider + method where the selected method can complete headlessly, starting with API-key methods and a documented credential input surface; preserve interactive behavior for OAuth, device/browser, setup-token, and custom methods unless they explicitly expose a safe non-interactive method. Update CLI help, docs at `docs/cli/models.md` and likely `docs/start/wizard-cli-automation.md`, focused tests for command behavior/CLI parsing, and `CHANGELOG.md` with user-facing attribution to the issue reporter. Do not conflate this with profile-id work in #40402/#49315 or remote gateway storage in #42291, do not trust/load untrusted workspace plugins beyond existing setup rules, and do not weaken SecretRef/ref-mode semantics or encourage plaintext secrets without documenting env/ref alternatives.
