## Summary

Keep open. Current main still rejects non-TTY `models auth login` before provider or method handling, and the command exposes no credential input flags, while the provider API-key non-interactive contract already exists through onboarding.

## What This Changes

Allow scripted provisioning of provider credentials through `openclaw models auth login` without requiring an interactive TTY, especially for API-key methods.

## Best Possible Solution

Add a narrow non-interactive API-key login path that reuses provider auth metadata and auth-profile persistence, updates CLI help and user docs, and leaves broader OAuth/token support to provider-owned headless contracts.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: with `process.stdin.isTTY` false, `modelsAuthLoginCommand` throws before any supplied provider, method, or existing API-key non-interactive plumbing can run.

## Solution Assessment

Yes for the API-key subset. The narrowest maintainable fix is to route explicit credential input through the existing provider `runNonInteractive` API-key contract; a generic OAuth/token flag would be underspecified for this issue.
