## Summary

Keep open. Current main still rejects non-TTY `models auth login` before provider or method handling, the login command exposes no credential-input option, and the existing provider API-key non-interactive path is only wired through onboarding rather than direct login.

## What This Changes

The issue requests scriptable credential input for `openclaw models auth login`, especially API-key provisioning, so provider auth can be configured without an interactive TTY.

## Best Possible Solution

Add a narrow non-interactive API-key login path that reuses provider auth metadata, the onboarding API-key resolver, and auth-profile persistence; update CLI help/docs and leave broader OAuth/token headless support to explicit provider-owned contracts.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: with `process.stdin.isTTY` false, `modelsAuthLoginCommand` throws at the top of the command before provider or method resolution, and the CLI has no credential flag that could bypass it.

## Solution Assessment

Yes for the API-key subset. The maintainable fix is to route explicit API-key input through existing provider `runNonInteractive` support rather than asking users to hand-edit `auth-profiles.json`; OAuth/token flags need separate provider-specific design.
