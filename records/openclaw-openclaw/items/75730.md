## Summary

Keep open, but the proposed implementation should not merge as-is. The PR addresses a real SecretRef-sensitive web_search area, yet it reimplements secret resolution synchronously inside the web-search runtime, walks the entire config tree, and bypasses the existing provider, active-surface, and file/env security contracts.

## What This Changes

The PR adds a synchronous recursive SecretRef resolver to `src/web-search/runtime.ts` and adds web-search runtime tests for file, env, snapshot, and plain-string credential cases.

## Best Possible Solution

Keep SecretRef materialization in the central secrets runtime and selected web-provider credential path. If an unresolved web_search snapshot still escapes in production, add a reproduction through `prepareSecretsRuntimeSnapshot` or the gateway reload path, then fix the collector or `resolveRuntimeWebProviderSelection` flow so only the active web_search credential target is resolved with `resolveSecretRefValues`.

## Reproduction Assessment

Unclear. The PR reproduces the failure by manually activating an unresolved snapshot, but current main’s normal preparation path resolves collected plugin SecretRefs before activation, so the production reproduction path is not yet high-confidence.

## Solution Assessment

No. The proposed sync walk is not the best fix because it duplicates SecretRef resolution inside web-search runtime and skips active-surface and provider security semantics; the narrower maintainable fix is in snapshot preparation or selected-provider credential resolution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Resolve only active web-search credentials:** `src/web-search/runtime.ts:108`
  - body: The new call recursively resolves every SecretRef in the selected config before provider execution. Current docs and runtime code keep non-selected web-search refs inactive, so this would materialize inactive provider keys and unrelated plugin secrets into the config handed to web-search providers. Keep resolution scoped to the selected credential target.
  - confidence: 0.9
- **[P2] Honor env provider allowlists:** `src/web-search/runtime.ts:91`
  - body: Resolving env refs with `process.env[value.id]` ignores `secrets.providers.<provider>.allowlist`, provider source mismatches, and missing/empty checks enforced by the central resolver. A configured env SecretRef can therefore read an env var that the selected provider config would reject.
  - confidence: 0.88
- **[P2] Use the central file SecretRef resolver:** `src/web-search/runtime.ts:78`
  - body: The sync file branch does not match file provider semantics: JSON-mode refs are JSON Pointers, but `parsed[ref.id]` only reads top-level keys, and `singleValue` ids are not validated. Valid file-backed refs such as `/providers/brave/apiKey` can resolve to `undefined` instead of the configured secret.
  - confidence: 0.87
