## Summary

Keep open: current main and v2026.6.10 still lack a per-agent Telegram outbound topic default. Existing explicit `threadId`/`:topic:` targets, `defaultTo`, and active-context inheritance cover parts of the workflow, but the requested default binding remains a Telegram config/product decision.

## What This Changes

The issue requests a Telegram config-level outbound topic/thread default per agent so proactive sends can target a forum topic without each call passing `threadId`.

## Best Possible Solution

Keep the current active-context inheritance and explicit topic-target support, then add a narrow Telegram-owned outbound-default contract only if maintainers approve the config surface, precedence rules, docs, and regression coverage.

## Reproduction Assessment

Yes for the missing config surface: current main and v2026.6.10 have no outbound topic binding field in the strict Telegram topic schema or topic config type. I did not live-reproduce the older nested proactive-send path on current main.

## Solution Assessment

Unclear: the requested `outbound: true` flag is plausible, but current code already has explicit topic targets, top-level `defaultTo`, and active-context inheritance while repository policy sets a high bar for new config/default surfaces.
