## Summary

Keep open: current main and v2026.5.22 still lack a Telegram config-level outbound topic default, while the existing auto-threading path only inherits an ambient tool context when one is present. This remains a real Telegram routing/product-contract request rather than cleanup-safe close material.

## What This Changes

The issue requests a Telegram per-agent outbound default topic/thread binding so proactive sends can target a configured forum topic without per-call `threadId`.

## Best Possible Solution

Add a Telegram-owned outbound default topic contract with deterministic precedence: explicit `threadId` or `:topic:` target first, active current topic context second, configured per-agent fallback last, plus duplicate validation, docs, and regression tests.

## Reproduction Assessment

Yes, at source level: the topic type/schema has no outbound default field, strict validation rejects unknown topic keys, and Telegram auto-threading only uses ambient tool context when present. The issue comment also gives live evidence from v2026.5.18 for a successful no-thread proactive send that was hard to find from the active topic.

## Solution Assessment

Unclear until maintainers choose the public config shape and ambiguity policy. The narrow maintainable direction is a Telegram-owned config contract and precedence rules, not a broad core default or silent override of current context.
