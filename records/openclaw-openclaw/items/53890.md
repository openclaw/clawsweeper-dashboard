## Label Justifications

- `impact:message-loss`: The issue is about proactive Telegram messages landing outside the expected forum topic and becoming effectively hard to find for users working in that topic.

## Summary

Keep open: current main and v2026.5.19 still lack a Telegram config-level outbound topic default, and the latest live issue comment describes the proactive no-thread send path still producing hard-to-find Telegram messages.

## What This Changes

Add a Telegram config-level per-agent outbound topic/thread default so proactive sends can target an agent's forum topic without passing `threadId` on every call.

## Best Possible Solution

Add a Telegram-owned outbound default topic contract with deterministic precedence: explicit `threadId` or `:topic:` target first, active current topic context second, configured per-agent fallback last, plus duplicate validation, docs, and regression tests.

## Reproduction Assessment

Yes, at source level: the topic type/schema has no outbound default field, strict validation rejects unknown topic keys, and Telegram auto-threading is context-only. The May 20, 2026 live note also reports a v2026.5.18 proactive send that succeeded without a topic thread.

## Solution Assessment

Unclear until maintainers choose the public config shape and ambiguity policy. The narrow maintainable direction is a Telegram-owned config contract and precedence rules, not a broad core default or silent override of current context.

- A boolean `outbound: true` shape is ambiguous if multiple topics in the same group claim the same agent, so duplicate-default validation and error behavior need a maintainer decision.
- Configured fallback precedence must not override an explicit `threadId`, a `:topic:` target, or a valid inherited active topic context.
- The live repro points at a nested/dynamic proactive message-tool path, so a config-only fallback may not cover all missing-current-context cases unless that outbound threading seam is audited too.
