## Summary

Keep open: current main has thread-inheritance machinery for active same-conversation sends, but it still lacks the requested Telegram config-level outbound topic binding, and the live issue comment shows a real no-thread delivery pain point that needs a product/API decision rather than cleanup closure.

## What This Changes

Add a Telegram per-agent default outbound topic/thread binding so proactive message sends can target a forum topic without each call supplying `threadId`.

## Best Possible Solution

Decide the product boundary first, then either document current thread inheritance as the intended answer or add a plugin-owned Telegram outbound default with explicit precedence, schema/docs coverage, and live Telegram proof.

## Reproduction Assessment

Yes for the source-level config gap: current main's strict Telegram topic schema has no `outbound` field. The v2026.5.18 live no-thread delivery report is credible, but I did not run a live current-main Telegram reproduction.

## Solution Assessment

Unclear. The requested flag is plausible, but current main also has active thread inheritance and thread-binding APIs, so maintainers should choose the canonical behavior before implementation.
