## Summary

Keep open: current main and v2026.5.22 still lack a Telegram config-level outbound topic default, and the remaining behavior is a product/config contract decision rather than cleanup-close material.

## What This Changes

The issue requests a Telegram config-level outbound topic default per agent so proactive sends can target the agent's forum topic without an explicit threadId.

## Best Possible Solution

Add a Telegram plugin-owned outbound default topic contract with deterministic precedence: explicit thread first, active same-chat context second, configured per-agent fallback last, plus duplicate validation, docs, and regression coverage.

## Reproduction Assessment

Yes, at source level: the current strict topic schema/type reject the requested config shape, and outbound thread resolution has no configured per-agent fallback. The live comment adds a real v2026.5.18 Telegram forum-topic send that succeeded without a thread and was hard to find.

## Solution Assessment

Unclear until maintainers choose the public config shape and ambiguity policy. The maintainable direction is a Telegram plugin-owned config contract and precedence rules, not a broad core default or silent override of explicit/current thread context.
