## Best Possible Solution

Keep the current same-context inheritance path, then settle a narrow Telegram outbound-default contract only if maintainers accept the new config surface, precedence rules, docs, and regression coverage.

## Solution Assessment

Unclear: `outbound: true` is plausible, but current code already favors active-context inheritance and repository policy sets a high bar for new config. Maintainers should decide whether the remaining no-context proactive case needs a config default, a warning/fail-closed behavior, or a broker/session-binding design.
