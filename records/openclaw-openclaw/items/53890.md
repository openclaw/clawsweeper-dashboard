## Label Justifications

- `impact:message-loss`: The issue is about proactive Telegram messages landing outside the expected forum topic and becoming effectively hard to find.

## Summary

Keep open: current main and the latest release still do not provide a Telegram config-level outbound topic default, and the existing send path only preserves an explicit or context-derived thread id. The live issue comment shows the remaining proactive-send failure mode, so this is not safe cleanup-close material.

## What This Changes

Request a per-agent Telegram topic/thread default so proactive outbound sends can target the configured forum topic without every call passing `threadId`.

## Best Possible Solution

Add a Telegram-owned outbound default topic contract with deterministic precedence: explicit thread first, active same-chat context second, configured per-agent fallback last, plus duplicate validation, docs, and regression coverage.

## Reproduction Assessment

Yes, at source level: current topic schema/type reject the requested config shape, and outbound thread resolution has no configured per-agent fallback. The live comment adds a real v2026.5.18 Telegram forum-topic send that succeeded without a thread and was hard to find.

## Solution Assessment

Unclear until maintainers choose the public config shape and ambiguity policy. The maintainable direction is a Telegram plugin-owned config contract and precedence rules, not a broad core default or silent override of explicit/current thread context.
