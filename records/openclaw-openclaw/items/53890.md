## Summary

Keep open: current main and the latest release still lack the requested Telegram config-level outbound topic default, while existing code only inherits a topic from active same-context tool metadata. This remains a Telegram product/config decision rather than conservative cleanup.

## What This Changes

The issue requests a per-agent Telegram default outbound forum topic/thread binding so proactive sends can target a configured topic without passing threadId each time.

## Best Possible Solution

Keep the current active-topic inheritance path, then add a narrow Telegram-owned outbound-default contract only if maintainers approve the config surface, precedence rules, docs, and regression coverage.

## Reproduction Assessment

Yes for the missing config surface: current main has no accepted outbound topic binding field and strict topic config rejects unknown fields. I did not live-reproduce the older nested proactive-send path on current main.

## Solution Assessment

Unclear: the requested outbound flag is plausible, but the repository has a high bar for new config/default surfaces. Maintainers should decide whether the remaining no-context proactive case needs a config default, a warning/fail-closed path, or a different session-binding design.
