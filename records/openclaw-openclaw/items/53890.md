## Summary

Keep open: current main and v2026.6.8 still support explicit Telegram topic targets and active-context thread inheritance, but they do not implement the requested config-level per-agent outbound topic default. The remaining work is a Telegram config/product contract decision, not conservative cleanup.

## What This Changes

The issue requests a per-agent Telegram config default that binds proactive outbound sends to a forum topic/thread without requiring each send call to pass `threadId`.

## Best Possible Solution

Keep the current active-topic inheritance path, then add a narrow Telegram-owned outbound-default contract only if maintainers approve the config surface, precedence rules, docs, and regression coverage.

## Reproduction Assessment

Yes for the missing config surface: current main and v2026.6.8 have no outbound topic binding field in the strict Telegram topic schema. I did not live-reproduce the older nested proactive-send path on current main.

## Solution Assessment

Unclear: the requested `outbound: true` flag is plausible, but current code already has active-context inheritance and repository policy sets a high bar for new config/default surfaces. Maintainers should decide whether the remaining no-context proactive case needs a config default, a warning/fail-closed path, or a different session-binding design.
