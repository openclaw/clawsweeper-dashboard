## Summary

Keep open: current main still has inbound per-topic Telegram agent routing and current-context auto-threading, but it does not provide a config-level per-agent outbound topic default for proactive sends. Recent live issue discussion on v2026.5.18 also reports the no-thread proactive-send path as still observable.

## What This Changes

The issue requests a Telegram configuration-level default outbound topic/thread binding per agent so proactive sends can target the expected forum topic without each call passing a threadId.

## Best Possible Solution

Add an additive Telegram-owned outbound default topic contract only after maintainers choose the config shape, duplicate-default validation, and precedence order: explicit target/threadId first, active topic context second, configured per-agent fallback last.

## Reproduction Assessment

Yes, at source level: current main has no outbound topic field in the Telegram topic type/schema, strict topic validation rejects unknown fields, and the auto-thread resolver only uses current tool context rather than configured per-agent defaults. The latest issue comment adds a live v2026.5.18 observation, but I did not run a live Telegram repro in this read-only review.

## Solution Assessment

Unclear until maintainers choose the public config contract and conflict behavior. The narrow maintainable direction is Telegram-owned policy plus the smallest outbound threading seam needed to see the active agent without changing explicit target/threadId precedence.
