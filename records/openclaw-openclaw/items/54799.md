## Summary

Keep this open. Current main still does not implement declarative `SKILL.md` trigger metadata, scheduler reconciliation from skill metadata, event-triggered skill launch, last-triggered skill status, or a skills audit command; the request crosses core skill loading, cron, lifecycle events, CLI status, and security policy, so it needs maintainer product/API review rather than conservative cleanup.

## What This Changes

Request optional `SKILL.md` frontmatter metadata for scheduled and event-driven skill triggers, automatic scheduler registration, last-triggered status in skills output, and a skills audit command.

## Best Possible Solution

Define a versioned `metadata.openclaw.triggers` contract for skills, validate schedule and event trigger forms, reconcile approved schedule triggers into owned cron jobs, define an approved lifecycle-event vocabulary, persist last/next trigger state, and expose that state through `openclaw skills list`, `openclaw skills info`, and a skills audit command.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature gap. The current-main verification path is high-confidence source/docs inspection showing the metadata type, resolver, status report, CLI, and docs do not expose the requested trigger contract.

## Solution Assessment

Unclear until maintainer design is settled. Declarative metadata is a plausible direction, but the safest solution needs a narrow, versioned, opt-in contract rather than letting arbitrary `SKILL.md` content schedule or event-trigger runs by default.
