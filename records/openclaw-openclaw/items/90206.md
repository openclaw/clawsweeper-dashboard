## Summary

Keep open: the PR targets a real Bedrock AWS SDK compaction bug, but the fallback veto still misses credential failures that are wrapped after a partial chunk succeeds, and the supplied proof is source-runtime only rather than live AWS SDK/Bedrock compaction proof.

## What This Changes

The PR adds `authMode: "aws-sdk"` request-auth provenance, lets the compaction safeguard proceed without static credentials for that mode, and adds a caller-controlled fallback veto for AWS SDK credential failures.

## Best Possible Solution

Carry AWS SDK credential-failure classification through every summarize/chunk/fallback path, then land only after redacted live Bedrock AWS SDK compaction proof or explicit maintainer acceptance of the remaining provider-level proof gap.

## Reproduction Assessment

Yes, at source level. Current main can return successful Bedrock `aws-sdk` request auth without `apiKey` or `headers`, and the current compaction safeguard cancels that shape; I did not run a live AWS_PROFILE/STS Bedrock repro.

## Solution Assessment

No, not as submitted. Carrying request-auth provenance is the right boundary, but the fallback veto must also see credential failures hidden by the partial-summary wrapper before this is the best fix.
