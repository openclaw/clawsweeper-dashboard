## Summary

Keep open: the PR takes the right provenance-based approach, but it currently lets `aws-sdk` auth bypass the no-credential guard even when the AWS SDK credential chain later fails, which can turn an auth failure into a successful fallback compaction instead of preserving history.

## What This Changes

The branch adds `authMode: "aws-sdk"` to `ResolvedRequestAuth`, has `ModelRegistry.getApiKeyAndHeaders()` return that provenance, and lets the compaction safeguard proceed for that auth mode without static API key or header material.

## Best Possible Solution

Keep the registry-to-safeguard `aws-sdk` provenance, but make AWS credential/signing failures cancel compaction rather than produce fallback summaries, then prove the happy path with redacted live AWS SDK credential-chain output.

## Reproduction Assessment

Yes. Source inspection shows current main can resolve Bedrock `auth: "aws-sdk"` as successful request auth with no `apiKey` or `headers`, and the current safeguard cancels that shape before compaction.

## Solution Assessment

No, not yet. Carrying explicit `aws-sdk` provenance is the right owner-boundary fix, but the patch also needs to keep credential/signing failures from becoming successful fallback compactions.
