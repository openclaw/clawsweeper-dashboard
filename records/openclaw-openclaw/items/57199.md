## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Widen the wake contract before forwarding agent IDs:** `src/gateway/server-methods/cron.ts:47-49`
  - body: The handler starts passing `agentId` into `context.cron.wake`, but current main's `CronServiceContract.wake` only accepts `mode`, `text`, and optional `sessionKey`. Refresh the service contract and generated/adapted surfaces before forwarding this field across the boundary.
  - confidence: 0.9
- **[P1] Preserve session-keyed next-heartbeat wakes:** `src/cron/service/timer.ts:1248-1256`
  - body: This timer shape only requests a heartbeat when `mode === "now"`. Current main intentionally treats `next-heartbeat` plus `sessionKey` as an immediate targeted wake; otherwise a session-keyed event can remain queued in a non-main session that the regular heartbeat never peeks.
  - confidence: 0.88
- **[P2] Keep wake params validation forward-compatible:** `src/gateway/protocol/schema/agent.ts:137-138`
  - body: The PR adds typed wake fields on top of an older closed `WakeParamsSchema` and keeps `sessionKey` as a plain string. Current main rejects empty session keys and keeps unknown wake metadata valid, so a refreshed schema should add `agentId` without losing those constraints.
  - confidence: 0.84
- **[P2] Regenerate wake clients for agent IDs:** `src/gateway/protocol/schema/agent.ts:137`
  - body: Adding a typed protocol field requires generated client follow-through. Current main's generated Swift `WakeParams` model only exposes `mode`, `text`, and `sessionkey`, so native clients and protocol checks would not have the new `agentId` surface.
  - confidence: 0.82
