## Summary

Keep open. Current main has the newer `--session-key` wake path but still does not expose `--agent-id`, so the PR addresses real remaining work; however the branch is stale against current wake semantics, has blocking correctness issues, and still lacks required real behavior proof.

## What This Changes

The PR adds `--agent-id` and `--session-key` to `openclaw system event` and threads those fields through wake schema validation, gateway cron handling, cron service wake plumbing, timer wake behavior, and CLI tests.

## Best Possible Solution

Refresh the PR or replace it with one additive wake-target contract update that preserves current sessionKey semantics while adding agentId end to end with generated protocol/docs and focused regression coverage.

## Reproduction Assessment

Yes. Source inspection on current main shows `--session-key` exists but `--agent-id` does not, and comparing the PR diff to current WakeParamsSchema, CronServiceContract, timer wake behavior, and tests gives a high-confidence source reproduction of the blockers.

## Solution Assessment

No. The direction is useful, but this exact patch is not the best merge shape because it is based on older wake semantics, omits the current public contract updates, and does not preserve targeted session wake behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Update the wake contract before passing agent IDs:** `src/gateway/server-methods/cron.ts:47-49`
  - body: The handler starts passing `agentId` through `context.cron.wake`, but the current cron contract only accepts `mode`, `text`, and optional `sessionKey`. A refreshed patch needs to widen the public cron contract and all adapters together before sending this field across the boundary.
  - confidence: 0.9
- **[P1] Preserve targeted session wakes for next-heartbeat:** `src/cron/service/timer.ts:1248-1256`
  - body: This timer shape only requests a wake when `mode === "now"`. Current main intentionally treats `next-heartbeat` plus `sessionKey` as an immediate targeted wake; otherwise a session-keyed event can sit in a non-main session queue that the regular heartbeat never peeks.
  - confidence: 0.88
- **[P2] Keep wake params forward-compatible:** `src/gateway/protocol/schema/agent.ts:137-138`
  - body: The PR adds typed wake fields on top of an older closed `WakeParamsSchema`. Current main has coverage requiring unknown wake metadata to remain valid, so the refreshed schema must keep `additionalProperties: true` while adding typed fields.
  - confidence: 0.84
- **[P3] Reject blank session keys at the schema boundary:** `src/gateway/protocol/schema/agent.ts:138`
  - body: `sessionKey` is declared as `Type.String()`, so direct RPC callers can send an empty string that validates and is later normalized away. Match current main's `NonEmptyString` behavior for this typed wake field.
  - confidence: 0.81
