## Summary

Keep open. Current main still lacks the requested `openclaw system event` targeting flags, but this PR is not merge-ready because the gateway cron contract is not updated, the `--session-key`-only path can enqueue under a different agent than the heartbeat wake, and the branch needs to be refreshed against current wake-schema/docs expectations.

## What This Changes

The PR adds `--agent-id` and `--session-key` to `openclaw system event`, extends wake validation and cron handling to carry those optional targets, forwards them through CronService wake/timer paths, and adds CLI forwarding tests.

## Best Possible Solution

Land a refreshed focused implementation that adds the CLI flags, updates the public cron wake contract, preserves wake metadata forward compatibility, resolves the wake target once for enqueue plus heartbeat, and documents the new operator-facing behavior with regression coverage.

## Reproduction Assessment

Yes. The contract failure is reproducible by source inspection: the PR passes extra fields through `context.cron.wake` while `CronServiceContract.wake` still accepts only `{ mode, text }`. The session-key-only routing bug is also reproducible by tracing `timer.wake` into `buildGatewayCronService` with a key like `agent:coding:discord:direct:123` and no `agentId`.

## Solution Assessment

No. The requested direction is valid, but the proposed patch is not the best merge shape until it updates the typed contract and uses one shared target resolver so enqueue and heartbeat wake cannot diverge.

Full review comments:

- **[P1] Update the cron wake contract before passing targets:** `src/gateway/server-methods/cron.ts:43-49`
  - body: The handler now passes `agentId` and `sessionKey` to `context.cron.wake`, but `context.cron` is typed as `CronServiceContract` and that contract still accepts only `{ mode, text }`. This object literal will fail the gateway typecheck unless the public cron wake contract is extended with the optional target fields.
  - confidence: 0.93
- **[P1] Resolve the session-key target before enqueueing:** `src/cron/service/timer.ts:1244-1247`
  - body: The PR advertises `--session-key` without `--agent-id`, but `timer.wake` forwards only the raw session key to enqueue. `buildGatewayCronService` resolves enqueue through `opts.agentId` and defaults to `main`, while heartbeat wake can derive `coding` from the same session key, so the event and wake can target different sessions.
  - confidence: 0.91
- **[P2] Keep wake params forward compatible:** `src/gateway/protocol/schema/agent.ts:136-138`
  - body: Current main intentionally accepts unknown wake metadata and has protocol tests for that behavior. When adding typed `agentId` and `sessionKey`, keep `additionalProperties: true`; otherwise external wake senders that attach opaque metadata will start failing validation after this stale branch is rebased.
  - confidence: 0.84
- **[P3] Reject blank session keys in wake params:** `src/gateway/protocol/schema/agent.ts:138`
  - body: `sessionKey` becomes a typed wake parameter but uses `Type.String()`, so direct RPC callers can send `""`, pass validation, and then have the value dropped by the falsy guards. Use the same non-empty validation style as `agentId` while keeping CLI whitespace normalization.
  - confidence: 0.79
- **[P3] Document the new system event targets:** `src/cli/system-cli.ts:60-62`
  - body: These are user-facing CLI flags, but the docs still say `system event` always enqueues on the main session and list no target flags. Update `docs/cli/system.md` and add the required user-facing changelog entry so users can discover the new behavior.
  - confidence: 0.76
