## Summary

Keep open. Current main has implemented the session-key-targeted wake path, but it still does not expose `openclaw system event --agent-id`, and this PR is now stale/conflicting relative to the current wake contract and session-key behavior.

## What This Changes

This PR adds `--agent-id` and `--session-key` to `openclaw system event` and threads those fields through wake validation, cron handling, timer wake logic, and CLI tests.

## Best Possible Solution

Refresh the PR against current main, preserve the shipped `--session-key` wake behavior, and either add `--agent-id` through the current wake contract and target resolver or intentionally keep session keys as the supported targeting API.

## Reproduction Assessment

Yes. Source inspection shows current main supports `--session-key` but not `--agent-id`, and the stale PR head can be traced to contract and wake-behavior mismatches without executing the CLI.

## Solution Assessment

No. The direction is useful, but this PR is not the best merge shape until it is rebased onto the current session-key implementation, updates the public wake contract coherently, and preserves forward-compatible schema behavior.

  - body: The handler passes `agentId` to `context.cron.wake`, but the current `CronServiceContract.wake` accepts only `mode`, `text`, and optional `sessionKey`.   - confidence: 0.88

  - confidence: 0.84

  - body: `sessionKey` is introduced as a typed wake parameter but uses `Type.String()`, so direct RPC callers can send an empty string that validates and is later dropped by normalization. Match current main's `NonEmptyString` behavior for this field.
  - confidence: 0.81
