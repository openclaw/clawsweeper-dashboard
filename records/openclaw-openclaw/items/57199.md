## Summary

Keep open: current main and the latest release still lack targeted system-event wake, but this branch is not mergeable as-is because it misses the cron service contract update, can split session-key-only enqueue and heartbeat targets, and lacks after-fix real behavior proof.

## What This Changes

This PR adds `--agent-id` and `--session-key` to `openclaw system event`, extends wake params and cron wake forwarding, and adds CLI forwarding tests.

## Best Possible Solution

Land a refreshed additive wake-target implementation that updates the public wake contract, CLI, Gateway schema and handler, CronServiceContract, service/timer target resolution, docs, and regression coverage while preserving default main-session behavior.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main accepts only mode/text, and the PR branch's session-key-only path can be traced to enqueue resolution by missing `agentId` while heartbeat can derive the session-key agent.

## Solution Assessment

No. The direction is useful, but this patch is not the best merge shape until it widens the public cron contract and resolves one canonical target for both enqueue and heartbeat.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Update the wake contract before passing targets:** `src/gateway/server-methods/cron.ts:43-49`
  - body: The handler now passes `agentId` and `sessionKey` to `context.cron.wake`, but `context.cron` is typed as `CronServiceContract` and that contract still accepts only `{ mode, text }`. A refreshed patch needs to widen `src/cron/service-contract.ts` first, or the gateway typecheck rejects this object literal.
  - confidence: 0.93
- **[P1] Resolve one target before enqueueing and waking:** `src/cron/service/timer.ts:1244-1247`
  - body: Forwarding a `sessionKey` without an `agentId` sends the raw session key to enqueue. Current Gateway cron enqueue resolution defaults from `opts.agentId`, while heartbeat wake can derive an agent from the same session key, so the event can land on the default agent while another agent is woken.
  - confidence: 0.91
- **[P2] Preserve forward-compatible wake metadata:** `src/gateway/protocol/schema/agent.ts:137-138`
  - body: Current main intentionally allows unknown wake params and has protocol tests for opaque external metadata. When adding typed target fields, keep `additionalProperties: true`; otherwise existing external wake senders that attach metadata can start failing validation.
  - confidence: 0.84
- **[P3] Reject blank session keys at the schema boundary:** `src/gateway/protocol/schema/agent.ts:138`
  - body: `sessionKey` becomes a typed wake parameter but uses plain `Type.String()`, so direct RPC callers can send an empty string, pass validation, and then have it dropped by falsy guards. Use the same non-empty validation style as `agentId` while keeping CLI whitespace normalization.
  - confidence: 0.79
