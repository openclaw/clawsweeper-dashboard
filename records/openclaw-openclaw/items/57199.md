## Summary

Keep open. The requested multi-agent wake targeting is still missing on current main and related issue #46886 remains open, but this PR is not merge-ready because it passes target fields through an unextended cron contract and leaves `--session-key`-only routing able to split enqueue and heartbeat targets.

## What This Changes

The PR adds `--agent-id` and `--session-key` to `openclaw system event`, threads them through wake schema and cron wake handling, and adds CLI forwarding tests.

## Best Possible Solution

Land a refreshed focused implementation that adds the CLI flags, extends the typed wake contract, resolves one canonical target for both enqueue and heartbeat, and documents the new operator-facing behavior with regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: current main lacks the flags entirely, and the PR branch's `sessionKey`-only path can be traced from `timer.wake` into `buildGatewayCronService` where enqueue and heartbeat resolve targets differently.

## Solution Assessment

No. The requested direction is valid, but this patch is not the best merge shape until it updates the public cron contract and shares target resolution so enqueue and heartbeat cannot diverge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Update the cron wake contract before passing targets:** `src/gateway/server-methods/cron.ts:43-49`
  - body: The handler now passes `agentId` and `sessionKey` to `context.cron.wake`, but `context.cron` is typed as `CronServiceContract` and that contract still accepts only `{ mode, text }`. This will fail the gateway typecheck unless the public cron wake contract is extended with the optional target fields.
  - confidence: 0.93
- **[P1] Resolve the session-key target before enqueueing:** `src/cron/service/timer.ts:1244-1247`
  - body: The advertised `--session-key`-without-`--agent-id` flow forwards only the raw session key to enqueue. Current gateway cron enqueue resolution defaults through `opts.agentId`, while heartbeat wake can derive the agent from the same session key, so the event can land on `main` while waking another agent.
  - confidence: 0.91
- **[P2] Keep wake params forward compatible:** `src/gateway/protocol/schema/agent.ts:136-138`
  - body: Current main intentionally accepts unknown wake metadata and has protocol tests for that behavior. When adding typed `agentId` and `sessionKey`, keep `additionalProperties: true`; otherwise stale-branch updates can reject existing external wake senders that attach opaque metadata.
  - confidence: 0.84
- **[P3] Reject blank session keys in wake params:** `src/gateway/protocol/schema/agent.ts:138`
  - body: `sessionKey` becomes a typed wake parameter but uses plain `Type.String()`, so direct RPC callers can send `""`, pass validation, and then have the value dropped by falsy guards. Use the same non-empty validation style as `agentId` while keeping CLI whitespace normalization.
  - confidence: 0.79
- **[P3] Document the new system event targets:** `src/cli/system-cli.ts:60-62`
  - body: These are user-facing CLI flags, but the docs still say `system event` enqueues on the main session and list no target flags. Update the system CLI docs and add the required changelog entry so operators can discover the new behavior.
  - confidence: 0.76
