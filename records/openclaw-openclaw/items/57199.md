## Summary

Keep open. The remaining `agentId` wake-target capability is still not implemented on current `main`, but this branch is stale against the merged `sessionKey` wake work and would regress current wake contract behavior unless refreshed.

## What This Changes

The PR adds `--agent-id` and `--session-key` to `openclaw system event` and forwards those fields through wake validation, gateway cron handling, and cron wake execution.

## Best Possible Solution

Refresh this PR or replace it with one additive wake-target update that adds `agentId` end to end while preserving current `sessionKey` validation, forward compatibility, and targeted wake semantics.

## Reproduction Assessment

Yes. Source inspection on current main shows `--agent-id` is absent while `--session-key` is already implemented, and comparing the PR diff to the current wake schema, service contract, and timer behavior gives a high-confidence source reproduction of the merge blockers.

## Solution Assessment

No. The direction is useful, but this exact patch is not the best merge shape because it is based on older wake semantics, omits the current public contract updates, and would regress targeted session-key wake behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Update the wake contract before passing agent IDs:** `src/gateway/server-methods/cron.ts:47-49`
  - body: The handler starts passing `agentId` into `context.cron.wake`, but the current public `CronServiceContract.wake` only accepts `mode`, `text`, and optional `sessionKey`. A refreshed patch needs to widen the contract and all adapters before sending this field across the boundary.
  - confidence: 0.9
- **[P1] Preserve targeted session wakes for next-heartbeat:** `src/cron/service/timer.ts:1248-1256`
  - body: This timer shape only requests a heartbeat when `mode === "now"`. Current main intentionally treats `next-heartbeat` plus `sessionKey` as an immediate targeted wake; otherwise a session-keyed event can sit in a non-main session queue that the regular heartbeat never peeks.
  - confidence: 0.88
- **[P2] Keep wake params forward-compatible:** `src/gateway/protocol/schema/agent.ts:137-138`
  - body: The PR adds typed wake fields on top of an older closed `WakeParamsSchema` and uses a plain string for `sessionKey`. Current main keeps unknown wake metadata valid and rejects empty session keys, so the refreshed schema should add `agentId` without losing those constraints.
  - confidence: 0.84
