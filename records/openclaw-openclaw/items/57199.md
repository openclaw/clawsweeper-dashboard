## Summary

Keep open. The requested agent targeting is still not implemented on current main and the related report at https://github.com/openclaw/openclaw/issues/46886 remains open, but this branch is stale against newer session-key wake semantics, is conflicting, and lacks real behavior proof.

## What This Changes

The PR adds `--agent-id` and `--session-key` to `openclaw system event`, then threads those values through wake validation, gateway cron handling, cron service/timer forwarding, and CLI tests.

## Best Possible Solution

Refresh against current main, preserve the shipped `sessionKey` wake semantics, and add `agentId` only through the typed wake contract, adapters, generated protocol surfaces, docs, and regression tests if maintainers want this public CLI/API surface.

## Reproduction Assessment

Yes. Source inspection on current main shows `--session-key` exists but `--agent-id` does not, and the PR diff can be compared against the current wake schema, contract, timer behavior, and tests to reproduce the stale merge blockers.

## Solution Assessment

No. The direction is useful, but this branch is not the best merge shape because it is based on older wake semantics and needs a contract-aware refresh plus real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Update the wake contract before forwarding agent IDs:** `src/gateway/server-methods/cron.ts:47-49`
  - body: The handler starts passing `agentId` into `context.cron.wake`, but current main's `CronServiceContract.wake` only accepts `mode`, `text`, and optional `sessionKey`. A refreshed patch needs to widen that contract and every adapter/generated surface together, or this boundary stays inconsistent.
  - confidence: 0.9
- **[P1] Derive the agent when only a session key is supplied:** `src/cron/service/timer.ts:1248-1250`
  - body: The PR claims `--session-key` can work without `--agent-id`, but forwarding only `sessionKey` into the enqueue path can resolve the event against the default agent while the heartbeat targets the keyed session. Preserve current main's resolver behavior that derives the agent from agent-prefixed session keys before enqueueing or waking.
  - confidence: 0.86
- **[P1] Preserve targeted session wakes for next-heartbeat:** `src/cron/service/timer.ts:1250-1252`
  - body: Current main intentionally treats `next-heartbeat + sessionKey` as an immediate targeted wake because the regular heartbeat watches the agent main session and would not peek the queued session event. This timer branch only wakes on `mode === "now"`, so the default mode can strand session-keyed events.
  - confidence: 0.88
- **[P2] Keep wake params forward-compatible:** `src/gateway/protocol/schema/agent.ts:137-138`
  - body: The PR adds typed wake fields on top of an older schema that still has `additionalProperties: false`. Current main has protocol coverage requiring unknown wake metadata to remain accepted, so the refreshed schema must keep that forward-compatible contract.
  - confidence: 0.84
- **[P3] Reject blank session keys at the schema boundary:** `src/gateway/protocol/schema/agent.ts:138`
  - body: `sessionKey` is added as `Type.String()`, allowing direct RPC callers to send an empty string that validates and is later dropped. Match current main's `NonEmptyString` behavior for this field.
  - confidence: 0.81
