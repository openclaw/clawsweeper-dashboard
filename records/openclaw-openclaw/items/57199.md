## Summary

Keep open. The missing `--agent-id` targeting is still not implemented on current main, but this PR is stale against the newer `--session-key` wake contract and would regress targeted session wake behavior unless refreshed.

## What This Changes

The PR adds `--agent-id` and `--session-key` flags to `openclaw system event` and forwards both through wake schema validation, gateway cron handling, cron service wake plumbing, timer wake behavior, and CLI tests.

## Best Possible Solution

Refresh the PR against current main, preserve the shipped sessionKey wake semantics, and add agentId only through the current wake contract, gateway adapters, generated protocol surfaces, docs, and regression tests if maintainers want that CLI/API surface.

## Reproduction Assessment

Yes. Source inspection shows current main supports `--session-key` but not `--agent-id`, and the PR diff can be compared directly against the current wake schema, contract, timer behavior, and tests to identify the stale merge blockers.

## Solution Assessment

No. The direction is useful, but this exact patch is not the best merge shape because it is based on an older wake contract and does not preserve current session-key targeted wake behavior or forward-compatible wake metadata.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Update the wake contract before passing agent IDs:** `src/gateway/server-methods/cron.ts:47-49`
  - body: The handler passes `agentId` to `context.cron.wake`, but the current gateway context is typed as `CronServiceContract`, whose `wake` method accepts only `mode`, `text`, and optional `sessionKey`. A refreshed patch needs to widen the public cron contract and every adapter together, or avoid passing an unsupported field through this boundary.
  - confidence: 0.9
- **[P1] Preserve targeted session wakes for next-heartbeat:** `src/cron/service/timer.ts:1244-1252`
  - body: With this timer shape, `--session-key` plus the default `next-heartbeat` mode only enqueues the event and does not request the targeted immediate wake that current main intentionally performs. The regular heartbeat watches the agent main session, so this can strand session-keyed events instead of delivering them.
  - confidence: 0.88
- **[P2] Keep wake params forward-compatible:** `src/gateway/protocol/schema/agent.ts:137-138`
  - body: The PR adds typed wake fields on top of an older `WakeParamsSchema` that still has `additionalProperties: false`. Current main has coverage requiring unknown wake metadata to remain accepted, so the refreshed schema must keep that forward-compatible contract while adding any typed fields.
  - confidence: 0.84
- **[P3] Reject blank session keys at the schema boundary:** `src/gateway/protocol/schema/agent.ts:138`
  - body: `sessionKey` is added as a typed wake parameter using `Type.String()`, so direct RPC callers can send an empty string that validates and is later dropped by normalization. Match current main's `NonEmptyString` behavior for this field.
  - confidence: 0.81
