## Summary

Keep open. Current main still routes Windows node install/start through the shared Scheduled Task implementation that targets `node.cmd` directly, and two open PRs now explicitly reference this issue with closing syntax, so the issue should stay open until a fix PR lands or is closed.

## What This Changes

The issue reports that `openclaw node install` and `openclaw node start` on Windows leave a persistent visible command window even though the node service is documented as a background service.

## Best Possible Solution

Land one focused Windows node launcher fix that hides the node Scheduled Task/start path while preserving service status, stop/restart, and Startup-folder fallback behavior, then close the issue from the merged PR.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: current main still installs `OpenClaw Node` as `node.cmd` through the interactive Scheduled Task path, and `openclaw node start` reaches the same restart/run path; only live Windows sign-in proof is still missing.

## Solution Assessment

Yes. The narrow maintainable solution is to hide the Windows node Scheduled Task launcher in `src/daemon/schtasks.ts`/node service plumbing, not to add a user option or rewrite all Windows service registration policy.
