## Summary

Keep open. Current main still routes Windows node service installs through the shared Scheduled Task path that runs `node.cmd` directly, while the node docs promise a background service; the linked draft fix PR should be reviewed or closed before this issue is resolved.

## What This Changes

Report that Windows `openclaw node install` and `openclaw node start` leave a persistent visible `openclaw-node` command window instead of running as a background service.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/81267 after real Windows Scheduled Task proof, then close this issue; if that PR is abandoned, implement the same narrow hidden-launcher fix in the Windows node service path.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes, source inspection on current main shows node installs bind `OpenClaw Node` to `node.cmd` and the shared Windows Scheduled Task create path runs that script directly; a live Windows sign-in run remains the final runtime proof.

## Solution Assessment

Is this the best way to solve the issue? Yes, the narrow hidden-launcher direction in the linked PR is the maintainable fix boundary; the issue should stay open until that PR is no longer draft and has real Windows Scheduled Task proof or is replaced.
