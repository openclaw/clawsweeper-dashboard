## Summary

Keep open. Current main still implements the reported Windows node service path as an interactive Scheduled Task targeting `node.cmd`, and two open PRs explicitly reference this issue with closing syntax, so the issue should remain open until a fix lands.

## What This Changes

The issue reports that `openclaw node install` on Windows registers `OpenClaw Node` to run `node.cmd`, leaving a persistent visible command window at login and when starting the node.

## Best Possible Solution

Land a node-only Windows hidden-launcher fix that preserves the existing service command/status/restart contract, prove it on a real Windows node install/sign-in or start flow, then close this issue as implemented with commit or release provenance.

## Reproduction Assessment

Yes. The reporter gave concrete Windows 11 steps and current main still maps the node service to `node.cmd` through the shared interactive Scheduled Task path; I did not run a live Windows login session.

## Solution Assessment

Yes. A scoped Windows scheduled-task launcher change in the daemon/node service surface is the narrow maintainable direction; this issue should wait for the linked implementation PR review and proof rather than close early.
