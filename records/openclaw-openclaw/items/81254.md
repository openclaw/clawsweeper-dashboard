## Summary

Keep open. Current main still installs the Windows node service as an interactive Scheduled Task targeting `node.cmd`, while the docs describe `openclaw node install` as a background user service; two open closing PRs now target this issue, so the issue should remain open until one lands.

## What This Changes

Windows `openclaw node install` should start the node host without leaving a persistent visible command window at login or during `openclaw node start`.

## Best Possible Solution

Land one node-specific hidden-launcher fix that preserves command parsing, status/start/stop/restart, and Startup-folder fallback behavior, then close this issue after the selected PR merges.

Yes. Source-level reproduction is high confidence: current main writes `node.cmd` directly into an interactive ONLOGON Scheduled Task, and the reporter supplied concrete Windows 11 output; I did not run a live Windows login in this read-only pass.

## Solution Assessment
