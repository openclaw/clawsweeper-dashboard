## Summary

Keep open. Current main still has separate Discord preview-streaming, block-streaming, and default progress-delivery paths, so the report is plausible, but the issue does not yet pin the misleading fragment to one exact runtime path or settled product policy.

## What This Changes

The issue asks OpenClaw to prevent Discord from emitting misleading mid-turn partial/block/progress fragments when streaming is disabled or configured for final-only output.

## Best Possible Solution

Add a focused Discord runtime regression that separates preview-off behavior, standalone tool progress, and block replies, then implement the chosen final-only policy without changing unrelated channel streaming semantics.

## Reproduction Assessment

No high-confidence live reproduction is available from the issue alone. Source inspection shows plausible current-main paths where Discord preview streaming is off but default progress messages or separately enabled block replies can still produce mid-turn channel output.

## Solution Assessment

Unclear. A narrow code change may be possible, but maintainers should first decide whether Discord `streaming.mode=off` means preview-only off or all mid-turn progress/block output off when other delivery controls are active.
