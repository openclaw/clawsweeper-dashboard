## Summary

Keep open. Current main still treats Discord preview streaming and normal block replies as separate paths, so the report is plausible, but the exact misleading-fragment behavior is not live-reproduced and the remaining fix depends on a maintainer policy decision about whether Discord `streaming.mode="off"` should also suppress separately configured block delivery.

## What This Changes

The issue asks OpenClaw to prevent Discord from emitting misleading mid-turn partial/block fragments when preview streaming is disabled or final-only output is intended.

## Best Possible Solution

Define the Discord final-only policy across preview streaming, block replies, and tool progress, then encode that choice in focused runtime coverage and docs.

## Reproduction Assessment

No high-confidence live reproduction is available. Source inspection shows a plausible current-main path where Discord preview streaming is off but block delivery can still emit mid-turn messages through the separate block-streaming defaults.

## Solution Assessment

Unclear. A narrow code fix may be possible, but maintainers should first decide whether Discord `streaming.mode="off"` means preview-only off or all mid-turn block/progress output off when block streaming is separately configured.
