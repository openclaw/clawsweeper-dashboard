## Summary

This PR should stay open: the linked silent Telegram/Gateway behavior is source-reproducible, but the proposed fallback is too broad and can send an extra source-channel `Done.` after a message tool has already delivered content. The external PR also still lacks required after-fix real behavior proof.

## What This Changes

The PR adds an empty-payload fallback in `src/auto-reply/reply/agent-runner.ts` that returns `Done.` when `runResult.meta?.toolSummary?.calls` is positive.

## Best Possible Solution

A corrected fix should acknowledge local tool-only side effects only when there is no text/media payload and no committed message-tool or other outbound delivery evidence, with focused regression coverage and redacted real Telegram or Gateway proof.

## Reproduction Assessment

Yes, source-level: current main returns `undefined` for an empty payload array after the run, matching the linked Telegram no-message report for tool-only turns. I did not run a live Telegram/Gateway reproduction in this read-only review.

## Solution Assessment

No. `toolSummary.calls > 0` is not the narrowest safe condition because message-send tools are also tool calls and already carry delivery evidence; the safer fix should gate the acknowledgement on absence of committed outbound delivery evidence or route through existing payload-building semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Skip the fallback after message-tool deliveries:** `src/auto-reply/reply/agent-runner.ts:1464-1468`

Gate the fallback on no committed message-tool delivery evidence, or route the acknowledgement through the existing delivery path.
