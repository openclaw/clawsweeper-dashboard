## Summary

Keep open: the reported silent tool-only turn is still source-reproducible on current main, but this PR's `toolSummary.calls > 0` fallback can duplicate message-tool deliveries and the external PR still lacks real Telegram/Gateway proof.

## What This Changes

This PR changes `runReplyAgent` to return `Done.` when an empty run result reports one or more tool calls.

## Best Possible Solution

A maintainable fix should acknowledge local tool-only side-effect turns only when no committed message-tool, block, cron, or approval delivery already occurred, with focused regression coverage and real Telegram/Gateway proof.

## Reproduction Assessment

Yes, source-level: current main still returns `undefined` for zero payloads after the run, which matches the linked Telegram no-message report. I did not establish a live Telegram/Gateway reproduction in this read-only review.

## Solution Assessment

No: `toolSummary.calls > 0` is too broad because message-tool sends are also tool calls and may already have delivered visible output. The safer fix is to gate the acknowledgement on absence of committed outbound delivery evidence or route it through the existing payload-building semantics.
