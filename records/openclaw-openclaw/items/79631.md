## Summary

Keep open: the reported silent tool-only turn is source-reproducible, but this PR's fallback is too broad and would add a source-channel `Done.` after successful message-tool deliveries; the PR also still lacks after-fix real Telegram/Gateway proof.

## What This Changes

The PR changes `src/auto-reply/reply/agent-runner.ts` to return `Done.` when an empty agent run reports one or more tool calls.

## Best Possible Solution

A maintainable fix should acknowledge local tool-only side effects only when no other outbound delivery occurred, preserve message-tool suppression evidence, add focused regression coverage, and include real Telegram/Gateway proof.

## Reproduction Assessment

Yes, source-level: current main returns `undefined` for an empty payload array after the run, which matches the linked Telegram no-message report. I did not run a live Telegram/Gateway reproduction in this read-only review.

## Solution Assessment

No: `toolSummary.calls > 0` is not the narrowest safe condition because message-tool sends are also tool calls and already carry delivery evidence for suppression. The safer fix is to gate the acknowledgement on the absence of committed message-tool or other outbound delivery evidence, or route it through the existing payload-building semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Skip the fallback after message-tool deliveries:** `src/auto-reply/reply/agent-runner.ts:1461-1465`
  - body: `hadToolCalls` is true for message tools as well as local side-effect tools. In this empty-payload branch the new return sends `Done.` before `buildReplyPayloads` can inspect `messagingToolSentTexts`, media URLs, or targets, so a run that already delivered content through a message tool can get an extra source-channel reply. Gate the fallback on no committed message-tool delivery evidence, or route the acknowledgement through the existing delivery path.
