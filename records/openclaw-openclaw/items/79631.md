## Summary

Keep open: this PR targets a real source-level silent tool-only reply bug, but the current diff can send an extra `Done.` after message-tool deliveries and still lacks required real Telegram/Gateway proof.

## What This Changes

Adds a `Done.` fallback in `runReplyAgent` when an empty payload run reports `toolSummary.calls > 0`.

## Best Possible Solution

Add a guarded acknowledgement for local tool-only side-effect turns only when no message-tool or outbound delivery already occurred, cover both acknowledgement and no-duplicate paths, and attach redacted real Telegram/Gateway proof.

## Reproduction Assessment

Yes, source-level: current main returns `undefined` for zero payloads and empty replies are normalized away, matching the linked Telegram no-message report. I did not run a live Telegram/Gateway reproduction in this read-only review.

## Solution Assessment

No: `toolSummary.calls > 0` is not the narrowest safe condition because message-send tools are also tool calls and may already have delivered content. The safer fix is to gate the acknowledgement on the absence of committed message-tool or other outbound delivery evidence, or route it through the existing delivery semantics.
