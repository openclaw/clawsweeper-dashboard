## Summary

## What This Changes

The PR adds a `Done.` fallback in `runReplyAgent` when `runResult.meta.toolSummary.calls` is positive and the agent run returns no reply payloads.

## Best Possible Solution

Use a policy-aware fallback that synthesizes an acknowledgment only when no visible or side-effect delivery already happened, automatic source delivery is allowed, and focused tests plus real Telegram proof cover automatic, message-tool-only, and already-sent cases.

## Reproduction Assessment

Yes, at source level: current main can reach the empty-payload branch and return undefined, while the linked issue describes the Telegram user path. I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

No for the current patch shape. It is a plausible mitigation for one automatic-delivery case, but the best fix must avoid duplicate side-effect acknowledgments and preserve or intentionally change the current `message_tool_only` contract.
