## Summary

Keep open: the linked bug is still source-reproducible on current main, but this PR's predicate is too broad and can add a duplicate `Done.` after a message tool already delivered output; it also still lacks required real Telegram/Gateway proof.

## What This Changes

The branch makes `runReplyAgent` return a `Done.` payload when an empty run result reports at least one tool call.

## Best Possible Solution

A safe fix should acknowledge local tool-only side-effect turns only when no message-tool, block, cron, approval, or other committed outbound delivery already occurred, with focused regression coverage and real Telegram/Gateway proof.

## Reproduction Assessment

Yes, source-level: current main normalizes empty replies away and returns `undefined` for zero payloads, matching the linked Telegram no-message path. I did not run a live Telegram/Gateway reproduction in this read-only review.

## Solution Assessment

No: `toolSummary.calls > 0` is too broad because messaging sends are tool calls with committed delivery evidence. The safer solution is to gate the acknowledgement on absence of committed outbound delivery or route it through the existing side-effect/dedupe semantics.
