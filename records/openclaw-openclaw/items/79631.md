## Summary

Keep open: the PR targets a real source-level silent-turn bug, but the current predicate is too broad and can add a duplicate `Done.` after a message tool already delivered output. The external PR also still lacks after-fix real Telegram/Gateway proof.

## What This Changes

The PR adds a `Done.` fallback in `runReplyAgent` when an empty reply result reports `toolSummary.calls > 0`.

## Best Possible Solution

Add a guarded acknowledgement for local tool-only side-effect turns only when no committed outbound/message-tool/block/cron/approval delivery already occurred, with focused regression coverage and real Telegram/Gateway proof.

## Reproduction Assessment

Yes, source-level: current main returns `undefined` for zero payloads, matching the linked Telegram no-message report. I did not establish a live Telegram/Gateway reproduction in this read-only review.

## Solution Assessment

No: `toolSummary.calls > 0` is too broad because message-tool sends are also tool calls and may already have delivered visible output. The safer fix is to gate the acknowledgement on absence of committed outbound delivery evidence.
