## Summary

Keep open: the linked Telegram bug is still source-reproducible on current main, but this PR should not merge as-is because `toolSummary.calls` also covers message-tool deliveries that may already have produced visible output, and no after-fix real Telegram/Gateway proof is present.

## What This Changes

The PR adds a `toolSummary.calls > 0` check in `runReplyAgent` and returns a `Done.` reply when an agent run produces no reply payloads after tool activity.

## Best Possible Solution

Add a guarded acknowledgement for local tool-only side-effect turns only when no committed outbound delivery evidence already exists, with focused regression coverage and real Telegram/Gateway proof.

## Reproduction Assessment

Yes, source-level: current main normalizes empty content away and returns `undefined` when `payloadArray` is empty, so a tool-only run with no final assistant text has no outbound reply unless another delivery path fires. I did not run a live Telegram/Gateway reproduction in this read-only review.

## Solution Assessment

No: `toolSummary.calls > 0` is too broad because message-tool sends are also tool calls and may already have delivered visible content. The safer fix should check for absence of committed outbound delivery evidence before adding an acknowledgement.
