## Summary

Keep open: the linked user problem is still not implemented on current main or the latest release, but this PR is not merge-ready because the `toolSummary.calls > 0` gate can emit an extra visible `Done.` after successful side-effect delivery and can change `message_tool_only` delivery policy without proof or maintainer approval.

## What This Changes

The PR adds a `Done.` acknowledgment in `runReplyAgent` when an agent run returns no payloads but reports one or more tool calls.

## Best Possible Solution

Use a policy-aware acknowledgment only when no visible or side-effect delivery already happened and the selected source-reply mode permits automatic visible delivery, then cover that choice with focused regression tests and real Telegram proof.

## Reproduction Assessment

Yes, at source level: current main can reach the empty-payload branch and return `undefined`, and the linked issue describes the Telegram user path. I did not establish a live Telegram reproduction in this read-only review.

## Solution Assessment

No for the current patch shape. It is a plausible mitigation, but the best fix must reuse existing delivery evidence and preserve or deliberately change the `message_tool_only` contract.
