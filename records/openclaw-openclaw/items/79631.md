## Summary

Keep open: current main still has the empty-payload no-delivery path, but this PR needs real behavior proof and has a duplicate-delivery edge case for message-tool turns.

## What This Changes

This PR changes `src/auto-reply/reply/agent-runner.ts` to return `Done.` when an agent run has no payloads but reports one or more tool calls.

## Best Possible Solution

Land a scoped fix that acknowledges local tool side effects only when no other outbound delivery occurred, preserves the existing message-tool suppression path, adds focused regression coverage, updates the changelog, and includes real Telegram/Gateway proof.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source-level, yes: current main returns `undefined` when `runResult.payloads` is empty after tool/task drain, matching the no-message path described in https://github.com/openclaw/openclaw/issues/79472; I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

Is this the best way to solve the issue? No: checking only `toolSummary.calls` is too broad because message-tool deliveries are tracked separately and the early return bypasses the existing dedupe/suppression path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Skip the fallback after message-tool deliveries:** `src/auto-reply/reply/agent-runner.ts:1461-1465`
  - body: `hadToolCalls` is true for message tools as well as local side-effect tools. In the empty-payload path this returns `Done.` before `buildReplyPayloads` can see `messagingToolSentTexts` or targets, bypassing the existing message-tool delivery suppression and adding an extra source-channel reply after a tool already sent outbound content. Gate the fallback on no committed message-tool delivery evidence, or route this through the existing delivery path.
  - confidence: 0.86
