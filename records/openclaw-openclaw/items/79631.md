## Summary

Keep open: the linked message-delivery problem still exists on current main and in the latest release, but this PR is not merge-ready because the fallback is keyed only on tool-call count, lacks real Telegram/live-runtime proof, and needs delivery-policy review before it can safely land.

## What This Changes

The PR adds a `Done.` fallback in `runReplyAgent` when an agent run reports tool calls but returns no reply payloads.

## Best Possible Solution

Use a policy-aware fallback: synthesize an acknowledgment only when no visible or side-effect delivery already happened, automatic source delivery is allowed, and focused tests plus real Telegram proof cover automatic, message-tool-only, and already-sent cases.

## Reproduction Assessment

Yes, from source: current main can reach `runReplyAgent` with an empty payload array and return undefined, and v2026.6.8 still has that path. The PR still lacks real Telegram proof for the user-visible transcript.

## Solution Assessment

No. The patch is a plausible mitigation for one automatic-delivery case, but the best fix must first avoid duplicate side-effect acknowledgments and preserve or intentionally change the current message-tool-only contract.
