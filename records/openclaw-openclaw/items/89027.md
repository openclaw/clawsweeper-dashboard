## Summary

Keep this PR open: the patch is a narrow, source-backed fix for the linked `claude-cli` empty-result fallback bug, but it still needs real Claude CLI behavior proof before merge because the supplied proof is only parser unit-test output.

## What This Changes

The PR adds a `completedTurn` signal to Claude stream-json CLI parsing and skips `empty_response` failover when a terminal result event completed with no assistant text.

## Best Possible Solution

Land the narrow completion-state guard after real Claude CLI proof, or maintainer live validation, shows a thinking-only completed turn no longer falls back to a different model while genuine empty subprocess output still fails.

## Reproduction Assessment

Yes from source inspection, though I did not run a live Claude CLI session. A blank Claude stream-json `type: "result"` currently parses to empty text and then trips the `empty_response` failover guard in `runPreparedCliAgent`.

## Solution Assessment

Yes, the proposed direction looks like the narrowest maintainable fix: carry completion state from the result parser to the fallback guard and keep uncompleted empty subprocess output as a failure. The remaining gap is live after-fix proof, not a different code shape.
