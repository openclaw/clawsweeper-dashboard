## Label Justifications

- `P3`: This is useful plugin-install ergonomics, but it is a speculative capability requiring product and security design rather than urgent regression repair.
- `impact:data-loss`: The proposed merge can overwrite persisted user settings and replace arrays, as demonstrated by review of the closed implementation PR.
- `impact:security`: Package-controlled configuration writes would cross a security boundary unless writable scope, consent, and sensitive-value handling are constrained.
- `impact:auth-provider`: The proposal explicitly includes provider selection and credential-like fields, so incorrect mutation can change authentication or routing behavior.

## Summary

The central capability is still missing on current main: manifests cannot declare an install-time configuration intent, and install persistence only updates plugin activation and install records. The linked implementation closed unmerged after concrete writable-scope and array-overwrite concerns, so the issue remains useful but requires a maintainer-owned product and security contract.

## What This Changes

Add a plugin-manifest contract that can propose and apply required OpenClaw configuration during plugin installation.

## Best Possible Solution

Extend the typed plugin setup contract with bounded configuration intents that use existing patch preview and validation machinery, require explicit operator approval, define per-field conflict and list semantics, avoid embedding secrets, and record provenance for reinstall and uninstall guidance.

## Reproduction Assessment

Not applicable; this is a request for a new plugin installation capability. Source inspection provides high-confidence confirmation that current manifests and install persistence do not expose that capability.

## Solution Assessment

No. The proposed unrestricted automatic deep merge is not the best solution; the maintainable path is a typed, scoped, previewable configuration-intent contract built on the current explicit patch and setup surfaces.
