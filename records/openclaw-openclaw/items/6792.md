## Summary

Keep open. Current main still lacks a manifest-level install-time `configPatch` contract, existing `configPatch` support is provider-auth-only, and the linked implementation PR is closed unmerged while the remaining design is product/security-sensitive.

## What This Changes

Request to let plugin manifests declare `configPatch` and have `openclaw plugins install` apply those config changes automatically with validation and visible install feedback.

## Best Possible Solution

Define a documented, auditable install-time plugin config intent contract that covers service, tool, provider, hook, and channel setup while validating before commit, showing an exact diff with a skip path, constraining allowed targets, and recording provenance for uninstall guidance across all install routes.

## Reproduction Assessment

Not applicable as a feature request rather than a bug reproduction. There is a high-confidence current-main verification path by inspecting the manifest type/loader, install result, install persistence, install callers, provider-auth prior art, and manifest docs.

## Solution Assessment

No. The raw automatic deep-merge proposal is not yet the best maintainable solution without guardrails; a constrained config-intent contract with validation, diff display, skip behavior, trust rules, array semantics, and provenance is safer.
