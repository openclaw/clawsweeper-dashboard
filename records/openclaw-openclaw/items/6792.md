## Summary

Keep open. Current main still lacks a manifest-level install-time config patch contract, and the remaining work is a product/security design decision rather than conservative cleanup.

## What This Changes

Request is to let plugin manifests declare an install-time `configPatch` that `openclaw plugins install` applies to `openclaw.json` automatically.

## Best Possible Solution

Define a documented, auditable install-time plugin config intent contract with allowed surfaces, validation before write, visible diff/skip behavior, conflict and array rules, install provenance, uninstall guidance, and coverage for all install routes.

## Reproduction Assessment

Not applicable as a feature request. Source inspection does give high confidence that current main lacks the requested manifest/install contract.

## Solution Assessment

No. The useful direction is clear, but the raw deep-merge manifest field proposed here is not the best final shape until OpenClaw defines the security, consent, conflict, array, and uninstall semantics.

## Security Review

Status: needs_attention

Summary: The requested contract is security-sensitive because third-party manifests would be able to influence persisted OpenClaw config during install.

Concerns:

- **[medium] Constrain install-time manifest config writes:**
  - body: A manifest-level patch mechanism needs explicit allowed surfaces, validation, user-visible diff/skip behavior, and provenance before it can safely write user config during plugin install.
  - confidence: 0.9
- **[medium] Avoid silent list replacement for allowlist fields:**
  - body: Using existing config patch array replacement for fields like `tools.alsoAllow` could drop previously configured plugin grants and change tool availability without a clear conflict prompt.
  - confidence: 0.86
