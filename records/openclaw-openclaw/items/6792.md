## Label Justifications

- `P3`: This is a useful but product-sensitive plugin install ergonomics enhancement, not an urgent broken runtime workflow.
- `impact:data-loss`: The requested feature would write user config during plugin install, and unresolved merge semantics could silently overwrite existing config entries.

- `impact:auth-provider`: The motivating examples and existing prior art involve provider/search routing, API-key-like fields, and provider auth config patch flows.

## Summary

Keep open. Current main still lacks a manifest-level install-time config patch contract, the direct implementation attempt is closed unmerged, and the remaining decision is the product/security shape for third-party config mutation rather than conservative cleanup.

## What This Changes

Request to let plugin manifests declare an install-time config patch that `openclaw plugins install` validates, applies, and reports automatically.

## Best Possible Solution

Define a documented, auditable plugin install config-intent contract with allowlisted surfaces, validation before write, visible diff and skip path, conflict and array rules, uninstall guidance, and tests across every install route.

## Reproduction Assessment

Yes at source level. A manifest containing `configPatch` would currently be ignored because manifests do not parse it, install results cannot carry it, and install persistence only records/enables plugins.

## Solution Assessment

No for the raw deep-merge proposal as written. The maintainable path is a constrained config-intent contract that keeps setup explicit, validates before commit, and records enough provenance for later review or uninstall guidance.
