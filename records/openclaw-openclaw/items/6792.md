## Label Justifications

- `P3`: This is a useful plugin-install ergonomics and API design request, but it is not an urgent shipped regression or broken core workflow.
- `impact:data-loss`: Automatic manifest patches can overwrite or drop existing user config values, especially list fields such as `tools.alsoAllow`, unless merge semantics are designed carefully.
- `impact:security`: Third-party install metadata mutating local operator config can affect tool, provider, and security posture, so the trust boundary is central to the issue.
- `impact:auth-provider`: The motivating examples include provider, model, and web search routing configuration that can influence provider selection and auth-adjacent behavior.

## Summary

Keep open. Current main and the latest release still lack a manifest-level install-time config patch contract, and the remaining work is a product/security decision about third-party manifests mutating operator config.

## What This Changes

Allow plugin manifests to declare install-time config changes that `openclaw plugins install` applies automatically instead of requiring manual `openclaw.json` edits.

## Best Possible Solution

Define a documented, auditable install-time plugin config intent contract with constrained allowed surfaces, validation before commit, exact diff plus skip path, conflict and array semantics, provenance for uninstall guidance, and coverage for every install route.

## Reproduction Assessment

Yes. Source inspection on current main and v2026.6.6 shows no manifest `configPatch` field, no install result payload for it, and no install persistence path that applies manifest-declared config changes.

## Solution Assessment

No for the raw deep-merge proposal as written. The best fix is a constrained config-intent contract with consent, validation, allowed-path policy, array semantics, and uninstall/audit provenance rather than silent free-form config writes.
