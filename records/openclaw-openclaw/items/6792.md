- `P3`: This is a low-urgency feature and product-contract proposal for plugin setup ergonomics, not a broken current workflow with an established fix contract.
- `impact:data-loss`: The requested behavior would write user config during plugin install, where merge semantics can preserve or overwrite existing `openclaw.json` values.

## Summary

Keep open: current main and the latest release still do not provide a manifest-level install-time config patch contract, and the closed implementation attempt at https://github.com/openclaw/openclaw/pull/43421 did not land. The claude-search-proxy example and the later tool-allowlist comment show real plugin setup friction, but the remaining work is the product and security design for third-party config mutation.

## What This Changes

This issue requests a plugin manifest field that lets `openclaw plugins install` apply validated config changes automatically instead of requiring users to edit `openclaw.json` by hand.

## Best Possible Solution

Define a documented, auditable install-time plugin config intent contract that validates before commit, shows an operator-visible diff with a skip path, records provenance for uninstall guidance, and covers conflict and array behavior across all install routes.

## Reproduction Assessment

Yes for the missing capability: current-main source inspection shows manifests do not expose `configPatch`, install results cannot carry it, and install persistence has no patch application path. This is a feature gap rather than a failing runtime repro.

## Solution Assessment

Unclear as proposed. Manifest-declared setup intent is plausible, but automatic raw deep-merge from third-party manifests is not the best final shape until maintainers define consent, conflict handling, allowed paths, validation, provenance, and uninstall guidance.
