Keep open. Current main still lacks a manifest-level install-time config mutation contract, the only direct implementation PR was closed unmerged, and the remaining question is a product/security contract for third-party config writes rather than a stale or already-implemented request.

## What This Changes

The issue requests a manifest-level `configPatch` field that plugin install would apply to user config automatically.

## Best Possible Solution

Define a documented, auditable install-time plugin config intent contract with allowed surfaces, validation before write, visible diff or skip behavior, conflict and array rules, provenance, uninstall guidance, and coverage for every install route.

## Solution Assessment

No, not as a raw arbitrary deep merge from third-party manifests. The maintainable path needs a constrained, auditable config intent contract with security, consent, conflict, array, and uninstall semantics defined first.
