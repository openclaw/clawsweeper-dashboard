## Label Justifications

- `P3`: This is a useful plugin-install ergonomics and API design request, but it is not an urgent broken workflow or shipped regression.
- `impact:data-loss`: Automatic manifest patches can overwrite or drop existing user config values, especially list fields such as `tools.alsoAllow`, unless merge semantics are designed carefully.
- `impact:security`: Third-party install metadata mutating local config can affect operator-controlled security posture and needs an explicit trust boundary.
- `impact:auth-provider`: The motivating examples include provider, model, and web search routing configuration that can influence provider selection and auth behavior.

## Summary

Keep open. Current main still does not expose a manifest-level install-time config patch contract, the only directly related implementation PR is closed unmerged, and the remaining work is a product/security decision about how much third-party manifests may mutate user config.

## What This Changes

Request to let plugin manifests declare install-time config changes that `openclaw plugins install` applies automatically instead of requiring users to edit `openclaw.json` manually.

## Best Possible Solution

Define a documented, auditable install-time plugin config intent contract that validates before commit, shows the exact diff with a skip path, limits or classifies allowed config surfaces, defines conflict and array semantics, records provenance for uninstall guidance, and covers every install route with regression tests.

## Reproduction Assessment

Yes for the missing capability: source inspection shows current main has no `configPatch` manifest field, no install result payload for it, and no install persistence path to apply it. This is a source-reproducible feature gap rather than a broken existing behavior contract.

## Solution Assessment

No for the raw proposal as written: an arbitrary manifest deep-merge during install is too broad without a trust, consent, and audit contract. The maintainable solution should be a constrained install-time config-intent API rather than silent free-form config writes.
