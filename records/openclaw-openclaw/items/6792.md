## Summary

Current main still lacks a manifest-level install-time plugin config intent path, and the request remains a real plugin ergonomics/product-security question rather than conservative cleanup. The direct implementation PR was closed unmerged, and the remaining decision is how third-party manifests may request user config changes safely.

## What This Changes

The issue requests an `openclaw.plugin.json` manifest field that can apply validated config changes during `openclaw plugins install`.

## Best Possible Solution

Define a documented, auditable plugin install config-intent contract with constrained surfaces, validation before write, visible diff and skip behavior, conflict/list rules, uninstall guidance, and tests across every install route.

## Reproduction Assessment

Yes, at source level. A manifest containing `configPatch` would currently be ignored because manifests do not parse it, install results cannot carry it, and install persistence only records/enables plugins.

## Solution Assessment

No for the raw deep-merge proposal as written. The maintainable solution is a constrained config-intent contract that keeps setup explicit, validates before commit, and records enough provenance for later review or uninstall guidance.
