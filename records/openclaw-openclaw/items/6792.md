## Summary

Keep open. Current main still lacks a manifest-level install-time config mutation contract, the direct implementation PR was closed unmerged, and the remaining decision is the product/security shape for third-party config writes rather than stale cleanup.

## What This Changes

The issue requests a manifest-level `configPatch` field that `openclaw plugins install` would apply to user config during plugin installation.

## Best Possible Solution

Define a documented, auditable install-time plugin config intent contract with allowed surfaces, validation before write, visible diff or skip behavior, conflict and array rules, provenance, uninstall guidance, and coverage for every install route.

## Reproduction Assessment

Not applicable as a feature request. Source and docs inspection show current main lacks the requested manifest/install contract, while provider-auth `configPatch` and manual `openclaw config patch` do not cover plugin install.

## Solution Assessment

No, not as an arbitrary deep merge from third-party manifests. The maintainable path needs a constrained, auditable config intent contract with security, consent, conflict, array, and uninstall semantics defined first.
