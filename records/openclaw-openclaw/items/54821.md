## Summary

Keep open: the capability is not implemented on current main, but the PR head is not merge-ready because it weakens reasoning visibility, conflicts with the merged raw-reasoning contract, and leaves pre-registered event grants alive on early dispatch failure.

## What This Changes

The branch adds a Gateway `thinking-events` client capability, cap-gated recipient routing for reasoning tokens, reasoning delta changes, gateway registration plumbing, tests, and generated API baseline updates.

## Best Possible Solution

Refresh or replace this with a design that decouples Gateway thinking events from the callback while preserving `thinkingLevel` visibility, raw core payloads, cap-gated delivery, and guaranteed recipient cleanup.

## Reproduction Assessment

Yes for the bug portion by source inspection: current main still requires `onReasoningStream` before emitting `stream:"thinking"`, while Gateway `chat.send` does not provide that callback. The cap-gated subscription behavior is new protocol surface rather than an existing-behavior reproduction.

## Solution Assessment

No. The direction may be useful, but this PR head is not the best merge path until it preserves visibility gates, keeps core thinking payloads raw, finalizes recipient grants on early failure, and gets maintainer agreement on the capability contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve reasoning visibility gates:** `src/agents/pi-embedded-subscribe.ts:77`
  - body: This sets streamed reasoning from `reasoningMode === "stream"` alone, dropping current main's `thinkingLevel` visibility guard. A client with the new cap could receive reasoning even when reasoning visibility is disabled, so the branch needs to keep the existing gate while decoupling the callback.
  - confidence: 0.9
- **[P2] Keep core thinking payloads raw:** `src/agents/pi-embedded-subscribe.ts:664-689`
  - body: Current main treats core reasoning `text` and `delta` as raw payloads and leaves Markdown formatting to channel edges. Reintroducing `formatReasoningMessage(...)` here would send formatted `Reasoning:` Markdown through the Gateway protocol and conflict with the merged raw-payload contract.
  - confidence: 0.88
- **[P2] Finalize pre-registered recipients on early failure:** `src/gateway/server-methods/chat.ts:1558`
  - body: `chat.send` now pre-registers thinking recipients before dispatch, but the early dispatch failure path only deletes the abort controller. If dispatch rejects before lifecycle events run, the run-to-connection grant survives until TTL and can accumulate or misroute if a run id is reused.
  - confidence: 0.76
