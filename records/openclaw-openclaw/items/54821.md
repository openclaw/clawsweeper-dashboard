## Summary

Keep this PR open, but do not merge it as-is. Current main still lacks cap-gated thinking subscriptions, so the work is not obsolete; the PR head still has blocking correctness and security issues around disabled thinking, delta-only chunks, and stale recipient cleanup.

## What This Changes

The PR adds a Gateway `thinking-events` client capability, routes thinking agent events to registered WebSocket recipients, decouples reasoning streaming from `onReasoningStream`, changes raw reasoning delta tracking, and updates related tests plus generated SDK baselines.

## Best Possible Solution

Land a corrected, rebased version of the cap-gated thinking-event design that preserves current-main visibility rules, explicitly separates delta-only from full-snapshot reasoning payloads, finalizes recipient registries on all failure paths, and documents/changelogs the Gateway capability.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: current main lacks `thinking-events` and requires `onReasoningStream`, while the PR head can be checked with a prior raw value `Checking` followed by a delta-only chunk ` files` and with a dispatch rejection before lifecycle emission.

## Solution Assessment

No. The cap-gated direction is the right shape, but this PR is not the best mergeable solution until the visibility guard, delta/full-snapshot contract, recipient cleanup, and public docs/changelog are fixed.

## Security Review

Status: needs_attention

Summary: The intended capability gate improves privacy, but the PR currently has concrete reasoning-token exposure risks around disabled thinking and stale subscribers.

Concerns:

- **[medium] Reasoning can stream when thinking is disabled:** `src/agents/pi-embedded-subscribe.ts:77`
  - body: The PR drops current main's `params.thinkingLevel !== "off"` guard from the reasoning streaming flag, so opted-in Gateway clients could receive reasoning tokens even when the session explicitly disables thinking visibility.
  - confidence: 0.8
- **[medium] Stale thinking subscribers after early dispatch failure:** `src/gateway/server-methods/chat.ts:1558`
  - body: `chat.send` pre-registers thinking recipients before dispatch, but if dispatch fails before lifecycle `end` or `error`, `thinkingEventRecipients.markFinal` is never called. The stale mapping can outlive the failed run and receive reasoning events for a reused run id.
  - confidence: 0.82
