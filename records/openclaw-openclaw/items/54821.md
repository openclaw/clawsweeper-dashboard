## Summary

Keep open: the Gateway thinking-events capability is not on current main, and the no-callback reasoning stream bug is still source-reproducible, but this PR head is not safe to merge because it drops the reasoning visibility gate, conflicts with the current raw-reasoning payload direction, and leaves pre-registered recipients alive on early dispatch failure.

## What This Changes

The PR adds a Gateway `thinking-events` client capability, routes thinking agent events to registered WebSocket recipients, decouples reasoning emission from `onReasoningStream`, changes reasoning delta handling, and updates Gateway/agent tests plus generated API baselines.

## Best Possible Solution

Refresh or replace this branch with a protocol-reviewed Gateway `thinking-events` design that preserves reasoning visibility, keeps core thinking payloads raw, finalizes recipient grants on all early-failure paths, and includes focused Gateway/agent coverage; if the new cap is not desired, land the narrower callback fix in https://github.com/openclaw/openclaw/pull/79687 instead.

## Reproduction Assessment

Yes for the bug portion: current main still requires `onReasoningStream` before `emitReasoningStream` can emit, while Gateway `chat.send` does not provide that callback. The cap-gated subscription behavior is a new protocol feature rather than a current-main bug reproduction.

## Solution Assessment

No. The capability direction may be valid, but this PR head drops the reasoning visibility gate, reintroduces formatted core thinking payloads, and leaves pre-registered recipient grants alive on early dispatch failures.

  - confidence: 0.8
