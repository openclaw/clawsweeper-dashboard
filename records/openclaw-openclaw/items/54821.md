## Summary

Keep this PR open: current main still lacks the requested `thinking-events` capability, but the branch has blocking source-level defects in reasoning visibility, delta accumulation, and early-failure recipient cleanup, plus missing public docs/changelog coverage.

## What This Changes

The PR adds a Gateway `thinking-events` client capability, thinking-event recipient routing, raw reasoning delta tracking, chat/agent registration plumbing, and related gateway tests/generated baselines.

## Best Possible Solution

Land a corrected, rebased cap-gated thinking-event implementation that preserves current visibility rules, defines a robust raw delta/full-snapshot contract, finalizes recipient registries on all failure paths, and documents/changelogs the new Gateway capability.

## Reproduction Assessment

Yes. The current-main source path shows `thinking-events` is absent and reasoning streaming still requires `onReasoningStream`; the PR-head defects are source-reproducible from the diff with a prior raw `Checking` plus delta-only ` files`, disabled thinking visibility, and dispatch rejection after pre-registration.

## Solution Assessment

No. The opt-in capability design is the right direction, but this patch is not the best mergeable solution until the visibility guard, delta/full-snapshot handling, recipient cleanup, docs, and changelog are fixed.

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve delta-only reasoning chunks:** `src/agents/pi-embedded-subscribe.ts:648-651`
  - body: This branch also receives native `thinking_delta` fallback chunks where `text` is only the new token. After an initial `"Checking"`, a later `" files"` chunk has no meaningful prefix/suffix overlap, so this replacement branch truncates the full thinking text instead of accumulating `"Checking files"`. The fix needs an explicit full-snapshot versus delta signal, or a fallback that appends known delta-only events.
  - confidence: 0.9
- **[P2] Keep the thinking visibility guard:** `src/agents/pi-embedded-subscribe.ts:77-85`
  - body: Current main only enables reasoning display/streaming when `params.thinkingLevel !== "off"`, but this branch enables `streamReasoning` solely from `reasoningMode === "stream"`. A session with thinking disabled could still emit reasoning tokens to opted-in Gateway clients, so preserve the existing visibility guard for both included and streamed reasoning.
  - confidence: 0.86
- **[P2] Clean up pre-registered recipients on dispatch failure:** `src/gateway/server-methods/chat.ts:1558-1560`
  - body: `chat.send` now registers tool/thinking recipients before dispatch, but if dispatch rejects before any lifecycle event is emitted, the catch/finally path never marks the recipient registries final. Those run-to-connection entries can stay alive for the registry TTL and can misroute events if a client run id is reused.
  - confidence: 0.85
- **[P3] Add the required changelog entry:** `src/gateway/protocol/client-info.ts:49`
  - body: This adds a user-visible Gateway capability and changes thinking-event delivery, but the PR does not update `CHANGELOG.md`. Repo policy requires user-facing `feat` and `fix` changes to include an active-version changelog entry before merge.
  - confidence: 0.93

  - confidence: 0.8

## Security Review

Status: needs_attention

Summary: The capability gate improves privacy in principle, but the current diff has concrete reasoning-token exposure and stale-subscriber risks.

Concerns:

- **[medium] Reasoning can stream when thinking is disabled:** `src/agents/pi-embedded-subscribe.ts:77`
  - body: The PR drops current main's `params.thinkingLevel !== "off"` guard from the reasoning streaming flag, so opted-in Gateway clients could receive reasoning tokens even when the session disables thinking visibility.
  - confidence: 0.82
