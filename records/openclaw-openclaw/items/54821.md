## Summary

Keep open: this PR addresses a real source-reproducible Gateway thinking-stream gap, but the current branch is not mergeable as-is because it bypasses reasoning visibility gates, reintroduces formatted core thinking payloads after the merged raw-payload refactor, and leaves pre-registered recipients alive on early dispatch failures.

## What This Changes

This PR adds a Gateway `thinking-events` capability, run-scoped recipient routing for thinking streams, and Embedded Pi reasoning-stream delta changes for WebSocket clients.

## Best Possible Solution

Land the narrow no-callback fix from https://github.com/openclaw/openclaw/pull/79687 if that is the desired behavior, and pursue cap-gated thinking events only after explicit protocol approval with visibility guards, raw payloads, cleanup, docs, and protocol tests.

## Reproduction Assessment

Yes, there is a high-confidence source-level reproduction path: current main requires `onReasoningStream`, while Gateway `chat.send` does not provide that callback. I did not run a live Gateway/model reproduction in this read-only review.

## Solution Assessment

No. The cap-gated direction may be viable, but this branch is not the best version until it preserves current visibility gates, keeps core thinking payloads raw, finalizes pre-registered recipients on early failure, and gets maintainer protocol approval.

  - confidence: 0.9

  - confidence: 0.88

  - body: `chat.send` now registers tool and thinking recipients before dispatch, but if `dispatchInboundMessage` rejects before lifecycle events are emitted, no `markFinal` call runs and the run-to-connection grant survives until TTL. Finalize those recipient registries in the early failure path too.
  - confidence: 0.78
