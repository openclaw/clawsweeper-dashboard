## Summary

Keep open: current main does not implement the `thinking-events` Gateway cap, and this PR addresses a real gap, but the current diff still has blocking correctness and privacy issues around reasoning visibility, delta handling, and recipient cleanup.

## What This Changes

The PR adds a Gateway `thinking-events` client capability, thinking-event recipient routing, raw reasoning delta tracking, chat/agent registration plumbing, and related tests/generated baselines.

## Best Possible Solution

Land a corrected, rebased cap-gated thinking-event implementation that preserves existing thinking visibility policy, uses an explicit raw full-snapshot versus delta contract, cleans recipient registries on all terminal paths, and updates Gateway protocol docs plus the changelog.

## Reproduction Assessment

Yes. Current main source shows the new cap is absent and reasoning streaming still depends on `onReasoningStream`; the remaining PR defects are source-reproducible from the diff without needing a live provider.

## Solution Assessment

No. The opt-in capability route is the right direction, but this patch is not mergeable until the visibility guard, delta/full-snapshot handling, early-failure cleanup, docs, and changelog are fixed.

- **[P2] Preserve delta-only reasoning chunks:** `src/agents/pi-embedded-subscribe.ts:648-649`
  - body: This branch also receives native `thinking_delta` fallback chunks where `text` is only the new token. After a prior raw value like `Checking`, a later ` files` chunk has little or no overlap, so this replacement branch resets the accumulated thinking text instead of producing `Checking files`. The fix needs an explicit delta/full-snapshot signal or a fallback that appends known delta-only events.
  - confidence: 0.9
- **[P2] Keep the thinking visibility guard:** `src/agents/pi-embedded-subscribe.ts:75`
  - body: Current main only streams reasoning when `params.thinkingLevel !== "off"`, but this change enables `streamReasoning` solely from `reasoningMode === "stream"`. A session with thinking visibility disabled can still emit reasoning tokens to opted-in Gateway clients, so preserve the existing visibility guard while decoupling the optional callback.
  - confidence: 0.86

  - body: `chat.send` now registers tool/thinking recipients before dispatch, but if dispatch rejects before any lifecycle event is emitted, the catch/finally path never marks those registries final. The stale run-to-connection entries can live for the registry TTL and may receive events if a client run id is reused.
  - confidence: 0.85
