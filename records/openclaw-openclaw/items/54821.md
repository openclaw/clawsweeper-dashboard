## Summary

Keep this PR open: the cap-gated routing direction is useful, but the branch is stale against current main and would regress current reasoning visibility and raw-payload contracts before merge.

## What This Changes

The branch adds a `thinking-events` gateway client capability, per-run thinking recipient registration, cap-only `stream: "thinking"` routing, reasoning delta state changes, and gateway tests/generated baseline updates.

## Best Possible Solution

Add `thinking-events` on top of current main's raw reasoning stream contract, preserving `canShowReasoning`, `silentExpected`, and Control UI visibility gates; coordinate the callback-only part with https://github.com/openclaw/openclaw/pull/79687.

## Reproduction Assessment

Yes, there is a high-confidence source-level reproduction path: current main lacks `thinking-events` and still requires `onReasoningStream` before emitting thinking events, while gateway `chat.send` does not provide that callback. I did not run a live model session in this read-only review.

## Solution Assessment

No. Cap-gated routing is the right direction, but this branch is not the best mergeable solution until it is rebased onto current raw reasoning behavior and preserves the current visibility/silent-turn safeguards.

  - body: Current main only streams reasoning when `thinkingLevel` allows it and `silentExpected` is false.   - confidence: 0.88

  - body: The new pre-dispatch registration runs before any lifecycle `end` or `error` can call `markFinal`. If `dispatchInboundMessage` rejects early, the catch/finally path only deletes the active run, leaving tool/thinking recipients alive until TTL and risking stale delivery if the client run id is reused.
  - confidence: 0.72
