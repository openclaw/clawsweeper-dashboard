## Summary

Keep open: current main still lacks the requested `thinking-events` Gateway capability, and this PR remains useful but is not mergeable as-is. The latest PR head still has source-reproducible correctness and privacy blockers around thinking visibility, delta-only chunks, and recipient cleanup after early dispatch failure.

## What This Changes

The PR adds a Gateway `thinking-events` client capability, per-run thinking-event recipient routing, raw reasoning delta tracking, chat/agent registration plumbing, and related tests/generated baselines.

## Best Possible Solution

Land a rebased cap-gated thinking-event implementation that preserves thinking visibility, robustly distinguishes full snapshots from deltas, finalizes recipient registries on all failure paths, and updates protocol docs/changelog.

## Reproduction Assessment

Yes. Current-main source shows the requested cap is absent and thinking streaming still requires `onReasoningStream`; the PR-head defects are source-reproducible from the changed branches and dispatch failure path.

## Solution Assessment

No. The opt-in capability design is the right direction, but this patch is not the best mergeable solution until it keeps current visibility guards, fixes delta/full-snapshot handling, and cleans up recipient registrations reliably.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Preserve the thinking visibility guard:** `src/agents/pi-embedded-subscribe.ts:77`
  - body: Current main only includes or streams reasoning when `params.thinkingLevel !== "off"`, but this PR head sets `streamReasoning` from `reasoningMode === "stream"` alone. An opted-in Gateway client could receive reasoning tokens for a session where thinking visibility is disabled, so carry the existing `canShowReasoning` guard into both included and streamed reasoning.
  - confidence: 0.9
- **[P2] Preserve delta-only reasoning chunks:** `src/agents/pi-embedded-subscribe.ts:648-651`
  - body: This non-overlap branch also receives native delta-only chunks. After a prior raw value like `Checking`, a later chunk ` files` has little prefix/suffix overlap, so the new replacement path drops the prior text instead of accumulating `Checking files`; the fix needs an explicit delta/full-snapshot signal or a safe append fallback for known delta events.
  - confidence: 0.89
- **[P2] Finalize recipients when dispatch fails early:** `src/gateway/server-methods/chat.ts:1553-1559`
  - body: `chat.send` now registers recipients before dispatch, but if `dispatchInboundMessage` rejects before lifecycle `end`/`error`, the catch/finally path never marks the tool or thinking recipient registries final. Those run-to-connection mappings can live until TTL and can misroute reused run IDs, so finalize/remove the pre-registered entries on this failure path.
  - confidence: 0.86
- **[P3] Document the new Gateway capability:** `src/gateway/protocol/client-info.ts:49`
  - body: Adding `thinking-events` changes the public connect/capability contract, but the PR does not update the Gateway protocol docs. Client authors need the public docs to describe that advertising this cap opts into thinking agent events.
  - confidence: 0.82
- **[P3] Add the required changelog entry:** `src/gateway/protocol/client-info.ts:49`
  - body: This is a user-visible `feat`/`fix` PR that changes Gateway event delivery, but it does not add a `CHANGELOG.md` entry. Repo policy requires an active-version changelog entry before merge.
  - confidence: 0.93
