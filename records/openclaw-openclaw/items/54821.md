## Summary

Keep this PR open, but it is not ready to merge: the capability-gated direction may be useful, yet the current branch is stale against main and would regress reasoning visibility and raw payload behavior.

## What This Changes

The PR adds a Gateway `thinking-events` client capability, run-scoped thinking-event recipients, and Embedded Pi reasoning-stream delta changes for WebSocket clients.

## Best Possible Solution

Choose the Gateway thinking-stream contract, then either land the narrow callback fix or rework this cap-gated design on current main with visibility gates, raw payloads, recipient cleanup, protocol docs, and focused tests.

## Reproduction Assessment

Yes. Source inspection shows current main still requires `onReasoningStream` before emitting `stream:"thinking"`, while Gateway `chat.send` does not provide that callback and current main has no `thinking-events` cap.

## Solution Assessment

No. The cap-gated direction may be viable, but this branch is not the best version until it preserves visibility and silent-turn gates, keeps core thinking payloads raw, and finalizes recipient registries on early failure.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve reasoning visibility gates:** `src/agents/pi-embedded-subscribe.ts:77`
  - body: Current main only streams reasoning when visibility allows it and silent turns are not expected. This branch enables `streamReasoning` from `reasoningMode` alone, so an opted-in Gateway client could receive reasoning that `thinkingLevel:"off"` or silent-turn handling should suppress.
  - confidence: 0.88
- **[P2] Keep core thinking payloads raw:** `src/agents/pi-embedded-subscribe.ts:688-689`
  - body: Current main keeps core thinking `text` and `delta` as raw strings, with formatting pushed to channel/display edges. Emitting `formatReasoningMessage(...)` as Gateway `data.text` would reintroduce formatted `Reasoning:\n_..._` payloads for clients consuming the core event stream.
  - confidence: 0.9
- **[P2] Finalize pre-registered recipients on failure:** `src/gateway/server-methods/chat.ts:1553-1558`
  - body: The branch registers tool and thinking recipients before dispatch, but if `dispatchInboundMessage` rejects before lifecycle events are emitted, the catch/finally path never marks those recipient registries final. Those entries can survive until TTL and may be reused with the same client run id.
  - confidence: 0.76
