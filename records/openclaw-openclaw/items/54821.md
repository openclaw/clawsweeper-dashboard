## Summary

Keep open: current main still lacks a cap-gated Gateway thinking-event subscription, so the unique protocol/security part is not implemented. The current PR head is not merge-ready because it predates the raw reasoning payload refactor, drops current reasoning visibility gates, and leaves a recipient-cleanup gap; the narrower open https://github.com/openclaw/openclaw/pull/79687 covers only the callback-decoupling part, not the capability contract.

## What This Changes

The branch adds a Gateway `thinking-events` client capability, a run-scoped thinking recipient registry, embedded reasoning delta changes, and related Gateway tests/baseline updates.

## Best Possible Solution

Rebase or replace this with a maintainer-approved Gateway thinking-events contract that preserves reasoning visibility gates, keeps core thinking payloads raw, finalizes run-scoped recipient grants on every dispatch failure path, and documents the protocol; if maintainers only want the narrow callback fix, land https://github.com/openclaw/openclaw/pull/79687 and explicitly close the cap-gated design track.

## Reproduction Assessment

Yes. The bug portion is source-reproducible because current main requires `onReasoningStream` before emitting `stream:"thinking"`, while Gateway `chat.send` does not pass that callback; I did not run a live current-main Gateway repro in this read-only review.

## Solution Assessment

No. Cap-gated delivery is a plausible design, but this PR head is not the best merge path until it is rebased, preserves current privacy/raw-payload contracts, and handles recipient cleanup on early failures.
