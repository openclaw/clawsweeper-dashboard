## Summary

Keep open: current main still lacks the cap-gated thinking-event subscription, and the linked request was closed in favor of this PR, but the branch is not mergeable as-is because it bypasses current reasoning visibility policy and conflicts with the merged raw-reasoning payload contract.

## What This Changes

The PR adds a gateway `thinking-events` client capability, recipient registry, and reasoning-stream changes so subscribed WebSocket clients can receive thinking events and deltas.

## Best Possible Solution

Rework the cap-gated gateway design on current main so it preserves reasoning visibility policy, keeps core thinking payloads raw, finalizes recipient registries on early failures, and documents/tests the protocol capability.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main still gates thinking emission on `onReasoningStream` and has no `thinking-events` client capability. I did not run a live reasoning model for this review.

## Solution Assessment

No. The cap-gated direction is plausible, but this branch should be rebased to preserve current visibility gates, keep raw thinking payloads, and clean up recipient registries on early dispatch failure.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve reasoning visibility gates:** `src/agents/pi-embedded-subscribe.ts:72-77`
  - body: Current main only includes or streams reasoning when `params.thinkingLevel !== "off"`, and silent turns are suppressed before emission. This branch derives `includeReasoning` and `streamReasoning` from `reasoningMode` alone, so a cap-subscribed gateway client can receive thinking text that current visibility policy hides.
  - confidence: 0.88
- **[P2] Keep thinking payloads raw:** `src/agents/pi-embedded-subscribe.ts:664-690`
  - body: The merged raw-reasoning contract keeps core thinking `text` and `delta` as raw strings, with display formatting pushed to channel edges. Emitting `formatReasoningMessage(rawAccumulated)` here would regress gateway clients back to formatted `Reasoning:\n_..._` payloads.
  - confidence: 0.9
- **[P3] Finalize recipient registries on dispatch failure:** `src/gateway/server-methods/chat.ts:1553-1559`
  - body: The chat path registers tool/thinking recipients before dispatch, but if dispatch rejects before lifecycle end/error, the catch/finally path does not mark those registries final. Those entries can remain until TTL and be reused with the same client run id.
  - confidence: 0.74

## Security Review

Status: needs_attention

Summary: The opt-in capability is a reasonable direction, but the current patch can expose reasoning tokens outside the existing visibility policy.

Concerns:

- **[medium] Reasoning visibility gates are bypassed:** `src/agents/pi-embedded-subscribe.ts:77`
  - body: The PR enables thinking streams from `reasoningMode` alone instead of preserving `canShowReasoning` and silent-turn suppression, so opted-in clients could receive reasoning tokens current main intentionally hides.
  - confidence: 0.86
- **[low] Stale thinking recipients may persist after early failure:** `src/gateway/server-methods/chat.ts:1558`
  - body: Pre-registered thinking recipients are not finalized if dispatch fails before lifecycle events, leaving run-to-connection entries alive until TTL and creating a possible misdelivery path if an idempotency run id is reused.
  - confidence: 0.7
