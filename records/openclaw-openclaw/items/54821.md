## Summary

Keep this PR open, but do not merge it as-is: current main still lacks the `thinking-events` capability path, while this branch conflicts with current reasoning visibility and raw-payload contracts introduced after the branch was opened.

## What This Changes

The PR adds a `thinking-events` gateway client capability, recipient registry, and reasoning stream changes so subscribed WebSocket clients can receive thinking events with incremental deltas.

## Best Possible Solution

Rework the cap-gated Gateway design on current main so it preserves reasoning visibility policy, keeps core thinking payloads raw, cleans up recipient registries on early failures, and documents/tests the protocol capability.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main has no `thinking-events` client cap and still requires `onReasoningStream` before emitting thinking events. I did not run a live reasoning model for this review.

## Solution Assessment

No. The cap-gated direction is plausible, but this branch is not the best version because it weakens visibility gates, conflicts with the raw thinking payload contract, and leaves early-failure cleanup incomplete.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve reasoning visibility gates:** `src/agents/pi-embedded-subscribe.ts:77`
  - body: Current main treats `thinkingLevel === "off"` and silent turns as gates before reasoning emission. This branch makes `streamReasoning` true from `reasoningMode` alone and the emitter lacks the `silentExpected` guard, so a `thinking-events` client could receive reasoning tokens that current main hides.
  - confidence: 0.88
- **[P2] Keep core thinking payloads raw:** `src/agents/pi-embedded-subscribe.ts:664-689`
  - body: Current main sends raw thinking `text` and `delta` from the core layer and leaves formatting to channel/display edges. Emitting `formatReasoningMessage(rawAccumulated)` here would regress Gateway clients back to formatted `Reasoning:\n_..._` payloads.
  - confidence: 0.9
- **[P3] Finalize recipients when dispatch fails early:** `src/gateway/server-methods/chat.ts:1702-1704`
  - body: Recipients are registered before dispatch, but if `dispatchInboundMessage` rejects before lifecycle end/error, this cleanup only deletes the abort controller. The tool/thinking recipient entries can remain until TTL and may be reused for the same idempotency run id.
  - confidence: 0.74
