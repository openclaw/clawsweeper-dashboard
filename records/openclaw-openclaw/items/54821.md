## Summary

Keep open. The thinking-events capability is not implemented on current main and the PR has useful real WebSocket proof, but the branch is not merge-ready because it conflicts with current raw reasoning and visibility contracts.

## What This Changes

Adds a `thinking-events` gateway client capability, recipient registry/routing, reasoning delta state changes, gateway tests, and generated SDK baseline updates.

## Best Possible Solution

Rebase the cap-gated gateway delivery onto current main's raw reasoning stream contract, preserving thinking visibility gates and adding focused routing coverage plus protocol documentation.

## Reproduction Assessment

Yes. Source inspection shows current main exposes only `tool-events`, still requires `onReasoningStream` before thinking events are emitted, and routes thinking as a generic non-tool event; I did not run a live model session.

## Solution Assessment

No. The cap-gated direction is right, but this branch must be rebased and changed to preserve current raw payloads, visibility gates, and recipient cleanup before it is the best maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve current reasoning visibility gates:** `src/agents/pi-embedded-subscribe.ts:77`
  - body: Current main only streams reasoning when it can be shown and also suppresses silent turns before emitting. This branch enables `streamReasoning` from `reasoningMode` alone and the emit path no longer checks `silentExpected`, so cap-subscribed clients can receive reasoning tokens that current main intentionally hides.
  - confidence: 0.86
- **[P2] Keep thinking-event payloads raw:** `src/agents/pi-embedded-subscribe.ts:664-689`
  - body: Current main's raw-reasoning refactor sends raw reasoning text and deltas from the core stream, leaving channel formatting to the edges. This branch formats accumulated reasoning in core before emitting thinking events/callbacks, which would regress gateway clients and channels that now consume raw reasoning payloads.
  - confidence: 0.88
- **[P3] Finalize pre-registered recipients on dispatch failure:** `src/gateway/server-methods/chat.ts:1553-1558`
  - body: The new pre-dispatch recipient registration can run before any lifecycle end/error event exists to call `markFinal`. If `dispatchInboundMessage` rejects early, the catch/finally path removes the active run but leaves tool/thinking recipient entries alive until TTL, which can accumulate and misroute if an idempotency run id is reused.
  - confidence: 0.72
