## Summary

Keep this PR open: current main still lacks the requested `thinking-events` Gateway capability, but the PR is dirty against main and has blocking reasoning-stream security/correctness issues plus missing protocol docs and changelog coverage.

## What This Changes

The PR adds a Gateway `thinking-events` client capability, routes thinking agent events to cap-registered WebSocket recipients, decouples reasoning streaming from `onReasoningStream`, changes raw reasoning delta tracking, and updates related tests plus generated SDK baselines.

## Best Possible Solution

Land a corrected, rebased version of the cap-gated thinking-event design that preserves thinking-off and hidden-run visibility rules, explicitly separates delta-only chunks from full snapshots, finalizes recipient registries on every failure path, and documents/changelogs the Gateway capability.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: current main lacks `thinking-events` and requires `onReasoningStream`, while the PR head can be checked with `thinkingLevel: "off"`, with prior raw text `Checking` followed by delta-only chunk ` files`, and with dispatch rejection after recipient pre-registration.

## Solution Assessment

No. The cap-gated direction is the right maintainable shape, but this branch is not mergeable and still needs security, delta-contract, cleanup, docs, and changelog fixes before it is the best solution.

The fix needs an explicit full-snapshot versus delta signal, or a fallback that appends known delta-only events.

## Security Review

Status: needs_attention

Summary: The intended capability gate improves privacy, but the PR currently has concrete reasoning-token exposure risks around disabled thinking and stale subscribers.

Concerns:

- **[medium] Reasoning can stream when thinking is disabled:** `src/agents/pi-embedded-subscribe.ts:77`
  - body: The PR drops current main's `params.thinkingLevel !== "off"` guard from `streamReasoning`, so opted-in Gateway clients could receive reasoning tokens even when the session explicitly disables thinking visibility.
  - confidence: 0.86
- **[medium] Stale thinking subscribers after early dispatch failure:** `src/gateway/server-methods/chat.ts:1558`
  - body: `chat.send` pre-registers thinking recipients before dispatch, but if dispatch fails before lifecycle `end` or `error`, `thinkingEventRecipients.markFinal` is never called. The stale mapping can outlive the failed run and receive reasoning events for a reused run id.
  - confidence: 0.82
