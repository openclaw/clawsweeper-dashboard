## Summary

Keep open for maintainer handling: the underlying Gateway reasoning-stream problem is still source-reproducible on current main, but this branch is not mergeable as-is because it bypasses current reasoning visibility gates, conflicts with the raw-payload direction already merged, and adds a security-sensitive capability contract that needs product/protocol review.

## What This Changes

The PR adds a Gateway `thinking-events` client capability, run-scoped thinking-event recipients, and embedded Pi reasoning delta changes for WebSocket clients.

## Best Possible Solution

Review and land the narrow callback-decoupling path in https://github.com/openclaw/openclaw/pull/79687 if it satisfies the bug, then handle cap-gated thinking-event delivery as a separate documented Gateway protocol decision with routing and cleanup tests.

## Reproduction Assessment

Yes. Source inspection shows current main still requires `onReasoningStream` for `streamReasoning`, while Gateway `chat.send` does not provide that callback, so callback-less WebSocket clients cannot receive live thinking events on that path.

## Solution Assessment

No. The branch combines a useful no-callback fix with a new capability/security contract and stale reasoning-payload assumptions; the safer path is a narrow callback fix plus a separately reviewed protocol change if maintainers still want cap-gated thinking events.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve reasoning visibility gates:** `src/agents/pi-embedded-subscribe.ts:72-77`
  - body: Current main gates reasoning visibility with `canShowReasoning`, but this branch enables both `includeReasoning` and `streamReasoning` from the mode alone. A client using the new cap could receive reasoning when `thinkingLevel: "off"` should hide it, so keep the visibility guard when decoupling the callback.
  - confidence: 0.9
- **[P2] Keep core thinking payloads raw:** `src/agents/pi-embedded-subscribe.ts:664-689`
  - body: Current main moved reasoning formatting out of `pi-embedded-subscribe` and sends raw thinking text/deltas from the core stream. Reintroducing `formatReasoningMessage(...)` here would send `Reasoning:\n_..._` over the Gateway event payload and conflict with the channel-edge formatting contract.
  - confidence: 0.87
- **[P2] Finalize recipients after early dispatch failures:** `src/gateway/server-methods/chat.ts:1558`
  - body: This pre-registers thinking recipients before dispatch, but if `dispatchInboundMessage` rejects before lifecycle end/error events, the catch/finally path never marks the recipient entry final. That leaves a stale run-to-connection grant alive until TTL and can accumulate after repeated early failures.
  - confidence: 0.76
