## Summary

Keep this PR open, but do not merge the current head. Current main still lacks the `thinking-events` Gateway capability, so the work is not obsolete, but the PR is merge-conflicting and would regress current raw reasoning and visibility/security behavior.

## What This Changes

The PR adds a Gateway `thinking-events` client capability, routes thinking agent events to cap subscribers, changes reasoning delta handling, and updates related tests/generated API baselines.

## Best Possible Solution

Refresh or replace this with a minimal Gateway thinking-events capability that preserves `thinkingLevel`/silent visibility gates, keeps core reasoning payloads raw, cap-gates delivery, finalizes run grants on all failure paths, and documents the protocol.

## Reproduction Assessment

Yes for the bug portion by source inspection: current main still requires `onReasoningStream` before emitting `stream: "thinking"`, while Gateway `chat.send` does not provide that callback. The cap-gated subscription behavior is new Gateway protocol surface rather than an existing-behavior reproduction.

## Solution Assessment

No. The capability direction is useful, but this PR head is stale/conflicting and would regress current raw-payload and visibility contracts; the safer path is a refresh on main with a narrow, documented Gateway cap design.

  - body: This enables streamed reasoning from `reasoningMode === "stream"` alone, dropping current main's `thinkingLevel` visibility guard. A client with the new cap could receive reasoning when reasoning visibility is disabled, so keep the existing visibility gate while decoupling the callback.
  - confidence: 0.9

  - confidence: 0.88

If dispatch fails before lifecycle end/error, those run-scoped grants remain until TTL; mark both recipient registries final in the catch/finally path.
  - confidence: 0.78
