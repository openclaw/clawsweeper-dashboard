## Summary

Keep this PR open: the opt-in Gateway `thinking-events` direction is useful and current main still lacks that capability, but the PR head needs repairs before merge because it can bypass the thinking visibility policy, reintroduce core-side reasoning formatting, corrupt delta-only chunks, and leave pre-registered recipients stale after early dispatch failure.

## What This Changes

The PR adds a Gateway `thinking-events` client capability, thinking-recipient routing, raw reasoning delta tracking, WebSocket cap registration, and related gateway tests/baselines.

## Best Possible Solution

Rebase and repair the opt-in Gateway thinking-event implementation on current main while preserving raw core reasoning streams, visibility policy, recipient cleanup, and public protocol documentation.

## Reproduction Assessment

Yes, at source level: current main lacks the `thinking-events` cap and still requires `onReasoningStream` for reasoning streaming, while the PR-head defects are visible in the touched code paths. I did not run a live model/WebSocket reproduction in this read-only review.

## Solution Assessment

No. The opt-in capability direction is right, but this patch is not the best mergeable solution until it preserves the thinking-off guard, raw reasoning contract, delta-only accumulation, early-failure cleanup, and Gateway protocol docs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve the thinking visibility guard:** `src/agents/pi-embedded-subscribe.ts:77`
  - body: Current main only streams reasoning when thinking visibility is enabled, but this patch sets `streamReasoning` solely from `reasoningMode === "stream"`. An opted-in Gateway client could receive reasoning tokens even when the session disables thinking visibility, so keep the `params.thinkingLevel !== "off"` guard in this path.
  - confidence: 0.91
- **[P2] Keep core reasoning payloads raw:** `src/agents/pi-embedded-subscribe.ts:664`
  - body: Current main moved reasoning formatting to channel edges, and tests now expect raw `onReasoningStream` text. Formatting `rawAccumulated` here would reintroduce `Reasoning:\n_..._` payloads into the core/Gateway thinking stream and break clients that apply raw deltas.
  - confidence: 0.88
- **[P2] Preserve delta-only reasoning chunks:** `src/agents/pi-embedded-subscribe.ts:648-651`
  - body: This branch also receives native delta-only chunks. With prior raw text `Checking` and the next chunk ` files`, the no-overlap branch treats the chunk as a replacement and drops the earlier token; use an explicit full-snapshot versus delta signal, or append for known delta-only events.
  - confidence: 0.86
- **[P2] Finalize pre-registered recipients on early failure:** `src/gateway/server-methods/chat.ts:1558`
  - body: `chat.send` now registers tool/thinking recipients before dispatch, but if `dispatchInboundMessage` rejects before any lifecycle `end` or `error`, the catch/finally path never marks those registries final. That leaves stale run-to-connection entries alive until TTL and can misroute events if a client run id is reused.
  - confidence: 0.86
- **[P3] Document the thinking-events capability:** `src/gateway/protocol/client-info.ts:47`
  - body: Adding `THINKING_EVENTS` changes the Gateway client handshake contract, but the PR only updates the generated baseline and protocol constant. Client authors need the public Gateway protocol docs to say that advertising this cap opts into thinking `agent` events.
  - confidence: 0.78
