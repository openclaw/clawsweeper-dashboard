## Summary

Keep this PR open: the underlying Gateway thinking-stream problem is still present on current main, but the PR head is stale against newer reasoning-stream behavior and still has merge-blocking protocol/security issues.

## What This Changes

The PR adds a Gateway `thinking-events` client capability, cap-gated thinking event routing, reasoning delta changes, and related gateway/agent tests plus generated API baseline updates.

## Best Possible Solution

Refresh the branch or replace it with a narrower design that preserves reasoning visibility gates, keeps Gateway thinking payloads raw, finalizes recipient grants on all failure paths, and documents the new client capability after maintainer protocol approval.

## Reproduction Assessment

Yes for the bug portion by source inspection: current main requires `onReasoningStream` before `stream:"thinking"` is emitted, while Gateway `chat.send` does not pass that callback. The cap-gated subscription behavior is new protocol surface rather than a broken existing contract reproduction.

## Solution Assessment

No. The direction may be useful, but this PR head is not the best merge path until it preserves current visibility gates, follows the raw core reasoning payload contract, cleans up recipient grants on early failures, and gets maintainer agreement on the Gateway capability contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve reasoning visibility gates:** `src/agents/pi-embedded-subscribe.ts:77`
  - body: This enables streamed reasoning from `reasoningMode === "stream"` alone, dropping current main's `canShowReasoning` guard.   - confidence: 0.91
- **[P2] Keep core thinking payloads raw:** `src/agents/pi-embedded-subscribe.ts:664-689`
  - body: Current main now treats core reasoning `text` and `delta` as raw payloads and leaves Markdown formatting to channel edges. Reintroducing `formatReasoningMessage(...)` here would send formatted `Reasoning:` Markdown through the Gateway protocol and conflict with that merged contract.
  - confidence: 0.89
- **[P2] Finalize recipients after early dispatch failures:** `src/gateway/server-methods/chat.ts:1558`
  - body: `chat.send` pre-registers thinking recipients before dispatch, but the catch/finally path only deletes the abort controller. If dispatch rejects before lifecycle events run, the run-to-connection grant survives until TTL and can accumulate or misroute if a run id is reused.
  - confidence: 0.77
