## Summary

Keep this PR open: the requested thinking-events capability is still relevant, but the branch has blocking regressions against current main’s reasoning visibility and raw-payload contracts.

## What This Changes

The PR adds a `thinking-events` gateway client capability, recipient registry, and agent reasoning-stream changes so subscribed WebSocket clients can receive thinking events and incremental deltas.

## Best Possible Solution

Rework the cap-gated gateway design on current main so it preserves reasoning visibility policy, keeps core thinking payloads raw, finalizes recipient registries on early failures, and documents/tests the protocol capability.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main still requires `onReasoningStream` before emitting thinking events and has no `thinking-events` client cap. I did not run a live reasoning model for this review.

## Solution Assessment

No. The cap-gated direction is plausible, but this branch is not the best version because it drops current visibility gates, conflicts with the raw thinking payload contract, and leaves early-failure cleanup incomplete.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve reasoning visibility gates:** `src/agents/pi-embedded-subscribe.ts:77`
  - body: Current main only streams reasoning when the visibility gate allows it, and silent turns are suppressed before emission. This change enables thinking streams from `reasoningMode` alone, so a cap-subscribed client could receive reasoning tokens that current main intentionally hides.
  - confidence: 0.88
- **[P2] Keep thinking payloads raw:** `src/agents/pi-embedded-subscribe.ts:664-689`
  - body: The merged raw-reasoning contract keeps core thinking `text` and `delta` as raw strings, with formatting pushed to channel/display edges. Emitting `formatReasoningMessage(...)` here would regress gateway clients back to `Reasoning:\n_..._` payloads.
  - confidence: 0.9
- **[P3] Finalize pre-registered recipients on dispatch failure:** `src/gateway/server-methods/chat.ts:1553-1559`
  - body: Recipients are registered before dispatch, but if `dispatchInboundMessage` rejects before lifecycle events are emitted, the catch/finally path only deletes the abort controller. Those recipient entries can stay alive until TTL and may be reused with the same idempotency run id.
  - confidence: 0.74
