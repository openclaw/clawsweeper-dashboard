## Summary

Keep open: the feature request is still valid and the PR has visible after-fix proof, but the branch is not mergeable as-is because it conflicts with current reasoning visibility and raw-payload contracts and leaves recipient cleanup incomplete.

## What This Changes

The PR adds a `thinking-events` gateway client capability, recipient registry, and reasoning-stream routing/delta changes so subscribed WebSocket clients can receive thinking events.

## Best Possible Solution

Rework the cap-gated gateway design on current main so it preserves reasoning visibility policy, keeps core thinking payloads raw, finalizes recipient registries on early failures, and documents/tests the protocol capability.

## Reproduction Assessment

Yes. Source inspection on current main shows gateway WebSocket runs still cannot receive thinking events without an `onReasoningStream` callback, and current main has no `thinking-events` client capability.

## Solution Assessment

No. The cap-gated direction is plausible, but this branch should be rebased or replaced so it preserves current visibility gates, raw thinking payloads, and cleanup behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve reasoning visibility gates:** `src/agents/pi-embedded-subscribe.ts:77`
  - body: Current main only streams reasoning when the visibility gate allows it and suppresses silent turns before emitting. This branch enables thinking streams from `reasoningMode` alone, so a cap-subscribed client could receive reasoning tokens that current main intentionally hides.
  - confidence: 0.88
- **[P2] Keep thinking payloads raw:** `src/agents/pi-embedded-subscribe.ts:664-689`
  - body: The merged raw-reasoning contract now keeps core thinking `text` and `delta` as raw strings, with formatting pushed to display edges. This branch emits `formatReasoningMessage(...)` as the core thinking `text`, which would regress gateway clients back to `Reasoning:\n_..._` payloads.
  - confidence: 0.9
- **[P3] Finalize pre-registered recipients on dispatch failure:** `src/gateway/server-methods/chat.ts:1553-1559`
  - body: Recipients are registered before dispatch, but if `dispatchInboundMessage` rejects before lifecycle events are emitted, the catch/finally path only deletes the abort controller. Those recipient entries can stay alive until TTL and may be reused with the same idempotency run id.
  - confidence: 0.74
