## Summary

Keep open: current main still lacks the proposed `thinking-events` gateway cap, so the PR is not obsolete, but the current branch is not mergeable as-is because it conflicts with current reasoning visibility gates and the raw-reasoning payload contract.

## What This Changes

The PR adds a `thinking-events` gateway client capability, recipient routing, reasoning-stream delta state, gateway registration plumbing, and related tests/generated baseline updates.

## Best Possible Solution

Rework or replace the cap-gated gateway delivery so it preserves `canShowReasoning` and `silentExpected`, keeps core thinking `text`/`delta` raw, cleans up recipient registries on early failures, and updates protocol docs/tests for the new cap if maintainers want that API.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: current main lacks `thinking-events`, still requires `onReasoningStream` before emitting thinking events, and only broadcasts thinking through the generic non-tool agent path. I did not run a live model session in this read-only review.

## Solution Assessment

No. The opt-in cap direction is plausible, but this branch is not the best implementation until it is reconciled with current raw reasoning payloads, existing visibility gates, and recipient cleanup behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

  - body: Current main only streams reasoning when `thinkingLevel` allows it and suppresses silent turns before emitting. This branch sets `streamReasoning` from `reasoningMode` alone, so a cap-subscribed client could receive reasoning tokens current main intentionally hides.
  - confidence: 0.86
- **[P2] Keep core thinking payloads raw:** `src/agents/pi-embedded-subscribe.ts:664-689`
  - body: The merged raw-reasoning refactor now treats core thinking `text` and `delta` as raw strings, leaving markdown formatting to channel edges. This branch formats the payload before emitting thinking events/callbacks, which would regress gateway and channel consumers back to `Reasoning:\n_..._` payloads.
  - confidence: 0.88
- **[P3] Finalize pre-registered recipients on dispatch failure:** `src/gateway/server-methods/chat.ts:1535-1558`
  - body: These new pre-dispatch registrations can happen before any lifecycle end/error event exists to call `markFinal`. If `dispatchInboundMessage` rejects early, the catch/finally path only removes the active run, leaving tool/thinking recipient entries alive until TTL and possibly reused with the same idempotency run id.
  - confidence: 0.72
