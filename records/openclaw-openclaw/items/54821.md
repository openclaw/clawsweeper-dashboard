## Summary

Keep open: the PR still contains unique cap-gated thinking-event work that is not on current main, but it is merge-conflicting and would regress current reasoning visibility and raw-payload contracts without a maintainer-led rework.

## What This Changes

The PR adds a `thinking-events` gateway client capability and recipient registry, routes thinking events to opted-in WebSocket clients, and rewrites reasoning-stream delta handling.

## Best Possible Solution

Rebase or replace this with a focused cap-gated gateway design that preserves `canShowReasoning` and `silentExpected`, keeps core thinking `text` and `delta` raw, finalizes recipient registries on early failures, and updates gateway protocol docs/tests alongside any new cap.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: current main lacks `thinking-events`, still requires `onReasoningStream` before emitting thinking events, and only broadcasts thinking through the generic non-tool agent path. I did not run a live model session in this read-only review.

## Solution Assessment

No. The opt-in capability direction is plausible, but this patch is no longer the best merge path until it is reconciled with current raw reasoning payloads, visibility gates, cleanup behavior, and gateway protocol documentation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve reasoning visibility gates:** `src/agents/pi-embedded-subscribe.ts:77`
  - body: Current main only streams reasoning when the reasoning visibility gate allows it and suppresses silent turns before emitting. This branch sets `streamReasoning` from `reasoningMode` alone and emits thinking events without the silent-turn guard, so a cap-subscribed client could receive reasoning tokens that current main intentionally hides.
  - confidence: 0.88
- **[P2] Keep thinking payloads raw:** `src/agents/pi-embedded-subscribe.ts:664-689`
  - body: The merged raw-reasoning refactor now treats core thinking `text` and `delta` as raw strings, with formatting pushed to display edges. This branch emits `formatReasoningMessage(...)` as the core thinking `text`, which would regress gateway and channel consumers back to `Reasoning:\n_..._` payloads.
  - confidence: 0.9
- **[P3] Finalize pre-registered recipients on failure:** `src/gateway/server-methods/chat.ts:1553-1558`
  - body: The branch registers tool/thinking recipients before dispatch, but if `dispatchInboundMessage` rejects before lifecycle events are emitted, the catch/finally path only deletes the abort controller. Those recipient entries stay alive until TTL and can accumulate or be reused with the same idempotency run id.
  - confidence: 0.74
