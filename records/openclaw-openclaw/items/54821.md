## Summary

Keep open: current main still has the source-level no-callback Gateway thinking-stream bug and no `thinking-events` capability, but this PR head is not mergeable as-is because it changes the reasoning-token protocol/security boundary and regresses current visibility/raw-payload expectations.

## What This Changes

The branch adds a Gateway `thinking-events` client capability, cap-gated thinking-event recipient routing, reasoning delta changes, and related gateway/agent tests plus generated API baseline updates.

## Best Possible Solution

Land either a narrow no-callback reasoning-stream fix or a refreshed cap-gated design that preserves visibility gates, raw core payloads, and recipient cleanup after explicit maintainer protocol review.

## Reproduction Assessment

Yes for the bug portion: source inspection shows current main requires `onReasoningStream` before emitting `stream:"thinking"`, while Gateway `chat.send` does not provide that callback. The cap-gated subscription behavior is new protocol surface rather than an existing-behavior reproduction.

## Solution Assessment

No. The broad cap-gated direction may be useful, but this PR head is not the best merge path until it preserves visibility gates, keeps core thinking payloads raw, finalizes recipient grants on all early failure paths, and gets maintainer agreement on the Gateway capability contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve reasoning visibility gates:** `src/agents/pi-embedded-subscribe.ts:77`
  - body: This sets streamed reasoning from `reasoningMode === "stream"` alone, dropping current main's `canShowReasoning` guard. A client with the new cap could receive reasoning even when `thinkingLevel` is off, so the branch needs to keep that visibility gate while decoupling the callback.
  - confidence: 0.9
- **[P2] Keep core thinking payloads raw:** `src/agents/pi-embedded-subscribe.ts:664-689`
  - body: Current main now treats core reasoning `text` and `delta` as raw payloads and leaves Markdown formatting to channel edges. Reintroducing `formatReasoningMessage(...)` here would send formatted `Reasoning:` Markdown through the Gateway protocol and conflict with that merged contract.
  - confidence: 0.88
- **[P2] Finalize recipients after early dispatch failures:** `src/gateway/server-methods/chat.ts:1558`
  - body: `chat.send` pre-registers thinking recipients before dispatch, but the catch/finally path only deletes the abort controller. If dispatch rejects before lifecycle events run, the run-to-connection grant survives until TTL and can accumulate or misroute if a run id is reused.
  - confidence: 0.76
