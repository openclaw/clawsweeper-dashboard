## What This Changes

The PR adds a Gateway `thinking-events` client capability, cap-scoped routing for thinking agent events, and reasoning-stream delta changes in the embedded Pi stream path.

## Best Possible Solution

Land a coordinated, rebased Gateway reasoning-stream contract that preserves `thinkingLevel` visibility, keeps core Gateway thinking payloads raw, documents any approved `thinking-events` capability, and finalizes recipient grants on every dispatch exit.

## Reproduction Assessment

Yes for the bug portion via source inspection: current main still gates `streamReasoning` on `onReasoningStream`, while Gateway `chat.send` passes no such callback. The `thinking-events` capability itself is a new protocol feature, not a failing current-main behavior.

## Solution Assessment

No, not as currently proposed. Cap-gated delivery may be a valid direction, but this branch should not land until it is rebased and preserves current visibility, raw payload, and recipient-cleanup contracts.
