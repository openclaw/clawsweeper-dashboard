## Summary

Keep open: the requested cap-gated thinking event path is still useful and not implemented on current main, but this PR head is stale against current raw-reasoning and visibility-gate behavior and should not merge as-is.

## What This Changes

The branch adds a `thinking-events` gateway client capability, recipient registry/routing, reasoning delta state changes, gateway tests, and generated baseline updates.

## Best Possible Solution

Add `thinking-events` on top of current main's raw reasoning stream contract, preserving can-show/silent/Control UI visibility gates and covering cap-only gateway delivery with focused tests and protocol documentation.

## Reproduction Assessment

Yes, from source inspection: current main exposes only `tool-events`, still requires `onReasoningStream` before thinking events are emitted, and routes non-tool agent events through the generic broadcast path. I did not run a live model session.

## Solution Assessment

No. The cap-gated routing direction is right, but this branch needs to be rebased and changed to preserve current raw payloads and reasoning visibility gates before it is the best maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Preserve current reasoning visibility gates:** `src/agents/pi-embedded-subscribe.ts:77`
  - body: Current main only streams reasoning when it can be shown, and it also suppresses silent-expected turns before emitting. This branch enables `streamReasoning` for every stream-mode run and its emit path has no `silentExpected` guard, so cap-subscribed clients could receive reasoning current main intentionally hides.
  - confidence: 0.86
- **[P2] Keep thinking-event payloads raw:** `src/agents/pi-embedded-subscribe.ts:664-688`
  - body: Current main's raw-reasoning refactor sends raw reasoning text and deltas from the core stream, leaving channel formatting to the edges. This branch formats accumulated reasoning in core before emitting thinking events/callbacks, which would regress clients and channels that now consume raw reasoning payloads.
  - confidence: 0.88
- **[P3] Clean up pre-registered recipients on dispatch failure:** `src/gateway/server-methods/chat.ts:1553-1558`
  - body: The new pre-dispatch recipient registration can run before any lifecycle end/error event exists to call `markFinal`. If `dispatchInboundMessage` rejects early, the finally path removes the active run but leaves tool/thinking recipient entries alive until TTL, which can accumulate and misroute if an idempotency run id is reused.
  - confidence: 0.72
