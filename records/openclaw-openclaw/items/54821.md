## Summary

Keep open for maintainer review: this PR still contains unique cap-gated Gateway protocol work, but the current head conflicts with current-main reasoning visibility and raw-payload contracts and has a recipient-cleanup gap.

## What This Changes

The PR adds a Gateway `thinking-events` client capability with run-scoped recipient routing and modifies Embedded Pi reasoning streaming/delta handling so subscribed WebSocket clients receive thinking events without an `onReasoningStream` callback.

## Best Possible Solution

Choose one Gateway thinking-stream contract: land the narrow callback fix separately, or rework this cap-gated design on current main with visibility gates, raw payloads, recipient cleanup, docs, and focused protocol tests.

## Reproduction Assessment

Yes. Source inspection shows current main still requires `onReasoningStream` before emitting `stream:"thinking"`, while Gateway `chat.send` does not provide that callback and current main has no `thinking-events` cap.

## Solution Assessment

No. The cap-gated direction may be viable, but this branch is not the best version until it preserves visibility/silent gates, keeps core thinking payloads raw, and finalizes recipient registries on early failure; the narrower callback-only fix is already covered by https://github.com/openclaw/openclaw/pull/79687.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve reasoning visibility gates:** `src/agents/pi-embedded-subscribe.ts:77`
  - body: Current main only streams reasoning when visibility allows it and silent turns are not expected. This branch enables `streamReasoning` from `reasoningMode` alone, so an opted-in Gateway client could receive reasoning that `thinkingLevel:"off"` or silent-turn handling should suppress.
  - confidence: 0.88
- **[P2] Keep thinking payloads raw:** `src/agents/pi-embedded-subscribe.ts:688-689`
  - body: Current main keeps core thinking `text` and `delta` as raw strings, with formatting pushed to channel/display edges. Emitting `formatReasoningMessage(...)` as Gateway `data.text` would reintroduce formatted `Reasoning:\n_..._` payloads for clients consuming the core event stream.
  - confidence: 0.9
- **[P2] Finalize recipients on dispatch failure:** `src/gateway/server-methods/chat.ts:1553-1559`
  - body: The branch registers tool and thinking recipients before dispatch, but if `dispatchInboundMessage` rejects before lifecycle events are emitted, the catch/finally path never marks those recipient registries final. Those entries can survive until TTL and may be reused with the same client run id.
  - confidence: 0.76
