## Summary

Keep open: current main still lacks the cap-gated Gateway thinking-event path, so this PR carries useful remaining work, but the patch is not mergeable as-is because it drops the thinking-off guard, corrupts delta-only reasoning chunks, and leaves stale recipients after early dispatch failure.

## What This Changes

This PR adds a Gateway `thinking-events` client capability, recipient registry/routing, WebSocket cap registration, raw reasoning delta tracking, and related gateway tests/generated baselines.

## Best Possible Solution

Rebase and repair the opt-in Gateway thinking-event implementation on current main, preserving raw reasoning streams, visibility policy, recipient cleanup, and public protocol documentation.

## Reproduction Assessment

Yes, at source level: current main lacks the `thinking-events` cap and still requires `onReasoningStream` for `streamReasoning`, while the PR-head defects are visible in the changed code paths. I did not run a live model/WebSocket reproduction in this read-only review.

## Solution Assessment

No. The opt-in capability direction is right, but this patch is not the best mergeable solution until it preserves the thinking-off guard, handles delta-only chunks, finalizes pre-registered recipients on early failures, and rebases around the raw-reasoning refactor.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Keep the thinking visibility guard:** `src/agents/pi-embedded-subscribe.ts:77`
  - body: Current main only streams reasoning when thinking visibility is enabled, but this patch sets `streamReasoning` solely from `reasoningMode === "stream"`. An opted-in Gateway client could receive reasoning tokens even when the session has thinking disabled, so keep the `params.thinkingLevel !== "off"` guard in the streaming path.
  - confidence: 0.9
- **[P2] Preserve delta-only reasoning chunks:** `src/agents/pi-embedded-subscribe.ts:648-651`
  - body: This replacement branch also receives native delta-only chunks. With prior raw text `Checking` and the next chunk ` files`, there is no significant prefix/suffix overlap, so the code replaces the accumulated text with ` files` and drops the earlier token. The fix needs an explicit full-snapshot versus delta signal, or an append fallback for known delta-only events.
  - confidence: 0.9
- **[P2] Clean up recipients when dispatch fails:** `src/gateway/server-methods/chat.ts:1558`
  - body: `chat.send` now registers tool/thinking recipients before dispatch, but if `dispatchInboundMessage` rejects before any lifecycle `end` or `error`, the catch/finally path never marks those registries final. That leaves stale run-to-connection entries alive until TTL and can misroute events if a client run id is reused.
  - confidence: 0.86
- **[P3] Document the new Gateway capability:** `src/gateway/protocol/client-info.ts:49`
  - body: Adding `THINKING_EVENTS` changes the client handshake contract, but the PR does not update the public Gateway protocol docs to explain that advertising this cap opts into thinking `agent` events. Client authors need that contract documented with the protocol surface.
  - confidence: 0.78
