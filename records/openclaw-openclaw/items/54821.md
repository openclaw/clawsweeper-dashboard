## Summary

Keep open for maintainer review: the Gateway WebSocket thinking-stream bug is real and current main still lacks the new cap, but this PR is not merge-ready because it weakens reasoning visibility, conflicts with the merged raw-payload direction, and leaves stale recipient grants on early dispatch failure.

## What This Changes

The PR adds a Gateway `thinking-events` client capability, run-scoped thinking-event recipient routing, and Embedded Pi reasoning-stream delta changes for WebSocket clients.

## Best Possible Solution

Land a narrow callback-decoupling fix that preserves `canShowReasoning` and raw payloads, then design and document any cap-gated `thinking-events` protocol separately with cleanup and routing tests.

## Reproduction Assessment

Yes. The bug portion is source-reproducible: current main requires `onReasoningStream`, while Gateway `chat.send` does not provide that callback; I did not run a live gateway repro in this read-only review.

## Solution Assessment

No. The cap-gated direction may be useful, but this patch is not the best form until it preserves reasoning visibility, keeps core payloads raw, finalizes early recipient grants, and gets an explicit protocol/security decision.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve reasoning visibility gates:** `src/agents/pi-embedded-subscribe.ts:77`
  - body: Current main only streams reasoning when visibility allows it. This branch enables `streamReasoning` from `reasoningMode` alone, so a Gateway client with the new cap could receive live reasoning even when `thinkingLevel: "off"` should suppress it.
  - confidence: 0.9
- **[P2] Keep core thinking payloads raw:** `src/agents/pi-embedded-subscribe.ts:664-689`
  - body: Current main now keeps core thinking event `text` and `delta` as raw strings after the raw-payload refactor. Emitting `formatReasoningMessage(...)` as Gateway `data.text` would reintroduce formatted `Reasoning:\n_..._` payloads for protocol clients.
  - confidence: 0.88
- **[P2] Finalize pre-registered recipients on dispatch failure:** `src/gateway/server-methods/chat.ts:1558`
  - body: `chat.send` registers tool and thinking recipients before dispatch, but if `dispatchInboundMessage` rejects before lifecycle events are emitted, no `markFinal` call runs and the run-to-connection grant survives until TTL. Finalize those recipient registries in the early failure path too.
  - confidence: 0.78
