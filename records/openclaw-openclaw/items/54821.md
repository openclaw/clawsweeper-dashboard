## Summary

Keep this PR open, but do not merge the current head. The cap-gated thinking-event idea is still unique work on top of current main, but the branch conflicts with current reasoning visibility and raw-payload behavior and needs maintainer protocol/security judgment before a refreshed implementation can land.

## What This Changes

The PR adds a Gateway `thinking-events` client capability, run-scoped thinking-event recipient routing, reasoning-stream delta changes, and related Gateway tests/baseline updates.

## Best Possible Solution

Refresh or replace this with a maintainer-approved Gateway thinking-events protocol that preserves reasoning visibility gates, keeps core thinking payloads raw, finalizes recipient grants on all dispatch failure paths, and documents the client capability; if maintainers only want the narrow callback fix, continue with https://github.com/openclaw/openclaw/pull/79687 instead.

## Reproduction Assessment

Yes for the bug portion: current main still requires `onReasoningStream` before emitting `stream:"thinking"`, while Gateway `chat.send` does not pass that callback. I did not run a live current-main Gateway repro in this read-only review.

## Solution Assessment

No. Cap-gated delivery may be the right direction, but this PR head is not the best merge path until the protocol/security contract is approved and the branch is updated for current visibility, raw-payload, and cleanup behavior.
