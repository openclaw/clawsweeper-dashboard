## Summary

Keep open. Current main still does not implement cap-gated Gateway thinking events, but the current PR head has blocking correctness/security issues and conflicts with the raw reasoning payload direction already merged on main.

## What This Changes

The PR adds a Gateway `thinking-events` client capability, run-scoped thinking-event recipients, and revised embedded Pi reasoning delta emission for WebSocket clients.

## Best Possible Solution

Land a narrow current-main implementation that decouples Gateway thinking delivery from the optional callback while preserving `canShowReasoning`, raw core payloads, run-scoped cleanup, and protocol tests for any new capability.

## Reproduction Assessment

Yes. Source inspection on current main shows Gateway thinking events still depend on `onReasoningStream`, which callback-less WebSocket Gateway runs do not supply, and the `thinking-events` capability does not exist.

## Solution Assessment

No. The cap-gated direction may be useful, but this PR is not the best merge path until it is rebased and reconciled with current visibility gates, raw payload contracts, and recipient cleanup behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve reasoning visibility gates:** `src/agents/pi-embedded-subscribe.ts:77`
  - body: This makes streamed reasoning depend on `reasoningMode === "stream"` alone. Current main also gates streamed reasoning on `canShowReasoning`, so a Gateway client advertising the new cap could receive thinking tokens when `thinkingLevel: "off"` should hide them.
  - confidence: 0.88
- **[P2] Keep core thinking payloads raw:** `src/agents/pi-embedded-subscribe.ts:688-689`
  - body: The PR sends `data.text` as `formatReasoningMessage(rawAccumulated)`. Current main moved formatting out of the core reasoning stream, so this would put `Reasoning:\n_..._` back onto the Gateway protocol instead of preserving raw text for clients to format at the edge.
  - confidence: 0.86
- **[P2] Finalize recipients after early dispatch failures:** `src/gateway/server-methods/chat.ts:1558`
  - body: `chat.send` pre-registers thinking recipients before dispatch, but an early `dispatchInboundMessage` rejection can happen before lifecycle end/error calls `markFinal`. That leaves stale run-to-connection grants alive until TTL and can accumulate during repeated early failures.
  - confidence: 0.74
