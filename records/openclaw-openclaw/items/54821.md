## Summary

Keep open: the PR addresses a real source-reproducible Gateway thinking-stream gap, but the current branch is conflicting and still introduces reasoning-token protocol/security regressions called out in the prior maintainer-side review.

## What This Changes

The PR adds a Gateway `thinking-events` client capability, run-scoped recipient routing, and Embedded Pi reasoning-stream delta changes for WebSocket clients.

## Best Possible Solution

Land a narrow implementation that keeps reasoning visibility gates and raw core payloads, then document and test any cap-gated Gateway thinking-event protocol before merge.

## Reproduction Assessment

Yes. The bug portion is source-reproducible because current main requires `onReasoningStream` while Gateway `chat.send` does not supply that callback; I did not run a live streaming-model repro in this read-only review.

## Solution Assessment

No. The cap-gated direction may be useful, but this branch is not the best merge shape until it preserves reasoning visibility, keeps core payloads raw, finalizes early recipient grants, and gets an explicit protocol/security decision.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve reasoning visibility gates:** `src/agents/pi-embedded-subscribe.ts:77`
  - body: Current main only streams reasoning when `thinkingLevel` allows it. This branch enables `streamReasoning` from `reasoningMode` alone, so a Gateway client with the new cap could receive reasoning tokens that `thinkingLevel: "off"` is supposed to hide.
  - confidence: 0.9
- **[P2] Keep core thinking payloads raw:** `src/agents/pi-embedded-subscribe.ts:664-689`
  - body: Current main moved core reasoning streams to raw text/delta payloads. Emitting `formatReasoningMessage(...)` as Gateway `data.text` would reintroduce formatted `Reasoning:\n_..._` payloads for protocol clients and conflict with the current edge-formatting direction.
  - confidence: 0.88
- **[P2] Finalize recipients on early dispatch failure:** `src/gateway/server-methods/chat.ts:1558`
  - body: `chat.send` registers tool and thinking recipients before dispatch, but if dispatch rejects before lifecycle events are emitted, no `markFinal` call runs and the run-to-connection grant survives until TTL. Finalize those registries in the early failure path too.
  - confidence: 0.78
