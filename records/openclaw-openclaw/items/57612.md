Full review comments:

- **[P1] Route DOCX parsing through the extractor contract:** `src/media/input-files.ts:400-405`
  - body: Current main routes document parsing through `extractDocumentContent` and the `document-extract` plugin, but this branch calls a new core DOCX helper directly from `input_file` and `read`. That creates a second untrusted document parser path with separate limits and ownership; move DOCX into the document-extractor contract or get explicit maintainer approval for a core exception.
  - confidence: 0.9
- **[P1] Check DOCX XML size before inflating it:** `src/media/docx-extract.ts:71-76`
  - body: The guard compares `documentXmlBytes.byteLength` only after `documentXmlFile.async("uint8array")` has accumulated the inflated XML. A small compressed DOCX can still expand into large memory before the limit applies, so the uncompressed bound needs to be enforced before or during extraction.
  - confidence: 0.91
- **[P2] Bound DOCX reads before full buffering:** `src/agents/pi-tools.read.ts:408`
  - body: For any `.docx` path, this calls `readBufferForToolPath` and loads the whole file before MIME detection or DOCX-specific limits. A large local document can bypass the adaptive read budget and pressure memory, so stat or bounded-read the file before parsing.
  - confidence: 0.87
