## Summary

Keep open. Current main still lacks native DOCX extraction for both `input_file` and `read`, so the PR addresses a real product gap, but the proposed implementation is not ready to merge because it inflates untrusted DOCX XML before the byte cap applies and changes the public `input_file` MIME contract without matching docs; it also needs maintainer judgment on routing this through the existing document-extractor plugin boundary.

## What This Changes

This PR adds a JSZip-based DOCX text extractor, allows DOCX in `input_file`, wires DOCX extraction into the `read` tool, and adds paragraph/table/entity/capping integration tests.

## Best Possible Solution

Land DOCX support through the existing document-extractor contract, preferably by extending the bundled `document-extract` plugin with a DOCX extractor, bounded ZIP/XML handling before materialization, default MIME/docs updates, and focused `input_file` plus `read` regression coverage.

## Reproduction Assessment

Yes. The current-main gap is reproducible by source inspection: `input_file` defaults exclude DOCX, native extraction only special-cases PDF, and the read wrapper has no DOCX path. The PR security issue is reproducible from the patch because the byte guard runs after `word/document.xml` is inflated.

## Solution Assessment

No, not as-is. The feature direction is useful, but the safer maintainable path should enforce limits before inflating XML and should use the plugin-owned document-extractor seam instead of adding a separate core-only parser.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Stream DOCX XML before enforcing byte caps:** `src/media/docx-extract.ts:76`
  - body: `documentXmlFile.async("uint8array")` inflates the whole `word/document.xml` entry before `maxXmlBytes` is checked. A small compressed DOCX can therefore allocate a large XML payload in `input_file` or `read` before the guard runs; add a preflight uncompressed-size check and/or streaming byte limit before decoding.
  - confidence: 0.9
- **[P3] Document DOCX input_file MIME support:** `src/media/input-files.ts:110`
  - body: Adding DOCX to the default `input_file` allowlist changes the public OpenResponses file contract, but the docs still list only text/markdown/html/csv/json/pdf. Please update the OpenResponses docs with the new MIME and behavior.
  - confidence: 0.82
