## Solution Assessment

No as-is. The maintainable direction is to use the current document-extractor contract and bounded parser limits rather than adding a parallel core-native DOCX parser on a conflicting branch.

Full review comments:

- **[P1] Route DOCX parsing through the extractor seam:** `src/media/input-files.ts:398-404`
  - body: Current main moved parser-backed document extraction into the `DocumentExtractorPlugin` contract, with PDF routed through `extractDocumentContent`. Calling a new DOCX parser directly from `input_file` creates a second parser ownership path that bypasses plugin enablement/deny policy and will drift from the existing document-extract boundary.
  - confidence: 0.88
- **[P1] Check XML size before accumulating DOCX content:** `src/media/docx-extract.ts:71`
  - body: `documentXmlFile.async("uint8array")` resolves only after JSZip has accumulated the full uncompressed entry, so the byte check that follows cannot prevent a small compressed DOCX from expanding into large memory. Use a bounded extraction path or stream/metadata guard before materializing `word/document.xml`.
  - confidence: 0.9
- **[P2] Avoid full-file reads for DOCX read normalization:** `src/agents/pi-tools.ts:393-394`
  - body: The read wrapper re-reads `.docx` files with `fs.readFile`, so large local archives are fully buffered before the adaptive read output cap can help. The parser path needs input-size enforcement before this full read, ideally through the shared document-extractor flow.
  - confidence: 0.82
