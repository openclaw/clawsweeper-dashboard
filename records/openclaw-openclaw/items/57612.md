## Summary

Keep open. Current main still does not implement native DOCX extraction for `input_file` or `read`, and the PR contains useful follow-up work from the discussion, but the latest diff still needs maintainer/author attention before merge because untrusted DOCX XML is inflated before the byte cap is enforced and the implementation likely needs to fit the existing plugin-owned document extraction seam.

## What This Changes

This PR adds a JSZip-based DOCX text extractor, allows DOCX for `input_file`, wires DOCX extraction into the `read` tool, and adds regression coverage for paragraphs, tables, entities, caps, `input_file`, and `read`.

## Best Possible Solution

Land DOCX support through the existing document-extractor contract, ideally in the bundled `document-extract` plugin, with DOCX MIME defaults/docs, bounded streaming or preflight ZIP/XML limits, resilient malformed-archive fallback, and focused `input_file` plus `read` tests preserving the useful coverage from this PR.

## Reproduction Assessment

Yes. The current-main gap is reproducible by source inspection: default `input_file` MIME support excludes DOCX, the file extraction path only special-cases PDF, and the read wrapper has no DOCX extraction branch. The PR security finding is also reproducible by code inspection because the proposed XML byte guard runs after JSZip has inflated `word/document.xml` into memory.

## Solution Assessment

No, not as-is. The requested DOCX text extraction is a good narrow feature, but the safest maintainable solution should enforce ZIP/XML limits before materializing the XML and route the extractor through the existing plugin-owned document extraction seam instead of adding a core-only parser.
