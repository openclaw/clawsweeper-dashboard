## Best Possible Solution

Implement DOCX as a bounded document-extractor capability, route `input_file` and `read` through that shared contract, enforce compressed and uncompressed limits before materialization, then add real behavior proof.

## Reproduction Assessment

Yes for the blocking findings, by source inspection. PR-head source shows direct core DOCX parsing and `documentXmlFile.async("uint8array")` before the byte check, and JSZip v3.10.1 source shows `async()` accumulates chunks before resolving; I did not run tests because this review is read-only.

## Solution Assessment

No. The safer maintainable path is a bounded DOCX document-extractor capability reused by `input_file` and `read`, not a separate core-native parser branch with post-inflation limits.
