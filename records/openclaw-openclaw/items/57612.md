## Summary

Keep open. Current main still lacks native DOCX extraction, but this PR is not merge-ready because the DOCX path can materialize untrusted ZIP/XML before effective limits, the read path adds an unbounded full-file buffer read for `.docx`, public docs are not updated, and contributor real-behavior proof is still missing.

## What This Changes

The PR adds a JSZip-based DOCX text extractor, routes DOCX through `input_file` and `read`, expands the default MIME allowlist, and adds focused regression tests.

## Best Possible Solution

Land DOCX support through the existing document-extractor seam or an explicitly approved core path, with pre-materialization compressed-file and XML-entry limits, public docs, and real behavior proof for both `read` and `input_file`.

## Reproduction Assessment

Yes for the blocking review findings: PR-head source shows full DOCX file/XML materialization before effective caps, and JSZip 3.10.1 source confirms `async()` accumulates the entry in memory. I did not run tests because this review is read-only.

## Solution Assessment

No, not as-is. The feature direction is useful, but the safer solution needs bounded ZIP/XML parsing, docs for the new public contract, and maintainer agreement on core-native parsing versus the document-extractor plugin boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Enforce DOCX XML limits before inflation:** `src/media/docx-extract.ts:71`
  - body: `documentXmlFile.async("uint8array")` inflates and accumulates the whole `word/document.xml` entry before `maxXmlBytes` is checked. A compressed DOCX that passes the outer file cap can still expand into a large XML payload in `input_file` or `read`, so enforce an uncompressed entry or streaming byte limit before materializing the XML.
  - confidence: 0.9
- **[P2] Bound DOCX reads before full-file buffering:** `src/agents/pi-tools.read.ts:408`
  - body: The `.docx` read branch calls `readBufferForToolPath` for the whole file after adaptive paging has already returned a bounded result, and there is no compressed file-size cap before parsing. Reading a large DOCX can now bypass the read tool's byte budget and load the full archive into memory; add a bounded/stat-checked read or fail back to the original result when the file is too large.
  - confidence: 0.86
- **[P3] Document the DOCX input_file contract:** `src/media/input-files.ts:106`
  - body: Adding the DOCX MIME type to `DEFAULT_INPUT_FILE_MIMES` changes the public OpenResponses `input_file` allowlist, but the docs still list only text/CSV/JSON/PDF and describe only PDF parsing. Please document the DOCX MIME type, extraction scope, and limits.
  - confidence: 0.84
