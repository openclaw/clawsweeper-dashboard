## Summary

Keep open. The PR addresses a real current-main DOCX gap, but it should not merge as-is because the new untrusted DOCX XML path inflates before enforcing the XML cap and still needs maintainer direction on whether this belongs in the existing document-extract plugin seam.

## What This Changes

This PR adds a JSZip-based DOCX text extractor, allows DOCX as a default `input_file` MIME, routes DOCX through `read`, and adds focused unit and integration tests.

## Best Possible Solution

Extend the bundled `document-extract` plugin with a bounded DOCX extractor, then wire `input_file` and `read` through that seam with docs, changelog, and focused regression coverage.

## Reproduction Assessment

Yes. Source inspection shows current main supports PDF-only native `input_file` extraction and has no DOCX `read` branch; the PR's security issue is visible in the patch because the XML byte cap is checked after JSZip inflates `word/document.xml`.

## Solution Assessment

No, not as-is. The feature direction is useful, but the maintainable path should enforce limits before XML materialization and should likely extend the existing document-extract plugin contract instead of adding a separate core-only parser.
