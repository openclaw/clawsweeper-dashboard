## Summary

Keep open. This PR targets a valid documented bug, but the latest diff still leaves the Control UI unable to derive DeepSeek V4 Pro `xhigh`/`max` from current lightweight `sessions.list` rows when the active session differs from defaults.

## What This Changes

This PR updates TUI and Control UI `/think` fallback logic and tests, plus a changelog entry, so session-specific thinking levels are preferred over mismatched defaults.

## Best Possible Solution

Revise the Control UI path so the active session gets plugin-owned thinking levels from a gateway-backed contract or deliberate active-row hydration, then cover the current lightweight-row case in slash-command and picker tests.

## Reproduction Assessment

Yes. Source inspection shows current `sessions.list` rows can carry empty thinking arrays while the browser fallback ignores provider/model, so a DeepSeek V4 Pro active session can still collapse to base levels without a server-provided row profile.

## Solution Assessment

No. The proposed TUI empty-array fallback is narrow, but the Control UI fallback still relies on a provider-agnostic browser helper instead of the gateway/plugin profile source needed for DeepSeek `xhigh` and `max`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Hydrate provider levels before slash-command fallback:** `ui/src/ui/chat/slash-command-executor.ts:669-672`
  - body: When the active session differs from defaults, current `sessions.list` rows are lightweight and carry empty thinking arrays. This fallback calls the browser `formatThinkingLevels`, but `ui/src/ui/thinking.ts` ignores provider/model and returns only base levels, so `/think` still omits and rejects `max` for a DeepSeek V4 Pro session unless the test stubs a populated row that the real endpoint does not provide.
  - confidence: 0.91
- **[P2] Use plugin-owned levels for the thinking picker:** `ui/src/ui/chat/session-controls.ts:196`
  - body: This picker fallback has the same empty-row problem: after skipping mismatched defaults, `listThinkingLevelLabels(provider, model)` in the browser returns only the base list. A DeepSeek V4 Pro active row from the current lightweight `sessions.list` path will still render without `xhigh` and `max`; the picker needs a gateway-backed/provider-profile source for the active session.
  - confidence: 0.89
