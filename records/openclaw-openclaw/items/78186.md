## Summary

Keep open: current main still blocks LINE webhook ACKs, and this PR is a focused ACK-first fix with sufficient terminal proof and green current-head checks; no repair lane is needed.

## What This Changes

This PR changes LINE Node, Express, and monitor webhook paths to send HTTP 200 after validation, run event processing asynchronously with error logging, update receive ACK metadata/tests, and add a changelog entry.

## Best Possible Solution

Land this ACK-first LINE fix once maintainers are ready, keeping signature and payload validation before ACK and treating durable queue or burst backpressure as separate follow-up work if production volume shows a need.

## Reproduction Assessment

Yes. Source inspection on current main shows a valid signed LINE POST with events awaits `handleWebhook` or `onEvents` before writing HTTP 200 in the Node, Express, and monitor handlers, so a pending agent turn keeps the ACK unsent.

## Solution Assessment

Yes. Acknowledging immediately after signature and payload validation is the narrow maintainable fix, matches the SDK `after_receive_record` policy and Telegram precedent, and avoids changing reply-token, dedup, or validation behavior.
