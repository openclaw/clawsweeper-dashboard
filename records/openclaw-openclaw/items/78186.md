## Summary

Keep open. This PR is a focused fix for a source-reproducible LINE delayed-ACK bug that is still present on current main, and I found no blocking correctness or security findings in the proposed patch.

## What This Changes

The PR changes LINE webhook handlers and ACK metadata to send HTTP 200 after trusted validation, process events asynchronously with runtime error logging, and adds regression tests plus a changelog entry.

## Best Possible Solution

Land this focused LINE ACK-first fix after maintainer approval, preserving pre-ACK signature/body/target validation and leaving durable queue or burst backpressure work to separate follow-ups.

## Reproduction Assessment

Yes. A signed LINE POST with events and a pending `handleWebhook` or `onEvents` promise exercises the current-main delay, and existing source/tests show the HTTP 200 is held until processing resolves.

## Solution Assessment

Yes. ACKing immediately after trusted validation and moving only agent processing behind a caught async boundary is the narrow maintainable fix, matches the receive ACK contract, and follows the existing Telegram webhook pattern.
