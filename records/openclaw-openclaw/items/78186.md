## Summary

Keep open for maintainer merge review. Current main still blocks the LINE webhook HTTP 200 behind event processing, while this PR applies the narrow ACK-first fix across the Node, Express, and monitor paths with matching tests, green CI, and sufficient real behavior proof.

## What This Changes

This PR changes LINE webhook handlers to send HTTP 200 after signature and payload validation, process events asynchronously with runtime error logging, update LINE receive ACK metadata/tests, and add a changelog entry.

## Best Possible Solution

Land the ACK-first LINE fix after normal maintainer review, keeping signature and payload validation before ACK and leaving durable queue/backpressure work as a separate follow-up if needed.

## Reproduction Assessment

Yes. A valid signed LINE POST whose `handleWebhook` or `onEvents` promise stays pending reproduces the current-main source path: HTTP 200 is not sent until that promise resolves.

## Solution Assessment

Yes. ACKing immediately after trusted validation is the narrowest maintainable fix for the LINE timeout, matches the existing message receive policy and Telegram precedent, and avoids changing signature validation, reply-token handling, or dedup semantics.
