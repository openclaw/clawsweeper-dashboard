## Summary

Keep open for maintainer landing: current main still blocks LINE webhook ACKs, this PR is the focused fix with sufficient real behavior proof and green PR checks, and I found no correctness or security blocker in the diff.

## What This Changes

The PR changes LINE webhook handlers, ACK metadata, tests, and changelog so HTTP 200 is sent after validation while agent processing continues asynchronously.

## Best Possible Solution

Land this focused LINE ACK-first fix after maintainer approval, preserving pre-ACK validation and leaving durable queue or burst backpressure work to a separate follow-up if needed.

## Reproduction Assessment

Yes. A signed LINE POST with events and a pending `handleWebhook` or `onEvents` promise is enough to show current main awaits event processing before HTTP 200 in all three LINE ingress paths; existing tests also assert the response remains unsent while processing is pending.

## Solution Assessment

Yes. ACKing immediately after signature/payload/target validation and moving only trusted event processing behind a caught async boundary is the narrow maintainable fix, matches the receive ACK contract, and follows the existing Telegram pattern.
