## Summary

Keep this PR open for normal maintainer review; the LINE webhook timeout bug is source-reproducible on current main, the patch is narrow, and I did not find a blocking correctness or security issue in the diff.

## What This Changes

The PR changes LINE Node, Express, and multi-account monitor webhook paths to acknowledge valid webhook requests before asynchronous agent processing, updates LINE ACK metadata/tests, and adds a changelog entry.

## Best Possible Solution

Land the ACK-first LINE webhook fix after maintainer approval, keeping signature and payload validation before ACK and leaving durable queue or burst backpressure design for a separate follow-up if production evidence warrants it.

## Reproduction Assessment

Yes. A signed LINE POST with events and a pending `handleWebhook` or `onEvents` promise on current main keeps HTTP 200 unsent because each LINE webhook path awaits agent processing before ACK.

## Solution Assessment

Yes. ACKing immediately after signature and payload validation, then running agent processing asynchronously with error logging, is the narrowest maintainable fix and matches the existing receive ACK policy plus Telegram precedent.
