## Summary

Keep open. Current main still blocks LINE HTTP 200 on event processing, and this PR is a focused ack-first fix across the Node, Express, and monitor webhook paths with matching ACK metadata, regression coverage, supplied real behavior proof, green CI, and no blocking review findings.

## What This Changes

The PR changes LINE webhook handlers to return HTTP 200 after signature/payload validation, process events asynchronously with runtime error logging, and updates ACK metadata, tests, and changelog.

## Best Possible Solution

Land this focused LINE ACK-first fix after maintainer approval, preserving pre-ACK validation and leaving durable queue or burst backpressure work to a separate follow-up if needed.

## Reproduction Assessment

Yes. A signed LINE POST with events and a pending `handleWebhook` or `onEvents` promise is enough: current main awaits that promise before sending HTTP 200 in all three LINE ingress paths.

## Solution Assessment

Yes. ACKing immediately after trusted signature/payload/target validation and moving only agent processing behind a caught async boundary is the narrow maintainable fix, matches the existing receive ACK policy contract, and follows the Telegram precedent.
