## Summary

Keep open: current main has Telegram request timeout machinery, but the reported CLI/action delete path still awaits `api.deleteMessage` through the generic retry runner without applying OpenClaw's per-method Telegram guard. Upstream grammY has a default 500-second client timeout, so the exact “forever” wording is not fully proven on current main, but the user-visible CLI hang remains a valid bounded-timeout bug.

## What This Changes

The issue asks for Telegram message deletion to fail with a bounded timeout instead of leaving `openclaw message delete --channel telegram` stuck when the Bot API call does not settle.

## Best Possible Solution

Apply the existing Telegram per-method request timeout policy to CLI/tool action clients, especially `deleteMessage`, while preserving no-op delete warnings and retry behavior for recoverable network errors.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection gives a clear reproduction shape: a non-settling `api.deleteMessage` promise is awaited through `retryAsync` without OpenClaw's per-method timeout guard, while upstream grammY would otherwise wait up to its 500-second default.

## Solution Assessment

No, adding a timeout directly to generic `retryAsync` is broader than necessary for this report. The narrower maintainable fix is to make Telegram-owned request wrapping use the existing method timeout table for action/send clients and cover `deleteMessage` with a focused regression test.
