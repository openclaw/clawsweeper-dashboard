## Summary

Keep open. Current main has a CLI-side gateway timeout and grammY-backed request timeout mitigation, but the Telegram delete helper still awaits a non-settling `api.deleteMessage` through the retry wrapper with no operation-level timeout, so the reported root cause is still source-plausible.

## What This Changes

The issue asks for Telegram message deletion to fail with a bounded timeout instead of hanging when the Bot API `deleteMessage` call never settles.

## Best Possible Solution

Add a Telegram adapter-level timeout or cancellation guard for delete/control API calls, preserving grammY throttling and avoiding generic retries for non-idempotent sends.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection shows a non-settling `api.deleteMessage` would remain awaited in `deleteMessageTelegram`/`retryAsync`, but I did not run a real Telegram DM delete against current main.

## Solution Assessment

No, the proposed generic `retryAsync` timeout is not clearly the safest fix because shared retry code also wraps non-idempotent sends. A narrower Telegram delete/control-operation timeout is the safer first repair.
