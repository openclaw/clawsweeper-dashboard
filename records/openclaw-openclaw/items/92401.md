## Label Justifications

- `P2`: The report describes noisy operational task health output and false session-state failure signals with limited but real operator impact.
- `impact:session-state`: The affected audit output can misclassify active subagent/task state as failed, stale, or historically blocked.

## Summary

Keep open: current `main` and the latest release still use fixed task and TaskFlow audit thresholds, do not carry subagent `runTimeoutSeconds` into task audit records, and do not provide the requested freshness/history/grouping contract. https://github.com/openclaw/openclaw/issues/83253 overlaps on retained historical task noise, but this issue also has distinct timeout-aware long-running subagent and audit-output requirements.

## What This Changes

The issue asks for `openclaw tasks audit` to stop reporting healthy long-running subagents and retained historical task records as active failures, and to reduce timestamp/audit noise.

## Best Possible Solution

Define one task-audit health contract that keeps current/live failures prominent, treats long-running subagents according to their configured deadline, preserves inspectable history behind explicit filters, and updates CLI/JSON/docs/tests together.

## Reproduction Assessment

Yes, at source level: current task and TaskFlow audits use fixed 30-minute stale thresholds while subagent `runTimeoutSeconds` lives only on subagent run records, not on the task records the audit reads. I did not run a live stateful gateway reproduction because this review is read-only.

## Solution Assessment

Unclear as a full solution: making stale checks timeout-aware is the narrowest bug-fix direction, but `--fresh`, `--include-historical`, grouping, and suppression defaults need a maintainer-approved CLI/JSON compatibility contract.
