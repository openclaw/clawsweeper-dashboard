## Summary

Keep open: current main and the latest release still use fixed task/TaskFlow audit thresholds and do not provide the requested fresh/historical split, so the false-positive audit problem is not already implemented. The related historical-noise issue at https://github.com/openclaw/openclaw/issues/83253 remains useful context but does not cover the timeout-aware long-running subagent case.

## What This Changes

Request that `openclaw tasks audit` reduce false positives by respecting subagent `runTimeoutSeconds`, aggregating timestamp noise, separating historical findings, and improving the human summary.

## Best Possible Solution

Define a maintainer-approved task health model where default audit output reports live operational risk, historical findings remain inspectable behind explicit filters, and long-running subagents are not marked stale before their configured timeout expires.

## Reproduction Assessment

Yes, source inspection gives a high-confidence repro path: create running task or TaskFlow records older than 30 minutes while the associated subagent timeout is longer, then `tasks audit` will use the fixed audit window rather than the configured timeout. I did not run a live gateway reproduction because this review is read-only.

## Solution Assessment

Unclear as a complete solution: respecting `runTimeoutSeconds` is the right narrow bug boundary, but grouping timestamp findings, adding fresh/historical flags, and changing default output need maintainer approval for CLI/JSON compatibility.
