## Summary

Keep open: current main and v2026.6.9 still use fixed task and TaskFlow audit thresholds, task audit records do not carry subagent runTimeoutSeconds, and the requested freshness/history/grouping contract is still missing. https://github.com/openclaw/openclaw/issues/83253 overlaps on retained historical task noise, while https://github.com/openclaw/openclaw/issues/44925 is adjacent on subagent delivery loss, but neither owns this issue's timeout-aware audit behavior.

## What This Changes

Requests a quieter, timeout-aware openclaw tasks audit that respects subagent runTimeoutSeconds, groups timestamp warnings, separates fresh from historical rows, and adds a severity-aware CLI summary.

## Best Possible Solution

Define one task-audit health model that carries timeout/deadline facts into audit decisions, separates current from historical findings, groups repeated timestamp noise, and keeps an explicit all/history mode for operators.

## Reproduction Assessment

Yes at source level: current main uses fixed 30 minute task and TaskFlow audit thresholds while subagent runTimeoutSeconds is stored on subagent run records and not on TaskRecord. I did not run the reporter's live multi-subagent workload in this read-only review.

## Solution Assessment

Unclear: the timeout-aware bug fix is well grounded, but the requested --fresh/default suppression, grouping, and human/JSON summary semantics need maintainer product direction before implementation.
