## Summary

Keep this issue open: current main and release v2026.6.5 retain the fixed 30-minute stale-running rule despite timeout-aware sibling logic, while the historical-noise portion should be coordinated with the existing canonical issue at https://github.com/openclaw/openclaw/issues/83253.

## What This Changes

Reduce false task-audit failures for long-running subagents and separate actionable current findings from timestamp and retained-history noise.

## Best Possible Solution

Persist or otherwise carry the canonical timeout/liveness fact into task state, repair timestamp anomalies at their write boundary, preserve delivery failures as a distinct signal, and implement the shared current-versus-retained presentation model through the canonical related issue.

## Reproduction Assessment

Yes, partially: current source provides a high-confidence reproduction of the fixed 30-minute stale-running decision and unsuppressed output, but not of the reporter's exact 154-row dataset or the asserted cause of its delivery_failed rows.

## Solution Assessment

No, not as one bundled implementation: timeout-aware stale classification is the right direction, but timestamp normalization, terminal delivery, grouping, and historical defaults require distinct root-cause and compatibility treatment.
