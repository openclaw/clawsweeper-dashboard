## Summary

Keep this open: current main still emits flat task and TaskFlow audit findings with fixed 30-minute staleness thresholds, while the historical-record concern overlaps but is not fully covered by an existing issue. The report’s timeout explanation for `delivery_failed` is not established, because that code represents failed terminal-update delivery rather than a running-task liveness timeout.

## What This Changes

Reduce noisy and misleading task-audit findings for long-running subagents and retained historical task or TaskFlow records.

## Best Possible Solution

Split the work into a focused audit-classification design: use canonical lifecycle deadlines for genuine active-staleness findings, preserve `delivery_failed` as a distinct terminal-delivery condition, aggregate repetitive anomalies for human output while retaining raw JSON diagnostics, and coordinate historical-state semantics with https://github.com/openclaw/openclaw/issues/83253.

## Reproduction Assessment

Partly yes: current source clearly reproduces fixed 30-minute thresholds and flat per-record findings, but it does not establish that `delivery_failed` is caused by long-running healthy execution, and the exact 154-finding live dataset was not replayed.

## Solution Assessment

No, not as one bundled implementation. Deadline-aware stale detection and human-output aggregation may be appropriate, but new freshness flags, default suppression, historical classification, and `delivery_failed` semantics require separate contract decisions and coordination with the existing historical-task issue.

## Risks / Open Questions

- The proposed timeout treatment conflates `delivery_failed`, which represents failed terminal-update delivery, with running-task liveness; changing that finding to wait for a run deadline could hide genuine delivery failures.
- Adding default freshness filtering or changing suppression behavior could alter output consumed by existing JSON watchdog scripts.
- Task records and TaskFlow records expose different lifecycle metadata, so deadline-aware classification needs a canonical cross-owner contract rather than copying subagent-specific timeout fields into every audit path.
- The claimed downstream data loss is plausible for an aggressive external watchdog, but no concrete current-main automation terminating or respawning healthy sessions was provided.
