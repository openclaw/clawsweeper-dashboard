## Label Justifications

- `P2`: The issue reports noisy operational task health output and false long-running subagent failure signals with limited but real operator impact.
- `impact:session-state`: The affected audit output can misclassify active subagent/task state as failed, stale, or historically blocked.

## Summary

Keep open: current `main` and the latest release still use fixed task/TaskFlow audit thresholds, do not expose subagent run deadlines on task audit records, and do not provide the requested freshness/history/grouping contract. The related historical-noise issue overlaps, but it does not cover the long-running subagent timeout behavior or the proposed audit-output changes.

## What This Changes

The issue asks for task audit semantics that distinguish healthy long-running subagents and recent failures from historical timestamp and retained-task noise.

## Best Possible Solution

Define one maintainer-approved task-audit health contract that keeps current/live failures prominent, treats long-running subagents according to their configured deadline, preserves inspectable history behind explicit filters, and updates CLI/JSON/docs/tests together.

## Reproduction Assessment

Yes, at source level: current task and TaskFlow audits use fixed stale thresholds, while subagent run timeout metadata stays on subagent run records and is not part of the task audit records. The full live state database from the report was not recreated in this read-only review.

## Solution Assessment

Unclear as a full solution: timeout-aware stale checks are the narrow bug-fix direction, but `--fresh`, `--include-historical`, grouping, and suppression defaults need a maintainer-approved CLI/JSON compatibility contract.
