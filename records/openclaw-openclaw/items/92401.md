## Summary

Keep open: current main and v2026.6.9 still use fixed task and TaskFlow audit thresholds, do not carry subagent runTimeoutSeconds into TaskRecord audit decisions, and lack the requested fresh/history/grouping CLI semantics. Related issues cover historical retention or broader delivery loss, but neither owns this issue's specific timeout-aware audit-health model.

## What This Changes

The issue requests quieter, timeout-aware `openclaw tasks audit` output for long-running subagents, timestamp noise grouping, freshness/history filtering, and severity-aware summaries.

## Best Possible Solution

Define one maintainer-approved task-audit health model that carries timeout/deadline facts into audit decisions, separates current from historical findings, groups repeated timestamp noise, and preserves an explicit all/history mode for operators.

## Reproduction Assessment

Yes, at source level: current main uses fixed 30-minute task and TaskFlow audit thresholds while subagent runTimeoutSeconds is stored on subagent run records and not on TaskRecord. I did not run the reporter's live four-subagent workload in this read-only review.

## Solution Assessment

Unclear: timeout-aware stale detection is well grounded, but the requested fresh/history defaults, grouping, summary semantics, and any persisted task-shape change need maintainer product direction before implementation.
