## Label Justifications

- `P2`: The issue describes a plausible false health signal in task/session monitoring with limited blast radius and no confirmed core outage.
- `impact:session-state`: The affected audit path can misclassify healthy long-running subagent or TaskFlow state as stale, which is session-state drift for operators and automation.

## Summary

Keep open: current main and latest release still expose fixed-threshold task and TaskFlow audit paths that can produce stale/dead-agent noise for long-running work, while related open work only covers parts of the requested behavior.

## What This Changes

The issue requests less noisy `openclaw tasks audit` output by respecting long-running subagent timeouts, grouping timestamp findings, filtering historical rows, and improving the human summary.

## Best Possible Solution

Define one maintainer-approved task health model that makes default audit/status output operationally current, preserves inspectable history behind explicit filters, and shares timeout-aware liveness rules with subagent run state.

## Reproduction Assessment

Yes, source-reproducible: current main flags running tasks and TaskFlows after fixed 30-minute audit thresholds, while the task audit path lacks `runTimeoutSeconds` and the CLI lacks fresh/historical filters. I did not reproduce the reporter's full live 154-finding state.

## Solution Assessment

Unclear as a complete solution: respecting timeout-aware liveness is the right direction, but grouping, default suppression, and new `--fresh` or historical flags need maintainer-approved CLI and JSON semantics.
