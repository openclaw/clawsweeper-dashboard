## Summary

Keep open: the requester-vs-target announce routing problem is still present on current main, but this PR head still has a prompt/destination mismatch and lacks contributor-supplied real behavior proof, so it should not merge as-is.

## What This Changes

The PR changes `sessions_send` A2A announce delivery to resolve the final send target from the requester session, separates target-side channel metadata, and adds requester-vs-target regression coverage.

## Best Possible Solution

Land a rebased narrow fix that aligns final delivery and model-facing announce prompt with the actual requester audience while preserving target provenance, threadId delivery, skip-token filtering, and regression coverage.

## Reproduction Assessment

Yes, at source level: current main resolves the announce target from the target/display session and the current sessions test expects `group:target`. I did not run tests in this read-only review.

## Solution Assessment

No, not as the current PR head. The requester-routing direction is plausible, but the announce prompt still describes target-channel delivery while the final send would go to the requester, and the external PR still needs real behavior proof.
