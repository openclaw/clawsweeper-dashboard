Keep this open. Current main still has the `unknown requestId` path, and the issue is recent with unresolved evidence about whether the reporter used a device ID/stale ID or a valid current request ID that disappeared during approval.

## What This Changes

The issue reports that `openclaw devices approve <deviceId>` fails with `unknown requestId` while device listing and gateway health still work.

## Best Possible Solution

Land or supersede https://github.com/openclaw/openclaw/pull/80779 for the stale-approve diagnostic, keep deterministic recovery tracked in https://github.com/openclaw/openclaw/issues/74484, and ask for exact list/approve evidence before changing approval semantics.

## Solution Assessment

Unclear. The narrow diagnostic fix in https://github.com/openclaw/openclaw/pull/80779 is maintainable for stale IDs, but a successful-approval fix needs the reporter's exact current pending request and command before changing the request-id contract.
