## Summary

Keep this open. Current main still lacks an operator-selectable readiness-aware queue mode; the public queue contract, schema, command surface, drain path, and docs remain based on the existing steer/followup/collect/interrupt modes and debounce/quiet-window flushing, while subagent and exec completion paths are only internal prior art.

## What This Changes

Request a new readiness-aware queue collection mode that buffers and coalesces status or completion notices until the session/run is ready, with a documented starvation fallback.

## Best Possible Solution

Define a documented core queue contract that waits on generic session/run readiness, coalesces only eligible status/completion notices, preserves existing debounce modes, and includes a bounded fallback.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a feature request; source and docs inspection give high-confidence verification that current main does not expose the requested public readiness-aware queue mode.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers define the readiness signal, eligible notice classes, coalescing rules, and starvation cap; the likely home is the shared queue contract rather than a Signal-specific workaround.
