## Summary

Keep this open. Current main still exposes only the existing timer/debounce-based queue modes, while the one readiness-style deferral path is internal subagent announce prior art rather than a public `messages.queue` mode for ordinary queued chat/status/completion notices.

## What This Changes

The issue requests a readiness-aware queue collection mode that buffers ordinary queued status/completion messages until the session is ready, then flushes or coalesces them safely.

## Best Possible Solution

Add a documented public queue option in the core auto-reply queue surface, with config types/schema/help, `/queue` parsing and command metadata, queue docs, focused tests, a generic session/run readiness signal, and a bounded fallback while preserving existing collect/debounce behavior.

## Reproduction Assessment

Not applicable as a feature request. Source-level verification is high confidence: current main still has only the existing public queue modes, and generic followup draining remains debounce/quiet-window based.

## Solution Assessment

Unclear until maintainers define the readiness contract. The requested surface belongs in core queue handling, but the exact mode name, readiness signal, participating async notices, and starvation cap need owner judgment before a narrow implementation is safe.
