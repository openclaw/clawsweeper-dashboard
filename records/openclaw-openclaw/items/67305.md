## Summary

Keep this issue open. Current main still exposes the existing timer/debounce-based queue modes and docs; the only readiness-style deferral I found is internal subagent announce delivery prior art, not a public `messages.queue` mode for ordinary queued chat/status/completion notices.

## What This Changes

Request to add an operator-selectable readiness-aware queue collection mode that buffers queued status/completion messages until the session is ready, then flushes or coalesces them with bounded starvation protection.

## Best Possible Solution

Add a documented public queue option in the core auto-reply queue surface, backed by config types/schema/help, /queue parsing and command metadata, queue docs, and focused tests. The drain condition should use a generic session/run readiness signal and define a bounded fallback while preserving current collect/debounce behavior for existing operators.

## Reproduction Assessment

Not applicable as a feature request. Source-level verification is high confidence: current main still has only the existing public queue modes and generic followup draining remains debounce/quiet-window based.

## Solution Assessment

Unclear until maintainers define the readiness contract. The requested surface is the right core queue area, but the exact mode name, readiness signal, participating async notices, and starvation cap need product/owner judgment before a narrow implementation is safe.
