## Summary

Keep this open because current main still exposes timer/debounce-based public queue modes, while the only readiness-style deferral found is internal subagent announce prior art rather than an operator-selectable queue mode for ordinary queued chat/status/completion notices.

## What This Changes

Add a readiness-aware queue collection mode that holds and coalesces queued notices until the target session is ready, with bounded starvation protection and documentation relative to existing collect/debounce behavior.

## Best Possible Solution

Add a documented public queue option in the core auto-reply queue surface that waits on a generic session/run readiness signal, coalesces only the intended notice classes, preserves existing collect/debounce behavior, and defines a clear starvation cap.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high-confidence verification that current main still lacks a public readiness-aware queue mode and generic followup draining remains debounce-based.

## Solution Assessment

Unclear until maintainers define the readiness contract. The requested surface belongs in core queue handling, but the mode name, readiness signal, participating notices, and fallback cap need product/owner judgment before implementation.
