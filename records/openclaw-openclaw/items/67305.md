## Summary

Keep this open. Current main still exposes only timer/debounce-based queue collection, and the requested readiness-aware mode would add a public queue/config contract that needs maintainer design before implementation.

## What This Changes

Request adds an operator-selectable readiness-aware queue mode that holds and coalesces status or completion notices until the session is ready to reply.

## Best Possible Solution

Add a documented messages.queue mode or option backed by a generic session/run readiness signal, with bounded starvation protection and compatibility with existing debounce modes.

## Reproduction Assessment

Not applicable: this is a feature/config request, not a report of broken existing behavior. Source and docs inspection show current main still only offers steer, followup, collect, and interrupt, with collect/followup draining after a debounce quiet window.

## Solution Assessment

Unclear until maintainers define readiness, notice eligibility, coalescing, and starvation semantics. If accepted, the maintainable direction is a core queue contract rather than a Signal-specific workaround.
