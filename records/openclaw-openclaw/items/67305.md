## Summary

Keep this open. Current main still only exposes timer/debounce-based public queue modes, while the readiness-style deferral found on main is internal subagent announce prior art rather than an operator-selectable queue mode for ordinary queued chat/status/completion notices.

## What This Changes

Requests a public readiness-aware queue collection mode or option that holds and coalesces queued notices until a session is actually ready to reply.

## Best Possible Solution

Define a public core queue contract that waits on generic session/run readiness, coalesces only intended notices, preserves existing collect/debounce behavior, and documents a bounded fallback.

## Reproduction Assessment

Not applicable as a feature request. Source, docs, and test inspection give high-confidence verification that current main lacks an operator-selectable readiness-aware queue mode and generic followup draining remains debounce-based.

## Solution Assessment

Unclear until maintainers define the contract. The request belongs near core queue handling, but the mode name, readiness signal, notice classes, and starvation cap need product/owner judgment before implementation.
