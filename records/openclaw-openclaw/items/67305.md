## Summary

Keep open. Current main still only exposes timer/debounce-based queue behavior, while the requested readiness-aware flush mode would add a new public queue/config contract that needs maintainer design.

## What This Changes

The issue requests an operator-selectable readiness-aware queue collection mode that buffers status/completion notices until a session is ready and then flushes or coalesces them safely.

## Best Possible Solution

Add a documented messages.queue mode or option backed by a generic session/run readiness signal, with bounded starvation protection and compatibility with existing debounce-based modes.

## Reproduction Assessment

Not applicable: this is a feature/config request, not a report of broken existing behavior. Source and docs inspection give a high-confidence implementation-gap check: current modes are steer, followup, collect, and interrupt, with collect/followup draining after debounce.

## Solution Assessment

Unclear until maintainers define readiness, notice eligibility, coalescing, and starvation semantics. If accepted, the maintainable direction is a core queue contract rather than a Signal-specific workaround.
