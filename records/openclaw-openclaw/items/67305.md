## Summary

Keep open. Current main still exposes the existing timer/debounce-based public queue modes, with no operator-selectable readiness-aware mode or public starvation-cap option. The internal subagent announce deferral is useful prior art, but it is not the requested generic `messages.queue` behavior for ordinary chat/status completion delivery.

## What This Changes

The issue requests an operator-selectable readiness-aware queue collection mode that holds and coalesces queued status/completion messages until the session is ready to reply.

## Best Possible Solution

Add a documented public queue option in the core auto-reply queue surface, backed by config types/schema/help, /queue parsing and command metadata, queue docs, and focused tests. The drain condition should use a generic session/run readiness signal and define a bounded fallback while preserving the current collect/debounce behavior for existing operators.

## Reproduction Assessment

Not applicable as a feature request, but the source-level verification is high confidence: current main has no public readiness-aware queue mode and still drains generic followups through debounce/quiet-time logic.

## Solution Assessment

Unclear until maintainers define the readiness contract. The requested public queue surface is the right area, but the exact mode name, participating async notices, readiness signal, and starvation cap need maintainer judgment before a narrow implementation is safe.
