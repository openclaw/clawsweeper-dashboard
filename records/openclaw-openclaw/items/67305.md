## Summary

Keep this open. Current main still lacks an operator-selectable readiness-aware queue mode; the public queue/config/docs surface remains `steer`, `followup`, `collect`, and `interrupt`, and generic followup/collect draining is still debounce-based rather than tied to session readiness.

## What This Changes

Add an operator-selectable readiness-aware queue collection mode that holds or coalesces status/completion messages until a session is ready, with documented starvation behavior.

## Best Possible Solution

Define a documented core queue contract that waits on a generic session/run readiness signal, coalesces only eligible status/completion notices, preserves existing collect/debounce behavior, and includes a bounded fallback.

## Reproduction Assessment

Not applicable: this is a feature request rather than a broken existing behavior. Source and docs inspection give high confidence that the requested public mode is absent from current main.

## Solution Assessment

Unclear until maintainers define the readiness signal, notice eligibility, coalescing rules, and starvation cap. The likely maintainable direction is a core queue contract, not a Signal-specific workaround.
