## Summary

Current main does not match the reported loader-wide failure: runtime plugin activation binds `api.registerCommand()` through the registry path, and existing regression coverage asserts activating loads publish plugin commands. The unbound `buildPluginApi()` call in `loader.ts` is the descriptor-only CLI metadata path, not the live command runtime.

## What This Changes

The issue reports that plugin command registration via `api.registerCommand()` is a no-op because `loader.ts` does not pass a `registerCommand` handler into `buildPluginApi()`.

## Best Possible Solution

Keep the current split: live runtime loads register plugin commands through `registry.ts`, while `cli-metadata` remains limited to CLI descriptor capture; investigate any external media plugin failure with plugin-specific logs and config.

## Reproduction Assessment

No high-confidence failing reproduction remains for the reported core path. Current source and regression tests show activating plugin loads bind and publish `api.registerCommand()`, while the cited missing handler is confined to descriptor-only CLI metadata loading.

## Solution Assessment

Yes, closing as not reproducible is the best cleanup outcome for the core report. Adding `registerCommand` to the CLI metadata loader would blur a documented non-runtime path, while the runtime path already has the required binding.
