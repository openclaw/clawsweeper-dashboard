## Summary

Keep open: current main and the latest release still show the recoverability gap, and https://github.com/openclaw/openclaw/pull/96651 is an open candidate fix rather than a merged implementation.

## What This Changes

The issue reports that Gateway `memory_search` can latch a fallback embedding provider after a remote provider outage and does not recover with soft reload once the primary provider is healthy again.

## Best Possible Solution

Review and land a corrected version of https://github.com/openclaw/openclaw/pull/96651, or an equivalent narrow recovery/invalidation fix, while preserving the provider/model identity guard.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main can enter fallback-active state from createEmbeddingProvider fallback results, then later searches keep the initialized provider state instead of retrying the primary provider. I did not run the live Ollama outage sequence in this read-only review.

## Solution Assessment

Yes in direction, but the linked candidate PR still needs its review blocker resolved. The best fix is a narrow recovery or cache-invalidation path for every fallback-active entry while keeping the existing index-identity guard intact.
