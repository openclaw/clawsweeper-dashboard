## Summary

Keep open: current main still has the fallback-active latch path, and the two closing PRs for this issue remain open and are not ancestors of main.

## What This Changes

The issue reports that Gateway `memory_search` can stay latched to a fallback embedding provider after a primary-provider outage, requiring a full process restart to recover.

## Best Possible Solution

Land one maintainer-chosen recovery implementation that re-resolves the configured primary provider without weakening the index identity guard, then close this issue from the merged fix.

## Reproduction Assessment

Yes at source level: current main can store `fallbackFrom` on the cached Gateway memory manager and skip primary provider re-resolution once `providerInitialized` is true. I did not run the reporter's live remote-provider outage sequence.

## Solution Assessment

Yes, the best fix is a narrow recovery or manager invalidation path that restores the configured provider while preserving the existing model-identity guard. Adding a new config knob or weakening vector identity checks would be a worse direction.
