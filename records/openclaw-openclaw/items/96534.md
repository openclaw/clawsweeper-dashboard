## Label Justifications

- `P2`: The issue affects memory search recovery after a provider outage for a bounded memorySearch fallback configuration.
- `impact:session-state`: The long-lived memory manager can retain stale fallback state until process restart, causing memory recall behavior to drift from the configured provider/index state.
- `impact:auth-provider`: The failure is tied to configured embedding provider fallback and provider recovery behavior.

## Summary

Keep open: current main still has a cached default memory manager that can be mutated into fallback-active embedding provider state, and the inspected hot-reload path does not clear or re-resolve that manager, so the reported no-recovery behavior remains distinct from the already-closed identity-guard issues.

## What This Changes

The issue reports that in-Gateway memory_search remains latched to a fallback embedding provider after a remote provider outage and does not recover after the primary provider returns or after soft config reload.

## Best Possible Solution

Make fallback activation recoverable by re-resolving the configured provider or invalidating the cached default memory manager on the relevant recovery/reload path, while preserving the model-identity guard and avoiding new config knobs.

## Reproduction Assessment

Yes at source level, but not by live execution in this read-only review. Current main caches the default memory manager, fallback activation mutates that manager into fallback-active state, and the inspected soft-reload path does not clear or re-resolve it.

## Solution Assessment

Yes, a narrow recovery/invalidation change in memory-core or the Gateway reload path appears to be the best fix. It should not add product surface or weaken the existing model/provider identity guard.
