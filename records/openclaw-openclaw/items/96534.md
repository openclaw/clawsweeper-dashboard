## Summary

Keep open: current main still has a source-reproducible recovery gap where a long-lived Gateway memory manager can stay on fallback-active embedding provider state after the primary provider recovers.

## What This Changes

The issue reports that Gateway `memory_search` can latch a fallback embedding model after a provider outage and only recover after a full process restart.

## Best Possible Solution

Make fallback-active embedding state recoverable in memory-core by re-resolving the configured primary provider on later searches or evicting the cached manager on relevant reload, while keeping mismatched vector search degraded to FTS-only.

## Reproduction Assessment

Yes, at source level: current main can store fallback-active provider state, skip primary reinitialization once initialized, and refuse another fallback transition once `fallbackFrom` is set. A live repro would use the reporter's Ollama outage sequence.

## Solution Assessment

Yes, the best fix is a narrow memory-core lifecycle/cache repair. Adding a new config option or weakening the index identity check would be the wrong layer.

## Repair Work Prompt

Fix the Gateway memory_search fallback latch after embedding provider outage. In current main, MemoryIndexManager.ensureProviderInitialized() treats a fallback EmbeddingProviderResult as initialized, activateFallbackProvider() refuses further action once fallbackFrom is set, and Gateway config reload does not close memory search managers for agents.defaults.memorySearch changes. Keep the fix inside memory-core provider lifecycle/cache invalidation; do not add a new config option and do not weaken the provider/model identity guard. Expected behavior: after the configured primary embedding provider is reachable again, memory_search re-resolves it or otherwise releases incompatible fallback state so vector search can resume without a full process restart; if the fallback model identity cannot match the existing index, degrade to FTS-only rather than using incompatible vectors. Add focused regression coverage around fallback-active recovery and reload/manager cache invalidation as appropriate.
