## Label Justifications

- `P2`: The issue affects memory search recovery after a provider outage for a bounded memorySearch fallback configuration.
- `impact:session-state`: The long-lived memory manager can keep stale provider state until process restart, causing memory recall state to drift from the configured/indexed provider.
- `impact:auth-provider`: The failure is tied to embedding provider fallback and recovery behavior for configured memory providers.

## Summary

Keep open: current main still mutates the cached memory manager into a fallback-active provider state and has no soft-reload or next-call path that clears it back to the configured provider, so the distinct recovery bug remains.

## What This Changes

The issue reports that Gateway `memory_search` stays pinned to a fallback embedding provider after a remote provider outage, and soft reload does not recover it.

## Best Possible Solution

Make fallback activation recoverable by re-resolving the configured provider or invalidating the cached memory manager on the relevant reload/recovery path, while preserving the existing identity guard and avoiding new config knobs.

## Reproduction Assessment

Yes at source level, but not by live execution in this read-only review. Current main activates and stores fallback provider state on the cached manager, blocks mismatched vector search, and lacks a soft-reload or retry path that releases the fallback identity.

## Solution Assessment

Yes, the best fix is a narrow recovery/invalidation change in the memory manager or Gateway reload path. It should not add a new config option or weaken the model/provider identity guard that prevents incompatible vector spaces.
