## Label Justifications

- `P2`: This is a bounded `memory_search` recovery bug after provider fallback outages, with clear source evidence and limited blast radius.
- `impact:session-state`: A long-lived memory manager can retain stale fallback provider state until process restart, causing memory recall state to drift from configured/indexed state.
- `impact:auth-provider`: The failure is tied to embedding provider fallback, primary-provider recovery, and model/provider identity resolution.

## Summary

Keep open: current main still has a source-reproducible recovery gap where the long-lived Gateway memory manager can remain in fallback-active embedding-provider state after the primary provider recovers, and no merged or open closing PR owns this exact latch/reload fix.

## What This Changes

Recover `memory_search` from a fallback embedding provider after a remote provider outage or soft reload without requiring a full Gateway restart.

## Best Possible Solution

Make fallback activation recoverable by re-resolving the configured provider on later calls or evicting/resetting affected memory managers on the relevant reload path, while keeping mismatched index identity fail-closed and avoiding new config surface.

## Reproduction Assessment

Yes, at source level: the cached default manager can store fallback-active provider state and later searches skip primary provider reinitialization. A live repro would use the reporter's Ollama outage sequence, but the current code path is clear enough to keep this open.

## Solution Assessment

Yes, a narrow recovery/invalidation fix is the best direction. The fix should re-resolve or evict fallback-stuck memory managers and keep the existing provider/model identity guard intact rather than weakening vector safety or adding a new operator knob.
