## Summary

Keep open: current main and the latest release still allow a long-lived builtin memory manager to remain in fallback-active provider state after a transient embedding-provider failure, and later searches skip primary provider re-resolution.

## What This Changes

The issue reports that `memory_search` can latch a fallback embedding provider after a remote provider outage and only recover after a full restart.

## Best Possible Solution

Make fallback activation recoverable by resetting or re-resolving the configured primary provider after fallback-active state, and invalidate the relevant cached memory manager on memorySearch reload without adding new config or weakening index identity safety.

## Reproduction Assessment

Yes at source level: current main can store fallback-active provider state on the cached manager, then later searches skip primary provider reinitialization. A live repro would still need the reported remote Ollama outage sequence.

## Solution Assessment

Yes, the best fix is a narrow memory manager recovery or cache invalidation change. The fix should keep the existing provider/model identity guard intact rather than implementing broad multi-index failover or adding another config knob.
