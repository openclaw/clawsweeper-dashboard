## Summary

Keep open: this PR targets a valid Windows QMD command-parsing bug, but it is not merge-ready because it repairs after `splitShellArgs` has already stripped separators and it also converts bare PATH commands like `qmd` into workspace-relative paths.

## What This Changes

The PR adds drive-relative path repair to QMD backend path resolution, routes `memory.qmd.command` through that resolver, and adds a backend-config regression test.

## Best Possible Solution

Land one canonical backend-config fix that preserves Windows absolute QMD command paths before POSIX-style parsing, keeps bare PATH commands unchanged, and includes redacted Windows QMD runtime proof.

## Reproduction Assessment

Yes, source-level. Current main sends `memory.qmd.command` through `splitShellArgs`, whose unquoted backslash branch consumes Windows separators before QMD spawn resolution; I did not run a live Windows QMD setup.

## Solution Assessment

No. Repairing drive-relative output after shell parsing is too late, and resolving every command token through `resolvePath` breaks PATH lookup; the safer fix preserves Windows absolute commands before splitting while leaving bare executables alone.
