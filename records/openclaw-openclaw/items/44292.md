## Summary

Keep open. Current main still uses a manual `FIELD_LABELS` table and per-key missing-label failures, while the directly related implementation PR https://github.com/openclaw/openclaw/pull/46304 is closed unmerged, so the requested developer workflow is still missing.

## What This Changes

Add a developer workflow that makes missing config field labels obvious and quick to fix when new config help keys are introduced.

## Best Possible Solution

Improve the existing parity test to aggregate missing root/help labels and print deterministic editable stubs for `src/config/schema.labels.ts`; defer a write-mode generator unless maintainers want broader tooling.

## Reproduction Assessment

Yes, by source inspection. Adding a `FIELD_HELP` key without a matching `FIELD_LABELS` entry still reaches the per-key `requireLabel()` throw path with no aggregated missing-key list or copy-paste-ready stub output.

## Solution Assessment

Yes. A narrow test-output improvement is the best first fix because it satisfies the acceptance criteria with a small test-only change; a generator or schema-driven label source would be broader than needed.
