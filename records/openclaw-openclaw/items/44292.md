## Summary

Keep open. Current main still has the same manual `FIELD_LABELS` workflow: the parity test reports one missing key at a time, the label table is hand-maintained, and the directly related implementation PR #46304 is closed unmerged.

## What This Changes

Add a developer workflow that makes missing config `FIELD_LABELS` entries obvious and quick to add when new config help keys are introduced.

## Best Possible Solution

Improve the existing parity test to aggregate missing root/help labels and print deterministic editable stubs for `src/config/schema.labels.ts`; defer a write-mode generator unless maintainers want a broader workflow.

## Reproduction Assessment

Yes. Source inspection on current main shows adding a `FIELD_HELP` key without a matching `FIELD_LABELS` entry still reaches a per-key `toBeDefined()` failure such as `missing label for help key X`, with no aggregated missing-key list or stub output.

## Solution Assessment

Yes. A narrow test-output improvement is the best first fix because it satisfies the acceptance criteria with a small test-only change; a write-mode generator or schema-driven label source would be more invasive.
