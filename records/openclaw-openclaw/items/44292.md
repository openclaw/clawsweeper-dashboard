## Summary

Keep open. Current main still uses a manual `FIELD_LABELS` table and the parity test still throws on the first missing label without aggregated diagnostics or copy-paste-ready stubs; the directly related implementation PR is closed unmerged, so the requested developer workflow is still missing.

## What This Changes

The issue requests an autofix, generator, or clearer parity-test workflow for config help keys that are missing corresponding `FIELD_LABELS` entries.

## Best Possible Solution

Improve the existing parity test to aggregate all missing root/help labels and print deterministic editable stubs for `src/config/schema.labels.ts`; leave a write-mode generator or schema-source refactor for a separate maintainer decision.

## Reproduction Assessment

Yes, by source inspection. Adding a new `FIELD_HELP` key without a matching `FIELD_LABELS` entry still reaches the per-key `requireLabel()` throw path with no aggregate missing-key list or copy-paste-ready stub output.

## Solution Assessment

Yes. A narrow test-output improvement is the best first fix because it satisfies the acceptance criteria while preserving the current label/help metadata contract; a generator would be broader than needed.
