## Summary

Keep open. Current main partially fixes the mixed tool/text projection path, but WebChat still reloads from a bounded recent-history window without a visible has-more/load-older contract, and newer comments point to compaction and restart-recovery variants that remain unresolved.

## What This Changes

The issue asks WebChat/Control UI to render assistant replies from the transcript reliably so it matches TUI behavior for the same session.

## Best Possible Solution

Make WebChat history truncation explicit and navigable, preferably by reusing the paginated session-history surface or extending chat.history with hasMore/cursor metadata plus a load-older UI, while keeping compaction and restart-recovery failures tracked separately.

## Reproduction Assessment

Yes, at source level: create a session with more than 100 displayable transcript messages and reload WebChat; Control UI asks chat.history for only 100 messages and replaces its local chat state from that bounded response. I did not run the exact Safari/long-transcript setup live in this read-only review.

## Solution Assessment

No, closing as fixed is not the best path. Current main has a useful projection fix, but the remaining maintainable solution is a truncation-aware WebChat history contract rather than another fixed-limit bump.
