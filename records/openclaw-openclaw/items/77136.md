## Summary

Keep open. Current main still has bounded WebChat history and render windows that can make long transcript history look like assistant replies disappeared, while the reporter's exact Safari/transcript case is not available to close as fixed or non-reproducible.

## What This Changes

The issue asks WebChat to render assistant messages from long transcript history consistently instead of leaving some replies missing from the browser view.

## Best Possible Solution

Define WebChat's long-history contract and implement it visibly, preferably with pagination/load-older behavior or server-provided truncation metadata so bounded history never presents as silent message loss.

## Reproduction Assessment

Partially: source inspection gives a high-confidence long-history path where WebChat reloads only the recent 200-message window and can hide older assistant rows while the transcript remains intact. The exact Safari transcript was not provided, so the narrower reported failure is not fully reproduced.

## Solution Assessment

Unclear. Raising the fixed limits alone is not the best maintainable fix; the safer direction is explicit truncation/has-more state or a load-older flow while preserving payload caps.
