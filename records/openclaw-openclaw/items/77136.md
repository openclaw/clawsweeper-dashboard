## Summary

Keep open: current main and the latest shipped tag still have a source-proven WebChat long-history path where the Control UI requests a bounded 100-message WebSocket history window, the RPC response has no truncation/page metadata, and the renderer can only warn about history hidden inside the payload it already received. The discussion also has active related evidence and an open pagination PR, so this is not safe cleanup closure.

## What This Changes

The issue reports WebChat failing to render some assistant messages from long sessions while the transcript and TUI view remain intact.

## Best Possible Solution

Define an explicit WebSocket chat.history long-history contract for Control UI, then keep compaction/restart-recovery variants on their narrower canonical issues.

## Reproduction Assessment

Yes, at source level: Control UI still requests only a 100-message WebSocket chat.history window, the RPC response has no hasMore/cursor metadata, and the renderer only warns when more than 100 messages are already in the client payload. I did not live-reproduce the reporter's Safari/DeepSeek transcript because the raw transcript was not provided.

## Solution Assessment

Unclear. A renderer-only notice would reduce confusion, but the maintainable fix likely needs a maintainer choice between additive WebSocket truncation metadata, cursor pagination, or a broader load-older UX.
