## Summary

Keep open. Current main still has fixed 200-message WebChat history and render windows, and the available evidence does not show a resolved long-transcript rendering path for the reported behavior.

## What This Changes

The issue reports that WebChat hides some assistant responses while the transcript and TUI still show the same session data.

## Best Possible Solution

Define WebChat long-history behavior and implement it visibly, either with load-older pagination or server-side truncation/count metadata so the browser never looks like it silently lost transcript-backed replies.

## Reproduction Assessment

Yes for the bounded-history path from source: Control UI requests 200 messages, Gateway returns a recent projected window, and the renderer only warns when the local array exceeds 200. No for the reporter's exact transcript shape because the transcript was not attached.

## Solution Assessment

Unclear. Raising the limit alone is not the best durable fix unless maintainers accept the payload tradeoff; pagination or explicit truncation metadata would be more maintainable.
