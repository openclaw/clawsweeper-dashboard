## Summary

Keep open. Current main still has a source-clear WebChat long-history path where the browser requests a bounded 100-message window and the renderer cannot tell that older server-side history was omitted, while the exact Safari transcript case remains unverified.

## What This Changes

The issue reports WebChat hiding or failing to render some assistant messages in long sessions even though the transcript and TUI still contain them.

## Best Possible Solution

Expose WebChat long-history state end to end, preferably with `hasMore`/cursor or load-older behavior aligned with the existing session-history pagination seam, so bounded history is visible and recoverable instead of looking lost.

## Reproduction Assessment

Yes, at source level: a session with more than 100 displayable messages will be returned to WebChat as an exact 100-message array, and the renderer's hidden-count notice cannot fire because it only sees that already-truncated array. I did not reproduce the exact Safari transcript without the reporter's data.

## Solution Assessment

Unclear. The renderer-only flag sketch in the discussion is useful, but the best maintainable fix needs an end-to-end truncation signal or load-older path; simply raising the fixed limits would not solve the underlying UX contract.
