## Label Justifications

- `P2`: This is a source-reproducible WebChat reliability bug with real user impact, but it is bounded to history rendering rather than core data loss.
- `impact:message-loss`: The user-visible symptom is assistant replies disappearing from WebChat even though the transcript still contains the data.
- `impact:session-state`: The failure is tied to reconstructing and bounding stored session history for the browser UI.

## Summary

Keep open: current main still has a source-proven WebChat long-history path where the client requests only 100 history rows, the renderer only detects hidden history inside rows it already received, and `chat.history` does not return a pagination or truncation signal for older omitted rows.

## What This Changes

The issue reports that WebChat/Control UI can make assistant replies appear to disappear in long sessions even though transcript data remains intact and TUI can still show the conversation.

## Best Possible Solution

Add an explicit bounded-history contract for WebChat, preferably a `chat.history` pagination or `hasMore` signal wired through Control UI to show and load older messages while preserving the current payload limits.

## Reproduction Assessment

Yes for source reproduction: current Control UI requests `chat.history` with limit 100, the renderer only warns when its received array exceeds 100, and the Gateway response has no older-history signal. I did not run a live browser reproduction in this read-only review.

## Solution Assessment

No current implementation on main fully solves it. The narrow renderer notice sketch is useful, but the best maintainable fix needs a maintainer choice between a minimal truncation signal and a broader pagination/load-older contract.
