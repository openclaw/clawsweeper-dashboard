## Summary

Keep open. Current main still has a WebChat reload path that replaces browser state with a fixed recent-history window, so long transcripts can still look like assistant replies disappeared even though the underlying transcript is intact.

## What This Changes

The issue requests WebChat to render assistant transcript history consistently with TUI for long sessions where transcript data remains intact.

## Best Possible Solution

Define WebChat's long-history contract and implement load-older or visible truncation state, preferably by reusing the existing paginated session-history metadata or adding equivalent chat.history metadata.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Partially: source inspection gives a clear current-main path for >200-message transcripts to be truncated on WebChat reload, but the exact Safari/DeepSeek transcript case is not attached.

## Solution Assessment

Is this the best way to solve the issue? Unclear: raising the fixed limit is not the best maintainable fix; maintainers should choose between WebChat pagination/load-older UX and explicit truncation metadata before implementation.
