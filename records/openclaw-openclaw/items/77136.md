## Summary

Keep open. Current main still has equal 100-message WebChat request/render windows and no `chat.history` truncation signal, so older assistant replies in long sessions can fall outside the browser view without the hidden-history notice; recent comments also point to separate compaction and restart-recovery variants that should not be collapsed into this issue.

## What This Changes

The issue asks WebChat to stop making long-session assistant replies appear lost when transcript data remains intact and the TUI can still show the conversation.

## Best Possible Solution

Define the WebChat long-history contract and wire an explicit truncation signal into the UI, with pagination or load-older behavior as the durable direction and a clear notice when only the recent bounded window is shown.

## Reproduction Assessment

Yes, at source level: with more than 100 transcript messages, WebChat asks for exactly 100 and the renderer only shows a hidden-history notice when the client already has more than 100. No live Safari transcript was available for the reporter's exact case.

## Solution Assessment

Unclear. The renderer-only sketch is useful but incomplete until maintainers choose how the controller/server communicates truncation; simply raising the fixed limit would not be the most maintainable fix.
