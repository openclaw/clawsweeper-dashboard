## Summary

Keep open. Current main still has a source-proven WebChat long-history path where the browser requests a 100-message tail, the gateway returns only that bounded window without `hasMore`/truncation metadata, and the renderer can only show the hidden-history notice when it receives more than the render limit; no merged fix or canonical superseding item closes this exact UI contract gap.

## What This Changes

The issue asks WebChat to render long-session assistant history consistently with the durable transcript/TUI instead of silently losing visible replies.

## Best Possible Solution

Define and implement a WebChat long-history contract so bounded `chat.history` responses surface older omitted messages, ideally through an additive `hasMore`/pagination signal or an equivalent load-older affordance wired into the Control UI notice.

## Reproduction Assessment

Yes, at source level: current main requests 100 WebChat history messages, returns a bounded recent tail without a truncation signal, and only renders a hidden-history notice when more than 100 messages reach the renderer. The exact Safari/DeepSeek transcript case was not reproduced because the raw transcript was not provided.

## Solution Assessment

Unclear. A renderer-only `serverHistoryTruncated` notice is a useful narrow fix sketch, but maintainers still need to choose between limit-plus-one probing, additive `hasMore` metadata, or a fuller pagination/load-older contract.
