## Summary

Keep open. Current main still has bounded WebChat history paths that can make older transcript messages absent from the browser view without a visible server-side truncation contract, and the exact Safari/DeepSeek transcript was not provided.

## What This Changes

The issue asks WebChat to render all assistant transcript messages consistently with TUI, especially after long sessions or reloads.

## Best Possible Solution

Define WebChat's long-history contract and implement visible pagination, load-older behavior, or server-side truncation metadata so bounded payloads do not look like lost assistant replies.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the bounded-history hypothesis from source: reloads request and render a 200-message window. No for the reporter's exact Safari/DeepSeek transcript because the transcript was not provided.

## Solution Assessment

Is this the best way to solve the issue? Unclear. A limit bump alone is not the narrowest maintainable fix; the safer direction is pagination or explicit truncation/count metadata, unless the missing transcript proves a separate projection bug inside the current window.
