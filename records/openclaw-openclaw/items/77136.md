## Summary

Keep open: current main still has a bounded WebChat history path that can omit older transcript rows without a client-visible older-page affordance, and the likely fix is the open maintainer-labeled pagination PR rather than merged code. The issue discussion correctly separates this from compaction and restart-recovery disappearances that are adjacent or already handled elsewhere.

## What This Changes

The issue asks WebChat to render long assistant history reliably from intact transcript data, matching the TUI instead of making older assistant replies appear lost.

## Best Possible Solution

Land or revise the maintainer-owned WebChat history pagination path so `chat.history` exposes additive older-page metadata and WebChat can load older visible transcript rows, while keeping compaction/restart-recovery issues separate.

## Reproduction Assessment

Yes for the bounded-history portion: source inspection shows WebChat requests only 100 rows and current `chat.history` gives no older-page metadata, so a transcript with more than 100 displayable rows cannot be fully reconstructed in WebChat. I did not run the reporter's Safari/Homebrew transcript live.

## Solution Assessment

No, current main is not the best final state because it only windows the returned rows and logs some truncation cases. The best current path is the additive Gateway/UI pagination work in https://github.com/openclaw/openclaw/pull/87111 after maintainer review and conflict resolution.
