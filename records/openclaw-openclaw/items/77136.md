## Summary

Keep open. Current main still has a source-reproducible WebChat bounded-history UX failure: the client requests 100 messages, the renderer only shows its hidden-history notice when it receives more than 100, and `chat.history` does not return a truncation or `hasMore` signal.

## What This Changes

The issue asks WebChat to stop making long-session assistant replies look lost when transcript data remains intact and TUI/session readers can still see the conversation.

## Best Possible Solution

Choose the long-history contract, then wire WebChat to an explicit older-history signal or load-older flow so bounded history is visible and cannot be mistaken for lost assistant replies.

## Reproduction Assessment

Yes for the bounded-history UX path: current source shows a 100-message client request feeding a 100-message renderer window, so a server-truncated long transcript cannot trigger the existing hidden-history notice. No for the exact Safari/DeepSeek transcript case because the raw transcript was not provided.

## Solution Assessment

Unclear. The renderer-only prop sketch is a good narrow start, but maintainers still need to choose whether the truncation fact comes from an additive `chat.history` response field, reuse of the session-history pagination contract, or a client-side limit-plus-one probe.
