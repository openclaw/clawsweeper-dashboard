## Summary

Keep open. Current main still has bounded WebChat history and render windows that can make older assistant replies disappear from the browser view, while the reporter’s exact Safari/transcript case is not available enough to close as fixed or non-reproducible.

## What This Changes

The issue requests that WebChat render assistant transcript history reliably instead of making some assistant replies disappear during long sessions or reloads.

## Best Possible Solution

Define WebChat’s long-history contract and implement it visibly, preferably with pagination or server-side truncation metadata so bounded history never looks like lost assistant replies.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Partially: source inspection shows a bounded-history path where WebChat reloads only the recent 200 messages, but the exact Safari transcript case has not been provided.

## Solution Assessment

Is this the best way to solve the issue? Unclear: simply raising the fixed limit is not the best maintainable fix; the better direction is explicit truncation/has-more state or a load-older flow while preserving payload caps.
