## Summary

Keep open. Current main still has a high-confidence bounded-history path where WebChat reloads only a 200-message tail, so this needs a maintainer decision on long-history UX/API rather than a cleanup close.

## What This Changes

The issue asks WebChat to keep assistant replies visible from intact long transcripts so browser history matches TUI behavior for long conversations.

## Best Possible Solution

Define WebChat's long-history contract and implement visible pagination/load-older behavior or truncation metadata, preferably reusing the existing session-history pagination primitives.

## Reproduction Assessment

Yes for the bounded-history path from source: a session with more than 200 projected messages reloads through a 200-message WebChat request and the gateway returns only the tail. The exact Safari/DeepSeek transcript case was not reproduced because the transcript was not provided.

## Solution Assessment

Unclear. Raising the fixed limit alone is not the best solution; the maintainable path is to expose `hasMore`/cursor state or a load-older flow while preserving payload caps.
