## Summary

Keep open. Current main still has bounded WebChat history paths that can make older assistant messages vanish from the browser view, and the report is too recent and concrete to close without deciding the long-history UX/API behavior.

## What This Changes

The issue requests that WebChat render assistant messages from long transcripts consistently with the intact session transcript and the TUI view.

## Best Possible Solution

Define WebChat's long-history contract and implement it visibly: paginate or load older transcript rows when feasible, or return server-side truncation metadata so the UI never looks like it lost assistant replies while preserving payload caps.

## Reproduction Assessment

Yes for the bounded-history hypothesis: source inspection shows WebChat reloads through a 200-message recent-history window, with a separate 200-message render window. No for the reporter's exact transcript shape because the transcript was not provided.

## Solution Assessment

Unclear. Raising a limit alone is not the best fix unless maintainers accept the payload tradeoff; a durable solution should expose truncation/counts or add a load-older path.
