## Summary

Keep open: current main still has a source-reproducible WebChat long-history path where older assistant replies can fall outside the bounded browser view without any truncation notice, and the recent discussion adds a useful fix sketch rather than evidence that the issue is resolved.

## What This Changes

The issue asks WebChat/Control UI to stop making assistant replies appear lost in long sessions and to make rendered history line up with the durable transcript/TUI behavior.

## Best Possible Solution

Define WebChat's bounded-history UX explicitly: show a reliable older-history notice when the server returns a truncated window, and preferably add a load-older or `hasMore` path without increasing unbounded browser payloads.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the current-main bounded-history/no-notice path: a transcript with more than 100 displayable messages can return exactly 100 messages to WebChat, while the renderer only shows the hidden-count notice when it receives more than 100. The exact reporter transcript/Safari case was not live-reproduced.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers choose the UX/API contract; a renderer-only prop is a useful piece, but the controller or gateway still has to supply reliable truncation state. A fixed limit increase alone would not be the best solution.
