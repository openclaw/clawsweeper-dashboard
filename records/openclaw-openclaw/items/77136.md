## Summary

Keep open: current main still has a source-proven WebChat long-history gap where Control UI requests and renders a bounded 100-message window without a `chat.history` truncation signal, so older assistant replies can look missing even though the transcript is intact. The discussion and the open pagination work are useful, but this is not implemented on a clean landing path yet.

## What This Changes

The issue reports that WebChat silently omits some assistant messages in long sessions while TUI and transcript data still show the full conversation.

## Best Possible Solution

Land a clean additive `chat.history` pagination or truncation contract for WebChat, with a visible load-older or older-history notice and regression coverage for server-truncated 100-message sessions.

## Reproduction Assessment

Yes from source inspection, though I did not run a live browser reproduction. A session with more than 100 visible messages can be truncated by the RPC request before the renderer sees it, and the renderer only warns about rows already present in its local array.

## Solution Assessment

Unclear as a final product path. The narrow renderer-only notice is plausible, but a clean additive pagination/load-older contract is the more complete fix and needs maintainer selection because there are overlapping open PRs.
