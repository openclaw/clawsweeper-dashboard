## Summary

Keep open. Current main still has a source-proven WebChat long-history path where assistant messages can fall outside the browser’s bounded history window without any truncation signal, and the later issue discussion adds a plausible narrow renderer fix sketch but not a maintainer-selected direction or merged fix.

## What This Changes

The issue requests that WebChat stop making intact transcript assistant messages look silently lost in long conversations, matching TUI visibility or clearly indicating bounded history.

## Best Possible Solution

Define the WebChat long-history contract, preferably by adding an explicit `chat.history` truncation signal or pagination/load-older flow and wiring the Control UI so bounded history never looks like silent message loss.

## Reproduction Assessment

Yes for the bounded-history/no-notice path: current source shows the browser requests 100 messages, the gateway tails only that many, and the renderer’s hidden-count notice only fires when it receives more than 100. The exact Safari/DeepSeek transcript case was not reproduced because the raw transcript was not provided.

## Solution Assessment

Unclear. A renderer-only `serverHistoryTruncated` notice is a useful narrow fix for the misleading UI, but fully satisfying the report likely needs a maintainer choice between additive `hasMore` metadata, limit-plus-one probing, or a real load-older/pagination contract.
