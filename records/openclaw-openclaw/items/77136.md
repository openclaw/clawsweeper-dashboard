## Label Justifications

- `P1`: Multiple user reports describe WebChat hiding assistant replies in real sessions, which breaks trust in an active agent/chat workflow even though durable transcript data may remain intact.
- `impact:message-loss`: The issue is about assistant messages being absent from the WebChat surface while still present in transcript/TUI history.
- `impact:session-state`: The visible WebChat state can diverge from durable session history because the client reconstructs only a bounded history window without a truncation signal.

## Summary

Keep open: current main still has a source-proven WebChat long-history path where Control UI asks for only 100 messages, Gateway returns a bounded tail without `hasMore`/cursor/truncation metadata, and the renderer can only warn when hidden history is already present in the received payload.

## What This Changes

WebChat should render or clearly account for all assistant messages present in durable transcript history instead of making older replies appear lost while TUI still shows them.

## Best Possible Solution

Define an explicit long-history contract for WebChat, preferably additive `chat.history` truncation metadata or pagination, then wire Control UI so older transcript history is either loadable or clearly disclosed instead of silently falling outside the visible window.

## Reproduction Assessment

Yes, at source level: Control UI requests exactly 100 `chat.history` messages, Gateway returns only that bounded tail without older-history metadata, and the renderer warning only fires when more than 100 visible messages are already in the payload. The exact Safari/DeepSeek transcript was not live-reproduced because the raw transcript was not provided.

## Solution Assessment

Unclear. A renderer-only notice could reduce confusion, but the durable fix needs maintainer choice between limit-plus-one probing, additive `hasMore`/cursor metadata, or full load-older pagination such as the still-open work in https://github.com/openclaw/openclaw/pull/72178.
