## Summary

Keep open: current main still has a source-proven WebChat long-history path where assistant messages can fall outside the browser's 100-message history window with no truncation signal, and the later renderer-only sketch is useful context but not a merged maintainer-selected fix.

## What This Changes

The issue asks WebChat to keep assistant transcript messages visible, or at least stop making bounded history look like silent message loss.

## Best Possible Solution

Define a clear WebChat long-history contract, preferably by adding explicit chat.history truncation metadata or a load-older/pagination path, then wire Control UI so bounded history never looks like silent assistant-message loss.

## Reproduction Assessment

Yes, for the bounded-history/no-notice path: current source shows the browser requests 100 messages, the gateway returns only that tail, and the renderer only warns when it receives more than 100. The exact Safari/DeepSeek transcript case was not reproduced because the raw transcript was not provided.

## Solution Assessment

Unclear. Jim Dawdy's renderer-only serverHistoryTruncated sketch is a useful narrow fix, but maintainers still need to choose between limit-plus-one probing, additive hasMore metadata, or a full load-older/pagination contract.
