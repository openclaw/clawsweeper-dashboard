## Summary

Keep open. Current main still has bounded WebChat history paths that can make older assistant messages vanish from the browser view, and the exact Safari/DeepSeek transcript was not provided to rule out a narrower projection bug.

## What This Changes

The issue asks WebChat to render assistant history from the transcript consistently with the TUI instead of dropping visible replies after reload or session switch.

## Best Possible Solution

Define WebChat's long-history contract and implement visible pagination/count metadata or a load-older path while preserving Gateway payload caps.

## Reproduction Assessment

Yes for the source-level bounded-history path: Control UI requests 200 messages, Gateway returns a recent 200-message window, and the renderer has its own 200-message display cap. No for the reporter's exact Safari/DeepSeek transcript shape because the transcript was not attached.

## Solution Assessment

Unclear. Raising a limit alone is unlikely to be the best maintainable fix; a durable solution should expose truncation/counts or add a load-older path so WebChat does not look like it lost transcript-backed replies.
