## Summary

Keep open: current `main` still has a plausible WebChat history/rendering gap for long sessions because the UI reload path is bounded to 200 messages and does not expose an older-history cursor to the WebChat client.

## What This Changes

The issue reports that WebChat intermittently fails to render some assistant messages even though the transcript and TUI show the full conversation.

## Best Possible Solution

Keep this open and add a bounded WebChat history pagination or older-message loading path that preserves response byte caps and makes omitted older transcript content visible to users.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction class: create a WebChat session with more than 200 visible transcript messages, reload/switch back, and the current client can only request the last 200. It is still unclear whether that accounts for every reported Safari disappearance without the reporter's transcript.

## Solution Assessment

No, current `main` is not the best solution for the reported behavior because it silently bounds WebChat reloads without a WebSocket cursor or older-history UI. A safer fix is bounded pagination or an explicit older-history affordance, not an unbounded transcript fetch.
