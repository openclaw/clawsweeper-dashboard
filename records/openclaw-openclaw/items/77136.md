## Summary

Keep open. Current main still has a credible WebChat long-history path where older assistant messages can disappear from the browser view without the transcript being lost, and the report is recent enough that it should not be closed without the reporter transcript or a maintainer decision on the long-history contract.

## What This Changes

The issue requests that WebChat render assistant messages from stored transcripts consistently with the same session viewed in TUI, especially after long conversations or reloads.

## Best Possible Solution

Define WebChat's long-history behavior explicitly, then add pagination or truncation/count metadata so the UI can either load older transcript rows or clearly show that only a bounded tail is being displayed.

## Reproduction Assessment

Yes for the bounded-history path from source inspection: a transcript with more than 200 visible messages can be reduced to the last 200 by chat.history and rendered without a server-truncation notice. No for the reporter's exact Safari/DeepSeek transcript shape because the transcript was not attached.

## Solution Assessment

Unclear. Raising the limit alone is not the best maintainable fix unless maintainers accept the payload tradeoff; adding explicit truncation metadata or a load-older path would address the user-visible loss while preserving safety bounds.
