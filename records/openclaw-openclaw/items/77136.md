## Summary

Keep open. Current main still has a plausible WebChat-only disappearance path: the chat tab reloads a fixed 200-message `chat.history` window and renders a separate 200-message local window, while the newer paginated session-history HTTP/SSE surface is not wired into the chat view.

## What This Changes

The issue asks WebChat to render assistant messages from long transcripts consistently with TUI when the transcript data is intact.

## Best Possible Solution

Define WebChat long-history behavior and implement it visibly, either by wiring chat to paginated history/load-older or by returning truncation metadata so bounded history never appears like lost assistant output.

## Reproduction Assessment

Yes for the bounded-window path: source and tests show a transcript over 200 visible messages will reload in WebChat as only the recent `chat.history` window. No for the reporter's exact Safari/DeepSeek transcript shape because the transcript was not provided.

## Solution Assessment

Unclear. Raising limits alone is not the best fix; the safer direction is to use the existing paginated session-history surface or add RPC metadata/UX copy while preserving payload caps.
