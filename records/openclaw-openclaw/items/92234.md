## Summary

The revised branch still does not complete the reported user path: it adds a public wizard field but neither regenerates the shared Swift protocol model nor updates a native wizard renderer, so in-repo RPC clients cannot use the surfaced URL. It also still makes every RPC-backed provider auth run report `isRemote: true`, which changes provider flow selection beyond presentation, and the contributor has supplied only mocked/seam proof rather than a real client/provider OAuth run.

## What This Changes

Adds optional OAuth challenge metadata to gateway wizard text steps and threads authorization URLs through OpenAI, Chutes, and Gemini provider flows.

## Best Possible Solution

Define a discriminated, renderable auth-challenge union; keep client presentation capability separate from host locality; regenerate native protocol models; update the intended wizard renderer; and land redacted proof of a real RPC OAuth round trip.

## Reproduction Assessment

Yes from source: current RPC wizard steps can request redirect input while the authorization URL is only logged, and the current native `WizardStep` model has no field for the proposed metadata. A live failing client run was not supplied.

No. Surfacing metadata is the right direction, but this PR is not the narrowest complete fix because it omits the consuming client rollout, permits invalid challenge shapes, and changes provider locality semantics.
