## Summary

Keep open: the PR targets a real OAuth wizard gap, but current head still conflates RPC auth presentation with host locality, so local RPC wizard OAuth can skip browser/callback behavior; proof remains mock-only.

## What This Changes

The PR adds an optional wizard prompter auth-presentation flag and threads OAuth authorization URLs into redirect-paste prompt text across shared provider flows and several plugins.

## Best Possible Solution

Keep host locality tied to `isRemoteEnvironment()`, pass auth-presentation needs separately into OAuth prompt construction, align the PR body/client contract with the actual protocol shape, and verify a real RPC/provider OAuth flow.

## Reproduction Assessment

Yes from source: current main and v2026.6.6 expose wizard text steps while remote OAuth helpers log the auth URL separately and prompt only for the redirect URL. I did not run a live Windows companion or real provider OAuth login.

## Solution Assessment

No. Adding URL-bearing prompt text is a plausible narrow fix, but redefining host locality for all auth-presenting RPC prompters is not the best owner-boundary solution.
