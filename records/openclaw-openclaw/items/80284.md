## Summary

Keep open. Current main still lacks a diagnostics-otel-owned handler for OTLPExporterError rejections, and an open linked PR already targets this exact crash with closing syntax, so this issue should stay open until that PR lands or is closed.

## What This Changes

The issue reports a process crash when diagnostics-otel OTLP exporter failures surface as Error, plain-object, or array-wrapped OTLPExporterError unhandled rejections.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/81085 if it passes merge gates, then close this issue from the merged plugin-local handler and regression tests.

## Reproduction Assessment

Yes, source-reproducible: enable diagnostics-otel with OTLP/HTTP exporters, trigger an exporter failure such as HTTP 410, and current main has no diagnostics-otel handler before the global fatal fallback. I did not run a live collector crash reproduction.

## Solution Assessment

Yes. The plugin-local runtime-env handler proposed by the issue and open PR is the narrowest maintainable fix because diagnostics-otel owns the OTLP dependency and core already exposes the handler seam.
