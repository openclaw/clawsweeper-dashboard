## Summary

Keep open. Current main still lacks a diagnostics-otel-owned OTLPExporterError rejection handler, and the same crash is already paired with the open fixing PR at https://github.com/openclaw/openclaw/pull/81085, so the issue should remain open until that PR lands or is replaced.

## What This Changes

The issue requests that diagnostics-otel suppress OTLPExporterError unhandled rejections from unavailable OTLP collectors instead of letting the OpenClaw process crash.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/81085 or an equivalent plugin-local handler with lifecycle cleanup and regression coverage, then close this issue as implemented.

## Reproduction Assessment

Yes, source-reproducible: configure diagnostics-otel with OTLP/HTTP and trigger an export that returns a non-retryable response such as HTTP 410; current main has no plugin-local handler and the core fatal fallback does not classify OTLPExporterError. I did not run a live collector crash in this read-only pass.

## Solution Assessment

Yes, the plugin-local handler direction is the narrowest maintainable fix because diagnostics-otel owns the OTLP dependency and the SDK already exposes registerUnhandledRejectionHandler. The open fixing PR is the current best path, so the issue should stay open until that PR lands.
