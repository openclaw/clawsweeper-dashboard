## Summary

Keep open. Current main still lacks diagnostics-otel handling for OTLPExporterError, and an open linked fix PR already owns the implementation path, so this issue should close only after that PR or a superseding fix lands.

## What This Changes

The issue reports that OTLPExporterError rejections from diagnostics-otel can bypass plugin handling and crash the OpenClaw process when an OTLP collector is unavailable.

## Best Possible Solution

Review and land the linked diagnostics-otel handler PR after adequate real unavailable-collector proof or maintainer override, then close this issue with the merged commit as implementation proof.

## Reproduction Assessment

Yes, source-reproducible: configure diagnostics-otel with an OTLP endpoint that returns a non-retryable response such as HTTP 410, trigger export, and current main has no plugin handler before the global fatal fallback. I did not run a live collector failure in this read-only review.

## Solution Assessment

Yes, the requested plugin-local runtime-env handler is the narrowest maintainable solution because diagnostics-otel owns the OTLP dependency and exporter behavior. Adding OTLPExporterError to the core transient allowlist would put plugin-specific policy in core.
