## Summary

Keep open. Current main still wires diagnostics-otel OTLP exporters without a plugin-scoped unhandled-rejection handler, while the core process handler would fall through to the fatal unhandled-rejection path for an OTLPExporterError that is not classified as transient.

## What This Changes

The report asks for OTLP exporter failures from the diagnostics-otel plugin to be treated as non-fatal when the collector is unavailable or cancelled.

## Best Possible Solution

Add a diagnostics-otel-owned rejection handler that recognizes OTLPExporterError in Error, plain-object, and array-wrapped forms, logs a bounded warning/exporter diagnostic, unregisters on plugin stop, and leaves the core transient allowlist generic.

## Reproduction Assessment

Yes, source-reproducible: enable diagnostics-otel against an OTLP/HTTP endpoint that returns a non-retryable response such as HTTP 410, trigger export, and current main has no plugin handler before the core fatal fallback. I did not run a live collector crash repro in this read-only review.

## Solution Assessment

Yes, the plugin-local handler direction is the narrowest maintainable solution because diagnostics-otel owns the OTLP dependency and the SDK already exposes registerUnhandledRejectionHandler for plugin-owned benign rejections.
