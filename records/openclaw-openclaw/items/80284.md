## Summary

Keep open. Current main still matches the reported crash path, and an open PR already references and implements the candidate diagnostics-otel fix, so the issue should stay open until that PR lands or is closed.

## What This Changes

The issue requests that diagnostics-otel suppress OTLPExporterError exporter rejections as non-fatal telemetry failures instead of letting them crash the OpenClaw process.

## Best Possible Solution

Review and land the linked diagnostics-otel handler PR, or close that PR and replace it with the same plugin-owned suppression plus focused regression coverage.

## Reproduction Assessment

Yes. This is source-reproducible: enable diagnostics-otel with an OTLP endpoint that returns a non-retryable response such as HTTP 410, and current main has no diagnostics-otel handler before the global fatal fallback for an unmatched OTLPExporterError.

## Solution Assessment

Yes. A diagnostics-otel-owned registerUnhandledRejectionHandler path is the narrowest maintainable fix because the plugin owns the OTLP dependency and the SDK already exposes the runtime hook.
