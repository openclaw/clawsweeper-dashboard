## Summary

Keep open. Current main still has the diagnostics-otel OTLP exporters without plugin-local OTLPExporterError rejection handling, and the open fix PR is the right implementation path until it merges.

## What This Changes

The issue reports that diagnostics-otel OTLP exporter failures can surface as unhandled promise rejections and crash OpenClaw when the collector is unavailable.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/81085 if merge gates pass, then close this issue once the plugin-local suppression and regression coverage are on main.

## Reproduction Assessment

Yes, source-reproducible: current main creates OTLP exporters in diagnostics-otel, checks plugin handlers before the fatal fallback, and does not classify OTLPExporterError or the reported array-wrapped shape as transient. I did not run a live unavailable-collector crash reproduction.

## Solution Assessment

Yes. A diagnostics-otel-owned runtime-env handler is the narrowest maintainable fix because the failure is exporter-specific and avoids teaching the core global transient allowlist about OTLP details.
