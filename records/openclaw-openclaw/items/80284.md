## Summary

Keep open. Current main still wires diagnostics-otel OTLP protobuf exporters without a plugin-local unhandled-rejection handler, the core fallback still exits on unmatched rejections, and https://github.com/openclaw/openclaw/pull/80292 is the open closing fix path rather than a merged implementation.

## What This Changes

The issue reports that diagnostics-otel OTLP exporter failures can surface as unhandled promise rejections and crash the OpenClaw process.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/80292, or a replacement with the same diagnostics-otel-owned handler, after real unavailable-collector proof or an explicit maintainer proof override.

## Reproduction Assessment

Yes, source-reproducible: enable diagnostics-otel with an OTLP endpoint that returns a non-retryable response such as HTTP 410, trigger export, and current main has no plugin handler before the core fatal fallback. I did not run a live collector crash repro in this read-only review.

## Solution Assessment

Yes, the narrow maintainable fix is a diagnostics-otel-owned registerUnhandledRejectionHandler that recognizes OTLPExporterError shapes and leaves the core transient allowlist generic. The linked open PR follows that direction but still needs normal PR handling before this issue closes.
