## Summary

Keep open. Current main still has the diagnostics-otel OTLP exporters wired without a plugin-local unhandled-rejection handler, while the core global handler would fall through to process exit for an OTLPExporterError that is not otherwise classified as transient.

## What This Changes

The issue requests that diagnostics-otel suppress and warn on OTLPExporterError rejections from unavailable OTLP collectors instead of allowing a process crash.

## Best Possible Solution

Add a diagnostics-otel-owned unhandled-rejection registration that recognizes OTLPExporterError in Error, plain-object, and array-wrapped shapes, logs a bounded warning, unregisters on stop, and leaves the core transient allowlist generic.

## Reproduction Assessment

Yes, source-reproducible: enable diagnostics-otel with an OTLP endpoint that returns a non-retryable response such as HTTP 410, trigger export, and the current plugin has no handler before the global fatal fallback. I did not run a live collector crash repro in this read-only pass.

## Solution Assessment

Yes, the issue’s plugin-local handler direction is the narrowest maintainable fix because diagnostics-otel owns the OTLP dependency and the SDK already exposes registerUnhandledRejectionHandler for plugin-owned benign rejections.
