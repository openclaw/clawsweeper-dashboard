## Summary

Keep open. Current main still has the diagnostics-otel exporter path without plugin-local OTLPExporterError handling, and an open linked PR already uses closing syntax for this issue, so the right cleanup point is after that PR lands or is closed.

## What This Changes

Suppress diagnostics-otel OTLPExporterError unhandled rejections when an OTLP collector is unavailable so telemetry export failures do not crash OpenClaw.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/80292 after sufficient real unavailable-collector proof or maintainer override, then close this issue as implemented.

## Reproduction Assessment

Yes, source-reproducible: enable diagnostics-otel with an OTLP endpoint that returns a non-retryable response such as HTTP 410, trigger export, and current main has no diagnostics-otel handler before the global fatal fallback. I did not run a live collector crash repro in this read-only review.

## Solution Assessment

Yes, the plugin-local handler direction is the narrowest maintainable fix because diagnostics-otel owns the OTLP dependency and the plugin SDK already exposes the rejection-handler seam. The open linked PR should be reviewed and proven rather than closing this issue early.
