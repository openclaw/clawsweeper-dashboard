## Summary

Keep open. Current main still has the diagnostics-otel OTLP exporter crash path unresolved, and the live issue is already linked to an open fixing PR with closing syntax, so the issue should remain open until that PR is merged or closed.

## What This Changes

The issue reports that diagnostics-otel OTLP exporter failures can surface as unhandled rejections and crash OpenClaw when an OTLP collector is unavailable or returns a non-retryable response.

## Best Possible Solution

Land the diagnostics-otel-owned unhandled-rejection handler and regression coverage through https://github.com/openclaw/openclaw/pull/81085, keep the core transient allowlist generic, then close this issue after the fix merges.

## Reproduction Assessment

Yes, source-reproducible: enable diagnostics-otel OTLP export, have the collector return a non-retryable response such as HTTP 410 during export, and current main has no diagnostics-otel handler before the core fatal fallback. I did not run a live collector crash repro in this read-only review.

## Solution Assessment

Yes. A diagnostics-otel-owned runtime-env handler is the narrowest maintainable fix because the OTLP dependency and exporter lifecycle are plugin-owned, and the linked open PR follows that direction.
