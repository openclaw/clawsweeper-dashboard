## Summary

Keep open. Current main still lacks diagnostics-otel-owned handling for OTLPExporterError, and a focused open pull request already links this issue with closing syntax, so the issue should remain open until that fix merges.

## What This Changes

Treat diagnostics-otel OTLPExporterError rejections as non-fatal plugin-owned telemetry export failures instead of letting the global rejection handler exit the process.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/81085 if it passes merge gates, then close this crash issue with the merge commit and release or main-only provenance.

## Reproduction Assessment

Yes, source-reproducible: enable diagnostics-otel against an OTLP endpoint returning a non-retryable status such as HTTP 410, trigger export, and current main has no plugin-local handler before the core fatal fallback. I did not run a live collector crash reproduction.

## Solution Assessment

Yes. A diagnostics-otel-owned runtime-env rejection handler is the narrowest maintainable fix because the OTLP dependency is plugin-owned and the core transient allowlist should stay generic.
