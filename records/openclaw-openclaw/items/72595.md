Keep open. Current main still lacks Feishu/Lark mixed-proxy docs and any Feishu-owned per-channel proxy policy, while the same-author docs mitigation at https://github.com/openclaw/openclaw/pull/75050 remains open and runtime direct/ambient egress needs maintainer judgment.

## What This Changes

Request a Feishu/Lark channel proxy policy so provider traffic can keep using ambient proxy env while Feishu token and WebSocket traffic can go direct or follow an explicit channel policy.

## Best Possible Solution

Land the docs-only NO_PROXY mitigation if maintainers accept it, and handle runtime work as an explicit Feishu-owned proxy policy covering Lark SDK HTTP, guarded fetch, WebSocket, managed service env boundaries, tests, and docs.

## Reproduction Assessment

Yes for the missing capability: current main has no Feishu proxy config key, no mixed-proxy docs, and WebSocket routing still follows ambient env. I did not live-reproduce the reporter's Windows proxy 400 response.

## Solution Assessment

Unclear. A docs-only NO_PROXY path is the narrowest low-risk mitigation, while a runtime proxyMode-style option is more complete but needs maintainer approval because it changes channel egress behavior.
