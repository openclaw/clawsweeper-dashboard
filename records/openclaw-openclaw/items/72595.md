## Summary

Keep open. Current main still lacks Feishu/Lark mixed-proxy troubleshooting docs and any Feishu-owned direct/ambient proxy policy, while the same-author docs PR remains the active mitigation path rather than a merged fix.

## What This Changes

Request a Feishu/Lark channel-specific proxy bypass or documented NO_PROXY guidance so Feishu token and WebSocket traffic can go direct while model-provider traffic continues using ambient proxy env vars.

## Best Possible Solution

Review and land #75050 if docs-only NO_PROXY guidance is the accepted mitigation. If runtime behavior is desired, implement a Feishu-owned proxy policy that applies consistently to createFeishuClient SDK HTTP calls, fetchWithSsrFGuard token/card paths, and createFeishuWSClient, with focused tests and docs.

## Reproduction Assessment

Yes. The issue gives a concrete Windows mixed-proxy setup, affected tenant_access_token endpoint, and observed HTTP/HTTPS proxy error; current source inspection confirms separate SDK HTTP, guarded fetch, and WebSocket transport paths without a shared policy.

## Solution Assessment

Unclear. Docs-only NO_PROXY guidance is the lowest-risk mitigation and already has open PR #75050, while a proxyMode-style runtime option is more complete but needs maintainer approval because it changes channel egress policy.
