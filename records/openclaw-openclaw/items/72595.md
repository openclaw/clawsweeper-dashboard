## Summary

Keep open. Current main still lacks Feishu/Lark mixed-proxy troubleshooting docs and any Feishu-owned proxy policy, and the same-author docs PR #75050 is still the active mitigation path rather than a merged fix.

## What This Changes

The request is to let Feishu/Lark channel traffic bypass or explicitly choose proxy routing while model-provider traffic can continue using ambient proxy environment variables.

## Best Possible Solution

Review and land #75050 if docs-only NO_PROXY guidance is accepted; otherwise implement a Feishu-owned proxy policy covering SDK HTTP, guarded fetch token/card calls, and WebSocket with focused tests and docs.

## Reproduction Assessment

Yes. The report gives a concrete Windows mixed-proxy setup, affected tenant_access_token endpoint, and observed HTTP/HTTPS proxy error; current source confirms separate SDK HTTP, guarded fetch, and WebSocket paths without a shared policy.

## Solution Assessment

Unclear. Docs-only NO_PROXY guidance is the lowest-risk mitigation and already has open PR #75050, while a proxyMode-style runtime option is more complete but needs maintainer approval because it changes channel egress policy.
