## Summary

Keep open. Current main still lacks Feishu mixed-proxy docs or a Feishu-owned direct/ambient proxy policy, and the same-author docs PR remains open as the active low-risk mitigation while a runtime bypass needs maintainer egress-policy review.

## What This Changes

Request Feishu/Lark-specific proxy guidance or channel policy so provider traffic can keep the ambient proxy while Feishu token and WebSocket traffic can bypass it when needed.

## Best Possible Solution

Land PR #75050 if docs-only NO_PROXY guidance is accepted; if runtime control is needed, add a Feishu-owned proxy policy covering SDK HTTP and WebSocket paths with tests and explicit egress semantics.

## Reproduction Assessment

Yes for the missing capability: current main has no Feishu proxy config key, createFeishuClient has no channel proxy override, and createFeishuWSClient follows ambient proxy env when configured. The exact Windows proxy error was not live-reproduced in this read-only review.

## Solution Assessment

Unclear. Docs-only NO_PROXY guidance is the narrowest mitigation and is already open in PR #75050, while a proxyMode-style runtime option is more complete but changes channel egress policy and needs maintainer approval.
