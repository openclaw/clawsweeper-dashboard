## Summary

Keep open. Current main still lacks Feishu/Lark mixed-proxy troubleshooting docs and lacks a Feishu-owned per-channel proxy policy; the same-author docs mitigation #75050 remains open rather than merged, so this should not be cleanup-closed yet.

## What This Changes

Request Feishu/Lark channel-specific proxy handling or documentation so model-provider traffic can keep using ambient proxy env while Feishu HTTP and WebSocket traffic can bypass it when needed.

## Best Possible Solution

Review and land #75050 if docs-only NO_PROXY guidance is the accepted mitigation. If runtime behavior is desired, implement a Feishu-owned proxy policy that applies consistently to createFeishuClient SDK HTTP calls, fetchWithSsrFGuard token/card paths, and createFeishuWSClient, with focused tests and docs.

## Reproduction Assessment

Yes. The issue gives a concrete Windows mixed-proxy setup, affected tenant_access_token endpoint, and observed HTTP/HTTPS proxy error; current source inspection confirms separate SDK HTTP, guarded fetch, and WebSocket transport paths without a shared Feishu policy.

## Solution Assessment

Unclear. Docs-only NO_PROXY guidance is the lowest-risk mitigation and already has open PR #75050, while a proxyMode-style runtime option is more complete but needs maintainer approval because it changes channel egress policy.
