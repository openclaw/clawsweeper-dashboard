## Summary

Keep open. Current main still lacks Feishu/Lark mixed-proxy documentation and any per-channel proxy policy, and the reporter already has an open docs-only PR for this exact area, so maintainers should decide whether docs guidance is enough or whether Feishu runtime egress policy should change.

## What This Changes

Request a Feishu/Lark channel-specific proxy bypass or policy so provider traffic can continue using ambient proxy env vars while Feishu token and WebSocket traffic can go direct when needed.

## Best Possible Solution

Review and land #75050 if NO_PROXY guidance is the accepted mitigation. If maintainers want runtime behavior, implement the policy inside the Feishu plugin and apply it consistently to createFeishuClient SDK HTTP calls, fetchWithSsrFGuard token/card requests, and createFeishuWSClient, with docs and focused tests.

## Reproduction Assessment

Yes. The issue gives a concrete Windows mixed-proxy setup, affected Feishu tenant_access_token endpoint, and observed HTTP/HTTPS proxy error; current-main source inspection confirms there is no channel-specific proxy option and that Feishu uses separate SDK HTTP, guarded fetch, and WebSocket transport paths.

## Solution Assessment

Unclear. Docs-only NO_PROXY guidance is the lowest-risk mitigation and already has an open PR, while a proxyMode-style runtime option is more complete but needs maintainer approval because it changes Feishu channel egress policy.
