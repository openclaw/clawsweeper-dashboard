## Summary

Keep open. Current main still lacks Feishu/Lark mixed-proxy documentation and a Feishu-owned direct/ambient proxy policy, while the same-author docs PR #75050 remains the active mitigation path rather than a merged fix.

## What This Changes

Request Feishu/Lark proxy guidance or channel-level proxy policy so Feishu token and WebSocket traffic can bypass ambient proxy settings while provider traffic keeps using them.

## Best Possible Solution

Review and land #75050 if docs-only NO_PROXY guidance is accepted; otherwise implement a Feishu-owned proxy policy covering SDK HTTP, guarded fetch, and WebSocket paths with focused tests and docs.

## Reproduction Assessment

Yes. The report gives a concrete Windows mixed-proxy setup, the tenant_access_token endpoint, and the observed HTTP/HTTPS proxy error; current source confirms the missing Feishu proxy policy and separate HTTP/WS/fetch paths.

## Solution Assessment

Unclear. Docs-only NO_PROXY guidance is the lowest-risk mitigation and already has an open PR, while a proxyMode-style runtime option is more complete but changes channel egress policy and needs maintainer review.
