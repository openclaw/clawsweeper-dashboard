## Summary

Keep open. Current main still lacks Feishu/Lark mixed-proxy troubleshooting docs and any Feishu-owned direct/ambient proxy policy, while the same-author docs PR #75050 remains an open mitigation rather than a merged fix.

## What This Changes

Request Feishu/Lark channel support or guidance for bypassing ambient proxy on channel traffic while provider traffic continues using proxy env.

## Best Possible Solution

End state should be either accepted docs guidance in #75050 or a Feishu-owned proxy policy that applies consistently to SDK HTTP, guarded fetches, and WebSocket with focused docs and tests.

## Reproduction Assessment

Yes. The report gives a concrete Windows mixed-proxy setup, affected tenant_access_token endpoint, and observed HTTP/HTTPS proxy error; current source confirms separate SDK HTTP, guarded fetch, and WebSocket paths without a shared policy.

## Solution Assessment

Unclear. Docs-only NO_PROXY guidance is the lowest-risk mitigation and already has open PR #75050, while a proxyMode-style runtime option is more complete but needs maintainer approval because it changes channel egress policy.
