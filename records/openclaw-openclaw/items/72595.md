## Summary

Keep open: current main and v2026.6.8 still lack a Feishu-owned mixed proxy policy or docs mitigation, and the related open PRs cover conflicting direct, ambient, and docs-only directions that need maintainer/security review.

## What This Changes

The issue requests Feishu/Lark channel-specific proxy handling so provider traffic can keep ambient proxy settings while Feishu token/API and WebSocket traffic can bypass or explicitly choose proxy routing.

## Best Possible Solution

Settle one Feishu-owned egress policy covering SDK REST/token HTTP, WS bootstrap/socket traffic, guarded-fetch call sites, docs, and focused tests before merging either docs-only or runtime proxy changes.

## Reproduction Assessment

Yes at source level: current main has no Feishu proxy config key or mixed-proxy docs, Feishu SDK HTTP is timeout-only, and WebSocket still follows the ambient proxy helper. I did not live-reproduce the Windows proxy 400 response.

## Solution Assessment

Unclear until maintainers choose the egress policy. Docs-only NO_PROXY guidance is the lowest-risk mitigation, while a runtime proxyMode or default change needs product and security approval.
