## Summary

Keep open. Current main still lacks Feishu/Lark mixed-proxy guidance and has no channel-level proxy policy; the issue is also paired with open docs PR #75050, so maintainers should choose docs-only mitigation versus runtime channel behavior before cleanup.

## What This Changes

Request Feishu/Lark channel support or documentation for routing channel HTTP/WebSocket traffic independently from model-provider proxy environment variables.

## Best Possible Solution

Keep this issue open while maintainers review #75050. If docs-only NO_PROXY guidance is accepted, land that PR and close afterward; if runtime behavior is desired, implement a Feishu-owned proxy policy that applies consistently to SDK HTTP token/API calls and WebSocket startup, with focused tests and docs.

## Reproduction Assessment

Yes. The issue gives a concrete Windows mixed-proxy setup, affected Feishu tenant_access_token endpoint, and observed HTTPS-port proxy error, and current-main source inspection confirms HTTP token/API and WebSocket routing still have separate, non-configurable proxy behavior.

## Solution Assessment

Unclear. Docs-only NO_PROXY guidance is the lowest-risk mitigation and already has an open PR, while a channel-level proxyMode would be more complete but changes channel network egress policy and needs maintainer choice first.
