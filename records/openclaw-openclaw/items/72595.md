## Summary

Keep open: current main and v2026.6.6 still lack a Feishu-owned mixed-proxy policy or docs mitigation, and the related docs/runtime PRs are open while maintainers still need to choose direct, ambient, or managed-proxy semantics.

## What This Changes

Request Feishu/Lark channel traffic to support a channel-specific proxy bypass or documented NO_PROXY path so provider traffic can keep using an environment proxy while Feishu token and WebSocket traffic can go direct when needed.

## Best Possible Solution

Settle one Feishu-owned egress policy, then land or reject the paired docs mitigation and runtime PRs so Lark SDK HTTP, WebSocket bootstrap/socket routing, managed proxy mode, NO_PROXY guidance, tests, and docs agree.

## Reproduction Assessment

Yes at source level: current main has no Feishu proxy config key or mixed-proxy docs, Feishu REST/token HTTP is timeout-only, and WebSocket still follows the ambient proxy helper. I did not live-reproduce the reporter's Windows proxy 400 response.

## Solution Assessment

Unclear. A docs-only NO_PROXY update is the lowest-risk mitigation, but a runtime proxyMode-style option needs maintainer approval because it changes credentialed channel egress behavior.
