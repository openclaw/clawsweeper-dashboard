## Summary

Keep open. Current main still lacks Feishu/Lark mixed-proxy docs and a Feishu-owned direct/ambient proxy policy; same-author PR #75050 is the active docs mitigation path, while runtime proxy bypass needs maintainer proxy/security judgment.

## What This Changes

Request Feishu/Lark channel support or documentation so provider traffic can keep using ambient proxy env while Feishu token/API/WebSocket traffic can bypass or explicitly choose proxy policy.

## Best Possible Solution

Review and land #75050 if docs-only NO_PROXY guidance is the accepted mitigation; otherwise implement a Feishu-owned proxy policy covering SDK HTTP, WebSocket, and guarded fetch paths with focused tests and docs.

## Reproduction Assessment

Yes. The issue gives a concrete Windows mixed-proxy setup, Feishu WebSocket mode, proxy env vars, the tenant_access_token endpoint, and the observed HTTP-to-HTTPS-port error; source inspection confirms the missing Feishu proxy policy.

## Solution Assessment

Unclear. Docs-only guidance is the lowest-risk near-term path and already has an open PR, while a proxyMode-style runtime option is more complete but changes channel egress behavior and needs explicit maintainer review.
