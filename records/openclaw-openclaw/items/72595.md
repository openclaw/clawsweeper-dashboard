## Summary

Keep open. Current main still lacks Feishu/Lark mixed-proxy troubleshooting docs and any Feishu channel proxy policy, and the same-author docs PR #75050 is still the active implementation candidate for the low-risk mitigation path.

## What This Changes

The issue requests Feishu/Lark channel proxy guidance or a per-channel proxy policy so provider traffic can keep using ambient proxy while Feishu/Lark HTTP and WebSocket traffic can bypass it when needed.

## Best Possible Solution

Review and land #75050 if docs-only NO_PROXY guidance is the accepted resolution. If runtime behavior is desired, implement a Feishu-owned proxy policy that applies consistently to createFeishuClient SDK HTTP calls, fetchWithSsrFGuard Feishu token/card paths, and createFeishuWSClient startup, with focused tests and docs.

## Reproduction Assessment

Yes. The issue gives a concrete Windows mixed-proxy setup, affected Feishu tenant_access_token endpoint, and observed HTTP/HTTPS proxy error; current source inspection shows separate SDK HTTP, guarded fetch, and WebSocket transport paths without a shared channel policy.

## Solution Assessment

Unclear. Docs-only NO_PROXY guidance is the lowest-risk mitigation and already has open PR #75050, while a proxyMode-style runtime option is more complete but needs maintainer approval because it changes channel egress policy.
