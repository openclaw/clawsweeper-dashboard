## Summary

Keep open. Current main still lacks Feishu/Lark mixed-proxy troubleshooting docs and has no Feishu-owned direct/ambient proxy policy; the same-author docs PR #75050 remains the active mitigation path rather than a merged fix.

## What This Changes

Request Feishu/Lark support for a documented or configured per-channel proxy policy so provider traffic can keep env proxy while Feishu token and WebSocket traffic can bypass it when needed.

## Best Possible Solution

Review and land #75050 if docs-only NO_PROXY guidance is accepted; otherwise implement a Feishu-owned proxy policy covering SDK HTTP, streaming-card guarded fetch, setup fetches, and WebSocket with focused tests and docs.

## Reproduction Assessment

Yes. The report gives a concrete Windows mixed-proxy setup, affected tenant_access_token endpoint, and observed HTTP/HTTPS proxy error; current source confirms separate SDK HTTP, guarded fetch, and WebSocket paths without a shared policy.

## Solution Assessment

Unclear. Docs-only NO_PROXY guidance is the lowest-risk mitigation and already has open PR #75050, while a proxyMode-style runtime option is more complete but changes channel egress policy and needs maintainer approval.
