## Summary

Keep open. Current main still has no Feishu/Lark mixed-proxy troubleshooting docs and no Feishu-owned proxy policy, while the same-author docs PR #75050 is open as an active mitigation path rather than a merged fix.

## What This Changes

The issue requests a Feishu/Lark channel-specific proxy policy or documentation so provider traffic can use ambient proxy while Feishu/Lark API and WebSocket traffic can bypass it.

## Best Possible Solution

Review and land #75050 if docs-only NO_PROXY guidance is accepted; otherwise define a Feishu-owned proxy policy that consistently covers SDK HTTP, WebSocket, and streaming-card fetch paths with tests and docs.

## Reproduction Assessment

Yes. The issue gives a concrete Windows mixed-proxy setup, proxy env vars, affected tenant_access_token endpoint, and observed HTTP/HTTPS proxy error; current source and upstream SDK inspection confirm the relevant Feishu transport paths exist.

## Solution Assessment

Unclear. NO_PROXY documentation is the lowest-risk immediate mitigation, while a proxyMode-style runtime option is more complete but needs maintainer approval because it changes channel egress policy.
