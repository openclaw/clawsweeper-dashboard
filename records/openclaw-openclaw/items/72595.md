## Summary

Keep open. Current main still lacks Feishu/Lark mixed-proxy troubleshooting docs and a channel-level proxy policy, and the issue is paired with open docs PR #75050, so maintainers should choose docs-only mitigation versus runtime proxy behavior before cleanup.

## What This Changes

Request to let Feishu/Lark channel traffic bypass or explicitly control ambient proxy settings while provider traffic continues using environment proxies, with consistent HTTP/WebSocket handling and docs/tests.

## Best Possible Solution

Review and land #75050 if docs-only NO_PROXY guidance is the accepted resolution. If runtime behavior is desired, implement a Feishu-owned proxy policy that applies consistently to createFeishuClient SDK HTTP calls, fetchWithSsrFGuard Feishu token/card paths, and createFeishuWSClient startup, with focused tests and docs.

## Reproduction Assessment

Yes. The issue provides a concrete Windows mixed-proxy setup, affected Feishu token endpoint, and observed HTTP/HTTPS proxy error; current source inspection shows no channel policy and separate SDK HTTP, guarded fetch, and WebSocket transport paths.

## Solution Assessment

Unclear. Docs-only NO_PROXY guidance is the lowest-risk mitigation and already has an open PR, while a proxyMode-style runtime option is more complete but needs maintainer approval because it changes channel egress policy.
