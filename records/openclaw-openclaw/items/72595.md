## Summary

Keep open. Current main still lacks Feishu mixed-proxy docs on the published Feishu page and lacks a Feishu-owned direct/ambient proxy policy; the same-author docs mitigation remains open, while runtime egress behavior overlaps narrower active Feishu proxy issues that need maintainer judgment.

## What This Changes

Request a Feishu/Lark channel proxy policy or documentation so provider traffic can keep using ambient proxy while Feishu token and WebSocket traffic can bypass it when needed.

## Best Possible Solution

Land the docs-only NO_PROXY mitigation if maintainers accept it, then handle runtime behavior as an explicit Feishu-owned proxy policy covering Lark SDK HTTP, guarded fetch, and WebSocket paths with tests and documentation, coordinated with https://github.com/openclaw/openclaw/issues/65799, https://github.com/openclaw/openclaw/issues/48949, and https://github.com/openclaw/openclaw/issues/78842.

Yes for the missing capability: current source shows no Feishu proxy config key, no mixed-proxy docs, no HTTP proxy override, and WebSocket routing still follows ambient env. I did not live-reproduce the reported Windows 400 response.

Unclear. The open docs PR is the narrowest low-risk mitigation, while a runtime direct/ambient proxy policy is more complete but needs maintainer approval because it changes channel egress behavior.
