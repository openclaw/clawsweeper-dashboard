## Summary

Keep open. Current main still lacks Feishu/Lark mixed-proxy troubleshooting docs and any Feishu-owned proxy policy, and the same-author docs PR #75050 remains the active low-risk mitigation path rather than a merged fix.

## What This Changes

Request a Feishu/Lark channel-specific proxy bypass or docs guidance so Feishu token and WebSocket traffic can go direct while model-provider traffic continues using ambient proxy env vars.

## Best Possible Solution

Review and land #75050 if docs-only NO_PROXY guidance is the accepted mitigation. If maintainers want runtime behavior, implement a Feishu-owned proxy policy that applies consistently to createFeishuClient SDK HTTP calls, fetchWithSsrFGuard token/card paths, and createFeishuWSClient, with docs and focused tests.

## Reproduction Assessment

Yes. The issue provides a concrete Windows mixed-proxy setup, endpoint, and HTTP/HTTPS proxy error, and current source inspection confirms separate Feishu HTTP, fetch, and WebSocket transport paths without a shared policy.

## Solution Assessment

Unclear. The linked docs PR is the narrowest low-risk mitigation, while a proxyMode-style runtime option is more complete but needs maintainer approval because it changes network egress policy.
