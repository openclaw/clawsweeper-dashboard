## Summary

Keep open. Current main still lacks Feishu/Lark mixed-proxy troubleshooting docs and an explicit Feishu-owned proxy policy; same-author PR #75050 remains the active docs mitigation, while a runtime direct/ambient option needs maintainer security review.

## What This Changes

Request Feishu/Lark channel proxy controls or NO_PROXY guidance so channel traffic can bypass ambient env proxy while provider traffic continues using it.

## Best Possible Solution

Review and land #75050 if docs-only NO_PROXY guidance is the accepted mitigation; otherwise implement a Feishu-owned proxy policy covering SDK HTTP, guarded fetch, and WebSocket paths with focused tests and docs.

## Reproduction Assessment

Yes. The report gives a concrete Windows mixed-proxy setup, endpoint, proxy env vars, and HTTP/HTTPS error, and current main confirms the missing Feishu proxy policy across the relevant paths.

## Solution Assessment

Unclear. Docs-only NO_PROXY guidance is the lowest-risk near-term mitigation, while proxyMode-style runtime behavior is more complete but changes channel egress policy and needs maintainer review.
