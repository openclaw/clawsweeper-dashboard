## Summary

Keep open. Current main and v2026.6.6 still lack a Feishu-owned mixed-proxy policy or merged mixed-proxy docs, the same-author docs mitigation at https://github.com/openclaw/openclaw/pull/75050 remains open and dirty, and the related runtime fix path at https://github.com/openclaw/openclaw/pull/86386 is open for the older canonical proxy bug rather than merged.

## What This Changes

Request Feishu/Lark channel traffic to support per-channel proxy bypass so provider traffic can keep using ambient proxy environment variables while Feishu token and WebSocket traffic can go direct or follow an explicit channel policy.

## Best Possible Solution

Settle the Feishu egress policy once, then land either a corrected docs-only NO_PROXY mitigation or a maintainer-approved runtime policy that covers both Lark SDK HTTP/token traffic and WebSocket/bootstrap traffic without changing existing proxy users by surprise.

## Reproduction Assessment

Yes for the missing capability from source inspection: current main has no Feishu proxy policy key or mixed-proxy docs, Feishu HTTP/token calls use the timeout-only SDK wrapper, and WebSocket still follows ambient proxy behavior. I did not live-reproduce the reporter's Windows proxy 400 response.

## Solution Assessment

Unclear as a product choice. A docs-only NO_PROXY path is the lowest-risk interim, while a runtime proxyMode-style option is more complete but requires maintainer approval because it changes credentialed channel egress semantics.
