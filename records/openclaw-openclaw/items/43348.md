## Summary

Keep this PR open. Current main still does not expose the proposed `acp_send` agent tool, `sessions_cancel` agent tool, or `sessions.cancel` Gateway RPC, but the PR is not merge-ready because the new cancellation path bypasses existing session visibility/A2A policy and the fire-and-forget send path accepts non-ACP targets where no ACP callback can deliver the result.

## What This Changes

The PR adds `acp_send` and `sessions_cancel` agent tools, registers their display/tool entries, and adds a Gateway `sessions.cancel` schema, validator, scope entry, and server handler backed by ACP session cancellation.

## Best Possible Solution

Rework the PR on current main so any agent-facing ACP cancel path uses the same session resolution, visibility, sandbox, and A2A policy model as existing session tools, validates that targets are ACP-backed before ACP-only operations, and updates protocol docs/tool catalog/tests around the new public surface.

## Reproduction Assessment

Yes. The cancellation issue can be reproduced with a focused unit/integration test where one agent receives or guesses another ACP session key and calls `sessions_cancel`; the PR path forwards that key to `acpManager.cancelSession()` without a visibility guard. The `acp_send` delivery issue can be reproduced by targeting a normal/subagent session and observing the accepted fire-and-forget run has no ACP callback path.

## Solution Assessment

No. The direction may be useful, but the current PR is not the safest implementation because it exposes cross-session cancellation and generic fire-and-forget dispatch without ACP-only validation. The narrower maintainable fix is to share the existing session access checks and add ACP metadata validation before invoking ACP-specific behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Enforce session visibility before cancelling:** `src/agents/tools/sessions-cancel-tool.ts:32-34`
  - body: `sessions_cancel` forwards the raw `sessionKey` to `sessions.cancel`, and the new handler only validates the key before calling `acpManager.cancelSession()`. A model that learns another ACP session key can cancel that run even when `tools.sessions.visibility` or A2A policy should block cross-session control, so the target needs to be resolved and guarded like `sessions_send` before cancellation.
  - confidence: 0.92
- **[P2] Reject non-ACP targets before dispatch:** `src/agents/tools/acp-send-tool.ts:218-238`
  - body: `acp_send` accepts any visible resolved session and sends an `agent` request with `deliver: false`, but only ACP sessions have the callback path this tool relies on. Targeting a normal or native subagent session can return `accepted` while dropping the reply, so validate the resolved target is ACP-backed before dispatching.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The PR introduces a new cross-session cancellation surface that currently bypasses existing session visibility and A2A authorization checks.

Concerns:

- **[high] Unscoped ACP session cancellation:** `src/agents/tools/sessions-cancel-tool.ts:32`
  - body: The proposed `sessions_cancel` tool and `sessions.cancel` handler accept a caller-provided key and call `acpManager.cancelSession()` without resolving whether that session is visible or controllable by the requesting agent. This can let an agent cancel another ACP session if the key appears in context or prior tool output.
