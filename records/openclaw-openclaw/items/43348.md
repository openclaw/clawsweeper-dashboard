## Summary

Keep open. Current main still does not implement the proposed `acp_send`, `sessions_cancel`, or `sessions.cancel` surfaces, so this PR is not obsolete; however, the branch is not merge-ready because it adds security-sensitive ACP/session control without the same visibility authorization used by existing session tools and is stale against current main.

## What This Changes

The PR adds `acp_send` and `sessions_cancel` agent tools, display metadata, and a Gateway `sessions.cancel` protocol/schema/scope/handler backed by `AcpSessionManager.cancelSession()`.

## Best Possible Solution

Rework this PR on current main so ACP-only agent tools share the existing session reference resolution, sandbox/tree/self/cross-agent visibility, and A2A policy model; validate that ACP-only operations target ACP-backed sessions; update the current tool display/config surfaces; and add focused tests for unauthorized cancel, non-ACP send rejection, and successful ACP callback delivery.

## Reproduction Assessment

Yes. The cancellation issue is reproducible with a focused tool/RPC test where one agent passes another ACP session key to `sessions_cancel`; the PR path forwards that key to `acpManager.cancelSession()` without the visibility guard used by `sessions_send`. The `acp_send` issue is reproducible by targeting a visible native/subagent session and observing an accepted fire-and-forget run with no ACP callback path for the reply.

## Solution Assessment

No. The feature direction may be useful, but the current PR is not the safest implementation because it exposes raw-key ACP cancellation and accepts non-ACP targets for a callback-dependent tool. The safer alternative is to factor or reuse the existing session access checks and add ACP session-key/metadata validation before invoking ACP-specific behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Enforce session visibility before cancelling:** `src/agents/tools/sessions-cancel-tool.ts:32-34`
  - body: `sessions_cancel` forwards the caller-provided `sessionKey` directly to `sessions.cancel`, and the new handler only requires a non-empty key before calling `acpManager.cancelSession()`. A caller that sees another ACP key can cancel a session that `tools.sessions.visibility` or A2A policy should hide, so resolve and guard the target like `sessions_send` before invoking cancel.
  - confidence: 0.92
- **[P1] Use the current TypeBox import:** `src/agents/tools/acp-send-tool.ts:2`
  - body: Current main has migrated schema imports to `typebox` and `package.json` no longer declares `@sinclair/typebox`. The new tool files import `@sinclair/typebox`, so this branch will fail module resolution after rebasing to current main; update both new tool imports to `typebox`.
  - confidence: 0.9
- **[P2] Reject non-ACP targets before dispatch:** `src/agents/tools/acp-send-tool.ts:218-238`
  - body: `acp_send` resolves any visible session and dispatches an internal `agent` run with `deliver: false`, but only ACP sessions have the callback path this tool relies on. Targeting a native/subagent session can return `accepted` while losing the reply, so validate the resolved target is ACP-backed before sending.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: Security review needs attention because the PR introduces cross-session ACP cancellation without session visibility authorization.

Concerns:

- **[high] Unscoped ACP session cancellation:** `src/agents/tools/sessions-cancel-tool.ts:32`
  - body: The proposed `sessions_cancel` tool and `sessions.cancel` handler accept a caller-provided key and call `acpManager.cancelSession()` without resolving whether that session is visible or controllable by the requesting agent. This can let an agent cancel another ACP session if the key appears in context or prior tool output.
  - confidence: 0.92
