## Summary

Keep open: current main still lacks the proposed ACP send/cancel tools, but this PR is not merge-ready because it exposes ACP cancellation without the existing session visibility guard, accepts non-ACP targets for a callback-dependent send path, uses a stale TypeBox import, and lacks the required changelog entry.

## What This Changes

The PR adds `acp_send` and `sessions_cancel` agent tools, display metadata, and a Gateway `sessions.cancel` protocol/schema/scope/handler backed by `AcpSessionManager.cancelSession()`.

## Best Possible Solution

Rework this PR on current main so ACP-only agent tools share existing session resolution, visibility, and A2A policy checks; validate ACP-backed targets before callback-dependent sends; keep the Gateway method scoped intentionally; add focused regression tests and a changelog entry.

## Reproduction Assessment

Yes. The cancellation issue is source-reproducible by following `sessions_cancel` from raw `sessionKey` input to `sessions.cancel` and then to `AcpSessionManager.cancelSession()` without the guard used by `sessions_send`; the `acp_send` issue is source-reproducible by targeting any visible non-ACP session and observing that the PR dispatches `agent` with `deliver: false` anyway.

## Solution Assessment

No. The feature direction may be useful, but this implementation is not the safest path because it puts ACP-specific control behind generic raw-key dispatch; the narrower maintainable fix is to reuse the current session access helpers and verify ACP metadata before invoking ACP-only behavior.
