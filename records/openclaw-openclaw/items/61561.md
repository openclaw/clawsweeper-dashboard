## Summary

Keep this PR open. The BlueBubbles satellite-message bug still appears valid on current main, but the patch is incomplete: documented `chat_guid:`/`guid:` targets bypass the new bare-GUID normalization and would still pass `iMessageLite` through to the send payload. The PR also needs the required user-facing changelog entry before merge.

## What This Changes

The PR adds a BlueBubbles helper to map `iMessageLite` chat GUID services to `iMessage`, applies it during webhook normalization and bare raw GUID target parsing, and adds monitor-normalize tests.

## Best Possible Solution

Revise this as a BlueBubbles-local fix with one canonical chatGuid service normalization helper used anywhere a chatGuid is parsed, sent, or compared, including prefixed target forms; keep core plugin APIs and unrelated channel behavior untouched.

## Reproduction Assessment

Yes. A focused parser/send-path check can reproduce the PR defect: `chat_guid:iMessageLite;-;+...` is parsed by the prefix parser before `parseRawChatGuid`, then returned directly by `resolveChatGuidForTarget`.

## Solution Assessment

No. The PR's direction is narrow and appropriate, but it is not the best complete fix until prefixed `chat_guid:`/`guid:` parsing and changelog coverage are handled through the same BlueBubbles-local path.
