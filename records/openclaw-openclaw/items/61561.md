## Summary

Keep open. The underlying BlueBubbles bug is real on current main, but this PR is not merge-ready because documented prefixed `chat_guid:` targets and `chat_guid:` allowlist entries still bypass the new normalization, and the user-facing fix lacks changelog coverage.

## What This Changes

The PR adds a BlueBubbles helper that maps `iMessageLite` chat GUID services to `iMessage`, applies it during webhook normalization and bare raw GUID parsing, and adds monitor-normalize tests.

## Best Possible Solution

Keep the fix BlueBubbles-local and canonicalize the chatGuid service component through one helper anywhere BlueBubbles parses, sends, compares, or formats chat GUIDs.

## Reproduction Assessment

Yes. The parser path is enough to reproduce the PR defect: `chat_guid:iMessageLite;-;+...` is returned by the prefix parser before `parseRawChatGuid`, then `resolveChatGuidForTarget` returns that value directly. The allowlist mismatch is similarly reproducible by comparing normalized inbound `iMessage;-;...` against a raw `chat_guid:iMessageLite;-;...` allowlist entry.

## Solution Assessment

No. The PR's direction is narrow and appropriate, but the complete solution should also normalize prefixed target parsing and allowlist comparison with the same BlueBubbles helper, then add changelog coverage.
