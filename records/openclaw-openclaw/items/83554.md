## Summary

Keep open. Current main supports observed in-memory group history, Discord message reads, and prompt hooks, but I did not find a native opt-in session-start platform scrollback preload; older exact requests were closed as stale or covered only the manual read-tool path, and the broader memory/context umbrella does not own this narrower channel-provider contract.

## What This Changes

Add an opt-in native channel-history preload that fetches recent platform messages for shared channels at session start or refresh time and injects them as session context.

## Best Possible Solution

Make a maintainer product/API call first: publish this as a ClawHub/plugin pattern if current hooks are enough, or add a narrow provider-owned channel-history preload contract with opt-in config, budgets, refresh/pruning, untrusted-context marking, docs, and focused tests.

## Reproduction Assessment

Not applicable: this is a feature request, not a broken current contract. The high-confidence check is source and docs inspection showing observed in-memory history and manual read hooks, but no automatic session-start platform scrollback preload.

## Solution Assessment

Unclear as requested: a core-wide `preloadHistory` knob is not obviously the best boundary. The safer path is a maintainer decision between a ClawHub/plugin implementation and a narrow provider-owned preload contract before code is written.

## Risks / Open Questions

- Native preload would inject untrusted historical channel content, so privacy, prompt-injection labeling, channel visibility, allowlists, and permission boundaries need explicit design.
- A generic cross-channel feature may need a provider-owned SDK/channel contract; placing Discord/Slack/Telegram-specific fetch policy directly in core could duplicate plugin ownership and drift.
- Prompt-build or session-start network fetches could add latency, rate-limit, or availability problems unless budgeted, throttled, cached, pruned, and fail-open as the issue describes.
