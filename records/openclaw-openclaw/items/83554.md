## Summary

Keep open. Current main and the latest release still provide observed-message history limits and plugin prompt hooks, but not a native opt-in platform scrollback preload at session start; the reporter’s Discord plugin workaround is useful evidence, and the remaining question is a product/API boundary decision.

## What This Changes

The issue requests an opt-in native channel-history preload that fetches recent platform messages for Discord and other shared channels before the first prompt build and after an inactivity refresh interval.

## Best Possible Solution

Make a maintainer product/API call first: either keep Discord scrollback preload as plugin/ClawHub work, or define a narrow provider-owned history-preload contract with budgets, throttling, untrusted-context marking, docs, and focused channel tests.

## Reproduction Assessment

Not applicable: this is a feature request, not broken existing behavior. Source and docs inspection show current main records observed channel messages in runtime maps but does not fetch native platform scrollback at session start.

## Solution Assessment

Unclear as requested: a core-wide `preloadHistory` knob is not obviously the best boundary. The maintainable path is either a provider/plugin contract or an explicit decision that this remains plugin/ClawHub-owned.

## Risks / Open Questions

- A native preload would inject historical channel content into prompt context, so permission scope, privacy expectations, and prompt-injection labeling need explicit design.
- A generic cross-channel feature may require a provider-owned SDK/config contract; doing it only in core or only in Discord could duplicate channel policy across plugins.
- Prompt-build-time platform fetches can add latency, rate-limit pressure, and availability failures unless budgets, throttling, pruning, and fail-open behavior are part of the contract.
