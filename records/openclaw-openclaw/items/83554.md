## Summary

Keep open. Current main still has rolling observed-message history, Discord read-message tools, and plugin prompt hooks, but no native opt-in platform scrollback preload at session start; the remaining work needs maintainer API, config, privacy, and prompt-injection boundary decisions.

## What This Changes

Add an opt-in native channel-history preload that fetches recent platform messages at session start or after an idle refresh, starting with Discord and ideally generalizing across channels.

## Best Possible Solution

Make a maintainer product/API call first: either keep Discord-specific preloading as ClawHub/plugin-owned work, or define a narrow provider-owned channel-history preload contract with explicit config, permission checks, budgets, untrusted-context envelopes, docs, and focused channel tests.

## Reproduction Assessment

Not applicable: this is a feature request, not a broken existing behavior. Current-main source/docs comparison shows no preloadHistory or refresh-budget setting and only rolling observed-message history.

## Solution Assessment

Unclear as requested: the desired outcome is valid, but a core-wide preload knob is not yet proven as the best boundary. The safer path is a maintainer decision between plugin-owned Discord behavior and a generic provider-owned preload contract.

## Risks / Open Questions

- A native preload can inject old untrusted channel content into model context, so envelope placement and prompt-injection labeling need explicit design.
- Provider-token REST fetching must respect channel permissions, allowlists, bot scopes, and private group/DM boundaries before it becomes native behavior.
- Prompt-build-time platform fetches can add latency, rate-limit pressure, and availability coupling unless budgets, throttling, pruning, and fail-open behavior are part of the contract.
