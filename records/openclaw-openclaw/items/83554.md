## Summary

Keep open. Current main has observed-message history buffers, Slack DM-specific backfill, plugin prompt hooks, and a focused Discord reconnect-backfill PR, but not the requested native opt-in shared-channel scrollback preload at session start.

## What This Changes

The issue requests native opt-in channel history preloading at session start or after an idle refresh for Discord and other shared channels.

## Best Possible Solution

Make the product/API boundary decision first; if native, define a channel-owned opt-in history-preload contract with budgets, throttling, pruning, fail-open behavior, untrusted-context marking, docs, and focused Discord plus at least one non-Discord test path.

## Solution Assessment

Unclear as requested: a core-wide `preloadHistory` knob is not obviously the right boundary. The maintainable direction is either a provider/channel-owned SDK contract or an explicit decision to keep this as plugin/ClawHub work.

## Risks / Open Questions

- A native preload would inject untrusted historical channel content into model context, so permission scope, privacy expectations, and prompt-injection labeling need explicit design.
- Prompt-build or session-start REST fetches can add latency, rate-limit pressure, and availability failures unless budgets, throttling, pruning, aborts, and fail-open behavior are part of the contract.
- A generic cross-channel knob could conflict with channel-local ownership unless maintainers define whether this belongs in provider plugins, a shared SDK contract, or ClawHub/plugin guidance.
