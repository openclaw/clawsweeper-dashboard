## Summary

Keep open. Current main still has observed-message channel history and plugin prompt hooks, plus a Slack thread-specific history fetch, but it does not implement the requested native opt-in platform scrollback preload for Discord or a generic cross-channel contract; the remaining work needs product/API and security boundary decisions before implementation.

## What This Changes

The issue requests native opt-in channel history preloading at session start or after idle refresh for Discord and other shared channels.

## Best Possible Solution

Make a maintainer product/API call first: either keep this as a ClawHub/plugin-owned Discord capability, or define a narrow provider-owned generic history-preload contract with opt-in config, budgets, throttling, untrusted-context marking, docs, and focused channel tests.

## Reproduction Assessment

Not applicable: this is a feature request, not broken existing behavior. Source and docs inspection show current main has observed-message history limits and plugin prompt hooks, but no native platform scrollback preload at session start.

## Solution Assessment

Unclear as requested: a core-wide `preloadHistory` knob is not obviously the best boundary. The maintainable path is likely either a provider-owned SDK/channel contract or an explicit decision to keep this as ClawHub/plugin work.

## Risks / Open Questions

- A native preload would inject untrusted historical channel content into model context, so permission scope, privacy expectations, and prompt-injection labeling need explicit design.
- Prompt-build-time platform fetches can add latency, rate-limit pressure, or availability failures unless the contract includes budgets, aborts, throttling, pruning, and fail-open behavior.
- A Discord-only implementation could drift from Slack, Telegram, and other channels unless maintainers choose a provider-owned generic contract or deliberately keep it plugin-local.
