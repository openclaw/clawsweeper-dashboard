## Summary

Keep this PR open: it targets a real linked Qwen/tool replay failure and the retry-disable direction is plausible, but the current repeated-thinking detector can inject a loop warning across unrelated user turns because it only looks at the last three assistant turns globally. The branch also needs the required user-facing changelog coverage before merge.

## What This Changes

The PR disables Pi SDK auto-retry by returning an in-memory retry-disabled prepared settings manager, then adds repeated-thinking loop-hint injection for both transport-message replay and OpenAI-completions payload construction.

## Best Possible Solution

Land a focused runtime fix that disables Pi's inner auto-retry only for embedded OpenClaw runs, scopes repeated-thinking detection to an uninterrupted tool-result loop or resets on real user turns, covers Qwen/openai-completions replay with focused tests, and records the user-facing fix in the changelog.

## Reproduction Assessment

Yes. The linked bug report gives a concrete Qwen/openai-completions replay pattern, and source inspection confirms current main leaves Pi auto-retry enabled while Qwen uses the completions builder path that currently bypasses OpenClaw transport-message transformation.

## Solution Assessment

No. The in-memory retry override and completions wiring are the right shape, but the repeated-thinking guard should not scan across unrelated user prompts before appending a synthetic user hint; it needs a narrower loop-scope check first.
