## Summary

Keep open: the PR targets a real source-level retry/replay bug, but it still has a concrete detector scoping bug, needs current-main test alignment, lacks contributor real behavior proof, and is now conflicting/stale against main.

## What This Changes

The PR disables embedded Pi SDK auto-retry through prepared runtime settings and adds repeated-thinking loop hints to agent transport message conversion, including the OpenAI completions path used by Qwen.

## Best Possible Solution

Land or consolidate a rebased fix that disables Pi SDK retry only for OpenClaw-managed runtime sessions, scopes any repeated-thinking hint to an active replay cycle, adds focused regression coverage, and leaves broader circuit-breaker policy to https://github.com/openclaw/openclaw/issues/78865.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level, but not with live proof here: the linked report gives the Qwen/Feishu replay shape, current main leaves Pi SDK retry enabled, and the pinned SDK removes the failed assistant turn before calling continue().

## Solution Assessment

Is this the best way to solve the issue? Not yet: disabling Pi auto-retry is the right narrow direction, but the repeated-thinking detector must reset/scope across real user turns and the branch needs current-main tests, proof, and consolidation with the overlapping retry PR.
