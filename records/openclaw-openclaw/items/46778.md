## Summary

Keep open. Current main still resolves non-media Feishu messages to the raw `message_id`, and the receive, debounce, and final persistent dedupe gates all consume that key, so same-logical text deliveries with different IDs can still be processed twice; the related reconnect and media reused-ID items cover different cases.

## What This Changes

The issue requests Feishu private-chat duplicate suppression when Feishu delivers the same logical message with different `message_id` values.

## Best Possible Solution

Add a Feishu-local logical text retry fingerprint used consistently by receive, debounce, and final persistent dedupe while preserving the existing media-aware key behavior.

## Reproduction Assessment

Yes, source-level. A focused check can feed two Feishu p2p text events with the same account, sender, chat/thread, `create_time`, and content but different `message_id` values; current main resolves them to different dedupe keys.

## Solution Assessment

Yes. The best fix is narrower than a bare content hash: combine stable Feishu event fields such as chat/thread, sender, timestamp, and normalized content, while preserving existing media/file uniqueness.
