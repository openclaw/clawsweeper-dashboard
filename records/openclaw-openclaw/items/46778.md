## Summary

Keep open. Current main and the latest shipped release still derive non-media Feishu text dedupe from the raw `message_id`, so the reported same-logical-message deliveries with different IDs can still pass the receive and final persistent dedupe gates.

## What This Changes

The issue asks for Feishu private-chat duplicate suppression when the same text message is delivered with different `message_id` values.

## Best Possible Solution

Add a Feishu-local logical text retry fingerprint, based on stable sender/chat/thread/create_time/content fields, and use it consistently across receive claims, debounce suppression, and final persistent dedupe while preserving the existing media-aware key behavior.

## Reproduction Assessment

Yes, source-level. A focused check can feed two Feishu p2p text events with the same account, sender, chat/thread, `create_time`, and content but different `message_id` values; current main resolves them to different dedupe keys.

## Solution Assessment

Yes. The narrowest maintainable fix is a Feishu-local logical text retry fingerprint at the existing dedupe key helper and its current call sites, not a core dedupe API change or the broad stale-message rejection approach in the unrelated Feishu mega-PR.
