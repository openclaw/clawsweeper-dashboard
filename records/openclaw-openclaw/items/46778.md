## Summary

Keep open. Current main still resolves non-media Feishu messages to raw `message_id`, and the receive, debounce, and final persistent dedupe gates all consume that key, so same-logical text events with different IDs can still be processed twice; the merged media-aware dedupe work addresses a different reused-ID media case.

## What This Changes

The issue requests Feishu private-chat dedupe for duplicate logical messages that arrive with different `message_id` values.

## Best Possible Solution

Add a Feishu-local logical text retry fingerprint that is used consistently by receive, debounce, and final persistent dedupe while preserving the existing media-aware keys.

## Reproduction Assessment

Yes, source-level. A focused test can feed two Feishu p2p text events with the same account, sender, chat/thread, `create_time`, and content but different `message_id` values; current main resolves them to different dedupe keys.

## Solution Assessment

Yes, with a narrower implementation than a bare content hash. The maintainable fix should combine stable Feishu event fields such as chat/thread, sender, timestamp, and normalized content, and keep the existing media/file uniqueness behavior intact.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/46778 by adding a conservative Feishu-local logical duplicate key for text/private-chat retry events that arrive with different `message_id` values. Use or extend `resolveFeishuMessageDedupeKey` so receive-time claims, debounce suppression, suppressed-ID recording, and final persistent dedupe all share the same key. The text key should include stable event identity such as account namespace, chat/thread/root context, sender id, `create_time`, message type, and normalized text content; do not use a bare content hash. Preserve the existing media-aware `message_id + file_key/image_key` behavior from https://github.com/openclaw/openclaw/issues/75057 and do not introduce new core APIs, config options, or a broad reconnect/offline replay policy for https://github.com/openclaw/openclaw/issues/41195. Add focused regression coverage for two p2p text events with identical logical fields and different IDs, plus a guard that distinct media variants still dispatch. Add a single-line `CHANGELOG.md` fix entry if the change is user-visible.
