## Summary

Keep open. Current main still uses the resolved Feishu dedupe key as bare `message_id` for non-media text messages, so the reported same-logical-text/private-chat duplicate with different IDs remains possible; the related closed reconnect and media-dedupe items cover different parts of the Feishu dedupe surface.

## What This Changes

The issue requests Feishu private-chat duplicate suppression when the same logical text message arrives with different `message_id` values.

## Best Possible Solution

Add a Feishu-local logical text retry fingerprint that is reused by receive, debounce, and final persistent dedupe, while preserving the current media-aware key behavior and avoiding a broad core dedupe change.

## Reproduction Assessment

Yes, source-level. A focused check can feed two Feishu p2p text events with the same account, sender, chat/thread, `create_time`, and content but different `message_id` values; current main resolves them to different dedupe keys.

## Solution Assessment

Yes. The narrowest maintainable fix is a Feishu-local logical text retry fingerprint based on stable event fields, not a bare content hash or shared core dedupe change.

## Repair Work Prompt

Fix the Feishu duplicate text-message bug tracked by https://github.com/openclaw/openclaw/issues/46778. Current main keys non-media Feishu text dedupe on bare `message_id`, so the same logical p2p text event can be processed twice when Feishu delivers it with different IDs. Implement a conservative Feishu-local logical text retry fingerprint in `extensions/feishu/src/dedupe-key.ts` and reuse it consistently in receive claims, debounce suppression, final persistent dedupe, and error-release paths already wired through `resolveFeishuMessageDedupeKey`. Preserve the current media-aware `message_id + file_key/image_key` behavior from https://github.com/openclaw/openclaw/pull/76408, and do not move this into shared core dedupe. Add regression coverage for two Feishu p2p text events with the same account, sender, chat/thread, `create_time`, and content but different `message_id` values, while retaining coverage that distinct media attachments with reused IDs still dispatch independently. Add a single-line changelog fix entry if the branch is a user-facing repair. Stop and ask for maintainer judgment if the implementation would require a config option or policy change for repeated identical user messages.
