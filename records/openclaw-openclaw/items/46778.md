## Summary

Keep open: current main and the latest release still dedupe ordinary Feishu text deliveries by the resolved `message_id`, so the reported same-message/different-`message_id` DM duplicate path remains source-reproducible. The linked Feishu mega-PR contains a related create-time idea, but it is open, conflicting, broad, and already flagged for message-delivery/security risks, so it is not a safe close or canonical landing path.

## What This Changes

The issue requests Feishu private-chat duplicate suppression when Feishu redelivers the same text message with a different `message_id`.

## Best Possible Solution

Land a narrow Feishu plugin fix that adds a bounded logical-message retry signature for duplicate text deliveries while preserving the existing media-aware `message_id` behavior.

## Reproduction Assessment

Yes, at source level: two equivalent Feishu text receive events with the same sender/chat/create_time/content but different `message_id` values resolve to different dedupe keys and can both be processed. I did not run a live Feishu replay in this read-only review.

## Solution Assessment

No, the reporter's broad content-hash fallback is not the best shape by itself because it can suppress distinct same-content messages. The best fix is a narrow, bounded Feishu retry signature using sender/chat/time/content fields and the existing plugin-state dedupe path.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/46778 with a narrow Feishu-plugin-only repair. Current main dedupes ordinary text by `resolveFeishuMessageDedupeKey(event)`, which falls back to `message_id`; duplicate Feishu DM deliveries with the same sender/chat/create_time/content but different `message_id` can both pass `createFeishuMessageReceiveHandler` and `handleFeishuMessage`. Add a bounded logical retry signature for text duplicate deliveries that preserves existing media-aware `message_id + file_key/image_key` behavior and avoids monotonic timestamp watermark drops. Keep the fix inside `extensions/feishu`, avoid core/plugin SDK changes unless absolutely necessary, do not add config flags, do not edit `CHANGELOG.md`, and put user-facing release-note context in the PR body or commit message. Add focused regression coverage for same sender/chat/create_time/content with different `message_id` being dropped, plus a guard that distinct legitimate messages are not suppressed.
