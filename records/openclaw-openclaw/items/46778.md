## Summary

Keep open. Current main and the latest shipped release still derive non-media Feishu text dedupe from the raw `message_id`, so same-logical-message deliveries with different IDs can pass the receive, debounce, and final persistent dedupe gates.

## What This Changes

The issue asks for Feishu private-chat duplicate messages with different `message_id` values to be deduped by a stable sender, timestamp, and content fallback.

## Best Possible Solution

Add a Feishu-local logical text retry fingerprint from stable sender, chat/thread, create_time, and normalized content fields, and use it consistently across receive claims, debounce suppression, and final persistent dedupe while preserving the existing media-aware key behavior.

## Reproduction Assessment

Yes, source-level. Feed two Feishu p2p text events with the same account, sender, chat/thread, `create_time`, and content but different `message_id` values; current main resolves them to different dedupe keys.

## Solution Assessment

Yes. The narrow maintainable fix is a Feishu-local logical text retry fingerprint at the existing dedupe helper and current call sites, not a new core dedupe API or a user-facing config option.
