- `impact:message-loss`: Duplicate inbound Feishu deliveries can be processed as separate user turns when the same logical message arrives with different IDs.

## Summary

Keep open. Current main and the latest release still use the raw Feishu message_id as the dedupe key for plain text messages, so the reported same-logical-message/different-ID duplicate path remains source-reproducible.

## What This Changes

The issue asks Feishu to suppress duplicate private-chat text deliveries when Feishu sends the same logical message with different message_id values.

## Best Possible Solution

Add a Feishu-local logical text retry fingerprint from stable sender, chat/thread, create_time, and normalized text, then use that key consistently through receive claims, debounce suppression, and final persistent dedupe while preserving existing media-aware keys.

## Reproduction Assessment

Yes, source-level. Feed two Feishu p2p text events for the same account with the same sender, chat/thread, create_time, and normalized content but different message_id values; the current key resolver returns different raw IDs and all downstream gates treat them as different messages.

## Solution Assessment

Yes. The maintainable fix is Feishu-local key normalization in or near resolveFeishuMessageDedupeKey plus regression coverage; no core dedupe API, user-facing config option, or broad recovery module is needed.
