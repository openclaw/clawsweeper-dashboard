## Summary

Keep open. Current main still dedupes non-media Feishu messages by the resolved message_id-only key, so repeated private text/content deliveries with fresh Feishu IDs can pass both the receive-time claim and final persistent dedupe gates; the closed related work covers restart warmup and same-ID media collisions, not this fresh-ID logical duplicate path.

## What This Changes

Request to prevent duplicate Feishu private messages when Feishu delivers the same content with a different message_id by adding a sender/time/content fallback dedupe key.

## Best Possible Solution

Add a Feishu-local logical-message fingerprint as a secondary dedupe key for non-media messages, preserving existing message_id and media-aware behavior while adding focused regression coverage for same sender/chat/time/content with different message IDs.

## Reproduction Assessment

Yes, source-level. Two valid Feishu text receive events for the same account/chat/sender/content with different message_id values resolve to different keys and pass the receive and final dedupe checks on current main.

## Solution Assessment

Yes, with a narrow boundary. The maintainable fix is Feishu-local: compute a short-window logical fingerprint from account/chat or thread, sender, create_time or a tight time bucket, message type, and normalized content, rather than changing shared core dedupe semantics.
