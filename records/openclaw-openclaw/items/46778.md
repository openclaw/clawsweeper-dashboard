## Summary

Current main still treats ordinary Feishu text deliveries with different `message_id` values as distinct, so the reported duplicate-processing gap remains. The related open PR proposes create-time deduplication but bundles many unrelated changes and lacks positive real-behavior proof, so it is not yet a safe canonical fix.

## What This Changes

Prevent equivalent Feishu private-message deliveries with different message IDs from being processed more than once.

## Best Possible Solution

Add one Feishu-owned logical retry key derived from a verified provider invariant, apply it consistently at the early claim and persistent finalization boundaries, and prove that fresh-ID retries collapse while intentional repeated text and distinct media remain deliverable.

## Reproduction Assessment

Yes at source level: two equivalent Feishu text events with different `message_id` values resolve to different keys and both pass the current-main receive and final dedupe gates. A live Feishu retry payload was not captured in this review.

## Solution Assessment

No, the proposed broad content hash is not the safest fix because it can collapse legitimate repeated messages. A narrower Feishu replay identity based on verified event or creation metadata is the maintainable direction.
