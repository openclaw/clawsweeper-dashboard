## Summary

Keep open. Current main and the latest release still key Feishu inbound duplicate suppression to `message_id`, so the reported same-logical-message/different-id path remains unresolved; a newer related Feishu dedupe PR addresses the inverse media case but does not close or implement this text/content fallback.

## What This Changes

The issue requests Feishu private-chat inbound dedupe that can suppress the same logical message even when Feishu delivers it with different `message_id` values.

## Best Possible Solution

Add a Feishu-owned logical duplicate fingerprint shared by the receive and final-processing paths, while preserving id/media-key uniqueness for #75057. Cover same text with different ids, true retries, distinct repeated text, and media cases in focused Feishu tests without moving service-specific policy into core.

## Reproduction Assessment

Yes. A focused unit reproduction can enqueue two Feishu text events for the same account, chat/thread, sender, `create_time`, and content with different `message_id` values; current source inspection shows both ids pass the early and persistent dedupe gates.

## Solution Assessment

Unclear in exact implementation, but the direction is Feishu-local rather than core-wide. The best fix should not be a plain content hash; it should compose stable event identity fields and account for the open media-dedupe work in #75078.
