## Summary

Keep open. Current main and the latest shipped tag still route plain Feishu text dedupe through raw `message_id`, so the reported same-logical-message/different-ID path remains source-reproducible and is not replaced by the broad open Feishu omnibus PR.

## What This Changes

The issue asks OpenClaw to suppress duplicate Feishu private-message deliveries when the same logical text message is pushed again with a different `message_id`.

## Best Possible Solution

Add Feishu-local logical text retry fingerprinting from stable sender, chat/thread, `create_time`, and normalized text content, while preserving the existing media-aware key behavior.

## Reproduction Assessment

Yes, source-level. Feed two Feishu p2p text events for the same account with the same sender, chat/thread, `create_time`, and normalized content but different `message_id` values; current key resolution and downstream gates treat them as different messages.

## Solution Assessment

Yes. The narrow maintainable fix is Feishu-local dedupe-key normalization plus regression coverage; no core dedupe API, user-facing config option, or broad recovery module is needed for this bug.
