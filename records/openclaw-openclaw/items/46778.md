## Summary

Keep open: current main and the latest release still dedupe ordinary Feishu text messages by the resolved `message_id` key, so the reported same-message/different-`message_id` DM duplicate path remains source-reproducible. The linked Feishu mega-PR mentions a create-time dedupe approach, but it is open, broad, and currently flagged as unsafe to merge, so it is not a close reason.

## What This Changes

The issue requests reliable Feishu inbound deduplication when the same private-message delivery arrives twice with different `message_id` values.

## Best Possible Solution

Land a narrow Feishu plugin fix that adds a bounded, lossless retry identity for duplicate text deliveries while preserving the existing media-aware `message_id` behavior and avoiding the broad open PR branch.

## Reproduction Assessment

Yes, at source level: two equivalent Feishu text events with the same sender/chat/create_time/content but different `message_id` values currently produce different dedupe keys and can both be processed. I did not run a live Feishu replay in this read-only review.

## Solution Assessment

No, the reporter's broad content-hash fallback is not the best shape by itself because it can suppress distinct same-content messages. The better fix is a narrow, bounded Feishu retry signature using documented event fields and existing plugin-state dedupe paths.
