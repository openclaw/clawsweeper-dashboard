## Summary

Keep open: current main and the latest release still dedupe ordinary Feishu text messages by the resolved `message_id`, so the reported same-message/different-`message_id` DM duplicate path remains source-reproducible. The related broad Feishu PR mentions create-time dedupe, but it is open, conflicting, missing proof, and flagged with delivery/security blockers, so it is not a safe close reason.

## What This Changes

The issue requests a Feishu inbound dedupe fix for duplicate private-chat deliveries where the same message arrives with different `message_id` values.

## Best Possible Solution

Land a narrow Feishu plugin fix that adds a bounded, lossless logical-message retry identity for duplicate text deliveries while preserving the existing media-aware `message_id` plus media-key behavior.

## Reproduction Assessment

Yes, at source level: two equivalent Feishu text events with the same sender, chat, create_time, and content but different `message_id` values produce different dedupe keys and can both pass the receive and finalization gates. I did not run a live Feishu replay in this read-only review.

## Solution Assessment

No, the reporter's broad content-hash fallback is not the best shape by itself because it can suppress distinct same-content messages. The better repair is a narrow Feishu-local retry signature with a short window and regression tests, after validating which Feishu fields stay stable across duplicate delivery.
