## Summary

Keep open. Current main and the latest release still resolve plain Feishu text dedupe to the raw `message_id`, so the reported same-logical-message/different-ID duplicate path remains source-reproducible; related closed Feishu items cover reconnect persistence or media reused-ID behavior, not this text retry gap.

## What This Changes

The issue asks Feishu to suppress duplicate private-chat deliveries when the same logical text message arrives more than once with different `message_id` values.

## Best Possible Solution

Add Feishu-local logical text retry fingerprinting from stable sender, chat/thread, `create_time`, and normalized text content, while preserving the existing media-aware key behavior and avoiding a new core API or config option.

## Reproduction Assessment

Yes, source-level. Feed two Feishu p2p text events for the same account with the same sender, chat/thread, `create_time`, and normalized content but different `message_id` values; current key resolution and downstream gates treat them as different messages.

## Solution Assessment

Yes. The narrow maintainable fix is Feishu-local dedupe-key normalization with regression tests; a broad recovery module, new config flag, or core dedupe API is not needed for this bug.
