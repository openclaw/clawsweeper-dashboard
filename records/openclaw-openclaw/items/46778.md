## Summary

Keep open. Current main still returns the raw Feishu `message_id` for non-media text dedupe, and the receive, debounce, and final persistent gates all consume that key, so same-logical text deliveries with fresh IDs can still be processed twice.

## What This Changes

The issue requests Feishu WebSocket/private-chat dedupe that suppresses duplicate logical text messages even when Feishu assigns different `message_id` values.

## Best Possible Solution

Add one Feishu-local logical text retry fingerprint used by the existing receive, debounce, and final persistent dedupe gates, while preserving the current media-aware key behavior.

## Reproduction Assessment

Yes, source-level. A focused check can feed two Feishu p2p text events with the same account, sender, chat/thread, `create_time`, and content but different `message_id` values; current main resolves them to different dedupe keys.

## Solution Assessment

Yes. The narrowest maintainable solution is a Feishu-local logical text retry key based on stable event fields, not a bare content-only hash and not a core dedupe API change.
