## Summary

Keep open: current main and the latest release still dedupe Feishu text receive events by the resolved message ID key, with media keys added only for media attachments. That leaves the reported same-logical-message/new-`message_id` path open, and https://github.com/openclaw/openclaw/issues/41195 was already closed in favor of this issue as the canonical remaining tracker.

## What This Changes

The issue reports duplicate Feishu private-chat processing when Feishu sends the same message content with different `message_id` values, defeating message-id dedupe.

## Best Possible Solution

Add a Feishu-local secondary logical-message fingerprint for text receive events with new IDs, bounded by account/chat or thread, sender, create_time, message type, and normalized content, while preserving existing message-id and media-aware dedupe.

## Reproduction Assessment

Yes, source-level: two Feishu text receive events with the same sender, chat/thread, create_time, and content but different `message_id` values resolve to different dedupe keys and pass both the receive and final processing gates. I did not run a live Feishu tenant because this was a read-only review.

## Solution Assessment

Yes, a narrow Feishu-local logical-message fingerprint is the maintainable fix shape; a broad content-only key or timestamp watermark would be riskier because it could suppress legitimate repeated messages or distinct media.
