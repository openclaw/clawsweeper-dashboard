## Summary

Keep open. Current main and the latest release still dedupe plain Feishu text messages by raw `message_id` at the receive, debounce, and final persistent gates, so the reported same-logical-message/different-ID path remains source-reproducible.

## What This Changes

The issue asks OpenClaw to suppress duplicate Feishu private-chat deliveries when the same logical text message arrives with different `message_id` values.

## Best Possible Solution

Add a Feishu-local logical text retry fingerprint from stable sender, chat/thread, `create_time`, and normalized text content, and feed that same key through receive claims, debounce suppression, and final persistent dedupe while preserving the existing media-aware key behavior.

## Reproduction Assessment

Yes, source-level. Feed two Feishu p2p text events for the same account with the same sender, chat/thread, `create_time`, and normalized content but different `message_id` values; the current key resolver returns different raw IDs and all downstream gates treat them as different messages.

## Solution Assessment

Yes. The maintainable fix is Feishu-local key normalization in or near `resolveFeishuMessageDedupeKey` plus regression coverage; no core dedupe API, user-facing config option, or broad recovery module is needed for this bug.
