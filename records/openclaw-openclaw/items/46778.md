## Summary

Current main still deduplicates ordinary Feishu text deliveries by `message_id` alone, so equivalent retries with different IDs can both reach dispatch. The related broad PR remains open and conflict-blocked, and its unrelated scope makes it unsuitable as the canonical repair; this issue should stay open for a focused plugin-owned fix.

## What This Changes

Add replay-aware Feishu inbound deduplication for equivalent private-message deliveries that arrive with different message IDs.

## Best Possible Solution

Use stable Feishu retry metadata at the plugin ingress to derive a bounded replay identity, preserve the existing media-specific and persistent dedupe paths, and test both changed-ID retries and legitimate repeated messages.

## Reproduction Assessment

Yes at source level: two ordinary Feishu text events with equivalent payloads but different `message_id` values resolve to different keys and can both pass the current processing claim. A captured live retry payload is still useful for choosing the correct stable transport field.

## Solution Assessment

No, content hashing alone is not the best fix because it can collapse intentional repeated messages. A retry-specific identity based on verified Feishu event/create-time semantics is narrower and safer.
