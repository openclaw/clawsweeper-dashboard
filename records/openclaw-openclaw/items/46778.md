## Summary

Keep open. Current main still keys non-media Feishu text dedupe on the raw `message_id`, and the receive, debounce, and final persistent dedupe gates all consume that key, so the reported same-logical private-chat text deliveries with different IDs can still be processed twice.

## What This Changes

The issue asks Feishu private-chat duplicate deliveries with different `message_id` values to be deduped by a stable logical text fingerprint instead of only by `message_id`.

## Best Possible Solution

Add a Feishu-local logical text retry fingerprint used consistently by receive, debounce, and final persistent dedupe, while preserving the current media-aware key behavior.

## Reproduction Assessment

Yes, source-level. A focused check can feed two Feishu p2p text events with the same account, sender, chat/thread, `create_time`, and content but different `message_id` values; current main resolves them to different dedupe keys.

## Solution Assessment

Yes. The narrowest maintainable fix is a Feishu-local logical text retry fingerprint based on stable event fields, not a bare content hash and not a core dedupe API change.
