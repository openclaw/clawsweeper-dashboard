## Summary

Keep open. Current main has media-aware Feishu dedupe, but text/private-chat events still fall back to raw `message_id`, so the reported same-logical-message-with-different-id case can still pass the receive, debounce, and final persistent dedupe gates; related media-id work does not cover this text retry case.

## What This Changes

The issue requests Feishu private-chat duplicate suppression when the same logical text message arrives twice with different `message_id` values.

## Best Possible Solution

Add a Feishu-local text retry fingerprint used consistently by receive, debounce, and final persistent dedupe while preserving the existing media-aware keys.

## Reproduction Assessment

Yes. A focused unit reproduction can feed two Feishu p2p text events with the same account, sender, chat/thread, `create_time`, and content but different `message_id`; source inspection shows current main resolves those to different dedupe keys.

## Solution Assessment

Partly. The right boundary is Feishu-local logical duplicate detection, but a bare content hash alone is too broad; the safer fix should combine stable event identity fields and keep media/file uniqueness intact.
