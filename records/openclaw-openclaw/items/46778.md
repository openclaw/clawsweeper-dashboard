## Summary

Keep open. Current main still resolves non-media Feishu text messages to the raw `message_id`, and the receive, debounce, and final persistent dedupe gates all consume that key, so same-logical private-chat text deliveries with different IDs can still be processed twice; the related closed reconnect and media reused-ID items cover different cases.

## What This Changes

The issue reports Feishu WebSocket DMs processing the same logical text message twice when Feishu delivers it with different `message_id` values, and asks for fallback dedupe beyond message ID.

## Best Possible Solution

Add a Feishu-local logical text retry fingerprint used consistently by receive, debounce, and final persistent dedupe, while preserving the current media-aware key behavior.

## Reproduction Assessment

Yes, source-level. A focused check can feed two Feishu p2p text events with the same account, sender, chat/thread, `create_time`, and content but different `message_id` values; current main resolves them to different dedupe keys.

## Solution Assessment

Yes. The narrowest maintainable fix is a Feishu-local logical text retry fingerprint, not a bare content hash, using stable event fields while keeping media/file uniqueness intact.
