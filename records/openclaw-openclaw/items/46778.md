## Summary

Keep open. Current main and the latest release still dedupe plain Feishu text events by the raw `message_id`, so the reported same-logical-message/different-ID path remains source-reproducible; the closed related Feishu issues cover reconnect persistence or the opposite media reused-ID case, not this remaining text retry gap.

## What This Changes

The issue reports duplicate Feishu private-message processing when Feishu sends the same logical text message with different `message_id` values, bypassing message_id-only dedupe.

## Best Possible Solution

Add Feishu-local logical text retry fingerprinting from stable sender, chat/thread, `create_time`, and normalized text content, while preserving the existing media-aware key behavior and avoiding a new core API or config option.

## Reproduction Assessment

Yes, source-level. Feed two Feishu p2p text events for the same account with the same sender, chat/thread, `create_time`, and normalized content but different `message_id` values; current key resolution and downstream gates treat them as different messages.

## Solution Assessment

Yes. The narrow maintainable fix is Feishu-local dedupe-key normalization with tests; a broad recovery module, new config flag, or core dedupe API is not needed for this bug.
