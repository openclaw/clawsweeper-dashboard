## Summary

Keep open. The report describes a docs-defined persistence guarantee, but current main source already routes cron tool and Gateway updates through `CronService.update()` and `persist()`, so the remaining question is a current-main reproduction with the exact `cron.status.storePath` and on-disk files rather than a cleanup close.

## What This Changes

The issue reports that cron tool add/update changes appear in the running Gateway but are not written to `cron/jobs.json`, so restarting the Gateway restores the old job definition.

## Best Possible Solution

Add focused current-main proof for Gateway cron add/update persistence against the configured `cron.store`; if that proof fails, fix the cron service store write path with regression coverage, otherwise document the store-path diagnostic needed for this failure mode.

## Reproduction Assessment

No. The issue has clear user steps, but source inspection on current main shows the cron tool and Gateway update path should call `persist()`, and I did not run a live Gateway/file restart repro in this read-only review.

## Solution Assessment

Unclear. A persistence fix is only the best solution if current-main live proof shows `jobs.json` is unchanged after Gateway `cron.add` or `cron.update`; otherwise the safer path is diagnostics around the configured store path and sidecar files.
