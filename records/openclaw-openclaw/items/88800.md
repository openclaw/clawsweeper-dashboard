## Label Justifications

- `P2`: This fixes a repeatable security-audit false positive and generated auth-state ambiguity without exposing actual secrets or breaking the core runtime.
- `merge-risk: 🚨 compatibility`: The branch changes the persisted generated API-key marker format and adds special handling for files written by older releases.
- `merge-risk: 🚨 auth-provider`: Model-registry request auth now interprets prefixed values as env SecretRefs, so incorrect parsing would prevent provider credentials from resolving.

## Summary

This PR remains necessary: current main still persists ambiguous bare env-reference names, while the branch consistently writes an explicit SecretRef env marker, resolves it at request time, and prevents the resulting false-positive audit finding without broadly exempting unrelated strings.

## What This Changes

The branch changes generated provider API keys from bare env-variable names to `secretref-env:` markers, resolves those markers in model-registry requests, and updates the secrets audit, tests, and docs for new and stale generated files.

## Best Possible Solution

Land one canonical `secretref-env:<ID>` generated-state representation, retain the exact-match stale-file audit bridge only for already-written files, and keep runtime resolution centralized through the existing SecretRef parser.

## Reproduction Assessment

Yes. Current-main source deterministically writes a bare env-reference name into generated provider auth, and the current audit classifies an unrecognized non-marker string there as plaintext; the linked issue also supplies a concrete released-version reproduction.

## Solution Assessment

Yes. A canonical explicit marker reuses the existing SecretRef parser, avoids persisting the resolved secret, preserves request-time env lookup, and is narrower than adopting structured SecretRef objects in a schema that currently requires string API keys.
