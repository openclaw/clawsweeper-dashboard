## What This Changes

The PR stores env-backed provider API keys in generated `models.json` as `secretref-env:<NAME>`, resolves those markers at request time, and narrows audit compatibility for matching stale bare env-name markers.

## Best Possible Solution

Keep one canonical env-reference marker and merge only after redacted real-path proof covers fresh generation, matching stale-file audit compatibility, unresolved-env failure, and successful provider request resolution.

## Reproduction Assessment

Yes, from source: current main writes a bare env name for an env-backed profile SecretRef, and deep audit classifies that non-marker string as plaintext. The supplied regression tests cover the path, but no real CLI reproduction was run after the patch.

## Solution Assessment

Yes, provisionally: reusing the existing `secretref-env:` parser is the narrowest way to align generation, audit, and request-time resolution, and the stale exception is provider- and ref-specific. Real upgrade-path proof is still required before merge.
