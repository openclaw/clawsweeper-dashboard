## Summary

Current main already fixes the buildx attestation verifier failure, the fix shipped in v2026.5.7, and the PR branch is now both conflicting and weaker than the shipped implementation because it would accept explicit wrong artifactType values when layers are in-toto.

## What This Changes

The PR changes the Docker attestation verifier to accept current buildx attestation manifests without the legacy Docker artifactType and adds fixtures for the new and malformed manifest shapes.

## Best Possible Solution

Keep the shipped v2026.5.7 verifier behavior, which accepts omitted artifactType while preserving rejection of present-but-wrong artifactType; track any stricter layer mediaType hardening separately if maintainers want it.

## Reproduction Assessment

No for current main: the cited GHCR image verifies successfully with the current script for linux/amd64 and linux/arm64, and source plus tests cover omitted artifactType. The original failure would reproduce only on the pre-fix verifier.

## Solution Assessment

Yes for current main, not for this PR branch. The shipped fix is the narrowest maintainable solution because it accepts omitted artifactType while preserving rejection of explicit wrong artifactType values.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve explicit artifactType rejection:** `scripts/verify-docker-attestations.mjs:101`
  - body: The new OR condition accepts a manifest that sets artifactType to an unexpected value as long as its layers use application/vnd.in-toto+json, so a malformed attestation that used to fail would pass. Current main's shipped fix accepts omitted artifactType but still rejects present-but-wrong values.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The current main fix is safe, but the PR diff would weaken supply-chain verifier validation for explicit wrong artifactType values if merged.

Concerns:

- **[medium] Verifier would accept explicit wrong artifactType:** `scripts/verify-docker-attestations.mjs:101`
  - body: Because the PR accepts either a matching artifactType or all in-toto layers, a manifest with an unexpected artifactType and in-toto layers would pass the verifier. Current main avoids that by allowing omission but rejecting present-but-wrong values.
  - confidence: 0.86
