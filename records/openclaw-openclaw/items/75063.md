## Summary

Current main and v2026.5.7 already fix the Docker buildx attestation verifier failure by accepting omitted artifactType while still rejecting explicit bad artifactType values; the PR branch is now conflicting and less strict than the shipped verifier.

## What This Changes

The PR changes the Docker attestation verifier and tests to accept buildx attestation manifests without artifactType when their layers are in-toto payloads.

## Best Possible Solution

Keep the shipped v2026.5.7 verifier behavior; open a separate narrow hardening issue only if maintainers want additional layer mediaType validation.

## Reproduction Assessment

No for current main: the cited GHCR image omits artifactType, and a current-main verifier probe returned no errors for the required linux/amd64 and linux/arm64 attestations. The old failure is still understandable from the PR body and historical fix, but it no longer reproduces on current main.

## Solution Assessment

Yes for the shipped implementation, not for this PR branch. Current main fixes the buildx output at the verifier boundary while preserving rejection of present-but-wrong artifactType values.
