## Summary

Keep open: the PR addresses a real misleading CLI diagnostic, but the proposed resolver can claim a tool is registered when it is only manifest-declared and not actually available for the current config.

## What This Changes

The PR adds a CLI unknown-subcommand diagnostic that maps a matching plugin-declared agent tool name to its manifest owner, wires it into `run-main`, adds tests, and updates the changelog.

## Best Possible Solution

Land the diagnostic only after it uses the same effective tool availability rules as agent tool discovery, while preserving the existing disabled-plugin and `plugins.allow` fallthrough messages.

## Reproduction Assessment

Yes, from source inspection. Current main falls through to the `plugins.allow` suggestion for the reported unknown tool-name shape, and the PR’s new failure mode is source-reproducible because it uses manifest ownership rather than active availability.

## Solution Assessment

No. The direction is right, but the maintainable fix should reuse or extract the active/effective plugin-tool availability path instead of treating `contracts.tools` as a registered runtime tool list.

## Review Findings
