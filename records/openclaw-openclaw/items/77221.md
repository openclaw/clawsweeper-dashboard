## Summary

Keep open: the PR targets a real CLI diagnostic bug, but the current patch should not merge because it treats manifest-declared tool ownership as proof that the tool is registered and available in the current runtime.

## What This Changes

The PR adds a tool-owner lookup to the unknown-subcommand CLI policy, wires it through runtime manifest metadata, adds tests, and records a changelog fix for tool names being mistaken for plugin ids.

## Best Possible Solution

Keep the diagnostic improvement, but make the tool-name branch depend on the same current-config availability semantics as runtime tool dispatch; otherwise preserve the existing plugin allow/disabled guidance.

## Reproduction Assessment

Yes, source-level: current main falls through to the `plugins.allow` message for an unknown root command under a restrictive allowlist, and the PR branch can be read to return the new tool-use message from `contracts.tools` before checking whether the owning plugin/tool is actually available. I did not run tests because this sweep is read-only.

## Solution Assessment

No. The best fix is narrower than the PR as written: resolve only loaded or currently available tool registrations, not every tool name declared in manifest metadata.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Gate tool-name messages on actual availability:** `src/cli/run-main-policy.ts:176-182`

Do not treat `contracts.tools` alone as registration.
