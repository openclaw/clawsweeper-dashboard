## Summary

Close as duplicate/superseded: #77220 is already open against the same #77214 CLI diagnostic bug and touches the same policy/runtime/changelog surfaces, so this PR should not remain as a competing fix branch.

## What This Changes

This PR adds manifest-declared tool-owner lookup to unknown CLI command diagnostics, wires the resolver through `run-main`, adds focused tests, and documents the change in the changelog.

## Best Possible Solution

Keep one corrected PR for #77214, with tool-owner diagnostics filtered through effective plugin/tool availability and preserving plugin-enable guidance when the owner is disabled or excluded.

## Reproduction Assessment

Yes. Source inspection shows current main falls through to the `plugins.allow` message for an unknown root token under a restrictive allowlist, and both competing PRs exercise that path with a manifest-declared tool name.

## Solution Assessment

No, not as-is. The direction is right, but this branch duplicates #77220 and its new resolver treats manifest-declared ownership as runtime registration without checking whether the owning plugin/tool is actually available.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Check tool availability before returning model-tool guidance:** `src/cli/run-main-policy.ts:188-193`
  - body: The new branch returns the agent-tool message for any `contracts.tools` match, but the production resolver reads a manifest registry that can include disabled or unavailable plugins. A user invoking a tool name from a plugin excluded by `plugins.allow`, disabled by `plugins.entries.<id>.enabled=false`, or gated by tool config would be told to use model tool-use even though the tool cannot be registered. Filter to an effective/available owner, or include owner enablement guidance, before returning this message.
  - confidence: 0.88
