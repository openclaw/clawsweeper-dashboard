## Summary

Keep open: current main still has the reported misleading CLI diagnostic, but this PR is not merge-ready because it treats manifest-declared tool ownership as active tool registration and can mislead users for disabled or config-gated tools.

## What This Changes

The PR adds a CLI unknown-subcommand branch that looks up plugin `contracts.tools`, returns an agent-tool-not-CLI message, adds tests, and updates the changelog.

## Best Possible Solution

Land the diagnostic after it uses the same active/effective plugin-tool semantics as agent tool discovery, preserving the existing enablement messages for excluded, disabled, optional, denied, or auth-unavailable tools.

## Reproduction Assessment

Yes, source-reproducible. Current main falls through to the `plugins.allow` suggestion for an unknown tool-like primary command, and the PR’s new wrong case is source-reproducible because it consults manifest metadata instead of active tool availability.

## Solution Assessment

No. The direction is right, but the maintainable fix should query effective tool availability or reuse the existing plugin tool discovery filters instead of treating `contracts.tools` ownership as registration.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Check active tool availability before claiming registration:** `src/plugins/manifest-command-aliases.runtime.ts:38-42`
  - body: `resolveManifestToolOwner` loads the manifest metadata registry and the new policy branch treats any `contracts.tools` match as a registered agent tool. That registry includes disabled plugins, and runtime tool discovery also filters optional, denied, and config-gated tools, so `openclaw <tool>` can now tell users to use model tool-use when the real fix is enabling/configuring the plugin or when the tool is unavailable. Please base this branch on effective tool availability instead of raw manifest ownership.

## Repair Work Prompt

On PR #77221, keep the CLI diagnostic goal from #77214 but replace the manifest-only `contracts.tools` owner check with active/effective plugin-tool availability semantics. The agent-tool-not-CLI message should fire only when the requested name is actually available to model tool-use for an enabled/loaded plugin under the current config. Preserve existing `plugins.allow` and `plugins.entries.<id>.enabled=false` fallthroughs for excluded or disabled plugins, and preserve unknown-name behavior. Reuse existing plugin tool discovery or availability helpers where possible; do not broaden plugin loading, change protected config behavior, or add a new config option. Add regression tests for an available tool, a disabled/excluded plugin tool, a metadata-gated tool without config/auth, and optional/denied tool policy. Keep the changelog entry aligned with repo policy.
