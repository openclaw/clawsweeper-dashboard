## Label Justifications

- `P1`: The issue reports current released cron behavior that can unexpectedly fire billable agent turns outside schedule guards.
- `impact:other`: The impact is unsafe automation execution and operator-visible cron behavior rather than message loss, auth, crash-loop, or session-state drift.

## Summary

Keep open: current main and v2026.6.8 still default omitted cron `runMode`/`mode` to `force`, so the schedule-guard bypass is source-reproducible; the fix needs maintainer judgment because docs and tests currently treat bare manual cron runs as force runs.

## What This Changes

Requests changing the cron MCP/agent `action=run` path so omitted `runMode` respects due scheduling, with follow-up observability for manual versus scheduled run records.

## Best Possible Solution

Decouple the agent/MCP safe default from explicit force behavior: make omitted agent `runMode` schedule-respecting or require explicit `force`, update tests/docs for that boundary, and handle trigger observability as a narrow run-history follow-up if accepted.

## Reproduction Assessment

Yes, from source inspection rather than a live run: omitted `runMode` becomes `mode: "force"`, and force-mode preflight skips enabled and schedule due checks. I did not execute a gateway reproduction because this review was read-only.

## Solution Assessment

Unclear until maintainers confirm the default semantics. A narrow agent/MCP default change is safer than changing the gateway/CLI bare `cron.run` contract without explicit compatibility discussion.
