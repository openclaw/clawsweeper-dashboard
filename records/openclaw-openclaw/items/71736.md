## Summary

Keep open; current main has a narrow data-only Control UI descriptor foundation and session-action plumbing, but not the broader contribution-slot vocabulary or browser rendering contract requested here. Recent discussion and related open follow-up work also treat this as unfinished design work rather than a cleanup candidate.

## What This Changes

Requests a declarative SDK/Gateway/Control UI contribution surface for plugin chat modes, approval cards, event classifiers, input guards, status surfaces, and safe host-owned rendering.

## Best Possible Solution

Keep this as the broad design tracker until maintainers approve a versioned, data-only contribution model with safe host-owned rendering, Gateway projection, action authorization, disablement semantics, and fixture/client coverage.

## Reproduction Assessment

Not applicable; this is an RFC/feature request, not a bug report. Source inspection against current main gives high confidence that the requested broader slot vocabulary is absent.

## Solution Assessment

Unclear; the descriptor-first, host-owned direction is plausible, but the exact SDK schema, Gateway projection, browser rendering, action model, and security boundary need maintainer design approval.

## Security Review

Status: needs_attention

Summary: The requested browser-facing plugin UI/action projection needs explicit security design before implementation.

Concerns:

- **[medium] Define the Control UI plugin trust boundary:**
  - body: Contribution slots could affect operator-visible rendering, composer enablement, session actions, status surfaces, and possibly plugin-owned browser content.
