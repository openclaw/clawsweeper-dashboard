## Summary

Keep open. Current main has the generic host-hook foundation and generic plugin approval card rendering, but it still does not expose the broader Control UI contribution-slot model requested here. The linked prototypes and follow-up PRs are useful adjacent design input, not a complete implementation of chat modes, input guards, event classifiers, status surfaces, native/panel vocabulary, or browser-side rendering semantics.

## What This Changes

Request a data-driven Control UI plugin contribution API for plugin chat modes, approval cards, event classifiers, input guards, and status surfaces.

## Best Possible Solution

Keep this as the broad Control UI contribution-slot design tracker until maintainers approve an additive, versioned, data-only descriptor model with gateway projection, host-owned browser components, plugin disablement semantics, action authorization, and fixture coverage for enabled/disabled plugins and plugin-originated UI actions.

## Reproduction Assessment

Not applicable; this is an RFC/feature request rather than a bug. The high-confidence verification path is source, docs, protocol, UI search, history, and related discussion review against current main, which shows only the generic descriptor foundation.

## Solution Assessment

Unclear. The requested direction is plausible, but the best first implementation should be a maintainer-approved manifest-first or tightly versioned runtime descriptor vocabulary with host-owned rendering, not arbitrary plugin browser code or a broad mutable UI registry.
