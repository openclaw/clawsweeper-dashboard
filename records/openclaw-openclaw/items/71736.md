## Summary

Keep open. Current main still has only the narrow Control UI descriptor foundation, while the requested chat modes, approval families, event classifiers, input guards, status surfaces, panels/native pages, and browser rendering contract remain design work; the active PRs cover useful slices but do not close the broader RFC.

## What This Changes

RFC asks for data-driven Control UI contribution slots for plugin chat modes, approval cards, event classifiers, input guards, and status surfaces.

## Best Possible Solution

Keep this as the broad design tracker until maintainers approve or deliberately split a versioned, data-only contribution model with gateway projection, host-owned rendering, authorized plugin actions, disablement semantics, browser security rules, and fixture coverage.

## Reproduction Assessment

Not applicable; this is an RFC/feature request rather than a bug report. The high-confidence verification path is source, docs, protocol schema, UI search, and related PR discussion showing only a narrow descriptor foundation on current main.

## Solution Assessment

Unclear. The descriptor-first, host-owned direction is plausible, but the maintainable solution needs maintainer-approved versioning, UI slot vocabulary, action authorization, and browser security boundaries before implementation.
