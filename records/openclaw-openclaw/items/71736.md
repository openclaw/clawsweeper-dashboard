## Summary

Keep open; current main has a narrow generic Control UI descriptor foundation, but not the broader contribution-slot model requested here. The live discussion also says the recent host-hook follow-ups and the `controlUiPanels` prototype are useful adjacent work, not a closure path for chat modes, approval families, event classifiers, input guards, status surfaces, panels/native pages, and browser rendering contracts.

## What This Changes

Request a data-driven SDK and Control UI contribution model for plugins to expose chat modes, approval cards, event classifiers, input guards, and status surfaces.

## Best Possible Solution

Keep this as the broad design tracker until maintainers approve an additive, versioned, data-only contribution model with gateway projection, host-owned rendering, authorized plugin actions, disablement semantics, and fixture coverage.

## Reproduction Assessment

Not applicable; this is an RFC/feature request rather than a bug report. The high-confidence verification path is source, docs, protocol schema, UI search, and related discussion showing only the generic descriptor foundation on current main.

## Solution Assessment

Unclear. The descriptor-first, host-owned direction is plausible, but the maintainable solution needs a maintainer-approved versioned contract rather than arbitrary plugin browser JavaScript or a broad mutable UI registry.
