## Summary

Keep open. Current main has a narrow generic Control UI descriptor foundation, but it does not implement the broader contribution-slot system requested here, and the live discussion plus related open PRs describe partial foundations rather than a completed design.

## What This Changes

Request data-driven Control UI plugin contribution slots for chat modes, approval cards, event classifiers, input guards, status/status-panel surfaces, and host-owned safe rendering.

## Best Possible Solution

Keep this issue as the broad Control UI contribution-slot design tracker until maintainers choose an additive, versioned, data-only descriptor model with typed schemas, gateway projection, host-owned browser components, plugin disablement semantics, and fixture coverage for enabled/disabled plugins plus plugin-originated actions.

## Reproduction Assessment

Not applicable; this is an RFC/feature request rather than a bug. The verification path is source, docs, protocol, UI, issue discussion, and related PR review against current main, which shows only the generic descriptor foundation.

## Solution Assessment

Unclear. The requested direction is plausible, but the maintainable solution should be a maintainer-approved manifest-first or tightly scoped runtime projection model with host-owned rendering, not arbitrary browser plugin code or an unbounded mutable UI registry.
