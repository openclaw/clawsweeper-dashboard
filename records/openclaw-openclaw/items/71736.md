## Summary

Keep open. Current main has a narrow generic Control UI descriptor projection, but it does not implement the requested contribution-slot vocabulary, browser-side consumers, manifest/runtime design, or security contract for plugin chat modes, input guards, event classifiers, status surfaces, panels/native pages, and richer approval renderers. The issue discussion also points to active partial follow-up PRs and an external `controlUiPanels` prototype as adjacent work rather than a closeout.

## What This Changes

Request a first-class, data-driven Control UI plugin contribution API for chat modes, approval cards, event classifiers, input guards, status surfaces, and related host-rendered UI descriptors.

## Best Possible Solution

Keep this as the broad Control UI contribution-slot design thread until maintainers choose an additive, versioned descriptor model. The eventual implementation should define typed schemas, gateway projection, Control UI consumers, disablement semantics, sanitization, plugin-originated actions, and fixture coverage for enabled/disabled plugins.

## Reproduction Assessment

Not applicable; this is an RFC/feature request rather than a bug report. The verification path is source, docs, tests, discussion, and current-main inspection, which show partial generic descriptors but no full contribution-slot implementation.

## Solution Assessment

Unclear. The requested direction is plausible, but the safest maintainable path needs maintainer API/product/security judgment, likely starting with manifest-first data descriptors and a host-owned component catalog rather than arbitrary plugin browser code.
