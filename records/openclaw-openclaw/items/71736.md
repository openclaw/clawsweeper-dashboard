## Summary

Keep open; current main still has only a narrow generic Control UI descriptor foundation, while the requested chat-mode, input-guard, event-classifier, status-surface, panel/native-page, and broader approval-slot model remains unimplemented and explicitly called out as partial in the discussion.

## What This Changes

Requests a first-class, data-driven Control UI plugin contribution API for chat modes, approval cards, event classifiers, input guards, and status surfaces.

## Best Possible Solution

Keep this as the broad RFC until maintainers approve an additive, versioned, data-only contribution model with gateway projection, host-owned rendering, authorized plugin actions, disablement semantics, and fixture coverage for enabled and disabled plugins.

## Reproduction Assessment

Not applicable; this is an RFC/feature request rather than a bug. The high-confidence verification path is current source, docs, protocol schema, UI search, and issue discussion showing only the generic descriptor foundation on main.

## Solution Assessment

Unclear. The direction is plausible, but the maintainable solution should be a maintainer-approved host-rendered contract, with any native/plugin-owned page work split behind explicit security and API decisions.

## Security Review

Status: needs_attention

Summary: Browser-facing plugin UI projection and plugin-originated UI actions need explicit trust, sanitization, and authorization boundaries.

- **[medium] Define the plugin UI and action trust boundary:**
