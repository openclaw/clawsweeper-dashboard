## Summary

Keep open. Current main has the generic host-hook and Control UI descriptor foundation, but it is still limited to generic session/tool/run/settings descriptors and does not implement the broader contribution-slot system requested here. The recent discussion and the linked controlUiPanels prototype are useful design input, but they are partial follow-up work rather than evidence that the RFC is complete.

## What This Changes

The issue requests first-class, data-driven Control UI contribution slots for plugin chat modes, approval cards, event classifiers, input guards, status surfaces, panels/native pages, and safe host-owned rendering.

## Best Possible Solution

Keep this as the broad Control UI contribution-slot design tracker until maintainers approve an additive, versioned, data-only descriptor model with gateway projection, host-owned browser components, plugin disablement semantics, action authorization, and fixture coverage for enabled/disabled plugins and plugin-originated UI actions.

## Reproduction Assessment

Not applicable; this is an RFC/feature request rather than a bug. The verification path is source, docs, protocol, UI search, history, and related discussion review against current main, which shows only the generic descriptor foundation.

## Solution Assessment

Unclear. The requested direction is plausible, but the best first implementation should be a maintainer-approved manifest-first or tightly versioned runtime descriptor vocabulary with host-owned rendering, not arbitrary plugin browser code or a broad mutable UI registry.
