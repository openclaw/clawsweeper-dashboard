## Summary

Keep open. Current main has a useful generic Control UI descriptor foundation, but it is still limited to four generic descriptor surfaces and does not implement the broader contribution-slot vocabulary requested here; the issue comments also clarify that the linked PRs and prototype work are adjacent, partial design inputs rather than a complete solution.

## What This Changes

Request a first-class, data-driven Control UI plugin contribution-slot SDK/protocol/UI surface for chat modes, approval cards, event classifiers, input guards, status surfaces, and plugin-owned actions.

## Best Possible Solution

Keep this as the broad Control UI contribution-slot design tracker until maintainers approve an additive, versioned, data-only descriptor model with gateway projection, host-owned rendering, disablement semantics, action authorization, and fixture coverage for enabled/disabled plugins and plugin-originated UI actions.

## Reproduction Assessment

Not applicable; this is an RFC/feature request rather than a bug. The high-confidence verification path is source, docs, protocol, UI search, related discussion, and current-main inspection, which shows only the generic descriptor foundation.

## Solution Assessment

Unclear. The requested direction is plausible, but the best first implementation needs maintainer-approved scope and should preserve a data-only, host-rendered browser boundary rather than exposing arbitrary plugin JavaScript or a broad mutable UI registry.
