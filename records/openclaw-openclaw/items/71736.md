## Summary

Keep open. Current main has the merged generic host-hook/UI descriptor foundation, but it still lacks the broader Control UI contribution-slot system requested here: chat-mode descriptors, input guards, event classifiers, status/panel/native-page slots, approval-card families, and a browser rendering contract. The latest issue discussion also says the active/adjacent work is partial rather than a closeout.

## What This Changes

The issue requests a data-driven plugin SDK and Control UI contribution-slot surface for plugin chat modes, approval cards, event classifiers, input guards, status surfaces, and related safe host-rendered UI descriptors.

## Best Possible Solution

Keep this as the broad Control UI contribution-slot design tracker until maintainers choose an additive, versioned, data-only descriptor model with typed schemas, gateway projection, host-owned browser components, plugin disablement behavior, and fixture coverage for enabled/disabled plugins and plugin-originated actions.

## Reproduction Assessment

Not applicable; this is an RFC/feature request rather than a bug. The verification path is source, docs, protocol, UI search, related PR state, and issue discussion review against current main, which shows only the generic descriptor foundation.

## Solution Assessment

Unclear. The direction is plausible, but the maintainable first implementation should be a maintainer-approved, manifest-first and data-only slot vocabulary rather than arbitrary browser plugin code or a broad mutable UI registry.
