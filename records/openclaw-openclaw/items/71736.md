## Summary

Keep open. Current main has the generic host-hook and Control UI descriptor foundation, but it still lacks the broader contribution-slot model requested here: plugin chat modes, input guards, event classifiers, status/panel/native-page slots, approval-card families, and browser-side rendering semantics. The latest issue discussion also treats #73384, #74483, #74853, #66944, and the `controlUiPanels` fork prototype as adjacent or partial rather than a closeout.

## What This Changes

Request a data-driven Control UI plugin contribution-slot SDK/API for chat modes, approval cards, event classifiers, input guards, status surfaces, and related safe host-owned UI descriptors.

## Best Possible Solution

Keep this as the broad Control UI contribution-slot design tracker until maintainers approve an additive, versioned, data-only descriptor model with typed schemas, gateway projection, host-owned browser components, plugin disablement behavior, and fixture coverage for enabled/disabled plugins and plugin-originated actions.

## Reproduction Assessment

Not applicable; this is an RFC/feature request rather than a bug. The verification path is source, docs, protocol, UI search, tests, and current issue discussion, which show only the generic descriptor foundation on current main.

## Solution Assessment

Unclear. The requested direction is plausible, but the best implementation needs maintainer-approved manifest/runtime boundaries, a browser rendering/security model, and a scoped initial slot vocabulary rather than a broad mutable UI registry or arbitrary plugin browser code.
