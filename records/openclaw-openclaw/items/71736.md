## Summary

Keep open; current main has the generic host-hook/UI descriptor foundation, but it still does not provide the broader Control UI contribution-slot model requested for chat modes, input guards, event classifiers, status surfaces, native panels/pages, or host-rendered approval families.

## What This Changes

The issue requests an additive SDK, gateway, and Control UI contract for data-driven plugin contribution slots across chat modes, approvals, event classification, composer guards, and status UI.

## Best Possible Solution

Keep this as the broad design tracker until maintainers approve an additive, versioned, data-only contribution model with gateway projection, host-owned rendering, action authorization, disablement semantics, and fixture coverage for enabled and disabled plugins.

## Reproduction Assessment

Not applicable; this is an RFC/feature request rather than a bug. The high-confidence verification path is source, docs, tests, protocol, UI search, history, and the related discussion, which show only the generic descriptor foundation on current main.

## Solution Assessment

Unclear. The requested direction is plausible, but the best implementation needs maintainer-approved scope and should preserve a data-only, host-rendered browser boundary rather than introducing arbitrary plugin JavaScript or a broad mutable UI registry.
