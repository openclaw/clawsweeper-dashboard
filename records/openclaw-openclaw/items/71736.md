## Summary

Keep open: current main and v2026.6.10 still provide only a narrow plugin Control UI descriptor/session-action surface, while the requested broader contribution-slot model remains an SDK, Gateway, Control UI, and browser-security design decision.

## What This Changes

Request for a declarative plugin Control UI contribution-slot API covering chat modes, approval/card rendering, event classifiers, input guards, status surfaces, and safe host-owned browser rendering.

## Best Possible Solution

Keep this as the maintainer-owned broad RFC and define a versioned, data-only or explicitly sandboxed Control UI contribution contract before merging implementation slices.

## Reproduction Assessment

Not applicable: this is an RFC for new SDK and Control UI contribution surfaces, not a bug report. Source and release inspection give high confidence that the requested broader slot model is absent from current main and v2026.6.10.

## Solution Assessment

Unclear until maintainers choose the API and trust model. The host-owned descriptor direction fits the repository, but the final manifest/runtime split, component catalog, and browser isolation rules need product and security decisions.
