## Best Possible Solution

Use a generic manifest activation/workspace-trust resolver for the configured memory slot, retain plugin-owned prompt guidance, and add an integration-style test against the real bundled `memory-core` manifest so the scoped Codex and Copilot loads preserve only an eligible selected memory plugin.

## Reproduction Assessment

Yes. Current main performs the late scoped registry load, and the supplied before-fix live trajectory shows `memory_search` and `memory_get` available while the registered Memory Recall section is absent after a fresh Telegram session.

## Solution Assessment

No, not at the current head. Preserving the selected memory capability and consuming its registered prompt is the correct ownership boundary, but a provider-owner resolver cannot select the real memory plugin; a generic trusted manifest resolver is the narrower working fix.
