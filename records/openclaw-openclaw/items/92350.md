## Summary

This PR addresses a real Codex memory-guidance regression, but the production code uses a provider-only resolver to select the configured memory plugin. Since `memory-core` declares no provider surface, the resolver returns no memory plugin and the scoped harness load can still discard the registration the patch intends to preserve.

## What This Changes

The PR preserves the configured memory plugin during scoped harness registry loads and adds registered memory-recall plus deferred-tool guidance to Codex workspace bootstrap instructions.

## Best Possible Solution

Resolve the configured memory slot through a generic manifest activation and workspace-trust path, then prove with non-mocked coverage that the real bundled `memory-core` plugin remains registered during scoped Codex and Copilot harness loads.

## Reproduction Assessment

Yes. Current main performs a later harness/provider-only scoped registry load, and the supplied before-fix trajectory shows memory tools available while the registered recall guidance is absent.

## Solution Assessment

No in its current form. Preserving the configured memory capability is the right ownership boundary, but a provider-only resolver cannot select the bundled memory plugin.
