## Summary

Keep open: current main partially addresses this hot path with sqlite-vec KNN and parallel BM25/vector merging, but the cache-size PRAGMA, FTS-prefilter rerank policy, and JSON TEXT embedding storage remain unresolved design and performance tradeoffs.

## What This Changes

The issue requests three memory-plugin SQLite optimizations: a larger SQLite page cache, FTS-prefiltered vector reranking, and removal or replacement of the JSON TEXT embedding column.

## Best Possible Solution

Keep this as a benchmark-backed memory-core design task, then split any accepted cache, retrieval, or schema migration changes into focused PRs with regression coverage.

## Reproduction Assessment

No, not as a high-confidence performance reproduction: no large-store cold-cache benchmark was run. Source inspection does confirm the current missing cache_size PRAGMA, current parallel hybrid behavior, and active JSON TEXT embedding fallback.

## Solution Assessment

Unclear: the proposed changes are plausible, but the narrowest maintainable solution needs benchmark evidence and maintainer decisions on RAM budget, recall-quality tradeoffs, and schema compatibility.

## Risks / Open Questions

- No benchmark or live large-store reproduction was run against current main, so the reported p95 and cold-cache latency impact remain unverified.
- A fixed 64 MiB page cache can increase per-agent connection memory usage and needs a maintainer-owned default/budget decision.
- FTS-prefiltered vector reranking may improve latency but can change recall semantics compared with the current parallel vector/BM25 weighted merge.
- Dropping or replacing chunks.embedding requires migration and fallback compatibility for installs where sqlite-vec is unavailable or degraded.
