## Summary

Keep open. Current main and the latest release still mount/pre-create the runtime container workspace at `/home/node/.openclaw/workspace` without a `/workspace` alias, the direct fix PR was closed unmerged, and the newer Docker/DooD sandbox issue is adjacent rather than a replacement for this runtime-container ergonomics request.

## What This Changes

The issue requests a `/workspace` symlink or equivalent container configuration so runtime agent containers can use `/workspace` instead of `/home/node/.openclaw/workspace`.

## Best Possible Solution

Choose one supported runtime workspace contract, then either implement `/workspace` consistently across Dockerfile, Compose, setup helpers, Podman/Quadlet, docs, and smoke coverage, or document `/workspace` as sandbox-only and keep runtime containers on `/home/node/.openclaw/workspace`.

## Reproduction Assessment

Yes, at source level. Current main and `v2026.6.6` show Docker, Compose, Docker setup, Podman, and Quadlet paths using `/home/node/.openclaw/workspace` without a `/workspace` alias; I did not run a live container in this read-only review.

## Solution Assessment

Unclear. The requested one-line symlink is plausible prior art, but the best fix needs a maintainer decision on whether `/workspace` is a runtime-container contract, a second bind mount, or sandbox-only documentation.
