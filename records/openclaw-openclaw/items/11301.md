## Summary

Keep open: current main and v2026.6.5 still keep Docker/Podman runtime workspaces at `/home/node/.openclaw/workspace`, while `/workspace` remains a sandbox-facing path; the linked implementation PR is closed unmerged and no merged replacement owns the exact request.

## What This Changes

The issue requests a `/workspace` alias in OpenClaw container environments so documented agent workspace paths also work outside sandboxed task containers.

## Best Possible Solution

Maintainers should decide whether runtime containers should support `/workspace` as a compatibility alias across Docker, Compose, setup helpers, Podman, and docs, or explicitly document `/workspace` as sandbox-only.

## Reproduction Assessment

Yes. Source inspection on current main and v2026.6.5 shows runtime container paths bind and configure `/home/node/.openclaw/workspace`, while `/workspace` is established only by sandbox configuration paths; I did not run a live container because this is a read-only review.

## Solution Assessment

Unclear. A simple image-level symlink is plausible, but the best fix must deliberately cover Docker Compose, generated setup, Podman/Quadlet, docs, and sandbox path semantics rather than changing only one image layer.
