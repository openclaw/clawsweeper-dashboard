## Summary

Keep open. Current main and v2026.6.5 still expose runtime container workspaces at `/home/node/.openclaw/workspace`, while `/workspace` remains a sandbox-container contract; the only related implementation PR was closed unmerged and was too narrow for all first-party container entry points.

## What This Changes

The issue requests a `/workspace` alias inside OpenClaw runtime containers pointing to `/home/node/.openclaw/workspace` so agent workspace paths work without the longer container path.

## Best Possible Solution

Define `/workspace` either as a supported runtime-container alias across Dockerfile, Compose/setup, Podman/Quadlet, docs, and smoke tests, or explicitly document it as sandbox-only and remove runtime-doc ambiguity.

## Reproduction Assessment

Yes. Source inspection shows current main consistently mounts runtime container workspaces at `/home/node/.openclaw/workspace`, contains no `/workspace` alias, and separately reserves `/workspace` for sandbox containers.

## Solution Assessment

No. The issue body's Dockerfile or entrypoint symlink is only a partial solution; the maintainable fix needs a product decision and cross-surface implementation or documentation update.
