## Summary

Current main and the latest release still keep runtime-container workspaces at `/home/node/.openclaw/workspace` while reserving `/workspace` for agent sandboxes; the closed Dockerfile-only PR did not land, so the request remains a valid but product-sensitive ergonomics decision.

## What This Changes

Add a `/workspace` alias in OpenClaw runtime containers for the canonical `/home/node/.openclaw/workspace` directory.

## Best Possible Solution

Keep `/home/node/.openclaw/workspace` as the canonical persisted runtime path, and either add a documented convenience alias consistently across Docker and Podman with rootless smoke coverage or explicitly document `/workspace` as sandbox-only and remove ambiguous agent guidance.

## Reproduction Assessment

Yes at source level: current main and v2026.6.5 mount runtime workspaces only at `/home/node/.openclaw/workspace`, while sandbox code separately uses `/workspace`. No live container was launched because this review is read-only.

## Solution Assessment

No, the proposed Dockerfile-only symlink is not yet the best demonstrated solution because it omits supported Podman and generated configuration paths and does not settle the runtime-versus-sandbox contract.
