## Summary

Keep open. Current main and the latest release still do not provide a `/workspace` alias in the Docker or Podman gateway/CLI runtime containers; the only first-party `/workspace` behavior remains sandbox-workdir behavior, so this is still a small but real container-contract product decision.

## What This Changes

The issue asks OpenClaw to add a `/workspace` alias in container runtimes so workspace files can be reached without using `/home/node/.openclaw/workspace`.

## Best Possible Solution

Choose one supported container workspace contract, then either implement `/workspace` consistently across Dockerfile, Compose, Docker setup, Podman, Quadlet, docs, and smoke coverage, or document `/workspace` as sandbox-only and keep runtime containers on `/home/node/.openclaw/workspace`.

## Reproduction Assessment

Yes, at source level. Current main and v2026.6.6 create and mount only `/home/node/.openclaw/workspace` for Docker/Podman runtime containers, while `/workspace` is documented and implemented as a sandbox container workdir.

## Solution Assessment

Unclear. The requested Dockerfile symlink is plausible but too narrow by itself; the best fix needs a maintainer decision on whether `/workspace` should be a runtime-container contract or remain sandbox-only.
