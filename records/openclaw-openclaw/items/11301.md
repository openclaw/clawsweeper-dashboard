## Summary

Keep open. Current main and v2026.6.5 still expose runtime container workspaces at `/home/node/.openclaw/workspace`, while `/workspace` remains a sandbox-container contract; the related implementation PR was closed unmerged and no canonical replacement is open.

## What This Changes

Request adding a `/workspace -> /home/node/.openclaw/workspace` alias to first-party container configuration so workspace documentation paths also work inside runtime containers.

## Best Possible Solution

Define `/workspace` either as a supported runtime-container alias across Dockerfile, Compose/setup, Podman/Quadlet, docs, and smoke tests, or explicitly document it as sandbox-only and remove runtime-doc ambiguity.

## Reproduction Assessment

Yes. Source inspection shows current main consistently mounts runtime container workspaces at `/home/node/.openclaw/workspace`, contains no runtime `/workspace` alias, and separately reserves `/workspace` for sandbox containers.

## Solution Assessment

No. The issue body's one-line Dockerfile symlink is a plausible partial repair, but the maintainable solution needs a product decision plus cross-surface Docker, Podman, docs, and smoke-test coverage.
