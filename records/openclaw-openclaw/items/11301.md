## Summary

Keep open: current main still does not provide a runtime-container `/workspace` alias, and the only linked implementation attempt was closed unmerged. The remaining question is a maintainer product/security decision about whether outer Docker/Podman containers should expose `/workspace` or whether docs should reserve that path for sandbox containers.

## What This Changes

Request to add a `/workspace -> /home/node/.openclaw/workspace` compatibility path in OpenClaw container configuration.

## Best Possible Solution

Maintain one explicit container path contract: either implement and test `/workspace` consistently across Dockerfile, Compose, Docker setup, Podman, Quadlet, and docs, or document that `/workspace` is sandbox-only and keep runtime containers on `/home/node/.openclaw/workspace`.

## Reproduction Assessment

Yes at source level. Current Docker, Compose, Docker setup, Podman, and Quadlet sources all expose the runtime workspace at `/home/node/.openclaw/workspace` and do not create `/workspace`; I did not run a live container because this review is read-only.

## Solution Assessment

Unclear. Adding an alias is plausible, but the best solution needs a maintainer choice because `/workspace` already has a sandbox meaning and Docker/Podman/runtime docs must stay aligned.
