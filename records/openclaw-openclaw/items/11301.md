## Summary

Current main and v2026.6.5 both retain the runtime-versus-sandbox path split, the only directly linked implementation closed unmerged from inactivity, and no viable canonical replacement owns the request. This remains a low-priority product decision rather than cleanup-close material.

## What This Changes

Add a `/workspace` alias in OpenClaw runtime containers that resolves to `/home/node/.openclaw/workspace`.

## Best Possible Solution

Define one deliberate container path contract: either support `/workspace` across all first-party runtime-container launch surfaces with ownership and smoke coverage, or declare it sandbox-only and remove ambiguous runtime guidance.

## Reproduction Assessment

Yes. Current main and v2026.6.5 consistently mount runtime workspaces at `/home/node/.openclaw/workspace` while `/workspace` is sandbox-specific, so the requested runtime alias is demonstrably absent from source.

## Solution Assessment

No. A one-line Dockerfile symlink is plausible but too narrow; the maintainable solution must explicitly cover Compose, generated setup, Podman/Quadlet, ownership, docs, and the sandbox path distinction.
