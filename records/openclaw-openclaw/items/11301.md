## Summary

Keep open. Current main and v2026.6.10 still expose runtime container workspaces at `/home/node/.openclaw/workspace` without a `/workspace` alias across Docker, Compose, Docker setup, Podman, and Quadlet, while https://github.com/openclaw/openclaw/pull/12764 was closed unmerged and https://github.com/openclaw/openclaw/issues/31331 tracks an adjacent Docker-out-of-Docker sandbox bind-source problem rather than this runtime alias contract.

## What This Changes

Request adding a `/workspace` symlink or equivalent alias for OpenClaw container runtimes so documented workspace paths work without using `/home/node/.openclaw/workspace`.

## Best Possible Solution

Maintainers should choose the runtime workspace contract, then either implement `/workspace` consistently across Dockerfile, Compose, Docker setup, Podman/Quadlet, docs, and smoke coverage, or document `/workspace` as sandbox-only.

## Reproduction Assessment

Yes, at source level. Current main and v2026.6.10 still show Docker, Compose, Docker setup, Podman, and Quadlet runtime workspace paths at `/home/node/.openclaw/workspace` with no `/workspace` alias; I did not run a live container in this read-only review.

## Solution Assessment

Unclear. The one-line symlink proposed in the closed prior PR is plausible prior art, but the best solution needs a maintainer decision on image symlink versus parallel bind mount versus documenting `/workspace` as sandbox-only.
