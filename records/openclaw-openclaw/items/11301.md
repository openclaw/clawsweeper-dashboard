## Summary

Keep open. Current main and v2026.6.5 still mount runtime container workspaces at `/home/node/.openclaw/workspace`, while `/workspace` remains a sandbox-specific path; the linked implementation PR was closed unmerged and only changed the Dockerfile, so it is prior art rather than a fix.

## What This Changes

The issue requests a `/workspace` alias in OpenClaw container environments so documented workspace paths work without using `/home/node/.openclaw/workspace`.

## Best Possible Solution

Define `/workspace` as either a supported runtime-container alias across Dockerfile, Compose/setup, Podman/Quadlet, docs, and smoke tests, or explicitly document it as sandbox-only and remove runtime-documentation ambiguity.

## Reproduction Assessment

Yes. Source inspection shows current main consistently mounts runtime workspaces at `/home/node/.openclaw/workspace`, contains no runtime `/workspace` alias, and separately reserves `/workspace` for sandbox containers; I did not run a live container during this read-only review.

## Solution Assessment

No for the issue body's Dockerfile-only suggestion. The maintainable solution is a deliberate runtime-container contract across Dockerfile, Compose/setup, Podman/Quadlet, docs, and smoke coverage, or a docs decision that `/workspace` is sandbox-only.
