## Summary

Keep open. Current main and the latest release still only expose the runtime container workspace at `/home/node/.openclaw/workspace`, while `/workspace` remains a sandbox workdir contract; the requested runtime alias is still a product/setup decision rather than an already-implemented cleanup.

## What This Changes

The issue requests a `/workspace` alias in OpenClaw container environments so documented workspace paths work without using `/home/node/.openclaw/workspace`.

## Best Possible Solution

Choose one explicit container workspace contract, then either implement and test `/workspace` consistently across Dockerfile, Compose, Docker setup, Podman, Quadlet, and docs, or document that `/workspace` is sandbox-only and keep runtime containers on `/home/node/.openclaw/workspace`.

## Reproduction Assessment

Yes at source level. Current main shows Dockerfile, Compose, Docker setup, Podman launcher, and Quadlet all target `/home/node/.openclaw/workspace` and do not provide a runtime-container `/workspace` alias; I did not run a live Docker container in this read-only review.

## Solution Assessment

Unclear as proposed. A one-line image symlink is a plausible prior attempt, but the best fix needs a maintainer-approved workspace contract that covers Docker, Compose, setup-generated overrides, Podman, Quadlet, docs, and sandbox path semantics together.
