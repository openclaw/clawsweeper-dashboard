## Summary

Keep open. Current main and latest release still use `/home/node/.openclaw/workspace` for Docker, generated Compose, Podman, and Quadlet runtime containers without a `/workspace` alias; the direct symlink PR was closed unmerged, and the open Docker/DooD sandbox issues are adjacent rather than a replacement for this runtime-container ergonomics request.

## What This Changes

The issue requests a `/workspace` runtime-container alias for `/home/node/.openclaw/workspace` so container users and agent instructions do not need the longer home path.

## Best Possible Solution

Choose one supported runtime workspace contract, then either implement `/workspace` consistently across Dockerfile, Compose, Docker setup, Podman/Quadlet, docs, and smoke coverage, or document `/workspace` as sandbox-only and keep runtime containers on `/home/node/.openclaw/workspace`.

## Reproduction Assessment

Yes, at source level. Current main and `v2026.6.8` show Docker, Compose, Docker setup, Podman, and Quadlet paths using `/home/node/.openclaw/workspace` without a `/workspace` runtime-container alias; I did not run a live container in this read-only review.

## Solution Assessment

Unclear. The one-line symlink from the closed PR is plausible prior art, but the best fix needs a maintainer decision on image symlink versus second bind mount versus documenting `/workspace` as sandbox-only.
