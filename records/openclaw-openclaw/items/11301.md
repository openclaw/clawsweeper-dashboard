## Summary

Keep open: current main and the latest release still lack a runtime-container `/workspace` alias, while Docker/Podman runtime paths continue to mount only `/home/node/.openclaw/workspace`. The related implementation PR was closed unmerged, and the remaining decision crosses runtime container behavior and the existing sandbox `/workspace` contract.

## What This Changes

The issue requests a `/workspace` symlink or equivalent container configuration so container agents can use `/workspace` for the persisted OpenClaw workspace.

## Best Possible Solution

Maintainers should choose the runtime-container workspace contract, then either add tested Docker/Podman `/workspace` alias coverage plus docs or explicitly document `/workspace` as sandbox-only.

## Reproduction Assessment

Yes. Source inspection on current main and `v2026.5.12` shows Docker/Podman runtime paths mount only `/home/node/.openclaw/workspace` and do not create `/workspace`, while sandbox code separately reserves `/workspace`.

## Solution Assessment

Unclear. The proposed one-line Dockerfile symlink addresses one image path, but the maintainable solution needs a coordinated Docker, Compose, setup-helper, Podman, Quadlet, docs, and sandbox-contract decision.
