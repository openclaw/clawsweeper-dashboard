## Summary

Keep open: this is a plausible security/config feature, but the PR is not mergeable as-is because it introduces SSRF policy plumbing with policy-agnostic caches, partially enforced remote-media config, a conflicting branch, and no real behavior proof.

## What This Changes

The branch adds opt-in SSRF policy config/schema/runtime plumbing for web_fetch, Gemini citation redirects, remote outbound media, browser policy typing, generated config baselines, and focused tests.

## Best Possible Solution

Land a rebased, security-reviewed version that preserves policy-specific cache identities, consistently wires or narrows remote-media SSRF config, and includes redacted real proof from a configured run.

## Reproduction Assessment

Yes, source-based. At PR head, web_fetch and Gemini search read policy-agnostic cache entries before the new SSRF policy affects the network path, and the branch accepts global remoteMedia policy while Slack/Telegram sender paths do not forward it.

## Solution Assessment

No. The direction is reasonable, but the branch should rebase on current main, keep cache keys policy-aware, and either enforce messages.remoteMedia.ssrfPolicy across all outbound media senders or scope the config to the channels that actually honor it.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Include SSRF policy in the web_fetch cache key:** `src/agents/tools/web-fetch.ts:552`
  - body: The branch forwards `params.ssrfPolicy` into guarded fetches, but cache hits are keyed only by URL/extract mode/max chars and happen before the guard runs. A permissive run can populate the cache and a later strict run can read that result during the TTL window, so include a stable SSRF-policy component in the cache key or bypass cache across policy changes.
  - confidence: 0.93
- **[P2] Add citation redirect policy to Gemini cache identity:** `extensions/google/src/gemini-web-search-provider.ts:228`
  - body: The branch passes `citationRedirectSsrFPolicy` into redirect resolution, but Gemini search caching still uses provider/query/count/model only. A permissive redirect resolution can be reused by stricter calls without rerunning the redirect SSRF guard, so include a normalized policy fingerprint in the Gemini cache key.
  - confidence: 0.88
- **[P2] Wire remoteMedia policy through every sender:** `src/config/zod-schema.session.ts:191-196`
  - body: The branch adds global `messages.remoteMedia.ssrfPolicy`, but sender paths such as Slack and Telegram still call media loaders without that policy. Operators would see channel-dependent behavior for the same config setting, so every outbound remote-media path needs the same option or the schema should stay narrower.
  - confidence: 0.86
