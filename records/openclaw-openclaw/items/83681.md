## Summary

Keep open: current main still has separate write paths where a visible delivery mirror can append an assistant `delivery-mirror` record to the transcript tail before the triggering inbound user turn is persisted, so the reported parentId/order inversion is plausible and not covered by an existing open canonical fix.

## What This Changes

The issue asks OpenClaw to preserve causal transcript order when a chat-sourced inbound user turn is mirrored later than an assistant delivery produced by that turn.

## Best Possible Solution

Add a transcript append contract that anchors same-rollout delivery mirrors after the triggering inbound user turn, then cover delayed-user-plus-immediate-delivery ordering with a focused regression test.

## Reproduction Assessment

Yes, source-reproducible but not live-reproduced here: a focused test can append a delivery mirror through `appendAssistantMessageToSessionTranscript` before the corresponding user turn is persisted through SessionManager and then assert the user parentId/order inversion. I did not run that test in this read-only review.

## Solution Assessment

No current fix is present on main. The best repair should be at the shared transcript append/mirror boundary, not in one channel, so all chat-backed message-tool deliveries preserve causal ordering.
