## Summary

Keep open. This PR has a protected `maintainer` label and the patch introduces blocking issues around who may mutate persistent Codex plugin policy and whether the new command reflects the effective global policy.

## What This Changes

The PR adds `/codex plugins list|enable|disable` command handling, config-file mutation wiring for Codex sub-plugin entries, help text, and unit coverage.

## Best Possible Solution

Land only after mutating commands are owner/admin gated, global and per-entry Codex plugin policy are handled consistently, and real app-server/config mutation proof is added.

## Reproduction Assessment

Yes for the review findings from source inspection: an authorized non-owner can reach the new mutating handler because `/codex` only requires auth and the handler ignores `ctx.senderIsOwner`, and the effective policy stays disabled when global `codexPlugins.enabled` is false. I did not run a live command because this was a read-only review.

## Solution Assessment

No. The command surface is useful, but the current patch is not the narrowest safe solution until it uses the established owner/admin boundary for config writes and reports or updates the effective Codex plugin policy rather than only the entry flag.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Gate plugin config writes to owners/admins:** `extensions/codex/src/command-plugins-management.ts:29-32`
  - body: `/codex` only requires an authorized sender, and this new handler ignores `ctx.senderIsOwner` before `enable` or `disable` mutates `openclaw.json`. Because Codex plugin entries can expose native apps with destructive actions, an allowlisted non-owner could change persistent plugin policy. Keep `list` read-only, but reject mutating verbs unless the caller is owner/admin-scoped.
  - confidence: 0.92
- **[P2] Reflect the global Codex plugin switch:** `extensions/codex/src/command-plugins-management.ts:57-61`
  - body: `enable` only flips the per-entry `enabled` field, but current main computes effective plugin enablement as `codexPlugins.enabled && entry.enabled !== false`. When the global flag is false or omitted, this command reports the plugin as enabled and `list` prints `ON` even though the next Codex conversation still receives disabled app policy. Either set the global flag intentionally or report/block the effective disabled state.
  - confidence: 0.88
