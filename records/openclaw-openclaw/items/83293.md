## Summary

Keep open: the feature is plausible and the PR now includes terminal proof, but the current head still permits non-owner config writes and reports entry-level enablement inconsistently with the global Codex plugin switch.

## What This Changes

Adds `/codex plugins list|enable|disable` for Codex sub-plugin config inspection and mutation, plus help text and command tests.

## Best Possible Solution

Land a revised PR that keeps `list` read-only for authorized callers, gates `enable`/`disable` to owner or `operator.admin`, and handles global Codex plugin enablement consistently with the documented runtime policy.

## Reproduction Assessment

Yes for the review findings from source inspection: `/codex` receives `senderIsOwner`, but the new handler discards `_ctx` before mutating config, and current main requires global `codexPlugins.enabled` for effective policy.

## Solution Assessment

No. The command surface is useful, but the patch is not the safest implementation until config writes follow the existing owner/admin boundary and the command reflects effective global-plus-entry policy.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Gate plugin config writes to owners or admins:** `extensions/codex/src/command-plugins-management.ts:31`
  - body: `/codex` only requires an authorized sender, and this handler discards `ctx` before `enable` or `disable` mutates `openclaw.json`. A non-owner who is allowed to run `/codex` could change persistent Codex plugin app policy, including plugins with destructive actions. Keep `list` read-only, but reject mutating verbs unless the caller is owner/admin-scoped.
  - confidence: 0.92
- **[P2] Reflect the global Codex plugin switch:** `extensions/codex/src/command-plugins-management.ts:58-60`
  - body: `enable` only flips the per-entry flag, while current runtime policy enables a Codex plugin only when global `codexPlugins.enabled` is true and the entry is not disabled. If the global switch is false or omitted, this command reports the plugin as enabled and `list` prints `ON`, but the next Codex conversation still receives disabled app policy. Either set the global switch intentionally or report/block the effective disabled state.
  - confidence: 0.88
