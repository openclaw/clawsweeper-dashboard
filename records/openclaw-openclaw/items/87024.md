## Summary

Keep open. Current main still has source-reproducible paths where `media://inbound/...` can be converted into a workspace-relative filesystem path before the media-store resolver sees it, and the related open pull request covers only the native loader path rather than the direct image tool path.

## What This Changes

The issue asks for `media://inbound/...` Telegram image references to resolve consistently through the media store in both native prompt image loading and direct `image` tool usage.

## Best Possible Solution

Resolve inbound media references before workspace-relative path handling in both `loadImageFromRef` and the direct `image` tool, then cover both paths with focused regressions and either expand or supersede the related open pull request cleanly.

## Reproduction Assessment

Yes from source, though not from a live Telegram run: with `workspaceDir` set, current main can pass `media://inbound/...` through ordinary workspace path resolution before the media-store resolver runs. The same source pattern exists in the native prompt image loader.

## Solution Assessment

No, current main is not the best end state because only `loadWebMedia` and shared helpers know how to resolve the URI, while callers can still rewrite it first. The maintainable fix is to preserve or centrally resolve media-store refs before any local path normalization in the affected callers.
