## Summary

Keep open: current main and the latest release still sort light Dreaming entries by `lastRecalledAt` before `recallCount`, and deep promotion still gates on aggregate `signalCount`; the only targeted PR is closed unmerged while the overlapping signal-policy issue remains open.

## What This Changes

The issue requests memory-core Dreaming to prioritize real recall evidence over daily/session refresh signals so valid recalled entries are not hidden from light reports or promotion.

## Best Possible Solution

Settle one memory-core promotion signal contract, then update light ordering, rank/apply gates, diagnostics, docs, and regression tests together.

## Reproduction Assessment

Yes. Source inspection and current tests show daily/session signals can refresh `lastRecalledAt` and satisfy aggregate promotion gates while `recallCount` remains zero; I did not run a live workspace reproduction.

## Solution Assessment

Unclear as a standalone sort-only fix. The source confirms the report, but the durable fix needs a maintainer decision on whether daily/session/grounded signals should satisfy recall-like promotion gates.
