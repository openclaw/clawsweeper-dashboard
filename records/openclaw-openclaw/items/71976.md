Keep open: current main and v2026.6.10 still contain the recency-first light-dreaming order and the aggregate signal-count promotion gate, while both implementation PRs for this cluster are closed unmerged. The source reproduction is clear, but current tests preserve daily-only and grounded-only promotion semantics, so maintainers still need to decide the memory-core signal contract.

## What This Changes

The issue reports that memory dreaming light-phase ranking and short-term promotion gates prefer recent daily/session signals over true recall signals, hiding real recall data and contributing to zero promotions.

## Best Possible Solution

Keep this as the broad canonical memory-core dreaming signal-policy bug, decide the intended recall/daily/grounded promotion contract, then land one focused fix that aligns light ordering, rank/apply gates, diagnostics, docs, and regression tests.

## Reproduction Assessment

Yes. Source inspection on current main shows light dreaming ingests daily/session signals and sorts by lastRecalledAt before recallCount, while rank/apply compare minRecallCount to aggregate signalCount; no live cycle was run in this read-only review.

## Solution Assessment

Unclear as a one-line fix. Recall-first light ordering is narrow, but changing the promotion gate needs maintainer direction because current tests intentionally allow daily-only and grounded-only evidence to satisfy default deep thresholds.
