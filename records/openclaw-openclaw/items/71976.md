## What This Changes

The issue asks memory-core Dreaming to prioritize real recall signals over daily/session freshness and mixed signal counts so valid short-term recall candidates are not hidden or filtered out.

## Best Possible Solution

Maintainers should decide the memory-core signal contract, then update light ordering, promotion gates/scoring, rehydration span if still required, and focused tests/docs in one bounded fix.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: light Dreaming still refreshes daily/session signals and sorts by lastRecalledAt before recallCount, and deep promotion still gates on totalSignalCount including daily and grounded signals. I did not run a live memory store reproduction because this review is read-only.

## Solution Assessment

Unclear as a mechanical fix: recall-first ordering is plausible, but current tests intentionally let daily-only signals qualify, so the best fix needs a maintainer decision on daily/session/grounded signal semantics before code changes.
