## What This Changes

The issue asks memory-core Dreaming to stop letting fresh daily/session signals bury genuinely recalled entries in light output and short-term promotion.

## Best Possible Solution

Settle one memory-core promotion signal contract, then update light ordering, rank/apply gates, diagnostics, docs, and focused regressions together instead of landing the old branch piecemeal.

## Reproduction Assessment

Yes. Current source still runs daily/session ingestion before light selection, updates lastRecalledAt for daily signals without incrementing recallCount, and sorts light entries by timestamp before recall count; I did not run a live workspace reproduction in this read-only review.

## Solution Assessment

Unclear as a standalone sort-only fix. The source confirms the reported behavior, but current tests intentionally allow daily-only promotion, so maintainers need to choose the final daily/session/grounded signal contract before a repair is safe.
