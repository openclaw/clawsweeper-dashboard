## Summary

Keep open. Current main still has the reported Memory Core behavior: Light refreshes daily/session signals before sorting by `lastRecalledAt`, Deep ranking/apply still gate on aggregate signal count, and rehydration can still drop unmatched candidates; related items cover pieces, but the light-sort and overall promotion contract are not fully superseded.

## What This Changes

The issue asks Memory Core dreaming to prioritize genuine recall data over daily/session freshness and avoid dropping valid promotion candidates during promotion rehydration.

## Best Possible Solution

Define one Memory Core promotion contract for true recalls, daily/session freshness, grounded evidence, Light ordering, scoring gates, and rehydration, then update implementation, docs, diagnostics, and regression tests together.

## Reproduction Assessment

Yes, at source level. Current main still shows recency-first Light sorting after daily/session refresh, aggregate Deep gates, daily-only eligibility tests, and the rehydration drop path; I did not replay the reporter's private memory store.

## Solution Assessment

Unclear as a simple patch. The proposed sort and span changes target real symptoms, but the durable fix needs to align with the open gate discussion at https://github.com/openclaw/openclaw/pull/74114 and the rehydration work tracked by https://github.com/openclaw/openclaw/issues/77707.
