## Summary

Keep open. Current main still contains the reported Memory Dreaming behavior, and open PR #74051 is explicitly linked with closing syntax as the active implementation candidate, so this issue should remain open until that PR or a focused replacement lands.

## What This Changes

The issue requests a Memory Core fix so real recall signals, not daily/session recency refreshes, drive Light dreaming visibility and deep promotion eligibility, with more robust rehydration for shifted snippets.

## Best Possible Solution

Review and land #74051, or a focused replacement, so real recalls drive Light ordering and recall-gate eligibility, daily/session signals can still support scoring without satisfying recall gates alone, rehydration handles shifted snippets, and tests/docs define the corrected Memory Dreaming contract.

## Reproduction Assessment

Yes. Source inspection on current main reproduces the mechanism: daily/session ingestion refreshes `lastRecalledAt`, Light sorts by that timestamp before `recallCount`, and deep promotion gates still use combined signal counts rather than real recall count alone.

## Solution Assessment

No for the issue's initial sort-only fix. The narrower maintainable direction is a Memory Core fix that separates recall-gate semantics from daily/session scoring, preserves intentional grounded evidence behavior, and hardens rehydration with focused tests and docs.
