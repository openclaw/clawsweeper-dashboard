Keep open: current main and v2026.6.10 still contain the recency-first light-dreaming sort and aggregate signal-count promotion gates, while the linked fix PRs are closed unmerged and the remaining signal contract needs maintainer direction.

## What This Changes

The issue asks memory-core dreaming to prioritize real recall-backed short-term entries over daily/session refreshes and to avoid promotion/rehydration behavior that hides valid recall data.

## Best Possible Solution

Keep this as the broad memory-core dreaming signal-policy bug, decide the intended recall/daily/grounded contract, then land one focused fix covering light ordering, rank/apply gates, diagnostics, docs, and regression tests.

## Reproduction Assessment

Yes. Source inspection on current main shows daily/session ingestion refreshes lastRecalledAt, light dreaming sorts by recency before recallCount, and rank/apply compare minRecallCount to aggregate signalCount; no live cycle was run in this read-only review.

## Solution Assessment

Unclear as a final fix. Recall-first light ordering is narrow, but the promotion gate needs maintainer direction because current tests intentionally allow daily-only and grounded-only evidence to satisfy default deep thresholds.
