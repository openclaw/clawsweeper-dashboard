## Summary

Keep open: current main and v2026.5.22 still implement non-loopback Gateway bind as IPv4-only, and the closed related PRs/documentation work do not provide native IPv6 or dual-stack listening. The remaining fix is clear in source, but it needs maintainer security and cross-platform product policy review before implementation.

## What This Changes

The issue requests native IPv6 or dual-stack listening support for Gateway LAN/custom bind modes.

## Best Possible Solution

Define the public Gateway IPv6 and dual-stack bind contract, then update bind resolution, listener creation, custom host validation, status/link output, security docs, and cross-platform socket tests together.

## Reproduction Assessment

Yes. Source inspection shows `gateway.bind=lan` resolves to `0.0.0.0`, non-loopback hosts are not expanded to IPv6 listeners, and `customBindHost` rejects IPv6 literals; no live socket proof was run.

## Solution Assessment

Unclear. Directly switching LAN to `::` would address some IPv6 deployments, but the safer maintainable path needs an explicit dual-stack/exposure policy and platform proof first.
