Unclear. Binding LAN directly to :: is a plausible fix, but the best solution needs an explicit IPv6/dual-stack exposure contract and cross-platform proof before changing a security-sensitive Gateway listener path.

## Risks / Open Questions

- Changing LAN/custom Gateway bind behavior could make the Gateway reachable on globally routable IPv6 interfaces, so auth, origin checks, trusted-proxy assumptions, startup warnings, and firewall guidance need security review.
- Dual-stack listening is platform- and container-sensitive; a direct lan-to-:: change may behave differently on Windows, hardened Linux ipv6Only settings, Docker, and host-network deployments.
- The best user-facing contract is not decided yet: maintainers need to choose between changing existing lan semantics, extending custom IPv6 literals, adding an explicit dual-stack mode, or documenting sidecar/proxy-only support.
