## Summary

Keep open: current main and v2026.6.6 still keep Gateway LAN/custom BYOH bind paths IPv4-only, and the related reports point back here as the canonical IPv6/dual-stack bind request. The requested network exposure change needs maintainer security and product direction before implementation.

## What This Changes

Request support for IPv6 or dual-stack listening when the OpenClaw gateway runs in LAN/custom bind mode.

## Best Possible Solution

Define the public Gateway IPv6/dual-stack bind contract first, then update bind resolution, listener creation, custom host validation, status/link/probe output, security docs, and cross-platform socket coverage together.

## Reproduction Assessment

Yes. Source inspection shows current main resolves LAN bind to `0.0.0.0`, does not expand non-loopback bind hosts to IPv6 listeners, and rejects IPv6 `customBindHost` values; no live socket repro was run in this read-only review.

## Solution Assessment

Unclear. Binding LAN directly to `::` is a plausible implementation, but the best solution needs an explicit IPv6/dual-stack exposure contract and cross-platform proof before changing a security-sensitive Gateway listener path.

## Risks / Open Questions

- Changing LAN/custom binding from IPv4-only to IPv6 or dual-stack can make the Gateway reachable on additional non-loopback interfaces, including globally routable IPv6 addresses.
- A direct `lan` -> `::` change has platform and deployment differences, including Windows/libuv behavior, Linux `bindv6only`, Docker networking, and firewall rules.
- Maintainers still need to choose the public contract: change existing `lan` semantics, allow IPv6 literals in `customBindHost`, add an explicit dual-stack mode, or keep sidecar/proxy-only support.
