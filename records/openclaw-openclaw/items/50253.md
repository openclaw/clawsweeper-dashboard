## Summary

Current main and the latest tagged release still keep Gateway LAN/custom bind on IPv4-only paths, so this remains a valid IPv6/dual-stack Gateway feature request. The related IPv6 issue was closed in favor of this issue, and the open docs PR only documents the limitation rather than implementing it.

## What This Changes

Request that Gateway LAN/custom bind support IPv6 or dual-stack listening instead of only IPv4.

## Best Possible Solution

Define the public Gateway IPv6/dual-stack bind contract, then update bind resolution, listener expansion, custom host validation, status/control UI links, security/config docs, and socket proof together.

## Reproduction Assessment

Yes. Source inspection shows current main resolves gateway.bind=lan to 0.0.0.0, does not add :: for non-loopback listeners, and rejects IPv6 customBindHost values; I did not run live sockets in this read-only review.

## Solution Assessment

Unclear. Directly switching LAN to :: may solve IPv6-only deployments, but the safer maintainable solution needs an explicit cross-platform dual-stack policy and exposure documentation.
