## Summary

Keep open: current main and the latest release still resolve Gateway LAN/custom bind paths as IPv4-only, and the related IPv6 issue was already closed as a duplicate of this canonical tracker. The remaining work is a gateway networking contract decision, not cleanup.

## What This Changes

Request IPv6 or dual-stack listening for Gateway bind/LAN mode, including `::` or specific IPv6 bind addresses.

## Best Possible Solution

Define a core Gateway IPv6/dual-stack bind contract, then update bind resolution, listener expansion, custom host validation, status/control UI links, security/config docs, and focused regression coverage together.

## Reproduction Assessment

Yes, source-reproducible: `gateway.bind=lan` resolves to `0.0.0.0`, non-loopback listen hosts are not expanded, and `gateway.bind=custom` rejects non-IPv4 literals on current main. I did not run a live socket repro because this review is read-only.

## Solution Assessment

Unclear. Allowing `::` or making LAN dual-stack may be the right direction, but maintainers should first choose the public cross-platform bind behavior and security expectations.
