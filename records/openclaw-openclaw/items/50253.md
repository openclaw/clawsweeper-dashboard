## Summary

Keep open. Current main still codifies the reported behavior: LAN bind resolves to IPv4 wildcard only, the listener does not add an IPv6 wildcard for non-loopback binds, and the config/runtime path rejects IPv6 custom bind hosts. The related #63743 has stronger network evidence for the same request, but it is closed, so this remains the live tracker.

## What This Changes

Request IPv6 listening support for OpenClaw Gateway bind mode, especially LAN/dual-stack or IPv6-only deployments.

## Best Possible Solution

Add explicit IPv6-capable Gateway bind behavior in core, with config schema/runtime resolver/server listen/docs/tests updated together and existing non-loopback auth and Control UI origin guardrails preserved. The maintainable end state should make clear whether IPv6 is opt-in through `customBindHost`, part of `lan`, or a separate bind mode.

## Reproduction Assessment

Yes. Current source and tests reproduce the issue by resolving `gateway.bind="lan"` to `0.0.0.0` only, and a live Node socket check showed that this IPv4 wildcard did not accept a `::1` connection in this environment.

## Solution Assessment

Unclear. The request is valid, but the best fix needs maintainer API judgment because a silent `lan` semantic change may broaden network exposure; an explicit IPv6-capable bind path may be safer.
