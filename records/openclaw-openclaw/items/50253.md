## Summary

Keep open: current main and v2026.5.22 still implement non-loopback Gateway bind as IPv4-only, and the closed related reports/PRs leave this as the canonical remaining IPv6/dual-stack bind request. The request is valid, but changing Gateway LAN/custom IPv6 exposure needs maintainer security and cross-platform policy review before an automated fix PR.

## What This Changes

Request to support IPv6 or dual-stack listening for Gateway LAN/custom bind mode.

## Best Possible Solution

Define the public Gateway IPv6/dual-stack bind contract, then update bind resolution, listener creation, custom host validation, status/link output, security docs, and cross-platform socket tests together.

## Reproduction Assessment

Yes. Source inspection shows `gateway.bind=lan` resolves to `0.0.0.0`, non-loopback hosts are not expanded to IPv6 listeners, and `customBindHost` rejects IPv6 literals; no live socket proof was run.

## Solution Assessment

Unclear. Directly switching `lan` to `::` could help some IPv6 deployments, but the safer maintainable solution needs an explicit IPv6/dual-stack exposure contract and cross-platform proof first.
