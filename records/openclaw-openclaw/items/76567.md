## Summary

Current `main` and the latest release already implement the requested configured-memory-slot runtime loading. The linked same-author fix branch is useful context, but the central behavior shipped through the merged canonical fix at https://github.com/openclaw/openclaw/pull/76393; the broader multi-slot memory PR remains a separate product/API track.

## What This Changes

The issue asks memory runtime and status checks to load or consult the configured `plugins.slots.memory` plugin before reporting no active memory runtime.

## Best Possible Solution

Keep the shipped slot-scoped runtime loader from https://github.com/openclaw/openclaw/pull/76393; broader memory-role architecture should continue separately in https://github.com/openclaw/openclaw/pull/88504.

## Reproduction Assessment

Yes for the historical bug: the canonical report and fix show standalone doctor/status paths could see an empty process-local memory runtime before loading the configured slot. Current `main` no longer has that failing source path because the helper loads the configured plugin id first.

## Solution Assessment

Yes. The shipped fix is the narrow owner-boundary solution: load only the configured memory slot through the existing runtime registry loader instead of adding a new config option or broad plugin activation.
