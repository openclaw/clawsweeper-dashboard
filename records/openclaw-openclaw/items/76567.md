## Label Justifications

- `P2`: The issue described a real memory-runtime diagnostics bug with limited blast radius, now fixed by a shipped narrow runtime change.
- `impact:session-state`: The affected behavior made configured memory providers appear unavailable in doctor/status paths, which is session and memory state visibility.

## Summary

Current main already solves the reported cold-registry memory runtime check: the helper now resolves `plugins.slots.memory`, loads only that configured plugin through the standalone runtime loader, and the fix is present in tagged release source.

## What This Changes

Runtime memory, doctor, and status checks should load the plugin configured in `plugins.slots.memory` before reporting that no active memory runtime exists.

## Best Possible Solution

Close this issue and keep the shipped narrow runtime fix from https://github.com/openclaw/openclaw/pull/76393; broader memory role-slot architecture remains separate in https://github.com/openclaw/openclaw/pull/88504.

## Reproduction Assessment

Yes for the historical bug: the linked reports and pre-fix code path show doctor/status could query `getMemoryRuntime()` before the configured memory plugin had registered in that CLI process. Current main no longer reproduces that source path because it loads the configured slot before returning no runtime.

## Solution Assessment

Yes. The merged solution in https://github.com/openclaw/openclaw/pull/76393 is the narrow owner-boundary fix: it loads only the configured memory slot instead of broad plugin activation or a new config workaround.
