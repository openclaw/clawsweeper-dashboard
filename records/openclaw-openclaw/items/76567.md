## Summary

Current main and the latest stable release already load the configured memory-slot plugin for runtime/status checks, so the reported false “no active memory runtime” path is implemented and can close.

## What This Changes

The issue asks memory runtime/status checks to consult the configured `plugins.slots.memory` plugin before reporting that no memory runtime is active.

## Best Possible Solution

Keep the shipped slot-scoped runtime-loader behavior and close this issue as implemented; the remaining cleanup is to handle the now-superseded linked PR separately.

## Reproduction Assessment

No for current main; the failing path is fixed. The old source path is high-confidence: the pre-fix helper only probed an already-loaded registry and returned no runtime when the configured memory plugin had not yet been loaded.

## Solution Assessment

Yes. The shipped helper uses the existing standalone registry loader with the configured plugin id only, which is the narrowest maintainable core fix for this runtime check.
