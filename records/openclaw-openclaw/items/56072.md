## Summary

Keep open. Current main still rolls stale daily/idle sessions into a new session and archives the old transcript without invoking the compaction memory-flush turn or exposing the requested reset policy controls; the related open PRs address the hook gap only as partial implementation candidates, not the broader reset-time memory-flush behavior requested here.

## What This Changes

The issue requests reset-time durable-memory preservation for daily/idle session rollover, including a compaction-style memory flush before transcript archive plus reset controls such as flush/archive-only and off modes.

## Best Possible Solution

Keep this issue open and decide the reset contract explicitly: add a core stale-rollover preservation path that can run the existing `agents.defaults.compaction.memoryFlush` silent turn before archive, add schema/type/docs coverage for reset policy controls such as flush/archive-only and off/manual-only, and either bridge or clearly document how internal command hooks, plugin lifecycle hooks, and reset-time flush semantics relate.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to seed a session whose `updatedAt` crosses the daily or idle freshness boundary and call `initSessionState`; current main creates a new session and archives the old transcript, while the only memory-flush call is later in the reply runner and the bundled hook remains command-only.

## Solution Assessment

Unclear. The requested core reset-time memoryFlush path is a plausible maintainable solution, but the open hook-based PRs are only partial fixes for session-memory persistence and do not implement the requested `onReset`/off policy or direct compaction-flush behavior.
