## Summary

Keep open: current main and the latest release still have the brittle unvalidated MCP tool listing path, while this PR adds unique bounded hardening. It should stay open for maintainer acceptance because the branch is maintainer-owned, protected, and intentionally compatibility-sensitive.

## What This Changes

The PR changes the shared MCP tools handler to materialize plugin tool descriptors through runtime schema projection, validate them with the MCP SDK `ToolSchema`, and quarantine invalid schemas from both listing and calling, with focused tests.

## Best Possible Solution

Land after maintainer acceptance that this shared MCP bridge should expose and call only tools satisfying runtime projection plus the MCP `ToolSchema` contract; add an explicit compatibility path first if invalid schemas must remain callable.

## Reproduction Assessment

Yes at source level: current main evaluates `"type" in params` while listing tools, so a proxy `has` trap for `type` can throw before `tools/list` returns. I did not rerun the repro in this read-only review, but the current code path and PR body before/after proof are consistent.

## Solution Assessment

Yes, with maintainer compatibility acceptance: reusing runtime schema projection and validating against the MCP SDK contract is the narrow boundary fix. The safer alternative, if compatibility must dominate, is an explicit tested compatibility mode rather than silently listing invalid tools.
