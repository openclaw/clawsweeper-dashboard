## Summary

Keep open. The PR addresses a real current-main gap and is the canonical follow-up for the closed source issue, but the proposed marker set still misses the default NVS install path, so it does not fully deliver the claimed all-version-manager behavior.

## What This Changes

The PR adds a bootstrap version-manager marker array for CA bundle auto-injection and adds tests for fnm, volta, asdf, n, nodenv, nodebrew, and nvs exec-path detection.

## Best Possible Solution

Update the version-manager detection to include default NVS paths, preferably by sharing or synchronizing the marker set across bootstrap, daemon audit, and runtime-path detection, then add focused tests and the required changelog entry before merge.

## Reproduction Assessment

Yes. On current main, any non-nvm version-manager exec path returns false because only `/.nvm/` is checked; on the PR branch, a default NVS path like `/home/user/.nvs/node/22.14.0/x64/bin/node` still returns false because the added marker is `/nvs/`.

## Solution Assessment

No. The direction is right, but the exact patch is not the best final fix because it copies an already-incomplete NVS marker; the safer repair is to include `/.nvs/` and keep related marker lists/tests aligned.
