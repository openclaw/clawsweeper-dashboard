## Summary

Keep open: current main has broad redaction helpers and some sanitized display/log paths, but provider error construction still has at least one raw error-message path and the OpenAI Responses client does not use the guarded fetch path that adjacent OpenAI Completions uses.

## What This Changes

The issue reports that unreachable custom provider endpoints may surface raw API keys through network error messages, request URLs, headers, logs, dashboards, or agent-visible previews.

## Best Possible Solution

Add a shared provider-error sanitation boundary before any provider/network error becomes session state, retry events, diagnostics, logs, or UI payloads, and cover URL query secrets plus Authorization headers with focused regression tests.

## Reproduction Assessment

No live high-confidence reproduction was run in this read-only pass. Source inspection does show credential-bearing provider requests can flow into raw provider error-message fields, so the issue is source-reproducible enough to keep open.

## Solution Assessment

Yes, the proposed direction is the right shape if implemented at the shared provider/network error boundary. Scattered UI-only redaction would be too narrow because raw session, retry, diagnostic, and log paths can consume provider error text before display projection.
