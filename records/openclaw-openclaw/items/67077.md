## Summary

Keep open. This is a valid focused contributor fix with sufficient real behavior proof, but current main still has the request-visible persistence cascade and the branch now needs maintainer rebase/adaptation after https://github.com/openclaw/openclaw/pull/80375 consolidated success bookkeeping into `markAuthProfileSuccess`.

## What This Changes

The PR wraps auth-profile last-good, usage, and failure bookkeeping saves in non-fatal warn/debug logging, adds EPERM regression coverage, and updates the changelog.

## Best Possible Solution

Rebase and adapt the patch so current `markAuthProfileSuccess` plus failure/cooldown bookkeeping persistence are best-effort, while credential and OAuth persistence failures remain fatal and EPERM regression coverage stays focused on the lock-null direct-save path.

## Reproduction Assessment

Yes at source level: current main awaits `markAuthProfileSuccess` after a successful embedded run, and both success and failure bookkeeping can still call `saveAuthProfileStore` without a local catch after the lock path returns null. I did not live-reproduce the Windows ReadOnly race.

## Solution Assessment

Yes, the proposed direction is the narrow maintainable fix because it catches only post-completion bookkeeping persistence failures instead of weakening `saveAuthProfileStore` globally. The implementation needs current-main adaptation around `markAuthProfileSuccess`.
