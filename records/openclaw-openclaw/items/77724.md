## Summary

Keep open: the Feishu local-media gap is source-reproducible on current main, but this PR is not mergeable as written because it reads `info.mediaAccess` from a reply dispatcher info object that currently only carries `kind`, so the patch should fail typecheck and would not receive roots in production. The external PR also only provides unit-test output, not after-fix live Feishu behavior proof.

## What This Changes

The PR wires Feishu reply media dispatch to pass authorized local file roots to `sendMediaFeishu`, adds a regression test, and adds a changelog entry.

## Best Possible Solution

Revise the PR to pass Feishu reply media roots from a typed existing source, likely agent-scoped media roots for the route, then show a live Feishu run where an allowed `file://` attachment uploads instead of being sent as text.

## Reproduction Assessment

Yes from source inspection, though not from a live Feishu run: current main omits `mediaLocalRoots` in Feishu reply media delivery while `sendMediaFeishu` only authorizes local reads when those roots are passed. The contributor's proof does not reproduce the real channel behavior after the fix.

## Solution Assessment

No. Forwarding local roots is the right direction, but `info.mediaAccess` is not part of the reply delivery contract, so the narrow maintainable fix is to use an existing typed Feishu/agent media-root source or explicitly extend the shared contract with coverage.

## Review Findings
