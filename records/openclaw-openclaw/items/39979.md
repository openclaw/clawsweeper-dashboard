## Summary

Keep open. Current main still has tool-name allow/deny, `tools.fs.workspaceOnly`, and command/executable allowlists, but no path-keyed RWX policy shared across file tools and exec; the closed unmerged implementation attempts and duplicate #68631 make this the canonical maintainer/secops design thread.

## What This Changes

Add a backward-compatible path-keyed RWX permission model for file tools, apply_patch, and exec so permissions inherit from target paths rather than approved binaries.

## Best Possible Solution

Keep this as the canonical thread and design a versioned filesystem permission contract that covers read, write, edit, apply_patch, and exec operands while preserving existing exec approval behavior and documenting platform limits.

## Reproduction Assessment

Yes for the current-state gap: source and test inspection show file tools are governed by `workspaceOnly`, and exec approvals match commands/executables rather than target paths. I did not run tests because this review was read-only.

## Solution Assessment

Unclear as a concrete implementation plan, but keeping this open is the best triage outcome. The path-RWX direction matches the observed gap, while the schema, OS enforcement, and delegation semantics need maintainer/secops design before a safe implementation PR.
