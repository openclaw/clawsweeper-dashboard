## Summary

Current main and the latest inspected release still expose binary-oriented exec approvals plus boolean workspace-only file guards, not a unified path-scoped RWX policy across exec and file tools. The linked implementation PRs were closed unmerged after security-review concerns, so this remains a valid security/product design issue rather than a cleanup-close candidate.

## What This Changes

The issue requests path-scoped read/write/execute permissions that apply consistently to exec, read, write, edit, and apply_patch while preserving existing exec allowlist behavior.

## Best Possible Solution

Keep this as the canonical security design thread until maintainers choose one policy contract that composes with existing exec approvals, workspaceOnly guards, sandboxing, and doctor/config migration rules.

## Reproduction Assessment

Yes, source inspection reproduces the gap: current main has only binary exec approvals and boolean workspaceOnly file guards, while docs explicitly say exec approvals are not a filesystem read-only policy.

## Solution Assessment

Unclear; the requested capability is valid, but the rejected implementation PRs show that the exact policy format, enforcement layers, and compatibility path need maintainer security/product design before implementation.
