## Summary

Keep open: current main still has executable-oriented exec approvals plus coarse workspace-only filesystem guards, not a shared path-keyed RWX policy for exec, file tools, and apply_patch. The closed implementation PRs are useful design history, and the duplicate path-policy issue was closed in favor of this issue, so this remains the canonical security/config design thread.

## What This Changes

Request path-keyed RWX permission rules shared by exec, read/write/edit, and apply_patch so filesystem access is scoped by target path rather than executable.

## Best Possible Solution

Define a maintainer-approved path-keyed RWX contract, then implement it across config/schema/protocol, exec approvals, file/apply_patch enforcement, platform sandbox behavior, docs, and focused security regression tests.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives a high-confidence current-state check: exec approvals are still allowlist/executable oriented and file tools still expose only workspaceOnly rather than path-keyed RWX rules.

## Solution Assessment

Unclear until maintainers choose the security/config contract. The requested direction is plausible, but the best solution should be a reviewed core policy design rather than reviving a closed broad implementation verbatim.
