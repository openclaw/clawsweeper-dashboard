## Summary

Keep open. Current main still has workspace-only filesystem guarding and executable/command allowlists, but no shared path-keyed RWX policy across read/write/edit/apply_patch and exec; the linked implementation attempts were closed unmerged and the later overlapping request was closed as a duplicate of this thread.

## What This Changes

Request a backward-compatible path-scoped RWX permission map that applies uniformly to filesystem tools and exec instead of relying on binary-level exec allowlists.

## Best Possible Solution

Keep this as the canonical design thread and specify a versioned filesystem permission contract that covers read, write, edit, apply_patch, and exec operands while preserving existing exec approval behavior and documenting platform limits.

## Reproduction Assessment

Yes for the current-state gap: source and test inspection show file tools are governed by workspaceOnly and exec approvals match commands/executables rather than target paths. I did not run tests because this was a read-only review.

## Solution Assessment

Unclear as a final implementation plan, but keeping this open is the right triage outcome. The path-RWX direction matches the observed gap, while schema shape, OS-level enforcement, and delegation semantics need maintainer/secops agreement before implementation.
