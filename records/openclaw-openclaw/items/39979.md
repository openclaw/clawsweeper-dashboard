## Summary

Keep open. Current main still has workspace-only filesystem policy and executable/command-based exec approvals, while the path-keyed RWX model across file tools and exec remains unimplemented; the related implementation PRs were closed unmerged, so this remains the canonical narrow design thread.

## What This Changes

Request to add a backward-compatible path-keyed RWX permission map shared by file tools, apply_patch, and exec instead of relying on binary-scoped exec allowlists.

## Best Possible Solution

Keep this as the canonical security/design thread and define a versioned path-permission contract shared by file tools, apply_patch, and exec, with explicit platform limits and compatibility for existing exec approvals.

## Reproduction Assessment

Yes for the current-state gap: source inspection and existing tests show file tools are governed by workspaceOnly and exec approvals authorize executable/command allowlists, not path operands. I did not run tests because this review is read-only.

## Solution Assessment

Unclear as a concrete implementation plan, but keeping this open is the best triage outcome. The path-RWX direction matches the missing boundary, while schema, delegation, and platform enforcement need maintainer/security design first.
