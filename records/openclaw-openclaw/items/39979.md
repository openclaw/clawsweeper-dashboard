## Summary

Keep open: current main still has executable-scoped exec approvals plus coarse workspace-only filesystem guards, not a shared path-keyed RWX policy for exec, read, write, edit, and apply_patch. The closed implementation PRs and the closed duplicate issue are useful design history, but this remains the canonical security/config design item.

## What This Changes

Add path-keyed read/write/execute permissions that apply consistently across exec and filesystem tools while preserving existing exec allowlist compatibility.

## Best Possible Solution

Define a maintainer-approved path-keyed RWX permission contract, then implement it across config/schema/protocol, exec approvals, read/write/edit/apply_patch enforcement, platform sandbox behavior, docs, and focused security regression tests.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives a high-confidence current-state check: exec approvals remain executable scoped and file tools remain governed by workspaceOnly, not path-keyed RWX rules.

## Solution Assessment

Unclear until maintainers choose the security/config contract. The requested direction is plausible, but the best solution should be a reviewed core policy design rather than reviving a closed broad implementation verbatim.
