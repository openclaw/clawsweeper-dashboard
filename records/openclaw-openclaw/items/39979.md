## Summary

Keep open: current main still exposes exec approvals as allowlist-shaped command policy and filesystem tools as workspace/workspaceOnly guards, not the requested path-keyed RWX policy shared across exec and file tools. The closed implementation PRs and related duplicate are useful design history, but the remaining change is a new security/config contract that needs maintainer review.

## What This Changes

Request a path-keyed read/write/execute filesystem permission map shared by file tools, apply_patch, and exec while preserving existing exec allowlist compatibility.

## Best Possible Solution

Define a maintainer-approved path-keyed RWX permission contract, then implement it consistently across config/schema/protocol, exec approvals/runtime, file tools, apply_patch, platform sandboxing, docs, and security regression tests.

## Reproduction Assessment

Not applicable as a feature/security hardening request. Source inspection gives a high-confidence current-state check: exec approvals remain allowlist-shaped and filesystem tools remain governed by workspaceOnly rather than a path-keyed RWX map.

## Solution Assessment

Unclear until maintainers choose the security and config contract. The requested direction is plausible, but the best path is a reviewed core policy design using the closed PR discussions as input rather than reviving a broad implementation verbatim.
