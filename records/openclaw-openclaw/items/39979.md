## Summary

Keep open: current main still exposes executable-oriented exec approvals plus `workspaceOnly` filesystem guards, not a path-keyed RWX policy shared by exec, read, write, edit, and apply_patch. The closed unmerged implementation PRs are useful design history, and https://github.com/openclaw/openclaw/issues/68631 was already closed in favor of this issue as the canonical remaining path-policy thread.

## What This Changes

The issue requests a path-keyed RWX permission map that applies consistently to exec and file tools while preserving existing exec allowlist compatibility.

## Best Possible Solution

Define a maintainer-approved path-keyed RWX permission contract, then implement it across exec, read/write/edit, apply_patch, config/schema/protocol/docs, platform sandbox fallbacks, and focused security regression tests.

## Reproduction Assessment

Not applicable as a feature request. Current-main source and tests provide a high-confidence state check: exec is still binary/safe-bin approval based, and filesystem tools still use coarse `workspaceOnly` behavior rather than path-keyed RWX rules.

## Solution Assessment

No implemented solution is present on current main. The requested direction is plausible, but the best path is a maintainer-owned security/config design before another implementation PR.
