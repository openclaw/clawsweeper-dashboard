## Summary

Keep open: current main and the latest release still provide executable allowlists plus `tools.fs.workspaceOnly` boolean guards, not a path-keyed RWX permission map shared by exec and file tools. The related implementation attempts are closed or partial open work, and the remaining change is a security-sensitive config and enforcement contract that needs maintainer product/security review.

## What This Changes

The issue requests a path-keyed read/write/execute permission model that applies uniformly to file tools and exec so permissions follow target paths rather than executable allowlist entries.

## Best Possible Solution

Define an owner-reviewed path permission contract first, then implement it consistently across config/schema/protocol, exec approvals/runtime, file tools, apply_patch, platform sandboxing, docs, and security regression proof.
