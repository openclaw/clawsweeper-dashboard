## Summary

Keep open: current main still has executable-scoped exec approvals plus boolean workspace-only filesystem guards, not the requested shared path-keyed RWX permission map. The closed implementation attempts and the closed duplicate are useful design history, but the current labels and code state still point to maintainer security/product review rather than cleanup closure.

## What This Changes

Request a backward-compatible path-keyed RWX permission map shared by exec and file tools, with deny/default semantics and per-agent scoping.

## Best Possible Solution

Define a maintainer-approved path-keyed RWX permission contract, then implement it across config/schema/protocol, exec approval/runtime, file-tool and apply_patch enforcement, platform sandboxing, docs, and security regression tests.

## Reproduction Assessment

Not applicable as a feature/security hardening request. Source inspection gives a high-confidence current-state check: exec approvals remain executable/allowlist scoped and file tools remain governed by workspaceOnly rather than path-keyed RWX rules.

## Solution Assessment

Unclear until maintainers choose the security and config contract. The requested direction is plausible, but the best path is a reviewed core policy design using the closed PR discussions as input rather than reviving a broad implementation verbatim.
