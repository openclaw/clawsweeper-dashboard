## Summary

Keep open: current main still has executable-scoped exec approvals plus boolean workspace-only filesystem guards, not the requested shared path-keyed RWX permission map. The closed implementation attempts and the closed duplicate are useful design history, but this remains the canonical security/config design thread.

## What This Changes

The issue requests a path-keyed filesystem permission model with read/write/execute bits enforced uniformly across exec and file tools.

## Best Possible Solution

Define a maintainer-approved path-keyed RWX permission contract, then implement it across config/schema/protocol, exec approval/runtime, read/write/edit/apply_patch enforcement, platform sandbox behavior, docs, and focused security regression tests.

## Reproduction Assessment

Not applicable as a feature/security hardening request. Source inspection gives a high-confidence current-state check: exec approvals remain executable/allowlist scoped and file tools remain governed by workspaceOnly rather than path-keyed RWX rules.

## Solution Assessment

Unclear until maintainers choose the security and config contract. The requested direction is plausible, but the best path is a reviewed core policy design using the closed PR discussions as input rather than reviving a broad implementation verbatim.
