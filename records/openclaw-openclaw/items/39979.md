## Summary

Keep open: current main and v2026.6.10 still provide boolean workspace-only filesystem guards, sandboxing, and exec approval allowlists, but not a shared path-scoped RWX permission map for file tools and exec. The linked implementation attempts are closed unmerged, and the remaining work needs security/product ownership rather than conservative cleanup.

## What This Changes

Request path-scoped read/write/execute permission policy shared by file tools and exec so access is enforced by target path instead of by binary allowlist alone.

## Best Possible Solution

Settle an owner-approved path permission contract first, including canonical config shape, migration/doctor behavior, file-tool and exec enforcement semantics, OS sandbox role, docs, and focused tests, then land one focused security-hardening implementation.

## Reproduction Assessment

Yes. Source inspection on current main and v2026.6.10 shows only boolean workspace-only filesystem policy plus exec approval/sandbox guardrails, not a path-scoped RWX policy shared across file tools and exec.

## Solution Assessment

No, not as an automated one-shot fix. The RWX direction fits the security gap, but maintainers need to settle the config, migration, threat model, and enforcement contract before implementation.
