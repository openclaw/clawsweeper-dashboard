## Summary

Keep open: current main and v2026.6.6 still expose binary/command-scoped exec controls plus sandbox/workspace boundaries, not a path-keyed RWX authorization map shared by file tools and exec. The two linked implementation attempts are closed unmerged after security-review concerns, while the duplicate cleanup at https://github.com/openclaw/openclaw/issues/68631 points back to this issue as the canonical thread.

## What This Changes

Request a core path-scoped read/write/execute permission model that applies uniformly to file tools and exec while preserving existing exec allowlist compatibility.

## Best Possible Solution

Keep this as the canonical maintainer design thread and land a security-reviewed core policy that has one canonical config shape, explicit upgrade/doctor behavior, file-tool and exec enforcement parity, OS-sandbox limits documented, and focused bypass tests.

## Reproduction Assessment

Not applicable as a bug reproduction, but the current-source gap is high-confidence: exec approvals are binary/command-scoped and file tools are guarded by workspace/root containment, with no shared path-RWX permission map.

## Solution Assessment

Unclear as a final design: path-scoped enforcement is the right problem area, but the exact config location, compatibility plan, and OS-sandbox semantics need maintainer security/product review before implementation.
