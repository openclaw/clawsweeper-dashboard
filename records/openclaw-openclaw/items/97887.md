## Summary

Keep open: current main still has a channel auto-reply CLI failure path where a reused Claude CLI binding can throw through `runCliAgentWithLifecycle` without clearing persisted `cliSessionIds`/`cliSessionBindings`, even though the separate command attempt path already has similar cleanup.

## What This Changes

Stale Claude CLI session IDs are not cleared after a FailoverError during an inbound cli-backend turn, causing repeated failed resumes.

## Best Possible Solution

Add the same reused-Claude-CLI binding cleanup to the channel `runCliAgentWithLifecycle` failure path, using the shared session-store helper and preserving existing missing-transcript and successful dropped-replacement behavior.

## Reproduction Assessment

Yes at source level, but not as a live Telegram/Claude CLI run in this review. The channel CLI failure path can throw through `runCliAgentWithLifecycle` and return a known failure reply without invoking the existing command-path stale binding cleanup.

## Solution Assessment

Yes, a focused cleanup in the channel CLI failure path is the best shape. The existing command-path implementation proves the helper and behavior are already accepted, so this should not become Telegram-only handling or a new config option.
