## Summary

Keep open: this PR is the focused fix candidate for the canonical group-session transcript attribution bug, and current main plus v2026.6.9 still omit sender metadata before JSONL persistence. The patch is a reasonable shared-session fix, but it intentionally makes senderId, senderName, and senderUsername durable transcript metadata, so the remaining blocker is maintainer privacy/security-boundary approval.

## What This Changes

The PR threads senderId, senderName, and senderUsername from reply session context into UserTurnInput and persists them under a __openclaw envelope in user-turn transcript JSONL when present.

## Best Possible Solution

Merge the shared transcript fix only after maintainers explicitly accept durable non-phone sender metadata in session JSONL; optional live Telegram proof would further de-risk the transport path.

## Reproduction Assessment

Yes, source-level. Telegram and the shared inbound context carry SenderId, SenderName, and SenderUsername, but current main and v2026.6.9 do not forward those fields into UserTurnInput or the persisted JSONL message builder.

## Solution Assessment

Yes for the implementation layer, with a maintainer policy gate. The shared user-turn transcript writer is the narrow owner boundary and excluding senderE164 is the safer shape, but maintainers still need to approve durable sender identity metadata.
