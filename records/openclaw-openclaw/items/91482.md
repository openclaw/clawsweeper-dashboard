## Summary

Keep this issue open: current main still exposes the bundled file-transfer `dir_list` tool without any paired-node availability signal, and the shared resolver still turns guessed values into `unknown node` errors when the node list is empty.

## What This Changes

Requests that `dir_list` stop misleading agents when no paired nodes exist and that its node parameter/error text clearly point to paired node ids from node status.

## Best Possible Solution

Keep the bundled tool, but make the shared file-transfer node path fail early with an explicit no-paired-nodes/local-tools message and tighten the `dir_list` descriptor so models do not treat it as a local directory lister.

## Reproduction Assessment

Yes for a source-level reproduction: with bundled file-transfer tools materialized and no paired nodes, `dir_list` remains registered and the shared resolver produces `node required` or `unknown node` for guessed node names. I did not run the full Windows/Ollama session live.

## Solution Assessment

Yes: the narrow maintainable fix is a shared file-transfer no-paired-nodes error plus descriptor text, because hiding the tool at discovery time would require broader async Gateway-state availability plumbing.

## Repair Work Prompt

Fix the bundled file-transfer tool UX reported in https://github.com/openclaw/openclaw/issues/91482. When `file_fetch`, `dir_list`, `dir_fetch`, or `file_write` are invoked and `listNodes()` returns no paired nodes, fail before `resolveNodeIdFromList` with a clear model-facing error that these tools operate only on paired nodes from `openclaw nodes status` and that local workspace directories should use local file/exec tools. Tighten the `dir_list`/file-transfer `node` parameter descriptions so they explicitly require an existing paired node id, display name, or IP and discourage `local`, `host`, `gateway`, or `auto`. Add focused Vitest coverage for the no-paired-nodes path, especially `dir_list`, and avoid adding config flags or hiding the tool through broader runtime discovery unless the existing synchronous plugin-tool path already supports it cleanly. Do not conflate this with the separate Windows node-host command-advertisement issue.
