Full review comments:

- **[P1] Constrain local health probes before sending auth:** `apps/macos/Sources/OpenClaw/GatewayProcessManager.swift:415-419`
  - body: `probeLocalGatewayHealth` builds its channel from `GatewayEndpointStore.localConfig()`, which can resolve custom/tailnet hosts and configured token/password auth. Because `GatewayChannelActor` sends that auth in the connect frame, automatic process-manager polling can leak credentials to a non-loopback endpoint; force this probe to loopback/no-auth or validate the endpoint before sending secrets.
  - confidence: 0.91
- **[P1] Require acknowledgement before marking it complete:** `apps/macos/Sources/OpenClaw/OnboardingView+Actions.swift:61-63`
  - body: `finish()` now persists the security acknowledgement unconditionally, but the page dots still allow jumping past page 0 without checking the toggle. A first-run user can skip to the final page, finish onboarding, and permanently mark the warning acknowledged without accepting it.
  - confidence: 0.9
- **[P1] Keep env and launchd auth in the node gate:** `apps/macos/Sources/OpenClaw/NodeMode/MacNodeModeCoordinator.swift:145-147`
  - body: The node-mode onboarding fallback only checks `openclaw.json` via `hasExistingLocalSetup(root:)`, while the actual local endpoint resolver also accepts `OPENCLAW_GATEWAY_TOKEN`, `OPENCLAW_GATEWAY_PASSWORD`, and launchd snapshot auth. Upgraded installs using those supported auth sources will be disconnected until onboarding is rerun.
  - confidence: 0.86
- **[P2] Add a recheck path for manual CLI installs:** `apps/macos/Sources/OpenClaw/OnboardingView+Pages.swift:747-748`
  - body: The copied manual install command is presented as an alternative, but this page only enables Next when `cliInstalled` is refreshed. Running the command in Terminal leaves users stuck on the page unless they leave and re-enter onboarding or use the in-app installer path.
  - confidence: 0.88
- **[P2] Preserve the injected readiness connection:** `apps/macos/Sources/OpenClaw/GatewayProcessManager.swift:380`
  - body: `waitForGatewayReady` now bypasses the `connection` property, so DEBUG tests using `setTestingConnection` no longer drive this readiness path and instead depend on a real local socket. Keep the injectable connection path or add an injectable probe so existing GatewayProcessManager tests remain deterministic.
  - confidence: 0.82
