## Summary

Keep open: current main and the latest shipped release still use the same text-block extraction path for `assistant_message_end.rawText`, related Gemini PRs do not claim to fix this exact post-tool long-final-answer case, and the report still needs raw Gemini/JSONL evidence before it can be closed or safely repaired.

## What This Changes

The issue reports that Google Gemini final answers longer than roughly 150-200 tokens can leave `assistant_message_end.rawText` empty after an `exec` tool call, causing an embedded run timeout.

## Best Possible Solution

Capture the failing Gemini raw-stream/SSE chunks, add focused Google transport and embedded-subscriber regression coverage for a tool call followed by a long final answer, then fix the provider parser or message-end fallback in that narrow path.

## Reproduction Assessment

No high-confidence live reproduction yet. Source inspection shows how `assistant_message_end.rawText` can become empty when the final assistant message lacks text blocks, but the issue still lacks the raw Gemini chunks needed to replay the exact failing path.

## Solution Assessment

Unclear until the failing chunks are captured. The narrowest maintainable fix should stay in the Google provider parser or the message-end rawText fallback, backed by a focused regression rather than a generic timeout change.
