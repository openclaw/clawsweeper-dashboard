## Summary

This should stay open: current main still has a plausible Google/Gemini parser-to-`assistant_message_end.rawText` gap, the issue is fresh and specific, and no merged PR or current test proves the reported post-tool long-response case is fixed.

## What This Changes

The issue reports that Gemini final responses longer than roughly 150-200 tokens can leave `assistant_message_end.rawText` empty after an `exec` tool call, causing embedded runs to time out.

## Best Possible Solution

Add focused Google transport and embedded-subscribe regression coverage for a tool call followed by a long Gemini final answer, then fix the parser or message-end fallback so final text survives without changing generic timeout policy.

## Reproduction Assessment

No, not at high confidence. The current source path from Gemini SSE parsing to `assistant_message_end.rawText` is clear, but live Gemini credentials or the reporter's raw JSONL are needed to reproduce the exact >150-token post-tool failure on current main.

## Solution Assessment
