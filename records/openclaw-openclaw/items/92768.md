## Label Justifications

- `P1`: A macOS OS update can force an already paired app through repeated approval/re-pairing, breaking a real gateway client workflow.
- `impact:auth-provider`: The report concerns paired-device metadata and gateway authorization for the macOS app.

## Summary

Keep open: current main and v2026.6.8 still leave `openclaw-macos` outside the same-family platform refresh path, and the issue is already paired with open PR https://github.com/openclaw/openclaw/pull/89702 as the candidate fix.

## What This Changes

The issue requests that macOS app platform version changes be treated as same-family metadata refreshes instead of forcing device re-pairing.

## Best Possible Solution

Review and land or replace the linked macOS gateway normalization PR so macOS app OS-version refresh follows the existing same-family mobile refresh behavior.

## Reproduction Assessment

Yes, by source inspection: macOS client IDs are not included in the platform-version normalization branch that suppresses metadata-upgrade pairing for same-family OS version changes. I did not run a live macOS upgrade scenario in this read-only review.

## Solution Assessment

Yes: the narrow maintainable fix is to extend the existing gateway metadata-pin normalization to macOS and cover it with resolver tests, while keeping device-family, public-key, role, and scope changes approval-bound.
