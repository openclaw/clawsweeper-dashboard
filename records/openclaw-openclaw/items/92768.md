## Label Justifications

- `P1`: The issue can break an already paired macOS app workflow after an OS update by forcing repeated metadata-upgrade approval.
- `impact:auth-provider`: The affected path is gateway paired-device identity metadata and whether a known native app must be re-approved.

## Summary

Keep open: current main and the latest release still leave `openclaw-macos` outside the same-family platform-version refresh path, and the issue is paired with an open closing PR that should be reviewed, proven, and landed or replaced before closure.

## What This Changes

The issue asks OpenClaw to stop forcing macOS app re-pairing when only the reported macOS platform version changes after an OS update.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/89702 or an equivalent gateway fix that treats `openclaw-macos` same-family OS-version changes as metadata refresh while keeping identity and permission changes approval-bound.

## Reproduction Assessment

Yes, source-reproducible: current main leaves `openclaw-macos` platform-version strings out of same-family refresh normalization, so a paired Mac app OS-version-only change reaches `metadata-upgrade` re-pairing. I did not run a live macOS reconnect in this read-only review.

## Solution Assessment

Yes: the narrow maintainable fix is to extend the existing native platform-version refresh decision to the macOS app while keeping non-native platform changes and device-family changes approval-bound.
