## Label Justifications

- `P1`: A normal macOS update can break an existing paired native app workflow by forcing metadata-upgrade re-approval.
- `impact:auth-provider`: The affected path is gateway device metadata pinning and pairing approval state for a native client.

## Summary

Keep open: current main and v2026.6.9 still only normalize iOS/iPadOS and Android platform-version refreshes, so a paired `openclaw-macos` client can still hit metadata-upgrade pairing after an OS-version-only platform change. The linked open PR is the active fix candidate and should be reviewed rather than queuing a separate repair.

## What This Changes

The issue asks for the gateway to treat macOS native app OS-version-only platform metadata changes as refreshes instead of forcing metadata-upgrade re-pairing.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/89702, or an equivalent gateway metadata-pinning fix, with focused resolver coverage and accepted macOS reconnect proof.

## Reproduction Assessment

Yes. Source inspection shows a paired `openclaw-macos` client with only a macOS platform-version string change does not hit the iOS/Android refresh normalization, so `platformMismatch` reaches `metadata-upgrade` pairing; I did not run a live macOS reconnect in this read-only review.

## Solution Assessment

Yes. Extending the existing metadata-pinning resolver to include macOS native app platform-version refreshes is the narrow server-side fix; changing only request TTLs or approval prompts would leave the stale metadata mismatch in place.
