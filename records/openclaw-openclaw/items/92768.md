## Label Justifications

- `P1`: The report describes a repeated macOS app re-approval loop after an OS update, breaking an already paired client workflow for real users.
- `impact:auth-provider`: The affected behavior is paired-device authorization metadata and the metadata-upgrade re-approval path for the macOS app.

## Summary

Keep open: current main and v2026.6.6 still normalize only iOS/iPadOS and Android platform-version labels, so `openclaw-macos` OS-version drift can still enter the `metadata-upgrade` pairing path; the linked open PR is the right implementation path and should land or be replaced before this issue closes.

## What This Changes

The issue requests that paired macOS app reconnects survive macOS platform-version-only changes without forcing metadata-upgrade re-pairing.

## Best Possible Solution

Keep this issue open and resolve it by landing https://github.com/openclaw/openclaw/pull/89702 or an equivalent gateway fix with focused macOS metadata-pinning coverage and acceptable reconnect proof.

## Reproduction Assessment

Yes. Source inspection shows `openclaw-macos` platform strings are not normalized into a same-family refresh, so a macOS version-only change can set `platformMismatch` and enter `metadata-upgrade`; I did not run a live macOS reconnect in this read-only review.

## Solution Assessment

Yes. Extending the existing server-side platform-version refresh decision to macOS is the narrowest maintainable fix; changing approval TTLs or the Mac prompt would leave the stale metadata mismatch in place.
