## Label Justifications

- `P1`: A macOS OS update can break an already paired app workflow by forcing repeated re-approval after a routine platform-version change.
- `impact:auth-provider`: The affected behavior is paired-device authorization metadata and the gateway re-approval path for the macOS app.

## Summary

Keep open: current main and v2026.6.8 still normalize same-family OS-version refreshes only for iOS/iPadOS and Android, so macOS app platform-version drift still reaches the metadata-upgrade re-pairing path. The open linked PR is the right owner for the fix path and should be reviewed or replaced before this issue closes.

## What This Changes

The issue asks OpenClaw to treat macOS app OS-version-only platform metadata changes as refreshable display metadata instead of forcing device re-pairing.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/89702 or an equivalent gateway fix with macOS metadata-pinning coverage and acceptable proof that device-family and public-key changes remain approval-bound.

## Reproduction Assessment

Yes, by source inspection: current main leaves `openclaw-macos` platform version strings out of same-family refresh normalization, so a paired Mac app OS-version-only change reaches `metadata-upgrade` re-pairing. I did not run a live macOS reconnect in this read-only review.

## Solution Assessment

Yes: extending the existing iOS/Android platform-version refresh decision to `openclaw-macos` is the narrow server-side repair. Changing request TTLs or UI prompts alone would leave the stale metadata mismatch in place.
