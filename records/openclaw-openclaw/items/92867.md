## Summary

Keep this PR open: it targets the right QMD command-normalization boundary for the linked Windows bug, but the guard is still too narrow for Windows UNC absolute paths and the supplied proof is string-level WSL output rather than an actual Windows OpenClaw/QMD run.

## What This Changes

The PR bypasses POSIX-style shell splitting for drive-letter Windows absolute `memory.qmd.command` values in the QMD memory backend resolver.

## Best Possible Solution

Land a narrow command-normalization fix at `resolveMemoryBackendConfig` that treats all Windows absolute command paths as verbatim before `splitShellArgs`, with focused regression coverage and real Windows QMD proof.

## Reproduction Assessment

Yes for source-level reproduction: current main still routes `memory.qmd.command` through a parser that consumes unquoted backslashes, matching the linked Windows report. I did not run a live Windows QMD setup in this read-only review.

## Solution Assessment

No, not yet: the fix is at the right boundary, but the current predicate omits UNC absolute paths and the proof is not a real Windows OpenClaw/QMD run. The safer solution is a complete Windows absolute-path guard plus focused regression coverage and Windows behavior proof.
