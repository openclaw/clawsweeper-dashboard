## What This Changes

The PR changes QMD memory backend command resolution so Windows drive-letter absolute `memory.qmd.command` values bypass POSIX-style shell argument splitting.

## Best Possible Solution

Land a command-normalization fix that treats all Windows absolute command paths as verbatim before shell parsing, with focused regression coverage and redacted Windows QMD behavior proof.

## Reproduction Assessment

Yes for source-level reproduction: current `main` still sends `memory.qmd.command` through `splitShellArgs`, whose unquoted backslash handling strips Windows separators. I did not run a live Windows QMD setup in this read-only review.

## Solution Assessment

No, not yet: the fix is at the right boundary, but the predicate omits UNC absolute paths and the PR body proof is not a real Windows OpenClaw/QMD run. The safer path is complete Windows absolute-path coverage plus focused regression tests and Windows behavior proof.
