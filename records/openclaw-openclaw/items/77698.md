## Summary

Keep open: current main still has the doctor false-warning path, but this PR leaves the linked exec SecretRef reproduction on the same stale warning text and lacks after-fix real behavior proof.

## What This Changes

The PR changes `runGatewayAuthHealth()` to resolve non-exec `gateway.auth.token` SecretRefs before warning, while leaving exec SecretRefs on the existing warning path.

## Best Possible Solution

Use a doctor-specific SecretRef check that resolves env/file refs, avoids executing exec providers without explicit policy, reports skipped exec refs as unverified rather than unavailable, and includes redacted real `openclaw doctor` proof.

## Reproduction Assessment

Yes, source-reproducible: configure local token auth with `gateway.auth.token` as a resolvable exec SecretRef and run `openclaw doctor`; current main detects the SecretRef but `resolveGatewayAuth()` ignores SecretRef object values and reaches the unavailable warning branch. I did not run a live exec provider in this read-only review.

## Solution Assessment

No. Resolving env/file refs before warning is the right direction, but this patch deliberately skips exec refs and emits the same unavailable diagnostic for the exact linked reproduction; the safer solution is a policy-aware exec path or an explicit unverified diagnostic.
