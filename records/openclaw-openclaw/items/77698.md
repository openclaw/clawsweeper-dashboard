## Summary

Keep open: the updated patch addresses the earlier command-execution concern by skipping exec SecretRefs, but it still preserves the stale unavailable warning for the linked resolved-exec repro and lacks required real behavior proof and changelog coverage.

## What This Changes

The PR updates `runGatewayAuthHealth()` to resolve env/file `gateway.auth.token` SecretRefs before warning, while leaving exec SecretRefs on the existing warning path.

## Best Possible Solution

Use a doctor-specific SecretRef path that resolves env/file refs, avoids executing exec refs without an explicit policy, reports exec refs as unverified rather than unavailable when not executed, adds regression coverage and a changelog entry, and verifies the result with real doctor output.

## Reproduction Assessment

Yes. Current main is source-reproducible: `runGatewayAuthHealth()` sends raw SecretRef-backed config to `resolveGatewayAuth()`, and that resolver intentionally ignores SecretRef objects before the doctor warning decision.

## Solution Assessment

No. Resolving env/file refs is the right direction, but preserving the old exec `currently unavailable` warning leaves the linked resolved-exec case unsolved; a policy-aware exec path or an explicit unverified diagnostic is safer.
