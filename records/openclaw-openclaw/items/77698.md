## Summary

Keep open: the linked doctor false warning is valid, but this patch would make `openclaw doctor` execute exec-backed SecretRef providers without an explicit allow-exec policy, and the external PR still lacks after-fix real behavior proof.

## What This Changes

The PR resolves `gateway.auth.token` SecretRefs inside `runGatewayAuthHealth()` before emitting the gateway-token-unavailable warning and includes the unresolved ref reason when resolution fails.

## Best Possible Solution

Use a doctor-specific, policy-aware SecretRef path that avoids persisting plaintext, does not execute exec providers without explicit consent, adds regression coverage for resolved and unresolved refs, and documents the shipped behavior in the changelog.

## Reproduction Assessment

Yes. Current main is source-reproducible: `runGatewayAuthHealth()` sends raw SecretRef-backed config to `resolveGatewayAuth()`, and that resolver intentionally ignores SecretRef objects before the doctor warning decision.

## Solution Assessment

No. Reusing the generic token resolver inside `doctor` is too broad because it can execute exec SecretRef providers without an allow-exec policy; the safer fix needs doctor-specific resolution rules or an explicit opt-in path.

## Review Findings

Overall correctness: patch is incorrect

Full review comments:

- **[P1] Gate exec SecretRef resolution before doctor runs it:** `src/flows/doctor-health-contributions.ts:172-176`
