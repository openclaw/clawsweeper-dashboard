## Summary

Keep open: the PR improves the env/file SecretRef path, but it still leaves the linked exec-SecretRef report with the same misleading unavailable warning and has no after-fix real behavior proof.

## What This Changes

The branch updates `runGatewayAuthHealth()` to resolve non-exec `gateway.auth.token` SecretRefs before emitting the unavailable warning and to include unresolved env/file reasons.

## Best Possible Solution

Resolve safe env/file Gateway token SecretRefs only after token auth is known to require a token, avoid implicit exec execution, and report exec refs as unverified or opt-in rather than unavailable.

## Reproduction Assessment

Yes. Source inspection plus reporter evidence gives a high-confidence path: configure local token auth with `gateway.auth.token` as a resolvable exec SecretRef, confirm it with `openclaw secrets audit --allow-exec`, then run `openclaw doctor`; current main reaches the warning because `resolveGatewayAuth()` ignores SecretRef values.

## Solution Assessment

No. Resolving env/file refs before warning is the right direction, but preserving the generic unavailable diagnostic for skipped exec refs leaves the linked exec-SecretRef report unresolved.
