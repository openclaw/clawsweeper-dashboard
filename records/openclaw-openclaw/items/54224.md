## Summary

Keep open. Current main already covers Slack channel and thread history through the generic `read` action with user-token-preferred reads, but Slack message search is still absent from current main and is being handled by overlapping open implementation PRs that need maintainer/security validation.

## What This Changes

Request to expose Slack read-only message retrieval actions for history, search, and thread summaries in user-token read-only Slack workflows.

## Best Possible Solution

Land one Slack-plugin-owned read-only retrieval surface: keep `read` as the history/thread solution, add a least-privilege `search` action only through the chosen PR path, and cover discovery, dispatch, runtime, CLI/docs, and live Slack proof.

## Reproduction Assessment

Not applicable as a bug reproduction. Source inspection gives a high-confidence gap check: current main has Slack `read`/`readMessages` for history and threads but no Slack search action, dispatch, runtime, or CLI surface.

## Solution Assessment

Unclear until maintainers choose between the focused search PR and the broader action-parity/RTS direction. The existing `read` path is the right solution for history/thread reads, while search should be added once with explicit least-privilege Slack API gating.
