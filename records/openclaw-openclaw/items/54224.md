## Summary

Keep open. Current main covers the history and thread-read parts through Slack's existing `read`/`readMessages` path with user-token-preferred read behavior, but Slack message search is still absent; the supplied timeline identifies open PR #65691 as the active implementation candidate for the remaining work.

## What This Changes

Request to expose read-only Slack message history, search, and thread retrieval actions for user-token read-only workflows without enabling Slack write capabilities.

## Best Possible Solution

Review and either land or close #65691. The desired end state is a Slack-owned generic `search` message action backed by Slack `search.messages`, advertised only for usable user-token read credentials, wired through the existing message action contract and CLI, documented in the Slack channel docs, and covered by focused discovery, dispatch, runtime, formatting, and least-privilege tests; the existing `read` action can remain the channel history and thread history solution.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: current Slack action discovery, dispatch, exports, docs, and runtime lack a Slack search handler, while the existing read path covers history/thread reads.

## Solution Assessment

Yes. Keeping this issue open for the linked search PR is the narrowest maintainable path; `read` should continue to own history/thread retrieval, and Slack `search.messages` should be added as a separate least-privilege read action rather than mixed with write gates or the separate RTS request in #38716.
