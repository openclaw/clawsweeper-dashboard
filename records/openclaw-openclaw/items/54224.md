## Summary

Keep open. Current main covers Slack channel history and thread reads through the existing `read`/`readMessages` path with user-token-preferred reads, but Slack message search is still not implemented or advertised. The related open PR #65691 is the active implementation candidate for the remaining `search.messages` work.

## What This Changes

Request to expose Slack read-only history, message search, and thread-history retrieval actions for user-token read-only workflows without enabling Slack writes.

## Best Possible Solution

Keep the existing Slack `read` action as the channel-history/thread-history solution, and land or replace #65691 with a Slack-owned `search` message action backed by Slack `search.messages`, exposed only when user-token read credentials are usable and covered by focused discovery, dispatch, CLI, docs, runtime, and least-privilege tests.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: current Slack discovery/runtime/CLI/docs do not expose Slack search, while the existing `read` path covers channel history and thread reads.

## Solution Assessment

Yes. Keeping this open for #65691 is the narrowest maintainable path; adding Slack search inside the existing message-action contract is preferable to a new core seam or mixing it with the separate RTS/action-token request in #38716.
