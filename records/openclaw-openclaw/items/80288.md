## Summary

Keep open: current main still intentionally sets the Codex stdio app-server child `HOME` to the per-agent Codex native home, and the requested `homeDir`/`OPENCLAW_CODEX_APP_SERVER_HOME_DIR` override is not present. The related implementation attempts at https://github.com/openclaw/openclaw/pull/80315 and https://github.com/openclaw/openclaw/pull/80318 are closed unmerged, while the older related `REAL_HOME` thread does not make dotfile-backed tools like `gh` use the real `HOME`.

## What This Changes

The issue requests a trusted-local Codex app-server config/env override that keeps `CODEX_HOME` isolated while setting the spawned child process `HOME` to the real OS user home.

## Best Possible Solution

Define one maintainer-approved trusted-local operator-home contract for the Codex plugin, keep default isolation unchanged, and update config schema, manifest UI hints, docs, regression tests, and live proof around that contract.

## Reproduction Assessment

Yes. Source inspection and existing tests give a high-confidence path: local stdio Codex app-server starts currently receive `<agentDir>/codex-home/home` as `HOME` unless an internal start-options env override is already supplied.

## Solution Assessment

Unclear. A gated `homeDir` override is plausible and narrower than changing the default, but maintainers need to choose the HOME/REAL_HOME isolation contract before implementation.
