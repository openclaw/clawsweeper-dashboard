## Label Justifications

- `P1`: The report describes a real user-facing Telegram/Codex workflow where one inbound message can be duplicated into a new turn or end in a generic failure after timeout.
- `impact:message-loss`: The failure duplicates user-visible message handling by replaying the same inbound transcript/media as if it were sent again.

- `impact:auth-provider`: The observed replay path is triggered through auth/profile rotation and configured model fallback handling after a timeout.

## Summary

Keep open: current main and v2026.5.20 still have a source-visible path where an assistant timeout can rotate auth/profile state and immediately retry the same run loop, while the report supplies real Telegram evidence of duplicate user-message replay.

## What This Changes

The issue reports that Codex app-server timeouts can combine retry, auth/profile rotation, and model fallback handling to resubmit the same Telegram voice-note transcript/media into a fresh Codex turn.

## Best Possible Solution

Make timeout retries replay-safe in the embedded runner: when an aborted/timed-out attempt has non-replay-safe metadata or visible side effects, do not resubmit the original user input; surface a single recovery/failure path or an explicit continuation marker, and document the Codex app-server timeout boundary if it is separate from agent timeoutSeconds.

## Reproduction Assessment

Yes, source-reproducible with high confidence: current main still classifies assistant timeouts for profile rotation, returns retry after advanceAuthProfile, and continues the run loop from params.prompt. I did not run the full 10-minute Telegram/Codex live reproduction in this read-only pass.

## Solution Assessment

Partly: the suggested replay suppression matches existing replayInvalid metadata and the runner's side-effect checks. The implementation should stay narrow so replay-safe auth/fallback retries still work, while unsafe timeout retries stop or resume with an explicit continuation signal instead of reusing the original user input.
