- `P1`: The reported release behavior can duplicate a user's inbound Telegram message into a fresh Codex turn after timeout/fallback, breaking a real channel workflow.
- `impact:message-loss`: The issue is about duplicated/replayed channel messages becoming visible to the user as if the same inbound message was sent twice.

## Summary

Close: current main contains a targeted Codex app-server recovery guard that prevents unsafe replay, suppresses duplicate user persistence on safe retry, and has regression tests; the fix is not in v2026.5.20 yet.

## What This Changes

Report requests preventing Codex app-server timeout/fallback retry from replaying the same Telegram user input as a fresh turn.

## Best Possible Solution

Keep the current main Codex app-server recovery guard and regression tests, then carry the fix into the next release for affected v2026.5.20 users.

## Reproduction Assessment

Yes for the reported shipped-release path: the issue includes concrete timeout/fallback steps and current source shows the matching retry/replay boundary. I did not run a live 10-minute Telegram/Codex timeout against current main; current main now has source-level guards covering that path.

## Solution Assessment

Yes. The current-main solution is bounded and maintainable: safe stdio client-close retries are allowed once with duplicate user persistence suppressed, while unsafe output/tool/idle-timeout cases fail instead of replaying the user message.
