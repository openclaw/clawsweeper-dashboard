## Summary

Keep open: this is a real security fix direction with sufficient proof, but the latest diff still ties the mutex-backed bootstrap-token verifier to the optional, loopback-exempt auth limiter, leaving default and loopback paths exposed.

## What This Changes

Adds a bootstrap-token auth rate-limit scope, gates pre-auth bootstrap-token verification through it, records/resets that bucket, adds unit and WebSocket integration coverage, and updates the changelog.

## Best Possible Solution

Use a dedicated bootstrap-token limiter constructed unconditionally before verifyDeviceBootstrapToken, with loopback non-exempt behavior or an explicitly approved equivalent, and cover no-config, default-loopback, browser-origin, and configured remote cases.

## Reproduction Assessment

Yes. Source inspection on current main shows a signed device identity with auth.bootstrapToken reaches verifyDeviceBootstrapToken before any bootstrap-token limiter, and the PR body supplies terminal proof for the configured limiter case.

## Solution Assessment

No. Mirroring the optional device-token limiter is too narrow for this mutex-backed pre-auth verifier; the maintainable fix is an always-present bootstrap-token limiter, or an explicitly approved narrower security policy.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

- **[P1] Use an always-on bootstrap-token limiter:** `src/gateway/server/ws-connection/auth-context.ts:167`
