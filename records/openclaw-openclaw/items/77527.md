## Summary

Keep open. The PR correctly targets a real pre-auth bootstrap-token DoS path, but the current patch only gates verification through the optional configured auth limiter, leaving the default non-browser and loopback-exempt paths able to reach the mutex-backed verifier.

## What This Changes

The PR adds a bootstrap-token auth-rate-limit scope around WebSocket connect bootstrap-token verification, plus regression tests and a changelog entry.

## Best Possible Solution

Create a bootstrap-token-only limiter unconditionally with loopback non-exempt behavior, pass it separately to the WS connect auth path, and cover unconfigured plus default-loopback cases without changing shared-secret or device-token limiter semantics.

## Reproduction Assessment

Yes. Source inspection shows current main accepts a valid signed device identity plus auth.bootstrapToken and reaches verifyDeviceBootstrapToken before any bootstrap-token limiter; the PR discussion also reports no-config and default-loopback attempts still hit verify under the patch.

## Solution Assessment

No. Reusing the optional authRateLimiter is not the best fix for a mutex DoS because the default unconfigured and loopback-exempt paths remain reachable; a dedicated always-on bootstrap limiter is the narrower maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Use an always-on limiter for bootstrap token checks:** `src/gateway/server/ws-connection/auth-context.ts:167`

Please pass a bootstrap-token-specific limiter that is created unconditionally with loopback non-exempt behavior.
