## Summary

Keep open: the branch narrows the configured and browser-origin attack path and now includes real terminal proof, but the diff still leaves unconfigured and default-loopback non-browser bootstrap-token verifies able to enter the mutex-backed verifier.

## What This Changes

The PR adds a bootstrap-token auth rate-limit scope, gates bootstrap-token verification with the existing auth limiter, records/resets that bucket, adds unit and WebSocket integration coverage, and updates the changelog.

## Best Possible Solution

Use a dedicated bootstrap-token limiter constructed unconditionally before verifyDeviceBootstrapToken, with loopback non-exempt behavior for this expensive pre-auth path and coverage for unconfigured, default-loopback, browser-origin, and configured remote cases.

## Reproduction Assessment

Yes. Source inspection on current main shows a signed device identity plus auth.bootstrapToken reaches verifyDeviceBootstrapToken before any limiter, and the PR discussion gives concrete forged-handshake counts for the remaining unprotected cases.

## Solution Assessment

No. Mirroring the optional device-token limiter is too narrow for a mutex-backed pre-auth verifier; the maintainable fix is a bootstrap-token-specific limiter that is always present and not loopback-exempt for this path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Use an always-on limiter for bootstrap-token verifies:** `src/gateway/server/ws-connection/auth-context.ts:167`
  - body: Because this branch only checks an existing `params.rateLimiter`, deployments without `gateway.auth.rateLimit` and loopback clients under the default `exemptLoopback:true` still call the mutex-backed `verifyDeviceBootstrapToken`. Add a bootstrap-token limiter that is always constructed and loopback-non-exempt for this verify path, then cover no-config and default-loopback cases.
