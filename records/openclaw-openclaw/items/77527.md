## Summary

Keep open: the PR targets a real pre-auth gateway DoS path, but the current patch still leaves bootstrap-token verification unthrottled when the normal auth limiter is absent or loopback-exempt, which needs maintainer/security decision before merge.

## What This Changes

The PR adds a bootstrap-token auth rate-limit scope, gates bootstrap-token verification through the existing auth limiter, records/resets that bucket, adds unit and in-process WebSocket tests, and updates the changelog.

## Best Possible Solution

Use a dedicated bootstrap-token limiter constructed unconditionally before verifyDeviceBootstrapToken, with loopback non-exempt behavior for this pre-auth mutex path and coverage for no-config, default-loopback, browser-origin, and configured remote cases.

## Reproduction Assessment

Yes. Source inspection on current main shows a signed device identity with auth.bootstrapToken reaches verifyDeviceBootstrapToken before any bootstrap limiter, and that verifier serializes under withLock; the PR body also supplies terminal proof for the configured-limiter path.

## Solution Assessment

No. Mirroring the optional device-token limiter is too narrow for this mutex-backed pre-auth verifier; the safer maintainable fix is a bootstrap-token-specific limiter that is always present and loopback-non-exempt, or an explicitly approved equivalent.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Use an always-on bootstrap-token limiter:** `src/gateway/server/ws-connection/auth-context.ts:167-183`
  - body: Because this gate only runs when `params.rateLimiter` is supplied, deployments without `gateway.auth.rateLimit` and loopback clients with the default `exemptLoopback:true` still enter `verifyDeviceBootstrapToken`. Use an always-created bootstrap-token limiter for this mutex-backed pre-auth path and add no-config/default-loopback coverage.
  - confidence: 0.9
