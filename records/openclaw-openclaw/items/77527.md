## Summary

Keep open: the PR targets the right pre-auth bootstrap-token mutex DoS path, but the current patch still depends on the optional auth limiter, leaving default unconfigured and loopback-exempt requests able to enter the expensive verify path; it also needs after-fix real behavior proof before merge.

## What This Changes

The PR adds a bootstrap-token auth-rate-limit scope, gates bootstrap-token verification, adds unit and WebSocket integration tests, and updates CHANGELOG.md.

## Best Possible Solution

Use a bootstrap-token-only limiter created unconditionally with exemptLoopback:false around the expensive verify path, add coverage for unconfigured and default loopback paths, and keep the changelog entry.

## Reproduction Assessment

Yes. Source inspection shows a signed device identity with auth.bootstrapToken reaches verifyDeviceBootstrapToken before any limiter on current main, and the author also confirmed the current patch still lets unconfigured and loopback-exempt attempts hit verify.

## Solution Assessment

No. The current diff mirrors the optional device-token limiter envelope, but this path protects a process-global mutex, so the narrow maintainable fix is a dedicated always-on bootstrap limiter.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Use an always-on limiter for bootstrap-token checks:** `src/gateway/server/ws-connection/auth-context.ts:167`
  - body: Because this check runs only when `params.rateLimiter` exists, deployments without `gateway.auth.rateLimit` and default loopback-exempt non-browser clients still enter `verifyDeviceBootstrapToken`, the mutex-backed fs path this patch is meant to protect. Please pass a bootstrap-token-specific limiter created unconditionally with loopback non-exempt behavior.
  - confidence: 0.9
