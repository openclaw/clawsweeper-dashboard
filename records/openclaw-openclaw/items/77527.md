## Summary

Keep open: the PR improves configured and browser-origin bootstrap-token attempts, but the proposed gate still depends on the optional auth limiter and leaves default non-browser and loopback-exempt paths reaching the mutex-backed verify work.

## What This Changes

The branch adds a bootstrap-token auth rate-limit scope, gates bootstrap-token verification with the existing auth limiter, adds unit and WebSocket integration coverage, and updates the changelog.

## Best Possible Solution

Use a dedicated bootstrap-token limiter constructed unconditionally with loopback non-exempt behavior before the expensive verify path, then cover unconfigured, default-loopback, browser-origin, and configured remote cases.

## Reproduction Assessment

Yes. Source inspection on current main shows a signed device identity plus auth.bootstrapToken reaches verifyDeviceBootstrapToken before any limiter, and the PR author confirmed the remaining no-config and default-loopback paths still hit verify on the branch.

## Solution Assessment

No. Mirroring the optional device-token limiter is too narrow for this mutex-backed path; the maintainable fix is a bootstrap-token-specific limiter that is always present and not loopback-exempt.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Use an always-on limiter for bootstrap-token verifies:** `src/gateway/server/ws-connection/auth-context.ts:167`
  - body: Because this branch only checks `params.rateLimiter`, deployments without `gateway.auth.rateLimit` and default loopback-exempt non-browser clients still enter `verifyDeviceBootstrapToken`, the global lock plus filesystem path this change is meant to protect. Please use a dedicated bootstrap-token limiter that is created unconditionally and does not exempt loopback.
