## Summary

Keep open: this addresses a real mutex-backed pre-auth DoS for configured and browser-origin traffic, but it still leaves no-config and default loopback paths able to enter `verifyDeviceBootstrapToken`; security-owner review should require an always-on bootstrap-specific limiter before merge.

## What This Changes

This PR adds a bootstrap-token auth rate-limit scope, gates bootstrap-token verification through the existing auth limiter, records/resets the new bucket, adds unit and WebSocket integration coverage, and updates `CHANGELOG.md`.

## Best Possible Solution

Add a dedicated bootstrap-token limiter constructed unconditionally, use it before `verifyDeviceBootstrapToken` with loopback exemption disabled, and cover no-config, default-loopback, browser-origin, and configured remote cases.

## Reproduction Assessment

Yes. Source inspection on current main shows a signed device identity plus `auth.bootstrapToken` reaches `verifyDeviceBootstrapToken`, and the verifier is protected by a module-level lock; the contributor also confirmed the PR branch still lets no-config and default-loopback attempts hit verify.

## Solution Assessment

No. Mirroring the optional device-token limiter is too narrow for this mutex-backed pre-auth verifier; the safer maintainable fix is an always-present bootstrap-token-specific limiter that does not honor the general loopback exemption.
