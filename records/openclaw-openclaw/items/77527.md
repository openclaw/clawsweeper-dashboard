## Summary

Keep open: the PR addresses a real pre-auth bootstrap-token mutex DoS path, but the current patch still leaves default/unconfigured and loopback-exempt non-browser paths able to reach the mutex-backed verify call, and it still needs after-fix real behavior proof.

## What This Changes

The PR adds a bootstrap-token auth rate-limit scope, gates bootstrap-token verification, adds unit and WebSocket integration tests, and updates the changelog.

## Best Possible Solution

Use a bootstrap-token-only limiter that is created unconditionally with loopback non-exempt behavior around the expensive verify path, then cover unconfigured and default-loopback paths with tests and keep the changelog entry.

## Reproduction Assessment

Yes. Source inspection on current main shows a signed device identity plus auth.bootstrapToken reaches verifyDeviceBootstrapToken before any limiter, and that verify path is protected by a module-wide lock with pairing-state I/O; the author also reproduced the remaining unconfigured and default-loopback gaps on the PR.

## Solution Assessment

No. The current diff mirrors the optional device-token limiter envelope, but this path protects a process-global mutex, so the narrow maintainable fix is a dedicated always-on bootstrap-token limiter with no loopback exemption for this scope.
