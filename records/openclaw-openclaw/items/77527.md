## Summary

Keep open: the PR identifies a real pre-auth bootstrap-token mutex DoS and adds useful coverage, but the current diff still leaves default unconfigured and default loopback-exempt paths able to reach the mutex-serialized verify path.

## What This Changes

The PR adds a bootstrap-token auth rate-limit scope, gates bootstrap-token verification, adds unit and WebSocket integration tests, and records a changelog entry.

## Best Possible Solution

Add a bootstrap-token-only limiter that is created unconditionally with loopback non-exempt behavior, and use it only around the expensive bootstrap-token verify path while preserving existing shared-secret and device-token rate-limit semantics.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main accepts a signed device identity plus auth.bootstrapToken, then enters verifyDeviceBootstrapToken before any bootstrap-token limiter; the PR also describes an integration test that exercises this path.

## Solution Assessment

No, not as currently written. Reusing the optional authRateLimiter is too narrow for a mutex DoS because the unconfigured and default loopback-exempt paths remain reachable; a dedicated always-on bootstrap limiter is the safer minimal fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Use an always-on limiter for bootstrap token checks:** `src/gateway/server/ws-connection/auth-context.ts:167`
  - body: This gate only runs when params.rateLimiter exists. In the default server path, rateLimiter is undefined unless gateway.auth.rateLimit is configured, and non-browser loopback clients are exempt by default, so those requests still enter verifyDeviceBootstrapToken, the mutex-serialized fs read/write path this PR is meant to protect. Please pass a bootstrap-token-specific limiter that is created unconditionally with loopback non-exempt.
  - confidence: 0.9
