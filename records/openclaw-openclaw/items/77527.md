## Summary

Keep open: the PR addresses the configured and browser-origin bootstrap-token path, but the latest diff still leaves the default no-rate-limit path and default loopback-exempt path able to enter the mutex-backed verifier.

## What This Changes

The PR adds a bootstrap-token auth rate-limit scope, gates bootstrap-token verification through the existing auth limiter, records/resets that bucket, adds unit and WebSocket integration coverage, and updates the changelog.

## Best Possible Solution

Use a dedicated bootstrap-token limiter constructed unconditionally before verifyDeviceBootstrapToken, with loopback non-exempt behavior for this expensive pre-auth path and coverage for no-config, default-loopback, browser-origin, and configured remote cases.

## Reproduction Assessment

Yes. Source inspection on current main shows a signed device identity with auth.bootstrapToken reaches verifyDeviceBootstrapToken before any bootstrap limiter, and that verifier uses the shared withLock-backed pairing-state path.

## Solution Assessment

No. Mirroring the optional device-token limiter is too narrow for this mutex-backed pre-auth verifier; the maintainable fix is a bootstrap-token-specific limiter that is always present and not loopback-exempt, or an explicitly approved equivalent.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Use an always-on limiter for bootstrap-token verifies:** `src/gateway/server/ws-connection/auth-context.ts:167`
