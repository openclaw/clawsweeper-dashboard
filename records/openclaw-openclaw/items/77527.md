## Summary

Keep open: this is a useful security hardening PR, but the current patch only gates bootstrap-token verification when the optional normal auth limiter is present, so default no-config and default loopback paths can still reach the mutex-backed verifier.

## What This Changes

The PR adds a bootstrap-token auth rate-limit scope, gates pre-auth bootstrap-token verification, records/resets bucket state, adds gateway tests, and updates the changelog.

## Best Possible Solution

Use a dedicated bootstrap-token limiter before verifyDeviceBootstrapToken that is always constructed and loopback-non-exempt, or an explicitly approved equivalent, with coverage for no-config, default-loopback, browser-origin, and configured remote cases.

## Reproduction Assessment

Yes. Source inspection shows a signed device handshake with auth.bootstrapToken reaches verifyDeviceBootstrapToken before any bootstrap limiter on current main, and that verifier serializes through withLock; the PR body also supplies configured-limiter runtime proof.

## Solution Assessment

No. Mirroring the optional device-token limiter is a useful partial fix, but the mutex-backed bootstrap-token verifier needs an always-present preflight limiter or an explicitly approved security-policy alternative.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Use an always-on limiter for bootstrap-token verifies:** `src/gateway/server/ws-connection/auth-context.ts:167`
  - body: Because this gate only runs when params.rateLimiter is supplied, deployments without gateway.auth.rateLimit still enter verifyDeviceBootstrapToken. The normal configured limiter also exempts loopback by default, so use an always-created, loopback-non-exempt bootstrap-token limiter or an explicitly approved equivalent.
  - confidence: 0.9
