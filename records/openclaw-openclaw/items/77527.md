## Summary

Keep open: current main still sends pre-auth bootstrap-token attempts into the mutex-protected bootstrap verifier without a bootstrap-token limiter, and the PR appears to address that path without a blocking review finding. It still needs maintainer/secops handling because it changes gateway pre-auth security behavior and the provided GitHub state reports the branch as unmergeable.

## What This Changes

The PR adds a bootstrap-token auth rate-limit scope, gates pre-auth bootstrap-token verification with check/record/reset behavior, adds unit and WebSocket integration coverage, and records a changelog fix.

## Best Possible Solution

Land a rebased version that rate-limits bootstrap-token verification before entering `verifyDeviceBootstrapToken`, keeps the regression coverage, and receives the normal gateway auth security review.

## Reproduction Assessment

Yes. Source inspection shows a valid device-signed connect frame with `auth.bootstrapToken` reaches `verifyDeviceBootstrapToken` on current main before any bootstrap-token-specific limiter; the PR’s integration test describes the same handshake path, though I did not run it in read-only mode.

## Solution Assessment

Yes. A separate bootstrap-token scope using the existing rate-limiter check/record/reset pattern is the narrowest maintainable fix for this pre-auth mutex-saturation path, with the remaining work limited to rebase, CI, and security review.
