## Summary

Keep this PR open: it targets a real macOS LaunchAgent restart bug, but the patch is not safe to merge because the new intent write can leave stale restart intent after pre-signal failures and can overwrite explicit restart intent options before the gateway SIGTERM handler consumes them.

## What This Changes

The PR moves gateway restart-intent writing into the macOS launchd restart path, adds an optional restartIntent service-control argument, and updates launchd tests for the extra runtime probe and intent file contract.

## Best Possible Solution

Keep the launchd owner-boundary fix, but write or preserve the intent only at the signal boundary, clear it on pre-signal failures, and pass restartIntent through every service.restart call that can reach restartLaunchAgent.

## Reproduction Assessment

Yes for a source-level reproduction path: current main shows SIGTERM routes through consumeGatewayRestartIntentPayloadSync, and launchd restart has pre-kickstart busy-port failure coverage. I did not run a live macOS LaunchAgent repro in this read-only review.

## Solution Assessment

No: moving the intent closer to launchd is a good owner-boundary direction, but this patch writes too early and does not preserve the richer restartIntent payload. The safer fix is to write/clear at the signal boundary and pass the intent through all restart callers.

Pass the intent through from lifecycle-core and the stale-pid retry path, then cover that the final file retains those fields.
