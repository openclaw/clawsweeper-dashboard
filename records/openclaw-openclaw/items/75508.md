## Summary

Keep open. Current main still matches the reported silent CLI path: `openclaw security audit` resolves SecretRefs and awaits the full audit before emitting human or JSON output, while plugin collector fallback still uses uncached metadata loading without an explicit manifest-only or runtime-deps-free audit mode. The related #75456 cluster is relevant, but this issue adds useful non-JSON, `--deep`, and direct `runSecurityAudit(...)` contrast that should stay attached until a fix lands.

## What This Changes

The issue reports that `openclaw security audit` and `openclaw security audit --deep` can appear to hang silently from the CLI even though direct installed `runSecurityAudit(...)` calls complete.

## Best Possible Solution

Ship a narrow audit-boundary fix that keeps plugin security collector coverage while making collector discovery read-only and bounded, avoids staging bundled runtime dependencies for discovery-only audit work, and adds visible diagnostics or progress for slow CLI preflight/audit phases. Fold the closely related #75456 hidden extension directory filtering only if it stays small and testable.

## Reproduction Assessment

Yes for the observable no-output path: current source shows the CLI waits through SecretRef resolution and the complete `runSecurityAudit` call before emitting output. The long hang duration itself still needs an install-dependent timed reproduction during the fix.

## Solution Assessment

Yes, the best direction is a targeted security-audit and metadata-loader repair, not a broad plugin-loader rewrite. The safer implementation is an explicit audit/read-only discovery path or runtime-deps opt-out with focused regression tests proving collector coverage is preserved.
