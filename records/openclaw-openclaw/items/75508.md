## What This Changes

The issue requests that `openclaw security audit` and `openclaw security audit --deep` avoid silently appearing hung on the CLI when the underlying `runSecurityAudit(...)` path can complete.

## Best Possible Solution

Keep #75456 as the canonical tracker and solve the audit boundary once there: make plugin security collector discovery avoid runtime-deps/module loading where possible, add visible or bounded diagnostics for slow audit phases, and filter hidden/backup installed-plugin directories in the audit scans.

## Reproduction Assessment

Yes for the source-backed no-output path: current main waits for SecretRef resolution and `runSecurityAudit(...)` before emitting normal audit output, and the collector fallback can still enter plugin metadata/runtime loading. The exact hang duration is install- and filesystem-dependent.

## Solution Assessment

Yes. Closing this into #75456 is the best maintenance path because both reports point at the same audit CLI/plugin-loading boundary, and one focused repair can preserve this issue's direct-`runSecurityAudit(...)` contrast without splitting the fix across two trackers.
