## Summary

Keep open. Current main still leaves the reported security-audit CLI boundary unresolved: the command can remain silent while SecretRef preflight and the full audit complete, and the plugin collector fallback still enters uncached plugin metadata loading without manifest-only/read-only loader flags. The closely related #75456 report should be fixed in the same cluster, but this issue preserves a distinct non-JSON/`--deep` no-output symptom and the direct-`runSecurityAudit(...)` contrast.

## What This Changes

The issue requests that `openclaw security audit` and `openclaw security audit --deep` stop appearing to hang silently when the installed direct `runSecurityAudit(...)` path completes.

## Best Possible Solution

Land a narrow audit-path fix: make CLI preflight and collector discovery bounded and visible, discover audit collectors without staging bundled runtime deps when possible, preserve plugin security coverage, and include #75456’s hidden backup-directory cleanup if the same PR owns the full security-audit hang cluster.

## Reproduction Assessment

Yes for the source-level no-output path: current main performs SecretRef preflight and the full audit before printing human-readable output, and the audit collector fallback can still trigger uncached plugin metadata/module loading. The exact long hang should be reproduced with a controlled slow-loader or packaged-install fixture during the fix.

## Solution Assessment

Yes, the best direction is a focused audit/metadata-loader change rather than a broad plugin-loader rewrite. The safer alternative to disabling collectors is to use manifest-only or explicit read-only collector discovery first, then load runtime collector modules only when actually needed and with progress/diagnostics around slow phases.
