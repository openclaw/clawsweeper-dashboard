## Summary

Keep open. Current main still has the plausible silent CLI path described in the issue: the command does not emit output until SecretRef resolution and the full audit finish, and plugin collector fallback still reaches uncached metadata loading without explicitly requesting manifest-only/runtime-deps-free discovery. The related #75456 report is a close repair cluster match, but this issue adds useful non-JSON/`--deep` and direct-`runSecurityAudit(...)` contrast that should stay attached until a fix lands.

## What This Changes

The issue reports that `openclaw security audit` and `openclaw security audit --deep` can sit silently on the CLI while direct `runSecurityAudit(...)` calls complete, likely through the plugin collector/runtime loading path.

## Best Possible Solution

Ship a narrow audit-boundary fix that keeps collector coverage but avoids staging bundled runtime deps just to discover collectors where possible, adds visible/bounded diagnostics for slow CLI preflight or audit phases, and folds the closely related #75456 hidden-extension-dir cleanup only if it remains small and testable.

## Reproduction Assessment

Yes for the observable no-output path: source shows the CLI waits through SecretRef resolution and `runSecurityAudit` before any output. The exact long hang duration is still install-dependent and should be confirmed with a timed fixture or local reproduction while implementing the fix.

## Solution Assessment

Yes, a targeted security-audit/metadata-loader change is the best direction; a broad plugin-loader rewrite is not needed to make this command bounded and visible. The safer alternative is to make the audit caller explicitly request read-only/manifest-only discovery or a runtime-deps opt-out, then add focused regression tests.
