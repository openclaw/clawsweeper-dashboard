## Summary

Keep open. Current main still has a plausible no-output CLI path for this report: `security audit` waits for command secret resolution and the full audit before printing anything, and the default audit collector fallback can still enter uncached plugin metadata/module loading with bundled runtime-dependency staging enabled. The related #75456 report already tracks a very close security-audit hang boundary, but this issue adds a useful non-JSON/`--deep` CLI symptom and direct-`runSecurityAudit` contrast that should be preserved in the same repair cluster rather than closed as a duplicate.

## What This Changes

The issue asks for `openclaw security audit` and `openclaw security audit --deep` to complete, fail clearly, or show useful progress instead of sitting silently when the CLI/plugin-loading path stalls.

## Best Possible Solution

Keep this issue open and fix the security-audit CLI boundary narrowly: make command preflight and plugin collector discovery bounded and visible, avoid staging bundled runtime deps just to discover audit collectors, preserve existing collector coverage, and link the implementation to #75456 and the broader plugin-loader stall reports.

## Reproduction Assessment

Yes for the observable no-output path from source: the CLI prints nothing until SecretRef resolution and `runSecurityAudit` return, and current collector discovery can still enter uncached plugin module/runtime-deps loading. The exact wall-clock hang remains install-dependent and should be confirmed with a focused regression test or timed fixture during the fix.

## Solution Assessment

Yes, the best fix is a narrow audit-path change rather than a broad plugin-loader rewrite: bound or bypass runtime-deps staging for audit collector discovery, add clear progress/diagnostic output for slow CLI preflight phases, and keep plugin activation semantics unchanged.
