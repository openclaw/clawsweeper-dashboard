## Summary

Keep open. Current main still matches the reported audit CLI reliability path: the command emits output only after SecretRef resolution and the full audit complete, while plugin collector fallback still reaches uncached metadata loading without an audit-safe manifest-only/runtime-deps-free mode. The related #75456 report is a close repair-cluster match, but this issue adds useful non-JSON, `--deep`, and direct `runSecurityAudit(...)` contrast that should stay attached until a fix lands.

## What This Changes

The issue reports that `openclaw security audit` and `openclaw security audit --deep` can appear to hang silently from the CLI even though direct installed `runSecurityAudit(...)` calls complete.

## Best Possible Solution

Ship a narrow audit-boundary fix that keeps plugin security collector coverage while making collector discovery read-only and bounded, avoids staging bundled runtime dependencies for discovery-only audit work, filters hidden/backup extension directories from audit scans, and adds visible diagnostics or progress for slow non-JSON CLI phases without breaking JSON output.

## Reproduction Assessment

Yes for the observable no-output path: current source shows the CLI waits through SecretRef resolution and the complete `runSecurityAudit` call before emitting output. The long hang duration itself is install-dependent and should be confirmed with a timed packaged/global-npm reproduction or focused loader fixture during the fix.

## Solution Assessment

Yes. A targeted security-audit and metadata-loader repair is the narrowest maintainable direction; a broad plugin-loader rewrite is not required to make this command bounded and visible.
