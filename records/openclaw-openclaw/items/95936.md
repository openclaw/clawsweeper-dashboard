## Summary

Keep open as a candidate for the shared same-workspace memory-index work, but the patch is not merge-ready: the updated head still leaves the central extraPaths reindex identity unchanged, changes the default persistent memory DB path without an upgrade path, and introduces unresolved session/privacy and SDK-surface risks.

## What This Changes

The PR changes memory search resolution so memory-source agents use a workspace-hash shared SQLite store and exports new shared-store registry helpers from the memory-host SDK.

## Best Possible Solution

Land a shared-memory design only after DB identity and index identity share one accepted contract, upgrade behavior is explicit, session visibility is enforced at the store/query boundary, and new SDK exports are either removed or documented as accepted API.

Yes. Source inspection shows current main still keys the active store per agent, while the PR head changes the DB filename only; current memory-core identity code and tests still treat extraPaths changes as index-scope changes.

## Solution Assessment

No. The workspace-hash idea is plausible, but this patch is not the best complete fix because it leaves the persisted index identity, upgrade path, session-boundary model, and SDK contract unresolved.

Full review comments:

- **[P1] Initialize sharedStorePath before returning the store:** `src/agents/memory-search.ts:72`
  - body: Adding `store.sharedStorePath` as a required field makes `mergeConfig()` return an invalid `ResolvedMemorySearchConfig`: the local `store` object is still built without that field and only patched later in `resolveMemorySearchConfig()`. This leaves the package failing strict type/build checks before the runtime code can use the value.
  - confidence: 0.95
- **[P1] Align the index identity with the shared-store hash:** `src/agents/memory-search.ts:493-498`
  - body: This changes only the SQLite filename. The memory-core index identity still hashes `extraPaths` and compares it to persisted `meta.scopeHash`, so changing `memorySearch.extraPaths` can still dirty the index or trigger a full reindex despite the PR's stated fix.
  - confidence: 0.93
- **[P1] Add an upgrade path before moving the memory DB:** `src/agents/memory-search.ts:497-498`
  - body: Overriding `resolved.store.databasePath` moves existing memory-source agents from their per-agent SQLite DB to a new shared DB path. Existing indexes can be stranded unless the PR adds a doctor migration/import path, explicit rebuild warning, or maintainer-approved upgrade plan.
  - confidence: 0.9
- **[P1] Keep session-enabled stores agent-scoped:** `src/agents/memory-search.ts:493`
  - body: The override runs for any config that includes `memory`, including `sources: ["memory", "sessions"]`. That can put per-agent transcript chunks into the same shared DB, while the CLI search path calls `manager.search()` directly instead of the agent tool's session visibility filter.
  - confidence: 0.88
- **[P2] Defer registry exports until the API is accepted:** `packages/memory-host-sdk/src/engine-storage.ts:55-58`
  - body: The PR exposes new registry helpers through the memory-host SDK even though no caller wires the registry and the shared-store lifecycle, migration, and access-control contract is unsettled. Keep these helpers private or add the maintainer-approved public API contract and compatibility tests before exporting them.
  - confidence: 0.82
