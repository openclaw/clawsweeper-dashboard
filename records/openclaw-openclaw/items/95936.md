## What This Changes

The PR redirects memory search storage to a shared SQLite path derived from workspace directory plus extraPaths, adds a small shared-store registry helper, exports new memory-host SDK helpers, and adds resolver tests.

## Best Possible Solution

Redesign the shared memory index on the current memory-core and memory-host SDK boundary with explicit source-set identity, agent-scoped session filtering, watcher ownership, and upgrade/rebuild proof before landing the linked issue fix.

No. A path override is only a plausible shortcut; the maintainable fix needs a first-class shared-index model with source identity, session membership filtering, watcher ownership, and upgrade behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Keep session-memory indexes out of the shared store:** `src/agents/memory-search.ts:414`
  - body: `computeSharedScopeHash(workspaceDir, resolved.extraPaths)` ignores the enabled source set and session owner, but the resolver redirects any config that includes `memory` to this shared DB. With `experimental.sessionMemory` enabled, same-workspace agents will write `source = 'sessions'` chunks into the same tables, and search only filters by source, so one agent can retrieve another agent's session memory. Keep session-enabled stores per-agent or add explicit agent membership/filtering before sharing.
  - confidence: 0.91
- **[P2] Preserve the existing memory index on upgrade:** `src/agents/memory-search.ts:416-417`
  - body: Switching the resolved store to `state/memory/shared-...sqlite` ignores the shipped per-agent SQLite indexes without any migration, rebuild policy, or upgrade proof. Existing users can hit an empty index and trigger a full re-embed on first search or sync, which is the expensive path this feature is intended to avoid.
  - confidence: 0.86
