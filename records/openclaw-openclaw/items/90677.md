## Label Justifications

- `impact:message-loss`: The reported failure is that one Google Chat app's message never arrives as an inbound event for the other agent.

## Summary

Keep open: current main still only handles Google Chat bot-loop protection after Google delivers a MESSAGE event, the docs still do not warn about the reported Chat-app-to-Chat-app delivery limit, and the bridge request remains a maintainer product/API decision.

## What This Changes

The issue requests a Google Chat docs warning and/or a channel-independent bridge for bot-to-bot agent conversations when Google Chat does not deliver app-authored messages to peer Chat apps.

## Best Possible Solution

Add a Google Chat and bot-loop docs warning now, then separately decide whether an opt-in bridge should reuse the existing sessions_send/tools.agentToAgent boundary.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No live Workspace repro was run, but current source makes the OpenClaw side source-reproducible because allowBots and botLoopProtection only run after a Google Chat MESSAGE event exists.

## Solution Assessment

Is this the best way to solve the issue? Partly: documenting the Google Chat limitation is the narrowest immediate fix, while an automatic bridge needs maintainer design around the existing sessions_send/tools.agentToAgent boundary.
