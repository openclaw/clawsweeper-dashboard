## Label Justifications

- `P2`: The issue blocks a scoped Google Chat multi-agent workflow without making ordinary Google Chat human-authored messages or the core runtime unusable.
- `impact:message-loss`: The reported failure is that a message authored by one Google Chat app never arrives as an inbound event for the other agent.

## Summary

Keep open: current main and v2026.6.8 still only apply Google Chat `allowBots` and bot-loop protection after a `MESSAGE` webhook exists, the docs do not warn that peer Chat-app messages may not arrive, and the linked Google Chat PR explicitly does not own this app-to-app delivery gap.

## What This Changes

Request to document Google Chat app-to-app message delivery limits and/or add an internal channel-independent A2A bridge for two agents bound to separate Chat apps.

## Best Possible Solution

Document the Google Chat event-delivery limitation now, and treat any automatic bridge as a separate opt-in A2A design that reuses existing sessions_send safety limits.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not with a fresh live Workspace run; source inspection does show OpenClaw has no path to process a peer app message unless Google first sends a `MESSAGE` event.

## Solution Assessment

Is this the best way to solve the issue? Partly: a docs warning is the narrow immediate fix, while any automatic bridge should reuse or extend the existing `sessions_send`/`tools.agentToAgent` boundary only after maintainer design.
