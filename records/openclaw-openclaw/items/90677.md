## Label Justifications

## Summary

Keep open: current main and the latest release still only handle Google Chat bot-loop behavior after an inbound `MESSAGE` webhook exists, while the issue reports that Google does not emit that webhook for another Chat app's message; the docs still do not call out that limitation, and the bridge request needs maintainer product direction.

## What This Changes

The issue asks OpenClaw to document Google Chat's app-to-app delivery limitation and/or add a channel-independent internal agent-to-agent bridge for multi-agent conversations.

## Best Possible Solution

Add a Google Chat and multi-agent docs warning for the app-to-app event gap, then handle any transparent bridge as a policy-gated core/session design that reuses or extends `sessions_send` and `tools.agentToAgent`.

## Reproduction Assessment

No live two-Chat-app reproduction was run, so this is not a strict reproduced bug. The source-level path is clear enough for triage: current main can only dispatch received Google Chat `MESSAGE` events, and the reporter's logs say that event is absent for app-authored messages.

## Solution Assessment

Partly. A docs clarification is the narrow immediate fix, but the optional transparent bridge needs maintainer product/API judgment and should preserve existing `sessions_send` and `tools.agentToAgent` boundaries if accepted.
