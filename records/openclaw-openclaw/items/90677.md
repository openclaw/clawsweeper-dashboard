## Label Justifications

- `P2`: The issue blocks a scoped Google Chat multi-agent workflow, but ordinary human-authored Google Chat messages and the core runtime are not unusable.
- `impact:message-loss`: The reported failure is that one agent's Google Chat app message never arrives as an inbound event for the other configured agent.

## Summary

Keep open: current main and v2026.6.9 still only process Google Chat bot-loop behavior after a delivered MESSAGE webhook exists, and the docs still do not warn that another Chat app's message may never arrive as an inbound event.

## What This Changes

The issue asks OpenClaw to document Google Chat's app-to-app delivery limitation and possibly add a channel-independent agent-to-agent bridge for addressed agent replies.

## Best Possible Solution

Document the Google Chat delivery limitation now, then design any bridge as an explicit opt-in A2A path that reuses existing sessions_send and tools.agentToAgent safety boundaries.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not with a fresh live Workspace run; source inspection does show OpenClaw cannot process a peer Chat-app message unless Google first sends a MESSAGE event.

## Solution Assessment

Is this the best way to solve the issue? Partly: a docs warning is the narrow maintainable immediate fix, while a transparent bridge needs maintainer design around A2A policy and loop semantics.
