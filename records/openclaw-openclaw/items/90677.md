## Label Justifications

- `impact:message-loss`: The reported failure is that a message from one Chat app never arrives as an inbound event for the other configured agent.

## Summary

Keep open: current main still only handles Google Chat bot-loop protection after an inbound Google Chat MESSAGE event exists, and the docs still do not warn that Chat-app-authored messages may not be delivered to another Chat app. The existing `sessions_send`/`tools.agentToAgent` path is a guarded tool-level A2A mechanism, not the automatic Google Chat Dev-to-QA bridge requested here.

## What This Changes

The issue requests documenting Google Chat app-to-app message delivery limits and possibly adding a channel-independent internal agent-to-agent bridge for multi-agent conversations.

## Best Possible Solution

Document the Google Chat limitation now, and handle any channel-independent Dev-to-QA bridge as an explicit guarded A2A design that reuses or extends `sessions_send`/`tools.agentToAgent` rather than hiding routing inside the Google Chat plugin.

## Reproduction Assessment

Partly. Source review shows OpenClaw only applies Google Chat `allowBots` and bot-loop protection after a Google Chat inbound `MESSAGE` webhook arrives, but this review did not run a live two-Chat-app Workspace reproduction.

## Solution Assessment

No for an automatic hidden bridge. The immediate docs clarification is narrow and maintainable, while any channel-independent bridge should be an explicit A2A/product design that respects existing session visibility and agent-to-agent policy.
