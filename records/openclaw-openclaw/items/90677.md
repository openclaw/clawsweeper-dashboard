## Summary

Keep open: current main and the latest release still only handle Google Chat bot-loop protection after an inbound Google Chat MESSAGE event exists, while the Google Chat docs still do not document the reported Chat-app-to-Chat-app event-delivery limitation and no automatic channel-independent bridge exists.

## What This Changes

The issue requests Google Chat documentation for Chat-app-authored message delivery limits and optionally a channel-independent internal agent-to-agent routing bridge.

## Best Possible Solution

Add a Google Chat and multi-agent docs note for the app-to-app event-delivery limitation, and treat any automatic bridge as a separate, policy-gated A2A design on top of existing sessions_send/tools.agentToAgent behavior.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not live; current source makes the failure source-reproducible because allowBots and botLoopProtection only run after Google Chat delivers an inbound MESSAGE event, matching the reporter's missing-event logs.

## Solution Assessment

Is this the best way to solve the issue? Partly: documenting the Google Chat limitation is the narrow immediate fix, while an automatic bridge is a broader product/API decision that should reuse or extend existing sessions_send and tools.agentToAgent boundaries.
