## Summary

Keep open: current main still documents and implements Google Chat bot-loop handling only after an inbound bot-authored event exists, but it does not document that Google Chat interaction webhooks may not deliver Chat-app-authored messages to another Chat app and it does not provide an automatic channel-independent bridge for the reported Dev-to-QA conversation use case.

## What This Changes

The issue asks OpenClaw to document Google Chat's app-to-app delivery limitation and optionally add a channel-independent agent-to-agent routing bridge.

## Best Possible Solution

Document the Google Chat interaction-event limitation now, and handle any channel-independent agent-to-agent bridge as a product/API design on top of the existing sessions_send and tools.agentToAgent policy.

## Reproduction Assessment

No. The report gives a plausible live reproduction and current source shows OpenClaw can only dispatch received Google Chat MESSAGE events, but I did not run a live two-Chat-app Workspace setup in this review.

## Solution Assessment

Unclear. A docs clarification is clearly the right immediate fix, but the optional automatic bridge needs maintainer product/API judgment and should reuse existing sessions_send/tools.agentToAgent boundaries if accepted.
