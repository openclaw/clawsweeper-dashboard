## Summary

Keep open: current main and v2026.6.8 still only process Google Chat bot-loop protection after Google delivers an inbound MESSAGE event, while the docs do not warn that peer Chat-app messages may never arrive and the requested transparent bridge is a product/API decision.

## What This Changes

The issue requests a Google Chat documentation warning and/or a channel-independent internal agent-to-agent bridge for two agents conversing when Google Chat does not deliver app-authored messages between Chat apps.

## Best Possible Solution

Add a Google Chat and bot-loop docs warning now, then decide separately whether existing tools.agentToAgent and sessions_send should grow an opt-in automatic bridge with loop and safety tests.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No live high-confidence Workspace repro was run; source inspection does show OpenClaw can only apply allowBots and botLoopProtection after a Google Chat MESSAGE event exists, matching the reporter's missing-event logs.

## Solution Assessment

Is this the best way to solve the issue? Partly: documenting the Google Chat limitation is the narrow immediate fix, while an automatic bridge should reuse or extend the existing sessions_send/tools.agentToAgent boundary only after maintainer design.
