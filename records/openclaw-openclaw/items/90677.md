## Summary

Keep open: current main and the latest release still only apply Google Chat `allowBots` and bot-loop handling after a `MESSAGE` event reaches the plugin, the docs still omit the reported Chat-app-to-Chat-app event limitation, and the existing A2A path is explicit `sessions_send` tooling rather than the automatic channel bridge requested.

## What This Changes

The issue asks OpenClaw to document that Google Chat app-authored messages do not reach another Chat app as inbound events and/or add a channel-independent agent-to-agent routing bridge.

## Best Possible Solution

Document the Google Chat app-to-app interaction-event limitation now, and require any automatic bridge to be a maintainer-designed extension of existing `sessions_send` and `tools.agentToAgent` safety semantics.

## Reproduction Assessment

No high-confidence live reproduction was established in this review. Source inspection gives a clear OpenClaw-side reproduction of the boundary: Google Chat `allowBots` and loop protection only run after an inbound `MESSAGE` event exists, so a missing app-to-app event cannot be recovered by current code.

## Solution Assessment

Unclear. A docs clarification is the narrow immediate fix, but the optional automatic bridge is not yet proven to be the best solution without maintainer product/API design against existing session-tool and A2A policy boundaries.
