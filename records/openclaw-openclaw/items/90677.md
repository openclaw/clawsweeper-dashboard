## Summary

Keep open: current main still depends on inbound Google Chat `MESSAGE` events before `allowBots` or bot-loop protection can run, the current docs still imply accepted bot-authored Google Chat events are guarded without documenting the app-to-app missing-event case, and the existing `sessions_send` A2A path is explicit and policy-gated rather than the requested transparent bridge.

## What This Changes

The issue requests documentation of Google Chat app-to-app delivery limits and optionally a channel-independent internal agent-to-agent bridge for two OpenClaw agents talking through Google Chat.

## Best Possible Solution

Document the Google Chat app-to-app event limitation now, and only add a channel-independent bridge after maintainers decide how it should reuse or extend `sessions_send`, `tools.agentToAgent`, and session visibility policy.

No high-confidence live reproduction was established. The reporter gives plausible logs, and current source confirms OpenClaw can only route received Google Chat `MESSAGE` events, but this review did not run a two-Chat-app Workspace setup.

## Solution Assessment
