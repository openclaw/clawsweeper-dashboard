## Label Justifications

- `P2`: The issue blocks a real but scoped Google Chat multi-agent workflow without making normal Google Chat human-authored messages or the core runtime unusable.
- `impact:message-loss`: The reported failure is that a message authored by one Google Chat app never arrives as an inbound event for the other agent.

## Summary

Keep open: current main and the latest release still depend on Google Chat delivering an inbound MESSAGE event before allowBots or bot-loop protection can run, while the docs do not warn about the reported Chat-app-to-Chat-app delivery gap and an automatic bridge remains a product decision.

## What This Changes

Document the Google Chat app-authored message delivery limitation and optionally add a channel-independent bridge for agent-to-agent conversations.

## Best Possible Solution

Add a Google Chat and multi-agent docs note for the app-to-app event-delivery limitation, and treat any automatic bridge as a separate gated A2A design built on the existing sessions_send/tools.agentToAgent boundary.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not live; source inspection shows OpenClaw can only act after a Google Chat MESSAGE event exists, matching the reporter's missing-event logs, but a fresh Workspace two-app run was not performed.

## Solution Assessment

Is this the best way to solve the issue? Partly: documenting the limitation is the narrow maintainable fix, while an automatic bridge should be a separately designed A2A feature rather than a Google Chat-only workaround.
