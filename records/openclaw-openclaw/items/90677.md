## Summary

Keep open: current main still relies on Google Chat delivering an inbound `MESSAGE` event before `allowBots` or bot-loop protection can run, and the docs still do not explain the reported Chat-app-to-Chat-app missing-event limitation. Existing `sessions_send` support is a policy-gated tool path, not the transparent channel-independent bridge requested here.

## What This Changes

The issue requests documentation and/or a channel-independent routing bridge for Google Chat agent-to-agent conversations where one Chat app does not receive events for messages sent by another Chat app.

## Best Possible Solution

Document the Google Chat inbound-event limitation now, and design any channel-independent bot-to-bot bridge on top of existing `sessions_send` and `tools.agentToAgent` policy rather than adding a Google Chat-only shortcut.

## Reproduction Assessment

No live high-confidence reproduction was run in this review. Source inspection does show that OpenClaw can only authorize and guard bot-authored Google Chat messages after Google delivers an inbound `MESSAGE` event, matching the reported failure mode.

## Solution Assessment

No, not as an automatic code fix. A docs clarification is the narrow immediate solution, while the optional bridge needs maintainer product/API review and should reuse the existing A2A policy boundary.
