## Summary

Keep open: the PR addresses a real gateway sampling bug, but the proposed short-window cache returns before checking event-loop delay, so rapid probes can hide delay-only starvation despite the new classifier contract saying short-window delay should still degrade.

## What This Changes

The PR adds a 1s minimum gateway event-loop health sample window, caches the last snapshot for rapid probes, adds focused Vitest coverage, and records a changelog fix.

## Best Possible Solution

Keep the 1s baseline preservation for utilization and CPU, but continue sampling and surfacing event-loop delay during short follow-up windows with monitor-level regression coverage.

## Reproduction Assessment

Yes, from source inspection. Current main resets rate baselines on every `snapshot()`, and the PR's changed `snapshot()` path returns cached data before reading delay metrics, so rapid health/status/readiness probes exercise the affected path.

## Solution Assessment

No. The 1s cache is the right direction for utilization and CPU, but the implementation should gate only rate-based classification while preserving fresh delay degradation.
