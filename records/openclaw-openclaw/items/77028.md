## Summary

Keep this PR open: current main still resets event-loop utilization and CPU baselines on every health snapshot, so the reported rapid-probe false degradation remains source-reproducible. The PR is a narrow gateway fix with focused coverage and no concrete functional or security finding from this review.

## What This Changes

The PR changes the gateway event-loop health monitor to reuse cached snapshots until a 1s sample window is available, factors reason classification into a helper, adds focused Vitest coverage, and updates the changelog.

## Best Possible Solution

Land the PR after CI or targeted gateway test proof confirms the 1s sampling contract and any branch-refresh churn stays limited to the changelog/current-main context.

## Reproduction Assessment

Yes, from source inspection. Current main recomputes and resets utilization/CPU baselines on every `snapshot()`, so rapid follow-up probes can classify sub-second rate samples as degraded and discard the longer sampling baseline.

## Solution Assessment

Yes. Holding the rate baseline until a full 1s window and reusing the last snapshot is the narrowest maintainable fix because it preserves sustained utilization/CPU detection without changing the public health schema or adding config.
