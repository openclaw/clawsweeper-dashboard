## Summary

Keep open. Current main still awaits bundle MCP setup before the model attempt and awaits MCP tool execution during the turn; related open issues cover circuit-breaker, retry-storm, and status-panel slices, but none supersedes the full fast-degradation contract for the main reply path.

## What This Changes

The issue asks for external MCP/plugin failures to degrade quickly instead of delaying or blocking the main assistant reply path.

## Best Possible Solution

Define an additive bundle-MCP resilience contract around catalog listing and tool execution: short bounded budgets, per-server health/cooldown state, optional degraded materialization, redacted diagnostics/status, and focused regression coverage.

## Reproduction Assessment

Yes, at source level: current main awaits MCP runtime creation, catalog materialization, SDK listTools, and SDK callTool on the reply/tool path. I did not run a live half-hung MCP server, so this is source-reproducible rather than a live runtime reproduction.

## Solution Assessment

Unclear on exact policy. The requested direction is maintainable, but the best implementation should put bounded degradation in the bundle MCP runtime/materializer and avoid treating a narrow timeout-only patch as the whole circuit-breaker and diagnostics solution.
