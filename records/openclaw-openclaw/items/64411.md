## Summary

Keep open. Current main still has source-proven MCP catalog and tool-call awaits on the assistant reply path, while the related open issues cover circuit-breaker, retry-storm, and status-panel slices rather than superseding the full fast-degradation contract.

## What This Changes

The issue requests bounded, optional degradation for unhealthy MCP/plugin capability so the main assistant reply path stays responsive.

## Best Possible Solution

Define an additive bundle-MCP resilience contract around catalog listing and tool execution: short bounded budgets, per-server health/cooldown state, optional degraded materialization, redacted diagnostics/status, and focused regression coverage.

## Reproduction Assessment

Yes, at source level: current main awaits MCP runtime creation, catalog materialization, SDK listTools, and SDK callTool on the reply/tool path. I did not run a live half-hung MCP server, so this is source-reproducible rather than a live runtime reproduction.

## Solution Assessment

Unclear on exact policy. The requested direction is maintainable, but the best implementation should put bounded degradation in the bundle MCP runtime/materializer and avoid treating a timeout-only patch as the whole circuit-breaker and diagnostics solution.
