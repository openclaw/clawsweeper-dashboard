## Summary

Keep this open. Current main partially degrades rejected MCP startup failures by logging and dropping a failed server, but slow or half-hung MCP catalog listing and tool execution are still directly awaited on the main reply path, and there is still no per-server circuit/cooldown state or degraded capability status that distinguishes channel health from external MCP health.

## What This Changes

The issue requests fast degradation for unhealthy optional MCP/plugin capabilities so slow or disconnected external calls do not block the main assistant reply path.

## Best Possible Solution

Add a core bundle-MCP resilience contract: bounded catalog and tool-call budgets, per-server failure cooldown/circuit state, degraded empty-tool materialization for unhealthy optional external capability, and diagnostics/status text that separates healthy channels from degraded external MCP/plugin capability.

## Reproduction Assessment

Yes. A deterministic reproduction can use an MCP server or test double whose listTools or callTool promise never settles; current main awaits those promises before model reply preparation or during tool execution, so the turn blocks until an outer run timeout rather than degrading quickly.

## Solution Assessment

Yes, the best first slice is a bounded core fix in the bundle-MCP runtime and materialization path, with hook timeout coverage extended only where reply-path hooks are still unbounded. Moving this to a plugin would not solve the core reply-path await.
