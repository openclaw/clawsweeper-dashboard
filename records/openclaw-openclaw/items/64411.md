## Label Justifications

- `P1`: An unhealthy external MCP can delay or suppress the main assistant reply path even when the channel and model path are otherwise healthy.
- `impact:message-loss`: The reported failure mode is delayed or missing assistant replies from a healthy channel because optional external capability blocks the turn.

## Summary

Keep open. Current main still gates the assistant reply/tool path on bundle-MCP runtime creation, catalog listing, and tool execution; related open issues cover generic circuit breaker, retry storm, and UI status slices, but no current implementation provides the full fast-degradation contract.

## What This Changes

Request bounded optional degradation for unhealthy MCP/plugin capability so the main assistant reply path stays responsive.

## Best Possible Solution

Define an additive bundle-MCP resilience contract around catalog listing and tool execution: bounded budgets, per-server health/cooldown state, optional degraded materialization, redacted diagnostics/status, and focused regression coverage.

## Reproduction Assessment

Yes at source level: current main awaits MCP runtime setup, catalog listing, and SDK tool execution on the reply/tool path. I did not replay a live half-hung MCP server, so this is source-reproducible rather than a live runtime reproduction.

## Solution Assessment

Unclear on exact policy. The requested direction is maintainable, but the best fix needs a designed resilience contract rather than a timeout-only patch.
