## Summary

Keep open. Current main still awaits bundle MCP catalog materialization before the model attempt and awaits bundle MCP tool execution during the turn; the related open issues cover circuit-breaker, retry-storm, and status-panel slices, but not the full fast-degradation contract for the main reply path.

## What This Changes

The issue asks OpenClaw to degrade unhealthy optional MCP/plugin capability quickly so the main assistant reply path stays responsive.

## Best Possible Solution

Define an additive bundle-MCP resilience contract around catalog listing and tool execution: bounded budgets, per-server health/cooldown state, optional degraded materialization, redacted diagnostics/status, and focused regression coverage.

## Reproduction Assessment

Yes, at source level: current main awaits MCP runtime creation, catalog materialization, listTools, and callTool on the reply/tool path. I did not run a live half-hung MCP server, so this is source-reproducible rather than a live runtime reproduction.

## Solution Assessment

Unclear on exact policy. The requested direction is maintainable, but the best implementation should put bounded degradation in the bundle MCP runtime/materializer and avoid treating a narrow timeout-only patch as the whole circuit-breaker and diagnostics solution.
