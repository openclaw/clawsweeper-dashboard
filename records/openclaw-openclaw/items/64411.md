## Summary

Keep open. Current main still awaits bundle MCP catalog materialization before the model attempt and directly awaits MCP tool calls, so a slow or never-settling MCP server can still block the assistant reply path; adjacent open issues cover narrower timeout, retry-storm, and status-panel pieces but do not supersede this broader degradation request.

## What This Changes

The issue requests bounded, observable degradation for unhealthy external MCP/plugin capability so the main assistant reply path stays responsive unless the current turn explicitly depends on that capability.

## Best Possible Solution

Implement a core bundle-MCP resilience contract around catalog listing and tool execution: bounded budgets, per-server failure/cooldown state, degraded empty-tool materialization for unhealthy optional MCP capability, and diagnostics that distinguish channel health from external MCP health.

## Reproduction Assessment

Yes, from source inspection: a `listTools`, `getCatalog`, or `callTool` promise that never settles will keep the current awaited reply path open before or during a turn. I did not run a live MCP because this was a read-only review, but the unbounded awaits are visible on current main.

## Solution Assessment

Yes for the code boundary: the narrow maintainable fix belongs around bundle MCP runtime/materialization, where the reply-path awaits occur. The exact timeout thresholds, config surface, circuit reset behavior, and UI/status diagnostics remain product choices.
