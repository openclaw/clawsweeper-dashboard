## Summary

Keep open. Current main still puts bundle MCP catalog materialization and tool execution on the assistant reply path, with only connection startup guarded; rejected startup errors degrade, but slow or never-settling catalog/tool promises can still block a turn.

## What This Changes

Request fast degradation for optional external MCP/plugin capability failures through bounded timeouts, cooldown/circuit state, and clear diagnostics so normal chat replies stay responsive.

## Best Possible Solution

Add a core bundle-MCP resilience contract around catalog listing and tool calls: bounded budgets, per-server failure cooldown/circuit state, degraded empty-tool materialization for unhealthy optional MCP capability, and diagnostics that separate channel health from external MCP health.

## Reproduction Assessment

Yes. A deterministic test double whose getCatalog/listTools or callTool promise never settles would exercise the current path: the runner awaits catalog materialization before the model attempt, and materialized tools await callTool during execution.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the core bundle-MCP runtime/materialization path because that is where the reply-path await occurs; moving this to ClawHub or a single plugin would not protect configured external MCP servers generally.
