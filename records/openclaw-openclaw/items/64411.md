## Summary

Keep open. Current main still awaits bundle MCP catalog materialization before model attempts and awaits bundle MCP tool calls during execution, while the related open items only cover narrower timeout, circuit-breaker, retry-storm, and status-panel slices rather than the full fast-degradation contract.

## What This Changes

The issue asks OpenClaw to keep the main assistant reply path responsive when optional external MCP/plugin capabilities are slow or unhealthy.

## Best Possible Solution

Define an additive bundle-MCP resilience contract around catalog listing and tool execution: bounded budgets, per-server health/cooldown state, optional degraded tool materialization, redacted diagnostics, and focused docs/tests.

## Reproduction Assessment

Yes, from source inspection: current main awaits MCP runtime creation, catalog materialization, listTools, and callTool on the reply/tool path. I did not run a live hanging MCP server in this read-only review, so the proof is source-level rather than live runtime proof.

## Solution Assessment

Unclear on exact policy, but the implementation boundary is clear: bundle MCP runtime and materialization should own fast degradation. The safer path is not to merge a narrow timeout-only patch as the full solution unless maintainers accept the remaining circuit-breaker and diagnostics gaps separately.
