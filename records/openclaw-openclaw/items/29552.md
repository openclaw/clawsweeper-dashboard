## Summary

Keep this open. Current main still resolves ordinary local Gateway calls env-first and has no config-first retry after `AUTH_TOKEN_MISMATCH`, so the reported pairing failure remains plausible despite adjacent newer-config and update hardening.

## What This Changes

The issue reports Gateway pairing failures after OpenClaw version/config drift, with logs showing both a newer-config warning and a gateway token mismatch.

## Best Possible Solution

Define the intended local Gateway auth contract and then either strengthen doctor/docs recovery for stale env/service tokens or implement a guarded loopback-only config-first retry for token mismatch with focused coverage.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: local call/probe auth can still prefer a stale `OPENCLAW_GATEWAY_TOKEN`, and the call path has no config-first retry after token mismatch.

## Solution Assessment

Unclear. A narrow local retry like the closed PR may be viable, but Gateway auth precedence is security-sensitive enough that a maintainer should choose the policy before automation changes it.
