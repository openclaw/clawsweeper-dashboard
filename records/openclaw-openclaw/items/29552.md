## Summary

Keep this open. Current main has newer-config guards, token-drift diagnostics, and trusted device-token retry handling, but the ordinary local Gateway call path still resolves one env-first credential set and does not retry with config-first `gateway.auth` credentials after `AUTH_TOKEN_MISMATCH`; tests and docs codify that behavior. The closed fallback PR would have addressed this path, while the open warning-text PR is adjacent and does not close the auth recovery gap.

## What This Changes

The issue requests fixing a version/config mismatch scenario where an older CLI or stale credential source causes local Gateway pairing to fail with a gateway token mismatch.

## Best Possible Solution

Define the intended local Gateway auth contract explicitly: either keep env-first local calls and improve doctor/docs recovery for stale env/service tokens, or add a narrowly guarded loopback-only retry after `AUTH_TOKEN_MISMATCH` that uses config-first `gateway.auth` credentials with regression coverage across call, credential precedence, client retry, onboarding, and daemon restart paths.
