## Summary

Keep this open. Current main has newer-config guards and adjacent version-reporting/update fixes, but the reported gateway pairing failure still maps to an unresolved local Gateway auth path: ordinary local calls remain env-first, credentials are resolved once, and `AUTH_TOKEN_MISMATCH` retry is still limited to trusted cached device-token recovery rather than a config-first `gateway.auth` retry.

## What This Changes

The issue requests fixing a version/config mismatch scenario where local Gateway pairing fails with a newer-config warning followed by a gateway token mismatch.

## Best Possible Solution

Define the intended local Gateway auth contract explicitly: either preserve env-first local calls and strengthen doctor/docs recovery for stale env or service tokens, or add a narrowly guarded loopback-only retry after `AUTH_TOKEN_MISMATCH` that uses config-first `gateway.auth` credentials with regression coverage across call, credential precedence, client retry, onboarding, and daemon restart paths.

## Reproduction Assessment

Yes. Current source and tests provide a high-confidence reproduction path for the stale-env half: local call/probe credential resolution prefers `OPENCLAW_GATEWAY_TOKEN` over `gateway.auth.token`, and the call path has no config-first retry after token mismatch.

## Solution Assessment

Unclear. The closed fallback PR describes a plausible narrow fix, but changing auth retry/precedence is security-sensitive; a maintainer should choose between documenting env-first recovery and implementing a guarded local-only retry.
