## Summary

Keep open: the PR addresses the documented `main` binding bug in the right area, but the current predicate also accepts and preserves malformed binding ids that fallback-normalize to `main`, so config validation and doctor repair are still not safe to merge.

## What This Changes

The branch changes config schema validation and doctor binding pruning to treat `DEFAULT_AGENT_ID` as valid for `bindings[].agentId` when `agents.list` is non-empty.

## Best Possible Solution

Allow explicit literal `main` bindings in both schema validation and doctor repair, keep fallback-normalized malformed ids rejected or pruned, and add focused regression coverage plus real config-load and doctor proof.

## Reproduction Assessment

Yes at source level: current main builds the allowed binding-agent set only from `agents.list`, while docs show `agentId: "main"` route bindings next to ACP agent entries. I did not run tests because this review is read-only.

## Solution Assessment

No: the proposed fix is in the right files, but comparing the normalized id to `DEFAULT_AGENT_ID` is too broad because normalization falls back to `main` for malformed values. The safer fix is a shared explicit-main predicate used by both schema validation and doctor pruning.
