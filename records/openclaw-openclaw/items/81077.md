## Summary

Keep open: current `main` still has the reported asymmetric help path, because unowned primary command rejection is skipped whenever the invocation is classified as help/version before the CLI reaches normal parsing.

## What This Changes

The issue asks for `openclaw <unknown-command> --help` to reject with an unknown-command error and nonzero exit instead of showing generic top-level help with exit 0.

## Best Possible Solution

Reject unowned primary commands before generic help rendering while preserving exit 0 for valid root, built-in, and plugin help paths.

## Reproduction Assessment

Yes. Source inspection on current `main` shows the exact path: unknown-root rejection is bypassed for help/version invocations, while the existing no-help unknown-root test only covers `openclaw foo`.

## Solution Assessment

Yes. The maintainable fix is a narrow CLI bootstrap change plus regression coverage for `openclaw <unknown>` and `openclaw <unknown> --help`, without changing valid command help behavior.
