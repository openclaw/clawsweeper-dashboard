## Summary

Keep open: current main still has the reported scheduler behavior, and the open closing PR needs maintainer review because it changes the documented QMD lexical-mode contract.

## What This Changes

The issue asks OpenClaw to automatically run `qmd embed` according to `memory.qmd.update.embedInterval` even when `memory.qmd.searchMode` is omitted or resolves to lexical search.

## Best Possible Solution

Settle the QMD lexical-mode contract: either keep `searchMode: "search"` as BM25-only and clarify `embedInterval` docs, or implement an explicit `embedInterval` override with resolved-config explicitness and aligned tests/docs.

## Reproduction Assessment

Yes, source-reproducible against current main: omitted `searchMode` resolves to `search`, and the QMD manager returns before scheduling or running `qmd embed`. Existing tests also assert no subprocess calls for `searchMode: "search"` plus `embedInterval: "5m"`.

## Solution Assessment

Unclear. The scheduler gap is narrow, but the fix direction needs maintainer product judgment because letting `embedInterval` override lexical mode conflicts with the current BM25-only docs and tests.
