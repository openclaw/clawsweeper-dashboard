## Summary

Keep open. Current main still resolves omitted QMD `searchMode` to lexical `search`, and both embed scheduling and embed execution return before `embedIntervalMs` is considered. Open PR #73512 is the active unmerged implementation candidate with closing syntax for this issue, so the issue should wait for that PR to be reviewed, landed, or closed rather than be resolved now.

## What This Changes

The issue asks OpenClaw to automatically run QMD embedding according to `memory.qmd.update.embedInterval` when QMD documents change, even when `searchMode` is omitted from config.

## Best Possible Solution

Resolve the scheduler contract through PR #73512 or an equivalent narrow change: either make explicit `embedInterval` opt into QMD embedding maintenance and update tests/docs, or keep lexical mode BM25-only and make the required `searchMode: "query"` or `"vsearch"` guidance clear in config/docs/status output.

## Reproduction Assessment

Yes. A high-confidence static and unit-level reproduction exists: configure QMD with omitted/default `searchMode: "search"` plus `embedInterval`, advance the manager timer, and current main's test expects no `update` or `embed` subprocess calls.

## Solution Assessment

Unclear. PR #73512 is the right narrow implementation candidate for the observable bug, but maintainers still need to decide whether explicit `embedInterval` should override the documented lexical-mode behavior.
