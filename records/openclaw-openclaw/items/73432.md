## Summary

Keep open. Current main still routes the reporter's omitted `searchMode` configuration to lexical `search`, and that path returns before QMD embed execution or timer scheduling can honor a positive `embedInterval`; the open closing PR is still unmerged and has a contract question to resolve.

## What This Changes

The issue requests automatic QMD embedding on the configured `memory.qmd.update.embedInterval` even when `memory.qmd.searchMode` is omitted or defaults to `search`.

## Best Possible Solution

Settle the lexical-mode QMD contract, then land https://github.com/openclaw/openclaw/pull/73512 or an equivalent narrow fix that tracks explicit `embedInterval` intent and aligns tests plus docs.

## Reproduction Assessment

Yes, source-level. Omitted `searchMode` resolves to `search`, and current manager guards prevent both embed timer creation and `qmd embed` execution before the positive `embedIntervalMs` can matter.

## Solution Assessment

Unclear. The linked PR targets the right scheduler gap, but the maintainable fix should preserve the documented BM25-only default and only override it when explicit user intent is represented in config.
