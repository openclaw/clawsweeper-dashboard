## Summary

Keep open. Current main and the latest release still make QMD lexical `search` mode skip embedding maintenance even when `embedInterval` resolves to a positive value, while the docs now describe that as the intended BM25-only contract; the remaining question is whether an explicit `update.embedInterval` should override that contract.

## What This Changes

The issue requests automatic QMD embedding on the configured update/embed cadence when `memory.qmd.searchMode` is omitted and defaults to lexical `search`.

## Best Possible Solution

Choose and document one QMD contract: either keep lexical `search` embedding-free and direct users to `query` or `vsearch`, or carry raw explicit `update.embedInterval` intent through resolved config and update runtime, tests, and docs together.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: omit `memory.qmd.searchMode`, configure QMD update/embed intervals, and current main resolves lexical `search` then returns from both embed execution and timer scheduling before the interval can matter.

## Solution Assessment

Unclear. The maintainable repair, if accepted, is not the closed guard-only PR shape; it should preserve explicit raw `embedInterval` intent and align docs/tests, or preserve the documented `query`/`vsearch` workaround instead.
