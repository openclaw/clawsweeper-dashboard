## Summary

Keep open. Current main still matches the reported behavior because lexical QMD `search` mode blocks both embed execution and the periodic embed timer before `embedIntervalMs` can opt in, and the linked PR #73512 is open with closing syntax for the same scheduler behavior. This should wait for maintainer review of that PR rather than be closed or queued as duplicate repair.

## What This Changes

The issue requests that OpenClaw automatically trigger QMD embedding according to `memory.qmd.update.embedInterval` when QMD-managed memory documents change.

## Best Possible Solution

Resolve the contract through PR #73512 or a maintainer replacement: either treat explicit `embedInterval` as an embedding opt-in and update tests/docs accordingly, or keep lexical mode BM25-only and improve config/status guidance so users know to choose `vsearch` or `query` for automatic embeddings.

## Reproduction Assessment

Yes. A high-confidence static and unit-level reproduction exists on current main: configure QMD with default or explicit `searchMode: "search"` plus `embedInterval`, advance the manager timer, and current code/tests expect no `qmd update` or `qmd embed` subprocess calls.

## Solution Assessment

Unclear. PR #73512 is a plausible narrow code fix for the observable user problem, but it changes the documented BM25-only behavior of lexical QMD mode, so maintainer review should decide whether the code or the docs/status guidance should move.
