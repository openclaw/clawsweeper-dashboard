## Summary

Keep open. Current main still gates QMD embedding before `embedInterval` is considered in lexical `search` mode, and the reporter's config omits `searchMode`, which resolves to that mode. An open, unmerged PR already references this issue with closing syntax and targets the same scheduler behavior, so this should wait for maintainer review of that PR rather than be closed or queued as a second repair.

## What This Changes

The issue requests that QMD automatically run embedding on the configured `memory.qmd.update.embedInterval` cadence after OpenClaw updates memory files, even when `searchMode` is omitted/defaulted.

## Best Possible Solution

Resolve the contract through PR #73512 or an equivalent narrow change: treat an explicitly configured `embedInterval` as an embedding opt-in, with matching tests and docs, or keep lexical mode BM25-only and update the issue with the required `searchMode` guidance.

## Reproduction Assessment

Yes. A high-confidence static and unit-level reproduction exists on current main: use QMD with omitted/default `searchMode: "search"` plus `embedInterval`, advance the manager timer, and the current test expects no `update` or `embed` subprocess calls.

## Solution Assessment

Unclear. The linked PR is the right narrow place to solve the observable scheduler bug, but the implementation needs maintainer agreement because current docs and tests intentionally define lexical `search` mode as skipping QMD embedding maintenance.
