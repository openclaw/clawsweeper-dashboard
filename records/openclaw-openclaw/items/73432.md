## Summary

Keep open. Current main still blocks QMD embedding before `embedInterval` is considered when QMD uses the default lexical `search` mode, and the reporter's config omits `searchMode`, which resolves to `search`. PR #73512 is still the active open implementation candidate with closing syntax for this issue, so this should wait for maintainer review or landing of that PR rather than be closed or queued as a second repair.

## What This Changes

The issue requests that QMD automatically run embedding on the configured `memory.qmd.update.embedInterval` cadence after OpenClaw updates memory files, even when `searchMode` is omitted/defaulted.

## Best Possible Solution

Resolve the contract through PR #73512 or an equivalent narrow change: either treat explicitly configured `embedInterval` as an embedding opt-in with matching tests/docs, or preserve BM25-only lexical mode and update the issue/PR with clear `searchMode` guidance.

## Reproduction Assessment

Yes. A high-confidence static and unit-level reproduction exists on current main: use QMD with omitted/default `searchMode: "search"` plus `embedInterval`, advance the manager timer, and the current test expects no `update` or `embed` subprocess calls.

## Solution Assessment

Unclear. The linked PR is the right narrow place to solve the observable scheduler bug, but the implementation needs maintainer agreement because current docs and tests intentionally define lexical `search` mode as skipping QMD embedding maintenance.
