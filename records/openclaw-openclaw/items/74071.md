## Summary

Keep open. Current main still implements `isAnnounceSkip` as strict trimmed equality, so multi-line output ending with a standalone `ANNOUNCE_SKIP` line is not suppressed; open PR #74136 is the active closing implementation candidate and should be reviewed or replaced before this issue closes.

## What This Changes

The issue requests that sub-agent announcements be suppressed when multi-line output ends with `ANNOUNCE_SKIP` on its own final line, not only when the entire trimmed output is exactly the token.

## Best Possible Solution

Land PR #74136 or an equivalent narrow patch that updates only the announce-skip predicate for a standalone final-line token, preserves `REPLY_SKIP` and `NO_REPLY` semantics, adds focused helper and announce-flow regression coverage, and aligns user-facing docs or changelog if maintainers make final-line `ANNOUNCE_SKIP` part of the supported contract.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is `isAnnounceSkip("summary\nANNOUNCE_SKIP")` on current main: the helper compares the whole trimmed string to the token and returns false, and both announce delivery paths rely on that helper.

## Solution Assessment

Yes. The narrow maintainable solution is to extend `isAnnounceSkip` for a standalone final `ANNOUNCE_SKIP` line while leaving unrelated reply-suppression tokens strict; the open linked PR is the current implementation candidate.
