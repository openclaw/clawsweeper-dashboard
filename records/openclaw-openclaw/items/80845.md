## Summary

Keep open for human review. Current main only has the synchronous realtime consult path, and this PR adds a new default-off voice-call email feature, but the PR head still returns an email promise before a deliverable recipient is verified, does not track the full background research chain through shutdown, and lacks inspectable proof that the Twilio-to-email path actually delivered mail.

## What This Changes

The PR adds optional voice-call background email delivery for explicit email requests or consult timeouts, with new realtime config fields, consult-tool delivery preference support, a delivery helper, and SDK exports.

## Best Possible Solution

Land this only after recipient availability is verified before acknowledgement, the full research-and-delivery chain is tracked through shutdown, exec-based delivery is explicitly approved and documented, and focused tests plus redacted Twilio-to-email proof are attached.

  - body: Both deferred paths can return an email acknowledgement while `recipientEmail` is unset. The helper only asks a later agent to look up an address in memory and warn on failure, so callers can be told an email is coming when no deliverable destination is known; resolve or require the address before acking, or keep the normal voice path.
