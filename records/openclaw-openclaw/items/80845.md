## Summary

Keep open: the feature direction is plausible, but the PR still has blocking correctness issues, insufficient real behavior proof for the full Twilio-to-email path, and a security/product concern around caller-triggered exec-based email delivery.

## What This Changes

This PR adds optional realtime voice-call background email delivery with email-first and consult-timeout paths, new realtime config fields, a background delivery helper, consult-tool schema and SDK exports, and an abortSignal passthrough.

## Best Possible Solution

Land this only after recipient availability is proven before acknowledgement, the whole research-and-delivery chain is tracked through shutdown, exec-based delivery is approved and documented, focused tests and CI pass, and redacted Twilio-to-email proof is attached.

## Reproduction Assessment

Not applicable as a user-reported bug because this is a new feature PR. The blocking review findings are source-reproducible from the PR head by following the deferred email paths in extensions/voice-call/src/runtime.ts and the delivery helper.

## Solution Assessment

No. The feature should not promise email before a recipient is known, should track the full background chain through shutdown, and should receive explicit approval for the exec-capable delivery path before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Verify a recipient before promising email delivery:** `extensions/voice-call/src/runtime.ts:443-480`
  - body: The email-first path returns an email acknowledgement even when `recipientEmail` is unset. The helper only asks a later agent to look up an address in memory and warn on failure, so callers can be told an email is coming with no known destination; resolve or require the recipient before returning this ack, or keep the voice path.
  - confidence: 0.88
- **[P2] Track the full background consult through shutdown:** `extensions/voice-call/src/runtime.ts:447-452`
  - body: This starts a research promise, but only the later email-delivery promise enters `pendingBackgroundDeliveries`. If the runtime stops while research is still running, `stop()` can flush an empty set and tear down resources before delivery is queued, so track the research-and-delivery chain as one background task.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The patch adds a default-off but caller-triggered background agent with exec access, which needs explicit security/product approval and documentation before merge.

Concerns:

- **[medium] Review the exec-capable email delivery path:** `extensions/voice-call/src/runtime.ts:93`
  - body: `EMAIL_DELIVERY_TOOLS_ALLOW` grants `exec` to the background email delivery agent so a phone-call-triggered flow can run local mail CLI commands, while the existing safe-read-only consult policy excludes exec. This may be acceptable as an opt-in design, but it is a concrete new command-execution surface.
  - confidence: 0.86
