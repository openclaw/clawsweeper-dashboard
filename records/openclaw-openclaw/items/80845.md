## Summary

This PR should stay open: it is a new feature with two source-reproducible blocking defects, insufficient real behavior proof for the promised email delivery path, and a caller-triggered exec surface that needs explicit product/security review.

## What This Changes

The PR adds optional realtime voice-call background email delivery through new realtime config fields, a consult-tool delivery preference, SDK exports, and a background email delivery helper.

## Best Possible Solution

Land this feature only after the branch proves a known recipient or falls back to voice before promising email, tracks the full background research-and-delivery chain through shutdown, receives explicit approval for exec-based delivery, adds focused coverage, and attaches redacted live proof of the full delivery path.

## Reproduction Assessment

Not applicable as a user-reported bug because this is a new feature PR. The blocking patch defects are source-reproducible from the PR head by following the deferred email branches in extensions/voice-call/src/runtime.ts and the delivery helper.

## Solution Assessment

No. The feature direction may be useful, but this patch should not promise email before recipient availability is proven, should track the full background chain, and should get explicit approval for the exec-capable delivery path before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Verify the recipient before returning the email ack:** `extensions/voice-call/src/runtime.ts:443-480`
  - body: Both deferred paths can tell the caller an email is coming while recipientEmail is unset. The helper only asks a later agent to search memory and warn on failure, so callers can get a false delivery promise; resolve or require the address before acking, or keep the normal voice path.
  - confidence: 0.88
- **[P2] Track the research promise for shutdown:** `extensions/voice-call/src/runtime.ts:447-452`

  - confidence: 0.85
