## Summary

Keep open. The async voice-call email-delivery feature is plausible, but the current branch still has blocking policy/security issues, missing inspectable real-behavior proof, and a timer cleanup bug confirmed from the PR head and current main contract.

## What This Changes

The PR adds asynchronous email delivery for realtime voice-call consults with new realtime config fields, delivery preference parsing, a background email helper, SDK exports, and a small consult-tool test update.

## Best Possible Solution

Design async voice-call delivery as an explicit reviewed capability that preserves the realtime policy boundary, redacts call content, clears timeout resources, documents the new config, and includes focused tests plus real Twilio/email proof.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature PR. The blockers are high-confidence from PR-head source review against the current realtime tool-policy contract.

## Solution Assessment

No. The async delivery direction may be useful, but this implementation is not the best path until the command-capable delivery policy is explicit, call content logs are redacted, and the timeout race is cleaned up.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve the realtime tool policy for email delivery:** `extensions/voice-call/src/background-email-delivery.ts:98-119`
  - body: The new email-delivery agent starts another `consultRealtimeVoiceAgent` run without passing `toolsAllow`, while the prompt asks that run to use exec-based email tools. Under `safe-read-only`, the normal realtime consult is constrained to read/web/memory tools, so this caller-triggered path can exceed the configured realtime policy.
  - confidence: 0.91
- **[P1] Stop logging caller prompts and consult answers:** `extensions/voice-call/src/runtime.ts:362-364`
  - body: This log stores JSON tool args and partial transcripts, and the new async paths also log question/result snippets. Voice calls can contain PII, customer data, or secrets, so these logs should keep only structural IDs/statuses or use a redaction boundary.
  - confidence: 0.89
- **[P2] Clear the timeout when the consult wins:** `extensions/voice-call/src/runtime.ts:489-490`
  - body: When the consult finishes before `backgroundConsultTimeoutMs`, the timeout created for the race remains scheduled until it fires. That can keep the process alive and accumulate timers on successful consults; store the handle and clear it when the consult settles.
  - confidence: 0.86
