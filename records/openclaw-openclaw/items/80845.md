The PR adds default-off background email delivery for realtime voice-call consults, with new realtime config fields, a delivery helper, consult-tool deliveryPreference support, SDK exports, and limited test updates.

## Best Possible Solution

Land only after the PR compiles, the deferred consult-and-email chain is tracked through shutdown, recipient availability or failure semantics are explicit before acknowledgement, focused tests cover the new branches, real Twilio-to-email proof is attached, and the exec-capable delivery path is approved or redesigned.

## Solution Assessment

No. The direction may be useful, but the current patch should not merge until it compiles, avoids promising email without a verified recipient path, tracks all deferred work, and gets explicit approval for the exec-capable delivery model.

  - body: The new ack says the system already knows where to send the email, but `recipientEmail` is optional and the delivery helper only tries a later memory lookup.   - confidence: 0.9

  - body: `stop()` only flushes promises added by `spawnEmailDeliveryAgent`, but the email-first and timeout branches return while the background research promise is still untracked. If the runtime stops before that consult resolves, lifecycle resources can shut down before delivery is even queued.
  - confidence: 0.86
