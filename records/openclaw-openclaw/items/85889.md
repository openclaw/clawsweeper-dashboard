## Summary

Keep open: the PR targets the right payload-builder seam and includes useful live-provider context, but the proposed clamp only changes an over-context request to `contextWindow - 1`, which still exceeds the remaining context for the linked high-input failure.

## What This Changes

The PR clamps resolved OpenAI Completions max token params for proxy-like endpoints when the resolved value is at least the model context window, and adds focused builder unit tests.

## Best Possible Solution

Thread a real remaining-output budget into the canonical OpenAI Completions payload path, or clamp at a nearby seam that already has prepared input-size facts, while preserving native/OpenAI-compatible defaults and adding regression coverage for the linked StepFun-style high-input case.

## Reproduction Assessment

Yes. From source inspection, a proxy-like model with `contextWindow=262144`, `maxTokens>=262144`, and any substantial prompt still emits a cap near the full context window, so the linked remaining-budget overflow remains reproducible by construction.

## Solution Assessment

No. The builder seam is a good place to avoid duplicate callers, but `contextWindow - 1` is not the remaining output budget and does not solve the reported strict-provider error once input tokens are present.
