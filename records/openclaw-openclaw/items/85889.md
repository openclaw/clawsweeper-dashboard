## Summary

Keep this PR open: it targets a real source-reproducible strict-provider failure in the canonical OpenAI Completions payload builder, but the latest estimator still undercounts non-Latin input and ignores OpenClaw's effective `contextTokens` cap.

## What This Changes

The PR adds a local remaining-context-budget clamp for proxy-like OpenAI Completions `max_completion_tokens` and adds unit coverage in the transport test file.

## Best Possible Solution

Keep the canonical builder seam, but compute the remaining budget from the effective context cap and the repository's CJK-aware token-pressure helper, then refresh focused tests and live proof for the latest head.

## Reproduction Assessment

Yes. Source inspection of current main shows the builder forwards the resolved output cap directly, and the linked issue/proof describes strict providers rejecting prompt plus output budgets that exceed their context.

## Solution Assessment

No. The builder seam is the right place, but the current implementation duplicates weaker token math instead of using the existing CJK-aware pressure logic and it ignores the documented effective `contextTokens` cap.

- **[P1] Use CJK-aware input token estimates:** `src/agents/openai-transport-stream.ts:3002`
  - body: This estimator still uses raw `chars / 4 * 1.25`, while `src/utils/cjk-chars.ts` documents that this undercounts CJK text by 2-4x and the pre-prompt pressure code already uses `estimateStringChars`.   - body: The new clamp reads only `model.contextWindow`, but OpenClaw documents and carries `contextTokens` as the effective runtime cap when it is lower than the native context window. For a proxy-like model configured with `contextTokens` to match a smaller server/runtime limit, this can still send an output budget that exceeds the intended remaining context; use the effective cap and add coverage for that case.
  - confidence: 0.84
