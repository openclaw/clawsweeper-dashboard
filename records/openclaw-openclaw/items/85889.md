## Summary

Keep open: the payload-builder seam is reasonable and the PR has useful live vLLM proof, but the patch only caps output tokens to `contextWindow - 1`; it does not fix the linked long-input failure where the provider requires `max_tokens <= contextWindow - inputTokens`.

## What This Changes

This PR clamps proxy-like OpenAI-completions max-token fields in `buildOpenAICompletionsParams` and adds unit coverage for proxy-like and native endpoint behavior.

## Best Possible Solution

Keep the builder-seam fix direction, but clamp the final outbound max-token field against a request-specific remaining context budget and cover long-prompt/history cases before treating the linked bug as fixed.

## Reproduction Assessment

Yes. The linked issue gives concrete failing math, and PR-head source inspection shows the patch would still emit `262143` for a `262144` context window regardless of already-used input tokens.

## Solution Assessment

No. The builder seam is the right place, but `contextWindow - 1` is only a partial guard; the maintainable fix needs an input-aware remaining-budget cap or a narrower PR scope.
