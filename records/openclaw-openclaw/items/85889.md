## Summary

Keep open: the linked bug is real and this PR targets the right payload-builder seam, but the new raw character estimator can still undercount non-Latin/CJK input and the visible proof does not show latest-head diagnostic output for that updated estimator path.

## What This Changes

The PR adds a proxy-like OpenAI Completions remaining-context clamp for `max_tokens`/`max_completion_tokens` in `buildOpenAICompletionsParams` and adds focused regression tests.

## Best Possible Solution

Reuse the existing CJK-aware token-pressure estimation path, or factor a shared helper for the builder, then add a non-Latin regression test and latest-head real-provider proof while preserving native OpenAI/OpenRouter behavior and smaller configured caps.

Yes. Source inspection shows current main and the latest release forward the full resolved output cap, and the linked StepFun report gives a concrete context-window/input-token failure shape; I did not replay live StepFun credentials.

## Solution Assessment

No. The builder seam is the right place to solve this, but the raw character estimator is not the best implementation because the repository already has CJK-aware token-pressure utilities for this exact undercount class.
