## Summary

Keep this PR open for normal maintainer review: the latest head fixes the earlier estimator and effective-context objections at the canonical payload-builder seam, and I did not find a blocking correctness or security issue. The remaining concern is an intentional compatibility tradeoff because proxy-like providers will receive smaller output caps when the old payload exceeded the remaining context budget.

## What This Changes

The PR adds a remaining-effective-context clamp for proxy-like OpenAI Completions `max_completion_tokens`/`max_tokens` payloads and adds focused transport regression tests.

## Best Possible Solution

Land the canonical builder-level clamp after maintainers accept the proxy-provider compatibility semantics and required checks pass, then let the linked bug close through the PR.
