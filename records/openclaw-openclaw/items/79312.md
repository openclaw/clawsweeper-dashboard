## Summary

Keep open: current main still has a source-backed status/usage gap for OpenAI Codex OAuth. Chat status resolves usage from the displayed model provider (`openai`) instead of the Codex runtime/auth provider, and CLI status usage relies on ambient config/default agent resolution instead of the status scan's resolved config.

## What This Changes

Restore OpenAI Codex OAuth usage/quota windows in chat `/status` and `openclaw status --usage`, or show a clear diagnostic when usage auth cannot be resolved.

## Best Possible Solution

Carry resolved status runtime facts into provider usage lookup: use `openai-codex` as the usage provider for Codex-harness OpenAI sessions, and call provider usage with the scanned config plus resolved default agent dir for CLI status.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: an `openai/gpt-5.5` status card with effective Codex runtime/auth cannot query `openai-codex` usage because the usage provider is derived from `openai`; the CLI half is source-backed by missing config/agentDir threading but still needs live OAuth proof.

## Solution Assessment

Is this the best way to solve the issue? Yes: reusing the existing status runtime/auth provider and provider-usage seams is narrower than adding a new status mode, config option, or cache layer.
