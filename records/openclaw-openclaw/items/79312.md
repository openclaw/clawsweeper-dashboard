## Summary

Keep open: current docs and code still support provider usage snapshots, but the status paths can miss OpenAI Codex OAuth because `/status` derives usage from the displayed model provider while CLI `status --usage` loads usage without an agent directory.

## What This Changes

Restore OpenAI Codex OAuth usage/quota windows in chat `/status` and `openclaw status --usage`, or show a clear diagnostic when usage auth cannot be resolved.

## Best Possible Solution

Use the same effective auth provider and agent scope already resolved for status auth labels when loading usage, so canonical `openai/*` Codex-runtime sessions request `openai-codex` usage without adding a new user setting.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: a Codex-runtime `openai/gpt-5.5` status card resolves auth as `openai-codex` but usage from `openai`, and CLI status usage omits the active agent directory. A focused unit test can reproduce both without live secrets.

## Solution Assessment

Yes, the narrow maintainable fix is to carry the prepared effective usage provider and agent directory into status usage loading. Avoid adding a config knob or a broad fallback cache.
