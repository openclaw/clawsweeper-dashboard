## Summary

Keep this PR open as a useful but not merge-ready implementation candidate. Current main still lacks importance-weighted temporal decay, but the branch targets the old `src/memory` layout and includes config/docs pieces that are either unwired or already shipped elsewhere.

## What This Changes

The PR adds an optional `importanceBoost` temporal-decay config, marker/file-pattern based importance detection, related memory-search schema/type wiring, session-memory `/reset` docs, and temporal-decay tests.

## Best Possible Solution

Keep the PR open only as source material. A mergeable version should port the feature into the active `memory-core` temporal-decay, hybrid, and manager paths; define deep-merge and validation semantics for `importanceBoost`; regenerate schema artifacts; update docs; and remove already-shipped session-memory reset changes plus unwired `periodicExtraction` unless that behavior is fully implemented.

## Reproduction Assessment

Not applicable as a feature PR rather than a reported runtime bug. Current-main verification is high-confidence: source, docs, schema, and active memory-core call paths show no `importanceBoost` support today.

## Solution Assessment

No. The requested feature is plausible, but this branch is not the best current implementation because it patches the retired memory layout, exposes unwired config, and mixes unrelated shipped reset-doc changes with the ranking feature.

Full review comments:

- **[P2] Port the boost into memory-core:** `src/memory/temporal-decay.ts:4-7`
  - body: Current main runs memory ranking through `extensions/memory-core`, but this patch adds the decay logic and tests under the old `src/memory` path. As a result, the proposed feature would not affect the active memory search implementation after a current-main port unless the memory-core temporal-decay, hybrid, and manager paths are updated too.
  - confidence: 0.93
- **[P2] Merge importanceBoost overrides deeply:** `src/agents/memory-search.ts:363-365`
  - body: This picks either the whole override object or the whole defaults object. A partial override such as only `boostFactor` drops default fields like `enabled` or `patterns`, so inherited importance weighting can silently turn off or change matching behavior.
  - confidence: 0.88
- **[P2] Reject blank importance patterns:** `src/config/zod-schema.agent-runtime.ts:721-722`
  - body: The schema accepts empty strings in `contentMarkers` and `filePatterns`, but `isImportantChunk()` uses substring matching. `includes("")` is always true, so one blank entry makes every result important and effectively suppresses normal decay.
  - confidence: 0.9
- **[P2] Do not accept unwired periodicExtraction config:** `src/config/zod-schema.agent-defaults.ts:123-131`
  - body: The new `periodicExtraction` schema accepts user config, but the runtime types and memory-flush resolver do not carry or read it. That creates a valid-looking setting that has no effect.
  - confidence: 0.86
- **[P2] Regenerate config schema artifacts:** `src/config/zod-schema.agent-runtime.ts:713-726`
  - body: Adding public config fields only to the Zod schemas leaves the generated schema/help/label artifacts stale, so schema lookup and UI-driven configuration will not expose the new settings.
  - confidence: 0.86
- **[P3] Align boostFactor validation with runtime behavior:** `src/config/zod-schema.agent-runtime.ts:718`
  - body: `z.number().positive()` accepts values between 0 and 1, but the implementation only applies the boost when `boostFactor > 1`. Those accepted values become silent no-ops instead of invalid config.
  - confidence: 0.84
