This PR adds an optional `importanceBoost` temporal-decay config with marker/path detection and tests, wires it through memory-search config schemas/types, and updates hook docs for `/reset` session-memory behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Port the boost into memory-core:** `src/memory/temporal-decay.ts:4-7`
  - body: Current main runs memory ranking through `extensions/memory-core`, but this patch adds the core decay logic under the old `src/memory` path. A current-main port would leave the active search implementation unchanged unless the memory-core temporal-decay, hybrid, and manager paths are updated.
  - confidence: 0.93
- **[P2] Merge importanceBoost overrides deeply:** `src/agents/memory-search.ts:363-365`
  - body: This selects either the whole override object or the whole defaults object. A partial agent override such as only `boostFactor` drops default fields like `enabled` or `patterns`, so inherited importance weighting can silently turn off or change matching behavior.
  - confidence: 0.88
- **[P2] Reject blank importance patterns:** `src/config/zod-schema.agent-runtime.ts:721-722`
  - body: The schema accepts empty strings in `contentMarkers` and `filePatterns`, but the implementation uses substring matching. Since `.includes("")` is always true, one blank entry makes every result important and suppresses normal decay across the result set.
  - confidence: 0.9
- **[P2] Do not accept unwired periodicExtraction config:** `src/config/zod-schema.agent-defaults.ts:123-131`
  - body: The new `periodicExtraction` schema validates user config, but the runtime types and memory-flush resolver do not carry or read it. That creates a valid-looking setting that has no effect.
  - confidence: 0.86
- **[P2] Regenerate config schema artifacts:** `src/config/zod-schema.agent-runtime.ts:713-726`
  - body: Adding public config fields only to the Zod schemas leaves the generated schema/help/label artifacts stale. `config.schema.lookup` serves the generated base schema, so the new settings would not appear in schema-driven tooling until those artifacts are refreshed.
  - confidence: 0.86
- **[P3] Align boostFactor validation with runtime behavior:** `src/config/zod-schema.agent-runtime.ts:718`
  - body: `z.number().positive()` accepts values between 0 and 1, but the implementation only applies the boost when `boostFactor > 1`. Those accepted values become silent no-ops instead of invalid config.
  - confidence: 0.84
