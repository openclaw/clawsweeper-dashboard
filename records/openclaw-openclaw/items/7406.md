## Summary

Keep open: current main still learns Telegram forum topic names, but the known topic name stops at Telegram prompt/context metadata and is not projected into stored session metadata, gateway session rows, or the Control UI dropdown. The linked implementation PRs are closed unmerged, and the remaining open UI cleanup PR is adjacent rather than a complete fix.

## What This Changes

Request to display Telegram forum topic sessions with human-readable group/topic names instead of raw `:topic:<id>` session keys in the session dropdown.

## Best Possible Solution

Add a generic explicit thread/session display-label path, populate it from Telegram cached topic names for forum-topic sessions, expose it through `sessions.list`, and have the UI prefer that label while preserving non-Telegram behavior.

## Reproduction Assessment

Yes, source-level: current main can set `TopicName` for a Telegram forum message, but the session metadata, gateway row, and UI selector code never consume `TopicName` as a display label. I did not run a live Telegram/UI reproduction in this read-only review.

## Solution Assessment

Yes. The maintainable fix is to reuse the existing generic `ThreadLabel` or an explicit session display-label path; Telegram-specific UI key parsing would create a second source of truth.
