## Summary

Keep open. Current main has the Telegram topic-name source and exposes `TopicName` in inbound context, but the known topic name is still not carried into persisted session display metadata, Gateway session rows, or the Control UI session selector.

## What This Changes

The issue requests human-readable Telegram forum topic names in the Control UI session dropdown instead of raw topic/session keys.

## Best Possible Solution

Add a narrow generic session display/thread-label path, have the Telegram plugin populate it from the cached forum topic name, expose it through `sessions.list`, and make the Control UI selector prefer that explicit label while preserving existing non-Telegram labels.

## Reproduction Assessment

Yes from source inspection: current main learns and exposes `TopicName`, but stored session metadata, Gateway rows, and Control UI display resolution do not consume it as a readable topic label. I did not run a live Telegram reproduction in this read-only cleanup review.

## Solution Assessment

Yes: the best fix is the generic thread/display-label propagation path already suggested in the issue, not a Telegram-only UI key parser or a new config `name` field.
