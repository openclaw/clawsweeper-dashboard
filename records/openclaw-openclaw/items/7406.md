## Summary

Keep open. Current main has the Telegram topic-name source and exposes `TopicName` to prompt metadata, but the value still does not flow into stored session metadata, Gateway session rows, or the Control UI session selector; the directly related implementation PRs are closed unmerged and the remaining open UI PR is only adjacent cleanup.

## What This Changes

Request to show Telegram forum topic sessions in the session dropdown with human-readable group and topic names instead of raw topic-qualified session keys.

## Best Possible Solution

Add an additive generic thread/topic display-label path in session metadata, populate it from Telegram's cached forum topic name for topic sessions, expose it through `sessions.list`, and have the Control UI selector prefer that explicit label while preserving existing non-Telegram fallbacks.

## Reproduction Assessment

Yes. Build or receive a Telegram forum-topic message after the topic name has been learned: current tests show `TopicName` reaches Telegram context and prompt metadata, while the inspected session metadata, Gateway row, and UI selector paths do not consume it.

## Solution Assessment

Yes. A generic display-label or `ThreadLabel` propagation path with Telegram supplying the known topic name is the narrowest maintainable fix; a UI-only key formatter or Telegram-specific core display rule would leave other session consumers inconsistent.
