## Summary

Keep open: current main still learns Telegram forum topic names, but the value stops at inbound context metadata and is not wired into stored session display metadata, gateway session rows, or the Control UI selector. Related duplicate issues were closed in favor of this canonical tracker, while the open UI cleanup PRs are adjacent visual work rather than the missing metadata bridge.

## What This Changes

The issue requests human-readable Telegram forum topic names in the session dropdown instead of raw topic session keys or indistinguishable group-only labels.

## Best Possible Solution

Add a generic session thread/display-label metadata path, populate it from Telegram cached TopicName for forum-topic sessions, expose it through sessions.list, and have the Control UI prefer that explicit label while preserving non-Telegram behavior.

## Reproduction Assessment

Yes. Source inspection on current main shows Telegram can produce TopicName for forum messages, but SessionEntry, GatewaySessionRow construction, and the Control UI selector do not consume it for display labels.

## Solution Assessment

Yes. The maintainable direction is a generic thread/display-label bridge from Telegram-owned topic metadata into session rows; a UI-only key formatter or Telegram-specific core parser would leave stored metadata incomplete.
