## Summary

Keep open. Current main has the Telegram topic-name data source and prompt metadata, but the value still does not flow into stored session display metadata, Gateway session rows, or the Control UI session selector; the open related UI PR is adjacent cleanup rather than the full metadata propagation fix.

## What This Changes

Show human-readable Telegram forum topic names in session dropdown labels instead of raw topic-qualified session keys.

## Best Possible Solution

Add an additive generic thread/topic display-label path in session metadata, populate it from Telegram cached forum topic names, expose it through sessions.list, and make the Control UI prefer that explicit label while preserving existing non-Telegram fallbacks.

## Reproduction Assessment

Yes. The source path is high-confidence: a Telegram forum-topic message can populate ctx.TopicName, while session metadata, Gateway row construction, and the UI selector do not consume that value.

## Solution Assessment

Yes. A generic display-label or ThreadLabel propagation path with Telegram supplying the known topic name is the narrowest maintainable direction; a UI-only key formatter or Telegram-specific core rule would leave other session consumers inconsistent.

Fix the Telegram forum-topic session label gap tracked by https://github.com/openclaw/openclaw/issues/7406.
