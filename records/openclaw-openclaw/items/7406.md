## Summary

Current main still learns Telegram forum topic titles but does not propagate them into stored session display metadata, Gateway session rows, or the Control UI selector. The direct implementation PRs are closed unmerged and live search found no open replacement, so this remains the canonical bounded tracker.

## What This Changes

Show human-readable Telegram forum topic names instead of raw or indistinguishable topic session labels in the Control UI session selector.

## Best Possible Solution

Propagate one explicit generic human thread label from the Telegram plugin through canonical session metadata and Gateway serialization, then make the Control UI prefer it while preserving all existing non-Telegram labels and fallbacks.

## Reproduction Assessment

Yes, source inspection provides a high-confidence reproduction path: Telegram emits `TopicName`, but the session metadata, Gateway row, and UI row contracts do not carry or consume it. A live Telegram setup was not executed during this read-only review.

## Solution Assessment

Yes. A generic explicit thread-label path is the narrowest maintainable solution; Telegram-specific UI key parsing or a new per-topic config name would duplicate policy and put ownership in the wrong layer.
