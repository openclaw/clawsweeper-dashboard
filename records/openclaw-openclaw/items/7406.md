## Label Justifications

- `P2`: The missing label impairs a real multi-topic Telegram workflow but has bounded blast radius and no availability or security impact.
- `impact:session-state`: Distinct Telegram topic sessions cannot be reliably identified because their displayed session metadata omits the known topic title.
- `impact:message-loss`: Indistinguishable topic entries can cause an operator to select and send through the wrong Telegram topic session.

## Summary

Keep open. Current main and v2026.6.5 learn Telegram forum topic titles, but the value still stops at inbound context and is not propagated into stored session display metadata, Gateway session rows, or the Control UI selector; all directly related implementation PRs are closed unmerged, so this remains the canonical tracker.

## What This Changes

Propagate known Telegram forum topic titles into persisted session display metadata and human-readable Control UI session labels.

## Best Possible Solution

Add one generic explicit thread/session display-label field, populate it from Telegram's existing cached topic title, persist and expose it through `sessions.list`, and make the Control UI prefer it without new config or Telegram-specific key parsing.

## Reproduction Assessment

Yes at source level: current main obtains `TopicName` but has no path carrying it into session display metadata or the Control UI selector. A live Telegram transport reproduction was not run during this review.

## Solution Assessment

Yes. A generic explicit thread/session label reuses the existing topic cache and established ownership boundaries, avoiding a new configuration surface, background synchronization, or Telegram-specific UI heuristics.
