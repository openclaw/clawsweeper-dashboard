## Summary

Keep open: current main still captures Telegram forum topic names for context, but does not carry them into stored session display metadata, gateway session rows, or the Control UI session selector. The related implementation attempts are closed unmerged, and the open UI cleanup PR is adjacent rather than a complete metadata propagation fix.

## What This Changes

Display Telegram forum-topic sessions in the session dropdown with human-readable group and topic names instead of raw topic IDs.

## Best Possible Solution

Add a generic explicit thread/session display label, populate it from Telegram cached topic names for forum-topic sessions, expose it through sessions.list, and have the UI prefer it while preserving existing non-Telegram labels.

## Reproduction Assessment

Yes, source-level. A Telegram forum message can set TopicName, but current session metadata, gateway rows, and the UI selector never consume TopicName as a display label; I did not run live Telegram/UI proof.

## Solution Assessment

Yes. The narrow maintainable fix is to reuse the generic ThreadLabel or explicit session display-label path; Telegram-specific UI key parsing would create a weaker second source of truth.
