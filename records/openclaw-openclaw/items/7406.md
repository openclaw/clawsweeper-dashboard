## Summary

Keep open: current main still learns Telegram forum topic names, but it does not carry them into persisted session display metadata, gateway session rows, or the Control UI selector label path. The earlier implementation PRs were closed unmerged, and the open UI cleanup PR is adjacent rather than a complete backend-to-UI fix.

## What This Changes

Requested Telegram forum-topic sessions should show human-readable topic names in the session dropdown instead of raw topic/session keys.

## Best Possible Solution

Add a generic session thread/display-label path, populate it from Telegram cached TopicName for forum-topic sessions, expose it through sessions.list/GatewaySessionRow, and make the Control UI prefer that explicit label without Telegram-only key parsing in core.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: a Telegram forum message can produce TopicName, but SessionEntry, gateway row construction, and the Control UI selector do not consume that value for display labels.

## Solution Assessment

Yes. The narrow maintainable solution is to reuse the existing topic-name cache and add a generic session display/thread-label bridge; a UI-only cleanup or Telegram-specific core formatter would leave the backend metadata gap unresolved.
