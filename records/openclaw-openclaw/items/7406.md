## Summary

Keep open: current main still learns Telegram forum topic names, but the value stops at inbound context metadata and is not persisted or exposed as session display metadata for gateway rows or the Control UI selector. The related duplicate reports are closed in favor of this canonical tracker, while the open UI cleanup PRs are adjacent and do not complete the metadata bridge.

## What This Changes

The issue asks for Telegram forum-topic sessions to show a human-readable group and topic label in the session dropdown instead of raw numeric topic session keys.

## Best Possible Solution

Add a generic thread/display-label metadata bridge from Telegram-owned TopicName into stored session entries, gateway sessions.list rows, and the Control UI selector while preserving existing non-Telegram labels.

## Reproduction Assessment

Yes. Source inspection on current main shows Telegram can produce TopicName for forum messages, but the session metadata, gateway row, and Control UI selector do not consume that value for display labels.

## Solution Assessment

Yes. A generic session thread/display-label bridge is the narrow maintainable direction; a UI-only key formatter or Telegram-specific core parser would leave stored metadata incomplete.
