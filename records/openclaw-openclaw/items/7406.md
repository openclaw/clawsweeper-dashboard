## Summary

Keep open. Current main already learns Telegram forum topic names and exposes them to prompt metadata, but it still does not promote the known topic name into persisted session display metadata, gateway session rows, or the Control UI session selector.

## What This Changes

The issue requests human-readable Telegram forum topic names in the session dropdown instead of raw or indistinct Telegram topic session keys.

## Best Possible Solution

Add a narrow generic thread/session display-label path, populate it from Telegram cached TopicName for forum-topic sessions, expose it through sessions.list, and make the Control UI prefer that explicit label while preserving non-Telegram labels.

## Reproduction Assessment

Yes, source-level: Telegram forum messages can set TopicName, but current session metadata, gateway rows, and the Control UI selector never consume TopicName as a display label. I did not run a live Telegram/UI repro in this read-only pass.

## Solution Assessment

Yes, the best fix is a generic explicit thread/session label path rather than parsing Telegram session keys in the UI. Prior unmerged PRs and the open synthetic-label cleanup do not carry the topic name through the full backend-to-UI chain.
