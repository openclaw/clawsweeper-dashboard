## Summary

Keep open: current main still learns Telegram forum topic names, but the value only reaches inbound metadata, prompt metadata, and hooks. Stored session metadata, gateway session rows, and the Control UI selector still have no display-label path that consumes the topic name, and the related implementation PRs are closed unmerged while the open UI cleanup PR is only adjacent.

## What This Changes

Request that Telegram forum-topic sessions show human-readable group/topic labels in the Control UI session dropdown instead of raw topic keys or indistinguishable group labels.

## Best Possible Solution

Add a generic explicit thread/session display label in session metadata, populate it from Telegram cached topic names for forum-topic sessions, expose it through sessions.list, and have the UI prefer it.

## Reproduction Assessment

Yes, source-level. A Telegram forum message can set TopicName, but current session metadata, gateway rows, and the UI selector never consume TopicName as a display label; I did not run live Telegram/UI proof.

## Solution Assessment

Yes. The narrow maintainable fix is to reuse the generic ThreadLabel/conversation-label or session display-label path; Telegram-specific UI key parsing or a core-only formatter would be weaker.
