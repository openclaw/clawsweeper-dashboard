## Summary

Keep open. Current main has the Telegram topic-name source and prompt metadata, but the value still does not flow into stored session metadata, Gateway session rows, or the Control UI session selector; the complete related PR attempts are closed unmerged, and the remaining open UI PR is only adjacent cleanup.

## What This Changes

The issue asks OpenClaw to show Telegram forum topic sessions with human-readable group/topic labels in the session dropdown instead of raw internal topic keys.

## Best Possible Solution

Add an additive generic thread/topic display-label path in session metadata, populate it from Telegram's cached forum topic name for topic sessions, expose it through sessions.list, and have the Control UI selector prefer that explicit label while preserving existing non-Telegram fallbacks.

## Reproduction Assessment

Yes. A Telegram forum-topic message with a learned topic name will populate TopicName in the Telegram context and prompt metadata, while the inspected session metadata, Gateway row, and UI selector paths do not consume that value.

## Solution Assessment

Yes. A generic display-label or ThreadLabel propagation path with Telegram supplying the known topic name is the narrowest maintainable fix; a UI-only key formatter or Telegram-specific core rule would leave other session consumers inconsistent.
