## Summary

Keep open. Current main captures Telegram forum topic names for inbound context and prompt metadata, but the value still does not flow into stored session display metadata, Gateway session rows, or the Control UI session selector; the open related UI PR is adjacent cleanup rather than the full metadata propagation fix.

## What This Changes

Request to show Telegram forum-topic sessions in the session dropdown with human-readable Telegram group/topic names instead of raw keys or numeric topic IDs.

## Best Possible Solution

Add an additive generic thread/topic display-label path through Telegram context, session metadata, Gateway sessions.list, and the Control UI selector, with focused regression coverage and a user-facing changelog entry.

## Reproduction Assessment

Yes. Source inspection on current main shows Telegram forum messages can populate ctx.TopicName, while SessionEntry, deriveGroupSessionPatch, Gateway row construction, and the UI selector do not consume that value for visible session labels.

## Solution Assessment

Yes. A generic display-label or ThreadLabel propagation path with Telegram supplying the known topic name is the narrowest maintainable direction; a UI-only key formatter or Telegram-specific core rule would leave other session consumers inconsistent.

Fix the Telegram forum-topic session-label gap tracked by https://github.com/openclaw/openclaw/issues/7406. Add a narrow additive generic thread/topic display-label path, have the Telegram plugin populate it from the existing cached `TopicName` for forum-topic sessions, expose it through Gateway session rows, and make the Control UI prefer that explicit label while preserving existing non-Telegram fallbacks. Keep core generic: no hard-coded Telegram behavior in shared session display code beyond plugin-supplied metadata.
