## Summary

Keep open: current main learns Telegram forum topic names, but the value still stops at inbound context/prompt metadata and is not carried into persisted session labels, gateway session rows, or the Control UI selector. The related closed PRs were not merged, and the remaining open UI cleanup PR is adjacent rather than a complete fix for Telegram topic names.

## What This Changes

The issue requests human-readable Telegram forum topic names in the session dropdown instead of raw topic-scoped session keys or generic group labels.

## Best Possible Solution

Add a generic session display/thread-label path, populate it from Telegram's cached TopicName for forum-topic sessions, expose it through sessions.list, and have the Control UI prefer it without hardcoded Telegram IDs or overriding explicit user labels.

## Reproduction Assessment

Yes, source-level: current main caches TopicName for Telegram forum topics, but session metadata, gateway rows, and UI display resolution never consume it. I did not run live Telegram or browser proof in this read-only review.

## Solution Assessment

Yes, but not as a Telegram-specific hardcoded UI formatter. The maintainable fix is a narrow generic display-label/thread-label metadata path that Telegram fills from cached TopicName while preserving existing non-Telegram labels.
