## Summary

Keep open: current main can learn Telegram topic names, but it still exposes them only as inbound context metadata; stored session metadata, gateway rows, and the Control UI selector still depend on label/displayName values built from group subject/id, so the user-visible topic-name label gap remains.

## What This Changes

Show Telegram forum-topic sessions in the session dropdown with human-readable group/topic names instead of raw or ambiguous topic IDs.

## Best Possible Solution

Add an explicit generic session thread/display-label path, populate it from Telegram’s cached forum TopicName for topic sessions, expose it through sessions.list, and have the Control UI prefer that label without hard-coding Telegram-only formatting in core.

## Reproduction Assessment

Yes, by source inspection rather than a live Telegram run: current main can produce TopicName for forum topics, but the stored SessionEntry, gateway row, and Control UI selector paths never consume it for display labels.

## Solution Assessment

Yes, the narrow maintainable direction is to reuse the existing topic-name source and add a generic session display/thread-label seam; open #51124 is useful adjacent UI cleanup but does not complete the backend metadata path.
