## Summary

Keep open: current main has a Telegram forum topic-name source, but the value still stops at inbound context metadata and is not persisted or preferred through session metadata, gateway rows, and the Control UI selector. The closed implementation PRs were not merged, and the open UI cleanup PR is adjacent rather than a complete backend-to-UI topic label fix.

## What This Changes

Show human-readable Telegram forum topic names in session dropdown labels instead of raw topic-scoped session keys.

## Best Possible Solution

Add a generic session thread/display-label path, populate it from Telegram cached TopicName for forum-topic sessions, expose it through sessions.list/GatewaySessionRow, and have the Control UI prefer that explicit label without hard-coding Telegram-only parsing in core.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: a Telegram forum message can produce TopicName, but SessionEntry, gateway row construction, and the Control UI selector do not consume that value for display labels. I did not run a live Telegram forum scenario in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is to reuse the existing topic-name cache and add a generic session display/thread-label seam; a UI-only cleanup or Telegram-specific core formatter would leave the backend metadata gap unresolved.
