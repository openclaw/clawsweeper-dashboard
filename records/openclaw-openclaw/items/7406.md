## Summary

Keep open: current main still captures Telegram forum topic names, but the value stops at Telegram inbound metadata and is not carried into stored session display metadata, gateway session rows, or the Control UI selector. Recent discussion also treats this issue as the canonical tracker; the linked implementation PRs are closed unmerged and the open UI cleanup PR is adjacent, not a complete fix.

## What This Changes

Show Telegram forum-topic sessions in the session dropdown with human-readable group/topic names instead of raw topic-qualified session keys.

## Best Possible Solution

Add a generic explicit thread/session display-label path, populate it from Telegram cached TopicName for forum-topic sessions, expose it through sessions.list, and make the Control UI prefer that label without parsing raw Telegram keys.

## Reproduction Assessment

Yes, source-level: Telegram forum messages can set TopicName, but current session metadata, gateway rows, and the Control UI selector never consume TopicName as a display label. I did not run a live Telegram/UI reproduction in this read-only pass.

## Solution Assessment

Yes. The maintainable direction is a generic explicit thread/session display label populated by Telegram from the cached topic name; Telegram-specific UI key parsing or a core-only format special case would be weaker.
