## Summary

Keep open: current main still learns Telegram TopicName, but the value stops at inbound context and is not persisted into session display metadata, exposed through gateway session rows, or rendered by the Control UI selector. The open UI cleanup at https://github.com/openclaw/openclaw/pull/68850 is adjacent visual cleanup, not the canonical metadata bridge for this issue.

## What This Changes

Display Telegram forum-topic sessions in the session dropdown with human-readable group/topic names instead of raw topic session keys.

## Best Possible Solution

Carry known Telegram topic names through a generic session thread/display label in session metadata and gateway rows, then have the Control UI selector prefer that explicit label while preserving existing non-Telegram labels.

## Reproduction Assessment

Yes, source-level. Current main can populate Telegram TopicName for forum messages, but the session metadata, gateway row, and Control UI selector paths do not consume it for visible labels.

## Solution Assessment

Yes. A generic session thread/display-label bridge is the narrow maintainable solution; UI-only key parsing or a Telegram-specific parser in core would leave stored metadata incomplete.
