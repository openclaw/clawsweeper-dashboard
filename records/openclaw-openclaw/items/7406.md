## Summary

Keep open. Current main has a persisted Telegram forum topic-name source and exposes `TopicName` in inbound context, but that value still is not propagated into durable session display metadata, gateway session rows, or the Control UI session selector.

## What This Changes

The issue requests human-readable Telegram forum topic names in session dropdown labels instead of raw Telegram topic session keys.

## Best Possible Solution

Add a narrow generic session display-label/thread-label propagation path, have Telegram populate it from the existing cached topic name for forum-topic sessions, and make gateway rows plus the Control UI selector prefer that explicit label without adding config or UI string parsing.

## Reproduction Assessment

Yes, at source level: current main has tests and code proving `TopicName` is available in Telegram context, while the session metadata, gateway row, and UI selector paths do not consume it. I did not run a live Telegram setup in this read-only review.

## Solution Assessment

Yes, the best fix is to reuse or extend the generic thread/display-label session metadata path. A Telegram-only UI parser or new config field would create a second source of truth and would be less maintainable.
