## Summary

Keep open. Current main has the Telegram topic-name data source, but the known topic name still only reaches inbound context/prompt metadata and is not promoted into persisted session label/displayName data used by sessions.list and the Control UI selector.

## What This Changes

The issue requests human-readable Telegram forum topic names in the session dropdown instead of raw or indistinguishable topic session labels.

## Best Possible Solution

Add a narrow generic thread/session display-label path, have Telegram populate it from cached TopicName for forum-topic sessions, expose it through sessions.list, and make the Control UI selector prefer that explicit label without changing non-Telegram labels.

## Reproduction Assessment

Yes, source-level: Telegram forum messages can set TopicName, but current session metadata, gateway rows, and the Control UI selector never consume TopicName as a display label. I did not run a live Telegram/UI repro in this read-only review.

## Solution Assessment

Yes, the best fix is the narrow generic label/thread-label path rather than a Telegram-specific UI parser or hardcoded topic map. Prior closed PRs either stayed unmerged, used stale paths, or did not preserve the metadata through the full session row and UI chain.
