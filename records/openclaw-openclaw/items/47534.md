## Summary

Keep open. Current main still lacks the requested `openclaw doctor` diagnostic or repair path for stale persisted `skillsSnapshot` prompt paths, while the issue discussion provides concrete persisted-state evidence and a narrow remediation boundary.

## What This Changes

Request `openclaw doctor` detect and guide remediation for stale session skill snapshots whose cached runtime paths no longer match the active OpenClaw install.

## Best Possible Solution

Add a focused doctor state diagnostic that resolves the active bundled skill root, scans persisted snapshot prompt/path metadata for inactive or temp-backed roots, reports cached-session metadata drift separately from live runtime health, and in repair mode backs up the store before invalidating only affected `skillsSnapshot` fields.

## Reproduction Assessment

Yes. Source inspection shows a session entry with stale `skillsSnapshot.prompt` paths would be loaded while doctor never checks that field, and the runtime reuse paths refresh only by version/filter predicates.

## Solution Assessment

Yes. A doctor/session-store diagnostic and explicit repair flow is the narrowest maintainable solution; hidden startup or load-time mutation would change long-lived session state outside an operator-requested repair pass.
