## Summary

Keep open. Current main still lacks a doctor diagnostic or repair path for stale persisted skill snapshot prompt paths, while the discussion and source inspection show a concrete, bounded session-store drift case.

## What This Changes

Request `openclaw doctor` to detect and safely remediate persisted session skill snapshots that reference old, missing, or temp-backed OpenClaw runtime roots.

## Best Possible Solution

Add a focused doctor state-integrity diagnostic that resolves the active bundled skills root, scans persisted `skillsSnapshot.prompt` paths for missing/temp/inactive roots, and in repair mode backs up the store before deleting only affected snapshot fields.

## Reproduction Assessment

Yes. Source inspection plus the reporter's `sessions.json` example give a high-confidence path: seed a session entry whose `skillsSnapshot.prompt` contains an old `/opt/homebrew` or `/tmp` OpenClaw `SKILL.md` path, then run current doctor; no doctor code reads that field.

## Solution Assessment

Yes. A doctor/session-store diagnostic with opt-in repair is the narrowest maintainable solution; startup or load-time mutation would change long-lived session state outside an operator-requested repair pass.

## Repair Work Prompt

Implement a focused fix for https://github.com/openclaw/openclaw/issues/47534. Add an `openclaw doctor` state-integrity diagnostic that resolves the active bundled skills directory/runtime root and scans persisted agent session stores for `skillsSnapshot.prompt` entries that reference missing, temp-backed (`/tmp`, `/private/tmp`), or inactive OpenClaw bundled skill roots. The output should distinguish healthy live runtime paths from stale cached session metadata. In repair mode, back up the session store and delete only the affected `skillsSnapshot` fields so they rebuild on the next run; do not wipe sessions or add startup/load-time mutation. Cover CLI and gateway session-store behavior with targeted tests and add user-facing docs/changelog if behavior changes.
