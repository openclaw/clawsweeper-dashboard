## Summary

Keep open. Current main still lacks a doctor diagnostic or repair for stale persisted `skillsSnapshot.prompt` path metadata, while runtime reuse still depends on snapshot version and skill-filter state rather than validating cached paths against the active bundled skills root; the related draft at https://github.com/openclaw/openclaw/pull/71497 is adjacent refresh-version work, not the requested doctor warning/remediation.

## What This Changes

Add an `openclaw doctor` diagnostic that detects stale cached session skill snapshots pointing at inactive, missing, or temp-backed runtime roots after OpenClaw runtime path changes.

## Best Possible Solution

Add a focused doctor state-integrity check that reports cached-session skill metadata drift separately from live runtime health and, in repair mode, invalidates only affected `skillsSnapshot` fields after backing up the store.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a session store entry whose `skillsSnapshot.prompt` contains an old install root or `/tmp` path while current main's doctor checks do not inspect that field and the runtime refresh predicate checks only version/filter state.

## Solution Assessment

Yes. A narrow doctor diagnostic plus safe snapshot invalidation is the best fit for this operator-facing drift case; the adjacent refresh-version PR may reduce some stale cases but does not surface or repair cached-session metadata drift for users.
