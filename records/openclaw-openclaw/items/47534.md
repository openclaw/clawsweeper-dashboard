## Summary

Keep open. Current main still lacks a doctor diagnostic or repair path for stale persisted `skillsSnapshot.prompt` paths, and the issue discussion gives concrete persisted-state evidence plus a safe narrow remediation.

## What This Changes

Request `openclaw doctor` detection and guided remediation for stale cached session skill snapshots after runtime root changes.

## Best Possible Solution

Add a focused doctor state-integrity check that resolves the active bundled skills root, scans persisted session snapshot prompts for missing/inactive/temp runtime roots, and in repair mode backs up the store before invalidating only affected `skillsSnapshot` fields.

## Reproduction Assessment

Yes. Source inspection shows a `sessions.json` entry with `skillsSnapshot.prompt` pointing at `/opt/homebrew`, `/tmp`, `/private/tmp`, or another inactive runtime root would still be loaded while doctor never checks that field.

## Solution Assessment

Yes. A doctor/session-store diagnostic with explicit repair is the narrowest maintainable direction; startup or load-time mutation would change long-lived session state outside an operator-requested repair pass.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/47534 by adding a focused `openclaw doctor` state-integrity diagnostic for stale persisted skills snapshots. Resolve the active bundled skills/runtime root, scan persisted session entries for cached `skillsSnapshot.prompt` paths that reference inactive install roots, missing roots, `/tmp`, or `/private/tmp`, and report this separately as stale cached session metadata while making clear that the live runtime may be healthy. In `doctor --fix`/repair mode, back up the session store and remove only the affected `skillsSnapshot` fields so the next turn rebuilds them; do not wipe whole sessions, mutate state at startup/load time, or make the check extension-specific. Update focused tests, docs for doctor output/repair behavior, and CHANGELOG.md because this is user-facing. Note that draft PR #71497 is adjacent restart-version work and should not be treated as covering this doctor diagnostic unless it has since grown the same check.
