## Summary

Keep open. Current main still lacks the requested doctor diagnostic: state integrity reads the session store for transcript/orphan checks but does not inspect persisted skill snapshots for stale bundled-skill paths, while both CLI and gateway paths can reuse persisted snapshots based on version and skill-filter state. The reporter also supplied concrete persisted-state evidence and a narrow manual remediation, so this is actionable.

## What This Changes

Add an `openclaw doctor` diagnostic and safe repair path for stale cached session skill snapshots after OpenClaw runtime roots change.

## Best Possible Solution

Add a focused doctor state diagnostic that resolves the active bundled skills directory, scans persisted session `skillsSnapshot.prompt` and `skillsSnapshot.resolvedSkills[].filePath` values for missing, inactive, or temp-backed bundled runtime roots, reports cached-session metadata drift separately from live runtime health, and in repair mode backs up the session store before deleting only affected snapshot fields.

## Reproduction Assessment

Yes. A persisted session entry with `skillsSnapshot.prompt` or `skillsSnapshot.resolvedSkills[].filePath` pointing at an old bundled runtime root is enough to verify the gap: current doctor reads the store but never inspects those fields, and runtime refresh only checks snapshot presence, version, and skill filter.

## Solution Assessment

Yes. A doctor/session-store diagnostic and explicit repair is the narrowest maintainable solution for operator-facing drift; startup/load-time mutation would be broader and would change long-lived session state outside an intentional repair flow.
