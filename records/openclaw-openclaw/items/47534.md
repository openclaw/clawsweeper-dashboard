## Summary

Keep open. Current main still lacks the requested `openclaw doctor` diagnostic/repair for stale cached session skill snapshots, while the issue discussion gives concrete persisted-state evidence and a narrow verified remediation path.

## What This Changes

Request an `openclaw doctor` check that detects and safely invalidates session-cached skill snapshots whose absolute paths point at old, inactive, missing, or temp-backed OpenClaw runtime roots.

## Best Possible Solution

Add a focused doctor state diagnostic that resolves the active bundled skills directory, scans persisted `skillsSnapshot.prompt` and `skillsSnapshot.resolvedSkills[].filePath` values for missing, inactive, or temp-backed bundled runtime roots, reports cached-session metadata drift separately from live runtime health, and in repair mode backs up the session store before deleting only affected snapshot fields.

## Reproduction Assessment

Yes. A session-store entry whose `skillsSnapshot.prompt` or `skillsSnapshot.resolvedSkills[].filePath` points at `/opt/homebrew`, `/tmp`, `/private/tmp`, or another inactive bundled runtime root is enough to verify the current gap because doctor loads the store but never checks those fields.

## Solution Assessment

Yes. A doctor/session-store diagnostic with an explicit repair flow is the narrowest maintainable solution; startup or load-time mutation would change long-lived session state outside an operator-requested repair pass.
