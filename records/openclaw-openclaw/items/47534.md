## Summary

Keep open. Current main still does not add the requested `openclaw doctor` diagnostic or repair path for stale cached session skill snapshots, while the issue discussion provides concrete persisted-state evidence and a narrow verified remediation.

## What This Changes

Add an `openclaw doctor` diagnostic and safe repair path for cached session `skillsSnapshot` metadata that references old, missing, or temp-backed OpenClaw runtime roots after the active bundled skills root changes.

## Best Possible Solution

Add a focused doctor state diagnostic that resolves the active bundled skills directory, scans persisted session `skillsSnapshot.prompt` and `skillsSnapshot.resolvedSkills[].filePath` values for missing, inactive, or temp-backed bundled runtime roots, reports cached-session metadata drift separately from live runtime health, and in repair mode backs up the session store before deleting only affected snapshot fields.

## Reproduction Assessment

Yes. A session-store entry with `skillsSnapshot.prompt` or `skillsSnapshot.resolvedSkills[].filePath` pointing at `/opt/homebrew`, `/tmp`, `/private/tmp`, or another inactive bundled runtime root is enough to verify the current gap because doctor loads the store but never checks those fields.

## Solution Assessment

Yes. A doctor/session-store diagnostic and explicit repair flow is the narrowest maintainable solution; startup or load-time migration would mutate long-lived session state outside an operator-requested repair pass.
