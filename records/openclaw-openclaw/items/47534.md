## Summary

Keep open. Current main still lacks a doctor diagnostic or repair for stale persisted `skillsSnapshot` path metadata, while the runtime can continue reusing cached snapshots based on version and skill-filter checks; the related open restart-version PR is adjacent but does not implement the requested doctor warning/remediation.

## What This Changes

Add an `openclaw doctor` diagnostic and safe remediation path for persisted session skill snapshots that reference inactive, missing, or temp-backed runtime roots after install or runtime path changes.

## Best Possible Solution

Add a focused doctor state-integrity check that resolves the active bundled skills directory, scans persisted session skill snapshot prompt/location data for inactive or temp-backed runtime roots, reports cached-session metadata drift separately from live runtime health, and optionally invalidates only affected snapshots after backing up the store.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a session store entry whose `skillsSnapshot.prompt` references `/tmp/openclaw`, `/private/tmp`, or an old install root while current main's doctor state-integrity path does not inspect that field and runtime refresh predicates compare only version/filter state.

## Solution Assessment

Yes. A narrow doctor diagnostic plus safe snapshot invalidation is the best fit for this operator-facing drift case; the restart-version invalidation PR is useful adjacent work but does not tell users that cached session metadata is stale.
