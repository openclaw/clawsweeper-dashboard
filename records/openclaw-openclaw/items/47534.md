## Summary

Keep open. Current main still lacks the requested doctor diagnostic or repair for stale persisted `skillsSnapshot.prompt` paths; adjacent snapshot refresh work and config-change invalidation do not cover runtime-root drift, and the issue discussion provides concrete persisted-state evidence plus a narrow remediation.

## What This Changes

Request that `openclaw doctor` detect and safely remediate stale cached session `skillsSnapshot` metadata that points at old or temporary OpenClaw runtime roots.

## Best Possible Solution

Add a focused doctor state-integrity check that reports cached-session metadata drift separately from live runtime health and offers an explicit, backed-up repair that invalidates only affected skill snapshot caches.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a `sessions.json` entry with `skillsSnapshot.prompt` pointing at an old `/opt/homebrew`, `/tmp`, or `/private/tmp` OpenClaw skill path would be loaded while doctor never inspects that field.

## Solution Assessment

Yes. A doctor/session-store diagnostic and explicit repair flow is the narrowest maintainable solution; startup or load-time mutation would change long-lived session state outside an operator-requested repair pass.

## Repair Work Prompt

Implement the stale session skills snapshot doctor diagnostic requested in https://github.com/openclaw/openclaw/issues/47534. Boundary: add a focused `openclaw doctor` state-integrity check that resolves the active bundled skills/runtime root, scans persisted session stores for `skillsSnapshot.prompt` and any intentionally supported legacy path-bearing snapshot fields that reference missing, inactive, `/tmp`, `/private/tmp`, or non-current bundled OpenClaw roots, and reports the condition as stale cached session metadata distinct from live runtime health. In `--fix`/`--repair`, use existing safe session-store update/backup patterns and invalidate only the affected `skillsSnapshot` fields so they rebuild on the next run; do not wipe sessions or transcripts, do not add startup/load-time migrations, and avoid hardcoded plugin ids. Update focused tests, doctor docs, and CHANGELOG because this is user-facing doctor behavior. Treat PR #71497 as adjacent context only, not as covering this diagnostic.

- Current `loadSessionStore` strips legacy `resolvedSkills`, so implementation should deliberately choose whether to scan raw JSON for legacy path fields or limit detection to persisted `skillsSnapshot.prompt`.
