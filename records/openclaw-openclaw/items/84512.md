## Summary

Keep open. The PR targets a real source-level plugin-tool failure and includes usable terminal proof, but the branch currently fails lint and adds a bare wildcard manifest contract without aligning the rest of plugin discovery, allowlist behavior, docs, or public types.

## What This Changes

Maps `contracts.tools: true` to `["*"]`, treats `*` as a wildcard in tool-contract validation, and adds focused manifest/tool-contract tests.

## Best Possible Solution

Choose one coherent manifest contract: keep concrete `contracts.tools` names with clearer diagnostics, or implement a maintainer-approved wildcard mode across normalization, registration validation, discovery/allowlist filtering, docs, types, and regression tests.

## Reproduction Assessment

Yes, source-level rather than live-run. Current main drops boolean `contracts.tools` during manifest normalization, and the registry then emits the reported missing-contract diagnostic before registering plugin tools.

## Solution Assessment

No. The branch fixes the local registration gate, but bare `*` is not propagated through the manifest-first discovery contract; the safer path is concrete names or a maintainer-approved wildcard design across all consumers.
