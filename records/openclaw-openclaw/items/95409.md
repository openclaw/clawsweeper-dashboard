## Summary

The issue describes a concrete Telegram rich-message formatting bug that still appears present on current main: the rich HTML preparation path sanitizes and limits nesting but does not materialize plain text-node newlines as rich-message line breaks. It is new, contributor-authored, unprotected, and not covered by the closed native-draft PR, so conservative cleanup should keep it open as a narrow repair candidate.

## What This Changes

Persistent Telegram rich final replies should preserve ordinary plain-text line breaks when rich messages are enabled, without changing the normal send/edit fallback path.

## Best Possible Solution

Add newline-preservation in the Telegram persistent rich HTML preparation path by converting ordinary text-node newlines to `<br>` or equivalent block structure while leaving preformatted/code content and rich-disabled fallback behavior unchanged.

## Reproduction Assessment

Yes, source inspection gives a credible reproduction path: enable Telegram rich messages and send plain line-separated text through the persistent rich final path, which currently forwards raw text-node newlines without a line-break conversion step. I did not run a live Telegram client reproduction in this read-only review, so confidence is medium rather than high.

## Solution Assessment

Yes, the proposed boundary is the right fix shape: normalize text-node newlines in the existing persistent rich HTML preparation path, not by reviving native draft streaming or changing the rich-disabled send/edit fallback.
