## Summary

Keep open: current main still sends opted-in Telegram persistent rich replies through `rich_message.html` after sanitize/nesting handling only, and an open closing PR now owns the narrow repair. This issue should stay open until that PR or another selected fix is reviewed and merged.

## What This Changes

The issue asks OpenClaw to preserve ordinary line breaks in opted-in Telegram persistent rich messages instead of visually collapsing structured assistant replies.

## Best Possible Solution

Keep this issue open as the canonical tracker, then review and land one narrow fix PR, preferably the linked closing PR if it proves the persistent rich-message path without changing rich-disabled fallback behavior.

## Reproduction Assessment

No, not as a live reproduction from this review. The source path is clear and the report includes real Telegram observations, but the current-main proof here is source-level rather than an independently run Telegram client repro.

## Solution Assessment

Yes, the best fix shape is to normalize ordinary text-node newlines inside the existing persistent rich HTML preparation path while leaving pre/code and the rich-disabled sendMessage fallback unchanged. Reviving native rich drafts would be the wrong layer for this issue.
