## Summary

Current main still source-reproduces the reported mismatch: Android can advertise `sms.search`, but gateway policy keeps `sms.search` dangerous/default-blocked unless `gateway.nodes.allowCommands` explicitly opts in. This should stay open because the remaining question is a security/product decision about SMS-read enablement, documentation, diagnostics, or phone-control arming.

## What This Changes

The issue asks for Android nodes with granted SMS permission to expose and allow `sms.search` in live sessions instead of rejecting it as not allowed.

## Best Possible Solution

Keep SMS reads behind explicit gateway policy, then add maintainer-approved docs, diagnostics, or a phone-control SMS-read arming path that makes the opt-in boundary clear before changing defaults.

## Reproduction Assessment

Yes, from source inspection: Android can advertise `sms.search`, while the gateway default Android allowlist deliberately excludes it unless policy configuration opts in. I did not run a live Android device in this read-only review.

## Solution Assessment

No, automatically allowing SMS search from Android permission alone is not the best fix under the current security policy. A safer solution is explicit gateway opt-in with clearer docs/diagnostics, or a maintainer-approved phone-control group for SMS reads.
