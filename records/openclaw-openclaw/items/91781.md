## Summary

Keep open. Current main and v2026.6.8 still source-reproduce the mismatch: Android can advertise `sms.search`, but the gateway deliberately treats it as a dangerous/default-blocked command unless `gateway.nodes.allowCommands` opts it in, so the remaining work needs maintainer product/security direction rather than conservative cleanup.

## What This Changes

The issue asks for Android SMS permission grants to make `sms.search` and related SMS commands visible and invokable in paired live sessions.

## Best Possible Solution

Keep SMS reads behind explicit gateway policy, then add maintainer-approved docs, diagnostics, or an SMS-read arming path that makes the opt-in boundary clear before changing defaults.

## Reproduction Assessment

Yes, from source inspection: Android can advertise `sms.search`, while the gateway default Android allowlist deliberately excludes it unless policy configuration opts in. I did not run a live Android device in this read-only review.

## Solution Assessment

No, automatically allowing SMS search from Android permission alone is not the best fix under the current security policy. A safer solution is explicit gateway opt-in with clearer docs/diagnostics, or a maintainer-approved phone-control group for SMS reads.
