## Summary

Current main still reproduces the reported mismatch from source: Android can advertise `sms.search`, but the gateway default Android command policy filters it out unless `gateway.nodes.allowCommands` explicitly opts in. The item should stay open because the remaining work is a security/product decision about SMS-read enablement, docs, or phone-control arming, not a stale or implemented report.

## What This Changes

The issue asks for Android nodes with granted SMS permission to expose and allow `sms.search` in live sessions instead of rejecting it as a disallowed node command.

## Best Possible Solution

Keep SMS reads behind explicit gateway policy, then make the supported opt-in path unmistakable through docs/diagnostics or a maintainer-approved phone-control arming group for SMS search.

## Reproduction Assessment

Yes, from source inspection: Android can advertise `sms.search`, while the gateway default Android allowlist deliberately excludes it unless policy configuration opts in. I did not run a live Android device in this read-only review.

## Solution Assessment

No, automatically allowing SMS search from Android permission alone is not the best fix under the current security policy. A safer solution is explicit gateway opt-in with clearer docs/diagnostics, or a maintainer-approved phone-control group for SMS reads.
