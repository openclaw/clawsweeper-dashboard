## Summary

Keep open: current main still treats `sms.search` as a high-risk Android node command that is filtered unless `gateway.nodes.allowCommands` explicitly opts it in. The report remains actionable because Android docs list `sms.search`, while the supported gateway policy or phone-control arming path for SMS search is not clear enough.

## What This Changes

The issue asks for Android SMS permission state and live gateway node command gating to agree so `sms.search` is advertised and allowed when appropriate.

## Best Possible Solution

Keep SMS search behind explicit operator control, then either document the required `allowCommands`/`denyCommands` setup clearly or add a maintainer-approved phone-control SMS-read arming path with focused gateway and Android capability tests.

## Reproduction Assessment

Yes. Source inspection shows Android can advertise `sms.search`, while current gateway policy and tests deny `sms.search` by default unless `gateway.nodes.allowCommands` opts it in; I did not run a live Android pairing repro in this read-only review.

## Solution Assessment

No, not as an automatic permission-to-allowlist change. The safer fix is an explicit maintainer-approved opt-in path for SMS reads, with docs or phone-control UX aligned to the existing high-risk command policy.
