## Summary

Keep open. Current main and the latest release still source-reproduce the reported mismatch: Android can advertise `sms.search`, but the gateway intentionally treats it as a dangerous node command that is not allowed unless `gateway.nodes.allowCommands` opts it in, so the remaining work is a product/security decision rather than conservative cleanup.

## What This Changes

The issue asks for Android SMS permission to make `sms.search` and related SMS commands visible and runnable in the live node session.

## Best Possible Solution

Keep SMS reads behind explicit gateway policy, then have maintainers choose a documented opt-in, diagnostic, or arming path for Android `sms.search` instead of auto-allowing it from OS permission alone.

## Reproduction Assessment

Yes, from source inspection: Android can advertise `sms.search`, while the gateway default Android allowlist deliberately excludes it unless policy configuration opts in. I did not run a live Android device in this read-only review.

## Solution Assessment

No, automatically allowing SMS search from Android permission alone is not the best fix under the current security policy. A safer solution is explicit gateway opt-in with clearer docs, diagnostics, or a maintainer-approved SMS-read arming path.
