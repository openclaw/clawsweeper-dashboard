## Label Justifications

- `P2`: The issue blocks a documented Android capability but affects an optional, bounded workflow rather than core availability.
- `impact:security`: The missing authorization path controls agent access to private SMS history on a paired device.

## Summary

Current main intentionally filters `sms.search` as a high-risk command, but the temporary phone-control workflow cannot authorize it and the Android documentation does not explain the required gateway opt-in. The issue therefore identifies a real security-sensitive product and documentation gap that still needs maintainer direction.

## What This Changes

Provide an explicit gateway authorization path for permission-backed Android `sms.search` and document it alongside the Android permission requirement.

## Best Possible Solution

Keep `sms.search` denied by default, add a distinct temporary SMS-read arming group or equally explicit operator-controlled workflow, and update Android and node documentation to show both the OS permission and gateway authorization steps.

## Reproduction Assessment

Yes at source level. Android can declare `sms.search`, the gateway filters it as dangerous, and no phone-control group can temporarily authorize it, matching the reported live-session absence and rejection.

## Solution Assessment

No, automatically injecting or allowing the command from Android permission is not the best solution because it bypasses gateway policy. Extending the explicit temporary authorization workflow and its documentation is the narrowest secure direction.
