## Summary

Current main still filters `sms.search` out of the live Android node command surface unless the gateway policy explicitly allows it; that matches the safe-default command policy, but the reported workflow remains actionable because the SMS docs and phone-control arming path do not make the `sms.search` opt-in path clear or easy.

## What This Changes

The issue reports that an Android node with SMS permission granted still does not advertise or allow `sms.search` in a live session.

## Best Possible Solution

Keep SMS commands behind explicit gateway policy, then make the supported path for `sms.search` unmistakable: document the required `allowCommands`/`denyCommands` changes and re-approval step, or add a maintainer-approved phone-control arming group for SMS reads.

## Reproduction Assessment

Yes from source: Android can declare `sms.search`, while the gateway default Android allowlist deliberately excludes it unless policy configuration opts in. I did not run a live Android device, but the current code and tests show the same rejection path described by the report.

## Solution Assessment

No, automatically allowing SMS solely because Android permission is granted is not the safest default under the current command policy. The better solution is either clearer docs/diagnostics for the explicit gateway opt-in or a maintainer-approved phone-control path for temporarily arming SMS search.
