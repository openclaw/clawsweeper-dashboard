## What This Changes

The issue reports that `openclaw infer tts convert --model xai/tts` times out locally while the same xAI TTS request succeeds through `--gateway`.

## Best Possible Solution

Align xAI speech credential policy and local/Gateway timeout behavior so the same valid xAI setup either succeeds locally and through Gateway or fails quickly with one clear diagnostic, backed by focused tests and live xAI proof.

## Reproduction Assessment

No live high-confidence reproduction was run in this review. Source inspection shows the reported local path still reaches xAI `/tts` through the guarded fetch path with the 30s default while Gateway gets a 120s call timeout, matching the reported timeout shape.

## Solution Assessment

Unclear: the issue is real enough to keep open, but the best fix depends on deciding whether xAI speech is OAuth-supported or API-key-only and whether local CLI TTS should inherit a longer timeout, expose the existing timeout contract, or fail earlier with a shared diagnostic.
