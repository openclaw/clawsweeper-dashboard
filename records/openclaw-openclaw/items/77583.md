## Summary

Keep open: the PR is useful source work, but it is not merge-ready because the advertised user-facing config key is not added to the strict config schema/generated schema, and the raw-transcript replay opt-in needs maintainer acceptance as a safety tradeoff.

## What This Changes

The PR adds a per-CLI-backend opt-in flag to reseed fresh CLI sessions from the raw transcript tail when no usable compaction summary exists, plus tests for that opt-in path.

## Best Possible Solution

If maintainers accept the safety tradeoff, make the flag a complete config contract with zod schema, generated schema/help, user docs, changelog, and validation coverage while preserving the default-off behavior.

## Reproduction Assessment

Yes for the PR blocker: source inspection shows the proposed config key is absent from the strict `CliBackendSchema` and generated schema. Reproducing would be `openclaw config validate` with the PR's example config; I did not run validation in this read-only review.

## Solution Assessment

No. The runtime fallback is narrow, but a user-facing backend option must be added to the validation/schema/docs contract, and maintainers should explicitly approve the raw-transcript replay policy before merge.
