## Summary

Keep open for normal maintainer review and merge handling. The patch has no blocking code finding and the real behavior proof now passes, but it deliberately expands `cron.retry` from the current documented one-shot-only contract to recurring jobs, which is a maintainer compatibility decision before merge.

## What This Changes

The PR changes recurring cron error handling to schedule retryable failures at configured backoff before the next natural cron slot, with updated cron retry docs/help/types and regression coverage.

## Best Possible Solution

Land the focused scheduler fix if maintainers accept recurring failures inheriting `cron.retry`, keeping the high-frequency backoff floor, exhausted-budget fallback, unresolved-next-run guard, docs, and regressions together.

## Reproduction Assessment

Yes at source level: current main's recurring error branch uses the later of natural next run and backoff, so a daily cron's natural slot dominates a short retry backoff. I did not run the live MiniMax tenant path.

## Solution Assessment

Yes, if maintainers accept the contract expansion: changing the recurring error scheduling branch while preserving high-frequency backoff and unresolved-next-run behavior is the narrow scheduler fix. The safer alternative is an opt-in recurring retry policy if default upgrade compatibility wins.
