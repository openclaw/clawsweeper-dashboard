## Summary

Keep open: the latest PR head fixes the outcome-sequence cursor follow-up, but the new guard still observes only after the embedded attempt returns, so it does not interrupt the active post-compaction tool loop it is meant to stop.

## What This Changes

The PR adds a default-on post-compaction loop guard for the pi embedded runner, with config/schema/docs/changelog updates and focused unit/integration tests.

## Best Possible Solution

Keep the guard/config surface, but evaluate the post-compaction verdict from the live tool-outcome path so the configured threshold aborts the active prompt instead of waiting for attempt completion.

## Reproduction Assessment

Yes, from source inspection. The PR observes after `runEmbeddedAttemptWithBackend` returns, while current source and docs show `activeSession.prompt()` runs the full LLM/tool loop before the outer runner resumes.

## Solution Assessment

No. The guard design is useful, but this hook point is too late for a persistent post-compaction loop; the narrow repair is to feed the guard from live tool outcome recording.

## Review Findings

- **[P1] Observe post-compaction outcomes before the attempt returns:** `src/agents/pi-embedded-runner/run.ts:1221-1265`
  - body: The guard is armed before the retry, but this observation block runs only after `runEmbeddedAttemptWithBackend` returns.
