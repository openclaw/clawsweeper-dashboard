## Summary

This PR should stay open because the latest head still evaluates the post-compaction guard only after the embedded prompt attempt returns, so it does not interrupt the active tool-call loop it is meant to stop.

## What This Changes

The PR branch adds a default-on post-compaction loop guard for the pi embedded runner, config/schema/docs/changelog updates, outcome sequence tracking, and focused unit/integration tests.

## Best Possible Solution

Keep the guard and config surface, but evaluate the verdict from live tool outcome recording so the threshold aborts the active post-compaction prompt loop, with a regression that proves the prompt has not returned yet.

## Reproduction Assessment

Yes. Source inspection shows the PR observes after `runEmbeddedAttemptWithBackend` returns, while current main awaits `activeSession.prompt(...)` for the full LLM/tool loop inside that attempt.

## Solution Assessment

No. The guard design is useful, but this hook point is too late; the narrow maintainable fix is to feed the guard from live tool outcome recording and abort the in-flight prompt when the verdict trips.
