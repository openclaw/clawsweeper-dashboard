## Summary

Keep open: the guard is a useful targeted safety net, but the latest PR head still evaluates it only after an embedded attempt returns, which is too late for the post-compaction loop it is meant to stop.

## What This Changes

The PR adds a default-on post-compaction loop guard for the pi embedded runner, plus config/schema/docs/changelog updates and focused unit/integration tests.

## Best Possible Solution

Keep the guard concept, but make the live tool-outcome recording path evaluate the armed post-compaction verdict and abort the active prompt on the triggering outcome.

## Reproduction Assessment

Yes, from source inspection: the PR observes outcomes only after `runEmbeddedAttemptWithBackend` returns, while current source and docs show `activeSession.prompt()` owns the full LLM/tool loop before the outer runner regains control.

## Solution Assessment

No. The current hook point is too late for a persistent post-compaction loop; the maintainable fix is to evaluate the guard during live tool outcome recording, while preserving the config/docs surface and trim-resilient sequencing.

## Review Findings

- **[P1] Observe post-compaction outcomes before the attempt returns:** `src/agents/pi-embedded-runner/run.ts:1224-1265`
