## Summary

Keep open: this PR still observes post-compaction tool outcomes only after the pi SDK prompt attempt returns, so it does not reliably stop the persistent in-attempt loop it is meant to bound.

## What This Changes

The PR adds a default-on post-compaction loop guard, config/schema/docs/changelog updates, diagnostic outcome sequencing, and focused unit/integration/config tests.

## Best Possible Solution

Move the post-compaction verdict into the live tool-outcome path so the third matching outcome interrupts the active prompt, while preserving the config/docs surface and trim-resilient sequencing.

## Reproduction Assessment

Yes, from source inspection. The PR observes after `runEmbeddedAttemptWithBackend` returns, while current docs and source show `activeSession.prompt()` runs the full LLM/tool loop before the outer runner sees the attempt result.

## Solution Assessment

No. The guard design is useful, but the current hook point is too late for a persistent post-compaction loop; the safer fix is to evaluate the guard during live tool outcome recording.
