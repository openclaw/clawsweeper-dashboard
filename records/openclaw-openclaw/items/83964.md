## Summary

Keep open: the linked release reports show a real Codex runtime-resolution failure, current code still relies on host `openclaw/plugin-sdk/*` imports, the latest checked package metadata still treats `openclaw` as an optional peer, and the open fix candidate at https://github.com/openclaw/openclaw/pull/84275 has not merged.

## What This Changes

Report that `@openclaw/codex` installs or updates can leave no resolvable host `openclaw` package for runtime SDK imports, causing `ERR_MODULE_NOT_FOUND` until peer-link repair is run.

## Best Possible Solution

Keep the documented host-peer model and make install, update, doctor, and package/update smoke checks reliably leave `@openclaw/codex` with a working plugin-local `node_modules/openclaw` link, unless maintainers explicitly approve changing Codex to install the host package directly.

## Reproduction Assessment

Yes for the shipped failure class: comments on this issue report live 2026.5.18 and 2026.5.19 reproductions, and current source shows the same runtime import fails if the Codex package lacks a working host `openclaw` link. I did not run a destructive current-main repro in this read-only review.

## Solution Assessment

Unclear as proposed by the open PR: a direct `openclaw` dependency fixes standalone npm imports but conflicts with the documented host-peer model. The safer default is to repair the managed peer-link/update path unless maintainers intentionally change the package contract.
