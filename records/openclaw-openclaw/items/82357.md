## Summary

Keep open: current `main` still routes unset Codex-backed direct chats through `message_tool_only`, which can suppress Feishu/Lark visible final replies and matches the reported `queuedFinal=false, replies=0` shape. This should stay with maintainers because current docs and tests intentionally describe the Codex direct-chat tool-only default, while the issue asks for Feishu/Lark DM compatibility with the older visible streaming path.

## What This Changes

Codex/GPT-backed Feishu/Lark direct messages are reported to bypass the visible streaming reply path unless `messages.visibleReplies` is explicitly set to `automatic`.

## Best Possible Solution

Make an explicit policy decision for Codex-backed Feishu/Lark DMs: either keep the tool-only default with clearer diagnostics/docs, or add a narrow direct-chat compatibility path with regression coverage and redacted live Feishu proof.

## Reproduction Assessment

Yes at source level: an unset Codex-backed direct chat resolves to `message_tool_only`, suppresses automatic final delivery, and current tests assert the resulting no-final-reply path. I did not run a live Feishu/Lark tenant reproduction.

## Solution Assessment

Unclear as an automated fix: restoring automatic visible streaming for Codex-backed Feishu DMs would override the documented Codex direct-chat default. Maintainers should decide the compatibility boundary before implementation.
