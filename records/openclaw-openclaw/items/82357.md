## Summary

Keep open: current main and shipped v2026.5.12 still let unset Codex-backed direct chats inherit `message_tool_only`, which matches the reported Feishu/Lark `queuedFinal=false, replies=0` shape. Resolving this would change or narrow the documented Codex direct-chat default, so this remains a maintainer/product decision rather than conservative cleanup.

## What This Changes

Restore visible streaming/final replies for Codex/GPT-backed Feishu/Lark DMs when `messages.visibleReplies` is unset, instead of requiring message-tool sends for ordinary replies.

## Best Possible Solution

Make an explicit maintainer policy decision for Codex-backed Feishu/Lark DMs, then either add a narrow compatibility path with regression coverage and redacted live proof or keep the tool-only default with clearer diagnostics/docs.

## Reproduction Assessment

Yes at source level: an unset direct Feishu/Lark turn using the Codex harness inherits `sourceVisibleReplies=message_tool`, resolves to `message_tool_only`, and current tests assert `queuedFinal=false` with no final reply. I did not perform a credentialed live Feishu tenant run.

## Solution Assessment

Unclear as an automated fix: restoring automatic visible replies for Codex-backed Feishu DMs would override the documented Codex direct-chat default. Maintainers should decide the compatibility boundary before a code change lands.
