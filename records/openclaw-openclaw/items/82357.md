## Summary

Keep open: current source and shipped tests make the reported behavior source-reproducible, but the same behavior is now documented as the Codex harness default for unset direct chats. Changing it needs a maintainer product decision about whether Feishu/Lark DMs should override that default or whether docs/doctor/onboarding should make the required `messages.visibleReplies: "automatic"` opt-out clearer.

## What This Changes

Restore Feishu/Lark direct-message Codex/GPT replies to the normal streaming visible-reply path unless the user explicitly configures message-tool-only output.

## Best Possible Solution

Decide the direct-chat visible-reply contract for Codex harness sessions, then either add a scoped Feishu/direct-chat exception or migration with regression coverage, or keep the current behavior and improve diagnostics/onboarding around `messages.visibleReplies: "automatic"`.

## Reproduction Assessment

Yes, at source level: a Feishu direct chat with an unset `messages.visibleReplies` and a Codex harness session resolves to `message_tool_only`, and the current regression test asserts `queuedFinal=false` with no final reply queued. I did not run a live Feishu WebSocket scenario.

## Solution Assessment

Unclear for automation: setting `messages.visibleReplies: "automatic"` is a valid workaround, but changing the default would reverse a documented Codex behavior. A maintainer should choose whether Feishu DMs need a scoped override/migration or whether the current tool-only default should remain.
