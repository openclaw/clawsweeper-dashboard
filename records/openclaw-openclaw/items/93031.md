## Summary

Kept open because the report describes a concrete latest-release cron upgrade failure, but source inspection does not support the stated root cause. Legacy jobs can indeed migrate with `agent_id` unset, yet both current `main` and the v2026.6.6 tag treat missing cron agent ids as default-agent state rather than a direct scheduler filter, so the actual failing upgrade path still needs reproduction.

## What This Changes

The issue reports that legacy `jobs.json` cron jobs imported into SQLite during the v2026.6.6 upgrade keep blank `agent_id` values and then silently stop dispatching.

## Best Possible Solution

Reproduce the v2026.5.28 to v2026.6.6 upgrade with a legacy job missing `agentId`, then fix the first migration, reload, or runtime path that prevents that job from dispatching under the correct effective agent.

## Reproduction Assessment

No high-confidence current-main reproduction was established. The report has concrete upgrade and SQL evidence, and source confirms blank `agent_id` can exist, but source inspection contradicts the claimed skip-by-agent scheduler filter.

## Solution Assessment

Unclear. A blanket migration to `agent_id='main'` is not proven to be the best fix because current cron code treats missing `agentId` as default-agent state and multi-agent installs may need a more precise ownership rule.
