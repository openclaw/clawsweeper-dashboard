## Summary

Keep open: current main and v2026.6.8 still use semantic LanceDB recall without freshness ranking or timestamped provenance, and the only direct fixing PR remains open and unmerged. The report is valid, but the fix shape needs maintainer calibration because recency ranking, source provenance, and live/history lookup have different product tradeoffs.

## What This Changes

The issue asks memory-lancedb to avoid stale answers for freshness-sensitive prompts by adding recency-aware recall and clearer recalled-memory provenance.

## Best Possible Solution

Land or replace the linked memory-lancedb fix after deciding the minimal product boundary: freshness-sensitive prompts should bias toward recent relevant memories, and recalled historical context should be distinguishable from live source/history lookup.

## Reproduction Assessment

Yes, by source inspection: current main embeds the latest user text, calls MemoryDB.search, and MemoryDB.search returns semantic vector results without timestamp ranking. I did not run a live Telegram/LanceDB repro in this read-only review.

## Solution Assessment

Unclear. memory-lancedb is the right owner boundary, but maintainers still need to choose whether createdAt reranking is enough or whether source timestamps, provenance, or live-history routing are required before closing this.
