## Summary

Keep open: current main and the latest release still use semantic-only memory-lancedb recall without freshness reranking or assistant-visible timestamp/provenance, and the linked implementation pull request remains open and conflicting.

## What This Changes

Request recency- and provenance-aware memory-lancedb recall so freshness-sensitive prompts do not answer from stale recalled memories as if they were live state.

## Best Possible Solution

Repair or replace the linked memory-lancedb PR on the current before_prompt_build path, rerank freshness-sensitive recall by timestamp, and surface timestamp/provenance caveats in injected memory context.

## Reproduction Assessment

Yes, at source level. Seed two matching memories where the older entry has stronger vector similarity, ask a latest/current prompt, and current main will inject db.search(vector, 3, 0.3) results without recency reranking or timestamp/provenance context.

## Solution Assessment

Yes, the requested direction is the right fix boundary for this plugin. The maintainable path is to keep the change in memory-lancedb, preserve the current before_prompt_build timeout/query normalization behavior, and add focused regression coverage.
