## Summary

Keep open: current main and v2026.5.22 still use semantic-only memory-lancedb recall without freshness reranking or assistant-visible timestamp/provenance, and the linked implementation PR remains open and conflicting.

## What This Changes

The issue requests freshness-aware memory-lancedb recall and provenance so freshness-sensitive prompts do not answer from older semantic memories as if they were live history.

## Best Possible Solution

Review, repair, and land the linked memory-lancedb PR or a narrow replacement that implements freshness-aware recall on before_prompt_build, preserves existing timeout/input bounds, and surfaces timestamp/provenance context.

## Reproduction Assessment

Yes, at source level. Seed two matching memories where the older item has stronger vector similarity, ask a latest/current prompt, and current main calls db.search(vector, 3, 0.3) and injects semantic order without recency reranking or timestamp/provenance context.

## Solution Assessment

Yes, the requested direction is the right fix for this plugin. The implementation should stay on the current before_prompt_build path, rerank freshness-sensitive candidates, and add provenance context without adding a new config knob for the minimal repair.
