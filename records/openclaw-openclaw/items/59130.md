## Summary

Keep open: current main still performs semantic-only LanceDB auto-recall and injects memory context without freshness intent handling, recency-aware reranking, source timestamp metadata, or live-vs-recall provenance. Open PR #59141 is already linked as an implementation candidate, so the issue should stay open until that PR lands, is replaced, or maintainers choose a different fix path.

## What This Changes

The issue asks memory-lancedb to prevent freshness-sensitive prompts from being answered from stale LanceDB recall without recency-aware ranking or assistant-visible provenance.

## Best Possible Solution

Keep the issue open and resolve it in memory-lancedb by adding freshness-intent detection, widening candidate recall when needed, reranking by createdAt or source timestamp where available, surfacing a clear recalled-memory provenance/caveat in the injected context, and adding a regression test where semantic similarity would otherwise select the older memory. Open PR #59141 is the current review path for that fix.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: seed two matching memories where the older one has the stronger vector match, ask a freshness-sensitive prompt, and current before_prompt_build will inject db.search(vector, 3, 0.3) results without recency reranking or provenance; I did not run tests because this review is explicitly read-only.

## Solution Assessment

Yes, the issue's requested direction is the right boundary: fix owner-specific recall behavior inside memory-lancedb first, with only a later generic seam if multiple memory providers need shared freshness/provenance contracts. The linked PR appears to be the current implementation candidate, while structured source provenance may remain a follow-up if the minimal createdAt rerank lands first.
