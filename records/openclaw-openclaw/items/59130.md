## Summary

Current main and the latest release still use semantic LanceDB recall without freshness-aware ranking or assistant-visible timestamp/provenance, and the linked closing PR is still open and conflicting rather than merged. This issue should stay open as the canonical report while maintainers review, refresh, or replace the linked fix path.

## What This Changes

The issue requests recency-aware and provenance-visible memory-lancedb recall so freshness-sensitive prompts do not answer from stale recalled memories as if they were live source lookups.

## Best Possible Solution

Refresh or replace the linked PR so the current before_prompt_build recall path gets a narrow recency/provenance behavior with regression coverage and maintainer-approved freshness semantics.

## Reproduction Assessment

Yes by source inspection and a focused mocked test path: current recall is semantic-only and can rank an older high-similarity memory ahead of a newer lower-similarity memory for a latest/current prompt. I did not establish a live Telegram/LanceDB reproduction in this read-only review.

## Solution Assessment

Unclear as a final product direction. Recency reranking with timestamp/provenance hints is a plausible mitigation, but maintainers still need to choose the intended boundary between memory recall, source metadata, and live/history lookup before this issue is fully solved.
