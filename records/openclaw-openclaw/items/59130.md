## Summary

Keep open: current main and v2026.5.19 still use semantic-only memory-lancedb auto-recall with no freshness reranking or assistant-visible timestamp/provenance, and the linked closing pull request remains open and conflicting.

## What This Changes

The issue asks memory-lancedb to avoid stale answers for freshness-sensitive prompts by adding recency-aware recall and clearer provenance for injected memories.

## Best Possible Solution

Land or replace the linked memory-lancedb fix on the current before_prompt_build path, overfetch and rerank freshness-sensitive recall by timestamp, and surface timestamp/provenance caveats in injected memory context.

## Reproduction Assessment

Yes, at source level. Seed two matching memories where the older item has stronger vector similarity, ask a latest/current prompt, and current main will call db.search(vector, 3, 0.3) and inject that semantic order without recency reranking or timestamp/provenance context.

## Solution Assessment

Yes, the requested direction is the right fix for this plugin. The implementation should stay on before_prompt_build, preserve current timeout/input bounds, rerank freshness-sensitive candidates, and add provenance context without requiring a new config option.
