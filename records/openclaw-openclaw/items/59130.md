## Summary

Current main and v2026.6.8 still use semantic-only LanceDB memory recall without freshness-aware ranking or assistant-visible timestamp/source provenance, and the linked fix PR is still open and conflicting. This issue should stay open as the canonical report while maintainers review, refresh, or replace that fix path.

## What This Changes

Requests memory-lancedb freshness-aware recall and provenance so latest/current prompts do not answer from stale semantic memories.

## Best Possible Solution

Refresh or replace the linked PR with a maintainer-approved memory-lancedb freshness/provenance behavior and focused regression coverage.

## Reproduction Assessment

Yes, by source inspection. Current auto-recall preserves semantic vector-search order and injects the top clean results, so a latest/current prompt can receive an older higher-similarity memory ahead of a newer lower-similarity one; I did not run a live Telegram/LanceDB repro in this read-only review.

## Solution Assessment

Unclear as a final product direction. Recency reranking with timestamp/provenance hints is a plausible mitigation, but maintainers still need to choose the intended boundary between memory recall, source metadata, and live/history lookup.
