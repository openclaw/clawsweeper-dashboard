## Summary

Keep open: current main and v2026.6.6 still use semantic-only memory-lancedb auto-recall without freshness ranking or per-memory provenance, and the linked fix PR is open but unmerged/conflicting.

## What This Changes

The issue requests recency-aware and provenance-aware LanceDB memory recall so freshness-sensitive prompts do not answer from stale recalled memories as if they were live source queries.

## Best Possible Solution

Land or replace the linked memory-lancedb fix with a maintainer-approved design that handles freshness intent, preserves recalled-memory provenance, and keeps live source/history lookup distinct from long-term memory context.

## Reproduction Assessment

Yes, as a high-confidence source-level path: current auto-recall searches LanceDB by semantic vector score and injects the returned memories without recency ranking or provenance. I did not execute a live LanceDB repro in this read-only review.

## Solution Assessment

Unclear. Recency-aware createdAt ranking is a plausible mitigation, but the issue also asks for provenance/source timestamps or live-history routing, which needs a maintainer product decision.
