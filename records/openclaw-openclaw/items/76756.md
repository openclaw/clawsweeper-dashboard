## Summary

Keep open. Current main still lacks an injected plugin runtime gateway accessor while the requested host RPCs exist for normal gateway clients; because this would add a public SDK seam with redaction, allowlist, and replay-contract implications, it is not cleanup-closeable.

## What This Changes

Request a plugin-sdk runtime gateway surface for plugins to read host RPCs such as `status` and `models.list`.

## Best Possible Solution

Add a narrow documented `runtime.gateway` namespace for read-only status/model queries, with explicit allowlisting, redaction rules, exported public types, and replay snapshot semantics.

## Reproduction Assessment

Not applicable; this is a feature request rather than a broken existing contract. Source inspection against current main confirms the requested `PluginRuntime` gateway namespace is absent while `status` and `models.list` are implemented for gateway clients.

## Solution Assessment

Unclear until maintainer API review. Curated read-only accessors look safer than exposing a broad `dispatchMethod` or generic request surface, especially because status redaction and replay snapshots need explicit policy.

## Risks / Open Questions

- The public SDK shape is unsettled: curated accessors and a generic allowlisted request API have different compatibility and replay contracts.
- `status` can expose sensitive runtime state, so plugin access needs explicit redaction and trust policy.
- The linked replay and type-export issues (#76758 and #76759) should be resolved or aligned before shipping the new surface.
- Open PR #46485 proposes a broader dispatcher, but that is not obviously the safest solution for this issue's read-only host-state use case.
