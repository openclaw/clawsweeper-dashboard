## Summary

Keep open: current main and v2026.6.8 still lack an ordinary plugin runtime Gateway accessor for `status`/`models.list`, while the existing dispatcher is intentionally reserved for authenticated plugin HTTP routes. This remains a focused plugin SDK and security-boundary design item, not stale cleanup or an implemented feature.

## What This Changes

Request a public plugin SDK surface that lets plugins invoke a small allowlisted set of host Gateway RPCs such as `status` and `models.list`.

## Best Possible Solution

Keep this issue as the narrow host-RPC plugin SDK design item and add a curated read-only Gateway accessor only after maintainer security/product approval of allowlist, auth, type, and replay semantics.

## Reproduction Assessment

Yes. Source inspection on current main shows no `PluginRuntime.gateway` surface, and the only in-process Gateway dispatcher rejects ordinary plugin runtime callers unless a route-scoped authenticated-request contract is present.

## Solution Assessment

Unclear. Curated read-only accessors look safer than a broad raw request API, but maintainers still need to decide allowlist, caller identity, authorization, public types, and replay/frozen snapshot semantics.
