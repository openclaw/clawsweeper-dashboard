## Summary

Keep open. Current main still lacks an injected plugin runtime gateway accessor, while the underlying Gateway RPCs exist for operator/TUI clients, so this remains a valid SDK feature and API-design request.

## What This Changes

Add a plugin-sdk runtime surface that lets plugins read selected host Gateway RPC data such as status and models.list.

## Best Possible Solution

Add a narrow documented api.runtime.gateway read-only namespace with status/model accessors, public result types, redaction defaults, and deterministic replay guidance aligned with #76758 and #76759.

## Reproduction Assessment

Not applicable; this is a feature request, not a failing existing behavior. Source inspection on current main confirms api.runtime has no gateway namespace while status and models.list exist for Gateway clients.

## Solution Assessment

Unclear until maintainer API review. A curated read-only facade looks safer than exposing a broad request method, but the allowlist, redaction policy, and replay contract need to be settled first.

## Security Review

Status: needs_attention

Summary: A future runtime gateway accessor would expose host state to in-process plugins, so it needs explicit allowlisting, redaction, and replay semantics.

Concerns:

- **[medium] Preserve gateway status redaction for plugins:** `src/gateway/server-methods/health.ts:126`
  - body: The status handler currently sets includeSensitive from operator.admin client scopes; an in-process runtime accessor would bypass that client auth layer unless it intentionally returns redacted output or enforces an equivalent trusted policy.
  - confidence: 0.86
- **[medium] Keep plugin RPC access allowlisted:** `src/gateway/server-methods-list.ts:4`
  - body: The Gateway protocol includes many read and mutating methods, so a generic runtime request surface should not expose arbitrary methods to plugins without a narrow allowlist and per-method policy.
  - confidence: 0.78
