## Summary

Keep open. Current main still lacks a plugin-runtime gateway accessor even though the underlying read-only Gateway RPCs exist; this is a new public SDK seam, not conservative cleanup closure.

## What This Changes

Request a plugin-sdk runtime gateway surface so plugins can read host status and model catalog RPCs.

## Best Possible Solution

Add a narrow documented api.runtime.gateway facade for read-only status/model queries, with allowlisting, redaction rules, exported public types, and replay snapshot semantics.

## Reproduction Assessment

Not applicable; this is a feature request rather than broken existing behavior. Source inspection confirms the runtime surface is absent while the requested Gateway RPCs exist for Gateway clients.

## Solution Assessment

Unclear until maintainer API review. Curated read-only accessors look safer than exposing a broad GatewayClient, but redaction, replay, and type-export contracts need to be settled first.

## Security Review

Status: needs_attention

Summary: The requested SDK surface would expose host runtime state to in-process plugins and needs an explicit allowlist plus redaction policy.

Concerns:

- **[medium] Preserve status redaction for plugin access:** `src/gateway/server-methods/health.ts:127`
  - body: The current status Gateway handler only includes sensitive fields when the client has operator.admin; an in-process plugin gateway facade must not bypass that trust check or leak redacted session paths/model defaults.
  - confidence: 0.86
- **[medium] Keep the gateway surface read-only and allowlisted:** `src/gateway/method-scopes.ts:69`
  - body: status and models.list are read-scoped today, but exposing a generic request method without a strict allowlist could give plugins a path toward mutating Gateway RPCs.
  - confidence: 0.78
