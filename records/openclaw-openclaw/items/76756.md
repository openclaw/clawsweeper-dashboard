## Label Justifications

- `P2`: This is a normal-priority plugin SDK capability gap with limited immediate blast radius, but it needs maintainer design before implementation.
- `impact:security`: The requested surface would change which host Gateway state and RPC behavior loaded plugin code can access and how those calls are authorized.

## Summary

Keep open: current main and the latest release still do not provide the requested plugin runtime accessors for host Gateway RPCs, and the closest existing dispatch helper is intentionally limited to authenticated plugin HTTP routes, so this remains a valid security-sensitive plugin SDK design item.

## What This Changes

The issue requests a public plugin SDK/runtime surface for plugins to call selected host Gateway RPCs such as `status` and `models.list`.

## Best Possible Solution

Keep this open as a focused plugin SDK design item and add narrow read-only Gateway accessors or an allowlisted request surface with caller attribution, authorization, replay semantics, public types, docs, and contract tests.

## Reproduction Assessment

Yes, at source level: current main exposes plugin registration of Gateway methods and host `status`/`models.list` handlers, but `PluginRuntime` has no public gateway caller and the only SDK dispatcher is restricted to authenticated plugin HTTP routes. No live runtime reproduction is needed to confirm this feature gap.

## Solution Assessment

Unclear. Curated read-only accessors look safer than a broad raw request API, but maintainers still need to choose the allowlist, caller identity, authorization, and replay contract before implementation.
