## Summary

Keep open. Current main still lacks an injected plugin runtime gateway accessor; the existing gateway methods and broad `plugin-sdk/gateway-runtime` client do not settle the safer public API, redaction, or replay contract.

## What This Changes

Request a curated plugin-sdk runtime gateway surface so plugins can read host status and model catalog data without using a broad GatewayClient or guessing response types.

## Best Possible Solution

Add a narrow documented `api.runtime.gateway` facade for read-only status/model queries with allowlisting, redaction policy, exported public types, and replay snapshot semantics.

## Reproduction Assessment

Not applicable; this is a feature request rather than broken documented behavior. Source inspection confirms there is no `api.runtime.gateway` namespace while `status` and `models.list` exist as Gateway RPC handlers.

## Solution Assessment

Unclear until maintainer API review. Curated accessors look safer than exposing a broad `GatewayClient.request`, but the implementation should first define allowlist, redaction, exported types, and replay semantics.

## Security Review

Status: needs_attention

Summary: The requested capability would expose host runtime state to plugins, so implementation needs explicit allowlist and redacted status semantics.

  - body: The current `status` handler derives `includeSensitive` from operator scopes; an in-process runtime accessor would bypass that client auth layer unless it intentionally returns redacted output or enforces an equivalent policy.
  - confidence: 0.85
- **[medium] Avoid exposing arbitrary Gateway RPCs:** `src/plugin-sdk/gateway-runtime.ts:20`
  - body: A plugin runtime facade should allowlist the intended read-only methods instead of handing plugins a broad request primitive that can reach other Gateway methods if credentials or scopes are widened.
