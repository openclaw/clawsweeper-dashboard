## Summary

Keep open. Current main still lets raw MCP embedded resource blocks cross the bundle-MCP tool boundary, while the provider replay paths only consume top-level text/image blocks. Related #68037/#68046 cover the separate HTTP MCP gateway normalizer, so this remains a narrow agent-side runtime bug.

## What This Changes

The issue requests that MCP tool-result content blocks with `type: "resource"` expose nested `resource.text` to the model instead of becoming an image placeholder.

## Best Possible Solution

Normalize MCP tool-result content as it enters OpenClaw's agent tool-result contract: convert embedded text resources to normal text blocks, keep valid image blocks intact, and use clear bounded fallbacks for unsupported MCP block variants.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: return a `CallToolResult` with `content: [{ type: "resource", resource: { text: "..." } }]` through bundle MCP, then replay it through the OpenAI-compatible conversion that reads no top-level text and reaches the placeholder path.

## Solution Assessment

Yes. Extracting `resource.resource.text` is correct, but the maintainable fix is central MCP-to-agent content normalization rather than duplicating special cases in every provider transport.
