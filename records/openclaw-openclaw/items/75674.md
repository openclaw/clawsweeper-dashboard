## Summary

Keep open. Current main still allows MCP embedded resource blocks to pass through the bundle-MCP tool boundary without extracting nested `resource.text`, and the OpenAI-compatible replay path can still turn that into the reported image placeholder. The related HTTP MCP gateway issue and PR cover a different endpoint, so they do not supersede this agent-side runtime bug.

## What This Changes

The issue asks OpenClaw to preserve text from MCP `EmbeddedResource` tool-result blocks by passing `resource.text` to the model instead of `(see attached image)`.

## Best Possible Solution

Normalize MCP tool-result content at the bundle-MCP to pi-agent boundary so text embedded resources become standard OpenClaw text tool-result blocks before persistence and provider replay, while leaving image handling intact and using explicit fallbacks for unsupported resource/blob variants.

## Reproduction Assessment

Yes. A source-level reproduction is clear: return an MCP `CallToolResult` with `content: [{ type: "resource", resource: { text: "..." } }]`; current main casts it through the materializer and later provider conversion sees no top-level text, matching the placeholder path.

## Solution Assessment

Yes. Extracting `resource.resource.text` is the right behavior, but the narrowest maintainable fix is at the MCP-to-OpenClaw tool-result normalization boundary rather than duplicating resource handling independently in every provider transport.
