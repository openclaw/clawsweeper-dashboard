## Summary

Keep open. Current main still passes MCP `resource` content through the bundle-MCP tool boundary without converting `resource.text`, and the downstream OpenAI/OpenAI-replay/Anthropic paths only extract top-level text or image blocks. The linked open PR #75714 has closing syntax for this issue, so the issue should remain open until that PR or an equivalent fix lands.

## What This Changes

MCP tool results containing `EmbeddedResource` blocks should expose `resource.text` to the model instead of becoming an image placeholder or empty tool output.

## Best Possible Solution

Land #75714 or an equivalent narrow fix that normalizes MCP `EmbeddedResource` content at the bundle-MCP to Pi `AgentToolResult` boundary, converting text resources into text blocks and adding focused regression coverage for the materializer and provider replay behavior.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to return `content: [{ type: "resource", resource: { text: "..." } }]` from a bundle-MCP tool; current main preserves that raw block, after which the OpenAI-compatible path extracts no text and emits the image placeholder.

## Solution Assessment

Yes, the boundary-normalization direction is the narrowest maintainable fix because Pi tool results only support text/image while MCP tool results can include resources. The open PR should still receive normal review and validation before merge.
