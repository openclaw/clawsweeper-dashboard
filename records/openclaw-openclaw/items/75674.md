## Summary

Keep open. Current main still passes MCP EmbeddedResource blocks through the bundle-MCP tool boundary without converting nested resource text into Pi text content, and downstream OpenAI-compatible, replay, and Anthropic paths only extract top-level text/image blocks. The linked open PR #75714 explicitly targets this issue, so this issue should remain open until that PR or an equivalent fix lands.

## What This Changes

MCP tool results containing EmbeddedResource blocks should expose resource.text to the model instead of surfacing the image placeholder.

## Best Possible Solution

Land #75714 or an equivalent narrow boundary fix that converts MCP EmbeddedResource text into Pi text tool-result content before provider transports see it, with focused regression tests for text resources and bounded behavior for non-text resources.

## Reproduction Assessment

Yes. A source-level reproduction is clear: have a bundle MCP tool return content with { type: "resource", resource: { text: "..." } }, then follow current main through toAgentToolResult and the provider conversion path, where no top-level text is found.

## Solution Assessment

Yes. Normalizing once at the bundle-MCP to Pi AgentToolResult boundary is the narrowest maintainable fix; patching each transport separately would leave the same class of bug for replay or future providers.
