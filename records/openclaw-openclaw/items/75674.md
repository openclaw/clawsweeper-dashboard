## Summary

Keep open. Current main still passes MCP `resource` tool-result blocks through the bundle-MCP boundary unchanged, and the downstream provider/replay paths still only extract top-level text or image content; the linked closing PR #75714 is open and unmerged, so this issue should wait for that PR to land, be revised, or be replaced.

## What This Changes

MCP tool results containing `EmbeddedResource` text should pass `resource.text` to the model instead of the `(see attached image)` placeholder.

## Best Possible Solution

Normalize MCP `EmbeddedResource` text into Pi text content at `src/agents/pi-bundle-mcp-materialize.ts`, add regression coverage, and review or replace #75714 as the focused implementation path.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: return `content: [{ type: "resource", resource: { text: "..." } }]` from a bundle MCP tool, and current main preserves that raw block until provider paths that only read top-level text/image content.

## Solution Assessment

Yes. Boundary normalization is the narrowest maintainable fix because Pi/provider transports already expect text/image tool-result content; patching each downstream transport would leave the MCP-to-Pi shape mismatch in place.
