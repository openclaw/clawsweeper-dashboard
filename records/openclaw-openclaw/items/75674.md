## Summary

Keep open. Current main still allows MCP embedded resource blocks to pass through the bundle-MCP tool boundary as raw tool-result content, and downstream OpenAI-compatible/OpenAI replay/Anthropic conversion paths only extract top-level text or image blocks, so the reported placeholder/empty-output path remains. Related #68037/#68046 cover the HTTP MCP gateway content normalization path, not this agent-side bundle-MCP/provider replay path.

## What This Changes

Handle MCP EmbeddedResource tool results so nested `resource.text` is delivered as normal tool-result text instead of the image placeholder.

## Best Possible Solution

Normalize MCP tool-result content at the bundle-MCP to Pi `AgentToolResult` boundary: convert text embedded resources with `resource.text` into text blocks, preserve supported image blocks, and use explicit bounded fallbacks for unsupported MCP block variants. Add focused regression coverage for the materializer and provider replay path touched by the fix.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: an MCP `CallToolResult` containing `content: [{ type: "resource", resource: { text: "..." } }]` passes through the materializer unchanged, then the OpenAI-compatible conversion finds no top-level text and emits the placeholder.

## Solution Assessment

Yes. Extracting `resource.resource.text` is the right behavior, and the narrowest maintainable fix is to normalize at the MCP-to-OpenClaw tool-result boundary, with replay compatibility only if persisted raw resources need support.
