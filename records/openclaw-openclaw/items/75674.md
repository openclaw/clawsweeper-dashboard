## Summary

Keep open. Current main still passes MCP `resource` tool-result blocks through the bundle-MCP boundary unchanged, while the downstream OpenAI/replay/Anthropic conversion paths only extract top-level text or image content; #75714 is an open unmerged closing PR and should be reviewed or replaced before this issue closes.

## What This Changes

The issue requests that MCP `EmbeddedResource` tool-result blocks expose `resource.text` to model/provider contexts instead of becoming the misleading image placeholder.

## Best Possible Solution

Normalize MCP `EmbeddedResource` blocks once at the bundle-MCP to Pi `AgentToolResult` boundary, add regression coverage for text resources, and keep provider transports generic.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: return `content: [{ type: "resource", resource: { text: "..." } }]` from a bundle MCP tool, and current main passes it through unchanged into provider paths that only read top-level text/image blocks.

## Solution Assessment

Yes. Boundary normalization is the narrowest maintainable fix because Pi/provider transports already expect text/image content; patching each transport separately would leave the MCP-to-Pi shape mismatch for future providers.
