## Label Justifications

- `P1`: The issue describes a user-facing Slack workflow where long-running work can finish or partially finish but leave no visible final message or status.
- `impact:message-loss`: The affected path can leave Slack replies or media delivery unresolved with only queue/log state visible to the operator.
- `impact:session-state`: The report involves long-running session state and recovery after restart, where the user cannot tell whether the run is active, aborted, or only delivery-failed.

## Summary

Keep open: current main and the latest stable tag still match the reported Slack delivery failure mode, and no broader canonical fix has landed. The only related open PR I found addresses a narrower logging gap for failed queue-state writes, not Slack reconciliation or a user-visible unresolved-delivery fallback.

## What This Changes

The issue requests visible, recoverable Slack behavior when long-running media replies partially fail, recovery refuses unsafe replay, and the user otherwise sees no final status.

## Best Possible Solution

Add Slack-owned unknown-send reconciliation where Slack can prove delivery state, plus a safe unresolved-delivery notice or manual recovery path through the existing channel-message lifecycle without enabling blind replay.

## Reproduction Assessment

Yes, at source level: a Slack queued delivery in `send_attempt_started` or `unknown_after_send` reaches recovery without `reconcileUnknownSend`, refuses replay, and moves/fails the entry. I did not run live Slack, but current tests and the latest stable tag encode the same behavior.

## Solution Assessment

Unclear pending owner decision: the right direction is the existing `reconcileUnknownSend` seam plus a visible unresolved-delivery path, while blind replay or a broad status rewrite would be riskier than necessary.
