## Label Justifications

- `P1`: The report describes a user-facing Slack workflow where long-running work may appear silent after delivery failure and restart recovery.
- `impact:message-loss`: The affected path can leave the Slack user without the final reply, error, or recovery notice.
- `impact:session-state`: The issue involves restart recovery and long-running session state becoming ambiguous to the user.

## Summary

Keep open: current main still matches the reported Slack failure family. Best-effort partial media delivery can leave a failed queue entry, recovery refuses started/unknown sends without channel reconciliation, and Slack does not currently declare `reconcileUnknownSend`, so the unresolved state is logged rather than made visible in the Slack thread.

## What This Changes

The issue asks OpenClaw to make Slack long-running delivery failures visible and recoverable when media delivery partially fails and restart recovery refuses unsafe replay.

## Best Possible Solution

Add Slack-owned unknown-send reconciliation where Slack can prove sent/not-sent, plus a safe unresolved-delivery notice or manual recovery path for ambiguous cases, with focused text-plus-media restart recovery coverage.

## Reproduction Assessment

Yes, at source level: a Slack queued delivery in `send_attempt_started` or `unknown_after_send` reaches the generic recovery refusal path because Slack does not declare `reconcileUnknownSend`. I did not run a live Slack setup, but current tests exercise the same no-reconciliation branch.

## Solution Assessment

Partly: the existing `reconcileUnknownSend` channel-message seam is the right boundary, and blind replay would be unsafe. The visible fallback notice and manual recovery policy need maintainer agreement before implementation.
