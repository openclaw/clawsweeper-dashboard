- `impact:message-loss`: The central symptom is a Slack reply or recovery notice not reaching the visible thread after delivery partially fails.
- `impact:session-state`: The user-visible state of a long-running session can drift from the actual completed or partially completed backend work.

## Summary

Keep open: current main still matches the reported Slack failure mode, where best-effort partial media delivery can leave a failed queued entry and restart recovery refuses replay because Slack has no unknown-send reconciler.

## What This Changes

Slack long-running final media delivery should surface a visible success, error, or recovery path when partial media sends become unreplayable after restart.

## Best Possible Solution

Add Slack-owned unknown-send reconciliation plus a safe unresolved-delivery notice or recovery action through the existing channel-message lifecycle, while preserving the no-blind-replay duplicate-safety rule.

## Reproduction Assessment

Yes, at source level: a Slack queued delivery in send_attempt_started or unknown_after_send reaches the recovery refusal path because Slack does not declare reconcileUnknownSend. I did not run a live Slack setup in this read-only review.

## Solution Assessment

Partly: the right boundary is the existing reconcileUnknownSend channel-message seam plus narrow Slack-visible unresolved-delivery surfacing. The exact notice/replay policy needs maintainer judgment to avoid duplicate Slack spam.
