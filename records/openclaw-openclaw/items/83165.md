## Label Justifications

- `P1`: The report describes a real Slack workflow where long-running replies can appear to vanish after delivery failure and restart recovery.
- `impact:message-loss`: The affected behavior can leave the Slack user without the final message, error, or recovery notice.

## Summary

Keep open: current main and v2026.5.18 still match the reported source-level failure mode, and the related Slack receipt PR is open and explicitly does not add Slack fetch-back/reconciliation.

## What This Changes

The issue asks for Slack delivery recovery, reconciliation, and visible failure/progress handling when long-running media replies partially fail.

## Best Possible Solution

Add Slack-owned unknown-send reconciliation where Slack APIs can prove delivery state, plus a conservative unresolved-delivery notice or manual recovery path with focused text-plus-media restart recovery coverage.

## Reproduction Assessment

Yes, at source level: a queued Slack delivery in send_attempt_started or unknown_after_send reaches the generic recovery refusal path because Slack does not declare reconcileUnknownSend. I did not run a live Slack setup.

## Solution Assessment

Partly: the existing reconcileUnknownSend channel-message seam is the right boundary, but the fallback notice/manual recovery policy needs maintainer judgment to avoid duplicate Slack posts.
