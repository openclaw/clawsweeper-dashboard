## Summary

Keep open: current source and tests confirm the no-blind-replay safety behavior, but Slack lacks a reconciliation implementation and there is no user-visible fallback notice when ambiguous best-effort media delivery is moved to failed recovery state.

## What This Changes

The issue requests safer Slack visibility and recovery when long-running runs end in partial media delivery failure and restart recovery cannot safely replay the queued send.

## Best Possible Solution

Add a maintainer-approved Slack recovery path that preserves no blind replay while making ambiguous failed delivery visible, ideally through `reconcileUnknownSend` where Slack can prove state and a clear fallback/manual recovery notice where it cannot.

## Reproduction Assessment

Yes, source-level only: current main can reach `partial delivery failure (bestEffort)`, then recovery refuses replay for `send_attempt_started` when the Slack adapter has no reconciliation. I did not reproduce the full long-running Slack session against a real workspace.

## Solution Assessment

Unclear as an implementation plan: the requested direction is sound, but the exact best fix needs maintainer judgment on Slack reconciliation versus fallback notices and manual recovery semantics.
