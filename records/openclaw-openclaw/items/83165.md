## Label Justifications

- `P1`: A real Slack Socket Mode workflow can leave long-running runs with no visible final success, error, or recovery signal after delivery partially fails.
- `impact:message-loss`: The issue is about Slack messages and media replies becoming invisible or unrecoverable from the user's thread after partial delivery failure.
- `impact:session-state`: The failure is tied to long-running session/restart recovery state such as `send_attempt_started`, active work drain, and ambiguous delivery queue entries.

## Summary

Keep open: current main and the latest shipped release still match the reported Slack failure mode, and related open work covers adjacent receipt/logging gaps but not Slack started-entry reconciliation or a user-visible unresolved-delivery fallback.

## What This Changes

The issue asks OpenClaw to make Slack long-running media delivery failures visible and recoverable when partial delivery leaves a started queue entry that recovery cannot safely replay.

## Best Possible Solution

Add Slack-owned unknown-send reconciliation plus a small unresolved-delivery notice or manual recovery path, with focused text-plus-media restart recovery tests and live Slack proof.

## Reproduction Assessment

Yes, at source level: a Slack queued delivery in `send_attempt_started` or `unknown_after_send` reaches the recovery refusal path because Slack does not declare `reconcileUnknownSend`. I did not run a live Slack workspace in this read-only pass.

## Solution Assessment

Partly: the best boundary is the existing channel-message `reconcileUnknownSend` seam plus narrow Slack-visible unresolved-delivery surfacing. Blind replay or a broad long-running status rewrite would be riskier than owner-scoped reconciliation and fallback policy.
