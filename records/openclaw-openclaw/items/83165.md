- `impact:message-loss`: The central symptom is that final Slack replies or recovery notices may not reach the user after partial media delivery failure.

## Summary

Keep open: current main still matches the reported failure mode. Slack best-effort media delivery can leave an ambiguous started queue entry, recovery refuses unsafe replay without Slack reconciliation, and the unresolved state is logged rather than surfaced clearly in the Slack thread.

## What This Changes

The issue asks OpenClaw to make Slack long-running delivery failures visible and recoverable when media delivery partially fails and queued recovery cannot safely replay.

## Best Possible Solution

Add Slack-owned unknown-send reconciliation where Slack APIs can prove delivery state, and add a narrow unresolved-delivery notice or manual recovery path for ambiguous text-plus-media deliveries while keeping blind replay disabled by default.

## Reproduction Assessment

Yes, at source level: a Slack queued delivery in send_attempt_started reaches the recovery refusal path because Slack does not currently provide reconcileUnknownSend. I did not run a live Slack Socket Mode reproduction, but current source and tests match the reported state transition.

## Solution Assessment

Partly: the maintainable boundary is the existing reconcileUnknownSend seam plus a Slack-visible unresolved-delivery notice or recovery action. A blind replay or broad status/progress rewrite would be riskier than the bug requires.
