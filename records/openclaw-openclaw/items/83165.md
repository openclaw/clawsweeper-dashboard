## Summary

Keep open: current main still matches the reported Slack failure mode, and the related open Slack delivery issues/PRs are adjacent rather than a complete superseding fix for partial media delivery plus recovery refusal.

## What This Changes

The issue asks OpenClaw to make Slack long-running runs visibly recoverable when partial media delivery leaves queued sends in an ambiguous state that recovery will not blindly replay.

## Best Possible Solution

Add Slack-owned unknown-send reconciliation where Slack can prove sent/not-sent, plus a safe unresolved-delivery notice or recovery path through the existing channel-message contract, without blind replaying ambiguous sends.

## Reproduction Assessment

Yes, at source level: current main can put a queued delivery into `send_attempt_started`/`unknown_after_send`, Slack does not declare `reconcileUnknownSend`, and recovery refuses replay without a Slack-visible fallback. I did not run a live Slack setup.

## Solution Assessment

Partly: the existing `reconcileUnknownSend` channel-message seam is the right boundary, and blind replay is not safe. The unresolved part is the maintainer policy for Slack API reconciliation, manual recovery, and when to post a fallback notice.
