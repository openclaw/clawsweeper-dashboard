## Summary

Keep open: current main still matches the reported Slack delivery-recovery failure mode, and the repair needs Slack-owned reconciliation plus a user-visible fallback policy rather than conservative cleanup closure.

## What This Changes

Make Slack long-running runs visibly recoverable when best-effort media delivery partially fails and recovery refuses to replay an uncertain send.

## Best Possible Solution

Add Slack-owned unknown-send reconciliation and a safe unresolved-delivery notice or manual recovery path through the existing channel-message seam, with focused text-plus-media restart recovery coverage and no blind replay.

## Reproduction Assessment

Yes, at source level: a Slack queued delivery in send_attempt_started or unknown_after_send reaches the generic refusal branch because Slack does not declare reconcileUnknownSend. I did not run a live Slack setup, so this is source-reproducible rather than live-reproduced.

## Solution Assessment

Partly: the right boundary is Slack-owned reconcileUnknownSend plus a narrow unresolved-delivery notice or recovery action. Blind replay or a broad status/progress rewrite would be riskier and needs maintainer product judgment.
