## Label Justifications

- `P1`: The report describes a real long-running Slack workflow where completed or partially completed work can become invisible to the user after delivery recovery fails.
- `impact:message-loss`: The central failure mode is a Slack reply or media payload that may not reach the thread and is not visibly recovered.
- `impact:session-state`: The issue concerns long-running session and recovery state after restart, where active work and queued delivery state diverge from user-visible status.

## Summary

Keep open: current main still matches the reported Slack visibility gap. Best-effort media failures can leave a queued delivery unresolved, recovery refuses started sends without adapter reconciliation, and Slack still does not declare the reconciliation capability needed to safely resolve the ambiguity.

## What This Changes

The issue asks for Slack long-running runs to surface partial media delivery failures, safely reconcile or recover unknown sends, and keep user-visible status when final delivery or tool/media handling fails.

## Best Possible Solution

Add Slack-owned unknown-send reconciliation plus a safe Slack-visible unresolved-delivery notice or manual recovery path, with focused coverage for text-plus-media restart recovery and without blind replay.

## Reproduction Assessment

Yes, at source level: a Slack delivery entry in send_attempt_started or unknown_after_send reaches the recovery refusal path because Slack does not declare reconcileUnknownSend. I did not run a live Slack setup, but current tests already cover the generic refusal behavior.

## Solution Assessment

Partly: the maintainable boundary is the existing channel-message reconcileUnknownSend seam plus a narrow Slack-visible unresolved-delivery fallback. Blind replay or a broad long-running-status rewrite would be riskier than fixing the delivery ambiguity and surfacing it.
