## Summary

Keep open: current main matches the reported failure mode. Best-effort partial media failures can leave a failed delivery entry, recovery refuses started Slack entries because Slack does not declare unknown-send reconciliation, and the current behavior is only logged rather than made visible in the Slack thread.

## What This Changes

Request to make Slack long-running media deliveries visible and recoverable when best-effort media send partially fails and startup recovery cannot safely replay.

## Best Possible Solution

Add Slack-owned unknown-send reconciliation and a safe unresolved-delivery notice or recovery path using the existing channel-message capability seam, with focused text-plus-media restart recovery coverage.

## Reproduction Assessment

Yes, at source level: a queued Slack delivery in `send_attempt_started` or `unknown_after_send` reaches the recovery refusal path because Slack does not declare `reconcileUnknownSend`. I did not run a live Slack setup, but current tests already exercise the generic failure branch.

## Solution Assessment

Partly: the right boundary is the existing `reconcileUnknownSend` channel-message seam plus narrow user-visible delivery-failure surfacing. A broad status/progress rewrite or blind replay would be a riskier fix.
