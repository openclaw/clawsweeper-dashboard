## Label Justifications

- `P1`: The issue describes a real user-facing Slack workflow where long-running runs can appear silent and final delivery state is not visible after partial media failure.
- `impact:message-loss`: The reported failure leaves Slack replies or recovery notices absent from the user-visible thread after delivery uncertainty.

## Summary

Keep open: current main still matches the central Slack recovery gap. Best-effort partial media delivery can leave a failed queue entry, recovery refuses started entries unless the channel can reconcile unknown sends, and Slack still declares no `reconcileUnknownSend` capability; the merged middleware fix in https://github.com/openclaw/openclaw/pull/83029 only addresses an adjacent post-processing symptom.

## What This Changes

The issue requests Slack-visible recovery, diagnostics, and safer delivery handling when long-running Slack runs hit partial media delivery failure or ambiguous tool/media failure states.

## Best Possible Solution

Implement Slack-owned `reconcileUnknownSend` where Slack receipts/history can prove sent or not-sent, and add a safe unresolved-delivery notice or manual recovery path for ambiguous media sends without blind replay.

## Reproduction Assessment

Yes, at source level: a Slack queued delivery in `send_attempt_started` or `unknown_after_send` reaches the recovery refusal path because Slack does not declare `reconcileUnknownSend`. I did not run a live Slack setup, so the platform-side partial-media sequence remains unverified.

## Solution Assessment

Unclear until a maintainer chooses the Slack recovery policy. The narrowest maintainable direction is to use the existing `reconcileUnknownSend` seam plus a bounded unresolved-delivery notice, not blind replay or a broad status rewrite.
