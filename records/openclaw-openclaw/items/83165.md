## Summary

Keep open: current main and the latest shipped release still match the reported Slack failure mode, and related threads do not supersede this narrower text-plus-media recovery gap. The source-repro path is clear, but the repair needs maintainer judgment around Slack reconciliation, fallback notices, and duplicate-send safety.

## What This Changes

The issue asks OpenClaw to make Slack long-running delivery failures visible and recoverable when best-effort media sends partially fail and crash recovery refuses replay without reconciliation.

## Best Possible Solution

Add Slack-owned unknown-send reconciliation where Slack readback can prove sent/not-sent, plus a safe unresolved-delivery notice or manual recovery path for ambiguous text-plus-media sends.

## Reproduction Assessment

Yes, at source level: a queued Slack delivery in `send_attempt_started` or `unknown_after_send` reaches the recovery refusal branch because Slack does not declare `reconcileUnknownSend`. I did not run a live Slack setup.

## Solution Assessment

Partly: the existing channel-message `reconcileUnknownSend` seam is the right boundary, but the fallback-notice and ambiguous-send policy need maintainer judgment before implementation.
