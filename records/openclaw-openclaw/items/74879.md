## Summary

Keep open: current main still has a blocking onboarding completion-cache path, and the manual completion command can still load the full subcommand/plugin command tree. Related #72910 is open, but its stated scope is update/doctor timeout/progress behavior rather than the full onboarding/manual completion hang reported here.

## What This Changes

Fix a recent Docker/umbrelOS onboarding regression where setup and shell-completion cache generation take minutes or hang around completion or channel setup.

## Best Possible Solution

Make automatic onboarding and doctor cache refresh bounded, core-only, or deferred, and make explicit `openclaw completion --write-state` observable and safe when full plugin-command completion is slow.

## Reproduction Assessment

Yes for a source-level reproduction path: fresh QuickStart without an installed completion cache reaches `setupWizardShellCompletion`, calls `ensureCompletionCacheExists`, and blocks on `completion --write-state`. The comments add shipped Docker evidence for the manual hang, but this review did not run a live timing repro.

## Solution Assessment

Unclear because there is no patch on this issue, and raising timeouts alone leaves onboarding/manual cache generation coupled to full plugin command loading. A narrower durable fix should avoid the full plugin tree on automatic setup paths and add bounded feedback for the manual path.
