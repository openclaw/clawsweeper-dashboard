## Summary

Keep open: current main still puts shell-completion cache generation on the onboarding finish path, and the cache helper still synchronously runs `completion --write-state` with a 30-second timeout while loading plugin commands unless an env escape hatch is set. The related merged PR improved the update timeout message, and the related open PR addresses update/doctor timeout symptoms, but neither proves that QuickStart onboarding avoids the blocking full-cache path.

## What This Changes

The issue reports a regression where recent Docker/umbrelOS onboarding takes minutes and shell completion times out compared with older releases.

## Best Possible Solution

Keep full plugin-command cache generation available for explicit `openclaw completion --write-state`, but make automatic onboarding use a bounded core-only cache path or defer full cache generation until after onboarding reaches the usable Control UI/dashboard surface.

## Reproduction Assessment

Yes for a source-level reproduction path: a fresh QuickStart with no installed completion reaches `setupWizardShellCompletion`, calls `ensureCompletionCacheExists`, and synchronously spawns `completion --write-state` with a 30-second timeout. The exact Docker/umbrelOS timing still needs live performance proof.

## Solution Assessment

Unclear for the related open PR because raising update/doctor timeouts does not remove the onboarding blocking path. A focused onboarding change that skips plugin command loading or defers cache generation is the narrower maintainable fix.

## Repair Work Prompt

Fix issue #74879 by removing the blocking full completion-cache path from automatic QuickStart onboarding. Current main awaits `setupWizardShellCompletion`, QuickStart calls `ensureCompletionCacheExists`, and that helper synchronously runs `completion --write-state` with a 30s timeout while plugin command loading is enabled. Preserve explicit `openclaw completion --write-state` behavior for full plugin commands. Prefer adding a small option/seam to `generateCompletionCache` / `ensureCompletionCacheExists` so onboarding can set `OPENCLAW_COMPLETION_SKIP_PLUGIN_COMMANDS=1`, or defer the full cache refresh until after onboarding reaches the Control UI/dashboard surface. Keep update/doctor behavior compatible with related #72910 and do not broaden plugin loading or change unrelated completion install semantics. Update focused tests for QuickStart and any helper contract changes, and add a single-line `CHANGELOG.md` fix entry with `Fixes #74879` and `Thanks @al-lac`.
