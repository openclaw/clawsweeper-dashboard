## Summary

Keep open: current main still keeps shell-completion cache generation on the QuickStart onboarding path, and that helper still runs a synchronous `completion --write-state` subprocess with the 30-second timeout and plugin command loading enabled by default.

## What This Changes

The issue reports a regression where OpenClaw 2026.4.26+ onboarding and shell completion cache generation are much slower than earlier releases, especially on Docker/umbrelOS.

## Best Possible Solution

Keep full plugin-command completion generation available for explicit `openclaw completion --write-state`, but make automatic onboarding use a bounded core-only cache path or defer full cache generation until after onboarding reaches the usable dashboard/control surface.

## Reproduction Assessment

Yes for a source-level reproduction path: a fresh QuickStart with no installed completion reaches `setupWizardShellCompletion`, calls `ensureCompletionCacheExists`, and synchronously spawns `completion --write-state`. The exact Docker/umbrelOS timing still needs live timing proof.

## Solution Assessment

Unclear because this issue has no patch and #72910 appears to address timeout/progress symptoms rather than removing the onboarding blocking path. A core-only or deferred onboarding cache generation path looks narrower than only raising the timeout.
