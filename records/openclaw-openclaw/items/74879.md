## Summary

Keep open: current main still leaves the reported onboarding slowdown path in place. Classic QuickStart onboarding awaits shell-completion setup, that setup auto-generates the cache without prompting, and the cache helper still runs the full `completion --write-state` subprocess with the 30-second timeout and without the core-only completion mode used by the update follow-up.

## What This Changes

The issue requests a fix for a regression where recent Docker/umbrelOS onboarding takes minutes and shell completion often times out after about 30 seconds.

## Best Possible Solution

Keep full plugin-command completion generation for explicit `openclaw completion --write-state`, but make automatic onboarding use a bounded core-only cache path or defer cache generation until after the user reaches the dashboard/control surface, with focused tests around QuickStart onboarding and completion cache generation.

## Reproduction Assessment

Yes for a source-level reproduction path: a fresh classic QuickStart with no installed completion reaches `setupWizardShellCompletion`, calls `ensureCompletionCacheExists`, and blocks on a synchronous `completion --write-state` subprocess. The exact Docker/umbrelOS timing still needs live timing proof.

## Solution Assessment

Unclear because there is no patch on this issue and #72910 appears to address update/doctor timeout behavior rather than the onboarding blocking path. Raising the timeout alone may reduce `ETIMEDOUT`, but it would still keep onboarding blocked longer than a core-only or deferred cache path.
