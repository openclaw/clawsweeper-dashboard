## Summary

Keep open: current main and the latest release still leave the onboarding/doctor completion-cache path tied to a 30s full completion rebuild, and the follow-up Docker comment reproduces the manual completion hang that the current code can still hit.

## What This Changes

The issue requests restoring fast onboarding and responsive shell completion after a regression in newer Docker installs.

## Best Possible Solution

Separate automatic onboarding/doctor cache refresh from full plugin-command completion loading, make manual cache generation bounded and observable, and collect channel-specific Docker timing if the selection stall persists.

## Reproduction Assessment

Yes for the shell-completion portion: source inspection shows onboarding/doctor can still invoke the slow completion-cache path, and the Docker follow-up reproduces the manual hang. The channel-selection stall remains only partially reproducible because the selected channel and timing breakdown are missing.

## Solution Assessment

Unclear because there is no patch on this issue and #72910 only covers part of the timeout/progress surface. The safer direction is a maintainer decision on core-only automatic cache refresh versus full manual plugin-command completion, then a narrow implementation.
