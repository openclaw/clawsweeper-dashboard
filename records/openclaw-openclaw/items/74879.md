## Summary

Keep open: current main still routes QuickStart onboarding through shell-completion cache generation, and that helper still spawns `completion --write-state` with the normal environment, 30-second timeout, and full plugin command registration unless an env flag is set. The related open PR #72910 only lengthens the update/doctor timeout and progress output; it does not remove the onboarding blocking path or address the manual full-plugin `completion --write-state` hang described in the latest comment.

## What This Changes

The issue asks to restore fast onboarding and shell completion behavior for newer Docker/Debian installs where completion cache generation stalls or times out.

## Best Possible Solution

Keep the issue open and make automatic onboarding/doctor completion-cache generation bounded and core-only or deferred so first-run setup reaches the dashboard promptly. Preserve explicit full plugin-command completion generation for users who request it, but make that manual path observable and fail boundedly when a plugin registrar hangs.

## Reproduction Assessment

Yes for the source-level path: a fresh QuickStart with no completion installed reaches `setupWizardShellCompletion`, calls `ensureCompletionCacheExists`, and blocks on a synchronous `completion --write-state` subprocess. The latest Docker comment also gives a concrete manual repro on 2026.4.29, though this review did not run a live Docker timing test.

## Solution Assessment

No current solution is merged. #72910 is useful but incomplete because raising update/doctor timeouts does not remove onboarding’s automatic full-cache generation or fix the direct manual `completion --write-state` hang; a core-only/deferred automatic path is the safer narrow fix.
