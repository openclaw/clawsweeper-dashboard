## Summary

Keep open: current main still puts shell-completion cache generation on the classic QuickStart onboarding path, and that helper still runs the 30-second synchronous `completion --write-state` subprocess without the core-only completion mode used by update follow-up.

## What This Changes

The issue reports a regression where Docker/umbrelOS onboarding in 2026.4.26+ takes minutes and shell completion often times out around 30 seconds compared with faster 2026.4.14 behavior.

## Best Possible Solution

Keep full plugin-command completion generation available for explicit `openclaw completion --write-state`, but make automatic onboarding use a bounded core-only cache path or defer full cache generation until after onboarding reaches the usable dashboard/control surface.

## Reproduction Assessment

Yes for a source-level reproduction path: a fresh classic QuickStart with no installed completion reaches `setupWizardShellCompletion`, calls `ensureCompletionCacheExists`, and synchronously spawns `completion --write-state` with the 30-second timeout. The exact Docker/umbrelOS timing still needs live timing proof.

## Solution Assessment

Unclear because this issue has no patch and #72910 appears to address timeout/progress symptoms rather than the onboarding blocking path. Raising the timeout alone may avoid `ETIMEDOUT`, but it can also keep onboarding blocked longer; a core-only or deferred onboarding cache path looks narrower.
