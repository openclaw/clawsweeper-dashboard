## Summary

Keep open: current main still leaves onboarding and doctor on the same blocking completion-cache path, while the related open PR appears to cover update/doctor timeout behavior rather than the full onboarding/manual `completion --write-state` hang reported in the latest comments.

## What This Changes

The issue reports a regression where OpenClaw onboarding and shell completion cache generation became much slower or timed out in Docker-based installs after the 2026.4.14 release line.

## Best Possible Solution

Keep automatic onboarding/doctor cache generation bounded and preferably core-only or deferred, while separately making explicit `openclaw completion --write-state` observable and non-hanging for full plugin-command completion rebuilds.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the source-level path: a fresh QuickStart with no completion profile reaches `setupWizardShellCompletion`, awaits `ensureCompletionCacheExists`, and spawns `completion --write-state`; the comments add shipped Docker 2026.4.29 evidence for the manual hang.

## Solution Assessment

Is this the best way to solve the issue? Unclear because there is no patch on this issue; increasing timeouts alone may reduce `ETIMEDOUT`, but the narrower maintainable fix should avoid blocking onboarding on full plugin-command completion generation.
