## Summary

Keep open: current main still keeps completion cache generation on the onboarding/doctor path, and the direct manual cache command still loads plugin CLI commands unless an internal env flag is set. The related open PR is narrower than the full Docker/manual/onboarding slowdown reported here.

## What This Changes

Report that onboarding on 2026.4.26+ and `openclaw completion --write-state` are much slower or hang compared with 2026.4.14, especially in Docker installs.

## Best Possible Solution

Decouple automatic onboarding and doctor cache refresh from full plugin CLI registration, make direct `completion --write-state` bounded and observable, then verify on the reported Docker/onboarding path.

## Reproduction Assessment

Yes for a high-confidence source-level path: fresh onboarding without an existing completion cache reaches `setupWizardShellCompletion`, awaits `ensureCompletionCacheExists`, and runs the synchronous `completion --write-state` child process. The issue comments add shipped 2026.4.29 Docker evidence for the manual hang, though I did not run live timing here.

## Solution Assessment

Unclear because there is no patch on this issue, and the related timeout PR does not cover the full manual plugin-command path. A durable fix should avoid full plugin loading for automatic setup/doctor work and add progress or a bounded failure mode for explicit manual cache generation.
