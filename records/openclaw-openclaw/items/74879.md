## Summary

Current main still leaves fresh onboarding on the synchronous shell-completion cache path, so the reported slowdown remains plausible and source-reproducible. The related open PR #72910 helps the timeout/update symptom, but it does not remove onboarding from the blocking full completion-generation path.

## What This Changes

The issue reports a regression where Docker-based onboarding on newer OpenClaw releases takes minutes and shell completion can time out instead of completing quickly as in older releases.

## Best Possible Solution

Keep full plugin-command cache generation for explicit completion repair or manual refreshes, but move automatic onboarding to a bounded path: either skip plugin commands during the onboarding cache write or defer full cache generation until after onboarding completes.

## Reproduction Assessment

Yes for a source-level reproduction path: fresh QuickStart onboarding reaches setupWizardShellCompletion, calls ensureCompletionCacheExists, and blocks on completion --write-state. The exact Docker/umbrelOS duration still needs live timing proof.

## Solution Assessment

Unclear because there is no patch on this issue, and the related PR mainly changes timeout/progress handling for update and doctor flows. A narrower onboarding fix would avoid or defer full plugin-command completion generation instead of only increasing the timeout.
