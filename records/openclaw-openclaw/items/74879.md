## Summary

Keep open: current main still keeps shell-completion cache generation on the QuickStart onboarding finish path, and that path can still run the full `completion --write-state` subprocess with the 30-second synchronous timeout reported by the issue. The related completion-cache PRs cover adjacent update/doctor messaging or timeout behavior, but they do not prove this onboarding regression is fixed on main.

## What This Changes

The issue reports that OpenClaw onboarding in 2026.4.26+ takes minutes and shell completion often times out around 30 seconds, compared with the faster 2026.4.14 flow.

## Best Possible Solution

Keep full plugin-command completion generation available for explicit `openclaw completion --write-state`, but make automatic onboarding use a bounded core-only cache path or defer full cache generation until after the user reaches a usable dashboard/control surface.

## Reproduction Assessment

Yes for a source-level reproduction path: a fresh QuickStart with no installed completion reaches `setupWizardShellCompletion`, calls `ensureCompletionCacheExists`, and synchronously spawns `completion --write-state` with the 30-second timeout. The exact umbrelOS/Docker runtime duration still needs live timing proof.

## Solution Assessment

Unclear because this issue has no patch, and the adjacent open PR appears focused on update/doctor timeout handling rather than removing the onboarding blocker. Raising the timeout alone may reduce `ETIMEDOUT`, but it can also keep onboarding blocked longer; a core-only or deferred onboarding cache path looks narrower.
