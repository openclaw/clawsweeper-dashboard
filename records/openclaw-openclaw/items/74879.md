## Summary

Keep open: current main still contains the reported onboarding/completion slowdown path, and later comments add fresh Docker/onboarding confirmations rather than making the report obsolete.

## What This Changes

The issue reports a regression where Docker-based onboarding and shell completion became minutes-long after 2026.4.26.

## Best Possible Solution

Move automatic onboarding/doctor cache generation off the full plugin-command path, likely by using a bounded core-only cache path or deferring cache generation, while keeping explicit full plugin completion either diagnostic and bounded or separately fixed.

## Reproduction Assessment

Yes for a source-level reproduction path: a fresh QuickStart with no installed completion reaches `setupWizardShellCompletion`, calls `ensureCompletionCacheExists`, and blocks on `completion --write-state`. Recent comments add live Docker/onboarding evidence, though this review did not time the Docker path locally.

## Solution Assessment

No current-main solution is sufficient: the update path has a core-only mitigation, but onboarding/doctor/manual completion still use the full write-state path. Raising the timeout alone may reduce `ETIMEDOUT` but would still leave onboarding waiting longer than necessary and does not address a manual command that can hang beyond 60 seconds.
