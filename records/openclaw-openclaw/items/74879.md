## Summary

Keep open: current main still keeps shell-completion cache generation on the onboarding path, and manual `completion --write-state` still eager-loads plugin commands unless an internal env flag is set. The 2026.4.29 Docker follow-up confirms the manual path can still hang, while related #72910 only covers part of the timeout/update surface.

## What This Changes

The issue reports a regression where Docker onboarding in 2026.4.26+ and shell completion cache generation take minutes or hang compared with 2026.4.14.

## Best Possible Solution

Bound automatic onboarding and doctor completion-cache generation away from full plugin command loading, make manual cache generation observable and bounded, and separately isolate any channel-selection hang if it persists.

## Reproduction Assessment

Yes for a source-level reproduction path: a fresh QuickStart without a completion cache reaches `setupWizardShellCompletion`, calls `ensureCompletionCacheExists`, and spawns `completion --write-state`. The shipped Docker comment adds a manual 2026.4.29 reproduction where that command stayed silent past 60 seconds.

## Solution Assessment

Unclear because there is no patch on this issue, and increasing timeouts alone would not address the manual full-plugin command path or the onboarding coupling. The safer fix is to use the existing core-only completion mode for automatic paths and add bounded feedback or an explicit mode for manual full-cache generation.
