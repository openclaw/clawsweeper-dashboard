## Summary

Keep open: current main still leaves the reported completion-cache slowdown path in place for classic onboarding and manual completion cache generation. The merged update mitigation is scoped to core-command cache refreshes, while onboarding/doctor still call the generic helper that spawns `completion --write-state` with a 30-second timeout and the manual command still eagerly registers plugin commands unless the private skip env var is set.

## What This Changes

The issue reports a regression where onboarding and shell completion cache generation became much slower in recent OpenClaw releases, especially Docker-based installs from 2026.4.26 onward.

## Best Possible Solution

Automatic setup paths should avoid blocking onboarding on full plugin-command completion generation: use a bounded core-only cache refresh or defer cache generation until after onboarding reaches the dashboard/control surface. The explicit `openclaw completion --write-state` path should also gain a non-silent failure/progress boundary for plugin command registration so doctor advice does not send users to a command that can hang indefinitely.

## Reproduction Assessment

Yes for a high-confidence source-level reproduction path: a fresh classic QuickStart with no installed completion calls `setupWizardShellCompletion`, which calls `ensureCompletionCacheExists`, which spawns `completion --write-state` with the doctor helper timeout. The later 2026.4.29 Docker comment also gives a direct manual reproduction for `openclaw completion --write-state` hanging beyond 60 seconds.

## Solution Assessment

No current-main solution is in place for the full reported behavior. The update path's core-only mitigation is useful, but raising or messaging the timeout alone is not the best complete fix while onboarding and manual full plugin completion can still block or hang silently.
