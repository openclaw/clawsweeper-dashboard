## Summary

Keep open: current main still leaves fresh QuickStart onboarding on a synchronous completion-cache generation path that can run the full plugin command tree under the same 30-second cache-write timeout. The related open PR is relevant to completion-cache timeouts, but its stated scope is update/doctor refresh behavior, so it does not yet close the onboarding-specific slowdown reported here.

## What This Changes

The issue requests restoring recent onboarding and shell-completion performance so OpenClaw 2026.4.26+ Docker/umbrelOS onboarding completes within seconds instead of minutes or timeout-prone completion-cache work.

## Best Possible Solution

Keep explicit `openclaw completion --write-state` as the full plugin-command rebuild path, but make automatic onboarding use a bounded core-only cache generation path or defer full plugin-command cache generation until after onboarding finishes, with focused tests and a user-facing changelog entry.

## Reproduction Assessment

Yes for a high-confidence source-level reproduction path: fresh QuickStart with no installed completion reaches `setupWizardShellCompletion`, auto-enables completion, and synchronously spawns `completion --write-state`. The exact multi-minute Docker/umbrelOS timing still needs live measurement if maintainers want performance numbers before merging a fix.

## Solution Assessment

No current-main solution is in place yet. Raising the timeout alone, as in the related PR's stated scope, may reduce `ETIMEDOUT` but is not the best onboarding fix because it can make the blocking onboarding step wait even longer instead of avoiding or deferring full plugin-command generation.

## Repair Work Prompt

Fix the onboarding shell-completion slowdown reported in https://github.com/openclaw/openclaw/issues/74879. Current main has QuickStart onboarding call `setupWizardShellCompletion`, which auto-enables completion and awaits `ensureCompletionCacheExists`; that helper spawns `openclaw.mjs completion --write-state` with a 30s timeout and full plugin command loading. Keep explicit `openclaw completion --write-state` as the full plugin-command rebuild path, but make automatic onboarding bounded and fast, preferably by adding an option/helper that sets `OPENCLAW_COMPLETION_SKIP_PLUGIN_COMMANDS=1` for onboarding cache generation or by deferring full plugin-command cache generation until after onboarding completes. Do not use a longer blocking onboarding timeout as the sole fix, do not change default explicit completion behavior, and coordinate with the related open PR #72910 if it has landed or changed the same helper. Add focused coverage showing QuickStart/onboarding uses the bounded/core-only path and existing explicit completion behavior remains intact. Add a single-line user-facing `CHANGELOG.md` fix entry crediting the issue reporter, not bot or maintainer handles.
