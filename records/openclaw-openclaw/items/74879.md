## Summary

Keep open: current main still leaves the reported completion-cache slowdown path in classic onboarding and doctor repair. The update follow-up has a core-only cache refresh, but onboarding/doctor still spawn `completion --write-state` with the 30-second helper timeout, and the manual command still loads plugin commands by default.

## What This Changes

The issue reports a regression in 2026.4.26+ where onboarding takes minutes and shell completion cache generation times out or hangs compared with earlier releases.

## Best Possible Solution

Keep automatic onboarding and doctor repair on a bounded core-only or deferred cache path, and make the user-facing manual recovery path unable to hang silently. Preserve full plugin-command completion only behind an explicit, observable path if plugin-owned command completions remain required.

## Reproduction Assessment

Yes. A source-level reproduction exists: fresh classic QuickStart reaches `setupWizardShellCompletion`, auto-calls `ensureCompletionCacheExists`, and the helper spawns `completion --write-state`; the latest discussion also reports the manual command hanging past 60 seconds on 2026.4.29 Docker.

## Solution Assessment

No current-main solution is in place. Raising the timeout in #72910 may reduce one update/doctor failure mode, but it does not by itself address onboarding blocking on cache generation or the manual full-plugin completion path that still hangs.
