## Summary

Keep open: current main still leaves the shell-completion/onboarding slowdown path unresolved, and the related open PR only addresses part of the update/doctor timeout surface while manual completion and channel-selection reports remain open.

## What This Changes

The issue reports a regression where Docker onboarding and shell completion became much slower in recent OpenClaw releases, especially around onboarding channel selection and `completion --write-state`.

## Best Possible Solution

Separate automatic onboarding/doctor completion-cache refresh from full plugin-command completion loading, make manual cache generation bounded and observable, and collect channel-specific Docker timing if channel selection still stalls.

## Reproduction Assessment

Yes for the shell-completion portion: source inspection and the Docker follow-up show the manual/full plugin-command path still exists on current main. The channel-selection stall is unclear without the selected channel, logs, or timing around plugin install/load/status refresh.

## Solution Assessment

Unclear because this issue has no patch and the related PR is only a partial mitigation. The narrow maintainable path needs maintainer direction on when completion generation should stay full-plugin versus bounded/core-only during onboarding, doctor, and manual rebuilds.
