## Summary

Keep open. Current main still lacks the Moonshot replay backfill this branch targets, and the functional approach is narrow, but the branch is not merge-ready because it omits the required user-facing changelog entry for a provider fix.

## What This Changes

The PR adds a Moonshot payload helper that injects empty `reasoning_content` on assistant messages with `tool_calls` when Moonshot thinking is enabled, plus regression tests for enabled, disabled, and pre-existing reasoning-content cases.

## Best Possible Solution

Land the Moonshot-only outgoing payload repair after adding the required Unreleased changelog fix entry, preserving existing `reasoning_content`, injecting an empty placeholder only when final Moonshot thinking is enabled for assistant tool-call replay, and leaving generic OpenAI-compatible replay policy unchanged.

## Reproduction Assessment

Yes. Current main source inspection plus #71491 gives a high-confidence path: a Moonshot thinking-enabled replay can contain assistant `tool_calls` without top-level `reasoning_content`, while Moonshot's thinking docs require preserving that field through multi-step tool calls. ([platform.moonshot.ai](https://platform.moonshot.ai/docs/guide/use-kimi-k2-thinking-model.en-US))

## Solution Assessment

Yes, once the changelog entry is added. The provider wrapper is the narrowest maintainable place for this compatibility repair and mirrors the existing DeepSeek V4 payload-patch pattern; preserving real reasoning through compaction would be better for quality, but the placeholder is a pragmatic rejection guard once history is already missing it.
