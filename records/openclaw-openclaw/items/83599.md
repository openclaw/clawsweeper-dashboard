- `impact:security`: The remaining fix direction changes sandbox and host-execution boundaries for Codex app-server native code-mode turns.

## Summary

Keep open: current main preserves the related network-egress fix, but it still intentionally sends active OpenClaw-sandboxed Codex turns through Codex `workspaceWrite`, which can require bubblewrap on Linux. The remaining request is not a duplicate of the closed egress issue because avoiding nested bwrap touches the Gateway-host sandbox boundary.

## What This Changes

The issue asks OpenClaw to avoid or document nested Codex bwrap use when Codex app-server is configured with `sandbox: "danger-full-access"` under an active Docker sandbox.

## Best Possible Solution

Define the supported Codex-under-OpenClaw-sandbox boundary: either keep the Codex bwrap requirement with clear preflight/docs, or add an explicit safe mode that moves execution into the outer sandbox before allowing no-bwrap operation.

## Reproduction Assessment

No high-confidence live current-main repro was run. Source inspection does show the active-sandbox Codex path still sends `workspaceWrite` sandbox policy, and upstream Codex uses bubblewrap/user namespaces for that Linux sandbox path.

## Solution Assessment

Unclear. Avoiding bwrap by honoring `danger-full-access` directly is not obviously safe while the app-server runs on the Gateway host; the maintainable fix needs a maintainer security/product decision first.
