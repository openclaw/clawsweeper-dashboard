## Summary

Keep open. Current main still cannot safely reconcile a QMD path/pattern conflict when QMD only reports the stale collection name without path/pattern metadata, so the reported `life` versus `life-main` boot/search failure remains a valid issue. The provided timeline shows PR #72297 as the active open implementation candidate, so this issue should stay open until that PR or an equivalent fix lands.

## What This Changes

The issue requests boot-time QMD collection registration to reconcile or clearly report stale collection names for the same path/pattern so memory search does not fail on startup.

## Best Possible Solution

Keep the issue open until PR #72297 or an equivalent memory-core change lands: preserve automatic rebind only for verified path/pattern matches, surface an explicit manual repair hint for unverified stale-name conflicts, and keep regression coverage for metadata-free QMD list output.

## Reproduction Assessment

Yes. The report includes concrete QMD stderr for the `life` versus `life-main` conflict, and current tests model the metadata-free collection-list plus add-conflict path that still produces a skipped-add warning.

## Solution Assessment

No for current main. The narrow maintainable solution is a metadata-gated automatic rebind plus a clear manual repair hint for unverified conflicts, rather than silently continuing or destructively removing a name parsed only from stderr.
