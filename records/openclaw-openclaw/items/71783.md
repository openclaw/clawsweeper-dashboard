## Summary

Keep open. Current main still leaves the reported metadata-free stale-name QMD conflict unresolved: it can only rebind a different collection name when QMD list output exposes matching path and pattern metadata, otherwise it logs a skipped add and searches continue to target the configured collection name. Open PR #72297 is the active implementation candidate, so this issue should stay open for review/landing rather than be closed or duplicated.

## What This Changes

The issue requests boot-time QMD collection reconciliation or clearer diagnostics when QMD already has the same path and pattern registered under a different collection name, causing searches for the configured name to fail.

## Best Possible Solution

Keep this open until PR #72297 or an equivalent memory-core change lands: preserve automatic rebind only for verified path and pattern matches, add an explicit stale-conflict repair hint when QMD exposes only the conflict name in stderr, and keep regression coverage for the metadata-free conflict path.

## Reproduction Assessment

Yes. The issue includes exact QMD stderr for the life versus life-main conflict, and current regression coverage models the metadata-free collection-list plus add-conflict path that still produces a skipped-add warning instead of recovery.

## Solution Assessment

Yes for the proposed direction, but no for current main. The narrow maintainable solution is a metadata-gated automatic rebind plus a manual repair hint for unverified conflicts, rather than deleting a collection name parsed only from stderr.
