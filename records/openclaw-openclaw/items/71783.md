## Summary

Keep open. Current main still leaves the metadata-free stale-name QMD conflict unresolved: it can rebind a different collection name only when QMD exposes matching path/pattern metadata, otherwise it logs a skipped add while later searches still target the configured collection name. PR #72297 remains the active open implementation candidate rather than a landed fix.

## What This Changes

The issue reports that a stale QMD collection registered under a different name for the same path/pattern prevents OpenClaw from registering and searching the configured collection name on boot.

## Best Possible Solution

Land PR #72297 or an equivalent memory-core change that keeps automatic rebind limited to verified path/pattern matches, parses or surfaces the stale QMD conflict name as a manual repair hint when metadata is missing, and keeps regression coverage for the metadata-free conflict path.

## Reproduction Assessment

Yes. The report includes concrete QMD stderr for the `life` versus `life-main` conflict, and current tests model the metadata-free collection-list plus add-conflict path that still produces a skipped-add warning.

## Solution Assessment

No for current main. The narrow maintainable fix is a metadata-gated automatic rebind plus an explicit manual repair hint for unverified conflicts, rather than silently continuing or deleting a collection based only on stderr.
