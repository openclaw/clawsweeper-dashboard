## Summary

Keep open. Current main still leaves the metadata-free stale-name QMD conflict unresolved: it can rebind a different collection name only when QMD exposes matching path and pattern metadata, otherwise it logs a skipped add and later searches still target the configured collection name. PR #72297 remains the active implementation candidate rather than a landed fix.

## What This Changes

The issue asks OpenClaw to reconcile or clearly surface QMD collections that already exist at the configured path/pattern under a stale different name, such as `life-main` versus `life`.

## Best Possible Solution

Keep this open until PR #72297 or an equivalent memory-core change lands: preserve automatic rebind only for verified path/pattern matches, parse or surface the stale conflict name from QMD stderr as a manual repair hint when metadata is missing, and keep regression coverage for the metadata-free conflict path.

## Reproduction Assessment

Yes. The issue gives exact QMD stderr for the `life` versus `life-main` conflict, and current tests model the metadata-free collection-list plus add-conflict path that still produces a skipped-add warning.

## Solution Assessment

No for current main. The narrow maintainable fix is a metadata-gated automatic rebind plus an explicit manual repair hint for unverified conflicts, rather than blindly deleting a name parsed only from stderr.
