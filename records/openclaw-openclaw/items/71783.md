## Summary

Keep open. Current main still leaves the metadata-free stale-name QMD conflict unresolved, and open PR #72297 is already the active implementation candidate for this issue.

## What This Changes

The issue asks OpenClaw to reconcile or clearly report QMD collection path/pattern conflicts when an existing QMD collection has the same path/pattern but a different name.

## Best Possible Solution

Review and land PR #72297, or an equivalent metadata-gated QMD conflict fix, then let the issue close after the implementation merges.

## Reproduction Assessment

Yes. The issue gives a concrete stale `life-main` versus configured `life` QMD stderr, and current main has a source-level path plus regression coverage where metadata-free list output followed by an already-exists add conflict logs `qmd collection add skipped`.

## Solution Assessment

Yes for keeping it open. The narrow maintainable solution is to preserve verified path/pattern rebinds while surfacing a clear manual repair hint when QMD only exposes the conflicting stale name; PR #72297 is already aimed at that path.
