## Summary

Keep open. Current main still does not fully resolve the reported QMD stale-name conflict: it can safely rebind when QMD list output proves a matching path and pattern, but a metadata-free list plus an add conflict still logs a skipped add and leaves searches targeting the configured collection name. Open PR #72297 is the active related implementation candidate, so the best maintainer action is review or land that PR rather than closing this issue now.

## What This Changes

Report requests QMD startup collection reconciliation or clearer diagnostics when configured collection `life` conflicts with an existing QMD collection at the same path/pattern under another name such as `life-main`.

## Best Possible Solution

Keep this open until PR #72297 or an equivalent memory-core change lands. The desired end state is verified automatic rebind for proven path/pattern matches plus an explicit stale-conflict repair hint when QMD only exposes the conflicting name in stderr, without deleting a collection based only on unverified stderr text.

## Reproduction Assessment

Yes. The issue includes exact QMD stderr for the `life` versus `life-main` conflict, and current unit coverage models the metadata-free list plus add-conflict path that still produces a skipped-add warning rather than recovery.

## Solution Assessment

Yes for the proposed safe direction, but no for current main. The narrow maintainable fix is metadata-gated automatic rebind plus a clear manual repair hint for unverified conflicts, not destructive removal based only on a parsed conflict name.
