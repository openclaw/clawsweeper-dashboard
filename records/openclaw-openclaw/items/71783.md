## Summary

Keep open. Current main still leaves the metadata-free stale-name QMD conflict unresolved: it can rebind a different collection name only when `collection list` exposes matching path/pattern metadata, then otherwise logs a skipped add while later searches still target the configured collection name. PR #72297 remains the active open implementation candidate rather than a landed fix.

## What This Changes

The issue requests that OpenClaw detect and reconcile an existing QMD collection with the same path/pattern but a different name, or surface a clearer repair hint when boot-time registration fails.

## Best Possible Solution

Keep this open until PR #72297 or an equivalent memory-core change lands: preserve automatic rebind only for verified path/pattern matches, parse or surface the stale conflict name from QMD stderr as a manual repair hint when metadata is missing, and keep regression coverage for the metadata-free conflict path.

## Reproduction Assessment

Yes. The report includes concrete QMD stderr for the `life` versus `life-main` conflict, and current tests model the metadata-free collection-list plus add-conflict path that still produces a skipped-add warning.

## Solution Assessment

No for current main. The narrow maintainable fix is a metadata-gated automatic rebind plus an explicit manual repair hint for unverified conflicts, rather than blindly deleting a name parsed only from stderr.
