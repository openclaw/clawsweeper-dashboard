## Summary

Keep open. Current main still preserves the reported failure mode for a stale QMD collection registered under a different name when QMD cannot provide path/pattern metadata, and open PR #72297 remains the active implementation candidate rather than a landed fix.

## What This Changes

The issue asks OpenClaw's QMD startup reconciliation to handle an existing collection registered at the same path/pattern under a different name, or surface the conflict clearly so searches do not fail with `Collection not found`.

## Best Possible Solution

Keep the issue open until PR #72297 or an equivalent memory-core change lands: keep automatic rebind limited to verified path/pattern matches, add a clear manual repair hint for unverified stale-name conflicts, and preserve regression coverage for the metadata-free conflict path.

## Reproduction Assessment

Yes. The issue includes concrete QMD stderr for the `life` versus `life-main` conflict, and current tests model the same metadata-free list plus add-conflict path that still produces a skipped-add warning.

## Solution Assessment

No for current main. The narrow maintainable solution is the linked implementation direction: metadata-gated automatic rebind plus an explicit manual repair hint, rather than silently continuing or deleting a stale name from stderr alone.
