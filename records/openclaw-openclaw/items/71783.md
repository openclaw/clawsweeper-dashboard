## Summary

Keep open. Current main still preserves the unresolved metadata-free stale-name QMD conflict path: it can safely rebind a different collection name only when QMD exposes matching path and pattern metadata, otherwise it logs a skipped add and searches continue to target the configured collection name. Open PR #72297 remains the active implementation candidate for this issue, so this should not be auto-closed or duplicated.

## What This Changes

Requests QMD boot collection registration to reconcile or clearly surface same-path/pattern collections registered under a stale different name, such as `life-main` when OpenClaw is configured to search `life`.

## Best Possible Solution

Keep this open until PR #72297 or an equivalent memory-core change lands: preserve automatic rebind only for verified path/pattern matches, surface the stale conflict name from QMD stderr as a manual repair hint when metadata is missing, and keep regression coverage for the metadata-free conflict path.

## Reproduction Assessment

Yes. The issue gives exact QMD stderr for the `life` versus `life-main` conflict, and current tests model the metadata-free collection-list plus add-conflict path that still produces a skipped-add warning.

## Solution Assessment

Yes for the proposed direction, but not for current main. The narrow maintainable fix is a metadata-gated automatic rebind plus an explicit manual repair hint for unverified conflicts, rather than blindly deleting a collection name parsed only from stderr.
