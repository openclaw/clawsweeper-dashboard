## Summary

Keep open. Current main still leaves the metadata-free stale-name QMD conflict unresolved: OpenClaw can rebind a different collection name only when QMD list output exposes matching path and pattern metadata, and otherwise logs a skipped add while later searches still target the configured collection name. Open PR #72297 remains the active implementation candidate rather than a landed fix.

## What This Changes

The issue asks OpenClaw to handle or clearly surface a QMD sidecar collection already bound to the target path/pattern under a different name, such as `life-main` instead of configured `life`.

## Best Possible Solution

Land PR #72297 or an equivalent memory-core change: keep automatic rebinds limited to verified path/pattern matches, parse or surface the stale conflict name from QMD stderr as a manual repair hint when metadata is missing, and retain regression coverage for the metadata-free conflict path.

## Reproduction Assessment

Yes. The issue gives exact QMD stderr for the `life` versus `life-main` conflict, and current tests model the metadata-free collection-list plus add-conflict path that still produces a skipped-add warning.

## Solution Assessment

Yes for the direction in PR #72297; no for current main. The narrow maintainable solution is to preserve metadata-gated automatic rebinds and add an explicit manual repair hint for unverified conflicts, rather than blindly deleting a name parsed only from stderr.
