## Summary

Keep open. The report matches a plausible current-main Control UI/WebChat state-loss path: session switching clears the active chat state before `chat.history` reloads, and there is no visible per-session message cache to restore in-flight or not-yet-durable messages when switching back.

## What This Changes

The issue asks OpenClaw to preserve WebChat/Control UI chat context when switching away from an active conversation and back.

## Best Possible Solution

Preserve per-session WebChat UI state across session switches while still reconciling with durable `chat.history` when it catches up, and add a regression test for switching away from an active or recently updated session and back.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection gives a focused likely reproduction path: populate visible messages in session A, switch to session B, switch back to A, and have `chat.history` return an empty or stale snapshot before the transcript catches up.

## Solution Assessment

Yes, the likely narrow fix belongs in the Control UI/WebChat session-switch state handling, not in `dmScope` configuration or channel routing. The fix should preserve local state per session and let durable `chat.history` replace it once current.
