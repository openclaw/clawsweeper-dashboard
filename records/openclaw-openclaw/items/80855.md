## Summary

Keep open. Current main still has a plausible WebChat/Control UI state-loss path: switching sessions clears visible chat state and run state before `chat.history` reloads, so an empty or stale reload cannot restore the switched-away session's in-flight or not-yet-durable messages.

## What This Changes

WebChat should preserve an active conversation when the user switches to another conversation and then switches back.

## Best Possible Solution

Preserve per-session Control UI/WebChat visible state across session switches, then reconcile it with durable `chat.history` when current data arrives, with regression coverage for switching away from an active or recently updated session and back.

## Reproduction Assessment

No live high-confidence reproduction was run. Source inspection gives a focused current-main path: populate visible messages in session A, switch to session B, switch back to session A, and have `chat.history` return empty or stale data after the UI cleared A's local state.

## Solution Assessment

Yes, the narrow maintainable fix belongs in Control UI/WebChat session-switch state handling, not in `dmScope`, channel routing, or provider behavior. It should preserve local per-session UI state while still allowing durable history and explicit resets to win.
