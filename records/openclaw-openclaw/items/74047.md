## Summary

Keep open. Current main has useful service-layout/status/doctor diagnostics and system-scoped restart guidance, but it still lacks a supported split user/system install migration contract and no source, docs, or test evidence explains the newer-install `/v1/chat/completions` timeout; the related open PR #70306 only covers one legacy unit-resolution gap.

## What This Changes

A Linux Gateway upgrade can leave the live systemd service pinned to an older system install while a newer user install and gpt-5.5 local runtime appear to work, making `/v1/chat/completions` unreliable after repointing.

## Best Possible Solution

Keep this as the canonical tracker until status/doctor/update/docs and Package Acceptance or Testbox cover split user/system Gateway migration, legacy unit resolution, and a gpt-5.5 `/v1/chat/completions` smoke after service promotion.

## Reproduction Assessment

No. The report has credible rollback evidence, but I found no current fixture or live package/service lane that installs an older system unit, promotes a newer user install, and reproduces the timeout.

## Solution Assessment

Unclear. The existing diagnostics and #70306's legacy unit fallback are plausible partial fixes, but the maintainable solution depends on the chosen split-install policy and proof of the Gateway timeout root cause.
