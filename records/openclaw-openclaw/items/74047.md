## Summary

Keep open. Current main has useful adjacent service-layout diagnostics, doctor/status surfacing, package-update restart checks, and Codex provider fixes, but it still does not prove or document a safe split user/system install migration for the reported managed Gateway path, nor does it reproduce the newer-install `/v1/chat/completions` timeout.

## What This Changes

The issue asks OpenClaw to clarify or fix split user/system install upgrades where the CLI and active Gateway service can run different OpenClaw versions and the newer service path becomes unreliable for `/v1/chat/completions`.

## Best Possible Solution

Keep this issue as the canonical tracker until OpenClaw has a documented split user/system install policy, status/doctor output that clearly reconciles CLI and live service install roots and versions, operator-safe handling for system-scoped and legacy units, and live or fixture-backed proof that a managed Linux Gateway can migrate to the intended newer install and reliably serve `/v1/chat/completions` with `openai-codex/gpt-5.5`.

## Reproduction Assessment

No. The report has credible host-state and rollback evidence, but current main does not include a high-confidence fixture or live run for the split user/system service repoint plus `/v1/chat/completions` timeout; the upgrade-survivor lane starts an updated foreground Gateway and checks health/status only.

## Solution Assessment

Unclear. The merged diagnostics, restart guidance, package-update checks, and Codex OAuth transport fixes are the right nearby surfaces, but the narrowest maintainable fix depends on which layer actually causes the managed Gateway timeout after repointing.
