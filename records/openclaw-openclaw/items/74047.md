## Summary

Keep open. Current main has useful gateway service-layout diagnostics and conservative doctor/update behavior, but it still does not define or prove a safe split user/system install migration, and the newer-install `/v1/chat/completions` timeout remains unisolated; related #70306 is also still open.

## What This Changes

The issue asks for a supported and reliable upgrade path when a Linux Gateway service remains pinned to an older system install while a newer user install is available.

## Best Possible Solution

Keep this as the canonical tracker until OpenClaw has a documented split user/system install policy, status/doctor output that reconciles CLI and live-service install roots and versions, operator-safe handling for legacy and system-scoped units, and live or package-acceptance proof that the intended newer install can serve `/v1/chat/completions` with `openai-codex/gpt-5.5` after migration.

## Reproduction Assessment

No. The report gives credible host-state and rollback evidence, but current main does not include a high-confidence fixture or live run that reproduces a pinned Linux Gateway service being migrated from an older system install to a newer user install and timing out on `/v1/chat/completions`.

## Solution Assessment

Unclear. The current diagnostics and restart guidance are the right nearby surfaces, but the best fix depends on whether the timeout comes from install-root drift, service environment/profile drift, systemd ownership, or provider/runtime behavior after migration.
