## Summary

Keep open. Current main has useful diagnostics for service entrypoints, package roots, and active systemd rewrite safety, but it still does not provide a proven split user/system install migration path or explain the reported newer-install `/v1/chat/completions` timeout; related PR #70306 remains open and PR #74920 was explicitly partial.

## What This Changes

Clarify and harden Linux Gateway upgrades where user and system installs diverge, including service path detection/migration and reliable `/v1/chat/completions` after promoting newer `openai-codex/gpt-5.5`.

## Best Possible Solution

Keep this as the canonical tracker until OpenClaw documents and validates split user/system install handling, reconciles CLI/status/doctor/update behavior for the live Gateway service root, and proves a managed Linux Gateway can migrate to the intended newer install while serving `openai-codex/gpt-5.5` through `/v1/chat/completions`.

## Reproduction Assessment

No. The report has credible rollback evidence, but I did not find a current-main fixture or live package/service run that repoints a legacy Linux Gateway service from an older system install to a newer user install and reproduces the timeout.

## Solution Assessment

Unclear. Current diagnostics and #70306 address parts of the operator visibility and legacy-unit problem, but the best fix depends on maintainer policy for split installs plus live Gateway/provider proof.
