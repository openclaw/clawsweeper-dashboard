## Summary

Keep open. Current main has strong adjacent diagnostics and package-update restart guardrails, but it still does not establish a supported split user/system install migration, does not auto-reconcile a running system-scoped Gateway unit, and does not reproduce or explain the reported `/v1/chat/completions` timeout after moving the service to the newer install.

## What This Changes

The issue asks OpenClaw to make split user/system Gateway upgrades understandable and reliable, including service-path warnings or migration guidance and safe promotion of `openai-codex/gpt-5.5` through `/v1/chat/completions`.

## Best Possible Solution

Use this as the canonical tracker until OpenClaw has an explicit split user/system install policy, status/doctor output that clearly reconciles CLI and live service install roots, operator-safe handling for system-scoped units, and live or fixture-backed proof that a managed Linux Gateway can move to the intended install and serve `/v1/chat/completions` reliably with the intended Codex model.

## Reproduction Assessment

No. The report gives credible manual host-state and rollback evidence, but current main does not include a high-confidence fixture or live lane that reproduces repointing a system/user split Gateway service and then exercising `/v1/chat/completions` with `openai-codex/gpt-5.5`.

## Solution Assessment

Unclear. The merged diagnostics and restart safeguards are the right adjacent surfaces, but the best fix depends on maintainer policy for split installs and on isolating whether the endpoint timeout is caused by service environment, install-root migration, or Codex transport behavior.
