## Summary

Keep open. Current main has useful adjacent diagnostics and restart guidance for gateway service layout drift, but it still does not define or prove a safe split user/system install migration for the reported managed Linux Gateway path, and the newer-install `/v1/chat/completions` timeout remains unisolated. Related merged PR #74920 is adjacent hardening, while open PR #70306 appears to cover only a narrower legacy-unit resolution slice.

## What This Changes

The issue asks OpenClaw to clarify or fix split user/system install upgrades so the live Gateway service tracks or warns about the intended install path and reliably serves `/v1/chat/completions` with `openai-codex/gpt-5.5` after migration.

## Best Possible Solution

Keep this as the canonical tracker until OpenClaw has a documented split user/system install policy, status/doctor output that reconciles CLI and live service install roots and versions, operator-safe handling for system-scoped and legacy units, and package-acceptance or live proof that a managed Linux Gateway can migrate to the intended newer install and serve `/v1/chat/completions` with `openai-codex/gpt-5.5`.

## Reproduction Assessment

No. The report has credible host-state and rollback evidence, but current main does not include a high-confidence fixture or live run for the split user/system service repoint plus `/v1/chat/completions` timeout.

## Solution Assessment

Unclear. The diagnostics and restart guidance on main are the right nearby surfaces, but the best fix depends on which layer causes the managed Gateway timeout after repointing and on maintainer policy for system-scoped service migration.
