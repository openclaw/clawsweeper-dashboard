## Summary

Keep open. Current main has useful adjacent diagnostics for Gateway service layout drift, update restart guidance, and OpenAI-compatible HTTP handling, but it still does not define or prove a safe split user/system install migration for the reported Linux Gateway path, and the newer-install `/v1/chat/completions` timeout remains unisolated.

## What This Changes

The issue asks for a supported upgrade/migration path when user and system OpenClaw installs diverge, plus clearer Gateway service warnings and reliable `openai-codex/gpt-5.5` promotion through `/v1/chat/completions`.

## Best Possible Solution

Keep this as the canonical tracker until OpenClaw has a documented split user/system install policy, status/doctor output that reconciles CLI and live service install roots and versions, operator-safe handling for system-scoped and legacy units, and live or package-acceptance proof that a managed Linux Gateway can migrate to the intended newer install and serve `/v1/chat/completions` with `openai-codex/gpt-5.5`.

## Reproduction Assessment

No. The report has credible host-state and rollback evidence, but current main does not include a high-confidence fixture or live run for repointing a pinned Linux Gateway service from an older system install to a newer user install and reproducing the `/v1/chat/completions` timeout.

## Solution Assessment

Unclear. The current diagnostics and restart guidance are the right nearby surfaces, but the best fix depends on whether the timeout comes from install-root drift, service environment/profile drift, systemd privilege handling, or provider/runtime behavior after migration.
