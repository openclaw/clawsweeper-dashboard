Current main still publishes the reported default piped installer path and still has an interactive onboarding handoff that can inherit the installer script stream; the linked implementation candidate is open and unmerged, so the issue should remain open pending that review or a replacement fix.

## What This Changes

Report that the documented `curl ... | bash` installer can hang on WSL2 by letting interactive setup consume the script stream and truncate `warn_shell_path_missing_dir`.

## Best Possible Solution

Make `install.sh` itself pipe-safe for interactive child commands, then align the public installer examples and add focused installer smoke coverage for the default piped path.

## Reproduction Assessment

No, not at high confidence from this read-only pass. The reporter gave concrete WSL2 steps and current main preserves a plausible stdin-inheritance path, but I did not run the mutating installer to reproduce the exact hang and truncation.

## Solution Assessment

No. A docs-only `bash -s --` change is incomplete while installer code can still pass the script stream to interactive onboarding; the safer fix is to route interactive handoffs through `/dev/tty` or a noninteractive path and then update docs.
