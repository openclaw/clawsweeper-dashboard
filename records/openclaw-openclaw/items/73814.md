## Summary

Keep open. Current main still publishes the reported bare `curl | bash` installer command, and `scripts/install.sh` still has an onboarding handoff that can inherit piped script stdin. Open PR #73830 references this issue with closing syntax, but it is still open and its docs-only direction does not fully address the installer-level stdin handling.

## What This Changes

The issue reports that the recommended `curl ... | bash` installer invocation can hang on WSL2 and leave a truncated `warn_shell_path_missing_di` command when interactive installer or onboarding reads consume the script stream.

## Best Possible Solution

Make `scripts/install.sh` robust under piped execution by ensuring every interactive prompt and onboarding handoff reads from `/dev/tty` or is skipped in noninteractive mode, then update the public install examples and add focused coverage for the piped interactive path. Keep the separate Node auto-installer report in #73837 and the finalization-probe hang in #62509 as adjacent but distinct follow-ups.

## Reproduction Assessment

Unclear. The issue provides concrete WSL2 steps and current source still shows a plausible stdin-consumption path, but this read-only review did not run the mutating installer end to end.

## Solution Assessment

No for the currently linked docs-only path. Changing examples to `bash -s --` may help invocation consistency, but the maintainable fix should harden installer child reads such as `openclaw onboard` against inherited piped stdin.
