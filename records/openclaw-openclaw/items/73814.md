## Summary

Keep open. Current main and the latest release still publish the reported bare `curl ... | bash` install command, and `install.sh` still has an onboarding handoff that checks for a TTY but invokes `openclaw onboard` without redirecting stdin from `/dev/tty`. The open linked PR #73830 is a closing candidate rather than a merged fix, so this issue should stay open for maintainer review of that PR or a replacement installer-level fix.

## What This Changes

The issue asks for the `install.sh` curl-pipe installer path to stop hanging or truncating script input when interactive prompts or onboarding read from stdin.

## Best Possible Solution

Make `install.sh` robust under piped execution by ensuring every interactive prompt and onboarding subprocess reads from `/dev/tty` or is skipped in noninteractive mode, then update the public install docs to the supported invocation and add focused coverage for the default piped installer path.

## Reproduction Assessment

Unclear. The issue gives concrete WSL2 steps and a minimal piped-bash check reproduces the underlying stdin-consumption mechanism, but this read-only review did not run the full mutating installer path.

## Solution Assessment

No. A docs-only `bash -s --` change is not the best complete fix because child reads can still inherit the script pipe; the maintainable fix belongs in installer stdin handling, with docs updated afterward.
