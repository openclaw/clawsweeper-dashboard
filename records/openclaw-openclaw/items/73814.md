## Summary

Keep open. Current main still publishes the affected piped install command, still has an installer onboarding handoff that can inherit the script stream, and open PR #73830 is already linked with closing syntax as the active fix candidate.

## What This Changes

The issue reports that the documented `curl ... | bash` installer path can hang under WSL2 because interactive installer work may read from the script stream and corrupt the remaining shell input.

## Best Possible Solution

Make the installer robust under piped execution by ensuring every interactive prompt and onboarding/doctor subprocess reads from `/dev/tty` or is run noninteractively with `/dev/null`, then align all public install snippets and add a focused installer regression for piped stdin behavior.

## Reproduction Assessment

Unclear. The issue gives concrete WSL2 steps and a minimal `bash -s --` pipe check reproduces the stdin-consumption mechanism, but this read-only review did not run the full mutating installer end to end.

## Solution Assessment

No, a docs-only `bash -s --` change is not the best complete fix. The safer maintainable fix is installer-level stdin isolation for interactive child processes, with docs updated to match the supported invocation.
