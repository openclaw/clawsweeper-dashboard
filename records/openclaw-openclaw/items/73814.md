## Summary

Keep open. Current main still publishes the reported piped installer command, the installer still has an onboarding path that checks for a TTY but runs `openclaw onboard` without redirecting stdin from `/dev/tty`, and open PR #73830 is still the linked closing PR candidate rather than a merged fix.

## What This Changes

The issue reports that the documented `curl ... install.sh | bash` installer path can hang on WSL2 and mangle `warn_shell_path_missing_dir` when interactive reads consume the script stream.

## Best Possible Solution

Make `install.sh` robust under piped execution by ensuring every interactive prompt and onboarding handoff reads from `/dev/tty` or is skipped in noninteractive mode, then update the public install docs and add a focused regression for the piped interactive path.

## Reproduction Assessment

Unclear. The issue provides concrete WSL2 steps and the stdin-consumption mechanism is demonstrable with a minimal piped bash command, but this read-only pass did not run the full mutating installer path.

## Solution Assessment

No, a docs-only `bash -s --` change is not the best complete fix. The safer maintainable fix is installer-level TTY isolation for prompt and onboarding subprocesses, with docs reflecting the supported invocation afterward.
