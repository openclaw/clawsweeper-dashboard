## Summary

Keep open. Current main still exposes the reported failure shape: the public install docs continue to recommend bare piped `bash`, and the bootstrap onboarding path can still run `openclaw onboard` with the installer script stream as inherited stdin. The linked PR #73830 remains open and unmerged, so this issue should stay open until that work is landed, revised, or replaced.

## What This Changes

Make the `install.sh` piped installer safe for interactive execution so prompts and onboarding cannot consume the incoming script stream, and align the public install docs with the supported invocation.

## Best Possible Solution

Make `scripts/install.sh` robust under piped execution by ensuring every interactive prompt and onboarding handoff reads from `/dev/tty` or is skipped in noninteractive mode, then update the public install examples and add focused coverage for the piped interactive path.

## Reproduction Assessment

Unclear. The issue has concrete WSL2 steps and current source still has a matching stdin-inheritance path, but this read-only review did not run the full mutating installer; existing smoke coverage uses `--no-prompt` and does not reproduce the default interactive path.

## Solution Assessment

No. A docs-only `bash -s --` change, as proposed in PR #73830, is not the best complete fix because child reads can still inherit the script stream; the narrow maintainable fix belongs in installer TTY handling first, with docs aligned afterward.
