## Summary

Keep open. Current main still publishes the reported bare `curl | bash` install command in first-run docs and installer examples, and the installer still has a bootstrap onboarding path that checks for `/dev/tty` but then runs `openclaw onboard` without redirecting stdin from it. The discussion already produced open PR #73830, but that PR is still unmerged and its docs-only `bash -s --` direction does not address the remaining installer-level stdin inheritance shown by the current code.

## What This Changes

The issue requests that the macOS/Linux installer work reliably when invoked through the documented `curl -fsSL https://openclaw.ai/install.sh | bash` pipeline instead of hanging or consuming its own script stream during interactive prompts.

## Best Possible Solution

Make `scripts/install.sh` robust under piped execution by ensuring every interactive prompt and onboarding handoff reads from `/dev/tty` or is skipped in noninteractive mode, then update all public install examples to the supported invocation and add focused coverage for the piped interactive path.

## Reproduction Assessment

Unclear. The issue gives concrete WSL2 steps and a local minimal piped-bash check confirms the stdin-consumption mechanism, but this read-only review did not run the full mutating installer path.

## Solution Assessment

No. A docs-only `bash -s --` change is not the best complete fix because child reads can still consume piped script stdin; the maintainable fix belongs in the installer’s TTY handling first, with docs aligned afterward.
