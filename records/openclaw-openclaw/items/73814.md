## Summary

Keep open. Current main still documents the reported default `curl ... | bash` installer path, the installer still has at least one onboarding subprocess path that can inherit the script pipe, and the open linked PR with closing syntax is not merged and has review feedback that its docs-only `bash -s --` change is not a complete fix.

## What This Changes

The issue reports that the documented piped installer command can hang and corrupt the incoming shell script stream on WSL2 when interactive installer prompts read from stdin.

## Best Possible Solution

Make the default installer path pipe-safe by routing all interactive child handoffs through `/dev/tty` or a tested noninteractive mode, align public installer examples/help, and add focused installer smoke coverage for the default piped path.

## Reproduction Assessment

No, not as a high-confidence live reproduction. The reporter provided concrete WSL2 logs and current source still shows a plausible stdin-inheritance path, but this read-only pass did not run the mutating installer end to end.

## Solution Assessment

No, the currently linked `bash -s --` docs-only proposal is not the best complete fix. The safer solution is installer-side `/dev/tty` handling or explicit noninteractive behavior, followed by consistent docs and smoke proof.
