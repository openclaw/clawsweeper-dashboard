## Summary

Keep open. Current main still publishes the reported piped installer command in user-facing docs, the installer still has an onboarding path that can inherit the script stream instead of `/dev/tty`, and the linked closing PR #73830 remains open rather than merged.

## What This Changes

The issue reports that running the documented `curl -fsSL https://openclaw.ai/install.sh | bash` installer can hang on WSL2 Ubuntu and later execute a truncated `warn_shell_path_missing_dir` function name after stdin is consumed by interactive installer behavior.

## Best Possible Solution

Make `install.sh` robust for piped execution by redirecting every interactive prompt and onboarding handoff to `/dev/tty` or explicitly using noninteractive stdin, then update all public install examples to the supported invocation and add focused coverage for the piped interactive path.

## Reproduction Assessment

Unclear. The issue has concrete WSL2 steps and the stdin-consumption mechanism is reproducible with a minimal piped bash command, but this read-only review did not run the mutating installer end to end.

## Solution Assessment

No. The open docs-only PR direction is not enough by itself; the narrow maintainable fix should be installer-level TTY isolation, with documentation adjusted after the behavior is safe.
