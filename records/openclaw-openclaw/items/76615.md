## Summary

Keep open. The PR fixes a real source-reproducible credential echo path and addresses the earlier default/session/changelog feedback, but the latest macOS CLI no-echo implementation uses `getpass(3)`, which can silently truncate longer credential inputs.

## What This Changes

This PR adds sensitive wizard text prompts, masks Clack and macOS session CLI credential input, marks onboarding credential call sites sensitive, preserves existing gateway credentials via confirm prompts, adds tests, and updates `CHANGELOG.md`.

## Best Possible Solution

Land the masking change with a macOS CLI no-echo reader that preserves full credential length, keeps the TypeScript sensitive prompt plumbing and existing-secret confirmations, and retains focused regression coverage plus the changelog entry.

## Reproduction Assessment

Yes. Source inspection on current main shows onboarding credential prompts call `prompter.text()`, and the Clack adapter renders that through normal `text()` input, so credential entry can echo today.

## Solution Assessment

No. The PR direction is right, but using `getpass(3)` for the macOS CLI sensitive branch is not the best mergeable solution because it can truncate inputs beyond the BSD 128-character limit; use a termios/no-echo reader or `readpassphrase` path with an explicit buffer instead.
