## Summary

Keep this PR open: the dependency-runtime scan direction is useful, but the diff has a blocking managed-npm scope regression and the external PR does not include after-fix real behavior proof.

## What This Changes

Adds post-install plugin package code scanning for installed runtime trees, including `node_modules` and hidden directories, plus a configurable scan cap and regression tests.

## Best Possible Solution

Keep the post-install runtime-code scan, but scope managed-npm validation to the candidate package and its newly materialized dependency closure or a baseline-safe equivalent so prior trusted or forced installs do not block unrelated installs.

## Reproduction Assessment

No live run was performed, but the behavior is source-reproducible: current main skips `node_modules` in the scanner and the PR would run the new scan against `npmRoot` for managed npm installs. A focused real repro would seed a trusted or forced dangerous plugin, then install a separate safe npm plugin without trust or force flags.

## Solution Assessment

No. The archive-install part is directionally right, but reusing `params.packageDir` for npm-spec installs is too broad because that value can be the shared managed npm root rather than the candidate runtime tree.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Limit installed scans to the candidate runtime tree:** `src/plugins/install-security-scan.runtime.ts:981`
  - body: For managed npm installs the caller passes `dependencyScanRootDir: npmRoot`, so this new scan walks the shared npm root, not just the package being installed. If an existing plugin was previously accepted with `--dangerously-force-unsafe-install` or trusted official provenance, its critical patterns are now rechecked under the next unrelated install's flags and can block that safe install. Scope this scan to the candidate package plus its materialized dependencies, or baseline existing findings.
  - confidence: 0.88
