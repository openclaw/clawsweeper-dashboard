## Summary

Keep open. Current main still has provider onboarding helpers and API-key auth paths that can set `agents.defaults.model.primary` unconditionally; the allowlist merge/clobber part appears already guarded in current code, but the requested primary/fallback protection is still missing.

## What This Changes

The issue requests that auto-onboard/provider preset configuration add provider models without overwriting an existing user-selected default model or fallback chain.

## Best Possible Solution

Provider-owned setup should remain additive by default: register provider models and aliases, preserve unrelated allowlist entries, and only fill `agents.defaults.model.primary` or fallbacks when those fields are absent. Explicit user actions such as `openclaw models set` or `models auth login --set-default` should continue to replace the default intentionally.

## Reproduction Assessment

Yes. A focused helper-level reproduction is to call `applyProviderConfigWithDefaultModelPreset` or a provider `applyConfig` with an existing `agents.defaults.model.primary`; current code routes through `applyAgentDefaultModelPrimary` and replaces it.

## Solution Assessment

No current fix is present. The narrow maintainable fix is to add a preserve-existing-primary path to provider preset/auth config application while leaving explicit set-default commands as the intentional override path.
