## Summary

Keep open. Current main still has the reported overwrite behavior in the shared provider preset path and in OpenAI’s provider-default helper; this is a real, narrow config-preservation bug rather than cleanup-closable stale work. The current shared helper does merge existing `agents.defaults.models`, so the custom-model stripping part needs targeted regression coverage, but the primary overwrite is directly visible in source and tests.

## What This Changes

The issue requests that auto-onboard/provider preset application merge new provider models without replacing an explicit `agents.defaults.model.primary` or fallback configuration.

## Best Possible Solution

Preserve explicit user model choices by making provider preset/default appliers additive for provider models and aliases, only setting `agents.defaults.model.primary` when no primary is configured or it is empty, while leaving explicit default-setting commands as the opt-in way to replace the primary.

## Reproduction Assessment

Yes. A source-level reproduction is clear: call `applyProviderConfigWithDefaultModelPreset` or `applyOpenAIConfig` with an existing `agents.defaults.model.primary`, and current code routes through `applyAgentDefaultModelPrimary`, which writes a new primary; the OpenAI test already asserts that overwrite.

## Solution Assessment

Yes, the narrow maintainable fix is in the shared provider-onboard/default-model helper path plus the OpenAI bypass, with tests proving existing primaries and custom allowlist entries survive preset application. A broader userOverrides framework is not needed before fixing this concrete clobber.
