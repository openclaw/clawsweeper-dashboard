## Summary

Keep open: current main still has provider preset paths that can replace an existing default primary, and the linked closing PR is open, unmerged, and conflicting.

## What This Changes

The issue reports provider auto-onboard and preset helpers replacing a user-set `agents.defaults.model.primary` and dropping custom model entries when new provider plugins are registered.

## Best Possible Solution

Repair, rebase, and land https://github.com/openclaw/openclaw/pull/75748, or an equivalent narrow fix, so provider presets add models without changing defaults unless the user takes an explicit default-setting path.

## Reproduction Assessment

Yes, at source level. Passing a config with an existing `agents.defaults.model.primary` through `applyProviderConfigWithDefaultModelPreset`, `applyOpenAIConfig`, or `applyMoonshotConfig` still reaches `applyAgentDefaultModelPrimary`; the reporter also confirmed the production update path.

## Solution Assessment

Yes. The narrow maintainable fix is to preserve existing primary and fallbacks during non-explicit provider preset/setup paths while keeping `openclaw models set` and `models auth login --set-default` authoritative.
