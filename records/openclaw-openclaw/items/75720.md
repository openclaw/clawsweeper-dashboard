## Summary

Keep open. Current main still has provider onboarding and provider API-key auth paths that can replace `agents.defaults.model.primary` during preset/config application; the default model map merge is guarded now, so the remaining actionable bug is preserving explicit primary/fallback selections unless the user intentionally asks to change them.

## What This Changes

Provider auto-onboarding and plugin default-model presets should add provider-owned model registrations without silently overwriting an explicitly configured `agents.defaults.model.primary` or fallback chain.

## Best Possible Solution

Provider-owned setup should stay additive by default: register provider models and aliases, preserve unrelated model entries, and only fill `agents.defaults.model.primary` or fallbacks when those fields are absent. Explicit user actions such as `openclaw models set`, `models auth login --set-default`, and legacy default migrations should remain intentional override paths.

## Reproduction Assessment

Yes. A focused source-level reproduction is to call `applyProviderConfigWithDefaultModelPreset` or a bundled provider `applyConfig` with an existing `agents.defaults.model.primary`; current main routes through `applyAgentDefaultModelPrimary` and replaces it.

## Solution Assessment

Yes, the best fix is a narrow helper-level change in provider onboarding/auth default application, with tests proving provider presets fill empty defaults but preserve explicit user defaults. It should not be solved by disabling bundled providers or changing static catalog discovery as the primary repair.
