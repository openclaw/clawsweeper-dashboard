## Summary

Keep open. The corrected report narrows this to a QA harness/mock-provider fidelity problem, and current main still shows the mock can plan `read` from prompt text while Codex app-server intentionally excludes `read` from OpenClaw dynamic-tool exposure; no merged fix or canonical superseding item resolves it yet.

## What This Changes

The issue asks qa-lab to stop treating a mock-provider planned `read` as Codex runtime evidence when the Codex app-server lane does not declare `read` as a dynamic tool.

## Best Possible Solution

Keep Codex workspace tools native while making qa-lab mock/provider-plan evidence compare only declared dynamic tools or explicitly model the native Codex read protocol.

## Reproduction Assessment

No, not as a high-confidence current-main runtime reproduction. Current main lacks `--runtime-pair`, but source inspection shows the mock provider can emit `read` from prompt text while Codex filters `read` out of dynamic tool exposure.

## Solution Assessment

Unclear. Gating mock `read` on declared tools is probably the narrow harness-fidelity fix, but maintainers should choose whether this scenario should instead model Codex-native read behavior before an automated repair changes semantics.
