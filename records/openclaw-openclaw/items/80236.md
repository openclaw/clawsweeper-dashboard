## Summary

Keep open. The updated report narrows this to a QA harness/mock-provider fidelity issue rather than a proven user-facing Codex app-server regression, and current main still shows the mock can plan `read` while Codex app-server intentionally excludes `read` from OpenClaw dynamic-tool exposure. No close reason applies because the behavior is not implemented on main or clearly superseded by a merged fix.

## What This Changes

The issue asks qa-lab to stop treating a mock-provider `read` plan as Codex app-server runtime evidence when `read` is not declared as an OpenClaw dynamic tool.

## Best Possible Solution

Make qa-lab mock-provider planning respect the effective declared tool list, or add explicit Codex-native read modeling for this scenario, while keeping Codex workspace tools out of OpenClaw dynamic-tool exposure.

## Reproduction Assessment

No. The report includes branch-level proof, and current main source makes the mock-provider read plan versus Codex-native tool boundary clear, but current main does not contain the `--runtime-pair` CLI needed for the exact Pi-vs-Codex reproduction.

## Solution Assessment

Unclear. Gating mock `read` on declared tools is likely the narrow harness-fidelity fix, but the issue also names Codex-native protocol modeling as an alternative, so maintainers should choose the boundary before implementation.
