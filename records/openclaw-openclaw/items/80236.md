## Summary

Keep open: current main still has the Codex-native workspace-tool contract and the mock QA provider can still plan direct `read` calls from approval prompts, so this remains a QA harness fidelity/native-proof gap rather than a closeable implemented fix.

## What This Changes

The issue asks the QA runtime-parity harness to stop treating mock-provider `read` planning as Codex app-server runtime evidence, by gating mock `read` on declared tools or modeling native Codex approval reads.

## Best Possible Solution

Keep the Codex app-server native workspace-tool contract intact, then either make the mock provider avoid undeclared `read` plans for Codex lanes or add a native/live approval-read fixture that proves the real Codex path.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Partly: source inspection shows the mock/Codex contract mismatch, and the issue includes fresh focused runtime-pair output from active PR heads. I do not have a high-confidence current-main end-to-end repro because the runtime-pair harness is not merged on current main.

## Solution Assessment

Is this the best way to solve the issue? Yes for the corrected direction: exposing `read` as an OpenClaw dynamic tool in the Codex lane would conflict with current tests and app-server ownership, so the narrow maintainable path is mock declared-tool gating or a native Codex approval-read proof fixture.
