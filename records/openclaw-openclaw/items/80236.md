This is a recent, concrete Codex runtime parity bug: the issue discussion and current source still line up with a post-approval `read` being planned while the Codex app-server path keeps `read` behind its native workspace-tool boundary, and the linked harness work is diagnostic rather than a fix.

## Solution Assessment

Unclear. The best fix should be boundary-preserving in Codex app-server/native-tool handling or the private-QA mock contract; blindly exposing all workspace tools as dynamic tools would fight the current contract.
