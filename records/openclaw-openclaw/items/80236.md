## Summary

Keep open. The report remains a concrete Codex runtime parity bug: current source still excludes `read` from OpenClaw dynamic tool exposure for Codex app-server while the QA scenario and mock provider expect a post-approval `read`, and the linked PRs are diagnostic or harness work rather than fixes.

## What This Changes

The issue asks OpenClaw to make Codex app-server approval followthrough return a real/equivalent `read` result instead of an unsupported-read result that can be synthesized as success.

## Best Possible Solution

Preserve Codex-native workspace-tool ownership, but make post-approval `read` return real file content or a structured failure that cannot be summarized as a successful review.

## Reproduction Assessment

No high-confidence current-main reproduction was run. The linked harness proof plus source inspection make the path source-reproducible: the scenario asks for `read`, the mock provider plans `read`, and Codex filters `read` out of OpenClaw dynamic tools as app-server-owned.

## Solution Assessment

Unclear. The best fix should be boundary-preserving in Codex app-server/native-tool handling or the private-QA mock contract; blindly exposing workspace tools as dynamic tools would fight the current contract.
