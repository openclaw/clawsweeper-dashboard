## Label Justifications

- `P1`: The PR touches a user-facing cron alert path and, as written, can unexpectedly enable failure-alert messages after reload.
- `merge-risk: 🚨 compatibility`: The diff changes the persisted meaning of existing all-null cron failure-alert rows, affecting stored job settings on upgrade or restart.
- `merge-risk: 🚨 message-delivery`: The unsafe decode would cause unconfigured jobs to request failure-alert messages that users did not enable.

## Summary

Keep this PR open but blocked: the reported cron failure-alert round-trip bug is real, but this patch decodes every all-null alert row as enabled and would turn jobs with no per-job alert config into default-alert jobs after reload. A stronger same-root PR is already open with the sentinel guard and tests, so maintainers should choose that path or require this branch to match it.

## What This Changes

This PR changes the cron failure-alert SQLite decoder to return an empty enabled alert config for all-null optional alert columns.

## Best Possible Solution

Preserve all three stored states by decoding `{}` only when `failure_alert_disabled === 0`, keep `null` as omitted config, add focused round-trip tests, and either update this branch or use https://github.com/openclaw/openclaw/pull/96601 as the canonical fix path.

## Reproduction Assessment

Yes, at source level: `bindFailureAlertColumns(undefined)` and `bindFailureAlertColumns({})` both write null optional alert columns, and this PR's unconditional `return {}` makes the omitted row decode as enabled. I did not run tests because this review is read-only.

## Solution Assessment

No. The best fix is the narrower sentinel check, `failure_alert_disabled === 0`, so defaults-only alerts survive without changing the meaning of rows where failure alerts were omitted.
