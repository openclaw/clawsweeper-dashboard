## Label Justifications

- `P1`: The PR touches a user-facing cron alert path and, as written, can unexpectedly enable failure-alert messages after reload.
- `merge-risk: 🚨 compatibility`: The diff changes the persisted meaning of existing all-null cron failure-alert rows, affecting upgrades and stored job settings.
- `merge-risk: 🚨 message-delivery`: The unsafe decode would cause unconfigured jobs to request failure-alert messages that users did not enable.

## Summary

Keep open; the PR targets a real cron persistence bug, but the patch is unsafe because it turns every all-null failure-alert row, including the existing no-config state, into an enabled defaults-only alert.

## What This Changes

The PR changes the cron failure-alert SQLite decoder to return an empty alert config when all optional alert columns are null.

## Best Possible Solution

Preserve all three stored states by decoding `{}` only when `failure_alert_disabled === 0`, keep `null` as no config, and cover disabled, undefined, defaults-only, and populated round trips in tests.

## Reproduction Assessment

Yes at source level: `bindFailureAlertColumns(undefined)` and `bindFailureAlertColumns({})` both produce null option columns, and the PR's unconditional `return {}` would make the no-config row decode as enabled. I did not run tests because this review is read-only.

## Solution Assessment

No. The best fix is the narrower sentinel check, `failure_alert_disabled === 0`, so defaults-only alerts survive without changing the meaning of rows where failure alerts were omitted.
