## Summary

Keep open. Current main still has the reported Unix/macOS behavior: listener PIDs from `lsof -Fpc` are filtered by the `c` command token before any full-argv verification, so a Node-backed gateway reported as `cnode` is dropped. The issue is also paired with open, unmerged PR #70681, which uses closing syntax and should remain the implementation path until it lands or is replaced.

## What This Changes

The issue requests that macOS/Unix stale-gateway PID discovery stop trusting `lsof`'s command token and instead verify listening PIDs using the process argv before cleanup or port-free polling.

## Best Possible Solution

Land PR #70681 or an equivalent narrow fix that treats `lsof` as a raw listener-PID source, verifies Unix/macOS candidates with full argv inspection before cleanup and port-free polling, preserves self and ancestor exclusions, and updates regression coverage for `cnode` gateway listeners and unrelated non-gateway listeners.

## Reproduction Assessment

Yes. The issue provides a concrete macOS reproduction, and current-main inspection gives a deterministic unit-level reproduction: feed `lsof -Fpc` output containing a real gateway PID with `cnode`, and the parser drops that PID before argv verification can inspect it.

## Solution Assessment

Yes. The proposed direction is the narrowest maintainable fix because it mirrors the Windows argv-verification path and avoids using `lsof`'s kernel command field as a gateway identity signal for Node-based processes.
