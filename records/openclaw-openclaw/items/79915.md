## Summary

Keep open: current main still lacks a diagnostic startup-grace guard, and an open PR now specifically references this issue with closing syntax, so the issue should remain open until that PR is reviewed and either merged or replaced.

## What This Changes

The issue asks for diagnostic liveness warning emission to be suppressed during cold start while keeping liveness sampling active for baseline resets.

## Best Possible Solution

Review and land the open startup-grace PR, or replace it with an equivalent narrow fix that suppresses warning emission during startup while preserving sampling and regression coverage.

## Reproduction Assessment

Yes, from source inspection: current main starts diagnostic liveness sampling at gateway startup and has no grace check before warning emission. A focused fake-timer test that injects an early liveness sample with queued work can reproduce the warning path without the reporter's Windows host.

## Solution Assessment

Yes. The narrowest maintainable solution is to add an internal startup grace window to diagnostic warning emission while keeping sampling active; the optional OPENCLAW_STARTUP_GRACE_MS knob should remain a separate follow-up unless maintainers explicitly want it now.
