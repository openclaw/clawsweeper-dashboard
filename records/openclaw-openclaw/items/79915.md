## Summary

Keep open: current main still starts diagnostic liveness at gateway startup without a startup-grace guard, so queued/open diagnostic work can still emit a WARN-level liveness warning during cold-start blocking. The related startup-grace PRs were closed unmerged, and the open threshold-tuning PR does not cover this missing grace window.

## What This Changes

The issue requests a diagnostic heartbeat startup grace window that suppresses liveness warning/event emission during cold start while continuing to sample liveness metrics.

## Best Possible Solution

Land a localized diagnostics change that adds an internal startup-grace boundary and leaves any user-configurable grace knob for a separate config decision.

## Reproduction Assessment

Yes, from source inspection: current main starts diagnostic liveness at gateway startup with no grace check, and queued diagnostic work plus a non-null liveness sample reaches the WARN path. A focused fake-timer test using `logMessageQueued` during the first heartbeat window should reproduce it without the reporter's Windows setup.

## Solution Assessment

Yes. The narrow maintainable fix is to add internal `startupGraceMs`, always run the liveness sampler for resets/baselines, and suppress only liveness event/WARN emission during grace; the optional env/config knob should be separate.
