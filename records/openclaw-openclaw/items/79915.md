## Summary

Keep open: current main still starts diagnostic liveness sampling during Gateway startup without a startup-grace guard, and the linked open fixing PR has not merged yet.

## What This Changes

Requests a diagnostic startup grace window so cold-start event-loop liveness samples do not emit false WARN logs before startup settles.

## Best Possible Solution

Land a corrected diagnostic startup-grace change that samples every heartbeat but suppresses liveness event/WARN emission during the grace window; keep any env/config knob as a separate product follow-up.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: start diagnostics, queue or keep open diagnostic work before the first heartbeat, return a non-null event-loop-delay sample, and current main reaches the warning path with no startup-grace check. I did not execute a live Windows cold-start run.

## Solution Assessment

Yes, the issue's core fix direction is the narrow maintainable one: add an internal startup-grace emission gate while keeping the sampler active for resets and baselines. The optional configurable grace window should stay separate unless maintainers want a new config surface.
