## Summary

Keep open: current main still has no diagnostic liveness startup-grace window, and the same gap is present in v2026.5.7. The issue is source-reproducible with queued/waiting diagnostic work during the first heartbeat window, while the two specific grace-window PRs that attempted the fix are both closed unmerged.

## What This Changes

Request adding a startup-grace window to the diagnostic liveness heartbeat so cold-start event-loop delay does not emit false WARN-level liveness messages while sampling continues.

## Best Possible Solution

Add an internal 60s diagnostic startup-grace boundary that keeps sampler resets running but suppresses liveness event and WARN emission until grace expires; leave a tunable env/config knob for a separate product decision.

## Reproduction Assessment

Yes, source-reproducible: with diagnostics enabled, queue diagnostic work before the first heartbeat tick and return an event-loop-delay liveness sample; current main has no grace check before WARN emission. I did not run a live Windows cold-start because this review is read-only.

## Solution Assessment

Yes; the narrowest maintainable fix is the prior PR shape: add an internal `startupGraceMs`, keep sampling for metric resets, and discard liveness reports only while startup grace is active. The optional env/config knob is a follow-up, not part of the bug fix.
