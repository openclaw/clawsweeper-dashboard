## Summary

Keep open: current main still has no diagnostic startup-grace guard, and the active/queued or sustained-delay warning path can still emit the cold-start WARN this issue reports. The related fix PRs were closed unmerged, and the merged active-only liveness-noise work does not cover queued work or sustained P99 delay during startup.

## What This Changes

The issue requests a diagnostic startup grace window so cold-start event-loop blocking does not emit false liveness warnings.

## Best Possible Solution

Add an internal diagnostic startup-grace boundary that continues running the liveness sampler every tick for metric resets/baselines, but suppresses liveness event and WARN emission until the grace window expires; leave the optional env/config knob as a separate product decision if needed.

## Reproduction Assessment

Yes, from source inspection: current main starts diagnostic liveness during gateway startup and has no grace check before warning/event emission. A focused fake-timer test with queued work and a non-null liveness sample during the first heartbeat window should reproduce the warning path without the reporter's Windows setup.

## Solution Assessment

Yes, the best fix is the narrow prior-PR shape: add an internal `startupGraceMs`, always run the sampler, and gate only liveness event/WARN emission during grace. The requested `OPENCLAW_STARTUP_GRACE_MS` knob is useful only as a later configurable enhancement, not required for the bug fix.
