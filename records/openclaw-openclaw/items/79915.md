## Summary

Keep open: current main still starts diagnostic liveness sampling at Gateway startup without a startup-grace emission guard, and the two focused startup-grace PRs were closed unmerged. The issue is narrow and actionable because the warning path is source-reproducible and the prior PR discussion already identified the important boundary: keep sampling during grace, suppress only event/WARN emission.

## What This Changes

The issue asks for a diagnostic heartbeat startup-grace window that suppresses false cold-start liveness warnings while preserving sampler resets and baselines.

## Best Possible Solution

Add an internal 60-second diagnostic startup grace, pass it from Gateway startup, keep the liveness sampler running every tick, and suppress only liveness event/WARN emission until the grace window expires.

## Reproduction Assessment

Yes, at source level: current main starts diagnostic liveness during Gateway startup, has no startup-grace check, and queued work plus a non-null liveness sample reaches the warning path. I did not run a live Windows cold-start reproduction in this read-only pass.

## Solution Assessment

Yes for the narrow fix direction: the maintainable solution is the prior PR shape with an internal grace window that gates only emission while preserving sampler resets. The optional env/config knob should remain a separate follow-up unless maintainers explicitly want that public surface now.
