## Summary

Keep open: current main still starts diagnostic liveness sampling during gateway startup without a startup-grace gate, and both prior fix PRs are closed unmerged. The narrow fix path is well understood from source inspection and the earlier PR discussion.

## What This Changes

The issue asks OpenClaw to suppress diagnostic liveness event/WARN emission during an initial startup-grace window while keeping the sampler running for metric resets, with an optional future tuning knob.

## Best Possible Solution

Add an internal 60-second diagnostic startup-grace boundary from gateway startup that samples every heartbeat but suppresses liveness event/WARN emission until grace expires.

## Reproduction Assessment

Yes, source-reproducible: current main starts the diagnostic heartbeat during gateway startup, has no grace check, and queued/waiting work plus a non-null liveness sample reaches the WARN path. A focused fake-timer diagnostic test can reproduce this without the reporter's Windows setup.

## Solution Assessment

Yes. The best fix is the prior corrected PR shape: add an internal `startupGraceMs`, always run the sampler for resets and baselines, and suppress only liveness event/WARN emission during grace; the env/config knob is a separate product/config follow-up.

Add an internal `startupGraceMs` option to `StartDiagnosticHeartbeatOptions`, pass `60_000` from `startGatewayServer` when diagnostics are enabled, and keep `sampleLiveness`/`sampleDiagnosticLiveness` running on every heartbeat so event-loop-delay metrics and CPU/ELU baselines reset during grace. Add a user-facing changelog fix entry because this changes gateway diagnostic warning behavior.
