## Summary

Keep open: current main still lacks a diagnostic startup-grace guard, and an open PR already references this issue with closing syntax, so the issue should remain open until that PR merges or is closed.

## What This Changes

Suppress diagnostic liveness warning/event emission during gateway startup grace while keeping liveness sampling active.

## Best Possible Solution

Review and land the open narrow diagnostic heartbeat grace PR, or close that PR and replace it with an equivalent fix before closing this issue.

## Reproduction Assessment

Yes. Current-main source shows gateway startup begins diagnostic liveness sampling without a grace check, and a focused fake-timer test with queued diagnostic work before the first heartbeat can exercise the warning path.

## Solution Assessment

Yes. The best fix is to add an internal startup-grace boundary that keeps the sampler running for resets and baselines while suppressing liveness event/WARN emission until grace expires; a public env/config knob is a separate enhancement.
