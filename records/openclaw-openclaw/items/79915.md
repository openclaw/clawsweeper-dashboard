## Summary

Keep open: current main still has no diagnostic startup-grace gate, so gateway startup can begin liveness sampling immediately and queued/open diagnostic work can still reach WARN emission during cold start. The two linked startup-grace PRs were both closed unmerged, so there is no canonical open fix to land yet.

## What This Changes

Request adding a diagnostic liveness startup-grace window so cold-start blocking does not emit false liveness warnings while sampler baselines continue to reset.

## Best Possible Solution

Add an internal diagnostic startup-grace boundary that keeps sampling every heartbeat but suppresses liveness event and WARN emission until the grace window expires.

## Reproduction Assessment

Yes, from source inspection: current main starts diagnostic liveness at gateway startup, has no grace check, and an existing queued-work test shows a first-window liveness sample reaches WARN emission. A focused fake-timer regression test can reproduce this without the reporter's Windows host.

## Solution Assessment

Yes. The narrow fix is to add an internal `startupGraceMs`, always call the sampler for metric resets/baselines, and gate only liveness event/WARN emission during startup grace; the env/config knob is a separate enhancement.
