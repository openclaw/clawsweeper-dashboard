## Summary

Keep open: current main still starts the diagnostic heartbeat without a startup grace window, and the related startup-grace PR was closed unmerged. The current max-spike/debug gating helps older idle-warning reports, but it does not implement the requested cold-start emission grace for sustained startup blocking with queued or active work.

## What This Changes

Add a startup grace window to diagnostic liveness warning emission so cold-start event-loop delay is sampled but not warned during the initial Gateway startup window.

## Best Possible Solution

Add an internal 60-second diagnostic startup-grace guard that preserves sampler resets while suppressing liveness event/WARN emission during Gateway cold start, and leave configurability and broader event-loop saturation work to follow-ups.

## Reproduction Assessment

Yes, by source inspection: current main starts the heartbeat during Gateway startup, has no grace option, and the existing queued-work test shows liveness warnings can emit on the first tick. I did not run a live Windows cold-start reproduction.

## Solution Assessment

Yes for the narrow repair direction: the prior unmerged PR’s emission-only grace is the right shape because it keeps sampler baselines fresh while suppressing startup noise. A public env/config knob is a separate product decision and should not be bundled into the first fix.

## Repair Work Prompt

Implement a narrow fix for https://github.com/openclaw/openclaw/issues/79915. Current main starts `startDiagnosticHeartbeat` from `src/gateway/server.impl.ts` without a grace window and `src/logging/diagnostic.ts` samples/emits liveness reports immediately. Add an internal `startupGraceMs` option to `StartDiagnosticHeartbeatOptions`, pass `60_000` from Gateway startup, always call the liveness sampler every tick, and discard only liveness event/WARN emission while `now < graceUntil`. Add focused regression coverage in `src/logging/diagnostic.test.ts` proving the sampler is called during grace, no `diagnostic.liveness.warning` event or `liveness warning:` WARN log is emitted inside grace with queued work, and warning resumes after grace expires. Add a single-line `CHANGELOG.md` fix entry. Do not add `OPENCLAW_STARTUP_GRACE_MS` or any public config knob in this PR; treat configurability as a follow-up product decision. Preserve existing liveness warning semantics after grace and avoid broad event-loop saturation changes.
