## Summary

Keep open: current main still starts diagnostic liveness sampling at gateway startup without any startup-grace guard, and the prior closed PR implements the narrow missing behavior but was never merged.

## What This Changes

The issue requests a startup grace window for diagnostic liveness warnings so normal cold-start blocking does not emit WARN-level noise.

## Best Possible Solution

Add an internal diagnostic startup-grace boundary that preserves sampler resets during startup but suppresses liveness event and WARN emission until the grace window expires.

## Reproduction Assessment

Yes, from source inspection: current main starts diagnostic liveness sampling at gateway startup and has no grace check before liveness event or warning emission. A focused fake-timer test with queued work during the first heartbeat window should reproduce the warning path without needing the reporter's Windows setup.

## Solution Assessment

Yes; the narrowest maintainable fix is the prior PR's shape: add an internal `startupGraceMs`, always run the sampler for metric resets, and discard the sample only while startup grace is active. The optional env/config knob is not needed for the bug fix and should be separate follow-up work if desired.
