## Label Justifications

- `P1`: The issue reports a current, user-facing memory workflow break where chat memory_search becomes unavailable after Active Memory timeouts even though the QMD backend remains healthy.
- `impact:session-state`: The affected state is the session/runtime memory manager lifecycle: a timed-out embedded recall can leave later chat memory_search calls using a closed shared manager.

## Summary

Keep open: current main and the latest release still have a source-reproducible lifecycle bug where Active Memory timeout cleanup can close the agent-scoped shared QMD memory_search manager, and the exact failure is narrower than the related active-memory/QMD reliability reports.

## What This Changes

The issue reports that active-memory timeout cleanup can close the shared QMD memory_search manager used by normal chat-session memory_search.

## Best Possible Solution

Narrow Active Memory timeout cleanup so request-scoped recall cancellation cannot close the shared/default QMD chat memory manager, while preserving a disposable or owner-scoped cleanup path for the retained local embedding resources fixed by https://github.com/openclaw/openclaw/pull/84048.

## Reproduction Assessment

Yes at source level: current main schedules Active Memory timeout cleanup, that cleanup closes the agent-scoped QMD cache entry, and normal chat memory_search uses that shared/default manager path. I did not run the reporter's macOS/QMD live setup.

## Solution Assessment

No, the current implementation is not the best solution because it uses agent-scoped teardown for a request-scoped Active Memory timeout. The maintainable fix is to isolate or narrow the cleanup so shared QMD chat memory stays available while previous timeout leak cleanup remains covered.
