## Summary

Keep open: current main and v2026.6.10 still route active-memory timeout cleanup into the full memory manager close path, and the linked fix PRs remain open rather than merged.

## What This Changes

The issue reports that active-memory timeout cleanup closes the shared QMD-backed memory_search manager, causing later chat memory_search calls to return closed-manager errors while CLI search still works.

## Best Possible Solution

Land one focused fix that preserves the long-lived shared QMD manager during request-scoped active-memory timeouts while retaining disposable local index/provider cleanup and regression coverage.

## Reproduction Assessment

Yes, at source level: active-memory timeout cleanup still delegates to the full close path, and memory-core closes the cached QMD manager that chat memory_search can reuse. I did not run the live QMD gateway scenario in this read-only review.

## Solution Assessment

Yes, the narrow maintainable fix is to separate request-scoped active-memory timeout cleanup from shared QMD manager retirement. The open linked PRs implement candidate shapes, so they should be reviewed or rejected before this issue closes.
