## Label Justifications

- `P1`: The issue describes a broken user-facing chat memory workflow after active-memory timeouts while the QMD backend remains healthy.
- `impact:session-state`: The affected state is the long-lived shared memory_search manager used by later chat-session memory calls.

## Summary

Keep open: current main and latest release still route active-memory timeout cleanup into the shared QMD manager close path, and the linked PRs are open candidate fixes rather than merged implementation.

## What This Changes

The issue reports that an active-memory recall timeout can close the shared QMD-backed memory_search manager, making chat memory_search unavailable while CLI QMD search still works.

## Best Possible Solution

Review and land one focused fix that keeps active-memory timeout cleanup from retiring the shared QMD manager, then close this issue after that fix reaches main.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: active-memory timeout cleanup calls closeActiveMemorySearchManager, which delegates to closeMemorySearchManager and closes the cached QMD manager for that agent. I did not run a live QMD gateway reproduction in this read-only review.

## Solution Assessment

Yes at the issue boundary: the maintainable fix is to narrow timeout cleanup so request-scoped active-memory cancellation does not tear down the shared QMD manager, without adding a user-facing config knob. The open linked PRs are candidate implementations that still need maintainer review and conflict resolution.
