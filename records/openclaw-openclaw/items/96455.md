## Label Justifications

- `P1`: The issue describes a released active-memory plus QMD workflow where chat memory_search can become unavailable after timeout while the backend and CLI search remain healthy.
- `impact:session-state`: The affected state is the agent-scoped shared memory_search manager and subsequent chat recall availability.

## Summary

Keep open: current main and the latest release still route active-memory timeout cleanup into the full memory manager close path, and the issue now has open candidate fix PRs rather than a merged implementation.

## What This Changes

The issue requests that active-memory timeouts cancel only the recall attempt and not close the shared QMD memory_search manager used by normal chat memory calls.

## Best Possible Solution

Land one focused fix that makes active-memory timeout cleanup release only disposable recall resources while preserving the shared QMD manager, then close this issue after the chosen PR merges.

## Reproduction Assessment

Yes. The current-main source path routes active-memory timeout cleanup into full manager close, and memory-core then closes the cached QMD manager that later chat memory_search calls can reuse; I did not run a live QMD gateway repro.

## Solution Assessment

Yes at the issue level. The narrow maintainable fix is to preserve the long-lived QMD manager during request-scoped active-memory timeout cleanup while keeping the disposable resource cleanup; the open PRs implement candidate shapes for that outcome.
