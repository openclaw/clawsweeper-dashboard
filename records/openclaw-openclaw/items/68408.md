## Summary

Keep this PR open because it targets a real remaining memory isolation gap for opt-in inline/both dreaming storage and legacy daily files, which the default-storage fix did not fully remove. The branch is not mergeable as-is: it still has a normalized-path bypass, a public Plugin SDK baseline mismatch, an ordinary-content formatting rewrite, and no after-fix real behavior proof.

## What This Changes

This PR adds a Plugin SDK markdown helper for stripping managed Light/REM dreaming blocks, applies it to memory-core QMD daily full and partial reads, and adds helper/read-path tests.

## Best Possible Solution

Land a corrected memory-core read-path filter that classifies daily files from the resolved workspace-relative path, strips only managed Light/REM blocks from full and paginated memory_get reads, preserves ordinary daily-note text when no block is removed, keeps dreaming internals on raw files, and updates the SDK baseline if the helper remains public.

## Reproduction Assessment

Yes. Source inspection shows current main returns raw daily-file text from QMD readFile(), while inline/both storage can still write managed Light/REM blocks into memory/YYYY-MM-DD.md; the PR's remaining bypass follows from resolveReadPath() canonicalizing a noncanonical request before the PR checks raw relPath.

## Solution Assessment

No. The read-path filtering direction is appropriate, but the implementation should classify the resolved canonical path, avoid no-op formatting rewrites, include the generated SDK baseline, and provide real behavior proof before merge.

Overall correctness: patch is incorrect

  - confidence: 0.91

  - confidence: 0.9

  - body: `stripDreamingManagedBlocks()` always rewrites runs of blank lines, even when no dreaming markers were removed. Daily memory reads should preserve ordinary user-authored notes exactly unless a managed block was actually stripped.
  - confidence: 0.86
