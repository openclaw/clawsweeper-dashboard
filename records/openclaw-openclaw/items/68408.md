## Summary

Keep this PR open: it targets a real memory-core read-path gap that is not implemented on current main, but the current head still has blocking correctness/API-contract issues and no contributor-provided real behavior proof.

## What This Changes

The PR adds a public Plugin SDK helper for stripping managed Light Sleep and REM dreaming blocks, applies it to memory-core QMD daily full and partial reads, and adds focused tests.

## Best Possible Solution

Land a corrected memory-core read-path filter that classifies daily files from the resolved canonical path, strips only managed Light/REM blocks from full and paginated memory_get reads, preserves ordinary daily-note text when no block is removed, keeps dreaming internals reading raw files, and updates the SDK API baseline if the helper remains public.

## Reproduction Assessment

Yes, at source level: current main returns raw QMD daily-file text from readFile(), while inline/both storage and legacy daily files can still contain managed Light/REM blocks. I did not run a runtime memory_get flow because this review is read-only.

## Solution Assessment

No, not as currently written. Filtering daily reads is the right direction, but this branch must classify the resolved canonical path, avoid no-op formatting rewrites, include the generated SDK baseline, and provide real behavior proof.
