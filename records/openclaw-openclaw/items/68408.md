## Summary

Keep open. Current main still returns daily memory files without stripping managed dreaming blocks, and this PR targets a real residual inline/legacy contamination path, but the branch should not merge as-is because it leaves a normalized-path bypass, omits the required Plugin SDK API baseline update, rewrites ordinary daily-note blank lines, and is currently dirty against its base.

## What This Changes

This PR adds a public `stripDreamingManagedBlocks()` markdown helper, applies it to QMD daily-memory full and partial reads, and adds helper/read-path tests.

## Best Possible Solution

Land a corrected memory-core read-path filter that classifies daily files from the resolved canonical workspace/collection-relative path, strips only managed Light/REM blocks from both full and paginated `memory_get` reads, preserves ordinary daily-note text when no managed block was removed, keeps dreaming internals reading raw files, and either keeps the helper local or updates the Plugin SDK API baseline and changelog.

## Reproduction Assessment

Yes. On current main, a daily file containing `<!-- openclaw:dreaming:light:start -->` is returned raw by `QmdMemoryManager.readFile()` for full and partial reads; on the PR branch, a normalized-equivalent request such as `memory/./2026-04-18.md` still resolves to the same file while skipping the raw-`relPath` daily-file check.

## Solution Assessment

No. The read-path filtering direction is right, but the patch should classify the resolved path, avoid changing blank lines when no managed block was stripped, and satisfy the Plugin SDK/changelog contracts before merge.
