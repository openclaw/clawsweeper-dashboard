## Summary

Keep open: this is a plausible fix for a real memory isolation bug, but the branch still has correctness blockers, lacks contributor-provided real behavior proof, and is currently conflicting with main.

## What This Changes

The PR adds a public Plugin SDK markdown helper for stripping managed Light/REM dreaming blocks, applies it to memory-core QMD daily full and partial reads, and adds focused tests.

## Best Possible Solution

Land a corrected memory-core read-path filter that classifies daily files from the resolved canonical path, strips only managed Light/REM blocks from full and paginated memory_get reads, preserves ordinary daily-note text when no block is removed, keeps dreaming internals reading raw files, and updates the SDK baseline if the helper remains public.

## Reproduction Assessment

Yes, at source level: memory_get delegates to QmdMemoryManager.readFile, current main returns raw daily-file text, and inline/both or legacy daily files can still contain managed Light/REM blocks. I did not execute a runtime memory_get flow because this review was read-only.

## Solution Assessment

No, not as currently written. Filtering daily reads is the right direction, but this branch must use the resolved canonical path, avoid no-op formatting rewrites, include the generated SDK baseline, refresh conflicts, and provide real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Normalize the resolved daily path before filtering:** `extensions/memory-core/src/memory/qmd-manager.ts:1216`
  - body: `resolveReadPath()` canonicalizes the file that will be read, but this check tests the raw request `relPath`. A request such as `memory/./2026-04-18.md` resolves to the same daily file while skipping the stripping branch, so managed Light/REM content can still leak through path syntax alone; base the daily-file decision on the resolved workspace-relative path for both full and partial reads.
  - confidence: 0.91
- **[P2] Update the Plugin SDK API baseline:** `src/plugin-sdk/memory-host-markdown.ts:65`
  - body: `memory-host-markdown` is a published Plugin SDK subpath, and this PR adds a new public export from it. Regenerate and commit `docs/.generated/plugin-sdk-api-baseline.sha256`, or keep the helper plugin-local, so the SDK contract check stays aligned.
  - confidence: 0.9
- **[P2] Only collapse blank lines after removing a block:** `src/plugin-sdk/memory-host-markdown.ts:75`
  - body: `stripDreamingManagedBlocks()` always rewrites runs of blank lines, even when no dreaming markers were removed. Daily `memory_get` reads should remain exact excerpts for ordinary user-authored notes; only normalize blank lines after a managed block was actually stripped.
  - confidence: 0.86
