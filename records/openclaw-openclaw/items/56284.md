## Summary

Keep open: current main and v2026.5.22 provide an explicit `--safe` restart path and health checks, but plain Windows Scheduled Task restart is still documented and implemented as the compatibility service-manager path, so the remaining question is a maintainer product decision rather than an already-implemented fix.

## What This Changes

The issue requests that Windows `openclaw gateway restart` drain active work and recover sessions instead of using the immediate Scheduled Task stop/start path.

## Best Possible Solution

Make an explicit maintainer decision: either promote the Windows default restart to the OpenClaw-aware safe path while preserving `--force`/escape behavior and adding native Windows proof, or keep plain service-manager restart documented and close this as the supported `--safe` workflow.

## Reproduction Assessment

Yes, at source level: current main and v2026.5.22 still route plain Windows Scheduled Task restart through `/End` plus `/Run`, while the OpenClaw-aware drain path is exposed separately through `--safe`. I did not run a native Windows live repro in this read-only pass.

## Solution Assessment

Unclear until maintainers choose the product contract for plain restart. The safest code direction is to preserve an explicit immediate/force path while making the default Windows operator path drain active work only if maintainers accept that compatibility change.
