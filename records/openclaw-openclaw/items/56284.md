## Summary

Keep open. Current main contains restart intent, restart-drain, health-check, and restart-aborted main-session recovery pieces, but the Windows Scheduled Task restart path still ends the task, terminates Gateway listeners, waits only briefly for port release, and starts the task again. The direct implementation PR #56292 is closed unmerged, and open PR #45044 addresses a related session-transcript preservation case rather than the Windows Scheduled Task drain contract.

## What This Changes

The issue requests that Windows Scheduled Task gateway restarts drain active work, write restart/session recovery state, wait for a healthy restart, and avoid losing sessions.

## Best Possible Solution

Define a single Windows Scheduled Task restart contract that coordinates with the running Gateway: persist restart/recovery state, let the Gateway-owned drain path run without immediate listener cleanup, wait for the restarted Gateway to become healthy, and ensure interrupted sessions are completed, resumed, or explicitly failed without silent loss.

## Reproduction Assessment

Unclear. The issue gives concrete Windows 11 Scheduled Task steps and current source shows a plausible forced-cleanup path, but a high-confidence reproduction still requires a live Windows run with active Gateway work to observe drain and session recovery behavior.

## Solution Assessment

Unclear. The proposed direct wiring to deferGatewayRestartUntilIdle and writeRestartSentinel is useful context, but the safer maintainable fix is likely to integrate the Scheduled Task service path with the existing gateway-owned restart intent, drain, restart-sentinel/session-recovery, and health mechanisms instead of adding a parallel Windows-only flow.
