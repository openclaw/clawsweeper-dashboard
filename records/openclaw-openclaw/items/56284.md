## Summary

Keep open: current main provides an OpenClaw-aware `openclaw gateway restart --safe` path and in-process restart draining, but the documented plain Windows Scheduled Task restart path is still the compatibility service-manager path and the Scheduled Task restart implementation still ends the task and terminates lingering gateway listeners. That leaves the reported plain-restart data/session/message-loss behavior as a maintainer product and upgrade-safety decision, not a proven implemented-on-main close.

## What This Changes

The issue asks OpenClaw to make Windows Scheduled Task gateway restarts wait for active work and preserve or recover session state instead of killing running tasks.

## Best Possible Solution

Have maintainers decide whether Windows plain restart should become OpenClaw-aware by default or whether the documented `--safe` path satisfies the report, then require native Windows Scheduled Task proof for the chosen behavior.

## Reproduction Assessment

No live current-main Windows Scheduled Task reproduction was run. Source inspection gives a credible source repro: docs distinguish `--safe` from plain restart, and the Windows Scheduled Task restart path still uses `/End` plus listener termination while tests assert that cleanup behavior.

## Solution Assessment

Unclear: the issue's proposed unconditional Windows graceful restart is plausible but not clearly the best default because current docs preserve plain restart as a compatibility service-manager path. The maintainable solution is an explicit maintainer decision plus Windows proof, not an automatic cleanup close.
