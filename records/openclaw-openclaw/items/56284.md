## Summary

Keep open: current main and v2026.5.22 still make the OpenClaw-aware drain path opt-in via `--safe`, while plain Windows Scheduled Task restart still uses the service-manager stop/start path and the mid-turn session reset predicate remains present in source.

## What This Changes

The issue asks for Windows `openclaw gateway restart` to wait for active work, preserve/recover session state, and avoid force-terminating the Scheduled Task gateway path.

## Best Possible Solution

Keep this as the canonical Windows restart/session-loss issue until maintainers choose the plain-vs-safe restart contract, then route Windows managed restarts through coordinated drain behavior with native proof and fix the mid-turn session reset regression with focused tests.

## Reproduction Assessment

Yes at source level: current main still has a plain Windows Scheduled Task restart path that terminates listeners with short grace, and the session manager still truncates header-plus-user/no-assistant files. I did not run a native Windows end-to-end reproduction in this read-only review.

## Solution Assessment

No, the current implementation is only a partial solution: `--safe` is maintainable for opt-in coordinated restarts, but plain restart remains documented and implemented as service-manager behavior. The durable fix needs maintainer approval for that contract plus Windows proof.
