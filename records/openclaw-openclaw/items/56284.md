## Summary

Keep open. Current main still routes Windows Gateway service restarts through the Scheduled Task restart path, which ends the task, performs short-grace listener cleanup, waits for port release, and starts the task again. Main has adjacent restart-intent, drain, health-check, and restart-aborted session-recovery pieces, but the direct implementation PR #56292 is closed unmerged and related open PRs #45044 and #52487 cover narrower session-preservation or duplicate-start races rather than the full Windows Scheduled Task drain contract.

## What This Changes

The issue requests that Windows Scheduled Task Gateway restarts drain active work, persist restart/session recovery state, wait for healthy restart, and avoid silent session loss.

## Best Possible Solution

Define one Windows Scheduled Task restart contract that coordinates with the running Gateway: persist restart/recovery state, let the Gateway enter its existing drain path without premature listener cleanup, wait for a healthy replacement, and make interrupted sessions complete, resume, or fail visibly instead of disappearing.

## Reproduction Assessment

Unclear. The issue gives concrete Windows 11 Scheduled Task steps and current source shows a plausible forced-cleanup path, but high-confidence reproduction still requires a live Windows run with active Gateway work to observe drain and session recovery behavior.

## Solution Assessment

Unclear. The issue's direct wiring proposal is useful context, but the best maintainable fix is likely to integrate the Scheduled Task service path with the existing restart intent, drain, sentinel/continuation, health, and recovery mechanisms rather than add a parallel Windows-only restart flow.
