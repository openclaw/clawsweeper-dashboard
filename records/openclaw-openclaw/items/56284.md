## Label Justifications

- `P1`: The report describes a Windows gateway restart workflow that can interrupt active work and session continuity for real users.
- `impact:data-loss`: The reported failure mode can discard in-flight task results or restart recovery state.
- `impact:message-loss`: Active replies or task outputs can be lost if restart happens before delivery drains.
- `impact:session-state`: The issue centers on preserving and recovering active session state across gateway restart.

## Summary

Close: current main and the latest release implement the accepted safe-restart alternative, with `openclaw gateway restart --safe` deferring restart until active OpenClaw work drains while plain `restart` remains the compatibility service-manager path.

## What This Changes

The issue requests a Windows-safe gateway restart path that waits for active work, coordinates session recovery, and proves restart health instead of forcefully ending Scheduled Task runs.

## Best Possible Solution

Keep the shipped `--safe` path as the OpenClaw-aware restart and document that plain restart remains the compatibility service-manager path.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path for the historical plain Scheduled Task behavior: the non-safe Windows service restart still uses `/End` then `/Run`. The current supported mitigation is the documented `--safe` path, which was not live-tested on Windows in this read-only review.

## Solution Assessment

Yes. The current implementation is the maintainable product choice: preserve plain restart compatibility and make the active-work drain explicit through `openclaw gateway restart --safe`, with `--force` and `--skip-deferral` as operator escape hatches.
