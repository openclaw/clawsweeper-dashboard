## Summary

Keep open: current main and v2026.5.22 include a safe restart path and restart health checks, but plain Windows Scheduled Task restart is still documented and implemented as the service-manager path, and the remaining decision is whether to change that compatibility behavior for Windows by default.

## What This Changes

The issue asks for Windows `openclaw gateway restart` to wait for active tasks, write restart recovery state, and preserve session/message state across Scheduled Task restarts.

## Best Possible Solution

Make an explicit maintainer decision: either promote the Windows default restart to the OpenClaw-aware safe path with `--force`/`--wait` compatibility and native Windows proof, or keep plain service-manager restart documented and close this as the supported `--safe` workflow.

## Reproduction Assessment

Yes, at source level: current main and v2026.5.22 still route plain Windows Scheduled Task restart through `/End` plus `/Run`, while the OpenClaw-aware drain path is exposed separately through `--safe`. I did not run a native Windows live repro in this read-only pass.

## Solution Assessment

Unclear until maintainers choose the product contract for plain restart. The safest code direction is to preserve an explicit immediate/force path while making the default Windows operator path drain active work only if maintainers accept that compatibility change.
