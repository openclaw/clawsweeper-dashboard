## Label Justifications

- `P1`: The reported default Windows restart path can interrupt active user work and session delivery, making it a significant reliability issue.
- `impact:session-state`: The report centers on session state not being preserved or recovered correctly across Windows gateway restart.
- `impact:data-loss`: Abruptly ending active tasks can lose task results, transcript progress, or restart recovery state.
- `impact:message-loss`: A forced restart can prevent pending replies or active task output from reaching the user.

## Summary

Keep this open: current main and v2026.6.6 now provide an explicit `openclaw gateway restart --safe` path, but the documented plain Windows Scheduled Task restart still keeps the service-manager `schtasks /End` then `/Run` behavior for compatibility. The remaining question is whether that default should change, which needs maintainer product/compatibility judgment.

## What This Changes

The issue asks for Windows Scheduled Task gateway restarts to drain active work and preserve or recover session state instead of forcefully ending active tasks.

## Best Possible Solution

Maintainers should choose the supported restart contract; if safe-by-default is desired, route plain Windows restart through the safe coordinator with Windows Scheduled Task proof, otherwise document and close around `--safe` as the no-loss command.

## Reproduction Assessment

No live high-confidence reproduction was run. Source inspection shows the plain Windows path still calls the service-manager restart while the active-work-draining path is only behind `--safe`.

## Solution Assessment

Unclear. The explicit `--safe` implementation is a plausible supported solution, but deciding whether the plain command should change default semantics needs maintainer compatibility judgment.
