## Label Justifications

- `P1`: The default Windows gateway restart path can interrupt active user work and session delivery, making this a significant real-user reliability bug.
- `impact:session-state`: The issue centers on session state not being preserved or recovered correctly across Windows gateway restart.
- `impact:data-loss`: Abruptly ending active work can lose task results, transcript progress, or restart recovery state.
- `impact:message-loss`: A forced restart can prevent pending replies or active task output from reaching the user.

## Summary

Keep this open: current main has an explicit safe restart path, but the reported plain Windows Scheduled Task restart still follows the service-manager `schtasks /End` then `/Run` path instead of waiting for active work to drain. The linked merged Windows restart PR fixed a duplicate `/Run` race, not the safe-by-default restart behavior requested here.

## What This Changes

The issue requests Windows Scheduled Task `openclaw gateway restart` to wait for active work, preserve restart/session recovery state, and return only after the Gateway is healthy.

## Best Possible Solution

Maintainers should choose and document the operator contract, then add Windows Scheduled Task proof and tests for the chosen default, `--safe`, and force/bypass restart behavior.

## Reproduction Assessment

Yes, source-reproducible at medium confidence: current main still routes plain Windows restart to `restartScheduledTask`, which directly ends and reruns the task instead of deferring on active work. I did not run a native Windows Scheduled Task live repro in this read-only pass.

## Solution Assessment

No clear best fix is selected yet. The existing `--safe` path is a plausible implementation building block, but deciding whether the plain command should use it by default needs maintainer product and compatibility judgment.
