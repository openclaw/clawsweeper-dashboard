## Summary

Keep open. Current main still assigns every Bedrock Mantle `/v1/models` discovery row the 32K fallback, so the PR addresses a real provider metadata gap; however, the branch needs repair before merge because its final 200K fallback overstates omitted 128K models, one listed Nemotron value disagrees with the merged Bedrock table, and the branch predates the merged Claude Opus 4.7 Mantle route.

## What This Changes

The PR changes `amazon-bedrock-mantle` discovery to assign known context windows to selected Mantle model IDs and raise the default discovered-model context window.

## Best Possible Solution

Land a rebased, tested Mantle-local implementation that maps known open-weight model IDs, preserves the current Claude Anthropic Messages handling, uses conservative fallback semantics, and documents the user-visible provider behavior in the changelog.

## Reproduction Assessment

Yes. The existing `discoverMantleModels` fetch seam can mock `/v1/models`: on current main, a returned ID such as `minimax.minimax-m2.5` gets 32K, while this PR's omitted 128K-model path would get the new 200K fallback.

## Solution Assessment

No, not as written. A known-context-window table is the right shape and matches the merged Bedrock approach, but this branch should not rely on a 200K fallback for omitted 128K Mantle models and must rebase around the current Claude route.
