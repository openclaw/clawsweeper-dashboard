## Summary

Keep open: this is a broad new continuation feature that is not on current main, and the latest head still has a blocking policy regression in the delivery-time recheck for lineage fanout. The proof is also not current-head matched for the changed gate path.

## What This Changes

The branch adds opt-in continuation tools/config, durable delegate and post-compaction dispatch, cross-session return targeting, trace propagation/OTel support, status/protocol/docs/tests, and a direct `uuid` dependency.

## Best Possible Solution

Keep the feature opt-in, make every enforcement point use the same targeting-policy helper so lineage fanout remains allowed by default, and require current-head real behavior proof plus maintainer approval for trusted cross-session delivery.

## Reproduction Assessment

Yes for the review finding by source inspection: with default `crossSessionTargeting: "disabled"`, the tool accepts `fanoutMode: "tree"`, but the normal child-completion delivery path rejects it as cross-session. No current-main bug reproduction applies because the continuation feature is not on main.

## Solution Assessment

No, not as currently proposed. The delivery-time recheck should reuse `hasCrossSessionDelegateTargeting` or equivalent semantics so `tree` lineage routing is allowed while `all` and non-self explicit targets remain blocked unless opted in.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Allow tree fanout under the default policy:** `src/agents/subagent-announce.ts:1231-1234`
  - body: `fanoutMode: "tree"` is documented and accepted as lineage routing when `crossSessionTargeting` is disabled, but this delivery-time recheck treats it as cross-session and returns before enqueueing the completion. A delegate accepted under the default policy can therefore drop its ancestor return at completion time; use the shared cross-session targeting helper here so only `all` and non-self explicit targets are blocked.
  - confidence: 0.93
