## Summary

Keep this PR open: exact head still has a merge-blocking Discord REST lane regression where standard user-visible writes can be dropped locally after 60 seconds, despite the PR and changelog framing stale dropping as background-work handling.

## What This Changes

The PR adds Discord REST priority lanes, gateway send validation and invalid-session jitter, command metadata comparison normalization, one-off component wait helpers, top-level component forwarding, tests, and a changelog entry.

## Best Possible Solution

Keep the Discord-local lane scheduler, but make stale dropping background-only for OpenClaw standard message writes and add a regression test proving standard sends, edits, and deletes stay queued until Discord accepts or rejects them.

## Reproduction Assessment

Yes. On PR head, use a one-worker `RequestClient`, block the first request, enqueue a standard webhook or non-GET channel-message write, advance fake time past 60000 ms, then release the worker; the standard write is rejected as stale.

## Solution Assessment

No. The plugin-local boundary is right, but default stale dropping on the standard lane is broader than the PR's user-facing policy; the safer fix is to remove or opt out `standard.staleAfterMs` unless maintainers explicitly choose that Carbon behavior.
