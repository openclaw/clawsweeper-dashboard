## Summary

Keep open: this PR is not obsolete and is scoped to the bundled Discord plugin, but the current head still can drop normal user-visible REST writes as stale after 60 seconds, so it needs a narrow repair before automerge.

## What This Changes

The PR adds Discord REST request lanes, gateway send validation, member-intent checks, invalid-session jitter, command metadata normalization, one-off component helpers, message-action component forwarding, tests, and a changelog entry.

## Best Possible Solution

Keep the Discord-local hardening, but make stale dropping background-only or otherwise opt-in so normal webhook and channel-message writes remain queued until sent or fail from Discord.

## Reproduction Assessment

Yes. A source-level reproduction is clear: use one REST worker, block the first request, enqueue a standard send/edit/delete route, advance time past 60000 ms, then release the worker and observe local stale rejection.

## Solution Assessment

No. The PR is scoped to the right Discord plugin surface, and the prior option-localization review is addressed, but applying a default stale timeout to standard writes is not the narrowest maintainable policy.
