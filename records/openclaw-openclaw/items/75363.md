## Summary

Keep this PR open. The Discord-local hardening is useful and has a changelog entry, but the latest head still has a concrete P2 blocker in the new component wait helper, and the PR is protected/automerge-tracked rather than closeable cleanup.

## What This Changes

The PR adds Discord REST priority lanes and stale-drop behavior, gateway send validation, command reconcile normalization, one-off component wait helpers, focused tests, and a changelog entry.

## Best Possible Solution

Keep the Discord plugin hardening, but repair the one-off component wait path so initial replies, deferred edits, and already-replied follow-ups all wait on the actual message that contains the components.

## Reproduction Assessment

Yes. A source-level reproduction is to call replyAndWaitForComponent after the interaction is already replied; reply() takes the follow-up path, followUp() does not use wait: true, and the helper falls back to @original so clicks on the new follow-up message time out.

## Solution Assessment

No. The Discord-local direction is the right scope, but the current helper implementation is not mergeable until it obtains the concrete follow-up message, likely by using webhook wait: true when a component wait needs the created message.
