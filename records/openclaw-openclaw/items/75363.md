## Summary

Keep this PR open: the latest PR head still applies stale dropping to normal Discord outbound writes, so the automerge path should wait for the narrow REST lane fix.

## What This Changes

The PR adds Discord REST priority lanes, stale background dropping, gateway payload/member-intent validation, invalid-session jitter, command metadata normalization, component wait helpers, top-level component forwarding, tests, and a changelog entry.

## Best Possible Solution

Keep the Discord-local hardening, but make stale dropping background-only and add a regression test proving standard sends, edits, and deletes stay queued until sent or rejected by Discord.

## Reproduction Assessment

Yes. Use a one-worker `RequestClient`, block the first request, enqueue a webhook or non-GET channel-message write, advance fake time past 60000 ms, then release the worker; PR-head code rejects the standard write as stale.

## Solution Assessment

No. The PR is scoped to the right Discord plugin code, but default stale dropping on the standard lane is broader than the stated background-work policy; the safer fix is to remove or opt out `standard.staleAfterMs`.
