## Summary

Keep this PR open. The latest head fixes the earlier unrelated `.agents` and changelog blockers, but the new one-off component wait helper can register against the wrong message for follow-up replies, so the patch should not merge as-is.

## What This Changes

This PR hardens Discord REST scheduling, gateway send validation, command reconciliation comparisons, component send forwarding, one-off component wait helpers, focused tests, and the changelog.

## Best Possible Solution

Keep the Discord-local hardening, repair the one-off component wait helper so it waits on the actual message it sends or edits in every response state, add a focused regression test, and let exact-head CI/automerge gate the result.

## Reproduction Assessment

Yes. A source-level reproduction is to call `replyAndWaitForComponent` after an interaction has already replied; `reply()` takes the follow-up path, the helper falls back to `fetchReply()`, and a click on the follow-up components will not resolve the registered wait.

## Solution Assessment

No. The broad Discord hardening is in the right plugin boundary, but the helper should obtain the concrete follow-up message, likely by using the webhook `wait` response for follow-ups instead of falling back to `@original`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Wait on the actual follow-up message:** `extensions/discord/src/internal/interactions.ts:231`
  - body: When `responseState` is already `replied`, `this.reply()` sends a follow-up via `createWebhookMessage` without `wait: true`, so `result` is not a message and this fallback fetches `@original`. The one-off wait is then registered on the old message, and clicks on components in the newly sent follow-up will time out.
