## Summary

Keep this PR open. The Discord hardening is on the right plugin surface, but the latest head still applies stale-drop behavior to the `standard` REST lane, so normal user-visible Discord writes can be rejected locally after 60 seconds.

## What This Changes

The PR adds Discord REST priority lanes, gateway send and member-request validation, command metadata comparison normalization, one-off component wait helpers, explicit component forwarding, tests, and a changelog entry.

## Best Possible Solution

Keep the Discord-local hardening, but make stale dropping background-only and add a regression test proving standard sends, edits, and deletes remain queued until sent or rejected by Discord.

## Reproduction Assessment

Yes. Use a one-worker `RequestClient`, block the first request, enqueue a webhook or non-GET channel-message write, advance time past 60000 ms, then release the worker; PR-head code will reject the standard write as stale.

## Solution Assessment

No. The PR is scoped to the right Discord plugin code, but default stale dropping on the standard lane is broader than the stated background-work policy; the safer fix is to remove or opt out `standard.staleAfterMs`.
