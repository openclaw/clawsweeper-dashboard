## Summary

Keep open: current main does not contain the Discord Carbon-parity hardening, and the automerge PR still has a narrow REST lane regression that should block merge until repaired.

## What This Changes

This PR updates the bundled Discord plugin REST scheduler, gateway send/session handling, command comparison normalization, one-off component helpers, top-level component forwarding, tests, and changelog.

## Best Possible Solution

Keep the Discord-local scheduler and hardening, but make stale dropping background-only by default and prove standard sends, edits, and deletes stay queued until Discord accepts or rejects them.

## Reproduction Assessment

Yes. On PR head, a one-worker `RequestClient` can block the first request, enqueue a standard webhook or non-GET channel-message write, advance fake time past 60000 ms, then release the worker and observe a local stale rejection.

## Solution Assessment

No. The plugin-local boundary is right, but default stale dropping on the standard lane is broader than the stated behavior; remove or opt out `standard.staleAfterMs` unless explicitly configured.
