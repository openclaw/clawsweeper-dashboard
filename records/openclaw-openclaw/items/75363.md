## Summary

Keep this PR open. The Discord-local hardening is mostly in the right area, but the latest head still registers one-off component waits against `@original` after an already-replied interaction sends a follow-up, so clicks on the new follow-up message can time out.

## What This Changes

The PR adds Discord REST priority lanes and stale-drop behavior, gateway send validation and invalid-session jitter, command metadata normalization, one-off component waits, focused tests, and a changelog entry.

## Best Possible Solution

Keep the Discord plugin hardening, but make `replyAndWaitForComponent()` register against the actual message containing the components for initial replies, deferred edits, and already-replied follow-ups.

## Reproduction Assessment

Yes. A source-level reproduction is to call `replyAndWaitForComponent()` after the interaction has already been replied; `reply()` takes the follow-up path, `followUp()` does not use webhook `wait: true`, and the helper falls back to `@original`.

## Solution Assessment

No. The Discord-local direction is the right scope, but this PR is not mergeable until follow-up waits obtain the concrete created message, likely by using webhook `wait: true` on that path.
