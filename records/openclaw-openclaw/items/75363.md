## Summary

Keep open. The latest head addresses the earlier option-localization and standard-lane stale-drop blockers, but the REST scheduler now wakes every 5 ms while work is only waiting behind an active same-bucket request, so the patch is not merge-ready.

## What This Changes

This PR adds Discord REST priority lanes and stale background drops, hardens gateway send/session/member handling, normalizes command reconciliation metadata, forwards explicit components, and adds one-off component wait helpers with tests and changelog coverage.

## Best Possible Solution

Keep the Discord-local hardening, but make queued same-bucket waits event-driven by relying on request completion and rate-limit timers rather than a fixed short poll.

## Reproduction Assessment

Yes. A same-bucket queue with one active long request and another pending request on PR head reaches the new active-bucket branch that returns `waitMs: 5` repeatedly until the first request finishes; I verified this by source and diff inspection under the read-only review constraint.

## Solution Assessment

No. The plugin-local design and previous repairs are on the right path, but the active-bucket scheduler branch should not schedule a 5 ms drain when no request can start.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87
