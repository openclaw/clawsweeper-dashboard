## Summary

Keep this PR open: the Discord hardening is scoped to the right plugin surface, but the current head still lets normal Discord writes expire locally through the new standard REST lane timeout, so it is not merge-ready.

## What This Changes

The PR changes the bundled Discord plugin to add REST priority lanes, gateway send validation, member-intent checks, invalid-session jitter, component wait helpers, command metadata normalization, tests, and a changelog entry.

## Best Possible Solution

Keep the Discord-local hardening, but constrain stale dropping to background requests so standard sends, edits, and deletes remain queued until sent or rejected by Discord.

## Reproduction Assessment

Yes. Use one REST worker, block the first request, enqueue a standard webhook or channel-message write, advance time past 60000 ms, then release the worker and observe the PR-head scheduler reject the write as stale.

## Solution Assessment

No. The PR is scoped to the right Discord plugin surface, but default stale dropping on the standard lane is broader than the stated background-work policy and should be removed or made opt-in for standard writes.
