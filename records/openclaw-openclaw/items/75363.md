## Summary

Keep this PR open. The Discord-local hardening is useful and includes a changelog entry, but the latest head still has a P2 one-off component helper regression that should block automerge.

## What This Changes

This PR adds Discord REST priority lanes/stale-drop behavior, gateway send validation, command comparison normalization, component wait helpers, focused tests, and a changelog entry.

## Best Possible Solution

Keep the Discord-local hardening, but repair the one-off component wait helper so it waits on the actual message sent or edited in every response state before automerge proceeds.

## Reproduction Assessment

Yes. A source-level reproduction is to call `replyAndWaitForComponent` after the interaction is already `replied`; `reply()` takes the follow-up path, no created message is returned, and the helper registers against `@original`.

## Solution Assessment

No. The broad fix belongs in the Discord plugin, but the helper needs to obtain the concrete follow-up message, likely by using webhook `wait: true` for follow-up sends that need component waits.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Wait on the actual follow-up message:** `extensions/discord/src/internal/interactions.ts:231`
  - body: When `responseState` is already `replied`, `this.reply()` sends a follow-up webhook message without `wait: true`, so `result` is not the new message and this fallback fetches `@original`. The wait is then registered on the old message, and clicks on components in the new follow-up will time out.
