## Summary

Keep open. Exact head 93716076e709 is not merge-ready because two source-level regressions remain in the Discord plugin: one-off component waits can target `@original` after follow-up replies, and the standard REST lane can stale-drop normal writes.

## What This Changes

The PR hardens the bundled Discord plugin by adding REST priority lanes, gateway payload and intent checks, command reconciliation normalization, one-off component waits, message-action component forwarding, tests, and a changelog entry.

## Best Possible Solution

Keep the Discord-local hardening, but repair the follow-up message wait and restrict stale dropping to obsolete background work before exact-head review and CI gate automerge.

## Reproduction Assessment

Yes. A source-level reproduction is clear: call `replyAndWaitForComponent` after the interaction has already replied, and queue a standard webhook or channel-message write behind a blocked worker or long rate limit for more than 60000 ms.

## Solution Assessment

No. The PR is scoped to the right Discord plugin surface, but the implementation should wait on the actual created follow-up message and avoid applying stale-drop policy to normal writes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Wait on the created follow-up message:** `extensions/discord/src/internal/interactions.ts:246`
  - body: When `reply()` routes to `followUp()` for an already-replied interaction, the follow-up webhook call does not request Discord to return the created message, so `replyAndWaitForComponent` falls back to `fetchReply()` and registers the one-off wait on `@original`. Component clicks on the new follow-up can then time out.
  - confidence: 0.9
- **[P2] Keep standard writes out of stale dropping:** `extensions/discord/src/internal/rest.ts:74`
  - body: The `standard` lane has a default stale timeout, and webhook plus channel-message writes are classified as `standard`. A normal send/edit/delete queued behind a blocked worker or long rate limit for more than 60000 ms can be rejected as stale even though only obsolete background work should be dropped.
  - confidence: 0.86
