## Summary

Keep this PR open; it is not merge-ready. The Discord-local scope, changelog entry, and option-localization repair are good, but the current head still has component follow-up waits registering on the wrong message and also stale-drops standard outbound writes despite describing stale dropping as background-only work.

## What This Changes

The PR updates the Discord plugin REST scheduler, gateway validation and reconnect handling, command deploy comparison, interaction component wait helpers, message action component forwarding, focused tests, and CHANGELOG.md.

## Best Possible Solution

Keep the Discord plugin hardening, but repair the one-off component helper to wait on the actual created follow-up message and restrict stale dropping to background/read-only REST work before rerunning the Discord tests and changed gate.

## Reproduction Assessment

Yes. Source-level repros are clear: call `replyAndWaitForComponent()` after the interaction has already replied, and queue a standard Discord write behind a blocked worker or rate limit past 60000 ms.

## Solution Assessment

No. The Discord-local hardening direction is the right scope, but the current implementation is not the narrowest maintainable solution until follow-up waits use the created message and standard sends/edits/deletes are not treated as stale background work.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Wait on the actual follow-up message:** `extensions/discord/src/internal/interactions.ts:231`
  - body: When `responseState` is already replied, `reply()` sends a follow-up webhook message, but `followUp()` does not pass `wait: true`, so `result` is not the new message and this falls back to `@original`. The one-off wait is then registered on the old response, causing clicks on components in the new follow-up to time out.
  - confidence: 0.9
- **[P2] Do not stale-drop standard Discord writes:** `extensions/discord/src/internal/rest.ts:74`
  - body: The default `standard` lane now has a 60000 ms stale timeout, and the classifier puts webhook POST/PATCH/DELETE calls plus channel message writes in that lane. Under backlog or a long rate limit, normal sends, edits, or deletes can be rejected as stale even though the PR only intends stale dropping for background REST work.
  - confidence: 0.78
