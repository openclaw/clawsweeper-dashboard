## Summary

Keep open: the latest head still lets normal Discord message/webhook writes expire locally on the standard REST lane, so the PR is not merge-ready even though the earlier option-localization review concern appears addressed.

## What This Changes

This PR changes the bundled Discord plugin to add REST priority lanes and stale background dropping, gateway send validation and invalid-session jitter, command comparison normalization, one-off component wait helpers, top-level component forwarding, tests, and a changelog entry.

## Best Possible Solution

Keep the Discord-local scheduler and hardening, but make stale dropping background-only by default while proving standard sends, edits, and deletes remain queued until Discord accepts or rejects them.

## Reproduction Assessment

Yes. On PR head, a one-worker `RequestClient` can block the first request, enqueue a standard webhook or non-GET channel-message write, advance fake time past 60000 ms, then release the worker and observe a local stale rejection.

## Solution Assessment

No. The plugin-local boundary is right, but default stale dropping on the standard lane is broader than the stated behavior; the safer fix is to remove or opt out `standard.staleAfterMs` unless explicitly configured.
