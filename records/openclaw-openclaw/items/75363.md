## Summary

Keep this PR open. Current main does not yet contain the Discord Carbon-parity hardening, and the latest PR head still has two P2 correctness blockers in component waits and REST stale-dropping behavior.

## What This Changes

This PR changes the bundled Discord plugin REST scheduler, gateway send validation, interaction component waits, command deployment comparison, message-action component forwarding, tests, and changelog.

## Best Possible Solution

Keep the Discord-local hardening, but make component waits bind to the actual created follow-up message and restrict stale dropping to background REST work before exact-head checks gate automerge.

## Reproduction Assessment

Yes. A source-level repro is clear: call `replyAndWaitForComponent()` after the interaction has already replied, and queue a standard webhook or channel-message write behind a blocked worker or rate limit for more than 60000 ms.

## Solution Assessment

No. The PR is scoped to the right Discord plugin surface, but the current implementation is not the best fix until follow-up waits target the created message and normal writes cannot be dropped as stale background work.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Wait on the created follow-up message:** `extensions/discord/src/internal/interactions.ts:231`
  - body: When `reply()` routes to `followUp()` for an already-replied interaction, the webhook call does not pass `wait: true`, so Discord can return 204 and this falls back to `@original`. The one-off wait is then registered on the old response, so clicks on components in the follow-up can time out.
  - confidence: 0.9
- **[P2] Keep standard writes out of stale dropping:** `extensions/discord/src/internal/rest.ts:74`
  - body: The default `standard` lane has a 60000 ms stale timeout, while webhook writes and channel message writes are classified as `standard`. Under backlog or a long rate limit, normal sends, edits, or deletes can be rejected as stale even though only obsolete background REST work should be dropped.
  - confidence: 0.86
