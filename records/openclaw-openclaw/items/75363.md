## Summary

Keep this automerge PR open. The latest head still has two Discord-local P2 regressions: one-off component waits can target the original response after a follow-up reply, and normal Discord writes can be stale-dropped under backlog or rate-limit delay.

## What This Changes

This PR adds Discord REST priority lanes and stale dropping, gateway payload/member-intent/invalid-session hardening, command deploy comparison normalization, component wait helpers, message action component forwarding, tests, and a changelog entry.

## Best Possible Solution

Keep the Discord hardening direction, but repair the component wait helper to wait on the actual created follow-up message and limit stale dropping to background REST work before exact-head review and checks gate automerge.

## Reproduction Assessment

Yes. The source-level repros are clear: call `replyAndWaitForComponent()` after the interaction has already replied, and queue a standard webhook or channel-message write behind a blocked worker or long rate limit for more than 60000 ms.

## Solution Assessment

No. The PR is scoped to the right Discord plugin surface, but the current implementation is not merge-ready until follow-up waits target the created message and standard writes are never aged out as stale background work.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Wait on the created follow-up message:** `extensions/discord/src/internal/interactions.ts:231`
  - body: When `reply()` routes to `followUp()` for an already-replied interaction, the webhook call does not pass `wait: true`, so `result` is not the newly-created message and this falls back to `@original`. The one-off wait is then registered on the old response, and clicks on components in the follow-up can time out.
  - confidence: 0.9
- **[P2] Keep standard writes out of stale dropping:** `extensions/discord/src/internal/rest.ts:74`
  - body: The default `standard` lane now has a 60000 ms stale timeout, while webhook POST/PATCH/DELETE and channel message writes are classified as `standard`. Under backlog or a long rate limit, normal sends, edits, or deletes can be rejected as stale even though only obsolete background REST work should be dropped.
  - confidence: 0.82
