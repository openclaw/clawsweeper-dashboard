## Summary

Keep this PR open. The Discord hardening is in the right owner boundary, but the exact current head still has a merge-blocking one-off component wait bug in the already-replied follow-up path.

## What This Changes

The PR adds Discord REST priority lanes and stale-drop behavior, gateway send validation and invalid-session jitter, command metadata normalization, one-off component wait helpers, focused tests, and a changelog entry.

## Best Possible Solution

Keep the Discord plugin hardening, but make the one-off wait helper register against the actual message containing the components for initial replies, deferred edits, and already-replied follow-ups.

## Reproduction Assessment

Yes. A source-level reproduction is to call replyAndWaitForComponent after the interaction is already replied; reply() takes the follow-up path, followUp() does not use wait: true, and the helper falls back to @original.

## Solution Assessment

No. The Discord-local direction is the right scope, but this implementation is not mergeable until follow-up waits obtain the concrete created message, likely by using webhook wait: true on that path.
