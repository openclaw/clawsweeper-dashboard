## Summary

Keep this PR open. The Discord-local hardening is useful and now has a changelog entry, but the latest head still has two P2 correctness blockers: one-off component waits can attach to the original response after a follow-up, and ordinary Discord writes can be stale-dropped after 60000 ms.

## What This Changes

The PR changes the Discord plugin REST scheduler, gateway validation and reconnect handling, command deploy comparison, interaction component wait helpers, message action component forwarding, focused tests, and CHANGELOG.md.

## Best Possible Solution

Keep the Discord hardening, but repair the component helper to wait on the actual created follow-up message and restrict stale dropping to background REST work before exact-head review and checks gate automerge.

## Reproduction Assessment

Yes. The source-level repros are clear: call `replyAndWaitForComponent()` after the interaction has already replied, and queue a normal webhook or channel-message write behind a blocked worker or long rate limit for more than 60000 ms.

## Solution Assessment

No. The PR is scoped to the right Discord plugin surface, but the current implementation is not merge-ready until follow-up waits target the created message and standard sends, edits, and deletes are not treated as stale background work.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Wait on the created follow-up message:** `extensions/discord/src/internal/interactions.ts:231`
  - body: When `responseState` is already replied, `reply()` sends a follow-up webhook message, but `followUp()` does not pass `wait: true`, so `result` is not the new message and this falls back to `@original`. The one-off wait is then registered on the old response, causing clicks on components in the new follow-up to time out.
  - confidence: 0.9
- **[P2] Keep standard writes out of stale dropping:** `extensions/discord/src/internal/rest.ts:74`
  - body: The default `standard` lane now has a 60000 ms stale timeout, and the classifier puts webhook POST/PATCH/DELETE calls plus channel message writes in that lane. Under backlog or a long rate limit, normal sends, edits, or deletes can be rejected as stale even though this hardening only intends stale dropping for background REST work.
  - confidence: 0.78
