## Summary

Keep open. The latest head is not implemented on current main and still introduces two Discord-local P2 regressions: follow-up component waits can bind to the original reply, and normal Discord writes can be dropped as stale work.

## What This Changes

The PR adds Discord REST priority lanes, gateway send hardening, command comparison normalization, one-off component wait helpers, message-action component forwarding, tests, and a changelog entry.

## Best Possible Solution

Keep the Discord-local hardening, but repair the two queue/wait regressions on the PR branch before exact-head review and CI can gate automerge.

## Reproduction Assessment

Yes. A source-level reproduction is clear: call replyAndWaitForComponent after the interaction has already replied, and queue a standard webhook or channel-message write behind a blocked worker or long rate limit for more than 60000 ms.

## Solution Assessment

No. The PR is scoped to the right Discord plugin surface, but the implementation should wait on the actual created follow-up message and reserve stale dropping for background work only.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Wait on the created follow-up message:** `extensions/discord/src/internal/interactions.ts:231`
  - body: When reply() routes to followUp() for an already-replied interaction, createWebhookMessage is called without wait=true, so this falls back to fetchReply() and registers the one-off wait on @original. Component clicks on the new follow-up can then time out.
  - confidence: 0.9
- **[P2] Keep standard writes out of stale dropping:** `extensions/discord/src/internal/rest.ts:74`
  - body: The default standard lane has a 60000 ms stale timeout, and webhook writes plus channel-message writes are classified as standard. Under backlog or a long rate limit, normal sends, edits, or deletes can be rejected as stale even though only obsolete background REST work should be dropped.
  - confidence: 0.86
