## Summary

Keep open: current `node.list` no longer directly scans agent directories, but the Control UI reconnect path still calls `agents.list`, and that handler synchronously enumerates every on-disk agent directory before filtering configured agents; the requested `doctor --fix` cleanup also remains a policy decision rather than implemented behavior.

## What This Changes

The issue requests that Gateway list RPCs avoid event-loop stalls from orphaned agent directories and that `openclaw doctor --fix` provide a safe cleanup path.

## Best Possible Solution

Split the resolution: avoid unnecessary hot-path agent-directory enumeration or cache/coalesce list work without regressing disk-only session discovery, then design a guarded `doctor --fix` archive/remove flow with explicit data-retention rules.

## Reproduction Assessment

Yes for the remaining source-level stall class: `agents.list` synchronously enumerates the full state `agents` directory during Control UI startup. No live high-confidence reproduction was run for the exact slow-storage 30-48s timing or the historical `node.list` attribution.

## Solution Assessment

No for a `node.list`-only repair. The maintainable path is to bound or avoid hot-path agent-dir discovery while preserving intentional disk-only session discovery, then handle orphan cleanup through an explicitly reviewed doctor repair policy.
