## Summary

Keep open. Current main no longer has `node.list` directly walking agent directories, but the remaining list-RPC stall is not fully solved: `sessions.list` still computes `configuredAgentsOnly` and then loads the combined store without passing that option, so configured-only callers can still pay for broad disk-only agent-store discovery before filtering; the related cache/coalescing PR is open, conflicting, and not a safe canonical fix yet.

## What This Changes

The issue reports that `node.list` and related list RPCs can block the gateway event loop for tens of seconds when many orphaned agent directories remain on disk.

## Best Possible Solution

Keep this as the canonical issue, repair configured-only `sessions.list` loading so default UI/list callers avoid disk-only discovery, preserve explicit broad Gateway/ACP discovery, and make a maintainer decision on whether `doctor --fix` should remove stale agent directories.

## Reproduction Assessment

No live high-confidence reproduction was run. Source inspection is enough to reproduce the remaining current-main path: a configured-only `sessions.list` request still loads broad discoverable agent stores before filtering, but the reported latency magnitude depends on slow storage and many stale directories.

## Solution Assessment

No, this should not be closed as solved yet. The narrow code fix appears to be passing the existing configured-only option into the combined-store loader, while the broader stale-directory cleanup still needs a maintainer compatibility decision.
