## Label Justifications

- `P1`: The original report describes 30-48 second event-loop stalls that made the gateway and Control UI unresponsive on a real host.
- `impact:crash-loop`: The issue is about process-level availability from long event-loop stalls and liveness warnings.
- `impact:session-state`: The affected orphaned agent directories can contain session/auth state, and current doctor behavior treats them as stateful before removal.

## Summary

Close as implemented: current main and v2026.6.6 make the reported Control UI `node.list` and default session-list startup path independent of orphaned per-agent directories; the remaining automatic deletion idea is a separate state-safety product decision.

## What This Changes

The issue requests that `node.list` stop blocking on orphaned per-agent directories and suggests a doctor cleanup path for stale agent dirs.

## Best Possible Solution

Keep the shipped bounded `node.list` and configured-only Control UI session listing, and treat any automatic orphan-dir deletion as a separate explicit state-migration/product decision.

## Reproduction Assessment

No for current main: the historical logs are credible, but current `node.list` no longer scans agent dirs and the Control UI session path defaults to configured agents only. I did not run a live low-power eMMC reproduction.

## Solution Assessment

Yes for the central bug: decoupling `node.list` from agent-dir/session discovery and narrowing default UI session listing is the maintainable hot-path fix. Automatic deletion is not the best default in this issue because orphan dirs can hold auth/session state.
