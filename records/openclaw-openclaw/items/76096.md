## Summary

Keep open: current main still has synchronous orphan-agent directory scans on the Control UI list hot paths, although the exact `node.list` attribution is likely an adjacent-RPC timing artifact because `node.list` itself only reads pairing state and connected-node registry data.

## What This Changes

The issue asks for `node.list`/Control UI list RPCs to avoid blocking on orphaned agent directories and for doctor to offer a safe cleanup path.

## Best Possible Solution

Make agent/session listing bounded or non-blocking while preserving intentional disk-only session visibility, then add a guarded doctor preview/fix flow for stale agent directories rather than deleting by default.

## Reproduction Assessment

Yes for the broader gateway stall: create many `~/.openclaw/agents/<id>` directories with only a few configured agents, then trigger Control UI `agents.list`/`sessions.list`; current main still has synchronous directory enumeration in those paths. The exact `node.list` handler is not a high-confidence reproducer because it does not scan agent directories.

## Solution Assessment

No: changing `node.list` alone would miss the root cause. The safer repair is to bound or cache configured-agent and session-store discovery, yield or async any unavoidable disk work, and add explicit doctor cleanup confirmation.
