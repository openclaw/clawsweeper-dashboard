## Label Justifications

- `P1`: The report describes Gateway list RPCs blocking the event loop for tens of seconds and making the control plane unresponsive for real users on low-power hosts.
- `impact:crash-loop`: The issue is about hangs/event-loop starvation and Gateway liveness failures, which are process availability problems.

## Summary

Keep open. Current main has mitigations around bounded/session-list behavior, and `node.list` itself now reads pairing/node-registry state rather than scanning agent directories directly, but the central list-RPC stall is not fully solved because `sessions.list` still broad-loads discoverable disk-only agent stores synchronously before configured-agent filtering, and the related cache/coalescing PR is still open rather than merged.

## What This Changes

The issue requests that Gateway list RPCs avoid blocking the event loop when stale on-disk agent directories exist, and asks for an automatic doctor cleanup path for orphaned agent directories.

## Best Possible Solution

Keep this issue as the canonical thread, then land a maintainer-reviewed fix that removes or amortizes blocking list-RPC filesystem discovery while preserving intentional disk-only ACP/session compatibility and adding a doctor cleanup path if stale directory removal is the chosen product behavior.

## Reproduction Assessment

No live high-confidence reproduction was run. Source inspection does show a credible current-main path: `sessions.list` still broad-loads synchronously discovered agent session stores before configured-agent filtering, but the reported 30-48s latency depends on slow storage and many stale directories.

## Solution Assessment

No, this should not be closed as solved yet. The best fix needs a maintainer decision around disk-only agent/session compatibility, first-call versus cached-list behavior, and whether `doctor --fix` should remove stale agent directories automatically.
