## Label Justifications

- `P1`: The report describes Gateway list RPCs blocking the event loop for tens of seconds and making the control plane unresponsive for real users on low-power hosts.
- `impact:crash-loop`: The issue is about event-loop starvation and Gateway liveness failures, which are process availability problems.

## Summary

Keep open. Current main moved `node.list` away from direct agent-directory scans, but the list-RPC stall is not fully resolved because `sessions.list` still performs broad synchronous discovered-agent session-store loading before applying the configured-agent filter, and the related cache/coalescing PR is still open and conflicting.

## What This Changes

Prevent Gateway list RPCs from blocking on orphaned agent directories and add a safe cleanup path for stale agent state.

## Best Possible Solution

Keep this as the canonical issue and land a maintainer-reviewed fix that avoids synchronous broad agent-store discovery on list RPCs, preserves intentional disk-only session visibility, and defines whether doctor cleanup is safe.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection does show a credible current-main path: `sessions.list` still synchronously discovers and loads disk-only agent session stores before configured-agent filtering, but the 30-48s stall depends on hardware and directory count.

## Solution Assessment

No, the issue should not be closed as solved yet. The safest solution needs a maintainer decision around disk-only session compatibility, first-call behavior, and whether automatic doctor cleanup should remove stale agent directories.
