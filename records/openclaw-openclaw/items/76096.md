## Summary

Keep open: current `node.list` itself does not scan agent directories, but the Control UI startup flow still calls `agents.list`, which synchronously enumerates all on-disk agent directories before filtering, and `doctor` still warns about orphaned agent directories without offering a guarded repair path.

## What This Changes

Prevent orphaned agent directories from making Control UI list RPCs block the gateway event loop, and add a cleanup path for stale agent directories.

## Best Possible Solution

Keep this open and split the fix into a narrow hot-path change that avoids unnecessary full agent-dir enumeration during `agents.list`, plus a separately reviewed doctor repair flow that archives or removes stale agent dirs only with clear confirmation.

## Reproduction Assessment

Yes for the remaining source-level stall class: `agents.list` synchronously enumerates the full state `agents` directory during the Control UI startup flow. No for the exact reported `node.list` attribution on current main, because the current handler no longer scans agent directories and no slow-storage live repro was run.

## Solution Assessment

No for a `node.list`-only repair. The maintainable path is to bound or avoid hot-path agent-dir discovery while preserving intentional disk-only session discovery, then handle doctor cleanup through an explicit guarded repair policy.
