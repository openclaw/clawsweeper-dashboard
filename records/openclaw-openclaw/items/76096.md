## Summary

Keep open: current `node.list` is not the direct orphan-agent-directory scanner, but current `main` still has synchronous `agents.list` and `sessions.list` discovery paths in the same Control UI startup flow, and `doctor` still only warns about orphaned agent directories instead of offering a guarded repair.

## What This Changes

The issue requests making Control UI list RPCs avoid scanning orphaned `~/.openclaw/agents/*` directories synchronously and adding a `doctor --fix` cleanup path for stale agent directories.

## Best Possible Solution

Make agent/session discovery bounded, cached, async/yielded, or scoped while preserving intentional disk-only session visibility, then add a guarded doctor repair/cleanup flow for stale agent directories.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not for the exact `node.list` attribution on current `main`, because the current handler does not scan agent directories. The broader Control UI stall class is source-reproducible through synchronous `agents.list` and `sessions.list` directory/store discovery in the same startup flow.

## Solution Assessment

Is this the best way to solve the issue? No for a `node.list`-only repair. The maintainable fix belongs in shared agent/session discovery plus guarded doctor cleanup, with regression coverage that preserves disk-only ACP/session-store visibility.
