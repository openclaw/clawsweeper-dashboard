## Summary

Keep open. The default-off TLS repair direction is plausible for #74848, but the new `gateway.remote.allowSystemTrustedPinRepair` option is only read from raw macOS config and is missing from the strict shared config contract, so enabling it would make canonical config invalid.

## What This Changes

The PR adds a default-off macOS companion opt-in for arbitrary system-trusted `wss://` stale TLS pin auto-repair, plus Swift tests and a changelog entry.

## Best Possible Solution

Make `gateway.remote.allowSystemTrustedPinRepair` a first-class, default-false config field across schema, exported types, help/labels, generated baselines, docs, and validation tests while preserving the strict default behavior.

## Reproduction Assessment

Yes. Current main source and tests show trusted public-host pin mismatches are rejected by default, and the PR-specific defect is reproducible by comparing the raw config read with the strict `gateway.remote` schema that omits the new key.

## Solution Assessment

No. The default-off opt-in is safer than broad default repair, but the implementation is incomplete until the new option is schema-valid and documented.
