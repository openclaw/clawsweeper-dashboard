## Summary

Keep open. The PR targets a real macOS stale TLS pin reconnect failure, but the current branch adds a security-sensitive opt-in config key that is read only by the macOS app and is not registered in the shared strict config schema/types/help surface, so an operator who sets it can break normal OpenClaw config validation. Inspectable real behavior proof and maintainer acceptance of the TLS trust-boundary tradeoff are still needed before merge.

## What This Changes

The PR adds a default-off macOS config opt-in for stale `wss://` gateway TLS pin auto-repair on system-trusted public hosts and adds focused Swift unit coverage.

## Best Possible Solution

Land this only as a default-off, documented opt-in after the shared config schema/types/help/docs accept the key, focused validation coverage proves the config is valid, and maintainers have proof for both the rotating-public-cert recovery path and the strict private/self-signed mismatch path.

## Reproduction Assessment

Yes, at source level. Current main only auto-repairs stale pins for loopback and `.ts.net` hosts, so a stored-pin mismatch on another public `wss://` host remains on the strict failure path; I did not run a live macOS certificate-rotation scenario in this read-only review.

## Solution Assessment

No. A default-off opt-in is a plausible mitigation, but this branch is not the best mergeable form until the new config key is part of the shared strict config contract and the TLS trust tradeoff has real macOS proof.
