## Summary

Keep open: the default-off opt-in is a safer direction than broad automatic repair, but the PR still introduces a new `gateway.remote` config key without registering it in the strict config contract, so the advertised setting would be invalid or invisible to schema consumers.

## What This Changes

The PR adds a macOS companion opt-in config flag for system-trusted arbitrary `wss://` stale TLS pin auto-repair, wires it into the stale-pin helper, adds Swift coverage, and updates the changelog.

## Best Possible Solution

Keep the default loopback/`.ts.net` behavior, and land the arbitrary-host system-trust fallback only as a first-class documented config key across schema, types, help, generated metadata, and tests.

## Reproduction Assessment

Yes. Current main source and tests show trusted `wss://gateway.example.com` pin mismatches are rejected, and the PR-specific defect is source-reproducible by comparing the new raw config read with the strict shared `gateway.remote` schema.

## Solution Assessment

No. The default-off opt-in is the right shape for the TLS trade-off, but this implementation is incomplete until the new key is registered in the canonical config contract instead of being read only by the macOS helper.
