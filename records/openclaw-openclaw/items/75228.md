## Summary

Keep open. The PR addresses the prior security concern by making arbitrary system-trusted remote pin repair default-off, but it is still incomplete because the new `gateway.remote.allowSystemTrustedPinRepair` flag is read by the macOS app while the canonical `gateway.remote` config schema and metadata do not define it. That makes the advertised opt-in invalid or invisible for schema-backed config tooling.

## What This Changes

This PR adds a default-off macOS `gateway.remote.allowSystemTrustedPinRepair` path for stale TLS pin auto-repair on arbitrary system-trusted `wss://` hosts, plus Swift coverage and a changelog entry.

## Best Possible Solution

Keep the default loopback/`.ts.net` repair boundary, and if maintainers accept the opt-in policy, make `gateway.remote.allowSystemTrustedPinRepair` a first-class config field across schema, exported types, help/labels, generated config artifacts, public docs, and validation coverage before merging.

## Reproduction Assessment

Yes. The original behavior is reproducible at code level because current main returns false for a non-loopback, non-`.ts.net` `wss://` pin mismatch with `systemTrustOk: true`; the PR defect is reproducible by comparing the new key read on the PR head with the strict current-main `gateway.remote` schema that omits it.

## Solution Assessment

No, not yet. The default-off opt-in is a safer direction than broadening repair by default, but the implementation is not merge-ready until the new config flag is supported by the canonical config contract and docs surfaces.
