## Summary

Keep open: the PR now uses a safer default-off opt-in for arbitrary system-trusted `wss://` stale-pin repair, but the branch still reads a new config key that is absent from the strict canonical config contract, so the advertised opt-in would make normal config validation fail.

## What This Changes

The PR adds a macOS companion `gateway.remote.allowSystemTrustedPinRepair` opt-in for stale TLS pin repair on arbitrary system-trusted `wss://` hosts, plus Swift tests and a changelog entry.

## Best Possible Solution

Keep the default loopback/`.ts.net` behavior and land the opt-in only after the new key is registered across schema, exported types, help/labels, generated config artifacts, docs, and validation coverage.

## Reproduction Assessment

Yes. Current main source and tests show trusted `wss://gateway.example.com` pin mismatches are rejected, and the PR-specific defect is source-reproducible by comparing the new raw config read with the strict shared `gateway.remote` schema.

## Solution Assessment

No. A default-off opt-in is safer than broad default repair, but this implementation is incomplete until the new key is a first-class documented config contract field.
