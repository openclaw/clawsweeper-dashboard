## Summary

Keep open: this PR addresses a source-reproducible macOS stale TLS pin reconnect loop, but the current branch reads a new `gateway.remote.allowSystemTrustedPinRepair` key without registering it in the strict config contract, so the advertised opt-in would make user config invalid.

## What This Changes

The PR adds a macOS companion opt-in config flag for auto-repairing stale TLS pins on arbitrary system-trusted `wss://` gateway hosts, with Swift tests and a changelog entry.

## Best Possible Solution

Keep the default loopback/`.ts.net` policy, and land the opt-in only after it is a first-class documented config key with schema, types, help metadata, generated baselines, docs, and validation coverage.

## Reproduction Assessment

Yes. Current main source and tests show trusted `wss://gateway.example.com` pin mismatches are rejected, and the PR-specific defect is source-reproducible by comparing the raw config read with the strict `gateway.remote` schema.

## Solution Assessment

No. A default-off opt-in is a maintainable direction for this TLS trade-off, but this branch is incomplete until the new key is registered and documented in the canonical config contract.
