## Summary

Keep open. The PR has moved the broader TLS pin repair behind a safer default-off opt-in, but the new config key is read from raw macOS config while the canonical `gateway.remote` schema/type/help surfaces still do not declare it, so the advertised setting would be schema-invalid for operators.

## What This Changes

The PR adds a default-off macOS companion config opt-in for arbitrary system-trusted `wss://` stale TLS pin auto-repair, adds Swift coverage, and updates the changelog.

## Best Possible Solution

Keep the default loopback/`.ts.net` repair boundary, and make the arbitrary-host fallback a documented, schema-valid, default-false config key across the shared contract, docs, generated baselines, and validation tests before merge.

## Reproduction Assessment

Yes. Current main source shows trusted `wss://gateway.example.com` pin mismatches are rejected while loopback and `.ts.net` are allowed; the PR-specific defect is reproducible by comparing its raw config read with the strict `gateway.remote` schema that omits the key.

## Solution Assessment

No. The default-off opt-in is the safer direction compared with broad default repair, but this implementation is incomplete until `gateway.remote.allowSystemTrustedPinRepair` is first-class in the config contract and docs.
