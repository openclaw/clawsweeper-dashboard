## Summary

Keep open. The default-off opt-in is a safer direction than broad stale-pin repair and addresses the macOS logic for #74848, but the PR reads and advertises a new `gateway.remote` config key that is absent from the strict shared config contract, generated schema metadata, and help surfaces.

## What This Changes

The PR adds a default-off macOS config flag to auto-repair stale gateway TLS pins for system-trusted arbitrary `wss://` hosts, with macOS unit coverage and a changelog entry.

## Best Possible Solution

Keep the default loopback/`.ts.net` behavior, and expose arbitrary-host repair only as a documented, schema-valid, default-false config key with generated schema/docs metadata and validation coverage.

## Reproduction Assessment

Yes. Current main has source-level coverage showing trusted `wss://gateway.example.com` pin mismatches are not auto-repaired, and the PR-specific defect is reproducible by comparing the new raw config read with the strict `gateway.remote` schema that omits the key.

## Solution Assessment

No. The opt-in approach is narrower than broad default repair, but this implementation is incomplete until `gateway.remote.allowSystemTrustedPinRepair` is first-class in the shared config contract and docs.
