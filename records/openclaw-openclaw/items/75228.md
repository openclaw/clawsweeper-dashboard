## Summary

Keep open. The default-off opt-in direction is safer than silently broadening TLS pin repair, but the branch still reads and advertises a new `gateway.remote.allowSystemTrustedPinRepair` key without registering it in the strict shared config contract, so users who enable it would create an invalid or unsupported config surface.

## What This Changes

The PR adds a default-off macOS companion opt-in for system-trusted arbitrary `wss://` stale TLS pin auto-repair, adds Swift coverage for the opt-in boundary, and updates the changelog.

## Best Possible Solution

Keep the default loopback/`.ts.net` repair boundary, and make the arbitrary-host system-trust fallback a documented, schema-valid, default-false operator opt-in across the canonical config schema, exported type, UI help/labels, generated docs baselines, public remote/TLS docs, and validation tests.

## Reproduction Assessment

Yes. Source inspection on current main shows trusted `gateway.example.com` pin mismatches are rejected while `.ts.net` and loopback are allowed; the PR-specific defect is reproducible by comparing its new raw-config read with the strict `gateway.remote` schema that omits the key.

## Solution Assessment

No. The default-off opt-in is the right safer shape compared with broad default repair, but this implementation is incomplete until `gateway.remote.allowSystemTrustedPinRepair` is a first-class config key accepted and documented by the shared contract.
