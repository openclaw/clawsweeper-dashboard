## Summary

Keep open: current main still limits stale TLS pin auto-repair to loopback and `*.ts.net`, and the linked public-host recovery path remains a real macOS policy question. The PR is not obsolete, but it still has a blocking config-contract bug, missing real behavior proof, and a TLS trust tradeoff that needs maintainer/security acceptance before merge.

## What This Changes

The PR adds a default-off macOS config flag for stale TLS pin repair on arbitrary system-trusted WSS hosts, plus Swift tests and a changelog entry.

## Best Possible Solution

Register and document the opt-in through the canonical config contract, route it through the current remote-gateway config helper, preserve strict defaults, and merge only after maintainer/security acceptance plus redacted real macOS proof.

## Reproduction Assessment

Yes for source-level reproduction: current main has a trusted public WSS pin-mismatch test that is not auto-repaired, and the TLS delegate cancels stored-pin mismatches. A live reproduction would need a macOS app install with a stale stored pin and a rotated public certificate.

## Solution Assessment

No. A default-off opt-in is a plausible direction, but this patch is incomplete until the config key is registered across schema/types/help/docs and maintainers accept the TLS trust boundary.
