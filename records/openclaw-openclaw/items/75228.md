## Summary

Keep open. The PR addresses a real macOS stale TLS pin recovery gap, and the default-off opt-in is safer than broadening auto-repair by default, but the branch still reads a new public config key that is not registered in the strict shared config schema/type/help/docs contract.

## What This Changes

The PR adds a default-off macOS opt-in for system-trusted stale TLS pin auto-repair on arbitrary `wss://` hosts, wires it into `MacNodeModeCoordinator`, adds Swift coverage, and updates the changelog.

## Best Possible Solution

Keep the default loopback/`.ts.net` repair boundary, and if maintainers accept the opt-in policy, make `gateway.remote.allowSystemTrustedPinRepair` a first-class config key across the shared schema, exported type, help/labels, generated schema/docs baselines, public remote/TLS docs, and validation coverage before merging.

## Reproduction Assessment

Yes. Current main has a code-level reproduction showing a non-loopback, non-`.ts.net` trusted `wss://` pin mismatch returns false; the PR defect is reproducible by source inspection because the branch reads a key that the strict `gateway.remote` schema does not accept.

## Solution Assessment

Not yet. The default-off opt-in is the right safer direction compared with silent broadening, but this implementation is incomplete until the new config setting is supported by the canonical config contract and docs surfaces.
