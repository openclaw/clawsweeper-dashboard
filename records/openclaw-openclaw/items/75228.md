## Summary

Keep open. The PR is useful and the latest head addressed the earlier broad-default TLS concern by adding a default-off opt-in, but it is not merge-ready because the new `gateway.remote.allowSystemTrustedPinRepair` key is read from macOS raw config while the shared `gateway.remote` config schema, exported type, help, labels, generated schema metadata, and public docs still omit it. OpenClaw config validation is strict, so the advertised operator flag would be schema-invalid or invisible to canonical config consumers.

## What This Changes

The PR adds a default-off macOS config flag for stale TLS pin auto-repair on arbitrary system-trusted `wss://` hosts, wires it into macOS node-mode repair, adds Swift coverage, and updates the changelog.

## Best Possible Solution

Register `gateway.remote.allowSystemTrustedPinRepair` as a first-class, optional, default-false config key across the shared schema, exported type, help/labels, generated schema/docs metadata, public remote/TLS docs, and validation tests, while keeping the default loopback/`.ts.net` auto-repair behavior unchanged.

## Reproduction Assessment

Yes. Source inspection on current main reproduces the original gap because trusted arbitrary public hosts are rejected by `shouldAutoRepairStaleTLSPin`, and the PR-specific blocker is reproducible by comparing the PR's new raw-config read with the strict `gateway.remote` schema that omits the key.

## Solution Assessment

No, not as implemented. The default-off opt-in is the safer shape compared with broad default repair, but the implementation is incomplete until the flag is accepted and documented by the canonical config contract.
