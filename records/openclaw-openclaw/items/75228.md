## Summary

Keep open. The opt-in approach is safer than broad default stale-pin repair and the macOS helper/tests match the intended behavior, but the PR introduces `gateway.remote.allowSystemTrustedPinRepair` only as a raw macOS config read while the shared strict config contract and generated metadata still omit that key.

## What This Changes

The PR adds a default-off macOS config flag to auto-repair stale gateway TLS pins for system-trusted arbitrary `wss://` hosts, with Swift test coverage and a changelog entry.

## Best Possible Solution

Keep the default loopback/`.ts.net` behavior, and expose arbitrary-host repair only as a documented, schema-valid, default-false config key with generated metadata and validation coverage.

## Reproduction Assessment

Yes. Current main source shows trusted `wss://gateway.example.com` pin mismatches are not auto-repaired, and the PR-specific defect is reproducible by comparing the new raw config read with the strict `gateway.remote` schema that omits the key.

## Solution Assessment

No. The opt-in direction is narrower than broad default repair, but this implementation is incomplete until `gateway.remote.allowSystemTrustedPinRepair` is first-class in the shared config contract and docs.
