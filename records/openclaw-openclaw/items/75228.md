## Summary

Keep open: the PR is a plausible macOS stale TLS pin recovery path, but the current head still reads a new `gateway.remote.allowSystemTrustedPinRepair` key that is absent from the strict config schema, exported types, labels/help, and docs. The contributor also needs to add visible after-fix real behavior proof before merge.

## What This Changes

The PR adds a default-off `gateway.remote.allowSystemTrustedPinRepair` opt-in for macOS stale TLS pin auto-repair on arbitrary system-trusted WSS hosts, plus Swift tests and a changelog entry.

## Best Possible Solution

Keep the default loopback/`*.ts.net` behavior, register and document the opt-in across the canonical config contract, then merge only after redacted real macOS proof shows the intended recovery and strict-path behavior.

## Reproduction Assessment

Yes for source-level reproduction. Current main has a test and source path where trusted public WSS pin mismatches are not auto-repaired, and the PR-specific defect is reproducible by comparing the new raw config read with the strict shared schema.

## Solution Assessment

No. A default-off opt-in is safer than broad default repair, but this implementation is incomplete until the key is registered in schema/types/help/docs and maintainers accept the TLS trust tradeoff.
