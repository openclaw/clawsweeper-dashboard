## Summary

Keep open. The PR addresses a real macOS remote TLS-pin pain point and the latest branch moves the risky behavior behind a default-off opt-in, but it still reads a new public config key that current main's strict shared config contract does not accept or document.

## What This Changes

The PR adds a macOS config opt-in for system-trusted stale TLS pin auto-repair on arbitrary `wss://` hosts, wires it into stale-pin repair, adds Swift coverage, and updates the changelog.

## Best Possible Solution

Keep the default loopback/`.ts.net` repair boundary, and if maintainers accept the opt-in policy, make `gateway.remote.allowSystemTrustedPinRepair` a first-class config key across schema, exported type, help/labels/hints, generated baselines, public remote/TLS docs, and validation coverage before merging.

## Reproduction Assessment

Yes. Current main has a code-level reproduction where a trusted non-loopback, non-`.ts.net` `wss://` pin mismatch returns false; the PR defect is reproducible by source inspection because the branch reads a key that the strict shared `gateway.remote` schema does not accept.

## Solution Assessment

Not yet. The default-off opt-in is the safer direction compared with silently broadening TLS pin repair, but the implementation is incomplete until the new config setting is supported by the canonical config contract and docs surfaces.
