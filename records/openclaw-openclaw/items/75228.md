## Summary

Keep open; current main still lacks the requested public-host stale-pin recovery, and this PR still has a merge-blocking config-contract gap after moving the behavior behind an opt-in.

## What This Changes

The branch adds a default-off `gateway.remote.allowSystemTrustedPinRepair` read in the macOS companion, threads it into stale TLS pin auto-repair, adds Swift tests, and updates the changelog.

## Best Possible Solution

Keep the default loopback/`.ts.net` repair boundary, make the arbitrary-host fallback a documented schema-valid default-off config key across shared config surfaces, and then have a maintainer accept or reject the TLS opt-in policy.

## Reproduction Assessment

Yes. Source inspection of current main shows trusted `wss://gateway.example.com` pin mismatches are rejected, and the PR-specific defect is reproducible from the branch diff because it reads a new raw config key that the strict shared schema does not declare.

## Solution Assessment

No. The default-off opt-in is safer than broad default repair, but this implementation is not the best mergeable shape until the new key is first-class in the config contract, docs, generated metadata, and validation tests.
