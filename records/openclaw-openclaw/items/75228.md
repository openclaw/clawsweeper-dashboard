## Summary

Keep open: the updated PR is in the safer default-off direction, but it still exposes a new macOS config opt-in only through the Swift reader while the shared strict config contract omits that key.

## What This Changes

The PR adds a default-off macOS companion config opt-in for system-trusted arbitrary `wss://` stale TLS pin auto-repair, plus Swift tests and a changelog entry.

## Best Possible Solution

Keep the default loopback/`.ts.net` policy, and make `gateway.remote.allowSystemTrustedPinRepair` a first-class documented config key across schema, exported types, help/labels, generated schema baselines, and validation tests before merge.

## Reproduction Assessment

Yes. Current main source and tests show trusted `wss://gateway.example.com` pin mismatches are rejected, and the PR-specific defect is source-reproducible by comparing the new raw config read with the strict shared `gateway.remote` schema.

## Solution Assessment

No. A default-off opt-in is safer than broad default repair, but this implementation is incomplete until the new key is registered and documented in the canonical config contract.
