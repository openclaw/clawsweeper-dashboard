## Summary

Keep open. The PR is useful for the linked macOS stale TLS pin bug, and the latest default-off opt-in avoids silently broadening the current repair policy. It is still incomplete because it reads and advertises `gateway.remote.allowSystemTrustedPinRepair` while the strict shared config schema, exported type, help/label metadata, and generated docs surfaces do not accept that key, so the advertised opt-in would make operator config invalid.

## What This Changes

The PR adds a default-off macOS `gateway.remote.allowSystemTrustedPinRepair` opt-in for repairing stale system-trusted remote `wss://` gateway TLS pins, plus Swift tests and a changelog entry.

## Best Possible Solution

Keep the current default loopback/`.ts.net` repair boundary, and land the opt-in only after `gateway.remote.allowSystemTrustedPinRepair` is a first-class config field across schema, exported types, help/labels, generated schema/docs baselines, public remote/macOS docs, and validation tests with the security trade-off stated clearly.

## Reproduction Assessment

Yes. Current main has a code-level reproduction for the linked bug because non-loopback, non-`.ts.net` `wss://` pin mismatches return false even with `systemTrustOk: true`; the PR blocker is reproducible by validating a config with `gateway.remote.allowSystemTrustedPinRepair` against the strict `gateway.remote` schema that lacks that property.

## Solution Assessment

No, not yet. A default-off opt-in is a narrower and safer direction than silently broadening stale-pin repair, but the PR is not complete until the new public config key is registered and documented through the canonical config contract.
