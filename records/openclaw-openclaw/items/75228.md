## Summary

Keep open. The PR is still relevant because current main and the latest release only auto-repair stale macOS gateway TLS pins for loopback and `.ts.net`, but the branch is not merge-ready because it adds a security-sensitive operator config key outside the shared config contract and lacks real macOS after-fix proof.

## What This Changes

The PR adds a default-off macOS config flag, `gateway.remote.allowSystemTrustedPinRepair`, so stale public `wss://` gateway TLS pins can be auto-repaired when the new chain is system-trusted.

## Best Possible Solution

Land a rebased version only after the opt-in is part of the shared config contract and maintainers have real macOS proof for both the repair and strict non-repair paths.

## Reproduction Assessment

Yes at source level: current main only repairs loopback and `.ts.net` stale pins, while the linked issue reports a public `wss://` host. I did not run a live macOS certificate-rotation scenario in this read-only review.

## Solution Assessment

No, not in its current form. A default-off opt-in is a plausible mitigation, but it is not the best mergeable OpenClaw fix until the config contract is aligned and the security-sensitive macOS behavior has real proof.
