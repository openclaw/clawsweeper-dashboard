## Summary

Keep open. Current main has already flattened the Twitch config JSON Schema and documents both single-account and multi-account setup, but the reported gateway restart loop still matches the source: Twitch `startAccount` awaits a monitor that returns immediately after registering handlers, and the shared channel manager treats any resolved channel task as a clean exit and schedules restarts.

## What This Changes

The issue reports that Twitch can authenticate after an update but never stabilizes, with config-schema friction, incomplete setup diagnostics, and `openclaw status --deep` showing the channel as enabled but never successfully connected.

## Best Possible Solution

Keep this issue open while #74773 is reviewed or closed, and land a Twitch-owned lifecycle fix that keeps the monitor task pending until abort or explicit stop while preserving the flat schema work already on main.

## Reproduction Assessment

Yes. There is a high-confidence source-level reproduction: a successful Twitch monitor setup returns from `monitorTwitchProvider`, and the shared gateway manager records a clean channel exit and restart-pending state when `startAccount` resolves.

## Solution Assessment

No. Current main uses the right schema direction, but the remaining restart loop is best solved narrowly in the Twitch plugin lifecycle rather than by weakening the shared gateway clean-exit restart behavior.
