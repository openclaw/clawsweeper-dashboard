## Summary

Keep open. The Twitch schema/docs part is already fixed on current main and in v2026.5.7, but the reported restart loop still matches current source: the Twitch monitor returns after handler setup and the shared channel manager treats that resolved task as a clean exit that enters auto-restart. The linked schema-only PR https://github.com/openclaw/openclaw/pull/74773 should not close this issue unless the remaining lifecycle failure is also addressed.

## What This Changes

The issue reports that the bundled Twitch channel can authenticate or appear enabled after an update but never reaches a stable connected state, with schema friction and weak diagnostics during setup.

## Best Possible Solution

Keep the shipped schema/docs fix, resolve the schema-only PR explicitly, and land a Twitch-owned lifecycle fix that keeps the gateway task pending until abort/disconnect while surfacing real connect and join failures through status diagnostics.

## Reproduction Assessment

Yes. There is a high-confidence source-level path: Twurple connect() is non-blocking, Twitch monitorTwitchProvider returns immediately after handler registration, and the shared channel manager restarts any non-manual resolved channel task.

## Solution Assessment

No. The current schema fix is useful but not sufficient; the narrow maintainable fix is in the Twitch plugin lifecycle, not in weakening the shared clean-exit restart policy.
