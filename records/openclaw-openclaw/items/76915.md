## Summary

Keep open: current main still has a source-reproducible path that can suppress post-message-tool assistant text, and the linked fix PR is closed unmerged. The remaining question is policy-sensitive because current docs and tests intentionally keep some same-target final replies private after a message-tool send.

## What This Changes

The issue requests delivery of assistant final text generated after a successful same-turn message tool send to the originating messaging channel.

## Best Possible Solution

Define one explicit same-turn policy, then implement it consistently in both streaming dedupe and final-payload suppression with regression coverage and docs updates.

Yes from source inspection: current main still contains both the reverse-substring suppression path and the same-target final-payload suppression path. I did not run a live Telegram reproduction, so this is source-reproducible rather than runtime-reproduced.

## Solution Assessment

Unclear. Delivering every post-tool final reply is not clearly the best fix because current message-tool-only docs and tests intentionally make some final text private; a safer fix needs a maintainer decision on duplicate versus distinct-commentary semantics.
