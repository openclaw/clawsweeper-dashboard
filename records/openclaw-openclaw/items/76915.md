## Summary

Keep open: current main still has a source-reproducible path that suppresses post-message-tool assistant text for the same channel target, but the requested behavior conflicts with the documented message-tool-only/private-final policy and needs maintainer product judgment before a repair PR.

## What This Changes

The issue asks OpenClaw to deliver distinct assistant final text to the originating messaging channel even after a successful `message(action=send)` tool call in the same turn.

## Best Possible Solution

Define one explicit policy for same-turn message-tool sends: either keep all channel-visible context inside the message tool payload/caption, or deliver only distinct nonduplicate final text while preserving duplicate suppression and documenting the mode differences.

## Reproduction Assessment

Yes from source inspection, but not from a live Telegram run: the current reply builder suppresses all final payloads after a same-target successful message-tool send, and an existing unit test covers a different final text being dropped.

## Solution Assessment

Unclear. Delivering every post-tool final reply is not clearly the best fix because current docs and prompts intentionally make message-tool-only final text private; a safer solution needs a maintainer decision on duplicate-vs-additional-text semantics.
