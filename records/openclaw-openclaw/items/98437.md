## Summary

Keep open: current main and the latest release still emit the MCP loopback conflict warning from the schema projection path, and the only directly related PR is still open, conflicting, and proof-blocked rather than a safe canonical fix.

## What This Changes

The issue asks OpenClaw to stop logging one MCP loopback schema-conflict warning per repeated bundle/session initialization and instead aggregate or deduplicate the diagnostic while preserving first-defined schema behavior.

## Best Possible Solution

Implement a focused Gateway MCP schema-projection or cache-lifecycle fix that preserves first-defined schema semantics while aggregating or deduplicating conflict diagnostics, with regression coverage and real loopback proof that repeated tool listing no longer floods logs.

## Reproduction Assessment

Yes for the source path, but not for the exact production count. Current main still warns on repeated non-mergeable MCP loopback union-schema field conflicts, and the loopback runtime rebuilds schemas on context/cache misses.

## Solution Assessment

Yes, the Gateway MCP schema projection/cache layer is the narrowest maintainable place to solve this. Renaming every field would be broader and compatibility-sensitive, while lifecycle-scoped aggregation or dedupe keeps the diagnostic signal without per-init spam.
