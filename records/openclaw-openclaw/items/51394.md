## Summary

Keep open: current main still lacks an agent-facing browser Network/intercept surface. The browser control server and CLI already expose passive request listing and response-body capture, but the requested interception capability remains a new security-sensitive API design.

## What This Changes

Request to expose CDP Network-domain-style request listing, request/response monitoring, and URL-pattern interception through the browser tool.

## Best Possible Solution

Add a narrow agent-facing passive request/response inspection surface in the browser plugin, and handle true interception as a separately scoped design with safety policy and regression tests.

## Reproduction Assessment

Yes. Source inspection on current main shows BrowserToolSchema and createBrowserTool do not accept network, requests, responsebody, or intercept actions, while lower-level /requests and /response/body routes exist.

## Solution Assessment

Unclear as proposed. Passive inspection can reuse the existing browser control routes, but raw CDP Network or interception exposure is broader than necessary and needs maintainer API/security design first.

## Security Review

Status: needs_attention

Summary: The issue requests browser traffic interception, which needs an explicit safety model before agent exposure.
