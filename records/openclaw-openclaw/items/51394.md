## Summary

Keep open: current main and v2026.5.22 still do not expose request listing, response-body lookup, CDP Network, Fetch, or interception through the agent-facing browser tool. The lower-level browser control API already has passive request and response-body routes, so the remaining work is a security-sensitive model-facing API decision rather than cleanup to close.

## What This Changes

The issue requests an agent-facing browser tool API for CDP/network request capture, response monitoring, and request interception during browser automation.

## Best Possible Solution

Expose a narrow passive request and response-body inspection surface through the browser tool first, then design any mutating CDP/Fetch interception separately with explicit security policy, redaction, scoping, and regression coverage.

## Reproduction Assessment

Yes for the source-level feature gap: BrowserToolSchema and createBrowserTool still do not accept or route network, requests, responsebody, CDP, Fetch, or intercept actions, while lower-level /requests and /response/body routes exist. I did not run a live browser session because this was a read-only review.

## Solution Assessment

Unclear as proposed. Passive inspection can reuse existing control routes, but raw CDP Network or mutating interception is broader than necessary and needs maintainer API/security design first.
