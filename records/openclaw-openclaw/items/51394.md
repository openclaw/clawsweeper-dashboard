## Summary

Keep open: current main and the latest release still do not expose agent-facing browser-tool actions for request listing, response bodies, CDP Network, Fetch, or interception. The lower-level browser-control API already has passive request and response-body routes, so the remaining work is a new security-sensitive model-facing API decision rather than cleanup to close.

## What This Changes

Request to expose browser-tool network inspection and request-interception capabilities so agents can discover SPA API traffic without page-side fetch hooks.

## Best Possible Solution

Expose a narrow passive request and response-body inspection action through the browser tool first, then design mutating CDP/Fetch interception separately with explicit security policy, redaction, scoping, and regression tests.

## Reproduction Assessment

Yes for the source-level feature gap: BrowserToolSchema and createBrowserTool still do not accept or route network, requests, responsebody, CDP, Fetch, or intercept actions, while lower-level /requests and /response/body routes exist. I did not run a live browser session because this was a read-only review.

## Solution Assessment

Unclear as proposed. Passive inspection can reuse existing control routes, but raw CDP Network or mutating interception is broader than necessary and needs maintainer API/security design first.
