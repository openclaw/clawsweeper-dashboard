## Summary

Keep open: current main and the latest release still lack a model-facing browser tool action for CDP Network interception, while the existing browser control/CLI paths only provide read-only request listing and response-body capture. The remaining request is a new security-sensitive browser capability that needs maintainer product and security design.

## What This Changes

Request to expose CDP Network-domain request capture/interception through the model-facing browser tool for SPA endpoint discovery and debugging.

## Best Possible Solution

Design this as a browser-plugin network inspection capability with explicit sensitive-data policy, profile capability rules, and wrapped/redacted model-facing output, reusing the existing `/requests` and `/response/body` paths where they fit.

## Reproduction Assessment

Yes. Source inspection reproduces the gap: the model-facing browser tool schema and act-kind union do not expose network, requests, responsebody, or intercept actions, while lower control routes only expose read-only request listing and response-body capture.

## Solution Assessment

## Risks / Open Questions

- A model-facing Network interception API can expose cookies, authorization headers, tokens, private endpoints, and response bodies unless redaction, permissions, profile restrictions, and output wrapping are designed first.

- Managed, raw/remote CDP, sandbox, node-hosted, and existing-session profiles may need different support levels; a single action shape could overexpose data or fail inconsistently.
