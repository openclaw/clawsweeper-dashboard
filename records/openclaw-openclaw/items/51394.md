## Summary

Keep open: current main and the latest release already have read-only browser request listing and response-body capture on the control/CLI path, but the requested model-facing browser-tool/CDP Network interception surface is still missing and needs product/security design before implementation.

## What This Changes

The issue requests a browser-tool API for enabling CDP Network capture/interception and listing captured requests during browser automation.

## Best Possible Solution

Keep this open for maintainer product/security design of a browser-plugin network surface that reuses the existing `/requests` and `/response/body` pieces where appropriate, and only adds interception with explicit sensitive-data handling and profile capability rules.

## Reproduction Assessment

Yes, source inspection reproduces the gap: the browser tool schema and action executor do not expose `network`, `requests`, `responsebody`, or `intercept`, while the lower control API only exposes read-only request listing and response-body capture. No live browser run was needed to prove the missing model-facing API surface.

## Solution Assessment

No, not as proposed yet: the idea is valid, but `browser act kind=network` should not be accepted without a maintainer-designed security boundary, profile capability matrix, and sensitive-output policy. A safer implementation would extend the browser plugin/control API deliberately and then expose only the approved subset to the model-facing tool.

## Risks / Open Questions

- A model-facing Network interception API can expose cookies, authorization headers, private endpoints, and response bodies unless redaction, permissioning, profile limits, and output wrapping are explicitly designed.
- The current control API can list requests and wait for a response body, but active interception or mutation of requests is a different capability with session-state and security implications.
- Existing-session, local-managed, remote CDP, sandbox, and node-hosted browser profiles may need different support levels; a one-size API could silently fail or overexpose data on some profiles.
