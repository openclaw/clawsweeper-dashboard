## Summary

Keep open: current main and the latest release still do not expose an agent-facing browser Network/intercept action. Passive request listing and response-body capture exist in the lower-level browser control API and CLI, but mutating request interception is a new browser-tool feature with unresolved product and security policy.

## What This Changes

The issue asks OpenClaw to expose CDP Network-domain capabilities through the browser tool so agents can inspect request activity and intercept matching browser requests.

## Best Possible Solution

Expose a narrow passive request/response inspection action through the browser tool first, and treat mutating CDP/Fetch interception as a separately designed API with target scoping, permission policy, SSRF interaction, and regression tests.

## Reproduction Assessment

Yes for the source-level feature gap. Current BrowserToolSchema and createBrowserTool do not accept network, requests, responsebody, intercept, CDP, or Fetch actions, while lower-level /requests and /response/body routes exist.

## Solution Assessment

Unclear as proposed. Passive inspection can reuse the existing control routes, but raw CDP Network or request interception exposure is broader than necessary and needs maintainer API/security design first.
