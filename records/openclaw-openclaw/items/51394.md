Keep open: current main and the latest release provide browser control/CLI request listing and response-body capture, but the requested model-facing browser tool API for CDP Network interception is still missing and needs maintainer product/security design.

## What This Changes

Request to expose CDP Network request inspection and interception through the model-facing browser tool for SPA debugging and API endpoint discovery.

## Best Possible Solution

Design this as a browser-plugin network capability with explicit sensitive-data policy, profile capability rules, and model-facing output wrapping, reusing the existing `/requests` and `/response/body` pieces where they fit.

## Reproduction Assessment

Yes. Source inspection reproduces the gap: the model-facing browser tool schema and act-kind union do not expose network, requests, responsebody, or intercept actions, while lower control routes only expose read-only request listing and response-body capture.

## Solution Assessment

No, not as proposed yet. `browser act kind=network` should wait for a maintainer-designed security boundary and profile capability matrix rather than directly exposing CDP Network interception to agents.

## Risks / Open Questions

- A model-facing Network interception API can expose cookies, authorization headers, tokens, private endpoints, and response bodies unless redaction, permissions, profile restrictions, and output wrapping are designed first.
- The existing CLI/control request listing and response-body routes are read-only; active interception or request mutation is a stronger capability with different security and session-state implications.
- Managed, raw/remote CDP, sandbox, node-hosted, and existing-session browser profiles may need different support levels, so a single action shape could overexpose data or fail inconsistently.
