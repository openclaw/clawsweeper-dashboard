## Summary

Keep open: current main and v2026.6.10 still lack the requested model-facing browser tool API for CDP Network request interception, even though lower-level browser-control request listing and response-body capture exist.

## What This Changes

The issue requests exposing CDP Network-domain request capture/interception through the model-facing browser tool API.

## Best Possible Solution

Design this as a browser-plugin network capability with explicit sensitive-data policy, profile capability rules, and model-facing output wrapping, reusing the existing read-only `/requests` and `/response/body` pieces where appropriate.

Yes. Source inspection reproduces the gap: current main records requests and exposes lower-level `/requests` and `/response/body`, but the model-facing browser tool schema and action switch have no `network`, `requests`, `responsebody`, or `intercept` action.

## Solution Assessment

## Risks / Open Questions

- A model-facing Network interception API can expose cookies, authorization headers, tokens, private endpoints, and response bodies unless redaction, permissions, profile restrictions, and output wrapping are designed first.

- Managed, raw/remote CDP, sandbox, node-hosted, and existing-session browser profiles may need different support levels, so a single action shape could overexpose data or fail inconsistently.
