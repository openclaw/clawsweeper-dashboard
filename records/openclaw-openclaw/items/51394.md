## Summary

Current main still leaves the requested agent-facing browser Network surface open. The browser control API and CLI expose passive request listing and response-body capture, but the browser tool schema/dispatcher still do not expose network, requests, responsebody, or intercept actions, and true request interception remains a separate safety/design gap.

## What This Changes

Expose CDP Network-domain style browser tool capabilities so agents can capture, inspect, and potentially intercept browser network requests during SPA automation.

## Best Possible Solution

Keep the lower-level control API in place, add documented browser tool actions for passive request listing and response-body capture, and split true request interception into a scoped browser-plugin design with safety tests before exposing it to agents.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to inspect BrowserToolSchema and createBrowserTool: current main accepts no network/requests/responsebody/intercept action even though GET /requests and POST /response/body exist in the control API.

## Solution Assessment

Unclear as proposed. Exposing a raw CDP Network domain directly is likely not the safest maintainable boundary; a narrow browser tool API that reuses existing passive control routes is safer, while interception should be designed separately.
