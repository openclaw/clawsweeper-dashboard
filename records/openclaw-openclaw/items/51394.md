## Summary

Keep open: current main still lacks an agent-facing browser Network surface, while the existing lower-level control API and CLI only cover passive request listing and response-body capture. The requested interception capability needs maintainer product and security design before automation should implement it.

## What This Changes

The issue asks to add browser tool/CDP Network actions for request capture, listing, response inspection, and URL-pattern interception during browser automation.

## Best Possible Solution

Expose a narrow passive browser tool API for request listing and response-body capture, then handle true interception as a separate scoped browser plugin design with safety tests.

## Reproduction Assessment

Yes. Source inspection on current main shows BrowserToolSchema and createBrowserTool do not accept network, requests, responsebody, or intercept actions, even though GET /requests and POST /response/body exist in the control API.

## Solution Assessment

Unclear as proposed. A raw CDP Network-domain exposure is not the narrowest maintainable boundary; passive request/response inspection can reuse existing routes, while interception needs a separate safety design.
