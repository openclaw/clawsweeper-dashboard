## Summary

Keep open: current main and v2026.5.22 still have passive browser-control request inspection routes and CLI docs, but the agent-facing browser tool does not expose request listing, response-body lookup, CDP Network, Fetch, or interception actions. The remaining work is a new security-sensitive browser API, not cleanup to close.

## What This Changes

Request to add agent-facing browser tool support for CDP Network request capture and interception so browser automation can discover SPA API traffic.

## Best Possible Solution

Expose a narrow passive requests and response-body browser-tool action first, then design mutating CDP/Fetch interception separately with explicit scoping, redaction, permission policy, SSRF interaction, and regression tests.

## Reproduction Assessment

Yes, at source level: BrowserToolSchema and createBrowserTool still do not accept or route network, requests, responsebody, CDP, Fetch, or intercept actions, while lower-level /requests and /response/body routes exist. I did not run a live browser session because this was a read-only review.

## Solution Assessment

Unclear as proposed. Passive inspection can reuse existing control routes, but raw CDP Network or mutating interception is broader than necessary and needs maintainer API/security design first.
