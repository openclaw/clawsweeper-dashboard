## Summary

Keep open. The PR is useful directionally, but it is not merge-ready: the branch still leaves the normal ACP manager path on a null audit sink, the file logger has correctness and retention gaps called out in prior review comments, and the branch is stale against current main ACP manager behavior.

## What This Changes

The PR branch adds an ACP control-plane audit logging module with file/null loggers, wires session init/close audit calls into AcpSessionManager, adds audit logger tests, and carries small unrelated config comment/schema-ordering changes.

## Best Possible Solution

Rework the feature on current main with an explicit ACP audit config surface, a production-wired sink, secure state-directory storage, best-effort lifecycle emission, deterministic retention, and focused tests for failure/race paths before considering merge.

## Reproduction Assessment

Yes. The main functional gap is reproducible by source inspection: the PR head's DEFAULT_DEPS uses createNullAuditLogger(), while getAcpSessionManager() constructs the manager without injecting a file sink, so normal ACP session init/close calls cannot persist audit records.

## Solution Assessment

No. The patch is not the best current solution because it adds audit plumbing without production wiring or secure storage semantics, and it wraps sensitive lifecycle paths in awaited audit calls instead of best-effort bookkeeping.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Wire the default manager to a real audit sink:** `src/acp/control-plane/manager.types.ts:149`
  - body: `getAcpSessionManager()` still constructs `new AcpSessionManager()` with default deps, and this added default is `createNullAuditLogger()`. Since no normal caller injects `FileAuditLogger`, the advertised session init/close audit trail is never persisted in production.
  - confidence: 0.93
- **[P2] Keep audit writes best-effort:** `src/acp/control-plane/manager.core.ts:347-358`
  - body: Both session init and close await `this.auditLogger.log()` from `finally`. If a real or injected audit sink rejects, a successful lifecycle operation can be turned into a failure or the original error can be masked, so audit emission should be caught and logged separately.
  - confidence: 0.88
- **[P2] Cap the buffer after failed flushes:** `src/acp/control-plane/audit/audit-logger.file.ts:97-100`
  - body: When `appendFile` fails, `flush()` requeues the full batch with `unshift`, while `log()` keeps pushing new entries and only schedules another async flush. A sustained disk or permission failure can grow memory without bound despite `maxBufferSize`.
  - confidence: 0.9
- **[P2] Create private audit log files:** `src/acp/control-plane/audit/audit-logger.file.ts:60-63`
  - body: Audit records can include client IPs, device/user IDs, session keys, details, and error messages, but the logger creates the directory and appends JSONL files without restrictive modes. Existing audit storage uses `0o700` directories and `0o600` files; this sink should do the same.
  - confidence: 0.86
- **[P2] Actually enforce retention:** `src/acp/control-plane/audit/audit-logger.file.ts:47-57`
  - body: `retentionDays` is part of the public defaults and the PR claims automatic pruning, but production construction only starts the flush timer and never calls `prune()`. Enabled loggers will keep JSONL files indefinitely unless some external caller does undocumented cleanup.
  - confidence: 0.84
- **[P2] Await storage setup before the first flush:** `src/acp/control-plane/audit/audit-logger.file.ts:60-63`
  - body: The constructor starts `fs.mkdir(...).catch(...)` but does not await or reuse that promise. An immediate `flush()` or `close()` on a new storage path can race directory creation and fail with `ENOENT`.
  - confidence: 0.83
- **[P2] Make queries atomic across flushes:** `src/acp/control-plane/audit/audit-logger.file.ts:225-239`
  - body: `flush()` removes entries from `this.buffer` before the append completes, while `query()` reads the buffer and then disk with no pending-flush set or id dedupe. Concurrent queries can miss in-flight events or return duplicates when a flush moves entries between the two passes.
  - confidence: 0.78
- **[P3] Return the documented pruned entry count:** `src/acp/control-plane/audit/audit-logger.file.ts:286-290`
  - body: The interface says `prune()` returns the number of entries pruned, but this implementation increments once per deleted file. Any metrics or compliance report using the return value will undercount multi-entry JSONL files.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff does not add third-party dependencies or workflow changes, but the new audit sink stores sensitive local audit records without restrictive file permissions or a clear state-dir contract.

Concerns:

- **[medium] Audit JSONL files use default permissions:** `src/acp/control-plane/audit/audit-logger.file.ts:60`
  - body: `AuditLogEntry` can contain client IPs, device/user identifiers, session keys, details, and error messages, but the file logger calls `mkdir` and `appendFile` without explicit private modes. Existing OpenClaw audit storage uses `0o700` directories and `0o600` files, so this new sink could expose sensitive audit data to other local users depending on umask.
  - confidence: 0.86
