Overall confidence: 0.94

Full review comments:

- **[P1] Preserve ACP runtime options during session init:** `src/acp/control-plane/manager.core.ts:252-266`
  - body: This rebuilds `initialRuntimeOptions` from only `cwd` and calls `ensureSession` without model/thinking. Current main forwards `input.runtimeOptions` and has regression coverage for persisted model/thinking/cwd, so this would silently ignore caller-selected ACP runtime settings.
  - confidence: 0.96
- **[P1] Keep current closeSession reset semantics:** `src/acp/control-plane/manager.core.ts:1166-1169`
  - body: This older close path calls `runtime.close` without `discardPersistentState` and lacks current main's fast-reset, prepareFreshSession, and extra recoverable-error branches. Existing cleanup/reset flows can leave runtime state stale or keep session slots blocked.
  - confidence: 0.94
- **[P1] Wire a real configured audit sink:** `src/acp/control-plane/manager.types.ts:149`
  - body: `DEFAULT_DEPS` installs `createNullAuditLogger()`, and the production singleton still constructs `new AcpSessionManager()` without custom deps. Normal ACP init/close calls therefore persist no audit trail, so the advertised security feature is inert for users.
  - confidence: 0.95
- **[P2] Isolate audit sink failures from ACP lifecycle:** `src/acp/control-plane/manager.core.ts:347-358`
  - body: The lifecycle `finally` blocks await `auditLogger.log()`. If an injected sink rejects, a successful init/close can fail or an original control-plane error can be replaced by the audit error; audit emission should be best-effort or policy-controlled.
  - confidence: 0.9
- **[P2] Create audit storage with private permissions:** `src/acp/control-plane/audit/audit-logger.file.ts:59-63`
  - body: The file sink creates directories and appends JSONL records with filesystem defaults. These records can include actor identifiers, client IPs, session keys, and error text, so directories/files should use explicit private modes.
  - confidence: 0.89
- **[P2] Cap buffered entries after failed flushes:** `src/acp/control-plane/audit/audit-logger.file.ts:243-246`
  - body: When `appendFile` fails, `flush()` requeues the whole batch with `unshift()` while new `log()` calls keep appending. Sustained disk or permission failures can grow memory beyond the advertised `maxBufferSize`.
  - confidence: 0.88
- **[P2] Synchronize queries with in-flight flushes:** `src/acp/control-plane/audit/audit-logger.file.ts:225-239`
  - body: `flush()` removes entries from `this.buffer` before `appendFile` completes, while `query()` scans only the buffer and completed files. Queries during that window can miss entries, and races can duplicate entries across memory and disk.
  - confidence: 0.86
- **[P2] Return pruned entry counts:** `src/acp/control-plane/audit/audit-logger.file.ts:286-290`
  - body: The interface says `prune()` returns the number of entries pruned, but the implementation increments once per deleted daily file. Metrics or compliance reports built on this value would undercount removals.
  - confidence: 0.84
- **[P2] Actually enforce retention pruning:** `src/acp/control-plane/audit/audit-logger.file.ts:47-57`
  - body: `retentionDays` is part of the advertised config, but production construction only starts a flush timer and never schedules or invokes `prune()`. File audit logs can grow without bound unless an external caller manually prunes them.
  - confidence: 0.82
