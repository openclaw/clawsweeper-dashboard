## Label Justifications

- `P2`: This is a confirmed, bounded Telegram transport bug that completely blocks attachment workflows on IPv4-broken/IPv6-working hosts.
- `impact:message-loss`: Inbound Telegram photos and files fail during download and never reach the agent workflow.

## Summary

Current main still allows qualifying Telegram fetch failures to promote an IPv4-only dispatcher even when the primary transport was built from explicit IPv6-capable network config. The closed related branches did not merge this fix, and https://github.com/openclaw/openclaw/issues/55099 was closed in favor of this issue as the canonical remaining fallback-policy repair.

## What This Changes

Prevent Telegram media downloads from switching to forced IPv4 when explicit Telegram network configuration requires IPv6-capable behavior.

## Best Possible Solution

Keep explicit `channels.telegram.network` address-family choices authoritative, while retaining bounded and recoverable IPv4 fallback only when policy came from defaults; cover both inverse network topologies and SSRF-pinned media fetching in regression tests.

## Reproduction Assessment

Yes, at source level. On current main, explicit IPv6-capable config controls the primary dispatcher, but a qualifying network failure can still promote a forced-IPv4 dispatcher.

## Solution Assessment

Yes, with refinement: the narrowest maintainable fix is to suppress forced-family fallback when address-family policy is explicit, while preserving tested adaptive fallback for default-derived policy; a new config switch is not needed.
