## Summary

Keep open: current main and the latest release still preserve explicit Telegram network config for the primary transport, but media downloads can retry through IPv4-only fallback dispatcher attempts after broad fetch/network failures. The linked broad Telegram PR targets this issue, but it remains open and has not landed the fix.

## What This Changes

The issue asks Telegram media downloads to stop overriding explicit IPv6-friendly `channels.telegram.network` settings with runtime IPv4-first fallback behavior.

## Best Possible Solution

Land a narrow Telegram transport policy fix that preserves explicit IPv6-friendly network settings during fallback, with focused media-download regression coverage.

## Reproduction Assessment

Yes, source-level. Current main deterministically feeds explicit Telegram network config into the primary dispatcher, then media-download retry can iterate IPv4-forced fallback dispatcher policies after qualifying fetch errors.

## Solution Assessment

Yes. The narrowest maintainable fix is to gate automatic IPv4-only fallback when explicit Telegram network config should be preserved; adding a new opt-out setting would be broader than the reported bug requires.
