## Summary

Keep open because current main still preserves explicit Telegram network config only for the primary transport, while media-download retry can still advance to IPv4-only fallback dispatcher attempts after broad fetch/network failures.

## What This Changes

Fix Telegram media downloads on IPv4-broken / IPv6-working hosts by preventing runtime IPv4 fallback from overriding explicit Telegram network configuration.

## Best Possible Solution

Land a narrow Telegram transport policy change that preserves explicit IPv6-friendly network settings during fallback, with focused media-download regression coverage.

## Reproduction Assessment

Yes, source-level. Current main deterministically resolves explicit Telegram network settings as config-sourced, then media download retry can still walk IPv4-only fallback dispatcher attempts after qualifying fetch errors.

## Solution Assessment

Yes. The narrowest maintainable fix is to gate sticky IPv4 fallback when explicit Telegram network config should be preserved; a new opt-out config is broader than needed unless maintainers want a larger policy surface.
