## Summary

Keep open. Current main still honors explicit Telegram network config for the primary transport, but it also still builds IPv4-only fallback dispatcher attempts that Telegram media downloads retry after broad fetch/network errors; the related implementation PR remains open and unmerged.

## What This Changes

Requests fixing Telegram media downloads on IPv4-broken/IPv6-working hosts by preventing runtime IPv4-only fallback from overriding explicit `channels.telegram.network` settings.

## Best Possible Solution

Land a narrow Telegram transport policy change that preserves explicit IPv6-friendly network settings during fallback, with focused media-download regression coverage.

## Reproduction Assessment

Yes, source-level. Current main deterministically resolves explicit Telegram network settings as config-sourced, then still exports IPv4-only fallback attempts that media downloads retry after qualifying fetch errors.

## Solution Assessment

Yes, the narrowest maintainable solution is to policy-gate sticky IPv4 fallback when explicit network config should be preserved; adding a new opt-out config should only be considered if maintainers want a broader fallback policy surface.
