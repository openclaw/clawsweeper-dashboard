## Summary

Keep open. Current main still honors explicit Telegram network config for the primary dispatcher, but it also still exports IPv4-only fallback dispatcher attempts for media downloads without checking whether the address-family policy came from explicit config; the linked batch PR remains open and unmerged.

## What This Changes

The issue requests that Telegram media downloads preserve explicit IPv6-friendly network configuration instead of switching to a runtime IPv4 fallback after fetch errors.

## Best Possible Solution

Preserve explicit Telegram network policy by suppressing or policy-gating sticky IPv4 fallback when autoSelectFamily or dnsResultOrder came from user config, with focused regression coverage for media downloads.

## Reproduction Assessment

Yes, source-level. Current main deterministically resolves explicit config as source=config, then still exports IPv4-only fallback dispatcher attempts into the Telegram media download path.

## Solution Assessment

No, current main is not the best solution because it can still force the one-direction IPv4 fallback after explicit IPv6-friendly config. The narrow maintainable fix belongs in the Telegram transport fallback policy and media regression tests.
