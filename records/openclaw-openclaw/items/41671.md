- `impact:message-loss`: The reported failure prevents inbound Telegram media from downloading and reaching the agent workflow.

## Summary

Keep open: current main and the latest shipped release still let retryable Telegram fetch/media failures advance from an explicitly configured IPv6-friendly primary transport into a forced IPv4-only fallback, so the reported message-loss bug remains source-reproducible.

## What This Changes

Telegram media downloads can fail on IPv4-broken, IPv6-working hosts when runtime retry fallback overrides explicit IPv6-friendly Telegram network config.

## Best Possible Solution

Gate the Telegram forced-IPv4 fallback on explicit network intent so IPv6-friendly config stays respected, while preserving default and explicitly configured IPv4 recovery paths with focused regressions for both directions.

## Reproduction Assessment

Yes, at source level: explicit config reaches resolveTelegramTransport, but retryable Telegram media/API failures still get fallback attempts that force IPv4-only behavior. I did not run a live Telegram host with broken IPv4 and working IPv6 in this read-only review.

## Solution Assessment

Yes, the narrow maintainable fix is to make the existing fallback policy respect explicit IPv6-friendly config rather than adding a new config knob first.
