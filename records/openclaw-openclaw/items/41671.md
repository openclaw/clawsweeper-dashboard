- `impact:message-loss`: The reported failure prevents inbound Telegram media from downloading and reaching the agent workflow.

## Summary

Keep open: current main and v2026.5.22 still build a forced IPv4-only Telegram fallback after broad fetch/network failures, even when explicit IPv6-friendly Telegram network config is recognized and passed into the transport.

## What This Changes

The issue reports Telegram media downloads failing on IPv4-broken/IPv6-working hosts because runtime fallback forces IPv4-only behavior despite explicit Telegram network config.

## Best Possible Solution

Gate sticky IPv4-only fallback so explicit IPv6-friendly Telegram network config is not overridden, while preserving default IPv4 recovery and explicit IPv4 operator intent.

## Reproduction Assessment

Yes, at source level: explicit Telegram network config reaches resolveTelegramTransport, but retryable fetch failures can still promote requests to forced IPv4-only dispatcher attempts. I did not run a live IPv4-broken / IPv6-working Telegram host in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is to make the existing fallback policy respect explicit IPv6-friendly config rather than adding a new setting or changing global Node/undici defaults first.
