- `impact:message-loss`: The reported failure prevents inbound Telegram media from downloading and reaching the agent workflow.

## Summary

Keep open: current main and the latest shipped tag still build Telegram fallback attempts that can promote retryable API and media fetches to a forced IPv4-only dispatcher even when the operator configured IPv6-friendly Telegram network settings.

## What This Changes

The issue requests that Telegram media/API download retry behavior stop overriding explicit `channels.telegram.network` settings with forced IPv4 fallback on IPv4-broken, IPv6-working hosts.

## Best Possible Solution

Refine Telegram transport fallback so explicit IPv6-friendly network config prevents forced IPv4-only fallback, while default broken-IPv6 recovery and explicit IPv4 settings continue to work.

## Reproduction Assessment

Yes, at source level: explicit Telegram network config reaches `resolveTelegramTransport`, but retryable fetch errors can still promote API and media fetch attempts to forced IPv4-only dispatchers. I did not run a live IPv4-broken / IPv6-working Telegram host in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is to make the existing fallback policy respect explicit IPv6-friendly config rather than adding a new setting or changing global Node/undici defaults first.
