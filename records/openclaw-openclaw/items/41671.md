## Summary

Keep open: current main still lets Telegram promote retryable transport failures to a sticky IPv4-only fallback even when explicit IPv6-friendly Telegram network config is present, so the reported media/API failure remains actionable.

## What This Changes

Telegram media downloads should keep honoring explicit IPv6-friendly `channels.telegram.network` settings instead of switching to sticky IPv4-only fallback after retryable network errors.

## Best Possible Solution

Make the Telegram transport fallback respect explicit network config, keep default IPv4 fallback for unconfigured broken-IPv6 hosts, and cover both directions with focused regression tests.

## Reproduction Assessment

Yes, source-level. Current main threads explicit Telegram network config into the primary transport, then still exposes fallback attempts that force IPv4-only behavior after retryable fetch failures.

## Solution Assessment

Yes. The narrow maintainable fix is to gate or reshape sticky IPv4 fallback when explicit IPv6-friendly Telegram network config is present, rather than adding a new config knob first.
