## Summary

Keep open: current main and the latest release still build an IPv4-forced Telegram fallback after retryable fetch errors even when explicit IPv6-friendly Telegram network config is recognized and passed into the transport. The related duplicate at https://github.com/openclaw/openclaw/issues/55099 was closed in favor of this issue, while https://github.com/openclaw/openclaw/issues/75574 tracks the mirror IPv6-broken / IPv4-working case that the fix must preserve.

## What This Changes

The issue reports Telegram media download failure on IPv4-broken but IPv6-working hosts because runtime retry fallback forces IPv4-first behavior despite explicit IPv6-friendly Telegram network config.

## Best Possible Solution

Refine Telegram transport fallback so explicit IPv6-friendly config prevents forced IPv4-only fallback, while default broken-IPv6 recovery and explicit IPv4 settings remain covered by focused regression tests.

## Reproduction Assessment

Yes, at source level: explicit Telegram network config reaches resolveTelegramTransport, but retryable fetch failures can still promote API and media fetch attempts to forced IPv4-only dispatchers. I did not run a live IPv4-broken / IPv6-working Telegram host in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is to make the existing fallback policy respect explicit IPv6-friendly config rather than adding a new setting or changing global Node/undici defaults first.
