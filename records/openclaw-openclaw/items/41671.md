## Summary

Keep open: current main still lets explicit IPv6-friendly Telegram network config reach the transport, then permits retryable failures to promote the transport to an IPv4-only sticky fallback. The related batch PR at https://github.com/openclaw/openclaw/pull/42059 is open and unmerged, so there is no implemented fix to close against.

## What This Changes

The issue requests that Telegram media/API fetch fallback stop overriding explicit `channels.telegram.network` settings on IPv4-broken, IPv6-working hosts.

## Best Possible Solution

Gate or reshape sticky IPv4 fallback so explicit IPv6-friendly Telegram network config remains respected, while preserving default IPv4 fallback for unconfigured broken-IPv6 hosts and adding focused regression tests.

## Reproduction Assessment

Yes, source-level. Current main threads explicit Telegram network config into the primary transport, then still exposes fallback attempts that force IPv4-only behavior after retryable fetch failures.

## Solution Assessment

Yes. The narrow maintainable fix is to make fallback policy respect explicit IPv6-friendly Telegram config first, rather than adding a new config knob before fixing the existing contract.
