## Label Justifications

- `P2`: The bug is scoped to Telegram transport but can block Telegram media/API delivery on a real network shape.
- `impact:message-loss`: The reported failure prevents inbound Telegram media from downloading and reaching the agent workflow.

## Summary

Keep open: current main and the latest release still build forced IPv4-only Telegram fallback attempts even when the primary transport honored explicit IPv6-friendly network config, so the reported media/API failure remains source-reproducible.

## What This Changes

Prevent Telegram runtime IPv4 fallback from overriding explicit IPv6-friendly `channels.telegram.network` settings during Telegram API and media fetch retries.

## Best Possible Solution

Refine Telegram transport fallback so explicit IPv6-friendly config prevents forced IPv4-only fallback, while default broken-IPv6 recovery and explicit IPv4 settings remain covered by tests.

## Reproduction Assessment

Yes, at source level: explicit Telegram network config reaches resolveTelegramTransport, but retryable fetch failures can still promote API and media fetch attempts to forced IPv4-only dispatchers. I did not run a live IPv4-broken / IPv6-working Telegram host in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is to make the existing fallback policy respect explicit IPv6-friendly config rather than adding a new setting or changing global Node/undici defaults first.
