## Label Justifications

- `P2`: This is a bounded Telegram transport bug that can completely block inbound media workflows on IPv4-broken / IPv6-working hosts.
- `impact:message-loss`: Inbound Telegram photos and files can fail during download and never reach the agent workflow.

## Summary

Keep open: current main and the latest release still allow Telegram’s runtime fallback to build a forced IPv4 dispatcher after explicit IPv6-capable network config, so the reported media-download failure remains valid and canonical.

## What This Changes

The issue asks Telegram media downloads to keep explicit `channels.telegram.network` address-family config authoritative instead of letting runtime fallback force IPv4 on IPv4-broken/IPv6-working hosts.

## Best Possible Solution

Keep explicit Telegram address-family config authoritative, while retaining bounded IPv4 fallback only for default-derived policy and covering both Bot API and SSRF media download paths in regression tests.

## Reproduction Assessment

Yes, at source level. Current main still applies explicit IPv6-capable config to the primary dispatcher, then allows a qualifying network failure to promote a forced IPv4 fallback dispatcher on the same transport/media path.

## Solution Assessment

Yes, with a narrow repair. The best fix is to suppress forced-family fallback when address-family policy came from explicit config, rather than adding a new switch or removing fallback for default-derived IPv6-broken hosts.

## Repair Work Prompt

Fix the Telegram fallback-policy bug tracked by https://github.com/openclaw/openclaw/issues/41671. Keep explicit `channels.telegram.network.autoSelectFamily` and `channels.telegram.network.dnsResultOrder` authoritative: when either address-family setting is explicitly configured, do not promote or export a forced IPv4-only fallback dispatcher for Telegram fetch/media downloads. Preserve the existing bounded IPv4 fallback behavior for default-derived policy so IPv6-broken / IPv4-working hosts still recover. Keep SSRF pinning, explicit-proxy trust rules, and existing dispatcher lifecycle behavior intact. Add focused regression coverage for explicit IPv6-capable config on the normal fetch path and the `saveRemoteMedia`/dispatcherAttempts media path. Do not add a new config option, env var, changelog entry, or broad refactor.
