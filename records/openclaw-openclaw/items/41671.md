## Summary

Keep open. Current main still honors explicit Telegram network config only for the primary dispatcher, then prepares sticky IPv4-only fallback attempts for direct transports without checking whether the address-family policy came from user config; media downloads consume those attempts after retryable fetch errors. The linked implementation PR remains open rather than merged, so it is not evidence to close this issue yet.

## What This Changes

The issue asks OpenClaw to stop Telegram media download retry fallback from overriding explicit IPv6-friendly `channels.telegram.network` settings with IPv4-only behavior.

## Best Possible Solution

Land a narrow Telegram transport change that suppresses or policy-gates sticky IPv4 fallback when explicit network settings should be preserved, with media download regression coverage and real Telegram transport proof.

## Reproduction Assessment

Yes, source-level. Current main deterministically resolves explicit Telegram network settings as config-sourced, then still exports IPv4-only fallback attempts that the media download path retries after qualifying fetch errors.

## Solution Assessment

No, current main is not the best solution because it can still force one-direction IPv4 fallback after explicit IPv6-friendly config. The narrow maintainable fix belongs in the Telegram transport fallback policy and media regression tests.
