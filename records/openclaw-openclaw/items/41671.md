## Summary

Keep open: current main and the latest release still build forced IPv4-only Telegram fallback attempts even when explicit IPv6-friendly Telegram network config is recognized and passed into the transport, so the reported media/API failure remains source-reproducible.

## What This Changes

The issue requests that Telegram media/API fallback stop overriding explicit IPv6-friendly `channels.telegram.network` settings with forced IPv4-only behavior.

## Best Possible Solution

Refine Telegram transport fallback so explicit IPv6-friendly config prevents forced IPv4-only fallback, while default broken-IPv6 recovery and explicit IPv4 settings remain covered by tests.

## Reproduction Assessment

Yes, at source level: explicit Telegram network config reaches resolveTelegramTransport, but retryable fetch failures can still promote API and media fetch attempts to forced IPv4-only dispatchers. I did not run a live IPv4-broken / IPv6-working Telegram host in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is to make the existing fallback policy respect explicit IPv6-friendly config rather than adding a new setting or changing global Node/undici defaults first.

Fix the Telegram transport fallback bug reported at https://github.com/openclaw/openclaw/issues/41671. In current main, explicit IPv6-friendly `channels.telegram.network` config is recognized and passed into `resolveTelegramTransport`, but retryable fetch/media errors can still promote to fallback dispatcher attempts that force `autoSelectFamily: false`, `dnsResultOrder: "ipv4first"`, and `forceIpv4: true`. Keep the fix inside the Telegram transport/media fallback boundary, likely `extensions/telegram/src/fetch.ts` plus focused tests. Do not add a new config knob first. Ensure explicit IPv6-friendly operator intent, especially `autoSelectFamily: true` and/or `dnsResultOrder: "verbatim"`, prevents forced IPv4-only fallback from API and media paths. Preserve default IPv4 recovery for unconfigured IPv6-broken hosts, preserve explicit IPv4 operator intent, preserve proxy/env-proxy behavior, and do not weaken SSRF media guards. Add regression coverage for explicit IPv6-friendly config, existing default fallback, and explicit IPv4 behavior. Consider the mirror issue https://github.com/openclaw/openclaw/issues/75574 and the duplicate context at https://github.com/openclaw/openclaw/issues/55099 while keeping this to one focused fix.
