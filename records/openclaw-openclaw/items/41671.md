## Summary

Keep open. Current main still distinguishes explicit Telegram network config, but direct Telegram transports still prepare sticky IPv4-only fallback attempts without checking whether the address-family policy came from config, and media downloads consume those attempts; open PR #42059 is still the existing implementation candidate rather than a reason to close.

## What This Changes

The issue asks OpenClaw to stop Telegram media-download retry fallback from overriding explicit `channels.telegram.network` IPv6-friendly settings with IPv4-only behavior.

## Best Possible Solution

Land a narrow Telegram transport fix, likely split from #42059, that suppresses or policy-gates sticky IPv4 fallback when explicit network settings should be preserved, with regression coverage for media downloads.

## Reproduction Assessment

Yes, source-level. Current main deterministically resolves explicit Telegram network settings as config-sourced, then still exports IPv4-only fallback attempts that the media download path retries after qualifying fetch errors.

## Solution Assessment

No, current main is not the best solution because it can still force one-direction IPv4 fallback after explicit IPv6-friendly config. The narrow maintainable fix belongs in the Telegram transport fallback policy and media regression tests.
