## Summary

Keep open: current main and v2026.5.20 still recognize explicit Telegram network config for the primary transport but can still promote Telegram API and media fetches into a forced IPv4-only fallback after retryable network errors.

## What This Changes

The issue asks OpenClaw to stop overriding explicit IPv6-friendly Telegram network settings with runtime IPv4-only fallback on IPv4-broken / IPv6-working hosts.

## Best Possible Solution

Refine Telegram transport fallback so explicit IPv6-friendly network config prevents forced IPv4-only fallback, while default broken-IPv6 recovery and explicit IPv4 settings continue to work.

## Reproduction Assessment

Yes, at source level: explicit Telegram network config reaches resolveTelegramTransport, but retryable fetch errors can still promote to forced IPv4 dispatcher attempts. I did not run a live IPv4-broken / IPv6-working Telegram host in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is to make the existing fallback policy respect explicit IPv6-friendly config rather than adding a new setting or changing global Node/undici defaults first.
