## Summary

Keep open. Current main still recognizes explicit Telegram network config, but it still constructs IPv4-only fallback dispatcher attempts without checking whether that policy came from user config, and the inbound media path still passes those fallback attempts into media downloads. The related batch PR remains open and unmerged, so there is not enough evidence for an implemented-on-main close.

## What This Changes

The issue requests that Telegram media downloads stop forcing IPv4-first fallback after explicit IPv6-friendly `channels.telegram.network` config is loaded.

## Best Possible Solution

Keep the issue open and land a narrow Telegram transport fix: when `channels.telegram.network.autoSelectFamily` or `channels.telegram.network.dnsResultOrder` was explicitly configured, do not arm the sticky IPv4 fallback that exports `family: 4` media dispatcher attempts. Add focused regression coverage in `extensions/telegram/src/fetch.test.ts` and `extensions/telegram/src/bot/delivery.resolve-media-retry.test.ts` for the explicit `autoSelectFamily: true` plus `dnsResultOrder: "verbatim"` media-download path.

## Reproduction Assessment

Yes. The issue provides concrete network reproduction steps, and current main has a deterministic source-level reproduction: resolve Telegram transport with explicit IPv6-friendly config, then inspect the exported fallback attempts passed into `fetchRemoteMedia`.

## Solution Assessment

No, current main is not the best way to solve the issue because it can still force an IPv4-only fallback despite explicit config. The maintainable fix is a local Telegram transport guard or explicit fallback policy that preserves user-selected address-family behavior for media downloads.
