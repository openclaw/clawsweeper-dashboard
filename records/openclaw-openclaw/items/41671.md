## Summary

Keep open. Current main still recognizes explicit Telegram network config for the primary transport, but direct Telegram media downloads can still retry through exported IPv4-only fallback attempts after broad fetch/network errors; the related implementation candidate remains open and is not merged.

## What This Changes

The issue reports that Telegram media download retries override explicit IPv6-friendly Telegram network configuration with sticky IPv4 fallback on IPv4-broken hosts.

## Best Possible Solution

Land a narrow Telegram transport policy change that suppresses or policy-gates sticky IPv4 fallback when explicit IPv6-friendly network settings should be preserved, with focused media-download regression coverage.

## Reproduction Assessment

Yes, source-level. Current main deterministically resolves explicit Telegram network settings as config-sourced, then still exports IPv4-only fallback dispatcher attempts that media downloads retry after qualifying fetch errors.

## Solution Assessment

No. Current main is not the best solution because it can still force one-direction IPv4 fallback after explicit IPv6-friendly config; the narrow fix belongs in Telegram transport fallback policy and media regression tests.
