## Summary

Keep open: current main and the latest release still recognize explicit Telegram network config for the primary transport but still build and promote a forced IPv4-only fallback path after broad retryable network errors. The closed batch PR that claimed this fix was not merged, and the newer open Telegram network issue covers the opposite IPv6-broken shape rather than superseding this report.

## What This Changes

The issue asks OpenClaw to stop Telegram runtime IPv4 fallback from overriding explicit IPv6-friendly `channels.telegram.network` settings during media/API fetch retries.

## Best Possible Solution

Refine the Telegram transport fallback policy so explicit IPv6-friendly network config prevents forced IPv4-only sticky fallback, while default broken-IPv6 recovery and explicit IPv4 settings continue to work with focused regression tests.

## Reproduction Assessment

Yes, at source level: explicit Telegram network config reaches `resolveTelegramTransport`, and retryable fetch errors can still promote requests to forced IPv4 dispatcher attempts. I did not run a live IPv4-broken / IPv6-working Telegram host in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is to make the existing fallback policy respect explicit IPv6-friendly config rather than adding a new setting or changing global Node/undici behavior first.
