## Summary

Keep open because current main still builds IPv4-only Telegram fallback attempts after broad fetch failures, including for media downloads, while the reporter's explicit IPv6-friendly Telegram network config is only honored on the primary transport path.

## What This Changes

The issue requests that Telegram media download fallback stop overriding explicit IPv6-friendly `channels.telegram.network` settings on hosts where IPv4 is broken and IPv6 works.

## Best Possible Solution

Land a narrow Telegram transport/media fallback change that preserves explicit IPv6-friendly network settings during fallback, with focused regression coverage for media download dispatcher attempts.

## Reproduction Assessment

Yes, source-level. Current main recognizes explicit Telegram network config, then still exposes media-download dispatcher attempts that include IPv4-only fallback after retryable fetch/network failures.

## Solution Assessment

Yes. The narrow maintainable fix is to gate or reshape sticky IPv4 fallback when explicit IPv6-friendly Telegram network config is present; adding a new fallback-policy config option would be broader than needed for this bug.
