- `impact:message-loss`: The reported failure prevents inbound Telegram media from downloading and reaching the agent workflow.

## Summary

Keep open: current main and the latest release still build a forced IPv4-only Telegram fallback path after broad fetch/network failures, even when explicit IPv6-friendly Telegram network config is recognized and threaded into the transport.

## What This Changes

The issue reports Telegram media/API failures on IPv4-broken, IPv6-working hosts because runtime fallback can override `channels.telegram.network` and force IPv4-first behavior.

## Best Possible Solution

Refine Telegram transport fallback so explicit IPv6-friendly config prevents forced IPv4-only fallback, while default broken-IPv6 recovery and explicit IPv4 settings remain covered by tests.

## Reproduction Assessment

Yes, at source level: explicit Telegram network config reaches `resolveTelegramTransport`, but retryable fetch failures can still promote requests to forced IPv4-only dispatcher attempts. I did not run a live IPv4-broken / IPv6-working Telegram host in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is to make the existing fallback policy respect explicit IPv6-friendly config rather than adding a new setting or changing global Node/undici defaults first.
