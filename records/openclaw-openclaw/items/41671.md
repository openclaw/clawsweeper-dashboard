## Summary

Keep open: current main and the latest release still build IPv4-only Telegram fallback attempts after honoring explicit network config, and inbound media downloads consume those attempts.

## What This Changes

The issue asks OpenClaw to stop overriding IPv6-capable Telegram network config with runtime IPv4-only fallback on IPv4-broken hosts.

## Best Possible Solution

Adjust Telegram transport fallback so explicit IPv6-capable network config is not followed by IPv4-only fallback, while preserving default IPv4 recovery for broken-IPv6 hosts.

## Reproduction Assessment

Yes at source level, but not with a live network host in this read-only review. Current main still creates IPv4-only fallback attempts after explicit Telegram network config, and media downloads use those attempts.

## Solution Assessment

Yes. The best fix is a narrow Telegram fallback-policy change plus regression tests, not a new config option by default.
