## Summary

Keep open: current main and the latest shipped tag still preserve explicit Telegram network config on the primary transport, but retryable Telegram fetch/media failures can still advance into a forced IPv4-only fallback. The closed related broad PR did not merge, and the closely related sticky IPv4 issue was closed as a duplicate pointing back here, so this remains the canonical fix target.

## What This Changes

The issue asks OpenClaw to stop Telegram runtime IPv4 fallback from overriding explicit IPv6-friendly `channels.telegram.network` settings on IPv4-broken, IPv6-working hosts.

## Best Possible Solution

Make Telegram fallback policy respect explicit IPv6-friendly network config before arming forced IPv4-only fallback, while preserving default IPv4 recovery for unconfigured broken-IPv6 hosts and explicit IPv4 operator intent.

## Reproduction Assessment

Yes, at source level. Current main threads explicit Telegram network config into the primary transport, then still allows retryable failures to enter a forced IPv4-only fallback chain used by Telegram API and media fetches.

## Solution Assessment

Yes. The narrow maintainable fix is to make the existing fallback respect explicit IPv6-friendly config instead of adding a new knob first or changing global Node/undici defaults.
