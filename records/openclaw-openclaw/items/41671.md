## Label Justifications

- `P2`: The bug is scoped to Telegram transport but can block Telegram media/API delivery on a real network shape.
- `impact:message-loss`: The reported failure prevents inbound Telegram media from downloading and reaching the agent workflow.

## Summary

Keep open: current main and the latest shipped tag still honor explicit Telegram network config only on the primary transport, while retryable Telegram API/media failures can still advance into a forced IPv4-only fallback. The related sticky IPv4 report was closed back to this issue as canonical, and the mirror IPv4-intent report remains open separately.

## What This Changes

The issue asks Telegram media/API fallback to respect explicit IPv6-friendly network config instead of forcing IPv4-only dispatchers after retryable fetch errors.

## Best Possible Solution

Refine Telegram transport fallback so explicit IPv6-friendly network config prevents forced IPv4-only fallback, while the default unconfigured broken-IPv6 recovery path and explicit IPv4 settings keep working.

## Reproduction Assessment

Yes, at source level: resolveTelegramTransport accepts explicit IPv6-friendly config for the primary dispatcher, then still exposes forced IPv4-only fallback attempts and promotes them on retryable fetch/network errors. I did not run a live Telegram host with broken IPv4 and working IPv6 in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is to make the existing fallback policy respect explicit IPv6-friendly config, rather than adding a new knob or changing global Node/undici defaults first.
