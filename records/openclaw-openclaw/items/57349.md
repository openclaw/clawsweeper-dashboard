## Summary

Keep open. Current main has added useful bounded transcript-read, session-list responsiveness, session-cache, and active-memory cache mitigations, but the long-uptime gateway OOM report is broader than those fixes and still has recent corroborating evidence across Docker, macOS, Control UI/WebChat, Telegram-only heavy compaction, and memory-plugin-heavy deployments. I do not see source or test evidence that current main has eliminated the remaining whole-store, polling, transcript-tree, and memory-plugin retention paths well enough to close this as implemented or unreproducible.

## What This Changes

The issue asks OpenClaw to stop long-running gateway heap exhaustion/OOM crashes associated with filesystem scans, node.list polling, session/transcript growth, browser/tool use, and memory/plugin runtime retention.

## Best Possible Solution

Keep this open as the broad gateway heap-exhaustion tracker while using #55334 for the sessions.json/session-store subproblem. The maintainable end state is a profile-backed set of bounded data paths: push/coalesced node refresh instead of hot polling, paged/lazy session-store access, tail/page transcript reads that also avoid tree-transcript full reads, bounded compaction/session runtime retention, and memory-plugin watcher/session/cache cleanup in the owning modules.

## Reproduction Assessment

Unclear for current main. The issue discussion gives concrete reproduction steps and multiple high-signal crash reports on recent released builds, but this read-only review did not run a long soak or capture heap snapshots against 4babd925c40d9eeaecee95482b630342540da4f9.

## Solution Assessment

No, closing as implemented would not be the best solution. The best path is to keep the issue open, collect a current diagnostics export plus heap snapshots from an affected build, and then land targeted fixes in the owner modules rather than treating the existing mitigations as a complete fix.
