## Summary

Keep open. Current main has useful mitigations around diagnostics, session-store maintenance, chat.history response caps, active-memory cache bounds, memory-core streaming paths, and lazy QMD startup, but it does not prove the broad long-uptime Gateway OOM is fixed. The discussion contains recent multi-environment reports, and current source still has whole-store, whole-transcript, polling, watcher, and runtime-retention paths that need profiling before conservative cleanup can close this.

## What This Changes

The issue reports long-uptime Gateway heap/RSS growth and OOM crashes during filesystem scans, node/session polling, heavy compaction/tool use, and memory-plugin-heavy deployments.

## Best Possible Solution

Keep this as the broad Gateway heap-exhaustion tracker until maintainers collect a diagnostics export, stability bundle, and heap snapshots from a current affected build. Then split proven retainers into narrow owner PRs: push/bounded node refresh, paged or lazy session listing, tail/page chat.history reads, compaction/session-state release, memory-core watcher/session cleanup, and bounded plugin caches, with #55334 remaining the focused session-store subtracker.

## Reproduction Assessment

Unclear for current main. The report has strong multi-user reproduction evidence on 2026.3.28, 2026.4.11, and 2026.4.14 builds, but this read-only review did not run a current f3145 soak or capture heap snapshots to reproduce the retained objects today.

## Solution Assessment

No, closing as implemented is not the best way to solve this issue yet. The maintainable path is to profile current main, identify the retained object families, and land narrow fixes in the owning Gateway/session/memory/plugin surfaces rather than treating partial mitigations as proof of a complete fix.
