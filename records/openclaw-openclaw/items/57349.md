## Summary

Keep open. Current main has several real mitigations, but the report is still broader than the landed fixes: the original Control UI polling path still exists, session listing still materializes combined stores, persisted session rows still carry large prompt/report payloads, and related production reports plus an open focused fix PR show the main session-store subproblem is not fully resolved.

## What This Changes

The issue reports long-uptime OpenClaw gateway heap exhaustion and OOM crashes during filesystem scanning, Control UI/WebChat polling, session/compaction activity, browser use, and memory-plugin-heavy deployments.

## Best Possible Solution

Keep this as the broad gateway heap-exhaustion tracker, use https://github.com/openclaw/openclaw/issues/55334 and https://github.com/openclaw/openclaw/pull/78259 for the session-store payload subproblem, and require profile-backed fixes for any remaining polling, compaction, browser, and memory-plugin retention paths.

## Reproduction Assessment

No high-confidence current-main end-to-end reproduction was established in this read-only review. Source inspection does show current memory-relevant paths that match the reports, especially whole-store session reads, persisted prompt payloads, and recurring `node.list` polling.

## Solution Assessment

No, closing or treating existing mitigations as a complete solution is not the best way to solve the issue. The maintainable path is to land the focused session-store fix, then use heap snapshots and targeted soak proof to split any remaining retention paths by owning module.
