## Summary

Keep open: current main still stores a failed Talk start as persistent UI state and renders it without a clear or dismiss action; the existing ClawSweeper comment already identified the same source path and I found no merged fix in the checked-out main.

## What This Changes

Make Control UI Talk-mode start failures dismissable or avoid launching Talk when no realtime voice provider is configured.

## Best Possible Solution

Keep Talk available, but make failed-start Talk errors transient or dismissable in the Control UI and add focused regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: Start Talk calls `talk.realtime.session`, the missing provider config rejects, and the catch path stores the error in `realtimeTalkDetail` and `lastError` for the chat view to render.

## Solution Assessment

Yes. A narrow Control UI clear/dismiss path is the best first fix; hiding or disabling Talk based on provider availability would require a more reliable provider-status contract.
