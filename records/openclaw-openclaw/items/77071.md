## Summary

Current main still has the reported persistent Talk error state: failed realtime startup stores the provider error in Control UI state, and the chat composer renders that value without any dismissal path.

## What This Changes

The issue asks for the Control UI Talk error shown when no realtime voice provider is configured to be dismissable or prevented by disabling/hiding Talk.

## Best Possible Solution

Add a focused Control UI error-clear path for failed Talk status while leaving realtime provider resolution and configuration contracts unchanged.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: a missing configured realtime provider makes `talk.realtime.session` reject, `toggleRealtimeTalk()` stores that message in `realtimeTalkDetail`, and `renderChat()` keeps rendering it with no dismiss control. I did not run a live browser reproduction in this read-only sweep.

## Solution Assessment

Yes: a narrow dismiss/clear path for Talk error state is the best first fix because it directly addresses the stuck UI without changing provider selection, config schema, or Talk availability policy.
