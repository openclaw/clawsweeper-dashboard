## Summary

Keep open: current main still leaves a failed Talk start error in persistent Control UI state, and the chat view renders that state without a dismiss or clear action.

## What This Changes

The issue reports that the Control UI Talk-mode error shown when no realtime voice provider is configured cannot be dismissed.

## Best Possible Solution

Keep Talk available, but make failed-start Talk errors transient or dismissible in the Control UI and cover that behavior with a focused regression test.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: Start Talk calls talk.realtime.session, missing provider config rejects, and the catch path stores UI-visible error state that has no dismiss action.

## Solution Assessment

Yes. A narrow Control UI state/rendering fix is the best first repair; hiding Talk based on provider availability would need a stronger config/status contract and is not necessary for this stale-error bug.
