## Summary

Keep open: current main still leaves the Talk failure in UI state after a failed start, and the chat view renders that state without any clear or dismiss path.

## What This Changes

Clicking Start Talk without a configured realtime voice provider leaves a persistent, non-dismissable error in the Control UI chat surface.

## Best Possible Solution

Keep Talk available, but make failed-start Talk errors transient or dismissable in the Control UI and cover that behavior with a focused regression test.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: Start Talk calls talk.realtime.session, a missing selected provider rejects, and the catch path stores realtimeTalkDetail plus lastError. The chat renderer then shows both error surfaces without a dismiss callback or clear button.

## Solution Assessment

Yes, a narrow Control UI state/rendering fix is the best first repair. Hiding or disabling Talk based on provider availability would need a more reliable config/status contract, while a dismiss/clear path directly fixes the stale error state.
