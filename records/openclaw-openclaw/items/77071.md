## Summary

Keep open: current main still stores a failed Talk start as persistent Control UI state and renders it without a dismiss or clear path, so none of the safe close reasons apply.

## What This Changes

The issue requests that failed Talk-mode startup errors in the Control UI be dismissible or avoided when no realtime voice provider is configured.

## Best Possible Solution

Keep Talk available, but make failed-start Talk errors transient or dismissible in the Control UI and cover the behavior with focused regression tests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: Start Talk calls talk.realtime.session, a missing configured provider rejects, toggleRealtimeTalk stores realtimeTalkDetail plus lastError, and renderChat displays both without a dismiss callback.

## Solution Assessment

Yes. A narrow Control UI state/rendering fix is the best first repair; hiding or disabling Talk based on provider availability would require a stronger config/status contract than this bug needs.
