## Summary

Keep open; current main still lacks the requested SKILL.md tool/toolset availability gates, and the remaining work is a core skill spec/API design question rather than cleanup.

## What This Changes

The issue requests SKILL.md metadata fields that include or hide skills based on available tools or toolsets.

## Best Possible Solution

Extend the public skill metadata parser, types, docs, and eligibility context with a well-defined tool/toolset availability contract and regression tests for requires and fallback behavior.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/spec request. Source inspection on current main confirms the requested metadata fields, parser mapping, eligibility inputs, and docs are still absent.

## Solution Assessment

Unclear until maintainers define the tool/toolset taxonomy and refresh timing. The likely maintainable direction is a core skill-loader/spec extension, because plugin-published skills flow through the same loader and current prompt hooks do not re-filter loaded skills.
