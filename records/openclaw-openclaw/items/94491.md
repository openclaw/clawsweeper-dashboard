## Summary

Keep open: this is a real cron empty-response regression, but the PR is not the best fix because it moves cron policy into the shared CLI runner with a lane substring heuristic and leaves the embedded cron runner sibling unhandled.

## What This Changes

The PR changes `src/agents/cli-runner.ts` so empty CLI output does not throw when `params.lane` contains `cron`.

## Best Possible Solution

Thread the existing silent-empty flag from the cron executor into both CLI and embedded cron runner calls, keep the shared runner flag-based, and validate the chosen cron policy with focused tests plus redacted real Gateway cron proof.

## Reproduction Assessment

Yes, at source level: current main sends cron CLI runs without `allowEmptyAssistantReplyAsSilent`, and the shared runner throws `empty_response` on empty trimmed output. I did not run a live scheduled cron tick in this read-only review.

## Solution Assessment

No. The PR likely mitigates the CLI cron symptom, but the best fix is to pass the existing silent-empty flag from the cron executor for both CLI and embedded cron paths instead of adding lane substring inference to the shared runner.
