## Label Justifications

- `P1`: The linked bug breaks recurring silent cron automation by marking intentionally empty scheduled runs as failures.
- `merge-risk: 🚨 compatibility`: The PR changes shared CLI runner behavior based on a free-form lane substring rather than the existing explicit silent-empty flag.
- `merge-risk: 🚨 message-delivery`: The changed guard can turn empty backend output into no-delivery success, affecting whether failures are surfaced or routed.

## Summary

Keep open: the linked cron regression is real, but this PR is not merge-ready because it changes the shared CLI runner to infer silent success from a free-form lane string instead of passing the existing cron-owned silent-empty flag.

## What This Changes

The PR changes `src/agents/cli-runner.ts` so empty assistant output is not treated as a backend failure when the run lane string contains `cron`.

## Best Possible Solution

Set the existing silent-empty flag at the cron executor boundary for both CLI and embedded cron runs, keep the shared runner guard flag-based, and validate with live cron/Gateway proof plus focused regression tests.

## Reproduction Assessment

Yes. Source inspection on current main shows `server-cron` enters the cron lane, the cron executor omits `allowEmptyAssistantReplyAsSilent`, and `runPreparedCliAgent` throws `empty_response` on empty output unless that flag is true.

## Solution Assessment

No. The PR likely mitigates the CLI cron-nested symptom, but the best fix is to pass the existing silent-empty flag from the cron executor for both CLI and embedded cron paths instead of adding lane substring inference to the shared runner.
