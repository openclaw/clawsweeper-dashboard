## Summary

Keep open: the PR targets a real cron regression, but it moves silent-empty policy into a shared CLI runner lane substring check, leaves the embedded cron sibling unaligned, and still needs real scheduled cron/Gateway proof.

## What This Changes

The PR adds a lane substring check in `src/agents/cli-runner.ts` so empty CLI output does not throw when `params.lane` contains `cron`.

## Best Possible Solution

Set the existing silent-empty flag at the cron executor boundary for both CLI and embedded cron runs, keep the shared runner guard flag-based, and validate the chosen cron policy with focused tests plus redacted real cron/Gateway proof.

## Reproduction Assessment

Yes, at source level: current main sends cron CLI runs without `allowEmptyAssistantReplyAsSilent`, and the shared runner throws `empty_response` on empty trimmed output. I did not run a live scheduled cron tick in this read-only review.

## Solution Assessment

No. The PR likely mitigates the CLI cron symptom, but the best fix is to pass the existing silent-empty flag from the cron executor for both CLI and embedded cron paths instead of adding lane substring inference to the shared runner.
