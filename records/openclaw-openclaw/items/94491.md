## Summary

Keep open: the cron empty-response bug is real, but this PR changes the shared CLI runner guard by lane substring instead of using the existing explicit silent-empty flag at the cron caller boundary, so it is not merge-ready.

## What This Changes

The PR changes `src/agents/cli-runner.ts` so empty assistant output does not throw when the lane string contains `cron`.

## Best Possible Solution

Set `allowEmptyAssistantReplyAsSilent` at the cron executor boundary for both CLI and embedded cron runs, keep the shared runner guard explicit-flag based, and validate with focused tests plus redacted live cron/Gateway proof.

## Reproduction Assessment

Yes, at source level. Current main can route cron CLI runs through `runCliAgent` without `allowEmptyAssistantReplyAsSilent`, and the shared CLI runner throws `empty_response` on empty trimmed output unless that flag is true.

## Solution Assessment

No. The PR likely mitigates the CLI cron symptom, but the maintainable fix is to pass the existing silent-empty flag from the cron executor for both CLI and embedded cron paths instead of inferring policy from a lane substring inside the shared runner.
