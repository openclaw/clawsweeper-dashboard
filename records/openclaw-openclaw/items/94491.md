## Label Justifications

- `P1`: The PR targets a current cron regression that can mark recurring silent scheduled jobs as failed on every tick.
- `merge-risk: 🚨 compatibility`: The diff changes empty-output failure semantics in a shared runner based on lane naming rather than an explicit caller-owned flag.

## Summary

Keep open: this is a focused candidate for a real cron regression, but the patch is not merge-ready because it bypasses the shared CLI empty-response guard with a lane-substring heuristic, leaves the embedded cron sibling on the old contract, and still has only test/mocked proof.

## What This Changes

The PR changes `src/agents/cli-runner.ts` so empty CLI assistant output is not treated as a failover error when the run lane string contains `cron`.

## Best Possible Solution

Pass the existing silent-empty flag from the cron executor for intended silent cron contexts in both CLI and embedded paths, keep the shared runner flag-based, and validate with focused tests plus live cron/Gateway proof.

## Reproduction Assessment

Yes, by source inspection. A cron CLI run can reach `runPreparedCliAgent` without `allowEmptyAssistantReplyAsSilent`, and empty trimmed CLI output then throws `FailoverError` with `reason: "empty_response"`.

## Solution Assessment

No. The PR likely mitigates the CLI cron symptom, but the better fix is to set the existing silent-empty flag at the cron executor boundary for both runner implementations instead of inferring policy from a lane substring.
