## Summary

Keep open: this is a focused fix for a real cron regression, but the patch is not merge-ready because it moves cron policy into the shared CLI empty-response guard and still lacks live scheduled-cron proof.

## What This Changes

The PR changes `src/agents/cli-runner.ts` so empty CLI output does not throw when the free-form lane string contains `cron`.

## Best Possible Solution

Pass the existing silent-empty flag at the cron executor boundary for the intended cron policy, cover both CLI and embedded cron paths, and validate with focused tests plus redacted live cron/Gateway proof.

## Reproduction Assessment

Yes, at source level: current main sends cron CLI runs without `allowEmptyAssistantReplyAsSilent`, and the runner throws `empty_response` on empty trimmed output. I did not run a live scheduled cron tick in this read-only review.

## Solution Assessment

No. The PR likely mitigates the CLI cron symptom, but the best fix is to pass the existing silent-empty flag from the cron executor for both CLI and embedded cron paths instead of adding lane substring inference to the shared runner.
