## Summary

Keep this PR open but do not merge as-is: the reported cron empty-response regression is real on current main and v2026.6.9, but this patch puts cron policy inference in the shared CLI runner instead of the cron executor boundary that also owns the embedded sibling path.

## What This Changes

The PR changes `src/agents/cli-runner.ts` so empty CLI output skips `FailoverError` when the free-form lane string contains `cron`.

## Best Possible Solution

Set the existing silent-empty flag at the cron executor boundary for both CLI and embedded cron runs, keep the shared runner guard flag-based, and validate with focused tests plus redacted live cron/Gateway proof.

## Reproduction Assessment

Yes. Source inspection shows current cron CLI runs can enter the runner with cron trigger/lane and no silent-empty flag, then the runner throws `empty_response` on empty trimmed output.

## Solution Assessment

No. The PR likely mitigates the CLI cron symptom, but the best fix is to pass the existing silent-empty flag from the cron executor for both CLI and embedded cron paths instead of adding lane substring inference to the shared runner.
