## Summary

Keep open: the PR targets a real cron empty-response regression, but the patch is at the shared CLI runner throw site and infers silence from a free-form lane substring instead of setting the existing caller-owned silent-empty flag at the cron executor boundary. That leaves the embedded cron sibling split and can make non-cron `cron`-named lanes silently succeed on empty backend output.

## What This Changes

The PR adds a `params.lane` substring check to the shared CLI empty-response guard so empty output does not throw when the lane contains `cron`.

## Best Possible Solution

Set the existing silent-empty flag at the cron executor boundary for both CLI and embedded cron runs, keep the shared runner guard flag-based, and validate the selected cron policy with focused tests plus redacted live cron/Gateway proof.

## Reproduction Assessment

Yes, at source level: current main sends cron CLI runs without `allowEmptyAssistantReplyAsSilent`, and the shared runner throws `empty_response` on empty trimmed output. I did not run a live scheduled cron tick in this read-only review.

## Solution Assessment

No. The PR likely mitigates the CLI cron symptom, but the best fix is to pass the existing silent-empty flag from the cron executor for both CLI and embedded cron paths instead of adding lane substring inference to the shared runner.
