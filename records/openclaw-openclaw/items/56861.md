## Summary

Keep open: the Signal direction is useful, but this PR is still not mergeable because quoted media is modeled against the wrong upstream shape, fetched quoted media bypasses OpenClaw's media ingestion contract, reply-to-bot identity matching is fragile, and the external-PR real behavior proof gate is still unmet.

## What This Changes

The PR adds Signal quoted-message media context, structured mention detection, reply-to-bot mention handling, and data-URI attachment sends for remote signal-cli setups.

## Best Possible Solution

Keep the useful Signal behavior, but port it onto current main with upstream-shaped quoted attachments, media routed through MediaPaths/MediaTypes, separate quote phone/UUID checks, focused tests, and redacted real Signal proof.

## Reproduction Assessment

Yes for the review findings: PR-head source inspection plus upstream signal-cli JSON records show the quoted attachment shape mismatch, and current-main source shows ReplyToMediaPath is not part of media normalization. I did not run a live Signal setup in this read-only review.

## Solution Assessment

No. The direction is useful, but the branch is not the best mergeable solution until it follows the upstream quote contract, uses OpenClaw's existing media ingestion fields, preserves identity matching across phone and UUID, and includes real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Read quoted media from the nested thumbnail:** `extensions/signal/src/monitor/event-handler.ts:797`
  - body: signal-cli represents quote attachments as entries with contentType/filename and an optional nested thumbnail; the id is on that thumbnail attachment, not the quoted attachment itself. Scanning only `quote.attachments[].id` means real quoted media will usually be skipped, so the headline reply-target media feature still fails.
  - confidence: 0.9
- **[P2] Route quoted media through MediaPaths:** `extensions/signal/src/monitor/event-handler.ts:219-220`
  - body: The PR stores fetched quoted media only in `ReplyToMediaPath` and `ReplyToMediaType`, but the model media pipeline normalizes attachments from `MediaPath(s)` and `MediaUrl(s)`. Even if the quote fetch succeeds, the media will not reach media understanding.
  - confidence: 0.92
- **[P2] Compare quote phone and UUID separately:** `extensions/signal/src/monitor/event-handler.ts:660`
  - body: `quote.author ?? quote.authorUuid` ignores the UUID whenever the deprecated author field is present. In UUID-only bot configurations, a reply to the bot can be treated as not mentioning the bot and then skipped by mention gating.
  - confidence: 0.86
- **[P3] Thread the resolved MIME type into data URIs:** `extensions/signal/src/send.ts:16-34`
  - body: The data URI helper re-derives MIME type from the local filename even though the resolved attachment content type is available at the call site. Extensionless or temporary files can be sent as `application/octet-stream` despite a known type.
  - confidence: 0.78
