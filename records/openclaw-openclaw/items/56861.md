## Summary

Keep open: the Signal feature direction is useful and not fully implemented on current main, but this branch is not mergeable as-is because source review still finds quoted-media, media-routing, and reply identity bugs, plus no real Signal proof.

## What This Changes

The PR modifies Signal inbound handling and send paths to add quoted-message media context, structured mention and reply-to-bot detection, and conditional data-URI attachments for non-local signal-cli base URLs.

## Best Possible Solution

Port the useful Signal changes onto current main using upstream-shaped quote attachments, existing MediaPaths/MediaTypes ingestion, API-mode-owned attachment encoding, focused tests, and redacted live Signal proof.

## Reproduction Assessment

Yes for the PR findings: PR-head source inspection plus upstream signal-cli quote types show the quoted media shape mismatch, and current-main source shows ReplyToMediaPath is not part of media normalization. I did not run a live Signal setup in this read-only review.

## Solution Assessment

No. The direction is reasonable, but this branch should not merge until it follows the upstream quote contract, routes media through existing ingestion fields, compares phone and UUID identity separately, and includes real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Read quoted media from the nested thumbnail:** `extensions/signal/src/monitor/event-handler.ts:797`
  - body: signal-cli represents quote attachments as JsonQuotedAttachment entries with an optional nested thumbnail, and the id lives on that nested JsonAttachment. Searching only quote.attachments[].id means real quoted media will usually be skipped, so the reply-target media feature still fails.
  - confidence: 0.9
- **[P2] Route quoted media through MediaPaths:** `extensions/signal/src/monitor/event-handler.ts:219-220`
  - body: The PR stores fetched quoted media only in ReplyToMediaPath/ReplyToMediaType, but the model media pipeline normalizes attachments from MediaPath(s) and MediaUrl(s). Even if the quote fetch succeeds, the media will not reach media understanding.
  - confidence: 0.92
- **[P2] Compare quote phone and UUID separately:** `extensions/signal/src/monitor/event-handler.ts:660`
  - body: quote.author ?? quote.authorUuid ignores the UUID whenever the legacy author field is present. In UUID-only bot configurations, a reply to the bot can be treated as not mentioning the bot and then skipped by mention gating.
  - confidence: 0.86
- **[P3] Thread the resolved MIME type into data URIs:** `extensions/signal/src/send.ts:16-34`
  - body: The data URI helper re-derives MIME type from the local filename even though the resolved attachment content type is available at the call site. Extensionless or temporary files can be sent as application/octet-stream despite a known type.
  - confidence: 0.78
