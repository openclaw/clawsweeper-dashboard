## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Read quoted media from quote thumbnails:** `extensions/signal/src/monitor/event-handler.ts:797`
  - body: signal-cli wraps quoted attachments as JsonQuotedAttachment, where the downloadable JsonAttachment with id is under thumbnail. Looking for a top-level id skips real quoted media before fetchAttachment can run.
  - confidence: 0.9
- **[P2] Route quoted media through MediaPaths:** `extensions/signal/src/monitor/event-handler.ts:219-220`
  - body: The patch stores quoted attachments only in ReplyToMediaPath/ReplyToMediaType, but OpenClaw media normalization consumes MediaPath(s) and MediaUrl(s). The agent will not receive the quoted image/audio for model media processing.
  - confidence: 0.92
- **[P2] Compare quote phone and UUID independently:** `extensions/signal/src/monitor/event-handler.ts:660`
  - body: Collapsing quote.author and quote.authorUuid skips UUID matching whenever the legacy author field is present. UUID-only bot configs can miss reply-to-bot and then drop mention-gated group replies.
  - confidence: 0.86
- **[P2] Do not infer remote Signal attachments from host name:** `extensions/signal/src/send.ts:163-164`
  - body: The localhost/127.0.0.1 check misclassifies SSH-tunneled or proxied remote native signal-cli daemons as local, so this path can still send file paths that the remote daemon cannot read.
  - confidence: 0.78
- **[P3] Preserve resolved MIME type in data URIs:** `extensions/signal/src/send.ts:16-34`
  - body: localFileToDataUri derives MIME type from the filename even though attachment resolution already has contentType. Extensionless cached files can be sent as application/octet-stream despite a known media type.
  - confidence: 0.78
