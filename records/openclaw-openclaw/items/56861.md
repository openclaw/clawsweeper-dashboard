## Summary

Keep this PR open: the Signal direction is useful, and a narrower open PR now covers the native mention subset at https://github.com/openclaw/openclaw/pull/69413, but this branch still has blocking source-level issues in quoted media ingestion and reply identity matching plus no redacted after-fix Signal runtime proof.

## What This Changes

The branch modifies the Signal plugin event handler/types and send path to add quoted-message media context, structured mention/reply-to-bot handling, and data URI encoding for non-local signal-cli attachments.

## Best Possible Solution

Port the useful Signal changes onto current main using upstream-shaped quote attachments, existing `MediaPaths`/`MediaTypes` ingestion, independent phone/UUID matching, API-mode-owned attachment encoding, focused tests, and redacted live Signal proof.

## Reproduction Assessment

Yes for the blocking PR findings: the PR-head source can be checked against upstream signal-cli quoted attachment types and current-main media normalization. I did not run a live Signal setup in this read-only review.

## Solution Assessment

No. The direction is reasonable, but this patch should not merge until it follows the upstream quote contract, routes quoted media through existing ingestion fields, compares phone and UUID identity separately, and includes real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Read quoted media from the thumbnail:** `extensions/signal/src/monitor/event-handler.ts:797`
  - body: signal-cli models quote attachments as `JsonQuotedAttachment` with the downloadable `JsonAttachment` under `thumbnail`. This line searches for a top-level `id` and passes that wrapper to `fetchAttachment`, so real quoted media is skipped.
  - confidence: 0.9
- **[P2] Route quoted media through MediaPaths:** `extensions/signal/src/monitor/event-handler.ts:219-220`
  - body: `ReplyToMediaPath` and `ReplyToMediaType` are not read by `normalizeAttachments`; media understanding consumes `MediaPath(s)`/`MediaUrl(s)`. A successfully downloaded quoted attachment will not reach model media processing.
  - confidence: 0.92
- **[P2] Compare quote phone and UUID independently:** `extensions/signal/src/monitor/event-handler.ts:660`
  - body: Using `quote.author ?? quote.authorUuid` means `authorUuid` is never checked when `author` is present. UUID-only bot configs can miss replies to the bot and then skip mention-gated group messages.
  - confidence: 0.86
- **[P3] Preserve the resolved MIME type in data URIs:** `extensions/signal/src/send.ts:16-34`
  - body: The helper re-derives MIME type from the filename even though `resolveOutboundAttachmentFromUrl` returns `contentType`. Extensionless cached files can be sent as `application/octet-stream` despite a known type.
  - confidence: 0.78
