## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve adapter-owned attachment encoding:** `extensions/signal/src/send.ts:163-164`
  - body: Current main routes Signal sends through `client-adapter.ts`, and container mode already converts file-path attachments to base64 data URIs in `client-container.ts`. Pre-converting attachments here based on `baseUrl` would hand a `data:` string to container mode, whose `filesToBase64DataUris` still calls `fs.readFile` on every attachment, so remote container sends fail instead of becoming compatible.
  - confidence: 0.9
- **[P2] Route quoted media through the inbound media pipeline:** `extensions/signal/src/monitor/event-handler.ts:869-871`
  - body: This stores downloaded quoted attachments only in `ReplyToMediaPath` and `ReplyToMediaType`, but current media understanding consumes `MediaPath`/`MediaPaths` or supplemental quote media resolved by `buildChannelInboundEventContext`. As written, prompts about a quoted image still will not receive that image unless a custom template reads the new ad-hoc keys.
  - confidence: 0.88
- **[P2] Compare quote author number and UUID independently:** `extensions/signal/src/monitor/event-handler.ts:660-661`
  - body: `quote.author ?? quote.authorUuid` discards `authorUuid` whenever Signal includes both fields. In a UUID-only bot identity setup, a reply to the bot can be misclassified as not an implicit mention because the phone value is compared first and the UUID path is never checked, causing mention-gated group replies to be skipped.
  - confidence: 0.84
