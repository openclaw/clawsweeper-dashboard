## Summary

Keep open: current main does not implement the requested Signal reply-media and remote attachment behavior, and the PR still has merge-blocking source-level defects plus missing real behavior proof.

## What This Changes

The PR adds Signal quoted-message media context, structured Signal mention matching, reply-to-bot mention handling, and data-URI attachment sends for remote signal-cli setups.

## Best Possible Solution

Port the useful Signal behavior onto current main by using the upstream quoted-attachment shape, appending quoted media to MediaPaths/MediaTypes, comparing quote phone and UUID independently, preserving local file-path sends, and adding focused Signal tests plus real redacted runtime proof.

## Reproduction Assessment

Yes for the review findings: source inspection shows the quoted attachment shape mismatch, the ReplyToMediaPath ingestion gap, and the quote author/UUID short-circuit. I did not run a live Signal setup in this read-only review.

## Solution Assessment

No. The direction is useful, but this branch is not the best mergeable solution because it models upstream quoted attachments incorrectly, bypasses the established media ingestion contract, and leaves UUID-only reply-to-bot detection fragile.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Read quoted media from the nested thumbnail:** `extensions/signal/src/monitor/event-handler.ts:760`
  - body: signal-cli exposes quoted attachments as entries with an optional nested `thumbnail`, not a top-level attachment `id`. Because this scans only `quote.attachments[].id`, quoted media from real Signal payloads will not be fetched, so the headline reply-target media feature still fails.
  - confidence: 0.9
- **[P2] Route quoted media through MediaPaths:** `extensions/signal/src/monitor/event-handler.ts:215-217`
  - body: The PR stores fetched quoted media only in `ReplyToMediaPath` and `ReplyToMediaType`, but the media-understanding pipeline reads `MediaPath(s)` and `MediaUrl(s)`. As written, even successfully fetched quoted media will be absent from model media ingestion.
  - confidence: 0.92
- **[P2] Compare quote phone and UUID separately:** `extensions/signal/src/monitor/event-handler.ts:657`
  - body: `quote.author ?? quote.authorUuid` ignores the UUID whenever the deprecated author field is present. In UUID-only bot configurations, a reply to the bot can be treated as not mentioning the bot and skipped by mention gating; compare the phone/author and UUID fields independently.
  - confidence: 0.86
- **[P3] Use the resolved MIME type for data URIs:** `extensions/signal/src/send.ts:16-35`
  - body: `localFileToDataUri` re-derives the MIME type from the local filename even though `resolved.contentType` is available at the call site. Temp or extensionless files can be sent as `application/octet-stream` despite a known content type, so thread the resolved type into the helper.
  - confidence: 0.78
