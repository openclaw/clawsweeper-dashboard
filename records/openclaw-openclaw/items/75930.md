## Summary

Keep open. Current main contains the prior #73198 bounded-queue mitigation, but cron Dream Diary generation still detaches narrative work and permits three simultaneous subagent LLM calls, so the reported strict-provider failure remains plausible and narrow enough to fix.

## What This Changes

Make cron Dream Diary narrative generation provider-safe across multiple workspaces instead of allowing concurrent detached LLM calls to rate-limit most narratives.

## Best Possible Solution

Make detached Dream Diary generation serialize by default, or add a narrowly scoped memory-core limiter/retry path that defaults safely for one-request providers.

## Reproduction Assessment

Yes. Configure multiple dreaming workspaces on a provider that rejects concurrent same-key requests, then trigger the cron path; current source allows three detached narrative subagent runs at once.

## Solution Assessment

No. The current three-wide limiter is a useful mitigation for unbounded fan-out, but it is not the narrowest safe solution for providers that only tolerate one active request per key.

Fix the Dream Diary cron narrative concurrency reported in https://github.com/openclaw/openclaw/issues/75930. Current main still detaches cron narrative generation and caps runDetachedDreamNarrative at three in-flight subagent calls, which can rate-limit strict providers across multiple workspaces. Keep the raw dreaming phase/workspace behavior intact, but make detached Dream Diary narrative generation provider-safe by serializing by default or adding a memory-core-local limiter/retry path that defaults to one active provider call. Avoid broad provider/core scheduler changes and do not add bundled provider-specific IDs. Update the focused detached narrative test that currently expects three in-flight calls, add/adjust coverage for multiple detached calls not exceeding the safe default, and preserve best-effort logging semantics.
