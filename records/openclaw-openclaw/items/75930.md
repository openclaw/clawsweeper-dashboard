## Summary

Keep open. Current main has the earlier unbounded-fanout mitigation, but cron Dream Diary generation still allows three concurrent detached subagent/provider runs, so the reported strict-provider failure remains plausible and actionable.

## What This Changes

The issue requests provider-safe Dream Diary narrative scheduling for cron-triggered multi-workspace dreaming so LLM narratives are not lost under concurrent same-key rate limits.

## Best Possible Solution

Make memory-core cron Dream Diary narrative generation provider-safe by default, preferably by serializing detached narrative provider calls or adding a narrowly scoped limiter that defaults to one without changing dreaming phase semantics.

## Reproduction Assessment

Yes. Configure multiple dreaming workspaces on a provider that rejects concurrent same-key requests and trigger the cron path; current main still permits three in-flight detached narrative runs, and the focused test asserts that behavior.

## Solution Assessment

Yes for a localized memory-core limiter/default change. Batching all workspaces into one diary call would change per-workspace output semantics, while broad provider retry/failover work is wider than this bug needs.
