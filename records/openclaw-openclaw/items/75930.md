## Summary

Keep open. Current main has the earlier unbounded fan-out mitigation, but the cron Dream Diary path still starts up to three detached narrative subagent runs at once, so the reported strict-provider rate-limit failure remains plausible and actionable.

## What This Changes

The issue requests provider-safe Dream Diary narrative scheduling for multi-workspace cron dreaming so all eligible workspaces get narrative entries instead of losing most calls to provider concurrency limits.

## Best Possible Solution

Make cron Dream Diary narrative generation provider-safe by serializing detached narrative calls by default, or by adding a narrowly scoped configurable limiter that defaults to one in-flight request.

## Reproduction Assessment

Yes. Configure multiple memory-core workspaces on a provider that rejects concurrent same-key requests, then trigger the cron dreaming path; current source starts up to three detached narrative subagent runs at once.

## Solution Assessment

Yes for the serialized/default-one limiter direction. The existing three-wide cap is useful for unbounded fan-out but is not sufficient for the provider class described here, while batching all workspaces would be a larger product change.
