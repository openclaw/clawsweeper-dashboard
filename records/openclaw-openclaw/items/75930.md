## Summary

Keep open. Current main still detaches cron Dream Diary narratives and allows three concurrent subagent/provider runs, so the prior unbounded-fanout fix does not fully address strict same-key provider rate limits across multiple workspaces.

## What This Changes

The issue asks memory-core cron Dream Diary narrative generation to avoid concurrent detached provider/subagent calls so every configured workspace can receive a `DREAMS.md` narrative entry.

## Best Possible Solution

Make cron Dream Diary narrative generation provider-safe in memory-core, preferably by serializing detached narrative subagent runs by default while preserving per-workspace and per-phase diary semantics.

## Reproduction Assessment

Yes. Configure multiple dreaming workspaces on a provider that rejects concurrent same-key requests and trigger the cron path; current main still permits three in-flight detached narrative runs and has a test asserting that concurrency.

## Solution Assessment

Yes for a localized memory-core limiter/default change. Batching workspaces into one diary call would alter output semantics, while broad provider retry/failover work is wider than this bug needs.
