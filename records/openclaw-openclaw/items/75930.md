## Summary

Keep open. Current `main` still allows three detached Dream Diary narrative subagent runs at once, so the strict provider rate-limit failure mode described here remains unresolved despite the earlier unbounded-fanout cap.

## What This Changes

The issue asks memory-core to make cron Dream Diary narrative generation provider-safe across multiple agent workspaces instead of launching concurrent detached narrative calls.

## Best Possible Solution

Serialize detached Dream Diary narrative runs by default, or enforce a provider-safe limiter that defaults to one concurrent narrative call while preserving per-workspace diary entries and existing phase output semantics.

## Reproduction Assessment

Yes, from source inspection rather than a live provider run. Configure multiple dreaming workspaces on a provider that rejects concurrent same-key requests and trigger the cron path; current `main` can still start three detached narrative subagent calls at once, and the focused test asserts that behavior.

## Solution Assessment

Yes. A localized memory-core serialization or provider-safe limiter is narrower than batching all workspaces into one diary entry or changing generic provider retry policy, and it directly addresses the remaining concurrency failure mode.
