## Summary

Keep open. Current main and the latest release no longer have unbounded 20+ narrative fan-out, but they still allow three detached Dream Diary subagent runs at once with a fixed 60s wait and no public concurrency/timeout setting, so the serial-provider failure reported here is only partially mitigated; the related timeout PR does not settle the concurrency policy.

## What This Changes

The issue asks OpenClaw to make cron-driven multi-workspace Dream Diary narrative generation reliable instead of silently missing `DREAMS.md` entries under concurrent provider calls.

## Best Possible Solution

Keep this as the canonical serial-provider Dream Diary concurrency item, decide the concurrency/timeout contract, and land one bounded memory-core change with schema/docs/tests coordinated with https://github.com/openclaw/openclaw/pull/78440.

## Reproduction Assessment

No high-confidence live reproduction was established in this review. The source path is clear on current main: cron sweeps detach narrative runs and the global queue still allows three concurrent subagent calls, but provider-specific rate limiting was not exercised.

## Solution Assessment

Unclear. The reported direction is valid, but the best fix needs a maintainer decision between a safer serial default and a user-facing concurrency/timeout config surface.
