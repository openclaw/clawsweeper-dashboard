## Summary

Keep open. Current main and the latest release include the earlier unbounded-fanout cap, but cron Dream Diary generation still allows three detached narrative subagent runs at once, which leaves the reported strict-provider failure mode unresolved for multi-workspace sweeps.

## What This Changes

The issue asks memory-core cron Dream Diary generation to stop launching concurrent detached LLM narrative calls across multiple agent workspaces so each workspace gets narrative output.

## Best Possible Solution

Serialize detached Dream Diary narrative runs by default, or otherwise enforce provider-safe concurrency in memory-core, while preserving per-workspace diary entries and existing phase output semantics.

## Reproduction Assessment

Yes, from source inspection rather than a live provider run. Configure multiple dreaming workspaces on a provider that rejects concurrent same-key requests and trigger the cron path; current main can still start three detached narrative subagent calls at once, and the focused test asserts that behavior.

## Solution Assessment

Yes. A localized memory-core serialization or provider-safe limiter is narrower than batching all workspaces into one diary entry or changing generic provider retry policy, and it directly addresses the remaining concurrency failure mode.
