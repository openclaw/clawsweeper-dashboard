## Summary

Keep open. Current main still detaches cron Dream Diary narratives and caps them at three concurrent runs, so the reported one-active-request provider failure remains plausible despite the earlier unbounded-fanout mitigation.

## What This Changes

The issue asks memory-core to stop cron-triggered Dream Diary narrative generation from launching concurrent detached LLM calls across workspaces because strict providers rate-limit most of them.

## Best Possible Solution

Default detached Dream Diary scheduling to one in-flight narrative, or expose a narrow limiter that defaults to one with retry/backoff, while keeping non-cron behavior unchanged.

## Reproduction Assessment

Yes. Configure multiple memory workspaces on a provider that rejects more than one same-key request, then trigger the cron path; current main can start up to three detached narrative subagent runs at once.

## Solution Assessment

Yes for the serialization/limiter direction. A provider-safe limiter defaulting to one is narrower than batching all workspaces and safer than the current three-wide default for strict providers.
