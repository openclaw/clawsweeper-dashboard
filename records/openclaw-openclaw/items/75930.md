## Summary

Keep open. Current main and the latest release still allow three detached Dream Diary narrative subagent runs at once, so strict same-key providers can still hit the reported failure mode even though the earlier unbounded fan-out was capped.

## What This Changes

Serialize or otherwise provider-safely limit cron Dream Diary narrative generation so multi-workspace sweeps do not launch concurrent same-key narrative calls that drop most diary entries.

## Best Possible Solution

Serialize detached Dream Diary narrative subagent runs by default, or use an equivalent provider-safe limiter defaulting to one, while preserving per-workspace diary entries and existing phase output semantics.

## Reproduction Assessment

Yes, from source inspection rather than a live provider run. Configure multiple dreaming workspaces on a provider that rejects concurrent same-key requests and trigger the cron path; current main can still start three detached narrative subagent calls at once, and the focused test asserts that behavior.

## Solution Assessment

Yes. A localized memory-core serialization or provider-safe limiter is narrower than batching all workspaces into one diary entry or changing generic provider retry behavior.
