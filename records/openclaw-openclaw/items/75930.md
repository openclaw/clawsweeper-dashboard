## Summary

Keep open. Current main mitigates the older unbounded fanout bug, but cron Dream Diary generation still allows three detached provider/subagent runs at once, so the strict-provider same-key rate-limit failure remains plausible and actionable.

## What This Changes

The issue requests making cron Dream Diary narrative generation provider-safe across multiple workspaces instead of allowing detached concurrent narrative calls that can rate-limit after the first run.

## Best Possible Solution

Make detached cron Dream Diary generation provider-safe in memory-core, preferably by serializing by default or adding a narrowly scoped limiter that defaults to one without changing phase semantics.

## Reproduction Assessment

Yes. Configure multiple dreaming workspaces on a provider that rejects concurrent same-key requests, then trigger the cron path; current main still permits three in-flight detached narrative runs and has a test asserting that limit.

## Solution Assessment

Yes for a localized memory-core limiter/default change. Batching all workspaces into one diary call would change per-workspace diary semantics, and broad provider retry work is wider than this bug needs.
