## Summary

Keep open. Current main has the earlier unbounded-fanout mitigation, but detached Dream Diary generation still allows three concurrent subagent/provider runs, so the reported strict-provider failure remains plausible and actionable.

## What This Changes

The issue asks memory-core dreaming to avoid firing concurrent cron Dream Diary narrative API calls across multiple workspaces because most calls are rate-limited and dropped.

## Best Possible Solution

Make cron Dream Diary narrative generation provider-safe in memory-core, preferably by defaulting detached narrative concurrency to one or by adding a narrowly scoped limiter that defaults to one-request providers without changing dreaming phase semantics.

## Reproduction Assessment

Yes. Configure multiple dreaming workspaces on a provider that rejects concurrent same-key requests, then trigger the cron path; current main still permits three in-flight detached narrative runs and has a test asserting that behavior.

## Solution Assessment

Yes for a localized memory-core limiter/default change. Batching all workspaces into one diary call would alter output semantics, while broad provider retry/failover changes are wider than this bug needs.
