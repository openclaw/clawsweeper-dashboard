## Summary

Keep open. Current main has the earlier unbounded-fanout mitigation, but cron Dream Diary generation still permits three concurrent detached subagent/provider calls, so strict same-key providers can still rate-limit multi-workspace sweeps.

## What This Changes

Request provider-safe Dream Diary narrative scheduling for cron dreaming sweeps so multiple workspaces do not run concurrent LLM narrative calls that rate-limit.

## Best Possible Solution

Make cron Dream Diary narrative generation provider-safe in memory-core, preferably by serializing detached narrative subagent runs by default while preserving existing phase outputs and model configuration semantics.

## Reproduction Assessment

Yes. Configure multiple dreaming workspaces on a provider that rejects concurrent same-key requests and trigger the cron path; current main still permits three in-flight detached narrative runs and has a test asserting that behavior.

## Solution Assessment

Yes for a localized memory-core limiter/default change. Batching all workspaces into one diary call would alter output semantics, while broad provider retry/failover changes are wider than this bug needs.
