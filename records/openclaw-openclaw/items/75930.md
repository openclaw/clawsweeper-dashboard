## Summary

Close as implemented. Current main and the shipped 2026.4.27 release route detached light/REM/deep Dream Diary generation through a shared FIFO cap of three, increase the narrative timeout to 60 seconds, and test the bounded queue/drain path, so the unbounded fire-and-forget fan-out described here is no longer current behavior.

## What This Changes

Request to serialize or limit cron-triggered Dream Diary narrative generation so multi-workspace dreaming does not fire provider calls simultaneously and silently lose entries.

## Best Possible Solution

Keep the shipped bounded detached-narrative queue; if a specific provider still needs one-at-a-time calls after upgrading, track that as a narrower configurable-concurrency or retry issue with provider logs.

## Reproduction Assessment

Yes. The report gives a concrete cron plus multi-workspace Dream Diary path, and current tests model the formerly unbounded detached queue by scheduling five narratives and proving only three can run at once.

## Solution Assessment

Yes. The existing fix targets the root cause by bounding detached cron narratives globally while preserving background cron behavior and avoiding a larger provider-specific retry system.
