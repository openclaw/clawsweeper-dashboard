## Summary

Keep open. Current main has the earlier #73198 mitigation, but it still allows three detached Dream Diary subagent runs at once, so the reported strict-provider failure remains plausible and actionable for providers that accept only one active request per key.

## What This Changes

The issue requests that cron-driven Dream Diary narrative generation avoid concurrent fire-and-forget LLM calls across multiple agent workspaces.

## Best Possible Solution

Make cron Dream Diary narrative generation provider-safe by serializing detached narrative calls by default, or by adding a narrowly scoped configurable limiter with retry/backoff that defaults safely for one-request providers.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: configure multiple dreaming workspaces on a provider that rejects concurrent same-key requests, then trigger the cron path that permits three in-flight detached narrative subagent runs.

## Solution Assessment

No. The existing three-wide limiter is a good fix for unbounded fan-out, but it is not sufficient for the reported provider class; a default concurrency of one or equivalent retrying limiter is the safer narrow repair.

## Repair Work Prompt

Fix the memory-core Dream Diary cron concurrency bug reported in https://github.com/openclaw/openclaw/issues/75930. Current main already caps detached Dream Diary narratives at 3 in `extensions/memory-core/src/dreaming-narrative.ts`, but strict providers such as DeepSeek/Moonshot can still reject concurrent calls from the same key, so cron dreaming across multiple workspaces should not run multiple narrative subagent/model calls at once by default. Keep the change narrowly inside memory-core dreaming narrative scheduling; do not change the core cron scheduler, provider SDKs, or workspace discovery.
