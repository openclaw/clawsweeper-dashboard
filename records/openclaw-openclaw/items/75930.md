## Summary

Keep open. Current main has the prior unbounded-fanout mitigation, but cron Dream Diary generation still permits three concurrent detached narrative subagent/provider calls, so strict same-key providers can still rate-limit multi-workspace sweeps.

## What This Changes

The issue requests making cron Dream Diary narrative generation safe for multi-agent provider rate limits by avoiding simultaneous detached narrative API calls across workspaces.

## Best Possible Solution

Make memory-core's cron Dream Diary queue provider-safe by serializing detached narrative subagent runs by default while preserving existing phase output, timeout, cleanup, and model fallback behavior.

## Reproduction Assessment

Yes, source-reproducible. Configure multiple dreaming workspaces on a provider that rejects same-key concurrent requests and trigger the cron path; current main still schedules detached narrative runs through a global queue with concurrency 3.

## Solution Assessment

Yes, a localized memory-core limiter/default change is the narrowest maintainable fix. Batching all workspaces into one diary call would alter output semantics, and broad provider retry/failover changes are larger than this bug needs.
