## Summary

Keep open. Current main and the latest release still keep the bundled session-memory archive on `/new` and `/reset` only; compaction has a separate lifecycle hook and memory-flush path, but neither writes the requested raw session-memory archive on compact.

## What This Changes

The issue requests that session-memory preservation run when compaction happens, not only on new/reset session commands.

## Best Possible Solution

Decide whether session-memory should complement memoryFlush and recent-continuity work; if yes, wire it through session:compact:before with the required workspace/session context, focused tests, and docs.

## Reproduction Assessment

Yes for the behavior gap. Source inspection shows session-memory handles only command:new and command:reset, while /compact goes through the separate compaction runtime and session:compact:before lifecycle.

## Solution Assessment

No for the literal command:compact-only proposal. If maintainers want this behavior, session:compact:before is the maintainable hook point so manual, automatic, and engine-owned compaction share one pre-compaction archival path.
