## Label Justifications

- `P2`: The issue describes a real memory-continuity improvement with meaningful user impact, but it is a product feature decision rather than an urgent regression.
- `impact:data-loss`: The reported gap can leave valuable prior conversation context unwritten before compaction changes transcript state.
- `impact:session-state`: The affected surface is session transcript compaction, memory persistence, and continuity across session state changes.

## Summary

Keep open. Current main and v2026.5.20 still keep the bundled session-memory archive scoped to `/new` and `/reset`; compaction emits separate lifecycle hooks, but the existing compact-before payload and closed continuity PR do not implement the requested archive-on-compact behavior.

## What This Changes

Request that bundled session-memory capture run before `/compact` or compaction so valuable transcript context is saved before compression.

## Best Possible Solution

Define one compact-time memory-continuity contract, likely on `session:compact:before`, then wire session-memory through it with the necessary workspace/session context, duplicate-write guards, focused tests, and docs.

## Reproduction Assessment

Yes, source-level. Current main shows session-memory only accepts `command:new` and `command:reset`, while `/compact` goes through the compaction runtime and `session:compact:before` lifecycle without writing the requested archive.

## Solution Assessment

No for the literal `command:compact`-only proposal. If maintainers want this behavior, `session:compact:before` or a dedicated memory lifecycle contract is the maintainable hook point so manual, automatic, and engine-owned compactions share one path.
