## Summary

Keep open: current main and v2026.6.10 still keep the bundled session-memory archive hook limited to `/new` and `/reset`, while compaction uses the separate `session:compact:before/after` lifecycle. The request remains useful, but the one-line `command:compact` proposal is not the right implementation shape without a maintainer decision about compact-time archival semantics and hook context.

## What This Changes

Request to trigger session-memory preservation when a session is compacted so valuable pre-compaction context is archived automatically.

## Best Possible Solution

Keep this issue open until maintainers choose the compact-time memory contract, then implement a narrow path that either teaches session-memory to consume `session:compact:before` with the needed context or adds a dedicated memory lifecycle seam with docs and regression coverage.

## Reproduction Assessment

Yes for the source-level gap: current main registers and handles session-memory only for `command:new` and `command:reset`, while compaction emits `session:compact:before/after`. I did not run a live compact session because this is a read-only review and the source/test contracts are enough to show the missing trigger.

## Solution Assessment

No: adding `command:compact` alone is not the best way to solve it because current compaction plumbing is session lifecycle based and also covers automatic compaction. The maintainable solution should use `session:compact:before` or a dedicated memory lifecycle API with the transcript/workspace context session-memory needs.
