## Label Justifications

- `impact:session-state`: The requested behavior concerns preserving session context before compaction changes the active transcript state.
- `impact:data-loss`: Users relying on session-memory archives can miss valuable pre-compaction details because the current hook does not run at compact time.

## Summary

Keep open: current main still limits the bundled session-memory archive hook to `/new` and `/reset`, while compaction uses separate `session:compact:*` lifecycle events and no merged or viable canonical item now owns this exact compact-time archive behavior.

## What This Changes

The issue requests adding compact-time triggering to the bundled session-memory hook so valuable transcript context is archived before `/compact` alters the session.

## Best Possible Solution

Define the product contract for compact-time memory archival, then wire session-memory through the existing `session:compact:before` lifecycle if maintainers want automatic pre-compaction persistence.

## Reproduction Assessment

Yes for source reproduction: current main registers and handles session-memory only for `command:new` and `command:reset`, while compaction emits `session:compact:before/after` through a different lifecycle path.

## Solution Assessment

Unclear as proposed: adding the behavior may be useful, but the event should likely be `session:compact:before` rather than a new `command:compact`, and maintainers need to decide whether automatic archive writes during compaction are the right default.
