## Summary

Keep open: current main and v2026.6.10 still limit the bundled session-memory archive hook to `/new` and `/reset`, while manual and Gateway compaction use the existing `session:compact:before/after` lifecycle. The preservation request remains useful, but the event contract and retention semantics need maintainer product review rather than a conservative cleanup close.

## What This Changes

Request that the bundled session-memory hook archive session context when `/compact` runs, so pre-compaction conversation details are persisted without requiring `/new` or `/reset`.

## Best Possible Solution

Define the compact-time memory archival contract, then wire session-memory through `session:compact:before` with metadata, docs, regression tests, and explicit retention/dedupe behavior if maintainers want automatic persistence.

## Reproduction Assessment

Yes, at source level: session-memory metadata and handler logic only register and accept `/new` and `/reset`, while compaction currently emits `session:compact:before/after`. I did not run a live compaction because this review is read-only.

## Solution Assessment

Unclear as proposed: the preservation gap is real, but `command:compact` is not the current lifecycle surface. The maintainable fix is likely a maintainer-approved `session:compact:before` session-memory contract rather than a new command event alias.
