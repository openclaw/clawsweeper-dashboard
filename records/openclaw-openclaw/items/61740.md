## Summary

Keep open. Current main has adjacent skill inventory, per-agent allowlists, hot skill snapshot refresh, and ClawHub/operator search, but it does not implement the requested agent-callable runtime skill discovery tool or structured SKILL.md capability metadata. The remaining request is broad enough to need maintainer product/API design rather than conservative cleanup.

## What This Changes

The issue requests runtime skill discovery, richer skill metadata, and tiered skill loading so agents can find relevant skills after session start.

## Best Possible Solution

Keep this open for maintainer design. A good end state would be a documented, backward-compatible discovery contract that lets agents search eligible installed skills at runtime, optionally bridges to ClawHub metadata, preserves per-agent allowlists, and treats third-party skill discovery/install as an explicit security boundary.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. The current-main gap is reproducible by inspection: skills are surfaced through a prompt snapshot and operator Gateway/CLI methods, with no registered agent `skill_search` tool or SKILL.md capability schema.

## Solution Assessment

Unclear. The proposal identifies a real gap, but it combines runtime search, semantic ranking, metadata schema, lazy loading, and tier policy; the best implementation likely needs maintainer API/product review before a narrow PR can be safely scoped.
