## Summary

Keep open. Current main has load-time skill snapshots, prompt XML, per-agent filters, hot refresh, and operator-facing ClawHub search, but it still lacks the requested agent-callable runtime skill matcher and structured skill capability metadata. The newer user comment is useful demand signal, and related open issues cover narrower pieces rather than superseding this core runtime-discovery request.

## What This Changes

The issue requests runtime skill discovery/matching, structured skill capability metadata, and tiered skill loading so agents can find relevant skills after session startup.

## Best Possible Solution

Design an additive, backward-compatible skill matching contract that preserves agent allowlists, separates installed/local/ClawHub sources, defines trusted metadata semantics, and splits tiering or prompt-budget work into narrower follow-ups if needed.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source inspection gives a high-confidence current-main gap: there is no skill_search/skill_match agent tool, and the existing skills.search RPC is an operator ClawHub search endpoint.

## Solution Assessment

Unclear. The proposal identifies a real gap, but it combines runtime matching, semantic ranking, SKILL.md metadata schema, lazy loading, and tier policy, so maintainers should define the API and trust boundary before a narrow implementation is scoped.
