## Summary

Keep open. Current main still exposes skills through load-time snapshots and prompt XML, plus operator-facing ClawHub search, but it does not provide the requested agent-callable runtime skill matcher or structured routing metadata; the later router prototype was closed as ClawHub/plugin scope and explicitly leaves this core API discussion open.

## What This Changes

Request to add runtime skill discovery/matching so agents can find relevant skills dynamically instead of relying only on the static available-skills prompt.

## Best Possible Solution

Design an additive runtime skill-matching contract that preserves per-agent allowlists, separates installed/local/ClawHub sources, defines trusted metadata semantics, and splits tiering or prompt-budget work into narrower follow-ups if needed.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/API request. Static inspection gives a high-confidence current-main gap: there is no agent-callable skill_search/skill_match tool, and the existing skills.search RPC is operator ClawHub search.

## Solution Assessment

Unclear. The proposal identifies a real gap, but it combines runtime matching, semantic ranking, metadata schema, lazy loading, and tier policy, so maintainers should define the API and trust boundary before implementation.
