## Summary

Keep open: current main has adjacent OpenRouter billed-total reconciliation, transcript-backed aggregation, and estimated-cost display, but it still does not expose provider-reported OpenRouter `usage.cost` or `cost_details` as stable agent runtime, `session_status`, or reply-usage metadata.

## What This Changes

Expose per-message OpenRouter cost data to agents and session surfaces so replies, including Telegram replies, can show exact provider-reported cost.

## Best Possible Solution

Define a provider-reported cost metadata contract with explicit source and units, capture it at the OpenRouter/OpenAI-compatible boundary, persist it through transcript/session/runtime metadata, and expose it through `session_status` or a dedicated usage tool for channels to render.

## Reproduction Assessment

Yes, at source level. Current main exposes token counters, local estimated-cost display, and OpenRouter billed-total reconciliation, but provider-reported `usage.cost` and `cost_details` are not carried into agent metadata or `session_status` details.

## Solution Assessment

Unclear until maintainers choose the public metadata contract. The maintainable solution is a provider-neutral cost/source/units schema first, with Telegram rendering as a consumer rather than the primary implementation boundary.
