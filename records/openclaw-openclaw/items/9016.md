## Summary

Keep open: current main still normalizes OpenRouter/OpenAI-compatible token usage and local estimated USD cost, but it does not expose OpenRouter provider-reported `usage.cost` or `cost_details` through agent metadata, session status, or response-footer data. The closed local-estimate work and the related open resolved-model and billed-cost issues are adjacent, not replacements for this runtime metadata request.

## What This Changes

Request to capture OpenRouter provider-reported usage cost fields and expose them through agent/session runtime surfaces.

## Best Possible Solution

Define an additive provider-reported usage-cost contract at the OpenAI-compatible/OpenRouter boundary, keep exact OpenRouter charges separate from `estimatedCostUsd`, and thread per-turn plus cumulative values through agent metadata, transcripts/session rows, `session_status` or a usage tool, and optional channel footers.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature request. Source inspection gives high-confidence proof that current main does not expose OpenRouter `usage.cost` or `cost_details` through agent metadata, session state, or response footer metadata.

## Solution Assessment

Yes on the requested direction, but not as a Telegram footer-only patch. The maintainable solution is an additive provider-cost metadata contract with explicit source and units, coordinated with the related OpenRouter resolved-model and billed-cost issues.
