## Label Justifications

- `P2`: This is a normal-priority provider observability enhancement with a bounded but cross-cutting runtime/session contract.
- `impact:session-state`: The requested per-call and cumulative charge values must be carried through agent and persisted session state.
- `impact:auth-provider`: The missing exact charge and upstream cost details originate in OpenRouter provider responses.

## Summary

Current main and the latest release still expose locally estimated cost, not OpenRouter’s provider-reported per-response charge; related issues cover broader token/model observability but do not own this exact metadata gap.

## What This Changes

Expose OpenRouter’s provider-reported per-response cost to agent/session runtime surfaces for cost-aware replies and tracking.

## Best Possible Solution

Define one provider-neutral reported-cost record at the response boundary, preserve source and units separately from estimated USD cost, aggregate it through canonical run/session usage, and expose it through the shared status/tool contract before adding optional channel presentation.

## Reproduction Assessment

Yes at source level: current main and v2026.6.5 route OpenRouter usage through token normalization and configured-price calculation without retaining the documented exact charge fields. A billed live request was not necessary to establish the missing data path, but should be part of implementation proof.

## Solution Assessment

No, implementing only an automatic Telegram footer would put provider accounting policy in the wrong layer. The narrow maintainable solution is a generic reported-cost metadata contract that status tools and channels can consume.
