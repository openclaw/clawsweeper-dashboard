## Summary

Keep open: current main has local estimated-cost surfaces, but it still lacks a provider-reported OpenRouter cost contract exposed to agent runtime or session metadata.

## What This Changes

Expose OpenRouter per-response usage cost, cost details, and cumulative per-turn cost data to agents so channels such as Telegram can show exact provider-reported costs.

## Best Possible Solution

Add a typed provider-reported usage field at the OpenAI-compatible response boundary, keep it separate from estimated USD cost, persist it through transcript/session metadata, and expose per-turn plus cumulative values through `session_status` or a dedicated usage tool with estimate fallback.

## Reproduction Assessment

Yes, as a source-level feature gap: current main routes OpenRouter through the shared OpenAI-compatible parser and does not carry `usage.cost` or `cost_details` into agent/session metadata. I did not run a live OpenRouter call in this read-only review.

## Solution Assessment

Yes directionally, but the best fix is a typed provider-reported usage contract rather than overloading `estimatedCostUsd` or the existing local cost object.
