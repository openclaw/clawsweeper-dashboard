## Summary

Keep this issue open. Current main captures and reconciles authoritative OpenRouter cost on the final assistant-message usage object, but the generic agent metadata and session persistence path still reduce usage to token counters and locally estimated USD cost; the linked issues cover adjacent model-resolution or broader session-usage work rather than this exact provider-reported-cost contract.

## What This Changes

Expose provider-reported OpenRouter per-response cost data to agent runtime and session usage surfaces.

## Best Possible Solution

Keep OpenRouter-specific reconciliation in the plugin, then carry a distinct source-tagged authoritative cost value through the established assistant usage, agent metadata, transcript, and session-status boundaries while preserving estimatedCostUsd as an explicitly separate fallback.

## Reproduction Assessment

Yes, at source level. Current main demonstrably places authoritative OpenRouter cost on final assistant-message usage and then converts runtime metadata and session persistence to token-only NormalizedUsage plus a local estimate, directly exposing the requested gap.

## Solution Assessment

The requested direction is valid, but appending Telegram text directly or overloading estimatedCostUsd would not be the best fix. The maintainable solution is to reuse the existing plugin-reconciled value through a generic, provenance-aware monetary usage contract shared by runtime and session consumers.
