## Summary

Keep this issue open. Current main now reconciles OpenRouter’s authoritative billed total into the final assistant-message usage object, but generic agent metadata still strips monetary fields and session persistence still calculates only a local estimate; the related issues cover broader session usage or resolved-model metadata rather than this exact provider-cost contract.

## What This Changes

Expose OpenRouter’s provider-reported per-response cost to agent runtime and session-visible usage surfaces.

## Best Possible Solution

Add a generic, source-labeled provider-cost shape to canonical assistant usage, preserve it through run aggregation and session persistence, and expose per-call and cumulative values through the shared usage/session tool contract; channels should only format that shared data when requested.

## Reproduction Assessment

Yes at source level. Current main clearly captures the authoritative OpenRouter total on assistant usage, then reduces downstream agent and session data to token counters and locally estimated cost; this is a missing capability rather than a failing documented contract.

## Solution Assessment

The requested outcome is valid, but a provider-specific Telegram footer is not the best core fix. The narrow maintainable solution is a generic provider-cost contract with explicit provenance and aggregation semantics, consumed by session tools and presentation layers.
