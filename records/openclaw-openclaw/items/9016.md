## Summary

Keep open: current main and v2026.6.8 still only provide local estimated cost and partial OpenRouter total-cost reconciliation, not a source/units-aware OpenRouter provider-reported cost surface for agent runtime or session status.

## What This Changes

Expose OpenRouter per-response usage cost and cost details to agents, sessions, and user-facing reply/status surfaces.

## Best Possible Solution

Define a provider-reported cost metadata contract with explicit source and units, capture it at the OpenAI-compatible/OpenRouter boundary, persist it through transcript/session/runtime metadata, and let `session_status` or a usage tool expose it for channels to render.

## Reproduction Assessment

Yes, as a source-level gap: current main exposes token counters and local estimated cost paths, while the OpenRouter raw `usage.cost` and `cost_details` fields are not carried into agent metadata or session status.

## Solution Assessment

Unclear until maintainers choose the provider-neutral metadata schema; the maintainable direction is a canonical runtime/session cost contract first, with Telegram rendering as a consumer of that contract.
