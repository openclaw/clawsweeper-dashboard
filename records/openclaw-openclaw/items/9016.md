## Summary

Keep open: current main still has adjacent estimated-cost reporting, but it does not carry OpenRouter provider-reported per-response cost fields into agent runtime metadata, session status, or response footers.

## What This Changes

The issue requests capturing OpenRouter response `usage.cost` / `cost_details` and exposing per-turn or cumulative exact cost data to agents and channel replies.

## Best Possible Solution

Add a reviewed provider-reported cost metadata contract that preserves OpenRouter exact cost fields separately from `estimatedCostUsd` and exposes per-turn plus cumulative values through session/runtime APIs.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a feature request; source inspection on current main confirms OpenRouter usage is normalized to token counters and local estimated cost without exact provider cost fields reaching agent/session surfaces.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers choose the API contract; the narrow maintainable direction is a distinct provider-reported cost object with explicit source and units rather than overloading existing estimated-cost fields.
