## Summary

Keep open: current main still has estimated-cost reporting and transcript aggregation, but it does not carry OpenRouter provider-reported `usage.cost` or `cost_details` into agent runtime metadata, session rows, or reply footers.

## What This Changes

Expose OpenRouter per-response cost data to agent/session runtime surfaces so replies, including Telegram replies, can show exact provider cost when available.

## Best Possible Solution

Add an additive provider-reported cost metadata contract with explicit source and units, then capture OpenRouter raw usage at the OpenAI-compatible response boundary and thread it through transcripts, session metadata, `EmbeddedPiAgentMeta`, `session_status` or a dedicated cost tool, and channel footers.

## Reproduction Assessment

Not applicable as a feature request. Source inspection on current main gives high-confidence proof that the requested OpenRouter provider-cost fields are not exposed on runtime, session, or footer surfaces.

## Solution Assessment

Unclear until maintainers choose the public metadata and cost-authority contract. The narrow maintainable direction is a distinct provider-reported cost object with explicit source, units, and fallback semantics rather than overloading existing `estimatedCostUsd` fields.
