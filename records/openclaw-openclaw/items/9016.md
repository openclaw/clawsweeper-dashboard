## Summary

Keep open. Current main has estimated-cost and transcript aggregation paths, but it still does not preserve OpenRouter provider-reported `usage.cost`, `cost_details.upstream_inference_cost`, or `prompt_tokens_details.cache_write_tokens` for agent/runtime metadata.

## What This Changes

Expose OpenRouter per-response usage cost data to agent runtime/session metadata so channels such as Telegram can report exact provider costs when available.

## Best Possible Solution

Add an additive provider-reported-cost path that captures OpenRouter raw usage at the OpenAI-compatible response/stream boundary, carries source and units through transcript/session usage and `EmbeddedPiAgentMeta`, and lets channels render exact OpenRouter values when available while keeping local estimates as fallback.

## Reproduction Assessment

Yes, at source level. Feed the shared OpenAI-compatible stream parser a final OpenRouter-style usage chunk with numeric `cost`, `cost_details.upstream_inference_cost`, and `prompt_tokens_details.cache_write_tokens`; current main normalizes tokens and local estimated cost but drops those provider fields.

## Solution Assessment

Yes. The narrow maintainable direction is additive capture and propagation of provider-reported OpenRouter usage metadata, separate from existing `estimatedCostUsd` and local transcript cost estimates.
