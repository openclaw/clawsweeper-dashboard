## Summary

Keep open: current main still has adjacent estimated-cost and usage-summary surfaces, but it does not expose OpenRouter provider-reported per-response `usage.cost` or `cost_details` through agent runtime/session metadata. This remains a feature/API contract decision rather than a stale or already-implemented cleanup item.

## What This Changes

Request to capture OpenRouter per-message usage cost and expose per-turn or cumulative cost data to agents, session status, and chat replies such as Telegram footers.

## Best Possible Solution

Define an additive provider-reported usage-cost contract at the OpenAI-compatible/OpenRouter boundary, keep it separate from `estimatedCostUsd`, and carry per-turn and cumulative values through assistant metadata, session status/tool output, and optional channel footers.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature request. Source inspection gives high-confidence proof that current main does not expose OpenRouter `usage.cost` or `cost_details` through agent meta, session state, or response footers.

## Solution Assessment

Yes on the direction, but not as a footer-only patch. The maintainable solution is an additive provider-cost metadata contract that keeps local estimated USD separate from exact OpenRouter charges and leaves naming/display policy to maintainer review.
