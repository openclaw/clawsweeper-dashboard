## Summary

Keep open: current main has local estimated-cost reporting, cache-write token parsing, and OpenRouter billed-total reconciliation, but it still does not expose OpenRouter provider-reported `usage.cost`, `cost_details.upstream_inference_cost`, source, or units through agent runtime/session metadata. The related open issues overlap on resolved model, auth-profile attribution, or structured usage APIs, but none replaces this provider-specific metadata gap.

## What This Changes

Expose OpenRouter provider-reported per-response cost and cost details to agent/session runtime so agents and channels can report exact per-reply costs.

## Best Possible Solution

Define a provider-reported cost metadata contract with explicit source and units, capture it at the OpenRouter/OpenAI-compatible boundary, persist it through transcript/session/runtime metadata, and expose it through `session_status` or a dedicated usage tool for channels to render.

## Reproduction Assessment

Yes, at source level. Current main exposes token counters, local estimated-cost display, and OpenRouter billed-total reconciliation, but raw OpenRouter `usage.cost` and `cost_details` are not carried into agent metadata or `session_status` details.

## Solution Assessment

Unclear until maintainers choose the public metadata contract. The maintainable solution is a provider-neutral cost/source/units schema first, with Telegram rendering as a consumer rather than the primary implementation boundary.
