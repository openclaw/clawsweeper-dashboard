## Summary

Keep open: current main has local estimated-cost reporting and transcript aggregation, but it still does not carry OpenRouter provider-reported `usage.cost` or `cost_details` into agent runtime metadata, session rows, or response footers. Related #7006, #51441, and #68066 are complementary visibility/cost-authority work, not replacements for this runtime metadata request.

## What This Changes

The issue requests capturing OpenRouter per-response cost fields and exposing them to agents/session surfaces so channels such as Telegram can display exact per-reply cost.

## Best Possible Solution

Define and implement an additive provider-reported cost metadata contract that preserves OpenRouter exact fields separately from local estimates and exposes per-turn plus cumulative values through runtime/session APIs.

## Reproduction Assessment

Not applicable as a feature request. Source inspection on current main gives high-confidence proof that the requested OpenRouter provider-cost fields are not exposed on the runtime, session, or footer surfaces.

## Solution Assessment

Unclear until maintainers choose the public metadata contract. The narrow maintainable direction is a distinct provider-reported cost object with explicit source, units, and fallback semantics rather than overloading existing estimated-cost fields.
