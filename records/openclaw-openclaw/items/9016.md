## Summary

Keep open: current main still has local estimated-cost plumbing, but OpenRouter provider-reported `usage.cost` and `cost_details.upstream_inference_cost` are not captured into agent runtime metadata, session status, or reply-footer inputs. The related local estimate, billed-cost, and resolved-model items are adjacent rather than replacements for this provider-cost runtime contract request.

## What This Changes

Expose OpenRouter per-response usage cost data to agent runtime and session metadata so agents or channels can report exact provider charges per reply.

## Best Possible Solution

Define an additive provider-reported cost contract at the OpenAI-compatible/OpenRouter boundary, keep exact provider charges separate from local `estimatedCostUsd`, and thread per-turn plus cumulative values through agent metadata, transcript/session usage, `session_status` or a usage tool, and optional channel footers.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature request, but source inspection gives high-confidence proof of the current gap. Current main and v2026.5.12 do not expose OpenRouter `usage.cost` or `cost_details` through agent metadata, session state, or response footer metadata.

## Solution Assessment

Yes on the requested direction, but not as a Telegram-footer-only patch. The maintainable solution is an additive provider-cost metadata contract with explicit source and units, coordinated with existing local estimated-cost, billed-cost reconciliation, and resolved-model surfaces.
