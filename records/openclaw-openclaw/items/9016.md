## Summary

Keep open: current main and v2026.6.8 still provide local estimated-cost and OpenRouter total-cost reconciliation, but not a provider-reported OpenRouter `usage.cost`/`cost_details` surface in agent runtime or session metadata.

## What This Changes

Request to capture OpenRouter response `usage.cost` and `cost_details` and expose per-turn or cumulative cost data to agents/session metadata, especially for Telegram reply footers.

## Best Possible Solution

Define a provider-reported cost metadata contract with explicit source and units, capture it at the OpenAI-compatible/OpenRouter boundary, persist it through transcript/session/runtime metadata, and expose it through `session_status` or a usage tool for channels to render.

## Reproduction Assessment

Yes, as a source-level gap: current main exposes token counters and local estimated cost paths, while raw OpenRouter `usage.cost` and `cost_details` are not carried into agent metadata or session status.

## Solution Assessment

Unclear until maintainers choose the provider-neutral metadata schema; the maintainable direction is a canonical runtime/session cost contract first, with Telegram rendering as a consumer.
