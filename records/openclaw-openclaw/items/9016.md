## Summary

Keep open. Current main has estimated-cost and transcript aggregation paths, but it still does not preserve OpenRouter provider-reported `usage.cost`, `cost_details.upstream_inference_cost`, or `prompt_tokens_details.cache_write_tokens` for agent/runtime metadata; the related estimated-cost work does not close this exact gap.

## What This Changes

Expose OpenRouter provider-reported per-message cost and cost-detail usage data to agent runtime, session metadata, and channel reply surfaces.

## Best Possible Solution

Add an additive provider-reported-cost path that captures OpenRouter raw usage at the OpenAI-compatible response/stream boundary, carries source and units through transcript/session usage and `EmbeddedPiAgentMeta`, and lets channels render exact OpenRouter values when available while keeping local estimates as fallback.

## Reproduction Assessment

Yes, at source level. Feed the shared OpenAI-compatible stream parser a final OpenRouter-style usage chunk with numeric `cost`, `cost_details.upstream_inference_cost`, and `prompt_tokens_details.cache_write_tokens`; current main normalizes tokens and local estimated cost but drops those provider fields.

## Solution Assessment

Yes. The narrow maintainable direction is additive capture and propagation of provider-reported OpenRouter usage metadata, separate from existing `estimatedCostUsd` and local transcript cost estimates.

Implement issue #9016 as a focused additive feature. Capture OpenRouter provider-reported usage from OpenAI-compatible final response/final stream usage chunks, including `usage.cost`, `usage.cost_details.upstream_inference_cost`, and `prompt_tokens_details.cache_write_tokens`.
