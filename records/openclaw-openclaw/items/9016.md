## Summary

Keep open. Current main has adjacent estimated-cost reporting, transcript aggregation, `/usage cost`, and session `estimatedCostUsd`, but it still drops OpenRouter's provider-reported `usage.cost`, `usage.cost_details.upstream_inference_cost`, and `prompt_tokens_details.cache_write_tokens` before agent/runtime metadata can expose them.

## What This Changes

Expose OpenRouter per-response usage cost and related usage details to agent runtime/session metadata so replies and tools can report exact provider costs when available.

## Best Possible Solution

Add an additive provider-reported-cost path: capture OpenRouter raw usage at the OpenAI-compatible response/stream boundary, preserve local estimated USD separately from provider-reported credits/cost details with explicit source and units, carry per-turn and cumulative values through transcript/session usage and `EmbeddedPiAgentMeta`, then expose them through `session_status` or a dedicated usage/cost tool for channels such as Telegram to render.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to feed the OpenAI-compatible stream parser a final OpenRouter-style chunk containing `usage.cost`, `usage.cost_details.upstream_inference_cost`, and `prompt_tokens_details.cache_write_tokens`; current main normalizes only token counts, drops those fields, and leaves runtime/session metadata token-only.

## Solution Assessment

Yes. The narrow maintainable direction is to capture provider-reported OpenRouter usage at the shared transport boundary and thread it as additive metadata; replacing the current estimate fields or relying only on post-hoc dashboard/generation lookup would not satisfy this runtime exposure request.
