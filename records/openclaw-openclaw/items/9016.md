## Summary

Keep open: current main and v2026.6.10 still expose normalized token usage, local estimated USD, and OpenRouter billed-total reconciliation, but not OpenRouter provider-reported `usage.cost`, `cost_details.upstream_inference_cost`, source, or units through agent/session runtime metadata. Related usage, resolved-model, auth-attribution, UI, and plugin-pricing items overlap but do not replace this focused provider-cost metadata gap.

## What This Changes

The issue requests capturing OpenRouter provider-reported per-response cost data and exposing per-turn/cumulative cost information to agent/session surfaces for optional channel display.

## Best Possible Solution

Define a provider-reported cost metadata contract with explicit source and units, capture it at the OpenRouter/OpenAI-compatible boundary, persist it through transcript/session/runtime metadata, and expose it through `session_status` or a dedicated usage tool for channels to render.

## Reproduction Assessment

Yes, at source level: current main exposes token counters, local estimated-cost display, and OpenRouter billed-total reconciliation, but raw OpenRouter `usage.cost`, `cost_details`, source, and units are not carried into agent metadata or `session_status` details.

## Solution Assessment

Unclear until maintainers choose the public metadata contract. The maintainable solution is a provider/session cost-source schema first, with Telegram or plugin rendering as downstream consumers.
