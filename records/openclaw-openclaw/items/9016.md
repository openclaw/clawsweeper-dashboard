## Summary

Keep open: current main still normalizes OpenAI-compatible token usage and computes local estimated cost, but it does not expose OpenRouter provider-reported `usage.cost` or `cost_details` through agent runtime metadata, session rows, or response footers. Recent maintainer and ClawSweeper comments also kept this issue open because the closed estimated-cost work does not cover exact OpenRouter provider cost metadata.

## What This Changes

Expose OpenRouter per-response provider cost data to agents and session/status surfaces so replies can include exact cost information when OpenRouter returns it.

## Best Possible Solution

Add an additive provider-reported cost metadata contract with explicit source and units, capture OpenRouter raw usage at the OpenAI-compatible response boundary, and thread it through transcripts, session metadata, `EmbeddedPiAgentMeta`, `session_status` or a dedicated cost tool, and channel footers.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection on current main gives high-confidence proof that OpenRouter raw cost fields are not exposed on the requested runtime, session, or footer surfaces.

## Solution Assessment

Unclear until maintainers choose the public metadata and cost-authority contract. The narrow maintainable direction is a distinct provider-reported cost object with explicit source, units, and fallback semantics rather than overloading existing `estimatedCostUsd` fields.
