## Summary

Keep open: current main and v2026.6.6 now partially address OpenRouter billed-cost capture by reconciling `message.usage.cost.total`, but they still do not expose provider-reported cost, cost source/units, or `cost_details` through agent runtime metadata, `session_status`, or the reply footer requested here.

## What This Changes

The issue requests provider-reported OpenRouter per-response cost data be captured and exposed to agent/session runtime metadata so replies, including Telegram replies, can show exact usage cost.

## Best Possible Solution

Define one provider-reported cost metadata record with source, units, availability, and provider detail fields, propagate it from OpenRouter response/stream completion through run metadata and session/status APIs, and let Telegram or other channels render it only after the generic contract exists.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.6 reconcile OpenRouter billed cost into assistant message usage, but the agent metadata, session_status result, and reply usage footer still do not expose provider-reported cost to agents. A live billed OpenRouter request was not needed to prove the missing runtime/session path, though it should be part of implementation proof.

## Solution Assessment

No, the current partial implementation is not the complete best solution for this issue. A transcript-only billed total is useful, but the maintainable solution is a generic provider-reported cost contract that keeps exact charges separate from local estimates and is explicitly exposed through agent/session surfaces.
