## Summary

Keep open: the PR targets a real current-main regression, but the proposed self-scoped `list` path can miss the current job because it filters only the first `cron.list` page; the maintainer-labeled authorization-boundary change also still needs CI and maintainer handling.

## What This Changes

The PR lets self-remove-scoped cron tool runs call `status` and a current-job-filtered `list`, adds cron tool regression tests, and updates cron docs and the changelog.

## Best Possible Solution

Land this PR or an equivalent patch after the self-scoped list lookup reliably finds the current job across pagination, CI is green, and maintainers accept the read-only permission boundary.

## Reproduction Assessment

Yes. Current main source and tests show self-remove-scoped `list` is rejected before gateway dispatch, and the linked regression report shows the same error for `status` and `list` in v2026.5.4 logs.

## Solution Assessment

No, not yet. The permission shape is narrow, but the self-scoped `list` implementation should page until the current job is found or use an id lookup before returning a current-job-only response.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Fetch enough pages before filtering self lists:** `src/agents/tools/cron-tool.ts:711-714`
  - body: In self-remove scope this calls `cron.list` once and then filters that page to the current job. The gateway caps a list page at 200 entries, so if this agent has more jobs and the current job sorts after the first page, the restricted run gets an empty list and still cannot inspect its own job metadata. Page until the job is found, or use/add an id lookup, before truncating the response.
