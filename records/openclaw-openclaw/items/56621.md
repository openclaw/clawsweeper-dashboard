## Summary

Keep open. Current main still lacks the requested shell CLI for approved pairing entries: the pairing namespace only exposes pending-request listing and approval, docs document only those commands, and channel status still formats `allow:` from the account snapshot rather than the pairing allowFrom store. The adjacent chat `/allowlist` command and pairing-store helpers are useful implementation building blocks, but they do not satisfy this issue’s requested `openclaw pairing list-approved` and `openclaw pairing revoke` surface.

## What This Changes

The issue requests CLI commands to list approved paired senders and revoke pairing-store approvals, plus merged allowlist visibility in channel status.

## Best Possible Solution

Add a conservative core CLI surface: `openclaw pairing list-approved <channel> [--account] [--json]` should merge config `allowFrom` entries with pairing-store approvals and label their source, while `openclaw pairing revoke <channel> <sender> [--account]` should remove pairing-store approvals only and clearly report config-owned entries. Update `channels status --probe` to show an effective merged allowlist or paired-store summary, and cover default-account, non-default-account, JSON, docs, and status formatting cases.

## Reproduction Assessment

Yes. A source/docs check against current main is enough: `registerPairingCli` only registers pending `list` and `approve`, `rg` finds no `list-approved` or pairing `revoke`, and the docs only describe those two commands.

## Solution Assessment

Yes, the requested CLI direction is the narrow maintainable solution, as long as revoke semantics stay limited to the pairing store and config-owned entries are reported separately. The existing store helpers and chat `/allowlist` command provide implementation guidance without requiring a new storage model.
