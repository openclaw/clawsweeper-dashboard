## Summary

Keep open: current main and v2026.6.10 still expose only pending-request listing and approval in the terminal pairing CLI, while approved-pairing listing/revocation remains only partially covered by lower-level store helpers and the in-chat `/allowlist` surface. The request is useful but security-sensitive, so it needs maintainer product direction before implementation.

## What This Changes

The issue requests terminal commands to list approved channel pairings and revoke paired senders, plus status output that reflects merged config and pairing-store allowlists.

## Best Possible Solution

Define one account-scoped admin pairing/contact management surface that coordinates terminal CLI behavior with `/allowlist`, channel status, and the broader contact-inventory design, with explicit redaction and authorization semantics.

## Reproduction Assessment

Yes, at source level. Current `openclaw pairing list` lists pending requests and `pairing approve` writes approved senders, while no terminal command lists or revokes approved pairings.

## Solution Assessment

Unclear until maintainers choose the product and security boundary. The proposed commands are plausible, but the best shape should avoid a second partial permission-management surface beside `/allowlist` and https://github.com/openclaw/openclaw/issues/48620.
