## Summary

Keep open: the bug is source-proven and the PR targets the right Signal reply path, but the patch leaves Signal media replies without the hook metadata that shared outbound and Telegram paths provide, and the supplied proof is only unit-test output rather than real after-fix Signal behavior.

## What This Changes

The PR adds a Signal monitor `message_sending` hook call before `deliverTextOrMediaReply` sends inbound replies, honoring cancellation and text rewrites.

## Best Possible Solution

Have Signal reply delivery match the shared outbound/Telegram hook event shape, add focused text and media hook coverage, then validate with a real Signal account and a registered hook that observes, rewrites, or cancels a DM reply.

## Reproduction Assessment

Yes. Source inspection on current main shows inbound Signal replies flow through `createSignalEventHandler` into `deliverReplies`, and `deliverReplies` sends through `sendMessageSignal` without `getGlobalHookRunner` or `runMessageSending`.

## Solution Assessment

Partially. Adding the hook in `extensions/signal/src/monitor.ts` is the right narrow fix, but this patch should also preserve existing hook metadata for media replies and provide real Signal proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Pass media URLs through the Signal sending hook:** `extensions/signal/src/monitor.ts:375-377`
