## Best Possible Solution

Run the hook for any sendable Signal reply, preserve empty-reply suppression, add focused Signal monitor regression tests, and require redacted Signal-path proof before merge.

Yes. Current main routes inbound Signal replies through `deliverReplies`, which sends through `sendMessageSignal` without `runMessageSending`; the PR’s remaining media-only gap is also source-reproducible from the `rawContent` guard.

## Solution Assessment

No, not as written. The hook placement is the right owner-local fix, but the guard should key off sendable content rather than non-empty text, with Signal-path proof and regression tests before landing.
