## Summary

Keep open: the PR is focused and in the right Signal delivery path, but the current guard only runs `message_sending` when reply text is truthy, so media-only Signal replies with no caption still bypass hook rewrite/cancel handling.

## What This Changes

The PR updates `extensions/signal/src/monitor.ts` to call the existing `message_sending` hook before Signal reply transport sends, applying cancel/content rewrite behavior and media metadata for non-empty text replies.

## Best Possible Solution

Run `message_sending` for every sendable Signal reply, including media-only replies with empty string content, skip only true no-content payloads, and add focused regression/proof coverage before landing and closing https://github.com/openclaw/openclaw/issues/81610.

## Reproduction Assessment

Yes, by source inspection. Current main and `v2026.5.20` route Signal inbound replies through `deliverReplies`, which sends via `sendMessageSignal` without invoking `message_sending`.

## Solution Assessment

Not yet. The owner-local hook placement matches the Telegram/shared outbound pattern, but the `rawContent` truthy guard is too narrow because media-only replies are sendable with empty text.
