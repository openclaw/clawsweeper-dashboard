## Summary

Keep open: this is a plausible narrow fix for a real Signal hook gap, but the patch still omits media URL metadata from the hook event and only supplies unit-test output instead of real Signal behavior proof.

## What This Changes

The PR imports the plugin runtime hook runner in `extensions/signal/src/monitor.ts` and calls `runMessageSending` during Signal reply delivery, honoring cancel and text rewrite results.

## Best Possible Solution

Have Signal reply delivery match the shared outbound and Telegram hook event shape, add focused text rewrite/cancel/media metadata coverage, and validate with a real Signal account plus a registered hook.

## Reproduction Assessment

Yes. Source inspection on current main shows inbound Signal replies flow through `createSignalEventHandler` into `deliverReplies`, and that function sends through `sendMessageSignal` without `getGlobalHookRunner` or `runMessageSending`.

## Solution Assessment

Partially. Adding the hook in `extensions/signal/src/monitor.ts` is the right narrow fix, but the patch should pass the resolved media URLs in hook metadata and supply real Signal proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

- **[P2] Pass media URLs through the Signal sending hook:** `extensions/signal/src/monitor.ts:375-377`
