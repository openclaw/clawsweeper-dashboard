## Summary

Keep open: the PR targets a real, source-proven Signal hook bypass, but the patch is not merge-ready because it runs hooks for empty replies and the supplied proof still does not exercise the patched Signal monitor path.

## What This Changes

The PR updates `extensions/signal/src/monitor.ts` so Signal `deliverReplies` calls the existing `message_sending` hook with Signal metadata and honors cancel/content rewrites before sending replies.

## Best Possible Solution

Keep the focused Signal monitor hook wiring, preserve empty-reply suppression, add targeted regression coverage, and require redacted Signal-path proof before landing and closing https://github.com/openclaw/openclaw/issues/81610.

## Reproduction Assessment

Yes, by source inspection: current main routes inbound Signal replies through `deliverReplies`, which sends via `sendMessageSignal` without the documented `message_sending` hook. The PR-specific empty-reply regression is also source-reproducible because reply text defaults to `""` before the new hook call.

## Solution Assessment

No, not as written. The owner-local hook placement is the right direction, but it should gate hooks on sendable content or otherwise preserve empty reply suppression, then be backed by Signal-path proof and focused tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Preserve empty-reply suppression before hooks:** `extensions/signal/src/monitor.ts:370`
  - body: `resolveSendableOutboundReplyParts` always returns `text` as a string, so this condition also runs for `{}` or blank-text replies when hooks are registered.   - confidence: 0.87
