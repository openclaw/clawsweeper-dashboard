## Summary

Keep open. Current main deliberately keeps trusted-proxy token auth rejected, local fallback password-only, and loopback proxy trust behind explicit opt-in; this PR broadens the gateway auth trust boundary and still needs maintainer/security direction.

## What This Changes

The PR changes gateway trusted-proxy auth to add loopback shared-secret/token fallback behavior, relax same-host loopback proxy handling, and expand gateway auth tests.

## Best Possible Solution

Keep the explicit current model unless maintainers revise it: password-only local-direct fallback, `trustedProxy.allowLoopback` for same-host reverse proxies, and any token/service identity work through the RFC with coordinated config, docs, and tests.

## Reproduction Assessment

Yes. The PR diff and current main tests provide source-level reproductions for token fallback, loopback proxy headers, local-direct detection, and config-guard ordering; I did not run tests because this was a required read-only review.

## Solution Assessment

No. The PR is broader than the maintained contract, and the safer path is to preserve explicit loopback opt-in and password-only fallback unless maintainers approve a service-identity/token design.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Require explicit opt-in for loopback proxy headers:** `src/gateway/auth.ts:301-302`
  - body: This exception accepts trusted-proxy identity from a loopback source based on Host and forwarding-header shape instead of `trustedProxy.allowLoopback`. Local processes can forge those inputs, so same-host loopback trusted-proxy auth becomes implicit rather than explicitly configured.
  - confidence: 0.9
- **[P2] Preserve raw loopback local-direct detection:** `src/gateway/auth.ts:151-153`
  - body: `isLocalDirectRequest` now depends on proxy-aware client IP resolution. When loopback is listed in `gateway.trustedProxies` and no forwarded client header is present, that resolver returns no client IP, so direct loopback requests may stop being classified as local-direct for other gateway paths.
  - confidence: 0.86
- **[P2] Align token fallback with startup validation:** `src/gateway/auth.ts:390-396`
  - body: The fallback accepts `auth.token` in trusted-proxy mode, but current startup validation rejects trusted-proxy configs that also set a shared token. The advertised token fallback is therefore not usable through normal config unless validation and docs change with it.
  - confidence: 0.88
- **[P2] Keep trusted-proxy config guards before fallback:** `src/gateway/auth.ts:528-537`
  - body: The local shared-secret fallback runs before the missing `trustedProxy` and empty `trustedProxies` checks. That lets a malformed trusted-proxy runtime auth object authenticate on loopback by shared secret instead of failing closed like current main.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff changes gateway authentication trust boundaries and introduces concrete loopback trusted-proxy and token-mode concerns.

Concerns:

- **[high] Implicit loopback trusted-proxy trust:** `src/gateway/auth.ts:301`
  - body: The PR would allow trusted-proxy identity headers from `127.0.0.1` based on Host and forwarding-header shape instead of requiring `trustedProxy.allowLoopback`; local processes can forge those inputs, weakening the explicit same-host proxy trust boundary.
  - confidence: 0.9
- **[medium] Trusted-proxy token fallback broadens auth mode:** `src/gateway/auth.ts:390`
  - body: The PR adds token fallback inside trusted-proxy mode while current main treats token and trusted-proxy as mutually exclusive. That changes credential eligibility for a security-sensitive auth mode without the matching startup/docs contract.
  - confidence: 0.84
- **[medium] Malformed trusted-proxy configs can authenticate by fallback:** `src/gateway/auth.ts:528`
  - body: Because fallback is evaluated before trusted-proxy config and proxy-list guards, direct callers can authenticate with a shared secret even when the trusted-proxy mode lacks the proxy contract that current main requires before any fallback.
  - confidence: 0.78
