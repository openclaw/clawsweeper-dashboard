## Summary

Keep open for human auth/security review: the branch still contains unique gateway auth changes, but the proposed patch has blocking trust-boundary issues and lacks contributor-provided real behavior proof.

## What This Changes

The PR changes gateway auth and tests to add trusted-proxy loopback shared-secret/token fallback plus same-host loopback proxy-header handling.

## Best Possible Solution

Keep the stricter trusted-proxy contract on main and handle internal/same-host callers through explicit allowLoopback, password fallback where supported, or a reviewed service-identity design.

## Reproduction Assessment

Yes, source-level. The PR head shows token fallback and implicit loopback proxy acceptance, while current main source, tests, and docs show those paths are intentionally fail-closed or require explicit opt-in.

## Solution Assessment

No. The PR broadens a security-sensitive auth boundary; the safer direction is explicit loopback opt-in and the RFC-backed local service identity path rather than another trusted-proxy/shared-secret fallback.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Require explicit opt-in for loopback proxy headers:** `src/gateway/auth.ts:301-302`
  - body: This treats a loopback trusted-proxy request as proxied whenever Host is non-localish and any forwarding header exists. Local processes can forge those inputs, so same-host proxy trust becomes implicit instead of gated by `trustedProxy.allowLoopback`.
  - confidence: 0.9
- **[P2] Preserve raw loopback local-direct detection:** `src/gateway/auth.ts:151-153`
  - body: `isLocalDirectRequest` now depends on `resolveRequestClientIp`; when the socket address is a configured trusted proxy and no forwarded chain is present, that resolver returns undefined, so direct loopback calls stop being localDirect for other auth/origin/pairing gates.
  - confidence: 0.86
- **[P2] Align trusted-proxy token fallback with startup validation:** `src/gateway/auth.ts:390-396`
  - body: The new fallback accepts `auth.token` in trusted-proxy mode, but startup rejects trusted-proxy configs that also set a token. This makes the advertised token fallback unusable through normal config while widening runtime behavior for constructed auth objects.
  - confidence: 0.88
- **[P2] Keep trusted-proxy config guards before fallback:** `src/gateway/auth.ts:528-537`
  - body: The fallback runs before missing `trustedProxy` and empty `trustedProxies` checks. A malformed trusted-proxy runtime object can therefore authenticate on loopback by shared secret where current main fails closed.
  - confidence: 0.78
