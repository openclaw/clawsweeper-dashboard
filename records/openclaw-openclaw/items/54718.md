## Summary

Keep this PR open for maintainer/security handling, but the patch should not merge as-is. Current main already shipped a narrower trusted-proxy model, while this branch reopens token fallback and implicit loopback proxy trust paths that conflict with the current startup/docs contract and the active gateway-auth RFC.

## What This Changes

The PR adds trusted-proxy local loopback shared-secret fallback behavior, including token/password fallback, loopback proxy-header handling changes, and expanded gateway auth tests.

## Best Possible Solution

Keep the explicit current model unless maintainers deliberately revise it: password-only local-direct fallback for trusted-proxy internal callers, explicit `trustedProxy.allowLoopback` for same-host reverse proxies, and token/service identity work handled through the RFC with coordinated config, docs, and tests.

## Reproduction Assessment

Yes for the review findings: the PR diff and current `authorizeGatewayConnect` unit cases provide a high-confidence source-level reproduction path for token fallback, loopback proxy headers, and local-direct detection. I did not run tests because this was a required read-only review.

## Solution Assessment

No. The PR is broader than the current maintained contract; the safer path is to keep explicit loopback opt-in and password-only fallback unless maintainers approve a new service-identity/token design through #69066.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Require explicit opt-in for loopback proxy headers:** `src/gateway/auth.ts:301-302`
  - body: This exception accepts trusted-proxy identity from a loopback source based on Host and forwarding-header shape instead of `trustedProxy.allowLoopback`. A local process can forge those inputs, so same-host loopback trusted-proxy auth becomes implicit rather than explicitly configured.
  - confidence: 0.9
- **[P2] Preserve raw loopback local-direct detection:** `src/gateway/auth.ts:151-153`
  - body: `isLocalDirectRequest` is changed to depend on proxy-aware client IP resolution. When loopback is listed in `gateway.trustedProxies` and no forwarded client header is present, that resolver can return no client IP, so direct loopback requests may stop being classified as local-direct.
  - confidence: 0.86
- **[P2] Align token fallback with startup validation:** `src/gateway/auth.ts:390-396`
  - body: The fallback accepts `auth.token` in trusted-proxy mode, but current startup validation rejects trusted-proxy configs that also set a shared token. The advertised token fallback is therefore not usable through normal config unless validation and docs change with it.
  - confidence: 0.88
- **[P2] Keep trusted-proxy config guards before fallback:** `src/gateway/auth.ts:519-526`
  - body: The local shared-secret fallback runs before the missing `trustedProxy` and empty `trustedProxies` checks. That lets a malformed trusted-proxy runtime auth object authenticate on loopback by shared secret instead of failing closed like current main.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff changes gateway authentication trust boundaries and introduces concrete loopback trusted-proxy and token-mode concerns.

Concerns:

- **[high] Implicit loopback trusted-proxy trust:** `src/gateway/auth.ts:301`
  - body: The PR would allow trusted-proxy identity headers from `127.0.0.1` based on Host and forwarding-header shape instead of requiring `trustedProxy.allowLoopback`; local processes can forge those inputs, weakening the explicit same-host proxy trust boundary.
  - confidence: 0.9
- **[medium] Trusted-proxy token fallback broadens auth mode:** `src/gateway/auth.ts:390`
  - body: The PR adds token fallback inside trusted-proxy mode while current main treats token and trusted-proxy as mutually exclusive. That changes credential eligibility for a security-sensitive auth mode without the matching startup/docs contract.
  - confidence: 0.84
- **[medium] Malformed trusted-proxy configs can authenticate by fallback:** `src/gateway/auth.ts:519`
  - body: Because fallback is evaluated before trusted-proxy config/proxy-list guards, direct callers can authenticate with a shared secret even when the trusted-proxy mode lacks the proxy contract that current main requires before any fallback.
  - confidence: 0.78
