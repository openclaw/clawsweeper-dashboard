## Summary

Keep open for maintainer review. The branch touches a real gateway auth pain point, but the current diff revives trusted-proxy token fallback that current source/docs reject, lets loopback proxy auth bypass the explicit `allowLoopback` opt-in, and lacks contributor real-behavior proof for the deployment path.

## What This Changes

The PR changes gateway trusted-proxy authentication to add local shared-secret fallback for loopback requests and to allow same-host loopback proxy requests through trusted-proxy header auth.

## Best Possible Solution

Keep token fallback rejected, keep same-host proxy trust behind the explicit `allowLoopback` contract, and have maintainers decide whether any remaining internal-caller gap belongs in the service-identity path rather than this branch.

## Reproduction Assessment

Yes, source inspection is enough for the review findings: current main rejects trusted-proxy plus token configs and requires `allowLoopback` for loopback proxy trust, while the PR patch changes both behaviors. I did not run tests because this review is read-only.

## Solution Assessment

No. The branch is not the best current fix because it bypasses documented auth guardrails; the safer path is an explicit maintainer-approved local password/service-identity design plus docs and tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep loopback proxy auth behind allowLoopback:** `src/gateway/auth.ts:299-300`
  - body: This changes loopback trusted-proxy handling so a request from `127.0.0.1` with a non-local Host and any forwarding context can reach proxy-header auth even when `trustedProxy.allowLoopback` is not enabled. Current main and the docs make `allowLoopback` the explicit opt-in because any local process can try to forge proxy identity headers; please preserve that opt-in instead of inferring trust from Host/header shape.
  - confidence: 0.89
- **[P2] Do not revive trusted-proxy token fallback:** `src/gateway/auth.ts:388-396`
  - body: The new fallback accepts `auth.token` in `trusted-proxy` mode, but current startup validation rejects that mixed config and the docs say token fallback is intentionally unsupported. These tests exercise a state the real gateway should not start with, and calling this helper directly would bypass the policy that PR 58371 added.
  - confidence: 0.88
- **[P2] Preserve trusted-proxy config validation before fallback:** `src/gateway/auth.ts:520-529`
  - body: The fallback runs before checking that `auth.trustedProxy` exists and that `trustedProxies` is non-empty, and the tests now expect token auth to succeed when those trusted-proxy prerequisites are missing. That turns an invalid trusted-proxy setup into shared-secret auth instead of failing closed, contrary to the current runtime contract.
  - confidence: 0.82
