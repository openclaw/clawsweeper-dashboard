## Summary

Keep open. The PR still contains security-sensitive gateway auth changes that conflict with current main's explicit trusted-proxy contract, and the remaining direction is an auth/product decision rather than conservative cleanup.

## What This Changes

This PR changes `src/gateway/auth.ts` and gateway auth tests to add trusted-proxy loopback shared-secret/token fallback and same-host loopback proxy handling.

## Best Possible Solution

Preserve the explicit fail-closed trusted-proxy model for now, use `trustedProxy.allowLoopback` only as an intentional same-host proxy opt-in, and settle local service/token identity through the RFC-backed design.

## Reproduction Assessment

Yes, source-level. The PR head and current-main source/tests show the token fallback, loopback proxy-header behavior, local-direct detection change, and config-guard ordering without needing a runtime setup; I did not run tests because this review is read-only.

## Solution Assessment

No. The PR is broader than the maintained auth contract; the safer path is explicit loopback opt-in plus a reviewed service/local identity design instead of implicit trusted-proxy token fallback.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Require explicit opt-in for loopback proxy headers:** `src/gateway/auth.ts:301-302`
  - body: This accepts trusted-proxy identity from a loopback source based on Host and forwarding-header shape instead of `trustedProxy.allowLoopback`. Local processes can forge those inputs, so same-host loopback trusted-proxy auth becomes implicit rather than explicitly configured.
  - confidence: 0.9
- **[P2] Preserve raw loopback local-direct detection:** `src/gateway/auth.ts:151-153`
  - body: `isLocalDirectRequest` now depends on proxy-aware client IP resolution. When loopback is listed in `gateway.trustedProxies` and no forwarded client header is present, that resolver returns no client IP, so direct loopback requests can stop being treated as local-direct by other gateway paths.
  - confidence: 0.86
- **[P2] Align token fallback with startup validation:** `src/gateway/auth.ts:390-396`
  - body: The fallback accepts `auth.token` in trusted-proxy mode, but current startup validation rejects trusted-proxy configs that also set a shared token. The advertised token fallback is therefore not usable through normal config unless the validation and docs contract changes too.
  - confidence: 0.88
- **[P2] Keep trusted-proxy config guards before fallback:** `src/gateway/auth.ts:528-537`
  - body: The local shared-secret fallback runs before the missing `trustedProxy` and empty `trustedProxies` checks. That lets a malformed trusted-proxy runtime auth object authenticate on loopback by shared secret instead of failing closed like current main.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff changes gateway authentication trust boundaries and needs explicit auth/security design review before merge.

Concerns:

- **[high] Implicit loopback trusted-proxy trust:** `src/gateway/auth.ts:301`
  - body: The PR would allow trusted-proxy identity headers from `127.0.0.1` based on Host and forwarding-header shape instead of requiring `trustedProxy.allowLoopback`; local processes can forge those inputs, weakening the explicit same-host proxy trust boundary.
  - confidence: 0.9
- **[medium] Trusted-proxy token fallback broadens auth mode:** `src/gateway/auth.ts:390`
  - body: The PR adds token fallback inside trusted-proxy mode while current main treats token and trusted-proxy as mutually exclusive. That changes credential eligibility for a security-sensitive auth mode without a matching startup/docs contract.
  - confidence: 0.84
- **[medium] Malformed trusted-proxy configs can authenticate by fallback:** `src/gateway/auth.ts:528`
  - body: Because fallback is evaluated before trusted-proxy config and proxy-list guards, direct loopback callers can authenticate with a shared secret even when the trusted-proxy mode lacks the proxy contract that current main requires before any fallback.
  - confidence: 0.78
