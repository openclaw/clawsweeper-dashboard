## Summary

Close: current main and release v2026.4.27 already shipped the maintained trusted-proxy fallback shape, with password-only local-direct fallback and explicit same-host loopback proxy opt-in. The PR’s remaining token fallback and implicit loopback trust changes conflict with the current startup validation, docs, and regression tests.

## What This Changes

The PR changes gateway trusted-proxy authentication to add local shared-secret/token fallback handling and same-host loopback proxy behavior, with new gateway auth tests.

## Best Possible Solution

Keep the shipped model: password-only local-direct fallback, explicit `trustedProxy.allowLoopback` for same-host reverse proxies, and any future token or service-identity work in the gateway auth RFC rather than this branch.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main tests reject trusted-proxy token fallback and accept password fallback, while the PR head changes those expectations and bypasses current startup/config guards.

## Solution Assessment

No. The PR diff is not the best solution now that current main shipped the narrower explicit loopback/password fallback contract; merging this branch would re-open token and loopback trust-boundary behavior that current docs and tests intentionally avoid.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Require explicit opt-in for loopback proxy headers:** `src/gateway/auth.ts:301-302`
  - body: This accepts trusted-proxy identity from a loopback source based on Host and forwarding-header shape instead of `trustedProxy.allowLoopback`. Local processes can forge those inputs, so same-host loopback trusted-proxy auth becomes implicit rather than explicitly configured.
  - confidence: 0.9
- **[P2] Preserve raw local-direct detection:** `src/gateway/auth.ts:151-163`
  - body: `isLocalDirectRequest` now depends on proxy-aware client IP resolution. When loopback is listed in `gateway.trustedProxies` and no forwarded client header is present, that resolver returns no client IP, so direct loopback requests can stop being classified as local-direct for other gateway paths.
  - confidence: 0.86
- **[P2] Align token fallback with startup validation:** `src/gateway/auth.ts:390-396`
  - body: The fallback accepts `auth.token` in trusted-proxy mode, but current startup validation rejects trusted-proxy configs that also set a shared token. The advertised token fallback is therefore not usable through normal config unless validation and docs change with it.
  - confidence: 0.88
- **[P2] Keep trusted-proxy guards before fallback:** `src/gateway/auth.ts:528-537`
  - body: The local shared-secret fallback runs before the missing `trustedProxy` and empty `trustedProxies` checks. That lets a malformed trusted-proxy runtime auth object authenticate on loopback by shared secret instead of failing closed like current main.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The PR changes gateway authentication trust boundaries and would weaken the shipped explicit loopback and token-mode separation.

Concerns:

- **[high] Implicit loopback trusted-proxy trust:** `src/gateway/auth.ts:301`
  - body: The PR would allow trusted-proxy identity headers from `127.0.0.1` based on Host and forwarding-header shape instead of requiring `trustedProxy.allowLoopback`; local processes can forge those inputs, weakening the explicit same-host proxy trust boundary.
  - confidence: 0.9
- **[medium] Trusted-proxy token fallback broadens auth mode:** `src/gateway/auth.ts:390`
  - body: The PR adds token fallback inside trusted-proxy mode while current main treats token and trusted-proxy as mutually exclusive. That changes credential eligibility for a security-sensitive auth mode without the matching startup/docs contract.
  - confidence: 0.84
- **[medium] Malformed trusted-proxy configs can authenticate by fallback:** `src/gateway/auth.ts:528`
  - body: Because fallback is evaluated before trusted-proxy config and proxy-list guards, direct callers can authenticate with a shared secret even when trusted-proxy mode lacks the proxy contract that current main requires before any fallback.
  - confidence: 0.78
