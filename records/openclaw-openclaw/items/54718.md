## Summary

This PR should close as superseded: the branch still adds trusted-proxy token fallback and implicit loopback proxy acceptance, while current main and the active RFC now steer this auth gap toward explicit loopback/service-identity design instead of rebasing this fallback path.

## What This Changes

The branch modifies `src/gateway/auth.ts` and `src/gateway/auth.test.ts` to add trusted-proxy local shared-secret fallback and same-host loopback proxy handling.

## Best Possible Solution

Keep the explicit trusted-proxy contract on main and route remaining local/internal caller auth through the canonical service-identity RFC.

## Reproduction Assessment

Yes. Source inspection is enough: the PR adds trusted-proxy token fallback and implicit loopback acceptance, while current main rejects mixed trusted-proxy token config and requires explicit `allowLoopback` for loopback trusted-proxy sources.

## Solution Assessment

No. This is not the best fix because it adds another trusted-proxy fallback branch; the safer path is the open service-identity design that keeps local service auth separate from user trusted-proxy auth.
