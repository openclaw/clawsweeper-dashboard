## Best Possible Solution

Separate the DM reply metadata target from thread/session routing, keep Mattermost DMs flat, add monitor-level regression coverage, and include redacted live Mattermost proof of the after-fix inbound payload.

## Reproduction Assessment

Yes, source-level. Current main intentionally clears `effectiveReplyToId` for direct Mattermost chats, and the PR passes that cleared value into a helper that exits when it is absent.

## Solution Assessment

No. The best fix is to separate reply metadata hydration from thread/session routing so direct-message metadata can be fetched without changing the documented flat-DM contract.
