## Summary

Keep this PR open, but it is not mergeable as-is: the new Mattermost reply-context helper is still wired to the delivery/threading reply id that direct messages intentionally suppress, so the DM path this PR targets still skips the lookup. The supplied proof is also terminal hygiene output rather than observable Mattermost DM behavior.

## What This Changes

The PR adds a Mattermost post-fetch helper, a bounded monitor cache, a reply-context helper, and tests intended to populate `ReplyToBody` and `ReplyToSender` for Mattermost DM replies.

## Best Possible Solution

Keep DM delivery/threading flat, hydrate quote metadata from a separate context-only target such as normalized `post.root_id`, add monitor-level DM regression coverage, and require redacted real Mattermost proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: Mattermost `D` channels become `direct`, direct chats force `effectiveReplyToId` to `undefined`, and the PR helper returns empty before fetching the actual DM reply target.

## Solution Assessment

No. The bounded fetch/helper direction is reasonable, but using `effectiveReplyToId` is not the right fix because that value intentionally suppresses DM threading; a separate context-only reply target is safer and narrower.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Use the actual DM reply target:** `extensions/mattermost/src/mattermost/monitor.ts:1540`
