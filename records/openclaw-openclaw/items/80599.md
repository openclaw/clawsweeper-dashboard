## Summary

Keep open: the PR targets a real Mattermost reply-context gap, but the patch currently hydrates from the same threading id that direct chats intentionally suppress, so the advertised DM path still does not populate the metadata. It also needs contributor-provided real Mattermost proof before merge.

## What This Changes

The PR adds a Mattermost post fetch helper, reply-context resolver, short-lived post cache, and tests to populate `ReplyToBody` and `ReplyToSender` for inbound Mattermost replies.

## Best Possible Solution

Keep DM sessions and outbound replies flat, but derive a separate reply-reference id from the inbound DM reply target for metadata hydration, add monitor-level regression coverage, bound the cache, and require redacted live Mattermost proof.

## Reproduction Assessment

Yes, by source inspection: current main suppresses `effectiveReplyToId` for direct chats, and this PR uses that suppressed value to trigger hydration. A focused monitor-path test with a direct-message post carrying `root_id` would reproduce the missing `ReplyToBody`/`ReplyToSender` without needing a product-mode change.

## Solution Assessment

No. The maintainable fix is to keep the existing non-threaded DM contract while passing a separate reference-only reply target into the hydration path; using `effectiveReplyToId` conflates metadata with threading and misses DMs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Use the DM reply target instead of the threading id:** `extensions/mattermost/src/mattermost/monitor.ts:1523`
  - body: The PR hydrates reply context from `effectiveReplyToId`, but direct chats return `undefined` before considering `threadRootId`. In the DM reply flow this makes `resolveMattermostReplyContext` exit immediately, leaving `ReplyToBody` and `ReplyToSender` unset. Keep outbound DM threading disabled, but pass a separate reply-reference id from the inbound reply target and cover the monitor path.
  - confidence: 0.93
- **[P2] Bound the post lookup cache:** `extensions/mattermost/src/mattermost/monitor.ts:800`
  - body: `postCache` is keyed by Mattermost post id and only checks TTL when a key is read; expired entries are never pruned and the map has no max size. A long-running gateway receiving replies to many unique posts can retain unbounded cache entries, so use a bounded cache or prune expired entries on insert/read.
  - confidence: 0.87

## Security Review

Status: needs_attention

Summary: The diff adds no dependency or secret-handling change, but the new high-cardinality post cache is unbounded and needs an eviction cap before merge.
