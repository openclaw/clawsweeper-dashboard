The PR adds Mattermost post lookup/caching plus reply-context hydration and tests intended to populate `ReplyToBody` and `ReplyToSender` for direct-message replies.

## Best Possible Solution

Keep Mattermost DM sessions and delivery non-threaded, but derive a separate reply-metadata target from the inbound post/root context, hydrate it through the current monitor resource/cache owner, add a monitor-level DM reply regression, and require redacted live Mattermost DM proof.

## Reproduction Assessment

Yes, source-reproducible: current main deliberately returns `undefined` for direct-message `effectiveReplyToId`, and the PR passes that value into the new hydration helper. A monitor-level DM reply fixture would show the new fields remain absent for the intended path.

## Solution Assessment

No. The maintainable fix is to separate the DM reply metadata target from the DM thread/session id, then cover that monitor path and prove it in a real Mattermost DM flow.
