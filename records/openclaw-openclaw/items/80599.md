## Best Possible Solution

Keep Mattermost DM sessions and delivery non-threaded, hydrate reply metadata from the inbound post's actual reply target/root id, add monitor-level DM coverage, and require redacted live Mattermost proof.

## Reproduction Assessment

Yes, source-reproducible: current main direct-message routing leaves `effectiveReplyToId` undefined and the PR uses that value for metadata hydration. I did not run a live Mattermost workspace in this read-only review.

## Solution Assessment

No. The maintainable fix is to separate the DM metadata reply target from the DM thread/session id, then cover that monitor path and prove it in a real DM flow.
