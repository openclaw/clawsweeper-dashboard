## Summary

Keep open: this is a plausible Mattermost bug-fix PR, but the current patch does not actually exercise the direct-message path it claims to fix because it hydrates reply context from `effectiveReplyToId`, which remains undefined for DMs. External real behavior proof is also still missing.

## What This Changes

The PR adds a Mattermost post lookup/cache and reply-context helper so inbound Mattermost messages can set `ReplyToBody` and `ReplyToSender`.

## Best Possible Solution

Derive reply-context metadata from the actual Mattermost reply target such as `post.root_id` while keeping DM delivery/threading top-level, then add a direct-message monitor regression that proves the finalized inbound context contains the quoted body and sender.

## Reproduction Assessment

Yes, from source inspection: a Mattermost direct-message reply with `post.root_id` still gets `effectiveReplyToId: undefined`, and the new helper returns before fetching. A focused test can reproduce this by feeding the monitor a direct post with a root id and asserting `ReplyToBody`/`ReplyToSender` are absent with the current patch.

## Solution Assessment

No. The helper/cache direction is reasonable, but the fix should use a separate reply-context target id for quote metadata instead of reusing the delivery/threading id that is intentionally suppressed for DMs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Use the DM root post as the reply context target:** `extensions/mattermost/src/mattermost/monitor.ts:1540`

Please derive a separate reply-context id from the Mattermost reply target while leaving DM delivery threading suppressed.
