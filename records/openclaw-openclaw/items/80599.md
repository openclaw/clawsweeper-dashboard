## Summary

This PR should stay open, but it is not merge-ready: the patch hydrates reply metadata from `effectiveReplyToId`, while current Mattermost direct-message code intentionally makes that value `undefined`, so the exact DM reply path described by the PR still will not fetch the replied-to post. The external real-behavior proof is also insufficient, and the live PR API reports the branch as dirty against base.

## What This Changes

The PR adds a Mattermost post fetch helper, reply-target context hydration, bounded post caching, and tests intended to populate `ReplyToBody` and `ReplyToSender` for Mattermost DM replies.

## Best Possible Solution

Keep DM sessions and delivery non-threaded, but hydrate reply metadata from the inbound post's actual reply target/root id, add a monitor-level DM regression test, rebase cleanly, and provide redacted live Mattermost proof before merge.

## Reproduction Assessment

Yes, source-reproducible: current direct-message code clears `effectiveReplyToId`, and the PR uses that cleared value as the reply-context lookup id. I did not run live Mattermost in this read-only review.

## Solution Assessment

No. The maintainable fix is to separate the DM metadata reply target from the DM thread/session id, then cover that monitor path and prove it in a real DM flow.
