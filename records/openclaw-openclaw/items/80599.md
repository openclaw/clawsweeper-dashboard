## Summary

Keep this PR open, but it is not merge-ready: the new hydration call is wired to `effectiveReplyToId`, which current Mattermost direct-message routing intentionally suppresses, and the supplied proof still does not show a real DM reply payload after the fix.

## What This Changes

The branch adds a Mattermost post fetch helper, a reply-context resolver, a bounded in-memory post cache in the Mattermost monitor, and tests for populating `ReplyToBody` and `ReplyToSender`.

## Best Possible Solution

Separate reply metadata hydration from thread/session routing, preserve flat Mattermost DMs by default, add monitor-level DM payload coverage, and require redacted live Mattermost proof before merge.

## Reproduction Assessment

Yes, source-level: current main suppresses `effectiveReplyToId` for direct Mattermost chats, and the PR passes that suppressed value into hydration. I did not run a live Mattermost workspace in this read-only review.

## Solution Assessment

No. The proposed helper/cache shape is plausible, but wiring it to `effectiveReplyToId` is not the best fix because that field owns thread/session routing and is intentionally absent for DMs; use a separate reply metadata target instead.
