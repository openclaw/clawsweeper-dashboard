## Summary

Keep open: the narrow Mattermost bugfix is still useful, but this branch is not merge-ready because it feeds the new reply-context helper from `effectiveReplyToId`, which current main intentionally leaves empty for direct messages, and the PR still lacks real Mattermost DM proof.

## What This Changes

The PR adds a Mattermost post fetch/cache, a reply-context helper, and tests to populate `ReplyToBody` and `ReplyToSender` for replied-to Mattermost posts.

## Best Possible Solution

Separate metadata hydration from thread/session routing: derive a DM reply-target id only for context metadata, preserve flat DM delivery and session behavior, avoid unintended non-DM lookups unless deliberately covered, and prove the after-fix inbound payload with real Mattermost evidence.

## Reproduction Assessment

Yes, source-reproducible: current main returns `undefined` for direct `effectiveReplyToId`, and the PR passes that value into the new helper, which exits before fetching a target. I did not run a live Mattermost workspace.

## Solution Assessment

No. The best fix is to pass a separate direct-message metadata target into the helper instead of reusing the routing/thread id that intentionally stays empty for DMs.
