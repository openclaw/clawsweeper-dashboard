## Summary

Keep open: the patch targets a real Mattermost DM reply-context gap, but the monitor wires the new hydrator to `effectiveReplyToId`, which is still intentionally `undefined` for direct chats, so the intended DM path never fetches the replied-to post. The supplied proof also shows branch hygiene rather than an observed Mattermost DM reply after the fix.

## What This Changes

The PR adds a Mattermost post-fetch helper, a reply-context hydrator/cache in the Mattermost monitor, and regression tests intended to populate `ReplyToBody` and `ReplyToSender` for replied-to posts.

## Best Possible Solution

Keep DM replies top-level for delivery, but hydrate context from the actual Mattermost reply target such as `post.root_id` through a context-only path, then cover that monitor-level DM flow and provide redacted real DM proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: Mattermost channel type `D` becomes `direct`, direct chats force `effectiveReplyToId` to `undefined`, and the new helper exits before lookup when that value is missing.

## Solution Assessment

No. The helper and bounded fetch direction are reasonable, but wiring it to `effectiveReplyToId` is not the narrowest correct fix because that value intentionally suppresses DM threading; use a separate context-only reply target while leaving delivery threading unchanged.
