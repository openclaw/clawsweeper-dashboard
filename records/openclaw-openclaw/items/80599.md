## Summary

Keep this PR open: the intended fix is valid and narrow, but the branch still wires reply-context hydration to the DM delivery/threading id that current Mattermost logic intentionally suppresses, so the target DM reply path still skips the new lookup.

## What This Changes

The PR adds a Mattermost post fetch helper, a reply-context resolver/cache, monitor wiring, and focused tests to populate `ReplyToBody` and `ReplyToSender` for replied-to Mattermost posts.

## Best Possible Solution

Hydrate reply context from a separate context-only Mattermost reply target such as `post.root_id`, while keeping DM delivery and session threading top-level.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: Mattermost channel type `D` becomes `direct`, direct chats suppress `effectiveReplyToId`, and the PR helper exits without fetching when that value is absent.

## Solution Assessment

No. The fetch helper and bounded cache are reasonable, but wiring hydration to `effectiveReplyToId` is not the narrowest correct fix because that value intentionally encodes delivery/threading policy rather than context lookup policy.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

- **[P1] Hydrate from the real DM reply target:** `extensions/mattermost/src/mattermost/monitor.ts:1540`

Pass a separate context-only reply target, such as the normalized Mattermost root id, while leaving delivery threading unchanged.
