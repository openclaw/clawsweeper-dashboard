## Summary

Keep open: the central DM reply-context fix is still wired to `effectiveReplyToId`, which current main intentionally clears for direct chats, so the target Mattermost DM reply path still cannot hydrate `ReplyToBody` or `ReplyToSender`. The supplied proof also shows branch hygiene, not observed after-fix Mattermost behavior.

## What This Changes

The PR adds a Mattermost post-fetch helper, reply-context hydration/cache in the monitor, and focused tests to populate `ReplyToBody` and `ReplyToSender` for DM replies.

## Best Possible Solution

Keep DM delivery threading unchanged, but hydrate reply metadata from a separate context-only target such as normalized `post.root_id`, then cover that monitor-level direct-chat path.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: Mattermost type `D` becomes `direct`, direct chats clear `effectiveReplyToId`, and the new helper exits before lookup when that value is absent.

## Solution Assessment

No. The bounded fetch helper is a reasonable direction, but wiring it to `effectiveReplyToId` conflates delivery threading with quote-context lookup; a separate context-only root id is the narrower maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Hydrate DM replies from the actual reply target:** `extensions/mattermost/src/mattermost/monitor.ts:1540`
  - body: For direct chats, current main makes `resolveMattermostEffectiveReplyToId()` return `undefined` before considering `post.root_id`. This PR passes that value into `resolveMattermostReplyContext()`, whose first branch returns `{}` when it is missing, so a Mattermost DM reply still never fetches the replied-to post and `ReplyToBody`/`ReplyToSender` stay absent. Pass a separate context-only target from the normalized root id while leaving delivery `ReplyToId` unchanged.
  - confidence: 0.94
