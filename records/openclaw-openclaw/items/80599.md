## Summary

Keep open, but this PR is not merge-ready: the patch hydrates reply metadata from `effectiveReplyToId`, while current Mattermost direct-message routing intentionally clears that value, so the stated DM reply-context path still exits without fetching the replied-to post and lacks real Mattermost proof.

## What This Changes

The PR adds a Mattermost post fetch/cache and reply-context helper to populate `ReplyToBody` and `ReplyToSender` for replied-to posts, with focused unit tests.

## Best Possible Solution

Derive a separate direct-message reply metadata target for hydration, keep Mattermost DM routing/session/delivery flat, avoid unintended non-DM lookups unless deliberately covered, and prove the after-fix inbound payload with redacted real Mattermost evidence.

## Reproduction Assessment

Yes, source-level: current main intentionally clears `effectiveReplyToId` for direct Mattermost chats, and the PR head passes that value into a helper that returns empty context when it is absent. I did not run a live Mattermost workspace.

## Solution Assessment

No. The best fix is to separate reply metadata hydration from thread/session routing, using a DM-only metadata target while preserving the documented flat-DM contract.
