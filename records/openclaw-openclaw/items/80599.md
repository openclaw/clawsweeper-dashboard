## Summary

Keep open: current main and v2026.6.8 still do not populate Mattermost `ReplyToBody`/`ReplyToSender`, so the narrow fix is useful, but this patch wires hydration to `effectiveReplyToId`, which is intentionally empty for DMs and can trigger lookups in non-DM threads; it also lacks real DM proof.

## What This Changes

The PR adds a Mattermost post fetch/cache plus reply-context helper and tests to populate `ReplyToBody` and `ReplyToSender` for replied-to Mattermost posts.

## Best Possible Solution

Separate metadata hydration from thread/session routing: use the actual DM reply target only for `ReplyToBody` and `ReplyToSender`, keep documented flat-DM delivery/session behavior, and prove it with monitor coverage plus redacted live Mattermost evidence.

## Reproduction Assessment

Yes, source-reproducible: current main returns `undefined` for direct `effectiveReplyToId`, and the PR passes that value into the new helper, which exits without a target. I did not run a live Mattermost workspace.

## Solution Assessment

No. The best fix is to pass a separate direct-message metadata target into the helper instead of reusing the routing/thread id.
