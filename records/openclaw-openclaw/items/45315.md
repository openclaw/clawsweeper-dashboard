## Summary

Keep open. The reported bug is valid and the screenshots are useful, but the patch is incomplete because routed abort replies still normalize a templated responsePrefix without any prefix context; current main also does not yet contain a complete fix.

## What This Changes

The PR pre-seeds reply-prefix context with config default model/provider/thinking values and adds helper-level tests for defaults and runtime overrides.

## Best Possible Solution

Carry a populated, runtime/session-aware ResponsePrefixContext through both routed fast-abort delivery and direct dispatcher fallback, with regression coverage for both paths.

## Reproduction Assessment

Yes, at source level: configure messages.responsePrefix with {model} or {thinkingLevel}, trigger stop before onModelSelected runs, and current main can normalize the abort reply without model/thinking context. I did not run a live channel in this read-only review.

## Solution Assessment

No. Config-default seeding improves one fallback path, but the maintainable fix should pass runtime/session-aware ResponsePrefixContext into both routed and direct fast-abort delivery.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9
