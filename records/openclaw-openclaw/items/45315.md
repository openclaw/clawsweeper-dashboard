## Summary

Keep open. The branch improves direct fast-abort replies by seeding config defaults, but it still leaves routed abort delivery able to normalize a templated response prefix without model/thinking context, so the central user-visible bug is only partially fixed.

## What This Changes

The branch pre-seeds `createReplyPrefixContext` with config default model/provider/thinking values and adds unit coverage for pre-seeding plus `onModelSelected` overrides.

## Best Possible Solution

Carry a populated, preferably session/runtime-aware `ResponsePrefixContext` through both direct dispatcher delivery and `routeReply`, then cover direct and routed fast-abort replies with templated prefix regression tests.

## Reproduction Assessment

Yes, at source level. Configure `messages.responsePrefix` with `{model}` or `{thinkingLevel}`, trigger `stop` or `/stop` before `onModelSelected` runs, and the current fast-abort/routed delivery paths can normalize the prefix without model/thinking context.

## Solution Assessment

No. Config-default pre-seeding is a useful partial fix, but the maintainable solution also needs to pass populated prefix context into routed delivery and prefer runtime/session metadata where available.
