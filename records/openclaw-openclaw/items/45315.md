## Summary

Keep open: current main still has the response-prefix abort bug, and this PR has real WhatsApp proof, but the patch is incomplete because routed abort delivery still normalizes templated prefixes without any prefix context.

## What This Changes

The branch pre-seeds reply-prefix context from config default model/provider/thinking values and adds unit tests for default seeding and `onModelSelected` overrides.

## Best Possible Solution

Carry a populated, preferably session/runtime-aware `ResponsePrefixContext` through both dispatcher fallback and `routeReply`, then cover direct and routed fast-abort replies with templated prefix regression tests.

## Reproduction Assessment

Yes, at source level. Configure `messages.responsePrefix` with `{model}` or `{thinkingLevel}`, trigger `stop` or `/stop` before `onModelSelected` runs, and current main can normalize the abort reply without model/thinking context.

## Solution Assessment

No. Config-default pre-seeding is a useful partial fix, but routed delivery still lacks prefix context and the durable fix should prefer runtime/session metadata where available.
