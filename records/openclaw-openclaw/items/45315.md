## Summary

Keep open. The reported bug is valid and the PR improves one direct abort path, but it is still incomplete because routed abort delivery can normalize a templated `responsePrefix` without a `responsePrefixContext`, leaving `{model}` or `{thinkingLevel}` literal.

## What This Changes

The PR pre-seeds `createReplyPrefixContext()` with config default model/provider/thinking values and adds unit coverage for the seeded context and `onModelSelected()` override.

## Best Possible Solution

Build a session/runtime-aware `ResponsePrefixContext` and pass it through both dispatcher fallback delivery and routed reply normalization, with direct and routed fast-abort regression coverage.

## Reproduction Assessment

Yes, at source level. Configure `messages.responsePrefix` with `{model}` or `{thinkingLevel}`, trigger `stop` before `onModelSelected()` runs, and current routed fast-abort delivery can normalize a templated prefix without interpolation context.

## Solution Assessment

No. Config-default pre-seeding improves direct dispatcher sends, but the maintainable fix must also pass populated prefix context into `routeReply()` and prefer runtime/session metadata where available.
