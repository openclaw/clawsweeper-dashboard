## Summary

Keep open: the reported bug is real and the PR has useful WhatsApp proof, but the patch is incomplete because routed abort delivery still normalizes templated prefixes without a `responsePrefixContext`.

## What This Changes

The PR pre-seeds reply-prefix context with config default model/provider/thinking values and adds unit coverage for seeded defaults and runtime overwrite behavior.

## Best Possible Solution

Carry a session/runtime-aware `ResponsePrefixContext` through both direct dispatcher delivery and routed `routeReply` normalization, with direct and routed fast-abort regression tests.

## Reproduction Assessment

Yes, at source level. Configure `messages.responsePrefix` with `{model}` or `{thinkingLevel}`, start a run, then send `stop` or `/stop` before `onModelSelected`; current main can send the abort reply through normalization without populated model/thinking context.

## Solution Assessment

No. Pre-seeding config defaults is a useful partial mitigation, but the maintainable fix also needs to pass resolved prefix context through routed abort delivery.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Pass prefix context into routed abort replies:** `src/channels/reply-prefix.ts:55-58`
  - body: The new seed only helps deliveries that use the channel dispatcher. Fast aborts first try `routeReplyToOriginating()`, and `routeReply()` normalizes with only `responsePrefix` and no `responsePrefixContext`, so routed conversations with `{model}` or `{thinkingLevel}` still leak raw placeholders. Please carry the seeded context into routed abort normalization and add a routed regression test.
  - confidence: 0.86
