## Best Possible Solution

Rebase and update the fix so abort replies use the active session/channel model context for both dispatcher and routed-origin delivery, with focused regression coverage.

No. Default seeding is a useful mitigation for simple direct replies, but the best fix should derive abort prefixes from active session/channel model selection and pass that context through routed delivery.

## Review Findings

Overall correctness: patch is incorrect

Full review comments:

- **[P2] Resolve abort prefixes from active model context:** `src/channels/reply-prefix.ts:44-49`
  - body: The new seed uses `resolveDefaultModelForAgent`, so fast-abort replies can show the config-default model/thinking even when the current session has a `/model` override or the channel has `channels.modelByChannel`. Normal reply selection honors those overrides before `onModelSelected`, but abort returns before that callback; derive this prefix from the same active session/channel selection instead.
  - confidence: 0.86
- **[P2] Pass prefix context into routed abort delivery:** `src/channels/reply-prefix.ts:44`
  - body: This context is only available to dispatcher paths. When `dispatch-from-config` sends the abort via `routeReplyToOriginating`, `routeReply` calls `normalizeReplyPayload` with `responsePrefix` but no `responsePrefixContext`, so cross-channel abort replies can still leak raw `{model}` or `{thinkingLevel}` placeholders. Pass the populated abort context through routed delivery too.
  - confidence: 0.82
