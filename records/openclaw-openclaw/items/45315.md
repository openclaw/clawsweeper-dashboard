## Summary

This PR should stay open because it fixes only the dispatcher-owned part of the fast-abort prefix path; routed abort delivery can still normalize a templated responsePrefix without any ResponsePrefixContext and leak the same placeholders. The WhatsApp before/after screenshots are useful real behavior proof for the partial change, but the patch still has a blocking routed-delivery gap before merge.

## What This Changes

The branch pre-seeds reply prefix context with config-default model/provider/thinking values and adds unit coverage for seeded defaults, runtime override, and empty-config fallback.

## Best Possible Solution

Carry a populated ResponsePrefixContext through both dispatcher and routed abort normalization, using available session/runtime metadata where practical, with regression coverage for direct and routed fast-abort replies.

## Reproduction Assessment

Yes, at source level. Configure messages.responsePrefix with {model} or {thinkingLevel}, start a run, then send stop or /stop before onModelSelected(); current main can normalize the abort reply without model/thinking context, and the template engine leaves missing values literal.

## Solution Assessment

No. Pre-seeding config defaults is a useful partial fallback, but the patch does not cover routed abort delivery where routeReply() normalizes without responsePrefixContext.
