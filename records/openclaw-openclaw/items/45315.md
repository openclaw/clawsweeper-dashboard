## Summary

Keep open. The reported abort-prefix bug is still real on current main, but this PR is not merge-ready because it seeds fallback prefix values from config defaults rather than the effective runtime/session model metadata that normal replies use, and it does not use catalog-backed thinking defaults.

## What This Changes

This PR pre-seeds `createReplyPrefixContext()` with config-default provider, model, and thinking values and adds tests for default seeding, runtime override, and fallback behavior.

## Best Possible Solution

Land one canonical fix that populates abort response-prefix context from the same effective runtime metadata used for normal replies, including session or parent-session model overrides, channel model overrides where available, and catalog-backed thinking defaults, with focused regression coverage for fast-abort templated replies.

## Reproduction Assessment

Yes. Configure `messages.responsePrefix` with `{model}` or `{thinkingLevel}`, start a run, then send `/stop`; current main returns through the fast-abort path before `onModelSelected()` fills the prefix context, so missing variables can remain literal.

## Solution Assessment

No. This PR is a useful partial improvement for the config-default case, but the narrowest maintainable fix is to seed or persist the effective runtime model/thinking metadata used by the actual run rather than guessing from static config defaults.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Seed abort prefixes from the effective session model:** `src/channels/reply-prefix.ts:42`
  - body: This seeds fast-abort prefix values from `resolveDefaultModelForAgent()`, but normal runs can use stored session or parent-session overrides and channel model overrides. Because fast abort returns before `onModelSelected()` runs, users who changed models can receive a resolved prefix with the wrong model. Thread or persist the effective runtime model metadata for abort replies instead of using only config defaults.
  - confidence: 0.87
- **[P2] Use catalog-backed thinking defaults before seeding:** `src/channels/reply-prefix.ts:43-47`
  - body: `resolveThinkingDefault()` is called without a catalog here, so reasoning-capable models with no explicit `thinkingDefault` can seed `off` while the normal selection path hydrates catalog metadata and may resolve a non-off default. Use the same catalog-backed or persisted runtime thinking value before formatting abort prefixes.
  - confidence: 0.82
