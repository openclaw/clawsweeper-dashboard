## Summary

Keep open: this PR targets a real source-reproducible abort-prefix bug and includes sufficient screenshot proof, but the patch only seeds dispatcher prefix context while routed abort replies can still normalize a templated prefix with no interpolation context.

## What This Changes

The PR pre-seeds `createReplyPrefixContext()` with config-derived provider/model/thinking defaults and adds unit coverage for default seeding, runtime override, and empty-config fallback.

## Best Possible Solution

Build a session/runtime-aware `ResponsePrefixContext` and pass it through both direct dispatcher delivery and `routeReply()` abort delivery, with direct and routed fast-abort regression tests while preserving later `onModelSelected` overrides.

## Reproduction Assessment

Yes, at source level. Configure `messages.responsePrefix` with `{model}` or `{thinkingLevel}`, start a run, then send `stop` or `/stop` before `onModelSelected`; current main can route the abort payload through normalization with a templated prefix and no prefix context.

## Solution Assessment

No. Pre-seeding config defaults improves the dispatcher fallback, but routed abort replies still normalize with `responsePrefix` and no `responsePrefixContext`, so the maintainable fix needs to carry the populated context into `routeReply()` as well.

## Review Findings
