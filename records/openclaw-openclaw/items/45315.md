## Summary

Keep open. The PR fixes part of the visible raw-placeholder problem, but the routed fast-abort path still normalizes a templated `responsePrefix` without any `responsePrefixContext`, so routed stop replies can still leak `{model}` and `{thinkingLevel}`.

## What This Changes

The PR pre-seeds `ResponsePrefixContext` with config-default provider, model, and thinking-level values in `createReplyPrefixContext()` and adds unit tests for seeded defaults and `onModelSelected()` overrides.

## Best Possible Solution

Carry a populated, session/runtime-aware `ResponsePrefixContext` into both dispatcher fallback and `routeReply`-routed fast-abort delivery, with regression coverage for routed stop replies and normal `onModelSelected()` override behavior.

## Reproduction Assessment

Yes, at source level. Configure `messages.responsePrefix` with `{model}` or `{thinkingLevel}`, trigger `stop` or `/stop` before `onModelSelected()` runs, and current routed abort delivery can normalize the reply without model/thinking context.

## Solution Assessment

No. Pre-seeding config defaults helps paths that consume `createReplyPrefixContext()` but does not fix `routeReply()` routed delivery, which still lacks `responsePrefixContext`.
