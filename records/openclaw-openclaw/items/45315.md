## Summary

Keep open: the PR is a useful partial bug fix with real WhatsApp proof, but it still does not carry a populated prefix context into routed fast-abort delivery, so routed stop replies can still leak `{model}` or `{thinkingLevel}`.

## What This Changes

The PR pre-seeds reply-prefix template context with config-default provider/model/thinking values and adds unit coverage for seeded defaults and runtime override.

## Best Possible Solution

Carry a populated runtime/session-aware ResponsePrefixContext into both dispatcher fallback and routeReply-routed fast-abort delivery, with regression coverage for routed stop replies and normal onModelSelected override behavior.

## Reproduction Assessment

Yes, at source level. Configure messages.responsePrefix with {model} or {thinkingLevel}, trigger stop or /stop before onModelSelected runs, and routed abort delivery can normalize the reply without model/thinking context.

## Solution Assessment

No. Pre-seeding config defaults helps dispatcher paths that use createReplyPrefixContext, but the routed abort path still normalizes without responsePrefixContext; the fix should plumb that context into routed delivery too.
