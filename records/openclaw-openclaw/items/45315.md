## Summary

Keep open: this is a valid bug fix with visible proof, but the patch still misses routed fast-abort delivery, so some stop replies can continue to leak `{model}` or `{thinkingLevel}` placeholders.

## What This Changes

The PR pre-seeds `ResponsePrefixContext` with config-default provider, model, and thinking level in `createReplyPrefixContext()` and adds unit coverage for seeded defaults and runtime override.

## Best Possible Solution

Carry a populated, runtime-aware `ResponsePrefixContext` into both dispatcher fallback and `routeReply`-routed fast-abort delivery, preferring session/runtime metadata when it is available.

## Reproduction Assessment

Yes, at source level. Configure `messages.responsePrefix` with `{model}` or `{thinkingLevel}`, send `stop` before `onModelSelected` runs, and the routed fast-abort path normalizes with no prefix context.

## Solution Assessment

No. Pre-seeding config defaults helps the dispatcher fallback path, but the maintainable fix must also pass populated prefix context into routed abort delivery.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Pass prefix context into routed abort replies:** `src/channels/reply-prefix.ts:51-57`

The new seeded values only help the dispatcher fallback path, so routed stop replies can still emit literal `{model}` or `{thinkingLevel}`; plumb the populated context into routed delivery and cover that path.
