## Summary

Keep open. The PR addresses the core symptom for direct dispatcher abort replies and includes real WhatsApp before/after proof, but the patch does not cover routed abort delivery where `routeReply` still normalizes a templated `responsePrefix` without any prefix context.

## What This Changes

The PR pre-seeds `ResponsePrefixContext` from config-default model/provider/thinking values and adds unit coverage for default seeding and `onModelSelected` overrides.

## Best Possible Solution

Carry a session/runtime-aware response prefix context through both direct dispatcher delivery and routed `routeReply` abort delivery, then cover direct and routed fast-abort replies with regression tests while preserving the later `onModelSelected` override.

## Reproduction Assessment

Yes, at source level. Configure `messages.responsePrefix` with `{model}` or `{thinkingLevel}`, start a run, then send `stop` or `/stop` before `onModelSelected`; the fast-abort path can normalize a templated prefix without populated model/thinking context.

## Solution Assessment

No. Pre-seeding config defaults is a useful partial fix, but the maintainable fix should also pass populated prefix context into routed abort normalization and avoid stale defaults for active session/channel/model metadata.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Pass prefix context through routed abort replies:** `src/channels/reply-prefix.ts:54-57`
  - body: The new seed only reaches dispatcher normalization. In the fast-abort path, `routeReplyToOriginating` is tried before the dispatcher fallback, and `routeReply` recomputes `responsePrefix` then calls `normalizeReplyPayload` without `responsePrefixContext`, so routed channels can still send literal `{model}` / `{thinkingLevel}` tokens. Please pass the populated prefix context into routed delivery or normalize the abort payload before routing, and add a routed fast-abort regression.
  - confidence: 0.87
