## Summary

Keep open: the PR targets a real user-visible bug, but the current patch only seeds the dispatcher prefix context and does not carry that context into `routeReply()`, so routed fast-abort replies can still send literal template variables.

## What This Changes

The PR pre-seeds `createReplyPrefixContext()` with config-derived provider/model/thinking defaults and adds unit tests for default seeding, runtime override, and fallback behavior.

## Best Possible Solution

Carry a populated `ResponsePrefixContext` into routed abort normalization, preferably using session/runtime metadata when available, and cover both direct and routed fast-abort paths while preserving later `onModelSelected()` overrides.

## Reproduction Assessment

Yes, at source level. Configure `messages.responsePrefix` with `{model}` or `{thinkingLevel}`, start a run, then send `stop` or `/stop` before `onModelSelected()`; current main can route the abort payload through normalization without prefix context.

## Solution Assessment

No. Pre-seeding config defaults helps dispatcher fallback, but routed abort replies still normalize with `responsePrefix` and no context, so the maintainable fix needs to pass a populated context into `routeReply()` as well.
