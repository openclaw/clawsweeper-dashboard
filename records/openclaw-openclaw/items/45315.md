## Summary

Keep open. The linked bug is real and current main still has source-level paths where abort replies can apply a templated responsePrefix without model/thinking context; this PR improves the direct fallback but is incomplete because routed abort replies still normalize without a responsePrefixContext.

## What This Changes

The PR pre-seeds reply-prefix context with config-default model/provider/thinking values and adds unit tests for default seeding, runtime override, and empty-config fallback.

## Best Possible Solution

Build a populated, session/runtime-aware ResponsePrefixContext and pass it through both dispatcher fallback delivery and routeReply normalization, with fallback defaults and direct plus routed fast-abort regression coverage.

## Reproduction Assessment

Yes, source-level. Configure messages.responsePrefix with {model} or {thinkingLevel}, trigger stop before onModelSelected, and current main can normalize the fast-abort reply with a templated prefix and no model/thinking context.

## Solution Assessment

No. Config-default pre-seeding improves the direct path, but the patch misses routed abort normalization and does not carry override-aware metadata through the delivery boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9
