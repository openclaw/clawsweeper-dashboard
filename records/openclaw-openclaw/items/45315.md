## Summary

Keep open: the reported bug is source-reproducible on current main, and this PR is a useful but incomplete fix because routed fast-abort delivery still normalizes a templated response prefix without template context.

## What This Changes

The PR pre-seeds reply-prefix context with config default model/provider/thinking values in `createReplyPrefixContext()` and adds unit tests for that helper.

## Best Possible Solution

Carry a populated, session/runtime-aware `ResponsePrefixContext` through both dispatcher fallback and `routeReply` fast-abort delivery, with direct and routed regression tests.

## Reproduction Assessment

Yes, at source level. Configure `messages.responsePrefix` with `{model}` or `{thinkingLevel}`, trigger `stop` or `/stop` before `onModelSelected()` runs, and current main can normalize the abort reply without model/thinking context.

## Solution Assessment

No. Config-default pre-seeding improves the dispatcher fallback path, but routed abort delivery still lacks prefix context and the durable fix should carry runtime/session metadata into both delivery paths.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Pass prefix context through routed abort delivery:** `src/channels/reply-prefix.ts:44-47`
  - body: Fast aborts try `routeReplyToOriginating()` before the dispatcher fallback, but `routeReply()` still normalizes `responsePrefix` without `responsePrefixContext`. This patch only seeds dispatcher-created contexts, so routed `/stop` replies can still leak `{model}` or `{thinkingLevel}`; carry a populated context into routed delivery and cover that path.
  - confidence: 0.9
