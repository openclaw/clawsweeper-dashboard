## Summary

## What This Changes

The PR pre-seeds reply-prefix context from configured default provider/model/thinking values and adds tests for default seeding plus later `onModelSelected` overrides.

## Best Possible Solution

Carry the effective model, provider, and thinking facts used by normal model selection into fast-abort prefix rendering, with a deliberate fallback only when active facts are unavailable.

## Reproduction Assessment

Yes, at source level. Current main routes fast-abort replies before `onModelSelected` fills dynamic prefix facts, and the inspected WhatsApp screenshot shows literal placeholders in the abort reply.

## Solution Assessment

No. Pre-seeding config defaults is a useful mitigation, but the best fix is to pass the effective active model-selection facts into fast-abort prefix rendering so abort replies match normal replies.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.96

Full review comments:

- **[P1] Seed abort prefixes from active model state:** `src/channels/reply-prefix.ts:44-56`
  - body: The new pre-seed uses `resolveDefaultModelForAgent`, but fast abort can happen after a session `/model`, parent-session, or channel model override is active. Normal replies honor those sources before the run; with this patch, abort replies can replace raw placeholders with stale config-default model/thinking values. Pass the effective selected facts into the prefix context and cover those override cases.
  - confidence: 0.96
- **[P2] Use catalog-backed thinking defaults:** `src/channels/reply-prefix.ts:45-49`
  - body: `resolveThinkingDefault` is called without a catalog here, while normal model selection resolves thinking defaults through configured, manifest, or runtime catalog data. Reasoning-capable models without explicit config can therefore show `off` in abort prefixes instead of the normal catalog-derived level.
  - confidence: 0.95
