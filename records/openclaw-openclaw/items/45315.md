## Summary

Keep open. The PR targets a real source-reproducible bug and has adequate visible proof, but the patch is still incomplete because routed abort replies can still normalize a templated response prefix without model/thinking context.

## What This Changes

The PR pre-seeds `createReplyPrefixContext()` with config-default model/provider/thinking values and adds tests for pre-seeding, runtime override, and fallback defaults.

## Best Possible Solution

Build a runtime/session-aware `ResponsePrefixContext` and pass it through both direct dispatcher delivery and `routeReply()` for fast-abort replies, with direct and routed regression coverage.

## Reproduction Assessment

Yes, at source level. Configure `messages.responsePrefix` with `{model}` or `{thinkingLevel}`, trigger `stop` or `/stop` before `onModelSelected()` runs, and current main can normalize the abort reply without model/thinking context.

## Solution Assessment

No. Config-default pre-seeding improves direct dispatcher aborts, but the maintainable fix also needs to carry populated prefix context into routed delivery and prefer runtime/session metadata where available.
