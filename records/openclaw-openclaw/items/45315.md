## Summary

Keep open: the branch fixes the dispatcher-local prefix context but still misses current main's routed fast-abort path, so routed stop replies can continue leaking raw `{model}` or `{thinkingLevel}` tokens; the branch is also currently conflicting with main.

## What This Changes

The branch pre-seeds `ResponsePrefixContext` with config-default provider, model, and thinking-level values in `createReplyPrefixContext()` and adds unit tests for seeded defaults plus `onModelSelected()` overrides.

## Best Possible Solution

Carry a populated, session/runtime-aware `ResponsePrefixContext` into both dispatcher fallback and `routeReply()` routed fast-abort delivery, with regression coverage for routed stop replies and normal `onModelSelected()` override behavior.

## Reproduction Assessment

Yes, at source level. Configure `messages.responsePrefix` with `{model}` or `{thinkingLevel}`, trigger `stop` or `/stop` before `onModelSelected()` runs, and the routed abort path can normalize the reply without model/thinking context.

## Solution Assessment

No. Pre-seeding config defaults helps callers that consume `createReplyPrefixContext()`, but the maintainable fix also needs to carry prefix context into routed abort delivery and cover that path with tests.
