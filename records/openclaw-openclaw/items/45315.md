## Summary

Keep open. The patch improves the direct abort case, and the WhatsApp screenshots are credible proof for that visible path, but routed abort replies can still normalize a templated response prefix without any model/thinking context and leak literal placeholders.

## What This Changes

Pre-seeds reply-prefix context from config-default model/provider/thinking values and adds unit coverage for default seeding, runtime override, and fallback defaults.

## Best Possible Solution

Carry a populated prefix context through both direct dispatcher delivery and routed abort normalization, seeded from available session/runtime model metadata and falling back to config defaults only when needed.

## Reproduction Assessment

Yes, at source level. Configure `messages.responsePrefix` with `{model}` or `{thinkingLevel}`, trigger `stop` before `onModelSelected()`, and current main can normalize direct or routed abort replies without model/thinking context.

## Solution Assessment

No. Config-default pre-seeding is a useful fallback, but the patch still misses routed abort normalization and should pass or build prefix context for `routeReply()` as part of the same fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Pass prefix context into routed abort replies:** `src/channels/reply-prefix.ts:34-47`

Please pass or build prefix context for routed normalization and add routed fast-abort coverage.
