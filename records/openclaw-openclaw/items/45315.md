## Summary

Keep open. The report is valid and the PR improves one path, but current main still has the bug and this patch does not carry prefix context into routed abort delivery, so routed stop replies can still leak literal template variables.

## What This Changes

The PR pre-seeds reply prefix context with default model, provider, and thinking-level values so abort replies do not display raw responsePrefix template placeholders.

## Best Possible Solution

Build a session/runtime-aware ResponsePrefixContext and pass it through both dispatcher fallback delivery and routeReply normalization, with direct and routed fast-abort regression coverage.

## Reproduction Assessment

Yes, at source level. Configure messages.responsePrefix with {model} or {thinkingLevel}, trigger stop before onModelSelected runs, and routed fast-abort normalization can still run without responsePrefixContext.

## Solution Assessment

No. Config-default pre-seeding improves the dispatcher fallback path, but it does not fix routed abort replies and can still disagree with runtime/session metadata.
