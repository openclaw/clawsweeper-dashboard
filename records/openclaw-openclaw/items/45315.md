## Summary

Keep open. The reported bug is valid on current main, and this PR is useful but incomplete because routed abort delivery can still normalize a templated `responsePrefix` without any model/thinking context.

## What This Changes

The PR pre-seeds reply prefix context with configured model/provider/thinking defaults and adds unit coverage for `createReplyPrefixContext()`.

## Best Possible Solution

Build a session/runtime-aware `ResponsePrefixContext` and pass it through both dispatcher fallback delivery and routed reply normalization, with direct and routed fast-abort regression coverage.

## Reproduction Assessment

Yes, at source level. Configure `messages.responsePrefix` with `{model}` or `{thinkingLevel}`, trigger `stop` or `/stop` before `onModelSelected()` runs, and current routed fast-abort delivery can normalize the abort reply with no interpolation context.

## Solution Assessment

No. The PR's config-default pre-seeding improves dispatcher-created contexts, but the maintainable fix must also carry populated prefix context into `routeReply()` and prefer runtime/session metadata where available.
