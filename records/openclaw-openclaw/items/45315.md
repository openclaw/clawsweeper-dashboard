## Summary

Keep open. The branch improves the direct fallback, but it still misses routed fast-abort delivery: current main's `routeReply()` normalizes `responsePrefix` without a `responsePrefixContext`, and this PR does not touch that path, so routed replies can still leak placeholders.

## What This Changes

The PR updates `src/channels/reply-prefix.ts` to seed `ResponsePrefixContext` from config defaults and adds `src/channels/reply-prefix.test.ts` coverage for seeded defaults, runtime override, and empty-config fallback.

## Best Possible Solution

Carry a populated `ResponsePrefixContext` through both dispatcher and routed fast-abort normalization, using session/runtime metadata where practical, with direct and routed regression tests.

## Reproduction Assessment

Yes, at source level. Configure `messages.responsePrefix` with `{model}` or `{thinkingLevel}`, trigger `stop` before `onModelSelected()`, and current main can normalize a routed abort reply with a templated prefix but no prefix context.

## Solution Assessment

No. Pre-seeding config defaults is a useful partial fallback, but the fix is incomplete until routed abort delivery passes or builds the same prefix context before normalization.
