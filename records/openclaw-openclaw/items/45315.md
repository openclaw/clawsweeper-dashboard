## Summary

Keep open: this is a real user-visible bug fix attempt with sufficient screenshot proof, but the patch is incomplete because routed fast-abort replies still normalize a templated responsePrefix without a ResponsePrefixContext, and the PR is currently merge-conflicting.

## What This Changes

The PR pre-seeds reply prefix context from config defaults in src/channels/reply-prefix.ts and adds createReplyPrefixContext unit coverage.

## Best Possible Solution

Build a session/runtime-aware ResponsePrefixContext and pass it through both dispatcher fallback delivery and routeReply normalization, with direct and routed fast-abort regression coverage.

## Reproduction Assessment

Yes, at source level. Configure messages.responsePrefix with {model} or {thinkingLevel}, trigger stop before onModelSelected runs, and the routed fast-abort path can still normalize the reply without responsePrefixContext.

## Solution Assessment

No. Config-default pre-seeding improves one fallback path, but the maintainable fix must also carry a populated context into routed reply normalization and prefer runtime/session metadata where available.
