## Summary

Keep open: current main and the latest release still have the abort-prefix gap, but this PR is not merge-ready because it seeds config defaults instead of the effective active model state used by normal replies.

## What This Changes

The PR pre-seeds reply-prefix context with configured default provider, model, and thinking level so early abort replies interpolate `{model}` and `{thinkingLevel}` instead of showing raw template tokens.

## Best Possible Solution

Carry effective model/provider/thinking state into fast-abort prefix rendering and keep a deliberate fallback for cases where active state is unavailable.

## Reproduction Assessment

Yes, at source level. Current main routes fast-abort replies before `onModelSelected` fills dynamic prefix facts, and the inspected WhatsApp proof shows the literal placeholders before the patch.

## Solution Assessment

No. Default seeding is a useful mitigation, but the best fix is to feed fast-abort rendering with the effective active model/provider/thinking state used by normal replies.

- **[P1] Seed abort prefixes from active model state:** `src/channels/reply-prefix.ts:44-56`
  - body: The new pre-seed uses configured defaults, but fast abort can happen after a session or parent-session `/model` override, or a channel `modelByChannel` override, is active.   - confidence: 0.95

  - body: `resolveThinkingDefault` is called without the catalog path normal model selection uses, so reasoning-capable models without explicit config can show `off` in abort prefixes instead of the normal catalog-derived level. Seed the prefix from the same catalog-backed thinking resolution used by normal model selection.
  - confidence: 0.94
