## Summary

Keep open: the reported bug is real and the branch improves the direct dispatcher fallback, but it does not carry the populated prefix context into routed abort replies, so supported template variables can still leak as literals.

## What This Changes

The PR pre-seeds `createReplyPrefixContext()` with config-derived provider/model/thinking defaults and adds unit tests for default seeding, runtime override, and fallback defaults.

## Best Possible Solution

Carry a populated, session/runtime-aware `ResponsePrefixContext` through both dispatcher fallback delivery and routed abort normalization, then cover direct and routed fast-abort paths while preserving later `onModelSelected` overrides.

## Reproduction Assessment

Yes, at source level. Configure `messages.responsePrefix` with `{model}` or `{thinkingLevel}`, start a run, then send `stop` or `/stop` before `onModelSelected`; current main can route the abort payload through normalization with no prefix context.

## Solution Assessment

No. Pre-seeding config defaults helps direct dispatcher fallback, but routed abort replies still normalize with `responsePrefix` and no context, so the maintainable fix needs to pass the populated context into `routeReply()` as well.
