## Summary

Keep open: this PR addresses a real responsePrefix abort bug and includes visible WhatsApp proof, but the patch is incomplete because routed fast-abort delivery still normalizes replies without a responsePrefixContext, so templated prefixes can remain literal on that path.

## What This Changes

The PR pre-seeds createReplyPrefixContext() with config-default model/provider/thinking values and adds unit coverage for those defaults plus onModelSelected overrides.

## Best Possible Solution

Build a session/runtime-aware ResponsePrefixContext and pass it through both dispatcher fallback delivery and routeReply normalization, with direct and routed fast-abort regression coverage.

## Reproduction Assessment

Yes, at source level. Configure messages.responsePrefix with {model} or {thinkingLevel}, trigger stop before onModelSelected runs, and current routed fast-abort delivery can still normalize with a templated prefix but no context.

## Solution Assessment

No. Config-default pre-seeding improves the dispatcher fallback path, but the maintainable fix needs to carry populated prefix context into routed reply normalization and prefer runtime/session metadata where available.
