## Summary

Keep open: the reported abort-prefix bug is still source-reproducible on current main and this PR has useful WhatsApp proof, but the branch is dirty and the patch only seeds config defaults, leaving active session/channel/routed abort state unresolved.

## What This Changes

The PR pre-seeds reply-prefix context with config default provider, model, and thinking values and adds tests for seeded defaults, runtime override, and empty config fallback.

## Best Possible Solution

Rebase on current `main` and resolve abort reply prefixes from canonical active model metadata for both dispatcher and routed replies, falling back to config defaults only when no session, channel, runtime, or catalog facts exist.

## Reproduction Assessment

Yes. Source inspection shows current `main` leaves model/thinking fields unset until `onModelSelected`, while fast abort can send before that callback; the PR screenshots also show the before/after WhatsApp symptom.

## Solution Assessment

No, not as-is. Pre-seeding config defaults fixes the simplest placeholder leak, but the existing dynamic-prefix contract calls for the actual selected model after overrides/fallbacks and routed abort normalization needs the same context.
