## Summary

Keep open. The PR improves the direct abort fallback path, but current main still has a routed abort path where `routeReply()` applies a templated `responsePrefix` without any `responsePrefixContext`, so routed `/stop` replies can still leak `{model}` or `{thinkingLevel}`.

## What This Changes

The PR pre-seeds reply prefix context with config-default model, provider, and thinking values and adds unit coverage for default seeding and runtime override.

## Best Possible Solution

Build a session/runtime-aware `ResponsePrefixContext` and pass it through both dispatcher fallback delivery and `routeReply()` normalization, with direct and routed fast-abort regression coverage.

## Reproduction Assessment

Yes, at source level. Configure `messages.responsePrefix` with `{model}` or `{thinkingLevel}`, trigger `stop` before `onModelSelected`, and current main can normalize the abort reply with missing template context, especially through the routed delivery path.

## Solution Assessment

No. Config-default pre-seeding improves the direct dispatcher path, but the maintainable fix must also pass a populated context into routed reply normalization and should prefer runtime/session metadata where available.
