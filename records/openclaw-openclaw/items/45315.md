## Summary

Keep open: current main still leaves model/thinking response-prefix variables unset before fast abort, and this PR’s static-default seed is a partial patch with correctness gaps already covered more directly by the open runtime-metadata draft #45454.

## What This Changes

The PR pre-seeds reply prefix template context with config-default provider, model, and thinking values and adds tests for default seeding, runtime override, and fallback behavior.

## Best Possible Solution

Land one canonical fix that formats abort prefixes from effective persisted/runtime model metadata, including session/channel overrides and catalog-backed thinking defaults, then close the paired issue and superseded PRs.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure `messages.responsePrefix` with `{model}` or `{thinkingLevel}`, start a run, then send `/stop`; fast abort sends the reply before `onModelSelected()` fills those fields, leaving literals in current main.

## Solution Assessment

No. The PR improves the config-default case, but the narrow maintainable fix should reuse effective runtime model/thinking metadata rather than guessing from static defaults before session, channel, and catalog resolution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Seed abort prefixes from the effective runtime model:** `src/channels/reply-prefix.ts:42`
  - body: This seeds fast-abort prefix values from `resolveDefaultModelForAgent()`, but normal runs can use stored session or parent-session overrides and channel model overrides. Because fast abort returns before `onModelSelected()` runs, users who changed models can receive a resolved prefix with the wrong model; thread or persist the effective runtime model metadata for abort replies instead of using only config defaults.
  - confidence: 0.87
- **[P2] Use catalog-backed thinking defaults before seeding:** `src/channels/reply-prefix.ts:43-47`
  - body: `resolveThinkingDefault()` is called without a catalog here, so reasoning-capable models with no explicit `thinkingDefault` can seed `off` while the normal selection path hydrates catalog metadata and may resolve a non-off default. Use the same catalog-backed or persisted runtime thinking value before formatting abort prefixes.
  - confidence: 0.82
