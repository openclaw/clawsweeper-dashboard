## Summary

Keep this PR open, but it is not merge-ready as-is. The direct abort-path fix is useful and the WhatsApp screenshots show the intended improvement, but the patch only seeds the dispatcher prefix context; routed abort replies still normalize `responsePrefix` without a context, so `{model}` and `{thinkingLevel}` can still leak in that path.

## What This Changes

The PR pre-seeds reply-prefix context from configured default model/provider/thinking values and adds tests for direct abort prefix interpolation and model-selection override behavior.

## Best Possible Solution

Use one shared session/runtime-aware `ResponsePrefixContext` and pass it through both dispatcher and `routeReply`, with focused tests for direct and routed fast-abort replies while preserving later `onModelSelected` overrides.

## Reproduction Assessment

Yes. Source inspection shows the fast-abort reply can run before `onModelSelected`, and missing prefix context values are preserved as literal placeholders; I did not run a live channel repro in this read-only review.

## Solution Assessment

No, not completely. Pre-seeding defaults is a useful narrow improvement for direct dispatcher delivery, but the best fix should also carry the same prefix context into routed reply normalization so all abort delivery paths resolve the template.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

- **[P2] Pass prefix context through routed abort replies:** `src/channels/reply-prefix.ts:34-44`

Thread the context into route-reply or normalize the abort payload with it before routing.
