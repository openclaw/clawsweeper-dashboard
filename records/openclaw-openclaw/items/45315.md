## Summary

Keep open. The bug is valid and the PR has real WhatsApp before/after proof, but the patch is still incomplete because routed abort replies can normalize a templated responsePrefix without any ResponsePrefixContext.

## What This Changes

The PR seeds reply-prefix context with config default model/provider/thinking values and adds unit tests for the seeded defaults and runtime override behavior.

## Best Possible Solution

Carry a session/runtime-aware ResponsePrefixContext through both dispatcher delivery and routeReply normalization, with direct and routed fast-abort regression tests.

## Reproduction Assessment

Yes, at source level. Configure messages.responsePrefix with {model} or {thinkingLevel}, start a run, then send stop or /stop before onModelSelected; current main can send the abort reply through normalization without populated model/thinking context.

## Solution Assessment

No. Pre-seeding config defaults is a useful partial mitigation, but the maintainable fix needs to pass resolved prefix context through routed abort delivery as well.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Pass prefix context into routed abort replies:** `src/channels/reply-prefix.ts:43-49`

Please carry the seeded context into routed abort normalization and add a routed regression test.
