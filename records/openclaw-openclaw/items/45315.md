Keep open. The PR is a useful focused improvement with sufficient WhatsApp before/after proof, but source review confirms routed fast-abort delivery can still normalize `responsePrefix` without model/thinking context, and the branch is currently conflicting with main.

## What This Changes

The branch pre-seeds `ResponsePrefixContext` from config defaults in `createReplyPrefixContext()` and adds unit tests for seeded defaults plus `onModelSelected()` overrides.

## Best Possible Solution

Carry a populated runtime/session `ResponsePrefixContext` through both dispatcher fallback and routed fast-abort delivery, with regression coverage for routed stop replies and normal `onModelSelected()` override behavior.

## Reproduction Assessment

Yes, at source level. Configure `messages.responsePrefix` with `{model}` or `{thinkingLevel}`, trigger `stop` or `/stop` before `onModelSelected()` runs, and current routed abort delivery can normalize a reply without model/thinking context.

## Solution Assessment

No. The PR's default seeding improves one path, but the maintainable fix needs to carry populated prefix context into routed fast-abort delivery and avoid treating config defaults as active session state.
