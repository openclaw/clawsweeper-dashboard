## Summary

Keep open. The underlying bug is source-reproducible on current main, and this PR is a useful focused attempt, but it is currently conflicting and does not carry the seeded prefix context into routed abort replies, so it should not be merged as-is.

## What This Changes

This PR pre-seeds `createReplyPrefixContext()` with default model/provider/thinking values and adds tests for pre-seeding plus `onModelSelected()` override behavior.

## Best Possible Solution

Land a rebased focused fix that builds a safe response-prefix context before fast aborts and passes it through both direct dispatcher delivery and routed reply normalization, with regression coverage for each path.

## Reproduction Assessment

Yes, at source level. Configure `messages.responsePrefix` with `{model}` or `{thinkingLevel}`, trigger `stop` or `/stop` before `onModelSelected()` runs, and current main can normalize the abort reply with missing context so placeholders remain literal.

## Solution Assessment

No, not as-is. Pre-seeding config defaults is a useful direct-path improvement, but the fix should also thread a populated prefix context into routed replies and preferably use available session/runtime model metadata rather than only global defaults.
