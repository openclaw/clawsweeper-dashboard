## Summary

Keep open: current main still has the documented abort-prefix bug, and this PR is useful but incomplete because routed abort delivery still normalizes templated prefixes without prefix context.

## What This Changes

The PR pre-seeds reply-prefix context from config defaults and adds createReplyPrefixContext tests for abort replies.

## Best Possible Solution

Build a session/runtime-aware ResponsePrefixContext and pass it through both direct dispatcher delivery and routeReply, with direct and routed fast-abort regressions.

## Reproduction Assessment

Yes, at source level. Configure messages.responsePrefix with {model} or {thinkingLevel}, trigger stop before onModelSelected runs, and current main can normalize the abort reply with missing model/thinking context.

## Solution Assessment

No. Config-default pre-seeding improves one path, but the narrow maintainable fix must also carry populated prefix context into routed delivery and prefer runtime/session metadata where available.
