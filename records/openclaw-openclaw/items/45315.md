## Summary

Keep open. This PR targets a real source-reproducible bug, but the current patch only seeds the reply-prefix context used by dispatcher-style delivery and still misses the routed fast-abort path, so it should not merge as-is.

## What This Changes

The PR pre-seeds `createReplyPrefixContext()` with config-default provider, model, and thinking-level values and adds unit coverage for seeded defaults and runtime override.

## Best Possible Solution

Carry a populated runtime/session-aware `ResponsePrefixContext` into both dispatcher fallback and `routeReply`-routed fast-abort delivery, with regression coverage for the routed stop path.

## Reproduction Assessment

Yes, at source level. Configure `messages.responsePrefix` with `{model}` or `{thinkingLevel}`, trigger `stop` or `/stop` before `onModelSelected()` runs, and routed abort delivery can normalize the reply without model/thinking context.

## Solution Assessment

No. Pre-seeding config defaults helps only paths that use `createReplyPrefixContext()` output; the routed abort path still calls `normalizeReplyPayload()` without a prefix context.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Pass prefix context into routed abort delivery:** `src/channels/reply-prefix.ts:51-57`
  - body: These seeded fields are only consumed by dispatcher paths that receive `createReplyPrefixContext()`'s context provider. Fast aborts first call `routeReplyToOriginating()`, and `routeReply()` normalizes with `responsePrefix` but no `responsePrefixContext`, so routed stop replies can still leak `{model}`/`{thinkingLevel}`. Plumb populated context into routed delivery and cover that path.
  - confidence: 0.9
