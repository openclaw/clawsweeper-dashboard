## Summary

Keep open. The reported bug is real and the PR has useful proof, but the patch is incomplete because routed fast-abort replies still normalize templated `responsePrefix` without any prefix context.

## What This Changes

The PR pre-seeds `createReplyPrefixContext()` with config default model/provider/thinking values and adds helper-level unit tests.

## Best Possible Solution

Carry a populated, runtime/session-aware `ResponsePrefixContext` through both direct dispatcher fallback and routed fast-abort delivery, with regression coverage for both paths.

## Reproduction Assessment

Yes, at source level. Configure `messages.responsePrefix` with `{model}` or `{thinkingLevel}`, trigger `stop` or `/stop` before `onModelSelected()` runs, and current main can normalize the abort reply without model/thinking context.

## Solution Assessment

No. Config-default pre-seeding improves the dispatcher fallback path, but routed abort delivery still lacks prefix context and the durable fix should carry runtime/session metadata into both delivery paths.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

This patch only seeds dispatcher-created contexts, so routed `/stop` replies can still leak `{model}` or `{thinkingLevel}`; carry a populated context into routed delivery and cover that path.
