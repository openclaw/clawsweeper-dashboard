## Summary

Keep open: this is a useful partial fix with sufficient WhatsApp before/after proof, but routed fast-abort delivery still normalizes templated prefixes without a response prefix context, so `{model}` and `{thinkingLevel}` can still leak in that path.

## What This Changes

The branch pre-seeds `createReplyPrefixContext()` with default model/provider/thinking values and adds unit tests for default seeding, runtime override, and no-config fallback.

## Best Possible Solution

Carry a populated, session/runtime-aware `ResponsePrefixContext` through both direct dispatcher delivery and routed abort delivery, with regression coverage for templated fast-abort prefixes in both paths.

## Reproduction Assessment

Yes, at source level. Configure `messages.responsePrefix` with `{model}` or `{thinkingLevel}`, start a run, then send `stop` or `/stop` before `onModelSelected()`; the current routed fast-abort path can normalize the abort reply with no prefix context.

## Solution Assessment

No. Default seeding is a useful partial fallback, but the narrow maintainable fix also needs to pass or build prefix context for `routeReply()` so routed abort replies cannot leak placeholders.
