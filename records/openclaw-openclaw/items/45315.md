## Summary

Keep open: the PR targets a real, source-reproducible reply-prefix bug and the screenshot proof is sufficient, but the branch is currently conflicting and the implementation seeds abort replies from config defaults instead of active model/session/channel state.

## What This Changes

The branch pre-seeds reply-prefix context with config default model/provider/thinking values and adds unit coverage for seeded defaults, runtime override, and empty-config fallback.

## Best Possible Solution

Rebase on current main and populate abort reply prefixes from canonical active model metadata first, with config defaults only as a fallback when no session, turn, channel, runtime, or catalog facts exist.

## Reproduction Assessment

Yes, at source level. Current main creates reply-prefix context without model/thinking data, fast abort sends before onModelSelected, and the resolver preserves unresolved placeholders.

## Solution Assessment

No, not as-is. Pre-seeding from config defaults fixes the simplest placeholder leak but bypasses active session, turn, channel, and catalog-backed model state that the existing model-selection path already uses.

- **[P2] Seed abort prefixes from active model state:** `src/channels/reply-prefix.ts:42`
