## Summary

Keep open. The PR is a useful partial fix and the inspected WhatsApp screenshots show the intended direct abort behavior, but routed abort delivery still normalizes a templated responsePrefix without any prefix context, so placeholders can still leak in routed replies.

## What This Changes

The branch pre-seeds response prefix context with configured default model/provider/thinking values and adds unit tests for default seeding, runtime override, and no-model fallback.

## Best Possible Solution

Carry a populated, session/runtime-aware `ResponsePrefixContext` through both direct dispatcher delivery and routed abort delivery, with regression tests for templated fast-abort prefixes in both paths.

## Reproduction Assessment

Yes, at source level. Configure `messages.responsePrefix` with `{model}` or `{thinkingLevel}`, start a run, then send `stop` or `/stop` before `onModelSelected()`; current main can normalize the abort reply with missing model/thinking context.

## Solution Assessment

No. Default seeding is a useful partial fallback, but the maintainable fix should also pass or build prefix context for `routeReply()` so routed abort replies cannot leak placeholders.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Pass prefix context into routed abort replies:** `src/channels/reply-prefix.ts:40-47`
  - body: This seeding only reaches dispatcher normalization. In fast abort, `dispatch-from-config` first calls `routeReplyToOriginating()`, and `routeReply()` resolves `responsePrefix` but normalizes without any `responsePrefixContext`, so routed `/stop` replies with `{model}` or `{thinkingLevel}` can still send literals. Please pass or build the prefix context for routed normalization and add a routed fast-abort regression test.
