## Summary

Keep open. The PR is a useful partial fix with sufficient WhatsApp before/after proof, but source review shows routed abort replies still normalize with `responsePrefix` and no `responsePrefixContext`, so templated `{model}` / `{thinkingLevel}` values can still leak in that delivery path.

## What This Changes

The PR pre-seeds `createReplyPrefixContext()` with configured default model, provider, and thinking level and adds tests for seeded defaults and runtime override behavior.

## Best Possible Solution

Carry a populated, session/runtime-aware `ResponsePrefixContext` through both direct dispatcher delivery and `routeReply()` normalization, with direct and routed fast-abort regression tests.

## Reproduction Assessment

Yes, at source level. Configure `messages.responsePrefix` with `{model}` or `{thinkingLevel}`, start a run, then send `stop` or `/stop`; current main can normalize an abort reply before `onModelSelected` fills the prefix context.

## Solution Assessment

No. Pre-seeding config defaults helps one path, but the PR does not pass prefix context into routed abort normalization, so the reported behavior can remain for routed replies.
