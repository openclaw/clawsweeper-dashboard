## Summary

Keep open. The PR is a useful focused bug fix and the WhatsApp before/after proof is sufficient, but the patch still misses routed abort delivery, so cross-provider/origin-routed stop replies can continue leaking `{model}` or `{thinkingLevel}` placeholders.

## What This Changes

The PR pre-seeds reply-prefix context from config defaults and adds focused tests for default seeding, runtime override, and empty-config fallback.

## Best Possible Solution

Carry a session/runtime-aware `ResponsePrefixContext` through both direct dispatcher delivery and `routeReply()` normalization, preserving `onModelSelected` overrides and covering direct plus routed fast-abort cases with regression tests.

## Reproduction Assessment

Yes, at source level. Configure `messages.responsePrefix` with `{model}` or `{thinkingLevel}`, start a run, then send `stop` or `/stop` before `onModelSelected`; current main can route the abort payload through normalization with a templated prefix and no prefix context.

## Solution Assessment

No. Pre-seeding config defaults helps the dispatcher fallback, but routed abort replies still normalize with `responsePrefix` and no `responsePrefixContext`, so the maintainable fix needs to pass context into `routeReply()` as well.
