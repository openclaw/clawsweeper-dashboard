## Summary

Keep open. The PR partially addresses the direct fast-abort placeholder leak and the inspected WhatsApp screenshots show the intended visible improvement, but the diff still leaves routed abort delivery normalizing a templated `responsePrefix` without `responsePrefixContext`, so routed `/stop` replies can still leak `{model}` or `{thinkingLevel}` literals.

## What This Changes

The PR pre-seeds reply-prefix context from config defaults in `createReplyPrefixContext()` and adds unit tests for default seeding, runtime override, and empty-config fallback.

## Best Possible Solution

Carry a populated, override-aware `ResponsePrefixContext` through both direct dispatcher delivery and routed abort normalization, with direct and routed fast-abort regression coverage.

## Reproduction Assessment

Yes, at source level. Configure `messages.responsePrefix` with `{model}` or `{thinkingLevel}`, trigger `stop` before `onModelSelected()`, and current main can normalize an abort reply with a templated prefix and no model/thinking context.

## Solution Assessment

No. Config-default pre-seeding is a useful fallback, but it is incomplete because routed abort delivery still normalizes without `responsePrefixContext` and override-aware metadata is not carried through.
