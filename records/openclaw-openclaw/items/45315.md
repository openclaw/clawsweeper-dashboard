## Summary

Keep open. The PR is a focused bug fix with useful WhatsApp before/after proof, but the patch only seeds the dispatcher prefix context and still leaves cross-provider routed abort replies normalizing without `responsePrefixContext`.

## What This Changes

The PR pre-seeds `createReplyPrefixContext()` with config-default model/provider/thinking values and adds unit coverage for default seeding and `onModelSelected()` overrides.

## Best Possible Solution

Carry a populated, session-aware `ResponsePrefixContext` through both dispatcher and routed reply normalization, with focused direct and routed fast-abort regression tests.

## Reproduction Assessment

Yes, at source level. Configure `messages.responsePrefix` with `{model}` or `{thinkingLevel}`, send `stop` before `onModelSelected()` runs, and the current fast-abort path can normalize an abort reply with missing prefix values.

## Solution Assessment

No. Pre-seeding config defaults is a useful direct-path mitigation, but the maintainable fix should also pass prefix context into `routeReply()` and cover the dispatch paths that actually emit abort replies.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Pass prefix context into routed abort delivery:** `src/channels/reply-prefix.ts:43-46`
  - body: This seeds the dispatcher-side prefix context, but fast aborts that go through `routeReplyToOriginating()` still enter `routeReply()`, which resolves `responsePrefix` and calls `normalizeReplyPayload()` without any `responsePrefixContext`. Cross-provider routed abort replies can therefore still send literal `{model}` / `{thinkingLevel}` tokens, so the fix needs to thread the context into routed normalization or normalize before routing.
  - confidence: 0.84
