## Summary

Keep open. Current main still has the abort response-prefix interpolation bug, and this PR is a useful partial fix, but seeding from static config defaults can report the wrong active model or thinking level for session/channel overrides and catalog-backed reasoning models; the open draft #45454 tracks a more complete runtime-metadata direction.

## What This Changes

The branch pre-seeds `createReplyPrefixContext()` with config-default provider, model, and thinking values and adds tests for default seeding, runtime override, and fallback behavior.

## Best Possible Solution

Land one canonical fix that formats abort response prefixes from the same effective runtime metadata used by normal replies, including stored session or parent-session model state and catalog-backed thinking defaults, with focused regression coverage for fast-abort templated replies.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure `messages.responsePrefix` with `{model}` or `{thinkingLevel}`, start a run, then send `/stop`; the fast-abort branch queues the abort reply before `onModelSelected()` fills the prefix context, so current main can leave those placeholders literal.

## Solution Assessment

No. The PR improves the config-default case, but the narrow maintainable fix is to use the effective runtime model/thinking metadata rather than guessing from static defaults before session, channel, and catalog resolution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Seed abort prefixes from the effective runtime model:** `src/channels/reply-prefix.ts:42`
  - body: This seeds fast-abort prefix values from `resolveDefaultModelForAgent()`, but normal runs can use stored session or parent-session overrides and channel model overrides. Because fast abort returns before `onModelSelected()` runs, users who changed models can receive a resolved prefix with the wrong model; thread or persist the effective runtime model metadata for abort replies instead of using only config defaults.
  - confidence: 0.87
- **[P2] Use catalog-backed thinking defaults before seeding:** `src/channels/reply-prefix.ts:43-47`
  - body: `resolveThinkingDefault()` is called without a catalog here, so reasoning-capable models with no explicit `thinkingDefault` can seed `off` while the normal selection path hydrates catalog metadata and may resolve a non-off default. Use the same catalog-backed or persisted runtime thinking value before formatting abort prefixes.
  - confidence: 0.82
