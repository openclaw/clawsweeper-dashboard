## Summary

Keep open. The reported bug is still source-reproducible on current main and the PR contains useful proof and a partial fix, but the patch is incomplete because routed abort replies still normalize a templated responsePrefix without a populated responsePrefixContext.

## What This Changes

The PR pre-seeds reply-prefix context with config default model/provider/thinking values and adds createReplyPrefixContext tests.

## Best Possible Solution

Build a session/runtime-aware ResponsePrefixContext and pass it through both direct dispatcher delivery and routeReply, with direct and routed fast-abort regression coverage.

## Reproduction Assessment

Yes, at source level. Configure messages.responsePrefix with {model} or {thinkingLevel}, trigger stop or /stop before onModelSelected runs, and current main can normalize the abort reply with missing model/thinking context.

## Solution Assessment

No. Config-default pre-seeding improves direct dispatcher abort replies, but the narrow maintainable fix also needs to carry populated prefix context into routed delivery and prefer runtime/session metadata where available.
