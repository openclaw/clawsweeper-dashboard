## Summary

Keep open: the reported bug and PR direction are valid, but the patch is incomplete because routed fast-abort delivery still normalizes templated prefixes without a ResponsePrefixContext. The linked bug remains open, and the broader alternate PR was closed without merge.

## What This Changes

The PR pre-seeds reply-prefix context with config-default provider/model/thinking values and adds unit coverage for seeded defaults plus runtime override.

## Best Possible Solution

Carry a populated runtime/session-aware ResponsePrefixContext into both dispatcher fallback and routeReply-routed fast-abort delivery, with regression coverage for the routed stop path.

## Reproduction Assessment

Yes, at source level. Configure messages.responsePrefix with {model} or {thinkingLevel}, trigger stop or /stop before onModelSelected runs, and routed abort delivery can normalize the prefix without model/thinking context.

## Solution Assessment

No. Pre-seeding config defaults helps the dispatcher fallback path, but the maintainable fix must also pass populated prefix context into routed delivery and should prefer runtime/session metadata when available.
