## Summary

Keep open. The PR is a useful partial fix, but routed fast-abort replies still go through `routeReply` without any response-prefix context, so `{model}` or `{thinkingLevel}` can still leak in visible abort replies.

## What This Changes

The PR pre-seeds reply-prefix context from config default model/provider/thinking values and adds unit tests for default seeding plus `onModelSelected` override behavior.

## Best Possible Solution

Carry a populated, session/runtime-aware `ResponsePrefixContext` through both dispatcher fallback and `routeReply` fast-abort delivery, with direct and routed regression tests.

## Reproduction Assessment

Yes, at source level. Configure `messages.responsePrefix` with `{model}` or `{thinkingLevel}`, trigger `stop` or `/stop` before `onModelSelected()` runs, and current main can normalize the abort reply without model/thinking context.

## Solution Assessment

No. Config-default pre-seeding improves the dispatcher fallback path, but routed abort delivery still lacks prefix context and the durable fix should carry runtime/session metadata into both delivery paths.
