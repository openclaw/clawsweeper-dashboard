Keep open: this is a focused source-level fix for the linked cron announce disclosure, and I found no definite correctness defect in the diff. It still needs after-fix real behavior proof before merge, and maintainers should consciously accept the failed-run delivery semantics change.

## What This Changes

The PR short-circuits cron announce delivery when an isolated run resolves to a fatal error payload and adds a regression test for that policy.

## Best Possible Solution

Suppress normal completion announce for fatal cron runs while preserving cron run status, diagnostics, verified message-tool accounting, and separate failure notifications.

## Reproduction Assessment

Yes, at source level: current `main` turns fatal `isError` cron payloads into `deliveryPayloads` and then calls `dispatchCronDelivery`, whose send path forwards non-empty payloads to the channel. I did not run a live Discord or Telegram provider-failure reproduction in this read-only review.

## Solution Assessment

Yes. The proposed boundary is the narrow maintainable fix: suppress normal completion announce after the existing fatal-state classification, while leaving run status and failure-notification machinery intact.
