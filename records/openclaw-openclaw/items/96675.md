## Summary

Keep open. Current main has adjacent gates for exec/plugin approvals, before-tool-call approval hooks, and Skill Workshop proposals, but it still lacks the requested owner-confirmed durable memory/evidence lifecycle; the remaining work is broad security/product design rather than cleanup.

## What This Changes

The issue requests optional owner-confirmation gates so assistant outputs, skill results, actions, memories, and reusable evidence cannot become trusted persistent context without explicit owner review.

## Best Possible Solution

Define a maintainer-approved responsibility/trust contract that reuses existing exec, plugin, and Skill Workshop gates, links the memory/source-routing slices to https://github.com/openclaw/openclaw/issues/7707, https://github.com/openclaw/openclaw/issues/87714, and https://github.com/openclaw/openclaw/issues/27445, and adds core/plugin API only where current seams cannot represent owner-confirmed memory or evidence state.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/security design request. Source inspection does give a high-confidence way to verify the gap: current memory contracts lack owner-confirmed/review/revocation state while existing action and skill gates cover only narrower surfaces.

## Solution Assessment

Unclear as a final implementation plan. The direction is plausible and aligned with safety goals, but the best solution needs maintainer product/security design to avoid duplicating existing approval hooks and to coordinate with the narrower memory, source-provenance, and sub-agent routing issues.
