## Summary

Keep open: the PR fixes the reported `0` and negative timeout case, but it still honors positive fractional values before flooring, so `timeoutMs: 0.5` becomes `0` and preserves the near-immediate timeout path.

## What This Changes

The PR changes gateway tool timeout normalization so non-positive `timeoutMs` overrides fall back to the 30s default and adds regression coverage for `0` and `-1`.

## Best Possible Solution

Normalize to a floored candidate first, only honor it when the floored value is at least `1`, and add a regression case for a positive fractional input below `1`.

## Reproduction Assessment

Yes, from source inspection: current `main` maps `timeoutMs: 0` to `1ms`, and the PR body reports a real WhatsApp smoke test where the patched behavior avoided the instant timeout. I did not run tests because this review is read-only.

## Solution Assessment

No, not yet: the direction is narrow and correct, but the guard must validate the floored timeout rather than the raw value. Otherwise positive fractions below `1` still produce a zero-ms override.
