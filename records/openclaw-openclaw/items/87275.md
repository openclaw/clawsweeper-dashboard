## Summary

Keep this PR open, but it is not merge-ready: the proposed change fixes only direct `resolveGatewayOptions` calls, while the reported model-facing message/gateway/cron paths still reject `timeoutMs: 0` before that fallback runs.

## What This Changes

The PR changes `resolveGatewayOptions` to treat finite non-positive `timeoutMs` values as absent and adds a direct helper regression test.

## Best Possible Solution

Normalize non-positive gateway timeouts at the shared gateway tool input boundary, update the schema/tests for that gateway-specific rule, and prove it through at least one real model-facing tool execution path.

## Reproduction Assessment

Yes, from source inspection rather than an executed test: current main rejects `timeoutMs: 0` in `readGatewayCallOptions` before message, gateway, or cron tool dispatch. Direct `resolveGatewayOptions({ timeoutMs: 0 })` still clamps to 1ms, but that is not the user-facing parser path.

## Solution Assessment

No. The PR changes a downstream helper only; the narrow maintainable fix is to put the zero-as-absent rule at the shared gateway option parsing boundary and cover a real tool execution path.
