## Summary

Keep open: current main still auto-inherits Slack thread context for same-channel message-tool sends, and the requested top-level same-channel opt-out is a real public API gap rather than already-implemented behavior.

## What This Changes

Request an explicit message-tool opt-out that lets Slack sends target the same parent channel as a new top-level post from inside a thread.

## Best Possible Solution

Add a documented, backward-compatible message-tool threading opt-out in the generic outbound contract and Slack adapter, with tests proving same-channel top-level delivery does not break default thread inheritance.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: invoke a Slack-threaded turn, send the message tool to the same channel without an explicit string threadId, and the resolver applies currentThreadTs. I did not run a live Slack workspace test.

## Solution Assessment

Unclear until maintainers choose the public API shape. The narrow maintainable solution is an explicit opt-out flag or sentinel handled before auto-thread resolution, not a Slack-only workaround after thread_ts has already been injected.
