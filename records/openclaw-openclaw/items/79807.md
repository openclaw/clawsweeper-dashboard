## Summary

Keep open: current main still auto-inherits Slack thread context for same-channel message-tool sends, and there is no documented or typed opt-out that lets an agent force a new top-level Slack post in that same parent channel.

## What This Changes

Request an explicit message-tool/Slack opt-out for sending a same-channel top-level message from inside an active Slack thread.

## Best Possible Solution

Add a documented, backward-compatible opt-out in the generic message-tool/outbound threading path and Slack adapter, with focused tests proving same-channel top-level delivery and unchanged default inheritance.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a same-channel Slack send from a threaded tool context resolves `currentThreadTs`, writes it as `threadId`, and dispatches it as Slack `threadTs`.

## Solution Assessment

Unclear until maintainers choose the public API shape. The maintainable direction is an explicit opt-out handled before auto-thread resolution, not a Slack-only cleanup after `threadTs` has already been injected.
