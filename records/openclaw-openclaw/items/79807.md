## Summary

Keep open: current main still auto-inherits the active Slack thread for same-channel message-tool sends, and no public opt-out exists for forcing a new top-level Slack post in that same channel.

## What This Changes

Request an explicit message-tool API to send a top-level Slack message to the current Slack channel from inside a thread without inheriting the thread timestamp.

## Best Possible Solution

Add a documented, backward-compatible message-tool opt-out handled before auto-thread resolution, with Slack and generic outbound tests proving same-channel top-level delivery and existing default thread inheritance.

## Reproduction Assessment

Yes, we have a high-confidence source-level reproduction: a same-channel Slack send from threaded context resolves currentThreadTs and applies it before dispatch. I did not run a live Slack workspace test.

## Solution Assessment

Unclear until maintainers choose the public API shape. The narrow maintainable direction is an explicit opt-out flag or sentinel handled before Slack auto-thread resolution, not a Slack-only workaround after thread_ts has already been injected.
