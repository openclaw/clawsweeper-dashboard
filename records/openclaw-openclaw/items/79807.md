## Summary

Keep open: current main still auto-inherits Slack thread context for same-channel message-tool sends, and the remaining work is a public opt-out contract rather than something already implemented or safely auto-closeable.

## What This Changes

Add a way for the shared Slack message tool to force a top-level same-channel Slack post from inside an existing Slack thread.

## Best Possible Solution

Add a documented, backward-compatible top-level opt-out in the generic outbound threading/message-tool contract, with Slack adapter behavior and regression tests preserving default same-thread inheritance.

## Reproduction Assessment

Yes from source inspection: a threaded Slack context produces `currentThreadTs`, a same-channel target resolves it as `threadId`, and Slack dispatch maps it to `thread_ts`. I did not run a live Slack workspace test.

## Solution Assessment

Unclear until maintainers choose the public API shape. The narrow maintainable direction is an explicit opt-out flag or sentinel handled before auto-thread resolution, not a Slack-only workaround after `thread_ts` has already been injected.
