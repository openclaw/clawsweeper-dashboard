## Summary

Keep open: current `main` still auto-inherits Slack thread context for same-channel message-tool sends from a threaded Slack turn, and no documented or tested API lets the agent force a new top-level same-channel Slack post.

## What This Changes

The issue requests an explicit message-tool opt-out for sending a top-level Slack message to the current parent channel while the agent is running inside a Slack thread.

## Best Possible Solution

Add a documented, backward-compatible message-tool opt-out that is consumed before generic auto-thread resolution, with focused Slack tests preserving default same-channel thread inheritance and proving explicit same-channel top-level delivery.

## Reproduction Assessment

Yes from source inspection: a threaded Slack context plus a same-channel `channel:C...` target resolves the current `threadTs` and forwards it into Slack dispatch. I did not run a live Slack workspace test.

## Solution Assessment

Unclear until maintainers choose the public API shape. The safer direction is a generic, documented opt-out flag or sentinel handled before Slack auto-thread resolution, not a Slack-only workaround after `thread_ts` has already been injected.
