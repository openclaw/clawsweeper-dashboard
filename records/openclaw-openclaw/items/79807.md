## Summary

Keep open: current main still auto-inherits Slack thread context for same-channel message-tool sends, and no documented API lets an agent force a new top-level Slack post from inside that thread.

## What This Changes

The issue asks for a Slack/message-tool opt-out that lets threaded agents send a new top-level message to the same Slack channel.

## Best Possible Solution

Add a documented, backward-compatible message-tool opt-out in the generic outbound threading path and Slack adapter, with focused Slack tests proving same-channel top-level delivery still leaves default thread inheritance intact.

## Reproduction Assessment

Yes from source inspection: a same-channel Slack send from a threaded context resolves and applies `currentThreadTs` before dispatch. I did not run a live Slack workspace test.

## Solution Assessment

Unclear until maintainers choose the public API shape. The narrow maintainable direction is an explicit opt-out flag or sentinel handled before auto-thread resolution, not a Slack-only workaround after `thread_ts` has already been injected.
