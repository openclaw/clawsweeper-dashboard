## Summary

Keep open: current main still auto-inherits the active Slack thread for same-channel message-tool sends, and solving the reported gap requires a new public opt-out API rather than maintainer cleanup.

## What This Changes

The issue requests a supported way for the shared message tool to send a top-level Slack message to the current parent channel when invoked from inside a Slack thread.

## Best Possible Solution

Add a documented, backward-compatible message-tool thread-inheritance opt-out in the generic outbound threading path, plumb it through Slack, and cover same-channel top-level sends while preserving default thread inheritance.

## Reproduction Assessment

Yes, from source inspection: a threaded Slack invocation targeting the same channel resolves `currentThreadTs`, injects it as `threadId`, and passes it to Slack as `threadTs`. I did not run a live Slack workspace test.

## Solution Assessment

Unclear until maintainers choose the public API shape. The maintainable direction is a generic explicit opt-out handled before auto-thread resolution, not a Slack-only workaround after `threadTs` has already been injected.
