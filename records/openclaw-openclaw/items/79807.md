## Summary

Keep open: current main still auto-inherits the active Slack thread for same-channel message-tool sends, and adding a top-level opt-out needs a maintained public API choice rather than cleanup closure.

## What This Changes

The issue requests an explicit way for the shared message tool to send a new top-level Slack message to the current parent channel from inside a Slack thread.

## Best Possible Solution

Add a documented, backward-compatible message-tool threading opt-out in the generic outbound path, then map it in Slack before auto-thread resolution while preserving default same-channel thread inheritance.

## Reproduction Assessment

Yes. Source inspection shows a threaded Slack context plus same-channel target resolves and applies the active `currentThreadTs` before Slack dispatch; I did not run a live Slack workspace test.

## Solution Assessment

Unclear until maintainers choose the public API shape. The narrow maintainable direction is an explicit opt-out handled before auto-thread resolution, not `reply_broadcast` or a Slack-only workaround after `thread_ts` is injected.
