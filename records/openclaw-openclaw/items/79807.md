## Summary

Keep open: current main still injects Slack thread context for same-channel message-tool sends from inside a Slack thread, and the remaining work is a public opt-out API decision rather than a duplicate close.

## What This Changes

Request an explicit message-tool way to send a new top-level Slack message to the current parent channel from a threaded Slack context.

## Best Possible Solution

Add a documented, backward-compatible message-tool opt-out in shared outbound threading, with the Slack adapter honoring it before auto-thread resolution and tests preserving default thread inheritance.

## Reproduction Assessment

Yes from source inspection: a threaded Slack context with MessageThreadId flows into resolveSlackAutoThreadId for same-channel targets and then into Slack threadTs. I did not run a live Slack workspace test.

## Solution Assessment

Unclear until maintainers choose the public API shape. The narrow maintainable direction is an explicit shared message-tool opt-out handled before Slack auto-thread resolution, not a Slack-only workaround after threadTs is injected.
