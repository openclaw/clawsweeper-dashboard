## Summary

Keep open: current main still auto-inherits Slack thread context for same-channel message-tool sends, and the remaining work is a public message-tool API opt-out rather than an already-implemented fix.

## What This Changes

The issue requests a way for Slack message-tool sends from inside a thread to post a new top-level message to the same parent channel.

## Best Possible Solution

Add a documented, backward-compatible message-tool opt-out handled before generic outbound auto-thread resolution, with Slack tests proving same-channel top-level delivery and default thread inheritance.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a Slack threaded turn builds `currentThreadTs`, same-channel message-tool sends resolve it through `resolveSlackAutoThreadId`, and Slack dispatch passes it as `thread_ts`.

## Solution Assessment

Unclear until maintainers choose the API shape. The maintainable direction is a generic message-tool opt-out or sentinel before auto-thread resolution, not a Slack-only workaround after `thread_ts` is injected.
