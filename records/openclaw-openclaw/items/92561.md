## Label Justifications

- `P1`: The issue is a high-confidence prompt-context security boundary problem in agent runtime behavior with an open but unmerged fix path.
- `impact:security`: Untrusted `AGENTS.md` or `CLAUDE.md` files outside the workspace can be injected into system prompt context.
- `impact:session-state`: The affected loader changes which instruction context is attached to agent sessions.

## Summary

Keep open: current main and the latest release still walk project context files up to the filesystem root, and the open linked fix PR has not merged yet.

## What This Changes

The issue requests bounding `loadProjectContextFiles()` ancestor discovery for `AGENTS.md`/`CLAUDE.md` to the trusted workspace root instead of the filesystem root.

## Best Possible Solution

Land a workspaceDir-bound context-loader fix with regression coverage for nested cwd loading, above-workspace exclusion, and existing no-context/embedded isolation paths.

## Reproduction Assessment

Yes. Source inspection shows `loadProjectContextFiles()` walks from `cwd` to `/`, reads matching context files, and the system-prompt builder injects those files as project instructions.

## Solution Assessment

Yes. Bounding the walk at the trusted `workspaceDir` is the narrow maintainable fix; the earlier home-directory and cwd-only variants are less correct because they either still allow parents above workspace or drop workspace-root context for nested cwd runs.
