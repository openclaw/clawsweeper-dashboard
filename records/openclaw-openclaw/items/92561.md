## Label Justifications

- `P1`: This is a high-confidence agent prompt-injection boundary issue affecting real shared-host or same-user multi-agent deployments.
- `impact:security`: The report concerns untrusted instruction files outside the workspace crossing into model-visible system prompt context.
- `impact:session-state`: The affected surface is automatic project-context injection into agent session prompts.

## Summary

Keep open: current `main` and the latest release still let `loadProjectContextFiles()` walk from `cwd` to the filesystem root, and the loaded files are passed into system prompt construction. The linked fix attempts are closed unmerged, and the repeated home-directory boundary proposal does not fully satisfy the workspace-boundary security report.

## What This Changes

Request to bound `loadProjectContextFiles()` ancestor discovery so `AGENTS.md` and `CLAUDE.md` files outside the workspace cannot be injected into agent prompts.

## Best Possible Solution

Use the actual workspace or project-root boundary for ancestor context discovery, preserve nested in-workspace context files, exclude parent/home/system files, and add focused regression coverage for those cases.

## Reproduction Assessment

Yes from source inspection: with `cwd` inside a workspace, the loop loads every ancestor `AGENTS.md` or `CLAUDE.md` until `/`. I did not run a live agent session in this read-only review.

## Solution Assessment

The requested workspace-boundary direction is right, but the exact owner boundary should be decided before implementation. The prior home-directory PR shape is not the best fix because it still allows `~/AGENTS.md` for workspaces below home and can break workspaces outside home.
