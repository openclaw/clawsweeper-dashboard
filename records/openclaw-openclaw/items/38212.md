## Summary

Keep open. Current main still leaves a credible raw `codex-cli/gpt-5.4` helper failure path: the session-memory slug helper forwards the configured provider/model into the embedded runner unchanged, and model resolution is still provider-scoped before auth alias handling can help. Adjacent auth/runtime hardening and doctor migration exist, but they do not prove that an already-loaded raw helper run is normalized or explicitly rejected before `Unknown model` can surface.

## What This Changes

The issue asks for raw `codex-cli/gpt-5.4` embedded/helper runs, especially session-memory slug generation, auth lookup, and xhigh thinking support, to behave consistently with `openai-codex/gpt-5.4`.

## Best Possible Solution

Maintainers should make the boundary explicit: either normalize legacy CLI-backed helper runs before embedded model resolution with focused regression coverage, or clearly reject/repair raw `codex-cli/*` primary refs for embedded helpers through doctor/config validation and docs so the session-memory hook cannot fail with a vague unknown-model error.
