## Summary

Keep open: current main and the latest shipped release still leave the raw `codex-cli/gpt-5.4` helper path governed by an unsettled runtime contract, while docs define `codex-cli/*` as doctor-repaired legacy config rather than a guaranteed runtime ref.

## What This Changes

Request that raw legacy `codex-cli/gpt-5.4` defaults work in embedded/helper paths such as session-memory slug generation with the same Codex model/auth handling as `openai-codex`.

## Best Possible Solution

Pick one contract for legacy `codex-cli/*` and implement it consistently: either helper/runtime normalization with regression coverage, or explicit doctor-only diagnostics that prevent the raw helper failure.

## Reproduction Assessment

Yes, source-reproducible: current main still lets the slug helper pass raw `codex-cli/gpt-5.4` into `runEmbeddedPiAgent`, whose model-resolution path can throw `model_not_found` for unresolved refs. I did not run a live `/new` or `/reset` helper because this review is read-only.

## Solution Assessment

Unclear. Direct normalization would address the symptom, but current docs make `codex-cli/*` a doctor-repaired legacy contract, so maintainers should decide that runtime policy before implementation.
