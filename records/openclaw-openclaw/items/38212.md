## Summary

Keep open: current main still has a source-reproducible raw `codex-cli/gpt-5.4` helper path, but current docs and doctor tests now define `codex-cli/*` as legacy config repaired to `openai/*` plus the Codex runtime. This needs a maintainer runtime-contract decision rather than an automatic compatibility patch or stale close.

## What This Changes

The issue asks for embedded/helper paths using `codex-cli/gpt-5.4` to resolve and authenticate consistently with the equivalent Codex-backed `openai-codex/gpt-5.4` path.

## Best Possible Solution

Pick and document one contract for legacy `codex-cli/*`: either confirm doctor-only repair and improve diagnostics, or add a narrow helper/runtime normalization with regression coverage.

## Reproduction Assessment

Yes, source-reproducible. A primary model of `codex-cli/gpt-5.4` still flows from the slug helper into provider-scoped embedded model resolution, where no raw `codex-cli` model owner is guaranteed before `model_not_found`.

## Solution Assessment

Unclear. Direct compatibility normalization would fix the symptom, but current docs and doctor tests prefer canonical `openai/*` refs plus `agentRuntime.id: "codex"`, so a maintainer should choose the contract first.
