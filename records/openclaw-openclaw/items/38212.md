## Summary

Keep open: current main still has a source-reproducible raw `codex-cli/gpt-5.4` helper-path gap. Auth alias and doctor migration support exist, but the slug helper still forwards the configured provider into provider-scoped embedded model resolution, and the remaining choice is whether to support raw legacy refs at runtime or require canonical migration.

## What This Changes

The issue asks for `codex-cli/*` helper and embedded model resolution, auth lookup, and xhigh thinking support to behave consistently with `openai-codex/*`.

## Best Possible Solution

Maintainers should settle the legacy `codex-cli/*` contract, then implement one shared compatibility or migration boundary with regression coverage instead of a slug-helper-only alias.

## Reproduction Assessment

Yes. Source inspection shows `agents.defaults.model.primary = codex-cli/gpt-5.4` can still reach `generateSlugViaLLM`, pass `codex-cli` into `runEmbeddedPiAgent`, and fail provider-scoped embedded model resolution before any auth alias can help.

## Solution Assessment

Unclear. Normalizing legacy refs before embedded helper runs would be narrow, but current docs and doctor logic prefer migration to canonical provider refs plus runtime policy, so this needs maintainer direction before an automated fix.
