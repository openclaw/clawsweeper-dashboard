## Summary

Keep open: current main still has a source-reproducible raw `codex-cli/gpt-5.4` helper-path gap. Auth aliasing and doctor migration now cover part of the report, but the slug helper still forwards the legacy provider into provider-scoped embedded model resolution, so maintainers need to decide whether raw legacy refs remain runtime-compatible or are doctor-only migration input.

## What This Changes

The issue asks for `codex-cli/gpt-5.4` embedded/helper runs to resolve, authenticate, and expose thinking support consistently with `openai-codex/gpt-5.4`.

## Best Possible Solution

Choose one shared boundary for legacy `codex-cli/*` refs: either canonicalize before helper model resolution or enforce doctor-only migration with clear diagnostics, then cover that contract with focused regression tests.

## Reproduction Assessment

Yes. Source inspection on current main shows `agents.defaults.model.primary = codex-cli/gpt-5.4` can still reach `generateSlugViaLLM`, pass `codex-cli` into `runEmbeddedPiAgent`, and fail provider-scoped model resolution before the auth alias path helps.

## Solution Assessment

Unclear. A runtime alias before helper resolution would be narrow, but current docs and doctor logic prefer canonical provider refs plus runtime policy, so maintainers should decide the compatibility contract first.
