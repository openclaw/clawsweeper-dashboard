## Summary

Keep open: current main still leaves the reported slug/helper path source-reproducible for raw `codex-cli/gpt-5.4`. The adjacent auth-alias and RuntimePlan work reduced nearby failures, but the helper still reaches embedded provider-scoped model resolution before a legacy CLI compatibility decision can help.

## What This Changes

The issue requests that raw `codex-cli/gpt-5.4` helper and embedded paths resolve and authenticate consistently with the OpenAI Codex-backed model family.

## Best Possible Solution

Choose and document one contract for legacy CLI refs in embedded helper paths, then add focused slug/helper regression coverage for either normalization before embedded resolution or a clear doctor-guided rejection path.

## Reproduction Assessment

Yes, source-reproducible. A config primary of `codex-cli/gpt-5.4` still flows from `generateSlugViaLLM` into provider-scoped `runEmbeddedPiAgent` model resolution, where missing raw `codex-cli` model ownership can produce `model_not_found`.

## Solution Assessment

Unclear. A small compatibility normalization could fix the symptom, but current docs and doctor migration prefer canonical refs plus runtime policy, so the narrowest maintainable solution needs a maintainer contract decision first.
