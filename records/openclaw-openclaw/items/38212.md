## Summary

Keep open: current main now treats legacy `codex-cli/*` as doctor-repaired config, but a raw `codex-cli/gpt-5.4` default can still flow into the helper embedded path without a guaranteed model/runtime owner. The remaining question is the runtime contract, not stale cleanup.

## What This Changes

The issue asks for `codex-cli/gpt-5.4` helper and embedded runs to resolve and authenticate like the equivalent `openai-codex/gpt-5.4` path, especially for session-memory slug generation.

## Best Possible Solution

Pick and document one contract for legacy `codex-cli/*`: either keep doctor-only migration and improve diagnostics, or add a narrow helper/runtime normalization with regression coverage.

## Reproduction Assessment

Yes, source-reproducible. A configured primary model of `codex-cli/gpt-5.4` still flows from the slug helper into embedded provider/model resolution, where current main does not guarantee a raw `codex-cli` model or harness owner before `model_not_found`.

## Solution Assessment

Unclear. Direct compatibility normalization would address the symptom, but current docs and doctor tests prefer canonical `openai/*` refs plus Codex runtime policy, so maintainers should choose the contract first.
