## Summary

Keep open: current main still allows the raw `codex-cli/gpt-5.4` slug-helper path to reach embedded model resolution, but current docs and doctor code now treat `codex-cli/*` as legacy config that should be migrated instead of a guaranteed runtime ref.

## What This Changes

The issue asks for `codex-cli/gpt-5.4` helper and embedded runs to resolve models, auth profiles, and thinking-level support consistently with the Codex-backed `openai-codex` path.

## Best Possible Solution

Pick and document one legacy `codex-cli/*` contract, then either normalize helper/runtime paths consistently or make doctor-only migration diagnostics prevent the raw helper failure.

## Reproduction Assessment

Yes. Source inspection shows the slug helper can pass raw `codex-cli/gpt-5.4` into `runEmbeddedPiAgent`, and that path still throws `model_not_found` when the selected provider/model cannot resolve.

## Solution Assessment

Unclear. Direct runtime normalization would address the symptom, but current docs and doctor code define `codex-cli/*` as legacy migration state, so maintainers should decide the intended contract first.
