## Summary

Keep open. Current main still has a credible raw `codex-cli/gpt-5.4` session-memory slug helper failure path: the helper forwards the configured provider/model unchanged into `runEmbeddedPiAgent`, and the embedded runner still performs provider-scoped model resolution before throwing `model_not_found`. The related fix PRs were closed unmerged, and the merged GPT-5.4 hardening work appears adjacent rather than proof that this exact raw helper-resolution path is fixed.

## What This Changes

The issue requests that `codex-cli/gpt-5.4` helper and embedded runs resolve and authenticate consistently with the Codex OAuth-backed `openai-codex/gpt-5.4` path.

## Best Possible Solution

Define one explicit compatibility contract for legacy `codex-cli/*` helper runs and prove it with regression coverage. The maintainable end state is either a shared normalization/migration boundary before session-memory invokes embedded model resolution, or a clear validation/doctor/docs path that prevents raw legacy refs from reaching helper hooks as runnable PI model refs.

## Reproduction Assessment

Yes, for a static reproduction path: configure `agents.defaults.model.primary` as `codex-cli/gpt-5.4`, let the session-memory slug hook call `generateSlugViaLLM`, and the helper forwards `codex-cli` into provider-scoped embedded model resolution. I did not run a live Gateway turn under the read-only constraint.

## Solution Assessment

Unclear because there is no active patch and the architecture now distinguishes provider refs from CLI runtime refs. A narrow fix should use a shared compatibility boundary or explicit validation rather than a slug-helper-only alias that could drift from doctor/runtime policy.
