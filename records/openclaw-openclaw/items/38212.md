## Summary

Keep open: current main has improved Codex routing and auth aliasing, but the exact raw `codex-cli/gpt-5.4` helper path still depends on an unresolved product contract between runtime compatibility and doctor-only legacy migration.

## What This Changes

The issue asks OpenClaw to make legacy `codex-cli/gpt-5.4` helper and embedded runs resolve, authenticate, and expose thinking support consistently with `openai-codex/gpt-5.4`.

## Best Possible Solution

Maintainers should choose and document one legacy `codex-cli/*` contract, then either normalize helper/runtime paths consistently or make doctor/preflight diagnostics prevent the raw helper failure before execution.

## Reproduction Assessment

Yes, at source level. The slug helper still forwards the configured provider/model into the embedded runner, and that runner still throws `model_not_found` when the selected provider/model cannot resolve.

## Solution Assessment

Unclear. Direct runtime normalization would address the symptom, but current docs and doctor migration code define `codex-cli/*` as legacy state, so maintainers should decide the intended contract before implementation.
