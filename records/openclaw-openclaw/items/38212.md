## Summary

Keep open because current main still has the raw legacy `codex-cli/gpt-5.4` helper path described in the report: the session-memory slug helper forwards the configured provider/model into embedded PI model resolution, and the current resolver still treats `codex-cli` as its own provider rather than the `openai-codex` model provider.

## What This Changes

The issue reports that legacy `codex-cli/gpt-5.4` primary-model refs can fail in embedded helper paths such as session-memory slug generation while `openai-codex/gpt-5.4` works.

## Best Possible Solution

Define one compatibility boundary for legacy `codex-cli/*` helper runs and cover it with regression tests, either by normalizing before embedded model resolution or by rejecting/migrating raw legacy refs before hooks run.

## Reproduction Assessment

Yes, from source inspection: configure `agents.defaults.model.primary` as `codex-cli/gpt-5.4`, let session-memory call `generateSlugViaLLM`, and the helper forwards `codex-cli` into provider-scoped embedded model resolution. I did not run a live Gateway turn under the read-only constraint.

## Solution Assessment

Unclear because there is no active patch and the current architecture separates model provider refs from CLI runtime refs. A maintainable fix should use a shared legacy-runtime compatibility or validation boundary rather than a slug-helper-only alias.
