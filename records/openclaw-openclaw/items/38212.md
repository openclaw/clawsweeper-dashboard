## Summary

Keep open: current main still has a source-reproducible `codex-cli/gpt-5.4` helper-path gap. The auth-alias work is present, but the session-memory slug helper still forwards the raw `codex-cli` provider into provider-scoped embedded model resolution, and current docs leave a maintainer decision between runtime compatibility and legacy-ref migration.

## What This Changes

The issue asks for legacy `codex-cli/gpt-5.4` helper and embedded runs to resolve, authenticate, and expose thinking levels like the `openai-codex/gpt-5.4` path.

## Best Possible Solution

Maintainers should settle the legacy `codex-cli/*` contract, then implement one shared compatibility or migration boundary with regression coverage instead of a slug-helper-only alias.

## Reproduction Assessment

Yes. From source inspection, configuring `agents.defaults.model.primary` as `codex-cli/gpt-5.4` makes `generateSlugViaLLM` pass `codex-cli` into `runEmbeddedPiAgent`, where provider-scoped model resolution can still return `model_not_found`.

## Solution Assessment

Unclear. Normalizing raw legacy refs before embedded helper runs would be narrow, but current docs also define `codex-cli/*` as legacy runtime syntax that should migrate to canonical provider refs, so maintainer policy should decide the boundary.
