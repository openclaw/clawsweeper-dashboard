## Summary

Keep open: current main still has a source-reproducible raw `codex-cli/gpt-5.4` helper path, while current docs and doctor coverage define `codex-cli/*` as legacy config that should be repaired to canonical `openai/*` plus the Codex runtime. The related GPT-5.4 hardening work improved auth/runtime parity, but it did not settle whether raw legacy refs should keep working at runtime or only through `openclaw doctor --fix`.

## What This Changes

The issue asks for legacy `codex-cli/gpt-5.4` helper and embedded runs to resolve and authenticate like the Codex-backed `openai-codex/gpt-5.4` path.

## Best Possible Solution

Pick and document one contract for legacy `codex-cli/*`: either confirm doctor-only repair as intended and close with that rationale, or add a narrow helper/runtime normalization or diagnostic with regression coverage.

## Reproduction Assessment

Yes, source-reproducible. A primary model of `codex-cli/gpt-5.4` still flows from the slug helper into provider-scoped embedded model resolution, where raw `codex-cli` is not guaranteed a model or harness owner before `model_not_found`.

## Solution Assessment

Unclear. Direct runtime normalization would fix the reported symptom, but current docs and doctor tests prefer canonical `openai/*` refs plus `agentRuntime.id: "codex"`, so a maintainer should choose the contract first.
