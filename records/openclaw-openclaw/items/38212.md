## Summary

Keep open: current main still leaves the reported helper path source-reproducible for raw `codex-cli/gpt-5.4`. Recent RuntimePlan, auth-alias, and doctor work cover adjacent pieces, but the slug helper still enters embedded model resolution before any compatibility decision can help.

## What This Changes

The issue asks OpenClaw to make helper and embedded runs treat `codex-cli/gpt-5.4` consistently with the Codex-backed `openai-codex/gpt-5.4` path.

## Best Possible Solution

Choose one contract for CLI-runtime helper paths, then encode it with focused tests: either normalize before embedded helper resolution or reject legacy/raw CLI refs with clear doctor-guided diagnostics.

## Reproduction Assessment

Yes, source-reproducible. A config primary of `codex-cli/gpt-5.4` still flows from `generateSlugViaLLM` into provider-scoped `runEmbeddedPiAgent` model resolution, where missing `codex-cli` model ownership can produce `model_not_found`.

## Solution Assessment

Unclear. A small normalization before helper model resolution could fix the reported path, but current docs and doctor migration prefer canonical refs plus explicit runtime policy, so the maintainable fix needs a maintainer contract decision first.
