- `P1`: The PR changes a live Feishu channel workflow where bot-to-bot replies can be silently dropped, newly delivered, or wrongly broadened depending on mention handling.
- `merge-risk: 🚨 compatibility`: The diff adds Feishu config/default behavior that changes how existing bot-authored Feishu webhook traffic is admitted or suppressed during upgrade.

## Summary

Keep open: this is useful Feishu bot-to-bot work with solid live proof, but the branch is not merge-ready because the new mention registry can learn broadcast mentions and later turn natural `@所有人` text into a real `@all` notification; GitHub also reports the PR conflicts with current base.

## What This Changes

The PR adds Feishu bot-authored inbound handling, bot sender mention backstops across text/card/chunked/streaming replies, outbound mention normalization, bot-name lookup, `allowBots` and `botLoopProtection` config/docs, and Feishu regression tests.

## Best Possible Solution

Filter Feishu broadcast mentions out of the mention registry, add focused regression coverage for that invariant, then rebase and have a Feishu owner explicitly accept the bot-admission default and live delivery tradeoffs.

## Reproduction Assessment

Yes for the review finding by source inspection: a Feishu broadcast mention can pass through the new registry seeding loop, and later registry-based `@Name` normalization can emit a real broadcast tag. I did not run a live Feishu repro.

## Solution Assessment

No, not yet: the Feishu-owned implementation is the right boundary, but it must preserve the existing no-cascade invariant by excluding broadcast mentions from registry learning before maintainers accept the defaults.
