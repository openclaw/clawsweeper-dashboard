## Summary

Keep open: the npm-global fix is pointed at the right bug, but this branch would misclassify supported pnpm global/package roots that carry the published `npm-shrinkwrap.json` and keep their pnpm lock/marker files outside the package root.

## What This Changes

The PR changes package-manager detection to prefer `npm-shrinkwrap.json` over `package.json.packageManager` when no same-root `pnpm-lock.yaml` exists, adds an npm shrinkwrap dependency-status fallback, and adds focused tests.

## Best Possible Solution

Keep the shrinkwrap npm fix, but first preserve pnpm-owned package roots by checking the owning node_modules/layout markers before falling back to npm for `npm-shrinkwrap.json`.

## Reproduction Assessment

Yes, from source inspection and reporter logs: current main reads `packageManager` before npm shrinkwrap evidence for the original npm bug, and this PR's new branch would classify pnpm-owned package roots with out-of-root pnpm markers as npm.

## Solution Assessment

No, this is only a partial fix. The narrow maintainable solution is to treat shrinkwrap as npm install evidence after excluding supported pnpm-owned package layouts, then keep the npm dependency-status fallback.
