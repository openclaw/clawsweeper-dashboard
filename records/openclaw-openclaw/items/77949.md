## Summary

Keep this PR open: the direction is useful, but the current patch can still hide subsequent immediate tool-start lines, can double-render native Codex tool items, lacks required real behavior proof, and is missing a user-facing changelog entry.

## What This Changes

The PR adds normalized Codex `stream: "tool"` events for dynamic/native tool activity with redacted payloads and makes Telegram progress drafts start explicit tool progress before delayed progress/status paths.

## Best Possible Solution

Keep the PR open until progress drafts render every immediate tool-start update once, native Codex tool progress is coalesced with existing item progress, `CHANGELOG.md` is updated, and real Telegram/Codex proof is attached.

## Reproduction Assessment

Yes for source-level reproduction of the PR regressions: the `startImmediately` branch returns after a no-op `startNow()` on already-started gates, and native Codex item starts now emit both item and tool streams consumed by Telegram. No live Telegram run was established.

## Solution Assessment

No: normalizing Codex tool progress is a reasonable fix direction, but the patch should render later immediate starts, avoid duplicate native item/tool draft lines, add the required changelog entry, and include real behavior proof before merge.
