## Summary

Keep open: current main still matches the reported Feishu Docker/package runtime regression. The Feishu plugin is ESM, it value-imports the Lark SDK, and the unified build externalization list still omits `@larksuiteoapi/node-sdk`, whose published ESM bundle contains a bare `__dirname` path.

## What This Changes

The issue reports a release/2026.5.2 Docker runtime regression where the Feishu plugin exits because bundled ESM code reaches `__dirname`.

## Best Possible Solution

Externalize `@larksuiteoapi/node-sdk` or otherwise preserve a compatible module boundary for Feishu package builds, then add regression coverage for the built Docker/package runtime path.

## Reproduction Assessment

Yes. The issue body gives the exact Docker gateway exception, and current main still has the Feishu ESM package, runtime Lark SDK imports, and build configuration that can bundle the SDK into the ESM plugin output.

## Solution Assessment

Yes. The narrowest maintainable fix is a build/package change for Feishu's plugin-local runtime dependency, with focused tests proving the built output no longer exposes the SDK's ESM `__dirname` trap.
