## Summary

Keep open. Current main still counts all session `.jsonl` files for the `openclaw memory status` denominator, while the session corpus used by indexing/export already excludes compaction checkpoint snapshots, so the reported mismatch remains valid.

## What This Changes

The issue requests that `openclaw memory status` exclude compaction checkpoint session snapshots from displayed session file totals.

## Best Possible Solution

Make the memory status session scan use the same usage-counted transcript predicate as indexing/export, with focused CLI status regression coverage and a user-facing changelog fix.

## Reproduction Assessment

Yes. Create one normal session `.jsonl` and one `<session>.checkpoint.<uuid>.jsonl` in an agent sessions directory; current source inspection shows status counts both in the denominator while the indexer lists only the usage-counted transcript.

## Solution Assessment

Yes. Reusing the existing session artifact predicate is the narrow maintainable fix; indexing checkpoint snapshots would broaden the memory corpus and conflict with the current checkpoint design.

## Repair Work Prompt

Fix the memory status file-count mismatch from https://github.com/openclaw/openclaw/issues/69759. Current `extensions/memory-core/src/cli.runtime.ts` `scanSessionFiles` counts every regular `.jsonl` in the agent sessions directory, while memory indexing/export uses `isUsageCountedSessionTranscriptFileName` and excludes compaction checkpoint snapshots. Update the status scan to count the same usage-counted session corpus, preserving plugin boundary rules by importing/exporting the predicate through the existing memory-core host runtime/SDK facade as needed; do not deep-import core internals from extension production code and do not index checkpoint snapshots. Add focused CLI status coverage with one normal transcript plus one `<session>.checkpoint.<uuid>.jsonl` snapshot so the `Indexed` and sessions `By source` denominators stay consistent. Add a single-line `CHANGELOG.md` fix entry credited to @xinzf. Related PR #65914 is adjacent QMD timeout/export coverage and should not be treated as resolving this issue unless it is updated to cover this exact CLI denominator.
