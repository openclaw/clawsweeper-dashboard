## Summary

Keep open. Current main still counts every session `.jsonl` file for the `openclaw memory status` denominator, while the indexing/export corpus uses the shared usage-counted transcript predicate that excludes compaction checkpoints, so the reported `Indexed: 3/4 files` mismatch remains valid.

## What This Changes

Make `openclaw memory status` report file totals from the same indexable session transcript corpus used by memory indexing, excluding compaction checkpoint snapshots from the displayed denominator.

## Best Possible Solution

Align the memory status session scan with the usage-counted transcript corpus already used by indexing/export, add focused CLI status coverage for a normal transcript plus a checkpoint snapshot, and include a user-facing changelog fix credited to @xinzf.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: put one normal session `.jsonl` and one `<session>.checkpoint.<uuid>.jsonl` in the agent sessions directory; current `scanSessionFiles` counts both while the indexer lists only the usage-counted transcript.

## Solution Assessment

Yes. The narrow maintainable fix is to reuse the existing session artifact predicate for status totals; indexing checkpoint snapshots would broaden the searchable corpus and does not match the current checkpoint design.

Fix the memory status file-count mismatch reported in https://github.com/openclaw/openclaw/issues/69759. Current `extensions/memory-core/src/cli.runtime.ts` counts all session files ending in `.jsonl` for `openclaw memory status`, but the memory indexing/export paths use `isUsageCountedSessionTranscriptFileName` and exclude `<session>.checkpoint.<uuid>.jsonl` compaction snapshots. Add focused CLI status coverage showing a normal transcript plus checkpoint snapshot reports the indexable denominator. Add a user-facing `CHANGELOG.md` fix entry crediting @xinzf. Keep this separate from PR #65914's QMD timeout/export work.
