## Summary

Current main still matches the reported failure path: the auto-update child only adds `OPENCLAW_AUTO_UPDATE`, while service markers from the gateway environment are merged into the spawned process and trip the package-update guard. The issue should stay open because https://github.com/openclaw/openclaw/pull/78494 is an open closing PR for the same bug, not a merged fix.

## What This Changes

The issue asks for the gateway auto-updater to strip `OPENCLAW_SERVICE_MARKER` and `OPENCLAW_SERVICE_KIND` when it spawns `openclaw update --yes`.

## Best Possible Solution

Land a narrow updater fix that unsets the gateway service markers only for the auto-update child process, with regression coverage, while preserving the manual in-service package-update guard.

## Reproduction Assessment

Yes, from source inspection: a launchd-managed gateway receives the service marker env, `runAutoUpdateCommand` does not unset it, `resolveCommandEnv` preserves omitted env keys, and the update guard blocks marked gateway-service envs. I did not perform a live macOS/npm update run.

## Solution Assessment

Yes, the narrowest maintainable fix is to strip `OPENCLAW_SERVICE_MARKER` and `OPENCLAW_SERVICE_KIND` in the auto-update spawn env while keeping `OPENCLAW_AUTO_UPDATE=1`; that matches existing env-merge semantics and the sibling marker-stripping helper used after core updates.
