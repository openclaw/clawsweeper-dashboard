## Summary

Current main still has a source-backed failure path: the gateway auto-update child is spawned from the service context with only `OPENCLAW_AUTO_UPDATE` added, while the package-update service guards still block gateway-service or gateway-ancestry execution. The issue should stay open while the linked closing PR is reviewed or replaced.

## What This Changes

The issue asks for gateway auto-update subprocesses to avoid inheriting service-marker state that causes package updates to be refused from managed gateway services.

## Best Possible Solution

Update or replace the linked fix so auto-update uses a service-safe package-update handoff that clears inherited gateway service state without weakening the manual in-service update guard, then cover both env-marker and managed-service preflight behavior with regression tests and redacted real-service proof.

## Reproduction Assessment

Yes. Source inspection provides a high-confidence path: a managed gateway receives service markers, `runAutoUpdateCommand` spawns the update child from that environment, `resolveCommandEnv` preserves inherited keys, and the update command blocks gateway-service or gateway-ancestry package updates.

## Solution Assessment

No for the narrow `undefined`-override proposal as written. The reported direction is right, but current main needs a fix that actually removes inherited marker keys and accounts for the managed-service ancestry preflight before claiming the auto-update child can reach package installation.
