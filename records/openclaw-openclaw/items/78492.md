## Summary

Current main still matches the reported updater failure path, and the linked closing PR is open rather than merged. The issue should stay open until the service-safe auto-update fix lands or is replaced.

## What This Changes

The issue asks for the gateway auto-updater to spawn `openclaw update --yes` without inherited gateway service marker env so package updates can run from managed service installs.

## Best Possible Solution

Land or rework the open updater fix so the gateway auto-update handoff can run safely outside the live gateway service context, while preserving the manual in-service package-update guard and adding regression plus managed-service proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: managed services set the marker env, the auto-update child does not unset it, env resolution preserves omitted keys, and the update guard blocks marked gateway-service envs; I did not run a live macOS/npm update.

## Solution Assessment

Unclear for the current linked PR. Stripping the marker env is necessary, but the final fix should also address or explicitly preserve the non-dry-run service ancestry guard so the managed auto-update path actually reaches package installation.
