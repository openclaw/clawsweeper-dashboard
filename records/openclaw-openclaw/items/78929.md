## Summary

Keep open: current main still contains the Telegram silent-fallback path that can rewrite a synthetic `NO_REPLY` into the reported text, and an open closing PR is already the right implementation vehicle.

## What This Changes

Telegram direct-message turns should not send the silent-reply rewrite text while a queued final reply is still in flight.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/79142 after redacted real Telegram proof, or implement the equivalent `queuedFinal` fallback/status guard with focused regression coverage if that PR is abandoned.

## Reproduction Assessment

Yes, source-reproducible: on current main, a turn with `dispatchResult.queuedFinal=true` but no observed `deliveryState` delivery can enter the fallback and synthesize `NO_REPLY`, which direct Telegram rewrites to visible canned text. I did not establish a live intermittent Telegram repro in this read-only review.

## Solution Assessment

Yes, the narrowest maintainable fix is to use the existing `queuedFinal` signal in the Telegram silent-fallback and status-response gates. The open PR already takes that direction, but should wait for real Telegram proof before merge.
