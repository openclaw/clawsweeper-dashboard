## Summary

Current main still contains the Telegram fallback path that can send a rewritten silent reply when `queuedFinal` is true but lane delivery is not marked delivered, and an open closing PR already targets this bug.

## What This Changes

The issue reports Telegram sending the rewritten silent-reply text before the actual delayed reply in tool-using turns.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/79142 or an equivalent narrow fix that treats `queuedFinal` as suppressing the Telegram silent-reply fallback, with regression coverage and real behavior proof before merge.

## Reproduction Assessment

Yes, by source inspection: a direct Telegram turn with `queuedFinal=true` and no observed `deliveryState.delivered` currently falls into the synthetic `NO_REPLY` fallback path, which rewrites to visible text. I did not live-send through Telegram in this read-only review.

## Solution Assessment

Yes, the linked PR's proposed direction is the narrowest maintainable fix: skip the silent-reply fallback and status-error path when the dispatcher already reports a queued final reply. The issue itself should remain open until that or an equivalent fix lands.
