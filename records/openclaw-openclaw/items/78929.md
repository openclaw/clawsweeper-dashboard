## Summary

Keep open: current main still has a source-reproducible Telegram fallback path that can rewrite `NO_REPLY` into a visible direct-message placeholder, and an open linked PR with closing syntax is already the right fix path.

## What This Changes

The issue reports an intermittent Telegram race where a rewritten silent-reply fallback is sent before the real reply during tool-using turns.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/79142, or an equivalent narrow `queuedFinal` guard, after real behavior proof; keep this issue open until a fix merges.

## Reproduction Assessment

Yes at source level: current main can have `queuedFinal=true` while Telegram delivery state is not marked delivered, then still create a rewritten `NO_REPLY` fallback. I did not live-reproduce the intermittent Telegram race in this read-only pass.

## Solution Assessment

Yes: the linked PR's `!queuedFinal` fallback guard is the narrowest fix for the current Telegram race and matches shared dispatch-result semantics. Because that PR is open and references this issue with closing syntax, the issue should stay open until the PR merges or is replaced.
