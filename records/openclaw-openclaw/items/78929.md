## Summary

Keep open: current main still has the Telegram silent-fallback path that can synthesize a direct-session `NO_REPLY` rewrite while a final reply is queued, and https://github.com/openclaw/openclaw/pull/79142 is the open implementation candidate for this issue.

## What This Changes

Suppress Telegram direct-session silent-reply filler when a real final reply is already queued or in flight.

## Best Possible Solution

Land or replace the narrow Telegram fallback fix so `queuedFinal` prevents synthetic silent-reply filler and status reactions treat queued finals as successful responses.

## Reproduction Assessment

Yes at source level: current main can assign `queuedFinal=true` and still enter the synthetic `NO_REPLY` fallback when delivery has not been observed. I did not run a live Telegram reproduction because this review is read-only and the report says the behavior is intermittent.

## Solution Assessment

Yes: using `queuedFinal` in the existing Telegram fallback/status gate is the narrow maintainable fix, and https://github.com/openclaw/openclaw/pull/79142 is already aimed at that path. The issue should remain open until that PR lands or maintainers choose an equivalent fix.
