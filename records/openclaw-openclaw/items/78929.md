## Summary

Keep this issue open: current main still has source-level paths where a direct Telegram `NO_REPLY` final payload is rewritten to canned visible text, and the existing pending-subagent fix only covers one related case.

## What This Changes

Telegram should avoid sending the canned silent-reply fallback before the real reply on tool-call turns.

## Best Possible Solution

Add a narrow suppression path for exact silent final payloads when the turn has already produced or is waiting on user-visible delivery, while preserving the direct-chat fallback for genuinely silent no-output turns.

## Reproduction Assessment

Yes at source level, but not as a full live reproduction: current tests and delivery code show a direct Telegram `NO_REPLY` final can become visible canned text after tool-result activity. The intermittent active-memory timing still needs live or regression-test confirmation.

## Solution Assessment

No, current main is not the complete fix for this report. The pending-subagent suppression from https://github.com/openclaw/openclaw/pull/69942 is the right pattern, but this path needs a broader pending-or-already-delivered real-reply signal without disabling direct fallback globally.
