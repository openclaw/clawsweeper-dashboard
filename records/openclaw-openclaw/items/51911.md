## Summary

Keep open: current main and the latest release still leave setup-token/manual token writes scoped to one selected agent store, while current main-store read-through only mitigates part of the multi-agent failure. The prompt/docs also do not fully explain the Claude setup-token dance, and the linked closing PR is still open, unmerged, and conflicting.

## What This Changes

The issue asks OpenClaw to make Anthropic setup-token onboarding clear, persist/validate the manual token profile, and make configured agents resolve the credential without manual duplication.

## Best Possible Solution

Settle the supported credential propagation contract, then either document main-store inheritance as the intended setup-token behavior or land a focused sync/validation fix with regression coverage for setup-token and paste-token.

## Reproduction Assessment

Source inspection gives a medium-confidence reproduction of the remaining setup path: setup-token and paste-token still write only one selected agent store, and the Anthropic prompt still asks only for the final setup-token. I did not run a live Anthropic multi-agent gateway repro, and current main-store read-through may avoid the exact startup failure.

## Solution Assessment

Unclear: static sibling sync is a plausible fix, but not clearly the best fix because current main already has main-store credential inheritance. The best path needs a maintainer decision on inheritance versus copying, plus targeted UX/validation coverage.
