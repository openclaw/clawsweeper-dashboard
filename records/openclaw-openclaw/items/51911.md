## Summary

Keep open: current main and v2026.6.6 still write Anthropic setup-token, generic plugin auth, and paste-token credentials to one selected agent store. Runtime main-store read-through and docs mitigate the missing-copy failure, but the setup-token UX, post-write validation, and credential propagation contract still need maintainer/security judgment before the linked conflicting PR or a replacement lands.

## What This Changes

The issue asks OpenClaw to make Anthropic setup-token onboarding reliable by clarifying the Claude CLI token flow, validating persistence, and making credentials available across multi-agent installs.

## Best Possible Solution

Settle the auth-store contract, then land or replace a focused fix that covers the chosen propagation/read-through model, setup-token prompt guidance, post-write validation, regression coverage, and live multi-agent proof.

## Reproduction Assessment

Partly: source inspection gives a high-confidence reproduction of the remaining write-path gap because setup-token and paste-token still write one selected agent store while OAuth has explicit sibling sync. I did not run a live Anthropic setup-token/gateway repro, and runtime read-through may now prevent the exact startup failure.

## Solution Assessment

Unclear: sibling sync is a plausible repair, but current main also has main-agent read-through and docs warn against cloning OAuth refresh tokens, so maintainers should choose the intended credential-sharing contract before implementation lands.
