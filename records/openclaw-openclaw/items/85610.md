## Summary

Current main now fixes the reported Gemini CLI API-key shadowing path: selected Google/Gemini API-key profiles are staged into an isolated Gemini CLI home, inherited Google/Gemini auth environment is cleared, and the fix is present on main after the latest stable release.

## What This Changes

The issue reports that `openclaw infer model run --local` fails for `google-gemini-cli` with `API_KEY_INVALID` even though the same Gemini API key works through REST and the standalone `gemini` CLI.

## Best Possible Solution

Keep the provider-owned Gemini CLI auth bridge on main and ship it in the next release so API-key-backed local Gemini CLI runs no longer inherit the wrong ambient Google credential.

## Reproduction Assessment

Yes. The issue gives concrete failing commands for an older OpenClaw build, and current source/history show the implicated ambient-auth path is now replaced by selected-profile staging; I did not rerun a live Gemini API-key prompt in this read-only review.

## Solution Assessment

Yes. The current fix is at the maintainable owner boundary: the Google plugin's Gemini CLI backend stages only the selected profile for the subprocess and avoids adding public SDK credential forwarding surface.
