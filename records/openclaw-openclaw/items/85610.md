## Label Justifications

- `P2`: This is a normal-priority auth-provider bug affecting a specific local Gemini CLI execution path rather than a core runtime outage.
- `impact:auth-provider`: The report concerns provider authentication, model runtime routing, and credential selection for `google-gemini-cli`.

## Summary

Current main already fixes the reported Gemini CLI API-key shadowing path by staging the selected OpenClaw Google/Gemini API-key profile into the Gemini CLI subprocess and clearing ambient auth variables before launch; the fix is main-only after v2026.6.8.

## What This Changes

The issue reports `openclaw infer model run --local` failing with `API_KEY_INVALID` for `google-gemini-cli` even though the same Gemini API key works through REST and the standalone Gemini CLI.

## Best Possible Solution

Keep the provider-owned Gemini CLI auth bridge on main and ship it in the next release so selected API-key profiles no longer lose to ambient Google/Gemini auth state.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: the issue gives concrete failing commands, and v2026.6.8 shows the Gemini CLI backend had no auth bridge while current main stages selected credentials. I did not rerun a live Gemini API-key request in this read-only review.

## Solution Assessment

Yes, the current fix is the best layer: it stays inside the Google plugin's bundled Gemini CLI backend, isolates selected credentials for the subprocess, and avoids adding a public Plugin SDK credential-forwarding contract.
