## Summary

Current main already fixes the reported Gemini CLI auth shadowing path: selected Google/Gemini API-key credentials are staged into an isolated Gemini CLI home, inherited Google/Gemini auth variables are cleared before spawning `gemini`, and the fix is current-main-only after the latest release.

## What This Changes

The issue reports local `google-gemini-cli` model runs rejecting a Gemini API key that succeeds through Gemini REST and the standalone `gemini` CLI.

## Best Possible Solution

Keep the provider-owned Gemini CLI auth bridge on main and ship it in the next release so local Gemini CLI runs use the selected OpenClaw credential instead of ambient machine auth.
