## Summary

Current main and the latest release no longer exercise the missing-`sharp` failure path: image processing now uses Rastermill from a root dependency, and in-limit PNG/JPEG/WebP media can bypass optimization safely before any backend work.

## What This Changes

The issue requests clearer handling of missing image optimization dependencies so the image tool does not fail every image with an opaque `Failed to optimize image` error.

## Best Possible Solution

Keep the Rastermill-backed implementation and close this report; any new image-processing backend failure should be filed with current-version logs rather than reopening the old missing-`sharp` package path.

## Reproduction Assessment

No current-main reproduction remains: the old missing-`sharp` path is source-reproducible from the reported 2026.4.x code and comments, but current main and `v2026.6.8` use Rastermill and preserve safe in-limit originals before optimization.

## Solution Assessment

Yes. Replacing the missing optional Sharp path with a declared root Rastermill backend plus guarded original-byte preservation is a better durable fix than only appending the swallowed Sharp error.
