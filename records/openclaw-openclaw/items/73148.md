## Summary

Keep open. Current main now preserves the underlying Sharp error in the optimizer message, but it still does not prove Sharp is available in packaged installs or provide a guarded fallback for normal image delivery when the backend is absent.

## What This Changes

The issue asks OpenClaw to avoid opaque image-tool and media-send failures when Sharp is missing by making the dependency available, surfacing the backend error, or falling back safely.

## Best Possible Solution

Choose and implement a supported media dependency policy: either ship or stage Sharp for `media-understanding-core` in packaged installs, or add a guarded original-image fallback for already-safe media, with regression coverage for image-tool and outbound channel paths.

## Reproduction Assessment

Yes from source, though not live-executed here. Current main still sends normal image media through the Sharp-backed optimizer while the root package does not declare Sharp as a runtime dependency, and the latest user comment reproduces the missing-Sharp path on OpenClaw 2026.5.2.

## Solution Assessment

Unclear. The diagnostic-only fix is useful but incomplete; the remaining fix needs a maintainer decision between native dependency packaging/staging and a conservative media fallback policy.
