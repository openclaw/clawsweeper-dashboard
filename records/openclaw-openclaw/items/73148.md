## Summary

Keep open. Current main has improved the diagnostic message, but it still does not guarantee `sharp` is present in packaged installs and normal image handling still fails when the Sharp-backed optimizer cannot load.

## What This Changes

The issue asks OpenClaw to stop opaque image optimization failures when `sharp` is absent and make the backend dependency either diagnosable, reliably available, or safely bypassed.

## Best Possible Solution

Keep the diagnostic improvement, then ensure `sharp` is available before packaged media-understanding public artifacts load or add a guarded under-cap original-image fallback that preserves existing byte and pixel safety checks.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: normal image media still enters the optimizer, the optimizer calls the Sharp-backed media-understanding ops, and missing `sharp` throws before image delivery; issue comments also reproduce this on 2026.4.25 through 2026.5.2.

## Solution Assessment

Unclear. The current diagnostic-only behavior is useful but incomplete; the safest complete fix needs maintainer choice on dependency staging versus a conservative media fallback.
