## Summary

Keep open. Current main fixed the fully opaque diagnostic by preserving resize failure details, but the packaged-install Sharp availability problem is still not resolved in source, and the normal image path still fails when the backend cannot load.

## What This Changes

The issue requests actionable image optimization errors and reliable Sharp availability for packaged installs where the image tool currently fails when Sharp is missing.

## Best Possible Solution

Guarantee a supported image backend is available before `media-understanding-core` image ops load, and keep the current cause-rich diagnostics; add a fallback only if maintainers accept that policy with resource guards intact.

## Reproduction Assessment

Yes. Source inspection shows the normal image path still flows into the Sharp-backed optimizer, while package metadata excludes extension `node_modules` and does not make Sharp a root runtime dependency; the issue comments also reproduce the failure on v2026.5.2.

## Solution Assessment

Unclear. The diagnostic change on main is a good partial fix, but the remaining package/runtime behavior needs a maintainer choice between dependency staging, a hard dependency, and a guarded fallback.
