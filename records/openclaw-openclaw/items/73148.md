## Summary

Keep open. Current main fixes the opaque diagnostic by preserving the Sharp import failure in the image optimizer error, but the source still does not guarantee Sharp is present in packaged installs or provide a guarded original-image fallback when the optimizer backend is unavailable.

## What This Changes

The issue asks OpenClaw to stop failing opaquely when `sharp` is absent and either install/stage Sharp reliably or expose the underlying dependency error.

## Best Possible Solution

Decide the package/runtime boundary, then either make `media-understanding-core` Sharp reliably available after normal install/update or add a guarded fallback for already-safe originals while keeping the actionable error detail.

## Reproduction Assessment

Yes for the remaining missing-backend failure: a packaged install without `sharp` still reaches the Sharp-backed resize boundary, and current source shows that failure propagates through image optimization. The old fully opaque message is no longer reproducible from source because the optimizer now includes failure detail.

## Solution Assessment

Unclear. The current diagnostic fix is narrow and useful, but the best complete fix needs maintainer judgment between dependency packaging, doctor/update repair expectations, and a conservative fallback policy.
