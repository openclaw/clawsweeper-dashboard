## Summary

Keep open. Current main has shipped the diagnostic part by preserving resize/import failure details, but the remaining missing-Sharp backend failure is still source-reproducible because image loading still requires the Sharp-backed optimizer and the package metadata does not make Sharp a root runtime dependency or provide a guarded original-image fallback.

## What This Changes

The issue requests that image optimization failures caused by missing `sharp` become actionable and/or that `sharp` be reliably available for image processing.

## Best Possible Solution

Decide the package/runtime boundary, then either make `media-understanding-core`'s Sharp dependency reliably available after normal packaged installs/upgrades or add a guarded under-cap original-image fallback while preserving the clearer error detail.

## Reproduction Assessment

Yes, source-reproducible for the remaining missing-backend failure: image loading still reaches the Sharp-backed optimizer, and current package metadata does not guarantee `sharp` is present in the root package. The old fully opaque error wording is no longer reproducible from source because the optimizer now includes failure detail.

## Solution Assessment

Unclear. The diagnostic fix is narrow and already shipped, but the complete fix needs a maintainer choice between dependency packaging and a guarded fallback policy.
