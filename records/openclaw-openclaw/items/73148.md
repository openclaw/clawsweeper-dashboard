## What This Changes

The issue reports that missing `sharp` makes the image tool fail with an opaque `Failed to optimize image` error and asks for reliable dependency handling or clearer diagnostics.

## Best Possible Solution

Resolve the linked implementation path into one accepted fix: land or explicitly supersede the optimizer fallback PR and the packaged runtime-dependency repair PR, then close this issue only after the accepted changes are on main with release provenance.

## Reproduction Assessment

Yes. The issue and follow-up comments provide concrete global/package install scenarios where `sharp` is absent, and current source confirms the non-`sips` path still loads `media-understanding-core` image ops and depends on `sharp` for resize/metadata work.

## Solution Assessment

Unclear until the open PRs are resolved. Keeping `sharp` plugin-owned and staged through the bundled runtime-deps system fits the current architecture better than moving it back to a root hard dependency, but fallback behavior and packaged doctor repair coverage still need maintainer resolution.
