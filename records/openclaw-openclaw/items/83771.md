## Summary

Keep open: the feature is useful and the supplied build/import proof supports the narrow behavior, but the patch introduces a Docker-only public pip package env name instead of following the existing image-scoped build-arg contract shared by Docker and Podman.

## What This Changes

The PR adds an opt-in Docker build argument and Docker setup pass-through for installing requested Python packages into locally built images.

## Best Possible Solution

Ship the opt-in pip package hook as an image-scoped build contract shared by Docker and Podman, with docs and focused contract coverage, while keeping unset/default builds unchanged.

## Reproduction Assessment

Not applicable; this is an additive local image-build option, not a reported regression. The supplied logs do verify the intended after state by importing `requests` and `humanize` from the built `openclaw:local` image.

## Solution Assessment

No; the useful behavior should be implemented through the existing image-scoped build-arg pattern and shared Docker/Podman setup path before the new config surface ships.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Use the shared image package env:** `scripts/docker/setup.sh:300`
  - body: Adding `OPENCLAW_DOCKER_PIP_PACKAGES` here makes the setup script persist a Docker-only public env key, while the existing image package hook is `OPENCLAW_IMAGE_APT_PACKAGES` and `scripts/podman/setup.sh` builds the same `Dockerfile` through `OPENCLAW_IMAGE_*` args. If this ships, Podman/local image users get no matching setup path and OpenClaw has to support two naming contracts for image package installs.
