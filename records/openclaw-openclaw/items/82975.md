## Summary

Close as implemented: current main and the v2026.5.18 release expose canonical OpenAI GPT-5.x `xhigh` thinking metadata through provider policy, and the native `/think` picker now reads those provider/model choices rather than falling back to `default, off`.

## What This Changes

The issue asks for canonical `openai/*` models routed through Codex auth to expose and accept `/think` choices such as `xhigh` at runtime.

## Best Possible Solution

No high-confidence current-main failing reproduction remains. Source inspection shows canonical `openai/gpt-5.5` now resolves to a provider thinking profile that includes `xhigh`, and v2026.5.18 ships the same policy.

## Solution Assessment

Yes. The maintainable fix is the current provider-owned thinking profile path, which lets the picker and directive validation share canonical OpenAI/OpenAI-Codex capability metadata instead of adding channel-specific special cases.
