## Summary

Keep open: current main still uses one resolved realtime provider config for voice-call bridge sessions, and per-agent realtime voice selection would add a new voice-call plugin config/API surface rather than fix broken documented behavior.

## What This Changes

Request per-agent OpenAI Realtime voice selection for the voice-call plugin so callback-transferred agents can speak with distinct voices.

## Best Possible Solution

Expose a documented, provider-aware per-agent realtime provider override in the voice-call plugin that preserves the existing global provider default and keeps provider-specific config ownership clear.

## Reproduction Assessment

Not applicable: this is a feature/config request, not broken documented behavior. Source inspection confirms current main passes one stored realtime provider config into every bridge session.

## Solution Assessment

Unclear: the proposed `realtime.agentVoices` map would address the immediate OpenAI voice case, but maintainers should first choose whether the API is voice-only, provider-generic, or tied into existing agent/TTS routing semantics.
