## Summary

Keep open: current main still creates every voice-call realtime bridge session with one resolved provider config, and per-agent OpenAI Realtime voice selection would add a new public voice-call plugin configuration/API surface rather than fix broken documented behavior.

## What This Changes

The issue requests per-agent OpenAI Realtime voice selection for the voice-call plugin, likely by mapping agent IDs to provider voice overrides.

## Best Possible Solution

Expose a documented, provider-aware per-agent realtime provider override in the voice-call plugin, preserving the existing global provider default and coordinating with per-call agent resolution so the selected agent can supply the session voice.

## Reproduction Assessment

Not applicable: this is a feature/config request, not broken documented behavior. Source inspection confirms current main passes one stored realtime provider config into every bridge session.

## Solution Assessment

Unclear: the proposed `realtime.agentVoices` map addresses the immediate OpenAI voice case, but maintainers should decide whether the API is voice-only, provider-generic, or tied into existing agent/TTS routing semantics.
