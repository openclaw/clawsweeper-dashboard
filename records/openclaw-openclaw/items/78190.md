## Summary

Keep open: current main and v2026.6.8 still use one provider-level realtime voice configuration for voice-call bridge sessions, and the related open routing PR only addresses preserving the call's agent identity, not a per-agent realtime voice config surface.

## What This Changes

Add per-agent OpenAI Realtime voice selection to the voice-call plugin so different agents can answer calls with distinct voices.

## Best Possible Solution

Define a documented, provider-aware per-agent realtime override in the voice-call plugin that preserves the existing global provider default and keys from settled call-level agent identity.

## Reproduction Assessment

Not applicable as a strict bug reproduction: this is a feature/config request. Source inspection does confirm the current-main behavior and v2026.6.8 behavior reuse one provider-level realtime voice config for every bridge session.

## Solution Assessment

Unclear: the requested agentVoices map solves the immediate OpenAI voice case, but maintainers should choose whether the permanent API is voice-only, provider-generic, or tied to the related per-call agent-routing work.
