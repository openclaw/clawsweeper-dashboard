## Summary

Keep open: current main still has only provider-global realtime provider configuration for voice-call bridge sessions, and the requested per-agent OpenAI Realtime voice selection would add a new public plugin config/API surface. The adjacent open routing PR may help determine the active agent per call, but it does not by itself add a per-agent realtime voice/provider override.

## What This Changes

The issue requests a voice-call plugin config surface that selects a different OpenAI Realtime voice per agent while preserving the global realtime provider voice as the default.

## Best Possible Solution

Expose a documented, provider-aware per-agent realtime provider override in the voice-call plugin while preserving the existing global provider default and coordinating with the per-call agent routing design.

## Reproduction Assessment

Not applicable: this is a feature/config request, not broken documented behavior. Source inspection confirms current main creates realtime bridge sessions with the same stored provider config for every call.

## Solution Assessment

Unclear: the proposed agentVoices map solves the immediate OpenAI voice case, but maintainers should choose whether the public API is voice-only, provider-generic, or composed with per-call agent routing semantics.
