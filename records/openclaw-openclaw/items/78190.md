## Summary

Keep open: current main and v2026.6.10 still use one provider-level realtime provider config for Voice Call bridge sessions, and the open routing PR is only partial overlap rather than the requested per-agent realtime voice override API.

## What This Changes

Add a voice-call realtime configuration surface for per-agent OpenAI Realtime voice overrides instead of using one provider-level voice for every bridge session.

## Best Possible Solution

Define a documented, provider-aware per-agent realtime override API in the voice-call plugin that preserves global provider defaults and keys from settled call-level agent identity.

## Reproduction Assessment

Yes, source-level: current main has no per-agent realtime voice config and constructs each realtime bridge session with the same providerConfig. This is not a strict bug reproduction because the request is for a new config capability.

## Solution Assessment

Unclear: the requested agentVoices map solves the immediate OpenAI voice case, but maintainers should choose whether the permanent API is voice-only, provider-generic, or tied to the related call-agent routing work.
