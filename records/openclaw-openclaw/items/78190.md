## Summary

Keep open: current main still resolves one realtime provider config for the voice-call runtime and reuses it for every bridge session, while the requested per-agent voice selection would add a new public plugin config/API surface.

## What This Changes

Request per-agent OpenAI Realtime voice selection in the voice-call plugin so routed agents can use distinct voices.

## Best Possible Solution

Expose a documented, provider-aware per-agent realtime provider override in the voice-call plugin while preserving the existing global provider default.

## Reproduction Assessment

Not applicable: this is a feature/config request, not broken documented behavior. Source inspection gives a high-confidence proof of current behavior: every bridge session receives the same stored realtime provider config.

## Solution Assessment

Unclear: the proposed agentVoices map solves the immediate OpenAI voice case, but maintainers should choose whether the public API is voice-only, provider-generic, or tied into existing agent/TTS routing semantics.
