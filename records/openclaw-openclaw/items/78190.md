## Summary

Keep open: current main and v2026.6.10 still use one provider-level realtime voice configuration for voice-call bridge sessions, and the related open routing PR only preserves call agent identity rather than adding the requested per-agent realtime voice config surface.

## What This Changes

The issue requests a per-agent OpenAI Realtime voice selection surface for the voice-call plugin so different agents can use distinct voices during calls.

## Best Possible Solution

Define a documented voice-call plugin API that preserves the global provider default while allowing provider-aware per-agent realtime overrides keyed from settled call-level agent identity.

## Reproduction Assessment

Yes, source-level: current main has no per-agent realtime voice config and constructs each realtime bridge session with the same providerConfig. This is not a strict runtime bug reproduction because the request is for a new config capability.

## Solution Assessment

Unclear: the requested `agentVoices` map solves the immediate OpenAI voice case, but maintainers should choose whether the permanent API is voice-only, provider-generic, or tied to the related call-agent routing work.
