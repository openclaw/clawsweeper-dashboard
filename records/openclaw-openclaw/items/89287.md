## Summary

Keep this PR open: the direction is useful, but the current matcher can still reject valid current-thread/topic delivery evidence, the new JSON field is undocumented, and the supplied proof is still a substituted harness rather than live threaded-channel delivery proof.

## What This Changes

The PR adds target metadata to durable delivery outcomes and makes subagent completion delivery accounting compare message-tool and automatic delivery evidence against provider, account, channel, and thread targets.

## Best Possible Solution

Preserve the existing source-delivery thread semantics while rejecting explicit wrong targets, document the new delivery-status target field, then verify with redacted live threaded-channel proof or maintainer override.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: current `sourceDeliveryTargetsMatch` accepts topic-encoded and `threadImplicit` threaded evidence, while the PR's exact matcher rejects those same shapes by checking only raw `target.threadId`. I did not run tests because this review is read-only.

## Solution Assessment

No. The best fix is to keep the existing source-delivery matcher semantics as the source of truth, then layer strict account/thread rejection only for explicit wrong or suppressed targets.
