## Label Justifications

- `P2`: The issue affects Telegram reply-context reliability with bounded channel/session impact rather than a core outage or emergency.
- `impact:session-state`: The requested contract is about preserving reply metadata through current-turn session and runtime context without drift.

## Summary

Keep open: current main and v2026.6.8 still split Telegram reply context between enriched `Body`/`ReplyTo*` data, plain `BodyForAgent`, and text-only current inbound prompt context, while the first-class runtime contract is still an open maintainer decision with an open candidate PR.

## What This Changes

The issue requests a durable first-class inbound quote/reply context contract for Telegram so reply metadata and agent-visible input cannot drift apart.

## Best Possible Solution

Review and land one maintainer-approved current-inbound reply metadata contract through the open runtime-context PR or a replacement, then align Telegram mapping, docs, and regression coverage around that single path.

## Reproduction Assessment

Yes for a source-level reproduction: current main still builds reply-aware Telegram `Body` and `ReplyTo*` fields while keeping `BodyForAgent` as the current message text and `CurrentInboundPromptContext` text-only. I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

Unclear until maintainers choose the durable runtime/API contract. The open PR is a plausible best path; a Telegram-only `BodyForAgent` enrichment is not clearly best because it would set semantics without aligning the structured runtime-context contract and docs.
