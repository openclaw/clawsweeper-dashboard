- `impact:message-loss`: The reported failure mode loses quoted/replied message context before the agent can use it.

## Summary

Keep open: current main already has meaningful Telegram reply metadata and prompt-context handling, but it still preserves the split this issue is about: Telegram builds a reply-aware `Body` and structured quote fields while `BodyForAgent` remains the plain current message text. That makes the remaining work a product/API contract decision, not a stale cleanup close.

## What This Changes

The issue asks OpenClaw to make Telegram quote/reply context a durable inbound contract so agent-visible input, structured reply metadata, hooks, docs, and regression coverage cannot drift apart.

## Best Possible Solution

Maintainers should choose and document one canonical inbound reply contract for Telegram and shared channels, then align `BodyForAgent`, structured reply fields, hook-visible semantics, docs, and regression tests to that contract.

## Reproduction Assessment

No high-confidence live reproduction was established in this read-only review. Source inspection does show the current split: Telegram sends reply-aware `Body` plus structured quote fields while `BodyForAgent` remains `bodyText`.

## Solution Assessment

Unclear until maintainers choose the contract. Replacing `BodyForAgent` with a concatenated reply-aware body is not obviously the best fix because current docs and code intentionally prefer structured untrusted context blocks for reply metadata.
