## Label Justifications

- `P2`: The issue affects Telegram reply comprehension and channel stability, but it is scoped to a bounded messaging-context contract and is not a crash, security bypass, or core outage.
- `impact:session-state`: The reported failure is agent context drift where the visible Telegram reply relationship can be lost before the agent reasons over the turn.
- `impact:message-loss`: The quoted or replied-to message context can be silently omitted from the agent-visible input even though the user replied in Telegram.

## Summary

Keep open: current main has meaningful Telegram reply metadata support, but the source still shows the split surface this issue calls out, with enriched reply context attached to `Body` and supplemental/prompt metadata while `BodyForAgent` remains the plain current message body. That makes this a real product/API contract question rather than a stale or already-implemented cleanup item.

## What This Changes

Request to make Telegram quote/reply context a first-class durable inbound contract so agent-visible body, structured metadata, prompt rendering, hooks, and regression coverage cannot drift apart.

## Best Possible Solution

Define one canonical inbound reply-context contract across the Telegram plugin and shared channel helper, then add regression coverage that proves reply metadata, prompt rendering, hook-visible bodyForAgent semantics, and final agent input cannot diverge.

## Reproduction Assessment

No, not as a high-confidence failing current-main runtime repro. Source inspection does confirm the split path: Telegram enriches `Body`/supplemental reply metadata while passing plain `bodyText` as `BodyForAgent`, but no live Telegram run proves current main drops the final prompt context.

## Solution Assessment

Unclear until maintainers choose the contract. Simply copying reply suffix text into `BodyForAgent` might fix one symptom but could duplicate untrusted context; the better direction is a single canonical structured contract with explicit rendering and hook semantics.
