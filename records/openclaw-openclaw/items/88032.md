## Summary

Keep open: current main and the latest release still keep Telegram reply context split between enriched Body/ReplyTo fields, plain BodyForAgent, and a text-only current inbound prompt context, while the runtime-contract PR is still open.

## What This Changes

The issue requests a durable, first-class inbound reply/quote context contract for Telegram instead of relying on separate prompt and runtime mapping behavior.

## Best Possible Solution

Review and land one maintainer-approved current-inbound reply metadata contract through the open runtime-context PR or a replacement, then align Telegram mapping, docs, and regression coverage.

## Reproduction Assessment

Yes for the source-level split: current main still builds reply-aware Telegram Body and ReplyTo fields while keeping BodyForAgent as current-message text. I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

Unclear until maintainers choose the durable contract. The open PR is a plausible path; a Telegram-only BodyForAgent concatenation is not clearly the best fix because it would set API semantics without aligning the runtime-context contract and docs.
