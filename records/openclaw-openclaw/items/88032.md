- `P2`: The issue affects Telegram reply-context reliability with bounded channel/session impact rather than a core outage or emergency.

## Summary

Keep open: current main and v2026.6.9 still split Telegram reply context across enriched Body/ReplyTo fields, plain BodyForAgent, and text-only current inbound context, while open PRs still need a maintainer contract choice.

## What This Changes

The issue asks for Telegram quote/reply context to become a durable first-class inbound/runtime contract instead of relying on separate prompt formatting and BodyForAgent mapping.

## Best Possible Solution

Land one maintainer-approved current-inbound reply contract, likely through https://github.com/openclaw/openclaw/pull/90745 or a smaller split after https://github.com/openclaw/openclaw/pull/89428, then align Telegram mapping, docs, and regression coverage around that single path.

## Reproduction Assessment

Yes for a source-level reproduction: current main and v2026.6.9 build reply-aware Telegram Body and ReplyTo fields while keeping BodyForAgent as current text and CurrentInboundPromptContext text-only. I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

Unclear until maintainers choose the durable runtime/API contract. A first-class structured runtime context is a better direction than local dist patching or Telegram-only BodyForAgent concatenation, but the canonical PR shape still needs maintainer selection.
