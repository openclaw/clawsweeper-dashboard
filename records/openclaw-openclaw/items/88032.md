## Summary

Keep open: current main and the latest release still keep Telegram reply context split between enriched `Body`/`ReplyTo*` fields and plain `BodyForAgent`, while the related runtime-contract PR is still open rather than merged.

## What This Changes

The issue requests a durable Telegram/shared-channel inbound reply context contract, structured metadata, deterministic agent input, docs, and regression coverage.

## Best Possible Solution

Review, revise if needed, and land one canonical current-inbound reply metadata contract through https://github.com/openclaw/openclaw/pull/90745 or a maintainer-approved replacement, then align Telegram mapping, hook/docs wording, and regression coverage; keep narrower allowlist behavior tracked at https://github.com/openclaw/openclaw/issues/82002.

## Reproduction Assessment

Yes from source inspection for the split contract: current main still builds reply-aware Telegram `Body` and structured `ReplyTo*` fields while `BodyForAgent` remains plain current-message text. I did not establish a live failing Telegram turn in this read-only review.

## Solution Assessment

Unclear until maintainers choose the durable contract. The open PR is a plausible path, while a one-off concatenation into Telegram `BodyForAgent` is not clearly the best fix because hook-visible semantics and docs need to stay compatible.
