## Label Justifications

- `P2`: The issue affects Telegram reply-context reliability with bounded channel/session impact rather than a core outage or emergency.
- `impact:session-state`: The requested contract is about preserving current-turn reply metadata through session and runtime context without drift.
- `impact:message-loss`: When the split regresses, the Telegram reply target context can be dropped from the agent-visible turn even though the user replied in Telegram.

## Summary

Keep open: current main still has the reported split between Telegram reply-aware `Body`, plain `BodyForAgent`, and text-only current inbound runtime context, while the direct runtime-contract candidate remains open and needs maintainer product/API direction.

## What This Changes

The issue requests a durable first-class Telegram quote/reply inbound contract so reply context survives channel ingress, runtime handoff, agent-visible context, and updates without local `dist/*` patching.

## Best Possible Solution

Choose and land one maintainer-approved current-inbound reply-context contract, then align Telegram mapping, docs, and regression coverage around that single path while keeping quoted text untrusted.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main and `v2026.6.9` build reply-aware Telegram `Body` and `ReplyTo*` fields while keeping `BodyForAgent` current-message-only and `CurrentInboundPromptContext` text-only. I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

Unclear until maintainers choose the durable runtime/API contract. A first-class structured runtime context is a better direction than local `dist/*` patching or Telegram-only `BodyForAgent` concatenation, but the canonical PR shape still needs maintainer selection.
