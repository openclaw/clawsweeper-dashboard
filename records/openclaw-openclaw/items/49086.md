## Summary

Current main still exposes the reported heartbeat timestamp footgun. The default workspace AGENTS template still tells agents to track periodic checks in `memory/heartbeat-state.json` with raw numeric Unix timestamps, while the current-time prompt line does not provide epoch seconds and the safer `HEARTBEAT.md` task-state path is documented but not connected from that template guidance.

## What This Changes

The issue asks OpenClaw to prevent agents from manually estimating Unix timestamps for heartbeat state, preferably by updating the default workspace template and/or moving timestamp handling into system-owned heartbeat state.

## Best Possible Solution

Update the default workspace guidance so interval heartbeat checks use `HEARTBEAT.md` `tasks:` blocks and OpenClaw-owned `heartbeatTaskState` by default. If `memory/heartbeat-state.json` remains mentioned, it should explicitly require epoch values to come from a system/tool call such as `date +%s` and warn agents never to calculate them manually.

## Reproduction Assessment

No high-confidence live reproduction exists for the exact LLM hallucination because it is probabilistic. Static reproduction of the unsafe condition is high confidence: current main still ships raw epoch heartbeat-state guidance without a tool-call instruction, and the prompt does not provide epoch seconds.

## Solution Assessment

Yes, a narrow template/docs repair is the best first fix because the failure is caused by bootstrap guidance that asks the model to own raw timestamp state. Adding current epoch injection or future-timestamp clamping can be considered as defense in depth, but the minimum maintainable fix is to stop steering agents toward manual epoch JSON.
