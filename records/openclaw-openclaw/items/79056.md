## Summary

Keep open: current main still records WhatsApp inbound activity before outbound echo filtering, and the linked open PR is the active fix path rather than a reason to close this issue early.

## What This Changes

WhatsApp self-sends through the Gateway should advance outbound activity without updating `channels status` inbound timestamps.

## Best Possible Solution

Land or revise https://github.com/openclaw/openclaw/pull/79057 so WhatsApp records inbound activity only for accepted inbound messages, with regression coverage for outbound echoes.

## Reproduction Assessment

Yes. The issue provides a concrete live WhatsApp Web reproduction on current main, and source inspection shows the inbound activity record happens before the existing outbound-echo filter.

## Solution Assessment

Yes. Moving inbound activity recording to the accepted-inbound boundary, after echo/history/enrichment/dedupe filters and before enqueue, is the narrowest maintainable fix for this status-accounting bug.
