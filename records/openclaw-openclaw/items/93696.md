## Summary

Keep open: the Matrix reasoning bug is real and worth fixing, but this PR is not merge-ready because it changes only the lower-level Matrix reply helper while current generic dispatch still drops `isReasoning` payloads first, and the added `msgtype` option is not accepted or forwarded by `sendMessageMatrix`. The diff also changes unflagged reasoning-looking text from suppressed to delivered, which is a reasoning-visibility regression risk.

## What This Changes

The PR changes Matrix reply delivery tests and helper logic to send reasoning-like replies as `m.notice` instead of suppressing them.

## Best Possible Solution

Use a narrow Matrix dispatch opt-in that lets explicit `isReasoning` payloads reach a Matrix send path that actually emits `m.notice`, while keeping unflagged reasoning-looking text suppressed and adding live Matrix proof before merge.

## Reproduction Assessment

Yes by source inspection: docs define `/reasoning on` as separate visible reasoning output, the runner can emit `{ isReasoning: true }` payloads, and current generic dispatch plus Matrix delivery suppress them before send. I did not run a live Matrix homeserver reproduction in this read-only review.

## Solution Assessment

No. This PR is a plausible lower-level change, but the best fix must include the dispatch-level Matrix opt-in and a Matrix send path that actually supports `m.notice`, while preserving suppression for unflagged reasoning-looking text.
