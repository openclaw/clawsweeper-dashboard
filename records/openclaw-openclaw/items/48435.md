## Summary

Keep open. Current main still has the Slack unfurl configuration gap: the strict Slack config schema and TypeScript config type do not expose the requested keys, outbound Slack `chat.postMessage` payloads omit `unfurl_links` and `unfurl_media`, and the related unfurl PRs found in the discussion/search are closed without merge.

## What This Changes

The issue requests Slack config options that let OpenClaw suppress Slack link and media unfurls on bot messages by passing `unfurl_links` and `unfurl_media` to `chat.postMessage`.

## Best Possible Solution

Add backward-compatible bundled Slack plugin/config support for `unfurlLinks` and `unfurlMedia` at top-level and per-account scope, inherit them through the existing account merge path, pass explicitly configured values to Slack `chat.postMessage` sends, and update tests, generated config metadata, Slack docs, and changelog without changing default Slack behavior when unset.

## Reproduction Assessment

Yes. This is a feature gap rather than a crash; the high-confidence reproduction is static on current main: the proposed config keys are absent from strict schema/types and no outbound `chat.postMessage` payload contains the corresponding Slack API arguments.

## Solution Assessment

Yes. A config-driven Slack implementation is the narrowest maintainable solution for this report because Slack supports the flags per `chat.postMessage`, existing account merging gives the right top-level/per-account override model, and hardcoding unfurls off would change existing behavior.
