## Summary

Keep open: the report is actionable, and the current code still has a plausible mismatch where a runtime catalog row marked `reasoning: false` can force `/think` validation to `off` before the OpenAI provider thinking profile is considered, matching the reported rejection even though the picker/profile advertises non-off levels.

## What This Changes

The issue reports that `/think` on Telegram for canonical `openai/gpt-5.5` shows non-off thinking options but rejects every non-off selection with `Use one of: off`.

## Best Possible Solution

Make `/think` menu and directive validation use one authoritative effective provider/model thinking profile for canonical OpenAI/Codex models, overriding stale provider-owned catalog metadata only where the owning plugin policy proves support.

## Reproduction Assessment

Yes, source-reproducible: the issue provides a concrete Telegram transcript, and the current/released code path reproduces the exact error whenever the thinking catalog supplies `openai/gpt-5.5` with `reasoning: false`. I did not run a live Telegram/Pi OAuth repro in this read-only review.

## Solution Assessment

No, the current implementation is not the best end state because a stale runtime catalog row can outrank the provider-owned OpenAI thinking profile. The safer fix is a narrow metadata-precedence or effective-target resolution change with regression coverage, not a global bypass of `reasoning: false`.
