## Summary

Keep open. Current main and v2026.6.6 intentionally fail closed for declared context-engine host incompatibility and preserve mixed-runtime configs with warnings, so the requested per-session legacy demotion is a compatibility/product decision rather than an already-implemented cleanup.

## What This Changes

Request per-session fallback to the legacy context engine when a globally selected context-engine plugin is incompatible with a specific agent runtime.

## Best Possible Solution

Decide the host-incompatibility policy first; if maintainers want mixed-fleet uptime, add a documented per-session legacy demotion with diagnostics and focused tests, otherwise preserve fail-closed behavior and improve operator guidance.

## Reproduction Assessment

Yes, at source level: generic CLI preparation asserts host support for non-legacy engines, and existing tests expect rejection for an engine requiring assemble-before-prompt. I did not run the live lossless-claw mixed-fleet scenario in this read-only pass.

## Solution Assessment

Unclear. Per-session fallback would address the reported uptime problem, but current docs and tests intentionally fail closed for declared host incompatibility, so maintainers need to choose the safety contract before a code change.
