## Label Justifications

- `P2`: This is a normal-priority Control UI rendering bug with a clear affected surface and limited blast radius, but it can make multiline command prompts hard to trust from Dashboard.

## Summary

This issue is still necessary: current main and the latest release still render chat message text through the Control UI markdown pipeline, whose code-block renderers add visible Copy-button chrome and whose indented-code parsing can split raw multiline shell snippets. Source inspection also shows the raw `chat.send` payload is sent separately, so the confirmed problem is the Dashboard rendering path rather than proven agent-input mutation.

## What This Changes

The issue requests byte-faithful Dashboard rendering for submitted multiline shell/heredoc commands without injected Copy text or indentation damage.

## Best Possible Solution

Keep the raw `chat.send` payload path unchanged, but make user-message rendering preserve multiline shell/heredoc text without injecting code-block Copy chrome while retaining assistant markdown/code-copy behavior.

## Reproduction Assessment

Yes for the visible render path: source and existing tests show current main renders user text through markdown-it and inserts Copy-button labels into rendered code-block markup. I did not establish a live failing Dashboard session, and source inspection suggests the agent request payload itself still receives the raw message.

## Solution Assessment

Yes, the maintainable fix is role-aware rendering or a no-chrome/plain-preserve mode for user messages, not changing the gateway `chat.send` request path. The repair should avoid removing useful assistant code-block copy controls.
