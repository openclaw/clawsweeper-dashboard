## Summary

Keep open. Current main still lacks the requested high-usage TTL bypass: there is no `forcePruneRatio` config/schema/docs surface, and the context-pruning hook still returns inside a fresh cache-TTL window before context usage is estimated. The linked implementation PR is closed unmerged, while the adjacent rolling-mode PR does not provide this thresholded cache-TTL behavior.

## What This Changes

Add optional `agents.defaults.contextPruning.forcePruneRatio` so context pruning can bypass cache TTL when estimated context usage meets a configured threshold.

## Best Possible Solution

Implement the feature as an additive, unset-by-default context-pruning knob: validate and document `forcePruneRatio`, carry it through effective settings and embedded runtime setup, estimate finite usage before the fresh-TTL early return, bypass TTL only when usage meets the threshold, and preserve existing cache-TTL behavior otherwise.

## Reproduction Assessment

Yes. Configure cache-TTL pruning with `lastCacheTouchAt` inside the TTL window and an over-threshold context; current `extension.ts` returns before `pruneContextMessages`, and the existing TTL reset test already proves the fresh-window skip path.

## Solution Assessment

Yes. The requested additive ratio is the narrowest maintainable fix because it keeps cache-TTL mode unchanged until context usage is high; the rolling-mode direction is broader and does not provide a thresholded TTL bypass.
