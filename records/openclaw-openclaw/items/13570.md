## Summary

Keep open: current main still lacks `forcePruneRatio`, and the remaining work is a new public context-pruning trigger/config choice rather than an already-implemented cleanup. The direct implementation PR is closed unmerged, while related rolling-pruning work remains open with a different API shape.

## What This Changes

Add an optional `agents.defaults.contextPruning.forcePruneRatio` setting so context pruning can bypass cache TTL when estimated context usage crosses a configured threshold.

## Best Possible Solution

Maintainers should choose the pruning trigger API, then implement the selected additive surface across settings, runtime gating, config schema/types, docs, changelog, and focused regression tests.

## Reproduction Assessment

Not applicable: this is an enhancement for a new config option rather than a broken existing contract. Source inspection verifies current main lacks `forcePruneRatio` and still gates cache-TTL pruning before usage estimation.

## Solution Assessment

Unclear: `forcePruneRatio` is a narrow additive knob, but https://github.com/openclaw/openclaw/pull/43238 proposes `mode: "rolling"` and https://github.com/openclaw/openclaw/issues/58284 proposes turn-age pruning for the same active-session pressure. The maintainable next step is API selection before implementation.
