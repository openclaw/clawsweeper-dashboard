## Summary

Keep open. Current main still lacks the requested high-usage TTL bypass: `forcePruneRatio` is absent from config/schema/docs/runtime, and cache-TTL pruning still returns inside a fresh TTL window before estimating context usage.

## What This Changes

Request to add optional `agents.defaults.contextPruning.forcePruneRatio` so cache-TTL pruning can bypass TTL when estimated context usage crosses a configured threshold.

## Best Possible Solution

Ship an additive, unset-by-default threshold through config, schema, docs, runtime wiring, and hook behavior while preserving normal cache-TTL pruning when usage is below the threshold.

## Reproduction Assessment

Yes, from source inspection. Configure cache-TTL pruning with `lastCacheTouchAt` inside `ttlMs` and an over-threshold context; current `extension.ts` returns at the TTL gate before `pruneContextMessages` can estimate usage.

## Solution Assessment

Yes. An optional threshold is the narrowest maintainable solution for this request because it preserves normal cache-TTL behavior below high usage; rolling mode is broader and not equivalent.
