## Summary

Keep open: current main still lacks `forcePruneRatio`, the active `cache-ttl` path still returns before usage estimation during a fresh TTL window, and the related direct implementation is closed unmerged. The open rolling-mode work at https://github.com/openclaw/openclaw/pull/43238 overlaps with the pressure problem but is a different public API.

## What This Changes

Request to add optional `agents.defaults.contextPruning.forcePruneRatio` so high estimated context usage can bypass the cache TTL pruning delay, with schema and test coverage.

## Best Possible Solution

Choose one pruning trigger API, then implement it in the active `pi-hooks` context-pruning path with aligned config types/schema, generated docs, changelog, and focused runtime/config tests while preserving existing `cache-ttl` behavior.

## Reproduction Assessment

Not applicable as a bug reproduction: this is an enhancement for a new config trigger. Source inspection on current main does confirm the requested field is absent and TTL-gated pruning still exits before usage estimation in a fresh TTL window.

## Solution Assessment

Unclear. `forcePruneRatio` is a coherent additive option, but https://github.com/openclaw/openclaw/pull/43238 and https://github.com/openclaw/openclaw/issues/58284 propose different rolling or turn-age APIs for the same active-session pressure, so maintainer API selection should come before implementation.
