## Summary

Keep open: current main still has TTL-gated `cache-ttl` context pruning only, with no `forcePruneRatio` runtime, schema, docs, or regression coverage. Open PR #43238 proposes a different `rolling` mode for the same active-session pressure problem, so maintainers should choose the public trigger API before implementation.

## What This Changes

Add optional `agents.defaults.contextPruning.forcePruneRatio` so cache-TTL context pruning can bypass TTL when estimated context usage is high.

## Best Possible Solution

Maintainers should choose the pruning trigger API, then implement the selected additive surface across settings, schema/generated docs, runtime TTL gating, docs, changelog, and focused tests.

## Reproduction Assessment

Not applicable: this is a feature/config request rather than a broken existing contract. Source inspection on current main shows the requested field is absent and the hook still returns inside fresh TTL windows before usage estimation.

## Solution Assessment

Unclear: `forcePruneRatio` is a narrow additive knob, but open PR #43238 proposes a different `rolling` public API for the same pressure problem. Maintainers should settle that API before a repair PR changes schema and docs.
