## Summary

Keep open. Current main still lacks the requested high-usage TTL bypass: `forcePruneRatio` is absent from config/schema/docs/runtime, and cache-TTL pruning still returns inside a fresh TTL window before context usage is estimated; the direct implementation PR is closed unmerged and the adjacent rolling-mode PR is not equivalent.

## What This Changes

Request an additive `contextPruning.forcePruneRatio` option that lets cache-TTL context pruning bypass the TTL gate when estimated context usage meets a configured threshold.

## Best Possible Solution

Implement an unset-by-default `forcePruneRatio` through config types, Zod schema, generated docs/help, effective settings, embedded runtime setup, and the context hook so TTL is bypassed only when finite estimated usage meets the threshold.

## Reproduction Assessment

Yes. Configure cache-TTL pruning with `lastCacheTouchAt` inside `ttlMs` and an over-threshold context; current `extension.ts` returns at the TTL gate before `pruneContextMessages` can estimate usage.

## Solution Assessment

Yes. The additive threshold is the narrowest maintainable solution for this request because it preserves normal cache-TTL behavior until context usage is high; rolling mode is broader and does not provide thresholded TTL bypass.
