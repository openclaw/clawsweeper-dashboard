## Summary

Keep open: current main still lacks the requested `forcePruneRatio` config/runtime/schema/docs surface, and the remaining active-session pruning problem overlaps with a broader open rolling/turn-based pruning discussion that needs API direction.

## What This Changes

Request to add an optional `agents.defaults.contextPruning.forcePruneRatio` threshold that bypasses cache-TTL pruning when estimated context usage is high.

## Best Possible Solution

Choose one active-session pruning trigger API, then implement it in the active `pi-hooks` runtime with aligned config types, strict schema, generated docs, public docs, changelog, and focused runtime/config tests.

## Reproduction Assessment

Not applicable as a bug reproduction because this is an enhancement request for a new config option. Source inspection on current main confirms the option is absent and the cache-TTL gate returns before usage estimation.

## Solution Assessment

Unclear: `forcePruneRatio` is a coherent threshold knob, but the same active-session pruning pressure is now represented by competing rolling and turn-age API proposals, so maintainers should choose the contract first.
