## Summary

Keep open: current main still lacks `forcePruneRatio`, and the remaining work is a new public pruning trigger/config choice rather than an already-implemented cleanup. The direct implementation PRs are closed unmerged, while the open rolling-mode PR overlaps with the same active-session pressure problem but proposes a different API.

## What This Changes

The issue requests an optional `agents.defaults.contextPruning.forcePruneRatio` setting that bypasses cache-TTL pruning when estimated context usage reaches a configured threshold.

## Best Possible Solution

Maintain a single selected pruning trigger API and implement it consistently across active hook settings/factory, config schema/types, docs, changelog, and focused runtime/config tests.

## Reproduction Assessment

Not applicable: this is an enhancement/new config option rather than a broken existing contract. Source inspection on current main shows the requested field is absent and `cache-ttl` pruning still exits before usage estimation during a fresh TTL window.

## Solution Assessment

Unclear: `forcePruneRatio` is a coherent additive knob, but https://github.com/openclaw/openclaw/pull/43238 proposes a different `mode: "rolling"` trigger for the same active-session pressure problem. The safer path is maintainer API selection before implementation.
