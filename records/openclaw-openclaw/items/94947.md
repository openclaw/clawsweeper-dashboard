## Summary

Keep open. Current main and the latest release still expose Skill Workshop as a single-target contract: `create` creates from the submitted new name, `update` accepts one existing `skill_name`, and there is no merged guard or multi-target patch action for update-shaped create content. The closed candidate pull request is unmerged, and the currently open related pull request addresses a different update-apply footgun.

## What This Changes

The issue requests preventing `skill_workshop action=create` from silently creating a sibling skill when proposal content targets existing skills, or adding an explicit multi-skill patch/update path.

## Best Possible Solution

Define the public Skill Workshop contract first, then land either a scoped create/revise guard with docs and regression tests or a maintainer-approved structured patch/update API covering agent, Gateway, proposal records, and docs.

## Reproduction Assessment

Yes. Source inspection shows `action=create` writes a proposal for the submitted new skill name and does not inspect proposal content for existing skill paths; no live MCP run was performed because this cleanup review is read-only.

## Solution Assessment

Unclear. The reported gap is real, but maintainers need to choose whether the best fix is a conservative create-time guard/docs change or a broader multi-target patch API.
