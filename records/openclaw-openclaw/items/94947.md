## Summary

Current main and the latest release still have the reported Skill Workshop shape: `create` targets a new skill name, `update` targets one existing skill, and there is no current guard or multi-skill patch action for update-shaped proposal content. No canonical duplicate or merged fix was found, so this issue remains necessary.

## What This Changes

The issue asks Skill Workshop's agent/MCP tool to avoid silently creating a sibling skill when proposal content is intended to patch existing skills.

## Best Possible Solution

Define the Skill Workshop contract so `action=create` cannot silently accept clearly update-shaped proposals, or add an explicit patch/update path, then cover the agent tool, service/Gateway semantics, docs, and regression tests.

## Reproduction Assessment

Yes, from source inspection: `action=create` creates a proposal for the requested sibling name and does not inspect proposal content for existing skill path references. I did not run the live MCP flow because this review is read-only.

## Solution Assessment

No, current behavior is not the best end state because existing guidance says create is for new skills but the tool still accepts update-shaped create proposals. The maintainable fix needs a maintainer choice between a conservative reject/warn guard and a broader multi-target patch API.
