## Summary

Keep open. Current main and latest release still let `create` target the requested new sibling without scanning proposal content for existing skill paths, and https://github.com/openclaw/openclaw/pull/96011 is open with closing syntax, so this issue should remain open until that PR lands or maintainers choose a different contract.

## What This Changes

Prevent `skill_workshop action=create` from silently creating a new sibling skill when proposal content is clearly intended to update existing skills.

## Best Possible Solution

Land a shared-service create guard with docs and tool-schema coverage if maintainers accept fail-closed semantics; otherwise replace it with an explicit multi-target patch/update API decision.

## Reproduction Assessment

Yes. Source inspection on current main shows `action=create` writes a proposal for the requested new skill name and does not inspect proposal content for existing `skills/<name>/...` targets; I did not run a live MCP call in this read-only review.

## Solution Assessment

Unclear until maintainers accept the public contract. The open shared-service guard is the narrowest current candidate, while a multi-target patch API would be a larger product/API change.
