## Summary

Keep open. Current `main` and latest release still let `skill_workshop` create proposals bind only to the new sibling skill name, while the open fix candidate at https://github.com/openclaw/openclaw/pull/96011 specifically owns the shared-service guard for this issue.

## What This Changes

The issue asks Skill Workshop to stop silently creating a new sibling skill when `action=create` proposal content is clearly intended to patch existing workspace skills.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/96011 or an equivalent shared-service guard, with docs that keep multi-target patching as a separate product/API decision.

## Reproduction Assessment

Yes. Source inspection shows the current create path binds to the new sibling name and does not inspect proposal content for existing skill paths; a live repro would create an existing workspace skill and then call `skill_workshop action=create` with proposal content referencing it.

## Solution Assessment

Yes for the narrow guard shape, with maintainer confirmation. Putting the rejection in `proposeCreateSkill` covers CLI, Gateway, and agent-tool callers through the shared service, while a multi-target patch action should remain a separate product decision.
