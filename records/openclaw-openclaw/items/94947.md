## Summary

Keep open. Current main and the latest release still have a single-target Skill Workshop contract: `create` creates a new workspace skill proposal, `update` targets one existing skill, and there is no create-time guard or multi-target patch action for proposals that are clearly update-shaped.

## What This Changes

The issue requests a Skill Workshop MCP/tool behavior change so update-shaped multi-skill proposals do not silently land as a new sibling skill.

## Best Possible Solution

Define the Skill Workshop contract so `create` cannot silently accept clearly update-shaped proposals, or add an explicit structured patch/update path, then cover the agent tool, Gateway/API shape, docs, and regression tests.

## Reproduction Assessment

Yes. Source inspection shows `action=create` writes a proposal for the requested new skill name and does not inspect proposal content for existing `skills/<name>/...` targets; I did not run a live MCP flow in this read-only review.

## Solution Assessment

Unclear. The report identifies a real product gap, but maintainers need to choose whether the best fix is a conservative create-time guard/docs change or a broader multi-target patch API.
