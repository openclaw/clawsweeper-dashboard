## Summary

Keep open. Current main and v2026.6.8 still expose a single-target Skill Workshop contract: `create` targets a new sibling skill, `update` targets one existing skill, and there is no guard or patch action for update-shaped multi-skill proposal content.

## What This Changes

Request that `skill_workshop` stop silently creating a new sibling skill when proposal content is clearly meant to patch existing skills.

## Best Possible Solution

Define the Skill Workshop contract so `create` cannot silently accept clearly update-shaped proposals, or add an explicit structured patch/update path, then cover the agent tool, Gateway/API shape, docs, and regression tests.

## Reproduction Assessment

Yes. Source inspection shows `action=create` writes a proposal for the requested new skill name and does not inspect proposal content for existing `skills/<name>/...` targets; I did not run a live MCP flow because this review is read-only.

## Solution Assessment

Unclear. The report identifies a real gap, but maintainers need to choose whether the best fix is a conservative create-time warning/rejection plus docs or a broader multi-target patch API.
