## Summary

Keep open. Current main and the latest release still expose the same single-target Skill Workshop contract: `create` targets a new sibling skill, `update` targets one existing skill, and no guard or structured multi-skill patch action prevents update-shaped proposal content from landing under a new skill name.

## What This Changes

The issue requests a safer `skill_workshop` contract for proposals that are clearly meant to patch existing skills instead of creating a new sibling skill.

## Best Possible Solution

Define the Skill Workshop contract so `create` cannot silently accept clearly update-shaped proposals, or add an explicit structured patch/update path with docs and regression coverage.

## Reproduction Assessment

Yes. Source inspection shows `action=create` writes a proposal for the requested new skill name and does not inspect proposal content for existing `skills/<name>/...` targets; I did not run a live MCP flow because this review is read-only.

## Solution Assessment

Unclear. The report identifies a real gap, but maintainers need to choose whether the best fix is a conservative create-time warning/rejection plus docs or a broader multi-target patch API.
