## Summary

Keep open. Current main still exposes a single-target Skill Workshop contract: `create` targets a new workspace skill, `update` targets one existing skill, and no current guard or patch action prevents update-shaped `create` proposals from landing as unused sibling skills.

## What This Changes

The issue asks Skill Workshop to avoid silently creating a new sibling skill when an MCP `create` proposal is clearly meant to patch existing skills.

## Best Possible Solution

Define the public Skill Workshop contract so update-shaped `create` proposals cannot silently land as sibling skills, either via a reviewed guard/docs path or an explicit multi-target patch API.

## Reproduction Assessment

Yes. Source inspection shows `action=create` binds the proposal to the requested new name and does not inspect proposal content for existing `skills/<name>/...` targets; I did not run a live MCP flow because this review is read-only.

## Solution Assessment

Unclear. The report identifies a real gap, but maintainers need to choose whether the best fix is a narrow create-time guard/docs update or a broader multi-target patch action.
