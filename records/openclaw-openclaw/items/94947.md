## Summary

Keep open. Current main and v2026.6.8 still make `skill_workshop action=create` target a new sibling skill, while `action=update` accepts one existing skill; the remaining work is a maintainer product decision about guard/docs behavior versus a new multi-target patch contract.

## What This Changes

The issue asks for `skill_workshop` to avoid silently creating a new sibling skill when proposal content is intended to patch one or more existing skills.

## Best Possible Solution

Define the Skill Workshop contract so `create` cannot silently accept clearly update-shaped proposals, or add an explicit structured patch/update path, then cover the agent tool, shared service, docs, and regression tests.

## Reproduction Assessment

Yes. Source inspection shows `action=create` writes a proposal for the requested new skill name and does not inspect proposal content for existing `skills/<name>/...` targets; I did not run a live MCP flow because this review is read-only.

## Solution Assessment

Unclear. The report identifies a real gap, but maintainers need to choose whether the best fix is a conservative create-time warning/rejection plus docs or a broader multi-target patch API.
