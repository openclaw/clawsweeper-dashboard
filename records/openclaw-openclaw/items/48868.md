## Summary

Keep this PR open: the underlying request is still valid on current main, but the proposed patch is incomplete because the status/default-log-file helper would still point users at the temp-root log path when the logger writes under the new log-dir override.

## What This Changes

The PR adds `OPENCLAW_LOG_DIR` handling to the Gateway logger and `OPENCLAW_TMP_DIR` handling to the trusted temp-directory resolver.

## Best Possible Solution

Land a refreshed implementation that uses one shared default log-directory/file helper for logger and status surfaces, reads the log override after CLI dotenv hydration, keeps the existing trusted temp-root checks for the temp override, and documents the supported operator contract.

## Reproduction Assessment

Yes. Source inspection on current main is enough for the base gap, and the PR regression is reproducible by comparing the patched logger default path with the unchanged `resolveConfiguredLogFilePath()` status path when `OPENCLAW_LOG_DIR` is set and `logging.file` is unset.

## Solution Assessment

No as-is. The direction is narrow and maintainable, but the implementation needs the status/default-log helper, tests, and docs updated before it is the best fix.
