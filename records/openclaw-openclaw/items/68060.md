Full review comments:

- **[P1] Add smartExtraction to the manifest schema:** `extensions/memory-lancedb/config.ts:139`
  - body: The runtime parser now accepts `smartExtraction`, but the plugin manifest still has `additionalProperties: false` and no `smartExtraction` property. Because plugin config is validated against the manifest before runtime loading, users who enable the new option will get invalid config instead of this feature.
  - confidence: 0.95
- **[P1] Preserve capture filtering before LLM extraction:** `extensions/memory-lancedb/index.ts:633-638`
  - body: The smart-extraction branch sends every collected user text to `extractMemories` before applying `shouldCapture` or `captureMaxChars`. That bypasses the existing prompt-injection, envelope-sludge, and length gates, so opt-in users can send and potentially store content the current capture path rejects.
  - confidence: 0.94
- **[P1] Escape existing memories before adding them to the prompt:** `extensions/memory-lancedb/smart-extractor.ts:109-112`
  - body: `existingMemories` are stored user-controlled strings, but this prompt concatenates them verbatim as instructions-adjacent text. The current recall path escapes memories and labels them untrusted; the extractor should use the same untrusted framing or another delimiter-safe representation before asking the model to deduplicate.
  - confidence: 0.9
- **[P2] Make extractor failures observable:** `extensions/memory-lancedb/smart-extractor.ts:202-203`
  - body: Any extraction exception currently returns an empty result, and the caller treats that as a normal no-op. An invalid model, auth failure, or provider outage would silently disable smart auto-capture, so this path should at least log/fallback in a way users and maintainers can diagnose.
  - confidence: 0.82
