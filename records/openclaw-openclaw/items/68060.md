## Summary

Keep this PR open, but it is not merge-ready: the proposed memory-lancedb smart extraction path still has manifest/config, capture-safety, prompt-trust, observability, and proof gaps that need maintainer-directed rework.

## What This Changes

This PR adds opt-in `smartExtraction` config, a new smart extractor module, parser tests, and an auto-capture branch that sends user messages to a chat model for atomic memory extraction.

## Best Possible Solution

Keep the PR open for focused memory-lancedb rework: align manifest, runtime config, UI hints, docs, and tests; run smart extraction only after the existing capture gate; treat stored memories as untrusted prompt data; make extractor failures observable or fall back safely; and require real runtime proof before merge.

## Reproduction Assessment

Yes. The blocking issues are source-reproducible: enabling the new config would hit the strict manifest before the runtime parser, and the smart branch visibly sends raw user text and existing memories to the extractor before the current capture/trust safeguards.

## Solution Assessment

No. Single-shot extraction may be a viable opt-in direction, but this implementation is not the narrowest maintainable merge path until it preserves the existing memory-lancedb config contract, safety gate, prompt-trust boundary, and operator diagnostics.

  - body: The runtime parser now accepts `smartExtraction`, but `openclaw.plugin.json` still has `additionalProperties: false` with no matching property.   - body: The smart branch passes every collected user text to `extractMemories` before applying `shouldCapture`, while the fallback branch applies that gate below. This bypasses `captureMaxChars` and the existing prompt-injection/content filters for the new outbound LLM call and storage path.

  - confidence: 0.91
- **[P2] Surface extractor failures:** `extensions/memory-lancedb/smart-extractor.ts:202-203`
  - body: If the chat completion fails because of auth, model, base URL, or provider outage, this catch returns an empty result and the caller treats it as a normal no-op. That silently disables smart auto-capture and gives operators no warning or fallback path.
  - confidence: 0.88
