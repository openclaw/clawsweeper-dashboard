## Summary

Keep open. The PR is useful source work for opt-in LLM memory extraction, but it is not merge-ready against current main because the new config is not wired through the strict manifest, the extraction path bypasses existing capture safety gates, and stored memories are interpolated as trusted prompt text.

## What This Changes

The PR adds opt-in `smartExtraction` config, a new `memory-lancedb` smart extractor, parser tests, and an auto-capture branch that sends user messages to a chat model for atomic memory extraction.

## Best Possible Solution

Keep the PR open for maintainer-directed rework in `memory-lancedb`: preserve current config contracts, add matching manifest/docs/changelog coverage, run smart extraction only after the existing capture gate, treat stored memories as untrusted prompt data, and make extractor failures observable or fall back safely.

## Reproduction Assessment

Yes. The load-time config problem is reproducible by enabling `smartExtraction` under the plugin config: the manifest schema is strict and lacks that property, and the loader validates the manifest before plugin runtime parsing. The safety issues are reproducible by inspection of the smart branch, which passes raw `texts` to the extractor before `shouldCapture`.

## Solution Assessment

No. Single-shot extraction may be a viable opt-in feature, but this implementation is not the best merge path until it preserves current memory-lancedb config compatibility and the existing safety/trust boundaries.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Sync smartExtraction with the manifest schema:** `extensions/memory-lancedb/config.ts:139`
  - body: The runtime parser accepts `smartExtraction`, but `openclaw.plugin.json` still has `additionalProperties: false` and no matching property. Users who add the new config will fail manifest validation in the plugin loader before this parser runs.
  - confidence: 0.95
- **[P1] Gate smart extraction through shouldCapture:** `extensions/memory-lancedb/index.ts:633-637`
  - body: The smart branch sends every collected user text to `extractMemories` before applying `shouldCapture`, while the fallback branch applies the existing gate below. This bypasses `captureMaxChars` and prompt-injection/content filters for the new outbound LLM call and possible storage path.
  - confidence: 0.96
- **[P1] Escape existing memories before prompting the extractor:** `extensions/memory-lancedb/smart-extractor.ts:109-110`
  - body: `existingMemories` are persisted user-controlled strings, but the extractor prompt interpolates them verbatim as bullet text. A stored instruction-like memory can steer extraction or dedup behavior, unlike the recall path that escapes memories and marks them as untrusted.
  - confidence: 0.91
- **[P2] Surface extractor failures instead of dropping capture:** `extensions/memory-lancedb/smart-extractor.ts:202-203`
  - body: If the chat completion fails because of auth, model, base URL, or provider outage, this catch returns an empty result and the caller treats it as a normal no-op. That silently disables smart auto-capture and gives operators no warning or fallback path.
  - confidence: 0.88
- **[P2] Document the new user-facing config:** `extensions/memory-lancedb/config.ts:257-279`
  - body: This adds user-visible `smartExtraction` config and behavior, but the PR does not update the memory-lancedb docs or changelog. The new model/API/base URL knobs need public docs and a release note before merge.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The diff introduces a new outbound LLM path for user conversation text and stored memories, and the current patch bypasses established capture and prompt-trust safeguards.

Concerns:

- **[high] Capture safety bypass before outbound LLM call:** `extensions/memory-lancedb/index.ts:633`
  - body: The smart extraction branch calls the chat extractor with raw collected user text before applying `shouldCapture` or `captureMaxChars`, so content the current plugin rejects can be sent to the extraction provider and possibly stored.
  - confidence: 0.96
