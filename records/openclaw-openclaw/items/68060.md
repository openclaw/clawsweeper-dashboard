## Summary

Keep open. Current main does not already implement smart extraction, so this contributor PR is not obsolete, but the proposed patch is not merge-ready: it breaks provider-backed memory-lancedb configs, exposes a config key that the manifest still rejects, and adds a security-sensitive LLM extraction path that bypasses existing capture and prompt-trust safeguards.

## What This Changes

The PR adds opt-in smartExtraction config to the memory-lancedb plugin, a new smart-extractor module with parser tests, and an auto-capture branch that calls a chat model to decompose user messages into atomic memories.

## Best Possible Solution

Keep this PR open for maintainer-directed rework in the memory-lancedb plugin: align manifest/config/UI/docs/changelog, preserve provider-auth compatibility, run smart extraction only after the existing capture gate, escape or otherwise isolate stored memories as untrusted prompt data, make extractor failures observable or fall back safely, and add focused tests for those contracts before merge.

## Reproduction Assessment

Yes. The config regression is reproducible with the existing provider-backed embedding config shape that omits embedding.apiKey; the manifest rejection follows from validating a smartExtraction config against the current strict manifest; and the safety bypass is visible from the PR branch calling extractMemories with raw texts before shouldCapture.

## Solution Assessment

No. The single-shot extraction feature may be a viable opt-in direction, but this implementation is not the best merge path until it preserves existing config, manifest, capture-safety, and prompt-trust contracts.

Overall confidence: 0.93

Full review comments:

- **[P1] Preserve provider-backed configs without apiKey:** `extensions/memory-lancedb/config.ts:202`
  - body: This always evaluates `resolveEnvVars(embedding.apiKey)` even when `smartExtraction` is absent and `embedding.apiKey` is undefined. Current memory-lancedb configs may rely on provider auth without a plugin apiKey, so those supported configs will throw during parsing before the plugin can load.
  - confidence: 0.96
- **[P1] Sync smartExtraction with the manifest schema:** `extensions/memory-lancedb/config.ts:139`
  - body: The runtime parser now accepts `smartExtraction`, but the plugin manifest still has `additionalProperties: false` without that property. Since plugin config is validated from the manifest before runtime parsing, users who enable this option can get an invalid-config load failure before the feature runs.
  - confidence: 0.94
- **[P1] Filter texts before smart extraction:** `extensions/memory-lancedb/index.ts:633-637`
  - body: The smart branch forwards every collected user text to `extractMemories` without applying `shouldCapture(..., { maxChars: cfg.captureMaxChars })`. That bypasses the existing prompt-injection and length filters, so content current main rejects can now be sent to the extraction model and stored.
  - confidence: 0.96
- **[P1] Escape existing memories before prompting the extractor:** `extensions/memory-lancedb/smart-extractor.ts:110`
  - body: `existingMemories` are persisted user-controlled strings, but the extractor prompt inserts them verbatim as bullet text. A stored memory containing instruction-like content can steer extraction or dedup behavior, unlike the recall path that escapes memory text and labels it untrusted.
  - confidence: 0.91
- **[P2] Surface extraction failures instead of dropping capture:** `extensions/memory-lancedb/smart-extractor.ts:203`
  - body: When the chat completion fails, this catch returns an empty result and the caller treats it as a normal no-op. Auth, model, base URL, or provider outages would silently disable smart auto-capture with no operator signal or fallback.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The diff introduces a new outbound LLM extraction path for user conversation text and persisted memories, and two concrete prompt-safety regressions need fixing before merge.

Concerns:

- **[high] Capture safety bypass before outbound LLM call:** `extensions/memory-lancedb/index.ts:633`
  - body: The smartExtraction branch sends raw collected user text to the chat extractor before applying shouldCapture or captureMaxChars, so content the current plugin rejects can leave the process and may be stored.
  - confidence: 0.96
- **[high] Stored memories inserted as trusted prompt text:** `extensions/memory-lancedb/smart-extractor.ts:110`
  - body: Existing memories are user-controlled persisted data but are interpolated into the extractor prompt without escaping or an untrusted-data boundary, creating a prompt-injection route specific to smart extraction.
  - confidence: 0.91
- **[medium] Silent extractor failures obscure provider problems:** `extensions/memory-lancedb/smart-extractor.ts:203`
  - body: The extractor swallows chat-completion errors and returns an empty result, making auth, endpoint, or model failures look like normal no-op capture and delaying detection of misrouted credentials or outages.
  - confidence: 0.86
