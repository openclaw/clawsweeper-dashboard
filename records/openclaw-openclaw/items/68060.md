## Review Findings

Overall confidence: 0.92

Full review comments:

- **[P1] Sync smartExtraction with the manifest schema:** `extensions/memory-lancedb/config.ts:139`
  - body: This adds `smartExtraction` to the runtime parser/UI hints, but `openclaw.plugin.json` still has `additionalProperties: false` without that property. The loader validates entry config against the manifest before runtime config parsing, so users who enable the new option can get an invalid-config plugin load failure instead of the feature.
  - confidence: 0.96
- **[P1] Gate smart extraction with capture filters:** `extensions/memory-lancedb/index.ts:633-638`
  - body: The smart branch sends raw `texts` to `extractMemories`, while the existing path only stores text after `shouldCapture(..., { maxChars: cfg.captureMaxChars })`. That bypasses length, relevant-memory, system-like content, trigger, and prompt-injection checks before sending content to the extraction model, so apply the same safety gate to candidates before the LLM call.
  - confidence: 0.95
- **[P1] Treat existing memories as untrusted prompt input:** `extensions/memory-lancedb/smart-extractor.ts:110`
  - body: `existingMemories` are stored user-controlled data, but this prompt injects them verbatim as bullets. A stored memory containing instruction-like text can steer extraction/dedup behavior; mirror the recall path's untrusted labeling and escaping/delimiting before including memories in the extractor prompt.
  - confidence: 0.92
- **[P2] Surface extraction failures instead of dropping capture:** `extensions/memory-lancedb/smart-extractor.ts:202-203`
  - body: When the chat completion fails, this catch returns an empty memory list and the caller treats it as a normal no-op. Invalid model/auth/base URL or provider outages would silently disable smart auto-capture, so log/fallback/bubble enough detail for operators to diagnose the configured feature.
  - confidence: 0.88
