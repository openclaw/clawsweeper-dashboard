## Summary

Keep open. Current main still emits the misleading sqlite-vec warning in the linked missing-provider path, and the PR is a focused warning-text/test/changelog fix with no blocking review findings from source inspection.

## What This Changes

The PR updates memory-core manager and CLI vector-degradation warnings to distinguish sqlite-vec load errors from missing embedding dimensions, adds a focused regression test, and adds a changelog entry.

## Best Possible Solution

Land this focused branch after maintainer review and targeted validation, preserving the loadError-based sqlite-vec distinction and the missing-provider regression coverage.

## Reproduction Assessment

Yes. Configure memory indexing with vector writes enabled but no embedding provider, then index chunks in FTS-only mode; current main writes chunks with vectorReady false and no sqlite-vec loadError, so the existing helper/CLI warning misattributes the cause.

## Solution Assessment

Yes. Using `vector.loadError` to decide whether to name sqlite-vec is the narrowest maintainable fix in the current design, because sqlite-vec load failures set that field while missing-provider FTS-only writes do not.
