## Summary

Keep open. Current main still classifies explicit HTTP 429 signals as rate_limit before honoring billing payloads, and assistant error formatting still checks transient/rate-limit copy before billing copy; the TeamoRouter comment is useful external failover context but does not fix OpenClaw's shared classifier path.

## What This Changes

The issue requests clearer Moonshot/Kimi billing-error handling when zero-balance API responses arrive as HTTP 429, plus recovery behavior that avoids provider lockout.

## Best Possible Solution

Make high-confidence billing signals override generic HTTP 429 and rate-limit classification in the shared helpers, preserve ordinary 429 behavior, add focused regression tests, and only split out broader lockout behavior if it remains after the classifier fix.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists by passing HTTP 429 text containing insufficient_balance, credit balance, or exhausted-balance wording through classifyFailoverReasonFromHttpStatus, classifyFailoverReason, or formatAssistantErrorText; current main takes the rate-limit path first.

## Solution Assessment

Yes. The narrow maintainable fix is to change shared classifier and formatter precedence with regression coverage; a Moonshot-only special case or broad circuit-breaker rewrite should wait unless real payloads prove the generic matcher cannot cover it.
