## Summary

Current main still preserves the core bug: any HTTP 429 is classified as `rate_limit` before explicit billing text can win. Adjacent lockout mitigations have improved, but the Moonshot/Kimi insufficient-balance path still needs a focused classifier fix.

## What This Changes

The issue asks OpenClaw to surface Moonshot/Kimi insufficient-balance failures as billing errors and avoid provider-failure lockout behavior.

## Best Possible Solution

Preserve explicit billing signals inside Moonshot/Kimi 429 payloads while leaving true 429 rate limits classified as `rate_limit`.

## Reproduction Assessment

Yes, for the current-main classifier path: `classifyFailoverReasonFromHttpStatus(429, <billing text>)` flows to the unconditional 429 `rate_limit` branch. I did not run tests or a live Moonshot account because this review was read-only.

## Solution Assessment

No, closing would not be the best outcome because the central misclassification remains. The narrow maintainable fix is to let explicit billing classifications take precedence over the generic 429 branch and add regression coverage for both billing and real rate-limit 429 payloads.
