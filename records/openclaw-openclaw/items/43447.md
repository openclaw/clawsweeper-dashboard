## Summary

Keep open: current main still classifies explicit HTTP 429 failures as `rate_limit` before honoring billing text, so the Moonshot/Kimi zero-balance path remains valid even though related model-switching and auto-failover lockout pieces have been addressed.

## What This Changes

The issue asks OpenClaw to surface Moonshot/Kimi insufficient-funds 429s as billing failures and avoid retry or provider-lockout behavior from treating them as ordinary rate limits.

## Best Possible Solution

Add a targeted Moonshot/Kimi classifier for HTTP 429 `exceeded_current_quota_error` and insufficient-balance payloads, with regression coverage, while preserving ordinary 429 rate-limit behavior and the already-fixed command-switching safeguards.
