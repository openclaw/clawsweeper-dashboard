## Summary

Keep open: current main still lets explicit HTTP 429 status classification win over Moonshot/Kimi billing-shaped payloads, so the zero-balance path can still surface as a rate-limit failure even though the model-switching and auto-override lockout pieces are already handled.

## What This Changes

The issue asks OpenClaw to distinguish Moonshot/Kimi 429 insufficient-balance or quota payloads from temporary rate limits and avoid the resulting session lockout.

## Best Possible Solution

Add a targeted Moonshot/Kimi classifier for 429 `exceeded_current_quota_error` and insufficient-balance payloads, with regression coverage that preserves ordinary 429 rate-limit behavior.

## Reproduction Assessment

Yes, at source level: an error signal with `status: 429` still returns `rate_limit` even when the message has billing-shaped Moonshot/Kimi content. I did not verify with a live depleted Moonshot account.

## Solution Assessment

Yes: the narrow maintainable path is provider-aware or equivalent status handling for Moonshot/Kimi billing payloads, not a new fallback router or another model-switching rewrite.

## Repair Work Prompt

Fix the remaining Moonshot/Kimi 429 billing misclassification from https://github.com/openclaw/openclaw/issues/43447. Current main returns `rate_limit` for explicit HTTP 429 before billing-shaped payloads can win; Moonshot/Kimi documents `exceeded_current_quota_error` under 429 for insufficient balance/quota. Implement the smallest provider-boundary or equivalent status-handling repair so Moonshot/Kimi 429 `exceeded_current_quota_error`, suspended-insufficient-balance, and check-account-balance payloads classify as `billing`, while ordinary 429 `rate_limit_reached_error`, RPM/TPM/TPD, concurrency, and engine-overloaded cases remain rate-limit or overloaded. Prefer keeping Moonshot-specific policy in the Moonshot plugin if the hook can run before the 429 override; otherwise thread provider context through the 429 status branch narrowly. Add regression coverage for failover classification and user-facing formatting. Do not add a new config option, broad fallback router, or generic all-provider 429 billing rewrite. Add a CHANGELOG Fixes entry if the repair lane owns landing.
