## Summary

Keep open. Current main still lets HTTP 429/rate-limit handling override explicit billing signals, so a Moonshot zero-balance response containing `insufficient_balance` or `credit balance` can still surface as a rate-limit failure; the TeamoRouter comment is useful external failover context but does not fix OpenClaw's shared classifier path.

## What This Changes

The issue requests clearer Moonshot/Kimi billing-error handling when zero-balance responses arrive as HTTP 429, plus prevention of retry/UI lockout paths.

## Best Possible Solution

Make high-confidence billing signals override generic HTTP 429/rate-limit classification and assistant copy in the shared failover helpers, preserving ordinary 429 handling and adding focused regression tests.
