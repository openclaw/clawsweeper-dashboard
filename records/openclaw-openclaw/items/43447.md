## Summary

Keep open. Current main still lets generic HTTP 429/rate-limit handling override explicit billing signals, so a Moonshot zero-balance response that includes `insufficient_balance`, `credit balance`, or similar text can still be shown and cooled down as a rate-limit failure; the TeamoRouter comment is useful external failover context but does not fix OpenClaw's classifier path.

## What This Changes

The issue requests clearer Moonshot/Kimi zero-balance handling when billing failures arrive as HTTP 429, plus prevention of retry and UI lockout behavior.

## Best Possible Solution

Make high-confidence billing signals override generic HTTP 429/rate-limit classification and assistant copy in the shared failover helpers, while preserving ordinary 429 behavior and adding focused regression tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists by passing `status: 429` with `insufficient_balance`, `credit balance`, or equivalent exhausted-balance text through `classifyFailoverReasonFromHttpStatus`, `classifyFailoverReason`, or `formatAssistantErrorText`; current main takes the rate-limit path first.

## Solution Assessment

Yes. The narrow maintainable repair is a shared classifier/formatter precedence change with tests; a Moonshot-only special case or broad circuit-breaker rewrite should wait unless the real payload proves the generic matcher cannot cover it.
