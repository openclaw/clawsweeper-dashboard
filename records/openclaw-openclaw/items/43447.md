## Summary

Keep open: current main has fixed several cascade/lockout contributors, but I did not find a Moonshot-specific fix for zero-balance HTTP 429 billing responses being classified as rate_limit.

## What This Changes

The issue asks OpenClaw to classify Moonshot billing/quota failures accurately, avoid background retry lockout, and allow model switching when the active provider is failing.

## Best Possible Solution

Add a targeted Moonshot/provider-boundary classifier and regression tests once the exact zero-balance 429 payload is confirmed, while preserving the existing fallback, reset, and slug-generation safeguards already on main.

## Reproduction Assessment

No high-confidence live reproduction is available. Source inspection shows current main still classifies HTTP 429 as rate_limit before Moonshot-specific billing handling, but the issue lacks the exact raw Moonshot zero-balance payload needed for a full reproduction.

## Solution Assessment

Unclear. The safest solution is a narrow provider-specific classifier backed by a real Moonshot payload, not a broad rule that reclassifies all HTTP 429 responses with quota-like wording.
