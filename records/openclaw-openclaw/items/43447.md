## Summary

Keep open: current `main` and the latest shipped release still classify every explicit HTTP 429 as `rate_limit` before billing-shaped Moonshot/Kimi payload text can win. The related `/models` lockout and broader circuit-breaker reports are closed or partially addressed, but this narrow provider error-classification bug remains source-reproducible.

## What This Changes

The issue requests that Moonshot/Kimi insufficient-funds responses returned as HTTP 429 be surfaced as billing/credits errors instead of rate-limit errors, without trapping the user on the broken provider.

## Best Possible Solution

Add a targeted Moonshot/Kimi billing-over-429 classifier or status-precedence exception, with regression coverage that preserves ordinary 429 rate-limit behavior.

## Reproduction Assessment

Yes, at source level. Current `main` computes a billing-shaped message classification but then returns `rate_limit` for any explicit HTTP 429 before that classification can become the final reason.

## Solution Assessment

Yes. The narrow maintainable fix is a provider/status-classification change for Moonshot/Kimi billing payloads, not a broad fallback router or another model-switching rewrite.
