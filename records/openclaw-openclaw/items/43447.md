## Summary

Keep open: current main still lets explicit HTTP 429 status classification win over billing-shaped Moonshot/Kimi payloads, so the zero-balance path can still surface as a rate-limit failure. The model-switch/reset lockout pieces have largely been handled by related merged fixes, but the provider error-classification bug remains narrow and source-reproducible.

## What This Changes

The issue asks OpenClaw to classify Moonshot/Kimi zero-balance 429 responses as billing failures and avoid the resulting misleading rate-limit lockout behavior.

## Best Possible Solution

Add a targeted Moonshot/Kimi classifier for 429 exceeded_current_quota_error and insufficient-balance payloads, while preserving ordinary 429 rate-limit behavior and the existing control-plane model-switch/reset fixes.

## Reproduction Assessment

Yes, at source level. An error signal with status 429 still returns rate_limit before billing-shaped Moonshot/Kimi message content can become the final failover reason; I did not run a live depleted-account Moonshot call.

## Solution Assessment

Yes: a narrow provider-owned or equivalently scoped classifier is the maintainable fix. A new fallback router, broad generic 429 reinterpretation, or another model-switching rewrite would be wider than the remaining bug.
