## Summary

Keep open: current main has fixed several lockout contributors, but the remaining Moonshot zero-balance path still appears valid because explicit HTTP 429 status classification overrides billing text before any Moonshot-specific handling can run.

## What This Changes

The issue asks OpenClaw to classify Moonshot zero-balance HTTP 429 responses as billing, avoid background slug-generation lockups, and keep model switching usable when the active provider is down.

## Best Possible Solution

Add a targeted Moonshot/Kimi 429 billing classifier with regression coverage for the documented exceeded_current_quota_error and insufficient-balance payloads, while preserving the existing command-switching and session-memory safeguards already on main.

## Reproduction Assessment

Yes at the source level: a Moonshot 429 signal carrying the documented insufficient-balance payload will still be classified as rate_limit because the status branch wins before billing classification. I did not run a live depleted-account Moonshot call.

## Solution Assessment

Yes for the remaining work: a narrow Moonshot/provider-boundary classifier or equivalent status-handling exception is the maintainable fix. Reopening broad fallback, reset, or /models behavior is not the best path because those pieces are already separately addressed on current main.
