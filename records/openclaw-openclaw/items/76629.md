## Summary

Keep open. Current main has partial runtime-context stripping and outbound delivery sanitization, but provider-bound replay still lacks the requested transient assistant-message pruning for standalone silent replies, stream-error fallback text, and copied runtime-context/event scaffold text.

## What This Changes

The issue asks OpenClaw to remove internal runtime-context, silent/no-op, and failed assistant artifacts from provider-bound replay history before the next model call.

## Best Possible Solution

Add a generic embedded-runner replay-hygiene helper before provider conversion that drops only OpenClaw-owned nonsemantic artifacts, with focused regression tests and separate doctor/repair handling for persisted polluted transcripts.

## Reproduction Assessment

Yes. Source-level reproduction is high-confidence: construct replay history with assistant `NO_REPLY`, `[assistant turn failed before producing content]`, or copied runtime-context/event preface text, and current main has no provider-bound pruning step that removes those assistant rows.

## Solution Assessment

Yes. The requested direction is the narrow maintainable fix if implemented as replay-boundary hygiene in the embedded runner; outbound sanitizer work and historical transcript repair should remain separate.

## Repair Work Prompt

Fix issue #76629 by adding provider-bound replay hygiene in the embedded runner. Add a generic helper near `normalizeAssistantReplayContent` / `sanitizeSessionHistory` that removes only OpenClaw-owned nonsemantic artifacts from the in-memory provider replay copy: `openclaw.runtime-context` custom messages, standalone silent/no-visible-reply assistant sentinel payloads such as exact `NO_REPLY`, standalone `[assistant turn failed before producing content]`, and copied assistant text that consists only of OpenClaw runtime-context/event scaffold prefaces. Do not strip user-authored mentions or substantive assistant replies that merely discuss these strings, do not add a config flag unless maintainers require it, and do not perform startup/load-time transcript migrations in core. Add regression coverage for `normalizeAssistantReplayContent` and `sanitizeSessionHistory`; update `docs/reference/transcript-hygiene.md` and `CHANGELOG.md` because this changes user-facing replay behavior. Treat #75726/#75755 as complementary outbound sanitizer work, not the fix for this issue.
