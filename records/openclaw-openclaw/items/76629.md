## Summary

Keep open. Current main has partial replay cleanup, but it still lacks the broader transient replay filter for standalone silent assistant payloads, copied runtime-context assistant text, and pending-final-delivery recovery text before those paths can affect model or channel context.

## What This Changes

The issue asks OpenClaw to prune internal runtime/control messages and no-op assistant artifacts before provider-bound replay context.

## Best Possible Solution

Add a core replay-boundary hygiene helper for OpenClaw-owned nonsemantic artifacts, wire it into provider replay and pending-final-delivery recovery paths, and leave the broader declarative sanitizer registry to the existing RFC thread.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: construct current-main session history with a standalone assistant `NO_REPLY`, copied runtime-context assistant preface, or pending-final-delivery text containing internal envelopes and the inspected paths do not fully prune them before replay/recovery use. I did not execute a live channel/provider reproduction.

## Solution Assessment

Yes, the best narrow fix is a generic replay/recovery hygiene helper using the existing runtime-context and envelope stripping contracts, plus focused regression tests. A broad RFC-style declarative registry may be valuable later but is not required to fix this reported bug.
