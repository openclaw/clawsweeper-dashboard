Full review comments:

- **[P1] Neutralize copied closing references before body updates:** `scripts/generate-microsoft-tracker.mjs:426`
  - body: `--update-pr-body` writes the generated tracker directly into a live PR body, but rows copy raw issue/PR titles. The current tracker includes titles with closing keywords plus issue refs, so this can create misleading associations or closure semantics for unrelated items when the PR body is refreshed; sanitize refs/closing keywords or remove the body-update mode before merge.
  - confidence: 0.92
- **[P1] Use tracked priority labels instead of keyword escalation:** `scripts/generate-microsoft-tracker.mjs:153-181`
  - body: `priorityFor` scans titles and all label text for words like `security`, `auth`, and `crash`, which promotes live P1/P2 items to P0/P1 because they carry labels such as `clawsweeper:needs-security-review` or `merge-risk: security-boundary`. Prefer explicit `P0`-`P3` labels before fallback keywords so the tracker does not misorder the maintainer queue.
  - confidence: 0.9
- **[P2] Remove Azure Speech from the Teams label match:** `scripts/generate-microsoft-tracker.mjs:69-72`
  - body: `plugin: azure-speech` is treated as an MS Teams label term and the first matched category wins, so Azure Speech/TTS PRs appear in the MS Teams queue even when they have no Teams label. Move that label term to Azure only or add explicit category disambiguation.
  - confidence: 0.89
- **[P2] Add a check target for the generated tracker:** `scripts/generate-microsoft-tracker.mjs:421-422`
  - body: The script writes `MICROSOFT_TRACKER.md`, but the branch adds no package script or `--check` mode to verify that committed markdown matches generator output. That violates the scripts guide for generated artifacts and leaves the tracker able to drift after manual refreshes.
  - confidence: 0.86
- **[P2] Point committed tracker instructions at the file:** `MICROSOFT_TRACKER.md:7`
  - body: The committed tracker tells maintainers to mark rows by editing "this PR body", which is only true for the PR description and wrong once `MICROSOFT_TRACKER.md` lands as a repository file. Make the generated wording target-aware or avoid committing PR-body-specific text as a file.
  - confidence: 0.88
- **[P3] Remove the release-owned changelog entry:** `CHANGELOG.md:9`
  - body: Root policy says normal PRs should not edit `CHANGELOG.md`; release generation owns that file. Move this release-note context into the PR body or eventual squash/merge message instead of landing a manual changelog line.
  - confidence: 0.95
