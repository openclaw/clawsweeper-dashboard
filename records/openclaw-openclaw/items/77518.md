## Summary

Keep open: current main still has the CLI-only transcript fallback and WebChat deliberately avoids mirroring normal agent-run final replies, but the exact embedded-runtime loss path is not proven because Pi AgentSession is supposed to persist `message_end` through `SessionManager`. The issue is recent, concrete, and concerns documented transcript durability, so it should not be swept closed.

## What This Changes

Fix WebChat embedded-runtime sessions that stream assistant replies but later reload from a transcript missing those assistant turns.

## Best Possible Solution

Add a guarded durability fallback or repair path for embedded WebChat/main-session runs that verifies the transcript is missing the Pi assistant turn before appending through the SessionManager-compatible transcript helper, with regression coverage for both missing and non-duplicated paths.

## Reproduction Assessment

No, there is not yet a high-confidence current-main reproduction. Source inspection confirms the CLI-only fallback and WebChat no-mirror branch, but upstream Pi AgentSession should persist embedded `message_end`, so the actual missing-assistant path still needs a focused failing test or live repro.

## Solution Assessment

Unclear as proposed: blindly treating embedded runs like CLI persistence is risky because normal Pi `message_end` persistence should already happen and WebChat tests guard against duplicate mirrors. The maintainable fix is a narrow fallback with duplicate detection after proving the missing transcript condition.
