## Summary

Keep open: current main still has the CLI-only post-turn transcript persistence gate, while embedded WebChat agent-run final replies are intentionally not mirrored by `chat.send`, so the reported assistant-turn drop remains actionable and is not covered by the older #8643 transcript-repair issue.

## What This Changes

The issue reports that embedded-runtime WebChat turns stream assistant replies but fail to persist them into session JSONL because post-turn transcript persistence is gated to CLI runner traces.

## Best Possible Solution

Persist final assistant turns for embedded WebChat/main-session runs when Pi did not write them, using existing transcript primitives with duplicate protection and regression coverage for reload history.

## Reproduction Assessment

No: I did not establish a high-confidence live reproduction in this read-only pass. Source inspection shows the current CLI-only gate, embedded runner trace value, WebChat’s no-mirror agent-run branch, and the issue provides two real-world incidents.

## Solution Assessment

Yes: the best direction is a narrow embedded-run persistence fallback or shared guarded persistence path with duplicate detection. Relabeling embedded runs as `cli` would blur runner lifecycle semantics and is less maintainable.

## Repair Work Prompt

Fix the embedded-runtime WebChat assistant-turn persistence regression reported at https://github.com/openclaw/openclaw/issues/77518. Current `main` only calls `persistCliTurnTranscript` for `executionTrace.runner === "cli"`; embedded Pi runs report `runner: "embedded"`, and WebChat does not mirror normal agent-run final replies because Pi is expected to persist assistant `message_end` turns. Implement a narrow durable persistence fallback/shared helper for embedded WebChat/main-session final assistant turns when Pi did not already write them. Preserve runner semantics; do not relabel embedded runs as `cli`. Reuse existing transcript append/idempotency/latest-equivalent duplicate protection or add a narrowly scoped guard so normal Pi-owned assistant turns are not duplicated. Add regression tests proving reload/history keeps the assistant turn for the embedded path and does not duplicate normal message_end entries. Add a user-facing `CHANGELOG.md` fix entry. Stop and ask if the implementation requires a new config option or product-policy choice.
