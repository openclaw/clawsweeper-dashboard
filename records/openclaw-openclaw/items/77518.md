## Summary

Keep open: current main still has the CLI-only post-turn persistence gate, and direct WebChat deliberately avoids mirroring normal agent-run final payloads because Pi is expected to persist them. The reported incidents line up with the current source shape, but I did not establish a clean live reproduction in this read-only pass.

## What This Changes

The issue reports that embedded-runtime WebChat turns can lose assistant transcript entries because post-turn transcript persistence only runs for CLI runner results.

## Best Possible Solution

Add a narrow, duplicate-safe embedded-run assistant persistence fallback for WebChat/main-session turns only when Pi did not already write the assistant turn, with regression coverage for reload history and normal message_end non-duplication.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection shows the CLI-only persistence gate, embedded runner trace value, WebChat's no-mirror agent-run branch, and two concrete incident reports from the issue.

## Solution Assessment

Yes, the best direction is a narrow embedded-run persistence fallback or repair path with duplicate detection. Relabeling embedded runs as "cli" would blur runner semantics and is less maintainable.

Fix https://github.com/openclaw/openclaw/issues/77518 as a narrow bug fix. Current main only calls persistCliTurnTranscript for executionTrace.runner === "cli" in src/agents/agent-command.ts, while embedded WebChat agent runs rely on Pi SessionManager persistence and chat.send intentionally does not mirror final agent-run payloads.
