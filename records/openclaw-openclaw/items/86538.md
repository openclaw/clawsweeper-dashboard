## Label Justifications

- `P1`: Multiple independent released-build reports show normal user turns and multi-subagent workflows repeatedly failing before reply or completion delivery.
- `impact:session-state`: A live per-session journal lock blocks subsequent legitimate state writes until they time out.
- `impact:message-loss`: Successful subagent results can fail to reach the requesting session and require manual recovery or polling.

## Summary

The issue is still necessary. Current main and v2026.6.5 retain a 60-second acquisition timeout while a healthy owner may hold the session lock for five minutes, and recent reports show both subagent delivery loss and ordinary queued turns failing in that window. The linked closing PR is still open and intentionally limits itself to preserving fallback text plus better owner diagnostics, so this issue should remain open for that PR's disposition and the underlying lock-ownership decision.

## What This Changes

Fix session write-lock contention that blocks active session turns and subagent completion delivery, while retaining completed child output and useful lock-owner diagnostics.

## Best Possible Solution

Keep the durable fallback and owner-diagnostic improvements, then make the per-session journal lock cover only the actual mutation critical section or route every same-session write through one lifecycle-owned serializer whose wait semantics match active turn duration; track the broader lock-lifetime redesign with https://github.com/openclaw/openclaw/issues/92395 without dropping this issue's delivery-durability acceptance criteria.

## Reproduction Assessment

Yes, at source level with high confidence: shipped and current defaults permit a healthy lock owner to remain valid for five minutes while a competing writer fails after one minute, matching several independent runtime logs. This read-only review did not execute a live current-main gateway scenario.

## Solution Assessment

No, the linked PR is a useful durability and diagnostics mitigation but explicitly leaves the underlying long-held lock untouched; the best maintainable solution must also correct the lock-scope or same-session serialization boundary.
