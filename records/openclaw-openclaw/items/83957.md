## Summary

Current main and the shipped v2026.5.19 release implement the structured-only cron denial classifier that this issue asked for: ordinary assistant prose such as “I could not run …” is no longer promoted to a fatal cron denial unless embedded structured denial metadata is present.

## What This Changes

The issue reports that isolated cron marked successful runs as errors when final assistant prose mentioned an optional helper that could not be run.

## Best Possible Solution

Keep the shipped structured-only denial contract and let the separate cron tool-hydration/recovered-status reports continue independently.

## Reproduction Assessment

No for current main: the old source path was real, but current main now tests that `I could not run the requested script.` remains non-fatal prose. I did not run a live cron because this review is read-only.

## Solution Assessment

Yes. The implemented structured metadata boundary fixes the false-positive without adding prompt markers or config knobs, while preserving real denials through typed failure signals.
