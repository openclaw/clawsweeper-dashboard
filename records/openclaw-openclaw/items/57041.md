## Summary

Keep this PR open, but do not merge it as submitted. The reported cleanup-pending window is still observable on current main, yet the branch is based on older subagent list/status paths and also intersects an open quota-semantics report that needs maintainer judgment before the active-run predicate is broadened.

## What This Changes

The PR adds `cleanupCompletedAt` and `suppressAnnounceReason` checks to subagent active-run predicates/status formatting and adds regression tests for the pending-cleanup status window.

## Best Possible Solution

Rebase the fix onto current main and centralize a single cleanup-pending active/status predicate used by the subagents tool, slash-command list/info targeting, status summary, and `countActiveRunsForSession`. The final behavior should explicitly distinguish short cleanup-pending runs from stale zombie descendants and include regression tests for direct cleanup-pending runs, suppressed steer/killed runs, and quota/list behavior alongside #66187.

Yes. A source-level reproduction is a `SubagentRunRecord` with `endedAt` set, `cleanupCompletedAt` absent, no `suppressAnnounceReason`, and no pending descendants; current main formats it as `done` and excludes it from direct active-run counts.

## Solution Assessment

Unclear as submitted. The direction addresses the duplicate-orchestrator race, but the branch needs a current-main rebase onto the moved helper paths and a maintainer decision on whether cleanup-pending ended runs should count against spawn quota.
