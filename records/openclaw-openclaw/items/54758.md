## Summary

Keep open: current main still has the generic pending-request close error, and the PR targets the right path, but the branch lacks real behavior proof and appears to reformat already formatted protocol-mismatch connect errors.

## What This Changes

The PR changes `GatewayBrowserClient` close handling to flush pending Control UI RPCs with structured connect-handshake errors and adds a node unit regression test.

## Best Possible Solution

Carry the original structured connect failure, or store unformatted error info, so pending requests get `gatewayCode` and `details` without reformatting messages; then add redacted real Control UI/gateway proof before merge.

## Reproduction Assessment

Yes, by source inspection: current main stores the structured connect error and then flushes pending requests with a generic close Error. The PR-specific formatting regression is also source-reproducible by wrapping an already formatted protocol mismatch error in a new `GatewayRequestError`.

## Solution Assessment

No as submitted: reusing the captured connect error is the right fix direction, but constructing a new `GatewayRequestError` from an already formatted message is unsafe. The safer path is to carry the original error object or store unformatted `GatewayErrorInfo`.
