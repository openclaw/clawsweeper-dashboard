## Summary

Keep this PR open: current main and v2026.6.8 still reject pending Control UI RPCs with a generic gateway-closed Error, while the PR narrowly reuses the existing GatewayRequestError path and adds focused coverage. The remaining maintainer decision is whether to accept the compatibility-sensitive pending-request rejection shape change.

## What This Changes

This PR changes the Control UI GatewayBrowserClient close handler so pending RPC promises reject with GatewayRequestError when a structured connect failure is available, and adds a node test for the PAIRING_REQUIRED path.

## Best Possible Solution

Land the typed GatewayRequestError propagation after maintainers accept it as the intended rejection shape for structured Control UI connect failures, while keeping generic Error for unstructured socket closes.

## Reproduction Assessment

Yes. Source inspection shows current main stores pendingConnectError but rejects pending RPC promises with a generic Error on close, and the PR's added node test exercises that exact failed-connect path.

## Solution Assessment

Yes. Reusing the existing GatewayRequestError at the flushPending boundary is narrower than app-level message parsing because pending RPC promises otherwise never receive the structured details.
