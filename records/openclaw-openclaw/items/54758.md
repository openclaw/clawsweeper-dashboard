The branch changes Control UI WebSocket close handling so pending RPC requests reject with GatewayRequestError carrying structured connect-error details, and adds a focused unit test.

## Best Possible Solution

Resolve the branch on top of the current notifyClose-based close handler, keep generic close errors when no structured connect error exists, and land only after maintainers accept the UI pending-request Error-shape change and decide whether sibling gateway-client parity is follow-up work.

## Solution Assessment

Yes, after conflict resolution and maintainer compatibility acceptance. Reusing GatewayRequestError at the existing flushPending boundary is the narrowest UI fix because failed response frames already use that class and the visible close path already carries structured details.
