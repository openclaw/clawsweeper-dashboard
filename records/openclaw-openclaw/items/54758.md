## Summary

This PR remains useful: current main still drops structured connect-handshake details when flushing pending Control UI RPC promises, while the PR routes the existing connect error into those pending rejections with focused coverage and passing checks. The only meaningful remaining concern is maintainer acceptance of the intentional error-shape compatibility change.

## What This Changes

The PR changes Control UI WebSocket close handling so pending RPC requests receive a structured GatewayRequestError after connect-handshake failures, and adds a focused node test for the PAIRING_REQUIRED path.

## Best Possible Solution

Keep the typed GatewayRequestError propagation and land after maintainers accept that structured connect failures should replace generic gateway-closed rejections for pending Control UI RPCs.

## Reproduction Assessment

Yes, from source inspection rather than a local test run. Current main stores pendingConnectError in handleConnectFailure, then rejects pending RPC promises with a generic Error on close; the PR's added node test exercises that path.

## Solution Assessment

Yes. Reusing the existing GatewayRequestError at the flushPending boundary is narrower than app-level message parsing because pending RPC promises otherwise never receive the structured details.
