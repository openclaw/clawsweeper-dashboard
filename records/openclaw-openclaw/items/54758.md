## What This Changes

The PR changes Control UI WebSocket close handling so pending RPC requests reject with a structured `GatewayRequestError` when a connect failure already has structured details, and adds a focused regression test.

## Best Possible Solution

Rebase the branch onto current main, preserve `notifyClose` callback isolation and generic fallback behavior, then land only after maintainers accept the Control UI error-shape change or explicitly track sibling gateway-client parity.

## Reproduction Assessment

Yes, from source inspection. Current main stores structured `pendingConnectError` during connect failure and then flushes pending requests with a generic `gateway closed` Error; I did not run tests because this review is read-only.

## Solution Assessment

Yes, after rebase and maintainer compatibility acceptance. Reusing `GatewayRequestError` at the existing `flushPending` boundary is the narrowest UI fix because failed response frames already use that class and the close path already carries structured details.
