## Summary

Keep this PR open: current main still drops structured connect errors when flushing pending Control UI RPCs, and the branch is a useful narrow fix, but it is currently conflicting with main and changes the pending-request rejection Error shape.

## What This Changes

The branch changes Control UI WebSocket close handling so pending RPC requests receive a structured GatewayRequestError for connect-handshake failures, with a focused regression test.

## Best Possible Solution

Resolve the conflict by applying the structured flush error on top of the current notifyClose close handler, keep generic close behavior when no connect error exists, and land after Control UI owner acceptance of the pending-request Error-shape change.

## Reproduction Assessment

Yes. Source inspection shows current main stores pendingConnectError after a failed connect response and then rejects pending requests with a generic gateway-closed Error instead of that structured error; I did not execute tests in this read-only review.

## Solution Assessment

Yes, after conflict resolution and compatibility acceptance. Reusing GatewayRequestError at the existing flushPending boundary is the narrowest maintainable fix because normal failed response frames already use that class and onClose already receives structured connect details.
