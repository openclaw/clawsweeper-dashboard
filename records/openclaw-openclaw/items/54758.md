## What This Changes

This PR changes `GatewayBrowserClient` close handling to reject pending Control UI requests with structured connect-handshake errors and adds a node unit regression test.

## Best Possible Solution

Carry the original structured connect failure, or store unformatted error info, so pending requests preserve `gatewayCode` and `details` without reformatting messages; require redacted real Control UI/gateway proof before merge.

## Reproduction Assessment

Yes, by source inspection rather than a live run: current main stores structured connect failure details but flushes pending requests with a generic close `Error`. The PR-specific formatting defect is also source-reproducible from the stored formatted message plus the `GatewayRequestError` constructor path.

## Solution Assessment

No. The fix direction is right, but wrapping the stored `connectError` in a new `GatewayRequestError` can double-format protocol mismatch messages; carrying the original error or raw unformatted fields is safer.
