## Best Possible Solution

Preserve the original `GatewayRequestError` instance or store raw gateway error fields for the pending flush, keep ordinary closes generic, and land only after redacted real Control UI/gateway proof.

## Reproduction Assessment

Yes. Source inspection shows current main stores structured connect failure details but flushes pending requests with a generic close `Error`; this read-only review did not run the mocked Vitest repro.

## Solution Assessment

No. The close-handler target is right, but the branch should not construct a new `GatewayRequestError` from an already formatted stored error; it should preserve the original error instance or raw unformatted fields.
