## Summary

Keep this PR open: it targets a real Control UI error-propagation gap, but it is not merge-ready because the current patch can reformat stored connect errors and the PR still lacks real behavior proof.

## What This Changes

This PR changes `GatewayBrowserClient` close handling so pending Control UI RPCs receive structured connect-handshake errors, and adds a node unit regression test.

## Best Possible Solution

Carry the original structured connect failure, or store unformatted error info, so pending requests get `gatewayCode` and `details` without reformatting messages; then add redacted real Control UI/gateway proof before merge.

## Reproduction Assessment

Yes, by source inspection: current main stores the structured connect error before close, then flushes pending requests with a generic close Error. The PR-specific formatting regression is also source-reproducible from the new wrapper call and current protocol mismatch formatter.

## Solution Assessment

No as submitted: reusing the captured connect error is the right direction, but constructing a new `GatewayRequestError` from an already formatted message is unsafe. The safer path is to carry the original error object or store unformatted `GatewayErrorInfo`.
