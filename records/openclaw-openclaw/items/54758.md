## Summary

Keep open: the PR addresses a valid, source-reproducible Control UI gateway bug and I found no blocking patch defect. The main merge consideration is the intentional compatibility change from a generic pending-request close error to a structured GatewayRequestError during connect failures.

## What This Changes

The PR changes the Control UI gateway client to reject pending RPC requests with structured GatewayRequestError metadata after connect-handshake failures, and adds a focused node test for that path.

## Best Possible Solution

Land the narrow close-handler propagation with its regression test once maintainers accept the typed-error compatibility change and the required checks are green.

## Reproduction Assessment

Yes. Current main stores pendingConnectError in handleConnectFailure but the close handler rejects pending requests with a generic gateway-closed Error; the added test is a focused reproduction path, though I did not run tests in this read-only review.

## Solution Assessment

Yes. The structured error is already captured at the browser gateway-client boundary for onClose, so converting that same value into the flushPending error is the narrowest maintainable fix; changing app-gateway formatting or the protocol helpers would be a less direct workaround.
