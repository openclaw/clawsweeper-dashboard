The branch changes Control UI WebSocket close handling so pending RPC requests receive GatewayRequestError with structured connect-error details and adds a focused unit test.

## Best Possible Solution

Resolve the conflict on top of the current notifyClose close handler, keep generic close behavior when no structured connect error exists, and land after Control UI owner acceptance of the pending-request Error-shape change.

## Reproduction Assessment

Yes, after conflict resolution and compatibility acceptance. Reusing GatewayRequestError at the existing flushPending boundary is the narrowest UI fix because failed response frames already use that class and the visible close path already consumes structured details.
