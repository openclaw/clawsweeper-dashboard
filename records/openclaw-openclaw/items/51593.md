## Summary

Keep open. Current main and v2026.5.19 still leave Moonshot opted out of replay tool-call-ID sanitization, while the replay-history and outbound stream replay sanitizers only run when the provider policy enables `sanitizeToolCallIds` plus `toolCallIdMode`; related post-close evidence on https://github.com/openclaw/openclaw/issues/84730 independently points at the same Moonshot policy gap.

## What This Changes

The issue reports Moonshot/Kimi failing WhatsApp group-chat tool turns with duplicate `exec:*` tool-call IDs while DMs and other providers continue working.

## Best Possible Solution

Enable a Moonshot-safe duplicate-ID rewrite for replayed history and live outbound requests while preserving native Kimi `functions.<name>:<index>` IDs; only revisit WhatsApp session scoping if that targeted repair does not fix the request shape.

## Reproduction Assessment

Yes, at source level. Current main and v2026.5.19 both keep Moonshot opted out of the policy-gated replay duplicate-ID sanitizer, while the reporter and related validation show Moonshot rejects duplicate `exec:*` IDs.

## Solution Assessment

Yes. The narrow maintainable fix is to adapt or enable the existing occurrence-aware sanitizer for Moonshot while preserving native Kimi IDs, rather than rewriting WhatsApp session ownership first.
