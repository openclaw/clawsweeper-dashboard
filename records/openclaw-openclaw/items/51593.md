## Summary

Keep open. Current main still keeps Moonshot/Kimi outside the tool-call-ID sanitizer, so duplicated raw OpenClaw IDs such as `exec:26` can still pass through replay and live stream requests even though the shared sanitizer now preserves native Kimi IDs.

## What This Changes

The issue requests that Moonshot/Kimi tool-call IDs remain unique in WhatsApp group-chat requests instead of causing Moonshot HTTP 400 duplicate-ID failures.

## Best Possible Solution

Enable a Moonshot-safe duplicate-ID sanitization path for replay and live outbound requests, preserving native Kimi IDs while rewriting repeated raw OpenClaw IDs and adding focused regression coverage.

## Reproduction Assessment

Yes for the source-level failure path: current main resolves Moonshot without the sanitizer flags, and the replay/live request paths only rewrite duplicate tool-call IDs when those flags are present. A live WhatsApp/Kimi run was not performed.

## Solution Assessment

Yes, the narrowest maintainable fix is to activate or adapt the existing sanitizer for Moonshot now that native Kimi IDs are preserved. A broad WhatsApp session-scoping rewrite should wait for evidence that duplicates are created before the outbound replay/stream layer can repair them.
