## Summary

Keep open. Current main still disables Moonshot's transcript tool-call ID sanitizer, so duplicate raw OpenClaw IDs such as `exec:26` can still pass through replay/live outbound contexts; the merged native-Kimi-ID and retry-corruption fixes solve adjacent problems, not this remaining WhatsApp group-chat symptom.

## What This Changes

Moonshot/Kimi tool calls from WhatsApp group chats should avoid duplicate `tool_call_id` values while continuing to work like direct messages.

## Best Possible Solution

Add a Moonshot-safe duplicate-ID rewrite path for replayed history and live outbound requests that disambiguates repeated raw OpenClaw IDs while preserving native Kimi `functions.<name>:<index>` IDs.

## Reproduction Assessment

Yes for a source-level reproduction: bundled Moonshot resolves without sanitizer flags, and both replay and live outbound duplicate-ID rewrites are gated behind those flags. A live WhatsApp/Kimi run was not established in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is to adapt or activate the existing occurrence-aware sanitizer for Moonshot without mangling native Kimi IDs; a broader WhatsApp session rewrite should wait for evidence that this does not repair the reported duplicate request IDs.
