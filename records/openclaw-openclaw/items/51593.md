## Summary

Keep open. Current main still opts the bundled Moonshot provider out of tool-call ID sanitization, so duplicate raw OpenClaw IDs can still bypass the existing replay and live-stream sanitizer paths for Kimi/Moonshot.

## What This Changes

The issue reports Moonshot/Kimi rejecting WhatsApp group-chat requests because duplicate OpenClaw tool-call IDs such as `exec:26` appear in the provider request.

## Best Possible Solution

Enable a Moonshot-safe duplicate-ID sanitization path for replay and live outbound requests, preserving native Kimi IDs while rewriting repeated raw OpenClaw IDs and adding focused regression coverage.

## Reproduction Assessment

Yes for a source-level reproduction: current main resolves Moonshot without sanitizer flags, and both replay and live outbound paths only rewrite duplicate tool-call IDs when those flags are present. A live WhatsApp/Kimi run was not established in this read-only review.

## Solution Assessment

Yes, the narrowest maintainable fix is to adapt or activate the existing sanitizer for Moonshot now that native Kimi IDs are preserved. A broader WhatsApp session-scoping rewrite should wait for evidence that duplicates are created before the outbound replay/stream layer can repair them.
