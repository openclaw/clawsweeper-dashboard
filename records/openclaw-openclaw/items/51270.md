## Summary

Keep open. The reported TUI command bug is source-reproducible, but this patch is not merge-ready because it replaces the shared model-command behavior with TUI-local shortcuts and has no after-fix real behavior proof.

## What This Changes

The PR adds TUI handling and tests so `/model list` opens the model selector and `/model status` prints `state.sessionInfo` instead of passing those words to `patchSession`.

## Best Possible Solution

Route TUI `/model list` and `/model status` through the shared model command behavior, or implement an equivalent adapter, while preserving `/model <provider/model>` patching and adding focused regression tests.

## Reproduction Assessment

Yes, at source level. Current main parses `/model status` or `/model list` as non-empty TUI model args and sends them to `patchSession`, while docs and shared directive tests define them as informational commands.

## Solution Assessment

No. The PR fixes the immediate token parsing but does it through TUI-local shortcuts that drop shared agent-aware model browsing and detailed status diagnostics; the safer fix is to reuse or match the shared command path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

TUI users can get a different catalog from the documented `/model list` behavior, so this should forward to or reuse the shared command handling instead of calling the context-free TUI model selector.

Please keep the shared status response, or implement an equivalent TUI adapter, so troubleshooting output is not lost.
