## Summary

Keep open: the TUI bug is still source-reproducible on current main, but this PR would lose shared `/model list` context and `/model status` diagnostics, and the external PR still lacks real behavior proof.

## What This Changes

The PR adds TUI `/model list` and `/model status` branches plus colocated command-handler tests.

## Best Possible Solution

Route TUI `/model list` and `/model status` through the shared session-aware model command behavior, or match it exactly, with focused regression tests and redacted real TUI proof.

## Reproduction Assessment

Yes. Source review on current main shows `/model list` and `/model status` parse as `model` with args `list`/`status`, and the TUI model case sends non-empty args to `patchSession` as model refs.

## Solution Assessment

No. The PR prevents the literal fallthrough, but `/model list` bypasses the session-aware shared path and `/model status` replaces full diagnostics with one cached line.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Route `/model list` through the session-aware path:** `src/tui/tui-command-handlers.ts:295-297`
  - body: The new `args === "list"` branch calls `openModelSelector()`, whose `client.listModels()` path carries no active agent, workspace, or session entry. The shared `/model list` behavior carries that context, so multi-agent or workspace-scoped auth setups can show the wrong catalog.
  - confidence: 0.88
- **[P2] Preserve full `/model status` diagnostics:** `src/tui/tui-command-handlers.ts:298-305`
  - body: The new `status` branch reports only cached `state.sessionInfo` or `unknown`. The shared status response includes selected versus active runtime model, default model, agent, auth file, provider auth, endpoint, and API details, so this regresses troubleshooting output.
  - confidence: 0.9
