## What This Changes

The PR adds a wrapper-level NO_PROXY check in `src/agents/tools/web-guarded-fetch.ts` so `useTrustedEnvProxy` requests matching NO_PROXY use strict guarded fetch instead of trusted env-proxy mode.

## Best Possible Solution

Do not merge this wrapper-only change as the canonical fix; either keep the documented strict NO_PROXY fallback or implement a maintainer-approved fetch-guard policy with focused SSRF tests, docs, and real proxy proof.

## Reproduction Assessment

Yes from source for the review finding: current main already avoids EnvHttpProxyAgent on NO_PROXY matches and then uses the strict SSRF path, which still blocks localhost/private targets. I did not run a live proxy/local-service reproduction in this read-only review.

## Solution Assessment

No. The wrapper-level check duplicates the lower-level proxy gate and does not solve the local/private strict fallback; the real decision belongs in the fetch-guard policy or in preserving the documented strict fallback.
