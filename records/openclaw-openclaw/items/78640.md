## Summary

Keep open: current main and v2026.6.6 still use retry-only `renameWithRetry` for memory-core atomic swaps, so persistent Windows `EPERM`/`EBUSY` can still exhaust and throw. The merged cleanup in https://github.com/openclaw/openclaw/pull/92891 addresses stale temp-file accumulation after hard restarts, but it explicitly leaves normal atomic swap behavior unchanged, and the copy/unlink fallback PRs are closed unmerged.

## What This Changes

The issue requests a rollback-safe memory-core recovery path for persistent Windows `EPERM`/`EBUSY` failures during `openclaw memory index --force`, building on earlier retry and temp-cleanup work.

## Best Possible Solution

Land one focused memory-core fix that preserves the old index while recovering from persistent Windows rename/unlink failures, reusing the current reindex lock and startup sweep instead of adding config or a CLI-only workaround.

## Reproduction Assessment

Yes for source-backed reproduction: the current `openclaw memory index --force` path reaches `runSafeReindex` and `renameWithRetry`, and injected persistent `EPERM`/`EBUSY` rename failures still exhaust and throw. I did not live-run the Windows machine scenario in this read-only review.

## Solution Assessment

Yes, the best layer is the memory-core atomic swap helper shared by forced and automatic reindexing. The fix should be rollback-safe with the current reindex lock/cleanup design, not a new config option or CLI-only workaround.
