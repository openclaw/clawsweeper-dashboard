## Summary

Keep open: current main still has the retry-only implementation merged for https://github.com/openclaw/openclaw/issues/64187. The persistent Windows `EPERM` case is not handled by the current memory-core rename helper, and https://github.com/openclaw/openclaw/pull/71611 is useful context but was closed unmerged and should not be revived verbatim.

## What This Changes

Request a memory-core fallback so `openclaw memory index --force` on Windows recovers when persistent `EPERM` survives the existing rename retry loop.

## Best Possible Solution

Teach memory atomic reindex to recover from persistent Windows `EPERM` after retry exhaustion with a guarded SQLite file-family fallback, covered by regression tests and a changelog entry.
