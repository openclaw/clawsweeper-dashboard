## Summary

Keep open: current main and v2026.5.12 still use retry-only memory-core atomic reindex moves, and persistent Windows EPERM/EBUSY/EACCES still throws after the retry budget. The recent Windows comment strengthens this as the canonical open tracker for the post-retry memory reindex fallback/cleanup work.

## What This Changes

The issue requests a persistent Windows fallback for memory-core atomic reindex file swaps when retry-only rename or cleanup handling exhausts on EPERM, EBUSY, or EACCES.

## Best Possible Solution

Add a guarded memory-core fallback/recovery path for persistent Windows atomic reindex rename and cleanup failures, with regression coverage for EPERM/EBUSY recovery, sidecars, rollback, tmp cleanup, and existing orphan handling.
