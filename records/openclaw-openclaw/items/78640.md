## Summary

Keep open: current main and the latest release still have retry-only memory-core atomic reindex renames, so a persistent Windows EPERM/EBUSY/EACCES failure still throws after the retry budget. This remains the canonical open tracker for the post-retry Windows memory reindex fallback work.

## What This Changes

The issue requests a guarded copy/unlink fallback for persistent Windows EPERM failures in memory-core atomic reindex after the retry-only fix proved insufficient.

## Best Possible Solution

Add a narrow, guarded memory-core fallback for persistent Windows rename failures, with regression coverage for EPERM recovery, sidecar moves, rollback, cleanup, and a changelog fix entry.

## Solution Assessment
