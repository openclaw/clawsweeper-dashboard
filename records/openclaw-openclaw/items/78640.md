## Summary

Keep open: current main contains the merged retry-only memory reindex fix, and this issue reports persistent Windows EPERM on v2026.5.5 and v2026.5.6 after that shipped. The requested copy/unlink fallback was explicitly not chosen for SQLite swaps, so this needs maintainer review instead of cleanup closure.

## What This Changes

The issue requests a copyFile/unlink fallback for memory-core atomic reindex when Windows EPERM persists after the existing rename retry.

## Best Possible Solution

Decide the SQLite-safe recovery policy, then land a memory-core fix that handles persistent Windows EPERM without corrupting the main SQLite file family and proves it on the reported Windows path.

## Reproduction Assessment

Yes, source-level: if `fs.rename` returns EPERM for all retry attempts, current `renameWithRetry()` throws. The issue also provides concrete Windows v2026.5.6 output, but I did not run a live Windows current-main reproduction.

## Solution Assessment

Unclear. The requested fallback matches prior PR 71611 and the exec-approvals precedent, but current main intentionally shipped retry-only for SQLite swaps, so maintainers should validate fallback safety or choose a stronger handle-release approach.
