## Summary

Keep open: current main and the latest release still use retry-only rename handling for memory-core atomic reindex swaps, and the tests still assert exhaustion on persistent transient rename errors instead of a copy/unlink recovery path. The Windows reports and the follow-up comment point to a real session-memory persistence failure, not a stale or already-implemented request.

## What This Changes

The issue requests a Windows-safe copy/unlink fallback and cleanup path for memory-core atomic reindex failures that persist after bounded rename retry.

## Best Possible Solution

Add a bounded, tested memory-core file-swap helper that can recover from persistent Windows rename failures without losing the existing index, and clean up stale temp files without adding config or changing the memory plugin API.

## Reproduction Assessment

Yes for source-backed reproduction: the current `openclaw memory index --force` path reaches `runSafeReindex` and `renameWithRetry`, which still exhausts persistent `EPERM`/`EBUSY` rename failures. I did not live-run the Windows scenario in this checkout.

## Solution Assessment

Yes: fixing the memory-core atomic file swap helper is the narrowest maintainable layer because both forced indexing and automatic reindexing share that helper. The safer shape is a tested fallback that preserves the old DB on failure rather than a config flag or CLI-only workaround.
