## Summary

Keep open: current main still handles persistent Windows memory-core rename failures with bounded retry only, so a repeated `EPERM` from the live SQLite file swap still throws instead of recovering. The report is valid, but the requested copy/unlink fallback should get maintainer design review because the previously closed fallback PR was explicitly superseded by a retry-only approach for SQLite swaps.

## What This Changes

The issue asks OpenClaw to recover from persistent Windows `EPERM` during memory atomic reindex swaps, likely with a guarded copy/unlink fallback after rename retries fail.

## Best Possible Solution

Keep this as the canonical tracker and design a memory-core persistent Windows lock recovery that preserves SQLite file-family rollback semantics, whether that is a guarded fallback or a safer coordinated swap strategy.

## Reproduction Assessment

Yes at source level: current main routes forced memory reindex through `runMemoryAtomicReindex()`, and a persistent `EPERM` from `renameWithRetry()` still throws after the retry budget. I did not establish a live Windows repro in this read-only review.

## Solution Assessment

Unclear as stated: the issue identifies the right owner path, but a copy/unlink fallback should not be applied verbatim after the prior retry-only maintainer decision. The best fix is a guarded, owner-approved persistent-lock strategy with regression coverage for the SQLite base/WAL/SHM family.
