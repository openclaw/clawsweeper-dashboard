## Label Justifications

- `P1`: The original bug could break forced memory reindexing on Windows and leave users with stale or frozen memory recall.
- `impact:data-loss`: Failed reindex publication could make newly indexed memory content unavailable from persistent searchable recall.
- `impact:session-state`: The affected path owns per-agent memory index state and reporter follow-up linked failures to stuck processing sessions.

## Summary

Current main removes the reported rename-based memory reindex swap and replaces it with a database-first shadow publish path, so the Windows EPERM/EBUSY failure surface is implemented away on main; the fix is not yet in v2026.6.8.

## What This Changes

The issue asks OpenClaw to stop Windows memory reindex failures caused by retry-only live SQLite file rename/unlink swaps that can leave stale memory indexes and orphan tmp files.

## Best Possible Solution

Keep the database-first reindex implementation on main, let follow-up cleanup PRs handle any remaining shadow sidecar hardening, and ship the fix in the next release.

## Reproduction Assessment

Yes for the historical/shipped path: reporter Windows logs and the v2026.6.8 source show the retry-only rename/unlink swap that can exhaust on EPERM/EBUSY. Current main no longer has that failing path, so I did not reproduce the failure against main.

## Solution Assessment

Yes: replacing the live-file swap with a shadow database plus table publication is a cleaner fix than adding another copy/unlink fallback, because it avoids renaming the live SQLite file at all and preserves the old index on late publish failure.
