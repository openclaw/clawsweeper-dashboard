## Summary

Keep open: current main and v2026.5.12 still retry memory-core atomic reindex renames and throw after the retry budget, so the reported persistent Windows `EPERM`/`EBUSY` rename failure is not fully addressed. The later cleanup-side fix reduces orphaned tmp-file failures, but it does not add the requested safe rename fallback.

## What This Changes

The issue requests a Windows-safe fallback for persistent memory-core atomic reindex rename failures after the retry-only fix still leaves `openclaw memory index --force` failing or leaking tmp SQLite files.

## Best Possible Solution

Add a guarded memory-core persistent-lock fallback or equivalent Windows-safe swap strategy that preserves SQLite sidecars, rollback, and temp cleanup semantics, while keeping stuck-session recovery tracked separately.

## Reproduction Assessment

Yes, source-reproducible: a persistent injected `EPERM`, `EBUSY`, or `EACCES` from `fileOps.rename` still exhausts `renameWithRetry()` and throws on current main. I did not run a live Windows locked-file reproduction in this read-only pass.

## Solution Assessment

Yes with refinement: the fix belongs in `extensions/memory-core/src/memory/manager-atomic-reindex.ts`, but it should be a tested SQLite file-family fallback or equivalent safe swap rather than a direct transplant of the closed PR.
