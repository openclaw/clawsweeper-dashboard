## Label Justifications

- `P1`: The original bug broke a real Windows memory reindex workflow and could leave user memory/search state stale until the database-first fix shipped.
- `impact:data-loss`: The failed reindex path could leave newly indexed memory absent from the durable searchable index, presenting as lost recall.
- `impact:session-state`: The issue and follow-up Windows logs tied failed memory sync/reindex state to stale per-agent memory and stuck processing sessions.

## Summary

Current main and the shipped v2026.6.9-beta.1+ release path have replaced the old rename-based memory reindex swap with a database-first shadow publish, so the reported Windows EPERM/EBUSY rename failure path no longer exists.

## What This Changes

The issue asks for memory-core forced reindexing on Windows to recover from persistent EPERM/EBUSY file locks instead of failing while renaming SQLite index files.

## Best Possible Solution

Keep the database-first memory reindex implementation and let narrower follow-up cleanup such as https://github.com/openclaw/openclaw/pull/95113 harden remaining shadow-sidecar cleanup without reopening the old whole-file swap design.

## Reproduction Assessment

Yes historically: the old source path and reporter logs showed `openclaw memory index --force` reaching a rename-based SQLite swap that could exhaust EPERM/EBUSY retries. Against current main, that failing path is not reproducible because the helper and file-swap publish path are gone.

## Solution Assessment

Yes. The database-first shadow publish is a better fix than adding another Windows fallback to the old whole-file rename path because it avoids replacing or renaming the live shared agent database.
