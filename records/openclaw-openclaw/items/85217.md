## Label Justifications

- `P1`: The report describes an active memory-search workflow becoming unusable for Apple Silicon users who opt into QMD `query` mode.
- `impact:crash-loop`: The affected path is triggered by a native QMD/node-llama-cpp subprocess abort in the Metal backend.
- `impact:session-state`: The observable product impact is lost memory recall context because valid QMD search results are discarded and replaced by an empty builtin fallback.

## Summary

Keep open: current main still rejects a non-zero QMD subprocess exit before parsing captured stdout, so the reported loss of valid QMD query results remains a plausible, narrow memory-search bug rather than a duplicate of the broader Metal crash discussion.

## What This Changes

The issue asks OpenClaw to preserve valid QMD query JSON emitted before a macOS Apple Silicon Metal cleanup crash exits the subprocess non-zero.

## Best Possible Solution

Handle direct QMD search subprocess failures by accepting well-formed search JSON emitted before a non-zero exit, while preserving failures for invalid output and non-search maintenance commands.

## Reproduction Assessment

We have a high-confidence source-level reproduction path by mocking a QMD child process that writes valid JSON and then closes non-zero; no live macOS Apple Silicon reproduction was run in this read-only review.

## Solution Assessment

Yes, the narrowest maintainable fix is to salvage validated QMD search JSON on non-zero direct search exits, not to change defaults or add a CPU-only config path as the first repair.
