## Label Justifications

- `P1`: The report describes a broken QMD query memory workflow on macOS Apple Silicon where valid search results are lost and active memory can return no recall.
- `impact:session-state`: The affected path is QMD-backed memory_search recall over durable memory and session collections.
- `impact:crash-loop`: The triggering failure is a repeatable native QMD/node-llama Metal SIGABRT in a spawned process during query cleanup.

## Summary

Keep open. Current main still rejects QMD direct-search subprocesses on non-zero exit before parsing stdout, then switches to the builtin index, so this is a concrete QMD result-handling bug rather than fully covered by the broader Apple Silicon Metal tracker at https://github.com/openclaw/openclaw/issues/44202.

## What This Changes

QMD direct query/search results are discarded when the subprocess exits non-zero after producing valid JSON stdout, causing macOS Apple Silicon Metal cleanup crashes to return no memory results.

## Best Possible Solution

Carry stdout, stderr, and exit status through direct QMD search failures, accept complete valid JSON search results from cleanup-time aborts, and leave CPU/Metal policy plus process isolation to the broader Apple Silicon tracker.

## Reproduction Assessment

Yes at source level: simulate a direct qmd query/search process that writes valid JSON to stdout and then closes non-zero, and current main rejects before parsing those results. A hardware-confirming repro still needs macOS Apple Silicon with the Metal cleanup crash.

## Solution Assessment

Yes for the narrow bug: preserving QMD stdout on failed direct search commands and parsing complete valid JSON is the most targeted fix. CPU-only controls, Metal policy, and worker isolation should stay with the broader Apple Silicon memory safety work.
