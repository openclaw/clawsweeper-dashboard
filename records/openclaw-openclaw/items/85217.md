## Label Justifications

- `P1`: The report describes opted-in QMD `query` memory search becoming unusable on macOS Apple Silicon because valid recall results are discarded.
- `impact:crash-loop`: The affected path is triggered by a native node-llama-cpp/ggml Metal subprocess abort during QMD cleanup.
- `impact:session-state`: The observable product impact is lost memory recall context when QMD results are discarded and the empty builtin fallback is used.

## Summary

Keep open: current main still rejects a failed direct QMD subprocess before the existing parser can validate captured stdout, so the reported discard of valid `query` results remains a narrow memory-search bug rather than a cleanup-close candidate.

## What This Changes

The issue reports that QMD `query` mode emits valid JSON search results on macOS Apple Silicon, then aborts during Metal cleanup, and OpenClaw discards those results because the subprocess exits non-zero.

## Best Possible Solution

Handle validated QMD search JSON emitted before a non-zero direct search exit, while preserving failures for invalid output and non-search maintenance commands.

## Reproduction Assessment

Yes for source-level reproduction: a mocked direct QMD search subprocess that writes valid JSON then closes non-zero follows the current rejection path. No live Apple Silicon Metal repro was run in this read-only review.

## Solution Assessment

Yes: the narrowest maintainable fix is to salvage only well-formed QMD search JSON from failed direct search invocations, not to add a new CPU-only config path as the first repair.
