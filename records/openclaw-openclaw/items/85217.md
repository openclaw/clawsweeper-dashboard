## Label Justifications

- `P1`: The report describes an active QMD `query` memory-search workflow becoming unusable for Apple Silicon users because valid results are discarded after a native abort.
- `impact:crash-loop`: The affected path is triggered by a native QMD/node-llama-cpp Metal backend abort during subprocess cleanup.
- `impact:session-state`: Discarding valid QMD search results can remove memory recall context and replace it with an empty builtin fallback.

## Summary

Keep open: current main still rejects a non-zero direct QMD subprocess exit before parsing captured stdout, so a `qmd query` run that emits valid JSON and then aborts during Metal cleanup can still lose usable memory-search results and fall back to builtin search.

## What This Changes

Allow QMD query-mode search results to be used when stdout contains valid JSON even if Metal cleanup makes the QMD subprocess exit non-zero.

## Best Possible Solution

Add a narrow QMD search-only path that can parse captured stdout after a non-zero direct `qmd query/search/vsearch` exit and accept only well-formed result JSON, with regression coverage for valid stdout plus SIGABRT and for invalid/no-output failures.

## Reproduction Assessment

Yes, at source level: current main rejects a direct QMD child process on non-zero close before parsing captured stdout. A mock child that writes valid QMD JSON and then closes non-zero would exercise the reported loss path; live Apple Silicon QMD proof was not run here.

## Solution Assessment

Yes: the narrow maintainable fix is to salvage validated QMD search JSON on non-zero direct search exits. Adding a CPU-only QMD policy knob would be a broader support decision and should not be the first repair for this result-discard bug.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/85217. Direct QMD search results are lost when `qmd query`, `qmd search`, or `qmd vsearch` writes valid JSON to stdout and then exits non-zero, such as SIGABRT during Metal cleanup. Keep the change bounded to QMD search subprocess handling in the memory host/manager path: expose captured stdout/stderr with non-zero exit metadata or add a search-specific runner so `QmdManager` can attempt `parseQmdQueryJson` only for direct search commands. Accept only well-formed QMD result arrays or existing no-results markers, and preserve fail-closed behavior for empty/invalid/truncated output, timeouts, spawn errors, update/embed/status/collection commands, and the mcporter path unless a focused test covers that path. Add regression tests for valid JSON plus non-zero close, invalid stdout plus non-zero close, and fallback-manager behavior when salvage is not possible. Add a concise changelog fix entry if this becomes a user-facing repair PR.
