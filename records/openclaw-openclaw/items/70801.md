## Summary

Keep open. Current main still uses the kickstart-first macOS LaunchAgent restart path, and the focused tests still assert that the normal restart path does not bootout/bootstrap before starting; the related shipped supervision-state work references this issue but does not implement the requested two-phase ordering.

## What This Changes

Request a macOS LaunchAgent restart sequence that stops/bootouts, waits for the previous process to exit, then bootstraps/kickstarts to avoid a restart port race.

## Best Possible Solution

Make an explicit maintainer decision on the macOS LaunchAgent restart contract; if accepted, implement the ordering change in the launchd owner module while preserving detached self-restart and installed-but-unloaded recovery with focused unit plus live macOS proof.

## Reproduction Assessment

Unclear for a high-confidence live reproduction because the reported failure depends on launchd and port timing. The acceptance gap is reproducible by inspection: current main still starts with `kickstart -k` before any explicit stop-and-wait sequence, and tests assert that behavior.

## Solution Assessment

Unclear pending maintainer judgment. The proposed two-phase sequence directly targets the race, but prior history shows both bootout/bootstrap and kickstart-first semantics have caused regressions, so the safest path needs launchd owner review and live macOS validation.
