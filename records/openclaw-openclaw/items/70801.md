## Summary

Keep open. Current main still preserves the kickstart-first macOS LaunchAgent restart contract, and the focused tests still assert that the healthy restart path does not bootout/bootstrap before starting. The adjacent merged supervision-state work references this issue but covers installed-but-unloaded recovery, not the requested two-phase stop/wait/start ordering.

## What This Changes

Request a macOS `restartLaunchAgent` refactor that stops the LaunchAgent, waits for the previous process to exit, then bootstraps and starts the service with ordering and detached-handoff coverage.

## Best Possible Solution

Make an explicit maintainer decision on the macOS LaunchAgent restart contract. If accepted, implement the ordering change in the launchd owner module while preserving detached self-restart and installed-but-unloaded recovery, with focused unit coverage plus live macOS validation.

## Reproduction Assessment

Unclear for a high-confidence live reproduction because the reported failure depends on launchd and port timing. The acceptance gap is reproducible by inspection: current main still starts with `kickstart -k` before any explicit stop-and-wait sequence, and tests assert that behavior.

## Solution Assessment

Unclear pending maintainer judgment. The proposed two-phase sequence directly targets the race, but prior history shows both bootout/bootstrap and kickstart-first restart semantics have caused regressions, so the safest route needs launchd owner review and live macOS proof.
