## Summary

Keep open. Current main still uses the kickstart-first macOS LaunchAgent restart path, and the focused unit coverage continues to assert that the healthy restart path does not bootout or bootstrap before starting. The adjacent merged supervision-state work references this issue but only covers installed-but-unloaded recovery, not the requested two-phase ordering change.

## What This Changes

Request a macOS `restartLaunchAgent` refactor to stop the LaunchAgent, wait for the previous process to exit, then bootstrap and kickstart the service with explicit tests for ordering and detached handoff behavior.

## Best Possible Solution

Make an explicit maintainer decision on the macOS LaunchAgent restart contract. If accepted, implement the ordering change in the launchd owner module while preserving detached self-restart and installed-but-unloaded recovery, with focused unit tests plus live macOS validation for normal restart, self-restart, disabled-state recovery, stale listener cleanup, and unloaded-service recovery.

## Reproduction Assessment

Unclear for a high-confidence live reproduction because the reported failure depends on launchd and port timing. The acceptance gap is reproducible by inspection: current main still starts with `kickstart -k` before any explicit stop-and-wait sequence, and tests assert that behavior.

## Solution Assessment

Unclear pending maintainer judgment. The proposed two-phase sequence directly targets the race, but prior history shows both bootout/bootstrap and kickstart-first restart semantics have caused regressions, so the best fix needs launchd owner review and live macOS proof.
