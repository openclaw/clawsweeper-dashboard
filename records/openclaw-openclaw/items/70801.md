## Summary

Keep open. Current main and the latest shipped release still use the kickstart-first macOS LaunchAgent restart path, while recent merged launchd work fixed adjacent stop/repair cases without implementing this issue's requested two-phase restart ordering.

## What This Changes

Request `restartLaunchAgent` to stop the macOS LaunchAgent, wait for the previous process to exit, then bootstrap/kickstart the new instance with explicit ordering coverage.

## Best Possible Solution

Have maintainers decide the macOS launchd restart contract; if accepted, implement the two-phase stop/wait/start sequence in `src/daemon/launchd.ts` while preserving detached handoff and recent supervision recovery behavior.

## Reproduction Assessment

No, not as a high-confidence live runtime reproduction. Source inspection does show the relevant ordering on current main and the latest release: normal restart reaches `kickstart -k` before any explicit `bootout` or previous-process wait.

## Solution Assessment

Unclear pending maintainer judgment. The proposed two-phase sequence directly targets the race, but the current kickstart-first path was introduced to avoid launchd unload regressions and is covered by tests, so it needs owner review and macOS validation.
