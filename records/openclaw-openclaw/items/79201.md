## Summary

Keep open: the PR targets a real auth/transport bug, but the proposed identity check does not cover the current PI SDK default stream wrapper that OpenClaw actually receives from `createAgentSession`. The protected `maintainer` label also makes this explicit maintainer-handled work rather than cleanup-close material.

## What This Changes

The PR adds an API-provider identity check in embedded stream resolution, a regression test for `getApiProvider(...).streamSimple`, and a changelog fix entry.

## Best Possible Solution

Handle the PI SDK-created default stream explicitly, or add a small PI/OpenClaw seam that marks it as default, so supported embedded runs use boundary-aware transports while explicit provider-owned and custom session streams still win.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main creates a PI session with the SDK default wrapper, then the resolver treats that wrapper as `session-custom` because it is not `undefined` or the top-level `streamSimple`. I did not run the live provider path in this read-only review.

## Solution Assessment

No. The PR recognizes `getApiProvider(api)` stream identities, but the current embedded-runner path receives PI coding-agent's anonymous default wrapper, so the main failing path likely remains unchanged.
