## Summary

Keep open. Current main still lacks per-agent dreaming configuration, and the supplied GitHub context shows open same-author PR #67420 uses closing syntax for this issue, so the issue should remain open until that PR is reviewed, landed, or closed.

## What This Changes

Add per-agent memory-core dreaming controls so operators can disable or schedule dreaming per agent instead of relying on one global sweep for all workspaces.

## Best Possible Solution

Keep this issue open while #67420 is reviewed; land a documented, schema-backed `agents.list[].dreaming.enabled` contract first and explicitly split or design per-agent schedules.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: the proposed `agents.list[].dreaming` shape is absent from the type/schema, while the resolver and runtime still process all configured agent workspaces.

## Solution Assessment

Unclear for the full request. Per-agent enablement is the narrow maintainable first step, but custom per-agent schedules likely need an explicit multi-cron ownership design rather than a resolver-only change.
