## Summary

Keep open. Current main still has only global memory-core dreaming configuration, and the paired open PR #67420 has not landed.

## What This Changes

Add per-agent memory-core dreaming controls so operators can disable dreaming or set schedules for individual agents instead of relying on one global sweep.

## Best Possible Solution

Land a documented, schema-backed `agents.list[].dreaming.enabled` opt-out wired through the resolver and tests, then split per-agent schedules into an explicit multi-cron design if maintainers keep that scope.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: the proposed `agents.list[].dreaming` shape is absent from the type and strict schema, while the resolver/runtime still process all configured agent workspaces.

## Solution Assessment

Unclear for the full request. Per-agent enablement is a narrow maintainable first step, but custom per-agent schedules need explicit ownership of multiple cron jobs rather than a resolver-only change.
