## Summary

Keep open. Current main still lacks schema, type, docs, and runtime support for `agents.list[].dreaming`, and open PR #67420 still references this issue with closing syntax. The discussion includes concrete multi-agent resource-pressure examples, so the maintainer path is to review the linked PR and decide whether per-agent schedules stay in this issue or split out.

## What This Changes

The issue requests per-agent memory-core dreaming overrides, including opt-out and custom schedules, so multi-agent setups can reduce memory pressure and stagger dreaming work.

## Best Possible Solution

Keep this issue open while #67420 is reviewed. The desired end state is a documented, schema-backed `agents.list[].dreaming` contract at least for `enabled`, wired through config validation, generated schema metadata, resolver behavior, docs, and tests; per-agent schedules should either be explicitly modeled as multiple scoped cron jobs or split into a narrower design follow-up.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: add the proposed `agents.list[].dreaming` shape and the strict schema/type surface has no field for it, while the resolver and runtime continue to process all configured agent workspaces.

## Solution Assessment

Unclear for the full request. Per-agent `enabled` is the narrowest maintainable first step, but custom per-agent schedules need explicit cron ownership design rather than being assumed as part of the same resolver change.
