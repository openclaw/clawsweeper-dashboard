## Summary

Keep open. Current main still only exposes global memory-core dreaming configuration, automatic dreaming still fans out across configured agent workspaces, and the issue remains paired with open implementation PR #67420, so the maintainer path is to review or close that PR rather than close this issue early.

## What This Changes

The issue requests per-agent memory-core dreaming enablement and custom schedules so operators can disable or stagger dreaming for specific agents.

## Best Possible Solution

Add a documented, schema-backed `agents.list[].dreaming` contract at least for `enabled`, thread it through config types/generated schema and `resolveMemoryDreamingWorkspaces`, and cover it with resolver/runtime/docs tests. If custom schedules remain in scope, model multiple scoped cron jobs explicitly or split scheduling into a separate design issue.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: add the proposed `agents.list[].dreaming` shape and the strict agent schema/type surface has no field for it, while the workspace resolver and runtime continue to process all configured agent workspaces.

## Solution Assessment

Unclear for the full request. Per-agent `enabled` is the narrowest maintainable first step, but per-agent custom schedules need explicit cron ownership design rather than being assumed as part of the same small resolver change.
