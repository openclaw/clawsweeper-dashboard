## Summary

Keep open. Current main and the latest release still expose only global memory-core dreaming configuration, while the same-author implementation PR that closes this issue remains open and conflicting, so this should stay tracked for maintainer PR or successor review.

## What This Changes

Request per-agent memory-core dreaming controls so operators can disable or schedule dreaming per agent instead of running one global sweep across every configured workspace.

## Best Possible Solution

Keep this issue open while https://github.com/openclaw/openclaw/pull/67420 or a successor defines a schema-backed per-agent dreaming contract and decides whether per-agent schedules belong in the same change or a follow-up.

## Reproduction Assessment

Yes for the feature gap by source inspection: current main has global dreaming config only, strict agent config has no `agents.list[].dreaming`, and the resolver still includes every configured agent. I did not run a live OOM scenario because the central missing behavior is a new config surface.

## Solution Assessment

Unclear for the full requested scope. A per-agent enabled opt-out is a narrow plausible path, but custom per-agent schedules require maintainer product judgment about multiple scoped cron jobs or a separate follow-up.
