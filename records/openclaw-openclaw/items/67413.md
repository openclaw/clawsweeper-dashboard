## Summary

Keep open. Current main and the latest release still expose global-only memory-core dreaming configuration and sweep all configured agent workspaces, while the linked same-author implementation PR is open, conflicting, and still needs maintainer API/config-scope review.

## What This Changes

Request per-agent memory-core dreaming controls so operators can disable or reschedule dreaming for selected agents instead of running one global sweep across all configured workspaces.

## Best Possible Solution

Keep this issue open until maintainers land or replace the linked PR with a documented, schema-backed per-agent opt-out or allowlist, and split per-agent schedules if they are not accepted in the first fix.

## Reproduction Assessment

Yes, at source level: configure multiple agents, enable memory-core dreaming, and current main resolves every configured agent workspace before looping through the full workspace list. I did not run a live cron sweep because this review is read-only and such a run writes memory artifacts.

## Solution Assessment

Unclear. A per-agent opt-out or global allowlist is a plausible minimum fix, but the permanent config location, default fallback behavior, and per-agent schedule scope need maintainer product/API review.
