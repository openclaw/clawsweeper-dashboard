## Label Justifications

- `P2`: This was a bounded session cleanup regression that reported success while leaving stale/capped session-state entries in place, without an immediate crash loop or security bypass.

## Summary

Close: current main routes non-dry-run session cleanup through the lifecycle mutation path, so `--enforce` now runs store maintenance for stale/cap cleanup even when no `--fix-*` removals are present.

## What This Changes

`openclaw sessions cleanup --enforce` should apply previewed `prune-stale` and `cap-overflow` actions even when `--fix-missing` or `--fix-dm-scope` do not mutate the store.

## Best Possible Solution

Keep the current lifecycle-seam implementation on main, ship it in the next release, and leave the separate model-run retention policy to https://github.com/openclaw/openclaw/issues/88632.

## Reproduction Assessment

Yes for affected beta/stable builds: the issue body and June 11 confirmation give a concrete cap-overflow path where dry-run planned cleanup but apply left the store unchanged. I did not establish a failing current-main path because current source now routes apply through forced storage maintenance.

## Solution Assessment

Yes. The current storage-owned lifecycle mutation is a better fix than making CLI callbacks duplicate stale/cap pruning, because it keeps entry removal, final referenced-session cleanup, and maintenance reporting in one store operation.
