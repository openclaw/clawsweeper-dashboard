## Label Justifications

- `P2`: The report was a normal-priority session cleanup bug with silent state maintenance failure but no crash loop or security boundary.
- `impact:session-state`: The affected behavior is persisted session cleanup and stale/capped session state remaining in the store.

## Summary

Close as implemented: current main and the v2026.6.6 release no longer have the reported result-count skip. The cleanup apply path calls `updateSessionStore` without `skipSaveWhenResult`, so store maintenance can prune/cap even when the inline `--fix-*` callback returns zero.

## What This Changes

`openclaw sessions cleanup --enforce` should apply previewed `prune-stale` and `cap-overflow` maintenance even when no `--fix-missing` or `--fix-dm-scope` mutation occurs.

## Best Possible Solution

Keep the shipped writer-backed cleanup behavior for cap/stale enforcement, and track model-run retention separately in the related lifecycle issue.

## Reproduction Assessment

No, not against current main: the reported no-op gate is gone because this cleanup call does not opt into `skipSaveWhenResult`, and the writer still runs maintenance after a zero callback result. The original affected-release transcript and June 11 confirmation were concrete enough to trace the old failure mode.

## Solution Assessment

Yes. The current design solves this at the shared store-writer layer, so cleanup does not need to duplicate prune/cap logic inside the CLI callback.
