## Label Justifications

- `impact:security`: The requested permission change affects the sandbox boundary and must preserve read-only skill instruction roots.

## Summary

Current `main` still treats `workspaceAccess: "none"` as read-only at both the Docker mount and filesystem-bridge policy layers, so the reported regression remains. The prior fix PRs were closed unmerged, while the later merged skill-integrity hardening intentionally retained a read-only whole-workspace assumption for `none`; changing that assumption now requires an explicit maintainer security/product decision and a replacement design that keeps skill roots immutable.

## What This Changes

Make the per-scope sandbox workspace writable for `workspaceAccess: "none"` without exposing the main agent workspace.

## Best Possible Solution

Approve and implement `none` as a writable per-scope sandbox workspace with no main-workspace mount, while adding explicit read-only overlays and bridge rules for all existing materialized skill roots and aligning tool, remote-bridge, and memory-flush write gates.

## Reproduction Assessment

Yes. Current main deterministically marks the Docker `/workspace` bind and filesystem-bridge workspace target writable only for `rw`, while `none` uses the isolated sandbox workspace and therefore fails write operations by construction.

## Solution Assessment

Not yet. The intended isolated-writable semantics are the right product direction to evaluate, but the earlier one-line-style fixes are incomplete after the merged skill-root hardening; the best fix must preserve immutable skill mounts across every filesystem path.

## Risks / Open Questions

- Making `none` writable without narrower protected mounts could let sandboxed agents rewrite materialized skill instructions, undoing the security guarantee added by https://github.com/openclaw/openclaw/pull/85591.
- Changing only the Docker bind would leave OpenClaw file tools, remote bridges, memory flush gating, or other callers inconsistent because several layers independently treat only `rw` as writable.
- The adjacent open request https://github.com/openclaw/openclaw/issues/59718 concerns writable custom host binds under `none`; it should remain a separate least-privilege policy decision rather than being implicitly broadened by this fix.
