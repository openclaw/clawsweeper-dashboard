## Label Justifications

- `P1`: The report describes a sandbox regression that can break write-heavy skills and agent workflows for users who select isolated `none` workspaces.
- `impact:session-state`: The bug affects per-session sandbox workspace state and whether agent writes, memory, outputs, and intermediate files can persist inside that workspace.
- `impact:security`: The issue is about sandbox filesystem isolation and the boundary between no main-workspace access and writable isolated workspace access.

## Summary

Keep open: current main and the latest release still mount the isolated `workspaceAccess: "none"` sandbox workspace read-only, and the filesystem bridge still permits writes only for `"rw"`; the remaining fix is security-sensitive and needs maintainer review.

## What This Changes

The issue asks for `workspaceAccess: "none"` sandbox sessions to hide the main workspace while keeping their isolated `/workspace` writable.

## Best Possible Solution

Keep this as the canonical tracker, have sandbox/security maintainers confirm the `none` contract, then implement one consistent policy across Docker/browser mounts, local and remote fs bridges, tool gating, memory/skill behavior, regression tests, and docs.

## Reproduction Assessment

Yes, source-level. Current main selects an isolated sandbox workspace for `none`, then mounts it read-only and rejects bridge writes unless access is `rw`; I did not run a live Docker sandbox in this read-only review.

## Solution Assessment

Yes at the direction level, pending maintainer confirmation of the security contract. A safe fix should not be a one-line mount suffix change; it should align mount metadata, both bridge paths, tool exposure, docs, and tests while preserving read-only skill and host-workspace boundaries.
