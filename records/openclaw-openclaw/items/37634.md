## Summary

Keep open: current main and the latest release still make the isolated `workspaceAccess: "none"` workspace read-only in Docker, local bridge, and remote bridge paths, while the exact fix remains an open PR needing review and proof.

## What This Changes

The issue asks for `workspaceAccess: "none"` to hide the host workspace while keeping the isolated sandbox `/workspace` writable.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/46300 or a focused replacement that treats only `ro` as read-only while keeping `none` confined to the isolated sandbox workspace across Docker, browser, local FS bridge, remote FS bridge, docs, and regression coverage.

## Reproduction Assessment

Yes, at source level. Current main maps `workspaceAccess: "none"` to an isolated sandbox workspace but then marks Docker, local bridge, and remote bridge write paths read-only.

## Solution Assessment

Yes on direction, but the repository is not fixed yet. The narrow maintainable solution is to treat only `ro` as read-only while preserving `none` isolation and bridge/path-safety checks across all sandbox backends.
