## Summary

Keep open: current main and v2026.5.12 still leave `workspaceAccess: "none"` read-only through the managed `/workspace` mount and both filesystem bridges, and the remaining fix path is already covered by an open implementation PR plus a protected adjacent sandbox write-tool policy PR.

## What This Changes

Make `workspaceAccess: "none"` keep the real agent workspace hidden while leaving the isolated sandbox `/workspace` writable.

## Best Possible Solution

Have maintainers choose the canonical `workspaceAccess: "none"` policy, then land or replace https://github.com/openclaw/openclaw/pull/46300 with a complete fix across managed mounts, local and remote bridges, tool exposure, memory/docs/tests, and reconcile https://github.com/openclaw/openclaw/pull/73385 with that decision.

## Reproduction Assessment

Yes, source-level. Current main maps `none` to an isolated sandbox workspace, but the managed mount and both filesystem bridges still allow writes only for `rw`; I did not run a live sandbox in this read-only review.

## Solution Assessment

Unclear until maintainers resolve the sandbox policy conflict. The writable-isolated-workspace direction is a narrow code fix if accepted, but it must be aligned with write-tool exposure, docs, memory behavior, and bridge parity.
