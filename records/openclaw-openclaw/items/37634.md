## Label Justifications

- `P1`: The current default isolated mode prevents filesystem-dependent skills and tools from writing session outputs, with multiple users reporting materially broken workflows.
- `impact:security`: The requested behavior changes a sandbox write boundary and must preserve the read-only protection of executable skill instructions.
- `impact:session-state`: The failure prevents the per-session isolated workspace from storing normal intermediate files, outputs, and other session-local state.

## Summary

Current main still makes `workspaceAccess: "none"` read-only across the managed workspace mount and filesystem bridge, so the reported behavior remains; however, docs and later merged skill-integrity hardening intentionally preserve that restriction, making the correct writable-isolation design a security and product decision rather than a safe automated fix.

## What This Changes

Allow a sandbox with `workspaceAccess: "none"` to write within its per-session isolated workspace without exposing the main agent workspace.

## Best Possible Solution

Approve an explicit isolated-but-writable `none` contract, then apply it consistently across every sandbox backend and file-tool bridge while preserving read-only overlays for all eligible skill instruction roots and documenting the security boundary.

## Reproduction Assessment

Yes. On current main, both the Docker mount builder and filesystem bridge mark the workspace writable only for `rw`, providing a high-confidence source reproduction of read-only `none`.

## Solution Assessment

Not yet. The requested isolated-but-writable outcome is reasonable, but the earlier fixes are incomplete against the subsequently merged skill-integrity model; the best fix must retain protected read-only submounts and align every backend and bridge.
