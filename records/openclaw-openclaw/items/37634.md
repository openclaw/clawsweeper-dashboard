## Summary

Keep open: current main and the latest release still treat `workspaceAccess: "none"` as read-only in the managed sandbox mount and local/remote write bridges, and the open fix path still needs maintainer security-policy review rather than cleanup closure.

## What This Changes

The issue requests that `workspaceAccess: "none"` hide the host workspace while keeping the isolated sandbox `/workspace` writable.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/46300 with a maintainer-approved fix that makes the chosen writable-`none` policy consistent across Docker/browser mounts, local and remote bridges, tool exposure, docs, tests, and redacted runtime proof.

## Reproduction Assessment

Yes, by source inspection. Current main maps `none` to an isolated sandbox workspace, but the managed mount plus local and remote bridges still allow writes only for `rw`; I did not run a live Docker or OpenShell sandbox in this read-only review.

## Solution Assessment

Unclear until maintainers resolve the sandbox policy. The narrow implementation is to make only `ro` read-only while preserving host isolation, but the final solution must align tool exposure, remote backend parity, docs, and security expectations.
