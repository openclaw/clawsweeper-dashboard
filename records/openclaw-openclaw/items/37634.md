## Summary

Keep open: current main still makes the isolated `workspaceAccess: "none"` workspace read-only through Docker mount construction and both filesystem bridge write checks, while open PR #46300 already targets this exact behavior.

## What This Changes

Request is to keep the host workspace hidden for `workspaceAccess: "none"` while making the isolated sandbox `/workspace` writable.

## Best Possible Solution

Review and land #46300, or an equivalent narrow patch, with aligned Docker mounts, local and remote FS bridge write policy, docs, and regression tests.

## Reproduction Assessment

Yes by source inspection, not by a live Docker run. Current main maps `workspaceAccess: "none"` to the sandbox workspace but marks it writable only when access is `"rw"`, matching the reported read-only `/workspace` failure.

## Solution Assessment

Unclear until maintainer policy review. The narrow requested direction is plausible, but current docs/tests also encode the read-only behavior, so the maintainable fix must update the full sandbox contract and validate #46300 or equivalent.
