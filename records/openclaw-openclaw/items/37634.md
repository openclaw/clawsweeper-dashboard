## Summary

Keep open: current main and the latest release still make the isolated `workspaceAccess: "none"` workspace read-only in the Docker mount and filesystem bridges, and the active fix path needs maintainer security-policy review rather than cleanup closure.

## What This Changes

The issue asks for `workspaceAccess: "none"` to keep the real agent workspace hidden while making the per-session sandbox `/workspace` writable.

## Best Possible Solution

Have maintainers choose the canonical `workspaceAccess: "none"` policy, then land or replace the open fix with a complete Docker/browser, local bridge, remote bridge, docs, tool-surface, and test update backed by redacted runtime proof.

## Reproduction Assessment

Yes, source-level. Current main maps `none` to an isolated sandbox workspace, but the managed mount plus local and remote bridges still allow writes only for `rw`; I did not execute a live sandbox in this read-only review.

## Solution Assessment

Unclear until maintainers resolve the sandbox policy conflict. The narrow implementation is to make only `ro` read-only while preserving host isolation, but the final behavior must align with tool exposure, remote backend parity, docs, and security expectations.
