## Label Justifications

- `impact:security`: The requested behavior changes how the sandbox boundary grants write access while keeping host workspace isolation intact.

## Summary

Keep open: current `main` still makes `workspaceAccess: "none"` use an isolated sandbox workspace but treats that `/workspace` mount and both filesystem bridges as read-only, so the reported write failure remains source-reproducible and needs maintainer security/product review rather than cleanup closure.

## What This Changes

The issue asks OpenClaw to keep the main agent workspace hidden for `workspaceAccess: "none"` while allowing writes inside the isolated per-session sandbox workspace.

## Best Possible Solution

Have maintainers confirm the sandbox contract, then land one coordinated core change where `none` keeps the main workspace hidden, makes only the isolated sandbox `/workspace` writable, leaves `ro` read-only, protects skill/source mounts, and updates Docker, browser, local/remote filesystem bridges, docs, and regression tests together.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: configure sandboxing with `workspaceAccess: "none"`, then any write under `/workspace` is blocked because the Docker mount, local bridge, and remote bridge all allow writes only for `rw`. I did not run a live Docker repro because this review is read-only.

## Solution Assessment

Unclear until maintainer review: isolated-writable `none` is a plausible fix for the reported regression, but the best implementation must reconcile current no-write tests/docs with sandbox security and skill immutability rules.
