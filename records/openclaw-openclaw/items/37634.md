## Summary

Keep this open: current main still makes the isolated `workspaceAccess: "none"` workspace read-only across the Docker/browser mount and both filesystem bridge paths, and the related fix PR #46300 is still open rather than merged.

## What This Changes

The issue asks for `workspaceAccess: "none"` to keep the main workspace hidden while allowing writes inside the per-session sandbox workspace.

## Best Possible Solution

Keep this focused tracker open and land or supersede #46300 with a complete fix across Docker/browser mounts, local and remote filesystem bridges, docs, and regression tests while preserving host-workspace isolation for `none` and read-only semantics for `ro`.

## Reproduction Assessment

Yes. Source inspection of current main shows `none` selects the isolated sandbox workspace and then the mount and filesystem bridges still mark that workspace read-only or reject writes unless access is `rw`.

## Solution Assessment

Yes in direction, but current main is not fixed. The narrow maintainable solution is to make only the isolated `none` workspace writable while keeping the main workspace unmounted and preserving `ro` as read-only.
