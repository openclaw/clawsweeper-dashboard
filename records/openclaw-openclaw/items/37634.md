## Summary

Keep open: current main and the latest release still treat `workspaceAccess: "none"` as read-only in the shared Docker/browser mount helper and local/remote filesystem bridges. An open fix PR targets this behavior, but it has not landed and the remaining change needs sandbox security review.

## What This Changes

The issue asks for `workspaceAccess: "none"` to keep the main workspace hidden while leaving the isolated per-session sandbox workspace writable.

## Best Possible Solution

Keep this issue open until a focused fix or replacement for https://github.com/openclaw/openclaw/pull/46300 makes only `ro` read-only across mounts and bridges, updates tests/docs, and includes sandbox runtime proof plus security review.

## Reproduction Assessment

Yes. Source inspection on current main shows `none` selects an isolated sandbox workspace, but the mount helper and both filesystem bridges still treat only `rw` as writable; the latest release and tests preserve the same behavior.

## Solution Assessment

Yes on direction, but it has not landed. The maintainable path is to make only `ro` read-only while preserving the isolated `none` workspace boundary and reconciling the related write-tool policy before merge.
