## Summary

Keep open: current main still maps `workspaceAccess: "none"` to an isolated sandbox workspace, but Docker/browser mounts plus local and remote bridge write checks still treat that workspace as read-only. The open fix path remains https://github.com/openclaw/openclaw/pull/46300, which should be reviewed, repaired, or replaced rather than duplicating work.

## What This Changes

The issue requests that `workspaceAccess: "none"` keep the host workspace hidden while leaving the isolated sandbox `/workspace` writable.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/46300 or a focused replacement that treats only `ro` as read-only for the isolated `none` workspace across Docker/browser mounts, local FS bridge, remote FS bridge, docs, and regression coverage.

## Reproduction Assessment

Yes. Source inspection on current main shows `none` selects the isolated sandbox workspace but still marks Docker/browser, local bridge, and remote bridge write paths read-only; a focused repro is configuring `workspaceAccess: "none"` and attempting a write under `/workspace`.

## Solution Assessment

Yes on direction, but the repository is not fixed yet. The narrow maintainable solution is to make only `ro` read-only while keeping `none` confined to the isolated sandbox workspace and preserving all mount/path guards.
