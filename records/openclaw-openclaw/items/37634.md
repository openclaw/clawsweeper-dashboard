## Summary

Keep this issue open. Current main still isolates `workspaceAccess: "none"` to the sandbox workspace, but the Docker/browser mount, local filesystem bridge, remote filesystem bridge, tests, and docs still treat that workspace as read-only or `rw`-only; the related open PR #46300 should be reviewed, landed, or superseded rather than closing the tracker now.

## What This Changes

The issue requests that `workspaceAccess: "none"` hide the main agent workspace while keeping the isolated sandbox `/workspace` writable.

## Best Possible Solution

Keep this issue as the focused tracker and land or supersede #46300 with a complete fix that treats only `workspaceAccess: "ro"` as read-only, keeps the main agent workspace unmounted for `none`, updates Docker/browser mounts plus local and remote filesystem bridges, and aligns docs and regression tests.

## Reproduction Assessment

Yes. The static current-main path is enough to reproduce with high confidence: `workspaceAccess: "none"` selects the isolated sandbox workspace, then the mount and fs-bridge write gates still mark it read-only or reject writes unless access is `rw`.

## Solution Assessment

Yes in direction, but current main is not solved. The narrow maintainable solution is to make the isolated `none` workspace writable while preserving `ro` read-only behavior and host-workspace isolation, with secops review of the permission boundary.
