## Summary

Keep open: current `main` and the latest release still make `workspaceAccess: "none"` read-only at the managed `/workspace` mount and local/remote filesystem bridges. The linked fix work is still open or stale, and there is now a competing protected PR about whether write tools should be exposed for `none`, so this needs coordinated maintainer review rather than cleanup closure.

## What This Changes

The issue asks for `workspaceAccess: "none"` to keep the real agent workspace hidden while leaving the isolated sandbox `/workspace` writable.

## Best Possible Solution

Have maintainers choose the canonical `workspaceAccess: "none"` policy, then land or replace https://github.com/openclaw/openclaw/pull/46300 with a complete fix across Docker/browser mounts, local and remote bridges, memory/docs/tests, and reconcile https://github.com/openclaw/openclaw/pull/73385 with that decision.

## Reproduction Assessment

Yes, source-level. Current main maps `none` to an isolated sandbox workspace, but the managed mount and both filesystem bridges still allow writes only for `rw`; I did not run a live sandbox in this read-only review.

## Solution Assessment

Unclear as a merge path until maintainers resolve the sandbox policy conflict. The writable-isolated-workspace direction is a narrow code fix if accepted, but it must be aligned with tool exposure, docs, and the protected write-tool PR.
