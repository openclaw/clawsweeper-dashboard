## Summary

Keep open: current main and v2026.5.12 still make `workspaceAccess: "none"` read-only in the managed `/workspace` mount and both filesystem bridges. Open PR https://github.com/openclaw/openclaw/pull/46300 targets this behavior, but it has not landed and still needs sandbox/security proof and parity with the remote bridge.

## What This Changes

The issue requests that `workspaceAccess: "none"` keep the host workspace hidden while making the isolated sandbox `/workspace` writable.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/46300 with a complete fix that makes only `ro` read-only across Docker/browser mounts and local/remote bridges, preserves host isolation, updates docs/tests, and includes redacted sandbox proof.

## Reproduction Assessment

Yes. Source inspection on current main shows `none` selects an isolated sandbox workspace, but the mount helper and local/remote write guards still permit writes only for `rw`; I did not run a live sandbox in this read-only review.

## Solution Assessment

Yes. The requested direction is the narrow maintainable fix, but the implementation needs complete local/remote parity, docs/tests, and sandbox security proof before merge.
