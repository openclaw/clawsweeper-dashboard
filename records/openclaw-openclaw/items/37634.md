## Summary

Keep open: current main and latest release still isolate `workspaceAccess: "none"` to the sandbox workspace but make the managed mount and FS bridges writable only for `rw`. Multiple users confirmed the workflow impact, and the open fix/policy PRs point in different directions, so this needs maintainer security/product review rather than cleanup closure.

## What This Changes

The issue asks for `workspaceAccess: "none"` to keep the real agent workspace hidden while making the isolated sandbox `/workspace` writable.

## Best Possible Solution

Keep this as the canonical tracker until maintainers choose the writable-`none` contract, reconcile tool exposure, land or replace the open fix PR, and update tests/docs plus local and remote sandbox proof.

## Reproduction Assessment

Yes, by source inspection. At the target SHA, `none` maps to the isolated sandbox workspace, but the Docker mount, local FS bridge, and remote FS bridge still allow writes only for `rw`; I did not run a live sandbox in this read-only review.

## Solution Assessment

Unclear until maintainers resolve the sandbox policy. The narrow code direction is to make only `ro` read-only while preserving host isolation, but the final solution must align write-tool exposure, remote backend parity, docs, tests, and runtime proof.
