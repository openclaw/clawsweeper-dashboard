- `impact:security`: The requested fix changes sandbox write authorization and must preserve the host workspace security boundary.

## Summary

Keep open: current main and the latest release still make `workspaceAccess: "none"` use the isolated sandbox workspace but treat that workspace as read-only in the managed mount and local/remote FS bridges. The remaining fix is policy-sensitive because https://github.com/openclaw/openclaw/pull/46300 moves toward writable `none` semantics while https://github.com/openclaw/openclaw/pull/73385 narrows write-tool exposure for `none`.

## What This Changes

The issue requests that `workspaceAccess: "none"` keep the host workspace hidden while leaving the per-session isolated sandbox `/workspace` writable.

## Best Possible Solution

Keep this issue as the canonical policy/bug tracker and land or replace https://github.com/openclaw/openclaw/pull/46300 only after maintainers choose the writable-`none` contract, reconcile related write-tool policy, update docs/tests, and prove local plus remote sandbox behavior.

## Reproduction Assessment

Yes, by source inspection. Current main maps `none` to an isolated sandbox workspace, but the managed Docker mount, local FS bridge, and remote FS bridge still allow writes only for `rw`; I did not run a live Docker or OpenShell sandbox in this read-only review.

## Solution Assessment

Unclear until maintainers resolve the sandbox policy. The narrow code direction is to make only `ro` read-only while preserving host isolation, but the final solution must align tool exposure, memory behavior, remote backend parity, docs, tests, and runtime proof.
