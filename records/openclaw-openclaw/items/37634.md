## Summary

Keep open: current main still makes `workspaceAccess: "none"` sandbox workspaces read-only while write/edit/apply_patch tools are still exposed for that mode. The open related PR #46300 appears to target this exact behavior, but it has not merged, so this issue should remain open for review or an equivalent fix.

## What This Changes

Request `workspaceAccess: "none"` to keep the host agent workspace hidden while leaving the isolated sandbox `/workspace` writable.

## Best Possible Solution

Land a focused sandbox fix that treats only `workspaceAccess: "ro"` as read-only across Docker/browser mounts, local and remote FS bridges, memory/docs, and regression tests while keeping the host agent workspace unmounted for `"none"`.

## Reproduction Assessment

Yes by source inspection: configure sandboxing with `workspaceAccess: "none"` and attempt a write to `/workspace`; current main mounts and resolves that workspace as read-only while still exposing write-capable tools. I did not run a live Docker session, so this is source-reproducible rather than runtime-reproduced.

## Solution Assessment

Yes: the narrow maintainable fix is to make `"ro"` the only read-only workspace mode and keep `"none"` isolated from the host workspace. The related open PR #46300 or an equivalent patch should update the mount, bridge, remote bridge, docs, and tests together.
