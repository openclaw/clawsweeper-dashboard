## Summary

Keep open: current main still treats `workspaceAccess: "none"` as read-only across the Docker mount and filesystem bridge, and an open fix PR already targets the same behavior. Because current docs and tests also encode the read-only semantics, this should be resolved through maintainer review of the sandbox policy rather than cleanup closure.

## What This Changes

Request that `workspaceAccess: "none"` keep the agent workspace hidden while leaving the isolated sandbox `/workspace` writable.

## Best Possible Solution

Review and land a narrow policy-consistent fix that makes only the isolated `none` sandbox workspace writable, while keeping host workspace access blocked and updating Docker mounts, FS bridge checks, memory behavior, docs, and tests together.

## Reproduction Assessment

Yes by source inspection, not by a live Docker run. `workspaceAccess: "none"` resolves to the sandbox workspace, but the mount and FS bridge both treat every non-`rw` mode as read-only, matching the reported `/workspace:ro` failure.

## Solution Assessment

Unclear until maintainer policy review. The suggested narrow direction is plausible, but current docs/tests also need to change and open PR #46300 is the right place to evaluate the full fix.
