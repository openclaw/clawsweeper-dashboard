## Summary

Current main still matches the reported failure path: `workspaceAccess: "none"` uses the isolated sandbox workspace, but Docker/browser mounts and both local and remote filesystem bridge write checks still treat anything other than `"rw"` as read-only. This should stay open while the related open fix PR #46300 is reviewed, landed, or replaced with a complete sandbox-permission fix.

## What This Changes

The issue requests that `workspaceAccess: "none"` keep the main agent workspace hidden while allowing writes inside the isolated per-session sandbox `/workspace`.

## Best Possible Solution

Land or supersede #46300 with a complete fix that makes only the isolated sandbox workspace writable for `workspaceAccess: "none"`, while preserving main workspace isolation and `"ro"` read-only behavior across Docker/browser mounts, local and remote FS bridges, docs, and regression tests.
