## Summary

Keep open: current main and the latest release still mount the isolated `workspaceAccess: "none"` sandbox workspace read-only, while exact fix attempts are closed unmerged and the open adjacent PR does not implement the requested mount semantics.

## What This Changes

Requests changing sandbox `workspaceAccess: "none"` so the isolated per-session `/workspace` remains writable while the main agent workspace stays unmounted.

## Best Possible Solution

Settle the sandbox policy, then align Docker/browser/remote mounts, fs-bridge writability, tool exposure, docs, and regression tests around one consistent `workspaceAccess: "none"` contract.

## Reproduction Assessment

Yes. Source inspection on current main shows `none` resolves to the isolated sandbox workspace and then mounts or bridges it as non-writable unless access is `"rw"`; I did not run a live Docker sandbox in this read-only review.

## Solution Assessment

Unclear until maintainers choose the security semantics. The isolated-but-writable direction is coherent, but it intentionally changes current tested/documented read-only behavior and must be applied across all sandbox backends and write gates.
