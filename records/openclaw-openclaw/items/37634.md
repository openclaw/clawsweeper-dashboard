## Summary

Keep open: current main and latest release still treat `workspaceAccess: "none"` as read-only for the isolated sandbox workspace, while earlier exact fix PRs are closed unmerged and the remaining choice is security-sensitive.

## What This Changes

The issue requests that `workspaceAccess: "none"` keep the main agent workspace hidden while leaving the per-session sandbox `/workspace` writable.

## Best Possible Solution

Settle the sandbox policy first, then make Docker/browser/SSH mounts, fs-bridge writability, tool exposure, docs, and regression tests express one consistent `none` meaning.

## Reproduction Assessment

Yes from source inspection: `none` resolves to the sandbox workspace, then the mount and fs bridge both mark it read-only unless access is `"rw"`. I did not run a live Docker sandbox, but the current code and tests make the failure path explicit.

## Solution Assessment

Unclear until maintainers choose the security semantics. The requested isolated-but-writable direction is coherent, but current docs/tests and adjacent tool-gating work also support a stricter read-only interpretation.
