- `impact:security`: The requested fix changes sandbox write authorization and must preserve the host workspace security boundary.

## Summary

Keep open: current main and the latest release still make `workspaceAccess: "none"` use an isolated sandbox workspace while the Docker mount and local/remote filesystem bridges treat that workspace as read-only; the open fix path at https://github.com/openclaw/openclaw/pull/46300 still needs maintainer sandbox-policy and security review alongside https://github.com/openclaw/openclaw/pull/73385.

## What This Changes

The issue asks for `workspaceAccess: "none"` to keep the real agent workspace hidden while making the isolated sandbox `/workspace` writable.

## Best Possible Solution

Have maintainers choose the canonical `workspaceAccess: "none"` policy, then land or replace https://github.com/openclaw/openclaw/pull/46300 with a complete fix across Docker/browser mounts, local and remote bridges, docs, memory/tool exposure, and tests while reconciling https://github.com/openclaw/openclaw/pull/73385.

## Reproduction Assessment

Yes, source-level: current main maps `none` to an isolated sandbox workspace, but managed mounts plus local and remote bridges still allow writes only for `rw`. I did not execute a live sandbox in this read-only review.

## Solution Assessment

Unclear until maintainers resolve the sandbox policy conflict. The narrow implementation is to make only `ro` read-only while preserving host isolation, but the final behavior must align with tool exposure, remote backend parity, docs, and security expectations.
