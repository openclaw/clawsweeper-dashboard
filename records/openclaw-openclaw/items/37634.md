## Summary

Keep open: current main and v2026.5.12 still treat `workspaceAccess: "none"` as read-only in the managed workspace mount and local/remote filesystem bridges. Open PR https://github.com/openclaw/openclaw/pull/46300 targets the behavior, but it has not landed and still needs sandbox/security validation.

## What This Changes

Make `workspaceAccess: "none"` hide the agent workspace while keeping the isolated sandbox `/workspace` writable.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/46300 with a complete fix that makes only `ro` read-only across Docker/browser mounts and local/remote bridges, preserves host isolation, updates docs/tests, and includes sandbox proof.

## Reproduction Assessment

Yes, source-reproducible on current main: `none` selects an isolated sandbox workspace, but the mount helper and local/remote bridges still permit writes only for `rw`. I did not run a live sandbox in this read-only review.

## Solution Assessment

Yes, the requested direction is the narrow maintainable fix, but it has not landed. The implementation should keep host workspace isolation intact and update remote bridge parity, tests, and docs before merge.
