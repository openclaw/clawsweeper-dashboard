## Summary

Keep open: current main and the latest release still make `workspaceAccess: "none"` workspaces read-only in the Docker/browser mount path and both local/remote bridge write guards, while the active fix path remains the open PR at https://github.com/openclaw/openclaw/pull/46300.

## What This Changes

The issue requests that `workspaceAccess: "none"` keep the real agent workspace hidden while leaving the per-session isolated sandbox `/workspace` writable.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/46300 or a focused replacement that treats only `ro` as read-only across Docker/browser mounts, local and remote FS bridges, memory/docs/tests, with runtime proof and sandbox security review.

## Reproduction Assessment

Yes. Source inspection on current main shows `none` selects an isolated sandbox workspace, but the mount helper, local bridge, and remote bridge still mark only `rw` writable; current tests also expect `none` to mount as read-only.

## Solution Assessment

Yes on direction, but it has not landed. The narrow maintainable fix is to make only `ro` read-only while preserving the isolated `none` workspace boundary and all mount/path guards.
