## Summary

Keep open: current main and the latest release still make `workspaceAccess: "none"` read-only in sandbox mount and bridge write paths, and the open fix PR has not merged.

## What This Changes

The issue requests that `workspaceAccess: "none"` keep the host workspace hidden while leaving the isolated sandbox `/workspace` writable.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/46300 or a focused replacement that makes only the isolated `none` workspace writable across Docker, browser, local FS bridge, remote FS bridge, docs, changelog, and regression coverage.

## Reproduction Assessment

Yes, at source level and with matching user Docker logs. Current main selects an isolated sandbox workspace for `none`, then marks Docker, local bridge, and remote bridge writes as writable only for `rw`.

## Solution Assessment

Yes on direction, but the repository is not fixed yet. The maintainable fix is to treat only `ro` as read-only while keeping `none` confined to the isolated sandbox workspace across all sandbox backends and docs/tests.
