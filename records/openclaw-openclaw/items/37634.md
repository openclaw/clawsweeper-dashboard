## Summary

Keep open: current main still routes `workspaceAccess: "none"` to an isolated sandbox workspace, but Docker `/workspace`, the local filesystem bridge, and the remote filesystem bridge still treat writes as allowed only for `workspaceAccess: "rw"`. The issue remains source-reproducible and security-sensitive, so it is not a cleanup-close candidate.

## What This Changes

The issue asks for `workspaceAccess: "none"` to hide the real agent workspace while keeping the per-session isolated `/workspace` writable.

## Best Possible Solution

Keep this as the canonical tracker until maintainers land a focused sandbox contract fix that makes only the isolated `none` workspace writable while preserving host isolation, protected skill behavior, bridge parity, docs, tests, and runtime proof.

## Reproduction Assessment

Yes, by source inspection. Current main selects the isolated sandbox workspace for `workspaceAccess: "none"`, but the Docker mount, local filesystem bridge, and remote filesystem bridge still allow writes only for `rw`; I did not run a live sandbox in this read-only review.

## Solution Assessment

Unclear until maintainers settle the sandbox policy. The narrow repair is to make only `ro` read-only while keeping `none` isolated, but the final change needs bridge parity, protected mount handling, docs, tests, and runtime proof.
