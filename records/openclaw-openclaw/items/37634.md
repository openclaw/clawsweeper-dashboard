## Summary

Keep open: current main and the latest release still make `workspaceAccess: "none"` read-only in Docker, local bridge, and remote bridge write paths, while the exact fix path remains an open PR.

## What This Changes

The issue asks for `workspaceAccess: "none"` to keep the host workspace hidden while leaving the isolated sandbox `/workspace` writable.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/46300 or a focused replacement that makes only the isolated `none` workspace writable across Docker, browser, local FS bridge, remote FS bridge, docs, changelog, and regression coverage.

## Reproduction Assessment

Yes. Source inspection on current main shows `none` uses the isolated sandbox workspace but marks Docker, local FS bridge, and remote bridge write paths read-only, and https://github.com/openclaw/openclaw/issues/59613 provides matching Docker logs.

## Solution Assessment

Yes on direction, but the repository is not fixed yet. The narrow maintainable solution is to treat only `ro` as read-only while keeping `none` confined to the isolated sandbox workspace across every sandbox backend.
