## Summary

Keep open: current main still makes `workspaceAccess: "none"` read-only in Docker/local/remote sandbox write paths even though that mode uses an isolated sandbox workspace, and an open implementation PR already targets this behavior.

## What This Changes

The issue asks for `workspaceAccess: "none"` to hide the main agent workspace while keeping the isolated sandbox `/workspace` writable.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/46300 or an equivalent focused fix that aligns Docker, browser, local FS bridge, remote FS bridge, docs, changelog, and regression coverage for isolated writable `none` workspaces.

## Reproduction Assessment

Yes, at source level and with matching user-provided runtime logs. Current main selects the isolated sandbox workspace for `none` but marks Docker, local bridge, and remote bridge writes as writable only for `rw`.

## Solution Assessment

Yes on direction, but incomplete until all sandbox backends and docs/tests are aligned. The narrow maintainable fix is to make only the isolated sandbox workspace writable for `none` while keeping the host agent workspace hidden.
