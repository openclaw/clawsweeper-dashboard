## Label Justifications

- `P1`: The issue describes a sandbox regression that can make I/O-heavy sandboxed sessions and skills unable to write to their isolated workspace.
- `impact:security`: The requested behavior changes sandbox filesystem read/write boundaries and must preserve host workspace isolation.
- `impact:session-state`: The affected path is the per-session sandbox workspace used for temporary outputs, caches, memory flushes, and session-local files.

## Summary

Keep open: current main and the latest release still make the isolated `workspaceAccess: "none"` sandbox workspace read-only at the mount and filesystem-bridge layers. The exact writable-`none` fix attempts are closed unmerged, and the remaining decision is a sandbox security/product contract rather than a cleanup close.

## What This Changes

Requests that `workspaceAccess: "none"` keep the real agent workspace hidden while allowing writes inside the isolated per-session sandbox `/workspace`.

## Best Possible Solution

Settle the sandbox policy, then make mounts, local and remote bridges, tool exposure, docs, and tests express one consistent `workspaceAccess: "none"` contract.

## Reproduction Assessment

Yes. Source inspection shows `none` selects the isolated sandbox workspace, then current mounts and filesystem bridges mark that workspace read-only unless access is `"rw"`; no live Docker sandbox was run in this read-only review.

## Solution Assessment

Unclear until maintainers choose the sandbox write policy. The requested isolated-but-writable behavior is coherent, but current docs and policy also treat `none` as read-only/no-write, so the best fix must first resolve that contract.
