## Summary

Keep open: current main still selects an isolated sandbox workspace for `workspaceAccess: "none"`, but Docker/browser mounts plus local and remote bridge write guards still treat that workspace as read-only. The active fix path is the open PR at https://github.com/openclaw/openclaw/pull/46300, so this issue should stay open until that PR or a focused replacement lands.

## What This Changes

The issue requests that `workspaceAccess: "none"` keep the real agent workspace hidden while allowing writes inside the isolated sandbox `/workspace`.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/46300 or a focused replacement that treats only `ro` as read-only across Docker/browser mounts, local and remote FS bridges, docs, and regression coverage while keeping `none` confined to the isolated sandbox workspace.

## Reproduction Assessment

Yes. Source inspection on current main shows `none` selects the isolated sandbox workspace, but Docker/browser mounts, local FS bridge, and remote FS bridge still mark non-`rw` workspace access read-only.

## Solution Assessment

Yes on direction, but the repository is not fixed yet. The narrow maintainable solution is to make only `ro` read-only while preserving all mount and path-boundary guards for the isolated `none` workspace.
