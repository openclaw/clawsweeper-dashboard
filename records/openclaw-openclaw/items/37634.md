## Label Justifications

- `P1`: The issue is a source-reproducible sandbox regression that breaks write-heavy skills and normal agent workflows for the default isolated workspace mode.
- `impact:security`: The requested behavior changes sandbox filesystem write authorization and must preserve the isolation boundary around the real agent workspace.
- `impact:session-state`: The read-only isolated workspace blocks session-local tool output, memory/bootstrap file writes, and other per-session filesystem state.

## Summary

Keep open: current main and the latest release still make `workspaceAccess: "none"` read-only at the Docker mount and filesystem-bridge layers, while the direct fix PRs were closed unmerged and the open related PR addresses a narrower fail-closed tool-surface policy rather than the requested writable isolated workspace.

## What This Changes

The issue asks OpenClaw to keep the real agent workspace hidden for `workspaceAccess: "none"` while making the isolated sandbox `/workspace` writable for tools and skills.

## Best Possible Solution

Settle the sandbox/security contract, then implement one current-main fix that makes only the isolated `none` sandbox workspace writable while preserving real-workspace isolation, read-only skill roots, bridge checks, docs, and regression proof.

## Reproduction Assessment

Yes, at source level: current main chooses the isolated sandbox workspace for `workspaceAccess: "none"`, then mounts and classifies that workspace as writable only when access is `"rw"`. I did not run a live Docker sandbox in this read-only review.

## Solution Assessment

Unclear until maintainers settle the sandbox policy. If writable `none` is accepted, the best fix is a cross-layer current-main change covering Docker/browser mounts, local and remote bridges, tool gating, memory/docs, and tests rather than another mount-only patch.
