## Summary

Keep open: current main and the latest release still isolate `workspaceAccess: "none"` to a sandbox workspace but make the managed `/workspace` mount and both filesystem bridges writable only for `rw`. Multiple users confirmed the workflow impact, and related open policy/fix work points in different directions, so this needs maintainer security/product review rather than cleanup closure.

## What This Changes

The issue requests that `workspaceAccess: "none"` keep the real agent workspace hidden while making the per-session isolated sandbox `/workspace` writable.

## Best Possible Solution

Keep this as the canonical tracker until maintainers choose the writable-`none` sandbox contract, reconcile write-tool exposure and bind behavior, then land a focused fix with Docker, local bridge, remote bridge, docs, tests, and runtime proof.

## Reproduction Assessment

Yes, by source inspection. Current main selects the isolated sandbox workspace for `workspaceAccess: "none"`, but the Docker mount, local FS bridge, and remote FS bridge still allow writes only for `rw`; I did not run a live sandbox in this read-only review.

## Solution Assessment

Unclear until maintainers resolve the sandbox policy. The narrow code direction is to make only `ro` read-only while preserving host isolation, but the final solution must align write-tool exposure, bind behavior, remote backend parity, docs, tests, and runtime proof.
