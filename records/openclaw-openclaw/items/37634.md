## Summary

Keep open: current main still makes the isolated `workspaceAccess: "none"` workspace read-only through Docker mount construction and both filesystem bridge write checks; the related fix PR is still open and unmerged.

## What This Changes

The issue asks for `workspaceAccess: "none"` to keep the host agent workspace hidden while making the isolated sandbox `/workspace` writable.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/46300, or an equivalent corrected patch, with aligned Docker, local bridge, remote bridge, docs, regression tests, and runtime proof.

## Reproduction Assessment

Yes by source inspection, not by a live runtime run. Current main selects the isolated sandbox workspace for `workspaceAccess: "none"`, but Docker and both FS bridge write gates only allow writes for `"rw"`, matching the reported read-only `/workspace` behavior.

## Solution Assessment

Yes in direction, but not complete on main. The narrow maintainable fix is the open PR or equivalent, completed across remote bridge parity, docs, tests, and validation before this issue closes.
