## Summary

Keep open: current main and the latest release still make `workspaceAccess: "none"` use an isolated sandbox workspace while the Docker mount and local/remote filesystem bridges treat that workspace as read-only, and the remaining fix needs maintainer sandbox-policy review because it affects write authorization boundaries.

## What This Changes

The issue asks OpenClaw to keep the real agent workspace hidden for `workspaceAccess: "none"` while leaving the isolated sandbox `/workspace` writable.

## Best Possible Solution

Have maintainers choose the canonical `workspaceAccess: "none"` policy, then land or replace https://github.com/openclaw/openclaw/pull/46300 with a complete fix across Docker/browser mounts, local and remote bridges, docs, memory/tool exposure, and tests while reconciling https://github.com/openclaw/openclaw/pull/73385.

## Reproduction Assessment

Yes, source-level. Current main maps `none` to an isolated sandbox workspace, but the managed mount plus local and remote bridges still allow writes only for `rw`; I did not execute a live sandbox in this read-only review.

## Solution Assessment

Unclear until maintainers resolve the sandbox policy conflict. The narrow implementation is to make only `ro` read-only while preserving host isolation, but the final behavior must align with tool exposure, remote backend parity, docs, and security expectations.
