## Summary

Keep open: current main and the latest release still make `workspaceAccess: "none"` workspaces read-only in the Docker/browser mount path and local/remote bridge write guards. The issue is source-reproducible, but the remaining fix needs maintainer sandbox/security review and coordination with the open fix PR.

## What This Changes

Requests that `workspaceAccess: "none"` hide the real agent workspace while keeping the isolated sandbox `/workspace` writable.

## Best Possible Solution

Keep this issue open until a focused fix or https://github.com/openclaw/openclaw/pull/46300 makes only `ro` read-only across Docker/browser mounts, local and remote FS bridges, docs, and regression tests with sandbox security review and runtime proof.

## Reproduction Assessment

Yes. Source inspection on current main shows `none` selects an isolated sandbox workspace, but the mount helper and both bridge write guards still treat only `rw` as writable; current tests and the latest release preserve that behavior.

## Solution Assessment

Yes on direction, but it has not landed. The maintainable fix is to make only `ro` read-only while preserving the isolated `none` workspace boundary, then reconcile the related write-tool policy work before merge.
