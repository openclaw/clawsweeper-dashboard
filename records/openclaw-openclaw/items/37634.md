## Label Justifications

- `P1`: The report describes a sandbox regression that can make I/O-heavy sandboxed sessions and skills unable to write to their isolated workspace.
- `impact:security`: The requested behavior changes sandbox filesystem read/write boundaries and must preserve host workspace isolation.
- `impact:session-state`: The affected path is the per-session sandbox workspace used for temporary outputs, caches, memory flushes, and session-local files.

## Summary

Keep open: current main and v2026.6.10 still treat the isolated `workspaceAccess: "none"` sandbox workspace as read-only at the mount and filesystem-bridge layers, while the exact writable-`none` fix PRs were closed unmerged and the remaining choice is sandbox-security policy.

## What This Changes

The issue requests that `workspaceAccess: "none"` keep the real agent workspace hidden while leaving the per-session isolated sandbox `/workspace` writable.

## Best Possible Solution

Settle the sandbox policy, then make mounts, local/remote bridges, tool availability, memory/docs behavior, and regression tests express one consistent `workspaceAccess: "none"` contract.

## Reproduction Assessment

Yes. Source inspection shows `none` selects the isolated sandbox workspace, then current mounts and filesystem bridges mark that workspace read-only unless access is `"rw"`; I did not run a live Docker sandbox in this read-only review.

## Solution Assessment

Unclear until maintainers choose the sandbox write policy. The requested isolated-but-writable behavior is coherent, but current security/policy docs also treat `none` as a read-only posture, so the best fix must first resolve that contract.
