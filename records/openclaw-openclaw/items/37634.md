## Label Justifications

- `P1`: The current sandbox behavior breaks write-capable workflows for users relying on isolated sandbox workspaces.
- `impact:security`: The requested fix changes sandbox filesystem write authorization and must preserve the host workspace isolation boundary.
- `impact:session-state`: The bug prevents sandbox-local session files, memory, and tool state from being written in `workspaceAccess: "none"` mode.

## Summary

Keep open: current main and the latest release still make `workspaceAccess: "none"` read-only in the managed `/workspace` mount and filesystem bridges, while related duplicate reports point back to this issue as the canonical tracker.

## What This Changes

The issue asks for `workspaceAccess: "none"` sandbox sessions to keep their isolated `/workspace` writable while still hiding the host agent workspace.

## Best Possible Solution

Land or replace the open writable-`none` fix with a maintainer-approved sandbox policy that keeps host workspace isolation while making the isolated sandbox workspace writable consistently across mounts, bridges, tools, docs, and proof.

## Reproduction Assessment

Yes, by source inspection. Current main maps `none` to an isolated sandbox workspace, but the managed mount plus local and remote bridges still allow writes only for `rw`.

## Solution Assessment

Unclear until maintainers settle the sandbox policy. The narrow likely fix is to make only `ro` read-only while preserving host isolation, but the final implementation must align Docker, local bridge, remote bridge, tool exposure, docs, and tests.
