## Summary

Keep open: the source path makes the reported Telegram sequencing bug plausible and the patch looks narrow, but this external draft PR only provides unit/format validation and still needs real Telegram behavior proof before merge.

## What This Changes

The PR routes Telegram text `/steer`, `/tell`, and `/queue` commands onto the per-chat control sequentialization lane and adds sequential-key unit coverage.

## Best Possible Solution

Land a narrow sequential-key fix after redacted live Telegram proof shows `/steer`, `/tell`, and `/queue` bypass a busy chat/topic lane without regressing command/reply ordering; keep the separate cross-lane targeting report at https://github.com/openclaw/openclaw/issues/81594 separate.

## Reproduction Assessment

Yes from source inspection: current main's Telegram sequential key only puts abort/status-style commands on the control lane, so these management text commands fall through to the normal chat/topic key. I did not establish a live Telegram reproduction in this read-only review.

## Solution Assessment

Yes, the proposed approach is the narrowest source-level fix because it extends the existing control-lane exception in `sequential-key.ts` instead of changing session routing. Merge should still wait for real Telegram proof because the behavior is timing-sensitive and adjacent to prior command/reply race fixes.
