## Summary

Close as duplicate/superseded: the same provider-timeout-to-idle-watchdog root cause is already tracked in the open canonical issue and an active focused PR; this report adds useful Docker/Ollama evidence but does not require a separate OpenClaw work item.

## What This Changes

The issue reports that `models.providers.<id>.timeoutSeconds` is accepted for a Docker-local Ollama provider but the embedded LLM idle watchdog still aborts after the hardcoded 120 seconds.

## Best Possible Solution

Keep the canonical timeout issue and active resolver PR as the single work path, and preserve this Docker/Ollama evidence there if maintainers need the host-gateway reproduction detail.

## Reproduction Assessment

Yes, at source level: with `baseUrl: "http://host.docker.internal:11434"` and a provider `requestTimeoutMs` above 120s, current `main` does not classify the endpoint as local and caps the idle watchdog at `DEFAULT_LLM_IDLE_TIMEOUT_MS`. I did not run a live Docker/Ollama cold-start replay in this read-only pass.

## Solution Assessment

Yes for cleanup: the maintainable path is one resolver fix under the canonical issue/active PR, not a second queueable duplicate. The fix should honor explicit provider timeouts while preserving the default 120s guard when users have not configured a provider timeout.
