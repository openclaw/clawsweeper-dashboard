## Label Justifications

- `P1`: A documented provider timeout path is ineffective for Docker-local Ollama, breaking agent replies for slow local models in the current release.
- `impact:message-loss`: Affected user messages can time out without an agent reply when the local model stays silent beyond 120 seconds.
- `impact:auth-provider`: The report centers on model-provider configuration and runtime provider request timeout behavior.

## Summary

Keep open: current `main` still treats Docker's documented `host.docker.internal` Ollama URL as non-local, so an explicit provider timeout is capped to the default 120s idle watchdog before the Ollama guarded fetch sees it.

## What This Changes

The issue asks for `models.providers.<id>.timeoutSeconds` to extend the LLM idle/fetch watchdog for slow Docker-local Ollama models instead of failing at 120 seconds.

## Best Possible Solution

Make the idle-timeout resolver honor documented Docker-local provider endpoints, especially `host.docker.internal`, while preserving the cloud-provider default guard and adding regression coverage for the configured provider timeout path.

## Reproduction Assessment

Yes for source-level reproduction: with `baseUrl: "http://host.docker.internal:11434"` and `modelRequestTimeoutMs: 1200000`, the resolver does not classify the endpoint as local and caps the idle watchdog at 120000ms. No live Docker/Ollama run was performed in this read-only pass.

## Solution Assessment

Yes, the narrow maintainable fix is in idle-timeout classification or explicit provider-timeout handling, with tests for `host.docker.internal` and no broad weakening of remote cloud idle safeguards.
