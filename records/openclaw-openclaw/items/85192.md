## Summary

Keep open: current main still gates the visible-answer continuation retry on `assessLastAssistantMessage(...) === "incomplete-text"`, and unsigned thinking-only turns do not meet that predicate. The DeepSeek normal `reasoning_content` path should be signed today, so the remaining blocker is a focused DeepSeek V4-Pro trace or fixture that resolves the mismatch between the reporter’s production failure and the source path.

## What This Changes

The issue asks OpenClaw to recover DeepSeek V4 reasoning-only turns when thinking blocks are not recognized as signed, so the run gets the visible-answer continuation retry instead of timing out or using a weaker fallback.

## Best Possible Solution

Capture or synthesize the exact DeepSeek V4-Pro replay shape, then add a narrow regression that makes that shape recover with the visible-answer continuation while preserving deliberate-silence and provider-specific replay boundaries.

## Reproduction Assessment

Yes for the narrow classifier gap: source inspection shows unsigned thinking-only blocks do not trigger the reasoning-only continuation retry. No for the full DeepSeek V4-Pro production failure yet, because the reported persisted `thinkingSignature` should be considered signed by current source.

## Solution Assessment

Unclear: the suggested direction is plausible, but the safest fix depends on whether the real failure is an actually unsigned block, a replay transformation mismatch, idle-timeout state, or group-chat silent-reply gating. The best path is fixture-backed recovery rather than broadening all unsigned thinking semantics.
