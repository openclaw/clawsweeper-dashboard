## Summary

Keep open: the source-level bug is real and the patch is mostly pointed at the right shared helper, but the external PR currently has only targeted unit-test output for proof and introduces a small edge where `null` rejections lose their previous message detail.

## What This Changes

The PR changes shared agent transport error handling to best-effort stringify non-`Error` transport rejections and adds regression coverage for BigInt and circular rejection values.

## Best Possible Solution

Keep the shared helper approach, preserve existing detail for JSON-serializable primitives such as `null`, and add redacted runtime or provider-harness proof that a malformed rejection emits a structured transport error event.

## Reproduction Assessment

Yes. Current main directly calls `JSON.stringify(error)` for non-`Error` transport failures, and a Node probe confirms the reported BigInt and circular values throw there.

## Solution Assessment

Mostly yes. The shared transport helper is the right narrow boundary, but the message serializer should preserve JSON-serializable `null` behavior and the PR still needs real behavior proof before merge.
