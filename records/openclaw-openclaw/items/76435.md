## Summary

Keep open. Current main still returns OpenAI attribution headers in the OpenAI Realtime browser session and the Control UI forwards them to the direct SDP fetch; a live CORS preflight for the same endpoint allows only authorization and content-type. The open closing PR #76457 is the right place to review the fix before this issue is closed.

## What This Changes

The issue asks OpenAI Realtime Talk WebRTC to keep server-side attribution while omitting browser-visible attribution headers from the direct SDP offer POST.

## Best Possible Solution

Land #76457 or an equivalent narrow provider-local fix that removes OpenAI attribution headers from browser `offerHeaders`, preserves Gateway-side attribution for server calls, updates focused tests, and adds the required changelog entry.

## Reproduction Assessment

Yes. Source inspection shows current main forwards `originator`/`version` into the browser SDP fetch, and a live preflight for the exact OpenAI endpoint rejects those headers by allowing only `authorization,content-type`.

## Solution Assessment

Yes. The narrow maintainable fix is provider-local: do not return OpenAI attribution headers in browser `offerHeaders` for the direct SDP exchange, while leaving server-side attribution intact.
