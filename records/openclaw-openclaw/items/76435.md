## Summary

Keep open. Current main and v2026.5.2 still return OpenAI attribution headers for the browser WebRTC SDP offer path, and the UI forwards them to the direct OpenAI fetch; a live CORS preflight only allows authorization and content-type. Open PR #76457 already references this issue with closing syntax, so the issue should stay open until that PR or an equivalent fix lands.

## What This Changes

The issue asks OpenClaw to stop sending OpenAI attribution headers on the browser-side OpenAI Realtime WebRTC SDP POST for Talk.

## Best Possible Solution

Land a narrow provider/UI fix that removes browser-visible OpenAI attribution headers from WebRTC SDP offers while keeping Gateway-side attribution intact.

## Reproduction Assessment

Yes. Current source constructs and forwards `originator`/`version` for the browser SDP request, and a live preflight for the exact OpenAI endpoint confirms those headers are not allowed.

## Solution Assessment

Yes. The narrow maintainable solution is provider-local/browser-offer filtering rather than changing the generic Talk session contract or adding a config option.
