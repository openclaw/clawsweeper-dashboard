## Summary

Keep open. Current main still exposes OpenAI attribution headers to the browser SDP POST path and a live CORS preflight rejects those headers; linked PR #76457 is an active closing fix attempt but has not merged, so this issue should remain open until that work lands or is replaced.

## What This Changes

The issue asks OpenClaw to remove server-side OpenAI attribution headers from the browser-side Realtime WebRTC SDP exchange while preserving Gateway-side session minting.

## Best Possible Solution

Review and land PR #76457 or an equivalent narrow OpenAI provider fix that removes browser-visible attribution headers, updates focused tests and changelog, and then lets the merged PR close this issue.

## Reproduction Assessment

Yes. Current main returns OpenAI browser `offerHeaders` containing attribution data and forwards them to the browser SDP `fetch`; a live preflight for that exact endpoint allows only `authorization,content-type`.

## Solution Assessment

Yes. The narrow maintainable direction is provider-local: keep server-side attribution, but do not return `originator`, `version`, or other server-only attribution headers in browser SDP `offerHeaders`.
