## Summary

Keep open. Current main and v2026.5.2 still put OpenClaw attribution headers into the browser-visible OpenAI WebRTC SDP offer path, and a live OpenAI CORS preflight allows only `authorization,content-type`, so this is a real current Talk bug rather than a duplicate of the earlier CSP/audio-input cleanup in #73427.

## What This Changes

The issue asks OpenClaw to stop sending OpenClaw attribution headers from the Control UI browser when posting OpenAI Realtime WebRTC SDP offers directly to `api.openai.com`.

## Best Possible Solution

Remove OpenAI attribution headers from the browser-side WebRTC SDP offer response path, keep Gateway-side attribution intact, update focused provider/UI tests, and add a user-facing changelog fix entry.

## Reproduction Assessment

Yes. Source shows `talk.realtime.session` can return `originator`/`version` in `offerHeaders` and the UI forwards them to the direct OpenAI SDP `fetch`; a live CORS preflight against that endpoint confirms OpenAI allows only `authorization,content-type`.

## Solution Assessment

Yes. The narrow maintainable fix is provider-local: stop returning OpenAI attribution headers in `offerHeaders` for the browser SDP exchange, while leaving server-side attribution and the generic WebRTC session contract alone.
