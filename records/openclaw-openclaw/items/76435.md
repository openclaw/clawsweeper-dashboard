## Summary

Keep open. Current main and the latest release still put OpenClaw attribution headers into the browser-visible OpenAI Realtime WebRTC SDP offer path, and a live CORS preflight for that endpoint only allows `authorization,content-type`.

## What This Changes

The issue asks OpenClaw to stop sending the `originator` attribution header from the Control UI browser SDP POST to OpenAI Realtime while keeping Gateway-side attribution intact.

## Best Possible Solution

Remove OpenAI attribution headers from the browser-side WebRTC SDP offer response path, update focused provider/UI regression tests, and add a user-facing changelog fix entry.

## Reproduction Assessment

Yes. Current main returns `originator`/`version` in OpenAI browser `offerHeaders`, the UI forwards them to the direct SDP `fetch`, and the live OpenAI CORS preflight rejects those requested headers.

## Solution Assessment

Yes. The narrow maintainable fix is provider-local: stop returning OpenAI attribution headers for the browser SDP exchange while leaving the generic WebRTC contract and Gateway-side OpenAI attribution intact.
