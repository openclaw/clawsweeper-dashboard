## Summary

Keep open. Current main and v2026.5.2 still expose OpenAI attribution headers to the browser WebRTC SDP offer path, a live CORS preflight rejects those headers, and open PR #76457 is already the linked fix attempt that should land or be replaced before this issue closes.

## What This Changes

Reports that Control UI Talk with OpenAI WebRTC sends OpenClaw attribution headers in the browser SDP POST, causing OpenAI CORS preflight failure.

## Best Possible Solution

Review and land PR #76457, or an equivalent provider-local fix that strips browser-only SDP offer headers while preserving Gateway-side OpenAI attribution.

## Reproduction Assessment

Yes. Source shows the OpenAI browser session can return `originator`/`version` in `offerHeaders` and the UI forwards them into browser `fetch`; a live preflight against the OpenAI calls endpoint allows only `authorization,content-type`.

## Solution Assessment

Yes. The narrowest maintainable fix is in the OpenAI realtime provider's browser offer-header filter, with focused provider/UI tests; a protocol change or config knob is unnecessary.
