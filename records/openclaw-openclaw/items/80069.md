## Summary

Keep open: the PR targets the right voice-call resolver for the linked session-continuity bug, but it needs after-fix real behavior proof and a small agent-id normalization fix before merge.

## What This Changes

The branch changes `resolveVoiceCallSessionKey` to generate `agent:<agentId>:voice:*` keys for per-phone and per-call voice-call sessions and updates resolver unit tests.

## Best Possible Solution

Keep the fix in the voice-call-owned resolver, but normalize the agent id with the same routing contract used by core and attach redacted real restart/session-continuity proof before merge.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: current main writes generated raw `voice:*` keys and startup migration canonicalizes bare keys to agent-scoped keys. I did not run a live telephony restart flow in this read-only review.

## Solution Assessment

No, not quite: the resolver is the right owner boundary, but the PR should normalize/sanitize the configured agent id before composing canonical keys and needs real behavior proof for the restart path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Normalize agent IDs before composing session keys:** `extensions/voice-call/src/config.ts:729`
  - body: This uses the configured `agentId` verbatim when building `agent:${agentId}:voice:*`. Core session-key code normalizes agent IDs to lowercase/path-safe values, and startup migration canonicalizes store keys with normalized ids. With `agentId: "Reception"` or a value with surrounding whitespace, this patch writes a key such as `agent:Reception:voice:15550001111`; after a Gateway restart the migration can rewrite it to `agent:reception:voice:15550001111`, so the next call looks up the mixed-case key and loses continuity again. Use the same `normalizeAgentId`/`sanitizeAgentId` contract available through the plugin SDK before constructing the prefix.
  - confidence: 0.89
