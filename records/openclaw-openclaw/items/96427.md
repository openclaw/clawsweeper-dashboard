## Summary

Keep this PR open as a useful narrow fix: current main still leaves stale run-id reverse lookups after active-run rebinding, and the branch fixes the private store invariant with focused coverage. No blocking findings were found in the patch.

## What This Changes

The PR updates the in-memory ACP session store so rebinding a session to a new active run removes the old run-id lookup, with a regression test.

## Best Possible Solution

Land the store-level cleanup with the regression test after required CI finishes, keeping one current run id per ACP session in the reverse lookup map.

## Reproduction Assessment

Yes. Source inspection of current main shows setActiveRun can leave the old run id in runIdToSessionId after rebinding; I did not run a local failing test because this review was constrained to read-only inspection.

## Solution Assessment

Yes. The reverse lookup map is private to createInMemorySessionStore, so deleting the previous mapping inside setActiveRun fixes the stale entry at the owner boundary; sweeping only on session removal would leave stale lookups visible until deletion.
