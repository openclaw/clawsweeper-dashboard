## Summary

Keep open. Current main has generic routed-delivery transcript mirroring, but QQBot’s plugin-owned direct dispatcher still sends successful replies through its own delivery callback, and I found no QQBot regression proving short delivered finals are mirrored into session JSONL for WebChat history.

## What This Changes

The issue asks for every successfully delivered QQBot assistant reply, including very short replies, to be present in the durable session history shown by WebChat.

## Best Possible Solution

Persist successful normal QQBot assistant finals through the existing idempotent session-transcript mirror or repair the Pi `message_end` write path, with coverage proving WebChat can reload the reply from session JSONL.

## Reproduction Assessment

No high-confidence live reproduction was run on current main. The source path is reproducible from inspection: QQBot can successfully send through its local dispatcher while WebChat reloads from session JSONL, and there is no QQBot regression test proving the delivered short final is persisted.

## Solution Assessment

Yes, the narrowest maintainable fix is to repair durable transcript ownership for successful QQBot finals rather than adding a WebChat-only display workaround. The existing delivery-mirror helper already has idempotency and duplicate-avoidance behavior that a QQBot-specific or generic channel-turn hook can reuse.
