## Summary

Keep open. Current main still has the QQBot live-delivery path separated from the Pi/session JSONL transcript path, and I found no current test, changelog entry, or landed fix proving successful short QQBot replies are persisted for WebChat history reloads.

## What This Changes

The issue reports that QQBot short replies can be delivered successfully to QQ but omitted from the durable session transcript, making them invisible in WebChat history.

## Best Possible Solution

Make successful normal QQBot assistant replies durable through the Pi transcript path, or add an idempotent `SessionManager.appendMessage` fallback only when Pi did not already persist the assistant turn.

## Reproduction Assessment

No high-confidence live current-main reproduction was run. The source path is reproducible from inspection: QQBot can deliver through the live dispatcher while WebChat reloads only from Pi/session JSONL, and current tests do not cover the combined persisted-transcript case.

## Solution Assessment

Yes. The narrow maintainable fix is to repair durable transcript ownership with regression coverage, not to add a WebChat-only display workaround or a blind mirror that can duplicate Pi-managed turns.
