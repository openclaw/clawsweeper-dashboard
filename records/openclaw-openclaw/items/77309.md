## Summary

Keep open. Current main still separates QQBot live delivery from Pi/WebChat durable transcript persistence, and I found no source, test, or release proof that very short QQBot assistant replies are always written to session JSONL after successful QQ delivery.

## What This Changes

The issue reports that short QQBot assistant replies can be sent successfully to QQ but not persisted to session history, so WebChat cannot show them.

## Best Possible Solution

Persist every successful normal QQBot assistant reply through Pi's durable transcript path, or add a narrowly guarded fallback only when Pi did not already write the assistant turn.

## Reproduction Assessment

No high-confidence live reproduction was established against current main. The source path is clear from QQBot delivery, Pi assistant-text handling, and WebChat's JSONL-only history reload contract, but the missing proof is an executable regression test for a short QQBot reply and the persisted transcript.

## Solution Assessment

Yes, the narrow maintainable direction is to repair the durable transcript ownership path rather than add a WebChat-only display workaround. Any fallback must avoid duplicating assistant turns already written by Pi.
