## Summary

Keep open. Current main still has a source-visible gap between successful QQBot live delivery and the durable Pi/session transcript that WebChat reloads, and the discussion now includes reporter confirmation that the short reply was delivered but never recorded.

## What This Changes

The issue reports that successful short QQBot assistant replies are delivered to QQ but not persisted to the session transcript, so WebChat history cannot show them.

## Best Possible Solution

Persist every successful normal QQBot assistant reply through the durable Pi transcript path, or add a narrowly guarded fallback only when Pi did not already write the assistant turn.

## Reproduction Assessment

No high-confidence live reproduction is available here. The source path is reproducible by inspection: QQBot can deliver a successful block/final reply while WebChat reloads only from the session JSONL, and current tests do not prove the short QQBot reply reaches that transcript.

## Solution Assessment

Yes. The narrowest maintainable fix is to repair durable Pi/session persistence for the normal assistant turn, not to add a WebChat-only history display workaround or blindly mirror all final delivery payloads.
