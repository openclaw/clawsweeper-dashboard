## What This Changes

The issue requests that short QQBot assistant replies successfully delivered to QQ also be written to the session transcript so WebChat history reloads show them.

## Best Possible Solution

Persist every successful normal QQBot assistant reply through the durable Pi transcript path, or add a narrowly guarded SessionManager-based fallback only when Pi did not already write the assistant turn.

## Solution Assessment

Yes, the best direction is to repair durable transcript ownership rather than add a WebChat-only display workaround. Any fallback should be idempotent and use the session manager path so it does not duplicate or sever Pi transcript history.
