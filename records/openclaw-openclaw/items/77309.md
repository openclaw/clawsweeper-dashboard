## Summary

Keep open. Current main still has a plausible QQBot live-delivery path without regression coverage proving very short successful replies are persisted into the durable Pi/session transcript that WebChat reloads from.

## What This Changes

QQBot short assistant replies should be written to the durable session transcript so WebChat history shows every successfully delivered reply.

## Best Possible Solution

Persist every successful normal QQBot assistant reply through Pi's durable transcript path, or add a narrowly guarded SessionManager-backed fallback only when Pi did not already write the assistant turn.

## Reproduction Assessment

No high-confidence live reproduction was run on current main. The source path is clear enough to guide a regression test: deliver a short QQBot assistant reply and assert the durable session transcript/history contains it.

## Solution Assessment

Yes. The maintainable fix is in durable transcript ownership, not a WebChat-only display workaround; any fallback must avoid duplicating Pi-managed assistant turns.
