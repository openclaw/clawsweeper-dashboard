## Summary

Keep open. Current main still has a QQBot path that can send successful live block replies while WebChat relies on the durable Pi/session transcript, and I did not find coverage or a shipped fix proving very short QQBot replies are persisted for history reloads.

## What This Changes

The issue reports that QQBot short assistant replies can be delivered to QQ but missing from the durable session transcript, so WebChat history cannot show them.

## Best Possible Solution

Persist every successful normal QQBot assistant reply through the same durable Pi transcript path, or add a narrowly guarded fallback only when Pi did not already write the assistant turn.

## Reproduction Assessment

No high-confidence live reproduction was run on current main, but the source path is clear: QQBot can deliver via the channel dispatcher while WebChat reloads only from session JSONL. The missing proof is an executable regression test covering a short QQBot reply and the persisted transcript.

## Solution Assessment

Yes, the narrowest maintainable solution is to repair the durable transcript ownership path, not add a WebChat-only display workaround. Any fallback should avoid duplicating Pi-managed assistant turns.
