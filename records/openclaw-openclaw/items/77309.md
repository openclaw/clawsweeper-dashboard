## Summary

Keep open. The report describes a real durability boundary: QQBot can successfully deliver a short assistant reply through the live channel dispatcher while WebChat reloads from the Pi/session JSONL transcript, and current main does not show a regression test or fix proving those delivered QQBot replies are always persisted.

## What This Changes

Ensure successful short QQBot assistant replies are written to durable session history so WebChat can reload them.

## Best Possible Solution

Make durable transcript ownership reliable for normal QQBot agent replies, preferably by fixing the Pi assistant-turn write path or adding an idempotent SessionManager-backed fallback only when Pi did not already persist the assistant turn.

## Reproduction Assessment

No high-confidence live reproduction was run on current main. The source path is clear from inspection: successful QQBot live delivery and WebChat durable history are separate paths, and current tests do not cover the delivered-short-reply-to-session-JSONL case.

## Solution Assessment

Yes, the best direction is a narrow durable-transcript fix, not a WebChat-only display workaround. Any fallback should use `SessionManager.appendMessage` or an existing wrapper and prove it does not duplicate Pi-written assistant turns.

## Repair Work Prompt

Fix the QQBot/WebChat transcript persistence regression from https://github.com/openclaw/openclaw/issues/77309. Observable bug: a successful short QQBot assistant reply can be delivered to QQ (`Sent markdown chunk`, API 200) but be absent from the durable session JSONL, so WebChat history reload cannot show it. Keep the fix narrow: persist every successful normal QQBot assistant reply through the Pi durable transcript path, or add a narrowly guarded fallback only when Pi did not already write the assistant turn. Do not raw-append Pi `type: "message"` entries to JSONL; use `SessionManager.appendMessage` or an existing wrapper that preserves parentId history. Do not add a WebChat-only display workaround, new config option, or QQBot-specific core policy if a generic channel/Pi seam is enough. Add regression coverage for a short text reply and duplicate-prevention/idempotency. Add a user-facing changelog fix entry if behavior changes.
