## Summary

Keep open. Current main still has the separated QQBot live-delivery and Pi durable-transcript paths described by the report, and I found no changelog entry, regression test, or shipped fix proving short QQBot replies are persisted for WebChat history reloads.

## What This Changes

QQBot short assistant replies that are successfully sent should also be persisted to the session transcript so WebChat history can reload them.

## Best Possible Solution

Persist every successful normal QQBot assistant reply through the durable Pi transcript path, or add a narrowly guarded `SessionManager.appendMessage` fallback only when Pi did not already write the assistant turn.

## Reproduction Assessment

No high-confidence live reproduction was run on current main. The source path is reproducible from inspection: QQBot can deliver the short reply through the channel dispatcher while WebChat reloads only from Pi/session JSONL, and current tests do not cover the combined persisted-transcript case.

## Solution Assessment

Yes, the narrowest maintainable fix is to repair durable transcript ownership, not to add a WebChat-only display workaround. Any fallback should be idempotent and use the session manager path so it does not duplicate or sever Pi transcript history.
