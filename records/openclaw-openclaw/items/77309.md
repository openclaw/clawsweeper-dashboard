## Summary

Keep open. Current main still separates successful QQBot live delivery from durable Pi/WebChat transcript persistence, and I found no source change, changelog entry, or regression test proving short QQBot replies are now written to session history.

## What This Changes

The issue requests that successful short QQBot assistant replies be persisted to session JSONL so WebChat history reloads show them.

## Best Possible Solution

Make normal successful QQBot assistant replies durable through the Pi transcript path, with an idempotent SessionManager-backed fallback only if Pi did not already write the assistant turn.

## Reproduction Assessment

No high-confidence live reproduction was run on current main. Source inspection does show the reported path: QQBot can deliver the reply while WebChat reloads only from Pi/session JSONL, and current tests do not cover the combined persisted-transcript case.

## Solution Assessment

Yes. The narrow maintainable fix is to repair durable transcript ownership or add a guarded idempotent append fallback, not to add a WebChat-only display workaround.
