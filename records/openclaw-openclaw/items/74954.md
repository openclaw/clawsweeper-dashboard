## Summary

Keep this PR open: the reported bug is still visible in current main and the branch targets the right provider-auth seam, but the patch still drops the provider-selected model when there is no inherited default model, lacks after-fix real behavior proof, and is currently conflicting.

## What This Changes

The branch changes provider auth/default-model handling and related tests so `setDefaultModel: false` stops returning provider default models as per-agent overrides during agent auth setup.

## Best Possible Solution

Rebase the branch and suppress `agentModelOverride` only when the original config already has an inherited primary, with coverage for both inherited-default and no-default `agents add` paths.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main returns `agentModelOverride` under `setDefaultModel: false`, and `agents add` writes that value into the agent entry.

## Solution Assessment

No. The provider-auth seam is the right place to fix this, but the PR should preserve the provider-selected override when no inherited primary exists instead of removing the override for every deferred provider-auth flow.
