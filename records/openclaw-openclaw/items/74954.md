## Summary

Keep open: the linked bug is valid and current main still has the agent override path, but this PR is not merge-ready because it suppresses the provider default override even when there is no existing default model to inherit, is currently conflicting, and lacks real behavior proof.

## What This Changes

The branch changes provider auth-choice model-default handling and related tests so `setDefaultModel: false` no longer returns a provider default `agentModelOverride` for `openclaw agents add`.

## Best Possible Solution

Rebase onto current main, suppress `agentModelOverride` only when an inherited default model already exists, keep the no-default provider override behavior, add focused `applyAuthChoice` and `agents add` coverage, and require redacted CLI proof before merge.

## Reproduction Assessment

Yes, at source level. Current main returns `agentModelOverride` from provider auth when `setDefaultModel=false`, and `openclaw agents add` writes that override into the agent config; I did not run the CLI because this review is read-only.

## Solution Assessment

No, not as-is. The PR targets the right seam, but the best fix must only suppress the override when there is an existing default model to inherit and must preserve the no-default provider fallback behavior.
