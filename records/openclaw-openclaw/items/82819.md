## Summary

Keep open: the PR targets a real CLI/Gateway SecretRef bug, but the current patch still scopes provider-override owner matching to bundled plugins, so installed external web providers selected with `--provider` can still have their plugin-scoped credentials treated as inactive. The PR also has the protected `maintainer` label, and the repro script prints API-key prefix material that should be removed before merge.

## What This Changes

The PR routes CLI web search/fetch SecretRef resolution through provider-aware command and gateway secret paths, adds protocol support for web provider overrides, preserves legacy Firecrawl fetch credentials, and adds regression/proof coverage.

## Best Possible Solution

Keep the provider-aware command SecretRef design, but make override owner resolution registry-aware across installed plugin origins and remove secret material from the repro output before landing.

## Reproduction Assessment

Yes for review purposes: the linked issue gives a concrete `openclaw infer web search --provider tavily` SecretRef failure, and the remaining PR defects are source-reproducible from the bundled-only owner filters and secret-prefix print. I did not execute the repro because this was a read-only review.

## Solution Assessment

No: the overall direction is the right owner boundary, but the implementation should match the origin-agnostic provider-owner resolution already used by command target selection and should not print any API-key material.
