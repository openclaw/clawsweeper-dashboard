## Summary

Keep open. Current main still resolves thinking hooks only from the startup-active provider registry, while the OpenAI plugin that owns `openai-codex` thinking policy is startup-lazy; the linked open PR #75295 is the implementation candidate and should be reviewed or landed before this issue closes.

## What This Changes

Gateway/provider thinking validation should accept `xhigh` for startup-lazy `openai-codex` GPT-5.4/GPT-5.5 models by resolving provider-owned thinking profiles outside the active startup registry.

## Best Possible Solution

Land PR #75295 or an equivalent generic provider-runtime fallback that resolves provider-owned thinking hooks for enabled startup-lazy providers, with regression coverage for session defaults/rows, `sessions.patch`, and agent command validation, without hardcoding OpenAI Codex policy into core.

## Reproduction Assessment

Yes. The issue and follow-up comments provide concrete OpenClaw 2026.4.27 configs and commands that reject `--thinking xhigh`, and current-main source still shows the same active-registry-only path when the OpenAI provider hook is not loaded.

## Solution Assessment

Yes. A generic provider-runtime fallback is the narrow maintainable direction because it preserves plugin-owned thinking policy and fixes the startup-lazy provider visibility gap for all providers using the same hook pattern.
