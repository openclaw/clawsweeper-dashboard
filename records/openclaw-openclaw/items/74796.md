## Summary

Keep open. Current main still resolves thinking profiles only from the active plugin registry, so a startup-lazy OpenAI provider can still fall back to base levels without `xhigh`; the issue also has open PR #75295 explicitly linked as the implementation candidate, so this should stay open until that PR or an equivalent fix lands.

## What This Changes

Ensure gateway and agent-session thinking validation advertises and accepts `xhigh` for `openai-codex/gpt-5.5` even when the OpenAI provider plugin is enabled but not startup-active.

## Best Possible Solution

Review and land #75295 or an equivalent generic provider-runtime fallback that loads provider-owned thinking hooks for enabled startup-lazy providers, with regression coverage for session defaults/rows, `sessions.patch`, and agent command validation without hardcoding OpenAI Codex rules in shared core.

## Reproduction Assessment

Yes. The issue and follow-up comment give concrete 2026.4.27 config and commands that hard-reject `--thinking xhigh`, and current-main source still shows the same active-registry-only resolver path when the provider hook is absent.

## Solution Assessment

Yes. The linked direction is the narrow maintainable fix: resolve provider-owned thinking policy through a generic runtime provider lookup while preserving active-registry precedence, rather than adding Codex-specific policy to gateway/session callers.
