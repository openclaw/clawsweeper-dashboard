## Summary

Keep open. Current main still resolves provider thinking hooks only from the active startup registry, so the startup-lazy OpenAI Codex provider can still lose its `xhigh` profile; open PR #75295 is already the linked implementation candidate and this issue should stay open until that PR lands or is closed.

## What This Changes

Gateway/session validation should include and accept `xhigh` for `openai-codex/gpt-5.5` even when the OpenAI provider plugin is enabled but not startup-active.

## Best Possible Solution

Land PR #75295 or an equivalent generic provider-runtime fallback that resolves provider-owned thinking hooks for enabled startup-lazy providers, with regression coverage for gateway defaults/session rows, `sessions.patch`, and agent command validation without hardcoding OpenAI Codex rules in shared core.

## Reproduction Assessment

Yes. The issue and follow-up comment provide concrete 2026.4.27 gateway/CLI reproduction steps, and current-main source still shows the active-only provider hook lookup that makes those steps fail when OpenAI is not startup-active.

## Solution Assessment

Yes. A generic fallback from the shared thinking resolver to the provider runtime hook layer is the narrow maintainable fix; hardcoding `openai-codex` or eagerly activating all provider plugins would cut across the existing plugin boundary.
