## Summary

Keep open. Current main still has the provider-profile visibility gap: thinking hooks are resolved only from the active startup registry, while the bundled OpenAI plugin is startup-lazy, so gateway/session validation can still miss the OpenAI Codex `xhigh` profile. The linked PR #75295 is the open implementation candidate and should be reviewed, landed, or closed before this issue is resolved.

## What This Changes

The issue reports that gateway and agent validation for `openai-codex/gpt-5.5` rejects or remaps `xhigh` because provider thinking hooks are unavailable before startup-lazy OpenAI plugin activation.

## Best Possible Solution

Land PR #75295 or an equivalent generic runtime-provider fallback so enabled startup-lazy provider plugins can supply `resolveThinkingProfile` to gateway defaults, session rows, `sessions.patch`, and agent command validation without hardcoding OpenAI Codex behavior in core.

## Reproduction Assessment

Yes. The issue and comments give concrete 2026.4.27 gateway, local, and subagent commands that reject `xhigh`, and current-main source still shows the same failure path when the OpenAI Codex provider hook is absent from the active registry.

## Solution Assessment

Yes. A generic provider-runtime lookup fallback is the narrow maintainable direction because the OpenAI Codex provider already owns the `xhigh` policy; the issue should remain open until the linked PR or an equivalent fix is merged and validated.
