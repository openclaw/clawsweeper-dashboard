## Summary

Keep open. Current main still has the reported provider-profile visibility gap, and open PR #75295 is already the linked implementation candidate for the same gateway/session validation path, so this issue should remain open until that PR lands or is closed.

## What This Changes

The issue requests that gateway/session thinking validation and metadata accept `xhigh` for `openai-codex/gpt-5.5` by resolving the startup-lazy OpenAI Codex provider thinking profile.

## Best Possible Solution

Land PR #75295 or an equivalent generic provider-runtime fallback that resolves provider-owned thinking hooks for enabled startup-lazy providers, with regression coverage for session defaults/rows, `sessions.patch`, and agent command validation, without hardcoding OpenAI Codex policy in gateway/core callers.

## Reproduction Assessment

Yes. The issue and comments provide concrete 2026.4.27 gateway, local agent, subagent, and installed-dist reproductions, and current main source still shows the same active-registry-only resolver path that omits the startup-lazy OpenAI Codex provider profile.

## Solution Assessment

Yes. A shared provider-runtime fallback in the thinking resolver is the narrow maintainable direction because it keeps `xhigh` model policy in the OpenAI Codex provider while fixing gateway/session callers generically.
