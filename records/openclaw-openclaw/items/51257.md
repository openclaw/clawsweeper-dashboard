## Summary

Keep open: current main still has the status display gap, but this PR needs a rebase, real behavior proof, and fixes for selector precedence plus untrusted friendly-name rendering before merge.

## What This Changes

This PR adds `label` and `displayName` fields to the status summary types and renders `displayName ?? label ?? key` in the CLI and TUI recent-session lists.

## Best Possible Solution

Rebase onto current `main`, thread friendly session metadata through the current status contracts, render a sanitized friendly label while preserving a resolvable selector, add focused status/TUI coverage, and avoid closing broader statusline/session-picker work unless it is also handled.

## Reproduction Assessment

Yes. Source inspection on current main shows cron labels are written, but `SessionStatus`/`GatewayStatusSummary` omit those fields and both CLI/TUI status render raw keys; I did not run a live TUI session in this read-only review.

## Solution Assessment

No. The direction is right, but the proposed `displayName`-before-`label` fallback can hide the resolvable label, and the raw friendly-name rendering needs terminal/TUI sanitization before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Prefer the resolvable session label in status output:** `src/commands/status.command.ts:607`
  - body: This displays `displayName` before `label` in the `Key` column, but follow-up session targeting resolves `key`, `sessionId`, or `label`, not arbitrary display names. For sessions with both a channel-derived display name and a unique stored label, status will show an unresolvable value; prefer the stored label before `displayName` or show the selector separately.
  - confidence: 0.86
- **[P2] Sanitize friendly session names before rendering them:** `src/tui/tui-status-summary.ts:79`
  - body: `label` and `displayName` can come from `sessions.patch`, spawn labels, or channel/thread metadata, and labels are only length-trimmed today. Interpolating them directly into CLI/TUI status output lets newlines or ANSI/control sequences alter the rendered status; run the friendly text through the existing terminal/TUI sanitizers before printing it.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The diff introduces unescaped user/channel-provided friendly names into terminal and TUI output.

Concerns:

- **[medium] Terminal control sequences can reach status output:** `src/tui/tui-status-summary.ts:79`
  - body: The new rendering path prints `label`/`displayName` directly, but those values can be user or channel supplied and current label validation does not remove newlines or ANSI/control sequences. This can spoof or obscure CLI/TUI status output unless sanitized before rendering.
  - confidence: 0.84
