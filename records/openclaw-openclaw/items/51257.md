## Summary

Keep open. The PR is a useful implementation candidate for the linked status-label feature, but it is stale against current main's active CLI status renderer, hides resolvable selectors in the Key column, prints untrusted friendly names without terminal sanitization, and still lacks real behavior proof.

## What This Changes

The PR threads session label/displayName metadata into status summary payloads and renders friendly names in CLI and TUI status session lists.

## Best Possible Solution

Land a corrected implementation that carries existing friendly session metadata through the status contracts, renders terminal-safe friendly text, and preserves a resolvable key or label for follow-up commands.

## Reproduction Assessment

Yes. Source inspection on current main shows session entries can store `label`/`displayName` and cron runs set labels, while status payload/types and active CLI/TUI status renderers still omit or render raw keys.

## Solution Assessment

No. Carrying session metadata through status is the right direction, but this patch needs to update the active shared CLI row builder, avoid unresolvable `displayName` values in the Key selector column, sanitize terminal-rendered names, and provide real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Move rendering to the active sessions row builder:** `src/commands/status.command.ts:610`
  - body: Current main renders `openclaw status` through `buildStatusCommandReportData()` and `buildStatusSessionsRows()`, which still use `sess.key`. This edits an older inline row-construction site, so after rebase the CLI status table would still show raw keys unless the display logic moves into the shared row builder.
  - confidence: 0.9
- **[P2] Keep the Key column resolvable:** `src/commands/status.command.ts:610`
  - body: `displayName` is not accepted by session resolution; follow-up commands can target `key`, `sessionId`, or `label`. Showing `displayName` in the Key column can give users a value they cannot pass back to session commands, so prefer a resolvable label/key there or show the display name separately.
  - confidence: 0.91
- **[P2] Sanitize printed session names:** `src/tui/tui-status-summary.ts:79`
  - body: Labels and display names can come from session patches or channel/thread metadata, and `parseSessionLabel` does not reject control characters. Rendering them raw in CLI/TUI status output can inject extra rows or terminal control sequences, so sanitize the chosen friendly name before interpolation.
  - confidence: 0.93

## Security Review
