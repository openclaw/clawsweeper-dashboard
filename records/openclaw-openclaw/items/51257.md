The PR adds optional session label/displayName fields to status summary/TUI types and renders friendly names in CLI and TUI status output.

Land a corrected implementation that carries existing label metadata through the active status contracts, renders sanitized friendly text, and keeps a resolvable key or label visible.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Move friendly names into the active row builder:** `src/commands/status.command.ts:610`
  - body: Current main renders CLI status through buildStatusCommandReportData() and buildStatusSessionsRows(), where Key still comes from sess.key. Updating the old inline map in status.command.ts will not affect CLI output after rebase, so put the friendly-name logic in the shared row builder.
  - confidence: 0.9
- **[P2] Keep the Key column resolvable:** `src/commands/status.command.ts:610`
  - body: sessions.resolve accepts key, sessionId, or label, not displayName. Showing displayName in the Key column can leave users with a visible value they cannot pass back to session commands, so prefer a resolvable label/key there or display displayName separately.
  - confidence: 0.91
- **[P2] Sanitize friendly session names before rendering:** `src/tui/tui-status-summary.ts:79`
  - body: Labels and display names can contain newlines, ANSI escapes, or other control characters from label patches or channel metadata. Raw interpolation into CLI/TUI status can inject rows or terminal controls, so sanitize to single-line terminal text before rendering.
  - confidence: 0.93
