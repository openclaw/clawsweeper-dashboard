## What This Changes

The PR threads optional session `label` and `displayName` fields through status summary/TUI types and renders recent status sessions with a `displayName → label → key` fallback.

## Best Possible Solution

Land a rebased replacement that carries friendly metadata through the active shared status contracts, sanitizes rendered names, preserves a resolvable key or label, and leaves broader statusline/session-picker scope tracked until covered.

## Reproduction Assessment

Yes. Source inspection on current main shows session entries can carry label/displayName metadata, but status summary, CLI status rows, and TUI `/status` omit those fields and render raw keys.

## Solution Assessment

No. The feature direction is sound, but this PR is not the best implementation as-is because it targets a stale CLI path, hides a resolvable selector behind `displayName`, omits sanitization, and lacks after-fix proof.
