## Summary

Keep open. Current main still has the user-facing gap, but the PR is not ready to merge because it updates an older CLI render path, hides resolvable session selectors when `displayName` wins, and prints untrusted friendly names without a single-line terminal sanitizer.

## What This Changes

The PR adds optional `label` and `displayName` fields to status payload/types, populates them from stored session entries, and renders recent sessions with a `displayName ?? label ?? key` fallback in CLI and TUI status output.

## Best Possible Solution

Land a corrected implementation that carries `label` and `displayName` through the status payload, updates the current shared CLI session-row builder and TUI formatter, displays friendly sanitized text without losing a resolvable key or label, and adds focused regression coverage for labeled sessions plus ANSI/control-character inputs.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: create or inspect a `SessionEntry` with `label` or `displayName`, then follow `getStatusSummary` into the CLI/TUI renderers; the fields are not present in `SessionStatus` and both renderers still use `key`.

## Solution Assessment

No. The PR addresses the right product problem, but the patch is not the best fix because it changes a stale CLI site, can hide the usable selector, and lacks terminal-output sanitization for untrusted friendly names.

  - confidence: 0.93
