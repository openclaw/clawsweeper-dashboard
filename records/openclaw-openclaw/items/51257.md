## Summary

Keep open. This PR addresses a real status-display UX gap, but it is not ready to merge because it targets a stale CLI status rendering path on current main, can put non-resolvable display names in the Key column, lacks terminal-output sanitization, and has no external after-fix behavior proof.

## What This Changes

The branch adds optional `label`/`displayName` fields to status payload types and renders those names ahead of raw session keys in CLI and TUI status summaries.

## Best Possible Solution

Land a corrected implementation that threads existing session label/displayName metadata through the active status contracts, renders sanitized friendly text without hiding a resolvable key or label, and explicitly scopes statusline/session-picker follow-up work.

## Reproduction Assessment

Yes. Source inspection on current main shows cron runs can set `sessionEntry.label`, while `SessionStatus`/`GatewayStatusSummary` recent rows still omit label/displayName and both active CLI/TUI status renderers use raw keys.

## Solution Assessment

No. Carrying friendly metadata through status is the right direction, but this patch needs to move the CLI change into the active row builder, preserve a resolvable selector, sanitize terminal-rendered names, and provide real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Move friendly names into the active row builder:** `src/commands/status.command.ts:610`
  - body: Current main renders CLI status sessions through `buildStatusCommandReportData()` and `buildStatusSessionsRows()`, which still uses `sess.key`. Updating the old inline `status.command.ts` map will not change CLI output after rebase, so put the friendly-name logic in the shared row builder instead.
  - confidence: 0.9
- **[P2] Keep the Key column resolvable:** `src/commands/status.command.ts:610`
  - body: `displayName` is not accepted by session resolution; follow-up commands resolve key, sessionId, or label. Showing `displayName` in the Key column can leave users with a visible value they cannot pass back to session commands, so prefer label/key there or show displayName separately.
  - confidence: 0.91
- **[P2] Sanitize friendly session names before rendering:** `src/tui/tui-status-summary.ts:79`
  - body: `label` and `displayName` can come from `sessions.patch` or channel/thread metadata, and label parsing does not reject newlines or ANSI escapes. Interpolating them raw into CLI/TUI status can inject rows or terminal controls; sanitize to single-line terminal text first.
  - confidence: 0.93
