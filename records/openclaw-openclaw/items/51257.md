## Summary

Keep this PR open: the feature gap is still valid on current main, but the proposed patch is stale against the active CLI status path and introduces resolvability and terminal-output safety issues that should be fixed before merge.

## What This Changes

The PR adds optional session `label` and `displayName` fields to status payload/types and renders recent sessions with a `displayName ?? label ?? key` fallback in CLI and TUI status output.

## Best Possible Solution

Land a corrected implementation that projects session `label` and `displayName` into the status payload, updates the current shared CLI row builder and TUI formatter, preserves a resolvable key/label selector, sanitizes friendly names before terminal rendering, and adds focused regression coverage plus a changelog entry.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: create or inspect a `SessionEntry` with `label` or `displayName`, then follow `getStatusSummary` into the CLI/TUI renderers; the fields are absent from the status payload and both active renderers still print `key`.

## Solution Assessment

No. The PR targets the right feature, but the current patch changes a stale CLI rendering site, hides resolvable labels behind non-resolvable display names, and prints untrusted friendly names without terminal sanitization.

  - body: Current main renders `openclaw status` through `buildStatusCommandReportData` and `buildStatusSessionsRows`, which still use `sess.key`. Updating this older row-construction site will not affect the active CLI table after rebase, so move the display-name logic into the shared row builder.

  - confidence: 0.93
