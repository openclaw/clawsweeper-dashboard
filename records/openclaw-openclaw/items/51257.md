## Summary

Keep open: the feature direction is useful and current main still omits friendly session metadata in status output, but this external PR still lacks real behavior proof and has correctness/security blockers in the active CLI renderer, selector semantics, and terminal output handling.

## What This Changes

The branch adds optional `label` and `displayName` fields to status payload/TUI types and renders recent sessions with a `displayName ?? label ?? key` fallback in CLI and TUI status output.

## Best Possible Solution

Land a corrected implementation that threads `label` and `displayName` through the status contracts, renders sanitized friendly names without replacing resolvable selectors, updates the active shared CLI/TUI renderers, and adds focused regression coverage.

## Reproduction Assessment

Yes. Source inspection on current main shows `SessionEntry` can contain `label`/`displayName`, but the status payload and active CLI/TUI status renderers still only carry and render `key`.

## Solution Assessment

No. Carrying existing metadata through status is the right direction, but this patch needs to update the active row builder, preserve a resolvable selector, and sanitize terminal-rendered friendly names before it is maintainable.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Move friendly-name rendering into the active row builder:** `src/commands/status.command.ts:610`
  - body: Current main renders `openclaw status` through `buildStatusCommandReportData()` and `buildStatusSessionsRows()`, which still use `sess.key`. Updating this older row-construction site will not affect the active CLI table after rebase, so the display logic needs to move into the shared row builder.
  - confidence: 0.9
- **[P2] Keep the Key column tied to a resolvable selector:** `src/commands/status.command.ts:610`
  - body: `displayName` is not accepted by session resolution, which accepts `key`, `sessionId`, or `label`. Showing `displayName` in the Key column can leave users with a value that follow-up session commands cannot target, so prefer a resolvable label/key there or show display name separately.
  - confidence: 0.91
- **[P2] Sanitize friendly session names before terminal output:** `src/tui/tui-status-summary.ts:79`
  - body: Labels and display names can come from session patches or channel/thread metadata. Rendering them raw in CLI/TUI status output can inject newlines or ANSI/control sequences, so pass the selected friendly name through the existing single-line terminal sanitizer before interpolation.
  - confidence: 0.93

## Security Review

Status: needs_attention

Summary: The diff prints session `label`/`displayName` values without terminal sanitization, creating a concrete terminal-output injection risk.

Concerns:

- **[medium] Sanitize terminal-rendered session names:** `src/tui/tui-status-summary.ts:79`
  - body: Labels and display names can be user- or channel-controlled; printing them raw in status output can inject newlines or ANSI/control sequences into a terminal. Use `sanitizeTerminalText` or equivalent single-line sanitization and cover control characters/ANSI escapes in regression tests.
  - confidence: 0.92
