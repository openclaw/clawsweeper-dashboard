## Summary

Keep open. Current main still has the reported status-display gap, and this PR is still useful source work, but the patch is not mergeable as-is because it misses the active CLI renderer, prints a non-resolvable `displayName` in the selector column, renders untrusted names without terminal sanitization, and lacks after-fix real behavior proof.

## What This Changes

The PR adds optional session `label` and `displayName` fields to status payload/types and renders recent sessions with a `displayName ?? label ?? key` fallback in CLI and TUI status output.

## Best Possible Solution

Land a corrected implementation that carries `label` and `displayName` through the status contract, renders a sanitized friendly label/key in the active CLI and TUI status paths, preserves a resolvable selector, and adds focused regression coverage.

## Reproduction Assessment

Yes. Source inspection on current main is enough: create or inspect a `SessionEntry` with `label` or `displayName`, then follow `getStatusSummary()` into `buildStatusSessionsRows()` and `formatStatusSummary()`; both current renderers still print only `key`.

## Solution Assessment

No. The PR targets the right feature, but the implementation needs to move display logic into the active shared CLI row builder, avoid non-resolvable display names in the selector column, and sanitize friendly names before rendering.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Move display logic into the active row builder:** `src/commands/status.command.ts:607`
  - body: Current main renders `openclaw status` through `buildStatusCommandReportData()` and `buildStatusSessionsRows()`, which still use `sess.key`. Updating this older row-construction site will not affect the active CLI table after rebase, so move the friendly-name logic into the shared row builder.
  - confidence: 0.88
- **[P2] Keep the printed selector resolvable:** `src/commands/status.command.ts:607`
  - body: The Key column now prefers `displayName`, but gateway resolution accepts `key`, `sessionId`, or `label`, not `displayName`. If a session has both a display name and label, the status output can show a value follow-up commands cannot target; prefer a label/key selector or show display name separately.
  - confidence: 0.91
- **[P2] Sanitize friendly names before terminal rendering:** `src/tui/tui-status-summary.ts:79`
  - body: Session labels and display names can come from session patches or channel/thread metadata. Rendering them raw can inject newlines or ANSI/control sequences into CLI/TUI status output, so pass the selected friendly name through the existing single-line terminal sanitizer before interpolation.
  - confidence: 0.93
