## Summary

Keep open. Current main still has the reported status-display gap, and this external PR is a plausible direction but remains unsafe to merge because it misses the active CLI row builder, can print untrusted friendly names raw, can hide resolvable selectors, and has no real behavior proof.

## What This Changes

The PR adds optional session label/displayName fields to status payload/TUI types and renders recent status sessions with a displayName-to-label-to-key fallback.

## Best Possible Solution

Land a corrected implementation that threads existing friendly metadata through the status contracts, renders sanitized friendly labels without replacing resolvable key/label selectors, updates the shared CLI/TUI row builders, and adds focused regression coverage.

## Reproduction Assessment

Yes, source-level reproduction is high confidence. Current main has stored label/displayName metadata and cron labels, but the status payload/types and active CLI/TUI status renderers still carry and render raw keys.

## Solution Assessment

No. Carrying session metadata through status is the right direction, but this patch needs to update the active shared CLI row builder, avoid unresolvable displayName values in the Key selector column, and sanitize terminal-rendered friendly names.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Move friendly-name rendering into the active row builder:** `src/commands/status.command.ts:610`
  - body: Current main renders `openclaw status` through `buildStatusCommandReportData()` and `buildStatusSessionsRows()`, which still use `sess.key`. Updating this older inline row-construction site will not affect the active CLI table after rebase, so move the display logic into the shared row builder.
  - confidence: 0.9
- **[P2] Keep the Key column tied to a resolvable selector:** `src/commands/status.command.ts:610`
  - body: `displayName` is not accepted by session resolution, which accepts `key`, `sessionId`, or `label`. Showing `displayName` in the Key column can leave users with a value follow-up session commands cannot target, so prefer a resolvable label/key there or show display name separately.
  - confidence: 0.91
- **[P2] Sanitize friendly session names before printing them:** `src/tui/tui-status-summary.ts:79`
  - body: Labels and display names can come from session patches or channel/thread metadata, and `parseSessionLabel` does not reject control characters. Rendering them raw in CLI/TUI status output can inject extra lines or terminal control sequences, so sanitize the chosen friendly name before interpolation.
  - confidence: 0.93

## Security Review

Status: needs_attention

Summary: The diff introduces a concrete terminal-output injection risk by printing session label/displayName values without single-line sanitization.

Concerns:

- **[medium] Sanitize terminal-rendered session names:** `src/tui/tui-status-summary.ts:79`
