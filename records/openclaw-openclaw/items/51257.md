## Summary

Keep open. Current main still has the requested user-facing gap, but this PR is not ready to merge because it targets an old CLI render site, can replace a usable session selector with a non-resolvable display name, and prints untrusted friendly names without terminal sanitization; it also lacks the required user-facing changelog entry.

## What This Changes

The PR adds optional `label` and `displayName` fields to status payload/types, populates them from session entries, and renders recent sessions with a `displayName ?? label ?? key` fallback in CLI and TUI status output.

## Best Possible Solution

Land a corrected implementation that carries `label` and `displayName` through `SessionStatus` and `GatewayStatusSummary`, updates the current shared CLI row builder and TUI `/status` formatter, preserves a resolvable selector when friendly text differs, sanitizes friendly names before terminal output, and adds focused regression coverage plus a changelog entry.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: create or inspect a `SessionEntry` with `label` or `displayName`, then follow `getStatusSummary` into the CLI/TUI renderers; the fields are absent from `SessionStatus`, and both active renderers still print `key`.

## Solution Assessment

No. The PR points at the right product problem, but the best fix should update the current shared CLI renderer, avoid hiding a resolvable key/label behind `displayName`, and sanitize friendly names before terminal rendering.
