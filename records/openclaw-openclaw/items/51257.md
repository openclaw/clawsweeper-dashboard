## Summary

Keep open. The requested status-label feature is still missing on current main, but this PR needs correction before merge because it updates a stale CLI rendering path, can show non-resolvable display names in the Key column, and prints untrusted friendly names without terminal sanitization; it also lacks real after-fix behavior proof.

## What This Changes

The branch adds label/displayName to status summary and TUI types and renders displayName/label ahead of raw session keys in CLI and TUI status session lists.

## Best Possible Solution

Land a corrected implementation that carries existing friendly session metadata through the active status contracts, sanitizes rendered names, preserves a resolvable label/key for follow-up commands, and handles the broader linked issue scope deliberately.

## Reproduction Assessment

Yes. Source inspection on current main shows cron sessions can already store label metadata, while SessionStatus/GatewayStatusSummary omit label/displayName and both active CLI/TUI status renderers still use raw session keys.

## Solution Assessment

No. Carrying metadata through status is the right direction, but this patch updates a stale CLI path, uses displayName where users expect a resolvable Key value, and prints untrusted friendly names without terminal sanitization.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Move rendering to the active status row builder:** `src/commands/status.command.ts:610`
  - body: Current main no longer renders CLI status through this inline session map; statusCommand delegates to buildStatusCommandReportData() and buildStatusSessionsRows(), which still uses sess.key. Move the friendly-name logic into the shared row builder so the CLI path actually changes after rebase.
  - confidence: 0.9
- **[P2] Keep the Key column resolvable:** `src/commands/status.command.ts:610`
  - body: displayName is not accepted by session resolution; follow-up commands can target key, sessionId, or label. Showing displayName in the Key column can give users a value they cannot pass back to session commands, so prefer a resolvable label/key there or show displayName separately.
  - confidence: 0.91
- **[P2] Sanitize friendly session names before printing:** `src/tui/tui-status-summary.ts:79`
  - body: label and displayName can come from session patches or channel/thread metadata, and label parsing does not reject control characters. Rendering them raw in CLI/TUI status output can inject extra rows or terminal control sequences, so sanitize the chosen friendly name before interpolation.
  - confidence: 0.93

## Security Review

Status: needs_attention

Summary: The diff introduces terminal-output injection risk by printing session label/displayName values without single-line sanitization.

Concerns:
