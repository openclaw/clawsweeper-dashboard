## Summary

Keep open. The feature is still useful and current main still omits friendly session metadata from status output, but this PR is not merge-ready because it lacks real behavior proof and has correctness/security blockers around active CLI rendering, selector resolvability, and terminal-safe output.

## What This Changes

The PR adds optional label/displayName fields to status summary and TUI types, then renders displayName or label before the raw key in CLI and TUI status session lists.

## Best Possible Solution

Land a corrected implementation that carries existing friendly session metadata through the status contracts, sanitizes rendered names, and keeps a resolvable label or raw key available for follow-up commands.

## Reproduction Assessment

Yes. Source inspection on current main shows cron sessions can store label metadata, but status payload/types and active CLI/TUI status renderers still omit or render raw session keys.

## Solution Assessment

No. Carrying metadata through status is the right direction, but this patch updates the wrong CLI rendering path, can display non-resolvable displayName values in the Key column, and prints untrusted friendly names without terminal sanitization.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Move rendering to the active status row builder:** `src/commands/status.command.ts:610`
  - body: Current main no longer renders CLI status through this inline session map; statusCommand calls buildStatusCommandReportData(), which delegates to buildStatusSessionsRows() and still uses sess.key. Move the friendly-name logic into the shared row builder so CLI status actually changes after rebase.
  - confidence: 0.9
- **[P2] Keep the Key column resolvable:** `src/commands/status.command.ts:610`
  - body: displayName is not accepted by session resolution; follow-up commands can target key, sessionId, or label. Showing displayName in the Key column can give users a value they cannot pass back to session commands, so prefer a resolvable label/key there or show displayName separately.
  - confidence: 0.91
- **[P2] Sanitize friendly session names before printing:** `src/tui/tui-status-summary.ts:79`
  - body: label and displayName can come from session patches or channel/thread metadata, and label parsing does not reject control characters. Rendering them raw in CLI/TUI status output can inject extra rows or terminal control sequences, so sanitize the chosen friendly name before interpolation.
  - confidence: 0.93

## Security Review

Status: needs_attention

Summary: The diff introduces a terminal-output injection risk by printing session label/displayName values without single-line sanitization.

Concerns:

- **[medium] Sanitize terminal-rendered session names:** `src/tui/tui-status-summary.ts:79`
  - body: label and displayName can be user- or channel-controlled; printing them raw in status output can inject newlines or ANSI/control sequences into a terminal. Use sanitizeTerminalText or equivalent single-line sanitization and add regression coverage for control characters/ANSI escapes.
  - confidence: 0.92
