## Summary

Keep this PR open: the feature gap is still present on current main, but the proposed patch has correctness and terminal-safety blockers and lacks external real behavior proof.

## What This Changes

The branch adds label/displayName fields to status payload types and renders displayName or label before raw session keys in CLI and TUI status session lists.

## Best Possible Solution

Land a corrected implementation that carries existing friendly session metadata through the active status contracts, sanitizes single-line terminal output, preserves a resolvable label/key selector, and handles the broader statusline/session-picker scope deliberately.

## Reproduction Assessment

Yes. Source inspection on current main shows cron runs can set sessionEntry.label, while SessionStatus/GatewayStatusSummary omit label/displayName and both active CLI/TUI status renderers still use raw keys.

## Solution Assessment

No. Carrying metadata through status is the right direction, but this patch updates a stale CLI path, can show a non-resolvable displayName in the Key column, and prints friendly names without terminal sanitization.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Move friendly names into the active row builder:** `src/commands/status.command.ts:610`
  - body: Current main routes CLI status through buildStatusCommandReportData() and buildStatusSessionsRows(), which still renders sess.key. Updating this stale inline map will not change CLI output after rebase, so the friendly-name logic needs to live in the shared row builder.
  - confidence: 0.9
- **[P2] Keep the Key column resolvable:** `src/commands/status.command.ts:610`
  - body: displayName is not accepted by session resolution; follow-up commands resolve key, sessionId, or label. Showing displayName in the Key column can leave users with a visible value they cannot pass back to session commands, so prefer label/key there or show displayName separately.
  - confidence: 0.91
- **[P2] Sanitize friendly session names before rendering:** `src/tui/tui-status-summary.ts:79`
  - body: label/displayName can come from session patches or channel/thread metadata, and label parsing does not reject newlines or ANSI escapes. Interpolating them raw into CLI/TUI status can inject rows or terminal controls; sanitize to single-line terminal text first.
  - confidence: 0.93

## Security Review

Status: needs_attention

Summary: The diff introduces terminal-output injection risk by printing session label/displayName values without single-line sanitization.

Concerns:

- **[medium] Sanitize terminal-rendered session names:** `src/tui/tui-status-summary.ts:79`
  - body: label and displayName can be user- or channel-controlled; printing them raw in status output can inject newlines or ANSI/control sequences into a terminal. Use sanitizeTerminalText or equivalent single-line sanitization and add regression coverage.
  - confidence: 0.92
- **[medium] Sanitize CLI Key column display names:** `src/commands/status.command.ts:610`
  - body: The CLI Key column would also interpolate displayName or label without sanitization, turning a previously raw internal key field into untrusted terminal-rendered text.
  - confidence: 0.9
