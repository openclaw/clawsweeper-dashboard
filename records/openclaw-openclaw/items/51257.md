## Summary

Keep open: current main still lacks the friendly session metadata projection, but this PR is not ready because it misses the active CLI row builder, can show unresolvable display names, prints unsanitized friendly names, and has no real behavior proof.

## What This Changes

The branch adds optional `label` and `displayName` fields to status payload/TUI types and renders recent sessions with a `displayName ?? label ?? key` fallback in CLI and TUI status output.

## Best Possible Solution

Land a corrected implementation that threads existing friendly metadata through the status contracts, renders sanitized friendly names while preserving a resolvable key or label, updates the shared CLI/TUI renderers, and adds focused regression coverage.

## Reproduction Assessment

Yes, source-level reproduction is high confidence. Current main has stored `label`/`displayName` metadata and cron labels, but status payload/types and active CLI/TUI status renderers still only carry and render raw keys.

## Solution Assessment

No. Carrying metadata through status is the right direction, but this patch needs to update the active shared CLI row builder, avoid putting unresolvable display names in the Key selector column, and sanitize terminal-rendered friendly names.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Move friendly-name rendering into the active row builder:** `src/commands/status.command.ts:610`
  - body: Current main renders `openclaw status` through `buildStatusCommandReportData()` and `buildStatusSessionsRows()`, which still use `sess.key`. Updating the older inline rows in this file will not change the active CLI table after rebase, so the display logic needs to move into the shared row builder.
  - confidence: 0.9
- **[P2] Keep the Key column tied to a resolvable selector:** `src/commands/status.command.ts:610`
  - body: Gateway resolution accepts `key`, `sessionId`, or `label`, but not `displayName`. Showing `displayName` in the Key column can leave users with a value that follow-up session commands cannot target, so prefer a resolvable label/key there or show display name separately.
  - confidence: 0.91
- **[P2] Sanitize friendly session names before printing them:** `src/tui/tui-status-summary.ts:79`
  - body: Labels and display names can come from session patches or channel/thread metadata, and `parseSessionLabel` does not reject control characters. Rendering them raw in CLI/TUI status output can inject extra lines or terminal control sequences, so sanitize the chosen friendly name before interpolation.
  - confidence: 0.93

## Security Review

Status: needs_attention

Summary: The diff introduces a concrete terminal-output injection risk by printing session labels/display names without single-line sanitization.
