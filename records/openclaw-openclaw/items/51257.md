## Summary

Keep open. The feature gap remains on current main and this PR targets the right area, but the patch is not ready: it changes a stale CLI rendering site, can show non-resolvable display names as selectors, prints untrusted friendly names without sanitization, and lacks after-fix real behavior proof.

## What This Changes

The branch adds optional session `label` and `displayName` fields to status payload/types and renders recent sessions with a `displayName ?? label ?? key` fallback in CLI and TUI status output.

## Best Possible Solution

Land a corrected implementation that threads `label` and `displayName` through the status contract, renders sanitized friendly names without replacing the resolvable key/label selector, updates the active shared CLI row builder and TUI formatter, and adds focused regression coverage.

## Reproduction Assessment

Yes. Source inspection is enough: a `SessionEntry` can contain `label` or `displayName`, but current main omits those fields from the status payload and both active CLI/TUI status renderers read only `key`.

## Solution Assessment

No. Carrying existing metadata through status is the right direction, but this patch needs to update the active row builder, preserve a resolvable selector, and sanitize terminal-rendered friendly names before it is maintainable.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Move display logic into the active status row builder:** `src/commands/status.command.ts:607`
  - body: Current main renders `openclaw status` through `buildStatusCommandReportData` and `buildStatusSessionsRows`, which still use `sess.key`. Updating this older row-construction site will not affect the active CLI table after rebase, so move the friendly-name logic into the shared row builder.
  - confidence: 0.9
- **[P2] Keep the Key column resolvable:** `src/commands/status.command.ts:607`
  - body: Using `displayName ?? label ?? key` in the Key column can show text that session commands cannot resolve, because gateway resolution accepts `key`, `sessionId`, or `label`, not `displayName`. Prefer a resolvable label/key there, or show display name separately from the selector.
  - confidence: 0.91
- **[P2] Sanitize friendly names before rendering them:** `src/tui/tui-status-summary.ts:79`
  - body: Session labels and display names can come from session patches or channel/thread metadata. Rendering them raw in CLI/TUI status output can inject newlines or ANSI/control sequences, so pass the selected friendly name through the existing single-line terminal sanitizer before interpolation.
  - confidence: 0.93
