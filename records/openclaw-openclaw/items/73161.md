## Summary

Keep open. The stale subagent cancellation fix is still useful and not implemented on current main, but this broad PR is not merge-ready because it mixes unrelated changes and introduces blocking runtime, security-audit, and generated-config-doc issues; the author has already acknowledged the review and plans to split the branch.

## What This Changes

The branch changes task cancellation/runtime packaging, Discord and Telegram liveness and outbound payload handling, message-tool action scoping, failover classification, security trust-model config/audit behavior, `.gitignore`, and coding-agent guidance.

## Best Possible Solution

Split the work into focused PRs: land the narrow stale subagent cancellation guard inversion with its task-registry test, leave current-main runtime packaging and Discord transport-activity handling in place, and reintroduce any trust-model or Telegram/message-tool changes separately with owner review, docs, generated baselines, and targeted tests.

## Reproduction Assessment

Yes. The stale-task bug has a high-confidence focused path: create a running subagent task and mock `killSubagentRunAdmin` to return `{ found: false, killed: false }`; current main returns `cancelled: false` instead of marking the task cancelled. The PR regressions are reproducible by comparing the synthetic wrapper to the real context-engine initializer and by inspecting the changed audit severity.

## Solution Assessment

No. The narrow task-registry guard inversion is the right shape for the stale cancellation bug, but this PR is not the best way to solve it because it bundles unrelated runtime, Discord, Telegram, payload, and security-config changes, including a packaged runtime regression and a security-audit demotion.

Overall correctness: patch is incorrect

- **[P1] Preserve the real subagent runtime initializer:** `scripts/runtime-postbuild.mjs:14-20`
  - body: The synthetic `subagent-registry.runtime.js` wrapper exports an `ensureContextEnginesInitialized` that only flips a local boolean. The real initializer registers the legacy context engine, so replacing it can make packaged subagent flows fail to resolve the default context engine.
  - confidence: 0.86

  - body: Demoting `gateway.trusted_proxies_missing` to `info` under `security.trustModel="personal-assistant"` hides an actionable reverse-proxy spoofing warning. A single-operator deployment still needs a warning when Control UI is exposed through a proxy without configured trusted proxies.
  - confidence: 0.78
- **[P2] Update generated config documentation artifacts:** `src/config/zod-schema.ts:455-462`
  - body: Adding top-level `security.trustModel` changes the public config surface, but the PR does not include the generated config documentation/baseline updates required by `pnpm config:docs:check`.   - confidence: 0.9
