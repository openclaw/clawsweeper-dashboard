## Summary

Keep this issue open. Current main still contains the reported WhatsApp target-boundary bug: non-WhatsApp provider prefixes can be stripped by generic E.164 normalization and then accepted by the WhatsApp outbound resolver, while cron failure delivery still relies on that selected channel resolver and gateway validation only checks that the channel is configured.

## What This Changes

The issue requests preventing cron/config-recovery failure delivery from interpreting Telegram chat IDs as WhatsApp phone numbers.

## Best Possible Solution

Harden the WhatsApp-owned target boundary so foreign provider prefixes such as telegram: and tg: are rejected before generic phone normalization, while preserving valid WhatsApp E.164, JID, group JID, and whatsapp: forms. Add regression coverage for the normalizer, outbound resolver, and cron failure delivery path.

## Reproduction Assessment

Yes. A focused unit check of normalizeWhatsAppTarget("telegram:1234567890") and resolveWhatsAppOutboundTarget with the same to value plus empty or wildcard allowFrom should reproduce the coercion path shown in current source.

## Solution Assessment

No current fix is present on main. The narrow maintainable fix belongs in the WhatsApp target resolver/normalizer boundary, with cron tests proving failure notifications cannot cross provider target formats.
