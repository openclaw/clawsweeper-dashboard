## Summary

Keep open. Current main still renders TUI image content as text placeholders and has no inline image protocol renderer or documented `tui.inlineImages` setting; the active implementation path is the open draft at https://github.com/openclaw/openclaw/pull/49022, not a merged fix.

## What This Changes

Add inline image rendering to `openclaw tui` for image-capable terminals such as iTerm2, Kitty, and Sixel, with graceful fallback and a possible config flag.

## Best Possible Solution

Review and either land or replace https://github.com/openclaw/openclaw/pull/49022 with a TUI-owned inline media renderer that keeps placeholder fallback, adds bounded local media handling, covers capability detection/config/docs, and includes focused tests.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: feed a TUI tool result containing an image content block and `ToolExecutionComponent` renders `[image/png ...]` text, while assistant/history paths still retain only string content.

## Solution Assessment

Unclear. The requested direction is reasonable, but the maintainable fix should be settled through the existing draft PR or a replacement that covers both tool-result and assistant media paths without weakening local-media safety.
