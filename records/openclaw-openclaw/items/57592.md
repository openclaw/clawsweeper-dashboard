## Summary

Keep open. Current main still does not render inline images in the TUI: tool-result image blocks are converted to bracketed text placeholders, assistant/chat rendering remains text-only, and the existing PR #49022 is an open draft implementation candidate rather than merged proof.

## What This Changes

Request support for inline image rendering in `openclaw tui` on iTerm2, Kitty, Sixel, and similar terminals with graceful fallback and a config flag.

## Best Possible Solution

Review and either land or replace PR #49022 with a TUI-owned inline media renderer that preserves placeholder fallback and includes capability detection, bounded local media handling, config/docs/changelog, and focused tests.

## Reproduction Assessment

Yes. A high-confidence static reproduction is available by feeding a TUI tool result with an image content block; current `ToolExecutionComponent` renders it as `[image/png ...]`, while assistant/history paths only retain text blocks.

## Solution Assessment

Unclear. The requested direction is reasonable, but the best implementation should be decided through the existing draft PR or a maintainer-scoped replacement that covers both tool-result and assistant media paths.
