## Summary

Keep open. Current main and the latest release still render TUI image content as text placeholders or `MEDIA:` text, while the matching implementation remains an open draft that needs maintainer product/security decisions and real terminal proof.

## What This Changes

Add inline image rendering to `openclaw tui` for image-capable terminals, with graceful placeholder fallback and a possible `tui.inlineImages` config control.

## Best Possible Solution

Review, repair, land, or replace https://github.com/openclaw/openclaw/pull/49022 with a TUI-owned inline media renderer that preserves placeholder fallback, constrains local media reads, documents any accepted config, and proves behavior in a real image-capable terminal.

## Reproduction Assessment

Yes, source-level: current `ToolExecutionComponent` converts image content into placeholder text, and `image_generate` returns media references without a TUI inline rendering path. I did not run a live image-capable terminal repro in this read-only review.

## Solution Assessment

Unclear. Inline terminal rendering is a plausible direction, but maintainers still need to settle supported protocols, config behavior, fallback semantics, local media safety, and proof expectations through the open draft or a replacement.
