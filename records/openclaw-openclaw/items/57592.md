## Summary

Keep open. Current main still renders TUI image outputs as text placeholders and has no inline terminal-image renderer, while the related implementation path remains an open draft PR with a maintainer label rather than a merged fix.

## What This Changes

Request inline image rendering in the TUI for image-capable terminals such as iTerm2, Kitty, and Sixel, with graceful placeholder fallback and an optional configuration mode.

## Best Possible Solution

Land or replace the existing draft with a TUI-owned inline media renderer that preserves placeholder fallback, defines the supported protocol/config contract, handles local media safely, and adds focused tests and docs.

## Reproduction Assessment

Yes for the current behavior: source inspection shows image content blocks reaching TUI tool results are rendered as `[mime size]` Markdown text on current main. This is a feature gap rather than a broken documented contract.

## Solution Assessment

Unclear. Inline terminal rendering is a plausible solution, but the maintainable path should be settled through the open draft PR or a refreshed replacement that covers capability detection, fallback behavior, config/docs, and local-media safety.
