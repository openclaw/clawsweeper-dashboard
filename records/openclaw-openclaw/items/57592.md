## Summary

Keep open. Current main still does not render TUI images inline: tool-result images become bracketed text placeholders and assistant/chat output flows through text-only markdown components, while an open maintainer-labeled PR already targets this exact TUI display gap.

## What This Changes

The issue requests inline image rendering in `openclaw tui` for iTerm2, Kitty, Sixel, and similar image-capable terminals with a graceful text fallback and config control.

## Best Possible Solution

Review and either land or replace PR #49022 with a TUI-owned renderer that preserves text fallbacks, detects supported terminals, covers config/docs/changelog, and tests tool-result plus assistant image paths.

## Reproduction Assessment

Yes. A high-confidence static reproduction is available: feed the TUI a tool result containing an image block and current `ToolExecutionComponent` renders `[mime size]` text, while assistant/chat rendering strips down to markdown text instead of retaining image blocks.

## Solution Assessment

Unclear. The requested direction is reasonable, but the best implementation should be decided through the existing inline-image PR or a maintainer-scoped replacement rather than a separate cleanup close.
