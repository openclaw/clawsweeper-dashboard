## Summary

Keep open. Current main still degrades TUI image content to text placeholders and has no inline-image config or docs; the concrete implementation path remains an open draft, maintainer-labeled PR, so cleanup should not close the request.

## What This Changes

Request inline terminal image rendering in the TUI for iTerm2, Kitty, and Sixel-capable terminals, with placeholder fallback and an auto/off/on config option.

## Best Possible Solution

Review and either land or replace https://github.com/openclaw/openclaw/pull/49022 with a TUI-owned inline media renderer that preserves placeholder fallback, documents/configures the behavior, and keeps bounded local media handling covered by tests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a TUI tool result containing an image block reaches `ToolExecutionComponent.extractText()`, which emits `[mime size]` text and renders it through Markdown only.

## Solution Assessment

Unclear. Inline rendering is a reasonable direction, but the maintainable fix should be settled through the existing draft PR or a replacement that covers protocol detection, config/docs, and media safety together.
