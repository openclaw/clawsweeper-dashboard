## Summary

Keep open. Current main and the latest release still render TUI image tool-result content as bracketed placeholder text, while the matching implementation path at https://github.com/openclaw/openclaw/pull/49022 remains an open draft that needs maintainer decisions and real terminal proof.

## What This Changes

The issue requests inline image rendering in `openclaw tui` for iTerm2, Kitty, and Sixel-capable terminals, with graceful placeholder fallback and an optional auto/on/off config.

## Best Possible Solution

Land or replace the open draft with a TUI-owned inline media renderer that preserves placeholder fallback, constrains local media reads, documents any accepted config, and proves behavior in at least one real image-capable terminal.

## Reproduction Assessment

Yes, source-level. Current `ToolExecutionComponent` converts image content entries to bracketed placeholder text, though I did not run a live image-capable terminal repro in this read-only review.

## Solution Assessment

Unclear. Inline terminal rendering is plausible, but maintainers still need to choose the supported protocols, config contract, fallback behavior, local media safety model, and real-terminal proof requirements.
