## Summary

Current main has not implemented inline TUI image rendering: tool result images still render as text placeholders, and the only implementation candidate I found is an open draft PR that still needs maintainer review/proof. This should stay open as the canonical TUI feature request rather than being closed as fixed or superseded.

## What This Changes

The issue requests inline image rendering in `openclaw tui` for image-capable terminals such as iTerm2, Kitty, and Sixel, with graceful fallback for unsupported terminals.

## Best Possible Solution

Keep this issue open as the canonical feature request and have maintainers decide the TUI inline-rendering shape, preferably by reviewing or replacing the existing draft implementation with a narrow, proven TUI-only change that preserves unsupported-terminal fallback and avoids unnecessary config surface.

## Reproduction Assessment

Yes at source level: current main routes TUI tool results through `ToolExecutionComponent`, whose image branch renders `[mime size]` placeholder text. I did not run a live image-generating TUI session because this review was read-only.

## Solution Assessment

Unclear: inline rendering in the core TUI is a plausible end state, but the terminal protocol scope, local media safety bounds, and proposed config flag need maintainer product review. The safer path is a narrow rendering implementation with fallback and real terminal proof before deciding on new config.
