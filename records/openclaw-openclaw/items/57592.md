## Summary

Keep open. Current main still renders TUI image content through text/Markdown paths, explicitly converts tool-result image blocks into bracketed placeholders, and has no iTerm2/Kitty/Sixel graphics path or inline-image config. The related draft PR #49022 remains an open implementation candidate, not shipped behavior.

## What This Changes

Request support for inline image rendering in `openclaw tui` on image-capable terminals, with capability detection/configuration and placeholder fallback.

## Best Possible Solution

Keep this issue open and use #49022 as prior work to review, land, or supersede with a core TUI renderer that carries image media through the display model, safely emits terminal-specific image output, preserves placeholder fallback, and documents/tests the behavior.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: pass an image tool-result content block through `ToolExecutionComponent` and it becomes bracketed placeholder text; current TUI assistant/tool display paths have no image-rendering component or protocol output.

## Solution Assessment

Yes for keeping it open. The maintainable path is a core TUI implementation because plugins cannot replace the built-in TUI chat/tool render pipeline; closing or moving this to ClawHub would leave the reported core surface unchanged.
