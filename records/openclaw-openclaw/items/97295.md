## Summary

Keep open: this is a useful candidate for the Feishu invalid-token bug, but it is not merge-ready because invalidation is registered from a module normal sends do not necessarily load, the retry can reuse the already captured SDK client/token cache, persistent token-invalid failures lose Feishu diagnostics, and no real Feishu proof is present.

## What This Changes

The PR adds Feishu 99991663/99991664 invalid-token detection, a cache-clear callback, one retry in `requestFeishuApi`, and retry coverage.

## Best Possible Solution

Move invalid-token recovery to the Feishu client/token owner boundary so every SDK-backed send/media path and streaming-card path invalidates the token it actually uses, retries once, preserves diagnostics, and includes focused tests plus redacted live Feishu proof.

## Reproduction Assessment

Yes at source level, not live: current main retries only rate-limit codes, and PR head can retry after a no-op invalidation or with the same captured SDK client. No live Feishu tenant recovery proof was provided or run in this read-only review.

## Solution Assessment

No: token-invalid classification is the right direction, but the current setter in `streaming-card.ts` plus clearing future client-cache entries is not the best owner-boundary fix. The fix should live with the Feishu client/token owner or make the retry closure use a freshly invalidated token/client path.

Persistent auth failures should stop after the single refresh attempt and wrap the actual second error.
