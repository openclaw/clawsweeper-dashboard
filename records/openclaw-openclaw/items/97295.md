Full review comments:

- **[P1] Clear the SDK tenant-token cache before retrying:** `extensions/feishu/src/client.ts:260`
  - body: This callback only clears OpenClaw's `clientCache`, but the Lark SDK stores `CTenantAccessToken` in its own module-level cache and new `Client` instances default to that cache. After 99991663/99991664, the refreshed client can still read the same stale tenant token and resend the invalid Authorization header, so the retry does not reliably fix the reported failure.
  - confidence: 0.97
- **[P1] Wire streaming-card sends into the retry path:** `extensions/feishu/src/streaming-card.ts:71`
  - body: `FeishuStreamingSession.start` still calls `requestFeishuApi` without `onTokenInvalid`; with the new helper behavior, token-invalid errors without that callback clear caches and then throw. The streaming-card registration therefore does not make card send/reply recovery happen, so this path can still drop the first message instead of retrying with fresh token state.
  - confidence: 0.91
- **[P1] Align token-invalid tests with the required callback:** `extensions/feishu/src/send.retry.test.ts:361`
  - body: The added retry tests call `requestFeishuApi(request, ..., NO_DELAY)` and expect recovery, but the final helper only continues after token-invalid errors when `options.onTokenInvalid` is set. As written, those tests reject on the first 99991663/99991664 instead of proving the branch's callback-based retry path.
  - confidence: 0.95
