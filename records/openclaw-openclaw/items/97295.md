## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.95

Full review comments:

- **[P1] Invalidate the SDK tenant token before retrying:** `extensions/feishu/src/client.ts:270-271`
  - body: The registered clearer only clears OpenClaw's `clientCache`. The Lark SDK defaults new `Client` instances to its shared `internalCache`, and `TokenManager` reads `CTenantAccessToken` from that cache under the appId namespace before injecting `Authorization`, so the retry can reuse the same invalid tenant token. Clear or bypass the SDK tenant-token cache for the affected app before retrying.
  - confidence: 0.96
- **[P1] Retry CardKit token failures where the token is used:** `extensions/feishu/src/streaming-card.ts:67-68`
  - body: This clearer only runs when `requestFeishuApi` sees 99991663/99991664, but the CardKit create/update/replace/note/close paths call `getToken()` and then `fetchWithSsrFGuard` directly. A CardKit invalid-token response still will not clear `tokenCache` or retry the operation.
  - confidence: 0.92
- **[P1] Remove the unused send-target client bindings:** `extensions/feishu/src/send.ts:618-619`
  - body: After the getter refactor, `sendMessageFeishu` and `sendCardFeishu` still destructure `client` without reading it. `tsconfig.extensions.json` enables `noUnusedLocals`, so the extension type/lint lane should fail until those bindings are removed.
  - confidence: 0.97
- **[P2] Limit token-invalid recovery to one retry:** `extensions/feishu/src/comment-shared.ts:221-222`
  - body: The token-invalid branch uses `attempt < FEISHU_SEND_MAX_RETRIES`, and that constant is 2 for rate limits. Persistent 99991663/99991664 failures therefore get two refresh retries after the first failure even though this issue and PR describe a single recovery retry.
  - confidence: 0.9
