## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Invalidate the SDK tenant token before retrying:** `extensions/feishu/src/client.ts:270-271`
  - body: This clearer only drops OpenClaw's `clientCache`. The Lark SDK defaults every new `Client` to its shared `internalCache`, and `TokenManager` reads `CTenantAccessToken` from that cache under the appId namespace before setting `Authorization`, so `opts.getClient()` can recreate a client that still sends the same invalid token. Clear or bypass the SDK cache entry for the affected app before retrying, and prove the second request fetches a fresh tenant token.
  - confidence: 0.95
- **[P2] Do not let token recovery fall out of the retry loop:** `extensions/feishu/src/comment-shared.ts:221-223`
  - body: The token-invalid branch can `continue` even on the final rate-limit attempt because it only checks `retriedTokenInvalid`, not whether the loop has another iteration. A sequence such as two rate-limit retries followed by 99991663 clears caches and exits the loop without running the promised token recovery retry, then throws the generic exhaustion error instead of the Feishu error.
  - confidence: 0.88
- **[P2] Cover CardKit token failures after creation:** `extensions/feishu/src/streaming-card.ts:304-307`
  - body: This one-off retry only handles invalid tokens while creating the CardKit card. The later update, replace, note, close, and cleanup paths still call `getToken()` and `fetchWithSsrFGuard` directly, so a token invalidation during a streaming response can keep dropping visible updates or closeout work without clearing `tokenCache`. Put the direct CardKit fetches behind a shared token-invalid retry helper or cover the same recovery at each direct token use.
  - confidence: 0.86
