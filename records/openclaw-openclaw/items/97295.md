## Summary

Keep open: this focused PR is still needed for the linked Feishu invalid-token bug, but the current patch is not merge-ready because it clears only the streaming-card token cache while normal send paths keep using the cached Lark SDK client/token, and repeated token-invalid failures can lose the Feishu error details.

## What This Changes

The PR adds Feishu 99991663/99991664 invalid-token detection, a cache-clear callback, one retry in `requestFeishuApi`, and retry tests.

## Best Possible Solution

Move invalid-token recovery to the Feishu client/token owner boundary so SDK-backed sends and streaming-card sends invalidate the token they actually use, retry exactly once, preserve diagnostics, and include redacted live Feishu proof.

## Reproduction Assessment

Yes at source level: current main and v2026.6.10 classify only rate-limit codes, while PR head retries a captured SDK client and clears only the streaming-card cache. No live Feishu reproduction was run in this read-only review.

## Solution Assessment

No: the shared retry helper is a plausible layer, but this implementation clears the wrong token owner for normal sends and mishandles persistent token-invalid failures. The narrower maintainable fix belongs at the Feishu client/token owner boundary plus explicit one-retry failure semantics.
