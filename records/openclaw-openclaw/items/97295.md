Full review comments:

- **[P1] Call the real client cache clearer before retrying:** `extensions/feishu/src/client.ts:283-284`
  - body: `requestFeishuApi` invokes registered cache clearers, but this registration only clears OpenClaw's `clientCache`. Normal send paths keep using the captured SDK client, so they never call the `clearClientCache()` logic that tries to expire the SDK tenant token and can retry with the same stale Authorization value.
  - confidence: 0.97
- **[P1] Expire the SDK token under the app namespace:** `extensions/feishu/src/client.ts:272`
  - body: The Lark SDK stores `CTenantAccessToken` with `namespace: this.appId`; this unnamespaced write on a throwaway `appId: "clear"` client does not expire the real app-scoped cached token. The retry can still reuse the stale tenant token unless invalidation targets the account's appId namespace.
  - confidence: 0.95
- **[P2] Limit token-invalid recovery to one retry:** `extensions/feishu/src/comment-shared.ts:221-222`
  - body: The token-invalid branch uses `attempt < FEISHU_SEND_MAX_RETRIES`, so persistent 99991663/99991664 errors get two retries after the first failure even though the issue and PR describe a single recovery retry. Track token-invalid retry consumption separately so permanently invalid credentials do not get extra token churn and API calls.
  - confidence: 0.86
- **[P2] Remove the proof script from plugin source:** `extensions/feishu/src/_dp_feishu_token_retry.sh:1`
  - body: This adds an executable network/test proof helper under `extensions/feishu/src`, which is plugin source/package content rather than runtime or test code. Keep this evidence in the PR body or an external artifact so the Feishu plugin does not ship proof-only scripts.
