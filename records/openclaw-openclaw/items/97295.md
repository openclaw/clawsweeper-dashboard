## Summary

## What This Changes

The PR adds Feishu 99991663/99991664 token-invalid detection, a cache-clear callback registered from streaming-card code, one retry in `requestFeishuApi`, and retry tests.

## Best Possible Solution

Move invalid-token recovery to the Feishu client/token owner boundary so every SDK-backed send/media path and CardKit streaming path invalidates the token it actually uses, retries once, preserves diagnostics, and includes focused tests plus redacted live Feishu proof.

## Reproduction Assessment

Yes at source level, not live: current main and v2026.6.10 classify only Feishu rate-limit codes, and PR head can retry after no invalidation or with the same captured SDK client/token cache. No live Feishu tenant recovery proof was provided in the PR body or comments.

## Solution Assessment

No: token-invalid classification is the right direction, but the setter in `streaming-card.ts` plus `clearClientCache()` is not the best owner-boundary fix for SDK-backed sends. The retry should invalidate the token cache actually used by the request or rebuild the request around a fresh token/client path.
