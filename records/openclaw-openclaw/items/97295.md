## Summary

Keep open: the PR is still not merge-ready because it clears OpenClaw's client map without invalidating the Lark SDK tenant-token cache used by the captured client, persistent token-invalid failures still lose Feishu diagnostics, and the posted proof does not show a real invalid-token send recovering.

## What This Changes

The branch adds Feishu token-invalid code detection, a global cache-clear callback registered from Feishu client/streaming-card modules, token-invalid retry handling in `requestFeishuApi`, and retry tests.

## Best Possible Solution

Land one canonical Feishu-local invalid-token recovery that invalidates the actual SDK tenant-token cache or re-resolves a fresh client/request on retry, covers streaming-card token cache separately, preserves error details, and includes redacted live recovery proof.

## Reproduction Assessment

Yes at source level: current main retries only Feishu rate-limit codes, and PR head still retries the same captured SDK client whose token manager reads cached tenant tokens. I did not run a live Feishu tenant invalid-token scenario in this read-only review.

## Solution Assessment

No: token-invalid classification belongs in the Feishu plugin, but clearing only `clientCache` through a global setter is not the best fix for SDK-backed sends. The safer fix is to invalidate the SDK token cache or rebuild/re-resolve the client for the retry and preserve diagnostics.
