The PR adds Feishu token-invalid code detection, a module-level token-cache clear callback, a retry branch in `requestFeishuApi`, and focused retry tests.

## Best Possible Solution

Invalidate the actual Feishu token owner used by each request path, retry exactly once, preserve original Feishu diagnostics on failure, and add focused tests plus redacted live Feishu proof.

## Reproduction Assessment

Yes at source level for the review findings: the PR retries the same SDK client closure while the Feishu SDK caches `tenant_access_token` internally, and persistent 99991663/99991664 errors bypass the original error wrapper after the first retry. No live Feishu reproduction was run in this read-only review.

## Solution Assessment

No: the shared retry helper is a plausible layer, but the current callback clears only the streaming-card cache while SDK sends use a different token cache. The better fix is token-owner invalidation plus explicit one-retry failure semantics.
