## Summary

Keep open: this PR targets the right P1 memory migration bug and includes credible real-runtime proof, but the patch adds legacy sidecar import to steady-state manager startup. That is not the best migration boundary under the repository’s stored-state policy and carries stale memory-state risk, so it needs maintainer review or a repair before merge.

## What This Changes

The branch adds an optional legacy sidecar SQLite import to memory schema setup, passes `memory/<agentId>.sqlite` from the memory-core manager, and adds a sidecar-import regression test.

## Best Possible Solution

Handle the legacy sidecar as a one-shot doctor/update migration with verification and skip markers, or get explicit maintainer approval for a narrowly bounded startup migration with tests for already rebuilt and intentionally cleared per-agent databases.

## Reproduction Assessment

Yes. Current main routes memory to the per-agent SQLite DB and only migrates legacy tables inside the opened DB, while https://github.com/openclaw/openclaw/issues/95495 provides upgrade evidence that the old sidecar remains intact but inactive.

## Solution Assessment

No. The row copy targets the right data, but wiring old sidecar import into steady-state runtime startup is a weaker layer than a one-shot doctor/update migration or an explicitly approved startup migration contract.
