## Label Justifications

- `P1`: The reported first outbound reply deterministically splits one real DM peer into two client-visible session identities and can make session reset target the wrong entry.
- `impact:session-state`: Inbound traffic and outbound delivery-mirror text are persisted under conflicting direct and group session keys for the same peer.

## Summary

Current main still contains the source-reproducible outbound fallback that turns a plugin-resolved `group` target into a group session unless the plugin is channel-only. The issue has a focused, clean, mergeable closing PR, so it remains necessary and should stay open until that PR is reviewed and merged or replaced.

## What This Changes

Make outbound replies for direct-only channel plugins reuse direct session routing instead of creating a second group-keyed session for the same peer.

## Best Possible Solution

Land a generic capability-constrained fallback that converts a resolved group guess to direct only when the plugin declares direct support and neither group nor channel support, with regression coverage for unresolved and pre-resolved targets and preferably a live WeChat session-key check.

## Reproduction Assessment

Yes at source level: current main clearly carries the target-kind-to-session-key path that produces the conflicting group entry for a direct-only plugin. This read-only review did not perform a live WeChat reproduction.

## Solution Assessment

Yes, the linked direction is the narrowest maintainable core fix: use the plugin's declared chat-type capabilities to correct an impossible group result, rather than hardcoding WeChat or adding session-store lookup to the outbound hot path.
