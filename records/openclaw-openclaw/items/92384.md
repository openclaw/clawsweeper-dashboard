## Label Justifications

- `P1`: The current-main routing path deterministically creates the wrong session identity for outbound replies on an affected direct-only channel.
- `impact:session-state`: One WeChat peer can be persisted under conflicting direct and group session keys, causing conversation and reset-state drift.
- `impact:message-loss`: Outbound delivery and conversation-control operations can target the phantom group session rather than the live WeChat DM.

## Summary

Current main still classifies an uninferred bare target as `group`, even when the channel declares direct messaging only, and then persists outbound mirror metadata under that generated group route. The related open report covers full session-key input rather than this issue’s bare native ID path, and no open PR found in the canonical search fixes this specific fallback.

## What This Changes

Fix outbound routing for direct-only WeChat peer IDs so replies and delivery-mirror records reuse the canonical direct session instead of creating a phantom group session.

## Best Possible Solution

Teach the generic resolver to honor an unambiguous single supported chat type—especially `direct`—before its legacy group fallback, add focused regression tests for direct-only plugin IDs, and leave provider-specific canonicalization available through the existing plugin hooks.

## Reproduction Assessment

Yes at source level. Current main demonstrably converts an uninferred bare target to `group`, carries that result into outbound session routing, and persists metadata under the resulting group session key; a live WeChat installation was not exercised during this read-only review.

## Solution Assessment

The issue's plugin-hook diagnosis is correct, but the safest OpenClaw-side repair is narrower than peer-ID session lookup: honor an unambiguous direct-only capability before the fallback and retain plugin hooks for provider-specific grammar.

## Repair Work Prompt

Fix the direct-only channel outbound target classification bug reported in https://github.com/openclaw/openclaw/issues/92384. In `src/infra/outbound/target-resolver.ts`, make the normalized fallback honor an unambiguous single supported plugin chat type, so a plugin whose `capabilities.chatTypes` is only `direct` resolves a bare native ID as a user/direct target rather than `group`. Keep explicit `preferredKind`, plugin `inferTargetChatType`, and explicit user/channel prefixes authoritative. Review `src/infra/outbound/outbound-session.ts` for consistent single-chat-type handling, but do not add peer-ID-only lookup-before-mint logic that could conflate valid direct and group sessions. Add focused regression tests for a direct-only plugin with `looksLikeId` and no inference hook. Treat https://github.com/openclaw/openclaw/issues/81723 as related context, but do not broaden this PR to full agent session-key parsing unless the same focused normalization change demonstrably fixes it. Do not add config, change the plugin SDK contract, edit external plugin source, or edit CHANGELOG.md.
