## Summary

Keep open: the underlying iMessage status bug is source-reproducible, but this PR should not merge as-is because it moves arbitrary channel probe error strings into non-sensitive health snapshots and still lacks live after-fix iMessage proof.

## What This Changes

The PR preserves redacted failed iMessage probe payloads in non-sensitive health snapshots, adds health/status regression coverage, and restores small CLI guidance output behavior.

## Best Possible Solution

Preserve the iMessage failure signal through an iMessage-owned or plugin-approved sanitized status field while keeping arbitrary probe diagnostics out of non-sensitive health snapshots.

## Reproduction Assessment

Yes at source level: a failing iMessage `chats.list` probe returns `{ ok: false, error }`, current main removes that probe when `includeSensitive=false`, and the formatter then falls back to `configured`. I did not live-run a macOS denied-Full-Disk-Access setup.

## Solution Assessment

No. Keeping the failure signal is the right direction, but copying generic `probe.error` strings into non-sensitive snapshots is broader than the safe redaction boundary; the safer fix should be iMessage-scoped or plugin-sanitized.
