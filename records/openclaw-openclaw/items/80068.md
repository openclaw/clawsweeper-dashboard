## Summary

Keep open: the bug fix is valid and the latest proof is sufficient, but the patch hard-codes iMessage-specific redaction and remediation policy inside core health code, which should stay plugin-agnostic.

## What This Changes

The PR changes iMessage RPC/health handling so Full Disk Access probe failures become canonical redacted failures in non-sensitive status output, adds regression tests and a changelog entry, and includes one small msteams test type guard.

## Best Possible Solution

Let the iMessage plugin project its known safe public probe failure through a generic status adapter hook, keeping generic probe errors hidden and keeping core health code free of bundled channel IDs.

## Reproduction Assessment

Yes, at source level. Current main can produce an iMessage `{ ok: false, error }` probe, then remove that probe from non-sensitive health output before formatting, and the linked report plus latest PR proof cover the denied Full Disk Access behavior.

## Solution Assessment

No. The behavior direction is right, but the current implementation puts iMessage-specific remediation policy in core; route the safe public failure projection through a generic plugin-owned status seam instead.
