## Summary

Keep open: the linked bug is source-reproducible and the PR is aimed at the right health/status path, but the current patch broadens non-sensitive health output to include arbitrary failed probe error text and its proof is limited to the test harness.

## What This Changes

The PR preserves failed iMessage probe details in non-sensitive health snapshots, adds health/status regression coverage and a changelog entry, and adjusts CLI help/error guidance.

## Best Possible Solution

Preserve an iMessage failure signal through a plugin-approved sanitized status field or narrowly scoped redacted probe payload, while keeping arbitrary channel probe errors out of non-sensitive health responses and covering that boundary in tests.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes, at source level: an iMessage `chats.list` probe failure returns `{ ok: false, error }`, current main removes that probe from non-sensitive snapshots, and the formatter can then fall back to configured output.

## Solution Assessment

Is this the best way to solve the issue? No in current form: preserving the failure signal is right, but copying generic probe `error` strings into non-sensitive snapshots is broader than the safe redaction boundary; the narrower fix should use plugin-approved sanitization or an iMessage-specific safe failure field.
