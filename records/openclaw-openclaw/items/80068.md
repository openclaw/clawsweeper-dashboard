## Summary

Keep this PR open: it targets the right source-reproducible iMessage health bug, but the current patch broadens non-sensitive health output by copying arbitrary failed probe errors, and the supplied proof is test-harness-only rather than after-fix real behavior proof.

## What This Changes

The PR preserves redacted failed probe details in non-sensitive health snapshots, adds iMessage health/status regression coverage, restores two CLI guidance behaviors, and updates the changelog.

## Best Possible Solution

Keep the focused health fix, but tighten it so includeSensitive=false only surfaces public/redacted probe failure text, then add real after-fix terminal/log proof for the status or health command path.

## Reproduction Assessment

Yes, at source level: current main can run the iMessage probe and then remove the failed probe payload from non-sensitive health output before the formatter sees it. I did not live-run a macOS denied-Full-Disk-Access setup.

## Solution Assessment

No, not as written: preserving a redacted failed probe is the right direction, but copying arbitrary probe error strings into non-sensitive snapshots weakens the existing redaction boundary. A safer fix should only expose plugin-provided public error text or apply a stronger shared redaction path before formatting.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Do not copy arbitrary probe errors into public health:** `src/commands/health.ts:93`
  - body: `buildNonSensitiveProbeFailure` now runs for every channel plugin and copies any failed probe's `error` into `includeSensitive: false` health snapshots. The previous path deliberately removed raw probe payloads there, and the probe contract only says `error?: string`, so bundled or third-party probes can include local paths, URLs, or credential-shaped details. Please only surface an explicitly public/redacted probe failure or pass the text through the shared non-sensitive redaction boundary before storing it.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The patch introduces a medium-risk data exposure by copying generic probe error text into non-sensitive health snapshots.

Concerns:

- **[medium] Raw probe errors can reach read-scoped health output:** `src/commands/health.ts:93`

Because plugin probe errors are not guaranteed public-safe, this can expose paths, URLs, or secrets to operator.read clients.
