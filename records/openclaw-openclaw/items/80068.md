## Summary

Keep open: this PR targets a valid source-reproducible bug from https://github.com/openclaw/openclaw/issues/79795, but the current patch is not ready because it exposes generic failed-probe error strings through non-sensitive health snapshots and still lacks live after-fix iMessage Full Disk Access proof.

## What This Changes

This PR preserves redacted failed probe payloads in non-sensitive health snapshots, adds iMessage/status regression tests and a changelog entry, and restores small CLI help/error guidance behavior.

## Best Possible Solution

Preserve the iMessage failure signal through a plugin-owned sanitizer or dedicated public-safe status field, with live redacted terminal/log proof before merge.

## Reproduction Assessment

Yes at source level: current main can run a failing iMessage probe, drop the failed probe for includeSensitive=false, and then format the channel as configured/OK. I did not establish a live macOS Full Disk Access denial run.

## Solution Assessment

No: preserving the failure signal is the right direction, but copying generic probe.error strings into non-sensitive snapshots is broader than the safe redaction boundary. A narrower plugin-approved or iMessage-specific sanitized payload is safer.
