## Summary

Keep open: the linked status bug is source-reproducible, but this PR still copies arbitrary failed-probe error strings into non-sensitive health snapshots and only supplies test-harness proof rather than a live after-fix iMessage/Gateway run.

## What This Changes

This PR preserves redacted failed probe payloads in non-sensitive health snapshots for iMessage, adds health/status regression tests and a changelog entry, and includes small CLI help/error guidance fixes.

## Best Possible Solution

Preserve the iMessage failure signal through a plugin-approved redacted payload or dedicated safe status field, while keeping arbitrary channel probe errors out of non-sensitive health snapshots.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: a failing iMessage probe returns { ok: false, error }, current main drops that probe for includeSensitive=false, and the formatter falls back to configured; I did not establish a live macOS denied-Full-Disk-Access run.

## Solution Assessment

Is this the best way to solve the issue? No: preserving the failure signal is the right direction, but copying generic probe.error strings into non-sensitive snapshots is broader than the safe redaction boundary; the safer fix should be iMessage-scoped or plugin-sanitized.
