## Summary

Keep open: the bug is real and the supplied terminal proof is sufficient, but this head should not merge as-is because the health command embeds iMessage-specific detection and remediation in core instead of using a plugin-owned or generic status seam.

## What This Changes

The PR preserves redacted iMessage Full Disk Access probe failures in health snapshots/status output, records matching RPC diagnostics in the iMessage client, and adds regression tests plus a changelog entry.

## Best Possible Solution

Keep the iMessage-specific Full Disk Access classification and public remediation in the iMessage plugin or behind a generic plugin status hook, with core preserving only plugin-declared safe probe output.

## Reproduction Assessment

Yes at source level: current main runs the probe but strips failed `probe` data from non-sensitive health snapshots, and the formatter then falls back to configured. The PR discussion also includes macOS denied-Full-Disk-Access terminal proof for the after-fix path.

## Solution Assessment

No as implemented. The behavior direction is right, but the current solution hard-codes iMessage policy in core; a plugin-owned safe projection or generic status hook is the narrower maintainable fix.
