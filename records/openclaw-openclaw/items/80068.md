## Summary

Keep open: the bug is valid and the branch now covers the old banner-to-generic-exit path, but it still conflicts with the maintainer decision to fix raw imsg stdout in imsg and can miss the clean JSON-RPC Full Disk Access error shape; after-fix proof is also still test-harness only.

## What This Changes

The PR preserves a canonical iMessage Full Disk Access probe failure in non-sensitive health snapshots/status output, adds iMessage client diagnostic normalization, regression tests, a changelog entry, and one msteams test type narrowing.

## Best Possible Solution

Keep the fix at the iMessage/imsg boundary: imsg should return clean JSON-RPC Full Disk Access errors, and OpenClaw should project that typed/redacted failure through non-sensitive health/status without raw diagnostics.

## Reproduction Assessment

Yes at source level: current main runs the probe, strips non-sensitive probe payloads, and the formatter needs `probe.ok === false` to report failure. I did not live-run a macOS denied-Full-Disk-Access setup.

## Solution Assessment

No. Preserving a redacted failed probe is the right direction, but this branch still adds OpenClaw-side banner classification that related maintainer review rejected and its allowlist can miss the intended clean imsg JSON-RPC Full Disk Access error shape.
