## Summary

Keep open: the fix direction matches the source-level root cause, but the patch currently re-exposes generic failed probe error text in non-admin health snapshots, and the supplied real behavior proof is only test-harness output.

## What This Changes

The PR keeps a redacted failed probe in non-sensitive health snapshots, adds iMessage/status regression tests, restores two CLI guidance expectations, and adds a changelog entry.

## Best Possible Solution

Preserve an iMessage failure signal through a plugin-approved redacted payload or equivalent safe status field, while keeping arbitrary channel probe errors out of non-sensitive health snapshots and covering both the iMessage status line and redaction boundary in tests.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: a failing iMessage `probeAccount` returns `{ ok: false, error }`, current main drops that probe for `includeSensitive: false`, and the formatter then falls back to configured/OK; I did not live-run macOS with Full Disk Access denied.

## Solution Assessment

Is this the best way to solve the issue? No in current form: preserving the failure signal is right, but copying generic probe `error` strings into non-sensitive snapshots is broader than the safe redaction boundary; the narrower fix should use plugin-approved sanitization or a dedicated safe failure field.
