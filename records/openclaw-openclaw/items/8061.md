## Summary

Keep open. Current main still exposes only the global subagent concurrency cap, while the requested per-target cap is missing from the per-agent config schema, generated schema, docs, and both native and ACP spawn dispatch paths. The issue discussion includes a second user with the same problem, and the related per-agent main-lane PR was closed unmerged and did not cover this subagent target cap.

## What This Changes

Add an optional per-agent subagent concurrency cap so one target agent can serialize spawned runs while other subagents remain parallel.

## Best Possible Solution

Implement a focused core config/runtime change under the existing per-agent subagents namespace, document and validate it, and enforce the target-specific limit alongside the global subagent lane cap for all spawn runtimes.

## Reproduction Assessment

Not applicable as a feature request, but the missing capability is verifiable with high confidence by inspecting the strict schema and runtime dispatch path: agents.list[] has no accepted per-target maxConcurrent and every spawn uses the shared subagent lane.

## Solution Assessment

Yes, with a small config-shape adjustment. The maintainable path is to use agents.list[].subagents.maxConcurrent, matching existing per-agent subagent settings, rather than adding a broader top-level field or solving unrelated lane policies.
