## Summary

Keep open: current main and v2026.6.6 still expose only a global subagent concurrency cap and a fixed `subagent` queue lane, so the requested target-agent cap remains missing and needs maintainer product direction before implementation.

## What This Changes

Request to add a per-target-agent subagent concurrency cap so one configured subagent can be serialized without limiting all subagent work globally.

## Best Possible Solution

Define one validated target-aware subagent limit under the per-agent subagent namespace, enforce it for native and ACP spawns beneath the global cap, and document precedence with focused serialization and cross-target parallelism tests.

## Reproduction Assessment

Yes, at source level. Current main and v2026.6.6 expose one global subagent resolver and one fixed subagent lane, while the strict per-agent type and schema expose no target-specific concurrency field.

## Solution Assessment

Unclear as originally proposed. The target-specific cap is the right capability, but the public key should likely fit the existing `agents.list[].subagents.*` namespace and must compose with related scheduler controls.
