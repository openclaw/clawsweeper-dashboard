## Summary

Keep open: current main and v2026.6.10 still only expose a global `agents.defaults.subagents.maxConcurrent` cap and fixed `subagent` lane, while this issue asks for a target-agent-specific cap with confirmed user demand.

## What This Changes

The issue requests a per-agent subagent concurrency limit so one configured target agent can be serialized without globally throttling all subagents.

## Best Possible Solution

Add one validated per-target-agent subagent cap, likely under `agents.list[].subagents`, and enforce it alongside the global cap for native and ACP spawns with docs and focused tests.

## Reproduction Assessment

Yes, at source level: current main has a strict per-agent schema with no target cap, one global subagent resolver, and fixed native/ACP dispatch to the `subagent` lane. I did not run a live TTS/local-model workload.

## Solution Assessment

Unclear as originally proposed: the capability is well-scoped, but the public key should likely fit the existing `agents.list[].subagents.*` namespace and must compose with related lane and scheduler controls.
