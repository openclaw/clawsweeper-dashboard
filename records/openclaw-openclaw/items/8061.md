## Summary

Keep open. Current main still has only a global subagent concurrency cap, and the discussion plus related closed PR show this needs maintainer design direction rather than cleanup closure.

## What This Changes

Add an optional per-agent subagent concurrency limit so a target agent such as a local TTS agent can be capped while other subagents remain parallel.

## Best Possible Solution

Add a validated per-target-agent subagent concurrency setting, document it, and enforce it for native and ACP spawns while preserving the global subagent cap.

## Reproduction Assessment

Yes. Source inspection is enough to reproduce the missing capability: strict per-agent config has no cap field, docs expose only the global subagent cap, and runtime dispatch uses one shared subagent lane.

## Solution Assessment

Unclear until maintainers choose the public config contract. The requested direction is valid, but agents.list[].subagents.maxConcurrent appears more consistent with the existing namespace than a top-level agents.list[].maxConcurrent.
