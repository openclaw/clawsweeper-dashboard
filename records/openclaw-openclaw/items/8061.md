## Summary

Keep open. Current main and v2026.5.12 still expose only a global subagent concurrency cap, with no typed, validated, documented, or enforced per-target-agent cap; the related open concurrency items cover adjacent scheduler scopes rather than superseding this request.

## What This Changes

The issue asks for an optional per-agent subagent concurrency cap so a target agent can serialize its own spawned runs while other subagents remain parallel.

## Best Possible Solution

Add a documented per-target subagent concurrency setting, likely under agents.list[].subagents if maintainers keep the existing namespace pattern, with strict schema/type/help/docs updates and native plus ACP sessions_spawn enforcement while preserving the global subagent lane cap.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this requests new config and scheduler behavior. Source inspection gives high-confidence proof that current main and v2026.5.12 lack the requested per-target subagent concurrency cap.

## Solution Assessment

Unclear as proposed. The capability is well scoped, but maintainers should choose the public config shape and define how the per-target cap composes with global subagent lanes and adjacent concurrency controls.
