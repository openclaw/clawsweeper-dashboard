## Summary

Current main and v2026.6.5 still expose only one global subagent concurrency cap; the target-agent limit remains a distinct, useful feature that needs maintainer agreement on its public contract.

## What This Changes

Add a per-agent subagent concurrency limit so resource-heavy target agents can serialize without reducing concurrency for unrelated agents.

## Best Possible Solution

Add one scheduler-owned cap keyed by canonical target-agent identity under the existing per-agent subagents configuration, acquired together with the global subagent budget across native and ACP spawns.

## Reproduction Assessment

Yes, from source with high confidence: agents.list[] has no validated target concurrency field, and both native and ACP child runs use the single globally capped subagent lane. A live overload run was not needed to confirm the missing capability.

## Solution Assessment

No final solution is approved yet. The target-agent limit matches the stated TTS use case, but the proposed top-level field should be replaced by a canonical nested shape with explicit precedence across existing concurrency scopes.
