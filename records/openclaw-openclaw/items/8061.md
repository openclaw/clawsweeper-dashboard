## Summary

Keep open. Current main still has only the global subagent concurrency cap plus a per-session active-child reject cap; it does not provide a typed, validated, documented, or enforced per-target-agent subagent concurrency setting.

## What This Changes

Request a per-target-agent subagent concurrency cap so a target like a local TTS agent can be limited to one active run while other subagents remain parallel.

## Best Possible Solution

Add a validated per-target-agent subagent concurrency setting, likely under agents.list[].subagents, expose it in docs/schema/tool discovery, and enforce it for native and ACP sessions_spawn while preserving the global subagent lane cap.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection on current main gives high-confidence proof that the per-target-agent cap is missing from config, docs, agents_list output, and native/ACP spawn enforcement.

## Solution Assessment

Unclear until maintainers choose the public config contract. The requested direction is valid, but agents.list[].subagents.maxConcurrent appears more consistent with existing per-agent subagent settings than a top-level agents.list[].maxConcurrent.
