## Summary

Current main and the latest release still expose one global subagent concurrency cap, while the strict per-agent configuration has no target-specific cap. The TTS/local-model resource-isolation use case remains concrete, and related open items address different scopes, so this issue is still necessary; maintainer judgment is needed to choose one composable public configuration and scheduling model.

## What This Changes

Add a per-target-agent concurrency limit for subagent spawns so constrained agents can serialize work while other agents remain parallel.

## Best Possible Solution

Define one target-aware limit under `agents.list[].subagents`, compose it as an additional ceiling beneath the existing global subagent cap, and enforce the same target-keyed accounting across native and ACP spawns with focused serialization and cross-target parallelism tests.

## Reproduction Assessment

Yes at source level. Current main has one global resolver and shared subagent lane, while the strict per-agent type and schema expose no target-specific concurrency field; this establishes the missing capability without requiring a live overload run.

## Solution Assessment

The target-specific limit is the right capability, but the issue's top-level `agents.list[].maxConcurrent` proposal is not yet clearly the best shape. Keeping subagent controls together under `agents.list[].subagents` and defining explicit composition with the global ceiling is the narrower, more maintainable direction.
