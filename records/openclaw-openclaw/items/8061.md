## Summary

Keep open. Current main and v2026.5.22 still only expose a global subagent concurrency cap plus per-requester active-child/depth guards; there is no typed, strict-schema, documented, or enforced per-target-agent cap, and the related open concurrency items cover adjacent scopes rather than replacing this request.

## What This Changes

Requests a per-agent subagent concurrency cap so a target agent such as a TTS/local-model worker can be limited while other subagents remain parallel.

## Best Possible Solution

Add a documented, validated per-target subagent concurrency setting, likely under agents.list[].subagents, and enforce it for native and ACP sessions_spawn alongside the existing global cap.

## Reproduction Assessment

Yes, for source-level capability proof: current main and v2026.5.22 lack the requested per-target maxConcurrent field in strict schema, exported types, docs, and runtime lane resolution. I did not attempt a live TTS/local-model reproduction.

## Solution Assessment

Unclear as proposed. The capability is well scoped, but maintainers should choose the public config shape and define how it composes with the global subagent lane, requester active-child limits, and adjacent lane work.
