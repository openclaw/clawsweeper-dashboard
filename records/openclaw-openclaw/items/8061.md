## Summary

Current main and v2026.6.5 still provide only a process-wide subagent concurrency lane, so the concrete request to serialize one constrained target agent while allowing other targets to run concurrently remains unmet; related open items address broader per-parent or lane isolation and do not supersede this narrower capability.

## What This Changes

Add an optional per-target-agent concurrency limit for `sessions_spawn` so constrained agents can serialize without reducing concurrency for other subagents.

## Best Possible Solution

Add a validated `agents.list[].subagents.maxConcurrent` target limit in the canonical spawn scheduler, enforce it for both native and ACP spawns alongside the existing global safety cap, and specify deterministic composition with parent and lane limits before implementation.

## Reproduction Assessment

Yes at source level: current main exposes only `agents.defaults.subagents.maxConcurrent` for one global lane, while strict per-agent config has no target-specific limit. A live overload run was not needed to establish the missing capability.

## Solution Assessment

Unclear. Nesting the option under `agents.list[].subagents` is the narrowest shape consistent with existing ownership, but maintainers must first define how target, parent, and named-lane limits compose.
