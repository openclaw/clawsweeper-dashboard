## Summary

Keep open. Current main and v2026.5.12 still expose only a global subagent concurrency cap; there is no typed, validated, documented, or enforced per-target-agent cap, and the related open concurrency issues cover different scheduler scopes rather than superseding this request.

## What This Changes

The issue requests a per-agent subagent concurrency cap so a target such as a local TTS agent can be limited without globally limiting all subagents.

## Best Possible Solution

Add a documented per-target subagent concurrency setting with agreed config shape, strict schema/type/help/docs updates, optional agents_list visibility, and enforcement for both native and ACP sessions_spawn while preserving the existing global subagent lane cap.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this requests new config and scheduler behavior. Source inspection gives high-confidence proof that current main and v2026.5.12 lack the requested per-target subagent concurrency cap.

## Solution Assessment

Unclear as proposed. The capability is well scoped, but maintainers should choose the public config shape and define how the per-target cap composes with global subagent lanes and adjacent concurrency controls.
