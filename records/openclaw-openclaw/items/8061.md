## Summary

Keep open. Current main and v2026.6.6 still expose one global subagent concurrency cap and no per-target-agent cap, while the reported TTS/local-model isolation use case is concrete and has second-user confirmation.

## What This Changes

The issue requests an optional per-agent subagent concurrency limit so one spawned target agent can serialize its runs without constraining all subagents globally.

## Best Possible Solution

Define one target-aware setting, likely under agents.list[].subagents, and enforce it as an additional ceiling beneath the existing global subagent cap across native and ACP spawns.

## Reproduction Assessment

Yes, at source level. Current main and v2026.6.6 have strict per-agent config without a target-specific concurrency field and one global subagent lane cap.

## Solution Assessment

Not exactly as proposed. The capability is useful, but maintainers should choose a subagent-scoped config shape and precedence with the global cap before implementation.
