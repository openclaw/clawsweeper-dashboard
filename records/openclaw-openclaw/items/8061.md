## Summary

Keep open. Current main still exposes only the global `agents.defaults.subagents.maxConcurrent` subagent lane cap plus per-session active-child limits; there is no typed, validated, documented, or enforced per-target-agent cap, and the related open lane items cover different scheduler scopes.

## What This Changes

The issue requests an optional per-agent subagent concurrency cap so one target agent can be serialized without limiting all subagents globally.

## Best Possible Solution

Add a validated per-target-agent subagent concurrency setting, likely under `agents.list[].subagents`, document it, and enforce it for native and ACP `sessions_spawn` while preserving the global subagent lane cap.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection gives high-confidence proof that the per-target cap is missing from config types, strict schema, docs, and spawn enforcement.

## Solution Assessment

Unclear until maintainers choose the public config contract and scheduler semantics. The requested direction is valid, but `agents.list[].subagents.maxConcurrent` appears more consistent with the existing per-agent subagent namespace than top-level `agents.list[].maxConcurrent`.
