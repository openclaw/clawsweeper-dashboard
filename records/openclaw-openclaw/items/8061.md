## Summary

Keep open. Current main still has only a shared subagent concurrency lane plus per-session active-child limits; there is no typed, validated, documented, discoverable, or enforced per-target-agent subagent concurrency cap, and the adjacent open lane-isolation work tracks a different main-command-lane problem.

## What This Changes

Request a per-target-agent subagent concurrency setting so one agent, such as a local TTS agent, can be limited without restricting other subagents.

## Best Possible Solution

Add a validated per-target-agent subagent concurrency setting, likely under agents.list[].subagents, document it, and enforce it for native and ACP sessions_spawn while preserving the global subagent lane cap.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection gives high-confidence proof that the per-target cap is missing from config types, strict schema, docs, and spawn enforcement.

## Solution Assessment

Unclear until maintainers choose the public config contract and scheduler semantics. The requested direction is valid, but agents.list[].subagents.maxConcurrent appears more consistent with the existing per-agent subagent namespace than a top-level agents.list[].maxConcurrent.
