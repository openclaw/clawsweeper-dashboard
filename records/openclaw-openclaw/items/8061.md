## Summary

Keep open: current main still exposes and enforces only the global subagent concurrency cap, while the requested per-target-agent limit remains absent; the related open items cover different lane scopes rather than this feature.

## What This Changes

The issue requests an optional per-agent subagent concurrency limit so one target agent, such as a local TTS agent, can be serialized while other subagents remain parallel.

## Best Possible Solution

Add a validated per-target-agent subagent cap under the agreed per-agent config namespace, document it, and enforce it alongside the existing global subagent lane limit.

## Reproduction Assessment

Not applicable as a feature request. The missing capability is source-verifiable: current strict config, docs, and spawn dispatch paths expose only the global subagent lane cap.

## Solution Assessment

Unclear until maintainers confirm the public API shape. The narrow maintainable direction appears to be an `agents.list[].subagents.maxConcurrent`-style setting that matches existing per-agent subagent controls rather than a new unrelated top-level key.
