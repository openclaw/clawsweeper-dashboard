## Summary

Keep open. Current main still has only a global subagent concurrency cap; per-agent config, schema, docs, agents_list output, and native/ACP spawn paths do not provide or enforce a per-target subagent maxConcurrent setting.

## What This Changes

Request adding an optional per-agent subagent concurrency limit so one target agent can serialize work while other subagents remain parallel.

## Best Possible Solution

Add a documented per-target subagent concurrency setting with agreed config shape, strict schema/type/help/docs updates, agents_list visibility if useful, and enforcement for both native and ACP sessions_spawn while preserving the existing global subagent lane cap.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this requests new config and scheduler behavior. Source inspection gives high-confidence proof that current main lacks the requested per-target subagent concurrency cap.

## Solution Assessment

Unclear as proposed. The capability is well scoped, but maintainers should choose the public config shape and how the per-target cap composes with the global subagent lane, active-child limits, and adjacent concurrency controls.
