## Summary

Keep open. Current main still only exposes a global subagent concurrency cap plus requester-session active-child limits; the requested per-target-agent cap remains absent, and the related concurrency items cover different scheduler scopes.

## What This Changes

Request to add an optional per-agent subagent concurrency cap in `agents.list` so one target agent can serialize spawns while other subagents remain parallel.

## Best Possible Solution

Add a documented, validated per-target-agent subagent cap under the existing subagents namespace, enforce it for native and ACP `sessions_spawn`, and cover capped and uncapped targets with focused tests.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection gives high-confidence proof that current main lacks the typed, schema, runtime, and docs surface for a per-target subagent cap.

## Solution Assessment

Unclear until maintainers choose the public config contract and scheduler semantics. The requested direction is valid, but `agents.list[].subagents.maxConcurrent` appears more consistent with current per-agent subagent controls than a top-level `agents.list[].maxConcurrent`.
