## Summary

Current main and v2026.6.6 still expose only one global subagent concurrency cap and one fixed `subagent` queue lane; the requested target-agent cap remains a real core config/runtime gap that needs maintainer product direction before implementation.

## What This Changes

The issue requests an optional per-agent subagent concurrency limit so one target subagent can be capped without globally limiting other subagents.

## Best Possible Solution

Define one target-aware subagent limit under the existing per-agent subagent namespace, enforce it as an additional ceiling beneath the global subagent cap for native and ACP spawns, and document precedence with focused serialization and cross-target parallelism tests.

## Reproduction Assessment

Yes at source level. Current main and v2026.6.6 have one global subagent resolver and one fixed subagent lane, while the strict per-agent type and schema expose no target-specific concurrency field.

## Solution Assessment

Unclear as originally proposed. The target-specific limit is the right capability, but `agents.list[].maxConcurrent` is likely too broad; `agents.list[].subagents.maxConcurrent` or a maintainer-selected equivalent would fit the existing namespace better.
