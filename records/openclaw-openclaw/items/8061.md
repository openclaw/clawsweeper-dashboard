## Summary

Keep open: current main and the latest release still expose only a global `agents.defaults.subagents.maxConcurrent` cap and fixed `subagent` dispatch lane, so the requested per-target-agent subagent cap remains missing and needs maintainer config-shape direction.

## What This Changes

Request to let a specific spawned target agent, such as a local TTS agent, have its own subagent concurrency limit while other subagents can still run in parallel.

## Best Possible Solution

Define one validated per-target subagent cap, likely under `agents.list[].subagents.maxConcurrent`, enforce it beneath the global cap for native and ACP spawns, and document/test cross-target parallelism plus capped-target serialization.

## Reproduction Assessment

Yes, at source level. Current main and `v2026.6.8` expose a strict per-agent schema with no target-specific concurrency field and dispatch all native/ACP subagent runs through the fixed `subagent` lane.

## Solution Assessment

Unclear as originally proposed. The target-specific cap is the right capability, but the key likely belongs under the existing `agents.list[].subagents.*` namespace and must compose with adjacent lane and global-cap proposals.
