## Summary

Keep open. Current main still only supports the global subagent concurrency cap; per-agent config, docs, `agents_list`, and native/ACP spawn dispatch do not expose or enforce a per-target-agent cap, and the related open issues cover adjacent but different scheduling scopes.

## What This Changes

Request adding a per-agent subagent concurrency cap so one target agent can be serialized while other subagents remain parallel.

## Best Possible Solution

Add a validated per-target-agent subagent concurrency setting, document it, and enforce it for native and ACP sessions_spawn while preserving the global subagent lane cap.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection gives high-confidence proof of the missing capability: strict per-agent config has no cap field and both native and ACP spawns use the shared subagent lane.

## Solution Assessment

Unclear until maintainers choose the public config contract. The requested direction is valid, but agents.list[].subagents.maxConcurrent appears more consistent with existing per-agent subagent settings than a top-level agents.list[].maxConcurrent.
