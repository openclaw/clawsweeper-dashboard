## Summary

Keep open. Current main still lacks a typed, validated, documented, or enforced per-target-agent subagent concurrency cap, and the existing related items cover adjacent but different scheduler scopes rather than superseding this request.

## What This Changes

The issue requests a per-agent subagent concurrency limit so one target agent can be capped without restricting other subagent targets.

## Best Possible Solution

Add a documented, validated per-target-agent subagent cap under the existing subagents namespace, enforce it for native and ACP `sessions_spawn`, and test capped targets serializing while uncapped targets still run concurrently.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection gives high-confidence proof that current main lacks the typed, schema, runtime, docs, and test surface for a per-target subagent cap.

## Solution Assessment

Unclear until maintainers choose the public config contract and scheduler semantics. The requested direction is valid, but the maintainable shape likely belongs under `agents.list[].subagents.*` rather than a top-level per-agent `maxConcurrent`.
