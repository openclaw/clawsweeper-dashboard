## Summary

Keep open. Current main and the latest shipped release still expose only a global subagent concurrency cap; the requested per-target-agent cap is absent from config schema, typed config, docs, agents_list output, and the native/ACP spawn scheduling path.

## What This Changes

Request to allow a per-target subagent maxConcurrent setting in agents.list so one subagent can be serialized while other subagents remain parallel.

## Best Possible Solution

Add a documented per-target subagent concurrency setting with agreed config shape, strict schema/type/help/docs updates, agents_list visibility if useful, and enforcement for both native and ACP sessions_spawn while preserving the existing global subagent lane cap.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this requests new config and scheduler behavior. Source inspection gives high-confidence proof that current main and v2026.5.12 lack the requested per-target subagent concurrency cap.

## Solution Assessment

Unclear as proposed. The capability is well scoped, but maintainers should choose the public config shape and how the per-target cap composes with global subagent lanes and adjacent concurrency controls.
