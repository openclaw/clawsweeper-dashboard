## Summary

Keep open. Current main still exposes only a global subagent concurrency cap, while per-agent config, strict schema validation, docs, agents_list output, and native/ACP spawn paths do not provide or enforce a per-target subagent maxConcurrent setting; the related open items cover different scheduler scopes and the earlier per-agent lane PR was closed unmerged for owner-backed design direction.

## What This Changes

Request a per-agent subagent concurrency cap so one target agent, such as a local TTS agent, can be limited without reducing parallelism for other subagents.

## Best Possible Solution

Add a documented per-target subagent concurrency setting with agreed config shape, strict schema/type/help/docs updates, and enforcement in native plus ACP spawn paths while preserving the global subagent lane cap.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this requests new config and scheduler behavior. Source inspection gives high-confidence proof that current main lacks the requested per-target subagent concurrency cap.

## Solution Assessment

Unclear as proposed. The capability is well scoped, but maintainers should choose the public config shape and how the per-target cap composes with the global subagent lane and related concurrency controls.
