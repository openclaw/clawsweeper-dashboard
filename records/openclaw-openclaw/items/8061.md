## Summary

Current main and v2026.6.5 still provide only one process-wide subagent concurrency budget, so the concrete target-agent resource-isolation request remains unresolved and is distinct from the related main-lane, per-parent, and lifetime spawn-limit proposals.

## What This Changes

Requests a per-agent subagent concurrency cap so resource-heavy target agents can serialize without restricting other subagents.

## Best Possible Solution

Define one validated per-target-agent subagent concurrency field and enforce it through a shared target-keyed scheduling layer that composes with the existing global `subagent` lane, with native and ACP tests plus documentation.

## Reproduction Assessment

Yes, at source level. Current main and v2026.6.5 expose only `agents.defaults.subagents.maxConcurrent`, while native and ACP spawns both use the shared `subagent` lane and no per-target field exists.

## Solution Assessment

Unclear. Per-target gating is the appropriate capability, but the proposed top-level field is not yet proven to be the best contract; the existing `agents.list[].subagents.*` namespace is more consistent, and scheduler lifecycle semantics need owner approval.
