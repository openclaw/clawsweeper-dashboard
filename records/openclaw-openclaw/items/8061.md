## Summary

Keep open: current main and the latest release still provide only one global subagent concurrency pool, while the related open issues cover top-level agent lanes, per-parent fanout, or lifetime spawn totals rather than this target-agent resource limit.

## What This Changes

Request a per-agent subagent concurrency limit so a constrained target agent can serialize work while other subagents remain parallel.

## Best Possible Solution

Add one typed target-concurrency setting under `agents.list[].subagents`, compose it with the existing global subagent pool in a shared scheduling owner, and prove capped and uncapped targets behave independently for embedded and ACP spawns.

## Reproduction Assessment

Yes, at source level. The strict per-agent schema has no concurrency field, and gateway startup configures only one shared global subagent lane cap.

## Solution Assessment

No for the exact top-level field proposed. The requested behavior is appropriate, but the existing configuration structure favors a subagent-specific field and one shared scheduler covering embedded and ACP paths.
