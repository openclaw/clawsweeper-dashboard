## Summary

Current main still provides only a global subagent concurrency cap, while each `agents.list[]` entry lacks a target-specific limit. The related open reports address different scopes, so this concrete resource-isolation request remains necessary but needs a maintainer decision on configuration shape and limiter precedence.

## What This Changes

Add a per-target-agent concurrency cap for spawned subagents so constrained runtimes can serialize independently while other targets remain parallel.

## Best Possible Solution

Add one shared target-agent concurrency gate, likely configured under `agents.list[].subagents`, and compose it with the global subagent lane and requester-child limit across native and ACP spawning.

## Reproduction Assessment

Yes at source level: current main directly shows one global subagent cap and no per-target-agent configuration or enforcement path. Because this is a missing capability rather than broken documented behavior, a failing runtime reproduction is not required.

## Solution Assessment

No, the exact proposed top-level field is not yet the best-established design. A nested per-agent subagent setting with shared target-keyed enforcement better matches the current configuration and runtime ownership, but maintainers must define precedence first.
