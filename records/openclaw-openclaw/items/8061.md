## Summary

Keep open. Current main still has only the global `agents.defaults.subagents.maxConcurrent` cap plus requester-session active-child limits; there is no typed, validated, documented, or enforced per-target-agent subagent concurrency cap, and nearby concurrency items cover different scheduler scopes.

## What This Changes

Requests an optional per-agent subagent concurrency limit so a target such as a local TTS agent can be serialized without throttling other subagents.

## Best Possible Solution

Add a documented, validated per-target-agent subagent cap under the existing subagents namespace, enforce it for native and ACP `sessions_spawn` alongside the global lane cap, and cover capped and uncapped targets with focused tests.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection provides high-confidence proof that current main lacks the typed, schema, runtime, and docs surface for a per-target subagent cap.

## Solution Assessment

Unclear until maintainers choose the public config contract and scheduler semantics. The requested direction is valid, but `agents.list[].subagents.maxConcurrent` appears more consistent with current per-agent subagent controls than a top-level `agents.list[].maxConcurrent`.
