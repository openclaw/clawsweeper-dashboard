## Summary

Keep open. Current main still exposes only a global subagent concurrency cap, while per-agent config, docs, and native/ACP spawn paths do not provide or enforce a per-target-agent cap; the discussion also shows another user confirmed the same need.

## What This Changes

The issue requests a per-agent subagent concurrency limit so one target agent, such as a local TTS agent, can be capped without restricting all subagents globally.

## Best Possible Solution

Add a validated per-target-agent subagent concurrency setting, document it, and enforce it for native and ACP spawns while preserving the global subagent cap.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection still gives high-confidence proof of the missing capability: strict per-agent config has no cap field, docs expose only the global cap, and runtime dispatch uses the shared subagent lane.

## Solution Assessment

Unclear until maintainers choose the public config contract. The requested direction is valid, but agents.list[].subagents.maxConcurrent appears more consistent with the existing per-agent subagent namespace than a top-level agents.list[].maxConcurrent.
