## Summary

Keep open. Current main still only supports a global subagent concurrency lane plus per-session active-child rejection; it does not provide a typed, validated, documented, discoverable, or enforced per-target-agent subagent concurrency cap.

## What This Changes

Add a per-agent subagent concurrency limit so a target such as a local TTS agent can be capped while other subagents remain parallel.

## Best Possible Solution

Add a validated per-target-agent subagent concurrency setting, likely under agents.list[].subagents, expose it in schema/docs/agents_list, and enforce it for native and ACP sessions_spawn while preserving the global subagent lane cap.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection gives high-confidence proof that the per-target-agent cap is missing from config, docs, discovery output, and native/ACP spawn enforcement.

## Solution Assessment

Unclear until maintainers choose the public config contract. The requested direction is valid, but agents.list[].subagents.maxConcurrent appears more consistent with the existing per-agent subagent namespace than a top-level agents.list[].maxConcurrent.
