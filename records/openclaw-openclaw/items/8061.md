## Summary

Keep open: current main and the latest release still only expose a global `agents.defaults.subagents.maxConcurrent` cap and fixed `subagent` lane, while this issue asks for a target-agent-specific subagent cap with confirmed user demand.

## What This Changes

The issue requests a per-target-agent subagent concurrency cap so one configured subagent can serialize while other subagents remain parallel.

## Best Possible Solution

Define one validated per-target-agent subagent cap, likely under the existing per-agent `subagents` namespace, and enforce it alongside the global cap with docs and focused tests.

## Reproduction Assessment

Yes, at source level. Current main has a strict per-agent schema with no target cap, one global subagent resolver, and fixed native/ACP dispatch to the `subagent` lane; I did not run a live TTS/local-model workload.

## Solution Assessment

Unclear as originally proposed. The capability is well-scoped, but the public key should likely fit `agents.list[].subagents.*` and must compose with related lane and scheduler controls.
