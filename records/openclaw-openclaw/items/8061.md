## Summary

Keep open. Current main and v2026.6.5 still only expose a global subagent concurrency cap plus requester-session active-child limits; there is no typed, strict-schema, documented, or enforced per-target-agent cap, and the related open concurrency items cover adjacent scopes rather than replacing this request.

## What This Changes

The issue requests an optional per-agent subagent concurrency cap so one target agent can be serialized without limiting all subagents globally.

## Best Possible Solution

Add a documented, validated per-target subagent concurrency setting, likely under agents.list[].subagents, and enforce it for both native and ACP sessions_spawn alongside the existing global subagent lane cap.

## Reproduction Assessment

Yes, for source-level capability proof: current main and v2026.6.5 lack the requested per-target maxConcurrent field in strict schema, exported types, docs, and runtime lane resolution. I did not attempt a live TTS/local-model reproduction.

## Solution Assessment

Unclear as proposed. The capability is well scoped, but maintainers should choose the public config shape and define how it composes with the global subagent lane, requester active-child limits, and adjacent lane work.
