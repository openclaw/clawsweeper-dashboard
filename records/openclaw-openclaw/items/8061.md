## Summary

Keep open: current main and v2026.6.8 still only support the global `agents.defaults.subagents.maxConcurrent` cap and a fixed `subagent` queue lane, so the requested per-target-agent cap remains missing and needs maintainer product direction for the public config shape.

## What This Changes

The issue requests an optional per-agent subagent concurrency cap so one target subagent, such as a local TTS worker, can be serialized without globally limiting all subagents.

## Best Possible Solution

Define one validated target-agent subagent cap under the per-agent subagent namespace, enforce it alongside the global cap for native and ACP spawns, document precedence, and test capped-target serialization with cross-target parallelism.

## Reproduction Assessment

Yes, at source level. Current main and v2026.6.8 expose a strict per-agent schema with no target cap, one global subagent resolver, and fixed native/ACP dispatch to the `subagent` lane.

## Solution Assessment

Unclear as originally proposed. The target-specific cap is the right capability, but the public key should likely fit the existing `agents.list[].subagents.*` namespace and must compose with adjacent scheduler controls.
