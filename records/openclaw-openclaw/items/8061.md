## Summary

Keep open. Current main and v2026.5.22 still provide only a global subagent concurrency cap plus requester-session active-child/depth guards; there is no typed, strict-schema, documented, or enforced per-target-agent subagent `maxConcurrent` setting, and related open concurrency items cover adjacent scopes rather than replacing this request.

## What This Changes

Request to add a per-agent subagent concurrency limit so one target agent, such as a local TTS agent, can be serialized while other subagents remain parallel.

## Best Possible Solution

Add a documented per-target subagent concurrency setting, likely under `agents.list[].subagents`, with schema/type/help/docs updates and native plus ACP enforcement alongside the existing global cap.

## Reproduction Assessment

Yes for source-level capability proof: current main lacks the requested per-target `maxConcurrent` field in strict schema, exported types, docs, and runtime lane resolution. I did not attempt a live TTS/local-model reproduction.

## Solution Assessment

Unclear as proposed. The capability is well scoped, but maintainers should choose the public config shape and define how it composes with the global subagent lane, requester active-child limits, and adjacent lane work.
