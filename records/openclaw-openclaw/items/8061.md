## Label Justifications

- `P2`: The missing capability affects resource isolation in real heterogeneous multi-agent deployments but is not an urgent outage or regression.
- `impact:session-state`: The requested control governs simultaneous subagent sessions and target-specific runtime pressure.

## Summary

Current main still provides only a process-wide subagent concurrency cap and a separate per-requester active-child limit; neither limits all concurrent runs targeting one resource-constrained agent, so the concrete TTS/local-model use case remains unmet.

## What This Changes

Add a per-target-agent cap for concurrent subagent execution while leaving other target agents free to use the global pool.

## Best Possible Solution

Add a canonical `agents.list[].subagents.maxConcurrent` target-cap contract, enforce it through shared spawn admission for embedded and ACP runtimes, and document its ordering with the global lane and requester-child limits.

## Reproduction Assessment

Yes. Current main’s strict per-agent schema, global-only resolver, and explicit limit-semantics tests provide a high-confidence source reproduction of the missing per-target capability.

## Solution Assessment

Yes in direction, but not in the exact proposed shape. The established `agents.list[].subagents.*` namespace plus shared native/ACP admission logic is narrower and more maintainable than a top-level field or an inbound command-lane mechanism.
