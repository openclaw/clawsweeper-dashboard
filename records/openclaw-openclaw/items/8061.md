## Summary

Keep open. Current main still exposes only a global subagent concurrency cap and active-child/session guardrails; it has no typed, validated, documented, or enforced per-target-agent subagent concurrency setting, and the related lane/concurrency items cover different scheduler scopes rather than superseding this issue.

## What This Changes

The issue requests a per-agent subagent concurrency limit so one target subagent can be serialized without globally throttling every subagent.

## Best Possible Solution

Add a documented per-target subagent concurrency setting with agreed config shape, schema/help/docs updates, and native plus ACP enforcement while preserving the global subagent lane cap.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this requests new config and scheduler behavior. Source inspection gives high-confidence proof that current main lacks the requested per-target subagent concurrency cap.

## Solution Assessment

Unclear as proposed. The capability is well-scoped, but maintainers should choose the public config shape and how the per-target cap composes with the global subagent lane and related concurrency controls.
