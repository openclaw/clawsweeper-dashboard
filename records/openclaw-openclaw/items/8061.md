## Summary

Current main still provides only one global subagent execution cap, so the resource-isolation use case remains unmet; this issue should stay open for a maintainer decision on the canonical config scope and scheduler composition.

## What This Changes

Add a per-target-agent subagent concurrency limit so a resource-constrained agent can serialize its runs while unrelated agents remain concurrent.

## Best Possible Solution

Define a nested per-agent subagent limit, enforce the minimum of target-specific and global limits through one scheduler-owned target key across native and ACP spawns, and prove uncapped targets still run concurrently.

## Reproduction Assessment

Yes at source level: current main's strict per-agent schema, global-only resolver, gateway lane setup, and public docs establish with high confidence that no target-specific cap exists. This is a missing capability rather than a broken existing contract needing a failing runtime reproduction.

## Solution Assessment

No, the proposed top-level field is not yet the best-supported API. Existing target spawn policy belongs under `agents.list[].subagents`, and enforcement should live in the canonical scheduler instead of adding another generic command-lane configuration path.
