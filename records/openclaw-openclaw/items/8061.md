## Summary

Keep open: current main and v2026.6.6 still expose only a global subagent concurrency cap and a fixed `subagent` lane, while the narrower per-target-agent cap requested here remains missing.

## What This Changes

Request a per-target-agent subagent concurrency cap so a fragile TTS/local-model subagent can be serialized while other subagents continue in parallel.

## Best Possible Solution

Keep this open for a maintainer-chosen config shape, then add a validated per-target subagent concurrency cap enforced alongside the global subagent lane for both native and ACP spawns, with docs and focused tests.

## Reproduction Assessment

Yes, at source level: current main's strict per-agent schema has no per-target subagent concurrency field, and both native and ACP spawned child runs use the fixed `subagent` lane. No live TTS/local-model setup is needed to prove the missing capability.

## Solution Assessment

Unclear as proposed. The capability is valid, but `agents.list[].maxConcurrent` may conflict with foreground agent concurrency, so maintainers should choose whether this belongs under `agents.list[].subagents.maxConcurrent` or another scheduler-owned shape.
