## Summary

Keep open. Current main and the latest release still expose only the global `agents.defaults.subagents.maxConcurrent` lane cap; the requested per-target-agent cap is not typed, accepted by the strict schema, documented, or enforced in the subagent scheduler.

## What This Changes

Add an optional per-agent subagent concurrency cap so one target agent can be limited without reducing concurrency for all subagents.

## Best Possible Solution

Add a validated per-target subagent concurrency setting under the existing per-agent subagent config namespace, enforce it alongside the global subagent lane cap, and document how it differs from per-parent and per-session limits.

## Reproduction Assessment

Yes, from source inspection: a config containing `agents.list[].subagents.maxConcurrent` or `agents.list[].maxConcurrent` has no accepted strict-schema field, and runtime resolution reads only the global default.

## Solution Assessment

Unclear until maintainers choose the public shape. The maintainable direction is likely `agents.list[].subagents.maxConcurrent`, because existing per-agent subagent controls already live under that namespace.
