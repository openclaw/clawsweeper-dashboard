## Summary

Keep open. Current main still only provides a global subagent concurrency lane plus per-session active-child rejection; there is no typed, validated, documented, discoverable, or enforced per-target-agent subagent concurrency cap, and the related open items cover different scheduler scopes.

## What This Changes

The issue requests a per-agent subagent concurrency cap so a target agent such as a local TTS worker can be limited without globally throttling all subagent spawns.

## Best Possible Solution

Add a validated per-target-agent subagent concurrency setting, likely under agents.list[].subagents, expose it in schema/docs/agents_list, and enforce it for native and ACP sessions_spawn while preserving the existing global subagent lane cap.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection gives high-confidence proof that the per-target-agent cap is missing from config, docs, discovery-facing surfaces, and native/ACP spawn enforcement.

## Solution Assessment

Unclear until maintainers choose the public config contract. The requested direction is valid, but agents.list[].subagents.maxConcurrent appears more consistent with the existing per-agent subagent namespace than a top-level agents.list[].maxConcurrent.
