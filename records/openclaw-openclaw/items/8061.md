## Summary

Current main still exposes only a global subagent concurrency cap; the requested per-target-agent resource isolation remains distinct from the related per-parent, per-session-total, named-lane, and inbound-agent lane proposals. The feature is useful, but its new public config shape and composition with native and ACP spawn limits need maintainer product judgment.

## What This Changes

Add a per-target-agent subagent concurrency limit so a constrained agent can serialize without reducing concurrency for unrelated subagents.

## Best Possible Solution

Add one canonical setting, preferably alongside existing `agents.list[].subagents` policy, that limits concurrent runs by resolved target agent while preserving the global subagent pool and applying consistently to native and ACP spawns.

## Reproduction Assessment

Yes at source level. Current main has a single global subagent cap and no typed, validated, or resolved per-target-agent cap, so the reported configuration limitation follows directly from the current implementation.

## Solution Assessment

Not yet. Per-target limiting is the right capability, but the proposed top-level field is less consistent than nesting it with existing per-agent subagent policy, and maintainers must define how it composes with the global pool and ACP sessions.
