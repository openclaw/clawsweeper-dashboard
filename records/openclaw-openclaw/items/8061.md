## Summary

Current main and v2026.6.5 still expose only one process-wide subagent concurrency cap, so the concrete need to serialize a constrained target agent while other targets remain parallel is unmet; the related open reports cover parent caps, named lanes, session totals, or inbound-agent lanes rather than this target-specific limit.

## What This Changes

Requests a per-target-agent subagent concurrency limit so one constrained agent can serialize while other subagents remain parallel.

## Best Possible Solution

Define a nested target-agent cap under the existing per-agent subagent controls, then enforce the minimum of the global and target limits through one shared native/ACP spawn scheduler with focused serialization and cross-target parallelism tests.

## Reproduction Assessment

Yes, at source level. The strict per-agent config surface cannot express a target cap, and every native subagent run uses the single globally capped subagent lane; no runtime execution was needed to establish that capability gap.

## Solution Assessment

No, not exactly as proposed. The capability is appropriate, but a top-level `agents.list[].maxConcurrent` would be ambiguous with top-level run concurrency; nesting it under `agents.list[].subagents` and sharing enforcement across native and ACP spawning is the narrower maintainable design.
