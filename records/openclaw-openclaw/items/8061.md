## Label Justifications

- `P2`: The missing isolation control affects resource-constrained multi-agent deployments but does not make the core runtime unusable.
- `impact:other`: The issue concerns target-specific resource saturation and concurrency isolation, which is a concrete operational impact outside the more specific owned categories.

## Summary

Current main and the latest release still expose only one global subagent concurrency cap, so the concrete TTS/local-model isolation use case remains unmet and is not owned by the adjacent concurrency reports.

## What This Changes

Request a per-target-agent subagent concurrency cap so one constrained agent can serialize runs without reducing concurrency for other agents.

## Best Possible Solution

Add `agents.list[].subagents.maxConcurrent` as the canonical target-agent setting and enforce an additional limiter keyed by resolved target agent across native and ACP spawns while retaining the global subagent lane.

## Reproduction Assessment

Yes at source level. Current main exposes only a global subagent cap and one shared subagent lane, so a target-specific concurrency policy cannot be configured or enforced.

## Solution Assessment

No for the exact proposed top-level field. A nested `agents.list[].subagents.maxConcurrent` setting and target-keyed limiter better match current ownership, but queue lifecycle and ACP parity require maintainer judgment.
