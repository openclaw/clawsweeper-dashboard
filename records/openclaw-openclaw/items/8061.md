## Summary

Keep open. Current main still exposes and enforces only the global subagent concurrency cap, and the related open/closed items cover different lane or per-parent limits rather than this per-target-agent cap.

## What This Changes

The issue requests a per-agent subagent concurrency limit so one target subagent, such as a local TTS agent, can be capped without reducing concurrency for other subagents.

## Best Possible Solution

Add a validated per-target-agent subagent concurrency setting, document it, and enforce it in both native and ACP spawn paths while preserving the existing global subagent lane cap.

## Reproduction Assessment

Yes. Source inspection on current main is enough to reproduce the missing capability: strict per-agent config has no cap field, generated schema omits it, and runtime dispatch uses one global `subagent` lane cap.

## Solution Assessment

Unclear until maintainers choose the config contract. The requested direction is reasonable, but `agents.list[].subagents.maxConcurrent` looks more consistent than the proposed top-level `agents.list[].maxConcurrent`.
