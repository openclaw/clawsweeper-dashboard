## Summary

Keep open: current main and v2026.6.10 still only expose a global subagent concurrency cap, while this issue asks for a target-agent-specific cap with confirmed user demand and no canonical merged fix.

## What This Changes

The issue requests an optional per-agent subagent concurrency limit so one configured target agent can be capped without serializing all other subagents.

## Best Possible Solution

Define one validated per-target-agent subagent cap, likely under the existing per-agent subagents namespace, and enforce it alongside the global cap with docs and focused native plus ACP tests.

## Reproduction Assessment

Yes, at source level. Current main has no per-target cap in the strict per-agent schema or typed config, and native plus ACP subagent dispatch still use the single global subagent lane.

## Solution Assessment

Unclear as originally proposed. The capability belongs in core scheduler/config code, but maintainers should choose a namespace-compatible config shape rather than adding a top-level agents.list[].maxConcurrent key by default.
