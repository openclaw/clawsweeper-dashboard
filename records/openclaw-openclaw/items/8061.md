## Summary

Keep open: current main and the latest release still expose only a global subagent concurrency cap, while the requested per-target-agent cap remains an unimplemented public config/runtime feature with confirmed user demand and maintainer-side product-shape questions.

## What This Changes

The issue requests a per-agent subagent concurrency limit so a target such as a local TTS agent can be serialized without globally limiting all subagents.

## Best Possible Solution

Add a validated per-target-agent subagent concurrency setting, likely under agents.list[].subagents, and enforce it for native and ACP sessions_spawn while preserving the existing global subagent lane cap as a process-wide safety limit.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature request. Source inspection gives high-confidence proof that current main lacks the requested per-target-agent config, validation, docs, and enforcement path.

## Solution Assessment

Unclear as a final design until maintainers choose the config shape and scheduler semantics. A per-agent subagents.maxConcurrent setting that composes with the global subagent lane cap looks like the narrowest maintainable direction if defaults preserve current behavior.
