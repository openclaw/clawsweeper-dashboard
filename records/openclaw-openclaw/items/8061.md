## Summary

Keep open. Current main still has only a global subagent lane cap, while per-agent config, strict schema validation, docs, agents_list output, and native/ACP spawn paths do not provide or enforce a per-target subagent maxConcurrent setting; the issue has a concrete TTS/local-model use case and another user confirmed the same problem.

## What This Changes

Add an optional per-agent subagent concurrency limit to agent list configuration so one target agent can be serialized without limiting all subagents globally.

## Best Possible Solution

Add a documented per-target subagent concurrency setting with agreed config shape, strict schema/type/help/docs updates, agents_list visibility if useful, and enforcement for both native and ACP sessions_spawn while preserving the existing global subagent lane safety cap.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this requests new config and scheduler behavior. Source inspection gives high-confidence proof that current main lacks the requested per-target subagent concurrency cap.

## Solution Assessment

Unclear as proposed. The capability is well scoped, but maintainers should choose the public config shape and how the per-target cap composes with the global subagent lane, active-child limits, and adjacent concurrency controls.
