## Summary

Keep open. Current main still exposes only a global subagent concurrency cap, while `agents.list[]` has no typed, strict-schema, documented, or enforced per-target subagent cap; the related open concurrency issues cover adjacent lane, parent, and lifetime-budget scopes rather than this TTS-style target-agent limit.

## What This Changes

Add an optional per-agent subagent concurrency cap so a target like a TTS agent can run only one instance while other subagents remain parallel.

## Best Possible Solution

Add a documented per-target subagent concurrency setting, likely under `agents.list[].subagents` if maintainers keep the namespace pattern, with schema/type/help/docs updates and native plus ACP enforcement alongside the global cap.

## Reproduction Assessment

Yes, for source-level capability proof: current main lacks any per-target maxConcurrent field in the strict schema, exported types, docs, and runtime resolver. No live TTS/local-model reproduction was attempted.

## Solution Assessment

Unclear as proposed. The capability is well scoped, but maintainers should choose the public config shape and define how a per-target cap composes with the global subagent lane and adjacent concurrency controls.
