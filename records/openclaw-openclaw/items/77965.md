## Summary

Keep open: the PR targets a real Feishu ambient-proxy failure, but the current diff is not mergeable as-is because it bypasses documented operator-managed proxy routing and still needs patched-branch real behavior proof plus docs/changelog cleanup.

## What This Changes

The PR changes Feishu client code/tests to ignore ambient proxy env for WebSocket unless `OPENCLAW_FEISHU_WS_USE_PROXY=1`, and also carries an unrelated generated base-config schema version change.

## Best Possible Solution

Bypass unmanaged ambient proxy env for Feishu WebSocket by default while still honoring `OPENCLAW_PROXY_ACTIVE=1` managed proxy routing, then document/changelog the Feishu-specific escape hatch with focused regression coverage.

## Reproduction Assessment

Yes, from source inspection: set HTTP(S) proxy env and `createFeishuWSClient()` currently creates and passes a proxy agent into the Lark `WSClient`. The linked issue and PR body add live failure/recovery logs, but I did not run a live Feishu gateway in this read-only review.

## Solution Assessment

No. Gating only on `OPENCLAW_FEISHU_WS_USE_PROXY=1` fixes unmanaged ambient env inheritance but breaks the documented managed-proxy path; the narrower maintainable fix should distinguish `OPENCLAW_PROXY_ACTIVE=1` from unrelated ambient proxy env.
