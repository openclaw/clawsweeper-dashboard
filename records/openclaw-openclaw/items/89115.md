## Label Justifications

- `P1`: The issue describes a real user-visible agent/channel workflow where a delivered reply can leave session state stuck and block later activity.
- `impact:message-loss`: A stuck pendingFinalDelivery flag can suppress or mis-handle later message delivery on the affected session.
- `impact:session-state`: The central failure is persisted pendingFinalDelivery state remaining set after successful final delivery.

## Summary

Current main still checks the dispatch abort signal after a successful final payload but before clearing pendingFinalDelivery, so the reported stuck-state race remains valid. The cited related issues cover adjacent heartbeat cleanup, orphan retry bounds, and stuck-recovery thresholds, but they do not own this user-message dispatch ordering window.

## What This Changes

The issue asks for the user-message dispatch success path to clear pendingFinalDelivery after successful final delivery even if a stuck-session abort arrives before post-delivery bookkeeping completes.

## Best Possible Solution

Make pendingFinalDelivery cleanup run once final delivery has succeeded, then surface any abort afterward, with a focused regression test for abort-after-send-before-clear.

## Reproduction Assessment

Yes for source-level reproduction: current main has a successful sendFinalPayload path followed by an abort check before pendingFinalDelivery cleanup, and stuck-session recovery can abort concurrently. I did not run a live gateway timing repro, so this is source-reproducible rather than runtime-reproduced.

## Solution Assessment

Yes, the suggested ordering/finally-style direction is the narrow maintainable fix: once final delivery has succeeded, cleanup should not be skipped by a later abort. A regression test should model abort after successful final delivery and assert the pending fields are cleared.
