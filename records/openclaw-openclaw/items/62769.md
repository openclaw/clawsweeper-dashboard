## Summary

Current main still rejects Telegram direct-message ACP configured bindings in the Telegram configured-binding adapter, and the docs still describe Telegram persistent ACP bindings as forum-topic-only. The related draft PR #68537 remains the active implementation candidate, so this issue should stay open until that PR or a narrow replacement lands.

## What This Changes

Support top-level Telegram ACP configured bindings for direct-message peers so dedicated Telegram DM bots can create or resume persistent ACP sessions instead of falling through to embedded runtime.

## Best Possible Solution

Land a rebased #68537 or narrow replacement that accepts positive Telegram direct-chat peer ids in the Telegram configured-binding adapter, preserves existing forum-topic behavior, updates Telegram/ACP docs, adds focused regression coverage, and includes the required changelog entry.

## Reproduction Assessment

Yes. Configure a top-level Telegram ACP binding with `match.peer.kind: "direct"` and a positive numeric `match.peer.id`; current main sends that id through the Telegram adapter, which returns null before the binding can match.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the Telegram plugin's configured-binding normalization and inbound matching path, with docs and tests, rather than in the generic ACP registry or core routing layer.
