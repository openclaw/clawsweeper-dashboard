## Summary

Keep open. Current main still rejects Telegram direct-message configured ACP bindings at the Telegram adapter layer, the public docs still describe Telegram persistent ACP bindings as forum-topic/group scoped, and #68537 is the open implementation candidate rather than a landed fix.

## What This Changes

Support top-level ACP configured bindings for Telegram direct-message peer IDs so DMs can create or resume persistent ACP harness sessions instead of falling through to the native runtime.

## Best Possible Solution

Land a rebased Telegram-channel fix, or a narrow replacement for #68537 if that branch is not practical to update, that accepts positive direct-chat peer ids in the Telegram configured-binding adapter, preserves current topic behavior, updates Telegram and ACP docs, and adds focused regression coverage plus a changelog entry.

## Reproduction Assessment

Yes. Configure a top-level Telegram ACP binding with `match.peer.kind: "direct"` and a positive numeric `match.peer.id`; current main passes that id into the Telegram provider, which cannot parse it as a topic and returns null before the binding can match.

## Solution Assessment

Yes, the best path is a narrow Telegram plugin fix rather than a core ACP registry change. The open PR direction is appropriate, but it needs the previously identified type/docs/changelog repair or replacement before this issue can close.
