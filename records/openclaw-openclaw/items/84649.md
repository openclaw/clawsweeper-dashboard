## Summary

Keep open: the performance direction is useful and the PR includes real local TUI cold-start measurements, but the diff drops current derived snapshot file-freshness behavior by caching with lookup-time registry state and replacing the manifest-edit regression expectation.

## What This Changes

The PR adds plugin metadata and registry memoization, threads snapshots/install records/discovery through provider and auto-enable paths, clears registry memo state on index writes/gateway close, and adds targeted plugin-system regression tests.

## Best Possible Solution

Keep the startup perf wins, but preserve derived snapshot watched-file freshness in the memo key/state and keep regression coverage for manifest edits before broader plugin startup validation.

## Reproduction Assessment

Yes for the review finding: current main has a focused test that edits a derived plugin manifest and expects cache invalidation, while the PR head stores the lookup-time registry state and removes that manifest-edit expectation. I did not rerun the full TUI cold-start profile locally.

## Solution Assessment

No as written. The overall approach of reducing repeated discovery is maintainable, but the memo must retain derived watched-file invalidation instead of making installed-index rewrite or process restart the only freshness signal.
