## Summary

Keep open: this protected maintainer PR has useful, well-supported startup-performance work, but source review found a workspace-inheritance regression in the new eager metadata-snapshot load that should be fixed before merge.

## What This Changes

The PR expands plugin metadata snapshot memoization to an 8-entry persisted/provided LRU and threads metadata snapshots/discovery through provider and auto-enable hot paths to reduce TUI/gateway cold-start plugin work.

## Best Possible Solution

Keep the performance refactor, but make snapshot, owner-resolution, and load-state paths share the same resolved workspace/env before landing.

## Reproduction Assessment

Yes for the review finding by source inspection: the new snapshot loads use params.workspaceDir while the existing load base still inherits the active registry workspace. I did not rerun the cold-start profiler in this read-only pass.

## Solution Assessment

No, not as-is: the snapshot threading direction is maintainable, but it should resolve and reuse the same inherited workspace for snapshot and provider load decisions before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.78

Full review comments:

- **[P2] Use the inherited workspace for snapshot resolution:** `src/plugins/providers.runtime.ts:348`
  - body: When callers omit `workspaceDir`, `resolvePluginProviderLoadBase` still inherits the active registry workspace, but the new metadata snapshot is loaded first with `params.workspaceDir` still undefined. Resolve the inherited workspace before loading the snapshot and use the same value for both paths.
