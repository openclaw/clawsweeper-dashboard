## Summary

Keep open: current main still lacks rate-limit headroom output, but this PR is not merge-ready because it is conflicting, lacks real-provider proof, drops the new flag on the routed CLI path, and adds a separate credentialed provider request outside the existing transport boundary.

## What This Changes

The PR adds an opt-in `--rate-limits` flag to `models status --probe`, parses provider rate-limit headers through a secondary request, and renders RPM/TPM headroom in table and JSON output.

## Best Possible Solution

Expose provider rate-limit headroom through the existing model probe/provider transport and auth contracts, wire both route-first and Commander CLI paths, and require redacted real-provider proof for table plus JSON output before merge.

## Reproduction Assessment

Yes for the PR defects: source inspection shows the route-first parser omits `rateLimits`, and the PR head makes a credentialed raw `fetch` outside the existing provider transport. The linked request itself is a new feature, not a broken current-main behavior.

## Solution Assessment

No. The feature direction is useful, but the current implementation should be reworked to reuse the existing provider transport/auth path and update route-first parsing rather than adding a parallel request path.

- **[P1] Forward the flag through route-first parsing:** `src/cli/models-cli.ts:91`
  - body: Normal startup can handle `models status` through `parseModelsStatusRouteArgs` before Commander.   - body: `probeRateLimits` posts API credentials with raw `fetch` to locally constructed URLs. That bypasses the guarded provider transport used by the successful probe, including SSRF/private-network policy, proxy/TLS overrides, local-service leases, configured headers/auth, timeouts, and diagnostics.
  - confidence: 0.9
- **[P2] Resolve follow-up credentials with the probe context:** `src/commands/models/list.probe.ts:558-563`
  - body: The follow-up resolver calls `resolveEnvApiKey(target.provider)` without the config/workspace/alias/evidence context used by the main probe, then falls back to raw custom-provider key checks. Rate-limit probing can therefore miss or use different credentials than the probe that just succeeded.
  - confidence: 0.86
- **[P2] Match custom provider config by normalized keys:** `src/commands/models/list.probe.ts:570-575`
  - body: `resolveCustomBaseUrl` only checks exact and already-normalized object keys. A config stored under a key like `OpenAI` can make the main probe use the custom endpoint while the rate-limit follow-up falls back to a default host and reports no headroom.
  - confidence: 0.82
