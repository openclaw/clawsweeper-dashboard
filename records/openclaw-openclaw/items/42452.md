## Summary

Keep open. Current main still does not expose provider rate-limit headroom for `models status --probe`, and the PR remains not merge-ready because it misses the route-first CLI path and adds a direct credentialed provider request outside the established transport and credential controls.

## What This Changes

The PR adds opt-in `--rate-limits` support to `models status --probe`, parses provider rate-limit headers, and surfaces RPM/TPM data in table and JSON probe output.

## Best Possible Solution

Rebase and repair the PR so both route-first and Commander paths carry the option, the rate-limit probe reuses provider transport and usable secret resolution, and the new public CLI/JSON contract is documented and changelogged.

## Reproduction Assessment

Yes. Source inspection is enough here: current main lacks the requested `--rate-limits` surface, and the PR diff does not update the route-first parser that normally invokes `modelsStatusCommand`.

## Solution Assessment

No. The user-facing idea is plausible, but the current implementation should be routed through existing CLI parsing, provider request policy, and usable credential resolution instead of a separately assembled raw fetch path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Forward --rate-limits through route-first parsing:** `src/cli/models-cli.ts:91`
  - body: `models status` is handled by the route-first parser before Commander on normal startup. Because `parseModelsStatusRouteArgs` does not return `rateLimits`, the advertised flag reaches `modelsStatusCommand` without the new option unless route-first routing is bypassed.
  - confidence: 0.9
- **[P2] Reuse provider request transport for rate-limit probes:** `src/commands/models/list.probe.ts:617-621`
  - body: The new follow-up call sends API credentials through a raw `fetch` with locally assembled headers. Normal provider traffic can depend on configured headers, proxy/TLS, private-network policy, guarded fetch behavior, and provider-owned transport hooks, so this can fail or route secrets differently even after the main probe succeeds.
  - confidence: 0.86
- **[P2] Resolve models.json keys with usable secret semantics:** `src/commands/models/list.probe.ts:562-563`
  - body: The models.json fallback uses `getCustomProviderApiKey`, which can return env names or managed SecretRef marker strings instead of a usable secret. Reuse the existing usable custom-provider key resolver so this probe does not omit credentials or send a marker when the main probe can authenticate.
  - confidence: 0.8
- **[P2] Match provider config by normalized entry keys:** `src/commands/models/list.probe.ts:570-575`
  - body: `resolveCustomBaseUrl` only checks exact and already-normalized provider keys. Current model resolution also scans entries whose keys normalize to the provider id, so configs keyed like `OpenAI` can make the main probe succeed while the rate-limit follow-up probes the default host.
  - confidence: 0.82
- **[P3] Add the required changelog entry:** `src/cli/models-cli.ts:91`
  - body: This adds a user-facing CLI flag and JSON probe field, but the PR does not update `CHANGELOG.md`. OpenClaw policy requires a changelog entry for user-facing `feat` changes before merge.
  - confidence: 0.78
