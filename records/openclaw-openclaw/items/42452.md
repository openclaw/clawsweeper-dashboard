## Summary

Keep open: current main still lacks provider rate-limit headroom in `models status --probe`, but this PR has blocking routed-CLI, auth/provider-transport, and proof issues before it can merge.

## What This Changes

The PR adds an opt-in `--rate-limits` flag to `models status --probe`, a `rateLimit` probe result field, RPM/TPM table formatting, and parsing tests.

## Best Possible Solution

Expose rate-limit headroom through the existing model probe/provider transport and auth contracts, wire the flag through both routed and Commander CLI paths, and prove table plus JSON output with redacted real-provider output.

## Reproduction Assessment

Yes for the PR defects: source inspection shows the routed parser drops the new flag, the follow-up request uses raw fetch, and custom auth/config lookup can diverge from the main probe path. The underlying request is a new feature, not a broken current-main behavior.

## Solution Assessment

No. The feature direction is useful, but the implementation should reuse OpenClaw's routed CLI, credential, and provider-transport contracts instead of adding a parallel provider request path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Forward --rate-limits through the routed parser:** `src/cli/models-cli.ts:91`
  - body: Normal startup can route `models status` through `parseModelsStatusRouteArgs` before Commander. This patch adds the flag only to the Commander path, so routed `openclaw models status --probe --rate-limits` reaches `modelsStatusCommand` without `rateLimits` and produces no RPM/TPM output.
  - confidence: 0.9
- **[P2] Use the provider transport for header probes:** `src/commands/models/list.probe.ts:617-622`
  - body: `probeRateLimits` posts credentials with raw `fetch` to locally constructed URLs. That bypasses configured request headers/auth overrides, proxy/TLS/private-network policy, guarded fetch behavior, local service leases, transport timeouts, and diagnostics used by the successful main probe.
  - confidence: 0.88
- **[P2] Resolve custom provider configs by normalized key:** `src/commands/models/list.probe.ts:570-575`
  - body: `resolveCustomBaseUrl` checks only exact and already-normalized keys. A config stored under a key like `OpenAI` can make the main probe use the custom endpoint while the rate-limit follow-up falls back to the default host and reports `-`.
  - confidence: 0.84
- **[P2] Resolve probe credentials through the usable auth path:** `src/commands/models/list.probe.ts:558-564`
  - body: The follow-up resolver calls `resolveEnvApiKey` without the same config/workspace context and falls back to raw `getCustomProviderApiKey`, which can miss resolved SecretRef/env-marker behavior that the main probe already supports.
  - confidence: 0.84
