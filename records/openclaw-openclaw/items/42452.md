## Summary

Keep open: current main still lacks rate-limit headroom in model probe output, and the linked feature issue was closed in favor of this PR, but the branch should not merge until the routed CLI flag, provider transport, credential handling, and real behavior proof gaps are addressed.

## What This Changes

The PR adds an opt-in `--rate-limits` flag to `models status --probe`, parses provider `x-ratelimit-*` headers, and shows RPM/TPM headroom in table and JSON probe output.

## Best Possible Solution

Expose rate-limit headroom through the existing model probe/provider transport contract, carry the new flag through both routed and Commander CLI paths, and prove table plus JSON output with focused tests and redacted real provider output.

## Reproduction Assessment

Yes for the PR defects: source inspection shows the route-first parser drops the new flag, the follow-up request uses a separate raw provider fetch, and custom auth/config lookup can diverge from the main probe path. The underlying user request is a new feature, not a broken behavior on current main.

## Solution Assessment

No. The feature direction is useful, but this patch should be reworked to reuse OpenClaw's routed CLI, credential, and provider-transport contracts instead of adding a parallel request path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Forward `--rate-limits` through the routed parser:** `src/cli/models-cli.ts:91`
  - body: Normal startup can route `models status` through `parseModelsStatusRouteArgs` before Commander. This patch adds the flag only to the Commander command, so routed `openclaw models status --probe --rate-limits` calls `modelsStatusCommand` without `rateLimits` and the new output stays disabled.
  - confidence: 0.9
- **[P2] Use the provider transport for header probes:** `src/commands/models/list.probe.ts:617-622`
  - body: `probeRateLimits` sends credentials through a raw `fetch` with locally built URLs and headers. That bypasses configured request headers/auth overrides, proxy/TLS/private-network policy, guarded fetch behavior, local service leases, transport timeouts, and model transport diagnostics used by the successful main probe.
  - confidence: 0.88
- **[P2] Resolve custom provider configs by normalized key:** `src/commands/models/list.probe.ts:570-575`
  - body: `resolveCustomBaseUrl` checks only exact and already-normalized keys. A config stored under a key like `OpenAI` can make the main probe use the custom endpoint while the rate-limit follow-up falls back to the default host and reports `-`.
  - confidence: 0.83
- **[P2] Resolve probe credentials through the usable auth path:** `src/commands/models/list.probe.ts:558-564`
  - body: The new follow-up resolver uses `resolveEnvApiKey` without the same config/workspace context and falls back to raw `getCustomProviderApiKey`. Targets admitted by `hasUsableCustomProviderApiKey`, such as env SecretRefs or marker-backed local configs, can succeed in the main probe but return no rate-limit data here.
  - confidence: 0.84
