## Summary

Keep open: current main does not implement the requested rate-limit probe surface, and the PR is still useful source work, but it is not mergeable as-is because the new flag misses route-first parsing and the added credentialed probe bypasses existing auth/request-policy seams.

## What This Changes

The PR adds an opt-in `models status --probe --rate-limits` CLI flag, rate-limit header parsing/types, an extra provider HTTP probe, RPM/TPM table columns, JSON probe data, parser/formatter tests, and a `.gitignore` log pattern.

## Best Possible Solution

Rebase and revise the PR around route-first argument parsing plus existing provider credential/request transport seams, then add tests, docs, and changelog coverage for the new CLI/JSON contract.

## Reproduction Assessment

Yes. Source inspection of the PR diff and current main shows normal route-first startup does not receive `rateLimits`, and the new models.json key path references helpers without imports; no live provider call is needed to reproduce those integration failures.

## Solution Assessment

No. The user-facing direction is plausible, but the current diff is not the narrowest maintainable solution until it reuses route-first parsing and OpenClaw's existing credential/request-policy transport controls.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Forward --rate-limits through route-first parsing:** `src/cli/models-cli.ts:91`
  - body: `models status` is routed by `tryRouteCli` before Commander on normal startup, but this PR only adds the flag to the Commander command. Since `parseModelsStatusRouteArgs` does not parse or return `rateLimits`, `openclaw models status --probe --rate-limits` can reach `modelsStatusCommand` without the new option unless route-first is disabled.
  - confidence: 0.9
- **[P1] Use the resolved custom-provider key seam:** `src/commands/models/list.probe.ts:562-563`
  - body: The new models.json fallback calls `getCustomProviderApiKey` and `isNonSecretApiKeyMarker` without importing them, so the patch will not typecheck. Importing those helpers directly would still use marker-level semantics instead of resolved env/SecretRef values, so this should reuse the existing usable custom-provider key resolver that the main probe path is aligned with.
  - confidence: 0.93
- **[P2] Reuse provider request transport for rate-limit probes:** `src/commands/models/list.probe.ts:617`
  - body: This sends API credentials with a raw `fetch` assembled locally. Existing provider traffic can rely on configured request headers, proxy/TLS, private-network policy, attribution headers, and provider-owned transport hooks, so the follow-up probe can leak different behavior or fail when the main probe succeeds.
  - confidence: 0.86
- **[P2] Match provider config by normalized entry keys:** `src/commands/models/list.probe.ts:572`
  - body: `resolveCustomBaseUrl()` checks only exact and already-normalized provider keys. Current model resolution also scans entries whose keys normalize to the provider id, so configs keyed like `OpenAI` can make the main probe succeed while the rate-limit follow-up falls back to the default host.
  - confidence: 0.82
- **[P3] Add the required changelog entry:** `src/cli/models-cli.ts:91`
  - body: This adds a user-facing `models status` flag and JSON field, but the PR does not update `CHANGELOG.md`. OpenClaw policy requires a changelog entry for user-facing `feat` changes before merge.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The diff introduces a direct credentialed provider fetch outside the existing provider request controls.

Concerns:
