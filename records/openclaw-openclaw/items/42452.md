## Summary

Keep open: current main still does not expose provider rate-limit headroom in `models status --probe`, and this PR is the active implementation candidate, but the branch is not mergeable yet because it has build/correctness issues, a provider-transport security concern, and no real provider proof.

## What This Changes

The PR adds an opt-in `--rate-limits` flag for `models status --probe` that surfaces provider rate-limit headroom in table and JSON probe output.

## Best Possible Solution

Expose rate-limit headroom through the existing model probe CLI/JSON contract while carrying the flag through route-first parsing and using the provider credential/request policy path, or capturing headers from the existing probe, with focused tests and redacted real provider proof.

## Reproduction Assessment

Yes for the PR defects: source inspection shows the route-first parser drops the new flag, the added key fallback references missing helpers, and the secondary provider request bypasses the established transport path. The underlying request is a new feature rather than a broken existing behavior on main.

## Solution Assessment

No. The direction is useful, but this patch should be reworked to stay on OpenClaw's route, credential, and provider-transport contracts before it is considered the maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Use the usable custom-key resolver:** `src/commands/models/list.probe.ts:558-564`
  - body: `resolveApiKeyForTarget` calls `getCustomProviderApiKey` and `isNonSecretApiKeyMarker` without importing them, and `isNonSecretApiKeyMarker` is not exported from `model-auth.ts`.   - body: Normal startup routes `models status` through the fast parser before Commander, but this patch only adds `--rate-limits` to the Commander command. `openclaw models status --probe --rate-limits` can therefore run with the flag dropped and no RPM/TPM output.
  - confidence: 0.9
- **[P2] Reuse provider transport for the header probe:** `src/commands/models/list.probe.ts:617-622`
  - body: `probeRateLimits` sends provider credentials through raw `fetch` with locally built URLs and headers. Existing model requests can rely on configured headers, auth overrides, proxy/TLS/private-network policy, guarded fetch behavior, local service leases, and transport timeouts, so this follow-up can route differently after the main probe succeeds.
  - confidence: 0.88
- **[P2] Match custom provider configs by normalized keys:** `src/commands/models/list.probe.ts:570-575`
  - body: `resolveCustomBaseUrl()` checks only exact and already-normalized provider keys. The main runtime also scans keys by `normalizeProviderId`, so configs keyed like `OpenAI` can make the main probe use a custom endpoint while the rate-limit follow-up probes the default host.
  - confidence: 0.83
