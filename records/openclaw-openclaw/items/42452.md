## Summary

Keep open: current main still lacks the requested rate-limit probe surface, and the PR remains useful but not mergeable because the latest head still has source-visible integration, typecheck, changelog, and credentialed-transport blockers.

## What This Changes

The PR adds an opt-in `models status --probe --rate-limits` flag, rate-limit header parsing/types, a secondary provider HTTP probe, RPM/TPM table columns, JSON probe data, and parser/formatter tests.

## Best Possible Solution

Rework the feature around existing provider credential/request transport seams or a last-seen-header mechanism, then route the flag through route-first parsing and add contract tests, docs, and changelog coverage for the public CLI/JSON surface.

## Reproduction Assessment

Yes. Source inspection of current main plus the PR patch is enough: normal startup routes `models status` before Commander, the PR patch calls unimported helpers, and its secondary request uses raw `fetch` instead of the provider request stack.

## Solution Assessment

No. The user-facing direction is plausible, but the current patch is not the best implementation until it reuses route-first parsing and OpenClaw's existing provider credential/request-policy transport controls.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Forward --rate-limits through route-first parsing:** `src/cli/models-cli.ts:91`
  - body: Normal startup calls `tryRouteCli` before Commander registers this option, but `parseModelsStatusRouteArgs` does not parse or return `rateLimits`. As a result, `openclaw models status --probe --rate-limits` can be routed without the new option and produce the old output.
  - confidence: 0.9
- **[P1] Use the resolved custom-provider key seam:** `src/commands/models/list.probe.ts:562-563`
  - body: `resolveApiKeyForTarget` calls `getCustomProviderApiKey` and `isNonSecretApiKeyMarker` without importing them, so the patch will not typecheck. Importing them directly would still miss the existing resolved SecretRef/env semantics, so this should reuse the usable custom-provider key resolver.
  - confidence: 0.93
- **[P2] Reuse provider request transport for rate-limit probes:** `src/commands/models/list.probe.ts:617`
  - body: The follow-up probe sends API credentials through a raw `fetch` assembled locally. Existing model requests can depend on configured headers, proxy/TLS, private-network policy, attribution headers, and provider-owned transport hooks, so this can fail or route differently when the main probe succeeds.
  - confidence: 0.87
- **[P2] Match provider config by normalized entry keys:** `src/commands/models/list.probe.ts:572`
  - body: `resolveCustomBaseUrl` only checks exact and already-normalized provider keys. Current model resolution also scans entries whose keys normalize to the provider id, so configs keyed like `OpenAI` can make the main probe use a custom endpoint while the rate-limit probe falls back to the default host.
  - confidence: 0.82
- **[P3] Add the required changelog entry:** `src/cli/models-cli.ts:91`
  - body: This adds a user-facing `models status` flag and JSON field but does not update `CHANGELOG.md`. Repo policy requires a changelog entry for user-facing feature changes before merge.
  - confidence: 0.84
