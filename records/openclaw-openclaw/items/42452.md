## Summary

Keep open: the feature is not on current main and this PR is a plausible implementation path, but the current head should not merge until route-first CLI parsing, provider transport/auth resolution, and real-provider proof are addressed.

## What This Changes

Adds a `--rate-limits` option to `models status --probe` that parses provider `x-ratelimit-*` headers into probe results, JSON output, and RPM/TPM table columns using a secondary provider request.

## Best Possible Solution

Expose rate-limit headroom through the existing model probe CLI/JSON contract while reusing provider credential and request policy, or capture headers from the existing probe, with route-first parsing, normalized config lookup, focused tests/docs, and redacted live proof.

## Reproduction Assessment

Yes for the PR defects: source inspection shows the route-first parser has no `rateLimits` field, and the PR head still uses a raw secondary provider `fetch` plus incomplete config-aware credential lookup. The underlying request is a new feature rather than a broken existing behavior on main.

## Solution Assessment

No. The user-facing direction is reasonable, but the patch should reuse OpenClaw's provider credential/request policy path or capture headers from the existing probe before adding the CLI/JSON surface.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Forward the flag through route-first parsing:** `src/cli/models-cli.ts:91`
  - body: Normal startup routes `models status` before Commander, but this patch only adds `--rate-limits` to the Commander command. The route-first parser still returns no `rateLimits` option, so `openclaw models status --probe --rate-limits` can run without the extra probes or RPM/TPM columns.
  - confidence: 0.9
- **[P2] Reuse provider transport for the header probe:** `src/commands/models/list.probe.ts:617-622`
  - body: `probeRateLimits` sends provider credentials through a raw `fetch` using locally built URLs and headers. Existing model requests can depend on configured headers, auth overrides, proxy/TLS/private-network policy, guarded fetch behavior, local service leases, and transport timeouts, so this request can fail or route differently after the main probe succeeds.
  - confidence: 0.88
- **[P2] Resolve custom provider keys through the usable key path:** `src/commands/models/list.probe.ts:558-564`
  - body: This fallback asks only plain env and then uses `getCustomProviderApiKey` directly, which drops SecretRefs, env-marker custom providers, and local marker-backed providers that the main probe can use. Those targets would probe successfully, then show no rate-limit data because the follow-up cannot resolve the same usable key.
  - confidence: 0.86
- **[P2] Match custom provider configs by normalized keys:** `src/commands/models/list.probe.ts:570-575`
  - body: `resolveCustomBaseUrl()` checks only exact and already-normalized provider keys. Current provider config resolution also scans entries whose keys normalize to the provider id, so configs keyed like `OpenAI` can make the main probe use a custom endpoint while this follow-up probes the default host.
  - confidence: 0.82
