## Summary

Keep open. Current main still lacks this feature, and this PR remains the canonical implementation thread for the closed linked feature request, but the patch is not merge-ready because route-first CLI parsing, provider credential/transport policy, normalized config lookup, and real behavior proof are still unresolved.

## What This Changes

The PR adds an opt-in `models status --probe --rate-limits` CLI/JSON surface, rate-limit header parsing/types, RPM/TPM table output, a secondary provider HTTP probe, and parser/formatter tests.

## Best Possible Solution

Expose rate-limit headroom through an approved CLI/JSON contract that reuses existing provider credential and request-transport seams, or a last-seen-header mechanism, with route-first parsing, normalized provider lookup, focused tests, docs/changelog coverage, and redacted real-run proof.

## Reproduction Assessment

Yes. Source inspection is enough: the normal route-first path has no `rateLimits` field, and the PR head shows a raw secondary provider fetch plus custom config lookup that diverges from current provider request behavior.

## Solution Assessment

No. The user-facing direction is plausible, but this patch is not the best implementation until it reuses OpenClaw's provider credential/request policy path or captures headers from the existing probe and wires the flag through route-first parsing.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Forward --rate-limits through route-first parsing:** `src/cli/models-cli.ts:91`
  - body: Normal startup runs `tryRouteCli` before Commander, but `parseModelsStatusRouteArgs` does not parse or return `rateLimits`. The new Commander-only option means `openclaw models status --probe --rate-limits` can route with `opts.rateLimits` undefined, so no RPM/TPM columns or rate-limit probe are produced.
  - confidence: 0.9
- **[P2] Use the resolved custom-provider key resolver:** `src/commands/models/list.probe.ts:562-564`
  - body: `resolveApiKeyForTarget` falls back to `getCustomProviderApiKey` and drops non-secret markers, so env SecretRef or known env-marker custom providers that the main probe can use resolve to null for the rate-limit probe. Reuse the usable custom-provider key resolver or the same auth resolution as the main probe.
  - confidence: 0.86
- **[P2] Reuse provider request transport for rate-limit probes:** `src/commands/models/list.probe.ts:617`
  - body: The follow-up probe sends API credentials through raw `fetch` with locally built headers and URLs. Existing model requests can depend on configured headers, auth overrides, proxy/TLS settings, private-network policy, attribution, and timeouts, so this can fail or route differently after the main probe succeeds.
  - confidence: 0.88
- **[P2] Match custom provider config by normalized keys:** `src/commands/models/list.probe.ts:572-574`
  - body: `resolveCustomBaseUrl` only checks exact and already-normalized provider keys. Current model resolution also scans entries whose keys normalize to the provider id, so configs keyed like `OpenAI` can make the main probe use a custom endpoint while the rate-limit probe falls back to the default host.
  - confidence: 0.82
