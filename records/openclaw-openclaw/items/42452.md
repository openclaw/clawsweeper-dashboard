## Summary

Keep open. Current main still does not expose probe rate-limit headroom, but this contributor feature PR is not merge-ready because it is dirty against current main, misses the route-first CLI path, and adds a credentialed provider HTTP call outside the existing request-control seams.

## What This Changes

This PR adds an opt-in `--rate-limits` flag for `openclaw models status --probe`, rate-limit header parsing/types, RPM/TPM table columns, JSON probe data, and parser/formatter tests.

## Best Possible Solution

Rebase and repair the PR around the existing route-first parser and provider request/credential transport seams, then document and changelog the new opt-in CLI/JSON contract.

## Reproduction Assessment

Yes. Source inspection shows current main lacks the requested `--rate-limits` probe surface, and the PR head/source plus current route-first parser show high-confidence integration failures without needing live provider calls.

## Solution Assessment

No. The user-facing shape is plausible, and prior review feedback on timeout cleanup and version-suffix path doubling appears addressed, but the current implementation is not the best maintainable path until it reuses route-first parsing and provider request/credential controls.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Forward --rate-limits through route-first parsing:** `src/cli/models-cli.ts:91`
  - body: `models status` is handled by `tryRouteCli` before Commander on normal CLI startup, but the PR only adds the flag to the Commander command. Since `parseModelsStatusRouteArgs` does not return `rateLimits`, `openclaw models status --probe --rate-limits` still reaches `modelsStatusCommand` without the new option unless route-first is disabled.
  - confidence: 0.9
- **[P2] Reuse provider request transport for the rate-limit probe:** `src/commands/models/list.probe.ts:617-621`
  - body: The new probe sends API credentials with a raw `fetch` and locally assembled headers. Existing provider traffic can rely on configured request headers, proxy/TLS, private-network policy, guarded fetch behavior, and provider-owned transport hooks, so this follow-up can route secrets differently or fail when the main probe succeeds.
  - confidence: 0.86
- **[P2] Resolve models.json keys with usable secret semantics:** `src/commands/models/list.probe.ts:562-563`
  - body: The models.json fallback calls `getCustomProviderApiKey`, which can return marker strings such as env names or managed SecretRef placeholders rather than a resolved secret. Reuse the existing usable custom-provider key resolver so the rate-limit request does not omit credentials or send a marker when the main probe can authenticate.
  - confidence: 0.8
- **[P2] Match provider config by normalized entry keys:** `src/commands/models/list.probe.ts:570-575`
  - body: `resolveCustomBaseUrl` checks only exact and already-normalized provider keys. Current main also scans entries whose keys normalize to the provider id, so configs keyed like `OpenAI` can make the main probe succeed while the rate-limit follow-up falls back to a default host.
  - confidence: 0.82
