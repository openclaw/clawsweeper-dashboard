## Summary

Keep open: current main still lacks this rate-limit headroom surface, but this PR is not mergeable as-is because it misses current routed CLI plumbing, diverges from existing credential/provider transport contracts, and lacks real provider proof.

## What This Changes

The PR adds an opt-in `--rate-limits` flag to `models status --probe`, a `rateLimit` probe result field, RPM/TPM table columns, direct provider header probing, and parser/formatter tests.

## Best Possible Solution

Expose rate-limit headroom through the existing model probe CLI/JSON contract while carrying the flag through route-first parsing and using the policy-aware provider transport, or capturing headers from the existing probe, with focused tests and redacted provider proof.

## Reproduction Assessment

Yes for the PR defects: source inspection of current main and the PR head shows the route-first parser lacks `rateLimits`, the new header probe uses raw `fetch`, and custom key/base URL resolution diverges from current contracts. The underlying request is a new feature rather than a broken existing behavior on main.

## Solution Assessment

No. The feature direction is useful, but the maintainable solution should reuse OpenClaw's routed CLI, credential, and provider-transport contracts instead of adding a parallel provider request path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Forward --rate-limits through the routed status parser:** `src/cli/models-cli.ts:91`
  - body: Current main routes `models status` through `parseModelsStatusRouteArgs` before Commander, but this patch only adds the new flag to the Commander command. In that normal path, `openclaw models status --probe --rate-limits` can drop the flag and skip the new RPM/TPM output and rate-limit probe entirely.
  - confidence: 0.9
- **[P2] Reuse provider transport for the header probe:** `src/commands/models/list.probe.ts:617-622`
  - body: `probeRateLimits` sends provider credentials through a raw `fetch` to locally built URLs. Existing model requests can rely on configured headers, auth overrides, proxy/TLS/private-network policy, guarded fetch behavior, local service leases, transport timeouts, and diagnostics, so the follow-up request can diverge immediately after a successful main probe.
  - confidence: 0.88
- **[P2] Resolve models.json keys with the usable-key helper:** `src/commands/models/list.probe.ts:562-564`
  - body: The new resolver only returns literal custom keys and ignores marker-backed or env SecretRef models.json keys that current status/probe logic treats as usable through `resolveUsableCustomProviderApiKey`. Those profiles can probe successfully but still show no rate-limit data.
  - confidence: 0.86
- **[P2] Match custom provider configs by normalized key:** `src/commands/models/list.probe.ts:570-575`
  - body: `resolveCustomBaseUrl()` only checks the exact provider key and the already-normalized key. The main runtime also scans configured keys by `normalizeProviderId`, so a config keyed as `OpenAI` can make the main probe hit a custom endpoint while the rate-limit follow-up falls back to the default host.
  - confidence: 0.83
