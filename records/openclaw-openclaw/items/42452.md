## Summary

Keep this PR open: current main does not implement the requested rate-limit probe surface, and the PR is still not merge-ready because the advertised flag is not wired through route-first startup, credential resolution is inconsistent with current auth seams, and the secondary provider request bypasses guarded provider transport.

## What This Changes

The PR adds an opt-in `models status --probe --rate-limits` CLI/JSON rate-limit headroom surface, parser/formatter tests, direct provider HTTP probes, and an `openclaw-*.log` ignore rule.

## Best Possible Solution

Rework the feature around existing provider credential/request transport seams or a last-seen-header mechanism, then add route-first parser coverage plus docs and changelog for the public CLI/JSON contract.

## Reproduction Assessment

Yes. Source inspection of current main plus the PR patch is enough: normal startup routes `models status` before Commander, the PR does not update the route-first parser, and the secondary probe path calls unimported helpers while using raw `fetch`.

## Solution Assessment

No. The user-facing direction is plausible, but the current patch is not the best solution until it reuses OpenClaw's existing credential and provider request-policy transport controls and wires the new flag through both CLI parsing paths.

  - confidence: 0.92

  - body: The follow-up rate-limit probe sends API credentials through a locally assembled raw `fetch`. Existing model traffic can depend on guarded fetch policy, configured proxy/TLS/private-network handling, attribution headers, timeout dispatchers, and provider-owned hooks, so this can fail or route differently after the main probe succeeds.
  - confidence: 0.87

  - body: `resolveCustomBaseUrl` checks only exact and already-normalized keys. Current model resolution also scans entries whose keys normalize to the provider id, so configs keyed like `OpenAI` can make the main probe use a custom endpoint while the rate-limit probe falls back to the default host.
  - confidence: 0.82
