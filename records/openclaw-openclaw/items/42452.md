## Summary

Keep open: current main still lacks provider rate-limit headroom output, and the linked feature issue was intentionally closed in favor of this PR, but the patch is not ready because it drops the new flag on the route-first path, bypasses existing provider transport and credential semantics, and lacks external real behavior proof.

## What This Changes

The PR adds an opt-in `--rate-limits` flag to `models status --probe`, returning provider rate-limit header data in CLI and JSON probe output.

## Best Possible Solution

Expose rate-limit headroom through the existing model probe CLI/JSON contract while reusing provider credential and request transport policy, or by capturing headers from the existing probe, with route-first parsing, normalized config lookup, focused tests/docs, and redacted live proof.

## Reproduction Assessment

Yes for the PR defects: source inspection shows the route-first parser has no `rateLimits` field, and the PR head still uses a raw secondary provider `fetch` plus incomplete custom-provider auth/config resolution. The original request is a new feature rather than a broken existing behavior.

## Solution Assessment

No. The user-facing direction is plausible, but the patch should reuse OpenClaw's provider credential/request policy path or capture headers from the existing probe before adding the CLI/JSON surface.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Forward the flag through route-first parsing:** `src/cli/models-cli.ts:91`
  - body: Normal startup routes `models status` before Commander, but this patch only adds `--rate-limits` to the Commander command. The route-first parser still returns no `rateLimits` option, so `openclaw models status --probe --rate-limits` can run without the extra probes or RPM/TPM columns.
  - confidence: 0.9
- **[P2] Reuse provider transport for the header probe:** `src/commands/models/list.probe.ts:617-622`
  - body: `probeRateLimits` sends provider credentials through raw `fetch` using locally built URLs and headers. Existing model requests can depend on configured headers, auth overrides, proxy/TLS/private-network policy, guarded fetch behavior, local service leases, and transport timeouts, so this request can fail or route differently after the main probe succeeds.
  - confidence: 0.88
- **[P2] Resolve custom provider keys through the usable key path:** `src/commands/models/list.probe.ts:562-564`
  - body: This fallback uses `getCustomProviderApiKey` directly and drops marker-backed credentials, so env SecretRefs, known env-marker custom providers, and local marker-backed providers that the main probe can use resolve to null for the rate-limit probe.
  - confidence: 0.86
- **[P2] Match custom provider configs by normalized keys:** `src/commands/models/list.probe.ts:570-575`
  - body: `resolveCustomBaseUrl()` checks only exact and already-normalized provider keys. Current provider config resolution also scans entries whose keys normalize to the provider id, so configs keyed like `OpenAI` can make the main probe use a custom endpoint while this follow-up probes the default host.
  - confidence: 0.82
