## Summary

Keep open. The feature is still absent from current main and the PR is a plausible implementation for the linked rate-limit headroom request, but the current PR head is not merge-ready: the diff has a compile blocker, the route-first CLI path drops the new flag, and the credentialed follow-up request bypasses existing provider request controls.

## What This Changes

This PR adds an opt-in `--rate-limits` models status probe flag, rate-limit header parsing/types, a secondary provider HTTP probe, RPM/TPM table columns, JSON result data, parser/formatter tests, and one `.gitignore` log pattern.

## Best Possible Solution

Repair the PR rather than close it: keep the opt-in user shape, but thread the option through route-first parsing/tests, reuse the existing credential/provider request transport seams for base URL, headers, proxy, TLS, and normalized provider lookup, then document and changelog the new CLI/JSON contract.

## Reproduction Assessment

Yes. Source and PR-diff inspection gives a high-confidence reproduction path: current main lacks the feature, and the PR head shows unresolved helper references plus a routed CLI parser that cannot pass `rateLimits` to `modelsStatusCommand`.

## Solution Assessment

No. The proposed opt-in surface is reasonable, but this patch is not the best maintainable implementation until it compiles and uses the same route-first CLI, credential resolution, and provider request controls as the existing probe path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Import the helpers used by rate-limit probing:** `src/commands/models/list.probe.ts:562-563`
  - body: `resolveApiKeyForTarget` calls `getCustomProviderApiKey` and `isNonSecretApiKeyMarker`, but the diff does not import either helper into this module. `getCustomProviderApiKey` is exported from `model-auth.ts` and the marker helper is exported from `model-auth-markers.ts`, so the PR head will fail to typecheck before the feature can run.
  - confidence: 0.95
- **[P1] Thread rate-limits through routed status args:** `src/cli/models-cli.ts:91`
  - body: `tryRouteCli` handles `models status` before Commander, and `parseModelsStatusRouteArgs` does not return a `rateLimits` option. Adding the Commander flag here means normal `openclaw models status --probe --rate-limits` execution reaches `modelsStatusCommand` without the new option, so no rate-limit probe or RPM/TPM columns run unless route-first is disabled.
  - confidence: 0.9
- **[P2] Route the extra fetch through provider request controls:** `src/commands/models/list.probe.ts:617-620`
  - body: The new rate-limit call sends credentials with a raw `fetch` and locally built headers. Existing provider traffic can depend on `models.providers.*.headers` and `models.providers.*.request` proxy/TLS controls, so this follow-up can fail for working providers or leave the configured network route; build it through the same provider request/transport seam as the main probe.
  - confidence: 0.82
- **[P2] Match configured providers by normalized key:** `src/commands/models/list.probe.ts:572`
  - body: `resolveCustomBaseUrl` checks only exact and already-normalized object keys. Current model resolution also scans entries with `findNormalizedProviderValue`, so a config keyed as `OpenAI` can make the main probe succeed while this follow-up falls back to the default host and reports missing rate-limit data.
  - confidence: 0.84
- **[P2] Resolve models.json keys with the existing credential path:** `src/commands/models/list.probe.ts:562-563`
  - body: After the imports are fixed, this fallback still reads `models.providers.*.apiKey` through `getCustomProviderApiKey`, which can return persisted env SecretRef ids or non-secret markers rather than the resolved secret value. Reuse the existing usable custom-provider key resolver semantics so a main probe that succeeds from models.json credentials does not send a marker or omit the key in the rate-limit request.
  - confidence: 0.76

## Security Review
