## Summary

Keep this PR open. Current main still does not expose provider rate-limit headroom from `openclaw models status --probe`, and the PR is a plausible implementation, but the patch is not merge-ready because the route-first CLI path would drop the new flag, custom provider base URL lookup diverges from current normalized provider matching, and the extra credentialed fetch bypasses existing provider request controls.

## What This Changes

This PR adds an opt-in `--rate-limits` models status probe path with rate-limit header parsing/types, direct provider fetches, RPM/TPM table columns, JSON result data, focused parser/formatter tests, and one `.gitignore` entry.

## Best Possible Solution

Keep the implementation direction but rebase it onto current main, thread the option through both Commander and route-first CLI parsing, reuse existing provider configuration/request seams for endpoint, header, proxy, TLS, and normalized provider lookup behavior, then document and test the table and JSON contract for the opt-in probe.

## Reproduction Assessment

Yes. Source inspection on current main reproduces the missing feature with high confidence: there is no `--rate-limits` flag, no probe result payload, and no RPM/TPM table output; the PR-specific route and custom-provider gaps are also traceable from the route parser and new helper code.

## Solution Assessment

No. The proposed user-facing shape is reasonable, but this patch is not the best maintainable solution until it aligns with the route-first CLI and existing provider request/config resolution instead of adding a separate partial transport path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Forward rate-limits through the routed status path:** `src/cli/models-cli.ts:88`
  - body: The route-first CLI path for `models status` runs before Commander and currently parses only the existing probe fields. This PR adds the Commander option, but without adding `rateLimits` to `parseModelsStatusRouteArgs`, `openclaw models status --probe --rate-limits` will still call `modelsStatusCommand` without the new option in normal routed execution, so no RPM/TPM columns or rate-limit probe run.
  - confidence: 0.89
- **[P2] Use normalized provider lookup for base URLs:** `src/commands/models/list.probe.ts:572-575`
  - body: `resolveCustomBaseUrl` checks only the exact key and the normalized key string. Current model/status resolution also scans configured provider entries by normalized key, so configs keyed as `OpenAI` or another normalized-compatible alias can make the main probe succeed while the rate-limit follow-up falls back to the default host and reports `-`. Reuse the existing normalized entry lookup pattern here.
  - confidence: 0.84
- **[P2] Apply provider request controls to the extra fetch:** `src/commands/models/list.probe.ts:617-622`
  - body: The new rate-limit call uses raw `fetch` with only locally built headers. That bypasses the existing provider request config surface for extra headers, proxy, TLS, and policy, so users whose main model probe succeeds through `models.providers.*.headers` or `models.providers.*.request` can have the follow-up request fail or leave the configured network route. Build this call through the same provider request/transport helpers or explicitly carry those controls.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The PR introduces a new credentialed provider HTTP path that does not apply OpenClaw's existing provider request proxy/TLS/header controls.

Concerns:

- **[medium] Raw rate-limit fetch bypasses provider request policy:** `src/commands/models/list.probe.ts:617`
  - body: `probeRateLimits` sends API credentials with a direct `fetch` using only the endpoint headers built in this file. Existing provider traffic can be governed by `models.providers.*.headers` and `models.providers.*.request` proxy/TLS controls; bypassing that path can route secrets differently from the successful main probe and should be fixed or explicitly justified before merge.
  - confidence: 0.78
