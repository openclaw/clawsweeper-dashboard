## Summary

Keep this PR open, but it is not merge-ready: current main still lacks the requested rate-limit headroom surface, while this branch is conflicting, missing real-provider proof, and introduces provider-auth/security-boundary regressions that need rework before review can continue.

## What This Changes

The PR adds an opt-in `--rate-limits` flag to `models status --probe`, a `RateLimitInfo` result type, direct provider follow-up requests, RPM/TPM table output, JSON output, and parsing tests.

## Best Possible Solution

Expose rate-limit headroom through the existing probe/provider transport and auth contracts, wire both Commander and route-first CLI paths, and require redacted real-provider table plus JSON proof before merge.

## Reproduction Assessment

Yes for the PR defects: source inspection shows the route-first parser omits `rateLimits`, and the PR head makes a credentialed raw `fetch` outside the existing provider transport. The linked request itself is a new feature, not a broken current-main behavior.

## Solution Assessment

No. The feature direction is useful, but the current implementation should reuse the existing provider transport/auth path and update route-first parsing rather than adding a parallel request path.

- **[P1] Forward --rate-limits through the fast parser:** `src/cli/models-cli.ts:88`
  - body: Normal startup can handle `models status` through `parseModelsStatusRouteArgs` before Commander.   - confidence: 0.93

  - body: `probeRateLimits` posts API credentials with raw `fetch` to locally constructed URLs.   - confidence: 0.9

  - body: The follow-up resolver calls env/custom credential helpers separately from the main probe context. Rate-limit probing can therefore miss or use different credentials than the probe that just succeeded, especially with configured workspace/env context, runtime auth preparation, or provider plugins.
  - confidence: 0.86
