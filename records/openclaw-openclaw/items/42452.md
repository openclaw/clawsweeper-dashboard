## Summary

Keep open: current main still lacks the requested `models status --probe` rate-limit output, but this PR is not merge-ready because the flag is missed by the route-first CLI path, the secondary credentialed request bypasses existing provider transport controls, provider config/auth resolution is incomplete, and real provider proof is absent.

## What This Changes

The PR adds an opt-in `--rate-limits` flag for model auth probes, a `rateLimit` result shape, RPM/TPM table columns, parsing/formatting helpers, and focused unit tests.

## Best Possible Solution

Expose rate-limit headroom through the existing model probe CLI/JSON contract while reusing provider credential and request transport policy, or by capturing headers from the existing probe, with route-first parsing, normalized provider lookup, focused tests, docs/changelog coverage, and redacted real-run proof.

## Reproduction Assessment

Yes for the PR defects. Source inspection shows the route-first parser has no `rateLimits` field, and the PR head shows a raw secondary provider `fetch` plus incomplete custom-provider auth/config resolution.

## Solution Assessment

No. The user-facing direction is plausible, but the patch should reuse OpenClaw's existing provider credential/request policy path or capture headers from the existing probe, then wire the option through the route-first parser.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Forward the flag through the route-first parser:** `src/cli/models-cli.ts:91`
  - body: Normal startup calls `tryRouteCli` before Commander, but this patch only registers `--rate-limits` on the Commander command. The fast `models status` route still parses no `rateLimits` value, so the advertised command can run with `opts.rateLimits` undefined and produce no RPM/TPM output or rate-limit probe.
  - confidence: 0.9
- **[P2] Reuse provider transport for the rate-limit request:** `src/commands/models/list.probe.ts:617`
  - body: `probeRateLimits` sends API credentials through raw `fetch` using locally built URLs and headers. Existing model requests can depend on configured headers, auth overrides, proxy/TLS/private-network policy, guarded fetch behavior, local service leases, and transport timeouts, so this follow-up request can fail or route differently after the main probe succeeds.
  - confidence: 0.88
- **[P2] Resolve custom provider keys through the usable key path:** `src/commands/models/list.probe.ts:562-564`
  - body: This fallback uses `getCustomProviderApiKey` directly and drops non-secret markers, so env SecretRefs, known env-marker custom providers, and local marker-backed providers that the main probe can use resolve to null for the rate-limit probe. Reuse the same usable custom-provider key resolution as the normal auth path.
  - confidence: 0.86
- **[P2] Match provider configs by normalized key:** `src/commands/models/list.probe.ts:572-574`
  - body: `resolveCustomBaseUrl()` only checks exact and already-normalized provider keys. Current provider config resolution also scans entries whose keys normalize to the provider id, so configs keyed like `OpenAI` can make the main probe use a custom endpoint while the rate-limit probe falls back to the default host.
  - confidence: 0.82
