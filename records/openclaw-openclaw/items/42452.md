## Summary

## What This Changes

The PR adds opt-in `models status --probe --rate-limits` output with a `rateLimit` result field, direct provider follow-up requests, RPM/TPM table columns, JSON wiring, parsing helpers, tests, and a log ignore pattern.

## Best Possible Solution

Expose rate-limit headroom by capturing headers through the existing probe/provider transport or an explicit approved transport seam, then wire the route-first CLI parser, table output, JSON output, and focused tests on a rebased branch.

## Reproduction Assessment

Yes for the PR blockers by source inspection: the patch has unresolved helper references, the current routed parser has no rate-limit option, and the PR adds a raw provider `fetch` after successful probes. The linked request itself is a new diagnostics feature, not a broken current-main behavior.

## Solution Assessment

No; this PR's separate HTTP probe is not the safest long-term boundary for credentialed provider traffic. The better solution is to extend the existing probe/provider transport or add an explicit captured-header seam, then expose that data in CLI table and JSON output.
