## Summary

Keep this PR open for maintainer review. Current main still lacks the requested rate-limit headroom surface, but the proposed branch has blocking correctness issues, missing real behavior proof, and a security-sensitive raw provider request path that should be reworked before merge.

## What This Changes

The PR adds `--rate-limits` to `models status --probe`, parses provider `x-ratelimit-*` headers into a new `rateLimit` result field, renders RPM/TPM columns, and adds parser/formatter tests.

## Best Possible Solution

Expose rate-limit headroom through an approved CLI/JSON contract that reuses the existing provider credential and request-transport seams, or captures headers from the existing probe, with route-first parsing, normalized provider lookup, focused tests, docs/changelog coverage, and redacted real-run proof.

## Reproduction Assessment

Yes. Source inspection is enough for the PR defects: current main lacks the feature, and the PR head shows a Commander-only flag plus a raw secondary provider fetch that diverges from existing route-first and provider transport behavior.

## Solution Assessment

No. The user-facing direction is plausible, but this patch is not the best implementation until it reuses OpenClaw's provider credential/request policy path or captures headers from the existing probe and wires the option through the fast route parser.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Forward --rate-limits through the fast route parser:** `src/cli/models-cli.ts:91`
  - body: Normal startup calls `tryRouteCli` before Commander, but `parseModelsStatusRouteArgs` does not parse or return `rateLimits`. Registering the option only on the Commander command means `openclaw models status --probe --rate-limits` can run through the fast route with `opts.rateLimits` undefined, producing no RPM/TPM output and no rate-limit probe.
  - confidence: 0.9
- **[P2] Resolve custom-provider keys through the usable key path:** `src/commands/models/list.probe.ts:562-564`
  - body: This fallback uses `getCustomProviderApiKey` directly and drops non-secret markers, so models.json env SecretRefs or known env-marker custom providers that the main probe can use resolve to null for the rate-limit probe. Reuse the same usable custom-provider key resolution as the normal auth path.
  - confidence: 0.86
- **[P2] Reuse provider transport for the rate-limit request:** `src/commands/models/list.probe.ts:617`
  - body: The follow-up probe sends API credentials through raw `fetch` with locally built headers and URLs. Existing model requests can depend on configured headers, auth overrides, proxy/TLS settings, private-network policy, attribution, and guarded fetch handling, so the rate-limit probe can fail or route differently after the main probe succeeds.
  - confidence: 0.88
- **[P2] Match provider configs by normalized key:** `src/commands/models/list.probe.ts:572-574`
  - body: `resolveCustomBaseUrl` only checks exact and already-normalized provider keys. Current provider config resolution also scans entries whose keys normalize to the provider id, so configs keyed like `OpenAI` can make the main probe use a custom endpoint while the rate-limit probe falls back to the default host.
  - confidence: 0.82
