## Summary

Keep open: current main still does not expose provider rate-limit headroom, and the linked feature issue was closed in favor of this PR, but the patch is not mergeable as-is because the fast CLI route misses the new flag, the secondary credentialed request bypasses existing provider transport controls, and external real behavior proof is still missing.

## What This Changes

The PR adds an opt-in `--rate-limits` model status probe option, a `rateLimit` probe result field, RPM/TPM table columns, parser/formatter helpers, tests, and a log ignore pattern.

## Best Possible Solution

Expose rate-limit headroom through the approved CLI/JSON contract while reusing the existing provider credential and request-transport path, or by capturing headers from the existing probe, with route-first parsing, normalized provider lookup, focused tests, docs/changelog coverage, and redacted real-run proof.

## Reproduction Assessment

Yes. Source inspection is enough for the PR defects: current startup runs the fast route before Commander, the current route parser has no `rateLimits` field, and the PR head shows a raw secondary provider `fetch` after credential resolution.

## Solution Assessment

No. The user-facing direction is plausible, but this patch should reuse OpenClaw's existing provider credential/request policy path or capture headers from the existing probe, then wire the option through the fast route parser.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Forward the flag through the fast route parser:** `src/cli/models-cli.ts:91`
  - body: Normal startup calls `tryRouteCli` before Commander, but this patch only registers `--rate-limits` on the Commander command. The fast `models status` route still parses no `rateLimits` value, so the advertised command can run with `opts.rateLimits` undefined and produce no RPM/TPM output or rate-limit probe.
  - confidence: 0.9
- **[P2] Reuse the provider transport for the rate-limit request:** `src/commands/models/list.probe.ts:617`
  - body: `probeRateLimits` sends API credentials through raw `fetch` with locally built URLs and headers. Existing model requests can depend on configured headers, auth overrides, proxy/TLS/private-network policy, guarded fetch behavior, local service leases, and transport timeouts, so this follow-up request can fail or route differently after the main probe succeeds.
  - confidence: 0.88
- **[P2] Resolve custom provider keys through the usable key path:** `src/commands/models/list.probe.ts:562-564`
  - body: This fallback uses `getCustomProviderApiKey` directly and drops non-secret markers, so env SecretRefs, known env-marker custom providers, and local marker-backed providers that the main probe can use resolve to null for the rate-limit probe. Reuse the same usable custom-provider key resolution as the normal auth path.
  - confidence: 0.86
- **[P2] Match provider configs by normalized key:** `src/commands/models/list.probe.ts:572-574`
  - body: `resolveCustomBaseUrl` only checks exact and already-normalized provider keys. Current provider config resolution also scans entries whose keys normalize to the provider id, so configs keyed like `OpenAI` can make the main probe use a custom endpoint while the rate-limit probe falls back to the default host.
  - confidence: 0.82
