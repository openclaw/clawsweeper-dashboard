## Summary

Keep open. Current main still has the reported divergence path: the gateway hydrates Dashboard availability from persisted update-check state and returns during the throttle window, while the CLI status command performs a live registry-backed availability check.

## What This Changes

The issue requests that Dashboard/Overview update availability stay consistent with `openclaw update status` when `update-check.json` is stale or missing a valid available version.

## Best Possible Solution

Fix the gateway-owned update-check cache path so Dashboard availability is driven by a valid registry result when needed, previous good availability is preserved across empty npm resolution, and empty results retry on a short backoff with focused regression coverage.

## Reproduction Assessment

Yes. Seed `update-check.json` with a current `lastCheckedAt` and stale or missing `lastAvailableVersion`, then call `runGatewayUpdateCheck`; current code returns before a live registry lookup while `openclaw update status` uses the registry-backed path.

## Solution Assessment

Yes. The narrow maintainable solution belongs in the gateway update startup/cache logic, with UI tests only to protect snapshot/event consumption; the CLI and Dashboard should not need separate workaround logic once the gateway snapshot is correct.
