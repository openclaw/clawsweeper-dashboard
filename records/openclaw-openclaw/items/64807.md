## Summary

Keep open. Current main still has the reported divergence path: the gateway hydrates Dashboard update availability from persisted state and returns during the throttle window, while the CLI status path performs a registry-backed check. The fresh-empty path also still refreshes `lastCheckedAt` and writes state when npm resolution returns no version, so this remains a narrow valid bug rather than stale cleanup.

## What This Changes

Ensure the Dashboard/Overview update banner reflects the same effective package update availability as `openclaw update status` even when `update-check.json` is stale or partially empty.

## Best Possible Solution

Fix the gateway-owned update-check cache path so a valid registry result drives Dashboard availability, stale persisted availability does not block a live refresh when it is no longer trustworthy, transient empty registry results preserve the previous good state, and the next retry happens on a short bounded backoff with regression coverage.

## Reproduction Assessment

Yes. Seed `update-check.json` with a current `lastCheckedAt` and stale or missing `lastAvailableVersion`, then call `runGatewayUpdateCheck` through the existing test seam; current code returns before a registry lookup, while `openclaw update status` uses the registry-backed path.

## Solution Assessment

Yes. The narrow maintainable fix belongs in `src/infra/update-startup.ts` with focused tests; the CLI and UI should not need separate workaround logic once the gateway snapshot state is corrected.
