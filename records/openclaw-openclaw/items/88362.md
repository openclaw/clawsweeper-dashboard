## Summary

Keep open. Current main does not provide enough proof to close this: the direct WhatsApp path appears to save inbound media once and propagate the saved path, while the Gateway/WebChat offload path is the one that mints separate `media://inbound` IDs, and no merged PR or current test proves the reported WhatsApp second-read failure is gone.

## What This Changes

The issue reports that WhatsApp inbound image `media://inbound` references can fail on a second read because the emitted media ID no longer resolves to a file.

## Best Possible Solution

Add a focused WhatsApp inbound-image regression that reads the emitted `media://inbound` ref twice across a follow-up or subagent-like context, then fix only the path that emits an unstable ID or deletes the referenced file.

## Reproduction Assessment

No. The issue gives concrete steps and an exact resolver error, but this read-only review did not run a live WhatsApp session and current source does not clearly reproduce the reported double-save path for direct WhatsApp inbound media.

## Solution Assessment

Unclear. Avoiding a double save is the right fix if Gateway offload is involved, but the WhatsApp path needs a targeted regression first so the repair does not change media retention or plugin/core ownership speculatively.
