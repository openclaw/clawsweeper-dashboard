## What This Changes

The issue reports that WhatsApp inbound images referenced as `media://inbound/...` can be read once by the `image` tool but fail on later turns because the media ID points at a transient second save.

## Best Possible Solution

Add a regression that exercises a WhatsApp-saved inbound image through prompt `media://` rendering and repeated `image` tool reads, then fix only the path that emits or cleans up a non-persistent ID.

## Reproduction Assessment

No. The issue has concrete steps and an exact resolver error, but I did not run a live WhatsApp session; source inspection shows the direct WhatsApp path should propagate the original saved file path, while Gateway base64 offload is the path that mints new `media://` IDs.

## Solution Assessment

Unclear. Avoiding a double save may be right for the Gateway offload path, but the WhatsApp inbound path needs a targeted regression first so the fix does not change media retention or plugin/core ownership speculatively.
