## Summary

Keep this open. Current main has substantial Telegram transport and polling hardening, but the issue is recent, maintainer discussion explicitly left the regional sticky-IPv4 fallback case open, and related #75377 reports adjacent Telegram polling stalls on v2026.4.29. There is not enough affected-host current-main evidence to close this as implemented or cannot-reproduce.

## What This Changes

Investigate and fix Telegram long-polling where `getUpdates` stops consuming updates while the Bot API remains reachable and the Telegram transport repeats sticky IPv4 fallback warnings.

## Best Possible Solution

Keep this as an active Telegram transport/liveness investigation until the affected host retests current main with `OPENCLAW_DIAG=telegram` through a full stall/rebuild cycle and an in-gateway `resolveTelegramFetch` probe for `getMe`/`getUpdates`. Once the failing layer is identified, patch the bundled Telegram plugin path with focused regression coverage and update Telegram troubleshooting docs if operator configuration is part of the fix.

## Reproduction Assessment

No high-confidence maintainer reproduction against current main. The reporter provided concrete v2026.4.22 steps and logs, but maintainers already observed successful current-main wrapper probes locally and identified the failure as host/network-region specific.

## Solution Assessment

Unclear. Current main has the right classes of hardening for HTTP/1 dispatchers, sticky fallback reuse, dirty transport rebuilds, and status liveness, but those changes are not proven sufficient for the original host; affected-host diagnostics are the safer next step before another transport change.
