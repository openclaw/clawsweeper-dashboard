## Summary

Keep open. Current main has substantial Telegram transport and polling hardening, but the maintainer comments and newer open related reports leave the affected-host sticky IPv4/getUpdates stall unresolved rather than proven fixed.

## What This Changes

The issue reports Telegram long-polling failing to consume getUpdates despite valid Bot API reachability, with repeated sticky IPv4 fallback warnings and pending updates accumulating.

## Best Possible Solution

Keep this as an active bundled Telegram reliability investigation. Retest current main on an affected host, isolate whether the failure is dispatcher, polling lifecycle, service environment, or supervisor restart, then land the smallest Telegram-plugin fix with targeted regression coverage and troubleshooting docs if operator action is required.

## Reproduction Assessment

No high-confidence maintainer reproduction against current main. The issue has concrete host steps and logs, but maintainers saw local current-main getMe/getUpdates wrapper success and still requested affected-host diagnostics because the failure is network/region-specific.

## Solution Assessment

Unclear. The suggested dispatcher hypotheses are plausible, but current main already has cached sticky fallback dispatchers, HTTP/1-only pool options, dirty transport rebuilds, and post-4.29 timeout hardening; the safer path is diagnostics before another transport change.
