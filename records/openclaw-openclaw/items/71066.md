## What This Changes

The issue reports Telegram long polling failing to consume `getUpdates` despite a valid token and reachable Bot API, leaving inbound messages undelivered while the gateway either loops on sticky IPv4 fallback or goes silent in isolated ingress.

## Best Possible Solution

Retest current main on the affected hosts with Telegram diagnostics and an in-gateway OpenClaw transport/getUpdates probe, then land or reject the narrower canonical fixes before closing this as implemented.

## Reproduction Assessment

No high-confidence full current-main reproduction exists for the affected-host sticky IPv4 failure. Source inspection does confirm a current default isolated-ingress liveness gap that can explain silent non-recovery, but that is only part of the broader report.

## Solution Assessment

Unclear. Current main contains partial transport, watchdog, and routing fixes plus an open narrow PR for isolated ingress liveness, so the safest path is focused retest and canonical fix review rather than a new speculative transport rewrite.
