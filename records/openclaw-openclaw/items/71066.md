## Summary

Keep open. Current main has substantial Telegram polling and transport hardening, and v2026.4.29 shipped a related timeout clamp, but the maintainer discussion and current code still do not prove that the reported macOS/India sticky IPv4 fallback loop is fixed on the affected host.

## What This Changes

The issue reports that Telegram long-polling never consumes Bot API getUpdates on a macOS host in India despite a valid token and successful curl/standalone undici probes, with repeated sticky IPv4 fallback warnings.

## Best Possible Solution

Keep the issue open until the affected host produces current-main or v2026.4.29 diagnostics. The desired end state is either confirmed healthy getUpdates completion through the OpenClaw gateway fetch path, or a focused Telegram plugin fix in the transport or polling lifecycle based on that evidence.

## Reproduction Assessment

No high-confidence current-main reproduction is available. The original 2026.4.22 report has concrete steps and logs, but the missing proof is an affected-host retest with Telegram diagnostics plus an OpenClaw fetch-wrapper getMe/getUpdates probe from the gateway environment.

## Solution Assessment

Unclear. The existing #71502 watchdog work, dirty transport rebuilds, sticky fallback handling, and timeout clamp are reasonable baseline fixes, but another code change would be speculative until diagnostics identify the failing layer.
