## Summary

Keep open. Current main has Telegram polling/network hardening in the right area, but the reported host-specific sticky IPv4 fallback loop has not been disproven; maintainer discussion explicitly left it open, and a newer related open report says similar Telegram polling stalls still occur on 4.29.

## What This Changes

The issue reports Telegram long-polling never consuming Bot API getUpdates despite a valid token and reachable API, with repeated sticky IPv4 fallback warnings and pending updates accumulating.

## Best Possible Solution

Keep the report open until the affected host is retested on current main or the latest release with Telegram diagnostics and an in-gateway resolveTelegramFetch/getUpdates probe. If it still fails, fix the Telegram plugin transport or polling lifecycle with focused regression coverage for fallback reuse, dirty transport rebuild, and getUpdates liveness reporting.

## Reproduction Assessment

Unclear. The issue has concrete reproduction steps and logs from OpenClaw 2026.4.22 on a specific macOS/India host, but maintainers only reproduced successful getMe/getUpdates locally; current-main affected-host logs are still missing.

## Solution Assessment
