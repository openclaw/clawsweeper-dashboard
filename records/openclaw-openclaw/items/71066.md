## Summary

Keeping this open. Current main contains several relevant Telegram polling/network hardening changes, but the maintainer comments already explain that local wrapper probes succeeding do not disprove the India-host sticky IPv4 fallback loop, and the newer #75377 report says related Telegram polling stalls still reproduce on 2026.4.29. There is not enough affected-host current-main evidence to close this as implemented or cannot-reproduce.

## What This Changes

The issue reports that Telegram long-polling stops consuming `getUpdates` inside the gateway while direct curl and standalone undici calls to Telegram succeed, with repeated sticky IPv4 fallback warnings.

## Best Possible Solution

Keep this as an active Telegram transport/liveness investigation until an affected host retests current main with `OPENCLAW_DIAG=telegram` and an in-gateway `resolveTelegramFetch` probe. Once the failing layer is identified, patch the bundled Telegram plugin owner path with a focused regression test and update the Telegram troubleshooting docs if operator action is needed.

## Reproduction Assessment

No high-confidence maintainer reproduction against current main. The report has concrete steps and logs, but maintainers already observed successful current-main wrapper `getMe`/`getUpdates` probes locally while noting that the failure is network/region-specific and still needs affected-host diagnostics.

## Solution Assessment

Unclear. The existing current-main fixes are directionally correct for transport liveness, HTTP/1 dispatchers, sticky fallback, and dirty rebuilds, but they are not proven sufficient for the reported host; the safer path is to gather current-main diagnostic logs before adding another transport change.
