## Summary

Keep open: current main still has the outbound metadata leak, but this draft branch is not safe to merge because it misses the current payload-plan parse point and adds an unrelated tokened CI fallback.

## What This Changes

The PR attempts to strip echoed inbound metadata before outbound media directive parsing, adds a regression test, and also changes CI and macOS-related files.

## Best Possible Solution

Land a focused current-main fix in the outbound payload plan that strips only copied leading inbound metadata before directive parsing, preserves MEDIA parsing, adds regression coverage, and leaves unrelated CI/macOS changes out.

## Reproduction Assessment

Yes, from source inspection: a ReplyPayload beginning with a Conversation info fenced JSON block followed by visible text and a MEDIA directive reaches createOutboundPayloadPlanEntry(), which parses raw payload.text before any inbound metadata strip. I did not run tests because this review is read-only.

## Solution Assessment

No, this PR is not the best way to solve the issue as-is. The narrow fix belongs at the current createOutboundPayloadPlanEntry() parse point with a leading-metadata-only strip, while the CI/macOS changes should be dropped from this bug fix.

  - body: The branch changes the older normalizeReplyPayloadsForDelivery() loop, but current main now routes delivery and JSON normalization through createOutboundPayloadPlanEntry(), where parseReplyDirectives(payload.text ?? "") still sees raw metadata. Rebase the fix so the strip happens before that current parse point.
  - confidence: 0.9

  - body: stripInboundMetadata() removes leading OpenClaw timestamp prefixes even when no metadata block is present. Applying it to every outbound payload can silently change legitimate assistant replies such as logs or transcripts; use a leading metadata-only outbound helper instead.
  - confidence: 0.82
