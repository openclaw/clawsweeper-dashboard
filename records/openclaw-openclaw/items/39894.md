## Summary

Keep this PR open. The underlying metadata leak is still plausible on the shared outbound payload-plan path, but the branch is not merge-ready because it edits an older normalization point, uses an over-broad sanitizer for arbitrary outbound text, and mixes in unrelated CI changes with a concrete token-exposure concern.

## What This Changes

The PR strips inbound metadata before outbound reply directive parsing, adds a metadata-plus-MEDIA regression test, and also changes CI PR-file detection, generated macOS Swift policy punctuation, and macOS CI Vitest pool selection.

## Best Possible Solution

Keep the canonical bug open and land one focused current-main patch in createOutboundPayloadPlanEntry() that strips only copied leading inbound metadata before parseReplyDirectives(), preserves MEDIA parsing and legitimate timestamp-prefixed outbound text, adds coverage in src/infra/outbound/payloads.test.ts, and drops the unrelated CI/macOS/test-runner changes from this branch.

## Reproduction Assessment

Yes. A high-confidence source reproduction is a ReplyPayload whose text starts with a Conversation info fenced JSON block followed by visible text and a MEDIA directive; current main reaches parseReplyDirectives() in createOutboundPayloadPlanEntry() before any outbound metadata strip.

## Solution Assessment

No, not as-is. The core idea is pointed at a real bug, but the patch should be rebased into the current payload-plan entry path, use an outbound-safe leading metadata strip instead of the broad display sanitizer, and remove unrelated CI/platform changes.
