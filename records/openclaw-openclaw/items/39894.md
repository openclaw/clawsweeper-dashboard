## Summary

Close as superseded: open PR #50520 now tracks the same #39847 outbound metadata leak with a focused current-main patch, while this draft branch is stale and carries unrelated CI/macOS/test-runner changes plus a CI token exposure concern.

## What This Changes

The PR strips inbound metadata in older outbound reply normalization, adds a metadata-plus-MEDIA regression test, and also changes CI secret-scan PR-file detection, generated macOS Swift policy formatting, and macOS CI Vitest pool selection.

## Best Possible Solution

Close this superseded draft and concentrate review/merge effort on #50520 or an equivalent focused current-main fix.

## Reproduction Assessment

Yes, from source inspection. A ReplyPayload beginning with a Conversation info fenced JSON block plus visible text/MEDIA reaches createOutboundPayloadPlanEntry(), and current main parses payload.text before stripping any inbound metadata.

## Solution Assessment

No, this PR is not the best path as-is. The narrower maintainable solution is the current-main approach in #50520: strip only leading inbound metadata at the payload-plan parse boundary, keep MEDIA parsing, add regression coverage, and avoid unrelated CI/macOS edits.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Avoid passing the token to PR-controlled code:** `.github/workflows/ci.yml:304-306`
  - body: The workflow exports GITHUB_TOKEN and then runs scripts/ci-list-pr-files.mjs from the PR checkout. A contributor can modify that script in the same branch, giving branch-controlled JavaScript direct token access; keep tokened PR-file listing in trusted workflow code or remove this fallback from the bug-fix PR.
  - confidence: 0.87
- **[P2] Move the strip into the current payload plan:** `src/infra/outbound/payloads.ts:49`
  - body: The branch changes the older normalizeReplyPayloadsForDelivery() loop, but current main routes delivery and JSON normalization through createOutboundPayloadPlanEntry(), where parseReplyDirectives(payload.text ?? "") still sees raw metadata. Rebase the fix so the strip happens before that current parse point.
  - confidence: 0.9
- **[P2] Preserve timestamp-prefixed outbound text:** `src/infra/outbound/payloads.ts:49`
  - body: stripInboundMetadata() removes leading OpenClaw timestamp prefixes even when no metadata block is present. Applying it to every outbound payload can silently change legitimate assistant replies such as logs or transcripts; use a leading metadata-only outbound helper instead.
  - confidence: 0.82
