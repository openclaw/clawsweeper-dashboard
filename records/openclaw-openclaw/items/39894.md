## Summary

Keep this draft PR open: the underlying outbound metadata leak is still present on current main, but the branch is not mergeable as-is because it patches an older normalization point, uses a broader display sanitizer on outbound text, and adds unrelated CI code that exposes the workflow token to PR-controlled JavaScript.

## What This Changes

The PR strips inbound metadata before reply directive parsing, adds a metadata-plus-MEDIA regression test, and also changes CI PR-file detection, generated macOS Swift policy punctuation, and macOS CI Vitest pool selection.

## Best Possible Solution

Land one focused current-main fix in createOutboundPayloadPlanEntry() that strips only copied leading inbound metadata before parseReplyDirectives(), keeps MEDIA parsing intact, adds coverage in src/infra/outbound/payloads.test.ts, and drops the unrelated CI/macOS/test-runner changes.

## Reproduction Assessment

Yes. A ReplyPayload whose text starts with a Conversation info fenced JSON block followed by visible text and a MEDIA directive reaches parseReplyDirectives() in createOutboundPayloadPlanEntry() without a pre-parse metadata strip on current main.

## Solution Assessment

No, not as-is. The fix should be rebased into the current payload-plan parse point and use the narrower leading metadata strip behavior, while removing unrelated CI/platform changes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Avoid passing the token to PR-controlled code:** `.github/workflows/ci.yml:304-306`
  - body: The workflow exports GITHUB_TOKEN and then executes scripts/ci-list-pr-files.mjs from the PR checkout. A contributor can modify that script in the same branch, so this gives branch-controlled JavaScript direct token access; keep tokened PR-file listing in trusted workflow code or remove this fallback from the bug-fix PR.
  - confidence: 0.87
- **[P2] Move the strip into the current payload plan:** `src/infra/outbound/payloads.ts:49`
  - body: The branch changes the older normalizeReplyPayloadsForDelivery() loop, but current main now routes delivery and JSON normalization through createOutboundPayloadPlanEntry(), where parseReplyDirectives(payload.text ?? "") still sees raw metadata. Rebase the fix so the strip happens before that current parse point.
  - confidence: 0.9
- **[P2] Preserve timestamp-prefixed outbound text:** `src/infra/outbound/payloads.ts:49`
  - body: stripInboundMetadata() removes leading OpenClaw timestamp prefixes even when no metadata block is present. Applying it to every outbound payload can silently change legitimate assistant replies such as logs or transcripts; use a leading metadata-only outbound helper instead.
  - confidence: 0.82
