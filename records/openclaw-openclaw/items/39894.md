## Summary

Keep open: current main still has the metadata leak path, and this PR targets the right bug, but the patch is not mergeable as-is because it patches a stale normalization point and mixes in unrelated CI/platform changes including a token exposure risk.

## What This Changes

The PR strips copied inbound metadata before outbound reply directive parsing, adds a metadata-plus-MEDIA regression test, and also changes CI PR-file detection plus macOS/test-runner files.

## Best Possible Solution

Land one focused current-main fix in createOutboundPayloadPlanEntry() that strips only copied leading inbound metadata before directive parsing, keeps MEDIA parsing intact, adds outbound payload-plan coverage, and drops the unrelated CI/macOS/test-runner changes.

## Reproduction Assessment

Yes. Source inspection shows a ReplyPayload whose text starts with a Conversation info fenced JSON block followed by visible text and a MEDIA directive reaches parseReplyDirectives() in createOutboundPayloadPlanEntry() before any metadata strip.

## Solution Assessment

No, not as proposed. The fix should move to the current payload-plan parse point and use a narrower leading metadata strip, while removing the unrelated CI/platform changes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Avoid passing the token to PR-controlled code:** `.github/workflows/ci.yml:304-306`
  - body: The workflow exports GITHUB_TOKEN and then executes scripts/ci-list-pr-files.mjs from the PR checkout. A contributor can modify that script in the same branch, so this gives branch-controlled JavaScript direct token access; keep tokened PR-file listing in trusted workflow code or remove this fallback from the bug-fix PR.
  - confidence: 0.87
- **[P2] Move the strip into the current payload plan:** `src/infra/outbound/payloads.ts:49`
  - body: The branch changes the older normalizeReplyPayloadsForDelivery() loop, but current main routes delivery and JSON normalization through createOutboundPayloadPlanEntry(), where parseReplyDirectives(payload.text ?? "") still sees raw metadata. Rebase the fix so the strip happens before that current parse point.
  - confidence: 0.9
- **[P2] Preserve timestamp-prefixed outbound text:** `src/infra/outbound/payloads.ts:49`
  - body: stripInboundMetadata() removes leading OpenClaw timestamp prefixes even when no metadata block is present. Applying it to every outbound payload can silently change legitimate assistant replies such as logs or transcripts; use a leading metadata-only outbound helper instead.
  - confidence: 0.82
