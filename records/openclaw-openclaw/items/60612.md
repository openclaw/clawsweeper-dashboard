## Summary

Keep open. Current main still intentionally warns when the Gateway service uses a version-manager Node path, only migrates to a supported system Node when one exists, and has no shipped durable runtime pin or acknowledgement surface; the ASDF/fnm follow-ups and open runtime-path PR make this an active product/config policy item rather than cleanup-close material.

## What This Changes

The issue asks for a durable way to pin the Gateway Node executable or intentionally acknowledge/suppress doctor warnings for NVM, ASDF, fnm, and similar version-manager runtimes.

## Best Possible Solution

Decide whether intentional version-manager Gateway runtimes are supported; if yes, land a documented durable runtime-path pin with install/repair/audit coverage and live supervisor proof, and suppress the warning only for an explicit acknowledged pin.

## Reproduction Assessment

Yes, source-level. A Gateway service whose Node executable is under NVM, ASDF, fnm, Volta, or similar while no supported system Node exists will still warn and fall back to that runtime, and current main exposes no durable pin or acknowledgement option.

## Solution Assessment

Unclear. The existing system-Node warning is coherent and documented; a durable explicit runtime pin is safer than blanket suppression, but maintainers need to decide whether that workflow is supported.
