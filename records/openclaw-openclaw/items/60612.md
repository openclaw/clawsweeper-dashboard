## Summary

Keep open: current main and v2026.6.10 still intentionally warn on version-manager daemon Node paths and do not expose a supported exact-runtime pin or warning acknowledgement path. The linked open PR is the closest candidate, but it is still a product/compatibility decision rather than an implemented fix.

## What This Changes

The issue asks for a supported way to permanently use or acknowledge a version-manager Node runtime for managed gateway services so doctor stops warning about regenerated NVM/ASDF/fnm paths.

## Best Possible Solution

Keep the current system-Node default unless maintainers approve an explicit daemon runtime pin or acknowledgement contract, then implement it consistently across gateway install, node install, doctor repair, docs, and regression tests.

## Reproduction Assessment

Yes, from source inspection: current main still flags version-manager daemon Node paths and has no registered exact runtime path or suppression option. I did not run a live launchd/NVM reproduction because this was a read-only review.

## Solution Assessment

No current implementation solves it. The open `--runtime-path` direction is plausible, but it adds a compatibility-sensitive CLI/env contract and needs maintainer approval plus live upgrade/service proof before it can be called the best fix.
