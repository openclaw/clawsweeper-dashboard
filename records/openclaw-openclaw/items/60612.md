## Summary

Keep open. Current main and v2026.5.12 still intentionally warn on version-manager Gateway Node paths, migrate only when a supported system Node exists, and expose no durable Node pin or warning acknowledgement; the ASDF and fnm comments show this is a broader runtime-policy request, not an NVM-only support case.

## What This Changes

Request a durable way to pin the Gateway Node executable or suppress/acknowledge doctor warnings when the managed Gateway intentionally uses NVM, ASDF, or fnm instead of Homebrew/system Node.

## Best Possible Solution

Keep the system-Node migration default, then add a documented durable Node-path pin or explicit acknowledgement only if maintainers decide intentional version-manager Gateway runtimes are supported.

## Reproduction Assessment

Yes, source-level. A Gateway service whose Node executable is under NVM, ASDF, fnm, Volta, or similar while no supported system Node exists will still warn and fall back to that runtime, and no durable pin or suppression option is exposed.

## Solution Assessment

Unclear. The existing system-Node preference is coherent and covered by tests, but adding a durable pin or warning acknowledgement is a maintainer product/config-policy decision rather than a narrow bug fix.
