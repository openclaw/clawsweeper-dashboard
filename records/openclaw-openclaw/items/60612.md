## Summary

Keep open: current main and v2026.6.8 still intentionally warn when the Gateway service uses version-manager Node paths and only migrate to system Node when available; the remaining request is a maintainer product/API decision, with an open but not-yet-merge-ready runtime-pin PR as the closest implementation path.

## What This Changes

The issue asks for a durable way to choose or acknowledge a version-manager Node runtime for managed Gateway services instead of receiving an unfixable doctor warning.

## Best Possible Solution

Have maintainers decide the daemon runtime policy, then either land a validated explicit runtime-pin path with live launchd/systemd proof or document that system Node is the supported way to clear this warning.

## Reproduction Assessment

Yes, source-level. A Gateway service command using an nvm/fnm/asdf-style Node path still triggers the version-manager warning, and current CLI/docs expose no supported runtime-path pin or warning acknowledgement path.

## Solution Assessment

Unclear. Explicit validated runtime pinning is a plausible durable solution, but warning suppression by itself is likely the wrong layer because current policy treats version-manager daemon runtimes as fragile.
