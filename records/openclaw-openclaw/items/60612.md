## Summary

Keep open. Current main and v2026.5.12 still intentionally warn on version-manager Gateway Node paths, migrate only when a supported system Node exists, and expose no durable Node pin or warning acknowledgement; the ASDF follow-up comment confirms the same policy gap beyond NVM.

## What This Changes

Request to make Gateway doctor’s version-manager Node warning permanently resolvable through an explicit Node path, warning suppression, or clearer supported guidance.

## Best Possible Solution

Keep the warning and system-Node migration default, then add a documented durable Node pin or explicit acknowledgement only if maintainers accept intentional version-manager Gateway runtimes as supported policy.

## Reproduction Assessment

Yes, source-level. A Gateway service whose first ProgramArguments entry points under NVM, ASDF, FNM, Volta, or similar while no supported system Node is available still warns and falls back to that runtime, and no durable pin/suppression option is exposed.

## Solution Assessment

Unclear. The existing system-Node preference is a coherent default, but accepting intentional version-manager runtimes needs a maintainer policy decision before adding a config knob or acknowledgement path.
