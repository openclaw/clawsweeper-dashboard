## Summary

Keep open: current main and v2026.6.10 still intentionally warn on version-manager Node daemon paths, current install code still prefers a supported system Node when available, and there is no exact runtime-pin or warning-acknowledgement contract yet. The ASDF/fnm follow-up comments confirm this is broader than NVM, and the closest implementation path is still the open, conflicting, proof-needed https://github.com/openclaw/openclaw/pull/82290.

## What This Changes

Request a durable way to pin or acknowledge version-manager Node for managed Gateway services so doctor does not repeatedly warn when NVM, fnm, or ASDF is intentional.

## Best Possible Solution

Have maintainers decide the daemon runtime policy, then either land a validated explicit runtime-pin or acknowledgement path with live launchd/systemd/schtasks upgrade proof, or document that supported system Node is the only way to clear this warning.

## Reproduction Assessment

Yes. Source inspection shows that a Gateway service command using an nvm/fnm/asdf/mise-style Node path still triggers the version-manager warning, while current CLI/docs expose no supported exact runtime pin or acknowledgement path.

## Solution Assessment

Unclear. Explicit validated runtime pinning is a plausible durable solution, but warning suppression by itself is likely the wrong layer because current policy treats version-manager daemon runtimes as fragile.
