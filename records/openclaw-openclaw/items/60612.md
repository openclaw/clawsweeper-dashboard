## Summary

Keep open: current main and the latest release still intentionally warn on version-manager Node daemon paths and expose no supported exact-runtime pin or warning acknowledgement path, while the closest fix is an open, not-merge-ready PR that needs maintainer product approval.

## What This Changes

The issue asks for a durable way to pin the Gateway service to an intended Node executable or suppress/clarify doctor warnings when a version-manager Node is intentionally used.

## Best Possible Solution

Have maintainers decide the daemon runtime policy, then either land a validated explicit runtime-pin path with live launchd/systemd proof or document that system Node is the supported way to clear this warning.

## Reproduction Assessment

Yes, source-level. A Gateway service command using an nvm/fnm/asdf/mise-style Node path still triggers the version-manager warning, and current CLI/docs expose no supported runtime-path pin or warning acknowledgement path.

## Solution Assessment

Unclear. Explicit validated runtime pinning is a plausible durable solution, but warning suppression by itself is likely the wrong layer because current policy treats version-manager daemon runtimes as fragile.
