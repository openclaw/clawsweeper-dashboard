## Summary

Current `main` partially improves new installs by preferring a supported system Node when one exists, but the reported no-system-Node case still has no durable Node pin or warning acknowledgement; this should stay open for an explicit Gateway runtime policy decision.

## What This Changes

Review the request to persistently choose or acknowledge an intentional version-manager Node runtime for the Gateway service so doctor does not warn indefinitely.

## Best Possible Solution

Keep the diagnostic by default, continue migrating to supported system Node when available, and add a documented durable escape hatch for intentional version-manager use across NVM, ASDF, FNM, Volta, and similar runtimes.

## Reproduction Assessment

Yes. A source-level reproduction is a Gateway service command whose first program argument is under `~/.nvm` or `~/.asdf` with no supported system Node available; current code and tests show the warning remains and the repair path has no migration target in that case.

## Solution Assessment

Unclear. The current default install behavior is better when system Node exists, but a durable Node pin or acknowledgement is a product/config contract decision rather than a narrow bug fix.
