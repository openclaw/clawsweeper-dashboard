## Summary

Keep open. Current main and the latest release still warn on version-manager Gateway Node paths, migrate only when a supported system Node is available, and expose no durable Node pin or acknowledgement surface; the ASDF and fnm follow-up comments show the same policy gap beyond NVM.

## What This Changes

Request a durable way to pin the Gateway Node executable or suppress/acknowledge doctor warnings for intentional version-manager Node runtimes.

## Best Possible Solution

Keep the system-Node migration default, then add a documented durable Node pin or explicit warning acknowledgement only if maintainers accept intentional version-manager Gateway runtimes as supported policy.

## Reproduction Assessment

Yes, source-level. A Gateway service whose first ProgramArguments entry points under NVM, ASDF, fnm, Volta, or similar while no supported system Node is available still warns and falls back to that runtime, with no durable pin or suppression surface exposed.

## Solution Assessment

Unclear. The current system-Node preference is coherent and documented, but satisfying the request requires a maintainer policy decision before adding a config knob, CLI flag, or acknowledgement path.
