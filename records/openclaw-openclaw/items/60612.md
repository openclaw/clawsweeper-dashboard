## Summary

Keep open: current main and v2026.6.10 still intentionally warn on version-manager Node daemon paths, and there is still no supported exact runtime pin or warning-acknowledgement contract. The ASDF/fnm follow-up comments confirm this is broader than NVM, and the closest implementation path remains the open runtime-path PR rather than merged behavior.

## What This Changes

The issue asks for a durable way to select or acknowledge an intentionally used version-manager Node path for managed gateway services so doctor does not keep warning about NVM/asdf/fnm paths.

## Best Possible Solution

Have maintainers decide the daemon runtime policy, then either land a refreshed and validated explicit runtime-pin or acknowledgement path with live supervisor proof, or document that supported system Node is the only supported way to clear this warning.

## Reproduction Assessment

Yes, source-level: a Gateway service command using an nvm/fnm/asdf/mise-style Node path still triggers the version-manager warning, and current CLI/docs expose no supported exact runtime pin or acknowledgement path. I did not run a live macOS launchd reproduction in this read-only cleanup review.

## Solution Assessment

Unclear. Explicit validated runtime pinning is a plausible durable solution, but warning suppression by itself is likely the wrong layer because current policy treats version-manager daemon runtimes as fragile.
