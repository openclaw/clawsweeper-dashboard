## Summary

Keep open. Current main still warns on Gateway services that use version-manager Node paths and only repairs to a supported system Node when one exists, so the remaining request needs a maintainer policy decision rather than cleanup closure.

## What This Changes

Add a durable way to pin or acknowledge an intentional version-manager Node runtime so doctor no longer reports an unfixable Gateway warning.

## Best Possible Solution

Keep the system-Node migration warning by default, then add a documented durable escape hatch or explicit Node pin only if maintainers accept intentional version-manager runtimes as supported policy.

## Reproduction Assessment

Yes. A source-level reproduction is a Gateway service whose first ProgramArguments entry points under NVM, ASDF, FNM, Volta, or similar while no supported system Node is available; current code and tests show that path still warns and falls back to the version-manager runtime.

## Solution Assessment

No current solution is complete. The supported-system-Node preference solves the easy migration case, but intentional version-manager use still needs a first-class acknowledgement, explicit Node pin, or documented wrapper policy.
