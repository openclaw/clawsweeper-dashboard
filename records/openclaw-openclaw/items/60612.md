## Summary

Keep open. Current main now prefers a supported system Node over an active version-manager runtime when one is available, but the reported no-system-Node case still falls back to the version-manager Node and keeps warning without a durable acknowledgement or explicit Node pin policy.

## What This Changes

Request a durable way to pin or acknowledge an intentional version-manager Node runtime for the Gateway doctor warning.

## Best Possible Solution

Keep the warning and automatic system-Node migration by default, then add a documented durable escape hatch for intentional version-manager runtimes if maintainers accept that policy.

## Reproduction Assessment

Yes. A source-level reproduction is a Gateway service command whose first program argument is under an NVM, ASDF, FNM, or similar Node path with no supported system Node available; current code and tests show that path still warns and falls back to the version-manager runtime.

## Solution Assessment

No current solution is complete. The system-Node preference fixes the easy migration case, but intentional version-manager use still needs a first-class acknowledgement, explicit Node pin, or clearly documented wrapper policy.
