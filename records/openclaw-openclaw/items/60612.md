## Summary

Keep open. Current main still treats Gateway services launched from NVM/ASDF-style Node paths as a recommended diagnostic warning and only repairs that state by migrating to a supported system Node when one is available; it still lacks a durable `--node-path` selector or scoped acknowledgement for intentional version-manager use.

## What This Changes

The issue asks for a permanent Gateway Node path pin, a way to suppress or acknowledge intentional NVM/ASDF use, or clear guidance that the warning is safe to ignore.

## Best Possible Solution

Add a documented Gateway service runtime policy that keeps the warning by default, continues automatic migration to supported system Node when available, and gives intentional users a narrow durable escape hatch for supported Node pinning or explicit acknowledgement across NVM, ASDF, FNM, Volta, and similar managers.

## Reproduction Assessment

Yes. A Gateway service command whose first program argument is under `~/.nvm` or `~/.asdf` reproduces the audit warning in current source, and the existing service-audit test covers the NVM warning path; if no supported system Node exists, the repair path has no migration target.

## Solution Assessment

No current solution is complete. Wrapper support can persist an externally managed launcher, but the maintainable product fix should be an explicit runtime pin or scoped acknowledgement rather than manual plist edits or an undocumented wrapper bypass.
