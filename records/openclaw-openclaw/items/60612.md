## Summary

Keep open. Current main now prefers a supported system Node when one exists, but it still warns on intentional NVM/ASDF-style Gateway runtimes and exposes no durable Node-path selector or warning acknowledgement; that remaining gap is a runtime policy/config feature, not a resolved bug.

## What This Changes

Request a durable way to choose the Gateway service Node path or suppress/acknowledge doctor warnings when intentionally using NVM/ASDF-style Node.

## Best Possible Solution

Keep the diagnostic by default, continue migrating to supported system Node when available, and add a documented explicit escape hatch if maintainers want to support pinned or acknowledged version-manager runtimes.

## Reproduction Assessment

Yes, source-level: a Gateway service whose first program argument is under `.nvm`, `.asdf`, `.fnm`, or similar markers triggers the audit warning, and doctor only repairs to a supported system Node. I did not run a live macOS LaunchAgent repro in this read-only review.

## Solution Assessment

No current solution is complete. The merged system-Node preference is the right default when system Node exists, but the remaining request needs an explicit product choice between Node pinning, warning acknowledgement, or documented wrapper-only guidance.
