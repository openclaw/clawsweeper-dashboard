## Summary

Keep open. Current main and v2026.5.18 still intentionally warn on version-manager Gateway Node paths, only migrate when a supported system Node exists, and expose no durable Node pin or warning acknowledgement; the ASDF/fnm comments and open runtime-path PR show this remains an active runtime-policy/config question.

## What This Changes

The issue asks for a durable way to pin or acknowledge an intentionally version-managed Gateway Node runtime so `openclaw doctor` does not keep warning after regenerated service files.

## Best Possible Solution

Expose and document a validated exact runtime pin for managed services if intentional version-manager runtimes are supported, preserve it through doctor/reinstall, and keep warning or acknowledgement semantics explicit.

## Reproduction Assessment

Yes, source-level. A Gateway service whose Node executable is under NVM, ASDF, fnm, Volta, or similar while no supported system Node exists will still warn and fall back to that runtime, and current main exposes no durable pin or acknowledgement option.

## Solution Assessment

Unclear pending maintainer policy. An explicit validated runtime-path contract, as proposed in the open PR, looks like the maintainable direction if maintainers accept intentional version-manager service runtimes.
