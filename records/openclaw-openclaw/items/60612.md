## Summary

Keep open. Current main still warns on version-managed Gateway Node paths and only auto-repairs that condition by migrating to a supported system Node when one exists; it does not expose a durable `--node-path`/config selector or a scoped acknowledgement for intentional NVM/ASDF use. The existing wrapper support is a useful persistent workaround for externally managed launch behavior, but it is not documented as the product answer to this doctor policy question, and the ASDF comment shows the request is broader than one stale NVM plist.

## What This Changes

The issue asks for a permanent way to pin an intentional Gateway Node executable, suppress/acknowledge the NVM/ASDF doctor warning, or clarify whether the warning is safe to ignore.

## Best Possible Solution

Add an explicit, documented Gateway service runtime policy that keeps the warning by default, continues automatic migration to supported system Node when available, and gives intentional users a durable narrow escape hatch: either pin a supported Node executable through the install/repair path or acknowledge this exact warning for version-manager use. The design should cover NVM, ASDF, FNM, Volta, and similar managers consistently and explain when the warning is operationally safe versus fragile.

## Reproduction Assessment

Yes. A Gateway service command whose first program argument is under `~/.nvm` or `~/.asdf` reproduces the audit warning in current source, and `src/daemon/service-audit.test.ts` already asserts the NVM warning path; if no supported system Node is present, the doctor repair path has no migration target.

## Solution Assessment

No current solution is complete. Wrapper support can persist an externally managed launcher, but the maintainable product fix should be an explicit runtime pin or scoped acknowledgement rather than relying on manual plist edits or an undocumented wrapper bypass for this warning.
