## Summary

Keep open. Current main and the latest shipped release still intentionally warn when the Gateway service executable is an NVM/ASDF-style Node path and only migrate automatically when a supported system Node is available; the issue discussion also shows ASDF users need the same suppression or acknowledgement path.

## What This Changes

The issue requests a durable way to pin the Gateway service Node path or suppress/acknowledge doctor warnings when intentionally using version-managed Node.

## Best Possible Solution

Keep the diagnostic by default, keep migrating to supported system or bundled Node when available, and add a documented durable opt-in for explicit Node pinning or scoped acknowledgement across NVM, ASDF, FNM, Volta, and similar managers.

## Reproduction Assessment

Yes. A source-level reproduction is a Gateway service command whose first program argument is under ~/.nvm or ~/.asdf with no supported system Node; current code emits the version-manager warning and has no CLI pin or acknowledgement path.

## Solution Assessment

No current solution is complete. Manual plist edits are overwritten by design and --wrapper can persist an external launcher, but a maintainable fix needs an explicit runtime policy for pinning or acknowledgement.
