## Summary

Keep open. Current main still has a source-backed unsafe recovery path: non-abort Telegram send failures stay queued, recovery replays saved payloads with mirror metadata, and routed replies still lack a replay-stable delivery-mirror idempotency key; #74508/#76989 only partially overlap with immediate Telegram retry behavior.

## What This Changes

The issue asks OpenClaw to prevent Telegram delivery queue recovery from duplicating visible sendMessage replies and delivery-mirror transcript entries.

## Best Possible Solution

Add a narrow recovery contract that quarantines or otherwise blocks replay of uncertain non-idempotent Telegram sends and gives routed delivery mirrors a stable idempotency key, while preserving safe generic recovery.

## Reproduction Assessment

Yes, at source level: enqueue a Telegram delivery with mirror metadata, record a non-abort sendMessage network failure via failDelivery, age it past backoff, then run recoverPendingDeliveries; current main rebuilds saved params and calls deliver with skipQueue: true. I did not run a live/test reproduction because this review is read-only.

## Solution Assessment

Yes, keeping this issue open is the right outcome. The narrow maintainable fix belongs in outbound queue recovery plus mirror idempotency; #74508/#76989 reduce related Telegram retry pressure but do not remove the persisted recovery replay path.
