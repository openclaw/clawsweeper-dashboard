## Summary

Source inspection still shows the generic delivery recovery path replaying eligible queued Telegram sends after ambiguous failures, while related Telegram retry work only narrows immediate send retries and does not quarantine the persisted queue entry.

## What This Changes

The issue requests safer Telegram delivery-queue recovery so uncertain Telegram `sendMessage` failures are not replayed into duplicate user-visible replies or duplicate `delivery-mirror` transcript entries.

## Best Possible Solution

Add a narrow recovery policy that treats Telegram queue entries with ambiguous non-idempotent send failures as unsafe to replay, moves or marks them failed, and preserves mirror idempotency coverage for any replay path that remains safe.

## Reproduction Assessment

Yes from source inspection: a Telegram direct send can be queued, fail with an ambiguous non-abort error, remain pending with `lastError`, and later be delivered again by `recoverPendingDeliveries`. I did not execute a live Telegram reproduction in this read-only review.

## Solution Assessment

No current-main solution fully solves the issue. The immediate Telegram retry policy is safer, but the generic recovery path still needs a Telegram-aware unsafe-replay guard plus regression coverage.

## Repair Work Prompt

Fix the Telegram delivery queue replay bug tracked by https://github.com/openclaw/openclaw/issues/51628. On current main, `deliverOutboundPayloads` records failed Telegram sends in `delivery-queue`, and `recoverPendingDeliveries` later replays all eligible entries even when `lastError` indicates an ambiguous non-idempotent Telegram `sendMessage` or media send failure. Add a focused recovery policy that treats Telegram entries whose previous error may have landed at Telegram as unsafe to replay and moves/quarantines them instead of sending again. Reuse or extend `extensions/telegram/src/network-errors.ts` semantics rather than inventing broad string matching in core if possible; keep core generic unless a small channel-owned classifier seam is needed. Preserve safe recovery for definite pre-connect failures and other channels. Add regression coverage in `src/infra/outbound/delivery-queue.recovery.test.ts` or an adjacent Telegram-aware outbound test that covers a persisted Telegram entry with `lastError: "Network request for 'sendMessage' failed!"`, verifies recovery does not call deliver, and verifies the entry is moved/marked failed. Add mirror-level coverage only if the implementation touches transcript behavior. Add a single-line changelog fix entry because this is user-facing. Do not change Telegram API tokens, retry config defaults, unrelated streaming behavior, or broad delivery queue architecture.
