## Label Justifications

- `impact:message-loss`: The reported recovery path can duplicate outbound Telegram replies during queue replay.
- `impact:session-state`: The report also shows duplicate `delivery-mirror` assistant entries corrupting the source session transcript.

## Summary

Keep open: current main has partial protections for newly marked unknown-send entries, but legacy Telegram delivery-queue rows with a network `lastError` and no `recoveryState` can still migrate into SQLite as unmarked pending entries and reach recovery replay.

## What This Changes

Prevent Telegram delivery recovery from blindly replaying ambiguous `sendMessage` queue entries and duplicating `delivery-mirror` transcript messages.

## Best Possible Solution

Keep this issue open until maintainers choose and implement a legacy ambiguous-send policy for old Telegram queue entries, then prove recovery replay and transcript-mirror idempotency together.

## Reproduction Assessment

Yes, source-level: a legacy Telegram queue row with network `lastError` and no `recoveryState` migrates with `recovery_state = null`, loads without `recoveryState`, and bypasses the unknown-send guard into recovery delivery.

## Solution Assessment

No for closing: current main fixes newly marked unknown-send paths, but the legacy migration/recovery policy remains undecided. The best fix should be a narrow legacy quarantine or reconciliation rule plus transcript idempotency proof, not a blanket transcript-mirror disable.
