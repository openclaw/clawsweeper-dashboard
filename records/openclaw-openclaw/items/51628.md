## Label Justifications

- `P2`: The bug can duplicate Telegram replies and transcript mirror entries, but it is bounded to recovery of failed outbound delivery queue entries.
- `impact:message-loss`: The issue is about incorrect Telegram delivery semantics where recovery can resend an old outbound reply.
- `impact:session-state`: The issue is about duplicate `delivery-mirror` entries making the session transcript diverge from the real conversation.

## Summary

Keep open. Current main protects newly written queue entries with explicit unknown-send recovery state, but legacy Telegram queue files from the reported release can still lack that state and fall into normal replay after backoff.

## What This Changes

The issue asks for Telegram delivery recovery to avoid replaying uncertain old `sendMessage` queue entries and to keep `delivery-mirror` transcript appends deduped across recovery.

## Best Possible Solution

Add a maintainer-approved legacy policy for ambiguous Telegram queue files, route those entries through reconciliation or quarantine instead of blind replay, and cover both recovery replay and transcript mirror dedupe with regression tests.

## Reproduction Assessment

Yes, at source level. A legacy Telegram queue entry with `retryCount`, a network `lastError`, no `recoveryState`, and elapsed backoff is normalized only with `lastAttemptAt` and can then enter the normal replay branch.

## Solution Assessment

Unclear as an implementation choice. The technical boundary is narrow, but maintainers need to choose whether legacy ambiguous Telegram sends should be quarantined, reconciled, or replayed because each changes user-visible recovery behavior.
