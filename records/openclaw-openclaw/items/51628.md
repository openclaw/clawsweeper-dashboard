## Summary

Keep open. Current main still stores failed Telegram outbound deliveries with only retry/backoff error text and later replays eligible queued payloads through recovery, while routed reply mirrors still lack a stable idempotency key. The existing Telegram send path is conservative for immediate non-idempotent retries, but that uncertainty policy is not carried into the write-ahead queue recovery path.

## What This Changes

The issue requests preventing Telegram delivery-queue recovery from replaying uncertain sendMessage failures and keeping delivery-mirror transcript appends idempotent across recovery/replay paths.

## Best Possible Solution

Implement a narrow outbound recovery contract: persist or derive uncertain-delivery state for non-idempotent Telegram sends, quarantine or require confirmation for uncertain sendMessage/send-media failures during startup and reconnect recovery, preserve safe recovery for definitely-unsent cases, and derive/persist a stable delivery-mirror idempotency key for the original queued delivery.

## Reproduction Assessment

Yes. A focused unit-level reproduction is clear: enqueue a Telegram delivery with mirror metadata, record a non-abort sendMessage network failure through failDelivery, make it backoff-eligible, and run recoverPendingDeliveries; current main will call deliver again with the saved payload and mirror metadata, and routed replies provide no stable mirror idempotency key.

## Solution Assessment

Yes, keeping this open is the right outcome. The maintainable fix should be in the outbound queue/recovery plus mirror metadata path, reusing Telegram's existing non-idempotent send-error classification instead of adding broad channel retry changes.
