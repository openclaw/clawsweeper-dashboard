## Label Justifications

## Summary

Keep open: current main now protects newly marked post-send queue states, but legacy Telegram delivery-queue rows from the reported release can still migrate without recovery state and reach recovery replay.

## What This Changes

The issue asks OpenClaw to prevent ambiguous Telegram `sendMessage` recovery from duplicating visible replies and `delivery-mirror` transcript entries.

## Best Possible Solution

Keep this issue open until legacy Telegram queue rows are either classified, quarantined, or migrated with recovery state, with regression coverage for recovery replay and transcript mirror idempotency.

## Reproduction Assessment

Yes. A legacy Telegram queued delivery with network `lastError` and no `recoveryState` can migrate with null recovery state, inflate without `recoveryState`, bypass the unknown-send guard, and reach the recovery delivery path.

## Solution Assessment

No for closing. Current main fixes newly marked post-send entries, but the legacy migration/recovery policy for old unmarked Telegram queue rows remains undecided.
