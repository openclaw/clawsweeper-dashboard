## Summary

Keep open. Current main fixed the main new-entry replay hazard by marking started/unknown platform sends and refusing blind recovery replay, but legacy queued Telegram failures from the reported release can still lack `recoveryState`; current normalization only backfills `lastAttemptAt`, so those entries can still fall through to `deliver(... skipQueue)` after backoff.

## What This Changes

The issue asks OpenClaw to avoid replaying ambiguous Telegram `sendMessage` queue entries and to prevent duplicate `delivery-mirror` transcript appends during recovery.

## Best Possible Solution

Keep the current lifecycle safety for newly written entries and add an explicit legacy/ambiguous queued-delivery recovery policy that uses adapter-owned reconciliation or quarantines entries without plugin-specific core shortcuts.

## Reproduction Assessment

Yes. At source level on current main, a legacy queued Telegram retry with `retryCount`/`lastError` and no `recoveryState` is normalized only with `lastAttemptAt`, then recovery falls through to `deliver(..., skipQueue: true)` after backoff; I did not run a test because this review is read-only.

## Solution Assessment

No. Closing as implemented would miss the legacy retry shape from the reported release; the safer solution is a maintainer-approved legacy unknown-send policy that preserves the current no-blind-replay contract without plugin-specific core branching.
