## Label Justifications

- `impact:message-loss`: The issue is about incorrect Telegram message delivery semantics caused by recovery replay.
- `impact:session-state`: The issue is about duplicate `delivery-mirror` entries corrupting the session transcript view.

## Summary

Keep open. Current main protects newly written unknown-send queue entries from blind replay, but legacy Telegram queue files from the reported release can still lack `recoveryState` and fall through to normal recovery replay after backoff.

## What This Changes

The issue asks OpenClaw to prevent Telegram delivery-queue recovery from replaying uncertain old `sendMessage` entries and duplicating `delivery-mirror` transcript content.

## Best Possible Solution

Add a maintainer-approved legacy policy for ambiguous Telegram queue files that routes them through reconciliation or quarantine, with regression coverage for recovery replay and transcript mirror dedupe.

## Reproduction Assessment

Yes, at source level. A legacy Telegram queue entry with `retryCount`, `lastError`, and no `recoveryState` is normalized only with `lastAttemptAt`; after backoff it can enter the normal replay branch.

## Solution Assessment

Unclear as an implementation choice. The safest technical fix is narrow, but maintainers need to choose the legacy quarantine-versus-replay policy because either option changes user-visible failure behavior.
