## Label Justifications

- `P2`: The bug can duplicate Telegram replies and transcript entries, but it is a bounded recovery edge case rather than a broad runtime outage.
- `impact:message-loss`: The issue is about duplicated user-visible Telegram delivery from recovery replay.
- `impact:session-state`: The issue is about duplicate `delivery-mirror` entries polluting the session transcript.

## Summary

Keep open. Current main protects newly written outbound sends with explicit unknown-send recovery state, but legacy Telegram queue files from the reported release can still lack that state and remain eligible for normal recovery replay after backoff.

## What This Changes

The issue requests safer Telegram delivery-queue recovery so ambiguous `sendMessage` failures do not replay old replies or append duplicate `delivery-mirror` transcript entries.

## Best Possible Solution

Keep the current unknown-send reconciliation for new entries and add a maintainer-approved legacy policy that either reconciles when possible or quarantines old Telegram network-failure queue files, with focused replay and transcript-mirror regression coverage.

## Reproduction Assessment

Yes, at source level. A legacy queued Telegram retry with `retryCount`, `lastError`, and no `recoveryState` is normalized only with `lastAttemptAt`; after backoff, recovery can call the normal replay path instead of reconciling unknown send state.

## Solution Assessment

Unclear as an implementation choice. Quarantine is safer for duplicate sends, but it may drop replies that never reached Telegram, so maintainers should choose the legacy replay-versus-quarantine policy before implementation.
