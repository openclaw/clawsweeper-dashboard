## Summary

Keep open. Current main fixed the main new-entry hazard by marking started or unknown platform sends and refusing blind recovery replay, but legacy queued Telegram failures from the reported release can still lack `recoveryState`; after backoff they can bypass reconciliation and be delivered again with `skipQueue`.

## What This Changes

Prevent Telegram delivery recovery from replaying ambiguous `sendMessage` queue entries and from duplicating `delivery-mirror` transcript appends.

## Best Possible Solution

Keep the current lifecycle safety for new entries and add a maintainer-approved legacy ambiguous-delivery policy that uses adapter reconciliation when available or quarantines old uncertain sends with focused regression coverage.

## Reproduction Assessment

Yes. At source level, a legacy queued Telegram retry with `retryCount`, `lastError`, and no `recoveryState` is normalized only with `lastAttemptAt`; once backoff elapses, recovery can call `deliver(..., skipQueue: true)` instead of reconciling unknown send state.

## Solution Assessment

No, not as a hard-coded Telegram replay block in core. The safer maintainable solution is an explicit legacy unknown-send policy that preserves the adapter-owned `reconcileUnknownSend` contract and avoids broad channel regressions.
