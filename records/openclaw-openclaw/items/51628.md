## Summary

Keep open. Current main has newer Telegram immediate-send retry safeguards, but the outbound queue recovery path still replays backoff-eligible queued Telegram deliveries after non-abort failures and still forwards mirror metadata without a stable routed-reply idempotency key; the May 3 comment correctly describes #74508 as only a partial overlap.

## What This Changes

The issue reports Telegram delivery recovery replaying uncertain sendMessage failures, causing duplicate user-visible replies and duplicate delivery-mirror transcript entries.

## Best Possible Solution

Add a narrow recovery contract that prevents replay of uncertain non-idempotent Telegram sends and gives delivery-mirror appends a replay-stable idempotency key while preserving safe recovery for definitely-unsent entries.

## Reproduction Assessment

Yes, at source level: enqueue a Telegram delivery with mirror metadata, record a non-abort sendMessage network failure via failDelivery, age it past backoff, then run recoverPendingDeliveries; current main will call deliver again with skipQueue and mirror metadata. I did not run this as a test because the review was read-only.

## Solution Assessment

Yes, keeping this open is the right outcome. The narrow maintainable fix belongs in outbound queue recovery plus mirror idempotency, while #74508 only reduces related Telegram retry pressure and does not remove the unsafe recovery replay path.
