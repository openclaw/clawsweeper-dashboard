## Summary

Keep open. Current main protects newly queued unknown sends with `recoveryState` and adapter reconciliation, but legacy Telegram queue entries from the reported release can still lack `recoveryState` and fall through to `skipQueue` recovery replay after backoff; the merged Telegram stream-preview work only partially addressed adjacent send retry conditions.

## What This Changes

Prevent Telegram delivery recovery from blindly replaying uncertain `sendMessage` queue entries and duplicating delivery-mirror transcript entries.

## Best Possible Solution

Keep the current lifecycle reconciliation for new entries and add a maintainer-approved legacy ambiguous-entry policy that reconciles when possible or quarantines old Telegram network-failure queue files, with focused replay and transcript-mirror regression coverage.

## Reproduction Assessment

Yes, at source level. A legacy queued Telegram retry with `retryCount`, `lastError`, and no `recoveryState` is normalized only with `lastAttemptAt`; once backoff elapses, recovery can call `deliver(..., skipQueue: true)` instead of reconciling unknown send state.

## Solution Assessment

No, this should not be closed as fixed by the stream-preview work. The maintainable fix is an explicit legacy unknown-send policy that preserves the adapter-owned `reconcileUnknownSend` contract and avoids broad channel regressions.
