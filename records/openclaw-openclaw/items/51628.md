- `impact:message-loss`: The issue is about duplicate Telegram outbound delivery after recovery replay.
- `impact:session-state`: The issue also reports duplicate `delivery-mirror` assistant entries in the source session transcript.

## Summary

Keep open. Current main and v2026.6.8 protect newly marked unknown-send entries, but legacy Telegram queue entries from the reported release can still migrate without `recoveryState` and reach the recovery replay path.

## What This Changes

The issue asks OpenClaw to stop replaying ambiguous Telegram `sendMessage` queue entries and prevent duplicate `delivery-mirror` transcript appends after recovery.

## Best Possible Solution

Keep this issue open until maintainers choose and implement a legacy ambiguous-send policy: migrate/quarantine or reconcile old Telegram `sendMessage` queue entries, then prove transcript-mirror idempotency across recovery.

## Reproduction Assessment

Yes, source-reproducible: a legacy Telegram queued delivery with network `lastError` but no `recoveryState` bypasses the unknown-send guard and reaches the recovery deliver path. I did not run a live Telegram replay in this read-only review.

## Solution Assessment

Unclear. New unknown-send entries are guarded, but the remaining best fix needs maintainer policy on whether legacy ambiguous Telegram sends should be quarantined, migrated into unknown state, or reconciled before replay.
