## Summary

Keep open. Current main and v2026.5.22 protect newly marked unknown-send entries, but legacy Telegram queue JSON from the reported release can still lack `recoveryState`; after backoff those entries fall through to normal replay instead of quarantine/reconciliation.

## What This Changes

The issue asks for Telegram outbound recovery to avoid replaying ambiguous queued `sendMessage` replies and to keep `delivery-mirror` transcript appends deduped across recovery.

## Best Possible Solution

Define the legacy ambiguous-send policy, then add a focused recovery/storage fix that treats old Telegram network-error queue entries as unsafe to blind replay or reconciles them before delivery, while preserving current mirror idempotency safeguards.

## Reproduction Assessment

Yes, at source level. A legacy retry entry with Telegram `lastError` and no `recoveryState` is normalized with `lastAttemptAt` only, and after backoff the recovery path falls through to `deliver(...)` instead of the unknown-send quarantine/reconciliation branch.

## Solution Assessment

Unclear until maintainers choose the legacy policy. The code boundary is narrow, but deciding whether old ambiguous sends should be quarantined, reconciled, or replayed is a product/upgrade-safety decision.
