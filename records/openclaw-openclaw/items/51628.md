## Summary

Keep open. Current main protects newly written unknown sends with `recoveryState` and adapter reconciliation, but legacy Telegram queue entries from the reported release can still lack that state and fall through to normal recovery replay after backoff; related TTL and outbound-dedupe work is still open and not a clean superseding fix.

## What This Changes

The issue asks OpenClaw to prevent Telegram delivery-queue recovery from replaying ambiguous `sendMessage` failures and duplicating delivery-mirror transcript entries.

## Best Possible Solution

Keep the current unknown-send reconciliation for new entries and add a maintainer-approved legacy policy that reconciles when possible or quarantines old Telegram network-failure queue files, with focused replay and transcript-mirror regression coverage.

## Reproduction Assessment

Yes, at source level. A legacy queued Telegram retry with `retryCount`, `lastError`, and no `recoveryState` is normalized only with `lastAttemptAt`; once backoff elapses, recovery can call `deliver(..., skipQueue: true)` instead of reconciling unknown send state.

## Solution Assessment

Unclear as an implementation choice. Quarantine is safer for duplicate sends, but it can drop replies that never reached Telegram, so maintainers need to choose the replay-versus-quarantine policy before implementation.
