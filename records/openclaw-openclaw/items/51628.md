## Summary

Keep open. Current main has conservative immediate Telegram send retry behavior and the related discussion notes useful partial mitigations, but the write-ahead delivery queue still stores failed outbound entries with only retry/backoff error metadata and later replays eligible payloads with mirror metadata, so the recovery-level duplicate delivery and transcript-mirror gap remains.

## What This Changes

Prevent Telegram delivery queue recovery from replaying ambiguous failed sendMessage entries and duplicating delivery-mirror transcript entries.

## Best Possible Solution

Add a narrow outbound recovery policy that marks or derives uncertain non-idempotent Telegram sends, quarantines them instead of replaying unless they are definitely unsent, and derives a stable delivery-mirror idempotency key from the original queued delivery.

## Reproduction Assessment

Yes from source inspection. Enqueue a Telegram delivery with mirror metadata, record a plain `Network request for 'sendMessage' failed!` through failDelivery, age it past backoff, and recoverPendingDeliveries will call deliver with skipQueue: true and the same mirror metadata; I did not run live Telegram in this read-only pass.

## Solution Assessment

Yes, keeping this open is the right outcome. The narrow maintainable fix belongs in the outbound queue recovery and mirror metadata path; #74508 and #66150 only address adjacent partial cases.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/51628. Current main persists outbound entries before Telegram sends, records failed sends with retryCount/lastAttemptAt/lastError only, and recoverPendingDeliveries/drainPendingDeliveries later replay saved payloads with skipQueue: true and mirror metadata. Implement a narrow recovery contract for non-idempotent Telegram sends: preserve or derive uncertain-delivery state from failed sendMessage/send-media outcomes, especially plain grammY `Network request for 'sendMessage' failed!`, timeout, reset, or other post-connect-ambiguous failures; quarantine or move those entries to failed instead of blindly replaying them; keep safe recovery for definitely-unsent pre-connect failures and non-Telegram channels. Persist or derive a stable delivery-mirror idempotency key from the original queued delivery/session/recipient and use it on every recovery replay path so transcript append stays idempotent. Add focused regression coverage in delivery-queue recovery/reconnect, deliver or route-reply, transcript, and Telegram network-error tests as needed. Do not add a config flag, broaden general channel retry policy, or disable crash recovery for definitely-unsent entries. Add a CHANGELOG.md entry because this fixes user-visible duplicate Telegram delivery.
