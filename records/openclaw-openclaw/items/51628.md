## Summary

Keep open. Current main and the latest release protect newly marked unknown-send queue entries, but legacy Telegram queue entries from the reported release can still migrate or load without `recoveryState`, bypass the unknown-send guard, and be replayed by startup/reconnect recovery; the merged related idempotency PR explicitly leaves this recovery policy unresolved.

## What This Changes

The issue reports Telegram recovery replaying ambiguous queued `sendMessage` replies, causing duplicate user delivery and duplicate `delivery-mirror` transcript entries.

## Best Possible Solution

Implement a maintainer-approved legacy ambiguous-send policy that migrates or quarantines old Telegram `sendMessage` network-error queue entries, or reconciles them before replay, while preserving transcript-mirror idempotency.

## Reproduction Assessment

Yes, source-reproducible: a legacy Telegram queued delivery with `lastError` but no `recoveryState` bypasses the unknown-send guard and reaches the recovery deliver path. I did not run a live Telegram replay in this read-only review.

## Solution Assessment

Unclear. New entries are guarded, but the remaining best fix needs maintainer policy on whether legacy ambiguous Telegram sends should be migrated, quarantined, or reconciled instead of blanket-disabling transcript mirroring.
