## Summary

Keep open. Current main protects newly marked unknown-send queue entries, but the legacy queue path from the reported release can still migrate or load without `recoveryState`, bypass the unknown-send guard, and be replayed after Telegram reconnect/startup recovery.

## What This Changes

The issue reports that Telegram delivery recovery can replay an ambiguous failed `sendMessage` and append duplicate `delivery-mirror` transcript entries.

## Best Possible Solution

Keep this open until maintainers choose and implement a legacy ambiguous-send policy: quarantine or migrate old Telegram `sendMessage` queue entries, or add a verified reconciliation path, while preserving transcript-mirror idempotency.

## Reproduction Assessment

Yes, source-reproducible: current main still lets legacy Telegram queue entries without `recoveryState` bypass the unknown-send guard and call the recovery `deliver` path. I did not run a live Telegram replay in this read-only review.

## Solution Assessment

Unclear. New entries are guarded, but the remaining best fix needs maintainer policy on whether legacy ambiguous Telegram sends should be migrated, quarantined, or reconciled instead of blanket-disabling transcript mirroring.
