## Summary

Keep open: the linked heartbeat bug is source-provable and this PR targets the right area, but the diff introduces blocking heartbeat delivery regressions and still lacks after-fix real behavior proof.

## What This Changes

The PR changes heartbeat pending-final-delivery handling in `agent-runner`, `get-reply`, and `heartbeat-runner` to skip or clear stuck heartbeat pending state.

## Best Possible Solution

Land a narrow heartbeat/session-state repair that prevents ack-only heartbeat text from entering pending replay, clears unreachable no-target or pseudo-target heartbeat state, preserves configured `ackMaxChars` and real-channel durable delivery, and adds focused regression coverage.

## Reproduction Assessment

Yes, source-reproducible: seed a heartbeat main session with `pendingFinalDeliveryText`; current `getReplyFromConfig` returns the stale text and updates `updatedAt` before the normal heartbeat prompt can run. I did not run a live gateway scenario in this read-only review.

## Solution Assessment

No: the heartbeat-specific direction is appropriate, but this patch must share the configured `ackMaxChars` contract and avoid clearing deliverable real-channel heartbeat alerts solely by retry count. It also needs real after-fix behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve configured ack thresholds:** `src/auto-reply/reply/agent-runner.ts:1911`
  - body: `runHeartbeatOnce` suppresses heartbeat replies using the resolved `ackMaxChars`, but this guard calls `stripHeartbeatToken` with the default threshold. With `ackMaxChars: 0`, existing coverage expects `HEARTBEAT_OK all good` to be deliverable; this code skips saving it for durable retry, so a failed send loses that visible heartbeat alert.
  - confidence: 0.88
- **[P2] Keep real heartbeat alerts pending through outages:** `src/auto-reply/reply/get-reply.ts:350-359`
  - body: This branch clears `pendingFinalDelivery` for every heartbeat once the attempt count reaches five, regardless of whether the pending text is a real alert for a configured channel. A channel outage lasting five heartbeat intervals will discard the alert before another delivery attempt; limit clearing to unreachable pseudo-target or ack-only state, or preserve durable retry for deliverable heartbeat notifications.
  - confidence: 0.8
