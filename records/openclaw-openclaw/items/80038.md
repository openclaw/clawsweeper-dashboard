## Summary

Keep open: this is a plausible fix for a source-proven heartbeat pending-final-delivery loop, but the patch currently drops durable retry for heartbeat replies that are visible under a stricter `ackMaxChars` setting and it lacks after-fix real behavior proof.

## What This Changes

The PR updates heartbeat pending-final-delivery handling in `agent-runner`, `get-reply`, and `heartbeat-runner` to avoid or clear stuck heartbeat replay state.

## Best Possible Solution

Land a narrow heartbeat-specific repair that prevents bare or effectively empty heartbeat acknowledgements from entering pending replay, clears unreachable heartbeat pending state, preserves configured `ackMaxChars` visibility and real-channel durable delivery, and adds focused regression coverage.

## Reproduction Assessment

Yes: current main source gives a deterministic path by seeding a heartbeat main session with `pendingFinalDeliveryText`; `getReplyFromConfig` replays it and updates freshness before normal heartbeat processing. I did not run a live gateway reproduction in this read-only review.

## Solution Assessment

No: the overall heartbeat-specific direction is right, but the proposed `agent-runner` guard must share the same `ackMaxChars` contract as `runHeartbeatOnce` or restrict itself to truly bare acknowledgements. The PR also needs real after-fix behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Respect configured heartbeat ack thresholds:** `src/auto-reply/reply/agent-runner.ts:1911`
  - body: `runHeartbeatOnce` decides whether a heartbeat reply is suppressible using the configured `ackMaxChars`, and existing coverage treats `HEARTBEAT_OK all good` as deliverable when `ackMaxChars` is `0`.
