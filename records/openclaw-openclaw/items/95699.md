## Summary

Keep this PR open: current main still has the per-access case-insensitive session-key scans, and the PR is a focused performance cleanup with no definite code-level regression found. It still needs maintainer-visible upgrade proof and real behavior proof because it removes a shipped lazy legacy-key fallback from session-state paths.

## What This Changes

The branch removes runtime case-insensitive session-store key scans and scanLegacyKeys plumbing, relying on canonical writes plus startup migration, and adds migration-focused coverage for mixed-case key convergence.

## Best Possible Solution

Land the cleanup only after maintainers accept startup migration as the single legacy-key owner and the PR includes redacted proof for both large-store performance and mixed-case upgrade convergence.

## Reproduction Assessment

Yes, source inspection shows current main still does a full Object.keys scan in findStoreKeysIgnoreCase from session lookup paths; I did not run a 5000-key benchmark, so the multi-second stall itself is source-reproducible rather than locally reproduced.

## Solution Assessment

Yes, this appears to be the right layer if maintainers accept migration-only legacy cleanup: writes and startup migration own canonicalization, while gateway hot paths stop rediscovering legacy case variants. The safer pre-merge state is to add real upgrade/performance proof before removing the fallback.
