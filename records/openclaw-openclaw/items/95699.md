## Label Justifications

- `P1`: The PR can break local TUI startup/history for existing users with legacy mixed-case session keys.
- `merge-risk: 🚨 compatibility`: The diff removes runtime compatibility fallback for shipped mixed-case session-store keys and depends on migrations instead.

## Summary

Keep open: the performance direction is sound, but the PR introduces a local TUI startup race after removing the runtime fallback, and the PR still needs real behavior proof before merge.

## What This Changes

The PR removes case-insensitive session-store key scans from runtime lookup paths and relies on startup/session migrations to canonicalize mixed-case keys.

## Best Possible Solution

Preserve the exact-only runtime fast path only after gateway and local TUI entry points finish or gate session-key migration before any session read, with a delayed-migration regression test and posted large-store timing proof.

## Reproduction Assessment

Yes from source inspection: the PR branch starts migrateOrphanedSessionKeys as a detached promise, then queues local TUI readiness, while the readiness path immediately calls session list/history methods that now use exact-only store lookups. A delayed migration plus a mixed-case legacy key is enough to reproduce the missing-session window.

## Solution Assessment

No. Removing the per-access scan is the right long-term shape for the gateway hot path, but the local TUI migration must be awaited or otherwise gated before the fallback disappears from all runtime reads.
