## Summary

Keep open: the Claude credential-collision bug is real, but this PR is still unsafe because it changes custom ACPX command env inheritance by default, misses canonical auth keys, lacks live patched-gateway ACP proof, and is currently conflicting with main.

## What This Changes

The PR adds default-on ACPX provider credential env scrubbing via an `env -u` wrapper, wires a new `acp.scrubProviderEnv` config field, and adds ACPX/config tests plus generated config baseline updates.

## Best Possible Solution

Rework the fix around canonical provider-env metadata, limit default-on stripping to maintainer-approved built-in harness launch paths, preserve custom-command inheritance unless explicitly opted in, and add redacted live gateway ACP proof.

## Reproduction Assessment

Yes for the PR-introduced compatibility regression: source inspection shows custom or unknown ACPX commands can be scrubbed by agent id rather than by a built-in harness boundary. The original Claude failure has plausible before evidence, but there is no live after-fix gateway spawn proof.

## Solution Assessment

No. Command decoration may be acceptable for built-in harnesses, but this implementation is too broad for custom commands and should reuse canonical provider auth keys instead of a new local table.
