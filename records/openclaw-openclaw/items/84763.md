## Best Possible Solution

Rework the fix around canonical provider-env metadata, limit default-on stripping to maintainer-approved built-in ACPX harness launch paths, preserve custom-command inheritance unless explicitly opted in, and add redacted live gateway ACP proof.

No. Command decoration may be acceptable for built-in harnesses, but this implementation is too broad for custom commands and should reuse canonical provider auth keys instead of a new local table.
