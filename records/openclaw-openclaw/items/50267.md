## Summary

Keep open: the OpenRouter image-attribution gap is still source-reproducible, but this PR's provider-only header injection bypasses route-aware policy and the branch now targets an old image path.

## What This Changes

This PR adds provider-derived attribution headers to the image understanding completion call for OpenRouter free-tier requests.

## Best Possible Solution

Apply route-aware OpenRouter attribution in the current image/direct-completion path and add regression coverage proving OpenRouter image requests receive attribution while custom OpenAI-compatible routes do not receive hidden OpenAI/OpenAI Codex headers.

## Reproduction Assessment

No live OpenRouter billing reproduction is present. The issue is source-reproducible by following the OpenRouter image provider into the generic image runtime and direct completion fallback.

## Solution Assessment

No. The proposed provider-only header lookup is not the safest fix; it should use the route-aware provider request policy or an OpenRouter-only direct-completion helper.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

- **[P2] Gate attribution on the resolved route:** `src/media-understanding/providers/image.ts:189-201`
