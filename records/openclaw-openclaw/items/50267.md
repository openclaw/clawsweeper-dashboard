## Summary

Keep open: the OpenRouter image attribution bug is still source-reproducible, but this PR is not safe to merge because it targets a removed file and uses provider-only header attribution instead of endpoint-aware request policy.

## What This Changes

The PR adds provider attribution headers to the old media-understanding image provider by resolving headers from `model.provider` and passing them into `complete()`.

## Best Possible Solution

Patch the live `src/media-understanding/image.ts` completion path with endpoint-aware request-policy headers, preserving native OpenRouter attribution while proving custom proxy base URLs do not receive those headers.

## Reproduction Assessment

Yes, by source inspection rather than live billing. Current main's image `complete()` call does not pass request-policy headers, and normal OpenRouter wrapper registration is not reached by `registerProviderStreamForModel` because the plugin exposes `wrapStreamFn` while that helper resolves `createStreamFn` or transport requirements.

## Solution Assessment

No. The PR intent is correct, but the patch should target the current image runtime and use endpoint-aware request policy instead of `resolveProviderAttributionHeaders(model.provider)` in a removed file.
