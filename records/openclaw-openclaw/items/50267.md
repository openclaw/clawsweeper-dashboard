## Summary

Keep open and require changes: the reported image-tool OpenRouter attribution gap is still valid on current main, but this PR edits a removed image-provider path and uses provider-only attribution instead of the endpoint-aware request policy.

## What This Changes

The PR imports `resolveProviderAttributionHeaders` in `src/media-understanding/providers/image.ts` and passes provider-derived attribution headers to that file's image `complete()` call.

## Best Possible Solution

Apply image-understanding completion headers through the existing endpoint-aware request-policy helper on the active `src/media-understanding/image.ts` call path, with regression coverage for native OpenRouter attribution and custom base URL non-injection.

## Reproduction Assessment

Yes. Source inspection of current main shows the active image completion path reaches `complete()` without `headers`; a focused mock of `complete()` with an OpenRouter image model can assert the missing `HTTP-Referer` and `X-OpenRouter-*` options.

## Solution Assessment

No. The PR intent is right, but the submitted patch targets a stale file and bypasses host-aware request policy; the narrower maintainable fix belongs at the active image completion call site.
