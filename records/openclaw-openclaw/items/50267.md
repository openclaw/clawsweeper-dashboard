## Summary

Keep open: the OpenRouter image/direct-completion attribution gap is still plausible on current main, but this PR should not merge as-is because it applies provider-only headers, lacks real behavior proof, and is now conflicting with the moved image-understanding path.

## What This Changes

The PR imports `resolveProviderAttributionHeaders` into the old image provider path and passes provider-derived headers into the image `complete()` options.

## Best Possible Solution

Use the route-aware provider request policy for image/direct-completion requests and add regression coverage for OpenRouter image headers plus custom OpenAI-compatible routes receiving no hidden OpenAI/OpenAI Codex headers.

## Reproduction Assessment

No high-confidence live OpenRouter billing reproduction is present. The missing and unsafe header behavior is source-reproducible by following the image `complete()` fallback, OpenRouter plugin registration, and provider attribution policy.

## Solution Assessment

No. Passing `resolveProviderAttributionHeaders(model.provider)` is not the narrow maintainable fix because it bypasses base URL/request-route gates; use `resolveProviderRequestPolicyConfig` or an equivalent route-aware direct-completion helper.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Gate attribution on the resolved request route:** `src/media-understanding/providers/image.ts:189`
  - body: This resolves attribution only from `model.provider`, so `openai` and `openai-codex` image models routed through a custom OpenAI-compatible `baseUrl` can start sending hidden `originator`/`version`/`User-Agent` headers to that proxy. The existing request policy gates those headers by the resolved route, and this path should use that same policy or a narrow OpenRouter-only helper.
  - confidence: 0.9
