## Summary

Keep open. The OpenRouter image-attribution gap is plausible and has related prior discussion, but this PR should not merge as written because it applies provider-only attribution headers to OpenAI/OpenAI-Codex image requests without the native-route/baseUrl gate used by chat, and the contributor has not provided after-fix real behavior proof.

## What This Changes

The PR imports `resolveProviderAttributionHeaders` and passes its result as `headers` to the image-model `complete()` call.

## Best Possible Solution

Use a route-aware provider request policy for media-understanding completions so verified OpenRouter routes get documented attribution without sending hidden OpenAI headers to custom proxies, then add focused regression coverage and real runtime proof.

## Reproduction Assessment

Yes for the PR-introduced regression: the diff and current attribution policy show that provider-only resolution adds OpenAI/OpenAI-Codex hidden headers without checking `baseUrl`. The original OpenRouter billing symptom was not live-reproduced here.

## Solution Assessment

No. A provider-only header resolver is not the best fix; the safer implementation should use `provider`, `api`, `baseUrl`, capability, and transport to apply only the attribution headers valid for the resolved route.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Gate image attribution by resolved route:** `src/media-understanding/providers/image.ts:201`
  - body: This line passes headers derived only from `model.provider`, so `openai` or `openai-codex` image models using a custom OpenAI-compatible `baseUrl` will start sending hidden `originator`/`User-Agent` attribution headers to that proxy. The chat path deliberately gates those headers through provider/api/baseUrl policy; please use the route-aware request policy helper, or an OpenRouter-only media attribution path, instead of the provider-only resolver.
