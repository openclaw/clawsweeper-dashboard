## Summary

Keep open: the reported image-tool attribution gap is valid on current main, but this PR is not mergeable because it edits an obsolete path and applies provider-only attribution instead of endpoint-aware request policy.

## What This Changes

The PR imports `resolveProviderAttributionHeaders` in the old image provider file and passes the resolved provider attribution headers into that file's image `complete()` call.

## Best Possible Solution

Patch the live `src/media-understanding/image.ts` completion helper to pass endpoint-aware request headers for the final resolved image model, with focused regression coverage for native OpenRouter attribution and custom proxy non-injection.

## Reproduction Assessment

Yes. Source inspection of current main shows `describeImagesWithModelInternal` reaches the direct image `complete()` call without `headers`, and the existing image test harness already mocks `complete()` for a focused OpenRouter header assertion.

## Solution Assessment

No. The PR intent is right, but the patch targets a removed file and bypasses endpoint-aware request policy; the safer fix belongs at the current image completion call site using provider/API/base URL/capability/transport policy.
