## Summary

Keep open: the reported billing/header bug is still plausible from current source, but this PR is not mergeable as-is because it patches an obsolete image path with provider-only attribution headers, lacks real behavior proof, and can leak attribution headers to custom proxy routes.

## What This Changes

The PR adds provider-derived attribution headers to the image completion call in `src/media-understanding/providers/image.ts`.

## Best Possible Solution

Patch the current `src/media-understanding/image.ts` direct completion path to apply endpoint-aware OpenRouter request headers, with regression coverage for native/default OpenRouter routes and custom proxy routes.

## Reproduction Assessment

Yes, by source inspection rather than live billing: current main's image `complete()` call has no direct request-policy headers, and the OpenRouter wrapper is registered as a stream wrapper that this path does not prove is applied.

## Solution Assessment

No. The intent is right, but the branch resolves headers from provider identity alone and should instead use the existing endpoint-aware request policy on the live image runtime.
