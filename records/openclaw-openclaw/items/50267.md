## Summary

Keep open: the PR targets a plausible image/OpenRouter attribution gap, but the proposed provider-only header injection is too broad and there is no after-fix real behavior proof.

## What This Changes

The PR imports provider attribution in the image understanding provider and passes resolved headers into the `complete()` options for image descriptions.

## Best Possible Solution

Use the request-aware attribution policy for image/direct completion requests and add regression coverage for both OpenRouter image headers and custom OpenAI-compatible base URLs receiving no hidden OpenAI headers.

## Reproduction Assessment

No high-confidence live reproduction is present. The gap and proposed regression are source-reproducible from the image `complete()` call and attribution policy, but the PR does not include real OpenRouter request or billing proof.

## Solution Assessment

No. Passing provider-only attribution is not the best fix because the existing request policy uses route/baseUrl gates; the patch should use that same policy or a narrow OpenRouter direct-completion helper.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Gate attribution on the resolved request route:** `src/media-understanding/providers/image.ts:189`
  - body: This uses `resolveProviderAttributionHeaders(model.provider)`, so `openai` and `openai-codex` image models routed through custom OpenAI-compatible `baseUrl`s will start sending hidden `originator`/`version`/`User-Agent` headers to that proxy. The chat path uses request-aware policy with `baseUrl` gating; this path should do the same or limit the fix to verified OpenRouter routes.
  - confidence: 0.9
