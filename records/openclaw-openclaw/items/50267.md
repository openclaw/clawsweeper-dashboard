## Summary

Keep open: the reported bug still looks valid, but this PR should not merge as-is because it edits an obsolete image-provider path, lacks real behavior proof, and uses provider-only headers that bypass the current endpoint-aware attribution policy.

## What This Changes

The PR imports `resolveProviderAttributionHeaders` in `src/media-understanding/providers/image.ts` and forwards resolved headers into the image `complete()` options.

## Best Possible Solution

Update the live `src/media-understanding/image.ts` completion path to apply endpoint-aware provider request headers for native/default OpenRouter image completions, with regression tests proving headers are present for OpenRouter and absent for custom proxy routes.

## Reproduction Assessment

Yes, by source inspection rather than live billing. Current main's live image completion call lacks direct request-policy headers, and the OpenRouter provider wrapper is registered as `wrapStreamFn`, not the `createStreamFn` path used by `registerProviderStreamForModel`.

## Solution Assessment

No. The PR intent is correct, but the patch should target the current image runtime and use endpoint-aware request policy instead of resolving attribution from `model.provider` in a removed file.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Route image completions through endpoint-aware headers:** `src/media-understanding/providers/image.ts:201`
  - body: The patch resolves attribution only from `model.provider` and passes it to `complete()`, which would send attribution headers on custom/proxy base URLs if this logic is moved to the live path. Please patch current `src/media-understanding/image.ts` with the shared endpoint-aware request policy and cover native OpenRouter versus proxy routes.
  - confidence: 0.93

## Security Review

Status: needs_attention
