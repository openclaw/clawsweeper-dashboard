## Summary

Keep open: the OpenRouter image-attribution gap is still source-reproducible on current main, but this PR is not safe to merge because it applies provider-only attribution instead of the route-aware request policy and lacks real behavior proof.

## What This Changes

The PR imports `resolveProviderAttributionHeaders` in the image provider path and passes resolved headers into the image `complete()` call.

## Best Possible Solution

Apply route-aware OpenRouter attribution to image/direct-completion requests and add regression coverage that OpenRouter image requests receive attribution while custom OpenAI-compatible routes do not receive hidden OpenAI/OpenAI Codex headers.

## Reproduction Assessment

No live OpenRouter billing reproduction is present. The issue is source-reproducible by following the image `complete()` fallback, OpenRouter provider registration, and route-aware attribution policy.

## Solution Assessment

No. The proposed `resolveProviderAttributionHeaders(model.provider)` fix is not the safest maintainable solution because it bypasses the resolved request route; the image path should use the route-aware request policy or a narrow OpenRouter direct-completion helper.
