## Summary

Keep open: the PR targets a real OpenRouter image-attribution gap, but the proposed provider-only header injection would also apply hidden OpenAI/OpenAI Codex attribution headers without the request-route gating used elsewhere, and the contributor has not provided after-fix real behavior proof.

## What This Changes

The PR adds provider attribution headers to the image understanding `complete()` call by resolving headers from `model.provider` and passing them in the completion options.

## Best Possible Solution

Use the route-aware provider request policy for image/direct-completion requests and add regression coverage for OpenRouter image headers plus custom OpenAI-compatible routes receiving no hidden OpenAI/OpenAI Codex headers.

## Reproduction Assessment

No high-confidence live reproduction is present. The attribution gap and the proposed regression are source-reproducible from the image `complete()` call, custom API registration behavior, and the existing route-aware attribution policy, but there is no real OpenRouter request or billing proof.

## Solution Assessment

No. Passing headers from `resolveProviderAttributionHeaders(model.provider)` is not the best fix because it bypasses the existing baseUrl/request-route gates; a request-aware image/direct-completion path is the narrower maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Gate attribution on the resolved request route:** `src/media-understanding/providers/image.ts:189`
  - body: This resolves attribution only from `model.provider`, so `openai` and `openai-codex` image models routed through a custom OpenAI-compatible `baseUrl` can start sending hidden `originator`/`version`/`User-Agent` headers to that proxy. The existing request policy gates those headers by the resolved route, and this path should use that same policy or a narrow OpenRouter-only helper.
  - confidence: 0.9
