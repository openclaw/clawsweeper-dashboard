## Summary

Keep this PR open: the underlying OpenRouter image-tool attribution bug still exists on current main, but the submitted patch is stale and uses the wrong attribution seam. The live image path does not pass request-policy headers into the direct `complete()` calls, while current attribution policy is endpoint-aware and should not be bypassed with provider-only header injection.

## What This Changes

The PR imports `resolveProviderAttributionHeaders` in the older image provider implementation and passes its provider-only result as `headers` to the image `complete()` call.

## Best Possible Solution

Rework the fix in `src/media-understanding/image.ts` so image-understanding direct completions resolve headers with the endpoint-aware provider request policy using the final model provider, API, base URL, `capability: "image"`, and `transport: "media-understanding"`; pass those headers through both direct completion attempts and add focused tests for OpenRouter native/default attribution and custom-base-url non-injection.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists by configuring an OpenRouter image-capable model through the generic image path and inspecting the mocked `complete()` options; current main reaches `src/media-understanding/image.ts:415` without any `headers`.

## Solution Assessment

No. The PR intent is right, but the submitted implementation is not the best fix because it patches an obsolete file and bypasses the current endpoint-aware request-policy helper; the safer solution is to resolve request headers at the live image completion call site.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Use endpoint-aware request headers for image completions:** `src/media-understanding/providers/image.ts:190`
  - body: This resolves attribution from `model.provider` alone, so `openrouter`, `openai`, and `openai-codex` image models can receive attribution headers even when routed through a custom or proxy `baseUrl`. Current request policy gates those headers by provider, API, base URL, capability, and transport; use that policy before passing `headers` to `complete()` so image requests do not diverge from chat or leak attribution headers to proxy routes.
  - confidence: 0.91
