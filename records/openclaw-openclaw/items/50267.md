## Summary

Keep open: the PR targets a plausible OpenRouter image/direct-completion attribution gap, but the patch is not safe as written because it applies attribution by provider id without the endpoint gating used by current main, and it still lacks contributor real-behavior proof.

## What This Changes

The PR imports provider attribution header resolution into the image understanding path and passes resolved headers into the `complete()` call.

## Best Possible Solution

Use the endpoint-aware request policy for direct image completions, add focused regression coverage for OpenRouter native versus custom base URLs, and require redacted real OpenRouter proof before merge.

## Reproduction Assessment

Yes for the review finding: the diff calls `resolveProviderAttributionHeaders(model.provider)`, while current policy and docs require attribution to be gated by resolved endpoint/base URL. The original charging symptom was not live-reproduced in this read-only review.

## Solution Assessment

No. The safer fix is to use `resolveProviderRequestAttributionHeaders` or the current request-policy transport path with provider, API, base URL, capability, and transport context instead of provider-only header resolution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Gate attribution by endpoint before passing headers:** `src/media-understanding/providers/image.ts:189`
  - body: This resolves attribution from `model.provider` alone, so an OpenRouter model repointed at a custom `baseUrl` still receives OpenRouter attribution headers, and OpenAI/OpenAI-Codex image models can receive hidden attribution headers even on non-native proxies. Current policy and docs gate these headers by resolved endpoint; pass the model API/base URL through the request attribution policy instead.
