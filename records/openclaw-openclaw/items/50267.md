## Summary

Keep open: this targets a plausible OpenRouter image-tool attribution bug, but the branch is stale and the current diff is too broad because it keys attribution only on `model.provider` instead of the resolved endpoint policy. The PR also still lacks after-fix real behavior proof.

## What This Changes

The PR imports `resolveProviderAttributionHeaders` and conditionally passes provider attribution headers into the image media-understanding `complete()` call.

## Best Possible Solution

Move the fix onto current main with an endpoint-aware direct-completion helper or `resolveProviderRequestPolicyConfig` path, then cover OpenRouter image requests and OpenAI proxy non-attribution in focused tests.

## Reproduction Assessment

No high-confidence live reproduction is present. Source inspection shows the direct image `complete()` path and the PR's intended header gap, but the PR does not include real OpenRouter output proving the before/after behavior.

## Solution Assessment

No. The intended direction is right, but this implementation should use the endpoint-aware attribution policy instead of `resolveProviderAttributionHeaders(model.provider)`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Gate image attribution headers on the resolved endpoint:** `src/media-understanding/providers/image.ts:201`
  - body: This adds attribution headers based only on `model.provider`. For an image model configured as `openai` or `openai-codex` with a custom OpenAI-compatible `baseUrl`, the image path would start sending hidden `originator`/`User-Agent` headers to that proxy, while the chat path gates those headers to verified native OpenAI/Codex hosts. Use the endpoint-aware provider request policy here so only verified OpenRouter/native routes receive attribution.
  - confidence: 0.86
