## Summary

Keep open: the PR targets a real attribution gap, but this branch should not merge as-is because it adds provider-only headers in an obsolete image path and can leak attribution headers to custom proxy routes. It also lacks after-fix real behavior proof.

## What This Changes

The PR imports `resolveProviderAttributionHeaders` in `src/media-understanding/providers/image.ts` and forwards resolved headers into image completion options.

## Best Possible Solution

Apply endpoint-aware provider request headers in the current `src/media-understanding/image.ts` direct-completion path, with regression coverage for native/default OpenRouter image requests and custom proxy routes.

## Reproduction Assessment

Yes, by source inspection rather than live billing. Current main’s image completion path reaches `complete()` without direct headers, and the OpenRouter wrapper registered in the plugin is not the `createStreamFn` path used by `registerProviderStreamForModel`.

## Solution Assessment

No. The intent is right, but the patch should target the current `src/media-understanding/image.ts` runtime and use endpoint-aware request policy instead of resolving headers from provider identity alone.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Route image attribution through endpoint policy:** `src/media-understanding/providers/image.ts:190-201`
  - body: This resolves attribution from `model.provider` and passes it directly to `complete()`, so OpenAI/OpenAI-Codex and OpenRouter models on custom `baseUrl` routes can receive headers that the endpoint-aware policy intentionally withholds. Please apply the current request-policy helper in the live image runtime so native/default OpenRouter gets attribution while proxy routes do not.
  - confidence: 0.93
