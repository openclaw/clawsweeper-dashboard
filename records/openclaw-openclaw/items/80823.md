## What This Changes

The PR changes CLI config set, patch, and unset success text to derive gateway restart hints from reload planning and adds CLI/gateway regression tests.

## Best Possible Solution

Plan CLI apply hints from the same before/after changed-path set the gateway reload machinery uses, while keeping conservative handling for plugin-entry paths where metadata is unavailable.

## Reproduction Assessment

Yes from source inspection: broad config replacements can request a hot root path while changing nested restart-owned fields. I did not run the command because this review was constrained to read-only inspection.

## Solution Assessment

No: delegating to buildGatewayReloadPlan() is the right direction, but using requested operation paths is not the same contract as the gateway reload path uses. The safer fix is to derive hints from actual before/after changed paths and then apply the conservative plugin-entry guard where needed.
