## Summary

Keep this PR open: the core fix is useful, but the patch still has a compatibility blocker because the CLI success hint can trust `buildGatewayReloadPlan()` without the plugin reload registrations that make plugin-owned restart prefixes conservative.

## What This Changes

The PR changes `config set`, `config patch`, and `config unset` success text to derive restart/no-restart hints from the gateway reload planner and adds CLI regression tests for agent, model, gateway, and mixed config writes.

## Best Possible Solution

Keep the agent/model hot hint, but make plugin-owned config hints use real plugin reload metadata when available and fall back to restart-required for plugin entry paths when the CLI has not loaded that metadata.

## Reproduction Assessment

Yes. Current main source shows `runConfigOperations()` and `runConfigUnset()` always print `Restart the gateway to apply.`, while gateway reload planning already marks agent/model paths as hot.

## Solution Assessment

No. Reusing the shared planner is the right direction for core paths, but this implementation needs plugin reload metadata or a conservative plugin-path fallback before it is the safest fix.

  - body: `config set`/`unset` normally run with plugin loading disabled, but this helper now trusts `buildGatewayReloadPlan()` for the user-facing hint.
