## Summary

Keep open: the latest head fixes set/patch changed-path planning, but direct `config unset` still plans the apply hint from only the requested root path, so broad unsets can under-warn operators when removed nested config requires a gateway restart.

## What This Changes

This PR updates CLI config set, patch, and unset success hints to use gateway reload planning and adds CLI/gateway tests for hot versus restart-required config paths.

## Best Possible Solution

Have every CLI post-write hint, including direct `config unset`, derive from actual before/after changed paths; leave the broader machine-readable apply-status contract to https://github.com/openclaw/openclaw/issues/83851.

## Reproduction Assessment

Yes. Source inspection at PR head shows direct `runConfigUnset` still plans from the requested root path while current reload rules require nested actual paths to distinguish hot roots from restart-owned children; I did not run commands because this review is read-only.

## Solution Assessment

No. Reusing the gateway reload planner is the right direction, but direct `config unset` needs the same actual before/after diff planning already added for set and patch.
