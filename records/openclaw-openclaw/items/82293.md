## Summary

Keep open. Current main still routes `/stop` and `/restart` through same-process AbortController, backend-cancel, timer, and SIGUSR1 paths, and a focused Node 24 probe confirms those callbacks wait behind synchronous event-loop work; the linked event-loop reports are adjacent symptoms, not a duplicate owner for this exact interrupt problem.

## What This Changes

The issue asks for `/stop` and `/restart` to interrupt an active agent promptly even when agent/model work is starving the Node event loop.

## Best Possible Solution

Keep the existing command contract, but move interrupt authority outside the starved loop or add an owner-approved watchdog/kill path with blocked-loop regression coverage.

## Reproduction Assessment

No, not as a live OpenClaw reproduction. Source inspection plus Node 24 probes make the failure mode credible, but the reported gateway/model delay still needs an end-to-end reproduction.

## Solution Assessment

Unclear. Worker isolation, a watchdog, and async hot-path cleanup are plausible, but the safest fix needs owner judgment on process isolation versus forced termination before implementation.
