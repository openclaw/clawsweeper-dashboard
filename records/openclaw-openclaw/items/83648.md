## Summary

Keep open: current main and v2026.6.8 still send the cron local-provider preflight probe without auth, but this PR is not merge-ready because it adds a parallel credential resolver that bypasses scoped provider auth and request-header handling.

## What This Changes

The PR adds API-key resolution and an optional Bearer Authorization header to cron local-provider preflight probes, with tests for literal config keys and authless providers.

## Best Possible Solution

Route cron preflight through the same scoped provider auth and request-header contract used by actual model requests, passing the selected cron agent context while preserving authless local providers.

Yes. Current main and v2026.6.8 send the local OpenAI-compatible preflight GET without headers, and the PR body includes live LiteLLM logs showing 401 before and 200 after the patch.

## Solution Assessment

No. Adding auth to the probe is the right direction, but this implementation is not the best fix because it hand-rolls credential and header resolution instead of reusing the scoped provider auth/request contract used by normal model calls.

- **[P1] Resolve provider keys through existing auth helpers:** `src/cron/isolated-agent/model-preflight.runtime.ts:170-172`
  - body: `apiKey` is a `SecretInput`, but this branch returns any non-empty string directly and later sends it as `Authorization: Bearer ...`.   - confidence: 0.94

  - body: This reads `resolveDefaultAgentDir(cfg)/auth-profiles.json` inside preflight, but the cron caller does not pass the selected agentDir/profile/workspace context. A non-default cron agent can therefore use the wrong default-agent credential or miss its own profile credential, unlike the actual model run.
  - confidence: 0.93
- **[P1] Reuse provider request auth and header resolution:** `src/cron/isolated-agent/model-preflight.runtime.ts:209-214`
  - body: The probe now only injects `Authorization: Bearer <apiKey>`, dropping configured `headers`, `authHeader`, and `request.auth`/`request.headers` behavior that normal provider requests support. Self-hosted endpoints requiring custom auth or tenant headers can still be marked unavailable even though the real model request would work.
  - confidence: 0.9
