## Summary

## What This Changes

## Best Possible Solution

Route cron preflight through the same scoped provider auth and request-header contract used by the actual model run, while keeping authless local providers unchanged.

## Solution Assessment

No. Adding auth to the probe is the right direction, but this implementation is not the best fix because it hand-rolls credential resolution instead of reusing the scoped provider auth/request contract used by normal model calls.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Resolve provider keys through existing auth helpers:** `src/cron/isolated-agent/model-preflight.runtime.ts:170-172`
  - body: `apiKey` is a `SecretInput`, but this branch returns any non-empty string directly and later sends it as `Authorization: Bearer ...`. Env templates, known marker values, or profile ids can therefore be sent as bearer text, while structured SecretRefs are skipped, so preflight still does not match normal provider auth.
  - confidence: 0.94
- **[P1] Keep preflight credentials scoped to the cron agent:** `src/cron/isolated-agent/model-preflight.runtime.ts:177-180`
  - body: This reads `resolveDefaultAgentDir(cfg)/auth-profiles.json` inside preflight, but the cron caller does not pass the selected agentDir/profile/workspace context. A non-default cron agent can therefore use the wrong default-agent credential or miss its own profile credential, unlike the actual model run.
  - confidence: 0.93
