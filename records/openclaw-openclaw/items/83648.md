## Summary

Keep this PR open: the bearer-auth preflight direction is useful and the supplied LiteLLM proof is real, but the current patch conflicts with OpenClaw's existing custom-header auth contract by adding a resolved Authorization bearer header when `request.auth.mode: "header"` is configured.

## What This Changes

The PR adds credential-aware cron provider preflight requests, error redaction, agent/workspace directory forwarding, and expanded tests for bearer and custom-header auth.

## Best Possible Solution

Keep the bearer preflight fix, but make header-mode preflight follow the existing provider request contract: custom-header auth should replace Authorization, not add a second credential header.

## Reproduction Assessment

Yes: current main's preflight fetch has no auth headers, and the linked LiteLLM proof shows the before/after `GET /models` behavior. I did not run a live LiteLLM repro locally in this read-only review.

## Solution Assessment

No: adding auth to the preflight is the right layer, but the current implementation is not the best fix until it reuses or matches the existing provider request auth semantics for custom-header mode.
