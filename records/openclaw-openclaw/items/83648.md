## Summary

Keep open: the reported cron preflight auth gap is real and the PR now has useful live LiteLLM proof, but the branch still hand-rolls credential resolution in ways that can send marker strings, miss supported auth-profile forms, and use the default agent's credential scope for non-default cron agents.

## What This Changes

The PR adds API-key lookup to cron local-provider preflight and sends a Bearer Authorization header on the GET probe when a key is found.

## Best Possible Solution

Thread the selected cron agentDir/profile context into preflight, reuse or extract the existing provider auth resolution path, suppress non-secret/no-auth markers before adding headers, remove the committed proof artifact, and cover literal, SecretRef, auth-profile, non-default-agent, and authless cases with focused tests.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main and v2026.5.18 send the cron local-provider preflight GET without headers, and the PR body includes live LiteLLM before/after logs for the reported 401 noise. I did not run a live LiteLLM instance in this read-only review.

## Solution Assessment

No, not as implemented. Adding auth to the probe is the right direction, but the maintainable fix should reuse scoped provider auth resolution instead of adding a separate raw config/auth-profiles parser.

Full review comments:

- **[P1] Resolve config SecretRefs before adding Authorization:** `src/cron/isolated-agent/model-preflight.runtime.ts:168`
  - body: `apiKey` is a `SecretInput`, and the LiteLLM/vLLM docs use values like `${LITELLM_API_KEY}`.   - body: `resolveDefaultAgentDir(cfg)` ignores the agentDir already resolved for the cron job. A non-default cron agent can therefore probe with the default agent's credentials, or with no credential, while the actual model run uses a different scoped auth store.
  - confidence: 0.9

  - body: Auth profiles support `keyRef`, `token`, `tokenRef`, OAuth refresh, profile ordering, and compatibility checks, but this loop only casts profiles to `{ provider, key }` and returns the first `key`. Reuse the existing provider/profile resolver so preflight stays in sync with normal inference auth.
