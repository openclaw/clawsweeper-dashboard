## Summary

Keep open; the cron preflight bug is plausible, but the current PR should not merge because it treats non-secret local auth markers as bearer credentials and introduces an unused parameter that conflicts with the core typecheck settings. The PR also needs stronger after-fix real behavior proof before merge.

## What This Changes

The PR changes cron isolated-agent model preflight to resolve provider auth and include an Authorization bearer header on local provider probe requests.

## Best Possible Solution

Resolve cron preflight auth through the shared provider auth path, but only send Authorization for real credentials, keep marker-backed local providers headerless, and add focused regression coverage plus real proof.

## Reproduction Assessment

Yes at source level: current main probes local providers with GET and no headers, and the PR targets that exact call path. I did not establish a live LiteLLM/vLLM reproduction in this read-only review.

## Solution Assessment

No; using shared auth resolution is the right direction, but this patch needs to reject non-secret marker values and remove the unused parameter before it is the maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Skip non-secret auth markers before adding Authorization:** `src/cron/isolated-agent/model-preflight.runtime.ts:170-172`
  - body: `resolveApiKeyForProvider()` can return local placeholder values such as `custom-local` or `ollama-local`, and normal model requests call `applyLocalNoAuthHeaderOverride()` so those markers do not become real Authorization headers. This helper treats every non-empty `apiKey` as a bearer token, so marker-backed authless local providers can now be probed with `Authorization: Bearer custom-local` instead of staying headerless. Please filter `isNonSecretApiKeyMarker()` values, or reuse the existing header-override semantics, before constructing the probe header.
  - confidence: 0.9
- **[P2] Remove the unused providerCfg parameter:** `src/cron/isolated-agent/model-preflight.runtime.ts:156`
  - body: The new `providerCfg` argument is never read, while `tsconfig.core.json` enables `noUnusedParameters`. As written, the core typecheck should fail on this added helper; remove the parameter or use it for the intended marker/header decision.
  - confidence: 0.92
