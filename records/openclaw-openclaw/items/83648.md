## Summary

Keep open: the PR targets a real cron/provider-auth nuisance and includes usable live LiteLLM log proof, but the branch hand-rolls credential discovery and can send marker strings or the wrong agent's credential.

## What This Changes

The PR adds API-key lookup to cron local-provider preflight and sends a Bearer Authorization header on the GET probe when a key is found.

## Best Possible Solution

Route cron preflight through the same scoped, request-ready provider auth and transport contract used for the actual model run, with regression coverage for literal keys, SecretRefs, auth profiles, agentDir scoping, and authless local providers.

## Reproduction Assessment

Yes, at source level: current main sends the local provider preflight GET without headers, and the PR body includes live LiteLLM before/after logs for the reported 401 noise. I did not run a live LiteLLM/vLLM instance in this read-only review.

## Solution Assessment

No. Adding auth to the probe is the right direction, but the proposed hand-rolled resolver is not the maintainable fix because it diverges from the scoped provider auth contract used by normal inference.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Resolve SecretInput before adding the bearer header:** `src/cron/isolated-agent/model-preflight.runtime.ts:166-168`
  - body: `apiKey` is a `SecretInput`, but this returns any non-empty string directly and later sends it as `Authorization: Bearer ...`. Documented configs such as `${LITELLM_API_KEY}` or normalized env-marker values can therefore send the marker name instead of the secret, so authenticated preflight still fails and the probe no longer matches normal inference auth.
  - confidence: 0.94
- **[P1] Keep preflight auth scoped to the cron agent:** `src/cron/isolated-agent/model-preflight.runtime.ts:173-176`
  - body: This helper reads `resolveDefaultAgentDir(cfg)/auth-profiles.json`, while cron preflight is called without the selected agentDir/profile. A cron run for a non-default agent can therefore use the wrong default-agent credential, miss the real agent's keyRef/tokenRef/OAuth credential, or send a credential to a local endpoint that the actual run would not use.
  - confidence: 0.93

## Security Review

Status: needs_attention

Summary: The patch introduces a new credential-sending path that bypasses the existing scoped auth resolver and can send the wrong credential or marker values to local/private endpoints.

- **[medium] Default-agent credential can be sent for another cron agent:** `src/cron/isolated-agent/model-preflight.runtime.ts:173`
