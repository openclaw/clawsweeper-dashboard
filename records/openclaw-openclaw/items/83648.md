## Summary

Keep open: current main and the latest release still send the cron local-provider preflight without auth, but this PR is not merge-ready because it hand-rolls provider credential resolution and can send placeholder or wrong-agent credentials to local/private provider endpoints.

## What This Changes

The branch adds a cron model-preflight API-key resolver, sends an optional Bearer header on local provider probes, and adds mocked tests for literal-key and authless cases.

## Best Possible Solution

Route cron preflight through the same scoped provider auth and request-header contract used by the selected cron model run, while preserving authless local providers.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.8 send the local OpenAI-compatible preflight GET with no headers, and the PR body includes live LiteLLM logs showing 401 before and 200 after. I did not run a live LiteLLM instance in this read-only review.

## Solution Assessment

No. Adding auth to the probe is the right direction, but this implementation is not the best fix because it duplicates credential parsing instead of reusing the scoped provider auth/request contract used by normal model calls.

  - body: `apiKey` is a `SecretInput`, but this returns any non-empty string directly and later sends it as `Authorization: Bearer ...`.
