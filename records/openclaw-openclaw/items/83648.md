## Summary

Keep open: the PR moves cron provider preflight in the right direction and has useful LiteLLM screenshot proof for the bearer-header path, but latest head still omits provider-level headers from the shared resolver call, so configured local proxies can still fail preflight while normal model requests succeed.

## What This Changes

The PR changes cron model-provider preflight to build probe headers through shared provider request helpers, pass agent/workspace dirs to credential lookup, redact preflight errors, and add auth parity tests.

## Best Possible Solution

Keep the shared request-auth delegation, but route preflight through the same provider-header credential contract as normal model requests or explicitly document and test any excluded header surfaces before merge.

## Reproduction Assessment

Yes, at source level: current main sends local cron provider probes as bare GET requests, and the PR proof screenshot shows the reported LiteLLM GET /models 401 changing to 200 after adding auth. I did not run a local provider in this read-only review.

## Solution Assessment

No, not yet. Delegating auth construction to the shared resolver is the right direction, but the current call is still too narrow because it omits provider-level headers that normal model execution can use.
