- `P2`: This is a normal-priority local cron/provider auth fix with limited blast radius, not a core outage.
- `merge-risk: 🚨 compatibility`: The PR can add bearer headers to authless or marker-backed local provider setups that currently run without such headers.

## Summary

Keep open: the missing preflight auth header is source-visible on current main, but this PR is not merge-ready because it hand-rolls credential resolution and can send env/SecretRef markers or default-agent profile credentials instead of the actual cron runtime credential.

## What This Changes

The PR adds API-key lookup to cron local-provider preflight and sends an Authorization bearer header on the GET /models or /api/tags probe when a key is found.

## Best Possible Solution

Resolve preflight auth through the existing scoped provider-auth contract, add Authorization only for request-ready real credentials, preserve headerless authless local providers, and cover the key auth-source cases with regression tests.

## Reproduction Assessment

Yes, at source level: current main sends the local provider preflight GET without headers, and the PR body includes LiteLLM before/after proxy logs for the symptom. I did not run a live LiteLLM/vLLM instance in this read-only review.

## Solution Assessment

No. Adding auth to the probe is the right direction, but this implementation should use the existing scoped auth contract and marker filtering instead of manually reading config, env, and auth-profiles.json.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Resolve config markers before adding Authorization:** `src/cron/isolated-agent/model-preflight.runtime.ts:166-168`
  - body: This returns `providerCfg.apiKey` directly, but documented provider config commonly stores `${LITELLM_API_KEY}` or its normalized env-var marker there. Sending that marker as `Bearer LITELLM_API_KEY` will not authenticate the probe and can keep the reported 401 noise for the common env-backed setup; resolve a request-ready key or skip the header instead.
  - confidence: 0.92
- **[P1] Use the scoped auth profile resolver:** `src/cron/isolated-agent/model-preflight.runtime.ts:173-183`
  - body: The helper reads `resolveDefaultAgentDir(cfg)` and scans only `profile.key`, so a non-default cron agent or token/keyRef/OAuth profile can probe with the wrong credential or no credential while the actual model run uses a different auth context. Thread the cron run's scoped auth context into preflight and reuse the existing resolver/helper rather than parsing auth-profiles.json here.
  - confidence: 0.88
