## Summary

Keep open: the reported cron preflight bug is real and the proof is strong, but this patch is not merge-ready because it duplicates provider-auth resolution and can use default-agent credentials during non-default cron runs.

## What This Changes

The PR adds a cron preflight API-key lookup, sends an Authorization header on local provider probes, and adds a root proof JPEG.

## Best Possible Solution

Use the selected cron agent context and shared provider auth/request contracts for the preflight header, add focused regression coverage, and keep proof artifacts in the PR rather than the repository.

## Reproduction Assessment

Yes at source/proof level: current main sends the local-provider preflight GET without headers, and the PR proof shows LiteLLM returning 401 before and 200 after. I did not run a live LiteLLM instance in this read-only review.

## Solution Assessment

No: adding a Bearer header is directionally right, but the implementation should reuse scoped provider auth/request resolution instead of parsing credentials separately.
