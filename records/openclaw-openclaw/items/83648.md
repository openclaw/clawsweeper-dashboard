## Summary

Keep open: the reported cron preflight auth bug is real and the live proof is useful, but this patch is not merge-ready because it adds a second credential/header resolver that bypasses existing scoped provider auth and request-header contracts.

## What This Changes

The PR adds a cron preflight API-key resolver, injects an Authorization Bearer header into local provider GET probes, and adds tests for literal config keys and authless providers.

## Best Possible Solution

Route cron preflight through the same scoped provider auth and request-header contract used by normal model calls, passing the selected cron agent context while preserving authless local providers.

## Solution Assessment

No. Adding auth to the probe is the right direction, but this implementation is not the best fix because it hand-rolls credential and header resolution instead of reusing the scoped provider auth/request contract used by normal model calls.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Resolve provider keys through existing auth helpers:** `src/cron/isolated-agent/model-preflight.runtime.ts:170-172`
  - body: `apiKey` is a `SecretInput`, but this branch returns any non-empty string directly and later sends it as `Authorization: Bearer ...`. Env-template strings, legacy markers, and profile ids can therefore be sent as bearer text, while structured SecretRefs are skipped, so preflight still does not match normal provider auth.
  - confidence: 0.94
- **[P1] Pass cron agent scope into preflight auth:** `src/cron/isolated-agent/model-preflight.runtime.ts:177-180`
  - body: This reads `resolveDefaultAgentDir(cfg)/auth-profiles.json` inside preflight, but the cron caller does not pass the selected agentDir/profile/workspace context. A non-default cron agent can therefore use the wrong default-agent credential or miss its own profile credential, unlike the actual model run.
  - confidence: 0.93
- **[P1] Reuse provider request auth and headers:** `src/cron/isolated-agent/model-preflight.runtime.ts:209-214`
  - body: The probe now only injects `Authorization: Bearer <apiKey>`, dropping configured `headers`, `authHeader`, and `request.auth`/`request.headers` behavior that normal provider requests support. Self-hosted endpoints requiring custom auth or tenant headers can still be marked unavailable even though the real model request would work.
  - confidence: 0.9
