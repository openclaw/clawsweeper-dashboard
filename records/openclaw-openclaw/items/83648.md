## Summary

Keep open: the target is plausible, but this patch is not merge-ready because it duplicates provider auth resolution, sends supported config marker strings as bearer tokens, misses supported auth-profile credential shapes, and still lacks inspectable real behavior proof.

## What This Changes

The PR modifies cron isolated-agent model preflight to resolve an API key and include a bearer Authorization header on local provider probe requests.

## Best Possible Solution

Route preflight through the shared provider auth path or a small request-ready helper, filter non-secret markers before adding Authorization, and cover literal keys, env markers, SecretRefs, auth profiles, and authless local providers with focused tests plus real proof.

## Reproduction Assessment

Yes at source level: current main sends the cron local-provider probe with no headers, and the PR targets that exact call path. I did not establish a live LiteLLM/vLLM reproduction in this read-only review.

## Solution Assessment

No: the current patch is not the best fix because it creates a partial second auth resolver instead of using the existing provider auth contract and marker filtering. A maintainable fix should share the runtime auth path and add regression coverage for real-key, marker-backed, SecretRef/profile, and authless cases.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Resolve config markers before sending bearer auth:** `src/cron/isolated-agent/model-preflight.runtime.ts:166-168`
  - body: This returns any string from `models.providers.<id>.apiKey` verbatim, so generated configs that store `VLLM_API_KEY` or other known env/local markers will send `Authorization: Bearer VLLM_API_KEY` instead of the real env value or no header. Normal model auth resolves or filters those markers, so authenticated preflight can still 401 while inference succeeds.
  - confidence: 0.92
- **[P2] Reuse profile resolution instead of parsing only key:** `src/cron/isolated-agent/model-preflight.runtime.ts:180-183`
  - body: The manual auth-profile scan only accepts `profile.key`; it ignores token profiles, OAuth access tokens, and `keyRef`/`tokenRef` secrets that the existing resolver supports. Those supported credential shapes will continue probing without auth even though normal inference has credentials.
  - confidence: 0.9
