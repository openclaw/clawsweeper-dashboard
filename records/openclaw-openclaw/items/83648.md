## Summary

Keep open: the cron preflight auth gap remains on current main and the latest release, but this PR is not merge-ready because it duplicates provider credential/header resolution and can use or send the wrong auth material.

## What This Changes

The PR adds a cron local-provider preflight API-key resolver, sends a Bearer Authorization header on GET probes, and adds two runtime preflight tests.

## Best Possible Solution

Route cron preflight through the same scoped provider auth and request-header contract used by normal model calls, passing the selected cron agent context while preserving authless local providers.

## Reproduction Assessment

Yes. Current main and v2026.6.9 send the local OpenAI-compatible preflight GET without headers, and the PR body includes live LiteLLM logs showing 401 before and 200 after the patch; I did not run a live LiteLLM server in this read-only review.

## Solution Assessment

No. Adding auth to the probe is the right direction, but this implementation is not the best fix because it hand-rolls credential and header resolution instead of reusing the scoped provider auth/request contract used by normal model calls.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Resolve provider keys through existing auth helpers:** `src/cron/isolated-agent/model-preflight.runtime.ts:170-172`
  - body: `apiKey` is a `SecretInput`, but this branch returns any non-empty string directly and later sends it as `Authorization: Bearer ...`. Env-template strings, legacy markers, and profile ids can therefore be sent as bearer text, while structured SecretRefs are skipped, so preflight still does not match normal provider auth.
  - confidence: 0.94
- **[P1] Pass cron agent scope into preflight auth:** `src/cron/isolated-agent/model-preflight.runtime.ts:177-180`
  - body: This reads `resolveDefaultAgentDir(cfg)/auth-profiles.json` inside preflight, but the cron caller has a selected agentDir/workspace/profile context that is not passed into preflight. A non-default cron agent can therefore use the wrong default-agent credential or miss its own profile credential, unlike the actual model run.
  - confidence: 0.93
- **[P1] Reuse provider request auth and headers:** `src/cron/isolated-agent/model-preflight.runtime.ts:209-214`
  - body: The probe now only injects `Authorization: Bearer <apiKey>`, dropping configured `headers`, `authHeader`, and `request.auth`/`request.headers` behavior that normal provider requests support. Self-hosted endpoints requiring custom auth or tenant headers can still be probed differently from the real model request.
  - confidence: 0.9
