## Summary

Keep open: the live proof supports the reported LiteLLM preflight log-noise bug, but the latest patch still duplicates provider auth resolution and can send credentials from the wrong agent scope instead of matching the credentials actual inference uses.

## What This Changes

This PR adds a cron local-provider preflight API-key resolver, sends a Bearer header on the probe, and commits a root screenshot proof artifact.

## Best Possible Solution

Pass the selected cron agent/profile context into preflight and reuse or extract the existing provider auth and request-header resolution so probe headers match the actual model request, with focused regression tests and proof kept out of the source tree.

## Reproduction Assessment

Yes, at source/proof level: current main sends the local provider preflight GET without headers, and the PR includes live LiteLLM logs and a screenshot showing 401 before and 200 after. I did not run a live LiteLLM instance in this read-only review.

## Solution Assessment

No. The best fix should reuse the existing scoped provider auth/request contract instead of adding a hand-rolled resolver that diverges from normal inference behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Use scoped provider auth for the probe:** `src/cron/isolated-agent/model-preflight.runtime.ts:173-183`
  - body: `resolveProbeApiKey()` reads `auth-profiles.json` from `resolveDefaultAgentDir(cfg)` and returns the first matching raw key. Cron has already resolved the selected `agentDir` before preflight, and normal inference uses that scoped agent/profile context, so a non-default cron agent can preflight with default-agent credentials and send the wrong token to the local provider. Pass the scoped cron agent/profile context into preflight and reuse the existing provider auth resolver before setting any Authorization header.
  - confidence: 0.91
- **[P2] Honor configured request auth for preflight:** `src/cron/isolated-agent/model-preflight.runtime.ts:202-211`
  - body: The probe now only builds `Authorization: Bearer ${apiKey}`, but model provider configs also support `authHeader` and `request.auth`/custom header overrides, with SecretRef resolution handled before `resolveProviderRequestConfig()` applies headers. Providers configured with those supported request-auth paths will still receive an unauthenticated `/models` probe, leaving preflight auth out of sync with actual model requests.
  - confidence: 0.84
- **[P3] Remove the committed proof image:** `evidence-preflight-fix.jpg:1`
  - body: `evidence-preflight-fix.jpg` is review evidence, not source. The PR body already links the proof, so committing the binary image at the repository root adds unrelated package/source churn that should be removed before merge.
