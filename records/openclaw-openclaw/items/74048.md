## Summary

Keep open: the linked ACP failure is still a valid user problem, but this PR is not mergeable because it changes the documented missing-`configOptionKeys` contract and can silently ignore requested runtime controls such as `approval_policy`. The newer affected-user report tied to #76564 supports the same root issue, so this needs a revised fix rather than cleanup closure.

## What This Changes

This PR changes ACP manager runtime-control application to send session config options only when a backend reports a non-empty `configOptionKeys` list, filtering out non-advertised options.

## Best Possible Solution

Revise or replace this PR with a capability-aware fix that preserves the core undefined-key contract, handles strict Claude ACP option support in the owner ACPX/adapter boundary, keeps unsupported permission controls visible, and adds focused tests plus a changelog entry.

## Reproduction Assessment

Yes for the PR regression from source inspection: the current manager fixture omits `configOptionKeys` and expects persisted runtime options to be sent, while this patch skips them all. The original ACP failure also has concrete live reproduction steps in #73875 and a matching affected-user report in #76564, but I did not run live ACP.

## Solution Assessment

No. The patch addresses the strict-backend symptom by redefining missing `configOptionKeys` and silently dropping unsupported controls; the safer path is a capability-aware ACPX/adapter fix that preserves the existing core contract and error visibility.
