## Summary

Keep this PR open, but do not merge the current patch as-is. The linked Claude ACP bug appears real and the PR touches the right runtime-control path, but the patch conflicts with the current `configOptionKeys` contract and silently hides unsupported advertised-key runtime controls, including permission-related controls.

## What This Changes

The PR changes ACP runtime-control setup so persisted config options are sent only when the backend advertises matching `configOptionKeys`, and skipped entirely when no key list is advertised.

## Best Possible Solution

Land a tested ACP runtime-control fix that prevents generic OpenClaw config IDs from breaking strict adapters like Claude ACP, while preserving clear failure or explicit warning when a backend advertises a strict key list and a requested runtime option is unsupported. The runtime capability contract should be clarified in types/tests, and any adapter-specific key mapping should live in the ACP/acpx boundary rather than silently changing all backends.

## Reproduction Assessment

Yes. A focused reproduction is a runtime that supports `session/set_config_option`, returns no `configOptionKeys`, and has persisted runtime options for `model`, `thinking`, `approval_policy`, or `timeout`; current main will call `setConfigOption` for each. The linked Claude ACP issue adds live adapter steps, though this read-only review did not run live ACP.

## Solution Assessment

No. The PR addresses the right failure area, but it is not the best fix because it changes the advertised-key path from explicit rejection to silent filtering and contradicts the documented undefined-key contract. A safer fix should decide the missing-key semantics explicitly and keep unsupported strict-list controls visible to callers.
