## Summary

Keep this PR open: the linked Claude ACP failure appears real and is supported by #73875 and the affected-user confirmation in #76564, but this patch changes the ACP capability contract and can silently drop persisted runtime controls, including permission policy, without tests or changelog coverage.

## What This Changes

This PR changes ACP runtime-control setup to only send `session/set_config_option` calls for advertised `configOptionKeys`, and to send none when no keys are advertised.

## Best Possible Solution

Revise the PR to preserve the existing missing-`configOptionKeys` permissive contract and unsupported-key errors while handling strict Claude/acpx keys explicitly, with focused ACP/acpx regression tests and a changelog entry.

## Reproduction Assessment

Yes for the PR regression by source inspection: the existing no-`configOptionKeys` manager fixture would stop receiving all persisted config-option calls under this patch. The original Claude failure is also concretely described in #73875 and confirmed by #76564, but I did not run a live Claude ACP session.

## Solution Assessment

No. The patch fixes the strict-backend symptom by redefining absent `configOptionKeys` as deny-all and silently filtering unsupported advertised keys; the safer fix is to keep the current contract and handle strict adapter mapping/filtering in the ACPX path.
