## Summary

Keep open: this PR targets the real ACP/Claude failure reported in #73875 and echoed by #76564, but the patch changes the existing ACP capability contract and can silently ignore runtime controls such as approval policy.

## What This Changes

The PR changes ACP runtime-control setup to send `session/set_config_option` only for advertised keys and to send none when the backend advertises no `configOptionKeys`.

## Best Possible Solution

Revise the fix to preserve the missing-`configOptionKeys` permissive contract, add a strict supported-key path for Claude/ACPX, keep unsupported controls visible, and include focused regression tests plus a changelog entry.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: the PR would stop the existing no-`configOptionKeys` ACP manager fixture from receiving persisted config options, while the pinned Claude ACP adapter source confirms unknown config IDs are rejected.

## Solution Assessment

No. The patch fixes the strict-backend symptom by redefining missing `configOptionKeys` and silently filtering unsupported keys; the narrower maintainable fix is capability-aware ACPX/Claude handling that preserves existing contract semantics.
