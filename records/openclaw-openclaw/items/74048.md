## Summary

Keep this PR open. It targets a real ACP runtime-control failure, but the patch is not merge-ready because it breaks the current no-`configOptionKeys` contract used by ACPX-backed controls and also changes strict advertised-key mismatches from explicit errors into silent drops for controls such as `approval_policy`.

## What This Changes

The PR modifies `src/acp/control-plane/manager.runtime-controls.ts` so persisted ACP config options are sent only when `configOptionKeys` contains the key, and no config options are sent when the backend reports no key list.

## Best Possible Solution

Decide and document the ACP `configOptionKeys` contract, then land a focused fix that prevents generic OpenClaw keys from reaching strict Claude ACP while preserving visible failures or warnings for unsupported requested controls. A maintainable path likely needs ACPX to expose the adapter's actual config option IDs or a narrow OpenClaw-side adapter mapping before manager-level filtering is applied.

## Reproduction Assessment

Yes. A high-confidence reproduction exists through the current manager path: use a runtime that supports `session/set_config_option` but omits `configOptionKeys`, plus persisted `model`, `thinking`, `approval_policy`, or `timeout`; current main calls `setConfigOption` for each, while this patch would call none. The paired Claude ACP issue adds live adapter evidence, but no live ACP run was performed in this read-only review.

## Solution Assessment

No. The PR changes the right function, but it is not the best fix because it contradicts the current omitted-key contract, would break existing no-key ACPX runtime-option behavior, and removes caller visibility for strict advertised-key mismatches.
