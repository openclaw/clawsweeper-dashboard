## Summary

Keep open: this PR addresses a real ACP/Claude failure tracked by #73875 and echoed by the affected user in #76564, but the current diff is not safe because it changes the optional `configOptionKeys` contract and can silently drop requested runtime controls such as `approval_policy`.

## What This Changes

The PR changes ACP runtime-control setup so `session/set_config_option` is sent only for keys in `configOptionKeys`, and no config options are sent when no keys are advertised.

## Best Possible Solution

Preserve the existing optional-key contract, add a capability-aware path for strict adapters such as Claude, keep unsupported requested controls visible, and cover the behavior with focused ACP manager/acpx tests plus a changelog entry.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path for the PR regression: the existing no-`configOptionKeys` manager fixture would stop receiving persisted config-option calls under the patch; the original Claude failure is concretely described in #73875 and #76564.

## Solution Assessment

No. The patch fixes the strict-backend symptom by redefining missing `configOptionKeys` and silently filtering unsupported advertised-key options; a safer fix should preserve the existing contract and introduce an explicit strict/advertised capability path.
