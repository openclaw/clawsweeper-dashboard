## Summary

Keep open: the linked Claude ACP bug is real, but this PR is not mergeable because it changes the current optional `configOptionKeys` contract and can silently drop requested runtime controls such as `approval_policy`. Current source, docs, and tests expect missing keys to remain permissive and unsupported advertised-key controls to fail visibly.

## What This Changes

This PR changes ACP manager runtime-control application to send session config options only when `configOptionKeys` is non-empty and to filter out options outside the advertised key set.

## Best Possible Solution

Preserve the existing ACP capability contract, add a strict supported-key path or Claude ACP adapter mapping/suppression in the ACPX plugin, keep unsupported permission controls visible, cover both manager and ACPX behavior with tests, and add the changelog entry.

## Reproduction Assessment

Yes for the regression introduced by the patch: source inspection plus the existing manager test fixture show that missing `configOptionKeys` currently must still send persisted `model`, `thinking`, `approval_policy`, and `timeout` options, while the PR would send none. The linked issue #73875 also provides concrete live Claude ACP reproduction steps for the original failure, though I did not run live ACP in this read-only review.

## Solution Assessment

No. The patch solves the Claude strictness symptom by redefining missing `configOptionKeys` globally and hiding unsupported controls; a narrower fix should distinguish unknown/permissive capability reporting from a strict supported-key list or adapter-owned mapping.
