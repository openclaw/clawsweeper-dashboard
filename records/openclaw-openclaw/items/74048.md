## Summary

Keep open: the PR targets a real Claude ACP failure tracked by #73875, but this implementation changes the existing ACP capability contract and can silently ignore requested runtime controls such as approval policy.

## What This Changes

The branch changes ACP runtime-control setup so `session/set_config_option` is sent only for keys in `configOptionKeys`, and sends no config options when no keys are advertised.

## Best Possible Solution

Revise or replace this with a capability-aware ACP fix that preserves the missing-`configOptionKeys` permissive contract, gives strict backends an explicit supported-key path, keeps unsupported controls visible, adds focused regression tests, and includes a changelog entry.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path for the PR regression: the existing no-`configOptionKeys` manager fixture would stop receiving all persisted config-option calls under the patch. The original Claude failure is also concretely described in #73875, though I did not run a live Claude ACP session in this read-only review.

## Solution Assessment

No. The patch fixes the strict-backend symptom by redefining missing `configOptionKeys` and silently filtering unsupported advertised-key options; the safer solution is to preserve the existing contract while adding an explicit strict capability path for Claude/acpx.
