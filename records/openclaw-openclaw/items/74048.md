## Summary

Keep open. The PR targets a real ACP/Claude failure, but the proposed fix changes the established `configOptionKeys` contract: current OpenClaw treats an empty or missing list as permissive, while this patch treats it as “send nothing.” That would silently drop configured model, thinking, approval policy, timeout, and backend extras for any ACP runtime that accepts keys without publishing a strict list.

## What This Changes

This PR changes `applyManagerRuntimeControls` so ACP config options are forwarded only when `configOptionKeys` is non-empty, and all config options are skipped when no keys are advertised.

## Best Possible Solution

Revise the PR so OpenClaw preserves the current permissive meaning of missing `configOptionKeys`, while still avoiding unsupported Claude ACP generic keys through an explicit strict capability signal or corrected ACPX/adapter capability reporting. Add focused manager/ACPX tests and the required changelog entry.

## Reproduction Assessment

Yes. The existing manager test fixture advertises `session/set_config_option` without `configOptionKeys` and expects persisted runtime options to be applied; under this patch that same path would skip every option because `advertisedKeys.size` is zero. The linked #73875 report also provides concrete live Claude ACP reproduction steps for the original failure.

## Solution Assessment

No. The patch solves the strict-backend symptom by redefining an existing capability contract, which breaks permissive backends and can silently ignore requested controls. A safer fix is to distinguish “unknown/permissive” from “strict list” and only filter when the backend has actually supplied a strict key set.
