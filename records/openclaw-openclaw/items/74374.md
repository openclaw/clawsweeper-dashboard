## Summary

Keep open. Current main fixes the adjacent DeepSeek V4 enabled-thinking replay gap, but it still does not cover this report's disabled-thinking composition: `params.thinking: false` is ignored, DeepSeek V4 defaults back to enabled thinking, and the later `extra_body` merge can overwrite `thinking` to disabled after `reasoning_content` has already been added. The issue also has two open closing PRs, so it should remain open until maintainers reconcile, land, or reject those candidates.

## What This Changes

The issue requests that DeepSeek v4-pro disabled-thinking requests avoid replaying an inconsistent `reasoning_content` history that triggers DeepSeek's 400 response on later turns.

## Best Possible Solution

Land one cohesive DeepSeek V4 fix that treats common disabled values as `off`, prevents `extra_body.thinking.type = "disabled"` from leaving `reasoning_content` in the final payload, and adds regression coverage for the reporter's exact `params.thinking: false` plus `extra_body` setup. If #74403 and #74418 are both useful, reconcile them into one tested contract rather than closing this issue before merge.

## Reproduction Assessment

Yes. The high-confidence static reproduction path is `params.thinking: false` falling through to DeepSeek V4's `high` default, the provider wrapper adding enabled-thinking replay fields, and `extra_body` later overwriting `thinking` to disabled while leaving `reasoning_content` in messages.

## Solution Assessment

No, current main is not yet the best solution for this issue. The narrow maintainable fix should cover both disabled-alias normalization and the post-`extra_body` payload collision, with explicit tests for the reported config.
