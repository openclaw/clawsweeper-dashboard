## Summary

Keep open. Current main has the adjacent DeepSeek V4 enabled-thinking replay fix, but the reported disabled-thinking and `extra_body` composition is still not fully covered. Two open closing PRs already target this issue, so it should remain open until maintainers reconcile, land, or reject those candidates.

## What This Changes

Prevent DeepSeek V4 Pro multi-turn requests with disabled thinking from sending or replaying an inconsistent `reasoning_content` payload that triggers provider 400 errors.

## Best Possible Solution

Land one cohesive DeepSeek V4 payload contract that normalizes common disabled aliases before provider wrapping and leaves the final outgoing payload internally consistent after `extra_body` overrides, with regression coverage for `params.thinking: false`, `"disabled"`, `"none"`, plain assistant replay, and tool-call replay.

## Reproduction Assessment

Yes. The high-confidence reproduction path is static: the reporter's disabled aliases are ignored by `resolveThinkingDefault`, DeepSeek V4 falls back to `high`, the wrapper can add enabled-thinking `reasoning_content`, and `extra_body` then overwrites only the top-level `thinking` field.

## Solution Assessment

No. Current main is not the best solution for this issue because it fixes enabled-thinking assistant replay but leaves the disabled-thinking alias and `extra_body` collision paths open; the safer path is to reconcile the open PRs into one normalized request-payload contract.
