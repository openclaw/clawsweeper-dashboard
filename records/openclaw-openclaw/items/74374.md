## Summary

Keep open. Current main contains the adjacent DeepSeek V4 enabled-thinking replay fix, but the reporter's disabled-thinking configuration is still not fully covered because `params.thinking: false` is ignored and `extra_body.thinking` can overwrite the provider wrapper after `reasoning_content` has already been added; the issue also has open closing PRs that should be reconciled before closure.

## What This Changes

The issue reports DeepSeek v4-pro 400 responses when thinking is configured disabled because `reasoning_content` from earlier assistant turns can remain inconsistent with the next request payload.

## Best Possible Solution

Land one cohesive DeepSeek V4 fix that normalizes common disabled values to `off`, keeps `extra_body.thinking.type="disabled"` from leaving stale `reasoning_content` in the final payload, and adds regression coverage for the reporter's exact config.

## Reproduction Assessment

Yes. The high-confidence path is static and deterministic: `params.thinking: false` is not normalized, DeepSeek V4 falls back to `high`, the provider wrapper can add replay `reasoning_content`, and `extra_body.thinking.type="disabled"` can later overwrite `thinking`; live intermittency depends on DeepSeek returning `reasoning_content` in prior turns.

## Solution Assessment

No. Current main is not the best solution yet because it covers the enabled-thinking replay gap but leaves the disabled-alias and post-`extra_body` collision paths unresolved; reconciling #74403 and #74418 into one tested contract is safer.
