## Summary

Keep open. Current main still leaves the reporter's disabled-thinking DeepSeek path uncovered: `params.thinking: false` is not normalized to `off`, and `extra_body.thinking` can still overwrite the final payload after DeepSeek wrapping; open closing PRs #74403 and #74418 should be reconciled before this issue closes.

## What This Changes

DeepSeek v4-pro sessions configured with disabled thinking should avoid replaying conflicting `reasoning_content` so follow-up requests do not fail with a 400.

## Best Possible Solution

Land one cohesive DeepSeek V4 fix that normalizes common disabled values, handles final `extra_body.thinking.type = disabled` collisions, and adds regression coverage for the reporter's exact config.

## Reproduction Assessment

Yes. The static payload path is high-confidence: `params.thinking: false` falls through to the DeepSeek V4 `high` default, the provider wrapper can add `reasoning_content`, and the later `extra_body` merge can set `thinking` back to disabled; the live provider error only depends on DeepSeek returning or requiring the replay field across turns.

## Solution Assessment

No. Current main is not the best solution yet; the safer route is to reconcile the two open closing PRs into a single tested contract for disabled aliases and post-merge payload cleanup.
