## Summary

Keep open. Current main still leaves the reporter's disabled-thinking configuration uncovered: `params.thinking: false` and `"disabled"` do not resolve to the canonical off path, and `extra_body.thinking` can overwrite the DeepSeek wrapper after `reasoning_content` has already been added. The two open closing PRs should be reviewed or reconciled before this issue is closed.

## What This Changes

Fix DeepSeek v4-pro multi-turn requests so disabled thinking does not produce an inconsistent payload that later fails with DeepSeek's `reasoning_content` 400 error.

## Best Possible Solution

Land one cohesive DeepSeek V4 regression fix that normalizes disabled thinking aliases and prevents post-`extra_body` payload collisions, with regression coverage for the reporter's exact config.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: `params.thinking: false` falls through to DeepSeek V4's `high` default, the provider wrapper injects `reasoning_content`, and `extra_body.thinking.type="disabled"` can overwrite the top-level thinking field afterward.

## Solution Assessment

No. Current main only covers the adjacent enabled-thinking replay case; the narrow maintainable solution is to reconcile #74403 and #74418 or an equivalent combined fix before closing this issue.
