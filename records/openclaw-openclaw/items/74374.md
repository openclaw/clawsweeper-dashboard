## Summary

Keep open. Current main still has the reported disabled-thinking collision path: `params.thinking: false` or `"disabled"` falls through, DeepSeek V4 defaults to enabled thinking, and post-provider `extra_body` can overwrite `thinking` to disabled after `reasoning_content` has been added.

## What This Changes

The issue asks OpenClaw to prevent DeepSeek V4 disabled-thinking multi-turn requests from replaying an inconsistent `reasoning_content` payload.

## Best Possible Solution

Reconcile and land one DeepSeek V4 payload-contract fix that normalizes disabled thinking aliases and handles post-`extra_body` disabled collisions with regression coverage.

## Reproduction Assessment

Yes. Source reproduction is high confidence: the reported config still resolves to enabled DeepSeek V4 thinking before `extra_body` can overwrite the request to disabled, leaving a path for replayed `reasoning_content` in a disabled-thinking payload.

## Solution Assessment

No. Current main is not the best solution yet; the narrow maintainable fix should cover both disabled alias normalization and deterministic cleanup after `extra_body` collisions.
