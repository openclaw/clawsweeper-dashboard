## Summary

Keep open. Current main still leaves the reporter's disabled-thinking configuration uncovered because per-model `params.thinking: false` and `"disabled"` are not normalized to off, and `extra_body.thinking` is applied after the DeepSeek wrapper can add replay `reasoning_content`; the linked open PRs #74403 and #74418 explicitly target this issue.

## What This Changes

DeepSeek V4 Pro disabled-thinking conversations should avoid an inconsistent replay payload that makes DeepSeek reject the next turn with a `reasoning_content` 400 error.

## Best Possible Solution

Land one cohesive DeepSeek V4 fix that normalizes disabled-thinking aliases and prevents post-`extra_body` payload collisions from leaving `reasoning_content` in disabled requests, with regression coverage for the reporter's exact config.

## Reproduction Assessment

Yes. The issue body gives a concrete `params.thinking: false` plus `extra_body.thinking.type="disabled"` setup, and current main still has the static path that resolves DeepSeek V4 to enabled thinking before `extra_body` overwrites the payload.

## Solution Assessment

No for current main. The narrow maintainable solution is to reconcile #74403 and #74418 or equivalent changes into one tested fix rather than closing this issue on the adjacent enabled-thinking replay repair.
