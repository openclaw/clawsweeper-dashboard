## Summary

Keep open. Current main only partially addresses the DeepSeek V4 replay problem; the reporter's `params.thinking: false` plus `extra_body.thinking.type=disabled` path can still produce an inconsistent request, and two open closing PRs already target the issue.

## What This Changes

DeepSeek V4 Pro conversations with disabled thinking should not send inconsistent `reasoning_content` history that causes 400 responses.

## Best Possible Solution

Implement and ship one DeepSeek V4 payload-contract fix that normalizes disabled thinking aliases and prevents disabled `extra_body.thinking` from leaving `reasoning_content` in the next request, with regression coverage.

## Reproduction Assessment

Yes. Static reproduction is high confidence: `params.thinking: false` is ignored, DeepSeek V4 defaults to high thinking, the wrapper can add `reasoning_content`, and `extra_body.thinking.type=disabled` is applied afterward.

## Solution Assessment

No. The current main implementation is not the full fix; the maintainable path should reconcile the alias normalization and post-`extra_body` payload cleanup covered separately by the linked PRs.
