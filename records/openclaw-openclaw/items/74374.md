## Summary

Keep open. Current main has the adjacent DeepSeek V4 enabled-thinking replay fix, but it still does not cover the reporter's disabled-thinking composition: `params.thinking: false`/`"disabled"` is not normalized to `off`, and `extra_body.thinking.type="disabled"` is applied after the DeepSeek wrapper without a final reasoning-content cleanup. Two open PRs already reference this issue with closing syntax, so the issue should remain open until maintainers reconcile, merge, or reject those candidates.

## What This Changes

The issue asks OpenClaw to prevent DeepSeek V4 `reasoning_content` from poisoning multi-turn history when DeepSeek thinking is disabled through per-model params and `extra_body`.

## Best Possible Solution

Land one cohesive fix that normalizes common disabled values (`false`, `disabled`, `none`) to `off` before provider wrapping and also prevents a later `extra_body.thinking.type="disabled"` override from leaving replayed `reasoning_content` in the outgoing DeepSeek V4 payload, with regression tests for the reporter's combined config.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: configure `deepseek/deepseek-v4-pro` with `params.thinking: false` plus `extra_body.thinking.type="disabled"`; current main resolves the thinking level to the DeepSeek default `high`, the provider wrapper backfills `reasoning_content`, and the later extra-body merge can overwrite only `thinking` to disabled.

## Solution Assessment

No. Current main is not the best solution for this issue because it fixes the enabled-thinking replay path but leaves disabled alias normalization and post-`extra_body` collision cleanup unresolved; the safer solution is to reconcile #74403 and #74418 into one consistent request-payload contract.
