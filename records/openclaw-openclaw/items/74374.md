## Summary

Keep open. Current main still has the reported DeepSeek disabled-thinking collision path: per-model `thinking: false` or `"disabled"` is not normalized to `off`, and `extra_body` can overwrite a provider-wrapped payload to disabled after `reasoning_content` handling has already run. Linked PRs #74403 and #74418 already target this issue, so the issue should stay open until one reconciled fix lands.

## What This Changes

The issue reports DeepSeek v4-pro multi-turn 400s when disabled thinking is configured but replayed assistant `reasoning_content` and `extra_body.thinking` produce an inconsistent DeepSeek payload.

## Best Possible Solution

Review and land one combined DeepSeek V4 payload-contract fix from the linked PRs, covering disabled alias normalization, post-`extra_body` cleanup, and regression tests for the reported config.

## Reproduction Assessment

Yes. Source reproduction is high confidence: `params.thinking: false` or `"disabled"` falls through to the DeepSeek V4 default, then `extra_body.thinking.type="disabled"` can overwrite a payload whose DeepSeek wrapper path has already handled `reasoning_content`.

## Solution Assessment

No. Current main is only a partial fix; the maintainable solution should reconcile disabled alias normalization with the post-`extra_body` DeepSeek cleanup and confirm the disabled replay behavior with regression coverage.
