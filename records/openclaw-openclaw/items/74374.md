## Summary

Keep open. Current main has the adjacent DeepSeek V4 enabled-thinking replay fix, but the reported disabled-thinking composition is still not fully covered: disabled aliases are not normalized before provider wrapping, and `extra_body.thinking.type="disabled"` can still overwrite the provider payload after DeepSeek wrapping. The issue is also paired with two open closing PRs, so it should remain open until maintainers reconcile, land, or reject those candidates.

## What This Changes

The issue asks OpenClaw to keep DeepSeek V4 disabled-thinking requests consistent by not replaying `reasoning_content` when thinking is disabled or by otherwise enabling thinking consistently.

## Best Possible Solution

Land one cohesive fix that normalizes common disabled aliases to `off` before provider wrapping and enforces a consistent disabled DeepSeek payload after `extra_body` overrides, with regression coverage for the reporter's `params.thinking: false` plus `extra_body.thinking.type=disabled` setup.

## Reproduction Assessment

Yes. The high-confidence reproduction path is static: `params.thinking: false` or `"disabled"` is ignored, DeepSeek V4 falls back to its `high` profile default, the wrapper backfills `reasoning_content`, and `extra_body` can then overwrite only the top-level `thinking` field to disabled.

## Solution Assessment

No. Current main is not the best solution for this issue because it fixes enabled-thinking assistant replay but leaves the disabled-thinking collision path open; the safer path is to reconcile the two open PRs into one normalized request-payload contract.
