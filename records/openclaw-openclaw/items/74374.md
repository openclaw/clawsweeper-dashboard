## Summary

Keep open. Current main covers the adjacent enabled-thinking DeepSeek V4 replay gap, and AmnAnon’s comment usefully confirms that direction, but the reporter’s disabled-thinking composition is still not fully covered: per-model `params.thinking: false`/`"disabled"` is not normalized to `off`, and `extra_body.thinking.type="disabled"` still overwrites the payload after the DeepSeek wrapper without a final cleanup. The issue also has two open closing PRs, #74403 and #74418, so it should remain open until maintainers reconcile, land, or reject those candidates.

## What This Changes

The issue requests a DeepSeek V4 disabled-thinking fix so stored or backfilled `reasoning_content` does not make follow-up `deepseek-v4-pro` requests fail with a 400 after `extra_body` disables thinking.

## Best Possible Solution

Reconcile #74403 and #74418 into one cohesive fix: normalize common disabled aliases to `off` before DeepSeek provider wrapping, enforce a consistent disabled DeepSeek payload after `extra_body` overrides, and add a regression test for the reporter’s `params.thinking: false` plus `extra_body.thinking.type="disabled"` setup.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists on current main: the reported `params.thinking: false` is ignored by `resolveThinkingDefault`, DeepSeek V4 falls back to `high`, the provider wrapper can add `reasoning_content`, and then `extra_body` overwrites only the top-level `thinking` field to disabled.

## Solution Assessment

No. Current main is not the best solution for this issue because it covers enabled-thinking replay but leaves the disabled-thinking collision path open; the safer path is a single reviewed fix that handles both alias normalization and final payload consistency.
