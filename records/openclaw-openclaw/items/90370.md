## Summary

Keep open as the canonical product/design tracker for PostgreSQL-capable internal storage. Current main and v2026.6.8 remain intentionally SQLite-only for OpenClaw-owned runtime state, the dual-backend abstraction issue was closed back to this issue, and the active RFC narrows toward SQLite snapshot durability rather than choosing PostgreSQL.

## What This Changes

The issue requests a configurable PostgreSQL alternative to SQLite for OpenClaw internal runtime storage, including shared state, task/session flows, plugin indexes, and related stores.

## Best Possible Solution

Keep this issue open as the canonical design tracker until maintainers choose between SQLite-only core storage, snapshot-based durability, or a backend-neutral storage abstraction with explicit config, dependency, transaction, and migration contracts.

## Reproduction Assessment

Not applicable; this is a request for a new PostgreSQL-capable storage backend, not a bug report with failing current behavior to reproduce. Source and release inspection confirm the current behavior remains SQLite-only.

## Solution Assessment

Unclear as proposed; a direct storage.type config switch would be too small for the current synchronous SQLite architecture. The maintainable path needs maintainer-approved backend abstraction, async conversion, config, dependency, transaction, backup, and migration design first.
