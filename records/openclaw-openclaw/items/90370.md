## Summary

Keep open as the canonical storage-backend product/design tracker. Current main and v2026.6.9 are still intentionally SQLite-only across shared state, per-agent state, plugin state, memory, Kysely execution, and transaction handling; the later dual-backend issue was closed back to this issue, and the snapshot/RFC discussion explicitly keeps SQLite authoritative while addressing durability separately.

## What This Changes

Feature request to support PostgreSQL as an optional replacement or peer backend for OpenClaw-owned internal SQLite storage.

## Best Possible Solution

Keep this issue open as the canonical storage-backend design item until maintainers choose SQLite-only core storage, snapshot-based durability, or a staged backend-neutral abstraction with explicit config, migration, credential, and transaction contracts.

## Reproduction Assessment

Not applicable; this is a feature/design request rather than a bug report. Source and release inspection confirm current behavior remains SQLite-only, but there is no failing current-main behavior to reproduce.

## Solution Assessment

Unclear as proposed; PostgreSQL support may be valuable for managed or enterprise deployments, but the maintainable path needs maintainer-approved backend, async transaction, config, dependency, credential, backup, and migration contracts first.
