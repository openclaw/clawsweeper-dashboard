## Summary

Keep open as the canonical product/design tracker for PostgreSQL or backend-neutral internal storage. Current main and the latest release remain SQLite-only across the shared state DB, per-agent DB, plugin state, memory-store config, and sync transaction/Kysely seams, while the related dual-backend issue was closed back to this issue and the newer snapshot/RFC discussion deliberately keeps SQLite as the local authoritative runtime database.

## What This Changes

The issue requests PostgreSQL as an optional or replacement backend for OpenClaw-owned internal storage such as task state, task flows, plugin indexes, and channel/runtime state.

## Best Possible Solution

Keep this issue open as the canonical storage-backend design item until maintainers choose SQLite-only core storage, snapshot-based durability, or a staged backend-neutral abstraction with explicit config, migration, credential, and transaction contracts.

## Reproduction Assessment

Not applicable; this is a feature/design request rather than a bug report. Source and release inspection confirm current behavior remains SQLite-only, but there is no failing current-main behavior to reproduce.

## Solution Assessment

Unclear as proposed; PostgreSQL support may be valuable for managed or enterprise deployments, but the maintainable path needs maintainer-approved backend, async transaction, config, dependency, credential, and migration contracts first.
