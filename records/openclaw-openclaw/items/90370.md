## Summary

Keep open as the canonical product/design tracker for PostgreSQL or backend-neutral internal storage. Current main and the latest release remain SQLite-only across shared state, per-agent state, plugin state, memory search, and transaction/Kysely seams; the related dual-backend issue was closed back to this issue, while the snapshot/RFC discussion intentionally keeps SQLite as the local authoritative runtime database.

## What This Changes

Request support for PostgreSQL as an optional or replacement backend for OpenClaw internal runtime storage instead of mandatory SQLite.

## Best Possible Solution

Keep this as the canonical storage-backend design item until maintainers choose SQLite-only core storage, snapshot-based durability, or a staged backend-neutral abstraction with explicit config, credential, migration, dialect, and transaction contracts.

## Reproduction Assessment

Not applicable; this is a feature/design request rather than a bug report. Source and release inspection confirm the requested PostgreSQL backend is absent, but there is no current-main failure to reproduce.

## Solution Assessment

Unclear as proposed; a single storage.type config knob would not cover sync-to-async transactions, schema dialects, plugin SDK contracts, memory/search, backup, or migrations. The maintainable path starts with a maintainer-approved storage architecture decision.
