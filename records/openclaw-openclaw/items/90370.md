## Summary

Keep open for maintainer product/API direction. Current main and v2026.6.8 still use SQLite-only shared and per-agent runtime state, the closed dual-backend issue points back to this issue as the canonical tracker, and the new RFC is a narrower durability discussion that intentionally keeps SQLite authoritative while running.

## What This Changes

The issue requests optional PostgreSQL-backed internal storage instead of SQLite-only OpenClaw-owned state.

## Best Possible Solution

Keep this as the canonical core storage-backend product/design tracker; if maintainers pursue it, start with an accepted RFC or design that defines the storage seam, config/secret contract, migration story, transaction semantics, and SQLite default behavior before any PostgreSQL implementation.

## Solution Assessment

No for the literal config-switch proposal as a narrow fix. The maintainable path is a maintainer-approved storage architecture/RFC that preserves SQLite as the default until an explicit backend contract and migration plan exist.
