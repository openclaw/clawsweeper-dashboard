## Label Justifications

- `P1`: The report describes a shipped memory_search workflow that can stay paused after repair while repeated rebuilds consume tens of GB.
- `impact:session-state`: The issue affects memory_search index identity and live manager state, leaving session memory recall unavailable despite a repaired on-disk index.
- `impact:crash-loop`: Abandoned temp DB families and stalled reindex processes can create disk-pressure and runtime availability failures for long-running gateways.

## Summary

Keep open: current main still has the core ingredients of the reported failure, including safe reindex exposing the shared manager to a temp DB, no evident stale temp-family sweep after process interruption, and separate CLI/default manager instances that can explain why CLI repair succeeds while live memory_search remains paused.

## What This Changes

The issue reports memory-core reindex thrash that leaks `main.sqlite.tmp-*` files and leaves the live `memory_search` manager paused after a successful CLI rebuild.

## Best Possible Solution

Memory-core should isolate safe reindex construction from the live search manager, clean abandoned temp DB families from interrupted runs, and refresh or invalidate cached default-manager identity after a successful CLI rebuild.

## Reproduction Assessment

Yes for a source-level reproduction path: current main still has the shared-manager temp DB swap, separate CLI/default managers, post-search paused-status gate, and no evident stale temp-family sweep. I did not run the reporter's large live index scenario, so confidence is medium rather than high.

## Solution Assessment

No; current main is not the best solution because it still leaves the reported state boundaries unresolved. The maintainable fix should be a bounded memory-core repair rather than a new config option or runtime workaround.
