## Summary

Keep open. Current main still has the misleading no-provider infer error, and this PR is a plausible narrow fix, but the patch misses the linked directory/unsupported-input reproduction because it only treats skipped decisions with selected attachments and zero attempts as missing provider configuration; no-attachment decisions still fall through to the path-blaming errors.

## What This Changes

The PR threads media-understanding runner decisions through file runtime results, maps infer image/audio no-provider skipped results to clearer configuration errors, adds regression tests, and updates the changelog.

## Best Possible Solution

Land a narrow version that reports missing provider configuration whenever no image/audio provider can be resolved, independent of whether the supplied path is a recognized media attachment, while preserving separate empty-output and unsupported-input errors once a provider is actually configured.

## Reproduction Assessment

Yes. The linked bug gives deterministic CLI commands for no-provider audio/image infer paths, and code inspection confirms current main still falls back to generic no-text errors. The PR covers media-looking skipped/no-attempt inputs but not the directory/no-attachment variant from #73569.

## Solution Assessment

No. Threading the runner decision through runtime is the right direction, but relying on the post-selection skipped/no-attempt shape misses no-provider cases that become `no-attachment`; provider availability should be checked before attachment filtering or carried as an explicit no-provider decision.
