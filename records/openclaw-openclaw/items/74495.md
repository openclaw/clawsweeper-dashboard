## Summary

Keep open. The PR fixes the selected-media no-provider path, but it still misses a concrete reproduction from the linked bug: directory or unsupported paths become a `no-attachment` media decision before provider resolution, so the new helper falls back to the old path-blaming error.

## What This Changes

The PR threads media-understanding decisions through file runtime results, maps no-provider infer image/audio results to clearer CLI errors, adds regression tests, and adds a changelog entry.

## Best Possible Solution

Land a narrow version that reports missing image/audio provider configuration independently of attachment selection, while preserving separate unsupported-input and empty-output errors once a provider is actually configured or called.

## Reproduction Assessment

Yes. The linked bug gives deterministic CLI commands for no-provider audio/image infer paths, and code inspection shows directory inputs become `no-attachment` before the PR's skipped/no-attempt detection can fire.

## Solution Assessment

No. Propagating the runner decision is a good direction, but the current helper is too late and too narrow; the missing-provider condition should be checked before attachment filtering or preserved as an explicit decision that survives the no-attachment path.
