## Summary

Keep open. The PR moves in the right direction, but it still misses the linked bug's directory and unsupported-input no-provider cases because those paths become `no-attachment` before the new missing-provider helper can see a skipped/no-attempt decision.

## What This Changes

The PR propagates media-understanding decisions through file runtime results and adds CLI errors, tests, and a changelog entry for image/audio no-provider skipped decisions.

## Best Possible Solution

Land a narrow fix that reports missing image/audio provider configuration whenever no provider can be resolved, while preserving separate unsupported-input and empty-provider-output errors once provider availability is known.

## Reproduction Assessment

Yes. The linked bug gives deterministic no-provider CLI commands, and source inspection shows unsupported paths are filtered into `no-attachment` before provider resolution, leaving the old generic error path reachable.

## Solution Assessment

No. Propagating the runner decision is useful, but the missing-provider check needs to happen before attachment filtering or be carried as an explicit no-provider signal that survives `no-attachment` outcomes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86
