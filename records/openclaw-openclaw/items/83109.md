## Summary

Keep open: current main still defaults normal Codex app-server threads to code-mode-only, v2026.5.12 shipped the same setting with @openai/codex 0.130.0, and two later live comments report that disabling the flags restores tool turns. The remaining decision is how to reconcile that regression with the intentionally documented native code-mode policy.

## What This Changes

The issue asks OpenClaw to stop forced Codex code-mode-only from wedging tool-using Codex-runtime turns and restore tool execution followed by a final reply.

## Best Possible Solution

Land an owner-approved Codex harness change that either completes native code-mode dynamic-tool execution reliably or disables/opts in code-mode-only, with focused regression coverage and real tool-using Codex turn proof.

## Reproduction Assessment

Yes for the released regression and source path: the issue gives a CLI repro, logs, rollout evidence, and a package bisect, while later comments confirm live beta.5/beta.6 failures. I did not live-run current main, so this is source-reproducible rather than a current-main live reproduction.

## Solution Assessment

Unclear: disabling both flags restored affected installs in the comments, but code-mode-only was an intentional Codex harness direction. The best fix needs owner choice between completing the native code-mode tool contract and disabling or making code-mode-only opt-in.
