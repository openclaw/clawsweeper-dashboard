## What This Changes

The issue reports that bundled Codex runtime threads force `features.code_mode` and `features.code_mode_only`, causing tool-using Codex app-server turns to hang after the 2026.5.10-beta.5/2026.5.12 upgrade.

## Best Possible Solution

Restore successful Codex-runtime tool turns with regression coverage and real Codex app-server proof; maintainers should decide whether to disable `code_mode_only` by default, make it configurable, or repair the native code-mode nested-tool completion path.

## Reproduction Assessment

Yes, there is a high-confidence reproduction path from source plus the reporter's live logs: a Codex-runtime tool-using turn on 2026.5.12 while current main still forces `features.code_mode_only`. I did not run the live model path during this read-only review, so the structured status is source-reproducible rather than locally reproduced.

## Solution Assessment

Unclear which fix is best without owner input. The issue correctly identifies broken shipped behavior, but the maintainable solution should be chosen between restoring the previous default, adding an override, or making OpenClaw's Codex code-mode nested-tool path reliably reach `turn/completed`.
