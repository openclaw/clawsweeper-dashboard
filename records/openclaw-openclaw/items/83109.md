## Summary

This should stay open: the report is a concrete released regression in the Codex runtime path, and current main still intentionally forces code-mode-only while the exact tool-turn completion behavior has not been proven fixed.

## What This Changes

Report that Codex-runtime tool-using turns hang after 2026.5.12 because the Codex plugin forces `features.code_mode` and `features.code_mode_only` on every thread.

## Best Possible Solution

Make Codex-runtime tool-using turns complete reliably, either by supporting the native code-mode tool contract end to end or by disabling/overriding code-mode-only until that path has live proof and regression coverage.

## Reproduction Assessment

Yes for the released regression path: the issue provides a concrete 2026.5.12 CLI command, gateway diagnostics, rollout JSONL, and a package bisect. I did not live-run current main, but source inspection confirms the forced code-mode-only config remains in the current Codex plugin.

## Solution Assessment

Unclear: flipping the flags is a plausible mitigation, but code-mode-only is an intentional harness policy from the related merged PR. The maintainable fix should be chosen by the Codex harness owners and proven with a real tool-using Codex turn.
