## Summary

This should stay open: the report describes a concrete released Codex-runtime regression, current main still forces code-mode-only, and I found no merged fix or canonical replacement issue that owns the same root cause.

## What This Changes

The issue reports that forced Codex code-mode-only thread config makes tool-using Codex-runtime turns hang after emitting `custom_tool_call` items.

## Best Possible Solution

Make Codex-runtime tool-using turns complete reliably, either by supporting native code-mode tool execution end to end or by disabling/overriding code-mode-only until that path has live proof and regression coverage.

## Reproduction Assessment

Yes for the released/source path: the issue provides a concrete CLI prompt, stalled-session diagnostics, rollout JSONL, a package bisect, and a mitigation, while current source still forces code-mode-only. I did not run a live current-main Codex turn in this read-only review.

## Solution Assessment

Unclear: flipping or making the flags overrideable is a plausible mitigation, but code-mode-only was an intentional harness policy. The maintainable fix should be chosen by Codex harness owners and proven with a real tool-using Codex turn.
