## Summary

Keep open: current main still enables Codex code-mode-only by default, tests and docs pin that behavior, and the later live comments report that disabling those flags restores tool turns. Current main has timeout mitigations for stale turns, but that does not prove the code-mode-only tool path now completes successfully.

## What This Changes

The issue asks OpenClaw to stop unconditionally forcing Codex code-mode-only, or otherwise make Codex-runtime tool turns complete instead of stalling.

## Best Possible Solution

Make Codex-runtime tool turns complete reliably, either by fully supporting the native code-mode tool contract end to end or by owner-approved disabling/overriding of code-mode-only with docs and regression proof kept aligned.

## Reproduction Assessment

Yes for released and beta builds: the issue has a concrete CLI repro, logs, rollout evidence, a package bisect, and two independent live comments confirming the same mitigation. For current main, source inspection shows the forced default remains, but I did not run a live Codex turn.

## Solution Assessment

Unclear as an implementation choice: flipping the flags is a proven local mitigation, but code-mode-only was an intentional harness policy. The maintainable fix should be selected by Codex harness owners and proven with a real tool-using Codex turn plus focused regression coverage.
