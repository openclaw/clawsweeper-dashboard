## Summary

Keep open: current main still forces Codex code-mode-only, and the later timeout/release guards do not prove tool-using Codex-runtime turns now complete successfully. The issue has concrete source and live-install evidence, but the fix changes an intentional Codex harness policy and should go through maintainer review.

## What This Changes

Issue reports that Codex-runtime tool-using turns hang because the bundled Codex plugin forces `features.code_mode_only=true` and prevents normal OpenClaw tool-turn completion.

## Best Possible Solution

Make Codex-runtime tool-using turns complete reliably, either by implementing the native code-mode tool contract end to end or by relaxing code-mode-only behind a maintainer-approved default/override with regression and live tool-turn proof.

## Reproduction Assessment

Yes, source-reproducible with high confidence: current main still forces the same code-mode-only config, and the issue plus follow-up comment provide commands, logs, rollout snippets, package bisect, and a live hotfix result. I did not run a live Codex turn in this read-only review, so I am not marking it as directly reproduced here.

## Solution Assessment

Unclear: disabling or making code-mode-only overridable is a plausible mitigation, but code-mode-only is documented and test-pinned as intentional harness policy. The maintainable fix should be chosen by Codex harness owners and proven with a real tool-using Codex-runtime turn.
