## Summary

Keep open: current `main` still defaults unrestricted Codex app-server turns to `features.code_mode_only: true`, and the newer merged policy work only disables that native surface for restricted allowlist/deny cases rather than the normal tool-using turn reported here.

## What This Changes

The issue requests fixing a released Codex-runtime regression where forced code-mode-only turns emit wrapped `custom_tool_call`/`exec` output and fail to complete tool-using agent turns.

## Best Possible Solution

Make normal Codex-runtime tool turns complete reliably, either by proving and fixing the Codex-native nested tool bridge end to end or by replacing forced code-mode-only with a safe compatibility/default policy plus live regression proof.

## Reproduction Assessment

Yes, source-reproducible with strong external live reports: current `main` still sends code-mode-only for unrestricted Codex turns, and commenters supplied before/after live installs where disabling the forced flags restored tool-turn completion. I did not personally run a live current-main Codex turn in this read-only sweep.

## Solution Assessment

Unclear: the local mitigation of flipping the flags is well-supported, but code-mode-only was intentionally introduced as the Codex-native tool path. The best fix needs a Codex harness owner decision and proof that either the native nested-tool bridge works or the compatibility/default policy is changed safely.
