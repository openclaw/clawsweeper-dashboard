## Summary

This should stay open: current main still forces Codex code-mode-only for thread start/resume, and the remaining question is not already implemented or duplicated by another open item. The report has strong source-level and released-behavior evidence, but the fix direction needs Codex harness owner judgment because the forced mode was merged and documented intentionally.

## What This Changes

The issue reports a released Codex-runtime regression where forced `features.code_mode_only` causes tool-using turns to emit JS `custom_tool_call` items and stall instead of completing OpenClaw dynamic tools.

## Best Possible Solution

Make Codex-runtime tool-using turns complete reliably, either by supporting the native code-mode dynamic-tool contract end to end or by changing the default/override policy with focused regression coverage and live proof.

## Reproduction Assessment

Yes for source-level and released-path reproduction: the issue provides a concrete 2026.5.12 command, logs, rollout evidence, and a package bisect, and current main still forces the same flags. I did not live-run the Codex turn in this read-only review, so this is source-reproducible rather than fully reproduced.

## Solution Assessment

Unclear: flipping the flags or merge order may be a valid mitigation, but code-mode-only is an intentional Codex harness policy from the merged related PR. The maintainable fix should be chosen by the Codex harness owners and proven with a real tool-using Codex turn.
