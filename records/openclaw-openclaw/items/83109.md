## Summary

Keep open: current main still forces Codex code-mode-only through the same merge order, and tests/docs pin that as intentional behavior. Adjacent watchdog fixes reduce stale-lane blast radius, but they do not prove successful tool-turn completion or settle whether this runtime policy should stay default-on.

## What This Changes

The issue asks OpenClaw to stop Codex-runtime tool turns from hanging when the bundled Codex plugin forces code-mode-only thread config.

## Best Possible Solution

Make Codex-runtime tool-using turns complete reliably, either by supporting the native code-mode tool contract end to end with live proof and regression coverage or by disabling/honoring an override for code-mode-only until that path is ready.

## Reproduction Assessment

Yes for the released regression path: the issue provides a concrete 2026.5.12 CLI command, gateway diagnostics, rollout JSONL, package bisect, and mitigation. I did not live-run current main, but source inspection confirms current main still forces code-mode-only while newer watchdogs may change the failure from indefinite stall to timeout/release.

## Solution Assessment

Unclear: flipping the flags off is a plausible mitigation, but code-mode-only was introduced and documented as an intentional Codex harness policy. The maintainable fix should be chosen by the Codex harness owners and proven with a real tool-using Codex turn.
