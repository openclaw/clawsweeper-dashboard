## Summary

This should stay open: current main still forces the Codex harness into code-mode-only for unrestricted Codex runs, and three live issue datapoints now tie that behavior to failed or wedged tool turns on shipped/beta installs.

## What This Changes

The issue reports a regression where Codex-runtime agents hang or fail on tool-using turns because the bundled Codex plugin forces `features.code_mode` and `features.code_mode_only` into thread config.

## Best Possible Solution

Make Codex-runtime tool-using turns complete reliably by either supporting Codex native code-mode tool continuation end to end or relaxing `code_mode_only` under an owner-approved default/override with focused regression coverage and live proof.

## Reproduction Assessment

Yes for a high-confidence source reproduction path: current main still forces code-mode-only for unrestricted Codex runs, and the issue now has shipped-release plus beta live confirmations that patching those flags off restores tool-turn completion. I did not run the live Codex turn in this read-only review.

## Solution Assessment

Unclear: flipping the flags is a proven mitigation in the reports, but code-mode-only was intentionally introduced for the Codex harness. The maintainable fix should be selected by the Codex harness owners and proven with a real tool-using Codex turn.
