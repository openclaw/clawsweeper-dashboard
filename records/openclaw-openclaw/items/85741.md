## Summary

This looks like a valid current-main bug: ACP WebChat turns can both persist an `acp-runtime` assistant row and later mirror the delivered final as `gateway-injected`, and the display projection does not dedupe that pair.

## What This Changes

Prevent WebChat/Control UI ACP chat turns from storing both `acp-runtime` and `gateway-injected` assistant transcript rows for the same reply.

## Best Possible Solution

Treat ACP WebChat turns as transcript-owning agent runs so the UI still receives the ACP final reply but `chat.send` does not append a second `gateway-injected` transcript row; add regression coverage for `chat.history` returning one assistant reply.

## Reproduction Assessment

Yes at source level: current main has a clear two-write path for ACP WebChat replies, and the report includes a concrete transcript plus `chat.history` verification command. I did not execute a live Cursor/acpx turn in this read-only review.

## Solution Assessment

Yes, the write-path fix is the best fit: ACP dispatch should mark or otherwise be handled as transcript-owned before WebChat final mirroring. A read-path filter could reduce symptoms, but by itself would leave duplicate stored transcript rows behind.
