## Summary

This remains a valid current-main bug: ACP WebChat replies can be persisted once by the ACP runtime and then mirrored again as a gateway-injected assistant row. The issue should stay open because https://github.com/openclaw/openclaw/pull/85775 is an open closing PR for the same report.

## What This Changes

WebChat/Control UI should show only one assistant reply for Cursor ACP sessions instead of rendering both the ACP-persisted and gateway-injected transcript rows.

## Best Possible Solution

Review and land or replace the linked fix so ACP WebChat turns produce one visible assistant reply while preserving normal non-duplicate gateway-injected WebChat delivery.

## Reproduction Assessment

Yes at source level: current main has a clear ACP transcript write plus WebChat gateway-injected append path, and the report includes concrete transcript rows plus a `chat.history` verification command. I did not run a live Cursor/acpx WebChat session in this read-only review.

## Solution Assessment

Yes for the issue direction: the durable fix is to ensure ACP WebChat turns produce one visible assistant reply with focused history coverage. The linked PR's projection-only approach may be acceptable, but maintainers should decide whether the write path should also avoid creating duplicate raw transcript rows.
