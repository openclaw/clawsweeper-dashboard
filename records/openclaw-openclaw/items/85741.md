## What This Changes

The issue reports duplicate WebChat/chat.history assistant replies for Cursor ACP sessions when both ACP runtime persistence and WebChat gateway-injected final delivery store the same response.

## Best Possible Solution

Land or adjust the linked fix so ACP-owned turns do not leave duplicate visible or stored assistant rows, while preserving normal gateway-injected WebChat replies and TTS/media supplements.

## Reproduction Assessment

Yes at source level: current main has a clear two-write path for ACP WebChat replies, and the report includes concrete transcript rows plus a chat.history verification command. I did not execute a live Cursor/acpx turn in this read-only review.

## Solution Assessment

Yes for the desired endpoint: ACP transcript-owned turns should not also be stored as gateway-injected final replies. A read-path filter can reduce the WebChat symptom, but a write-path repair is cleaner if maintainers want the stored transcript fixed too.
