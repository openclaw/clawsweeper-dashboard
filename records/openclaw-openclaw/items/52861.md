## Summary

Keep open: current main and v2026.5.22 still only partially mitigate the reported per-turn overhead. Tool Search and compact skills exist, but direct tool exposure remains the documented default, skills are budget-filtered rather than relevance-selected, workspace bootstrap still defaults to per-turn injection, and the latest discussion explicitly calls out workspace-file injection as unresolved.

## What This Changes

Reduce repeated per-turn prompt overhead by conditionally injecting tool schemas, lazily exposing skills, indexing or streaming workspace bootstrap files, and compacting long-session context.

## Best Possible Solution

Keep this as the umbrella performance/design thread and land focused, measured changes for tool routing, skill visibility, workspace bootstrap policy, and compaction behavior after maintainer decisions on defaults and compatibility.

## Reproduction Assessment

Yes at source level: current main still defaults to broad tool exposure, budget-based skill prompting, and always-on workspace bootstrap injection unless explicit controls are configured. I did not rerun the live Feishu/Lark token-count measurement in this read-only review.

No as one automatic fix: the proposal spans tool schema routing, public skill activation, workspace bootstrap semantics, and compaction policy. The maintainable path is incremental, measured changes after maintainer agreement on compatibility and defaults.
