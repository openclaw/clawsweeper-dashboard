## Summary

Keep open. Current main has partial prompt-cost mitigations for skills, bootstrap files, and compaction, but it still registers the whole policy-filtered effective tool set as model-facing custom tool schemas for a run, so the central conditional per-turn tool injection request remains unimplemented.

## What This Changes

Request to reduce static per-turn prompt and context overhead by injecting tool schemas, skill descriptions, workspace files, and old conversation context conditionally or on demand.

## Best Possible Solution

Keep this as the umbrella prompt-overhead tracker and split the work into policy-preserving pieces: add a relevance layer after effective tool policy but before custom tool registration, leave the skills catalog on the existing compact/on-demand path, and handle bootstrap streaming/indexing as a separate context-engine or bootstrap-injection design.

## Reproduction Assessment

Yes for the remaining implementation state: on current main, following `attempt.ts` to `splitSdkTools` to `toToolDefinitions` shows every effective tool is registered with its schema. A live Feishu session is not required to reproduce that source-level behavior, though it would be useful for measuring final savings.

## Solution Assessment

Unclear for a concrete implementation because no patch is proposed. The requested direction is plausible, but the maintainable fix must preserve effective tool policy and provider compatibility rather than adding an early message-keyword filter.
