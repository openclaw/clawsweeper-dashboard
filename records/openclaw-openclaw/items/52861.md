## Summary

Keep open. Current main has real prompt-cost mitigations for skills, bootstrap files, and compaction, but the central request remains unimplemented: the embedded runner still resolves a run-level effective tool set and forwards every effective tool schema to Pi custom-tool registration, with no per-turn semantic relevance router.

## What This Changes

The issue asks OpenClaw to reduce repeated per-turn prompt cost by conditionally injecting relevant tool schemas, lazily exposing skills, bounding workspace bootstrap context, and compacting long conversations.

## Best Possible Solution

Keep this as the umbrella prompt-overhead tracker and split follow-up work into policy-preserving pieces: add a per-turn relevance layer after effective tool policy but before custom-tool registration, keep skills on the bounded catalog path, and treat bootstrap streaming/indexing as separate context-engine design.
