## Summary

Keep open. Current main and the latest release still contain the WebChat runtime-policy exclusion, and the nearby tests assert the reported main-session, unsandboxed fallback; the remaining question is the intended WebChat operator/security boundary, not whether the behavior exists.

## What This Changes

The issue reports that `session.dmScope: "per-channel-peer"` is ignored for WebChat because direct WebChat contexts stay on `agent:main:main` instead of resolving to a per-channel direct-peer session.

## Best Possible Solution

Define a consistent WebChat isolation contract. If WebChat should participate in secure DM/runtime-policy isolation, add a WebChat-aware session-key/runtime-policy path with regression coverage that preserves external delivery-route protections; if WebChat is intentionally an internal operator exception, document that exception in WebChat, routing, session, and security docs.

## Reproduction Assessment

Yes. The current resolver and unit test provide a high-confidence reproduction path: a direct WebChat context on `agent:main:main` resolves back to `agent:main:main`, and `non-main` sandbox mode remains unsandboxed. I did not run tests because this review was required to keep the checkout read-only.

## Solution Assessment

Unclear. Removing `channel !== "webchat"` is the narrow local edit, but the surrounding tests and delivery guards show WebChat is intentionally treated as an internal surface, so the safest solution depends on maintainer product/security direction.
