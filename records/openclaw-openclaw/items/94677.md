## Summary

Keep open: current main has push-completion guidance and yield-specific protection, but the reported non-persistent Monitor path still plausibly loses a terminal completion when its spawning turn is superseded, and I did not find a current-main fix or canonical issue that owns this exact lifecycle loss.

## What This Changes

Non-persistent Monitor tasks should not lose their completion notification when a new user turn starts before the watched condition fires.

## Best Possible Solution

Define and implement the Monitor lifecycle contract so pending non-persistent monitors either survive until condition or timeout, or reliably deliver a terminal cancellation event when superseded.

## Reproduction Assessment

No, not with high confidence in this read-only pass. Source inspection shows yielded child completions are protected from abort-as-kill while other aborted terminals become killed, which matches the reported failure shape but does not isolate the exact Monitor owner path.

## Solution Assessment

Unclear. The narrowest maintainable fix is likely at the Monitor/subagent lifecycle boundary, but maintainers need to choose whether non-persistent monitors survive turn supersession or emit a guaranteed terminal cancellation event.
