## Summary

Keep open: current main still contains source-level paths that can classify non-yielded aborted monitor-like work as killed and can still treat a pending completion handoff as failed, while the exact non-persistent Monitor lifecycle across a superseding user turn has not been live-reproduced or resolved by a merged canonical fix.

## What This Changes

The issue asks OpenClaw to prevent non-persistent Monitor tasks from silently losing completion when a later user turn supersedes the spawning turn, or to document and surface that cancellation lifecycle.

## Best Possible Solution

Maintainers should define the non-persistent Monitor lifecycle explicitly, then either preserve monitor completion through intervening user turns until condition or timeout, or guarantee a visible terminal cancellation event and documentation when supersession is intentional.

## Reproduction Assessment

No high-confidence live reproduction was run in this read-only review. Source inspection shows a plausible killed path for non-yielded aborted work and a current completion-handoff loss path, but the exact persistent:false Monitor owner path still needs live confirmation.

## Solution Assessment

Unclear. Current main has useful completion retry and reconciliation machinery, and related open PRs address pending direct announces, but the best fix depends on the maintainer decision for non-persistent Monitor survival versus terminal cancellation delivery.
