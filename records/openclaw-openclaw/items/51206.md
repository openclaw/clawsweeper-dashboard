## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Reject private imports before dependency allowlists:** `src/plugins/import-policy.ts:371-372`
  - body: `allowedPackages.has(barePackage)` returns before the private `openclaw/*` guard, so a plugin can declare `openclaw` in its package manifest and still import `openclaw/runtime`. That defeats the boundary this policy is meant to enforce; reject `openclaw` and private subpaths before package allowlists, and add a regression test for that manifest shape.
  - confidence: 0.96
- **[P1] Deny computed module specifiers:** `src/plugins/import-policy.ts:269-274`
  - body: The scanner only matches literal specifiers, so `require("openclaw/" + "runtime")` or `import(getSpecifier())` can pass validation and then execute through the actual loader. A security-sensitive import boundary needs runtime resolution enforcement or AST parsing that fails closed on non-literal module specifiers.
  - confidence: 0.93
- **[P1] Resolve overlay targets before allowing them:** `src/plugins/import-policy.ts:389-396`
  - body: The runtime-overlay exception returns success for any raw `../../../dist/` prefix before normalizing the final path, so `../../../dist/../../...` can widen the allowed filesystem boundary. Resolve the target first and only allow it when the normalized real path stays under the intended canonical dist root.
  - confidence: 0.9
- **[P1] Cover current CLI metadata plugin loads:** `src/plugins/loader.ts:1150-1153`
  - body: The policy is inserted only before the main plugin runtime load in this old branch. Current main also loads plugin CLI metadata modules through the plugin module loader, so a rebase must enforce or explicitly exempt the same boundary there or plugins can still execute unchecked code through that path.
  - confidence: 0.86
- **[P2] Support Node directory package resolution:** `src/plugins/import-policy.ts:318-324`
  - body: Local resolution only tries direct files and `index.*`, but Node also resolves `require("./dir")` through `dir/package.json` `main` or exports. This can reject valid in-plugin imports and break existing plugins even though runtime module resolution would succeed.
  - confidence: 0.82
- **[P2] Support package.json directory resolution for local imports:** `src/plugins/import-policy.ts:318-324`
  - body: Node resolves `require("./dir")` through `./dir/package.json` before falling back to index files, but this resolver only tries direct files and `index.*`. Valid in-package helper directories can therefore be rejected even though runtime module resolution would load them.
  - confidence: 0.86
