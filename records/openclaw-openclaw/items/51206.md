## Review Findings

Overall confidence: 0.87

Full review comments:

- **[P1] Reject OpenClaw private paths before dependency allowlists:** `src/plugins/import-policy.ts:371-372`
  - body: `resolveBarePackageName("openclaw/runtime")` is `openclaw`, so a plugin can declare `openclaw` in its dependencies and return from the allowlist branch before the private `openclaw/*` rejection below ever runs. That bypasses the main boundary this PR is trying to enforce; move the private OpenClaw guard before package allowlists and add a regression test.
  - confidence: 0.97
- **[P1] Reject computed module specifiers before execution:** `src/plugins/import-policy.ts:273-274`
  - body: The policy only collects literal regex matches like `require("...")` and `import("...")`. A plugin can build the same forbidden target with `require("openclaw/" + "runtime")` or another computed specifier, the scan reports no use, and `getJiti` still executes the module afterward, so this is not an enforceable import boundary yet.
  - confidence: 0.93
- **[P1] Validate plugin-local dependency entrypoints:** `src/plugins/import-policy.ts:455`
  - body: For declared bare packages, validation stops at the allowlist check and then `continue`s, so code inside `pluginRoot/node_modules/<declared-package>` is never scanned. A plugin can ship or depend on a declared package whose entrypoint imports private OpenClaw internals, and the loader will execute it despite the advertised policy.
  - confidence: 0.9
- **[P1] Constrain runtime overlay imports after resolution:** `src/plugins/import-policy.ts:389-396`
  - body: `isRuntimeOverlayForwardImport` allows any specifier starting with `../../../dist/`, and this branch returns `null` before resolved-path containment checks run. A runtime overlay import can add traversal after that prefix and bypass the canonical dist-root boundary; resolve and verify the final path instead of accepting the raw prefix.
  - confidence: 0.88
- **[P2] Convert file URLs with fileURLToPath before path checks:** `src/plugins/import-policy.ts:399-400`
  - body: Using `new URL(specifier).pathname` for filesystem decisions leaves percent-encoding intact, while Node converts file URLs to real paths before loading. That mismatch can make encoded dot segments or platform-specific file URL forms pass policy checks for a different path than the runtime loads; use `fileURLToPath` and then apply the existing path-safety helpers.
  - confidence: 0.83
- **[P1] Add an upgrade-safe enforcement path:** `src/plugins/loader.ts:1150-1160`
  - body: This hooks the new validator into every plugin load before module execution, and the PR body acknowledges existing plugins may rely on private imports or undeclared reach-through. As written there is no staged warning mode, migration/doctor path, docs update, or packaged third-party plugin proof, so existing setups can fail closed immediately on upgrade.
  - confidence: 0.9
- **[P3] Remove the release-owned changelog entry:** `CHANGELOG.md:55`
  - body: Repository policy says `CHANGELOG.md` is release-generated and normal PRs should carry release-note context in the PR body or commit message instead. This branch should drop the direct changelog edit and keep the behavior summary in the PR text.
  - confidence: 0.86
