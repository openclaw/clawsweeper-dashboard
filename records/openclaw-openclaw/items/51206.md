## Summary

## What This Changes

The PR adds a loader-time plugin import policy, invokes it before plugin module evaluation, adds loader tests for allowed and blocked imports, and adds a changelog entry.

## Best Possible Solution

Rebase onto current `main`, move enforcement to a shared module-resolution or module-load boundary that covers every plugin entry path, close the bypasses, and include an explicit maintainer-approved compatibility rollout for existing plugins.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path for the main blockers: a plugin can declare `openclaw` and import `openclaw/runtime`, use computed specifiers, or hide imports in plugin-local dependency code while the static policy does not reject those paths.

## Solution Assessment

No. The proposed static scanner is a plausible hardening layer, but the best fix is a shared loader/resolution boundary with explicit bypass coverage and an upgrade-safe rollout.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Reject private OpenClaw imports before allowlists:** `src/plugins/import-policy.ts:371-372`
  - body: `allowedPackages.has(barePackage)` returns before the private `openclaw/*` check, so a plugin can add `openclaw` to its package dependencies and import `openclaw/runtime` while passing the new policy. Move the private OpenClaw rejection ahead of package/self allowlists and add a regression test for that bypass.
  - confidence: 0.96
- **[P1] Reject non-literal module specifiers:** `src/plugins/import-policy.ts:256-279`
  - body: The policy only records literal import/require strings, so code such as `require("openclaw/" + "runtime")` or `import(getSpecifier())` is invisible to validation and still runs through the loader. A security-boundary gate needs to reject or runtime-enforce computed specifiers before plugin evaluation.
  - confidence: 0.9
- **[P1] Scan plugin-local dependency entrypoints:** `src/plugins/import-policy.ts:440-455`
  - body: Declared bare imports are allowed and then skipped with `continue`, so a plugin can declare a local dependency whose entrypoint imports private OpenClaw internals or escapes the package root. Resolve declared dependencies from the importer and validate plugin-root dependency code before treating the graph as safe.
  - confidence: 0.89
- **[P1] Convert file URLs with fileURLToPath:** `src/plugins/import-policy.ts:399-401`
  - body: Using `new URL(specifier).pathname` leaves percent-encoding in the path used for policy decisions, while Node converts file URLs to filesystem paths by decoding them. Use `fileURLToPath()` for `file:` specifiers before normalization so encoded dot segments cannot disagree with runtime resolution.
  - confidence: 0.86
- **[P1] Add an upgrade-safe enforcement path:** `src/plugins/loader.ts:1150-1160`
  - body: This call makes the new policy fail closed for every plugin before module load, but the PR does not include a staged rollout, compatibility mode, docs/contract update, or proof across existing bundled and representative external plugins. Existing plugins that currently load can become hard failures on upgrade.
  - confidence: 0.88
- **[P2] Constrain overlay imports after resolving the target:** `src/plugins/import-policy.ts:63-75`
  - body: The runtime-overlay exception allows any specifier with the raw `../../../dist/` prefix before resolving the rest of the path, so traversal after that prefix can escape the intended canonical dist root. Resolve and verify the target stays inside the canonical dist root instead of allowlisting by string prefix.
  - confidence: 0.82
