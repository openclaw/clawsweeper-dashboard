## Summary

Close as ClawHub/plugin scope: current main already exposes managed TaskFlow authoring through plugin-owned runtime APIs, Webhooks routes, and the Lobster tool path, while core TaskFlow CLI/agent tools remain intentionally inspect-focused.

## What This Changes

The issue requests exposing TaskFlow managed-mode operations to agents through CLI commands, an agent tool, or sessions_spawn integration.

## Best Possible Solution

Keep managed TaskFlow mutation behind plugin-owned tools or trusted automation routes built on the existing session-owned runtime API, and publish any generic taskflow tool through ClawHub rather than OpenClaw core.

## Reproduction Assessment

Not applicable; this is a feature request for a new agent-accessible TaskFlow surface, not a report of broken existing behavior.

## Solution Assessment

Yes for a ClawHub/plugin-scope outcome; current main already exposes the needed managed-flow API to plugin-owned tools and trusted Webhooks routes, while a generic core tool would add a default orchestration surface that VISION.md discourages.
