## Summary

Keep open: this PR is not merge-ready because the NO_PROXY branch catches SSRF policy failures and falls back to a bare direct dispatcher, turning currently blocked localhost/private/internal web_fetch targets into reachable direct requests. Current main and v2026.6.8 document and implement NO_PROXY matches as strict pinned fallback, so local/private access needs maintainer security/product approval rather than this implicit bypass.

## What This Changes

The branch changes src/infra/net/fetch-guard.ts so trusted env-proxy mode detects NO_PROXY matches and creates a direct dispatcher, falling back to a bare direct agent if pinned resolution fails.

## Best Possible Solution

Keep NO_PROXY matches under the strict SSRF contract unless maintainers explicitly approve a documented, tested local-access policy with live web_fetch proof.

## Reproduction Assessment

Yes, from source for the patch defect: resolvePinnedHostnameWithPolicy can reject private/internal targets, and this PR catches that failure and creates a direct agent. I did not run a live proxy/local-service reproduction in this read-only review.

## Solution Assessment

No. The direct fallback is not the best fix because it makes NO_PROXY an implicit SSRF bypass; the safer path is strict fallback by default or an explicit maintainer-approved local-access policy.
