## Summary

Keep open: this PR targets the right shared fetch-guard area, but it is not merge-ready because the new NO_PROXY direct path can catch SSRF policy failures and fall back to a bare direct dispatcher.

## What This Changes

The PR modifies `src/infra/net/fetch-guard.ts` so trusted env-proxy requests excluded by NO_PROXY use a direct dispatcher instead of the env proxy path.

## Best Possible Solution

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction for the patch defect: `resolvePinnedHostnameWithPolicy()` can reject blocked targets, and the PR catches that failure before creating a direct agent. I did not run a live proxy plus local-service web_fetch in this read-only review.

## Solution Assessment

No. The shared `fetch-guard` layer is the right place to settle the behavior, but this patch is not the best fix because it makes NO_PROXY an implicit SSRF bypass rather than preserving strict fallback or implementing a maintainer-approved scoped policy.
