## Summary

Keep this PR open for maintainer review, but the current patch is not merge-ready because the new NO_PROXY direct path can turn SSRF policy failures into bare direct network access.

## What This Changes

The PR changes `src/infra/net/fetch-guard.ts` so trusted environment proxy mode can create a direct dispatcher when `NO_PROXY` matches the target URL.

## Best Possible Solution

Preserve strict DNS-pinned SSRF behavior for NO_PROXY matches; if maintainers want local-service web_fetch access, add an explicit private-network policy surface with docs and tests instead of making NO_PROXY an implicit bypass.

## Reproduction Assessment

Yes for the review finding from source: `resolvePinnedHostnameWithPolicy()` throws on blocked private/internal targets, and the PR catches that failure and creates a direct agent. I did not run a live proxy/local-service reproduction in this read-only review.

## Solution Assessment

No. The maintainable fix is to keep NO_PROXY matches on the strict pinned path or rethrow SSRF policy failures; direct private-network web_fetch access needs an explicit maintainer-approved policy surface.
