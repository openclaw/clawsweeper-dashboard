## Summary

Keep open: this is a real Inworld memory-safety hardening with sufficient terminal proof, but the patch should not merge as-is because the new fixed wire-body cap can still return decoded audio above the configured/default generated-audio cap, and the public SDK export needs maintainer acceptance.

## What This Changes

The PR bounds Inworld TTS success and error response reads, re-exports `readResponseTextSnippet` from the focused response-limit Plugin SDK subpath, adds Inworld regression tests, and updates the SDK surface budget.

## Best Possible Solution

Land after Inworld keeps bounded wire/error reads, enforces the configured decoded-audio cap like sibling speech providers, and maintainers accept the focused SDK helper export.

## Reproduction Assessment

Yes. Current main and `v2026.6.10` use unbounded `response.text()` on the Inworld TTS success/error paths, and the PR body includes terminal proof that the patched path cancels an oversized stream instead of buffering it.

## Solution Assessment

No, not as-is. Bounding at the Inworld HTTP boundary is the right layer, but the patch should also enforce the generated-audio cap after base64 decoding so the fixed 32 MiB wire cap does not bypass the configured/default audio limit that sibling providers honor.
