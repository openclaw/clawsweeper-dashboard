## Summary

Keep open: current main and the latest release still let memory-search-disabled paths suppress or replace dreaming diagnostics, while current tests show dreaming can run with memory search disabled.

## What This Changes

The issue requests clearer doctor/status output that distinguishes embedding memory search from the memory-core dreaming pipeline.

## Best Possible Solution

Keep the embedding-memory-search disabled status, but add an independent memory-core dreaming status path that reports enabled state, scheduled run data, and available recent-run evidence even when embedding memory search is disabled.

## Reproduction Assessment

Yes by source inspection, though I did not run a live doctor command in this read-only review. The current code returns `Memory search disabled` or omits the `dreaming` payload when the memory search manager is missing, while current tests prove dreaming still runs with `memorySearch.enabled: false`.

## Solution Assessment

Yes: the maintainable fix is to decouple dreaming diagnostics from the embedding memory-search manager and make the wording explicit about embedding search versus memory-core dreaming. This should not add config or change dreaming execution semantics.
