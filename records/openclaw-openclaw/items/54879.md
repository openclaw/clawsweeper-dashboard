## Summary

Keep open: current main and v2026.6.6 still do not expose a CDP/IME/native text insertion browser action. The action schema and route accept only the existing action kinds, and current text entry still routes through Playwright type/fill or Chrome MCP fill; the FSB comment is useful supporting evidence, but the public API name, driver support matrix, focus semantics, and security behavior need maintainer judgment.

## What This Changes

Add a browser act action that uses CDP/native text insertion for IME and contenteditable editors like the X composer while preserving existing browser profiles and attached sessions.

## Best Possible Solution

Add a focused browser act action for native/CDP text insertion where supported, keep existing type/fill semantics unchanged, and document/test explicit unsupported behavior for drivers that cannot send native text.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection does show the current browser action set has no CDP/IME insertion action and existing text input routes through fill/type paths.

## Solution Assessment

Unclear for final API shape: the requested browser action layer is plausible, but maintainers still need to choose the public action name, supported-driver contract, focus semantics, and security guards.
