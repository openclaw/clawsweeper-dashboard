## Summary

Keep open: current main and the latest release still lack a CDP `Input.insertText` browser action, while the issue discussion and linked FSB datapoint support this as a real browser automation capability gap that needs a maintainer naming/support decision.

## What This Changes

Request a new browser act action that inserts text through CDP `Input.insertText` for IME/contenteditable editors instead of relying on type, fill, paste, or DOM text tricks.

## Best Possible Solution

Add a documented, additive browser act kind that focuses the target tab or element and uses CDP `Input.insertText` where supported, with explicit unsupported errors for drivers that cannot provide native text insertion.

## Reproduction Assessment

Yes for the source-level capability gap: current main and `v2026.5.20` expose only the existing browser act kinds and route text input through Playwright type/fill or Chrome MCP fill. I did not reproduce the live X/Twitter Korean composer symptom.

## Solution Assessment

Yes directionally: a separate additive native insertion action is safer than changing `type` or `fill` semantics for every caller. Maintainers still need to choose the public name and unsupported-driver behavior.
