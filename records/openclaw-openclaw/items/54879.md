## Summary

Keep open: current main and the latest release still do not expose a CDP/IME text insertion browser action, and the existing text paths still route through Playwright fill/type or Chrome MCP fill. The FSB comment is useful supporting evidence for a separate focused action with explicit unsupported-driver behavior, but the public action name, driver support matrix, focus semantics, and security behavior need maintainer judgment.

## What This Changes

The issue requests a new browser act capability that inserts text through CDP `Input.insertText` for IME/contenteditable editors such as the X composer.

## Best Possible Solution

Add a focused browser act action for native text insertion through CDP where supported, keep existing type/fill semantics unchanged, and document/test explicit unsupported behavior for drivers that cannot send native text.

## Reproduction Assessment

No high-confidence live reproduction of the X/Korean IME failure was established. Source inspection does reproduce the central missing capability: current main has no CDP/IME insert-text act kind and routes existing text input through fill/type paths.

## Solution Assessment

Unclear for final API shape: the requested browser layer is plausible and supported by the CDP contract, but maintainers still need to choose the public action name, driver support contract, focus semantics, and guard behavior.
