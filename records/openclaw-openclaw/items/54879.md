## Summary

Keep open. Current main still lacks a browser `act` kind, request type, route dispatch, CLI command, or public docs surface for CDP-backed IME/native text insertion; the existing text paths still use Playwright `fill`/`type` or Chrome MCP `fill`. The issue is recent, concrete, and the prior ClawSweeper review correctly identified the same remaining browser plugin capability gap.

## What This Changes

Requests an additive browser action that focuses a target element/tab and inserts text through CDP `Input.insertText` for IME/non-Latin contenteditable rich editors.

## Best Possible Solution

Add a documented, additive browser action for native text insertion that focuses the requested target and sends CDP `Input.insertText` for managed/raw CDP profiles, with schema, route normalization, CLI/docs, changelog, and focused regression coverage. Existing-session profiles should preserve the same public contract by implementing an equivalent path only when available or returning a precise unsupported error.

## Reproduction Assessment

Yes for the OpenClaw capability gap: current main rejects any proposed insertion kind at `/act` before normalization, and the only available text paths route through Playwright or Chrome MCP fill. The live X/Twitter editor behavior was not reproduced in this read-only review.

## Solution Assessment

Yes, an additive `act` kind is the narrowest maintainable direction because it preserves existing `type`/`fill` behavior while giving contenteditable/IME workflows a native insertion path. Overloading `type` would make existing semantics less predictable across drivers.
