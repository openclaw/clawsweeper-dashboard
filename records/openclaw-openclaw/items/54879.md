## Summary

Keep open: current main and the latest release still expose only the existing type/fill-style browser input paths and no CDP `Input.insertText` action, so the reported browser automation capability gap remains. The linked FSB datapoint supports the requested direction, but the public action name and driver support policy need maintainer decision.

## What This Changes

Request a new browser act action that inserts text via CDP `Input.insertText` for IME/contenteditable editors where type, fill, or paste paths are unreliable.

## Best Possible Solution

Add a documented, additive browser act kind that focuses the target tab or element and uses CDP `Input.insertText` where supported, with explicit unsupported errors for drivers that cannot provide native insertion.

## Reproduction Assessment

Yes for the source-level capability gap: current main and `v2026.5.19` expose only the existing act kinds and route text input through Playwright type/fill or Chrome MCP fill. I did not reproduce the live X/Twitter Korean composer symptom.

## Solution Assessment

Yes directionally: a separate additive native text insertion action is a better fit than changing `type` or `fill` semantics. Maintainers still need to choose the public name and unsupported-driver behavior.
