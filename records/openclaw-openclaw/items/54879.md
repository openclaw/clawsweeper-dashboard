## Summary

Keep open: current main still lacks a browser action schema entry, normalization path, or managed/existing-session dispatch for CDP/IME text insertion, and the FSB comment reinforces that this should be a separate native insertion action rather than a change to existing type/fill behavior.

## What This Changes

The issue requests a CDP-backed browser text insertion action for rich contenteditable editors, especially IME/non-Latin input in surfaces like the X composer.

## Best Possible Solution

Add a documented additive browser act kind that focuses the target/ref and sends CDP Input.insertText where supported, with explicit unsupported errors for drivers that cannot provide native insertion.

## Reproduction Assessment

Yes for the OpenClaw capability gap: current main has no schema entry, accepted act kind, normalizer branch, or dispatch path for CDP/native text insertion. The specific X/Twitter Korean composer behavior was not live-reproduced here.

## Solution Assessment

Yes in direction, but not ready for automated implementation until maintainers choose naming and support policy. A separate additive act kind is safer than changing existing type/fill semantics for all editors.
