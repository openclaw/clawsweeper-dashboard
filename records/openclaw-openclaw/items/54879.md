## Summary

Keep open: current main still has no accepted browser act kind, schema field, CLI command, managed-browser dispatch, or existing-session dispatch path for CDP/IME-native text insertion. The issue discussion is also useful: the linked FSB data point supports keeping this separate from existing type/fill behavior, but the public action name and driver support policy need maintainer product/API review before implementation.

## What This Changes

The issue requests an additive CDP-backed browser text insertion action for rich contenteditable editors and IME/non-Latin input.

## Best Possible Solution

Add a documented, additive browser act kind that focuses the requested target and uses CDP Input.insertText where the driver can send native text, while preserving type/fill semantics and returning clear unsupported behavior elsewhere.

## Reproduction Assessment

Yes for the OpenClaw capability gap: current main has no schema entry, accepted act kind, normalization branch, managed dispatch, existing-session dispatch, CLI command, or docs surface for CDP/IME-native insertion. The exact X/Twitter Korean composer symptom was not live-reproduced here.

## Solution Assessment

Yes in direction, with product/API details still open: a separate additive act kind is narrower and safer than changing type or fill semantics for every editor. Maintainers still need to choose the public name and unsupported-driver behavior.
