## Summary

Keep open. Current main still has no browser act kind or dispatch path for CDP/IME text insertion, and the fresh FSB comment plus repository search support this as a real additive browser capability rather than an obsolete report.

## What This Changes

Add a CDP-backed browser act action for IME/native text insertion into contenteditable rich editors.

## Best Possible Solution

Introduce a documented additive browser act action that focuses the target/tab and sends CDP Input.insertText where supported, with explicit unsupported behavior or a separate driver path where native text insertion cannot be sent.

## Reproduction Assessment

Yes for the OpenClaw capability gap: current main has no accepted act kind and no dispatch path for CDP/native text insertion. The site-specific X/Twitter Korean editor symptom was not live-reproduced here.

## Solution Assessment

Yes in direction: an additive browser act kind is narrower than changing type/fill semantics and matches the external FSB data point. The unresolved part is maintainer agreement on naming and driver support policy.
