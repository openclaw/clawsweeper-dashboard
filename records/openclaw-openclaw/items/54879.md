## Summary

Keep open: current main and v2026.6.10 still expose only the existing browser act text paths, while the issue and the FSB data point describe a credible rich-editor IME gap that needs maintainer API and live-proof decisions.

## What This Changes

The issue requests a CDP-backed browser act text insertion capability for contenteditable rich editors where type, fill, paste, or DOM insertion do not reliably commit editor state.

## Best Possible Solution

Add a focused native text insertion browser act kind in the browser plugin, using CDP where supported, with explicit unsupported-driver behavior and live rich-editor/IME proof.

## Reproduction Assessment

No high-confidence live reproduction of the X/Korean IME failure was established. Source inspection confirms the central missing capability: current main has no CDP/IME insert-text act kind and routes text insertion through type/fill paths.

## Solution Assessment

Unclear for final API shape: the browser act layer is plausible, and FSB is a useful external data point, but maintainers still need to choose the public action contract, driver support, focus semantics, and guard behavior.
