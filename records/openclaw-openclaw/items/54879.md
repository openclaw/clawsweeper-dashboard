## Summary

Keep open: current main and the latest release still do not expose a CDP/IME text insertion browser action, and the existing text paths still route through Playwright fill/type or Chrome MCP fill. The FSB comment is useful supporting evidence for a separate action, but the public action name, driver support matrix, and security behavior need maintainer judgment.

## What This Changes

Request a new browser act action that inserts text through CDP/IME semantics for rich contenteditable editors such as the X composer.

## Best Possible Solution

Add a focused browser act action for native text insertion through CDP where supported, keep existing type/fill semantics unchanged, and document/test explicit unsupported behavior for drivers that cannot send native text.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection does show the current action set has no CDP/IME insertion action and existing text input routes through fill/type paths.

## Solution Assessment

Unclear for final API shape: the requested layer is plausible and likely right, but maintainers still need to choose the public action name, driver support contract, and guard behavior.
