## Summary

Keep open: current main and the latest release still lack a native CDP/IME text insertion browser action, and the existing text paths continue to route through Playwright fill/type or Chrome MCP fill. The FSB comment is useful supporting context for keeping this separate from type/fill, but the public action name and unsupported-driver contract still need maintainer judgment.

## What This Changes

The issue requests a new browser act action that inserts text through CDP/native-text semantics for IME and rich contenteditable editors.

## Best Possible Solution

Add a narrow browser action that focuses the selected tab or element and uses a native CDP text insertion path where supported, with explicit unsupported-driver errors and regression coverage for rich editors.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the missing OpenClaw capability: current main has no native insert-text action in the schema or server enum, and text still routes through type/fill paths.

## Solution Assessment

Is this the best way to solve the issue? The separate-action direction looks right because it avoids changing type/fill semantics, but maintainers still need to choose the API name and driver support contract.
