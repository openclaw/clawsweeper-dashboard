## Summary

Keep open. Current main and the latest shipped release still lack a browser act kind, typed request, route dispatch, CLI command, or docs surface for CDP-backed IME/native text insertion, so this remains a real additive browser plugin capability request.

## What This Changes

Request a browser act/CLI action that focuses a target and inserts text through CDP Input.insertText for IME/contenteditable editors.

## Best Possible Solution

Introduce a documented additive browser act action that focuses the target element/tab and sends CDP Input.insertText where supported, with precise fallback or unsupported behavior for existing-session profiles.

## Reproduction Assessment

Yes for the OpenClaw capability gap: current main rejects any proposed insertion kind at /act, and the only existing text paths route through Playwright type/fill or Chrome MCP fill. The specific live X/Twitter Korean editor behavior was not reproduced here.

## Solution Assessment

Yes, an additive browser act kind is the narrowest maintainable direction because it preserves existing type/fill behavior while adding an IME/native insertion path. The unresolved part is the public naming and driver policy, not whether the current implementation already handles it.
