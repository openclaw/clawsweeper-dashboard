## Summary

Current main still has no browser action, schema entry, or dispatch path for CDP/IME/native text insertion, so the reported contenteditable editor capability gap remains valid and should not be closed as cleanup.

## What This Changes

The issue requests an additive browser act action that inserts text via CDP `Input.insertText` for rich contenteditable editors where paste, type, and fill do not reliably commit IME or non-Latin text.

## Best Possible Solution

Add a documented, additive browser act kind that focuses the target tab/element and uses CDP `Input.insertText` where supported, with explicit unsupported behavior for drivers that cannot provide native insertion.

## Reproduction Assessment

Yes for the OpenClaw capability gap: current main has no schema entry, accepted act kind, or dispatch path for CDP/native text insertion. The X/Twitter Korean composer symptom itself was not live-reproduced here.

## Solution Assessment

Yes in direction, but maintainer agreement is still needed on public action naming and driver-support policy. A separate additive browser act kind is narrower and safer than changing existing `type` or `fill` semantics for all editors.
