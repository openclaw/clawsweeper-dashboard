## Summary

Keep open: current main and v2026.5.22 still expose only the existing browser act input paths and route text through Playwright type/fill or Chrome MCP fill, so the requested CDP `Input.insertText` capability remains missing. The linked FSB comment supports keeping native insertion separate from type/fill, but the public action name and unsupported-driver behavior need maintainer decision.

## What This Changes

Request a new CDP/IME text insertion action in the browser act surface for rich contenteditable editors where type, fill, or paste do not reliably commit text.

## Best Possible Solution

Add a documented, additive browser act kind that focuses the target tab or element and uses CDP `Input.insertText` where supported, with explicit unsupported errors for drivers that cannot provide native insertion.

## Reproduction Assessment

Yes for the source-level capability gap: current main and v2026.5.22 expose no CDP/native insertion act kind and route text through Playwright type/fill or Chrome MCP fill. I did not reproduce the live X/Twitter Korean composer symptom.

## Solution Assessment

Yes directionally: a separate additive native insertion action is safer than changing `type` or `fill` semantics for all callers. Maintainers still need to choose the public name and unsupported-driver behavior.
