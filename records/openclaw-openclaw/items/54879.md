## Summary

Current main still lacks a browser action, act-kind validation path, or dispatch implementation for CDP/IME-style text insertion, so this remains a valid feature request rather than conservative cleanup.

## What This Changes

The issue requests an additive browser action that inserts text through CDP/IME-style native text insertion for contenteditable editors where paste, type, or fill can leave editor state stale.

## Best Possible Solution

Add a documented, additive browser act kind that focuses the target tab or element and uses CDP Input.insertText where supported, while returning an explicit unsupported error for drivers that cannot provide native insertion.

## Reproduction Assessment

Yes for the OpenClaw capability gap: current main has no schema entry, accepted act kind, client request type, normalization branch, or managed/existing-session dispatch for CDP/native text insertion. The live X/Twitter Korean composer symptom was not reproduced in this read-only review.

## Solution Assessment

Yes directionally, but it should stay a separate additive action rather than changing existing type or fill semantics. Maintainers still need to decide the public action name and unsupported-driver behavior before a fix PR is safe.
