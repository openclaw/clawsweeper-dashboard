## Summary

Keep open. Current main still lacks a browser `act` kind, request type, route dispatch, CLI command, or docs surface for CDP-backed IME/native text insertion, and the remaining behavior is a real additive browser plugin capability gap rather than an obsolete report.

## What This Changes

Request to add an additive browser action that inserts text via CDP `Input.insertText` for IME/contenteditable editors while preserving existing browser profile workflows.

## Best Possible Solution

Add a documented additive browser `act` action that focuses the target and sends CDP `Input.insertText` on managed/raw CDP profiles, with existing-session support only if the driver can preserve equivalent semantics or a precise unsupported error otherwise.

## Reproduction Assessment

Yes for the OpenClaw capability gap: current main rejects any proposed insertion kind at `/act`, and the only text paths route through Playwright `fill`/`type` or Chrome MCP `fill`. The specific live X/Twitter Korean editor behavior was not reproduced in this read-only review.

## Solution Assessment

Yes, an additive `act` kind is the narrowest maintainable direction because it preserves existing `type`/`fill` semantics while adding a native insertion path for rich editors. The remaining choice is the public action name and exact driver policy.
