## Summary

Keep open: current main and v2026.6.8 still do not expose a CDP/IME native text insertion browser action, while the issue and FSB comment describe a real browser automation gap that needs maintainer API and safety decisions before implementation.

## What This Changes

Add a browser `act` action that focuses a target tab/element and inserts text through CDP `Input.insertText` for IME/contenteditable rich-editor workflows.

## Best Possible Solution

Add a focused native text insertion browser action via CDP where supported, keep existing type/fill semantics unchanged, and document/test explicit unsupported behavior for drivers that cannot send native text.

## Reproduction Assessment

No high-confidence live reproduction of the X/Korean IME failure was established. Source inspection does reproduce the central missing capability: current main has no CDP/IME insert-text act kind and routes existing text input through type/fill paths.

## Solution Assessment

Unclear for final API shape: the browser layer is plausible and supported by the CDP contract, but maintainers still need to choose the public action name, driver support contract, focus semantics, and guard behavior.
