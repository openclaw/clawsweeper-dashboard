## Summary

Keep open. Current main and the latest release still lack a separate CDP/IME text insertion browser action, and the latest discussion adds relevant external implementation experience supporting a distinct native insertion path rather than changing `type` or `fill`.

## What This Changes

Add a CDP-backed browser act text insertion action for IME/contenteditable editors such as the X composer.

## Best Possible Solution

Introduce a documented additive browser act kind that focuses the target tab/element and sends CDP `Input.insertText` where supported, with explicit unsupported behavior for drivers that cannot send native text.

## Reproduction Assessment

Yes for the OpenClaw capability gap: source inspection shows proposed insertion kinds are rejected and the only current text paths are Playwright type/fill or Chrome MCP fill. The site-specific X/Twitter Korean editor behavior was not live-reproduced here.

## Solution Assessment

Yes, a separate browser act kind is the narrowest maintainable direction because it preserves existing `type` and `fill` semantics while adding a native insertion path. The remaining decision is the public action name and driver policy.
