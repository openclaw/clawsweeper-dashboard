## Summary

Keep open. Current main has the Baileys media hotfix and root/package runtime-deps work, but the remaining external staged runtime-deps reuse path can still accept an unpatched Baileys install as materialized without checking the `messages-media.js` hotfix content before WhatsApp loads.

## What This Changes

The issue requests hardening WhatsApp outbound media upload so Baileys encrypted temp-file `ENOENT` failures cannot crash the Gateway, with preserved dispatcher compatibility, configurable timeout behavior, and consistent runtime-deps selection.

## Best Possible Solution

Keep this open until OpenClaw verifies and repairs the Baileys media hotfix in every selected external bundled runtime-deps root before WhatsApp runtime loading, or upgrades to an upstream Baileys version that removes the encrypted temp-file race. The implementation should keep core runtime-deps behavior generic where possible and keep WhatsApp/Baileys-specific policy owner-scoped.

## Reproduction Assessment

Yes. The issue gives concrete CLI reproduction steps and crash logs, and a source-level reproduction of the remaining gap is to create a generated external runtime-deps root with matching Baileys version/entry files but unpatched `messages-media.js`, which current materialization accepts without a hotfix-content check.

## Solution Assessment

No. The current package-root postinstall hotfix is useful but incomplete because it does not prove that the selected external staged runtime-deps root contains the same Baileys encrypted-media and dispatcher hotfixes.
