## Summary

Keep open. Current main contains the Baileys media hotfix and a package-root postinstall patcher, but the external staged runtime-deps reuse path can still select a matching Baileys root without verifying that `messages-media.js` has the encrypted-stream and dispatcher hotfixes. The linked WAHA item is an optional alternate channel direction, not a substitute for hardening the bundled Baileys-backed WhatsApp path.

## What This Changes

The issue requests hardening WhatsApp outbound media uploads so Baileys encrypted temp-file upload failures cannot crash the Gateway, while preserving dispatcher bypass, configurable upload timeout, and reliable runtime-deps patch selection.

## Best Possible Solution

Add a narrow verifier or repair step so every selected staged runtime-deps root used for WhatsApp is proven to contain the Baileys media hotfix before runtime loading, or upgrade Baileys to an upstream version that removes the race and make the materialization check prove that selected roots contain that version/content.

## Reproduction Assessment

Yes. The issue has concrete CLI reproduction and crash logs, and the source-level reproduction is clear: create a matching external staged root for Baileys `7.0.0-rc.9` with unpatched `messages-media.js`; current materialization and sibling reuse can accept it.

## Solution Assessment

No. The current package-root postinstall hotfix is useful but not sufficient because it does not verify or repair the actual external staged root that may be selected before WhatsApp loads.
