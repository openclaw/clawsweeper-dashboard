## Summary

Keep open. Current main has the Baileys media hotfix and dispatcher guard for installed package roots, but the bundled runtime-deps reuse path can still select a compatible sibling external root using only dependency metadata/version/entry-file checks, without proving that the reused Baileys `messages-media.js` contains the hotfix before WhatsApp loads it.

## What This Changes

The issue asks OpenClaw to prevent WhatsApp outbound media upload crashes from Baileys encrypted temp-file ENOENTs, preserve fetch dispatcher compatibility, make upload timeout configurable, and clarify staged runtime-deps reuse.

## Best Possible Solution

Add a narrow verifier or repair step so every selected external staged runtime-deps root used for WhatsApp/Baileys media upload is checked for the encrypted-stream finish hotfix and dispatcher guard before reuse, or is repaired/reinstalled before the WhatsApp runtime loads it. Keep the generic runtime-deps boundary clean by reusing/extracting the existing hotfix logic rather than switching uploads to full-buffer reads or hard-coding broad WhatsApp policy in unrelated core paths.

## Reproduction Assessment

Yes. The issue provides concrete CLI steps and crash logs, and current source gives a high-confidence source reproduction shape: a sibling external staged root with matching Baileys metadata but unpatched `messages-media.js` can pass materialization and be reused.

## Solution Assessment

Yes, a selected-root verifier/repair is the narrowest maintainable fix. The reporter's full-buffer workaround avoids the race locally, but current shipped fixes intentionally keep upload streaming to avoid large-media RSS/OOM regressions, so the better OpenClaw fix is to make the selected runtime copy provably patched.
