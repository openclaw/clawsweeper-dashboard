## Summary

Keep open. Current main has the package-root Baileys media hotfix and postinstall patcher, but the staged runtime-deps reuse path still treats a prior external root as materialized from dependency metadata, installed versions, and entry files alone; it does not verify that the selected Baileys `messages-media.js` contains the media hotfix before WhatsApp loads.

## What This Changes

The issue reports a WhatsApp outbound media upload Gateway crash from Baileys encrypted temp-file reads and asks for crash containment, a reliable Baileys upload hotfix, dispatcher handling, configurable timeout, and staged runtime-deps selection hardening.

## Best Possible Solution

Add a narrow verification or repair step so any selected external staged runtime-deps root containing Baileys is checked for the encrypted-media and dispatcher hotfix before WhatsApp runtime loading, or rejected/repaired if it cannot be proven safe.

## Reproduction Assessment

Yes. The issue provides CLI reproduction steps and crash logs; a source-level reproduction is to seed a compatible previous external runtime-deps root with matching Baileys metadata but unpatched `messages-media.js`, which current reuse logic can accept because materialization checks specs/version/entry files only.

## Solution Assessment

No. The current postinstall/package-root hotfix is useful but not sufficient because a reused external staged root can bypass it; the safer fix is to make the selected root itself prove or receive the Baileys hotfix before runtime use.
