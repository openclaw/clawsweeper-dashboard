## Summary

Keep open. Current main has the Baileys media hotfix and package-root postinstall patcher, but the staged runtime-deps reuse path can still accept a matching external root without verifying that Baileys `messages-media.js` contains the hotfix before WhatsApp loads.

## What This Changes

The issue asks OpenClaw to prevent WhatsApp outbound media sends from crashing the Gateway on Baileys encrypted temp-file `ENOENT` failures and to make the related upload/runtime-deps behavior robust.

## Best Possible Solution

Add a narrow verifier/repair step so every selected external staged runtime-deps root is checked for the Baileys media hotfix before WhatsApp runtime loading, or upgrade to an upstream Baileys version that removes the race and then verify all staged roots select that version.

## Reproduction Assessment

Yes. The issue gives concrete CLI reproduction steps and crash logs; a focused source-level reproduction is to create a generated external runtime-deps root with matching Baileys package metadata but unpatched `messages-media.js`, which current materialization can accept and reuse.

## Solution Assessment

No. The current package-root patch/postinstall path is useful but not sufficient because it does not prove that the selected external staged runtime-deps root contains the same Baileys encrypted-media and dispatcher fixes.
