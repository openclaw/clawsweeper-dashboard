## Summary

Keep open. Current main carries the Baileys media patch/hotfix machinery, but the staged external runtime-deps path still accepts compatible roots by manifest/spec/version/entry-file checks without proving the selected Baileys `messages-media.js` contains the encrypted-stream finish and dispatcher hotfixes. That leaves the reporter's stale-runtime-copy crash scenario source-plausible and narrow enough for a focused repair.

## What This Changes

The issue requests hardening WhatsApp outbound media uploads so Baileys encrypted temp-file `ENOENT` failures cannot crash the gateway, while preserving the dispatcher bypass, improving timeout behavior, and handling stale staged runtime-deps copies.

## Best Possible Solution

Ensure any staged runtime-deps root used for WhatsApp either comes from a dependency flow that applies the Baileys media patch or is content-verified and repaired before the WhatsApp runtime can load it. If upstream Baileys ships the equivalent fix, upgrade and remove the local hotfix path instead of carrying duplicate patch policy.

## Reproduction Assessment

Yes. The issue gives a live CLI reproduction and crash logs, and the remaining current-main gap can be reproduced with a generated external runtime-deps root containing matching `@whiskeysockets/baileys` version/entry files but an unpatched `lib/Utils/messages-media.js`.

## Solution Assessment

No. The current package-root pnpm patch and postinstall helper are useful but incomplete because external staged runtime-deps reuse can still accept an unverified Baileys install; the safer path is to verify or repair the selected staged root before materialization/load succeeds.
