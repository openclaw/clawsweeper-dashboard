## Summary

Keep open. Current main has the Baileys media hotfix helper and newer runtime-deps materialization checks, but the external staged runtime-deps reuse path still accepts compatible package versions and entry files without verifying that the selected Baileys `messages-media.js` contains the OpenClaw hotfix before WhatsApp loads.

## What This Changes

The issue requests hardening WhatsApp outbound media uploads so Baileys encrypted temp-file `ENOENT` failures cannot crash the Gateway, while preserving dispatcher bypass behavior and improving runtime-deps/timeout handling.

## Best Possible Solution

Add a targeted Baileys hotfix verification/repair step for every selected external bundled runtime-deps root before WhatsApp imports Baileys, or invalidate and reinstall roots that lack the encrypted-stream and dispatcher safeguards. Keep the generic runtime-deps materialization contract package-based, with WhatsApp/Baileys-specific checks isolated to the hotfix owner path and covered by a stale-external-root test.

## Reproduction Assessment

Yes. The issue provides a concrete CLI media-send reproduction and crash logs, and the remaining source-level reproduction is to create a compatible external staged runtime-deps root containing Baileys `7.0.0-rc.9` and entry files but an unpatched `messages-media.js`; current materialization would still accept that shape.

## Solution Assessment

No. The current package-root postinstall hotfix and generic runtime-deps entry-file verification are useful but incomplete for reused external staged roots; the narrower maintainable fix is to verify or repair the Baileys hotfix content at runtime-deps root selection before WhatsApp loads it.
