## Summary

Keep open. Current main contains the Baileys media hotfix and dispatcher guard, but the staged runtime-deps reuse path can still accept a sibling external root based on dependency metadata and entry files without verifying that Baileys `messages-media.js` contains the hotfix before WhatsApp loads.

## What This Changes

The issue requests hardening WhatsApp outbound media uploads so Baileys encrypted temp-file `ENOENT` failures cannot crash the Gateway across staged dependency roots.

## Best Possible Solution

Add a narrow verifier/repair step so every selected external staged runtime-deps root for `@whiskeysockets/baileys` is checked for the encrypted-stream finish hotfix and dispatcher guard before reuse, or is repaired/reinstalled before WhatsApp runtime loading.

## Reproduction Assessment

Yes. The issue includes concrete CLI media-send steps and gateway crash logs, and current source provides a focused reproduction shape: a matching external staged root with Baileys metadata but unpatched `messages-media.js` can pass materialization and be reused.

## Solution Assessment

Yes. A selected-root verifier/repair is the narrowest maintainable fix because current main's package-root patcher is useful but does not prove the actual reused external root contains the same Baileys hotfix.
