## Summary

Keep open. Current main has the Baileys media hotfix and newer pruning for unknown/legacy runtime-deps roots, but it still accepts materialized external runtime-deps roots based on dependency specs, installed versions, and entry files only, so a same-hash staged root with unpatched Baileys media code can still be reused before WhatsApp loads.

## What This Changes

The issue asks OpenClaw to prevent WhatsApp outbound media sends from crashing the Gateway on Baileys encrypted temp-file ENOENT races and to ensure staged runtime-deps copies carry the media-upload hotfix.

## Best Possible Solution

Keep this open until OpenClaw verifies or repairs the Baileys media hotfix in every selected/generated/reused bundled runtime-deps root before WhatsApp runtime loading, or upgrades to an upstream Baileys release that removes the encrypted temp-file race without local patching.

## Reproduction Assessment

Yes. The issue provides concrete CLI steps and crash logs; source inspection also gives a focused reproduction path: create a version-compatible external runtime-deps root containing unpatched Baileys `messages-media.js` and observe that current materialization/reuse checks accept it without hotfix verification.

## Solution Assessment

No. The current package-root hotfix plus root pruning is useful but incomplete because materialized same-hash external runtime-deps roots can still be trusted without proving the Baileys encrypted-media and dispatcher hotfixes are present.
