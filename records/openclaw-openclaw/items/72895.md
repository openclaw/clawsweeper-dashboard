## Summary

Keep open. Current main contains the Baileys media hotfix and dispatcher guard, but the runtime-deps loader can still reuse a sibling external staged root based on dependency metadata, installed version, and entry-file checks without verifying that the selected Baileys `messages-media.js` contains the hotfix before WhatsApp loads.

## What This Changes

The issue reports a WhatsApp outbound media gateway crash on Baileys temporary encrypted media files and asks OpenClaw to prevent the crash, preserve the fetch dispatcher guard, make upload timeout configurable, and avoid stale staged runtime-deps copies.

## Best Possible Solution

Add a narrow verifier/repair step so every selected external staged root containing `@whiskeysockets/baileys` is checked for the encrypted-stream finish hotfix and dispatcher guard before reuse; if the check fails, reinstall or repair that root before WhatsApp runtime loading.

## Reproduction Assessment

Yes. The issue has concrete CLI steps and crash logs, and current source gives a high-confidence synthetic reproduction: an external sibling staged root with matching Baileys metadata but unpatched `messages-media.js` can pass materialization and be reused.

## Solution Assessment

Yes. A selected-root verifier/repair is the narrowest maintainable fix because postinstall/package-root patching already exists, while the unresolved risk is choosing an older external root whose content was never verified.
