## Summary

Keep open: the branch targets a valid linked bug, but it is not merge-ready because the new config-only gate can expose `sessions_spawn` in embedded TUI without a proven execution path, misses documented per-agent allowlists, and the submitted proof is code inspection rather than real TUI behavior.

## What This Changes

The PR changes `src/agents/openclaw-tools.ts` so embedded tool construction includes `sessions_spawn` when `agents.defaults.subagents.allowAgents` is a non-empty array.

## Best Possible Solution

Gate embedded `sessions_spawn` on the effective per-agent/default allowlist and a proven execution path: gateway-bound embedded runs can expose it, while local mode needs real dispatch support or explicit gating/docs.

## Reproduction Assessment

Yes, by source inspection: current `main` sets local TUI into embedded mode and excludes `sessions_spawn` unless `allowGatewaySubagentBinding` is true. I did not run a live Windows/npm TUI reproduction.

## Solution Assessment

No. `createOpenClawTools` is a plausible materialization layer, but the condition should mirror effective requester policy and a supported spawn execution context rather than exposing a Gateway-dependent tool from config alone.
