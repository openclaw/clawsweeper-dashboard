## Summary

Keep open: the core cache design is useful, but the PR is not merge-ready because an exported fingerprint shim can turn uncacheable auth state into a reusable catalog cache key.

## What This Changes

The PR adds content-based models-config cache invalidation, canonical SQLite auth-store fingerprinting, provider-scoped `targetProvider`/`targetModel` short-circuiting, plugin catalog validation, and regression/proof scripts around those paths.

## Best Possible Solution

Preserve the PR's fail-closed cache design across all exported fingerprint callers by making uncacheable auth state uncacheable for model-catalog cache keys too, then land only after the models-config owner accepts the compatibility and security-boundary semantics.

## Reproduction Assessment

Yes, from source: when auth state is uncacheable, the exported fingerprint helper returns a stable per-agent sentinel, and both catalog callers use that value for cache lookup before revalidation. I did not run a live failing test because this review is read-only.

## Solution Assessment

No: the core `ensureOpenClawModelsJson` fail-closed path is the right direction, but the exported fingerprint shim reopens a stale catalog-cache hit path for model-catalog callers.
