## Summary

Keep this PR open because the models-config work is still useful, but latest head has a blocking cache-key bug in the restored source-fingerprint shim: unreadable auth state can still key persisted model-catalog caches outside the central fail-closed ensure path.

## What This Changes

The branch replaces mtime-based models-config readiness with content-based auth/models/plugin-catalog outcomes, adds provider-scoped targetProvider/targetModel short-circuiting, threads caller hints, and adds focused tests plus runtime proof scripts.

## Best Possible Solution

Fix the uncacheable source-fingerprint contract so unreadable auth state cannot key persisted catalog caches, add caller-level regression coverage, then have the models-config owner review the exact-head compatibility and security tradeoff before merge.

## Reproduction Assessment

Yes. Source inspection shows an unreadable auth store makes buildModelsJsonSourceFingerprint return a stable `uncacheable:${agentDir}` value, and two catalog callers use that value for persisted cache reads before the fail-closed ensure path can run.

## Solution Assessment

No. The central content-outcome and scoped-cache design is the right direction, but the compatibility shim must expose non-cacheability instead of a stable cache key, or its callers must skip persisted cache reads and writes for that outcome.
