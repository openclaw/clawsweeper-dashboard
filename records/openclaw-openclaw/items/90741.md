## Summary

Keep this PR open. The overall cache-boundary direction is useful and not implemented on current main, but the PR still has a concrete auth/provider cache bug: `loadModelCatalog` can keep and later write with a stale pre-ensure catalog key when the post-ensure source becomes non-cacheable.

## What This Changes

## Best Possible Solution

Fix the stale catalog-key transition and add the missing regression, then keep owner review for the broader cache/fail-closed semantics on a mergeable green head.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: initial source fingerprint is cacheable, `prepareOpenClawModelsJsonSource` returns `cacheable: false`, and `loadModelCatalog` keeps the old `catalogKey` because the update block only runs for a truthy refreshed key.

## Solution Assessment

No. The overall design is plausible, but the current implementation must clear the persisted catalog key when the post-ensure source becomes non-cacheable; otherwise it violates the PR’s own cache-safety contract.
