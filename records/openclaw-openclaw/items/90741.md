## Solution Assessment

No, not yet: the main cache design is the right layer, but uncacheable auth state must not become a reusable persisted catalog cache key. The safer fix is to carry a cacheable/non-cacheable result to callers or skip catalog cache reads and writes when no trustworthy fingerprint exists.

- **[P1] Do not reuse catalog cache keys for uncacheable auth state:** `src/agents/models-config.ts:1857`
  - body: `buildModelsJsonSourceFingerprint` maps an unreadable or oversized auth store to the stable string `uncacheable:${agentDir}`. Please make this state non-cacheable for the persisted catalog cache rather than returning a reusable key.
