## Summary

Keep this PR open for revision: it targets a source-reproducible cron reminder delivery gap, but the current draft/conflicting patch misses current delivery-context routing, can promote explicit nested no-delivery jobs, and lacks real behavior proof.

## What This Changes

The PR changes cron add handling to promote reminder-like chat-created `systemEvent` jobs into isolated `agentTurn` announce jobs and adds cron-tool regression tests.

## Best Possible Solution

Land a rebased cron-tool fix that promotes only implicit active-chat reminders while preserving explicit internal/no-delivery jobs, supported wrappers, currentDeliveryContext routing, and existing explicit delivery fields.

## Reproduction Assessment

Yes, source-level. Current main defaults implicit `systemEvent` cron jobs to `sessionTarget="main"` and only infers announce delivery for `agentTurn` jobs, so a chat reminder created as a system event can remain internal with `deliveryStatus="not-requested"`.

## Solution Assessment

No, not as written. The fix direction is narrow, but the patch should use currentDeliveryContext-first inference, preserve explicit delivery fields, and inspect the unwrapped job input before promoting.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve current delivery context for promoted reminders:** `src/agents/tools/cron-tool.ts:451`
  - body: The promotion path computes `inferredDelivery` only from `opts.agentSessionKey`. Current main infers from `currentDeliveryContext` first and supports context-only delivery, so promoted Matrix/Telegram reminders can lose provider casing, accountId, or threadId, and can fail to infer delivery when no session key is present.
  - confidence: 0.9
- **[P2] Respect explicit fields inside supported job wrappers:** `src/agents/tools/cron-tool.ts:239`
  - body: This guard only checks the outer raw job for `sessionTarget` or `delivery`. Normalization accepts wrapped inputs like `{ data: { ... } }`, so a nested `sessionTarget: "main"` or `delivery: { mode: "none" }` reminder can still be promoted into a user-visible isolated `agentTurn`; check the unwrapped job before promotion.
  - confidence: 0.86
- **[P2] Keep explicit delivery fields ahead of inferred defaults:** `src/agents/tools/cron-tool.ts:518-519`
  - body: The PR merges `{ ...delivery, ...inferredDelivery }`, so inferred accountId/threadId values can overwrite explicit delivery fields when only the target is missing. Current main preserves explicit delivery details while filling inferred targets; keep inferred defaults first and explicit delivery second.
  - confidence: 0.83
