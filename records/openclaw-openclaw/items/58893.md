## Summary

Keep open: this PR targets a real cron reminder delivery bug, but the patch is still draft/conflicting, lacks real behavior proof, and introduces routing/internal-delivery regressions around current delivery context, wrapped inputs, and explicit delivery precedence.

## What This Changes

The PR promotes reminder-like chat-created cron `systemEvent` jobs into isolated `agentTurn` announce jobs and adds cron-tool regression tests.

## Best Possible Solution

Land a rebased cron-tool fix that promotes only implicit active-chat reminders while preserving explicit internal/no-delivery jobs, supported wrappers, currentDeliveryContext routing, and explicit delivery fields.

## Reproduction Assessment

Yes, source-level. Current main defaults implicit `systemEvent` cron jobs to `sessionTarget="main"` and only infers announce delivery for `agentTurn` jobs, so a chat-created reminder represented as a system event can remain internal with delivery not requested.

## Solution Assessment

No, not as written. The fix direction is narrow, but the patch should infer from current delivery context first, inspect the unwrapped job input before promoting, and preserve explicit delivery fields over inferred defaults.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve current delivery context for promoted reminders:** `src/agents/tools/cron-tool.ts:451`
  - body: The promotion path computes `inferredDelivery` only from `opts.agentSessionKey`. Current main infers from `currentDeliveryContext` first and supports context-only delivery, so promoted Matrix/Telegram reminders can lose provider casing, accountId, or threadId, and can fail to infer delivery when no session key is present.
  - confidence: 0.9
- **[P2] Respect explicit fields inside supported job wrappers:** `src/agents/tools/cron-tool.ts:239`
  - body: This guard checks only the outer raw job for `sessionTarget` or `delivery`. Normalization accepts wrapped inputs like `{ data: { ... } }`, so a nested `sessionTarget: "main"` or `delivery: { mode: "none" }` reminder can still be promoted into a user-visible isolated `agentTurn`; check the unwrapped job before promotion.
  - confidence: 0.86
- **[P2] Keep explicit delivery fields ahead of inferred defaults:** `src/agents/tools/cron-tool.ts:518-519`
  - body: The PR merges `{ ...delivery, ...inferredDelivery }`, so inferred accountId/threadId values can overwrite explicit delivery fields when only the target is missing. Current main preserves explicit delivery details while filling inferred targets; keep inferred defaults first and explicit delivery second.
  - confidence: 0.83
