## Summary

Keep open. Current main still leaves systemEvent cron jobs on the main/internal path, so the delivery problem this PR targets is not implemented there, but the PR branch is draft, dirty against current main, and its new promotion path is stale relative to current delivery-context handling. The SonicBotMan comment adds useful cron-operations context, but it describes scheduler drift rather than the reminder delivery bug this patch changes.

## What This Changes

The PR changes cron tool add handling to promote active-chat systemEvent reminders without explicit sessionTarget or delivery into isolated agentTurn announce jobs and adds cron-tool tests for promotion, explicit main reminders, and context history.

## Best Possible Solution

Implement chat reminder delivery as a narrow cron-tool behavior that converts only active-chat, implicit reminder creations into isolated agentTurn announce jobs while preserving explicit sessionTarget, delivery.mode="none", webhook delivery, account/thread IDs, and currentDeliveryContext. The final patch should include regression coverage for Matrix casing/thread context and keep docs/tool guidance aligned with the chosen default.

## Reproduction Assessment

Yes. Source inspection provides a high-confidence path: create a cron.add from an active chat with payload.kind="systemEvent" and no explicit delivery; current main normalizes it to sessionTarget="main" and never enters the agentTurn delivery inference block.

## Solution Assessment

No, not as written. The PR's direction is plausible, but the implementation should be rebased onto current main and use inferDeliveryFromContext(opts.currentDeliveryContext) before session-key inference so promoted reminders keep the exact live route.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Preserve current delivery context when promoting reminders:** `src/agents/tools/cron-tool.ts:451`
  - body: The new promotion path builds `inferredDelivery` only from `agentSessionKey`, so promoted reminders from active chats can lose the preserved current route. Current main intentionally prefers `currentDeliveryContext` because Matrix room IDs are case-sensitive and threads/accounts may not round-trip through the internal session key. Use `inferDeliveryFromContext(opts.currentDeliveryContext) ?? inferDeliveryFromSessionKey(...)` here and add a promoted-reminder regression test for that case.
