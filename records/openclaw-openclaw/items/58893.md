## Summary

Keep open: current main still has a source-level chat reminder delivery gap, but this draft PR is conflicting, lacks required real behavior proof, and introduces correctness and privacy regressions around delivery context and explicit internal/no-delivery jobs.

## What This Changes

The PR promotes implicit chat-created cron `systemEvent` reminders into isolated `agentTurn` announce jobs and adds cron-tool regression tests.

## Best Possible Solution

Land a rebased cron-tool fix that promotes only implicit active-chat reminders while preserving explicit internal/no-delivery jobs, supported wrappers, `currentDeliveryContext` routing, and explicit delivery fields, with real chat proof.

## Reproduction Assessment

Yes, source-level. Current main defaults implicit `systemEvent` cron jobs to `sessionTarget="main"` and only infers announce delivery for `agentTurn` jobs, so a chat-created reminder represented as a system event can remain internal with delivery not requested.

## Solution Assessment

No, not as written. The fix direction is narrow, but the patch should infer from current delivery context first, inspect the unwrapped job input before promotion, and preserve explicit delivery fields over inferred defaults.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve current delivery context for promoted reminders:** `src/agents/tools/cron-tool.ts:451`
  - body: The promotion path computes `inferredDelivery` only from `opts.agentSessionKey`. Current main infers from `currentDeliveryContext` first, so promoted Matrix/Telegram reminders can lose provider casing, accountId, or threadId, and can fail when only context delivery is available.
  - confidence: 0.9
- **[P2] Respect explicit fields inside supported job wrappers:** `src/agents/tools/cron-tool.ts:239`
  - body: This guard checks only the outer raw job for `sessionTarget` or `delivery`. Normalization accepts wrappers like `{ data: { ... } }`, so a nested `sessionTarget: "main"` or `delivery: { mode: "none" }` reminder can still be promoted into a user-visible isolated `agentTurn`.
  - confidence: 0.86
- **[P2] Keep explicit delivery fields ahead of inferred defaults:** `src/agents/tools/cron-tool.ts:518-519`
  - body: The PR merges `{ ...delivery, ...inferredDelivery }`, so inferred accountId/threadId values can overwrite explicit delivery fields when only the target is missing. Current main fills inferred defaults first and keeps explicit delivery details second.
  - confidence: 0.83
