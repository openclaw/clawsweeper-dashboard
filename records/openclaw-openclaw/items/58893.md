## Summary

Keep open, but not merge-ready: source review confirms the reminder-delivery problem is plausible on current main, while this patch can misroute delivery, expose explicitly internal jobs, and still lacks real chat proof.

## What This Changes

The PR promotes reminder-like chat-created systemEvent cron jobs without explicit delivery metadata into isolated agentTurn announce jobs and adds cron-tool tests.

## Best Possible Solution

Rework or replace this with a rebased cron-tool fix that promotes only truly implicit active-chat reminders, preserves explicit internal/no-delivery and delivery fields, uses current delivery context, and includes real chat proof.

## Reproduction Assessment

Yes, at source level. Current main defaults missing-sessionTarget systemEvent cron jobs to main and only infers announce delivery for agentTurn jobs, so a chat-created reminder represented as a systemEvent can remain internal with delivery not requested.

## Solution Assessment

No, not as written. The fix should use current delivery context, inspect explicit fields after supported wrapper normalization, preserve explicit delivery fields over inferred defaults, and include real chat proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve current delivery context for promoted reminders:** `src/agents/tools/cron-tool.ts:451`
  - body: The promotion path infers delivery only from opts.agentSessionKey. Current main resolves currentDeliveryContext first, so promoted Matrix, Telegram, LINE, or threaded reminders can lose provider casing, accountId, threadId, or fail when only live context delivery is available.
  - confidence: 0.9
- **[P2] Respect explicit fields inside supported job wrappers:** `src/agents/tools/cron-tool.ts:239`
  - body: This guard checks only the outer raw job for sessionTarget or delivery, but normalization accepts wrappers like { data: { ... } } and { job: { ... } }. A nested sessionTarget="main" or delivery={mode:"none"} reminder can still be promoted into a user-visible job.
  - confidence: 0.86
- **[P2] Keep explicit delivery fields ahead of inferred defaults:** `src/agents/tools/cron-tool.ts:518-519`
  - body: The PR merges { ...delivery, ...inferredDelivery }, so inferred accountId, threadId, or target values can overwrite explicit delivery fields when only part of the announce target is missing. Current main fills inferred defaults first and keeps explicit details second.
  - confidence: 0.83
