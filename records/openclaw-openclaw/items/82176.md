## Summary

Keep open: the feature is plausible and the PR does reuse `sessions_list` visibility filtering before file reads, but the new preview path bypasses the existing session-history sanitization/redaction boundary and the supplied proof is test/mock-only.

## What This Changes

This PR adds a built-in `sessions_search` agent tool plus catalog, policy, display, docs, and tests for exact transcript search over visible sessions.

## Best Possible Solution

Add the search tool only after previews and searchable returned text follow the `sessions_history` safety boundary, with regression coverage for tool-result/scaffolding and credential redaction plus real OpenClaw tool-run proof.

## Reproduction Assessment

Yes for the review finding: source inspection shows the PR reads raw transcript text and returns previews from the raw matching line. I did not run a live PR-branch repro because this review is read-only and the branch diff was inspected remotely.

## Solution Assessment

No. Reusing `sessions_list` for visibility is the right direction, but the implementation should sanitize or redact the searched/returned preview content before exposing it as a built-in session tool.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Sanitize transcript previews before returning them:** `src/agents/tools/sessions-search-tool.ts:141-146`
  - body: The search path reads each transcript as raw JSONL text and builds the returned preview from the raw matching line. That means a match inside a tool result, thinking payload, model-control token, or token-like secret can be returned directly, while the existing `sessions_history` surface strips tool messages by default and always redacts credential-like text. Please search/preview a sanitized representation or at least apply the same redaction/filtering before returning previews.
  - confidence: 0.9

## Security Review

Status: needs_attention
