## Summary

Keep open. Current main still exposes a flat, budgeted `<available_skills>` catalog and tells the model to read at most one selected `SKILL.md` up front; it does not implement skill dependency metadata, dependency-edge parsing, layered runtime expansion, `openclaw skills graph --dot`, or a `skills.maxDepth` config contract. The related token-overhead reports overlap with the motivation, but this issue asks for a distinct graph/depth API that still needs maintainer product and extension-contract design.

## What This Changes

Request a Skill Graph feature that parses skill dependencies, expands dependent skills on demand by configurable depth, and adds DOT graph output to reduce token overhead.

## Best Possible Solution

Define a backward-compatible skill dependency metadata contract first, then implement graph construction and prompt expansion that respect existing eligibility, allowlists, prompt budgets, exposure rules, plugin-shipped skills, and docs, with DOT output and config/schema coverage after maintainer approval.

## Reproduction Assessment

Not applicable: this is a feature request, not a bug report. The high-confidence check is current-main source, docs, tests, and CLI inspection showing the requested graph, DOT CLI, dependency metadata, and depth config surfaces are absent.

## Solution Assessment
