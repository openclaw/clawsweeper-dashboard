## Summary

Keep open. This is a focused, still-needed DuckDuckGo parser fix: current main and the latest release still pass unchecked numeric entities to `String.fromCodePoint`, while the PR adds a plugin-local range/surrogate guard, focused tests, and sufficient terminal proof. I found no patch-introduced correctness or security finding; broad CI still needs maintainer rerun or triage before merge.

## What This Changes

The PR validates DuckDuckGo numeric HTML entity code points before decoding them and adds regression coverage for invalid, surrogate-range, and valid supplementary entities.

## Best Possible Solution

Land the plugin-local decoder guard after required CI/main validation, preserving malformed numeric entity text without changing plugin contracts, config, cache, network behavior, or shared core decoding paths.

## Reproduction Assessment

Yes. Current main routes DuckDuckGo title, URL, and snippet text through `decodeHtmlEntities`, and that helper passes unchecked numeric entities to `String.fromCodePoint`; a Node primitive check confirms the out-of-range and surrogate failure modes.

## Solution Assessment

Yes. The plugin-local helper guard is the narrow maintainable fix because all affected DuckDuckGo result fields already share this decoder, and moving behavior into shared core would cross the plugin boundary without evidence of a missing SDK seam.
