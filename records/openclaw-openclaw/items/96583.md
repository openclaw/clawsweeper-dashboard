## Summary

Keep open: current main and the latest release still decode DuckDuckGo numeric HTML entities without a code-point guard, while this PR adds a narrow plugin-local guard, focused tests, and sufficient terminal proof. I found no blocking patch findings; the remaining gate is ordinary maintainer/CI validation.

## What This Changes

The PR validates DuckDuckGo numeric HTML entity code points and preserves invalid or surrogate-range entities while keeping valid supplementary entities decodable.

## Best Possible Solution

Land the plugin-local guard after required CI and maintainer validation, keeping the behavior limited to malformed numeric entity preservation without changing plugin contracts, config, cache, or network behavior.

## Reproduction Assessment

Yes. Current main routes DuckDuckGo title, URL, and snippet text through decodeHtmlEntities, and that helper passes unchecked numeric entities to String.fromCodePoint; a Node primitive check confirms the out-of-range and surrogate failure modes.

## Solution Assessment

Yes. The helper-level plugin-local guard is the narrow maintainable fix because all affected DuckDuckGo result fields share this decoder, and a shared/core decoder change would cross the plugin boundary without evidence of a missing SDK seam.
