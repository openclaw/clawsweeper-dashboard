## What This Changes

The PR adds `spaceType` and `singleUserBotDm` to the Google Chat space payload type, routes inbound messages through a modern DM-versus-group classifier, and adds monitor tests for direct, space, group-chat, legacy, and conflicting payload cases.

## Best Possible Solution

Land the focused classifier and tests after maintainers accept the proof override and ensure the final merge text does not close the broader Google Chat Space issue prematurely.

## Reproduction Assessment

Yes at source level: current `main` and `v2026.6.9` use only legacy `space.type`, so a modern direct-message payload with `spaceType: DIRECT_MESSAGE` and no legacy `type` takes the group branch. Reporter diagnostics in the PR discussion support the user-visible symptom, but I did not run a fresh live Google Chat tenant reproduction.

## Solution Assessment

Yes, this is the best narrow fix: classify the already parsed Google Chat `space` once in the monitor before both access policy and route resolution consume `isGroup`. Moving the decision into only the access policy or only the route builder would leave the other caller with a mismatched direct/group decision.
