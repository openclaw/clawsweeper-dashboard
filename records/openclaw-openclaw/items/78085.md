## Summary

Keep open: current main and `v2026.6.9` still lack this exact CLI parser alias fallback, but the PR is not merge-ready because proof is only unit-test-level and the patch extends a second usage-alias list instead of using the shared usage normalizer.

## What This Changes

Adds `prompt_tokens`/`promptTokens` and `completion_tokens`/`completionTokens` fallbacks to the CLI usage parser plus a JSONL regression test for llama.cpp-shaped usage.

## Best Possible Solution

Use the shared usage normalization contract for CLI JSON/JSONL usage, keep focused regression coverage, and require redacted real behavior proof or an explicit maintainer proof override before merge.

## Reproduction Assessment

Yes at source level: a CLI JSON/JSONL payload with `prompt_tokens` and `completion_tokens` reaches `toCliUsage()`, while current main and `v2026.6.9` only read the input/output aliases there. I did not run a live llama.cpp or CLI-backed provider setup in this read-only review.

## Solution Assessment

No: the direction is right, but adding another alias chain in `toCliUsage()` is not the best fix because `src/agents/usage.ts` already centralizes these provider fields. A narrow adapter to the shared normalizer would reduce drift while preserving the CLI parser contract.
