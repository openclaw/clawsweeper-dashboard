Keep open: current main and the latest release still miss the CLI parser alias path, so the PR addresses a real remaining usage-accounting gap, but it is not merge-ready because real behavior proof is missing and the patch duplicates shared usage normalization logic.

## What This Changes

The PR adds `prompt_tokens`/`completion_tokens` fallbacks to CLI usage parsing and a JSONL regression test for llama.cpp-shaped output.

## Best Possible Solution

Route CLI JSON/JSONL usage through the shared `normalizeUsage()` contract or document a tested intentional divergence, keep the regression coverage, and require redacted real behavior proof or a maintainer proof override before merge.

## Reproduction Assessment

Yes at source level: a CLI JSON/JSONL payload with `prompt_tokens` and `completion_tokens` reaches `toCliUsage()`, while current main and `v2026.6.10` only read input/output aliases there. I did not run a live llama.cpp or CLI-backed provider setup in this read-only review.

## Solution Assessment

No: the bug and test direction are valid, but adding another alias chain in `toCliUsage()` is not the best fix because `src/agents/usage.ts` already centralizes these provider fields. A narrow adapter to the shared normalizer would reduce drift while preserving the CLI parser contract.
