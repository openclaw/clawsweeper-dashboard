## Summary

Keep open: current main still routes direct browser `/navigate` attachment URLs through URL-only Playwright `page.goto()` handling that rethrows the upstream `Download is starting` abort. The shipped isolated-download work is useful adjacent context, but it covers explicit and click-triggered downloads rather than this direct navigation response path.

## What This Changes

The issue asks for browser `/navigate` on CDP-backed attachment URLs to persist the file and return successful download metadata instead of propagating Playwright's download-starting navigation abort.

## Best Possible Solution

Implement the missing browser-plugin navigation-download path by observing a real download around guarded navigation, treating the download-starting abort as success only when a download is captured, persisting through the managed downloads directory, returning additive download metadata from `/navigate`, and preserving navigation policy checks.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: Playwright v1.60.0 throws `Download is starting` for `response.isDownload`, and current `/navigate` rethrows non-policy `page.goto()` errors while returning only `{ url }`. I did not run a live browser in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the browser plugin's guarded Playwright navigation/download handling, reusing the existing managed download storage and explicit waiter semantics while preserving SSRF checks.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/48045 in the browser plugin. Direct `/navigate` to a URL that returns `Content-Disposition: attachment` currently goes through `navigateViaPlaywright` -> `gotoPageWithNavigationGuard` -> `page.goto()` and rethrows Playwright's `Download is starting` abort while returning only URL metadata. Add a narrow navigation-download path: arm/observe a real Playwright download around guarded navigation, treat the download-starting abort as success only when a download was captured, save via the existing managed downloads directory/fs-safe path, return additive `download` metadata from `/navigate`, and keep SSRF/navigation policy behavior unchanged. Do not reintroduce the old https://github.com/openclaw/openclaw/pull/48132 approach: no nonexistent `BrowserContext.setDefaultDownloadPath`, no reverse-import cycle with download helpers, and no double-save while explicit wait/download helpers are active. Add focused regression coverage and a changelog entry if the fix is user-visible.
