## Summary

Keep open. Current main covers adjacent explicit and click-triggered browser downloads, but the direct Playwright/CDP `/navigate` path is still URL-only, still rethrows non-policy `page.goto()` errors, and has no CDP download protocol handling or regression coverage for attachment navigation.

## What This Changes

The issue asks OpenClaw's browser tool to handle direct download-triggering navigation over Playwright CDP by saving the file and returning download metadata instead of surfacing a `Download is starting` failure.

## Best Possible Solution

Add a focused browser-plugin fix for direct attachment navigation: arm or observe a download around guarded navigation, treat the download-start abort as success only when a real download is captured and saved under the guarded downloads directory, return structured download metadata from `/navigate`, and keep existing navigation policy and path-safety checks intact.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a `POST /navigate` to a `Content-Disposition: attachment` URL through raw Playwright/CDP: Playwright raises `Download is starting`, and current OpenClaw rethrows it instead of returning a saved download result.

## Solution Assessment

No. Current main is not the best solution because it fixes adjacent explicit/click download paths while leaving direct attachment navigation on the URL-only error path; the safer repair is localized to browser-plugin navigation/download handling plus regression coverage.
