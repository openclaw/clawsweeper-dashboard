## Summary

Keep open: current main still leaves direct browser `/navigate` attachment downloads on a URL-only path that rethrows Playwright's download-starting abort. The shipped managed/click download rewrite is useful adjacent context, but it does not complete this navigation response path.

## What This Changes

The issue asks for CDP-backed browser navigation to save attachment downloads and return success/download metadata instead of failing with Playwright's "Download is starting" error.

## Best Possible Solution

Add a localized browser-plugin navigation download path that captures the real download around guarded navigation, persists it through the guarded downloads directory, returns additive download metadata from `/navigate`, and keeps blocked navigations fail-closed.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: `POST /navigate` reaches guarded `page.goto()`, Playwright throws `Download is starting` for attachment navigations, and the route can only return `{ url }`; I did not run a live CDP browser in this read-only review.

## Solution Assessment

Yes. The narrowest maintainable fix is in browser-plugin Playwright/CDP navigation handling: arm or observe a download around guarded navigation, treat the download-starting abort as success only after a real capture, and preserve navigation-policy checks.
