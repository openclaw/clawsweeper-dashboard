## Summary

Keep open. Current main still has the direct Playwright/CDP `/navigate` path returning URL-only metadata and rethrowing non-policy `page.goto()` failures, while the shipped #64558 work covers adjacent managed/click-triggered download handling rather than attachment navigation.

## What This Changes

The issue requests that browser `/navigate` over Playwright CDP detect attachment downloads, save them to the configured download directory, and report success instead of throwing `Download is starting`.

## Best Possible Solution

Add a localized browser-plugin navigation download path that captures and saves attachment downloads through the existing guarded download directory, returns structured metadata from `/navigate`, and preserves current policy checks and explicit download waiter behavior.

## Reproduction Assessment

Yes, at source level. `POST /navigate` to a `Content-Disposition: attachment` URL still flows through `navigateViaPlaywright` and `gotoPageWithNavigationGuard`, which rethrows non-policy `page.goto()` errors and exposes only URL metadata; I did not run a live browser in this read-only sweep.

## Solution Assessment

No, current main is not the complete solution for this issue. The narrow maintainable fix is to keep the existing managed/click-download capture and add a direct navigate/download success path with regression coverage.
