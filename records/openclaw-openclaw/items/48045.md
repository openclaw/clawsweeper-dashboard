## Summary

Keep open: current main still leaves direct browser `/navigate` attachment downloads on a URL-only path, while the shipped managed/click download repair does not handle the `page.goto()` download-starting failure path or return download metadata.

## What This Changes

The issue asks for browser CDP download-triggering navigations to persist the file and return a successful response with download metadata instead of surfacing Playwright's `Download is starting` error.

## Best Possible Solution

Implement a localized browser-plugin navigation/download repair that captures the real download, persists it through the guarded downloads directory, returns additive download metadata from `/navigate`, and keeps policy-denied navigations fail-closed.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: `POST /navigate` reaches guarded `page.goto()`, a Playwright attachment navigation raises `Download is starting`, and the route can only return `{ url }`. I did not run a live CDP browser in this read-only review.

## Solution Assessment

Yes. The narrowest maintainable fix is in browser-plugin Playwright/CDP navigation handling: arm or observe a download around guarded navigation, treat download-starting as success only after a real capture, and preserve navigation-policy checks.
