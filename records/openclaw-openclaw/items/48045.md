## Summary

Keep open: current main and the latest release still leave direct browser `/navigate` attachment URLs on a URL-only Playwright path that rethrows the `Download is starting` abort. Pearcekieser's note about `41ed7fa535` is useful context, but that shipped rewrite covers managed/click-triggered downloads, not the direct navigation-download response path reported here.

## What This Changes

The issue asks for CDP-backed browser navigation to handle attachment downloads as successful saved downloads instead of surfacing Playwright's download-starting abort.

## Best Possible Solution

Implement the missing browser-plugin navigation-download path: observe or arm a real download around guarded navigation, treat the download-starting abort as success only after capture, persist through the guarded downloads directory, return additive download metadata from `/navigate`, and keep the current navigation policy checks intact.

## Reproduction Assessment

Yes. Source inspection plus Playwright 1.60.0 source gives a high-confidence path: direct `/navigate` calls guarded `page.goto()`, Playwright throws `Download is starting` for attachment navigations, and OpenClaw currently only returns `{ url }`.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the browser plugin's guarded Playwright navigation/download handling, reusing the existing managed downloads directory and waiter semantics rather than adding a config option or reviving the closed broken PR approaches.
