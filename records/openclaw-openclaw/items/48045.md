## Summary

Keep open: current main and the latest release still leave direct browser `/navigate` attachment URLs on a URL-only Playwright path that rethrows Playwright's `Download is starting` abort. The shipped managed-download work is useful adjacent context, but it covers click-triggered/explicit download handling rather than this direct navigation-download response path.

## What This Changes

The issue reports that browser CDP attachment navigations fail with Playwright's download-starting error and do not save or report the downloaded file.

## Best Possible Solution

Implement the missing browser-plugin navigation-download path: arm or observe a real download around guarded navigation, treat the download-starting abort as success only when a real download is captured, persist the file through the existing downloads directory, return additive download metadata from `/navigate`, and preserve navigation policy checks.

## Reproduction Assessment

Yes. Source inspection plus Playwright 1.60.0 source gives a high-confidence path: direct `/navigate` calls guarded `page.goto()`, Playwright throws `Download is starting` for attachment navigations, and OpenClaw currently only returns `{ url }`.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the browser plugin's guarded Playwright navigation/download handling, reusing the existing managed downloads directory and waiter semantics rather than reviving the closed broken PR approaches.
