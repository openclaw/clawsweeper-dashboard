## Summary

Keep open: current main and v2026.5.20 still leave direct browser `/navigate` attachment URLs on a URL-only Playwright path that rethrows Playwright's download-starting abort. The managed-download rewrite discussed in https://github.com/openclaw/openclaw/pull/64558 is useful adjacent context, but it covers click-triggered and explicit-waiter downloads rather than this direct navigation response path.

## What This Changes

The issue asks for browser CDP navigations to attachment/download URLs to save the file and return successful download metadata instead of failing or dropping the download.

## Best Possible Solution

Implement the missing browser-plugin navigation-download path by observing a real download around guarded navigation, treating the download-starting abort as success only when a download is captured, persisting through the managed downloads directory, returning additive download metadata from `/navigate`, and preserving navigation policy checks.

## Reproduction Assessment

Yes, at source level: direct `/navigate` calls guarded `page.goto()`, Playwright v1.60.0 throws `Download is starting` for attachment navigations, and OpenClaw currently returns only `{ url }`. I did not run a live browser in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the browser plugin's guarded Playwright navigation/download handling, reusing existing managed download storage and explicit waiter semantics while preserving the SSRF guard.
