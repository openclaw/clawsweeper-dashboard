## Summary

Keep open. Current main still leaves direct browser `/navigate` attachment downloads on a URL-only path: adjacent managed/click download capture exists, but guarded `page.goto()` still rethrows non-policy download-starting errors and the route cannot return download metadata.

## What This Changes

The issue requests browser CDP navigation downloads to persist files and report a successful `download.triggered` result instead of failing with Playwright's `Download is starting` error.

## Best Possible Solution

Implement a localized browser-plugin navigate/download repair that captures the real download, persists it through the guarded downloads directory, returns additive download metadata from `/navigate`, and leaves normal navigation and policy-deny failures fail-closed.

## Reproduction Assessment

Yes. From source inspection, `POST /navigate` to a `Content-Disposition: attachment` URL still reaches guarded `page.goto()`, rethrows non-policy errors, and can only return `{ url }`; I did not run a live CDP browser in this read-only review.

## Solution Assessment

Yes. The narrowest maintainable fix is in browser-plugin Playwright/CDP navigation handling: arm or observe a download around guarded navigation, treat download-starting as success only after a real capture, and preserve navigation-policy checks.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/48045. In the browser plugin Playwright/CDP path, direct `/navigate` to a URL whose response is a download, such as `Content-Disposition: attachment`, should not fail with Playwright's `Download is starting` error and should not silently discard the file. Implement a localized repair around `navigateViaPlaywright`, `gotoPageWithNavigationGuard`, and the `/navigate` route: arm or observe a real download around guarded navigation, persist it through the existing guarded downloads directory/path helpers, and return additive `download` metadata while preserving the existing `{ url }` behavior for normal navigation. Only treat the download-starting abort as success when an actual download is captured; rethrow ordinary navigation errors; preserve SSRF/final redirect policy checks and blocked-navigation cleanup. Do not revive the old unsafe approaches from https://github.com/openclaw/openclaw/pull/48132 or https://github.com/openclaw/openclaw/pull/28978, including `BrowserContext.setDefaultDownloadPath`, reverse-importing private helpers, double-saving explicit waiters, or skipping final navigation validation. Add focused regression coverage for direct attachment navigation plus existing guard behavior. Add a changelog entry because this is user-visible.
