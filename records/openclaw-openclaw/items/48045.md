## Summary

Keep open: current main and the latest release still send direct browser `/navigate` attachment URLs through URL-only Playwright navigation, so Playwright's `Download is starting` abort remains a failure path. The shipped managed-download work is useful adjacent coverage for explicit and click-triggered downloads, but it does not add `/navigate` download metadata or a guarded success path for attachment responses.

## What This Changes

The issue asks for CDP-backed browser navigation to attachment/download URLs to save the file and return success metadata instead of failing with Playwright's download-starting error.

## Best Possible Solution

Add a guarded browser-plugin `/navigate` download path that observes a real download around navigation, treats `Download is starting` as success only when that download is captured, persists through the managed downloads directory, returns additive download metadata, and preserves navigation policy checks.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: Playwright v1.60.0 throws `Download is starting` for `response.isDownload`, and current `/navigate` rethrows non-policy `page.goto()` errors while returning only URL metadata. I did not run a live browser in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the browser plugin's guarded Playwright navigation/download handling, reusing the existing managed download storage and explicit waiter semantics while preserving SSRF checks.
