## Summary

Keep open: current main and v2026.5.19 still leave direct browser `/navigate` attachment URLs on a URL-only Playwright path that rethrows Playwright's download-starting navigation abort. The shipped managed-download work is useful adjacent context, but it covers explicit/click-triggered downloads rather than this direct navigation-download response path.

## What This Changes

The issue asks for CDP-backed browser navigation to treat Content-Disposition attachment downloads as successful, persist the file, and return download metadata instead of surfacing Playwright's `Download is starting` error.

## Best Possible Solution

Implement the missing browser-plugin navigation-download path by observing a real download around guarded navigation, treating the download-starting abort as success only when a download is captured, persisting through the managed downloads directory, returning additive download metadata from `/navigate`, and preserving navigation policy checks.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: direct `/navigate` calls guarded `page.goto()`, Playwright v1.60.0 throws `Download is starting` for attachment navigations, and OpenClaw currently returns only `{ url }`. I did not run a live browser in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the browser plugin's guarded Playwright navigation/download handling, reusing existing managed download storage and explicit waiter semantics while preserving the SSRF guard.
