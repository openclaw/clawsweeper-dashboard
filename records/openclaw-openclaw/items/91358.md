## Label Justifications

- `P1`: A large-session compaction can make an active agent session appear frozen and block client chat sends for many minutes.
- `impact:crash-loop`: The issue is about a session becoming effectively unavailable during repeated long compaction/lock windows.
- `impact:session-state`: The affected surface is session transcript compaction and JSONL write-lock state.

## Summary

Keep open: current main and v2026.6.1 still use a 900s compaction timeout, derive compaction-held session locks from that timeout plus grace, and clients can still hit 60s session-file-locked timeouts; no related item owns this exact compaction/lock loop.

## What This Changes

Reports that large local session compaction can monopolize a session JSONL write lock while clients fail with session-file-locked errors.

## Best Possible Solution

Keep this open for a maintainer-approved core runtime fix that keeps live session writes responsive during long compaction, ideally by snapshotting/applying compaction without holding the transcript lock through model summarization.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source inspection gives a high-confidence contention path for 900s compaction locks versus 60s acquire timeouts, but I did not live-run a 10-16 MB session compaction to reproduce the full repeated loop.

## Solution Assessment

Is this the best way to solve the issue? Unclear; simply changing the 900s or 60s defaults is a compatibility-sensitive mitigation, while the stronger fix is to avoid holding the live transcript write lock during model summarization.
