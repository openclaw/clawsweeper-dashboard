## Summary

Keep this PR open: it is a useful focused fix for the shared auth-label helper, but it still leaves a sibling `/model status` auth-label path able to report env auth while runtime and the updated `/status`/`/models` helper report the literal `models.json` provider-entry key.

## What This Changes

The PR updates `resolveModelAuthLabel` to label literal per-entry custom provider `apiKey` bindings as `api-key (models.json)` before env fallback and adds regression coverage for that precedence.

## Best Possible Solution

Keep one credential-source precedence invariant across `/status`, `/models`, and `/model status`, preferably by aligning or consolidating the sibling resolver while preserving env-before-late-custom fallback when no literal per-entry binding exists.

## Reproduction Assessment

Yes, at source level. Runtime resolves literal provider-entry keys before fallback auth, and the unchanged `/model status` resolver still checks env before custom provider keys.

## Solution Assessment

No, not yet. The shared helper fix is the right local change for `/status` and `/models`, but the best fix also aligns or consolidates the sibling `/model status` resolver.
