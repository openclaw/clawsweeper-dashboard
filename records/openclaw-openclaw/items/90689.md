## Summary

Keep open: this is a focused fix for the linked custom-provider auth display bug, and the PR includes useful terminal proof, but the new label ordering can misreport `models.json` for provider entries where runtime auth still uses an env key.

## What This Changes

The PR changes agent auth-label resolution so custom provider `models.json` keys are reported before environment fallbacks, with regression coverage for the `zai-light` shared-base-url case.

## Best Possible Solution

Make `resolveModelAuthLabel` follow the same credential-source decision as `resolveApiKeyForProvider`, keeping the `zai-light` regression while adding sibling coverage for env-candidate providers with configured `apiKey` values.

## Reproduction Assessment

Yes, at source level: current `resolveModelAuthLabel` and `resolveApiKeyForProvider` use separate precedence paths, and the linked issue narrows the symptom to status display. I did not run a live Telegram repro in this read-only review.

## Solution Assessment

No. The PR is a plausible `zai-light` fix, but the best fix is to mirror runtime credential-source precedence instead of broadly moving every literal/custom config key ahead of env labels.
