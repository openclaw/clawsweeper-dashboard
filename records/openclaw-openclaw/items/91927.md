## Summary

Keep open: current main and the latest release still do not expose any `diagnostics.otel.sessionAttribute`-style option or session grouping span attribute, and the new Latitude comment strengthens the product signal while leaving the privacy, naming, and propagation policy for maintainers to decide.

## What This Changes

Requested opt-in diagnostics-otel support for exporting a non-raw session identifier on OTel spans so Langfuse and similar backends can group traces by session.

## Best Possible Solution

Have maintainers choose a default-off diagnostics-otel session grouping contract, likely using a vendor-neutral attribute or clearly documented Langfuse mapping, while keeping raw session keys redacted and adding config/docs/tests for the approved privacy model.

## Reproduction Assessment

Yes for the source-level feature gap: current main has no accepted session grouping config key and no `langfuse.session.id`, `session.id`, or `gen_ai.conversation.id` emission path in diagnostics-otel. I did not run a live backend because the missing attribute path is directly visible in source, docs, and tests.

## Solution Assessment

Unclear as written: an opt-in hashed identifier is plausible, but maintainers still need to choose the privacy contract, vendor-neutral versus Langfuse-specific attribute naming, and root-only versus propagated span behavior.
