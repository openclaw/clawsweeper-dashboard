Keep this PR open for maintainer review: current main still does not promote MiniMax XML tool calls into structured toolUse blocks, while this mixed-scope branch is stale/unmergeable, has blocking MiniMax parser bugs, and overlaps the still-open gateway reload follow-up in #43803.

## Best Possible Solution

Split this into focused current-main work: handle any remaining MiniMax XML output by promoting invocations through the existing toolUse execution path with order-independent XML attribute parsing and standard entity decoding, and handle browser-profile config.patch restart behavior in the #43803 reload-classification path rather than the old combined gateway change.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Parse XML attributes in any order:** `src/agents/pi-embedded-utils.ts:41`
  - body: `parseMinimaxToolCallXml` only recognizes `<invoke>` and `<parameter>` when `name` is the first attribute. Valid XML such as `<invoke id="x" name="Bash">` is skipped, after which promotion produces no `toolUse` block and the sanitizer strips the intended call.
  - confidence: 0.9
- **[P2] Decode XML entities before building tool input:** `src/agents/pi-embedded-utils.ts:48-51`
  - body: Parameter text is copied directly from the XML capture, so encoded values like `&amp;`, `&lt;`, or `&quot;` reach tools unchanged. Decode standard XML entities before assigning `toolUse.input` values so commands and file contents match what the model encoded.
  - confidence: 0.88
