## Summary

Keep open. This maintainer draft has a useful paired-device-state fix, but it unconditionally removes the documented `openclaw logs --follow` file fallback and would close a broader stale-log issue without implementing the active journal/provenance behavior that issue still asks for.

## What This Changes

The PR changes `openclaw logs` to use a passive backend Gateway identity for implicit local `logs.tail`, disables configured-file fallback in `--follow`, and updates tests, docs, and changelog.

## Best Possible Solution

Keep the passive backend identity change, but replace stale `--follow` file fallback with a service-aware live source or explicit provenance path, or have maintainers intentionally accept fail-closed follow behavior and keep the broader stale-log issue tracked separately.

## Reproduction Assessment

Yes at source level: current main routes implicit local logs through the CLI Gateway RPC path and falls back to configured file logs, while the PR test and code show `--follow` now retries and exits instead. I did not run the command because this review is read-only.

## Solution Assessment

No. The passive backend identity is a good fit for the paired-state side effect, but the unconditional fallback removal is not the best complete fix for the stale-log issue without a live service-aware source, provenance, or explicit maintainer acceptance of fail-closed behavior.
