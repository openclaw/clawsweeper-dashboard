## Summary

Keep this PR open: the passive backend identity change fits the existing loopback backend contract, but the `--follow` fallback removal is still a merge-blocking compatibility decision because it can replace log output with retry-then-error behavior and may close a broader live-systemd-log/provenance request prematurely.

## What This Changes

The PR changes `openclaw logs` to use a passive backend Gateway client for implicit local log reads, disables configured-file fallback while following, and updates logs tests, docs, and changelog.

## Best Possible Solution

Keep the passive backend identity for implicit local log reads, but either add active service-aware follow fallback with provenance or explicitly split the broader stale-log issue before landing a narrower fail-closed behavior.

## Reproduction Assessment

Yes at source level: current main uses normal CLI identity and configured-file fallback, while the PR diff shows passive backend identity and no fallback in `--follow`. I did not run the CLI or live systemd cutover scenario because this review is read-only.

## Solution Assessment

No, not as a complete fix for both linked issues. The passive backend identity is the right direction for the paired-device side effect, but unconditional fail-closed follow behavior needs maintainer acceptance or a service-aware live fallback before it can safely resolve the stale-log report.
