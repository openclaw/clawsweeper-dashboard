## Summary

Keep open. The branch targets the right memory-status false-clean gap, but it starts the session drift scan fire-and-forget, so plain status can read `dirty` before `sessionsDirty` is set; the PR body also does not show a real after-fix `openclaw memory status` run for a configured sessions source.

## What This Changes

Adds a status-mode memory manager branch that calls `markSessionStartupCatchupDirtyFiles()` for session sources without starting a full sync.

## Best Possible Solution

Make the status path wait for the lightweight session drift detection before reporting status, then add status-path regression coverage and redacted terminal proof for plain `openclaw memory status`.

## Reproduction Assessment

Yes, at source level. Plain status creates a transient `purpose: "status"` manager and then reads synchronous dirty flags, while the proposed helper awaits session file listing before setting `sessionsDirty`.

## Solution Assessment

No. Reusing the existing session/index comparison is the right narrow direction, but firing it from construction without awaiting it is not ordered before `status()` reads the dirty flag.
