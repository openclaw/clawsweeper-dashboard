## Summary

Keep open. The PR uses the right existing session/index comparison direction, but it starts the check fire-and-forget from construction while plain `openclaw memory status` reads `manager.status()` synchronously, so the false-clean behavior can remain.

## What This Changes

Adds a status-mode memory manager branch that calls `markSessionStartupCatchupDirtyFiles()` for `sessions` sources without starting a full sync.

## Best Possible Solution

Order the lightweight session drift check before a status-purpose manager can report status, cover the actual status path, and add redacted after-fix CLI proof.

## Reproduction Assessment

Yes, at source level. Plain status builds a transient `purpose: "status"` manager, then immediately reads synchronous dirty flags while the proposed helper performs asynchronous listing/stat work before setting `sessionsDirty`.

## Solution Assessment

No. Reusing the existing session/index comparison is the narrow owner-boundary direction, but launching it with `void` from the constructor is not ordered before status is read.
