## Summary

Keep open: current main still has the source-reproducible stale runtime model cache path, the latest release contains the same behavior, and the linked closing PR is still open rather than merged.

## What This Changes

The issue asks for `/new` and `/reset` to stop carrying stale persisted `session.model` and `modelProvider` runtime cache fields after the configured primary model changes.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/77339 or an equivalent narrow repair that clears stale auto-reply runtime `modelProvider` and `model` cache fields on reset while preserving explicit user selections.

## Reproduction Assessment

Yes. Seed a persisted channel session entry with `modelProvider` and `model` but no explicit override, trigger `/new`, and the current auto-reply reset path can preserve those fields before model resolution prefers them over the configured default.

## Solution Assessment

Yes. The narrowest maintainable fix is to clear only the runtime cache fields in the existing auto-reply reset cleanup while leaving explicit `modelOverride` and `providerOverride` selection logic intact.
