## Summary

Keep open: current main and v2026.6.8 still have a source-reproducible auto-reply reset path that can preserve stale runtime model fields, and the linked closing fix at https://github.com/openclaw/openclaw/pull/77339 is still open rather than merged.

## What This Changes

The issue reports that auto-reply `/new` and `/reset` can keep stale persisted `model`/`modelProvider` runtime cache fields after `agents.defaults.model.primary` changes.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/77339 or an equivalent narrow repair that clears stale auto-reply runtime `modelProvider` and `model` cache fields on reset while preserving explicit user selections.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: seed a channel session row with `modelProvider` and `model` but no explicit overrides, trigger `/new`, and the reset merge preserves those old fields before model resolution prefers them over the configured default.

## Solution Assessment

Yes. The narrowest maintainable fix is to clear only the implicit runtime cache fields in the existing auto-reply reset path while leaving explicit `modelOverride` and `providerOverride` preservation intact.
