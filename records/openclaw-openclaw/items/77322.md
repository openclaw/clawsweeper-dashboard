## Summary

The report matches current main source: the auto-reply `/new` path creates a fresh `sessionEntry` without runtime model fields, then merges it over the old stored entry, which preserves stale `model`/`modelProvider` when no explicit override exists. PR #69419 only covered `modelOverride`/`providerOverride`/auth override preservation, so this issue remains actionable.

## What This Changes

The issue requests that channel sessions without explicit model overrides stop carrying stale persisted `model`/`modelProvider` values across `/new` after `agents.defaults.model.primary` changes.

## Best Possible Solution

Clear or replace `model` and `modelProvider` during auto-reply session rollover, then let the next turn persist the model resolved from the current default unless a user override is present.

## Reproduction Assessment

Yes, from source inspection: seed an auto-reply session entry with plain `model`/`modelProvider` and no override fields, send authorized `/new`, and the merge at `src/auto-reply/reply/session.ts:773` preserves the old runtime model fields.

## Solution Assessment

Yes, the narrow maintainable fix is to clear runtime model cache fields on reset and add a regression test beside the existing #69301 override test; changing provider/default policy or adding configuration is unnecessary.
