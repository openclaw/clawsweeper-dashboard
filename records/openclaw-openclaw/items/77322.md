## Summary

Keep open: current main still has a source-reproducible path where plain runtime model/modelProvider fields can survive an auto-reply /new reset and continue to beat the updated configured default; an open closing PR already tracks the fix.

## What This Changes

The issue reports that channel session entries keep stale plain model/modelProvider runtime cache fields after agents.defaults.model.primary changes and /new starts a fresh session.

## Best Possible Solution

Review and land #77339 or an equivalent narrow fix that deletes or resets plain runtime model/modelProvider fields when auto-reply /new creates the replacement session, while preserving explicit user overrides.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: seed a channel session row with plain model/modelProvider and no overrides, change the configured default, then run auto-reply /new; the reset entry omits those fields and the store merge preserves the old values.

## Solution Assessment

Yes. The narrow maintainable fix is at the reset/create persistence boundary: clear stale runtime model cache fields when no explicit override is being preserved, with regression coverage for the auto-reply persisted store path.
