## Summary

Keep this PR open for maintainer review: the scoped hydration path is useful, proof-backed, and has no concrete line-level correctness finding, but it changes session reply delivery to withhold a provenance-backed display-truncated reply when full hydration is unavailable.

## What This Changes

This PR adds `chat.message.get` hydration for display-truncated `chat.history` assistant rows in `sessions_send` reply delivery paths, with embedded gateway support and regression coverage.

## Best Possible Solution

Land this as the scoped `sessions_send` hydration fix only after maintainer acceptance of the unavailable-hydration fail-closed behavior, while leaving broad sentinel sanitization to the separate sanitizer work.

## Reproduction Assessment

Yes, source and PR proof give a high-confidence path: create a long assistant transcript row, observe bounded `chat.history` append `...(truncated)...`, then have `sessions_send` read that preview as the latest reply. I did not rerun the gateway proof locally in this read-only review, but the PR body includes a real CLI gateway plus WebSocket proof for the changed path.

## Solution Assessment

Yes, the best fix shape is to reuse the existing `chat.message.get` projection boundary rather than increasing `chat.history` payloads or globally deleting marker-looking text. The remaining question is not a code defect, but whether maintainers accept the fail-closed behavior when hydration is unavailable.
