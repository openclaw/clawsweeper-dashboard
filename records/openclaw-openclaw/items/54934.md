## Summary

Keep open: the remaining helper/edit empty-card gap is still source-reproducible on current main, but this PR should not merge as-is because it reopens the raw Feishu `card` send-action surface and lacks contributor-owned real behavior proof.

## What This Changes

The branch adds a Feishu empty-card predicate, applies it in send/edit helpers and send/thread-reply action handling, and adds focused unit tests.

## Best Possible Solution

Refresh the branch to keep narrow runtime/helper empty-card validation with focused tests, remove raw send-action `ctx.params.card` handling, preserve the presentation contract, and add redacted real Feishu behavior proof.

## Reproduction Assessment

Yes, at source level: current main still lets `sendCardFeishu({ card: {} })` stringify the empty object and lets `editMessageFeishu({ card: {} })` treat it as an interactive card. The original message-tool send path no longer reproduces because current main builds cards from `presentation` instead of raw `params.card`.

## Solution Assessment

No. Helper-level validation is the narrow maintainable fix, but importing runtime send code into the channel entrypoint and reading raw `ctx.params.card` in send/thread-reply conflicts with the current presentation-only contract.
