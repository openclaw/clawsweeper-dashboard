## Summary

Keep open. The helper-level validation is still useful because current main can pass `{}` through Feishu card send/edit helpers, but this PR is not merge-ready because it also revives obsolete raw-card handling in the shared message action path and lacks the required changelog entry.

## What This Changes

The PR adds Feishu empty-card validation in send/edit helpers and send/thread-reply action handling, plus tests for rejection and text fallback.

## Best Possible Solution

Keep Feishu native-card validation at the helper/runtime boundary, preserve the current presentation-based message action contract, and add focused tests plus a single-line changelog fix entry.

## Reproduction Assessment

Yes for the remaining helper gap: current main source shows `sendCardFeishu({ card: {} })` stringifies and sends the card, and `editMessageFeishu({ card: {} })` treats it as a card via `Boolean(card)`. The old raw send-action reproduction is no longer high-confidence because send/thread-reply now builds cards from `presentation`.

## Solution Assessment

No. The helper-guard direction is the narrow maintainable fix, but the channel import/raw `ctx.params.card` hunk is stale against current main and should be removed or replaced with validation that stays behind the runtime/static utility boundary.
