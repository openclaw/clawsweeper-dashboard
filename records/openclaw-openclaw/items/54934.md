## Summary

Keep open: the helper-level empty-card hardening is still useful, but this PR is stale against current Feishu message sending and adds validation around an obsolete raw `ctx.params.card` send path.

## What This Changes

The PR adds Feishu empty-card validation in the message action dispatcher and send/edit helpers, with unit tests for card-only rejection and text fallback.

## Best Possible Solution

Keep current presentation-based message sending, add empty-card rejection at Feishu native helper/direct edit-send boundaries, and keep runtime helpers out of the channel entrypoint.

## Reproduction Assessment

Yes. Current-main source inspection gives a high-confidence split: `message(action=send, card={})` no longer reaches card sending, while `sendCardFeishu({ card: {} })` and `editMessageFeishu({ card: {} })` remain unguarded.

## Solution Assessment

No. The helper guard direction is narrow and maintainable, but the channel action hunk targets obsolete `ctx.params.card` behavior and the new static `send.ts` import crosses the lazy runtime boundary.
