## Best Possible Solution

Keep the empty-card guard at the Feishu helper/runtime boundary with focused tests, but leave send/thread-reply on the shared `presentation` contract and keep any predicate out of the eager channel entrypoint.

## Solution Assessment

No. Helper validation is the narrow maintainable fix, but the PR’s channel hunk imports runtime send code eagerly and reintroduces raw card handling into the shared send action.
