## Summary

Keep open. Current main has already removed raw `card` handling from Feishu `send`/`thread-reply`, but `sendCardFeishu` and `editMessageFeishu` still accept an empty object; this PR cannot merge as-is because its channel hunk targets obsolete send logic and should be rebased to the native helper/outbound surfaces.

## What This Changes

The PR adds Feishu empty-card validation in the message action dispatcher and send/edit helpers, plus unit tests for card-only rejection, text fallback, and direct helper rejection.

## Best Possible Solution

Preserve the shipped presentation-based Feishu message send path, drop the obsolete raw `ctx.params.card` send/action hunk, and apply any remaining empty-card validation only to native helper/direct edit/send surfaces with focused tests.

## Reproduction Assessment

Yes. Current-main code inspection gives a high-confidence split: `message(action=send, card={})` no longer reaches `sendCardFeishu`, while direct unit-level calls to `sendCardFeishu({ card: {} })` or `editMessageFeishu({ card: {} })` still follow the unguarded helper paths.

## Solution Assessment

No. The helper guard is a reasonable narrow hardening direction, but this PR is not the best current solution because it patches the old raw-card message send path instead of rebasing onto the current presentation/native-helper boundary.
