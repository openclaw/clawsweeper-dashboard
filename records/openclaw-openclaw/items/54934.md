## Summary

Keep open: the remaining helper-level Feishu empty-card validation is still useful, but this PR should not merge as-is because it reopens raw `card` handling in the shared send action and breaks the current lazy runtime boundary. It also still needs contributor-owned real behavior proof beyond mocked unit tests.

## What This Changes

Adds a Feishu card-payload predicate, applies it in send/thread-reply and send/edit helper paths, and adds unit tests for empty-card rejection and text fallback.

## Best Possible Solution

Refresh the PR to keep helper/runtime empty-card validation with focused tests while removing the raw-card send-action branch and preserving the presentation-only shared message contract.

## Reproduction Assessment

Yes, at source level: current main still stringifies `sendCardFeishu({ card: {} })` and treats `editMessageFeishu({ card: {} })` as an interactive card. The original raw-card send-action path is no longer current because send/thread-reply now build cards from `presentation`.

## Solution Assessment

No. Helper-level validation is the narrow maintainable fix, but this PR also imports runtime send code into the channel entrypoint and reopens raw `ctx.params.card` handling in send/thread-reply.
