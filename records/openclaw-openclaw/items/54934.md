## Summary

Keep open: there is still a narrow Feishu helper/edit validation gap on current main, but this PR should not merge as-is because it restores raw `card` handling in the send action path that current code and docs have moved to semantic `presentation`.

## What This Changes

The PR adds Feishu empty-card validation to send/edit helpers and message action handling, with tests for card-only rejection and empty-card-plus-text fallback.

## Best Possible Solution

Keep the focused empty-card guard in Feishu runtime/helper/edit code with tests, avoid reintroducing raw `ctx.params.card` for send/thread-reply, and require redacted real Feishu proof before merge.

## Reproduction Assessment

Yes at source level: current main still lets `sendCardFeishu({ card: {} })` stringify the empty object and lets `editMessageFeishu({ card: {} })` treat it as an interactive card. The original message-tool send path no longer reproduces because current main builds cards from `presentation` instead of raw `params.card`.

## Solution Assessment

No. Helper-level empty-card validation is the maintainable fix, but the PR’s send-action raw-card branch and eager runtime import should be removed to preserve the current presentation contract.
