## Summary

Keep open: the PR addresses a real remaining Feishu native-card helper gap, but the branch is stale against current main and adds channel-level raw-card handling that conflicts with the current presentation-based message action path and lazy runtime boundary.

## What This Changes

The PR adds Feishu empty-card validation in send/edit helpers and channel send/thread-reply handling, plus unit tests for rejection and text fallback.

## Best Possible Solution

Keep the current presentation-based message action path, add empty-card rejection at Feishu native helper/direct edit boundaries, add focused tests and a changelog entry, and avoid reintroducing raw provider-native `card` params to shared message actions.

## Reproduction Assessment

Yes for the remaining helper gap, and no for the old channel send-path behavior. Current main no longer sends raw `message(action=send, card={})`, but `sendCardFeishu({ card: {} })` and `editMessageFeishu({ card: {} })` still have a clear source-level path to bad Feishu API payloads.

## Solution Assessment

No. The helper-guard direction is narrow and maintainable, but the channel import and raw `ctx.params.card` guard are not the best fix against current main; keep validation in the native helper/runtime layer or a small static helper without exposing raw card fields in the message action path.
