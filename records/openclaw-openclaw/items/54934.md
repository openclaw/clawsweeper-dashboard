## Summary

Keep open. The helper-side bug still exists on current main, but this branch is stale against the presentation-based Feishu send path and would reintroduce raw card handling in the shared message action surface.

## What This Changes

The PR adds Feishu empty-card validation in send/edit helpers and message action handling, with unit tests for rejection and text fallback.

## Best Possible Solution

Keep message actions presentation-based, add empty-card rejection at Feishu native send/edit helper boundaries, cover that with focused tests, and add a single active changelog fix entry.

## Reproduction Assessment

Yes for the remaining helper gap: `sendCardFeishu({ card: {} })` and `editMessageFeishu({ card: {} })` still have source-level paths to bad interactive payloads. The older `message(action=send, card={})` path no longer reproduces on current main because sends build cards from `presentation`.

## Solution Assessment

No. The helper-guard direction is right, but the channel import and `ctx.params.card` guard are not the best fix against current main; validation should stay in the native helper/runtime layer or a small runtime-safe utility.
