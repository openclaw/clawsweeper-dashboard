## Summary

Keep open: the helper-level Feishu empty-card bug still exists on current main, but this PR also adds stale raw-card handling in the channel entrypoint and weakens the lazy runtime boundary, so it needs a narrow repair before merge.

## What This Changes

The PR adds Feishu card-payload validation in send/edit helpers and the message action dispatcher, plus tests for empty-card rejection and text fallback.

## Best Possible Solution

Keep Feishu native-card validation at the helper/runtime boundary, preserve the presentation-based send/thread-reply contract, and add focused tests plus one changelog fix entry.

## Reproduction Assessment

Yes for the remaining helper gap: current main shows sendCardFeishu({ card: {} }) stringifies and sends the card, and editMessageFeishu({ card: {} }) treats it as a card via Boolean(card). The old raw send-action reproduction is no longer high-confidence because send/thread-reply now builds cards from presentation.

## Solution Assessment

No, not as written. The helper guard direction is the narrow maintainable fix, but the channel import/raw ctx.params.card hunk should be removed or moved behind a lightweight static utility that does not load send.ts from channel.ts.
