## Summary

Keep open. Current `main` still has the narrow status-rendering gap described here: equivalent `anthropic/*` and `claude-cli/*` routes only borrow the active auth label when the selected auth is `unknown`, so a concrete selected-route env API-key label can still win over active Claude CLI OAuth.

## What This Changes

The issue requests that `/status` show the active Claude CLI OAuth auth path instead of an unrelated Anthropic env API key when a selected Anthropic model is routed through the `claude-cli` runtime alias.

## Best Possible Solution

Update the status renderer to prefer the active non-unknown auth label for equivalent CLI runtime aliases even when the selected route has a concrete env-key label, and add a regression with `ANTHROPIC_API_KEY` plus active `claude-cli` OAuth.

## Reproduction Assessment

Yes for the core renderer path, from source inspection: selected `api-key (...)` auth is treated as concrete and prevents the active OAuth label from being used for equivalent runtime aliases. I did not establish a full live current-main gateway reproduction.

## Solution Assessment

Yes, the narrowest maintainable fix is in `status-text.ts` and `status-message.ts`, with tests for the env-key shadow case. Changing model auth resolution or legacy runtime policy would be broader than the reported display bug.
