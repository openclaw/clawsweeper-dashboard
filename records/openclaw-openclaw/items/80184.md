## What This Changes

The issue requests that `/status` show the active `claude-cli` OAuth auth label instead of an unrelated Anthropic env API-key label when the runtime-alias route is actually using Claude CLI OAuth.

## Best Possible Solution

Land one canonical narrow fix that prefers the active non-unknown CLI-runtime auth label for equivalent runtime aliases while preserving normal API-key status for non-alias routes.

## Reproduction Assessment

Yes for the core renderer path: current source still lets a concrete selected `api-key` label outrank active OAuth for equivalent `claude-cli` runtime aliases. I did not establish a full live gateway reproduction in this read-only review.

## Solution Assessment

Yes. A focused status renderer/auth-label precedence fix with regression coverage is the narrow maintainable solution; a broader runtime config migration or auth resolver redesign is not required for this display-only bug.
