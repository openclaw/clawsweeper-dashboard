## Summary

Keep open. Current main still has the narrow status-rendering gap: equivalent `anthropic/*` and `claude-cli/*` routes only borrow the active auth label when the selected auth is `unknown`, so a concrete selected-route env API-key label can still shadow active Claude CLI OAuth.

## What This Changes

The issue reports that `/status` displays Anthropic env API-key auth instead of active Claude CLI OAuth when `ANTHROPIC_API_KEY` is present alongside a `claude-cli` runtime route.

## Best Possible Solution

Update the status renderer to prefer the active non-unknown auth label for equivalent CLI runtime aliases even when the selected route has a concrete env-key label, with focused regression coverage.

## Reproduction Assessment

Yes for the core renderer path: source inspection shows a selected `api-key (...)` auth label is treated as concrete and prevents the active OAuth label from being used for equivalent runtime aliases. I did not establish a full live current-main gateway reproduction.

## Solution Assessment

Yes. The narrow maintainable fix is in `status-text.ts` and `status-message.ts`, plus tests for the env-key shadow case; changing model auth resolution or legacy runtime policy would be broader than the reported display bug.
