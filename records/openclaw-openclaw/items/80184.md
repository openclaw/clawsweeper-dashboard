## What This Changes

The issue requests that `/status` show active Claude CLI OAuth instead of an unrelated Anthropic env API-key label when both are present.

## Best Possible Solution

Merge a narrow status-renderer fix that prefers active non-unknown OAuth for equivalent CLI runtime aliases over an unused selected-route env API-key label, with regression coverage for the env-key shadow case.

## Reproduction Assessment

Yes for the core renderer path: current `main` still treats selected `api-key` auth as concrete and therefore does not borrow the active Claude CLI OAuth label for equivalent runtime aliases. I did not establish a full live gateway reproduction in this read-only pass.

## Solution Assessment

Yes. The narrow maintainable fix is in status-label and cost-display precedence for runtime-alias-equivalent routes, plus a focused regression for `ANTHROPIC_API_KEY` with active `claude-cli` OAuth; a broader auth policy rewrite is not required for this display bug.
