## Summary

Keep open. Current main still has the narrow `/status` renderer gap: equivalent `anthropic/*` and `claude-cli/*` routes only borrow the active auth label when the selected auth is `unknown`, so a concrete selected-route env API-key label can still shadow active Claude CLI OAuth.

## What This Changes

The issue reports that `/status` can display an Anthropic env API-key auth label even when the active equivalent Claude CLI runtime is using OAuth.

## Best Possible Solution

Update the status renderers to prefer the active non-unknown auth label for equivalent CLI runtime aliases even when the selected route has a concrete env-key label, and add regressions for the env-key shadow case.

## Reproduction Assessment

Yes for the core renderer path: source inspection shows selected `api-key (...)` auth is treated as concrete and prevents the active OAuth label from being used for equivalent runtime aliases. I did not establish a live current-main gateway reproduction in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is in `status-text.ts` and `status-message.ts`, with focused tests for selected env API-key plus active `claude-cli` OAuth; changing model auth resolution or legacy runtime policy would be broader than this display bug.
