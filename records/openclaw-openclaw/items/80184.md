## Summary

Keep open. Current main still has the source-backed status-rendering gap: a concrete selected `api-key` auth label can shadow the active OAuth label for an equivalent `claude-cli` runtime alias, and the linked fix PR at https://github.com/openclaw/openclaw/pull/80260 is still open rather than merged.

## What This Changes

Fix `/status` so an unused `ANTHROPIC_API_KEY` env key does not make a selected Anthropic route display `api-key` when the active equivalent `claude-cli` route uses OAuth.

## Best Possible Solution

Land a canonical narrow status-renderer fix, ideally through https://github.com/openclaw/openclaw/pull/80260 or a maintainer equivalent, so equivalent CLI runtime aliases prefer the active non-unknown OAuth label and suppress unused API-key cost display with regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: call the shared status renderer with selected `anthropic/<model>`, active `claude-cli/<model>`, `modelAuth: "api-key"`, and `activeModelAuth: "oauth"`; current main keeps the selected API-key label and cost mode.

## Solution Assessment

Yes. The narrowest maintainable solution is to adjust the status renderer’s equivalent-runtime-alias auth choice, with regression coverage for selected env API-key plus active Claude CLI OAuth; changing broader auth resolution policy is not required for this display bug.
