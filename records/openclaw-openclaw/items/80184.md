## Summary

Keep open. Current main still lets a concrete selected-route env API-key label win over the active Claude CLI OAuth label for equivalent `anthropic/*` and `claude-cli/*` status routes.

## What This Changes

The issue reports that `/status` can display `api-key (env: ANTHROPIC_API_KEY)` even when the active execution path is Claude CLI OAuth.

## Best Possible Solution

Update the status renderer to prefer the active non-unknown auth label for equivalent CLI runtime aliases even when the selected route has a concrete env-key label, with a regression covering `ANTHROPIC_API_KEY` plus active Claude CLI OAuth.

## Reproduction Assessment

Yes for the core renderer path: source inspection shows a selected `api-key (...)` auth label is treated as concrete and prevents active OAuth from being displayed for equivalent runtime aliases. I did not establish a full live gateway reproduction.

## Solution Assessment

Yes. The narrow maintainable fix is in `src/status/status-text.ts` and `src/status/status-message.ts` with focused status regressions; changing auth resolution or legacy runtime policy would be broader than this display bug.
