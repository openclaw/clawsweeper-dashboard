## Summary

Keep open. Current main still has the reported narrow `/status` display gap: for runtime-alias-equivalent Anthropic and Claude CLI routes, a concrete selected-route env API-key label can still win over the active Claude CLI OAuth label.

## What This Changes

The issue asks `/status` to show the active Claude CLI OAuth auth path instead of an Anthropic env API-key label when Claude CLI is the actual runtime.

## Best Possible Solution

Prefer the active non-unknown auth label for equivalent CLI runtime aliases in the status renderer, with focused regression coverage for `ANTHROPIC_API_KEY` plus active `claude-cli` OAuth.

## Reproduction Assessment

Yes for the core renderer path: source inspection shows a selected `api-key (...)` label is treated as concrete and prevents active OAuth from replacing it for equivalent runtime aliases. I did not establish a full live current-main gateway reproduction.

## Solution Assessment

Yes. The narrow maintainable fix is in `src/status/status-text.ts` and `src/status/status-message.ts`, with tests for the env-key shadow case; changing auth resolution or legacy runtime selection would be broader than this display bug.
