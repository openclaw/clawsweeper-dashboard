## Summary

Keep open: current main still has a source-visible path where the selected Anthropic env-key label can win over the active claude-cli OAuth label for equivalent runtime aliases, so this is a real display bug rather than a duplicate of the already-fixed unknown-auth case.

## What This Changes

The issue requests that `/status` show the active claude-cli OAuth auth path instead of an unrelated Anthropic env API key when the selected Anthropic model is routed through the claude-cli runtime.

## Best Possible Solution

Make equivalent CLI runtime routes use the active runtime auth label and auth mode for status display and cost/usage gating, while preserving selected-provider env-key labels for real non-alias routes.

## Reproduction Assessment

Yes, at source level: selected `anthropic/<model>` plus active `claude-cli/<model>` can leave selected auth as env-derived `api-key` because the active-auth override only handles `unknown`. I did not run live Claude CLI OAuth in this read-only checkout.

## Solution Assessment

Yes, the narrow maintainable fix is to extend the existing runtime-alias equivalence handling so active non-unknown auth wins for equivalent CLI routes even when selected auth resolves to an env API key; this should not require new config.
