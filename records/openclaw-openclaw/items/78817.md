## Summary

Keep open: the feature is not on current main, and the PR is still a plausible plugin-local direction, but the current head has blocking runtime/config/test defects, lacks docs for the new setting, and only provides unit-test proof rather than a real auto-capture run.

## What This Changes

The PR adds a `captureMinChars` setting to the memory-lancedb config and manifest, passes it into `shouldCapture`, lowers the default capture floor to 4, and adds schema tests.

## Best Possible Solution

Keep the setting plugin-local, make `shouldCapture` honor the resolved minimum end to end, validate min/max after defaults are applied, align schema/manifest/tests/docs, and require redacted runtime proof of a short message being captured.

## Reproduction Assessment

Yes for the patch defects: source inspection of PR head shows the stale 10-character floor, the default-max validation gap, and the mismatched expected error string. The requested capability itself is a feature, not a broken current-main bug with an existing contract.

## Solution Assessment

No as submitted; the plugin-local config boundary is right, but the implementation must honor resolved limits, fix the tests, document the setting, and prove the runtime capture path before merge.

  - body: The new `minChars` check still falls through to the unchanged later guard that rejects non-CJK text shorter than 10 characters.   - body: The implementation and manifest use a 1-10000 range, but this test expects `captureMinChars must be between 1 and 1000`. The extension test suite will fail unless the intended range and expected message are made consistent.
  - confidence: 0.99
