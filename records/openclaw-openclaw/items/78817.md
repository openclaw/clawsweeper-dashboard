## Summary

Keep open: the feature is not on current main, and the PR still has correctness gaps in the runtime lower-bound handling and config validation. ## What This Changes

The PR adds a `captureMinChars` option to the memory-lancedb plugin config, manifest, runtime capture filter, and tests, and lowers the default minimum capture length to 4.

## Best Possible Solution

Keep the setting plugin-local, use one resolved min/max contract across config, manifest, runtime, and tests, preserve the trigger and prompt-injection safeguards, and require real proof with a trigger-bearing short message before merge.

## Reproduction Assessment

Yes for the PR defects: source inspection of the PR diff against current main shows the stale 10-character guard and the default-max validation gap. The requested capability itself is a feature, not a broken current-main contract.

## Solution Assessment

No as submitted; the plugin-local config boundary is appropriate, but the implementation must honor the resolved minimum everywhere, validate against the resolved maximum, align the test text, and provide stronger real-behavior proof.

  - body: The new `minChars` check is followed by the existing guard that rejects non-CJK text shorter than 10 characters. - **[P2] Validate min against the resolved max:** `extensions/memory-lancedb/config.ts:155-160`
  - body: This comparison only runs when `captureMaxChars` is explicitly set, but parsing later defaults max to 500. A config like `{ captureMinChars: 1000 }` can parse into a range where no text can satisfy both bounds, so compare against the resolved max value.
  - confidence: 0.94

  - body: The implementation and manifest use a 1-10000 range, but the added test expects `captureMinChars must be between 1 and 1000`. Because `toThrow(string)` matches substrings, this can pass while asserting the wrong public range; make the expected message match the intended contract.
  - confidence: 0.9
