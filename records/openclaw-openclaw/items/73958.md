## Summary

Keep open. The PR addresses a real Feishu/Lark routing bug that is still present on current main, and the diff is scoped to the Feishu plugin, focused tests, and changelog. One narrow cleanup remains: after the PR stops using `threadReply` in the dispatcher, that parameter is still exposed at the dispatcher boundary and passed by callers even though it no longer affects routing.

## What This Changes

The branch changes Feishu group reply routing so `root_id` anchors group thread replies, `thread_id`-only contexts honor `replyInThread`, the dispatcher no longer forces `replyInThread` from `threadReply`, and focused Feishu tests plus a changelog entry are updated.

## Best Possible Solution

Land this PR, or an equivalent Feishu-plugin replacement patch, after removing the unused `threadReply` dispatcher parameter and corresponding call-site/test plumbing. Keep the broader disabled-as-hard-off behavior in #39507, typing-reaction targeting in #67783, and rootless-thread fallback in #75024 as separate review decisions unless maintainers intentionally combine them.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: a Feishu group event with `thread_id`, no `root_id`, and `replyInThread: "disabled"` flows through `resolveFeishuGroupSession` to `replyInThread: true`, and the current test asserts that behavior.

## Solution Assessment

Yes. The proposed fix stays inside the Feishu plugin and preserves shared contracts; the safer refinement is to remove the now-inert `threadReply` dispatcher boundary rather than keeping a metadata field that no longer affects output.
