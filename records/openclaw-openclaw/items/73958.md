## Summary

Keep this PR open. Current main still forces Feishu/Lark thread delivery from any inbound `thread_id` or `root_id`, while this PR is a scoped Feishu-plugin fix for the reported routing behavior; the remaining issue is a narrow inert-parameter cleanup in the PR diff before merge/automation.

## What This Changes

The branch changes Feishu/Lark reply routing so root-id anchored group replies stay threaded, thread-id-only group events obey `replyInThread`, DM quote replies stop forcing thread delivery, and the affected Feishu tests plus changelog are updated.

## Best Possible Solution

Land this PR, or an equivalent Feishu-plugin patch, after removing or precisely documenting the inert dispatcher `threadReply` plumbing. Keep the broader disabled-as-hard-off policy in #39507 and the typing-reaction target fix in #67783 on their separate review paths.

Yes. A high-confidence static reproduction exists on current main: a group event with only `thread_id` and `replyInThread: "disabled"` flows through `threadReply` into `replyInThread: true`, and the current Feishu bot test asserts that forced behavior.

## Solution Assessment

Yes. The PR fixes the behavior at the Feishu owner boundary, preserves existing send helpers and API contracts, and adds focused regression coverage; the safer refinement is a mechanical cleanup of the now-inert dispatcher `threadReply` parameter.
