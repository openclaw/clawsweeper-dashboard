## Summary

Keep open. Current main still does not implement the requested heartbeat session reroute, and this PR remains a plausible implementation candidate for the linked bug, but the proposed migration path still has a concrete source-session canonicalization/data-isolation blocker before it should merge.

## What This Changes

The PR prioritizes configured `heartbeat.session` over the originating forced session and adds cross-session system-event migration, restore markers, event context tags, dedup bypass support, and session-routing regression tests.

## Best Possible Solution

Rebase the fix onto current main and keep the behavior narrow: configured `heartbeat.session` should win for event-triggered heartbeats, but any queue migration should first resolve a canonical, same-agent source session key and then preserve only eligible events with `trusted`, delivery context, dedup identity, and skip-restore behavior covered by focused tests.

## Reproduction Assessment

Yes. The source-level path is high-confidence: exec completion queues a system event in the originating session, requests heartbeat with that session key, and current `resolveHeartbeatSession()` returns the forced session before consulting configured `heartbeat.session`.

## Solution Assessment

No, not yet. The direction is right, but the proposed patch is not the best mergeable form until the source queue key is canonicalized and agent-scoped before draining, then rebased against current heartbeat preflight behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Canonicalize the source session before draining queues:** `src/infra/heartbeat-runner.ts:463-468`
  - body: The migration path uses `params.forcedSessionKey` raw when deciding which queue to drain, while routing canonicalizes and agent-scopes the same key first. If a wake carries an alias or a key for another agent/session namespace, preflight can drain the wrong queue and re-enqueue prompt-visible events into the configured heartbeat session. Normalize with the same `toAgentStoreSessionKey`/`canonicalizeMainSessionAlias`/agent-id checks before calling `drainSystemEventEntries`.
  - confidence: 0.87

## Security Review

Status: needs_attention

Summary: The PR introduces cross-session movement of prompt-visible system events and still drains the source queue from an uncanonicalized session key.

Concerns:

- **[medium] Unscoped source queue migration:** `src/infra/heartbeat-runner.ts:463`
  - body: Because `originatingSessionKey` is derived directly from `params.forcedSessionKey`, a mismatched or alias wake key can cause events from the wrong queue to be moved into the heartbeat session. That can expose or strand system-event text across session boundaries.
