## Summary

Keep open. The PR targets a real source-reproducible heartbeat.session precedence bug, but the current patch still has a session-boundary migration flaw and lacks after-fix real behavior proof.

## What This Changes

The PR reorders heartbeat session resolution, migrates system events between originating and configured heartbeat sessions with skip-restore handling, tags wake-event sources, extends system-event queue options, and adds heartbeat session-routing tests.

## Best Possible Solution

Rebase or replace this PR with a narrow fix that lets configured heartbeat.session win while migrating or restoring only canonical same-agent source queues, then add focused regression coverage and real Telegram or terminal proof.

## Reproduction Assessment

Yes, source-level. Exec completion can enqueue and wake heartbeat with the originating session key, and current main resolves that forced key before the documented heartbeat.session configuration.

## Solution Assessment

No. The precedence reorder is the right direction, but the current migration path drains the raw forced session key before applying the canonical same-agent validation used for routing.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Canonicalize the source session before draining queues:** `src/infra/heartbeat-runner.ts:463-468`
  - body: The migration source is `params.forcedSessionKey?.trim()`, while routing normalizes aliases and verifies same-agent ownership before accepting a forced key. A wake carrying an alias or another agent's key can drain that raw queue and re-enqueue prompt-visible events into the configured heartbeat session, so normalize and reject mismatched source keys before any drain.
  - confidence: 0.87
