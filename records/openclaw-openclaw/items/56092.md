## Summary

Keep open. This PR targets a source-reproducible heartbeat session-routing bug, but the current patch still has a blocking session-boundary migration defect and lacks required after-fix real behavior proof.

## What This Changes

The PR reorders heartbeat session resolution, migrates/restores system events between sessions, tags wake-event sources with context keys, extends system-event options, and adds heartbeat session-routing tests.

## Best Possible Solution

Rebase or replace this PR with a narrow fix where configured heartbeat.session wins and any migrated source queue is canonicalized, same-agent scoped, covered by regression tests, and proven with redacted Telegram or terminal runtime evidence.

## Reproduction Assessment

Yes, source-level. Current main lets an exec completion enqueue and wake heartbeat with the originating session key, and resolveHeartbeatSession consumes that forced key before the documented heartbeat.session configuration.

## Solution Assessment

No. The precedence reorder is the right direction, but the migration path drains params.forcedSessionKey before applying the same canonical same-agent validation used by routing; the safer path is to canonicalize and reject mismatched source keys before any drain.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Canonicalize the source session before draining queues:** `src/infra/heartbeat-runner.ts:463-468`
  - body: The migration source is params.forcedSessionKey?.trim(), while resolveHeartbeatSession normalizes aliases and verifies same-agent ownership before accepting a forced key. A wake carrying an alias or another agent's key can drain that raw queue and re-enqueue prompt-visible events into the configured heartbeat session, so normalize and reject mismatched source keys before any drain.
  - confidence: 0.87
