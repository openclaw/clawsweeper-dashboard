## Summary

Keep open: current main still has the configured-heartbeat-session precedence problem, and this PR is useful but not mergeable as-is because its migration path can drain a raw, unscoped source queue into the heartbeat session.

## What This Changes

The PR makes configured `heartbeat.session` win over forced wake sessions, adds cross-session system-event migration/restore markers, tags wake sources, extends system-event options, and adds heartbeat session-routing tests.

## Best Possible Solution

Rebase onto current main and keep the fix narrow: configured `heartbeat.session` should win, and any event migration must use canonical same-agent source keys while preserving current typed wake, delivery context, trust, dedup, and skip-restore behavior.

## Reproduction Assessment

Yes. Source inspection shows exec completion queues a system event in the originating session, requests a scoped heartbeat with that same key, and current `resolveHeartbeatSession()` returns the forced key before consulting `heartbeat.session`.

## Solution Assessment

No, not yet. The priority reorder is the right direction, but the migration path must canonicalize and agent-scope the source session before moving prompt-visible queue entries.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Canonicalize the source session before draining queues:** `src/infra/heartbeat-runner.ts:463-468`
  - body: The migration path derives `originatingSessionKey` from raw `params.forcedSessionKey` and drains that queue, while session resolution normalizes and same-agent checks the same key before routing. A wake carrying an alias or another agent's key can move prompt-visible events out of the wrong queue and into the configured heartbeat session; normalize with the same session-key flow and reject mismatched sources before draining.
