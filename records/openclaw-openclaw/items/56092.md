## Summary

Keep open. The PR targets a real current-main routing bug, but the proposed migration still has a session/data-isolation blocker because it drains the raw originating key before canonicalizing and agent-scoping it.

## What This Changes

The PR prioritizes configured `heartbeat.session` over the originating forced session and adds cross-session system-event migration, restore markers, context tags, dedup bypass support, and session-routing tests.

## Best Possible Solution

Rebase onto current main and keep the fix narrow: configured `heartbeat.session` should win, and any event migration must use canonical same-agent source keys while preserving delivery context, trust, dedup identity, and skip-restore coverage.

## Reproduction Assessment

Yes. Source inspection shows exec completion queues a system event in the originating session, requests a heartbeat with that same session key, and current `resolveHeartbeatSession()` checks that forced key before `heartbeat.session`.

## Solution Assessment

No, not yet. The direction is correct, but the patch is not the best mergeable form until source queue migration is canonicalized and agent-scoped, then rebased onto the current heartbeat runner.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Canonicalize the source session before draining queues:** `src/infra/heartbeat-runner.ts:461-466`
  - body: The migration path derives `originatingSessionKey` from raw `params.forcedSessionKey` and drains that queue, while routing normalizes and agent-scopes the same key first.
