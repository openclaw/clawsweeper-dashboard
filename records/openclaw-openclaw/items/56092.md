## Summary

Keep open: this PR targets a valid current-main heartbeat/session-routing bug, but the proposed migration still has a blocking source-session scoping issue and the external PR lacks real after-fix runtime proof.

## What This Changes

The PR moves heartbeat.session resolution ahead of forced session keys, migrates/restores system events between source and heartbeat sessions, tags wake-event context keys, extends system-event options, and adds session-routing tests.

## Best Possible Solution

Rebase onto current main and keep the configured heartbeat-session precedence, but canonicalize and same-agent-validate the source session before queue migration while preserving current wake-source, trust, dedup, delivery-context, and restore semantics.

## Reproduction Assessment

Yes. Source inspection shows current main can enqueue exec notifyOnExit events under an originating session while resolveHeartbeatSession still prefers forcedSessionKey before heartbeat.session; I did not run a live Telegram/heartbeat setup in this read-only review.

## Solution Assessment

No. The precedence reorder is the right direction, but the proposed migration should normalize and same-agent-scope the source session before any drain, then be rebased onto the current typed wake/source implementation.

## Review Findings

Overall correctness: patch is incorrect
