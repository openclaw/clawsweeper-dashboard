## Summary

Keep open. The PR targets a real source-reproducible heartbeat session bug, but it still needs contributor real behavior proof and the migration code introduces an unscoped session-boundary risk before it is safe to merge.

## What This Changes

The PR makes configured `heartbeat.session` take precedence over system-event wake session keys, migrates/restores queued system events across sessions, tags wake-event sources, extends system-event metadata, and adds heartbeat session-routing tests.

## Best Possible Solution

Update or replace this PR with a narrow fix where configured `heartbeat.session` wins, migrated source queues are canonicalized and same-agent scoped before draining, regression tests cover alias/mismatched-agent cases, and the contributor provides redacted Telegram or terminal runtime proof.

## Reproduction Assessment

Yes, source-level. Current main lets an exec completion enqueue and wake heartbeat with the originating session key, and `resolveHeartbeatSession()` consumes that forced key before the documented `heartbeat.session` configuration.

## Solution Assessment

No. The precedence reorder is the right direction, but the migration path drains `params.forcedSessionKey` before applying the same canonical same-agent validation used by routing; the safer solution is to canonicalize and reject mismatched source keys before any drain.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Canonicalize the source session before draining queues:** `src/infra/heartbeat-runner.ts:463-468`
