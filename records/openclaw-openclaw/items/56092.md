## Summary

Keep open. The PR targets a valid source-reproducible heartbeat session-routing bug, but the current PR head still has a concrete cross-session queue-migration defect and lacks required after-fix real behavior proof.

## What This Changes

The PR reorders heartbeat session resolution, migrates and restores system events across heartbeat queues, tags wake-event sources with context keys, extends system-event options, and adds heartbeat session-routing tests.

## Best Possible Solution

Rebase or replace this PR with a narrow fix that lets configured heartbeat.session win while migrating/restoring only canonical same-agent source queues, then add focused regression coverage and real Telegram or terminal proof.

## Reproduction Assessment

Yes, source-level. Exec completion can enqueue and wake heartbeat with the originating session key, and current main resolves that forced key before the documented heartbeat.session configuration.

## Solution Assessment

No. The precedence reorder is the right direction, but the current migration path drains the raw forced session key before applying the same canonical same-agent validation used for routing.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Canonicalize the source session before draining queues:** `src/infra/heartbeat-runner.ts:463-468`
  - body: The migration source is params.forcedSessionKey?.trim(), while routing normalizes aliases and rejects mismatched agent sessions before using a forced key. A wake carrying an alias or another agent's key can drain that raw queue and re-enqueue prompt-visible events into the configured heartbeat session, so normalize and reject mismatched source keys before any drain.
