## Summary

Keep open. The underlying heartbeat.session precedence bug is still source-reproducible on current main, but this PR is not merge-ready because its migration path still drains a raw forced session key, the branch is conflicting/stale, and contributor real behavior proof is missing.

## What This Changes

The PR reorders heartbeat session resolution, migrates/restores pending system events between sessions, tags wake-event sources with context keys, extends system-event options, and adds heartbeat session-routing tests.

## Best Possible Solution

Rebase or replace this with a narrow fix where configured heartbeat.session wins, migrated source queues are canonicalized and same-agent scoped, current wake metadata is preserved, and redacted real behavior proof is added.

## Reproduction Assessment

Yes, source-level. Current main lets exec completion enqueue and wake heartbeat with the originating session key, and resolveHeartbeatSession consumes that forced key before the documented heartbeat.session configuration.

## Solution Assessment

No, not as submitted. The precedence reorder is the right direction, but queue migration must validate the source session key before draining and the branch needs rebase plus real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Canonicalize the source session before draining queues:** `src/infra/heartbeat-runner.ts:463-468`
  - body: The migration source is `params.forcedSessionKey?.trim()`, while routing normalizes aliases and verifies same-agent ownership before accepting a forced key. A wake carrying an alias or another agent's key can drain that raw queue and re-enqueue prompt-visible events into the configured heartbeat session, so normalize and reject mismatched source keys before any drain.
  - confidence: 0.87
