## Summary

Keep open: the PR targets a real documented Discord timing bug, but it is not merge-ready because the external PR has no after-fix real Discord proof and its stale/conflicting diff needs to preserve current main's final-delivery-failure error path.

## What This Changes

The PR changes Discord status-reaction cleanup delay to read `doneHoldMs` and `errorHoldMs` from `cfg.messages.statusReactions.timing` instead of always using `DEFAULT_TIMING`.

## Best Possible Solution

Rebase onto current main, preserve `finalDeliveryFailed` as an error outcome while using configured terminal hold timing, add focused Discord regression tests, and provide redacted real Discord proof.

## Reproduction Assessment

Yes, source-reproducible with high confidence: on current main, `removeAckAfterReply: true` plus custom `messages.statusReactions.timing.doneHoldMs` reaches the Discord finalizer and sleeps with `DEFAULT_TIMING.doneHoldMs` before clearing. The linked issue also includes a real Discord observation of the done emoji being cleared almost immediately despite custom hold values.

## Solution Assessment

No, not as submitted. The direction is the narrow right fix, but the PR must be rebased over current main and keep `finalDeliveryFailed` in the error timing branch before it is the best maintainable solution.
