## Summary

Keep open. Current main still lacks a Slack `thread.replyOnToolUse`-style config or runtime branch; Slack reply placement is still driven by `replyToMode` plus incoming thread context, and the linked wrong-thread bug is adjacent rather than a substitute for this opt-in policy.

## What This Changes

The issue requests an opt-in Slack config that threads agent replies after tool use while leaving purely conversational replies at top level.

## Best Possible Solution

Approve the Slack UX first, then add an opt-in Slack-owned schema/types/docs/tested dispatch change that preserves defaults and handles delivery consistently across normal replies, streaming, previews, DMs, channels, and inherited thread context.

## Reproduction Assessment

Not applicable: this is a new Slack config/policy request, not a broken existing contract. Source inspection on current main verifies there is no tool-use-based reply-routing branch or config key to exercise.

## Solution Assessment

Unclear. The tool-use signal is plausible, but current Slack reply planning and native streaming choose thread targets before `onToolStart`, so maintainers should decide the UX and timing before implementation.
