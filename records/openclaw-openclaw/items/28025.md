## Summary

Keep open: current main still selects Slack reply placement from fixed pre-turn policy, while the requested tool-sensitive mode is absent and remains a valid channel-ergonomics feature. It needs maintainer product judgment because native streaming requires a destination thread before tool usage is known, and the new policy must define precedence with existing reply modes and explicit thread context.

## What This Changes

Add an opt-in Slack reply policy that threads agent turns invoking one or more tools while keeping tool-free conversational replies at top level.

## Best Possible Solution

Extend the Slack plugin’s single canonical reply-policy model with an explicit tool-sensitive mode only after defining deterministic precedence and a streaming contract, then document and test channel, DM, existing-thread, fresh-install, and upgrade behavior.

## Reproduction Assessment

Not applicable as a bug reproduction: this issue requests a new conditional routing policy. Current source and docs provide high-confidence proof that Slack presently chooses reply placement from fixed configuration before the turn, not from completed tool usage.

## Solution Assessment

No, the proposed boolean by itself is not yet the best solution. The maintainable path is one Slack reply-policy contract that explicitly resolves precedence and either buffers output or defines an earlier signal compatible with native streaming.
