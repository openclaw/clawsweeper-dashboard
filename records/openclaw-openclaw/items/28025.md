## Summary

Keep open: current main and v2026.6.8 still do not provide tool-use-sensitive Slack reply placement, and the requested behavior adds a new config/reply-policy surface that needs maintainer product direction before implementation.

## What This Changes

Request a Slack option that threads agent replies only when the turn used tools while leaving purely conversational replies at top level.

## Best Possible Solution

Define one Slack reply-policy contract for tool-sensitive threading, preferably within the existing reply-mode model, then update schema, docs, and routing/dispatch tests together.

## Reproduction Assessment

Yes, at source level: current main has no requested config key and selects Slack reply placement from fixed reply policy before completed-turn tool usage is known. This is not a failing bug repro; it is confirmation that the requested feature is absent.

## Solution Assessment

Unclear. The proposed standalone boolean is plausible, but the safer solution is a maintainer-approved Slack reply-policy design that defines precedence and streaming behavior before code changes.
