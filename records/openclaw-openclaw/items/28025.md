## Summary

Keep open: current main and the latest release still lack tool-use-sensitive Slack reply routing, and the requested config/product surface needs maintainer direction before implementation.

## What This Changes

Requests an opt-in Slack reply policy that threads agent replies after tool use while leaving purely conversational replies at top level.

## Best Possible Solution

Define one Slack reply-policy contract for tool-sensitive threading, then update schema, docs, dispatch/routing tests, and streaming fallback behavior together.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction of the feature gap: current Slack config and docs lack the requested key, and dispatch chooses reply placement from fixed reply policy before tool callbacks run.

## Solution Assessment

Unclear; the standalone boolean is plausible but not proven best. A safer solution is a maintainer-approved Slack reply-policy design that defines precedence with existing reply modes and streaming behavior.
