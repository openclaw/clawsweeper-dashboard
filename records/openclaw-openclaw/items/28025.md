## Summary

Keep open. Current main still lacks the requested Slack `replyOnToolUse`-style config and the remaining work is a new Slack reply-routing UX policy, not a cleanup close candidate.

## What This Changes

Request a Slack config option that routes agent replies into the triggering Slack thread only when the session used one or more tools, while leaving purely conversational replies top-level.

## Best Possible Solution

Have maintainers approve the Slack UX semantics first, then add a Slack-owned schema/types/docs/tested routing change that preserves current defaults and composes with DMs, channels, inherited threads, message tools, previews, and native streaming.

## Reproduction Assessment

Not applicable: this is a new Slack config and reply-routing policy request, not a broken existing contract. Source inspection verifies there is no tool-use-based Slack reply-routing key or branch to exercise on current main.

## Solution Assessment

Unclear: the opt-in tool-use heuristic is plausible, but maintainers need to decide the timing and UX semantics before implementation because current Slack reply planning and native streaming choose targets before tool callbacks fire.
