## Summary

Keep open. Current main still routes Slack replies from `replyToMode` and existing thread context, and there is no tool-use-based config or routing path; this is a new Slack plugin behavior/config decision rather than an implemented fix or duplicate.

## What This Changes

Request an opt-in Slack config that threads agent replies only for turns that invoked tools while leaving conversational replies at top level.

## Best Possible Solution

Define the Slack plugin routing policy first, then add an opt-in config/schema/docs/tested implementation in the Slack reply planner and dispatch path.

## Reproduction Assessment

Not applicable: this is a feature/config request, not broken existing behavior. Source inspection shows current-main Slack routing is mode/context based, while tool-use hooks only update status/progress.

## Solution Assessment

Unclear. The proposed signal is plausible, but maintainers should decide the Slack semantics and timing before implementation because it would change reply routing around streaming and existing reply modes.
