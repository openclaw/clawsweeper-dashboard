## Summary

Current main still has only static Slack reply-threading controls, so this feature remains open; the related open items overlap on channel clutter and reply modes but do not implement the requested tool-use-sensitive placement.

## What This Changes

The issue requests an opt-in Slack policy that routes tool-using agent replies into a thread while leaving purely conversational replies at the top level.

## Best Possible Solution

Define one maintainer-approved Slack reply-policy contract for tool-sensitive threading, then update schema, docs, routing/streaming behavior, and focused Slack regression tests together.

## Reproduction Assessment

Yes, at source level for feature absence: current Slack threading and config code are static reply-mode based and lack a tool-use-sensitive reply-routing branch. This is a feature gap, not a broken existing behavior contract.

## Solution Assessment

No, not as an automatic fix lane. The proposed default-off boolean is plausible, but the best fix needs maintainer-approved precedence with existing Slack reply modes, streaming/status timing, explicit top-level opt-outs, DMs, channels, and existing thread sessions.
