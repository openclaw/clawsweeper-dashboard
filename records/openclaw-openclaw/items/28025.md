## Summary

Keep open: current main and v2026.6.5 still lack opt-in Slack threading based on whether a turn used tools. Existing Slack routing is driven by fixed reply modes before the agent turn runs, so this needs a maintainer product decision for timing, streaming behavior, and precedence before implementation.

## What This Changes

Request an opt-in Slack setting that routes agent replies into a thread only when the turn used tools, while leaving purely conversational replies at top level.

## Best Possible Solution

Define the Slack-owned routing contract first, then implement it through prepare/dispatch with coverage for channel roots, DMs, existing threads, streaming, and reply-mode precedence.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a new opt-in Slack routing mode. Source and release inspection confirm the requested capability is absent on current main and v2026.6.5.

## Solution Assessment

No, not as fully specified yet: the desired UX is plausible, but the proposed pre-stream decision cannot depend on observed tool activity without a product choice about buffering, delaying, or changing streaming behavior.
