## Summary

Keep open. Current main still lacks a Slack `thread.replyOnToolUse` configuration or equivalent tool-use-gated reply target, and the Slack reply planner is still driven by reply mode and incoming thread state rather than whether the turn invoked tools.

## What This Changes

Request a Slack config option that routes agent replies into a Slack thread only when the turn invoked tools, while leaving purely conversational replies at top level.

## Best Possible Solution

If accepted, implement this inside the Slack plugin as a default-false typed config option, derive the reply target before the first Slack delivery or streaming action, preserve explicit thread replies and current reply-mode behavior, and update Slack docs, generated config metadata, and focused threading/dispatch tests.

## Reproduction Assessment

Yes for current behavior. A source-level check on current main shows the config key is absent, the strict schema would reject it, and Slack reply targeting uses `replyToMode` plus incoming thread state rather than a per-turn tool-use signal.

## Solution Assessment

Unclear until maintainer product review accepts the behavior. If accepted, the best implementation boundary is the Slack plugin/config surface with a generic per-turn signal only where needed, not a broad core migration or direct Slack API bypass.
