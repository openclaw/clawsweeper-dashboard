## Summary

Keep open. Current main still has no Slack `thread.replyOnToolUse` or equivalent tool-use-gated reply target, and Slack reply planning remains driven by `replyToMode` plus incoming thread state while tool callbacks only update progress/status.

## What This Changes

The issue requests a default-false Slack config option that threads agent replies only when the turn used tools and leaves purely conversational replies top-level.

## Best Possible Solution

If accepted, implement this in the Slack plugin as a default-false typed config option, decide the reply target before the first Slack delivery or stream action, preserve existing explicit-thread and reply-mode behavior, and update docs, generated config metadata, and focused Slack threading tests.

## Reproduction Assessment

Yes for current behavior, but not as a bug reproduction. Source inspection on current main shows the proposed config key is absent from the strict Slack thread schema and reply routing has no tool-use input.

## Solution Assessment

Unclear until maintainers accept the product behavior. The maintainable boundary would be the Slack plugin/config surface plus only the smallest generic per-turn tool-use signal needed, not direct Slack API bypass.
