## Summary

Keep open: current main and v2026.6.5 still lack tool-use-conditioned Slack threading, and the request remains distinct from the related thread-misrouting fixes. The desired UX is plausible, but it requires a product decision because Slack chooses its thread target before tool activity becomes known.

## What This Changes

Add an opt-in Slack policy that threads replies only when the agent invokes one or more tools, while keeping no-tool responses top-level.

## Best Possible Solution

Define one Slack-owned reply-mode contract that states when the tool-use decision becomes final and how it interacts with existing modes, then implement it through the existing prepare/dispatch path with channel, DM, existing-thread, preview, and native-streaming coverage.
