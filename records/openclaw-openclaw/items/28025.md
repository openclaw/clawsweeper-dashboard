## Summary

Keep open: current main and the latest release still lack a Slack tool-use-sensitive reply policy, and the request adds config/product surface that needs maintainer direction before implementation.

## What This Changes

Request a Slack config option that sends agent replies in a thread when the turn used tools and keeps purely conversational replies top-level.

## Best Possible Solution

Define one Slack reply-policy contract for tool-sensitive threading, preferably within the existing reply-mode model, then update schema, docs, and dispatch/routing tests together.

## Reproduction Assessment

Yes, from source inspection rather than a live Slack repro: current main has no requested config key and selects Slack reply placement from fixed reply policy before completed-turn tool usage is known.

## Solution Assessment

No, the proposed standalone boolean is not yet clearly the best solution. The safer path is a maintainer-approved Slack reply-policy design that defines precedence and streaming behavior before code changes.
