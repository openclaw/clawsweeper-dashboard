## Summary

Keep open: current main and v2026.6.6 still choose Slack reply placement from fixed `replyToMode` policy before tool usage is final. The request remains a valid Slack ergonomics feature, but it needs maintainer product direction because it adds config surface and must define streaming, DM/channel, and existing-thread precedence.

## What This Changes

Add an opt-in Slack reply policy/config that threads agent replies only for tool-using turns while leaving purely conversational replies top-level.

## Best Possible Solution

Define one Slack reply-policy contract for tool-sensitive threading, including precedence and streaming fallback, then update config/schema/docs and tests after maintainer approval.

## Reproduction Assessment

Not applicable as a bug reproduction: this issue requests a new conditional Slack reply policy. Source inspection confirms current main lacks that policy and still routes from fixed reply configuration.

## Solution Assessment

No, the proposed boolean alone is not yet the best solution. The safer path is a single Slack reply-policy design that defines precedence with existing modes and what happens to native streaming when the tool-use signal arrives late.
