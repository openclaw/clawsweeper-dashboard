## Summary

Keep open. Current main and v2026.5.19 still lack a Slack `replyOnToolUse` or equivalent tool-use-conditioned reply-routing option, and the request adds new Slack config and UX policy rather than fixing a defined existing contract. The related Slack misrouting fix in https://github.com/openclaw/openclaw/pull/83634 is useful context, but it addresses duplicate inbound delivery and fail-closed thread fallback, not this opt-in auto-threading feature.

## What This Changes

The issue requests an opt-in Slack setting that routes agent replies into a thread after tool use while keeping tool-free conversational replies top-level.

## Best Possible Solution

Have maintainers approve the Slack UX semantics first, then implement a Slack-owned opt-in config/types/schema/docs/routing change that preserves default false and covers DMs, channels, inherited threads, message-tool sends, previews, and native streaming.

## Reproduction Assessment

Not applicable: this is a new Slack config and routing-policy request rather than a broken existing contract. Source inspection verifies current main has no replyOnToolUse or equivalent path to exercise.

## Solution Assessment

Unclear: the opt-in tool-use heuristic is plausible, but maintainers need to decide timing and UX semantics because Slack reply planning, typing status, and native streaming currently choose targets before tool callbacks fire.
