## Summary

Keep this open: current main still lacks the requested side-effect-free cron and skills testing surface, and the remaining work spans CLI, Gateway protocol, cron runtime, skill execution, config, and shared tool-interception policy.

## What This Changes

The issue requests a dry-run/testing framework for cron jobs and skills, including CLI commands, validation, mocked tool calls, and CI-friendly output.

## Best Possible Solution

Design an additive implementation: cron test/validate RPC plus CLI, a skill test runner with explicit input and CI output semantics, and a coordinated tool-interception contract aligned with the related global dry-run proposal.

No, this is not ready as a narrow automated fix. The direction is useful, but the best solution needs maintainer design across protocol, runtime side effects, skills execution, config, and the related global dry-run proposal.
