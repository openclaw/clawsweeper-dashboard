## Summary

Current main still lacks side-effect-free cron test/validate commands and an input-driven skill test runner. The request is directionally aligned with OpenClaw’s safety and test-infrastructure goals, but the proposed all-in-one mock/read-only framework crosses CLI, Gateway protocol, cron execution, skill semantics, plugin/client tools, delivery, and security boundaries, so it needs an explicit maintainer product decision rather than an autonomous fix.

## What This Changes

Add CI-friendly, side-effect-free validation and test commands for cron jobs and skills, with tool-call previews or mocks instead of real execution.

## Best Possible Solution

Define a narrow additive cron validation/preview API and CLI that reuses canonical job parsing and delivery resolution without starting an agent run; track skill execution harness semantics and universal tool interception as coordinated but separate designs, with explicit guarantees, unsupported side effects, JSON output, and no new persistent config until the runtime contract is approved.

## Reproduction Assessment

Yes at source level. Current main’s CLI inventory, Gateway handler, protocol schema, and docs show that cron run queues real execution and that neither cron nor skills exposes the requested side-effect-free test command; this is a missing capability rather than a failing existing contract.

## Solution Assessment

No, the proposed combined mocking framework is not yet the best implementation shape. A bounded cron validation/preview surface is maintainable, but arbitrary skill execution, mock responses, read-only exceptions, and global tool interception need separately approved contracts to avoid false safety guarantees and duplicated ownership with https://github.com/openclaw/openclaw/issues/41418.

## Risks / Open Questions

- A generic mock layer could promise safety while missing side effects in plugin hooks, client tools, channel delivery, subprocesses, or code paths outside the selected interception point.
- Mock responses can materially change model reasoning, so a dry run may not predict the tools, arguments, ordering, output, or duration of a real run.
- The proposed persistent `testing` configuration block would add several core config/default surfaces before their ownership, upgrade behavior, and security semantics are established.
- A nominal read-only API mode cannot be treated as universally safe without per-tool or provider contracts for what constitutes a read and what metadata or remote state may still be affected.
