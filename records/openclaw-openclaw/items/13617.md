- `impact:security`: The issue concerns enforcing a boundary that prevents tools, external APIs, and message delivery from producing unintended real-world side effects.

## Summary

Current main adds useful cron delivery previews and a Skill Workshop, but still has no side-effect-free cron execution test, input-driven skill runner, or request-scoped tool interception contract; the related global dry-run design remains open and requires coordinated product and security decisions.

## What This Changes

Add side-effect-free cron and skill test commands with validation, mocked or read-only tool behavior, and CI-friendly output.

## Best Possible Solution

Reuse current cron delivery-preview and Skill Workshop primitives for static inspection, then add request-scoped cron and skill test execution only after defining one explicit no-side-effects tool and delivery contract with the global dry-run discussion.

## Reproduction Assessment

Yes at source level: the current CLI, Gateway schema and handler, skills commands, and shared tool adapter clearly establish that the requested execution modes do not exist. This is a capability gap rather than a failure of an existing documented contract.

## Solution Assessment

No, not exactly as proposed: cron and skill inspection should reuse current preview and workshop primitives, while execution interception should be request-scoped and share one security contract with https://github.com/openclaw/openclaw/issues/41418 instead of adding a global process flag or broad persistent mock configuration.

## Security Review

Status: needs_attention

Summary: The feature requires an explicit security model for what counts as a side effect and how interception remains request-scoped across concurrent agent, plugin, and delivery paths.

Concerns:

- **[medium] Define a complete no-side-effects boundary:**
  - body: Blocking only calls at the shared tool adapter may not cover message delivery, hooks, session mutations, or other work outside that funnel, so the contract must enumerate and test every allowed and suppressed effect.
  - confidence: 0.92
- **[medium] Avoid ambiguous read-only tool classification:**
  - body: A configurable read-only allowlist can misclassify tools whose apparent reads trigger authentication, billing, remote mutations, downloads, or plugin-defined behavior; maintainers need a typed capability policy rather than names alone.
  - confidence: 0.88
