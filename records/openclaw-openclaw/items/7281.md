## Summary

Keep open: current main still lacks a configurable post-failure fallback channel chain for hook mapping delivery, and the nearby `fallbackChannel`, `channel: "last"`, `bestEffort`, and `failureDestination` mechanisms cover different pre-delivery or failure-notification semantics.

## What This Changes

The issue requests a configurable fallback channel chain for hook mappings when the primary delivery channel fails.

## Best Possible Solution

Define a typed ordered hook delivery fallback contract and carry it through config schema/help, mapping transforms, hook dispatch, cron delivery storage, outbound retry-after-primary-failure handling, docs, and focused regression tests.

## Reproduction Assessment

Yes, source-level. Current main rejects the proposed fallback keys in strict hook config validation, carries only one `channel`/`to` through hook mapping and dispatch, and sends one resolved cron delivery target.

## Solution Assessment

Unclear pending maintainer product direction. The requested direction is plausible, but the maintainable fix should be a typed ordered fallback contract rather than treating `channel: "last"`, `bestEffort`, `failureDestination`, or pre-delivery selection helpers as equivalent.
