## Summary

Keep open. Current main still has a single hook delivery target, while the requested post-failure failover would add a new public hook config contract and delivery semantics that need maintainer product/API choice.

## What This Changes

The issue requests a configurable fallback delivery channel or ordered fallback list for hook mappings when the primary hook delivery channel fails.

## Best Possible Solution

Define a typed ordered hook-delivery failover contract in core, propagate it through hook mapping and cron delivery, and attempt alternates only after the primary outbound send fails, with docs and regression coverage.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection gives high-confidence verification that current main lacks the requested config key and propagation path.

## Solution Assessment

Unclear until maintainers choose the public API shape. The narrow maintainable direction is a typed failover list, not reusing existing pre-delivery fallback or failure-notification fields.
