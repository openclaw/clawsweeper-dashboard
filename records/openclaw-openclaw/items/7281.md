## Summary

Keep open. Current main still lacks configurable hook-mapping fallback delivery, and the existing discussion correctly treats this as a new public hook config and delivery-semantics decision rather than cleanup-close material.

## What This Changes

The issue requests a hook mapping fallback delivery option so a failed primary channel can retry through another channel.

## Best Possible Solution

Define a typed ordered hook-delivery failover contract, then propagate it through hook config/schema, mapping transforms, hook dispatch, CronDelivery, outbound delivery, docs, and focused tests.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection on current main confirms the proposed hook mapping keys and post-failure retry loop are absent.

## Solution Assessment

Unclear until maintainers choose the public API shape. An explicit ordered fallback target contract is likely the maintainable direction, but it should stay distinct from channel="last", bestEffort, and failureDestination.
