## Summary

Keep open: current main still models hook delivery as one target and the existing `last`, `bestEffort`, `failureDestination`, and outbound selection helpers do not provide post-failure hook delivery failover.

## What This Changes

Add a hook mapping `fallbackChannel` or ordered `fallbackChannels` option so delivery can try another channel after the primary channel fails.

## Best Possible Solution

Define a typed ordered hook delivery fallback contract and carry it through config schema/help, hook mapping transforms, dispatch payloads, cron delivery storage, outbound post-failure retry handling, docs, and focused regression tests.

## Reproduction Assessment

Yes, source-level. The proposed fallback keys are rejected by strict hook config validation, hook mapping and dispatch carry only one `channel`/`to`, and cron delivery sends to one resolved target.

## Solution Assessment

Unclear pending maintainer product direction. The requested direction is plausible, but the durable solution should be a typed ordered fallback contract rather than treating `channel: "last"`, `bestEffort`, `failureDestination`, or existing pre-delivery fallback helpers as equivalent.
