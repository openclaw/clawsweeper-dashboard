## Summary

Current main and v2026.6.5 still support only one hook delivery target; same-target retries and failure notifications do not provide successful-result failover, and the closed unmerged recovery PR is not a replacement. This remains useful work but needs maintainer product judgment before implementation.

## What This Changes

Add configurable fallback delivery targets for hook mappings when the primary channel send fails.

## Best Possible Solution

Design a typed ordered list of complete delivery targets, with explicit recipient, account, and thread data plus idempotent attempt and status semantics, then carry it consistently through hook config, mapping, cron persistence, outbound delivery, tests, and docs.

## Reproduction Assessment

Yes at source level. Current main strictly models one hook channel and target, persists one cron destination, and only retries that same destination for transient failures.

## Solution Assessment

No, not as the proposed channel-only field. The best solution needs complete fallback target identity plus explicit ordering, idempotency, transcript, and ambiguous-send semantics.

## Risks / Open Questions

- A fallback channel identifier alone does not identify the correct recipient, account, thread, or session and could misroute content.
- The contract must define ordered attempts, same-target transient retries, transcript mirroring, delivery status, and all-targets-failed behavior.
