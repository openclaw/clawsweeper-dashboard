## Summary

Keep open. Current main still lacks configurable post-failure fallback delivery for hook mappings, and this is a new public hook config/delivery semantics feature rather than an already-fixed bug.

## What This Changes

Request to add a `fallbackChannel` or ordered `fallbackChannels` option for hook mappings when primary delivery fails.

## Best Possible Solution

Define an ordered, typed hook delivery fallback contract and propagate it through hook config/schema, mapping transforms, hook dispatch, cron delivery, outbound failure handling, docs, and focused regression tests.

## Reproduction Assessment

No, not as a bug reproduction. Source inspection gives high confidence that the requested config keys and post-primary-send fallback path are absent on current main.

## Solution Assessment

Unclear until maintainers choose the public API and failure semantics. An explicit ordered fallback target contract is likely maintainable, but it must stay distinct from `channel: "last"`, `bestEffort`, and `failureDestination`.

## Risks / Open Questions

- The public API shape is unresolved: a single `fallbackChannel` may be insufficient if fallback targets need `to`, `accountId`, and `threadId` too.
- Automatic fallback can disclose hook output to an unintended recipient unless target validation and failure classification are fail-closed.
- WebChat fallback may need separate design because current docs describe WebChat/internal sessions differently from reusable outbound channels.
