- `impact:message-loss`: When primary hook delivery fails, the generated hook notification can reach no user-visible channel.

## Summary

Keep open: current main and the latest release still lack hook-mapping post-failure delivery failover, and the existing fallback code is pre-delivery target selection rather than an ordered retry chain after send failure.

## What This Changes

Add a configurable fallback delivery target or ordered target list for hook mappings when `deliver: true` primary channel delivery fails.

## Best Possible Solution

Define a shared ordered delivery-failover contract at the cron/outbound boundary, then let hook mappings populate it through typed config, docs, and tests after maintainers settle retry and safety semantics.

## Reproduction Assessment

Yes, at source level: hook config, schema, mapping resolution, hook dispatch, cron delivery, and outbound send selection expose one delivery target and no post-failure retry chain. I did not run a live channel repro because this read-only review is assessing a missing capability.

## Solution Assessment

No, not as an immediate hook-only `fallbackChannel` field. The safer maintainable solution is a shared ordered delivery-failover contract whose retry, target-pairing, duplicate-delivery, and failure-reporting semantics are decided first.
