## Summary

Keep open. Current main still has single-target hook delivery; the fallback-related helpers I found are pre-delivery route selection helpers, not post-failure failover from one hook delivery channel to another.

## What This Changes

The issue requests configurable `fallbackChannel` or ordered `fallbackChannels` support for hook mappings when the primary delivery channel fails.

## Best Possible Solution

Define a typed ordered hook-delivery failover contract in core, distinguish it from `channel: "last"`, `bestEffort`, and `failureDestination`, propagate it through hook mapping and cron delivery, then cover primary-fails/fallback-succeeds and all-targets-failed behavior in tests and docs.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: a proposed `fallbackChannel` mapping key is rejected by the strict hook mapping schema, and no fallback field can propagate through mapping, dispatch, `CronDelivery`, or the final outbound send path.

## Solution Assessment

Unclear until maintainers choose the public config shape. The requested direction is plausible and core-owned, but the narrow maintainable solution should be an explicit typed failover contract rather than reusing existing pre-delivery fallback or failure-notification fields.
