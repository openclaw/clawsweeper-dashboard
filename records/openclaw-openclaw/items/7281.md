## Summary

Keep open: current main still models hook-mapping delivery as a single target, and adding post-failure channel fallback is a new public hook config and retry-policy feature that needs maintainer product direction.

## What This Changes

The issue requests a hook-mapping fallbackChannel or fallbackChannels option so delivered hook replies can try another channel after the primary channel send fails.

## Best Possible Solution

Define a typed ordered hook delivery fallback contract and carry it through config schema/help, mapping resolution, dispatch payloads, cron delivery storage, outbound retry/failure handling, docs, and focused regression coverage.

## Reproduction Assessment

Yes, at source level: the proposed keys are rejected by strict config validation, the hook mapping and dispatch contract carries only one channel/to target, and cron delivery sends to one resolved target. I did not run a live channel outage because the missing public contract is visible on current main.

## Solution Assessment

Unclear pending maintainer product decision. The requested direction is plausible, but the maintainable solution should be a typed ordered fallback chain rather than treating channel: "last", bestEffort, failureDestination, or existing outbound fallbackChannel helpers as equivalent.
