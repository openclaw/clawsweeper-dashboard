## Summary

Keep open: current main still models hook delivery as one `channel`/`to` target, while the requested ordered post-failure fallback would add new public hook config and retry semantics.

## What This Changes

Request adds `fallbackChannel` or `fallbackChannels` to hook mappings so `deliver: true` can retry alternate channels after the primary channel delivery fails.

## Best Possible Solution

Define a typed ordered hook delivery fallback contract and carry it through config schema/help, mapping transforms, dispatch payloads, cron delivery storage, outbound retry-after-primary-failure handling, docs, and focused regression tests.

## Reproduction Assessment

Yes, source-level. The proposed fallback keys are rejected by strict hook config validation, hook mapping and dispatch carry only one `channel`/`to`, and cron direct delivery sends one resolved target.

## Solution Assessment

Unclear pending maintainer product direction. The requested direction is plausible, but the durable solution should be a typed ordered fallback contract rather than treating `channel: "last"`, `bestEffort`, `failureDestination`, or existing pre-delivery fallback helpers as equivalent.
