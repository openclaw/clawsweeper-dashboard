## Label Justifications

- `P2`: This is a valid delivery-reliability enhancement with limited blast radius, but it needs product semantics before implementation rather than urgent regression repair.
- `impact:message-loss`: When primary hook delivery fails after an agent response is generated, the requested notification can reach no user-visible channel.

## Summary

Keep open: current `main` and the latest release still model hook delivery as a single target, and the existing `fallbackChannel` code is pre-delivery target selection rather than post-failure hook failover. The related broad recovery PR was closed unmerged, and this still needs maintainer decisions about ordered fallback semantics.

## What This Changes

Requests a new `fallbackChannel` or `fallbackChannels` hook-mapping option so failed primary hook delivery can retry an alternate channel.

## Best Possible Solution

Define an ordered delivery-failover contract at the cron/outbound boundary, then expose typed hook-mapping config that populates it with documented retry, idempotency, and failure-reporting semantics.

## Reproduction Assessment

Yes. Source inspection shows a one-target path from hook config/schema through mapping, hook dispatch, cron delivery, and direct outbound send; I did not run a live WhatsApp-to-webchat failure scenario.

## Solution Assessment

No, not as a hook-only field. The maintainable fix is a shared ordered delivery-failover contract that hook mappings can use after retry, idempotency, unknown-outcome, and reporting semantics are decided.
