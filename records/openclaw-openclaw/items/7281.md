## Summary

Keep open. Current main still lacks hook-mapping fallback delivery support, and the request is a new public hook config plus post-failure delivery behavior rather than an already-implemented bug fix.

## What This Changes

Add a hook mapping fallback delivery option so a failed primary channel send can retry an alternate channel such as WebChat.

## Best Possible Solution

Define a typed ordered hook-delivery failover contract in core, then propagate it through config/schema, mapping transforms, hook dispatch, CronDelivery, outbound delivery, docs, and focused regression tests.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection on current main confirms the proposed hook mapping keys and post-failure retry loop are absent.

## Solution Assessment

Unclear until maintainers choose the public API shape. An explicit ordered fallback target contract is the likely maintainable direction, but it should not reuse channel: "last", bestEffort, or failureDestination for different semantics.
