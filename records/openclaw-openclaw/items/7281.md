## Summary

Keep open. Current main still models hook delivery as one target and has no typed fallback channel chain for hook mappings; the existing fallback helpers are pre-delivery target selection or failure-notification paths, not retry-after-send-failure semantics.

## What This Changes

Add `fallbackChannel` or `fallbackChannels` to hook mappings so a `deliver: true` hook can retry alternate channels when the primary outbound delivery fails.

## Best Possible Solution

Define a typed, documented hook-delivery fallback contract, carry it through config schema, mapping transforms, hook dispatch payloads, `CronDelivery`, and direct cron delivery, then retry alternates only after primary outbound send failure while preserving clear all-targets-failed reporting.

## Reproduction Assessment

Yes. The static reproduction is high-confidence: add the proposed fallback field to a hook mapping and current schema/path handling cannot accept or propagate it; a mocked primary delivery failure would still exercise only the single resolved cron target.

## Solution Assessment

Unclear until maintainers choose the public API shape. The requested direction is valid, but `channel: "last"`, `bestEffort`, existing `fallbackChannel` variables, and `failureDestination` do not solve this specific post-failure hook delivery failover.
