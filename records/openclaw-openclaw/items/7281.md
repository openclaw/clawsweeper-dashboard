- `impact:message-loss`: The reported failure mode is generated hook output not reaching the user when the configured primary channel is unavailable.

## Summary

Keep open: current main and v2026.5.22 still model hook delivery as one channel/target, and the related closed PR did not land a post-failure hook fallback chain.

## What This Changes

Request adding `fallbackChannel` or ordered `fallbackChannels` to hook mappings so `deliver: true` hook output can try alternate channels after the primary send fails.

## Best Possible Solution

Define an owner-approved ordered hook delivery fallback contract and thread it through config schema/types, mapping transforms, hook dispatch, cron delivery storage, outbound send failover, docs, and focused regression tests.

## Reproduction Assessment

Yes, source-level. Current main rejects the proposed fallback keys, carries only one channel/to route through mapping and dispatch, and sends one resolved cron delivery target.

## Solution Assessment

Unclear pending maintainer product direction. A typed ordered fallback chain is plausible, but it should not be confused with `channel: "last"`, `bestEffort`, `failureDestination`, or pre-delivery selection helpers.
