## Summary

Keep open. Current main still lacks hook-mapping `fallbackChannel`/`fallbackChannels` support, and the remaining work is a new public config and delivery-semantics feature rather than a stale or already-implemented bug fix.

## What This Changes

The issue requests a configurable fallback channel or ordered fallback chain for hook mappings when the primary `deliver: true` channel send fails.

## Best Possible Solution

Define an ordered, typed hook delivery fallback contract and propagate it through hook config/schema/help, mapping transforms, `HookAgentDispatchPayload`, `CronDelivery`, outbound retry handling, docs, and focused regression tests.

## Reproduction Assessment

No, not as a bug reproduction; this is a feature request for a new hook mapping option. Source inspection gives high confidence that the requested config keys and post-primary-send retry path are absent on current main.

## Solution Assessment

Unclear until maintainers choose the public API and failure semantics. An explicit ordered fallback target contract is the maintainable direction, but it should stay distinct from `channel: "last"`, `bestEffort`, and `failureDestination`.
