## Summary

Keep open: current main still models hook-mapping delivery as a single channel/target, and this request adds a new public fallback delivery contract that needs maintainer product direction before implementation.

## What This Changes

The issue requests `fallbackChannel` or `fallbackChannels` on hook mappings so failed primary delivery can retry alternate channels.

## Best Possible Solution

Define a typed ordered hook delivery fallback contract and carry it through config schema/help, mapping resolution, hook dispatch, cron delivery storage, outbound retry/failure handling, docs, and regression coverage.

## Reproduction Assessment

Yes, at source level: the proposed keys are rejected by strict config validation, hook mapping and dispatch carry only one `channel`/`to`, and cron delivery sends to one resolved target. I did not run a live outage because the missing contract is visible on current main.

## Solution Assessment

Unclear pending maintainer product decision. The requested direction is plausible, but the maintainable fix should be a typed ordered fallback chain rather than treating `channel: "last"`, `bestEffort`, `failureDestination`, or target-selection helpers as equivalent.
