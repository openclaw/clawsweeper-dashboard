## Summary

Keep open. Current main still models hook delivery as a single target, and the existing fallback-related helpers are pre-delivery route selection or failure-notification paths rather than post-failure failover from one hook delivery channel to another.

## What This Changes

Issue requests a configurable `fallbackChannel` or ordered fallback channel list for hook mappings when primary hook delivery fails.

## Best Possible Solution

Define a typed ordered hook-delivery failover contract in core, propagate it through hook mapping and cron delivery, and try alternates only after primary delivery fails, with docs and focused regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: a proposed `fallbackChannel` mapping key is rejected by the strict hook mapping schema, and no fallback field can propagate from mapping to cron delivery or outbound send.

## Solution Assessment

Unclear until maintainers choose the public API shape. The requested direction is plausible, but the maintainable solution should be an explicit typed failover contract rather than reusing existing pre-delivery fallback or failure-notification fields.
