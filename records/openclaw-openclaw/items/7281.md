## Summary

Keep open. Current main still models hook delivery as a single target, while the requested post-failure fallback chain is a new public hook config and delivery behavior that needs maintainer API/product choice.

## What This Changes

Request adds `fallbackChannel` or `fallbackChannels` to hook mappings so failed primary hook delivery can try alternate channels.

## Best Possible Solution

Define a typed ordered hook-delivery failover contract in core, propagate it through hook mapping and cron delivery, then attempt alternates only after primary outbound delivery fails, with docs and regression coverage.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection gives high confidence that current main lacks the requested config key and propagation path.

## Solution Assessment

Unclear until maintainers choose the public API shape. The narrow maintainable direction is a typed ordered fallback list, not reusing existing `channel: "last"`, `bestEffort`, or failure-notification fields.
