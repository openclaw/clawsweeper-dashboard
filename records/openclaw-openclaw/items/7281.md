## Summary

Keep open: current main still has no configurable post-failure fallback chain for hook mapping delivery. The related recovery PR is open and broad, but it does not settle the public hook config/API decision for this issue.

## What This Changes

Add `fallbackChannel` or ordered `fallbackChannels` to hook mappings so delivery can try an alternate channel after the primary channel fails.

## Best Possible Solution

Define an owner-approved ordered hook delivery fallback contract and carry it through config schema/help, mapping transforms, hook dispatch, cron delivery storage, outbound retry-after-primary-failure handling, docs, and focused regression tests.

## Reproduction Assessment

Yes, source-level. Current main rejects the proposed fallback keys in strict hook mapping validation, carries only one `channel`/`to` through mapping and dispatch, and sends one resolved cron delivery target.

## Solution Assessment

Unclear pending maintainer product direction. The requested fallback chain is plausible, but the maintainable solution should be a typed ordered contract rather than treating `channel: "last"`, `bestEffort`, `failureDestination`, or pre-delivery selection helpers as equivalent.

## Risks / Open Questions

- A fallback chain needs maintainer decisions on public config shape, ordering, and which failures trigger failover: target resolution, channel unavailable, transport send errors, partial sends, or all of them.
- Retrying another channel after an ambiguous send result can duplicate or misroute notifications unless error classification and idempotency are defined carefully.
- The open related recovery PR is broad and unmerged, so treating it as the canonical solution would leave this hook-specific API decision unresolved.
