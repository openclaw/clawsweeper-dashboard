- `impact:message-loss`: The reported problem is lost hook notification delivery when the configured primary channel is unavailable and no fallback channel can be tried.

## Summary

Keep open: current main still models hook mapping delivery as a single channel/target, and the existing `fallbackChannel`, `channel: "last"`, `bestEffort`, and `failureDestination` paths do not provide the requested post-failure hook failover chain.

## What This Changes

The issue requests `fallbackChannel` or ordered `fallbackChannels` support for hook mappings so `deliver: true` can try another channel after the primary delivery fails.

## Best Possible Solution

Define a typed ordered hook delivery fallback contract and carry it through config schema/help, mapping transforms, hook dispatch, cron delivery storage, outbound retry-after-primary-failure handling, docs, and focused regression tests.

## Reproduction Assessment

Yes, source-level: current main rejects the proposed fallback keys in strict hook config validation, carries only one `channel`/`to` through hook mapping and dispatch, and sends one resolved cron delivery target. I did not run live delivery because this was a read-only review and the missing contract is visible in source.

## Solution Assessment

Unclear pending maintainer product direction. The requested fallback chain is plausible, but the maintainable fix should be a typed ordered fallback contract rather than treating `channel: "last"`, `bestEffort`, `failureDestination`, or pre-delivery selection helpers as equivalent.
