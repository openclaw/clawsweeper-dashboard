## Summary

Keep open: current main still lacks configurable post-failure hook delivery failover, and adding it would introduce a new public hook config and retry policy that needs maintainer product direction.

## What This Changes

The issue asks for `deliver: true` hook mappings to support a `fallbackChannel` or ordered fallback chain when the primary channel delivery fails.

## Best Possible Solution

Define a typed ordered hook delivery fallback contract and carry it through config schema/help, mapping resolution, dispatch payloads, cron delivery storage, outbound retry/failure handling, docs, and focused regression coverage.

## Reproduction Assessment

Yes, at source level: the proposed fallback keys are rejected by strict hook config validation, hook mapping and dispatch carry only one `channel`/`to`, and cron delivery sends to one resolved target. I did not run a live channel outage because the missing public contract is visible on current main.

## Solution Assessment

Unclear pending maintainer product decision. The requested direction is plausible, but the maintainable solution should be a typed ordered fallback contract rather than treating `channel: "last"`, `bestEffort`, `failureDestination`, or existing pre-delivery fallback helpers as equivalent.
