## Summary

Keep open: current main and the latest release still lack configurable post-failure hook delivery failover, and the related broad delivery/fallback work does not replace this hook-specific config contract.

## What This Changes

Request to add `fallbackChannel` or ordered fallback channels to hook mappings so delivered hook agent replies can try another channel after the primary send fails.

## Best Possible Solution

Define an owner-approved ordered hook delivery fallback contract and wire it through config schema/types, mapping transforms, hook dispatch, cron delivery storage, outbound send failover, docs, and focused regression tests.

## Reproduction Assessment

Yes, source-level. Current main rejects the proposed fallback mapping keys, carries only one `channel`/`to` route through hook mapping and dispatch, and sends one resolved cron delivery target.

## Solution Assessment

Unclear pending maintainer product direction. A fallback chain is plausible, but the best fix should be a typed ordered delivery contract rather than overloading `channel: "last"`, `bestEffort`, `failureDestination`, or existing pre-delivery fallback helpers.
