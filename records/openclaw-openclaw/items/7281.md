## Summary

The central user problem remains unsolved on current main and in the latest release: hook mappings still carry one delivery destination, while existing target-selection fallback and failure notifications do not resend the original hook result after a failed primary send. No viable canonical issue or merged/open proof-positive PR supersedes this request, but the new cross-channel behavior and config contract require maintainer product judgment.

## What This Changes

Add configurable ordered fallback delivery destinations for hook mappings when the primary outbound delivery attempt fails.

## Best Possible Solution

Define one generic typed ordered delivery plan at the cron/outbound owner boundary, require explicit channel/target pairs, attempt alternates deterministically only after send failure, and report one clear final status while keeping failure notifications separate.

## Reproduction Assessment

Yes at source level: current main has a strict single-target hook mapping contract and a one-target delivery path, with no post-send alternate chain. This is a feature gap rather than a reproduced regression in already-defined behavior.

## Solution Assessment

No, the literal hook-only `fallbackChannel` field is not yet the best proven solution. The maintainable direction is a shared ordered delivery-plan contract with explicit privacy, status, and failure-notification semantics.
