## Summary

Keep open: current main and latest release still lack a typed hook delivery fallback chain, and the existing fallback helpers cover pre-delivery target selection or failure notifications rather than retrying an alternate hook delivery channel after the primary send fails.

## What This Changes

The issue requests a configurable `fallbackChannel` or ordered `fallbackChannels` option for hook mappings when primary hook delivery fails.

## Best Possible Solution

Define an owner-approved ordered hook delivery fallback contract and thread it through hook config schema/types, mapping transforms, dispatch payloads, cron delivery storage, outbound send failover, docs, and focused regression tests.

## Reproduction Assessment

Yes, source-level: current main rejects the proposed fallback keys in strict hook mapping validation, carries only one `channel`/`to` route through mapping and dispatch, and sends one resolved cron delivery target.

## Solution Assessment

Unclear pending maintainer product direction. The requested fallback chain is plausible, but it should be a typed ordered contract rather than reusing `channel: "last"`, `bestEffort`, `failureDestination`, or pre-delivery selection helpers as if they were equivalent.
