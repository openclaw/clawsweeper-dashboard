## Summary

Keep open. Current main still models hook delivery as a single resolved channel/target, while the requested post-failure fallback chain would add a new public hook config and delivery contract.

## What This Changes

Add a `fallbackChannel` or ordered `fallbackChannels` option for delivered hook mappings so failed primary delivery can try another channel.

## Best Possible Solution

Define a typed ordered hook-delivery failover contract in core, propagate it through mapping and cron delivery, and document how it differs from existing last-route and failure-notification behavior.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection on current main confirms the proposed config keys and post-failure fallback loop are absent.

## Solution Assessment

Unclear until maintainers choose the public API shape. The narrow maintainable direction is an explicit ordered fallback target contract rather than reusing `channel: "last"`, `bestEffort`, or `failureDestination`.
