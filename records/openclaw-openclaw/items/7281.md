- `impact:message-loss`: When primary hook delivery fails after the agent response is generated, the requested notification can reach no user-visible channel.

## Summary

Keep open: current main and v2026.6.8 still lack hook-mapping post-failure delivery failover, and the existing fallback paths are pre-delivery target selection or failure notification rather than an ordered retry chain after a primary send fails.

## What This Changes

The issue requests a `fallbackChannel` or ordered `fallbackChannels` option for hook mappings when `deliver: true` primary channel delivery fails.

## Best Possible Solution

Define ordered delivery failover at the cron/outbound boundary, then expose typed hook mapping config only after retry, unknown-outcome, target-pairing, duplicate-send, and failure-reporting semantics are settled.

## Reproduction Assessment

Yes, source-level. Current main has a clear one-target path from hook config through mapping, hook dispatch, cron delivery, and outbound send; I did not run a live WhatsApp/webchat repro in this read-only review.

## Solution Assessment

No, not as a standalone hook-only field. The maintainable solution is a shared ordered delivery-failover contract that hook mappings can populate after the retry and safety semantics are decided.
