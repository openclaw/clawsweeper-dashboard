## Summary

Keep this issue open: current main still supports only one hook delivery destination, the existing fallback mechanisms have different semantics, and no merged or viable canonical PR implements post-failure hook failover. The feature is worthwhile, but its destination shape and duplicate-safe delivery guarantees require maintainer product judgment.

## What This Changes

Add configurable ordered fallback destinations for hook-generated messages when delivery to the primary channel fails.

## Best Possible Solution

Define an ordered list of complete fallback destinations at the shared hook-to-cron delivery boundary, then continue only proven-unsent payloads after definite failures and report the final destination or aggregate failure explicitly.

## Reproduction Assessment

Yes, at source level: current hook mappings, dispatch payloads, cron announce delivery, and outbound sending carry one destination and provide no post-failure alternate destination chain. A live failure was not run during this read-only review.

## Solution Assessment

No, the proposed bare `fallbackChannel` field is underspecified because alternate transports may need different targets and partial sends need duplicate-safe handling. Ordered complete destination objects at the shared delivery boundary are the safer solution.

## Risks / Open Questions

- A channel-only fallback field cannot safely express transport-specific recipients, accounts, threads, or other destination identity required by the alternate channel.
- If the primary transport sends some payloads before failing, retrying the complete response through a fallback can duplicate already-delivered content.
- Maintainers must define which failures permit failover, whether fallback is ordered or parallel, and how all-destinations-failed status appears in hook summaries and cron history.
