## Summary

Keep open. Current main still has no hook-mapping fallback delivery field or post-failure retry chain, and the request is a new public hook config/delivery behavior rather than a broken existing contract.

## What This Changes

Request adding `fallbackChannel` or ordered `fallbackChannels` to hook mappings so delivery can retry another channel after the primary channel fails.

## Best Possible Solution

Define a typed ordered hook-delivery failover contract in core, propagate it through config/schema/help, mapping transforms, hook dispatch, CronDelivery, outbound delivery, docs, and focused success/failover/all-failed tests.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection on current main confirms the proposed hook mapping keys and post-failure delivery retry loop are absent.

## Solution Assessment

Unclear until maintainers choose the public API shape. An explicit ordered fallback target contract is the maintainable direction, but it should not reuse channel="last", bestEffort, or failureDestination for different semantics.
