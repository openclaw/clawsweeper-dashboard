## Label Justifications

- `P2`: This is a valid delivery reliability enhancement with bounded blast radius, but it needs product semantics before implementation.
- `impact:message-loss`: A generated hook response can fail to reach any user-visible channel when the selected channel is unavailable.

## Summary

Keep open: current main and the latest release still lack hook-mapping post-failure delivery failover. Existing `fallbackChannel` identifiers are pre-delivery selection helpers, and `failureDestination` is failure-notification routing, not an ordered retry chain after a hook delivery send fails.

## What This Changes

Request a `fallbackChannel` or ordered `fallbackChannels` hook mapping option so generated hook replies can retry another channel when primary delivery fails.

## Best Possible Solution

Define shared ordered delivery-failover semantics at the cron/outbound boundary, then expose them through typed hook mapping config, docs, and tests after maintainers settle the product behavior.

## Reproduction Assessment

Yes. At source level, hook config, strict schema, mapping resolution, hook dispatch, cron delivery, and direct outbound delivery all expose one delivery target and no post-failure retry chain.

## Solution Assessment

No, not as an immediate hook-only `fallbackChannel` field. The safer maintainable solution is a shared ordered delivery-failover contract whose retry, target-pairing, duplicate-delivery, and failure-reporting semantics are decided first.
