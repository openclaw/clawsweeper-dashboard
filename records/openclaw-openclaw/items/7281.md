## Summary

Keep open. Current main still models hook delivery as a single channel/target, and the existing fallback helpers are pre-delivery channel selection or failure-notification paths rather than post-failure hook delivery failover.

## What This Changes

The issue requests a new `fallbackChannel` or ordered `fallbackChannels` option for hook mappings when primary delivery fails.

## Best Possible Solution

Define an explicit ordered hook delivery fallback contract and carry it through config schema/help, mapping resolution, dispatch payloads, cron delivery storage, outbound retry/failure handling, docs, and focused regression coverage.

## Reproduction Assessment

Yes, at source level: adding the proposed key to a hook mapping would be rejected by the strict schema, and the current hook-to-cron delivery path has no field or loop for fallback targets. I did not run a live WhatsApp/WebChat failure scenario because this was a read-only review and the missing contract is visible in source.

## Solution Assessment

Unclear pending maintainer product decision. If accepted, the maintainable direction is a typed ordered fallback chain through the existing hook and cron delivery contract rather than treating current `channel: "last"`, `bestEffort`, or `failureDestination` as equivalent.
