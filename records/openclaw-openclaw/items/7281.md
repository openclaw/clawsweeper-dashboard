## Summary

Keep open. Current main still models hook delivery as one channel/target and has no post-failure fallback chain for hook mappings; the existing `fallbackChannel` helpers are pre-send channel selection fallbacks, not retry-after-delivery-failure behavior.

## What This Changes

Add a `fallbackChannel` or `fallbackChannels` hook mapping option so delivered hook runs can retry another channel after the primary send fails.

## Best Possible Solution

Define an ordered, typed hook delivery fallback contract and propagate it through hook config/schema/help, mapping transforms, `HookAgentDispatchPayload`, `CronDelivery`, outbound send retries, docs, and focused regression tests.

## Reproduction Assessment

No, not as a bug reproduction; this is a feature request. Source inspection on current main gives high confidence that the proposed hook mapping keys and post-failure retry path are absent.

## Solution Assessment

Unclear until maintainers choose the public API and failure semantics. An explicit ordered fallback target contract is the maintainable direction, but it should stay distinct from `channel: "last"`, `bestEffort`, and `failureDestination`.
