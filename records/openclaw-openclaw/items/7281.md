## Summary

Keep open: current main still lacks hook-mapping post-failure delivery failover, and the related open/closed items do not replace this specific hook config request.

## What This Changes

The issue requests a hook-mapping fallback delivery channel or ordered fallback list when the primary hook delivery channel fails.

## Best Possible Solution

Define ordered delivery failover at the cron/outbound boundary, then expose typed hook mapping config after retry, unknown-outcome, duplicate-send, target-pairing, and failure-reporting semantics are settled.

## Reproduction Assessment

Yes. Source inspection shows a one-target path from hook config through mapping, hook dispatch, cron delivery, and outbound send; I did not run a live WhatsApp/webchat failure scenario.

## Solution Assessment

No, not as a hook-only field. The maintainable solution is a shared ordered delivery-failover contract that hook mappings can populate after retry and safety semantics are decided.
