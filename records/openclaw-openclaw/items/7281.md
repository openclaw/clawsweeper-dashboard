- `impact:message-loss`: The reported problem is lost hook notification delivery when the configured primary channel is unavailable and no fallback channel can be tried.

## Summary

Keep open: current main still lacks configurable post-failure fallback channels for hook mapping delivery. The code and docs still model one hook delivery route, while adjacent `channel: "last"`, `bestEffort`, and `failureDestination` behavior cover different pre-delivery or failure-notification semantics.

## What This Changes

Add a `fallbackChannel` or `fallbackChannels` hook mapping option so `deliver: true` can retry alternate channels after primary delivery fails.

## Best Possible Solution

Define a typed ordered hook delivery fallback contract and carry it through config schema/help, mapping transforms, hook dispatch, cron delivery storage, outbound retry-after-primary-failure handling, docs, and focused regression tests.

## Reproduction Assessment

Yes, source-level. Current main rejects the proposed fallback keys in strict hook config validation, carries only one `channel`/`to` through hook mapping and dispatch, and sends one resolved cron delivery target.

## Solution Assessment

Unclear pending maintainer product direction. The requested fallback chain is plausible, but the maintainable fix should be a typed ordered contract rather than treating `channel: "last"`, `bestEffort`, `failureDestination`, or pre-delivery selection helpers as equivalent.
