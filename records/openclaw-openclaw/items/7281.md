## Summary

Keep open: current main still lacks configurable post-failure fallback delivery for hook mappings, and the existing channel selection, `bestEffort`, and `failureDestination` mechanisms cover different semantics.

## What This Changes

The issue requests a `fallbackChannel` or ordered `fallbackChannels` option for hook mappings so failed primary delivery can retry through another channel.

## Best Possible Solution

Define a typed ordered hook delivery fallback contract and carry it through config schema/help, mapping transforms, hook dispatch, cron delivery storage, outbound retry-after-primary-failure handling, docs, and focused regression tests.

## Reproduction Assessment

Yes, source-level. Current main rejects the proposed fallback keys, carries only one `channel`/`to` through hook mapping and dispatch, and sends one resolved cron delivery target.

## Solution Assessment

Unclear pending maintainer product direction. The requested direction is plausible, but the maintainable solution should be an explicit ordered fallback contract rather than reusing `channel: "last"`, `bestEffort`, `failureDestination`, or pre-delivery selection helpers as post-failure failover.
