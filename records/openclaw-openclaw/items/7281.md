## Summary

Keep open: current main still lacks hook-mapping delivery failover, and the requested `fallbackChannel`/`fallbackChannels` behavior would add a new public hook config and retry policy that needs maintainer product direction.

## What This Changes

Add an ordered fallback channel option for hook mappings so hook-delivered agent replies can try another channel when the primary delivery target fails.

## Best Possible Solution

Define a typed ordered hook delivery fallback contract and carry it through config schema/help, mapping resolution, dispatch payloads, cron delivery storage, outbound retry/failure handling, docs, and focused regression coverage.

## Reproduction Assessment

Yes, at source level: proposed fallback keys are rejected by strict config validation, hook mapping and dispatch carry only one channel/to target, and cron delivery sends to one resolved target. I did not run a live channel outage because the missing public contract is visible on current main.

## Solution Assessment

Unclear pending maintainer product decision. The requested direction is plausible, but the maintainable solution should be a typed ordered fallback chain rather than treating `channel: "last"`, `bestEffort`, `failureDestination`, or existing outbound fallback helpers as equivalent.
