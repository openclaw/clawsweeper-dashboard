## Summary

Keep open. Current main still has no hook-mapping `fallbackChannel` or ordered fallback delivery chain; the existing fallback helpers are pre-delivery channel selection paths, not retry-after-primary-send-fails behavior.

## What This Changes

The issue requests configurable fallback channel delivery for hook mappings when the primary outbound channel fails.

## Best Possible Solution

Define an ordered, typed hook delivery fallback contract and propagate it through hook config/schema/help, mapping transforms, `HookAgentDispatchPayload`, `CronDelivery`, outbound send retries, docs, and focused regression tests.

## Reproduction Assessment

No, not as a bug reproduction; this is a feature request. Source inspection on current main gives high confidence that the proposed mapping keys and post-failure retry path are absent.

## Solution Assessment

Unclear until maintainers choose the public API and failure semantics. An explicit ordered fallback target contract is the maintainable direction, but it should stay distinct from `channel: "last"`, `bestEffort`, and `failureDestination`.
