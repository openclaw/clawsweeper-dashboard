## Label Justifications

- `P2`: This is a valid delivery reliability enhancement with bounded blast radius, but it requires product semantics rather than urgent regression repair.
- `impact:message-loss`: When primary hook delivery fails after the agent response is generated, the notification can reach no user-visible channel.

## Summary

Keep open: current main and v2026.6.6 still lack hook-mapping post-failure delivery failover, and existing fallback paths are pre-delivery selection or failure-notification mechanisms rather than an ordered retry chain after a send failure.

## What This Changes

The issue requests a `fallbackChannel` or ordered `fallbackChannels` hook-mapping option so generated hook responses can be delivered through another channel when the primary channel send fails.

## Best Possible Solution

Define a shared ordered delivery-failover contract at the cron/outbound boundary, then let hook mappings populate it through typed config, docs, and tests after maintainers settle retry and safety semantics.

## Reproduction Assessment

Yes. At source level, hook config, strict schema, mapping resolution, hook dispatch, cron delivery, and final outbound send all expose one delivery target and no post-failure retry chain.

## Solution Assessment

No, not as an immediate hook-only `fallbackChannel` field. The safer maintainable solution is a shared ordered delivery-failover contract whose retry, target-pairing, duplicate-delivery, and failure-reporting semantics are decided first.
