## Summary

Keep open: current main still lacks a hook-mapping fallback delivery contract, and the existing fallback helpers only select or report a single route rather than trying an ordered post-failure channel chain.

## What This Changes

The issue requests a `fallbackChannel` or `fallbackChannels` hook mapping option so a failed primary delivery can retry through another channel.

## Best Possible Solution

Define a typed ordered hook delivery fallback contract and carry it through config schema/help, mapping resolution, dispatch payloads, cron delivery storage, outbound retry/failure handling, docs, and focused regression coverage.

## Reproduction Assessment

Yes, at source level: the proposed keys are rejected by the strict schema, the hook mapping/dispatch contract carries only one `channel`/`to`, and cron direct delivery sends to one resolved target. I did not run a live channel outage because the missing contract is visible in current main.

## Solution Assessment

Unclear pending maintainer product decision. The requested direction is plausible, but the maintainable implementation should be a typed ordered fallback chain rather than treating `channel: "last"`, `bestEffort`, or `failureDestination` as equivalent.
