## Summary

Keep open. Current main still models hook delivery as one channel/target, and the existing fallback-named paths are pre-delivery selection or failure-notification helpers rather than post-failure hook delivery failover.

## What This Changes

The issue requests a configurable hook delivery fallback channel or ordered fallback chain when the primary hook delivery channel fails.

## Best Possible Solution

Define and document an ordered hook delivery fallback contract, then carry it through hook config/schema, mapping transforms, HookAgentDispatchPayload, CronDelivery, outbound delivery, and focused failover tests.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source, docs, and tests on current main give high-confidence proof that the proposed mapping keys and post-failure retry loop are absent.

## Solution Assessment

Unclear until maintainers choose the public API and failure semantics. An explicit ordered fallback target contract looks like the maintainable direction, but it should remain distinct from pre-delivery channel selection, bestEffort delivery, and failureDestination notifications.
