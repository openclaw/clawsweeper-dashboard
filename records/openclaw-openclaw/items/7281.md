## Label Justifications

- `P2`: This is a valid reliability improvement with bounded user impact, but it requires product semantics rather than an urgent regression repair.
- `impact:message-loss`: When primary hook delivery fails, the generated notification can reach no user-visible channel.

## Summary

Current main and the latest release still support only one hook delivery target, while the related recovery PR closed unmerged and the newer durable-final issue covers delivery outcomes rather than cross-channel retry. This remains a valid feature request, but the safest implementation requires a maintainer product decision on a shared delivery-failover contract rather than a hook-only config field.

## What This Changes

Requests configurable fallback delivery channels for hook mappings when the primary channel send fails.

## Best Possible Solution

Define an ordered, typed delivery-target list at the shared cron/outbound owner boundary, then let hook mappings populate it with fail-closed target validation, deduplication, per-attempt outcomes, and explicit all-targets-failed behavior.

## Reproduction Assessment

Yes at source level: current hook config, normalized payload, cron delivery, and outbound dispatch expose one delivery target and no retry-after-failure chain. No runtime test was run because this read-only review is assessing a missing capability rather than a broken existing contract.

## Solution Assessment

No, not yet as a hook-specific `fallbackChannel` field; the narrower maintainable direction is a shared ordered delivery-target contract whose retry, target-pairing, and duplicate-delivery semantics are decided first.
