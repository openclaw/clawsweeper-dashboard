- `impact:message-loss`: When primary hook delivery fails, the generated notification can reach no user-visible channel.

## Summary

Keep open: current main and the latest release still model hook delivery as a single channel/target, while the related recovery PR closed unmerged and the broader durable-final issue does not replace this focused hook failover request.

## What This Changes

The issue requests configurable hook mapping fallback channels that retry another delivery channel after the primary hook delivery channel fails.

## Best Possible Solution

Define an ordered, typed delivery-target list at the shared cron/outbound boundary, then let hook mappings populate it with explicit validation, deduplication, per-attempt outcomes, and documented all-targets-failed behavior.

## Reproduction Assessment

Yes, at source level: hook config, schema, mapping resolution, hook dispatch, cron delivery, and outbound send selection expose one delivery target and no post-failure retry chain. I did not run a live channel repro because this read-only review is assessing a missing capability.

## Solution Assessment

No, not as an immediate hook-only fallbackChannel field. The safer maintainable solution is a shared ordered delivery-failover contract whose retry, target-pairing, and duplicate-delivery semantics are decided first.
