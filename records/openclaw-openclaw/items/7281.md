## Label Justifications

- `P2`: This is a normal-priority reliability enhancement with a clear hook delivery surface, but it requires public API and retry-semantics decisions before implementation.
- `impact:message-loss`: The reported failure mode is missed hook notifications when the configured primary delivery channel is unavailable and no alternate channel can be tried.

## Summary

Keep open: current main and the latest release still model hook mapping delivery as one channel/target, while the existing fallback helpers are pre-delivery selection paths rather than post-failure hook failover. The related recovery PR was closed unmerged and does not supersede this public hook config/API decision.

## What This Changes

Request adds fallbackChannel or fallbackChannels to hook mappings so deliver:true can try another channel after the primary delivery channel fails.

## Best Possible Solution

Define an owner-approved ordered hook delivery fallback contract and carry it through config schema/help, mapping transforms, hook dispatch, cron delivery storage, outbound retry-after-primary-failure handling, docs, and focused regression tests.

## Reproduction Assessment

Yes, source-level. Current main rejects the proposed fallback keys in strict hook mapping validation, carries only one channel/to through mapping and dispatch, and sends one resolved cron delivery target.

## Solution Assessment

Unclear pending maintainer product direction. The requested fallback chain is plausible, but the maintainable solution should be a typed ordered contract rather than treating channel="last", bestEffort, failureDestination, or pre-delivery selection helpers as equivalent.
