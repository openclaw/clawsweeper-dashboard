## Summary

Keep open: current main and the latest release still model hook delivery as one channel/target, and the related closed PR did not land a post-failure hook delivery fallback chain.

## What This Changes

The issue requests a `fallbackChannel` or ordered `fallbackChannels` hook mapping option that retries another delivery channel after the primary hook delivery send fails.

## Best Possible Solution

Define an owner-approved ordered hook delivery fallback contract and thread it through config schema/types, mapping transforms, hook dispatch, cron delivery storage, outbound send failover, docs, and focused regression tests.

## Reproduction Assessment

Yes, source-level: current main rejects the proposed fallback keys, carries only one channel/to route through mapping and dispatch, and sends one resolved cron delivery target.

## Solution Assessment

Unclear pending maintainer product direction. A fallback chain is plausible, but it should be a typed ordered contract rather than overloading `channel: "last"`, `bestEffort`, `failureDestination`, or pre-delivery fallback helpers.
