## Summary

Keep open. Current main still models hook delivery as one channel/target, and the existing `fallbackChannel` names in outbound code are pre-delivery target-selection helpers rather than post-failure hook failover.

## What This Changes

Add a hook mapping fallback channel option so delivered hook agent responses can retry an alternate channel when the primary send fails.

## Best Possible Solution

Define an ordered hook delivery fallback contract, then propagate it through hook config/schema, mapping transforms, `HookAgentDispatchPayload`, `CronDelivery`, outbound delivery, docs, and focused tests.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection on current main confirms the proposed hook mapping keys and post-failure retry loop are absent.

## Solution Assessment

Unclear until maintainers choose the public API shape. An explicit ordered fallback target contract looks like the maintainable direction, but it should stay distinct from `channel: "last"`, `bestEffort`, and `failureDestination`.
