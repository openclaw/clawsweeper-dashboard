## Summary

Keep open: current main and the latest release still model hook delivery as one channel/target, and the related closed PR did not land a post-failure hook delivery fallback chain.

## What This Changes

Add a configurable hook mapping fallback channel or ordered fallback channel list when primary hook delivery fails.

## Best Possible Solution

Define an owner-approved ordered hook delivery fallback contract and thread it through config schema/types, mapping transforms, hook dispatch, CronDelivery storage, outbound send failover, docs, and focused regression tests.

## Reproduction Assessment

Yes, source-level: current main rejects the proposed fallback keys, carries only one channel/to route through mapping and dispatch, and sends one resolved cron delivery target.

## Solution Assessment

Unclear pending maintainer product direction. The fallback chain is plausible, but it should be a typed ordered contract rather than overloading channel="last", bestEffort, failureDestination, or pre-delivery fallback helpers.
