## Summary

Keep open. Current main still lacks hook-mapping `fallbackChannel`/`fallbackChannels` support, and this is a new public hook config plus delivery failover behavior rather than an already-defined bug fix.

## What This Changes

Add a configurable fallback channel or ordered fallback channel list for hook mappings when `deliver: true` primary delivery fails.

## Best Possible Solution

Define a typed ordered hook-delivery failover contract in core, propagate it through config/schema, mapping transforms, hook dispatch, `CronDelivery`, outbound delivery, docs, and focused regression tests.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection on current main confirms the proposed hook mapping keys and post-failure delivery retry loop are absent.

## Solution Assessment

Unclear until maintainers choose the public API shape. An explicit ordered fallback target contract is the maintainable direction, but it should not reuse `channel: "last"`, `bestEffort`, or `failureDestination` for different semantics.
