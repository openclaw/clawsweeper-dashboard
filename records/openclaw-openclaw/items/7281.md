## Summary

Keep open. Current main still has single-target hook delivery, and the existing fallback-related code covers pre-delivery target selection, failure notifications, or best-effort behavior rather than retrying an alternate hook delivery channel after the primary send fails.

## What This Changes

Add a hook mapping option such as `fallbackChannel` or `fallbackChannels` so delivered hook agent results can retry alternate channels when the primary channel send fails.

## Best Possible Solution

Define a typed hook-delivery fallback contract, propagate it through config schema, mapping transforms, hook dispatch payloads, `CronDelivery`, and direct cron delivery, then document and test primary-fails/fallback-succeeds and all-targets-failed behavior.

## Reproduction Assessment

Yes. The source reproduction is high-confidence: the proposed fallback field is rejected by the strict hook mapping schema and no fallback field can propagate through mapping, dispatch, `CronDelivery`, or the final `deliverOutboundPayloads` call.

## Solution Assessment

Unclear until maintainers choose the public API shape. The requested direction is valid, but `channel: "last"`, `bestEffort`, `failureDestination`, and existing `fallbackChannel` helper variables do not solve post-failure hook delivery failover.
