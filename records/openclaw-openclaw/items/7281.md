## Summary

Keep open: current main still lacks a hook-mapping delivery fallback contract, and the existing fallback helpers only choose one target before sending rather than trying an ordered chain after primary delivery fails.

## What This Changes

Add a configurable `fallbackChannel` or `fallbackChannels` option for hook mappings so failed primary delivery can retry through alternate channels.

## Best Possible Solution

Define a typed ordered hook delivery fallback contract and carry it through config schema/help, mapping resolution, dispatch payloads, cron delivery storage, outbound retry/failure handling, docs, and focused regression coverage.

## Reproduction Assessment

Yes, at source level: the proposed keys are rejected by strict config validation, the hook mapping/dispatch contract carries only one `channel`/`to`, and cron delivery sends to one resolved target. I did not run a live channel outage because the missing contract is visible in current main.

## Solution Assessment

Unclear pending maintainer product decision. The requested direction is plausible, but the maintainable solution should be a typed ordered fallback chain rather than treating `channel: "last"`, `bestEffort`, or `failureDestination` as equivalent.
