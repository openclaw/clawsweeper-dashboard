## Label Justifications

- `P2`: This is a normal-priority reliability feature affecting a bounded hook delivery workflow but requiring cross-cutting design work.
- `impact:message-loss`: When the primary transport is unavailable, a successfully generated hook notification can remain undelivered.

## Summary

Current main and v2026.6.5 still lack post-failure hook delivery failover, and no viable canonical issue or merged PR owns it. The request remains useful, but it adds a public config/fallback surface and cross-channel delivery policy, so maintainers should decide the destination and safety semantics before implementation.

## What This Changes

Add configurable ordered alternate destinations for hook-generated output when its primary channel delivery fails.

## Best Possible Solution

Define an ordered typed destination list with explicit channel, account, recipient, and thread fields; carry it through the existing cron delivery plan and attempt the next destination only after a classified primary failure, with clear duplicate and all-destinations-failed reporting.

## Reproduction Assessment

Yes at source level: current main carries one hook channel/recipient into one cron delivery plan, so there is no path that can retry the successful result on an alternate channel after a send failure. A live disconnected-channel run was not performed in this read-only review.

## Solution Assessment

No final implementation shape is approved yet. An ordered typed destination model is safer and more maintainable than a channel-only fallback because alternate transports also need recipient, account, thread, ordering, and duplicate-send semantics.

- Adding fallbackChannel versus fallbackChannels creates a durable public config contract that must align across schema, docs, transforms, cron persistence, and delivery behavior.
