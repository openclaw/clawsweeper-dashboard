## Summary

Keep open: current main and v2026.6.8 still lack hook-mapping post-failure delivery failover, and the existing fallback paths are pre-delivery selection, best-effort session downgrade, or failure notification rather than an ordered retry chain after a primary send fails.

## What This Changes

Request to add `fallbackChannel` or `fallbackChannels` to hook mappings so a failed primary channel delivery can retry an alternate channel.

## Best Possible Solution

Define ordered delivery failover at the shared cron/outbound delivery boundary, then expose typed hook mapping config only after retry, duplicate-send, target-pairing, and failure-reporting semantics are settled and tested.

## Reproduction Assessment

Yes, source-level. Current main has a clear one-target path from hook config through mapping, hook dispatch, cron delivery, and outbound send; I did not run a live WhatsApp/webchat repro in this read-only review.

## Solution Assessment

No, not as a standalone hook-only field. The maintainable solution is a shared ordered delivery-failover contract that hook mappings can populate after retry and safety semantics are decided.
