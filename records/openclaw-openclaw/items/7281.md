## Summary

Keep open: current main and v2026.6.10 still lack hook-mapping post-failure delivery failover, and the existing fallback paths are pre-delivery target selection, session-only best-effort fallback, or failure notification rather than an ordered retry chain after the primary send fails.

## What This Changes

The issue requests configurable fallback delivery channels for hook mappings when the primary channel send fails.

## Best Possible Solution

Define ordered delivery failover at the cron/outbound boundary, then expose typed hook mapping config after retry, unknown-outcome, duplicate-send, target-pairing, and failure-reporting semantics are settled.

## Reproduction Assessment

Yes, source-level. Current main has a clear one-target path from hook config through mapping, hook dispatch, cron delivery, and outbound send; I did not run a live WhatsApp/webchat failure scenario in this read-only review.

## Solution Assessment

No, not as a standalone hook-only field. The maintainable solution is a shared ordered delivery-failover contract that hook mappings can populate after retry and safety semantics are decided.
