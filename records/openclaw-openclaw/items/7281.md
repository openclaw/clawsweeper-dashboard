## Summary

Keep this open: current main and the latest release still support only a single hook delivery target, while the remaining work is a compatibility-sensitive delivery fallback design rather than stale cleanup.

## What This Changes

Request adding `fallbackChannel` or `fallbackChannels` to hook mappings so `deliver: true` hook responses can retry another channel after the primary send fails.

## Best Possible Solution

Keep this open for a maintainer-owned typed delivery-failover contract that threads ordered fallback targets through hook mappings, hook dispatch, cron delivery, outbound failure handling, docs, and regression tests.

## Reproduction Assessment

Yes for the missing capability via source inspection: current main has no accepted fallback hook config field and the dispatch path carries one delivery target. I did not run a live WhatsApp failure scenario.

## Solution Assessment

Unclear; the proposed field is plausible, but the best solution should define shared post-failure delivery semantics instead of bolting a hook-only fallback onto one mapping path.
