## Label Justifications

- `P2`: This is a normal-priority reliability enhancement for automated notifications with a limited, configuration-dependent blast radius.
- `impact:message-loss`: A successful hook run can produce no user-visible notification when its single configured delivery channel is unavailable.

## Summary

Current main still cannot express post-failure hook delivery failover, and the existing cron failure destination sends failed-run notifications rather than retrying the successful hook output; keep this open for a maintainer product decision on one shared delivery policy.

## What This Changes

Add configurable alternate delivery channels for hook mappings so generated hook output can be delivered after the primary channel send fails.

## Best Possible Solution

Define one shared typed delivery policy for primary completion delivery, ordered alternate attempts, and final failure notification, then carry it from hook mapping through cron dispatch with focused success, failover, deduplication, and all-targets-failed tests.

## Reproduction Assessment

Yes at source level: current main models only one hook delivery channel and target, so the requested post-send alternate route cannot be configured or propagated.

## Solution Assessment

No, the proposed hook-only field is not yet proven to be the best fix; maintainers should first define shared semantics with cron completion delivery, `bestEffort`, and `failureDestination`.
