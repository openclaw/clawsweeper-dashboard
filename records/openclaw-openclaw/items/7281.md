## Summary

Keep open: current main still models delivered hook mappings as a single `channel`/`to` target, and the existing `last`, `bestEffort`, `failureDestination`, and outbound `fallbackChannel` helpers do not provide post-failure hook delivery failover. This is a real feature request that needs maintainer product direction before implementation.

## What This Changes

Requests a `fallbackChannel` or `fallbackChannels` hook mapping option so hook agent delivery can retry another channel after the primary channel send fails.

## Best Possible Solution

Define a typed ordered hook delivery fallback contract and carry it through config schema/help, hook mapping transforms, dispatch payloads, `CronDelivery` storage, outbound post-failure retry handling, docs, and focused regression tests.

## Reproduction Assessment

Yes, at source level. The proposed fallback keys are rejected by strict hook config validation, hook mapping and dispatch carry only one `channel`/`to`, and cron delivery sends to one resolved target.

## Solution Assessment

Unclear pending maintainer product direction. The requested direction is plausible, but the durable solution should be a typed ordered fallback contract rather than treating `channel: "last"`, `bestEffort`, `failureDestination`, or existing pre-delivery fallback helpers as equivalent.
