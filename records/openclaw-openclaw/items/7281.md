## Summary

Keep open. Current main still has only a single hook delivery target and the recent issue discussion correctly distinguishes this from existing pre-delivery channel selection, best-effort session fallback, and failure-notification destinations.

## What This Changes

Request to add a configurable fallback channel or fallback chain for hook mappings when the primary delivered hook response cannot be sent.

## Best Possible Solution

Define an ordered, typed hook delivery fallback contract and propagate it through hook config/schema/help, mapping transforms, HookAgentDispatchPayload, CronDelivery, outbound retry handling, docs, and focused regression tests.

## Reproduction Assessment

No, not as a bug reproduction; this is a feature request for a new hook mapping option. Source inspection gives high confidence that the requested config keys and post-primary-send retry path are absent on current main.

## Solution Assessment

Unclear until maintainers choose the public API and failure semantics. An explicit ordered fallback target contract is likely the maintainable direction, but it must stay distinct from channel:"last", bestEffort, and failureDestination.
