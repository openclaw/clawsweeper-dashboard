## Summary

Keep open: current main still has no configurable post-failure fallback chain for hook mapping delivery, and the related recovery PR remains open, broad, and does not settle the public hook config/API decision.

## What This Changes

The issue requests a `fallbackChannel` or ordered `fallbackChannels` hook mapping option so failed primary delivery can retry another channel.

## Best Possible Solution

Define an owner-approved ordered hook delivery fallback contract, carry it through config schema/help, mapping transforms, hook dispatch, cron delivery storage, outbound retry-after-primary-failure handling, docs, and focused regression tests.

## Reproduction Assessment

Yes, source-level. Current main rejects the proposed fallback keys in strict hook mapping validation, carries only one `channel`/`to` through mapping and dispatch, and sends one resolved cron delivery target.

## Solution Assessment

Unclear pending maintainer product direction. The requested fallback chain is plausible, but the maintainable solution should be a typed ordered contract rather than treating `channel: "last"`, `bestEffort`, `failureDestination`, or pre-delivery selection helpers as equivalent.
