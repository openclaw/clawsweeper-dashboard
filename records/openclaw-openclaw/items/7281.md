## Summary

Keep open: current main still lacks a hook-mapping delivery fallback contract, and the existing fallback helpers resolve one channel before send rather than trying an ordered fallback after primary delivery failure.

## What This Changes

The issue requests a `fallbackChannel` or `fallbackChannels` hook mapping option so hook-delivered replies can fail over when the primary channel send fails.

## Best Possible Solution

Define a typed ordered hook delivery fallback contract and carry it through config schema/help, mapping resolution, dispatch payloads, cron delivery storage, outbound retry/failure handling, docs, and focused regression coverage.

## Reproduction Assessment

Yes, at source level: the proposed keys are rejected by strict config validation, the hook mapping and dispatch contracts carry only one `channel`/`to`, and cron delivery sends to one resolved target. I did not run a live WhatsApp outage because the missing public contract is visible in current main.

## Solution Assessment

Unclear pending maintainer product decision. The requested direction is plausible, but the maintainable solution should be a typed ordered fallback chain rather than treating `channel: "last"`, `bestEffort`, `failureDestination`, or outbound pre-selection helpers as equivalent.
