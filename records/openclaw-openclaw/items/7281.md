## Summary

Keep open. Current main still has single-target hook delivery, and the existing fallback-related helpers cover pre-delivery target selection, session fallback, or failure notifications rather than retrying another hook delivery channel after a failed primary send.

## What This Changes

Add a `fallbackChannel` or ordered fallback channel option to hook mappings so failed primary hook delivery can retry another channel.

## Best Possible Solution

Define and document a typed hook-delivery failover contract, then propagate it through config, mapping, dispatch, cron delivery, and outbound send handling with focused regression coverage.

## Reproduction Assessment

Yes. The source reproduction is high-confidence: the proposed fallback field is rejected by the strict hook mapping schema and no fallback field can propagate through mapping, dispatch, `CronDelivery`, or the final outbound delivery call.

## Solution Assessment

Unclear until maintainers choose the public API shape. The requested direction is valid, but existing `channel: "last"`, `bestEffort`, `failureDestination`, and outbound fallback helper logic do not solve post-failure hook delivery failover.
