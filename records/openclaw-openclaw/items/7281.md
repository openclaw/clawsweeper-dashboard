## Summary

Keep open. Current main still only models one hook delivery channel/target, the strict hook mapping schema rejects the proposed fallback keys, and the final cron delivery path fails or best-effort logs instead of trying an ordered alternate channel chain.

## What This Changes

Add `fallbackChannel` or `fallbackChannels` to hook mappings so `deliver: true` hooks can retry alternate channels after primary delivery fails.

## Best Possible Solution

Define a typed ordered hook-delivery failover contract and wire it through hook config/schema, mapping transforms, hook dispatch, cron delivery, outbound send behavior, docs, and focused tests for primary success, fallback success, and all-targets-failed reporting.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection of current main confirms the proposed hook mapping keys and post-failure retry loop are absent.

## Solution Assessment

Unclear until maintainers choose the public API shape. An explicit ordered fallback target contract is likely the maintainable direction, but it should not reuse `channel: "last"`, `bestEffort`, or `failureDestination` for different semantics.
