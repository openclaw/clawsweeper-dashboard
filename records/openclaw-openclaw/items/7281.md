## Summary

Keep open. Current main still has a single-target hook delivery contract, and the requested post-failure fallback chain remains a new public hook config and delivery behavior rather than an implemented fix.

## What This Changes

The issue requests a configurable fallback channel or ordered fallback channels for hook mapping delivery when the primary channel fails.

## Best Possible Solution

Define a typed ordered hook-delivery failover contract in core, propagate it through config/schema, mapping transforms, hook dispatch, `CronDelivery`, outbound delivery, docs, and regression tests for primary-fails/fallback-succeeds and all-targets-failed behavior.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection on current main confirms the proposed hook mapping keys and post-failure delivery retry loop are absent.

## Solution Assessment

Unclear until maintainers choose the public API shape. The maintainable direction is an explicit ordered fallback target contract, not reusing `channel: "last"`, `bestEffort`, or `failureDestination` for different semantics.
