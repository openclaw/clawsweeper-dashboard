## Summary

Keep open: current main and the latest shipped tag still lack a configurable post-failure fallback channel chain for hook mapping delivery, and the related open recovery PR is broad, unmerged, and not a replacement for this public hook config decision.

## What This Changes

The issue requests a configurable fallback delivery channel or ordered fallback chain for hook mappings when the primary channel send fails.

## Best Possible Solution

Define an owner-approved ordered hook delivery fallback contract and carry it through config schema/help, mapping transforms, hook dispatch, cron delivery storage, outbound retry-after-primary-failure handling, docs, and focused regression tests.

## Reproduction Assessment

Yes, source-level for the missing capability. Current main rejects the proposed fallback keys in strict hook mapping validation, carries only one `channel`/`to` through mapping and dispatch, and sends one resolved cron delivery target.

## Solution Assessment

Unclear pending maintainer product direction. The requested fallback chain is plausible, but the maintainable solution should be a typed ordered contract rather than treating `channel: "last"`, `bestEffort`, `failureDestination`, or pre-delivery selection helpers as equivalent.
