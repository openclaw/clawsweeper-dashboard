- `impact:auth-provider`: The issue concerns OAuth prompt lifecycle and provider-owned callback behavior for the OpenAI/ChatGPT auth path.

## Summary

Keep this open: current main and the latest release still start a remote manual paste prompt while the browser callback can win, and there is still no explicit callback-settled or prompt-cancellation contract. The related merged wording PR documented the caveat, and the later OAuth centralization kept the behavior plugin-owned, but neither implemented the lifecycle signal requested here.

## What This Changes

The issue requests an explicit callback-settled signal so remote OpenAI Codex OAuth can dismiss manual paste input after browser callback completion.

## Best Possible Solution

Add a narrow provider-owned callback-settled or cancellable-prompt contract in the OpenAI plugin/generic OAuth seam, with tests for paste-only and SSH-forwarded callback paths and no token-exchange inference or config flag by default.

## Reproduction Assessment

Yes, source-level. Current main starts remote paste input immediately, the OAuth flow can complete through the browser callback, and no callback-settled or prompt-cancellation signal reaches the remote manual prompt path.

## Solution Assessment

Yes as a direction, but the exact API still needs maintainer choice. A provider-owned signal plus a generic prompt cancellation seam is cleaner than delaying paste input, forcing paste, or inferring success from token exchange.
