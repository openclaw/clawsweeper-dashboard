## Summary

Keep open. Current main still rejects external-target symlinked personal/project `.agents/skills` directories, and the PR addresses that behavior, but it needs maintainer approval because it relaxes a skill-loading containment boundary and lacks the required docs/changelog follow-through.

## What This Changes

The PR adds a logical-path containment fallback for personal and project `.agents/skills` sources, plus tests for symlinked personal/project skills and managed-source rejection.

## Best Possible Solution

Land this PR or an equivalent narrow patch only if maintainers explicitly want personal and project `.agents/skills` symlinked directories to be supported when their real targets live outside the configured root; keep strict physical containment for managed, bundled, workspace, extra, and plugin-owned roots, then document and release-note the exception.

## Reproduction Assessment

Yes. Current main discovers symlinked child directories, then rejects their `SKILL.md` paths when `resolveContainedSkillPath` realpaths them outside the configured root; a `~/.agents/skills/foo -> /external/foo` setup should hit that path.

## Solution Assessment

Unclear. The code change is narrow for the requested workflow, but it relaxes a containment boundary and is broader than the configured-root-only approach in #45912, so maintainers should choose the policy before merge.
