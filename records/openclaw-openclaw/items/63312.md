## Summary

Keep open. Current main still rejects the symlinked personal/project `.agents/skills` workflow this PR targets, and the PR is a narrow implementation candidate, but it should not merge yet because it relaxes a skill-loading containment boundary without docs, has Windows-unsafe symlink tests, and lacks the required user-facing changelog entry.

## What This Changes

The PR allows logical-path containment for personal and project `.agents/skills` symlinks while keeping other skill sources on realpath containment, and adds regression tests for personal/project symlink loading plus managed-source rejection.

## Best Possible Solution

Land this PR or an equivalent narrow patch only after maintainers explicitly approve the personal/project `.agents/skills` external-target symlink policy. The landed version should keep strict physical containment for managed, bundled, workspace, extra, and plugin-owned roots; make the symlink tests cross-platform-safe; document the exception in the public Skills security docs; and add the required changelog entry.

## Reproduction Assessment

Yes. Current main discovers symlinked skill directories, then rejects their `SKILL.md` when the real target is outside the configured root; the source trace through `listChildDirectories`, `loadSkills`, and `resolveContainedSkillPath` gives a high-confidence reproduction path without running mutating tests.

## Solution Assessment

Unclear. The implementation is narrow for the requested `.agents/skills` exception, but it is more permissive than the related configured-root-only symlink approach in #45912 and needs maintainer approval plus docs/test/changelog follow-through before it is the best maintainable fix.
