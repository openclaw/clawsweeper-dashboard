## Summary

Keep this PR open. Current main still lacks core MCP SecretRef support and the PR addresses the earlier review comments plus the changelog gap, but the new runtime collector treats stdio MCP env SecretRefs as active even for env keys that the launch path will drop for startup safety.

## What This Changes

The PR adds SecretInput support for core `mcp.servers.*.env.*` and `headers.*`, with schema/type changes, runtime resolution, audit/configure handling, target registry entries, docs, changelog, and tests.

## Best Possible Solution

Land this focused implementation after the MCP SecretRef collector mirrors the existing stdio env blocked-key policy and secops/codeowner review signs off on the credential surface.

## Reproduction Assessment

Yes. A source-level reproduction is a stdio MCP server with `env.NODE_OPTIONS` set to a SecretRef: the PR collector marks it active and resolves/requires it, while the existing stdio launch path drops `NODE_OPTIONS` before spawning.

## Solution Assessment

No, not yet. The overall approach is the right narrow fix for core MCP SecretRefs, but the runtime collector must skip or mark inactive blocked stdio env keys before this is the safest maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Full review comments:

- **[P2] Mark blocked MCP env refs inactive:** `src/secrets/runtime-config-collectors-core.ts:217-224`
