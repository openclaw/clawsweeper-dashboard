## What This Changes

The PR adds `card_msg_content_type: "user_card_content"` to Feishu single-message and thread-history reads and asserts those request params in `send.test.ts`.

## Best Possible Solution

Land a Feishu-owned fix that requests full card JSON through a type-safe SDK/raw-request boundary, preserves parser fallbacks, adds request-param regression tests, includes a changelog entry, and shows redacted live Feishu read proof.

## Reproduction Assessment

Yes from source and upstream docs, but not from a live tenant. Current main omits `card_msg_content_type`, while Feishu documents `user_card_content` as required for original card JSON on both affected APIs.

## Solution Assessment

No, not as written. Adding the parameter is the right narrow direction, but the implementation must account for the current SDK type surface and the PR still needs real behavior proof plus the required changelog entry.
