## Summary

Keep open. Current main still lacks the Telegram adapter-level silent-reply guard, and open PR #68304 is the active implementation candidate with closing syntax for this issue.

## What This Changes

The issue requests a Telegram send adapter guard that suppresses bare `NO_REPLY` messages before the Telegram API path, matching Slack’s defensive guard.

## Best Possible Solution

Review and land #68304 or an equivalent narrow Telegram plugin patch that imports the shared helper, suppresses only exact bare silent replies before Telegram API work, preserves media/buttons/thread behavior, and adds focused adapter coverage.

## Reproduction Assessment

Yes, at source level: `sendMessageTelegram("123", "NO_REPLY", opts)` currently has no adapter-level silent guard and reaches the normal text-send path when no media/buttons are present. I did not run a live Telegram send in this read-only review.

## Solution Assessment

Yes. A narrow adapter-level guard is the smallest maintainable fix because it matches Slack’s established behavior without disabling Telegram preview streaming globally; the active PR should be checked for media and inline-button bypass semantics before merge.
