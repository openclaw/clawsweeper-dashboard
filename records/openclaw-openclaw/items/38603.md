## Summary

Keep open: current main still lacks the Telegram send-level `NO_REPLY` guard, and https://github.com/openclaw/openclaw/pull/68304 remains the active closing implementation candidate.

## What This Changes

The issue requests a Telegram `sendMessageTelegram` guard that suppresses exact `NO_REPLY` sends before any Telegram API call, matching Slack's send-level behavior.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/68304 or an equivalent narrow Telegram guard with regression coverage; keep broader stream-buffering behavior as a separate follow-up if still needed.

## Reproduction Assessment

Yes. Source inspection on current main shows a direct `sendMessageTelegram(..., "NO_REPLY", ...)` path has no exact silent-token early return before the Telegram `api.sendMessage` path, while Slack has that guard.

## Solution Assessment

Yes for the defensive-coding gap. The narrow maintainable fix is to add the Telegram-owned exact-sentinel guard with focused regression coverage, without changing global streaming behavior.
