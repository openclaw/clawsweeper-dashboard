## Summary

Keep open: current main still lacks the Telegram send-level exact `NO_REPLY` guard, and https://github.com/openclaw/openclaw/pull/68304 is already the active closing implementation candidate.

## What This Changes

Telegram should suppress exact bare `NO_REPLY` sends in `sendMessageTelegram` before any Telegram API call, matching the Slack send-level guard.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/68304 or an equivalent narrow Telegram-owned guard with regression coverage, keeping broader stream-buffering changes separate unless still needed.

## Reproduction Assessment

Yes. Source inspection on current main shows a direct `sendMessageTelegram(..., "NO_REPLY", ...)` has no exact silent-token early return before the Telegram `api.sendMessage` path, while Slack has that guard.

## Solution Assessment

Yes for the defensive send-level gap. The narrow maintainable fix is a Telegram plugin guard using the existing exact silent-token helper with focused tests; stream buffering should remain a separate follow-up only if the flash remains.
