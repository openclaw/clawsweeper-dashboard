## Summary

Current main already prevents the reported Telegram flash at the actual streaming boundary: uppercase `NO_REPLY` lead fragments such as `NO` are suppressed before partial delivery, and the Telegram draft lifecycle clears unfinalized previews. The matching fix merged through https://github.com/openclaw/openclaw/pull/33169 and shipped in `v2026.3.7`, so the later send-adapter proposals are redundant rather than the best fix.

## What This Changes

Prevent Telegram live previews or direct sends from briefly exposing content from responses ultimately suppressed as silent replies.

## Best Possible Solution

Keep the shared pre-delivery prefix suppression and Telegram draft cleanup as the canonical fix; do not add a duplicate send-adapter guard unless a separate direct-send leak is newly reproduced.

## Reproduction Assessment

No on current main: the exact `NO` lead fragment is classified as a silent-token prefix before partial delivery, and Telegram clears non-finalized previews; focused source tests cover those paths.

## Solution Assessment

No, the suggested `sendMessageTelegram` guard is not the best primary fix because it runs after the preview could already be visible; the shipped shared streaming-boundary fix addresses the actual flash mechanism.
