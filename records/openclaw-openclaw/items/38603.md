## Summary

Keep open: current main still lacks the Telegram send-level `NO_REPLY` guard, and an open PR already links this issue as its closure target.

## What This Changes

The issue asks for Telegram outbound sends to suppress a bare `NO_REPLY` sentinel before calling the Telegram API, matching the Slack guard.

## Best Possible Solution

Review and land #68304 or an equivalent narrow Telegram send-level guard with focused regression coverage; keep any broader stream-buffering behavior as a separate follow-up if needed.

## Reproduction Assessment

Yes by source inspection: a direct `sendMessageTelegram(..., "NO_REPLY", ...)` has no exact silent-token early return before the Telegram API send path, while Slack does. I did not run tests because this sweep is read-only and the source path is sufficient to establish the missing guard.

## Solution Assessment

Yes for the defensive-coding gap: add an exact-sentinel guard in Telegram send when there is no media or inline keyboard, plus a send test proving no API call occurs. That is narrower than changing global stream behavior and matches the established Slack contract.
