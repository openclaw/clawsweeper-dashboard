## Summary

Keep open. Current main still lacks the Telegram adapter-level silent-reply guard, and open PR #68304 is the active implementation candidate that explicitly closes this issue.

## What This Changes

Add a Telegram send adapter guard so bare `NO_REPLY` text is suppressed before any Telegram API call, matching Slack's existing behavior.

## Best Possible Solution

Review and land #68304 or an equivalent narrow Telegram plugin patch that imports the shared helper, suppresses only exact bare silent replies before Telegram API work, preserves media/buttons/thread behavior, returns a synthetic suppressed result, and adds focused adapter coverage.

## Reproduction Assessment

Yes, at source level: `sendMessageTelegram("123", "NO_REPLY", opts)` on current main passes the nonempty text path and reaches chunked `api.sendMessage` delivery because no adapter-level silent guard exists. I did not run a live Telegram turn in this read-only review.

## Solution Assessment

Yes. The narrow adapter guard is the most maintainable immediate fix because it matches Slack’s established pattern without changing Telegram preview streaming semantics; broader pre-sentinel buffering should remain separate.
