## Summary

Keep open. Current main still lacks the requested `isSilentReplyText` guard in `sendMessageTelegram`, while Slack has the parity guard and the shared SDK helper is available; the linked implementation PR #68304 is still the proper place to land or replace this fix.

## What This Changes

The issue requests a Telegram send adapter guard that suppresses bare `NO_REPLY`/silent-reply payloads before Telegram API calls, matching Slack behavior.

## Best Possible Solution

Keep the tracker open and land #68304 or an equivalent narrow Telegram plugin patch that imports the shared silent-reply helper, suppresses only exact bare silent replies before any Telegram API work, preserves media/buttons/thread behavior, returns a synthetic suppressed result, and adds focused `extensions/telegram/src/send.test.ts` coverage. Broader preview buffering should stay separate.

## Reproduction Assessment

Yes. A direct `sendMessageTelegram("123", "NO_REPLY", opts)` path on current main has no adapter-level guard and will proceed toward normal text chunk sending; the existing Telegram final-delivery suppression tests do not exercise that reusable adapter entry point.

## Solution Assessment

Yes. The narrow adapter guard is the best immediate fix because it matches Slack’s shipped pattern and closes the defensive gap without changing Telegram streaming semantics; any pre-sentinel preview buffering would be a separate, broader behavior change.
