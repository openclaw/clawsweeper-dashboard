## Summary

Keep open. Current main still lacks the requested Telegram send-adapter `isSilentReplyText` guard, Slack still has the parity guard, and open PR #68304 remains the paired implementation candidate with closing syntax for this issue. The reporter's agent-side workaround helps the observed flash, but it does not close the upstream adapter gap.

## What This Changes

Add a defensive `isSilentReplyText` guard to Telegram's `sendMessageTelegram` adapter so bare `NO_REPLY` payloads are suppressed before Telegram API calls, matching Slack's behavior.

## Best Possible Solution

Review and land PR #68304 if maintainers accept the narrow adapter parity fix, or replace it with an equivalent Telegram plugin patch that suppresses only exact bare silent replies before API calls while preserving media and reply-markup behavior. Broader preview-buffering semantics should remain separate from this focused adapter guard.
