## What This Changes

The issue asks for Telegram sends of an exact `NO_REPLY` sentinel to be suppressed before calling the Telegram API, matching the Slack adapter guard.

## Best Possible Solution

Review and land the open Telegram-owned guard PR or an equivalent narrow fix with regression coverage; keep broader live-stream buffering separate unless the flash remains after the send-level guard.

## Solution Assessment

Yes for the defensive send-level gap. The narrow maintainable solution is to add the Telegram plugin guard using the existing exact silent-token helper and focused tests; broader stream buffering should remain a separate follow-up if still needed.
