## Summary

Keep open: this PR is a useful fix direction for the linked local-embedding dirty-status bug, but the patch applies the `local.modelPath` identity rule to every local-transport embedding adapter instead of the llama.cpp/local adapter contract, and the supplied proof is mock-only. Related open PRs exist for the same issue, but none is merged and proof-positive enough to safely supersede this PR.

## What This Changes

The PR changes memory-core status identity resolution and tests so local embedding indexes configured with only `memorySearch.local.modelPath` compare against the local model path instead of `settings.model`.

## Best Possible Solution

Scope the status mirror to the llama.cpp/local adapter identity contract, preserve `settings.model` for other local transports unless a new adapter capability says otherwise, then require real local-GGUF CLI proof before merge.

## Reproduction Assessment

Yes, source-level: current main persists the initialized local provider model path but status-only identity expects `settings.model` or fallback before provider initialization. I did not run the real Docker/GGUF CLI path in this read-only review.

## Solution Assessment

No. The fix belongs in the status identity mirror, but keying it off `transport: "local"` is broader than the current public embedding-provider contract supports; it should be scoped to the llama.cpp/local adapter contract or an explicit capability.
