## Summary

This PR should stay open because the targeted llama.cpp bug is real and the proof is strong, but the current patch uses `transport: "local"` as a proxy for `local.modelPath` model identity and can regress generic or third-party local embedding providers that still use `settings.model`.

## What This Changes

The PR changes memory-core status-only index identity resolution and tests so a local embedding index built from `local.modelPath` is no longer reported dirty immediately after rebuild.

## Best Possible Solution

Keep the llama.cpp `local.modelPath` fix, but preserve generic local adapter semantics by scoping the modelPath identity rule to the official llama.cpp/local adapter or a typed adapter identity contract, with regression coverage for both paths.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: current main's status-only comparison expects `settings.model` or the default while the llama.cpp adapter stores `local.modelPath` as `meta.model`. The PR body also includes before/after terminal output on the same real local index.

## Solution Assessment

No, not as written: the fix is in the right status identity layer, but `transport: "local"` is too broad as the signal for modelPath-based identity. The safer fix is to scope the rule to the llama.cpp adapter or add an explicit adapter identity contract.
