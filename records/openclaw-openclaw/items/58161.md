## Label Justifications

- `P2`: This is concrete bundle-MCP approval work with real integration impact, but it is not a core outage or regression blocking all users.
- `impact:security`: The issue concerns moving destructive or privileged MCP actions from coarse config or model-visible prompts to a user-approved security boundary.

## Summary

Keep open: current main and the latest release still execute embedded Pi bundle-MCP tools as direct `callTool` results with no MCP elicitation capability or handler, and the related consent-envelope work is still an open product/security alternative rather than a shipped replacement.

## What This Changes

The issue requests MCP elicitation support in the embedded Pi bundle-MCP client so MCP tools can request user approval during execution.

## Best Possible Solution

Land or replace the related approval work with a security-approved bundle-MCP design that either implements MCP-native elicitations or explicitly chooses the consent-envelope compatibility path, with documented interactive and non-interactive behavior.

## Reproduction Assessment

Yes, for a source-level baseline: a bundle-MCP server that needs `elicitation/create` during `tools/call` reaches the current direct `callTool` and materializer path without an advertised elicitation capability or handler. I did not run a live MCP server in this read-only review.

## Solution Assessment

Unclear. Bridging through OpenClaw's existing approval UX is the right area, but maintainers still need to choose MCP-native elicitation support, the consent-envelope contract in https://github.com/openclaw/openclaw/issues/78308 and https://github.com/openclaw/openclaw/pull/78303, or an explicit compatibility layer.
