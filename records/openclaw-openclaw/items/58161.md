## Summary

Keep open: current main and the latest release still pass embedded bundle-MCP tool calls straight through without MCP-native `elicitation/create` support, while the related consent-envelope PR is open and needs maintainer security/product judgment rather than closing this native-elicitation request early.

## What This Changes

Request support for MCP elicitations in the embedded Pi bundle-MCP client so plugin-backed MCP tools can ask users for dynamic runtime approval.

## Best Possible Solution

Review and land, or replace, https://github.com/openclaw/openclaw/pull/78303 with a security-approved bundle-MCP approval design that documents whether MCP-native elicitations are implemented, intentionally superseded, or still tracked separately.

## Reproduction Assessment

Yes for a source-level baseline: a bundle-MCP server that requires `elicitation/create` during `tools/call` reaches the current direct `callTool` and materializer path without an advertised elicitation capability or handler. This is a feature/security-boundary request rather than a strict broken-contract bug.

## Solution Assessment

Unclear. Bridging user approval through existing OpenClaw approvals is the right area, but maintainers still need to choose MCP-native elicitation support, the consent-envelope contract in https://github.com/openclaw/openclaw/issues/78308, or an explicit compatibility layer.
