## Summary

Keep open: current main still lacks embedded Pi bundle-MCP elicitation handling, and the active consent-envelope proposal is useful related work but is still open and does not settle MCP-native elicitation compatibility.

## What This Changes

The issue requests MCP elicitation support in the embedded Pi bundle-MCP client so plugin-backed MCP tools can request runtime user approval.

## Best Possible Solution

Design and land a security-reviewed embedded bundle-MCP elicitation bridge, or explicitly consolidate this request into the active consent-envelope design with documented MCP-native compatibility and fallback behavior.

## Reproduction Assessment

Yes for a source-level baseline: a bundle MCP server that sends `elicitation/create` during `tools/call` would run through the current direct `callTool` path without an advertised elicitation capability or handler. This is not a runtime bug reproduction because the item is a feature and security-boundary request.

## Solution Assessment

Unclear. Bridging through OpenClaw approvals is likely the right runtime area, but maintainers still need to choose MCP-native elicitation, the custom consent-envelope path in https://github.com/openclaw/openclaw/issues/78308, or a documented compatibility layer.
