## Summary

Keep open: current main still lacks embedded Pi bundle-MCP elicitation support, and the newer consent-envelope discussion is useful related work but not a merged or exact MCP-native replacement.

## What This Changes

Support MCP draft elicitation requests in the embedded Pi bundle-MCP client so MCP tools can ask users for dynamic approval during execution.

## Best Possible Solution

Design and land a security-reviewed embedded bundle-MCP elicitation bridge, or explicitly consolidate this request into the active MCP consent-envelope design with a clear decision on MCP-native compatibility.

## Reproduction Assessment

Yes for the current baseline: a bundle MCP server that sends `elicitation/create` during `tools/call` would hit the direct `callTool` flow without an advertised elicitation capability or handler. This is source-reproducible rather than a runtime bug reproduction because the item is a feature/security-boundary request.

## Solution Assessment

Unclear. Bridging dynamic approval through OpenClaw approvals is the likely direction, but maintainers still need to choose MCP-native elicitation, the custom consent-envelope path in https://github.com/openclaw/openclaw/issues/78308, or a documented compatibility layer between them.
