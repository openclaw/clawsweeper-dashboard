## Summary

Keep open: current main and the latest release still route embedded bundle-MCP tool calls through a direct `callTool` path without standard MCP elicitation handling, while the related consent-envelope issue and PR remain active product/security review rather than merged replacement behavior.

## What This Changes

The issue requests MCP elicitation support in the embedded Pi/bundle MCP client so plugin-backed MCP tools can request runtime user approval instead of relying on coarse static destructive-action config.

## Best Possible Solution

Keep this issue open until maintainers choose and land one MCP approval design that covers the embedded bundle-MCP runtime with exact call binding, safe decline/timeout behavior, noninteractive fallback semantics, tests, docs, and release/upgrade guidance.

## Reproduction Assessment

Yes. Source inspection on current main and v2026.6.8 shows the embedded bundle-MCP tool path calls the MCP client and forwards the result without a generic runtime elicitation/approval interception path.

## Solution Assessment

Unclear. Reusing `plugin.approval.*` at the bundle-MCP runtime/materialization boundary is plausible, but maintainers still need to decide whether the final contract should be standard MCP elicitations, the consent envelope, or a narrower bridge with specific upgrade behavior.
