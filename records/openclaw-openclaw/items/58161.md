## Summary

Keep open: current `main` and the latest release still expose embedded Pi bundle-MCP tools through a direct `callTool` path with no MCP elicitation or approval bridge, while the related consent-envelope work remains open for maintainer security/product review rather than being a shipped replacement.

## What This Changes

The issue requests MCP elicitation support in the embedded Pi MCP client so plugin-backed MCP tools can ask users for runtime approval instead of relying on static destructive-action policy.

## Best Possible Solution

Make an explicit maintainer decision on MCP-native elicitations versus the consent-envelope compatibility path, then land or replace the related approval work with docs, focused tests, and security-reviewed interactive and non-interactive semantics.

## Reproduction Assessment

Yes. A source-level check is enough for the current baseline: embedded Pi creates a plain bundle-MCP client, calls `client.callTool`, and forwards the result to the agent without an elicitation or approval handler; I did not run a live MCP server in this read-only review.

## Solution Assessment

Unclear. Reusing OpenClaw's approval UX is the right area, but maintainers still need to choose MCP-native elicitation support, the consent-envelope design in https://github.com/openclaw/openclaw/issues/78308 and https://github.com/openclaw/openclaw/pull/78303, or a compatibility layer between them.
