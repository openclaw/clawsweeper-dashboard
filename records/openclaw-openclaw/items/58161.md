## Summary

Keep open: current main still does not advertise or handle MCP elicitations in the embedded Pi bundle-MCP runtime, and the related consent-envelope proposal is open but does not settle MCP-native elicitation compatibility.

## What This Changes

Request support for MCP draft elicitations in the embedded Pi MCP client so bundle-MCP tools can ask users for dynamic runtime approval.

## Best Possible Solution

Land a security-reviewed embedded bundle-MCP design that either implements MCP-native elicitations directly or explicitly supersedes them with a documented consent-envelope compatibility contract.

## Reproduction Assessment

Yes for a source-level baseline: an embedded bundle-MCP server that sends `elicitation/create` during `tools/call` would go through the current direct `callTool` and materializer path without an advertised elicitation capability or handler. This is a feature/security-boundary request, not a strict runtime bug reproduction.

## Solution Assessment

Unclear. Bridging through existing OpenClaw approvals is likely the right area, but maintainers still need to choose MCP-native elicitation, the open consent-envelope contract in https://github.com/openclaw/openclaw/issues/78308, or a documented compatibility layer.
