## Summary

Keep open: current main and v2026.5.22 still run embedded Pi bundle-MCP tools through a direct `callTool` path with no MCP elicitation handler, while the related consent-envelope implementation remains an open maintainer security/product decision rather than a shipped fix.

## What This Changes

The issue requests MCP elicitation support in the embedded Pi MCP client so bundle-MCP tools can ask users for dynamic approval during execution.

## Best Possible Solution

Review and land or replace https://github.com/openclaw/openclaw/pull/78303 with a security-approved embedded bundle-MCP approval design, explicitly documenting whether OpenClaw supports MCP-native elicitations, the consent-envelope contract, or both.

## Reproduction Assessment

Yes. Source inspection shows current main returns bundled MCP `CallToolResult` values directly to the agent without advertised elicitation capability or an `elicitation/create` handler; I did not run a live MCP server in this read-only review.

## Solution Assessment

Unclear. Reusing OpenClaw's approval UX is the right area, but maintainers still need to choose MCP-native elicitation support, the consent-envelope contract, or an explicit compatibility layer.
