## Summary

Keep open: current main still lacks MCP-native elicitation support in the embedded Pi bundle-MCP runtime, and the related consent-envelope work is still an open product/security path rather than a merged replacement for MCP `elicitation/create` compatibility.

## What This Changes

Add MCP elicitation handling to the embedded Pi bundle-MCP runtime so MCP servers can request interactive user approval during tool execution.

## Best Possible Solution

Land a security-reviewed bundle-MCP approval design that either implements MCP-native elicitations directly or explicitly supersedes them with a documented consent-envelope compatibility contract.

## Reproduction Assessment

Yes for a source-level baseline: an embedded bundle-MCP server that sends `elicitation/create` during `tools/call` would hit the current direct `callTool` and materializer path without an advertised elicitation capability or handler. This is a feature/security-boundary request, not a strict bug reproduction.

## Solution Assessment

Unclear. Bridging through existing OpenClaw approvals is the right area, but maintainers still need to choose MCP-native elicitation, the open consent-envelope contract in https://github.com/openclaw/openclaw/issues/78308, or a documented compatibility layer.
