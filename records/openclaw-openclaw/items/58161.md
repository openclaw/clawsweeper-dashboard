## Summary

Keep open: current main still runs embedded bundle-MCP tools through a direct MCP `callTool` path with no elicitation or approval bridge, while the related consent-envelope work remains open and unmerged for maintainer security/product review.

## What This Changes

Support MCP elicitations in the embedded Pi MCP client so plugin-backed MCP tools can request per-action runtime approval from the user.

## Best Possible Solution

Keep this issue open until maintainers choose and land a bundle-MCP approval design, likely by reviewing the open consent-envelope work or replacing it with an MCP-spec-native bridge through existing `plugin.approval.*` surfaces.

## Reproduction Assessment
