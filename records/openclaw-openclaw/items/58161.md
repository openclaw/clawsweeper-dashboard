## Summary

Keep open: current main and the latest release still pass embedded bundle-MCP tool calls straight through without MCP-native elicitation handling, while the related consent-envelope PR remains an open security/product decision rather than a shipped replacement for this request.

## What This Changes

The issue requests MCP elicitation support in the embedded Pi bundle-MCP client so plugin-backed tools can ask users for dynamic runtime approval.

## Best Possible Solution

Land or replace the related approval work with a security-approved bundle-MCP design that either implements MCP-native elicitations or explicitly documents the consent-envelope compatibility path, including interactive and non-interactive behavior.

## Reproduction Assessment

Yes, for a source-level baseline: a bundle-MCP server that needs `elicitation/create` during `tools/call` reaches the current direct `callTool` and materializer path without an advertised elicitation capability or handler. I did not run a live MCP server in this read-only review.

## Solution Assessment

Unclear. Bridging through existing OpenClaw approval UX is the right area, but maintainers still need to choose MCP-native elicitation support, the consent-envelope contract in https://github.com/openclaw/openclaw/issues/78308, or an explicit compatibility layer.
