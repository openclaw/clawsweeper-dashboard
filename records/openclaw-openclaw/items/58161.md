## Summary

Keep open: current main still lacks MCP-native elicitation support in the embedded Pi bundle-MCP runtime, and the related consent-envelope proposal remains open rather than a merged replacement for MCP `elicitation/create` compatibility.

## What This Changes

The issue requests MCP elicitation support in the embedded Pi bundle-MCP client so plugin-backed MCP tools can request dynamic runtime approval from the user.

## Best Possible Solution

Land a security-reviewed bundle-MCP approval design that either implements MCP-native elicitations directly or explicitly supersedes them with a documented consent-envelope compatibility contract.

## Reproduction Assessment

Yes for a source-level baseline: an embedded bundle-MCP server that sends `elicitation/create` during `tools/call` would hit the current direct `callTool` and materializer path without an advertised elicitation capability or handler. This is a feature/security-boundary request, not a strict bug reproduction.

## Solution Assessment

Unclear. Bridging through existing OpenClaw approvals is the right area, but maintainers still need to choose MCP-native elicitation support, the open consent-envelope contract in https://github.com/openclaw/openclaw/issues/78308, or a documented compatibility layer.
