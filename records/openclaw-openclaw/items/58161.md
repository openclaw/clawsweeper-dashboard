## Summary

Keep open. Current main still has a basic embedded Pi bundle-MCP client path that lists and calls tools, but it does not advertise MCP elicitation support, register an elicitation request handler, or bridge elicitation outcomes into OpenClaw's approval UX. Adjacent approval infrastructure and Codex-specific elicitation handling exist, but they do not solve this runtime request.

## What This Changes

Request support for MCP elicitation requests in the embedded Pi bundle-MCP client so plugin-backed MCP tools can ask users for per-action runtime approval.

## Best Possible Solution

Add a generic, security-reviewed embedded bundle-MCP elicitation bridge that advertises only supported modes, maps approve/deny/cancel/timeout into MCP `accept`/`decline`/`cancel`, rejects or URL-gates sensitive input, documents interactive versus non-interactive behavior, and adds focused runtime/materialization tests plus docs.

## Reproduction Assessment

Yes. A bundle MCP server that sends `elicitation/create` during `tools/call` would hit the current direct `session.client.callTool(...)` path without an advertised elicitation capability or handler; this can be verified statically from the current runtime and test coverage.

## Solution Assessment

Unclear. Bridging elicitations through OpenClaw plugin/native approvals is the likely direction, but the safest implementation needs maintainer design for form versus URL mode, sensitive data handling, non-interactive fallback, and exact response mapping before coding.
