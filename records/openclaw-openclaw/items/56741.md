## Summary

Keep open. The PR targets a valid current-main Telegram authorization bug, but it is not merge-ready because the changed path would still send a visible disabled-command reply where the linked issue expects non-owning bots to silently ignore the command, and the external PR lacks required real Telegram proof.

## What This Changes

The PR flips `useTopicAndGroupOverrides` to true in `extensions/telegram/src/bot-native-commands.ts` so native Telegram slash commands evaluate topic/group policy overrides.

## Best Possible Solution

Native command auth should use effective topic/group policy, silently ignore commands blocked by topic/group disabled policy for non-owning bots, preserve intended account-level disabled-command replies, add focused regression coverage, and include redacted real Telegram proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure a Telegram forum topic or inherited group override with `groupPolicy: "disabled"` while the account/default policy would otherwise allow commands, then send a native slash command; current main passes the topic config but disables topic/group override resolution for native commands.

## Solution Assessment

No. Flipping the resolver flag is necessary, but the current PR routes effective disabled policy into an existing visible reply branch instead of matching the silent disabled-topic behavior requested by https://github.com/openclaw/openclaw/issues/56699 and used by regular messages.
