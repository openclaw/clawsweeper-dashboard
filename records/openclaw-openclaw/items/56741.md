## Summary

Keep this PR open: current main still has the Telegram native-command policy bypass, and the branch is a narrow implementation candidate rather than obsolete cleanup. The main remaining issues are maintainer judgment on visible disabled-command replies versus silent drops, plus the required changelog/test proof before merge.

## What This Changes

The PR changes Telegram native command authorization so slash commands use topic/group-aware `groupPolicy` resolution instead of account/default-only policy resolution.

## Best Possible Solution

Merge this patch or an equivalent current-layout change after maintainers decide the disabled-topic command behavior, with a focused regression test for topic/group `groupPolicy: "disabled"` and an Unreleased Fixes changelog entry.

## Reproduction Assessment

Yes. Configure a Telegram forum topic or group override with `groupPolicy: "disabled"` while the account/default policy would otherwise allow groups, then send a native command such as `/reset`; current main passes `useTopicAndGroupOverrides: false`, so the resolver skips that override.

## Solution Assessment

Yes for the authorization bypass: reusing the existing resolver with `useTopicAndGroupOverrides: true` is the narrowest maintainable fix and matches regular-message handling. It is unclear whether this fully satisfies the linked issue's no-noise expectation, because existing command-denial behavior sends a visible disabled-command reply.
