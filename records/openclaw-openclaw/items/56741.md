## Summary

Keep open. The flag flip targets the right Telegram authorization path, but the branch still sends a visible disabled-command reply for topic/group `groupPolicy: "disabled"` instead of silently dropping non-owning bots as the linked report and regular-message path expect, and the external PR has no real Telegram proof attached.

## What This Changes

This PR changes Telegram native command authorization to evaluate topic and group `groupPolicy` overrides by flipping `useTopicAndGroupOverrides` to `true`.

## Best Possible Solution

Native commands should use topic/group-aware policy resolution, silently drop commands blocked by effective topic/group disabled policy, add focused regression coverage, and attach redacted real Telegram proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure a Telegram forum topic or inherited group override with `groupPolicy: "disabled"` while account/default policy would otherwise allow commands, then send a native slash command; current main skips that override for native commands.

## Solution Assessment

No. Flipping the flag is necessary, but this branch still routes disabled topic/group overrides into a visible disabled-command reply and lacks focused coverage plus real Telegram proof.
