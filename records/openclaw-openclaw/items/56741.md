## Summary

Keep open. Current main still bypasses topic/group `groupPolicy` overrides for Telegram native commands, and the PR’s code change targets the implicated path. The patch looks functionally correct, but it needs the required changelog entry, focused regression coverage, and a maintainer call on visible disabled-command replies versus silent drops before merge.

## What This Changes

The PR changes Telegram native command authorization so slash commands evaluate `groupPolicy` with topic and group override resolution enabled.

## Best Possible Solution

Merge this patch or an equivalent current-layout change after adding a single-line `CHANGELOG.md` fix entry and a focused regression test that exercises a forum topic or group override where account/default policy would otherwise allow the command. If maintainers want the exact no-noise behavior from #56699, keep the authorization fix and add a separate silent-drop path for topic/group-policy-disabled native commands before closing the linked issue.

## Reproduction Assessment

Yes. Configure a Telegram forum topic with `topicConfig.groupPolicy: "disabled"` while account/default policy allows groups, then send `/reset` or `/status`; current main’s command path passes `useTopicAndGroupOverrides: false`, so it ignores that topic policy.

## Solution Assessment

Yes for the authorization bypass: reusing the existing group-policy resolver with `useTopicAndGroupOverrides: true` is the narrowest maintainable code fix and matches regular-message policy resolution. It is unclear whether it fully satisfies the linked issue’s desired UX until maintainers decide between the existing disabled-command reply and a silent drop.
