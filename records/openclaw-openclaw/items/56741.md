## Summary

Keep open: the PR targets a real Telegram native-command authorization bug on current main, but it still needs contributor-supplied real behavior proof and likely leaves disabled-topic commands visibly replying where the canonical report asked non-owning bots to stay silent.

## What This Changes

The PR flips Telegram native command authorization to evaluate topic/group `groupPolicy` overrides when handling `/reset`, `/new`, `/status`, and related native commands.

## Best Possible Solution

Make native Telegram commands use the topic/group-aware policy path, add focused regression coverage for effective disabled topic/group policy, and align disabled-topic command handling with regular-message silent-drop behavior unless maintainers intentionally choose reply semantics.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure a forum topic or inherited group override with `groupPolicy: "disabled"` while account/default policy would otherwise allow commands, then send a native command; current main skips that override via `useTopicAndGroupOverrides: false`.

## Solution Assessment

No. Flipping the flag is the right narrow policy-evaluation fix, but this branch should also settle disabled-topic reply semantics, add focused coverage, and attach redacted real Telegram proof before merge.
