## Summary

Keep open. The PR applies the right policy-lookup flag, but it still routes disabled topic/group commands to a visible disabled-command reply while the linked canonical report asks non-owning bots to ignore them, and the external Telegram authorization change still lacks real Telegram proof.

## What This Changes

The PR changes Telegram native command authorization to pass `useTopicAndGroupOverrides: true` when evaluating `groupPolicy` for slash commands.

## Best Possible Solution

Update or replace the PR so native commands use topic/group-aware policy resolution, disabled-topic commands are dropped consistently with regular messages unless maintainers explicitly choose reply semantics, focused regression coverage is added, and redacted real Telegram proof is attached.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: configure a Telegram forum topic or inherited group override with `groupPolicy: "disabled"` while account/default policy would otherwise allow commands, then send a native slash command. Current main passes topic/group config into the helper but disables topic/group override resolution for native commands; I did not run a live Telegram group in this read-only review.

## Solution Assessment

No. Flipping the flag is the necessary policy-selection piece, but this branch still sends a visible disabled reply for disabled-topic commands and lacks focused regression coverage plus required real Telegram proof.
