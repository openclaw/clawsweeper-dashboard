## Summary

Close as superseded by the canonical issue: the branch targets a real Telegram native-command policy bug, but it is not the right landing path because disabled topic overrides would still produce visible denial replies and real Telegram proof is missing.

## What This Changes

The PR changes Telegram native command authorization so group-policy checks use topic and group `groupPolicy` overrides.

## Best Possible Solution

Keep the canonical issue open for a replacement fix that honors topic/group overrides, silently drops disabled scoped commands for non-owning bots, preserves intended account-level command denials, and includes focused tests plus real Telegram forum proof.

## Reproduction Assessment

Yes, at source level. Current main still skips topic/group `groupPolicy` overrides for native commands while the helper and regular message sibling show exactly why scoped disabled policy is bypassed; I did not run a live Telegram repro in this read-only review.

## Solution Assessment

No. Flipping the flag is necessary but incomplete because disabled topic/group overrides would flow into a visible denial reply; the safer fix must distinguish scoped disabled policy and drop those commands silently.

## Review Findings

Full review comments:

- **[P1] Drop disabled-topic commands silently:** `extensions/telegram/src/bot-native-commands.ts:305`
  - body: Changing this flag makes topic/group `groupPolicy: "disabled"` reach the existing disabled-policy branch, which sends `Telegram group commands are disabled.`. In the reported multi-bot forum setup, every non-owning bot would still post in the topic; regular message handling drops disabled-policy matches without sending. The fix needs to distinguish scoped disabled overrides and return without a reply while preserving any intended account-level denial behavior.
