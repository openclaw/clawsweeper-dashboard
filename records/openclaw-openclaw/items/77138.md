## Summary

Keep open: current `main` still has a source-level path where `device.pair.approve` can fail with `missing scope: operator.read`, while the documented local pairing fallback does not clearly cover that failure mode in Docker/local bootstrap. This touches the trusted local approval boundary, so it should get maintainer review rather than a cleanup close.

## What This Changes

The issue reports that Docker-local `openclaw devices approve <requestId>` cannot bootstrap a first operator pairing request because approval requires the same operator scope it is meant to grant.

## Best Possible Solution

Add a narrow, tested bootstrap behavior that preserves remote scope containment while making the documented trusted local/Docker approval path succeed or emit a precise remediation command.

## Reproduction Assessment

Yes from source inspection: a pending `role: operator` request for `operator.read` approved by a session that only has `operator.pairing` reaches the current approval guard and returns `missing scope: operator.read`. I did not run the Docker scenario live in this read-only sweep.

## Solution Assessment

No close is appropriate yet. The maintainable direction is not to relax the gateway guard globally, but to decide the exact loopback/Docker bootstrap fallback or remediation and cover it with CLI plus gateway pairing tests.
