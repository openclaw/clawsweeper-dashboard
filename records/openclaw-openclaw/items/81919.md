## Summary

Keep open. The retry hang is source-plausible and the linked Telegram issue is still open, but this PR is not mergeable as written because it adds a generic 30s channel timeout that can override Telegram’s configured timeout contract, exposes a retry option without wiring the strict config schema/types, and lacks after-fix real behavior proof.

## What This Changes

The PR adds `perCallTimeoutMs` to `RetryConfig`, sets 30s defaults for shared channel and Discord retry runners, and adds retry unit tests.

## Best Possible Solution

Revise toward a bounded Telegram delete/control-operation timeout or cancellation path that preserves existing Telegram timeout and retry contracts, then attach redacted real CLI or Telegram proof.

## Reproduction Assessment

Source-reproducible, but not live-reproduced here: current main awaits `fn()` directly in `retryAsync`, and `deleteMessageTelegram` passes `api.deleteMessage` through that runner. A high-confidence live reproduction would run the reported `openclaw message delete --channel telegram ...` flow against a real Telegram DM and show the hang or fixed timeout behavior.

## Solution Assessment

No, not as written. A bounded timeout is directionally useful, but it should be scoped to the Telegram operation or aligned with existing timeout/config contracts instead of adding an unconfigurable generic channel cap.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Honor Telegram timeout config before adding a generic cap:** `src/infra/retry-policy.ts:17`
  - body: This 30s default is inherited by `createTelegramRetryRunner` for Telegram requests via `createTelegramRequestWithDiag`, so `channels.telegram.timeoutSeconds` values above 30s or slow send/media operations can now fail in `retryAsync` before the configured grammY timeout. Please derive the cap from Telegram operation/config, or keep the shared default disabled and opt in only for the affected delete/control calls.
  - confidence: 0.86
- **[P2] Expose the new retry field through config contracts:** `src/infra/retry.ts:18`
  - body: `perCallTimeoutMs` is exported on `RetryConfig` and the PR says it is overridable, but `RetryConfigSchema` is strict and `OutboundRetryConfig` still omit it, so channel config cannot set or disable it. Add the field to the schema/types/docs/generated metadata, or keep it as an internal-only option that is not advertised as config-driven.
  - confidence: 0.83
