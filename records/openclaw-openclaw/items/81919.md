## Summary

Keep this PR open: the updated branch fixed the earlier config-schema finding, but the shared 30s channel retry default still cuts across Telegram bot reply delivery, which currently has a 60s or configured-higher timeout contract. The external PR also still lacks after-fix real Telegram/CLI proof before merge.

## What This Changes

The PR adds a configurable per-call timeout to retry handling, enables channel/Discord retry timeout defaults, and updates config schema/types, generated metadata, and retry tests.

## Best Possible Solution

Keep the PR open, make the retry cap honor Telegram bot delivery timeout contracts as well as direct send/delete helpers, then require redacted real Telegram CLI proof showing the delete path times out instead of hanging.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not live here: the source path is clear because current main awaits an unresolved retry attempt and Telegram delete routes through that runner, but high-confidence runtime reproduction still needs the reported Telegram DM delete command against a real account.

## Solution Assessment

Is this the best way to solve the issue? No, not yet. The timeout approach is directionally right, but the current default still preempts Telegram bot delivery's existing 60s or configured-higher timeout path; the safer fix is to pass the effective Telegram timeout into that retry runner or scope the default to the affected control/delete path.
