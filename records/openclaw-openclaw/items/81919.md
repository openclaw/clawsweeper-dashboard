## Summary

This PR should stay open: it addresses a real source-level retry hang path, but the current branch still changes Telegram bot delivery timeout behavior and lacks required after-fix real Telegram or CLI proof.

## What This Changes

The PR adds `perCallTimeoutMs` retry support, defaults channel and Discord retry runners to a 30s per-call cap, exposes the field in config schema/types/metadata, and expands retry/config tests.

## Best Possible Solution

Keep the per-call timeout direction, but ensure every Telegram retry runner that covers visible delivery uses the effective Telegram timeout contract before the generic fallback, then require redacted real CLI or Telegram proof of the delete hang timing out instead of wedging.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not live here: the source path is clear from `retryAsync` awaiting a never-settling call and Telegram delete routing through it, but the real Telegram DM delete command still needs to be run against a real account to prove the failure and the fix.

## Solution Assessment

Is this the best way to solve the issue? No, not yet. A per-call cap is directionally right, but this branch must preserve Telegram delivery's existing timeout contract before the new generic cap and still needs real behavior proof.
