## Summary

Keep open. The bug path is source-plausible and the timeout direction is useful, but this diff applies a shared 30s channel cap that can preempt Telegram's configured request timeout, leaves the advertised retry option outside strict config contracts, and lacks after-fix real behavior proof.

## What This Changes

The branch adds `perCallTimeoutMs` to the shared retry runner, enables 30s channel and Discord retry defaults, and extends retry unit coverage.

## Best Possible Solution

Implement a scoped timeout or cancellation guard for Telegram delete/control API calls, align any public retry option with schema/docs if kept, then prove it with a redacted real CLI or Telegram run.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection and the linked diagnostics show a plausible non-settling `api.deleteMessage` path through `retryAsync`, but a real Telegram DM delete run is still needed for high-confidence proof.

## Solution Assessment

No, not as written. The outer timeout idea is useful, but the shared 30s default can preempt Telegram's configured timeout and non-idempotent request semantics; a scoped Telegram control-call guard or a fully wired config contract is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Keep the timeout scoped to safe Telegram operations:** `src/infra/retry-policy.ts:17`
  - body: Adding `perCallTimeoutMs: 30_000` to the shared channel defaults affects every Telegram helper using `createChannelApiRetryRunner`, including non-idempotent sends. A slow send or configured `channels.telegram.timeoutSeconds` above 30s can now be reported as failed while the original request keeps running in the background, which can produce confusing failures or duplicate manual retries. Please keep the generic default disabled and opt in only for delete/control calls, or derive the cap from the Telegram operation/config.
  - confidence: 0.88
- **[P2] Expose the retry timeout through the config contract:** `src/infra/retry.ts:19`
  - body: `perCallTimeoutMs` is exported on `RetryConfig` and the PR says channel retry config can override it, but `RetryConfigSchema` is strict and `OutboundRetryConfig` still omit the key. User channel config would reject or type-drop the advertised field, so either add schema/types/docs/generated metadata or keep this as an internal-only option.
  - confidence: 0.84
