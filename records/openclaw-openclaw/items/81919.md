## Summary

Current main fixes the linked Telegram delete hang with a narrower method-aware Telegram request-timeout wrapper, so this broader retry-runner PR is no longer the best path to merge.

## What This Changes

The PR adds a per-call timeout option to retryAsync/channel retry defaults, exposes it in retry config schema/types, applies 30s defaults to Discord/channel retry runners, and adds retry/config tests.

## Best Possible Solution

Keep the current Telegram method-aware request-timeout wrapper on main and avoid merging the broader generic retry timeout branch unless a separate owner-approved retry contract change is opened.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No live current-main reproduction was run; source inspection shows current main now wraps Telegram delete Bot API fetches with a 15s method timeout and has focused regression coverage for that path.

## Solution Assessment

Is this the best way to solve the issue? Yes for current main, no for this PR branch. The main fix bounds the Telegram request at the fetch layer with method-aware timeouts, while the PR's generic retry cap is broader and can interfere with delivery timeout contracts.
