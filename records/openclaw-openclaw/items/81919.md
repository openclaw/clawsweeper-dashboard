## Summary

Keep this PR open: the latest head appears to address the earlier P2 design findings, but this external Telegram transport fix still lacks after-fix real Telegram/CLI proof, and the newly public retry timeout option needs a small docs follow-up.

## What This Changes

The PR adds `perCallTimeoutMs` to shared retry config, applies per-call timeout defaults to channel and Discord retry runners, forwards Telegram `timeoutSeconds` into retry resolution, and adds retry/config tests.

## Best Possible Solution

Land the bounded retry timeout only after the public retry docs mention the new knob and redacted terminal/log/recording proof shows the Telegram delete command timing out and surfacing instead of hanging.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source-reproducible but not live-reproduced here: current main awaits `fn()` directly and the Telegram delete command path routes `api.deleteMessage` through that runner; live confidence still requires the real Telegram DM CLI run from https://github.com/openclaw/openclaw/issues/81908.

## Solution Assessment

Is this the best way to solve the issue? Mostly yes: the outer timeout race is a narrow fix that avoids larger AbortController plumbing, and the follow-up commit now respects Telegram timeout config and strict schema contracts; it still needs docs and live proof before merge.
