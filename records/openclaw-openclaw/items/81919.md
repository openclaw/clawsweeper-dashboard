## Summary

Keep open. The branch targets a real source-reproducible hang, but its timeout only races the promise and can leave the underlying grammY/fetch request plus grammY's own 500s timer alive, so it may not fix the reported CLI-exit failure; it also lacks required after-fix real behavior proof.

## What This Changes

The PR adds an optional per-call timeout to `retryAsync`, enables a 30s timeout for channel API and Discord retry defaults, and adds retry timeout coverage.

## Best Possible Solution

Put the timeout where the Telegram/Discord HTTP request can be aborted, preferably by reusing or extending the existing Telegram method-specific fetch timeout guard, then prove the real CLI/Telegram delete flow exits promptly.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection shows a non-settling `api.deleteMessage` promise is still directly awaited through current `retryAsync`, which is enough to make the reported hang source-reproducible.

## Solution Assessment

No. The current PR shape is broader than necessary and does not cancel the timed-out operation; a cancellable channel request or fetch-layer timeout is the narrower maintainable fix for the reported CLI hang.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Abort requests when the per-call timeout fires:** `src/infra/retry.ts:120`
  - body: `runWithPerCallTimeout` stops awaiting `fn()`, but it leaves the original request running. For the linked Telegram path, grammY 1.42.0 creates its own 500s request timeout and abort controller, so each timed-out `api.deleteMessage` attempt can keep the Node process alive long after this wrapper rejects, and retries can leave several live requests behind. That still misses the reported CLI-exit failure; wire an `AbortSignal` or channel transport timeout where the Telegram request is issued instead of orphaning it.
  - confidence: 0.86
