## Summary

Keep open: the PR addresses a real source-visible hang class, but the patch introduces a generic 30s channel API cap that can override Telegram timeout configuration and adds a retry option without aligning strict config contracts. It also lacks required after-fix real behavior proof.

## What This Changes

The PR adds an optional per-call timeout to `retryAsync`, enables a 30s timeout for channel and Discord retry defaults, and adds retry unit coverage.

## Best Possible Solution

Keep the PR open, preserve a bounded timeout for the reported hung delete path, and revise the design so Telegram timeoutSeconds/retry config contracts remain honored before asking for real CLI or Telegram proof.

## Reproduction Assessment

Source-reproducible, but not live-reproduced here: current main awaits `fn()` directly in `retryAsync`, and the linked Telegram delete path routes `bot.api.deleteMessage` through that runner. A high-confidence live proof would run the reported `openclaw message delete --channel telegram ...` command against a real Telegram DM and show the hang or timeout behavior.

## Solution Assessment

No, not as written. The outer timeout idea is directionally useful, but a generic 30s channel default conflicts with existing Telegram timeout configuration and the new retry option is not wired through the strict config contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Honor Telegram timeout config before adding a generic cap:** `src/infra/retry-policy.ts:17`
  - body: This 30s default is inherited by `createTelegramRetryRunner` for every Telegram request via `createTelegramRequestWithDiag`, so a user with `channels.telegram.timeoutSeconds` above 30s or a slow media/send operation now gets a `retryAsync` timeout before the configured grammY/client timeout can finish. Please derive the cap from the Telegram operation/config or keep the generic default disabled and opt in only for the affected delete/control calls.
  - confidence: 0.86
- **[P2] Expose the new retry field through config contracts:** `src/infra/retry.ts:19`
  - body: `perCallTimeoutMs` is exported on `RetryConfig` and the PR says it is overridable, but `RetryConfigSchema` is strict and `OutboundRetryConfig` still omit this key, so channel config cannot actually set or disable it. Add the field to the schema/types/docs/generated metadata, or keep it as an internal-only option that is not advertised as config-driven.
  - confidence: 0.83
