## Summary

Keep open: current main and the latest release still hardcode xAI image generation private-network access to false, so the reported provider request/browser SSRF opt-ins cannot unblock the xAI image path. The recent fake-IP comment is useful evidence that the policy question may be broader than xAI, but no merged or open PR currently owns this exact xAI image-provider gap.

## What This Changes

The issue asks xAI image generation to honor explicit private-network SSRF opt-ins instead of forcing private-network access off.

## Best Possible Solution

Keep SSRF default-deny, make the xAI image provider honor the existing explicit opt-in contract with focused deny-by-default and opt-in tests, and track broader fake-IP/default-provider-policy work separately.

## Reproduction Assessment

Yes. Source inspection on current main shows xAI image generation passes a hardcoded false private-network policy into the guarded provider HTTP path even when config contains the reported opt-in.

## Solution Assessment

Unclear as a final implementation until maintainers choose the exact accepted opt-in surface. The maintainable direction is to reuse the existing provider request/browser SSRF contracts rather than add a new global bypass in this narrow fix.
