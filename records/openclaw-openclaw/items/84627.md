## Summary

Keep open: current main still hardcodes xAI image generation private-network access to false, so the reported opt-in settings cannot reach the HTTP fetch guard. The related broader media-provider private-network PR is closed unmerged, and the open Comfy report tracks a different plugin-specific DNS/private-hostname behavior rather than this xAI image path.

## What This Changes

The issue asks xAI image generation to honor existing explicit private-network SSRF opt-in settings instead of forcing `resolveAllowPrivateNetwork` to false.

## Best Possible Solution

Keep SSRF default-deny, then make the xAI image provider honor only the approved existing explicit opt-in surface with tests for deny-by-default and opt-in behavior.

## Reproduction Assessment

Yes by source inspection: configuring xAI image generation with a private `baseUrl` still reaches `resolveAllowPrivateNetwork: () => false`. I did not run a live xAI proxy, so this is source-reproducible rather than runtime-reproduced.

## Solution Assessment

Unclear as a final patch until maintainers approve the exact SSRF opt-in surface. The narrow maintainable direction is to reuse the existing provider/private-network contract, not add a new global bypass.
