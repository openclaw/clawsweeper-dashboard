## Summary

Keep open: current main and the latest release still force xAI image-generation private-network access off, while the shared provider request contract already has an explicit opt-in. No linked closing PR or canonical duplicate currently resolves the xAI image path, and the remaining decision is security-sensitive.

## What This Changes

The issue asks xAI image generation to honor explicit private-network SSRF opt-ins instead of always denying private-network image endpoints.

## Best Possible Solution

Keep SSRF default-deny, wire xAI image generation into the existing explicit provider private-network opt-in contract, and add regression coverage for both denial by default and opt-in success.

## Reproduction Assessment

Yes by source inspection: xAI image generation resolves the configured base URL but passes a hardcoded private-network denial into the shared provider HTTP request path. I did not run a live xAI proxy, so this is source-reproducible rather than runtime-reproduced.

## Solution Assessment

Unclear as a final patch until maintainers confirm whether xAI images should honor the global browser SSRF opt-in, the provider request opt-in, or both. The maintainable direction is to reuse the existing provider HTTP/private-network contract rather than adding a new escape hatch.
