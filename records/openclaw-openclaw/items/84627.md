## Summary

Keep open: current main still hardcodes xAI image generation private-network access to false, while the documented provider request contract already has an explicit `request.allowPrivateNetwork` opt-in. The related OpenAI fix is useful precedent, but it does not cover the xAI image path and no canonical fix PR is open.

## What This Changes

The issue asks xAI image generation to honor explicit private-network opt-in for custom/private base URLs instead of always failing the SSRF guard.

## Best Possible Solution

Keep SSRF default-deny, then update the xAI image provider to honor the existing explicit private-network opt-in contract with focused tests for deny-by-default and opt-in behavior.

## Reproduction Assessment

Yes by source inspection: an xAI image request with a private `baseUrl` and explicit opt-in still reaches `resolveAllowPrivateNetwork: () => false`. I did not run a live private xAI proxy, so this is source-reproducible rather than runtime-reproduced.

## Solution Assessment

Unclear as a final patch until maintainers confirm the exact SSRF opt-in surface. The narrow maintainable direction is to reuse the existing provider request/private-network contract rather than adding a new config key or broad bypass.
