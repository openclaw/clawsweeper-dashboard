## Summary

Keep open: current main and the latest stable release still force xAI image-generation private-network access off before the documented model-provider request opt-in can be honored; the related OpenAI and Comfy threads do not supersede this xAI-specific path.

## What This Changes

The issue asks OpenClaw to let xAI image generation honor explicit private-network SSRF opt-ins for a configured private xAI-compatible base URL.

## Best Possible Solution

Keep SSRF default-deny, then update the xAI image provider to honor the established explicit opt-in path with focused tests for deny-by-default and approved opt-in behavior.

## Reproduction Assessment

Yes, by source inspection: xAI image generation with a private `baseUrl` and opt-in still reaches a provider path that hardcodes `allowPrivateNetwork` to false. I did not run a live xAI proxy in this read-only review.

## Solution Assessment

Unclear as a final patch until maintainers confirm the exact SSRF opt-in surface. The narrow maintainable direction is to reuse the existing provider request/private-network contract rather than adding a new config key or broad global bypass.
