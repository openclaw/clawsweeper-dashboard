## Summary

Keep open: current main still forces xAI image generation private-network access off, while the shared provider request contract already defines an explicit `request.allowPrivateNetwork` opt-in. The related OpenAI private-endpoint fix is useful precedent, but no current code or canonical open item fully resolves the xAI image path.

## What This Changes

The issue asks xAI image generation to honor private-network opt-in settings instead of always denying private/internal image endpoint URLs.

## Best Possible Solution

Keep SSRF default-deny, then update the xAI image provider to honor the existing explicit private-network opt-in contract with focused tests for deny-by-default and opt-in behavior.

## Reproduction Assessment

Yes by source inspection: configuring xAI image generation with a private `baseUrl` and opt-in still reaches a provider path that hardcodes `allowPrivateNetwork` to false. I did not run a live xAI proxy, so this is source-reproducible rather than runtime-reproduced.

## Solution Assessment

Unclear as a final patch until maintainers confirm the exact SSRF opt-in surface, but the narrow maintainable direction is to reuse the existing provider request/private-network contract rather than adding a new config key or broad global bypass.
