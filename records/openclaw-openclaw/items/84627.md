## Summary

Keep open: current main and v2026.5.19 still force xAI image generation private-network access off, while the provider request contract and docs already define an explicit private-network opt-in.

## What This Changes

The issue asks xAI image generation to honor explicit private-network SSRF opt-in for configured private xAI-compatible endpoints.

## Best Possible Solution

Keep SSRF default-deny, then wire xAI image generation to the existing explicit opt-in contract with regression coverage for default-deny, provider request opt-in, and browser-policy opt-in behavior.

## Reproduction Assessment

Yes by source inspection: xAI image generation accepts a configured private `baseUrl` path but forces `allowPrivateNetwork` to false before the shared HTTP guard runs. I did not run a live xAI proxy, so this is source-reproducible rather than runtime-reproduced.

## Solution Assessment

Unclear as a final patch until maintainers confirm the SSRF opt-in precedence. The narrow maintainable direction is to reuse the existing provider request and browser private-network opt-ins without adding a new config key or broad bypass.
