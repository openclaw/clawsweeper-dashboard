No; the channel-local resolver is a good boundary, but this implementation applies preview-mode suppression to implicit defaults and native command replies where the existing block default should remain unless a live preview actually owns delivery.

Full review comments:

- **[P1] Keep slash-command block defaults without a preview stream:** `extensions/discord/src/monitor/native-command-agent-reply.ts:52`
  - body: This native-command path only passes `disableBlockStreaming` into the dispatcher; it never installs an `onPartialReply` callback or draft preview controller. With the new resolver, a Discord account using `agents.defaults.blockStreamingDefault: "on"` and default or partial Discord streaming will have block streaming turned off for slash-command replies without any live preview path replacing those chunks, so existing command deliveries lose their configured streaming behavior.
  - confidence: 0.9
- **[P1] Preserve Telegram's implicit global block default:** `extensions/telegram/src/preview-streaming.ts:29-30`
  - body: When Telegram streaming is unset, `resolveTelegramPreviewStreamMode({})` returns the channel default `partial`. This helper then treats that implicit default as an active preview mode and returns `false`, so accounts that only set `agents.defaults.blockStreamingDefault: "on"` and no Telegram streaming config stop inheriting the global block default. Distinguish explicit channel preview config from Telegram's implicit default and add a regression test for the unset-config upgrade case.
  - confidence: 0.88
