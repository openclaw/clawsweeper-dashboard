## Summary

Keep open. Current main still has a plausible source-level path where the canonical Flash id misses Google’s forward-compat model resolver and falls back to a text-only configured model, while the repo manifest and docs expect Google media understanding to use `gemini-3-flash-preview` for images.

## What This Changes

The issue reports that Telegram image attachments are dropped for `google/gemini-3-flash-preview` on 2026.5.7 because media-understanding decides the model does not support images.

## Best Possible Solution

Patch the Google provider’s model resolver so canonical `gemini-3-flash-preview` resolves as the Flash family with image input, then add regression coverage for media-understanding/native image resolution.

## Reproduction Assessment

Yes, source-reproducible: with a configured Google provider and no explicit discovered model entry, `google/gemini-3-flash-preview` can miss dynamic resolution, synthesize a text-only fallback, and then hit the media-understanding `Model does not support images` guard. I did not run a live Telegram/Gemini flow in this read-only review.

## Solution Assessment

Yes, the narrowest maintainable fix is provider-owned: make the Google Flash forward-compat resolver accept the canonical `gemini-3-flash-preview` request id after alias normalization, with tests proving image input is preserved. A broad core fallback would be riskier because model capability ownership belongs in the Google provider metadata/resolver.
