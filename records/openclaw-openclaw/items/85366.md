## Label Justifications

- `P1`: The report describes current-release multi-minute gateway unresponsiveness that can block prompts, channel events, and active session work after restart.
- `impact:crash-loop`: The issue is about process-level availability from event-loop saturation during gateway startup sidecars.

## Summary

Keep open: current main still has source-level paths that can make ACP identity reconciliation and session-lock cleanup scale with the number and size of persisted session stores, and no exact merged fix or canonical duplicate covers the ACP startup sidecar wall-time report. The exact 450-460 s incident was not reproduced locally, but the report has fresh release logs and the current code still matches the implicated shape.

## What This Changes

The issue requests bounded, non-blocking startup maintenance for ACP identity reconciliation and stale session-lock cleanup on installs with many persisted sessions.

## Best Possible Solution

Keep this open and choose a maintainer-approved startup maintenance design that makes ACP session-store discovery, pending identity reconciliation, and stale-lock cleanup incremental or off the hot event loop, with metrics proving bounded startup delay on large session corpora.

## Reproduction Assessment

Partially: source inspection gives a high-confidence path for unbounded session-count growth, because ACP discovery synchronously loads session stores and both reconcile and lock cleanup still run serially. The exact 450-460 s wall time was not reproduced without the reporter's large macOS ACP session corpus.

## Solution Assessment

Unclear as a final fix: bounded concurrency may help the per-session reconcile loop, but the scan itself uses synchronous session-store loading and the session-lock path overlaps broader startup-readiness policy. The best solution is a deliberate bounded background-maintenance design, not a blind parallel loop.
