## Label Justifications

- `P2`: The issue is a normal-priority gateway responsiveness bug with high impact on large ACP installs but a limited startup-window blast radius.
- `impact:crash-loop`: The reported event-loop saturation makes the gateway effectively hung for minutes during startup.
- `impact:session-state`: The implicated paths reconcile ACP session identity metadata and clean persisted session locks.

## Summary

Current main still matches the reported startup shape: ACP identity reconciliation and session-lock cleanup are background sidecars, but both do serial per-session/per-directory work on the gateway event loop, so this is not cleanup-close material.

## What This Changes

The issue asks OpenClaw to prevent ACP identity reconciliation and session-lock cleanup from monopolizing the gateway event loop during startup on installs with many persisted sessions.

## Best Possible Solution

Move the startup sidecars to bounded, yielding background work with focused regression coverage for large ACP/session-lock fanout while preserving per-session serialization semantics.

## Reproduction Assessment

Yes from source, but not from a live local install: current main still performs ACP identity reconciliation and session-lock cleanup as serial startup sidecar work matching the reported phases and growth profile.

## Solution Assessment

Yes for the bounded repair direction: the maintainable path is to add explicit concurrency and yielding around these existing sidecar loops without changing ACP/session metadata contracts.
