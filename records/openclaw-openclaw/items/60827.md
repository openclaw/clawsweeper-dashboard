## Summary

Keep open: current main and v2026.5.7 still leave Docker sandbox PID, memory, swap, and CPU limits unset unless configured, and the related open PR only covers the PID limit. This is valid security hardening, but the default budgets and opt-out semantics need maintainer security/product approval before implementation.

## What This Changes

The issue requests default Docker sandbox resource limits so new sandbox containers get PID, memory, swap, and CPU caps unless overridden.

## Best Possible Solution

Approve explicit Docker sandbox resource budgets, then implement resolver defaults with schema/type/help/docs/changelog updates and tests proving defaults apply while global and per-agent overrides still win.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to resolve Docker sandbox config with no global or per-agent resource settings; the resolved values remain undefined and Docker create args omit the corresponding limit flags.

## Solution Assessment

Unclear until maintainers choose the budgets and opt-out contract. The resolver/create-args path is the narrow implementation surface, but the policy values need explicit approval first.
