## Summary

Keep open. Current main and the latest release still leave Docker sandbox PID, memory, swap, and CPU limits unset unless an operator configures them, and the related open PR only addresses the PID cap portion.

## What This Changes

Add approved default Docker sandbox CPU, memory, swap, and PID limits so new sandbox containers are resource-capped unless explicitly overridden.

## Best Possible Solution

Approve explicit Docker sandbox resource budgets and opt-out semantics, then implement them in the sandbox resolver with aligned schema, types, help/docs, changelog, and tests while preserving global and per-agent override precedence.

## Reproduction Assessment

Yes. A source-level reproduction is to resolve Docker sandbox config with no global or per-agent resource settings, then pass that config to `buildSandboxCreateArgs`; current main leaves the values undefined and emits no resource-limit flags.

## Solution Assessment

Unclear until maintainers choose the default budgets. The narrow implementation point is the Docker sandbox config resolver, but the PID-only related PR is partial and the broader default policy needs security/product approval first.
