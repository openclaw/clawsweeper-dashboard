## Summary

Keep open. Current main and the latest tagged release still resolve Docker sandbox PID, memory, swap, and CPU limits only from configured global or per-agent values, so new sandbox containers remain uncapped by default. The related PR #44098 is useful prior art but is open, unmerged, and only addresses PID limits.

## What This Changes

Request to add default Docker sandbox CPU, memory, and PID resource limits for new sandboxed agents while preserving operator overrides.

## Best Possible Solution

Maintainers should approve explicit Docker sandbox resource budgets, then implement them in the sandbox Docker config resolution path with constants, schema/docs/help alignment, changelog if user-facing, and tests proving defaults apply while global and per-agent overrides still win.

## Reproduction Assessment

Yes. A source-level reproduction is to resolve Docker sandbox config with no `agents.defaults.sandbox.docker` or per-agent resource settings and observe that `pidsLimit`, `memory`, `memorySwap`, and `cpus` stay undefined, causing Docker create args to omit the corresponding flags.

## Solution Assessment

Unclear until maintainers choose the default budget. The narrow maintainable code path is the resolver plus Docker create-args/config tests; the open pids-only PR is not sufficient for the broader CPU and memory request.
