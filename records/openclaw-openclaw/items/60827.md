## Summary

Keep open. Current main and the latest tagged release still leave Docker sandbox PID, memory, swap, and CPU limits unset unless an operator configures them, and the related open PR only addresses the PID-limit subset.

## What This Changes

The issue requests default Docker sandbox CPU, memory, and PID limits so new agents are resource-capped without per-agent configuration.

## Best Possible Solution

Approve explicit Docker sandbox resource budgets and opt-out semantics, then implement them in the sandbox config resolver with schema/types, docs, changelog, and tests proving defaults apply while configured overrides still win.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to resolve Docker sandbox config with no global or per-agent Docker resource settings and observe `pidsLimit`, `memory`, `memorySwap`, and `cpus` remain undefined, which makes Docker create args omit the corresponding limit flags.

## Solution Assessment

Unclear until maintainers approve the default budgets. The resolver/create-args path is the narrow implementation point, but the product and security policy values should be decided before a fix PR is safe.
