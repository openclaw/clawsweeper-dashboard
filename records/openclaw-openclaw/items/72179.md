## Summary

Keep this PR open, but do not merge it as-is. Current main already ships the overlapping `cron_changed` plugin hook for gateway cron lifecycle events, so adding `cron_lifecycle` would split the public plugin contract; the remaining useful part is a field-level rework, especially `agentId`, against the shipped hook surface.

## What This Changes

The PR branch adds a new public `cron_lifecycle` plugin hook, emits gateway cron start/finish events through it, documents the hook, and adds hook-runner and gateway cron tests.

## Best Possible Solution

Rework the branch around current main’s shipped `cron_changed` contract: add accepted missing fields such as `agentId` to the public cron job/event projection with docs, tests, and changelog, or get an explicit maintainer decision to replace/deprecate `cron_changed` before introducing another hook name.

## Reproduction Assessment

Not applicable as a feature/API PR rather than a runtime bug. The high-confidence verification path is static inspection: current main exposes, documents, tests, and emits `cron_changed`, while the PR adds a second overlapping `cron_lifecycle` hook.

## Solution Assessment

No. Adding `cron_lifecycle` beside the shipped `cron_changed` hook is not the narrowest maintainable solution; extending or clarifying `cron_changed` avoids splitting the public plugin contract unless maintainers intentionally choose a new hook name.
