## Summary

Keep this PR open, but do not merge it as-is. Current main already ships an overlapping `cron_changed` plugin hook for gateway-owned cron lifecycle events, so adding `cron_lifecycle` would split the public plugin contract; the PR still usefully highlights a likely field-level gap around cron `agentId` exposure.

## What This Changes

The PR adds a typed `cron_lifecycle` plugin hook, gateway start/finish emissions for cron jobs, hook documentation, and focused plugin/gateway tests.

## Best Possible Solution

Rework the branch around the current `cron_changed` contract: add accepted missing fields such as `agentId` there with docs, tests, and a changelog entry, or make an explicit maintainer decision to replace/deprecate that hook before introducing a new name.

## Reproduction Assessment

Not applicable as a feature/API PR rather than a runtime bug. The high-confidence verification path is static inspection of current main: `cron_changed` is present, documented, tested, and emitted from gateway cron handling, while `agentId` is still absent from the public cron job projection.

## Solution Assessment

No. Adding `cron_lifecycle` beside the shipped `cron_changed` hook is not the narrowest maintainable solution; extending or clarifying `cron_changed` avoids splitting the public plugin API unless maintainers choose a replacement hook deliberately.
