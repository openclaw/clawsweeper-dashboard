## Summary

Keep this PR open, but do not merge it as-is. Current main already ships the overlapping `cron_changed` plugin hook, so adding `cron_lifecycle` would split one gateway cron event stream across two public plugin contracts unless maintainers intentionally choose a replacement/deprecation path.

## What This Changes

The PR adds a typed `cron_lifecycle` hook, emits gateway cron `started`/`finished` events to it, documents the hook, and adds focused gateway/plugin tests.

## Best Possible Solution

Rework the branch around the current `cron_changed` contract by adding accepted missing fields, docs, tests, and changelog, or make an explicit maintainer decision to replace/deprecate that hook first.

## Reproduction Assessment

Not applicable as a feature/API PR rather than a runtime bug. The high-confidence verification path is static inspection of current main: `cron_changed` is present, documented, tested, and emitted from gateway cron handling, while the PR adds a second hook name for overlapping start/finish events.

## Solution Assessment

No. Adding `cron_lifecycle` alongside the shipped `cron_changed` hook is not the narrowest maintainable solution; extending or deliberately replacing `cron_changed` is safer for plugin authors.
