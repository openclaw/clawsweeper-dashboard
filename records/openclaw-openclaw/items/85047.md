## Label Justifications

- `P2`: This is a normal-priority TUI correctness/config issue with limited blast radius but real user confusion during long turns.
- `impact:session-state`: The watchdog can clear the active TUI run state while the backend run is still alive, creating visible session-state drift.

## Summary

Keep open: current main still has a 30s TUI watchdog with an internal context option but no user-facing config plumbing, and source inspection shows a silent long tool call can still expire the timer while the run is active.

## What This Changes

The issue requests preventing the TUI 30s streaming watchdog from false-warning during live long-running tool turns and exposing the existing watchdog timeout as user config.

## Best Possible Solution

Keep the lost-final recovery behavior, but make the watchdog distinguish live in-flight tool progress from a stale run and only add a public timeout knob after choosing the canonical TUI config location.

## Reproduction Assessment

Yes, source-reproducible: an active run arms a 30s timer, tool events only rearm it when events arrive, and expiry clears the active run plus posts the reported warning. I did not run a live TUI repro because this review was read-only.

## Solution Assessment

Unclear: a timeout knob is plausible, but the safer product fix may be in-flight tool awareness plus a bounded stale-run fallback. Maintainers should decide the public config location before exposing a new key.
