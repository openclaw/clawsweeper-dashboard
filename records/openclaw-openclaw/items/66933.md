## Summary

Keep this PR open: it is recent, not implemented on current main, and it adds a real new tool-capability config surface. The current patch still has a backward-compatibility bug in sender-key parsing that can drop an existing legacy sender policy, so it should not merge as-is.

## What This Changes

This PR adds global and per-agent per-sender tool policies with channel-scoped sender keys across config/schema/docs/core and bundled tool filtering, plus an unrelated cron target guard.

## Best Possible Solution

Keep the PR open, preserve legacy sender-id matching for malformed `channel:` keys, separate or justify the cron guard, then rebase and land only after the tool-capability semantics and targeted checks are green.

## Reproduction Assessment

Yes for the review finding via source inspection: current main sends untyped keys through the legacy id fallback, while PR head classifies `channel:alice` as typed and drops it before fallback. I did not run tests because this review is read-only.

## Solution Assessment

No. The overall direction is plausible, but the current parser is not the safest implementation until malformed `channel:` typed keys preserve legacy sender-id behavior or the typed parser only accepts the full `channel:<channelId>:<senderId>` form.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Preserve legacy `channel:` sender IDs:** `src/config/types.tools.ts:230`
  - body: Adding `channel` to the typed prefixes makes an existing legacy key like `channel:alice` enter the channel parser. Because the value has no second colon, `normalizeChannelSenderKey` returns empty and `parseSenderPolicyKey` drops the entry before the legacy id fallback, so an existing sender-specific deny/allow can silently stop applying. Only treat `channel:` as typed for the full `channel:<channelId>:<senderId>` form, or fall back to legacy id matching when channel normalization fails.
  - confidence: 0.88
