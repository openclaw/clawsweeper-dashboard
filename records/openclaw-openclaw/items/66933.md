## Summary

Keep open. Current main still lacks global and per-agent sender-scoped tool policy, so the PR is not obsolete, but the branch is not merge-ready: the new channel-scoped matcher uses raw provider strings where OpenClaw already has canonical channel normalization, and the public config-surface bookkeeping still needs to be completed before this security-sensitive tool boundary lands.

## What This Changes

The PR adds global and per-agent `toolsBySender` config, explicit `channel:<provider>:<senderId>` keys, and sender-specific policy filtering across core, plugin, and bundled tool resolution.

## Best Possible Solution

Keep the PR open and finish it on the contributor branch if maintainers accept the direction: canonicalize channel identities before sender-key matching, update generated config docs metadata and changelog, add focused regression tests for each filtering path, and document the final no-sender wildcard semantics.

## Reproduction Assessment

Yes for the main blocking behavior: a static reproduction is `toolsBySender: { "channel:msteams:alice": { deny: ["exec"] } }` with `messageProvider: "teams"` and `senderId: "alice"`; the PR builds `teams:alice` while existing channel normalization would resolve `teams` to `msteams`. No live test was run because this was a read-only review.

## Solution Assessment

No, not as-is. The feature direction is plausible, but the maintainable fix should reuse OpenClaw's canonical channel normalization and complete the config docs, changelog, and targeted test coverage before merging.
