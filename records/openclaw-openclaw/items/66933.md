## Summary

Keep this PR open, but do not merge it yet: the branch is close on the core/global path, while the advertised channel-scoped group policy still fails in plugin-owned Slack/MS Teams group resolvers because the canonical channel id is not threaded through that SDK context.

## What This Changes

This PR adds global/per-agent `toolsBySender`, channel-scoped sender-key matching, tool-filtering/schema/docs/test updates, and a small cron guard for untargeted system events.

## Best Possible Solution

Thread the canonical channel id through `ChannelGroupContext` and all plugin-owned group `toolsBySender` resolvers, then keep the docs/schema/tests aligned around that single channel-scoped contract.

## Reproduction Assessment

Yes for the blocking review finding. Source inspection shows channel-scoped keys require `messageProvider`, while Slack/MS Teams plugin group resolvers receive and pass no provider value, so those keys cannot match there.

## Solution Assessment

No. The proposed direction is maintainable, but the current patch is incomplete until plugin-owned group policy paths get the same canonical channel id as the core/global paths.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Thread channel ids into plugin group policy:** `src/agents/pi-tools.policy.ts:564`
  - body: Passing `messageProvider` only into the core fallback leaves loaded channel plugins without the provider value needed by the new `channel:<channelId>:<senderId>` bucket. Slack and MS Teams resolve group `toolsBySender` through plugin callbacks that pass only sender fields, so a config such as `channel:slack:U1: { deny: ["exec"] }` falls through and leaves the restricted tools visible. Add the canonical channel id to the plugin group context and use it in the plugin resolvers/tests before advertising this key format for those group configs.
