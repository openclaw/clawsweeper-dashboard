## Summary

Keep this issue open. Current main has useful adjacent fixes for specific loop subcases and better Control UI tool-event plumbing, but the default web UI problem is not fully resolved: repetitive loop detection is still disabled by default, there is no documented `tools.maxToolCalls` knob, and the report also covers stale refreshes, silent terminal states, interruptions, and unsafe config edits.

## What This Changes

The issue reports runaway tool-call loops, missing visible progress, stale Control UI updates, silent terminal states, interruptions, and unsafe config edits in the web UI on OpenClaw 2026.3.13.

## Best Possible Solution

Keep this open and route it to agent runtime plus Control UI/Gateway maintainers. The end state should be a default-safe runaway guard or hard per-turn tool-call circuit breaker with visible progress/failure summaries, an end-to-end web UI regression for repeated tool calls and final refresh, accurate docs for supported knobs, and maintainer judgment on safeguards for agent-initiated config writes.

## Reproduction Assessment

No high-confidence reproduction exists for the full issue: the report has screenshots and a loose `just run openclaw` path but no deterministic prompt/session JSONL for the reported provider/model. Related #34574 and #72587 provide strong subcase evidence, but they do not reproduce the broader default web UI behavior.

## Solution Assessment

No: the shipped adjacent fixes are useful but not the best complete solution for this issue because they cover opt-in loop detection and specific validation loops. A safer solution is a default-visible runaway guard plus UI terminal-state coverage and documentation of the real supported config surface.
