## Summary

Keep open: source inspection supports a real gap for catalog-provided provider headers when they exist only on the implicit provider catalog path, even though explicit `models.providers.<id>.headers` is already covered.

## What This Changes

The issue asks OpenClaw to merge provider catalog-level `headers` into resolved `model.headers` before `wrapStreamFn` runs, or to document the required workaround.

## Best Possible Solution

Hydrate existing catalog provider headers into the resolved runtime model before provider stream wrappers run, with regression coverage for an implicit provider catalog and no new config knob.

I did not execute a live custom plugin in this read-only review, so the status is source-reproducible rather than fully reproduced.

## Solution Assessment

Yes, the narrow maintainable direction is to propagate the already-supported catalog header contract into resolved runtime models before `wrapStreamFn`. A doc-only workaround would leave the manifest/provider catalog contract inconsistent with runtime behavior.
