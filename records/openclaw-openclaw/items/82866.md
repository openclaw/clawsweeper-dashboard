## Summary

Keep open. The PR targets a real auth-sensitive startup path, but current main already generates a runtime token before the server binds and rejects truly unauthenticated non-loopback runtime config; this patch over-blocks loopback container startup and the existing runtime-token bootstrap path, and it lacks real Docker/live proof.

## What This Changes

The PR changes gateway CLI startup guards and adds tests so `--allow-unconfigured` only bypasses config validation for non-container loopback, and so non-loopback binds without explicitly configured token/password auth are rejected before gateway startup.

## Best Possible Solution

Preserve the existing runtime-token and loopback-only semantics unless maintainers explicitly retire them, and if Docker defaults need tightening, make that change at the Docker/config startup boundary with live container proof and focused regression coverage.

## Reproduction Assessment

No, not for the full security claim. Source inspection confirms the Dockerfile uses `--allow-unconfigured` and the CLI guard is bypassed, but it also shows current main generates a runtime token and rejects truly unauthenticated non-loopback runtime config; live Docker proof is still needed for the reported unauthenticated access.

## Solution Assessment

No, not as written. The patch is broader than the proven issue because it blocks local-only loopback startup inside containers and removes the existing runtime-token bootstrap path before maintainer policy has been settled.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Keep loopback-only container starts eligible for the bypass:** `src/cli/gateway-cli/run.ts:263-264`
  - body: This treats every container as unsafe even when the effective bind is `loopback`. The Dockerfile comments and gateway bind docs say loopback inside Docker bridge networking is not reachable from the host through published ports, so `openclaw gateway --bind loopback --allow-unconfigured` would now fail even though it remains local-only and outside the reported exposure path.
  - confidence: 0.91
- **[P2] Preserve the runtime token bootstrap path:** `src/cli/gateway-cli/run.ts:225-228`
  - body: This helper now blocks non-loopback default token mode whenever no token/password is explicitly configured, so `startGatewayServer` never reaches the existing startup path that generates a random runtime token. Current main has `canBootstrapToken` for this case and docs/tests describe generated runtime-token startup, so removing it is a behavior-policy change rather than a narrow fix for unauthenticated exposure.
  - confidence: 0.86
