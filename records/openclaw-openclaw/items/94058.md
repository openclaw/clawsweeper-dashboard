## Summary

Keep open: current main and the latest release still have the source path where a non-fatal isolated agent result can be overwritten by a delivery-phase error, and the direct fixing pull requests are still open rather than merged.

## What This Changes

The issue reports scheduled isolated cron `agentTurn` runs where the isolated session succeeds and produces a final assistant message, but the outer cron run is persisted as `status=error` with `deliveryStatus=not-delivered`.

## Best Possible Solution

Review and land one narrow cron finalization fix that keeps isolated execution status separate from delivery status while preserving delivery diagnostics and backoff semantics; leave broader `warning` UI/status-model work to the related product-direction issue.

## Reproduction Assessment

No live scheduled reproduction was established in this read-only review, but the current source path is high-confidence: non-fatal isolated output can return a delivery `status: "error"` as the cron run status.

## Solution Assessment

Yes, the narrow fix belongs in cron finalization: keep the agent execution outcome and delivery outcome separate there, while maintainers decide whether the user-visible status should remain `ok` plus `not-delivered` or become a broader warning state.
