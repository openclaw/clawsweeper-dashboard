## Summary

Keep open. Current main still awaits Telegram audio download/transcription and auto-TTS in the reply path, and the remaining priority queue, streaming, caching, and voice/text concurrency work needs product and architecture decisions rather than conservative cleanup.

## What This Changes

Request to improve Telegram voice-message latency and throughput with priority handling, streaming transcription/TTS, caching, and concurrent voice/text processing.

## Best Possible Solution

Design a generic media/TTS pipeline state contract first, then add a Telegram-owned implementation that preserves authorization, transcript trust, session/topic ordering, and explicit non-blocking delivery semantics.

## Reproduction Assessment

Yes at source level: current main awaits Telegram STT preflight, awaits auto-TTS before delivery, and serializes per-lane inbound work. I did not run a live Telegram/local-TTS latency benchmark, so the exact latency targets are not reproduced.

## Solution Assessment

No single narrow patch is the best way to solve this issue. The maintainable path is a designed pipeline/state contract with Telegram integration, not a Telegram-only shortcut that bypasses ordering, authorization, or transcript-trust boundaries.
