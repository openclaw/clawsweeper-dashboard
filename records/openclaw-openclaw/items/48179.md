## Summary

Keep open. Current main has Telegram audio preflight and voice-note TTS support, but STT/TTS are still awaited and same-key inbound Telegram work is serialized; the requested priority queue, streaming/cached pipeline, and non-blocking voice/text concurrency remain product/runtime work.

## What This Changes

The issue requests lower-latency Telegram voice handling through priority queuing, streaming transcription, TTS/transcription caching, and concurrent voice/text processing.

## Best Possible Solution

Design a Telegram/generic media-processing pipeline that preserves authorization and session ordering while adding explicit stage state, timing, STT/TTS concurrency, and cache policy before implementation.

## Reproduction Assessment

Yes at source level for the blocking architecture: Telegram awaits STT preflight during context construction, awaits TTS before media payload delivery, and serializes same-key inbound work. No live Telegram/macOS latency benchmark was run.

## Solution Assessment

No as a one-off Telegram fast path. The maintainable fix needs an explicit media-pipeline design that preserves auth, ordering, session state, telemetry, and cache semantics.
