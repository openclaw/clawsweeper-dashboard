## Summary

Keep open. Current main and the latest release still have the broad Telegram voice-pipeline gap: voice body resolution can await STT before reply dispatch, Telegram inbound work still serializes same-key processing, and TTS delivery still awaits synthesis; the open typing-cue PR only covers a narrow slice.

## What This Changes

The issue requests lower-latency Telegram voice-message handling through priority queueing, streaming STT/TTS, caching, and concurrent voice/text processing.

## Best Possible Solution

Keep this as the canonical broad Telegram voice-pipeline design tracker, land narrow proven fixes separately, and require stage timing/instrumentation plus live Telegram evidence before queueing, streaming, caching, or async TTS changes.

## Reproduction Assessment

Yes at source level for the blocking pieces: Telegram can await audio preflight before direct typing, inbound work is same-key serialized, and TTS synthesis is awaited before delivery. I did not establish a live timing benchmark for the stated latency targets.

## Solution Assessment

No as one broad implementation sweep. The best path is to split bounded work: keep https://github.com/openclaw/openclaw/issues/58887 and https://github.com/openclaw/openclaw/pull/96278 on the typing cue, keep https://github.com/openclaw/openclaw/issues/42539 on delivery mode, and design instrumentation/queueing before changing pipeline concurrency.
