Keep open. Current main and v2026.6.6 still use awaited Telegram audio preflight, same-key inbound serialization, and synchronous auto-TTS reply payload generation; the remaining request is broad media/TTS pipeline design, not stale cleanup.

## What This Changes

The issue requests lower-latency Telegram voice-message handling through priority queuing, streaming transcription, caching, and non-blocking voice/TTS concurrency.

## Best Possible Solution

Design a generic media/TTS pipeline state and delivery policy first, then add a Telegram-owned implementation that preserves authorization, transcript trust, session/topic ordering, and explicit non-blocking voice behavior.

## Reproduction Assessment

Yes at source level: current main awaits Telegram STT preflight, awaits auto-TTS before reply delivery, and serializes same-key inbound work. I did not run a live Telegram/local-TTS latency benchmark, so the exact latency targets remain unmeasured.

## Solution Assessment

No single narrow patch is the best way to solve this issue. A maintainable fix needs a designed pipeline/state contract with Telegram integration, not a Telegram-only shortcut that bypasses ordering, authorization, transcript-trust, or TTS preference boundaries.
