## Summary

Keep open. Current main has Telegram audio preflight and voice-note TTS, but it still uses awaited STT/TTS paths and keyed inbound serialization; the requested priority queue, streaming/cached voice pipeline, and non-blocking voice/text concurrency remain product and architecture work.

## What This Changes

Request to improve Telegram voice-message latency and throughput through priority handling, streaming transcription/TTS, caching, and concurrent voice/text processing.

## Best Possible Solution

Design a generic media/TTS pipeline state model first, then add a Telegram-owned implementation that preserves transcript trust, session/topic ordering, and explicit non-blocking TTS/voice behavior.

## Reproduction Assessment

Yes at source level: current main awaits Telegram STT preflight, awaits auto-TTS before reply delivery, and serializes same-key inbound work. I did not run a live Telegram/local-TTS latency benchmark, so the exact latency target is unmeasured here.

## Solution Assessment

No single narrow patch is the best way to solve the issue. The maintainable path is a designed pipeline/state contract with Telegram integration, not a Telegram-only shortcut that bypasses existing ordering, authorization, or transcript-trust boundaries.
