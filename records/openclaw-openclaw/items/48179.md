Request to optimize Telegram voice-message processing with priority handling, streaming transcription, TTS caching, and concurrent voice/text processing.

## Best Possible Solution

Keep this as the canonical broad design tracker, land or close the narrow typing-cue PR separately, and design an instrumented async STT/TTS pipeline with explicit state-stage proof and Telegram live benchmarks before implementation.

## Reproduction Assessment

Yes at source level: current main awaits voice preflight transcription and awaits TTS synthesis in the reply path, with normal Telegram updates serialized by chat/topic key. No live benchmark proves the exact latency targets or macOS arm64 timings.

## Solution Assessment

Unclear as a single implementation. The best path is to split narrow UX work from the broader async pipeline design, because priority queues, streaming STT, caching, and background TTS need maintainer decisions about ordering, state visibility, and delivery semantics.
