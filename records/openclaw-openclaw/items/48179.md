## Summary

Keep open. Current main and v2026.6.6 still have awaited Telegram STT preflight, awaited auto-TTS before reply delivery, and serialized same-lane inbound processing; the requested priority queue, streaming/caching, and non-blocking voice/text concurrency remain broad product/architecture work rather than a cleanup close.

## What This Changes

The issue requests a lower-latency Telegram voice pipeline with priority handling, streaming transcription, caching, and non-blocking voice/text concurrency.

## Best Possible Solution

Design a generic media/TTS pipeline state model first, then add a Telegram-owned implementation that preserves authorization, transcript trust, session/topic ordering, and explicit non-blocking delivery semantics.

## Reproduction Assessment

Yes at source level: current main awaits Telegram STT preflight, awaits auto-TTS before final delivery, and serializes per-lane inbound work. I did not run a live Telegram/local-TTS latency benchmark, so the exact latency targets are not reproduced.

## Solution Assessment

No single narrow patch is the best way to solve this issue. The maintainable path is a designed pipeline/state contract with Telegram integration, not a Telegram-only shortcut that bypasses ordering, authorization, or transcript-trust boundaries.
