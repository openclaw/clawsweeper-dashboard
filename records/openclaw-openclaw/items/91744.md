## Label Justifications

- `P1`: Affected Linux gateway plus remote Mac iMessage setups can deliver an attachment path that Codex cannot read, breaking a real channel-to-agent media workflow.
- `impact:message-loss`: The message text can arrive while the attached image content is effectively lost to the agent because the primary media path points at an unreachable Mac filesystem path.

## Summary

Keep open: current main and the latest release still derive plugin-bound Codex media metadata from raw remote iMessage paths before the remote-cache rewrite, and the linked closing PR is open rather than merged.

## What This Changes

The issue asks OpenClaw to expose staged local remote-cache paths, not raw macOS Messages attachment paths, in final agent/Codex-visible metadata for remote iMessage media.

## Best Possible Solution

Review, repair if needed, and land https://github.com/openclaw/openclaw/pull/91803 or an equivalent fix that stages remote iMessage media before plugin-bound Codex metadata is constructed.

## Reproduction Assessment

Yes. Current main source shows remote iMessage can emit raw MediaPaths with MediaRemoteHost, dispatch builds plugin-bound claim metadata before staging, and Codex consumes that metadata as local image path input.

## Solution Assessment

Yes. The maintainable fix is to reuse the existing stageSandboxMedia rewrite before plugin/Codex metadata construction; teaching Codex about channel-specific raw macOS paths would put the repair in the wrong layer.
