## Summary

Keep open: current main still constructs plugin-bound attachment metadata before remote staging rewrites the path, and open PR https://github.com/openclaw/openclaw/pull/91803 explicitly owns the focused repair. The earlier https://github.com/openclaw/openclaw/issues/87089 fix covered media-understanding order, not this earlier plugin-claim path. Likely related people: @steipete and Vincent Koc for recent core dispatch work, both at medium confidence.

## What This Changes

Propagate the staged local cache path for remote iMessage attachments into plugin-bound and Codex-visible media metadata instead of exposing the original macOS Messages path.

## Best Possible Solution

Stage and rewrite remote media once before the first plugin or agent consumer, preserve the existing remote-root and size protections, and mark successful staging so the later reply path cannot repeat the transfer.

## Reproduction Assessment

Yes. Current-main source provides a high-confidence reproduction path: plugin metadata is derived from the raw remote path, and Codex maps that path directly to localImage before later reply staging can rewrite the context.

## Solution Assessment

Yes in direction. Rewriting the canonical inbound context before its first plugin consumer is narrower and safer than adding a Codex-specific cache search or fallback; the linked PR still needs review of earlier-transfer failure semantics.
