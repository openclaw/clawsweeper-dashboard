## Summary

Keep open: current main and the latest release still have the source-reproducible bug path, and the linked implementation PR that closes this issue remains open and mergeable rather than merged.

## What This Changes

The issue asks for remote iMessage attachments to expose the staged local remote-cache path to Codex/tool metadata instead of the raw macOS Messages attachment path.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/91803 if maintainers accept the earlier SCP-staging tradeoff, then close this issue as implemented by that PR.

## Reproduction Assessment

Yes. The source path is clear: remote iMessage supplies raw MediaPaths plus MediaRemoteHost, dispatch builds plugin-bound inbound claim metadata before staging, and Codex converts those paths into localImage inputs.

## Solution Assessment

Yes. Staging remote inbound media before hook-context and inbound-claim construction is the narrow owner-boundary fix; making Codex understand raw macOS Messages paths would put channel-specific SCP handling in the wrong layer.
