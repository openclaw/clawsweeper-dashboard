## Label Justifications

- `P1`: Remote iMessage photo attachments can reach the agent with an unreachable Mac-only path, breaking a real channel-to-Codex workflow for affected Linux gateway setups.
- `impact:message-loss`: The text message is delivered, but the attached image content can be unavailable to the agent because the visible media path points at an unreadable remote Mac location.

## Summary

Keep open: current main still lets remote iMessage raw attachment paths enter plugin-bound Codex metadata before the staged cache rewrite, and the linked closing PR is open rather than merged.

## What This Changes

Remote iMessage attachments should expose the staged local remote-cache path to Codex/tool-facing media metadata instead of the raw macOS Messages attachment path.

## Best Possible Solution

Review, repair if needed, and land https://github.com/openclaw/openclaw/pull/91803 or an equivalent fix so remote iMessage media is staged before plugin-bound Codex metadata is constructed.

## Reproduction Assessment

Yes. Current main source shows remote iMessage can emit raw MediaPaths with MediaRemoteHost, dispatch builds plugin-bound claim metadata before staging, and Codex consumes that metadata as localImage path input.

## Solution Assessment

Yes. The maintainable fix is to reuse the existing stageSandboxMedia rewrite before plugin/Codex metadata construction; making Codex understand channel-specific raw macOS Messages paths would put the repair in the wrong layer.
