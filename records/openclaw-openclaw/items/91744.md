## What This Changes

The issue reports remote iMessage attachments being SCP-staged locally while final Codex-visible media metadata still points to the raw macOS Messages attachment path.

## Best Possible Solution

Review, repair if needed, and land https://github.com/openclaw/openclaw/pull/91803 or an equivalent fix that stages remote iMessage media before plugin-bound Codex metadata is constructed.

## Reproduction Assessment

Yes. Source inspection shows remote iMessage can emit raw MediaPaths with MediaRemoteHost, dispatch builds plugin-bound Codex claim metadata before staging, and Codex consumes that path as a local image input.

## Solution Assessment

Yes. The best fix is to reuse OpenClaw’s existing remote media staging before plugin/Codex metadata construction; teaching Codex about channel-specific raw macOS paths would put the repair in the wrong layer.
