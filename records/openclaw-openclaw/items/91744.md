## Summary

Keep open: current main still has the source-level path that can build plugin-bound Codex media metadata from raw remote iMessage paths before staging rewrites them, and the linked implementation PR is open rather than merged.

## What This Changes

The issue reports that remote iMessage attachments are SCP-staged locally but final Codex/tool-facing media metadata can still expose the original macOS Messages attachment path.

## Best Possible Solution

Review and land the linked staging-order fix, or an equivalent replacement, so remote iMessage media is staged before plugin-bound Codex claim metadata is constructed while original Mac paths remain debug/original metadata only.

## Reproduction Assessment

Yes. Current main source shows remote iMessage can populate raw `MediaPaths`, dispatch builds plugin-bound claim metadata before staging, and Codex maps those paths into `localImage` inputs; I did not run a live remote Mac setup in this read-only review.

## Solution Assessment

Yes. The best fix is to reuse the existing `stageSandboxMedia()` rewrite contract before hook-context and inbound-claim construction; making Codex chase channel-specific raw macOS Messages paths would put the repair in the wrong layer.
