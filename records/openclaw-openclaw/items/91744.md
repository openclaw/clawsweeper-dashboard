## Label Justifications

- `P1`: Remote iMessage photo attachments can reach the agent with an unreachable path, breaking a real channel-to-agent workflow for affected Linux gateway setups.
- `impact:message-loss`: The attachment content is effectively lost to the agent/tool path when the staged file exists but the visible metadata still points at the Mac-only path.

## Summary

Keep open: current main still has a pre-Codex-claim path that can expose the raw macOS attachment path. The prior fix covers get-reply media understanding, but plugin-bound Codex inbound claims are built before remote media staging, and Codex forwards metadata paths as localImage inputs.

## What This Changes

The issue asks OpenClaw to expose staged local cache paths, not raw macOS Messages attachment paths, in final Codex-visible metadata for remote iMessage attachments.

## Best Possible Solution

Centralize remote inbound media staging before every agent/plugin dispatch surface that consumes media metadata, then cover remote iMessage to Codex binding with a regression test.

## Reproduction Assessment

Yes, this is source-reproducible from the remote iMessage plus Codex binding path: raw MediaPaths are converted into an inbound claim event before any staging code can rewrite them. I did not run a live remote Mac scenario in this read-only review.

## Solution Assessment

Yes, the reported fix direction is the right boundary, but it needs to cover dispatch-from-config inbound claim metadata rather than only the get-reply media-understanding path. Teaching Codex to chase raw /Users paths would be the wrong layer.
