## Label Justifications

- `P1`: Remote iMessage attachments can reach Codex as unreadable Mac filesystem paths, breaking a real channel-to-agent media workflow.
- `impact:message-loss`: The message text can arrive while the attached image content is effectively lost to the agent because the primary media path is unreachable.

## Summary

Keep open: current main still builds plugin-bound Codex media metadata from the raw remote iMessage context before dispatch-side staging can rewrite it, and the linked closing PR is open rather than merged.

## What This Changes

Ensure staged remote iMessage attachments expose the local remote-cache path as the primary Codex/tool-visible media path instead of the raw macOS Messages attachment path.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/91803 or an equivalent dispatch-layer fix that stages remote iMessage media before plugin-bound Codex metadata is constructed.

## Reproduction Assessment

Yes. Source inspection shows remote iMessage can emit raw MediaPath(s) with MediaRemoteHost, current main builds plugin-bound Codex claim metadata before staging, and Codex consumes that metadata as localImage filesystem input.

## Solution Assessment

Yes. The best fix is to reuse OpenClaw remote media staging before plugin/Codex metadata construction; teaching Codex about channel-specific raw macOS Messages paths would put the repair in the wrong layer.
