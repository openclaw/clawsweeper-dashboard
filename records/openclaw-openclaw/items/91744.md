## Label Justifications

- `P1`: The bug breaks a real remote iMessage-to-Codex attachment workflow by giving the agent an unreadable media path.
- `impact:message-loss`: The message can arrive while the attached image content is effectively lost to the agent because the primary path points at a remote Mac filesystem.

## Summary

Keep open: current main still has the dispatch/plugin metadata ordering gap, and the open closing PR is the right candidate path but has not merged yet.

## What This Changes

Remote iMessage attachments can be staged into the local remote cache but still expose the raw macOS Messages path as the Codex-visible media path.

## Best Possible Solution

Review, repair if needed, and land https://github.com/openclaw/openclaw/pull/91803 or an equivalent fix that stages remote iMessage media before plugin-bound Codex metadata is constructed.

## Reproduction Assessment

Yes. Current main source shows remote iMessage can provide raw MediaPath/MediaPaths with MediaRemoteHost, dispatch builds plugin-bound claim metadata before dispatch-level staging, and Codex consumes that metadata as localImage path input.

## Solution Assessment

Yes. The maintainable fix is to reuse the existing stageSandboxMedia rewrite before plugin/Codex metadata construction; teaching Codex channel-specific raw macOS Messages paths would put the repair in the wrong layer.
