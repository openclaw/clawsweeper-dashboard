## Summary

Keep open: current main still lets raw remote iMessage attachment paths reach plugin-bound Codex media metadata before the existing staging rewrite, and the linked closing PR is open rather than merged.

## What This Changes

The issue asks for staged remote iMessage attachment cache paths to be propagated into final agent/Codex-visible media metadata instead of raw macOS Messages paths.

## Best Possible Solution

Review, repair if needed, and land https://github.com/openclaw/openclaw/pull/91803 or an equivalent fix that stages remote iMessage media before plugin-bound Codex metadata is constructed.

## Reproduction Assessment

Yes. Current main source shows remote iMessage can emit raw MediaPaths with MediaRemoteHost, dispatch builds plugin-bound claim metadata before staging, and Codex consumes that metadata as local image path input.

## Solution Assessment

Yes. The maintainable fix is to reuse the existing stageSandboxMedia rewrite before plugin/Codex metadata construction; making Codex understand channel-specific raw macOS Messages paths would put the repair in the wrong layer.
