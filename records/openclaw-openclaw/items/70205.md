The branch aggregates gateway `usage.cost` summaries across configured agents, adds gateway/UI regression coverage, preserves date-only Control UI usage labels, and adds a changelog entry.

## Best Possible Solution

Keep the aggregation and date-label fixes, but rebase onto current main and preserve range presets, durable per-agent cache/cacheStatus behavior, bounded cache semantics, and `sessions.usage` lineage rollups.

## Reproduction Assessment

Yes. Current main source shows the scope asymmetry and the PR body plus inspected screenshots show the before/after Control UI symptom for daily token/cost data.

## Solution Assessment

No, not as-is. Aggregating `usage.cost` across configured agents is the right narrow fix, but the head must be rebased and adapted to current main’s range, cacheStatus, and lineage contracts before it is mergeable.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Preserve sessions.usage cache and lineage handling:** `src/gateway/server-methods/usage.ts:837-845`
  - body: Current main rolls up `includedSessionIds` and calls `loadSessionCostSummaryFromCache` so the UI receives lineage totals and cacheStatus. This head calls raw `loadSessionCostSummary` once per listed session, which would drop historical lineage totals and stale/refresh visibility from `sessions.usage`.
  - confidence: 0.91
- **[P2] Route aggregated usage.cost through the durable cache:** `src/gateway/server-methods/usage.ts:454-486`
  - body: The new aggregation path calls raw `loadCostUsageSummary` for each resolved agent, bypassing `loadCostUsageSummaryFromCache` and its refresh/cacheStatus contract. Route each agent through the durable cached loader and merge the returned cacheStatus so the UI keeps stale/refresh state and the gateway avoids direct rescans.
  - confidence: 0.9
- **[P2] Forward range presets into usage.cost:** `src/gateway/server-methods/usage.ts:587-593`
  - body: Current main accepts `params.range` for `usage.cost`, but this handler rebuild omits it when calling `parseDateRange`. Callers using presets such as `7d`, `90d`, or `all` would get the default window instead of the selected range.
  - confidence: 0.92
