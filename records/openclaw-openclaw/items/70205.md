## Summary

Keep open: the aggregation direction and UI date-label fix are useful, and the PR has real behavior proof, but the current head would regress newer main behavior in usage range presets, durable cost cache handling, and sessions.usage lineage/cacheStatus reporting.

## What This Changes

The PR updates gateway usage.cost aggregation across configured agents, adds focused gateway/UI tests, preserves Control UI date-only usage labels, and adds a changelog entry.

## Best Possible Solution

Adapt the aggregation onto current main so usage.cost resolves configured agents, uses the durable per-agent cache with merged cacheStatus, forwards range presets, and leaves the broader ledger and provider cost-normalization work to their own threads.

## Reproduction Assessment

Yes. Source inspection shows current main still has the reported scope asymmetry: sessions.usage enumerates configured agents while usage.cost reads one unscoped/default transcript cost summary path, and the PR body includes before/after UI screenshots for that symptom.

## Solution Assessment

No, not as-is. Aggregating usage.cost across configured agents is the right narrow direction, but this head must preserve current-main range presets, durable cache/cacheStatus behavior, and sessions.usage lineage rollups before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Route each agent through the durable cost cache:** `src/gateway/server-methods/usage.ts:454-486`
  - body: The new aggregation helper calls raw loadCostUsageSummary for each resolved agent, bypassing loadCostUsageSummaryFromCache and its refresh/cacheStatus contract. That would lose stale/refresh visibility and force direct rescans instead of using the durable transcript aggregate cache; route each agent through the cached loader and merge the returned cacheStatus.
  - confidence: 0.9
- **[P2] Forward range presets into usage.cost:** `src/gateway/server-methods/usage.ts:587-593`
  - body: This handler rebuild omits params.range when calling parseDateRange, even though current main accepts presets such as 7d, 90d, and all. Callers using those presets would get the default date window instead of their selected range.
  - confidence: 0.92
- **[P2] Preserve sessions.usage lineage and cache status:** `src/gateway/server-methods/usage.ts:837-845`
  - body: Current main rolls usage across includedSessionIds and calls loadSessionCostSummaryFromCache so cacheStatus reaches the UI. This head calls raw loadSessionCostSummary once per listed session, which would drop historical lineage totals and stale/refresh reporting from sessions.usage.
  - confidence: 0.91
