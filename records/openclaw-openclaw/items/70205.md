## Summary

Keep open: the PR targets a valid multi-agent usage.cost scope bug and includes useful proof, but the current head would regress current-main usage range and durable cache/cacheStatus behavior before merge.

## What This Changes

The PR changes gateway usage.cost to resolve and merge cost summaries across configured agents, adds focused gateway/UI tests, preserves date-only usage labels in the Control UI, and adds a changelog entry.

## Best Possible Solution

Keep the aggregation and date-label direction, but adapt it onto current main by preserving range presets, durable per-agent cache/status handling, and sessions.usage lineage/cacheStatus contracts.

## Reproduction Assessment

Yes. Source inspection shows current main still has the scope asymmetry: sessions.usage enumerates configured agents while usage.cost uses one unscoped cached summary path, and the PR body includes before/after real UI screenshots for the symptom.

## Solution Assessment

No, not as-is. Aggregating usage.cost across configured agents is the right narrow fix, but this head must preserve current-main range presets, durable cache/cacheStatus behavior, and sessions.usage lineage rollups before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Preserve sessions.usage cache and lineage handling:** `src/gateway/server-methods/usage.ts:837-845`
  - body: Current main rolls up usage across includedSessionIds and calls loadSessionCostSummaryFromCache so cacheStatus is returned to the UI. This head calls raw loadSessionCostSummary once per listed session, so merging it would drop session lineage totals and stale/refresh visibility from sessions.usage.
  - confidence: 0.91

  - body: The new aggregation path calls raw loadCostUsageSummary for each resolved agent, bypassing loadCostUsageSummaryFromCache and its refresh/cacheStatus contract. Route each agent through the durable cached loader and merge the returned cacheStatus so the UI keeps stale/refresh state and the gateway avoids direct rescans.
  - confidence: 0.9

  - body: Current main accepts params.range for usage.cost, but this handler rebuild omits it when calling parseDateRange. Callers using presets such as 7d, 90d, or all would get the default window instead of their selected range.
  - confidence: 0.92
