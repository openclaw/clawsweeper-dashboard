## Summary

Keep open: this is the canonical focused fix for the multi-agent `usage.cost` scope mismatch, and the PR includes useful after-fix UI proof, but the current head would regress durable usage cache behavior, range presets, and `sessions.usage` lineage/cache status from current `main`.

## What This Changes

The PR aggregates unscoped gateway `usage.cost` across configured agents, adds focused gateway/UI tests, fixes date-only usage labels in the Control UI, and updates the changelog.

## Best Possible Solution

Land a rebased/adapted version that aggregates `usage.cost` over configured agents while preserving current-main cacheStatus, range parsing, and session lineage behavior; leave provider cost normalization and dedicated usage ledger work to their separate threads.

## Reproduction Assessment

Yes. Source inspection gives a concrete current-main path: configure multiple gateway agents with transcript usage, then compare unscoped `usage.cost` with `sessions.usage`; the PR body also includes before/after dashboard screenshots for that symptom.

## Solution Assessment

No, not as-is. Aggregating `usage.cost` across configured agents is the right narrow direction, but this head must preserve current-main durable cache/cacheStatus behavior, range presets, and session lineage rollups before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Route each agent through the durable cost cache:** `src/gateway/server-methods/usage.ts:454-486`
  - body: The new aggregation path calls raw `loadCostUsageSummary` for each resolved agent, bypassing `loadCostUsageSummaryFromCache` and its refresh/cacheStatus contract. That loses stale/refresh visibility and forces direct rescans instead of using the durable aggregate cache; route each agent through the cached loader and merge the returned cacheStatus.
  - confidence: 0.9
- **[P2] Forward range presets into usage.cost:** `src/gateway/server-methods/usage.ts:587-593`
  - body: This handler rebuild omits `params.range` when calling `parseDateRange`, even though current main accepts presets such as `7d`, `90d`, and `all`. Callers using those presets would silently get the default date window instead of their selected range.
  - confidence: 0.92
- **[P2] Preserve sessions.usage lineage and cache status:** `src/gateway/server-methods/usage.ts:837-845`
  - body: Current main rolls usage across `includedSessionIds` and calls `loadSessionCostSummaryFromCache` so cacheStatus reaches the UI. This head calls raw `loadSessionCostSummary` once for the listed session, which drops historical lineage totals and stale/refresh reporting from `sessions.usage`.
  - confidence: 0.91
