## Summary

Keep open: the aggregation direction is useful and has real behavior proof, but the current PR head still regresses existing `usage.cost` range presets and bypasses the durable cost-cache path on current main.

## What This Changes

The PR changes gateway `usage.cost` to resolve configured agent scope, merge per-agent cost summaries with agent-scoped cache keys, add gateway/UI coverage, and fix Control UI date-only usage labels.

## Best Possible Solution

Keep the aggregation and UI label fix, but update the gateway path to pass `params?.range`, aggregate through `loadCostUsageSummaryFromCache` per resolved agent, merge cache status, and preserve agent-scoped cache keys.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure multiple gateway agents with usage-bearing transcripts, call `usage.cost` and `sessions.usage` over the same selected range, and compare current main's single-scope cost path against the PR's aggregation path.

## Solution Assessment

No, not as-is. The focused aggregation approach is the right fix direction, but it must preserve current-main range preset parsing and durable cached-loader behavior before merge.
