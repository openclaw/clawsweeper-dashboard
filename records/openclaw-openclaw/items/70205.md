## Summary

Keep open: the underlying multi-agent usage-cost bug is valid and the PR has useful proof, but the current head is not merge-ready because it misses current usage-cost range and durable-cache contracts.

## What This Changes

This PR changes gateway usage.cost to resolve configured agent scopes and merge per-agent cost summaries, adjusts Usage UI date-only labels, adds focused tests, and updates the changelog.

## Best Possible Solution

Keep the aggregation and date-label behavior, but adapt it to current durable usage cache by merging per-agent cached summaries and cacheStatus while preserving range presets.

## Reproduction Assessment

Yes. Source inspection shows current main still scopes usage.cost to the existing cached summary path while sessions.usage discovers configured agents, and the PR body screenshots show the visible No data symptom before and populated charts after.

## Solution Assessment

No, not as-is. Aggregating across configured agents is the right narrow fix, but the branch must preserve params?.range and route each per-agent summary through loadCostUsageSummaryFromCache with cacheStatus merging.

## Review Findings

Overall correctness: patch is incorrect

Full review comments:

- **[P2] Forward range presets into usage.cost:** `src/gateway/server-methods/usage.ts:587-593`
  - body: Current main accepts range presets for usage.cost, but this handler rebuild omits params?.range when calling parseDateRange. Callers using presets such as 7d, 90d, or all would fall back to the default window and get cost totals for the wrong selected range.
  - confidence: 0.92
- **[P2] Use the durable cost cache per agent:** `src/gateway/server-methods/usage.ts:454-486`

Route each resolved agent through the durable cached loader and merge statuses so the gateway avoids direct rescans and the UI keeps stale/refresh visibility.
  - confidence: 0.9
