## Summary

Keep open: this is a focused bug-fix PR with sufficient real-behavior proof, but the current head still drops current-main range preset handling and bypasses the durable usage-cost cache path.

## What This Changes

The PR changes gateway `usage.cost` to aggregate cost summaries across configured agents, scopes explicit `agentId` requests, updates cache keys/tests, and fixes Control UI date-only daily labels.

## Best Possible Solution

Keep the cross-agent aggregation and UI date-label fix, but preserve current-main `range` parsing and durable cache/cacheStatus semantics before merge.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure multiple gateway agents with usage-bearing transcripts, call `sessions.usage` and `usage.cost` for the same range, and current main fans out only `sessions.usage`.

## Solution Assessment

No, not as-is. The aggregation direction is the right narrow fix, but the PR branch should forward `range` and fan out through `loadCostUsageSummaryFromCache` with merged cache status.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Forward range presets into usage.cost:** `src/gateway/server-methods/usage.ts:587-593`
  - body: Current main forwards `range: params?.range` into `parseDateRange`, but this rewrite drops it. Calls using presets such as `7d`, `90d`, or `all` would fall back to the default cost window while `sessions.usage` still honors the selected range.
  - confidence: 0.92
- **[P2] Use the durable cache for per-agent cost scans:** `src/gateway/server-methods/usage.ts:454-486`
  - body: Current main serves gateway cost summaries through `loadCostUsageSummaryFromCache` with sync refresh and `cacheStatus` handling, but this helper fans out through raw `loadCostUsageSummary`. That regresses durable aggregate cache behavior and can force full transcript scans again, so each per-agent load should use the cached loader and merge cache status.
  - confidence: 0.88
