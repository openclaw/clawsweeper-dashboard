## Label Justifications

- `P2`: This is a normal-priority Telegram reply-context bug with bounded channel/session impact and no crash or security signal.
- `impact:message-loss`: A selected quote can cause the rest of the replied-to message to be omitted from the next prompt.
- `impact:session-state`: The omitted reply target affects what context the next agent turn can recover, especially after reply turns or session resets.

## Summary

Keep open: current main and the latest release still render Telegram selected quote text as the only inline reply target, and the full replied-to body is skipped when that inline context exists. This is source-reproducible, but the exact prompt labels and shared budget for emitting both quote and body need maintainer product direction before an implementation lane should start.

## What This Changes

The issue requests changing Telegram inline reply-context framing so `ReplyToQuoteText` does not suppress the full `ReplyToBody` when the selected quote is only a span of the replied-to message.

## Best Possible Solution

Preserve selected-quote primacy, render only the quote when it is effectively the body, and add a bounded full-message excerpt when the quote is a strict substring or materially diverges from the body.

## Reproduction Assessment

Yes, source-reproducible: Telegram can populate `ReplyToQuoteText` separately, current main renders only that selected quote, and the full `ReplyToBody` block is skipped when inline Telegram context exists. I did not run a live Telegram quote-selection reproduction in this read-only review.

## Solution Assessment

Mostly yes: the proposed direction is the narrow maintainable fix, but maintainers should choose the exact labels and per-block budget before code lands. A new config option is not needed for this bug.
