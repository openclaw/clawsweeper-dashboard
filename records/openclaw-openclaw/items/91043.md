## Label Justifications

- `P2`: This is a normal-priority core reply-context bug with bounded but real impact on Telegram selected-quote replies.
- `impact:message-loss`: The reported behavior drops the rest of the replied-to message from the next prompt when Telegram supplies a selected quote span.
- `impact:session-state`: The missing reply target context can make the next agent turn reason from incomplete conversation state.

## Summary

Keep open: current main and the latest release still render Telegram `ReplyToQuoteText` as the inline reply context and suppress the separate `ReplyToBody` block whenever that inline context exists. The remaining work is valid, but the discussion identifies unresolved prompt-contract choices for quote/body labels and budget allocation.

## What This Changes

The issue asks OpenClaw to preserve full replied-to message context when Telegram supplies a selected `ReplyToQuoteText` span that differs from `ReplyToBody`.

## Best Possible Solution

Have a maintainer choose the quote/body prompt contract, then update `resolveInlineReplyQuote` and focused tests so selected quote spans keep primacy without hiding the relevant full-message excerpt.

## Reproduction Assessment

Yes from source: current main still uses `ReplyToQuoteText` before `ReplyToBody`, and the fallback body block is skipped when the Telegram inline context exists. I did not run live Telegram quote-selection proof in this read-only review.

## Solution Assessment

Unclear until maintainers choose the prompt contract. The proposed direction is plausible and narrow, but label wording and budget allocation should be decided before implementation.
