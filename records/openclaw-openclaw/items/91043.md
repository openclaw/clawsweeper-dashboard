## Label Justifications

- `P2`: This is a normal-priority reply-context bug with real but bounded impact on multi-turn channel behavior.
- `impact:message-loss`: A selected reply quote can cause the rest of the replied-to message to be omitted from the next prompt.
- `impact:session-state`: The omitted reply target context affects what the next agent turn can recover after a reply or reset.

## Summary

Keep open: current main and the latest release still short-circuit Telegram selected quote text over the full replied-to body, and the active related PR at https://github.com/openclaw/openclaw/pull/87909 still has the same nullish-coalescing shape for this exact follow-up. The issue is valid, but the best next step is maintainer review or amendment of that active reply-context PR rather than closing this as already handled.

## What This Changes

The issue requests preserving Telegram selected-quote primacy while also including a bounded full-body excerpt when the selected quote differs from the full replied-to message.

## Best Possible Solution

Preserve selected-quote primacy, render only the quote when it equals the body, and add a bounded full-message excerpt when the selected quote is a strict substring or materially diverges from the replied-to body.

## Reproduction Assessment

Yes, source-reproducible: Telegram can populate ReplyToQuoteText separately, current main renders only that selected quote, and the separate ReplyToBody block is skipped when the inline Telegram context exists. I did not run a live Telegram quote-selection reproduction in this read-only review.

## Solution Assessment

Mostly yes: the proposed direction is the narrow maintainable fix, but maintainers should choose the exact prompt labels and per-block budget before it lands. A new config knob is not needed for this bug.
