## Label Justifications

- `P2`: This is a normal-priority reply-context bug with bounded but real impact on Telegram multi-turn behavior.
- `impact:message-loss`: Selected quote text can cause the rest of the replied-to message to be omitted from the next prompt.
- `impact:session-state`: The omitted replied-to body affects the state/context the next agent turn can recover from the user reply.

## Summary

Keep open: current main and the latest release still render Telegram inline reply context by choosing sanitized ReplyToQuoteText before ReplyToBody, and the current test suite still asserts that the full reply body block is suppressed when selected quote text exists. The report is valid, but the exact prompt labels and budget split need maintainer product direction before a fix lane should start.

## What This Changes

The issue asks OpenClaw to preserve full Telegram replied-to message context when selected quote text is present instead of rendering only ReplyToQuoteText.

## Best Possible Solution

Choose the quote/body prompt contract, then update inbound-meta to render the selected quote plus a bounded full-body excerpt when they differ, with regression tests for equal, substring, divergent, and short-snippet cases.

## Reproduction Assessment

Yes, source-reproducible: Telegram context can carry ReplyToQuoteText separately from ReplyToBody, current main renders only the selected quote, and then suppresses the separate reply body block. I did not run a live Telegram quote-selection repro in this read-only review.

## Solution Assessment

Mostly yes: preserving selected-quote primacy while adding a bounded full-body excerpt is the narrow maintainable direction. The maintainer decision is the exact labels and budget split, not whether the source-level gap exists.
