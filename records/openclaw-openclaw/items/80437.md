## Summary

Keep open: current main still has a source-backed path where the Discord native command deploy GET times out after 15s, aborts the deploy without a transient retry, and only logs a non-fatal warning while the reporter now says the failure repeats and native commands are unusable.

## What This Changes

Investigate a Discord native slash-command deployment timeout that repeats on startup and leaves native slash commands unusable.

## Best Possible Solution

Add a focused Discord deploy resilience fix for transient GET/update timeouts during native command reconciliation, preserving the non-fatal background startup behavior and the existing 429/daily-create-limit safeguards.

## Reproduction Assessment

No high-confidence live reproduction was established in this read-only review. We do have a source-backed current-main path for the reported timeout: the deploy reconcile GET uses the 15s Discord REST timeout, AbortError is not retried, and the deploy is logged as a warning.

## Solution Assessment

Yes for the likely direction, with one caveat: the narrow maintainable fix is deploy-specific retry/backoff for transient REST timeouts, not a gateway startup rewrite or a new deploy-mode config. The final patch should also prove that 429 handling does not regress into a retry burst.
