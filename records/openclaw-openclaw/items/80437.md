## Summary

Keep open. The report is low-detail, but current source still has a clear cache-miss path where Discord native command reconcile performs `GET /applications/{appId}/commands` through a 15s REST timeout, and the active deploy-policy work at https://github.com/openclaw/openclaw/issues/75888 / https://github.com/openclaw/openclaw/pull/75961 is related but does not fully cover a first-deploy or cache-miss GET timeout.

## What This Changes

The issue reports that Discord native slash-command deployment timed out on the REST GET of application commands in OpenClaw 2026.5.7.

## Best Possible Solution

Clarify whether this happens on first deploy/cache miss or repeated restarts, then either land the Discord deploy-policy work for redundant deploys or add targeted timeout/retry/diagnostic handling around `listApplicationCommands`.

## Reproduction Assessment

Yes at source level, not as a live Discord reproduction. Current main calls `deployCommands({ mode: "reconcile" })`, reaches `getCommands()` on cache miss, and uses a 15s REST abort, which matches the reported timeout shape.

## Solution Assessment

Unclear. Closing this as covered by the existing deploy-policy issue would be too broad because that work targets redundant deploys and policy controls, while this report could also be a first-cache-miss REST timeout needing retry, timeout, or diagnostic changes.
