## Summary

Keep open. Current main still has a source path that can emit the reported Discord deploy GET timeout when `/applications/{appId}/commands` exceeds the fixed 15s REST timeout; related deploy-policy work is adjacent but not a proven duplicate or shipped fix for this latest 2026.5.7 report.

## What This Changes

The issue asks for Discord native slash-command deployment to succeed instead of timing out on the startup REST GET in OpenClaw 2026.5.7 Docker.

## Best Possible Solution

Make Discord command deployment resilient to slow REST reads by avoiding unnecessary deploy checks, keeping deploy fingerprints scoped correctly, and adding focused retry/timeout/backoff diagnostics for the GET path without changing unrelated channel behavior.

## Reproduction Assessment

Partial/source-level yes. A slow Discord `GET /applications/{appId}/commands` exceeding 15s follows current main's fixed REST timeout into the exact deploy error log, but the real Docker/network condition was not reproduced live.

## Solution Assessment

Unclear. The adjacent deploy-policy issue and PR are likely relevant, but maintainers should decide whether this needs a separate REST timeout/retry fix, improved diagnostics, or should be folded into that deploy-policy work.
