## Summary

Keep open: current main and the latest release still have the source path that emits the doctor `--fix` trailer on a clean non-repair run, and an open linked PR already closes this issue but is currently conflicting.

## What This Changes

The issue reports that clean `openclaw doctor --non-interactive` runs print a non-actionable `openclaw doctor --fix` trailer even when there are no pending config changes.

## Best Possible Solution

Land one canonical doctor-flow fix so clean no-write doctor runs end at `Doctor complete.` while real pending repairs still show actionable guidance.

## Reproduction Assessment

Yes, at source level: current main's settled no-write doctor path logs the trailer solely because `--fix` was not requested. The issue also includes concrete shipped-output evidence from a clean rerun.

## Solution Assessment

Yes, a focused doctor-flow fix is the right shape. The maintainable solution is either to thread `pendingChanges` to `runWriteConfigHealth` as the open PR does, or to remove the duplicate trailer and keep fix hints centralized in `finalizeDoctorConfigFlow`.
