## What This Changes

The issue asks for `openclaw doctor` to print the `--fix` trailer only when there are real pending config changes to apply.

## Best Possible Solution

Review and land the narrow doctor config-flow fix that threads real pending-change state into the write health contribution and adds regression coverage for clean, pending, and `--fix` paths.

Yes. Source inspection on current `main` shows a clean no-write doctor run can reach `runWriteConfigHealth` with `shouldRepair: false`, which still emits the trailer without checking whether any repair is pending.

## Solution Assessment

Yes. The best fix is the narrow existing direction from the linked PR: reuse the doctor config flow's pending-change state to gate the generic trailer, with focused tests for both clean and pending runs.
