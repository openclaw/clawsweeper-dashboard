## Summary

This PR addresses a real duplicate plugin-load path and should stay open for normal review/repair rather than cleanup closure; the remaining issues are narrow PR hygiene and cache-key robustness, not a reason to close it.

## What This Changes

The PR caches plugin CLI registration entries on the Commander program by `primary` and adds tests for repeat registration and primary-change reload behavior.

## Best Possible Solution

Land the registration-layer cache after adding the required changelog entry and tightening or explicitly scoping the cache identity so direct helper calls cannot reuse entries for changed load inputs.

## Reproduction Assessment

Yes. Current main shows each registration call reaches the uncached loader, and the PR's added same-program repeat-call unit test is a focused reproduction of the duplicate-load contract.

## Solution Assessment

Yes, the registration-layer cache is the narrowest fix for the reported `completion --write-state` slowdown because the lower registry load intentionally uses `cache: false`; the safer final shape is to include all load-context inputs in the cache identity or restrict the cache to the validated-config wrapper.
