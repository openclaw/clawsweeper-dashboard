## Summary

Current main still reloads plugin CLI entries on every registration call, and this PR targets the right layer for the reported completion-cache slowdown. I would keep it open; the only concrete review blocker I found is the required changelog entry for a user-facing CLI fix.

## What This Changes

The PR caches plugin CLI registration entries on each Commander program instance by `primary` and adds unit coverage for repeat registration and primary-change reload behavior.

## Best Possible Solution

Land this narrow registration-layer cache after adding the missing changelog entry and letting CI or the targeted plugin CLI tests validate the exact PR head.

## Reproduction Assessment

Yes. On current main, the source path shows each registration call reaches `loadPluginCliRegistrationEntriesWithDefaults`, while completion builds subcommands and then calls plugin registration again; the PR’s proposed unit test is a high-confidence reproduction of the duplicate load contract.

## Solution Assessment

Yes, for the reported `completion --write-state` slowdown, caching loaded registration entries on the Commander program is narrower than changing the lower-level plugin registry cache that intentionally uses `cache: false`. A conservative follow-up would include config/env/loader options in the cache identity only if maintainers want the exported helper to support same-program re-registration with different inputs.
