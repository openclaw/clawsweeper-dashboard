## Label Justifications

- `P1`: The PR targets an urgent package-update regression that can leave existing users stuck during beta package upgrade handoff.
- `impact:security`: The diff changes trusted ClawHub-to-npm fallback behavior for plugin packages, which is a package-source security boundary.
- `impact:crash-loop`: The affected update path can leave the managed gateway/update flow unable to complete and restart cleanly.
- `merge-risk: 🚨 compatibility`: Merging with the remaining metadata gap can break legacy configs during upgrade when an old parent resumes after candidate doctor writes.
- `merge-risk: 🚨 security-boundary`: The PR changes package artifact fallback rules and official npm fallback gating, which CI alone does not fully settle.
- `merge-risk: 🚨 availability`: The changed package update and gateway restart path can make installed runtimes fail to update or restart if the handoff is wrong.

## Summary

Keep this PR open, but it is not merge-ready: the update-handoff direction is useful and now has real upgrade proof, yet the branch still future-stamps configs that never had `meta.lastTouchedVersion`, which can make the older parent reject the config when it resumes.

## What This Changes

The PR changes update/doctor handoff metadata handling, ClawHub-to-npm fallback gating, managed package-root selection, Crabbox wrapper option parsing, regression tests, and the changelog.

## Best Possible Solution

Preserve old-parent-readable metadata for both existing and absent `lastTouchedVersion` values during candidate doctor writes, keep the official-only npm fallback and managed-root handling, then rerun focused upgrade proof on the final head.

## Reproduction Assessment

Yes, source-level with strong external proof: the PR and companion discussion show the beta.4 package-upgrade handoff failure, and the current branch still has a clear source path for the no-metadata variant. I did not rerun the Docker upgrade lane in this read-only review.

## Solution Assessment

No, not yet. The overall approach is the right owner-boundary fix, but it must preserve missing legacy metadata explicitly before the package handoff can be considered safe.
