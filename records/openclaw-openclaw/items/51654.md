## Summary

Keep open. Current main still lacks a public `sessions_spawn` env parameter and does not thread session env through ACP spawn/manager initialization, while the requested credential-bearing behavior needs explicit maintainer policy before implementation.

## What This Changes

Request adding an optional per-spawn env map for ACP `sessions_spawn` sessions that merges with or overrides the ACP child process environment.

## Best Possible Solution

Define a documented ACP-only env contract, validate string keys and values, redact values from persisted/log-visible tool inputs, then thread the sanitized map through tool, spawn, manager, and runtime initialization.

## Reproduction Assessment

Yes. Static inspection of current main shows the requested env field is absent from the public tool schema, tool execution, ACP spawn params, and manager initialization, while the lower runtime type already has an env seam.

## Solution Assessment

Unclear until maintainer policy is settled. The narrow code seam is `AcpRuntimeEnsureInput.env`, but the maintainable fix must first decide credential redaction and whether explicit session env may override stripped provider auth variables.

## Security Review

Status: needs_attention

Summary: The requested feature would carry credentials into ACP child processes and needs explicit redaction plus provider-env override policy before implementation.

  - confidence: 0.86

  - body: ACP client startup currently strips known provider auth variables for protected paths. Session-level env support needs an explicit rule for whether caller-supplied env can reintroduce or override those stripped variables.
  - confidence: 0.82
