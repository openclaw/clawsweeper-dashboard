## Summary

Keep open: current main still lacks a public `sessions_spawn.env` contract and does not forward per-session env through the ACP spawn, manager, and acpx path. The request is plausible, but raw credential-bearing env passthrough needs maintainer product/security policy before implementation.

## What This Changes

The issue requests an optional per-call environment map for `sessions_spawn({ runtime: "acp" })` so ACP harness subprocesses can run with session-specific provider keys and endpoint configuration.

## Best Possible Solution

Define an explicit ACP-only session env or SecretRef-aware harness env contract with validation, redaction, merge/override rules, persistent-session reuse semantics, docs, tests, and acpx compatibility proof.

## Risks / Open Questions

- Credential-bearing session env values could leak into transcripts, persisted tool-call inputs, task logs, stream logs, or diagnostics unless redaction is designed and tested with the new contract.
- Caller-supplied env may conflict with existing provider-auth stripping, skill env stripping, and ACPX_AUTH-style conventions; maintainers need to define precedence and allow/deny behavior.
- Persistent ACP session reuse must account for env identity or explicitly reject env changes for reused sessions, otherwise two requested env contexts could share one harness session.
- The docs already state ACP runs on the host runtime rather than inside the OpenClaw sandbox, so this feature must not be presented as a hostile multi-tenant isolation boundary.
