- `impact:security`: The requested behavior is explicitly about passing API keys and endpoint env vars into spawned ACP processes.

## Summary

Keep this open: current main and the latest checked release still do not expose a `sessions_spawn.env` parameter or pass per-session environment variables into ACP initialization. The request is useful but security/product-sensitive because it would let a model-facing tool inject provider credentials or endpoint overrides into host ACP harness subprocesses.

## What This Changes

The issue requests an optional per-call `env` map for `sessions_spawn({ runtime: "acp" })` that would be merged into ACP/acpx subprocess environment for that session.

## Best Possible Solution

Design a deliberate ACP session environment override contract that validates/redacts sensitive keys, preserves current credential-stripping safeguards by default, and forwards approved values through `sessions_spawn` -> `spawnAcpDirect` -> ACP manager -> backend with focused tests and docs.

## Reproduction Assessment

Yes, source-level only: current main and `v2026.6.6` do not expose `sessions_spawn.env`, and ACP manager initialization does not forward env into `runtime.ensureSession`. I did not run a live ACP spawn because the missing surface is visible in the schema and call chain.

## Solution Assessment

No for the raw requested shape as written: a direct `Record<string,string>` passthrough from a model-facing tool is not clearly the safest product boundary. The better path is an explicit, reviewed environment override contract with redaction, validation, backend support, and tests for credential isolation.
