## Summary

Keep this open: current main and the latest release still do not expose a `sessions_spawn.env` ACP parameter, and the requested raw env passthrough remains product/security-sensitive because it would let a model-facing tool inject provider credentials and endpoint overrides into host ACP harness processes.

## What This Changes

Request support for per-spawn/session environment variable overrides when creating ACP sessions through `sessions_spawn`.

## Best Possible Solution

Design a credential-safe ACP session environment override contract, then wire approved values through `sessions_spawn` -> `spawnAcpDirect` -> ACP manager -> backend with tests, docs, and redaction behavior.

## Reproduction Assessment

Yes, source-level: current main and `v2026.6.6` do not expose `sessions_spawn.env`, and ACP manager initialization does not forward env into `runtime.ensureSession`. A live ACP spawn is not necessary to prove the missing surface.

## Solution Assessment

No for the raw requested shape as written: direct `Record<string,string>` passthrough from a model-facing tool is not the safest boundary. The better path is an explicit environment override contract with validation, redaction, backend support, and credential isolation tests.
