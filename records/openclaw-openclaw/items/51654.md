## Label Justifications

- `P2`: This is useful normal-priority ACP feature work with limited blast radius, but it needs maintainer policy because it can carry credentials.
- `impact:security`: The requested env map can contain API keys, endpoints, and OAuth-style secrets that cross process and persistence boundaries.
- `impact:auth-provider`: The primary use cases are per-session API keys, provider endpoints, model/provider testing, and harness auth routing.
- `impact:session-state`: The request changes what environment context is associated with each spawned ACP session.

## Summary

Keep open. Current main still has no public `sessions_spawn.env` contract or forwarding path into ACP session creation, and the remaining work is security-sensitive credential/environment policy rather than conservative cleanup.

## What This Changes

Request adding an optional env map to `sessions_spawn` for ACP sessions so per-spawn variables can be merged or overridden in the acpx subprocess.

## Best Possible Solution

Define an explicit ACP-only `sessions_spawn.env` or SecretRef-aware harness env contract with validation, redaction, merge/override rules, tests, docs, and acpx compatibility proof before forwarding it to spawned harnesses.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. The feature gap is high-confidence from source inspection: the public tool and ACP spawn/manager path omit env while only a lower OpenClaw runtime type has an env seam.

## Solution Assessment

Unclear until maintainers define the credential boundary. Passing raw env through the ACP pipeline is plausible, but the safer maintainable direction is an explicit env or SecretRef contract with redaction and precedence rules.
