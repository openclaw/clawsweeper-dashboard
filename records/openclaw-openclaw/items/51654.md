## Label Justifications

- `P2`: This is a concrete ACP/session capability request with security and auth implications, but it is not an urgent regression or outage.
- `impact:security`: The requested env injection can pass secrets and host process settings into ACP subprocesses.
- `impact:auth-provider`: The motivating examples are per-session API keys, base URLs, and provider selection/auth behavior.
- `impact:session-state`: The feature needs defined behavior for persistent ACP session reuse when environment state differs between spawns.

## Summary

Keep open: current main and the latest release still do not expose a public `sessions_spawn.env` parameter or forward env through the ACP spawn manager into acpx. The remaining work is security-sensitive because ACP child-process env inheritance has been intentionally restricted before, so this needs maintainer product/security review before implementation.

## What This Changes

Request to let ACP `sessions_spawn` calls provide per-session environment variables that override or merge into the spawned acpx subprocess environment.

## Best Possible Solution

Define a narrow, explicit ACP session env contract that validates/redacts credential inputs, forwards them through the manager and acpx runtime, and defines whether env changes force fresh persistent sessions rather than silently reusing prior state.

## Reproduction Assessment

Yes. Source inspection shows `sessions_spawn` has no `env` parameter and the ACP manager does not forward env to `runtime.ensureSession`; I did not run a live ACP spawn because this was a read-only review.

## Solution Assessment

No direct implementation shape is settled yet. The narrow maintainable solution needs an explicit security/product decision for env validation, redaction, merge/override rules, and persistent-session reuse before a fix PR is safe.

## Risks / Open Questions

- A generic env map can carry API keys, endpoint overrides, proxy settings, and auth-provider state into host-side ACP subprocesses, so the allowlist/redaction/SecretRef policy needs maintainer approval.
- Persistent ACP sessions can be reused by command, cwd, and resume identity today; env-sensitive reuse semantics are not defined, so a later spawn with different credentials could otherwise attach to the wrong runtime state.
- The existing lower `AcpRuntimeEnsureInput.env` seam is not enough by itself because public tool schema, docs, storage/metadata, manager forwarding, acpx subprocess behavior, tests, and log redaction all need to agree.
