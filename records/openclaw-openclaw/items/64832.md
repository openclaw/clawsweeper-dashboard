## Summary

Keep this PR open. Current main still lacks the requested reset-archive path for rotated isolated cron and heartbeat transcripts, and the PR discussion includes useful production evidence plus the related #65203 fresh-transcript fix; however, this branch is not mergeable as-is because it conflicts with current main's stable cron-session persistence and its reset archive guard can still skip legacy run-history rows.

## What This Changes

The PR adds archive-on-rotation helpers and tests so prior isolated cron and heartbeat transcripts are renamed into `.reset` archives, while separating heartbeat suffix-collapse deletion archival from reset archival.

## Best Possible Solution

Keep the shipped fresh-session behavior, then land a narrow archive-on-rotation implementation that captures the prior entry before overwrite, persists only current main's stable cron session state, archives the prior transcript with `reason: "reset"` and `restrictToStoreDir: true`, and ignores legacy cron run-history rows when deciding whether the just-rotated transcript is still actively referenced.

## Reproduction Assessment

Yes. A high-confidence reproduction for the PR blocker is a session store with a prior stable cron session entry plus an older `...:run:<id>` row that still references the prior `sessionId`; the PR's all-rows reference set makes `archiveRemovedSessionTranscripts` skip the reset rename.

## Solution Assessment

No. The archive-on-rotation direction is right, but the current PR is not the best mergeable solution until it rebases onto current main's stable-key persistence and narrows the reset reference set so historical run rows do not block orphan archival.

Build the reset reference set from active rows or otherwise exclude run-history keys for this rotation archive path.
  - confidence: 0.91
