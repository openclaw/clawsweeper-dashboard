## Summary

Keep open. Current main has the #65203 fresh-transcript behavior but still lacks the orphaned-transcript reset archival this PR targets; the PR is useful, but its reset archive helper can skip the common cron run-key case because it treats historical `...:run:<id>` rows as active references.

## What This Changes

The PR adds helpers and cron/heartbeat wiring with tests to archive prior isolated-session transcript files as `.reset` archives after rotation.

## Best Possible Solution

Land a narrow archive-on-rotation implementation that captures the prior entry before overwrite, archives it after persistence with `reason: "reset"` and `restrictToStoreDir: true`, and excludes historical cron run rows from the reset-archive reference set.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is a store with the prior stable cron session entry plus an older `...:run:<id>` row that still references the same prior `sessionId`; the PR helper includes that row in `referencedSessionIds`, so the archive primitive skips the reset rename.

## Solution Assessment

No. The direction is right, but the patch is not the best mergeable solution until reset archival computes active references only or otherwise ignores historical cron run rows while leaving the generic archive guard intact.

## Review Findings

Overall correctness: patch is incorrect
