## Summary

This PR is still addressing a real outbound-delivery bug, but it is not merge-ready: the new fallback uses the sanitized target while the plan still treats the request as explicit, which can drop session account/thread context and leave the delivery boundary unsafe.

## What This Changes

The PR adds reserved literal-target handling for `current`, `self`, `this`, and `me` in session target resolution and changes agent delivery planning to use the sanitized resolver target.

## Best Possible Solution

Make reserved strings a single delivery-plan decision: use the sanitized resolver result for target, mode, account, and thread, fail clearly when no bound target exists, and prove the downstream path with real gateway/Telegram or equivalent imported-runtime output.

## Reproduction Assessment

Yes for the source-level path: current main accepts raw `explicitTo` strings, and the PR shows the reserved-target path through session and agent delivery planning. I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

No. The resolver guard is a useful direction, but the plan should consume the sanitized `baseDelivery` mode/account/thread consistently instead of only replacing `resolvedTo`.
