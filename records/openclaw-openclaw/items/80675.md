## Summary

Keep open: the intended timeout fix is plausible, but the patch reintroduces the partial-assistant-fragment case that current tests intentionally suppress, and the supplied proof does not show the actual surfaced UI/error payload.

## What This Changes

The PR changes the embedded PI agent prompt-timeout branch to always return an explicit timeout error payload and prepend prior payloads before that error.

## Best Possible Solution

Preserve prior tool/media output on prompt timeout while continuing to suppress incomplete assistant fragments, then add focused runner coverage for both cases.

## Reproduction Assessment

Yes for the introduced regression: source inspection plus existing test coverage show that a prompt timeout with partial assistant text must not surface that fragment. I did not run the test because this review was read-only.

## Solution Assessment

No. The direction is close, but the implementation should filter partial assistant payloads instead of prepending every payload, and it should include a regression test for tool/media payloads plus timeout.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Filter partial fragments before appending timeout:** `src/agents/pi-embedded-runner/run.ts:2538`
  - body: The new timeout branch prepends every `payloadsWithToolMedia` entry before the timeout error. Since `buildEmbeddedRunPayloads()` turns non-empty `assistantTexts` into payloads, a stream timeout with only a partial assistant fragment will surface that incomplete text again, contradicting the existing regression test that requires only the timeout error. Preserve the successful tool/media payloads without re-emitting partial assistant text.
  - confidence: 0.94
