## Summary

Keep open: the PR targets a real prompt-timeout ghost-turn bug, but the current patch would also re-surface incomplete assistant fragments and the supplied proof does not show the changed UI or returned timeout payload.

## What This Changes

The PR changes `src/agents/pi-embedded-runner/run.ts` so prompt timeouts always return an explicit timeout error payload while prepending prior attempt payloads.

## Best Possible Solution

Preserve successful tool/media output on prompt timeout, append the explicit timeout error, continue suppressing incomplete assistant fragments, and add focused runner coverage for that combined case.

## Reproduction Assessment

Yes. Source inspection shows current main skips the timeout-error branch when a prompt timeout already has tool/media payloads, and the PR-introduced partial-fragment regression is covered by an existing runner test; no tests were run because this review is read-only.

## Solution Assessment

No. The direction is right, but the patch should filter partial assistant payloads while preserving successful tool/media payloads, then add regression coverage for that exact timeout shape.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Filter partial fragments before appending timeout:** `src/agents/pi-embedded-runner/run.ts:2538`
  - body: The new timeout branch prepends every `payloadsWithToolMedia` entry before the timeout error. Since `buildEmbeddedRunPayloads()` turns non-empty `assistantTexts` into payloads, a stream timeout with only a partial assistant fragment will surface that fragment again and violate the existing partial-fragment regression test. Preserve tool/media payloads without re-emitting assistant text fragments.
  - confidence: 0.94
