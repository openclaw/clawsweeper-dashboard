## Summary

Keep open: the PR addresses the right browser safety gap, but the new pre-dispatch helper still lets malformed credential-like HTTP(S) URLs cross the local/proxy request boundary, and the external PR proof is test-only rather than real browser behavior.

## What This Changes

The PR adds browser navigation userinfo rejection, invalid-URL credential redaction, and focused tests for browser tool and Playwright navigation guard paths.

## Best Possible Solution

Keep the shared guard approach, but make the browser-tool pre-dispatch path reject malformed credential-like HTTP(S) strings too, then add real runtime proof with secrets redacted.

## Reproduction Assessment

Yes. Current main source shows credential-bearing browser navigation URLs are not checked before dispatch, and the PR helper’s malformed-URL gap is reproducible from the added helper logic.

## Solution Assessment

No, not as-is. The shared guard is the right fix point, but the pre-dispatch helper should not swallow malformed credential-like HTTP(S) inputs, and external proof should exercise a real browser path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Reject malformed credential URLs before dispatch:** `extensions/browser/src/browser/navigation-url-userinfo.ts:18-21`
  - body: The new helper only throws for URLs that `new URL()` can parse. For malformed secret-bearing strings such as `https://user:secret@`, it catches the parse error and returns, so `readTargetUrlParam()` still lets `open`/`navigate` send the raw value to `browserOpenTab` or `callGatewayTool` before the shared guard can produce a redacted error. Reject credential-like `http(s)://...@` inputs in this pre-dispatch path too.
  - confidence: 0.84
