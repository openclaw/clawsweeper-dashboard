## Summary

Keep open. Current main has partial operator-managed HTTP proxy support and several surface-specific proxy paths, but the requested first-class system/custom HTTP plus SOCKS model remains incomplete across top-level config, web/browser overrides, and reviewed bypass semantics.

## What This Changes

Request to add first-class system/custom HTTP and SOCKS proxy configuration, including top-level config, environment fallback, per-agent/per-tool overrides, and runtime proxy injection.

## Best Possible Solution

Keep this as the proxy umbrella and split reviewed follow-ups for SOCKS support, web/provider/browser override surfaces, per-agent environment policy, and bypass semantics while preserving SSRF guardrails.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the missing capability from source inspection: current main rejects top-level SOCKS proxy URLs, lacks web/browser explicit proxy config fields, and intentionally guards explicit browser proxy routing under strict SSRF policy.

## Solution Assessment

Is this the best way to solve the issue? No as a single broad automated patch; the maintainable path is staged, reviewed core and plugin work with explicit SSRF, egress, browser routing, credential, and bypass policy decisions.
