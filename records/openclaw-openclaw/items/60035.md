## Summary

Keep open. Current main has partial process-wide HTTP forward-proxy and env-proxy support, but the broader request still lacks first-class top-level SOCKS support, explicit user-facing noProxy semantics, per-agent/per-web-tool overrides, and a browser proxy design that preserves SSRF guardrails; open PR #63633 is a narrower Telegram media SOCKS fix and does not supersede this umbrella request.

## What This Changes

The issue requests first-class system/custom HTTP and SOCKS proxy support with top-level config, standard env fallback, per-agent and per-tool overrides, and runtime wiring for web, browser, plugin, and provider network calls.

## Best Possible Solution

Keep this as the umbrella and split narrower design/implementation tasks for SOCKS, noProxy, web/provider/agent overrides, and browser policy while preserving the current managed-proxy and SSRF guarantees.

## Reproduction Assessment

Not applicable as a runtime bug, but yes for verifying the missing capability: current source and tests show top-level socks5:// proxyUrl is rejected, web tool config lacks proxy overrides, and browser proxy routing is constrained by strict SSRF policy.

## Solution Assessment

No, this is not best handled as one broad automated change. The maintainable path is a maintainer-scoped design that extends proxy support in small pieces without weakening current egress and SSRF boundaries.
