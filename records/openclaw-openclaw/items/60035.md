## Summary

Keep open. Current main and v2026.6.6 cover useful HTTP/HTTPS managed proxy and provider-specific proxy paths, but the requested first-class SOCKS support, custom noProxy/bypass config, per-agent/tool overrides, and browser-profile proxy contract remain broader than the shipped surface.

## What This Changes

The issue requests first-class system/custom HTTP and SOCKS proxy configuration across CLI/config, environment fallback, per-agent and per-tool/browser overrides, and runtime network agent wiring.

## Best Possible Solution

Keep the shipped HTTP/HTTPS managed proxy as the baseline, then split a maintainer-approved proxy design into explicit surfaces for SOCKS, env precedence, bypass policy, browser egress, per-agent/tool overrides, and credential handling.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes, as a source-level missing-surface check: current main rejects SOCKS for top-level managed proxy config and lacks first-class per-agent, per-tool, and browser-profile proxy fields.

## Solution Assessment

Is this the best way to solve the issue? No, not as one broad patch; the maintainable path is a staged product/security design that preserves existing managed proxy and provider proxy behavior while adding narrow, tested contracts.
