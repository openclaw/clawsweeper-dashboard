## Summary

Keep open. Current main has partial process-wide HTTP forward-proxy and env-proxy support, but the broader request still has unresolved SOCKS top-level config, explicit noProxy semantics, per-agent/per-web-tool overrides, and a browser proxy UX that preserves SSRF guarantees. Open PR #63633 is a narrow Telegram SOCKS fix for #60472 and does not supersede this broader feature request.

## What This Changes

The issue requests first-class system and custom HTTP/SOCKS proxy configuration in OpenClaw, including standard proxy environment fallback plus per-agent, per-tool, and browser override support.

## Best Possible Solution

Keep this issue open as the umbrella until maintainers split or design the remaining proxy work: retain the shipped process-wide HTTP forward-proxy baseline, decide explicit SOCKS/noProxy semantics, add narrow web/provider/agent override surfaces where maintainable, and keep browser proxy routing tied to explicit SSRF-safe policy.

## Reproduction Assessment

Not applicable as a runtime bug, but yes for verifying the missing capability: current source and tests show top-level socks5:// proxyUrl is rejected, web tool config lacks proxy overrides, and browser docs/code intentionally avoid automatic env-proxy routing.

## Solution Assessment

No; the current implementation is a useful partial baseline, not a complete solution to this issue. The safer solution is a maintainer-scoped design that extends proxy support without bypassing SSRF and browser navigation protections.

## Security Review

Status: needs_attention

Summary: The issue is security-sensitive because proxy and noProxy overrides alter outbound egress routing and can affect SSRF enforcement boundaries.

Concerns:

- **[medium] Design proxy overrides around SSRF enforcement:** `extensions/browser/src/browser/navigation-guard.ts:117`
  - body: Browser proxy routing hides the final connect target from the Node process, and current code blocks explicit browser proxy routing under strict SSRF policy unless private-network access is intentionally enabled. Any SOCKS, noProxy, per-agent, or per-tool design should preserve that boundary.
  - confidence: 0.88
- **[medium] Avoid destination bypass lists weakening managed proxy mode:** `docs/security/network-proxy.md:50`
  - body: The managed proxy docs say OpenClaw clears no_proxy, NO_PROXY, and GLOBAL_AGENT_NO_PROXY while active because destination-based bypasses can skip the filtering proxy. Adding user-facing noProxy semantics needs a deliberate SSRF policy model.
  - confidence: 0.84
