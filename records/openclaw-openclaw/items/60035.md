## Summary

Keep open. Current main has partial process-wide HTTP forward-proxy and env-proxy support plus some provider request overrides, but the broader request still lacks first-class top-level SOCKS support, explicit user-facing noProxy semantics, per-agent/per-web-tool overrides, and a browser proxy design that preserves SSRF guardrails; open PR #63633 is a narrower Telegram media SOCKS fix for #60472 and does not supersede this umbrella request.

## What This Changes

Request first-class system/custom HTTP and SOCKS proxy support, including config/env fallback and per-agent, web-tool, and browser overrides.

## Best Possible Solution

Keep this as the umbrella and split maintainer-scoped tasks for top-level SOCKS, noProxy semantics, agent/tool/provider overrides, and browser policy while preserving the managed proxy and SSRF guarantees.

## Reproduction Assessment

Not applicable as a single runtime bug, but yes for verifying the missing capability. Current source and tests show top-level socks5:// proxyUrl is rejected, web tool schemas lack proxy overrides, and browser proxy routing is blocked under strict SSRF policy.

## Solution Assessment

No, this is not best solved as one broad automated change. The maintainable path is a design-reviewed sequence of smaller proxy extensions with explicit SSRF, noProxy, browser, and plugin-contract boundaries.
