## Summary

Keep open. Current main has partial process-wide HTTP/env proxy and provider-request proxy support, but it still rejects top-level SOCKS proxy URLs, lacks first-class web tool/per-agent/browser proxy override design, and keeps explicit noProxy-style bypasses security-sensitive rather than generally configurable.

## What This Changes

Add first-class system/custom HTTP and SOCKS proxy configuration with env fallback, per-agent/per-tool/browser overrides, and runtime proxy injection.

## Best Possible Solution

Keep this as the umbrella, then split maintainer-scoped work for top-level SOCKS/noProxy semantics, provider and web-tool overrides, per-agent environment policy, and browser proxy routing while preserving current SSRF guardrails.

## Reproduction Assessment

Yes for verifying the missing capability, though this is a feature request rather than a runtime bug. Current schema/tests reject top-level socks5:// proxyUrl, web tool types expose no proxy override, and browser navigation explicitly blocks proxy-routed strict SSRF mode.

## Solution Assessment

No, this should not be solved as one broad automated patch. The maintainable path is a staged proxy design with security review for bypass semantics and browser egress behavior.
