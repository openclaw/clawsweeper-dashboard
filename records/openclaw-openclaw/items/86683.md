## Summary

This is a focused regression-fix PR with good package-level proof, but the new ancestor fallback currently depends on an explicit `argv1` parameter while the production lazy runtime call does not pass one, so the branch can still miss the package-root-unavailable path it is meant to repair.

## What This Changes

The PR adds plugin runtime module resolution diagnostics, ancestor runtime fallback candidates, plugin/runtime context on lazy resolution failures, and targeted SDK-alias tests.

## Best Possible Solution

Keep the diagnostic improvements, but make the ancestor fallback cover the production no-arg lazy runtime path and add a regression test for that default call shape before merge.

## Reproduction Assessment

Yes for the PR defect from source inspection: the branch's production lazy runtime call passes no `argv1`, while the new fallback only scans explicit `params.argv1`. I did not run a live packaged repro in this read-only review.

## Solution Assessment

No, not yet. The resolver is the right place for the fix, but it should use the default startup argv hint or pass that hint from the loader so the real packaged startup path exercises the fallback.
