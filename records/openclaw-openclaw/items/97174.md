## Summary

Keep open: current main still lacks registry-backed recovery for plugin interactive handlers, and this PR is a narrow plausible fix, but native Telegram callback behavior still needs redacted real-setup proof before merge.

## What This Changes

This PR stores plugin interactive handlers on PluginRegistry, restores live registry handlers before callback fallback, snapshots that state for loader rollback, and adds Telegram/core regression tests plus docs wording.

## Best Possible Solution

Land the registry-backed recovery after redacted Telegram proof shows code-agent inline callbacks route to the plugin handler, clear or update the buttons, and do not send fallback callback_data text.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not with a live Telegram run yet; source inspection shows the current-main path where an unmatched process-global handler falls through to Telegram callback text, and the PR adds targeted regression tests for that path.

## Solution Assessment

Is this the best way to solve the issue? Yes for the code shape: registry-backed recovery with loader rollback symmetry is the narrow owner-boundary fix; Telegram-specific hardcoding or a new config path would be worse.
