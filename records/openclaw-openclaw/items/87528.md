## Summary

Keep this PR open, but not merge-ready: it fixes a real assistant-media filename problem, yet the implementation applies `attachment` to a mixed preview/download endpoint and can regress existing inline media previews.

## What This Changes

The PR adds a basename-derived `Content-Disposition: attachment` header with `filename` and `filename*` to Control UI assistant-media responses.

## Best Possible Solution

Make assistant-media disposition MIME-aware: add a safe attachment filename for document-like downloads while preserving inline behavior for image/audio/video previews, with focused gateway tests and browser proof.

## Reproduction Assessment

Yes, at source level. Current main omits `Content-Disposition` on assistant-media streams, and WebChat document links use that route, but I did not run a live browser save-dialog repro in this read-only review.

## Solution Assessment

No. The proposed patch fixes the filename symptom by forcing `attachment` on the whole route, but the maintainable fix should target document/download responses and preserve inline behavior for previewable media.

Full review comments:

- **[P1] Keep Previewable Assistant Media Inline:** `src/gateway/control-ui.ts:608-610`
  - body: This route also feeds WebChat `<img>`, `<audio>`, and `<video>` previews. Adding `Content-Disposition: attachment` to every successful response can make existing previews download instead of render, so gate attachment behavior to document-like responses or preserve inline handling for previewable media.
  - confidence: 0.9
- **[P2] Encode `filename*` Value Characters:** `src/gateway/control-ui.ts:610`
  - body: `filename*` uses RFC 5987 extended-value syntax, but `encodeURIComponent` leaves characters such as apostrophes, parentheses, and asterisks unescaped. Filenames containing those characters can produce an invalid extended parameter, so use stricter encoding or the helper shape from the related broader PR.
  - confidence: 0.82
