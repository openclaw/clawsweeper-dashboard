## Summary

Keep open: the underlying webchat download filename bug is real on current main and the latest release, but this PR is not merge-ready because it applies `attachment` to a shared preview/download route and still lacks browser-level proof for the download and preview behavior.

## What This Changes

The PR adds a basename-derived `Content-Disposition: attachment` header to successful Control UI assistant-media responses.

## Best Possible Solution

Gate `Content-Disposition` to document or non-inline MIME responses, use a safe RFC 5987/6266 header builder with an ASCII fallback, add focused gateway/UI tests, and require redacted browser proof.

## Reproduction Assessment

Yes at source level: current main and `v2026.6.8` serve assistant-media without `Content-Disposition` while document links use that route. I did not run a live browser save-dialog reproduction in this read-only review.

## Solution Assessment

No as submitted. The patch fixes the filename symptom by forcing `attachment` on the whole route, but the best fix should target document/non-inline responses and preserve browser-native previews for image, audio, and video.
