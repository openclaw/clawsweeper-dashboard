## Summary

Keep open: current main and the latest release still lack the requested assistant-media filename hint, but this PR is not merge-ready because it forces `attachment` on a route also used for inline media and the supplied proof only verifies response headers.

## What This Changes

The PR adds a basename-derived `Content-Disposition: attachment` header with `filename` and `filename*` parameters to every successful Control UI assistant-media response.

## Best Possible Solution

Gate `Content-Disposition` to document or non-inline MIME responses, build the RFC 5987/6266 header with a safe fallback, add focused gateway/UI tests, and require redacted browser proof.

## Reproduction Assessment

Yes at source level: current main and `v2026.6.8` serve assistant-media without `Content-Disposition` while document links use that route. I did not run a live browser save-dialog reproduction in this read-only review.

## Solution Assessment

No as submitted. The patch fixes the filename symptom by forcing `attachment` on the whole route, but the maintainable fix should target document/non-inline responses and preserve image/audio/video previews.
