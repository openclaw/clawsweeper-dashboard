## Summary

Keep this PR open because the reported webchat download-name bug still exists on current main and the latest release, but the submitted patch is not merge-ready: it forces `attachment` on the shared assistant-media route used by inline previews, has filename* encoding gaps, and the supplied terminal proof does not show the browser save-dialog result or unchanged preview behavior.

## What This Changes

This PR adds a basename-derived `Content-Disposition: attachment` header with `filename` and `filename*` to every successful Control UI assistant-media response.

## Best Possible Solution

Use targeted non-inline disposition and a standards-safe filename helper while preserving image/audio/video previews, with focused gateway/UI tests and browser proof.

## Reproduction Assessment

Yes, at source level. Current main and `v2026.6.6` serve assistant-media without `Content-Disposition` and render document links through that route, but I did not run a live browser save-dialog repro in this read-only review.

## Solution Assessment

No. The patch fixes the filename symptom by forcing `attachment` on the whole route, but the best fix should target document/non-inline responses and preserve browser-native previews for image, audio, and video.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Keep previewable media inline:** `src/gateway/control-ui.ts:609-610`
  - body: This endpoint is also used as the `src` for local image, audio, and video previews in webchat. Setting every successful response to `attachment` can make existing previews download instead of rendering; gate this header to document/non-inline responses or another explicit download path.
  - confidence: 0.9
- **[P2] Encode filename* with RFC 5987-safe characters:** `src/gateway/control-ui.ts:610`
  - body: `encodeURIComponent` leaves characters such as single quotes, parentheses, and `*` unescaped, which are not safe in the extended `filename*` value syntax. Use a helper that tightens the percent-encoding and supplies a readable ASCII fallback before interpolating the header.
  - confidence: 0.82
