## Summary

Keep open: current main and the latest release still lack the requested assistant-media download filename header, but this PR is not merge-ready because it adds `attachment` to the whole mixed preview/download route and still needs safer filename encoding plus browser-level proof.

## What This Changes

The PR adds a basename-derived `Content-Disposition: attachment` header to successful `handleControlUiAssistantMediaRequest` responses.

## Best Possible Solution

Gate `Content-Disposition` to document or non-inline MIME responses, use a safe RFC 5987/6266 header builder with an ASCII fallback, add focused gateway/UI coverage, and provide redacted browser proof for downloads plus inline previews.

## Reproduction Assessment

Yes at source level: current main and `v2026.6.9` serve assistant-media without `Content-Disposition` while document links use that route. I did not run a live browser save-dialog reproduction in this read-only review.

## Solution Assessment

No as submitted. The patch fixes the filename symptom by forcing `attachment` on the whole route, but the best fix should target document/non-inline responses and preserve browser-native previews for image, audio, and video.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Keep previewable media inline:** `src/gateway/control-ui.ts:606-610`
  - body: This endpoint is also used as the `src` for local image, audio, and video previews in webchat. Setting every successful response to `attachment` can make existing previews download instead of render; gate this header to document/non-inline responses or another explicit download path.
  - confidence: 0.9
- **[P2] Encode filename* with RFC 5987-safe characters:** `src/gateway/control-ui.ts:610`
  - body: `encodeURIComponent` leaves characters such as apostrophes, parentheses, and `*` unescaped, which are delimiter-sensitive in the extended `filename*` value syntax. Use a helper that tightens percent-encoding and supplies a readable ASCII fallback before interpolating the header.
  - confidence: 0.82
