## Summary

Keep open: the filename problem is still present on current main and the latest release, but this patch is not merge-ready because it forces `attachment` on a mixed endpoint that also backs inline image/audio/video previews and the proof does not show the browser download outcome or preserved previews.

## What This Changes

The PR adds a basename-derived `Content-Disposition: attachment` header with `filename` and `filename*` to every successful Control UI assistant-media response.

## Best Possible Solution

Add targeted document/download filename disposition with RFC 5987-safe encoding while preserving inline image/audio/video responses, and cover both gateway response headers and UI attachment rendering.

## Reproduction Assessment

Yes at source level. Current main and v2026.6.6 lack `Content-Disposition` on assistant-media responses, and the UI routes document links through that endpoint; I did not run a live browser save-dialog repro in this read-only review.

## Solution Assessment

No. The PR is a plausible document filename fix, but it is too broad because the same endpoint serves previewable image/audio/video media and the filename encoding still has RFC 5987 edge cases.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep previewable assistant media inline:** `src/gateway/control-ui.ts:609-610`
  - body: This endpoint is also used as the `src` for image, audio, and video previews in webchat. Setting every successful response to `attachment` can make browsers download media instead of rendering it, so gate the attachment disposition to document/download responses or explicitly preserve inline handling for previewable media.
  - confidence: 0.9
- **[P2] Encode filename* with RFC 5987-safe escaping:** `src/gateway/control-ui.ts:610`
  - body: `filename*` uses extended parameter syntax, but `encodeURIComponent` leaves characters such as `'`, `(`, `)`, and `*` unescaped. Filenames containing those characters can produce an invalid extended value, so tighten the encoding helper before relying on it for browser filenames.
  - confidence: 0.82
