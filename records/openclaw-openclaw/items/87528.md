## Summary

Keep open: the underlying WebChat download bug is still present on current main and the latest release, but this PR is not merge-ready because it forces attachment behavior on a mixed preview/download route and does not prove unchanged preview behavior.

## What This Changes

The PR adds a basename-derived `Content-Disposition: attachment` header to successful Control UI `assistant-media` responses.

## Best Possible Solution

Set safe Content-Disposition at the gateway boundary only for document or otherwise non-inline MIME responses, preserve image/audio/video previews, and add focused gateway/UI coverage plus redacted browser or live-output proof.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.10 stream assistant-media documents without Content-Disposition, while the Control UI uses that route for document links. I did not run a live browser save-dialog reproduction in this read-only review.

## Solution Assessment

No as submitted. The gateway response is the right layer for filename disposition, but the best fix must narrow attachment behavior to document/non-inline MIME responses and preserve previewable media.

- **[P1] Keep previewable media inline:** `src/gateway/control-ui.ts:606-610`
  - body: This new header is set on every successful assistant-media response. The same URL is used by the Control UI as image/audio/video `src` and document `href`, so forcing `attachment` route-wide can turn existing preview/open flows into downloads; classify by MIME or use an explicit download path.
  - confidence: 0.9

  - body: The `filename*` value interpolates raw `encodeURIComponent` output; Node shows it leaves apostrophes, parentheses, and `*` unescaped, which are delimiter-sensitive in extended filename values. Add a helper that encodes those attr chars and covers edge filenames.
  - confidence: 0.82
