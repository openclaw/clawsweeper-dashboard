## Summary

Keep open: the filename bug is real, but this patch is not safe to merge because it forces `attachment` on a route also used for inline image/audio/video previews, and the supplied proof does not show the browser filename outcome or preserved preview behavior. The broader related implementation at https://github.com/openclaw/openclaw/pull/77912 shows the safer direction, but it is still open and conflicting, so it cannot supersede this PR yet.

## What This Changes

The PR adds a basename-derived `Content-Disposition: attachment` header with `filename` and `filename*` to every successful Control UI assistant-media response.

## Best Possible Solution

Preserve inline media previews while adding targeted document/download filename behavior with stricter `filename*` encoding and focused gateway/UI coverage.

Yes at source level. Current main lacks `Content-Disposition` on the assistant-media response, and the UI routes local document links through that endpoint; I did not run a live browser save-dialog repro in this read-only review.

## Solution Assessment

No. The PR fixes the filename symptom by forcing attachment on the whole route, but the safer fix targets document/download responses and preserves image/audio/video inline behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep previewable assistant media inline:** `src/gateway/control-ui.ts:607-610`
  - body: This endpoint is also used as the `src` for image, audio, and video previews in webchat. Setting every response to `attachment` tells clients to save rather than process the media normally, so existing previews can stop rendering; gate attachment behavior to document/download responses or preserve inline handling for previewable media.
  - confidence: 0.9
- **[P2] Escape extended filename value characters:** `src/gateway/control-ui.ts:608`
  - body: `filename*` uses RFC 5987-style value syntax, but `encodeURIComponent` leaves characters like `'`, `(`, `)`, and `*` unescaped. Filenames containing those characters can produce an invalid extended parameter, so tighten the encoding for `filename*` or use a dedicated helper with coverage.
  - confidence: 0.82
