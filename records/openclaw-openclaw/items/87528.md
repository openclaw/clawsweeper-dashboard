## Summary

Keep open: the filename bug is still present on current main and in the latest release, but this PR is not merge-ready because it applies `attachment` to a mixed assistant-media endpoint that webchat also uses for inline image, audio, and video previews. The broader related work at https://github.com/openclaw/openclaw/pull/77912 is useful context but is still open and conflicting, so it is not a safe superseding landing path yet.

## What This Changes

The PR adds a basename-derived `Content-Disposition: attachment` header with `filename` and `filename*` to successful Control UI assistant-media responses.

## Best Possible Solution

Keep previewable media inline while adding targeted document/download filename behavior, standards-compliant filename encoding, and regression coverage for document downloads plus image/audio/video previews.

## Reproduction Assessment

Yes at source level: current main and `v2026.6.6` serve assistant-media without `Content-Disposition`, and webchat builds document/media URLs through that route. I did not run a live browser save-dialog repro in this read-only review.

## Solution Assessment

No. The PR fixes the filename symptom by forcing `attachment` on the whole route, but the maintainable fix should target document/download responses and preserve inline behavior for image/audio/video previews.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep previewable assistant media inline:** `src/gateway/control-ui.ts:610`
  - body: This endpoint is also used as the `src` for image, audio, and video previews in webchat. Setting every successful response to `attachment` tells clients to save instead of process normally, so existing previews can stop rendering or playing; gate attachment behavior to document/download responses or otherwise preserve inline handling for previewable media.
  - confidence: 0.9
- **[P2] Percent-encode extended filename value characters:** `src/gateway/control-ui.ts:610`
  - body: `filename*` uses RFC 5987 value syntax, but `encodeURIComponent` leaves characters like single quote, parentheses, and `*` unchanged. Filenames containing those characters can produce an invalid extended parameter, so tighten the encoder for `filename*` or reuse a helper that follows RFC 5987 value-chars.
  - confidence: 0.82
