## Summary

Keep this PR open because the underlying webchat download-filename bug is still present on current main and the latest release, but the proposed route-wide attachment header is not merge-ready for the shared preview/download endpoint.

## What This Changes

The PR adds a basename-derived `Content-Disposition: attachment` header to successful Control UI assistant-media responses.

## Best Possible Solution

Use targeted attachment disposition for document/non-inline MIME responses, keep image/audio/video previews inline, add focused gateway/UI coverage, and require redacted browser proof for the download filename flow.

## Reproduction Assessment

Yes at source level: current main and `v2026.6.8` serve assistant-media without `Content-Disposition` while document links use that route. I did not run a live browser save-dialog reproduction in this read-only review.

## Solution Assessment

No. The patch fixes the filename symptom by forcing `attachment` on the whole route, but the best fix should target document/non-inline responses and preserve browser-native previews for image, audio, and video.

  - body: This endpoint is also used as the `src` for local image, audio, and video previews in webchat. Setting every successful response to `attachment` can make existing previews download instead of rendering; gate this header to document/non-inline responses or another explicit download path.
  - confidence: 0.9
- **[P2] Encode filename* with RFC 5987-safe characters:** `src/gateway/control-ui.ts:610`
  - body: `encodeURIComponent` leaves characters such as single quotes, parentheses, and `*` unescaped, which are not safe in the extended `filename*` value syntax. Use a helper that tightens percent-encoding and supplies a readable ASCII fallback before interpolating the header.
  - confidence: 0.82
