## Summary

Keep open: the document-download filename problem is real, but this PR applies `attachment` to a mixed preview/download endpoint and can regress existing image/audio/video previews. A broader related PR has the safer targeted shape, but it is still unmerged, so this PR is not obsolete enough to close automatically.

## What This Changes

The PR adds a basename-derived `Content-Disposition: attachment` header with `filename` and `filename*` to every successful Control UI assistant-media response.

## Best Possible Solution

Keep previewable media inline while adding targeted document/download filename behavior, ideally by repairing this branch to match the narrower approach already explored in https://github.com/openclaw/openclaw/pull/77912 or by closing this PR after that path lands.

## Reproduction Assessment

Yes at source level. Current main lacks `Content-Disposition` on the assistant-media response and webchat document links use that route, but I did not run a live browser save-dialog repro in this read-only review.

## Solution Assessment

No. The PR fixes the filename symptom by forcing `attachment` on the whole route, but the maintainable fix should target document/download responses and preserve inline behavior for image/audio/video previews.

Full review comments:

- **[P1] Keep previewable assistant media inline:** `src/gateway/control-ui.ts:609-610`
  - body: This endpoint is also used as the `src` for image, audio, and video previews in webchat. Setting every response to `attachment` tells clients to save rather than process normally, so existing previews can stop rendering; gate attachment behavior to document/download responses or preserve inline handling for previewable media.
  - confidence: 0.9
- **[P2] Percent-encode extended filename value characters:** `src/gateway/control-ui.ts:610`
  - body: `filename*` uses RFC 5987 value syntax, but `encodeURIComponent` leaves characters like `'`, `(`, `)`, and `*` unescaped. Filenames containing those characters can produce an invalid extended parameter, so tighten the encoding for `filename*` or use an existing helper.
  - confidence: 0.82
