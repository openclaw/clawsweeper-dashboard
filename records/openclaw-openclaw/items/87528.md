## What This Changes

The PR adds a basename-derived `Content-Disposition: attachment` header with `filename` and `filename*` to every successful Control UI assistant-media response.

## Best Possible Solution

Add the filename hint only for non-previewable document/download responses, preserve inline image/audio/video assistant media, and cover both gateway headers and chat rendering with focused tests.

Yes at source level. Current main and v2026.6.6 lack `Content-Disposition` on assistant-media responses while webchat document links use that route, but this read-only review did not run a live browser save-dialog repro.

## Solution Assessment

No. The PR fixes the filename symptom by forcing `attachment` on the whole route, but the maintainable fix should target document/download responses and preserve inline image/audio/video previews.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep previewable assistant media inline:** `src/gateway/control-ui.ts:625-628`
  - body: This endpoint is also used as the `src` for image, audio, and video previews in webchat. Setting every response to `attachment` tells clients to save rather than process normally, so existing previews can stop rendering; gate attachment behavior to document/download responses or preserve inline handling for previewable media.
  - confidence: 0.91
- **[P2] Use a real RFC 5987 encoder for filename*:** `src/gateway/control-ui.ts:627`
  - body: `filename*` uses RFC 5987 value syntax, but `encodeURIComponent` leaves characters such as single quote, parentheses, and `*` unescaped. Filenames containing those characters can produce an invalid extended parameter, so tighten the encoder or reuse a helper that percent-encodes the full `attr-char` exclusions.
  - confidence: 0.84
