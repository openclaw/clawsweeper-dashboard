## Summary

Keep open: current main and the latest release still lack the requested assistant-media filename hint, but this PR is not merge-ready because it forces `attachment` on a route that also backs inline image, audio, and video previews.

## What This Changes

The PR adds a basename-derived `Content-Disposition: attachment` header to successful `handleControlUiAssistantMediaRequest` responses.

## Best Possible Solution

Add attachment disposition only for document or non-inline MIME responses, preserve image/audio/video inline previews, use a small safe filename-header helper, and cover both paths with focused gateway/UI tests plus browser proof.

## Reproduction Assessment

Yes at source level: current main and `v2026.6.8` serve assistant-media without `Content-Disposition`, while document links use that route. I did not run a live browser save-dialog reproduction in this read-only review.

## Solution Assessment

No. The patch fixes the filename symptom by forcing `attachment` on the whole route, but the best fix should target document/non-inline responses and preserve browser-native previews for image, audio, and video.
