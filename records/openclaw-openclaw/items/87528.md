## Summary

Keep open: the download-filename problem is real, but this patch should not merge as-is because it applies an attachment disposition to a route that also powers inline media previews, and the PR still lacks observable real behavior proof.

## What This Changes

The PR adds a Content-Disposition header to Control UI assistant-media responses using the served local filename.

## Best Possible Solution

Preserve inline rendering for previewable media while adding filename metadata/download behavior for documents, then prove the Control UI browser flow with focused tests and real evidence.

## Reproduction Assessment

Yes at source level: current main's assistant-media response lacks Content-Disposition, and webchat document links use that route. I did not run a live browser repro in this read-only review.

## Solution Assessment

No; the patch uses a global attachment disposition on a mixed preview/download endpoint. The narrower maintainable fix should condition or split disposition behavior and test both downloads and previews.
