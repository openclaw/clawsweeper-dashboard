## Summary

Keep this PR open: the filename bug is real, but this branch forces `attachment` on the mixed assistant-media route and can change existing image/audio/video previews; the related broader PR is still open and conflicting, so it is not a safe supersession target yet.

## What This Changes

The branch adds a basename-derived `Content-Disposition: attachment` header with `filename` and `filename*` to successful Control UI assistant-media responses.

## Best Possible Solution

Add filename-aware downloads for document/binary assistant media while preserving inline image/audio/video behavior, either by repairing this branch to be MIME-aware or by making the broader related PR viable and using it as the canonical fix.

## Reproduction Assessment

Yes, at source level. Current main and v2026.6.6 lack Content-Disposition on assistant-media responses, and the PR defect is also source-reproducible because the new header is unconditional on a route used by inline media elements.

## Solution Assessment

No. Adding a filename hint is the right direction, but the best fix is MIME-aware and should preserve image/audio/video previews while adding attachment disposition for document or binary downloads.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep previewable assistant media inline:** `src/gateway/control-ui.ts:608-610`
  - body: This endpoint also serves image, audio, and video assistant media used by browser-native webchat previews. Because the new header is unconditional, those existing responses can be treated as downloads instead of inline media; gate `attachment` to non-previewable document/binary responses.
  - confidence: 0.92
- **[P2] Use a strict encoder for filename*:** `src/gateway/control-ui.ts:610`
  - body: `filename*` uses RFC 5987-style syntax, but `encodeURIComponent` leaves apostrophes, parentheses, and asterisks unescaped; an apostrophe in a filename can break the extended parameter value. Use a dedicated RFC 5987 encoder for `filename*` and a sanitized ASCII quoted fallback for `filename`.
  - confidence: 0.8
