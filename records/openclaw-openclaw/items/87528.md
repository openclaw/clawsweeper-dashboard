## Summary

Keep this PR open, but not merge-ready: it addresses a real assistant-media filename gap that is still present on current main and latest release, but the patch applies `attachment` to a route that also powers inline image/audio/video previews and still needs stronger browser-level proof.

## What This Changes

The PR adds a basename-derived `Content-Disposition: attachment` header to successful `handleControlUiAssistantMediaRequest` responses in `src/gateway/control-ui.ts`.

## Best Possible Solution

Gate `Content-Disposition: attachment` to document or non-inline MIME responses, use a safe RFC 5987/6266 header builder with an ASCII fallback, add focused gateway/UI coverage, and require redacted browser proof before merge.

## Reproduction Assessment

Yes at source level: current `origin/main` and `v2026.6.9` serve assistant-media without `Content-Disposition`, while document attachments use that route. I did not run a live browser save-dialog reproduction in this read-only review.

## Solution Assessment

No as submitted. The patch fixes the filename symptom by forcing `attachment` on the whole route, but the maintainable fix should target document or non-inline responses and preserve browser-native previews for image, audio, and video.
