## Summary

Keep open: this is a plausible fix for the misleading Memory Wiki fs-safe diagnostic, but the hardlink-directory guard is ineffective for fs.Stats-shaped `isFile()` results and the contributor proof is still test-only.

## What This Changes

The PR narrows Memory Wiki imported-source fs-safe error wrapping and adds a bridge regression for non-symlink write failures while trying to skip hardlink cleanup for non-files.

## Best Possible Solution

Normalize the stat shape before the hardlink branch, only unlink regular multi-link files, add coverage for an empty directory at the page path, and keep the targeted fs-safe diagnostic behavior.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: current `writeImportedSourcePage` wraps every `FsSafeError` from the write path with symlink wording, and the PR's added test outlines a bridge import followed by a directory collision. I did not run tests because this review is read-only.

## Solution Assessment

No, not as written. The fs-safe diagnostic direction is right, but the non-file guard should call or normalize `isFile`; otherwise the directory cleanup path can still run for fs.Stats-shaped results.
