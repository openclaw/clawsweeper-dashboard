Keep open: current main and v2026.6.10 still expose the backend `exec.approval.list` RPC but not the requested `openclaw approvals pending` CLI/docs surface, and the open implementation candidates still need maintainer/security-boundary review before this issue can close.

## What This Changes

The issue requests a new `openclaw approvals pending` CLI command, with JSON and optional watch output, to list pending exec approval requests and show the requested command text.

## Best Possible Solution

Land a narrow CLI implementation that reuses the existing approval-list/resolve authority path, renders sanitized table and JSON output, adds focused CLI/docs coverage, and defers `--watch` unless maintainers scope it in.

## Reproduction Assessment

Yes. Source inspection of current main shows `openclaw approvals` only registers policy-oriented `get`, `set`, and `allowlist` commands while the backend `exec.approval.list` RPC already exists.

## Solution Assessment

Yes for the narrow direction, but it is not implemented on main. The maintainable solution is a CLI wrapper over the existing approval-list authority path with authorization, sanitization, tests, docs, and live proof rather than a second pending-approval protocol/store.
