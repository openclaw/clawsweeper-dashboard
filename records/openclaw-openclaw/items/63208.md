## Label Justifications

- `P2`: The issue blocks memory-wiki users from indexing organized vault content, but it is a bounded plugin feature rather than an emergency runtime failure.
- `impact:session-state`: Skipped nested wiki pages can leave the memory-wiki digest/search context incomplete relative to the user's vault state.

## Summary

Keep open: current main and v2026.5.18 still only enumerate direct Markdown children in memory-wiki page-group directories, and the active recursive/page-group PR is open rather than merged.

## What This Changes

The issue requests recursive memory-wiki indexing for Markdown files under page-group subdirectories, with possible path-derived IDs or a recursion config.

## Best Possible Solution

Define the nested wiki page contract, then land a shared deterministic recursive Markdown collector across compile, query/get, status, and lint while preserving source-page metadata and filesystem safety.

## Reproduction Assessment

Yes, at source level. The compile, query/get, and status paths each read only direct directory entries, so a Markdown file under sources/<subdir>/page.md is skipped; I did not run a live vault compile because this review is read-only.

## Solution Assessment

Unclear for the final implementation. Recursive collection is the right direction, but maintainers still need to choose the page-id, raw-source/frontmatter, configurability, and filesystem traversal contract before merging a fix.
