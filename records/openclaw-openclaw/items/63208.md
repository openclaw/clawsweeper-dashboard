## Summary

Keep open. Current main and latest release still scan only direct Markdown children in memory-wiki page-group directories, and the open recursive-scanning PR is blocked rather than a landed fix.

## What This Changes

The issue requests recursive memory-wiki discovery for Markdown files nested below page-group directories such as `sources/<topic>/page.md`.

## Best Possible Solution

Add a shared memory-wiki-local recursive Markdown collector for existing page-group directories, reuse it in compile/query/status, and separately decide path-derived IDs or configurable page groups.

## Reproduction Assessment

Yes, at source level. Place Markdown under `sources/<subdir>/page.md`; compile, search/get, and status each enumerate only direct Markdown children, so the nested page is skipped.

## Solution Assessment

Unclear for the broader open proposal. Recursive discovery is the right core direction, but configurable page groups and path-derived IDs need maintainer API/product direction before landing.
