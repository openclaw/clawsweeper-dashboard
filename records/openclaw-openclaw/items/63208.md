## Summary

Keep open. Current main and the latest release still enumerate only direct Markdown children in memory-wiki page-group directories, and the open PR that now attempts recursive/configurable groups is not a merged or usable implementation path yet.

## What This Changes

Request recursive memory-wiki discovery for Markdown files nested under sources and other page-group subdirectories while preserving useful vault organization.

## Best Possible Solution

Define the nested-page and raw-source contract, then land a normal-source implementation that shares one deterministic recursive Markdown collector across compile, query/get, status, and any affected lint paths.

## Reproduction Assessment

Yes, at source level. Put Markdown under sources/<subdir>/page.md; compile, query/get, and status each inspect only direct page-group children, so the nested page is skipped on current main and v2026.5.12.

## Solution Assessment

Unclear for the final implementation. Recursive collection is the right direction, but the current open PR is not mergeable as-is and maintainers still need to choose the page ID, raw-source, and configurability contract.
