## Summary

Keep open. Current main still uses one-level Markdown discovery for memory-wiki compile, wiki search/get, and status counts, and the recent user confirmation shows the nested Obsidian-vault use case remains active.

## What This Changes

The issue requests recursive memory-wiki Markdown discovery so nested vault pages can be compiled and queried without flattening directories.

## Best Possible Solution

Add a shared memory-wiki-local recursive Markdown collector for page-group directories, reuse it in compile/query/status, and document/test nested pages while avoiding new config or path-derived ID semantics unless maintainers explicitly choose them.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: place a valid Markdown page under `sources/<subdir>/page.md`; compile, search/get, and status each enumerate only direct Markdown files, so the nested page is skipped.

## Solution Assessment

Yes. A shared plugin-local recursive collector is the narrowest maintainable direction; adding an opt-in config flag or path-derived page IDs should be separate only if maintainers want that compatibility policy.
