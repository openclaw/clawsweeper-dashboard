## Summary

Keep open: current main and v2026.6.1 still enumerate only direct Markdown children in memory-wiki compile/search/status, while the requested recursive vault import is a real Obsidian-style workflow request that needs maintainer decisions around defaults, path-derived IDs, and collisions.

## What This Changes

memory-wiki users want `openclaw wiki compile` to recursively index Markdown under wiki group subdirectories such as `sources/topic/file.md` while preserving useful path-derived page identity.

## Best Possible Solution

Define recursive wiki page discovery across compile, search, status, and lint with explicit path-id collision rules before changing the default or adding any new config surface.

## Reproduction Assessment

Yes from source inspection: current compile/search/status enumerators read only direct Markdown children under each wiki group directory. I did not run the CLI because this review is read-only, but the code path matches the reported behavior.

## Solution Assessment

Unclear as a direct patch: recursive discovery is plausible, but the best fix needs maintainer choices for default behavior, path-derived page IDs, duplicate names, and whether lint/status/search change in lockstep.
