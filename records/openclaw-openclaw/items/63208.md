## Summary

Keep open. Current main still enumerates only direct Markdown children for memory-wiki compile, local search/get discovery, and status counts, so nested vault pages remain unsupported; the remaining work is narrow and plugin-owned, but it is a user-facing recursive-discovery capability rather than an implemented close candidate.

## What This Changes

Support recursive discovery of Markdown pages under memory-wiki page-group directories so nested source and Obsidian-style vault layouts participate in compile, search/get, status, and generated artifacts.

## Best Possible Solution

Add a shared plugin-local recursive Markdown collector for the existing page-group directories, reuse it in compile, query/get, and status, document nested layout support, and cover nested pages in focused memory-wiki tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to place a valid Markdown page under `sources/<subdir>/page.md`; current main only reads direct page-group children with `entry.isFile()`, so the nested file is skipped before compile, search/get, or status can see it.

## Solution Assessment

Yes. A shared memory-wiki-local recursive collector is the narrowest maintainable direction; it does not need a core/plugin SDK change, and an opt-in config flag is only needed if maintainers want compatibility gating.

## Repair Work Prompt

Implement recursive Markdown page discovery for the memory-wiki plugin. Add or extract one plugin-local collector that walks only the existing page-group directories (`sources`, `entities`, `concepts`, `syntheses`, `reports`), preserves deterministic ordering, skips `index.md`, and avoids symlink/vault escape behavior. Reuse it for compile page summaries, local query/get discovery, and status counts. Add focused tests for nested source pages appearing in compile outputs/digests, `wiki_search`/`wiki_get` local discovery, and status page/source counts. Update `docs/plugins/memory-wiki.md`, `docs/cli/wiki.md`, and `CHANGELOG.md` for user-facing nested page support. Do not add a core/plugin SDK API, do not broaden scanning outside the configured vault/page groups, and only add an `ingest.recursive` config option if maintainer review explicitly asks for an opt-in gate.
