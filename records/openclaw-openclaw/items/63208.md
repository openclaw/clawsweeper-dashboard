## Summary

Keep open. Current main still uses flat direct-child Markdown discovery in memory-wiki compile, search/get, and status, and the live issue has a recent second-user confirmation with no closing PR or protected-label reason to suppress normal follow-up.

## What This Changes

The issue requests recursive memory-wiki Markdown discovery so nested vault folders, such as Obsidian-style subdirectories under `sources/`, are indexed instead of only root-level Markdown files.

## Best Possible Solution

Add one plugin-local recursive Markdown collector for existing memory-wiki page-group directories and reuse it in compile, search/get, and status, with docs and focused nested-page tests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: put a valid Markdown page under `sources/<subdir>/page.md`; compile, search/get, and status each enumerate only direct Markdown files, so the nested page is skipped on current main.

## Solution Assessment

Yes. A shared memory-wiki-local recursive collector is the narrowest maintainable solution; an opt-in config flag or generated path-ID contract should be added only if maintainers explicitly want that compatibility policy.
