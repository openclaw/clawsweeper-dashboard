## Summary

Keep open. Current main still has the reported memory-wiki limitation: compile, local query/get fallback, and status all enumerate only direct Markdown children of each page-group directory, so nested `sources/.../*.md` files remain invisible. This is a narrow bundled-plugin bug, not a ClawHub/core-scope close, and the related open issue in the provided context tracks a different lint/frontmatter question rather than this recursive discovery gap.

## What This Changes

Request recursive discovery for Markdown files under memory-wiki page-group subdirectories, especially nested `sources/` imports from existing note/vault layouts.

## Best Possible Solution

Add a memory-wiki-owned recursive Markdown page collector for the existing page-group directories, share it across compile, local query/get fallback, and status counts, keep deterministic relative paths, skip `index.md`, and update focused plugin tests plus memory-wiki docs without adding a new core or plugin-SDK seam.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to place a Markdown page under `sources/<subdir>/` and run wiki compile/search/status; current main only enumerates direct entries with `fs.readdir` plus `entry.isFile()`, so the nested page cannot enter those paths.

## Solution Assessment

Yes. A shared extension-owned recursive collector is the narrowest maintainable fix; a new core API, external ClawHub plugin, or broader lint/raw-source redesign is not required for this issue.
