## Summary

Keep open. Current main still enumerates only direct Markdown children for memory-wiki compile, local query/get discovery, and status counts, so the reported nested `sources/.../*.md` workflow remains unsupported and is narrow enough for a focused plugin-owned fix.

## What This Changes

Support recursive nested Markdown pages under memory-wiki page-group directories such as `sources/<topic>/*.md` while preserving user directory organization.

## Best Possible Solution

Add one memory-wiki-owned recursive Markdown collector for the existing page-group directories and reuse it in compile, local search/get discovery, and status. Keep the core/plugin SDK unchanged, document nested page support, and add focused tests for nested source pages in indexes, digests, search/get, and counts.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to put a Markdown page under `sources/<subdir>/` and run compile/search/status; current main only reads direct `fs.readdir` entries with `entry.isFile()`, so the nested file is never enumerated.

## Solution Assessment

Yes. A shared plugin-local recursive collector is the narrowest maintainable fix; an opt-in config flag is only needed if maintainers want compatibility gating, and a new core API is not required.
