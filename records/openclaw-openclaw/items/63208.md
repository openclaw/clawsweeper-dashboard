## Summary

Keep open. Current main still only enumerates direct Markdown children for memory-wiki compile, local search/get, and status counts, while the issue and a second user report describe a real nested-vault use case that is not implemented yet.

## What This Changes

Support recursive Markdown page discovery under memory-wiki page-group directories so nested Obsidian-style vault layouts can be compiled, searched, fetched, and counted.

## Best Possible Solution

Add a shared memory-wiki-local recursive Markdown collector for page-group directories, reuse it in compile, query/get, and status, then document nested layouts and add focused regression coverage.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to put a valid Markdown page at `sources/<subdir>/page.md`; current main only reads direct file entries, so that page is skipped before compile, search/get, and status can see it.

## Solution Assessment

Yes. A shared plugin-local recursive collector reused across compile, query/get, and status is the narrowest maintainable direction; an opt-in config flag is only needed if maintainers want compatibility gating.
