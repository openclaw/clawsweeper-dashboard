## Summary

Keep open. Current main still lets a shared-session heartbeat model override become the persisted runtime `model`/`modelProvider`, and the status surfaces continue to read those same fields; the docs now describe the bleed/workaround but do not implement the requested boundary.

## What This Changes

Prevent heartbeat and internal async runs from making a normal user session appear switched to the heartbeat runtime model.

## Best Possible Solution

Add run-kind-aware session accounting so heartbeat, cron, and internal no-op follow-up runs either avoid overwriting normal user-turn runtime model fields or write separate background diagnostic fields that status labels distinctly.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: configure `agents.defaults.heartbeat.model` with the default shared heartbeat session, run a heartbeat, and inspect the session entry or status card; current main resolves and persists the heartbeat model into fields that status reads.

## Solution Assessment

Yes. The narrow maintainable fix is to thread `isHeartbeat` or a broader `runKind` through run accounting and status display, rather than reverting generic runtime-model status for normal turns.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/74217. Current main lets shared-session heartbeat runs with `agents.defaults.heartbeat.model` persist their provider/model into the same `SessionEntry.model` / `modelProvider` fields used for normal user-turn status, so `session_status` and related status surfaces can make the user session look switched to the heartbeat model. Implement a narrow run-kind-aware boundary: thread `isHeartbeat` or a broader `runKind` from heartbeat/follow-up run setup into session usage/accounting, and ensure heartbeat/internal/no-op background runs do not overwrite the normal user-turn runtime model fields, or store them in clearly labeled background diagnostic fields that status can distinguish. Preserve runtime model visibility for normal user turns and real fallback runs. Do not remove `heartbeat.model`, do not change user-selected model overrides, and do not make a startup/load-time migration. Add focused regression tests for a shared heartbeat model override leaving the normal session model/status unchanged, plus a control proving normal user-turn runtime/fallback model updates still appear. Update heartbeat/status docs only if the visible behavior changes, and add a single-line `CHANGELOG.md` fix entry because this is user-facing.
