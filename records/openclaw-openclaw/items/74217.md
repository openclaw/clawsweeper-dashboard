## Summary

Keep open. Current main still resolves the configured heartbeat model for heartbeat turns, then persists that run's provider/model through the shared session usage path; the status surfaces continue to read those same persisted runtime fields. Existing docs/tests now diagnose this as heartbeat model bleed and suggest `isolatedSession` as a workaround, but they do not add the requested persistence or status boundary.

## What This Changes

The issue asks OpenClaw to prevent heartbeat/internal async-completion runtime models from making normal user sessions and status output look switched to the heartbeat model.

## Best Possible Solution

Add run-kind-aware session accounting: heartbeat, cron, and internal async-completion/no-op runs should either skip overwriting normal user-turn `model`/`modelProvider` fields on shared sessions or write separate diagnostic-only background runtime fields that status labels clearly.

## Reproduction Assessment

Yes. A high-confidence source-level path exists: configure shared-session `agents.defaults.heartbeat.model`, run a heartbeat, then inspect the session entry or status card; current main forwards that model into the run and persists it into the fields status reads.

## Solution Assessment

Yes. The narrow maintainable fix is to thread `isHeartbeat` or a broader `runKind` into session persistence/status accounting and add regression coverage; reverting runtime-model status would regress useful normal-run diagnostics.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/74217. Heartbeat/internal async-completion runs using `agents.defaults.heartbeat.model` can overwrite the normal shared session `model`/`modelProvider`, which makes `/status` and `session_status` report the heartbeat model for user turns. Add a run-kind-aware persistence boundary, likely by threading `isHeartbeat` or `runKind` through `persistRunSessionUsage`/`persistSessionUsageUpdate` from `agent-runner` and `followup-runner`, so heartbeat/cron/internal no-op runs do not overwrite normal user-turn runtime model fields. Preserve normal user-turn runtime model visibility from #55425, and do not add bundled plugin ids or owner-specific policy to core. If background run diagnostics are still needed, store them in separate clearly labeled fields/status text. Add focused regression coverage for heartbeat model override persistence/status and update docs/changelog as needed; a user-facing fix should add a `CHANGELOG.md` Fixes entry crediting @Sitham94.
