## Summary

Keep open. Current main still has a source-visible path where a shared-session heartbeat can select `agents.defaults.heartbeat.model`, persist that run's `providerUsed`/`modelUsed` into the normal session entry, and have status surfaces read those persisted runtime fields as the session model. The related #55425 status work is useful but does not supersede this bug because it intentionally makes status trust persisted runtime model fields rather than separating heartbeat/internal-run telemetry.

## What This Changes

Prevent heartbeat or internal async-completion runs using `agents.defaults.heartbeat.model` from making the user-facing session status/runtime model look switched to that heartbeat model.

## Best Possible Solution

Add a run-kind-aware runtime metadata boundary: normal user turns and real fallback selections should continue updating user-facing session runtime model fields, while heartbeat, cron, and internal async-completion runs either do not overwrite those fields on shared user sessions or write their runtime model to a separate internal diagnostic field that status can label distinctly.

## Reproduction Assessment

Yes. Configure a shared-session heartbeat with `agents.defaults.heartbeat.model` and `isolatedSession` unset, let the heartbeat run, then inspect the session entry or `session_status`; current main has a direct source path from heartbeat model selection to persisted `modelProvider`/`model` fields used by status.

## Solution Assessment

Yes. The narrow maintainable fix is to propagate run kind into persistence/status accounting and keep internal-run model telemetry separate, preserving #55425's normal-turn runtime visibility instead of reverting it or relying on `heartbeat.isolatedSession` as the only answer.
