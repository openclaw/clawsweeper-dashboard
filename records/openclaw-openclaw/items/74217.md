## Summary

Keep open. Current main still routes the configured heartbeat model into the active run model and then persists that model/provider through shared session usage accounting, while the status surfaces intentionally read those persisted runtime fields. The related status PRs improved runtime-model visibility for normal runs, but they do not add the missing heartbeat/internal-run persistence boundary.

## What This Changes

The issue requests that heartbeat or internal async-completion runs stop overwriting the user-facing session model/status, or that status separately display internal-run runtime model metadata.

## Best Possible Solution

Add run-kind-aware session accounting: heartbeat, cron, and internal async-completion runs should not overwrite the normal user-turn model/modelProvider fields on shared sessions, or should write separate diagnostic-only runtime fields that status labels clearly as internal/background metadata.

## Reproduction Assessment

Yes. A source-level reproduction path is clear: configure a shared-session heartbeat with agents.defaults.heartbeat.model, let a heartbeat run, then inspect the session entry or status card; current main forwards that model into the active run and persists it into the same fields that status reads.

## Solution Assessment

Yes. The narrow maintainable direction is to thread isHeartbeat or a broader runKind into persistence/status accounting and add regression coverage; relying on heartbeat.isolatedSession is only a workaround, and reverting runtime-model status would regress useful normal-run visibility.
