## Summary

Keep open. Current main still supports shared-session heartbeat model overrides, forwards the heartbeat-selected provider/model into the normal reply run, and persists that run identity through the shared session usage path that status surfaces read. The newer structured heartbeat response tool improves heartbeat delivery semantics, but it does not add the requested persistence/status boundary.

## What This Changes

Prevent heartbeat or internal async-completion runs using `agents.defaults.heartbeat.model` from making a normal user session status/runtime model look switched to the heartbeat model.

## Best Possible Solution

Keep normal user-turn runtime model reporting, but make session accounting run-kind-aware so heartbeat, cron, and internal async/no-op runs cannot overwrite the user-facing selected/runtime session model unless they are intentionally operating as a user turn. If background runtime diagnostics are useful, store and label them separately in status.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction path exists: configure a shared-session `agents.defaults.heartbeat.model`, run a heartbeat, then inspect the session entry or status; current main forwards the heartbeat model and persists it into the fields status reads.

## Solution Assessment

Yes. The narrow maintainable direction is to thread `isHeartbeat` or a broader `runKind` into session persistence/status accounting and add regression coverage, while preserving the useful normal-run runtime model behavior from #55425.

## Repair Work Prompt

Fix openclaw/openclaw#74217. Heartbeat/internal async-completion runs that use `agents.defaults.heartbeat.model` must not make a normal user-facing session look switched to the heartbeat model. Thread `isHeartbeat` or a broader `runKind` through `persistRunSessionUsage` / `persistSessionUsageUpdate` from the main reply path and followup path. For heartbeat, cron, and internal no-op/async-completion runs on shared sessions, do not overwrite the normal `sessionEntry.model` / `modelProvider` fields, or add separate diagnostic-only background runtime fields that `/status` and `session_status` label clearly. Preserve #55425 behavior for normal user turns and real model fallback diagnostics. Keep `isolatedSession` as a workaround, not the required fix. Update focused tests for heartbeat model override persistence/status behavior, add a changelog fix entry, and update `docs/gateway/heartbeat.md` if the documented workaround changes. Do not change provider normalization, fallback selection, or default model routing semantics beyond the accounting/status boundary.
