## Summary

Keep open. Current main still preserves heartbeat runtime models only in the agent-command session-store path; the auto-reply and followup session-usage path can still persist the heartbeat run provider/model into the session row that status displays.

## What This Changes

The issue reports that heartbeat/internal async runs can persist their heartbeat model into the user-facing session runtime model shown by status.

## Best Possible Solution

Thread heartbeat-aware model preservation through the auto-reply and followup session usage writes so heartbeat turns preserve prior runtime model/context while normal user turns continue updating session usage.

## Reproduction Assessment

Yes, at source level. A heartbeat with a configured heartbeat model resolves that model as the active run provider/model, and the auto-reply persistence path still writes those values into the shared session entry that `session_status` reads.

## Solution Assessment

Yes. The narrow maintainable fix is to carry the existing heartbeat-preserve intent into `persistRunSessionUsage` and `persistSessionUsageUpdate`, rather than masking only the status display after session state is polluted.
