## Summary

Keep open. Current main contains a partial heartbeat-preservation fix, but the normal post-run usage accounting still persists the heartbeat run provider/model into the shared session fields that status reads.

## What This Changes

The issue requests that heartbeat/internal async runs using `agents.defaults.heartbeat.model` stop making a user-facing session appear switched to the heartbeat model in stored runtime state and status output.

## Best Possible Solution

Thread run-kind-aware session accounting through the shared post-run persistence path so heartbeat/background turns can record usage diagnostics without overwriting the user-turn runtime model fields that status displays.

## Reproduction Assessment

Yes. A source-level reproduction path is high confidence: configure a shared-session heartbeat with `heartbeat.model`, let it produce normal usage metadata, then the post-run `persistRunSessionUsage` path writes that provider/model into the session entry that `session_status` displays.

## Solution Assessment

Yes. The narrow maintainable fix is to extend the existing `preserveRuntimeModel` idea into `persistRunSessionUsage`/`persistSessionUsageUpdate`, rather than special-casing status or relying only on the embedded-run session-store helper.
