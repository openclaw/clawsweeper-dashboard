## Summary

Keep open. Current main preserves heartbeat runtime model only in the direct agent-command session-store path, while the WhatsApp/direct auto-reply path still persists the heartbeat run's provider/model into the shared session entry and `session_status` reads that runtime identity back.

## What This Changes

The issue asks that heartbeat/internal runs stop overwriting user-facing session runtime model fields, or that status distinguish heartbeat runtime from normal-turn model state.

## Best Possible Solution

Thread heartbeat/run-kind-aware session accounting through auto-reply persistence so background heartbeat runs preserve the prior user-session runtime model while normal user turns continue updating runtime and usage state.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to run a heartbeat with `agents.defaults.heartbeat.model` through the auto-reply path and inspect the session entry or `session_status`; current main still writes the heartbeat run's provider/model through `persistSessionUsageUpdate`.

## Solution Assessment

Yes. The narrow maintainable fix is to carry the existing heartbeat-preserve intent into `persistRunSessionUsage` and `persistSessionUsageUpdate`, rather than masking only the status display after the session store has already been polluted.
