## Summary

Keep open. Current main still supports shared-session heartbeat model overrides, resolves the heartbeat model for heartbeat turns, persists the run provider/model through the generic session-usage path, and status reads those same runtime fields; the docs now acknowledge the bleed/workaround but do not add the requested persistence or status boundary.

## What This Changes

Prevent heartbeat/internal async-completion runs from making the user-facing session status/runtime model look switched to the heartbeat model.

## Best Possible Solution

Add run-kind-aware session accounting so heartbeat, cron, and internal/no-op async-completion runs either do not overwrite normal user-turn `model`/`modelProvider` on shared sessions or write clearly labeled background diagnostic fields that status can distinguish.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction path exists: configure `agents.defaults.heartbeat.model` with the default shared heartbeat session, run a heartbeat, then inspect the session entry or status card; current main forwards and persists that heartbeat model into the fields status reads.

## Solution Assessment

Yes. The narrow maintainable fix is to thread `isHeartbeat` or a broader `runKind` into session persistence/status accounting and add regression coverage, while preserving normal user-turn runtime-model diagnostics.

## Repair Work Prompt

Fix the heartbeat model bleed reported in https://github.com/openclaw/openclaw/issues/74217. Current main resolves `agents.defaults.heartbeat.model` for heartbeat turns, then persists that provider/model into the shared session's `model`/`modelProvider`, causing `/status` and `session_status` to look switched even when normal turns use the configured default. Add a generic run-kind-aware session accounting seam, preferably by threading `isHeartbeat` or `runKind` through `persistRunSessionUsage`/`persistSessionUsageUpdate`, so heartbeat/internal no-op/background runs on shared sessions do not overwrite the normal user-turn runtime model fields. Preserve normal user-turn runtime model diagnostics, explicit per-session model overrides, fallback/runtime status behavior, isolated heartbeat sessions, and CLI session bindings. If separate background diagnostic model fields are introduced, make status label them clearly rather than replacing the primary user-turn model line. Add focused regression tests for a shared-session heartbeat model override not overwriting user-facing runtime fields/status, and update `CHANGELOG.md` because this is user-facing. Treat #74257 only as related context; do not broaden this PR into Telegram delivery/approval leak handling unless the same small seam directly covers it.
