## Summary

Keep open. Current main still lets heartbeat-selected provider/model values flow through the auto-reply/followup session-usage persistence path, while the existing heartbeat preservation guard only covers the agent-command session-store path.

## What This Changes

Prevent heartbeat/internal runs using `agents.defaults.heartbeat.model` from persisting that runtime model into normal user-session model/status fields.

## Best Possible Solution

Thread heartbeat-aware session accounting through auto-reply and followup persistence so heartbeat turns preserve prior runtime model/context while normal user turns continue updating runtime and usage state.

## Reproduction Assessment

Yes, source-level. A heartbeat run with a configured heartbeat model reaches `runReplyAgent`, which then persists that run provider/model through `persistSessionUsageUpdate`; I did not run a live WhatsApp heartbeat because this review is read-only.

## Solution Assessment

Yes. The narrow maintainable fix is to carry heartbeat-preserve intent into `persistRunSessionUsage`/`persistSessionUsageUpdate`, rather than hiding the polluted value only in `session_status`.
