## Summary

Keep open. Current main still has a source-level path where a heartbeat run resolves `agents.defaults.heartbeat.model` as the active run model, then persists that provider/model into the shared session runtime fields that `session_status` and `/status` read. The newer #75452/#75567 work is clearly related, but that open PR is aimed at the `src/agents/command` session-store path and does not remove the auto-reply/session-usage path shown for this WhatsApp heartbeat report.

## What This Changes

The issue requests preventing heartbeat or internal async-completion model overrides from overwriting, or being reported as, the normal user session runtime model.

## Best Possible Solution

Add run-kind-aware runtime metadata accounting in the auto-reply path: heartbeat, cron, and internal async-completion runs should either preserve the previous user-turn runtime model fields or persist their runtime model into separate diagnostic fields, while normal user turns and real fallback selections continue updating status.

## Reproduction Assessment

Yes. Configure a shared session with `agents.defaults.heartbeat.model` different from the normal default model, trigger a heartbeat, then inspect the session entry or status output; current main forwards the heartbeat model into run usage persistence and status reads that same runtime identity.

## Solution Assessment

Yes for the requested direction, but not for current main. The narrow maintainable fix is to thread run kind or `isHeartbeat` into `persistRunSessionUsage`/`persistSessionUsageUpdate` and keep internal-run model telemetry separate; relying on `heartbeat.isolatedSession` is only a workaround.
