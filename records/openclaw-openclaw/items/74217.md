## Summary

Keep open. Current main still resolves `heartbeat.model` into the active heartbeat run, persists that provider/model through the generic shared-session usage path, and status surfaces read those same runtime fields; the recent docs and reset-message coverage diagnose heartbeat model bleed but do not isolate it.

## What This Changes

Prevent heartbeat/internal async-completion model overrides from making normal user sessions and `/status` appear switched to the heartbeat runtime model.

## Best Possible Solution

Add run-kind-aware session accounting so heartbeat/background turns either do not overwrite normal user-turn `model`/`modelProvider` or write separately labeled background diagnostic fields that status can distinguish.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction path exists: configure `agents.defaults.heartbeat.model` on the default shared heartbeat session, run a heartbeat, then inspect the session entry or status card; current main forwards and persists that heartbeat model into the fields status reads.

## Solution Assessment

Yes. The narrow maintainable fix is to thread `isHeartbeat` or a broader run kind into session persistence/status accounting and add regression coverage, rather than relying on context-overflow guidance as the user-facing boundary.

## Repair Work Prompt

Fix issue https://github.com/openclaw/openclaw/issues/74217. Heartbeat/internal async-completion runs using `agents.defaults.heartbeat.model` currently persist their runtime provider/model into the shared user session fields that `/status` and `session_status` treat as the normal runtime model. Add run-kind-aware session accounting so heartbeat/background runs do not overwrite normal user-turn `model`/`modelProvider`, or store them in separate labeled diagnostic fields that status can distinguish. Keep normal user-turn runtime fallback/status diagnostics intact, preserve explicit `/model` override precedence, and do not change heartbeat scheduling or delivery semantics. Likely files: `src/infra/heartbeat-runner.ts`, `src/auto-reply/reply/get-reply.ts`, `src/auto-reply/reply/agent-runner.ts`, `src/auto-reply/reply/followup-runner.ts`, `src/auto-reply/reply/session-run-accounting.ts`, `src/auto-reply/reply/session-usage.ts`, `src/agents/tools/session-status-tool.ts`, `src/status/status-message.ts`, plus targeted tests and `CHANGELOG.md` because this is user-visible. Add regression coverage for a shared-session heartbeat with a different heartbeat model and no per-session override, asserting the heartbeat run uses the override but the user-facing session/status model remains the normal configured/default model or clearly labels the background model separately.
