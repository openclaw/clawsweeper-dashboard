## Summary

Keep open. Current main still routes heartbeat `heartbeat.model` into the run, then persists `providerUsed` and `modelUsed` into the shared session row without an `isHeartbeat` or internal-run discriminator; the status surfaces read those persisted runtime fields, so the user-facing confusion remains valid.

## What This Changes

The issue requests that heartbeat/internal async runs stop overwriting or misleading user-facing session `model`/`modelProvider` and status metadata.

## Best Possible Solution

Add a run-kind-aware metadata boundary: heartbeat, cron, and internal async completion runs should not overwrite user-turn `model`/`modelProvider` on shared session rows, or they should persist their runtime model in a separate internal diagnostic field that status labels separately.

## Reproduction Assessment

Yes. Configure a shared-session heartbeat with `agents.defaults.heartbeat.model` and `isolatedSession` unset/false, let the heartbeat run, then inspect the session entry or status output; current main forwards the heartbeat model but drops the heartbeat marker before session runtime persistence.

## Solution Assessment

Yes. The narrowest maintainable fix is to thread run kind or `isHeartbeat` into session usage persistence/status accounting and keep internal-run model telemetry separate; using `heartbeat.isolatedSession` is only a workaround and reverting runtime-model status would regress normal debugging behavior.

## Repair Work Prompt

Fix the heartbeat/internal runtime model bleed described in openclaw/openclaw#74217. Current main forwards `heartbeatModelOverride` and `isHeartbeat`, but `agent-runner` persists `modelUsed`/`providerUsed` into shared session `model`/`modelProvider` without preserving run kind, so status can show a heartbeat model as the user session model. Add a run-kind-aware boundary so heartbeat/internal async completion/cron-style internal runs do not overwrite user-turn runtime model fields on shared session rows, or persist/display them in a clearly separate internal diagnostic field. Preserve #55425 behavior for normal user turns and real fallback selections: status should still show persisted runtime model when it came from ordinary user traffic/fallback routing. Do not make `heartbeat.isolatedSession` the only fix. Add focused regression coverage for a shared-session heartbeat with `agents.defaults.heartbeat.model` and no explicit session model override. Update docs only if the status/diagnostic contract changes. Add a single-line `CHANGELOG.md` fix entry crediting @Sitham94 unless maintainer guidance says otherwise.
