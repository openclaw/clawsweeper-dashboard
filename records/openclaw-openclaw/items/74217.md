## Summary

Keep open. Current main still supports a heartbeat-specific model override and forwards it into the normal reply path, then persists the resulting provider/model through shared session usage fields that status surfaces read; the existing docs and tests acknowledge heartbeat model bleed as a diagnostic/workaround case rather than fixing the persistence boundary.

## What This Changes

The issue reports that heartbeat or internal async-completion runs using `agents.defaults.heartbeat.model` can overwrite the user-facing session runtime model shown by `/status` and `session_status`.

## Best Possible Solution

Add run-kind-aware session accounting so heartbeat, cron, and internal no-op/background completion runs either do not overwrite the normal user-turn `model`/`modelProvider` fields on shared sessions or write separate diagnostic-only background runtime fields that status labels clearly.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction path exists: configure a shared-session `agents.defaults.heartbeat.model`, run a heartbeat, then inspect the session entry or status card; current main forwards the heartbeat model into the run and persists it into fields that status reads.

## Solution Assessment

Yes. The narrow maintainable fix is to thread `isHeartbeat` or a broader `runKind` into session persistence/status accounting and add regression coverage; reverting runtime-model status wholesale would lose useful normal-run diagnostics.
