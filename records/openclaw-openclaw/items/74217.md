## Summary

Keep open. Current main still fixes heartbeat runtime-model preservation only in the agent-command session-store path; the auto-reply path can still persist the heartbeat run provider/model into the shared session entry that `session_status` displays.

## What This Changes

Heartbeat/internal runs should not overwrite user-facing session runtime model fields or make status report a heartbeat model as the normal session model.

## Best Possible Solution

Thread heartbeat-aware session accounting through auto-reply persistence so heartbeat turns preserve the prior runtime model/context while normal user turns keep updating runtime and usage state.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to run a heartbeat with `agents.defaults.heartbeat.model` through the auto-reply path and inspect the session entry or `session_status`; current main still writes that run's provider/model through `persistSessionUsageUpdate`.

## Solution Assessment

Yes. The narrow maintainable fix is to carry the existing heartbeat-preserve intent into `persistRunSessionUsage` and `persistSessionUsageUpdate`, rather than masking only the status display after the session store has already been polluted.
