## Summary

Keep open. Current main still routes heartbeat model selection through the WhatsApp/direct auto-reply path and persists the run's `providerUsed`/`modelUsed` into the shared session entry, while `session_status` reads those runtime fields back as the displayed model when no explicit override exists.

## What This Changes

Heartbeat/internal async-completion runs should not make a normal user-facing session status look switched to the heartbeat model.

## Best Possible Solution

Thread heartbeat/run-kind-aware session accounting through auto-reply persistence so background heartbeat runs preserve the prior user-session runtime model while normal user turns continue updating runtime and fallback state.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to run a heartbeat with `heartbeat.model` through the WhatsApp/direct auto-reply path and inspect the session entry or `session_status`; current main still persists the heartbeat run's provider/model through `persistSessionUsageUpdate`.

## Solution Assessment

Yes. The narrow maintainable fix is to carry the existing heartbeat-preserve intent into `persistRunSessionUsage`/`persistSessionUsageUpdate`, rather than masking only the status display after the session store has already been polluted.
