## Summary

Keep open. Current main has a partial heartbeat-preserve fix for the `openclaw agent` session-store path, but the reported WhatsApp/direct heartbeat path still persists the heartbeat run's `modelUsed`/`providerUsed` through auto-reply session usage, and status still reads those runtime fields.

## What This Changes

Prevent heartbeat or internal async runs using `agents.defaults.heartbeat.model` from overwriting the normal user session runtime model shown by status.

## Best Possible Solution

Thread run-kind-aware session accounting through the auto-reply persistence path so heartbeat/background turns preserve the prior user-session runtime model while normal user turns and fallback changes still update it.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction remains: configure a shared-session heartbeat model, run the heartbeat through the WhatsApp/direct auto-reply path, and inspect the session entry or `session_status`; current main still writes the heartbeat provider/model through `persistSessionUsageUpdate`.

## Solution Assessment

Yes. The maintainable fix is run-kind-aware persistence/status accounting, reusing the recent preserve-runtime-model intent but applying it to the auto-reply session usage path that the reported heartbeat uses.
