## Summary

Keep open. Current main still lets a heartbeat run resolve `agents.defaults.heartbeat.model` as the active run model, then pass that model/provider into shared session usage persistence; the status surfaces then read the persisted runtime model fields. The merged #55425/#65807 status work is useful for normal runtime visibility, but it does not add the missing heartbeat/internal-run persistence boundary.

## What This Changes

Prevent heartbeat or internal async-completion runs from making a normal user session and status output appear switched to the heartbeat runtime model.

## Best Possible Solution

Add run-kind-aware runtime metadata accounting: heartbeat, cron, and internal async-completion runs should not overwrite the user-turn `model`/`modelProvider` fields on shared sessions, or they should persist/display their runtime identity in separate diagnostic fields while normal user turns and fallback selections continue to update status.

## Reproduction Assessment

Yes. A source-level reproduction path is clear: configure a shared-session heartbeat with `agents.defaults.heartbeat.model`, let the heartbeat run, then inspect the session entry or status card; current main forwards the heartbeat model into session usage persistence and status reads that same persisted runtime identity.

## Solution Assessment

Yes. The narrowest maintainable fix is to thread run kind or `isHeartbeat` into persistence/status accounting and keep internal-run model telemetry separate; relying on `heartbeat.isolatedSession` is only a workaround and reverting runtime-model status would regress the normal debugging behavior added by #55425.
