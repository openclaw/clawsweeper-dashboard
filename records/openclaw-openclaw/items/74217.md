## Summary

Keep open. Current main still has the reported persistence boundary bug: heartbeat runs can resolve a heartbeat-specific model, pass that model through the normal reply execution path, and persist it into the shared session runtime model fields that status then reads as the user-facing session model. The existing diagnostic/docs now acknowledge “heartbeat model bleed,” but they only give a workaround and do not stop the misleading status/session state.

## What This Changes

The issue asks OpenClaw to stop heartbeat or internal async-completion model selection from overwriting the normal user session runtime model, or to display internal/background runtime model metadata separately from normal user-turn status.

## Best Possible Solution

Add run-kind-aware session accounting so heartbeat, cron, and internal async-completion executions do not overwrite normal user-turn model/modelProvider fields on shared sessions. If internal runtime model metadata is useful, store it in separate diagnostic fields and label it separately in status output.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction path exists: configure a shared-session heartbeat with agents.defaults.heartbeat.model, allow a heartbeat/internal run, then inspect the session entry or status output; current main still threads that model through the normal persistence path.

## Solution Assessment

Yes. The narrow maintainable fix is to thread isHeartbeat or a broader runKind into session persistence/status accounting and add regression coverage; relying on isolatedSession is only a workaround and reverting runtime-model status would regress normal debugging visibility.

## Repair Work Prompt

Fix the heartbeat/internal-run model persistence leak reported in https://github.com/openclaw/openclaw/issues/74217. Current main resolves agents.defaults.heartbeat.model for heartbeat runs, then agent-runner persists providerUsed/modelUsed through persistSessionUsageUpdate into the shared session model/modelProvider fields that session_status and /status read as the normal runtime model. Add a run-kind-aware boundary so heartbeat, cron, and internal async-completion/no-op runs do not overwrite normal user-turn runtime model fields on shared sessions, or store those internal runtime models separately as diagnostic-only metadata with clear status labeling. Keep the fix generic: do not add channel-specific or provider-specific behavior, do not make heartbeat.isolatedSession the only solution, and do not revert the normal runtime-model status behavior from #55425. Add focused regression coverage and a CHANGELOG.md fix entry with reporter attribution because the behavior is user-facing.
