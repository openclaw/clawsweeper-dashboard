## Summary

Keep open. Current main preserves heartbeat runtime model only in the agent-command session-store path; the auto-reply path can still persist a heartbeat run's provider/model into the shared session entry that `session_status` prefers.

## What This Changes

Heartbeat/internal runs using `agents.defaults.heartbeat.model` should not make normal user session status or runtime fields report the heartbeat model.

## Best Possible Solution

Thread heartbeat-aware runtime-model preservation through auto-reply session accounting so heartbeat turns preserve the prior user-session model identity while normal turns keep updating runtime and usage state.

## Reproduction Assessment

Yes, at source level. A heartbeat with a different `heartbeat.model` reaches auto-reply with the heartbeat provider/model, and current main still writes those values through `persistSessionUsageUpdate` into the shared session entry that `session_status` displays.

## Solution Assessment

Yes. The narrow maintainable fix is to carry the already-established heartbeat preserve semantics into `persistRunSessionUsage` and `persistSessionUsageUpdate`, rather than masking the status display after the session store has been polluted.
