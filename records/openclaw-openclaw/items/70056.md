## Summary

Keep open: this PR targets a real gateway stop-timeout bug, but the proposed direct cleanup still has a lifecycle race, lacks real behavior proof, and is paired with an open reporter issue in an unresolved stop-timeout cluster.

## What This Changes

The PR deletes `store.aborts`/`store.tasks` on `stopChannel` timeout, marks the runtime stopped with `lastStopAt`, and updates the focused test to expect restart after timeout.

## Best Possible Solution

Define a guarded gateway lifecycle recovery contract that can recover from a never-settling timed-out channel task while preventing old task callbacks or plugin transports from corrupting or duplicating the replacement lifecycle.

## Reproduction Assessment

Yes, at source level. Current main has the task-map start guard, timeout state path, health-monitor stop-then-start caller, and focused tests showing the timeout boundary, but I did not run tests in this read-only review.

## Solution Assessment

No. The direct map cleanup fixes the no-op symptom, but it is not the safest solution without task identity/generation fencing or an accepted force-retire contract for old lifecycle side effects.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Guard old task callbacks before freeing the slot:** `src/gateway/server-channels.ts:576-577`
  - body: Deleting `store.tasks` here lets `startChannelInternal` launch a replacement while the original `trackedPromise` can still run its catch/finally/restart chain. If the old task later settles, those callbacks can mark the replacement stopped or enqueue restart bookkeeping for the wrong lifecycle, so task-owned writes need an identity or generation guard before this slot is reusable.
  - confidence: 0.9
