## Summary

Keep this PR open. Current main still has the stop-timeout bug the PR targets, but the proposed cleanup releases the task slot while the old task can still run unguarded callbacks, so it needs revision before merge rather than cleanup closure.

## What This Changes

The PR changes the gateway `stopChannel` timeout path to delete stored abort/task entries, mark the account stopped with `lastStopAt`, and updates the channel test to expect a successful restart after timeout.

## Best Possible Solution

Revise the lifecycle fix so timeout recovery can release or replace the task slot only with task identity or generation guards around all task-owned writes, restart bookkeeping, and cleanup. The final patch should include a regression where the timed-out task settles after a successful replacement start, plus the required changelog entry.

## Reproduction Assessment

Yes. Current main has a deterministic unit-test shape: start a never-settling account task, stop it past `CHANNEL_STOP_ABORT_TIMEOUT_MS`, then observe that `startChannel` is skipped and runtime remains `running: true`; the PR race can be reproduced by letting that old task settle after a replacement start.

## Solution Assessment

No. The PR fixes the stale map entry symptom but is not the best complete fix because it deletes the current task entry before guarding callbacks owned by the old `trackedPromise`; a task identity or generation guard is the safer narrow solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Guard stale task callbacks before releasing the slot:** `src/gateway/server-channels.ts:579-580`
  - body: Deleting `store.tasks` in the timeout branch lets a replacement start while the original `trackedPromise` is still pending. When that old task later settles, its `.catch`, `.finally`, and restart chain can still write runtime state or restart bookkeeping for the same account without proving task ownership, so it can mark the replacement stopped or enqueue another restart. Guard those task-owned callbacks with an identity/generation check and cover delayed settlement after restart.
  - confidence: 0.9

  - body: This changes user-facing gateway recovery behavior for stuck channel stops, but the PR does not update `CHANGELOG.md`. Repo policy requires user-facing fixes to include a changelog entry before merge.
