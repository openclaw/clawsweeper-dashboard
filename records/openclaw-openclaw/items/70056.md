## Summary

Keep open. Current main still has the reported stop-timeout dead-channel behavior, and this PR targets the right gateway lifecycle code, but the patch is not safe to merge as written because releasing the task slot before guarding the old task's callbacks lets a delayed timed-out task overwrite replacement runtime state.

## What This Changes

The PR changes `stopChannel` timeout handling to delete stored abort/task entries, mark the account stopped with `lastStopAt`, and update the gateway channel test to expect restart after a stop timeout.

## Best Possible Solution

Revise the PR so timeout cleanup permits recovery while task-owned callbacks are identity- or generation-guarded before writing runtime state, restart bookkeeping, or cleanup. Land that with delayed-settle regression coverage and a changelog entry for the gateway fix.

## Reproduction Assessment

Yes. Current main has a deterministic unit-test shape for the no-restart bug, and the remaining PR race can be reproduced by letting the timed-out task settle after a successful replacement start and asserting the replacement remains running.

## Solution Assessment

No. The PR fixes the stale map entry but releases the slot before guarding old task callbacks; the narrower maintainable fix is release plus task ownership/generation guards and delayed-settle regression coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Guard stale task callbacks before releasing the slot:** `src/gateway/server-channels.ts:579`
