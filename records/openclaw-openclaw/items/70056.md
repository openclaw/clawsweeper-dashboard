## Summary

Keep open: this is a plausible paired bug-fix PR for an open gateway stop-timeout issue, but the patch is not mergeable because it releases the task slot before stale task callbacks are fenced and it has no real behavior proof.

## What This Changes

The PR changes `stopChannel` timeout handling to delete abort/task entries, mark the account stopped, record `lastStopAt`, and update the focused restart test.

## Best Possible Solution

Implement a guarded gateway lifecycle recovery fix that preserves manual/recovery semantics and fences old task callbacks or defines an accepted force-retire contract, then prove it with tests and redacted real gateway proof.

## Reproduction Assessment

Yes, at source level: current main shows the task-map start guard and old task callback chain, and this PR's timeout deletion allows a replacement before the old promise settles. I did not run tests in this read-only review.

## Solution Assessment

No. The direct cleanup fixes one stale-entry symptom, but it is not the safest fix without task identity or generation fencing and it conflicts with current main's manual-versus-recovery timeout split.
