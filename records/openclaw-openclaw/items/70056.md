## Summary

Keep open: the PR addresses a real gateway stop-timeout bug, but the patch frees the account slot while the old task can still run lifecycle callbacks, and current main now has a later recovery-stop split that this branch does not preserve cleanly.

## What This Changes

The PR changes `stopChannel` so the stop-timeout branch deletes stored abort/task entries, marks the runtime as stopped, records `lastStopAt`, and updates the focused timeout restart test.

## Best Possible Solution

Use a guarded gateway lifecycle fix that preserves current manual-versus-recovery timeout semantics and fences task-owned status and restart callbacks by task identity or generation.

## Reproduction Assessment

Yes, source-level: current main shows the timeout branch, the task-map start guard, and focused timeout tests; the PR-specific regression follows when the old task settles after this patch has released the account slot.

## Solution Assessment

No. The PR fixes part of the stale-entry symptom, but freeing the task slot without identity fencing is not the safest fix and it needs to be reconciled with current recovery-stop behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Guard old task callbacks before releasing the slot:** `src/gateway/server-channels.ts:576-577`
  - body: Deleting `store.tasks` here lets `startChannelInternal` launch a replacement while the original `trackedPromise` can still run its catch/finally/restart chain. If the old task later settles, those callbacks can mark the replacement stopped or enqueue restart bookkeeping for the wrong lifecycle, so task-owned writes need an identity or generation guard before this slot is reusable.
  - confidence: 0.9
