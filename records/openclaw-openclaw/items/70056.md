## Summary

Keep open: this PR targets a real current-main gateway bug, but the proposed timeout cleanup still allows stale task callbacks to mutate replacement channel state and it is missing the required changelog entry.

## What This Changes

The PR changes the gateway stop-timeout path to delete stored task/abort entries, mark the channel stopped with `lastStopAt`, and update the server-channel test to expect a successful restart after timeout.

## Best Possible Solution

Revise the lifecycle fix so timeout recovery releases or replaces the task slot only with task identity or generation guards around all task-owned runtime writes, restart bookkeeping, and cleanup, then add a delayed-settlement regression test and changelog entry.

## Reproduction Assessment

Yes. Current main has a deterministic unit-test shape: start a never-settling account task, stop past `CHANNEL_STOP_ABORT_TIMEOUT_MS`, then observe `startChannel` is skipped and runtime stays `running: true`; the PR race is reproduced by letting the timed-out task settle after a replacement start.

## Solution Assessment

No. The PR fixes the stale map entry symptom, but deleting the task slot before guarding callbacks owned by the old `trackedPromise` is not the safest complete fix; task identity or generation guards are the narrow maintainable path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Guard stale task callbacks before freeing the slot:** `src/gateway/server-channels.ts:579`
  - body: Deleting `store.tasks` in the timeout branch lets a replacement start while the original `trackedPromise` is still pending. When that old task later settles, its catch/finally/restart chain can still write runtime state or enqueue restart bookkeeping for the same account without proving task ownership, so it can mark the replacement stopped or start an extra restart cycle. Guard those task-owned callbacks with an identity/generation check and cover delayed settlement after restart.
  - confidence: 0.9
- **[P3] Add the required changelog entry:** `src/gateway/server-channels.ts:578-579`
  - body: This changes user-facing gateway recovery behavior for stuck channel stops, but the PR does not update `CHANGELOG.md`. Repo policy requires user-facing fixes to include a changelog entry before merge.
  - confidence: 0.82
