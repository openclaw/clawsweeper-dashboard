## Summary

Keep open: this is a focused gateway bug-fix PR, but the current patch is not safe to merge because it frees the task slot while the old task callback chain can still mutate a replacement lifecycle, and it still lacks real behavior proof.

## What This Changes

The branch deletes abort/task store entries on stop timeout, marks the runtime stopped with `lastStopAt`, and updates the stop-timeout restart test.

## Best Possible Solution

Revise or replace this with a guarded gateway lifecycle fix that preserves the merged recovery-stop split, fences stale task-owned status/restart callbacks by identity or generation, and includes focused regression coverage plus real gateway proof.

## Reproduction Assessment

Yes, source-level. Current main exposes the stale-task path through the timeout branch, task-map start guard, and focused tests; the PR-specific race follows when the old task settles after this patch has allowed a replacement start.

## Solution Assessment

No. The patch addresses stale map entries, but releasing the slot before fencing the old task's callback chain is not the narrowest maintainable fix; a task-identity or generation guard is needed before replacement starts are safe.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Guard old task callbacks before releasing the slot:** `src/gateway/server-channels.ts:576-577`
  - body: Deleting `store.tasks` here lets `startChannelInternal` treat the account as free while the old `trackedPromise` can still run its catch/finally/restart chain. If that old task settles after a replacement start, those callbacks can mark the replacement stopped or enqueue restart bookkeeping for the wrong lifecycle; guard task-owned writes and restarts by identity or generation before freeing this slot.
  - confidence: 0.9
