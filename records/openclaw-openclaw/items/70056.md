## Summary

Keep open: this PR targets a real gateway stop-timeout bug, but it is not ready to merge because freeing the task slot while the old promise still owns callbacks can corrupt a replacement lifecycle. Current main partially fixed health-monitor recovery through a later merged PR, but the paired issue remains open and this external PR also still lacks real behavior proof.

## What This Changes

The PR changes `stopChannel` timeout handling to delete stored abort/task entries, mark the runtime stopped with `lastStopAt`, and update the stop-timeout restart test.

## Best Possible Solution

Update or replace this PR with a guarded gateway lifecycle fix that can retire timed-out tasks without stale callbacks mutating a newer account lifecycle, preserves the recovery-stop behavior already on main, and includes targeted regression coverage plus real gateway/runtime proof.

## Reproduction Assessment

Yes, source-level. The current `server-channels` timeout branch, task-map start guard, and focused tests show the stale task path, and the PR-specific race is reproducible by letting the old task settle after a replacement start.

## Solution Assessment

No. The branch addresses stale map entries, but freeing `store.tasks` before guarding old task-owned callbacks is not the narrowest safe fix because it can let the old lifecycle overwrite the replacement state.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Guard old task callbacks before releasing the slot:** `src/gateway/server-channels.ts:576-577`
  - body: Deleting `store.tasks` here makes `startChannelInternal` treat the account as free while the old `trackedPromise` still runs its catch/finally/restart chain. If that old task settles after a replacement start, those callbacks can mark the replacement stopped or enqueue restart bookkeeping for the wrong lifecycle; guard task-owned writes and restarts by identity or generation before freeing this slot.
  - confidence: 0.9
