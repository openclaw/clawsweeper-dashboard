## Summary

Keep open: current main has the fast embedded model-resolution path, but it still synchronously loads runtime plugins on embedded-runner and auto-reply dispatch hot paths, and there is no source/test/release evidence proving the Raspberry Pi Slack latency regression is fixed.

## What This Changes

The issue requests preventing expensive plugin/runtime/provider discovery from blocking Slack Socket Mode replies on Raspberry Pi-class gateways after upgrading beyond 2026.4.21.

## Best Possible Solution

Preserve the fast model-resolution path and move any remaining Slack warm-message runtime/plugin/model discovery work to cached startup, background, or a narrow low-resource gateway path with regression coverage.

## Reproduction Assessment

Yes: the report includes concrete setup steps, version comparison, missed Slack pong logs, timing traces around ensureOpenClawModelsJson, and a workaround trace. I did not live-reproduce it on Raspberry Pi hardware in this read-only review.

## Solution Assessment

Unclear: the model-resolution fast path is already the right fix for the models.json delay, but it does not prove the broader runtime/plugin loading pressure is solved. The safer path is maintainer profiling of the Slack hot path before changing loader semantics.
