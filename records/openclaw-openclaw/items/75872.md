## Summary

Keep open: current main has the fast embedded model-resolution path and adjacent tool/runtime mitigations, but I do not see high-confidence proof that the Raspberry Pi Slack missed-pong latency regression is fixed. The model-discovery normalization path still runs plugin hook cascades without a local cache, and runtime plugin loading remains reachable from inbound reply handling.

## What This Changes

The issue asks OpenClaw to prevent expensive plugin/runtime/provider/model discovery work from blocking Raspberry Pi Slack Socket Mode message handling after the 2026.4.21 release line.

## Best Possible Solution

Keep this as Slack/Pi evidence in the gateway performance cluster, profile a warm Slack Socket Mode turn with startup and event-loop diagnostics, then move any confirmed synchronous plugin/model/tool discovery off the hot path or behind a bounded cache.

## Reproduction Assessment

Yes: the report includes concrete hardware, versions, channel, model, missed-pong logs, comparative traces, and a workaround trace. I did not live-reproduce it on Raspberry Pi hardware in this read-only review.

## Solution Assessment

Unclear: the skipPiDiscovery fast path and current tool/runtime caching are plausible partial fixes, but they do not prove the broader post-2026.4.21 Pi Slack regression is gone. A profiling-backed narrow fix is safer than closing from adjacent mitigations.
