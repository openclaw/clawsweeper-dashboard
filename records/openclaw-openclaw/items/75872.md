The fast embedded model-resolution-first path is present, but the related normalization-cache mitigation discussed in the thread is not on current main, and runtime/plugin loading remains on Slack message dispatch paths enough that this should stay with the active gateway performance cluster.

## What This Changes

Investigate Raspberry Pi Slack gateway latency after 2026.4.21, especially model/plugin discovery and runtime loading that can block Socket Mode pongs during message handling.

## Best Possible Solution

Keep the issue open as Slack/Pi evidence in the gateway performance cluster, preserve the existing fast model-resolution path, and profile the remaining warm-message runtime/plugin/model-discovery work before moving it to cached startup/background paths or a narrow low-resource gateway mode.
