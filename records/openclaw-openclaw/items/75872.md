## Summary

Keep open: current main proves the fast embedded model-resolution path, but it does not prove the broader post-2026.4.21 Raspberry Pi Slack gateway latency regression is fixed. The remaining suspected runtime/plugin loading pressure is still plausible on Slack message paths and needs profiling before a safe code change can be chosen.

## What This Changes

The issue reports multi-minute Slack Socket Mode replies and missed pongs on Raspberry Pi gateways after 2026.4.21, likely tied to expensive runtime/plugin/model discovery work on the message path.

## Best Possible Solution

Treat this as a focused performance investigation: profile warm Slack Socket Mode messages on Raspberry Pi-class Linux, keep the existing fast model-resolution path, and move any remaining expensive runtime/plugin/model discovery off normal message handling through caching, startup/background work, or a narrow low-resource gateway mode.

## Reproduction Assessment

Yes: the issue body provides a high-confidence reproduction setup, version comparison, logs, and timing traces. I did not live-reproduce it on Raspberry Pi hardware during this read-only review.

## Solution Assessment

Unclear: the current fast model-resolution path is the right solution for the models.json delay, but it is not enough evidence for the broader post-2026.4.21 runtime/plugin latency. The safer fix is profiling-backed removal of only the remaining synchronous hot-path work.
