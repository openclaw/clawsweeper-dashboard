## Summary

Keep open: the report is fresh, detailed, and not proven fixed by current main. Current main has the fast embedded model-resolution path that avoids `models.json` generation when dynamic resolution succeeds, but the broader post-2026.4.21 Pi Slack latency and runtime/plugin loading pressure is not tied to a shipped or current-main fix.

## What This Changes

The issue reports a Raspberry Pi Slack gateway latency regression after OpenClaw 2026.4.21, where gateway/plugin/model-discovery work appears to block Slack Socket Mode responsiveness and cause multi-minute replies.

## Best Possible Solution

Keep the issue open and treat it as a focused performance investigation: preserve the existing fast model-resolution path, identify any remaining Slack message-path runtime/plugin/model discovery work that can block the event loop on low-resource Linux hosts, and move that work to cached startup/background paths or a narrow low-resource gateway mode with regression coverage.

## Reproduction Assessment

Yes: the report includes concrete setup steps, version comparison, missed Slack pong logs, timing traces around `ensureOpenClawModelsJson(...)`, and a narrow workaround trace. I did not live-reproduce it on Raspberry Pi hardware during this read-only review.

## Solution Assessment

Unclear: the fast model-resolution-first path is already the right direction for the `models.json` part, but it is not sufficient evidence for the broader post-2026.4.21 runtime-root/mirror latency. A safer fix is to profile the Slack hot path and remove only the remaining synchronous plugin/runtime/model discovery work from normal message handling.
