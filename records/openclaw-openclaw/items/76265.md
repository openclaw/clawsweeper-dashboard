## Summary

Keep open: current main still routes isolated cron execution from the unresolved canonical provider, so `anthropic/*` plus `agentRuntime.id: "claude-cli"` can take the embedded Anthropic path while the normal command/chat paths already resolve the CLI runtime first.

## What This Changes

Fix isolated cron agent-turn execution so canonical `anthropic/*` model refs honor `agentRuntime.id: "claude-cli"` instead of routing to the embedded Anthropic API.

## Best Possible Solution

Resolve the execution provider in `createCronPromptExecutor` before CLI detection, pass the resolved provider through CLI session lookup and `runCliAgent`, and add a focused cron regression test for canonical Anthropic refs with `agentRuntime.id: "claude-cli"`.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: with provider `anthropic` and configured runtime `claude-cli`, current isolated cron checks `isCliProvider("anthropic")` and falls through to embedded execution, while the existing resolver would return `claude-cli`.

## Solution Assessment

Yes. Reusing `resolveCliRuntimeExecutionProvider` is the narrowest maintainable fix because it aligns isolated cron with existing command and chat execution without adding a new mapping contract.
