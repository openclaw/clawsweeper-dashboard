## Summary

Keep open: current main still makes the isolated cron executor decide CLI versus embedded execution from the unresolved canonical provider, so `anthropic/*` with `agentRuntime.id: "claude-cli"` can bypass the configured Claude CLI runtime while normal agent turns already resolve that mapping.

## What This Changes

The issue reports that isolated cron execution routes canonical Anthropic Claude models through the embedded Anthropic API instead of the configured Claude CLI runtime.

## Best Possible Solution

Teach isolated cron to resolve the effective CLI runtime provider before CLI detection and session lookup, while preserving canonical provider/model selection for embedded execution and telemetry.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: with `providerOverride` resolved to `anthropic` and config setting `agentRuntime.id: "claude-cli"`, current isolated cron calls `isCliProvider("anthropic")` and falls into the embedded path, while the existing resolver would return `claude-cli`.

## Solution Assessment

Yes. Reusing `resolveCliRuntimeExecutionProvider` in the isolated cron executor is the narrowest maintainable fix because it aligns cron with the already-covered command and auto-reply paths without adding a new model/runtime contract.
