## Summary

Keep open: current main still has the isolated cron executor decide CLI vs embedded execution from the canonical provider alone, so `anthropic/*` with `agentRuntime.id: "claude-cli"` can bypass the CLI runtime even though normal agent turns already resolve that mapping.

## What This Changes

The issue reports that isolated cron execution routes `anthropic/claude-*` jobs through the embedded Anthropic Messages API instead of the configured `claude-cli` runtime when `agentRuntime.id` is `claude-cli`.

## Best Possible Solution

Teach isolated cron to resolve the execution provider through `resolveCliRuntimeExecutionProvider` before CLI detection, then cover canonical `anthropic/*` plus `agentRuntime.id: "claude-cli"` with a focused cron test.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: with provider `anthropic` and configured runtime `claude-cli`, the current cron callback checks `isCliProvider("anthropic")` and takes the embedded path, while the existing resolver would return `claude-cli`.

## Solution Assessment

Yes. The reporter's proposed direction mirrors the already-tested chat/command path and is the narrowest maintainable fix if scoped to isolated cron execution-provider selection.
