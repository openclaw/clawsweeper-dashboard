## Summary

Keep open: current main still routes isolated cron CLI detection from the unresolved canonical provider, so `anthropic/*` plus `agentRuntime.id: "claude-cli"` can fall through to embedded Anthropic execution while the command/chat paths already resolve the CLI runtime mapping.

## What This Changes

The issue asks OpenClaw to make isolated cron runs honor a configured Claude CLI agent runtime when the cron payload uses canonical `anthropic/claude-*` model refs.

## Best Possible Solution

Align isolated cron with command and auto-reply execution by resolving the execution provider before CLI detection, while preserving canonical model/provider persistence and fallback behavior.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: with provider `anthropic` and `agentRuntime.id: "claude-cli"`, current isolated cron checks `isCliProvider("anthropic")` and takes the embedded branch, while the existing resolver would return `claude-cli`.

## Solution Assessment

Yes. Reusing `resolveCliRuntimeExecutionProvider` before isolated cron's CLI branch is the narrowest maintainable fix because it applies the existing runtime contract instead of adding a new cron-specific mapping.
