## Summary

Keep open. The report is recent, specific, and still matches current main: the macOS companion chat path still hosts the shared native SwiftUI chat view, and that view still binds `scrollPosition` while repeatedly writing the bottom sentinel during load, sends, message changes, pending runs, and streaming. I found no current-main source, docs, changelog, or linked PR evidence that fixes scrollbar-thumb dragging hangs; the closest related Control UI scroll PR was web-only and closed unmerged.

## What This Changes

The issue reports that dragging the macOS companion app chat scrollbar thumb rapidly in v2026.4.22 freezes the UI and requires force-quit.

## Best Possible Solution

Fix the native macOS `OpenClawChatView` scroll behavior so programmatic bottom-follow does not fight active user scrollbar-thumb interaction, while preserving bottom-follow for initial load, sending, newly appended user messages, pending runs, and streaming assistant output. If SwiftUI `scrollPosition` remains unstable under thumb dragging, use a macOS-specific `NSScrollView`-backed seam rather than broad shared-chat churn.

## Reproduction Assessment

Yes for triage: the issue gives concrete steps against v2026.4.22 on macOS 26.4.1, and current main still uses the same native SwiftUI scroll architecture. This review did not independently live-reproduce the freeze on macOS.

## Solution Assessment

Not applicable as a patch review because there is no proposed fix. The narrowest maintainable direction is a macOS-aware guard around bottom-pinning during active user scroll interaction, with an `NSScrollView` replacement only if the SwiftUI binding remains the source of the hang.
