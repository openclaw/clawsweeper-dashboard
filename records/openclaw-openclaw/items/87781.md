## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Scope response deltas before refreshing this turn:** `extensions/codex/src/app-server/run-attempt.ts:1281-1294`
  - body: When multiple Codex runs share the leased app-server client, `CodexAppServerClient` broadcasts every notification to every active handler, and unscoped `response.*.delta` notifications have no thread/turn id for `isCodexNotificationOutsideActiveRun` to filter. This new `|| isNativeResponseStreamDelta` path can let another run's stream keep a wedged turn alive and prevent completion-stall recovery; only count these deltas after associating them with the active turn or a turn-owned response/item id.
  - confidence: 0.88
- **[P2] Only clear bindings owned by the timed-out turn:** `extensions/codex/src/app-server/run-attempt.ts:1900`
  - body: Thread lifecycle intentionally starts transient threads while preserving an existing binding when native tools are disabled or dynamic tools are temporarily unavailable. This unconditional clear deletes the previous valid binding even when the timed-out transient thread was never written as the session binding, so the next normal turn loses native thread continuity; clear only when the binding points at the timed-out thread or when this attempt wrote/resumed it.
  - confidence: 0.9
- **[P2] Align the timeout default across docs and metadata:** `docs/plugins/codex-harness.md:540`
  - body: The runtime now defaults `postToolRawAssistantCompletionIdleTimeoutMs` to at least `5 * 60_000`, but this docs row says `60000` and the plugin metadata still says it falls back to the assistant completion idle timeout. That leaves the public docs and settings/help surface out of sync with the behavior operators will get during upgrades.
  - confidence: 0.94
