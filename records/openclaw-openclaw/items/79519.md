## Summary

Keep open. The provider wildcard direction is useful and the earlier empty-catalog/exact-row issues appear addressed, but the latest PR head still introduces a blocking `models.list` responsiveness regression by bypassing the existing read-only catalog timeout for wildcard configs.

## What This Changes

Adds `provider/*` entries to `agents.defaults.models` and wires model visibility through shared selection, Gateway `models.list`, slash commands, Discord pickers, docs, changelog, and focused tests.

## Best Possible Solution

Land provider wildcard visibility after preserving the non-`all` `models.list` read-only/timeout behavior and keeping the existing edge-case coverage for empty wildcards and mixed exact rows.

## Reproduction Assessment

Yes, for the PR defect by source inspection. Configure `agents.defaults.models` with a `provider/*` entry and call default/configured `models.list` while full catalog discovery is slow or stalled; PR head selects the unbounded `readOnly: false` path instead of the existing 750ms fallback.

## Solution Assessment

No as submitted. The `provider/*` contract is a reasonable compact fix, but the implementation should materialize wildcard provider visibility without making picker-sized `models.list` calls wait on full discovery.
