## Summary

Keep open: the PR addresses a real provider-scoped model visibility gap and includes sufficient Gateway live-output proof, but the latest head still has correctness blockers that can broaden a configured wildcard policy outside the selected providers.

## What This Changes

The PR adds literal `provider/*` entries to `agents.defaults.models` and wires that visibility through shared model selection, Gateway `models.list`, `/models`, Discord picker startup, docs, changelog, and focused tests.

## Best Possible Solution

Enforce provider-scoped visibility consistently across every model catalog consumer, or choose a Discord picker pagination/search fix instead of adding partial config semantics.

## Reproduction Assessment

Yes, source-reproducible: current main has exact allowlists or broad discovery only, while the PR-head blockers can be exercised with `buildAllowedModelSet` for wildcard-only empty catalogs and mixed wildcard/exact defaults outside policy.

## Solution Assessment

No as submitted: extending the existing model map with literal `provider/*` is a plausible narrow direction, but the patch must materialize wildcard restrictions for all consumers and avoid legacy default inclusion once a wildcard policy is present.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve wildcard restrictions when no rows are materialized:** `src/agents/model-selection-shared.ts:812-813`
  - body: When a config contains only `provider/*` entries and the incoming catalog has no rows for those providers, this branch can return `allowAny: false` with an empty `allowedCatalog`. Existing consumers such as `src/tui/embedded-backend.ts` treat an empty allowed catalog as unrestricted and fall back to the full catalog, so a configured provider filter can leak every provider.
  - confidence: 0.91
- **[P2] Do not re-add defaults outside mixed wildcard policies:** `src/agents/model-selection-shared.ts:790-791`
  - body: In a mixed allowlist such as `openai-codex/*` plus `vllm/qwen-local`, any `defaultKey` is still added because `exactModelRefs.length > 0`, even when the default provider is neither wildcarded nor exact-listed. That widens the provider-scoped policy the PR is adding; preserve legacy default inclusion only for exact-only policies or explicit default coverage.
  - confidence: 0.86
