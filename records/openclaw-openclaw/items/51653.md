## Summary

Keep open for maintainer handling. The PR targets a real metadata-leak class, but the proposed payload-plan change is not mergeable as-is because it applies the broad inbound display sanitizer to all outbound text and would alter legitimate timestamp-prefixed replies; current main also has later assistant/replay sanitization that may supersede much of the original need.

## What This Changes

The PR imports `stripInboundMetadata` into outbound payload normalization, wraps formatted delivery text with it, and adds a regression test for Conversation/Sender metadata blocks.

## Best Possible Solution

Keep the assistant/replay sanitizer that already exists on main, then add a narrow outbound-only sanitizer only if maintainers want defense-in-depth at the payload-plan boundary. That sanitizer should remove generated metadata sentinel blocks and terminal untrusted-context suffixes without stripping legitimate timestamp-prefixed reply text, and the remaining work should be consolidated with #39847/#50520.

## Reproduction Assessment

Yes. The metadata-leak scenario is represented by the PR test, and the blocking regression is reproducible by passing a timestamp-prefixed payload through the PR branch because `stripInboundMetadata()` strips timestamp prefixes by design.

## Solution Assessment

No. Applying the inbound display sanitizer at the shared outbound plan boundary is broader than necessary; a safer fix would be an outbound-specific copied-metadata sanitizer that preserves ordinary assistant content such as logs and transcripts.
