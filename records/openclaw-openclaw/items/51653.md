## Summary

Keep this PR open, but do not merge the current patch as-is. Current main still lacks shared outbound stripping for copied inbound metadata, so the underlying bug remains valid, but this PR applies the broader inbound UI sanitizer to every outbound assistant reply and can remove legitimate leading timestamp text.

## What This Changes

The PR imports `stripInboundMetadata` into outbound payload normalization, wraps formatted reply text with it, and adds a regression test for stripping Conversation info and Sender metadata blocks before delivery.

## Best Possible Solution

Add an outbound-safe sanitizer at the shared outbound payload plan boundary that removes only copied inbound metadata sentinel blocks and terminal untrusted-context suffixes needed for delivery safety, while preserving legitimate assistant text such as timestamp-prefixed logs. Cover both stripping and timestamp preservation in `src/infra/outbound/payloads.test.ts`, then consolidate this work with #39847/#50520.

## Reproduction Assessment

Yes. A metadata-prefixed reply payload can be traced through current `createOutboundPayloadPlan()` and delivery projection without a sanitizer, and the PR regression is supported by existing `stripInboundMetadata()` tests that assert leading timestamp prefixes are removed.

## Solution Assessment

No. The current PR chooses the broad inbound display sanitizer, but the narrower maintainable fix is an outbound-only metadata block sanitizer that does not apply inbound timestamp cleanup to assistant-authored output.
