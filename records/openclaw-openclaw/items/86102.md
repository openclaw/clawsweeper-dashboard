## What This Changes

Request `openclaw doctor` to warn and continue when the cron job store exists but is unreadable by the runtime user.

## Best Possible Solution

Handle cron-store read/access failures inside the doctor cron health path by emitting a clear warning with the store path and continuing, while keeping runtime cron store loading strict for the scheduler.

Yes from source inspection: an unreadable existing cron store causes `loadCronStore` to rethrow, and the legacy cron doctor contribution is awaited without a catch. I did not execute a live permission-based repro in this read-only review.

## Solution Assessment

Yes, warning and returning from the doctor cron check is the narrow maintainable fix. The safer boundary is to avoid changing scheduler/runtime `loadCronStore` semantics and keep the non-fatal behavior specific to doctor diagnostics.
