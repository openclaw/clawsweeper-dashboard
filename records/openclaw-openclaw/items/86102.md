## Summary

Keep open: current main still lets unreadable cron-store read errors escape the doctor cron check, so `openclaw doctor` can abort before later diagnostics run. This is a narrow, source-backed doctor resilience bug with no duplicate or merged fix found.

## What This Changes

Request `openclaw doctor` to warn and continue when the cron job store exists but cannot be read.

## Best Possible Solution

Keep cron runtime loading strict, but make the doctor cron health path catch cron-store read/access failures, report a clear warning with the store path, and continue later checks.

## Reproduction Assessment

Yes from source inspection: an unreadable existing cron store causes `loadCronStore` to rethrow, and the legacy cron doctor contribution awaits that call without a catch. I did not execute a chmod-based live reproduction in this read-only review.

## Solution Assessment

Yes, warning and returning from the doctor cron check is the narrow maintainable fix. The safer boundary is to avoid changing scheduler/runtime `loadCronStore` semantics and keep the non-fatal behavior specific to doctor diagnostics.
