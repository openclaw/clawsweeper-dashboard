## Summary

Keep open. Current main and the latest release still have the reported Android foreground-service crash path: the Android app targets SDK 36, declares and starts a dataSync foreground service, and calls ServiceCompat.startForeground without handling Android 15+ quota exhaustion or timeout callbacks. The report is concrete, matches Android's documented exception behavior, and I found no merged fix or linked open PR that supersedes it.

## What This Changes

Prevent the Android companion app from crashing when NodeForegroundService hits Android 15/16 dataSync foreground-service quota and startForeground throws ForegroundServiceStartNotAllowedException.

## Best Possible Solution

Harden the Android service lifecycle so quota exhaustion cannot crash the app: handle startForeground denial, implement Android 15+ onTimeout handling, stop or demote the service cleanly, avoid restart loops, and document any remaining background-connection limitation.

## Reproduction Assessment

Yes. The high-confidence reproduction path is targetSdk 36 plus dataSync foreground-service quota exhaustion followed by NodeForegroundService.onCreate calling unguarded ServiceCompat.startForeground; live proof should use Android 15/16 with FGS_INTRODUCE_TIME_LIMITS and a reduced data_sync_fgs_timeout_duration.

## Solution Assessment

Yes for the fix direction. The narrow maintainable solution belongs in the Android NodeForegroundService and voice-mode lifecycle, with tests and device validation, rather than changing gateway connectivity code or hiding the crash by lowering SDK or removing service declarations.
