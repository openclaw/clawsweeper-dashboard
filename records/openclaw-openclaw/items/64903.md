## Summary

Keep open. Current `main` and the latest release still use a targetSdk 36 `dataSync` foreground service and call `ServiceCompat.startForeground` from `NodeForegroundService` without denial handling or `Service.onTimeout`, matching the reported Android 15/16 quota crash path.

## What This Changes

The issue asks OpenClaw to stop the Android companion app from crashing when `NodeForegroundService` starts after the `dataSync` foreground-service quota is exhausted.

## Best Possible Solution

Harden `NodeForegroundService` so Android 15/16 `dataSync` quota exhaustion stops or demotes the service without crashing or restart loops, with targeted tests and device/emulator proof.

## Reproduction Assessment

Yes, at source level. The current app targets SDK 36, declares/runs a `dataSync` foreground service, and calls `ServiceCompat.startForeground` without handling the documented quota-denial exception; a live Android 15/16 quota test is still needed for runtime proof.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the Android foreground-service lifecycle: handle `startForeground` denial, implement Android 15+ `Service.onTimeout`, stop or demote cleanly, and avoid restart loops rather than changing unrelated gateway behavior.
