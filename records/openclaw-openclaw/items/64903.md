## Summary

Keep open. Current main still starts the Android node as a targetSdk 36 `dataSync` foreground service and calls `ServiceCompat.startForeground` from `NodeForegroundService` without handling quota-denial or Android 15+ timeout callbacks, so the reported crash path remains live.

## What This Changes

The issue requests hardening the Android companion app so `NodeForegroundService` does not crash when Android denies `startForeground` after the `dataSync` foreground-service quota is exhausted.

## Best Possible Solution

Update the Android foreground-service lifecycle so quota exhaustion cannot crash the app: handle `startForeground` denial, implement Android 15+ `Service.onTimeout`, stop or demote the service cleanly, avoid restart loops, and add targeted tests plus Android 15/16 runtime proof.

## Reproduction Assessment

Yes. The issue has a high-confidence source-level reproduction path: target SDK 36, a `dataSync` foreground-service type, exhausted `dataSync` quota, and an unguarded `ServiceCompat.startForeground` call during `NodeForegroundService.onCreate`; final proof should use Android 15/16 timeout controls on a device or emulator.

## Solution Assessment

Yes on the fix area, with one policy detail still to choose. The narrow maintainable solution belongs in `NodeForegroundService` lifecycle handling and tests, with docs or UX updates only if Android background connection behavior must change.
