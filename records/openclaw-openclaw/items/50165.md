## Label Justifications

- `P2`: This is a normal-priority lifecycle/status correctness bug with meaningful workflow impact but no confirmed crash loop, security bypass, or unusable core runtime.
- `impact:session-state`: The report is about parent/subagent task state reaching a terminal-looking status while child-owned work can still be running.
- `impact:message-loss`: False completion can cause users or parent agents to stop waiting, miss the actual result, or report prematurely.

## Summary

Keep this open: current main still has a source-reproducible lifecycle gap where subagent completion follows the child agent-run result while child-started background exec/process work remains on a separate liveness surface. An overlapping open PR targets the likely implementation direction, but it is still open, conflicting, and blocked on proof/review issues.

## What This Changes

The issue asks OpenClaw to keep subagent/session state non-terminal until delegated background work or user-relevant output is actually complete.

## Best Possible Solution

Define one lifecycle contract, then either surface child-owned background processes as separate active work in `/tasks` and `session_status`, or defer subagent finalization through a generic owned-work liveness seam with explicit timeout, cancellation, cleanup, and regression coverage.

## Reproduction Assessment

Yes at source level, but not as a live harness reproduction. A child subagent can background an exec process that returns status:"running" and a process sessionId, while the subagent task finalizes from the child agent-run outcome without checking that process session.

## Solution Assessment

Unclear until maintainers decide the lifecycle contract. A generic owned-work liveness model or explicit active CLI task surfacing is safer than a narrow subagent-only wait hack.
