## Label Justifications

- `P2`: This is a normal-priority lifecycle/status correctness bug with workflow impact but no confirmed crash loop, security bypass, or unusable core runtime.
- `impact:session-state`: The reported failure is inaccurate parent/subagent task state reaching a terminal-looking status while child-owned work can still be running.
- `impact:message-loss`: False completion can cause users or parent agents to stop waiting, miss the actual result, or report prematurely.

## Summary

Keep this open: current main still finalizes subagent task state from the child agent-run outcome while child-owned background exec/process sessions live in a separate registry, and the related background-exec liveness PR is open with unresolved review/proof blockers.

## What This Changes

The issue asks OpenClaw to avoid showing delegated subagent work as completed until the real user-relevant background work is actually finished.

## Best Possible Solution

Define one lifecycle contract, then either gate subagent finalization through a generic owned-work liveness seam or make child-owned background processes first-class active work in task/status views without pretending the whole subagent is done.

## Reproduction Assessment

Yes at source level, but not as a live harness reproduction. A child subagent can start a background exec that returns status="running" and a process sessionId, then the subagent run can be finalized from the agent-run outcome without checking that process session.

## Solution Assessment

Unclear because no accepted patch is available and the correct fix depends on a lifecycle contract decision. A coherent owned-work liveness model or explicit active-work surfacing is safer than a narrow subagent-only wait hack.
