- `P2`: The issue is a normal-priority lifecycle/status correctness bug with real workflow impact but no current crash, security bypass, or high-confidence live repro in this review.
- `impact:session-state`: The report is about parent/subagent session state showing terminal completion while child-owned work may still be running.
- `impact:message-loss`: False completion can cause parent agents or users to stop waiting, miss follow-up delivery, or report results before the user-relevant output is ready.

## Summary

Keep this open. Current main and the latest shipped release still split subagent completion from child-owned background exec/process liveness; the backup workaround discussion does not address the OpenClaw lifecycle-state gap, and the open exec-liveness PR is only a partial, unmerged path.

## What This Changes

The issue asks for subagent/task status to remain in progress until delegated user-relevant work, including background exec/process work, is actually finished.

## Best Possible Solution

Define one lifecycle contract, then either surface child-owned background exec/process sessions as separate active work in task/status views or gate subagent finalization through a generic owned-work liveness seam with timeout, cancellation, cleanup, and regression coverage.

## Reproduction Assessment

Yes at source level, but not as a live harness reproduction. The code path is clear: a subagent can start a background exec, receive a running process session id, finish its agent run, and have the subagent task finalized without checking that process session.

## Solution Assessment

Unclear because no patch is proposed and the correct behavior is a runtime contract decision. A coherent owned-work liveness model or separate active-work surfacing is safer than a narrow subagent-only wait hack.
