## Label Justifications

- `P2`: The issue describes a credible subagent lifecycle reliability gap with limited blast radius and a product-contract blocker rather than an urgent outage.
- `impact:session-state`: The reported false completion means parent/subagent task state can become terminal while delegated child-owned work may still be active.
- `impact:message-loss`: False completion can make users or parent agents stop monitoring before the final result or follow-up output is actually ready.

## Summary

Keep open: current main has descendant-subagent announce gating, task status visibility, and a narrower Codex-native premature-success fix, but the generic subagent path can still finalize from the child agent-run outcome while child-owned background exec/process work is tracked separately and may outlive that turn.

## What This Changes

The issue asks for subagent/session/task status to remain in progress until delegated work and user-relevant side effects are actually complete.

## Best Possible Solution

Define the lifecycle contract explicitly, then implement one coherent path: either keep subagent completion agent-run authoritative and surface child-owned process liveness as separate active work, or defer subagent finalization through a generic owned-process liveness seam with timeout, cancellation, cleanup, and regression coverage.

## Reproduction Assessment

No, not with high confidence from a controlled run. The live issue has behavioral reports and current source shows a credible path where a subagent can complete from the agent-run result while background exec/process work remains in a separate registry.

## Solution Assessment

Unclear. Current main handles descendant subagent and Codex-native completion ordering cases, but the remaining generic background-process semantics need maintainers to choose whether those processes block subagent completion or remain separately visible active work.
