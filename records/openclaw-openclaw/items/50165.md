## Label Justifications

- `P2`: This is a normal-priority lifecycle/status correctness bug with workflow impact but no crash, security bypass, or live reproduction in this review.
- `impact:session-state`: The report is about parent/subagent session state reaching a terminal-looking status while child-owned work can still be running.
- `impact:message-loss`: False completion can cause users or parent agents to stop waiting and miss the user-relevant result or follow-up delivery.

## Summary

Keep this open: current main still finalizes subagent runs and task state from the child agent-run result while child-owned background exec/process sessions remain in a separate process registry, and the related fix PR is open but not merged.

## What This Changes

Ensure subagent completion/status does not appear terminal before delegated background work or its user-relevant result is actually finished.

## Best Possible Solution

Define one lifecycle contract, then either surface child-owned background processes as active work in task/status views or gate subagent finalization through a generic owned-work liveness seam with timeout, cancellation, cleanup, and regression coverage.

## Reproduction Assessment

Yes at source level, but not as a live harness reproduction. A child subagent can start a background exec that returns status="running" and a process sessionId, then the subagent run can be finalized from the agent-run outcome without checking that process session.

## Solution Assessment

Unclear because no patch is proposed and the correct behavior is a runtime contract decision. A coherent owned-work liveness model or separate active-work surfacing is safer than a narrow subagent-only wait hack.
