## Label Justifications

- `P2`: This is a normal-priority lifecycle/status correctness bug with workflow impact but no confirmed crash loop, security bypass, or unusable core runtime.
- `impact:session-state`: The reported failure is inaccurate parent/subagent task state reaching a terminal-looking status while child-owned work can still be running.
- `impact:message-loss`: False completion can cause users or parent agents to stop waiting, miss the actual result, or report prematurely.

## Summary

Keep this open: current main still makes the subagent task look terminal from the child agent-run outcome, while plain background exec/process sessions remain a separate liveness surface. The overlapping background-exec liveness PR is still open and not a merged implementation path.

## What This Changes

The issue reports that subagent lifecycle/status can show completion while child-owned delegated background work is still running.

## Best Possible Solution

Define one lifecycle contract and implement it coherently through either owned-work liveness gating or first-class child CLI task/status surfacing, with regression coverage across subagent finalization, background exec, process removal, and session_status.

## Reproduction Assessment

Yes at source level, but not as a live harness reproduction. A child subagent can background an exec process that returns status:"running" and a process sessionId, while the subagent task finalizes from the child agent-run outcome without checking that process session.

## Solution Assessment

Unclear until maintainers decide the lifecycle contract. A generic owned-work liveness model or explicit active CLI task surfacing is safer than a narrow subagent-only wait hack.
