## Label Justifications

- `P2`: The issue describes a real subagent lifecycle reliability bug with bounded blast radius and a product-contract blocker.
- `impact:session-state`: The reported false completion means parent/subagent state can disagree with whether delegated work is still active.
- `impact:message-loss`: False completion can cause users or parent agents to stop monitoring before final results or follow-up messages are ready.

## Summary

Keep open: current main and v2026.6.10 still allow subagent terminal state to follow the child agent-run result while child-owned background exec/process sessions are tracked separately, so the reported false-done lifecycle contract is not fully solved.

## What This Changes

The issue reports that subagent completion/status can appear finished before delegated background work or user-relevant output is actually done.

## Best Possible Solution

Define whether child-owned background processes are part of subagent completion or separate active work, then make subagent status, task status, and process visibility follow that one lifecycle contract.

## Reproduction Assessment

Yes at source level, but not as a live run: current code can finalize the subagent from the agent-run wait outcome while scoped background exec/process sessions remain separate. I did not run a live Windows or Linux reproduction in this read-only review.

## Solution Assessment

No: current fixes handle descendant subagent announce timing and active process references in prompts, but they do not settle the background-process completion contract. The narrowest maintainable solution needs a maintainer decision on that contract first.
