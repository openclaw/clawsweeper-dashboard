Keep open: the report matches a credible current source-level SecretRef startup gap for the externalized Discord channel, and the central Discord/token and channel-secret collector paths have not changed since v2026.5.2.

## What This Changes

Discord startup should accept a SecretRef-backed `channels.discord.token` in `@openclaw/discord` instead of failing before gateway readiness.

## Best Possible Solution

Materialize installed/external channel plugin SecretRefs into the active runtime snapshot before channel startup, with a Discord regression and changelog entry.

## Solution Assessment

Yes, the best fix is the generic runtime-snapshot path for installed channel plugin secret contracts; a Discord-only gentle-token fallback can be a defensive backstop, but it should not replace startup materialization.

## Repair Work Prompt

Fix the Discord external channel startup SecretRef regression reported in https://github.com/openclaw/openclaw/issues/76371. The expected boundary is that `channels.discord.token` and account token SecretRefs from installed/external channel plugins are materialized into the active runtime snapshot before `startAccount`/`resolveDiscordAccount` reads credentials. Prefer a generic channel-secret-contract path for installed channel plugins over a Discord-only hardcode; add a Discord-specific regression for an env SecretRef token on startup. A defensive Discord resolver change is acceptable only if it preserves the contract that runtime credential consumers should normally receive a resolved snapshot. Include a user-facing `CHANGELOG.md` fix entry. Do not add a new config option, do not persist resolved plaintext secrets, and do not weaken unsupported SecretRef surface policy.
