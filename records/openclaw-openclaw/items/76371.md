## Summary

Keep open. The report matches a current source-level startup gap for packaged/externalized Discord: the Discord token resolver still throws on unresolved SecretRefs, while startup secret collection only sees bundled/bootstrap channel secret contracts and the core package excludes the Discord dist tree needed for that bundled fallback.

## What This Changes

The issue reports that `@openclaw/discord@2026.5.2` crashes during channel startup when `channels.discord.token` is an env-backed SecretRef that remains unresolved.

## Best Possible Solution

Make external channel secret contracts available to startup secret resolution before channel startup, then add a regression proving an externalized Discord env SecretRef becomes a string before `resolveDiscordAccount` runs.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path for a packaged install where Discord is externalized: `channels.discord.token` remains a SecretRef because the bundled contract fallback is unavailable, then `resolveDiscordAccount` calls the strict token resolver during `startChannel`.

## Solution Assessment

Yes. The best fix is at the startup secret-contract boundary for externalized channel plugins; changing Discord token normalization alone would be a useful defensive backstop but would leave the same class of externalized-channel startup bug elsewhere.

## Repair Work Prompt

Fix the externalized Discord startup SecretRef regression reported at https://github.com/openclaw/openclaw/issues/76371. In packaged/external-plugin startup, `channels.discord.token` env SecretRefs must be resolved before Discord `plugin.config.resolveAccount` calls `resolveDiscordToken`; do not add a new config option or persist plaintext tokens. Prefer a generic startup secret-contract path that can load installed/external channel `secret-contract-api` or otherwise make official external channel secret contracts available before `prepareSecretsRuntimeSnapshot` finishes. Add a focused regression that simulates the packaged externalized Discord path where bundled `dist/extensions/discord` is unavailable and proves `channels.discord.token` resolves to a string before channel startup/account resolution. Consider a small Discord-local defensive guard only as a backstop, not as the sole architecture fix. Add a user-facing changelog fix entry and keep plugin wording as `plugin`, not `extension`.
