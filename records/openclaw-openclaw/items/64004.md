## Summary

Keep open. Current main still exposes the unresolved Control UI latency surface: Sessions navigation routes through the same sessions refresh path, Overview still overlaps several secondary refreshes, and the slow `node.list`, `cron.runs`, `sessions.usage`, and `usage.cost` request paths remain active without a landed click-to-render attribution or decoupling fix. The related #71631 is broader and open, but it has not merged a fix or explicitly superseded this concrete bug.

## What This Changes

The issue reports that the Control UI can remain visibly stalled after opening Sessions even when `sessions.list` returns in tens of milliseconds, with slow `node.list`, `cron.runs`, `usage.cost`, and `sessions.usage` calls observed in the same windows.

## Best Possible Solution

Keep this as a concrete Control UI performance bug until a landed change measures the full tab-click to visible-render path and prevents slow secondary usage/node/cron/status refreshes from blocking or obscuring immediate Sessions navigation feedback. A good end state would include per-action timing attribution plus lazy, cancellable, stale-guarded, or paged refreshes around the named slow surfaces.

## Reproduction Assessment

Yes for the symptom-level report: the issue includes concrete Windows Server/npm-global steps and repeated Gateway timing logs across 2026.4.9 and v2026.4.15. No for root-cause isolation yet: the checkout has source evidence but no live click-to-render trace tying the stall to a specific layer.

## Solution Assessment

Unclear because there is no proposed patch and the source still shows several plausible latency paths. The safest next solution is instrumentation first, followed by narrow decoupling or lazy/cancellable refresh changes once the blocking layer is identified.
