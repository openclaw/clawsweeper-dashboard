## Summary

Keep open. Current main still has the same unresolved Control UI latency surface: Sessions navigation still awaits the Sessions loader, Overview still launches several secondary refreshes together, and the slow RPC families named in the report remain active without a landed click-to-visible-render attribution or decoupling fix.

## What This Changes

The issue reports Control UI Sessions navigation remaining slow even when `sessions.list` itself returns quickly, with `node.list`, `cron.runs`, `sessions.usage`, and `usage.cost` showing much higher latency during the same windows.

## Best Possible Solution

Keep this as a concrete Control UI performance bug until a landed change measures the full tab-click to visible-render path and prevents slow secondary usage, node, cron, status, or debug refreshes from blocking or obscuring immediate Sessions feedback.

## Reproduction Assessment

Yes for the symptom-level report: the issue includes concrete Windows Server/npm-global steps and repeated Gateway timing logs across 2026.4.9 and v2026.4.15. No for root-cause isolation: current source inspection identifies plausible paths, but there is no live trace tying the stall to one layer.

## Solution Assessment

Unclear because there is no proposed patch and current main still shows multiple plausible latency paths. The safest direction is instrumentation first, followed by narrow lazy/cancellable/stale-guarded refresh changes once the blocking layer is known.
