## Summary

Keep open. Current main still creates and updates the native Windows Gateway Scheduled Task with plain `schtasks /Create` or `/Change` and no battery-power setting override, while Microsoft documents the relevant Task Scheduler battery defaults as stopping tasks on battery and disallowing battery starts. The report matches a narrow Windows daemon/service bug rather than an obsolete or duplicate crash report. ([learn.microsoft.com](https://learn.microsoft.com/en-us/windows/win32/taskschd/tasksettings-stopifgoingonbatteries))

## What This Changes

The issue reports that the npm-installed OpenClaw Gateway daemon exits every time a Windows 10 laptop switches from AC power to battery power.

## Best Possible Solution

Add a Windows Scheduled Task install/update repair that explicitly sets `DisallowStartIfOnBatteries=false` and `StopIfGoingOnBatteries=false` while preserving the existing user, privilege, script, fallback, and restart behavior.

## Reproduction Assessment

Yes: the focused reproduction path is a native Windows laptop with the Gateway installed as the managed Scheduled Task, then switching from AC to battery. I did not run a live battery test in this read-only review, but current code and Microsoft Task Scheduler defaults line up with the reported trigger. ([learn.microsoft.com](https://learn.microsoft.com/en-us/windows/win32/taskschd/tasksettings-stopifgoingonbatteries))

## Solution Assessment

Yes, the best fix is in the Windows daemon Scheduled Task owner path, not the generic Gateway loop. The narrow maintainable approach is to set the two Task Scheduler power flags for new and existing tasks and cover both paths with daemon tests.

Fix the Windows Gateway daemon battery-power crash reported in openclaw/openclaw#59299. Current `src/daemon/schtasks.ts` creates native Windows Gateway tasks with plain `schtasks /Create` and updates existing tasks with `/Change /TR`, which leaves Task Scheduler power defaults in place. Add focused tests in the schtasks install/update area proving both new-task creation and existing-task update/repair include the battery settings; keep fallback tests passing.
