## Label Justifications

- `impact:data-loss`: The requested editor would let users and agents modify workspace files, so conflict handling and refresh semantics can affect persisted document contents.

## Summary

Keep open: current main supports render-only `[embed]` Canvas previews, native A2UI action bridges, and a separate allowlisted Files editor, but not a bidirectional Control UI Canvas markdown editor with diff-to-agent context. The remaining work needs maintainer product and security design rather than conservative cleanup closure.

## What This Changes

Requests a live Canvas-based markdown editor in Control UI that loads workspace files, accepts user edits, refreshes after agent writes, and exposes diffs to the next agent turn.

## Best Possible Solution

Define a security-reviewed Canvas editor and agent-event contract that keeps untrusted rendering isolated, scopes file writes to approved workspace files, and connects diffs to next-turn context before implementing core UI.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request. Source inspection gives high-confidence absence proof because current main renders `[embed]` Canvas content as URL previews and lacks the requested bidirectional editor/diff path.

## Solution Assessment

Unclear as proposed. The issue's shape is plausible, but maintainers should first choose the browser-to-agent event contract, file-write boundary, and whether the permanent owner is Canvas plugin, core Control UI, or a gateway API.
