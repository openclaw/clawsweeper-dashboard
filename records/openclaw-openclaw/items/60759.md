## Summary

Keep open: current main and v2026.5.19 still have plugin config schemas with root-relative `$ref` values while the schema composer nests those schemas unchanged, so editor validation can still fail. The target-repo fix PRs found for this are closed unmerged, and the only open closing PR is in a fork rather than a mergeable target-repo PR.

## What This Changes

The issue asks for `openclaw config schema` to emit plugin schema `$defs` where `#/$defs/...` references can resolve, instead of nesting them under the qqbot config schema.

## Best Possible Solution

Land a generic schema composer fix that hoists or namespaces plugin/channel `$defs`, rewrites local root refs safely, handles collisions, and adds regression coverage for both plugin and channel schemas.

## Reproduction Assessment

Yes, at source level: current manifests still contain `$defs` with `#/$defs/...` refs, and current schema composition places those schemas under plugin/channel config nodes without hoisting or rewriting. I did not run the CLI because this review had to keep the checkout read-only.

## Solution Assessment

No landed solution exists on current main. A generic composer fix is the maintainable path; a qqbot-only manifest edit or manual `$defs` move would leave the plugin/channel schema contract fragile.
