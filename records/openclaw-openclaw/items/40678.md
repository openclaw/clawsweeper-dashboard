- `impact:session-state`: The issue is about whether a TUI session reflects activity from the same or other channel sessions.
- `impact:message-loss`: Messages and agent responses can exist in transcripts or other channels while missing from the live TUI view until reload or log tailing.

## Summary

Keep open: current main and v2026.6.8 still do not provide an opt-in TUI observe/mirror mode, and the remaining work needs product and security direction rather than cleanup closure.

## What This Changes

The issue requests an opt-in TUI observe or mirror mode for iMessage, Telegram, cron, and other channel activity while preserving TUI session isolation.

## Best Possible Solution

Define a default-off observe/mirror policy for TUI visibility, implement active-session live tailing through the existing sessions.messages.subscribe path, and keep session isolation as the default.

## Reproduction Assessment

Yes at source level for the narrower live-tail gap: Gateway and Control UI use session.message, while TUI neither subscribes to nor dispatches that event. The broad observe/mirror behavior is a feature request, not a runnable current-main bug reproduction.

## Solution Assessment

Unclear as a full feature: active-session live tailing should use sessions.messages.subscribe, but cross-channel observe/mirror needs maintainer product and security policy first.
