## Summary

Keep open: current main still has no TUI observe/mirror option, and the terminal TUI does not subscribe to or handle live `session.message` events even though the Gateway exposes that event surface. The closed related unified-sync issue points back here as the canonical tracker, so there is no duplicate/superseding item to close against.

## What This Changes

Request an opt-in TUI mode or config option that mirrors or observes iMessage, Telegram, cron, and other channel activity without reverting isolated `/new` TUI sessions.

## Best Possible Solution

Preserve `/new` isolation, then design an explicit TUI observer/live transcript path around the existing Gateway session-message subscription surface, with separate policy for read-only observation versus visible outbound cross-channel mirroring.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source-reproducible but not live-reproduced here: current main lacks the requested TUI mode and does not wire `session.message` subscriptions into the terminal TUI, while the 2026.5.3 Telegram/TUI comment provides concrete live verification steps.

## Solution Assessment

Is this the best way to solve the issue? Yes in direction: an opt-in observer/live transcript mode is safer than reverting TUI session isolation. The exact UX, config surface, and cross-channel visible-send policy still need maintainer design before automation should implement it.
