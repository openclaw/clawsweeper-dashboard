## Summary

Keep this open. Current main still has isolated TUI sessions and active-session filtering, and I did not find an opt-in TUI observe or mirror mode for other channel or session activity. The request is valid, but it needs maintainer product/security design because it would intentionally widen transcript read visibility across channels.

## What This Changes

The issue requests an opt-in TUI observe or mirror mode that can display iMessage, Telegram, cron, and sub-agent activity for the same agent without reverting TUI session isolation.

## Best Possible Solution

Add an explicit authorized TUI observe surface that keeps default `/new` isolation, separates read-only observation from interactive sending, labels channel/session/run origin, supports bounded recent previews plus live updates, and documents the privacy model.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: current main has no observe/mirror TUI option, loads history only for `state.currentSessionKey`, and ignores live chat events whose session key does not match the active TUI session.

## Solution Assessment

Unclear. The requested UX is reasonable, but the best fix is not a bare flag that widens event filters; it should be a distinct observe/read-authorization path coordinated with #41548 and #73402.
