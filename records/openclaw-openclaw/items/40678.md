## Summary

Keep open: current main still has isolated TUI sessions, single-session history loading, and active-session event filtering, with no opt-in TUI observe or mirror mode. The related broader sync issue and active-session subscription PR do not terminally resolve this read-only cross-channel visibility request, and the remaining design is privacy-sensitive.

## What This Changes

Add an opt-in TUI observe or mirror mode that shows iMessage, Telegram, cron, and other channel activity for the same agent without reverting per-TUI session isolation.

## Best Possible Solution

Add an explicit authorized TUI observe surface that keeps default `/new` isolation, separates read-only observation from interactive sending, labels channel/session/run origin, supports bounded recent previews plus live updates, and documents the privacy model.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: current main has no observe/mirror TUI option, loads history only for `state.currentSessionKey`, and ignores chat events whose session key does not match the active TUI session.

## Solution Assessment

Unclear. The requested UX is reasonable, but the best fix is not a bare flag that widens event filters; it should be a distinct observe/read-authorization path coordinated with #41548 and #73402.
