## Summary

Keep open: current main still lacks an opt-in TUI observe/mirror mode, and the latest same-session Telegram report is not superseded by a merged fix. Source inspection shows the TUI remains scoped to the active session and is not wired to the Gateway session-message subscription surface.

## What This Changes

Request an opt-in TUI mode or setting to display iMessage, Telegram, cron, and other channel activity while preserving isolated TUI sessions.

## Best Possible Solution

Keep `/new` isolation, add an explicit TUI observer or live-session subscription path, and gate visible cross-channel mirroring behind a maintainer-approved opt-in policy.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source-reproducible but not live-reproduced here: current main has no observe/mirror option, the TUI does not subscribe to `session.message`, and the May 4 Telegram/TUI comment gives a concrete live path for maintainer verification.

## Solution Assessment

Is this the best way to solve the issue? The requested opt-in observe direction is safer than reverting `/new` isolation, but the exact read-only UX, CLI/config surface, and visible cross-channel send-mirroring policy need maintainer design before implementation.
