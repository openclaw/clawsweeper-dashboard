## Label Justifications

- `P2`: This is a concrete TUI/session visibility improvement with real workflow impact, but it has a log-follow workaround and does not make core runtime unusable.

## Summary

Keep open: current main and latest release still lack an opt-in TUI observe/mirror mode, while the related active-session live-tail bug is narrower and remains tracked separately at https://github.com/openclaw/openclaw/issues/38829. The broad cross-channel visibility request needs product and security direction before implementation.

## What This Changes

Request an opt-in TUI mode or setting that mirrors or observes iMessage, Telegram, cron, and other channel activity in the terminal UI without undoing per-client TUI session isolation.

## Best Possible Solution

Define a default-off, authorization-aware TUI observe/mirror policy, land the narrower active-session live-tail fix through the existing `session.message` subscription path, and keep broad cross-channel fanout behind maintainer-approved product and security rules.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.8 expose no TUI observe/mirror option, and the current TUI does not consume the Gateway `session.message` event used for live transcript updates. I did not run a live Telegram/iMessage-to-TUI reproduction in this read-only review.

## Solution Assessment

Unclear as a full feature: the narrow active-session live-tail fix should use `sessions.messages.subscribe`, but cross-channel observe/mirror behavior needs an explicit product and security policy before adding a CLI flag or config knob.
