## Summary

Keep open: current main still has a source-backed Brave web_search cancellation gap that fits the reported Gateway timeout/hang class. The parser-runaway theory from the later comment is not supported by current Brave source because the response parser is called once and there is no retry loop.

## What This Changes

The issue reports Gateway hangs and CPU saturation after configuring Brave-backed web_search on OpenClaw 2026.5.2.

## Best Possible Solution

Brave should honor the existing managed web_search cancellation contract in both web and llm-context modes, with regression tests at the provider boundary.

## Reproduction Assessment

No, not as a high-confidence live Gateway hang. Source inspection does show a current-main reproduction of the cancellation gap: the agent signal reaches the provider contract, then Brave drops it before guarded fetch.

## Solution Assessment

Yes for the actionable defect: forwarding the existing AbortSignal through Brave is the narrowest maintainable fix. Content-type validation can be considered as follow-up hardening, but switching providers or adding a circuit breaker is not source-backed as the first repair.
