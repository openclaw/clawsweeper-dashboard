## Summary

Keep open: current main still drops the managed web_search AbortSignal at the Brave provider boundary, which plausibly fits the reported Gateway timeout/hang class and is not already implemented or superseded. The parser-runaway comment was considered, but current Brave source shows single response parsing and no retry loop, so replacing Brave or adding a circuit breaker is not the narrow first fix.

## What This Changes

The issue reports Gateway/Discord hangs and CPU saturation after configuring Brave-backed web_search on OpenClaw 2026.5.2.

## Best Possible Solution

Propagate the existing web_search AbortSignal through the Brave provider and both Brave endpoint modes, then add focused Brave cancellation regression coverage without adding a new provider or configuration knob.

## Reproduction Assessment

No; there is no live log or deterministic failing Brave response, but source inspection gives a clear current-main reproduction of the cancellation gap from agent tool to Brave fetch.

## Solution Assessment

Yes for the actionable defect: forward the existing provider execution signal through Brave's runtime. Content-type validation can be considered as local hardening later, but replacing Brave or adding a circuit breaker is not source-backed as the first fix.
