## Summary

Keep open: current main still drops the managed web_search abort signal at the Brave provider boundary, which fits the reported Gateway timeout/hang class even though the exact CPU saturation has not been reproduced. The later parser-runaway theory is not supported by the current Brave source because there is no parser retry loop.

## What This Changes

The issue reports OpenClaw 2026.5.2 hanging and restarting after Brave web_search configuration on Ubuntu, with removing Brave config restoring TUI and channel behavior.

## Best Possible Solution

Propagate the existing web_search AbortSignal through the Brave provider and both Brave endpoint modes, then add focused Brave cancellation regression coverage without adding a new provider or configuration knob.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No; there is no live log or deterministic failing Brave response, but source inspection shows the current-main cancellation signal is dropped before Brave fetches.

## Solution Assessment

Is this the best way to solve the issue? Yes for the actionable defect: forward the existing provider execution signal through Brave's runtime; content-type validation can be considered as local hardening, but replacing Brave or adding a circuit breaker is not source-backed as the first fix.
