## Summary

Keep open: current main still drops the managed `web_search` abort signal at the Brave provider boundary, which is a plausible contributor to the reported Gateway timeout/hang class. The parser-runaway theory in the comments is not supported by the current Brave source because Brave parses each response once and has no retry loop.

## What This Changes

The issue reports that configuring Brave Search for `web_search` on OpenClaw 2026.5.2 can saturate CPU or hang the Gateway until Brave config is removed.

## Best Possible Solution

Brave should honor the existing `web_search` cancellation contract through the provider and endpoint fetch layers, with focused regression coverage and no new provider or config knob.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No; there are no logs, live Brave response, or deterministic Gateway hang reproduction, but current-main source inspection clearly shows the cancellation signal is dropped before Brave fetches.

## Solution Assessment

Is this the best way to solve the issue? Yes for the actionable defect: forward the existing provider execution signal through Brave's runtime and guarded fetch calls, using the Gemini provider's current pattern as the local precedent.
