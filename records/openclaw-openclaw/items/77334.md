## Summary

Keep open: current main still loses the managed `web_search` abort signal at the Brave provider boundary, which is a plausible source-backed contributor to the reported Gateway timeout/hang class. The parser-runaway theory in the later comment is not supported by current Brave source because there is no parser retry loop.

## What This Changes

The issue reports Gateway hangs and CPU saturation after configuring Brave-backed `web_search` on OpenClaw 2026.5.2.

## Best Possible Solution

Propagate the existing `AbortSignal` through the Brave provider and both Brave endpoint modes, then add focused Brave cancellation regression coverage; defer broader provider switching or circuit-breaker work unless logs prove a separate failure mode.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No; there is no live log or deterministic failing Brave response for the reported CPU saturation, but source inspection clearly shows current main drops the cancellation signal before Brave fetches.

## Solution Assessment

Is this the best way to solve the issue? Yes for the actionable defect: forwarding the existing provider execution signal through Brave is the narrowest maintainable fix and does not require a new provider, config knob, or product-policy change.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/77334. Current main carries the agent `AbortSignal` from `createWebSearchTool` through `runWebSearch` into provider execution, but the Brave provider ignores the execution context and the Brave runtime does not pass a signal into `withTrustedWebSearchEndpoint`/`withSelfHostedWebSearchEndpoint`. Keep the fix plugin-local: thread the existing context signal through `createBraveToolDefinition`, `executeBraveSearch`, `runBraveLlmContextSearch`, and `runBraveWebSearch`, and pass it to both Brave endpoint helper calls. Add focused regression coverage for both `web` and `llm-context` Brave modes proving the same `AbortSignal` reaches the guarded endpoint/fetch path. Do not add a new provider, config option, fallback policy, or circuit breaker unless new logs prove a separate malformed-response root cause. Add a single-line user-facing `CHANGELOG.md` fix entry.
