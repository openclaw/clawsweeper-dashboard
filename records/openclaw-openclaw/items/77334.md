## Summary

Keep open: current main still drops the managed `web_search` abort signal at the Brave provider boundary, which remains a plausible source-backed contributor to the reported Gateway timeout/hang class. The later parser-runaway suggestion is not supported by the current Brave source because Brave parses each response once and has no retry loop.

## What This Changes

The issue reports that enabling Brave web search API configuration on OpenClaw 2026.5.2 can saturate CPU, hang Discord/Gateway processing, and recover after removing Brave-related `web_search` config.

## Best Possible Solution

Propagate the existing `web_search` AbortSignal through Brave provider execution and both Brave endpoint modes, then add focused Brave cancellation regression coverage without adding a new provider or configuration knob.

## Reproduction Assessment

No; the exact CPU saturation/Gateway restart is not reproduced from the report alone. Source inspection does show a current-main cancellation path mismatch: core passes a signal into provider execution, but Brave drops it before its guarded fetches.

## Solution Assessment

Yes for the actionable defect; forwarding the existing provider execution signal through Brave is the narrowest maintainable fix. Replacing Brave, adding a circuit breaker, or switching search vendors is not justified by current source evidence as the first repair.

Current core already passes `{ signal }` from `createWebSearchTool` to `runWebSearch` to provider `execute`; update Brave so `createBraveToolDefinition` accepts the execution context, passes the signal into `executeBraveSearch`, and both `runBraveLlmContextSearch` and `runBraveWebSearch` pass it to `withTrustedWebSearchEndpoint`/`withSelfHostedWebSearchEndpoint`. Add focused regression coverage for both Brave `web` and `llm-context` modes proving the signal reaches the fetch helper/request path, and keep the change plugin-local except for tests if needed.
