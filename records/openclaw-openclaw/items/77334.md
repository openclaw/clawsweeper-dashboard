## Summary

Keep open: current main still has a narrow Brave web_search cancellation gap that can plausibly fit the reported Gateway timeout/hang path, although the exact CPU saturation has not been reproduced from logs or a live failing Brave response.

## What This Changes

The issue reports that configuring Brave for web_search can make Discord/Gateway runs hang or saturate CPU until Brave-related config is removed.

## Best Possible Solution

Keep the issue open and fix the Brave provider to forward web_search execution AbortSignal through both Brave transport modes, with focused regression coverage and only narrow defensive response parsing if tests prove it is part of the failure.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No; the report lacks logs and a deterministic Brave API failure, but source inspection shows current main drops the existing web_search AbortSignal before Brave fetch execution.

## Solution Assessment

Is this the best way to solve the issue? Yes for the actionable defect: propagate the existing provider execution signal through Brave instead of changing Discord, Gateway restart behavior, adding a config option, or switching search vendors.
