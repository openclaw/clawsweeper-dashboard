## Summary

Keep open: current main still drops the managed `web_search` abort signal at the Brave provider boundary, which fits the reported Gateway timeout/hang class even though the exact CPU saturation was not reproduced. The later parser-runaway theory is not supported by current Brave source because there is no retry loop around Brave response parsing.

## What This Changes

Investigate and fix Gateway hangs or CPU saturation after configuring Brave API-backed `web_search` on OpenClaw 2026.5.2.

## Best Possible Solution

Propagate the existing `web_search` `AbortSignal` through Brave's provider and runtime for both `web` and `llm-context`, then add focused Brave cancellation regression coverage and a changelog entry.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No; there are no logs or deterministic Brave responses, and I did not establish a live Gateway hang. Source inspection does show a clear current-main cancellation plumbing bug: core passes a signal and Brave drops it before fetch.

## Solution Assessment

Is this the best way to solve the issue? Yes for the actionable defect: forward the existing signal through Brave's local provider/runtime boundary. A new SERP provider, circuit breaker, or provider switch would be broader than the evidence currently supports.
