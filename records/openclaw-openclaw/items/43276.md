## Summary

Keep this PR open. Current main mitigates the default group/channel failure by making visible group replies message-tool-only, but the documented legacy `automatic` mode still injects the same same-group message-tool prohibition that blocks attachment workflows; the PR is directionally useful but needs a narrow rebase onto the current delivery-mode split.

## What This Changes

The PR rewrites the group-chat prompt to allow same-group message-tool sends for files, images, and attachments, and adds unit tests for the prompt content.

## Best Possible Solution

Update the current `automatic` group/channel prompt so text replies remain normal automatic replies while files, images, and attachments may use `message(action=send)` to the same group; preserve the existing `message_tool_only` prompt and add mode-specific tests for both branches.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure or request `sourceReplyDeliveryMode: "automatic"` for a group/channel turn and inspect `buildGroupChatContext`; current main still emits the blanket same-group message-tool ban while the outbound message action supports media sends.

## Solution Assessment

No, not as-is. The original one-line PR was correct for the older helper shape, but current main needs a mode-aware rebase that changes only the legacy `automatic` branch and keeps the newer tool-only default behavior intact.
