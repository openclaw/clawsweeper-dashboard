## Summary

Keep this PR open, but it is not merge-ready: the central reliability fix is useful, yet the exact merge result can still terminate an in-process restart after the new watchdog timeout and it changes the documented live-probe contract without a settled compatibility path.

## What This Changes

## Best Possible Solution

Keep the zombie-shutdown defense, but cancel or suppress the watchdog when an in-process restart continues, make the live-probe shutdown contract an explicit maintainer-approved behavior with docs/manifests aligned, and deduplicate retry logging before merge.

## Solution Assessment

Is this the best way to solve the issue? Not yet: the watchdog and stricter probe are plausible defenses, but the implementation must be restart-aware and the live-probe compatibility contract needs an explicit maintainer decision or a narrower internal preflight path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Cancel the watchdog before in-process restarts continue:** `src/gateway/server-close.ts:887`
  - body: `armWatchdog(...)` returns a cancel handle, but the close handler discards it after every clean close. The run-loop can then take the existing in-process restart path, start a new gateway in the same Node process, and keep the event loop alive; about 5 seconds later this old shutdown watchdog still calls `process.exit(0)`, killing the restarted gateway. Capture/cancel the handle when restart continues, or skip this watchdog for restart closes that do not hand off to a fresh process.
  - confidence: 0.91
- **[P1] Get maintainer approval for the live-probe 503 contract:** `src/gateway/server-http.ts:324-327`
  - body: Current `/health` and `/healthz` are documented and tested as unauthenticated liveness probes that return 200 while the process is live, and the Docker/Kubernetes examples use `/healthz` for liveness. Returning 503 during shutdown may be the desired supervisor signal, but it changes an operator-facing contract and can make existing monitors or orchestrators treat intentional drain/stop as unhealthy. Either narrow this to the supervised preflight path or update/approve the public liveness contract and matching docs/manifests/tests.
  - confidence: 0.86
- **[P3] Deduplicate zombie detection logs per recovery cycle:** `src/cli/gateway-cli/run.ts:462-468`
  - body: This warn and restart-trace event fires inside the retry loop, so with the default 5s retry and 30s timeout a normal draining previous gateway can emit several identical `gateway.preflight.zombie_detected` events before the timeout. Guard it with a per-cycle boolean or emit one summary on timeout to avoid noisy alerts and inflated trace events.
  - confidence: 0.82
