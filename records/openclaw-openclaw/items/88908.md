## Summary

Keep this PR open: the zombie-shutdown defense is aimed at a real gateway availability failure, but the current patch should not merge because the new post-shutdown watchdog is also armed for same-process restarts and can kill the restarted gateway.

## What This Changes

The PR adds a gateway post-shutdown forced-exit watchdog, shutdown-aware 503 responses for live health probes, stricter supervised lock-recovery health checks, zombie-detection logs/traces, and focused regression tests.

## Best Possible Solution

Keep the two-layer zombie defense, but make the forced-exit watchdog restart-aware or cancellable before same-process startup resumes, and cover both terminal shutdown and in-process restart with regression tests.

## Reproduction Assessment

Source-level yes: current main returns 200 for live probes and lock recovery accepts any status below 500, matching the reported zombie-deferral path. I did not run a full live zombie gateway reproduction in this read-only review.

## Solution Assessment

No, this is not the best fix yet: the 503 probe and forced-exit direction are plausible, but the watchdog must not remain armed across same-process restarts. A safer fix keeps the zombie guard for terminal shutdown while cancelling or suppressing it for in-process restart continuation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Cancel the watchdog before in-process restart continues:** `src/gateway/server-close.ts:879`
  - body: The close handler now arms a 5s process-exit watchdog after every clean close and discards the returned cancel handle. `run-loop.ts` has fallback paths that re-enter `startGatewayServer` in the same Node process, so a normal in-process restart leaves the old timer armed; once the new gateway is running and keeping the event loop alive, that timer can call `process.exit(0)`. Capture/cancel the handle on restart, or only arm this watchdog for terminal shutdown/full-process handoff paths.
  - confidence: 0.9
- **[P3] Deduplicate the unhealthy-probe trace per recovery attempt:** `src/cli/gateway-cli/run.ts:462-468`
  - body: This warn/trace runs on every retry while the lock is held and `probeHealth` is false. A previous gateway can legitimately stay draining for multiple retry intervals, so this can emit duplicate zombie alerts and trace events before timeout. Guard it once per supervised recovery cycle or emit a single timeout summary.
  - confidence: 0.83
