## Summary

Keep open: the underlying bug is valid, but this patch can select the wrong configured model cap by matching only the final slash segment, and the supplied proof does not convincingly show the fixed OpenClaw runtime path.

## What This Changes

The PR adds fallback model-id matching in the context-window guard, two regression tests for provider-scoped and bare IDs, and a changelog fix entry.

## Best Possible Solution

Keep the PR open, replace the fallback with active-provider prefix strip/prepend matching that preserves nested model IDs, add a collision regression test, and update the PR body with valid redacted terminal/log proof from the fixed path.

## Reproduction Assessment

Yes for source reproduction: current main exact-matches `models.providers.<provider>.models[].id` while the embedded runner forwards the runtime `modelId`, so provider-scoped runtime IDs can miss bare configured context caps. I did not run a live current-main agent session in this read-only review.

## Solution Assessment

No: bare/scoped equivalence is the right direction, but this patch also matches final slash segments. The safer fix is to strip only the active provider prefix from either side, preferably with normalized provider semantics, and keep the nested model ID intact.
