## Summary

Keep open: this PR targets a valid context-window bug, but the proposed fallback overmatches nested model IDs and the PR body only shows unit-test output, not real after-fix behavior proof.

## What This Changes

The PR adds provider-prefixed and bare model ID fallback matching in the context-window guard, two regression tests, and a changelog entry.

## Best Possible Solution

Keep this PR open, replace the fallback with provider-prefix-scoped matching that preserves nested model IDs, add a same-final-slug regression test, and require real after-fix OpenClaw proof before merge.

## Reproduction Assessment

Yes for source reproduction: current main uses exact model ID equality in `resolveContextWindowInfo` while the runner forwards the runtime `modelId`, so scoped-vs-bare IDs can miss config. I did not run a live agent reproduction in this read-only review.

## Solution Assessment

No: provider-scoped matching is the right direction, but this implementation should strip only the active provider prefix and keep the remaining nested model ID intact.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep fallback matching scoped to the active provider:** `src/agents/context-window-guard.ts:56`
  - body: This fallback reduces both IDs to their final path segment before matching. For providers like OpenRouter where model IDs are already `owner/model`, a request for `openrouter/openai/foo` can match the first configured `anthropic/foo` because both end in `foo`, applying the wrong `contextTokens`. Strip only the active provider prefix and keep the rest of the model ID intact.
  - confidence: 0.86
