## Summary

Keep open. Current main still routes heartbeat runs without an explicit `heartbeat.model` through stored session model overrides, so an `agent:main:main` auto-fallback override from an older default can continue to win after `agents.defaults.model.primary` changes; the linked closing PR is still open and should be reviewed or superseded before this issue closes.

## What This Changes

The issue asks OpenClaw to stop heartbeat sessions from pinning stale auto-fallback model overrides after `agents.defaults.model.primary` changes across gateway restarts.

## Best Possible Solution

Heartbeat model selection should clear or skip only stale direct `modelOverrideSource: "auto"` overrides whose recorded fallback metadata no longer matches the current configured default, while preserving explicit `heartbeat.model`, user and legacy overrides, parent inheritance, and unchanged-default auto-fallback persistence.

## Reproduction Assessment

Yes. Seed `agent:main:main` with a direct stored provider/model override marked `modelOverrideSource: "auto"`, change the configured default, and run heartbeat without `heartbeat.model`; current main still applies the stored override because `hasResolvedHeartbeatModelOverride` remains false.

## Solution Assessment

No for current main. The maintainable path is the narrow stale-heartbeat auto-fallback detection proposed by the linked PR or an equivalent targeted replacement, not a broad session reset or removal of normal auto-fallback persistence.
