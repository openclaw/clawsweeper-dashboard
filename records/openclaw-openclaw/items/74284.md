## Summary

Keep open. Current main still allows heartbeat runs without an explicit `heartbeat.model` to reuse stored session model overrides, including auto-fallback overrides created under an older default, and #74932 is already the open targeted implementation candidate linked to close this issue.

## What This Changes

The issue requests that autonomous heartbeat runs stop pinning `agent:main:main` to an auto-fallback model selected under an older `agents.defaults.model.primary` after the config changes and the gateway restarts.

## Best Possible Solution

Land or supersede #74932 with a narrow rule that clears only stale direct heartbeat auto-fallback session overrides after the configured default changes, while preserving explicit `heartbeat.model`, user selections, legacy overrides, parent overrides, and unchanged-default auto fallback persistence.

## Reproduction Assessment

Yes. Seed `agent:main:main` with `providerOverride`, `modelOverride`, and `modelOverrideSource: "auto"` from an older default, change `agents.defaults.model.primary`, then run heartbeat without `heartbeat.model`; current main applies the stored override because no explicit heartbeat override was resolved.

## Solution Assessment

Yes for the linked fix direction. A targeted stale direct heartbeat auto-fallback check is narrower and safer than resetting the whole session or clearing all stored model overrides.
