## Summary

Keep open. Current main still routes heartbeat runs without an explicit `heartbeat.model` through the normal stored session override path, and existing tests intentionally preserve auto fallback overrides until reset. The linked fix PR #74932 uses closing syntax but is still open and unmerged, so this issue should remain the tracker until that PR lands or is superseded.

## What This Changes

The issue reports that autonomous heartbeat runs can keep using a stale persisted auto fallback model after `agents.defaults.model.primary` changes and the gateway restarts.

## Best Possible Solution

Review and land or supersede #74932 with a narrow heartbeat-path fix that clears only stale direct `modelOverrideSource: "auto"` fallback overrides when the recorded selected model no longer matches the current configured default, while preserving explicit `heartbeat.model`, user/legacy overrides, parent inheritance, and unchanged-default auto fallback persistence. Treat the transcript growth note as a separate follow-up unless it remains reproducible after the current heartbeat isolation work.

## Reproduction Assessment

Yes. A source-level reproduction is clear: seed `agent:main:main` with a direct auto fallback `modelOverride`, change `agents.defaults.model.primary`, then run heartbeat without explicit `heartbeat.model`; current main passes no heartbeat override and reapplies the stored model override.

## Solution Assessment

Yes, the linked fix direction is the narrowest maintainable approach: handle stale direct heartbeat auto fallback overrides in model selection, with regression tests around changed defaults and preserved non-stale/user/parent cases. The issue should stay open until that implementation is merged or replaced.
