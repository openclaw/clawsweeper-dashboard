## Summary

Keep open. Current main still applies stored session model overrides during heartbeat runs when no explicit `heartbeat.model` is configured, and the supplied context shows open PR #74932 uses closing syntax for this issue. That PR is the active implementation path, so this issue should stay open until maintainers land or supersede it.

## What This Changes

The issue requests heartbeat runs without explicit `heartbeat.model` to stop reusing stale direct auto-fallback session model overrides after `agents.defaults.model.primary` changes.

## Best Possible Solution

Review and either land or supersede PR #74932 with focused coverage that clears stale direct `modelOverrideSource: "auto"` heartbeat overrides after the configured default changes while preserving explicit `heartbeat.model`, user-selected overrides, legacy overrides, parent overrides, and unchanged-default auto fallback persistence.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a session entry with a direct auto fallback override and older `fallbackNoticeSelectedModel`, no configured `heartbeat.model`, then a heartbeat tick after the default model changes; current main keeps applying that stored override.

## Solution Assessment

Yes in direction. The narrow fix belongs in the model-selection/session-override seam for direct heartbeat auto fallback state, not in broad startup migration or blanket override clearing; the open PR claims that boundary and should be reviewed on its own diff.
