## Summary

This should stay open: current main still documents and configures OTEL content capture, and the diagnostics-otel plugin can map content fields, but the core model/tool completion events only emit timing, size, and metadata. The remaining work is real, with a privacy-sensitive design choice around where content capture is gated.

## What This Changes

Request to populate optional model and tool content fields in diagnostic events so diagnostics.otel.captureContent can export prompt, response, tool input, tool output, and system-prompt attributes.

## Best Possible Solution

Add the missing content fields through a privacy-aware core contract: gate content emission on the existing captureContent policy, update the SDK event types and focused diagnostics tests, and keep the OTEL plugin responsible for per-field export filtering.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: enable diagnostics.otel.captureContent and run model/tool traces, and the OTEL plugin has content-field readers but the current completion events never populate those fields. I did not run a live OTLP trace because this review is read-only.

## Solution Assessment

No, not as written: adding optional fields is the right direction, but always populating content before the exporter filter is not the safest boundary. The implementation should make the opt-in capture decision before sensitive content enters diagnostic event payloads.
