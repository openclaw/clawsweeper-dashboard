## Summary

Keep open: this PR addresses a real Discord authorization gap, but the new guard is too broad and can block existing owner CLI/manual guild-admin actions that run without `toolContext` or `requesterSenderId`.

## What This Changes

The PR centralizes Discord privileged guild-admin action detection and rejects non-Discord requester identities for those actions while preserving read-only guild lookup behavior.

## Best Possible Solution

Keep the Discord-owned cross-provider rejection, but scope it to tool contexts with a present non-Discord provider and add regression coverage proving CLI/manual no-context owner actions still reach the runtime.

## Reproduction Assessment

Yes, at source level: current main drops non-Discord requester IDs before Discord guild/moderation runtime, while the runtime permits missing sender context; the PR tests cover the cross-provider rejection path. I did not run tests in this read-only review.

## Solution Assessment

No, not as written. The Discord plugin is the right owner boundary, but the guard should distinguish present non-Discord tool contexts from absent CLI/manual context instead of failing both.
