## What This Changes

The PR centralizes Discord privileged guild-admin action classification and rejects non-Discord-origin privileged guild admin or moderation actions unless a trusted Discord sender identity is present, while preserving read-only guild lookups.

## Best Possible Solution

Merge a Discord-owned fail-closed guard after maintainers accept the compatibility impact and the contributor adds redacted real after-fix proof that privileged cross-provider actions are rejected while read-only guild lookups still work.

## Reproduction Assessment

Yes from source inspection, though I did not execute a live repro: current main only requires trusted sender identity for Discord-origin turns, and the Discord runtime skips permission checks when senderUserId is absent. The PR adds focused tests for the Telegram-origin channel-delete and timeout paths.

## Solution Assessment

Yes, the fix is in the right owner boundary: Discord owns which actions require a Discord sender identity, and the handler owns rejecting non-Discord origins before runtime permission checks. A core provider-specific block would be a worse fit because it would move Discord policy out of the plugin surface.
