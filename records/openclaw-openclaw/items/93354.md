## Summary

Keep open: current main still has the cross-provider Discord requester gap this PR is trying to close, but the current head is not merge-ready because its shared guard can reject owner/manual no-context admin actions before the Discord handler sees the owner trust bit.

## What This Changes

The PR centralizes Discord privileged guild-admin action detection, requires trusted requester identity for those privileged actions across origins, rejects non-Discord origins in the Discord guild-admin handler, and adds Discord action tests.

## Best Possible Solution

Keep the Discord-owned cross-provider rejection, but scope the shared trusted-requester guard so explicit non-Discord contexts reject while owner/manual no-context calls reach the handler with committed dispatcher-level regression coverage.

## Reproduction Assessment

Yes. Source inspection shows PR head makes the shared adapter guard require `requesterSenderId` before the Discord handler can inspect `senderIsOwner`, so no-context owner/manual admin actions are blocked at dispatch.

## Solution Assessment

No. The Discord plugin is the right owner boundary, but the current guard is too high in the call stack and the handler type contract is incomplete.
