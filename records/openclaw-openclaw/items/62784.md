## Summary

Keep this open. Current main still has only partial Slack Socket Mode mitigations: ping timeout tuning and reconnect backoff exist, but multi-account Slack Socket Mode startup is still concurrent and reaches `app.start()` without per-account desynchronization.

## What This Changes

The issue asks OpenClaw to survive multi-agent Slack Socket Mode startup storms by avoiding simultaneous WebSocket connects and reconnect cascades.

## Best Possible Solution

Add a Slack-owned, abortable startup desynchronization before Socket Mode opens when multiple enabled Slack Socket Mode accounts are configured, while keeping generic channel concurrency and existing timeout/backoff behavior unchanged.

## Reproduction Assessment

Unclear for the full crash because it requires several live Slack apps and host pressure. Yes for the trigger path: current main starts channel accounts through `Promise.all`, then Slack reaches `app.start()` without per-account startup delay.

## Solution Assessment

No, current main is only a partial mitigation. The narrow maintainable fix is Slack-specific startup jitter before Socket Mode start, not connection pooling or a broad channel-manager concurrency change.
