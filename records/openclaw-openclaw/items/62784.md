## Label Justifications

- `P1`: The report describes repeated production gateway crashes from an existing multi-account Slack Socket Mode workflow.
- `impact:crash-loop`: The central impact is repeated SIGKILL/SIGTERM gateway termination under Slack Socket Mode startup load.

## Summary

Keep this open. Current main and v2026.5.19 include useful Slack Socket Mode timeout, native reconnect, and backoff mitigations, but the initial multi-account Socket Mode startup path still has no Slack-owned stagger before `app.start()`, so the reported startup fanout remains a plausible availability bug.

## What This Changes

The issue reports that 8+ Slack Socket Mode accounts start concurrently, hit simultaneous pong timeouts, and can crash-loop the gateway, requesting staggered startup and reconnect hardening.

## Best Possible Solution

Add a Slack-owned, abortable startup desynchronization before Socket Mode `app.start()` when multiple enabled Socket Mode accounts start, while keeping current timeout/native-reconnect/backoff behavior and leaving transport isolation or pooling to separate follow-ups.

## Reproduction Assessment

No for the full SIGKILL cascade, which requires multiple live Slack apps and host pressure. Yes at source level for the trigger: current main can hand off multiple Slack account starts without staggering, and the Socket Mode path immediately calls `app.start()`.

## Solution Assessment

Yes for the narrow startup-stagger slice. Exponential reconnect/backoff is already present, while connection pooling or worker-thread transport isolation would be broader work better tracked separately.
