## Summary

Keep open: current main has useful Slack Socket Mode mitigations, including native reconnect and a 15s default pong timeout, but it still opens Slack Socket Mode connections without Slack-owned startup desynchronization for multi-account gateways.

## What This Changes

The issue asks OpenClaw to survive 8+ concurrent Slack Socket Mode account startups by avoiding synchronized pong-timeout and reconnect storms.

## Best Possible Solution

Add a Slack-owned, abortable initial startup stagger before Socket Mode opens when multiple enabled Slack Socket Mode accounts are configured, while leaving connection pooling and global gateway startup policy to separate maintainer decisions.

## Reproduction Assessment

No for the full SIGKILL cascade because it requires several live Slack apps and host pressure. Yes at source level for the trigger: current main can hand off Slack account monitors without awaiting/staggering the long-lived provider startup, and the Slack socket open path has no initial delay.

## Solution Assessment

No, current main is only partially mitigated. The narrow maintainable repair is Slack-specific startup desynchronization before Socket Mode app.start(), not connection pooling or a broad global gateway policy unless maintainers choose that separately.
