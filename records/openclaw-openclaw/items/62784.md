## Label Justifications

- `P1`: The report describes repeated production gateway crashes from an existing multi-account Slack workflow.
- `impact:crash-loop`: The central impact is repeated SIGKILL/SIGTERM gateway termination under Slack Socket Mode startup load.

## Summary

Keep this open. Current main and v2026.5.18 include useful Slack Socket Mode mitigations, but they still do not stagger the initial per-account WebSocket open before `app.start()`, so the reported multi-account startup fanout remains a plausible availability bug.

## What This Changes

The issue asks OpenClaw to survive multi-agent Slack Socket Mode startup by avoiding simultaneous per-account WebSocket connection and reconnect storms.

## Best Possible Solution

Add a Slack-owned, abortable startup desynchronization before Socket Mode `app.start()` when multiple enabled Socket Mode accounts are configured, while leaving connection pooling and broader transport isolation to separate follow-ups.

## Reproduction Assessment

No for the full SIGKILL cascade, which requires several live Slack apps and host pressure. Yes at source level for the trigger: current main can start Slack account monitors without initial staggering, and the Socket Mode path immediately calls `app.start()`.

## Solution Assessment

Yes for the narrow startup-stagger slice. A Slack-owned abortable delay before Socket Mode `app.start()` is the maintainable fix boundary; connection pooling or worker-thread transport isolation would be broader product work.
