## Label Justifications

- `P1`: The report describes repeated gateway event-loop freezes that make multi-agent gateways unresponsive and cause model idle timeouts in normal use.
- `impact:crash-loop`: The issue is about process-level availability: long synchronous Node event-loop hangs in the gateway hot path.

## Summary

Keep open. Current main has same-key process memoization for `loadPluginMetadataSnapshot`, but it is still a single-slot cache, so alternating multi-agent workspace/config lookups can evict each other and re-enter the synchronous installed-plugin crawl path.

## What This Changes

The issue asks for bounded multi-entry memoization of `loadPluginMetadataSnapshot` so multi-agent gateways do not repeatedly rebuild installed-plugin metadata synchronously.

## Best Possible Solution

Extend the existing process-local snapshot memo into a small bounded multi-entry cache keyed by the current memo key, preserving clone-on-return and freshness checks, with regression coverage for alternating workspace/config lookups.

## Reproduction Assessment

Yes at source level. A focused test can call `loadPluginMetadataSnapshot` with workspace/config key A, then B, then A, and assert the current single-slot memo reloads A instead of retaining multiple entries.

## Solution Assessment

No, current main is only a partial fix for the reported multi-agent case. The maintainable fix is a bounded multi-entry memo using the existing key and freshness machinery rather than another caller-specific cache.
