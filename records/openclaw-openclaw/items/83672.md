## Label Justifications

- `P1`: The report describes repeated gateway event-loop freezes that make multi-agent gateways unresponsive and cause model idle timeouts in normal use.
- `impact:crash-loop`: The issue is about process-level availability: long Node event-loop hangs in the gateway hot path.

## Summary

Keep open. Current main has a narrow process-local memo for repeated identical `loadPluginMetadataSnapshot` calls, but it is still a single-slot cache, so alternating multi-agent workspace/config keys can evict each other and re-enter the synchronous crawl path described in the report.

## What This Changes

The issue requests preventing repeated synchronous plugin metadata/index rebuilds on multi-agent gateways by making `loadPluginMetadataSnapshot` caching effective across multiple workspace/config keys.

## Best Possible Solution

Extend the existing process-local metadata snapshot memo into a small bounded multi-entry cache keyed by the current memo key, with the same freshness checks, cloning behavior, and clear lifecycle, plus regression coverage for alternating workspace/config lookups.

## Reproduction Assessment

Yes at source level: a focused test can call `loadPluginMetadataSnapshot` with workspace/config key A, then B, then A, and observe that the single-slot memo forces the A lookup to reload. I did not run a live gateway profile in this read-only pass.

## Solution Assessment

No, current main is not the complete fix for the reported multi-agent case. The maintainable path is to reuse the existing memo key/freshness machinery but retain a bounded set of entries instead of only the most recent one.
