## Summary

Keep open: the PR targets a valid active-memory status bug, but the latest patch passes the full runtime config into a helper that expects normalized active-memory plugin config, so targeted agents would also report `off`.

## What This Changes

The PR changes the active-memory session status command to account for configured agent scope, adds a regression test, and records a changelog entry for the linked bug report.

## Best Possible Solution

Keep the fix plugin-owned: refresh or resolve the live active-memory plugin config, check agent eligibility against that normalized config, cover both targeted-on and untargeted-off status cases, then merge after command-path proof is supplied.

## Reproduction Assessment

Yes. Source inspection shows the current session status branch can report `on` for an untargeted agent because it only checks the session disabled toggle, while docs and the recall hook require `config.agents` targeting.

## Solution Assessment

No. The requested direction is narrow and appropriate, but this patch uses the full `OpenClawConfig` where normalized active-memory plugin config is required, which would make status report `off` even for targeted agents.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Use normalized active-memory config for eligibility:** `extensions/active-memory/index.ts:2868`
  - body: The status path now calls `isEnabledForAgent(liveConfig, agentId)`, but `liveConfig` is the full `OpenClawConfig` returned by `api.runtime.config.current()`, while `isEnabledForAgent` expects the normalized active-memory plugin config. Because the full config has no top-level active-memory `enabled` field and its `agents` field is the core agents object, targeted agents will be reported as `off` too. Refresh or resolve the plugin config before this check and add a targeted-agent status assertion.
  - confidence: 0.94
