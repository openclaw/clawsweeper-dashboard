## Summary

Keep this PR open: current main still has the reported Dreaming ingestion gap, but the PR head is not merge-ready because its runId fallback can classify an explicitly owned non-cron transcript as cron-owned and skip real user content.

## What This Changes

The PR adds session-id-aware transcript classification, Dreaming sessionFilter config/docs/tests, internal-system provenance filtering for heartbeat/cron/exec prompts, and small CI/format cleanup changes.

## Best Possible Solution

Keep the PR’s session-id/filter direction, but make cron classification honor the resolved session-key owner when a runId collides with a registered non-cron sessionId, then land it with the existing changelog/docs coverage.

## Reproduction Assessment

Yes. A sessions.json with an explicit non-cron entry using sessionId shared-id and a cron entry whose runId is also shared-id makes lookupSessionKeyForTranscriptPath resolve the explicit session while isCronRunTranscriptPath still returns true on the PR head.

## Solution Assessment

No, not as currently implemented. The session-id-aware helper approach is the right maintainable direction, but the runId fallback must not classify by a bare id set when the reverse ownership map says that id belongs to a non-cron session.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Avoid treating runId collisions as cron transcripts:** `packages/memory-host-sdk/src/host/session-files.ts:223`
  - body: The runId fallback adds runIdFromKey to cronRunSessionIds even when that id is already registered as another session's sessionId. In that collision, lookupSessionKeyForTranscriptPath resolves the explicit non-cron session, but isCronRunTranscriptPath still returns true because it checks the bare id set, so Dreaming can skip a real user transcript. Avoid adding colliding runIds to the cron set or make cron classification follow the resolved owner.
