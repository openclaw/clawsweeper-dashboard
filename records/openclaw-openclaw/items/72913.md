## Summary

Keep this PR open, but do not merge it as-is: current main still has the Dreaming cron-transcript exclusion gap, and this PR targets the right area, but the new runId fallback can misclassify a non-cron transcript when its sessionId collides with a cron runId.

## What This Changes

The PR extends memory-host transcript classification with session-id/run-id lookup, adds memory-core Dreaming session filters with docs/schema/tests, marks heartbeat/cron/exec prompts as internal-system provenance, and removes stale raw-fetch allowlist entries.

## Best Possible Solution

Keep the PR open, make cron classification follow the resolved session-key owner when a runId collides with another sessionId, then rerun the targeted memory-host and memory-core tests plus the changed gate before merge.

## Reproduction Assessment

Yes. A sessions store with an explicit non-cron entry using `sessionId: "shared-id"` and a cron entry whose `:run:` id is also `shared-id` makes `lookupSessionKeyForTranscriptPath` resolve the explicit session while `isCronRunTranscriptPath` still returns true because `shared-id` was added to `cronRunSessionIds`.

## Solution Assessment

No, not as currently implemented. The overall session-id-aware helper direction is maintainable, but the runId fallback should not classify by a bare id set when the reverse ownership map says that id belongs to a non-cron session.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Avoid treating runId collisions as cron transcripts:** `packages/memory-host-sdk/src/host/session-files.ts:223-225`
  - body: The runId fallback adds `runIdFromKey` to `cronRunSessionIds` even when that id is already registered as another session's `sessionId`. In that collision, `lookupSessionKeyForTranscriptPath` correctly resolves the explicit non-cron session, but `isCronRunTranscriptPath` still returns true because it checks the id set directly, so Dreaming will skip a real user transcript. Please make cron classification follow the owning session key or avoid adding colliding runIds to the cron set.
  - confidence: 0.93
