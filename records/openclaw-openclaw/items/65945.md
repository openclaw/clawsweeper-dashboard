## Summary

Keep open. Current main still treats Node `agent.request` attachments as agent/model input rather than delivery payloads, and automatically forwarding them to IM recipients would add a new gateway delivery contract with authorization, media validation, and suppression semantics.

## What This Changes

Request an opt-in or fallback that forwards images from Node `agent.request` attachments to the target IM user when `deliver: true`, even if the agent response does not include media.

## Best Possible Solution

Add an explicit, versioned protocol opt-in only after maintainer approval, carry sanitized delivery-ready media references through the Node request path, and inject them into outbound payloads only for authorized `deliver: true` routes when the agent reply intentionally leaves media empty.

## Reproduction Assessment

Yes for the current behavior by source inspection, but not as a broken existing contract. Node attachments become `images`/`imageOrder` for agent input, while outbound delivery is built only from the agent result payloads.

## Solution Assessment

Unclear. An explicit opt-in field is plausible, but default fallback forwarding would be a broader and riskier product/security change than the current agent-controlled delivery contract.
