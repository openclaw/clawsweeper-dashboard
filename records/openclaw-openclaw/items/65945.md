## Label Justifications

- `impact:security`: The requested behavior would forward Node-supplied media across a credentialed chat delivery boundary and needs trust-boundary review.

## Summary

Keep open: current main and v2026.6.10 still accept Node `agent.request` attachments as agent/model input, but outbound delivery only sends media from the agent reply payload. The requested auto-forward behavior is a new security-sensitive gateway protocol/media-delivery contract, not conservative cleanup.

## What This Changes

The issue requests an explicit way for Node `agent.request` image attachments to be forwarded to the chat recipient when `deliver` is true, even if the agent does not include media in its reply.

## Best Possible Solution

Approve a narrow explicit additive protocol opt-in only if maintainers want `agent.request` to act as a media relay; otherwise keep media delivery agent-controlled and direct relay use cases on existing send surfaces.

## Reproduction Assessment

Yes, by source inspection: current `agent.request` attachment handling feeds images into the agent/model while outbound delivery only sends media from the agent reply payload. This is not a broken existing contract because the current protocol has no auto-forward opt-in.

## Solution Assessment

Unclear. A narrow explicit opt-in may be the right implementation if maintainers approve the product direction, but automatic fallback forwarding when the agent returns no media would be a broader security and policy change than the current agent-controlled delivery contract.
