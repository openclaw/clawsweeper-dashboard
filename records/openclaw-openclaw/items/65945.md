## Summary

Keep open. Current main still treats Node `agent.request` attachments as agent/model input, while outbound delivery is built from the agent reply payloads, so auto-forwarding would be a new gateway delivery contract rather than cleanup of an already-implemented path.

## What This Changes

Request an opt-in or fallback that forwards Node `agent.request` image attachments to the target IM recipient when `deliver: true`, even if the agent response has no media.

## Best Possible Solution

Add an explicit, additive protocol opt-in only after maintainer approval, carry sanitized delivery-ready media references through `agent.request`, and inject them into outbound payloads only for authorized `deliver: true` routes when the agent intentionally leaves media empty.

## Reproduction Assessment

Yes for the current behavior by source inspection: node attachments become `images`/`imageOrder` for the agent, and empty reply payloads are suppressed before outbound send. This is not a broken existing contract because the current schema has no auto-forward opt-in.

## Solution Assessment

Unclear. A narrow explicit opt-in is plausible, but default fallback forwarding would be a broader product and security change than the current agent-controlled delivery contract.
