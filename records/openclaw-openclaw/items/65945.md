## Label Justifications

- `P2`: This is a concrete gateway/media delivery improvement with limited blast radius, but it needs product and security scoping before implementation.
- `impact:message-loss`: The reported user impact is that media attached to a delivered Node agent request is not sent to the target chat unless the agent echoes it.
- `impact:security`: The requested change would forward Node-supplied media across a delivery boundary and needs authorization and media-validation rules.

## Summary

Keep open. Current main still treats Node `agent.request` attachments as agent/model input, while outbound delivery is built from the agent reply payloads, so auto-forwarding images would add a new gateway media-delivery contract rather than closeable cleanup.

## What This Changes

The issue requests an opt-in or fallback that automatically forwards image attachments from Node `agent.request` calls to the target IM user when `deliver: true`.

## Best Possible Solution

Add an explicit, additive protocol opt-in only if maintainers approve it, carry sanitized delivery-ready media references through `agent.request`, and keep default delivery agent-controlled while direct relay use cases continue to use the direct send surface.

## Reproduction Assessment

Yes for the current behavior by source inspection: Node attachments become agent input, and outbound sends are built only from agent reply payloads. This is not a broken existing contract because the current protocol has no auto-forward opt-in.

## Solution Assessment

Unclear. A narrow explicit opt-in is plausible, but default fallback forwarding would be a broader product and security change than the current agent-controlled delivery contract.
