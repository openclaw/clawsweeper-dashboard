## Summary

Keep open. Current main still treats Node `agent.request` image attachments as agent/model input only; there is no protocol field or delivery path that auto-forwards the original attachment when the agent reply has text-only or no media.

## What This Changes

Request to add a Node `agent.request` option or fallback that forwards image attachments to the selected IM recipient when `deliver: true`, even if the agent response lacks media.

## Best Possible Solution

Keep this open until maintainers approve the contract. A maintainable implementation would add an explicit opt-in field, carry sanitized delivery-ready media references through the gateway command path, and inject them only when `deliver: true`, the target is resolved and authorized, the channel supports that media type, and the agent did not provide its own media.

## Reproduction Assessment

Yes. A code-level reproduction is high-confidence: send a Node `agent.request` payload with image attachments and `deliver: true`; the current path forwards only `images`/`imageOrder` into the agent and has no original attachment media to deliver when the agent reply lacks media.

## Solution Assessment

Unclear. The requested direction is plausible, but a default fallback is riskier than an explicit opt-in field with authorization and channel-capability checks, so maintainer product/security approval should come before implementation.
