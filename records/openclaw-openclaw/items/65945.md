## Summary

Keep open. Current main still treats Node `agent.request` image attachments as agent/model input and delivers only media produced by the agent reply; the requested outbound auto-forwarding contract is not present, and the prior ClawSweeper keep-open comment remains accurate against the newer main SHA.

## What This Changes

The issue requests an opt-in or fallback path that forwards Node-provided image attachments to the target IM channel when `agent.request` uses `deliver: true`, even if the agent returns only text or no media.

## Best Possible Solution

Keep this open until maintainers approve the contract. A maintainable implementation would add an explicit opt-in field, carry sanitized delivery-ready media references through the gateway command path, and inject them only when `deliver: true`, the target is resolved and authorized, the channel supports that media type, and the agent did not provide its own media.

## Reproduction Assessment

Yes. A code-level reproduction is available on current main by sending a Node `agent.request` payload with image attachments and `deliver: true`; the request path forwards only `images`/`imageOrder` into the agent, and delivery has no original-attachment payload to send when the agent reply lacks media.

## Solution Assessment

Unclear. The requested direction is plausible, but a default fallback would be riskier than an explicit opt-in field with authorization and channel-capability checks, so maintainer product/security approval should come before implementation.
