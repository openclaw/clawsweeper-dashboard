## Summary

Keep open. Current main still treats Node event image attachments as agent/model input only; forwarding those attachments to chat recipients would require a new explicit gateway delivery contract plus maintainer product/security approval.

## What This Changes

The issue requests an opt-in way for Node `agent.request` image attachments to be forwarded to the target chat recipient when `deliver: true` even if the agent does not echo the media.

## Best Possible Solution

Add an explicit opt-in protocol/API field only after maintainer approval, carry sanitized delivery-ready media references through the gateway command path, and inject them only for authorized `deliver: true` routes when the agent reply does not already include media.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a Node `agent.request` with image attachments and `deliver: true` becomes `images`/`imageOrder` for the agent, while delivery later sends only agent reply payloads and returns without delivery when the agent has no payloads.

## Solution Assessment

Unclear. The feature direction is plausible, but the safer solution is an explicit opt-in field with authorization and channel/media checks rather than a default fallback that forwards every request attachment.
