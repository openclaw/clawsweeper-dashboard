## Label Justifications

- `P2`: This is a concrete gateway/media delivery enhancement with limited blast radius but non-trivial product and security scoping.
- `impact:message-loss`: The reported user impact is that image attachments on a delivered Node agent request are not sent to the chat unless the agent echoes media.
- `impact:security`: The requested behavior would forward Node-supplied media across a chat delivery boundary and needs trust-boundary review.

## Summary

Keep open because current main still treats `agent.request` attachments as agent/model input and outbound delivery is still built from agent reply payloads, so the requested media relay remains a new gateway delivery contract that needs maintainer product and security review.

## What This Changes

Request to add an `agent.request` option or fallback that automatically forwards original image attachments to the target chat when `deliver: true`, without requiring the agent to echo media in its reply.

## Best Possible Solution

Approve a narrow, explicit, additive protocol opt-in only if maintainers want `agent.request` to become a media relay; otherwise keep automatic delivery agent-controlled and direct relay use cases on the existing send surface.

## Reproduction Assessment

Yes by source inspection: current `agent.request` attachment handling feeds images into the agent/model, while outbound delivery only sends media that appears in the agent reply payload. This is not a broken existing contract because the current protocol has no auto-forward opt-in.

## Solution Assessment

Unclear. A narrow explicit opt-in may be the best implementation if maintainers approve the product direction, but fallback forwarding when the agent returns no media would be a broader security and policy change than the current agent-controlled delivery contract.
