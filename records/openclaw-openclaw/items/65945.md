## Summary

Keep open. Current main still treats Node `agent.request` attachments as agent input, and automatically forwarding them to IM recipients would add a new gateway delivery contract with security-sensitive opt-in, authorization, and suppression semantics.

## What This Changes

Request adding an `agent.request` delivery option that auto-forwards Node-supplied image attachments to the target IM recipient when `deliver` is true.

## Best Possible Solution

Add an explicit protocol opt-in only after maintainer approval, carry sanitized delivery-ready media references through the Node-to-agent path, and inject them into outbound payloads only for authorized `deliver: true` routes when the agent reply has no media.

## Reproduction Assessment

Yes for the current behavior by source inspection, but not as a broken existing contract. A Node `agent.request` attachment is converted into `images`/`imageOrder`, while outbound delivery later uses only the agent result payloads and suppresses an empty payload list.

## Solution Assessment

Unclear. An explicit opt-in forwarding field is plausible, but a default fallback that forwards every request attachment would be a riskier product and security change.
