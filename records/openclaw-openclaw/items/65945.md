## Summary

Keep open. Current main still treats Node `agent.request` attachments as model input only, and forwarding those attachments to chat recipients would add a new gateway delivery contract with security-sensitive product choices.

## What This Changes

Request an opt-in way for Node `agent.request` attachments to be forwarded to the target chat user when `deliver` is true even if the agent reply has no media.

## Best Possible Solution

Add an explicit opt-in protocol/API field only after maintainer approval, carry sanitized delivery-ready media references through the gateway command path, and inject them only for authorized `deliver: true` routes when the agent reply does not already include media.

## Reproduction Assessment

Yes for the current behavior, by source inspection rather than a live gateway run. A Node `agent.request` with image attachments is converted into `images`/`imageOrder`, while outbound delivery later uses only the agent's reply payloads and suppresses an empty payload list.

## Solution Assessment

Unclear. An explicit opt-in forwarding field is plausible, but a default fallback that forwards every request attachment would be a riskier product and security change.
