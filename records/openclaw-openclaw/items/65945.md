## Label Justifications

- `P2`: This is a concrete Gateway/media delivery enhancement with limited blast radius but non-trivial product and security scoping.
- `impact:security`: The requested behavior would forward Node-supplied media across a chat delivery boundary without an agent-authored media reply.
- `impact:message-loss`: The reported user impact is that delivered Node request image attachments are not sent to the chat unless the agent echoes media.

## Summary

Keep open because current main and the latest release still treat Node `agent.request` attachments as agent/model input, while outbound delivery is built only from the agent reply payloads. The requested auto-forward behavior remains a new security-sensitive Gateway protocol and media-delivery contract that needs maintainer product review.

## What This Changes

Requests an opt-in or default behavior that forwards image attachments from Node `agent.request` with `deliver: true` to the IM recipient even when the agent reply contains only text or no media.

## Best Possible Solution

Approve a narrow, explicit, additive protocol opt-in only if maintainers want `agent.request` to relay media; otherwise keep automatic media delivery agent-controlled and route dumb relay use cases through the existing send/media surfaces.

## Reproduction Assessment

Yes, by source inspection: current `agent.request` attachment handling feeds images into the agent/model, while outbound delivery only sends media from agent reply payloads. This is not a broken existing contract because the current protocol has no auto-forward opt-in.

## Solution Assessment

Unclear. A narrow explicit opt-in may be the right implementation if maintainers approve the product direction, but fallback forwarding whenever the agent returns no media would be a broader security and policy change than the current agent-controlled delivery contract.
