## Summary

Keep open. Current main still treats Node `agent.request` image attachments as agent/model input only; auto-forwarding them to IM recipients would need a new explicit gateway delivery contract plus maintainer product and security approval.

## What This Changes

Request to add opt-in or fallback forwarding of image attachments from Node `agent.request` deliveries to the target IM user when the agent reply omits media.

## Best Possible Solution

Add an explicit opt-in protocol/API field after maintainer approval, carry sanitized delivery-ready media references through the command path, and forward only on authorized `deliver: true` routes when the agent reply has no media.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: Node `agent.request` attachments become `images`/`imageOrder` for the agent, while outbound delivery only sends media present in the agent's reply payloads.

## Solution Assessment

Unclear. The feature direction is plausible, but the safest maintainable solution is an explicit opt-in field with authorization, media validation, channel capability checks, and no default attachment fallback.
