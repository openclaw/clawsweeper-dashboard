- `impact:security`: The requested behavior would forward Node-supplied media across a credentialed IM delivery boundary and needs trust-boundary review.

## Summary

Keep open because current main and v2026.6.10 still do not implement auto-forwarding for Node `agent.request` attachments; the request is a new security-sensitive delivery contract, not a fixed or stale bug.

## What This Changes

Request adds a way for Node `agent.request` with `deliver: true` to forward original image attachments to the IM recipient even when the agent reply has no media.

## Best Possible Solution

Approve a narrow explicit additive gateway protocol opt-in only if maintainers want `agent.request` to act as a media relay; otherwise keep automatic delivery agent-controlled and direct relay use cases on `openclaw message send` or the message tool.

## Reproduction Assessment

Yes. Source inspection shows `agent.request` attachments are parsed into `message`, `images`, and `imageOrder` for agent input, while outbound delivery sends only media present in the agent reply payloads.

## Solution Assessment

Unclear. A narrow explicit opt-in is safer than automatic fallback forwarding, but maintainers still need to decide whether this belongs in `agent.request` or should remain on the existing direct send and message-tool surfaces.

## Risks / Open Questions

- Auto-forwarding Node-supplied media would cross a credentialed chat delivery boundary without an agent-authored media reply unless maintainers define explicit opt-in, target authorization, media limits, audit behavior, and override semantics.
- Fallback forwarding when the agent returns no media could conflict with the current agent-controlled delivery contract and `message_tool_only` visible-reply semantics.
- The existing direct send and message-tool surfaces already cover some dumb relay cases, so maintainers need to decide whether `agent.request` should grow a parallel media relay contract.
