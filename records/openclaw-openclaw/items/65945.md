## Label Justifications

- `P2`: This is a concrete gateway/media delivery enhancement with limited blast radius but non-trivial product and security scoping.
- `impact:security`: The requested behavior would forward Node-supplied media across a credentialed chat delivery boundary and needs trust-boundary review.
- `impact:message-loss`: The reported user impact is that image attachments on a delivered Node agent request are not sent to the chat unless the agent echoes media.

## Summary

Keep open because current main and v2026.6.8 still route Node `agent.request` attachments into agent input and only deliver media that appears in the agent reply payload; the requested auto-forward behavior is a new security-sensitive delivery contract, not stale cleanup.

## What This Changes

Request an `agent.request` opt-in that automatically forwards Node-supplied image attachments to the target chat when `deliver: true`, even if the agent reply has no media.

## Best Possible Solution

Approve a narrow explicit additive protocol opt-in only if maintainers want `agent.request` to act as a media relay; otherwise keep automatic delivery agent-controlled and direct relay use cases on the existing send surfaces.

## Reproduction Assessment

Yes, by source inspection: current `agent.request` attachment handling feeds images into the agent/model, while outbound delivery only sends media from the agent reply payload. This is not a broken existing contract because the current protocol has no auto-forward opt-in.

## Solution Assessment

Unclear. A narrow explicit opt-in may be the right implementation if maintainers approve the product direction, but automatic fallback forwarding when the agent returns no media would be a broader security and policy change than the current agent-controlled delivery contract.
