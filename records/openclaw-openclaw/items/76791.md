## Summary

Keep open: this PR fixes the request-header half of the Google Chat auth regression, but the reporter retested it and still hits the same certificate-retrieval path on response headers. The patch needs a response-header repair and changelog entry before merge.

## What This Changes

The PR adds a Google Chat Gaxios request interceptor that wraps non-native request header bags in `Headers` and adds unit coverage for request-header normalization and native `Headers` passthrough.

## Best Possible Solution

Keep the fix in `extensions/googlechat` by preserving native `Headers` semantics for both request and response objects across the shared Google auth transport, with regression coverage and a single-line changelog entry.

## Reproduction Assessment

Yes, source and reporter-assisted: configuring Google Chat app-url auth and sending an inbound message exercises `OAuth2Client.verifyIdToken()`, and the reporter retested this PR with a new `res?.headers.get is not a function` failure on the same certificate path. I did not run a live webhook locally.

## Solution Assessment

No, not yet: normalizing request headers is a narrow and appropriate direction, but the patch does not cover the response headers that google-auth reads after certificate retrieval. The safer solution is to repair that response side and keep the SSRF-guarded transport boundary intact.
