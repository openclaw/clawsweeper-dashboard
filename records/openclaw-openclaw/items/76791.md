## Summary

Keep this PR open: it addresses the first Google Chat auth crash on request headers, but the latest reporter feedback says the branch still fails during certificate retrieval on `res?.headers.get`, and the pinned google-auth-library code does read response headers after `transporter.request()`. The PR also still lacks the required user-facing changelog entry.

## What This Changes

The PR adds a Google Chat Gaxios request interceptor that converts plain-object request headers back to native `Headers` and adds unit coverage for request-header normalization and passthrough.

## Best Possible Solution

Complete the plugin-owned Google auth transport fix so both request and response headers remain native `Headers` through certificate retrieval, add focused regression coverage, and add the single-line `CHANGELOG.md` Fixes entry.

## Reproduction Assessment

Yes, at source level with reporter confirmation: Google Chat app-url auth reaches `OAuth2Client.verifyIdToken()` through `getGoogleAuthTransport()`, the pinned dependency reads both request and response header methods, and the PR branch comment reports the remaining response-header crash. I did not run a local live webhook repro.

## Solution Assessment

No. Normalizing request headers in the Google Chat transport is the right boundary, but the branch is incomplete because certificate retrieval still needs response headers with `.get()` and the required changelog entry is missing.
