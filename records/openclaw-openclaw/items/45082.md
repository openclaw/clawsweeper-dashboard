## Summary

Current main still appears to have the reported Mattermost thread-context gap, and the issue is explicitly paired with open closing PRs, so it should stay open for PR review or replacement rather than be swept closed.

## What This Changes

The issue reports that Mattermost `message(action="send")` and cron/heartbeat replies from thread-scoped sessions lose the active thread when no explicit `replyTo` is provided.

## Best Possible Solution

Keep the issue open and resolve it by reviewing or replacing the linked Mattermost PRs with a current-main fix that adds plugin-owned tool-context/auto-thread handling and per-thread delivery-context persistence with focused regression tests.

## Reproduction Assessment

Yes. The issue has concrete Mattermost `replyToMode: "all"` steps and current-main source shows the send action lacks a `toolContext.currentThreadTs` fallback while the cron path still only records last-route context for direct sessions.

## Solution Assessment

No, this issue should not be closed now. The right direction is a current-main Mattermost fix through the linked PRs or a replacement; closing before those open closing PRs are resolved would be premature.
