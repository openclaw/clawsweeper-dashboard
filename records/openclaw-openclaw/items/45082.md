## Summary

Keep open. Current main still lacks the Mattermost-owned thread context needed for implicit same-thread message-tool sends, and the cron/heartbeat route persistence path still only updates last-route metadata for DMs. The issue is also paired with open closing PRs that target these exact paths, so it should remain open until those PRs land or are replaced.

## What This Changes

The issue requests Mattermost proactive `message` tool sends and cron/heartbeat wake replies to inherit the active thread context when a session is thread-scoped and no explicit reply target is supplied.

## Best Possible Solution

Mattermost should provide guarded plugin threading context for same-target tool sends and persist thread-aware last-route metadata for channel/thread sessions, with regressions for explicit `replyTo`, implicit same-thread sends, cross-channel suppression, `replyToMode`, and cron/heartbeat wake delivery.

## Reproduction Assessment

Yes. The issue gives concrete Mattermost steps, and static current-main inspection shows the same failure path: the plugin send action has no implicit thread fallback and the route persistence path skips non-DM sessions.

## Solution Assessment

Yes. The narrow maintainable direction is a Mattermost plugin fix plus route metadata persistence, using the existing channel threading seams rather than adding a core special case for one bundled plugin.
