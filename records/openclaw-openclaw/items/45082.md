## What This Changes

Ensure Mattermost proactive message-tool and cron/heartbeat deliveries inherit the active thread context instead of posting at the channel root or in a DM.

## Best Possible Solution

Refresh or replace the linked work so the message tool receives current Mattermost thread context through the existing threading contract and scheduled routes are persisted on the exact thread session entry, then verify both flows in a real Mattermost setup.

## Solution Assessment

Yes in direction, but not yet in the current linked patch shapes: the narrow maintainable solution is to use existing threading context and persist delivery state on the exact thread-scoped session, avoiding plugin-local caches and shared-main-session contamination.
