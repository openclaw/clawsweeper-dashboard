## Summary

Keep open. Current main still lacks the Mattermost message-tool thread inheritance hook and still omits non-direct Mattermost last-route persistence, while open closing PRs already cover the reported paths and have not landed.

## What This Changes

The issue reports that Mattermost proactive `message` sends and cron/heartbeat wake replies should inherit the active thread context when no explicit reply target is supplied.

## Best Possible Solution

Land or replace the linked Mattermost work with a current-main fix that covers both message-tool thread inheritance and cron/heartbeat route persistence, coordinated with https://github.com/openclaw/openclaw/pull/52120, https://github.com/openclaw/openclaw/pull/52236, and related route work in https://github.com/openclaw/openclaw/pull/77442.

## Reproduction Assessment

Yes, at source level. Current main lacks the Mattermost `buildToolContext`/`resolveAutoThreadId` hooks for message-tool sends and still omits non-direct Mattermost route persistence while carrying thread context in inbound sessions.

## Solution Assessment

Yes in direction, but the issue should remain open until the linked implementation path lands or is replaced. The maintainable fix belongs in Mattermost-owned thread context plus safe session route persistence, not in closing this report ahead of the open closing PRs.
