## Summary

Keep open. Current main still lacks Mattermost-owned thread context for implicit same-thread message-tool sends, and the wake/cron route path still does not persist channel/thread delivery route metadata through Mattermost inbound session recording; open closing PRs #52120 and #52236 already target these paths.

## What This Changes

The issue requests that Mattermost proactive message-tool sends and cron or heartbeat wake replies inherit the active thread context when no explicit reply target is provided.

## Best Possible Solution

Review and land or replace the open Mattermost PRs so the plugin builds thread-aware tool context for message sends and persists channel/thread delivery context for wake routing.

## Reproduction Assessment

Yes, as a source-reproducible current-main path rather than a live Mattermost run. The issue gives concrete steps, docs define the expected thread-session behavior, and the current send-action and session-recording code still show the missing thread context paths.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the Mattermost plugin and its session-route persistence, using existing threading seams rather than adding a core Mattermost special case.
