## Summary

This issue is still necessary: current main and the latest release still leave the implicit Mattermost thread-context path unresolved, and the linked implementation PRs remain open/stale rather than merged.

## What This Changes

Mattermost proactive message-tool sends and cron/heartbeat wake replies should inherit the active thread context instead of posting to the channel root or a DM when no explicit reply target is provided.

## Best Possible Solution

Land a refreshed Mattermost-owned fix that supplies thread-root tool context for implicit message-tool sends and safely persists thread-scoped delivery context for scheduled wake replies without contaminating main-session routes.

## Reproduction Assessment

Yes from source inspection: a thread-scoped Mattermost message-tool send without explicit replyTo still reaches code paths that use replyToId/currentMessageId rather than the active thread root, and the cron route recording remains DM-gated. I did not run a live Mattermost send in this read-only review.

## Solution Assessment

No, current main is not the best end state yet; https://github.com/openclaw/openclaw/pull/73060 covers explicit threadId only, while the stale linked PRs need refresh or replacement for the implicit active-thread and cron/heartbeat cases.
