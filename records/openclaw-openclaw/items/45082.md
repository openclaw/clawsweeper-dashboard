## Summary

Keep open. Current main and the latest release still lack the Mattermost route-time tool-thread hooks and non-direct route persistence needed for this issue, while the linked fixes are open, abandoned, draft, or only partial.

## What This Changes

The issue reports that Mattermost proactive `message` tool sends and cron/heartbeat wake replies lose active thread context and post at the channel root or DM instead of the current thread.

## Best Possible Solution

Consolidate the linked Mattermost work into a current-main fix that feeds implicit Mattermost thread IDs through shared route-time message-action threading and persists safe non-direct session delivery context, with focused tests and redacted live Mattermost proof.

## Reproduction Assessment

Yes, at source level: current main lacks Mattermost `buildToolContext`/`resolveAutoThreadId`, the handler only reads explicit reply ids, and non-direct Mattermost route persistence is still skipped; no live Mattermost workspace was run in this read-only review.

## Solution Assessment

No complete fix is on main. The right direction is Mattermost-owned route-time threading hooks plus safe shared or active-session route persistence; the existing abandoned PRs should be consolidated or replaced.
