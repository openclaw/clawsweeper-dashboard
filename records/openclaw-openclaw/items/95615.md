## Summary

Keep open: the PR targets a real CLI-backed cron regression, but the current patch has blocking security-boundary correctness issues and still lacks after-fix real behavior proof.

## What This Changes

The PR adds a persisted cron `toolsAllowIsDefault` flag and drops flagged default `toolsAllow` caps when scheduled `agentTurn` runs resolve to a CLI backend.

## Best Possible Solution

Keep CLI runs fail-closed unless the stored cap is proven non-restrictive, explicitly update or clear the persisted default flag on every `toolsAllow` change, and add real CLI cron proof plus regression coverage.

## Reproduction Assessment

Yes. Current main passes cron payload `toolsAllow` into CLI-backed runs, and `prepareCliRunContext` rejects any defined `toolsAllow`; I did not run a live cron job in this read-only review.

## Solution Assessment

No. Runtime provider resolution is the right layer, but this patch drops creator-surface caps that may be restrictive and does not safely clear stale default flags when explicit `toolsAllow` changes.
