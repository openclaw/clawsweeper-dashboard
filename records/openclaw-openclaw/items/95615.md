## Summary

Keep this PR open, but not merge-ready: the reported CLI-backed cron failure is source-reproducible, yet the patch can drop a policy-restricted creator-surface cap on CLI and can lose the new default flag during later updates; contributor real-run proof is also still missing.

## What This Changes

The PR adds a persisted `toolsAllowIsDefault` cron payload flag and drops flagged default `toolsAllow` values for CLI-backed scheduled `agentTurn` runs while preserving explicit restrictions.

## Best Possible Solution

Keep explicit or policy-restricted caps fail-closed on CLI, drop only a maintainer-approved non-restrictive default, preserve the flag across updates, and require redacted real CLI cron proof before merge.

## Reproduction Assessment

Yes, at source level: current main auto-stamps cron `toolsAllow`, forwards it to CLI-backed runs, and `prepareCliRunContext` rejects any defined runtime allow-list. I did not run a live cron job in this read-only review.

## Solution Assessment

No. Runtime provider resolution is the right layer, but this implementation conflates restricted creator-surface defaults with unrestricted defaults and loses the new flag on some updates.
