- `merge-risk: 🚨 compatibility`: The diff adds a persisted cron payload flag and changes upgrade/restart behavior for scheduled `agentTurn` jobs.

## Summary

Keep open: the PR targets a real CLI-backed cron regression, but the current patch drops creator-surface caps on CLI even when that cap is the restricted tool boundary, and it can also lose the new persisted default flag during later cron-tool updates.

## What This Changes

The PR adds a persisted `toolsAllowIsDefault` flag for cron `agentTurn` payloads and drops flagged default `toolsAllow` caps when a scheduled run resolves to a CLI backend.

## Best Possible Solution

Preserve fail-closed behavior for restricted creator-surface caps, drop only caps proven non-restrictive if maintainers accept that policy, persist the flag through later updates, and add real CLI cron proof.

## Reproduction Assessment

Yes, from source inspection: current main auto-stamps/passes cron `toolsAllow` into CLI-backed runs, and `prepareCliRunContext` rejects any defined `toolsAllow`. I did not run a live cron job in this read-only review.

## Solution Assessment

No. Runtime provider resolution is the right layer, but this patch conflates unrestricted defaults with restricted creator tool surfaces and does not preserve the new flag through later updates.

  - body: The cap created by https://github.com/openclaw/openclaw/pull/91499 is not only a user-requested restriction; it can be the creator's effective restricted tool surface. When a restricted creator schedules a CLI-backed turn with no explicit `toolsAllow`, this branch drops that cap and runs without the fail-closed guard in `prepareCliRunContext`, reopening the authorization boundary instead of preserving or rejecting an unenforceable restriction.
  - confidence: 0.9
- **[P2] Preserve the default flag through updates:** `src/agents/tools/cron-tool.ts:566-569`
  - body: A job created by this patch with `toolsAllowIsDefault: true` loses that flag when the cron tool later updates the job without an explicit `toolsAllow`: this path passes only `existingPayload.toolsAllow`, so `capCronAgentTurnToolsAllow` treats the array as an explicit restriction and deletes the flag. After reload, the same CLI-backed scheduled turn again passes `toolsAllow` to `prepareCliRunContext` and fails.
  - confidence: 0.88
