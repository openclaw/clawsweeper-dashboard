## Summary

Keep open. Current main still lacks the requested `gateway.version` JSON field and still drops the `status.runtimeVersion` payload on the `--require-rpc` path; the open closing PR at https://github.com/openclaw/openclaw/pull/73225 remains the right implementation path to review or land.

## What This Changes

The issue asks for `openclaw gateway status --json` and MCP `gateway_status` output to expose the running Gateway version for post-update and CLI/service skew checks.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/73225, or an equivalent narrow patch, so daemon status exposes `gateway.version` from handshake metadata with `status.runtimeVersion` as the read-probe fallback and the MCP/status wrapper shares that structured field.

## Reproduction Assessment

Yes. Source inspection on current main shows `GatewayStatusSummary` omits `gateway.version`, and the `--require-rpc` probe discards the `status.runtimeVersion` payload that already exists in the status RPC.

## Solution Assessment

Yes. Reusing existing handshake `server.version` metadata and falling back to the existing `status.runtimeVersion` payload is the narrow maintainable fix; no new RPC method, auth model, config option, or migration is needed.
