## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Split long multiline payloads before writing PRIVMSG:** `extensions/irc/src/client.ts:249`
  - body: With `textChunkLimit` raised to 16384, a response chunk can contain a single long markdown/code line. This branch writes that whole line as one tagged `PRIVMSG`, which still has to fit IRC's line-size limit; compliant servers can truncate or reject it. Split long logical lines into IRC-safe frames and use `draft/multiline-concat` for continuations before closing the batch.
  - confidence: 0.94
- **[P2] Keep LF normalization out of generic command sanitizing:** `extensions/irc/src/protocol.ts:147`
  - body: `sanitizeIrcOutboundText` is still used for NickServ, USER, QUIT, and other raw command payloads, but this change preserves standalone `\n` and leaves `sendRaw` to remove it entirely. That regresses current behavior where newline separators become spaces, so multiline message bodies need a separate sanitizer instead of changing the generic command sanitizer.
  - confidence: 0.86
- **[P2] Honor server multiline byte limits:** `extensions/irc/src/client.ts:325-333`
  - body: The `draft/multiline` capability advertises a required `max-bytes` value, but this handler only checks whether the token exists and later batches up to the static 16384 default. Servers that advertise a lower limit can reject otherwise valid OpenClaw replies; parse the capability value and clamp multiline batches to it.
  - confidence: 0.82
