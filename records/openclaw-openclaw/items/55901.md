## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Negotiate batch before using multiline batches:** `extensions/irc/src/client.ts:333`
  - body: `draft/multiline` depends on the IRCv3 `batch` capability, but this code only requests `draft/multiline` and then sends `BATCH` frames once that single capability is ACKed. Servers that require `batch` to be negotiated can reject or ignore those frames, so multiline delivery becomes server-dependent instead of protocol-safe.
  - confidence: 0.87
- **[P1] Honor the server's multiline max-bytes limit:** `extensions/irc/src/channel.ts:339`
  - body: The server advertises `draft/multiline` with a required `max-bytes` value, but the PR hard-codes a 16384 text chunk limit and never parses the advertised limit. A server advertising a smaller limit can reject otherwise valid OpenClaw replies with `MULTILINE_MAX_BYTES`, so the chunk limit needs to come from the negotiated capability or fall back safely.
  - confidence: 0.84
- **[P2] Split long lines inside multiline batches:** `extensions/irc/src/client.ts:249`
  - body: Each newline-delimited row is written as one tagged `PRIVMSG`, but the ordinary IRC message-section limit still applies to every row. A long URL or code line can exceed the per-line frame limit even though the total batch is allowed, causing truncation or rejection; split long rows with the multiline concat tag or an equivalent IRC-safe line strategy before writing.
  - confidence: 0.9
