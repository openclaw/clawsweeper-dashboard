## Summary

Keep this PR open: current main still lacks IRC draft/multiline support, but the branch is not merge-ready because it negotiates only `draft/multiline`, emits long multiline lines as single IRC frames, broadens newline preservation through a shared sanitizer, and lacks the required changelog/test/current-main integration work.

## What This Changes

The PR adds an IRC newline-preserving chunker, IRCv3 tag parsing, draft/multiline CAP negotiation, BATCH-based outbound/inbound multiline handling, and a larger IRC outbound chunk target.

## Best Possible Solution

Land a rebased IRC plugin change that negotiates both `batch` and `draft/multiline`, emits byte-safe frames with `draft/multiline-concat` for long logical lines, scopes newline preservation to message bodies, updates the current outbound adapter, and adds focused tests plus a changelog entry.

## Reproduction Assessment

Yes. Current main's feature gap is reproducible by static inspection: IRC outbound text is newline-flattened and emitted as plain `PRIVMSG`. The PR regression is reproducible by sending a multiline response containing one long URL or code line after multiline is enabled; the PR head writes that line as one tagged `PRIVMSG` over the IRC frame limit.

## Solution Assessment

No, not as-is. `draft/multiline` is the right owner-local direction for preserving IRC markdown replies, but this implementation must first satisfy the IRCv3 batch dependency, frame-size rules, sanitizer call-site safety, current-main adapter shape, tests, and changelog policy.
