## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Split long batched lines before writing:** `extensions/irc/src/client.ts:249`
  - body: In the multiline branch, each newline-separated logical line is written as one tagged `PRIVMSG`. A long URL or code line can exceed IRC's per-line budget and be truncated or rejected; split overlong logical lines inside the batch and use `draft/multiline-concat` for continuations so reconstructed text does not gain extra newlines. ([ircv3.net](https://ircv3.net/specs/extensions/multiline?utm_source=openai))
  - confidence: 0.93
- **[P2] Request the batch capability with draft/multiline:** `extensions/irc/src/client.ts:333`
  - body: The IRCv3 draft/multiline spec depends on the `batch` capability, but this code sends `CAP REQ` for `draft/multiline` only. Request and confirm the required capability set before sending or buffering `BATCH` messages. ([ircv3.net](https://ircv3.net/specs/extensions/multiline?utm_source=openai))
  - confidence: 0.84
- **[P2] Honor advertised multiline limits:** `extensions/irc/src/client.ts:332-333`
  - body: The LS handler only checks whether the capability string contains `draft/multiline` and ignores the required `max-bytes` value and optional `max-lines` value. With `textChunkLimit` raised to 16384, servers advertising lower limits can reject complete batches with multiline limit errors. ([ircv3.net](https://ircv3.net/specs/extensions/multiline?utm_source=openai))
  - confidence: 0.88
- **[P2] Handle multiline-concat when rebuilding batches:** `extensions/irc/src/client.ts:462`
  - body: The receiver joins every buffered batch message with LF. Valid multiline senders use `draft/multiline-concat` to split a long logical line without adding a line break, so current inbound handling corrupts those messages by inserting extra newlines. ([ircv3.net](https://ircv3.net/specs/extensions/multiline?utm_source=openai))
  - confidence: 0.86
- **[P2] Keep newline preservation out of generic command sanitization:** `extensions/irc/src/protocol.ts:147-148`
  - body: `sanitizeIrcOutboundText` is also used for NickServ, USER, and QUIT payloads that are later sent through `sendRaw`. Preserving standalone LF here means those non-message commands have newlines stripped and tokens joined instead of safely separated; keep LF preservation local to PRIVMSG multiline handling or flatten non-message callers.
  - confidence: 0.78
