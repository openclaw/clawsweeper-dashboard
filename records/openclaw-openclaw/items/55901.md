## Summary

Keep this PR open, but do not merge it as-is. Current main still lacks IRCv3 draft/multiline support, so the contribution is not obsolete; the PR remains relevant but needs protocol fixes, current-main rebase work, and focused tests before it is a safe bundled IRC plugin change.

## What This Changes

This PR adds IRCv3 draft/multiline CAP negotiation, IRCv3 tag and BATCH parsing, newline-preserving IRC chunking/sanitization, and a larger outbound text chunk target for multiline markdown replies.

## Best Possible Solution

Land a rebased IRC plugin implementation that negotiates all required IRCv3 capabilities, keeps every emitted IRC frame within safe byte limits, preserves markdown line breaks through draft/multiline, retains legacy fallback chunking, and adds focused unit tests for CAP/BATCH send and receive behavior plus command-argument sanitization.

## Reproduction Assessment

Yes. For the main PR finding, connect to a server advertising draft/multiline and send a multiline reply containing one long code or URL line; PR head writes that line as one tagged PRIVMSG, which exceeds standard IRC frame limits. The current-main feature gap is also reproducible by static inspection: all outbound newlines are flattened and no CAP/BATCH path exists.

## Solution Assessment

No, not yet. draft/multiline is the right product direction for preserving IRC markdown replies, but this implementation needs required batch CAP negotiation, per-frame splitting, current-main outbound adapter integration, and tests before it is the narrowest maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Negotiate batch before using draft/multiline:** `extensions/irc/src/client.ts:333`
  - body: The multiline spec depends on the `batch` capability, but this branch only requests `draft/multiline` and then starts sending `BATCH` frames. Servers that enforce CAP dependencies can reject or ignore the batch flow even though `multilineCap` is set, so request and ACK both capabilities before enabling this path.
  - confidence: 0.86
- **[P2] Split long multiline payload lines:** `extensions/irc/src/client.ts:249`
  - body: When multiline is active, each newline-delimited payload is written as a single tagged `PRIVMSG`. A long markdown code line or URL can exceed IRC's per-frame limit and be truncated or rejected; split long logical lines into safe frames, using `draft/multiline-concat` where needed to preserve the original line.
  - confidence: 0.92
- **[P3] Flatten LF for non-message command arguments:** `extensions/irc/src/protocol.ts:147-148`
  - body: `sanitizeIrcOutboundText` now preserves standalone `\n`, but the same helper still feeds NickServ, QUIT, and USER command arguments before `sendRaw` removes newlines entirely. Inputs with LF or escaped `\n` now concatenate tokens instead of the old space-flattened value, so keep newline preservation message-specific or flatten LF at non-message call sites.
  - confidence: 0.75
