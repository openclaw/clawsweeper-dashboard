## Summary

Keep open: current main still lacks IRC draft/multiline support, and this PR remains a plausible IRC-plugin feature but is not mergeable because the head is conflicting and still violates key IRCv3 multiline requirements around batch negotiation, advertised limits, long-line handling, concat reconstruction, and sanitizer scope.

## What This Changes

The PR adds IRCv3 draft/multiline CAP negotiation, LF-preserving IRC chunking and sanitization, outbound BATCH sending, and inbound multiline batch aggregation to the bundled IRC plugin.

## Best Possible Solution

Land a rebased IRC plugin implementation that negotiates batch plus draft/multiline, parses advertised max limits, emits byte-safe multiline-concat frames, reconstructs incoming concat batches correctly, keeps generic command sanitization safe, and adds focused IRC regression tests.

## Reproduction Assessment

Yes. Current main's missing feature is source-reproducible from the outbound IRC code flattening newlines into ordinary PRIVMSG frames, and the PR blockers are source/spec reproducible from the PR head and the IRCv3 multiline contract.

## Solution Assessment

No, not as-is. The IRC plugin is the right place for the feature, but the patch needs required batch negotiation, advertised-limit handling, byte-safe multiline-concat splitting, concat-aware receive reconstruction, and narrower LF preservation before it is the maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Request batch before enabling multiline:** `extensions/irc/src/client.ts:333`
  - body: The IRCv3 multiline spec depends on the `batch` capability, but this branch only requests `draft/multiline` and then sends BATCH frames. Request and require ACKs for both capabilities before enabling multiline sends.
  - confidence: 0.91
- **[P2] Honor advertised multiline limits:** `extensions/irc/src/client.ts:325-333`
  - body: The `draft/multiline` capability value carries required `max-bytes` and optional `max-lines` limits, but this branch only checks for the capability name and uses a hard-coded 16384 chunk limit. Parse the advertised values and keep emitted batches within them so servers do not reject valid-looking sends.
  - confidence: 0.87
- **[P2] Split long multiline lines with concat:** `extensions/irc/src/client.ts:249`
  - body: When multiline is active, each logical line is written as one tagged PRIVMSG. Long markdown code lines or URLs can exceed IRC's per-frame limit, so split long logical lines into safe frames and use `draft/multiline-concat` where the original line must be preserved.
  - confidence: 0.93
- **[P2] Preserve concat semantics on receive:** `extensions/irc/src/client.ts:462`
  - body: Incoming multiline batches are reconstructed with `messages.map(...).join("\n")`, so a server-sent `draft/multiline-concat` continuation would incorrectly gain a newline. Track that tag for buffered batch lines and concatenate those fragments without inserting LF.
  - confidence: 0.84
- **[P3] Keep generic sanitizer flattening LF:** `extensions/irc/src/protocol.ts:147-148`
  - body: `sanitizeIrcOutboundText` now preserves standalone LF, but NickServ, QUIT, and USER construction also use this helper before `sendRaw` strips newlines. Keep the generic sanitizer flattening CR/LF, or add a message-specific sanitizer for multiline payloads, so non-message command values do not concatenate tokens.
  - confidence: 0.82
