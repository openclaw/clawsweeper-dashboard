## Summary

Keep this PR open but do not merge it as-is. Current main still lacks IRC draft/multiline support, and the contribution is useful, but the latest diff still misses the required `batch` CAP negotiation, can exceed IRC frame limits for long multiline payload lines, changes a generic sanitizer used by non-message commands, and lacks the required user-facing changelog entry.

## What This Changes

The PR adds IRCv3 tag parsing, draft/multiline CAP negotiation, BATCH-based inbound/outbound multiline handling, a newline-preserving IRC chunker, and a larger IRC outbound chunk target.

## Best Possible Solution

Land a rebased IRC plugin change that negotiates both `batch` and `draft/multiline`, emits byte-safe multiline frames, scopes newline preservation to message payloads, preserves legacy fallback chunking, and adds focused tests plus a changelog entry.

## Reproduction Assessment

Yes. Current main's feature gap is reproducible by static inspection: IRC outbound text is newline-flattened and sent as plain `PRIVMSG`; the PR regressions are reproducible with a server requiring `batch`, a multiline response containing one long logical line, or a NickServ/USER/QUIT argument containing LF.

## Solution Assessment

No, not as-is. `draft/multiline` is the right IRC-plugin direction, but this patch needs the IRCv3 dependency negotiation, frame splitting, sanitizer scoping, tests, and changelog policy fixes before it is the narrowest maintainable solution.

Full review comments:

- **[P2] Negotiate batch before enabling multiline:** `extensions/irc/src/client.ts:330`
  - body: The multiline spec depends on the `batch` capability, but this branch requests only `draft/multiline` and then sends `BATCH` frames. Servers that enforce CAP dependencies can reject or ignore this flow, so request and require ACKs for both capabilities before enabling multiline sends.
  - confidence: 0.9
- **[P2] Split long multiline payload lines:** `extensions/irc/src/client.ts:249`
  - body: When multiline is active, each newline-delimited payload is written as one tagged `PRIVMSG`. A long code line or URL can exceed IRC's per-frame limit and be truncated or rejected; split long logical lines into safe frames, using `draft/multiline-concat` where needed to preserve the original line.
  - confidence: 0.92
- **[P3] Keep LF preservation scoped to messages:** `extensions/irc/src/protocol.ts:147`
  - body: `sanitizeIrcOutboundText` now preserves standalone LF, but this helper also feeds NickServ, QUIT, and USER command arguments before `sendRaw` removes newlines entirely. Inputs with LF now concatenate tokens instead of the old space-flattened value, so keep newline preservation message-specific or flatten LF at non-message call sites.
  - confidence: 0.78

  - body: This is a user-facing IRC feature, but the diff does not update `CHANGELOG.md`. Add a single-line entry in the active section before landing.
