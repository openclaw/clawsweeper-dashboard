## Summary

Keep this PR open but not merge-ready: the IRC multiline feature is useful and has visible happy-path proof, but the current patch still misses protocol-required negotiation, advertised limits, and concat-aware send/receive handling that can corrupt or drop IRC message content.

## What This Changes

This PR adds IRCv3 `draft/multiline` support in the IRC plugin by preserving outbound newlines, negotiating a multiline capability, sending and receiving BATCH frames, parsing IRCv3 tags, and changing IRC chunking limits.

## Best Possible Solution

Land a rebased IRC plugin implementation that negotiates both `batch` and `draft/multiline`, obeys advertised multiline limits, handles concat-aware byte-safe frames in both directions, keeps LF preservation message-scoped, and adds focused regression tests.

## Reproduction Assessment

Yes for the PR blockers: the PR head source and IRCv3 multiline spec show the missing `batch` negotiation, advertised limit handling, and concat-aware splitting/reconstruction paths. Current main does not implement the feature, so this is not a current-main bug reproduction.

## Solution Assessment

No, not as-is. The feature belongs in the IRC plugin, but the implementation needs protocol-complete negotiation, byte/line limit enforcement, concat handling, and a rebase onto the current IRC outbound boundary before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Negotiate batch before enabling multiline:** `extensions/irc/src/client.ts:333`
  - body: The IRCv3 multiline spec depends on the `batch` capability, but this handler requests only `draft/multiline` before later sending BATCH frames. Request and require ACKs for both capabilities before setting `multilineCap`.
  - confidence: 0.92
- **[P2] Honor advertised multiline limits:** `extensions/irc/src/client.ts:325-333`
  - body: Servers advertise `draft/multiline` with required `max-bytes` and optional `max-lines` values, but this code only checks for the capability name and then sends up to the fixed 16384 chunk limit. Parse those values and keep each batch within the server's advertised limits.
  - confidence: 0.9
- **[P2] Split long multiline lines with concat:** `extensions/irc/src/client.ts:249`
  - body: The multiline path writes each logical line as one tagged `PRIVMSG`, so a long code line or URL can still exceed IRC's per-frame limit and be truncated or rejected. Split long logical lines into byte-safe frames and mark continuations with `draft/multiline-concat`.
  - confidence: 0.93
- **[P2] Preserve concat continuations on receive:** `extensions/irc/src/client.ts:462`
  - body: Incoming multiline batches are always reconstructed with `join("\n")`, so a `draft/multiline-concat` continuation would gain an extra newline. Track the concat tag for buffered batch lines and append those fragments without inserting LF.
  - confidence: 0.88
- **[P3] Keep LF preservation out of generic commands:** `extensions/irc/src/protocol.ts:147-148`
  - body: `sanitizeIrcOutboundText` now preserves standalone LF, but NickServ, QUIT, and USER construction also use this helper before `sendRaw` strips newlines entirely. Keep the generic sanitizer flattening CR/LF, or introduce a message-specific sanitizer only for multiline payloads.
  - confidence: 0.82
