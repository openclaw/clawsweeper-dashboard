## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Split long multiline batch lines:** `extensions/irc/src/client.ts:249`
  - body: The multiline path writes each newline-separated `line` as one tagged `PRIVMSG`. IRCv3 still requires each protocol line to fit within the IRC line limit, and the spec recommends splitting long logical lines; with the new 16KB chunk limit, a long URL or code line can be truncated or rejected. Split long lines into IRC-safe frames, using `draft/multiline-concat` when the displayed logical line must remain joined.
  - confidence: 0.93
- **[P2] Negotiate the required batch capability:** `extensions/irc/src/client.ts:333`
  - body: This requests only `draft/multiline`, but the IRCv3 multiline spec says the `batch` capability must also be negotiated before multiline batches are used. Enable multiline only after the required capability set is advertised and ACKed; otherwise stricter servers or bridges can reject or ignore the BATCH traffic.
  - confidence: 0.86
- **[P2] Honor advertised multiline limits:** `extensions/irc/src/client.ts:325-333`
  - body: The CAP LS handler only checks whether the capability string contains `draft/multiline`, while the spec requires servers to advertise `max-bytes` and may advertise `max-lines`. Sending fixed 16KB batches without parsing those limits can produce `MULTILINE_MAX_BYTES` or `MULTILINE_MAX_LINES` failures on valid servers, so split or cap batches using the negotiated values.
  - confidence: 0.84
