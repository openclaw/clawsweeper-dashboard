## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Split long batched lines before writing:** `extensions/irc/src/client.ts:249`
  - body: The multiline branch writes each newline-separated line as one tagged PRIVMSG. A long URL or code line can exceed IRC's per-line frame budget and be truncated or rejected; split overlong logical lines inside the batch and preserve reconstruction with draft/multiline-concat.
  - confidence: 0.94
- **[P2] Negotiate the required batch capability:** `extensions/irc/src/client.ts:333`
  - body: The IRCv3 multiline contract depends on the batch capability, but this code sends CAP REQ for draft/multiline only. Request and confirm the required capability set before sending or buffering BATCH messages.
  - confidence: 0.9
- **[P2] Honor advertised multiline limits:** `extensions/irc/src/client.ts:332-333`
  - body: The LS handler only checks whether the capability string contains draft/multiline and ignores the advertised max-bytes and max-lines values. With the channel chunk cap raised to 16384, servers advertising lower limits can reject complete batches.
  - confidence: 0.91
- **[P2] Preserve concat semantics when rebuilding batches:** `extensions/irc/src/client.ts:462`
  - body: The receiver joins every buffered batch message with LF. Valid multiline senders use draft/multiline-concat to split a long logical line without adding a line break, so current inbound handling corrupts those messages by inserting extra newlines.
  - confidence: 0.88
- **[P2] Keep LF preservation out of generic IRC sanitizer:** `extensions/irc/src/protocol.ts:148`
  - body: sanitizeIrcOutboundText also feeds USER, QUIT, and NickServ commands that later go through sendRaw. Preserving standalone LF here lets sendRaw remove it without a separator, turning values such as secret\nJOIN into joined tokens; keep LF preservation local to PRIVMSG multiline sending.
  - confidence: 0.8
