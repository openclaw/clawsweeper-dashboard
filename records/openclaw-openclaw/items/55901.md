## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Negotiate batch before enabling multiline sends:** `extensions/irc/src/client.ts:333`
  - body: The IRCv3 multiline spec depends on the `batch` capability, but this branch only requests `draft/multiline` and then starts sending `BATCH` frames after that ACK. Request and require both `batch` and `draft/multiline` before setting `multilineCap`, otherwise compliant servers can reject or ignore the new flow.
  - confidence: 0.9
- **[P2] Honor advertised multiline size limits:** `extensions/irc/src/client.ts:325-333`
  - body: `CAP LS 302` advertises `draft/multiline` with required `max-bytes` data, but this code only checks for the substring and the channel still passes up to 16384 bytes into one batch. Parse the advertised limits and split or fall back before the server returns `MULTILINE_MAX_BYTES` or `MULTILINE_MAX_LINES`.
  - confidence: 0.86
- **[P1] Split long multiline lines before writing PRIVMSG:** `extensions/irc/src/client.ts:249`
  - body: With `textChunkLimit` raised to 16384, a single markdown/code line can be far larger than an IRC frame. This path writes that whole line as one tagged `PRIVMSG`, so long URLs or code lines can be truncated or rejected; split long logical lines into IRC-safe frames using `draft/multiline-concat`.
  - confidence: 0.94
- **[P2] Keep LF flattening for generic IRC commands:** `extensions/irc/src/protocol.ts:147-148`
  - body: `sanitizeIrcOutboundText` is still used for NickServ, USER, QUIT, and other raw command payloads, but preserving standalone `\n` leaves `sendRaw` to remove it entirely. That changes current behavior from separating tokens with a space to collapsing them, so multiline message bodies need a separate sanitizer.
  - confidence: 0.87
