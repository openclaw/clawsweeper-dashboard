## Summary

Keep open: current main still lacks IRC draft/multiline support, and the PR is useful, but the head still has source-provable IRCv3 protocol blockers that should be fixed before merge.

## What This Changes

The PR adds IRC draft/multiline CAP negotiation, IRCv3 tag parsing, LF-preserving outbound handling, a newline-preserving IRC chunker, and BATCH send/receive support.

## Best Possible Solution

Land a rebased IRC plugin implementation that negotiates batch plus draft/multiline, obeys advertised limits, handles concat-aware byte-safe frames in both directions, keeps LF preservation message-scoped, and adds focused regression tests.

## Reproduction Assessment

Yes. The PR blockers are source-reproducible from the head: current main lacks multiline support, while the PR head conflicts with IRCv3 multiline requirements around batch negotiation, limits, and concat handling.

## Solution Assessment

No, not as-is. The feature belongs in the IRC plugin, but the implementation needs protocol-complete negotiation, advertised-limit enforcement, concat-aware splitting/reconstruction, and scoped newline sanitization before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Negotiate batch before enabling multiline:** `extensions/irc/src/client.ts:333`
  - body: The IRCv3 multiline extension depends on the batch capability, but this handler requests only draft/multiline before sending BATCH frames. Request and require ACKs for both capabilities before enabling multiline sends.
  - confidence: 0.91
- **[P2] Honor advertised multiline limits:** `extensions/irc/src/client.ts:325-333`
  - body: draft/multiline is advertised with max-bytes and may include max-lines, but this code only checks for the capability name and keeps a fixed 16384 chunk limit. Parse the advertised values and keep each batch within the server's limits.
  - confidence: 0.89
- **[P2] Split long multiline lines with concat:** `extensions/irc/src/client.ts:249`
  - body: When multiline is active, each logical line is written as one tagged PRIVMSG. Long code lines or URLs can exceed the IRC frame limit, so split long logical lines into byte-safe frames and mark continuations with draft/multiline-concat where needed.
  - confidence: 0.93
- **[P2] Preserve concat continuations on receive:** `extensions/irc/src/client.ts:462`
  - body: Incoming multiline batches are reconstructed with join("\n"), so a draft/multiline-concat continuation would incorrectly gain a newline. Track that tag for buffered batch lines and concatenate those fragments without inserting LF.
  - confidence: 0.86
- **[P3] Keep newline preservation out of generic commands:** `extensions/irc/src/protocol.ts:147-148`
  - body: sanitizeIrcOutboundText now preserves standalone LF, but NickServ, QUIT, and USER construction also use this helper before sendRaw strips newlines. Keep the generic sanitizer flattening CR/LF, or add a message-specific sanitizer for multiline payloads.
  - confidence: 0.82
