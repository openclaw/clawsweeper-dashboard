## Summary

Keep this PR open, but it still needs changes before merge: current main does not support IRC draft/multiline, the PR has useful real-world proof, and the remaining blockers are concrete IRCv3 protocol/chunking issues in the proposed patch.

## What This Changes

The PR adds IRC draft/multiline capability negotiation, LF-preserving chunking, BATCH send/receive handling, IRCv3 tag parsing, and newline sanitizer changes for IRC messages.

## Best Possible Solution

Land a rebased IRC plugin implementation that negotiates batch plus draft/multiline, obeys advertised multiline limits, emits and receives concat-aware byte-safe frames, keeps LF preservation message-scoped, and adds focused regression tests.

## Reproduction Assessment

Yes. Current main's missing feature is source-reproducible from the IRC send path and sanitizer, and the remaining PR blockers are reproducible from the PR source against the IRCv3 multiline contract.

## Solution Assessment

No, not as-is. The IRC plugin is the right home for this feature, but the patch needs batch negotiation, advertised-limit handling, concat-aware send/receive behavior, and message-scoped LF sanitization before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Negotiate batch with multiline:** `extensions/irc/src/client.ts:333`
  - body: The multiline spec depends on the `batch` capability, but this branch requests only `draft/multiline` and then emits BATCH frames. Request and require ACKs for both `batch` and `draft/multiline` before enabling multiline sends.
  - confidence: 0.91
- **[P2] Honor advertised multiline limits:** `extensions/irc/src/client.ts:325-333`
  - body: `draft/multiline` is advertised with required `max-bytes` and optional `max-lines` values, but this code only checks for the capability name and uses a fixed 16384 chunk limit. Parse the advertised values and keep batches within them so servers do not reject valid-looking sends.
  - confidence: 0.88
- **[P2] Split long multiline lines with concat:** `extensions/irc/src/client.ts:249`
  - body: When multiline is active, each logical line is written as one tagged PRIVMSG. Long code lines or URLs can exceed IRC's per-frame limit, so split long logical lines into byte-safe frames and mark continuations with `draft/multiline-concat` where the original line must be preserved.
  - confidence: 0.93
- **[P2] Preserve concat continuations on receive:** `extensions/irc/src/client.ts:462`
  - body: Incoming multiline batches are reconstructed with `messages.map(...).join("\n")`, so a server-sent `draft/multiline-concat` continuation would incorrectly gain a newline. Track that tag for buffered batch lines and concatenate those fragments without inserting LF.
  - confidence: 0.84
- **[P3] Keep LF preservation message-scoped:** `extensions/irc/src/protocol.ts:147-148`
  - body: `sanitizeIrcOutboundText` now preserves standalone LF, but NickServ, QUIT, and USER construction also use this helper before `sendRaw` strips newlines. Keep the generic sanitizer flattening CR/LF, or add a message-specific sanitizer for multiline payloads, so non-message command values do not concatenate tokens.
  - confidence: 0.82
