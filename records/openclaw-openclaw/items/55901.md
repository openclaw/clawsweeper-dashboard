## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Negotiate batch before enabling multiline:** `extensions/irc/src/client.ts:333`
  - body: The multiline spec depends on the `batch` capability, but this request only asks for `draft/multiline` and enables multiline sends from that ACK alone. Servers that require `batch` negotiation can reject or ignore the BATCH frames, so the client should request/confirm both capabilities before using multiline.
  - confidence: 0.9
- **[P2] Honor advertised multiline limits:** `extensions/irc/src/client.ts:325-333`
  - body: The CAP parser reduces the advertisement to an `includes("draft/multiline")` check and discards required `max-bytes` and recommended `max-lines` values. With the channel limit raised to 16384, the client can emit batches that the server is required to fail for exceeding its advertised limits.
  - confidence: 0.88
- **[P2] Split long batch lines with concat:** `extensions/irc/src/client.ts:249`
  - body: Each newline-separated line is written as one tagged PRIVMSG without checking the IRC frame size. A long code line or URL can still exceed the 512-byte IRC line limit, so multiline sends need byte-safe splitting with `draft/multiline-concat` rather than one raw write per line.
  - confidence: 0.91
- **[P2] Honor concat tags when rebuilding batches:** `extensions/irc/src/client.ts:462`
  - body: Incoming batches are always rebuilt with `join("\n")`, but `draft/multiline-concat` fragments must be appended directly to the previous line without a line feed. Messages from conforming clients using concat will be delivered to OpenClaw with extra newlines inserted.
  - confidence: 0.87
- **[P2] Keep generic IRC sanitization newline-safe:** `extensions/irc/src/protocol.ts:147-148`
  - body: `sanitizeIrcOutboundText` now preserves standalone LF for every caller, including NickServ, USER, and QUIT paths that still pass through `sendRaw`, where LF is deleted rather than converted to a separator. Preserve newlines only in the message-specific multiline path and keep generic command text flattening CR/LF safely.
  - confidence: 0.74
