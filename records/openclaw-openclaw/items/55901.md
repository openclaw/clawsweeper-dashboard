## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Split long batched lines before writing:** `extensions/irc/src/client.ts:249`
  - body: In the multiline branch, each newline-separated line is written as one tagged `PRIVMSG`. A long URL or code line can exceed IRC's per-line budget and be truncated or rejected; split overlong logical lines inside the batch and preserve reconstruction with `draft/multiline-concat`.
  - confidence: 0.93
- **[P2] Negotiate batch and enforce multiline limits:** `extensions/irc/src/client.ts:333`
  - body: The IRCv3 draft/multiline spec depends on the `batch` capability and advertises a required `max-bytes` value, but this handler only checks for `draft/multiline` and requests only that capability. Servers advertising lower limits can reject complete batches once this PR raises the outbound chunk cap.
  - confidence: 0.88
- **[P2] Preserve concat semantics when rebuilding batches:** `extensions/irc/src/client.ts:462`
  - body: The receiver joins every buffered batch message with LF. Valid multiline senders use `draft/multiline-concat` to split a long logical line without adding a line break, so current inbound handling corrupts those messages by inserting extra newlines.
  - confidence: 0.86
- **[P2] Keep newline preservation out of generic IRC sanitizing:** `extensions/irc/src/protocol.ts:147-148`
  - body: `sanitizeIrcOutboundText` also feeds USER, QUIT, and NickServ commands that later go through `sendRaw`. Preserving standalone LF here lets `sendRaw` strip it without a separator, turning values such as `secret\nJOIN` into joined tokens; keep LF preservation in the PRIVMSG multiline path or flatten non-message callers before `sendRaw`.
  - confidence: 0.78
