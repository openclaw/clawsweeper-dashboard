## Summary

This PR should stay open because current main still lacks IRCv3 draft/multiline support, but the proposed implementation still has source-proven protocol blockers around required capability negotiation, advertised limits, per-frame splitting, and concat reconstruction.

## What This Changes

The PR adds IRCv3 tag parsing, LF-preserving IRC chunking, CAP negotiation, draft/multiline BATCH send/receive handling, and newline sanitizer changes in the IRC plugin.

## Best Possible Solution

Land a rebased IRC plugin implementation that negotiates batch plus draft/multiline, obeys advertised multiline limits, emits and receives concat-aware byte-safe frames, keeps LF preservation message-scoped, and adds focused regression tests.

## Reproduction Assessment

Yes. Current main's missing multiline capability is source-reproducible from the IRC send path and sanitizer, and the PR blockers are reproducible from the PR source against the IRCv3 multiline contract.

## Solution Assessment

No, not as-is. The IRC plugin is the right home for this feature, but the patch needs protocol-complete negotiation, advertised-limit handling, concat support, and safer per-frame splitting before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Negotiate batch before sending multiline batches:** `extensions/irc/src/client.ts:333`
  - body: The multiline extension depends on the IRCv3 `batch` capability, but this branch only requests `draft/multiline` before emitting `BATCH` commands and `@batch` tagged messages. Request and require ACKs for both `batch` and `draft/multiline` before enabling multiline sends so strict servers do not reject the frames.
  - confidence: 0.91
- **[P2] Honor advertised multiline limits:** `extensions/irc/src/client.ts:325-333`
  - body: `draft/multiline` is advertised with required `max-bytes` and optional `max-lines` values, but the client only checks for the capability name and uses a fixed 16384 limit. Parse the advertised values and keep outgoing batches within those server limits to avoid rejected sends on compliant servers.
  - confidence: 0.88
- **[P2] Split long multiline lines with concat:** `extensions/irc/src/client.ts:249`
  - body: When multiline is active, each logical line is written as one tagged `PRIVMSG`, so a long URL or code line can still exceed IRC's per-frame size limit. Split long logical lines into byte-safe frames and mark continuations with `draft/multiline-concat` when the original line must be preserved.
  - confidence: 0.93
- **[P2] Preserve concat continuations on receive:** `extensions/irc/src/client.ts:462`
  - body: Incoming multiline batches are reconstructed with `messages.map(...).join("\n")`, so a server-sent `draft/multiline-concat` continuation would incorrectly gain a newline. Track the concat tag for buffered batch entries and concatenate those fragments without inserting LF.
  - confidence: 0.84
- **[P3] Keep LF preservation message-scoped:** `extensions/irc/src/protocol.ts:147-148`
  - body: `sanitizeIrcOutboundText` now preserves standalone LF, but NickServ, QUIT, and USER construction also use this helper before `sendRaw` strips newlines. Keep the generic sanitizer flattening CR/LF, or add a message-specific sanitizer for multiline payloads, so non-message command values do not concatenate tokens.
  - confidence: 0.82
