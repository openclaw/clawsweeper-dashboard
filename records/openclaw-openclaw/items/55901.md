## Summary

This PR should remain open because current main still lacks IRC draft/multiline support and the branch contains useful IRC-plugin feature work, but the latest diff still has protocol correctness blockers around capability negotiation, advertised multiline limits, per-frame concat handling, receive-side reconstruction, and sanitizer scope.

## What This Changes

The PR adds IRC draft/multiline CAP negotiation, LF-preserving IRC chunking/sanitization, BATCH send/receive handling, and IRCv3 message tag parsing for the IRC plugin.

## Best Possible Solution

Land a rebased IRC plugin implementation that negotiates batch plus draft/multiline, obeys advertised server limits, splits byte-safe frames with concat support, preserves concat correctly on receive, and keeps LF preservation scoped to message payloads.

## Reproduction Assessment

Yes for the PR blockers: they are source-reproducible from the latest diff against the IRCv3 multiline contract and current main’s IRC client behavior. Current main also clearly lacks the requested multiline feature, so there is no implemented-on-main close path.

## Solution Assessment

No, not as-is. The feature belongs in the IRC plugin, but this implementation needs protocol-complete negotiation, advertised-limit handling, concat-aware splitting/reconstruction, and a narrower sanitizer change before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Negotiate batch before enabling multiline:** `extensions/irc/src/client.ts:333`
  - body: The IRCv3 multiline extension depends on the `batch` capability, but this handler requests only `draft/multiline` before the send path starts emitting `BATCH` frames. Request and require ACKs for both capabilities before setting `multilineCap` true.
  - confidence: 0.91
- **[P2] Honor advertised multiline limits:** `extensions/irc/src/client.ts:325-333`
  - body: `draft/multiline` is advertised with `max-bytes` and may include `max-lines`, but the PR only checks for the capability name and keeps a fixed 16384 chunk limit. Parse the advertised values and keep each outgoing batch within the server's limits.
  - confidence: 0.89
- **[P2] Split long multiline lines with concat:** `extensions/irc/src/client.ts:249`
  - body: When multiline is active, each logical line is written as one tagged `PRIVMSG`. Long code lines or URLs can exceed the IRC frame limit, so split long logical lines into byte-safe frames and mark continuations with `draft/multiline-concat` where needed.
  - confidence: 0.93
- **[P2] Preserve concat continuations on receive:** `extensions/irc/src/client.ts:462`
  - body: Incoming multiline batches are reconstructed with `join("\n")`, so a `draft/multiline-concat` continuation would incorrectly gain a newline. Track that tag for buffered batch lines and concatenate those fragments without inserting LF.
  - confidence: 0.86
- **[P3] Keep newline preservation out of generic commands:** `extensions/irc/src/protocol.ts:147-148`
  - body: `sanitizeIrcOutboundText` now preserves standalone LF, but NickServ, QUIT, and USER construction also use this helper before `sendRaw` strips newlines. Keep the generic sanitizer flattening CR/LF, or add a message-specific sanitizer for multiline payloads.
  - confidence: 0.82
