## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Split batched IRC lines before writing:** `extensions/irc/src/client.ts:249`
  - body: The multiline path writes each logical line as one tagged PRIVMSG. Long URLs or code lines can still exceed IRC's per-frame budget and be truncated or rejected, so split overlong logical lines inside the batch and use draft/multiline-concat when preserving one logical line.
  - confidence: 0.93
- **[P2] Honor advertised multiline limits:** `extensions/irc/src/client.ts:325-333`
  - body: The CAP handler only checks for draft/multiline as a boolean, but servers advertise max-bytes and may advertise max-lines. With the 16384-character outbound chunk cap, this can send batches that strict servers reject with multiline limit failures.
  - confidence: 0.9
- **[P2] Preserve concat semantics when rebuilding batches:** `extensions/irc/src/client.ts:462`
  - body: Joining every buffered batch message with `\n` corrupts valid draft/multiline-concat splits, where the tagged line must be appended to the previous line without a separator.
  - confidence: 0.88
- **[P2] Keep LF preservation out of generic IRC commands:** `extensions/irc/src/protocol.ts:147-148`
  - body: `sanitizeIrcOutboundText` also feeds NickServ, USER, and QUIT commands that later pass through `sendRaw`. Preserving standalone LF here lets `sendRaw` strip it without a separator, so keep newline preservation local to PRIVMSG multiline handling or flatten non-message callers before writing commands.
  - confidence: 0.8
