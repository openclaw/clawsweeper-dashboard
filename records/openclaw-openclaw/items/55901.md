## Summary

Keep open: current main still lacks IRC multiline support, but this PR is not ready to merge because the head implementation still misses required IRCv3 multiline semantics and has a shared sanitizer regression.

## What This Changes

The PR adds IRCv3 draft/multiline CAP negotiation, BATCH send/receive handling, a newline-preserving IRC chunker, tag parsing, and LF-preserving IRC outbound sanitization.

## Best Possible Solution

Land a rebased IRC plugin change that negotiates the required capabilities, emits and receives byte-safe draft/multiline frames, scopes LF preservation to message payloads, and adds focused IRC tests.

## Reproduction Assessment

Yes. The current-main feature gap is source-reproducible because IRC output flattens newlines and sends plain PRIVMSG frames, and the remaining PR blockers are visible in the head source against the IRCv3 multiline spec.

## Solution Assessment

No, not as-is. draft/multiline is the right IRC-plugin direction, but this patch needs batch negotiation, byte-safe frame splitting, concat-aware inbound reconstruction, and message-scoped LF handling before it is the maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Negotiate batch before enabling multiline:** `extensions/irc/src/client.ts:333`
  - body: The multiline spec depends on the `batch` capability, but this branch requests only `draft/multiline` and then sends `BATCH` frames. Servers that enforce capability dependencies can reject or ignore that flow, so request and require ACKs for both capabilities before enabling multiline sends.
  - confidence: 0.91
- **[P2] Split long multiline frames safely:** `extensions/irc/src/client.ts:249`
  - body: When multiline is active, each newline-delimited payload is written as one tagged `PRIVMSG`. A long markdown code line or URL can exceed IRC's per-line limit and be truncated or rejected; split long logical lines into safe frames, using `draft/multiline-concat` where needed to preserve the original line.
  - confidence: 0.92
- **[P2] Honor multiline concat when rebuilding batches:** `extensions/irc/src/client.ts:462`
  - body: Incoming batches are always rebuilt with `messages.map(...).join("\n")`, but `draft/multiline-concat` means the line should join directly with the previous line. Track that tag per buffered message so long logical lines are not delivered with spurious line breaks.
  - confidence: 0.88
- **[P3] Scope LF preservation to message bodies:** `extensions/irc/src/protocol.ts:147-148`
  - body: `sanitizeIrcOutboundText` now preserves standalone LF, but NickServ, QUIT, and USER command arguments still pass through this helper before `sendRaw` strips newlines. Inputs with LF now concatenate tokens instead of the old space-flattened value, so keep LF preservation message-specific or flatten LF at non-message call sites.
  - confidence: 0.8
