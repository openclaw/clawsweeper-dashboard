## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Split batched IRC lines before writing:** `extensions/irc/src/client.ts:249`
  - body: The multiline path writes each newline-separated line as one tagged PRIVMSG. A long URL or code line can exceed IRC's per-line budget and be truncated or rejected; split overlong logical lines inside the batch and preserve reconstruction with draft/multiline-concat. ([ircv3.net](https://ircv3.net/specs/extensions/multiline))
  - confidence: 0.93
- **[P2] Honor advertised multiline batch limits:** `extensions/irc/src/client.ts:325-333`
  - body: The LS handler only checks whether the capability string contains draft/multiline and ignores the required max-bytes value and optional max-lines value. With textChunkLimit raised to 16384, servers advertising lower limits can reject complete batches. ([ircv3.net](https://ircv3.net/specs/extensions/multiline))
  - confidence: 0.9
- **[P2] Preserve concat semantics when rebuilding batches:** `extensions/irc/src/client.ts:462`
  - body: The receiver joins every buffered batch message with LF. Valid multiline senders use draft/multiline-concat to split a long logical line without adding a line break, so current inbound handling corrupts those messages by inserting extra newlines. ([ircv3.net](https://ircv3.net/specs/extensions/multiline))
  - confidence: 0.87
- **[P2] Keep newline preservation out of generic commands:** `extensions/irc/src/protocol.ts:147-148`
  - body: sanitizeIrcOutboundText also feeds USER, QUIT, and NickServ commands that later go through sendRaw. Preserving standalone LF here lets sendRaw strip it without a separator, turning values such as secret\nJOIN into joined tokens; keep LF preservation in the PRIVMSG multiline path or flatten non-message callers before sendRaw.
  - confidence: 0.8
