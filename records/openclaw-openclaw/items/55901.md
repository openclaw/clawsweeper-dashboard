## Summary

The PR is still useful but not merge-ready: the inspected screenshot proves happy-path multiline rendering, while the current branch still leaves IRCv3 multiline protocol and delivery-limit gaps that can drop, truncate, or mis-reconstruct IRC messages.

## What This Changes

The branch adds an IRC markdown-preserving chunker plus client/protocol/control-character changes to negotiate `draft/multiline`, send newline-containing replies as IRCv3 BATCH frames, and reconstruct incoming multiline batches.

## Best Possible Solution

Keep the feature direction, but land it only after the IRC client enforces the IRCv3 multiline contract, byte-safe frame limits, advertised server limits, concat reconstruction, and focused regression tests on the current outbound-base boundary.

## Reproduction Assessment

Yes for the PR blockers: the branch source plus the IRCv3 spec show unguarded per-line batch writes, incomplete capability/limit handling, and LF-only reconstruction. This is not a current-main bug reproduction because current main does not implement `draft/multiline`.

## Solution Assessment

No as currently proposed: the user-visible direction is good, but the implementation needs to preserve current IRC frame safety and follow the multiline/batch capability contract before it is the narrow maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Split batch lines before writing PRIVMSG frames:** `extensions/irc/src/client.ts:249`
  - body: The multiline path now sends each newline-delimited markdown line as one tagged `PRIVMSG`. A long code line or URL can exceed IRC's per-frame limit and be truncated or rejected, so split long batch payloads into IRC-safe frames before writing them.
  - confidence: 0.9
- **[P2] Require batch before enabling multiline batches:** `extensions/irc/src/client.ts:333`
  - body: `draft/multiline` depends on IRCv3 BATCH support, but the branch enables multiline sends from the multiline ACK alone. Request and confirm the `batch` capability before sending BATCH frames.
  - confidence: 0.84
- **[P2] Honor advertised multiline limits:** `extensions/irc/src/client.ts:325-333`
  - body: The CAP parser only checks whether the capability string contains `draft/multiline` and then raises the channel limit to 16384. Parse and enforce the advertised `max-bytes` and `max-lines` values so servers do not reject oversized batches.
  - confidence: 0.86
- **[P2] Honor concat tags when rebuilding batches:** `extensions/irc/src/client.ts:462`
  - body: Incoming multiline batches are always reconstructed with `join("\n")`. Lines tagged with `draft/multiline-concat` must be joined directly to the previous line, otherwise long logical lines are delivered to OpenClaw with extra line breaks.
  - confidence: 0.87
- **[P2] Keep generic IRC sanitization single-line:** `extensions/irc/src/protocol.ts:145`
  - body: `sanitizeIrcOutboundText` is also used for NickServ, QUIT, and other non-message commands, but the PR now preserves standalone LF and relies on `sendRaw` to strip it. Use a message-specific multiline sanitizer so command payloads continue normalizing CR and LF to separators instead of collapsing tokens.
  - confidence: 0.78
