## Summary

Keep open: current main and the latest release still flatten IRC newlines, and this PR is useful but not mergeable as-is because the PR head does not fully satisfy the IRCv3 draft/multiline contract for prerequisite capability negotiation, advertised limits, or concat-aware long-line handling.

## What This Changes

The PR changes the IRC plugin to preserve markdown line breaks by adding an IRC chunker, IRCv3 tag parsing, LF-preserving sanitization, and draft/multiline BATCH send/receive handling.

## Best Possible Solution

Land a rebased IRC plugin implementation that negotiates the full IRCv3 prerequisite set, enforces server-advertised limits, preserves long logical lines with concat, scopes LF preservation to message payloads, and covers the behavior with focused tests.

## Reproduction Assessment

Yes, source-reproducible: current main and v2026.5.12 flatten IRC newlines, while the PR head's send/receive paths conflict with IRCv3 draft/multiline requirements for batch negotiation, advertised limits, and concat handling.

## Solution Assessment

No, not as-is. The feature belongs in the IRC plugin, but this branch needs protocol-complete negotiation, limit handling, concat-aware framing/reconstruction, and message-scoped sanitization before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Negotiate batch before enabling multiline:** `extensions/irc/src/client.ts:333`
  - body: The IRCv3 draft/multiline spec depends on the batch capability, but this path only requests draft/multiline and then enables BATCH sends. Request and require ACKs for both capabilities before multilineCap can become true.
  - confidence: 0.91
- **[P2] Honor advertised multiline limits:** `extensions/irc/src/client.ts:325-333`
  - body: Servers advertise draft/multiline with a required max-bytes value and recommended max-lines, but this handler only does a substring check while the channel uses a fixed 16384 limit. Parse the advertised values and cap batches to them.
  - confidence: 0.89
- **[P2] Split long multiline lines with concat:** `extensions/irc/src/client.ts:249`
  - body: Each logical line is written as one tagged PRIVMSG. A long URL or code line can exceed IRC frame limits; split byte-safely and mark continuations with draft/multiline-concat so the original line is reconstructed.
  - confidence: 0.93
- **[P2] Preserve concat continuations on receive:** `extensions/irc/src/client.ts:462`
  - body: The receive path joins every buffered PRIVMSG with LF, so a server-sent draft/multiline-concat continuation gains an extra newline. Track that tag per buffered part and concatenate continuation text directly.
  - confidence: 0.86
- **[P3] Keep LF preservation out of generic IRC commands:** `extensions/irc/src/protocol.ts:147-148`
  - body: sanitizeIrcOutboundText is also used for NickServ, QUIT, and USER values; preserving bare LF there lets sendRaw remove the newline and concatenate tokens. Keep the generic sanitizer flattening CR/LF and add a message-only sanitizer for multiline payloads.
  - confidence: 0.82
