## Summary

This PR should stay open because the feature is not implemented on current main and the branch still has protocol correctness blockers that can corrupt or reject multiline IRC messages. The IRC plugin is the right place for the feature, but the current patch needs more work before merge.

## What This Changes

The PR adds IRCv3 draft/multiline support to the IRC plugin by preserving newline-aware chunking, parsing IRCv3 tags, negotiating multiline CAP support, and sending or receiving multiline messages through BATCH frames.

## Best Possible Solution

Land a rebased IRC plugin implementation that negotiates batch plus draft/multiline, obeys advertised multiline limits, emits and receives concat-aware byte-safe frames, keeps LF preservation scoped to message payloads, and adds focused regression coverage.

## Reproduction Assessment

Yes for source-level behavior: current main lacks multiline IRC sending, and the PR blockers are visible from the PR source against the IRCv3 multiline contract. I did not run a live IRC server reproduction in this read-only review.

## Solution Assessment

No, not as-is. The IRC plugin is the right home for multiline support, but the patch must handle batch negotiation, advertised limits, concat-safe frame splitting, receive-side concat reconstruction, and sanitizer scoping before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Negotiate batch with multiline:** `extensions/irc/src/client.ts:333`
  - body: The multiline spec depends on the IRCv3 `batch` capability, but this branch requests only `draft/multiline` and then emits BATCH frames. Request and require ACKs for both `batch` and `draft/multiline` before enabling multiline sends.
  - confidence: 0.91
- **[P2] Honor advertised multiline limits:** `extensions/irc/src/client.ts:325-333`
  - body: `draft/multiline` is advertised with a required `max-bytes` value and optional `max-lines`, but this code only checks for the capability name and uses a fixed 16384 chunk limit. Parse the advertised values and keep batches within them so servers do not reject valid-looking sends.
  - confidence: 0.88
- **[P2] Split long multiline lines with concat:** `extensions/irc/src/client.ts:249`
  - body: When multiline is active, each logical line is written as one tagged PRIVMSG. Long code lines or URLs can exceed IRC's per-frame limit, so split long logical lines into byte-safe frames and mark continuations with `draft/multiline-concat` where the original line must be preserved.
  - confidence: 0.93
- **[P2] Preserve concat continuations on receive:** `extensions/irc/src/client.ts:462`
  - body: Incoming multiline batches are reconstructed with `messages.map(...).join("\n")`, so a server-sent `draft/multiline-concat` continuation would incorrectly gain a newline. Track that tag for buffered batch lines and concatenate those fragments without inserting LF.
  - confidence: 0.84
- **[P3] Keep LF preservation message-scoped:** `extensions/irc/src/protocol.ts:147-148`
  - body: `sanitizeIrcOutboundText` now preserves standalone LF, but NickServ, QUIT, and USER construction also use this helper before `sendRaw` strips newlines. Keep the generic sanitizer flattening CR/LF, or add a message-specific sanitizer for multiline payloads, so non-message command values do not concatenate tokens.
  - confidence: 0.82
