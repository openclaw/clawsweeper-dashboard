## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Split long multiline batch lines:** `extensions/irc/src/client.ts:249`
  - body: This loop sends each newline-delimited chunk as one tagged `PRIVMSG`. IRC multiline still rides on ordinary IRC message frames, and the IRCv3 draft calls out the 512-byte line limit plus `draft/multiline-concat` for splitting long logical lines. A long URL or code line inside a 16k chunk can be truncated or rejected instead of delivered.
  - confidence: 0.93
- **[P2] Honor advertised multiline limits:** `extensions/irc/src/client.ts:332-333`
  - body: The CAP handler only checks whether the accumulated capability text contains `draft/multiline`, but the IRCv3 draft advertises required `max-bytes` and recommended `max-lines` values. With the channel default raised to 16384, OpenClaw can emit a batch that a server advertised as smaller and receive a FAIL or dropped delivery.
  - confidence: 0.86
- **[P2] Bound incoming multiline batches:** `extensions/irc/src/client.ts:449-450`
  - body: Incoming `draft/multiline` batches are buffered until `BATCH -id`, but there is no byte limit, line limit, or timeout cleanup. A server can start a multiline batch and stream many tagged messages without ending it, growing memory on the IRC client path that previously processed lines one at a time.
  - confidence: 0.82
