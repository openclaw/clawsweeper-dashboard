## Summary

Keep this PR open. Current main still lacks IRCv3 draft/multiline support, so the contribution is not obsolete, and the discussion shows the author already addressed several earlier review concerns; however, the latest head is still not merge-ready because it misses required batch capability negotiation, leaves long multiline frames unsplit, changes a shared sanitizer in a way that can corrupt non-message command arguments, and needs to be rebased onto the current IRC outbound adapter structure.

## What This Changes

The PR adds IRC draft/multiline CAP negotiation, IRCv3 tag and BATCH parsing, newline-preserving outbound chunking/sanitization, incoming multiline reconstruction, and a larger IRC outbound text chunk target.

## Best Possible Solution

Land a rebased IRC plugin implementation that negotiates both required IRCv3 capabilities, preserves markdown line breaks through draft/multiline, keeps every emitted IRC frame within safe byte limits, retains legacy fallback chunking, keeps command-argument sanitization safe, and adds focused unit coverage plus a changelog entry for the user-facing feature.

## Reproduction Assessment

Yes. Current main's feature gap is reproducible by static inspection: outbound IRC text is newline-flattened and only plain PRIVMSG frames are emitted. The main PR regression is also reproducible by sending a multiline response with one long code or URL line to a server advertising draft/multiline; the PR head writes that line as one tagged PRIVMSG over the standard IRC frame limit.

## Solution Assessment

No, not as-is. draft/multiline is the right direction for preserving IRC markdown replies, but the implementation needs batch CAP negotiation, per-frame splitting with draft/multiline-concat for long logical lines, current-main outbound adapter integration, sanitizer cleanup, tests, and changelog coverage before it is the narrowest maintainable fix.

- **[P2] Negotiate batch before using draft/multiline:** `extensions/irc/src/client.ts:333`
  - body: The multiline spec depends on the `batch` capability, but this branch only requests `draft/multiline` and then starts sending `BATCH` frames. Servers that enforce CAP dependencies can reject or ignore the batch flow even though `multilineCap` is set, so request and ACK both capabilities before enabling this path.
  - confidence: 0.88
- **[P2] Split long multiline payload lines:** `extensions/irc/src/client.ts:249`
  - body: When multiline is active, each newline-delimited payload is written as a single tagged `PRIVMSG`. A long markdown code line or URL can exceed IRC's per-frame limit and be truncated or rejected; split long logical lines into safe frames, using `draft/multiline-concat` where needed to preserve the original line.
  - confidence: 0.92
- **[P3] Flatten LF for non-message command arguments:** `extensions/irc/src/protocol.ts:147-148`
  - body: `sanitizeIrcOutboundText` now preserves standalone `\n`, but the same helper still feeds NickServ, QUIT, and USER command arguments before `sendRaw` removes newlines entirely. Inputs with LF or escaped `\n` now concatenate tokens instead of the old space-flattened value, so keep newline preservation message-specific or flatten LF at non-message call sites.
  - confidence: 0.76
