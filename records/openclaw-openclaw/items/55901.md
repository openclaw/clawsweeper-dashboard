## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Negotiate batch before enabling multiline sends:** `extensions/irc/src/client.ts:333`
  - body: The IRCv3 multiline spec depends on the `batch` capability, but this branch only requests `draft/multiline` and then sends `BATCH` frames after that ACK. Request and require both capabilities before setting the multiline send path active, otherwise compliant servers can reject or ignore the new flow.
  - confidence: 0.93
- **[P1] Split long multiline lines before writing PRIVMSG:** `extensions/irc/src/client.ts:249`
  - body: With the outbound chunk default raised to 16384, a single markdown/code line can exceed an IRC frame. This path writes that whole line as one tagged `PRIVMSG`, so long URLs or code lines can be truncated or rejected; split long logical lines into IRC-safe frames using `draft/multiline-concat`.
  - confidence: 0.95
- **[P2] Honor advertised multiline size limits:** `extensions/irc/src/client.ts:325-333`
  - body: `CAP LS 302` can advertise values like `draft/multiline=max-bytes=...`, but this code only substring-checks the capability and the channel can pass up to 16384 bytes into one batch. Parse `max-bytes` and `max-lines`, then split or fall back before servers reject over-limit batches.
  - confidence: 0.88
- **[P2] Keep LF preservation out of command sanitization:** `extensions/irc/src/protocol.ts:147-148`
  - body: `sanitizeIrcOutboundText` is also used for NickServ, USER, QUIT, and other raw command payloads, but preserving standalone LF leaves `sendRaw` to remove it without a separator. That changes current behavior from separating tokens with a space to collapsing them, so multiline message bodies need a separate sanitizer path.
  - confidence: 0.88
- **[P2] Honor multiline concat when reconstructing batches:** `extensions/irc/src/client.ts:462`
  - body: Incoming multiline batches are always reconstructed with `join("\n")`, but the spec says lines tagged `draft/multiline-concat` are directly joined with the previous line. Preserve that tag in the batch buffer so long split lines are reconstructed without inserted line breaks.
  - confidence: 0.84
