## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Split batched IRC lines before writing:** `extensions/irc/src/client.ts:249`
  - body: The multiline path writes each newline-separated logical line as one tagged PRIVMSG. Long URLs or code lines can still exceed IRC's per-line frame budget and be truncated or rejected; split overlong logical lines inside the batch and preserve reconstruction with draft/multiline-concat.
  - confidence: 0.93
- **[P2] Honor advertised multiline limits:** `extensions/irc/src/client.ts:325-333`
  - body: The CAP handler only checks whether the accumulated capability text contains draft/multiline, but the server advertises required max-bytes and recommended max-lines limits. With the PR's 16384-character outbound chunk cap, servers advertising lower limits can reject complete batches.
  - confidence: 0.9
- **[P2] Preserve concat semantics when rebuilding batches:** `extensions/irc/src/client.ts:462`
  - body: The receiver joins every buffered batch message with \n. Valid multiline senders use draft/multiline-concat to split a long logical line without adding a line break, so current inbound handling corrupts those messages by inserting extra newlines.
  - confidence: 0.88
- **[P2] Keep newline preservation out of generic commands:** `extensions/irc/src/protocol.ts:147-148`
  - body: sanitizeIrcOutboundText also feeds NickServ, USER, and QUIT commands that later go through sendRaw. Preserving standalone LF here lets sendRaw strip it without a separator, corrupting values such as secret\nJOIN; keep LF preservation in the PRIVMSG multiline path or flatten non-message callers before sendRaw.
  - confidence: 0.8
