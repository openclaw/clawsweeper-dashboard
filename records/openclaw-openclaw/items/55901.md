## Summary

Keep this PR open: it is useful source work for the bundled IRC plugin and current main still lacks draft/multiline support, but the branch is not mergeable as-is because the implementation misses required IRCv3 multiline protocol behavior and has focused chunking/sanitizer defects.

## What This Changes

The branch adds an IRC-specific chunker, IRCv3 tag parsing, LF-preserving sanitizer changes, and draft/multiline BATCH send/receive handling so IRC messages can preserve markdown line breaks.

## Best Possible Solution

Land a rebased IRC plugin implementation that negotiates `batch` plus `draft/multiline`, obeys advertised limits, handles concat-aware byte-safe frames in both directions, keeps LF preservation message-scoped, and adds focused regression tests.

## Reproduction Assessment

Yes, source-reproducible for the PR blockers. Current main and v2026.5.12 show ordinary PRIVMSG sending with CR/LF flattening, while the PR head source conflicts with the IRCv3 draft/multiline contract.

## Solution Assessment

No, not as-is. The feature belongs in the IRC plugin, but the patch needs batch negotiation, advertised-limit handling, concat-aware byte-safe send/receive behavior, and message-scoped newline sanitization before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Negotiate batch before enabling multiline:** `extensions/irc/src/client.ts:333`
  - body: The IRCv3 multiline spec depends on `batch` and says it must be negotiated, but this handler requests only `draft/multiline` before sending `BATCH` frames. Request and require ACKs for both capabilities before enabling multiline sends. ([ircv3.net](https://ircv3.net/specs/extensions/multiline?utm_source=openai))
  - confidence: 0.91
- **[P2] Honor advertised multiline limits:** `extensions/irc/src/client.ts:325-333`
  - body: `draft/multiline` is advertised with required `max-bytes` and recommended `max-lines` values, but this code only checks for the capability name and uses a fixed 16384 chunk limit. Parse the advertised values and keep batches within server limits so valid-looking sends are not rejected. ([ircv3.net](https://ircv3.net/specs/extensions/multiline?utm_source=openai))
  - confidence: 0.89
- **[P2] Split long multiline lines with concat:** `extensions/irc/src/client.ts:249`
  - body: When multiline is active, each logical line is written as one tagged `PRIVMSG`. Long code lines or URLs can exceed IRC's per-frame limit, so split long logical lines into byte-safe frames and mark continuations with `draft/multiline-concat` where the original line must be preserved. ([ircv3.net](https://ircv3.net/specs/extensions/multiline?utm_source=openai))
  - confidence: 0.93
- **[P2] Preserve concat continuations on receive:** `extensions/irc/src/client.ts:462`
  - body: Incoming multiline batches are reconstructed with `messages.map(...).join("\n")`, so a server-sent `draft/multiline-concat` continuation would incorrectly gain a newline. Track that tag for buffered batch lines and concatenate those fragments without inserting LF. ([ircv3.net](https://ircv3.net/specs/extensions/multiline?utm_source=openai))
  - confidence: 0.86
- **[P3] Keep newline preservation out of generic commands:** `extensions/irc/src/protocol.ts:147-148`
  - body: `sanitizeIrcOutboundText` now preserves standalone LF, but NickServ, QUIT, and USER construction also use this helper before `sendRaw` strips newlines. Keep the generic sanitizer flattening CR/LF, or add a message-specific sanitizer for multiline payloads, so non-message command values do not concatenate tokens.
  - confidence: 0.82
