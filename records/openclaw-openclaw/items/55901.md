## Summary

Keep open: current main and the latest release still flatten IRC newlines, so the feature is not implemented, but this PR should not merge as-is because the draft/multiline protocol handling is incomplete and the branch is currently conflicting.

## What This Changes

The PR adds IRCv3 draft/multiline CAP negotiation, newline-preserving IRC chunking, BATCH send/receive handling, tag parsing, and newline sanitizer changes for IRC messages.

## Best Possible Solution

Land a rebased IRC plugin implementation that negotiates batch plus draft/multiline, obeys advertised limits, preserves byte-safe concat semantics on send/receive, keeps newline preservation message-scoped, and adds focused regression tests.

## Reproduction Assessment

Yes, source-reproducible. Current main and v2026.5.12 show ordinary PRIVMSG sending with CR/LF flattening, while the PR blockers are visible in the PR head source against the IRCv3 draft/multiline contract.

## Solution Assessment

No, not as-is. The IRC plugin is the right home for the feature, but the patch needs batch negotiation, advertised-limit handling, concat-aware byte-safe send/receive behavior, and message-scoped newline sanitization before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Negotiate batch before enabling multiline:** `extensions/irc/src/client.ts:333`
  - body: The draft/multiline spec depends on the `batch` capability, but this handler requests only `draft/multiline` and then emits `BATCH` frames. Request and require ACKs for both `batch` and `draft/multiline` before enabling multiline sends.
  - confidence: 0.91
- **[P2] Honor advertised multiline limits:** `extensions/irc/src/client.ts:323-333`
  - body: `draft/multiline` is advertised with required `max-bytes` and recommended `max-lines` values, but this code only checks for the capability name and uses a fixed 16384 chunk limit. Parse the advertised values and keep batches within server limits so valid-looking sends are not rejected.
  - confidence: 0.88
- **[P2] Split long multiline lines with concat:** `extensions/irc/src/client.ts:249`
  - body: When multiline is active, each logical line is written as one tagged `PRIVMSG`. Long code lines or URLs can exceed IRC's per-frame limit, so split long logical lines into byte-safe frames and mark continuations with `draft/multiline-concat` where the original line must be preserved.
  - confidence: 0.93
- **[P2] Preserve concat continuations on receive:** `extensions/irc/src/client.ts:462`
  - body: Incoming multiline batches are reconstructed with `messages.map(...).join("\n")`, so a server-sent `draft/multiline-concat` continuation would incorrectly gain a newline. Track that tag for buffered batch lines and concatenate those fragments without inserting LF.
  - confidence: 0.84
- **[P3] Keep newline preservation out of generic commands:** `extensions/irc/src/protocol.ts:147-148`
  - body: `sanitizeIrcOutboundText` now preserves standalone LF, but NickServ, QUIT, and USER construction also use this helper before `sendRaw` strips newlines. Keep the generic sanitizer flattening CR/LF, or add a message-specific sanitizer for multiline payloads, so non-message command values do not concatenate tokens.
  - confidence: 0.82
