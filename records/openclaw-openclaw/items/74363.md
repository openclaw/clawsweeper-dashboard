## Label Justifications

- `P1`: False failed/lost subagent runs can break active Mission Control or Workshop workflows and misreport child work that continued running.
- `impact:session-state`: The issue is about subagent run/session lifecycle state drifting from the backing session's actual running state.
- `impact:message-loss`: False terminalization can suppress the useful child completion signal back to the parent workflow.

## Summary

This remains a valid core subagent lifecycle bug on current main. Source inspection shows the pending wait branch still drops supervision, clean close 1000 is still absent from the recoverable wait classifier, and a stale active run with a still-running backing session can still fall through to the lost-context error unless `abortedLastRun` is set.

## What This Changes

The issue asks OpenClaw to recover subagent lifecycle supervision after clean gateway close, pending `agent.wait`, or missing in-memory run context while the backing session is still running.

## Best Possible Solution

Land one canonical subagent lifecycle hardening path that re-arms pending waits, treats clean reconnect closes as recoverable where appropriate, and reattaches or re-waits still-running backing sessions without weakening fail-closed handling for missing, terminal, stale, or drifted sessions.

## Reproduction Assessment

Yes from source inspection: current main still returns on pending waits and still lost-contexts running backing sessions without `abortedLastRun`. I did not run a live gateway restart reproduction in this read-only review.

## Solution Assessment

Yes, the requested repair direction is bounded and matches the current source gaps. The safest fix should keep retries bounded and preserve explicit fail-closed behavior for terminal, missing, stale, or drifted sessions.
