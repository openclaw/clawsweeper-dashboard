## Summary

Keep open. Current main still matches the three reported lifecycle gaps: clean gateway close 1000 is not classified as recoverable, pending waits return without re-arming supervision, and a still-running backing session can fall through to lost-context finalization.

## What This Changes

The issue asks OpenClaw to treat clean gateway close, pending wait results, and still-running backing sessions as recoverable subagent lifecycle states instead of false terminal failures.

## Best Possible Solution

Implement a focused agents-runtime fix in the existing wait classifier, subagent wait manager, and stale sweep reattach path while preserving fail-closed behavior for truly missing, terminal, stale, drifted, or orphaned backing sessions.

## Reproduction Assessment

## Solution Assessment

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/74363 with a narrow agents-runtime patch. On current main, `src/gateway/call.ts` formats clean closes as `gateway closed (1000 normal closure): ...`, but `src/agents/run-wait.ts` only classifies 1006/transport-style failures as recoverable; add the clean-close case without making gateway timeouts recoverable. In `src/agents/subagent-registry-run-manager.ts`, keep supervising when `agent.wait` returns `pending` by scheduling a guarded follow-up wait similar to the recoverable-error path. In `src/agents/subagent-registry.ts`, when sweep loses live run context but the child session store entry is still `status: "running"`, reattach/re-wait instead of finalizing as `subagent run lost active execution context`. Preserve fail-closed behavior for missing session entries, missing session ids, terminal statuses, stale unended runs, aborted recovery markers, drifted entries, and true orphan cases. Coordinate with related #74131 and #75462; do not broaden provider behavior or redesign gateway lifecycle. Add focused tests and a `CHANGELOG.md` fix entry.
