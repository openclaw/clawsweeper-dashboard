## Summary

Keep this open: current main and the latest release still contain all three reported control-flow gaps, and the open related hardening PR only partially overlaps rather than closing the clean-close, pending-wait, and running-session reattach behavior.

## What This Changes

The issue requests subagent lifecycle recovery so clean gateway closes, pending waits, and missing in-memory run contexts do not falsely mark still-running child work as failed or lost.

## Best Possible Solution

Extend the agents lifecycle fix so clean gateway close 1000 is recoverable, pending waits re-arm supervision, and stale in-memory runs reattach or re-wait when the backing session store still says running, with focused regression coverage and a temporary loopback gateway proof before landing.

## Reproduction Assessment

Yes, at source level: current main still lacks gateway close 1000 in the recoverable classifier, returns on pending waits without rearming, and lets a running backing session fall through to lost-context completion. I did not run a live gateway restart repro in this read-only review.

## Solution Assessment

No, current main is not the best end state because it only handles some timeout, 1006, and aborted-session edges. The maintainable fix should keep permanent failures fail-closed while rearming or reattaching only for explicitly recoverable lifecycle states.
