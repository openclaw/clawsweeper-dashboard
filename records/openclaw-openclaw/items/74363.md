## Summary

Keep open. Current main still has the reported source-level subagent lifecycle gaps: clean gateway close 1000 is not classified as recoverable, pending waits return without rearming supervision, and a non-terminal running backing session can still fall through to lost-context finalization.

## What This Changes

The issue asks for subagent lifecycle recovery to treat clean gateway close 1000 and pending wait results as recoverable, and to reattach when the backing session is still running.

## Best Possible Solution

Implement a focused agents-runtime fix in the existing recoverable wait classifier, subagent wait supervision, and stale sweep reattach path while preserving fail-closed behavior for missing, terminal, stale, drifted, or truly orphaned sessions.

## Reproduction Assessment

Yes. There is a high-confidence source-level path on current main: close 1000 is not recoverable, pending waits return without scheduling another wait, and a present running backing session can still reach lost-context finalization.

## Solution Assessment

Yes. The narrowest maintainable repair is to extend the existing wait classifier, reuse the existing delayed wait retry pattern for pending waits, and add a running-session reattach/re-wait guard before lost-context finalization.

## Repair Work Prompt

Fix issue #74363 in the agents runtime. On current main, `src/agents/run-wait.ts` treats gateway close 1006 as recoverable but not clean close 1000; `src/agents/subagent-registry-run-manager.ts` returns on `wait.status === "pending"` without rearming `waitForSubagentCompletion`; and `src/agents/subagent-registry.ts` can finalize an unended run as `subagent run lost active execution context` even when the backing child session store entry is still `status: "running"`. Add a focused fix that: treats `gateway closed (1000...)` as recoverable for subagent wait supervision without turning gateway timeout into recoverable; schedules a guarded follow-up wait when `agent.wait` returns pending; and, during stale sweep, reattaches/re-waits when the backing session is still running while preserving fail-closed behavior for missing, terminal, stale, drifted, or aborted cases. Add regression coverage for close 1000 recovery, pending wait rearming, running-session reattachment, and the existing terminal/missing-session behavior. Coordinate with adjacent open PRs #74131 and #75462 but do not broaden into spawn readiness, announce delivery, cleanup retry, or unrelated lifecycle refactors. Add a user-facing CHANGELOG.md fix entry because this changes runtime behavior.
