## Summary

Keep open. Current main still lacks the three requested safeguards: clean gateway close 1000 is not recoverable for waits, pending `agent.wait` results drop subagent supervision, and the stale sweep path can still finalize an unended run as lost instead of reattaching when the backing session remains running.

## What This Changes

The issue requests subagent lifecycle hardening for clean gateway closes, pending `agent.wait` results, and missing in-memory run contexts while the backing session is still running.

## Best Possible Solution

Implement the missing agents-runtime guards in the existing wait classifier, subagent wait manager, and stale sweep path while preserving fail-closed behavior for truly missing, terminal, stale, or drifted backing sessions.

## Reproduction Assessment

Yes. There is a high-confidence source-level reproduction path on current main: close 1000 is not classified recoverable, pending waits return without a follow-up waiter, and a still-running backing session can still reach the lost-context finalization path.

## Solution Assessment

Yes. The reporter's suggested boundary matches the existing recoverable-wait, wait-supervision, and stale-sweep seams and is narrower than changing gateway lifecycle semantics globally.

Keep the change scoped to agents runtime behavior: add clean gateway close 1000 to the recoverable agent-wait classifier in `src/agents/run-wait.ts`; make `waitForSubagentCompletion()` in `src/agents/subagent-registry-run-manager.ts` keep supervising when `agent.wait` returns `pending` by scheduling a bounded follow-up wait; and update the stale sweep/lost-context path in `src/agents/subagent-registry.ts` so an unended run with a backing session entry still marked `running` is reattached/re-waited instead of finalized as lost.
