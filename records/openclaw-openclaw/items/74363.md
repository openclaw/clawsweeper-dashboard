## Summary

Keep open. Current main still matches the reported subagent lifecycle gaps: clean gateway close code 1000 is not classified as recoverable, pending `agent.wait` results still return without rearming supervision, and stale active registry entries can still fall through to a lost-context completion even when the child session entry is present and `running`.

## What This Changes

The issue requests subagent lifecycle recovery hardening so clean gateway closes, pending waits, and missing in-memory run contexts do not falsely fail or lose active child work.

## Best Possible Solution

Implement a focused agents-runtime fix that treats clean gateway-close wait interruptions as recoverable for subagent supervision, schedules another wait after pending results, and re-waits or reattaches when stale sweep sees a non-terminal running backing session while preserving missing, terminal, drifted, aborted, and stale failure semantics.

## Reproduction Assessment

Yes. Current main has a high-confidence source-level reproduction path: clean close 1000 reaches the classifier but is not recoverable, pending waits exit supervision, and a still-running session can fall through to the lost-context completion path.

## Solution Assessment

Yes. The narrowest maintainable solution is in `src/agents/run-wait.ts`, `src/agents/subagent-registry-run-manager.ts`, and `src/agents/subagent-registry.ts` with focused regression tests, not a broad gateway lifecycle rewrite.
