## Summary

Keep open. Current main still matches the reported subagent lifecycle gaps: clean gateway close code 1000 is not classified as recoverable, pending waits return without rearming supervision, and a stale in-memory run can still be completed as lost even when the persisted child session is still running.

## What This Changes

The issue requests subagent lifecycle recovery for clean gateway closes, pending wait results, and stale active runs whose backing session is still running.

## Best Possible Solution

Implement a focused agents-runtime fix that treats clean gateway-close wait interruptions as recoverable for subagent supervision, schedules another wait after pending results, and re-waits or reattaches when stale sweep finds a non-terminal running backing session while preserving missing, terminal, drifted, and stale failure semantics.

## Reproduction Assessment

Yes. There is a high-confidence source-level reproduction path on current main: a clean gateway close 1000 reaches the classifier but is not recoverable, pending wait exits supervision, and a still-running session can fall through to the lost-context completion path.

## Solution Assessment

Yes. The narrowest maintainable solution is in `src/agents/run-wait.ts`, `src/agents/subagent-registry-run-manager.ts`, and `src/agents/subagent-registry.ts` with regression coverage for the three reported branches, rather than a broad gateway lifecycle rewrite.
