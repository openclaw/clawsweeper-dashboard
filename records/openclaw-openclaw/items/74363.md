## Summary

Keep this open: current main still has the reported subagent lifecycle gaps, and the open related hardening PR only overlaps part of the lifecycle area rather than implementing the clean-close, pending-wait, and running-session reattach behavior described here.

## What This Changes

Reports that subagent lifecycle supervision can falsely mark child runs failed or lost after a clean gateway close, a pending wait result, or missing in-memory context while the backing session is still running.

## Best Possible Solution

Extend the subagent lifecycle fix so clean gateway close 1000 is recoverable, pending waits re-arm supervision, and stale in-memory runs reattach or re-wait when the backing session store still says running, with focused regression coverage for each branch.

## Reproduction Assessment

Yes at source level: current main still returns on pending waits, lacks clean close 1000 in the recoverable wait classifier, and can route a stale active run to lost-context even when the backing session entry is running. I did not run a live gateway restart repro in this read-only review.

## Solution Assessment

No, current main is not the best end state because it only handles some timeout/1006/restart-aborted edges. The maintainable fix should keep permanent failures fail-closed while re-arming or reattaching only for the explicitly recoverable lifecycle states described here.
