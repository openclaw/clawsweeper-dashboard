## Summary

Keep open. Current main and the latest release still match the reported subagent lifecycle gaps: clean gateway close 1000 is not classified as recoverable, pending waits return without rearming supervision, and the stale active-run sweep can still mark a run lost even when the persisted child session is still running. Adjacent open PRs #74131 and #75462 are relevant but do not make this implemented on main.

## What This Changes

The issue requests that subagent supervision recover after clean gateway close, pending wait results, and stale in-memory run context when backing session state remains active.

## Best Possible Solution

Implement a focused agents-runtime fix: treat clean gateway close 1000 wait interruptions as recoverable for subagent supervision, schedule another wait after pending results, and re-wait or reattach when stale sweep finds a non-terminal running backing session while preserving existing missing, terminal, aborted, drifted, and stale failure semantics.

## Reproduction Assessment

Yes. There is a high-confidence source-level reproduction path on current main: feed a `gateway closed (1000 normal closure)` wait error, an `agent.wait` pending result, or a stale active run with `sessionEntry.status === "running"` and no live run context, and the inspected branches still produce the reported dropped supervision or false lost/error behavior.

## Solution Assessment

Yes. The narrowest maintainable solution belongs in `src/agents/run-wait.ts`, `src/agents/subagent-registry-run-manager.ts`, and `src/agents/subagent-registry.ts` with regression tests, rather than a broad gateway lifecycle rewrite.
