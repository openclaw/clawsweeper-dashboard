## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.98

Full review comments:

- **[P1] Do not delete rebuilds solely because they exceed five minutes:** `extensions/memory-core/src/memory/manager-sync-ops.ts:151-152`
  - body: A legitimate reindex can run longer than five minutes, especially the large or OOM-prone rebuilds this PR cites. Any later sync from another gateway or CLI process will treat that still-active `.tmp-*` database as stale and unlink it, discarding the rebuild and causing another memory outage. Use an explicit ownership or liveness protocol, or lifecycle-owned recovery, rather than file age alone.
  - confidence: 0.98
- **[P2] Remove the stale database sidecars with the base file:** `extensions/memory-core/src/memory/manager-sync-ops.ts:140-145`
  - body: The filter explicitly skips `.backup-*-wal`, `.backup-*-shm`, `.tmp-*-wal`, and `.tmp-*-shm`, so the cleanup leaves the SQLite sidecars that interrupted swaps create. This fails the stated housekeeping goal and can strand WAL data after deleting only its base database; reuse the existing complete-artifact removal helper or remove each set together.
  - confidence: 0.99
- **[P2] Test the new checkpoint before the existing close checkpoint:** `extensions/memory-core/src/memory/manager.wal-checkpoint.test.ts:76-77`
  - body: The test calls `manager.close()` before moving the WAL away, but `closeMemoryDatabase()` on current main already runs `wal_checkpoint(TRUNCATE)`. Therefore this test passes even if the new checkpoint in `writeMeta()` is removed and does not prove the claimed crash window. Observe the database before close or inject the checkpoint boundary so the regression fails on current main.
  - confidence: 0.99
