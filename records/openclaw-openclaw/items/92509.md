## Summary

This PR should not merge as written: its process-kill evidence removes the persistent WAL before reopening, unlike OpenClaw’s real startup path, so it does not establish that committed metadata is lost or that a synchronous TRUNCATE checkpoint fixes the reported memory failure.

## What This Changes

The PR adds a best-effort `wal_checkpoint(TRUNCATE)` immediately after memory index metadata writes and adds tests asserting that checkpoint call and normal close/reopen persistence.

## Best Possible Solution

Reproduce the failure by killing a real OpenClaw process during the relevant sync or reindex phase, preserve the database together with its WAL/SHM files, and then fix the observed owner boundary—likely atomic-reindex/live-manager state—rather than checkpointing every metadata update.

## Reproduction Assessment

No. The supplied live script removes the WAL/SHM recovery files before inspection, while the added repository test performs an orderly manager close; neither reproduces a hard-killed OpenClaw restart against the untouched database files.

## Solution Assessment

No. The proposed checkpoint addresses main-file portability after discarding the WAL, but current OpenClaw and SQLite treat the WAL as persistent database state; the narrower maintainable fix should follow a production-faithful reproduction of the actual missing-metadata path.
