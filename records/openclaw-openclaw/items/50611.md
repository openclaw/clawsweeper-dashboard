- `impact:session-state`: The affected gate can prevent long-running sessions from flushing durable memory before compaction, which impacts preserved session context.

## Summary

Keep open. Current main and the latest release still allow `reserveTokensFloor` plus the memory-flush soft threshold to make the effective token gate non-positive, which disables the token-based memory-flush/preflight path; same-repo fix attempts are closed unmerged, and adjacent open memory-flush work addresses a different 0/0 counter bug.

## What This Changes

Fix memory flush and preflight compaction so oversized `reserveTokensFloor` values do not make the flush threshold non-positive and disable token-based flushing.

## Best Possible Solution

Land a focused current-main fix that centralizes a context-window-aware positive threshold for both token-based memory flush and preflight compaction, with regression coverage for `reserveTokensFloor >= contextWindowTokens` and explicit maintainer-approved compatibility behavior.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: with positive token usage and `reserveTokensFloor + softThresholdTokens >= contextWindowTokens`, the shared gate returns `null`, causing memory flush and token-based preflight compaction to skip.

## Solution Assessment

Unclear as final policy. A shared context-window-aware threshold helper is the narrow technical layer, but maintainers still need to choose clamp, warning, validation, or doctor behavior for existing oversized reserve configs.
