## Summary

Keep open. Current main still disables both token-based memory flush and preflight compaction when the computed threshold is non-positive, and the provided context shows open PR #50694 explicitly targets this issue with closing syntax, so the issue should wait for that PR or a replacement to land.

## What This Changes

The issue reports that `reserveTokensFloor` equal to the model context window disables memory compaction instead of leaving a usable flush threshold.

## Best Possible Solution

Keep this issue open and review or land PR #50694, or replace it with a maintainer-preferred fix that guarantees a positive token-based flush and preflight threshold with focused regression coverage for equal and greater-than-context reserve values.

## Reproduction Assessment

Yes, from source inspection. Calling `shouldRunMemoryFlush` or `shouldRunPreflightCompaction` with positive token usage, `contextWindowTokens` equal to `reserveTokensFloor`, and any nonnegative `softThresholdTokens` produces a non-positive threshold and returns false.

## Solution Assessment

Unclear. A shared clamp or validation guard is the right direction, but maintainers still need to choose the durable policy and location so memory flush and preflight compaction stay consistent.
