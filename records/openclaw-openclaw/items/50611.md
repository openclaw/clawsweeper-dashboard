## Summary

Keep open. Current main still disables token-based memory flush and preflight compaction when the computed threshold is non-positive, and open PR #50694 explicitly targets this issue with closing syntax, so the issue should wait for that PR or a maintainer-preferred replacement.

## What This Changes

The issue asks OpenClaw to prevent memory flush and preflight compaction from being disabled when reserveTokensFloor is equal to or greater than the context window.

## Best Possible Solution

Land #50694 or a replacement that applies one shared positive-threshold policy for memory flush and preflight compaction, with regression coverage for reserve values equal to and greater than the context window.

## Reproduction Assessment

Yes, by source inspection. Calling shouldRunMemoryFlush or shouldRunPreflightCompaction with positive token usage, contextWindowTokens equal to reserveTokensFloor, and any nonnegative softThresholdTokens produces a non-positive threshold and returns false.

## Solution Assessment

Unclear. A shared clamp or validation guard is the right fix shape, but maintainers still need to choose the durable policy and location so memory flush and preflight compaction remain consistent.
