## Summary

Keep open. Current main still disables the token-based memory flush and preflight-compaction gates when the computed threshold is non-positive, and open PR #50694 is already the unmerged implementation candidate that explicitly targets the same bug.

## What This Changes

The issue requests that memory flush and preflight compaction keep a positive trigger when `reserveTokensFloor` is equal to or greater than the context window.

## Best Possible Solution

Keep this issue open and review or land PR #50694, or replace it with a maintainer-preferred fix that guarantees a positive token-based flush and preflight-compaction threshold for equal and greater-than-context reserve values, with focused regression tests for both paths.

## Reproduction Assessment

Yes. A focused static reproduction is to call `shouldRunMemoryFlush` or `shouldRunPreflightCompaction` with positive token usage, `contextWindowTokens` equal to `reserveTokensFloor`, and any nonnegative `softThresholdTokens`; current main computes a non-positive threshold and returns false.

## Solution Assessment

Unclear. Clamping to preserve a positive threshold is the right direction, but current main has not implemented it and maintainers should choose the narrow policy location so memory-flush and preflight-compaction behavior remain consistent.
