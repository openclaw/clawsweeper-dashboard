## Summary

Keep open. Current main and the latest release still disable the token-based memory-flush/preflight compaction gate when the threshold is non-positive, and the canonical target-repo fix path at https://github.com/openclaw/openclaw/pull/74010 remains open and conflicting.

## What This Changes

The issue reports that memory flush/compaction does not trigger when `reserveTokensFloor` is configured equal to the model context window.

## Best Possible Solution

Refresh and land https://github.com/openclaw/openclaw/pull/74010, or an equivalent narrow patch that resolves an effective context-capped reserve once and applies it to both compaction gates with regression coverage for equal-to-context and greater-than-context reserve values.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: positive token usage plus `contextWindowTokens=200000`, `reserveTokensFloor=200000`, and a nonnegative soft threshold produces `threshold <= 0`, which disables the token-based flush/preflight gate.

## Solution Assessment

Yes for triage. Keeping this issue open while the replacement PR is refreshed or replaced is the maintainable path; the code fix should centralize effective reserve calculation instead of adding separate clamps at each caller.
