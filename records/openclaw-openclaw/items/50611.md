## Summary

Keep open. Current main and the latest release still drop the token-based memory-flush/preflight decision when the effective threshold is non-positive, while same-root fix attempts are closed or fork-conflicting and the final clamp/default/warning behavior still needs maintainer judgment.

## What This Changes

The issue reports that memory flush never triggers when `reserveTokensFloor` is configured equal to the model context window.

## Best Possible Solution

Land a focused current-main fix that centralizes a context-window-aware effective reserve or positive threshold for both token-triggered memory flush and preflight compaction, with regression coverage for `reserveTokensFloor >= contextWindowTokens` and explicit maintainer-approved compatibility behavior.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: with positive token usage and `reserveTokensFloor + softThresholdTokens >= contextWindowTokens`, the shared gate returns null before token usage can trigger memory flush or preflight compaction.

## Solution Assessment

Unclear as a final policy. A shared context-window-aware threshold helper is the narrow technical shape, but maintainers still need to choose the compatibility behavior for existing oversized reserve configs.
