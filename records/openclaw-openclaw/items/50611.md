## Summary

Keep open. The inspected source still disables token-based memory flush and preflight compaction when the computed threshold is non-positive, and the provided GitHub context shows open PR #50694 explicitly targets this issue with closing syntax, so the issue should remain open until that PR or a replacement lands.

## What This Changes

The issue requests a guard so `reserveTokensFloor` values equal to or greater than the context window cannot prevent memory flush or preflight compaction from triggering.

## Best Possible Solution

Keep the issue open and land PR #50694 or a maintainer-preferred replacement that guarantees a positive token-based trigger for both memory flush and preflight compaction when reserve and soft-threshold settings consume the context window, with regression tests for equality and greater-than-context reserve values.

## Reproduction Assessment

Yes. A focused static reproduction is to call `shouldRunMemoryFlush` or `shouldRunPreflightCompaction` with positive token usage, `contextWindowTokens` equal to `reserveTokensFloor`, and any nonnegative `softThresholdTokens`; the inspected source resolves a non-positive threshold and returns false.

## Solution Assessment

Unclear. A shared clamp or validation guard is the right direction, but maintainers still need to choose whether the policy belongs in the flush-plan builder, the shared gate helper, or config validation so both runtime paths stay consistent.
