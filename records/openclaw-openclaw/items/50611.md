## Summary

Keep open: current main and the latest release still let the memory-flush token gate return no runnable threshold when the effective reserve floor consumes the context window, and the same-repo fix attempts are closed unmerged while the remaining open fork candidates are conflicting.

## What This Changes

The issue requests that memory flush still trigger when `reserveTokensFloor` is configured equal to the model context window instead of silently disabling the flush gate.

## Best Possible Solution

Land a narrow same-repo fix that keeps the effective memory-flush threshold positive when the configured reserve floor reaches the context window, with regression coverage and no new config surface unless maintainers explicitly choose validation or warning semantics.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: pass `contextWindowTokens: 200000`, `reserveTokensFloor: 200000`, `softThresholdTokens: 4000`, and a positive token count into `shouldRunMemoryFlush`, and the computed threshold path returns no runnable gate. I did not execute a test because this review is read-only and tests may write artifacts.

## Solution Assessment

Unclear as a product decision: a gate-level effective reserve clamp is the smallest maintainable bug fix, but maintainers should decide whether misconfigured `reserveTokensFloor` should also be normalized in the memory-core plan or surfaced as a config warning.
