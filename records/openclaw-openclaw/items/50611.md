## Summary

Keep open. Current main and the latest release still return no token-based memory flush when the effective threshold is non-positive, and the same-repo fix attempts are closed unmerged while remaining linked work needs a maintainer policy choice.

## What This Changes

The issue requests that memory flush still trigger, or that `reserveTokensFloor` be normalized, when `reserveTokensFloor` is configured equal to the active context window.

## Best Possible Solution

Land a focused current-main fix that centralizes a context-window-aware effective reserve or positive memory threshold for both preflight compaction and token-triggered memory flush, with regression coverage for `reserveTokensFloor >= contextWindowTokens`.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: with positive token usage, `contextWindowTokens` equal to `reserveTokensFloor`, and any nonnegative soft threshold, the shared gate returns null before token usage can trigger memory flush or preflight compaction.

## Solution Assessment

Unclear as a final policy. A shared context-window-aware threshold helper is the narrow technical shape, but maintainers still need to choose the compatibility behavior for existing oversized `reserveTokensFloor` configs.
