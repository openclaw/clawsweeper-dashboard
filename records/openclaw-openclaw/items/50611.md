## Summary

Keep open. Current main and the latest release still collapse the token-based memory-flush and preflight-compaction threshold to zero when the reserve consumes the context window, and https://github.com/openclaw/openclaw/pull/50694 remains the open closing-linked implementation path.

## What This Changes

The issue reports that memory flush never triggers when `reserveTokensFloor` is configured equal to the model context window.

## Best Possible Solution

Keep this issue open and review, rebase, validate, and land https://github.com/openclaw/openclaw/pull/50694 or an equivalent shared threshold fix that covers both preflight compaction and runtime memory flush.

## Reproduction Assessment

Yes. By source inspection on current main, `contextWindowTokens=200000`, `reserveTokensFloor=200000`, and the default nonnegative soft threshold produce a zero threshold, causing the shared gate to return `null` instead of flushing or preflight compacting.

## Solution Assessment

Yes for keeping this issue open while the closing-linked PR remains unmerged. The narrow maintainable solution is a shared context-window-aware positive-threshold helper used by both runtime paths, with regression coverage for equal and greater-than-context reserve values.
