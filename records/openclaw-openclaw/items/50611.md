## Summary

Keep open. Current main and v2026.6.10 still drop the token-based memory flush/preflight path when the effective threshold is non-positive, while the same-repository fix attempts are closed unmerged and the broader open compaction-budget PRs only partially overlap this specific bug.

## What This Changes

The issue asks OpenClaw to avoid disabling memory flush when `reserveTokensFloor` is configured equal to or above the active context window.

## Best Possible Solution

Land a focused current-main fix that centralizes a context-window-aware effective reserve or positive threshold for both preflight compaction and token-triggered memory flush, with regression coverage for equal and greater-than-context reserve values.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: with positive token usage, `reserveTokensFloor >= contextWindowTokens`, and nonnegative soft threshold, the shared gate returns null before token usage can trigger memory flush or preflight compaction.

## Solution Assessment

Unclear as a final policy. A shared threshold/effective-reserve helper is the narrow technical shape, but maintainers still need to choose the compatibility behavior for existing oversized `reserveTokensFloor` configs.
