## Summary

Keep open. Current main and the latest release still allow `reserveTokensFloor` to make the shared token gate threshold non-positive, and the direct closing PR remains open and unmerged.

## What This Changes

The report asks for memory flush and preflight compaction to still trigger when `reserveTokensFloor` is configured equal to or above the model context window.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/50694 or a replacement that centralizes an effective positive threshold/reserve calculation for both preflight compaction and runtime memory flush, with regression coverage for equal-to-context and greater-than-context reserve values.

## Reproduction Assessment

Yes. Source inspection gives a direct path: with positive token usage, `contextWindowTokens=200000`, `reserveTokensFloor=200000`, and a nonnegative soft threshold, the shared gate returns `null` before token-based flush or preflight compaction can trigger.

## Solution Assessment

Yes for triage: keeping this open while the closing-linked PR is unmerged is the right outcome. The maintainable fix is a shared effective-reserve or positive-threshold calculation rather than separate ad hoc clamps at each caller.
