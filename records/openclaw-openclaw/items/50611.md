## Summary

Keep open. Current main and the latest release still let the raw reserve floor collapse the token-based memory-flush/preflight threshold, and the open closing-linked PR remains unmerged.

## What This Changes

Fix memory-flush and preflight compaction so `reserveTokensFloor >= contextWindow` cannot suppress token-based compaction.

## Best Possible Solution

Use one shared effective-reserve or positive-threshold calculation for both preflight compaction and runtime memory flush, with regression coverage for reserve values equal to and above the context window.

## Reproduction Assessment

Yes. Source inspection on current main gives a direct path: with a positive token count, `contextWindowTokens=200000`, `reserveTokensFloor=200000`, and any nonnegative soft threshold, the shared gate returns `null` instead of allowing token-based flush or preflight compaction.

## Solution Assessment

Yes for keeping this issue open while the closing-linked PR is unmerged. The maintainable fix is to centralize the effective threshold/reserve calculation rather than adding separate ad hoc clamps at each call site.
