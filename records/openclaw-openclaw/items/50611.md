## Summary

Keep open. Current `main` and `v2026.6.8` still disable the token-based memory-flush/preflight path when `reserveTokensFloor` plus the soft threshold leaves a non-positive threshold; prior same-repo fix PRs are closed unmerged, and the open fork PRs that reference this issue are conflicting and need maintainer direction rather than cleanup closure.

## What This Changes

The issue asks OpenClaw to prevent `reserveTokensFloor` values equal to or above the context window from disabling memory-flush and preflight-compaction thresholds.

## Best Possible Solution

Keep this issue open until maintainers land a focused current-main fix that guarantees positive token-based memory-flush and preflight-compaction thresholds for oversized reserve settings, with regression coverage and compatibility/docs alignment for the chosen policy.

## Reproduction Assessment

Yes, at source level: with `contextWindowTokens=200000`, `reserveTokensFloor=200000`, and the default 4000 soft threshold, the gate computes a non-positive threshold and returns no flush/preflight action. I did not run tests because this review was read-only.

## Solution Assessment

Unclear as proposed: the report's 25% default is one plausible policy, but the best maintainable fix should normalize the effective reserve below the context window or enforce a positive threshold after maintainers choose the compatibility behavior.
