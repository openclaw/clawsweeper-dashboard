## Summary

Keep open. Current main and v2026.6.6 still carry an uncapped `reserveTokensFloor` into the memory-flush/preflight threshold, and the non-positive-threshold branch still disables token-triggered flush; the linked fork PRs reference this issue but are open, conflicting, and unmerged.

## What This Changes

The issue reports that setting `reserveTokensFloor` equal to the model context window leaves memory-flush/preflight thresholds non-positive so memory flushing never runs.

## Best Possible Solution

Land a maintainer-preferred fix that uses one canonical effective-reserve or threshold calculation for both memory flush and preflight compaction, with regression coverage for equal and greater-than-context reserve values.

## Reproduction Assessment

Yes, source-reproducible: with `contextWindowTokens=200000`, `reserveTokensFloor=200000`, and `softThresholdTokens=4000`, the gate computes a zero threshold and returns `null` before `shouldRunMemoryFlush` can trigger. I did not run a live test because this review is read-only.

## Solution Assessment

No, not as a final direction yet: defaulting to 25% is one plausible policy, but the best fix is a shared effective-reserve or positive-threshold rule across memory flush and preflight compaction. The open linked PRs need maintainer review for clamp semantics and upgrade behavior.
