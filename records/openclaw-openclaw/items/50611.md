## Summary

Keep open. Current main and the latest release still collapse the token-based memory-flush/preflight gate to `null` when the configured reserve plus soft threshold consumes the context window, and https://github.com/openclaw/openclaw/pull/50694 remains an open closing-linked implementation candidate.

## What This Changes

The issue reports that memory flush/preflight compaction never triggers when `reserveTokensFloor` equals or exceeds the context window.

## Best Possible Solution

Refresh, land, or replace https://github.com/openclaw/openclaw/pull/50694 with one shared positive-threshold calculation used by both preflight compaction and runtime memory flush, plus regression coverage for reserve values equal to or above the context window.

## Reproduction Assessment

Yes. Source inspection on current main gives a direct path: `contextWindowTokens=200000`, `reserveTokensFloor=200000`, and any nonnegative soft threshold make the shared gate return `null` instead of flushing or preflight compacting.

## Solution Assessment

Yes for keeping this issue open while the closing-linked PR remains unmerged. The maintainable fix is a shared context-window-aware positive-threshold helper used by both preflight compaction and runtime memory flush.
