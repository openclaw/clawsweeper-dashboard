## Summary

Keep open. Current main still disables the token-based memory-flush and preflight compaction gates when `reserveTokensFloor` reaches the context window, the latest release has the same gate, and the related replacement PR is still open and conflicting.

## What This Changes

The issue reports that setting `reserveTokensFloor` at or above the model context window prevents memory-flush and preflight compaction token gates from firing.

## Best Possible Solution

Refresh and land https://github.com/openclaw/openclaw/pull/74010, or an equivalent narrow patch that computes an effective context-capped reserve once and feeds both compaction gates with regression coverage for equal-to-context and greater-than-context reserve values.

## Reproduction Assessment

Yes. Source inspection on current main gives a direct path: with positive token usage, `contextWindowTokens=200000`, `reserveTokensFloor=200000`, and a nonnegative soft threshold, the shared gate returns `null` before token-based flush or preflight compaction can trigger.

## Solution Assessment

Yes for triage. Keeping this open until a refreshed related PR or replacement lands is the maintainable path; the code fix should centralize effective reserve calculation instead of adding separate clamps at each caller.
