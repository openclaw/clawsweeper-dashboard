## Summary

Keep open. Current main and the latest release still collapse the token-based memory-flush and preflight compaction gate to a non-positive threshold for the reported configuration, and https://github.com/openclaw/openclaw/pull/50694 remains an open closing-linked implementation candidate.

## What This Changes

The issue reports that memory flushing and compaction fail to trigger when `reserveTokensFloor` is configured equal to the model context window.

## Best Possible Solution

Keep this issue open and review, refresh, land, or replace https://github.com/openclaw/openclaw/pull/50694 with a shared positive-threshold calculation used by both preflight compaction and runtime memory flush, plus regression coverage for reserve values equal to or above the context window.

## Reproduction Assessment

Yes. Source inspection on current main gives a direct path: `contextWindowTokens=200000`, `reserveTokensFloor=200000`, and a nonnegative soft threshold make the shared gate return `null` instead of allowing token-based flush or preflight compaction.

## Solution Assessment

Yes for keeping this issue open while the closing-linked PR remains unmerged. The maintainable fix is a shared context-window-aware positive-threshold helper used by both preflight compaction and runtime memory flush, not a separate uncoordinated repair lane.
