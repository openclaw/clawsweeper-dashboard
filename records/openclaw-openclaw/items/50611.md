## Summary

Keep open: current main still returns no memory-flush/preflight-compaction gate state when the computed threshold is non-positive, and the same direct fix attempts in the target repository were closed unmerged. The remaining work is a real core session-state bug, but the exact clamp or validation policy needs maintainer direction because active compaction-setting PRs overlap the same config semantics.

## What This Changes

The issue reports that memory flush never triggers when `reserveTokensFloor` is set equal to the model context window because the computed threshold becomes non-positive.

## Best Possible Solution

Land one maintainer-approved context-window-aware reserve/threshold rule shared by token-based memory flush and preflight compaction, with regression coverage for `reserveTokensFloor >= contextWindowTokens` and no unrelated config semantic changes.

## Reproduction Assessment

Yes at source level: with positive token usage and `contextWindowTokens <= reserveTokensFloor + softThresholdTokens`, the current gate produces a non-positive threshold and `shouldRunMemoryFlush` returns `false`. I did not run a local test because this review is read-only and source inspection is sufficient for the gate behavior.

## Solution Assessment

Unclear. A shared effective-threshold helper is likely the right owner boundary, but maintainers still need to choose the public behavior for oversized configured reserves and overlapping compaction-budget semantics.
