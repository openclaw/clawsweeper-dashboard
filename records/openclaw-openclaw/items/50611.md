## Summary

Keep open. Current main and the latest release still disable token-based memory flush/preflight when the computed threshold is non-positive, and the open implementation candidate at https://github.com/openclaw/openclaw/pull/50694 explicitly targets this issue but is still unmerged.

## What This Changes

The issue requests that memory flush still trigger when `reserveTokensFloor` is configured equal to the model context window instead of making the token threshold unreachable.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/50694 or a replacement that centralizes context-window-aware threshold resolution for both preflight compaction and memory flush, with regression coverage for equal and greater-than-context reserve values.

## Reproduction Assessment

Yes, by source inspection on current main. With `contextWindowTokens` and `reserveTokensFloor` both 200000 plus any positive `softThresholdTokens`, the shared gate computes a zero threshold and returns `null`, so both token-based memory flush and preflight compaction return false.

## Solution Assessment

Yes for keeping this issue open while the linked implementation candidate is reviewed. The maintainable fix shape is a shared positive-threshold policy used by both runtime paths, not another disconnected repair job.
