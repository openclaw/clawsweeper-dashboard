## Summary

Keep open. Current main and v2026.5.20 still disable the token-based memory flush/preflight path when `reserveTokensFloor` collapses the threshold to zero, and an open closing-linked PR remains the active fix path.

## What This Changes

The issue reports that `reserveTokensFloor` equal to the context window prevents token-based memory flush and preflight compaction from triggering.

## Best Possible Solution

Use one shared effective-reserve or positive-threshold calculation for both runtime memory flush and preflight compaction, with regression coverage for `reserveTokensFloor` equal to and above `contextWindowTokens`.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: with positive tokens, `contextWindowTokens=200000`, `reserveTokensFloor=200000`, and any nonnegative soft threshold, the shared gate returns `null` instead of allowing token-based flush or preflight compaction.

## Solution Assessment

Yes for keeping this issue open until a fix lands. The maintainable fix is a shared effective-reserve or positive-threshold calculation, not separate ad hoc clamps in each caller.
