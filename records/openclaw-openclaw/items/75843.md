## Summary

Keep open. Current main still adds a hidden 30s Active Memory setup grace to the configured 15s recall timeout, uses the inflated 45s value for both the plugin watchdog and the `before_prompt_build` hook registration, and has tests that explicitly assert that behavior. This remains a narrow plugin-owned timeout/performance bug, not a duplicate of the earlier fix that made per-handler hook timeouts possible.

## What This Changes

The issue requests that Active Memory stop silently extending its configured prompt-build recall timeout with a hidden 30s setup grace that can hold the main command lane for 45s by default.

## Best Possible Solution

Fix this inside the Active Memory plugin so the visible `config.timeoutMs` is the default blocking budget for prompt-build recall and the main lane. If cold-start setup grace is still desired, make it an explicit documented advanced config field with manifest/schema/tests instead of silently extending the hook timeout.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: default config resolves `timeoutMs` to 15_000, current main adds `DEFAULT_SETUP_GRACE_TIMEOUT_MS` of 30_000, and the plugin registers `before_prompt_build` at 45_000 ms with tests asserting that exact value.

## Solution Assessment

Unclear. The reporter's one-line default-grace change is a plausible narrow fix, but the best maintainable patch should update the intentional setup-grace tests and decide whether the grace is removed by default or exposed as an explicit plugin option.
