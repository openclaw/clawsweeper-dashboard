## Summary

This issue should stay open because current main still matches the reported false-positive class: `test_speech` treats fresh bridge output bytes as `speechOutputVerified`, while the Google Meet health contract still has no loopback RMS, WebRTC sender level, or participant-audibility signal. The report is useful, but the remaining fix is a Google Meet plugin diagnostic/product decision rather than a conservative cleanup close.

## What This Changes

The issue asks Google Meet Chrome talk-back to distinguish byte-output success from evidence that non-silent audio reaches the Meet microphone/sender path, with docs or backend improvements for BlackHole/SoX failures.

## Best Possible Solution

Keep this in the Google Meet plugin by splitting TTS generation, command-output bytes, local loopback or sender evidence, and audibility limits in status/docs, then separately evaluate WebAudio/MediaStream talk-back as a supported backend if live validation supports it.

## Reproduction Assessment

Yes for the source-level false-positive class: current main can set `speechOutputVerified` from fresh `lastOutputBytes` without loopback, sender-level, or participant-audibility proof. No for the exact macOS live-meeting silence incident because this read-only review did not run a live Google Meet/BlackHole reproduction.

## Solution Assessment

Unclear. The narrowest maintainable first step is Google Meet plugin diagnostics and docs that stop overclaiming byte-output as audibility; a WebAudio/MediaStream talk-back backend may be better long term but needs product and live-validation review.
