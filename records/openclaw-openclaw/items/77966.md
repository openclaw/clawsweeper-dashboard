## Summary

Keep open: current main and the latest release still define `speechOutputVerified` as fresh bridge output bytes, not proof that non-silent audio reached the Meet sender path or remote participants. The related closed Google Meet items cover observe/listen health, PCM format, barge-in, and Linux guardrails, but none owns this narrower BlackHole/SoX false-positive talk-back health gap.

## What This Changes

The issue asks Google Meet Chrome talk-back health to stop reporting verified speech when the BlackHole/SoX command path only wrote bytes and participants heard silence.

## Best Possible Solution

Keep this as Google Meet plugin-owned health work: add explicit diagnostics for TTS generation, output-command bytes, loopback or WebRTC sender evidence, and remote-audibility limits, then document and live-validate the BlackHole/SoX failure mode before considering a browser-level backend.

## Reproduction Assessment

Partially. The health-semantics part is source-reproducible because current main can set `speechOutputVerified` from fresh output bytes only; the full live remote-silence path still needs a macOS BlackHole/SoX Google Meet reproduction.

## Solution Assessment

No. Current main documents the byte-based limitation and has adjacent audio hardening, but the maintainable fix needs a product decision on additive health fields, remediation docs, and possibly a browser-level audio backend rather than only renaming the existing byte check.
