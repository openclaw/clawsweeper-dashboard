## Summary

The current source still matches the reported false-positive class: `testSpeech` marks `speechOutputVerified` when bridge output bytes increase, while the Chrome health contract has no loopback RMS, Meet sender level, or participant-audibility field. The existing docs describe the narrower byte-based contract, so this is useful but needs maintainer product judgment rather than an automatic close or repair.

## What This Changes

The issue requests stronger Google Meet Chrome talk-back verification or diagnostics so `speechOutputVerified` does not imply participant-audible speech when the BlackHole/SoX path only wrote output bytes.

## Best Possible Solution

Keep this in the Google Meet plugin: separate TTS generation, command-output bytes, local loopback or sender evidence, and remote-audibility limits in diagnostics/docs, then evaluate WebAudio/MediaStream talk-back as a distinct supported backend decision.

## Reproduction Assessment

Yes for the source-level false-positive class: current main can set `speechOutputVerified` from fresh `lastOutputBytes` without sender-path or loopback-level proof. No for the exact remote-silence incident because that macOS/Meet setup was not live reproduced here.

## Solution Assessment

Unclear. The narrowest maintainable first step is plugin-owned diagnostic/docs clarification; making WebAudio/MediaStream injection a supported backend may be the durable fix but needs maintainer product and live-validation judgment.
