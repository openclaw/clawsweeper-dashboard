## Summary

Keep open: current main still defines `speechOutputVerified` as fresh bridge output bytes, not proof that Meet participants heard audio, and the related closed items only cover listen health, PCM quality, barge-in, or Linux guardrails.

## What This Changes

The issue reports Google Meet Chrome agent talk-back marking `speechOutputVerified` true from BlackHole/SoX output bytes while remote participants hear silence, and asks for stronger diagnostics, docs, or a non-BlackHole talk-back path.

## Best Possible Solution

Keep this as Google Meet plugin-owned work: add a health model that separates TTS generation, command output bytes, virtual-device loopback/sender evidence, and remote-audibility limits, then document the BlackHole failure mode and validate with a live Meet path.

## Reproduction Assessment

Yes for the health-semantics part: current source shows `speechOutputVerified` can become true solely from fresh output bytes. No for full live reproduction: I did not run a macOS BlackHole/SoX Google Meet session with a remote listener against current main.

## Solution Assessment

No. Current main documents the byte-based limitation and has adjacent audio hardening, but it does not yet solve the requested stronger proof or supported remediation for silent BlackHole/SoX talk-back.
