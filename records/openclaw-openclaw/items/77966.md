## Summary

Keep open. Current main still treats fresh command-bridge output bytes as `speechOutputVerified`; this report describes a narrower live talk-back false-positive that is not closed by the prior listen-first, fidelity, barge-in, or Linux-backend issues.

## What This Changes

Request that Google Meet Chrome talk-back distinguish command-output bytes from evidence that audio reached the Meet microphone/sender path, with docs or an alternate backend for BlackHole/SoX false positives.

## Best Possible Solution

Keep this in the Google Meet plugin: add explicit diagnostics/docs that separate TTS bytes, command-output bytes, local loopback or sender evidence, and remote-audibility limits, then evaluate a WebAudio/MediaStream backend as a separate supported transport decision.

## Reproduction Assessment

Yes for the false-positive health class from source: current main can set `speechOutputVerified` from fresh `lastOutputBytes` without checking BlackHole loopback level or Meet sender audio. The exact remote-silence setup was not live-run in this read-only review.

## Solution Assessment

Unclear. The narrowest maintainable first step is plugin-owned diagnostic/docs clarification; replacing the BlackHole path with WebAudio/MediaStream injection may be useful but needs maintainer product and live-validation judgment.
