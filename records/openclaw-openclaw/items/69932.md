## Summary

Keep open. Current main still has the BlueBubbles monitor typing start, restart, and stop paths calling `sendBlueBubblesTyping` without this PR's monitor-level Private API guard, and the current monitor tests still lack the disabled-Private-API regression case. The PR addresses Greptile's earlier test and stop-typing comments, but it remains open, dirty against the base branch, and overlaps the still-open helper/read-receipt work in #70799, so this needs maintainer reconciliation rather than cleanup closure.

## What This Changes

The PR wraps BlueBubbles monitor typing start, restart, and stop calls in the existing `privateApiEnabled` check and adds a monitor test that mocks Private API disabled.

## Best Possible Solution

Rebase this PR, keep the current-main monitor hardening intact, land the monitor-level typing guards plus disabled-path test if maintainers accept the `null`-as-disabled behavior, and reconcile any remaining mark-read/helper behavior with #70799 in the same review thread or a clearly split follow-up.

## Reproduction Assessment

Yes. The source path is clear on current main: with a BlueBubbles account whose cached Private API status is false or unknown, monitor processing can still reach unguarded typing start/restart/stop calls; the PR's focused unit reproduction is to mock `isBlueBubblesPrivateApiEnabled` as false and assert `sendBlueBubblesTyping` is not called.

## Solution Assessment

Unclear. The PR is a narrow plausible fix for the noisy disabled-Private-API monitor callbacks, but the `null` cache semantics and overlap with #70799 need maintainer choice before calling it the final best shape.
