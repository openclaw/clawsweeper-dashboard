## Summary

Keep open: current main and v2026.6.6 still expose only service-account Google Chat credentials and a single `chat.bot` token path, while reactions, media upload, and email-alias DM lookup still need delegated user-auth design rather than another stale cleanup close.

## What This Changes

Request adds optional Google Chat user OAuth so reactions, attachment uploads, and email-alias proactive DMs can use user-scoped credentials while service-account auth remains for webhook and bot operations.

## Best Possible Solution

Add a plugin-owned, SecretRef-aware delegated OAuth credential model with minimum per-operation scopes, explicit identity routing, service-account auth preserved for webhook/bot operations, and deterministic skip or warning behavior when user authorization is absent.

## Reproduction Assessment

Yes, at source level. Current main rejects any `userOAuth` shape, builds only `chat.bot` service-account tokens, and still routes reaction, upload, and DM lookup helpers through that one token path; no live GCP credentials were used in this read-only review.

## Solution Assessment

No, not as a raw `clientSecret` and `refreshToken` config sketch or automatic fix. The maintainable solution needs plugin-local dual authentication with SecretRefs, minimum operation-specific scopes, explicit sender identity, and approved fallback policy.
