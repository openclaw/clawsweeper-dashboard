## Summary

Keep open: current main still lacks Google Chat user OAuth configuration and scoped user-auth routing for outbound reactions, attachment upload, reaction typing, and email-alias DM lookup; the implementation remains service-account-only for those paths.

## What This Changes

The issue requests optional Google Chat user OAuth credentials alongside service-account auth so user-scoped operations such as reactions, media uploads, and proactive DMs can work.

## Best Possible Solution

Implement this in the Google Chat plugin as a secret-aware optional user OAuth credential model with scoped user-auth clients for user-only operations, service-account auth retained for bot/webhook flows, explicit fallbacks, docs, tests, and live Workspace validation.

## Reproduction Assessment

Yes, at source level: the strict config schema has no user OAuth shape, credential resolution is service-account-only, and upload/reaction/DM helpers all use the same chat.bot service-account token path. I did not run a live Google Workspace scenario in this read-only review.

## Solution Assessment

Yes as a product direction, but no implementation exists yet. The maintainable solution is a plugin-owned OAuth credential and scoped-client design rather than treating the inbound reaction issue or the external Drive-link workaround as a substitute.
