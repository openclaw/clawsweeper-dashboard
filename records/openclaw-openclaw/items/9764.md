## Summary

Keep open: current main and v2026.6.6 still expose Google Chat as a service-account channel with a single chat.bot auth path, while the requested reactions, attachment upload, reaction typing, and proactive-DM user behavior need a maintainer-approved delegated OAuth design.

## What This Changes

Add optional Google Chat user OAuth credentials so reactions, media uploads, reaction typing, and proactive DM lookup can use user-scoped Google Chat APIs while service-account auth remains for bot/webhook flows.

## Best Possible Solution

Approve a secret-aware delegated OAuth credential model for the bundled Google Chat plugin, route only user-scoped operations through that client, keep service-account auth for webhook/app flows, and document/test explicit fallback behavior when user OAuth is absent.

## Reproduction Assessment

Yes, at source level rather than a live Google Workspace run: current main has no user OAuth config, resolves only service-account credentials, and creates all Google Chat API tokens with the chat.bot scope. Official Google Chat API docs support the reported scope gap for reactions and attachment upload.

## Solution Assessment

Unclear as an implementation plan: the requested direction is plausible, but the exact credential shape, SecretRef handling, scopes, consent flow, sender attribution, and fallback semantics need maintainer product/security approval before implementation.
