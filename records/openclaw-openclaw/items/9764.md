## Summary

Keep open. Current main still lacks Google Chat user OAuth config and scoped OAuth client routing, so reactions, attachment upload, reaction typing, and email-alias DM behavior remain blocked behind the existing service-account-only auth path.

## What This Changes

Request optional Google Chat user OAuth support alongside service-account auth for reactions, media uploads, reaction typing, and proactive DM/email-alias lookup.

## Best Possible Solution

Add plugin-owned, secret-aware Google Chat user OAuth support with scoped per-operation clients while preserving service-account webhook and bot sends, plus documented fallback behavior when OAuth is absent.

## Reproduction Assessment

Yes, at source level. Current main has no accepted userOAuth config, no refresh-token credential resolver, and runtime auth remains fixed to service-account chat.bot, so the requested behavior cannot be enabled without code changes.

## Solution Assessment

Yes directionally, but not as an automated cleanup fix. The maintainable path is a Google Chat plugin-owned OAuth credential model with scoped clients after maintainer decisions on scopes, consent, fallback warnings, and defaults.
