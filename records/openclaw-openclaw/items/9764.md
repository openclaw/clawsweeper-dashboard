## Summary

Keep open. Current main still only accepts service-account Google Chat credentials and builds a chat.bot-scoped auth client; the related inbound-reaction issue and the open grant_type regression PR do not implement the requested user OAuth support for reactions, uploads, or email-alias DM lookup.

## What This Changes

The issue requests optional Google Chat user OAuth support so reactions, media uploads, and proactive email-alias DMs can use user-level scopes while service-account auth remains available.

## Best Possible Solution

Add plugin-owned, secret-aware Google Chat user OAuth support for the operations that require it, keep service-account auth as the default for webhook and bot-level sends, and document explicit fallback behavior when OAuth is absent.

## Reproduction Assessment

Yes, at source level. Current main has no accepted userOAuth config shape, no refresh-token credential resolution, and runtime auth remains fixed to service-account chat.bot, so the requested behavior cannot be enabled without code changes.

## Solution Assessment

Yes directionally, but not as an automated cleanup fix. The maintainable path is a Google Chat plugin-owned OAuth credential model with scoped per-operation clients and documented service-account fallback, after maintainer decisions on scopes, consent, and defaults.
