## Summary

Keep open: current main and v2026.6.8 still lack Google Chat user OAuth credential/config support, so reactions, attachment upload, reaction typing, and email-alias proactive DMs remain a valid product/security design item rather than cleanup.

## What This Changes

The issue requests optional Google Chat user OAuth credentials and auth routing for reactions, attachment uploads, reaction typing, and proactive DM/email-alias operations while keeping service-account webhook auth.

## Best Possible Solution

Approve a plugin-owned, secret-aware delegated OAuth design for Google Chat, route only user-scoped operations through that client, keep service-account auth for webhook/app flows, and document/test explicit fallback behavior when user OAuth is absent.

## Reproduction Assessment

Yes, at source level: current main has no user OAuth config, resolves only service-account credentials, rejects non-service-account credential objects, and creates Google Chat API tokens with the chat.bot scope. I did not run a live Google Workspace test in this read-only cleanup pass.

## Solution Assessment

Unclear as a concrete implementation until maintainers choose the OAuth product/security model. The requested split between service-account app auth and user OAuth is plausible, but the exact credential shape, SecretRef handling, scopes, sender attribution, and fallback semantics need approval first.
