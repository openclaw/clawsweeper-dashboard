## Summary

Keep open. Current main still has Google Chat wired to service-account credentials and the `chat.bot` scope, while the requested outbound reactions, media upload, reaction typing, and email-alias DM support require a new user OAuth credential/config surface and maintainer choices around secrets, scopes, and fallback behavior.

## What This Changes

The issue requests optional Google Chat user OAuth credentials for outbound reactions, media uploads, reaction typing, and proactive DMs while retaining service-account behavior.

## Best Possible Solution

Add plugin-owned, secret-aware Google Chat user OAuth support with scoped per-operation clients while preserving service-account webhook verification and bot-level sends, plus documented fallback behavior when OAuth is absent.

## Reproduction Assessment

Yes, at source level. Current main has no accepted `userOAuth` config, no refresh-token credential resolver, service-account-only credential validation, and runtime auth fixed to `chat.bot`.

## Solution Assessment

Yes directionally, but not as an automated cleanup fix. The maintainable path is a Google Chat plugin-owned OAuth credential model with scoped clients after maintainer decisions on scopes, consent, fallback warnings, and defaults.

## Risks / Open Questions

- Adding user OAuth introduces secret-bearing refresh-token config, so schema sensitivity, SecretRef resolution, redaction, token refresh, consent scopes, and guarded Google auth transport need explicit design.
- Current Google Chat action discovery can expose `upload-file` and reaction actions when only service-account credentials exist, so users may continue hitting runtime 403s until OAuth-aware routing or fallback warnings are implemented.
- The open Google Chat all-messages/threading work at https://github.com/openclaw/openclaw/issues/44347 may share OAuth design questions, but the closed inbound-reaction issue at https://github.com/openclaw/openclaw/issues/73681 is not a substitute for outbound user OAuth support.
