## Label Justifications

- `P2`: This is a real Google Chat capability gap with limited channel scope, not an emergency or current core-runtime outage.
- `impact:message-loss`: The request covers unsupported reaction, attachment upload, and proactive DM delivery paths that can fail or be skipped in Google Chat.
- `impact:auth-provider`: The central missing surface is user OAuth credential resolution and routing alongside the existing service-account auth path.

## Summary

Keep open. Current main and latest v2026.5.12 still use service-account `chat.bot` auth for Google Chat, so the requested user OAuth credential model and per-operation routing for reactions, media upload, and email-alias DMs are not implemented; related Google Chat issues and the external workaround do not supersede this work.

## What This Changes

Request optional Google Chat user OAuth support so reactions, attachment uploads, reaction typing, and proactive DMs can use user-scoped Chat APIs while service-account webhook auth remains available.

## Best Possible Solution

Add plugin-owned, secret-aware Google Chat user OAuth support with scoped per-operation clients, service-account webhook/bot auth preserved, and documented fallback behavior when OAuth is absent.

## Reproduction Assessment

Yes, at source level. Current main has no accepted `userOAuth` config, no refresh-token resolver, service-account credential validation only, and runtime auth fixed to `chat.bot`; I did not run a live Google Workspace repro.

## Solution Assessment

Yes directionally, but not as an automated cleanup fix. The maintainable path is a Google Chat plugin-owned OAuth credential model with scoped clients after maintainer decisions on scopes, consent, fallback warnings, and defaults.
