## Summary

Keep open. Current main still implements Google Chat with service-account credentials and the `chat.bot` scope only, so reactions, attachment upload, and email-alias DM lookup still lack the requested user OAuth credential and routing contract.

## What This Changes

Add optional Google Chat user OAuth credentials and scoped auth routing for reactions, media uploads, and email-alias proactive DMs while preserving service-account webhook and bot auth.

## Best Possible Solution

Add plugin-owned, secret-aware Google Chat user OAuth support with scoped per-operation clients, preserved service-account webhook/bot auth, and documented fallback behavior when OAuth is absent.

## Reproduction Assessment

Yes, at source level. Current main has no accepted `userOAuth` config, no refresh-token resolver, service-account credential validation only, and runtime auth fixed to `chat.bot`; I did not run a live Google Workspace repro.

## Solution Assessment

Yes directionally, but not as an automated cleanup fix. The maintainable path is a Google Chat plugin-owned OAuth credential model with scoped clients after maintainer decisions on scopes, consent, fallback warnings, and defaults.
