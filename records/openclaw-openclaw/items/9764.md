## Summary

Keep open: current main still has Google Chat wired around service-account credentials and a single `chat.bot` token path, while the issue asks for a new user OAuth credential model and scoped routing for user-only operations. The adjacent open grant-type PR and the closed inbound-reaction issue do not supersede this outbound/user-auth feature.

## What This Changes

Add optional user OAuth support to Google Chat for reactions, media uploads, and proactive direct messages while retaining service-account auth for existing bot/webhook flows.

## Best Possible Solution

Implement this in the Google Chat plugin as optional, secret-aware user OAuth with scoped user-auth clients for user-only operations, service-account auth retained for bot/webhook flows, explicit fallbacks, docs, tests, and live Workspace validation.

## Reproduction Assessment

Yes, at source level: current main has no user OAuth config/schema, resolves only service-account credentials, and routes reactions, attachment upload, and email-target DM lookup through the same `chat.bot` token path. I did not run a live Google Workspace repro in this read-only review.

## Solution Assessment

Yes as a product direction, but there is no implementation yet. The maintainable path is plugin-owned OAuth credential modeling and scoped-client routing, not treating the external workaround, inbound-reaction issue, or grant-type PR as substitutes.
