## Label Justifications

- `P2`: This is a normal-priority Google Chat channel capability gap with clear user impact but limited blast radius and required design work.
- `impact:message-loss`: The missing auth path prevents or breaks outbound Google Chat reactions, media uploads, and some proactive DM delivery paths.
- `impact:auth-provider`: The request is centered on new Google Chat OAuth credential resolution and per-operation auth-scope routing.

## Summary

Keep open. Current main and the latest release still have Google Chat wired to service-account credentials and the `chat.bot` scope only, with no user OAuth config or scoped routing for reactions, uploads, reaction typing, or email-alias DMs.

## What This Changes

Add optional Google Chat user OAuth credentials and scoped routing for reactions, attachment uploads, reaction typing, and proactive DM/email-alias lookup while preserving service-account auth.

## Best Possible Solution

Add plugin-owned, secret-aware Google Chat user OAuth support with scoped per-operation clients, service-account webhook/bot auth preserved, and documented fallback behavior when OAuth is absent.

## Reproduction Assessment

Yes, at source level. Current main accepts service-account config only, has no refresh-token resolver, hardcodes `chat.bot` auth, and routes the affected operations through that same account token path.

## Solution Assessment

Yes directionally, but not as an automated cleanup fix. The maintainable path is a Google Chat plugin-owned OAuth credential model with scoped clients after maintainer decisions on scopes, consent, fallback warnings, and defaults.
