## Summary

Keep open. Current main and the latest shipped tag still expose Google Chat service-account auth only, with no accepted user OAuth config, refresh-token credential resolver, or scoped OAuth client path for reactions, media upload, reaction typing, or email-alias DM lookup.

## What This Changes

This issue requests optional Google Chat user OAuth support alongside service-account auth for reactions, attachment uploads, reaction typing, and proactive DMs.

## Best Possible Solution

Add plugin-owned, secret-aware Google Chat user OAuth support with scoped per-operation clients, service-account webhook/bot preservation, and documented fallback behavior when OAuth is absent.

## Reproduction Assessment

Yes, at source level. Current main has no accepted userOAuth config, no refresh-token credential resolver, and runtime auth remains fixed to service-account chat.bot, so the requested behavior cannot be enabled without code changes.

## Solution Assessment

Yes directionally, but not as an automated cleanup fix. The maintainable path is a Google Chat plugin-owned OAuth credential model with scoped clients after maintainer decisions on scopes, consent, fallback warnings, and defaults.
