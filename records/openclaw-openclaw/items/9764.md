## Summary

Keep open: current main still has Google Chat user operations wired through service-account credentials and the `chat.bot` scope only, so user OAuth for outbound reactions, attachment uploads, reaction typing, and email-alias DM lookup remains unimplemented.

## What This Changes

Add optional Google Chat user OAuth credentials and route user-scoped operations through them while keeping service-account auth for webhook and bot flows.

## Best Possible Solution

Implement this in the Google Chat plugin with a strict secret-aware user OAuth credential model, scoped OAuth clients for user-only operations, service-account auth retained for bot/webhook flows, fallback warnings, docs, tests, and live Workspace validation.

## Reproduction Assessment

Yes, at source level: current main has no user OAuth schema/config, credential resolver, or OAuth-client routing, and the affected API helpers still use the service-account access token path. I did not establish a live Google Workspace reproduction in this read-only review.

## Solution Assessment

Yes as a product direction, but no implementation exists on current main. The maintainable path is Google Chat plugin-owned user OAuth and scoped-client routing, not closing this as covered by inbound reaction handling or the external Drive-upload workaround.
