## Summary

Current main still has no Google Chat user OAuth credential model or scoped client routing: reactions, attachment uploads, and direct-message lookup all share the service-account `chat.bot` token path. The request remains useful but needs maintainer product and security decisions because it adds secret-bearing configuration, OAuth setup/refresh behavior, and capability fallback policy.

## What This Changes

Add optional Google Chat user OAuth credentials and route reactions, media uploads, and proactive direct messages through user-scoped authentication while retaining service-account behavior.

## Best Possible Solution

Implement a plugin-owned, SecretRef-capable user OAuth credential model and scoped OAuth client, route only user-scope-required operations through it, preserve service-account webhook and bot operations, and expose capabilities or diagnostics that accurately reflect whether user OAuth is configured.

## Reproduction Assessment

Yes at source level: current main routes the affected operations through one `chat.bot` service-account token path, and the docs acknowledge that reaction mode requires unsupported user OAuth. This read-only review did not run a live Google Chat API reproduction.

## Solution Assessment

Yes on the ownership boundary, but not yet on the exact design: user OAuth belongs in the Google Chat plugin, while maintainers still need to approve the secret model, setup flow, operation routing, and absent-OAuth behavior.

## Risks / Open Questions

- OAuth client secrets and refresh tokens require SecretRef support, redaction, secure token refresh, and the existing SSRF-guarded Google transport; storing raw secrets in the sketched config would be unsafe.
- Adding a second credential path can create ambiguous operation routing or account inheritance unless the credential and capability model is closed and tested for multi-account setups.
- The fallback policy is product-sensitive: current actions can advertise reactions and uploads even when the configured service account cannot execute them, so maintainers must choose between capability gating, explicit errors, or documented degradation.
