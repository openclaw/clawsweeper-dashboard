## Summary

Keep open: current main still has Google Chat wired around service-account credentials and a `chat.bot` token, with no user OAuth config, refresh-token credential path, or scoped user-auth routing for reactions, attachment upload, reaction typing, or email-alias DM lookup. The external merged workaround in https://github.com/mmackelprang/claude-code-channels/pull/2 is useful prior art, but it uses Google Drive links outside this repository and does not implement the requested OpenClaw Google Chat OAuth support; https://github.com/openclaw/openclaw/issues/73681 tracks the opposite inbound-reaction gap, not this outbound/user-auth work.

## What This Changes

The issue requests optional Google Chat user OAuth support for reactions, media uploads, reaction typing, and proactive DMs while retaining service-account auth for existing bot/webhook flows.

## Best Possible Solution

Implement this in the Google Chat plugin as optional, secret-aware user OAuth with scoped user-auth clients for user-only operations, service-account auth retained for bot/webhook flows, explicit fallbacks, docs, tests, and live Workspace validation.

## Reproduction Assessment

Yes, at source level: current main lacks a user OAuth config/schema and routes the named operations through the same service-account `chat.bot` token path. I did not establish a live Google Workspace reproduction in this read-only review.

## Solution Assessment

Yes as a product direction, but no implementation exists yet. The maintainable solution is a Google Chat plugin-owned OAuth credential model and scoped-client routing rather than treating the inbound reaction issue or an external Drive-link workaround as substitutes.
