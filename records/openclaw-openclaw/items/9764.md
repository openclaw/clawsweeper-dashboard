## Summary

Keep open: current main still implements Google Chat with service-account credentials and a single `chat.bot` auth path, while this issue asks for a new user OAuth credential model and scoped routing for reactions, attachment uploads, reaction typing, and email-alias DM lookup. Prior comments and the merged external Google Drive workaround are useful prior art, but they do not ship the requested OpenClaw Google Chat OAuth support.

## What This Changes

Request optional Google Chat user OAuth credentials and scoped user-auth routing for reactions, media uploads, reaction typing, and proactive DM/email-alias lookup.

## Best Possible Solution

Implement this in the Google Chat plugin as optional, secret-aware user OAuth with scoped user-auth clients for user-only operations, service-account auth retained for bot/webhook flows, explicit fallbacks, docs, focused tests, and live Workspace validation.

## Reproduction Assessment

Yes, at source level: current main has no user OAuth config/schema, resolves only service-account credentials, and routes reactions, attachment upload, and DM lookup through the same `chat.bot` token path. I did not establish a live Google Workspace reproduction in this read-only review.

## Solution Assessment

Yes as a product direction, but there is no implementation yet. The maintainable path is plugin-owned OAuth credential modeling and scoped-client routing, not treating the external Drive-link workaround or the closed inbound-reaction issue as substitutes.

## Risks / Open Questions

- Adding user OAuth introduces refresh-token and client-secret handling, so schema strictness, SecretRef support, redaction, token refresh, consent scopes, and scoped-client caching need deliberate design.
- Google Chat currently advertises reaction/media actions when service-account credentials exist, so users can still hit Google authorization failures until OAuth-aware routing or explicit fallback warnings are implemented.
- No live Google Workspace smoke was run in this read-only review; final implementation needs real Workspace proof for reaction create/list, attachment upload, reaction typing fallback, and email-alias DM lookup.
