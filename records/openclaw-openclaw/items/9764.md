## Summary

Keep this issue open: current main still has service-account-only Google Chat credentials and routing, so outbound reactions, attachment upload, and email-alias direct-message lookup do not have the requested user OAuth path.

## What This Changes

Add optional Google Chat user OAuth credentials and scoped routing for reactions, media uploads, and proactive direct-message lookup while retaining service-account flows.

## Best Possible Solution

Implement the support inside the bundled Google Chat plugin with a secret-aware user OAuth credential model, scoped clients for user-only operations, service-account auth retained for webhook and bot flows, docs, tests, and live Workspace validation.

## Reproduction Assessment

Yes, at source level: current main exposes upload, reaction, and direct-message lookup paths, but their bearer token comes from the service-account `chat.bot` scope. I did not run a live Google Workspace API scenario in this read-only review.

## Solution Assessment

Yes as a product direction, but not yet as an implementation. The maintainable path is a plugin-owned OAuth credential model with scoped routing and fallback semantics, not treating inbound reaction work or an external Drive-link workaround as a substitute.
