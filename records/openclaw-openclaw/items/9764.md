## Summary

Keep open: current main still lacks Google Chat user OAuth configuration and scoped user-auth routing for outbound reactions, attachment upload, reaction typing, and email-alias DM lookup; the bundled plugin remains service-account-only for those paths.

## What This Changes

Add optional Google Chat user OAuth credentials and route reactions, attachment uploads, reaction typing, and proactive DM lookup through scoped user-auth clients while preserving service-account webhook and bot flows.

## Best Possible Solution

Implement this in the Google Chat plugin as a secret-aware optional user OAuth credential model with scoped user-auth clients for user-only operations, service-account auth retained for bot and webhook flows, explicit fallback behavior, docs, tests, and live Workspace validation.

## Reproduction Assessment

Yes, at source level: the strict config schema has no user OAuth shape, credential resolution is service-account-only, and upload/reaction/DM helpers all use the same chat.bot token path. I did not establish a live Google Workspace reproduction in this read-only review.

## Solution Assessment

Yes as a product direction, but no implementation exists yet. The maintainable solution is plugin-owned OAuth credential and scoped-client routing rather than treating the inbound reaction issue or the external Drive-link workaround as a substitute.
