## Summary

Keep open. Current main still has Google Chat wired to service-account credentials and the `chat.bot` scope only, while the requested reactions, media upload, reaction typing, and email-alias DM paths need a new user OAuth credential/config surface and maintainer decisions around secrets, scopes, and fallback behavior.

## What This Changes

The issue requests optional Google Chat user OAuth support alongside service-account auth for reactions, attachment uploads, reaction typing, and proactive DMs by email alias.

## Best Possible Solution

Add plugin-owned, secret-aware Google Chat user OAuth support with scoped per-operation clients while preserving service-account webhook verification and bot-level sends, plus documented fallback behavior when OAuth is absent.

## Reproduction Assessment

Yes, at source level. Current main has no accepted `userOAuth` config, no refresh-token credential resolver, service-account-only credential validation, and runtime auth fixed to `chat.bot`, so the requested behavior cannot be enabled without code changes.

## Solution Assessment

Yes directionally, but not as an automated cleanup fix. The maintainable path is a Google Chat plugin-owned OAuth credential model with scoped clients after maintainer decisions on scopes, consent, fallback warnings, and defaults.
