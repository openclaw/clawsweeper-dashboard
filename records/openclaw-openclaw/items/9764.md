## Summary

Keep open. Current main still only accepts Google Chat service-account credentials and builds runtime auth with the `chat.bot` scope, while the requested reactions, upload, and email-alias DM paths need a new secret-bearing user OAuth design rather than cleanup closure.

## What This Changes

The issue requests optional Google Chat user OAuth support for operations that service-account auth cannot perform, including reactions, attachment uploads, and proactive DMs.

## Best Possible Solution

Add plugin-owned, secret-aware Google Chat user OAuth support with scoped per-operation clients, preserving service-account webhook/bot behavior and documenting fallback when OAuth is absent.

## Reproduction Assessment

Yes, at source level. Current main has no accepted `userOAuth` config shape, no refresh-token credential resolver, and runtime auth remains fixed to service-account `chat.bot`, so the requested behavior cannot be enabled without code changes.

## Solution Assessment

Yes directionally, but not as an automated cleanup fix. The maintainable path is a Google Chat plugin-owned OAuth credential model with scoped clients and documented service-account fallback after decisions on scopes, consent, and defaults.
