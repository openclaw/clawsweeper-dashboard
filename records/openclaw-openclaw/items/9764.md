## Label Justifications

- `P2`: Google Chat remains usable for basic service-account text/webhook paths, but missing user OAuth blocks several plugin operations.
- `impact:security`: The requested fix would add secret-bearing OAuth refresh-token credentials and broader Google Chat scopes.
- `impact:message-loss`: Attachment sends, reaction typing, and reaction operations can fail or be skipped for Google Chat users.
- `impact:auth-provider`: The issue is centered on provider authentication mode, SecretRef resolution, and scope routing for Google Chat.

## Summary

Keep open. Current main and v2026.5.22 still have Google Chat wired to service-account credentials and the single `chat.bot` scope, while the requested user OAuth credential model and scoped routing for reactions, media upload, reaction typing, and email-alias DM lookup are still missing.

## What This Changes

Add optional Google Chat user OAuth credentials and route user-auth-only operations such as reactions, attachment uploads, reaction typing, and email-alias DM lookup through those scoped credentials.

## Best Possible Solution

Keep this open and design a Google Chat plugin-owned optional user OAuth credential model with SecretRef support, scoped token selection for user-auth-only operations, service-account defaults preserved, explicit fallback behavior, docs, tests, and live Workspace proof.

## Reproduction Assessment

Yes, at source level. Current main has no user OAuth schema or resolver branch, and the outbound upload/reaction/direct-message helpers all share the hardcoded service-account `chat.bot` token path.

## Solution Assessment

Unclear as an implementation until maintainers choose the credential model and fallback policy. Optional user OAuth inside the Google Chat plugin is directionally right, but schema sensitivity, scopes, and upgrade behavior need deliberate review.
