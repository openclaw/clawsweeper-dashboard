## Summary

Keep open. Current main still has Google Chat wired for service-account credentials and the `chat.bot` scope only, while the reported outbound reaction, upload, and email-alias DM operations still route through that same auth path.

## What This Changes

Add optional Google Chat user OAuth credentials and scoped auth routing for reactions, media uploads, reaction typing, and proactive DMs while keeping service-account auth for webhook and bot operations.

## Best Possible Solution

Implement this in the bundled Google Chat plugin as a strict, secret-aware user OAuth credential path with per-operation scoped clients and documented fallback when OAuth is absent.

## Reproduction Assessment

Yes for source-level reproduction: current main has service-account-only config, credential resolution, and `chat.bot` auth, and the affected upload/reaction/DM helpers all use that account path. A live Google Chat 403 was not rerun in this read-only review.

## Solution Assessment

Unclear pending maintainer/security design review, but the requested direction is the right boundary: plugin-owned OAuth credentials and scoped auth routing are narrower than a core migration and are not covered by the inbound reaction issue.
