## Summary

Keep open. Current main still accepts only Google Chat service-account credentials and builds runtime auth with the `chat.bot` scope, while the requested reaction, attachment upload, and email-alias DM paths need user-auth scoped behavior and a new secret-bearing config surface. ([developers.google.com](https://developers.google.com/workspace/chat/create-reactions))

## What This Changes

Add optional Google Chat user OAuth credentials and scoped routing for reactions, attachment uploads, reaction typing, and email-alias proactive DMs while preserving service-account webhook and bot flows.

## Best Possible Solution

Add plugin-owned, secret-aware Google Chat user OAuth support with scoped per-operation clients while preserving service-account webhook verification and bot-level sends, plus documented fallback behavior when OAuth is absent.

## Reproduction Assessment

Yes, at source level. Current main has no accepted `userOAuth` config, no refresh-token credential resolver, service-account-only credential validation, and runtime auth fixed to `chat.bot`; I did not run a live Google Workspace repro.

## Solution Assessment

Yes directionally, but not as an automated cleanup fix. The maintainable path is a Google Chat plugin-owned OAuth credential model with scoped clients after maintainer decisions on scopes, consent, fallback warnings, and defaults.
