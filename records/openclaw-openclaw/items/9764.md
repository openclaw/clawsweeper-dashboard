## Summary

Keep open: current main and the latest provided release still expose Google Chat as a service-account based channel with a `chat.bot` token path, while this issue asks for a new user OAuth credential model and scoped routing for user-only Google Chat operations. The external Google Chat workaround and the closed inbound-reaction issue are useful context, but neither implements this outbound/user-auth feature in OpenClaw.

## What This Changes

Add optional Google Chat user OAuth support for reactions, attachment uploads, reaction typing, and proactive DM lookup while preserving service-account auth for bot/webhook flows.

## Best Possible Solution

Implement this in the Google Chat plugin as optional, secret-aware user OAuth with scoped user-auth clients for user-only operations, service-account auth retained for bot/webhook flows, explicit fallbacks, docs, tests, and live Workspace validation.

## Reproduction Assessment

Yes, at source level: current main has no user OAuth config/schema, resolves only service-account credentials, and routes reactions, attachment upload, and DM lookup through the same `chat.bot` token path. I did not establish a live Google Workspace reproduction in this read-only review.

## Solution Assessment

Yes as a product direction, but there is no implementation yet. The maintainable path is plugin-owned OAuth credential modeling and scoped-client routing, not treating the external Drive-link workaround or the closed inbound-reaction issue as substitutes.

## Risks / Open Questions

- No live Google Workspace smoke was run in this read-only review; final implementation needs real Workspace proof for reaction create/list, attachment upload, reaction typing fallback, and email-alias DM lookup.
