## Summary

Keep open. Current main still has service-account-only Google Chat config, credential resolution, and `chat.bot` runtime auth; upstream Google Chat docs confirm reactions and attachment upload require user authentication, so this needs plugin-owned OAuth design plus maintainer product/security review. The external Google Chat work in the discussion is useful prior art, but it is not shipped here, and the related inbound-reaction thread does not cover this outbound/user-auth gap.

## What This Changes

Request optional Google Chat user OAuth support so reactions, attachment uploads, and email-alias proactive DMs can use user-scoped Chat API credentials while service-account auth remains for webhook and bot operations.

## Best Possible Solution

Add plugin-owned, secret-aware Google Chat user OAuth support with scoped per-operation clients, preserving service-account webhook/bot behavior and documenting fallback when OAuth is absent.

## Reproduction Assessment

Yes, at source level. Current main has no accepted userOAuth config shape, no refresh-token credential resolver, and runtime auth remains fixed to service-account chat.bot, so the requested behavior cannot be enabled without code changes.

## Solution Assessment

Yes directionally, but not as an automated cleanup fix. The maintainable path is a Google Chat plugin-owned OAuth credential model with scoped clients and documented service-account fallback after decisions on scopes, consent, and defaults.
