## Summary

Keep open. Current main still lacks a Google Chat user OAuth credential model and continues to route reactions, attachment uploads, and direct-message lookup through the existing service-account-only auth path, while recent issue discussion explicitly confirms users still want this fixed in the bundled Google Chat plugin.

## What This Changes

Add optional Google Chat user OAuth support for reactions, media uploads, reaction-based typing, and proactive direct messages while retaining service-account auth for existing bot/webhook behavior.

## Best Possible Solution

Keep this in the bundled Google Chat plugin and add an operation-scoped, secret-aware user OAuth credential path for reactions, attachment uploads, reaction typing, and findDirectMessage/email-alias proactive DMs, while preserving service-account auth for webhook verification and existing bot sends.

## Reproduction Assessment

Yes for the source-level behavior: current main has strict service-account-only config and auth, and no user OAuth branch for the affected operations. A live Google Chat 403 repro was not rerun because the redacted environment/profile check found no Google credentials.

## Solution Assessment

Unclear until maintainer design review, but the issue's direction is right: this should be solved in the Google Chat plugin with scoped OAuth clients and graceful fallback, not by closing as ClawHub or by adding a broad core migration.
