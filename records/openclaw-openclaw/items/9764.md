## Summary

Keep this issue open: current main still lacks Google Chat user OAuth credentials and still routes reactions, uploads, and direct-message lookup through the service-account auth path.

## What This Changes

Request optional Google Chat user OAuth support for reactions, media uploads, reaction typing, and proactive direct messages while retaining service-account auth for existing bot/webhook behavior.

## Best Possible Solution

Implement this in the bundled Google Chat plugin with strict user OAuth credential support, secret-aware resolution, scoped OAuth clients for user-required operations, service-account auth retained for webhook/bot sends, tests, docs, and clear fallback warnings.

## Reproduction Assessment

Yes, at source level: current main exposes upload/reaction/direct-message paths, but Google Chat access tokens are always created from the service-account chat.bot scope. I did not run a live Google Workspace API scenario.

## Solution Assessment

No current fix is present. The requested direction is plausible, but the maintainable solution needs a Google Chat plugin-owned OAuth credential model and scoped routing rather than treating inbound reaction work in #73681 as a substitute.
