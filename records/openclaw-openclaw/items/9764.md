## Summary

Keep open: current main still has no Google Chat user OAuth config or scoped user-auth routing, so the requested reactions, attachment upload, reaction typing, and email-alias DM paths remain unimplemented.

## What This Changes

The issue requests optional Google Chat user OAuth credentials for reactions, media uploads, reaction typing, and proactive DMs while retaining service-account auth for bot/webhook flows.

## Best Possible Solution

Implement this in the Google Chat plugin as an optional, secret-aware user OAuth credential model with scoped user-auth clients for user-only operations, service-account auth retained for bot/webhook flows, explicit fallback behavior, docs, tests, and live Workspace validation.

## Reproduction Assessment

Yes, at source level: current main lacks a user OAuth config/schema and routes the named user-only operations through the same service-account `chat.bot` token path. I did not establish a live Google Workspace reproduction in this read-only review.

## Solution Assessment

Yes as a product direction, but no implementation exists yet. The maintainable solution is plugin-owned OAuth credential and scoped-client routing rather than treating the inbound reaction issue or the external Drive-link workaround as a substitute.
