## Summary

Keep open. Current main still lacks a Google Chat user OAuth credential surface and scoped OAuth routing, so reactions, media uploads, reaction typing, and email-alias proactive DMs remain tied to the service-account-only path.

## What This Changes

The issue requests optional Google Chat user OAuth support for reactions, media uploads, reaction typing, and proactive DMs while preserving service-account behavior.

## Best Possible Solution

Add plugin-owned, secret-aware Google Chat user OAuth support with scoped per-operation clients, service-account webhook/bot auth preserved, and documented fallback behavior when OAuth is absent.

## Reproduction Assessment

Yes, at source level. Current main accepts service-account config only, has no refresh-token resolver, hardcodes `chat.bot` auth, and routes the affected operations through that same account token path.

## Solution Assessment

Yes directionally, but not as an automated cleanup fix. The maintainable path is a Google Chat plugin-owned OAuth credential model with scoped clients after maintainer decisions on scopes, consent, fallback warnings, and defaults.
