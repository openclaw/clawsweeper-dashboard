## Summary

Keep this issue open: current main still lacks Google Chat user OAuth credentials and continues to route reactions, uploads, and direct-message lookup through the service-account auth path.

## What This Changes

Add optional Google Chat user OAuth support for reactions, media uploads, reaction typing, and proactive DMs alongside existing service-account auth.

## Best Possible Solution

Implement this in the bundled Google Chat plugin with strict user-OAuth config, secret-aware credential resolution, scoped OAuth clients for only the user-required operations, service-account auth retained for webhook and bot-level sends, docs, tests, and clear fallback warnings.

## Reproduction Assessment

Yes, at source level: current main exposes upload/reaction paths but `getGoogleChatAccessToken` always uses the service-account `chat.bot` scope. I did not run a live Google Workspace API scenario.

## Solution Assessment

No current fix is present. The requested direction is plausible, but the maintainable solution needs a Google Chat plugin-owned OAuth credential model and scoped routing rather than treating #73681's inbound reaction work as a substitute.
