## Summary

Keep open: current main still has only Google Chat service-account authentication and no user OAuth config, refresh-token credential path, or scoped user-auth routing for reactions, attachment upload, reaction typing, or email-alias DM lookup.

## What This Changes

Add optional Google Chat user OAuth support alongside service-account auth for user-scoped reactions, media uploads, and proactive direct-message lookup.

## Best Possible Solution

Implement this in the Google Chat plugin as optional, secret-aware user OAuth with scoped user-auth clients for user-only operations, service-account auth retained for bot/webhook flows, explicit fallbacks, docs, tests, and live Workspace validation.

## Reproduction Assessment

Yes, at source level: current main lacks a user OAuth config/schema and routes the named user-only operations through the same service-account chat.bot token path. I did not establish a live Google Workspace reproduction in this read-only review.

## Solution Assessment

Yes as a product direction, but no implementation exists yet. The maintainable solution is plugin-owned OAuth credential and scoped-client routing rather than treating the inbound-reaction issue or service-account transport fixes as substitutes.
