## Summary

Keep open. Current main still has Google Chat user-auth work missing: config, schema, credential resolution, and runtime auth are service-account-only, while reactions, media upload, and email-alias DM lookup need OAuth design and maintainer product/security decisions.

## What This Changes

The issue requests optional Google Chat user OAuth support for reactions, media uploads, and proactive DMs while retaining service-account auth for bot/webhook operations.

## Best Possible Solution

Add plugin-owned, secret-aware Google Chat user OAuth support for the operations that require it, keep service-account auth as the default for webhook and bot-level operations, and document explicit fallback behavior when OAuth is absent.

## Reproduction Assessment

Yes, at source level. Current main has no accepted userOAuth config shape, no refresh-token credential resolution, and runtime auth remains fixed to service-account chat.bot, so the requested behavior cannot be enabled without code changes.

## Solution Assessment

Yes directionally, but not as an automated cleanup fix. The maintainable path is a Google Chat plugin-owned OAuth credential model with scoped per-operation clients and documented service-account fallback, after maintainer decisions on scopes, consent, and defaults.
