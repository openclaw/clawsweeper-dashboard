## Label Justifications

- `P2`: This is a normal-priority Google Chat channel capability gap with clear user impact, limited plugin blast radius, and required maintainer design work.
- `impact:message-loss`: The missing auth path blocks or breaks outbound Google Chat reactions, media uploads, and some proactive DM delivery paths.
- `impact:auth-provider`: The request is about adding OAuth credential resolution and routing Google Chat operations through the correct auth scope.
- `impact:security`: The proposed config introduces client secrets and refresh tokens that need deliberate SecretRef, redaction, and token-refresh handling.

## Summary

Keep open. Current main still has Google Chat wired to service-account credentials and the `chat.bot` scope only, so outbound reactions, media upload, reaction typing, and email-alias DM lookup still lack the requested user OAuth path; the external Google Chat upload PR is useful prior art but is not shipped in OpenClaw.

## What This Changes

Add optional Google Chat user OAuth credentials and route reactions, attachment uploads, reaction typing, and email-alias DM lookup through scoped user tokens while preserving service-account webhook and bot auth.

## Best Possible Solution

Add plugin-owned, secret-aware Google Chat user OAuth support with scoped per-operation clients, service-account webhook/bot auth preserved, and documented fallback behavior when OAuth is absent.

## Reproduction Assessment

Yes, at source level. Current main accepts only service-account credential config, validates credentials as `service_account`, hardcodes `chat.bot`, and sends upload/reaction/direct-message calls through that token path; no live Workspace repro was run.

## Solution Assessment

Yes directionally, but not as an automated cleanup fix. The maintainable path is a Google Chat plugin-owned OAuth credential model with scoped clients after maintainer decisions on scopes, consent, fallback warnings, and defaults.
