## Summary

The requested capability remains absent on current main and in v2026.6.5: Google Chat still has one service-account `chat.bot` token path for reactions, uploads, and direct-message lookup, while Google’s API contract requires user authentication for reaction creation and email-alias direct-message lookup. No open or merged OpenClaw PR now owns this work, so the issue should remain open for a maintainer-approved OAuth security and product contract.

## What This Changes

Add optional user OAuth credentials and operation routing to the bundled Google Chat plugin for reactions, media uploads, and email-alias proactive direct messages while retaining service-account webhook and bot behavior.

## Best Possible Solution

Approve and implement a plugin-owned OAuth credential model using existing SecretRef patterns, choose minimum per-operation scopes, preserve service-account app authentication for webhook verification and bot-attributed sends, and return explicit capability warnings when user OAuth is absent.

## Reproduction Assessment

Yes, at source level with high confidence: current main has only a `chat.bot` service-account token path, while the implemented reaction and email-alias lookup operations require user authentication under Google’s API contract. No live credentialed Google Chat test was performed in this read-only review.

## Solution Assessment

The split-auth direction is the best architectural shape, but the issue’s exact scope list is incomplete and maintainers must define the permanent credential, consent, sender-attribution, and fallback contract before implementation.

## Risks / Open Questions

- A new refresh-token path would expand secret storage, redaction, consent, token-refresh, and revocation responsibilities and needs explicit security ownership.
- Using user authentication for message creation changes sender attribution and restricts message content compared with app authentication, so routing must remain operation-specific rather than replacing the existing bot path.
- The proposed `chat.messages` scope is not sufficient by itself for email-alias direct-message lookup; Google documents `chat.spaces` or `chat.spaces.readonly` for that user-authenticated operation.
