## Summary

Keep this issue open. Current main and the latest release remain service-account-only, and no canonical issue or implementation PR has replaced this still-missing Google Chat capability; the linked external upload work is useful prior art, while the closed inbound-reaction report addressed a different direction of reaction handling.

## What This Changes

Add optional Google Chat user OAuth for reactions, media uploads, and proactive direct messages while retaining service-account authentication for bot operations.

## Best Possible Solution

Keep service-account authentication for webhook verification and supported bot operations, add a secret-aware plugin-owned user OAuth client for only the operations that require it, and gate or clearly report unavailable capabilities when user credentials are absent.

## Reproduction Assessment

Yes, from source. Current main has only service-account credential resolution, a single `chat.bot` token path, and an explicit reaction-mode fallback because user OAuth is unsupported; this read-only review did not exercise a live Google Workspace OAuth tenant.

## Solution Assessment

Unclear. A plugin-owned user OAuth route is the correct architectural boundary, but maintainers must first approve the credential schema, exact operation set, consent/scopes, and absent-credential behavior.
