## Summary

Keep open. Current main and the latest shipped release still route Google Chat through service-account credentials and the `chat.bot` scope, while the requested reactions, media upload, reaction typing, and email-alias DM paths need a user-auth credential model and scoped routing.

## What This Changes

The issue requests optional Google Chat user OAuth support for reactions, media uploads, reaction typing, and proactive DM lookup while preserving service-account webhook/bot behavior.

## Best Possible Solution

Implement this inside the Google Chat plugin as a secret-aware user OAuth credential model with scoped clients for the affected operations, while keeping service-account webhook verification and default bot sends compatible.

## Reproduction Assessment

Yes, at source level. Current main accepts service-account config only, has no refresh-token resolver, hardcodes the `chat.bot` auth path, and routes the affected operations through that same token path.

## Solution Assessment

Yes directionally, but not as an automated cleanup fix. The maintainable path is a Google Chat plugin-owned OAuth model with maintainer decisions on scopes, consent, fallback warnings, and default behavior.
