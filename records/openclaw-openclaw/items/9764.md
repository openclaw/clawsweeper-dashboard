## Summary

Keep open: current main and v2026.6.6 still expose only service-account Google Chat credentials and a single `chat.bot` token path, while the requested reactions, media upload, and email-alias DM behavior needs a maintainer-approved delegated OAuth design.

## What This Changes

Add optional Google Chat user OAuth credentials so reactions, media uploads, and proactive DM lookup can use user-scoped Google Chat API permissions while service-account auth remains available.

## Best Possible Solution

Approve a secret-aware delegated OAuth credential model for the bundled Google Chat plugin, route only the user-scoped operations through that client, keep service-account auth for webhook/app flows, and document/test explicit fallback behavior when user OAuth is absent.

## Reproduction Assessment

Yes, from source rather than a live Google Workspace run: current main has no user OAuth config, resolves only service-account credentials, and creates all Google Chat API tokens with the `chat.bot` scope. Official Google Chat API docs confirm the reported reactions/media/email-alias paths need user-scoped authentication for the requested behavior.

## Solution Assessment

Unclear as an implementation plan: the direction is plausible, but the exact credential shape, SecretRef handling, scopes, consent flow, sender attribution, and fallback semantics need maintainer product/security approval before a fix PR should be queued.
