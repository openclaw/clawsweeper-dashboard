## Summary

Keep open: current main still has service-account-only Google Chat credential resolution and `chat.bot`-scoped access tokens, so the requested user OAuth path for reactions, media upload, reaction typing, and email-alias DM lookup remains missing.

## What This Changes

Add optional Google Chat user OAuth credentials and scoped routing for reactions, media uploads, reaction typing, and proactive DMs while retaining service-account webhook and bot flows.

## Best Possible Solution

Implement this as Google Chat plugin-owned OAuth support with strict secret-aware config, scoped user-auth clients for user-only operations, service-account auth retained for webhook and bot flows, explicit fallback warnings, docs, tests, and live Workspace validation.

## Reproduction Assessment

Yes, at source level: upload, reaction, and `users/<email>` DM lookup all share the service-account `chat.bot` token path, and the strict config schema has no user OAuth shape. I did not run a live Workspace scenario in this read-only pass.

## Solution Assessment

Yes as a product direction, but no implementation exists yet. The maintainable path is a plugin-owned OAuth credential model with scoped routing and fallback semantics, not treating inbound reaction work or external Drive-link prior art as a substitute.
