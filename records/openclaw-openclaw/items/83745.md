## Summary

Keep open: current `main` still has a documented and plugin-advertised `openclaw memory` surface, but `memory` is not a core/sub-CLI root and only works if plugin CLI registration actually installs the plugin command before Commander parses it. The supplied v2026.5.18 runtime output is consistent with the current dispatch shape, and the related https://github.com/openclaw/openclaw/issues/81077 tracks a different unknown-command help fallthrough case rather than this advertised `memory` root mismatch.

## What This Changes

The issue reports that documented `openclaw memory` commands are advertised by help/docs but rejected by runtime dispatch as an unknown command.

## Best Possible Solution

Make the advertised `openclaw memory` surface and dispatch contract agree: either route/register it reliably for the active memory plugin, or hide/adjust help and docs when the owning plugin cannot provide that CLI.

## Reproduction Assessment

Yes, we have a high-confidence source-level reproduction path: docs and plugin metadata advertise `openclaw memory`, while core dispatch has no built-in `memory` root and depends on plugin CLI registration before Commander parses the command. I did not run the live command in this read-only review.

## Solution Assessment

Unclear whether the best fix is to make `openclaw memory` a stable generic command for active memory plugins or to narrow the advertised docs/help to the plugin that owns it. Maintainers should settle that boundary before implementation.
