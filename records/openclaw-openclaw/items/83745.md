## Summary

Keep open: current main still has a documented `openclaw memory` surface, but the runnable command is owned by `memory-core` and the memory-slot policy can disable that plugin when Honcho owns the slot, matching the reporter's live rejection.

## What This Changes

The issue asks for the advertised `openclaw memory` CLI root to execute reliably, or for help/docs to stop advertising it when unavailable.

## Best Possible Solution

Make help, docs, and dispatch agree by either defining a generic active-memory CLI contract or only advertising `openclaw memory` when the owning plugin can actually register it.

## Reproduction Assessment

Yes. The reporter provided a live v2026.5.18 repro, and current source still shows the advertised command is memory-core-owned while slot policy can prevent memory-core from registering it.

## Solution Assessment

Unclear. The maintainable fix is to align the advertised and runnable surfaces, but maintainers need to choose whether that means a generic active-memory CLI or memory-core-specific docs/help.
