## Summary

Keep open: current main still advertises `openclaw memory` through docs/help and `memory-core` metadata, but dispatch only works when the owning plugin CLI is registered for that root; the reporter’s live v2026.5.18 Honcho-slot repro is consistent with the current plugin-owned command boundary.

## What This Changes

The issue reports that documented `openclaw memory` status/search workflows are advertised but rejected as unknown when a non-default memory plugin such as Honcho owns the memory slot.

## Best Possible Solution

Make the advertised memory CLI contract and dispatch behavior agree for active memory plugins, with a maintainer-owned choice between a reliable generic `openclaw memory` router or narrower docs/help tied to the plugin that actually owns the command.

## Reproduction Assessment

Yes, there is a high-confidence source-level reproduction path: docs and plugin metadata advertise `openclaw memory`, while core dispatch has no built-in `memory` root and depends on plugin CLI registration. The reporter also added a sanitized live v2026.5.18 repro showing the advertised root rejected as unknown.

## Solution Assessment

Unclear until maintainers decide the ownership boundary. The narrow maintainable outcome is to align help/docs with dispatch, but choosing a generic active-memory command versus plugin-specific CLI roots is a product/API decision.
