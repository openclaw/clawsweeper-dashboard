## Summary

Keep this open. Current main and the latest release still preserve account-scoped Discord DM allowlist precedence, denied DMs return before normal inbound routing, and the union-style fix was closed unmerged because the access-control semantics need design approval.

## What This Changes

The issue asks for allowlisted Discord direct messages in a multi-account setup to be logged, routed, and answered instead of silently dropped while outbound DMs and guild messages still work.

## Best Possible Solution

Define the intended Discord multi-account DM access contract, then update account/root allowlist resolution, denied-DM diagnostics, tests, docs, changelog, and live Discord proof to match that contract.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not for the full reported four-account live Discord setup; source inspection does show the narrowed path where an account allowFrom overrides root allowFrom and a denied DM returns null before routing or normal activity logging.

## Solution Assessment

Is this the best way to solve the issue? Unclear: unioning global and account allowFrom is plausible for the reported case, but it changes authorization semantics, so maintainers should decide inherit, override, union, or explicit DM-receiving-account behavior first.
