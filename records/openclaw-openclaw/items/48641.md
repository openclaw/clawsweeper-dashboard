## Summary

Keep this open. Current main and the latest shipped release still preserve account-scoped Discord DM allowlist precedence, the union-style fix PR was closed unmerged after the author noted the semantics need more design thought, and the exact four-account DM drop is not fully explained by a closed or shipped fix.

## What This Changes

The issue asks for Discord DMs from an allowlisted user to be logged, routed to the agent session, and answered instead of being silently dropped while outbound DMs and guild messages work.

## Best Possible Solution

Define the intended Discord multi-account DM allowlist contract, including inherit, override, union, or explicit DM-receiving account behavior, then align code, tests, docs, and changelog without unintended access widening.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No for the full reported four-account Discord setup; yes for the narrowed source path where account-level `allowFrom` overrides root `allowFrom` and a denied DM returns `null` before routing.

## Solution Assessment

Is this the best way to solve the issue? Unclear: unioning global and account `allowFrom` is a plausible narrow code change, but it changes access-control semantics, so the safer path is an explicit maintainer decision on the contract first.
