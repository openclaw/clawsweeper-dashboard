## Summary

Keep open: current main and v2026.6.10 still encode account-level Discord `allowFrom` precedence, and the direct fix PR was closed unmerged because the union-versus-override authorization semantics need maintainer/security direction.

## What This Changes

The issue asks for allowlisted inbound Discord DMs to be logged, routed to the agent, and answered instead of being silently dropped in a multi-account setup where outbound DMs and guild messages work.

## Best Possible Solution

Keep this as the canonical report until maintainers define and document Discord root/account `allowFrom` inheritance, then implement the chosen contract with focused tests and live Discord proof.

## Reproduction Assessment

Yes, but only at source level for the suspected account/root override shape: an account-level `allowFrom` hides root `allowFrom` and can deny a globally allowlisted sender. The reporter's exact live multi-account drop still needs live Discord reproduction because their example also lists the user under `accounts.default.allowFrom`.

## Solution Assessment

Unclear as currently proposed: merging global and account allowlists is plausible, but it changes authorization semantics. The best fix needs a maintainer/security decision on whether root `allowFrom` is a baseline or only an inherited default.
