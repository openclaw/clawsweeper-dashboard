## Summary

Keep open. Current main still gives Discord account-local `allowFrom` precedence over root `allowFrom`, denied DMs return before inbound activity/routing, and the linked union-style fix at https://github.com/openclaw/openclaw/pull/49365 was closed unmerged because the access semantics need design review.

## What This Changes

The issue reports allowlisted Discord direct messages being dropped in a multi-account setup while outbound DMs and guild channel messages continue to work.

## Best Possible Solution

Define the intended Discord multi-account DM access contract, then align allowlist resolution, diagnostics, tests, docs, and live proof without unintentionally widening account-scoped access.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No for the full live four-account Discord setup; yes at source level for the narrower path where account-local `allowFrom` overrides root `allowFrom` and a denied DM returns before routing or activity logging.

## Solution Assessment

Is this the best way to solve the issue? Unclear: unioning root and account `allowFrom` would address one narrowed failure mode, but it changes access-control semantics, so maintainers should choose override, union, inheritance, or explicit DM-receiving-account behavior first.
