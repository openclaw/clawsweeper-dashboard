## Summary

Keep open. The reporter's updated sanitized config gives a concrete source-reproducible regression path: current main and the shipped 2026.5.12 release validate Discord channel config against a strict schema that accepts `enabled` but not the legacy `allow` key, while docs and doctor migration still acknowledge `allow` on per-guild channel entries.

## What This Changes

The issue reports that upgrading from 2026.4.23 to 2026.5.12 rejects Discord per-guild per-channel config containing a legacy `allow` key.

## Best Possible Solution

Align the Discord config contract so documented or legacy per-channel `allow` entries either pass through a maintainer-approved compatibility path or fail with a clear doctor migration route, without silently drifting docs, schema, and doctor behavior apart.

## Reproduction Assessment

Yes, from source inspection. The reporter's concrete config uses `allow`, current main and v2026.5.12 use a strict Discord per-channel schema without `allow`, and plugin validation turns that mismatch into an `invalid config` issue.

## Solution Assessment

Unclear. The closed PR's normalization is a plausible narrow functional fix, but the repository policy says legacy repair belongs in doctor, so a maintainer should choose the compatibility boundary before another repair PR repeats that approach.
