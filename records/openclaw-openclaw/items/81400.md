## Summary

Keep open: current main still has no merged fix, and the linked closed PR points to a real Discord config-validation gap around legacy per-channel `allow` aliases. The remaining question is maintainer ownership of the compatibility boundary, not cleanup eligibility.

## What This Changes

Discord per-guild per-channel config that worked in 2026.4.23 is reported to fail gateway startup after upgrading to 2026.5.12 beta, especially legacy channel entries under `channels.discord.guilds.*.channels.*`.

## Best Possible Solution

Maintainers should choose the config-contract path, then either add a plugin-owned pre-validation compatibility hook for Discord channel `allow` aliases or keep doctor-only migration and update docs/diagnostics so users are not led to startup-blocking config.

## Reproduction Assessment

Yes, for the legacy `allow` variant: a current-main config with `channels.discord.guilds.*.channels.*.allow` reaches strict bundled Discord schema validation, which accepts `enabled` but not `allow`. The reporter's exact config is missing, so the proof is source-based rather than a live gateway repro.

## Solution Assessment

Unclear. The closed PR's core shim is functionally narrow, but it duplicates Discord doctor compatibility inside core validation; a plugin-owned validation compatibility seam would better match the documented owner boundary if maintainers want startup-time acceptance.
