## Summary

Keep open. Current main still starts the interactive `openclaw onboard` wizard without an upfront duration or timeline expectation, while the flow can proceed through provider auth, channels, workspace/session setup, search, skills, hooks, daemon install, and health checks; the linked implementation PR was closed unmerged.

## What This Changes

The issue asks OpenClaw to show an upfront duration or timeline expectation when `openclaw onboard` starts.

## Best Possible Solution

Add a concise early classic-wizard note and matching user docs that describe variable setup duration and major long-running phases before the main prompts begin.

## Reproduction Assessment

Yes. Source inspection gives a deterministic path from `openclaw onboard` to `runSetupWizard`, where startup and QuickStart copy lack duration guidance while later setup phases can run substantial work.

## Solution Assessment

Yes, with wording adjustment. A narrow UX/docs/test change is the maintainable fix; avoid treating the reporter's 40-minute observation as a universal ETA unless maintainers have measured timing data.

## Repair Work Prompt

Fix the onboarding expectation-setting gap for this issue. Add conservative upfront copy to the classic interactive `openclaw onboard` flow so users see before the main prompts that setup duration varies and provider auth, channel setup, daemon install, skills, optional dependencies, and network/provider steps can take a while. Do not promise a universal hard-coded 40-minute ETA unless maintainers have measured data; prefer variable-duration wording and a short phase list. Mirror the expectation in the user-facing CLI onboarding docs. Add or update focused tests for the wizard copy. Add a user-facing `CHANGELOG.md` entry because this is onboarding UX copy. Do not change unrelated onboarding behavior, plugin install policy, daemon semantics, provider defaults, or non-interactive config behavior.
