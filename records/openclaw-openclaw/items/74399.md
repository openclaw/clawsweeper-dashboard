## Summary

Keep open. Current main still routes classic `openclaw onboard` into a wizard that starts without any upfront duration expectation, the onboarding docs do not mention expected setup time, and the linked implementation PR #75622 was closed unmerged.

## What This Changes

Request an upfront ETA or timeline warning for `openclaw onboard` so new users know setup can take a long time.

## Best Possible Solution

Add a concise early classic-wizard note and matching onboarding docs that set a variable-duration expectation without promising one universal runtime.

## Reproduction Assessment

Yes. Source inspection gives a deterministic path from `openclaw onboard` to `runSetupWizard`, where startup and QuickStart copy lack duration guidance while later setup phases can run substantial work.

## Solution Assessment

Yes, with conservative wording. A small UX/docs/test/changelog change is the narrowest maintainable path; exact 40-minute language should be avoided unless maintainers have measured timing data.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/74399 with a conservative UX/docs update. Add a concise upfront note in the default classic `openclaw onboard` wizard before users commit to the longer setup path, explaining that first-time setup duration varies and can take a while because provider auth, channels, daemon/health checks, search, skills, plugins, hooks, and optional dependencies may run. Do not promise an exact universal 40-minute ETA unless maintainers have measured data; prefer variable-duration wording. Add focused wizard test coverage that proves the note is shown early. Mirror the expectation in the CLI onboarding docs/reference and add a user-facing CHANGELOG.md entry. Do not rewrite unrelated onboarding flow, do not change credential handling, and do not broaden this into modern Crestodian onboarding unless the existing code path makes that a small shared-copy reuse.
