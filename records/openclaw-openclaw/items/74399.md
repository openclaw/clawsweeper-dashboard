## Summary

Keep open. Current main still starts the classic `openclaw onboard` wizard without an upfront duration or long-running setup expectation, and the public onboarding docs list the setup phases without ETA guidance. The request is narrow, current, and suitable for a focused UX/docs repair rather than closure.

## What This Changes

Add an upfront ETA or long-running setup expectation to `openclaw onboard` so beginners know onboarding may take a while.

## Best Possible Solution

Add concise early expectation-setting copy to the classic CLI onboarding wizard before the main prompts, and mirror it in the public CLI onboarding docs. The copy should explain that provider auth, channel setup, daemon install, skills/plugins, optional dependencies, and network conditions can make setup take several minutes or longer, without promising a hard 40-minute duration.

## Reproduction Assessment

Yes. Source inspection gives a deterministic reproduction path: default `openclaw onboard` reaches `runSetupWizard`, whose first user-visible outputs have no duration note, while the onboarding docs list the setup steps without time guidance.

## Solution Assessment

Yes, with adjusted wording. A small wizard/docs/changelog change is the narrowest maintainable fix; the exact proposed 40-minute prompt should be softened unless maintainers have timing data.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/74399 by adding conservative upfront duration expectation copy to the default classic `openclaw onboard` wizard. The observable gap is that `runSetupWizard` starts with the header/intro/risk acknowledgement and no note that setup may take a while, even though later phases can include provider auth, channels, daemon install, web search, skills/plugins, optional dependencies, hooks, and finalization. Keep the implementation scoped to CLI onboarding UX/docs/changelog. Do not hard-code a 40-minute ETA unless maintainers provide timing data; prefer wording like setup can take several minutes or longer depending on selected providers/channels, daemon install, optional skills/plugins, and network conditions. Likely files: `src/wizard/setup.ts`, `src/wizard/setup.test.ts`, `docs/cli/onboard.md`, `docs/start/wizard.md`, `docs/reference/wizard.md`, and `CHANGELOG.md`. Add or update focused tests around the early note. Because this is user-facing onboarding copy, add a single-line changelog entry with appropriate non-bot contributor attribution. Do not change runtime setup behavior, provider/channel selection, daemon installation, plugin install mechanics, or modern Crestodian behavior unless the same copy can be mirrored without expanding scope.
