## Summary

Keep open. Current main still starts classic `openclaw onboard` without an upfront duration or timeline expectation, the public onboarding docs do not mention expected setup duration, and the linked implementation PR was closed unmerged.

## What This Changes

Add an upfront ETA/timeline expectation to `openclaw onboard`, warning that fresh setup can take a long time.

## Best Possible Solution

Add concise early classic-wizard copy and matching onboarding docs that set a variable-duration expectation without promising one universal runtime.

## Solution Assessment

Yes, with wording adjustment. A small UX/docs/test change is the narrowest maintainable solution; exact 40-minute language should be avoided unless maintainers have measured timing data.

Fix https://github.com/openclaw/openclaw/issues/74399 with a focused UX/docs PR. Add an early note in the classic CLI onboarding wizard (`src/wizard/setup.ts`) before the main setup prompts that sets expectations that fresh setup can take a while and may include provider auth, channel setup, daemon install, web search, skills, plugins, hooks, and health checks. Do not promise a universal 40-minute runtime unless backed by measured data; use variable-duration wording. Add/adjust focused wizard tests in `src/wizard/setup.test.ts` and update user-facing onboarding docs (`docs/start/wizard.md`, and `docs/cli/onboard.md` or `docs/reference/wizard.md` if the wording belongs there). Add a single-line `CHANGELOG.md` entry if the final change is user-facing. Do not change onboarding behavior, provider/channel setup logic, daemon install semantics, or plugin installation policy. Treat closed PR https://github.com/openclaw/openclaw/pull/75622 as context only, not as shipped code.
