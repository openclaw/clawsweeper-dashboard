## Summary

Keep open. Current main still starts the classic interactive onboarding wizard without an upfront duration or long-running setup expectation, the public onboarding docs list phases without timing guidance, and the linked implementation PR #75622 is closed unmerged rather than shipped.

## What This Changes

Request asks for `openclaw onboard` to show an upfront ETA or timeline warning so new users know setup may take a long time.

## Best Possible Solution

Add a concise early expectation-setting note to the classic CLI wizard and mirror it in the public onboarding docs. The wording should describe the phases that can take time and use conservative ranges such as several minutes or longer rather than promising an exact 40-minute duration without timing data.

## Reproduction Assessment

Yes. Source inspection gives a deterministic reproduction path: default `openclaw onboard` reaches the interactive wizard start sequence, which currently has no ETA or long-running setup note before prompts, and the docs omit equivalent guidance.

## Solution Assessment

Yes, with adjusted wording. A focused CLI/docs/test/changelog update is the narrowest maintainable fix; a fixed 40-minute claim should be avoided unless maintainers have platform-specific timing evidence.

## Repair Work Prompt

Implement a conservative repair for https://github.com/openclaw/openclaw/issues/74399. Add an early expectation-setting note to the classic `openclaw onboard` wizard in `src/wizard/setup.ts`, before users make setup choices, explaining that fresh local setup can take several minutes and may take longer when provider auth, channels, daemon/service install, skills, plugins, optional runtime dependencies, or network downloads are selected. Do not hard-code a guaranteed 40-minute ETA unless there is timing evidence. Add focused coverage in `src/wizard/setup.test.ts` proving the note appears early. Update the public onboarding docs, especially `docs/start/wizard.md` and likely `docs/cli/onboard.md` or `docs/reference/wizard.md`, and add a user-facing `CHANGELOG.md` entry with `Thanks @sanjarcode`. Treat closed PR #75622 as context only; do not blindly copy exact duration ranges from it. Keep the change scoped to classic CLI onboarding unless maintainers explicitly request Crestodian copy too.
