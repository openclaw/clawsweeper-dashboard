## Summary

Keep open. Current main still routes default interactive onboarding through the classic wizard without an upfront duration or ETA note, the relevant public onboarding docs list the phases without time guidance, and the related implementation PR is closed unmerged rather than a shipped fix.

## What This Changes

Request to show an upfront duration or ETA expectation during `openclaw onboard` because first-time setup can take a long time.

## Best Possible Solution

Add a concise early note in the classic CLI onboarding wizard that sets variable-duration expectations and names the major slow phases, mirror the same guidance in the public onboarding docs, cover it with focused wizard tests, and add a user-facing changelog entry.

## Reproduction Assessment

Yes. Source inspection gives a deterministic reproduction path: default interactive `openclaw onboard` reaches `runSetupWizard`, whose startup and QuickStart notes lack ETA or long-running setup copy while later setup phases can perform slow work.

## Solution Assessment

Yes, with adjusted wording. A narrow CLI UX, docs, test, and changelog update is the maintainable fix; an exact universal 40-minute promise would be less robust than phase-based variable-duration guidance.

## Repair Work Prompt

Fix openclaw/openclaw#74399 by adding conservative onboarding duration expectation copy for the default classic CLI wizard. Add an early `prompter.note` or equivalent in `src/wizard/setup.ts` before the main setup choices so first-time users see that model/auth, channels, daemon setup, skills, plugins, optional dependency installs, and network checks can take several minutes or longer. Do not hard-code an unsupported universal 40-minute guarantee; use variable-duration wording and, if helpful, short phase bullets. Add focused coverage in `src/wizard/setup.test.ts` that verifies the note appears for the classic interactive wizard. Mirror the expectation in the public onboarding docs, likely `docs/start/wizard.md`, `docs/cli/onboard.md`, and/or `docs/reference/wizard.md` / `docs/start/wizard-cli-reference.md` where the setup phases are listed. Add a user-facing `CHANGELOG.md` entry under the active version with `Thanks @sanjarcode`. Do not change non-interactive behavior, provider/channel setup semantics, daemon install behavior, or unrelated onboarding copy.
