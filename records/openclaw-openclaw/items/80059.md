## Summary

Keep this PR open for normal maintainer merge review. Current main still swallows async stdout/stderr broken-pipe stream errors on the gateway path, while this focused PR has supplied real Linux systemd proof and I found no blocking correctness or security finding.

## What This Changes

The PR changes console capture to call `process.exit(0)` for async stdout/stderr `EPIPE`/`EIO` stream errors and updates the focused regression test.

## Best Possible Solution

Land the narrow handler and regression-test change after the existing green checks are accepted, so a gateway process with broken output streams exits instead of spinning.

## Reproduction Assessment

Yes. Current main's async stream handler returns on `EPIPE`/`EIO` and gateway startup enables that handler; the PR body provides a concrete Linux systemd restart proof, though I did not rerun the live scenario in this read-only review.

## Solution Assessment

Yes. Exiting with code 0 from the async broken-pipe handler is the narrowest maintainable fix for a process that has lost stdout/stderr, while non-broken-pipe errors still throw.
