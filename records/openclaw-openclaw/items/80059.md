## Summary

Keep open: current main still swallows async stdout/stderr EPIPE/EIO in console capture while gateway startup enables that handler. The PR is narrow, has supplied real terminal proof, and I did not find a blocking correctness or security issue.

## What This Changes

The PR changes console capture to exit with code 0 when stdout or stderr emits EPIPE/EIO, and updates the focused regression test to expect that exit.

## Best Possible Solution

Land the narrow handler and regression-test change after normal maintainer merge handling so a gateway process with broken stdout/stderr exits instead of spinning.

## Reproduction Assessment

Yes. Current main's stream error listener returns on EPIPE/EIO while gateway startup enables console capture, and the PR body supplies Linux systemd strace plus after-fix terminal output for that path.

## Solution Assessment

Yes. Exiting with code 0 from the async stdout/stderr broken-pipe handler is the narrowest maintainable fix for a process that has lost its own output stream, while non-broken-pipe errors still throw.
