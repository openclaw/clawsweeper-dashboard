## Summary

Keep open: this is a focused, valid bug-fix PR. Current main still swallows async broken-pipe stream errors on the gateway logging path, the PR body supplies real Linux systemd proof, and I found no blocking correctness or security issues.

## What This Changes

The PR makes console capture exit with code 0 when stdout/stderr emit async EPIPE/EIO errors and updates the focused console-capture test to assert that behavior.

## Best Possible Solution

Land the narrow handler and regression test after normal CI so a gateway with closed stdout/stderr does not continue spinning after service restart.

## Reproduction Assessment

Yes. Current main's handler returns on EPIPE/EIO while gateway startup installs console capture, and the PR body supplies before/after real systemd proof for that path.

## Solution Assessment

Yes. Exiting from the async stdout/stderr broken-pipe handler is the narrowest maintainable fix for a process that has lost its output stream, while non-broken-pipe errors still throw.
