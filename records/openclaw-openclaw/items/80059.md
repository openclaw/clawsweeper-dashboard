## Summary

Keep open: current main still swallows async stdout/stderr EPIPE/EIO stream errors, and this focused PR changes the implicated logging path with no blocking code finding. The external PR still needs stronger after-fix real behavior proof before merge.

## What This Changes

The PR changes console capture to call `process.exit(0)` on stdout/stderr `EPIPE`/`EIO` stream errors and updates the async EPIPE test to expect exit.

## Best Possible Solution

Merge a narrow logging-layer fix that exits when process stdout/stderr are broken, with regression coverage and redacted real systemd proof showing the orphaned gateway exits.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: current `enableConsoleCapture()` returns on stdout/stderr EPIPE/EIO stream errors, while the PR body describes a Raspberry Pi systemd restart with repeated `write(2) = -1 EPIPE`. I did not run that hardware/systemd scenario here.

## Solution Assessment

Yes for the async stream-error path: changing the logging handler is the narrowest implicated fix, and it avoids moving policy into gateway watchers or systemd-specific code. It should merge only after after-fix real behavior proof demonstrates the service exits rather than spins.
