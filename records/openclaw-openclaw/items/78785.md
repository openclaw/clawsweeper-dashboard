## Summary

Keep this issue open: current main still reuses stored `claude-cli` session bindings after aborts and non-`session_expired` failover errors, which matches the reported wedged-session failure mode.

## What This Changes

The issue requests invalidating the stored Claude CLI resume session after abort or failover errors so the next turn starts a fresh CLI process.

## Best Possible Solution

Clear the stored `claude-cli` binding after aborts and reused-session failover/timeouts, using the existing session-store helper and regression coverage, without adding a config knob.

## Reproduction Assessment

Yes: source inspection plus the provided logs give a high-confidence repro path by seeding a stored `claude-cli` binding, making the reused turn throw `AbortError` or timeout `FailoverError`, and observing that current main leaves the binding intact. I did not execute that repro because this review is read-only.

## Solution Assessment

Yes, if scoped to reused `claude-cli` sessions: clearing through `clearCliSessionInStore` on abort and session-tainting failover is the narrow maintainable fix. A broader clear on every provider or every failover reason would be less safe.
