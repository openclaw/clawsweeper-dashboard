## Summary

Keep this issue open: current main still reuses stored `claude-cli` session bindings after aborts and non-`session_expired` failover errors. The linked fix PR was closed unmerged, and later live-runtime comments strengthen the report rather than superseding it.

## What This Changes

The issue requests invalidating the stored `claude-cli` resume session after `AbortError` or non-`session_expired` `FailoverError` so the next turn starts fresh instead of reusing a wedged session.

## Best Possible Solution

Add a narrow `claude-cli` clear-only path for abort and reused-session failover/timeout errors, with regression coverage that proves the stored binding is removed before the original error is rethrown.

## Reproduction Assessment

Yes, source inspection plus the reporter's live logs give a high-confidence reproduction path: seed a reusable `claude-cli` binding, make the reused turn throw `AbortError` or timeout `FailoverError`, and observe current main retaining the binding. I did not execute the live Claude CLI path during this read-only review.

## Solution Assessment

Yes, the proposed direction is the narrow maintainable fix if it clears only the affected `claude-cli` stored binding and preserves the existing no-retry/rethrow contract for abort and timeout paths. A broader watchdog configuration change would not address the poisoned resume-session state.

Include a changelog entry because the fix is user-visible reliability behavior.
