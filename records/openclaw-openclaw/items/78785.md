## Summary

Keep open. The report is credible: current main has reusable Claude CLI session storage and recovery for missing transcripts/session_expired, but I did not find a path that clears the stored claude-cli binding after AbortError or non-session_expired FailoverError, so the next turn can still reuse the same stored resume id.

## What This Changes

Invalidate the stored claude-cli resume session id after AbortError or FailoverError so the next turn starts fresh instead of reusing a wedged CLI session.

## Best Possible Solution

Add a claude-cli scoped cleanup path that clears only the affected provider's CLI session binding and legacy ids on AbortError or non-session_expired FailoverError from a reused session, with regression coverage proving the next turn does not resume the stale id.

## Reproduction Assessment

Yes, source-reproducible with high confidence, though I did not run a live Claude CLI session. A focused test can seed a claude-cli binding, make a reused CLI turn throw AbortError or FailoverError, and assert the stored binding is cleared before the next turn.

## Solution Assessment

Yes. Clearing the provider-owned session binding at the session-store boundary after reused-session AbortError/FailoverError is narrower than changing watchdog or fallback policy and fits the existing missing-transcript/session_expired recovery pattern.
