## Summary

Keep open: current main still only clears stored CLI sessions for `session_expired`, so `AbortError` and timeout/non-`session_expired` failover errors can leave a stored `claude-cli` binding eligible for reuse; the linked fix PR is closed unmerged, and the latest production logs strengthen the report.

## What This Changes

Request automatic invalidation of stored `claude-cli` resume session ids after `AbortError` or non-`session_expired` `FailoverError` so the next turn starts fresh.

## Best Possible Solution

Add a clear-only invalidation path for the active reused `claude-cli` binding on abort and non-`session_expired` failover, backed by focused regression coverage and a changelog entry.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence current-main path: seed a stored `claude-cli` binding with an existing transcript, make the reused turn throw `AbortError` or timeout `FailoverError`, and observe that current code rethrows without clearing the binding; I did not execute the live Claude CLI repro in this read-only review.

## Solution Assessment

Yes. Clearing through the existing `clearCliSessionInStore` helper before rethrow is the narrow maintainable fix, as long as the existing `session_expired` retry behavior stays intact and the change remains scoped to the active CLI binding.
