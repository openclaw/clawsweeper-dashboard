## Summary

Keep open: the production bug still exists on current main, but this branch is stale and incomplete because it does not carry the host media read callback through `thread-reply` and its tests target a file no longer present on main.

## What This Changes

The PR updates Discord `thread-reply` attachment parsing to accept `media`, then `path`, then `filePath`, passes `mediaLocalRoots` into the action call, and adds alias precedence tests under the old shared action test path.

## Best Possible Solution

Land a current-main Discord extension fix that normalizes `thread-reply` attachment aliases with `media > path > filePath` precedence, forwards `mediaAccess`, `mediaLocalRoots`, and `mediaReadFile` through the active adapter/parser/runtime boundary, moves coverage into `extensions/discord/src` tests, and adds the required changelog entry.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: on current main, a Discord `message` tool call with `action: "thread-reply"` and `path` or `filePath` reaches a handler that reads only `media`, matching the linked production report where the tool returns success without an attachment.

## Solution Assessment

No. The alias fallback is the right narrow direction, but this PR is not the best merge as-is because it omits `mediaReadFile`, leaves coverage in a removed test path, and needs the repo-required changelog entry.
