## Summary

Keep this PR open: it targets a real Discord thread-reply attachment bug, but the current branch is stale against the active extension layout and still misses the host media read callback needed for agent-scoped local files.

## What This Changes

The PR updates Discord `thread-reply` attachment parsing to accept `media`, then `path`, then `filePath`, passes `mediaLocalRoots` into the thread-reply action call, and adds alias precedence tests under the old shared action test file.

## Best Possible Solution

Land a current-main Discord extension fix that normalizes `thread-reply` attachment aliases, carries both `mediaLocalRoots` and `mediaReadFile` through the adapter/runtime boundary, moves coverage into active extension tests, and adds the required changelog entry.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: call Discord `message(action="thread-reply")` with `path` or `filePath`; current `main` reads only `media`, matching the paired production report where the tool returned success without an attachment.

## Solution Assessment

No. The alias fallback is the right narrow direction, but this branch is not the best merge as proposed because it omits `mediaReadFile` from the `threadReply` action options and places coverage in a removed test file.
