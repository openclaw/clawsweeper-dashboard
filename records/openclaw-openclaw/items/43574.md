## Summary

Keep this PR open. The reported Discord `thread-reply` attachment bug is still present on current `main`, and the PR direction is useful, but the branch is stale and incomplete: it adds the alias fallback while still missing the active `mediaReadFile` forwarding path, and its coverage lands in a test file that current `main` no longer contains.

## What This Changes

The PR updates Discord `thread-reply` attachment parsing to accept `media`, then `path`, then `filePath`, passes `mediaLocalRoots` into the thread-reply action call, and adds alias precedence tests under the old shared action test file.

## Best Possible Solution

Land a current-main Discord extension fix that normalizes `thread-reply` attachment aliases with `media > path > filePath` precedence, carries both `mediaLocalRoots` and `mediaReadFile` through the adapter/parser/runtime boundary, moves coverage into the active Discord extension tests, and adds the required user-facing changelog credit.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: call Discord `message(action="thread-reply")` with `path` or `filePath`; current `main` reads only `media`, so `mediaUrl` is undefined even though the paired bug report documents production replies returning success without attachments.

## Solution Assessment

No. The alias fallback is the right narrow direction, but this branch is not the best merge as proposed because it does not carry the `mediaReadFile` callback through the active adapter/runtime path and its tests target a removed shared test file.
