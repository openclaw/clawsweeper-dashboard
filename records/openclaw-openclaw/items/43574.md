## Summary

Keep this PR open: the Discord `thread-reply` attachment alias bug still exists on current `main`, and the branch takes the right narrow direction, but it is not merge-ready because it omits the `mediaReadFile` option hop and adds coverage to a legacy test file that is absent on current `main`.

## What This Changes

The PR updates Discord `thread-reply` parsing to accept `media`, then `path`, then `filePath`, passes `mediaLocalRoots` into the `threadReply` action call, and adds alias precedence tests under the old shared action test file.

## Best Possible Solution

Land a current-main Discord extension fix that normalizes `thread-reply` attachment aliases, forwards the full media action options through the handler/runtime boundary, covers active extension tests, and adds the required changelog entry.

## Reproduction Assessment

Yes. Source inspection on current `main` shows `message(action="thread-reply")` reads only `media`, while the Discord `send` path accepts `path` and `filePath`; a call with only `filePath` reaches `threadReply` with `mediaUrl` unset.

## Solution Assessment

No. The alias fallback is the right narrow fix direction, but this branch is not the best merge as-is because it does not forward `mediaReadFile` and its tests live in a removed shared core test file instead of active Discord extension coverage.
