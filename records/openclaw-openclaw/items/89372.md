## Summary

Keep open: current main and v2026.5.28 do not provide a typed Gateway updater for `subagents.allowAgents`, while the runtime and docs already treat that field as the policy gate for cross-agent subagent spawning. The request is valid core API design work, but the method shape and permission boundary need maintainer review before implementation.

## What This Changes

Requested a typed Gateway API for adding or patching `agents.list[].subagents.allowAgents` after clients create coordinator and worker agents.

## Best Possible Solution

Keep this open for a narrow maintainer-approved Gateway API that validates coordinator and target agent ids, dedupes and preserves existing allowlist entries, writes through the retrying config mutation path, and adds protocol/server tests plus docs without broadening subagent orchestration policy.

## Reproduction Assessment

Not applicable as a bug repro; this is a missing typed API surface. Source inspection confirms current Gateway method schemas and registration have no typed allowlist updater, while the runtime already consumes `subagents.allowAgents`.

## Solution Assessment

Unclear pending maintainer design. A typed Gateway method looks like the right owner boundary compared with raw `config.patch`, but method semantics and permission scope should be approved before code is queued.
