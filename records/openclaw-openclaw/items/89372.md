## Summary

Keep open: current main and v2026.5.28 still expose only the basic typed agent Gateway methods plus raw admin config mutation, while the subagent allowlist is already a real runtime policy gate. This is valid core Gateway API design work, but the public method shape and security scope need maintainer review before implementation.

## What This Changes

The issue requests a typed Gateway API for updating `agents.list[].subagents.allowAgents` without forcing clients to use raw `config.patch`.

## Best Possible Solution

Add an additive, narrowly scoped `operator.admin` Gateway method under the agent policy surface that validates requester and target agent ids, dedupes and merges allowlist entries through the existing config mutation path, and covers protocol/server/docs behavior.
