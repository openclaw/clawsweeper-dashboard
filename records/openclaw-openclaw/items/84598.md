## Summary

Keep open: current main and v2026.5.18 still count all normalized memory-wiki claim rows while omitting `id` on ID-less rows, so the reported claim-count ambiguity is real. Source inspection also suggests these are ordinary ID-less `WikiClaim` entries, not a separate cache-row type, so the fix should clarify claim rows versus stable claim IDs without breaking existing `claimCount` consumers.

## What This Changes

The issue asks memory-wiki compiled/status metrics to distinguish total claim rows from stable claim-ID rows.

## Best Possible Solution

Keep `claimCount` compatible as total claim rows, add or derive explicit claim-ID and ID-less claim counts, and update labels/docs to say `claim rows` versus `claim IDs` where users see the metric.

## Reproduction Assessment

Yes, at source level. Any vault page with one claim that has an `id` and one text-only claim will produce a `claimCount` of 2 while only one compiled row has a stable claim ID.

## Solution Assessment

Yes, with a compatibility-preserving adjustment. Counting only ID-bearing rows in the existing `claimCount` would be riskier; the maintainable fix is to keep total claim rows and expose/label stable claim-ID counts separately.
