## Summary

Keep open: current main and the latest release still count all structured memory-wiki claim rows as `claimCount` and render that number as plain claims, while claim IDs remain optional, so the reported ambiguity is still present and needs a compatible maintainer decision.

## What This Changes

Clarify memory-wiki digest/status counts so stable claim-ID counts are not conflated with cache or claim rows.

## Best Possible Solution

Keep `claimCount` compatible as the total structured claim-row count, then add explicit ID-bearing and idless/cache-row metrics or update prompt/status wording and docs so stable claim IDs are not conflated with rows.

## Reproduction Assessment

Yes. Source inspection is enough: current main counts all `page.claims` rows while `claims.jsonl` conditionally omits `id`, so a vault with idless claims reproduces the reported count mismatch.

## Solution Assessment

Unclear as originally framed. Recounting `claimCount` as only ID-bearing claims could be incompatible, so the safer solution is additive metrics or wording/docs that preserve the shipped field semantics.
