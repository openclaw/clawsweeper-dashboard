## Summary

Keep open: current main and v2026.6.8 still compute memory-wiki `claimCount` from all structured claim rows while `claims.jsonl` only includes `id` when a claim has one. The merged palace UI fix clarified one visible label as claim rows, but the digest and prompt contract remain unresolved, and the related manifest PR is still open and blocked.

## What This Changes

The issue asks memory-wiki to distinguish claim rows from stable claim IDs in compiled digest, status, and prompt-facing counts.

## Best Possible Solution

Choose a compatible digest contract: keep `claimCount` as row count and add explicit ID-bearing/idless row metrics plus prompt/docs wording, or approve a breaking semantic change with migration and docs.

## Reproduction Assessment

Yes by source inspection. A vault with structured claims lacking `id` reproduces the mismatch because current main counts `page.claims.length` while `claims.jsonl` omits `id` for those rows.

## Solution Assessment

Unclear as a direct semantic change. Reusing `claimCount` for only ID-bearing claims could break existing digest consumers, so additive metrics and clearer wording are the safer maintainable path unless maintainers approve a breaking contract.
