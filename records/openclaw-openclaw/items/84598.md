## Summary

Keep open: current main still counts all structured memory-wiki claim rows as `claimCount` while `claims.jsonl` can emit rows without an `id`, so the ambiguity reported here remains. The root cause is source-reproducible, but changing the shipped `claimCount` meaning is compatibility-sensitive and needs a maintainer contract decision.

## What This Changes

The issue asks memory-wiki to stop presenting total structured claim rows as stable claim IDs by counting ID-bearing claims separately or relabeling the metric.

## Best Possible Solution

Keep the shipped row-count contract compatible, then add explicit ID-bearing and idless row counts or relabel user-facing copy to `claim rows` so stable claim IDs are not conflated with cache rows.

## Reproduction Assessment

Yes. Source inspection shows `claimCount` is computed from all structured claim rows while `claims.jsonl` conditionally omits `id`, so a vault with idless claims will reproduce the reported mismatch.

## Solution Assessment

Unclear as a direct semantic change. The safer path is an additive or wording-only contract update unless maintainers explicitly decide to redefine `claimCount`.
