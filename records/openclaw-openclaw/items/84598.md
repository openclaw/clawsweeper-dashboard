## Summary

Keep open: current main and the latest release still compute memory-wiki `claimCount` from all structured claim rows while `claims.jsonl` only emits `id` for claims that have one. The related manifest PR may cover adjacent cache/claim validation work, but it is open and blocked, so it is not a merged resolution.

## What This Changes

The issue asks memory-wiki to distinguish total claim rows from rows with stable claim IDs in digest/status/prompt-facing metrics.

## Best Possible Solution

Preserve `claimCount` as the compatible row count unless maintainers decide otherwise, then add explicit ID-bearing and idless row metrics plus prompt/docs wording that distinguishes claim rows from stable claim IDs.

## Reproduction Assessment

Yes. Source inspection of current main shows `claimCount` is computed from `page.claims.length` while `claims.jsonl` only writes `id` when `claim.id` exists, so a vault containing idless claims reproduces the row-vs-ID mismatch.

## Solution Assessment

Unclear as a direct semantic change. Reusing `claimCount` for only ID-bearing claims could break existing digest consumers; additive metrics and clearer wording are the safer maintainable path unless maintainers approve a contract change.
