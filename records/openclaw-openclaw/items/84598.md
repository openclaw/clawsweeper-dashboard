## Summary

Keep open: current main and the latest release still count all structured memory-wiki claim rows as `claimCount` while `claims.jsonl` omits `id` for idless claims, so the ambiguity remains. The likely fix needs a maintainer contract choice because changing `claimCount` from row count to ID-bearing count could break digest consumers.

## What This Changes

The issue asks memory-wiki to distinguish stable claim IDs from idless claim/cache rows in digest and prompt/status counts.

## Best Possible Solution

Keep `claimCount` compatible as the total structured claim-row count, then add explicit ID-bearing/idless metrics or relabel prompt/status copy so users can distinguish row totals from stable claim IDs.

## Reproduction Assessment

Yes. Source inspection shows current main computes `claimCount` from all `page.claims` while `claims.jsonl` omits `id` when a claim lacks one, so any vault with idless claims will reproduce the mismatch.

## Solution Assessment

Unclear as proposed. Counting only IDs would be a compatibility-sensitive semantic change; the safer maintainable direction is additive ID/row metrics plus clearer wording.
