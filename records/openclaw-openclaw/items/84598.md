## Summary

Keep open: current main still conflates structured claim rows with stable claim IDs in the memory-wiki digest/prompt wording, but the shipped `claimCount` field is also an existing machine-facing contract, so the fix should be an additive clarification rather than an automatic semantic change.

## What This Changes

The issue asks memory-wiki to stop presenting idless compiled claim rows as if they were stable claim IDs.

## Best Possible Solution

Keep `claimCount` compatible as the total structured claim row count, then add explicit ID-bearing/idless counts or relabel prompt/status copy so stable claim IDs are not conflated with rows.

## Reproduction Assessment

Yes. Source inspection shows `claimCount` is computed from all structured claim rows while `claims.jsonl` conditionally omits `id`, so a vault with idless claims will reproduce the reported count mismatch.

## Solution Assessment

Unclear as proposed. Counting only IDs may be incompatible with the shipped digest contract; the safer maintainable solution is additive ID/row metrics plus clearer wording.
