## Summary

Keep open: current main and v2026.6.8 still count every structured memory-wiki claim row in `claimCount` while `claims.jsonl` and lookup behavior allow rows without stable claim IDs. The merged palace UI change fixed one visible label, but the machine-facing digest and prompt contract still need a maintainer compatibility decision.

## What This Changes

The issue asks memory-wiki claim counts to distinguish stable claim IDs from idless claim/cache rows.

## Best Possible Solution

Choose a compatible digest contract: keep `claimCount` as row count and add explicit ID-bearing/idless metrics plus prompt/docs wording, or approve a breaking semantic change with migration and docs.

## Reproduction Assessment

Yes by source inspection. A wiki page with a structured claim lacking `id` is normalized as a claim, counted by `page.claims.length`, and emitted to `claims.jsonl` without an `id`.

## Solution Assessment

Unclear as a direct semantic change. Reusing `claimCount` for only ID-bearing claims could break stable digest consumers, so additive metrics and clearer wording are the safer maintainable path unless maintainers approve a breaking contract.
