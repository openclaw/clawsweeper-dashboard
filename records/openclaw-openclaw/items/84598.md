## Summary

Keep open: current main and the latest release still compute memory-wiki `claimCount` from all structured claim rows while `claims.jsonl` only emits `id` when a claim has one. The merged palace UI fix clarified one visible label as claim rows, but the digest and prompt contract still need a maintainer decision before changing or extending the metric.

## What This Changes

The issue asks memory-wiki to stop presenting total structured claim-row counts as if every row has a stable claim ID.

## Best Possible Solution

Preserve `claimCount` as the compatible row count, then add explicit ID-bearing and idless row metrics plus prompt/docs wording that distinguishes claim rows from stable claim IDs.

## Reproduction Assessment

Yes. Source inspection of current main shows `claimCount` is computed from `page.claims.length` while `claims.jsonl` only writes `id` when `claim.id` exists; a vault containing idless claims reproduces the row-vs-ID mismatch.

## Solution Assessment

Unclear as a direct semantic change. Reusing `claimCount` for only ID-bearing claims could break existing digest consumers; additive metrics and clearer wording are the safer maintainable path.
