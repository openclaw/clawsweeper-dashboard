## Label Justifications

## Summary

Current main and the v2026.6.1 release already handle the reported macOS LaunchAgent restart misroute: the restart path detects the current launchd service before stale-port cleanup, the detector trusts OpenClaw service markers plus the configured launchd label when `XPC_SERVICE_NAME` is nonmatching, and regression tests cover the `XPC_SERVICE_NAME="0"` handoff case.

## What This Changes

The issue requests robust macOS LaunchAgent self-restart detection when an exec-spawned child sees `XPC_SERVICE_NAME="0"` instead of the gateway launchd label.

## Best Possible Solution

Keep the shipped marker-based launchd detection; if a user still reproduces this, refresh the LaunchAgent service environment and reopen with the child process values for `OPENCLAW_SERVICE_MARKER`, `OPENCLAW_SERVICE_KIND`, and `OPENCLAW_LAUNCHD_LABEL`.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not as a failing current-main path; instead, current source and tests model the reported `XPC_SERVICE_NAME="0"` environment and prove the fixed handoff behavior.

## Solution Assessment

Is this the best way to solve the issue? Yes; the shipped fix keeps the decision in the launchd current-service boundary and trusts OpenClaw-owned service metadata rather than treating arbitrary nonmatching XPC labels as safe.
