## Summary

Keep open for maintainer review. Current main still does not implement `sessions_broadcast`, and the linked feature request was consolidated into this PR, but the patch is not merge-ready because it adds a core cross-session injection tool with an invalid Gateway provenance value, current-main dependency drift, and missing restricted-surface policy coverage.

## What This Changes

The PR adds a core `sessions_broadcast` agent tool, registers it in the catalog and default tool factory, and adds unit tests for broadcasting system-event messages to active sessions with optional channel filtering.

## Best Possible Solution

If maintainers want this capability in core, land it as a policy-aware sibling of `sessions_send`: use the current `typebox` dependency, use an existing valid provenance kind or intentionally extend the Gateway protocol and prompt handling, add matching subagent/Gateway HTTP/realtime/tool-mutation restrictions and regression tests, and document the trust boundary. If the desired product is operator notification rather than agent-to-agent context injection, fold the work into the admin-scoped design tracked in #53228.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: the PR sends `inputProvenance.kind: "broadcast"` through the validated Gateway `agent` method while current main only accepts three provenance kinds, and the policy gap is visible by comparing the new default tool registration with the existing `sessions_send` deny surfaces.

## Solution Assessment

No. The branch is a useful implementation sketch and it addressed the first round of automated feedback, but it is not the narrowest maintainable solution until the dependency, Gateway provenance, cross-session trust, restricted-surface policy, and docs/changelog contracts are resolved.
