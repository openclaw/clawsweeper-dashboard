## Summary

Keep this PR open but do not merge it as-is: the latest head still introduces a core cross-session broadcast tool that conflicts with current provenance validation, bypasses existing session-tool safety boundaries, and lacks real behavior proof.

## What This Changes

The PR adds a new core `sessions_broadcast` agent tool, registers it in the sessions tool catalog/default tool set, and adds mocked tests for broadcasting messages to active sessions.

## Best Possible Solution

Keep this as an implementation candidate only after it uses supported provenance semantics or a documented protocol extension, preserves `sessions_send` safety boundaries, follows current lane/dependency contracts, and includes redacted live proof plus maintainer product/security sign-off.

## Reproduction Assessment

Yes for the blocking review findings by source inspection: the latest PR head still sends an invalid provenance kind, can target thread sessions, exposes broadcast without a subagent deny rule, uses a shared nested lane, and imports an undeclared TypeBox package. I did not run a live broadcast because this review is read-only.

## Solution Assessment

No, not as currently patched. The maintainable path is to reuse or formally extend the gateway provenance contract, mirror existing `sessions_send` safety rules where applicable, and require live proof after source-level blockers are fixed.
