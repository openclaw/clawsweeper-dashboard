## Summary

Keep this PR open as the canonical feature candidate, but do not merge it as currently patched: current main still lacks `sessions_broadcast`, and the branch has concrete gateway-contract, subagent-policy, thread-targeting, lane-scoping, and stale dependency issues.

## What This Changes

The PR adds a core `sessions_broadcast` agent tool, registers it in the sessions tool catalog/default tool set, and adds mocked tests for broadcasting messages to active sessions.

## Best Possible Solution

Keep this PR as the implementation candidate only after it uses supported provenance semantics, preserves subagent and thread-session restrictions, follows current lane/dependency contracts, and demonstrates a redacted live broadcast between real sessions.

## Reproduction Assessment

Yes for the blocking review findings by source inspection: the PR sends a provenance kind rejected by the current gateway schema, bypasses current subagent/thread targeting boundaries, and uses stale lane/dependency contracts. I did not run a live broadcast in this read-only review.

## Solution Assessment

No, not as currently patched. The maintainable path is to reuse or formally extend the gateway provenance contract, mirror existing `sessions_send` safety rules where applicable, and add live proof after the source-level blockers are fixed.

Overall confidence: 0.96

Full review comments:

- **[P1] Use a valid provenance kind for broadcast sends:** `src/agents/tools/sessions-broadcast-tool.ts:142`
  - body: `sessions_broadcast` sends `inputProvenance.kind: "broadcast"`, but the gateway `agent` schema only accepts the existing provenance kinds, so each target send can be rejected as invalid params before the target session starts. Use `inter_session` with `sourceTool: "sessions_broadcast"` or add a documented protocol extension.
  - confidence: 0.94
- **[P1] Deny sessions_broadcast for subagent sessions:** `src/agents/openclaw-tools.ts:189`
  - body: Registering the tool in the default OpenClaw tool set exposes a new cross-session injection path to subagents, while the current always-denied subagent tools only include `sessions_send`. Add `sessions_broadcast` to that policy and cover it with a regression test.
  - confidence: 0.9
- **[P1] Skip thread-scoped sessions before broadcasting:** `src/agents/tools/sessions-broadcast-tool.ts:118`
  - body: The target loop broadcasts to every visible session key, but current `sessions_send` rejects thread-scoped chat sessions so inter-agent coordination cannot inject into active human-facing Slack or Discord threads. Apply the same thread-session guard before adding a broadcast target.
  - confidence: 0.88
- **[P2] Scope each broadcast lane to the target session:** `src/agents/tools/sessions-broadcast-tool.ts:140`
  - body: The PR uses the shared `AGENT_LANE_NESTED` lane for all targets, but current session sends use `resolveNestedAgentLaneForSession(resolvedKey)` so one target's nested run does not block another session. Use the same per-target lane helper for each broadcast call.
  - confidence: 0.84
- **[P2] Import TypeBox from the current package:** `src/agents/tools/sessions-broadcast-tool.ts:2`
  - body: Current main has migrated to the `typebox` package and no longer declares `@sinclair/typebox`; this new file's import will fail after rebasing or merging onto current main. Match the surrounding tools and import `Type` from `typebox`.
  - confidence: 0.87
