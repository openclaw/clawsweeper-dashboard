## Summary

Keep this PR open for normal maintainer review. The implementation shape is plausible and I found no blocking correctness bug, but it adds a new public plugin hook and currently leaves the hook semantics undocumented.

## What This Changes

The PR adds a new read-only `inbound_observed` typed plugin hook and emits it from Telegram’s `requireMention` skip path, with hook mapping and focused tests.

## Best Possible Solution

Land a documented, read-only observational hook after maintainer confirmation of the plugin privacy contract, keeping Telegram emission narrow and preserving `message_received` semantics.

## Reproduction Assessment

Yes, by source inspection: current main reaches the Telegram mention-skip return without a typed plugin hook, while the branch inserts `inbound_observed` before that return. I did not run tests because this review was required to keep the checkout read-only.

## Solution Assessment

Mostly yes: a dedicated read-only hook is safer than moving `message_received` earlier and changing existing hook semantics. The remaining cleanup is to document the new public hook and get maintainer sign-off on its privacy/API boundary.
