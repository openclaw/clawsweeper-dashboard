## Summary

Keep open: the PR adds a useful hook, but the Telegram emission is not merge-ready because it drops forum-topic origin targets for the new observed event, and the skipped-message visibility contract still needs maintainer privacy/API approval.

## What This Changes

Adds a public `inbound_observed` plugin hook, maps its payload, emits it for Telegram `requireMention`-skipped group messages, and updates docs plus focused tests.

## Best Possible Solution

Preserve forum-topic origin targets in `inbound_observed` by passing the existing Telegram inbound origin target into the emitted canonical context, add a forum-topic skip regression test, and get maintainer approval for the skipped-message plugin visibility contract before merge.

## Reproduction Assessment

Yes. Source inspection shows current main has no typed plugin hook on the requireMention skip path, and the PR's new helper recomputes the observed event target in a way that loses forum-topic origin information.

## Solution Assessment

No, not yet. A dedicated read-only hook is a maintainable direction, but this implementation needs to preserve Telegram forum-topic targets and receive maintainer approval for the new public privacy boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve Telegram topic targets in observed events:** `extensions/telegram/src/bot-message-context.body.ts:82`
  - body: This helper recomputes the observed event target as the parent chat even though `resolveTelegramInboundBody` already receives `originatingTo` from `buildTelegramInboundOriginTarget`, which encodes forum topics. For skipped forum-topic messages, archive plugins will see `conversationId`/`metadata.originatingTo` as `telegram:<chat>` instead of `telegram:<chat>:topic:<id>`, collapsing separate topics. Pass the existing origin target into the helper and cover a forum-topic skip.
  - confidence: 0.86
