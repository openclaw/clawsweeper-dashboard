## Summary

Keep open: current main and the latest release still leave the standalone Skills tab default-agent-only, while the report provides fresh multi-agent evidence after the older related issue was closed for inactivity. The remaining fix crosses a UI/API scope decision because the same top-level page also has install/update flows that still target default/global state.

## What This Changes

Add agent-scoped viewing to the Web UI Skills page so workspace skills for non-default agents are visible.

## Best Possible Solution

Decide the intended scope for the top-level Skills tab, then make status and mutations consistently agent-scoped or keep per-agent management in Agents > Skills with clearer guidance on the standalone tab.

## Reproduction Assessment

Yes, at source level: the standalone Skills tab calls `skills.status` with `{}`, and the server falls back omitted `agentId` to the default agent. The latest release shows the same top-level call, but I did not run a live Windows multi-agent UI session.

## Solution Assessment

Unclear as a final product direction. Passing `agentId` into the status call is the narrow read-only fix, but the page also exposes default-scoped install/update flows, so maintainers should choose the page scope before implementation.
