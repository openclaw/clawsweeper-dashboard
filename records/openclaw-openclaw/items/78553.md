## Summary

Keep open: current main still leaves the standalone Web UI Skills tab scoped to the default agent, while the backend and Agents panel already have agent-aware skill status paths. The requested selector is plausible, but the page also has default-scoped install/update mutations, so this needs a maintainer UX/API scope decision before implementation.

## What This Changes

The issue requests an agent selector on the Web UI Skills page so users can view workspace skills for non-default agents.

## Best Possible Solution

Decide whether the standalone Skills tab should become fully agent-scoped for reads and mutations, or keep per-agent skill management inside Agents > Skills and label the standalone tab as default/global scope.

## Reproduction Assessment

Yes, at source level: the standalone Skills tab calls `skills.status` with `{}`, and the gateway falls back omitted `agentId` to the default agent. The latest release shows the same top-level controller call.

## Solution Assessment

Unclear as a final product direction. Passing `agentId` only into status is the narrow read fix, but the same page exposes default-scoped mutations, so the maintainable solution needs a full page-scope decision first.
