## Summary

Keep open: current main still leaves the standalone Web UI Skills tab default-agent scoped, while the backend and the Agents panel already support agent-scoped skill status. The requested UI change is valid, but the same standalone page also contains default-scoped install/update/API-key flows, so this needs a maintainer UX/API decision before implementation.

## What This Changes

Request an agent selector on the Web UI Skills page and pass the selected `agentId` to `skills.status` so non-default workspace skills are visible.

## Best Possible Solution

Decide the standalone Skills tab scope, then implement consistent agent selection for status and either agent-scoped mutations or read-only non-default views with focused controller/view coverage.

## Reproduction Assessment

Yes, at source level: the standalone Skills tab calls `skills.status` with `{}`, and the server defaults an omitted `agentId` to the default agent. I did not run a live Windows Control UI session, but current source directly matches the reported behavior.

## Solution Assessment

Unclear as a final product direction: passing `agentId` is the narrow read-only status fix, but the same page still has install/update/API-key flows whose RPC contracts target default/global state. A maintainable fix should settle that UI/API scope first.
