## Summary

Keep open: current main still leaves the standalone Web UI Skills tab default-agent-only, while the backend and Agents panel already have agent-aware skill status paths. No canonical duplicate or merged fix was found, and the same top-level page has default-scoped install/update mutations, so this needs a maintainer UX/API scope decision before implementation.

## What This Changes

Request to add agent selection to the Web UI Skills page so workspace skills can be viewed for any configured agent instead of only the default agent.

## Best Possible Solution

Decide whether the standalone Skills tab should become consistently agent-scoped for both reads and mutations, or keep agent-specific management in Agents > Skills and clarify the standalone tab’s default/global scope.

## Reproduction Assessment

Yes, at source level: the standalone Skills tab calls `skills.status` with `{}`, and the gateway falls back omitted `agentId` to the default agent. The latest release has the same top-level controller call.

## Solution Assessment

Unclear as a final product direction. Passing `agentId` only into the status call is the narrow read fix, but the page also exposes default-scoped mutations, so maintainers should choose the full page scope first.
