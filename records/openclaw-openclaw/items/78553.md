## Summary

Keep open: current main confirms the reported standalone Skills tab still fetches `skills.status` without an `agentId`, even though the backend and the Agents panel already have agent-scoped skills support.

## What This Changes

The issue requests multi-agent support on the Web UI Skills tab so users can inspect workspace skills for any configured agent instead of only the default agent.

## Best Possible Solution

Decide whether the standalone Skills tab should become agent-scoped; if yes, make status and mutating skill actions consistently target the selected agent or mark non-default views read-only while preserving the existing Agents-panel workflow.

## Reproduction Assessment

Yes, at source level: the standalone Skills tab calls `skills.status` with `{}`, and the server defaults omitted `agentId` to the default agent. I did not run a live Windows Control UI session, but the code path directly matches the report.

## Solution Assessment

Unclear as a final product direction: passing `agentId` to `skills.status` is the narrow read-only fix, but the full page also has install/update flows that are still default-agent scoped. A maintainable solution should resolve that UI/API scope before implementation.
