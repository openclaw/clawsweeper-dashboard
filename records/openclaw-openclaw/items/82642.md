## Summary

Keep open: source inspection supports the reported iMessage slash-command acknowledgement gap, but this branch classifies all authorized iMessage turns as text commands and lacks after-fix real iMessage proof.

## What This Changes

The PR adds `CommandSource: "text"` to iMessage inbound context whenever the inbound decision is command-authorized and adds a focused `/new` context test.

## Best Possible Solution

Carry an iMessage command-presence fact or structured `CommandTurn` from the inbound decision, emit a text-slash turn only for actual authorized control commands, and cover both slash and ordinary authorized messages.

## Reproduction Assessment

Yes from source inspection: current main detects `hasControlCommandInMessage` but only forwards `CommandAuthorized` into the iMessage context, so authorized slash commands can finalize without a text command source. I did not establish a live iMessage failing path.

## Solution Assessment

No: using `decision.commandAuthorized` alone is broader than the command-source contract. The safer fix should require an actual control command or pass a structured text-slash `CommandTurn` from the inbound decision.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Gate CommandSource on actual slash commands:** `extensions/imessage/src/monitor/inbound-processing.ts:1034`
