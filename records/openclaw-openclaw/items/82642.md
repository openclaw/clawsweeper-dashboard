## Summary

Keep open: this targets a real iMessage slash-command acknowledgement gap, but the patch is too broad because `commandAuthorized` is an authorization fact, not proof that the inbound text was a slash command, and the PR still lacks required real iMessage behavior proof.

## What This Changes

The PR adds `CommandSource: "text"` to authorized iMessage inbound contexts and adds focused coverage for an authorized `/new` turn.

## Best Possible Solution

Carry explicit text-command state from the iMessage inbound decision and set `CommandSource` or structured `CommandTurn` only for actual authorized control commands, with focused regression coverage and redacted real iMessage proof.

## Reproduction Assessment

Yes from source inspection: current main can dispatch authorized iMessage slash commands with `CommandAuthorized` but no `CommandSource`, while source-reply policy treats explicit text commands specially. I did not establish a live iMessage failing path.

## Solution Assessment

No: setting `CommandSource` from `decision.commandAuthorized` alone is broader than the command-source contract. The safer fix should require an actual control command or carry a structured command turn from the inbound decision.
