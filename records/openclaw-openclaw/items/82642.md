## Summary

Keep open: the patch uses command authorization as a command-source signal, which can misclassify ordinary authorized iMessage messages as explicit text slash commands and bypass source-reply visibility policy; it also still lacks real iMessage behavior proof.

## What This Changes

The PR sets iMessage inbound `CommandSource` to `"text"` when `decision.commandAuthorized` is true and adds focused inbound-processing coverage for `/new`.

## Best Possible Solution

Carry an actual text-command fact from iMessage inbound decision, or emit a structured `CommandTurn`, and only mark authorized slash commands as text command turns with real iMessage acknowledgement proof.

## Reproduction Assessment

Yes from source inspection: current main lacks an iMessage text-command source for authorized slash commands, and the PR condition would also match authorized non-command turns. I did not establish a live iMessage failing path.

## Solution Assessment

No: `decision.commandAuthorized` is too broad for the command-source contract. The fix should require an actual control command or pass a structured command turn from the inbound decision.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Gate CommandSource on actual slash commands:** `extensions/imessage/src/monitor/inbound-processing.ts:1034`
  - body: `decision.commandAuthorized` is an authorization fact, not proof that the inbound iMessage was a slash command. Because the command gate can authorize a sender when `hasControlCommandInMessage` is false, this line can make ordinary authorized messages normalize as explicit `text-slash` turns and force automatic source replies.
  - confidence: 0.86
