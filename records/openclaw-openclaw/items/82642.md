## Summary

Keep open: the PR targets a plausible iMessage slash-command acknowledgement bug, but the patch marks every authorized iMessage dispatch as a text slash command and still lacks real iMessage behavior proof.

## What This Changes

The PR sets iMessage inbound `CommandSource` to `text` whenever an inbound decision is command-authorized and adds focused `/new` context coverage.

## Best Possible Solution

Carry an actual iMessage text-command fact into the inbound context, or emit a structured `CommandTurn`, and only mark authorized slash-command turns as text commands with real `/new` acknowledgement proof.

## Reproduction Assessment

Yes from source inspection: current iMessage inbound context carries `CommandAuthorized` but no text-command source, while source reply delivery only forces automatic replies for explicit command turns. I did not run a live iMessage reproduction.

## Solution Assessment

No: `decision.commandAuthorized` is too broad to stand in for a slash-command source. The safer fix is to carry `hasControlCommandInMessage` or a structured command turn and gate `CommandSource` on that actual command fact.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Gate CommandSource on actual slash commands:** `extensions/imessage/src/monitor/inbound-processing.ts:1034`
  - body: `decision.commandAuthorized` is an authorization fact, not proof that the inbound iMessage was a slash command. Because the command gate can authorize a sender when `hasControlCommandInMessage` is false, this can normalize ordinary authorized iMessage messages as explicit text-slash turns and force automatic source replies.
  - confidence: 0.88
