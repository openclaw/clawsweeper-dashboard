## Summary

Keep open: the bug-fix direction is plausible, but the patch currently over-classifies any authorized iMessage turn as a text command and the external PR lacks real iMessage behavior proof before merge.

## What This Changes

The PR sets iMessage inbound `CommandSource` to `"text"` for authorized commands and adds focused inbound-processing coverage for `/new`.

## Best Possible Solution

Emit text command source or a structured text-slash `CommandTurn` only when the iMessage inbound message actually contains an authorized control command, while preserving normal delivery policy for ordinary authorized messages.

## Reproduction Assessment

Yes from source inspection: current main lacks iMessage `CommandSource` for authorized slash commands, but the PR's proposed condition would also match authorized non-command turns. I did not establish a live iMessage failing path.

## Solution Assessment

No: using `decision.commandAuthorized` alone is broader than the command-source contract. The safer fix should additionally require an actual control command or carry a structured command turn from the inbound decision.

Overall correctness: patch is incorrect

- **[P1] Gate CommandSource on actual slash commands:** `extensions/imessage/src/monitor/inbound-processing.ts:1034`
