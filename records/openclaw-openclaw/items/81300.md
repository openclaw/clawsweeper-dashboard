## Summary

Keep open: this is a plausible Codex app-server bug fix with real thread/start behavior proof, but the patch still relies on `thread/resume` config for existing running threads even though upstream Codex ignores config overrides in that path.

## What This Changes

The PR maps Codex session `reasoningLevel` values `on` and `stream` into `model_reasoning_summary: "auto"` for Codex app-server thread start/resume and adds focused thread-lifecycle tests.

## Best Possible Solution

Resolve the Codex reasoning summary once from the session preference and apply it on the effective turn path, while preserving off/unset passthrough so user Codex config is not clobbered.

## Reproduction Assessment

Yes, source inspection plus the PR body's redacted live logs give a high-confidence reproduction for the new-thread path: `sessions.patch` with `reasoningLevel: stream` should make Codex emit `summary_text` events that OpenClaw already projects. The existing-running-thread gap is source-reproducible from upstream Codex ignoring `thread/resume` config overrides while OpenClaw does not pass `summary` on `turn/start`.

## Solution Assessment

No, not yet. The thread/start mapping is the right shape, but the patch should use Codex's per-turn `summary` override or intentionally restart/update the thread when the preference changes instead of relying on ignored running-thread resume config.

## Review Findings
