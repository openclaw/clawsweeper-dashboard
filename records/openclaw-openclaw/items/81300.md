## Summary

Keep this PR open: the intended Codex reasoning-summary plumbing is directionally useful, but the patch currently loses the distinction between an unset reasoning preference and the normal resolved default of `off`, so it can clobber a user's Codex config. The PR also has only unit-test evidence and still needs real after-fix behavior proof before merge.

## What This Changes

The PR maps Codex session `reasoningLevel` values into `model_reasoning_summary` in Codex app-server thread start and resume config, with focused tests for the mapping.

## Best Possible Solution

Preserve explicit reasoning intent through the Codex harness path so explicit `stream`/`on` sets `auto`, explicit `off` sets `none`, and an unset preference leaves `model_reasoning_summary` absent; then add focused regression tests and real Codex app-server proof.

## Reproduction Assessment

Yes, by source inspection: current reply resolution defaults missing reasoning state to `off`, and the PR maps that value to `model_reasoning_summary: "none"`. I did not run a live Codex app-server turn in this read-only review.

## Solution Assessment

No, not as written. The mapping for `stream`/`on` is the right shape, but the implementation should distinguish explicit `off` from the default unset path before writing a Codex config override.

- **[P2] Preserve unset reasoning before forcing summaries off:** `extensions/codex/src/app-server/thread-lifecycle.ts:303-306`

Please pass explicit/raw reasoning state into this path, or otherwise distinguish explicit `off` from the default before writing the config patch.
