## Summary

Keep this PR open. The status-routing direction is useful, and the unchanged PR head still addresses a real gap in current main, but the proposed resolver field drops inherited `agents.defaults.contextTokens` and the user-visible `/status` fix still needs the required changelog entry.

## What This Changes

The PR adds `contextTokens` to resolved agent config, changes `/status` to prefer the target session entry's persisted context window, and adds focused regression tests.

## Best Possible Solution

Keep the targeted target-session `/status` change, make the new resolved config field return the effective inherited cap with `entry.contextTokens ?? agentDefaults?.contextTokens`, add inherited-default regression coverage, and record the user-visible fix in `CHANGELOG.md`.

## Reproduction Assessment

Yes. On the PR diff, a config with `agents.defaults.contextTokens` and an agent entry without its own override would make `resolveAgentConfig(cfg, "main")?.contextTokens` return `undefined` because the new field reads only `entry.contextTokens`.

## Solution Assessment

No, not yet. The `/status` routing change is the right narrow shape, but the resolver must preserve inherited defaults before callers can rely on the new field as the effective context-window cap.
