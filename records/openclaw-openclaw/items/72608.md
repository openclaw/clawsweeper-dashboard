## Summary

Keep open. Current main still passes the caller context window into `/status`, and the PR is directionally right, but the resolver addition drops inherited `agents.defaults.contextTokens` and the user-visible fix needs a changelog entry.

## What This Changes

The PR adds `contextTokens` to resolved agent config, switches `/status` to use the target session entry's persisted context window, and adds focused regression tests.

## Best Possible Solution

Keep the targeted `/status` target-session change, return `entry.contextTokens ?? agentDefaults?.contextTokens` from the resolver, add inherited-default coverage, and record the fix in `CHANGELOG.md` while leaving the separate model-limit cap work to #39859.

## Reproduction Assessment

Yes. Build `handleStatusCommand` params with `params.contextTokens = 272000` and a target session entry containing `contextTokens: 500000`; current main passes `272000`, while the PR fixes that path. For the PR regression, a config with only `agents.defaults.contextTokens` would still resolve `undefined` because the new field reads only `entry.contextTokens`.

## Solution Assessment

No, not yet. The `/status` routing change is the narrow maintainable fix, but the resolver must preserve inherited defaults before callers can rely on the new field as the effective context-window cap.
