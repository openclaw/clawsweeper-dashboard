## Summary

Keep open. Current main still has the target-session `/status` context-window gap, and the PR is a useful narrow fix, but it needs one inherited-default repair plus the required changelog entry before merge. The related #39859 work covers capping an effective window by a known model limit, which is adjacent but does not supersede this target-session routing/resolved-config change.

## What This Changes

The PR adds `contextTokens` to resolved agent config, routes `/status` to the target session entry's persisted context window, and adds focused regression tests for those paths.

## Best Possible Solution

Keep the targeted target-session `/status` change, make the new resolved config field return the effective inherited cap with `entry.contextTokens ?? agentDefaults?.contextTokens`, add inherited-default regression coverage, and record the user-visible fix in `CHANGELOG.md` without folding in the separate #39859 model-window cap work.

## Reproduction Assessment

Yes. On current main, a target session entry with `contextTokens: 500000` and caller params with `contextTokens: 272000` still sends `272000` to `buildStatusReply`; on the PR diff, a config with only `agents.defaults.contextTokens` would make `resolveAgentConfig(cfg, "main")?.contextTokens` return `undefined`.

## Solution Assessment

No, not yet. The `/status` routing change is the right narrow shape, but the resolver must preserve inherited defaults before callers can rely on the new field as the effective context-window cap.
