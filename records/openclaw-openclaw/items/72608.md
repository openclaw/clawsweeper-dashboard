## Summary

Keep open. The PR addresses a valid agents/status context-window gap, but the proposed resolver field still drops an inherited `agents.defaults.contextTokens` value and the user-visible `/status` fix is missing the required changelog entry.

## What This Changes

The PR adds `contextTokens` to resolved agent config, routes `/status` with the target session context window, and adds regression tests for both code paths.

## Best Possible Solution

Keep the targeted `/status` routing change, make the resolver expose the effective context-token cap with `entry.contextTokens ?? agentDefaults?.contextTokens`, add inherited-default regression coverage, and add one Unreleased/Fixes changelog line while leaving the broader model-limit capping work to #39859.

## Reproduction Assessment

Yes. A defaults-only config with `agents.defaults.contextTokens` and an agent entry lacking its own override reproduces the resolver gap, and a `/status` command whose stored target session has a different `contextTokens` value exercises the routing path.

## Solution Assessment

No, not yet. The status routing direction is maintainable, but the resolver addition should preserve inherited defaults before callers can safely treat it as an effective context-window cap.
