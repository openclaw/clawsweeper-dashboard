## Summary

Keep this PR open. The target-session `/status` direction is small and compatible, but the resolver addition is still incomplete because it exposes only the per-agent `contextTokens` value and drops an inherited `agents.defaults.contextTokens` cap; the PR also lacks the required changelog entry for a user-visible status/config fix.

## What This Changes

This PR adds `contextTokens` to resolved agent config, routes `/status` with the target session entry's context window, and adds focused regression tests for both paths.

## Best Possible Solution

Keep the targeted `/status` routing fix, make the resolver return the effective inherited cap with `entry.contextTokens ?? agentDefaults?.contextTokens`, add inherited-default regression coverage, and leave the broader model-limit capping work to #39859.

## Reproduction Assessment

Yes. The review finding is reproducible on the PR diff with a config that sets only `agents.defaults.contextTokens` and an agent entry without its own override, then reads `resolveAgentConfig(cfg, id)?.contextTokens`.

## Solution Assessment

No, not yet. The `/status` change is a narrow maintainable direction, but the resolver field must preserve inherited defaults before callers can safely treat it as the effective context-window cap.
