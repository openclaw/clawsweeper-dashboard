## Summary

Keep open. Current main has the merged Slack/Teams preview-removal fix, but the remaining Slack double-render behavior is still plausible from source: the label-only event is drained into the reply body while the embedded runner can also split the same prompt-local prefix into a hidden runtime-context message, and no current regression proves the header appears only once.

## What This Changes

The issue asks OpenClaw to stop rendering the same Slack inbound `System:` event header twice in one model-facing turn.

## Best Possible Solution

Add a focused Slack-like model-boundary regression that counts the inbound `System:` header, then adjust prompt assembly so the label-only event appears exactly once while the raw body is still delivered once.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Medium source confidence: current source shows Slack queues the event, reply assembly prepends it into the user body, and the embedded runner can split prompt-local runtime context, but I did not run live Slack or a provider-payload capture.

## Solution Assessment

Is this the best way to solve the issue? The report's direction is right, but the fix should be anchored by a model-boundary regression and should not blindly remove Slack's label-only enqueue because that event is still useful context.
