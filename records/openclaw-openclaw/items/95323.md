## Summary

Keep open. Current main has the preview-removal part from the related merged PR, but the reported remaining problem is a distinct Slack prompt-assembly bug: a drained inbound system event is still routed through the reply body path while the embedded runner can also lift the same prefix into a runtime-context message, and there is no focused model-boundary regression proving it appears only once.

## What This Changes

The issue requests that Slack inbound system-event headers render only once per assembled model turn after the inbound preview text was removed.

## Best Possible Solution

Add a focused Slack-like model-boundary regression that counts the inbound `System:` header, then adjust the owning prompt assembly so the label-only inbound event appears in exactly one model-visible place while the raw user body remains delivered once.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not with a live Slack run in this read-only pass, but the issue includes a concrete model-facing payload and current source shows the two relevant render paths that need a one-occurrence regression.

## Solution Assessment

Is this the best way to solve the issue? The report's direction is right, but the fix should be driven by a model-boundary regression rather than removing Slack's enqueue path blindly, because the enqueue is still useful as a label-only runtime event.
