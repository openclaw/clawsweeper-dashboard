## Summary

Keep open: current main still lacks provider guarded-fetch env-proxy selection, while the latest PR head is a narrow, tested fix with a changelog entry and green public checks. I found no discrete functional or security defect in the proposed patch.

## What This Changes

The PR updates provider model guarded fetches to enter trusted env-proxy mode when HTTP(S) proxy env vars apply and no explicit provider dispatcher policy exists, adds regression tests for both decision paths, and adds an Unreleased changelog entry.

## Best Possible Solution

Land this PR or an equivalent narrow patch after standard merge gating, preserving the shared guarded-fetch helper and proxy-env predicate rather than changing global fetch behavior or weakening explicit provider transport policies.

## Reproduction Assessment

Yes. The focused reproduction is to set HTTPS_PROXY, build a provider model fetch with no explicit dispatcher policy, and assert that guarded fetch options enter trusted_env_proxy mode; current main inspection shows that path remains strict.

## Solution Assessment

Yes. Reusing withTrustedEnvProxyGuardedFetchMode behind shouldUseEnvHttpProxyForUrl and the no-dispatcherPolicy guard is the narrowest maintainable fix, and the PR tests cover both the env-proxy path and explicit dispatcher preservation.
