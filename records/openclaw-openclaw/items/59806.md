## Summary

Keep open: current main still has the reported Feishu token-request failure mode. The Feishu plugin now depends on the Lark SDK 1.62.0, but that SDK still swallows a rejected tenant token POST before destructuring, while OpenClaw's Feishu client wrapper only adds timeout/User-Agent behavior and WebSocket proxy support. Related #48949 and open PR #41943 cover a concrete proxy-triggered version, but they do not supersede this broader SDK token-error report.

## What This Changes

The issue requests that Feishu token acquisition failures surface the original HTTP/token error instead of crashing with `tenant_access_token` destructuring from `undefined`.

## Best Possible Solution

Fix the Feishu-owned SDK token boundary so rejected token requests preserve the original HTTP/proxy cause, coordinate that with explicit Feishu/Lark/custom-domain proxy semantics, and add focused regression coverage plus troubleshooting docs. Avoid patching installed node_modules at runtime; use an approved dependency patch/update or a maintainable Feishu client adapter.

## Reproduction Assessment

Yes for the reported symptom: SDK 1.62.0 still has a static path where a rejected tenant token POST is caught without rethrowing and then destructured. The exact Windows live setup was not reproduced locally, but #48949 provides a concrete proxy-triggered reproduction for the same user-visible error.

## Solution Assessment

No current solution is in place on main. The narrow maintainable fix should be Feishu-owned and deliberate about proxy policy; PR #41943 may be useful source work for the proxy trigger, but it does not by itself settle the broader swallowed token-error behavior.
