## Summary

Keep open: current main still contains the Feishu tenant-token failure mode. The bundled SDK moved to 1.63.1, but upstream still swallows rejected tenant-token POSTs before destructuring, and OpenClaw’s Feishu wrapper still delegates token/API HTTP through the SDK default instance without preserving the original failure.

## What This Changes

The issue asks OpenClaw to stop Feishu tenant-token failures from being hidden behind the upstream SDK’s `tenant_access_token` destructuring error.

## Best Possible Solution

Fix the Feishu-owned token/REST HTTP boundary so rejected tenant-token requests preserve the real HTTP/proxy cause, coordinating with https://github.com/openclaw/openclaw/issues/77323 and https://github.com/openclaw/openclaw/issues/48949 instead of patching installed `node_modules`.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: current main delegates token POSTs to the Lark SDK default HTTP instance, and `@larksuiteoapi/node-sdk@1.63.1` still catches rejected tenant-token POSTs without rethrowing before destructuring. I did not run a live Windows/Feishu credential scenario.

## Solution Assessment

No current solution is in place on main. The reported installed-SDK patch is not the best repository fix; the maintainable path is an owner-local Feishu adapter or upstream SDK update with regression coverage, coordinated with the open proxy transport trackers.
