## Summary

Keep open: current main still routes Feishu tenant-token acquisition through the Lark SDK path that can swallow the real rejected POST error and then fail with the reported `tenant_access_token` destructuring message. The dependency has moved to `@larksuiteoapi/node-sdk` 1.63.1, but that is still the latest published version and still contains the same token-manager catch-without-rethrow pattern.

## What This Changes

The issue asks OpenClaw to stop Feishu initialization from hiding the real tenant-token HTTP failure behind a confusing `tenant_access_token` destructuring crash.

## Best Possible Solution

Fix the Feishu-owned token acquisition boundary so rejected tenant-token requests surface the original HTTP/proxy cause, coordinate with https://github.com/openclaw/openclaw/issues/48949 and https://github.com/openclaw/openclaw/pull/41943, then add focused regression coverage and troubleshooting docs.

## Reproduction Assessment

Yes at source level: make the Lark SDK tenant-token POST reject and the published 1.63.1 token manager still catches without rethrowing before destructuring the result. The exact reporter Windows setup was not live-run.

## Solution Assessment

No current solution is in place on main. The maintainable fix should preserve the original token HTTP failure at the Feishu SDK boundary and settle proxy semantics explicitly, not patch installed `node_modules` at runtime.
