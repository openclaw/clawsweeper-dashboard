## Summary

Keep open. The linked bug is source-reproducible on current main, but this PR is not safe to merge as submitted because it removes the documented Slack thread-root media inheritance path, lacks inspectable real behavior proof, and is conflicting.

## What This Changes

The PR changes Slack message preparation so thread replies never hydrate media from the thread starter, and adds tests asserting starter media is not resolved for new or existing thread sessions.

## Best Possible Solution

Land a rebased Slack fix that stops repeated parent-media rehydration after the initial thread context is recorded while preserving reply-owned media and intentional first-turn root-media context, or explicitly update the Slack contract and docs if maintainers choose the never-hydrate policy.

## Reproduction Assessment

Yes, source-level. Current main hydrates Slack starter files into threadStarterMedia for a text-only thread reply with no direct media, then forwards that fallback as effective inbound media; I did not run live Slack because this review is read-only.

## Solution Assessment

No, not as submitted. The broad never-hydrate fix removes a documented first-turn/thread-root media path; the safer solution is to stop repeated rehydration after session context exists or make an explicit maintainer-approved contract change.
