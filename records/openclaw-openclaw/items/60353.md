## Summary

Keep open. Current main still re-injects Slack thread-starter media for text-only replies, but this external PR needs maintainer handling because it lacks inspectable live proof, has typecheck/docs blockers, and changes a documented thread-root media contract.

## What This Changes

The PR changes Slack thread context preparation to stop hydrating thread-starter media for replies, uses only direct media in `prepareSlackMessage`, and adds thread-context tests.

## Best Possible Solution

Settle and document the Slack thread-root media contract, then land a rebased Slack plugin fix that stops repeated parent-media reinjection while preserving reply-owned media and any intended first-turn root-media behavior.

## Reproduction Assessment

Yes, source-level. On current main, a text-only Slack thread reply with starter files and no direct media hydrates `starter.files` into `threadStarterMedia`, then forwards that fallback through the inbound media fields.

## Solution Assessment

No, not as submitted. The PR targets the right Slack prepare path, but it needs a maintainer decision on the documented thread-root media contract, a current-main rebase, typecheck cleanup, docs alignment, and inspectable live proof.
