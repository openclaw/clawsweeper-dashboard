## Summary

Keep open. The PR targets a real current-main Slack media leak, but its unconditional never-hydrate change breaks the shipped and documented Slack thread-root media inheritance path; the branch also lacks inspectable real behavior proof and still has typecheck blockers.

## What This Changes

The PR changes Slack thread context preparation to stop hydrating thread-starter media for replies, uses only direct media in prepared Slack messages, and adds thread-context tests for starter-media suppression.

## Best Possible Solution

Land a rebased Slack fix that stops repeated parent-media downloads on existing thread replies while preserving first/new-session thread-root media inheritance and reply-owned media, with docs, tests, and real Slack proof aligned.

## Reproduction Assessment

Yes, source-level. Current main hydrates starter.files into threadStarterMedia before checking whether the thread session already exists, then forwards that fallback through the Slack inbound media fields for text-only replies.

## Solution Assessment

No, not as submitted. The narrow maintainable fix should prevent repeated hydration after the thread session exists while preserving the shipped thread-root media inheritance contract, or else explicitly change that documented product behavior with maintainer approval.
