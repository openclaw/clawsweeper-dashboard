## Summary

Keep open. The branch targets the real Slack parent-media duplication reported in https://github.com/openclaw/openclaw/issues/60335, but the current patch removes a documented Slack thread-root media workflow, is conflicting with current main, and still lacks inspectable real behavior proof.

## What This Changes

The PR changes Slack thread preparation so thread replies never hydrate media from the thread starter, removes the starter-media fallback from prepared media selection, and adds tests for that suppression.

## Best Possible Solution

Land a rebased Slack fix that stops repeated parent-media rehydration after the intended thread-root context path, preserves reply-owned media and documented first-turn root-media context, or explicitly updates docs/tests if maintainers choose to remove that contract.

## Reproduction Assessment

Yes, source-level. Current main hydrates `starter.files` into `threadStarterMedia` for a text-only Slack thread reply and then forwards that fallback as effective inbound media; I did not run live Slack proof in this read-only review.

## Solution Assessment

No as written. The bug is real, but the never-hydrate implementation is broader than the existing documented contract; a narrower rebased fix or an explicit maintainer-approved contract change is safer.
