## Summary

Keep open. Current main appears to have fixed the Feishu text extraction and mention-normalized command probing side, but the shared command authorization path still denies group slash commands when `commands.useAccessGroups=true` and no command allowlist is configured; PR #49958 is an open closing implementation candidate, so this issue should remain open until that PR or an equivalent fix lands.

## What This Changes

Feishu group chat slash commands such as `/new`, `/status`, and `/model` should either execute as OpenClaw control commands or remain visible as normal text instead of being lost as empty-content events.

## Best Possible Solution

Land PR #49958 or an equivalent narrow shared command-gating fix that treats an absent command allowlist as unrestricted for ordinary group commands, while preserving fail-closed behavior when an allowlist is configured and the sender is not allowed; keep the existing Feishu parser normalization and add regression coverage for the no-allowlist group case.

## Reproduction Assessment

Yes. A source-level reproduction is available: Feishu group messages with no per-group or top-level `allowFrom` pass an unconfigured authorizer into `resolveCommandAuthorizedFromAuthorizers`, and current tests assert that this returns false when `useAccessGroups` is true.

## Solution Assessment

Unclear for the exact PR until maintainers review it, but the direction is the narrowest maintainable path: fix the shared command gate with tests instead of adding Feishu-specific parser work, because the parser side is already covered on current main.
