## Summary

Keep open: current main still records unpaired MS Teams DMs and returns without sending the shared pairing challenge or logging the pair code. The linked repair PR #62812 is closed unmerged, and the open Microsoft tracker #74163 still lists this as unresolved.

## What This Changes

The issue asks MS Teams `dmPolicy: "pairing"` to surface unpaired DM requests with code-bearing logs and an in-band pairing reply instead of only recording and dropping them.

## Best Possible Solution

Update the MS Teams personal-DM pairing rejection path to reuse `pairing.issueChallenge`, preserving the current conversation-reference capture and drop semantics while sending the standard Teams pairing reply and logging sender, label, and code on first contact.

## Reproduction Assessment

Yes. A personal MS Teams message from an unapproved AAD sender with `channels.msteams.dmPolicy: "pairing"` reaches the `access.decision === "pairing"` branch, upserts a pairing request, and returns before any Teams reply; the current authz harness constructs this path with a mocked `sendActivity`.

## Solution Assessment

Yes. Reusing `pairing.issueChallenge` is the narrowest maintainable fix because it already owns create-if-missing behavior, first-created-only replies, standard pairing text, code-bearing creation callbacks, and reply-error handling used by other channel plugins.
