## Summary

Keep open: current main still records unpaired MS Teams DMs and then returns without sending the shared pairing challenge or logging the code. The linked repair PR #62812 is closed unmerged, so a narrow replacement fix is still needed.

## What This Changes

Unpaired MS Teams direct messages under `dmPolicy="pairing"` should surface the pairing code instead of only recording the request and dropping the message.

## Best Possible Solution

Use the scoped `pairing.issueChallenge` helper in the MS Teams personal-DM pairing rejection path, preserving conversation-reference capture and drop semantics while sending the standard first-contact reply and logging code-bearing creation and reply-failure details.

## Reproduction Assessment

Yes. A mocked personal Teams message with `channels.msteams.dmPolicy: "pairing"` and no allowlist reaches the current `access.decision === "pairing"` branch, calls `pairing.upsertPairingRequest`, and returns without `sendActivity`.

## Solution Assessment

Yes. Reusing the existing `issueChallenge` helper is the narrowest maintainable fix because it already implements create-if-missing behavior, first-created-only replies, code callbacks, and reply-error handling used by other channel plugins.
