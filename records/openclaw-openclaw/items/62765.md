## Label Justifications

- `P1`: The default MS Teams DM pairing workflow accepts real first-contact messages but gives users no in-channel pairing challenge and no agent processing.

## Summary

Keep open: current main still records MS Teams unpaired DM pairing requests and drops the message without sending the shared in-band pairing challenge; the targeted repair PR was closed unmerged, so the actionable bug remains.

## What This Changes

The issue requests that MS Teams `dmPolicy: "pairing"` surface unpaired sender pairing requests with useful logging and an in-band Teams pairing reply instead of accepting and dropping the DM.

## Best Possible Solution

MS Teams should reuse `pairing.issueChallenge` for unpaired personal DMs while preserving conversation-reference capture, authorization decisions, repeat-request behavior, and existing drop semantics after the challenge is issued.

## Reproduction Assessment

Yes, by source inspection. A personal MS Teams message from an unapproved sender under `dmPolicy: "pairing"` reaches the pairing branch, records a request, and returns before any Teams reply path is called.

## Solution Assessment

Yes. Reusing `pairing.issueChallenge` is the narrowest maintainable fix because the existing helper centralizes request creation, first-time reply text, code-bearing callbacks, and reply-error handling.
