## Summary

Keep open: current `main` still records and drops unpaired MS Teams personal DMs without sending the pairing challenge, and the linked repair PR is closed unmerged. This is a narrow repair candidate because the shared pairing challenge helper already provides the missing one-time reply and code-bearing creation callback.

## What This Changes

The issue requests that MS Teams pairing-mode DMs surface pending pair codes through logging and an in-band Teams reply instead of silently recording and dropping unpaired senders.

## Best Possible Solution

Update the MS Teams personal-DM pairing rejection path to use `pairing.issueChallenge`, preserving conversation-reference capture and drop semantics while sending the standard Teams pairing reply on first contact and logging sender, label, and code.

## Reproduction Assessment

Yes, from current source inspection. A personal MS Teams message from an unapproved sender with `channels.msteams.dmPolicy: "pairing"` reaches the `pairing.upsertPairingRequest` branch and returns before any Teams reply; the existing authz test harness can exercise that path with a mocked `sendActivity`.

## Solution Assessment

Yes. Reusing `pairing.issueChallenge` is the narrowest maintainable fix because it already implements create-if-missing behavior, one-time replies, code-bearing creation callbacks, and reply-error handling used by other channel plugins.
