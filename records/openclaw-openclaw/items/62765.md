## Summary

Current main still leaves the reported MS Teams pairing UX bug unresolved: unpaired personal DM senders are recorded in the pairing store and then dropped, but the Teams handler never sends the generated pairing challenge and its creation log omits the code. The linked fix PR proposed the right shared-helper direction, but it is closed unmerged and the implementation is not present on main.

## What This Changes

The issue asks for MS Teams `dmPolicy: "pairing"` to surface unpaired DM pairing requests through an info log and in-band Teams pairing challenge instead of silently returning HTTP 200 with no visible reply.

## Best Possible Solution

Update the MS Teams personal-DM pairing rejection path to use the existing shared pairing challenge helper, preserving conversation reference capture and drop semantics while sending the standard Teams reply on first contact and logging sender, label, and code. Add focused regression coverage so the path cannot return to store-only silent drops.

## Reproduction Assessment

Yes. A personal MS Teams message from an unapproved sender with `channels.msteams.dmPolicy: "pairing"` reaches the current `pairing.upsertPairingRequest` branch and returns before any Teams reply; the existing authz unit harness can reproduce the path by mocking `sendActivity` and the pairing-store result.

## Solution Assessment

Yes. Reusing `pairing.issueChallenge` is the narrowest maintainable fix because the shared helper already implements create-if-missing behavior, first-created-only replies, code-bearing creation callbacks, and reply-error handling used by other channel plugins.
