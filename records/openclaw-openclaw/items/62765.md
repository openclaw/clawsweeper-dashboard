## Summary

Keep open: current main still records an unpaired MS Teams DM sender and returns without sending a pairing challenge, while the linked repair PR is closed unmerged. This is a narrow replacement-fix candidate because the shared pairing challenge helper already provides the missing reply and creation-log behavior.

## What This Changes

The issue asks MS Teams pairing-mode DMs from unpaired senders to emit an operator-visible log and/or in-band pairing challenge instead of silently recording and dropping them with HTTP 200.

## Best Possible Solution

Update the MS Teams personal-DM pairing rejection path to use the existing shared pairing challenge helper, preserving conversation-reference capture and drop semantics while sending the standard Teams reply on first contact and logging sender, label, and code.

## Reproduction Assessment

Yes. A personal MS Teams message from an unapproved sender with `channels.msteams.dmPolicy: "pairing"` reaches the current `pairing.upsertPairingRequest` branch and returns before any Teams reply; the existing authz unit harness can reproduce this with a mocked `sendActivity`.

## Solution Assessment

Yes. Reusing `pairing.issueChallenge` is the narrowest maintainable fix because it already implements create-if-missing behavior, first-created-only replies, code-bearing creation callbacks, and reply-error handling used by other channel plugins.
