## Summary

Keep open. Current main still has a credible path where provider-prefixed Google model ids from configured Google models can reach the native Gemini transport and be sent as `models/google/...`, while overlapping fixes remain open in #41398 and #41517.

## What This Changes

The issue reports that `google/gemini-3.1-flash-lite-preview` works in the main session but fails with 404s when inherited by sub-agents or reset sessions.

## Best Possible Solution

Land one consolidated provider-owned fix that strips only redundant active-provider prefixes before Google alias lookup and native Gemini URL construction, while preserving nested model ids for wrapper providers.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure `models.providers.google` with `api: google-generative-ai`, a Gemini API `baseUrl`, and a model id like `google/gemini-3-flash-preview`, then let sub-agent inheritance or session reset resolve that configured model into the native Google transport.

## Solution Assessment

No. The attempted direction of forcing a `models/` prefix in built output is not the maintainable fix; the safer solution is scoped Google/same-provider prefix normalization before alias lookup and transport URL construction.
