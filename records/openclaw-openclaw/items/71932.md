## Summary

Keep open. Current main still preserves provider-prefixed Google model ids through the configured-model path, and the native Gemini URL builder can send them as `models/google/...`; the two related fix PRs are closed unmerged rather than shipped.

## What This Changes

The issue reports Google Gemini native transport 404s when sub-agents or session resets inherit provider-prefixed model ids such as `google/gemini-3.1-flash-lite-preview`.

## Best Possible Solution

Add provider-owned Google model-id normalization that sends bare Gemini ids to native Google transport while preserving wrapper-provider nested ids, with regression tests for configured models, sub-agent inheritance, and reset.

## Reproduction Assessment

Yes, from source inspection. Configure `models.providers.google` with `api: google-generative-ai` and a configured model id like `google/gemini-3.1-flash-lite-preview`; current resolver paths can preserve that id and the Google transport then constructs `models/google/...`.

## Solution Assessment

Yes, the best direction is a narrow Google-owned normalization and transport fix. A generic same-provider strip is riskier because other providers can have meaningful nested model ids.
