## Summary

Keep open: current main and the latest release still have a source-visible Google native transport path where a configured `google/gemini-*` model id can survive into the request path as `models/google/...`. The overlapping fix PRs were closed unmerged, so this issue remains the canonical tracker.

## What This Changes

The issue asks OpenClaw to stop returning Google 404s when sub-agent spawn or session reset uses provider-prefixed Gemini model ids.

## Best Possible Solution

Land one Google-owned normalization fix that strips only redundant active-provider `google/` prefixes before Google model matching and native transport URL construction, while preserving proxy/nested ids and adding sub-agent and reset regression coverage.

## Reproduction Assessment

Yes, source-reproducible: a configured Google provider model id like `google/gemini-*` can be matched and preserved, then the native Google URL builder can turn it into `models/google/...`. I did not run live Google API traffic or a real sub-agent spawn.

## Solution Assessment

Yes. A provider-scoped Google normalization and transport guard is the narrowest maintainable solution; a generic same-provider strip would risk breaking valid nested ids for proxy or provider-local models.
