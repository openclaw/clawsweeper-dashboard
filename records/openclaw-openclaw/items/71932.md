## Summary

Keep open. Current main still has a credible Google inline-provider path where a configured `google/<model>` id is matched, preserved as the runtime model id, and sent to the native Google endpoint as `models/google/<model>`, which matches the reported 404 class and the corroborating config comment.

## What This Changes

The issue asks to fix Google Gemini model ids such as `google/gemini-3.1-flash-lite-preview` failing with 404 when inherited by sub-agents or reused after session reset.

## Best Possible Solution

Land one consolidated fix from #41398, #41517, or an equivalent narrow replacement that strips only redundant active-provider prefixes before Google alias lookup and native URL construction while preserving valid nested model ids for wrapper providers.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure `models.providers.google` with `api: google-generative-ai`, `baseUrl: https://generativelanguage.googleapis.com/v1beta`, and a model entry like `google/gemini-3-flash-preview`, then resolve that model through sub-agent inheritance or session reset into the native Google transport.

## Solution Assessment

No. Current main is not the best solution because it still lets provider-prefixed inline Google ids reach the Google URL builder; the safer fix is a scoped same-provider prefix strip with regression coverage for inline Google config, sub-agent inheritance, and reset.
