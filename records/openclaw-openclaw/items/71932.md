## Summary

Keep open. Current main still has a credible Google same-provider prefix path: configured Google model entries such as `google/gemini-...` can be matched and preserved as the resolved model id, while the native Gemini transport only protects `models/` and `tunedModels/` before prepending `models/`, matching the reported 404 class. The later issue comments add concrete config evidence, and the overlapping fixes are still represented by open related PRs rather than a merged implementation.

## What This Changes

The issue reports that Google Gemini models work in the main session but sub-agent spawning or session reset can send an invalid provider-prefixed Gemini model path and receive a 404 Model Not Found response.

## Best Possible Solution

Keep this open and consolidate the Google prefix work into one narrow fix that removes only redundant active-provider prefixes before native Google model matching or URL construction, preserves valid nested model ids for wrapper providers, and adds regression coverage for sub-agent spawn/reset plus configured Google model entries written as `google/<model>`.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure `models.providers.google` with `api: google-generative-ai` and a configured model id like `google/gemini-3-flash-preview`, then resolve an inherited bare Google model through the native sub-agent or reset path and inspect that current transport can build `models/google/...`.

## Solution Assessment

No. Current main is not the best solution because it still lets redundant same-provider prefixes survive into the native Google transport; the safer path is a scoped prefix-strip fix coordinated with #41398 or #41517.
