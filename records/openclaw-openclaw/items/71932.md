## Summary

Keep open. Current main still has a credible Google native-transport path where a provider-prefixed inline Google model id is preserved and then sent as `models/google/<model>`, and the overlapping fixes in #41398 and #41517 are still open rather than merged. 

## What This Changes

Fix Google sub-agent and session-reset model resolution so configured Google models like `google/gemini-3.1-flash-lite-preview` reach the Google Generative AI API as provider-local model ids.

## Best Possible Solution

Land one scoped fix that strips only redundant active Google provider prefixes before Google alias lookup and native Google request URL construction, while preserving valid nested model ids for OpenRouter/custom wrapper providers and adding regression coverage for sub-agent spawn and session reset.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure `models.providers.google` with `api: google-generative-ai`, `baseUrl: https://generativelanguage.googleapis.com/v1beta`, and a model id such as `google/gemini-3.1-flash-lite-preview`, then resolve it through a sub-agent or reset path into the native Google transport.

## Solution Assessment

No. Current main is not the best solution because it still allows the redundant Google provider prefix to survive; the safer path is to consolidate #41398 and #41517 into one narrow prefix-normalization fix with transport-level coverage.
