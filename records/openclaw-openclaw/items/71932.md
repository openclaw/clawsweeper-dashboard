## Summary

Keep open. Current main still has a credible Google provider-config path where a same-provider model id stored as `google/<model>` can be preserved into the native Gemini transport and sent as `models/google/<model>`, while the two overlapping fix PRs are still open.

## What This Changes

Fix Google Gemini 404s when sub-agent spawn or session reset reuses provider-prefixed Google model ids such as `google/gemini-3.1-flash-lite-preview`.

## Best Possible Solution

Land one consolidated Google-owned normalization fix that strips only redundant same-provider prefixes before Google alias lookup and native Gemini URL construction, while preserving nested model ids for wrapper providers.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure `models.providers.google` with `api: google-generative-ai`, a Gemini API `baseUrl`, and a model entry id like `google/gemini-3-flash-preview`, then let sub-agent inheritance or reset resolve that configured Google model into the native Gemini transport.

## Solution Assessment

No. Current main is not yet the best solution because provider-prefixed configured Google ids can still reach the Google URL builder; the safer fix is a scoped provider or manifest normalization rule with regression coverage for sub-agent and reset paths.
