## Summary

Keep open. Current main still has a credible path where a configured Google model id like `google/gemini-3-flash-preview` is matched as an inline model, preserved as the resolved model id, and then used by the native Gemini transport to build `models/google/...`, matching the reported 404 class and the corroborating follow-up config.

## What This Changes

The issue asks OpenClaw to stop returning Google Generative AI 404s when sub-agents or session reset inherit provider-prefixed Gemini model ids such as `google/gemini-3.1-flash-lite-preview`.

## Best Possible Solution

Land one focused Google/model-resolution fix that removes redundant same-provider prefixes before native Google matching and URL construction, preserves wrapper-provider nested ids, and covers sub-agent spawn/reset inheritance with regression tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure the Google provider with `api: google-generative-ai` and a configured model id `google/gemini-3-flash-preview`, then resolve an inherited bare Google model through the native sub-agent or reset path and inspect that current transport can build `models/google/...`.

## Solution Assessment

No. Current main is not the best solution because it still lets a redundant same-provider prefix survive into the native Google transport; the safer path is a scoped prefix-strip fix coordinated with #41398 or #41517.
