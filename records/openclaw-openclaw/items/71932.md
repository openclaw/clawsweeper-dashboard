## Summary

Keep open. Current main still has a credible path where a configured same-provider Google model id such as `google/gemini-3-flash-preview` can be preserved into the native Gemini transport and emitted as `models/google/...`, and the overlapping fixes in #41398 and #41517 are still open.

## What This Changes

The issue asks OpenClaw to make Google Gemini model refs work consistently for sub-agent spawn and session reset instead of returning Gemini API 404s.

## Best Possible Solution

Keep this open and land one consolidated fix that strips only redundant same-provider prefixes before Google alias lookup and native Gemini URL construction while preserving nested model ids for wrapper providers.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure `models.providers.google` with `api: google-generative-ai`, a Gemini API `baseUrl`, and a model id like `google/gemini-3-flash-preview`, then let sub-agent inheritance or reset resolve that model into the native Google transport.

## Solution Assessment

No. Current main is not the best solution because it still allows provider-prefixed inline Google ids to reach the Google URL builder; the safer path is a scoped provider-owned prefix strip with regression coverage for sub-agent inheritance and reset.
