## Summary

Keep open. Current main still has a credible Google native transport path where provider-prefixed configured Google model IDs can survive model resolution and be sent to the Gemini API as `models/google/...`, while the overlapping prefix-normalization PRs are still open rather than shipped.

## What This Changes

The issue requests fixing Google Gemini model resolution so sub-agent spawn or session reset no longer 404s when using provider-prefixed Gemini preview model IDs.

## Best Possible Solution

Land one focused Google/model-resolution fix that strips only redundant active-provider prefixes before native Google model matching and URL construction, preserves valid nested IDs for wrapper providers, and adds regression coverage for configured Google models plus sub-agent spawn/reset inheritance.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure `models.providers.google` with `api: google-generative-ai` and a model entry whose id is `google/gemini-3.1-flash-lite-preview`, then resolve an inherited or bare Google model through the native sub-agent/reset path and observe that the transport can build `models/google/...`.

## Solution Assessment

No. Current main is not the best solution because the redundant same-provider prefix can still survive into the native Google transport; the safer path is a scoped prefix-strip fix coordinated with #41398 or #41517.
