## Summary

Keep open. Current main still has a credible path where a configured Google model id written as `google/<model>` can be matched and preserved by the embedded runner, then sent through the native Google Generative AI transport as `models/google/<model>`. The recent follow-up comment provides a concrete provider config matching that path, and the related fixes are still open rather than merged.

## What This Changes

The issue reports that sub-agent spawn or session reset with a Google/Gemini model that works in the main session can fail with a 404 because the effective model id is sent to the Google API with the wrong prefix.

## Best Possible Solution

Normalize only redundant active-provider prefixes before Google model matching and native transport URL construction, preserve valid nested/pass-through model ids for wrapper providers, and add regression coverage for sub-agent spawn/reset inheritance with configured Google model entries written as `google/<model>`.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure `models.providers.google` with `api: google-generative-ai`, `baseUrl: https://generativelanguage.googleapis.com/v1beta`, and a configured model id like `google/gemini-3-flash-preview`, then follow sub-agent/reset model resolution into the native Google URL builder.

## Solution Assessment

No. Current main is not the best solution because it can preserve a redundant same-provider prefix into the native Google transport; the safer path is a narrow same-provider prefix strip coordinated with the existing open Google prefix PRs.
