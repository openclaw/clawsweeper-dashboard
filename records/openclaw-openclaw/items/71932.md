## Summary

Keep open. Current main still has a credible Google native transport path where a configured model id such as `google/gemini-3.1-flash-lite-preview` can be preserved through inline model resolution and sent as `models/google/...`; the overlapping fixes in #41398 and #41517 are still open implementation candidates, not shipped proof.

## What This Changes

The issue asks OpenClaw to fix Google Gemini 404s when sub-agents or session reset inherit a Google model configured with a `google/gemini-...` model id.

## Best Possible Solution

Keep this issue open and consolidate the Google prefix work into one narrow fix that strips only redundant active-provider prefixes before Google model matching and native transport URL construction, preserves valid nested model ids for wrapper providers, and adds regression coverage for sub-agent spawn/reset plus configured Google model entries whose ids are written as `google/<model>`.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure `models.providers.google` with `api: google-generative-ai` and a model entry whose id is `google/gemini-3.1-flash-lite-preview`, then resolve an inherited or bare Google model through the native sub-agent/reset path and observe that the transport can build `models/google/...`.

## Solution Assessment

No. Current main is not the best solution because the redundant same-provider prefix can still survive into the native Google transport; the safer path is a scoped prefix-strip fix coordinated with #41398 or #41517.
