## Summary

Keep open. Current main still has a source-level path where a configured Google model entry such as `google/gemini-3-flash-preview` can be matched for provider `google`, preserved as the model id, and sent to the native Gemini transport as `models/google/...`; the overlapping fixes in #41398 and #41517 are still open and should be consolidated rather than closing this as implemented.

## What This Changes

The issue reports Google Gemini model IDs working in the main session but failing with 404 when sub-agents spawn or sessions reset.

## Best Possible Solution

Land one narrow Google/provider-prefix fix that strips only redundant active-provider prefixes before Google alias lookup and native Gemini URL construction while preserving nested wrapper-provider model ids, with regression coverage for sub-agent inheritance, reset, and configured Google model entries.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure `models.providers.google` with `api: google-generative-ai`, a Gemini API `baseUrl`, and a model entry whose id is `google/gemini-3-flash-preview` or `google/gemini-3.1-flash-lite-preview`, then resolve provider `google` plus the bare model into the native Google transport.

## Solution Assessment

No. Current main is not the best solution because the inline configured-model path can still preserve the `google/` prefix into the transport; the safer fix is provider-scoped prefix stripping with tests that keep OpenRouter-style nested ids intact.
