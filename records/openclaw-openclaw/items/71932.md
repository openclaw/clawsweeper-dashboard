## Summary

Keep open. Current main still has a source-visible Google provider path where a configured `google/gemini-*` model id can survive into native Gemini URL construction as `models/google/...`, and the overlapping prefix-fix PRs are closed unmerged rather than shipped.

## What This Changes

The issue asks OpenClaw to stop 404s when Google Gemini models are used from sub-agent spawn or session reset with provider-prefixed model ids.

## Best Possible Solution

Land a narrow Google-owned normalization fix that strips only redundant `google/` before configured-model resolution and native Google transport URL construction, preserves valid nested proxy ids, and adds sub-agent/reset regression coverage.

## Reproduction Assessment

Yes. The bug is source-reproducible on current main: a configured Google model id like `google/gemini-*` can match inline config, remain prefixed in the resolved model, and be sent to the native transport as `models/google/...`; I did not send live Google API traffic.

## Solution Assessment

Yes. A provider-scoped Google normalization and transport fix is the narrowest maintainable path; a generic same-provider strip would be riskier because provider-native nested ids are valid elsewhere.
