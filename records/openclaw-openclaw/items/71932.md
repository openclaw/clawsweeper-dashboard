## Summary

Keep open: current main and the latest release still have a source-visible path where a configured Google model row such as `google/gemini-*` can be resolved as a native Google model and then sent to the Generative AI transport as `models/google/...`; the overlapping fix PRs were closed unmerged.

## What This Changes

The issue reports Google Gemini provider-prefixed model IDs returning 404 during sub-agent spawn or session reset.

## Best Possible Solution

Land one Google-owned normalization fix that strips only redundant `google/` from native Google model IDs before configured-model resolution and native transport URL construction, with sub-agent/reset regression coverage and no broad provider-prefix policy change.

## Reproduction Assessment

Yes, source-level: a configured Google provider model row with `id: "google/gemini-*"` can match inline config, remain prefixed on the resolved model, and be passed to the native Google transport as `models/google/...`. I did not run live Google API traffic or the reporter's exact sub-agent spawn payload.

## Solution Assessment

Yes. A provider-owned Google normalization and transport guard is the narrowest maintainable fix; a generic same-provider strip would risk breaking valid nested model IDs for proxy providers.
