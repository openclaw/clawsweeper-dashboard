## Summary

Keep open: current main and the latest release still have a source-visible path where a configured Google model row such as `google/gemini-*` can resolve into the native Google transport as `models/google/...`; the overlapping fix PRs were closed unmerged.

## What This Changes

The issue reports Google Gemini sub-agent/reset requests returning 404 when configured model IDs include a redundant `google/` prefix.

## Best Possible Solution

Land one Google-owned normalization fix that strips only redundant `google/` from native Google model IDs before matching and transport URL construction, preserves proxy nested IDs, and adds sub-agent/reset/configured-model regression coverage.

## Reproduction Assessment

Yes, source-level: a configured Google provider model row with `id: "google/gemini-*"` can match inline config, remain prefixed on the resolved model, and be passed to native Google transport as `models/google/...`. I did not run live Google API traffic or the reporter's exact sub-agent spawn payload.

## Solution Assessment

Yes. A provider-owned Google normalization and transport guard is the narrowest maintainable fix; generic same-provider stripping is riskier because valid proxy and provider-native nested IDs exist.
