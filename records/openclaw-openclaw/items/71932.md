## Summary

Keep open. Current main still has a credible source-level path where a configured Google inline model id like `google/gemini-3-flash-preview` can match the active Google provider but remain prefixed until the native Gemini URL builder emits `models/google/...`; overlapping PRs #41398 and #41517 should be consolidated rather than closing this issue.

## What This Changes

Fix Google Gemini API 404s when sub-agent spawn or session reset carries a provider-prefixed Google model id into the native Gemini transport.

## Best Possible Solution

Land one consolidated fix that strips only redundant active-provider prefixes before Google alias lookup and native Gemini URL construction, preserves cross-provider nested ids such as OpenRouter models, and adds regression coverage for sub-agent spawn, reset, and configured Google inline model entries.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure `models.providers.google` with `api: google-generative-ai`, a Gemini API `baseUrl`, and a model entry id like `google/gemini-3-flash-preview`, then resolve `provider=google` with the bare model through sub-agent or reset model selection; the resolved model can retain `google/...` and the transport will build `models/google/...`.

## Solution Assessment

No. Current main normalizes some stored overrides and bare Gemini aliases, but it does not normalize the preserved inline Google model id before the native transport; the safer fix is scoped provider-owned prefix stripping with tests that preserve wrapper-provider nested ids.
