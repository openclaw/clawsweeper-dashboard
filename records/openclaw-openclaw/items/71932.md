## Summary

Keep open: current main and the latest release still have a source-visible path where a configured Google model row like `google/gemini-*` can be retained and then sent to the native Google stream endpoint as `models/google/gemini-*`. The related prefix-fix PRs were closed unmerged, and no open PR currently closes this issue.

## What This Changes

Fix Google model references such as `google/gemini-3.1-flash-lite-preview` so sub-agents and session reset do not build invalid native Google API model paths that 404.

## Best Possible Solution

Land one narrow Google-owned normalization guard that removes only redundant native `google/` prefixes before Google model matching and native URL construction, while preserving proxy nested ids and adding sub-agent/reset regression coverage.

## Reproduction Assessment

Yes, source-level: a configured Google provider model row with `id: "google/gemini-*"` can match inline config, remain prefixed, and then be passed to the native Google transport as `models/google/...`. I did not run live Google traffic or the reporter's exact sub-agent spawn payload.

## Solution Assessment

Yes. A provider-owned Google normalization and transport guard is the narrowest maintainable fix; generic same-provider stripping is riskier because valid proxy and provider-native nested ids exist.
