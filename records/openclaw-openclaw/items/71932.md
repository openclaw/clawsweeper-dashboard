## Summary

Keep open. Current main still has a source-level path where a provider-owned Google model entry such as `google/gemini-3-flash-preview` or `google/gemini-3.1-flash-lite-preview` can be matched and preserved, then sent to the native Gemini transport as `models/google/...`; overlapping fix PRs #41398 and #41517 are open and should be consolidated rather than closing this as implemented.

## What This Changes

The issue reports Google Gemini API 404s when sub-agents or session reset use configured `google/gemini-3.1-flash-lite-preview` model refs that may be sent to the native Google API with an invalid provider prefix.

## Best Possible Solution

Land one narrow Google/provider-prefix fix that strips only redundant active-provider prefixes before Google model matching and native Gemini URL construction, preserves wrapper-provider nested ids, and adds regression coverage for sub-agent inheritance/reset plus configured Google model entries.

## Reproduction Assessment

Yes, by source inspection. Configure `models.providers.google` with `api: google-generative-ai`, a Gemini API base URL, and a configured model id like `google/gemini-3-flash-preview`; current main can preserve that id into the native Google transport where it becomes `models/google/...`.

## Solution Assessment

No. Current main is not the best solution because the configured inline-model path can still preserve the `google/` prefix; the safer fix is provider-scoped prefix stripping with tests that keep OpenRouter-style nested ids intact.
