## Summary

Keep open. Current main still has a credible native Google model-resolution path where a configured `google/<gemini-model>` id can be matched and preserved, then sent to the Gemini transport as `models/google/<gemini-model>`, which matches the reported 404 class. The issue also has fresh corroborating config evidence and overlapping open fix PRs, so it should be consolidated rather than closed.

## What This Changes

The issue asks OpenClaw to stop returning Google Generative AI 404s when sub-agents or session reset inherit provider-prefixed Gemini model ids such as `google/gemini-3.1-flash-lite-preview`.

## Best Possible Solution

Land one focused Google/model-resolution fix that removes redundant same-provider prefixes before native Google model matching and URL construction, keeps cross-provider wrapper ids intact, and adds regression coverage for configured Google models plus sub-agent spawn/reset inheritance.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure the Google provider with `api: google-generative-ai` and a model entry whose id is `google/gemini-3-flash-preview`, then resolve an inherited bare Google model through the native sub-agent or reset path and inspect that the current transport can build `models/google/...`.

## Solution Assessment

No. Current main is not the best solution because it still allows the redundant same-provider prefix to survive into native Google transport; the safer path is a scoped prefix-strip fix coordinated with #41398 or #41517.
