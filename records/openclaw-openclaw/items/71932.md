## Summary

Keep open: current main still has a source-visible Google provider path where configured `google/gemini-*` model ids can survive into the native Gemini request URL as `models/google/...`, and the earlier overlapping fix PRs were closed unmerged.

## What This Changes

The issue reports Google Gemini 404s when sub-agents or session reset use provider-prefixed Google model ids such as `google/gemini-3.1-flash-lite-preview`.

## Best Possible Solution

Land a Google-owned normalization fix that strips only redundant `google/` from Google native model ids before configured-model resolution and native transport URL construction, while preserving valid proxy/namespaced ids and adding sub-agent/reset coverage.

## Reproduction Assessment

Yes, source-reproducible: a configured Google provider model id like `google/gemini-*` can match inline config, remain prefixed in the resolved model, and be sent to native Google transport as `models/google/...`. I did not run live Google API traffic or a real `sessions_spawn` payload in this read-only review.

## Solution Assessment

Yes. A provider-scoped Google normalization and transport fix is the narrowest maintainable path; a generic same-provider strip would risk breaking valid nested ids for proxy or provider-local models.
