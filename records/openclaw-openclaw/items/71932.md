## Summary

Keep open. Current main still has a credible source-level path where a Google provider-owned model entry written as `google/<model>` can be matched and preserved, then passed to the native Google Generative AI URL builder as `models/google/<model>`, matching the reported 404 class and the corroborating follow-up config.

## What This Changes

The issue asks OpenClaw to fix Google Gemini model resolution so sub-agent spawn/reset paths do not hit 404s when using provider-prefixed Gemini model refs such as `google/gemini-3.1-flash-lite-preview`.

## Best Possible Solution

Keep this issue open and consolidate the Google prefix handling into one narrow fix that strips only redundant active-provider prefixes before Google model matching and native transport URL construction, while preserving valid nested model ids for wrapper providers and adding regression coverage for sub-agent spawn/reset inheritance.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure `models.providers.google` with `api: google-generative-ai`, `baseUrl: https://generativelanguage.googleapis.com/v1beta`, and a configured model id like `google/gemini-3-flash-preview`, then follow the native sub-agent/reset model resolution path into the Google transport URL builder.

## Solution Assessment

No. Current main is not the best solution because it still allows a redundant same-provider prefix to survive into native Google URL construction; the safer fix is a scoped same-provider prefix strip coordinated with #41398 or #41517.
