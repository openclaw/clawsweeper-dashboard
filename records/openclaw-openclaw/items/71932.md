## Summary

Keep open. Current main still has a credible source-level path where a Google inline model configured as `google/<model>` is matched, preserved as the runtime model id, and then sent through the native Google URL builder as `models/google/<model>`, matching the reported 404 class and the corroborating config comment.

## What This Changes

The issue reports Google Generative AI 404s for Gemini models during sub-agent spawn or session reset when a provider-prefixed Google model id reaches the native Google request path.

## Best Possible Solution

Land one consolidated fix that normalizes redundant same-provider Google prefixes before alias lookup and native Google URL construction, with regression coverage for configured inline Google models, sub-agent inheritance, and session reset.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure `models.providers.google` with `api: google-generative-ai`, `baseUrl: https://generativelanguage.googleapis.com/v1beta`, and a configured model id such as `google/gemini-3-flash-preview`, then resolve that model through a sub-agent or reset path into the native Google transport.

## Solution Assessment

No. Current main is not the best solution because it still lets the redundant provider prefix survive on configured inline Google models; the safer direction is a scoped same-provider prefix strip coordinated with the existing prefix-fix PRs.
