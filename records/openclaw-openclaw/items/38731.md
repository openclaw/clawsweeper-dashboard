## Summary

Keep this PR open. Current main still lacks the requested high-frequency cron advisory, and the linked issue remains open, but the PR is not safe to merge as-is because the gateway warning uses the logger API incorrectly and the guardrail is limited to add paths while edit/update paths can still create the same sub-30-minute schedule state.

## What This Changes

This PR adds non-blocking warnings for sub-30-minute `--every` cron schedules in CLI `cron add` and gateway `cron.add`.

## Best Possible Solution

Implement one shared high-frequency `every` schedule advisory that is used consistently by CLI add/edit and gateway add/update, keeps short intervals valid, preserves CLI JSON stdout, uses the current logger signature, and adds focused regression coverage for both CLI and gateway paths.

## Reproduction Assessment

Yes. Current-main source inspection gives a high-confidence reproduction path: create or edit a cron job with `--every 15m`, and the CLI/gateway code paths forward the valid schedule without any high-frequency warning.

## Solution Assessment

No. The PR addresses the right user problem but is not the best mergeable solution because it adds add-only checks, misses edit/update, and uses the gateway logger with reversed arguments; a shared helper plus add/edit and add/update coverage is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Use the gateway logger signature:** `src/gateway/server-methods/cron.ts:115-118`
  - body: `context.logGateway.warn` is typed as `(message, meta)`, but this call passes the metadata object first. That makes the changed path fail typechecking and can treat an object as the console message instead of logging the advisory.
  - confidence: 0.91
- **[P2] Warn when cron schedules are updated too:** `src/gateway/server-methods/cron.ts:114-119`
  - body: The PR only warns during `cron.add`, but `cron.update` accepts `patch.schedule`, so API callers can still change an existing job to `everyMs < 30m` with no gateway warning. Mirror the advisory on the update path when the patch carries a short `every` schedule.
  - confidence: 0.88
- **[P2] Share the CLI warning with cron edit:** `src/cli/cron-cli/register.cron-add.ts:131-135`
  - body: The CLI warning is added only to the add path, while `cron edit --every 15m` resolves a direct schedule and sends `cron.update`. Put the advisory behind the shared schedule resolution so add and edit behave consistently.
  - confidence: 0.86
