## Summary

Keep this PR open. Current main still lacks the requested high-frequency cron warning, but the PR is stale/dirty against main and remains incomplete: it only covers add paths and its gateway warning uses the logger API incorrectly.

## What This Changes

The PR adds non-blocking sub-30-minute `every` cron schedule warnings to CLI `cron add` and gateway `cron.add`.

## Best Possible Solution

Implement a shared high-frequency `every` schedule advisory used consistently by CLI add/edit and gateway add/update, keep short intervals valid, preserve machine-readable CLI stdout, use the current logger signature, add focused regression coverage, and include the required user-facing changelog entry.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: create or edit a cron job with `--every 15m`, or call gateway `cron.add`/`cron.update` with `{ kind: "every", everyMs: 900000 }`; the schedule is valid and no high-frequency warning is emitted.

## Solution Assessment

No. The PR addresses the right user problem, but this patch is not the best mergeable solution because it is add-only, misses edit/update parity, uses `context.logGateway.warn` with the wrong argument order, and is dirty against current main.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Use the gateway logger signature:** `src/gateway/server-methods/cron.ts:115-118`
  - body: `context.logGateway.warn` is typed as `(message, meta)`, but this call passes the metadata object first. That makes the changed path fail typechecking and can treat an object as the console message instead of logging the advisory.
  - confidence: 0.91
- **[P2] Warn when cron schedules are updated too:** `src/gateway/server-methods/cron.ts:114-119`
  - body: The PR only warns during `cron.add`, but `cron.update` accepts `patch.schedule`, so API callers can still change an existing job to `everyMs < 30m` with no gateway warning. Mirror the advisory on the update path when the patch carries a short `every` schedule.
  - confidence: 0.88
- **[P2] Share the CLI warning with cron edit:** `src/cli/cron-cli/register.cron-add.ts:131-135`
  - body: The CLI warning is added only to the add path, while `cron edit --every 15m` resolves a direct schedule and sends `cron.update`. Put the advisory behind shared schedule resolution or otherwise cover both add and edit consistently.
  - confidence: 0.86
