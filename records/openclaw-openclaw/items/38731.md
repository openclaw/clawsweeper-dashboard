## Summary

Keep open. Current main still lacks the requested high-frequency cron advisory, and this PR is paired with the still-open cron guardrail report, but the patch is not mergeable as-is because it covers only create/add paths, uses the gateway logger with reversed arguments, and targets a pre-refactor CLI schedule parser.

## What This Changes

The PR adds non-blocking sub-30-minute `every` cron schedule warnings to CLI `cron add` and gateway `cron.add`.

## Best Possible Solution

Implement one shared high-frequency `every` schedule advisory that is used by CLI add/edit and gateway add/update, keep short intervals valid, use the current logger signature, add focused regression coverage, and include the required user-facing changelog entry.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: create or edit a cron job with `--every 15m`, or call gateway `cron.add`/`cron.update` with `{ kind: "every", everyMs: 900000 }`; current main accepts those schedules and does not emit the requested advisory.

## Solution Assessment

No. The PR targets the right guardrail, but the best fix should be rebased onto current main's shared schedule helper and should cover add/edit plus add/update parity instead of adding an add-only check with a reversed gateway logger call.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Use the gateway logger signature:** `src/gateway/server-methods/cron.ts:115-118`
  - body: `context.logGateway.warn` is typed as `(message, meta)`, but this call passes the metadata object first. That makes the changed path fail typechecking and can treat an object as the log message instead of recording the advisory.
  - confidence: 0.92
- **[P2] Warn when cron schedules are updated too:** `src/gateway/server-methods/cron.ts:114-119`
  - body: The PR only warns during `cron.add`, but `cron.update` accepts `patch.schedule`, so API callers can still change an existing job to `everyMs < 30m` with no gateway warning. Mirror the advisory on the update path when the patch carries a short `every` schedule.
  - confidence: 0.88
- **[P2] Share the CLI warning with cron edit:** `src/cli/cron-cli/register.cron-add.ts:131-135`
  - body: The CLI warning is added only to the add path, while `cron edit --every 15m` resolves a direct schedule and sends `cron.update`. Put the advisory behind the shared schedule resolution or otherwise cover both add and edit consistently.
  - confidence: 0.87
