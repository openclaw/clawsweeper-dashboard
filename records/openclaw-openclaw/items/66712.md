## Summary

Keep open. Current main still lacks the scheduled-slot delivery guard, and the latest PR head addresses the earlier threadId collision concern, but this user-facing cron delivery change is still missing the required changelog entry and some public checks were still running at review time.

## What This Changes

The PR adds a process-local scheduled-slot cron delivery dedup cache, resets it in the double-announce tests, adds same-slot/different-runStartedAt and different-threadId regression coverage, and propagates `deliveryAttempted` from startup catch-up outcomes.

## Best Possible Solution

Add the required single-line changelog entry, let the pending checks finish, then proceed with normal maintainer review/merge for the same-process slot guard while leaving distributed multi-gateway coordination to the open HA tracking issue.

## Reproduction Assessment

Yes. The main bug is reproducible by calling `dispatchCronDelivery` twice for the same job, same `job.state.nextRunAtMs`, and same delivery target but different `runStartedAt`; the existing execution key differs, so current main sends twice. The PR adds that regression test and a second test for different thread IDs.

## Solution Assessment

Yes for the same-process problem after the changelog follow-up. The slot key complements the existing execution key and now includes threadId, while persistent cross-process dedupe is a separate product/architecture problem tracked elsewhere.
