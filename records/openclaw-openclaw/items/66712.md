## Summary

Keep this PR open, but do not merge it as-is. Current main still lacks the scheduled-slot delivery guard and the PR addresses a real same-slot/different-runStartedAt replay gap, but the proposed slot key omits `threadId`, so it can collapse distinct topic/thread destinations for the same chat slot.

## What This Changes

The PR adds an in-memory scheduled-slot cron delivery dedup cache, resets it in double-announce tests, adds a same-slot/different-runStartedAt regression test, and propagates `deliveryAttempted` from startup catch-up outcomes.

## Best Possible Solution

Keep the slot-level guard, but make the slot destination identity match the existing direct-delivery identity by including normalized `threadId`, then add a regression test for two deliveries to the same chat/account/slot with different thread IDs before merging.

## Reproduction Assessment

Yes. The main bug is reproducible with two `dispatchCronDelivery` calls for the same job and `job.state.nextRunAtMs` but different `runStartedAt`; the PR's new test covers that path. The review finding is reproducible by varying only `resolvedDelivery.threadId` between those two calls after the first slot delivery is recorded.

## Solution Assessment

No, not as-is. The scheduled-slot guard is the right narrow approach, but the key should include `stringifyRouteThreadId(delivery.threadId)` so topic/thread destinations are not collapsed.
