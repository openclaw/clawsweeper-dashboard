## Summary

Keep open. Current main still only has execution-scoped direct cron delivery idempotency, while this PR adds the missing scheduled-slot dimension and startup catch-up outcome propagation; the branch is currently dirty/unmergeable rather than obsolete.

## What This Changes

Adds a process-local scheduled-slot cron delivery dedup cache with regression coverage, a changelog entry, and `deliveryAttempted` propagation from startup catch-up outcomes.

## Best Possible Solution

Rebase the branch onto current main, keep the process-local slot guard plus tests and changelog entry, then run the targeted cron dispatch test and changed gate before merge; leave distributed multi-gateway coordination to #13613.

## Reproduction Assessment

Yes, source-level. Calling `dispatchCronDelivery` twice for the same job, same `job.state.nextRunAtMs`, and same destination but different `runStartedAt` values gives different execution keys on current main, so both direct sends can proceed.

## Solution Assessment

Yes for the same-process scheduled-slot replay gap. The PR complements the existing execution key with a slot key and preserves threadId; persistent cross-process dedupe is a separate HA design tracked by #13613.
