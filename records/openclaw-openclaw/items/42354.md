## Summary

Keep open. Current main contains a partial Feishu WebSocket repair, but source inspection plus the pinned Lark SDK contract show the monitor still treats `start()` resolution as readiness, does not observe SDK `onReady`/`onError` callback failures, and still leaves the Lark international WebSocket-vs-webhook path ambiguous.

## What This Changes

The issue reports that Feishu/Lark WebSocket long connection mode fails with `1000040351 system busy` and `PingInterval` errors, preventing the Lark console from detecting the app connection.

## Best Possible Solution

Finish the Feishu-owned WebSocket lifecycle by using SDK-supported readiness/failure callbacks or the SDK `LarkChannel` readiness wrapper, add a bounded initial handshake timeout and retry path for terminal callback failures, and document or enforce the supported Lark international transport mode.

## Reproduction Assessment

Yes for the remaining code-level failure mode: a mocked SDK client whose `start()` returns before a later `onError` would still be treated as started by current main. Live reproduction against Feishu/Lark is unclear because this review did not have usable channel credentials.

## Solution Assessment

No, current main is only a partial solution. The maintainable fix should be a Feishu plugin lifecycle wrapper that observes SDK readiness/failure and a clear Lark international setup/docs decision, rather than relying on an extra constructor property the pinned SDK does not consume.
