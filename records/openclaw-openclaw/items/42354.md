Current main and shipped v2026.5.7 cover the reported Feishu/Lark WebSocket failure: the Feishu client now carries heartbeat defaults, resolves the Lark domain explicitly, and the transport recreates the WebSocket client after SDK retry exhaustion.

## What This Changes

The issue asks OpenClaw to make Feishu/Lark WebSocket long-connection mode establish reliably instead of looping with `1000040351` and `PingInterval` errors.

## Best Possible Solution

Keep the shipped Feishu plugin client/transport implementation and use the separate reconnect-backoff follow-up at https://github.com/openclaw/openclaw/pull/73998 only for broader tuning outside this startup failure.

## Reproduction Assessment

Yes, at source level. The reported failure maps to the Feishu `WSClient` construction/domain/reconnect path, and current tests now exercise the heartbeat defaults plus terminal SDK retry-exhaustion recovery; I did not run a live Feishu/Lark tenant connection.

## Solution Assessment

Yes. The fix stays inside the Feishu plugin client and transport, uses the SDK callback contract, keeps domain selection in the existing `channels.feishu.domain` config, and avoids a core channel special case.
