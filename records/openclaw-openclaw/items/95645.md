## Summary

This issue should stay open: current main and the latest release still route `ChatType: "channel"` through the group prompt context, while the trusted metadata block reports `chat_type: "channel"`. The report is narrow, source-reproducible, and not covered by a merged fix or canonical replacement.

## What This Changes

The issue asks OpenClaw to stop describing Mattermost public/private channel turns as a “group chat” when the authoritative `chat_type` metadata distinguishes `channel` from `group`.

## Best Possible Solution

Make the shared group/channel prompt context chat-type-aware or neutral so `channel` turns do not contradict trusted metadata, while leaving the separate Mattermost target-inference work to https://github.com/openclaw/openclaw/issues/95646.

## Reproduction Assessment

Yes, from source: a `ChatType: "channel"` turn enters `buildGroupChatContext`, which still emits “group chat,” while trusted metadata emits `chat_type: "channel"`. I did not run a live Mattermost channel instance.

## Solution Assessment

Yes, the best fix is a narrow prompt-context repair: choose wording from the actual `ChatType` or use neutral shared-chat language that cannot be confused with the enum.
