## Summary

Keep open: the reported sentinel-token tool failure is plausible and partly supported, but this branch is not merge-ready because it mixes in a sensitive Telegram export command, mutates URLs inside the shared SSRF guard, rewrites shell text too broadly, and does not wire the file-path sanitizer onto current main’s file-tool path.

## What This Changes

The PR adds sentinel-token sanitizers for web_fetch URLs, guarded-fetch URLs, bash exec commands, and file-tool path args, plus a bundled portable-context Telegram command that exports local USER/MEMORY/session context.

## Best Possible Solution

Narrow this PR to a current-main-compatible sanitizer at the tool boundary: keep URL cleanup in web_fetch only, constrain bash/file-path cleanup to observed leading leaked prefixes or fail closed, wire file paths through the active read/write/edit wrappers, and move portable-context to a separate ClawHub/plugin security review if still desired.

## Reproduction Assessment

No for a full live reproduction in this review; yes at source level for the raw failure path. Current main passes raw exec commands and raw web_fetch/guarded-fetch URLs into the shell/URL parser, while the PR body supplies redacted live Gemma logs for bash and file-path leaks.

## Solution Assessment

No. The narrow fix should live at the specific tool input boundary and be rebased onto current wrappers; the shared SSRF guard mutation and bundled Telegram export are broader product/security changes than this bug fix needs.
