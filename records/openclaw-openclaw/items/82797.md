## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Constrain command cleanup to leaked prefixes:** `src/agents/tools/sanitize-command.ts:29`
  - body: This replacement runs over the whole shell string, so a valid command such as `printf '<|im_start|>' > token.txt` is changed before bash sees it. The fix is meant to recover malformed tool-call prefixes; keep cleanup to the leading leaked prefix or another unambiguous argument boundary.
  - confidence: 0.86
- **[P1] Keep sanitizer out of shared guarded fetch:** `src/infra/net/fetch-guard.ts:174`
  - body: `fetchWithSsrFGuard` is used by provider transports, media downloads, auth/API clients, downloads, plugins, and web tools. Sanitizing here can turn an invalid sentinel-prefixed URL into a real outbound request while preserving caller headers, proxy mode, and SSRF policy; apply URL recovery at the `web_fetch` argument boundary instead.
  - confidence: 0.9
- **[P1] Split the context export out of this PR:** `extensions/portable-context/index.ts:221-224`
  - body: This registers `/export_context` as a bundled command that reads USER.md, MEMORY.md, and recent session JSONL from hard-coded local agent paths and returns a Telegram document. That is a separate sensitive data-export feature with auth/product review needs, so it should not ship inside a sanitizer fix.
  - confidence: 0.92
- **[P2] Document the new loop-guard setting:** `src/config/zod-schema.agent-runtime.ts:465`
  - body: The PR adds `genericRepeatBlockThreshold` to the config schema/types and new environment knobs, but the docs/help/label surfaces for `tools.loopDetection` are not updated. Keep the public config contract aligned if this loop guard remains in the branch.
  - confidence: 0.78
