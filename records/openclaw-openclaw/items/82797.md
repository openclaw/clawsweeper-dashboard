## Summary

## What This Changes

The PR adds sanitizer helpers for bash, web-fetch, file/search tool arguments, changes generic tool-loop blocking, and adds a bundled `/export_context` portable-context command.

## Best Possible Solution

Land a narrow sanitizer at the affected tool boundaries with explicit logging and regression tests, keep shared guarded fetch strict, and split portable-context plus loop-guard policy into separate reviewed PRs.

## Reproduction Assessment

Yes for the source-level failure shape: current main forwards bash command strings raw and rejects sentinel-prefixed URLs through `new URL(params.url)`, so leaked sentinel prefixes can break tool execution. I did not reproduce a live Gemma/Telegram session in this read-only review.

## Solution Assessment

No. A focused boundary sanitizer is plausible, but this PR is not the best shape because it also adds a sensitive new command, moves URL recovery into shared guarded fetch, and changes loop-blocking policy.

- **[P1] Split the context export out of this fix:** `extensions/portable-context/index.ts:222-224`
  - body: This registers a new bundled `/export_context` command that reads USER.md, MEMORY.md, recent session JSONL, calls a hard-coded local vLLM endpoint, and sends the result as a Telegram document.   - confidence: 0.93
- **[P1] Keep URL cleanup out of guarded fetch:** `src/infra/net/fetch-guard.ts:174`
  - body: `fetchWithSsrFGuard` is a shared network primitive, so sanitizing `params.url` here can convert malformed input that currently fails closed into an outbound request using the caller's headers, proxy mode, and SSRF policy. Keep recovery at the `web_fetch` tool boundary where the model-facing leak occurs and preserve strict validation for other guarded-fetch callers.
  - confidence: 0.9
- **[P1] Preserve literal tokenizer strings in shell commands:** `src/agents/tools/sanitize-command.ts:29`
  - body: The sanitizer removes `<|...|>` patterns globally, so commands that intentionally inspect tokenizer/control tokens, such as grepping for `<|im_start|>`, will be rewritten before bash sees them. The fix should be prefix-scoped to observed leaked sentinels or otherwise require an explicit recovery path rather than mutating literal command content.
  - confidence: 0.86
- **[P2] Require no-progress evidence before blocking repeats:** `src/agents/tool-loop-detection.ts:593-598`
  - body: This blocks a generic tool call after five identical arguments in a row even when previous results have not proven the call is making no progress. Current main only hard-blocks generic repeats after identical no-progress outcomes, so loop-detection-enabled setups can lose legitimate repeated reads/status checks under this change.
  - confidence: 0.8
