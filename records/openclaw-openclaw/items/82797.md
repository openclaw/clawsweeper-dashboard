## Summary

Keep this PR open, but do not merge it as-is. The reported Gemma sentinel-token failure is plausible and useful to fix, but the patch rewrites a shared SSRF/network primitive and strips token-like shell content globally, so it needs a narrower revision plus complete web-fetch proof.

## What This Changes

The PR adds sentinel-token sanitizers for `web_fetch` URL inputs and bash exec command inputs, wires them into `web_fetch`, the shared SSRF fetch guard, and bash exec, and adds sanitizer unit tests.

## Best Possible Solution

Keep sanitization at the LLM tool-argument boundaries, make both sanitizers leading-prefix-only, leave `fetchWithSsrFGuard` strict, and add regressions for invalid guard URLs plus literal shell-token content.

## Reproduction Assessment

Yes, at source level: current main passes bash exec commands into the execution path and validates `web_fetch` URLs with `new URL`, so leading sentinel text can produce the reported shell or URL failure. I did not establish a live Gemma reproduction in this review.

## Solution Assessment

No. Sanitizing malformed LLM tool arguments is a reasonable fix direction, but this patch applies URL sanitization inside a shared SSRF primitive and rewrites token-looking shell content beyond the malformed leading prefix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep URL sanitizing out of the shared fetch guard:** `src/infra/net/fetch-guard.ts:171`
  - body: `fetchWithSsrFGuard` is an SDK-facing shared primitive used by plugin and provider callers with caller-supplied `init`. Rewriting `params.url` here means malformed input such as `junkhttps://attacker.example` can become a real outbound request instead of failing validation, potentially carrying headers or bodies to the embedded URL. Keep this repair at the `web_fetch` tool boundary and leave the generic guard strict.
  - confidence: 0.88
- **[P2] Limit command sanitizing to leading leaked prefixes:** `src/agents/tools/sanitize-command.ts:27-32`
  - body: These replacements remove token-looking text anywhere in the command. A legitimate command like `rg '<|assistant|>' docs` would be changed to search for an empty string, producing different and potentially much broader output. Restrict the repair to the malformed leading leakage pattern and preserve quoted or literal command content.
  - confidence: 0.84
- **[P3] Log URL sanitizer changes as promised:** `src/agents/tools/web-fetch.ts:507`
  - body: The PR body says URL sanitizer changes emit `webfetch.sanitize_special_tokens`, but the `web_fetch` path silently mutates `params.url`. If this sanitizer remains, log only when the URL actually changes so operators can monitor the same leak signal as bash exec.
  - confidence: 0.78
