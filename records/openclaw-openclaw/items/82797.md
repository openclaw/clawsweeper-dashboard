## Summary

Keep open: the reported Gemma/sentinel-token failure is plausible and the PR body now includes some live bash sanitizer proof, but the patch still has blocking correctness and security issues in shared fetch and command rewriting paths.

## What This Changes

The PR adds URL and shell-command sanitizers for leaked `<|...|>`-style tokens, wires them into `web_fetch`, the shared SSRF fetch guard, and bash exec, and adds sanitizer unit tests.

## Best Possible Solution

Keep sanitization at the `web_fetch` and bash tool-argument boundaries, make it leading-prefix-only, preserve strict shared fetch-guard validation, and add regressions for invalid guard URLs plus literal shell-token content.

## Reproduction Assessment

Yes, at source level: current main passes bash exec commands through to the execution path and validates `web_fetch` URLs with `new URL`, so leading sentinel text can produce the reported shell or URL failure. I did not reproduce a live Gemma session in this review.

## Solution Assessment

No. Sanitizing LLM tool arguments is a plausible fix direction, but this patch changes a shared network primitive and rewrites token-looking shell content beyond the malformed leading prefix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep URL sanitizing out of the shared fetch guard:** `src/infra/net/fetch-guard.ts:171`
  - body: `fetchWithSsrFGuard` is an SDK-facing shared primitive used by plugins and providers with caller-supplied `init`. Rewriting `params.url` here means malformed input such as `junkhttps://attacker.example` becomes a real outbound request instead of failing validation, potentially carrying headers or bodies to an embedded URL. Keep this repair at the `web_fetch` tool boundary and leave the generic guard strict.
  - confidence: 0.88
- **[P2] Limit command sanitizing to malformed leading prefixes:** `src/agents/tools/sanitize-command.ts:27-32`
  - body: These replacements remove token-looking text anywhere in the command. A legitimate command like `rg '<|assistant|>' docs` would be changed to search for an empty string, producing different and potentially much broader output. Restrict the repair to the malformed leading leakage pattern and preserve quoted/literal command content.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The patch introduces a concrete network-boundary regression by sanitizing URLs inside the shared SSRF fetch guard.

Concerns:

- **[medium] Shared fetch guard can redirect malformed input to a real URL:** `src/infra/net/fetch-guard.ts:171`
  - body: Because `sanitizeUrlInput` selects the first embedded `http(s)` scheme and `fetchWithSsrFGuard` forwards caller-provided request init, malformed external input can become an outbound request to a different destination instead of being rejected.
