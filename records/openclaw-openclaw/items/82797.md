## Summary

Keep open: the reported sanitizer problem is plausible and useful, but this PR is not merge-ready because it mixes the fix with a new Telegram context-export plugin, rewrites valid inputs in security-sensitive paths, conflicts with current main, and still lacks complete real behavior proof.

## What This Changes

The PR adds sanitizer helpers and tests for leaked LLM sentinel tokens in bash exec, web_fetch/guarded-fetch URLs, and file/search tool arguments, plus a bundled Telegram `/export_context` plugin.

## Best Possible Solution

Land a narrow sanitizer-only change that preserves literal command compatibility, keeps URL cleanup at the web_fetch tool boundary, rebases file/search argument cleanup onto current main wrappers, and moves `/export_context` to a separate ClawHub or security-reviewed plugin path with live Telegram proof.

## Reproduction Assessment

Yes for the source-level failure shape: current main passes raw exec commands and web_fetch URLs into bash/URL parsing, and the PR body supplies redacted Gemma runtime logs for bash and file-path leaks. I did not independently reproduce the live model leak or the web_fetch path.

## Solution Assessment

No. The sanitizer direction is useful, but this implementation is too broad, stale against current main, and bundled with an unrelated sensitive Telegram export feature.
