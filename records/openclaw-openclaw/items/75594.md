## Summary

Keep open. The docs-only direction is valid for the linked sandbox setup confusion, but this draft PR has a protected maintainer label and the new npm-install Docker build fallback should not ship with a mutable Debian base image when the checked-in sandbox Dockerfile is digest-pinned.

## What This Changes

The PR updates sandboxing-related docs and the changelog to say sandbox setup scripts require a source checkout and to add npm-install fallback guidance for building the default sandbox image.

## Best Possible Solution

Keep the source-checkout clarification, but change the npm-install fallback so it preserves the same supply-chain posture as `Dockerfile.sandbox`, preferably by using the exact digest-pinned `FROM` reference or by directing users to an equivalent pinned build path. After that, the linked docs/changelog updates are a narrow and maintainable fix for the reported setup confusion.

## Reproduction Assessment

Yes. Current main documents source-only sandbox setup scripts, `package.json` does not publish those scripts in the npm package, and the runtime missing-image hint also names `scripts/sandbox-setup.sh`. That is enough source-level evidence for the npm global install failure path described in #75485.

## Solution Assessment

No, not as currently written. The docs-only scope is the right narrow fix, but the inline Dockerfile should match the repo's pinned sandbox base image policy before this is the best way to solve the issue.
