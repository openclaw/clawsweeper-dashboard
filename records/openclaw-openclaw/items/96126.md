The issue asks status outputs to distinguish the configured/base model from the active fallback model and show why the fallback was selected.

Resolve this by landing or replacing the focused status fix so both `openclaw status` and `session_status` show configured-vs-active fallback model state without mislabeling runtime-only snapshots.

Yes. Source inspection shows `modelOverrideSource: "auto"` is excluded from model-selection reasons, and the CLI/session status renderers then suppress the requested configured-vs-active fallback explanation.

## Solution Assessment

Yes at the issue level. The maintainable fix belongs in the status summary and status-message renderers, with care to preserve runtime-only snapshot behavior and cover both CLI status surfaces.
