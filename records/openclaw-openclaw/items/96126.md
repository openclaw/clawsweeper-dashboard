## Summary

Keep open: current main has the raw fields needed to compare configured and selected models, but auto fallback rows still intentionally suppress `modelSelectionReason`, and the human status renderers only split configured/selected output for pinned user overrides.

## What This Changes

Request to make `session_status` and `openclaw status` distinguish the configured/base model from the active fallback model and expose a safe fallback reason.

## Best Possible Solution

Teach the shared status summary and status card to classify auto fallback separately from user-pinned overrides, render configured/base and active/fallback labels with the persisted safe reason, and keep the single `Model:` line when no fallback is active.

## Reproduction Assessment

Yes, source-reproducible: current tests build an auto fallback entry with a configured/selected mismatch and assert `modelSelectionReason` stays null, while the formatter filters out null reasons. I did not run a live gateway fallback because this review is read-only.

## Solution Assessment

Yes, the best path is a focused additive status-display change that reuses existing fallback notice/provenance state instead of changing model routing or adding configuration.
