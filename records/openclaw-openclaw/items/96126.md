The issue asks status output to distinguish the configured base model from the active fallback model and explain why fallback was selected.

Resolve this by reviewing the linked PR or a narrow replacement that updates both `openclaw status` and `session_status` without warning on runtime-only snapshots.

Unclear at the issue level. The linked PR targets the right files, but the best fix must cover both human status surfaces and avoid misleading runtime-only rows.
