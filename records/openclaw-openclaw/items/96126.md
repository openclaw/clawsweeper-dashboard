## Summary

Keep open: current main still hides the auto-fallback configured-vs-active model explanation in status output, and the linked closing PR is open and unmerged rather than an implemented fix.

## What This Changes

Request that status output distinguish the configured/base model from the active fallback model and expose the fallback reason when model failover is active.

## Best Possible Solution

Land or replace the linked focused status fix so both `openclaw status` and `session_status` show configured primary, active fallback, and reason while keeping runtime-only snapshots quiet.

## Reproduction Assessment

Yes. Source inspection shows current main leaves `modelSelectionReason` null for auto fallback overrides, and both status renderers only explain configured-vs-selected mismatches through the user-pinned or reason-present paths.

## Solution Assessment

Unclear at the issue level. The linked PR targets the right status files, but the best fix must cover both command status and `session_status` while preserving the runtime-only no-warning invariant.
