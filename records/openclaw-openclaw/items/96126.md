## Summary

Keep open: current main and latest release still hide auto-fallback model selection from the status mismatch explanation, and the open linked PR is the candidate fix path rather than a reason to close this issue before merge.

## What This Changes

The issue asks for `openclaw status` and `session_status` to show configured/base model, active fallback model, and fallback reason distinctly.

## Best Possible Solution

Keep the issue open until the linked PR or a replacement lands a focused fix for both `openclaw status` and `session_status` with configured-vs-active fallback labeling and proof.

## Reproduction Assessment

Yes, by source inspection: auto fallback entries use `modelOverrideSource: "auto"`, while current status summary and `session_status` mismatch copy are gated on user-pinned model selections. I did not run a live fallback scenario in this read-only review.

## Solution Assessment

Unclear for the open PR implementation, but the issue direction is right: status renderers should distinguish configured primary from auto-selected fallback without treating fallback state as a user pin.
