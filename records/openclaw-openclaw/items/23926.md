## Label Justifications

- `P1`: Third-party ClawHub skill installs and updates can currently bypass enforceable local policy hooks before untrusted skill content lands on disk.
- `impact:security`: The issue is about install-time security boundaries, scanner enforcement, and auditability for untrusted third-party skills.

## Summary

Keep open: current main and the latest release still only partially cover the requested lifecycle policy. The existing `before_install` path can block some skill installer flows, but native ClawHub skill install/update still disables scanning and the hook surface has no post-install audit observer.

## What This Changes

The issue requests blocking pre-install and observational post-install lifecycle hooks for skill installation so scanners and audit tools can enforce policy around third-party skills.

## Best Possible Solution

Define a backward-compatible skill install lifecycle API covering native ClawHub install/update, gateway/UI ClawHub installs, a blocking pre-copy policy point, and a post-install audit observer with source, target, timestamp, result, and outcome metadata.

## Reproduction Assessment

Yes. This is source-reproducible as a feature gap: current main shows native ClawHub install/update passing `scan: false`, while the hook surface exposes `before_install` but no post-install outcome observer.

## Solution Assessment

Unclear. Extending `before_install` or adding explicit skill lifecycle hooks is plausible, but maintainers need to choose the public API shape, blocking semantics, audit payload, UI behavior, and OpenClaw-versus-ClawHub ownership boundary first.
