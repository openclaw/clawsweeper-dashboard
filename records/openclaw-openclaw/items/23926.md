## Label Justifications

- `P1`: Third-party ClawHub skill installs and updates can currently bypass enforceable local policy hooks before untrusted skill content lands on disk.
- `impact:security`: The issue is about install-time security boundaries, scanner enforcement, and auditability for untrusted third-party skills.

## Summary

Keep open: current main still lacks the requested lifecycle coverage. Native ClawHub skill install/update continues to bypass the scanner-backed install policy path, and the plugin hook surface still has no post-install audit observer, so this is an active security/product API design item rather than a closeable stale or duplicate report.

## What This Changes

The issue requests blocking skill pre-install and observational post-install lifecycle hooks so plugins can scan, warn, block, or audit skill installs.

## Best Possible Solution

Define a documented, backward-compatible skill install lifecycle API covering native ClawHub install/update, gateway/UI ClawHub installs, a blocking pre-copy policy point, and a post-install audit observer with source, target, timestamp, result, and outcome metadata.

## Reproduction Assessment

Yes, as a source-verifiable feature gap rather than a runtime bug: current main shows native ClawHub install/update calling the archive installer with scanning disabled while the hook surface exposes only `before_install` and no post-install outcome observer.

## Solution Assessment

Unclear. Extending `before_install` or adding explicit skill lifecycle hooks is plausible, but maintainers need to choose the public API shape, blocking semantics, audit payload, UI behavior, and OpenClaw-versus-ClawHub ownership boundary first.
