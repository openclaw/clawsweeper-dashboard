## Label Justifications

- `P1`: Third-party ClawHub skill installs can currently bypass enforceable local install-policy hooks before untrusted skill content lands on disk.
- `impact:security`: The issue is about security boundaries for untrusted skill installation, scanner enforcement, and auditability.

## Summary

Keep open. Current main and v2026.5.12 still only partially cover the requested lifecycle policy: scanner-backed dependency/upload installs can run `before_install`, but native ClawHub install/update still passes through an unscanned copy path and the hook surface still has no post-install audit observer.

## What This Changes

Request adding blocking skill pre-install and observational post-install hook events so plugins can enforce scan policy and record audits around skill installation/update.

## Best Possible Solution

Define a documented, backward-compatible skill install lifecycle API that covers native ClawHub install/update, gateway/UI ClawHub installs, a blocking pre-copy policy point, and a post-install observer with source, target, timestamp, result, and outcome metadata.

## Reproduction Assessment

Yes, as a source-verifiable feature gap rather than a runtime bug: current main shows native ClawHub install/update calling the archive installer with `scan: false`, while the hook surface exposes only `before_install` and no post-install outcome observer.

## Solution Assessment

Unclear. Extending `before_install` or adding explicit skill lifecycle hooks is plausible, but maintainers need to choose the public API shape, audit payload, UI behavior, and OpenClaw-versus-ClawHub ownership boundary first.
