## Label Justifications

- `P1`: Third-party skill installation is a live supply-chain security workflow, and the missing lifecycle/audit hook blocks standard scanner and enterprise review integrations.
- `impact:security`: The issue is about scanning, blocking, and auditing untrusted skills before and after they are accepted onto a user's machine.

## Summary

Keep open: current main covers the blocking pre-install side through `security.installPolicy` and `before_install`, but the requested post-install/audit observer is still absent and needs product/security review before becoming a public plugin contract.

## What This Changes

The issue requests blocking `skill:pre-install` and observational `skill:post-install` hook events for skill installation and update flows so scanners can block or audit third-party skills.

## Best Possible Solution

Keep `security.installPolicy` and `before_install` as the blocking policy seams, then add a documented non-blocking skill install/update completion hook with minimal redacted metadata and explicit observer-failure semantics.

## Reproduction Assessment

Yes, at source level. Current main shows pre-install blocking policy exists, while the hook registry and install/update success paths still have no post-install observer event.

## Solution Assessment

Partly. Reusing `security.installPolicy` and `before_install` is the narrowest answer for blocking, but the remaining audit capability should be a carefully specified non-blocking completion hook rather than the original parallel event names unchanged.
