## Label Justifications

- `P1`: Third-party skill installation is a live supply-chain security boundary, and the remaining audit lifecycle gap affects security tooling and enterprise enforcement.
- `impact:security`: The issue concerns blocking untrusted skill installation and exposing reliable security audit events around installed code.

## Summary

Current main now routes native ClawHub skill installs and updates through the shared blocking `before_install` policy path, resolving the original pre-install gap. The requested observational post-install/audit hook remains absent from both the typed plugin surface and canonical lifecycle, leaving meaningful security lifecycle work that requires maintainer product and security decisions.

## What This Changes

Add blocking pre-install and observational post-install hook events across supported skill install and update paths.

## Best Possible Solution

Keep `before_install` as the single generic blocking policy seam for every OpenClaw-managed skill install/update, and add one documented non-blocking completion event at the canonical committed lifecycle boundary with typed source, target, install/update outcome, timestamps, redaction rules, and explicit observer-failure semantics; coordinate standalone ClawHub CLI coverage with its owning repository.

## Reproduction Assessment

Yes at source level. The canonical archive installer and typed hook registry show that blocking pre-install policy now runs, while the successful ClawHub lifecycle has no public post-install observer event.

## Solution Assessment

Partly. Reusing the existing generic `before_install` hook is the narrowest maintainable solution for blocking; the remaining post-install capability should be a carefully specified observational extension rather than adding the originally proposed parallel event family unchanged.
