## Label Justifications

- `P1`: Third-party ClawHub skill installs can currently bypass enforceable local policy hooks before untrusted skill content lands on disk.
- `impact:security`: The issue is about security boundaries for untrusted skill installation, scanner enforcement, and auditability.

## Summary

Keep open. Current main and the latest release still only partially cover the requested lifecycle policy: scanner-backed dependency/upload installs can run `before_install`, but native ClawHub install/update still uses an unscanned copy path and the public hook surface still lacks a post-install audit observer.

## What This Changes

The issue requests install lifecycle hook events so plugins can block suspicious skill installs before copy and observe post-install audit metadata.

## Best Possible Solution

Define a documented, backward-compatible skill install lifecycle API that covers native ClawHub install/update, gateway/UI ClawHub installs, a blocking pre-copy policy point, and a post-install audit observer with source, target, timestamp, result, and outcome metadata.

## Reproduction Assessment

Yes, as a source-verifiable feature gap rather than a runtime bug: current main shows native ClawHub install/update calling the archive installer with scanning disabled while the hook surface exposes only `before_install` and no post-install outcome observer.

## Solution Assessment

Unclear. Extending `before_install` or adding explicit skill lifecycle hooks is plausible, but maintainers need to choose the public API shape, blocking semantics, audit payload, UI behavior, and OpenClaw-versus-ClawHub ownership boundary first.
