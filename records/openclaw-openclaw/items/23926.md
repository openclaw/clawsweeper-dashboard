## Label Justifications

- `P1`: Third-party ClawHub skill installs and updates can still bypass enforceable local policy hooks before untrusted skill content lands on disk.
- `impact:security`: The issue is about install-time security boundaries, scanner enforcement, and auditability for untrusted third-party skills.

## Summary

Keep open: current main still only partially covers install-time skill policy. Gateway-managed skill dependency and upload installs can use the generic blocking `before_install` hook, but native ClawHub install/update still bypasses that scan path before copying skill files and the hook catalog has no post-install observer with audit/outcome metadata.

## What This Changes

The request is to add blocking `skill:pre-install` and observational `skill:post-install` hook events so plugins can scan, warn, block, or audit skill installs across ClawHub, sync, UI, and plugin-shipped flows.

## Best Possible Solution

Define a backward-compatible skill install lifecycle contract that covers native ClawHub install/update plus Gateway/UI installs with a blocking pre-copy policy point and a post-install audit observer carrying source, target, timestamp, result, and outcome metadata.

## Reproduction Assessment

Yes. This is source-reproducible as a feature gap: current main shows native ClawHub install passing `scan: false`, while the hook surface exposes only `before_install` and no post-install outcome observer.

## Solution Assessment

Unclear. Extending `before_install` may be the narrowest path, but maintainers still need to choose the public lifecycle API, blocking semantics, audit payload, UI behavior, and OpenClaw-versus-ClawHub ownership boundary.
