## Label Justifications

- `P1`: Third-party skill installation is a live supply-chain security workflow, and the missing lifecycle observer blocks standard scanner and enterprise audit integrations.
- `impact:security`: The issue is about scanning, blocking, and auditing untrusted skills before or after they are accepted onto a user's machine.

## Summary

Keep open: current main and v2026.6.10 cover much of the blocking pre-install side through `security.installPolicy` and `before_install`, but there is still no plugin-visible post-install/audit observer and the remaining security contract needs maintainer product review.

## What This Changes

Request to add blocking `skill:pre-install` and observational `skill:post-install` hook events for skill install/update flows, including ClawHub and UI installs.

## Best Possible Solution

Keep the existing blocking policy seams, then add a documented non-blocking skill install/update completion hook with minimal redacted metadata, explicit ordering and failure semantics, and focused coverage for ClawHub, gateway, UI, and update paths.

## Reproduction Assessment

Yes, at source level. Current main exposes `before_install` only, while the ClawHub success path writes install metadata and returns without dispatching a post-install observer.

## Solution Assessment

Partly. Reusing `security.installPolicy` and `before_install` is the narrowest blocking solution, but the remaining audit capability should be specified as a non-blocking completion hook rather than accepting the original event names and payload unchanged.
