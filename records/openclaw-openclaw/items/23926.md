## Label Justifications

- `P1`: Third-party skill installation is a live supply-chain security boundary, and the missing audit lifecycle hook blocks standard enforcement and enterprise review tooling.
- `impact:security`: The issue concerns scanning, blocking, and auditing untrusted skill installation before user machines accept third-party code and instructions.

## Summary

Keep open: current main covers the blocking side through the generic `before_install` install-policy path, but the requested post-install/audit observer is still missing from the typed hook surface and the committed ClawHub lifecycle. The VettAI/Snyk context makes the security use case concrete, but the remaining hook contract needs maintainer product and security decisions before implementation.

## What This Changes

The issue requests blocking pre-install and observational post-install lifecycle hooks for ClawHub, UI, and plugin-shipped skill installation flows.

## Best Possible Solution

Keep `before_install` as the single blocking policy seam, then add one documented non-blocking completion event at the committed install/update boundary with typed source, target, outcome, timestamps, redaction rules, and observer-failure semantics.

## Reproduction Assessment

Yes, at source level. Current main shows ClawHub installs run the generic blocking policy path, while the typed hook registry and install lifecycle still have no post-install observer event.

## Solution Assessment

Partly. Reusing `before_install` is the narrowest maintainable solution for blocking, but the remaining audit capability should be a carefully specified non-blocking completion hook rather than adding the originally proposed parallel event family unchanged.
