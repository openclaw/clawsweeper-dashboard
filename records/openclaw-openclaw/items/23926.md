## Label Justifications

- `P1`: Third-party ClawHub skill installs and updates can still bypass enforceable local policy hooks before untrusted skill content lands on disk.
- `impact:security`: The issue is about install-time security boundaries, scanner enforcement, and auditability for untrusted third-party skills.

## Summary

Keep open: current main still only partially covers install-time skill policy. The generic `before_install` hook can block some gateway-managed skill dependency and upload paths, but native ClawHub install/update still disables the scan path before copying skill files and there is no post-install observer carrying audit/outcome metadata.

## What This Changes

Add blocking `skill:pre-install` and observational `skill:post-install` hook events for skill install/update flows so plugins can scan, block, warn, and audit untrusted skill installs.

## Best Possible Solution

Define a backward-compatible skill install lifecycle API that covers native ClawHub install/update plus gateway/UI installs with a blocking pre-copy policy point and a post-install audit observer carrying source, target, timestamp, result, and outcome metadata.

## Reproduction Assessment

Yes. This is source-reproducible as a feature gap: current main shows native ClawHub install passing `scan: false`, while the hook surface exposes only `before_install` and no post-install outcome observer.

## Solution Assessment

Unclear. Extending `before_install` is plausible, but maintainers still need to choose the public lifecycle API, blocking semantics, audit payload, UI behavior, and OpenClaw-versus-ClawHub ownership boundary.
