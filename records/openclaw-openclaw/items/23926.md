## Label Justifications

- `P1`: Third-party ClawHub skill installs and updates can still bypass enforceable local policy hooks before untrusted skill content lands on disk.
- `impact:security`: The issue concerns install-time security boundaries, scanner enforcement, and auditability for untrusted third-party skills.

## Summary

Keep open: current main still only partially covers install-time skill policy. The existing `before_install` hook can block some gateway-managed skill dependency and upload paths, but native ClawHub install/update still bypasses that scan path and there is no post-install observer with audit/outcome metadata.

## What This Changes

The issue requests blocking pre-install and observational post-install skill lifecycle hooks for security scanning and audit use cases.

## Best Possible Solution

Define a backward-compatible skill install lifecycle API that covers native ClawHub install/update and gateway/UI installs with a blocking pre-copy policy point plus a post-install audit observer carrying source, target, timestamp, result, and outcome metadata.

## Reproduction Assessment

Yes. This is source-reproducible as a feature gap: current main shows native ClawHub install passing `scan: false`, while the hook surface exposes only `before_install` and no post-install outcome observer.

## Solution Assessment

Unclear. Extending `before_install` is plausible, but maintainers still need to choose the public lifecycle API, blocking semantics, audit payload, UI behavior, and OpenClaw-versus-ClawHub ownership boundary.
