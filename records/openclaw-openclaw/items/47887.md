## Summary

Keep open. Current main still has source-reproducible iOS LAN/plaintext and mixed-auth behavior that conflicts with the report and with some docs, while merged PR #77162 intentionally hardened setup-code issuance without closing the broader LAN/bootstrap-auth discussion and open PR #65185 still targets part of this issue.

## What This Changes

The issue asks OpenClaw to stop iOS LAN `ws://` onboarding from being forced to TLS and to make setup-code/bootstrap/password auth work reliably for iOS gateway pairing.

## Best Possible Solution

Keep this issue open until maintainers settle the private-LAN versus public/Tailscale policy and either land or replace #65185 with a coordinated Swift, TypeScript, docs, and regression-test update.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence reproduction: manual iOS `.local`/LAN hosts still resolve to TLS, the setup-code parser rejects non-loopback `ws://`, and Swift mixed auth sends bootstrap before password; I did not run a live iOS device repro in this read-only review.

## Solution Assessment

Unclear. PR #65185 is a plausible partial repair for the iOS LAN and Swift precedence pieces, but merged #77162 deliberately chose stricter setup-code issuance, so the best fix needs maintainer agreement across iOS, device-pair, setup-code generation, and docs.
