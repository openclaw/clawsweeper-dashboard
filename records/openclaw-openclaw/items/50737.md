## Summary

Keep open. The underlying bug is still real on current main, but this PR should not merge as-is because it fixes the behavior from a stale wrapper/save path instead of the active Anthropic transport and current onboarding config helpers.

## What This Changes

The PR adds Anthropic-compatible authHeader runtime and onboarding changes plus tests so custom anthropic-messages providers can use Authorization bearer auth instead of x-api-key.

## Best Possible Solution

Rework the fix on current main so the Anthropic transport itself chooses Authorization-only versus x-api-key from the resolved provider request/authHeader contract, and make onboarding verification plus saved custom-provider config use the same decision while preserving native Anthropic and OAuth defaults.

## Reproduction Assessment

Yes. Configure a custom provider with api: "anthropic-messages" and authHeader: true, then capture the runtime fetch or inspect a focused transport test; current main can carry Authorization in model.headers while createAnthropicMessagesClient still adds x-api-key from apiKey, and onboarding verification always builds x-api-key.

## Solution Assessment

No. The PR's direction is useful, but the safer current-main solution belongs in createAnthropicMessagesTransportStreamFn/createAnthropicMessagesClient and onboard-custom-config rather than an extra-params wrapper plus one-off verification inference.
