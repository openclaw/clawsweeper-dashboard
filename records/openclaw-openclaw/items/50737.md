## Summary

Keep open: the PR addresses a valid documented `authHeader` bug, but the current patch is not merge-ready because it can remove the selected runtime credential before the active Anthropic transport uses it and can verify Bearer-only endpoints without saving Bearer mode.

## What This Changes

The PR adds an Anthropic runtime wrapper and onboarding probe changes so `anthropic-messages` providers with `authHeader: true` use `Authorization: Bearer <apiKey>` instead of `x-api-key`, with focused tests.

## Best Possible Solution

Rebase or replace the branch with a narrow fix that implements Bearer-mode selection inside the Anthropic transport and custom-provider verification/save helpers, while preserving native Anthropic, OAuth, env/profile auth, and explicit `authHeader: false` behavior.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure a custom `anthropic-messages` provider with `authHeader: true`, then current main still builds `x-api-key` in the active Anthropic transport and onboarding verification uses `x-api-key`; the PR patch can additionally clear `options.apiKey` before that transport runs.

## Solution Assessment

No. The direction is right, but the maintainable fix belongs at the Anthropic transport/client auth decision point and in the onboarding verification/save contract, not in an extra-params wrapper that strips the selected credential before the active transport sees it.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep the API key available to Anthropic transport:** `src/agents/pi-embedded-runner/anthropic-stream-wrappers.ts:371`
  - body: The normal embedded path resolves `anthropic-messages` through `createAnthropicMessagesTransportStreamFn`, which throws when `options.apiKey` is empty before it can build a request. This wrapper clears `apiKey` and only adds an `Authorization` header, so custom providers whose key comes from config/auth profiles can fail with `No API key for provider` or fall back to an ambient env key instead of sending the selected Bearer credential. Put the Bearer/x-api-key choice inside the Anthropic transport while retaining the resolved key.
  - confidence: 0.9
- **[P1] Persist inferred Bearer auth before saving providers:** `src/commands/onboard-custom.ts:818-822`
  - body: When `resolveAnthropicVerificationAuthHeader()` makes verification succeed with `Authorization: Bearer`, that decision is only used for the probe. The later save path still writes a provider without `authHeader: true`, so onboarding can report success for a Bearer-only endpoint and then save config that fails on the first runtime request.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The patch changes outbound credential headers and currently can drop or substitute the selected runtime credential before Anthropic request construction.

Concerns:

- **[medium] Do not clear the selected credential before transport auth:** `src/agents/pi-embedded-runner/anthropic-stream-wrappers.ts:371`
  - body: Setting `apiKey` to `undefined` outside the active Anthropic transport can make the request fail before it is sent or fall back to provider env lookup instead of the resolved config/auth-profile credential. Bearer mode should be implemented where the transport decides between `x-api-key` and `Authorization`, with tests proving Authorization-only output and no credential fallback drift.
