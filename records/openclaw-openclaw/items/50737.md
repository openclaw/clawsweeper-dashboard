## Summary

Keep open. The reported bug is real on current main, but this PR is not merge-ready: the branch is stale and unmergeable, the runtime change is attached to the extra-params wrapper instead of the active Anthropic transport, and the onboarding change can verify with Bearer auth without saving that decision for the resulting provider.

## What This Changes

The PR adds an Anthropic auth-header stream wrapper, wires it through pi embedded runner extra-params, changes custom onboarding verification to infer Bearer auth for Anthropic-compatible endpoints, and adds focused runtime/onboarding tests.

## Best Possible Solution

Rework the fix on current main in the active Anthropic transport and custom onboarding config helpers: `authHeader: true` should make custom `anthropic-messages` runtime requests use Authorization-only auth, verification should use the same decision, newly inferred Bearer mode should be saved or preserved, and native Anthropic/OAuth/default behavior should remain unchanged with focused tests.

## Reproduction Assessment

Yes. Configure a custom provider with `api: "anthropic-messages"` and `authHeader: true`, then capture the runtime and onboarding verification requests; current source shows the runtime still derives `x-api-key` from `options.apiKey`, and verification still builds `x-api-key` probes.

## Solution Assessment

No. The direction is right, but this patch solves credential mode in a stale wrapper layer and does not persist inferred Bearer auth in the saved provider config; the safer solution belongs in the active Anthropic transport plus the onboarding save contract.
