## Summary

Keep this PR open: it targets a real documented `authHeader` gap for Anthropic-compatible providers, but the branch is still not mergeable as-is because verification can succeed without persisting Bearer auth for the saved provider and the external PR still lacks real after-fix behavior proof.

## What This Changes

The PR adds Anthropic-compatible Bearer-auth handling for `authHeader: true` providers in runtime stream wrapping and custom endpoint verification, with focused tests.

## Best Possible Solution

Implement Anthropic Bearer-mode selection in the transport/onboarding config surfaces that own credential serialization, persist inferred Bearer mode when a verified endpoint requires it, and add focused regression coverage for runtime headers plus custom endpoint verification/save behavior.

## Reproduction Assessment

Yes, from source inspection rather than a live run: current docs define the `authHeader` contract, current onboarding still builds `x-api-key` for Anthropic probes, and the transport path still needs proof that Bearer mode omits `x-api-key`. I did not run tests because this review is read-only.

## Solution Assessment

No. The PR addresses the right bug, but the current patch only uses inferred Bearer auth for verification and does not ensure the saved provider preserves that auth mode; the safer fix belongs in the transport and custom onboarding config/save path.
