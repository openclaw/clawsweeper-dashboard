## Summary

Keep open. The reported authHeader bug is real on current main and this PR is still a plausible source contribution, but the branch is not merge-ready: it lacks real behavior proof and still routes credential handling outside the active Anthropic transport/save path.

## What This Changes

The PR adds Anthropic-compatible Bearer auth handling for `authHeader: true` providers in runtime wrapping and custom endpoint verification, with focused tests.

## Best Possible Solution

Move Anthropic Bearer-mode selection into the Anthropic transport and custom onboarding config/save helpers, then add focused regression tests proving `authHeader: true` sends only `Authorization: Bearer` and persists that mode where onboarding infers it.

## Reproduction Assessment

Yes, from source inspection rather than a live run. Current main can inject Authorization for `authHeader: true`, while the active Anthropic transport still adds `x-api-key` from `apiKey` and custom Anthropic verification still builds `x-api-key` only.

## Solution Assessment

No. The PR identifies the right bug, but the maintainable fix belongs in the Anthropic transport and onboarding config/save path rather than an extra-params wrapper plus probe-only inference.
