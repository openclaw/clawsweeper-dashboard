## Summary

Keep this PR open because it targets a real auth-header bug, but the branch is conflicting and the proposed fix still puts credential routing in the wrong layer with no real Bearer-only gateway proof.

## What This Changes

The PR adds an extra-params Anthropic auth wrapper plus onboarding verification changes so `anthropic-messages` providers with `authHeader: true` use Bearer auth, with focused tests.

## Best Possible Solution

Rework this on current main by moving Bearer-mode selection into the Anthropic transport and custom onboarding config helpers, then cover runtime, verification, and saved-config behavior with focused regression tests.

## Reproduction Assessment

Yes. Source inspection on current main shows `authHeader: true` can inject Authorization, while the active Anthropic transport still serializes `x-api-key` from `apiKey` and custom Anthropic verification still builds `x-api-key` only.

## Solution Assessment

No. The PR identifies the right bug, but the runtime fix belongs in `createAnthropicMessagesClient`/transport request config and the onboarding save path, not an extra-params wrapper plus probe-only inference.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Move Bearer auth into the Anthropic transport:** `src/agents/pi-embedded-runner/extra-params.ts:364-365`
  - body: Adding the Bearer switch as an extra-params stream wrapper does not fix the active Anthropic transport contract: the transport still owns request credential serialization and builds `x-api-key` from `apiKey`. Put the `authHeader` mode switch where `createAnthropicMessagesClient` decides between `apiKey` and bearer auth, and prove the request sends exactly one credential header.
  - confidence: 0.89
- **[P1] Persist inferred authHeader when saving endpoints:** `src/commands/onboard-custom.ts:811-820`
  - body: The onboarding change can infer Bearer auth for verification, but that decision is not carried into the saved custom provider config. A Bearer-only endpoint can verify successfully and then be saved without `authHeader: true`, causing the first runtime request to fall back to API-key auth.
  - confidence: 0.9
