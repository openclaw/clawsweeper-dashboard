## Summary

Keep this PR open: it addresses a real documented authHeader bug for Anthropic-compatible providers, but the current patch is stale against main and still needs a transport-level rework before it can safely merge.

## What This Changes

The PR adds an Anthropic extra-params auth wrapper plus onboarding verification changes so `anthropic-messages` providers with `authHeader: true` use Bearer auth, with focused runtime and onboarding tests.

## Best Possible Solution

Put Bearer-mode selection in the Anthropic transport and onboarding config helpers, then add regression coverage proving `authHeader: true` sends Authorization only, omits `x-api-key`, and saves compatible provider config.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure a custom `anthropic-messages` provider with `authHeader: true`, then current main injects Authorization but the Anthropic transport still builds `x-api-key`, and onboarding verification still uses `x-api-key` only.

## Solution Assessment

No. The PR direction is valid, but the implementation should be in `createAnthropicMessagesClient`/transport request config and `onboard-custom-config`, not an extra-params wrapper that can bypass the active transport and a probe-only inference path.

## Review Findings

  - body: Adding this wrapper makes `agent.streamFn` no longer undefined or `streamSimple`, so stream resolution can skip the active boundary-aware Anthropic transport and fall back to a wrapper around the simple stream. Put the auth mode switch where the transport builds `x-api-key`/Authorization so the real runtime path proves exactly one credential header.
  - confidence: 0.89

A Bearer-only endpoint can verify successfully and then be saved without `authHeader: true`, causing the first runtime request to fall back to API-key auth.
  - confidence: 0.9
