## Summary

Keep open: the reported `authHeader` behavior is a real documented bug for Anthropic-compatible paths, but this PR is stale against current main, still has an onboarding save-path correctness bug, and lacks required real behavior proof for a credential-routing change.

## What This Changes

The PR adds Anthropic-compatible Bearer auth handling for `authHeader: true` providers in the old PI stream wrapper and custom onboarding verification tests/flow.

## Best Possible Solution

Port the fix to current `src/agents/anthropic-transport-stream.ts` and `src/commands/onboard-custom-config.ts`, persist inferred Bearer mode when onboarding saves it, and prove the real request path with redacted evidence.

## Reproduction Assessment

Yes, by source inspection: docs define the `authHeader` contract, current main still builds Anthropic runtime requests with `x-api-key` when `apiKey` is present, and onboarding verification still builds Anthropic probes with `x-api-key`. I did not run a live gateway in this read-only review.

## Solution Assessment

No. The PR targets the right behavior, but it is stale against current main's transport/onboarding split and the proposed onboarding flow does not persist inferred `authHeader: true` into the saved provider config.
