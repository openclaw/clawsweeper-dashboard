## Summary

Keep open: this is a valid existing authHeader bug surface, but the branch is not ready because current main only partially overlaps the runtime fix and the PR still leaves a save-path failure for inferred Bearer-mode providers.

## What This Changes

The PR adds Anthropic-compatible Bearer auth handling for `authHeader: true` providers in runtime stream wrapping and custom endpoint verification, with focused tests.

## Best Possible Solution

Move the fix into current main's Anthropic transport and custom onboarding config/save surfaces so `authHeader: true` uses Authorization-only credential routing and any inferred Bearer mode is persisted, with focused regression tests and redacted real request proof.

## Reproduction Assessment

Yes, by source inspection: docs define the `authHeader` contract, current main's Anthropic verification builder still emits `x-api-key`, and the Anthropic transport still adds `x-api-key` whenever `apiKey` is passed. I did not run a live gateway in this read-only review.

## Solution Assessment

No. The PR targets the right bug, but the current patch is stale against main and its inferred Bearer-mode verification is not carried into the saved provider config; the safer fix belongs in the current transport and onboarding config/save path.
