## Summary

Keep open: the documented `authHeader` behavior is still not implemented for the active `anthropic-messages` runtime and verification paths on current main, and this PR still needs rework before it is mergeable.

## What This Changes

The PR adds runtime and onboarding handling so `anthropic-messages` providers with `authHeader: true` use `Authorization: Bearer <apiKey>` instead of `x-api-key`, with focused tests.

## Best Possible Solution

Implement Bearer-mode selection inside the Anthropic transport and custom-provider verification/save helpers using the resolved provider `authHeader` contract, while preserving native Anthropic, OAuth, and explicit override behavior.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure a custom `anthropic-messages` provider with `authHeader: true`, then the current runtime can still build `x-api-key` from `options.apiKey` and onboarding verification always uses `x-api-key`.

## Solution Assessment

No. The PR direction is right, but the maintainable fix should be in `createAnthropicMessagesTransportStreamFn`/`createAnthropicMessagesClient` and `onboard-custom-config` rather than an extra-params wrapper plus probe-only inference.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Persist inferred authHeader before saving endpoints:** `src/commands/onboard-custom.ts:818-822`
  - body: When verification succeeds because the PR inferred Bearer auth, the later save still writes the custom provider without `authHeader: true`. That can make onboarding report success for a Bearer-only endpoint and then save a config that fails on the first runtime request.
  - confidence: 0.9
- **[P2] Move Bearer auth into the Anthropic transport:** `src/agents/pi-embedded-runner/extra-params.ts:364-365`
  - body: The active Anthropic runtime still expects `options.apiKey` and serializes that key through the transport. Wrapping from extra-params and clearing/replacing headers outside the transport can either leave `x-api-key` behavior in place or remove the key before the transport can authenticate; put the mode switch where `createAnthropicMessagesClient` builds auth headers.
  - confidence: 0.86
