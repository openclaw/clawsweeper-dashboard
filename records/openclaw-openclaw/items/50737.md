## Summary

Keep open: current main still has the authHeader gap, and this PR is useful, but it should not merge as-is because it patches stale wrapper/onboarding paths instead of the current Anthropic transport and config helpers.

## What This Changes

The PR adds runtime and onboarding handling so custom anthropic-messages providers with authHeader true use Authorization bearer auth and drop x-api-key, with focused tests.

## Best Possible Solution

Implement Bearer-mode selection inside the Anthropic transport and custom-provider verification/save helpers using the resolved provider authHeader/request contract, while preserving native Anthropic and OAuth defaults.

## Reproduction Assessment

Yes. Configure a custom provider with api: "anthropic-messages" and authHeader: true, then inspect or capture the runtime fetch; source shows current main can add Authorization while still emitting x-api-key, and onboarding verification always builds x-api-key.

## Solution Assessment

No. The PR direction is useful, but the narrow maintainable fix belongs in createAnthropicMessagesTransportStreamFn/createAnthropicMessagesClient and onboard-custom-config rather than an extra-params wrapper plus probe-only inference.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Persist inferred authHeader before saving endpoints:** `src/commands/onboard-custom.ts:818-822`
  - body: When verification succeeds because the PR inferred Bearer auth, the later save still writes the custom provider without authHeader: true. That leaves the first real runtime call able to fall back to x-api-key even though onboarding just proved the endpoint needs Authorization.
  - confidence: 0.9
- **[P2] Move bearer auth into the active Anthropic transport:** `src/agents/pi-embedded-runner/extra-params.ts:365`
  - body: Wrapping agent.streamFn from extra-params is stale for current main: anthropic-messages normally runs through the boundary-aware Anthropic transport, which owns apiKey-to-x-api-key serialization. Put the Authorization-only decision there so the active runtime path suppresses x-api-key without bypassing current transport behavior.
  - confidence: 0.86
