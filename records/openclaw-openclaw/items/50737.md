## Summary

Keep this PR open: the reported `authHeader` behavior is a real existing contract gap for Anthropic-compatible runtime and onboarding paths, but this branch is stale against current main, has a save-path correctness bug, and still lacks real request proof for the credential-routing change.

## What This Changes

The PR adds Anthropic-compatible Bearer auth handling for `authHeader: true` providers in the old PI stream wrapper and custom onboarding verification flow.

## Best Possible Solution

Port the behavior to the current Anthropic transport and onboarding config paths, clear the conflicting `x-api-key`/API-key mode for `authHeader: true`, persist inferred Bearer mode when onboarding saves it, and add focused tests plus redacted live request proof.

## Reproduction Assessment

Yes, by source inspection: docs define the `authHeader` contract, current main still builds Anthropic runtime requests with `x-api-key` when `apiKey` is present, and onboarding verification still builds Anthropic probes with `x-api-key`. I did not run a live gateway in this read-only review.

## Solution Assessment

No. The PR targets the right behavior, but it is stale against current main's transport/onboarding split and the proposed onboarding flow does not persist inferred `authHeader: true` into the saved provider config.
