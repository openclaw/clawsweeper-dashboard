## Label Justifications

- `P2`: This is a normal-priority stability and ergonomics feature for runaway agent runs, with meaningful impact but a required product/API decision before implementation.
- `impact:session-state`: Runaway tool loops can keep a session in an active tool-calling state instead of converging to a terminal user-visible response.
- `impact:message-loss`: The reported Discord case reached typing timeout and restart interruption without delivering the useful tool result to the user.

## Summary

Keep open. Current main and v2026.6.5 still do not expose or enforce a global/per-agent hard tool-turn or tool-call budget; `runRetries` and `loopDetection` are adjacent safeguards, but they do not force a final response after a configured count.

## What This Changes

Request to add agent-level `maxTurns`, `maxToolCalls`, and `toolCallTimeout` configuration for bounding runaway tool-call iterations.

## Best Possible Solution

Design one canonical core agent-run guardrail that enforces approved hard budgets before the next tool execution, with typed default/per-agent config, clear exhaustion semantics, docs, and focused runtime tests.

## Reproduction Assessment

Yes for the missing capability: current strict schemas and runtime hooks show no global/per-agent hard `maxTurns` or `maxToolCalls` budget. I did not replay the live KIMI/NVIDIA loop, but the source-level absence is high confidence.

## Solution Assessment

Unclear until maintainers choose the permanent config/API shape and exhaustion semantics. If approved, the maintainable direction is core agent-run enforcement before tool execution, not prompt-only instructions.
