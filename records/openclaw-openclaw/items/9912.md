## Summary

Current main and the latest release still lack a global or per-agent hard model/tool iteration budget, so this remains necessary rather than implemented or superseded.

## What This Changes

Request a typed `maxTurns`, `maxToolCalls`, and tool-response-timeout style configuration surface that stops runaway agent tool-call loops and returns a final user response.

## Best Possible Solution

Define one typed, opt-in agent-run budget policy with global defaults and per-agent overrides, enforce it at the canonical model/tool loop before the next tool executes, and specify deterministic terminal response semantics that preserve the last useful result.

## Reproduction Assessment

Yes for the missing capability: current strict schemas and the before-tool/runtime code show no absolute max-turn or max-tool-call budget. I did not reproduce the provider-specific KIMI loop live, so the production behavior remains log-backed rather than locally reproduced.

## Solution Assessment

Unclear as proposed. A hard runtime budget is the right enforcement layer, but the final key names, default mode, timeout-after-first-tool semantics, and forced-response behavior need maintainer design before implementation.
