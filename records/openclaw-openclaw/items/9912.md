## Summary

Keep open. Current main still lacks a strict global/per-agent hard `maxTurns`, `maxToolCalls`, or `toolCallTimeout` budget; existing loop detection is pattern-based, and related open issues cover narrower or broader variants rather than replacing this canonical config request.

## What This Changes

Request an agent-level config budget to cap model-tool iterations/tool calls and force a response when exhausted.

## Best Possible Solution

Define an additive agent-run budget contract under `agents.defaults` and `agents.list[]`, enforce counters before the next tool/model continuation, document precedence with loop detection/timeouts/OpenResponses, and cover exhaustion behavior with focused config/runtime tests.

## Reproduction Assessment

Not applicable as a strict bug reproduction. Source and docs on current main give a high-confidence absence check for the requested config and enforcement surface; the exact provider loop was not replayed.

## Solution Assessment

Unclear until maintainers choose the public API and semantics. A defaults/per-agent hard budget is plausible, but field names, timeout-after-first-tool behavior, and forced-response semantics need product decisions before implementation.
