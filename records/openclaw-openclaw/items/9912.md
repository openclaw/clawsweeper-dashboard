## Summary

Keep open: current main still lacks a strict global/per-agent `maxTurns`, `maxToolCalls`, or `toolCallTimeout` control. The nearby safeguards are disabled-by-default pattern loop detection, provider/tool-specific `maxTurns` fields, and an OpenResponses `max_tool_calls` field that is documented as ignored, so this remains a distinct feature/config request rather than an implemented cleanup close.

## What This Changes

Request an agent-level config budget that caps model-tool iterations or total tool calls and forces a final response when the limit is reached.

## Best Possible Solution

Add a typed global/per-agent hard iteration budget under `agents.defaults` and per-agent config, enforce it before the next tool execution or continuation turn, document its precedence, and keep per-channel budget/delegation work tracked separately in #13271.

## Reproduction Assessment

Not applicable: this is a feature request, not a failure of an existing documented config contract. Source inspection on current main gives high confidence that the requested agent-level budget fields and counter-based enforcement path are absent.

## Solution Assessment

Unclear until maintainers settle the config/API shape and exhaustion semantics. A hard budget is plausible, but implementing it safely requires product decisions before a narrow fix PR is appropriate.
