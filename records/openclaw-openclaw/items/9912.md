## Summary

Keep open: current main still lacks the requested global or per-agent hard budget for model-tool turns, total tool calls, and timeout-after-first-tool-call behavior. Existing controls cover whole-run/model timeouts plus optional pattern-based loop detection, and OpenResponses `max_tool_calls` is accepted but not enforced.

## What This Changes

Request a global or per-agent config budget that limits agent iterations, total tool calls, and post-tool-call response latency to prevent runaway tool loops.

## Best Possible Solution

Add typed strict global and per-agent hard budgets, enforce counters before the next tool execution, define timeout-after-first-tool-call semantics, and document/test precedence with existing loop detection and timeout controls.

## Reproduction Assessment

Not applicable as a feature request. The production log scenario is credible, and the current-main absence is high-confidence from schema, docs, and runtime inspection rather than a failing regression test.

## Solution Assessment

Unclear until maintainers settle the exact policy. A hard runtime budget in core is the right direction, but the narrow maintainable implementation needs explicit scope, precedence, and fallback behavior before automation should write it.
