## Summary

Keep this issue open as the canonical global/per-agent hard tool-iteration budget request. Current main adds configurable failure-retry limits and increasingly capable pattern-based loop detection, but neither provides the requested unconditional tool-call or model/tool-turn cap for an otherwise successful run.

## What This Changes

Requests configurable per-agent limits for model/tool iterations and timeout behavior to stop runaway agent runs before users lose a response.

## Best Possible Solution

Add one typed global/per-agent guardrail contract enforced immediately before tool execution, with separately defined turn/run counters, deterministic exhaustion outcomes, and documented interaction with existing retry, timeout, and loop-detection controls.

## Reproduction Assessment

Yes at source level: current main has no unconditional configurable total-turn or total-tool-call cap, and the existing controls demonstrably cover failure retries or detected repetition instead. The exact KIMI/Discord five-minute timing incident was not live-reproduced.

## Solution Assessment

No, not yet as specified: runtime enforcement is the correct boundary, but the three proposed options overlap existing timeout/retry controls and leave counter scope and safe exhaustion behavior ambiguous. A single coherent guardrail contract should be designed before implementation.
