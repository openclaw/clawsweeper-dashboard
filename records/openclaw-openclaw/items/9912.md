## Summary

Keep open. Current main still lacks a global or per-agent hard max-turn/max-tool-call budget, and adjacent safeguards are pattern-based or retry-oriented rather than the requested public config and forced final-response behavior.

## What This Changes

Add global/per-agent config options such as `maxTurns`, `maxToolCalls`, and `toolCallTimeout` to cap agent tool-loop iterations and force a final response.

## Best Possible Solution

Keep this as the canonical global/per-agent hard iteration-budget request and design an additive typed config plus runtime enforcement path that composes with existing loop detection, retry limits, and scoped-budget follow-ups.

No, not as a high-confidence live reproduction of the KIMI/NVIDIA loop; that provider behavior was not replayed. Source inspection does give a high-confidence reproduction of the missing global/per-agent config and enforcement surface on current main.

## Solution Assessment

Unclear. A hard global/per-agent budget is plausible, but maintainers need to choose the public config shape, enforcement point, timeout-after-first-tool-call semantics, and final-response behavior before implementation is safe.
