- `impact:message-loss`: The reported agent may complete useful tool work but continue iterating until no final response is delivered.

## Summary

Current main still lacks a general per-agent or global hard budget for model/tool iterations. Existing protections cover repetitive-loop patterns, post-compaction recurrence, and whole-run timeouts, while related open items address distinct per-channel or Ralph-loop scopes; this issue should remain the canonical product-decision item for a core run budget.

## What This Changes

Requests configurable hard limits on agent turns, tool calls, and post-tool-call duration to stop runaway agent loops and force a user-visible response.

## Best Possible Solution

Introduce one typed, run-scoped budget owned by the embedded runner, enforce it before executing the next tool, and normalize exhaustion through the canonical terminal-outcome and channel-delivery path; treat broader session and per-channel policies as separate follow-up scopes.

## Reproduction Assessment

Yes at source level, though no live current-main run was performed. The strict schemas, documented ignored OpenResponses field, and pattern-based before-tool guard establish that current main has no generic aggregate turn or tool-call budget matching the reported need.

## Solution Assessment

The missing capability is well supported, but the proposed three independent keys are not yet the best proven API. A single run-budget model with explicit counting, precedence, and terminal behavior is narrower and less likely to create competing controls.

## Risks / Open Questions

- Maintainers must define whether the budget counts model round trips, individual parallel tool invocations, retries, compaction attempts, nested tools, or some combination.
- Exhaustion behavior is product-sensitive: blocking the next tool, injecting a model-visible warning, forcing another model response, or returning a prior tool result have different safety and channel-delivery consequences.
- A timeout measured after the first tool call overlaps whole-run and provider request timeouts and could create conflicting deadlines without explicit precedence.
- Session-wide limits would require durable state ownership, while a run- or turn-scoped limit can remain local; combining these scopes would substantially enlarge the design.
