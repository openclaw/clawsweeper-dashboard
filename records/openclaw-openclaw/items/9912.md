## Summary

Keep open: current main still does not implement the requested global or per-agent hard budget for model-tool turns, total tool calls, or timeout-after-first-tool-call behavior. The existing controls are whole-run/model timeouts plus optional pattern-based loop detection, and the OpenResponses `max_tool_calls` field remains accepted but ignored.

## What This Changes

The issue requests `maxTurns`, `maxToolCalls`, and `toolCallTimeout` style agent configuration to stop runaway tool-call iterations and force a user-visible response.

## Best Possible Solution

Add a typed, strict global and per-agent hard budget for embedded agent runs, enforce turn and tool-call counters before executing the next tool, define timeout-after-first-tool-call semantics, document precedence with existing timeouts and loop detection, and cover exhaustion/final-response behavior with focused tests.

## Reproduction Assessment

Not applicable as a feature request. The production logs provide a credible model-specific runaway-loop scenario, and current-main absence is reproducible by inspecting the schema, docs, and before-tool-call enforcement path.

## Solution Assessment

Unclear until maintainers settle the exact exhaustion policy. A hard runtime budget is the right product direction for this request, but the safest implementation needs an explicit design for scope, precedence, and user-visible fallback behavior before a repair PR is queued.
