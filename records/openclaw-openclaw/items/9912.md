## Label Justifications

- `P2`: Runaway tool cycles can strand real agent turns and waste resources, but this remains a bounded product enhancement rather than an emergency core outage.
- `impact:session-state`: The reported behavior leaves an agent run trapped in repeated model-tool cycles without reaching a clean terminal state.
- `impact:message-loss`: The reported run can consume the turn without delivering the available tool result or a final user response.

## Summary

Current main and the latest release still lack a global or per-agent hard model/tool iteration budget; existing safeguards detect specific repetitive patterns or stop the whole run by elapsed time, so the central request remains necessary and distinct from the per-channel delegation request at https://github.com/openclaw/openclaw/issues/13271.

## What This Changes

Add configurable global and per-agent limits for model-tool turns, total tool calls, and possibly elapsed time after the first tool call, with graceful final-response behavior when exhausted.

## Best Possible Solution

Define one canonical agent-run budget contract with defaults and per-agent overrides, count model-tool turns and executed calls at the embedded before-tool-call boundary, specify parallel/retry/subagent accounting and precedence with existing safeguards, and terminate through the normal agent-run outcome path so channels receive a clear final result.

## Reproduction Assessment

Yes at the source level: current main has pattern-based loop detection and whole-run timeouts but no hard arbitrary-call or model-tool-turn budget, directly confirming the capability gap. The original NVIDIA/Kimi timing incident itself was not rerun live.

## Solution Assessment

Unclear: a hard runtime budget is the right class of safeguard, but the proposed three independent knobs and forced-output behavior are not yet the best-defined solution; maintainers should first settle one coherent accounting and terminal-outcome contract.
