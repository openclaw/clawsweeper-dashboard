## Summary

Keep open. Current main still lacks a strict global/defaults or per-agent hard `maxTurns`, `maxToolCalls`, or `toolCallTimeout` budget; adjacent loop detection, run retries, and OpenResponses parsing do not implement the requested forced-stop behavior.

## What This Changes

The issue requests agent-level config to cap model-tool turns, total tool calls, and timeout-after-first-tool behavior for runaway agent loops.

## Best Possible Solution

Define an additive typed agent-run budget contract under `agents.defaults` and `agents.list[]`, enforce counters in the trusted model-tool runtime before another tool execution or continuation, and document how it composes with existing loop detection, retries, and timeouts.

## Reproduction Assessment

Not applicable as a strict bug reproduction. Source, docs, live issue context, and related searches give a high-confidence absence check for the requested config and enforcement path, but the reported provider loop was not replayed.

## Solution Assessment

Unclear until maintainers choose the public API and semantics. A defaults/per-agent hard budget is plausible, but field names, timeout-after-first-tool behavior, blocking versus warning shape, and forced-response handling require product decisions.
