## Label Justifications

- `P2`: This is a validated core-runtime improvement with meaningful but intermittent runaway-loop and response-delivery impact, not an emergency regression.
- `impact:session-state`: Unbounded model/tool iteration can leave an agent run stuck or interrupted without a clean terminal state.
- `impact:message-loss`: Reported runs can exhaust typing or runtime limits without delivering the useful tool result or any final response.

## Summary

Current main still lacks a hard global or per-agent model/tool iteration budget, and the issue discussion documents multiple real-world cases where prompt guidance and pattern loop detection do not guarantee a final user response. This remains a product-sensitive core runtime request: maintainers must choose budget scope, exhaustion semantics, and precedence with existing timeouts and loop detection.

## What This Changes

Add configurable global and per-agent caps for model/tool iterations, total tool calls, and elapsed time after tool use, with graceful final-response behavior when a limit is reached.

## Best Possible Solution

Define one typed opt-in budget policy at agent defaults with per-agent override, enforce counters at the canonical embedded model/tool loop before the next tool executes, and specify a deterministic terminal outcome that preserves the last useful result and delivers a user-visible response.

## Reproduction Assessment

Yes for the missing capability: current source and strict schemas show no absolute iteration counter, while the supplied production logs and follow-up report describe observable runaway or unwanted post-tool turns. A deterministic current-main live reproduction depends on model behavior, so this is source-reproducible rather than universally reproduced.

## Solution Assessment

Unclear. A hard runtime budget is the right enforcement layer, but the proposed three independent keys and forced-response semantics need consolidation and a maintainer decision before implementation.

- A default-on cap could truncate legitimate long tasks, while an opt-in cap adds another configuration surface; this requires explicit product judgment and upgrade-safe defaults.
