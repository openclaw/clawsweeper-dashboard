## What This Changes

The issue requests `maxTurns`, `maxToolCalls`, and related agent-level configuration to force an agent to stop tool-looping and produce a response.

## Best Possible Solution

Define one typed, opt-in agent-run budget policy with global defaults and per-agent overrides, enforce it before the next tool executes in the canonical runtime loop, and specify deterministic terminal response semantics that preserve the last useful result.

## Reproduction Assessment

Yes for the missing capability: current strict schemas and before-tool/runtime code show no absolute max-turn or max-tool-call budget. The specific KIMI/NVIDIA loop remains production-log-backed rather than live-reproduced in this review.

Unclear as proposed. A hard runtime budget is the right enforcement layer, but the final key names, default mode, timeout-after-first-tool semantics, and forced-response behavior need maintainer design before implementation.
