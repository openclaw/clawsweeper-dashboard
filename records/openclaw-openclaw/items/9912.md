## Summary

Keep open: current main and the latest release still lack a global or per-agent hard agent iteration/tool-call budget, and the remaining work is a product-sensitive config/runtime contract rather than a cleanup close.

## What This Changes

Request configurable agent-level `maxTurns`, `maxToolCalls`, and `toolCallTimeout` limits to stop runaway model-tool loops and preserve a final user response.

## Best Possible Solution

Define one typed, opt-in agent-run budget policy with global defaults and per-agent overrides, enforce it before the next tool executes in the canonical runtime loop, and ship docs plus focused exhaustion/final-response tests.

## Reproduction Assessment

Yes for the missing capability: current strict schemas, loop-detection config, and the shared agent loop show no absolute max-turn or max-tool-call budget. The specific KIMI/NVIDIA loop remains production-log-backed rather than live-reproduced in this review.

## Solution Assessment

Unclear as proposed. A hard runtime budget is the right enforcement layer, but the final key names, default mode, timeout-after-first-tool semantics, and forced-response behavior need maintainer design before implementation.
