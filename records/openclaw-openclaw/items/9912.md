## Summary

Keep open: current main still lacks the requested global or per-agent hard iteration budget for model-tool turns, total tool calls, and timeout-after-first-tool-call behavior. Existing controls are whole-run/model timeouts, optional pattern-based loop detection, and an accepted-but-ignored OpenResponses `max_tool_calls` field.

## What This Changes

Request a `maxTurns`, `maxToolCalls`, and `toolCallTimeout` style agent configuration to stop runaway tool-call loops and force a response.

## Best Possible Solution

Add typed strict global and per-agent hard budgets, enforce counters before the next tool execution, define timeout-after-first-tool-call semantics, and document/test precedence with existing loop detection and timeout controls.

## Reproduction Assessment

Not applicable as a feature request. The production log scenario is credible, and the current-main absence is high-confidence from schema, docs, and runtime inspection rather than a failing regression test.

## Solution Assessment

Unclear until maintainers settle the exact policy. A hard runtime budget in the embedded/tool-call runtime is the right direction, but the narrow maintainable implementation needs explicit scope, precedence, and fallback behavior before automation should write it.
