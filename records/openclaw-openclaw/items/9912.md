## Summary

Keep open. Current main and the latest shipped tag still do not provide the requested global or per-agent hard max-turn/max-tool-call budget; the adjacent protections are retry bounds, pattern-based loop detection, and timeouts, so this still needs maintainer product/API review.

## What This Changes

Request a configurable agent-level max-turn, max-tool-call, or after-first-tool timeout budget to stop runaway tool-call loops and force a response.

## Best Possible Solution

Keep this as the canonical global/per-agent hard iteration-budget request and design an additive typed config plus runtime enforcement path that composes with existing loop detection, retry limits, and scoped-budget follow-ups.

## Reproduction Assessment

No, not as a strict live bug reproduction; the specific KIMI/NVIDIA loop was not replayed. Source, docs, issue-search, and release-tag checks give a high-confidence reproduction of the missing global/per-agent hard budget on current main.

## Solution Assessment

Unclear. A hard global/per-agent budget is plausible, but maintainers need to choose the public config shape, enforcement point, timeout semantics, and final-response behavior before implementation is safe.
