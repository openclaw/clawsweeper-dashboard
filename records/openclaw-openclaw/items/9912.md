## Summary

Keep open. Current main still lacks a global or per-agent hard iteration budget, and the existing safeguards are pattern-based loop detection plus fixed retry/timeouts rather than configurable `maxTurns`, `maxToolCalls`, or `toolCallTimeout` behavior.

## What This Changes

Request an agent-level config option to cap model-tool turns, total tool calls, and post-tool-call response time for runaway agent loops.

## Best Possible Solution

Add typed global and per-agent budget config, enforce counters before executing the next tool, define timeout-after-first-tool-call semantics, and document/test how these hard limits interact with existing loop detection and timeouts.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection gives high confidence the capability is absent from current main: the strict agent schemas lack the requested fields and the available guard is pattern-based loop detection.

## Solution Assessment

Unclear until maintainers decide the config shape and exhaustion semantics. The narrow technical boundary should be agent config plus pre-tool enforcement, with docs and tests covering precedence.
