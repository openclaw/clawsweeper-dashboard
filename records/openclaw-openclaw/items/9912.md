## Summary

Keep open. Current main still lacks a strict global/defaults or per-agent hard `maxTurns`, `maxToolCalls`, or `toolCallTimeout` budget; adjacent loop detection and run-retry surfaces do not implement the requested config or forced-response behavior.

## What This Changes

The issue requests agent-level hard iteration limits such as `maxTurns`, `maxToolCalls`, and `toolCallTimeout` to stop runaway model-tool loops and force a final response.

## Best Possible Solution

Define an additive agent-run budget contract under `agents.defaults` and `agents.list[]`, enforce counters in the trusted model-tool runtime before the next tool execution or continuation, and document how it composes with loop detection, run retries, provider timeouts, and OpenResponses.

## Reproduction Assessment

Not applicable as a strict bug reproduction. Source, schema, docs, live issue context, and related-item searches give a high-confidence absence check for the requested config and enforcement path, but the reported provider loop was not replayed.

## Solution Assessment

Unclear until maintainers choose the public API and semantics. A defaults/per-agent hard budget is plausible, but field names, timeout-after-first-tool behavior, blocking versus warning shape, and forced-response handling are product decisions.
