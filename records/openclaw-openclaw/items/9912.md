- `impact:message-loss`: The reported runaway tool loop can prevent a Discord response from being delivered after successful tool execution.

## Summary

Keep open. Current main and the latest release still lack a global or per-agent hard `maxTurns`/`maxToolCalls` budget; the existing guards are pattern-based loop detection, whole-run retry/timeouts, or ignored OpenResponses compatibility fields, so the requested behavior remains unsolved.

## What This Changes

Add global and per-agent config such as `maxTurns`, `maxToolCalls`, and `toolCallTimeout` to force agent runs to stop tool iteration and produce a response after configured limits.

## Best Possible Solution

Design an additive `agents.defaults` plus per-agent hard-budget contract, enforce counters before the next tool execution, document precedence with existing loop detection and timeouts, and keep per-channel auto-delegation work separate.

## Reproduction Assessment

Yes for source-level absence: current main and `v2026.5.22` lack the requested global/per-agent hard budget in schema, docs, resolver, and before-tool enforcement paths. No high-confidence live replay of the specific KIMI/NVIDIA loop was performed in this read-only review.

## Solution Assessment

Unclear. A hard budget is plausible, but maintainers need to choose the public config shape, enforcement point, timeout-after-first-tool-call semantics, and final-response behavior before implementation is safe.
