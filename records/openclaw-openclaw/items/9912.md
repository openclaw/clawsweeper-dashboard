## Summary

Keep open. Current main and the latest release still do not expose or enforce a global/per-agent hard tool-turn or tool-call budget; existing loop detection, runtime timeouts, xAI tool-local `maxTurns`, and ignored OpenResponses `max_tool_calls` do not satisfy the requested agent-run exhaustion behavior.

## What This Changes

The issue requests agent-level configuration such as `maxTurns`, `maxToolCalls`, and `toolCallTimeout` to stop runaway tool-call loops and force a response.

## Best Possible Solution

Design an additive `agents.defaults` plus per-agent hard-budget contract, enforce counters before the next tool execution, document precedence with existing loop detection and timeout behavior, and keep per-channel budget/autodelegation work separate.

## Reproduction Assessment

No for a high-confidence live replay of the KIMI/NVIDIA loop; that provider behavior was not exercised in this read-only review. Yes for source-level absence: current main and `v2026.5.22` lack the requested global/per-agent hard budget in schema, docs, resolver, and before-tool enforcement paths.

## Solution Assessment

Unclear. A hard budget is plausible, but maintainers need to choose the public config shape, enforcement point, timeout-after-first-tool-call semantics, and final-response behavior before implementation is safe.
