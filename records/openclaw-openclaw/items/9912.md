## Summary

Keep open: current main and v2026.6.6 still lack the requested global or per-agent hard model/tool iteration budget, and the remaining work needs a maintainer product decision for a new config/runtime contract.

## What This Changes

Request a global or per-agent `maxTurns`/`maxToolCalls`/tool-call timeout budget that stops runaway agent model-tool loops and returns a useful response.

## Best Possible Solution

Define one typed, opt-in agent-run budget policy with global defaults and per-agent overrides, enforce it before the next tool executes in the canonical embedded/tool-call loop, and specify deterministic terminal response semantics that preserve the last useful result.

## Reproduction Assessment

Yes for the missing capability: current strict schemas and the before-tool/runtime code show no absolute max-turn or max-tool-call budget. I did not reproduce the provider-specific KIMI loop live, so the production behavior remains log-backed rather than locally reproduced.

## Solution Assessment

Unclear as proposed. A hard runtime budget is the right enforcement layer, but the final key names, default mode, timeout-after-first-tool semantics, and forced-response behavior need maintainer design before implementation.
