## Summary

Keep open: the PR targets a real session-file prompt ownership race and includes useful proof, but the submitted holder lifecycle can leave a session file permanently marked prompt-active after the compaction-wait release path.

## What This Changes

The PR adds a session-file-scoped embedded prompt holder registry in `src/agents/pi-embedded-runner/run/attempt.session-lock.ts` and regression coverage for two controllers sharing one session JSONL file.

## Best Possible Solution

Keep the file-scoped prompt serialization, but pair holder registration and release across every lock-drop path or split the API so only true provider prompt windows register a holder, with regressions for provider streaming and post-prompt compaction waits.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current `attempt.ts` calls `releaseForPrompt()` before compaction wait and later reacquires via `withSessionWriteLock()`, while the patch only clears the new holder from `reacquireAfterPrompt()`.

## Solution Assessment

No as submitted. The file-scoped guard is the right direction, but the implementation should either scope it to provider prompt submission or clear it from all lock reacquisition paths.
