The PR adds a per-session-file prompt-holder guard for embedded runner prompt lock releases, separates compaction idle waits from provider prompt windows, and adds session-lock regressions.

## Best Possible Solution

Serialize provider prompt windows by session file with a loop that re-checks active ownership after reacquiring the session write lock, while keeping compaction idle waits out of prompt-holder ownership.

Full review comments:

- **[P1] Re-check the holder after waiting for a prompt turn:** `src/agents/pi-embedded-runner/run/attempt.session-lock.ts:729`
  - body: When two controllers are waiting on the same existing prompt holder, both `waitForPromptSessionFileTurn()` calls can return as soon as that first holder releases. The first waiter then registers its own holder and releases the file lock, but the second waiter has already left the wait loop; after it acquires the lock it calls `registerPromptSessionFileHolder()` and overwrites the first waiter's holder instead of waiting for it. That still allows overlapping prompt windows and can strand any waiter that observed the overwritten promise, so this needs another holder check after reacquiring the session write lock.
