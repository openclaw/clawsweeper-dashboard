## Label Justifications

- `P1`: The PR addresses a real dropped-reply race, but the current patch can also stall future same-session agent work if merged as-is.
- `merge-risk: 🚨 session-state`: The new per-file holder state can become stale and misrepresent a session file as still prompt-active.
- `merge-risk: 🚨 message-delivery`: A stale holder can block later prompt releases on the same transcript and prevent expected channel replies from being produced.
- `merge-risk: 🚨 availability`: Waiters block on a holder promise with no timeout if the non-provider release path never resolves it.

## Summary

Keep open: the branch targets the right core reliability problem, but the patch registers a per-file prompt holder from a controller method that is also used for non-provider compaction waits, which can leave a stale holder and block later same-session runs indefinitely.

## What This Changes

The PR adds a session-file-scoped embedded prompt holder registry in the agent session-lock controller and a regression test for two prompt holders sharing one session JSONL file.

## Best Possible Solution

Keep the file-scoped prompt serialization, but pair holder registration and release across every controller path that can drop the write lock, with regression coverage for provider streaming and post-prompt compaction waits.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current `attempt.ts` calls `releaseForPrompt()` before compaction wait and later reacquires via `withSessionWriteLock()`, while the patch only clears the new holder from `reacquireAfterPrompt()`.

## Solution Assessment

No as submitted. The file-scoped guard is the right direction, but the implementation should either scope it to provider prompt submission or clear it from all lock reacquisition paths.
