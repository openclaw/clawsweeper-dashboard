## Label Justifications

- `P1`: The PR targets a broken core agent/session workflow and currently introduces a same-session hang risk in that path.
- `merge-risk: 🚨 session-state`: The new process-local holder can stale a session file's prompt-active ownership state after an existing non-prompt reacquire path.
- `merge-risk: 🚨 availability`: A stale holder would make later same-file prompt releases wait indefinitely, stalling affected embedded agent runs.

## Summary

Keep this PR open, but not merge-ready: the fix is focused and the supplied real-behavior proof covers the provider prompt path, while source review shows the new prompt-holder map can leak across the existing compaction-wait release path and block later same-session runs.

## What This Changes

The PR adds a session-file-scoped prompt-holder registry to the embedded attempt session lock controller and a same-session regression test.

## Best Possible Solution

Keep the file-scoped prompt guard, but release or avoid registering the holder for every non-provider reacquire path so all existing session-lock lifecycles drain deterministically.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: call releaseForPrompt for the compaction-wait style path, reacquire with withSessionWriteLock, then start a second same-file controller and observe it waiting on the stale holder.

## Solution Assessment

No, not as submitted. The registry is the right fix direction, but it must either be scoped to provider prompt windows or be cleared from every later reacquire/cleanup path before merge.
