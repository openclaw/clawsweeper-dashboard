## Summary

Keep open for maintainer review: the code change is a focused Feishu permission-tool gate that matches the existing account-routing model, but this PR has a protected `maintainer` label, adds an intentional fail-closed compatibility change, and the external PR body only provides mocked/local validation rather than real behavior proof.

## What This Changes

The branch resolves the selected Feishu account before `feishu_perm` execution, rejects that tool when the selected account has `tools.perm` disabled, and adds contextual plus explicit-account routing regression tests.

## Best Possible Solution

Land the resolved-account `perm` gate after maintainers accept the intentional fail-closed behavior and the contributor adds redacted real-runtime proof for the changed Feishu permission-tool path.

## Reproduction Assessment

Yes. Source inspection shows current main registers `feishu_perm` from any enabled account's `perm` setting and then creates a client for the resolved account without checking that account's `tools.perm`; the PR's added tests encode the contextual and explicit disabled-account cases, though I did not run them in this read-only review.

## Solution Assessment

Yes. The PR mirrors the existing Bitable per-account execution gate and keeps the fix in the Feishu permission-tool owner boundary; changing the shared `createFeishuToolClient` helper for all tool families would be broader and riskier because wiki, drive, and doc do not currently share the same fail-closed per-account contract.
