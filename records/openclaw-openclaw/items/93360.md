- `merge-risk: 🚨 compatibility`: The patch intentionally changes existing multi-account runtime behavior from permissive execution to a fail-closed error when the selected account has permission tooling disabled.

## Summary

Keep open. Current main and the latest release still lack the selected-account `perm` gate, and the patch looks like a correct narrow fix, but it has a protected `maintainer` label, no real Feishu/OpenClaw runtime proof, a fail-closed compatibility impact, and a broader sibling PR at https://github.com/openclaw/openclaw/pull/93363 that maintainers may prefer.

## What This Changes

The PR resolves the selected Feishu account before `feishu_perm` execution, checks that account's `tools.perm` setting before client creation, and adds contextual plus explicit-account routing tests.

## Best Possible Solution

Choose one Feishu account-gating path, require real runtime proof or an explicit proof override, and explicitly accept the fail-closed upgrade behavior before merge.

## Reproduction Assessment

Yes, source-reproducible. Current main and v2026.6.6 register `feishu_perm` from any enabled account's `perm` setting and then create a client for the resolved account without checking that account's `tools.perm`; I did not run tests in this read-only review.

## Solution Assessment

Mostly yes for the narrow permission-tool bug. The patch mirrors the existing Bitable selected-account gate, but maintainers should decide whether the broader shared family-gate path in https://github.com/openclaw/openclaw/pull/93363 is the better final shape.
