## Summary

Keep this PR open: the reported Anthropic env-routing problem is real, but the current patch still treats default catalog URLs as explicit base URLs and the external PR lacks real after-fix proxy/PDF proof.

## What This Changes

The PR exports an Anthropic Messages URL resolver with `ANTHROPIC_BASE_URL` fallback, adds resolver unit tests, and makes native Anthropic PDF requests use that resolver.

## Best Possible Solution

Make Anthropic base URL resolution source-aware so explicit provider config wins, default catalog URLs remain env-overridable, streaming and native PDF share the same normalization, and both paths have focused regression tests plus redacted real proxy/PDF proof.

## Reproduction Assessment

Yes, source-level. Current main ignores `ANTHROPIC_BASE_URL`, and the PR helper still receives the default catalog `https://api.anthropic.com` value as a non-empty `baseUrl`, so the env-only path remains bypassed.

## Solution Assessment

No. Sharing URL normalization is useful, but the fix needs to distinguish explicit provider config from default catalog metadata before request URL construction.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Treat default catalog URLs as env-overridable:** `src/agents/anthropic-transport-stream.ts:489`
  - body: Default Anthropic catalog models already carry `baseUrl: "https://api.anthropic.com"`, and the resolver passes that value into this helper when no provider config exists. Because this new precedence treats any non-empty `baseUrl` as explicit, setting only `ANTHROPIC_BASE_URL` still routes normal Anthropic streaming and native PDF requests to the public endpoint. Move the env fallback before default catalog URL assignment, or carry base URL provenance so configured URLs still win.
  - confidence: 0.92
