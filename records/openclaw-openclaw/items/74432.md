## Summary

Keep this PR open: it targets a valid Anthropic provider-routing bug, but the current patch still does not fix the default resolved-model path and the external PR lacks after-fix real behavior proof.

## What This Changes

The branch exports an Anthropic Messages URL resolver with `ANTHROPIC_BASE_URL` fallback, adds resolver tests, and routes native Anthropic PDF fetches through that resolver.

## Best Possible Solution

Preserve explicit provider config precedence, but make default Anthropic catalog URLs env-overridable before they become indistinguishable from user-configured `baseUrl`, and cover both streaming and native PDF paths.

## Reproduction Assessment

Yes, from source inspection: resolve a default Anthropic catalog model with `ANTHROPIC_BASE_URL` set and no provider `baseUrl`, then observe that streaming and native PDF receive `https://api.anthropic.com` as `model.baseUrl` before the PR helper can use the env fallback.

## Solution Assessment

No. Sharing a URL normalizer is useful, but the fix needs source-aware base URL provenance or a resolver-layer change so catalog defaults do not override `ANTHROPIC_BASE_URL` while explicit provider config still wins.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Treat catalog defaults as env-overridable:** `src/agents/anthropic-transport-stream.ts:489`
  - body: Default Anthropic models already resolve to `https://api.anthropic.com` before this helper is called, and both streaming and native PDF pass that value as `baseUrl`. Because this chain treats any non-empty `baseUrl` as explicit, `ANTHROPIC_BASE_URL` is still ignored on the default no-config path this PR is meant to fix. Move the fallback to a source-aware resolver layer or pass base-url provenance into this helper.
