## Summary

Keep open. Current main still ignores ANTHROPIC_BASE_URL on the Anthropic streaming and native-PDF paths, and this PR is useful, but the latest patch is not mergeable because default Anthropic catalog models already carry https://api.anthropic.com as model.baseUrl, so the new env fallback is bypassed in the main production path.

## What This Changes

The PR exports and tests an Anthropic Messages URL resolver with ANTHROPIC_BASE_URL fallback and reuses that resolver for native Anthropic PDF requests.

## Best Possible Solution

Land a revised provider-routing fix that distinguishes user-configured Anthropic base URLs from catalog defaults, honors a non-empty ANTHROPIC_BASE_URL for default Anthropic streaming and native PDF calls, preserves explicit config precedence, shares one /v1/messages normalizer, and adds the required release-note entry.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: resolve a default Anthropic catalog model with ANTHROPIC_BASE_URL set and no models.providers.anthropic.baseUrl; model.baseUrl is already the public Anthropic URL, so the PR resolver treats it as explicit and never reaches the env fallback.

## Solution Assessment

No. Reusing one URL resolver is the right direction, but applying the env fallback only after model.baseUrl has been populated is too late for default Anthropic catalog models; the safer fix is to account for catalog-default provenance in model resolution or before the transport/PDF call.
