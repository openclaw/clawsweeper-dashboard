## Summary

Keep open. The PR addresses a real Anthropic routing gap, but the latest branch still applies the env fallback after the production path has already converted the catalog default into `model.baseUrl`, so default Anthropic streaming and native PDF calls would continue to treat `https://api.anthropic.com` as explicit and skip `ANTHROPIC_BASE_URL`; it also lacks the required changelog entry.

## What This Changes

The PR exports an Anthropic messages URL resolver with explicit/env/default precedence, adds resolver unit tests, and reuses that resolver for native Anthropic PDF requests.

## Best Possible Solution

Land a revised provider-routing fix that distinguishes user/provider-configured Anthropic base URLs from catalog defaults, honors a non-empty `ANTHROPIC_BASE_URL` for default Anthropic streaming and native PDF calls, preserves explicit config precedence, shares one `/v1/messages` normalization path, and adds focused regression tests plus an Unreleased Fixes changelog entry.

## Reproduction Assessment

Yes. With a default Anthropic catalog model and no `models.providers.anthropic.baseUrl`, current model resolution supplies `model.baseUrl: "https://api.anthropic.com"`; the PR helper treats that as explicit and therefore never consults `ANTHROPIC_BASE_URL` on the production streaming/PDF path.

## Solution Assessment

No. Reusing a shared URL normalizer is the right direction, but the current PR applies the env fallback after source provenance is lost; the safer fix is to resolve the env-backed default where catalog defaults are chosen, or pass explicit provenance into the resolver.
