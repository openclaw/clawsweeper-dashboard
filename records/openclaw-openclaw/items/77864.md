## Summary

Keep open: current main still has the Active Memory empty-status path, and an open linked PR already targets the likely warning-only `memory_search` root cause with closing syntax, so this issue should wait for that PR to be reviewed, proven, and merged or rejected.

## What This Changes

Active Memory should stop returning empty recall payloads for every query when memory search is degraded but still usable.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/78083 after it has deterministic warning-only watcher coverage and real after-fix Active Memory runtime proof, or replace it with an equivalent narrow fix if maintainers choose a different status/caching policy.

Yes, from source: a `memory_search` tool result with `warning` but no hard `disabled` or `error` currently satisfies the terminal unavailable condition and returns `status=empty`. I did not live-run the reporter’s exact Minimax/local-model setup.

## Solution Assessment

Yes for the current fix direction: hard unavailable handling should key off `disabled` or `error`, not advisory warning text. The issue itself should stay open because the closing PR is still open and still needs proof/review before merge.
