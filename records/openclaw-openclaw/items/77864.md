## Summary

Keep open: current main still has source-reproducible Active Memory paths that can produce terminal `status=empty summaryChars=0`, and https://github.com/openclaw/openclaw/pull/78083 explicitly closes this issue but is still open and blocked on proof/review.

## What This Changes

The issue reports that Active Memory repeatedly logs `status=empty summaryChars=0` for every query across several models, making the feature consume CPU/tokens without observable recall benefit.

## Best Possible Solution

Keep this issue open until https://github.com/openclaw/openclaw/pull/78083 or an equivalent replacement lands, then leave the broader `status=empty` taxonomy cleanup to https://github.com/openclaw/openclaw/issues/79812.

## Reproduction Assessment

Yes. From current source, a warning-only `memory_search` tool result becomes `debug.warning`, is classified as unavailable, and can win the terminal watcher race as `status=empty summaryChars=0`; I did not live-run the reporter's provider setup.

## Solution Assessment

Yes for the narrow bug: hard unavailability should be keyed off `disabled` or `error`, not an advisory warning. The open linked PR is the right direction but should not close this issue until it merges, and the broader no-useful-memory status work belongs in the linked follow-up issue.
