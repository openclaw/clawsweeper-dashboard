## Summary

Current main and shipped releases already fix the Active Memory `status=empty` behavior by splitting normal no-relevant-memory, unavailable, failed, timeout, and successful recall paths; the earlier narrow fix PR was superseded by the broader landed implementation.

## What This Changes

The issue requests that Active Memory stop returning `status=empty summaryChars=0` for all recall attempts and produce useful recall or accurate status output instead.

## Best Possible Solution

Keep the shipped Active Memory status split and close this report; any fresh current-release `status=empty` or timeout symptom should be filed as a new versioned report with logs and configuration.

## Reproduction Assessment

No current-main failure path remains: the old warning/zero-hit/no-summary paths are covered by current source and regression tests, and they no longer produce sticky `status=empty` behavior. The historical bug was source-proven and addressed by the landed status-classification fix.

## Solution Assessment

Yes. The landed implementation is the narrow maintainable fix because it stays inside `extensions/active-memory`, updates status classification and cache eligibility, and adds adjacent regression coverage without adding config or moving ownership into core.
