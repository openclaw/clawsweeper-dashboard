## Summary

Keep open: current main and the latest release still have the hidden 30s embedded-run restart cap, while this focused PR removes that cap, adds the missing forced SIGUSR1 deferral-timeout intent plumbing, and has no concrete correctness finding. The remaining blocker is maintainer policy judgment on the compatibility and availability tradeoff of honoring unbounded configured restart waits for embedded runs.

## What This Changes

The PR removes the embedded-run-specific 30s restart drain cap, passes the configured restart drain budget to embedded runs, and carries forced SIGUSR1 deferral-timeout intent through restart handling with focused tests.

## Best Possible Solution

Land the focused fix if maintainers accept the documented restart budget as the single contract for embedded runs; otherwise revise toward an explicit, documented embedded-run watchdog with tests.

## Reproduction Assessment

Yes, at source level. Current main and `v2026.5.22` cap active embedded-run restart drain at 30s even when the documented restart drain budget is longer or unbounded; I did not run local tests because this review is read-only.

## Solution Assessment

Yes, if maintainers accept the documented drain-budget contract. A separate embedded-run watchdog is a valid alternative policy choice, but it should be explicit and documented rather than hidden behind the current cap.
