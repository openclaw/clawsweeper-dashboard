- `P2`: This is a normal-priority gateway restart bug fix with limited but real operator impact.
- `merge-risk: 🚨 compatibility`: Existing setups that implicitly relied on the 30s embedded-run abort behavior will see longer or unbounded waits after merge.
- `merge-risk: 🚨 availability`: Unbounded restart waits can keep a gateway restart from reaching close or respawn until the active embedded run drains.

## Summary

Keep open. Current main still caps active embedded restart draining at 30 seconds despite documented longer or unbounded restart budgets, and this PR is a focused fix with supplied real runtime proof; the remaining question is maintainer acceptance of longer or unbounded embedded-run waits.

## What This Changes

The PR removes the embedded-run-specific 30s restart drain cap, passes the configured drain budget to embedded runs, and carries forced SIGUSR1 restart intent through deferral-timeout paths with regression coverage.

## Best Possible Solution

Land the focused restart-drain fix after maintainers explicitly accept that configured and unbounded restart budgets govern embedded runs too, or revise to a documented explicit embedded watchdog if that policy is preferred.

## Reproduction Assessment

Yes from source inspection: current main resolves the configured restart budget but clamps active embedded runs to a hidden 30s wait, while docs and the embedded-run wait helper both support unbounded waits. I did not run a live repro because this review is read-only.

## Solution Assessment

Yes, if the documented restart budget is intended to be authoritative; the PR removes a duplicate hidden cap rather than adding a parallel policy. If maintainers still want an embedded-run watchdog, the safer solution is an explicit documented cap.
