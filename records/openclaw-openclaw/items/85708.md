## Label Justifications

- `P2`: This is a focused gateway restart correctness fix with limited blast radius but real operator impact for long embedded runs.
- `merge-risk: 🚨 compatibility`: The PR changes existing restart behavior for setups that implicitly relied on the hidden 30s embedded-run cap.
- `merge-risk: 🚨 availability`: Unbounded restart waits can keep the gateway in a draining restart state indefinitely when an embedded run does not finish.

## Summary

Keep this PR open: current main still has the hidden 30s embedded-run restart cap, and the patch is a focused fix with adequate terminal proof and regression coverage. The remaining decision is maintainer acceptance of the availability/compatibility tradeoff for truly unbounded embedded-run waits.

## What This Changes

This PR removes the 30s embedded-run restart cap, passes the configured restart drain timeout to embedded runs, and carries SIGUSR1 deferral-timeout force intent through restart handling with focused tests.

## Best Possible Solution

Land the focused fix if maintainers accept the documented restart budget as the single source of truth for embedded runs; otherwise revise toward an explicit, documented embedded-run watchdog.

## Reproduction Assessment

Yes. Current main still routes active embedded runs through a 30s cap even when the documented restart drain budget is longer or unbounded.

## Solution Assessment

Yes, if maintainers accept the existing documented drain-budget contract. A separate explicit watchdog would be a different product choice, not a mechanical bug fix.
