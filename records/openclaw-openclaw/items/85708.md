## Label Justifications

- `P2`: This is a focused gateway restart correctness fix with limited blast radius but real operator impact for long embedded runs.
- `merge-risk: 🚨 compatibility`: The PR changes existing restart behavior for users who unknowingly depended on the hidden 30s embedded-run abort grace.
- `merge-risk: 🚨 availability`: Unbounded restart waits can leave the gateway in a draining restart state indefinitely while an embedded run remains active.

## Summary

Keep open for maintainer review: the patch appears focused and correct, with sufficient terminal proof and green current-head checks, but merging should explicitly accept the compatibility and availability tradeoff of letting embedded runs follow unbounded restart budgets.

## What This Changes

This PR removes the hidden 30s embedded-run restart cap, passes the configured restart drain timeout to embedded runs, and threads forced SIGUSR1 restart intents through deferral-timeout handling with regression tests.

## Best Possible Solution

Land the focused fix if maintainers accept the documented restart budget as the single source of truth for embedded runs; otherwise revise toward an explicit, documented embedded-run watchdog with tests.

## Reproduction Assessment

Yes. Current-main source shows active embedded runs are capped at 30s even when the documented restart drain budget is longer or unbounded; I did not run tests because this review is read-only.

## Solution Assessment

Yes, if maintainers accept the single restart-budget contract. A separate embedded-run watchdog is a valid alternative policy choice, but it should be explicit and documented rather than preserved as a hidden cap.
