- `merge-risk: 🚨 compatibility`: The PR changes existing restart behavior for installations that implicitly depended on the hidden 30s embedded-run cap.
- `merge-risk: 🚨 availability`: Unbounded restart waits can keep the gateway in a draining restart state indefinitely when an embedded run does not finish.

## Summary

Keep open for maintainer review: current main still has the hidden 30s embedded-run restart cap, and this PR appears to make a focused, tested fix, but merge needs an explicit maintainer call on whether unbounded restart waits should apply to embedded runs.

## What This Changes

The PR removes the embedded-run-specific 30s restart cap, passes the configured restart drain timeout to embedded runs, and adds SIGUSR1 restart intent plumbing plus regression tests.

## Best Possible Solution

Land this focused fix if maintainers accept the documented restart drain budget as the single source of truth; otherwise revise toward an explicit, documented embedded-run watchdog with tests.

## Reproduction Assessment

Yes from source inspection. Current main passes the configured restart drain budget to active tasks but caps active embedded runs at 30 seconds, while the docs describe the budget as applying to in-flight operations and allow 0 for indefinite waits.

## Solution Assessment

Yes if maintainers accept the documented drain-budget contract. A separate explicit watchdog would be a policy choice, not a mechanical correctness fix.
