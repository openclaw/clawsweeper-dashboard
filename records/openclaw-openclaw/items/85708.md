## Summary

Keep this PR open for maintainer review: it is a focused fix for a source-reproducible gateway restart mismatch, has sufficient terminal proof, and I found no blocking code or security findings. The remaining question is an intentional policy/upgrade choice: whether active embedded runs should inherit the documented unbounded restart budget or keep an explicit watchdog.

## What This Changes

The PR removes the hidden 30s active embedded-run restart cap, passes the configured restart drain timeout to embedded runs, and carries SIGUSR1 deferral-timeout force intent through the gateway restart loop.

## Best Possible Solution

Land the focused fix if maintainers accept the documented restart budget as the single source of truth for embedded runs; otherwise revise toward an explicit documented embedded-run watchdog.

Yes, if maintainers accept the existing documented drain-budget contract as the expected behavior. A separate embedded-run watchdog would be a different policy choice and should be explicit rather than hidden.
