## What This Changes

The PR adds `OOMPolicy=continue` to generated Linux systemd gateway units, aligns two public systemd examples, and covers the directive in the renderer test.

## Best Possible Solution

Land the focused generator, docs, and test change if maintainers accept the child-OOM availability behavior, while keeping the broader Linux gateway memory-policy work in the linked open issues.

## Reproduction Assessment

Yes, source-reproducible: current generated units omit `OOMPolicy=continue`, systemd documents that `continue` keeps the unit running after an OOM-killed unit process, and OpenClaw already raises eligible child `oom_score_adj`. I did not re-trigger a live host OOM.

## Solution Assessment

Yes for the narrow child-OOM restart problem: the canonical systemd unit generator plus matching docs and renderer coverage is the right layer. The broader gateway heap-growth and heavy-exec resource-boundary issues remain separate work.
