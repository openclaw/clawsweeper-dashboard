## Summary

Keep open: this is a plausible but broad first-party desktop feature, current main does not already implement it, and the PR still has source-verifiable contract/security blockers plus missing real behavior proof.

## What This Changes

This PR adds a Tauri v2 React/Rust desktop companion under `apps/desktop`, registers workspace/root desktop scripts, updates lockfiles, and wires tray status, Gateway health, restart, and embedded Control UI behavior.

## Best Possible Solution

Land a desktop MVP only after maintainers approve the first-party Tauri and restart model, then reuse canonical config, Control UI URL, auth, and restart contracts with synchronized docs, labeler coverage, and Linux/Windows validation proof.

Yes for the static review findings: the PR-head files can be compared directly with current main's config reader, Control UI link helper, Gateway runtime config, platform docs, and labeler. No full desktop runtime reproduction is available from the discussion because the contributor did not provide a desktop smoke run or build/typecheck proof beyond cargo fmt.

## Solution Assessment

No. A Tauri companion may be a viable direction, but this branch should not land while it reimplements canonical Gateway/config contracts, omits supported Control UI URL behavior, and restarts through an ambient shell command.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Load Gateway config through canonical contracts:** `apps/desktop/src-tauri/src/main.rs:296-383`
  - body: This reads `openclaw.json` with strict `serde_json` and hand-extracts selected fields, so valid OpenClaw configs using JSON5, `$include`, env substitution, or SecretRef-managed Gateway auth can be treated as missing or different from the running Gateway. The desktop app should consume an existing config snapshot, CLI contract, or Gateway dashboard contract instead of parsing a partial Rust copy.
  - confidence: 0.9
- **[P2] Include the configured Control UI base path:** `apps/desktop/src-tauri/src/main.rs:492-504`
  - body: `build_dashboard_url()` always embeds `/`, but current Gateway config supports `gateway.controlUi.basePath` and the canonical dashboard link helper appends that path. When the Control UI is mounted under a path such as `/openclaw`, the iframe opens the wrong route.
  - confidence: 0.9
- **[P2] Permit the dashboard URLs this app can generate:** `apps/desktop/src-tauri/tauri.conf.json:14`
  - body: The Rust URL builder can produce HTTPS dashboard URLs and custom-host URLs, but this CSP only allows HTTP/WS localhost and 127.0.0.1 frame/connect origins. Valid generated dashboard URLs can be blocked by the WebView policy.
  - confidence: 0.86
- **[P2] Avoid restarting through an ambient shell command:** `apps/desktop/src-tauri/src/main.rs:538-558`
  - body: The tray/UI restart action invokes `openclaw gateway restart` through `cmd` or `sh -lc`, so a first-party desktop control executes whichever `openclaw` the ambient shell resolves. Resolve a trusted CLI path or use a controlled Gateway restart surface before exposing this as a GUI action.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The diff introduces a concrete command-execution concern in the new desktop restart action.

Concerns:

- **[medium] Ambient shell restart execution:** `apps/desktop/src-tauri/src/main.rs:538`
  - body: `run_gateway_restart()` invokes `cmd /C openclaw gateway restart` on Windows and `sh -lc "openclaw gateway restart"` elsewhere. A first-party desktop app should not execute a PATH/cwd-hijackable command for a GUI restart action; it should resolve a trusted CLI path or use a controlled Gateway restart surface.
  - confidence: 0.88
