## Summary

Keep open. Current main and the latest release still miss the reported top-level `detail` error shape, and an open linked fix PR references this issue with closing syntax, so the issue should stay open until that PR lands or is replaced.

## What This Changes

The issue reports that OpenAI Codex/FastAPI-style `detail` error JSON can be forwarded into user chat, especially Telegram, instead of being formatted or suppressed.

## Best Possible Solution

Land, rebase, or replace the linked shared formatter fix with regression coverage; then revisit provider probe suppression only if unconfigured Codex errors still originate upstream.

## Reproduction Assessment

Yes for the parser leak: source inspection on current main shows top-level string `detail` payloads are neither classified nor message-extracted, so a short raw error can be returned unchanged. I did not live-reproduce the full Raspberry Pi/Telegram provider-spawn trigger.

## Solution Assessment

Yes for the immediate parser repair: recognizing top-level string `detail` in the shared assistant error formatter is the narrow channel-agnostic fix. Credential-gating unconfigured provider probes should remain a separate follow-up if it still reproduces after formatting is fixed.
