## Summary

Keep open. Current main still does not recognize top-level string `detail` payloads in the shared assistant error parser, so the reported raw JSON can still fall through; open PR #45216 remains the active linked implementation candidate.

## What This Changes

The issue asks OpenClaw to prevent OpenAI Codex/FastAPI `detail` error payloads from leaking as raw JSON into Telegram/user chat, especially for users without Codex configured.

## Best Possible Solution

Land or replace PR #45216's shared parser/formatter fix for top-level string `detail` payloads with regression tests, then add provider/ACP credential gating only if unconfigured Codex probes still fire.

## Reproduction Assessment

Yes for the source-level parser leak: `{"detail":"..."}` is not classified or message-extracted by current main, and short unclassified assistant errors can be returned raw. The exact live Telegram spawn/probe trigger was not reproduced in this read-only review.

## Solution Assessment

Yes for the first fix: recognizing top-level `detail` in the shared parser/formatter is the narrow maintainable repair across chat surfaces. If unconfigured Codex requests still occur afterward, that should be handled in provider/ACP credential gating rather than Telegram-specific filtering.
