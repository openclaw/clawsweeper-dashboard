## Summary

Keep this open: current main and v2026.6.6 have broad user-facing raw-error suppression, but the exact top-level `detail` payload still is not recognized as a structured API error, and the broader request to silently discard unconfigured-provider failures needs maintainer policy judgment.

## What This Changes

The issue asks OpenClaw to prevent OpenAI/Codex provider errors, especially top-level `detail` JSON payloads, from appearing in Telegram/user chat when the user did not configure that provider.

## Best Possible Solution

Decide the provider/channel policy first, then land a narrow shared fix that recognizes top-level string `detail` as provider error detail while keeping raw provider bodies out of user-facing payloads and applying any unconfigured-provider suppression only at the maintained routing boundary.

## Reproduction Assessment

Yes for a source-level parser gap: current main still does not classify a top-level string `detail` JSON object as an API error payload. No for the full live Raspberry Pi/Telegram trigger; I did not reproduce that environment or the unconfigured-provider spawn path.

## Solution Assessment

Unclear as a complete fix. Adding `detail` parsing is a narrow repair for the raw JSON shape, but the request to silently discard unconfigured-provider failures needs a maintainer decision because current shipped behavior uses generic user-facing failure copy instead.
