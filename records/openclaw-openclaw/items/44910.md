## Summary

Current main still fails to classify the reported FastAPI-style `{"detail":"..."}` payload, so raw JSON can bypass the established assistant-error formatting path. The matching upstream PR was closed unmerged, while its fork copy remains open and explicitly targets this issue; keep the issue open until that implementation is repaired and landed or replaced. The broader error-policy issue is related but does not subsume this narrow parser regression.

Likely related people: Vincent Koc (high confidence, current formatter refactor and recent area work), Peter Steinberger (high confidence, introduced the raw API error suppression behavior), and 助爪 (medium confidence, recent Codex server-error sanitization work).

## What This Changes

Recognize and safely handle Codex/FastAPI `detail` error payloads so raw provider JSON is not forwarded into user chat, including when Codex was not intentionally configured.

## Best Possible Solution

Land a focused parser regression fix for string `detail` payloads with false-positive coverage, then separately trace the resolved provider/model and credential path so Codex is never invoked merely because it exists in a catalog; keep detailed errors in logs and apply the maintainer-approved generic reply or silence policy at the channel boundary.

## Reproduction Assessment

Yes at source level: on current main and v2026.6.5, the exact `{"detail":"..."}` payload is not recognized by `parseApiErrorPayload` and can bypass raw-error formatting. The separate claim that an unconfigured Codex path initiated the request is not yet reproduced.

## Solution Assessment

Partly: narrow `detail` recognition is the best maintainable repair for the demonstrated JSON leak, but the linked patch only reformats the provider detail and does not prove the unexpected Codex invocation or settle whether the channel should reply generically or remain silent.
