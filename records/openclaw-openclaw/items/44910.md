## Summary

Keep open. Current main and the latest release still miss the reported top-level string `detail` error shape, and the linked OpenClaw fix PR is still open with closing syntax, so this issue should remain open until that PR or a replacement lands.

## What This Changes

The issue reports that Codex/OpenAI provider error payloads can leak raw JSON into user-facing Telegram chat when the provider was not intentionally configured or used.

## Best Possible Solution

Rebase, land, or replace the linked shared formatter fix with focused regression coverage; then investigate provider credential-gating only if unconfigured Codex errors still originate upstream.

## Reproduction Assessment

Yes for the source-level parser leak: current main still does not classify or message-extract a top-level string `detail` payload, so the reported short raw JSON can fall through unchanged. I did not live-reproduce the full Raspberry Pi/Telegram spawn trigger.

## Solution Assessment

Yes for the immediate parser repair: recognizing top-level string `detail` in the shared assistant error formatter is the narrow channel-agnostic fix. Broader suppression of unconfigured provider probes should stay separate unless the problem still reproduces after formatting is fixed.
