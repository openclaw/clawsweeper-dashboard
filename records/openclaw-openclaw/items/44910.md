## What This Changes

The issue requests preventing OpenAI Codex/FastAPI `detail` error payloads from leaking as raw JSON into Telegram/user chat when Codex is unconfigured.

## Best Possible Solution

Land, rebase, or replace the linked shared formatter fix with regression coverage; then revisit provider probe suppression only if errors still originate from unconfigured Codex paths.

## Reproduction Assessment

Yes for the parser leak: source inspection shows current main does not classify or extract top-level string `detail` payloads, and short unclassified errors can still return raw. I did not live-reproduce the Raspberry Pi/Telegram spawn trigger.

## Solution Assessment

Yes for the immediate fix: recognizing top-level string `detail` in the shared assistant error formatter is the narrow channel-agnostic repair. Credential-gating unconfigured provider probes should remain a separate follow-up only if it still reproduces after the formatter fix.
