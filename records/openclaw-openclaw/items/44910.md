## Summary

Keep this open: current main and v2026.6.6 broadly prevent raw assistant error bodies from being sent verbatim, but the exact top-level `detail` payload is still not parsed as structured provider error data and the requested silent discard of unconfigured-provider failures still needs maintainer policy judgment.

## What This Changes

The issue requests that OpenAI/Codex provider errors, including top-level `detail` JSON, not leak into Telegram chat when the user has not configured that provider.

## Best Possible Solution

Decide the provider/channel policy, then land a narrow shared `detail` parser repair plus any unconfigured-provider suppression at the maintained routing boundary with live Telegram or equivalent proof.

## Reproduction Assessment

Yes for the source-level parser gap: current main still does not classify top-level string `detail` JSON as an API error payload; no for the full live Raspberry Pi Telegram unconfigured-provider trigger, which was not reproduced here.

## Solution Assessment

Not yet as a complete fix: parser-only PR #622 is a useful narrow repair, but it formats the `detail` body rather than deciding whether unconfigured-provider failures should be silently discarded.
