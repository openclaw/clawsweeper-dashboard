## Summary

Keep open. Current main and `v2026.5.22` still miss the reported top-level string `detail` payload shape, the target-repo fix PR is closed unmerged, and the only open linked fix is a cross-repo fork PR rather than a landed OpenClaw change.

## What This Changes

The issue reports that Codex/OpenAI provider error payloads can leak as raw JSON into user chat and asks OpenClaw to stop forwarding errors from unconfigured providers.

## Best Possible Solution

Land or promote a focused formatter fix for top-level string `detail` payloads with regression coverage; if unconfigured Codex probing still reproduces afterward, track that as provider-routing follow-up alongside https://github.com/openclaw/openclaw/issues/39612 or a narrower provider issue.

## Reproduction Assessment

Yes for source-level reproduction: current main does not recognize or message-extract a top-level string `detail` payload, and the short raw-error fallback remains reachable. I did not live-reproduce the full Raspberry Pi/Telegram spawn trigger.

## Solution Assessment

Yes for the immediate leak: recognizing top-level string `detail` in the shared assistant error formatter is the narrow channel-agnostic fix. Broader suppression of unconfigured provider probes should stay separate unless the issue still reproduces after the formatter path is fixed.
