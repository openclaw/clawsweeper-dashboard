## What This Changes

The issue reports that Codex/OpenAI error payloads like `{"detail":"..."}` are forwarded verbatim into user chat even when Codex is not configured.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/45216 with focused regression coverage for top-level string `detail` payloads, then investigate provider credential-gating separately only if unconfigured Codex errors still originate upstream.

## Solution Assessment

Yes for the immediate leak: recognizing top-level string `detail` in the shared assistant error formatter is the narrow channel-agnostic fix. Broader suppression of unconfigured provider probes should stay separate unless the issue still reproduces after the formatter path is fixed.
