Keep open. Current main and the latest release still miss the reported top-level string `detail` payload shape, and the open linked fix PR at https://github.com/openclaw/openclaw/pull/45216 owns the focused formatter repair with closing syntax.

## What This Changes

The issue requests that Codex/OpenAI provider errors shaped as top-level `detail` JSON not be forwarded verbatim into user chat, especially for users without Codex configured.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/45216 with focused regression coverage for top-level string `detail` payloads, then track provider-probe suppression separately only if it still reproduces.

## Reproduction Assessment

Yes for source-level reproduction: current main does not recognize or message-extract a top-level string `detail` payload, and the short raw-error fallback remains reachable. I did not live-reproduce the full Raspberry Pi/Telegram spawn trigger.

## Solution Assessment

Yes for the immediate leak: recognizing top-level string `detail` in the shared assistant error formatter is the narrow channel-agnostic fix. Broader suppression of unconfigured provider probes should stay separate unless the issue still reproduces after the formatter path is fixed.
