## Summary

Keep open. Current main still does not recognize top-level string `detail` API error payloads in the shared assistant error parser, so the reported raw JSON can still fall through; https://github.com/openclaw/openclaw/pull/45216 remains the active linked fix candidate.

## What This Changes

The issue asks OpenClaw to stop forwarding unconfigured OpenAI/Codex provider error payloads, especially `{"detail":"..."}`, into user chat.

## Best Possible Solution

Land, rebase, or replace the linked shared parser fix with regression tests, then add provider credential gating only if unconfigured Codex probes still fire afterward.

## Reproduction Assessment

Yes for the source-level parser leak: current main does not classify or message-extract a top-level string `detail` payload, and short unclassified assistant errors can still be returned raw. The exact live Telegram spawn/probe trigger was not reproduced in this read-only review.

## Solution Assessment

Yes for the first repair: recognizing top-level `detail` in the shared parser/formatter is the narrow maintainable fix across chat surfaces. If unconfigured Codex requests still occur afterward, that should be handled in provider/ACP credential gating rather than Telegram-specific filtering.
