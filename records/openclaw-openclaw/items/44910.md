## Summary

Keep open. Current main still misses the reported top-level string `detail` error shape, the latest release has the same parser gap, and the open linked PR still lists this issue as closing work.

## What This Changes

The issue reports that Codex/OpenAI provider error payloads using a top-level `detail` field can be forwarded verbatim into a user's Telegram chat.

## Best Possible Solution

Land or replace the linked shared formatter fix with focused regression coverage, then investigate provider credential-gating only if unconfigured Codex errors still originate upstream.

## Reproduction Assessment

Yes for the source-level parser leak: current main still does not classify or message-extract a top-level string `detail` payload, and the short raw-error fallback remains reachable. I did not live-reproduce the full Raspberry Pi/Telegram spawn trigger in this read-only pass.

## Solution Assessment

Yes for the immediate formatter repair: recognizing top-level string `detail` in the shared assistant error formatter is the narrow channel-agnostic fix. Broader suppression of unconfigured provider probes should stay separate unless the problem still reproduces after formatting is fixed.
