## Summary

Keep this open: current main still lacks structured handling for top-level FastAPI-style `detail` error payloads, the Codex provider catalog still attempts live app-server discovery unless explicitly disabled, and the linked parser PR is open but conflicting while the broader unconfigured-provider suppression policy remains unresolved.

## What This Changes

The issue requests that OpenAI Codex/provider errors from unconfigured providers stop leaking raw JSON into Telegram/user chat and be gated or suppressed appropriately.

## Best Possible Solution

Land a conflict-free fix that recognizes `detail` payloads as provider errors and pair it with a maintainer-approved Codex discovery/auth policy so unconfigured-provider failures do not produce user-visible chat replies.

## Reproduction Assessment

We have a medium-confidence source reproduction for the parser gap: current main does not classify top-level `{detail: "..."}` payloads as API errors. We do not yet have a high-confidence live reproduction of the full no-credentials Telegram leak on current main.

## Solution Assessment

No, the current linked parser-only direction is useful but incomplete. The best fix also needs a maintainer decision on whether Codex model discovery should be credential-gated, disabled by default, or allowed with safe internal-only failure handling.
