## Summary

Keep open. Current main still misses the reported top-level `detail` API error shape in the shared assistant error parser, so short Codex/FastAPI JSON can still fall through as user-visible raw error text; open PR #45216 remains the active linked implementation candidate.

## What This Changes

The issue asks OpenClaw to prevent unconfigured OpenAI/Codex provider errors, especially `{"detail":"..."}` payloads, from leaking into Telegram chat.

## Best Possible Solution

Land or replace the shared parser/formatter fix that recognizes top-level string `detail` payloads with regression coverage, then separately add provider/ACP credential gating if unconfigured Codex probes still fire.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the parser leak from source: a short `{"detail":"..."}` payload is not classified or message-extracted and can fall through to raw assistant error text. The live Telegram spawn/probe trigger remains unverified.

## Solution Assessment

Is this the best way to solve the issue? Yes for the first fix: recognizing `detail` in the shared parser/formatter is the narrowest cross-surface repair. If unconfigured Codex requests still occur afterward, the follow-up belongs in provider/ACP credential gating rather than Telegram-specific filtering.
