## Summary

Keep this open: current main and the latest release still do not parse top-level FastAPI-style `detail` error payloads as structured provider errors, the reported no-credentials Telegram trigger was not live-reproduced, and an open linked PR already claims the narrow parser repair while leaving the silent unconfigured-provider policy unresolved.

The issue reports that raw OpenAI Codex provider errors can be forwarded into user chat for users who have not configured Codex/OpenAI.

## Best Possible Solution

Define the provider/channel policy, then land or replace the linked `detail` parser repair with focused regression tests and live Telegram or equivalent proof that unconfigured-provider failures do not leak raw or confusing replies.

## Solution Assessment

No, not as a cleanup close. The linked parser repair is a useful narrow piece, but the complete fix needs a maintainer decision about silent suppression versus generic user-facing failure handling for unconfigured-provider errors.
