## Summary

Keep this open: current main and the latest release have generic protection against exact raw assistant-error passthrough, but top-level `detail` payloads are still not parsed as structured provider errors, the requested silent handling for unconfigured-provider failures remains a provider/channel policy decision, and an open linked PR still claims the parser repair.

## What This Changes

The issue asks OpenClaw to prevent OpenAI/Codex provider errors, especially top-level `detail` JSON from unconfigured providers, from reaching Telegram or other user chats.

## Best Possible Solution

Define the provider/channel policy, then land or replace the narrow `detail` parser repair with focused tests and live Telegram or equivalent proof that unconfigured-provider failures do not produce raw or confusing user-visible replies.

## Reproduction Assessment

Partly. Source inspection shows the top-level `detail` parser gap and the current generic raw-passthrough guard, but the original Raspberry Pi Telegram/unconfigured-provider trigger was not live-reproduced.

## Solution Assessment

No, not as a cleanup close. The parser repair is useful, but the complete fix needs a maintainer decision about silent suppression versus generic user-facing failure handling for unconfigured-provider errors.
