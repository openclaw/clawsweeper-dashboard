## Summary

Keep open: current main and the latest release partially mitigate raw provider-error delivery with a generic user-facing guard, but the specific top-level Codex/FastAPI `detail` payload still is not parsed structurally, the requested unconfigured-provider silence is still a product/routing decision, and an open cross-repository PR remains linked as a candidate parser fix.

## What This Changes

The issue reports that OpenAI Codex/FastAPI-style provider errors can leak as raw JSON into Telegram chat for users who did not configure OpenAI or Codex.

## Best Possible Solution

Keep the centralized safe delivery guard, add or land a focused top-level `detail` parser repair with regression coverage, and have maintainers decide whether unconfigured-provider failures should be prevented earlier, silently dropped, or shown as generic diagnostics.

## Reproduction Assessment

Partly yes: current source still shows that a bare top-level `detail` JSON payload is not recognized as a structured API error. I did not establish the full current-main Raspberry Pi/Telegram live trigger in this read-only pass.

## Solution Assessment

Unclear: recognizing top-level string `detail` in the shared assistant error formatter is the narrow parser repair, but silently discarding unconfigured-provider errors needs a maintainer decision about provider discovery and channel silence.
