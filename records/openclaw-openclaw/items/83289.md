## Label Justifications

- `P2`: The issue is a real cross-platform bug in a bounded SSH helper surface, but it does not make the core runtime unusable for all users.
- `impact:security`: The requested fix is about choosing a trusted executable for an SSH subprocess rather than relying on untrusted path resolution.

## Summary

Keep open: current `main` and the latest release still hard-code `/usr/bin/ssh` in both SSH helper paths, so the Windows OpenSSH failure remains source-reproducible.

## What This Changes

The issue asks for SSH config and tunnel helpers to resolve a trusted platform SSH executable instead of hard-coding `/usr/bin/ssh`.

## Best Possible Solution

Add one shared internal SSH executable resolver for POSIX and Windows trusted locations, reuse it from both helpers, and keep arbitrary PATH fallback out of the default behavior.

## Reproduction Assessment

Yes. Source inspection on current `main` shows both helper paths spawn `/usr/bin/ssh`, and release-tag inspection shows the same behavior in `v2026.5.18`.

## Solution Assessment

Yes, the requested shared trusted resolver is the right maintainable shape. The unsafe alternative would be a plain PATH search or duplicated ad hoc platform branches in each helper.
