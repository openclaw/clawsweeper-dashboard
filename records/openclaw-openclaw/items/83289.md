## Summary

Keep open: current main still hard-codes `/usr/bin/ssh` in both SSH helper paths, so the Windows OpenSSH failure is source-reproducible. The remaining work is a narrow platform-aware resolver, but it should preserve the repository’s trusted-binary policy.

## What This Changes

SSH config and tunnel helpers should resolve a trusted platform SSH executable instead of assuming `/usr/bin/ssh`.

## Best Possible Solution

Add one internal SSH executable resolver that covers POSIX and Windows trusted locations, reuse it from both helpers, and keep arbitrary PATH fallback out of the default path.

## Reproduction Assessment

Yes, source-reproducible: on a host without `/usr/bin/ssh`, both helper calls try to spawn that nonexistent path. A focused regression test can mock platform/filesystem/spawn behavior without needing a full Windows machine.

## Solution Assessment

Yes, the requested direction is the maintainable shape: resolve a trusted platform SSH binary once and share it. The unsafe alternative would be a plain PATH search or duplicated ad hoc platform branches in each helper.
