## Summary

Keep open: current main and the latest release still hard-code `/usr/bin/ssh` in both SSH helper paths, and the open closing PR at https://github.com/openclaw/openclaw/pull/93030 is the active fix candidate rather than a reason to close this issue early.

## What This Changes

SSH config and tunnel helpers should resolve a trusted platform SSH executable instead of assuming the Unix-only `/usr/bin/ssh` path, with a clear diagnostic when none is available.

## Best Possible Solution

Keep this issue open as the canonical bug until https://github.com/openclaw/openclaw/pull/93030 lands or maintainers choose an equivalent trusted-resolver fix with security-boundary approval.

## Reproduction Assessment

Yes. Source inspection of current main shows both helpers still spawn `/usr/bin/ssh`, which fails on Windows hosts where built-in OpenSSH lives under `C:\Windows\System32\OpenSSH\ssh.exe`; I did not run a live Windows repro in this read-only review.

## Solution Assessment

Yes for the requested code direction: use the existing trusted system-binary resolver rather than PATH lookup or duplicated platform branches. The exact `System32\OpenSSH` trust expansion still needs maintainer security-boundary acceptance before the linked fix lands.
