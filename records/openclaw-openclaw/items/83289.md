## Summary

The report is valid on current main: the SSH config and tunnel helpers still hard-code `/usr/bin/ssh`, while OpenClaw already has a trusted system-binary resolver intended for this class of spawn. This is narrow enough for a focused fix PR, but it should stay open until the helpers resolve SSH through the trusted resolver and cover Windows OpenSSH explicitly.

## What This Changes

SSH config and tunnel helpers should resolve a trusted platform SSH executable instead of hard-coding `/usr/bin/ssh`.

## Best Possible Solution

Use the trusted system-binary resolver for SSH, include the Windows OpenSSH system directory in the trusted search path, and return a clear missing-SSH diagnostic from both config resolution and tunnel startup.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: on a host without `/usr/bin/ssh`, the config and tunnel helpers pass that Unix path directly to `spawn`. A live Windows run was not executed in this read-only pass.

## Solution Assessment

Yes, the maintainable fix is narrow: resolve `ssh` through the existing trusted system-binary mechanism, extend that mechanism for the Windows OpenSSH directory if needed, and add focused regression coverage instead of adding a user-facing config knob.
