## Summary

Keep open: current main still hard-codes `/usr/bin/ssh` in both SSH helper paths, so the reported Windows/non-FHS failure remains source-reproducible. The fix shape is narrow, but the trusted executable locations are a subprocess security boundary that should get maintainer review before automation changes it.

## What This Changes

The issue asks SSH config and tunnel helpers to resolve a trusted platform SSH executable instead of assuming `/usr/bin/ssh`.

## Best Possible Solution

Add one shared SSH executable resolver that reuses or extends `resolveSystemBin`, covers trusted POSIX and Windows OpenSSH locations, returns a clear missing-binary diagnostic, and is used by both SSH helpers without arbitrary PATH fallback.

## Reproduction Assessment

Yes, source-reproducible: on any host without `/usr/bin/ssh`, both helper paths attempt that absolute executable. A focused regression test can mock platform/filesystem/spawn behavior without requiring a live Windows machine.

## Solution Assessment

Yes, the requested direction is the maintainable fix shape, but the implementation should route through the trusted system-binary policy rather than a bare PATH lookup. Maintainer review should confirm the Windows OpenSSH trust roots before code changes land.
