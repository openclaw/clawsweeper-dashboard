## Label Justifications

- `P2`: This is a normal-priority portability and security-boundary bug in a documented SSH gateway path with limited blast radius.
- `impact:security`: The fix changes trusted subprocess binary selection, where an unsafe PATH fallback could weaken the command-execution trust boundary.

## Summary

Keep open: current main still spawns `/usr/bin/ssh` from both SSH helper paths, so the Windows OpenSSH portability failure remains source-reproducible and should be fixed with a trusted platform resolver.

## What This Changes

The issue asks SSH config and tunnel helpers to resolve a trusted platform-specific SSH executable instead of hard-coding a Unix-only path.

## Best Possible Solution

Add one internal trusted SSH executable resolver shared by both helpers, covering POSIX and Windows OpenSSH locations while avoiding arbitrary PATH trust by default.

## Reproduction Assessment

Yes, source-reproducible: on any host without `/usr/bin/ssh`, both SSH helper calls attempt to spawn that nonexistent absolute path. A focused regression test can mock platform, filesystem, and spawn behavior without requiring a live Windows host.

## Solution Assessment

Yes on the requested direction: a shared trusted platform resolver is the maintainable fix. The earlier PATH-only approach in https://github.com/openclaw/openclaw/pull/20330 is not the best fit because current safe-bin policy explicitly avoids trusting arbitrary PATH by default.
