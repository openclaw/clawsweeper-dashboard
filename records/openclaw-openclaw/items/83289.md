## Summary

Keep open: current main still spawns `/usr/bin/ssh` from both SSH helper paths, so the Windows OpenSSH failure remains source-reproducible. A prior closed PR tried the simpler PATH-based fix, but the current issue correctly needs a trusted platform resolver rather than a bare PATH fallback.

## What This Changes

The issue requests platform-aware SSH helper resolution and clear missing-binary diagnostics instead of hard-coding `/usr/bin/ssh`.

## Best Possible Solution

Add one internal SSH executable resolver over trusted POSIX and Windows locations, reuse it from both helpers, and return a clear diagnostic when no trusted SSH client is available.

## Reproduction Assessment

Yes, source-reproducible: both current helper paths call `spawn` with `/usr/bin/ssh`, which cannot reach Windows OpenSSH in the Windows system location. A focused regression test can mock platform and executable resolution without a live Windows machine.

## Solution Assessment

Yes, the requested resolver shape is the maintainable fix, provided it uses trusted platform locations rather than a plain PATH search. The prior PATH-only PR is useful context but should not be adopted as-is.
