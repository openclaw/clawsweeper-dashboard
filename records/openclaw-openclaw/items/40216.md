## Summary

Keep open: the underlying CJK context-pruning bug still has source-level gaps on current main, but this PR should not merge as-is because it edits the obsolete `pi-extensions` path, mixes weighted soft-trim length with raw extraction units, and lacks after-fix real behavior proof.

## What This Changes

The PR adds a local CJK-aware character estimator to the legacy `pi-extensions` context-pruning pruner, replaces several token-estimation `.length` calls, and adds estimator tests.

## Best Possible Solution

Port the CJK-aware sizing fix to the active `src/agents/pi-hooks/context-pruning` pruner using `src/utils/cjk-chars.ts`, handle soft-trim budgets/extraction and tool-call arguments consistently, add focused regressions, and require redacted real behavior proof before merge.

## Reproduction Assessment

Yes, at source level. On current main, the active pruner still uses raw `p.length` for soft-trim eligibility and raw `JSON.stringify(...).length` for tool-call arguments, so CJK-heavy tool results or arguments remain underestimated; no live runtime test was run in this read-only review.

## Solution Assessment

No. The direction is right, but changing the legacy `pi-extensions` pruner is not the maintainable fix; the fix should target the active `pi-hooks` pruner and keep threshold, extraction, and accounting units consistent.
