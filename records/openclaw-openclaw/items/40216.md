## Summary

Keep open: the CJK token-estimation bug is still source-reproducible, but this PR is not mergeable as-is because it patches the removed `pi-extensions` pruner while current main runs the `pi-hooks` pruner and still has CJK gaps in soft-trim eligibility and serialized tool-call argument sizing.

## What This Changes

The PR adds a local weighted CJK/non-Latin string estimator to the old context-pruning pruner, applies it to token-estimation paths, and adds estimator tests.

## Best Possible Solution

Port the useful behavior to `src/agents/pi-hooks/context-pruning`, reuse `src/utils/cjk-chars.ts`, and add focused regressions for CJK-heavy soft-trim decisions plus serialized tool-call arguments.

## Reproduction Assessment

Yes, at source level. Current main can count a CJK-heavy prunable tool result below `softTrim.maxChars` in raw UTF-16 units while its weighted size exceeds the threshold, and assistant tool-call arguments with CJK text are still sized with raw `JSON.stringify(...).length`.

## Solution Assessment

No. The requested direction is right, but this PR patches an obsolete path and the soft-trim change needs a consistent weighted-budget extraction strategy in the active `pi-hooks` pruner.
