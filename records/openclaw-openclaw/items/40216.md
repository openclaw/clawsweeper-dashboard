## Summary

Keep open: the CJK pruning bug is still source-reproducible on current main, but this PR is not mergeable as-is because it edits the removed `pi-extensions` path and its soft-trim change mixes weighted eligibility with raw extraction budgets.

## What This Changes

The PR branch adds a local CJK/non-Latin weighted estimator to the old `pi-extensions` context pruner, applies it to token-estimation paths, and adds estimator tests.

## Best Possible Solution

Port the useful behavior to `src/agents/pi-hooks/context-pruning`, reuse `src/utils/cjk-chars.ts`, and add focused regressions for CJK-heavy tool results and serialized tool-call arguments.

## Reproduction Assessment

Yes, at source level. On current main, a CJK-heavy prunable tool result can have raw length below `softTrim.maxChars` while its weighted size exceeds the threshold, and assistant tool-call arguments with CJK text are still counted with raw `JSON.stringify(...).length`.

## Solution Assessment

No. The requested direction is right, but the PR patches an obsolete path and changes soft-trim eligibility without also making head/tail extraction use the same budget unit.
