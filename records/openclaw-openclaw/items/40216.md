## Summary

Keep open: the underlying CJK pruner bug still has source-level gaps in current main, but this PR is not mergeable as-is because it edits the removed `pi-extensions` pruner, leaves soft-trim unit semantics unsafe, and lacks real behavior proof.

## What This Changes

The branch adds a local CJK/non-Latin weighted string-length estimator to the old `pi-extensions` context-pruning pruner and tests it.

## Best Possible Solution

Port the CJK-aware sizing fix to `src/agents/pi-hooks/context-pruning`, reuse `src/utils/cjk-chars.ts`, make soft-trim budgets and extraction use one consistent unit, and add focused regressions.

## Reproduction Assessment

Yes, at source level. On current main, the active pruner still uses raw UTF-16 length for soft-trim eligibility and serialized tool-call arguments, so CJK-heavy tool results or arguments remain underestimated; no runtime test was run because this was a read-only review.

## Solution Assessment

No. The direction is right, but this branch edits obsolete `pi-extensions` files; the maintainable fix is a narrow port to the active `pi-hooks` pruner using the shared estimator with consistent soft-trim units.
