## Summary

Keep open. The PR is useful source work for a real CJK context-pruning gap, but it is not mergeable as-is: current main moved the active runtime to `src/agents/pi-hooks/context-pruning`, while this PR edits the removed `pi-extensions` path, and its soft-trim change mixes weighted length estimates with raw UTF-16 slicing budgets.

## What This Changes

The PR adds a CJK/non-Latin weighted string estimator to the context pruner, uses it in message and tool-result token estimation paths, updates soft-trim length estimation, and adds estimator tests under the old `pi-extensions` context-pruning path.

## Best Possible Solution

Port the useful behavior to the active `src/agents/pi-hooks/context-pruning` implementation, reuse the existing `src/utils/cjk-chars.ts` helper, apply weighted estimation to serialized tool-call arguments and soft-trim eligibility, and make soft-trim extraction enforce the same weighted budget unit with regression tests.

## Reproduction Assessment

Yes. A source-level reproduction exists on current main: use a CJK-heavy prunable tool result whose raw length is below `softTrim.maxChars` but whose weighted estimate exceeds it, or an assistant tool call with CJK arguments; both active paths still use raw string length.

## Solution Assessment

No. The requested direction is right, but this PR is not the best implementation because it patches an obsolete path and changes soft-trim eligibility without updating raw head/tail extraction semantics.
