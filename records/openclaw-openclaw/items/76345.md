## Summary

Keep open. The PR is aimed at the right bug and the exact-head checks visible through the GitHub API are green, but the breaker state is still scoped inside one `runEmbeddedPiAgent` invocation, so model fallback can create a fresh counter and exceed the advertised five-call cap across the fallback chain.

## What This Changes

The PR adds a five-attempt zero-output idle-timeout breaker helper, run-loop integration, unit tests, and a changelog entry for the embedded agent retry loop.

## Best Possible Solution

Land a breaker whose state survives every retry path that can spend another paid model call, including model fallback, with tests covering fallback-chain exhaustion and reset semantics.

## Reproduction Assessment

Yes, by source inspection: current main has only the per-attempt stream idle watchdog and a broad 32-160 run-loop backstop, while the linked bug supplies concrete zero-output timeout storm evidence. The PR-specific gap is also source-reproducible because model fallback calls `runEmbeddedPiAgent` again and resets the PR's in-function breaker state.

## Solution Assessment

No, not as currently written. The helper boundary is maintainable for retries within one embedded-run invocation, but the state needs to be hoisted or passed through the model-fallback scope, or the changelog and behavior claim must be narrowed.
