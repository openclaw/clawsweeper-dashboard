## Summary

Keep open. The PR targets a real paid-call retry-loop bug, but the proposed breaker is scoped to a single idle-timeout wrapper while current main recreates that wrapper inside the outer embedded-run retry loop, so the cap can reset before solving the reported cost spike.

## What This Changes

This PR adds a default five-timeout circuit breaker to `streamWithIdleTimeout` plus wrapper-level unit tests for consecutive idle-timeout retry storms.

## Best Possible Solution

Move the consecutive idle-timeout cap to state that spans embedded-run retry attempts, preserve the successful-chunk reset and existing abort path, and document the user-facing fix in the changelog.

## Reproduction Assessment

Yes. Source inspection shows current main's outer retry loop creates a fresh embedded attempt and idle-timeout wrapper, while the PR counter is wrapper-local; I did not run a live provider or Vitest.

## Solution Assessment

No. The wrapper-local counter is not the narrowest reliable fix for the reported retry-loop cost spike; the breaker state should live in, or be shared across, the retry lifecycle that performs repeated embedded attempts.
