## Summary

Keep open. The PR targets a valid cost-control bug, but the proposed breaker is scoped to one `streamWithIdleTimeout` wrapper while OpenClaw's outer run retry path creates a fresh embedded attempt and wrapper per retry, so the advertised cap can reset before it bounds the real paid-call loop; it also lacks the required changelog entry.

## What This Changes

Adds a default five-timeout circuit breaker to `streamWithIdleTimeout` plus wrapper-level unit tests for consecutive idle-timeout retry storms.

## Best Possible Solution

Move the consecutive idle-timeout cap to state that spans `runEmbeddedPiAgent` retry attempts, keep slow responsive streams and existing failover behavior working, and add regression coverage plus a changelog entry.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: the PR's counter is wrapper-local, while current main recreates the embedded attempt and idle-timeout wrapper inside the outer retry loop. I did not run a live provider or Vitest in this read-only review.

## Solution Assessment

No. A wrapper-local counter is not the best fix for the reported outer retry-loop cost spike; the cap should live at, or be shared across, the retry lifecycle that performs repeated embedded attempts.
