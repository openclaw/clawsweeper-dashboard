## Summary

Keep open. Current main still lacks a dedicated LLM idle-timeout cost breaker, and this PR is directionally useful, but the patch does not yet preserve the breaker state across the model-fallback boundary and has an unresolved output-token semantics issue.

## What This Changes

The branch adds a run-loop idle-timeout breaker helper, tests, and changelog entry to halt after five consecutive idle-timeout attempts with zero output tokens.

## Best Possible Solution

Carry the breaker state through the model-fallback boundary, add coverage for fallback reset behavior, and settle the output-token versus visible-output semantics before landing.

## Reproduction Assessment

Yes, by source inspection. Current main only has a broad run-loop retry guard and same-model idle retry state, while the PR path shows the proposed state resets when model fallback creates a fresh embedded run.

## Solution Assessment

No, not yet. The run-loop helper is the right local shape, but the state must survive the actual model-fallback path or the advertised five-call bound becomes per fallback model rather than per incident.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Keep breaker state across model fallback:** `src/agents/pi-embedded-runner/run.ts:769`
  - body: The breaker state is allocated inside runEmbeddedPiAgent, but model fallback calls runAgentAttempt again, which creates a fresh runEmbeddedPiAgent for each fallback model. When the breaker trips and fallback is configured, the next fallback starts with consecutive=0, so the advertised five-call cap across the profile/model chain becomes five calls per fallback model. Hoist or carry this state through the model-fallback scope and add regression coverage for that path.
  - confidence: 0.87
- **[P3] Use a liveliness signal that excludes partial tool args:** `src/agents/pi-embedded-runner/run.ts:1183`
  - body: run.ts feeds aggregate attemptUsage.output into the breaker. That count can include completion/output tokens from streamed tool-call arguments, so a model that emits partial tool args and then stalls can reset the breaker without producing completed user-visible output. Count a completed/visible assistant signal or explicitly cover the chosen semantics before relying on this reset.
  - confidence: 0.72
