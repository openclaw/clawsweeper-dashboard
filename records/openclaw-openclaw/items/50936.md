## Summary

Keep open. Current main still drops `location.update` node events, so the PR addresses a real gateway/iOS gap, and the latest head fixes the earlier prompt-delivery and iOS info-level GPS logging concerns. It still queues node-originated location data as a trusted system event, which crosses the existing node-event prompt trust boundary, and it is missing the required changelog entry for a user-facing feature.

## What This Changes

The PR adds iOS significant-location logging, a gateway `location.update` node-event handler that enqueues a location system event and wakes heartbeat, heartbeat prompt/reason support for `location:*` events, and node/gateway documentation.

## Best Possible Solution

Land a hardened version of the feature: keep the finite coordinate validation, sanitized `source`, explicit location heartbeat prompt, and iOS debug-level coordinate logging, but enqueue the node-originated summary with `trusted: false`, add focused gateway and heartbeat regression tests, preserve current main heartbeat safety behavior during rebase, and add the required changelog entry.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: current main iOS emits `location.update`, while current main gateway has no matching handler and falls through the default case without enqueueing or waking heartbeat.

## Solution Assessment

No, not as-is. The PR is the right narrow product direction, but the node-originated location payload needs to use the existing untrusted system-event boundary before this is the maintainable fix.

## Security Review

Status: needs_attention

Summary: The diff introduces a prompt-trust boundary issue by queueing node-provided location data as trusted system context.

Concerns:

- **[medium] Node location event is trusted by default:** `src/gateway/server-node-events.ts:612`
