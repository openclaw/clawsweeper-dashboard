## Summary

Keep this PR open. Current main still has the iOS producer for `location.update` while the gateway drops that event, so the PR is not obsolete; however, the current PR head is not merge-ready because it queues node-provided location text as a trusted system event. The later commits did address the earlier GPS info-log and source-string sanitization comments, but the remaining trust-boundary issue needs maintainer review before merge.

## What This Changes

The PR adds iOS logging for significant location changes, a gateway `location.update` node-event handler, heartbeat wake/prompt classification for location events, and docs for passive node location updates.

## Best Possible Solution

Land a hardened version of this feature: validate finite coordinates, sanitize all node-controlled display fields, enqueue the location summary as untrusted prompt data, preserve current heartbeat untrusted-event ownership handling, add focused gateway and heartbeat tests, and keep the docs aligned with the shipped passive-location behavior.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: on current main the iOS app emits `location.update`, and a matching `node.event` reaches `handleNodeEvent` with no `location.update` case, so it falls through the default path without enqueueing or waking heartbeat.

## Solution Assessment

No, not as-is. The PR takes the right product direction and fixes the missing prompt path, but the new node-originated system event should follow the existing untrusted node-event pattern before this is the narrowest maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Mark location events untrusted before queuing:** `src/gateway/server-node-events.ts:612-615`
  - body: `enqueueSystemEvent` defaults entries to trusted, so this new node-originated location summary will be formatted as `System:` and will not trigger the current untrusted-event downgrade path. Match the notification and exec event handlers by passing `trusted: false` when queueing location updates.
  - confidence: 0.89

## Security Review

Status: needs_attention

Summary: The PR introduces a prompt-trust boundary concern by queueing node-provided location data as a trusted system event.

Concerns:

- **[medium] Node location event is trusted by default:** `src/gateway/server-node-events.ts:612`
  - body: The new handler queues a summary built from node event payload fields without `trusted:false`; because system events default to trusted, a compromised or buggy paired node can put data into trusted `System:` prompt context instead of the existing `System (untrusted):` path used for node notifications and exec events.
  - confidence: 0.89
