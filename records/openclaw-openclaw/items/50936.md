## Summary

Keep open: current main still lacks gateway support for the iOS `location.update` event, but this PR is not merge-ready because the new node-originated location system event remains trusted, real behavior proof is missing, and the branch needs to be rebased onto current heartbeat wake plumbing.

## What This Changes

The PR adds iOS/gateway logging, a gateway `location.update` node-event handler that queues a location system event and wakes heartbeat, heartbeat prompt/provider handling, and node location documentation.

## Best Possible Solution

Land a rebased implementation that validates and sanitizes location payloads, queues them as untrusted system events, uses current heartbeat wake/source plumbing, embeds the location event text for the agent, and documents the node event contract.

## Reproduction Assessment

Yes, source-reproducible: current main's iOS monitor sends `location.update`, while targeted source search shows no gateway or heartbeat handler. I did not run a live iOS device or simulator path in this read-only review.

## Solution Assessment

No, not as currently proposed. The direction is narrow and useful, but omitting `trusted: false` violates the existing node-event trust boundary and the branch needs rebase onto the current wake/source contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Mark location events as untrusted:** `src/gateway/server-node-events.ts:612-615`
  - body: This event comes from `node.event`, but the new enqueue call omits `trusted: false`, so `enqueueSystemEvent` defaults it to trusted and heartbeat will not force owner privileges off for node-supplied location text. Notification and remote exec node events already pass `trusted: false`; do the same here and cover the location path.
  - confidence: 0.9
