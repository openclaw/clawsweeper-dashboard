## Summary

Keep open. Current main still drops the iOS `location.update` event, but this PR should not merge as-is because the latest head queues node-originated location text as trusted system context and still needs the required changelog entry.

## What This Changes

The PR adds iOS significant-location logging, a gateway `location.update` node-event handler, heartbeat prompt/reason support for location events, and node/gateway documentation.

## Best Possible Solution

Land a hardened version that marks location events untrusted, keeps the sanitized source and explicit heartbeat prompt path, adds focused gateway/heartbeat regression coverage, rebases onto current main, and includes a single-line changelog entry.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: current main iOS emits `location.update`, while current main gateway has no matching handler and falls through the default case without enqueueing or waking heartbeat.

## Solution Assessment

No, not as-is. The PR is the right narrow direction, but node-originated location payloads must use the existing untrusted system-event boundary and the branch needs changelog/rebase cleanup before merge.

## Security Review

Status: needs_attention

Summary: The diff introduces a prompt-trust boundary issue by queueing node-provided location data as trusted system context.

Concerns:

- **[medium] Node location event is trusted by default:** `src/gateway/server-node-events.ts:612`
