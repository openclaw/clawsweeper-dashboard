## Summary

Keep open. Current main still lacks gateway handling while iOS already sends the event, so this PR addresses a real gap, but it should not merge as-is because node-originated location text remains trusted/partly unsanitized and no after-fix real behavior proof is attached.

## What This Changes

This PR adds iOS logging, gateway `location.update` node-event handling, heartbeat wake/prompt/provider plumbing, tests, and node/gateway docs for passive location updates.

## Best Possible Solution

Land a rebased implementation that validates location payloads, sanitizes node-controlled text, queues location summaries as untrusted system events, uses current typed heartbeat wake/prompt contracts, and documents the node event behavior.

## Reproduction Assessment

Yes, source-level. Current main's iOS monitor sends `location.update`, while targeted gateway/heartbeat searches show no current handler or prompt path; I did not run a live iOS/gateway flow in this read-only review.

## Solution Assessment

No, not as currently proposed. The direction is narrow and useful, but the branch must preserve the untrusted node-event boundary, neutralize system markers in node-controlled text, and rebase onto the current heartbeat wake contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Mark location updates as untrusted:** `src/gateway/server-node-events.ts:612-615`
  - body: This event is accepted from `node.event`, but the enqueue call omits `trusted: false`. `enqueueSystemEvent` defaults to trusted, while current notification and exec node events explicitly mark node-supplied text untrusted so heartbeat can force owner privileges off.
  - confidence: 0.92

  - confidence: 0.82
