## Summary

Keep open: current main still has the iOS `location.update` sender without gateway/heartbeat handling, and this PR is useful, but the proposed patch is not merge-ready because it lacks after-fix real behavior proof and does not yet preserve the current untrusted system-event boundary.

## What This Changes

The PR adds iOS logging plus gateway, heartbeat, tests, and docs plumbing so passive `location.update` node events enqueue system events and wake heartbeat.

## Best Possible Solution

Land a rebased implementation that validates and sanitizes `location.update` payloads, queues summaries as untrusted system events, wakes heartbeat through the current typed contract, adds focused gateway/heartbeat coverage, and includes redacted real behavior proof.

## Reproduction Assessment

Yes, source-level. Current main's iOS monitor sends `location.update`, and targeted current-main inspection shows no gateway or heartbeat handling for that event; no live iOS/gateway run was performed.

## Solution Assessment

No, not as currently proposed. The direction is useful, but the branch must preserve the untrusted node-event boundary, sanitize system markers in node-controlled text, and rebase onto the current heartbeat wake contract before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Mark location updates as untrusted:** `src/gateway/server-node-events.ts:612-615`
  - body: `location.update` accepts payloads from `node.event` and enqueues model-visible location text, but this enqueue call does not set the owner-downgrade fields used by current notification and exec node events. Add `forceSenderIsOwnerFalse: true` and the compatibility `trusted: false` marker so device-supplied location text cannot be treated as owner-authored.
  - confidence: 0.92
- **[P2] Sanitize source before building location summaries:** `src/gateway/server-node-events.ts:596`
  - body: The node-controlled `source` field is compacted but not run through `sanitizeInboundSystemTags`, so marker text such as `[System Message]` can survive into model-visible summaries and logs. Sanitize before compacting or interpolating it, matching the existing notification and exec event paths.
  - confidence: 0.84
