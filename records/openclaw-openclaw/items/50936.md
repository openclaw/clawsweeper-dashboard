## What This Changes

The PR wires iOS `location.update` node events into the gateway, heartbeat wake/prompt handling, docs, and iOS-side logging.

## Best Possible Solution

Land a rebased gateway-owned handler that validates the location payload, treats node-supplied event text as untrusted, embeds the location data in the heartbeat prompt, updates docs/tests, and includes redacted real runtime proof.

Yes, source inspection gives a clear reproduction path: current iOS sends `location.update`, while current main's gateway handler has no matching case. I did not run a live iOS/gateway setup, so this is source-reproducible rather than fully reproduced.

## Solution Assessment

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Mark location events untrusted:** `src/gateway/server-node-events.ts:612-615`
  - body: `location.update` payloads come from connected nodes, but this enqueues the summary without `trusted: false`, so `enqueueSystemEvent` defaults it to trusted. Current node-originated notification/exec events are explicitly untrusted, and heartbeat uses that flag to force `senderIsOwner=false`; with this PR a node-driven location heartbeat can keep owner-level routing. Pass `trusted: false` for the location event too.
  - confidence: 0.87
- **[P2] Neutralize system tags in the source field:** `src/gateway/server-node-events.ts:595-596`
  - body: `compactNotificationEventText` only collapses whitespace and caps length; it does not rewrite `[System Message]` or line-leading `System:` markers the way current notification/exec node-event paths do. Since `source` is node-provided and is embedded into the heartbeat prompt, sanitize it with `sanitizeInboundSystemTags` before compacting.
  - confidence: 0.84
