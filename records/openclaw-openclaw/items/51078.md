## Summary

Keep open: the Android bug-fix intent is still valid on current main, but this PR is not mergeable as-is because it lacks after-fix real Android proof, is conflicting, and introduces a Screen tab slot move that can dispose and recreate the WebView on first open.

## What This Changes

The PR threads an optional Android chat session-list limit through load/bootstrap and renders `ScreenTabScreen` on first Screen tab selection.

## Best Possible Solution

Rebase and land a revised Android fix that keeps one bootstrap `sessions.list` request at 200 and renders `ScreenTabScreen` from a single active-or-started keep-alive slot, with focused Android proof attached.

## Reproduction Assessment

Yes, at source level. Current main still shows the 50-session initial bootstrap path versus 200-session manual refresh, and the Screen tab first-open path can be traced through `screenTabStarted` and `HomeTab.Screen -> Unit`; I did not run a device repro.

## Solution Assessment

No. Passing the session limit through the existing bootstrap path is the right owner boundary, but the Screen tab fallback should be a single stable composition slot and the PR still needs rebase plus real Android proof.
