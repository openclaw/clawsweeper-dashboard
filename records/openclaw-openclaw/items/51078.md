## Summary

Keep this PR open. Current main still lacks both intended Android behavior fixes, but the latest PR patch still has a blocking Screen tab composition bug and needs normal Android review before merge.

## What This Changes

The PR adds an optional Android chat session-list limit through the load/bootstrap path and renders the Screen tab on first visit before the keep-alive Screen tab has started.

## Best Possible Solution

Land a revised Android fix that keeps the chat session-limit change on the single bootstrap `sessions.list` path, composes the Screen tab in one stable keep-alive slot from first open onward, and includes either focused regression coverage or explicit Android device verification plus the required changelog entry.

## Reproduction Assessment

Yes. The Screen tab review finding is reproducible from the Compose state transition: first Screen selection renders the PR fallback while `screenTabStarted` is false, then `LaunchedEffect(activeTab)` flips it true and moves `ScreenTabScreen` into the keep-alive block; current source shows `CanvasScreen` destroys its WebView on dispose.

## Solution Assessment

No. The revised chat path is the right direction, but the Screen fix should render through a single stable slot such as `screenTabStarted || activeTab == HomeTab.Screen`, rather than adding a separate active-branch fallback that is immediately replaced.
