## Summary

Keep this PR open: the chat-session limit direction is sound after the author removed the racing UI refresh, but the Screen-tab hunk still introduces a WebView teardown/recreate cycle on first open and the user-facing Android fix still needs a changelog entry.

## What This Changes

The PR threads an optional Android chat session-list limit through MainViewModel, NodeRuntime, and ChatController, and adds a first-open fallback render for the Android Screen tab.

## Best Possible Solution

Land a revised Android fix that passes the 200-session initial-load limit through the single existing bootstrap fetch path, renders the Screen tab from one stable keep-alive composition slot, and records the user-facing fix in the changelog.

## Reproduction Assessment

Yes, at source level. Current main shows the initial chat load still uses the 50-session bootstrap fetch while manual refresh uses 200, and the PR Screen fallback can be traced through Compose state from first Screen selection to a WebView-disposing slot move.

## Solution Assessment

No. The chat-limit change belongs in the existing bootstrap path, but the Screen-tab fix should use one stable composition condition, such as started-or-active in the keep-alive slot, instead of adding a second active-branch fallback.
