## Summary

## What This Changes

The PR threads an optional Android chat session-list limit through initial chat loading and renders `ScreenTabScreen` on the first Screen tab visit.

## Best Possible Solution

Land a revised Android fix that sends the 200-session initial-load limit through the single bootstrap request and composes the Screen tab from one active-or-started keep-alive slot, with focused Android proof attached.

## Reproduction Assessment

Yes, at source level. Current main still loads initial chat sessions through the 50-limit bootstrap path while manual refresh uses 200, and the first Screen tab render path can be traced through `screenTabStarted`, `HomeTab.Screen -> Unit`, and `CanvasScreen` disposal.

## Solution Assessment

No. The chat-limit threading is the right owner path after review, but the Screen-tab fix should render from one stable keep-alive slot and the PR still needs after-fix real Android proof.
