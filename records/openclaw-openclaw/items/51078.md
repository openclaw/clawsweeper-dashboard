## Summary

Keep open. Current main still lacks the intended Android Screen first-open behavior and the larger initial chat session-list load, but the proposed Screen-tab fix still moves a WebView-backed composable between composition slots and needs a changelog entry before merge.

## What This Changes

The PR threads an optional Android chat session-list limit through MainViewModel/NodeRuntime/ChatController and adds an active Screen-tab fallback render on first visit.

## Best Possible Solution

Land a revised Android fix that keeps the chat session-limit change on the single bootstrap `sessions.list` path, composes the Screen tab through one stable keep-alive slot from first open onward, and includes the required changelog entry plus focused verification.

## Reproduction Assessment

Yes. The Screen-tab defect is reproducible from the Compose state transition: first open mounts the fallback while `screenTabStarted` is false, then `LaunchedEffect(activeTab)` flips the flag and moves `ScreenTabScreen`; `CanvasScreen` destroys its WebView on dispose.

## Solution Assessment

No. The chat direction is the narrow maintainable path, but the Screen fix should use a single stable composition condition such as started-or-active instead of adding a second active-branch fallback slot.
