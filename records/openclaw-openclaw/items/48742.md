## Summary

Keep this PR open. Current main and the latest release still preserve `notify: true` with only a warning when `cron.webhook` is unset, so the central doctor cron bug is not implemented on main; however, the PR is not merge-ready because it bundles that useful fix with unrelated provider secret-normalization changes that would weaken the existing `models.json` plaintext-secret guard.

## What This Changes

The PR changes doctor cron migration to remove legacy `notify: true` when `cron.webhook` is unset, adds related doctor cron tests, and also changes provider API-key normalization plus unrelated test files.

## Best Possible Solution

Land a focused doctor cron change that removes inert legacy `notify: true` when no `cron.webhook` exists, keeps clear warnings for cases where existing delivery semantics are being preserved, and covers the no-webhook cases in `src/commands/doctor-cron.test.ts`; leave provider secret normalization to its own reviewed issue or PR.

## Reproduction Assessment

Yes. A stored cron job with `notify: true`, `delivery.mode: "none"`, and no configured `cron.webhook` reaches the current unset-webhook branch, which only warns and leaves `notify` in the store.

## Solution Assessment

No, not as submitted. The doctor cron part is the right narrow direction, but the combined PR is not the best solution because the unrelated provider API-key changes can regress plaintext-secret protection and should be split out.
