## Summary

Keep open: the cron doctor bug still exists on current main and this PR contains the useful cron-side fix, but the branch is not mergeable as submitted because it also changes provider secret normalization in a way that can reintroduce plaintext API-key persistence and it is currently conflicting with main.

## What This Changes

The branch removes legacy cron `notify: true` fields when `cron.webhook` is unset, adds cron tests, changes provider API-key normalization/tests, and touches unrelated WhatsApp and abort tests.

## Best Possible Solution

Land or replace this with a focused cron-doctor fix that removes inert legacy `notify: true` when no `cron.webhook` exists, preserves explicit delivery semantics with clear warnings, adds targeted cron regression coverage, and leaves provider secret normalization unchanged unless reviewed separately.

## Reproduction Assessment

Yes. Source inspection on current main shows a stored cron job with `notify: true`, `delivery.mode: "none"`, and no `cron.webhook` reaches the warning branch without deleting `notify`; no live doctor run was performed because this review is read-only.

## Solution Assessment

No, not as submitted. The cron hunk is the right narrow direction, but the PR bundles an unrelated provider secret-normalization policy change and stale test edits into a conflicting branch.

## Review Findings

- **[P1] Keep env-secret reverse lookup active without sourceProviders:** `src/agents/models-config.providers.ts:529`
  - body: This guard skips the resolved-env reverse lookup whenever `sourceProviders` is absent, so normalization paths without a source snapshot no longer convert a resolved `OPENAI_API_KEY` value back to the env marker.
