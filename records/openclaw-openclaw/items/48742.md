## Summary

Keep this PR open. Current main still leaves legacy `notify: true` unresolved when `cron.webhook` is unset, so the doctor bug is real, but the PR is not merge-ready because it bundles the useful doctor fix with provider secret-normalization changes that can weaken existing `models.json` plaintext-secret protection.

## What This Changes

The PR updates doctor cron migration/tests to remove legacy `notify: true` when no webhook is configured, changes provider API-key normalization/tests, and includes unrelated WhatsApp and abort test edits.

## Best Possible Solution

Land a focused doctor cron change that removes inert legacy `notify: true` when no `cron.webhook` exists, preserves explicit delivery semantics with clear warnings, and covers the no-webhook cases in `src/commands/doctor-cron.test.ts`; leave provider secret normalization to its already-shipped marker-protection path or a separate security-reviewed proposal.

## Reproduction Assessment

Yes. A stored cron job with `notify: true`, `delivery.mode: "none"`, and no configured `cron.webhook` reaches the current no-webhook branch, which only warns and leaves `notify` in the store.

## Solution Assessment

No, not as submitted. The doctor cron part is the right narrow approach, but combining it with provider API-key normalization changes is not the safest maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Keep env-secret reverse lookup active without sourceProviders:** `src/agents/models-config.providers.ts:529`
  - body: This gates the resolved-env reverse lookup on `params.sourceProviders`, so callers without a source snapshot no longer convert a resolved `OPENAI_API_KEY` value back to the env marker. That can persist plaintext provider secrets into `models.json` and regresses the existing #38757 guard; keep the reverse lookup or split this provider change out.
  - confidence: 0.87

## Security Review

Status: needs_attention
