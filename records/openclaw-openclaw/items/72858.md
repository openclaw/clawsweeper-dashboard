## Summary

Keep open. Current main supports Control UI exec when host exec policy is opened, but the remaining allowlist-miss path is an independent host exec approval boundary, and changing browser-surface host exec semantics needs maintainer product/security direction.

## What This Changes

The issue asks whether WebChat/Control UI exec should run after elevated sender allow rules and `tools.byProvider.webchat` allow `exec`/`process`, or whether another policy layer is blocking it.

## Best Possible Solution

Clarify the contract in docs/schema first; if browser-surface host exec grants are desired, add an explicit surface-policy path that preserves host exec allowlist and approval enforcement.

## Reproduction Assessment

Yes, from source inspection rather than a live WebChat run. Configure `tools.byProvider.webchat` while the active model provider is not `webchat`, keep host exec in `security=allowlist` without a matching allowlist or durable approval, and current main reaches the host `allowlist miss` path.

## Solution Assessment

Unclear. Treating `tools.byProvider.webchat` as a Control UI host exec grant is not the safest narrow fix; docs/schema clarification or an explicit surface-policy mechanism would be safer unless maintainers choose broader semantics.
