## Summary

Keep open: current main and v2026.6.10 still separate model-provider tool availability from host exec allowlist enforcement, so deciding whether browser-surface policy should ever broaden host exec remains a maintainer security/product question.

## What This Changes

The issue asks whether WebChat/Control UI exec should honor elevated sender and provider tool policy settings instead of still failing with an allowlist miss.

## Best Possible Solution

Maintainers should either document the current boundary more directly or design an explicit reviewed browser/channel exec policy that preserves host allowlist and approval guardrails.

## Reproduction Assessment

Yes, source-level. Current main still resolves `tools.byProvider` from the selected model provider/model while gateway-host exec throws `exec denied: allowlist miss` under allowlist security without a matching allowlist or durable approval.

## Solution Assessment

Unclear. Reinterpreting `tools.byProvider.webchat` as host exec authority is not the safest default; the maintainable path is maintainer-owned clarification or an explicit reviewed browser/channel exec policy.
