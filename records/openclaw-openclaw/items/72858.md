## Summary

Keep open. Current main supports the narrowed diagnosis: Control UI/WebChat is not hard-blocked from exec, but `tools.byProvider.webchat` is not a Control UI/WebChat surface grant; current code resolves `tools.byProvider` against the selected model provider/model, while host exec allowlist/approval enforcement is a separate security layer. The remaining work is a security-sensitive contract decision and documentation/schema-help mismatch, not an implemented fix or a clean duplicate.

## What This Changes

The issue asks OpenClaw to clarify or fix why Control UI/WebChat exec still returns an allowlist miss after configuring elevated sender allow rules and `tools.byProvider.webchat` tool policy.

## Best Possible Solution

Align the public contract first: make schema help, generated metadata, docs, and Control UI copy consistently describe `tools.byProvider` as selected model provider/model policy. If maintainers want Control UI/WebChat-specific host exec grants, add a separate explicit surface-scoped policy seam with regression tests around host exec allowlist and approval behavior.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: configure `tools.byProvider.webchat` while the active model provider is not `webchat`, keep `tools.exec.security=allowlist` without a matching host allowlist or durable approval, and current main will not apply that provider policy before the gateway allowlist path can throw `exec denied: allowlist miss`.

## Solution Assessment

Unclear. Treating `tools.byProvider.webchat` as permission to open host exec from the browser is not the safest narrow fix; contract clarification is safer unless maintainers explicitly approve a separate surface-policy design.
