## Summary

Keep open. Current main still supports the narrowed diagnosis: Control UI exec is not blanket-blocked, but `tools.byProvider.webchat` is not applied as a WebChat/Control UI surface grant, and `exec denied: allowlist miss` remains a separate host exec allowlist boundary.

## What This Changes

The issue asks OpenClaw to clarify or fix why WebChat/Control UI exec stays denied in allowlist mode after configuring elevated sender rules and `tools.byProvider.webchat` tool policy.

## Best Possible Solution

Align the public contract first: schema help, generated metadata, docs, and Control UI copy should consistently state that `tools.byProvider` keys the selected model provider/model. If maintainers want browser/WebChat-specific exec grants, add a separate explicit surface-policy seam with tests that preserve host exec allowlist and approval boundaries.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: configure `tools.byProvider.webchat` while the active model provider is not `webchat`, keep `tools.exec.security=allowlist` without a matching host allowlist or durable approval, and current main will not apply that provider policy before the gateway allowlist path can throw `exec denied: allowlist miss`.

## Solution Assessment

Unclear. Treating `tools.byProvider.webchat` as permission to open host exec from the browser is not the safest narrow fix; the safer default is contract clarification unless maintainers explicitly approve a separate surface-policy design.
