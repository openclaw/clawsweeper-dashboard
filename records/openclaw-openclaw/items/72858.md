## Summary

Keep open. Source inspection supports the reporter's narrowed diagnosis: Control UI exec is not blanket-blocked, but `tools.byProvider.webchat` does not currently mean browser-surface host exec permission, and the remaining `allowlist miss` path is the independent host exec approval boundary.

## What This Changes

The issue asks whether WebChat/Control UI exec should honor `tools.byProvider.webchat` after elevated sender allow rules, because allowlist-mode host exec still returns `exec denied: allowlist miss`.

## Best Possible Solution

Clarify the contract in schema/docs first; if browser-surface exec grants are desired, add a deliberately named surface-policy path that preserves host exec allowlist and approval enforcement.

## Reproduction Assessment

Yes, from source inspection rather than a live WebChat run. Configure `tools.byProvider.webchat` while the active model provider is not `webchat`, keep host exec in `security=allowlist` without a matching allowlist or durable approval, and current main reaches the host `allowlist miss` path.

## Solution Assessment

Unclear. Reinterpreting `tools.byProvider.webchat` as a Control UI host exec grant is not the safest narrow fix; docs/schema clarification or an explicit surface-policy mechanism would be safer unless maintainers choose broader semantics.
