## Summary

Keep open. Current main still treats `tools.byProvider` as selected model-provider policy, while Control UI/WebChat identity and gateway host `exec` allowlist enforcement remain separate security boundaries; changing that behavior needs maintainer product/security direction rather than conservative cleanup.

## What This Changes

The issue asks whether WebChat/Control UI `exec` denial in allowlist mode is caused by a WebChat-specific restriction or by provider/tool-policy application.

## Best Possible Solution

Clarify docs/schema help that `tools.byProvider` is model-provider keyed, and only add Control UI/WebChat host `exec` grants through an explicit reviewed sender or browser-surface policy that preserves allowlist and approval enforcement.

## Reproduction Assessment

Yes, at source level. Configuring `tools.byProvider.webchat` while the active model provider is not `webchat`, then running host `exec` under allowlist mode without a matching allowlist or durable approval, reaches the current `exec denied: allowlist miss` path.

## Solution Assessment

Unclear. Reinterpreting `tools.byProvider.webchat` as a Control UI host `exec` grant is not the safest narrow fix; docs/schema clarification or an explicit reviewed browser-surface policy would be safer.
