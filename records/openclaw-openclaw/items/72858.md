## Summary

Keep open. Current main supports the narrowed diagnosis: Control UI exec is not blanket-blocked, but `tools.byProvider.webchat` is not applied as a browser-surface grant, and the remaining `allowlist miss` path is the separate host exec allowlist/approval boundary.

## What This Changes

The issue asks OpenClaw to clarify or fix why WebChat/Control UI host exec still hits `exec denied: allowlist miss` after configuring WebChat elevated allow rules and `tools.byProvider.webchat` exec grants.

## Best Possible Solution

Clarify the public contract first, then add an explicitly named browser/surface exec policy only if maintainers want Control UI grants separate from model-provider policy while preserving host allowlist and approval enforcement.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: use `tools.byProvider.webchat` while the selected model provider is not `webchat`, keep `tools.exec.security = "allowlist"` without a matching allowlist or durable approval, and current main reaches the gateway `allowlist miss` path.

## Solution Assessment

Unclear. Treating `tools.byProvider.webchat` as permission to open host exec from Control UI is not the safest narrow fix; documentation/schema clarification or a new explicit surface-policy seam would be safer.
