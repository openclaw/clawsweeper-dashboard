## Summary

Keep open. Current main supports the narrowed diagnosis: Control UI exec is not blanket-blocked, but `tools.byProvider.webchat` still does not map to the browser surface and the remaining `allowlist miss` is an independent host exec approval/allowlist boundary that needs maintainer policy/security direction.

## What This Changes

The issue asks whether Control UI/WebChat exec should honor elevated sender allow rules plus `tools.byProvider.webchat`, or whether the remaining allowlist miss is an internal policy bug.

## Best Possible Solution

Clarify the public contract first: keep `tools.byProvider` model-provider keyed if that is intended, and add a separately named browser/surface policy only if maintainers explicitly want Control UI host exec grants beyond existing approvals.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: configure `tools.byProvider.webchat` while the selected model provider is not `webchat`, keep `tools.exec.security = "allowlist"` without a matching allowlist or durable approval, and current main reaches the host `allowlist miss` path.

## Solution Assessment

Unclear. Treating `tools.byProvider.webchat` as a Control UI host exec grant is not the safest narrow solution; documentation/schema clarification or an explicit surface-policy seam would be safer unless maintainers intentionally choose broader semantics.
