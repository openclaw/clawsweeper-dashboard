## Summary

Keep open. Current main supports the reporter's narrowed diagnosis: Control UI exec is not blanket-blocked, but `tools.byProvider.webchat` is not a browser-surface host exec grant, and the remaining `allowlist miss` is enforced by the independent host exec allowlist/approval boundary.

## What This Changes

The issue asks whether WebChat/Control UI exec should honor configured elevated sender and provider tool policy when host exec is otherwise denied by allowlist mode.

## Best Possible Solution

Clarify the config contract so model-provider tool policy and host exec allowlist/approval policy are visibly separate; if browser-surface grants are desired, add an explicit surface policy that still preserves host exec enforcement.

## Reproduction Assessment

Yes, from source inspection rather than a live WebChat run. Configure `tools.byProvider.webchat` while the active model provider is not `webchat`, keep host exec in `security=allowlist` without a matching allowlist or durable approval, and current main can still reach the host `allowlist miss` path.

## Solution Assessment

No, reinterpreting `tools.byProvider.webchat` as a Control UI host exec grant is not the safest narrow fix. A docs/schema clarification or a deliberately named surface-policy path would be more maintainable unless maintainers intentionally choose broader semantics.
