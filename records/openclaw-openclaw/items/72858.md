## Summary

Keep open. Current main still separates selected model-provider tool policy from WebChat/Control UI surface identity, and the remaining denial is the independent host exec allowlist boundary rather than a blanket Control UI exec block.

## What This Changes

The issue asks whether Control UI/WebChat exec should honor `webchat`-scoped provider tool policy and elevated sender rules instead of ending in host allowlist denial.

## Best Possible Solution

Clarify the public contract for `tools.byProvider` versus browser surfaces and host exec approvals; if browser-surface grants are desired, add an explicit policy path that preserves `tools.exec.security`, allowlist, and approval enforcement.

## Reproduction Assessment

Yes, from source inspection rather than a live WebChat run. Configure `tools.byProvider.webchat` while the active model provider is not `webchat`, keep host exec in `security=allowlist` without a matching allowlist or durable approval, and current main still reaches the gateway allowlist miss path.

## Solution Assessment

Unclear. Reinterpreting `tools.byProvider.webchat` as a Control UI host-exec grant is not the safest narrow fix; docs/schema clarification or a deliberately named surface-policy mechanism would be safer unless maintainers choose broader semantics.
