## Summary

Keep open. Current main supports the reporter's narrowed diagnosis: Control UI exec is not blanket-blocked, but `tools.byProvider.webchat` is model-provider policy, not a browser-surface host exec grant, and the remaining denial is the separate host exec allowlist boundary.

## What This Changes

Clarify or fix WebChat/Control UI host exec policy so elevated sender and tool policy configuration does not still produce allowlist-miss denials.

## Best Possible Solution

Document that `tools.byProvider` keys are model providers, then add an explicit browser-surface exec policy only if maintainers want that capability while preserving host allowlist and approval enforcement.

## Reproduction Assessment

Yes, from source inspection rather than a live WebChat run. With an active model provider other than `webchat`, `tools.byProvider.webchat` is not selected, and allowlist-mode host exec without a matching allowlist or durable approval reaches `exec denied: allowlist miss`.

## Solution Assessment

Unclear. Reinterpreting `tools.byProvider.webchat` as a Control UI host exec grant would blur model-provider policy with browser-surface authorization; docs/schema clarification or an explicit surface-policy mechanism is safer.
