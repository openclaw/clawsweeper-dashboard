## What This Changes

The issue asks whether Control UI/WebChat exec should still be blocked by host exec allowlist policy after elevated sender rules and provider tool policy are configured.

## Best Possible Solution

Maintainers should either clarify the current contract in docs/schema, or design an explicit browser-surface exec policy that preserves host allowlist and approval enforcement.

## Reproduction Assessment

Yes. Source inspection shows that WebChat/Control UI identity, model-provider tool policy, and host exec allowlist enforcement are separate, and an allowlist miss is still reachable when `tools.exec.security` is allowlist without a matching allowlist or durable approval.

## Solution Assessment

Unclear. Reinterpreting `tools.byProvider.webchat` as host exec authority is not the safest fix; the maintainable path is either clearer documentation or a reviewed explicit browser-surface exec policy.
