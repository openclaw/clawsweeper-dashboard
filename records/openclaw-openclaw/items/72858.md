## Summary

Keep open. Current main supports the narrowed diagnosis: Control UI exec is not blanket-blocked, but `tools.byProvider.webchat` is still model-provider policy rather than browser-surface policy, and the remaining `allowlist miss` path is the host exec allowlist/approval boundary.

## What This Changes

Clarify or fix why Control UI exec in allowlist mode can still return `exec denied: allowlist miss` after elevated sender rules and `tools.byProvider.webchat` grants are configured.

## Best Possible Solution

Clarify the public contract first: keep `tools.byProvider` model-provider keyed, and add an explicit browser/surface policy only if maintainers intentionally want Control UI-specific host exec grants with approval and allowlist enforcement preserved.

## Reproduction Assessment

Yes. A high-confidence static path exists: Control UI connects as `openclaw-control-ui`/`webchat`, tool policy evaluates the selected model provider, and host exec throws `allowlist miss` when `security=allowlist` lacks a matching allowlist entry or durable approval.

## Solution Assessment

Unclear. Treating `tools.byProvider.webchat` as a Control UI host-exec grant is not the safest narrow solution; docs/schema clarification or a separate surface-policy seam would be safer unless maintainers choose broader semantics.
