## Summary

Keep open. Current main supports the narrowed diagnosis: Control UI exec is not blanket-blocked, but `tools.byProvider.webchat` is not a browser/WebChat surface grant, and the remaining `allowlist miss` path is an independent host exec approval/allowlist boundary that needs explicit maintainer policy direction.

## What This Changes

The issue asks whether WebChat/Control UI exec should honor a `tools.byProvider.webchat` grant or whether the remaining `exec denied: allowlist miss` is an intended separate exec policy boundary.

## Best Possible Solution

Align the public contract first: if `tools.byProvider` is model-provider keyed, update schema help, generated metadata, docs, and Control UI copy to say that clearly. If browser/WebChat-specific exec grants are desired, add a separately named surface-policy mechanism with tests that preserve the host exec allowlist and approval boundary.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: configure `tools.byProvider.webchat` while the active model provider is not `webchat`, keep `tools.exec.security = "allowlist"` without a matching host allowlist or durable approval, and current main will not apply that provider policy before the gateway can throw `exec denied: allowlist miss`.

## Solution Assessment

Unclear. Treating `tools.byProvider.webchat` as permission to open host exec from the browser is not the safest narrow solution; contract clarification or an explicit surface-policy seam would be safer unless maintainers intentionally choose broader semantics.
