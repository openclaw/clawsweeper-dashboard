## Summary

Keep open. Current main still supports the reporter's narrowed diagnosis: Control UI can run exec when host exec policy is opened, but `tools.byProvider.webchat` is not a WebChat/Control UI surface grant; it resolves against the selected model provider/model, while schema help still describes channel/surface semantics. The remaining work is a security-sensitive contract decision, not an implemented fix or a clean duplicate.

## What This Changes

The issue asks whether WebChat/Control UI exec should honor elevated sender allow rules plus `tools.byProvider.webchat` grants in allowlist mode, or whether a separate internal restriction is intentional.

## Best Possible Solution

Align the public contract first: schema help, generated metadata, docs, and Control UI copy should consistently say `tools.byProvider` keys the selected model provider/model. If maintainers want WebChat/Control UI-specific grants, add a separate explicit surface-policy seam with tests that preserve host exec allowlist and approval boundaries.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: configure `tools.byProvider.webchat` while the active model provider is not `webchat`, keep `tools.exec.security=allowlist` without a matching host allowlist or durable approval, and current main will not apply that provider policy before the gateway allowlist path can throw `exec denied: allowlist miss`.

## Solution Assessment

Unclear. Treating `tools.byProvider.webchat` as permission to open host exec from the browser is not the safest narrow fix; the safer path is contract clarification unless maintainers explicitly approve a separate surface-policy design.
