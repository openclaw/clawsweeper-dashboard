## Summary

Keep open. Current main supports the narrowed diagnosis: Control UI exec is not blanket-blocked, but `tools.byProvider.webchat` is model-provider policy while the remaining `allowlist miss` path is a separate host exec allowlist/approval boundary that needs maintainer product/security direction.

## What This Changes

The issue asks OpenClaw to clarify or fix why WebChat/Control UI exec still returns `exec denied: allowlist miss` after elevated sender and `tools.byProvider.webchat` tool allowances are configured.

## Best Possible Solution

Align the public contract first: clarify that `tools.byProvider` is model-provider keyed, and if browser-surface exec grants are desired, add an explicit surface-policy mechanism with tests preserving host allowlist and approval enforcement.

## Reproduction Assessment

Yes. Configure `tools.byProvider.webchat` while the active model provider is not `webchat`, keep `tools.exec.security = "allowlist"` without a matching allowlist or durable approval, and current main reaches the host `allowlist miss` path.

## Solution Assessment

Unclear. Treating `tools.byProvider.webchat` as a Control UI host exec grant is not the safest narrow fix; documentation/schema clarification or a deliberately named surface-policy seam would be safer unless maintainers choose broader semantics.
