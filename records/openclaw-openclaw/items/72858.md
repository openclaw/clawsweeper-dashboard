## Summary

Keep open. Current main supports the narrowed diagnosis: Control UI exec is not blanket-blocked, but `tools.byProvider.webchat` is model-provider policy while the remaining `allowlist miss` path is enforced by the separate host exec allowlist/approval boundary.

## What This Changes

The issue asks whether WebChat/Control UI exec should honor elevated sender and `tools.byProvider.webchat` policy instead of still failing with `exec denied: allowlist miss`.

## Best Possible Solution

Align the public contract first: clarify provider-keyed tool policy versus browser-surface host exec policy, and add an explicit surface-policy mechanism only if maintainers want that capability while preserving host allowlist and approval checks.

## Reproduction Assessment

Yes. The reporter’s follow-up plus current code give a high-confidence path: use `tools.byProvider.webchat` while the active model provider is not `webchat`, keep host exec in `allowlist` without a matching allowlist or durable approval, and gateway exec reaches the `allowlist miss` path.

## Solution Assessment

Unclear. Treating `tools.byProvider.webchat` as a Control UI host exec grant is not the safest narrow fix because `byProvider` is model-provider keyed; docs/schema clarification or a deliberately named surface-policy seam would be safer.
