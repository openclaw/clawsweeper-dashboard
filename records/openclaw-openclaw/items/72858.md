## Summary

Keep open. Current main supports the narrowed diagnosis: Control UI exec is not blanket-blocked, but `tools.byProvider.webchat` is not a browser-surface grant, and the remaining `allowlist miss` path is a separate host exec approval/allowlist boundary that needs maintainer policy direction.

## What This Changes

The issue asks whether WebChat / Control UI exec should honor elevated sender and `tools.byProvider.webchat` policy for host `exec`, or whether the denial is an internal restriction or policy bug.

## Best Possible Solution

Align the public contract first: clarify whether `tools.byProvider` is strictly model-provider keyed, and if browser/WebChat exec grants are desired, add a separately named surface-policy mechanism with tests preserving host allowlist and approval enforcement.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: configure `tools.byProvider.webchat` while the active model provider is not `webchat`, keep `tools.exec.security = "allowlist"` without a matching allowlist or durable approval, and current main reaches the host `allowlist miss` path.

## Solution Assessment

Unclear. Treating `tools.byProvider.webchat` as permission to open host exec from Control UI is not the safest narrow solution; documentation/schema clarification or an explicit surface-policy seam would be safer unless maintainers intentionally choose broader semantics.
