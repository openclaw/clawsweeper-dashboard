## Summary

Keep open: current main still separates model-provider tool policy, Control UI/WebChat client identity, and host exec allowlist enforcement; resolving this as requested needs explicit maintainer security/product direction rather than cleanup closure.

## What This Changes

The issue asks whether WebChat/Control UI host exec should run when WebChat sender elevation and `tools.byProvider.webchat.alsoAllow` are configured, despite allowlist-mode denials.

## Best Possible Solution

Confirm the current provider/sender/host-exec separation in docs and schema help, or add an explicit reviewed browser-surface exec policy that preserves host allowlist and approval enforcement.

## Reproduction Assessment

Yes, source-level. Configuring `tools.byProvider.webchat` while the selected model provider is not `webchat`, then running host exec in allowlist mode without a matching allowlist or durable approval, reaches the current `exec denied: allowlist miss` path.

## Solution Assessment

Unclear. Reinterpreting `tools.byProvider.webchat` as Control UI host exec authority is not the safest narrow fix; a docs/schema clarification or explicit reviewed browser-surface policy would be safer.
