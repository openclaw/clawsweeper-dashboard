## Summary

Keep open: current main still separates WebChat/Control UI sender identity, model-provider tool visibility, and host exec allowlist enforcement. The reporter’s follow-up rules out a blanket Control UI exec block, but changing browser-surface policy so `tools.byProvider.webchat` can affect host `exec` allowlist behavior needs maintainer security/product direction.

## What This Changes

The issue asks OpenClaw to clarify or fix WebChat/Control UI exec policy so configured elevated sender rules and `tools.byProvider.webchat` do not still produce `exec denied: allowlist miss`.

## Best Possible Solution

Maintainers should clarify the current contract in docs/schema, or design an explicit reviewed browser/channel exec policy that preserves host allowlist and approval enforcement.

## Reproduction Assessment

Yes, source-level. Current main shows `tools.byProvider` is resolved from model provider/model while the gateway host exec path still denies allowlist misses when `tools.exec.security` is allowlist and no command allowlist or durable approval matches.

## Solution Assessment

Unclear. Reinterpreting `tools.byProvider.webchat` as host exec authority is not the safest fix; the maintainable path is either clearer documentation/schema wording or an explicit reviewed browser-surface exec policy.
