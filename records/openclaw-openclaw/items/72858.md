## Summary

Keep open. Current main supports the narrowed diagnosis: Control UI exec is not blanket-blocked, but `tools.byProvider.webchat` is not a browser-surface grant, and the remaining `allowlist miss` path is a separate host exec approval/allowlist boundary that needs maintainer policy direction.

## What This Changes

Clarify or fix why Control UI/WebChat exec still reports `allowlist miss` after elevated sender rules and provider-level tool policy are configured.

## Best Possible Solution

Align the public contract first: clarify that `tools.byProvider` is model-provider keyed, and add a separately named surface-policy mechanism only if maintainers intentionally want browser/WebChat-specific exec grants.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: configure `tools.byProvider.webchat` while the active model provider is not `webchat`, keep `tools.exec.security = "allowlist"` without a matching allowlist or durable approval, and current main reaches the host `allowlist miss` path.

## Solution Assessment

Unclear. Treating `tools.byProvider.webchat` as permission to open host exec from Control UI is not the safest narrow solution; documentation/schema clarification or an explicit surface-policy seam would be safer unless maintainers choose broader semantics.
