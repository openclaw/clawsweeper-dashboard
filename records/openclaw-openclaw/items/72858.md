## Summary

Keep open. Current main supports the narrowed diagnosis: Control UI exec is not blanket-blocked, but `tools.byProvider.webchat` is model-provider policy, while the remaining `allowlist miss` comes from the separate host exec approval/allowlist boundary and needs maintainer security/product direction.

## What This Changes

The issue asks OpenClaw to clarify or fix why Control UI/WebChat exec still hits `allowlist miss` after elevated sender and `tools.byProvider.webchat` settings are added.

## Best Possible Solution

Document `tools.byProvider` as model-provider keyed and, if browser-surface exec grants are desired, add an explicit surface-policy mechanism that preserves host approvals and allowlist enforcement.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: configure `tools.byProvider.webchat` while the active model provider is not `webchat`, keep `tools.exec.security = "allowlist"` without a matching allowlist or durable approval, and current main reaches the host `allowlist miss` path.

## Solution Assessment

Unclear because there is no patch yet. Treating `tools.byProvider.webchat` as a Control UI surface grant is not the safest narrow fix; docs/schema clarification or an explicit surface-policy seam would be safer.
