## Summary

Keep open. Current main supports the narrowed diagnosis: Control UI exec is not blanket-blocked, but `tools.byProvider.webchat` is not a browser/WebChat surface grant, and `exec denied: allowlist miss` remains an independent host exec allowlist/approval boundary. The remaining ambiguity is a security-sensitive product/docs contract issue, especially because generated schema help still describes provider policy as channel/surface-scoped.

## What This Changes

The issue asks whether Control UI/WebChat `exec` should honor `tools.byProvider.webchat` plus elevated sender configuration, or whether the remaining allowlist miss is an intentional separate host exec policy boundary.

## Best Possible Solution

Align the contract before changing behavior: schema help, generated metadata, docs, and Control UI copy should consistently say whether `tools.byProvider` is model-provider keyed. If maintainers want browser/WebChat-specific exec grants, add a separately named surface-policy seam with tests that keep host exec allowlists and approvals independent.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: configure `tools.byProvider.webchat` while the active model provider is not `webchat`, keep `tools.exec.security = "allowlist"` without a matching host allowlist or durable approval, and current main will not apply that provider policy before the gateway can throw `exec denied: allowlist miss`.

## Solution Assessment

Unclear. Treating `tools.byProvider.webchat` as a browser surface grant is not the safest narrow fix; contract clarification, or an explicit surface-policy mechanism if maintainers want that feature, is safer.
