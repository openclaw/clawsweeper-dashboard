## Summary

Keep open. Current main still treats `tools.byProvider` as selected model-provider policy, while Control UI/WebChat identity and host `exec` allowlist enforcement are separate boundaries; changing that behavior needs maintainer product/security direction.

## What This Changes

The issue asks whether WebChat/Control UI `exec` should become runnable when elevated sender allow rules and `tools.byProvider.webchat.alsoAllow` are configured.

## Best Possible Solution

Clarify docs/schema help that `tools.byProvider` is model-provider keyed, and only add Control UI/WebChat host `exec` grants through an explicit reviewed surface or sender policy that preserves host allowlist and approval enforcement.

## Reproduction Assessment

Yes. From source inspection, configuring `tools.byProvider.webchat` while the active model provider is not `webchat`, then running host `exec` in allowlist mode without a matching allowlist or durable approval, reaches the current `exec denied: allowlist miss` path.

## Solution Assessment

Unclear. Reinterpreting `tools.byProvider.webchat` as a Control UI host `exec` grant is not the safest narrow fix; docs/schema clarification or an explicit reviewed browser-surface policy would be safer.
