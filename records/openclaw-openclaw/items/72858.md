## Summary

Keep open. Current main still supports the reporter's narrowed diagnosis: Control UI exec is not blanket-blocked, but `tools.byProvider.webchat` is resolved as selected model provider/model policy, not as a WebChat/Control UI surface grant, while schema help still implies channel/surface semantics. The remaining work is a security-sensitive contract decision, not an implemented fix or a clean duplicate of the related allowlist/approval issues.

## What This Changes

The issue asks OpenClaw to clarify or fix why WebChat/Control UI host exec still hits `exec denied: allowlist miss` after elevated sender rules and `tools.byProvider.webchat` are configured.

## Best Possible Solution

Align the public contract first: schema help, generated metadata, docs, and Control UI copy should consistently say `tools.byProvider` keys the selected model provider/model. If maintainers want WebChat/Control UI-specific host exec grants, add a separate explicit surface-policy seam with regression tests that preserve host allowlist and approval boundaries.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: configure `tools.byProvider.webchat` while the active model provider is not `webchat`, keep `tools.exec.security=allowlist` without a matching host allowlist or durable approval, and current main will not apply that provider policy before the gateway allowlist path can throw `exec denied: allowlist miss`.

## Solution Assessment

Unclear. Treating `tools.byProvider.webchat` as permission to open host exec from the browser is not the safest narrow fix; the safer path is contract clarification unless maintainers explicitly approve a separate surface-policy design.
