## Summary

Keep open. Current main supports the narrowed diagnosis: Control UI exec is not blanket-blocked, but `tools.byProvider.webchat` is model-provider policy while the remaining `allowlist miss` path is enforced by the separate host exec allowlist/approval boundary.

## What This Changes

The issue asks OpenClaw to clarify or fix why WebChat/Control UI `exec` remains denied in allowlist mode after elevated sender rules and `tools.byProvider.webchat` tool policy are configured.

## Best Possible Solution

Clarify the public contract that `tools.byProvider` is model-provider keyed, and add an explicit browser-surface exec policy only if maintainers choose that behavior while preserving host allowlist and approval enforcement.

## Reproduction Assessment

Yes. Configure `tools.byProvider.webchat` while the active model provider is not `webchat`, keep `tools.exec.security = "allowlist"` without a matching allowlist or durable approval, and current main reaches the host `allowlist miss` path; the reporter also confirmed `security = "full"` makes Control UI exec work.

## Solution Assessment

Unclear. Treating `tools.byProvider.webchat` as Control UI host exec permission is not the safest narrow fix; documentation/schema clarification or a deliberately named surface-policy seam would be safer unless maintainers choose broader semantics.
