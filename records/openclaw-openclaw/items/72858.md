## Summary

Keep open. Current main still treats `tools.byProvider` as selected model-provider policy while host exec allowlist enforcement remains a separate boundary, so changing WebChat/Control UI exec semantics needs maintainer product/security direction.

## What This Changes

The issue asks whether WebChat/Control UI exec should honor configured elevated sender and provider tool policy when host exec otherwise reports an allowlist miss.

## Best Possible Solution

Clarify docs/schema help that `tools.byProvider` is model-provider keyed, and only add browser-surface or sender exec grants if maintainers design them to preserve host allowlist and approval enforcement.

## Reproduction Assessment

Yes, from source inspection rather than a live browser run: configure `tools.byProvider.webchat` while the active model provider is not `webchat`, keep host exec in allowlist mode without a matching allowlist or durable approval, and current main reaches the host `allowlist miss` path.

## Solution Assessment

Unclear. Reinterpreting `tools.byProvider.webchat` as a Control UI host exec grant is not the safest narrow fix; docs/schema clarification or an explicit reviewed surface/sender policy would be safer.
