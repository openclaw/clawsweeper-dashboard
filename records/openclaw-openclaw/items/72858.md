## Summary

Keep open. Current main still separates Control UI/WebChat sender identity, model-provider tool policy, and host exec allowlist enforcement; the remaining question is product/security semantics rather than a shipped fix or a clean duplicate.

## What This Changes

The issue asks whether WebChat / Control UI exec should honor configured sender elevation and `tools.byProvider.webchat` policy instead of still failing with an exec allowlist miss.

## Best Possible Solution

Clarify the contract in docs/schema first; if browser-surface exec grants are desired, add an explicit surface-policy design that preserves host exec allowlist and approval boundaries.

## Reproduction Assessment

Yes, from source inspection rather than a live WebChat run. Configure `tools.byProvider.webchat` while the selected model provider is not `webchat`, keep host exec in `security=allowlist` without a matching allowlist or durable approval, and current main can still reach `exec denied: allowlist miss`.

## Solution Assessment

Unclear. Reinterpreting `tools.byProvider.webchat` as a Control UI host exec grant is not the safest narrow fix; docs/schema clarification or an explicit browser-surface policy would be safer unless maintainers choose broader semantics.
