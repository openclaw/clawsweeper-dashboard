## Summary

Keep open. Current main supports the narrowed diagnosis: Control UI exec is not blanket-blocked, but `tools.byProvider.webchat` is not applied as a browser/WebChat surface grant, and `exec denied: allowlist miss` remains a separate host exec allowlist/approval boundary.

## What This Changes

The issue asks whether WebChat/Control UI exec should honor `tools.byProvider.webchat` or whether the observed `exec denied: allowlist miss` is an intentional separate host exec policy layer.

## Best Possible Solution

Align the public contract first: schema help, generated metadata, docs, and any Control UI copy should consistently say whether `tools.byProvider` is model-provider keyed. If browser/WebChat-specific exec grants are desired, add a separate explicit surface-policy seam with tests that preserve host exec allowlist and approval checks.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: configure `tools.byProvider.webchat` while the active model provider is not `webchat`, keep `tools.exec.security = "allowlist"` without a matching host allowlist or durable approval, and current main will not apply that provider policy before the gateway can throw `exec denied: allowlist miss`.

## Solution Assessment

Unclear. Treating `tools.byProvider.webchat` as permission to open host exec from the browser is not the safest narrow fix; contract clarification or a separately named surface-policy mechanism would be safer unless maintainers explicitly choose broader semantics.
