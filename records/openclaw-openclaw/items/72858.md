## Summary

Keep open. Current main still treats `tools.byProvider` as selected model-provider policy and host exec allowlist enforcement as a separate boundary; Control UI exec can work with full exec security, but WebChat surface-policy semantics remain a product/security/docs question.

## What This Changes

The issue asks whether WebChat/Control UI host exec should honor `tools.byProvider.webchat` plus elevated sender allow rules instead of failing with `exec denied: allowlist miss`.

## Best Possible Solution

Clarify schema/help/docs around model-provider versus WebChat surface keys, and only add a browser-surface or sender-policy exec grant if maintainers explicitly design it to preserve host allowlist and approval enforcement.

## Reproduction Assessment

Yes, from source inspection rather than a live WebChat run: configure `tools.byProvider.webchat` while the active model provider is not `webchat`, keep host exec in allowlist mode without a matching allowlist or durable approval, and current main reaches the host `allowlist miss` path.

## Solution Assessment

Unclear. Reinterpreting `tools.byProvider.webchat` as a Control UI host exec grant is not the safest narrow fix; docs/schema clarification or an explicit surface/sender policy would be safer unless maintainers choose broader semantics.
