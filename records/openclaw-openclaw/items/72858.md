## Summary

Keep open. Current main supports the narrowed diagnosis: Control UI/WebChat exec is not blanket-blocked, but `tools.byProvider.webchat` is not a browser-surface grant, and the observed `allowlist miss` path is still governed by the independent host exec allowlist/approval boundary. The remaining problem is a security-sensitive contract/docs decision because current schema help can still be read as channel- or surface-keyed.

## What This Changes

The issue asks OpenClaw to clarify or fix why WebChat/Control UI exec still reports `exec denied: allowlist miss` after configuring elevated sender allow rules and `tools.byProvider.webchat` grants.

## Best Possible Solution

Align the public contract first: update schema help, generated config metadata, docs, and any Control UI copy to say `tools.byProvider` is keyed by selected model provider/model id. If maintainers want browser/WebChat-specific host exec grants, add a separately named surface-policy mechanism with regression tests that keep host allowlist and approval checks independent.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: configure `tools.byProvider.webchat` while the active model provider is not `webchat`, keep effective host exec security in `allowlist` without a matching host allowlist or durable approval, and current main can still throw `exec denied: allowlist miss`.

## Solution Assessment

Unclear. Treating `tools.byProvider.webchat` as a Control UI/browser exec grant is not the safest narrow fix; schema/docs clarification or a new explicit surface-policy seam is safer unless maintainers intentionally choose broader semantics.
