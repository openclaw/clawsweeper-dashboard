## Summary

Keep open. Current main still supports the narrowed diagnosis from the discussion: Control UI can reach exec when host exec policy is opened, but `tools.byProvider.webchat` is not currently a WebChat/Control UI surface grant; it resolves against the selected model provider/model, while the host exec allowlist remains an independent approval boundary and generated schema help still implies channel/surface semantics.

## What This Changes

The issue asks OpenClaw to clarify or fix why WebChat/Control UI exec remains denied in allowlist mode after configuring elevated sender allow rules and `tools.byProvider.webchat` tool policy.

## Best Possible Solution

Align the public contract first: schema help, generated metadata, docs, and any Control UI config copy should consistently say whether `tools.byProvider` keys model providers/models or channel surfaces. If maintainers want WebChat/Control UI-specific host exec grants, add a separate explicit surface-policy seam with tests that preserve host exec allowlist and approval boundaries.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: configure `tools.byProvider.webchat` while the active model provider is not `webchat`, keep `tools.exec.security=allowlist` without a matching host allowlist or durable approval, and current main will not apply that provider policy before the gateway allowlist path can throw `exec denied: allowlist miss`.

## Solution Assessment

Unclear. Reinterpreting `tools.byProvider.webchat` as a Control UI permission is not the safest narrow fix; the safer first step is contract clarification, or an explicitly named surface-policy design if maintainers want browser-surface grants.
