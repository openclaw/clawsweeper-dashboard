## Summary

Keep open. Current main has gained a foreground stale-reply fence for buffered inbound dispatch, but it still lacks the unified per-session run generation/currentness contract, broad emission fencing, pre-tool currentness gate, and default new-message takeover requested here; active PRs https://github.com/openclaw/openclaw/pull/70363, https://github.com/openclaw/openclaw/pull/75226, and https://github.com/openclaw/openclaw/pull/73365 are the right review paths rather than a cleanup close.

## What This Changes

The issue requests coordinated run interruption semantics so `/stop` or a newer message invalidates the active run and fences stale side effects across replies, tools, queues, and subprocesses.

## Best Possible Solution

Keep this as the canonical core interruptibility tracker and coordinate the focused PRs until generation/currentness, stale-output fences, tool gates, process cancellation, takeover semantics, and regression coverage are all aligned.

## Reproduction Assessment

Yes. The gap is source-reproducible on current main by following the primitive abort path, missing ReplyOperation currentness contract, unfenced block-reply handler, and streaming-only active-run steering checks.
