## Summary

Keep open. Current main still has useful abort, queue cleanup, steering, and process-cancel primitives, but it does not implement the requested unified per-session generation invalidation, broad stale-output fencing, pre-tool currentness gate, or default new-message takeover. The same-author draft PR #70363 remains open and explicitly partial, and #75226 only addresses another part of the active-run steering surface.

## What This Changes

Request stronger run interruptibility so `/stop` or a new message invalidates the active run, cancels owned process/tool work, and prevents stale visible output.

## Best Possible Solution

Keep this as the canonical core interruptibility tracker. Review or split #70363, coordinate the narrower active-steering work in #75226, then land focused core changes for a session/run generation primitive, stale-output fences across reply/typing/followup/tool-progress/event emission paths, a Pi pre-tool currentness gate, process cancellation tied to the invalidated run scope, and regression tests proving stale runs cannot emit visible effects or launch further tools.

## Reproduction Assessment

Yes. The issue is source-reproducible on current main by inspecting the missing run-generation/currentness symbols, the primitive-only `/stop` path, unfenced reply delivery/runtime callbacks, and the boundary-based default steering docs.

Not applicable as a patch. The requested direction is still a plausible core fix, but the maintainable path is coordinated maintainer review of the open partial PRs rather than closing this issue or launching a competing automated repair.
