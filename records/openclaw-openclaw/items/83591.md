## Summary

Keep open. Current main still creates and handles the Discord Gateway WebSocket in-process, while Telegram has the isolated worker and durable spool pattern; the merged Discord ack-ordering work does not supersede this narrower ingress isolation request.

## What This Changes

The issue requests Discord gateway ingress isolation from main event-loop stalls, using an isolated-worker and durable-spool pattern similar to the existing Telegram fix.

## Best Possible Solution

Keep this as the canonical Discord design issue and land a maintainer-approved plugin-owned gateway ingress isolation path that preserves heartbeat, resume, sequence, replay, and saturation proof.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection shows Discord WebSocket receipt and dispatch remain in-process while the issue provides production liveness logs, so the coupling is source-reproducible but not runtime-measured here.

## Solution Assessment

Unclear as a final implementation. The Telegram worker/spool pattern is a strong precedent, but Discord needs a channel-specific design that preserves persistent Gateway semantics before a fix PR is safe.
