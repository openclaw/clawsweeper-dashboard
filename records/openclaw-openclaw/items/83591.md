## Summary

Keep open. Current main and the latest v2026.6.8 release still create and handle the Discord Gateway WebSocket in-process, while Telegram has the worker/spool isolation pattern this issue asks to mirror; the related Discord ack-ordering PR did not supersede the ingress-isolation request.

## What This Changes

The issue requests moving Discord Gateway WebSocket ingress off the main event loop with durable pickup/spooling comparable to Telegram's isolated polling worker.

## Best Possible Solution

Keep this as the canonical Discord ingress-isolation tracker and land a maintainer-approved Discord gateway worker/queue design with heartbeat, resume, sequence, replay, and live saturation proof.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection shows Discord WebSocket receipt and listener dispatch remain in-process while the issue provides production liveness logs, so the coupling is source-reproducible but not current-main runtime reproduced here.

## Solution Assessment

Unclear as a final implementation. The Telegram worker/spool pattern is a strong precedent, but Discord needs a channel-specific design that preserves Gateway heartbeat, resume, sequence, and replay semantics before a fix PR is safe.
