## Summary

Keep open. Current main still creates and handles the Discord Gateway WebSocket in the plugin runtime, while Telegram has the isolated worker and durable spool precedent; the related merged Discord ack-ordering PR does not implement this ingress isolation request.

## What This Changes

Request to isolate Discord Gateway ingress from main event-loop stalls using a worker and durable pickup queue pattern analogous to Telegram isolated ingress.

## Best Possible Solution

Keep this as the canonical Discord Gateway ingress isolation item and land a maintainer-approved plugin-owned design with live saturation proof preserving heartbeat, resume, sequence, and replay semantics.

## Reproduction Assessment

No high-confidence live current-main reproduction was run. Source inspection shows Discord WebSocket receipt and dispatch remain in-process while the issue and comments provide production stall evidence, so the coupling is source-reproducible but not live-reproduced here.

## Solution Assessment

Unclear as a final implementation. The Telegram worker/spool pattern is a strong precedent, but Discord needs a channel-specific Gateway design that preserves persistent WebSocket semantics before a fix PR is safe.
