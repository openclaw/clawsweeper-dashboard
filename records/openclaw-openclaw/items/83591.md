## Summary

Keep open. Current main still runs Discord gateway WebSocket receive and dispatch in the main plugin process; the Telegram isolated-worker/spool fix is present only on Telegram, so this Discord latency/resilience request remains actionable but needs design review rather than cleanup close.

## What This Changes

Discord ingress is reported to stall behind main event-loop work and the issue requests a Telegram-style isolated gateway worker with durable inbound spooling.

## Best Possible Solution

Design a Discord-owned ingress isolation path that keeps WebSocket receive, heartbeat/resume state, and durable inbound event capture independent of main-loop stalls while preserving current Discord ordering and listener semantics.

## Reproduction Assessment

Yes, at source level: Discord gateway receive currently runs on the main Node event loop, so a main-loop stall necessarily delays WebSocket message callbacks and dispatch. I did not establish a live Discord reproduction for the exact 1-4 s production latency profile.

## Solution Assessment

Unclear. Mirroring Telegram is directionally reasonable, but Discord has different WebSocket heartbeat/resume, interaction, voice, and event-ordering semantics, so the best fix needs maintainer design review before automation.
