## Label Justifications

- `P2`: The issue describes a credible Discord channel latency/reliability problem with limited blast radius, not a crash loop, security bypass, or unusable core runtime.
- `impact:message-loss`: The reported failure mode is inbound Discord messages being delayed or perceived as not picked up while the main event loop is stalled.

## Summary

Keep open. Current main and the latest shipped release still run Discord Gateway WebSocket ingress and dispatch in the main process; the Telegram worker/spool fix exists only on the Telegram path, so this remains credible Discord channel reliability work rather than cleanup.

## What This Changes

The issue requests Discord ingress isolation comparable to the merged Telegram ingress worker and durable spool work.

## Best Possible Solution

Design a Discord-owned isolated Gateway ingress path or equivalent durable pickup boundary that preserves Discord session, heartbeat, resume, and replay semantics while decoupling message pickup from main-thread stalls.

## Reproduction Assessment

No. Source inspection shows Discord WebSocket pickup and gateway dispatch are still main-process code, which supports the reported event-loop coupling, but no controlled live or fake-gateway saturation repro was run for the exact latency claim.

## Solution Assessment

Unclear as a final design. The Telegram worker/spool pattern is a strong precedent, but Discord needs a channel-specific transport design for heartbeat, sequence, resume, reconnect, and durable dispatch semantics before implementation.
