## Summary

Keep open. Current main and the latest release still create and handle the Discord Gateway WebSocket in the main process, while the isolated worker plus durable ingress spool remains Telegram-only; related broader event-loop issues do not supersede this narrower Discord ingress isolation request.

## What This Changes

The issue requests a Discord-owned isolated ingress worker and durable inbound spool so Discord message pickup is not delayed by main event-loop stalls.

## Best Possible Solution

Keep this open for a maintainer-approved Discord plugin design that isolates Gateway WebSocket receipt from main-thread stalls with durable replay/spooling or an equivalent channel-safe transport boundary.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection shows Discord WebSocket receipt and dispatch remain on the main process while the issue provides production liveness logs, so the coupling is source-reproducible but not runtime-measured here.

## Solution Assessment

Unclear as a final implementation. The Telegram worker/spool pattern is a strong precedent, but Discord needs a channel-specific design that preserves heartbeat, resume, sequence, and replay semantics before a fix PR is safe.
