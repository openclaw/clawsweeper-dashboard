## Label Justifications

- `P2`: The report is a credible Discord channel latency/reliability problem with limited blast radius, but it is not a crash loop, security bypass, or unusable core runtime.
- `impact:message-loss`: The issue concerns Discord inbound messages being delayed behind event-loop stalls and perceived as not being picked up by the bot.

## Summary

Keep open. Current main and the reported v2026.5.12 release still run Discord Gateway WebSocket ingress and listener dispatch inside the main process; the isolated worker plus durable spool pattern exists for Telegram only, so this is credible remaining Discord channel hardening work rather than implemented cleanup.

## What This Changes

The issue requests moving Discord ingress to an isolated worker with durable local spooling, mirroring the Telegram polling fix, so message pickup is not delayed by main event-loop stalls.

## Best Possible Solution

Design a Discord-owned isolated gateway ingress worker with durable event spooling, or an equivalent channel-level transport design, that preserves Discord Gateway resume/heartbeat semantics while decoupling message pickup from main-thread stalls.

## Reproduction Assessment

No high-confidence live reproduction was run in this review. Source inspection shows Discord WebSocket payload handling and listener dispatch remain on the main process, which supports the reported coupling but does not measure the reported production latency directly.

## Solution Assessment

Unclear as a final design. The Telegram worker/spool pattern is a strong precedent, but Discord needs a channel-specific design for gateway session state, heartbeat ACKs, resume behavior, durable dispatch, and replay semantics before implementation.
