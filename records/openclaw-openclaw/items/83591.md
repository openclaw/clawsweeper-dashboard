## Label Justifications

- `P2`: The issue is a credible Discord channel reliability and latency problem with limited blast radius, but it is not a crash loop, security bypass, or unusable core runtime.
- `impact:message-loss`: The report is about inbound Discord messages being delayed behind event-loop stalls and perceived as not being picked up by the bot.

## Summary

Keep open. Current main and the latest v2026.5.18 release still create and process the Discord Gateway WebSocket in the main process, while the worker plus durable spool pattern exists only for Telegram; duplicate search did not find a canonical Discord fix PR taking this over.

## What This Changes

The issue requests Discord Gateway ingress isolation and durable local spooling similar to the merged Telegram ingress worker fix.

## Best Possible Solution

Design a Discord-owned isolated gateway ingress worker or equivalent durable queue that preserves Discord Gateway heartbeat, resume, ordering, and dispatch semantics, then prove message pickup during deliberate main-thread saturation.

## Reproduction Assessment

No, not as a high-confidence live repro. Source inspection shows current-main Discord WebSocket receipt and gateway dispatch remain on the main event loop, but this review did not measure the reported production latency under saturation.

## Solution Assessment

Unclear as a final design. The Telegram worker/spool fix is a strong precedent, but Discord needs a channel-specific transport design for WebSocket session state, heartbeat ACKs, resume behavior, durable dispatch, and replay semantics.
