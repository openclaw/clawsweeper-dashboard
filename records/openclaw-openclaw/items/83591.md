## Summary

Keep open. Current main still handles Discord Gateway WebSocket ingress and listener dispatch in the main process, while the referenced isolated worker plus durable spool pattern exists for Telegram only; the remaining work is credible but needs Discord-specific design and controlled saturation proof.

## What This Changes

The issue requests Discord gateway ingress isolation with a durable local queue so message pickup is less coupled to main event-loop stalls.

## Best Possible Solution

Design a Discord-owned isolated gateway ingress worker or equivalent durable ingress buffer that preserves Discord Gateway heartbeat, sequence, resume, and replay semantics, then prove pickup during a deliberate main-thread stall.

## Reproduction Assessment

No high-confidence live reproduction was run in this review. Source inspection confirms current main receives Discord WebSocket payloads on the main process and only queues listener work after same-process dispatch, which supports the reported coupling.

## Solution Assessment

Unclear as a final design. The Telegram worker/spool pattern is a strong precedent, but Discord needs a channel-specific design for Gateway heartbeat, resume, sequence, durable dispatch, and replay behavior.
