## Summary

Keep open. Current main still wires `openclaw tasks maintenance` only to task-registry and Task Flow maintenance, while cron session-registry pruning remains a separate timer reaper. PR #74023 is the open implementation candidate with closing syntax for this issue, so the issue should stay open until that PR or an equivalent fix lands.

## What This Changes

Request `openclaw tasks maintenance --apply` to prune stale cron session-registry entries older than seven days while preserving entries for jobs still running in `jobs-state.json`.

## Best Possible Solution

Add a focused command-level session-registry maintenance pass that previews and applies stale cron run-session pruning, skips sessions for jobs still marked running, reports counts in text and JSON output, updates the user-facing docs and changelog, and covers preview/apply/running/non-cron cases in targeted tests.

## Reproduction Assessment

Yes. Source inspection against current main reproduces the gap: `tasksMaintenanceCommand` has no session-registry branch, and the only cron run-session pruning is the separate `sweepCronRunSessions` runtime path.

## Solution Assessment

Yes. The requested narrow extension to `tasks maintenance` is the maintainable boundary; the existing timer reaper is useful runtime cleanup but does not provide the operator-facing preview/apply sweep requested here.
