## Summary

Keep open. Current main still limits `openclaw tasks maintenance` to task-registry and Task Flow maintenance, while the linked implementation PR remains open and unmerged; the issue should stay open for normal review and landing of that PR or an equivalent focused fix.

## What This Changes

`openclaw tasks maintenance --apply` should also preview/apply conservative stale cron session registry pruning while protecting jobs currently running in `jobs-state.json`.

## Best Possible Solution

Land PR #74023 or an equivalent focused change that adds preview/apply session-registry maintenance to the tasks command, preserves currently running cron jobs from durable job state, updates text and JSON output plus docs/changelog, and locks the behavior with targeted command tests.

## Reproduction Assessment

Yes. Source inspection against current main reproduces the gap: the maintenance command has no session-registry branch, and the only existing cron session reaper runs from the cron timer path.

## Solution Assessment

Yes. A command-level sweep with explicit jobs-state protection is the narrow maintainable direction; because PR #74023 is already linked with closing syntax, the issue should wait for review and landing of that PR or an equivalent replacement.
