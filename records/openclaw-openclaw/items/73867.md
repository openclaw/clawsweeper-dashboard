## Summary

Keep open because current main still lacks command-level session-registry pruning, and open PR #74023 is the linked implementation candidate with closing syntax for this issue.

## What This Changes

Add a conservative `tasks maintenance --apply` sweep that prunes stale cron session-registry entries while preserving entries for currently running jobs.

## Best Possible Solution

Land #74023 or an equivalent focused command-level maintenance sweep with preview/apply output, running-job protection, docs, changelog, and regression coverage.

## Reproduction Assessment

Yes. Source inspection on current main shows `tasksMaintenanceCommand` has no session-registry pruning path, while the existing cron session reaper is a separate runtime timer cleanup.

## Solution Assessment

Yes. A focused `tasks maintenance` preview/apply sweep with jobs-state running protection is the narrow maintainable direction; the background retention reaper alone does not provide explicit operator cleanup behavior.
