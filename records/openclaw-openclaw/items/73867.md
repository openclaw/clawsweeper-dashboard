## Summary

Keep open. Current main still wires `openclaw tasks maintenance` only to task-registry and Task Flow maintenance, while cron session pruning remains a separate timer reaper; the discussion also points to open PR #74023 as the active closing implementation candidate, so this issue should stay open until that PR or an equivalent fix lands.

## What This Changes

Request command-level `openclaw tasks maintenance --apply` cleanup for stale cron session-registry entries older than seven days while preserving entries for jobs still marked running in durable cron state.

## Best Possible Solution

Add a focused command-level session-registry maintenance pass that previews and applies stale cron run-session pruning, skips any job still running according to durable cron state, reports counts in text and JSON output, documents the behavior, and covers preview/apply/running/non-cron cases in targeted tests.

## Reproduction Assessment

Yes. Source inspection against current main reproduces the gap: `tasksMaintenanceCommand` has no session-registry branch, and the only cron run-session pruning is the timer-driven `sweepCronRunSessions` path.

## Solution Assessment

Yes. The requested narrow maintenance-command extension is the right boundary; the existing timer reaper is useful runtime cleanup but does not provide the operator-facing preview/apply sweep requested here.
