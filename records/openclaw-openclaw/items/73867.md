## Summary

Keep open because current main still lacks a `tasks maintenance` session-registry sweep, and the linked implementation PR #74023 remains open and unmerged with closing syntax for this issue.

## What This Changes

The issue asks for `openclaw tasks maintenance --apply` to conservatively prune stale session-registry entries while preserving sessions still referenced by the running set in `jobs-state.json`.

## Best Possible Solution

Land #74023 or an equivalent focused command-level sweep that adds preview/apply session-registry cleanup, running-job protection, docs, changelog, and regression coverage.

## Reproduction Assessment

Yes. Source inspection against current main reproduces the gap: `tasksMaintenanceCommand` has no session-registry maintenance path, while cron run session pruning exists only as a separate timer reaper.

## Solution Assessment

Yes. A `tasks maintenance` preview/apply sweep with `jobs-state.json` running-set protection is the narrow maintainable solution, and the existing linked PR should be reviewed rather than duplicating the work.
