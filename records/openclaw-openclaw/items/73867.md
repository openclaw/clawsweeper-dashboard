## Summary

Keep open. Current main still does not wire session-registry pruning into `openclaw tasks maintenance`; the existing cron session reaper is a separate timer-path cleanup, and PR #74023 is still open as the implementation candidate that explicitly links this issue for closure.

## What This Changes

The issue requests adding conservative session-registry pruning to `openclaw tasks maintenance --apply`, using a 7-day stale threshold while preserving cron jobs currently running according to durable job state.

## Best Possible Solution

Review and land PR #74023, or an equivalent focused change, so `openclaw tasks maintenance` previews and applies stale cron run session-registry cleanup while preserving currently running cron jobs and documenting the new JSON/text output.

## Reproduction Assessment

Yes. Source inspection on current main reproduces the gap: `tasksMaintenanceCommand` has no session-registry maintenance branch, and the only cron run session pruning is the separate cron timer reaper.

## Solution Assessment

Yes. A narrow command-level sweep with durable jobs-state running protection is the right maintainable boundary; it should be handled by the existing open PR or a replacement if that PR is not landable.
