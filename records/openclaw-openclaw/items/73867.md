## Summary

Keep open. Current main still does not add a command-level session-registry sweep to `openclaw tasks maintenance`, and the linked closing PR remains open, so this issue should stay open for normal review and landing of that PR or an equivalent focused fix.

## What This Changes

The issue requests that `openclaw tasks maintenance --apply` also prune stale session-registry entries using conservative age and running-job protection criteria.

## Best Possible Solution

Land PR #74023 or an equivalent focused change that adds preview/apply session-registry maintenance to `tasksMaintenanceCommand`, preserves sessions for jobs currently running in durable job state, updates text and JSON output plus `docs/cli/tasks.md`, and adds targeted command tests.

Yes. Source inspection against current main reproduces the gap: the maintenance command has no session-registry branch or output field, while existing cron session pruning is wired through the cron timer path instead.
