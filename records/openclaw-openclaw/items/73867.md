## Summary

Keep open. Current main still scopes `openclaw tasks maintenance` to task and Task Flow maintenance, while cron session-registry pruning remains a separate timer-driven cleanup path. PR #74023 is already open and unmerged with closing syntax for this issue, so the issue should stay open until that PR or an equivalent fix lands.

## What This Changes

Request to extend `openclaw tasks maintenance --apply` so it also previews and applies conservative stale cron session-registry pruning while preserving entries for jobs still running in `jobs-state.json`.

## Best Possible Solution

Add a focused command-level session-registry maintenance pass that previews and applies stale cron run-session pruning, skips jobs still marked running in durable cron state, reports counts in text and JSON output, documents the behavior, and covers preview/apply/running/non-cron cases in targeted tests.

## Reproduction Assessment

Yes. Source inspection against current main reproduces the gap: `tasksMaintenanceCommand` has no session-registry branch, and the only cron run-session pruning is the timer-driven `sweepCronRunSessions` path.

## Solution Assessment

Yes. The requested command-level extension is the right boundary because the existing timer reaper is useful runtime cleanup but does not provide the operator-facing preview/apply sweep requested here; PR #74023 or an equivalent patch should still be reviewed before merge.
