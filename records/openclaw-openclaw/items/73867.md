## Summary

Keep open. Current main still limits `openclaw tasks maintenance` to task-registry and Task Flow maintenance, while cron session pruning only runs from the cron timer path and does not provide a command-level, jobs-state-protected session registry sweep. The linked implementation PR #74023 remains open, so this issue should wait for that PR or an equivalent focused fix to land.

## What This Changes

Request that `openclaw tasks maintenance --apply` also preview/apply conservative pruning of stale cron session registry entries while protecting sessions for cron jobs still running in `jobs-state.json`.

## Best Possible Solution

Land PR #74023 or an equivalent focused change that adds preview/apply session-registry maintenance to `tasksMaintenanceCommand`, preserves cron sessions for jobs still running in durable job state, updates text and JSON output plus docs/changelog, and covers preview/apply behavior with targeted command tests.

## Reproduction Assessment

Yes. Source inspection against current main reproduces the gap: `tasksMaintenanceCommand` has no session-registry maintenance branch, and the only cron session pruning path is the timer-driven reaper.

## Solution Assessment

Yes. A command-level preview/apply sweep with explicit jobs-state protection is the narrow maintainable path; because PR #74023 already targets this issue with closing syntax, the next step is to review and land that PR or a focused equivalent rather than open a duplicate repair.
