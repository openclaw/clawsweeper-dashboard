## Summary

Close as implemented on current main: the merged OpenRouter normalization change makes `openrouter/minimax/minimax-m3` use the upstream `minimax/minimax-m3` capability key, and the existing OpenAI-compatible transport already captures and replays OpenRouter `reasoning` / `reasoning_details` fields.

## What This Changes

The issue asks OpenClaw to preserve OpenRouter MiniMax M3 `reasoning` and `reasoning_details` across plain-text and tool-call assistant history.

## Best Possible Solution

Keep the current OpenRouter-owned normalization and replay paths; users on v2026.6.6 should retest with a build containing the main-only fix or the next release.

## Reproduction Assessment

Yes at source level. v2026.6.6 used the provider-qualified OpenRouter id for capability lookup while OpenRouter's live catalog key is unprefixed and reasoning-capable, which explains the reported `model.reasoning` false/drop path without needing a local credentialed run.

## Solution Assessment

Yes. The best fix is the current OpenRouter plugin normalization path because it corrects provider-owned model metadata before the shared replay code makes capture and history-preservation decisions.
