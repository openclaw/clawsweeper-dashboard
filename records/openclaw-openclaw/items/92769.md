## Label Justifications

- `P1`: The report describes a user-facing regression where provider reasoning history is silently lost across agent turns for a real OpenRouter model.
- `impact:data-loss`: Returned reasoning state is dropped from subsequent request history in the reported release behavior.
- `impact:session-state`: The lost reasoning changes multi-turn transcript/context continuity and causes the model to answer inconsistently.
- `impact:auth-provider`: The affected path is provider/model routing for MiniMax M3 through OpenRouter.

## Summary

Current main already handles the reported OpenRouter MiniMax M3 shape: `reasoning_details` text deltas are preserved as thinking for plain-text and tool-call turns, then replayed as OpenRouter-compatible reasoning on the next request. The fix is current-main-only and is not contained in the latest `v2026.6.6` release tag.

## What This Changes

The issue reports that MiniMax M3 via OpenRouter drops returned `reasoning`/`reasoning_details` from subsequent assistant message history for both plain-text and tool-call turns.

## Best Possible Solution

Close this issue as implemented on current main and let the next release carry the OpenRouter reasoning replay fix to release users.

## Reproduction Assessment

Yes for the reported release behavior: the issue includes concrete OpenRouter payload evidence, and current tests cover the same `reasoning_details` plain-text and tool-call shapes. I did not run a live MiniMax M3 request in this read-only pass.

## Solution Assessment

Yes. Preserving provider reasoning in the OpenAI-compatible transport parser and replay sanitizer is the right boundary because it covers both visible and tool-call turns immediately before the next Chat Completions request is built.
