## Summary

This PR targets a real P1 Telegram message-loss bug, but it does not yet fix the complete failure path: the newly propagated debouncer rejection is caught by the outer Telegram message handler, so update tracking still records success and the spool consumer can still delete the failed update. The overlapping https://github.com/openclaw/openclaw/pull/92233 identifies this same gap, but remains open and is not yet a safe superseding target.

## What This Changes

The PR rethrows Telegram dispatch failures, adds an opt-in inbound-debouncer error-rethrow mode for Telegram, and updates focused unit tests to expect rejection.

## Best Possible Solution

Use one canonical per-update failure outcome at the Telegram update/spool boundary, preserve the intended user-visible fallback, and prove that failed claimed updates are released or dead-lettered while only completed updates are deleted.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: dispatch rejects, the debouncer rethrows, `handleInboundMessageLike` catches and returns, the tracker records completion, and the spool consumer deletes the claim.

## Solution Assessment

No. The PR repairs two inner swallow sites but not the outer Telegram handler that determines whether update tracking and spool retention observe failure.
