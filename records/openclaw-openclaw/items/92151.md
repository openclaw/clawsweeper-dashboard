## What This Changes

The PR makes Telegram message-processing and selected inbound-debouncer flush errors rethrow toward callers so failed spooled updates can be retried.

## Best Possible Solution

Use one Telegram-owned failure outcome that reaches both the update tracker and isolated-ingress spool consumer, releases or dead-letters failed claims without bypassing replay eligibility, and assigns user-visible fallback delivery to exactly one layer.

## Reproduction Assessment

Yes. The current-main and branch control flow provides a high-confidence source reproduction: the proposed rejection is caught by `handleInboundMessageLike`, the tracker records completion, and the spool consumer deletes the claim after `handleUpdate()` resolves.

## Solution Assessment

No. The proposed rethrow stops two early swallows but not the later Telegram handler catch, and it lacks tracker-safe replay and single-owner fallback handling; maintainers must also choose between this narrow direction and the competing tracker-aware design.
