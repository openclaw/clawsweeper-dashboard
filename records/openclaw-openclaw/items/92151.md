## Summary

This PR fixes the immediate, non-buffered failure path, but it does not preserve the spool retry contract when Telegram uses timer-backed inbound debounce. The related https://github.com/openclaw/openclaw/pull/92233 is also open and not yet a safe superseding target, so this branch should remain open for a focused repair.

## What This Changes

The PR rethrows Telegram message-dispatch failures and adds an opt-in shared-debouncer mode intended to propagate flush failures to the isolated-ingress spool consumer.

## Best Possible Solution

Make Telegram's update tracker or spool consumer own a per-update terminal outcome across immediate and buffered dispatch, deleting a claim only after confirmed completion and releasing or dead-lettering the exact failed update without changing other channels' debounce defaults.

## Reproduction Assessment

Yes. Set Telegram inbound debounce above zero and process a claimed spooled update: the source deterministically shows `enqueue` returning after scheduling while dispatch runs later on the timer-owned promise.

## Solution Assessment

No. Rethrowing is sufficient for immediately awaited flushes, but caller rejection is not a complete durability contract when batching detaches dispatch from the originating update promise.
