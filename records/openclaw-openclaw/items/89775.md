## Summary

Keep this PR open: current main and the latest release still read `entry.factory` directly, while the branch is a focused viable guard with regression coverage; it is maintainer-authored/draft and should proceed through normal maintainer landing rather than cleanup close.

## What This Changes

The PR hardens Codex app-server extension factory listing so unreadable or malformed active-registry entries are skipped while healthy factories are returned in order.

## Best Possible Solution

Land the narrow guard and regression test after maintainer review, required validation, and any needed draft/rebase cleanup, without changing the plugin API or Codex protocol surface.

## Reproduction Assessment

Yes. By source inspection, current main reads `entry.factory` during listing, so a throwing accessor in the active registry can abort discovery before healthy factories are returned; the branch adds a focused regression test for that path.

## Solution Assessment

Yes. The best fix is to harden the discovery boundary itself, because changing registration would not cover corrupted active-registry rows and moving the guard later would still allow the default factory listing to throw.
