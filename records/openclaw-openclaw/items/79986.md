## Summary

Keep open: current main still has a source-level raw subagent tool-output fallback that can feed the requester completion event, and this issue adds recent Telegram `mode=run` evidence plus a separate duplicate-progress symptom that is not fully explained by the canonical auto-compaction report.

## What This Changes

The issue reports Telegram DM subagent `mode=run` completions exposing raw child tool output and duplicating the parent progress acknowledgement.

## Best Possible Solution

Consolidate the existing subagent announce PRs into one fix that prevents raw tool/toolResult text from becoming a successful completion result across direct capture and frozen-result refresh, while preserving bounded timeout/error summaries and adding Telegram-oriented regression coverage.

## Reproduction Assessment

Yes for the raw-output half from source: current main still selects latest tool/toolResult text when assistant output is empty, and tests assert that fallback. The duplicate Telegram progress bubble is not yet independently reproduced.

## Solution Assessment

No; the current requester-agent handoff is safer than direct raw-send, but it can still carry raw child tool output as completion data. The safer fix is in the shared subagent output/capture contract, not in Telegram delivery alone.
