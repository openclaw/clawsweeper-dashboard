## Summary

Keep this PR open: current main does not contain the requested runtime auth override API, but the PR still has source-level correctness and security-routing blockers plus no real plugin-run proof from the external contributor.

## What This Changes

The PR adds a public plugin API for runtime model-provider auth overrides, wires it into auth and media selection, and adds registry/runtime tests.

## Best Possible Solution

Land a revised additive SDK/auth seam that normalizes provider ids, applies per-call transport overrides in the actual runner path, preserves model context and fallback behavior, documents the public API, and includes maintainer/security approval plus real plugin-run proof.

## Reproduction Assessment

Not applicable for the feature request itself; yes for the review blockers, because PR-head source shows raw provider matching, empty model fallback, unapplied transport fields, and first-entry image selection without needing a live run.

## Solution Assessment

No as-is; a dedicated additive API is plausible, but this patch needs normalized matching, real application of base URL/header overrides, model-aware callback context, safer media fallback behavior, docs, and real behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Normalize provider IDs before matching auth overrides:** `src/agents/model-auth.ts:392`
  - body: Override matching uses raw string equality, but current auth resolution accepts aliases such as `aws-bedrock` for `amazon-bedrock`. A plugin registered for the canonical provider can be skipped and OpenClaw may fall back to normal credentials instead of the intended brokered auth; normalize both requested and registered provider ids, including the availability check.
  - confidence: 0.9
- **[P2] Apply transport overrides in the runner path:** `src/agents/model-auth.ts:427-428`
  - body: Returning `baseUrl` and `providerRequestHeaders` from auth resolution does not make the main LLM runner use them: the auth controller still forwards only API key, mode, and profile to runtime auth preparation and then stores the key. A plugin that returns a proxy URL or required headers will still send the brokered key through the configured/default provider route, so these fields need to be threaded into the model/request setup that actually performs inference.
  - confidence: 0.86
- **[P2] Thread known model IDs into auth override callbacks:** `src/agents/model-auth.ts:399`
  - body: The callback receives `params.modelId ?? ""`, but media execution resolves provider auth without passing the selected entry model. Model-scoped credential brokers can then see an empty model id and return null or the wrong credential; thread known models through resolver calls or make the context explicitly optional.
  - confidence: 0.84
- **[P2] Preserve image fallback for tentative override candidates:** `src/media-understanding/runner.ts:594`
  - body: `resolveAutoImageModel` uses the override-aware candidate list but returns only `keyEntries[0]`. If the first provider is conditionally override-backed and fails at request time, callers of this single-model helper never consider later key-backed providers; preserve ordered fallback behavior or exclude tentative override-only providers here.
  - confidence: 0.76

## Security Review

Status: needs_attention

Summary: No dependency or workflow supply-chain issue was found, but the new trusted plugin auth override seam still has concrete auth-routing concerns.

Concerns:

- **[medium] Alias calls can bypass brokered auth overrides:** `src/agents/model-auth.ts:392`
  - body: Raw provider-id matching means an alias-form request can skip a registered canonical override and use default credentials instead, undermining plugins intended to broker all auth for a provider family.
  - confidence: 0.84
- **[medium] Transport overrides can be ignored after credential resolution:** `src/agents/model-auth.ts:427`
  - body: The PR returns per-call base URL and header overrides from auth resolution, but the main runner path does not consume those fields from the resolved auth object, which can route a brokered credential to the wrong endpoint or omit required proxy headers.
  - confidence: 0.82
- **[low] Model-scoped broker context can be erased:** `src/agents/model-auth.ts:399`
  - body: Side paths can call the new override surface with an empty model id even when the selected model is known, which can weaken model-scoped broker authorization or route a request through unintended credentials.
  - confidence: 0.75
