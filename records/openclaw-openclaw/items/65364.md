## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Normalize provider IDs before matching overrides:** `src/agents/model-auth.ts:392`
  - body: The override loop compares raw provider ids with `reg.override.providers.includes(provider)`, but current auth paths normalize aliases such as `aws-bedrock` to `amazon-bedrock`. A plugin registered for the canonical provider can be bypassed by an alias-form request, falling back to non-override auth; normalize both registration ids and incoming provider ids in resolution and availability checks.
  - confidence: 0.88
- **[P2] Pass real model context to override callbacks:** `src/agents/model-auth.ts:399`
  - body: The API promises a model id, but missing values are converted to `""` and known-model call sites still call `resolveApiKeyForProvider` without passing the model. Model-scoped brokers will see an empty id and can return null or wrong credentials, so thread `modelId` through call sites that know it or make the callback field explicitly optional.
  - confidence: 0.82
- **[P2] Preserve fallback when resolving auto image auth:** `src/media-understanding/runner.ts:594`
  - body: `resolveAutoImageModel` builds an ordered key-entry list for override-backed providers but returns only `keyEntries[0]`. Because override registration is only a may-be-available signal, direct callers can fail on a conditional broker and never consider later providers with real keys.
  - confidence: 0.8
