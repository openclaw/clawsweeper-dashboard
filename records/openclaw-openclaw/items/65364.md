## Summary

Keep this PR open. Current main still lacks the proposed plugin SDK API, and the related feature issue was already treated as superseded by this PR, but the branch is not merge-ready because the latest diff still has concrete auth-routing and fallback gaps in the new public runtime-auth seam.

## What This Changes

The branch adds a plugin SDK registration API for provider runtime auth overrides, stores override registrations in plugin registry/runtime state, applies returned auth/base URL/headers during model auth, adjusts media auto-selection fallback, and adds focused tests.

## Best Possible Solution

Land a revised additive plugin-SDK/core seam that lets trusted plugins provide per-call provider credentials, base URL, and request headers without replacing provider implementations, while preserving canonical provider matching, accurate model context, conservative availability semantics, ordered fallback behavior, documentation, changelog coverage, and changed-gate proof.

## Reproduction Assessment

Not applicable for the feature request itself; yes for the review blockers, because the latest PR diff shows raw provider matching, empty-string model fallback, and first-entry image selection while current main shows normalized provider handling and side-path resolver calls.

## Solution Assessment

Is this the best way to solve the issue? Unclear as-is. A dedicated additive SDK registration seam is a plausible maintainable direction, but this patch should normalize provider ids, thread known model ids, preserve image fallback behavior, and update docs/changelog before it is the best implementation to merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Normalize provider IDs before matching overrides:** `src/agents/model-auth.ts:390`
  - body: The override loop matches registrations with raw string equality. Current auth resolution accepts normalized aliases such as `aws-bedrock` for `amazon-bedrock`, so a plugin registered for the canonical provider can be skipped and OpenClaw may fall back to normal credentials instead of the intended brokered auth. Normalize both requested and registered provider ids, including the availability check.
  - confidence: 0.88
- **[P2] Pass known model IDs into override callbacks:** `src/agents/model-auth.ts:399`
  - body: The callback receives `params.modelId ?? ""`, but several side paths already know the selected model and still call `resolveApiKeyForProvider` without passing it. Model-scoped credential brokers can then see an empty model id and return null or the wrong credential. Thread the known model through those resolver calls or make the context explicitly optional.
  - confidence: 0.82
- **[P2] Keep image auto selection from pinning tentative providers:** `src/media-understanding/runner.ts:604`
  - body: `resolveAutoImageModel` now uses the override-aware candidate list but immediately returns `keyEntries[0]`. If the first provider is only conditionally override-backed and fails at request time, callers of this single-model helper never consider later key-backed providers. Preserve ordered fallback behavior or keep tentative override-only providers out of this single-result path.
  - confidence: 0.76

## Security Review

Status: needs_attention

Summary: The diff does not add dependency, workflow, or package-resolution supply-chain changes, but it introduces a trusted plugin auth-override capability with concrete auth-routing gaps that need attention before merge.

Concerns:

  - body: Raw provider-id matching means an alias-form request can skip a registered canonical override and use default credentials instead, undermining the policy boundary for plugins intended to broker all auth for a provider family.
