## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Route override headers through request policy:** `src/agents/model-auth.ts:772`
  - body: This merges plugin-provided headers directly into the model, bypassing the current runtime request override path that sanitizes and normalizes transport headers. A plugin auth API should return the existing request override shape and flow through `sanitizeRuntimeProviderRequestOverrides` / `resolveProviderRequestConfig` so header precedence and forbidden transport policy stay centralized.
  - confidence: 0.86
- **[P2] Normalize provider IDs before matching overrides:** `src/agents/model-auth.ts:392`
  - body: Override matching uses raw string equality, while provider auth resolution elsewhere normalizes aliases such as `aws-bedrock` to `amazon-bedrock`. A plugin registering the canonical provider can be skipped for alias callers, and the availability check has the same mismatch.
  - confidence: 0.88
- **[P2] Pass real model context to override callbacks:** `src/agents/model-auth.ts:399`
  - body: The new callback promises a `modelId`, but this fallback sends an empty string whenever the caller did not thread the model through `resolveApiKeyForProvider`. Several side paths call that resolver directly, so model-scoped brokers can return the wrong credential or `null` even when the selected model is known upstream.
  - confidence: 0.82
- **[P2] Preserve fallback for auto image auth selection:** `src/media-understanding/runner.ts:594`
  - body: The PR teaches key resolution to return an ordered candidate list, but `resolveAutoImageModel` still collapses it to the first entry. If that first provider is only conditionally override-backed and later fails at request time, image callers such as sticker description can miss later providers with usable credentials.
  - confidence: 0.78
