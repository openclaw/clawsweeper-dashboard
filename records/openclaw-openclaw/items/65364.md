## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Normalize provider IDs before matching overrides:** `src/agents/model-auth.ts:392`
  - body: The new override loop compares raw provider ids with `reg.override.providers.includes(provider)`, but current auth code accepts aliases such as `bedrock` and `aws-bedrock` for `amazon-bedrock`. A plugin registered for the canonical provider can be bypassed by an alias-form request, falling back to non-override auth and weakening the brokered-auth path; normalize both the registration ids and incoming provider in resolution and availability checks.
  - confidence: 0.88
- **[P2] Pass real model context to override callbacks:** `src/agents/model-auth.ts:399`
  - body: The API promises a `modelId`, but missing values are converted to `""` and several known-model call sites still use `resolveApiKeyForProvider` directly. Model-scoped brokers will see an empty id and can return null or the wrong credentials, so thread `modelId` through call sites that know it or make the callback field explicitly optional.
  - confidence: 0.82
- **[P2] Preserve fallback when resolving auto image auth:** `src/media-understanding/runner.ts:594`
  - body: The PR builds an ordered key-entry list for override-backed providers but `resolveAutoImageModel` returns only `keyEntries[0]`. Because override registration is only a may-be-available signal, direct callers that use this single active image model can fail on a conditional broker and never consider later providers with real keys.
  - confidence: 0.8
