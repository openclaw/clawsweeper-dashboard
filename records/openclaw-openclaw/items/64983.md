## Summary

Keep open. Current main has partial coverage for Telegram dispatch failures and context-overflow fallback text, but the cron no-op timeout and gateway PID/port ownership behavior still need maintainer decisions before this can be closed conservatively.

## What This Changes

The issue reports three reliability symptoms: a silent Telegram no-reply after context overflow, cron failures when a no-op job should produce `NO_REPLY`, and gateway PID/listening-port ownership mismatch.

## Best Possible Solution

Keep this open as an umbrella or split it into narrower follow-ups: add a Telegram regression once sanitized traces identify the exact assistant-error path, define a cron terminal no-op contract that bypasses unnecessary final model generation for explicit no-send outcomes, and decide whether gateway ownership repair belongs in restart/doctor/status/startup flows.

## Reproduction Assessment

Unclear overall. Static current-main inspection gives a high-confidence path for the cron timeout gap and gateway diagnostic-only behavior, but the precise Telegram dropped-reply reproduction still depends on the reporter’s sanitized JSONL or timestamps.

## Solution Assessment

No. A single bundled fix is not the best path; this crosses Telegram delivery, cron execution semantics, and gateway process ownership, so the maintainable solution is separate subsystem work with explicit contracts.
