## Summary

Keep open. Current main has partial coverage for the Telegram symptom through dispatch fallback and embedded context-overflow recovery, but the bundled cron and gateway behaviors are still not fully implemented: isolated cron still treats provider/model failures before final output as job errors, and gateway status still diagnoses PID/port mismatch while cleanup is limited to restart/update flows.

## What This Changes

Request to make Telegram, cron, and gateway reliability failures fail visibly or self-heal instead of dropping replies, false-failing no-op cron runs, or leaving mismatched gateway ownership.

## Best Possible Solution

Keep this open and split it into narrower follow-ups: one Telegram trace-verification item, one cron terminal no-op contract for explicit no-send business outcomes, and one gateway ownership reconciliation decision for status/doctor/restart behavior.

## Reproduction Assessment

Unclear overall. Static current-main checks give high confidence for the cron timeout gap and gateway diagnostic-only behavior, but the precise Telegram dropped-reply path still needs the reporter's sanitized JSONL/timestamps.

## Solution Assessment

No. A single bundled fix is not the best path; the maintainable solution is separate subsystem changes with an explicit cron no-op contract and a gateway reconciliation policy.
