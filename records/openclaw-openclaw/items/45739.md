## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Use the shared run-wait helpers:** `src/agents/acp-spawn-parent-stream.ts:282-297`
  - body: Current main already centralizes `chat.history` assistant extraction, fingerprints, and `agent.wait` normalization in `src/agents/run-wait.ts`. Reimplementing those gateway reads here creates a second source of truth for filtering and terminal-state semantics, so the rebased fallback should call the shared helpers instead of parsing raw gateway responses locally.
  - confidence: 0.86
- **[P1] Keep polling history before terminal completion:** `src/agents/acp-spawn-parent-stream.ts:381-384`
  - body: When `agent.wait` returns `ok`, this path does one forced history read, flushes, emits completion, and disposes. If transcript persistence lags the terminal wait result, the parent can still receive completion without the child assistant output, which is the failure mode this fallback is supposed to recover.
  - confidence: 0.84
- **[P1] Set the local-assistant flag after text is relayed:** `src/agents/acp-spawn-parent-stream.ts:452-459`
  - body: This marks `sawLocalAssistantEvent` before checking whether the assistant event has non-empty relayable text. A whitespace or otherwise empty assistant event can disable history backfill even though no output reached the parent, leaving the relay stuck on start/stall-only notices.
  - confidence: 0.88
- **[P2] Re-check the history baseline after awaiting:** `src/agents/acp-spawn-parent-stream.ts:300-309`
  - body: `primeHistoryBaseline` can race with the first fallback poll: the poll can initialize and relay a newer history reply while the priming read is still in flight, then the priming call overwrites `lastHistoryAssistant` with an older value. Re-check `historyBaselineReady` after the await before writing baseline state.
  - confidence: 0.82
