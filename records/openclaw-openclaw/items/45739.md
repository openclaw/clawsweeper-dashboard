## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Retry history before disposing after terminal state:** `src/agents/acp-spawn-parent-stream.ts:381-384`
  - body: Both terminal fallback paths perform a single forced history read and then emit completion/dispose. If ACP terminal state is observed before the child transcript is persisted, this still permanently misses the assistant output in the missing-local-event case this PR is meant to recover; keep the relay alive for a bounded retry/read-after-terminal window before completing.
  - confidence: 0.9
- **[P2] Set local-output state only after relaying text:** `src/agents/acp-spawn-parent-stream.ts:453-459`
  - body: `sawLocalAssistantEvent` is set before validating that the assistant event has relayable final text. An empty, whitespace-only, or commentary-only local event can suppress history backfill even though the parent received no progress output; gate this state on a successful non-commentary relay.
  - confidence: 0.9
- **[P2] Recheck the history baseline after awaiting:** `src/agents/acp-spawn-parent-stream.ts:304-309`
  - body: `primeHistoryBaseline()` can race the first fallback poll: the poll may initialize and relay the real reply while the priming read is in flight, then the slower priming completion overwrites `lastHistoryAssistant` with stale data and allows a duplicate relay later. Recheck `historyBaselineReady` after the await before writing baseline state.
  - confidence: 0.88
- **[P2] Deduplicate delayed local deltas after history backfill:** `src/agents/acp-spawn-parent-stream.ts:459`
  - body: If the history fallback emits the full assistant text before delayed local assistant deltas arrive, the local path can append overlapping content again because it does not compare against already-relayed history output. Add a guard or regression test for history-first, local-second delivery.
  - confidence: 0.8
