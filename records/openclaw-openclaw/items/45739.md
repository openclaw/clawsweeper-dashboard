Full review comments:

- **[P1] Reuse the current run-wait helpers:** `src/agents/acp-spawn-parent-stream.ts:282-290`
  - body: Current main already centralizes `chat.history` assistant snapshots and `agent.wait` normalization in `src/agents/run-wait.ts`. This patch adds a separate parser/caller path in the relay, which would drift on fingerprinting, timeout/error status handling, and duplicate detection after rebase.
  - confidence: 0.86
- **[P1] Preserve commentary filtering on rebase:** `src/agents/acp-spawn-parent-stream.ts:450-459`
  - body: Current main suppresses `assistantPhase === "commentary"` before relaying child text. The PR-head handler routes every non-empty assistant payload through `recordAssistantOutput`, so a rebase that keeps this shape can leak commentary progress back to the parent session.
  - confidence: 0.84
- **[P2] Recheck the history baseline after awaiting:** `src/agents/acp-spawn-parent-stream.ts:304-309`
  - body: `primeHistoryBaseline` can race the first fallback poll: the poll may initialize and relay the real reply, then the slower priming read overwrites `lastHistoryAssistant` with stale data, causing duplicate relay output later.
  - confidence: 0.9
- **[P2] Mark local assistant output only after relaying text:** `src/agents/acp-spawn-parent-stream.ts:453-459`
  - body: `sawLocalAssistantEvent` is set before `recordAssistantOutput` confirms non-empty relayable text. A whitespace-only assistant event can suppress history backfill even though the parent saw no progress output.
  - confidence: 0.87
- **[P2] Retry history before completing gateway fallback:** `src/agents/acp-spawn-parent-stream.ts:381-384`
  - body: When `agent.wait` reports success, the fallback performs one history read, emits completion, and disposes. If `chat.history` lags behind run completion, assistant output can still be lost permanently in the missing-local-events path this patch is meant to recover.
  - confidence: 0.74
