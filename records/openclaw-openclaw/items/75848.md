## Summary

Keep open. Current main still treats an explicit `expectsCompletionMessage:false` as a non-completion announce mode rather than a no-handoff mode: the flag is parsed and stored, but completion cleanup still invokes the announce flow and dispatch can queue, steer, or direct-deliver into the requester session. Existing tests currently assert that behavior, so this remains a valid narrow bug rather than cleanup-ready.

## What This Changes

The issue asks for `sessions_spawn` with `expectsCompletionMessage:false` to suppress the subagent completion handoff so the parent session keeps its active context.

## Best Possible Solution

Add an explicit no-completion-handoff path for `sessions_spawn` false mode that still records the run outcome, performs browser/runtime/attachment cleanup, and preserves default announce behavior when the flag is omitted or true; add regression coverage for active requester queue/steer suppression and direct fallback suppression.

## Reproduction Assessment

Yes. The high-confidence source-level reproduction is to trace a `sessions_spawn` call with `expectsCompletionMessage:false`: current main registers false, then `completeSubagentRun` still starts announce cleanup, and dispatch can queue or direct-deliver into the requester session. I did not run tests because this review was required to stay read-only.

## Solution Assessment

Yes. A narrow registry lifecycle or announce-dispatch boundary fix is the best maintainable direction; changing the default globally or relying on docs would not preserve cleanup while preventing the parent-session handoff.
