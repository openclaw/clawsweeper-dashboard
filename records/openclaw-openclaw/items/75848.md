## Summary

Keep this issue open. Current main parses and stores `expectsCompletionMessage: false`, but the completion cleanup path still runs the announce flow and the false branch can still enqueue or steer the requester session under the default queue mode, which matches the reported parent-context displacement risk.

## What This Changes

The issue asks for `sessions_spawn({ expectsCompletionMessage: false })` to behave as a fire-and-forget subagent spawn that does not steer or displace the parent session when the child finishes.

## Best Possible Solution

Honor explicit fire-and-forget spawns by preventing the child completion from injecting, queueing, or steering the requester session, while preserving cleanup/bookkeeping and any deliberately separate announce/recovery paths that still need external delivery.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is an active requester session with default `messages.queue.mode` (`steer`), a completed registered subagent run whose record has `expectsCompletionMessage: false`, and lifecycle cleanup enabled; current main still enters the announce flow and the false dispatch branch can call `queueEmbeddedPiMessage`.

## Solution Assessment

Yes, the narrow maintainable direction is to gate the completion announce at the lifecycle or dispatch boundary for explicit fire-and-forget runs, with regression coverage that asserts no queue/direct parent wake occurs while cleanup still completes. The repair should avoid a broad removal of all false-mode announce calls because other internal paths use the same boolean for different delivery semantics.
