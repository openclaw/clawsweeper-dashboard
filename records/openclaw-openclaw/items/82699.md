## Summary

Needs changes before merge: the linked CLI SecretRef bug is real and the PR is aimed at the right path, but the updated target helper still includes unrelated plugin SecretInput targets and can make web commands resolve or fail on non-web credentials.

## What This Changes

The PR routes `infer web search` and `infer web fetch` through command secret resolution, adds a capability-web target helper, focused tests, a coverage allowlist entry, and a repro script.

## Best Possible Solution

Keep command-time secret resolution for web search/fetch, but derive exact web-only target ids and preserve selected-provider inactive semantics.

## Reproduction Assessment

Yes. Source inspection shows current main reads raw config for CLI web search/fetch, while the documented web SecretRef contract requires selected-provider resolution; the remaining PR blocker is also source-reproducible from the broad plugin target prefix.

## Solution Assessment

No. Resolving before invoking the web runtime is the right direction, but the target set must be exact web credential ids rather than all plugin config SecretInputs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Limit web targets to web credential ids:** `src/cli/command-secret-targets.ts:65`
  - body: The new capability target filter accepts every registry id under `plugins.entries.`. The registry also creates targets from bundled plugin `configContracts.secretInputs`, such as voice-call Twilio/realtime/tts keys, so `infer web search` can still resolve or fail on unrelated plugin credentials before the selected web provider runs. Keep this helper to `tools.web.search`, `tools.web.fetch`, and exact `webSearch.apiKey`/`webFetch.apiKey` plugin ids, with a regression that includes a non-web plugin secret.
  - confidence: 0.9
