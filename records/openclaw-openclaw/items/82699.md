## Summary

Keep open: the underlying CLI SecretRef bug is valid and the fix direction is right, but the latest PR head still has a blocking web-target scoping regression and needs real CLI-path proof before merge.

## What This Changes

The PR routes `infer web search` and `infer web fetch` through command secret resolution, adds a capability-web target helper, focused tests, and a repro script.

## Best Possible Solution

Keep command-time secret resolution for web search/fetch, but limit target ids to `tools.web.search`/`tools.web.fetch` and exact plugin `webSearch.apiKey`/`webFetch.apiKey` surfaces, with a non-web plugin SecretRef regression and real CLI proof.

## Reproduction Assessment

Yes. Source inspection on current main shows the CLI web search/fetch path reads raw runtime config, while the docs require selected web provider SecretRefs to resolve and inactive provider refs to remain non-fatal.

## Solution Assessment

No. Resolving before invoking the web runtime is the right direction, but the proposed helper still admits all `plugins.entries.*` registry targets instead of web-only credentials.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Limit web command targets to web credentials:** `src/cli/command-secret-targets.ts:65`
  - body: The new helper accepts every registry id below `plugins.entries.` after the exact web checks. Because the registry also includes non-web plugin `configContracts.secretInputs` such as voice-call Twilio/realtime/TTS keys, `infer web search` and `infer web fetch` can now resolve or fail on unrelated plugin secrets before the selected web provider runs. Drop the broad plugin prefix and add a regression with a non-web plugin config SecretRef.
  - confidence: 0.9
