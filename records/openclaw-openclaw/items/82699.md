## Summary

Keep open: this targets a valid CLI SecretRef bug in the right area, but the latest patch still lets web commands resolve every plugin config secret via a broad prefix, and the supplied proof does not exercise the final CLI path.

## What This Changes

The PR routes `infer web search` and `infer web fetch` through command-time secret resolution, adds a capability-web target helper, focused tests, a coverage allowlist entry, and a repro script.

## Best Possible Solution

Resolve only exact web search/fetch credential targets before local web commands, preserve inactive provider semantics, and add after-fix proof from the actual CLI command.

## Reproduction Assessment

Yes. Source inspection shows current main reads raw config for CLI web search/fetch while the documented web SecretRef contract requires selected-provider resolution; the remaining PR blocker is also source-reproducible from the broad plugin target prefix.

## Solution Assessment

No. Command-time secret resolution is the right direction, but the target helper must be narrowed to exact web credential ids instead of all plugin entries.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Limit web targets to web credential ids:** `src/cli/command-secret-targets.ts:65`
  - body: The new capability target filter still accepts every registry id under `plugins.entries.`. Registry entries also include non-web plugin SecretInputs such as voice-call Twilio/realtime/TTS keys, so `infer web search` can resolve or fail on unrelated plugin credentials before the selected web provider runs. Drop the broad prefix and add a regression with a non-web plugin secret.
  - confidence: 0.9
