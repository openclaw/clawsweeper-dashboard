## Summary

Keep open. The PR is aimed at the right CLI SecretRef bug, and the author did respond to the first review, but the latest head still lets web commands match every `plugins.entries.*` secret target and the supplied proof does not exercise the actual final CLI path.

## What This Changes

The PR routes CLI web search and web fetch through command secret resolution and adds a capability-web target helper, focused tests, and a repro script.

## Best Possible Solution

Keep command-time secret resolution for web search and fetch, but derive exact web-only target ids and prove the real CLI path with selected-provider inactive semantics preserved.

## Reproduction Assessment

Yes. Current main source passes raw config into local web search/fetch, and the linked issue provides a concrete CLI command and unresolved plugin `webSearch.apiKey` error; the remaining PR blocker is also source-reproducible from the broad plugin target prefix.

## Solution Assessment

No. Resolving config before invoking web runtimes is the right direction, but this implementation still over-selects plugin secrets and needs real proof from the final CLI path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Limit web targets to web credential ids:** `src/cli/command-secret-targets.ts:65`
  - body: `isCapabilityWebSecretTargetId` still accepts every registry id under `plugins.entries.`. The registry includes non-web plugin `configContracts.secretInputs`, such as voice-call Twilio/realtime/TTS keys, so `infer web search` can resolve or fail on unrelated plugin secrets before the selected web provider runs. Drop the broad plugin prefix and add a regression with a non-web plugin SecretInput target.
  - confidence: 0.9
