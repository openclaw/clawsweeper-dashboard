## Summary

Keep open: the reported CLI SecretRef bug is real and the fix direction is right, but the current PR head still lets capability web commands match every `plugins.entries.*` secret target, so unrelated plugin credentials can be materialized or made fatal. The supplied proof is also insufficient because it runs a helper script, not the actual CLI command or the final scoped target helper.

## What This Changes

The PR routes CLI web search/fetch through command-time SecretRef resolution, adds a capability-web target helper with focused tests, and adds a repro script.

## Best Possible Solution

Narrow capability web target selection to exact `tools.web.*` and plugin `webSearch.apiKey`/`webFetch.apiKey` credential ids, add a non-web plugin secret regression, and provide redacted proof from the actual CLI command.

## Reproduction Assessment

Yes. Source inspection shows current main reads raw config for CLI web search/fetch while the documented SecretRef contract requires scoped provider resolution, and the remaining PR blocker is visible in the current diff.

## Solution Assessment

No. Command-time secret resolution is the right seam, but the helper must not match the broad `plugins.entries.` prefix and the proof should exercise the actual CLI path.
