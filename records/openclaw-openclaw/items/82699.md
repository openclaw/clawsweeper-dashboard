## Summary

Keep open: this PR targets a real CLI SecretRef bug, but the final helper still accepts every plugin config secret under `plugins.entries.`, so web search/fetch can resolve or fail on unrelated plugin credentials. The supplied proof is also helper-level output rather than the final CLI web command path.

## What This Changes

The PR routes CLI web search/fetch through command-time SecretRef resolution, adds a capability-web target helper, focused tests, and a repro script.

## Best Possible Solution

Land one canonical fix that resolves CLI web search/fetch SecretRefs through command-time resolution using exact web credential targets only, with a non-web plugin secret regression and real CLI-path proof.

## Reproduction Assessment

Yes. Current main source shows CLI web search/fetch reading raw config, and the linked issue provides a concrete failing command and unresolved SecretRef error; I did not execute the live command in this read-only review.

## Solution Assessment

No. Resolving before web runtime execution is the right direction, but the helper must drop the broad `plugins.entries.` prefix and include only exact `webSearch.apiKey`/`webFetch.apiKey` plugin targets plus web legacy targets.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Limit web command targets to web credential ids:** `src/cli/command-secret-targets.ts:65`
  - body: The new helper still accepts every registry id under `plugins.entries.`. Because the registry also exposes non-web plugin `configContracts.secretInputs` such as voice-call Twilio and TTS keys, `infer web search` can resolve or fail on unrelated plugin credentials before the selected web provider runs. Keep this helper to web search/fetch credential ids and add a non-web plugin secret regression.
  - confidence: 0.9
