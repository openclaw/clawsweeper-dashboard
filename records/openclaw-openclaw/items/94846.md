- `impact:message-loss`: The reported final assistant output was not delivered even though a delivery target resolved successfully.

## Summary

Keep open: current main and v2026.6.9 still have the source-level path where recovered isolated cron tool warnings can remain fatal and return before normal delivery dispatch; three open closing PRs already target the issue, so it should stay open until one lands or is superseded.

## What This Changes

The issue reports scheduled isolated agentTurn cron delivery being skipped before dispatch when a recovered early tool error is still classified as a fatal run outcome.

## Best Possible Solution

Review and land one consolidated cron resolver/finalizer fix that treats recovered non-terminal tool warnings as non-fatal when final assistant output exists, reaches normal delivery dispatch, and preserves true fatal run errors with explicit delivery-not-attempted diagnostics.

## Reproduction Assessment

Yes, source-level: current main still routes the reported payload/run-level-error shape to hasFatalErrorPayload and can return before dispatchCronDelivery. I did not live-run the scheduled Feishu repro in this read-only pass.

## Solution Assessment

Unclear which open PR is the best final implementation, but the fix belongs in the cron payload outcome and finalization boundary. A new automated fix PR is not the best path while three open closing PRs already target the same issue.
