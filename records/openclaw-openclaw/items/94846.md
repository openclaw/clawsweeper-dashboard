## Summary

Keep open: current main still contains the cron payload-outcome and finalization gates that can classify recovered early tool errors as fatal and return before dispatch. Several open PRs already target this issue, so maintainers should choose or consolidate the fix rather than queue another repair branch.

## What This Changes

The issue asks for scheduled isolated cron agent turns that recover from an early tool error and produce final output to still reach delivery dispatch instead of being recorded as fatal and not delivered.

## Best Possible Solution

Review and land one narrow cron payload-outcome/finalization fix that treats recovered isolated runs with final output as non-fatal for delivery while preserving true fatal structured-error suppression.

## Reproduction Assessment

Yes, at source level: current main still gates recovered tool-warning recovery on channel final-text preference and still returns before dispatch when that warning is classified as fatal. I did not run the full scheduled WSL2/Feishu scenario in this read-only review.

## Solution Assessment

Yes, the best fix belongs in the existing cron payload-outcome/finalization boundary, not in a new config flag or separate channel workaround. The remaining decision is which overlapping linked PR should be adopted or consolidated.
