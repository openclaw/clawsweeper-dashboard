## Summary

Keep open: current main and v2026.6.9 still have the classifier and finalizer path that can treat recovered tool-warning output as fatal before delivery dispatch, and three open PRs already use closing syntax for this issue.

## What This Changes

Scheduled isolated cron agentTurn runs that recover from an early tool error and produce final output should still attempt configured delivery instead of recording a fatal not-delivered result.

## Best Possible Solution

Land one narrow cron payload-outcome fix that treats recovered tool warnings with final output as non-fatal for delivery while preserving true fatal structured-error suppression and real run-level failures.

## Reproduction Assessment

Yes at source level: current main still gates recovered cron tool-warning recovery on channel final-text preference and returns before dispatch when the warning remains fatal. I did not run the live WSL2/Feishu scheduled setup in this read-only review.

## Solution Assessment

Yes: the narrow maintainable fix is to correct cron payload outcome classification so recovered tool warnings with final output do not suppress dispatch. A new config flag or broad cron status redesign is larger than this specific bug requires.
