## Summary

Keep open: current main has a z.ai/GLM-specific Pi auto-compaction mitigation, but it still does not disable Pi auto-compaction merely because the effective compaction mode is `safeguard`, and open PR #73839 still targets this issue with closing syntax.

## What This Changes

Disable Pi SDK auto-compaction when OpenClaw safeguard compaction is the effective mode so the two compaction lifecycles do not block each other.

## Best Possible Solution

Land #73839 or an equivalent patch that threads the effective compaction mode into the existing Pi auto-compaction guard across embedded, CLI, and reload paths while preserving the z.ai-specific mitigation and adding regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure `agents.defaults.compaction.mode = "safeguard"` with the legacy context engine and a non-z.ai provider, and current main still leaves Pi auto-compaction enabled because the guard never sees the mode.

## Solution Assessment

Yes. Extending the existing guard to include the effective safeguard mode is the narrow maintainable fix; retrying around `already_compacted_recently` would leave two compaction owners competing.
