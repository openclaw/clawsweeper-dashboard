## Summary

Keep open: current main has a z.ai/GLM-specific mitigation, but it still does not disable Pi auto-compaction simply because the effective compaction mode is `safeguard`, and open PR #73839 remains the active unmerged fix for this issue.

## What This Changes

The issue asks OpenClaw to disable or coordinate Pi SDK auto-compaction when `agents.defaults.compaction.mode` resolves to `safeguard` so OpenClaw safeguard compaction is not blocked by recent Pi compaction entries.

## Best Possible Solution

Land #73839 or an equivalent patch that passes the effective safeguard mode into `applyPiAutoCompactionGuard` across embedded, CLI, and reload paths while preserving the #76056 z.ai-specific guard and adding regression coverage.

## Reproduction Assessment

Yes. Source inspection reproduces the remaining path: configure `agents.defaults.compaction.mode = "safeguard"` with the legacy context engine and a non-z.ai provider, and current main leaves Pi auto-compaction enabled because the guard never receives the compaction mode.

## Solution Assessment

Yes. Extending the existing Pi auto-compaction guard to treat effective `safeguard` mode as OpenClaw-owned compaction is the narrow maintainable fix; retrying around `already_compacted_recently` would keep two compaction owners competing.
