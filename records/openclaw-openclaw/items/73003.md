## Summary

Keep open: current main has a z.ai-specific Pi auto-compaction guard, but it still does not disable Pi auto-compaction based on explicit safeguard mode, and open PR #73839 references this issue with closing syntax.

## What This Changes

Request disabling or coordinating Pi SDK auto-compaction when OpenClaw safeguard compaction owns the session compaction lifecycle.

## Best Possible Solution

Land #73839 or an equivalent mode-aware guard that disables Pi auto-compaction whenever the effective compaction mode is safeguard across embedded, CLI, and post-reload paths, with focused regression tests.

## Reproduction Assessment

Yes. A source-level reproduction is `agents.defaults.compaction.mode = "safeguard"` with the legacy context engine and a non-z.ai provider: safeguard loads, but the Pi auto-compaction guard sees no `ownsCompaction` flag or mode input and can leave Pi auto-compaction enabled.

## Solution Assessment

Yes. Extending the existing guard to use the effective compaction mode is the narrow maintainable fix; retrying after `already_compacted_recently` would leave two compaction owners competing.
