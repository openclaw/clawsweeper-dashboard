## Summary

Keep open: current main still only disables Pi auto-compaction when a context engine declares `ownsCompaction`, while safeguard mode is enabled through a separate compaction-mode path. The issue is also paired with open PR #73839 using closing syntax, so it should stay open until that PR or an equivalent patch lands.

## What This Changes

The issue requests disabling Pi SDK auto-compaction when OpenClaw safeguard compaction owns the compaction lifecycle, to prevent repeated `already_compacted_recently` failures.

## Best Possible Solution

The desired end state is one compaction-ownership policy in `pi-settings` that treats safeguard mode and provider-forced safeguard as OpenClaw-owned compaction, applies that policy in both embedded and CLI paths, and has focused tests for default, safeguard, provider, and context-engine-owned cases.

## Reproduction Assessment

Yes. Configure `agents.defaults.compaction.mode = "safeguard"` with the default legacy context engine: current main installs the safeguard extension, but the Pi auto-compaction guard still returns false because `ownsCompaction` is unset.

## Solution Assessment

Yes. Threading the effective compaction mode into the existing Pi auto-compaction guard is the narrow maintainable fix; channel-specific workarounds or retry loops would leave the two compaction owners competing.
