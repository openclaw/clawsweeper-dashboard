## Summary

Keep open: current main still has the reported safeguard/Pi auto-compaction mismatch, and open PR #73839 is already the linked implementation candidate with closing syntax. The issue should stay open until that PR or an equivalent patch lands.

## What This Changes

The issue asks OpenClaw to disable or coordinate Pi SDK auto-compaction when `compaction.mode = "safeguard"` so OpenClaw-owned safeguard compaction is not blocked by Pi's recent-compaction state.

## Best Possible Solution

Land #73839 or an equivalent narrow patch that centralizes the policy in `pi-settings`, passes the effective compaction mode through embedded and CLI paths, disables Pi auto-compaction for safeguard/provider-owned compaction, and adds focused regression coverage for default, safeguard, and context-engine-owned cases.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: configure `agents.defaults.compaction.mode = "safeguard"` without an owning context engine, then the current guard returns false because it only sees `ownsCompaction`, while the safeguard extension is enabled independently and Pi threshold auto-compaction remains enabled.

## Solution Assessment

Yes. Threading the effective compaction mode into the existing Pi auto-compaction guard is the narrow maintainable fix; channel-specific workarounds or retry logic would leave the underlying competing compaction ownership unresolved.
