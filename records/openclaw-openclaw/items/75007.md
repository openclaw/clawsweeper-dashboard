## Summary

Keep open: current main still stages the backup manifest under ambient os.tmpdir(), and the macOS service TMPDIR path can sit inside the state directory that backup recursively archives; the focused repair PR #75031 is closed unmerged.

## What This Changes

The issue asks for backup creation to stage its manifest outside ~/.openclaw/tmp so macOS LaunchAgent backups contain only one root manifest.json.

## Best Possible Solution

Backup creation should stage manifest.json in an OpenClaw-managed temp root outside every selected source, preserve the exactly-one-root-manifest verifier invariant, and add a regression for TMPDIR inside state.

## Reproduction Assessment

Yes. The high-confidence reproduction is source-level: run a normal state/full backup with TMPDIR=<stateDir>/tmp, where the staged manifest sits under the recursive state source and is also passed explicitly to tar.

## Solution Assessment

Yes. Reusing resolvePreferredOpenClawTmpDir() with a source-containment guard is the narrow maintainable fix; service TMPDIR behavior and backup verification should stay strict.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/75007. Current `createBackupArchive()` stages `manifest.json` with `fs.mkdtemp(path.join(os.tmpdir(), "openclaw-backup-"))`; when macOS LaunchAgent sets `TMPDIR=<stateDir>/tmp`, the manifest is under the recursively archived state source and can appear twice. Use `resolvePreferredOpenClawTmpDir()` or an equivalent containment-checked OpenClaw temp root for backup staging, remove the unused `node:os` import if applicable, and ensure the final staging directory is outside every selected backup source. Add focused regression coverage that sets `TMPDIR` inside the test state dir, creates a real backup, and asserts exactly one root `manifest.json` entry while `backupVerifyCommand()` succeeds. Do not loosen backup verification, change macOS service TMPDIR behavior, or add generic backup exclude behavior here. Add a user-facing CHANGELOG.md fix entry.
