## Summary

Keep open. Current main still uses ambient `os.tmpdir()` for backup manifest staging while normal backup plans include the state directory and macOS service `TMPDIR` resolves under that state directory, so the duplicate-manifest path remains; related PR #75031 was closed unmerged.

## What This Changes

The report asks for `openclaw backup create` to stage its manifest outside `~/.openclaw` when LaunchAgent `TMPDIR` points at `~/.openclaw/tmp`.

## Best Possible Solution

Backup creation should stage transient files in a trusted temp root outside every backup source, preserve the one-root-manifest verifier invariant, and add regression coverage for `TMPDIR` under state.

## Reproduction Assessment

Yes. The source trace is high-confidence: set `TMPDIR` to `<stateDir>/tmp`, let backup creation write the staged manifest there, and the tar input can include it both explicitly and through the recursive state source.

## Solution Assessment

Yes. Reusing `resolvePreferredOpenClawTmpDir()` or an equivalent containment-checked OpenClaw temp root is the narrowest maintainable fix; changing the macOS service `TMPDIR` policy or relaxing verification would be broader and would not address backup self-inclusion.

## Repair Work Prompt

Fix openclaw/openclaw#75007. `openclaw backup create` currently stages `manifest.json` with `fs.mkdtemp(path.join(os.tmpdir(), "openclaw-backup-"))`; when the macOS LaunchAgent sets `TMPDIR` to `<stateDir>/tmp`, that staging directory is inside the state source and tar can archive the manifest twice. In `src/infra/backup-create.ts`, stage backup transient files in `resolvePreferredOpenClawTmpDir()` or an equivalent trusted OpenClaw temp root outside backup sources, and remove the unused `os` import if applicable. Add a regression test in `src/infra/backup-create.test.ts` that sets `TMPDIR` to a temp path under the test state dir, creates a backup, lists archive entries, and asserts exactly one root `manifest.json`; make sure the env is restored. Keep the verifier invariant; do not change the macOS service TMPDIR policy or relax duplicate-entry verification. Add a user-facing `CHANGELOG.md` fix entry. Use PR #75031 only as source context; it is closed unmerged.
