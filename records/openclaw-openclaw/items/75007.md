## Summary

Keep open. Current main still stages the backup manifest under ambient `os.tmpdir()`, while macOS service startup can set `TMPDIR` inside the archived state directory; the focused repair PR #75031 is closed unmerged.

## What This Changes

The issue asks for `openclaw backup create` to avoid packaging duplicate root `manifest.json` entries when `TMPDIR` points inside `~/.openclaw/tmp`.

## Best Possible Solution

Backup creation should stage `manifest.json` in a temp root proven outside all selected backup sources while preserving the verifier’s exactly-one-root-manifest invariant.

## Reproduction Assessment

Yes. The high-confidence path is source-level: run a normal state/full backup with `TMPDIR=<stateDir>/tmp`, where the staged manifest is under the recursive state source and is also passed explicitly to tar.

## Solution Assessment

Yes. Reusing `resolvePreferredOpenClawTmpDir()` with an added containment check is the narrow maintainable fix; service `TMPDIR` behavior and backup verification should stay strict.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/75007. Current `createBackupArchive()` stages `manifest.json` with `fs.mkdtemp(path.join(os.tmpdir(), "openclaw-backup-"))`; when macOS LaunchAgent sets `TMPDIR=<stateDir>/tmp`, the manifest is under the recursively archived state source and can appear twice. Use `resolvePreferredOpenClawTmpDir()` or an equivalent containment-checked OpenClaw temp root for backup staging, remove the unused `node:os` default import if applicable, and ensure the final staging directory is outside every selected backup source. Add focused regression coverage that sets `TMPDIR` inside the test state dir, creates a real backup, and asserts exactly one root `manifest.json` entry while `backupVerifyCommand()` succeeds. Do not loosen backup verification, change macOS service `TMPDIR` behavior, or add generic backup exclude behavior. Add a user-facing `CHANGELOG.md` fix entry.
