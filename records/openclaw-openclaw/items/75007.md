## Summary

Keep open. Current main still stages the backup manifest under ambient `os.tmpdir()` while macOS service startup can set `TMPDIR` inside the OpenClaw state directory, so the tar input can include the explicit root manifest and then walk the staging manifest again through the recursive state payload. The focused fix PR #75031 was closed without merging, so the bug remains actionable.

## What This Changes

Ensure `openclaw backup create` does not package its temporary manifest staging file when `TMPDIR` points inside `~/.openclaw/tmp`.

## Best Possible Solution

Backup creation should stage internal scratch files under an OpenClaw-managed temp root outside the planned backup sources, with regression coverage for service-style `TMPDIR=<state>/tmp` so the archive never captures its own staging directory.

## Reproduction Assessment

Yes. The source-level path is high confidence: Darwin service env sets TMPDIR under state, backup staging uses `os.tmpdir()`, and tar includes both the explicit manifest path and the recursive state source.

## Solution Assessment

Yes. Reusing the existing preferred OpenClaw temp resolver, plus a focused containment regression test, is the narrowest maintainable fix; changing the service TMPDIR would be broader and affect unrelated SQLite temp behavior.
