## Summary

Keep open. Current main still has the reported backup staging path, and the provided timeline shows PR #75031 is the active unmerged closing PR for this issue, so closing before that PR or an equivalent fix lands would be premature.

## What This Changes

The issue requests that `openclaw backup create` stop placing its temporary manifest staging directory under service `TMPDIR` when that path is inside the OpenClaw state directory.

## Best Possible Solution

Land PR #75031 or an equivalent focused change so backup manifest staging always happens outside every included backup source, with a regression test proving the service `TMPDIR`-inside-state scenario does not add a staged `manifest.json` under the payload.

## Reproduction Assessment

Yes. Current-main source inspection gives a high-confidence reproduction path: macOS service `TMPDIR` points under state, backup staging uses `os.tmpdir()`, and tar receives both the explicit manifest and the recursive state source. A live archive was not created because this review is read-only, but the path-level behavior is direct.

## Solution Assessment

Yes in direction. Replacing ambient `os.tmpdir()` with the existing preferred OpenClaw temp resolver, or otherwise checking that the staging root is outside all included assets, is the narrow maintainable fix; the linked PR is the active candidate to review for that outcome.
