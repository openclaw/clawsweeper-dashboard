## Summary

Close as implemented. Current main and the v2026.5.22 release contain the SSE stream behavior needed for this report: official OpenAI SSE streams are no longer sanitized, and the remaining sanitizer keeps reading split frames until it can emit a parser-visible event.

## What This Changes

The issue asks OpenClaw to preserve valid OpenAI Responses SSE frames so embedded OpenAI runs do not hang until timeout.

## Best Possible Solution

Keep the shipped provider transport behavior and close this report; any remaining Azure memory-tool stall should stay in its separate open issue.

## Reproduction Assessment

Yes for the historical bug: the issue and fix diff isolate the ReadableStream pull path that returned without emitting when an SSE frame was split. I did not run a live OpenAI repro during this read-only review, but current source and regression coverage make the old failure mode clear.

## Solution Assessment

Yes. The current implementation is narrower than a runtime bypass: official OpenAI streams are left unmodified, and sanitized streams keep reading until a parser-visible event or close.
