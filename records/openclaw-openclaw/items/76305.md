Fix OpenAI Responses streaming so the SSE sanitizer preserves valid frames without stalling embedded OpenAI runs.

## Repair Work Prompt

Fix the OpenAI Responses SSE sanitizer stall reported in https://github.com/openclaw/openclaw/issues/76305. Current `sanitizeOpenAISdkSseResponse()` in `src/agents/provider-transport-fetch.ts` can return from `pull()` without enqueueing when an upstream chunk ends before the first SSE boundary, leaving the OpenAI SDK reader pending. Update the sanitizer to keep reading until it can enqueue a complete readable SSE block or close/error, or replace it with an equivalent `TransformStream`-style implementation. Preserve the existing behavior that drops event-only and blank-data malformed frames before the OpenAI SDK parser sees them. Add regression coverage in `src/agents/provider-transport-fetch.test.ts` for a valid OpenAI Responses SSE frame split across chunks before `\n\n`, proving `Stream.fromSSEResponse()` yields the event instead of hanging. Do not add a permanent product bypass env var as the fix. Include a `CHANGELOG.md` fix entry if the change is user-facing under repo policy. Related context: #76174 and #75824 show broader OpenAI embedded timeout symptoms.
