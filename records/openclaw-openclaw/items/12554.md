## Summary

Keep this issue open: current main now has session-scoped process tracking and internal scope cancellation, but still lacks an authorized Gateway/RPC/CLI surface for operators to inspect and terminate the exec processes associated with a stored session. The related shutdown report concerns gateway cleanup rather than this missing operator capability.

## What This Changes

Add session-scoped process inspection and termination controls for active exec children.

## Best Possible Solution

Add an explicit Gateway protocol and CLI operation that resolves the canonical session scope, lists managed process records, cancels matching active runs through the supervisor, returns per-run outcomes, and documents authorization without changing default `sessions.abort` behavior.

## Reproduction Assessment

Yes, at the source level. Current main has session-scoped process records and internal scope cancellation, while the public session protocol still exposes only active-run abort and ordinary session maintenance.

## Solution Assessment

Not exactly as proposed: the supervisor-backed direction is appropriate, but an additive authorized process-control contract is safer than silently broadening `sessions.abort`; maintainers must settle its scope and lifecycle semantics first.
