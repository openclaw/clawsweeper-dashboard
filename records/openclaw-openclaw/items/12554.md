- `impact:session-state`: The issue is about active session-scoped exec work surviving after the session is stopped, aborted, or timed out.

## Summary

Keep open: current main still lacks a Gateway/RPC/CLI operator surface to list or terminate supervised exec children for an arbitrary session, and the existing abort and HTTP kill paths cover chat/subagent control rather than session-wide exec cleanup.

## What This Changes

The issue requests a `sessions_kill` or equivalent CLI/RPC surface that lists and terminates all running exec processes associated with a session.

## Best Possible Solution

Design an additive Gateway protocol and CLI operator contract that resolves the canonical session key, lists supervised exec/process records for that session scope, terminates them through supervisor cancellation with process-tree fallback where appropriate, returns per-run status, and documents authorization and isolation rules without changing default `sessions.abort` behavior.

## Reproduction Assessment

Yes for the feature gap at source level: current Gateway/CLI surfaces lack `sessions.kill` or exec-listing, and existing tests show background exec sessions survive a tool abort. I did not run a live macOS process-tree reproduction in this read-only review.

## Solution Assessment

Unclear for the exact API shape; the requested direction is valid, but the maintainable solution needs an additive, explicitly authorized session exec-control contract rather than overloading the current abort behavior silently.
