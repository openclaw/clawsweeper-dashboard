## Label Justifications

- `impact:session-state`: The issue concerns lifecycle ownership and cleanup of subprocesses associated with a session.

## Summary

The requested operator capability remains absent on pinned current main: `sessions.abort` controls active agent runs, while process-scope listing and cancellation remain separate lower-level primitives. This is a valid feature that needs maintainer product and security decisions before implementation.

## What This Changes

Add session-scoped controls to list and terminate active exec child processes, including process-tree cleanup.

## Best Possible Solution

Add an additive, operator-authorized Gateway API that resolves a canonical session to managed process scopes, lists active records, cancels them through the supervisor with process-tree fallback, and returns per-process outcomes; implement CLI commands as clients of that API without silently changing existing abort behavior.

## Reproduction Assessment

Yes at source level. Pinned current main exposes only run-abort parameters and no session process-list or process-kill method, while separate supervisor scope APIs show where the missing capability would need to connect.

## Solution Assessment

The requested outcome is appropriate, but the example CLI commands alone are not the best implementation boundary. A single additive Gateway contract with explicit authorization and adapter semantics should own the behavior, with CLI acting as a client.

## Risks / Open Questions

- A public session-to-process API could reveal or terminate processes outside the caller’s authorized scope unless canonical session ownership and operator permissions are enforced.
- Supervisor cancellation may not cover every child, PTY, sandbox, or detached process uniformly; platform-specific process-tree fallback and partial-failure reporting need an explicit contract.
- Changing existing `sessions.abort` to kill subprocesses by default could unexpectedly terminate work for current clients, so the behavior should remain additive unless maintainers intentionally approve a compatibility change.
