## Label Justifications

- `P2`: The issue is a real process-control and session-operator gap, but it is a bounded feature/design item rather than an immediate outage.
- `impact:session-state`: The requested behavior controls active work associated with a session and affects whether a stopped or deleted session still has running child activity.
- `impact:security`: A session-wide exec list/kill surface must enforce authorization and isolation so operators cannot see or terminate unrelated session processes.

## Summary

Keep open: current main and v2026.6.5 still have scoped process-supervisor primitives, but no operator Gateway/RPC/CLI surface that lists and terminates all exec children for an arbitrary session.

## What This Changes

Request an operator-facing sessions kill/list capability that terminates all running exec child processes spawned by a session.

## Best Possible Solution

Add an additive Gateway protocol and CLI surface that resolves the canonical session scope, lists active supervised exec runs, cancels that scope through supervisor-owned termination, returns per-run status, and documents authorization/isolation without changing default sessions.abort semantics.

## Reproduction Assessment

Yes, at source level: current main exposes scoped process primitives, but the session RPC schema, Gateway method list, abort handler, HTTP kill route, and CLI do not provide the requested session-wide exec list/kill path. I did not run a live macOS child-process reproduction.

## Solution Assessment

Unclear as an implementation because the requested direction belongs in core session/process control, but maintainers still need to choose the exact RPC/CLI contract, authorization scope, and termination semantics.
