## Summary

Keep open: current main and the latest release still do not expose a session-wide operator API or CLI command to list and terminate exec child processes by stored session. The lower-level supervisor and process-scope primitives exist, but the remaining work is a security-sensitive Gateway/RPC/CLI contract decision rather than completed implementation.

## What This Changes

Request an operator-facing sessions_kill or exec-list capability to terminate active exec child processes associated with a session.

## Best Possible Solution

Land a maintainer-approved additive Gateway protocol and CLI flow that resolves canonical session keys, lists scoped managed exec records, terminates matching active runs with safe fallback, returns per-run status, and documents authorization and isolation rules.

## Reproduction Assessment

Yes, at source level. Current main and v2026.6.5 have no sessions.kill or exec-list method, abort kill flag, or CLI equivalent, while existing controls are model-facing process tools, active chat aborts, or subagent-run controls.

## Solution Assessment

Yes for the direction, but only after maintainer design. Reusing supervisor scope primitives behind an additive operator contract is maintainable; silently changing sessions.abort or the HTTP subagent kill route would be riskier.
