## Label Justifications

- `P2`: The issue is a real session/process-control gap with bounded blast radius, but it requires design rather than an emergency runtime fix.
- `impact:session-state`: The requested behavior controls whether active work associated with a session can outlive an abort, timeout, or operator stop action.
- `impact:security`: A session-wide exec list/kill operator needs explicit authorization and isolation so callers cannot inspect or terminate unrelated session processes.

## Summary

Keep open: current main still has lower-level scoped process primitives and a subagent HTTP kill route, but it does not provide the requested Gateway/RPC/CLI operator surface to list and terminate exec children for an arbitrary stored session.

## What This Changes

The issue requests a session-level command or abort enhancement that lists and kills running exec child processes spawned by a session.

## Best Possible Solution

Add an additive Gateway protocol and CLI surface that resolves the canonical session scope, lists active supervised exec runs, cancels that scope through supervisor-owned termination, returns per-run status, and documents authorization/isolation rules without changing default sessions.abort semantics.

## Reproduction Assessment

Yes, at source level: current main exposes scoped process primitives, but the session RPC schema, Gateway method descriptors, abort handler, HTTP kill route, and CLI do not provide the requested session-wide exec list/kill path. I did not run a live macOS child-process reproduction.

## Solution Assessment

Unclear as an implementation shape: the requested capability belongs in core session/process control, but maintainers still need to choose the exact RPC/CLI contract, authorization scope, and termination semantics.
