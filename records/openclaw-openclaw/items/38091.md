## Summary

Keep open: current main still has the active-run reconnect gap. The Stop/keepalive work is present, but the Control UI reconnect hello path still clears in-flight chat state and the Gateway hello snapshot still has no active-run recovery metadata; open PR https://github.com/openclaw/openclaw/pull/41067 remains the focused implementation candidate.

## What This Changes

The issue reports that Control UI WebSocket reconnects can terminate or lose an active AI chat session instead of preserving in-flight state.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/41067 with a rebased narrow fix that exposes active chat run metadata in the Gateway hello snapshot and reconciles Control UI in-flight chat state across reconnects, while keeping outbound queue/ACK work in https://github.com/openclaw/openclaw/issues/45952 separate.

## Reproduction Assessment

Yes, source-level: seed `chatRunId`/`chatStream`, process a successful reconnect hello, and current `main` clears both while the hello snapshot has no active-run recovery field. I did not live-reproduce the WSL2 or macOS timing that produces `error=terminated`.

## Solution Assessment

Yes for keeping this issue open and routing through the existing focused PR rather than opening a duplicate repair lane. The maintainable fix is an additive Gateway snapshot contract plus deterministic Control UI recovery of the active run.
