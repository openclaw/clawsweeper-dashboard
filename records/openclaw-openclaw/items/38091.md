## Summary

Keep open: current main still has source-level behavior that marks a pre-disconnect chat run as interrupted/killed after reconnect, while the related recovery PR is open, conflicting, and not a merged implementation.

## What This Changes

The issue reports that OpenClaw UI WebSocket reconnects can terminate active AI sessions and lose in-flight session state.

## Best Possible Solution

Define the reconnect contract for active chat runs, then land or replace the active-run snapshot/UI reconciliation work so reconnect adopts still-running sessions instead of marking them killed.

## Reproduction Assessment

Yes at source level: current reconnect hello handling clears active local run state as killed/interrupted, and the adjacent test asserts that behavior. No live WSL2/macOS reproduction was attempted in this read-only review.

## Solution Assessment

Unclear for the open candidate PR because it is conflicting and not fully re-reviewed here. The maintainable solution appears to be a gateway-owned active-run snapshot plus UI reconciliation, not unconditional local run killing on reconnect.
