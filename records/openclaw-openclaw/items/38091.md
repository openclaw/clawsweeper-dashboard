## Label Justifications

- `P1`: Transient reconnects can make an active user conversation appear terminated and expose controls that may interrupt or replace the still-running turn.
- `impact:session-state`: The current reconnect path discards active run identity and changes the local session projection to killed.

## Summary

Keep this issue open: current main and v2026.6.6 still deliberately clear the Control UI’s active run on reconnect and mark the session killed, even though the gateway now exposes canonical in-flight run data that the UI does not consume. The linked recovery PR remains open, conflicting, and without positive real-behavior proof, so it is not a safe superseding close target.

## What This Changes

Prevent Control UI WebSocket reconnects from terminating or losing the state of active chat runs.

## Best Possible Solution

Make the gateway’s canonical in-flight snapshot the reconnect authority: restore matching run identity, buffered text, and busy controls, and mark a run interrupted only when the refreshed server state confirms it is no longer active.

## Reproduction Assessment

Yes at source level: current main contains a deterministic reconnect-hello path, backed by a focused test, that clears an active run and marks its session killed. This review did not repeat the intermittent WSL2 or macOS network disconnect in a live browser.

## Solution Assessment

No, unconditional local interruption is no longer the best fix because the gateway exposes authoritative in-flight run state. Reconnect should reconcile against that state before clearing the UI run.
