## Summary

Keep open. Current main still clears active Control UI/WebChat run state after a reconnect hello, and the Gateway hello snapshot has no active-run recovery metadata; open PR #41067 is a relevant implementation candidate but is not merged, while #45952 tracks a different outbound message queue/ACK gap.

## What This Changes

The issue requests Control UI/WebChat reconnect resilience so an in-flight AI chat run is not terminated or lost when the Gateway WebSocket reconnects.

## Best Possible Solution

Add an additive Gateway reconnect recovery contract, either activeChatRuns in the hello snapshot or a dedicated reconnect RPC, then have the Control UI preserve/reconcile chatRunId, busy state, and stream state until the run is known terminal, with regression coverage for WebSocket reconnect during an active chat.send.

## Reproduction Assessment

Yes. The issue includes concrete logs, and current main has a focused code-level reproduction: seed chatRunId/chatStream, reconnect/emit hello, and observe the current hello path clearing active state.

## Solution Assessment

Unclear for the exact open PR until its full diff is reviewed, but the direction is right: server-side active-run metadata must be surfaced through an additive protocol path and reconciled by the Control UI rather than clearing active state on hello.
