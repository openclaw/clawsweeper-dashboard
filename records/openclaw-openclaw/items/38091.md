## Summary

Keep this issue open. Current main still clears local in-flight chat state on reconnect hello instead of recovering an active run, the gateway hello snapshot still lacks an active-run recovery payload, and the related recovery PR is open and conflicting rather than a landed fix.

## What This Changes

The issue reports that OpenClaw UI WebSocket reconnects can terminate active webchat sessions and lose in-flight session state.

## Best Possible Solution

Define and land a reconnect-safe active chat run recovery contract, likely by exposing active abortable chat runs in the gateway hello/session state and making Control UI adopt that state instead of clearing it as interrupted.

## Reproduction Assessment

Yes, source-level. Current main's reconnect hello path explicitly clears any local active chat run as interrupted, and the regression test expects that behavior; I did not run a live WSL2/macOS repro.

## Solution Assessment

Unclear. The activeChatRuns direction in the related PR is plausible because Gateway already tracks active runs, but maintainers still need to confirm the durable recovery contract and check the no-reconnect `terminated` reports before treating it as the final fix.
