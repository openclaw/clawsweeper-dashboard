## Summary

Keep open: current main and the latest release still clear local Control UI chat-run state on reconnect hello and mark the session killed. ## What This Changes

The issue reports that Control UI WebSocket reconnects can terminate active webchat runs and lose session/message state.

## Best Possible Solution

Have maintainers choose or repair the recovery design so reconnect hello can preserve or reconcile active webchat run state from gateway-owned active-run data without losing queued sends or falsely killing sessions.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: current main clears `chatRunId` and marks the session killed on reconnect hello when local run or stream state exists, and the adjacent test asserts that behavior. I did not run a live WSL/macOS reconnect E2E in this read-only review.

## Solution Assessment

Unclear: the current reset was a reasonable gateway-restart fallback, but it is not the best solution for transient reconnects. The maintainable fix likely needs gateway-owned active-run reconciliation plus real Control UI proof, not another local-only UI state clear.
