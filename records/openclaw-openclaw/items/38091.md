## Label Justifications

- `P1`: A shipped Control UI reconnect path can lose active-run state during an ordinary user workflow and permit interruption or conflicting chat actions.
- `impact:session-state`: The reconnect handler declares the selected local run killed without first reconciling the gateway-owned active-run state.
- `impact:message-loss`: Clearing the in-flight stream can hide the pending assistant response and allow actions that interfere with its delivery.

## Summary

Keep this issue open: current main and the latest v2026.6.5 release still discard active Control UI run state after a reconnect, while the related recovery PR remains open and unmerged. The discussion may also contain a second embedded-run termination bug because one report observed `terminated` without a nearby disconnect.

## What This Changes

Prevent Control UI WebSocket reconnects from discarding active-run state and leaving in-flight conversations interrupted or unrecoverable.

## Best Possible Solution

Replace the unconditional reconnect reset with gateway-authoritative active-run reconciliation that preserves busy and abort semantics, and track no-disconnect embedded terminations separately if they still reproduce.

## Reproduction Assessment

Yes at source level: reconnecting while a local run or stream exists deterministically reconciles it as interrupted/killed and clears it on current main and v2026.6.5. A live reproduction is still needed to prove that this path causes every reported embedded `terminated` event.

## Solution Assessment

Yes for the confirmed reconnect-state defect: gateway-authoritative active-run recovery is narrower and safer than trying to eliminate all transient WebSocket reconnects. Independent terminations without matching disconnect evidence should not be forced into the same repair without further proof.
