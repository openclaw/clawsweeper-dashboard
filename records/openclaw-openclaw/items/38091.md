## Summary

Keep open: current main still only partially addresses this reconnect family. The merged keepalive/Stop work helps connection stability and abort availability, but Control UI still clears local in-flight chat state on reconnect and the Gateway hello snapshot still lacks active run metadata; https://github.com/openclaw/openclaw/pull/41067 remains the focused implementation candidate.

## What This Changes

The issue reports that OpenClaw Control UI WebSocket reconnects can terminate or lose an in-progress AI chat session, especially in local WebChat setups.

## Best Possible Solution

Land or replace the focused active-run reconnect recovery work so Gateway exposes in-flight chat run metadata in the hello snapshot and Control UI reconciles busy/run state after reconnect, while keeping outbound queue/ACK delivery work separate.

## Reproduction Assessment

Yes for a high-confidence source-level path: seed an in-flight Control UI chat run, process a reconnect hello snapshot, and current main clears local run state while the snapshot has no active run recovery field. I did not live-reproduce the WSL2 `terminated` timing.

## Solution Assessment

Yes for keeping this open and routing through the existing focused implementation candidate. The maintainable fix is an additive active-run recovery contract plus Control UI reconciliation; the current keepalive/Stop recovery is helpful but incomplete for this report.
