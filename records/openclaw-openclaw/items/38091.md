## Summary

Keep open: current main still resets Control UI chatRunId/chatStream on reconnect hello, and the Gateway hello snapshot schema/builder still does not expose active chat-run metadata; open PR #41067 remains the focused implementation candidate.

## What This Changes

The issue reports that Control UI/WebChat WebSocket reconnects terminate or lose in-flight AI chat sessions instead of preserving active run state.

## Best Possible Solution

Land or refresh #41067, or an equivalent narrow fix, so Gateway exposes active run metadata in hello-ok and Control UI preserves/reconciles in-flight chat state across reconnect while keeping #45952’s outbound queue/ACK work separate.

## Reproduction Assessment

Yes. A high-confidence source-level path is to seed chatRunId/chatStream, process a reconnect hello, and observe current main clearing both; the existing test still asserts that clearing behavior.

## Solution Assessment

Yes for keeping this issue open, but not for starting a duplicate repair branch. The narrow maintainable solution is an additive reconnect recovery contract plus Control UI reconciliation, and #41067 already targets that path.
