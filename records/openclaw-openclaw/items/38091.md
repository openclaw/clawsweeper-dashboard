## Summary

Keep open: current main still has a source-level reconnect path that clears in-flight Control UI chat run state, while the Gateway hello snapshot still lacks active-run recovery metadata. The existing open PR https://github.com/openclaw/openclaw/pull/41067 remains the focused implementation thread, and related issue https://github.com/openclaw/openclaw/issues/45952 covers a separate outbound send queue/ACK loss path.

## What This Changes

OpenClaw UI webchat should keep WebSocket sessions stable and preserve or recover in-flight AI chat state across reconnects instead of ending active sessions with `terminated`.

## Best Possible Solution

Land or replace the focused recovery work so Gateway hello snapshots expose active chat run metadata and Control UI deterministically reconciles in-flight run state across reconnects, while keeping outbound send queue/ACK work in https://github.com/openclaw/openclaw/issues/45952 separate.

## Reproduction Assessment

Yes, source-reproducible with high confidence: seed `chatRunId`/`chatStream`, process a reconnect hello, and current main clears both while the hello snapshot has no active-run recovery field. I did not live-reproduce the WSL2/browser timing that produced the original `terminated` logs.

## Solution Assessment

Yes for keeping this issue open and routing through the existing focused recovery PR rather than closing it or opening a duplicate repair. Current main's keepalive and Stop recovery is useful but incomplete for preserving active chat run state across reconnects.
