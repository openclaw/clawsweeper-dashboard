## Label Justifications

- `P1`: The report describes a regression that silently stopped Feishu direct-message replies for real users after upgrade.
- `impact:message-loss`: The symptom is successful model output being suppressed before a channel-visible reply is delivered.

## Summary

Current main resolves the reported Feishu DM reply-loss path by deriving direct/source visible-reply defaults from the selected model/runtime instead of the Feishu source channel alone; the remaining Codex tool-gated default is now explicit maintainer policy with a documented `messages.visibleReplies: "automatic"` opt-out.

## What This Changes

The issue requests restoring visible Feishu direct-message replies after v2026.5.16-beta.6 returned `queuedFinal=false, replies=0` despite successful model output.

## Best Possible Solution

Keep the current-main selected-model visible-reply behavior, use `messages.visibleReplies: "automatic"` for operators who want old-mode automatic Codex direct replies, and open a new issue only if current main still reproduces Feishu `replies=0` with a non-Codex selected model.

## Reproduction Assessment

Yes, at source level for the historical regression: the reported logs match the shared dispatch path where `message_tool_only` suppresses final replies, and current tests cover the selected-model precedence that fixes the non-Codex direct path. I did not run a live Feishu tenant repro in this read-only review.

## Solution Assessment
