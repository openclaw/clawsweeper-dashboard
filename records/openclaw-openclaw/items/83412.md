## Summary

Keep open: this is a fresh, concrete Feishu message-loss regression with two-host rollback evidence, and current main still contains the direct-chat message-tool-only suppression path that can produce `queuedFinal=false, replies=0` after successful assistant output. Related issues and the open command-only PR are useful context, but they do not fully resolve ordinary Feishu direct-message replies.

## What This Changes

The issue reports Feishu websocket direct messages generating assistant output but delivering no replies after upgrading from v2026.5.16-beta.3 to v2026.5.16-beta.6.

## Best Possible Solution

Keep this as the canonical Feishu beta.6 direct-message tracker, then decide whether Feishu direct chats need a compatibility fallback to automatic visible replies or clearer diagnostics/migration guidance around the current message-tool-only default.

## Reproduction Assessment

Yes at source level: current main has a direct-chat harness default path that resolves to `message_tool_only`, suppresses automatic final replies, and is covered by a test asserting `queuedFinal=false` with no `sendFinalReply`. I did not run a live Feishu websocket tenant reproduction.

## Solution Assessment

Unclear for an automated fix: restoring automatic Feishu DM replies would likely solve the user-visible regression, but it changes behavior around the documented Codex direct-chat default. Maintainers should choose the compatibility boundary before a fix PR changes defaults.
