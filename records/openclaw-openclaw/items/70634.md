## Summary

Keep open. Current main has static foreground/background queue priority, but agent-origin `sessions_send`/A2A work still enters the embedded runner as user-triggered foreground work, so the reported human-vs-agent loop starvation remains a distinct scheduling-policy gap.

## What This Changes

Request origin-aware command-lane scheduling so human channel messages can run ahead of agent-originated loop traffic in multi-agent conversations.

## Best Possible Solution

Keep this as the canonical human-vs-agent queue scheduling issue and design a tested origin-aware priority policy that separates human channel input from `sessions_send` loop traffic while preserving FIFO and fairness boundaries.

## Reproduction Assessment

Yes for a source-level path, but not for a live current-main scenario: `sessions_send` traffic carries inter-session provenance while still entering embedded execution as `trigger: "user"`, sharing foreground priority with human turns.

## Solution Assessment

No, current main is only a partial mitigation: it separates user/manual turns from maintenance triggers, not human-originated channel work from agent-originated `sessions_send` loop work.
