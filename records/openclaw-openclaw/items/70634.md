## Summary

Keep open. Current main and the latest release have generic foreground/background queue priority, but the specific human-vs-agent origin split remains missing for `sessions_send` loop traffic.

## What This Changes

The issue asks for human-originated messages to outrank agent-originated loop traffic in agent lane queues so operator messages are not indefinitely delayed.

## Best Possible Solution

Define a source-aware scheduler policy at queue admission or the channel/session boundary so true human ingress outranks agent-originated inter-session work, then prove it against `sessions_send` loops and existing foreground/background scheduling.

## Reproduction Assessment

No live high-confidence reproduction was established in this read-only review. The remaining scheduler gap is source-reproducible: current main can identify `sessions_send` as inter-session provenance but still schedules its embedded run as `trigger: "user"`.

## Solution Assessment

No, the current generic foreground/background priority is only a partial adjacent fix. The best fix needs explicit human-vs-agent origin policy instead of treating inter-session agent traffic as ordinary user-triggered foreground work.
