## Summary

Keep open. Current main and the latest release have useful foreground/background lane priority, but `sessions_send` agent-to-agent traffic still enters the agent run path as foreground user work, so the requested human-vs-agent origin split remains missing.

## What This Changes

The issue requests queue scheduling that lets human-originated channel messages overtake agent-originated loop traffic in an agent lane.

## Best Possible Solution

Keep this as a core scheduler-policy issue and design a tested origin-aware priority path that preserves foreground human turns while demoting or bounding agent-to-agent loop traffic.

## Reproduction Assessment

Yes for source reproduction: current main maps gateway agent runs, including `sessions_send`-started A2A runs, to `trigger: "user"`, and the queue priority code only sees foreground/normal/background. No live Discord multi-agent loop was run in this read-only review.

## Solution Assessment

No, the current implementation is not the complete fix; the merged priority work only separates manual/user turns from cron/maintenance/background work. The best fix needs an explicit origin-aware scheduler policy rather than relying on the existing trigger mapping.
