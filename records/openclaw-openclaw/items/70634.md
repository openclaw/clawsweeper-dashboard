## Label Justifications

- `P2`: The issue affects operator responsiveness in multi-agent channel setups but is bounded to queue/session scheduling behavior.
- `impact:message-loss`: Human-originated messages can be left unread behind continuing agent-originated loop traffic, which is effectively message delivery loss to the agent turn.

## Summary

Keep open. Current main and the latest release have generic foreground/background queue priority, but agent-originated `sessions_send` traffic is still marked as inter-session provenance while entering embedded execution as `trigger: "user"`, so it receives the same foreground scheduling as true human ingress.

## What This Changes

The issue requests source-aware queue routing so human-originated messages are not starved behind agent-to-agent loop traffic.

## Best Possible Solution

Define a source-aware scheduler policy that distinguishes true human/channel ingress from inter-session agent work, then prove human turns win over agent-loop traffic without breaking per-priority FIFO, A2A ping-pong limits, or background starvation handling.

## Reproduction Assessment

Yes, source-reproducible for the remaining scheduler gap: current main can identify `sessions_send` as inter-session provenance but still schedules it as `trigger: "user"`. I did not establish a live six-agent Discord reproduction in this read-only review.

## Solution Assessment

No, the current generic foreground/background priority is not the best complete solution for this issue because it cannot distinguish human ingress from agent-originated loop traffic. The best fix needs an explicit origin policy, likely at or before the queue admission boundary.
