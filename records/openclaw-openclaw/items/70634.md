## Summary

Keep open. Current main has generic foreground/background queue priority, but source review still shows agent-origin sessions_send/A2A traffic entering embedded runs as trigger=user, so it is not separated from human channel input.

## What This Changes

The issue asks OpenClaw to prevent human-originated messages from being starved behind agent-to-agent loop traffic in per-agent command queues.

## Best Possible Solution

Keep this as the canonical human-vs-agent queue scheduling issue and design a tested origin-aware priority policy that separates human channel input from sessions_send loop traffic without breaking FIFO and fairness boundaries.

## Reproduction Assessment

Yes for a source-level path, not for a live current-main scenario: sessions_send/A2A messages carry inter_session provenance but still run with trigger=user, so they share foreground priority with human turns.

## Solution Assessment

No, current main is only a partial mitigation: it separates manual/user turns from background maintenance, not human-originated channel work from agent-originated sessions_send loop work.
