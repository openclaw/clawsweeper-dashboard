## Label Justifications

- `P2`: The report describes a real multi-agent scheduling problem with limited blast radius and a policy-sensitive fix, not an emergency core outage.
- `impact:message-loss`: Human channel messages can be effectively buried behind agent-originated turns and fail to reach the agent in time.

## Summary

Keep open. Current main and v2026.5.18 now have generic foreground/background command-queue priority, but they still do not distinguish human channel ingress from `sessions_send`/`inter_session` agent-loop work because both enter the embedded runner as `trigger: "user"`; the KinthAI production comment also supports that this needs a source-aware queue policy rather than generic priority alone.

## What This Changes

The issue asks OpenClaw to prioritize human-originated messages ahead of agent-originated lane work during multi-agent loops.

## Best Possible Solution

Define a source-aware queue priority contract that ranks human/channel ingress above `inter_session`/`sessions_send` work on the same session or nested lane, preserves FIFO within each source class, and coordinates aging/back-pressure with the existing priority queue design.

## Reproduction Assessment

Yes, at source level. Current main and v2026.5.18 classify both human ingress and `sessions_send`/`inter_session` work as `trigger: "user"`, so agent-originated queued work can remain in the same foreground FIFO class ahead of later human messages.

## Solution Assessment

Unclear as a concrete patch. A source-aware priority contract is the right direction, but maintainers still need to decide the exact ranking, active-run behavior, queue aging, and loop back-pressure policy.
