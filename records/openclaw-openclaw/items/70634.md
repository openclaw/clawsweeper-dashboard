## Summary

Keep open. Current main has foreground/background queue priority, but `sessions_send`/A2A traffic still carries inter-session provenance while entering embedded execution as a `user` trigger, so it can share foreground priority with human turns instead of being classified as agent-origin loop work.

## What This Changes

The issue requests human-originated messages to take priority over agent-originated loop traffic in per-agent command queues so operator messages cannot be starved.

## Best Possible Solution

Keep this as the canonical human-vs-agent scheduling issue and design a generic origin-aware policy that keeps human/channel-origin turns foreground while demoting or separately classifying inter-session agent-origin loop work before it enters the target session lane.

## Reproduction Assessment

No live high-confidence reproduction was run in this read-only review. The source path is reproducible: `sessions_send` traffic is marked as inter-session provenance but still enters embedded execution as a `user` trigger, sharing foreground session priority with human turns.

## Solution Assessment

No, current main is only a partial mitigation. It separates user/manual turns from maintenance triggers, but not human-originated channel work from agent-originated `sessions_send` loop work; the better fix is origin-aware scheduling at the ingress/runtime boundary.
