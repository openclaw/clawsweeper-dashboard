## Summary

Keep open. Current main and v2026.6.8 include foreground/background queue priority, but the remaining user problem is still unresolved because `sessions_send` agent-to-agent runs carry inter-session provenance without a queue-level human-vs-agent origin priority split.

## What This Changes

The issue requests origin-aware queue scheduling so human channel messages reach an agent ahead of sustained agent-originated loop traffic.

## Best Possible Solution

Keep this as the canonical human-vs-agent queue scheduling issue and design a tested origin-aware priority policy that separates human channel input from `sessions_send` loop traffic without breaking per-priority FIFO ordering.

## Reproduction Assessment

Yes for source reproduction: current `main` carries only foreground/normal/background queue priority, while `sessions_send` target runs carry inter-session provenance that is not consumed by the queue priority mapping. I did not run a live Discord multi-agent loop in this read-only review.

## Solution Assessment

No, current `main` is only a partial mitigation: it separates user/manual turns from background maintenance, not human-originated channel work from agent-originated `sessions_send` loop work. The best fix needs an explicit origin-aware scheduler policy and focused regression proof.
