## Label Justifications

- `P2`: The issue is a normal-priority user-visible usage accounting bug with bounded blast radius to dashboard/session reporting.
- `impact:session-state`: The bug mis-associates archived transcript lineage with the visible session usage state shown by the dashboard and `sessions.usage` APIs.
- `maturity:stable`: The affected `gateway-runtime` Gateway RPC APIs and Events surface is scored M4 Stable, and the docs list the affected usage/session usage RPCs.

## Summary

Keep open: current main and v2026.6.10 include reset/deleted archives in the global usage aggregate, but the per-session detail, cache, time-series, and log paths still resolve the active transcript first and scan one file, so same-stem archives remain omitted until the linked closing PR lands.

## What This Changes

The issue requests archive-aware cost/usage accounting so `.jsonl.reset.*` and related reset/deleted transcript files are included in dashboard and per-session spend totals after `/new` or `/reset`.

## Best Possible Solution

Land or replace the linked backend lineage fix so per-session summary, cache, time-series, and logs aggregate active plus reset/deleted same-stem transcripts while preserving `.bak` and checkpoint exclusion, then track broader archive shapes separately if maintainers want them covered.

## Reproduction Assessment

Yes, at source level. Current main resolves and scans one active transcript for per-session usage detail, so an active `<id>.jsonl` plus same-stem `.jsonl.reset.*` archive reproduces the undercount path described in the issue.

## Solution Assessment

Yes for the current routing decision. The best fix is a backend session usage lineage implementation in the loader/cache path, not a UI-side correction or a new config option; the open closing PR is the current candidate but needs refresh/review before merge.
