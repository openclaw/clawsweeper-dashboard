## Summary

Keep open. Current main includes reset/deleted archives in the broad `usage.cost` scan, but the per-session `sessions.usage` path still resolves a session id to one transcript and returns the active `.jsonl` before archive siblings, so active-plus-reset session totals remain undercounted.

## What This Changes

The issue requests cost/usage aggregation to include archived `.jsonl.reset.*` transcripts created by `/new` or `/reset` alongside active `.jsonl` session files.

## Best Possible Solution

Centralize per-session transcript segment resolution so one session id contributes all usage-counted active, reset, and deleted transcript segments across summaries, timeseries, logs, exports, and dashboard totals while preserving existing exclusions for backup, checkpoint, and trajectory artifacts.

## Reproduction Assessment

Yes. The issue gives a concrete active-plus-reset file pair with expected and observed costs, and current source statically reproduces the gap because `loadSessionCostSummary({ sessionId })` resolves and scans only one transcript when the active file exists.

## Solution Assessment

Yes. A shared resolver/aggregation fix in the usage infrastructure is the narrow maintainable path; a UI-only adjustment would leave RPC totals, detail views, logs, and exports inconsistent.
