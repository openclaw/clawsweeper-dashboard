## Summary

Keep open: current `origin/main` still has the single-pass Claude CLI transcript probe and both resume paths still treat an immediate false result as `missing-transcript`. The paired fix at https://github.com/openclaw/openclaw/pull/81048 is open, mergeable, and uses closing syntax, so this issue should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue requests a bounded retry and negative diagnostics for the Claude CLI transcript-content probe so a just-created JSONL without an assistant record does not force a cold session restart.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/81048 or an equivalent bounded retry and diagnostic fix, then close this issue while keeping broader context-loss work in https://github.com/openclaw/openclaw/issues/77974 tracked separately.

## Reproduction Assessment

Yes, at source level: a current-main Claude project JSONL that exists but lacks an assistant record makes the helper return false, and both resume callers treat that as `missing-transcript`. I did not run a live Claude CLI flush-timing scenario in this read-only review.

## Solution Assessment

Yes, the paired PR’s one-shot retry gated on `fileExists` plus negative diagnostics is the narrow maintainable fix for this false-negative path. Keeping the issue open until that PR merges is the right tracking outcome.
