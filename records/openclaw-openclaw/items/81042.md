## Summary

Keep open: current main still has a one-pass Claude CLI transcript-content probe, and both resume paths treat a transient false result as missing-transcript invalidation. The live issue remains paired with the open closing PR https://github.com/openclaw/openclaw/pull/81048, which is unmerged and conflicting, so closing before a reviewed fix lands would be premature.

## What This Changes

Make Claude CLI session transcript verification tolerate transcript flush or creation latency and log negative probes so valid resume state is not cleared.

## Best Possible Solution

Land a reviewed fix via https://github.com/openclaw/openclaw/pull/81048 or a replacement that makes Claude CLI resume validation freshness-aware, preserves stale-binding protection, and covers both resume call sites with regression tests and redacted live proof.

## Reproduction Assessment

Yes at source level: a Claude project transcript that is missing or has no assistant record makes `claudeCliSessionTranscriptHasContent` return false, and both resume paths convert that false result into missing-transcript invalidation. I did not run a live Claude CLI timing race in this read-only pass.

## Solution Assessment

Yes, keeping this issue open while the linked closing PR is reviewed is the narrow maintainable path. The implementation should fix transcript validation and session-id capture without weakening the stale-session guard.
