Keep this issue open. The report is fresh and concrete, current main still runs the same Discord provider startup and gateway lifecycle path, and the shipped Discord 429 work plus later adjacent Discord/runtime-deps changes do not prove the Raspberry Pi arm64 post-startup CPU wedge is fixed.

## What This Changes

The issue reports that enabling the Discord channel on Raspberry Pi arm64 with OpenClaw 2026.4.26 wedges the gateway at about 100% CPU after Discord provider startup, while disabling Discord restores healthy gateway probes.

## Best Possible Solution

Validate current main or the next build on the reported Raspberry Pi arm64 setup with Discord enabled and disabled. If the wedge still reproduces, narrow it with voice/voiceStates disabled, startup diagnostics, gateway probes, logs, and CPU sampling, then fix the confirmed Discord plugin startup/lifecycle/REST/runtime-deps path with focused regression coverage.
