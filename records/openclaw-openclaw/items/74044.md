## Summary

Close as implemented on main. The current Discord startup path now leaves Discord voice and `GuildVoiceStates` off for text-only/default configs, which matches the root cause described in the related completed #73753 report and addresses the reported post-startup CPU wedge without disabling Discord entirely. The fix is present on current main but was not in the latest `v2026.4.29` release checked here.

## What This Changes

The issue reports that enabling the Discord channel on OpenClaw 2026.4.26 caused a Raspberry Pi arm64 Gateway to become unresponsive and spin near 100% CPU after Discord provider startup, while disabling Discord restored Gateway health.

## Best Possible Solution

Keep the current explicit-opt-in Discord voice behavior for text-only bots. If a new report reproduces high CPU with `channels.discord.voice.enabled=true`, handle that as a focused Discord voice-runtime issue with CPU samples and voice-state diagnostics rather than reopening the default Discord startup path.

## Reproduction Assessment

Yes for the original 2026.4.26 behavior: the report and #73753 give concrete setup, CPU, probe, and disabled-Discord/no-voice-state recovery evidence. Current-main verification is source/test-backed rather than live Raspberry Pi validation, and the default path that caused the CPU pressure no longer enables voice or `GuildVoiceStates`.

## Solution Assessment

Yes for the reported default/text-only Discord startup problem. Making Discord voice explicit opt-in is the narrowest maintainable fix because it preserves voice for configured users while removing default voice-state traffic and voice runtime load from ordinary Discord bots.
