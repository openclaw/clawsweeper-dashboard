## Summary

Keep this issue open. The report is fresh and concrete, and current main still has the same Discord startup surface active; adjacent Discord 429, gateway READY, and runtime-deps fixes reduce plausible failure modes but do not prove the reported Raspberry Pi arm64 post-startup 100% CPU Gateway wedge is fixed.

## What This Changes

The issue reports that enabling the Discord channel on Raspberry Pi arm64 with OpenClaw 2026.4.26 wedges the Gateway at about 100% CPU after provider startup, while disabling `channels.discord.enabled` restores normal Gateway status probes.

## Best Possible Solution

Validate current main or the next build on the reported Raspberry Pi arm64 setup with Discord enabled and disabled. If the wedge still reproduces, narrow with `channels.discord.voice.enabled=false`, `channels.discord.intents.voiceStates=false`, startup diagnostics, Gateway status probes, and CPU sampling, then fix the confirmed Discord plugin startup/lifecycle/REST/runtime-deps path with focused regression coverage.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the reporter's 2026.4.26 setup because the issue gives concrete platform, config, startup, probe, CPU, and disabled-Discord recovery steps; no for current main because that same live Raspberry Pi arm64 and Discord-token path was not exercised here.

## Solution Assessment

Is this the best way to solve the issue? Unclear because there is no proposed patch and the current fixes are adjacent rather than proven. The narrow maintainable path is live validation first, then a Discord-plugin or startup-runtime repair only after the hot path is identified.
