## Summary

Keep this issue open. The report is fresh, concrete, and still describes an unproven Discord runtime wedge on Raspberry Pi arm64; current main contains adjacent Discord rate-limit hardening, but the inspected code, docs, changelog, tests, and issue discussion do not prove that the reported post-startup 100% CPU gateway wedge is fixed.

## What This Changes

The issue reports that enabling the Discord channel on OpenClaw 2026.4.26 on Raspberry Pi arm64 wedges the gateway after startup until Discord is disabled.

## Best Possible Solution

Validate current main or the next build on the reported Raspberry Pi arm64 setup with a real Discord bot token. If the wedge remains, narrow the repair inside the Discord plugin startup, gateway lifecycle, REST cooldown, or voice-state path, then add focused regression coverage for the confirmed loop and document only verified workarounds.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the reported 2026.4.26 environment because the report gives platform, version, config, startup, probe, CPU, and disabled-Discord recovery steps; no for current main because the same live Raspberry Pi arm64 and Discord-token path was not exercised here.

## Solution Assessment

Is this the best way to solve the issue? Unclear. There is no proposed patch, and the current Discord rate-limit/lifecycle guards are plausible mitigations rather than proof; the safest solution path is live validation first, then a narrow Discord-plugin fix if the wedge still reproduces.
