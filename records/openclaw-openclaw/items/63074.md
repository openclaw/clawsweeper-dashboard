## Summary

Keep open. Current main still classifies non-`/` bind sources as `non_absolute` before checking blocked credential paths, while this PR targets a real security-audit classification gap for Windows home credential binds. The latest PR diff appears narrow and the earlier platform-awareness review was addressed by adding raw `HOME`/`USERPROFILE` roots, but this is security-sensitive sandbox policy and should get maintainer review rather than automated closure.

## What This Changes

The PR moves sandbox blocked-path classification ahead of the non-absolute fallback for Windows credential binds, adds Windows home-root coverage and regression tests, and includes several Windows CI/test portability adjustments around Mattermost policy exports and path snapshots.

## Best Possible Solution

Land or replace this with a narrow security-reviewed sandbox change that classifies Windows home credential bind sources as critical dangerous binds without accidentally enabling generic Windows drive-letter bind mounts; handle full Windows bind acceptance separately through the broader Windows bind PR.

## Reproduction Assessment

Yes. ## Solution Assessment

Yes for the stated classification bug: checking normalized blocked paths before the fallback and including raw Windows home env roots is the narrow maintainable fix. It should not be treated as the final solution for general Windows bind support, which needs config-schema and allowed-root policy follow-through.
