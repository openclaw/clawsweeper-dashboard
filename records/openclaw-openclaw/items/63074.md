## Summary

Keep open. Current main still classifies Windows-style home credential bind sources as `non_absolute` before dangerous-bind classification and does not include raw `USERPROFILE` roots in the blocked home-root cache; the PR is a focused security fix candidate, but it should go through maintainer/security review rather than cleanup closure.

## What This Changes

The PR reorders sandbox bind classification for Windows credential sources, adds raw Windows home environment roots to the blocked path set, adds regression coverage, and includes Windows portability fixes for Mattermost policy exports plus a few path-sensitive tests/scripts.

## Best Possible Solution

Land or replace this with a narrow security-reviewed sandbox fix that classifies Windows home credential bind sources as critical dangerous binds while leaving generic Windows bind support to the broader Windows bind work.

## Reproduction Assessment

Yes. The current-main source path shows the non-absolute return before blocked-path classification, and the provided PR test uses a concrete Windows credential bind string to exercise the expected critical classification. I did not run tests because this review is read-only.

## Solution Assessment

Yes for the narrow classification bug. Checking normalized blocked paths before the fallback and adding raw Windows home env roots is the maintainable fix; broader Windows bind acceptance should remain a separate policy/config-schema decision.
