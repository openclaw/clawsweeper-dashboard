## Summary

Keep open: the remaining Windows USERPROFILE credential-root hardening is still a valid source-level bug, but this stale branch should not merge as-is because current main now accepts drive-absolute Windows binds and Mattermost access-group policy that the PR would regress.

## What This Changes

The PR changes sandbox bind classification for Windows credential paths and adds related Windows-portability adjustments in tests, Mattermost policy exports, and package verification scripts.

## Best Possible Solution

Rebase onto current main, keep only the raw Windows USERPROFILE credential-root hardening with focused sandbox/audit tests, and preserve the current Windows bind and Mattermost auth contracts.

## Reproduction Assessment

Yes, source-level. With HOME or OPENCLAW_HOME pointing away from USERPROFILE, current blocked-root construction does not include raw USERPROFILE even though Windows credential binds can be drive-absolute; I did not run a live Windows gateway in this read-only pass.

## Solution Assessment

No. Adding raw USERPROFILE to blocked home roots is the right narrow direction, but the current branch should be rebased onto the merged Windows bind policy and current Mattermost auth code instead of merging the stale fallback/refactor.
