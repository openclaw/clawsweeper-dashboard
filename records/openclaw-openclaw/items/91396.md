## Summary

Keep open: current main and the latest release still do not propagate configured third-party embedding model limits into the OpenAI/OpenAI-compatible memory embedding path, and the linked attempted fix was closed unmerged.

## What This Changes

SiliconFlow/OpenAI-compatible memory embeddings should honor smaller configured embedding input limits so `openclaw memory index` does not send oversized chunks and fail with HTTP 400.

## Best Possible Solution

Resolve configured embedding input limits on the shared custom OpenAI-compatible memory provider route, preserve built-in OpenAI limits, avoid SiliconFlow hardcodes or SSRF bypasses, and add route-level regression coverage plus redacted live provider proof when available.

## Reproduction Assessment

Yes by source inspection for the root cause: current providers without `maxInputTokens` still fall back to 8192, and the reported custom OpenAI-compatible route does not expose configured model limits. A credentialed SiliconFlow call was not run here, so the live HTTP 400 itself was not reproduced.

## Solution Assessment

No, the current implementation is not the best solution because it only handles built-in OpenAI model limits. The safer fix is a generic configured-model limit resolver on the custom OpenAI-compatible memory provider route, not a transport bypass or SiliconFlow-specific hardcode.
