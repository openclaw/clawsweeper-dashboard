## Repair Work Prompt

Fix the SiliconFlow/OpenAI-compatible memory embedding limit bug for https://github.com/openclaw/openclaw/issues/91396. Current main resolves custom `models.providers.<id>.api: "openai-completions"` memory providers to the generic `openai-compatible` adapter, but `src/plugins/openai-compatible-embedding-provider.ts` does not expose a configured `maxInputTokens`, so `packages/memory-host-sdk/src/host/embedding-model-limits.ts` falls back to 8192 and can send chunks larger than BGE/SiliconFlow accepts. Add a narrow configured model limit lookup on the actual generic OpenAI-compatible provider route, preferring the configured model's `maxTokens` and falling back to `contextWindow` when the model id matches after provider-prefix normalization. Do not hardcode SiliconFlow or bypass SSRF. Add regression coverage through `createEmbeddingProvider` or the generic adapter showing a custom provider/model with `maxTokens: 512` produces provider.maxInputTokens and chunk splitting honors it; keep existing OpenAI-native limits intact. Changelog is release-owned; do not edit `CHANGELOG.md`.
