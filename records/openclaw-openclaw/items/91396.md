## Summary

Keep open: current main and the latest release still lack a configured input-limit path for third-party OpenAI-compatible embedding model metadata, and the linked fix PR was closed unmerged after review found it missed the generic route.

## What This Changes

Memory indexing should honor third-party OpenAI-compatible embedding model input limits so SiliconFlow BAAI/bge-large-zh-v1.5 does not receive oversized chunks.

## Best Possible Solution

Resolve configured embedding input limits on the shared OpenAI-compatible/custom-provider route, add route-level regression coverage, and close only after redacted live SiliconFlow memory-index proof or an equivalent provider-limit proof.

## Reproduction Assessment

Yes for the source-level path: custom OpenAI-compatible providers can still produce a provider object without maxInputTokens, so chunking falls back to 8192. A credentialed SiliconFlow run is still needed for end-to-end live reproduction.

## Solution Assessment

No current implementation fully solves it. The earlier PR was plausible but too narrow because the reported custom-provider route resolves through the generic OpenAI-compatible adapter, not only the OpenAI memory adapter.
