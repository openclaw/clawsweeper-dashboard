## Summary

Keep open: current main and v2026.6.6 still do not resolve configured third-party embedding input limits, and the linked fix PR was closed unmerged rather than becoming a canonical fix.

## What This Changes

The issue asks OpenClaw memory indexing to honor third-party OpenAI-compatible embedding model input limits so SiliconFlow BAAI/bge-large-zh-v1.5 does not receive oversized chunks and fail with HTTP 400.

## Best Possible Solution

Resolve configured embedding input limits on the actual memory provider route, preferably from configured model maxTokens/contextWindow for both OpenAI and generic OpenAI-compatible adapters, then add route-level regression coverage and redacted SiliconFlow memory-index proof.

## Reproduction Assessment

No live current-main reproduction was run because SiliconFlow credentials are not available here. Source inspection and the reporter's logs give a clear repro path: third-party models expose no maxInputTokens, so chunk enforcement falls back to 8192 before calling the configured provider.

## Solution Assessment

No, the repository is not fixed yet, and the closed linked PR was too narrow because it did not cover the custom OpenAI-compatible provider route. The best fix belongs in the shared configured-provider limit resolution path with route-level tests.
