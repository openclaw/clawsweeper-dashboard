## Summary

Keep open: current main and v2026.6.6 still do not expose configured model input limits on the generic OpenAI-compatible memory embedding path, and the only linked fix PR was closed unmerged after review found it missed that route.

## What This Changes

The issue reports SiliconFlow memory embedding HTTP 400s when configured as an OpenAI-compatible provider because OpenClaw can send chunks above the model's smaller embedding input limit.

## Best Possible Solution

Resolve the embedding input-limit contract on the shared OpenAI-compatible/custom-provider route, add route-level regression coverage, and capture redacted live SiliconFlow memory-index proof before closing.

## Reproduction Assessment

Yes for the source-level failure path: current main routes custom OpenAI-compatible providers through an adapter with no `maxInputTokens`, then chunking falls back to 8192. A live SiliconFlow 400 still needs credentials to reproduce end to end.

## Solution Assessment

No current fix is in place. Reading configured model limits is plausible, but the earlier closed PR was too narrow because it did not cover the shared custom OpenAI-compatible provider route.
