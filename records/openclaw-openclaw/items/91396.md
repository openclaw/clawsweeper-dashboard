## Summary

Keep open: current main and the latest release still lack a configured input-limit path for third-party OpenAI-compatible embedding model metadata, so SiliconFlow/BGE chunks can still fall back to the remote default limit.

## What This Changes

The issue requests a fix for SiliconFlow/OpenAI-compatible memory embeddings whose configured model input limit is lower than OpenClaw's default remote embedding chunk limit.

## Best Possible Solution

Resolve configured embedding input limits on the shared OpenAI-compatible/custom-provider route, add route-level regression coverage, and validate with redacted SiliconFlow or equivalent provider-limit proof.

## Reproduction Assessment

Yes for the source-level path: custom OpenAI-compatible providers can still produce a provider object without `maxInputTokens`, so chunking falls back to 8192. A credentialed SiliconFlow run is still needed for end-to-end live reproduction.

## Solution Assessment

No current implementation fully solves it. The best fix should resolve configured embedding input limits on the shared custom/OpenAI-compatible provider path rather than hardcoding one third-party model or only changing the OpenAI adapter.
