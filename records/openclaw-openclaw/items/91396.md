## Summary

Keep open: the reporter’s follow-up narrowed this to configured embedding input limits, and current main still leaves both the OpenAI memory adapter and the generic OpenAI-compatible route without a configured-model maxInputTokens fallback for this SiliconFlow setup.

## What This Changes

The issue asks OpenClaw memory indexing to honor SiliconFlow/OpenAI-compatible embedding model limits so oversized chunks are not sent upstream and rejected with HTTP 400.

## Best Possible Solution

Teach the shared memory embedding provider path to honor configured model input limits before chunking, then prove the SiliconFlow/OpenAI-compatible indexing path with focused tests and redacted live output.

## Reproduction Assessment

No live high-confidence reproduction was run because SiliconFlow credentials are needed. Source inspection does reproduce the remaining current-main gap: configured custom OpenAI-compatible models can still expose no maxInputTokens and fall back to 8192.

## Solution Assessment

No current solution has landed. The best fix is shared configured-limit resolution across the OpenAI and generic OpenAI-compatible memory paths, not the closed PR’s OpenAI-helper-only change.
