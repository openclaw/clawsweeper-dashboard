## Summary

Keep open: current main and the latest release still do not expose configured third-party model limits as `maxInputTokens` for the affected memory embedding paths, and the only linked fix PR was closed unmerged after review found it missed the custom-provider route.

## What This Changes

The issue reports SiliconFlow memory indexing failures when third-party OpenAI-compatible embedding model input limits are not honored before remote embedding requests.

## Best Possible Solution

Resolve configured embedding input limits on the shared OpenAI-compatible/custom-provider memory path, preserve native OpenAI limits, add regression coverage through `createEmbeddingProvider` or memory indexing, and collect redacted live SiliconFlow or equivalent provider proof before closing.

## Reproduction Assessment

Partly. The reporter provided concrete live steps and current source shows the missing `maxInputTokens` path, but this review did not run a credentialed SiliconFlow indexing request to reproduce the HTTP 400 directly.

## Solution Assessment

No current fix is present. The best fix is a shared custom-provider/OpenAI-compatible limit-resolution change with route-level tests, not a hardcoded SiliconFlow entry or an OpenAI-adapter-only patch.
