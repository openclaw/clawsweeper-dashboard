## Summary

Keep open: current main and the latest release still do not propagate configured third-party embedding input limits into the OpenAI-compatible memory embedding provider path, so SiliconFlow/BGE chunks can still fall back to the generic 8192 limit.

## What This Changes

The issue requests that memory indexing honor SiliconFlow/OpenAI-compatible embedding model input limits instead of sending oversized chunks that return HTTP 400.

## Best Possible Solution

Resolve configured embedding input limits in the shared custom/OpenAI-compatible provider path, add route-level regression coverage, and validate with redacted SiliconFlow or equivalent provider-limit proof.

## Reproduction Assessment

Yes for the source-level path: custom OpenAI-compatible providers can still produce a provider object without `maxInputTokens`, so memory chunking falls back to 8192. A credentialed SiliconFlow run would be needed for live end-to-end reproduction.

## Solution Assessment

No current implementation fully solves it. The best fix should handle the shared custom/OpenAI-compatible route rather than hardcoding one third-party model or only changing the OpenAI-native adapter.
