## Summary

Keep open. Current main still has the exact local-media allowlist rejection, but the delivery-queue permanent-error classifier does not match it and recovery/live failure paths still route non-matching errors through retry state. The issue remains a narrow, valid repair candidate; the secondary Telegram `message is too long` idea overlaps broader open delivery-recovery reports and should not block the local-media fix.

## What This Changes

Classify `Local media path is not under an allowed directory` delivery failures as permanent so queued outbound Telegram media sends do not waste retry budget on deterministic path-policy rejections.

## Best Possible Solution

Add a focused permanent delivery classification for the stable local-media allowlist rejection prefix, plus policy and recovery/drain tests proving existing queued entries with that deterministic `lastError` move to `failed/` without consuming the remaining retry budget. Keep enqueue-time validation, `lastErrorClass`, and broad Telegram 400/text-splitting work as separate follow-ups unless maintainers explicitly combine the scopes.

## Reproduction Assessment

Yes. A queued Telegram payload with a relative local `mediaUrls` entry reaches `loadWebMedia`, `assertLocalMediaAllowed` can throw the reported message, and current recovery/live failure handling only treats it as permanent if the existing regex classifier matches, which it does not.

## Solution Assessment

Yes for the title-scoped issue. The narrow maintainable solution is to extend the existing delivery-queue permanent-error policy and test the recovery behavior; earlier enqueue validation is useful but broader and not required to stop the retry-budget waste.
