## Summary

Keep open. Current main still has the exact local-media allowlist rejection path, but the delivery-queue permanent-error classifier does not match that message, so recovery continues to use retry/backoff instead of moving the entry directly to failed/. Related local-media work fixed broader attachment paths, and an exact referenced commit exists, but that change is not present on current main.

## What This Changes

Classify deterministic delivery failures such as local-media allowlist rejection and overlong Telegram messages as permanent delivery errors so queued items stop consuming retry budget.

## Best Possible Solution

Add a focused delivery-queue permanent-error classification for the local-media allowlist rejection, with policy and recovery tests proving the entry moves to failed/ without further retries. The exact overlong-message pattern can be included in the same narrow classifier PR if covered by tests, while enqueue-time validation and a new `lastErrorClass` field should remain separate follow-ups.

## Reproduction Assessment

Yes. A queued Telegram payload with a relative local `mediaUrls` entry reaches `loadWebMedia`, then `assertLocalMediaAllowed`, which throws the reported message; current recovery classification misses that message and routes it through `failDelivery` retry/backoff.

## Solution Assessment

Yes. The narrow maintainable fix is to extend the existing permanent-error classifier and add focused policy/recovery coverage; earlier enqueue validation would be useful but is broader than the retry-budget bug reported here.
