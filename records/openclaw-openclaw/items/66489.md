## Summary

Keep open. Current main still throws the reported local-media allowlist error, but the delivery-queue permanent-error classifier does not match it, so recovery falls through to retry/backoff; the related broader PR #74656 is closed and unmerged, and the later comment reports the same class still reproducing on v2026.4.29.

## What This Changes

The issue requests classifying deterministic delivery failures, especially the Telegram local-media allowlist rejection and overlong-message errors, as permanent so queued entries stop consuming retry budget.

## Best Possible Solution

Land a focused classifier repair in the existing delivery-queue recovery policy, with regression coverage for the local-media allowlist error and the overlong Telegram message error. Keep enqueue-time validation, `lastErrorClass`, and broader service-specific classifications as separate follow-ups unless each gets a precise test-backed pattern.

## Reproduction Assessment

Yes. The issue's queued Telegram media payload can reach `loadWebMedia` and `assertLocalMediaAllowed`, which still throws the exact reported message; current recovery classification misses that message and falls back to `failDelivery` retry/backoff.

## Solution Assessment

Yes for the immediate bug. Extending the existing permanent-error classifier with focused tests is the narrowest maintainable fix; earlier enqueue-time validation would be useful but is broader than the retry-budget failure reported here.
