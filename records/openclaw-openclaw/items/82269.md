## Summary

Keep open: current main still has a real mismatch where model-picker provider data reports a Pi-first runtime choice for OpenAI while the execution and status paths resolve the same official OpenAI model route to Codex.

## What This Changes

Runtime metadata produced for model-picker provider data should show the effective Codex harness for default official OpenAI models instead of a hard-coded Pi choice.

## Best Possible Solution

Derive picker runtime readback from the same resolveAgentHarnessPolicy path and centralized runtime labels used by execution/status, while preserving explicit provider/model runtime overrides and browse-only picker behavior.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction: buildModelsProviderData returns the Pi default choice for openai/gpt-5.5 while resolveAgentHarnessPolicy resolves that provider/model/config path to Codex.

## Solution Assessment

Yes. The narrow maintainable fix is to reuse the effective harness-policy and shared label contract for picker metadata, not to revive session runtime override persistence or add a new config surface.
