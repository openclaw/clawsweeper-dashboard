## Summary

Keep open. The linked bug is still source-reproducible on current main, and this PR is a plausible fix path, but the current branch is not merge-ready because it still calls the removed heartbeat wake API, has a direct `global` wake edge that can strand an event, and lacks inspectable real behavior proof.

## What This Changes

The branch threads wake-hook `agentId` and `sessionKey` through direct and mapped HTTP wake dispatch, adds session-policy guards, and adds gateway hook regression tests.

## Best Possible Solution

Land a rebased narrow fix that threads wake mapping targets through the existing hook session-policy path, uses the current heartbeat wake API, rejects undrainable `global` targets, and preserves focused gateway regressions.

## Reproduction Assessment

Yes. Source inspection on current main shows a high-confidence path: configure a wake mapping with `agentId` or `sessionKey`, then the mapping builder and mapped wake request branch drop those fields before `dispatchWakeHook` queues into the main session.

## Solution Assessment

No, not yet. Threading wake targets through the existing hook session-policy path is the right direction, but this branch needs the current heartbeat API, the literal `global` per-sender edge fixed, and inspectable proof before it is the narrow mergeable solution.
