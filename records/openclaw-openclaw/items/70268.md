## Summary

Keep this PR open. It is a useful focused fix for the wake-hook routing bug tracked at https://github.com/openclaw/openclaw/issues/64556, but the latest head still has a drain-safety bug for explicit `global` wake session keys outside global session scope, and the after-fix real behavior proof is not inspectable.

## What This Changes

The PR adds `sessionKey` and `agentId` routing for direct and mapped HTTP wake hooks, enforces hook session policy for those targets, and expands gateway hook tests.

## Best Possible Solution

Keep this PR open, reject or consistently canonicalize explicit `sessionKey: "global"` outside global session scope for both direct and mapped wake paths, add a focused regression, then let maintainers handle docs and changelog during landing.

## Reproduction Assessment

Yes. Current main is source-reproducible: wake mappings drop `agentId` and `sessionKey` before dispatch, and `dispatchWakeHook` queues the main session. The remaining PR bug is also source-reproducible with an explicit `sessionKey: "global"` in non-global session scope.

## Solution Assessment

No, not yet. The approach is the right narrow hook-session-policy path, but explicit literal `global` must either be rejected outside global scope or queued and woken through the same canonical key before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Reject global wake keys outside global scope:** `src/gateway/server/hooks-request-handler.ts:290-293`
  - body: `sessionKey: "global"` is accepted when `session.scope` is not `"global"`. `dispatchWakeHook` queues that literal key, but heartbeat resolution treats a forced `global` as invalid for per-sender scope and falls back to the main agent session, so the 200 response can leave the wake event undrained. Please reject `global` outside global scope or canonicalize both queue and heartbeat to the same key for direct and mapped wake paths.
  - confidence: 0.88
