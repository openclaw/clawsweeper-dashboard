## Summary

Keep open. Current main still hardcodes HTTP wake hooks to the main session, so the PR is addressing a real gateway bug, but the current PR head still has a drain-safety bug for explicit `global` wake keys outside global session scope and is missing required release/docs updates.

## What This Changes

The PR adds `sessionKey` and `agentId` routing for direct and mapped HTTP wake hooks, canonicalizes wake dispatch targets, enforces request/prefix/global-scope policy, and expands gateway hook tests.

## Best Possible Solution

Keep the PR open and land the focused HTTP wake-session routing fix after explicit `global` wake keys are made drain-safe outside global session scope, the regression is added, and the changelog/docs are updated without expanding the scope to cron, CLI, WS, or system-event wake routing.

## Reproduction Assessment

Yes. Current main reproduces by source inspection: direct wake drops `sessionKey`, mapped wake actions do not retain routing metadata, and dispatch always queues the main session. The PR's remaining bug reproduces with non-global `session.scope` plus direct or unmapped explicit wake `sessionKey: "global"`, which queues one key and wakes another.

## Solution Assessment

No, not yet. The PR is the right narrow approach for the hook bug, but it needs to reject or consistently canonicalize explicit `global` wake routing outside global scope, then document and record the user-facing behavior change.

Full review comments:

- **[P2] Reject global wake keys outside global scope:** `src/gateway/server/hooks-request-handler.ts:288-291`
  - body: This guard only rejects non-global keys when `session.scope` is `"global"`, so a normal per-sender gateway can still accept `sessionKey: "global"`. `dispatchWakeHook` then enqueues under `global`, while `requestHeartbeatNow({ sessionKey: "global" })` is resolved by the heartbeat runner as `agent:<default>:global`, so the 200 response can leave the wake event undrained. Please reject `global` outside global scope or canonicalize both queue and heartbeat to the same key.
  - confidence: 0.88

  - confidence: 0.84

Please update the webhook/config docs so operators know the new fields and global-scope limits.
