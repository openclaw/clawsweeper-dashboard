## Summary

Keep open. Current main still has a source-reproducible duplicate-final UI bug, but this PR should not merge as-is because it introduces concrete correctness and security blockers outside the narrow UI fix.

## What This Changes

The PR adds Control UI processed-runId deduplication while also changing Discord monitor code, Google Vertex/gaxios auth compatibility, sandbox tool policy/config types, and AGENTS gateway instructions.

## Best Possible Solution

Land a narrow UI-only fix that deduplicates processed terminal runIds for both own-run and other-run final append paths, preserves the dedupe set across same-session history reloads, clears it only on real session/agent/history resets, and adds focused regression coverage plus changelog coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction: call `handleChatEvent` with a renderable own-run `final`, observe `chatRunId` clear, then call it again with the same runId and payload; current main can append the assistant message twice.

## Solution Assessment

No. RunId-based terminal dedupe is the right direction, but this PR misses other-run final appends, clears dedupe state during same-session history reloads, and bundles unrelated risky changes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Remove the broken core Discord copies:** `src/discord/monitor/provider.lifecycle.ts:8-10`
  - body: The new `src/discord/monitor` files import siblings such as `../gateway-logging.js` and `../monitor.gateway.js`, but current main keeps Discord code under `extensions/discord` and the PR does not add those `src/discord` modules. This unrelated addition will fail module resolution/typecheck and should be removed from this UI dedupe PR or split into a complete Discord change.
  - confidence: 0.94
- **[P2] Preserve processed run IDs across history reloads:** `ui/src/ui/controllers/chat.ts:89-93`
  - body: Clearing `processedChatRunIds` in `loadChatHistory()` reopens the retransmit case this fix targets: a terminal handler can mark a run processed, a same-session reload clears the set, and a replayed final with the same runId can append again. Preserve the set for the active session and clear it only on real session/agent switches or explicit history clear.
  - confidence: 0.9
- **[P2] Mark other-run finals before returning:** `ui/src/ui/controllers/chat.ts:273-280`
  - body: The new guard only suppresses runIds already inserted into `processedChatRunIds`, but the other-run/sub-agent final branch appends `finalMessage` and returns without recording `payload.runId`. A retransmitted sub-agent final will therefore pass the guard and duplicate the appended message.
  - confidence: 0.88
- **[P2] Avoid creating a global window shim in Node:** `src/infra/gaxios-fetch-compat.ts:191-206`
  - body: This UI fix also adds a process-wide `globalThis.window` in a Node compatibility layer. That permanently changes `typeof window` for later dependencies and can route unrelated libraries into browser-only branches; the gaxios workaround should not mutate global browser-detection state and does not belong in this PR.
  - confidence: 0.87
