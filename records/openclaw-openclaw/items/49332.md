## Summary

Keep open. Current main still has the duplicate-final replay path, and this PR is not mergeable as-is because it misses other terminal paths, clears its dedupe state too aggressively, and bundles unrelated runtime/Discord/sandbox changes including a security-sensitive global window shim.

## What This Changes

The branch adds Control UI processed-runId tracking for terminal chat-event deduplication and also changes Discord monitor code, Google Vertex/gaxios auth compatibility, sandbox tool policy, config types, and repo instructions.

## Best Possible Solution

Land a narrow UI-only fix that tracks processed terminal runIds for the active session, covers own-run and other-run/sub-agent terminal append paths, preserves the set through same-session history reloads, clears it only on real session/agent/history resets, and includes focused regression tests plus changelog coverage.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to call `handleChatEvent` with a renderable own-run `final`, observe `chatRunId` clear, then call it again with the same `runId` and payload; current main appends the assistant message both times and the gateway does not reload history for renderable finals.

## Solution Assessment

No. RunId-based terminal dedupe is the right direction, but this PR is not the best implementation because it misses other-run final appends, clears dedupe state during same-session history reloads, and bundles unrelated changes outside the UI fix boundary.

  - body: Clearing `processedChatRunIds` in `loadChatHistory()` reopens the retransmit case this fix targets: a terminal handler can mark a run processed, a same-session reload clears the set, and a replayed final with the same `runId` can append again.   - confidence: 0.9

  - confidence: 0.88

  - body: This UI fix also adds a process-wide `globalThis.window` in a Node compatibility layer.
