## Summary

Keep this PR open because current main still lacks completed-run dedupe for retransmitted renderable terminal Control UI chat events, but the proposed branch should not merge as-is due to unrelated runtime/Discord/config changes plus concrete correctness and security findings in the diff.

## What This Changes

The PR adds Control UI processed-runId tracking for terminal chat-event deduplication and also changes Discord monitor code, sandbox tool policy, Google Vertex/gaxios auth compatibility, config types, and repo instructions.

## Best Possible Solution

Land a narrow UI-only fix that records processed terminal runIds for the active session, covers own-run `final`/`aborted` and other-run/sub-agent final append paths, preserves the completed-run set through same-session history reloads, clears it only on real session/agent switches or history clear, and adds focused regression tests for replaying the same renderable final event.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to call `handleChatEvent` with an own-run renderable `final`, observe `chatRunId` clear, then call it again with the same runId and payload; current main appends the assistant message a second time and the gateway path does not reload history for renderable finals.

## Solution Assessment

No. RunId-based terminal dedupe is the right direction, but this PR is not the best implementation because it clears the dedupe set on same-session history reload, misses other-run/sub-agent final appends, and bundles unrelated runtime and infrastructure changes.

  - body: The new `src/discord/monitor` files import siblings such as `../gateway-logging.js` and `./format.js` that are not added under `src/discord`; current main keeps these modules under `extensions/discord`. - **[P2] Keep processed run IDs across history reloads:** `ui/src/ui/controllers/chat.ts:89-93`
  - body: Clearing `processedChatRunIds` in `loadChatHistory()` reopens the reconnect/jitter case this fix targets: a terminal handler can mark a run processed, a same-session history reload clears the set, and a retransmitted final with the same runId can append again. Preserve the set for the active session and clear it only on real session/agent switches or explicit history clear.
  - confidence: 0.9
- **[P2] Mark other-run finals before returning:** `ui/src/ui/controllers/chat.ts:273-280`
  - body: The new guard only suppresses runIds that were previously inserted, but the other-run/sub-agent final branch appends `finalMessage` and returns without recording `payload.runId`. A retransmitted sub-agent final therefore passes the guard and duplicates the appended message.
  - confidence: 0.88
- **[P2] Avoid creating a global window shim in Node:** `src/infra/gaxios-fetch-compat.ts:191-206`
  - body: This UI fix also adds a process-wide `globalThis.window` in a Node compatibility layer. That permanently changes `typeof window` for later dependencies and can send unrelated libraries down browser-only paths; the gaxios workaround should not mutate global browser-detection state and does not belong in this PR.
  - confidence: 0.87
