## Summary

Keep open: the current release state still supports the reported regression. Current main has protocol-4 clients and a web cache mitigation, but `appcast.xml` still advertises the macOS app at 2026.5.7 while the latest stable GitHub release is 2026.5.12 with no macOS assets, so the stale macOS/web client explanation is still actionable and not cleanup-closeable.

## What This Changes

The issue reports that upgrading to OpenClaw v2026.5.12 leaves web and macOS app clients using protocol 3 against a protocol 4 gateway, producing WebSocket protocol mismatch failures.

## Best Possible Solution

Publish or promote the signed 2026.5.12 macOS app and appcast to match the protocol-4 gateway release, verify the web stale-cache recovery path, and keep protocol-3 non-probe rejection unchanged unless maintainers explicitly choose a compatibility window.

## Reproduction Assessment

Yes. A stale web/macOS client advertising `minProtocol=maxProtocol=3` against the current protocol-4 gateway reaches the source rejection path, and current release metadata shows the macOS app updater still points at 2026.5.7.

## Solution Assessment

No, closing this as already fixed is not the best path. The web service-worker fix is relevant, but the release/appcast state still needs maintainer-owned macOS publish verification or repair.
