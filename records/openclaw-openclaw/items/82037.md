## Summary

Keep open: current main still rejects protocol-3 webchat/non-probe clients and tests assert that behavior, while current UI/native clients advertise protocol 4. The report's logs match a stale 2026.5.7 client connecting to a 2026.5.12 gateway, so this needs maintainer handling rather than cleanup close.

## What This Changes

The issue reports that upgrading the gateway to 2026.5.12 while web/Mac app clients remain at 2026.5.7 causes protocol-mismatch connection failures.

## Best Possible Solution

Decide the intended upgrade contract, then either ensure update flows leave served web/native clients on protocol 4 or add a deliberate stale-client compatibility/upgrade message path without weakening the protocol boundary accidentally.

## Reproduction Assessment

Yes, source-reproducible: a webchat/control-ui connect with minProtocol=maxProtocol=3 against current main's protocol v4 handshake reaches the protocol-mismatch close path. I did not run a live macOS/Brave package upgrade because the checkout review is read-only.

## Solution Assessment

No, closing as expected behavior would be premature because a covered protocol rejection is colliding with a real update sequence reported against the latest release. The best fix needs a maintainer decision between compatibility, update-path repair, app-update handling, and clearer recovery guidance.
