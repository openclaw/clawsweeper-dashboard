## Summary

Keep open: current main has a likely web-side mitigation for stale Control UI bundles, but the report still describes a latest-release upgrade path where web/macOS app clients can end up speaking protocol 3 to a protocol 4 gateway. The remaining decision spans stale-client handling, served UI cache/update behavior, and the reporter's `ws://` versus `http://` narrowing, so this is not a conservative cleanup close yet.

## What This Changes

The issue reports that after updating the gateway to 2026.5.12, web and macOS app clients using a `ws://` path fail the WebSocket handshake with protocol mismatch errors while `http://` works.

## Best Possible Solution

Verify the current-main service-worker fix against the reported upgrade path, then either ship it with clear recovery guidance or add a deliberate macOS/web endpoint normalization or stale-client recovery path with regression coverage.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: a webchat/control-ui connect payload with `minProtocol=maxProtocol=3` against current protocol 4 reaches the `protocol mismatch` close path. The reporter also supplied live logs and narrowed the setup to `ws://` versus working `http://`, but I did not run the macOS package-upgrade path in this read-only review.

## Solution Assessment

No, closing this as already solved is premature. The current-main service-worker change is a plausible web-side fix, but maintainers still need to verify the release/app upgrade path and decide whether stale protocol clients should get compatibility, upgrade guidance, or strict rejection.
