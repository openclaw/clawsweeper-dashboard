## Summary

Keep open: current main has a relevant web cache mitigation, but the issue still has live release evidence for a protocol-3 client hitting a protocol-4 gateway, and the later `ws://` versus `http://` narrowing has not been validated or resolved in source.

## What This Changes

The issue reports a macOS upgrade regression where OpenClaw 2026.5.12 rejects web and app WebSocket clients speaking protocol 3 while the gateway expects protocol 4.

## Best Possible Solution

Verify the upgrade path with preserved browser/app state, then either ship the service-worker fix with clear recovery diagnostics or add a deliberate macOS endpoint normalization/stale-client recovery path with regression coverage.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: a non-probe connect payload with `minProtocol=maxProtocol=3` against current protocol 4 reaches the `protocol mismatch` close path. The reporter supplied matching live logs, but I did not run the macOS package-upgrade flow.

## Solution Assessment

No, closing as fixed is premature. Current main has a plausible web-side mitigation, but it is not in v2026.5.12 and does not prove the app `ws://` path or stale-client policy is resolved.
