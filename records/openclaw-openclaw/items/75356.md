## Summary

Keep this PR open: it addresses a valid Control UI/WebChat heartbeat-session bug, but the current PR head still preserves stale heartbeat `sessionId` values and is missing the required user-facing changelog entry.

## What This Changes

The PR projects isolated-heartbeat base-session metadata into Gateway/UI session rows, hides marker-backed heartbeat rows from the Control UI picker, normalizes chat sends toward the base session, adds focused tests, and restores a tool-display serializer export.

## Best Possible Solution

Keep the marker-backed base-session routing approach, but drop stale heartbeat session ids whenever the UI or Gateway rewrites an isolated heartbeat target to its base session, then add the required changelog entry and focused regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main can send a heartbeat key/id pair, and the PR still leaves a stale heartbeat `currentSessionId`/requested `sessionId` in the send path after base-key normalization.

## Solution Assessment

No, not yet. The marker-backed normalization is the right narrow design, but the implementation must also discard stale heartbeat transcript ids and document the user-visible fix in the changelog.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Clear stale ids when rewriting heartbeat sessions:** `ui/src/ui/controllers/chat.ts:522-529`
  - body: After `normalizeActiveChatSessionKey` rewrites a marker-backed heartbeat selection to the base key, this still sends `state.currentSessionId`. In the pruned-base case that id belongs to the heartbeat transcript, so the user send can still write to the heartbeat transcript even though the request key is now the base session. Clear or re-resolve the session id whenever the key is rewritten, and add a regression with a stale heartbeat `currentSessionId`.
  - confidence: 0.9
- **[P2] Drop requested ids after server heartbeat remaps:** `src/gateway/server-methods/chat.ts:1976-1981`
  - body: When Gateway remaps a marker-backed heartbeat key to its base key, `entry` can be missing because the base row was pruned. Falling back to `requestedSessionId` then preserves the heartbeat transcript id, so the server-side guard does not actually prevent stale heartbeat-key sends from landing in the heartbeat transcript. Track the remap and ignore the requested id unless it is known to belong to the resolved base row.
  - confidence: 0.86
- **[P2] Add the required changelog entry:** `ui/src/ui/controllers/chat.ts:599`
  - body: This is a user-visible Control UI/WebChat routing fix, but the PR head's `CHANGELOG.md` has no `#75203` or heartbeat-session routing entry. Add a single-line entry under the active Unreleased Fixes section before merge.
  - confidence: 0.94
