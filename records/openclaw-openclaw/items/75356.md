## Summary

Keep open: this PR targets a real Control UI/WebChat routing bug and uses the existing isolated-heartbeat marker, but the current head still forwards a stale heartbeat transcript session id after rewriting the session key and still lacks the required changelog entry.

## What This Changes

The PR projects heartbeat base-session markers to Gateway/UI session rows, hides marker-backed heartbeat rows in the Control UI picker, routes chat history/send/abort/events to the base key, adds focused tests, and restores a tool-display serializer export.

## Best Possible Solution

Keep the marker-backed routing approach, but normalize the session key and backing session id as one state transition so user transcript writes always target the base session, then land with changelog and regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a marker-backed heartbeat-only row plus stale currentSessionId is enough for the PR head to rewrite the key while still forwarding the heartbeat transcript id.

## Solution Assessment

No, not yet. Marker-backed normalization is the narrow maintainable fix, but this patch must also clear or re-resolve stale heartbeat session ids and add the required changelog entry.
