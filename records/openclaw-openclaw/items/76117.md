## Summary

Keep open: current main still leaves unconfigured local Ollama native requests without catalog-backed `num_ctx` and still applies the 120-second non-cron idle watchdog unless provider timeout is configured. The issue is already paired with open implementation PRs, so cleanup should wait for a maintainer-selected fix.

## What This Changes

Restore reliable local Ollama agent/tool behavior after upgrading past v2026.4.26, especially for qwen3.6:35B through gateway webchat.

## Best Possible Solution

Keep the issue open and review #76181 as the likely narrower fix, with any chosen solution preserving explicit `params.num_ctx`, documenting the provider-timeout workaround, and adding regression coverage for native Ollama context delivery and local idle-watchdog behavior.

## Reproduction Assessment

Unclear. Source inspection reproduces the unconfigured current-main paths: native Ollama leaves `options.num_ctx` unset and non-cron model streams use the 120-second watchdog unless provider timeout is configured, but I did not live-run the reported Ollama/qwen3.6:35B:a3b gateway webchat scenario.

## Solution Assessment

Unclear. A provider-aware Ollama/context fix like #76181 appears narrower than a global timeout bump, but the exact runtime-context and local-watchdog policy should be chosen by maintainers because it changes memory and hang behavior.
