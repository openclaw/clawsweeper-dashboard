## Summary

Keep open: current main still has the 120-second default LLM idle watchdog on non-cron model streams, and the issue is paired with open implementation PRs rather than a merged fix. Source inspection supports the reported timeout path, but the full Ollama/qwen tool-use degradation still needs live or trajectory proof.

## What This Changes

The issue reports a regression where local Ollama/qwen3.6:35B runs through gateway webchat degrade after v2026.4.26/v2026.4.29, allegedly because the 120-second LLM idle watchdog interrupts slow local model thinking.

## Best Possible Solution

Keep the issue open, use the provider-scoped timeout setting as the current workaround, and review the provider-aware local watchdog fix in #76181 or a maintainer-chosen equivalent with regression coverage and docs/changelog follow-through.

## Reproduction Assessment

Unclear. Source inspection and focused tests give a high-confidence reproduction path for the 120-second no-chunk watchdog, but I did not establish the full live Ollama/qwen3.6:35B:a3b gateway webchat behavior from the report.

## Solution Assessment

Unclear. A provider-aware local timeout policy looks narrower than a global default bump, but the exact semantics for disabling or extending the watchdog on local providers need maintainer review.
