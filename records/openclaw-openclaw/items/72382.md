## Summary

Keep open. Current main still has the reported installer path: it can warn that npm's global bin directory is missing from the parent shell PATH, keep verification opt-in, and still print a successful install banner before `openclaw` is runnable from that shell.

## What This Changes

The issue asks the curl/bash installer to leave Ubuntu users with a usable `openclaw` command after reporting a successful install.

## Best Possible Solution

Make installer finalization self-verifying and actionable so a successful install either leaves a directly usable CLI in the user's shell path or clearly reports the exact command/PATH export needed before onboarding.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: run the npm installer where npm's global bin directory is absent from the parent shell PATH and do not pass `--verify`; current main can warn but still finish with the success banner. I did not run a live Ubuntu 26.04 install in this read-only review.

## Solution Assessment

Unclear on exact exit and copy semantics, but the narrow maintainable fix belongs in `scripts/install.sh`: preserve npm prefix repair while making the final CLI path or required PATH export explicit, and fail only when the package or CLI truly cannot be resolved.

## Repair Work Prompt

Fix openclaw/openclaw issue #72382: the curl/bash installer can report success after an npm install while the user's parent shell still cannot run `openclaw` because npm's global bin directory is missing from PATH. Keep the change scoped to installer finalization and tests. In `scripts/install.sh`, make the final state self-verifying/actionable: use the resolved CLI path when possible, print the exact PATH export or direct command when the parent shell lacks the npm bin directory, and avoid leaving users with only a plain success banner when `openclaw` will be `command not found` in their shell. Do not add a new config option, do not change package manager ownership policy, and do not touch unrelated install flows. Add or update focused coverage in `test/scripts/install-sh.test.ts` and, if needed, the install smoke assertions so the parent-shell PATH case is not masked. Add a `CHANGELOG.md` fix entry because this is user-facing installer behavior.
