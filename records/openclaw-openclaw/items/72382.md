## Summary

Keep open. Current main still has the same installer UX path: npm installation can complete with a missing npm global bin PATH warning and a success banner, leaving the parent shell unable to run `openclaw`.

## What This Changes

Improve the curl/bash installer so a completed install leaves `openclaw` usable or gives a non-success/actionable final state when the CLI is not on PATH.

## Best Possible Solution

Make the installer end state self-verifying and actionable: after npm install, either leave a command the user can run immediately, make the required PATH export unmistakable in the final success path, or fail verification when the CLI cannot be resolved in the intended user shell.

## Reproduction Assessment

Yes, for the reported behavior class. Current source still supports a path where the installer resolves `openclaw` inside its own process after modifying PATH, while the parent shell remains unable to run `openclaw` without the npm global bin directory.

## Solution Assessment

Yes, the narrow maintainable solution is in the installer and its focused smoke/unit coverage. This should stay in `scripts/install.sh` and related tests/docs rather than becoming a broader CLI or package-manager change.
