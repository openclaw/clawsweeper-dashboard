## Summary

Keep open. Current main still matches the reported installer failure mode: the Unix installer can warn that npm's global bin directory is missing from the user's original PATH, skip install verification by default, and still print the successful install banner before `openclaw` is guaranteed to be runnable from the user's shell.

## What This Changes

The issue reports that the curl/bash installer can claim success after npm install while `openclaw` remains unavailable on the user's shell PATH.

## Best Possible Solution

Make the installer success state match what the user can do next: either ensure `openclaw` is runnable from the user's next shell command, or convert the missing original PATH case into clear action-required output before the success banner. Add a focused installer test for npm installs where the original PATH lacks the npm global bin directory; update docs only if the user-facing recovery guidance changes.

## Reproduction Assessment

Yes. Source inspection plus a read-only shell invocation reproduces the core behavior: the installer can emit the npm global bin PATH warning while default verification still returns success.

## Solution Assessment

Yes. A narrow `scripts/install.sh` behavior fix with focused tests is the best path; docs-only guidance would leave the default installer success state misleading.
