## Summary

Current main still leaves the reported installer success/PATH mismatch in place: the Unix installer can warn that npm's global bin directory is missing from the user's original PATH, skip verification by default, and still print the success banner before `openclaw` is guaranteed to be runnable from the user's shell.

## What This Changes

The issue reports that the macOS/Linux curl installer says it succeeded on Ubuntu 26.04 but leaves `openclaw` unavailable on PATH.

## Best Possible Solution

Make the installer success state match what the user can do next: either ensure `openclaw` is runnable from the user's next shell command, or convert the missing original-PATH case into clear action-required output before any success banner. Add focused installer coverage for npm installs where the original PATH lacks the npm global bin directory.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: run the installer through the npm method with an original PATH that lacks npm's global bin directory, and the code can warn while default verification still returns success.

## Solution Assessment

Yes. A narrow `scripts/install.sh` behavior fix with focused tests is the maintainable solution; docs-only guidance would leave the default installer success state misleading.
