## Summary

Keep open. Current main still matches the reported Unix installer failure mode: the default curl installer can warn that the original shell PATH lacks npm's global bin directory, skip post-install verification by default, and still print the success banner, while the existing non-root smoke explicitly adds that bin directory after the installer exits and does not prove the reporter's next `openclaw` command works.

## What This Changes

The issue reports that the Unix curl installer can report a successful Ubuntu install even though `openclaw` is not available in the user's shell afterward.

## Best Possible Solution

Make the Unix installer's normal success path match what the user can run next: either ensure the CLI is discoverable from the user's next shell without manual PATH repair, or turn the missing-original-PATH case into a clear action-required/non-success outcome with focused installer coverage and matching troubleshooting docs.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: run the npm-method installer with the npm global bin directory absent from the parent shell PATH and `OPENCLAW_VERIFY_INSTALL` unset; current main can warn about that PATH gap and still continue to a success banner.

## Solution Assessment

Yes. A narrow fix in `scripts/install.sh` with targeted installer and smoke coverage is the best path; docs-only guidance or optional verification would leave the misleading default success path in place.
