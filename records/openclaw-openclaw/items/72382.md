## Summary

Keep open. Current main still has the installer UX path where npm installation can warn that the npm global bin directory is missing from the original PATH and then print a success banner, leaving the parent shell unable to run `openclaw`.

## What This Changes

Improve the curl/bash installer so a completed install leaves `openclaw` usable or gives an actionable non-success/path state.

## Best Possible Solution

Make the installer end state self-verifying and actionable: either leave an immediately runnable command/path, make the required PATH export unmistakable in the final state, or fail verification when the CLI cannot be resolved for the intended user shell.

## Reproduction Assessment

Yes, for the reported behavior class. Current source has a source-level reproduction path: original PATH lacks npm's global bin, verification is off by default, the installer warns, and success messaging can still follow.

## Solution Assessment

Yes, the narrow maintainable solution is an installer UX/verification fix in `scripts/install.sh` with focused installer tests and docs updates, not a broader CLI or package-manager rewrite.
