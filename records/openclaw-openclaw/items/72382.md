## Summary

Keep open. Current main still has an installer UX path where the npm global bin warning can be followed by a successful completion message, leaving a user with `openclaw: command not found` in the parent shell.

## What This Changes

The issue reports that the curl installer on Ubuntu 26.04 says installation succeeded but `openclaw` is not available afterward.

## Best Possible Solution

Make the installer end state self-verifying and actionable: do not leave users with a plain success banner when the CLI is only available through an npm bin directory missing from their parent-shell PATH, and update the focused installer tests/docs around that behavior.

## Reproduction Assessment

Yes, for the reported behavior class. Current source emits the same PATH warning, keeps verification opt-in by default, and can finish with success messaging even though the parent shell may still report `openclaw: command not found`.

## Solution Assessment

Unclear for the exact final copy, but the narrow maintainable solution is in `scripts/install.sh`: preserve npm prefix repair while making the post-install CLI path or required PATH export explicit, and fail only when the package or CLI truly cannot be resolved.

## Repair Work Prompt

Fix the installer behavior reported in https://github.com/openclaw/openclaw/issues/72382. On Linux npm installs, when the npm global bin directory is missing from the original shell PATH, the curl installer currently warns and then can still end with a plain success message, after which the user's parent shell reports `openclaw: command not found`. Keep the existing npm prefix repair behavior, but make the final state actionable: after resolving `OPENCLAW_BIN`, either print an exact runnable command path plus a persistent PATH export/new-shell instruction when the parent shell cannot see the npm bin, or fail only when the package/CLI truly cannot be resolved. Avoid broad package-manager changes, dependency changes, release workflow changes, or changing default install method. Add focused tests in `test/scripts/install-sh.test.ts` for the missing-original-PATH case and for the verify/failure behavior. Update relevant install docs if user-facing guidance changes. Add a user-facing `CHANGELOG.md` fix entry.
