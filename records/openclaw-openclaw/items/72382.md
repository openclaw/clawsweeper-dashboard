## Summary

Keep open: current main still has the installer success-state mismatch reported here, with the PATH warning based on the caller shell, installer-local command resolution, opt-in verification, and a success banner before command availability is guaranteed in the user's next shell.

## What This Changes

The issue reports that the curl/bash installer can print success after an npm global bin PATH warning while `openclaw` remains unavailable as a shell command.

## Best Possible Solution

Make the installer's success state match the user's next action by either making the npm global bin PATH durable before success or treating the missing caller-shell PATH as action-required output, with focused regression coverage.

## Reproduction Assessment

Yes. The high-confidence reproduction path is the default npm installer flow with an original PATH that lacks npm's global bin directory: current main warns about that PATH, resolves `openclaw` only inside the installer process, skips verification by default, and still prints success.

## Solution Assessment

Yes. A narrow `scripts/install.sh` behavioral fix plus focused installer tests is the maintainable path; docs-only troubleshooting would leave the installer success banner misleading.
