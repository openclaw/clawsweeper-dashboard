## Summary

Keep open: current main still allows the Unix npm installer to warn that npm's global bin directory is missing from the caller's PATH, resolve `openclaw` only inside the installer process, skip verification by default, and print a success banner before the command is guaranteed to work in the user's next shell.

## What This Changes

The issue asks for the curl bash installer to leave `openclaw` usable after a reported successful Ubuntu npm install instead of ending with `openclaw: command not found`.

## Best Possible Solution

Make the installer success state match what the user can do next: either persist or require the needed PATH change before success, or stop with action-required output and focused regression coverage.

## Reproduction Assessment

Yes. The high-confidence reproduction path is the default npm installer flow with an original PATH that lacks npm's global bin directory: the installer warns, can resolve `openclaw` in its own process, skips verification by default, and still prints success.

## Solution Assessment

Yes. A narrow `scripts/install.sh` behavior fix with focused script tests is the maintainable solution; docs-only guidance would leave the installer success state misleading.

The observable problem is: on an npm install where the caller's original PATH lacks npm's global bin directory, `scripts/install.sh` can warn about the missing PATH, resolve `openclaw` only inside the installer process, skip verification by default, print `OpenClaw installed successfully`, and then the user's shell still reports `openclaw: command not found`. Keep the fix narrowly scoped to the installer success/PATH behavior and tests.
