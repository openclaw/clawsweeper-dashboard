## Summary

Keep open: current main still has the installer success/PATH mismatch described in the report. The npm install path warns when npm's global bin directory was missing from the original shell PATH, but the installer can still resolve the binary inside its own mutated PATH and print the success banner while post-install verification is opt-in.

## What This Changes

The issue reports that the macOS/Linux curl installer says OpenClaw installed successfully even though `openclaw` is not available afterward on Ubuntu 26.04.

## Best Possible Solution

Make the installer's reported final state match the user's next command: after npm installs, either ensure a durable executable path such as a wrapper on an already-present PATH, or treat the missing original-PATH case as action-required output before any success banner. Add focused tests for an npm install where the installer can find `${npm_bin}/openclaw` internally but the original shell PATH cannot.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is an npm install where `ORIGINAL_PATH` lacks `npm_global_bin_dir`: the installer warns, resolves `openclaw` through its own mutated PATH, skips verification by default, and still prints the success banner.

## Solution Assessment

Yes. A narrow `scripts/install.sh` behavior fix with installer tests is the best path; documentation alone would leave the default success state misleading.
