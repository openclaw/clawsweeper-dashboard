## Summary

Keep open. Current main still has the reported installer UX path: the installer can warn that the npm global bin is missing from the parent shell PATH, leave verification disabled by default, and then print a success banner even though `openclaw` may still be unavailable in the user's terminal.

## What This Changes

The issue reports that the Linux curl-bash installer can say installation succeeded after npm fallback/PATH warnings even though `openclaw` is not available in the user's shell.

## Best Possible Solution

Make the installer final state self-verifying and actionable so success means the CLI was resolved, or the user gets an explicit binary path/PATH export before leaving the install flow.

## Reproduction Assessment

Yes, for the reported behavior class. Current source defaults post-install verification off, warns about a missing npm global bin in the original PATH, and can still print the success banner afterward.

## Solution Assessment

Unclear on exact exit/copy semantics, but the narrow maintainable solution belongs in `scripts/install.sh` finalization and targeted installer smoke coverage rather than a broader install-system rewrite.

## Repair Work Prompt

Fix issue https://github.com/openclaw/openclaw/issues/72382. The observable bug is that the Linux curl-bash installer can finish with `OpenClaw installed successfully` after warning that the npm global bin is missing from the user's original PATH, leaving the parent shell with `openclaw: command not found`. Keep the fix bounded to installer finalization/PATH verification and focused tests. In `scripts/install.sh`, make the final state self-verifying and actionable: preserve npm prefix repair, resolve the installed CLI or bin link explicitly, avoid a plain success banner when the user's next shell cannot run `openclaw`, and give concrete PATH/export or direct-binary guidance.
