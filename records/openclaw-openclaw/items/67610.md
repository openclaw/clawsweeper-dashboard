## Label Justifications

- `P1`: When present, this bug breaks sandbox-backed agent turns for users who configure plugin-provided sandbox backends.
- `impact:other`: The issue affects sandbox backend plugin registration and runtime sandbox execution, a maintainer-visible runtime surface outside the more specific owned impact labels.

## Summary

Current main and shipped releases already fix the reported sandbox backend registry split: sandbox backend registrations now use a process-wide global registry, so plugin-loaded module instances and runtime lookup share the same backend map.

## What This Changes

The issue requests that `requireSandboxBackendFactory` find sandbox backends registered by plugins during gateway startup instead of throwing a not-registered error during agent execution.

## Best Possible Solution

Keep the shipped global registry implementation and close this issue; the same-author open PR can be handled separately as obsolete against current main.

## Reproduction Assessment

Yes, by source inspection against affected releases: older tags used a module-local backend `Map`, while plugin/runtime split module instances could register and look up different maps. Current main no longer has that failing shape.

## Solution Assessment

Yes. Fixing the core sandbox backend registry is the narrow maintainable solution because it preserves the plugin SDK contract and avoids backend-specific workarounds.
