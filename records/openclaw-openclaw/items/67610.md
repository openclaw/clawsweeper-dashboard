## Summary

Keep open. Current main still uses a module-local sandbox backend registry while plugin code registers through the SDK and sandbox context later resolves the configured backend through that registry; the reporter also has open PR #67623 for this same bug, so this issue should stay open until a reviewed fix lands.

## What This Changes

A custom sandbox backend plugin can register during startup, but sandboxed TUI turns later fail because `requireSandboxBackendFactory` cannot find the configured backend id.

## Best Possible Solution

Keep this issue open and review #67623 or a maintainer-authored equivalent that makes plugin-registered sandbox backends visible to core sandbox context resolution across plugin loader module instances, with a focused regression test for SDK-side registration followed by core lookup.

## Reproduction Assessment

Yes. The report gives a concrete path: install a custom sandbox backend plugin, configure `agents.defaults.sandbox.backend` to that id, restart the gateway, and send a sandboxed TUI turn; current source still has the module-local registry shape that can explain the reported missing backend.

## Solution Assessment

Yes on direction, not yet as a landed fix. A shared process registry plus a split-module regression test is the narrow maintainable fix, but this issue should remain open while #67623 or a replacement receives maintainer/security review.
