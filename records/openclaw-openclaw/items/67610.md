## Summary

Keep open. Current main still uses a module-local sandbox backend registry while plugin code registers through the SDK and sandbox context later resolves the configured backend through that registry; the reporter also has open same-author PR #67623 for this bug.

## What This Changes

The issue reports that a sandbox backend plugin registers successfully during startup, but `requireSandboxBackendFactory` cannot find that backend when a TUI turn resolves the configured sandbox backend.

## Best Possible Solution

Keep this issue open and review #67623 or a maintainer-authored replacement that makes plugin-registered sandbox backends visible across plugin loader module instances, with a focused regression test for SDK-side registration followed by core lookup.

## Reproduction Assessment

Yes. The report gives a concrete path: install a custom sandbox backend plugin, configure `agents.defaults.sandbox.backend` to that id, restart the gateway, and send a sandboxed TUI turn; current source still has the module-local registry shape that can explain the reported missing backend.

## Solution Assessment

Yes on direction, not as completed state. A shared process-wide registry plus a split-module regression test is the narrow maintainable fix, but current main has not landed it yet.
