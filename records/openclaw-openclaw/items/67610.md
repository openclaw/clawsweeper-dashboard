## Label Justifications

- `P1`: A configured sandbox backend can fail agent turns at runtime after plugin registration succeeds, breaking a real agent execution workflow for affected users.

## Summary

Keep open. Current main and the latest stable release still use a module-local sandbox backend registry, so the reported split-module plugin registration failure remains source-reproducible, and the issue is already paired with an open same-author fix PR.

## What This Changes

The issue asks for plugin-registered sandbox backend factories to remain discoverable when `requireSandboxBackendFactory` resolves the configured backend during an agent turn.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/67623 or an equivalent process-global sandbox backend registry fix with split-module/Jiti coverage and real external plugin proof, then close this issue as implemented.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: plugin code can register through `openclaw/plugin-sdk/sandbox` into one module-local map while sandbox runtime lookup calls `requireSandboxBackendFactory` from another module graph.

## Solution Assessment

Yes on direction, but it is not landed. A process-global sandbox backend registry with split-module/Jiti regression coverage is the narrow maintainable fix; backend-specific config or startup migration logic would not address the root cause.
