## Label Justifications

- `P1`: A configured sandbox backend plugin can register successfully at startup but fail at agent-turn runtime, breaking a real sandboxed agent workflow.

## Summary

Keep open. Current main and the latest release still use a module-local sandbox backend registry, so the reported split-module plugin registration failure remains source-reproducible, and an open paired fix PR already targets the right code path.

## What This Changes

Registering a custom sandbox backend plugin should make `requireSandboxBackendFactory` find that backend during agent turns.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/67623 or an equivalent process-global sandbox backend registry fix with split-module regression coverage and real external plugin proof.

## Reproduction Assessment

Yes, source-level. Plugin code registers through `openclaw/plugin-sdk/sandbox` into a module-local map, while `resolveSandboxContext` later calls `requireSandboxBackendFactory` from whichever module instance runtime sees.

## Solution Assessment

Yes on direction, not yet landed. The narrow maintainable fix is to share the sandbox backend registry across module instances and add split-module coverage, not add backend-specific config or startup migration logic.
