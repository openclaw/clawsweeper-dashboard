## Summary

Keep open. Current main and the latest release still use a module-local sandbox backend registry, so the reported plugin-registration split remains source-reproducible, and an open paired fix PR already tracks the repair.

## What This Changes

The issue reports that a custom sandbox backend plugin can register successfully during startup but later fails lookup through `requireSandboxBackendFactory` when a TUI message is sent.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/67623 or an equivalent process-global sandbox backend registry fix with split-module coverage and real external plugin proof, then close this issue as implemented.

## Reproduction Assessment

Yes. Source inspection provides a high-confidence path: plugin code registers through `openclaw/plugin-sdk/sandbox` into a module-local map, while `resolveSandboxContext` later calls `requireSandboxBackendFactory` from a potentially separate module graph.

## Solution Assessment

Yes on direction, not yet landed. The narrow maintainable solution is to make the sandbox backend registry process-global with split-module or Jiti-style regression coverage, rather than adding backend-specific config or startup migration logic.
