## Summary

Current main and the latest release already use a process-wide sandbox backend registry, so the reported split-module registration failure is fixed and no separate core bug remains.

## What This Changes

The issue reports that a plugin-registered sandbox backend is visible during gateway startup but later fails `requireSandboxBackendFactory` lookup when an agent turn starts.

## Best Possible Solution

Keep the shipped process-wide sandbox backend registry; the paired stale PR can be closed separately as superseded by current main and v2026.6.10.

## Reproduction Assessment

Yes, source-level. The historical failure path is clear from the issue and related OpenShell report: plugin registration used the sandbox SDK, while runtime lookup read a separate module-local registry; current main no longer has that failing registry shape.

## Solution Assessment

Yes. The current implementation fixes the core shared registry boundary instead of adding plugin-specific workarounds, which is the narrow maintainable solution for custom and externalized sandbox backends.
