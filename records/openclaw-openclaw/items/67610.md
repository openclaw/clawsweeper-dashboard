## Label Justifications

- `P1`: The original bug broke plugin-backed sandbox execution for affected users at message time after apparently successful gateway startup.
- `impact:other`: The issue affects the plugin sandbox backend registry, which is maintainer-visible runtime extensibility outside the more specific owned impact buckets.

## Summary

Current main already fixes the reported sandbox backend registry split: the registry is now stored on `globalThis` under `Symbol.for("openclaw.sandboxBackendFactories")`, so plugin-side registration and runtime lookup share the same process-wide map. The fix landed in merged OpenShell PR https://github.com/openclaw/openclaw/pull/84788 and first shipped in v2026.5.20-beta.2, with stable v2026.5.20 and latest v2026.6.6 also containing it.

## What This Changes

The issue asks for plugin-registered sandbox backends to remain discoverable by `requireSandboxBackendFactory` during agent turns after gateway startup registration succeeds.

## Best Possible Solution

Keep the shipped process-global sandbox backend registry and treat the older open workaround PR as redundant cleanup rather than a remaining issue fix.

## Reproduction Assessment

Yes, source-level for the affected versions: the pre-fix registry was a module-local map while plugins registered through the sandbox SDK and runtime lookup called `requireSandboxBackendFactory`. Current main no longer reproduces that failure because the registry is `globalThis`-backed.

## Solution Assessment

Yes. Moving the registry itself to a process-wide `Symbol.for` map is the narrow generic fix and avoids plugin-specific config, loader special cases, or new public API surface.
