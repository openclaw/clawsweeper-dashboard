## Summary

Keep open. Current main and the latest release still keep sandbox backend registrations in a module-local map, while runtime lookup uses that same registry path; the remaining work is already paired with an open same-author fix PR rather than a merged implementation.

## What This Changes

The issue reports that a plugin-registered sandbox backend is logged as registered during startup but later cannot be found by `requireSandboxBackendFactory` when a TUI message runs.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/67623 or an equivalent narrow fix that shares sandbox backend registrations across plugin/core module instances, with regression coverage for the split-module path.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: plugin registration writes to a module-local registry, while runtime sandbox context later reads from `requireSandboxBackendFactory` on the configured backend. I did not run the full external plugin gateway flow, so this is source-reproducible rather than live-reproduced.

## Solution Assessment

Yes on direction, but not complete yet. A process-wide registry using a direct `globalThis[Symbol.for(...)]` store or an explicitly approved equivalent is the narrow maintainable fix, with coverage before this issue closes.
