## Summary

Keep open: current main and v2026.6.8 still drop Matrix DMs configured with `dm.policy="open"` but no effective wildcard before dispatch, and the block reason is only emitted through verbose logging. The reporter’s follow-up proves dispatch, replies, and E2EE work once `dm.allowFrom` is set, so the remaining work is a security/product decision rather than cleanup closure.

## What This Changes

The issue asks OpenClaw to make Matrix DM access self-diagnosing when `dm.policy="open"` lacks `dm.allowFrom`, or redefine that policy so allowed DMs dispatch and replies deliver.

## Best Possible Solution

Keep the explicit wildcard contract, then add a redacted, bounded Matrix or shared ingress diagnostic so this invalid open-DM configuration is visible through doctor/status/startup or runtime warning without changing authorization semantics.

## Reproduction Assessment

Yes, at source level. Current main routes Matrix DMs through the shared ingress gate, and `dm.policy="open"` with no wildcard or sender allowlist resolves to `dm_policy_not_allowlisted` before dispatch.

## Solution Assessment

Unclear until maintainers choose the policy shape. The safest direction is to preserve explicit wildcard authorization and make the invalid or surprising Matrix config self-diagnosing rather than making open DMs implicitly public.
