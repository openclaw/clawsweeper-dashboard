## Summary

Keep open: the reporter’s follow-up narrows the Matrix symptoms to a current source-proven DM authorization gate, not E2EE transport loss. Current main and v2026.6.6 still require an explicit `dm.allowFrom` match or `"*"` for `dm.policy="open"`, and Matrix only emits the blocked-DM reason through verbose logging, so the remaining work needs a maintainer security/product choice rather than cleanup closure.

## What This Changes

The issue asks Matrix DMs with `dm.policy="open"` to dispatch and reply reliably, or to make the allowlist gate and required wildcard self-diagnosing instead of appearing as silent message loss.

## Best Possible Solution

Keep the explicit wildcard security contract, then add Matrix-specific validation, doctor/status guidance, or warn-level operator diagnostics so `dm.policy="open"` without an effective wildcard is caught before DMs appear lost.

## Reproduction Assessment

Yes at source level. Current main routes Matrix DMs through the shared ingress gate, and `dm.policy="open"` with no wildcard or sender allowlist resolves to `dm_policy_not_allowlisted` before dispatch.

## Solution Assessment

Unclear until maintainers choose the policy shape. The safest direction is not to make open DMs implicitly public, but to preserve the wildcard contract and make invalid or surprising Matrix configs self-diagnosing.
