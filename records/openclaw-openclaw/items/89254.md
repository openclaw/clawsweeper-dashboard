## Summary

Keep open: the reporter narrowed the original Matrix E2EE symptoms to a DM access-policy drop, and current main still source-proves that `dm.policy="open"` without a matching wildcard or sender allowlist blocks Matrix DMs with only verbose logging. The remaining work needs a maintainer security/product decision on validation, warning, or policy semantics.

## What This Changes

The issue asks Matrix to stop presenting `dm.policy="open"` with empty `dm.allowFrom` as silent message loss and to clarify or change the open-DM allowlist behavior.

## Best Possible Solution

Keep the explicit wildcard security contract, then add Matrix-specific validation, doctor/status guidance, or warn-level operator diagnostics so manual `dm.policy="open"` without an effective wildcard is caught before DMs appear lost.

## Reproduction Assessment

Yes at source level. Current main routes Matrix DMs through the shared ingress gate, and `dm.policy="open"` with no wildcard or sender allowlist resolves to `dm_policy_not_allowlisted` before dispatch.

## Solution Assessment

Unclear until maintainers choose the policy shape. The safest direction is not to make open DMs implicitly public, but to preserve the wildcard contract and make invalid or surprising Matrix configs self-diagnosing.
