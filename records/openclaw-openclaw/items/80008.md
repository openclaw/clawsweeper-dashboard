## Summary

Keep this PR open for maintainer API/security review rather than cleanup close. The delivery gap described by the author is plausible, but the current patch still exposes ACP dispatch through a global plugin flag and misses existing caller-scoped policy and ACP explicit-turn semantics.

## What This Changes

The PR adds `api.runtime.acp.spawn()` and `api.runtime.acp.prompt()`, a new `plugins.allowAcpSpawn` config key, schema/types, tests, and plugin runtime mocks.

## Best Possible Solution

Land a rebased version only after extracting a leaf ACP runtime contract, gating ACP access by calling plugin identity, preserving explicit ACP turn semantics, and documenting the operator opt-in.

## Reproduction Assessment

Yes for the PR review blockers: source inspection shows the global gate and missing explicit ACP turn marker, and the earlier ClawSweeper comment reported the import-cycle architecture failure. As a feature PR, there is no current-main user bug reproduction requirement.

## Solution Assessment

No, not yet. Routing ACP prompts through the gateway delivery path is the right direction, but the public plugin runtime surface should first align with existing caller-scoped runtime policy and ACP dispatch semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Break the ACP runtime type import cycle:** `src/plugins/runtime/types-core.ts:1`
  - body: Importing `SpawnAcp*` types from `../../agents/acp-spawn.js` pulls the public plugin runtime type surface back into the ACP/agents implementation graph. Extract a leaf DTO/contract for the runtime API instead of importing the agent implementation module from `types-core.ts`.
  - confidence: 0.9
- **[P1] Scope ACP permission to the calling plugin:** `src/config/types.plugins.ts:70`
  - body: `plugins.allowAcpSpawn` is global, so enabling ACP for one trusted plugin gives every loaded third-party plugin access to `api.runtime.acp.spawn()` and `prompt()`. This should follow the existing `plugins.entries.<id>` plus runtime plugin identity model before exposing host-side ACP dispatch.
  - confidence: 0.9
- **[P2] Mark plugin ACP prompts as explicit turns:** `src/plugins/runtime/runtime-acp.runtime.ts:52-68`
  - body: `promptPluginAcp()` sends the gateway `agent` request without `acpTurnSource: "manual_spawn"`, so follow-up prompts are treated as automatic dispatch and can fail when `acp.dispatch.enabled=false`. Native ACP spawn sets this marker for explicit operator-initiated turns.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The patch introduces privileged plugin-to-ACP dispatch with a global opt-in rather than caller-scoped authorization.

Concerns:

- **[high] Global ACP gate grants all plugins spawn access:** `src/config/types.plugins.ts:70`
  - body: With `plugins.allowAcpSpawn: true`, any loaded plugin can spawn or continue ACP-backed coding sessions with channel delivery. Since ACP can launch host-side harnesses, this permission should be tied to the calling plugin identity or an equivalent per-plugin trust contract.
  - confidence: 0.9
