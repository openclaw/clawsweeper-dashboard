## Summary

This PR should stay open because current main still lacks the requested channel-delivering `PluginRuntime.acp` surface, and the contributor supplied useful downstream delivery proof. The current patch should not merge as-is because it exposes privileged ACP dispatch behind a global plugin flag, couples plugin runtime types to the agents implementation graph, and omits the native explicit ACP turn marker.

## What This Changes

The PR adds `api.runtime.acp.spawn()` and `api.runtime.acp.prompt()` to the plugin runtime, guarded by a new `plugins.allowAcpSpawn` config key, with schema, mock, and runtime tests.

## Best Possible Solution

Land a revised API only after ACP DTOs live in a leaf contract, ACP authority is scoped to the calling plugin identity, explicit ACP turn metadata is preserved, and the operator opt-in is documented.

## Reproduction Assessment

Yes for the review blockers: PR-head source and CI show the global ACP gate, the plugin-runtime type back-edge, and the missing explicit ACP turn marker. The requested runtime namespace is a new feature, so there is no current-main bug reproduction requirement.

## Solution Assessment

No, not yet. Gateway-backed ACP delivery is a plausible direction, but this implementation should be revised so authorization, type ownership, and ACP dispatch semantics match existing plugin runtime boundaries.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Scope ACP permission to the calling plugin:** `src/config/types.plugins.ts:70`
  - body: `plugins.allowAcpSpawn` is process-wide, so enabling ACP for one trusted plugin gives every loaded third-party plugin access to `api.runtime.acp.spawn()` and `prompt()`. ACP dispatch can start host-side agent sessions and deliver into channels, so this permission should be tied to the calling plugin identity or another explicit per-plugin trust contract before exposing it in the runtime.
  - confidence: 0.9
- **[P1] Break the ACP runtime type import cycle:** `src/plugins/runtime/types-core.ts:1`
  - body: Importing `SpawnAcp*` types from `../../agents/acp-spawn.js` pulls the public plugin runtime type surface back into the agents implementation graph. The PR head's architecture job reports a Madge cycle ending through this file, so move the DTOs to a leaf contract module or define plugin runtime DTOs locally.
  - confidence: 0.93
- **[P2] Mark plugin ACP prompts as explicit turns:** `src/plugins/runtime/runtime-acp.runtime.ts:52-68`
  - body: `promptPluginAcp()` calls the gateway `agent` method without `acpTurnSource: "manual_spawn"`, so follow-up prompts into ACP sessions can be treated like automatic dispatch instead of explicit operator/plugin-initiated turns. Native ACP spawn uses this marker, and plugin prompts should preserve the same dispatch policy.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The patch exposes privileged ACP dispatch to plugins through a global opt-in rather than caller-scoped authorization.

Concerns:

- **[high] Global ACP gate grants all plugins spawn access:** `src/config/types.plugins.ts:70`
