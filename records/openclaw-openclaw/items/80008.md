## Summary

Keep open: the requested ACP runtime delivery surface is still not on current main and this newer PR supersedes the older branch, but the current patch needs human API/security review before merge because it exposes privileged ACP dispatch through a global plugin flag and misses explicit ACP turn semantics.

## What This Changes

The PR adds `api.runtime.acp.spawn()` and `api.runtime.acp.prompt()`, a `plugins.allowAcpSpawn` config key, schema/runtime mocks, and plugin runtime tests.

## Best Possible Solution

Land a rebased version only after extracting a leaf ACP runtime contract, deriving caller identity from plugin runtime scope, gating ACP access per plugin, preserving explicit ACP turn semantics, and documenting the opt-in.

## Reproduction Assessment

Yes for the review blockers: source inspection of the PR head and current main shows the global ACP gate, the type-boundary back edge, and the missing explicit ACP turn marker. As a feature PR, there is no current-main user bug reproduction requirement.

## Solution Assessment

No, not yet. Routing plugin ACP prompts through the gateway delivery path is directionally reasonable, but the proposed public API should not merge until the type boundary, authorization scope, and ACP dispatch policy match existing runtime contracts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Break the ACP runtime type import cycle:** `src/plugins/runtime/types-core.ts:1`
  - body: Importing `SpawnAcp*` types from `../../agents/acp-spawn.js` pulls the public plugin runtime type surface back into the agents/ACP implementation graph. Move the DTOs to a leaf contract or define plugin-runtime-specific request/result types so the runtime type facade stays acyclic.
  - confidence: 0.9
- **[P1] Scope ACP permission to the calling plugin:** `src/config/types.plugins.ts:70`
  - body: The new `plugins.allowAcpSpawn` flag is global, so enabling ACP for one trusted plugin gives every loaded plugin access to host-side ACP spawn and prompt dispatch. Mirror the existing `plugins.entries.<id>` plus plugin runtime identity model before exposing this capability.
  - confidence: 0.92
- **[P2] Mark plugin ACP prompts as explicit turns:** `src/plugins/runtime/runtime-acp.runtime.ts:52-68`
  - body: `promptPluginAcp()` sends a gateway `agent` request without `acpTurnSource: "manual_spawn"`, so follow-up prompts into ACP sessions are treated as automatic dispatch and can fail when `acp.dispatch.enabled=false`. Native ACP spawn sets this marker for explicit operator/plugin-initiated turns.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The patch adds a privileged plugin-to-ACP dispatch surface whose authorization model is too broad for third-party plugins.

Concerns:

- **[high] Global ACP gate grants all plugins spawn access:** `src/config/types.plugins.ts:70`
  - body: With `plugins.allowAcpSpawn: true`, any loaded plugin can spawn or continue ACP-backed coding sessions with channel delivery. Since ACP can launch host-side coding harnesses, the permission should be tied to the calling plugin identity or an equivalent per-plugin trust contract before merge.
  - confidence: 0.92
