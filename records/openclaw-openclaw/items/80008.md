## Summary

Keep this PR open, but it should not merge in its current shape. The feature is still absent from current main and the contributor provided useful real runtime logs, but the branch introduces a public privileged ACP runtime surface with an import-cycle failure, a global authorization gate, and a prompt path that misses the explicit ACP turn marker.

## What This Changes

The PR adds `api.runtime.acp.spawn()` and `api.runtime.acp.prompt()`, a `plugins.allowAcpSpawn` config key, schema/type/mock updates, and plugin runtime tests.

## Best Possible Solution

Land a rebased version only after extracting a leaf ACP runtime contract, gating ACP access by caller plugin identity under `plugins.entries.<id>`, preserving explicit ACP turn semantics, and documenting the operator opt-in.

## Reproduction Assessment

Yes for the review blockers: source inspection and PR CI show the import-cycle failure, global authorization gate, and missing explicit ACP turn marker. As a feature PR, there is no current-main user bug reproduction requirement.

## Solution Assessment

No, not yet. Routing plugin ACP prompts through the gateway delivery path is the right direction, but the public API should not merge until the type boundary, authorization scope, and ACP dispatch policy match existing runtime contracts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Break the ACP runtime type import cycle:** `src/plugins/runtime/types-core.ts:1`
  - body: Importing `SpawnAcp*` types from `../../agents/acp-spawn.js` pulls the public plugin runtime type surface back into the ACP/agents graph, and the PR head's architecture job fails with a Madge cycle ending through `src/plugins/runtime/types-core.ts`. Extract a leaf contract or define runtime DTOs that do not import agent implementation modules.
  - confidence: 0.93
- **[P1] Scope ACP permission to the calling plugin:** `src/config/types.plugins.ts:70`
  - body: The new `plugins.allowAcpSpawn` flag is global, so enabling ACP for one trusted plugin gives every loaded third-party plugin access to `api.runtime.acp.spawn()` and `prompt()`. Mirror the existing `plugins.entries.<id>` plus plugin runtime identity model before exposing host-side agent dispatch.
  - confidence: 0.9
- **[P2] Mark plugin ACP prompts as explicit turns:** `src/plugins/runtime/runtime-acp.runtime.ts:52-67`
  - body: `promptPluginAcp()` calls the gateway `agent` method without `acpTurnSource: "manual_spawn"`, so follow-up prompts into ACP sessions are treated as automatic dispatch and fail when `acp.dispatch.enabled=false`. Native ACP spawn sets this marker, and plugin prompts should do the same if they are explicit operator/plugin-initiated turns.
  - confidence: 0.88
