## Summary

Keep this PR open, but it is not ready to merge: current main still lacks the proposed plugin runtime ACP API, while the PR head has a CI-proven import-cycle blocker plus unresolved ACP trust-model and dispatch-policy issues.

## What This Changes

The PR adds `api.runtime.acp.spawn()` and `api.runtime.acp.prompt()`, a `plugins.allowAcpSpawn` config key, schema/types, runtime mocks, and plugin runtime tests.

## Best Possible Solution

Land a revised version only after extracting a leaf ACP runtime contract, gating ACP access by calling plugin identity, preserving explicit ACP turn semantics, and documenting the operator opt-in.

## Reproduction Assessment

Yes for the review blockers: PR CI shows the architecture import-cycle failure, and source inspection shows the global ACP gate plus missing explicit-turn marker. This is a feature PR, so there is no current-main bug reproduction requirement.

## Solution Assessment

No, not yet. Routing through the gateway delivery path is the right direction, but the implementation should not merge until the type boundary, authorization scope, and ACP dispatch policy match current runtime contracts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Break the ACP runtime type import cycle:** `src/plugins/runtime/types-core.ts:1`
  - body: Importing `SpawnAcp*` types from `../../agents/acp-spawn.js` pulls the public plugin runtime type surface back into the ACP/agents graph, and the current PR CI fails `check:architecture` with a Madge cycle through `src/plugins/runtime/types-core.ts`. Please extract a leaf DTO/contract instead of importing agent implementation types here.
  - confidence: 0.94
- **[P1] Scope ACP permission to the calling plugin:** `src/config/types.plugins.ts:70`
  - body: `plugins.allowAcpSpawn` is global, so enabling ACP for one trusted plugin gives every loaded third-party plugin access to `api.runtime.acp.spawn()` and `prompt()`. ACP can launch host-side coding harnesses, so this should use the existing plugin identity and `plugins.entries.<id>` trust model before exposing the runtime surface.
  - confidence: 0.92
- **[P2] Mark plugin ACP prompts as explicit turns:** `src/plugins/runtime/runtime-acp.runtime.ts:52-68`
  - body: `promptPluginAcp()` sends the gateway `agent` request without `acpTurnSource: "manual_spawn"`. Current ACP turns without that marker use dispatch policy and fail when `acp.dispatch.enabled=false`, while native ACP spawn marks explicit manual turns, so plugin prompts should preserve the same policy semantics.
  - confidence: 0.89
