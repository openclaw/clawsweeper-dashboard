## Summary

Keep open: current main still lacks the requested ACP plugin runtime namespace, and the PR has credible downstream proof, but the proposed patch should not merge until the import cycle, caller-scoped authorization, and ACP explicit-turn semantics are fixed.

## What This Changes

The PR adds `api.runtime.acp.spawn()` and `api.runtime.acp.prompt()`, a `plugins.allowAcpSpawn` config/schema key, runtime mocks, and plugin runtime tests.

## Best Possible Solution

Land a revised API with leaf ACP DTO types, caller-scoped `plugins.entries.<id>.acp` authorization, explicit `acpTurnSource: "manual_spawn"` prompt dispatch, and matching SDK/config docs and tests.

## Reproduction Assessment

Yes for the PR review blockers: live CI shows the Madge import-cycle failure, and source inspection shows the global gate and missing explicit ACP turn marker. Not applicable as a current-main bug because this is a new feature PR.

## Solution Assessment

No, not yet. Routing through the gateway delivery path is the right direction, but the patch should use a leaf type contract, caller-scoped ACP permission, and native explicit-turn semantics before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Break the ACP runtime type import cycle:** `src/plugins/runtime/types-core.ts:1`
  - body: Importing `SpawnAcp*` types from `../../agents/acp-spawn.js` pulls the public plugin runtime type surface into the agents/ACP implementation graph, and the current PR head fails `check-additional-runtime-topology-architecture` with a Madge cycle ending at this file. Extract leaf DTO types instead of importing the agent implementation module.
  - confidence: 0.94
- **[P1] Scope ACP permission to the calling plugin:** `src/config/types.plugins.ts:70`
  - body: The new `plugins.allowAcpSpawn` flag is global, so enabling ACP for one trusted plugin gives every loaded third-party plugin access to spawn and continue ACP-backed coding sessions. Mirror the existing `plugins.entries.<id>` plus plugin runtime identity model before exposing this host-side dispatch surface.
  - confidence: 0.91
- **[P2] Mark plugin ACP prompts as explicit turns:** `src/plugins/runtime/runtime-acp.runtime.ts:52-68`
  - body: `promptPluginAcp()` calls the gateway `agent` method without `acpTurnSource: "manual_spawn"`, so follow-up prompts into ACP sessions use dispatch policy and can fail when `acp.dispatch.enabled=false`. Native ACP spawn sets this marker; plugin-initiated follow-ups should do the same if they are explicit turns.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The diff introduces a privileged plugin-to-ACP dispatch surface with a global opt-in instead of caller-scoped authorization.

Concerns:
