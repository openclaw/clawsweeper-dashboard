## Summary

Keep open: the feature addresses a plausible plugin delivery gap and the PR body has real runtime logs, but the current patch should not merge because it exposes privileged ACP dispatch behind a global plugin flag, creates a plugin-runtime type import cycle that fails CI, and omits the explicit ACP turn marker used by the native ACP path.

## What This Changes

The PR adds `api.runtime.acp.spawn()` and `api.runtime.acp.prompt()` to the plugin runtime, backed by a new `plugins.allowAcpSpawn` config key plus schema, tests, and runtime mock updates.

## Best Possible Solution

Land a revised plugin ACP runtime API only after moving ACP DTOs to a leaf contract, scoping ACP authority to the calling plugin identity, preserving explicit ACP turn semantics, and documenting the operator opt-in.

## Reproduction Assessment

Yes for the review blockers: PR head source and CI logs show the import cycle, global authorization gate, and missing explicit ACP turn marker. It is not a current-main bug reproduction because this is a new API/config feature PR.

## Solution Assessment

No, not yet. Routing plugin ACP prompts through the gateway delivery path is plausible, but the implementation should first match existing plugin identity, architecture, and ACP dispatch-policy contracts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Break the ACP type import cycle:** `src/plugins/runtime/types-core.ts:1`
  - body: Importing `SpawnAcp*` types from `../../agents/acp-spawn.js` makes the public plugin runtime type surface depend on the ACP/agents implementation graph. The PR head's architecture job reports a Madge cycle ending at this file, so move these DTOs to a leaf contract or define plugin-runtime DTOs without importing agent implementation modules.
  - confidence: 0.93
- **[P1] Scope ACP permission to the calling plugin:** `src/config/types.plugins.ts:70`
  - body: `allowAcpSpawn` is a global `PluginsConfig` flag, so enabling ACP for one trusted plugin gives every loaded plugin access to ACP spawn/prompt with channel delivery. This should follow the existing `plugins.entries.<id>` plus plugin runtime identity model before exposing host-side coding-agent dispatch.
  - confidence: 0.9
- **[P2] Preserve explicit ACP turn policy:** `src/plugins/runtime/runtime-acp.runtime.ts:52-68`
  - body: `promptPluginAcp()` calls gateway `agent` without `acpTurnSource: "manual_spawn"`. Native ACP spawn sets that marker, and `agent-command` uses it to apply explicit-turn policy; without it, plugin follow-up prompts into ACP sessions are treated as automatic dispatch and fail when `acp.dispatch.enabled=false`.
  - confidence: 0.88

## Security Review
