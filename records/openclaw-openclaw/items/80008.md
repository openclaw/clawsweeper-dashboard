## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Break the ACP runtime type import cycle:** `src/plugins/runtime/types-core.ts:1`
  - body: Importing `SpawnAcp*` types from `../../agents/acp-spawn.js` pulls the public plugin runtime type surface into the ACP agent implementation graph, and the latest head still has a failing runtime topology architecture job. Define a leaf ACP runtime contract instead of importing from the agent implementation module.
  - confidence: 0.93
- **[P1] Scope ACP permission to the calling plugin:** `src/config/types.plugins.ts:70`
  - body: `plugins.allowAcpSpawn` is global, so enabling ACP for one trusted plugin gives every loaded third-party plugin access to `api.runtime.acp.spawn()` and `prompt()`. This should follow the existing `plugins.entries.<id>` plus plugin runtime identity model before exposing host-side ACP dispatch.
  - confidence: 0.91
- **[P2] Mark plugin ACP prompts as explicit turns:** `src/plugins/runtime/runtime-acp.runtime.ts:52-68`
  - body: `promptPluginAcp()` calls the gateway `agent` method without `acpTurnSource: "manual_spawn"`, so follow-up prompts can be treated as automatic ACP dispatch and fail when `acp.dispatch.enabled=false`. Native ACP spawn sets this marker for the same explicit operator/plugin-initiated turn path.
  - confidence: 0.88
