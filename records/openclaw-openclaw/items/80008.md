## Summary

Keep open: this active external feature PR has useful after-fix logs, but the current head still has blocking design, security, and architecture issues around privileged ACP dispatch from plugins.

## What This Changes

The PR adds a gated `api.runtime.acp` plugin runtime namespace with spawn/prompt helpers, config schema/type support, runtime mocks, and tests.

## Best Possible Solution

Keep the ACP plugin runtime seam only if maintainers want this core API, and land it with caller-scoped plugin authorization, explicit ACP turn semantics, a cycle-free type contract, docs, and focused runtime coverage.

## Reproduction Assessment

Yes for the review blockers: source inspection and CI logs show the global gate, missing explicit ACP turn marker, and architecture cycle on the PR head. I did not run local tests because this was a read-only review and `pnpm` is unavailable.

## Solution Assessment

No, not yet. Routing plugin ACP prompts through the gateway delivery path is plausible, but the current implementation is not the narrowest maintainable solution until the privilege model, policy marker, and type boundary are fixed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Scope ACP permission to the calling plugin:** `src/config/types.plugins.ts:65-70`
  - body: The new top-level `plugins.allowAcpSpawn` flag is checked without plugin identity, so enabling ACP for one trusted plugin enables every loaded third-party plugin to spawn or prompt ACP agent turns. Existing sensitive runtime policies are under `plugins.entries.<id>` and the runtime proxy already carries `pluginId`; gate this per calling plugin before exposing agent dispatch.
  - confidence: 0.9
- **[P2] Mark plugin ACP prompts as explicit turns:** `src/plugins/runtime/runtime-acp.runtime.ts:52-67`
  - body: `promptPluginAcp()` sends the gateway `agent` request without `acpTurnSource: "manual_spawn"`. Current `agent-command` treats missing markers as automatic dispatch, so plugin follow-up prompts are blocked when operators set `acp.dispatch.enabled=false`, unlike native ACP spawn.
  - confidence: 0.88
- **[P2] Move ACP DTOs out of the runtime implementation graph:** `src/plugins/runtime/types-core.ts:1`
  - body: Importing `SpawnAcp*` types from `../../agents/acp-spawn.js` pulls the plugin runtime contract into the ACP/agents implementation graph, and CI reports a Madge cycle through `agents/acp-spawn` and `plugins/runtime/types-core`. Put the DTOs in a leaf contract file or define a narrow plugin-facing shape so `check:architecture` can pass.
  - confidence: 0.86
