## Summary

Keep open. The downstream channel-delivery gap is plausible and the PR includes useful real logs, but the current patch still exposes a privileged ACP surface through a global plugin flag, creates an architecture import cycle, and omits the explicit ACP turn marker used by the existing gateway policy.

## What This Changes

Adds `api.runtime.acp.spawn()`/`prompt()`, a `plugins.allowAcpSpawn` config key, schema/tests/mocks, and a lazy ACP runtime helper.

## Best Possible Solution

Land a rebased version only after extracting leaf ACP runtime contracts, gating ACP access per plugin identity, preserving explicit ACP turn semantics, and documenting the operator opt-in.

## Reproduction Assessment

Yes for the review blockers: PR CI reproduces the import-cycle failure, and source inspection shows the global gate and missing `acpTurnSource`. As a feature PR, there is no current-main bug reproduction requirement.

## Solution Assessment

No, not yet. Routing plugin ACP prompts through the gateway delivery path is the right direction, but the public API should be shaped around the existing per-plugin trust model and ACP explicit-turn policy before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Extract ACP runtime types from a leaf contract:** `src/plugins/runtime/types-core.ts:1`
  - body: Importing `SpawnAcp*` from `../../agents/acp-spawn.js` pulls the public plugin runtime type surface back into the agents/ACP graph, and the PR CI now fails the Madge architecture check with a cycle ending at this file. Define a leaf DTO/contract for the plugin runtime instead of importing the agent implementation module.
  - confidence: 0.93
- **[P1] Scope ACP access to the calling plugin:** `src/config/types.plugins.ts:67-70`
  - body: `plugins.allowAcpSpawn` is global, so enabling ACP for one trusted plugin grants every loaded third-party plugin `api.runtime.acp.spawn()` and `prompt()`. This privileged host-side agent dispatch surface should follow the existing `plugins.entries.<id>` plus runtime plugin identity model.
  - confidence: 0.9
- **[P2] Mark plugin ACP prompts as explicit turns:** `src/plugins/runtime/runtime-acp.runtime.ts:52-68`
  - body: `promptPluginAcp()` calls the gateway `agent` method without `acpTurnSource: "manual_spawn"`, so follow-up prompts into ACP sessions are treated as automatic dispatch and can fail when `acp.dispatch.enabled=false`. Native ACP spawn sets this marker for the same explicit operator/plugin-initiated turn path.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The diff introduces privileged plugin-to-ACP dispatch with a global opt-in instead of caller-scoped authorization.
