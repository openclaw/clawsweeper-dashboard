## Summary

Keep open: the feature is still absent from current main and the PR includes useful real-run proof, but the current head should not merge because it creates an architecture cycle, exposes ACP through a global plugin flag, and omits the explicit ACP turn marker used by the native spawn path.

## What This Changes

The PR adds `api.runtime.acp.spawn()` and `api.runtime.acp.prompt()`, a `plugins.allowAcpSpawn` config/schema key, runtime mocks, and plugin runtime tests.

## Best Possible Solution

Land a revised version only after extracting a leaf ACP runtime contract, scoping ACP authority to the calling plugin, preserving explicit ACP turn semantics, and documenting the operator opt-in.

## Reproduction Assessment

Yes for the review blockers: source inspection plus PR CI reproduce the import cycle, global authorization gate, and missing explicit ACP turn marker. It is not a current-main bug reproduction because this PR proposes a new plugin runtime capability.

## Solution Assessment

No, not yet. Routing plugin ACP prompts through the gateway delivery path is the right direction, but the public API should not merge until the type boundary, authorization scope, and ACP dispatch policy match existing runtime contracts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Break the ACP runtime type import cycle:** `src/plugins/runtime/types-core.ts:1`
  - body: Importing `SpawnAcp*` types from `../../agents/acp-spawn.js` pulls the public plugin runtime type surface back into the agents graph, and the PR head's architecture job now fails with a Madge cycle ending at this file. Extract a leaf ACP runtime contract or DTOs instead of importing agent implementation types here.
  - confidence: 0.94
- **[P1] Scope ACP permission to the calling plugin:** `src/config/types.plugins.ts:70`
  - body: The new `plugins.allowAcpSpawn` flag is global, so enabling ACP for one trusted plugin gives every loaded plugin access to `api.runtime.acp.spawn()` and `prompt()`. This should mirror the existing `plugins.entries.<id>` plus plugin runtime identity model before exposing host-side ACP dispatch.
  - confidence: 0.91
- **[P2] Mark plugin ACP prompts as explicit turns:** `src/plugins/runtime/runtime-acp.runtime.ts:52-68`
  - body: `promptPluginAcp()` calls the gateway `agent` method without `acpTurnSource: "manual_spawn"`, so follow-up prompts are handled as normal ACP dispatch and can fail when `acp.dispatch.enabled=false`. Native ACP spawn sets this marker for explicit operator-initiated turns; plugin prompts need the same semantics if maintainers accept this surface.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The diff introduces a privileged plugin-to-ACP dispatch surface but gates it with a global plugin flag instead of caller-scoped authorization.

Concerns:

- **[high] Global ACP gate grants all plugins spawn access:** `src/plugins/runtime/runtime-acp.runtime.ts:26`
  - body: With `plugins.allowAcpSpawn: true`, any loaded plugin can spawn or continue ACP agent sessions with channel delivery. Since ACP can launch host-side coding harnesses, the permission should be tied to the calling plugin identity or another explicit per-plugin trust contract.
  - confidence: 0.91
