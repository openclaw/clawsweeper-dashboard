## Summary

Keep open: current main still lacks the proposed ACP plugin runtime API, and the use case is plausible, but the PR head still has blocking architecture, authorization, and ACP dispatch-policy issues before merge.

## What This Changes

The branch adds an injected `api.runtime.acp` namespace with `spawn` and `prompt`, a global `plugins.allowAcpSpawn` config flag, schema/type/mock updates, and plugin runtime tests.

## Best Possible Solution

Land a rebased version only after extracting a leaf ACP runtime contract, gating ACP access by calling plugin identity, preserving explicit ACP turn semantics, and documenting the operator opt-in.

## Reproduction Assessment

Yes for the review blockers: source inspection and live PR status show the architecture boundary issue, global authorization gate, and missing explicit ACP turn marker. As a feature PR, there is no current-main user bug reproduction requirement.

## Solution Assessment

No, not yet. Routing plugin ACP prompts through the gateway delivery path is the right direction, but the public API should not merge until the type boundary, authorization scope, and ACP dispatch policy match existing runtime contracts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Break the ACP runtime type import cycle:** `src/plugins/runtime/types-core.ts:1`
  - body: Importing `SpawnAcp*` types from `../../agents/acp-spawn.js` pulls the public plugin runtime type surface back into the ACP/agents implementation graph, and the PR head still has a failed runtime topology architecture check. Extract a leaf contract or define runtime DTOs that do not import agent implementation modules.
  - confidence: 0.93
- **[P1] Scope ACP permission to the calling plugin:** `src/config/types.plugins.ts:70`
  - body: The new `plugins.allowAcpSpawn` flag is global, so enabling ACP for one trusted plugin gives every loaded third-party plugin access to `api.runtime.acp.spawn()` and `prompt()`. Mirror the existing `plugins.entries.<id>` plus plugin runtime identity model before exposing host-side agent dispatch.
  - confidence: 0.9
- **[P2] Mark plugin ACP prompts as explicit turns:** `src/plugins/runtime/runtime-acp.runtime.ts:52-68`
  - body: `promptPluginAcp()` calls the gateway `agent` method without `acpTurnSource: "manual_spawn"`, so follow-up prompts into ACP sessions are treated as automatic dispatch and fail when `acp.dispatch.enabled=false`. Native ACP spawn sets this marker, and plugin prompts should do the same if they are explicit operator/plugin-initiated turns.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The diff exposes privileged plugin-to-ACP dispatch; the concrete concern is that authorization is global rather than scoped to the calling plugin.

Concerns:

- **[high] Global ACP gate grants all plugins spawn access:** `src/config/types.plugins.ts:70`
  - body: With `plugins.allowAcpSpawn: true`, any loaded plugin can spawn or continue ACP-backed coding sessions with channel delivery. Since ACP can launch host-side coding harnesses, this permission should be tied to the calling plugin identity or an equivalent per-plugin trust contract.
  - confidence: 0.9
