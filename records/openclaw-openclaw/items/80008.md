## Summary

Keep open. The PR targets a real plugin ACP channel-delivery gap that the earlier issue discussion did not fully settle, and the copied before/after logs are useful proof, but the current patch still introduces a privileged host-side ACP surface with a global plugin flag, fails architecture CI through a plugin runtime type import cycle, and omits the explicit ACP turn marker used by native ACP spawn.

## What This Changes

The PR adds `api.runtime.acp.spawn()` and `api.runtime.acp.prompt()`, a `plugins.allowAcpSpawn` config/schema key, runtime wiring, mocks, and tests for plugin ACP dispatch with channel delivery.

## Best Possible Solution

Land a revised API only after ACP DTOs live in a leaf contract, ACP access is authorized per calling plugin, plugin prompts preserve explicit-turn semantics, and public plugin runtime docs explain the operator opt-in and security model.

## Reproduction Assessment

Yes for the review blockers: PR CI and source inspection show the import-cycle failure, and code inspection shows the global gate and missing `acpTurnSource` marker. There is no current-main bug reproduction requirement because this is a feature PR.

## Solution Assessment

No, not yet. Routing ACP prompts through the gateway delivery path is a plausible direction, but the public runtime API should not merge until the type boundary, caller-scoped authorization, explicit ACP turn policy, and docs are aligned with existing runtime contracts.

Full review comments:

- **[P1] Extract ACP runtime types into a leaf contract:** `src/plugins/runtime/types-core.ts:1`
  - body: Importing `SpawnAcp*` types from `../../agents/acp-spawn.js` pulls the public plugin runtime type surface into the ACP/agents implementation graph, and the current PR head fails the architecture job with a Madge cycle ending at `src/plugins/runtime/types-core.ts`. Define the runtime DTOs in a leaf contract or another acyclic public type module before exposing them here.
  - confidence: 0.94
- **[P1] Scope ACP access to the calling plugin:** `src/config/types.plugins.ts:70`
  - body: The new `plugins.allowAcpSpawn` flag is global, so enabling ACP for one trusted plugin gives every loaded plugin access to host-side ACP spawn and prompt. This should mirror the existing `plugins.entries.<id>` plus plugin runtime identity model before exposing a privileged agent-dispatch surface.
  - confidence: 0.91
- **[P2] Mark plugin ACP prompts as explicit turns:** `src/plugins/runtime/runtime-acp.runtime.ts:52-68`
  - body: `promptPluginAcp()` sends the gateway `agent` request without `acpTurnSource: "manual_spawn"`, so follow-up prompts into ACP sessions use automatic dispatch policy and fail when `acp.dispatch.enabled=false`. Native ACP spawn sets this marker, and plugin-initiated ACP prompts should preserve the same explicit-turn semantics.
  - confidence: 0.89
- **[P3] Document the ACP plugin runtime contract:** `src/plugins/runtime/types-core.ts:314-326`
  - body: This adds a public `api.runtime.acp` namespace and a new operator config key, but the plugin runtime docs do not describe the methods, opt-in setting, or host-side privilege warning. Add the public documentation before shipping this plugin-facing feature.
  - confidence: 0.79

## Security Review
