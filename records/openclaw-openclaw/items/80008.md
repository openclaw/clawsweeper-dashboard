## Summary

Keep open for maintainer API/security review. The feature is useful and the PR body includes real runtime proof, but the current patch exposes privileged ACP dispatch through a global plugin flag, still pulls agent implementation types into the public plugin runtime type surface, and omits the explicit ACP turn marker used by native ACP spawn.

## What This Changes

The PR adds `api.runtime.acp.spawn()` and `api.runtime.acp.prompt()`, a `plugins.allowAcpSpawn` config/schema key, plugin runtime mocks, and runtime tests for gated ACP dispatch from plugins.

## Best Possible Solution

Land a rebased version only after extracting a leaf ACP runtime contract, gating ACP access by calling plugin identity, preserving explicit ACP turn semantics, and documenting the operator opt-in.

## Reproduction Assessment

Yes for the review blockers: the PR diff and current-main source show the global authorization gate and missing explicit ACP turn marker, and the PR head has a failing architecture check. This is not a current-main bug reproduction because the PR is adding a new feature.

## Solution Assessment

No, not as currently proposed. Routing plugin ACP prompts through the gateway delivery path is the right direction, but the public API should not merge until the type boundary, authorization scope, and ACP dispatch policy match existing runtime contracts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Break the ACP runtime type import cycle:** `src/plugins/runtime/types-core.ts:1`
  - body: Importing `SpawnAcp*` types from `../../agents/acp-spawn.js` pulls the public plugin runtime type surface back into the agents/ACP implementation graph, and the PR head's architecture job is failing. Move these DTOs to a leaf contract or define narrow runtime-facing types so `types-core.ts` stays acyclic.
  - confidence: 0.93
- **[P1] Scope ACP access to the calling plugin:** `src/config/types.plugins.ts:70`
  - body: `plugins.allowAcpSpawn` is root-wide, so enabling ACP for one trusted plugin exposes host-side `api.runtime.acp.spawn()` and `prompt()` to every loaded plugin. Mirror the existing `plugins.entries.<id>` trust model plus plugin runtime identity before exposing channel-delivered agent dispatch.
  - confidence: 0.9
- **[P2] Mark plugin ACP prompts as explicit turns:** `src/plugins/runtime/runtime-acp.runtime.ts:52-68`
  - body: `promptPluginAcp()` calls the gateway `agent` method without `acpTurnSource: "manual_spawn"`, so plugin-initiated follow-up prompts take the automatic dispatch policy path and can fail when `acp.dispatch.enabled=false`. Native ACP spawn sets this marker for the same explicit-turn path.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The patch introduces a privileged plugin-to-ACP dispatch surface with a global opt-in instead of caller-scoped authorization.

Concerns:

- **[high] Global ACP gate grants all plugins spawn access:** `src/config/types.plugins.ts:70`
  - body: With `plugins.allowAcpSpawn: true`, any loaded plugin can spawn or continue ACP agent sessions with channel delivery. Since ACP can launch host-side coding harnesses, permission should be tied to the calling plugin identity or another explicit per-plugin trust contract.
  - confidence: 0.9
