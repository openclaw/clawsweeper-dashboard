## Summary

Keep open: the feature addresses a real plugin delivery gap and includes live log proof, but the current patch exposes privileged ACP dispatch through a global plugin flag and misses the explicit ACP turn marker needed by current dispatch policy.

## What This Changes

The PR adds `api.runtime.acp.spawn()` and `api.runtime.acp.prompt()` to plugin runtime, a `plugins.allowAcpSpawn` config/schema key, runtime tests, and mock typing.

## Best Possible Solution

Expose ACP channel delivery only behind caller-scoped plugin authorization, preserve explicit ACP turn semantics for prompt calls, and document the operator opt-in and security model.

## Reproduction Assessment

Yes for the review blockers: source inspection of the PR head shows a global ACP gate and a gateway prompt call without the explicit ACP turn marker. The end-user feature itself is not a bug reproduction, but the PR body provides real downstream logs for the delivery gap.

## Solution Assessment

No, not yet. Routing through the gateway delivery path is the right direction, but the permission boundary and ACP policy marker need to be fixed before this is the maintainable core API.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Scope ACP permission to the calling plugin:** `src/config/types.plugins.ts:70`
  - body: The new `plugins.allowAcpSpawn` flag is global, so enabling ACP for one trusted plugin enables every loaded plugin to call `api.runtime.acp.spawn()` and `prompt()`. Current sensitive runtime policies are scoped through `plugins.entries.<id>` and runtime plugin identity; gate ACP the same way before exposing host-side agent dispatch.
  - confidence: 0.88
- **[P2] Mark plugin ACP prompts as explicit turns:** `src/plugins/runtime/runtime-acp.runtime.ts:52-67`
  - body: `promptPluginAcp()` sends a gateway `agent` request without `acpTurnSource: "manual_spawn"`, so prompts into existing ACP sessions are treated like automatic dispatch and will fail when `acp.dispatch.enabled=false`. Native ACP spawn sets this marker, and current policy keeps explicit turns available when only automatic dispatch is disabled.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff adds a privileged plugin-to-ACP dispatch surface, but the authorization gate is global rather than scoped to the calling plugin.

Concerns:

- **[medium] Global ACP gate grants all plugins dispatch access:** `src/plugins/runtime/runtime-acp.runtime.ts:26`
  - body: With `plugins.allowAcpSpawn: true`, any loaded plugin can spawn or prompt host-side ACP agent turns with channel delivery; the safer boundary is a per-plugin policy under `plugins.entries.<id>` using the existing runtime plugin identity scope.
  - confidence: 0.88
