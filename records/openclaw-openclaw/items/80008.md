## Summary

Keep this PR open: the ACP channel-delivery gap is real and the proposed runtime seam is plausible, but the patch has a security/API gating concern plus source-level compile and test blockers.

## What This Changes

The PR adds `api.runtime.acp.spawn()` and `api.runtime.acp.prompt()` to plugin runtime, guarded by a new `plugins.allowAcpSpawn` config flag, with config types/schema, mocks, and runtime tests.

## Best Possible Solution

Keep the ACP runtime seam if maintainers want it, but make the privilege policy caller-scoped, fix the compile/test blockers, and document the new plugin runtime and config contract.

## Reproduction Assessment

Yes for the review blockers: the PR diff contains two `createRuntimeAcp()` implementations, and the prompt tests mock `loadConfig()` while the helper calls `getRuntimeConfig()`. Source inspection is enough to identify these failures without mutating the checkout.

## Solution Assessment

No, not yet. Routing ACP prompts through the gateway delivery path is the right direction, but the patch needs a scoped privilege model plus mechanical compile and test fixes before it is maintainable.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Scope ACP spawn permission to the calling plugin:** `src/config/types.plugins.ts:70`
  - body: The new top-level `plugins.allowAcpSpawn` flag is checked without plugin identity, so enabling ACP for one trusted plugin enables every loaded third-party plugin to dispatch agent turns with channel delivery. Existing sensitive runtime policies are under `plugins.entries.<id>` and runtime scope already carries `pluginId`; gate this per calling plugin before exposing agent spawning.
  - confidence: 0.86
- **[P2] Remove the duplicate ACP runtime factory:** `src/plugins/runtime/index.ts:161-167`
  - body: This hunk adds a second `createRuntimeAcp()` implementation in the same module after adding the same helper near the lazy loader. TypeScript will reject the duplicate implementation, so the PR cannot pass typecheck or build until one copy is removed.
  - confidence: 0.98
- **[P2] Mock getRuntimeConfig in ACP prompt tests:** `src/plugins/runtime/index.test.ts:534-535`
  - body: The new ACP helper imports and calls `getRuntimeConfig()`, but these two tests only mock `loadConfig()`. After the duplicate factory is fixed, these cases will still see the real/default config and fail at the allowAcpSpawn gate before reaching the gateway assertions.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The diff introduces a privileged plugin-to-agent dispatch surface and currently gates it with a global flag rather than caller-scoped plugin policy.

Concerns:

- **[medium] Global ACP spawn gate grants authority to all plugins:** `src/config/types.plugins.ts:70`
  - body: `plugins.allowAcpSpawn` is a top-level flag, so once enabled any loaded plugin can call the new ACP helpers; existing sensitive plugin runtime policies use `plugins.entries.<id>` with scoped plugin identity, which would be a safer boundary for this new agent-spawning capability.
  - confidence: 0.86
