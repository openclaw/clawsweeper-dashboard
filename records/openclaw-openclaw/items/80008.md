## Summary

Keep open. Current main still lacks the channel-delivering `api.runtime.acp` surface this PR targets, but the patch should not merge yet because it exposes privileged ACP dispatch through a global plugin gate, keeps a plugin-runtime type back-edge into agent implementation code, and omits the explicit ACP turn marker used by native ACP spawn.

## What This Changes

The PR adds `api.runtime.acp.spawn()` and `api.runtime.acp.prompt()`, a `plugins.allowAcpSpawn` config/schema key, runtime mocks, and plugin runtime tests for ACP-backed channel delivery from plugins.

## Best Possible Solution

Rework this as a caller-scoped ACP runtime contract with per-plugin policy, leaf ACP DTO types, explicit ACP turn semantics, and documentation for the operator opt-in.

## Reproduction Assessment

Yes for the review blockers: source inspection of the PR head and current-main code shows the global gate, missing ACP turn marker, and type-boundary issue, and PR CI reports an architecture failure. There is no current-main bug reproduction requirement because this is a feature/API PR.

## Solution Assessment

No, not yet. Routing through the gateway delivery path is the right direction, but the public API should use caller-scoped trust, acyclic leaf contracts, and native explicit-turn semantics before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Break the ACP runtime type import cycle:** `src/plugins/runtime/types-core.ts:1`
  - body: `PluginRuntimeCore` imports `SpawnAcp*` types from `../../agents/acp-spawn.js`, pulling the public plugin runtime type surface back into the agent implementation graph; the PR head also has a failing runtime-topology architecture check. Move the DTOs to a leaf contract module before exposing this API.
  - confidence: 0.92
- **[P1] Scope ACP permission to the calling plugin:** `src/config/types.plugins.ts:67-72`
  - body: The new `plugins.allowAcpSpawn` flag is process-wide, so enabling ACP for one trusted plugin gives every loaded plugin access to host-side ACP spawn/prompt with channel delivery. Use the existing plugin identity scope and `plugins.entries.<id>` policy pattern, or an equivalent caller-scoped trust contract.
  - confidence: 0.9
- **[P2] Mark plugin ACP prompts as explicit turns:** `src/plugins/runtime/runtime-acp.runtime.ts:52-68`
  - body: `promptPluginAcp()` sends a gateway `agent` request into an existing ACP session without `acpTurnSource: "manual_spawn"`, so it follows the automatic dispatch policy and can fail when `acp.dispatch.enabled=false`; native ACP spawn sets this marker for explicit turns.
  - confidence: 0.88
