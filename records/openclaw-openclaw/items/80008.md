## Summary

Keep open: current main still lacks the proposed channel-delivering ACP runtime namespace, and the PR remains useful but not mergeable because the head still fails architecture checks and exposes privileged ACP dispatch through a global plugin flag.

## What This Changes

The PR adds `api.runtime.acp.spawn()` and `api.runtime.acp.prompt()` with a `plugins.allowAcpSpawn` config gate, schema/types, mocks, and plugin runtime tests.

## Best Possible Solution

Land a rebased version only after extracting leaf ACP runtime DTOs, enforcing ACP access through caller-scoped plugin trust, preserving explicit ACP turn semantics, and documenting the operator opt-in.

## Reproduction Assessment

Yes for the PR blockers: source inspection and PR CI reproduce the import-cycle failure, global authorization gate, and missing ACP turn marker. As a feature PR, there is no current-main bug reproduction requirement.

## Solution Assessment

No, not as written. Routing plugin ACP prompts through the gateway delivery path is the right direction, but the public API should not merge until the type boundary, authorization scope, and ACP dispatch policy match existing runtime contracts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Break the ACP runtime type import cycle:** `src/plugins/runtime/types-core.ts:1`
  - body: Importing `SpawnAcp*` from `../../agents/acp-spawn.js` makes the public plugin runtime type surface point back into agent implementation code, and the PR head's architecture job reports a Madge cycle ending at this file. Move the DTOs to a leaf contract or define plugin runtime params without importing the agent implementation module.
  - confidence: 0.94
- **[P1] Scope ACP permission to the calling plugin:** `src/config/types.plugins.ts:70`
  - body: The new `plugins.allowAcpSpawn` flag is global, so enabling ACP for one trusted plugin gives every loaded third-party plugin access to ACP spawn and prompt. Mirror the existing `plugins.entries.<id>` trust model and enforce it with the plugin runtime caller identity before exposing host-side agent dispatch.
  - confidence: 0.91
- **[P2] Mark plugin ACP prompts as explicit turns:** `src/plugins/runtime/runtime-acp.runtime.ts:52-68`
  - body: `promptPluginAcp()` calls the gateway `agent` method without `acpTurnSource: "manual_spawn"`, so follow-up prompts into ACP sessions use dispatch policy and can be rejected when ACP dispatch is paused. Native ACP spawn sets this marker for explicit turns, and plugin prompt should do the same if maintainers accept this API.
  - confidence: 0.89

## Security Review

Status: needs_attention

Summary: The patch introduces a privileged plugin-to-ACP dispatch surface with a global opt-in instead of caller-scoped authorization.

Concerns:

- **[high] Global ACP gate grants all plugins spawn access:** `src/config/types.plugins.ts:70`
  - body: With `plugins.allowAcpSpawn: true`, any loaded plugin can spawn or continue ACP-backed coding sessions with channel delivery. Since ACP can launch host-side coding harnesses, this permission should be tied to the calling plugin identity or an equivalent per-plugin trust contract.
  - confidence: 0.91
