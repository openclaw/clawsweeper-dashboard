## Summary

Keep open: the delivery gap is real and current main still lacks an injected `api.runtime.acp` surface, but this patch is not merge-ready because it introduces a privileged global ACP plugin gate, creates an architecture import cycle, and misses the explicit ACP turn marker used by native ACP spawn.

## What This Changes

The branch adds plugin runtime `acp.spawn` and `acp.prompt`, a `plugins.allowAcpSpawn` config gate, schema/type/mock updates, and runtime tests.

## Best Possible Solution

Land a rebased version only after extracting a leaf ACP runtime contract, gating ACP spawn and prompt per calling plugin identity, preserving explicit ACP turn semantics, and documenting the operator opt-in.

## Reproduction Assessment

Yes for the review blockers: source inspection plus PR CI show the import cycle, global authorization gate, and missing explicit ACP turn marker. As a feature PR, there is no current-main bug reproduction requirement.

## Solution Assessment

No, not yet. Routing plugin ACP prompts through the gateway delivery path is the right direction, but the public API should wait for a caller-scoped trust model and a leaf type boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Break the ACP runtime type cycle:** `src/plugins/runtime/types-core.ts:1`
  - body: Importing `SpawnAcp*` from `../../agents/acp-spawn.js` puts the public plugin runtime type surface back into the ACP/agents graph, and PR CI now reports a Madge cycle ending at this file. Define a leaf ACP runtime DTO/contract that both sides can import without pulling in agent implementation modules.
  - confidence: 0.93
- **[P1] Scope ACP access to the calling plugin:** `src/config/types.plugins.ts:70`
  - body: The new `plugins.allowAcpSpawn` flag is top-level, so enabling ACP for one trusted plugin gives every loaded plugin access to host-side ACP spawn and prompt with channel delivery. Use the existing runtime plugin identity plus `plugins.entries.<id>` policy pattern before exposing this capability.
  - confidence: 0.91
- **[P2] Mark plugin ACP prompts as explicit ACP turns:** `src/plugins/runtime/runtime-acp.runtime.ts:52-68`
  - body: `promptPluginAcp()` sends the gateway `agent` request without `acpTurnSource: "manual_spawn"`, so ACP follow-up prompts are treated as automatic dispatch and fail when `acp.dispatch.enabled=false`. Native ACP spawn sets this marker, and plugin-initiated ACP prompts should preserve the same explicit-turn policy.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The patch introduces a privileged plugin-to-ACP dispatch surface, but the authorization gate is global instead of tied to the calling plugin.
