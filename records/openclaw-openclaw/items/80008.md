## Summary

Keep open: the proposed gateway-delivered ACP runtime API addresses a real plugin gap, but the current patch still has blocking API/security and architecture issues before it can merge.

## What This Changes

The PR adds `api.runtime.acp.spawn()` and `api.runtime.acp.prompt()`, a `plugins.allowAcpSpawn` config key, schema/types, runtime mocks, and plugin runtime tests.

## Best Possible Solution

Land a rebased version only after extracting a leaf ACP runtime contract, gating ACP access by calling plugin identity, preserving explicit ACP turn semantics, and documenting the operator opt-in.

## Reproduction Assessment

Yes for the review blockers: PR CI and source inspection show the import cycle, global ACP authorization, and missing explicit ACP turn marker. As a feature PR, there is no current-main bug reproduction requirement.

## Solution Assessment

No. Routing through the gateway delivery path is the right direction, but the public API should not merge until the type boundary, authorization scope, and ACP dispatch policy are aligned with existing runtime contracts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Break the ACP runtime type import cycle:** `src/plugins/runtime/types-core.ts:1`
  - body: Importing `SpawnAcp*` types from `../../agents/acp-spawn.js` pulls the public plugin runtime type surface back into the ACP/agents graph, and the PR head's architecture lane fails with a Madge cycle ending through this file. Extract a leaf contract or define runtime DTOs that do not import agent implementation modules.
  - confidence: 0.94
- **[P1] Scope ACP permission to the calling plugin:** `src/config/types.plugins.ts:70`
  - body: The new `plugins.allowAcpSpawn` flag is global, so enabling ACP for one trusted plugin gives every loaded plugin access to host-side ACP spawn and prompt. Use the existing plugin runtime identity plus `entries.<id>` policy model before exposing this privileged dispatch surface.
  - confidence: 0.92
- **[P2] Mark plugin ACP prompts as explicit turns:** `src/plugins/runtime/runtime-acp.runtime.ts:52-68`
  - body: `promptPluginAcp()` calls the gateway `agent` method without `acpTurnSource: "manual_spawn"`, so follow-up prompts into ACP sessions take the automatic dispatch policy path and fail when `acp.dispatch.enabled=false`. Native ACP spawn sets this marker for explicit operator/plugin turns.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The diff introduces a privileged plugin-to-ACP dispatch surface with a global opt-in rather than caller-scoped authorization.
