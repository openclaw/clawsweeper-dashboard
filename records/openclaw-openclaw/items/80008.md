## Summary

Keep open: current main still lacks the requested plugin ACP runtime surface, and the PR remains a meaningful feature proposal, but the current patch still has blocking architecture, ACP policy, and plugin trust-boundary issues.

## What This Changes

The PR adds `api.runtime.acp.spawn()` and `api.runtime.acp.prompt()`, a global `plugins.allowAcpSpawn` config key, schema/types, runtime mocks, and plugin runtime tests.

## Best Possible Solution

Land a revised version only after moving ACP DTOs to a leaf contract, authorizing ACP access by calling plugin identity, preserving explicit ACP turn semantics, and documenting the operator opt-in.

## Reproduction Assessment

Yes for the review blockers: PR head source and CI show the import cycle, global authorization gate, and missing explicit ACP turn marker. As a feature PR, there is no current-main bug reproduction requirement for the requested new API.

## Solution Assessment

No, not yet. Routing plugin ACP prompts through the gateway delivery path is plausible, but this implementation should not merge until the type boundary, caller-scoped authorization, and ACP dispatch policy match existing runtime contracts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Break the ACP runtime type import cycle:** `src/plugins/runtime/types-core.ts:1`
  - body: Importing `SpawnAcp*` types from `../../agents/acp-spawn.js` pulls the public plugin runtime type surface back into the ACP/agents graph. The PR head's architecture job reports a Madge cycle ending through this file, so extract a leaf contract or define runtime DTOs that do not import agent implementation modules.
  - confidence: 0.93
- **[P1] Scope ACP permission to the calling plugin:** `src/config/types.plugins.ts:70`
  - body: `plugins.allowAcpSpawn` is global, so enabling ACP for one trusted plugin gives every loaded third-party plugin access to `api.runtime.acp.spawn()` and `prompt()`. This should follow the existing `plugins.entries.<id>` plus plugin runtime identity model before exposing host-side agent dispatch.
  - confidence: 0.9
- **[P2] Mark plugin ACP prompts as explicit turns:** `src/plugins/runtime/runtime-acp.runtime.ts:52-68`
  - body: `promptPluginAcp()` calls the gateway `agent` method without `acpTurnSource: "manual_spawn"`, so follow-up prompts into ACP sessions are treated as automatic dispatch and fail when `acp.dispatch.enabled=false`. Native ACP spawn sets this marker, and plugin prompts should do the same if they are explicit operator/plugin-initiated turns.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The patch exposes privileged plugin-to-ACP dispatch behind a global opt-in instead of caller-scoped authorization.

Concerns:

- **[high] Global ACP gate grants all plugins spawn access:** `src/config/types.plugins.ts:70`
  - body: With `plugins.allowAcpSpawn: true`, any loaded plugin can spawn or continue ACP agent sessions with channel delivery. Since ACP can launch host-side coding harnesses, the permission should be tied to the calling plugin identity or another explicit per-plugin trust contract.
  - confidence: 0.9
