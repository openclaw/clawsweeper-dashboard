## Summary

Keep open: the underlying ACP channel-delivery API gap is still plausible on current main, but this PR should not merge until the plugin ACP trust model and two source-level blockers are resolved.

## What This Changes

The PR adds a new `api.runtime.acp` plugin runtime namespace, `plugins.allowAcpSpawn` config/schema support, test-helper mocks, and runtime tests for ACP spawn/prompt delivery.

## Best Possible Solution

Land a revised API only after extracting leaf ACP runtime DTOs, gating ACP access by caller identity or an equivalent per-plugin capability, preserving explicit ACP turn semantics, and documenting the operator opt-in.

## Reproduction Assessment

Yes for the review blockers: PR-head source shows the global gate, agent-implementation type import, and missing ACP turn marker, and live PR checks show the architecture lane failing. There is no current-main bug reproduction requirement because this is a feature/API PR.

## Solution Assessment

No, not yet. Routing ACP prompts through the gateway delivery path is the right direction, but the current patch needs a leaf type boundary, caller-scoped authorization, and the manual-spawn turn marker before it is maintainable.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Break the ACP runtime type import cycle:** `src/plugins/runtime/types-core.ts:1`
  - body: `types-core.ts` is the public plugin runtime type surface, but this import reaches into `../../agents/acp-spawn.js`. That pulls the plugin runtime back into the agents graph and the PR's architecture lane is failing; define leaf ACP runtime DTOs or move the shared contract out of the agent implementation module.
  - confidence: 0.93
- **[P1] Scope ACP permission to the calling plugin:** `src/config/types.plugins.ts:70`
  - body: `plugins.allowAcpSpawn` is global, so enabling ACP for one trusted plugin gives every loaded plugin access to `api.runtime.acp.spawn()` and `prompt()`. This is a host-side agent dispatch surface and should follow the existing `plugins.entries.<id>` plus plugin runtime identity model before it is exposed.
  - confidence: 0.9
- **[P2] Mark plugin ACP prompts as explicit turns:** `src/plugins/runtime/runtime-acp.runtime.ts:52-67`
  - body: `promptPluginAcp()` sends a gateway `agent` request without `acpTurnSource: "manual_spawn"`. Native ACP spawn sets that marker, and `agent-command` uses it to allow explicit turns when dispatch is paused; plugin follow-ups should preserve the same semantics.
  - confidence: 0.88

## Security Review
