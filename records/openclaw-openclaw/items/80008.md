## Best Possible Solution

Land a revised ACP runtime surface only after ACP DTOs live in a leaf contract, ACP permission is authorized by calling plugin identity, plugin prompts preserve explicit ACP turn semantics, and the operator opt-in is documented.

## Solution Assessment

No, not yet. Routing plugin ACP prompts through the gateway delivery path is plausible, but this implementation should not merge until the type boundary, caller-scoped authorization, and ACP dispatch policy match existing runtime contracts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Move ACP DTOs out of the agent implementation graph:** `src/plugins/runtime/types-core.ts:1`
  - body: `PluginRuntimeCore` is a plugin-facing contract, but this import pulls ACP types from `../../agents/acp-spawn.js`. Current main has been splitting runtime contract leaf types to avoid these back-edges, and the earlier review reported a Madge cycle on this path; define the ACP DTOs in a leaf contract instead of importing the agent implementation module.
  - confidence: 0.9
- **[P1] Scope ACP permission to the calling plugin:** `src/config/types.plugins.ts:70`
  - body: `plugins.allowAcpSpawn` is process-wide, so enabling ACP for one trusted plugin gives every loaded plugin access to ACP spawn/prompt. Existing sensitive plugin runtime policies use `plugins.entries.<id>` plus runtime plugin identity; ACP dispatch should follow that pattern before exposing host-side agent spawning.
  - confidence: 0.92
- **[P2] Mark plugin ACP prompts as explicit turns:** `src/plugins/runtime/runtime-acp.runtime.ts:52-67`
  - body: `promptPluginAcp()` calls the gateway `agent` method without `acpTurnSource: "manual_spawn"`, so follow-up prompts into ACP sessions are treated like automatic dispatch and can fail when `acp.dispatch.enabled=false`. Native ACP spawn sets this marker, and plugin-initiated follow-ups should preserve the same explicit-turn semantics.
  - confidence: 0.88
