## Summary

Keep open. Current main still has the active-run presence gap: typing keepalive ticks re-send channel typing but do not refresh the run-side TTL, so a quiet healthy run can still lose presence before completion; the remaining work is already represented by the open targeted PR #71828.

## What This Changes

The issue requests a shared active-run presence lifecycle that keeps already-configured typing, reaction, or progress surfaces alive until the run has ended and reply dispatch is idle.

## Best Possible Solution

Keep this issue open and review, land, or replace PR #71828 with an equivalent narrow internal fix that refreshes the run-side typing TTL during active keepalive ticks, preserves stale-presence cleanup, respects final-only/progress configuration, and adds the active-run-beyond-TTL regression test.

## Reproduction Assessment

Yes. A `createTypingController` configured with a 6 second keepalive interval and a 10 second TTL will tick at 6 seconds without refreshing the TTL, then the TTL cleanup path can stop typing at 10 seconds while the run is still active.

## Solution Assessment

Yes for the proposed Phase 1 direction. A narrow internal TTL refresh or active-run lease tied to existing keepalive behavior is the maintainable fix because it preserves current channel/config semantics and avoids introducing default compaction or status-message disclosure.
