## Summary

Current main still has the active-run typing liveness gap: keepalive ticks call the channel typing action but do not refresh the run-side TTL, so a quiet active run can still clean up typing before completion. The issue should stay open because PR #71828 is already open with the narrow missing runtime change and regression test.

## What This Changes

The issue requests lifecycle-owned active-run presence so configured typing, reaction, or existing progress surfaces stay visible until the run has ended and reply dispatch is idle.

## Best Possible Solution

Keep the issue open and review, land, or replace PR #71828 with an equivalent narrow fix that refreshes the internal typing TTL on active keepalive ticks, preserves stale-presence cleanup, respects final-only/progress configuration, and adds the active-run-beyond-TTL regression test.

## Reproduction Assessment

Yes. By code inspection, a controller with typingIntervalSeconds 6 and typingTtlMs 10000 will start the keepalive loop, tick at 6 seconds without refreshing the TTL, and then call cleanup at 10 seconds while the run is still active; the open PR adds exactly that failing regression shape.

## Solution Assessment

Yes for Phase 1. A narrow internal TTL refresh tied to existing keepalive ticks is the maintainable first fix because it preserves current typing/progress configuration and avoids new default compaction or status-message disclosure surfaces.
