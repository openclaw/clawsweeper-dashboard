## Summary

Current main and the v2026.5.2 release now keep configured typing/presence alive through long active runs by refreshing the typing TTL on keepalive ticks and stopping only after run completion plus dispatch idle, while preserving the existing final-only/progress constraints.

## What This Changes

Keep already-configured active-run presence surfaces alive until the run has completed and reply dispatch is idle.

## Best Possible Solution

Close this as shipped, keep the internal lifecycle fix, and leave broader operator-visibility or SDK-surface proposals to the separate open follow-ups such as #47591 and #48355.

## Reproduction Assessment

No current-main failing reproduction remains. The original 6 second keepalive / 10 second TTL source-level scenario is now covered by `typing-persistence.test.ts` and current source refreshes the TTL before cleanup can fire during an active run.

## Solution Assessment

Yes. The current implementation is the narrow Phase 1 fix: it refreshes existing typing/presence lifecycle signals, preserves stale-presence cleanup, and avoids changing default compaction disclosure or final-only delivery semantics.
