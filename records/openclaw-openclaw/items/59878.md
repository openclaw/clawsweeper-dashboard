## Summary

Keep open. Current main has fixed adjacent diagnostic recovery and active-restart marking, but the exact stale-row path still remains: `sessions.abort` can return `no-active-run` without healing the persisted `running` row, and restart recovery still skips running rows that were never marked `abortedLastRun`.

## What This Changes

The issue reports stale session lanes that remain persisted as `running` after a dead run and are not cleared by `sessions.abort` or gateway restart.

## Best Possible Solution

Add conservative stale-row reconciliation for `sessions.abort` no-active outcomes and bounded startup handling for markerless stale running rows, reusing active run and lane facts to avoid clearing live work.

## Reproduction Assessment

Yes, source-level with high confidence. Seed a persisted session row with `status: "running"` and no active abort controller, then `sessions.abort` can return `no-active-run` without updating the row, and restart recovery skips the same row unless `abortedLastRun` is already true.

## Solution Assessment

Yes. The narrow maintainable fix is stale persisted-row reconciliation in the abort and restart ownership paths, while preserving the existing diagnostic recovery and active-restart marker behavior.
