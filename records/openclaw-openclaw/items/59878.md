## Summary

Keep open. Current main still leaves the reported stale-state gap: `sessions.abort` can report `no-active-run` without healing a persisted `running` row, and restart recovery remains gated to rows already marked `abortedLastRun`.

## What This Changes

The issue asks OpenClaw to clear stale `running` session state after a dead run through `sessions.abort`, gateway restart reconciliation, and ideally bounded timeout recovery.

## Best Possible Solution

Add a conservative session-owned reconciliation path that heals stale `running` rows on `sessions.abort` and bounded startup recovery only after proving no active in-memory owner exists.

## Reproduction Assessment

Yes. A source-level reproduction is clear: create or load a persisted session entry with `status: "running"` and no active `chatAbortControllers` entry, then call `sessions.abort`; current code returns `no-active-run` and does not update the row, while restart recovery skips the same row unless `abortedLastRun` is already true.

## Solution Assessment

No. Current main’s diagnostic lane reset and marker-gated restart recovery are useful partial fixes, but the narrow maintainable solution is explicit stale-session reconciliation in the session abort/startup ownership paths.
