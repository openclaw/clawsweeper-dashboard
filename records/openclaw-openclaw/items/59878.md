## Summary

Keep open. Current main has partial diagnostic stuck-session recovery, but the specific stale `running` row path still remains: `sessions.abort` can return `no-active-run` without healing persisted session state, and restart recovery still requires `abortedLastRun` before it will inspect a running row.

## What This Changes

The issue asks for stale running session state to be cleared by `sessions.abort`, reconciled on gateway restart, and covered by bounded timeout recovery.

## Best Possible Solution

Add conservative stale-row reconciliation in the `sessions.abort` no-active path and bounded startup recovery for markerless stale running rows, reusing active-run and lane facts to avoid clearing live work.

## Reproduction Assessment

Yes, source-level with high confidence. Seed a persisted session entry with `status: "running"` and no active `chatAbortControllers` entry, then call `sessions.abort`; current main returns `no-active-run` without updating the row, and restart recovery skips the same row unless `abortedLastRun` is already true.

## Solution Assessment

Yes. The narrow maintainable solution is explicit stale-session reconciliation in the abort and restart ownership paths, while leaving the broader model/tool stall policy to the separate canonical recovery issue.
