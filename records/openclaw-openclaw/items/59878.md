## Summary

Keep open. Current main still leaves the manual stale-running recovery gap: `sessions.abort` can return `no-active-run` without healing the persisted session row, and restart recovery still scans only rows already marked `abortedLastRun`; the newer diagnostics recovery work helps adjacent stuck-lane cases but does not satisfy this contract.

## What This Changes

The issue requests recovery for session lanes that remain persisted as running after the underlying run has already died.

## Best Possible Solution

Add conservative session-owned stale-row reconciliation for `sessions.abort` and bounded startup recovery, using current active-run/lane facts to avoid clearing legitimately active work.

## Reproduction Assessment

Yes, source-level with high confidence. Seed or load a persisted session entry with `status: "running"` and no active `chatAbortControllers` entry, then call `sessions.abort`; current main returns `no-active-run` and does not update the row, while restart recovery skips the same row unless `abortedLastRun` is already true.

## Solution Assessment

Yes, the narrow maintainable fix is explicit stale-session reconciliation in the abort and restart ownership paths. Avoid folding this into broader model/tool stall policy unless that becomes necessary to prove no active owner exists.
