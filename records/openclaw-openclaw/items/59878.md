## Summary

Keep open. Current main has partial recovery for active abort controllers, diagnostic-only stale lane release, and restart recovery for lock-marked `abortedLastRun` main sessions, but it still does not satisfy the reported manual recovery contract: `sessions.abort` can return `no-active-run` without updating a persisted `running` session row, and restart recovery explicitly skips ordinary `running` rows that lack the restart-aborted marker.

## What This Changes

The issue asks OpenClaw to recover session lanes and persisted session state when a run dies but the session remains `status: "running"`, including through `sessions.abort`, gateway restart, and a bounded lane timeout.

## Best Possible Solution

Add a conservative stale-run reconciliation path: when `sessions.abort` targets a session whose active run is already gone but whose persisted row is still `running`, mark that row terminal or abort-recoverable, emit a session change, and release only the matching stale session lane after proving no active embedded/reply work owns it. Gateway startup should also reconcile bounded stale `running` rows that cannot have live in-memory owners, with tests for the no-active-run and restart cases.

## Reproduction Assessment

Yes. A source-level reproduction is clear: create or load a persisted session entry with `status: "running"` and no active `chatAbortControllers` entry, then call `sessions.abort`; current code returns `no-active-run` and has no store update path, while restart recovery skips the same row unless `abortedLastRun` is already true.

## Solution Assessment

No, current main is not the best complete solution. The existing recovery mechanisms are useful but partial; the safer fix is a narrow session-owned reconciliation path rather than relying on `/new` or diagnostic-only lane cleanup.
