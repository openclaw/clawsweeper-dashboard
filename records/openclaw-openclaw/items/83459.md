## Summary

Keep open. Current main still has a source-reproducible offline CLI cleanup bug: `openclaw agents delete` falls back from Gateway to a helper that shells out to `trash`, while the official runtime image does not install `trash-cli`; an open linked PR targets this issue, so the issue should remain open until that PR or a replacement lands.

## What This Changes

The issue reports that Docker-based `openclaw agents delete` removes the agent from config but leaves workspace, agent, or session directories behind when the `trash` command is unavailable.

## Best Possible Solution

Land a focused fix that routes offline CLI/reset cleanup through the existing fs-safe Trash helper or equivalent shared behavior, with regression tests and Docker proof for missing `trash` plus preserved shared-workspace retention.

## Reproduction Assessment

Yes. Source inspection shows the offline CLI path invokes a PATH `trash` binary and the default Docker runtime image does not install one, which matches the reporter's Docker steps.

## Solution Assessment

No, adding `trash-cli` to the default image is not the best repository-level fix. The maintainable path is to reuse the existing fs-safe Trash helper so Docker, minimal Linux, CLI, and Gateway cleanup share one deletion contract.
