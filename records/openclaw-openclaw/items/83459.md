Keep open. Current main and the latest tagged release still use a PATH-resolved `trash` command for the offline CLI cleanup path, the official Dockerfile still does not install `trash-cli`, and the only directly linked fix PR was closed unmerged.

## What This Changes

The issue reports that the official Docker image lacks a `trash` command, so `openclaw agents delete` can remove config while leaving agent workspace, agent directory, or session files behind.

## Best Possible Solution

Move the offline CLI cleanup path to the existing safe no-PATH trash behavior or an equivalent fallback, then prove both unique-workspace deletion and shared-workspace retention in Docker.

## Reproduction Assessment

Yes, by high-confidence source inspection: current main shells out to `trash`, and the official Dockerfile package list does not include a provider for that executable. I did not execute the Docker flow in this read-only review.

## Solution Assessment

Yes, the best repair is to reuse the existing safe trash helper or add an equivalent no-PATH fallback in the shared CLI helper rather than only adding `trash-cli` to the default image. The fix should include focused unit coverage and Docker proof for the reported user path.
