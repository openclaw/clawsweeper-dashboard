## Summary

Keep open. Current main and the latest stable tag still have the offline CLI cleanup path depending on a PATH `trash` executable that the official Docker image does not install; the newer parent-directory residual is real source evidence too, but the linked open PR is not a complete clean fix for the original Docker `trash` gap.

## What This Changes

Ensure Docker/CLI agent deletion actually cleans up agent workspace/state files when `trash` is unavailable, and address the later-reported empty parent agent directory residual.

## Best Possible Solution

Implement one shared cleanup helper for CLI and Gateway that works without PATH `trash`, preserves shared-workspace retention, cleans the standard `agents/<id>` parent directory when safe, and proves the unique-workspace Docker flow plus shared-workspace retention.

## Reproduction Assessment

Yes by source inspection: the offline CLI path calls only the `trash` executable, and the official Docker image package list does not install it. I did not run the Docker container in this read-only review.

## Solution Assessment

The suggested Docker dependency would help one image, but the better maintainable fix is a shared fs-safe cleanup path that aligns CLI and Gateway behavior without relying on PATH `trash`; the later parent-directory residual should be handled in the same narrow cleanup surface.
