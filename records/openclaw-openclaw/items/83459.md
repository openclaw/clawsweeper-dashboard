The issue requests that `openclaw agents delete` in the official Docker image physically clean up owned agent files instead of depending on a missing external `trash` binary.

## Best Possible Solution

Land the narrow no-PATH cleanup fix from the linked PR or an equivalent current-main patch, keep shared-workspace retention intact, and handle the parent agent-directory cleanup as a separate scoped decision.

## Reproduction Assessment

Yes by source inspection: the offline CLI cleanup helper still shells out to a PATH `trash` command while the official Docker runtime package list does not include a provider for that command. I did not run a live Docker reproduction in this read-only review.

## Solution Assessment

Yes for the linked PR direction: reusing the in-process fs-safe trash path is narrower and more consistent than adding a Docker-only dependency. The later parent-directory cleanup should stay separate unless maintainers intentionally combine the scopes.
