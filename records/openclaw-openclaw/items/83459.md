## Summary

Keep open. Current main still has an offline CLI deletion path that depends on a PATH-resolved `trash` binary, while the root Dockerfile does not install one by default; the Gateway path already uses the safer fs-safe trash helper, so this is a valid narrow bug rather than stale cleanup.

## What This Changes

The issue asks for `openclaw agents delete` in official Docker images to actually remove or trash agent workspace/state files when the `trash` binary is unavailable.

## Best Possible Solution

Make the offline CLI deletion path reuse the existing safe `movePathToTrash` behavior, then prove unique-workspace cleanup and shared-workspace retention in focused tests and Docker smoke coverage.

## Reproduction Assessment

Yes by source inspection, though not by executed live Docker proof in this read-only review. The offline CLI path calls a missing `trash` binary and the official Dockerfile does not install one by default, matching the reporter's steps when the Gateway delete path is not used.

## Solution Assessment

No, adding `trash-cli` only to the Dockerfile is narrower than the actual bug surface but leaves minimal Linux/offline CLI behavior dependent on PATH. The best fix is to share the existing fs-safe trash helper between CLI and Gateway deletion paths while preserving documented Trash semantics.
