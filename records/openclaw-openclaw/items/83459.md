## Summary

Keep open. Current main and v2026.5.12 still have an offline CLI deletion path that depends on a `trash` binary the root Dockerfile does not install, while the Gateway path already uses the built-in fs-safe trash helper; prior related Linux fallback reports and PRs were closed without merging a fix.

## What This Changes

The issue reports that the official Docker image lacks a trash implementation, so `openclaw agents delete --force` can prune config while leaving agent workspace/state directories behind.

## Best Possible Solution

Make the offline CLI deletion path use the existing safe `movePathToTrash` behavior, then add focused unit coverage plus a Docker smoke that proves unique agent workspace/state cleanup without deleting shared workspaces.

## Reproduction Assessment

Not with executed live proof in this read-only review, but yes by high-confidence source inspection: the non-Gateway CLI path calls a `trash` binary and the official Dockerfile does not install one. The reporter’s Docker steps match that code path when the Gateway is unavailable or not used for deletion.

## Solution Assessment

The suggested Docker dependency would address official images, but the narrower maintainable fix is to reuse the existing fs-safe trash helper so CLI and Gateway deletion share the same no-PATH fallback. That should be paired with Docker proof for both unique-workspace deletion and shared-workspace retention.
