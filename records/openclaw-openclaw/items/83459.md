## Summary

Keep open. Current main still has the reported offline CLI path: `openclaw agents delete` falls back from Gateway to a helper that shells out to `trash`, while the official Docker runtime package list does not install `trash-cli`; the right next step is to review the open closing PR at https://github.com/openclaw/openclaw/pull/84394 rather than closing this issue early.

## What This Changes

The issue reports that official Docker images lack the PATH `trash` binary required by the offline `openclaw agents delete` cleanup path, leaving agent/workspace/session directories after config deletion.

## Best Possible Solution

Land the narrow fs-safe-helper fix in https://github.com/openclaw/openclaw/pull/84394 for the Docker missing-trash failure, then handle the parent-directory cleanup as a separate, focused agent-delete change if maintainers want that behavior.

## Reproduction Assessment

Yes by source inspection: current main's offline CLI delete path calls a PATH-resolved `trash` command, and the Docker runtime package list does not install a provider for that command. I did not run the Docker repro in this read-only review.

## Solution Assessment

Yes, the best available fix direction is the open PR's bounded switch from external `trash` to the existing fs-safe trash helper for the CLI path, with focused tests and Docker no-`trash` proof. Adding `trash-cli` to the image alone would leave non-Docker minimal Linux users and CLI/Gateway behavior divergence behind.
