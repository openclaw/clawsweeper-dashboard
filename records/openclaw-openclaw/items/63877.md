## Summary

Keep open. Current main still rejects the reported self-hosted Firecrawl base URL with the exact HTTPS and official-host checks, and the issue is already paired with open implementation PRs that should be reviewed and consolidated rather than closing the report early.

## What This Changes

Allow Firecrawl search/scrape and web_fetch fallback configuration to use a local self-hosted Firecrawl baseUrl over HTTP/private-network hosts instead of only the hosted Firecrawl endpoint.

## Best Possible Solution

Add one narrow Firecrawl plugin change that explicitly supports trusted self-hosted endpoints, allows HTTP only for private/local targets via the existing SSRF SDK seam, uses the trusted web-tools endpoint path only when private access is intended, keeps hosted Firecrawl strict by default, and updates Firecrawl tests plus public docs.

## Reproduction Assessment

Yes. The current helper path is deterministic: `resolveEndpoint("http://host.openshell.internal:3002/v1", "/v2/search")` reaches the HTTPS error, and an HTTPS non-`api.firecrawl.dev` host reaches the host-allowlist error; existing tests assert the same rejection pattern.

## Solution Assessment

No. Current main does not solve the self-hosted Firecrawl use case; the maintainable direction is an explicit self-hosted/private-network policy rather than removing Firecrawl endpoint checks broadly.
