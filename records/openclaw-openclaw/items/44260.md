## Summary

Keep open. The PR addresses a real file-permission pain point, but the patch is incomplete because it only applies the readable default to host workspace Write/Edit operations, lacks after-fix real behavior proof, and now targets pre-refactor fs-safe files that no longer match current main.

## What This Changes

The PR threads an optional file mode through the old fs-safe host write path and passes `0o644` from host workspace Write/Edit call sites while leaving the internal default at `0o600`.

## Best Possible Solution

Decide the user-facing file-mode policy, then update both host workspace and sandbox Write/Edit new-file paths against the current `@openclaw/fs-safe` APIs while preserving `0o600` for internal/sensitive writes and existing target modes.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current host workspace Write/Edit calls `root.write` without a mode and current sandboxed Write/Edit calls `bridge.writeFile`, while the relevant write helpers default new files to `0o600` unless a mode is deliberately supplied.

## Solution Assessment

No. Passing `0o644` only at the host workspace call sites is not the best complete fix because sandboxed Write/Edit uses a different bridge path, and current main requires adapting the change to the external fs-safe wrapper rather than the removed old helper files.
