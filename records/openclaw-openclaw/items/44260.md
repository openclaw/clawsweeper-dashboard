## Summary

Keep open. The PR targets a real file-permission problem, but the current branch is incomplete for sandboxed Write/Edit flows, is stale against the current fs-safe extraction, and lacks contributor-provided real behavior proof before merge.

## What This Changes

The PR adds optional file-mode plumbing to safe workspace writes, passes `0o644` from host Write/Edit operations, and changes one pinned-helper fallback to nullish mode handling.

## Best Possible Solution

Rework the change against current main by applying an explicit user-document mode through host and sandbox Write/Edit paths, preserving existing target modes and private internal defaults, with focused regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: host workspace Write/Edit still omit a mode when calling fs-safe, and sandbox Write/Edit still route through `SandboxFsBridge.writeFile` with no mode while new sandbox temp files are created as `0o600`.

## Solution Assessment

No. The PR's host-only override is directionally correct, but the maintainable fix must also thread mode through sandbox bridge writes and adapt to the current externalized fs-safe API.
