## Summary

Keep this PR open. The branch addresses part of the requested behavior for host workspace Write/Edit, but current main still lacks the change and the PR remains incomplete because sandboxed Write/Edit routes through a bridge and mutation helper that cannot receive a file mode and still creates replacement files with `0o600`.

## What This Changes

The PR adds optional file-mode plumbing to the safe write path, passes `0o644` from host workspace Write/Edit calls, and changes the pinned helper mode fallback to nullish coalescing.

## Best Possible Solution

Revise the PR so both host workspace and sandboxed user-facing Write/Edit create new regular files with `0o644`, while internal and sensitive write paths keep `0o600` defaults and existing-file mode preservation remains intact. Add focused permission regression tests for host safe writes, pinned helper mode handling, and sandbox bridge writes coordinated with #44077.

## Reproduction Assessment

Yes. The high-confidence reproduction path is a sandboxed session with workspace writes enabled: `pi-tools.ts` routes Write/Edit to the sandboxed tools, those call `bridge.writeFile` without a mode, and the sandbox mutation helper creates replacement files with `0o600`; #44077 provides concrete stat/EACCES evidence for that path.

## Solution Assessment

No. The PR's host-only mode override is a useful start, but the maintainable fix needs nullable-safe mode plumbing through both host safe-write and sandbox bridge paths, plus tests proving `0o644` user-facing creation and `0o600` internal defaults.
