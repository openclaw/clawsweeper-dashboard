## Summary

Keep open. The file-permission problem is still real on current main, but this PR is not merge-ready because it is dirty against main, targets old fs-safe internals, lacks real behavior proof, and leaves sandbox Write/Edit behavior inconsistent with the advertised fix.

## What This Changes

The branch adds a mode option to the legacy fs-safe write pipeline and passes `0o644` from host Write/Edit workspace writes.

## Best Possible Solution

Approve or defer a scoped user-output permission policy; if approved, implement it through current host `root.write` and sandbox bridge paths with focused permission tests and real `stat` proof.

## Reproduction Assessment

Yes at source level: current host workspace Write/Edit omits a mode, `@openclaw/fs-safe@0.2.7` defaults new writes to `0o600`, and sandbox Write/Edit routes through a bridge with no mode contract. I did not run a live Write/Edit session in this read-only review.

## Solution Assessment

No, this branch is not the best merge shape as-is. The safer path is a maintainer-approved user-output permission policy implemented against current `root.write` and sandbox bridge contracts, with explicit proof that internal state stays private.
