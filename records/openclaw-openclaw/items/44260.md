## Summary

Keep open. The branch targets a real user-facing file-permission problem, but it is not mergeable as-is: current main moved the host write path to the published `@openclaw/fs-safe` API, sandboxed Write/Edit still has no mode contract for new files, and the broader permission policy is still tracked in https://github.com/openclaw/openclaw/issues/51660.

## What This Changes

The branch threads an optional mode through the legacy fs-safe write helper and passes `0o644` from host Write/Edit call sites.

## Best Possible Solution

Settle the user-facing file-permission policy, then update current-main host `root.write` calls and sandbox bridge write contracts consistently while preserving restrictive internal writes.

## Reproduction Assessment

Yes, at source level. Current workspace-only host Write/Edit omits `mode`, `@openclaw/fs-safe` 0.2.4 defaults missing write modes to `0o600`, and sandbox Write/Edit has no mode field for new files.

No. The current branch is not the best fix as-is because it targets obsolete in-repo fs-safe helper code, updates only host Write/Edit, and leaves sandbox new-file behavior plus product policy unresolved.
