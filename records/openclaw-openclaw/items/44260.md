## Summary

Keep open for maintainer handling: the PR is a plausible permission fix, but the current branch is incomplete for sandboxed Write/Edit, targets stale safe-write internals, lacks contributor real behavior proof, and overlaps the open permission-policy discussion.

## What This Changes

The PR adds a mode option to the old safe-write pipeline and passes `0o644` from host Write/Edit operations for newly created user-facing files.

## Best Possible Solution

Define a maintainer-approved user-output file-mode policy, then implement it through the active `@openclaw/fs-safe` root write path and sandbox bridge/mutation path with focused permission tests while keeping internal state private.

## Reproduction Assessment

Yes, at source level. Current main shows workspace-only host Write/Edit omits a mode, sandbox Write/Edit has no mode field, and sandbox new-file creation still uses `0o600`; no live scenario was run because this was a read-only review.

## Solution Assessment

No. The PR direction is plausible for host-only Write/Edit, but it does not cover sandboxed Write/Edit and should be reconciled with the broader policy tracked at https://github.com/openclaw/openclaw/issues/51660.
