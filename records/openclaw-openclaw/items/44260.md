## Summary

Keep open. The PR targets a real user-facing Write/Edit file-permission pain point, but the current branch is incomplete for sandboxed Write/Edit, is stale against the current external `@openclaw/fs-safe` wrapper layout, and still lacks required real behavior proof.

## What This Changes

This PR threads a file mode through the old in-repo fs-safe write path and passes `0o644` from host workspace Write/Edit call sites.

## Best Possible Solution

Decide the user-facing file-mode policy, then implement it through current `@openclaw/fs-safe` and sandbox bridge write APIs while preserving `0o600` for internal/sensitive files and existing target modes.

## Reproduction Assessment

Yes. Source inspection shows current host workspace Write/Edit omits a mode and sandboxed Write/Edit uses `bridge.writeFile`, while the relevant write helpers default new files to `0o600` unless a mode is supplied; I did not run live chmod/stat proof in this read-only review.

## Solution Assessment

No. The current PR is not the best complete fix because it covers only host workspace Write/Edit, misses sandboxed Write/Edit, and needs adaptation to the current external fs-safe wrapper rather than the removed old helper files.
