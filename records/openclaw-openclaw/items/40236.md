## Summary

Keep open. Current main has adjacent cron timeouts, retry/backoff, failure alerts, run-log/session retention, and `cron_changed` observation, but it still lacks the requested per-job lifecycle policy/API; the related open items cover pieces rather than superseding this combined feature request.

## What This Changes

Request job-level cron lifecycle support for per-job timeout, retry/auto-cleanup policies, and start/complete/failure hooks.

## Best Possible Solution

Have cron owners design one additive lifecycle contract that reuses existing cron primitives where possible, routes observation through `cron_changed` unless job-local handlers are explicitly approved, and splits security-reviewed hooks from retry/cleanup policy work.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable: this is a feature/API request, and source inspection shows the requested lifecycle config/schema is absent from current main.

## Solution Assessment

Is this the best way to solve the issue? Unclear. The nested `lifecycle` object is plausible, but the maintainable path should first reconcile it with existing timeout, retry, alert, retention, run-log, and plugin-hook surfaces plus the active hook PRs.
