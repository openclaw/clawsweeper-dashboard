## Summary

Keep open. Current main has useful adjacent cron lifecycle pieces, including agent-run timeouts, cron_changed plugin events, one-shot cleanup, retry backoff, failure alerts, and retention pruning, but it does not implement the requested per-job lifecycle policy/API for timeoutMs, maxRetries, autoCleanup after runs/status/TTL, or job-configured onStart/onComplete/onFailure hooks.

## What This Changes

The issue requests cron job lifecycle management with per-job timeout, retry/cleanup policy, and start/complete/failure hooks.

## Best Possible Solution

Have cron owners design an additive cron contract that reuses existing primitives where possible: keep payload timeoutSeconds or generalize it deliberately, add any per-job retry/cleanup fields with schema/CLI/docs/tests, and route lifecycle observation through the existing plugin cron_changed surface unless job-local handlers are explicitly desired.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable: this is a feature/API request, and source inspection shows missing lifecycle configuration rather than broken existing documented behavior.

## Solution Assessment

Is this the best way to solve the issue? Unclear. The proposed lifecycle object is plausible, but the maintainable path should first reconcile it with existing timeoutSeconds, cron.retry, failureAlert/failureDestination, deleteAfterRun, cron_changed, sessionRetention, and runLog pruning semantics.
