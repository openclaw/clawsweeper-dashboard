## Summary

Keep open: current main and latest release still skip persisted pairing-store entries when Feishu direct messages run under `dmPolicy: "allowlist"`, while the reported non-default account store file is still a real account-scoped input path.

## What This Changes

The issue reports that Feishu non-default-account DM approvals stored in `credentials/<channel>-<accountId>-allowFrom.json` stop being honored after npm upgrades, causing approved DMs to receive a pairing challenge again.

## Best Possible Solution

Maintainers should choose the allowlist trust contract, then either add a deliberate runtime change with Feishu/shared tests and docs, or keep allowlist config-only and add docs or doctor guidance for store-only approvals.

## Reproduction Assessment

Partially. Source inspection gives a high-confidence path for the allowlist/store mismatch on current main, but I did not run the reported Windows npm-upgrade sequence live.

## Solution Assessment

Unclear. The straightforward runtime change is not automatically the best fix because it would reverse a tested access-control boundary; maintainers should choose between a contract change and a docs/doctor migration path first.
