## Summary

Keep open: current main and v2026.6.6 still skip the persisted pairing allow store when Feishu `dmPolicy` is `allowlist`, while Feishu has account-scoped store plumbing and public docs still conflict on whether allowlist should trust paired-store entries.

## What This Changes

Issue reports that Feishu account-scoped DM allowlist approval is forgotten after npm updates even though the credentials allowFrom file remains on disk.

## Best Possible Solution

Decide the DM allowlist contract explicitly, then either keep allowlist config-only with docs/doctor migration guidance for store-only approvals or deliberately restore paired-store reads for allowlist with security review and focused shared/Feishu tests.

## Reproduction Assessment

Partially. Source inspection gives a high-confidence path for the allowlist/store mismatch, but I did not run a live Windows npm-upgrade reproduction for the update-specific trigger.

## Solution Assessment

Unclear. The obvious runtime fix is not automatically the best fix because it would reverse a tested access-control boundary; maintainers should first choose between a docs/doctor migration path and a deliberate authorization-contract change.
