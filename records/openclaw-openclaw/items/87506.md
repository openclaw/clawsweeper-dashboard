## Summary

Keep open: current main has account-scoped Feishu pairing-store plumbing for pairing mode, but the reported allowlist plus paired-store upgrade scenario is not settled because runtime currently skips pairing-store reads under `dmPolicy="allowlist"` while public docs still describe allowlist as accepting the paired store.

## What This Changes

The issue reports Feishu non-default DM approvals stored in `credentials/feishu-lux-allowFrom.json` being ignored after global npm updates, forcing re-approval.

## Best Possible Solution

Decide and align the Feishu direct-DM contract: either restore documented paired-store authorization for allowlist mode with upgrade tests, or clarify/migrate users to config `allowFrom` for allowlist and `pairing` mode for store-backed approvals.

## Reproduction Assessment

Partially: source inspection shows the Feishu non-default pairing store is account-scoped, but the shared ingress layer deliberately does not read that store in `dmPolicy="allowlist"`. I do not have a high-confidence live reproduction of the Windows npm-update sequence itself.

## Solution Assessment

Unclear: a narrow code change could make allowlist mode read the store, but that conflicts with current tests and authorization semantics. The safer path is a maintainer decision on the intended contract before implementation.
