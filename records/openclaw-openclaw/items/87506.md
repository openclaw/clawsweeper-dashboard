## Summary

Keep open: current main still has a Feishu allowlist and pairing-store contract mismatch. The Feishu path has account-scoped store plumbing, but shared channel authorization intentionally ignores pairing-store entries when `dmPolicy` is `allowlist`, while part of the public gateway docs still describes allowlist as accepting the paired allow store.

## What This Changes

The issue asks OpenClaw to preserve Feishu non-default account DM approvals across npm updates when the approved sender remains in the account-scoped pairing credentials file.

## Best Possible Solution

Resolve the allowlist contract explicitly: either keep allowlist as config-only and update docs/doctor guidance for migrated pairing-store entries, or intentionally restore paired-store reads for allowlist with security review and focused Feishu/shared-channel tests.

## Reproduction Assessment

Partially. Source inspection gives a high-confidence path for the core allowlist/store mismatch, but no live Windows npm-upgrade reproduction was established for the update-specific trigger.

## Solution Assessment

Unclear. Blindly making allowlist read the pairing store is not clearly the best fix because it reverses a tested access-control boundary; the safer path is a maintainer decision between docs/doctor migration and a deliberate runtime contract change.
