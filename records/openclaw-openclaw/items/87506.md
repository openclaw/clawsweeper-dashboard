## What This Changes

The issue asks for Feishu non-default account DM approvals stored in `credentials/<channel>-<accountId>-allowFrom.json` to survive npm updates without re-approval.

## Best Possible Solution

Keep one explicit contract: preferably preserve strict allowlist runtime and make the update/doctor path reliably migrate account-scoped pairing-store entries into canonical `allowFrom`, unless maintainers intentionally approve runtime trust of pairing-store approvals for allowlist mode.
