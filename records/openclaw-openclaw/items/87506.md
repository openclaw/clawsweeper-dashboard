## Summary

Keep open: current main and v2026.6.8 still do not honor account-scoped Feishu pairing-store approvals in allowlist-mode runtime; doctor can migrate the store into config, but the reported manual npm-update path remains a maintainer security/product decision.

## What This Changes

The issue reports that Feishu allowlist-mode DM approvals for a non-default account stop being honored after an npm upgrade even though the account-scoped allowFrom credentials file remains on disk.

## Best Possible Solution

Maintainers should choose one explicit contract: either allowlist mode deliberately trusts approved pairing-store DM entries at runtime, or approvals remain pairing-only and the update/doctor/docs path reliably migrates store-only senders into canonical config allowFrom.

## Reproduction Assessment

Yes, by source inspection: Feishu passes the account-scoped store reader, but shared direct-message ingress skips store reads and store matches under `dmPolicy: "allowlist"`. I did not run the Windows npm-global upgrade path live.

## Solution Assessment

Unclear. Doctor repair is the safer upgrade path already present, but it does not make the reported manual npm update plus gateway restart recognize the stored sender without an extra repair step.
