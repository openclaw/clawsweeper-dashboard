## Summary

Keep open: the direction matches the linked shutdown bug, but the current head still treats any resolved `chat.abort` as successful and the supplied real-behavior proof is for an older head, so this needs another patch and proof pass before merge.

## What This Changes

The PR adds SIGINT/SIGTERM abort propagation through `openclaw agent` and `GatewayClient`, sends `chat.abort` for Gateway agent runs, adds focused tests, and updates CLI/cron docs.

## Best Possible Solution

Keep the signal bridge, but make abort cleanup succeed only when Gateway reports the run was actually aborted or add a server-side pre-registration cancellation path, then refresh current-head real Gateway/CLI proof.

## Reproduction Assessment

Yes, from source inspection. Current main has server-side `chat.abort` registration for agent runs, but the one-shot `openclaw agent` path does not pass a caller abort signal or record the accepted run id while waiting for the final Gateway response.

## Solution Assessment

No, not merge-ready yet. The PR uses the right caller/Gateway boundary, but it must handle the `aborted:false` pre-registration window and refresh proof on the current head.

Full review comments:

- **[P2] Verify chat.abort actually stopped the run:** `src/commands/agent-via-gateway.ts:207-215`
  - body: `abortAcceptedGatewayAgentRunWithRequest` returns `true` whenever the abort RPC resolves. But `chat.abort` returns `{aborted:false, runIds:[]}` when the run id is not registered; because Gateway request handlers run concurrently and `agent` does async setup before registering in `chatAbortControllers`, a SIGTERM in that window can send `chat.abort` too early, mark cleanup successful, and skip the fallback while the original run later continues. Check the response and retry, or add a pre-registration cancel path, before treating the active-connection abort as successful.
