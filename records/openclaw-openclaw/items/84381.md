## What This Changes

The PR threads caller abort signals and accepted-run callbacks through the Gateway client path, has `openclaw agent` send `chat.abort` for accepted runs on SIGTERM/SIGINT, adds focused tests, and updates CLI/cron docs.

## Best Possible Solution

Merge this Gateway signal-abort direction after CI and Gateway owner review, while keeping the documented hard-kill supervisor backstop for non-draining shutdowns.

## Reproduction Assessment

Yes. Source inspection shows current main waits for final Gateway agent replies without passing a caller abort signal or exposing the accepted run id, while current main already has server-side `chat.abort` support for accepted `agent` RPC runs.

## Solution Assessment

Yes. Threading `AbortSignal` and an accepted-run callback through `callGateway`/`GatewayClient` and reusing existing `chat.abort` authorization is narrower and more maintainable than adding a new public knob or separate abort API.
