## Summary

Keep this PR open: it targets a real current-main Gateway agent shutdown gap, but the latest patch can still miss `chat.abort` when the signal arrives before the accepted callback records the run id, and the supplied real-environment proof is for an older head.

## What This Changes

The PR threads caller abort signals and accepted-run callbacks through the Gateway client path, adds `openclaw agent` SIGINT/SIGTERM cleanup via `chat.abort`, updates focused Gateway/agent tests, and documents cron/systemd hard-kill backstops.

## Best Possible Solution

Keep the Gateway client signal-abort design, but seed abort cleanup from the known idempotency/run id before `onAccepted`, add focused signal-before-accepted coverage, and refresh current-head real behavior proof before merge.

## Reproduction Assessment

Yes, from source inspection. Current main has server-side `chat.abort` support for agent runs, but the one-shot agent CLI does not pass a caller abort signal or record the accepted run id before waiting for the final Gateway response.

## Solution Assessment

No, not quite yet. The overall design is the right boundary, but the PR should use the known idempotency/run id before the accepted callback and then rerun proof on the latest head.
