## Summary

Keep this PR open for maintainer handling, but it is not merge-ready: it adds a real ACPX durability feature that is still absent from current main, while the patch is based on the older wrapper-owned subprocess loop and has unresolved credential-persistence and abort-cleanup findings.

## What This Changes

The PR adds wrapper-owned ACPX per-run `route.json`, `terminal.json`, `stdout.log`, and `stderr.log` artifacts with strict terminal finalization, pre-spawn abort handling, collision-safe run IDs, and related tests.

## Best Possible Solution

Define durable terminal-truth and stderr artifacts in the supported `acpx` runtime, then expose only sanitized OpenClaw wrapper metadata with explicit privacy, retention, and regression coverage.

## Reproduction Assessment

Yes for the PR findings: source inspection of PR head `bb45eb5acb517c8f5755f816778fc0b4cbfb40fb` shows raw route args persisted, MCP payloads carrying `mcpServers`, and the abort-listener cleanup gap. Current main does not have the requested artifact feature, so there is no existing shipped behavior failure to reproduce.

## Solution Assessment

No. The proposed implementation is not the best solution as-is because it adds a reusable durability contract in the OpenClaw wrapper after the runtime boundary moved upstream, and it persists unsanitized route metadata.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Redact secret-bearing route args before persisting:** `extensions/acpx/src/runtime-internals/run-artifacts.ts:191`
  - body: `route_args` stores the spawned ACPX args verbatim, but MCP-enabled runs can pass a `--agent` command with a base64url payload containing `mcpServers` env values. Persist a sanitized route view or omit sensitive flags so credentials are not copied into durable artifacts.
  - confidence: 0.92
- **[P2] Move artifact creation into the abort cleanup scope:** `extensions/acpx/src/runtime.ts:572-584`
  - body: The abort listener is registered before `createRunArtifacts`, but listener removal only happens in the later `finally`. If artifact setup throws, a long-lived signal retains a stale listener that can call `cancel` later and accumulate across retries.
  - confidence: 0.86
- **[P3] Avoid duplicate pre-spawn cancellation:** `extensions/acpx/src/runtime.ts:606-607`
  - body: If the signal fires after the listener is attached but before the post-artifact abort check, `onAbort` has already called `cancelOnAbort`; this branch calls it again. Gate or remove the second call to avoid duplicate cancellation side effects.
  - confidence: 0.73
