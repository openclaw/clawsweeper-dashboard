## Summary

Keep this PR open: current main and the latest release still have the QMD Windows path-mangling bug, and the patch is a likely-correct fix at the parser boundary, but it still needs Windows/OpenClaw behavior proof and focused regression coverage before merge.

## What This Changes

The PR updates QMD backend command parsing so drive-letter and UNC-style Windows absolute `memory.qmd.command` values bypass shell-argument splitting and keep backslashes.

## Best Possible Solution

Land this focused parser fix after adding drive-letter and UNC backend-config regression tests plus redacted Windows QMD command-resolution proof, then close the canonical linked issue.

## Reproduction Assessment

Yes, source-level. Current main and `v2026.6.8` still route `memory.qmd.command` through `splitShellArgs`, whose unquoted backslash handling strips Windows separators; I did not run a live Windows QMD setup.

## Solution Assessment

Yes, mostly. Preserving Windows absolute command paths before POSIX-style shell parsing is the right boundary, but this PR still needs focused regression tests and real Windows/OpenClaw proof before it is merge-ready.
