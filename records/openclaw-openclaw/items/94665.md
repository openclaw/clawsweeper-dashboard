## Summary

Keep this PR open: the patch is aimed at the right QMD command-normalization boundary, but it still needs contributor-supplied Windows/OpenClaw behavior proof and focused regression coverage before merge.

## What This Changes

The PR changes QMD memory backend command parsing so drive-letter and UNC-style Windows absolute command paths bypass shell-argument splitting and keep their backslashes.

## Best Possible Solution

Land a focused parser fix after adding drive-letter and UNC backend-config regression tests plus redacted Windows QMD command-resolution proof, then close the linked canonical issue.

## Reproduction Assessment

Yes. Source inspection on current main shows raw Windows backslashes in `memory.qmd.command` are consumed by `splitShellArgs` before the resolved command reaches QMD spawn resolution; I did not run a live Windows QMD setup.

## Solution Assessment

Yes, mostly. Preserving Windows absolute command paths before POSIX-style shell parsing is the right boundary, but this PR still needs focused regression tests and real Windows/OpenClaw proof before it is the best merge-ready fix.
