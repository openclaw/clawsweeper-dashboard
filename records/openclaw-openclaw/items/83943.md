Related threads cover parts of the problem, but this warm-turn growth needs profiling before it can be safely folded into a narrower fix.

## What This Changes

The issue reports a 5.x regression where `session-resource-loader` latency grows across warm Feishu/Lark turns on Windows with MiniMax OAuth, while 4.23 stayed stable.

## Best Possible Solution

Keep this open until maintainers profile one slow current-main loader call, then route the fix either to the Pi resource-loader fast path, the active transcript pre-open boundary, or a narrower combined performance fix with Windows proof.

## Reproduction Assessment

No high-confidence live reproduction was run here. The reporter’s timing logs map to a current-main loader span that still opens/repairs the active session file and reloads Pi resources each attempt, but the exact hot frame needs a CPU profile on the reported setup.

## Solution Assessment

Unclear: caching the whole loader or adding a broad timeout is not proven safe. The best next step is to profile the slow span and then fix the narrow source, preserving inline compaction extensions and existing transcript/session semantics.
