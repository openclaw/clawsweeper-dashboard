## Summary

Keep open. Current main still lacks a typed per-agent delivery default; newer session delivery-context plumbing helps preserve explicit request/session routes, but it does not provide an agent-level fallback for fresh programmatic, webchat-routed, or cron-triggered runs.

## What This Changes

The issue requests an agent-level default delivery channel and target so cross-session or programmatic message calls can send without reconstructing long channel IDs.

## Best Possible Solution

Define a typed, documented agents.list[].delivery contract using the existing DeliveryContext-style fields, validate it in config/schema/help/docs, carry it through resolved agent config, and apply it only as the lowest-precedence trusted fallback after explicit message arguments and per-run/session delivery context.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: agents.list[].delivery is not in the strict agent schema, and targetless message(action=send) still throws unless toolContext.currentChannelId is populated from existing route context.

## Solution Assessment

No. The minimal fallback sketch is not the best final shape because it omits accountId/threadId and explicit precedence; a DeliveryContext-shaped fallback with a documented trust boundary is safer.
