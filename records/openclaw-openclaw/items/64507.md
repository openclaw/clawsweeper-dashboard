## Summary

Keep open. Current main still has only request/session-scoped delivery context, not a typed per-agent delivery default, so targetless message sends from fresh programmatic sessions still cannot infer a target from agent config.

## What This Changes

The issue requests an agent-level default delivery target and channel for programmatic, webchat-routed, cron, or cross-session message sends.

## Best Possible Solution

Define a documented agents.list[].delivery contract using DeliveryContext-style channel, to, accountId, and threadId fields, validate it in config/schema/help, carry it through resolved agent config, and apply it only as the lowest-precedence fallback after explicit args and trusted run/session delivery context.

## Reproduction Assessment

Yes. Source inspection on current main shows agents.list[].delivery is absent from the strict schema/resolved config, and message.send with no target still throws when toolContext.currentChannelId is unavailable.

## Solution Assessment

Unclear. The requested direction is reasonable, but the two-field defaultChannel/defaultTarget sketch is not clearly the best API because the existing routing contract also needs accountId, threadId, and explicit precedence rules.
