## Summary

Keep open. Current main still lacks an agent-level delivery default contract, targetless `message.send` still depends on explicit args or already-captured routing context, and the related cron-default issue only covers cron job defaults rather than cross-session agent runs.

## What This Changes

Request adding per-agent delivery defaults so programmatic or cross-session agent runs can infer message channel and target without repeating long IDs.

## Best Possible Solution

Define a typed, documented `agents.list[].delivery` contract using `DeliveryContext`-style fields, validate it in config/schema/help, carry it through resolved agent config, and apply it only as the lowest-precedence trusted fallback after explicit message args and per-run/session delivery context.

## Reproduction Assessment

Yes for the current limitation by source inspection: the proposed config key is absent from strict agent schema/resolution, and targetless `message.send` still throws when no explicit or trusted routing context exists. I did not run a live channel repro because this is a new config-surface request, not a regression in an existing contract.

## Solution Assessment

Unclear. An agent-level fallback is plausible, but the final shape should use the existing delivery-context vocabulary and define precedence before adding an implicit cross-channel routing knob.
