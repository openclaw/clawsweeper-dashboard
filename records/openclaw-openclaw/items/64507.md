## Summary

Keep open. Current main preserves trusted session/run delivery context, but it still lacks a typed per-agent delivery default, so fresh programmatic/cross-session runs without origin context cannot infer a message target from agent config.

## What This Changes

Request adding per-agent delivery defaults so programmatic, webchat-routed, cron, or cross-session agent runs can infer message tool channel and target without repeating long IDs.

## Best Possible Solution

Define a documented agents.list[].delivery contract using DeliveryContext-style channel, to, accountId, and threadId fields, validate it in config/schema/help, carry it through resolved agent config, and apply it only as the lowest-precedence fallback after explicit message arguments and trusted run/session delivery context.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: agents.list[].delivery is absent from the strict agent schema, and message.send without an explicit target still throws when toolContext.currentChannelId is unavailable.

## Solution Assessment

Unclear. The requested direction is reasonable, but the two-field defaultChannel/defaultTarget sketch is not clearly the best final API because current delivery routing also needs accountId, threadId, and explicit precedence rules.
