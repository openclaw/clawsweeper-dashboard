## Summary

Keep open. Current main still lacks a typed per-agent delivery default; the newer session-level `deliveryContext` paths preserve trusted routed context, but they do not add an `agents.list[].delivery` fallback for fresh programmatic sessions with no origin target.

## What This Changes

The issue requests a typed per-agent delivery default so programmatic, cron, and webchat-routed sessions can send messages without reconstructing long channel target IDs.

## Best Possible Solution

Define a typed, documented `agents.list[].delivery` contract using DeliveryContext-style fields such as `channel`, `to`, `accountId`, and `threadId`, validate it in config/schema/help/docs, carry it through resolved agent config, and apply it only as the lowest-precedence trusted fallback after explicit message arguments and run/session delivery context.

## Reproduction Assessment

Yes. The source-level reproduction remains high-confidence: `agents.list[].delivery` is absent from the strict agent schema, and a targetless `message.send` still throws when no `toolContext.currentChannelId` is available.

## Solution Assessment

No, the minimal two-field sketch is not the best final shape. A DeliveryContext-style contract with clear precedence and trust boundaries is safer than only falling back to `defaultChannel` and `defaultTarget`.
