## Summary

Keep open. Current main still has no agent-level delivery default contract, and target inference for the message tool still depends on explicit arguments or existing trusted session/origin routing; adding this would be a new config surface with routing precedence choices.

## What This Changes

Request an agent-level default delivery target/channel so programmatic, webchat-routed, cron, or cross-session runs can infer message-tool destinations when no origin route is present.

## Best Possible Solution

Define a documented `agents.list[].delivery` fallback using DeliveryContext-style fields, validate it in config/schema/help, carry it through resolved agent config, and apply it only after explicit message args and trusted run/session context.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: `agents.list[].delivery` is absent from the strict schema and `message.send` without an explicit target still throws when `toolContext.currentChannelId` is unavailable.

## Solution Assessment

Unclear. The requested direction is useful, but the two-field sketch is not clearly the safest final API because existing delivery routing also needs accountId, threadId, and explicit precedence rules.
