## Summary

Keep open. Current main still lacks an agent-level delivery default contract, and the existing discussion plus source review show this is a real cross-session delivery feature that needs maintainer API and precedence decisions before implementation.

## What This Changes

Request adding per-agent delivery defaults so targetless message-tool sends from programmatic, webchat-routed, or internal sessions can fall back to configured channel delivery context.

## Best Possible Solution

Define a typed, documented `agents.list[].delivery` contract using the existing delivery-context concepts, carry it through resolved agent config, and apply it only as the lowest-precedence trusted fallback after explicit message args and run/session context.

## Reproduction Assessment

Yes for the current limitation by source inspection: the requested agent config key is absent and targetless `message.send` still depends on explicit args or inferred tool context. I did not run a live channel reproduction because this is a new public config surface rather than a documented behavior regression.

## Solution Assessment

Unclear. Agent-level fallback delivery is plausible, but maintainers should first choose the public config shape and precedence relative to explicit tool arguments, existing delivery context, cron delivery defaults, and per-session routing.
