## Summary

Keep open. Current main still lacks a typed per-agent delivery default, and targetless message sends still rely on explicit arguments or existing trusted run/session origin context rather than an agent-level fallback.

## What This Changes

The issue requests an agent-level default delivery target and channel fallback for programmatic or cross-session message calls that lack origin context.

## Best Possible Solution

Define a documented agents.list[].delivery contract using DeliveryContext-style channel, to, accountId, and threadId fields, validate it in config/schema/help, carry it through resolved agent config, and apply it only as the lowest-precedence fallback after explicit message arguments and trusted run/session delivery context.

## Reproduction Assessment

Yes for the current limitation, from source inspection rather than a live run: the strict agent schema has no delivery field, buildThreadingToolContext derives target context only from origin/session data, and message.send still throws when no target/currentChannelId exists. This is a feature/API gap, not a broken documented behavior repro.

## Solution Assessment

Unclear. The requested direction is reasonable, but the final API should probably use the existing DeliveryContext shape and explicit precedence rules rather than only defaultChannel/defaultTarget strings.
