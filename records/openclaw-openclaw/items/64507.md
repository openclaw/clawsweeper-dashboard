## Summary

Keep open. Current main still has no agent-level delivery default contract, and targetless message sends still rely on explicit arguments or trusted run/session routing context rather than a per-agent fallback; the request is valid but needs config/API precedence decisions.

## What This Changes

The issue requests a per-agent default delivery target/channel so programmatic, webchat-routed, and cross-session agent runs can send messages without reconstructing long channel target IDs.

## Best Possible Solution

Define a typed, documented `agents.list[].delivery` fallback using the existing `DeliveryContext` vocabulary, validate and resolve it with agent config, and apply it only after explicit message args and trusted run/session delivery context.

## Reproduction Assessment

Yes for the current limitation: source inspection shows the proposed `agents.list[].delivery` key is absent from types/schema and targetless `message.send` still throws when no explicit or trusted session/run target exists. I did not run a live channel reproduction because this is a new config-surface request rather than a regression in an existing documented contract.

## Solution Assessment

Unclear. An agent-level fallback is plausible, but the final API should use the existing delivery-context model and define precedence before adding an implicit cross-channel delivery knob.
