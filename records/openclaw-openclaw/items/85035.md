## Summary

Keep open: the queue-hook API is useful and the PR includes credible terminal proof, but the patch currently bypasses existing prompt/conversation plugin trust gates and misses `queue_after_enqueue` observability when a normal follow-up enqueue is blocked.

## What This Changes

The PR adds public plugin hook types, docs, SDK baseline updates, and agent-runner/embedded-runner wiring for `queue_before_enqueue` and `queue_after_enqueue`.

## Best Possible Solution

Land a bounded version that treats queue hooks as prompt/conversation-sensitive plugin APIs, preserves the existing opt-in trust controls, emits after hooks for every follow-up queue decision, and keeps the SDK docs/tests aligned.

## Reproduction Assessment

Yes for the review findings by source inspection: the registry only enforces hook trust policy through the existing classifier lists, and the PR adds prompt-bearing queue hooks outside those lists. The blocked follow-up path also returns before the new after-hook call, while the embedded blocked path calls it.

## Solution Assessment

No. The hook seam can be the right SDK direction, but this implementation needs trust-policy classification and blocked-decision after-hook consistency before it is the safest maintainable solution.
