## Summary

Keep open: the security hardening is directionally right, but this PR is not merge-ready because it leaves the transport-level cross-origin 307 redirect test asserting the old body replay behavior and still lacks real redirect proof.

## What This Changes

The PR changes MCP HTTP fetch redirect handling to disable cross-origin unsafe POST body/method replay and adds focused MCP HTTP fetch regression tests.

## Best Possible Solution

Keep the shared secure default for MCP, update the transport-level redirect expectation, and add redacted real HTTP redirect plus representative MCP compatibility proof before merge.

## Reproduction Assessment

Yes from source inspection: current main opts MCP HTTP fetches into unsafe cross-origin replay, and the shared guard preserves 307/308 bodies only when that flag is true. I did not run local tests because this review is read-only, but PR CI also shows the changed behavior by failing the old transport expectation.

## Solution Assessment

Yes, removing the MCP-only unsafe replay override is the right owner-boundary fix because the shared guard already owns redirect policy and the provider transport uses the secure default. The PR is incomplete until the transport-level coverage and real behavior proof are aligned.
