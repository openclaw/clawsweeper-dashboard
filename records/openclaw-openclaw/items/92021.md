## Summary

The patch is a narrow, source-backed correction of a recent MCP HTTP security regression, and no actionable code defect was found; keep it open because the external-PR gate still requires after-fix proof from a real redirecting MCP setup.

## What This Changes

The PR disables unsafe cross-origin method/body replay in the MCP HTTP fetch wrapper and adds regression tests for cross-origin 307 and 302 POST redirects.

## Best Possible Solution

Keep the shared fetch guard as the single redirect-policy owner, remove the MCP-only unsafe replay exception, and require one redacted live redirect trace before merge.

## Reproduction Assessment

Yes. Current main explicitly enables unsafe cross-origin replay in `buildMcpHttpFetch`, and the shared guard deterministically preserves or removes the method/body based on that option; the missing evidence is real-network proof rather than a source reproduction path.

## Solution Assessment

Yes. Removing the isolated MCP exception is narrower and more maintainable than adding another redirect implementation, matches the sibling provider transport, and remains compatible with the pinned SDK contract.
