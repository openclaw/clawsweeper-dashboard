Keep open. ## What This Changes

The PR threads cron session metadata into embedded agent tool context and downgrades fresh isolated cron loop-warning diagnostics from warning to info, with focused before-tool-call coverage.

## Best Possible Solution

Make loop-warning emission single-owner and context-aware for fresh isolated cron runs, preserving critical blocks and non-isolated warnings, then attach redacted real gateway log proof before merge.

## Reproduction Assessment

Yes. Source inspection on the PR head shows `detectToolCallLoop` still emits warn-level `[agents/loop-detection]` messages before the new caller-side downgrade runs, and current main/latest release still have the same warning paths.

## Solution Assessment

No. This is a plausible partial mitigation, but the best fix is to make the detector warning emission itself context-aware or move warning logging to one caller-owned path so the reported err-log source is actually downgraded.
