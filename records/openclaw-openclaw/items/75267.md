## Summary

Keep this PR open, but do not merge it as-is. The current main bug is real, and the patch is aimed at the right picker code, but the diff introduces actionable picker-state regressions by hiding the configured supplement without confirming an actual provider-scoped option set and by clamping an invalid initial value to the first option, which can be a manual/setup action rather than a model. It also needs the required changelog entry for a user-facing fix.

## What This Changes

The PR changes `src/flows/model-picker.ts` to hide mismatched configured-model supplements in provider-scoped default pickers, clamp invalid initial values to displayed options, and adds model-picker regression tests in `src/commands/model-picker.test.ts`.

## Best Possible Solution

Keep the fix local to `src/flows/model-picker.ts`, but update it so configured supplements are hidden only when there is an actual provider-scoped model set, invalid initial values prefer a displayed model row instead of action entries, empty option lists fall back before calling the select prompt, and the user-facing fix is recorded in the changelog.

## Reproduction Assessment

Yes. Current main can be reproduced by configuring an alias such as `sonnet` for `claude-cli/claude-sonnet-4-6`, then opening `promptDefaultModel` with `preferredProvider: "openai"` and `allowKeep: false`; the configured Claude key is appended and preselected outside the OpenAI rows. The PR regressions are reproducible from source by adding manual/setup options or auth-filtered rows after the patch, which can make the fallback initial value an action or leave no options.

## Solution Assessment

No. The proposed direction is close, but the implementation should clamp to the first displayed model value, preserve the configured supplement when no actual provider scope exists, and avoid calling `prompter.select` with an empty option array.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Keep the supplement when the picker is not scoped:** `src/flows/model-picker.ts:804`
  - body: `maybeFilterModelsByProvider` only narrows to the preferred provider when matching rows exist; otherwise it leaves the full catalog unscoped. This guard checks only `preferredProvider`, so a current configured model that is not in the catalog disappears even when there is no provider-scoped option set to replace it. Gate this on the actual scoped state, such as `hasPreferredProvider`, so fallback/unscoped pickers do not make the current model unselectable.
  - confidence: 0.82
- **[P2] Prefer a model row when clamping initialValue:** `src/flows/model-picker.ts:828`
  - body: When `allowKeep` is false and manual or provider-setup actions are present, `options[0]` can be `__manual__` or a setup action because those entries are prepended before model rows. A provider-scoped picker with a hidden configured alias and valid model rows can therefore default into an action path instead of an in-scope model; choose the first displayed model value and handle the no-option case before calling `select`.
  - confidence: 0.9
