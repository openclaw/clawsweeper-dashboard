## Summary

Keep open: the interactive model-picker bug is valid on current main, but this PR still has two P2 correctness blockers and lacks after-fix real behavior proof.

## What This Changes

The PR changes `promptDefaultModel` to suppress out-of-scope configured model supplements, clamp invalid `initialValue`, and add model-picker regression tests.

## Best Possible Solution

Keep this focused interactive model-picker fix, but update it to preserve fallback mode, choose a real displayed model or manual fallback, and include real behavior proof.

## Reproduction Assessment

Yes. Source inspection shows current main can append a configured `claude-cli` alias and initialize the picker to it with `preferredProvider: "openai"`, `allowKeep: false`, and `includeManual: false`.

## Solution Assessment

No. The PR targets the right code path, but it should gate supplement hiding on an actual scoped-provider list and avoid falling back to action rows or an empty options list.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Gate supplement hiding on actual provider scope:** `src/flows/model-picker.ts:804`
  - body: The new provider check hides the configured supplement whenever `preferredProvider` is set and the resolved provider differs. `maybeFilterModelsByProvider` intentionally falls back to the full list when no preferred-provider rows exist, so this makes the current configured model unselectable in that fallback path; gate this on `hasPreferredProvider` or an equivalent actual-scope signal.
  - confidence: 0.86
- **[P2] Choose a model row for invalid initial values:** `src/flows/model-picker.ts:828`
  - body: When the configured value is removed, `options[0]` can be `__manual__`, `__keep__`, or a provider setup action because those rows are prepended before model rows. Prefer the first displayed model option and avoid calling select with an empty options list.
  - confidence: 0.82
