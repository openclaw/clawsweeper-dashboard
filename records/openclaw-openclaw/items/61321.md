## Summary

Keep this issue open. Current main still preserves the reported same-agent old-root absolute `sessionFile` path, and PR #61322 is already open with closing syntax for the same fix path.

## What This Changes

The issue requests remapping persisted same-agent absolute `sessionFile` paths from an old state root into the current agent sessions directory before transcript writes or write-locking.

## Best Possible Solution

Land PR #61322 or an equivalent narrow resolver change that remaps only same-agent canonical old-root session paths into the active state dir, with regression coverage preserving cross-agent and unsafe-path behavior.

## Reproduction Assessment

Yes. The issue gives concrete state and CLI steps, and current-main static inspection reproduces the failure path: the resolver returns the old absolute path unchanged, then the write lock creates that path's directory.

## Solution Assessment

Yes. Fixing this in session path resolution before persistence and write-locking is the narrowest maintainable solution; a broad startup migration would miss runtime resolution paths and has more compatibility risk.
