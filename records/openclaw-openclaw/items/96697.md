## Summary

Keep open: current main and the latest release still unconditionally register plugin CLI commands inside `registerNodesCli`, while `nodes` otherwise inherits the no-plugin command-path policy; the prior JSON-output fix only moved noisy logs to stderr and did not remove the slow plugin/runtime load.

## What This Changes

The issue asks for `openclaw nodes status/list` to stay on the lightweight built-in CLI path without loading plugin CLI/runtime registration.

## Best Possible Solution

Centralize nodes plugin CLI loading behind argv/command-path policy so built-in `nodes status/list` skip plugin loads while plugin-owned `openclaw nodes <plugin>` subcommands still work through metadata/lazy registration.

## Reproduction Assessment

Yes, there is a high-confidence source-level reproduction path: current main still calls plugin CLI registration from `registerNodesCli` even though the `nodes` command policy inherits `loadPlugins: "never"`. I did not execute the timed container command in this read-only sweep.

## Solution Assessment

Yes, the best fix is to guard or relocate the unconditional nodes plugin registration so built-in `status/list` avoid plugin loads while plugin-owned nested nodes commands such as Canvas remain available when requested.
