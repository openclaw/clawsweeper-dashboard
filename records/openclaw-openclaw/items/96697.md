## Summary

Keep open: current main and the latest release still unconditionally register plugin CLI commands inside `registerNodesCli`, while the command-path policy otherwise keeps `nodes` on the no-plugin default; this is a source-backed regression and a narrow fix candidate.

## What This Changes

The issue requests keeping `openclaw nodes status/list` on the lightweight built-in CLI path by skipping plugin CLI/runtime registration for those commands.

## Best Possible Solution

Make `nodes status/list` honor the existing command-path plugin-load policy, skipping plugin CLI/runtime registration for no-plugin paths while adding regression coverage that prevents the built-in nodes commands from loading plugins again.

## Reproduction Assessment

Yes for source-level reproduction: current main unconditionally registers plugin CLI commands inside `registerNodesCli`, and release source shows this block absent in v2026.5.7 but present in v2026.5.28 and v2026.6.10. ## Solution Assessment

Yes, the reporter's suggested direction is the maintainable fix: remove or guard the unconditional nodes plugin registration with the existing command-path policy. The fix should update the current test that expects plugin registration for `nodes list --json`.
