## Summary

Keep open. Current main still collapses Slack explicit and implicit mention detail into `WasMentioned` and prompt-visible `was_mentioned`, and the protected `maintainer` label means this should stay with maintainer/API review rather than cleanup closure.

## What This Changes

Request Slack inbound context preserve explicit mention target/source metadata and avoid presenting implicit thread wakes as an unqualified bot mention.

## Best Possible Solution

Add an additive Slack mention-source structure through prepare, finalized context, and prompt metadata while preserving `WasMentioned` compatibility; coordinate any stricter wake policy with the existing strict mention design discussion.

## Reproduction Assessment

Yes, source-level. Current main can mark a thread reply as `WasMentioned=true` from `reply_to_bot` or `bot_thread_participant` even when the only explicit Slack user mention targets someone else, and the prompt then exposes only `was_mentioned: true`.

## Solution Assessment

Unclear. Additive mention-source metadata is the narrowest compatible direction, but changing implicit wake behavior or `WasMentioned` semantics needs maintainer product/API judgment and should stay aligned with https://github.com/openclaw/openclaw/issues/62066.
