## Summary

Keep open. The report is recent and concrete, and current main still has multiple Telegram Bot API setup calls before the first long-polling `getUpdates`; I did not find source, docs, tests, or post-release history proving the high-RTT startup silence is fixed.

## What This Changes

The issue asks to fix a 2026.5.2 Telegram regression where high-RTT hosts time out during startup setup and never reach long-polling `getUpdates`.

## Best Possible Solution

Harden Telegram startup so slow or transient `getMe`, command-menu, and webhook-cleanup calls either retry with visible operator logs or safely reach long polling, with focused regression coverage for the reported startup ordering.

## Reproduction Assessment

No, there is not yet a high-confidence live or simulated reproduction of the full high-RTT silence on current main. The source path and production logs identify the pre-`getUpdates` setup gates, but a targeted test or live network simulation still needs to pin down the exact wedge.

## Solution Assessment

Unclear. The requested config knob should not be the first fix; the narrower maintainable solution is to harden the existing Telegram startup and retry path before adding a new user-facing timeout option.
