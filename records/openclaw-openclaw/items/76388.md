## Summary

Keep open. The report is recent and concrete, and current main still has Telegram startup paths where `getMe`, command-menu sync, and webhook cleanup can fail before the first successful `getUpdates`; existing tests cover pieces of recovery but not the reported high-RTT silent startup wedge.

## What This Changes

Fix the Telegram polling regression where high-RTT hosts on 2026.5.2 can fail to reach long-polling startup after Bot API setup timeouts.

## Best Possible Solution

Add a focused Telegram startup recovery fix that makes pre-poll `getMe`, command-menu, and webhook-cleanup timeouts bounded, visible, and unable to permanently prevent long polling, with regression coverage for the no-`getUpdates` sequence.

## Reproduction Assessment

No high-confidence live reproduction was established. The production logs and current source make the pre-`getUpdates` setup path clear, but the exact silent high-RTT wedge still needs a simulated or live regression test.

## Solution Assessment
