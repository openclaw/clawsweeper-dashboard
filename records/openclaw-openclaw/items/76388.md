## Summary

Keep open. The report is recent, concrete, and still lines up with current `main`: Telegram polling is documented as long-polling, but the current startup path can still spend multiple Bot API control-plane calls before the first `getUpdates`, and existing tests do not cover the exact high-RTT/silent-startup sequence.

## What This Changes

Telegram polling should start reliably on high-RTT hosts in 2026.5.2 instead of going silent after startup/control-plane Bot API timeouts.

## Best Possible Solution

Telegram polling startup should either reach `getUpdates` after recoverable control-plane timeouts or expose an actionable retry/failure state in logs, status, and doctor, with regression tests for delayed `getMe`, command-menu sync, and webhook cleanup.

## Reproduction Assessment

No, not as a high-confidence live reproduction. Source inspection confirms the pre-`getUpdates` gates and the tests cover related retry behavior, but the exact high-RTT silent wedge still needs a targeted simulated or live reproduction.

## Solution Assessment

Unclear as written. The requested config knob is not the best first step; the narrower maintainable solution is to harden the existing Telegram startup/retry path and make failures visible without adding configuration unless necessary.
