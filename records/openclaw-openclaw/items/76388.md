## Summary

Keep open. Current main still routes Telegram long polling through pre-`getUpdates` setup gates, and the newer source/docs/tests do not prove the reported high-RTT startup silence is fixed.

## What This Changes

The issue asks to fix a 2026.5.2 Telegram regression where startup setup can time out on high-RTT hosts and never reach long-polling `getUpdates`.

## Best Possible Solution

Harden Telegram startup so slow or transient `getMe`, command-menu, and webhook-cleanup calls recover with visible diagnostics and reliably reach long polling, with regression coverage for the startup ordering.

## Reproduction Assessment

No, not with high confidence. The production logs and current source identify the pre-`getUpdates` setup gates, but a live or simulated current-main high-RTT failure path still needs to pin down the exact wedge.

## Solution Assessment

Unclear. A new `channels.telegram.startupTimeoutMs` knob should not be the first fix; the narrower maintainable path is to repair and test the existing Telegram startup/retry/diagnostic flow.

## Repair Work Prompt

Fix the Telegram polling startup regression described in https://github.com/openclaw/openclaw/issues/76388. Keep the boundary inside the Telegram plugin: harden pre-poll startup so transient or slow `getMe`, command-menu, and `deleteWebhook`/`getWebhookInfo` calls cannot leave polling silently dead, and make retry/fallback logs visible enough for operators. Prefer repairing the existing startup/retry behavior over adding `channels.telegram.startupTimeoutMs`; stop and ask if a new config option or product policy is required. Add regression coverage for setup calls timing out before the first `getUpdates`, including command-menu sync and webhook cleanup behavior. Update `CHANGELOG.md` because this is user-facing. Do not move Telegram-owned behavior into core or change unrelated channel/provider startup policy.
