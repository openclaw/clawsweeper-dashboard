## Summary

Keep open. The report is recent and concrete, Telegram long polling is documented as default behavior, and current main still performs startup Bot API work before the first `getUpdates`; existing tests preserve parts of that pre-poll gating and do not prove the high-RTT startup silence is fixed.

## What This Changes

The issue reports that OpenClaw 2026.5.2 can log Telegram startup and a Bot API `getMe` timeout but never enter `getUpdates` polling on high-RTT VPS hosts, while 2026.4.29 works with the same setup.

## Best Possible Solution

Harden the Telegram-owned startup path so bounded `getMe`, native command-menu sync, and webhook cleanup failures are visible and either retry cleanly or allow safe long-poll startup, with regression coverage for the pre-`getUpdates` timeout sequence.

## Reproduction Assessment

No high-confidence live reproduction was established in this read-only review. The production logs and current source give a credible pre-`getUpdates` path, but a focused simulated high-RTT/control-plane-timeout test is still needed.

## Solution Assessment

Unclear. The config-knob suggestion should not be the first fix; the narrower maintainable path is to harden and test the existing Telegram startup and retry behavior inside the Telegram plugin.

## Repair Work Prompt

Fix the Telegram polling startup regression tracked by https://github.com/openclaw/openclaw/issues/76388. Current main runs a gateway `getMe` probe, starts fire-and-forget native command menu sync, then awaits polling-session `deleteWebhook` before `run(bot)` can issue `getUpdates`. Implement a narrow Telegram-owned fix so recoverable `getMe`, `deleteMyCommands`/`setMyCommands`, and `deleteWebhook` timeouts cannot leave the provider silent before first `getUpdates`: make failures visible at normal operator log level, retry/rebuild transport where appropriate, and continue to long polling when webhook cleanup is already satisfied or safely non-blocking. Do not add `channels.telegram.startupTimeoutMs` or move behavior into core unless maintainers explicitly choose that product/config direction. Add regression coverage for a high-RTT/control-plane timeout sequence before first `getUpdates`, including the command-menu path with a large command set, and add a user-facing changelog entry if runtime behavior changes.
