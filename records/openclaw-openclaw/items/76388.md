## Summary

Keep open. Current main still routes Telegram long polling through pre-`getUpdates` setup gates, and the available source, docs, and tests do not prove the reported high-RTT 2026.5.2 startup silence is fixed.

## What This Changes

The issue reports a Telegram polling startup regression where high-RTT hosts on 2026.5.2 time out during pre-poll Bot API setup and never reach `getUpdates`.

## Best Possible Solution

Harden Telegram polling startup so slow or transient `getMe`, command-menu, and webhook-cleanup calls recover with visible diagnostics and cannot prevent the first long-poll cycle, backed by a simulated regression test.

## Reproduction Assessment

No, not with high confidence. The production logs and source identify the pre-`getUpdates` gates, but I did not establish a deterministic live or simulated current-main failure path for the exact startup silence.

## Solution Assessment

Unclear. A new `channels.telegram.startupTimeoutMs` knob should not be the first fix; the narrower maintainable path is to repair and test the existing Telegram startup, retry, and diagnostic flow.

## Repair Work Prompt

Fix the Telegram polling startup regression reported in https://github.com/openclaw/openclaw/issues/76388. Add a simulated regression first for a high-RTT/timeout startup path where pre-poll Bot API setup (`getMe`, native command menu sync, or `deleteWebhook`/`getWebhookInfo`) times out or fails before `run()` starts, and assert that OpenClaw logs actionable retry diagnostics and still reaches or retries toward `getUpdates` without going silently idle. Keep the fix inside `extensions/telegram`; do not add a new `channels.telegram.startupTimeoutMs` config knob unless the existing startup/retry flow cannot be made reliable. Update `CHANGELOG.md` because this is user-facing Telegram reliability work, and avoid changing unrelated plugin externalization or upgrade-path behavior.
