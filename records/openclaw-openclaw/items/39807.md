## Summary

Keep open: current main still resolves inline provider API keys without a cooldown identity, and billing/auth failure marking still returns before recording anything when no auth profile ID exists. The open linked PR is the active fix candidate, so this issue should close only after that PR or an equivalent provider-level cooldown fix lands.

## What This Changes

The issue requests billing/auth cooldown backoff for providers configured only with inline `models.providers.<id>.apiKey` credentials.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/88709, or an equivalent maintainer-approved provider-level inline API-key cooldown fix, then close this issue after merge.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure an inline `models.providers.<id>.apiKey`, trigger a billing/auth failure, and current main resolves the key without a `profileId` while failure marking returns early when `profileId` is absent.

## Solution Assessment

Unclear at the issue level. A provider-level inline-key usage ID appears to be the right owner boundary, but maintainers still need to accept the compatibility-sensitive fail-closed cooldown contract before the linked fix merges.
