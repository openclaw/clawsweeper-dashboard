## Summary

Keep open. Current main still resolves explicit inline provider API keys without any profile or cooldown identity, while billing cooldown recording and fallback pre-skip remain profile-id based.

## What This Changes

The issue reports that billing failures from inline `models.providers.<id>.apiKey` credentials bypass auth-profile cooldowns and retry indefinitely instead of falling through to fallbacks.

## Best Possible Solution

Add a stable provider-scoped, non-secret cooldown identity for inline API-key credentials and carry it through auth resolution, failure recording, and fallback pre-skip while preserving explicit API-key precedence.

## Reproduction Assessment

Yes, source-level. Configure a provider only with `models.providers.<id>.apiKey`: current main resolves that credential without `profileId`, uses `[undefined]` as the profile candidate, and the billing cooldown path is profile-id based.

## Solution Assessment

No current patch is under review. A refreshed fix should use the closed synthetic-profile PR as context, but it needs to preserve current explicit API-key precedence, avoid persisting secrets, and cover both failure recording and fallback pre-skip.
