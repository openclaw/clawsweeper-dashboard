## Label Justifications

## Summary

Keep open. Current main still resolves literal inline `models.providers.<id>.apiKey` credentials without a profile identity, and the runner's auth-failure marker returns before recording cooldown state when no profile id exists; the open linked fix PR is the active path for this issue.

## What This Changes

The issue asks for billing/auth cooldown backoff to apply to providers configured only with inline `models.providers.<id>.apiKey` credentials.

## Best Possible Solution

Keep this issue open while maintainers review https://github.com/openclaw/openclaw/pull/88709; if that PR lands, close this issue as implemented by the merge.

## Reproduction Assessment

Yes, at source level: a literal inline provider key resolves without `profileId`, and the runner exits the auth-failure marker before calling the profile cooldown recorder. I did not reproduce with a live exhausted paid account.

## Solution Assessment

Yes for routing: the best next step is to finish review of the linked PR instead of creating a duplicate fix. The provider-level cooldown direction fits the missing profile identity, but the fail-closed auth-provider contract still needs maintainer acceptance.
