## What This Changes

Issue reports that inline provider API keys are not covered by auth billing cooldowns, causing repeated provider retries after 402 billing failures.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/88709 or an equivalent provider-level cooldown fix, then close this issue once merged code proves inline API-key billing failures are recorded and skipped during cooldown.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure an inline `models.providers.<id>.apiKey`, trigger a billing/auth failure, and current main resolves the key without a `profileId` while the failure marker returns when no profile id is present.

## Solution Assessment

Yes for the repair boundary: provider-level inline-key cooldown state is the maintainable fix because auth-profile cooldowns cannot cover credentials that intentionally have no profile. The linked PR is the active candidate, but maintainers still need to accept the compatibility-sensitive fail-closed behavior before merge.
