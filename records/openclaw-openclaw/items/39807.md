## Label Justifications

- `P1`: The report describes a live billing/auth failure loop that can repeatedly hit an exhausted paid provider and block fallback for affected agents.
- `impact:auth-provider`: The affected path is provider credential resolution, billing cooldown state, and fallback routing for inline API-key auth.

## Summary

Keep open: current main and v2026.5.22 still leave inline provider API keys outside billing cooldown tracking, while the open closing PR is the active but unmerged fix path.

## What This Changes

Apply billing-error cooldown/backoff to providers configured only with inline `models.providers.<id>.apiKey` credentials so repeated 402 failures stop retrying the same exhausted provider and can fall through to fallbacks.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/80482 with one provider-level cooldown path for literal and env-backed inline API keys, then close this issue after merged source or release proof.

## Reproduction Assessment

Yes, source-level. Configure `models.providers.<id>.apiKey` without an auth profile, trigger a billing-classified failure, and current main resolves no `profileId`, clears profile candidates, then skips failure marking because `profileId` is absent.

## Solution Assessment

Yes for issue routing. Keeping this open while the linked fix PR is reviewed or replaced is the narrowest maintainable path; closing should wait for merged source and release/main proof.
