## Summary

Keep open. Current main still leaves inline `models.providers.<id>.apiKey` auth outside the profile-based billing cooldown path, and the linked implementation PR is closed unmerged.

## What This Changes

The issue requests billing-error cooldown/backoff for providers configured only with inline API keys so failed 402 providers are skipped in favor of fallbacks.

## Best Possible Solution

Add a stable provider-scoped, non-secret cooldown identity for inline API-key credentials and wire it through auth resolution, failure recording, and fallback pre-skip without persisting inline key values.

## Reproduction Assessment

Yes, source-level. Configure a provider only with `models.providers.<id>.apiKey`: current main resolves it without `profileId`, runs with `[undefined]` as the profile candidate, and only records or pre-skips cooldowns by profile id.

## Solution Assessment

Not applicable to a current patch. The closed PR’s synthetic-profile direction is useful context, but a refreshed fix should preserve explicit API-key precedence and never persist secrets from inline config.
