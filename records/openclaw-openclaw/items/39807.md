## Summary

Keep open. Current main still resolves inline provider API keys without a cooldown identity and skips failure marking when no auth profile id exists; https://github.com/openclaw/openclaw/pull/80482 is the open implementation path but is not merged.

## What This Changes

The issue asks OpenClaw to apply billing-error cooldown/backoff to inline `models.providers.<id>.apiKey` credentials, not only auth-profile-backed providers.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/80482 or an equivalent provider-level inline API-key cooldown fix, including literal inline keys and SecretRef-backed inline keys, then close this issue from merged main or release proof.

## Reproduction Assessment

Yes, source-level: configure `models.providers.<id>.apiKey` without an auth profile, trigger a billing-classified failure, and current main resolves no `profileId`, clears explicit-config profile candidates, then skips failure marking because `profileId` is absent.

## Solution Assessment

Yes for triage: keeping this issue open while the linked provider-level cooldown fix is reviewed, updated, or replaced is the maintainable path because current main still lacks inline-key cooldown storage and skip logic.
