## Summary

Keep open. Current main still leaves inline `models.providers.<id>.apiKey` credentials without a durable cooldown identity, so billing-classified failures cannot be persisted and later turns cannot pre-skip the disabled inline primary.

## What This Changes

The issue requests billing-error cooldown/backoff for providers configured only with inline `models.providers.<id>.apiKey` credentials.

## Best Possible Solution

Add a generic, provider-agnostic, non-secret cooldown identity for inline API-key credentials and use it consistently in auth failure recording plus fallback pre-skip logic.

## Reproduction Assessment

Yes, source-level. Configure a provider with only `models.providers.<id>.apiKey`: current main resolves it with no `profileId`, skips failure recording for missing ids, and has no profile-order entry for later fallback pre-skip.

## Solution Assessment

Not applicable to a current patch. The narrow maintainable fix is a stable non-secret inline-provider cooldown identity, either as a synthetic profile or equivalent provider-level usage key consumed by both failure recording and model fallback.

## Repair Work Prompt

Fix the inline API-key billing cooldown gap for this issue. Inline `models.providers.<id>.apiKey` credentials currently resolve without a durable `profileId`, `maybeMarkAuthProfileFailure()` skips missing ids, `markAuthProfileFailure()` only persists stats for existing profiles, and model fallback pre-skip only checks auth-profile order. Add a generic, provider-agnostic, non-secret cooldown identity for inline provider credentials and feed it through both billing/auth failure recording and fallback candidate pre-skip. Do not add MiniMax-specific handling, do not store secret key material in usage identity, and do not change unrelated provider precedence. Use closed PR #39836 only as historical context, not as a blind patch. Add regression coverage for inline configured API keys, persisted billing `disabledUntil`, preserving existing profile behavior, and fallback skipping to secondary models after billing disable. Add a user-facing changelog fix entry unless maintainers decide this is internal-only.
