## Summary

Keep open: current main and latest release still leave inline provider API keys outside the auth-profile cooldown path, and the linked open PR is the active fix candidate rather than a merged implementation.

## What This Changes

The issue asks OpenClaw to apply billing cooldown/backoff and fallback skipping to inline `models.providers.<id>.apiKey` providers that have no auth profile.

## Best Possible Solution

Review and land or close https://github.com/openclaw/openclaw/pull/88709 as the canonical fix, with explicit maintainer acceptance of the inline-key cooldown compatibility contract.

## Reproduction Assessment

Yes, source-reproducible: inline `models.providers.<id>.apiKey` credentials resolve without a profile id, and current main returns before recording auth/billing failure state when `profileId` is missing. I did not run a live exhausted-provider 402.

## Solution Assessment

Yes, the best current direction is provider-scoped inline-key usage state rather than synthetic auth profiles, as in the linked PR. The remaining question is maintainer acceptance of the fail-closed cooldown and persisted usage namespace.
