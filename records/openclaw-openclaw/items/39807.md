## Summary

Keep open: current `main` still resolves inline `models.providers` API keys without an auth-profile identity, so billing failures have no profile key to mark disabled; the related fix PR was closed unmerged.

## What This Changes

The issue requests billing-error cooldown/backoff for providers configured only with inline API keys in `models.providers`.

## Best Possible Solution

Add a generic non-secret cooldown identity for inline provider credentials so true 402 billing failures set `disabledUntil` and selection skips to fallback/profile paths, with regression coverage and no MiniMax-specific special case.

## Reproduction Assessment

Yes from source inspection: inline provider keys resolve without `profileId`, then the billing failure path returns before marking cooldown state. I did not run a live provider 402 in this read-only review.

## Solution Assessment

Yes: the narrow maintainable fix is to make inline credentials participate in the existing auth cooldown contract, either via a synthetic non-secret profile identity or equivalent provider-level cooldown key. The closed related PR sketches that direction but is not merged and should be reworked against current `main`.

## Repair Work Prompt

Fix the inline API-key billing cooldown bug tracked by https://github.com/openclaw/openclaw/issues/39807. Providers resolved from `models.providers.<provider>.apiKey` currently run without an auth-profile id, so 402/billing failures do not call `markAuthProfileFailure` and never get `disabledUntil`. Implement a generic, non-secret cooldown identity for inline provider credentials or an equivalent provider-level cooldown path; do not persist real inline API keys into `auth-profiles.json`, do not special-case MiniMax, and keep existing auth-profile behavior compatible. Use #39836 only as prior art, not as a patch to apply blindly. Add regression tests proving inline `apiKey` billing failures are marked disabled, active disable windows are not extended by repeated failures, configured `auth.cooldowns` billing settings apply, and fallback/profile selection skips the disabled inline credential. Add a user-facing changelog entry because this fixes runaway provider retries.
