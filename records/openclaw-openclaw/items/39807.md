## Summary

Keep open. Current main still resolves inline `models.providers.<id>.apiKey` credentials without a durable auth-profile or provider cooldown identity, so billing-classified failures cannot be persisted and later turns cannot pre-skip the disabled inline primary; the related implementation PR #39836 was closed unmerged.

## What This Changes

The issue asks for billing/auth cooldown backoff to apply to inline `models.providers.<id>.apiKey` providers so 402 failures do not retry indefinitely.

## Best Possible Solution

Add a generic, provider-agnostic cooldown identity for inline API-key credentials and feed it through auth failure recording plus fallback pre-skip logic without MiniMax-specific handling or storing secret key material.

## Reproduction Assessment

Yes, source-level. Configure a provider with only `models.providers.<id>.apiKey`: current main resolves the key with no `profileId`, skips failure recording for missing profile ids, and has no profile-order entry for later fallback pre-skip.

## Solution Assessment

Not applicable to a current patch. The narrow maintainable fix is a stable non-secret inline-provider cooldown identity, either as a synthetic profile or equivalent provider-level usage key consumed by both failure recording and model fallback.
