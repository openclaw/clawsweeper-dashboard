## Summary

Keep open. Current main still leaves inline configured provider API keys outside the profile-backed billing cooldown path, and https://github.com/openclaw/openclaw/pull/80482 is the active unmerged fix path with a recorded P2 review blocker.

## What This Changes

The issue reports that provider billing failures for inline `models.providers.<id>.apiKey` credentials can retry indefinitely instead of entering billing cooldown/backoff.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/80482 with a stable provider-level cooldown identity for inline API keys, covering literal keys and configured env SecretRefs, then close this issue from merged implementation proof.

## Reproduction Assessment

Yes, source-level. Configure a provider only with `models.providers.<id>.apiKey`: current main resolves that credential without a profile id, skips profile candidates for explicit config API-key auth, and records billing failures only through profile-backed state.

## Solution Assessment

Yes for routing. Keeping this issue open while the linked PR is fixed, reviewed, or replaced is the narrow maintainable path; closing before merge would leave the reported retry/backoff bug unresolved on main.
