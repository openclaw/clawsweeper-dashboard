## Summary

Keep open. Current main still leaves inline provider API keys outside the auth-profile cooldown path, and https://github.com/openclaw/openclaw/pull/80482 is the active open closing PR rather than a merged fix.

## What This Changes

Apply billing-failure cooldown/backoff to providers configured only with inline `models.providers.<id>.apiKey` credentials.

## Best Possible Solution

Review and land or replace the linked PR with provider-level cooldown identity for all configured inline API keys, including SecretRef/env form, then close from merged main or release proof.

## Reproduction Assessment

Yes, source-level. Configure a provider through `models.providers.<id>.apiKey` with no auth profile; current main resolves that credential without a `profileId`, clears profile candidates for explicit config API-key auth, and skips failure marking when no `profileId` exists.

## Solution Assessment

Yes for routing: keeping this issue open while the linked PR is repaired, reviewed, or replaced is the narrow maintainable path. The implementation should use provider-level cooldown state for inline keys rather than closing before merged proof exists.
