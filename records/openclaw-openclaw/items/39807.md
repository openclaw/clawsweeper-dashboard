## Summary

Keep open. Current main still resolves explicit inline `models.providers.<id>.apiKey` credentials without a profile/cooldown identity, so billing-classified failures can be raised but cannot be durably recorded and skipped on later runs.

## What This Changes

The issue asks OpenClaw to apply billing-error backoff/cooldown to inline API-key providers, not only providers backed by auth profiles.

## Best Possible Solution

Add a stable, provider-agnostic, non-secret cooldown identity for inline API-key credentials and feed it through auth resolution, failure recording, and fallback pre-skip logic.

## Reproduction Assessment

Yes, source-level. Configure a provider only with `models.providers.<id>.apiKey`: current main resolves it without `profileId`, runs with `[undefined]` as the profile candidate, and the billing cooldown path is profile-id based.

## Solution Assessment

Not applicable to a current patch. The closed PR’s synthetic-profile direction is useful context, but the refreshed fix should preserve current explicit API-key precedence and avoid persisting secrets.
