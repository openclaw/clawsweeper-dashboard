## Label Justifications

- `P1`: The report describes an urgent provider-auth failure loop that can leave agents unresponsive while repeatedly hitting a billing-disabled provider.
- `impact:auth-provider`: The affected path is provider credential resolution, billing cooldown state, and fallback routing for inline API-key auth.

## Summary

Keep open. Current main and the latest release still leave inline `models.providers.<id>.apiKey` credentials without a cooldown identity, and the linked fix remains open rather than merged.

## What This Changes

The issue asks OpenClaw to apply billing/402 cooldown backoff to providers configured with inline `models.providers.<id>.apiKey` credentials so exhausted keys stop being retried and fallbacks can run.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/80482 or an equivalent provider-level inline API-key cooldown fix that covers literal inline keys and SecretRef-backed inline keys, then close with merged-main or release proof.

## Reproduction Assessment

Yes, source-level. Configure `models.providers.<id>.apiKey` without an auth profile, trigger a billing-classified failure, and current main resolves no `profileId`, clears profile candidates, then skips failure marking because `profileId` is absent.

## Solution Assessment

Yes for the issue path. Provider-level cooldown state for inline API-key credentials is the maintainable fix shape, but the linked PR still needs update/review before the issue is implemented on main.
