## Summary

This issue should stay open: current main and the latest release still route inline provider API keys outside the profile-backed cooldown path, while an open closing PR already owns the concrete fix candidate.

## What This Changes

The issue asks OpenClaw to apply billing/auth cooldown backoff to providers configured only with inline `models.providers.<id>.apiKey` credentials so repeated 402 failures stop retrying the same exhausted key.

## Best Possible Solution

End state should be a maintainer-approved provider-level cooldown path for inline API keys that skips exhausted inline credentials while preserving healthy stored auth-profile fallback behavior.

## Reproduction Assessment

Yes from source inspection, but not by executing a live exhausted paid-provider account in this read-only review. Current main still resolves inline provider API keys without a `profileId`, and the runner/auth-profile failure marker skips missing profile IDs.

## Solution Assessment

Yes on direction, unclear on final acceptance. Provider-scoped cooldown for inline credentials is the maintainable boundary, but the exact fail-closed and persisted-namespace contract should be accepted through the linked PR review before this issue closes.
