## Summary

Keep open: current `main` and the latest release still have the source-level inline-key cooldown gap, and the open linked PR is the correct implementation candidate to review rather than closing this issue early.

## What This Changes

The issue asks for billing/auth cooldown backoff to apply to inline `models.providers.<id>.apiKey` credentials so 402 failures stop retrying the same exhausted provider and can proceed to fallback handling.

## Best Possible Solution

Review and either land or reject https://github.com/openclaw/openclaw/pull/88709; close this issue only after the accepted inline-key cooldown fix reaches `main`.

## Reproduction Assessment

Yes from source: inline provider API-key auth resolves without a `profileId`, and the failure marker returns before cooldown recording when no profile id exists. I did not run a live exhausted-provider 402 because this is a read-only review.

## Solution Assessment

Yes, the provider-scoped cooldown approach in the open linked PR is the narrowest current fix shape because it avoids synthetic auth profiles while preserving healthy stored-profile fallback. The remaining question is maintainer acceptance of the compatibility-sensitive auth contract.
