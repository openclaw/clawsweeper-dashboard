## Summary

Keep open. Current main still records billing cooldowns through stored auth profile IDs, while inline provider API keys can resolve without a profile ID; the open linked PR is the active fix path and should decide the remaining compatibility-sensitive behavior.

## What This Changes

The issue requests billing/auth cooldown backoff for providers configured only with inline `models.providers.<id>.apiKey` credentials so 402 failures do not retry indefinitely.

## Best Possible Solution

Review and either land or reject https://github.com/openclaw/openclaw/pull/88709; close this issue only after the accepted inline-key cooldown behavior reaches main.

## Solution Assessment

Yes for the active direction: provider-scoped inline-key cooldown tracking avoids synthetic profiles and preserves stored-profile fallback. The remaining question is maintainer acceptance of the fail-closed inline-key cooldown contract.
