## Summary

Keep open: current main and the latest release still only record billing cooldowns through auth profile IDs, while inline provider API-key auth resolves without a profile ID. The open linked PR is the current fix candidate, so this issue should close only after that PR, or a maintainer-approved variant, lands.

## What This Changes

The issue requests billing/auth cooldown coverage for providers configured only with inline `models.providers.<id>.apiKey` credentials so 402 failures stop retrying the same exhausted provider.

## Best Possible Solution

Keep this issue open while https://github.com/openclaw/openclaw/pull/88709 is reviewed; land that PR or a maintainer-approved variant, then close this issue as implemented.

## Reproduction Assessment

Yes, source-reproducible: inline provider API-key auth on current main has no `profileId`, and the embedded runner returns before recording cooldown when no profile ID exists. I did not run a live exhausted MiniMax 402.

## Solution Assessment

Yes for the current direction: provider-scoped inline-key usage state without synthetic auth profiles is the narrowest fix path found. The remaining question is maintainer acceptance of the compatibility-sensitive fail-closed cooldown contract.
