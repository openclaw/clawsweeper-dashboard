## Summary

Keep this issue open: current main and the latest release still lack provider-level cooldown tracking for inline `models.providers.<id>.apiKey` credentials, and the active closing work is the open, clean PR at https://github.com/openclaw/openclaw/pull/88709.

## What This Changes

Inline API-key providers should enter the same billing/auth cooldown path as auth-profile-backed providers after 402 or insufficient-credit failures.

## Best Possible Solution

Review and land or reject https://github.com/openclaw/openclaw/pull/88709; after a merge, close this issue as implemented with the provider-level inline-key cooldown proof.

## Reproduction Assessment

Yes from source: current main records billing cooldowns only through auth profile IDs, while inline provider API-key credentials can resolve without any profile ID. I did not run a live paid-provider 402 against an exhausted account.

## Solution Assessment

Yes for the current fix direction, with one maintainer decision still needed. The linked provider-scoped cooldown approach avoids synthetic auth profiles and preserves stored-profile fallback, but maintainers need to accept the fail-closed inline-key cooldown contract.
