## Label Justifications

- `P1`: The report describes an urgent provider-auth failure loop that can make agents unresponsive and repeatedly spend provider calls.
- `impact:auth-provider`: The affected path is provider credential resolution, billing cooldown state, and fallback routing for inline API-key auth.

## Summary

Keep open. Current main still leaves inline configured provider API keys outside the auth-profile cooldown identity, so billing failures can be retried across turns; the open linked PR is the right implementation lane but has not merged.

## What This Changes

Apply billing backoff/cooldown to inline `models.providers.<id>.apiKey` credentials so 402 billing failures stop immediately reusing exhausted providers and can fall through to configured fallbacks.

## Best Possible Solution

Review, repair, and land https://github.com/openclaw/openclaw/pull/80482 or an equivalent provider-level inline API-key cooldown fix, including env SecretRef coverage, then close this issue from merged main or release proof.

## Reproduction Assessment

Yes, source-level. Configure `models.providers.<id>.apiKey` without an auth profile, trigger a billing-classified failure, and current main resolves no `profileId`, clears profile candidates for explicit config auth, then skips failure marking because `profileId` is absent.

## Solution Assessment

Yes for triage. Keeping this issue open while the linked PR is repaired, landed, or replaced is the maintainable path; closing before merged proof would leave the reported billing retry loop unresolved on main.
