## Summary

Keep open. Current main still resolves explicit inline `models.providers.<id>.apiKey` credentials without a durable cooldown identity, so billing-classified failures can be raised but cannot be recorded and skipped on later turns; the related fix attempt at https://github.com/openclaw/openclaw/pull/39836 is closed and unmerged.

## What This Changes

The issue asks OpenClaw to apply billing backoff and fallback skipping to providers configured only with inline `models.providers.<id>.apiKey` credentials.

## Best Possible Solution

Add a stable, non-secret inline-credential cooldown identity and route it through auth resolution, failure recording, and fallback pre-skip while preserving explicit configured-key precedence.

## Reproduction Assessment

Yes, source-level. Configure a provider only with `models.providers.<id>.apiKey`: current main resolves auth without `profileId`, runs with `[undefined]` as the profile candidate, and only records/skips cooldowns by profile id.

## Solution Assessment

Not applicable to a current patch. The synthetic-profile direction in https://github.com/openclaw/openclaw/pull/39836 is useful context, but a refreshed fix should preserve current precedence and persist only a non-secret cooldown identity.
