## Summary

Keep open. Current main still resolves inline configured API keys without a durable auth-profile identity, so billing-classified failures cannot be persisted into cooldown state and fallback pre-skipping still only reasons over stored auth profiles. The linked implementation PR #39836 was closed unmerged, so it does not supersede this issue.

## What This Changes

This issue asks OpenClaw to apply billing cooldown/backoff behavior to providers configured only with inline `models.providers.<id>.apiKey` credentials.

## Best Possible Solution

Add a provider-agnostic, non-secret cooldown identity for inline API-key credentials and have both failure recording and fallback candidate skipping consume that identity. The fix should not store plaintext inline keys in auth-profile state and should remain generic rather than MiniMax-specific.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: configure a provider with only `models.providers.<id>.apiKey`, trigger a billing-classified failure, and the resolved auth has no `profileId` for `maybeMarkAuthProfileFailure()` or model fallback pre-skip logic to use.

## Solution Assessment

Not applicable to a current patch. The best implementation direction is a stable inline-provider cooldown identity or equivalent provider-level usage key shared by auth resolution, failure persistence, and fallback skipping.
