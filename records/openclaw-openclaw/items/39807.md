## Summary

This should stay open: current main and the latest release still only record billing cooldowns through auth-profile identities, while the inline API-key case has no profile identity. The open linked PR is the active fix path and should be reviewed or revised before this issue closes.

## What This Changes

The issue asks for inline `models.providers.<id>.apiKey` credentials to participate in billing-error cooldown/backoff and fallback skipping instead of retrying a 402 provider indefinitely.

## Best Possible Solution

Review the open linked PR and either land its provider-level inline API-key cooldown contract or ask for a revised compatibility shape before closing this issue.

## Reproduction Assessment

Yes, from source inspection: inline provider API keys resolve without a profile/cooldown identity, and the runner drops auth/billing failure marking when `profileId` is absent. I did not run a live exhausted-provider 402 because this was a read-only review and intentionally causing paid-provider billing failure is not appropriate.

## Solution Assessment

Unclear until maintainer review of the linked PR completes. Provider-level cooldown tracking for inline credentials appears to be the right boundary, but the fail-closed inline-key behavior and persisted `inline-api-key:<provider>` namespace need explicit maintainer acceptance.
