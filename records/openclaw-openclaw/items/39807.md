## Summary

This issue is still necessary because current `main` does not yet record cooldown state for inline provider API keys with no auth profile, and the open linked PR is the active implementation path rather than a merged fix.

## What This Changes

Inline `models.providers.<id>.apiKey` credentials should enter auth/billing cooldown after 402-style billing failures instead of being retried indefinitely.

## Best Possible Solution

Review and either land or explicitly reject https://github.com/openclaw/openclaw/pull/88709; once it merges, this issue can close as implemented.

## Reproduction Assessment

Yes from source: current `main` records billing cooldowns through auth profile IDs, while the reported inline provider key path has no `profileId`. I did not run a live exhausted paid-provider 402.

## Solution Assessment

Yes for the current direction: the open PR's provider-scoped cooldown avoids synthetic auth profiles and preserves stored-profile fallback. The remaining question is maintainer acceptance of the fail-closed cooldown contract.
