## Summary

Keep open: current main and the latest release still resolve inline provider API keys without a cooldown identity, and auth failure marking still no-ops when no profile ID exists. The open linked PR is the active fix candidate and should be reviewed or landed before this issue closes.

## What This Changes

Billing/auth cooldown should apply to inline `models.providers.<id>.apiKey` provider credentials so 402 billing failures do not retry indefinitely and can fall through to fallback models.

## Best Possible Solution

Land a maintainer-approved provider-level inline API-key cooldown fix, preserving healthy stored-profile fallback and closing this issue only after the linked PR or an equivalent fix merges.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure an inline `models.providers.<id>.apiKey`, hit a billing failure, and current main resolves the key without a `profileId` while failure marking returns early when `profileId` is absent.

## Solution Assessment

Unclear at the issue level. A provider-level inline-key usage ID is the right boundary to evaluate and the open PR is the active candidate, but maintainers still need to accept its auth compatibility shape before merge.
