## Summary

Keep open. Current main still leaves explicit inline `models.providers.<id>.apiKey` credentials without a durable profile/cooldown identity, so billing-classified failures can be thrown as failover errors but cannot be persisted and pre-skipped on later turns.

## What This Changes

Apply auth cooldown/backoff handling to providers configured only with inline `models.providers.<id>.apiKey` credentials.

## Best Possible Solution

Add a provider-agnostic, non-secret cooldown identity for inline API-key credentials and use it consistently in auth resolution, failure recording, and fallback pre-skip logic.

## Reproduction Assessment

Yes, source-level. Configure a provider only with `models.providers.<id>.apiKey`: current main resolves it with no `profileId`, uses `[undefined]` as the profile candidate, and later billing failure/cooldown logic is profile-id based.

## Solution Assessment

Not applicable to a current patch. The narrow maintainable fix is a stable non-secret inline-provider cooldown identity, implemented either as a synthetic profile or equivalent provider-level usage key consumed by both failure recording and model fallback.
