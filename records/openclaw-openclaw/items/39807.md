## Summary

Keep this issue open. Current main and the latest release still have the no-profile inline API-key cooldown gap, and https://github.com/openclaw/openclaw/pull/88709 is an open closing PR that owns the implementation path.

## What This Changes

The issue reports that inline `models.providers.<id>.apiKey` credentials do not enter billing cooldown after 402 errors, causing repeated retries instead of backoff and fallback.

## Best Possible Solution

Review and land or explicitly reject https://github.com/openclaw/openclaw/pull/88709, preserving healthy stored-profile fallback behavior while adding provider-level cooldown coverage for literal, env-marker, and SecretRef inline keys.

## Reproduction Assessment

Yes from source, but not from a live paid-provider 402 run in this read-only review. Current main resolves inline provider API keys without a profile id, and both the runner wrapper and cooldown store return before recording failure state when no profile id exists.

## Solution Assessment

No, closing this issue now would be premature because the linked closing PR is still open. The best fix shape is the provider-level inline-key cooldown path under review, rather than forcing users to migrate every inline key into an auth profile as the permanent solution.
