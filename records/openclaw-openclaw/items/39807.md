## Summary

Keep open: current main still only records billing cooldown through auth profile IDs, while the reported inline provider API-key path has no profile ID. The open linked PR is the active fix candidate and should be reviewed/landed before this issue closes.

## What This Changes

The issue reports that inline `models.providers.<id>.apiKey` credentials without auth profiles do not enter billing cooldown, causing repeated 402 retries instead of skipping to fallback credentials or providers.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/88709 or an equivalent provider-level inline-key cooldown fix that records auth/billing failures while preserving healthy stored-profile fallback.

## Reproduction Assessment

Yes from source inspection: inline API-key credentials currently resolve without a profile id, and the failure marker returns before recording cooldown when `profileId` is missing. I did not run a live paid-provider 402 because that would intentionally trigger a billing failure.

## Solution Assessment

Yes, the best current path is the open linked provider-level cooldown PR if maintainers accept its compatibility contract. A new ClawSweeper repair PR would duplicate the active implementation path.
