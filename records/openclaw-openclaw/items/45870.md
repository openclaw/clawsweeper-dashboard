## Summary

Keep open: the Windows test cleanup is useful, but the current PR still introduces source-level regressions in home-prefix expansion and Windows executable validation, so it should not merge as-is.

## What This Changes

The PR updates Windows-specific path and schtasks tests and modifies executable lookup, home-prefix expansion, and stable Homebrew Node path matching helpers.

## Best Possible Solution

Keep the Windows and schtasks test normalization, preserve current home and executable resolution semantics, and add focused regressions for unmatched tilde tokens, restricted PATHEXT, and stable-node casing.

## Reproduction Assessment

Yes. Source inspection of the PR diff gives helper-level reproductions: `~alice/bin/tool` is resolved after a non-match, and caller PATHEXT is applied to already-qualified executables.

## Solution Assessment

No. The schtasks assertion update is aligned with current runtime, but the production helper changes should be narrowed or covered by regressions instead of changing approval and executable semantics as a side effect.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve unmatched tilde tokens:** `src/infra/home-dir.ts:73`
  - body: Wrapping the replacement in `path.resolve(...)` also resolves inputs the regex leaves untouched. A value like `~alice/bin/tool` becomes cwd-absolute, and callers feed this helper into exec allowlist and command path logic, so an unresolved token can silently target a different filesystem path.
  - confidence: 0.92
- **[P2] Keep PATHEXT out of explicit file checks:** `src/infra/executable-path.ts:41`
  - body: Passing caller env into `isExecutableFile` makes Windows reject an already-found file when its extension is not in PATHEXT. With `PATHEXT=.CMD`, a concrete `node.exe` path can return `undefined` even though the executable exists; PATHEXT should drive suffix search, not invalidate qualified paths.
  - confidence: 0.88
- **[P3] Normalize formula casing before fallback:** `src/infra/stable-node-path.ts:32`
  - body: The Cellar matcher is case-insensitive, but the fallback still checks `formula === "node"`. A path like `Cellar/Node/.../node.exe` matches only because of the `i` flag and then skips the `<prefix>/bin/node.exe` fallback.
  - confidence: 0.62

## Security Review

Status: needs_attention

Summary: Needs attention because the PR changes executable and approval-path helper semantics, though it does not touch workflows, dependencies, lockfiles, or release scripts.

Concerns:

- **[low] Home-prefix expansion can retarget approval paths:** `src/infra/home-dir.ts:73`
  - body: The proposed `path.resolve(...)` wrapper can turn an unmatched `~user/...` token into a cwd-relative absolute path. Because this helper feeds exec allowlist and command-resolution paths, that can change what an approved pattern or command target refers to.
  - confidence: 0.82
- **[low] PATHEXT gating can deny explicit executables:** `src/infra/executable-path.ts:41`
  - body: Forwarding caller PATHEXT into `isExecutableFile` makes explicit files depend on extension-search policy, which can break host exec resolution under restricted environments and create surprising allowlist misses for concrete binaries.
  - confidence: 0.78
