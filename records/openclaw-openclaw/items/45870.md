## Summary

Keep open: the schtasks test direction matches current runtime behavior, but the PR still introduces source-reproducible regressions in home-prefix expansion and Windows executable validation that should block merge.

## What This Changes

The PR updates Windows-specific path and schtasks tests and changes executable lookup, home-prefix expansion, and stable Homebrew Node path matching helpers.

## Best Possible Solution

Retain the Windows and schtasks assertion normalization, but narrow the production helper changes so existing home expansion and executable resolution semantics are preserved with focused regressions.

## Reproduction Assessment

Yes. Source inspection of the PR diff gives helper-level reproductions: `~alice/bin/tool` is resolved after a non-match, and caller `PATHEXT` is applied to already-qualified executables.

## Solution Assessment

No. The schtasks assertion update is aligned with runtime behavior, but changing approval-path and executable semantics as a side effect is not the narrowest maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Preserve unmatched tilde tokens:** `src/infra/home-dir.ts:73`
  - body: Wrapping the replacement in `path.resolve(...)` also resolves inputs the regex intentionally leaves untouched. A value like `~alice/bin/tool` becomes cwd-absolute, and callers feed this helper into exec allowlist and command path logic, so an unresolved token can silently target a different filesystem path.
  - confidence: 0.92
- **[P2] Keep PATHEXT out of explicit file checks:** `src/infra/executable-path.ts:41`
  - body: Passing caller env into `isExecutableFile` makes Windows reject an already-found file when its extension is not in `PATHEXT`. With `PATHEXT=.CMD`, a concrete `node.exe` path can return `undefined` even though the executable exists; `PATHEXT` should drive suffix search, not invalidate qualified paths.
  - confidence: 0.88
- **[P3] Normalize formula casing before fallback:** `src/infra/stable-node-path.ts:32`
  - body: The Cellar matcher is case-insensitive, but the fallback still checks `formula === "node"`. A path like `Cellar/Node/.../node.exe` matches only because of the `i` flag and then skips the `<prefix>/bin/node.exe` fallback.
  - confidence: 0.62

## Security Review

Status: needs_attention

Summary: The PR does not touch workflows, dependencies, or release scripts, but it changes executable and approval-path helper semantics in ways that can affect command targeting.

Concerns:

- **[low] Home-prefix expansion can retarget approval paths:** `src/infra/home-dir.ts:73`
  - body: The proposed `path.resolve(...)` wrapper can turn an unmatched `~user/...` token into a cwd-relative absolute path. Because this helper feeds exec allowlist and command-resolution paths, that can change what an approved pattern or command target refers to.
  - confidence: 0.82
- **[low] PATHEXT gating can deny explicit executables:** `src/infra/executable-path.ts:41`
  - body: Forwarding caller `PATHEXT` into `isExecutableFile` makes explicit files depend on extension-search policy, which can break host exec resolution under restricted environments and create surprising allowlist misses for concrete binaries.
  - confidence: 0.78
