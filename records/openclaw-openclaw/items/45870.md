Keep open: the useful Windows assertion cleanup is not ready because the PR still introduces helper-semantic regressions in executable and home path handling, and it lacks contributor-supplied real Windows behavior proof.

## What This Changes

The PR updates Windows-specific schtasks/path tests and changes home-directory expansion, Windows executable detection, and stable Homebrew Node path resolution.

## Best Possible Solution

Rebase the branch, keep the Windows schtasks/test expectation normalization, preserve current unmatched-tilde and executable-validation contracts, add focused regression coverage, and provide redacted Windows proof.

## Reproduction Assessment

Yes for the PR-introduced blockers: source inspection shows unmatched `~user` inputs being resolved after a failed home-prefix replacement and caller PATHEXT being applied to already-found executable files. I did not run Windows tests because this review is read-only.

## Solution Assessment

No. The schtasks assertion update is directionally correct, but the patch should not change approval-path and executable-resolution semantics as a side effect of Windows test cleanup.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve unmatched tilde tokens:** `src/infra/home-dir.ts:73`
  - body: `expandHomePrefix` should not resolve the whole string after a failed replacement. Unsupported inputs like `~alice/bin/tool` are currently left literal by this helper, but the PR turns them into cwd-absolute paths, which can change direct allowlist-pattern matching.
  - confidence: 0.92
- **[P2] Keep caller PATHEXT out of file validation:** `src/infra/executable-path.ts:41`
  - body: Forwarding caller env into `isExecutableFile` makes an already-found Windows executable depend on that caller's PATHEXT. A sanitized env such as `PATHEXT=.CMD` can reject a concrete `.exe` path even though suffix lookup, not file validation, is the part that should use caller PATHEXT.
  - confidence: 0.88
- **[P3] Normalize formula casing before fallback:** `src/infra/stable-node-path.ts:32`
  - body: The Cellar matcher becomes case-insensitive, but the fallback still checks `formula === "node"`. A matched `Cellar/Node/.../node.exe` path would skip the default-formula bin fallback introduced by the same change.
  - confidence: 0.62
