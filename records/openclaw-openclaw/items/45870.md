## Summary

Keep open. The Windows schtasks assertion cleanup is supported by current runtime behavior, but the branch still introduces production path-helper changes with concrete regressions in tilde and PATHEXT handling.

## What This Changes

This PR updates Windows schtasks/path test expectations and changes home-prefix expansion, executable detection, and Homebrew Node path stabilization helpers.

## Best Possible Solution

Refresh or replace the branch with the Windows/schtasks assertion cleanup while preserving current tilde and executable-resolution semantics, then add focused regressions for the helper edge cases.

## Reproduction Assessment

Yes. The diff gives helper-level reproductions: an unmatched token such as ~alice/bin/tool becomes cwd-absolute under the PR, and a qualified Windows executable can be rejected when caller PATHEXT is narrower than the file extension.

## Solution Assessment

No. The schtasks test cleanup is the right direction, but the current patch is not the narrowest maintainable fix because it changes production home-prefix and executable validation semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve unmatched tilde tokens:** `src/infra/home-dir.ts:73`
  - body: Wrapping the replacement in `path.resolve(...)` also resolves inputs that the regex intentionally leaves untouched. A value like `~alice/bin/tool` becomes cwd-anchored, and callers use this helper for executable and allowlist paths, so an unresolved token can silently target a different filesystem path.
  - confidence: 0.92
- **[P2] Bypass PATHEXT checks for qualified executables:** `src/infra/executable-path.ts:41`
  - body: Passing caller env into `isExecutableFile` makes Windows reject an already-found file when its extension is not in PATHEXT. With `PATHEXT=.CMD`, a concrete `node.exe` path can return `undefined` even though the executable exists; PATHEXT should drive suffix search, not invalidate qualified paths.
  - confidence: 0.88
- **[P3] Normalize formula casing before fallback:** `src/infra/stable-node-path.ts:32`
  - body: The Cellar matcher is case-insensitive, but the default formula fallback still checks `formula === "node"`. A path that only matched because of the `i` flag, such as `Cellar/Node/.../node.exe`, skips the prefix bin fallback.
  - confidence: 0.62

## Security Review

Status: needs_attention

Summary: Needs attention because the diff changes executable and approval-path helper semantics, though it does not touch workflows, dependencies, lockfiles, or release scripts.

Concerns:

- **[low] Home-prefix expansion can retarget approval paths:** `src/infra/home-dir.ts:73`
  - body: The proposed `path.resolve(...)` wrapper can turn an unmatched `~user/...` token into a cwd-relative absolute path. Because this helper feeds exec allowlist and command-resolution paths, that can change what an approved pattern or command target refers to.
  - confidence: 0.82
- **[low] PATHEXT gating can deny explicit executables:** `src/infra/executable-path.ts:41`
  - body: Forwarding caller PATHEXT into `isExecutableFile` makes explicit files depend on extension-search policy, which can break host exec resolution under restricted environments and create surprising allowlist misses for concrete binaries.
  - confidence: 0.78
