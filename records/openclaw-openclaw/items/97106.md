## Summary

Keep open: the branch is a focused attempt to fix https://github.com/openclaw/openclaw/issues/97069, but inferring the reason from `params.ask` in one reply helper can mislabel real ask-always approval prompts when callers only pass filtered `allowedDecisions`. It also leaves the broader renderer/protocol reason path unresolved.

## What This Changes

The PR changes `src/infra/exec-approval-reply.ts` and its test so missing Allow Always is described as non-persistable-command copy whenever `ask` is not `always`.

## Best Possible Solution

Carry a typed Allow Always unavailable reason, or at least the policy context that produced it, from the exec approval producer through `ExecToolDetails`/gateway request data into every renderer before changing user-facing copy.

## Reproduction Assessment

Yes, from source with high confidence. Current main removes Allow Always for both ask-always policy and one-shot persistence, and this PR's new branch lacks the producer-side context needed to distinguish those cases on every call path.

## Solution Assessment

No. The narrow copy change is a plausible partial patch, but the maintainable fix is to carry a real unavailable reason or policy context through the approval data model and update the sibling renderers that share the same invariant.
