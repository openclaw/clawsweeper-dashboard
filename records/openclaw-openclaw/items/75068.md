## Summary

Keep open. Current main still rejects Brave `llm-context` freshness and date filters before dispatch even though Brave documents those parameters; the related PR #51005 targets the narrow fix but has not merged and appears to need rebase or replacement against the active runtime. The `ui_lang` part remains less certain because Brave’s LLM Context docs do not list that parameter.

## What This Changes

The issue requests that Brave `llm-context` web_search accept `freshness` and `date_after`/`date_before`, and clarify or relax `ui_lang` and `count` handling.

## Best Possible Solution

Keep the issue open until PR #51005 or a replacement lands the provider-owned runtime fix: remove the llm-context freshness/date preflight guards, wire date filters into Brave freshness ranges with correct cache keys, update Brave and web-search docs plus changelog, add focused Brave tests, and treat `ui_lang` and `count` as explicit scope decisions.

## Reproduction Assessment

Yes. With a Brave API key and Brave configured as `mode: "llm-context"`, calling `web_search` with `freshness`, `date_after`, or `date_before` deterministically returns the local `unsupported_*` payload before any Brave network request; this is visible directly in `executeBraveSearch`.

## Solution Assessment

Unclear for the whole report. The narrow best fix is to enable documented freshness/date filters in the active Brave runtime; `ui_lang` is not documented by Brave for LLM Context, and `count` can be handled as a small follow-up or included only if maintainers accept that scope.
