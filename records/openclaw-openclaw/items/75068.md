## Summary

Keep open. Current main still rejects Brave `llm-context` `freshness` and `date_after`/`date_before` before the provider request is sent, while Brave's current LLM Context docs list `freshness`, custom date ranges, and `count` as supported request parameters. The already-linked PR #51005 is the right active implementation path, but it has not merged, so the issue is not implemented on main.

## What This Changes

The issue requests that Brave `web_search` in `llm-context` mode stop rejecting documented freshness and date filters, and asks maintainers to reconsider `ui_lang` and `count` handling for that mode.

## Best Possible Solution

Keep the issue open until #51005 or a replacement lands a provider-owned fix: allow documented Brave LLM Context freshness/date filters, convert date bounds into Brave freshness ranges, keep cache keys distinct, update Brave/web docs and changelog, and add focused Brave provider tests. Treat `ui_lang` and `count` as explicit scope decisions rather than folding them into the freshness/date bug without maintainer agreement.

## Reproduction Assessment

Yes. With Brave configured as `mode: "llm-context"`, `web_search` with `freshness`, `date_after`, or `date_before` deterministically returns local `unsupported_*` payloads before any network call; this is visible directly in `executeBraveSearch`.

## Solution Assessment

Partly. The narrow best fix is to unblock Brave-documented freshness/date filters in the Brave plugin runtime; removing the `ui_lang` guard is less clearly supported because Brave's LLM Context docs do not document that parameter.
