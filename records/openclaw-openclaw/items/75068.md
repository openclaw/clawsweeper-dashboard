## Summary

Keep open. Current main still returns local `unsupported_freshness` and `unsupported_date_filter` responses for Brave `llm-context` mode before the request can reach Brave, while Brave's LLM Context documentation lists `freshness` and custom date ranges as supported. The related PR #51005 is open and targets the same bug, so this should stay open until that PR is landed, repaired, or replaced.

## What This Changes

The issue requests that Brave `web_search` in `llm-context` mode stop falsely rejecting `freshness`, `date_after`, `date_before`, and possibly `ui_lang`.

## Best Possible Solution

Land, rebase, or replace the existing provider-owned fix so documented `freshness` and date-range filters are wired through the active Brave runtime with focused tests, docs updates, and a changelog entry; keep `ui_lang` as an explicit product/API decision rather than bundling it into the same mechanical fix.

## Reproduction Assessment

Yes. With a configured Brave API key and Brave mode set to `llm-context`, calling `web_search` with `freshness`, `date_after`, or `date_before` deterministically returns the local `unsupported_*` response before any Brave network call.

## Solution Assessment

Partly. Unblocking documented `freshness` and date ranges in the active Brave runtime is the narrow maintainable fix, but blindly allowing `ui_lang` is not clearly the best solution because upstream LLM Context docs do not list it as a supported parameter.
