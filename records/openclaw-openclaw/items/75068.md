## Summary

Keep open. Current main still preflight-rejects Brave `llm-context` freshness and date filters before dispatch, while Brave's own LLM Context reference documents `freshness` and custom date ranges; the related open PR has not merged.

## What This Changes

The issue asks OpenClaw to stop rejecting Brave `llm-context` `freshness` and date filters, and to reconsider `ui_lang`, so supported Brave parameters can reach the `web_search` LLM Context endpoint.

## Best Possible Solution

Patch the Brave plugin runtime so documented `freshness` and `date_after`/`date_before` values are normalized into Brave's LLM Context `freshness` query parameter, add focused provider tests and docs/changelog updates, and keep `ui_lang` intentionally rejected unless maintainers explicitly choose to support undocumented behavior.

## Reproduction Assessment

Yes. With a Brave API key value configured and Brave mode set to `llm-context`, a `web_search` call containing `freshness`, `date_after`, or `date_before` deterministically returns the local `unsupported_*` payload before any Brave network request.

## Solution Assessment

Partly. Unblocking freshness/date filters in `extensions/brave` is the narrow provider-owned fix, but blindly allowing `ui_lang` is not clearly the best path because the official LLM Context API reference does not list that parameter.
