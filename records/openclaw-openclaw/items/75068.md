## Summary

Keep open. Current main still preflight-rejects Brave `llm-context` `freshness` and date filters before dispatch, while Brave documents `freshness` and custom date ranges for that endpoint; the related PR #51005 targets this but remains open and unmerged.

## What This Changes

The issue asks OpenClaw to stop rejecting Brave `web_search` `llm-context` `freshness`, `date_after`/`date_before`, and possibly `ui_lang`, with optional `count` pass-through.

## Best Possible Solution

Land or replace the existing provider-owned fix so Brave `llm-context` accepts documented freshness/date filters, converts date bounds into Brave freshness ranges with correct cache separation, updates Brave and web-search docs plus changelog, and keeps `ui_lang` rejected or explicitly re-scoped by maintainer decision.

## Reproduction Assessment

Yes. With Brave configured for `mode: "llm-context"`, a `web_search` call containing `freshness`, `date_after`, or `date_before` deterministically returns the local `unsupported_*` payload before any Brave network request.

## Solution Assessment

Unclear for the full report. The narrow best fix is to enable documented freshness/date filters in the active Brave runtime; `ui_lang` and `count` need explicit scope decisions because only `count` is documented for the LLM Context endpoint.
