## Summary

Keep open. Current main still preflight-rejects Brave `llm-context` `freshness`, `date_after`, and `date_before` before the Brave request is sent, while Brave documents `freshness` and custom date ranges for the LLM Context endpoint; the related open PR #51005 targets the same bug but is stale/dirty against the active runtime split.

## What This Changes

The issue requests that Brave `web_search` in `llm-context` mode stop falsely rejecting documented freshness/date filters, consider `count` pass-through, and clarify or reconsider `ui_lang` handling.

## Best Possible Solution

Land, rebase, or replace the existing provider-owned fix in the active Brave runtime so documented `freshness` and date filters reach the LLM Context endpoint with deliberate one-sided range behavior, focused tests, Brave/web docs updates, and the required changelog entry; keep `ui_lang` rejected unless maintainers explicitly choose to support undocumented tolerance.

## Reproduction Assessment

Yes. With any Brave API key value configured and Brave set to `mode: "llm-context"`, a `web_search` call containing `freshness`, `date_after`, or `date_before` deterministically returns the local `unsupported_*` payload before any Brave network request.

## Solution Assessment

Partly. Removing the freshness/date guards and wiring documented ranges through the active runtime is the right narrow fix, but blindly allowing `ui_lang` or copying the existing one-sided date conversion without explicit validation/semantics would not be the best solution.
