## Summary

Keep open. Current main and the latest supplied release still reject Brave `llm-context` freshness and date filters before calling Brave, while Brave's current LLM Context docs list `freshness`, `count`, and custom date ranges; the open related PR is an implementation path, not a merged fix. The `ui_lang` part remains less certain because it is not in Brave's documented LLM Context parameter contract.

## What This Changes

The issue asks OpenClaw to stop falsely rejecting Brave `llm-context` `freshness`, `date_after`/`date_before`, and possibly `ui_lang` parameters in `web_search`.

## Best Possible Solution

Land or replace the provider-owned fix for documented freshness/date support: remove the Brave `llm-context` preflight guards for freshness/date filters, convert `date_after` and `date_before` into Brave freshness ranges, keep cache keys and tests aligned, and update the Brave/web-search docs. Leave `ui_lang` rejected or explicitly documented unless Brave adds it to the LLM Context contract; treat `count` as a small accepted-scope follow-up if maintainers want parity with the documented endpoint.

## Reproduction Assessment

Yes. With Brave configured in `llm-context` mode and an API key present, a `web_search` call containing `freshness`, `date_after`, or `date_before` deterministically returns the local `unsupported_*` payload before any network request.

## Solution Assessment

Partly. The narrow freshness/date direction in PR #51005 is the maintainable fix for the documented Brave contract; unblocking `ui_lang` merely because the API may ignore it is not clearly the best solution without maintainer/API-contract approval.
