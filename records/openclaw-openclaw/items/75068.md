## Summary

Keep open. Current main still rejects Brave `llm-context` `freshness` and date filters before dispatch, while Brave's current LLM Context docs list `freshness` with shortcut and custom range support; the `ui_lang` portion remains less certain because that parameter is not listed for the LLM Context endpoint. ([api-dashboard.search.brave.com](https://api-dashboard.search.brave.com/api-reference/summarizer/llm_context/get)) The related PR #51005 targets the narrow freshness/date fix but is still open, so this issue should not be closed yet.

## What This Changes

The issue asks OpenClaw to stop preflight-rejecting Brave `llm-context` `freshness`, `date_after`/`date_before`, and possibly `ui_lang` parameters for `web_search`, with optional `count` pass-through.

## Best Possible Solution

Land or replace the existing provider-owned fix so Brave `llm-context` accepts documented freshness and date filters, converts date bounds into Brave freshness ranges, keeps cache keys distinct, updates Brave/web-search docs and changelog, and adds focused Brave runtime tests. Keep `ui_lang` rejected or clarify it unless maintainers accept live evidence that passing it through is desirable.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: with Brave configured for `llm-context`, any `web_search` call containing `freshness`, `date_after`, or `date_before` returns the local `unsupported_*` payload before a network request can be made.

## Solution Assessment

Unclear for the full issue, but the best narrow solution is clear for freshness/date filters: remove the Brave `llm-context` guards and wire date ranges into the existing LLM Context request path. `ui_lang` and `count` should be handled as explicit scope choices because Brave documents `count` but not `ui_lang` for this endpoint.
