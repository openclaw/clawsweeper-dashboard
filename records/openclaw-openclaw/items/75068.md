## Summary

Keep open. Current main still rejects Brave `llm-context` `freshness` and date-range filters locally before dispatch, while the upstream Brave LLM Context docs list `freshness` and custom ranges as supported; the related open PR #51005 targets the same bug but is currently dirty/outdated against main, so this should stay open until that PR is repaired, landed, or replaced.

## What This Changes

The issue asks OpenClaw to stop falsely rejecting Brave `llm-context` `freshness` and `date_after`/`date_before`, reconsider `ui_lang`, and optionally wire `count` through to the LLM Context endpoint.

## Best Possible Solution

Repair and land the existing provider-owned fix path, or replace it with a narrow PR against `extensions/brave/src/brave-web-search-provider.runtime.ts` that unblocks documented `freshness` and date ranges, updates the relevant Brave/web docs and tests, and treats `ui_lang` separately because it is not documented by Brave for LLM Context.

## Reproduction Assessment

Yes. With a configured Brave API key and Brave mode set to `llm-context`, calling `web_search` with `freshness`, `date_after`, or `date_before` deterministically returns the local `unsupported_*` response before any Brave network call.

## Solution Assessment

Partly. Removing the local `freshness` and date-range guards and wiring date ranges into the active Brave runtime is the narrow maintainable fix; blindly passing `ui_lang` through is not clearly best because upstream LLM Context docs do not list it.
