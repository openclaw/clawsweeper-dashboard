## Summary

Keep open. Current main still rejects Brave `llm-context` `freshness` and date filters before the Brave request is built, while Brave's current LLM Context documentation lists `freshness`, custom date ranges, and `count`; open PR #51005 targets the narrow freshness/date portion but remains unmerged and appears based on the older pre-runtime file layout.

## What This Changes

The issue asks OpenClaw's Brave `web_search` provider to stop falsely rejecting supported `llm-context` freshness/date parameters, reconsider `ui_lang`, and optionally pass `count` through.

## Best Possible Solution

Land or replace the provider-owned fix so Brave `llm-context` accepts documented freshness/date filters, converts date bounds into Brave freshness ranges with cache separation, updates Brave/web-search docs and changelog, and leaves `ui_lang` and `count` behavior explicit rather than accidental.

## Reproduction Assessment

Yes. With Brave configured for `mode: "llm-context"`, static inspection shows a `web_search` call containing `freshness`, `date_after`, or `date_before` deterministically returns a local `unsupported_*` payload before any Brave network request.

## Solution Assessment

Unclear for the full request. The narrow maintainable fix is in the Brave provider runtime for freshness/date forwarding; `ui_lang` and `count` need explicit scope decisions because their documentation/support status differs.
