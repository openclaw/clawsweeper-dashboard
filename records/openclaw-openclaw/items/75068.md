## Summary

Keep open. Current main still locally rejects Brave `llm-context` `freshness`, `date_after`, and `date_before` before the Brave request is sent, while Brave documents `freshness` and custom date ranges for the LLM Context endpoint; related PR #51005 targets the same bug but is still open. The `ui_lang` part needs a separate maintainer call because Brave's documented LLM Context parameters do not include it.

## What This Changes

Allow Brave `web_search` in `llm-context` mode to accept and forward freshness/date filters, and reconsider or clarify `ui_lang` handling.

## Best Possible Solution

Land, rebase, or replace the provider-owned fix in the active Brave runtime so documented `freshness` and date filters reach the LLM Context endpoint with explicit date-range behavior, focused tests, docs updates, and a changelog entry. Keep `ui_lang` rejected or document it as unsupported unless maintainers intentionally accept Brave's undocumented tolerance.

## Reproduction Assessment

Yes. With Brave configured in `llm-context` mode and any non-empty Brave API key, a `web_search` call containing `freshness`, `date_after`, or `date_before` deterministically returns the local `unsupported_*` payload before any Brave network request.

## Solution Assessment

Unclear. Removing the freshness/date guards and wiring documented Brave ranges through the active runtime is the narrow maintainable path, but allowing `ui_lang` is not clearly the best fix because the official LLM Context parameter list omits it.
