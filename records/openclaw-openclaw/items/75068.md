## Summary

Keep open. Current main still returns local `unsupported_freshness` and `unsupported_date_filter` errors for Brave `llm-context` before reaching Brave, while Brave documents `freshness` and custom date ranges on the LLM Context endpoint. The `ui_lang` part is less certain because Brave does not list it for that endpoint, and open PR #51005 is still the right implementation path to review or replace rather than closing this issue now.

## What This Changes

The issue requests Brave `llm-context` `web_search` to accept documented freshness/date filters, reconsider the `ui_lang` rejection, and optionally pass `count` through to the LLM Context endpoint.

## Best Possible Solution

Keep this issue open until PR #51005 or a replacement lands a provider-local Brave runtime fix: allow documented freshness/date filters in `llm-context`, convert date bounds into Brave freshness ranges, keep cache keys distinct, update Brave/web-search docs and changelog, add focused Brave tests, and make explicit scope decisions for `ui_lang` and `count`.

## Reproduction Assessment

Yes. With Brave configured as `mode: "llm-context"` and an API key available, calling `web_search` with `freshness`, `date_after`, or `date_before` deterministically returns the local `unsupported_*` payload before any fetch; the source path shows that behavior directly.

## Solution Assessment

Unclear for the whole report. The narrow freshness/date fix is best handled inside the Brave provider runtime, but `ui_lang` is not documented by Brave for LLM Context and `count` is additional scope that should be accepted or split deliberately.
