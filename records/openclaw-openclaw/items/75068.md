## Summary

Keep open. Current main still rejects Brave `llm-context` `freshness` and `date_after`/`date_before` before dispatch, while Brave’s current LLM Context docs list `freshness` and custom date ranges; the related PR #51005 targets the narrow fix but is still open. The `ui_lang` portion should remain a separate scope decision because Brave’s LLM Context docs do not list that parameter.

## What This Changes

The issue requests that Brave `llm-context` `web_search` stop rejecting documented freshness/date filters, optionally pass `count`, and reconsider the `ui_lang` rejection.

## Best Possible Solution

Keep this issue open until PR #51005 or a replacement lands the provider-owned fix: remove the llm-context freshness/date guards, convert `date_after`/`date_before` into Brave freshness ranges, keep cache keys distinct, update Brave/web docs and changelog, and add focused Brave tests. Treat `ui_lang` and `count` as explicit scope decisions rather than folding them in accidentally.

## Reproduction Assessment

Yes. With Brave configured in `llm-context` mode, passing `freshness`, `date_after`, or `date_before` deterministically returns the local `unsupported_*` payload before any Brave request; this is visible in `executeBraveSearch`, so a live API key is not required to reproduce the OpenClaw-side failure.

## Solution Assessment

Unclear for the whole report. The narrowest maintainable fix is to unblock documented freshness/date filters in the Brave plugin runtime; `ui_lang` should stay rejected or be changed only after maintainers accept Brave’s undocumented silent-ignore behavior as acceptable.
