## What This Changes

The PR replaces usage-bar alias and map membership checks in `src/auto-reply/usage-bar/translator.ts` with `Object.hasOwn` so inherited Object.prototype keys do not count as table hits.

## Best Possible Solution

Land a small translator fix that uses own-property lookup, adds a prototype-key regression test, and includes redacted proof of the renderer path.

## Reproduction Assessment

Yes. Source inspection shows current main checks user-authored alias and case tables with `in`, so values like `toString` or `constructor` can hit Object.prototype instead of an authored table entry or `_default`; I did not run tests because this review is read-only.

## Solution Assessment

Yes. `Object.hasOwn` is the narrowest maintainable fix for lookup-table membership here; normalizing every template table to a null-prototype object would touch more code for the same invariant.
