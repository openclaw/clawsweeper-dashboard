## Label Justifications

- `P1`: Feishu users are seeing internal reasoning in chat and sometimes unreadable replies, which is a broken user-facing channel workflow with privacy impact.
- `impact:security`: The issue is about exposing internal model reasoning that docs treat as debug-only and potentially sensitive.
- `impact:message-loss`: The production follow-up says the Feishu card can bury or lose the actual answer while thinking tokens flicker or replace content.

## Summary

Keep open: current main has useful guards for flagged reasoning and streamed partial snapshots, but Feishu final/static delivery still appears to pass unflagged `<thinking>` text through, matching the reported workaround that non-streaming card mode still leaks.

## What This Changes

The issue requests that Feishu replies hide model thinking/reasoning content and show only the final answer.

## Best Possible Solution

Sanitize Feishu final/static card and text delivery with the same reasoning-tag stripping used for streamed partials, while keeping explicit `/reasoning stream` previews opt-in and covered by tests plus live Feishu proof.

## Reproduction Assessment

Yes at source level, but not live: current Feishu partial snapshots strip reasoning tags while final/static delivery still forwards unflagged text through `resolveSendableOutboundReplyParts`; the reporters also provided two Feishu/Xiaomi environments.

## Solution Assessment

Yes, the best fix appears to be in Feishu display delivery rather than Xiaomi API plumbing because related provider fixes handle `reasoning_content` passthrough while this report reproduces in the channel card rendering layer.
