## Summary

Keep open. The force-pushed PR now matches `spawnedBy`, but it still enumerates only the parent session store, so normal ACP spawns targeting a different harness agent can remain orphaned after parent reset/delete.

## What This Changes

The PR adds gateway reset/delete cleanup that closes direct child ACP session entries found in the parent store, plus reset/delete/concurrency tests and a changelog entry.

## Best Possible Solution

Keep the concurrent best-effort close shape, but enumerate ACP-backed children across resolved agent stores and close direct children matched by `spawnedBy` or `parentSessionKey`, with reset/delete tests and refreshed real-path proof.

## Reproduction Assessment

Yes, source-reproducible: current main closes only the reset/delete target ACP key, while `sessions_spawn(runtime="acp")` writes child ACP rows with `spawnedBy` under the target ACP agent key/store. I did not run a live multi-hour process leak reproduction in this read-only review.

## Solution Assessment

No: the current PR is still too narrow because it only scans the parent store. The maintainable fix should use the existing all-agent ACP enumeration or equivalent target-store resolution before closing children.
