## Summary

Keep open: the patch is aimed at the right gateway/ACP cleanup gap, but as written it misses the actual `sessions_spawn(runtime: "acp")` child rows and adds a serial timeout risk to reset/delete.

## What This Changes

The PR lists session-store entries during gateway reset/delete, closes matching child ACP runtimes, and adds a reset regression test plus changelog entry.

## Best Possible Solution

Match the actual ACP child lineage contract (`spawnedBy` and any supported `parentSessionKey` rows), close children with bounded best-effort latency, and cover both reset and delete using a regression shape that mirrors real ACP `sessions_spawn` rows.

## Reproduction Assessment

Yes. Source inspection shows current main closes only one target ACP session during reset/delete, while the real ACP spawn path stores child ownership in `spawnedBy`; the PR's filter misses that path.

## Solution Assessment

No. Enumerating child ACP sessions is the right direction, but this implementation filters on the wrong lineage field and waits serially on best-effort child cleanup.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Match ACP spawnedBy children during cleanup:** `src/gateway/session-reset-service.ts:582`
  - body: The ACP `sessions_spawn` path creates the child row with `spawnedBy: requesterInternalKey`, and the subagent child patch only adds depth/role fields. This filter only accepts `entry.parentSessionKey === parentKey`, so the reported run-mode ACP children are skipped and the new test passes only because it seeds a shape the spawn path does not write. Include `entry.spawnedBy === parentKey` or normalize the lineage before closing.
  - confidence: 0.96
- **[P2] Bound child cleanup latency:** `src/gateway/session-reset-service.ts:586-592`
  - body: This awaits each child close in sequence. `closeAcpRuntimeForSession` can wait the cleanup timeout for cancel and again for close, so a parent with many stuck ACP children can spend tens or hundreds of seconds in reset/delete before returning even though child cleanup is meant to be best-effort. Run children concurrently under a bounded aggregate or detach late cleanup while preserving the parent result.
  - confidence: 0.86
