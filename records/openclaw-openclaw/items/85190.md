## Summary

Keep open: the PR fixes a real gateway/ACP cleanup gap, but the new child cleanup loop awaits every child close in series, so stuck child ACP sessions can make reset/delete stall for one cleanup timeout per child before responding.

## What This Changes

The PR adds child ACP session enumeration/cleanup during gateway session reset/delete, a reset regression test, and a changelog entry.

## Best Possible Solution

Keep the direct-child ACP cleanup, but make it genuinely bounded and best-effort so parent reset/delete latency cannot scale with the number of stuck children, then cover that timeout behavior in tests.

## Reproduction Assessment

Yes, from source inspection: current main closes only the parent ACP runtime during `cleanupSessionBeforeMutation`, while child session rows are stored with `parentSessionKey`. I did not run a live process repro in this read-only review.

## Solution Assessment

No, not yet: enumerating direct ACP children is the right fix shape, but awaiting each child close sequentially is not the best way to keep reset/delete best-effort and bounded.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Bound child ACP cleanup latency:** `src/gateway/session-reset-service.ts:588-593`
  - body: `closeChildAcpRuntimesForParent` awaits every child close in series before the reset/delete mutation can finish. Because `closeAcpRuntimeForSession` has 15s timeout paths and the reported failure mode includes many orphaned ACP children, a stuck set can make `/reset` or `sessions.delete` hang for minutes even though this cleanup is described as best-effort. Please run the child closes concurrently under a shared bound, or enqueue them in background, while preserving the parent close result.
