## Summary

Keep open: the PR addresses the earlier `spawnedBy` lineage gap, but the cleanup helper still scans only the parent session store while ACP spawns can persist child rows in the target agent's store, so normal cross-agent `sessions_spawn(runtime="acp")` children can remain orphaned. The supplied terminal proof exercises synthetic rows in one store, not the actual reset/delete path with target-agent store routing.

## What This Changes

The PR adds gateway reset/delete cleanup that enumerates matching child ACP entries, closes them concurrently, and adds reset/delete/concurrency tests plus a changelog entry.

## Best Possible Solution

Keep the concurrent best-effort cleanup shape, but enumerate the relevant agent session stores and close direct ACP children matched by `spawnedBy` or `parentSessionKey`, with reset/delete tests and proof covering cross-agent ACP spawns.

## Reproduction Assessment

Yes, source-reproducible: current main closes only the reset/delete target ACP entry, while ACP spawn creates child keys under `targetAgentId` and records the requester in `spawnedBy`. I did not run the multi-hour memory-growth scenario in this read-only review.

## Solution Assessment

No: the proposed cleanup direction is close, but scanning only the parent store is not sufficient for the actual ACP spawn storage contract. The maintainable fix should use existing agent store discovery or equivalent routing so target-agent child entries are selected and closed.
