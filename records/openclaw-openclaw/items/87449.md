## Summary

Keep open: the PR is a focused Mattermost message-loss fix, but the new boundary rotation can still drop or overwrite a block when the previous draft update is pending or in flight, so it is not merge-ready yet.

## What This Changes

This PR adds a Mattermost draft-preview boundary controller, wires block-mode text/tool boundaries to create new preview posts, and lets the off-preview inbound path honor explicit block streaming.

## Best Possible Solution

Make Mattermost block-preview boundaries flush or await the current draft update before forcing a new post, add delayed-request regression coverage, and then verify the visible Mattermost path.

## Reproduction Assessment

Yes. Current main source shows the inbound Mattermost preview path edits a single draft post and hard-disables block streaming; the linked report and PR proof give concrete off-preview and block-mode sequences, though I did not run a live Mattermost server here.

## Solution Assessment

No. The direction is right and keeps the existing config surface, but synchronous boundary rotation is not the safest fix because it can clear or race pending draft delivery before starting the next post.

- **[P1] Flush the draft before rotating preview posts:** `extensions/mattermost/src/mattermost/draft-stream.ts:74-75`
  - body: `noteBoundary()` can run while the prior draft update is still pending or its `POST /posts` is in flight. Because `forceNewMessage()` clears pending text and resets the current post id without waiting, a fast tool/assistant boundary can either drop the previous block or let the in-flight create set `streamPostId` after the boundary, causing the next block to edit that post instead of creating a separate Mattermost post. Make the boundary path await/flush the current draft before forcing a new message and add a delayed-request regression test.
