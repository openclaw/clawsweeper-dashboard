## Summary

Keep open: the PR is still relevant because current main does not preserve Mattermost block-preview boundaries, but the new boundary flush can duplicate posts when another cumulative partial update is pending while the first Mattermost POST is still in flight.

## What This Changes

The PR changes Mattermost inbound draft preview so block mode rotates posts at text/tool boundaries, preview-off blockStreaming can send normal block replies, and docs/tests plus a draft-loop helper are updated.

## Best Possible Solution

Fix the pending-create boundary race, then land one canonical Mattermost repair after maintainers accept the visible-post compatibility change and the SDK helper surface.

## Reproduction Assessment

Yes, source-level: current main keeps one Mattermost draft post for non-off preview modes and forces block streaming off on the inbound path. The new PR-specific race is also source-reproducible from the cumulative partial-reply contract plus the delayed create path.

## Solution Assessment

No, not yet: the PR targets the right owner boundary, but the current forceNewMessage implementation can create a duplicate sealed post under a slow create plus pending partial. The safer shape is to preserve or await the created post id before deciding whether the sealed pending text is an edit or a create.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P1] Seal pending partials onto the in-flight post:** `extensions/mattermost/src/mattermost/draft-stream.ts:167-168`
  - body: When the first Mattermost `POST /posts` is still in flight, another cumulative `onPartialReply` can leave newer text pending for the same block. This code clears `streamPostId` and bumps `postGeneration` before waiting, so the in-flight create is not allowed to install its returned id; the boundary then sees no `sealedPostId` and sends the pending text as a second `POST` instead of editing the just-created post. A slow server can leave users with both a stale partial post and the sealed block post, so the boundary flush needs to retain the created id or wait before deciding create vs edit.
  - confidence: 0.86
