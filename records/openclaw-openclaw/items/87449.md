## Summary

Keep open: the PR addresses a real Mattermost message-loss path, but the current patch can duplicate a block under delayed first-post creation, and the remaining compatibility/SDK-surface choices still need maintainer acceptance before merge.

## What This Changes

The PR changes Mattermost draft-preview delivery, docs, and tests so block-mode previews rotate posts at text/tool boundaries, preview-off `blockStreaming` is honored, and `DraftStreamLoop` exposes pending-text capture.

## Best Possible Solution

Land one repaired Mattermost-local boundary fix that preserves one visible post per completed block under delayed creates, keeps the existing streaming config contract, and explicitly tracks the separate error-response deletion follow-up.

## Reproduction Assessment

Yes. Current main and v2026.6.6 still force the inbound Mattermost path through one editable draft post and `disableBlockStreaming: true`, and the PR body supplies before/after HTTP output for the production client path.

## Solution Assessment

No as-is. The Mattermost-local boundary direction is the right layer, but the patch needs a delayed-create plus pending-partial repair before it is the best fix.

- **[P2] Avoid duplicating a block when the first create is still in flight:** `extensions/mattermost/src/mattermost/draft-stream.ts:168`

Add that delayed-create + pending-partial regression and keep one visible post per completed block.
