## Summary

Keep open: this is a focused Mattermost message-loss fix with useful after-fix proof, but the patch currently rotates preview posts for both tool start and tool update phases, which can over-split repeated tool progress updates into extra Mattermost posts.

## What This Changes

The PR adds Mattermost draft-preview boundary rotation for existing `streaming.mode: "block"`, restores block replies when preview streaming is off, and adds focused draft-stream tests.

## Best Possible Solution

Keep the existing config contract and default behavior, but rotate Mattermost block preview posts only at real text/tool block boundaries while keeping repeated tool progress updates on the same preview post.

## Reproduction Assessment

Yes from source inspection: current main forces `disableBlockStreaming: true` on the inbound Mattermost preview path and the draft stream edits one `streamPostId` until cleared. I did not run a live Mattermost server, but the PR body's loopback output demonstrates the same POST-vs-PUT failure mode at the API boundary.

## Solution Assessment

No as-is. The overall approach is narrow and uses the existing `streaming.mode` contract, but the tool boundary needs to avoid rotating posts for `phase: "update"` events before this is the best merge shape.

  - body: `onToolStart` is invoked for both `phase: "start"` and `phase: "update"` from the reply runner, but this new boundary runs before checking the phase.
