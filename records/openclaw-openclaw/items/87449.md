## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep boundary handling synchronous before awaiting:** `extensions/mattermost/src/mattermost/monitor.ts:1950`
  - body: The embedded subscriber dispatches `onAssistantMessageStart`/`onReasoningEnd` with `void`, so this callback can resume after a later block has already called `noteUpdate()` or updated `lastPartialText`. Awaiting the split before resetting local state can clobber the newer block's partial state; `onToolStart` has the same ordering issue because progress is not emitted until the old post seal completes, so a fast next block can create the draft first and then get overwritten by late tool progress. Start the split synchronously, update new-block state/progress immediately, and only then await/catch the split without mutating newer state.
  - confidence: 0.88
