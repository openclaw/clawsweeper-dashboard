## Summary

Keep open: the underlying command-normalization regression is valid, but this PR does not yet prove or fully fix the real runtime path because session setup still collapses command-payload newlines before dispatch, and the external proof is unit-test-only.

## What This Changes

The branch reattaches text after the first newline during slash-command normalization and adds skill-command regression tests plus a changelog entry.

## Best Possible Solution

Land a narrow fix that preserves multiline command payloads through the real channel-turn/session command path for `/skill` and direct skill commands, with runtime-path regression coverage and real behavior proof.

## Reproduction Assessment

Yes from source inspection: current main drops the newline tail in `normalizeCommandBody()`, while the skill parser can accept multiline args if they reach it. I did not execute tests because this was a read-only review.

## Solution Assessment

No. Reattaching the tail in the normalizer is part of the right fix, but the PR's command-context test bypasses session setup, where command bodies are still flattened before skill dispatch.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Preserve newlines through session setup:** `src/auto-reply/reply/commands-context.test.ts:75`
  - body: This test feeds the raw Telegram-style body directly into `buildCommandContext`, but the real reply path first computes `triggerBodyNormalized` with `stripStructuralPrefixes(commandSource)`, which collapses all whitespace to spaces. A runtime command like `/skill@openclaw demo\nfirst\nsecond` can still reach skill dispatch without the multiline structure this PR is trying to preserve, so please cover the session/init path and keep command payload newlines intact there.
  - confidence: 0.86
- **[P2] Update command parsers for multiline aliases:** `src/auto-reply/commands-registry-normalize.ts:86`
  - body: Reattaching the tail for every arg-accepting alias makes cases such as `/side\nfirst\nsecond` normalize to `/btw\n...`, but existing handlers like `extractBtwQuestion()` still match arguments with `.*` and will not parse multiline text once those newlines survive. Either narrow the preservation to skill dispatch or update the affected arg parsers and handler tests.
  - confidence: 0.79
