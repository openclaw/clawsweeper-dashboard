## Summary

Current main still has the reported mismatch: the WhatsApp listener-down error is returned to the caller while the queued delivery remains pending for recovery. Because an open linked PR already uses closing syntax for this issue, the issue should stay open until that PR is reviewed, proven, and merged or closed.

## What This Changes

The issue reports duplicate WhatsApp group posts when sends that returned UNAVAILABLE remain in delivery recovery and replay after gateway restart.

## Best Possible Solution

Review and land the linked fix PR if its implementation and real behavior proof check out; otherwise replace it with the same narrow policy that a listener-down UNAVAILABLE send is terminal to recovery.

## Reproduction Assessment

Yes. Source inspection on current main shows the exact path: enqueue before dispatch, WhatsApp throws a plain listener-unavailable error, failDelivery leaves the entry pending, and recovery later replays pending entries.

## Solution Assessment

Yes. Treating listener-down UNAVAILABLE as terminal for the recovery queue is the narrowest maintainable fix because it preserves the caller-facing failure semantics and keeps recovery focused on uncertain/crash-after-send cases.
