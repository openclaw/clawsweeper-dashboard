## Summary

Current main still appears to have the reported bug: listener-down WhatsApp sends throw an ordinary error after the durable entry is enqueued, and delivery recovery can replay pending entries. Keep this issue open because https://github.com/openclaw/openclaw/pull/79388 is an open closing PR that still needs review/rebase/proof rather than pre-merge closure.

## What This Changes

The issue asks OpenClaw to stop silently replaying WhatsApp sends that already returned caller-visible UNAVAILABLE failures.

## Best Possible Solution

Review, rebase, and validate https://github.com/openclaw/openclaw/pull/79388 or an equivalent replacement so listener-down WhatsApp sends are terminal to delivery recovery while crash/unknown-outcome replay remains intact.

## Reproduction Assessment

Yes, as a source-level reproduction rather than a live WhatsApp run. Current main enqueues before dispatch, records the listener-down error with failDelivery, and recovery later retries pending entries, matching the reporter's logs.

## Solution Assessment

Yes. Making the listener-down dispatch failure terminal for the queue is the narrow maintainable fix; changing the caller-visible semantics to deferred delivery would be a broader protocol/product change.
