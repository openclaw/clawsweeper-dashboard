## Summary

Keep open. Current main still has the reported mismatch: WhatsApp listener-unavailable sends return `UNAVAILABLE` to the caller while the queued entry remains eligible for recovery, and the linked fix PR is open but unmerged.

## What This Changes

Ensure WhatsApp listener-unavailable `UNAVAILABLE` sends are not silently replayed by delivery-recovery after restart or reconnect.

## Best Possible Solution

Make listener-unavailable WhatsApp dispatch a terminal queue outcome, keep crash/unknown-after-send recovery intact, and cover both new dispatches and legacy pending entries with regression tests.

## Reproduction Assessment

Yes. The current-main source path is high-confidence: enqueue happens before dispatch, the missing WhatsApp listener throws a plain error, `failDelivery` preserves the pending entry, and recovery/reconnect drain can replay it.

## Solution Assessment

Yes, the open linked PR appears to target the narrow right boundary by making listener-down failures terminal while preserving recovery for crash/unknown-send cases. The issue should remain open until that PR lands or is replaced.
