## Summary

Keep open. Current main still keeps listener-unavailable WhatsApp sends in the durable queue and recovery/reconnect drain can replay them, while https://github.com/openclaw/openclaw/pull/79388 is an open closing PR, so this issue should stay open until that PR lands or is rejected.

## What This Changes

The issue reports duplicate WhatsApp group posts because delivery recovery replays sends that already returned UNAVAILABLE to the caller.

## Best Possible Solution

Keep this issue paired with https://github.com/openclaw/openclaw/pull/79388 and land a fix that makes the caller-visible send result and durable recovery state agree without regressing reconnect recovery for genuinely deferred background sends.

## Reproduction Assessment

Yes from source inspection and reporter logs. A live WhatsApp run was not attempted in this read-only review, but current main still records the UNAVAILABLE failure as a pending queue entry that recovery or reconnect drain can later deliver.

## Solution Assessment

Yes, the issue should remain open while the linked closing PR is reviewed. The narrow fix direction is to make pre-send listener-unavailable failures terminal to recovery, or otherwise report a deferred status to callers, with regression coverage for the old reconnect-drain behavior.
