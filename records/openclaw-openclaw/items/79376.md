## Summary

Keep this open: current main still has a source-reproducible path where a WhatsApp listener-unavailable send is recorded as a retryable pending delivery and can later be replayed, and the linked fix PR is still open rather than merged.

## What This Changes

The issue reports that WhatsApp sends returning UNAVAILABLE can remain pending in delivery recovery and later replay as duplicate posts after reconnect or restart.

## Best Possible Solution

Review, update, and land the linked fix PR or an equivalent targeted change that makes listener-unavailable sends either terminal for recovery or explicitly deferred to callers.

## Reproduction Assessment

Yes, from source inspection and the provided live logs. Current main records a WhatsApp listener-unavailable failure with failDelivery, leaves it pending, and both startup recovery and the WhatsApp reconnect drain can replay it later.

## Solution Assessment

Yes, the linked terminal-failure direction is the narrowest maintainable fix because it keeps the caller-visible UNAVAILABLE response consistent with the recovery queue state. The issue should stay open until that PR or an equivalent fix merges.
