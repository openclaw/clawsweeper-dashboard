## Summary

Keep open: current main and the latest release still allow a configured primary TTS provider to fall through to another configured provider, while the linked merged PRs only added diagnostics, compatibility, and adjacent provider registration fixes. The remaining question is a maintainer policy decision because current docs describe other configured providers as fallback options.

## What This Changes

The issue reports that a configured ElevenLabs TTS provider is invoked but OpenClaw ultimately plays OpenAI audio instead.

## Best Possible Solution

Define the explicit-provider TTS fallback contract, then implement it consistently across auto TTS, `/tts audio`, agent tool, streaming, telephony, status output, and docs.

## Reproduction Assessment

Yes from source for the remaining behavior: current main still builds a fallback provider chain and can return a later provider after the configured primary fails or is skipped. The exact ElevenLabs account/API/playback symptom was not live-reproduced.

## Solution Assessment

No final fix is present yet. The diagnostics and registration fixes are useful, but the best fix needs a maintainer decision on whether explicit provider selection should disable cross-provider fallback or keep it as documented fallback behavior.
