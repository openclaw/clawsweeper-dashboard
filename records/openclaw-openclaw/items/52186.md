## Summary

Keep open. Current main has fallback diagnostics and bundled ElevenLabs registration, but ordinary Auto-TTS and `/tts audio` still allow a configured ElevenLabs primary attempt to fall through to another provider; changing that into strict selected-provider behavior is a product/config policy decision, not conservative cleanup.

## What This Changes

The issue requests that configured ElevenLabs TTS audio be used instead of OpenClaw playing OpenAI audio after ElevenLabs appears to process the request.

## Best Possible Solution

Define an explicit TTS provider fallback/stickiness policy, likely with a config-backed strict-provider mode, then align Auto-TTS, `/tts audio`, gateway conversion, docs, and regression tests.

## Reproduction Assessment

Yes for the user-visible fallback symptom: source inspection shows a configured primary provider can be skipped or fail and `synthesizeSpeech` can return a later provider with `fallbackFrom`. The exact ElevenLabs-portal success claim was not live-reproduced.

## Solution Assessment

No. The merged diagnostics and registration fixes are useful but do not decide whether selected TTS providers should fail closed or continue automatic fallback; that policy needs maintainer direction first.
