## Summary

Keep open. Current main has better fallback diagnostics and provider registration fixes, but config-selected Auto-TTS and /tts audio still use the fallback provider chain, so the remaining provider-stickiness behavior is unresolved and needs maintainer policy judgment plus live/provider proof.

## What This Changes

The issue asks OpenClaw TTS to play ElevenLabs audio when `messages.tts.provider` is set to `elevenlabs` instead of falling through to OpenAI.

## Best Possible Solution

Define and document the TTS fallback/stickiness contract, preferably preserving documented fallback by default while adding an explicit strict-provider mode if needed, then update Auto-TTS, /tts audio, gateway status/conversion, diagnostics, docs, and regression tests consistently.

## Reproduction Assessment

Yes for the source-level fallback symptom: a configured provider can fail or be skipped, and Auto-TTS plus /tts audio can return a later configured provider with `fallbackFrom` set. No live reproduction proved the reporter's exact ElevenLabs-success-then-OpenAI-playback path.

## Solution Assessment

Unclear until maintainers choose the fallback policy. Diagnostics-only work is not the full fix, but an unconditional strict-provider change would conflict with the documented fallback contract and may need an explicit config mode.
