## Summary

Keep open. Current main has the diagnostics and provider-registration fixes, but ordinary Auto-TTS and `/tts audio` still allow an ElevenLabs primary failure or skip to fall through to another configured provider such as OpenAI; deciding whether that should fail closed is a maintainer product-policy call.

## What This Changes

The issue requests that configured ElevenLabs TTS playback not end up using the OpenAI voice after an ElevenLabs attempt.

## Best Possible Solution

Choose an explicit TTS fallback policy for selected providers and apply it consistently across Auto-TTS, `/tts audio`, gateway conversion, diagnostics, docs, and regression tests.

## Reproduction Assessment

Yes for the user-visible fallback symptom: source shows a configured primary provider can fail or be skipped and `synthesizeSpeech` can return a later fallback provider with `fallbackFrom` set. The exact ElevenLabs-portal success claim was not live-reproduced.

## Solution Assessment

No. The merged diagnostics and registration changes are useful, but the remaining issue needs a maintainer decision on whether selected TTS providers should fail closed or keep automatic fallback behavior.
