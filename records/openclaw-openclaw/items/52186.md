## Summary

Keep open: current main has the diagnostic pieces from the linked TTS PRs, but it still allows a configured primary TTS provider to fall through to another configured provider when the primary attempt is skipped or fails. The remaining provider-stickiness/fallback-policy behavior is real and was also treated as the canonical remaining work by the related closed issue.

## What This Changes

The issue asks OpenClaw to play the configured ElevenLabs TTS audio instead of silently or unexpectedly using an OpenAI fallback voice.

## Best Possible Solution

Define the TTS fallback contract in speech-core, then implement it consistently across auto TTS, command TTS, streaming, and telephony so configured-provider behavior is either sticky/fail-visible or intentionally fallback-capable with clear user control.

## Reproduction Assessment

Yes, source-reproducible but not live-reproduced: current main still builds a fallback provider chain for TTS and can return a later provider as a successful attempt. The exact ElevenLabs account/API/playback failure mode from the report still needs live confirmation.

## Solution Assessment

No final fix is present yet. The shipped diagnostics are useful but not sufficient for the expected behavior; the best fix needs a maintainer decision on whether explicit provider selection should disable cross-provider fallback or expose that as an intentional policy.
