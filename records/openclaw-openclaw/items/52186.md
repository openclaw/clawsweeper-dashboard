## Summary

Keep open. Current main has the fallback diagnostics and ElevenLabs registration fixes, but the selected TTS provider can still fall through to another configured provider, so the remaining stickiness behavior is not implemented and needs a maintainer policy decision.

## What This Changes

The issue requests that TTS play the configured ElevenLabs voice instead of falling back to OpenAI after ElevenLabs activity is observed.

## Best Possible Solution

Choose and document an explicit TTS fallback/stickiness policy, then apply it consistently across Auto-TTS, /tts audio, gateway conversion/status, diagnostics, and regression tests.

## Reproduction Assessment

Yes for the fallback symptom from source inspection: a configured primary provider can fail or be skipped and synthesizeSpeech can return a later provider with fallbackFrom set. The reporter's exact ElevenLabs portal-success path was not live-reproduced.

## Solution Assessment

Unclear until maintainers choose the intended fallback policy. The current diagnostics-only implementation is not the full fix, but a one-off automated change would risk contradicting the documented fallback contract.
