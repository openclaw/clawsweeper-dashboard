## Summary

Keep open: current main and the latest release still intentionally try fallback TTS providers after the selected provider fails or is skipped, while the linked merged PRs only added diagnostics, compatibility, and ElevenLabs registration fixes. The remaining work is a maintainer policy decision about whether explicit/configured TTS providers should be sticky or whether the documented fallback behavior should remain.

## What This Changes

The issue reports that ElevenLabs is configured and receives TTS requests, but OpenClaw ultimately plays audio from a fallback provider such as OpenAI.

## Best Possible Solution

Make an explicit maintainer decision for TTS provider stickiness versus documented fallback, then update speech-core behavior, status output, docs, and regression coverage together.

## Reproduction Assessment

No high-confidence live reproduction is available for the original Ubuntu/ElevenLabs setup. Source inspection does show that current main can fall through from a configured primary TTS provider to another configured provider when the primary attempt fails or is skipped.

## Solution Assessment

Unclear until maintainers decide the policy. The existing diagnostics and registration fixes are useful partial fixes, but the best remaining solution must explicitly choose provider stickiness or preserve documented fallback behavior.
