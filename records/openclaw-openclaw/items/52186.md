## Summary

Keep open. Current main and v2026.6.1 still use the configured TTS provider as the first entry in a fallback chain; the merged related work improved diagnostics and compatibility, but did not resolve the provider-stickiness/fallback-policy behavior reported here.

## What This Changes

The issue asks OpenClaw TTS to play the configured ElevenLabs voice instead of falling back to an OpenAI voice after an ElevenLabs attempt.

## Best Possible Solution

Define a maintainer-approved TTS fallback policy that makes configured-provider behavior predictable, then update speech-core, status output, tests, and docs to match that policy.

## Reproduction Assessment

Yes for the remaining fallback behavior from source inspection: current synthesis still tries later providers after a primary failure or skip and can report the later provider as successful. No live ElevenLabs/OpenAI run was performed for the reporter's exact setup.

## Solution Assessment

Unclear. Diagnostics are already improved, but forcing sticky ElevenLabs behavior or adding a strict no-fallback mode is a product and compatibility decision because fallback providers are currently documented behavior.
