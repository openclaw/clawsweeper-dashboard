## Summary

Keep open. Current main and the latest release have better fallback diagnostics, but the remaining complaint is still unresolved: configured Auto-TTS and `/tts audio` can still try later speech providers after the selected provider fails or is skipped, and changing that behavior needs maintainer policy judgment.

## What This Changes

The issue asks OpenClaw TTS to play the configured ElevenLabs voice instead of falling through to OpenAI when ElevenLabs is selected and processing requests.

## Best Possible Solution

Define and document the TTS fallback/stickiness contract, preserving documented fallback by default unless maintainers choose an explicit strict-provider mode, then update Auto-TTS, `/tts audio`, gateway conversion/status, diagnostics, docs, and regression tests consistently.

## Reproduction Assessment

Yes for the source-level fallback symptom: configured Auto-TTS and `/tts audio` can still return a later provider with `fallbackFrom` set. No live run reproduced the reporter's exact ElevenLabs-success-then-OpenAI-playback path.

## Solution Assessment

Unclear until maintainers choose the fallback policy. Diagnostics-only work is not the full fix, but unconditional strict-provider behavior would conflict with the documented fallback contract unless introduced as an explicit mode or policy change.
