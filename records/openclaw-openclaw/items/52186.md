## Summary

Keep open. Current main has the diagnostics and provider-registration fixes, but Auto-TTS and `/tts audio` still let a selected ElevenLabs primary failure or skip fall through to another configured provider such as OpenAI; the remaining fail-closed/stickiness behavior needs maintainer policy judgment.

## What This Changes

The issue asks OpenClaw TTS to play ElevenLabs audio when ElevenLabs is the configured provider instead of silently falling through to another provider such as OpenAI.

## Best Possible Solution

Choose and document an explicit TTS fallback policy, then apply it consistently across Auto-TTS, `/tts audio`, gateway conversion/status, diagnostics, and regression tests.

## Reproduction Assessment

Yes for the fallback symptom from source inspection: a configured primary provider can fail or be skipped and `synthesizeSpeech` can return a later provider with `fallbackFrom` set. The reporter's exact ElevenLabs-portal success path was not live-reproduced.

## Solution Assessment

No, the current partial fix is not the full solution because it improves visibility but keeps automatic fallback behavior. The narrow maintainable next step is a maintainer-approved fallback/stickiness policy rather than an automated one-off code change.
