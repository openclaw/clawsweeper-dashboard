## Summary

Keep open. Current main and v2026.5.22 still default to a TTS fallback provider chain; diagnostics and explicit gateway conversion improved, but Auto-TTS and `/tts audio` can still surface OpenAI as the successful provider after configured ElevenLabs fails or is skipped, and changing that behavior needs a maintainer fallback-policy decision.

## What This Changes

The issue asks OpenClaw TTS to honor configured ElevenLabs playback instead of producing or playing OpenAI audio after an ElevenLabs attempt.

## Best Possible Solution

Define and document the TTS fallback/stickiness contract, preserve documented fallback by default unless maintainers choose an explicit strict-provider mode, then update Auto-TTS, `/tts audio`, gateway conversion/status, diagnostics, docs, and regression tests consistently.

## Reproduction Assessment

Yes for the source-level fallback symptom: configured Auto-TTS and `/tts audio` can still return a later provider with `fallbackFrom` set. No live run reproduced the reporter's exact ElevenLabs-success-then-OpenAI-playback path.

## Solution Assessment

Unclear until maintainers choose the fallback policy. Diagnostics-only work is not the full fix, but unconditional strict-provider behavior would conflict with the documented fallback contract unless introduced as an explicit mode or policy change.
