## Summary

Keep open. Current main has fallback diagnostics and ElevenLabs registration fixes, but automatic TTS still allows the selected provider to fall through to another configured provider, so the remaining provider-stickiness behavior is unresolved and needs maintainer policy judgment.

## What This Changes

The issue asks OpenClaw TTS to play the configured ElevenLabs provider output instead of a fallback OpenAI voice when ElevenLabs is selected and API activity is observed.

## Best Possible Solution

Choose and document the intended TTS fallback/stickiness contract, then implement it consistently across Auto-TTS, /tts audio, gateway conversion/status, diagnostics, and regression tests.

## Reproduction Assessment

Yes for the fallback symptom from source inspection: a configured primary provider can be skipped or fail and synthesizeSpeech can return a later configured provider with fallbackFrom set. The reporter's exact ElevenLabs portal-success path was not live-reproduced.

## Solution Assessment

Unclear until maintainers choose the intended fallback policy. A diagnostics-only implementation is not the full fix, but an automated strict-provider change would risk contradicting the documented fallback contract.
