## Summary

Keep open. Current main has the diagnostics and provider-registration improvements, but ordinary automatic TTS and `/tts audio` still allow a configured ElevenLabs primary to fall through to another configured provider after a primary failure, matching the remaining behavior called out in the discussion.

## What This Changes

The issue requests that configured ElevenLabs TTS not result in OpenAI voice playback after ElevenLabs is selected and called.

## Best Possible Solution

Choose an explicit TTS provider fallback policy and implement it consistently across automatic replies, `/tts audio`, gateway conversion, status diagnostics, docs, and regression tests. If automatic fallback should remain the default, add a documented strict-provider or fail-closed option for users who need ElevenLabs pinning and billing predictability.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to configure ElevenLabs as the selected primary with another provider configured, then force the primary synthesis/readiness path to fail; current source and contract tests show `synthesizeSpeech` can return the fallback provider with `fallbackFrom` set. The original live ElevenLabs portal scenario was not re-run.

## Solution Assessment

No. The merged diagnostics and registration fixes are useful, but they do not fully solve the reported behavior because configured automatic and slash TTS paths still allow provider fallback by design.
