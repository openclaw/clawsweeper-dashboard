## Summary

Keep open. Current main still lacks an explicit update subprocess PATH repair for launchd-minimal Homebrew npm updates, so the reported `env node` shim failure remains a valid narrow updater reliability bug even though adjacent package-update restart behavior has been hardened.

## What This Changes

The issue requests that macOS launchd-triggered self-updates for Homebrew npm installs resolve Node correctly when npm is invoked from a minimal launchd PATH.

## Best Possible Solution

Add a shared, tested update subprocess env builder used by both gateway and CLI global package-update paths. It should preserve service PATH hardening while adding the active runtime bin, the owning npm prefix bin, Homebrew runtime/bin candidates, and system dirs needed for npm's `env node` shebang, without adding project-local bins.

## Reproduction Assessment

Yes. A Homebrew global npm install launched from launchd's minimal PATH is a high-confidence reproduction path, and current source inspection shows the update subprocess env still lacks explicit Node/Homebrew PATH repair for that case.

## Solution Assessment

No. The issue's hard-coded shell `PATH` export would likely avoid the immediate failure, but the best fix is a shared env helper at the global package-update boundary with regression coverage for gateway and CLI paths.
