## Summary

Keep open. The report is concrete and still maps to a narrow updater reliability bug: current main can choose the owning Homebrew npm shim, but the shared global-update environment builder does not add the active runtime bin, owning npm prefix bin, or Homebrew Node/bin paths when the update subprocess starts from a launchd-minimal PATH.

## What This Changes

The issue requests that macOS launchd-triggered self-updates from Homebrew npm installs provide a PATH that lets npm's `env node` shim resolve Node instead of looping with npm exit 127.

## Best Possible Solution

Add a shared, tested update subprocess env builder and use it from gateway and CLI global package-update paths so minimal launchd environments get the active runtime bin, owning npm prefix bin, Homebrew Node/bin locations, and system dirs without adding project-local bins or weakening service PATH hardening.

## Reproduction Assessment

Yes. A macOS Homebrew global npm install with PATH `/usr/bin:/bin:/usr/sbin:/sbin`, package root `/opt/homebrew/lib/node_modules/openclaw`, and owning npm `/opt/homebrew/bin/npm` should exercise the failing npm shim path; current source inspection shows no update env repair for that case.

## Solution Assessment

No. The issue's hard-coded shell PATH export would likely avoid the immediate failure, but the maintainable fix is a shared update env helper used by every global package-update subprocess path with focused regression coverage.
