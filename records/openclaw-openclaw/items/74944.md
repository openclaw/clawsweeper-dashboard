## Summary

Keep open. The issue remains a valid, narrow updater reliability bug: current main still passes a global-install subprocess env that does not add the active Node runtime directory, the owning npm prefix bin, or Homebrew Node/bin paths needed under launchd's minimal PATH, so an absolute Homebrew npm shim can still fail resolving `env node`.

## What This Changes

The issue requests making macOS launchd-submitted OpenClaw self-updates work on Homebrew Node/npm installs instead of failing with `env: node: No such file or directory` and npm exit 127.

## Best Possible Solution

Add a shared update subprocess env builder and use it from both gateway and CLI global package-update paths. It should include the active runtime bin, owning npm prefix bin, Homebrew Node/bin candidates, and system dirs while avoiding project-local bins and preserving the existing service PATH hardening posture.

## Reproduction Assessment

Yes. A Homebrew global npm install with a launchd-style PATH of `/usr/bin:/bin:/usr/sbin:/sbin` and an absolute `/opt/homebrew/bin/npm` command exercises the failing npm shim path; current code inspection shows no env repair for that case.

## Solution Assessment

No. The issue's hard-coded shell `export PATH=...` workaround addresses the immediate failure, but the maintainable solution is a shared, tested env helper used by every global package-update subprocess path.
