## Summary

Keep open. Current main still routes gateway and CLI global package updates through an env builder that does not add the active runtime directory, owning npm prefix bin, or Homebrew bin paths for launchd-minimal environments, so the reported Homebrew npm shim failure remains actionable.

## What This Changes

The issue reports that launchd-submitted self-update jobs on macOS Homebrew installs invoke the owning npm shim without a PATH that can resolve Homebrew Node, causing npm exit 127 retry loops.

## Best Possible Solution

Add a shared update subprocess env builder used by both gateway and CLI package-update paths. It should include the active runtime directory, the owning npm prefix bin, Homebrew Node/bin candidates, and system dirs while keeping project-local bins out, with focused tests for a launchd-minimal Homebrew npm shim case.

## Reproduction Assessment

Yes. The reporter provided a concrete macOS Homebrew launchd repro, and current-main source inspection shows the same path can run `/opt/homebrew/bin/npm` without adding a PATH entry that lets its `env node` shebang resolve Homebrew Node.

## Solution Assessment

No. The issue's hard-coded shell PATH workaround would likely avoid the immediate failure, but the best maintainable fix is a shared, tested env builder for update subprocesses so all global package-update paths behave consistently.
