## Summary

Close as implemented on current main. The current update path bootstraps PATH for launchd/minimal environments with the active runtime directory plus Homebrew defaults before update subprocesses run, and command execution merges that process environment into npm invocations, so Homebrew's `env node` npm shim no longer depends on launchd's default `/usr/bin:/bin:/usr/sbin:/sbin` PATH. The implementation is present in shipped release `v2026.4.29`.

## What This Changes

Report a macOS launchd self-update failure where Homebrew's absolute `npm` shim cannot resolve `env node` under launchd's minimal PATH.

## Best Possible Solution

Keep the current shared PATH-bootstrap behavior for updater subprocesses and direct users stranded on older broken versions to the documented installer/manual npm recovery path; no additional core change is needed for current main.

## Reproduction Assessment

Yes. The original failure has a high-confidence reproduction on older builds: launchd's minimal PATH plus a Homebrew npm shim with `env node`. Against current main, source inspection shows the active runtime directory is added before update subprocesses run, so that specific failure should not reproduce.

## Solution Assessment

Yes. The current approach is more maintainable than a hard-coded updater-script PATH export because it centralizes launchd/minimal PATH repair at process startup and lets both gateway and CLI update subprocesses inherit the same active-runtime/Homebrew-aware environment.
