## Summary

Keep open. Current main still has the narrow updater environment gap described in the report: the global package update path can select the Homebrew-owned npm shim, but the shared install environment builder does not add the active Node runtime directory, the owning npm prefix bin, or Homebrew Node/bin paths when the caller starts from a launchd-minimal PATH.

## What This Changes

The issue requests a reliable macOS Homebrew self-update path so launchd-started npm updates do not fail with `env: node: No such file or directory` when npm is invoked from a minimal PATH.

## Best Possible Solution

Add a shared, tested update subprocess env builder and use it from gateway and CLI global package-update paths so minimal launchd environments get the active runtime bin, owning npm prefix bin, Homebrew Node/bin locations, and system dirs without weakening service PATH hardening.

## Reproduction Assessment

Yes. A source-level reproduction is clear: use a Homebrew-style global package root and npm shim with PATH `/usr/bin:/bin:/usr/sbin:/sbin`; current env construction passes that minimal PATH through to the npm install step without adding a Node-resolving directory.

## Solution Assessment

No. The issue's hard-coded shell export would address the immediate Homebrew failure, but the better fix is a shared updater env helper used by all global package update subprocess paths with focused regression coverage.
