## Summary

Keep open. Current main still has the reported updater reliability gap: global package-update subprocesses can select the owning npm shim, but the shared update environment does not add the active runtime directory, owning npm prefix bin, or Homebrew runtime bins when launched from launchd's minimal PATH.

## What This Changes

The issue reports that launchd-submitted self-update jobs on Homebrew macOS installs fail because npm's `env node` shebang cannot resolve Node from launchd's minimal PATH.

## Best Possible Solution

Add a shared, tested update subprocess env builder and use it from both gateway and CLI global package-update paths so minimal launchd environments get the active runtime bin, the owning npm prefix bin, Homebrew runtime/bin locations, and system dirs without adding project-local bins or weakening service PATH hardening.

## Reproduction Assessment

Yes. A Homebrew global npm install launched from a launchd-minimal PATH should reproduce the npm shim failure; current source inspection shows the update env still lacks the PATH repair needed for that setup.

## Solution Assessment

No. The issue's hard-coded shell PATH export would likely avoid the immediate failure, but the maintainable fix is a shared env helper used by every global package-update subprocess path with focused regression coverage.
