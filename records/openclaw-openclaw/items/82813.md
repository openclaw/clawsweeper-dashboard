## Summary

Keep open: current main has generic externalized-plugin repair paths, but the Discord upgrade report is against the latest shipped release and source inspection still shows a concrete CLI diagnostic gap where `channels list` can hide a configured-but-missing official external channel.

## What This Changes

Upgrading from 2026.4.x to 2026.5.x should preserve or loudly repair a configured Discord channel after Discord moves to the external `@openclaw/discord` plugin.

## Best Possible Solution

Keep the post-core repair direction, add Discord-specific regression coverage for configured same-id externalization, and make `channels list` show the same install/doctor hint that status already shows for configured-but-missing official external channel plugins.

## Reproduction Assessment

No, not for the full upgrade flow: I did not run a real 2026.4.27 to 2026.5.12 packaged upgrade. The CLI inconsistency is source-reproducible because `channels list` can still emit `no configured chat channels` when a configured official external channel has no installed owner.

## Solution Assessment

Unclear: current main's post-core repair is the right implementation direction, but the issue should not close until Discord-specific coverage and the `channels list` diagnostic gap are addressed or explicitly split to a canonical follow-up.
