  - body: Retries are allowed from `failed` and `lost`, but this branch passes `parentFlowId` into the child spawn before moving the flow out of a terminal state. Current link guards reject terminal parent flows, so the retry can fail linkage or leave detached child work before the wait patch runs.
  - confidence: 0.91
- **[P1] Forward requester origin for subagent retries:** `src/tasks/task-executor.ts:572-574`
  - body: The subagent retry context only includes `agentSessionKey`, while thread-bound spawns need the original channel/to/thread fields for `subagent_spawning`. Forward `flow.requesterOrigin` as the ACP retry path does or retry can fail thread binding or route to the wrong surface.
  - confidence: 0.9
- **[P1] Prioritize cancellation before completion sync:** `src/tasks/task-registry.ts:947-949`
  - body: The managed completion path does not check `cancelRequestedAt`, so a child terminal update can mark the flow succeeded/failed before cancellation finalization gets a chance to apply. Gate completion when cancellation is requested, or apply cancellation before child completion.
  - confidence: 0.88
- **[P2] Require wait identifiers when matching child tasks:** `src/tasks/task-registry.ts:815-825`
  - body: When `waitJson` specifies a `runId` or `childSessionKey`, this matcher still accepts a terminal task that omits that identifier. That can close a multi-child flow from the wrong task outcome; require every present wait identifier to be present and equal on the task.
  - confidence: 0.86
- **[P2] Use retry eligibility for guidance text:** `src/tasks/task-flow-guidance.ts:41-47`
  - body: This guidance marks every managed `failed` or `lost` flow as retryable, but the same PR records non-retryable launches such as attachment-backed spawns. Use the retry eligibility resolver before advertising `retry_available`.
  - confidence: 0.85
- **[P2] Discard stale TaskFlow loads after session switches:** `ui/src/ui/controllers/task-flows.ts:23-33`
