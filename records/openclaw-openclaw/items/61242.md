Full review comments:

- **[P1] Use the declared TypeBox package:** `src/agents/tools/sessions-spawn-tool.ts:1`
  - body: This file imports from @sinclair/typebox, but the repo declares and uses typebox. With pnpm's isolated dependency layout, this can fail module resolution before the TaskFlow path runs.
  - confidence: 0.96
- **[P1] Preserve ACP inherited-tool policy checks:** `src/agents/tools/sessions-spawn-tool.ts:314-335`
  - body: The ACP branch now calls spawnAcpDirect without the current-main unsupported inherited tool deny/allow checks or policy forwarding. That can let an ACP child spawn proceed outside the parent session's tool restrictions.
  - confidence: 0.92
- **[P2] Keep current sessions_spawn controls in the schema:** `src/agents/tools/sessions-spawn-tool.ts:68-83`
  - body: The static schema drops current-main fields and gating such as taskName, context, lightContext, dynamic ACP availability, and thread-aware mode restrictions. Add taskFlow on top of the current schema instead of replacing the contract.
  - confidence: 0.91
- **[P1] Reopen failed flows before subagent retry spawns:** `src/tasks/task-executor.ts:569-570`
  - body: The retry resolver allows failed/lost flows, but this branch sends parentFlowId into spawnSubagentDirect while the parent flow is still terminal. Subagent registration rejects terminal parent flows, so retries can fail after starting detached work.
  - confidence: 0.92
- **[P1] Pass requester origin into subagent retries:** `src/tasks/task-executor.ts:572-574`
  - body: The subagent retry path reconstructs context with only agentSessionKey, while spawnSubagentDirect uses requester-origin fields for thread and channel binding. Threaded or channel-bound stored launches can lose their target on retry.
  - confidence: 0.88
- **[P1] Reopen failed flows before ACP retry linking:** `src/tasks/task-executor.ts:631-658`
  - body: The ACP retry path passes parentFlowId into spawnAcpDirect and may also link the spawned task before the failed/lost parent flow is made nonterminal. Current link validation rejects terminal parent flows, leaving retry work detached or failed.
  - confidence: 0.9
- **[P1] Give cancellation precedence over completion sync:** `src/tasks/task-registry.ts:963-977`
  - body: The completion path does not check cancelRequestedAt, so a child terminal outcome can finalize the managed flow before cancellation sync runs. A cancelled managed flow can then report succeeded/failed/lost instead of cancelled.
  - confidence: 0.9
- **[P2] Require wait identifiers when matching child completions:** `src/tasks/task-registry.ts:815-825`
  - body: matchesManagedChildWaitState only rejects mismatches when both sides provide an identifier. A flow waiting on a specific runId or childSessionKey can still be closed by another linked terminal task that omits that field.
  - confidence: 0.87
- **[P2] Base retry guidance on actual retry eligibility:** `src/tasks/task-flow-guidance.ts:41-47`
  - body: getTaskFlowGuidance reports retry_available for every failed or lost managed flow, but the retry resolver can reject missing launch metadata, active children, cancellation, or attachment-backed launches. Use the same eligibility logic for guidance.
  - confidence: 0.9
- **[P2] Use neutral waiting text for non-child managed waits:** `src/tasks/task-flow-registry.ts:266-276`
  - body: The status message hardcodes child-task wording for waiting flows. Managed flows can wait on other conditions, so the user-facing update should check waitJson.kind or fall back to neutral text.
  - confidence: 0.84
- **[P2] Discard stale flow loads after session switches:** `ui/src/ui/controllers/task-flows.ts:23-33`
  - body: loadLatestTaskFlow returns while a load is in flight and then writes the awaited response using mutable state. If the user switches sessions during that request, the old flow can populate the new chat and actions can target stale state.
  - confidence: 0.87
- **[P3] Remove localized docs edits from this repo:** `docs/zh-CN/tools/subagents.md:65-76`
  - body: The scoped docs guide says localized docs under docs/<locale> are not maintained here. Keep the English source change and glossary/update path instead of editing the generated zh-CN page directly.
  - confidence: 0.82
