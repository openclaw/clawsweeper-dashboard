Full review comments:

- **[P1] Preserve inherited ACP tool restrictions:** `src/agents/tools/sessions-spawn-tool.ts:327-335`
  - body: This ACP spawn context no longer forwards `inheritedToolAllowlist` or `inheritedToolDenylist`, and the branch also removes the current pre-spawn inherited-tool guards. A restricted parent session can therefore create an ACP child without the current tool boundary.
  - confidence: 0.92
- **[P1] Keep the current sessions_spawn contract:** `src/agents/tools/sessions-spawn-tool.ts:68-123`
  - body: The static schema replaces current main's dynamic schema and description path, dropping current fields such as `taskName`, `context`, and `lightContext` plus ACP/thread availability gating. Merging this over current main would regress supported model-facing tool behavior instead of only adding TaskFlow support.
  - confidence: 0.9
- **[P1] Reopen terminal flows before retry spawn:** `src/tasks/task-executor.ts:552-570`
  - body: Retry eligibility includes `failed` and `lost`, but the retry path passes `parentFlowId` into the child spawn before moving the flow out of a terminal status. Task registration rejects terminal parent-flow links, so failed/lost retries can fail after starting work or leave detached work.
  - confidence: 0.88
- **[P1] Forward requester origin on subagent retry:** `src/tasks/task-executor.ts:572-574`
  - body: The subagent retry context only carries `agentSessionKey`, so `spawnSubagentDirect` cannot reconstruct the original channel/thread requester origin. Retrying a thread-bound child flow can therefore fail binding or deliver progress to the wrong surface.
  - confidence: 0.86
- **[P2] Require wait identifiers for managed child matches:** `src/tasks/task-registry.ts:815-825`
  - body: This matcher only rejects a mismatch when both sides provide the identifier. If a flow waits on a specific `runId` or `childSessionKey`, a later linked task with that field missing can still finalize the flow with the wrong child outcome.
  - confidence: 0.84
- **[P1] Apply cancellation before child completion finalization:** `src/tasks/task-registry.ts:1120-1131`
  - body: `updateTask` runs managed completion finalization before cancellation finalization. When cancellation has been requested and the child then exits, the flow can become `succeeded` or `failed` first, after which the cancellation sync no longer applies.
  - confidence: 0.87
- **[P2] Use retry eligibility for user guidance:** `src/tasks/task-flow-guidance.ts:41-47`
  - body: This guidance labels every managed `failed` or `lost` flow as retryable, but the retry path can mark stored launches as non-retryable, such as attachment-backed spawns. The CLI/runtime UI can therefore advertise a retry action that the operation will reject.
  - confidence: 0.83
- **[P2] Discard stale task-flow loads after session switches:** `ui/src/ui/controllers/task-flows.ts:23-33`
  - body: When a flow load is in flight, this returns instead of starting a request for the new session, and the old response writes directly into shared state. A chat switch can show the prior session's flow card and action buttons until another refresh happens.
  - confidence: 0.86
