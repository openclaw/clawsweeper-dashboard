## Summary

Keep open: the central managed child TaskFlow retry/control-surface feature is not on current main, but the PR head still has source-reproducible retry, cancellation, matching, guidance, UI, and docs-policy blockers plus no real behavior proof.

## What This Changes

The PR adds managed child TaskFlow creation from `sessions_spawn(taskFlow=...)`, child-completion syncing, progress notifications, retry guidance, CLI/runtime/gateway/UI retry controls, docs, and tests.

## Best Possible Solution

Revise the branch into a smaller mergeable TaskFlow v1 slice with atomic retry reopen/linking, requester-origin-safe retries, cancellation precedence, strict child matching, eligibility-based guidance, stale-response UI guards, English-only docs updates, and real behavior proof.

## Reproduction Assessment

Yes. Source inspection of the PR head gives high-confidence reproduction paths for failed/lost retry linkage, requester-origin loss, cancellation being overwritten by child completion, loose child matching, retry guidance mismatch, and stale UI loads after session switches.

## Solution Assessment

No. The product direction is plausible, but this patch is not mergeable until the state transitions, retry context, matching, guidance, UI request-tokening, localized-docs policy issue, and real behavior proof are addressed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Reopen terminal flows before retry child spawns:** `src/tasks/task-executor.ts:569`
  - body: `resolveRetryableManagedChildTaskFlow` allows retries from `failed` and `lost`, but this path passes `parentFlowId` into a new child spawn while the flow is still terminal. Task creation/linking rejects terminal parent flows, so failed/lost retries can fail after starting work or detach the child from the flow. Move the flow to a nonterminal retrying/waiting state before spawning/linking, or make reopen plus child link atomic.
  - confidence: 0.91
- **[P1] Pass requester origin into subagent retry spawns:** `src/tasks/task-executor.ts:572-574`
  - body: The subagent retry call passes only `agentSessionKey`, but `spawnSubagentDirect` reconstructs channel/account/thread requester origin from the context object. Stored launches with `thread: true` can lose their original thread binding and fail even though the flow is otherwise retryable. Forward `flow.requesterOrigin` fields as the ACP retry path does.
  - confidence: 0.87
- **[P1] Give cancellation precedence over completion sync:** `src/tasks/task-registry.ts:1121-1130`
  - body: `updateTask` runs managed completion sync before cancellation sync, and the completion path does not skip flows with `cancelRequestedAt`. If a user cancels a managed flow and the child later exits, the flow can be finalized as succeeded/failed/lost before cancellation is observed. Check cancellation first or make completion sync bail when cancellation was requested.
  - confidence: 0.89
- **[P2] Require wait identifiers when matching child completions:** `src/tasks/task-registry.ts:815-825`
  - body: `matchesManagedChildWaitState` only rejects mismatches when both the wait state and task provide an identifier. If `waitJson` names a specific `runId` or `childSessionKey`, a different linked terminal task that lacks that field still matches and can close the flow with the wrong outcome. Require every identifier present in `waitJson` to be present and equal on the task.
  - confidence: 0.86
- **[P2] Base retry guidance on actual retry eligibility:** `src/tasks/task-flow-guidance.ts:41-47`
  - body: `getTaskFlowGuidance` reports `retry_available` for every failed or lost managed flow, but the retry resolver can reject non-retryable launch metadata, missing launch data, active child tasks, or cancellation. That makes CLI/UI guidance contradict the retry section. Derive guidance from the same eligibility logic or suppress retry guidance when retry is blocked.
  - confidence: 0.9
- **[P2] Discard stale flow loads after session switches:** `ui/src/ui/controllers/task-flows.ts:23-33`
  - body: `loadLatestTaskFlow` returns early while a load is in flight and writes the result after awaiting using mutable `state.sessionKey`. If the user switches sessions during that request, the old response can populate the new chat with the previous session's flow card until another refresh. Capture the requested session key or a request token and ignore stale responses while allowing a new-session load to proceed.
  - confidence: 0.87
- **[P3] Avoid child-task wording for generic waiting flows:** `src/tasks/task-flow-registry.ts:265-276`
  - body: `formatManagedFlowStatusMessage` emits `Waiting on child task` for any managed flow that enters `waiting`, while managed flows can wait on non-child conditions. Restrict this wording to `waitJson.kind === "child_task"` or use neutral waiting copy for other managed wait states.
  - confidence: 0.78
- **[P3] Remove localized docs edits from this repo:** `docs/zh-CN/tools/subagents.md:65-76`
  - body: The docs guide says localized docs under `docs/<locale>/**` are not maintained in this repo. Adding the zh-CN copy here can drift from the English source and generated docs pipeline; keep the English docs change here and let the docs publish/i18n process handle translations.
  - confidence: 0.84
