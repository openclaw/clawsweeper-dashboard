## Summary

Keep open: current main does not implement the PR’s managed child TaskFlow UX, and the proposed branch still has concrete correctness blockers around retry linking, cancellation precedence, matching, guidance, UI stale loads, and changelog policy.

## What This Changes

The PR adds managed child TaskFlow creation from `sessions_spawn`, child-completion sync, progress/retry guidance, retry operations, gateway/CLI/runtime/UI controls, docs, and tests.

## Best Possible Solution

Revise the branch into a mergeable TaskFlow v1 slice with atomic retry reopen/linking, preserved cancellation intent, strict child matching, requester-origin-safe retries, session-safe UI loads, eligibility-based guidance, neutral wait copy, and a required changelog entry.

## Reproduction Assessment

Yes. Source inspection gives high-confidence repro paths: retrying failed/lost managed flows links through a terminal parent, a cancel-requested flow can be finalized by later child completion, and switching sessions during an in-flight latest-flow request can apply stale UI state.

## Solution Assessment

No. The product direction is reasonable, but the current implementation is not the best solution until the retry/cancel state transitions, child matching, guidance consistency, UI request tokening, and changelog gap are fixed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Reopen terminal flows before retry child spawns:** `src/tasks/task-executor.ts:569`
  - body: `resolveRetryableManagedChildTaskFlow` allows failed/lost retries, but the retry path passes `parentFlowId` into child spawn while the flow is still terminal. Task creation rejects terminal parent flows, so retry can fail after starting work or detach the child from the flow. Move the flow to a nonterminal retrying/waiting state before spawning/linking, or make reopen and link atomic.
  - confidence: 0.91
- **[P1] Pass requester origin into subagent retry spawns:** `src/tasks/task-executor.ts:572-574`
  - body: The subagent retry call only passes `agentSessionKey`, but `spawnSubagentDirect` reconstructs channel/account/thread requester origin from context. Stored `thread: true` launches can lose their original thread binding and fail even though the flow is otherwise retryable. Forward `flow.requesterOrigin` fields as the ACP retry path does.
  - confidence: 0.87
- **[P1] Give cancellation precedence over completion sync:** `src/tasks/task-registry.ts:1121-1130`
  - body: `updateTask` runs managed completion sync before cancellation sync, and the completion path does not skip flows with `cancelRequestedAt`. If a user cancels a managed flow and the child later exits, the flow can be finalized as succeeded/failed/lost before cancellation is observed. Check cancellation first or make completion sync bail when cancellation was requested.
  - confidence: 0.89
- **[P2] Require wait identifiers when matching child completions:** `src/tasks/task-registry.ts:815-825`
  - body: `matchesManagedChildWaitState` only rejects mismatches when both sides provide an identifier. If `waitJson` names a specific `runId` or `childSessionKey`, a different linked terminal task that lacks that field can still close the flow with the wrong outcome. Require every identifier present in `waitJson` to be present and equal on the task.
  - confidence: 0.86
- **[P2] Base retry guidance on actual retry eligibility:** `src/tasks/task-flow-guidance.ts:41-47`
  - body: `getTaskFlowGuidance` reports `retry_available` for every failed or lost managed flow, but the retry resolver can reject missing or non-retryable launch metadata, active child tasks, or cancellation. Derive guidance from the same eligibility logic or suppress retry guidance when retry is blocked.
  - confidence: 0.9
- **[P2] Discard stale flow loads after session switches:** `ui/src/ui/controllers/task-flows.ts:23-33`
  - body: `loadLatestTaskFlow` returns early while a load is in flight and writes the result after awaiting with mutable `state.sessionKey`. If the user switches sessions during that request, the old response can populate the new chat with the previous session's flow card until another refresh. Capture the requested session key or a request token and ignore stale responses while allowing a new-session load to proceed.
  - confidence: 0.87
- **[P3] Avoid child-task wording for generic waiting flows:** `src/tasks/task-flow-registry.ts:265-276`
  - body: `formatManagedFlowStatusMessage` emits child-task-specific waiting copy for any managed flow that enters `waiting`. Managed flows can wait on other conditions, so restrict this wording to `waitJson.kind === "child_task"` or use neutral waiting copy for non-child wait states.
  - confidence: 0.78
- **[P2] Add the required changelog entry:** `docs/automation/index.md:76`
  - body: This PR adds user-facing TaskFlow behavior across CLI, docs, gateway, runtime, and Control UI, but the changed file list does not include `CHANGELOG.md`. Add an Unreleased changelog entry for the feature before merge.
  - confidence: 0.84
