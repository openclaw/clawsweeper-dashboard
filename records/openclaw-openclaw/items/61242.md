## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Apply managed-flow cancellation before child completion wins:** `src/tasks/task-registry.ts:1121-1130`
  - body: `updateTask` runs managed completion before managed cancellation, so a child that exits after a user requested cancellation can make the flow terminal as `succeeded`/`failed`/`lost`; the cancellation sync then returns because the flow is already terminal. Run the cancellation path first or skip completion when `cancelRequestedAt` is set so user cancellation remains visible.
  - confidence: 0.91
- **[P1] Require wait identifiers when matching child completions:** `src/tasks/task-registry.ts:815-825`
  - body: When `waitJson` contains a `runId` or `childSessionKey`, a terminal task that lacks that identifier still matches because the code only rejects when both sides are present and different. That can finalize a managed flow from the wrong linked task; treat missing required identifiers as non-matches.
  - confidence: 0.9
- **[P2] Forward requester origin when retrying subagent children:** `src/tasks/task-executor.ts:572-574`
  - body: The subagent retry branch calls `spawnSubagentDirect` with only `agentSessionKey`, but subagent spawning derives requester origin and thread binding from `agentChannel`, `agentTo`, and `agentThreadId`. Retry of a thread-bound child can therefore lose the original requester target; forward `flow.requesterOrigin` like the ACP retry branch does.
  - confidence: 0.88
- **[P2] Base retry guidance on real eligibility:** `src/tasks/task-flow-guidance.ts:41-47`
  - body: This helper labels every failed or lost managed flow as `retry_available`, even when stored launch metadata has `retryable: false`, launch details are missing, cancellation is requested, or an active child still exists. Reuse the retry eligibility resolver or a shared helper so `nextAction` and the actual retry result cannot contradict each other.
  - confidence: 0.9
- **[P2] Discard stale TaskFlow loads after session switches:** `ui/src/ui/controllers/task-flows.ts:23-33`
  - body: `loadLatestTaskFlow` drops reloads while one request is in flight and then writes the old response without checking the requested `sessionKey`. Switching chats during a slow request can show the previous session's flow card until another event reloads it; capture the session key/request token and ignore stale responses or allow a new-session fetch.
  - confidence: 0.86
