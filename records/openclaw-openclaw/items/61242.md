## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Reopen terminal flows before subagent retry spawn:** `src/tasks/task-executor.ts:569-570`
  - body: `resolveRetryableManagedChildTaskFlow` allows failed/lost flows, but this spawn still passes `parentFlowId` while the parent flow is terminal. The task registry rejects linking into terminal flows, so failed/lost subagent retries can fail before the flow is moved back to `waiting`; reopen the flow or defer linking before starting the retry.
  - confidence: 0.92
- **[P1] Forward requester origin on subagent retries:** `src/tasks/task-executor.ts:572-574`
  - body: The retry context only includes `agentSessionKey`, while `spawnSubagentDirect` reconstructs requester origin and thread binding from the context fields. Thread-bound retries can lose their original channel/thread target and fail or route incorrectly; forward the stored requester origin as the ACP retry path does.
  - confidence: 0.87
- **[P1] Reopen terminal flows before ACP retry linking:** `src/tasks/task-executor.ts:631`
  - body: The ACP retry path also sends `parentFlowId` into the spawn before the flow is moved out of a terminal failed/lost state. Because `linkTaskToFlowById` rejects terminal parent flows, ACP retries can start work that cannot be linked cleanly back to the managed flow.
  - confidence: 0.84
- **[P1] Prioritize cancellation before completion finalization:** `src/tasks/task-registry.ts:948-949`
  - body: Managed completion does not check `cancelRequestedAt`, so a child exiting after cancellation can finalize the flow as succeeded/failed/lost before the cancellation sync runs. That prevents the requested cancel from being reflected; skip completion when cancellation is requested or apply cancellation first.
  - confidence: 0.86
- **[P2] Require wait identifiers before finalizing flows:** `src/tasks/task-registry.ts:815-825`
  - body: This matcher only rejects mismatches when both sides provide an identifier. If `waitJson` names a runId or childSessionKey and a terminal task lacks that field, the flow can be finalized from the wrong linked child; require any identifier present in `waitJson` to also be present and equal on the task.
  - confidence: 0.9
- **[P2] Use retry eligibility for retry guidance:** `src/tasks/task-flow-guidance.ts:41-47`
  - body: `getTaskFlowGuidance` labels every failed/lost managed flow as retryable, but this PR also records non-retryable launch metadata such as attachment-backed spawns. That can show `Retry available` while the actual retry command says it is blocked; derive guidance from the same eligibility helper.
  - confidence: 0.84
- **[P2] Discard stale flow loads after session switches:** `ui/src/ui/controllers/task-flows.ts:23-24`
  - body: If the user switches chats while a flow load is awaiting, this early return skips the new fetch and the old response can later populate `taskFlowDetail` for the wrong session. Capture the requested session key or use a request token so stale responses are ignored and the current session still loads.
  - confidence: 0.91
