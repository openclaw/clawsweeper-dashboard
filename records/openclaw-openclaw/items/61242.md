## Summary

Keep open: current main does not yet implement the requested managed child TaskFlow UX, and the PR branch still has blocking correctness defects in retry, cancellation, child matching, and UI session switching. The earlier ACP registration failure noted in review was addressed by follow-up commit be6cfe9392930d4dc73e9be8c1c7fb3658018501, but later blockers remain.

## What This Changes

The PR adds managed child TaskFlow creation from `sessions_spawn`, child-completion synchronization, progress and retry guidance, retry operations, CLI/gateway/runtime/UI controls, docs, and tests.

## Best Possible Solution

Revise the branch into a mergeable TaskFlow v1 control-surface slice: keep the managed `sessions_spawn(taskFlow=...)` and retry concepts, but make retry state transitions atomic with child linking, preserve cancellation intent, require precise child identifiers, derive guidance from actual eligibility, make Control UI flow loads session-safe, and add the required changelog entry before changed-gate validation.

## Reproduction Assessment

Yes. Source inspection gives high-confidence repro paths: retrying a failed/lost managed child flow passes `parentFlowId` while the flow is still terminal, a cancel-requested flow can be completed by a later child terminal update, and switching UI sessions during `loadLatestTaskFlow` can apply a stale response.

## Solution Assessment

No. The product direction is plausible, but the current patch is not the best way to solve it until it reopens flows before spawn/link, gives cancellation precedence, uses strict wait matching, bases guidance on retry eligibility, and ignores stale UI loads.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Reopen terminal flows before retry child spawns:** `src/tasks/task-executor.ts:569`
  - body: `resolveRetryableManagedChildTaskFlow` allows retries from `failed` and `lost`, but this retry path passes `parentFlowId` into the child spawn while the flow is still terminal. Task linking rejects terminal parent flows, so retries can fail before relaunch or start detached from the flow. Move the flow back to a nonterminal retrying/waiting state before spawning/linking the child, or link only after a successful reopen.
  - confidence: 0.91
- **[P1] Pass requester origin into subagent retry spawns:** `src/tasks/task-executor.ts:572-574`
  - body: Subagent retries call `spawnSubagentDirect` with only `agentSessionKey`, so the spawn cannot reconstruct the original channel/account/thread requester origin. Stored launches with `thread: true` can lose thread binding and fail even though the flow is otherwise retryable. Forward `flow.requesterOrigin` fields here as the ACP retry path does.
  - confidence: 0.87
- **[P1] Give cancellation precedence over completion sync:** `src/tasks/task-registry.ts:1121`
  - body: `updateTask` runs managed completion sync before cancellation sync, and the completion path does not skip flows with `cancelRequestedAt`. If a user cancels a managed flow and the child later exits, the flow can be finalized as succeeded/failed/lost before cancellation is observed. Check cancellation first or make completion sync bail when cancellation was requested.
  - confidence: 0.89
- **[P2] Require wait identifiers when matching child completions:** `src/tasks/task-registry.ts:815-825`
  - body: `matchesManagedChildWaitState` only rejects mismatches when both the wait state and task provide an identifier. If `waitJson` names a specific `runId` or `childSessionKey`, a different linked terminal task that lacks that field still matches and can close the flow with the wrong outcome. Require any identifier present in `waitJson` to be present and equal on the task.
  - confidence: 0.86
- **[P2] Base retry guidance on actual retry eligibility:** `src/tasks/task-flow-guidance.ts:41-47`
  - body: `getTaskFlowGuidance` reports `retry_available` for every failed or lost managed flow, but the retry resolver can reject flows with non-retryable launch metadata, missing launch data, or an active child task. That makes CLI/UI guidance contradict the retry section. Derive guidance from the same eligibility logic or suppress retry guidance when retry is blocked.
  - confidence: 0.9
- **[P2] Discard stale flow loads after session switches:** `ui/src/ui/controllers/task-flows.ts:23-33`
  - body: `loadLatestTaskFlow` returns early while a load is in flight and writes the result after awaiting using mutable `state.sessionKey`. If the user switches sessions during that request, the old response can populate the new chat with the previous session's flow card until another refresh. Capture the requested session key or request token and ignore stale responses while allowing a new-session load to proceed.
  - confidence: 0.87
- **[P2] Add the required changelog entry:** `docs/automation/index.md:76`
  - body: This PR adds user-facing TaskFlow behavior across CLI, docs, gateway, runtime, and Control UI, but the file list does not include `CHANGELOG.md`. Repo policy requires user-facing `feat` changes to add an Unreleased changelog entry with appropriate contributor credit.
  - confidence: 0.84
