## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Reopen terminal flows before retry child spawns:** `src/tasks/task-executor.ts:569-574`
  - body: `retryManagedChildTaskFlowUnchecked` allows failed/lost retries, but it passes `parentFlowId` into a new child spawn before the flow is moved out of a terminal state. Current task linking rejects terminal parent flows, so retry can start or attempt work that cannot be linked back to the managed flow. Reopen the flow before spawning/linking, or make reopen plus child link atomic.
  - confidence: 0.91
- **[P1] Pass requester origin into subagent retry spawns:** `src/tasks/task-executor.ts:572-574`
  - body: The subagent retry branch reconstructs context with only `agentSessionKey`, while `spawnSubagentDirect` uses requester-origin fields for channel/thread binding. Stored launches that used threaded delivery can lose their original target and fail even though the flow is otherwise retryable. Forward `flow.requesterOrigin` fields as the ACP retry path does.
  - confidence: 0.87
- **[P1] Give cancellation precedence over completion sync:** `src/tasks/task-registry.ts:1121-1130`
  - body: `updateTask` runs managed completion sync before managed cancellation sync, and the completion path does not skip flows with `cancelRequestedAt`. If a user cancels a managed flow and the child later exits, the flow can be finalized as succeeded/failed/lost before cancellation is applied. Check cancellation first or make completion sync bail after cancellation is requested.
  - confidence: 0.89
- **[P2] Require wait identifiers when matching child completions:** `src/tasks/task-registry.ts:815-825`
  - body: `matchesManagedChildWaitState` only rejects mismatches when both the wait state and task provide an identifier. If `waitJson` names a specific `runId` or `childSessionKey`, a different linked terminal task that lacks that field can still close the flow with the wrong outcome. Require every identifier present in `waitJson` to be present and equal on the task.
  - confidence: 0.86
- **[P2] Base retry guidance on actual retry eligibility:** `src/tasks/task-flow-guidance.ts:41-47`
  - body: `getTaskFlowGuidance` reports `retry_available` for every failed or lost managed flow, but the retry resolver can reject missing or non-retryable launch metadata, active child tasks, cancellation, or attachment-backed launches. Derive guidance from the same eligibility logic or suppress retry guidance when retry is blocked.
  - confidence: 0.9
- **[P2] Discard stale flow loads after session switches:** `ui/src/ui/controllers/task-flows.ts:23-33`
  - body: `loadLatestTaskFlow` returns early while a load is in flight and writes the result after awaiting using mutable `state.sessionKey`. If the user switches sessions during that request, the old response can populate the new chat with the previous session's flow card until another refresh. Capture the requested session key or a request token and ignore stale responses while allowing a new-session load to proceed.
  - confidence: 0.87
- **[P3] Avoid child-task wording for generic waiting flows:** `src/tasks/task-flow-registry.ts:265-276`
  - body: `formatManagedFlowStatusMessage` emits child-task-specific waiting copy for any managed flow that enters `waiting`, while managed flows can wait on non-child conditions. Restrict this wording to `waitJson.kind === "child_task"` or use neutral waiting copy for other managed wait states.
  - confidence: 0.78
- **[P3] Remove localized docs edits from this repo:** `docs/zh-CN/tools/subagents.md:65-76`
  - body: The scoped docs guide says localized docs under `docs/<locale>/**` are not maintained in this repo. Keep the English docs change here and let the docs publish/i18n process handle translations so the zh-CN copy does not drift.
  - confidence: 0.84
