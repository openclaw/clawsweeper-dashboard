Overall confidence: 0.89

Full review comments:

- **[P1] Preserve inherited ACP tool restrictions:** `src/agents/tools/sessions-spawn-tool.ts:327-335`
  - body: The ACP branch now calls `spawnAcpDirect` with a context that omits the parent's inherited tool allow/deny lists. Current main checks and patches those restrictions for ACP children, so this would let a child spawn escape a parent-session tool policy.
  - confidence: 0.93
- **[P1] Keep the current sessions_spawn contract:** `src/agents/tools/sessions-spawn-tool.ts:68-123`
  - body: This PR replaces the newer dynamic `sessions_spawn` schema and routing surface with an older static schema, dropping current behavior such as ACP availability gating, `taskName`, `context`/`lightContext`, completion-owner routing, group metadata, and normalized model handling. Rebase this code on current main instead of reintroducing the older contract.
  - confidence: 0.88
- **[P1] Reopen terminal flows before retry spawn:** `src/tasks/task-executor.ts:552-570`
  - body: Retries from `failed` or `lost` are allowed, but the subagent retry path passes `parentFlowId` into `spawnSubagentDirect` while the flow is still terminal. Current parent-flow linking rejects terminal parents, so retry can fail after starting or attempting child work instead of cleanly moving the flow back to waiting first.
  - confidence: 0.9
- **[P1] Pass requester origin into subagent retry spawns:** `src/tasks/task-executor.ts:572-574`
  - body: The subagent retry context only passes `agentSessionKey`, unlike the ACP branch that reconstructs channel/account/to/thread fields from `flow.requesterOrigin`. Thread-bound retries can therefore lose the original requester target and fail binding or closeout delivery.
  - confidence: 0.86
- **[P1] Apply cancellation before completion finalization:** `src/tasks/task-registry.ts:1120-1130`
  - body: `updateTask` runs completion sync before cancellation sync, and `syncManagedFlowCompletionFromTask` does not gate on `cancelRequestedAt`. A child that exits after the user requests cancellation can finalize the flow as succeeded/failed/lost first, preventing the cancellation outcome from being reflected.
  - confidence: 0.87
- **[P2] Require wait identifiers when matching child tasks:** `src/tasks/task-registry.ts:815-825`
  - body: `matchesManagedChildWaitState` only rejects mismatches when both the flow wait state and task provide an identifier. If the flow waits on a specific runId or childSessionKey and a later terminal task omits that field, the flow can be closed from the wrong child outcome.
  - confidence: 0.84
- **[P2] Align guidance with retry eligibility:** `src/tasks/task-flow-guidance.ts:41-47`
  - body: `getTaskFlowGuidance` advertises `retry_available` for every managed `failed` or `lost` flow even when launch metadata explicitly marks the flow non-retryable, such as attachment-backed launches. The user-facing next action should use the same retryability decision as the retry command.
  - confidence: 0.82
- **[P2] Discard stale task-flow loads after session switches:** `ui/src/ui/controllers/task-flows.ts:23-33`
  - body: `loadLatestTaskFlow` suppresses a new fetch while another is in flight and then writes the old response into shared state without checking the requested sessionKey. Switching chats during a load can show the previous session's flow card and make retry/cancel buttons target stale UI state.
  - confidence: 0.86
