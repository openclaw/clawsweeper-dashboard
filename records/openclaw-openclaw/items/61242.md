## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Preserve inherited ACP tool restrictions:** `src/agents/tools/sessions-spawn-tool.ts:327-335`
  - body: Current main rejects unsupported inherited ACP tool allow/deny policy and passes the inherited lists into `spawnAcpDirect`; this branch's ACP context omits both lists. A parent session with inherited tool restrictions can therefore spawn an ACP child that no longer receives the same restrictions, so preserve the current checks/ctx fields before layering TaskFlow support.
  - confidence: 0.92
- **[P1] Forward requester origin on subagent retry:** `src/tasks/task-executor.ts:572-574`
  - body: Managed subagent retry relaunches with only `agentSessionKey: flow.ownerKey`, but `spawnSubagentDirect` reconstructs thread/channel binding from the context object. Thread-bound retries can lose the original requester channel/thread and fail binding or route closeout incorrectly; forward `flow.requesterOrigin` fields here as the ACP retry path does.
  - confidence: 0.88
- **[P1] Apply cancellation before child completion finalization:** `src/tasks/task-registry.ts:1121-1130`
  - body: `updateTask` runs managed completion sync before cancellation sync, and the completion path does not skip flows with `cancelRequestedAt`. If a user cancels and the child exits, the flow can be finalized as succeeded/failed/lost before cancellation has a chance to apply, so cancellation is never reflected.
  - confidence: 0.87
- **[P2] Require wait identifiers when matching children:** `src/tasks/task-registry.ts:815-825`
  - body: When a flow waits on a specific `runId` or `childSessionKey`, this matcher only rejects a mismatch if the terminal task also has that identifier. A linked terminal task missing the identifier can close the wrong flow; require every identifier present in `waitJson` to be present and equal on the task.
  - confidence: 0.84
- **[P2] Discard stale task-flow loads after session switches:** `ui/src/ui/controllers/task-flows.ts:23-33`
  - body: If the user switches sessions while `loadLatestTaskFlow` is in flight, the early return prevents a new load and the old response can still populate `taskFlowDetail`. Capture the requested session key or use a request token so stale responses cannot show controls for the wrong chat.
  - confidence: 0.82
- **[P2] Base guidance on real retry eligibility:** `src/tasks/task-flow-guidance.ts:41-47`
  - body: This returns `retry_available` for every managed `failed` or `lost` flow even when the stored launch marks retry as unsafe, such as attachment-backed spawns. Use the same eligibility check as the retry command so the UI and CLI do not advertise a retry that will be rejected.
  - confidence: 0.8
