## Summary

Keep this open: current main still has a WebChat reconnect-window delivery gap because normal submits are dropped or failed before any durable outbound queue/replay path can use the existing `chat.send` ACK/idempotency contract.

## What This Changes

The issue asks WebChat to preserve outbound messages across WebSocket reconnects with client-side queueing, ACK reconciliation, retry, and delivery error feedback.

## Best Possible Solution

Add a focused Control UI outbound-send lifecycle for normal `chat.send` that captures the message and stable idempotency key before transport, retries transient closed/not-connected or close-before-ACK failures after reconnect hello, reconciles `started`/`in_flight`/`ok`, and shows bounded failed-delivery feedback.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to submit while `host.connected` is false, or to close the browser WebSocket before the `chat.send` ACK; current main returns or rejects without durable queueing or replay.

## Solution Assessment

Yes. The narrowest maintainable fix is a Control UI queue/retry path using the existing Gateway ACK/idempotency contract; a Gateway protocol redesign is not needed unless implementation proves an ACK state is missing.

## Repair Work Prompt

Fix the WebChat reconnect-window outbound delivery gap from https://github.com/openclaw/openclaw/issues/45952. Add a focused Control UI send lifecycle for normal `chat.send`: capture the outbound message, attachments, session id/key, and a stable idempotency key before any transport call; retry transient `gateway not connected`, closed-socket-before-ACK, or reconnect-window failures after reconnect hello; reconcile gateway `started`, `in_flight`, and `ok` responses without duplicating agent turns; and show bounded failed-delivery feedback after retry exhaustion. Keep the fix in the Control UI unless a concrete missing generic Gateway contract is found. Do not broaden this PR into full browser-refresh conversation/draft persistence from #51549 or the broader active-run termination behavior from #38091. Preserve attachment payloads long enough for retry and release them on success, cancellation, or final failure. Add a user-facing `CHANGELOG.md` fix entry.
