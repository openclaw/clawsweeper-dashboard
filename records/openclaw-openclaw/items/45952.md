## Label Justifications

- `P1`: The issue describes silent loss of user-sent WebChat messages during reconnect, which can break an active chat workflow for real users.
- `impact:message-loss`: The reported failure is a user message that never reaches the gateway and is silently lost.
- `impact:session-state`: The reconnect window affects active WebChat run/session continuity and pending user input state.

## Summary

Keep open: current main still has a source-proven WebChat reconnect-window message-loss path. Normal sends are not captured while disconnected, in-flight `chat.send` RPCs are rejected on socket close without replay, and the documented ACK/idempotency contract is not yet wired into a Control UI outbound-send queue.

## What This Changes

WebChat should preserve client-sent messages through a WebSocket reconnect window by queueing unACKed sends and replaying them after reconnect.

## Best Possible Solution

Add a focused Control UI outbound-send queue for normal `chat.send` that captures the message before transport, preserves the original idempotency key through ACK, retries transient closed/not-connected failures after reconnect hello, and shows bounded failed-delivery feedback.

## Reproduction Assessment

Yes, at source level: current main returns from normal sends while disconnected and rejects in-flight RPCs on WebSocket close without replay. I did not establish a live browser timing reproduction in this read-only pass.

## Solution Assessment

Yes. The narrow maintainable fix is to use the existing `chat.send` ACK/idempotency contract in the Control UI send path, not to add a new gateway protocol or fold this into broader refresh, history, or draft persistence work.
