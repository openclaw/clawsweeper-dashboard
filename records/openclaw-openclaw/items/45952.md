## Summary

Keep open: current main and v2026.5.20 still have the WebChat reconnect-window delivery gap. Normal sends can return before queueing while disconnected, pending browser RPCs are rejected on socket close without method-aware replay, and the existing `chat.send` ACK/idempotency contract is not yet wired into a Control UI outbound queue.

## What This Changes

The issue requests a client-side WebChat outbound queue with ACK-based retry for messages submitted during WebSocket reconnects.

## Best Possible Solution

Add a focused Control UI outbound-send queue for normal `chat.send` that captures before transport, preserves one idempotency key through ACK, retries transient closed/not-connected failures after reconnect hello, and surfaces bounded failed-delivery feedback.

## Reproduction Assessment

Yes, at source level: current main returns from normal sends while disconnected and rejects in-flight WebSocket RPCs on close without replay. I did not run a live browser timing repro in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is to use the existing `chat.send` ACK/idempotency contract in the Control UI send path, not to add a new gateway protocol or fold this into broader refresh/history/draft persistence work.
