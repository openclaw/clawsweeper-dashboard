## Summary

Keep open: current main still has a source-reproducible WebChat data-loss window for normal sends during transient WebSocket reconnects. The server ACK/idempotency contract already exists, but the Control UI does not retain a normal `chat.send` payload/key until ACK or replay it after reconnect.

## What This Changes

The issue requests client-side queuing, ACK-based retry, and failed-delivery feedback for WebChat messages sent during WebSocket reconnects.

## Best Possible Solution

Add a focused Control UI outbound-send queue for normal `chat.send` that captures the payload before transport, preserves a stable idempotency key through ACK, retries transient closed/not-connected failures after reconnect hello, and shows bounded failed-delivery state.

## Reproduction Assessment

Yes, source-level: disconnected normal submits return before queueing, and a socket close while `chat.send` is pending rejects the RPC and clears send state without replay. I did not run a live browser timing reproduction in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is to use the existing server ACK/idempotency contract from the Control UI send path, not to add a new gateway protocol or fold this into broader refresh/draft persistence work.
