## Summary

Keep open: current main and the latest shipped tag still have a source-proven WebChat reconnect-window delivery gap. Normal sends are not captured while disconnected, pending browser RPCs are rejected on socket close without method-aware replay, and the existing `chat.send` ACK/idempotency contract is not yet wired into a Control UI outbound queue.

## What This Changes

This issue requests durable WebChat outbound delivery across WebSocket reconnects by queueing normal sends client-side, preserving ACK/idempotency state, retrying after reconnect, and showing failed-delivery feedback.

## Best Possible Solution

Implement a focused Control UI outbound-send queue for normal `chat.send` that captures before transport, preserves the same run/idempotency key until ACK, retries transient closed/not-connected failures after reconnect hello, and surfaces bounded failed-delivery feedback.

## Reproduction Assessment

Yes, at source level: current main returns from normal sends while disconnected and rejects in-flight WebSocket RPCs on close without replay. I did not run a live browser timing repro in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is to use the existing `chat.send` ACK/idempotency contract in the Control UI send path, not to add a new gateway protocol or fold this into broader refresh/history/draft persistence work.
