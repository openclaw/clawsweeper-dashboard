## Summary

Keep this open. Current main still has the WebChat reconnect-window delivery gap: disconnected submits return before queueing, pending browser RPCs are rejected when the socket closes, and normal reconnect does not replay an unACKed `chat.send`, even though the Gateway already exposes ACK/idempotency semantics that a focused UI repair can use.

## What This Changes

The issue requests a Control UI/WebChat outbound message queue with server ACK/idempotency retry so messages submitted during a WebSocket reconnect are not silently lost.

## Best Possible Solution

Add a narrow Control UI outbound-send lifecycle for normal `chat.send`: capture the message and stable idempotency key before transport, retry transient closed/not-connected or close-before-ACK failures after reconnect hello, reconcile `started`/`in_flight`/`ok` ACKs, and show bounded failed-delivery feedback without redesigning the Gateway protocol unless the existing ACK contract proves insufficient.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to submit while `host.connected` is false, or to close the socket before the `chat.send` ACK; current main either returns before queueing or rejects the pending RPC and clears the send state without replay.

## Solution Assessment

Yes. The best repair path is a focused UI queue/retry change that uses the existing Gateway ACK/idempotency contract; a Gateway protocol redesign is not needed for the first fix unless implementation finds a missing ACK state.
