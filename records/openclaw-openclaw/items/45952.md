## Summary

Keep open: current main still has a source-proven WebChat reconnect-window message-loss path. Normal sends are not captured while disconnected, in-flight `chat.send` RPCs are rejected on socket close without replay, and the existing server ACK/idempotency contract is not yet wired into a durable Control UI outbound queue.

## What This Changes

WebChat messages submitted during a transient WebSocket reconnect should be queued, ACK-tracked, retried after reconnect, and surfaced as failed instead of silently disappearing.

## Best Possible Solution

Add a focused Control UI outbound queue for normal `chat.send` that captures payload plus stable idempotency key before transport, keeps it until ACK, retries transient closed/not-connected failures after reconnect hello, and shows bounded failed-delivery feedback.

## Reproduction Assessment

Yes, at source level: current main has disconnected normal submits returning before queue capture and socket-close RPC rejection clearing attempted sends without replay. I did not establish a live browser timing reproduction in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is to use the existing `chat.send` ACK/idempotency contract in the Control UI send path, not to add a new gateway protocol or fold this into broader refresh, draft, or history persistence work.
