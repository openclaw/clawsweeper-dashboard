## Summary

Keep open: current main and the latest release still drop the normal WebChat send path during ordinary reconnect windows. The source path returns before queueing while disconnected, rejects in-flight RPCs on socket close, and only resumes the chat queue for seq-gap reconnects, while the documented `chat.send` ACK/idempotency contract is already available for a focused UI retry fix.

## What This Changes

The issue asks for WebChat user sends to survive WebSocket reconnect windows with client-side retention, ACK-backed retry, and failed-delivery feedback.

## Best Possible Solution

Add a focused Control UI outbound-send queue for normal `chat.send` that captures before transport, preserves one idempotency key through ACK, retries transient closed/not-connected failures after reconnect hello, and surfaces bounded failed-delivery feedback.

## Reproduction Assessment

Yes at source level: current main returns before queueing when disconnected, rejects/flushes in-flight RPCs on socket close, and only drains queued chat for seq-gap reconnects. I did not run a live browser timing repro in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is to use the existing `chat.send` ACK/idempotency contract in the Control UI send path rather than adding a new gateway protocol or folding this into broader refresh/history/draft persistence work.
