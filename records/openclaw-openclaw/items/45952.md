## Summary

Keep open: current main still has a source-proven WebChat data-loss window during transient WebSocket reconnects. Normal sends are discarded before capture when the UI is disconnected, in-flight RPCs are rejected on socket close without replay, and the existing server ACK/idempotency contract is not yet wired into a durable Control UI outbound queue.

## What This Changes

The issue requests durable WebChat outbound delivery across WebSocket reconnects with client-side queueing, server ACK tracking, retry on reconnect, and failed-delivery feedback.

## Best Possible Solution

Add a focused Control UI outbound-send queue for normal `chat.send` that captures payload and stable idempotency key before transport, keeps it until ACK, retries transient closed/not-connected failures after reconnect hello, and surfaces bounded failed-delivery state.

## Reproduction Assessment

Yes, at source level: disconnected normal submits return before queue capture, and socket-close RPC rejection clears the attempted send without replay. I did not establish a live browser timing reproduction in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is to use the existing `chat.send` ACK/idempotency contract in the Control UI send path, not to add a new gateway protocol or fold this into broader refresh, draft, or history persistence work.
