## Summary

Keep open: current main still has a source-reproducible WebChat data-loss window for normal sends during transient WebSocket reconnects. The server already has an ACK/idempotency contract, but the Control UI does not retain a normal `chat.send` payload/key until ACK or replay it after reconnect.

## What This Changes

Ensure WebChat normal sends survive transient WebSocket reconnects with client-side retention, ACK-based retry, and visible failed-delivery feedback.

## Best Possible Solution

Add a focused Control UI outbound-send queue for normal `chat.send` that captures the payload before transport, retains a stable idempotency key through ACK, retries transient closed/not-connected failures after reconnect hello, and surfaces bounded failed-delivery state.

## Reproduction Assessment

Yes, source-level. A disconnected normal submit returns before queueing, and a socket close while `chat.send` is pending rejects the RPC and clears send state without replay; I did not run a live browser timing repro in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is to use the existing server ACK/idempotency contract from the Control UI send path, not to add a new gateway protocol unless that contract proves insufficient.
