## Summary

Current main still lacks durable normal WebChat `chat.send` delivery across a WebSocket reconnect window. The UI can return before enqueueing while disconnected, rejects in-flight RPCs on socket close, and only resumes the chat queue for seq-gap recovery; the related open PR #73154 is adjacent queue persistence work for #51549, not a complete ACK-backed normal-send replay fix.

## What This Changes

Request client-side WebChat send queueing, ACK handling, retry on reconnect, and visible failed-delivery feedback for messages submitted during WebSocket reconnects.

## Best Possible Solution

Add a focused Control UI outbound `chat.send` queue that captures the send before transport, preserves one idempotency key through ACK, retries after reconnect hello, and surfaces bounded failed-delivery UI while reusing the existing gateway ACK/dedupe contract.

## Reproduction Assessment

Yes, by source inspection: current main has a clear reconnect-window path where normal submits are not queued, pending RPCs are rejected on close, and normal reconnect does not replay unACKed sends. I did not run browser/Vitest proof because this review was read-only.

## Solution Assessment

Yes. The narrow maintainable fix is a client-side outbound send queue using the existing `chat.send` ACK/idempotency contract; adding a new server protocol should only be considered if that existing contract proves insufficient.
