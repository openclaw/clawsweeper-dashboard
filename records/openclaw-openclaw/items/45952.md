## Summary

Keep open: current main still has a source-reproducible WebChat data-loss window during transient WebSocket reconnects. Normal sends are not captured before the connected-state gate, reconnect-window RPCs are rejected and cleared without replay, and the existing server ACK/idempotency contract is not yet used by a durable Control UI outbound queue.

## What This Changes

The issue requests durable WebChat send handling across transient WebSocket reconnects with client-side queuing, server ACK/idempotency, retry, and failed-delivery feedback.

## Best Possible Solution

Add a focused Control UI outbound-send queue for normal `chat.send` that captures payload plus stable idempotency key before transport, keeps it until ACK, retries transient closed/not-connected failures after reconnect hello, and surfaces bounded failed-delivery state.

## Reproduction Assessment

Yes, at source level: disconnected normal submits return before any queue capture, and socket-close RPC rejection clears send state without replay. I did not establish a live browser timing reproduction in this read-only pass.

## Solution Assessment

Yes. The narrow maintainable fix is to use the existing `chat.send` ACK/idempotency contract from the Control UI send path, not to add a new gateway protocol or fold this into broader refresh/draft/history persistence.
