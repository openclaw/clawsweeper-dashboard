## Summary

Current main still has the reported WebChat reconnect-window data-loss path: disconnected submits return before any queueing, in-flight RPCs are rejected on socket close, and normal reconnect does not replay unACKed `chat.send` messages even though the gateway already has ACK/idempotency support.

## What This Changes

Ensure WebChat `chat.send` submissions survive WebSocket disconnect/reconnect by queueing unACKed sends, reusing idempotency keys, retrying on reconnect, and surfacing failed delivery.

## Best Possible Solution

Add a focused Control UI outbound-send queue for normal `chat.send` that captures sends before transport, preserves a stable idempotency key until ACK/terminal status, retries after reconnect hello, and shows bounded failed-delivery feedback without changing the gateway protocol unless the existing ACK contract proves insufficient.

## Reproduction Assessment

Yes, at source level: `handleSendChat` drops disconnected submits, and a socket close rejects pending `chat.send` RPCs without retaining the original message/idempotency key for replay. I did not run a live browser reproduction because this review is read-only.

## Solution Assessment

Yes, the narrowest maintainable fix is to use the existing server ACK/idempotency contract from the Control UI send path. PR #73154's localStorage busy-queue persistence is useful adjacent work, but it does not by itself solve unACKed normal sends during reconnect.
