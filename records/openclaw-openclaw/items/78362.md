## Summary

Keep open: current main still serves the Control UI with `script-src 'self'` while browser UI code imports Zod v4 schemas, and upstream Zod probes `Function` unless `jitless` is enabled. The config-fetch 401 portion appears partly covered or expected on current main, but the CSP/Zod problem remains source-reproducible.

## What This Changes

The issue asks OpenClaw to stop Control UI CSP eval violations from browser-side Zod and avoid or gracefully handle initial 401s from the runtime config fetch.

## Best Possible Solution

Initialize browser-side Zod in jitless mode before Control UI schemas can parse, preserve the strict CSP, and leave the authenticated config endpoint contract intact unless focused proof shows an avoidable valid-token 401 remains.

## Reproduction Assessment

Yes, at source level: current main still combines strict Control UI CSP, browser-side Zod schemas, and upstream Zod code that uses or probes `Function` unless `jitless` is enabled. I did not run a live browser repro in this read-only review, so this is source-reproducible rather than live-reproduced.

## Solution Assessment

Yes: configuring Control UI Zod usage for jitless operation is narrower and safer than adding `unsafe-eval` to the admin UI CSP. The config-fetch side should only change if a focused current-main test proves a valid dashboard token still produces an avoidable 401.
