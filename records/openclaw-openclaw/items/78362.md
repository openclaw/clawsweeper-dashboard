## Summary

Keep open: current main still has the strict Control UI CSP, browser-side Zod object schemas on the startup/settings path, and no Control UI `z.config({ jitless: true })` setup; the linked `unsafe-eval` PR was closed as the wrong security tradeoff, and the config-fetch 401 half is only partly mitigated by current auth retry logic.

## What This Changes

The issue reports Control UI Chrome CSP `eval` violations from Zod `Function()` use plus an initial authenticated `/__openclaw/control-ui-config.json` 401.

## Best Possible Solution

Initialize browser-side Zod in jitless mode before Control UI schemas can be constructed or parsed, keep the strict CSP, and only touch the bootstrap config fetch path if current browser proof shows valid credentials still produce an avoidable 401.

## Reproduction Assessment

Yes, at source level: current main serves strict `script-src 'self'`, imports and parses Zod object schemas in the Control UI startup/settings path, and has no jitless initialization. I did not perform a live browser reproduction in this read-only pass.

## Solution Assessment

Yes: configuring Control UI Zod for jitless operation is the narrowest maintainable repair and preserves the admin UI CSP. The already-closed https://github.com/openclaw/openclaw/pull/78366 approach is broader because it allows eval-equivalent code execution for the whole Control UI.
