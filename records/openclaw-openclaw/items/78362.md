## Summary

Keep open: current main still has the strict Control UI CSP and browser-bundled Zod schemas without a jitless initialization, so the central CSP/eval report remains actionable; the config-fetch 401 side is partly covered by newer auth retry/post-hello handling and related closed reports.

## What This Changes

The issue reports Control UI CSP violations from Zod's Function constructor and an initial 401 when fetching the Control UI config JSON.

## Best Possible Solution

Keep the strict Control UI CSP, initialize Zod in jitless mode before browser schemas are constructed, and treat the config-fetch 401 as covered unless a current-main live repro shows a remaining auth/bootstrap failure.

## Reproduction Assessment

Yes for the CSP side from source and dependency inspection: current main bundles Zod schemas into the Control UI while serving a CSP that blocks `Function`/eval, and zod@4.4.3 uses a JIT path unless configured jitless. I did not run a live browser reproduction.

## Solution Assessment

No, the closed `unsafe-eval` PR was not the best fix because it relaxes the Control UI CSP for every script. The narrower maintainable solution is a Control UI Zod jitless initialization plus focused tests and browser proof.
