## Summary

Keep open: current main and the latest release still serve a strict Control UI CSP while browser-side Zod schemas are parsed without a jitless setup, and the linked unsafe-eval workaround was closed unmerged as the wrong security tradeoff.

## What This Changes

The issue asks for the Control UI to avoid Zod `Function()` CSP violations under the strict admin UI CSP and to stop or gracefully handle the initial unauthorized config fetch.

## Best Possible Solution

Initialize browser-side Zod in jitless mode before Control UI schemas can be constructed or parsed, keep the strict CSP, and only adjust the bootstrap config fetch path if current browser proof shows the auth retry still leaves a real failure.

## Reproduction Assessment

Yes, at source level: current main serves `script-src 'self'`, imports and parses Zod object schemas in the Control UI startup/settings path, and has no jitless initialization. I did not establish a live browser reproduction in this read-only pass.

## Solution Assessment

Yes: configuring Control UI Zod for jitless operation before first schema access is the narrowest maintainable fix and preserves the admin UI CSP. The already-closed https://github.com/openclaw/openclaw/pull/78366 approach is broader because it permits eval-equivalent code execution for the whole Control UI.
