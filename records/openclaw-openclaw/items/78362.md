## Summary

Keep open: current main and the latest release still serve the Control UI with a strict CSP while browser UI code imports Zod without a jitless setup, so the central CSP/eval report remains actionable. The config-fetch 401 side is partly covered by current endpoint aliases, auth-candidate retry, and post-hello refresh behavior, so it should not expand the fix without a fresh browser repro.

## What This Changes

The issue reports Control UI CSP eval violations from Zod's Function constructor plus an initial unauthorized `/__openclaw/control-ui-config.json` fetch.

## Best Possible Solution

Keep the strict Control UI CSP, initialize or otherwise run Zod without dynamic code before browser-side Control UI schema parsing, and only change config-fetch behavior if a current-main browser repro shows a remaining config panel failure.

## Reproduction Assessment

Yes for the CSP half from source and dependency inspection: current main and v2026.6.10 serve strict `script-src` while browser UI imports Zod 4.4.3 without an observed jitless setup. The config-fetch half is not fully reproduced because current main has newer endpoint compatibility, auth retry, and post-hello refresh behavior.

## Solution Assessment

Yes for the safer direction: configure Zod jitless or otherwise avoid its dynamic-code path in the Control UI bundle while preserving strict CSP. Adding `unsafe-eval` would be broader than needed and was already rejected in the related closed PR.
