## Summary

Keep open: current main still combines a strict Control UI CSP with browser-side Zod schemas and no jitless initialization, so the CSP violation remains source-reproducible. The config-endpoint 401 is partly expected and partly covered by current auth retry behavior, but the CSP/Zod bug still needs a safe fix.

## What This Changes

The issue asks to stop Chrome CSP eval violations in the Control UI caused by Zod's Function-based JIT path and to address an initial bootstrap config 401.

## Best Possible Solution

Initialize Zod in jitless mode for the Control UI before browser schemas can parse, keep the strict CSP, and only adjust the bootstrap fetch path if a focused current-main valid-token 401 remains reproducible.

## Reproduction Assessment

Yes, at source level: current main serves a strict script-src while the Control UI bundle imports Zod schemas and lacks jitless initialization, and Zod v4.4.3 probes or uses Function unless jitless is set. I did not run a live browser repro in this read-only pass.

## Solution Assessment

Yes: using Zod's jitless configuration is the narrow maintainable fix because it targets the dependency behavior without weakening the admin UI CSP. The config-fetch side should stay on the authenticated endpoint contract unless a valid-token 401 is proven on current main.
