## Summary

Keep open: current main still combines strict Control UI `script-src 'self'`, browser-side Zod object schemas, and no Control UI `z.config({ jitless: true })` initialization. The config-fetch 401 portion is partly covered by current auth/retry behavior, but the CSP/Zod problem remains source-reproducible.

## What This Changes

The issue reports Control UI CSP `eval` violations from browser-side Zod JIT plus a `/__openclaw/control-ui-config.json` 401 on initial load.

## Best Possible Solution

Initialize browser-side Zod in jitless mode before Control UI schemas can parse, keep the strict CSP, and only change the bootstrap fetch path if focused current-main proof shows a valid credential still produces an avoidable 401.

## Reproduction Assessment

Yes, at source level: current main serves strict `script-src 'self'`, imports and parses Zod schemas in the Control UI bundle, and uses Zod 4.4.3 without `jitless`. I did not establish a live browser reproduction in this read-only pass.

## Solution Assessment

Yes: configuring Control UI Zod usage for jitless operation is the narrowest maintainable fix and avoids weakening the admin UI CSP. Adding `unsafe-eval` would hide the warning by broadening script execution policy and is not the preferred repair.
