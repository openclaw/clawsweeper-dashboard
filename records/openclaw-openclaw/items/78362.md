## Summary

Keep open: current main and the latest release still serve a strict Control UI CSP while browser UI code imports Zod without a jitless initialization, so the central CSP/eval report remains actionable. The config-fetch 401 side appears partly addressed by current auth-candidate retry and post-hello bootstrap refresh behavior, but that does not close the remaining CSP/Zod work.

## What This Changes

The issue asks OpenClaw to stop Control UI CSP eval violations from Zod's Function-based JIT path and avoid the initial authenticated config fetch failure.

## Best Possible Solution

Keep the strict Control UI CSP, initialize Zod in jitless mode before any browser-side Control UI schema parsing, and only reopen the config-fetch portion if a current-main live repro shows a remaining authenticated bootstrap failure.

## Reproduction Assessment

Yes for the CSP half from source and dependency inspection: current main and v2026.6.8 bundle browser-side Zod while serving a CSP that blocks `Function`/eval, and zod@4.4.3 uses or probes JIT unless configured jitless. The config-fetch half is not fully reproduced here because current main has newer auth retry and post-hello refresh code.

## Solution Assessment

Yes, the issue's safer direction is the best current fix: configure Zod jitless in the Control UI bundle instead of adding `unsafe-eval` to the gateway CSP. The 401 side should stay scoped to a fresh repro because current code already retries authenticated bootstrap fetches and refreshes after hello.
