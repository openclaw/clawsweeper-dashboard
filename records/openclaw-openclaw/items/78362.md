## Summary

Keep open: current main and the latest release still serve the Control UI with strict `script-src` while browser UI code imports Zod 4.4.3 without a jitless setup, so the CSP/eval half remains actionable. The config-fetch 401 half is partly mitigated by current auth-candidate retry, post-hello refresh, and endpoint compatibility aliases, so further auth changes need fresh browser proof.

## What This Changes

The issue reports Control UI browser CSP violations from Zod using the Function constructor and an initial unauthorized config fetch during token-authenticated startup.

## Best Possible Solution

Keep strict Control UI CSP, configure Zod jitless before browser-side schema parsing, add focused regression coverage and browser proof, and only change config-fetch behavior if current-main proof shows a remaining 401 user failure.

## Reproduction Assessment

Yes for the CSP half from source and dependency inspection: current main and v2026.6.10 serve strict `script-src`, browser UI imports Zod 4.4.3, and no jitless setup is present. The initial 401 half is not fully reproduced because current main has newer auth retry, endpoint aliases, and post-hello refresh behavior.

## Solution Assessment

Yes for the safer direction: configure Zod jitless or otherwise avoid its dynamic-code path in the Control UI bundle while preserving strict CSP. Adding `unsafe-eval` is broader than needed and the related pull request using that approach was closed unmerged.
