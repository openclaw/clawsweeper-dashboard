## What This Changes

This PR adds a Discord lifecycle guard that treats abort-time live `reconnect-exhausted` gateway events as clean shutdown, adds a focused regression test, and records a changelog note.

## Best Possible Solution

Land this PR, or an equivalent narrow change, in the Discord plugin lifecycle boundary with the focused regression test and contributor credit preserved; the shared gateway health-monitor policy does not need to change for this race.

## Reproduction Assessment

Yes. The high-confidence path is an active Discord gateway lifecycle whose abort signal fires, then a live `reconnect-exhausted` event reaches the lifecycle handler before the outer `finally` sets `lifecycleStopping`; current main returns `stop` and rejects through `waitForDiscordGatewayStop`.

## Solution Assessment

Yes. This is the narrowest maintainable fix I found: use the existing abort signal in the Discord lifecycle handler to identify expected shutdown and leave unrelated supervisor, core channel health-monitor, and restart behavior untouched.
