## Summary

Keep this PR open: it addresses a real Codex native hook relay gap, but the current patch is not merge-ready because the approval bridge can run before-tool policy twice after a native relay no-decision, and the supplied proof is still focused tests rather than real runtime behavior.

## What This Changes

The PR extends Codex native hook relay config to public `/btw` side-question runs, forwards app-server command approvals through the active relay, normalizes `exec_command` policy input, and adds focused tests.

## Best Possible Solution

Keep the scoped relay extension, change the approval bridge so a handled native no-decision reaches the app-server/plugin approval path without re-running before-tool policy, add focused regression coverage for that path, and require redacted live runtime proof before merge.

## Reproduction Assessment

Yes from source inspection: current main enables the native relay for normal Codex app-server attempts but not public side questions, and approval handling does not receive the active relay. I did not establish a live failing `/btw` run in this read-only review.

## Solution Assessment

No, not as written: the design direction is appropriate, but the approval bridge currently re-enters before-tool policy after `nativeHook.invoke` returns no decision. The patch also still needs contributor-supplied real behavior proof for the runtime path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Avoid re-running approval policy after native relay no-decisions:** `extensions/codex/src/app-server/approval-bridge.ts:330-344`
  - body: When `nativeHook.invoke` returns no decision, the relay has already executed `runBeforeToolCallHook` for the same command. Falling through to the report-mode call here runs hooks/trusted policy a second time; approval-required policies can prompt from the relay and then be declined by the fallback, and loop/history state is counted twice. Treat a handled no-decision as falling through to the app-server/plugin approval route without another before-tool call, or make the relay invocation use a report-only/no-side-effect path.
  - confidence: 0.88
