## Summary

Keep open. Current main has useful related hardening, including reload deferral and safe pre-connect Telegram send retries, but ordinary Telegram same-surface replies still flow through an in-process dispatcher and direct Bot API delivery rather than a durable journaled path, so the reported hot-reload loss mode is not proven fixed.

## What This Changes

The issue asks OpenClaw to prevent silent loss of in-flight Telegram replies when hot reload overlaps sendChatAction or Telegram delivery network failures.

## Best Possible Solution

Add a focused same-surface Telegram hot-reload/network-failure regression, then either journal final Telegram replies through shared durable delivery primitives before Bot API delivery or surface a retryable hard failure consistently in Telegram, UI, and session state.

## Reproduction Assessment

Yes. A high-confidence regression can construct a Telegram inbound turn where Provider and Surface are both telegram, inject sendChatAction or Bot API delivery failure during a reload window, and assert the final same-surface reply path has no durable replay entry.

## Solution Assessment

Yes. The narrow maintainable direction is to cover the Telegram dispatcher path directly and reuse the shared durable outbound/session-delivery primitives where they preserve Telegram preview/final semantics; a Telegram-only queue should be a fallback, not the first design.
