## Summary

Keep open. Current main has restart deferral, fallback attempts, and a durable routed outbound queue, but ordinary Telegram-to-Telegram auto-replies still use the in-process dispatcher and direct Telegram delivery path, so the maintainer-identified hot-reload gap remains valid.

## What This Changes

The issue asks for Telegram in-flight replies during hot reload or sendChatAction/network failure to be durably replayed or fail visibly instead of being silently lost.

## Best Possible Solution

Add a focused same-surface Telegram hot-reload/network-failure regression, then reuse shared durable delivery primitives or surface a retryable failure consistently in Telegram, UI, and session state.

## Reproduction Assessment

Yes. A high-confidence regression can simulate Telegram inbound context with Provider and Surface both set to telegram, force sendChatAction or Bot API delivery failure during a reload window, and assert that current main has no durable replay for the final same-surface reply.

## Solution Assessment

Yes. The narrow maintainable fix is to cover the Telegram dispatcher path directly and either journal final replies through existing shared delivery queues or fail the turn visibly; a separate Telegram-only queue should be avoided unless shared primitives cannot preserve preview/final semantics.
