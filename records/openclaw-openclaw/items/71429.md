## Summary

Keep open. Current main still routes ordinary Telegram-to-Telegram replies through the in-process dispatcher and direct Telegram delivery path, while the durable write-ahead queue remains on the routed outbound path; the maintainer discussion explicitly says this Telegram hot-reload gap is not closed.

## What This Changes

Prevent Telegram same-surface hot-reload or sendChatAction delivery failures from silently dropping in-flight final replies.

## Best Possible Solution

Add a focused Telegram same-surface hot-reload/network-failure regression, then either journal final replies through an existing shared durable delivery primitive before Bot API delivery or fail the turn visibly with retry guidance in Telegram, UI, and session state.

## Reproduction Assessment

Yes. A high-confidence regression can construct a Telegram inbound turn with Provider and Surface both set to telegram, inject sendChatAction or Bot API delivery failure during a reload window, and assert that current main has no durable replay for the final same-surface reply.

## Solution Assessment

Yes. The narrow maintainable path is to cover the Telegram dispatcher path directly and reuse shared delivery/recovery primitives where they preserve Telegram semantics; a Telegram-only queue should be a fallback, not the first design.
