## Summary

Keep open. Current main still sends ordinary Telegram same-surface replies through the in-process dispatcher and direct Telegram delivery path, while the durable outbound/restart-continuation queues cover other paths; the maintainer discussion explicitly leaves this Telegram hot-reload gap unresolved.

## What This Changes

The issue requests reliable handling for Telegram in-flight replies during hot reload when sendChatAction or Bot API delivery fails, either through durable replay or explicit visible failure.

## Best Possible Solution

Add a focused same-surface Telegram hot-reload/network-failure regression, then route final Telegram replies through shared durable delivery primitives where safe or mark the turn visibly failed with retry guidance in Telegram, UI, and session state.

## Reproduction Assessment

Yes. A high-confidence regression can simulate a Telegram inbound turn with Provider and Surface both set to telegram, inject sendChatAction or Bot API delivery failure during a reload window, and assert that current main has no durable replay for the final same-surface reply.

## Solution Assessment

Yes. The narrow maintainable solution is to cover the Telegram dispatcher path directly and reuse shared durable delivery primitives or explicit failure state; a separate Telegram-only queue should be a fallback only if shared semantics cannot preserve preview/final behavior.
