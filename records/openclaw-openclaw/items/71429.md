## Summary

Keep open. Current main still leaves normal Telegram-to-Telegram auto-replies on the in-process dispatcher and direct Bot API delivery path, while the durable queues cover routed outbound and restart-continuation paths; the maintainer comments already identify this remaining gap and the relevant source still matches it.

## What This Changes

Requests reliable handling for Telegram auto-replies during gateway hot reload when `sendChatAction` or Bot API delivery fails, either by durable replay or a visible retryable failure instead of silent loss.

## Best Possible Solution

Keep this open until a focused regression covers same-surface Telegram replies during reload/network failure, then reuse the existing durable outbound/session delivery primitives where safe or convert the failed turn into a visible retryable state across Telegram, UI, and session state.

## Reproduction Assessment

Yes. A high-confidence regression can simulate a Telegram inbound turn with `Provider` and `Surface` both `telegram`, inject `sendChatAction` or direct Bot API delivery failure during a reload window, and assert the final same-surface reply is not durably replayed on current main.

## Solution Assessment

Yes. The narrow maintainable direction is to reuse the existing durable delivery primitives or add an explicit visible abort path; a Telegram-only journal should be avoided unless the shared queue cannot preserve Telegram preview/final semantics safely.
