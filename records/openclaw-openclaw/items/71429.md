## Summary

Keep open. Current main has related hardening for reload deferral, streaming preview edits, and best-effort Telegram error fallbacks, but normal same-surface Telegram final replies can still go through the in-process dispatcher to direct Bot API delivery without a durable write-ahead entry. The maintainer discussion also explicitly left this report open for that remaining Telegram delivery boundary.

## What This Changes

The issue requests durable replay or a visible hard failure for in-flight Telegram outbound replies that are lost during gateway hot reload or Telegram `sendChatAction`/network delivery failure.

## Best Possible Solution

Keep this open until a focused regression covers a Telegram same-surface turn during reload or Bot API failure, then route final replies through an existing durable delivery primitive where safe or record a visible retryable failure across Telegram, UI, and session state when safe replay is not possible.

## Reproduction Assessment

Yes. A high-confidence regression can simulate a Telegram inbound context with `Provider` and `Surface` both `telegram`, force `deliverReplies` or typing/direct Bot API delivery failure during a reload window, and assert current main leaves no durable queue entry or durable visible failure state.

## Solution Assessment

Yes for the proposed direction, not for the current implementation. The maintainable fix should reuse shared outbound/session delivery primitives or an explicit visible abort path rather than adding an isolated Telegram-only journal that can drift from preview/final semantics.
