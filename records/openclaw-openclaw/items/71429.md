## Summary

Keep open. Current main still routes ordinary Telegram-to-Telegram auto-replies through the in-process reply dispatcher and direct Telegram Bot API delivery, while the durable outbound/session queues cover routed or restart-continuation paths; the maintainer discussion explicitly called out this same remaining gap.

## What This Changes

The issue requests durable replay or an explicit visible failure for Telegram in-flight outbound replies when hot reload and sendChatAction/network failures occur.

## Best Possible Solution

Keep the issue open until normal same-surface Telegram replies are covered by a focused reload/network-failure regression and either journaled through an existing durable pre-delivery path or converted into a visible retryable failure across Telegram, UI, and agent state.

## Reproduction Assessment

Yes. A high-confidence regression can simulate a Telegram inbound turn where Provider and Surface are both telegram, force sendChatAction or direct Bot API delivery failure during a reload window, and assert that current main has no durable replay for the final same-surface reply.

## Solution Assessment

Yes. The maintainable direction is to reuse the existing durable outbound/session delivery primitives or add an explicit visible abort path; a Telegram-only ad hoc journal should be avoided unless the shared queue cannot preserve preview/final semantics safely.
