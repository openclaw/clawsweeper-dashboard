## Summary

Keep open. Current main still has related restart deferral and durable outbound/session queues, but ordinary Telegram-to-Telegram auto-replies still bypass those queues and rely on in-process direct delivery; the maintainer discussion already identified this gap and I found no current-main fix for it.

## What This Changes

Requests reliable handling for normal Telegram auto-replies during gateway hot reload when sendChatAction or direct Bot API delivery fails, by durably replaying the reply or surfacing a visible retryable failure instead of silently dropping it.

## Best Possible Solution

Keep this open until a focused regression covers same-surface Telegram auto-replies during reload/network failure, then either route final Telegram replies through an existing durable delivery primitive before Bot API delivery or convert the failure into a visible retryable state in Telegram, UI, and agent/session state.

## Reproduction Assessment

Yes. A high-confidence regression can simulate a Telegram inbound turn with Provider and Surface both set to telegram, inject sendChatAction or direct Bot API delivery failure during a reload window, and assert that current main does not durably replay the final same-surface reply.

## Solution Assessment

Yes. Reusing the existing outbound/session delivery primitives or adding an explicit visible abort path is the narrowest maintainable direction; a separate Telegram-only journal should be avoided unless the shared queue cannot preserve Telegram preview/final semantics safely.
