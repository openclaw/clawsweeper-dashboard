## Summary

Keep open. Current main still routes ordinary Telegram-to-Telegram replies through an in-process dispatcher and direct Telegram Bot API delivery, while the durable delivery queues cover routed outbound and restart-continuation paths; the maintainer discussion explicitly identified this remaining gap and the current source still matches it.

## What This Changes

The issue reports silent loss of in-flight Telegram replies when gateway hot reload overlaps with sendChatAction or Telegram Bot API network failures.

## Best Possible Solution

Keep this issue open until normal same-surface Telegram replies are covered by a focused reload/network-failure regression and either journaled before Bot API delivery through an existing durable delivery primitive or converted into a visible retryable failure in Telegram, UI, and agent state.

## Reproduction Assessment

Yes. A high-confidence regression can simulate a Telegram inbound turn where Provider and Surface are both telegram, fail sendChatAction or direct Bot API delivery during a reload window, and assert that current main has no durable replay for the final same-surface reply.

## Solution Assessment

Yes. The best direction is to reuse the existing generic outbound/session delivery primitives or add an explicit visible abort path; a Telegram-only ad hoc journal should be avoided unless the shared queue cannot preserve Telegram preview/final semantics safely.
