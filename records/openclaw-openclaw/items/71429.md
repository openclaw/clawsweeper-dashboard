## Summary

Keep open. Current main still sends ordinary Telegram-to-Telegram replies through the in-process dispatcher and direct Bot API delivery path, while the durable write-ahead queue is on the routed outbound path; the maintainer discussion explicitly says the remaining Telegram hot-reload gap is not closed.

## What This Changes

The issue asks OpenClaw to prevent silent loss of in-flight Telegram replies during gateway hot reload or sendChatAction/network failure, either by durable replay or visible retryable failure.

## Best Possible Solution

Add a focused same-surface Telegram hot-reload/network-failure regression, then reuse shared durable delivery primitives where safe or convert undeliverable turns into a visible retryable failure across Telegram, UI, and session state.

## Reproduction Assessment

Yes. A high-confidence regression can simulate a Telegram inbound turn with Provider and Surface both set to telegram, inject sendChatAction or Bot API delivery failure during a reload window, and assert current main has no durable replay for the final same-surface reply.

## Solution Assessment

Yes. The narrow maintainable path is to cover the Telegram dispatcher path directly and either journal final replies through existing shared delivery queues or fail the turn visibly; a new Telegram-only durable queue should be avoided unless shared primitives cannot preserve Telegram preview/final semantics.
