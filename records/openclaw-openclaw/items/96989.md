## Label Justifications

- `P2`: This is normal-priority provider stream hardening with a focused correctness blocker and no evidence of an active widespread outage.
- `merge-risk: 🚨 compatibility`: The SSE cap can reject existing valid streams based on transport chunk coalescing, and the JSON synthesis cap changes oversized responses to fail closed.
- `merge-risk: 🚨 availability`: A valid OpenAI-compatible provider streaming response can fail before complete SSE events are processed.

## Summary

Keep open: the PR addresses a real unbounded-buffer provider stream problem, but the SSE sanitizer cap is placed before complete events are drained, so a valid coalesced transport chunk can be rejected as oversized.

## What This Changes

The PR adds local caps and tests for provider-transport SSE sanitization and streaming JSON-to-SSE synthesis.

## Best Possible Solution

Keep the caps, but drain complete SSE events first and enforce the 64 KiB limit only on the remaining unterminated tail; separately accept or tune the 16 MiB JSON synthesis cap.

## Reproduction Assessment

Yes, by source inspection: current main appends provider SSE text until a boundary and accumulates streaming JSON until EOF with no cap. I did not run a current-main repro in this read-only pass, but the PR body includes live node:http output for the oversized bad paths.

## Solution Assessment

No. The cap is the right mitigation, but the SSE guard placement is not the best fix because it should process complete events before applying the unterminated-tail limit.
