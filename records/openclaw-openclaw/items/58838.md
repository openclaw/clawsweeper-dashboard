## Summary

This issue is still necessary: current main and the latest release still retry each staged split without a circuit breaker, while https://github.com/openclaw/openclaw/pull/86900 is an open linked fix candidate rather than a merged implementation.

## What This Changes

The issue requests stopping staged compaction after repeated summarizer fallback failures so an unavailable summarizer does not burn tokens across every split.

## Best Possible Solution

Review and land or reject https://github.com/openclaw/openclaw/pull/86900; the desired end state is a typed staged-compaction fallback breaker that preserves current partial-summary recovery while avoiding repeated unavailable-summarizer calls.

## Reproduction Assessment

Yes, from source inspection. Current `summarizeInStages()` still calls `summarizeWithFallback()` for every planned split, and `summarizeChunks()` can spend three retry attempts per chunk before returning the generic fallback path.

## Solution Assessment

Yes as a likely implementation direction, but this issue should stay open until the linked PR lands or maintainers reject its policy tradeoff. Typed fallback state is a maintainable way to avoid string matching; the fixed cutoff remains a maintainer session-state decision on the PR.
