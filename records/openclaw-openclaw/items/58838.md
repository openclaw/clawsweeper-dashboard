## Summary

Keep open. Current main and the latest release still have the staged compaction loop call the fallback summarizer for every split without a consecutive-failure breaker, and the remaining work is already owned by an open linked PR with closing syntax.

## What This Changes

The issue asks OpenClaw to stop repeated compaction summarization attempts after consecutive summarizer failures so long conversations do not burn tokens across every staged split.

## Best Possible Solution

Review and either land or reject https://github.com/openclaw/openclaw/pull/86900 as the canonical fix, with regression coverage proving that staged compaction stops after consecutive generic fallbacks while preserving useful partial summaries.

## Reproduction Assessment

Yes. Source inspection shows `summarizeInStages()` still calls `summarizeWithFallback()` for every staged split, and each split can reach `summarizeChunks()` with a three-attempt retry budget before returning the generic fallback.

## Solution Assessment

Yes in direction: a focused staged fallback breaker or equivalent typed fallback result is the narrowest maintainable fix. Current main only preserves partial summaries after mid-chain failures; it does not stop repeated generic fallback work across staged splits.
