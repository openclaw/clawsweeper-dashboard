## Summary

Keep open. Current main and v2026.6.6 still process every staged compaction split through the same fallback path with a per-chunk retry budget, and the open linked PR is the active fix path for the remaining token-burn behavior.

## What This Changes

The issue requests a compaction guard so staged summarization stops after consecutive summarizer failures instead of retrying every chunk or split.

## Best Possible Solution

Current main should stop staged compaction after consecutive generic fallback results while preserving the partial-summary behavior already on main; once the linked PR lands, this issue can close as implemented.

## Reproduction Assessment

Yes, from source inspection. `summarizeInStages()` still calls `summarizeWithFallback()` for every planned split, and each split can still reach `summarizeChunks()` with a three-attempt retry budget before returning the generic fallback string.

## Solution Assessment

Yes in direction: a focused staged fallback breaker or equivalent typed fallback result is the narrowest maintainable fix. Current main only preserves partial summaries after mid-chain failures; it does not stop repeated generic fallback work across staged splits.
