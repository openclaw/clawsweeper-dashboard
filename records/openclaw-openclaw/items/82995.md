## Summary

Keep open: this is a focused Gemini transport bug fix, but the current patch rejects the documented Gemini 3 skip-validator fallback in the OpenAI-compatible path and needs changes before merge.

## What This Changes

Adds strict Base64 validation before forwarding Gemini thought signatures through the native Google and OpenAI-compatible transport serializers.

## Best Possible Solution

Validate only real captured signatures, preserve the documented Gemini 3 fallback sentinel, add regression coverage for both invalid real signatures and fallback preservation, and coordinate any native Google replay changes with the open related repair.

## Reproduction Assessment

Yes. The review finding is source-reproducible: for a Gemini 3 OpenAI-compatible replay with no captured signature, `fallbackSig` becomes `skip_thought_signature_validator`, and the new guard rejects that value before writing it.

## Solution Assessment

No. The native validation direction is reasonable, but the OpenAI-compatible change validates the fallback sentinel as if it were a real signature; the safer fix is to sanitize captured signatures while preserving the documented fallback path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve the Gemini skip-validator fallback:** `src/agents/openai-transport-stream.ts:2827-2832`
  - body: For Gemini 3, `fallbackSig` is intentionally set to `skip_thought_signature_validator` when no captured signature exists. This new guard runs after `sigById.get(id) ?? fallbackSig`, and the sentinel is not Base64, so the OpenAI-compatible path now drops the fallback and sends tool calls without `extra_content.google.thought_signature`. That regresses the existing fallback behavior and can bring back the missing-`thought_signature` 400s the fallback was meant to avoid.
  - confidence: 0.91
