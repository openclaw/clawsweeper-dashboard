## Summary

Keep open: this PR targets a real Gemini thought-signature failure, but the latest patch still leaves native text/thinking signatures unsanitized and can suppress the Gemini 3 compat fallback after rejecting a bad captured signature.

## What This Changes

The PR adds truncation-footprint checks to native Google tool-call thought-signature sanitization and OpenAI-compatible tool-call signature injection, plus an empty README newline.

## Best Possible Solution

Use one shared truncation sanitizer for every same-route Gemini thoughtSignature field, fall back to the Gemini 3 skip-validator dummy when a captured function-call signature is rejected, add focused regression coverage, and remove the README trigger-only newline.

## Reproduction Assessment

Yes, at source level. Current main and the PR head still expose raw native text/thinking signature forwarding, and the Google contract plus PR logs support the 400 failure mode; I did not run the live Gemini replay locally.

## Solution Assessment

No. The truncation-filter direction is maintainable, but the current patch misses native text/thinking signatures and should preserve Gemini 3's documented fallback when a captured compat function-call signature is invalid.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Fall back when rejecting bad compat signatures:** `src/agents/openai-transport-stream.ts:2829-2833`
  - body: For Gemini 3 compat models, `fallbackSig` exists because omitting a function-call signature can fail validation. With this change, a same-route truncated value from `sigById` wins over the fallback and then `continue`s here, so the outgoing tool call gets no `thought_signature`; after rejecting a captured value, use the skip-validator fallback when it is available.
  - confidence: 0.86
- **[P2] Guard native text and thinking signatures too:** `extensions/google/transport-stream.ts:215-223`
  - body: The new sanitizer is only applied through the native tool-call path. Same-route text and thinking blocks still attach `block.textSignature` and `block.thinkingSignature` directly, so a compaction-truncated signature on either part type can still be forwarded to Gemini and reproduce the 400 this PR is meant to prevent.
  - confidence: 0.9
