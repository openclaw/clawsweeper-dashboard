Keep open: the PR targets a real WebChat media-history bug, but the patch does not yet cover the reported image-tool `media://` path, misses multiple-attachment transcript markers, adds a likely typecheck failure, and has no real WebChat proof.

## What This Changes

The PR exports and updates `extractTranscriptUserText` to strip simple media markers, adds unit tests for that helper, and skips workspace-relative joining for `media://` URLs in `loadWebMediaInternal`.

## Best Possible Solution

Fix the patch so media-note normalization covers all generated marker forms, preserve `media://` refs before any workspace join in the image-tool/media-loader boundary, add focused single/multiple upload and workspaceDir regressions, then attach real WebChat proof.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: WebChat image uploads can persist transcript media notes, and `image-tool.ts` can rewrite `media://inbound/...` under `workspaceDir` before `loadWebMedia` runs. I did not execute a live WebChat flow in this read-only review.

## Solution Assessment

No, this is not yet the best fix because one half of the patch is applied after the failing caller has already corrupted the media URI, and the transcript text normalizer does not cover multiple generated marker forms.
