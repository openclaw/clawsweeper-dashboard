## Summary

Keep open: the Mattermost recovery problem is real and the screenshot proof is sufficient, but the patch still treats an empty pending-history window as recovery even though OpenClaw clears that window after normal successful turns.

## What This Changes

The PR adds a Mattermost thread fetch helper, seeds empty in-memory thread histories from the server before inbound context assembly, and adds unit coverage for that backfill behavior.

## Best Possible Solution

Keep the Mattermost-owned server backfill, but gate it on a real recovery, new-session, or missing-session condition and add regression coverage proving active thread follow-ups do not fetch when only the pending-history window is empty.

## Reproduction Assessment

Yes, at source level. Current main keeps Mattermost thread context in a process-local map and has no server-thread fetch, so restart or session clear can leave an inbound thread turn with no prior thread messages.

## Solution Assessment

No, not as written. The owner boundary and Mattermost API are plausible, but the trigger needs a recovery-aware gate because an empty pending-history map is also normal after successful active thread replies.
