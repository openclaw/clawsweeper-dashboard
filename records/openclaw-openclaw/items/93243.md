## Summary

Keep open: the bug is real and the screenshot proof is sufficient, but the patch is not merge-ready because an empty Mattermost history window is also normal after successful active-thread turns, so this can rehydrate full server thread context and await an extra REST call outside the intended recovery case.

## What This Changes

The PR adds a Mattermost thread fetch helper, seeds empty thread history windows from `GET /posts/{root}/thread` before inbound context assembly, and adds unit coverage for the helper.

## Best Possible Solution

Keep the Mattermost-owned server backfill, but gate it on a real recovery, new-session, or missing-context signal and cover both recovery and active-thread steady state with focused tests.

## Reproduction Assessment

Yes, at source level. Current main keeps Mattermost thread context in a process-local map and has no server-thread fetch, so restart or session clear can leave an inbound thread turn with no prior thread messages.

## Solution Assessment

No, not as written. The Mattermost-owned fetch/backfill boundary is plausible, but the trigger must be recovery-aware because an empty pending-history window is also normal after successful active thread replies.
