## Summary

Keep open: current main still has the reported channel-lifecycle path, and the related issues track adjacent event-loop starvation/timeout work rather than this Slack reconnect poison state.

## What This Changes

The issue reports that Slack socket-mode reconnect is suppressed after a channel stop timeout leaves the account in manual-stop state.

## Best Possible Solution

Add a narrow gateway lifecycle distinction between explicit manual stops and recovery/restart stops, then cover timed-out recovery stops without weakening the duplicate-task guard.

## Reproduction Assessment

Yes, from source inspection: an abort-ignoring channel task can make `stopChannel()` time out, after which `manuallyStopped` suppresses both the tracked task's auto-restart and future health-monitor checks. I did not run a live Slack socket scenario in this read-only pass.

## Solution Assessment

No, the current timeout behavior is not the best solution; the safer repair is to scope manual-stop persistence to user-initiated stops and let health-monitor restarts recover after timeout without launching duplicate tasks.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/77651. In the gateway channel lifecycle, a health-monitor/recovery restart must not permanently poison a channel account as manually stopped when `stopChannel()` times out waiting for an abort-ignoring task. Preserve explicit user/operator manual-stop behavior and the existing duplicate-task guard. Prefer an internal stop reason/options seam or monitor-specific restart helper over clearing `manuallyStopped` for every stop. Add regression coverage for a timed-out health-monitor restart path that can recover on later task settlement or a later health pass, plus coverage that explicit manual stops still suppress auto-restart and that a second account task is not started while the original task remains unresolved. Likely touch `src/gateway/server-channels.ts`, `src/gateway/channel-health-monitor.ts`, `src/gateway/server-channels.test.ts`, and `src/gateway/channel-health-monitor.test.ts`; add a `CHANGELOG.md` fix entry because this is user-facing Slack/channel recovery behavior. Do not change Slack Socket Mode transport tuning or add a config option.
