## Summary

Keep open: current main still contains the source-level state trap where a timed-out channel stop can leave the account marked manually stopped and block health-monitor recovery.

## What This Changes

Request a fix so Slack Socket Mode channels recover after a channel stop timeout triggered by event-loop starvation instead of remaining manually stopped.

## Best Possible Solution

Add a restart-aware recovery path for health-monitor-triggered channel restarts that clears or bypasses the manual-stop gate only for recovery stops, with regression coverage for the stop-timeout path.

## Reproduction Assessment

Yes at source level: current main follows the reported path through `stopChannel`, the health monitor, and the restart gates. I did not live-replay Slack heartbeat starvation on macOS during this read-only review.

## Solution Assessment

No for the broad Option A as written, because clearing the manual-stop marker for every stop timeout could restart channels after a true manual stop. The maintainable fix should scope the cleanup or bypass to health-monitor restart intent.

## Repair Work Prompt

Fix OpenClaw issue https://github.com/openclaw/openclaw/issues/77651. Current main lets a health-monitor restart call `stopChannel`, which always marks the channel account as manually stopped; if `waitForChannelStopGracefully` times out, the old task remains tracked, `startChannel` returns early, and later cleanup/health-monitor checks keep the Slack account from reconnecting. Implement a restart-aware stop/recovery path in the gateway channel manager and/or health monitor so monitor-triggered recovery after a stop timeout can clear or bypass the manual-stop gate and eventually reconnect. Preserve true user/admin manual stops, gateway shutdown, reload, and duplicate-task protection while the old task is still tracked. Do not add a config option. Add regression coverage for stop timeout followed by health-monitor recovery, and update the existing stop-timeout expectations only as needed. Add a user-facing `CHANGELOG.md` fix entry.
