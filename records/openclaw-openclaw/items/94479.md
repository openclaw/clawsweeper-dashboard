## Summary

Current main and the latest release already solve the observable Control UI problem: submitted queued sends render in the chat thread before the `chat.send` ACK or assistant stream. The issue was reported against older `2026.4.21-5`, while the shipped pending-send renderer covers the user-visible behavior without adding a second transient `chatMessages` source of truth.

## What This Changes

The issue asks for Control UI user messages to appear immediately after send instead of waiting for the assistant response to start streaming.

## Best Possible Solution

Keep the shipped queued-send thread renderer as the pending-send display path; only open new work if maintainers want a separate internal `chatMessages` state-contract change.

## Reproduction Assessment

No current-main failing reproduction remains. Current source and shipped E2E coverage defer `chat.send` and verify the prompt appears in `.chat-thread` before the ACK resolves.

## Solution Assessment

Yes, the current implementation is the best current solution for the reported user-visible problem. It renders the pending turn from the existing queue state instead of duplicating transient messages into the persisted transcript state.
