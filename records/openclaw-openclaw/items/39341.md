## Summary

Keep open: current main and the latest release still have the Discord security-audit false positive, and the issue is already paired with an open closing PR that should be reviewed, revised, and landed or replaced before this issue closes.

## What This Changes

The issue requests that the Discord security audit honor top-level `channels.discord.allowFrom` when deciding whether native slash commands have an owner allowlist.

## Best Possible Solution

Review, revise, and land or replace https://github.com/openclaw/openclaw/pull/72467 with a resolver-based audit patch plus regression coverage for top-level and named-account precedence cases.

## Reproduction Assessment

Yes, at source level. With native Discord commands enabled, a configured guild, no guild/channel users allowlist, no credential-store or legacy allowlist, and only top-level `channels.discord.allowFrom`, current main leaves `ownerAllowFromConfigured` false and reaches the `no_allowlists` warning path.

## Solution Assessment

Yes on direction, with one refinement. The maintainable fix should reuse `resolveDiscordAccountAllowFrom({ cfg, accountId })` plus credential-store entries rather than adding another direct merged-config read.
