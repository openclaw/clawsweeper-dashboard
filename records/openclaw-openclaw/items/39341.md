## Summary

Keep open: current main still omits top-level `channels.discord.allowFrom` from the Discord native-command `no_allowlists` owner check, and an open exact-scope PR already targets this issue with closing syntax.

## What This Changes

The issue asks the Discord security audit to treat top-level `channels.discord.allowFrom` as an owner allowlist for the native slash-command `no_allowlists` warning.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/72467 or an equivalent narrow Discord plugin change that aligns the audit owner allowlist check with docs/runtime and adds regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: native commands enabled, a configured guild, no guild/channel users allowlist, no credential-store or legacy allowlist, and only top-level `channels.discord.allowFrom` leaves `ownerAllowFromConfigured` false.

## Solution Assessment

Yes. The narrow maintainable fix is to include top-level/effective Discord `allowFrom` in `ownerAllowFromConfigured` and add focused regression coverage; the open PR follows that direction but has not merged.
