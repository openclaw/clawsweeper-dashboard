## Summary

Keep open: current main and the latest release still have the Discord security-audit false positive, and an open exact-scope PR references this issue with closing syntax, so the issue should close only after that PR or an equivalent corrected fix merges.

## What This Changes

The issue reports that the Discord security audit ignores top-level `channels.discord.allowFrom` when deciding whether native slash commands have an owner allowlist.

## Best Possible Solution

Land or replace the linked PR with a resolver-based patch that makes the audit use `resolveDiscordAccountAllowFrom({ cfg, accountId })` plus store allowlist entries, with regression coverage for top-level and account-precedence cases.

## Reproduction Assessment

Yes, at source level. With native Discord commands enabled, a configured guild, no guild/channel users allowlist, no credential-store or legacy allowlist, and only top-level `channels.discord.allowFrom`, current main leaves `ownerAllowFromConfigured` false and still reaches the warning.

## Solution Assessment

Yes, the requested fix direction is the narrow maintainable path, but it should use the existing effective Discord allowFrom resolver rather than another direct merged-config read. The issue should remain open while the linked implementation PR is reviewed or replaced.
