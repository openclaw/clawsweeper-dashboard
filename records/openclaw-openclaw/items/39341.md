## Summary

Keep open: current main and v2026.5.2 still omit top-level `channels.discord.allowFrom` from the Discord `no_allowlists` audit check, and open PR #72467 explicitly targets this issue with closing syntax.

## What This Changes

The issue asks for Discord security audit to suppress `discord.commands.native.no_allowlists` when canonical `channels.discord.allowFrom` contains a valid user ID.

## Best Possible Solution

Land PR #72467 or an equivalent narrow change so the Discord audit uses the same effective account `allowFrom` source as runtime authorization, with regression coverage and changelog retained.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: native Discord commands enabled, a guild configured, no guild/channel users allowlist, no legacy/store allowlist, and only top-level `channels.discord.allowFrom` leaves `ownerAllowFromConfigured` false and emits the warning.

## Solution Assessment

Yes. The narrow maintainable fix is to include effective top-level/account `allowFrom` in `ownerAllowFromConfigured` and cover that case in `extensions/discord/src/security-audit.test.ts`; no new core API or config option appears necessary.
