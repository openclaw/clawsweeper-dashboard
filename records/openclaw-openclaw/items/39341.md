## Summary

Keep open because current main still has the reported Discord security-audit false positive, and the issue is already paired with open exact-scope PR #72467 using closing syntax. Closing this issue before that PR or an equivalent fix lands would drop a valid regression report.

## What This Changes

The issue asks for the Discord security audit to treat top-level `channels.discord.allowFrom` as an owner allowlist when deciding whether to emit the native slash-command `no_allowlists` warning.

## Best Possible Solution

Land PR #72467 or an equivalent narrow change so the Discord audit uses the same effective account allowFrom source as runtime authorization, with the top-level allowFrom regression test and changelog coverage kept in the fix.

## Reproduction Assessment

Yes. Static inspection is enough: a config with Discord native commands, a configured guild, no guild/channel users allowlist, and only top-level `channels.discord.allowFrom` still leaves `ownerAllowFromConfigured` false on current main.

## Solution Assessment

Yes. The narrow maintainable fix is to include top-level or effective account `allowFrom` in `ownerAllowFromConfigured` and cover that case in `extensions/discord/src/security-audit.test.ts`; no core API change appears necessary.
