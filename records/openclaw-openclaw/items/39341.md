## Summary

Keep open: current main and the latest release still have the reported Discord audit false positive, and an open exact-scope PR already references this issue with closing syntax, so this should close only after that fix or an equivalent corrected fix merges.

## What This Changes

The Discord security audit should treat top-level `channels.discord.allowFrom` as an owner allowlist for the native slash-command `no_allowlists` warning.

## Best Possible Solution

Land or replace the linked PR with a patch that computes the audit owner allowlist from `resolveDiscordAccountAllowFrom({ cfg, accountId })` plus the credential-store allowlist, with regression coverage for top-level and account-precedence behavior.

## Reproduction Assessment

Yes, at source level. With native Discord commands enabled, a configured guild, no guild/channel users allowlist, no credential-store or legacy allowlist, and only top-level `channels.discord.allowFrom`, current main leaves `ownerAllowFromConfigured` false and still emits the warning.

## Solution Assessment

Yes, the requested direction is the narrow maintainable fix. The implementation should use the existing effective Discord allowFrom resolver so audit behavior matches runtime account precedence instead of adding another direct config read.
