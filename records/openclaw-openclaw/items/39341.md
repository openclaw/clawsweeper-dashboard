## Summary

Keep open: current main and v2026.5.2 still omit top-level `channels.discord.allowFrom` from the Discord `no_allowlists` owner check, and open PR #72467 explicitly targets this issue with closing syntax.

## What This Changes

The issue requests that Discord security audit suppress the slash-command no-allowlists warning when canonical top-level `channels.discord.allowFrom` is configured.

## Best Possible Solution

Land PR #72467 or an equivalent narrow change so the Discord audit uses the effective account/root allowFrom source and includes regression coverage for the top-level path.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: native commands enabled, a guild configured, no guild/channel users allowlist, no credential-store or legacy allowlist, and only top-level `channels.discord.allowFrom` leaves `ownerAllowFromConfigured` false.

## Solution Assessment

Yes. The narrow maintainable fix is to include effective top-level/account `allowFrom` in `ownerAllowFromConfigured` and add a focused Discord security-audit regression test; no new config or product decision is needed.
