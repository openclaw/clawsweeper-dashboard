agents on a shared OpenClaw gateway and the lack of budget enforcement at the gateway level is the single biggest operational risk.\n\n**What we built as a stopgap** (and what the gateway should provide natively):\n\n**Three-tier quota hierarchy**: namespace → user → agent. Each level has:\n- `daily_limit_millicents`: Hard cap on daily spend\n- `monthly_limit_millicents`: Hard cap on monthly spend  \n- `per_request_ceiling`: Maximum cost for any single model call\n\n**Pessimistic allocation**: When an agent spawns a sub-task, the *estimated ceiling cost* is deducted from the budget immediately. If the actual cost is lower, the difference is refunded. This prevents the scenario where 10 concurrent tasks each estimate they can spend the full remaining budget.\n\n**Enforcement point**: The check happens in the gateway's model dispatch path, *before* the request reaches the provider. A budget-exceeded request gets a structured error (`BudgetExhaustedError` with `remaining_budget`, `requested_estimate`, `next_reset_at`) instead of silently hitting the provider and hoping for the best.\n\n**What the current `session-cost-usage.ts` is missing**:\n1. No enforcement — it tracks but doesn't block\n2. No hierarchy — it's per-session, not per-agent or per-user\n3. No pessimistic allocation — concurrent tasks can overshoot\n4. No structured error — the agent doesn't know *why* it was blocked or *when* it can retry\n\n**Cost attribution**: We also track per-task costs (not just per-agent) using proportional runtime allocation.
