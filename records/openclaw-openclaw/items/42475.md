## Summary

Current main still lacks per-agent spend enforcement at each model dispatch. The existing `before_agent_run` plugin gate can block only before a run, while model-call hooks are observational and expose completed usage after calls, so a plugin cannot safely enforce loops or reserve concurrent spend without a new generic admission contract; this remains a maintainer product/API decision rather than an autonomous fix.

## What This Changes

Requests optional per-agent daily and monthly model-spend budgets enforced before gateway model dispatch, with block, warn, or pause behavior and operator status visibility.

## Best Possible Solution

Add a narrow generic pre-model-call admission hook with agent/run/call/model identity, estimated maximum usage, atomic reserve/commit/release semantics, and structured block results; then implement configurable cost budgets as a plugin unless maintainers explicitly choose core ownership.

## Reproduction Assessment

Not applicable; this is a clearly specified request for new enforcement behavior, and source inspection confirms the current system provides cost observability but not that capability.

## Solution Assessment

No, not as proposed: reusing post-hoc session totals at a gateway check is insufficient for concurrent and multi-call runs; the narrowest maintainable path is a generic atomic per-call admission contract followed by plugin-owned budget policy unless core ownership is explicitly approved.

## Risks / Open Questions

- A run-level transcript-sum check can overshoot during concurrent calls or multiple model passes because actual usage arrives after dispatch.
- Unknown, stale, or provider-specific pricing requires an explicit fail-open versus fail-closed policy before hard blocking can be safe.
- Budgets across delegation, fallbacks, retries, and child agents need an ownership rule and atomic reservation model; otherwise costs can be misattributed or caps bypassed.
- Adding daily/monthly/action fields directly to core agent config would expand a compatibility-sensitive config surface before the generic extension boundary is settled.
