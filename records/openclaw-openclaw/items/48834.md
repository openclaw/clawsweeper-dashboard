## Summary

Keep open: the Feishu feature work is plausible and still has useful ACP block-delivery changes, but this PR is not merge-ready because it lacks real behavior proof and the patch has account-scoping and core/plugin-boundary problems against current main.

## What This Changes

The branch adds Feishu `/focus` and `/unfocus` support, Feishu thread-binding lifecycle/config hooks, ACP block delivery changes, session lifecycle handling, and session-binding/SDK updates.

## Best Possible Solution

Rebase onto current main's generic channel conversation-binding APIs, resolve Feishu sender scope from the active account config, keep core/plugin boundaries intact, and add redacted live Feishu proof.

## Reproduction Assessment

Yes, for the blocking review bug at source level: the PR helper reads only top-level `channels.feishu`, while live Feishu routing uses merged account config. I did not run a live Feishu tenant reproduction.

## Solution Assessment

No. The current PR direction should be updated to use current main's generic channel conversation-binding seams instead of adding Feishu-specific core imports and a separate account-unaware resolver.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Resolve Feishu sender scope from the active account:** `src/auto-reply/reply/feishu-context.ts:94-111`
  - body: `isFeishuSenderScopedTopic()` only reads the top-level Feishu config, but live Feishu routing resolves `groupSessionScope` from the merged account config. In a multi-account setup with an account-level or per-group override, `/focus`, `/unfocus`, `/session`, and reset can bind or mutate `chat:topic` while inbound routing uses `chat:topic:sender`, or the reverse.
  - confidence: 0.93
- **[P2] Keep Feishu lifecycle updates behind channel binding APIs:** `src/auto-reply/reply/commands-session.ts:1-4`
  - body: This imports `extensions/feishu/src/thread-bindings.js` directly from a core command file, which makes core depend on a bundled plugin implementation. Route lifecycle updates through the channel conversation-binding/plugin API instead so Feishu remains plugin-owned and other channels do not need core-specific imports.
  - confidence: 0.88
