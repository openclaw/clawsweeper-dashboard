## Summary

Keep open: current main still has the reported legacy Codex mirrored-history budget gap, while the earlier budget-aware projection work only covers the active context-engine branch.

## What This Changes

The issue reports that Codex legacy mirrored-history rebuilds omit the budget-derived projection cap, causing high-context Codex sessions to fall back to a 24,000-character rendered history cap.

## Best Possible Solution

Make the legacy mirrored-history projection use the same budget-derived maxRenderedContextChars calculation as the active context-engine path, with focused regression coverage for that fallback branch.

## Reproduction Assessment

Yes at source level: configure no active context engine with prior user history and a Codex start/rebuild, and the legacy branch reaches projectContextEngineAssemblyForCodex without maxRenderedContextChars. I did not execute a live high-window Codex session in this read-only review.

## Solution Assessment

Yes: passing the existing budget-derived cap into the legacy fallback is the narrowest maintainable fix. It should not try to solve the separate native-thread churn investigation in https://github.com/openclaw/openclaw/issues/84086.
