## Summary

Keep open: current main and the latest release still use the unbounded image success-body read, but this PR is not merge-ready because the generic 16 MiB JSON cap can reject documented large base64 image responses.

## What This Changes

## Best Possible Solution

Use the shared bounded reader with an image-sized response cap or an explicit documented 16 MiB product limit, plus provider-path valid-large and overflow coverage and mode-specific labels.

## Solution Assessment

No as written: the shared bounded reader is the right layer, but the default 16 MiB cap and mode-insensitive label are not the safest final shape for image JSON responses.
