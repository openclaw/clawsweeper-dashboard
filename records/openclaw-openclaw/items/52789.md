## Summary

Keep open. Current main still does not implement cron `delivery.mode: "session"`, and the remaining work is a new public cron delivery/API mode with authorization, provenance, diagnostics, UI, and docs decisions rather than a conservative cleanup close.

## What This Changes

Add a cron `delivery.mode: "session"` option that deterministically routes cron output to another session through internal agent-to-agent delivery instead of relying on model-issued `sessions_send` tool calls.

## Best Possible Solution

Keep this open for maintainer product/API review; if approved, add an additive cron session-delivery contract that reuses `sessions_send` authorization and provenance semantics and updates cron runtime, gateway schema, UI, docs, and tests together.

## Reproduction Assessment

Not applicable as a feature request. Static current-main checks against `origin/main` show cron types, planner, gateway schema, Control UI, and docs still expose only `none`, `announce`, and `webhook`, so the requested capability is absent with high confidence.

## Solution Assessment

Unclear until maintainers decide the public API and policy semantics. A framework-owned mode is plausible for reliability, but the maintainable path should reuse or mirror `sessions_send` visibility, A2A policy, provenance, thread restrictions, and diagnostics instead of adding a loose cron-only send branch.
