## Label Justifications

- `P2`: This is a normal-priority additive cron delivery feature with clear workflow value but limited blast radius and explicit product/security review needs.
- `impact:message-loss`: The motivating failure is unreliable internal cron result delivery when model-owned `sessions_send` calls omit required parameters.
- `impact:session-state`: The requested destination is another OpenClaw session, so the design affects inter-session context and transcript state.
- `impact:security`: Framework-owned cross-session delivery must preserve existing `sessions_send` authorization, visibility, and provenance boundaries.

## Summary

Keep open. Current main and the latest release still expose only `none`, `announce`, and `webhook` cron delivery modes, while the related openclaw-agents workaround confirms the workflow need rather than replacing native support.

## What This Changes

Requested addition of a cron `delivery.mode: "session"` that routes completed cron output internally through `sessions_send` to another session key.

## Best Possible Solution

If approved, add an additive `session` cron delivery contract that routes through the existing `sessions_send` authorization and provenance path, with cron storage, parser, gateway schema, CLI/UI, docs, and regression tests updated together.

## Reproduction Assessment

Not applicable as a feature request. Source inspection of current main and `v2026.5.22` shows the cron delivery contract still only supports `none`, `announce`, and `webhook`, so the requested mode is absent with high confidence.

## Solution Assessment

Unclear until maintainers decide the API and policy semantics. Framework-owned delivery is plausible for reliability, but the maintainable path should reuse `sessions_send` rather than add a separate cron-only cross-session send branch.
