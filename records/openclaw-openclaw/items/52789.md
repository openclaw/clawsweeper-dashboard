## Summary

Keep open. Current main still has no cron `delivery.mode: "session"`; the cron type, protocol schema, normalizer, planner, docs, and UI-facing form state remain limited to `announce`, `webhook`, and `none`, while `sessions_send` remains an agent tool path with its own visibility and A2A policy checks.

## What This Changes

Requests a new cron delivery mode that framework-delivers completed cron job output internally to a target session using `sessions_send`-equivalent behavior instead of visible chat delivery, webhooks, or model-issued tool calls.

## Best Possible Solution

Keep this issue open for maintainer product/API review. If approved, add a first-party cron `session` delivery mode across the CronDelivery type, Gateway protocol schema, normalizer, planner, CLI/UI/docs, and focused tests, with a framework-owned dispatch path that sends the final cron summary to a target session while preserving `sessions_send`-equivalent provenance, visibility, A2A policy checks, and delivery diagnostics.

## Reproduction Assessment

Not applicable as a feature request, but current-main verification is high confidence: static source, docs, and tests show only `announce`, `webhook`, and `none` are accepted or documented for cron delivery. No runtime mutation or tests were run because this was a read-only review.

## Solution Assessment

Unclear pending maintainer decision. If OpenClaw wants this behavior, a framework-owned cron delivery mode is the maintainable direction, but it should be designed against the existing `sessions_send` visibility/provenance contract rather than added as a loose channel-delivery variant.

## Risks / Open Questions

- Target identity and authorization semantics are unresolved: `delivery.to` would need to preserve the same session visibility, sandbox, and agent-to-agent policy guarantees that `sessions_send` currently enforces inside the tool path.
- The dispatch model needs maintainer design: a cron-owned mode could enqueue an internal event, start a target session turn, or reuse the A2A flow, and those choices affect visibility, retries, provenance, and failure diagnostics.
- This is a public cron API/protocol/UI/docs change, so a partial implementation could create config shapes that older clients or validators reject.
