## Label Justifications

- `P2`: This is a normal-priority additive cron delivery feature with clear workflow value but bounded blast radius and explicit product/security review needs.
- `impact:session-state`: The requested destination is another OpenClaw session, so the design affects inter-session context and transcript state.

- `impact:message-loss`: The motivating failure is unreliable internal cron result delivery when model-owned `sessions_send` calls omit required parameters.

## Summary

Keep open. Current main and the latest release still expose cron delivery as `none`, `announce`, or `webhook`; the requested framework-owned `session` delivery mode is not implemented and needs explicit product/security approval before a fix PR should be queued.

## What This Changes

The issue requests a new cron `delivery.mode: "session"` that delivers completed job output internally through `sessions_send` instead of visible channel delivery or model-authored tool calls.

## Best Possible Solution

After explicit product/security approval, add an additive `session` cron delivery contract that reuses the existing `sessions_send` authorization and provenance path, or record a maintainer decision that the broader output-routing issue should own the session-target API.

## Reproduction Assessment

Yes for the current limitation by source inspection: current main's cron type, parser, planner, Gateway schema, plugin scheduled-turn API, and docs all expose only `none`, `announce`, and `webhook`. A runtime repro is not needed to show that `delivery.mode: "session"` is not a supported contract today.

## Solution Assessment

Unclear until maintainers choose the API and security semantics. The narrow maintainable direction is to make cron delivery reuse the existing `sessions_send` policy/provenance path rather than adding a second cross-session send implementation.
