## Label Justifications

## Summary

Keep open: current main still exposes cron completion delivery as `none`, `announce`, or `webhook` only, and the requested internal session delivery contract remains a product and security decision rather than an already-implemented cleanup.

## What This Changes

The issue requests a new cron `delivery.mode: "session"` that delivers completed cron output internally to another session via `sessions_send` instead of a visible channel or model-authored tool call.

## Best Possible Solution

Design a maintainer-approved cron-to-session output contract that composes with A2A dispatch, defines authorization and provenance, then update schema, runtime delivery, docs, UI/protocol surfaces, and focused tests together.

## Reproduction Assessment

Yes. Current main source confirms the requested public mode is absent from cron types, parser, protocol schema, runtime planning, UI/docs, and tests; a live run is not needed to reproduce the missing contract.

## Solution Assessment

Unclear. Framework-owned session delivery is plausible, but maintainers still need to choose the permanent cron output API shape, A2A relationship, authorization, and transcript semantics before implementation.
