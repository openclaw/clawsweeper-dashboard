## Summary

Keep open. Current main and the latest release still only support cron delivery modes `none`, `announce`, and `webhook`; the requested framework-owned session delivery mode remains missing and needs maintainer product/security review before implementation.

## What This Changes

The issue requests a new cron `delivery.mode: "session"` that delivers cron output internally through `sessions_send` instead of visible channel delivery or model-issued tool calls.

## Best Possible Solution

Define a maintainer-approved cron session-delivery contract that reuses existing session visibility and agent-to-agent authorization rules, then implement the schema, runner, docs, and regression tests together.

## Reproduction Assessment

Yes, source-reproducible: current main and v2026.6.1 only accept `none`, `announce`, and `webhook` cron delivery modes. No runtime test was run because the missing mode is clear from the parser, type contract, docs, and delivery-plan source.

## Solution Assessment

Unclear as proposed. A `session` mode is a plausible solution to the model-tool-call reliability gap, but the best fix depends on maintainer decisions about cron-owned cross-session authorization, provenance, and whether this belongs in core delivery or another existing session/task completion path.
