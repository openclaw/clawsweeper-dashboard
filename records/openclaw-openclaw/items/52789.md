## Summary

Keep open. Current main still limits cron delivery to `none`, `announce`, and `webhook`; adding `session` would be a new public cron delivery mode that crosses protocol, UI, docs, and `sessions_send` authorization/provenance semantics rather than a cleanup-close item.

## What This Changes

Request adding cron `delivery.mode: "session"` so completed cron output can be delivered internally to a target session through `sessions_send`-style agent-to-agent delivery.

## Best Possible Solution

Keep this open for maintainer product/API design; if approved, implement a first-party cron session delivery mode across cron schema, planner, dispatch, Control UI, docs, tests, and session policy/provenance handling.

## Reproduction Assessment

Not applicable as a feature request. Static current-main verification is high-confidence: the source types, gateway schema, planner, Control UI, tests, and docs still expose only `none`, `announce`, and `webhook`.

## Solution Assessment

Unclear until maintainer design. A framework-owned mode is plausible for reliability, but the safest implementation should preserve `sessions_send` visibility, thread-target rejection, inter-session provenance, A2A behavior, and diagnostics rather than adding a loose dispatch branch.
