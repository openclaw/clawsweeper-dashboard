- `impact:message-loss`: The motivating problem is silent or unreliable internal cron result delivery when models miss `sessions_send` parameters.

## Summary

Keep open. Current main still limits cron delivery to `none`, `announce`, and `webhook`, and the requested internal session delivery mode would add a new cron/API behavior that needs maintainer product and security-boundary review.

## What This Changes

The issue requests a new cron `delivery.mode: "session"` that sends completed cron output internally to another session through `sessions_send`.

## Best Possible Solution

If maintainers approve the product behavior, add an additive cron session-delivery contract that reuses `sessions_send` authorization, visibility, provenance, diagnostics, timeout, and target-validation semantics across runtime, protocol, UI, docs, and tests.

## Reproduction Assessment

Not applicable as a feature request. Static current-main checks show cron types, parser, planner, gateway schema, Control UI, tool description, and docs still expose only `none`, `announce`, and `webhook`, so the requested capability is absent with high confidence.

## Solution Assessment

Unclear until maintainers decide the API and policy semantics. A framework-owned mode is plausible for reliability, but the maintainable path should reuse `sessions_send` behavior rather than adding a loose cron-only send branch.
