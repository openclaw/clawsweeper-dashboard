## Summary

Keep open. Current main still does not implement cron `delivery.mode: "session"`; the request is a new public cron delivery mode with API, UI, docs, authorization, provenance, and failure-semantics decisions, not an already-fixed cleanup item.

## What This Changes

Request to add a public cron `session` delivery mode that sends completed job output to a target session through framework-owned `sessions_send`-style routing.

## Best Possible Solution

Keep this open for maintainer product/API review; if approved, implement the mode across cron types, normalization/planning, dispatch, gateway protocol, Control UI, docs, and regression tests while preserving `sessions_send` authorization and provenance semantics.

## Reproduction Assessment

Not applicable as a feature request. Static current-main checks show the cron contract, planner, UI, tests, and docs still accept only `none`, `announce`, and `webhook`, so the requested mode is absent with high confidence.

## Solution Assessment

Unclear until maintainers decide the public API and policy semantics. A framework-owned mode is plausible for reliability, but the maintainable path should reuse or mirror `sessions_send` visibility, A2A policy, provenance, thread restrictions, and diagnostics.
