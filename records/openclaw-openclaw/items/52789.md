- `impact:message-loss`: The motivating workflow is silent or unreliable internal cron result delivery when models miss `sessions_send` tool parameters.
- `impact:session-state`: The requested destination is another OpenClaw session, so the design affects inter-session context, transcript provenance, and session routing.

## Summary

Keep open. Current main still exposes cron delivery as `none`, `announce`, and `webhook` only; the underlying `sessions_send` capability exists, but wiring it into cron as a framework-owned delivery mode is a new cron/API/policy feature rather than cleanup closure.

## What This Changes

The issue requests a new cron `delivery.mode: "session"` that sends completed job output internally to a target OpenClaw session via `sessions_send`.

## Best Possible Solution

If maintainers approve the product behavior, add an additive cron session-delivery contract that reuses `sessions_send` authorization, visibility, provenance, timeout, diagnostics, and thread-target rules across runtime, protocol, UI, docs, and tests.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source inspection on current main shows the requested mode is absent from cron types, parser, planner, gateway schema, Control UI, and docs with high confidence.

## Solution Assessment

Unclear until maintainers decide the public API and policy semantics. A framework-owned delivery mode is plausible for reliability, but the maintainable path should reuse `sessions_send` behavior rather than adding a cron-only send branch.
