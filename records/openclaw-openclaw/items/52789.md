## Label Justifications

- `P2`: This is a normal-priority multi-agent cron reliability feature with real user impact, but it is bounded and needs product/security direction before implementation.
- `impact:message-loss`: The report says cron results can be silently dropped when weaker models fail to call `sessions_send` correctly.
- `impact:session-state`: The requested mode would write cron output into another agent session and affect transcript/context routing semantics.
- `impact:security`: Invisible framework-owned cross-session delivery requires explicit authorization, visibility, and provenance decisions.

## Summary

Keep open: current main and the latest release still expose cron completion delivery only as `none`, `announce`, or `webhook`, and related open issues do not fully own this cron-specific session-delivery contract plus its security and A2A semantics.

## What This Changes

The issue requests a new cron `delivery.mode="session"` that framework-delivers completed cron output internally to a target session via `sessions_send`.

## Best Possible Solution

Design a maintainer-approved cron-to-session output contract that composes with one-way A2A dispatch, defines authorization and provenance, then update schema, runtime delivery, docs, and focused tests together.

## Reproduction Assessment

Yes. Source inspection shows current main and `v2026.6.10` exclude `session` from cron delivery types, protocol schema, model-facing tool enums, runtime planning, and docs; no live run is needed to confirm the public mode is absent.

## Solution Assessment

Unclear. Framework-owned session delivery is plausible, but maintainers still need to choose the durable cron output API shape, A2A relationship, and security model before implementation.
