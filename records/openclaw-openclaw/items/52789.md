## Summary

Keep open. Current main still lacks cron `delivery.mode: "session"`; the existing workaround confirms the workflow is useful, but it does not provide the requested deterministic framework-owned delivery mode.

## What This Changes

The issue requests a new cron `delivery.mode: "session"` that sends job summaries internally through `sessions_send` instead of visible channel delivery or model-authored tool calls.

## Best Possible Solution

If approved, add an additive cron session-delivery contract that reuses the existing `sessions_send` authorization, visibility, provenance, diagnostics, timeout, and thread-target rules across runtime, protocol, UI, docs, and tests.

## Reproduction Assessment

Not applicable as a feature request. Static current-main checks show cron types, parser, planner, gateway schema, Control UI, and docs still expose only `none`, `announce`, and `webhook`, so the requested capability is absent with high confidence.

## Solution Assessment

Unclear until maintainers decide the public API and policy semantics. A framework-owned mode is plausible for reliability, but the maintainable path should reuse `sessions_send` behavior rather than adding a loose cron-only send branch.
