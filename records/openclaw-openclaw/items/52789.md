## Label Justifications

- `P2`: This is a normal-priority additive cron delivery feature with clear user value but limited blast radius and explicit product/security review needs.
- `impact:message-loss`: The motivating failure is unreliable internal cron result delivery when model-owned `sessions_send` calls omit required parameters.
- `impact:session-state`: The requested destination is another OpenClaw session, so the design affects inter-session context, transcript provenance, and session state.

## Summary

Keep open. Current main still lacks a cron `session` delivery mode, there is no closing PR, and the external openclaw-agents workaround confirms the workflow need while still treating native OpenClaw support as future simplification.

## What This Changes

The issue requests a new cron delivery mode that sends completed cron output into another OpenClaw session through the `sessions_send` path instead of visible channel delivery or model-authored tool calls.

## Best Possible Solution

If maintainers approve it, add an additive `session` cron delivery contract that routes through the existing `sessions_send` authorization/provenance path and updates cron storage, protocol schema, CLI/UI, docs, and regression tests together.

## Reproduction Assessment

Not applicable as a feature request. Source inspection shows the cron type, parser, planner, gateway schema, Control UI model, and docs still expose only `none`, `announce`, and `webhook`, so the requested capability is absent with high confidence.

## Solution Assessment

Unclear until maintainers decide the API and policy semantics. Framework-owned delivery is plausible for reliability, but the maintainable path should reuse `sessions_send` rather than add a loose cron-only cross-session send branch.
