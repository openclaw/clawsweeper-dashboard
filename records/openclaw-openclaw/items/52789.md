## Summary

Keep open. Current main still does not implement cron `delivery.mode: "session"`; the remaining work is a new public cron delivery mode with protocol/UI/docs and session-authorization semantics, not a stale or already-fixed issue.

## What This Changes

The issue requests a new cron delivery mode that sends completed job output internally to another session via `sessions_send` instead of visible chat or webhook delivery.

## Best Possible Solution

Keep this open for maintainer product/API review; if approved, implement a first-party cron session delivery mode across cron types, gateway schema, normalization, planner/dispatch, Control UI, docs, and tests with `sessions_send`-equivalent provenance, policy checks, and diagnostics.

## Reproduction Assessment

Not applicable as a feature request. Static current-main verification is high confidence: source, protocol schema, UI types, UI options, and docs all still limit cron delivery to announce/webhook/none.

## Solution Assessment

Unclear pending maintainer decision. A framework-owned mode is plausible for reliability, but it should be designed against the existing `sessions_send` visibility/provenance contract rather than added as a loose delivery branch.
