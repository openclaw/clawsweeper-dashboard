## Label Justifications

- `P2`: This is a normal-priority additive cron delivery feature with clear workflow value but bounded blast radius and explicit product/security review needs.
- `impact:message-loss`: The motivating failure is unreliable internal cron result delivery when model-owned `sessions_send` calls omit required parameters.
- `impact:session-state`: The requested destination is another OpenClaw session, so the design affects inter-session context and transcript state.
- `impact:security`: Framework-owned cross-session delivery must preserve existing `sessions_send` authorization, visibility, sandbox, and provenance boundaries.

## Summary

Keep open. Current main and the latest release still expose cron delivery as `none`, `announce`, or `webhook`; the requested framework-owned `session` mode is not implemented and needs explicit product/security approval before a fix PR should be queued.

## What This Changes

Add a new cron `delivery.mode: "session"` that routes completed cron output internally to a target session via `sessions_send` instead of using visible channel announce, webhook, or model-owned tool calls.

## Best Possible Solution

After explicit product/security approval, add an additive `session` cron delivery contract that delegates through the existing `sessions_send` policy/provenance path and updates schemas, CLI/Gateway surfaces, docs, and focused regression tests together.

## Reproduction Assessment

Not applicable as a feature request. Source inspection of current main and `v2026.5.22` gives high confidence that cron delivery still only supports `none`, `announce`, and `webhook`.

## Solution Assessment

Unclear until maintainers decide the API and security semantics. The narrow maintainable direction is to make cron delivery reuse `sessions_send` policy/provenance rather than adding a separate cross-session send path.
