## Summary

Keep open. Current main still lacks cron session delivery across source, protocol, UI, docs, and tool prompt surfaces, and this is a new public cron/session delivery contract that needs maintainer product/API design.

## What This Changes

Request add `delivery.mode: "session"` for cron jobs so finished cron output is delivered internally to a target session via `sessions_send`.

## Best Possible Solution

Keep this open for maintainer product/API review; if approved, implement a first-party cron session delivery mode across cron types, normalization/planning/dispatch, gateway protocol, Control UI, docs, and tests with `sessions_send`-equivalent auth, provenance, and diagnostics.

## Reproduction Assessment

Not applicable; this is a feature request. Static current-main checks are high confidence because the cron type, protocol schema, delivery planner, Control UI, docs, and prompt snapshot all still enumerate only announce/webhook/none.

## Solution Assessment

Unclear pending maintainer design. A framework-owned mode is plausible for reliability, but the maintainable fix must preserve `sessions_send` visibility, requester ownership, provenance, timeout, and failure semantics rather than adding a loose delivery branch.
