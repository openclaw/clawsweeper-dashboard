## Summary

Keep open. Current `main` still exposes cron delivery as `none`, `announce`, or `webhook` only, and adding framework-owned internal session delivery needs maintainer API and policy review rather than cleanup closure.

## What This Changes

The issue requests a new cron `delivery.mode: "session"` that delivers completed cron output through internal session delivery instead of visible chat announcement or model-invoked tools.

## Best Possible Solution

If maintainers approve the product behavior, add an additive cron session-delivery contract that reuses `sessions_send` authorization, visibility, provenance, diagnostics, timeout, and thread-target rules across runtime, protocol, UI, docs, and tests.

## Solution Assessment

Unclear until maintainers decide the public API and policy semantics. A framework-owned mode is plausible for reliability, but the maintainable path should reuse `sessions_send` behavior rather than adding a loose cron-only send branch.
