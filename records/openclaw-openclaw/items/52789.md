## Label Justifications

- `P2`: The request is a normal-priority feature for multi-agent cron workflows with bounded but real delivery/session impact.
- `impact:message-loss`: The issue reports internal cron results can be dropped when cheaper models fail to call `sessions_send` correctly.
- `impact:session-state`: The requested delivery path would write cron output into another agent session and affect transcript/session routing semantics.
- `impact:security`: Framework-owned invisible internal session delivery needs authorization and visibility decisions before implementation.

## Summary

Keep open: current main and v2026.6.6 still expose only `none`, `announce`, and `webhook` cron delivery modes, so `delivery.mode="session"` remains unimplemented and needs maintainer product/security direction.

## What This Changes

Add a cron `delivery.mode="session"` option that sends completed cron output to an internal target session through framework-owned delivery instead of model-authored tool calls.

## Best Possible Solution

Design a maintainer-approved cron-to-session delivery contract that coordinates with the broader A2A dispatch semantics, then update the cron types, gateway protocol schema, CLI/tool input, runtime dispatch, docs, and focused security/session tests together.

## Reproduction Assessment

Yes at source level: current main and v2026.6.6 accept only `none`, `announce`, and `webhook` cron delivery modes across types, schemas, CLI/tool surfaces, runtime planning, and docs. No live run was needed because `session` is excluded by the enum/schema path.

## Solution Assessment

Unclear: framework-owned session delivery is a plausible solution, but the exact API and behavior need maintainer choice on cron config shape, authorization, target validation, transcript provenance, and alignment with A2A dispatch.
