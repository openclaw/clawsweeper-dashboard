## Label Justifications

- `P2`: This is a normal-priority multi-agent cron reliability feature with real user impact but bounded scope and required product direction.
- `impact:message-loss`: The report says cron results can be silently dropped when models fail to call `sessions_send` correctly.
- `impact:session-state`: The requested mode would write cron output into another agent session and affect transcript/context routing semantics.
- `impact:security`: Invisible framework-owned cross-session delivery requires explicit authorization, visibility, and provenance decisions.

## Summary

Keep open: current `main` and `v2026.6.10` still expose cron completion delivery only as `none`, `announce`, or `webhook`. The broader cron output-routing issue and A2A dispatch issue overlap, but neither fully owns the cron-specific `delivery.mode="session"` contract plus authorization/provenance decisions.

## What This Changes

The issue requests a new cron delivery mode that framework-delivers completed cron output into another internal session via `sessions_send` instead of relying on the model to call the tool.

## Best Possible Solution

Design a maintainer-approved cron-to-session output contract that composes with A2A dispatch, defines authorization and provenance, then update schema, runtime delivery, docs, and focused tests together.

## Reproduction Assessment

Yes at source level: current `main` and `v2026.6.10` exclude `session` from cron delivery types, protocol schema, model-facing cron tool enums, runtime planning, and docs. A live run was not needed to confirm the public mode is absent.

## Solution Assessment

Unclear: framework-owned session delivery is a plausible solution, but maintainers still need to choose the permanent cron output API shape, A2A relationship, and security model before implementation.
