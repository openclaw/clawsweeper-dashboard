## Summary

Keep open: current main and v2026.6.6 still expose only `none`, `announce`, and `webhook` cron delivery modes, while the linked external workaround still relies on prompting agents to call `sessions_send`. The request is useful but needs maintainer product and security direction before implementation.

## What This Changes

Add a cron `delivery.mode="session"` path that reliably sends completed cron output into an internal target session via the existing session-send machinery.

## Best Possible Solution

Keep this open for maintainers to design one canonical internal delivery contract that lets cron reliably write final output to an allowed target session without visible channel noise or model-dependent tool parameters.

## Reproduction Assessment

Yes, for the missing capability. Current main and v2026.6.6 both restrict cron delivery modes to `none`, `announce`, and `webhook`, so `delivery.mode="session"` has no accepted source or schema path.

## Solution Assessment

Unclear as an implementation choice. A framework-owned session delivery mode is plausible and better than prompt-only tool calls, but maintainers still need to choose the public API, authorization model, provenance, and relationship to one-way A2A dispatch.
