## Label Justifications

- `P2`: This is a normal-priority feature for multi-agent cron workflows with real but bounded delivery and session-routing impact.
- `impact:message-loss`: The report says internal cron results can be silently dropped when cheaper models fail to call `sessions_send` correctly.
- `impact:session-state`: The requested mode would write cron output into another agent session and affect transcript/context routing semantics.
- `impact:security`: Invisible framework-owned cross-session delivery needs explicit authorization, visibility, and provenance decisions before implementation.

## Summary

Keep open: current main and latest release still expose cron delivery only as `none`, `announce`, or `webhook`, and the requested internal session delivery mode remains a new public cron/session contract that needs maintainer product and security direction.

## What This Changes

The issue requests a new cron `delivery.mode: "session"` that deterministically sends cron output to an internal session instead of relying on model-issued `sessions_send` calls.

## Best Possible Solution

Design a maintainer-approved cron-to-session delivery contract aligned with A2A dispatch semantics, then update cron types, protocol schema, agent tool/CLI input, runtime delivery, docs, and focused authorization/session tests together.

## Reproduction Assessment

Yes, source-level: current main accepts only `none`, `announce`, and `webhook` across the cron type, protocol schema, agent tool enum, runtime planner, and docs. A live run was not needed because `session` is excluded by the public schema and enum path.

## Solution Assessment

Unclear: framework-owned delivery is a plausible fix, but maintainers need to choose the API shape, authorization model, transcript provenance, and relationship to the broader A2A dispatch contract before implementation.
