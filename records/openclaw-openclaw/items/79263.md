## Summary

Keep open: current main and the latest release still load process env, CWD `.env`, state-dir `~/.openclaw/.env`, and legacy `~/.config/openclaw/gateway.env`, but not the reporter's home-level service dotenv; deciding whether to widen credential loading, add a migration, or document the unsupported path needs maintainer security/product judgment.

## What This Changes

The issue asks whether bare local CLI inference should keep loading a home-level service dotenv for provider API keys, or document/diagnose the post-upgrade local-transport auth change.

## Best Possible Solution

Keep the documented state-dir dotenv and gateway env paths as the default, then have maintainers choose between targeted diagnostics/docs for home-level service-dotenv users or a narrow, explicitly reviewed compatibility migration.

## Reproduction Assessment

Yes for the source path: a key present only in `/home/openclaw/.env` is not loaded by current CLI dotenv code, while provider env fallback works once the key is in `process.env`, `$OPENCLAW_STATE_DIR/.env`, or the legacy gateway env path. I did not run a live packaged `opencode-go` request with real credentials.

## Solution Assessment

Unclear. Restoring home-level dotenv loading may match the reporter's old smoke-test setup, but docs/diagnostics or doctor-guided migration to `~/.openclaw/.env` may be the safer maintainable fix.
