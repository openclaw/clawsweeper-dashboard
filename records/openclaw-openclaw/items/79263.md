## Summary

Keep open: current main supports process-env and state-dir dotenv auth for `opencode-go`, but it still does not load the reporter's `~/ .env` style service file for bare local CLI smoke tests, so the remaining behavior needs an explicit maintainer decision rather than cleanup closure.

## What This Changes

The issue asks whether local `openclaw infer model run` should restore `~/ .env`/shell-env auth fallback for bare CLI smoke tests, or document/direct users to the supported gateway or state-dir dotenv path.

## Best Possible Solution

Maintain the documented state-dir dotenv and gateway paths, and either add a targeted diagnostic/doc note for `~/ .env` service-file users or explicitly approve a narrow compatibility migration if maintainers want that file to be supported.

## Reproduction Assessment

Yes for the narrow current-main source path: local `model run` loads CWD `.env` plus `~/.openclaw/.env`/state-dir dotenv, then falls back to process env for provider auth, so a key present only in `~/ .env` will not be found. I did not run a live packaged `opencode-go` request with real credentials.

## Solution Assessment

Unclear. Restoring `~/ .env` loading may match the reporter's old smoke-test setup, but the safer maintainable path may be a clearer error/docs or a doctor-guided migration to `~/.openclaw/.env` rather than widening ambient credential loading.
