## Summary

Keep open: current main still resolves provider keys from process-visible env and documented dotenv sources, but bare local CLI probes do not read `/home/openclaw/.env` as a service-style env file. Maintainers need to decide whether this should be documented/error-guided behavior or a deliberate expansion of the local CLI env-source boundary.

## What This Changes

The issue requests clarification or restoration of bare local CLI model auth when an API key lives only in a home/service `.env` file after v2026.4.29.

## Best Possible Solution

Clarify supported local CLI env sources and missing-auth guidance, and only add service env-file parity if maintainers explicitly approve that trust boundary.

## Reproduction Assessment

No live high-confidence reproduction was run. Source inspection gives a deterministic current-main path: a key present only in `/home/openclaw/.env` is not visible to local model auth unless that file is also CWD `.env`, state-dir `.env`, process env, config `env`, shell import, or an auth profile.

## Solution Assessment

Unclear: restoring implicit `/home/openclaw/.env` reads would expand the bare CLI trust boundary. A narrower maintainable path is docs and missing-auth message clarity unless maintainers decide local CLI should mirror the service environment file.
