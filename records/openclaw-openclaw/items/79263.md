## Summary

Keep open: current main and v2026.6.8 still support provider env auth when the key reaches the CLI process or documented trusted dotenv locations, but they do not load the reporter's home-level service dotenv; changing that credential-source boundary needs maintainer security/product judgment.

## What This Changes

The issue asks OpenClaw to clarify, document, or restore local CLI provider credential loading for bare `openclaw infer model run` when the API key exists only in a home-level service `.env` after upgrades.

## Best Possible Solution

Keep the documented trusted state-dir dotenv behavior as the default, then either improve diagnostics/docs or add a narrow security-reviewed compatibility migration if maintainers explicitly sponsor home-level service-dotenv support.

## Reproduction Assessment

Yes for the source-level path: a key present only in a home-level service dotenv is not loaded by current CLI dotenv code, while provider env fallback works once the key is in `process.env`, `$OPENCLAW_STATE_DIR/.env`, or the legacy gateway env path. I did not run a live packaged `opencode-go` request with real credentials.

## Solution Assessment

Unclear. Restoring home-level dotenv loading may match the reporter's old smoke-test setup, but docs/diagnostics or doctor-guided migration to `~/.openclaw/.env` may be the safer maintainable fix.
