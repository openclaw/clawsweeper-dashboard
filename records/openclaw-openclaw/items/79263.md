## Summary

Keep open: current main and v2026.6.9 still do not load a home-level `~/.env` for bare local CLI inference, while docs point users to trusted sources such as process env, `~/.openclaw/.env`, config `env`, or shell import. The reporter confirms the workaround is enough, but closing this as working-as-intended still needs a maintainer security/product decision.

## What This Changes

The issue asks whether local CLI inference should load a home-level service dotenv such as `~/.env`, restore the earlier smoke-test behavior, or document/direct users to the supported credential source.

## Best Possible Solution

Keep the documented state-dir dotenv behavior as the default, then have maintainers decide whether to improve diagnostics/docs or sponsor a security-reviewed compatibility migration for home-level service dotenv setups.

## Reproduction Assessment

Yes for the source-level path: current main and v2026.6.9 only load CWD `.env`, state-dir `.env`, and the scoped gateway compatibility file, not `~/.env`. I did not run a live `opencode-go` request with real credentials.

## Solution Assessment

Unclear: restoring home-level dotenv loading would match the reporter's smoke-test setup, but docs, diagnostics, or doctor-guided migration to `~/.openclaw/.env` may be the safer maintainable solution.
