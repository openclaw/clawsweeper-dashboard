## Summary

Keep open: current main supports process-visible `OPENCODE_API_KEY` and the documented dotenv sources, but it still does not treat a home/service file such as `/home/openclaw/.env` as an automatic bare local CLI auth source. The remaining decision is whether to clarify the boundary and auth error or intentionally add service env-file parity.

## What This Changes

Request to restore or document bare local CLI model auth behavior when provider API keys live only in a service/home `.env` file rather than process env or auth profiles.

## Best Possible Solution

Clarify supported local CLI env sources and improve the missing-auth guidance, adding explicit service env-file parity only if maintainers accept that trust boundary.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: if `OPENCODE_API_KEY` exists only in `/home/openclaw/.env` and that file is not CWD `.env`, state-dir `.env`, `~/.config/openclaw/gateway.env`, shell-imported, or in an auth profile, local model auth will not see it. I did not run a live provider request.

## Solution Assessment

Unclear: implicitly loading a home/service `.env` file from bare CLI runs would expand trusted auth sources and needs maintainer policy approval. The narrow maintainable path is docs/error-message clarity unless maintainers choose service env-file parity.
