## Summary

Keep open: current main still has local CLI env auth fallback, but only for env sources the CLI actually loads. The reported service env-file setup remains outside that boundary, so maintainers need to decide whether to document/improve the missing-auth guidance or intentionally add service-env parity.

## What This Changes

Request clarification or restoration of bare local `openclaw infer model run` API-key resolution when the key lives in a home-level service env file rather than process env, state-dir dotenv, config env, shell import, or auth profiles.

## Best Possible Solution

Maintainers should decide the env trust boundary, then either document the supported local CLI sources with a more directive missing-auth message or add a narrowly scoped, safe service-env import path.

## Reproduction Assessment

Yes, by source inspection: a local infer run with the key only in an unsupported service env file and no auth profile/config/process env will reach the missing-key branch. I did not run the live provider call, but the code path and reported error shape line up.

## Solution Assessment

Unclear: restoring implicit service-env reads may not be the safest default because current docs define narrower sources. The maintainable next step is a maintainer decision between docs/error-message clarity and a deliberate env-loading expansion.
