## Summary

Keep open: current main still supports process-visible provider env fallback and documented dotenv paths, but it does not read a bare home/service env file like `/home/openclaw/.env` unless that file is also the current working directory `.env`. The remaining choice is maintainer policy: document/improve the auth error or intentionally add service env-file parity for bare local CLI probes.

## What This Changes

The issue asks whether local CLI inference should restore or document auth lookup from a home/service env file after upgrades changed smoke-test behavior.

## Best Possible Solution

Clarify the supported local CLI env sources and missing-auth guidance, and only add bare home/service env-file parity if maintainers explicitly want that trust boundary.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection gives a clear failure path when the key exists only in `/home/openclaw/.env` and is not visible through process env, CWD `.env`, `~/.openclaw/.env`, `$OPENCLAW_STATE_DIR/.env`, `~/.config/openclaw/gateway.env`, config `env`, shell import, or an auth profile.

## Solution Assessment

Unclear: restoring implicit reads from a bare home/service env file is not obviously the safest default because current docs and tests define narrower trusted sources. A maintainable first step is docs/error-message clarity unless maintainers decide bare CLI should mirror the service environment file.
