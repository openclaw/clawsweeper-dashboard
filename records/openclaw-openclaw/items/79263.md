## Summary

Keep open: current main and the latest release still support provider env auth when the key reaches the process or documented dotenv locations, but they do not load the reporter's home-level service dotenv; changing that trust boundary needs maintainer security/product judgment.

## What This Changes

The issue asks whether local `infer model run` should keep finding provider API keys from a home-level service dotenv after v2026.4.29, or at least document/direct the new behavior.

## Best Possible Solution

Keep the documented state-dir dotenv and gateway env paths as the default until maintainers choose either clearer diagnostics/docs or a narrow, security-reviewed compatibility migration for home-level service-dotenv users.

## Reproduction Assessment

Yes for the source path: a key present only in `/home/openclaw/.env` is not loaded by current CLI dotenv code, while provider env fallback works once the key is in `process.env`, `$OPENCLAW_STATE_DIR/.env`, or the legacy gateway env path. I did not run a live packaged `opencode-go` request with real credentials.

## Solution Assessment

Unclear. Restoring home-level dotenv loading may match the reporter's old smoke-test setup, but docs/diagnostics or doctor-guided migration to `~/.openclaw/.env` may be the safer maintainable fix.
