## Summary

Keep open: current main still supports env-backed local model auth, but only from documented process, current-directory, state-dir, config, compatibility gateway env, or opt-in shell import sources. The reported operator service env file remains outside that boundary, and changing it needs maintainer judgment rather than cleanup closure.

## What This Changes

The issue asks OpenClaw to restore or explicitly document bare local CLI auth behavior for operator home dotenv files after the v2026.4.29 upgrade changed smoke-test behavior.

## Best Possible Solution

Clarify the supported local CLI env sources and missing-auth guidance, then add service env-file parity only if maintainers explicitly approve that trust boundary.

## Reproduction Assessment

Yes at source level, but not by live provider execution. A key present only in an operator service env file outside process env, current-directory `.env`, state-dir `.env`, config `env`, opt-in shell import, or auth profiles will not reach `resolveEnvApiKey`, producing the reported missing-auth path.

## Solution Assessment

Unclear. The current implementation is internally consistent with the docs, but restoring broader implicit env-file reads is a trust-boundary/product decision; the safer first step is clearer docs or a more directive missing-auth message.
