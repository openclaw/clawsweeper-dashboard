## Summary

Keep open: current main still supports process-visible provider env fallback, but it does not treat a separate service EnvironmentFile-style env file as a bare local CLI source. The remaining question is a maintainer policy choice between docs/error-message clarification and intentionally broadening local CLI env loading.

## What This Changes

The issue asks whether local `openclaw infer model run` probes should read the same service env file as the Gateway, or at least document and clarify the missing-auth behavior after v2026.4.29.

## Best Possible Solution

Clarify the supported local CLI env sources and improve the missing-auth hint; add service-env-file parity only if maintainers explicitly want that trust boundary for bare CLI probes.

## Reproduction Assessment

Yes at source level: a key present only in a service EnvironmentFile-style env file outside the documented CWD/state/config/shell-import sources will not be visible to local `infer model run`. I did not establish live provider-side proof in this checkout.

## Solution Assessment

Unclear: restoring implicit service-env-file reads is not obviously the narrowest or safest fix. A docs/error-message clarification is the maintainable first step unless maintainers decide local CLI should intentionally mirror Gateway service env loading.
