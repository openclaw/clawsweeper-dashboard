## Summary

Keep open: current main still has a provider env fallback for local model auth, but the reported setup uses a service/home env file path that current docs and code do not treat as a bare CLI env source. This needs maintainer judgment on whether to document the boundary, improve the missing-auth message, or intentionally add service env-file parity.

## What This Changes

The issue asks OpenClaw to restore or clearly document bare local CLI inference auth behavior when an OpenCode key exists only in a service/home env file after upgrading.

## Best Possible Solution

Clarify the supported local CLI env sources and missing-auth guidance, and only add service env-file parity if maintainers explicitly want that trust boundary for bare CLI probes.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection supports the reported failure when the key exists only in `/home/openclaw/.env` and is not visible through process env, CWD `.env`, `~/.openclaw/.env`, config `env`, optional shell import, or an auth profile.

## Solution Assessment

Unclear: restoring implicit reads from a home/service env file is not obviously the safest fix because current docs define narrower env sources. A maintainable first step is docs/error-message clarity unless maintainers decide bare CLI should mirror the gateway service environment.
