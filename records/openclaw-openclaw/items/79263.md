## Summary

Keep open: current main still supports env-backed model auth, but only from process-visible env and documented dotenv/config sources. The reported `/home/openclaw/.env` service-file parity remains a maintainer policy/security-boundary decision, not a conservative cleanup close.

## What This Changes

The issue asks OpenClaw to clarify or restore bare local CLI auth env loading for `openclaw infer model run` after upgrades stopped picking up a home/service `.env` file.

## Best Possible Solution

Clarify supported local CLI env sources and missing-auth guidance, and only add service env-file parity if maintainers explicitly accept that trust boundary.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection supports the failure when the key exists only in `/home/openclaw/.env` and is not present in process env, CWD `.env`, state-dir `.env`, config `env`, optional shell import, or an auth profile.

## Solution Assessment

Unclear: implicitly reading a home/service env file is not obviously the safest fix. A docs/error-message clarification is the narrow path unless maintainers decide local CLI should mirror service EnvironmentFile loading.
