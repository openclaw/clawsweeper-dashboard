## Label Justifications

- `P2`: This is a concrete plugin installation compatibility bug with a confusing WhatsApp recovery failure, but the blast radius is limited to external/floating plugin installs.
- `impact:auth-provider`: The reported failure happens during WhatsApp relink recovery and turns an auth repair step into a plugin-load API mismatch.

## Summary

Keep open: current main and latest release still only enforce `openclaw.install.minHostVersion` for direct npm/package installs, while the WhatsApp package advertises a newer plugin API/OpenClaw peer requirement than that install floor.

## What This Changes

The issue asks plugin install to reject or choose compatible WhatsApp plugin package versions when the package requires a newer OpenClaw/plugin SDK than the running core.

## Best Possible Solution

Add a generic direct npm/package install compatibility check for OpenClaw/plugin API peer requirements before persisting the plugin, and add regression coverage for the WhatsApp version-skew case.

## Reproduction Assessment

Yes, source-reproducible: a host such as `2026.5.10-beta.1` satisfies WhatsApp's `minHostVersion >=2026.4.25` while the same package declares `openclaw`/plugin API `>=2026.5.24`. The direct npm path reaches only the min-host gate before install; I did not run a live install because this review is read-only.

## Solution Assessment

No, the current implementation is not the best solution because the direct npm path lacks the compatibility gate already present for ClawHub. The maintainable fix is generic package-install compatibility validation, not a WhatsApp-only recovery special case.
