## Label Justifications

- `P2`: This is a concrete plugin installation compatibility bug with limited blast radius to external/floating plugin installs, but it can make WhatsApp auth recovery fail with a confusing plugin-load error.
- `impact:auth-provider`: The reported failure happens during WhatsApp relink/auth recovery and turns a channel auth problem into an incompatible plugin load failure.

## Summary

Keep open: current main and the latest shipped release still let the direct npm install path validate only `openclaw.install.minHostVersion`, while the WhatsApp package advertises a newer OpenClaw/plugin API requirement. The report is source-reproducible, and the ClawHub path already shows the intended fail-closed compatibility behavior.

## What This Changes

Request that floating WhatsApp plugin installs choose a core-compatible package, refuse with a clear compatibility error, or avoid overriding the bundled plugin with an incompatible external package.

## Best Possible Solution

Keep this issue open and land a narrow generic installer compatibility guard for direct npm/package installs, with WhatsApp version-skew regression coverage and no WhatsApp auth behavior changes.

## Reproduction Assessment

Yes, source-level: a 2026.5.10-beta.1 host satisfies WhatsApp's `minHostVersion` floor but not the package's `openclaw` peer or `openclaw.compat.pluginApi` requirement. The direct npm install validation path only enforces `minHostVersion`, so the mismatch remains source-reproducible without a live install.

## Solution Assessment

Yes: a generic direct package-install compatibility gate is the narrow maintainable fix and matches the existing ClawHub behavior. A WhatsApp-only workaround or docs-only note would leave the same failure mode for other official external plugins.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/85869 by adding a generic compatibility guard for direct npm/package plugin installs. The guard should reject or roll back an install before persisting/enabling the plugin when package metadata declares an OpenClaw/plugin API requirement newer than the running host, including `openclaw.compat.pluginApi`, `openclaw.compat.minGatewayVersion` or an equivalent `peerDependencies.openclaw` range. Preserve the existing ClawHub behavior, do not special-case WhatsApp auth flows, and avoid broad provider/channel policy changes. Add regression coverage for the WhatsApp version-skew shape where `minHostVersion` is older than the plugin API/peer requirement, and add/update a single CHANGELOG.md Fixes entry if this becomes a maintainer-owned fix PR.
