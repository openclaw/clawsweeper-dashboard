## Summary

Keep open: current main still only gates package installs on `openclaw.install.minHostVersion`, while the WhatsApp package advertises a newer OpenClaw/plugin API requirement than that install floor. The reported floating npm install skew remains source-reproducible, and no canonical duplicate or merged fix currently owns this exact install-compatibility guard.

## What This Changes

Request that `openclaw plugins install @openclaw/whatsapp` either select a host-compatible plugin version or fail clearly before installing a WhatsApp package built for a newer OpenClaw core/plugin SDK.

## Best Possible Solution

Add a generic package-install compatibility gate that rejects or avoids npm plugin packages whose declared OpenClaw/plugin API requirement is newer than the running host, and align WhatsApp metadata/tests with that contract.

## Reproduction Assessment

Yes, from source inspection: a 2026.5.10-beta.1 host satisfies WhatsApp's `>=2026.4.25` install floor but not its current `>=2026.5.24` OpenClaw/plugin API requirement. The npm install validator only enforces the lower install floor, so the mismatch remains source-reproducible without a live install.

## Solution Assessment

Yes, the requested compatibility guard is the right boundary. The maintainable fix is generic npm package compatibility validation or compatible-version selection, not a WhatsApp-only recovery special case.
