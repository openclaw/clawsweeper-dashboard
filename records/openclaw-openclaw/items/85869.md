## Label Justifications

- `P2`: This is a concrete plugin installation compatibility bug with a confusing WhatsApp recovery failure, but it is limited to external or floating plugin installs.
- `impact:auth-provider`: The reported failure occurs while recovering WhatsApp auth and can leave the channel plugin unable to load because it expects a newer core/plugin SDK.

## Summary

Keep open: current main still lets the direct npm plugin install path validate only `openclaw.install.minHostVersion`, while the current WhatsApp package advertises newer OpenClaw/plugin API requirements that an older host would not satisfy.

## What This Changes

Guard floating `@openclaw/whatsapp` installs so older OpenClaw cores either receive a compatible plugin package or get a clear compatibility error before the install is persisted.

## Best Possible Solution

Add a generic package-install compatibility gate that rejects package metadata requiring a newer OpenClaw/plugin SDK before persisting the install, and align WhatsApp metadata and regression tests with that contract.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: a `2026.5.10-beta.1` host satisfies WhatsApp's enforced `>=2026.4.25` install floor while the same package says its OpenClaw/plugin API requirement is `>=2026.5.22`. I did not run a live npm install because this review is read-only.

## Solution Assessment

Yes, the best fix is a generic installer compatibility check, not a WhatsApp-only workaround or a docs-only warning. The existing ClawHub compatibility guard shows the maintainable fail-closed shape to mirror for direct package installs.

## Repair Work Prompt

Fix OpenClaw plugin package install compatibility for https://github.com/openclaw/openclaw/issues/85869. On direct npm/package installs, before persisting an installed plugin, inspect `package.json` metadata and reject packages whose OpenClaw compatibility requirement is newer than the running host; at minimum align/enforce `openclaw.install.minHostVersion` with `openclaw.compat.pluginApi` and `peerDependencies.openclaw` when present, or add a generic package-compat helper that understands those fields. Preserve the existing ClawHub compatibility behavior and do not add WhatsApp-specific special cases. Add regression coverage for a host like `2026.5.10-beta.1` installing a fixture package with `minHostVersion >=2026.4.25` plus plugin API/OpenClaw peer `>=2026.5.22`; expected result is a clear compatibility error and no persisted install/root shadow. Update WhatsApp metadata/docs/tests as needed, and add a `CHANGELOG.md` Fixes entry for the user-visible install guard.
