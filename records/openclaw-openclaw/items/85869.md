## Summary

Keep open: current main and the latest release still let the direct npm/package install path rely on `openclaw.install.minHostVersion` while WhatsApp advertises a newer OpenClaw peer/plugin API requirement, so the reported version-skew install remains source-reproducible. The later issue comment also confirms this as a real production pain point and points to a compatibility-guard pattern rather than changing the product direction.

## What This Changes

The issue requests a compatibility guard so floating WhatsApp plugin installs cannot pull a package requiring a newer OpenClaw core/plugin SDK than the running host.

## Best Possible Solution

Add a generic direct npm/package install compatibility check that fails before persistence when package metadata declares an OpenClaw/plugin API or peer requirement newer than the running host, and align WhatsApp metadata/tests with that contract.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: a host such as `2026.5.10-beta.1` satisfies the current WhatsApp `minHostVersion` floor while failing the package's `>=2026.5.24` peer/API requirement, and direct npm install only checks the floor.

## Solution Assessment

No, current main is not the best solution because direct npm/package install lacks the compatibility gate already present for ClawHub. The maintainable repair is a generic package-install compatibility check plus metadata coverage.
