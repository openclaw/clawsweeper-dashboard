## Summary

Keep open. Current main and the latest release still expose Discord reactions only as routed system events, while the public plugin hook contract has no reaction add/remove hook names, handler entries, runner methods, or docs; the open wake-policy PR is adjacent and currently conflicting, not a replacement for plugin hook delivery.

## What This Changes

Request to expose Discord reaction add/remove events through the public Hooks system for plugin automation.

## Best Possible Solution

Define a maintainer-approved observation-only reaction hook contract, then wire Discord through the existing authorized reaction notification path while leaving agent-turn wakeups in the separate wake-policy track.

## Reproduction Assessment

Yes, at source level. Current main handles Discord reactions and queues system events, but the public hook registry, handler map, runner, docs, and tests have no reaction add/remove hook path.

## Solution Assessment

Unclear as an implementation until maintainers decide the public contract. Dedicated observation-only hooks look cleaner than overloading `message_received`, but hook names, payload shape, enrichment, and cross-channel scope still need approval.
