## Summary

Keep open. Current main still handles accepted Discord reactions as passive routed system events, but the public plugin hook catalog, handler map, runner, and docs do not expose reaction add/remove hooks; the May 7 production workaround and linked proposal add useful API-design evidence without resolving the request.

## What This Changes

Expose Discord reaction add/remove events through the plugin Hooks system so plugins can observe reaction-driven automation signals.

## Best Possible Solution

Define an additive, documented, observation-only reaction hook contract that emits after existing Discord reaction authorization and routing, carries structured reaction metadata, and stays separate from reaction-triggered agent wake behavior tracked elsewhere.

## Reproduction Assessment

Not applicable as a feature/API request. Source and test inspection gives high confidence that current main accepts Discord reactions and queues system events, while the plugin hook catalog and runner expose no reaction hook path.

## Solution Assessment

Unclear until maintainers choose the public API. A dedicated additive reaction hook appears safer than forcing reactions through message_received because it preserves message hook semantics and allows structured cross-channel metadata later.
