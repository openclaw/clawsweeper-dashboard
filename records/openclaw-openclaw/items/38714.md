## Summary

Keep open. Current main and the latest release still route Discord reaction notifications as passive system events, while the requested plugin-visible reaction hook remains missing and needs an API contract decision; the adjacent open inbound-activity PR does not cover reaction add/remove events.

## What This Changes

Expose Discord reaction add/remove events through the plugin Hooks system so plugins can run reaction-based automation.

## Best Possible Solution

Choose a public reaction event contract first, then wire accepted Discord reaction add/remove events through the plugin SDK with docs and focused tests while preserving the current passive system-event behavior.

## Reproduction Assessment

Yes. Current main shows Discord reactions are handled and queued as system events, while the plugin hook catalog and runner do not expose reaction hook names or dispatchers.

## Solution Assessment

Unclear until maintainers choose the public contract. Dedicated `reaction_added`/`reaction_removed` hooks are plausible, but a generic cross-channel reaction event hook may age better than a Discord-only seam.
