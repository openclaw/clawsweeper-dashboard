## Summary

Keep open. Current main still exposes accepted Discord reactions as routed system events only, while the plugin and internal hook catalogs still lack a reaction event; the recent proposal and linked workaround add useful design detail rather than making the issue obsolete.

## What This Changes

The issue requests Discord reaction add/remove events to be exposed through the hooks system so plugins can automate reaction-driven workflows.

## Best Possible Solution

Add an additive, documented observation hook contract for accepted reaction events after existing channel authorization and reaction-notification filters, with structured payload tests and docs, while leaving agent wake-up semantics to the separate canonical issue.

## Reproduction Assessment

Yes, by source inspection: an accepted Discord reaction reaches `enqueueSystemEvent`, while ordinary inbound messages call plugin `message_received` and internal `message:received` hooks. I did not run a live Discord event in this read-only review.

## Solution Assessment

Unclear until maintainers settle the public API. A dedicated observation hook is a safer direction than forwarding reactions as synthetic `message_received` events because it preserves message semantics and leaves room for structured cross-channel reaction metadata.
