## Summary

Keep open. Current `main` and v2026.6.6 still expose Discord reaction notifications only as passive system events, while the public plugin hook names, runner, and docs have no reaction add/remove hook; the production workaround/proposal strengthens the case, but this still needs maintainer API/product approval before implementation.

## What This Changes

Request to expose Discord reaction add/remove events through the plugin Hooks system so plugins can automate reaction-driven workflows.

## Best Possible Solution

Define one narrow observation-only plugin reaction hook contract, then wire Discord through the existing authorized reaction notification path with docs and focused tests; keep agent-turn wakeups in the separate feature track.

## Reproduction Assessment

Yes. Source inspection shows Discord reactions are accepted and enqueued as system events, while the public plugin hook names and `message_received` dispatch path have no reaction hook entry point.

## Solution Assessment

Unclear until maintainers approve the public plugin API shape. A dedicated observation-only reaction hook is cleaner than overloading `message_received`, but payload, enrichment, filtering, and cross-channel scope need product review.
