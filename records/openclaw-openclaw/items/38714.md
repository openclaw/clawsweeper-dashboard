## Summary

Keep open. Current main still turns accepted Discord reactions into routed system events, while the public hook catalog, handler map, runner, and docs do not expose a reaction add/remove hook; the May 7 proposal and reference plugin are useful design input, but they are not merged implementation.

## What This Changes

The issue requests Discord reaction add/remove events to be exposed through the plugin Hooks system for reaction-driven automations.

## Best Possible Solution

Define a stable additive observation-only reaction hook API emitted after existing Discord reaction filters pass, with structured metadata, runner support, focused Discord/plugin tests, and docs aligned with the adjacent cross-channel reaction work.

## Reproduction Assessment

Not applicable: this is a feature/API request, not a broken existing contract. Source inspection gives high confidence that accepted Discord reactions enqueue system events while the plugin hook catalog and runner expose no reaction hook path.

## Solution Assessment

Unclear until maintainers choose the public API. A dedicated additive reaction hook appears safer than overloading message_received because it preserves message hook semantics and leaves room for structured cross-channel metadata.
