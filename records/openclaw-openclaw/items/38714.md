## Summary

Keep open. Current main and v2026.5.12 still expose accepted Discord reactions as queued system events, while the public plugin hook catalog, handler map, runner, registration guard, and hook docs have no reaction hook path; the May 7 production workaround/proposal is useful design evidence but still needs maintainer API/product review.

## What This Changes

Add Discord reaction add/remove events to the public plugin Hooks system so plugins can automate on reaction metadata.

## Best Possible Solution

Define a stable additive observation-only reaction hook emitted after existing Discord reaction filters pass, with structured metadata, runner/registry support, focused Discord/plugin tests, and public docs; keep agent wake behavior separate unless maintainers explicitly combine the contracts.

## Reproduction Assessment

Not applicable: this is a feature/API request, not a broken documented behavior. Source inspection on current main shows accepted Discord reactions enqueue system events while the plugin hook catalog and runner expose no reaction hook path.

## Solution Assessment

Unclear until maintainers choose the public API. A dedicated additive reaction hook appears safer than overloading message_received because it preserves message hook semantics and leaves room for structured cross-channel metadata.
