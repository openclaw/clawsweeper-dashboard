## Summary

Keep open. Current main still turns accepted Discord reactions into passive system events, while the public plugin hook catalog, handler map, runner, and docs still have no reaction add/remove hook; the May 7 production-backed proposal is useful API-design input but does not resolve the request.

## What This Changes

The issue requests Discord reaction add/remove events to be exposed through the plugin Hooks system for reaction-driven automation.

## Best Possible Solution

Define an additive, observation-only reaction hook contract, then emit it from the Discord reaction listener after existing access and reaction-notification checks with structured payload types, docs, and focused tests.

## Reproduction Assessment

Not applicable: this is a feature/API request rather than a broken documented behavior. Source inspection gives high confidence that current main queues accepted Discord reactions as system events but has no plugin reaction hook path.

## Solution Assessment

Unclear until maintainers choose the public API. A dedicated additive reaction hook appears more maintainable than overloading message_received, but naming, payload, and channel scope are still product/API decisions.
