## Summary

Keep open. Current main still exposes Discord reactions only as queued system events, not as public plugin hook events; the May 7 production workaround and proposal add useful design evidence but do not replace a merged hook API.

## What This Changes

Expose Discord reaction add/remove events to the plugin Hooks system so plugins can automate reaction-driven workflows.

## Best Possible Solution

Define a stable additive observation-only reaction hook API emitted after existing Discord reaction filters pass, with structured metadata, runner support, focused Discord/plugin tests, and docs aligned with adjacent cross-channel reaction work.

## Reproduction Assessment

Not applicable: this is a feature/API request, not a broken documented behavior. Source inspection on current main gives high confidence that accepted Discord reactions enqueue system events while the plugin hook catalog and runner expose no reaction hook path.

## Solution Assessment

Unclear until maintainers choose the public API. A dedicated additive reaction hook appears safer than overloading message_received because it preserves message hook semantics and leaves room for structured cross-channel metadata.
