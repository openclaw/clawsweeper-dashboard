## Summary

Keep open. This remains the canonical Discord plugin-hook/API request: current main still turns accepted Discord reactions into passive system events, but the public plugin hook catalog, handler map, runner surface, and docs do not expose reaction add/remove hooks; the recent production-backed workaround proposal is useful design input rather than a reason to close.

## What This Changes

Expose Discord reaction add/remove events through the plugin Hooks system so plugins can react to emoji actions without synthetic message parsing.

## Best Possible Solution

Define an additive, documented reaction event contract that fires after existing Discord reaction authorization and route resolution, carries structured reaction metadata, updates plugin hook docs/tests, and keeps agent wake-up behavior separate.

## Reproduction Assessment

Not applicable as a feature/API request. Source inspection gives high confidence that accepted Discord reactions are detected and queued as system events while current plugin hooks remain message-only for this path.

## Solution Assessment

Unclear until maintainers choose the public API. A dedicated additive reaction hook/event contract looks safer than forwarding reactions as synthetic message_received events because it preserves hook semantics and leaves room for structured cross-channel metadata.
