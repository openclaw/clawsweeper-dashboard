## Summary

Keep open. Current main and the latest release still expose accepted Discord reactions as routed system events, while the public plugin Hook catalog, handler map, runner, registration guard, and docs have no reaction hook path; the May 7 production workaround/proposal is useful design evidence but still needs maintainer API/product review.

## What This Changes

The issue requests Discord reaction add/remove events in the plugin Hooks system so plugins can automate reaction-driven workflows.

## Best Possible Solution

Define a stable additive observation-only reaction hook emitted after accepted Discord reaction filters pass, with structured metadata, runner/registry support, focused Discord/plugin tests, and public docs; keep agent wake behavior separate unless maintainers explicitly combine the contracts.

## Reproduction Assessment

Not applicable: this is a feature/API request, not broken documented behavior. Source inspection on current main shows accepted Discord reactions enqueue system events while the plugin hook catalog and runner expose no reaction hook path.

## Solution Assessment

Unclear until maintainers choose the public API. A dedicated additive reaction hook appears safer than overloading message_received because it preserves message hook semantics and leaves room for structured cross-channel metadata; the open inbound_activity PR is adjacent but does not add reaction add/remove hooks.
