## Summary

Keep open. Current main still routes Discord reaction add/remove events into routed session system events only; the plugin hook catalog, plugin hook runner, internal hook docs, and plugin hook docs still have no reaction event surface. Related #52658 and #60805 were closed toward this canonical hook/API gap, while #17840 remains the separate opt-in agent-turn follow-up.

## What This Changes

Expose Discord reaction add/remove notifications through the Hooks system so plugins or HOOK.md automation can react to them.

## Best Possible Solution

Add an additive, documented reaction event contract that fires after existing Discord reaction authorization and route resolution, carries structured reaction metadata, updates plugin/internal hook docs and tests, and leaves agent wake-up semantics to #17840.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: DiscordReactionListener reaches enqueueSystemEvent only, while normal inbound messages invoke plugin message_received and internal message:received hooks.

## Solution Assessment

Unclear until maintainers choose the public API. The maintainable direction is an additive reaction hook/event contract; forwarding reactions as synthetic message_received events alone risks muddling hook semantics and cross-channel payloads.
