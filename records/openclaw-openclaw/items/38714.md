## Summary

Keep open. Current main and the latest shipped tag still route accepted Discord reactions into system events only, while the public plugin hook catalog, handler map, runner, registration guard, and docs have no reaction add/remove hook; the May 7 production workaround/proposal is useful design input but still needs maintainer API review.

## What This Changes

The issue requests Discord reaction add/remove support in the public plugin Hooks system so plugins can automate on user reactions.

## Best Possible Solution

Define a stable additive observation-only reaction hook emitted after accepted Discord reaction filters pass, with structured metadata, runner/registry support, focused Discord/plugin tests, and public docs while keeping agent wake behavior separate unless maintainers explicitly combine the contracts.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/API request, not broken documented behavior. Source inspection gives high-confidence proof that accepted Discord reactions enqueue system events while no reaction hook name, handler map entry, or runner method exists.

## Solution Assessment

Unclear until maintainers choose the public API. A dedicated additive reaction hook appears safer than overloading message_received, and the adjacent inbound_activity PR does not replace reaction add/remove observation hooks.
