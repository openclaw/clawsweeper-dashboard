## Summary

Keep open. Current main and the latest release still turn accepted Discord reactions into queued system events, while the public plugin hook catalog, handler map, runner, and registration guard have no reaction hook path; the May 7 production workaround/proposal is useful design evidence but still needs maintainer API/product review.

## What This Changes

Request to expose Discord reaction add/remove events as public plugin hooks so automations can respond to accepted reactions.

## Best Possible Solution

Define a stable additive observation-only reaction hook emitted after existing Discord reaction filters pass, with structured metadata, runner/registry support, focused Discord/plugin tests, and public docs; keep agent wake behavior separate unless maintainers explicitly combine the contracts.

## Reproduction Assessment

Not applicable: this is a feature/API request, not broken documented behavior. Source inspection on current main gives high confidence that accepted Discord reactions enqueue system events while the plugin hook catalog and runner expose no reaction hook path.

## Solution Assessment

Unclear until maintainers choose the public API. A dedicated additive reaction hook appears safer than overloading message_received because it preserves message hook semantics and leaves room for structured cross-channel metadata.
