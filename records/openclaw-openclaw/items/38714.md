## Summary

Keep open. Current main still has Discord reaction handling and passive system-event enqueueing, but the public plugin hook surface does not expose reaction add/remove hooks; the May 7 production-backed proposal adds useful API-design context rather than resolving the request.

## What This Changes

The issue requests Discord reaction add/remove events in the plugin Hooks system so plugins can automate reaction-driven workflows.

## Best Possible Solution

Define an additive, documented observation-only reaction hook contract that emits after existing Discord reaction authorization and route resolution, carries structured reaction metadata, and keeps agent wake-up behavior separate from plugin observation.

## Reproduction Assessment

Not applicable as a feature/API request. Source inspection gives high confidence that accepted Discord reactions are queued as system events while the current plugin hook catalog and runner surface have no reaction hook path.

## Solution Assessment

Unclear until maintainers choose the public API. A dedicated additive reaction hook looks safer than forcing reactions through message_received because it preserves current message hook semantics and leaves room for structured cross-channel metadata.
