## Summary

Keep open. Current main still exposes accepted Discord reactions as queued system events, not as public plugin hook events, and the recent production-workaround proposal adds useful design evidence without replacing a merged hook API.

## What This Changes

Expose Discord reaction add/remove events through the plugin Hooks system so plugins can automate on user reactions.

## Best Possible Solution

Define a stable additive observation-only reaction hook API emitted after existing reaction filters pass, with structured metadata, runner support, focused Discord/plugin tests, and public docs.

## Reproduction Assessment

Not applicable: this is a feature/API request, not a broken documented behavior. Source inspection on current main gives high confidence that accepted Discord reactions enqueue system events while the plugin hook catalog and runner expose no reaction hook path.

## Solution Assessment

Unclear until maintainers choose the public API. A dedicated additive reaction hook appears safer than overloading message_received because it preserves message hook semantics and leaves room for structured cross-channel metadata.
