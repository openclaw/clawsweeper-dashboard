## Summary

Keep open. Current main still accepts Discord reaction events only into the system-event path, while the public plugin hook catalog, handler map, runner, registration guard, and docs have no reaction add/remove hook; the May 7 production workaround/proposal is useful design evidence but still needs maintainer API review.

## What This Changes

Request to expose Discord reaction add/remove events through the public plugin Hooks system so plugins can automate reaction-driven workflows.

## Best Possible Solution

Define a stable additive observation-only reaction hook emitted after accepted Discord reaction filters pass, with structured metadata, runner/registry support, focused Discord/plugin tests, and public docs while keeping agent wake behavior separate unless maintainers explicitly combine the contracts.

## Reproduction Assessment

Not applicable: this is a feature/API request, not broken documented behavior. Source inspection on current main shows accepted Discord reactions enqueue system events while the plugin hook catalog and runner expose no reaction hook path.

## Solution Assessment

Unclear until maintainers choose the public API. A dedicated additive reaction hook appears safer than overloading message_received, and the adjacent inbound_activity PR does not replace reaction add/remove observation hooks.
