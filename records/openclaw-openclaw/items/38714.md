## Summary

Keep open. Current main still treats accepted Discord reactions as routed system events, but neither the plugin hook catalog nor the internal hook event surface exposes reaction add/remove events; recent discussion added a concrete production-backed proposal that still needs maintainer API direction.

## What This Changes

The issue requests first-class Discord reaction add/remove events in the Hooks system for plugin automation.

## Best Possible Solution

Add an additive, documented reaction event contract that fires after existing reaction authorization and route resolution, carries structured reaction metadata, updates plugin/internal hook docs and tests, and keeps agent wake-up semantics separate.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: an accepted Discord reaction reaches enqueueSystemEvent, while ordinary inbound messages are the path that fires plugin message_received and internal message:received hooks.

## Solution Assessment

Unclear until maintainers choose the public API. A dedicated additive reaction hook/event contract looks safer than forwarding reactions as synthetic message_received events because it preserves hook semantics and leaves room for structured cross-channel metadata.
