## Label Justifications

- `P2`: This is a normal-priority extension gap that blocks useful reaction-driven plugin workflows but does not break ordinary Discord conversations.
- `impact:message-loss`: Authorized Discord reaction signals reach OpenClaw but are suppressed from registered plugin automation handlers.

## Summary

Current main still receives authorized Discord reactions only as passive system events and exposes no reaction event through the typed plugin hook API. The request remains distinct from reaction-triggered agent turns, but maintainers need to approve the durable public hook contract before implementation.

## What This Changes

Add observation-only plugin hook events for Discord reaction additions and removals so plugins can run reaction-driven automation without starting an agent turn.

## Best Possible Solution

Approve one typed, observation-only reaction event contract in the public plugin SDK, then have channel-owned listeners emit it after ingress authorization while preserving existing system-event and agent-turn behavior.

## Reproduction Assessment

Yes, at source level. Current main receives and authorizes Discord add/remove reactions, then queues a system event, while the typed plugin hook registry has no reaction event.

## Solution Assessment

Yes for the architectural direction, but not yet for the exact proposed contract. A dedicated portable observation hook is narrower and more maintainable than synthetic message_received events or automatic agent turns, while naming, payload, gating, and initial channel coverage need maintainer approval.

## Risks / Open Questions

- A Discord-specific public hook could fragment plugin semantics across Discord, Slack, Matrix, and other channels that expose reactions differently.
- The hook must remain observation-only and must not accidentally wake an agent or duplicate the opt-in behavior tracked at https://github.com/openclaw/openclaw/issues/17840.
- Using message_received for reactions would conflate distinct event types and create a competing representation alongside the existing passive system event.
