## Summary

Keep open. Current main still exposes Discord reactions as routed system events, not plugin hook events; the May 7 concrete proposal and linked reference plugin are useful design evidence, but they are not a merged OpenClaw implementation.

## What This Changes

Add Discord reaction add/remove events to the plugin Hooks system for reaction-driven automation.

## Best Possible Solution

Define a stable additive observation-only reaction hook API emitted after existing Discord reaction filters pass, with structured metadata, runner support, focused Discord/plugin tests, and docs aligned with adjacent cross-channel reaction work.

## Reproduction Assessment

Not applicable: this is a feature/API request, not a broken existing contract. Source inspection gives high confidence that accepted Discord reactions enqueue system events while the hook catalog, handler map, runner, and docs expose no reaction add/remove hook path.

## Solution Assessment

Unclear until maintainers choose the public API. A dedicated additive reaction hook appears safer than overloading message_received because it preserves message hook semantics and leaves room for structured cross-channel metadata.
