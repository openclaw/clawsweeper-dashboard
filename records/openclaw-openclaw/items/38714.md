## Summary

Keep open. Current main and v2026.6.8 still expose Discord reactions as routed system events, while the public plugin hook names, runner, docs, and tests show no reaction add/remove hook; the May 7 production-workaround proposal strengthens the request but leaves public API shape and cross-channel scope for maintainer decision.

## What This Changes

Request a typed plugin hook surface so Discord reaction add/remove events can trigger plugin automation instead of only becoming passive system events.

## Best Possible Solution

Define a maintainer-approved observation-only reaction hook contract, then wire Discord through the existing authorized reaction notification path while leaving agent-turn wakeups to the separate reaction-triggered-turn issue.

## Reproduction Assessment

Yes, from source inspection. Current main registers and handles Discord reaction events, but the handler only enqueues system events and the public plugin hook registry has no reaction hook name or runner; I did not run a live Discord scenario in this read-only review.

## Solution Assessment

No, not as an immediate autonomous fix PR. A dedicated observation-only hook appears cleaner than overloading message_received, but maintainers still need to choose hook names, payload shape, enrichment boundaries, and cross-channel scope.
