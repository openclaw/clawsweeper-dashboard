## Summary

Keep open. Current main and v2026.6.5 still handle authorized Discord reactions as passive system events without a typed plugin reaction hook, and the adjacent open inbound-activity PR does not directly resolve the requested reaction hook contract.

## What This Changes

The issue requests typed plugin hook support for Discord reaction add/remove events so plugins can build reaction-driven automation.

## Best Possible Solution

Approve one portable, observation-only reaction hook contract in the public plugin SDK, then have channel-owned reaction listeners emit it after existing authorization and notification filters while preserving passive system-event behavior.

## Reproduction Assessment

Yes, at source level. Current main receives and authorizes Discord add/remove reactions, then queues a system event, while the typed plugin hook registry has no reaction event to register.

## Solution Assessment

Yes for the architectural direction, but unclear for the exact contract. A dedicated portable observation hook is more maintainable than synthetic `message_received` events or automatic agent turns, but names, payload, gating, and cross-channel scope need maintainer approval.
