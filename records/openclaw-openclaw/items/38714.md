## Summary

Keep open. Current main still treats accepted Discord reactions as routed system events, but it does not expose reaction add/remove events through the public plugin hook catalog, typed handler map, hook runner, or plugin docs; the May 7 production workaround and reference plugin are useful API-design evidence, not a merged OpenClaw implementation.

## What This Changes

Add Discord reaction add/remove events to the plugin Hooks system so plugin automations can observe user reactions.

## Best Possible Solution

Define an additive, observation-only reaction hook contract that emits after existing Discord reaction authorization and routing, carries structured reaction metadata, updates plugin docs/tests, and stays separate from reaction-triggered agent wake behavior tracked in https://github.com/openclaw/openclaw/issues/17840.

## Reproduction Assessment

Not applicable as a feature/API request. Source inspection gives high confidence that current main accepts Discord reactions and queues system events, while the public hook catalog, handler map, runner, and docs expose no reaction hook path.

## Solution Assessment

Unclear until maintainers choose the public API. A dedicated additive reaction hook appears safer than forcing reactions through message_received because it preserves message hook semantics and leaves room for structured cross-channel metadata.
