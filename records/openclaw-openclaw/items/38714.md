## Summary

Keep open. Current main and v2026.6.6 still route authorized Discord reaction add/remove notifications only into passive session system events; the public plugin hook inventory, runner, and docs expose message lifecycle hooks but no typed reaction event. The production workaround and concrete cross-channel proposal strengthen the use case, while the separate open agent-turn request does not cover observation-only plugin automation.

## What This Changes

Add typed plugin hook events for Discord reaction additions and removals so plugins can observe and automate from reaction activity.

## Best Possible Solution

Define one portable, observation-only reaction hook contract in the public plugin SDK, then have channel owners emit it after existing ingress authorization without synthesizing text, starting agent runs, or adding a second Discord client.

## Reproduction Assessment

Yes, at source level. Current main has a Discord reaction listener that queues system events and a separate typed plugin hook runner, with no reaction hook or adapter between them; no live Discord setup is needed to establish that API gap.

## Solution Assessment

Yes directionally, but not yet at contract detail. A dedicated observation-only event is narrower and safer than synthetic `message_received` events or agent-turn triggering, while naming, payload, filtering, and cross-channel scope need maintainer approval.
