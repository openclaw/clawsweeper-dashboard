## Summary

Keep open. Current main still routes accepted Discord reaction add/remove events into passive system events, but the public plugin hook catalog, hook handler map, runner, and docs do not expose reaction hooks; the recent production-backed proposal gives useful design input rather than making the item stale or superseded.

## What This Changes

Request support for Discord reaction add/remove events in the plugin Hooks system so plugins can build reaction-driven automation.

## Best Possible Solution

Add an additive, documented reaction event contract that fires after existing Discord reaction authorization and route resolution, carries structured reaction metadata, updates plugin hook docs/tests, and keeps agent wake-up semantics separate.

## Reproduction Assessment

Not applicable as a feature/API request. Source inspection still gives high confidence that accepted Discord reactions are detected and queued as system events, while current plugin hooks remain message-only for this path.

## Solution Assessment

Unclear until maintainers choose the public API. A dedicated additive reaction hook/event contract looks safer than forwarding reactions as synthetic message_received events because it preserves hook semantics and leaves room for structured cross-channel metadata.
