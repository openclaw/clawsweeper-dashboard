## Summary

Keep open. Current main still routes Discord reaction add/remove events into session system events only, while the plugin and internal hook surfaces still expose message hooks but no reaction event; related #52658 and #60805 were closed toward this canonical API gap, while #17840 remains the separate opt-in agent-turn follow-up.

## What This Changes

The issue requests Discord reaction add/remove events to be exposed through plugin or internal hooks so automations can respond to user reactions.

## Best Possible Solution

Add an additive, documented reaction event contract that fires after existing Discord reaction authorization and route resolution, carries structured reaction metadata, updates plugin/internal hook docs and tests, and keeps agent wake-up semantics separate.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: DiscordReactionListener reaches enqueueSystemEvent, while ordinary inbound messages invoke plugin message_received and internal message:received hooks.

## Solution Assessment

Unclear until maintainers choose the public API. A dedicated additive reaction hook/event contract is safer than forwarding reactions as synthetic message_received events because it preserves hook semantics and leaves room for structured cross-channel metadata.
