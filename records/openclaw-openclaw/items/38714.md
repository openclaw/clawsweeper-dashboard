## Summary

Keep open. Current main still detects Discord reactions and queues them as routed system events, but the public plugin hook contract has no reaction add/remove hook names, handler entries, runner methods, docs, or hook-delivery tests; the open wake-policy PR is adjacent rather than a replacement for plugin hook delivery.

## What This Changes

The issue requests Discord reaction add/remove events be exposed through the plugin Hooks system for reaction-driven automation.

## Best Possible Solution

Define a maintainer-approved observation-only reaction hook contract, then wire Discord through the existing authorized reaction notification path while leaving agent-turn wakeups in the separate wake-policy track.

## Reproduction Assessment

Yes, at source level. Current main registers and handles Discord reaction events, but accepted reactions are queued as system events while the public plugin hook registry, handler map, runner, docs, and tests have no reaction add/remove hook path.

## Solution Assessment

No, not as an immediate autonomous fix PR. A dedicated observation-only hook looks cleaner than overloading `message_received`, but maintainers still need to choose hook names, payload shape, enrichment boundaries, and cross-channel scope.
