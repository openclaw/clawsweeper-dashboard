## Summary

Keep open. Current main still routes accepted Discord reactions into session-scoped system events, but the public plugin hook catalog, handler map, runner, and docs do not expose reaction add/remove hooks; the May 7 proposal and reference plugin are useful design input, not merged implementation.

## What This Changes

Request to expose Discord reaction add/remove events through the plugin Hooks system so plugins can automate reaction-driven workflows.

## Best Possible Solution

Define a stable additive observation-only reaction hook API emitted after existing Discord reaction filters pass, with structured metadata, runner support, focused Discord/plugin tests, and docs aligned with adjacent cross-channel reaction work.

## Reproduction Assessment

Not applicable: this is a feature/API request, not a broken existing contract. Source inspection gives high confidence that accepted Discord reactions enqueue system events while the plugin hook catalog and runner expose no reaction hook path.

## Solution Assessment

Unclear until maintainers choose the public API. A dedicated additive reaction hook appears safer than overloading message_received because it preserves message hook semantics and leaves room for structured cross-channel metadata.
