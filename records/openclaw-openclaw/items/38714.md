## Summary

Keep open. Current main still routes Discord reaction add/remove events into session system events only; the plugin hook catalog, hook runner, internal hook types, and public hook docs still have no reaction event surface. Related #52658 and #60805 were already closed toward this canonical hook/API gap, while #17840 separately tracks opt-in reaction-triggered agent turns.

## What This Changes

Expose Discord reaction add/remove events through the Hooks system so plugins or hook automation can respond to user emoji reactions.

## Best Possible Solution

Add an additive, documented reaction event contract that fires after existing Discord reaction authorization/filtering and route resolution, carries structured reaction metadata, updates plugin and internal hook docs/types/tests, and leaves reaction-triggered agent turns to #17840.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: DiscordReactionListener reaches enqueueSystemEvent only, while the normal inbound message path is the one that invokes plugin message_received and internal message:received hooks.

## Solution Assessment

Unclear until maintainer API choices are made. The maintainable direction is an additive reaction hook/event contract, but the exact names, payload shape, and cross-channel portability should be settled before an automated fix PR changes the public hook surface.
