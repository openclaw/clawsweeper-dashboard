## Summary

Keep open. Current main still receives Discord reaction add/remove events and queues accepted reactions as system events, but the public plugin hook contract still has no reaction hook names, handler map entries, runner methods, or docs; the adjacent wake-policy PR does not replace plugin hook delivery.

## What This Changes

Request to expose Discord reaction add/remove events through the plugin Hooks system.

## Best Possible Solution

Define a maintainer-approved observation-only reaction hook contract, then wire Discord through the authorized reaction notification path while leaving agent-turn wakeups in the separate wake-policy track.

## Reproduction Assessment

Yes, at source level. Current main registers Discord reaction listeners and queues accepted reactions as system events, while the public plugin hook names, handler map, runner surface, docs, and tests have no reaction add/remove hook path.

## Solution Assessment

No, not as an immediate autonomous fix. A dedicated observation-only hook looks cleaner than overloading `message_received`, but maintainers still need to choose hook names, payload schema, enrichment boundaries, and cross-channel scope.
