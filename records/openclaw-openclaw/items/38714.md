## Summary

Keep open. Current main still accepts Discord reaction gateway events and queues them as system events, but the public plugin hook catalog, typed handler map, runner, registration guard, and docs do not expose reaction add/remove hooks; the May 7 production workaround/proposal is useful design input, not an implemented core contract.

## What This Changes

Expose Discord reaction add/remove notifications as typed plugin hook events so plugins can automate reaction-driven workflows.

## Best Possible Solution

Define a stable additive observation-only reaction hook emitted after accepted Discord reaction filters pass, with structured metadata, runner/registry support, focused Discord and plugin tests, and public docs while keeping agent wake behavior separate unless maintainers explicitly combine the contracts.

## Reproduction Assessment

Not applicable: this is a feature/API request, not broken documented behavior. Source inspection on current main shows accepted Discord reactions enqueue system events while the public plugin hook catalog and runner expose no reaction hook path.

## Solution Assessment

Unclear until maintainers choose the public API contract. A dedicated additive observation hook appears safer than overloading `message_received`, and the adjacent `inbound_activity` PR does not cover reaction add/remove events.
