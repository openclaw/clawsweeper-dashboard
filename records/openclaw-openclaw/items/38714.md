## Summary

Keep open. Current main still accepts Discord reactions only as routed system events and exposes no plugin hook name, typed payload, runner method, or docs entry for reaction add/remove; the May 7 proposal and external reference plugin are useful design evidence, not a merged OpenClaw implementation.

## What This Changes

Add Discord reaction-added and reaction-removed events to the public plugin Hooks system so plugins can automate on user reactions.

## Best Possible Solution

Define a stable, additive, observation-only reaction hook API emitted after existing Discord reaction authorization, with structured metadata, focused Discord/plugin tests, and docs that keep it separate from reaction-triggered agent turns.

## Reproduction Assessment

Not applicable as a feature/API request. Source inspection gives high confidence that current main queues accepted Discord reactions as system events while the public hook catalog, handler map, runner, and docs expose no reaction hook path.

## Solution Assessment

Unclear until maintainers choose the public API. A dedicated additive reaction hook appears safer than forcing reactions through message_received because it preserves message hook semantics and leaves room for structured cross-channel metadata.
