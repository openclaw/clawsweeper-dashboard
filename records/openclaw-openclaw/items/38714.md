## Summary

Keep open. Current main still detects Discord reaction add/remove events, but routes them only into session system events; neither the plugin hook catalog/runner nor the internal hook docs expose a reaction event surface. The related closed issues point back to this hook/API gap, while #17840 remains a separate agent-turn wake-up request.

## What This Changes

Add Discord reaction add/remove support to the OpenClaw hooks system so automation can respond to user reactions.

## Best Possible Solution

Add an additive, documented reaction event contract after existing Discord reaction authorization and route resolution, with structured reaction metadata, plugin/internal hook docs, regression tests, and no implicit agent wake-up behavior.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: DiscordReactionListener reaches enqueueSystemEvent only, while normal inbound messages invoke plugin message_received and internal message:received hooks.

## Solution Assessment

Unclear until maintainers choose the public API. The maintainable direction is an additive reaction hook/event contract; treating reactions only as synthetic message_received events risks muddying hook semantics and payload portability.
