## Summary

Keep open. Current main still receives accepted Discord reaction add/remove events and queues them as system events, but the public plugin hook catalog, typed handler map, runner, docs, and Discord reaction listener do not expose or dispatch reaction hooks; the May production workaround proposal is useful design input, not an implemented OpenClaw contract.

## What This Changes

Request to expose Discord reaction add/remove events through the plugin Hooks system so plugins can automate reaction-driven workflows.

## Best Possible Solution

Define a generic observation-only reaction hook contract in the Plugin SDK, then wire Discord through it with docs and focused reaction listener tests while keeping reaction-triggered agent turns in the separate agent-wake track.

## Reproduction Assessment

Yes. Source inspection shows accepted Discord reactions enqueue system events, while the current plugin hook catalog and runner do not define or dispatch reaction hooks.

## Solution Assessment

No, not yet. The issue's direction is valid, but the best fix needs a maintainer-approved public hook contract rather than a private Discord-only workaround.
