## Summary

Keep open: current main fixes adjacent OpenAI-compatible MiMo/Kimi replay paths, but the reported Anthropic-messages `xiaomi-token-plan-ams` route remains unhandled. The pinned pi-ai Anthropic stream parser only recognizes native Anthropic thinking events, while OpenClaw's Xiaomi/MiMo wrappers and fallback matchers are scoped to `xiaomi` or `openai-completions` paths.

## What This Changes

The issue requests support for Anthropic-compatible endpoints that emit OpenAI-style `reasoning_content` deltas so MiMo/MiniMax thinking can be preserved and replayed without follow-up 400s.

## Best Possible Solution

Add an owner-approved Anthropic-messages compatibility path for Xiaomi/MiniMax-style `reasoning_content`, preferably by normalizing those deltas into thinking blocks before transcript replay while keeping OpenAI-compatible replay fixes separate.

## Reproduction Assessment

Yes. The issue has a high-confidence source and report-based reproduction path: use `xiaomi-token-plan-ams/mimo-v2.5` over `anthropic-messages` with thinking enabled, then observe the provider's 400 when `reasoning_content` is not replayed; I did not run a live credentialed request here.

## Solution Assessment

Unclear. The adjacent OpenAI-compatible replay fixes are not sufficient for this Anthropic stream path, and the best implementation needs maintainer choice between disabling thinking for affected routes and implementing proper SSE `reasoning_content` translation/upstream pi-ai support.
