## Summary

Keep this PR open. Current main still has the early-abort responsePrefix interpolation gap, and this draft contains useful source work, but it is not merge-ready because routed abort replies still normalize without a prefix context and the PR was written against legacy channel paths that have since moved under bundled plugins.

## What This Changes

The PR seeds responsePrefix template context from persisted session runtime metadata, threads sessionKey into reply-prefix creation call sites, and adds regression coverage for stored metadata and post-selection override behavior.

## Best Possible Solution

Land one canonical current-main fix for #45314 that populates response-prefix context before early stop/abort reply normalization across direct and routed delivery, updates the current extension/pipeline call sites, preserves onModelSelected as the later authoritative override, and adds focused tests for direct aborts, routed aborts, stored runtime metadata, missing metadata fallback, and post-selection override behavior.

## Reproduction Assessment

Yes. Configure messages.responsePrefix with {model} and/or {thinkingLevel}, start an agent turn, then send stop or /stop; the current fast-abort path returns before onModelSelected and the template resolver leaves missing values as literal placeholders.

## Solution Assessment

No, not yet. Persisted session metadata is a plausible source for early abort context, but this PR still misses routed reply normalization and is stale relative to the current plugin-channel paths; a current-main canonical fix should cover both direct and routed delivery.
