## Summary

Keep this PR open: the underlying abort-prefix bug still appears valid on current main, but this draft patch is stale against the current plugin-channel layout and still misses routed abort replies; #45315 is an overlapping open fix path that maintainers should reconcile.

## What This Changes

This PR seeds responsePrefix template context from stored session runtime metadata, threads sessionKey into reply-prefix creation call sites, and adds regression coverage for abort-prefix interpolation.

## Best Possible Solution

Land one canonical current-main fix for #45314 that prepares response-prefix context before early abort normalization across direct and routed delivery, updates the current plugin-channel/gateway call sites, preserves onModelSelected as the later authoritative override, and adds focused direct/routed regression tests.

## Reproduction Assessment

Yes. The source path is clear: configure messages.responsePrefix with {model} or {thinkingLevel}, start an agent turn, then send stop or /stop; the fast-abort path returns before onModelSelected and unresolved template values remain literal.

## Solution Assessment

No. Persisted session metadata is a plausible seed for early abort context, but this PR still misses routeReply normalization and targets legacy channel files rather than the current extension-based call sites.
