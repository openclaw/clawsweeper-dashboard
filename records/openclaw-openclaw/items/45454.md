## Summary

Keep this PR open. Current main still has the early-abort responsePrefix interpolation gap, and this draft remains useful context, but it is stale against the channel plugin move and needs maintainer choice against the competing open implementation before it can land.

## What This Changes

The PR seeds responsePrefix template context from persisted session runtime metadata, threads sessionKey into several reply-prefix creation call sites, and adds regression tests for stored metadata and onModelSelected override behavior.

## Best Possible Solution

Land one canonical current-main fix for #45314 that populates response-prefix context before early stop/abort reply normalization across direct and routed delivery, preserves onModelSelected as the later authoritative override, and adds focused tests for direct aborts, routed aborts, stored runtime metadata, missing metadata fallback, and post-selection override behavior.

## Reproduction Assessment

Yes. Configure messages.responsePrefix with {model} and/or {thinkingLevel}, start an agent turn, then send stop or /stop; current main’s fast-abort path returns before onModelSelected and uses an unseeded prefix context, matching the linked bug.

## Solution Assessment

No, not yet. The persisted-session approach is promising and preserves useful prefixes, but this draft needs a current-main rebase, routeReply coverage, and maintainer choice versus #45315 before it is the best maintainable fix.
