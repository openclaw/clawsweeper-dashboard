## Summary

Keep open. Current main still does not have the native assistant-message copy affordance, but this PR should not merge as-is because its clipboard payload can diverge from the trace-aware text actually rendered in the chat UI and expose hidden assistant trace content.

## What This Changes

The branch modifies `apps/shared/OpenClawKit/Sources/OpenClawChatUI/ChatMessageViews.swift` to add assistant-message hover state and a `CopyMessageButton` that writes message text to `NSPasteboard` or `UIPasteboard` with temporary copied-state feedback.

## Best Possible Solution

Land a revised native copy affordance that derives its clipboard payload from the same visible `AssistantTextParser` segments used for rendering, respects `showsAssistantTrace`, remains usable on macOS and iOS, and includes focused Swift coverage plus the required changelog entry.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is an assistant message like `<think>secret</think><final>answer</final>` with `showsAssistantTrace == false`: current rendering shows only the response segment, while this PR passes the raw primary text to the copy button.

## Solution Assessment

No. The feature direction is reasonable, but the implementation should copy visible/rendered assistant content rather than raw payload text; a shared helper around `AssistantTextParser.segments(from:includeThinking:)` would be the safer maintainable path.
