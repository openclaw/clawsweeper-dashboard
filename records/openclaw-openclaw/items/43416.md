## Summary

Keep this PR open. The feature is useful and is not already implemented on current main, but the branch still has a blocking privacy/correctness issue: it copies the raw assistant payload while the existing renderer can intentionally hide thinking/trace segments.

## What This Changes

The PR adds assistant-message hover state plus a `CopyMessageButton` in `OpenClawChatUI` that writes assistant message text to `NSPasteboard` or `UIPasteboard` with temporary copied-state feedback.

## Best Possible Solution

Keep the native copy affordance, but derive its clipboard payload from the same visible assistant segments used for rendering, respecting `showsAssistantTrace`, and add focused `OpenClawChatUI` coverage for hidden-trace and visible-trace copy text.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is an assistant message like `<think>secret</think><final>answer</final>` with `showsAssistantTrace == false`: current rendering displays only the response segment, while the PR passes the raw primary text to the pasteboard button.

## Solution Assessment

No. The feature direction is reasonable, but the implementation should copy rendered/visible assistant content rather than raw payload text; a shared helper around `AssistantTextParser.segments(from:includeThinking:)` would be the safer maintainable path.

## Review Findings

Overall correctness: patch is incorrect
