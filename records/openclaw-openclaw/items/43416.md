## Summary

Keep open: this is a plausible native chat UX feature, but the branch still has a blocking raw-payload clipboard leak and the screenshot does not prove the click or clipboard behavior. Current main also has no equivalent Swift OpenClawChatUI copy button, so this is not closeable as implemented.

## What This Changes

The branch adds a SwiftUI CopyMessageButton below assistant messages in OpenClawKit with NSPasteboard/UIPasteboard support, hover visibility on macOS, and temporary copied-state feedback.

## Best Possible Solution

Keep the copy-button UX only if clipboard text is derived from the same visible assistant segments as rendering, with focused parser/copy coverage and redacted native app proof.

## Reproduction Assessment

Yes for the blocking review finding: source inspection shows the PR passes raw `primaryText` into `CopyMessageButton`, while current main renders assistant text through `AssistantTextParser.segments(..., includeThinking: showsAssistantTrace)`. A message like `<think>internal</think><final>answer</final>` would render only the response with traces off but copy the raw payload.

## Solution Assessment

No. The copy affordance is a reasonable UX direction, but the implementation should pass rendered visible assistant text to the pasteboard and then demonstrate the fixed native flow.
