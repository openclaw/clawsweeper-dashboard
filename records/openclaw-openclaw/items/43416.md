## Summary

Keep open. The feature is useful, but the current patch copies raw assistant payload text to the clipboard while the renderer can hide thinking segments, and the PR still lacks proof that the real native copy flow works after the fix.

## What This Changes

Adds a shared Swift chat UI copy button for assistant messages using NSPasteboard/UIPasteboard with hover visibility on macOS and temporary checkmark feedback.

## Best Possible Solution

Keep the copy-button feature, but derive clipboard text from the same visible assistant segments used by rendering and add focused parser/copy coverage plus redacted native app proof.

## Reproduction Assessment

Yes for the blocking review finding: source inspection shows the PR copies raw `primaryText`, while current main hides thinking segments through `AssistantTextParser` when traces are off. I did not run a live macOS or iOS app workflow in this read-only review.

## Solution Assessment

No. A copy affordance is a reasonable UX direction, but the implementation should copy the visible rendered assistant text rather than the raw payload.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Copy only visible assistant text:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/ChatMessageViews.swift:228`
  - body: Assistant rendering goes through `AssistantTextParser.segments(..., includeThinking: showsAssistantTrace)`, which can hide `<think>` content when traces are off. This button receives raw `primaryText`, so copying a message like `<think>internal</think><final>answer</final>` would place hidden reasoning and tags on the clipboard instead of only the visible answer.
  - confidence: 0.93
