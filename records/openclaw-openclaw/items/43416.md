## Summary

Keep open: the feature is useful, but the PR still copies the raw assistant payload instead of the rendered visible assistant text, and the attached proof does not show the clipboard result or the fixed hidden-text path.

## What This Changes

The PR adds a SwiftUI copy button to assistant messages in OpenClawKit with hover visibility, macOS/iOS clipboard writes, and temporary copied-state feedback.

## Best Possible Solution

Copy from the same visible `AssistantTextParser` response segments used for rendering, add focused coverage for hidden thinking/tagged output, and provide redacted native app proof showing hover, click, pasted output, and copied-state feedback.

## Reproduction Assessment

Yes for the blocking review finding: source inspection shows the PR passes raw `primaryText` into `CopyMessageButton`, while current main renders assistant text through `AssistantTextParser.segments(..., includeThinking:)`. A message like `<think>internal</think><final>answer</final>` would render only `answer` with traces off but copy the raw payload.

## Solution Assessment

No. The copy affordance is a reasonable UX direction, but this implementation should copy only the rendered visible assistant text and then demonstrate the fixed native flow.
