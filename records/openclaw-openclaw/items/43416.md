## Summary

Keep open. The PR is focused, but the current patch copies raw assistant payload text while current main intentionally strips hidden reasoning segments before rendering, so the copy action can put non-visible `<think>` content and tags on the clipboard. The attached screenshot shows the UI affordance, but not enough after-fix proof for the copy/paste and checkmark behavior.

## What This Changes

The PR adds a copy control to shared Swift chat assistant message bubbles, writing the message text to NSPasteboard/UIPasteboard with temporary checkmark feedback.

## Best Possible Solution

Keep the copy-button feature scoped, but copy the same visible assistant response text produced by the renderer and add focused parser/copy coverage plus redacted real macOS/iOS behavior proof.

## Reproduction Assessment

Yes for the blocking review finding from source inspection: current main hides thinking segments through `AssistantTextParser`, while the PR sends raw `primaryText` to the pasteboard. I did not run a live macOS/iOS app workflow in this read-only review.

## Solution Assessment

No. A copy button is a reasonable UX direction, but this implementation should copy the rendered visible assistant text rather than the raw payload; deriving copy text from `AssistantTextParser` is the narrower maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Copy only rendered assistant text:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/ChatMessageViews.swift:228`
  - body: Assistant rendering goes through `AssistantTextParser.segments(..., includeThinking: showsAssistantTrace)`, which drops `<think>` content when traces are off. This button receives the raw `primaryText`, so a message like `<think>internal</think><final>answer</final>` would display only `answer` while copying the hidden reasoning and tags to the clipboard. Please pass the same visible response text used by the renderer.
  - confidence: 0.93
