## Summary

Keep open. The native chat copy affordance is a plausible UI feature, but the patch still copies raw assistant payload text that the renderer can intentionally hide, and the attached proof only shows the icon rather than the copy/paste result.

## What This Changes

The PR adds a SwiftUI copy button to native assistant chat messages with macOS/iOS pasteboard writes, macOS hover visibility, and temporary checkmark feedback.

## Best Possible Solution

Keep the UI affordance, but derive the copied string from the same visible assistant segments used by the renderer and add proof showing the button, success state, and pasted visible-only clipboard text.

## Reproduction Assessment

Yes for the blocking review finding: source inspection shows the PR passes raw `primaryText` to the pasteboard while current rendering filters hidden thinking segments through `AssistantTextParser`. I did not run the native app in this read-only review.

## Solution Assessment

No. The copy button direction is reasonable, but the implementation should copy the rendered visible assistant text for the current trace setting rather than the raw payload.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Copy the rendered assistant text:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/ChatMessageViews.swift:229`
  - body: `CopyMessageButton` receives raw `primaryText`, but assistant rendering goes through `AssistantTextParser.segments(..., includeThinking: showsAssistantTrace)`. With traces off, `<think>...</think>` content is hidden in the UI while this copy path still writes it and its tags to the pasteboard, so the button can expose content the user did not see. Build the copied string from the visible response segments for the current trace setting.
  - confidence: 0.91
