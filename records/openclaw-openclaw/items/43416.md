## Summary

Keep open: current main does not include this native assistant-message copy affordance, but the PR still copies raw assistant payload text to the pasteboard while the renderer hides thinking segments when traces are off, and the attached screenshot does not prove the click, success state, or pasted clipboard contents.

## What This Changes

Adds a SwiftUI assistant-message copy control in OpenClawKit that writes to macOS/iOS pasteboards, appears on macOS hover, and shows temporary checkmark feedback.

## Best Possible Solution

Keep the copy affordance, but derive the copied string from the same visible assistant segments used by the renderer and add focused proof for hover/tap, feedback, and pasted visible-only text.

## Reproduction Assessment

Yes for the blocking review finding: source inspection shows the PR passes raw `primaryText` to the pasteboard while current rendering filters hidden thinking segments through `AssistantTextParser`. I did not run the native app in this read-only review.

## Solution Assessment

No. The copy-button direction is reasonable, but the implementation should copy rendered visible assistant text for the current trace setting rather than the raw payload.

## Review Findings

Overall correctness: patch is incorrect
