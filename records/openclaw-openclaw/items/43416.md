## Summary

Keep open. Current main still does not have the native Swift assistant-message copy affordance, and this PR still has a blocking clipboard correctness/security issue because it copies raw assistant payload text rather than the rendered visible text; its screenshot proof is also insufficient for merge.

## What This Changes

The PR adds a SwiftUI copy button below assistant messages in the shared OpenClawKit chat UI, with hover visibility, platform clipboard writes, and temporary copied-state feedback.

## Best Possible Solution

Implement the copy action from the same visible assistant segments used by rendering, add focused coverage for hidden thinking/tagged output, and provide redacted native app proof showing copy, paste, and copied-state feedback.

## Reproduction Assessment

Yes for the blocking review finding: source inspection shows the PR passes raw `primaryText` into `CopyMessageButton`, while current main renders assistant text through `AssistantTextParser.segments(..., includeThinking: showsAssistantTrace)`. A message like `<think>internal</think><final>answer</final>` would render only `answer` with traces off but copy the raw payload.

## Solution Assessment

No. The copy affordance is a reasonable UX direction, but this implementation should pass rendered visible assistant text to the pasteboard and then demonstrate the fixed native flow.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9
