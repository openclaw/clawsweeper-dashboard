## Summary

Keep open. The feature is useful, but the PR still copies raw assistant payload text to the pasteboard while current rendering can hide thinking segments, and the attached proof only shows the icon rather than the copy flow or pasted visible-only text.

## What This Changes

Adds a SwiftUI copy button to OpenClawKit assistant message bubbles with macOS hover visibility, iOS/macOS pasteboard writes, and temporary checkmark feedback.

## Best Possible Solution

Keep the copy affordance, but derive the copied string from the same visible assistant segments used by the renderer and add focused proof for hover/tap, feedback, and pasted visible-only text.

## Reproduction Assessment

Yes for the blocking PR finding: source inspection shows the PR passes raw `primaryText` to the pasteboard while current rendering filters hidden thinking segments through `AssistantTextParser`. I did not run the native app in this read-only review.

## Solution Assessment

No. The copy-button direction is reasonable, but the implementation should copy rendered visible assistant text for the current trace setting rather than the raw payload.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Copy only visible assistant text:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/ChatMessageViews.swift:229`
  - body: `CopyMessageButton` receives raw `primaryText`, but assistant rendering goes through `AssistantTextParser.segments(..., includeThinking: showsAssistantTrace)`. With traces off, `<think>...</think>` content is hidden in the UI while this copy path still writes it to the pasteboard, so derive the copied string from the visible segments for the current trace setting.
