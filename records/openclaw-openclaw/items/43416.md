## Summary

Keep open. The native chat copy affordance is useful, but the current diff copies the raw assistant payload even when the UI intentionally hides thinking segments, so it needs a visible-text fix and stronger real behavior proof before merge.

## What This Changes

The branch adds a SwiftUI copy button to native assistant message bubbles in OpenClawChatUI, writing message text to NSPasteboard/UIPasteboard with hover and copied-state feedback.

## Best Possible Solution

Keep the feature, but derive copied text from the same visible assistant segments used for rendering, add a focused parser/copy regression, and require redacted real proof showing paste output after the fix.

## Reproduction Assessment

Yes for the review finding from source. An assistant message with hidden `<think>` content and `showsAssistantTrace == false` renders only response segments, while this PR passes raw `primaryText` to the pasteboard path.

## Solution Assessment

No. The feature direction is reasonable, but the copy source should come from `AssistantTextParser.segments(..., includeThinking: showsAssistantTrace)` so the clipboard matches visible content instead of raw payload text.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Copy only the visible assistant text:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/ChatMessageViews.swift:228`
  - body: Assistant messages are rendered through `AssistantTextParser.segments(..., includeThinking: showsAssistantTrace)`, which drops `<think>` content when traces are hidden, but the new button receives raw `primaryText`. With traces off, clicking copy can put hidden thinking/tags on the clipboard even though the bubble only showed the final response. Build the copied text from the same visible segments used by the renderer.
