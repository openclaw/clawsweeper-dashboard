## Summary

Keep open: current main still allows the gateway hot-reload path to continue after a channel stop timeout, and this PR is a focused fix, but it still needs contributor-side real behavior proof plus a branch refresh/check resolution before merge.

## What This Changes

The PR adds an opt-in channel stop timeout failure mode and uses it from gateway hot reload so reload does not start over a timed-out channel stop.

## Best Possible Solution

Refresh the branch, add redacted real setup proof for a timed-out channel stop during hot reload, rerun the focused gateway validation, and land if maintainers accept the fail-fast behavior.

## Reproduction Assessment

Yes, at source level: a channel task that ignores abort makes stopChannel time out and return, and current hot reload awaits that non-throwing stop before continuing toward plugin replacement or restart. I did not run a live gateway scenario in this read-only review.

## Solution Assessment

Yes for the hot-reload slice: the opt-in failOnTimeout path is narrow and avoids changing normal manual/recovery stop semantics. The wider stale-task health-monitor recovery contract should stay with https://github.com/openclaw/openclaw/issues/71412 and https://github.com/openclaw/openclaw/issues/70024.
