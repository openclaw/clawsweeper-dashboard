## Summary

Keep open for normal PR review: this is a focused fix for a source-visible `maxEntries` subagent eviction bug, and I found no blocking correctness or security findings in the proposed patch.

## What This Changes

The PR adds a session-maintenance preserve-key provider for active and pending subagent child sessions and wires those keys into save-time and load-time session-store pruning/capping, with regression coverage.

## Best Possible Solution

Land this preserve-key approach after maintainer approval and CI, then handle archive fallback or disk-budget lifecycle protection separately only if maintainers want broader coverage beyond the reported `maxEntries` race.

## Reproduction Assessment

Yes, source inspection gives a high-confidence current-main reproduction path: force enforced maintenance over `maxEntries`, leave a completed or pending-delivery subagent child row outside `activeSessionKey`, and capping can remove it before announce/result delivery reads it. I did not run a live Slack reproduction during this read-only review.

## Solution Assessment

Yes, this is the narrow maintainable fix for the reported bug because it reuses the existing `preserveKeys` contract instead of adding Slack-specific delivery behavior or changing cap defaults. The broader archive fallback and instrumentation ideas can remain follow-ups.
