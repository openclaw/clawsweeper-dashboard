## Summary

Keep open: this PR targets a valid source-reproducible bug and the preserve-key approach fits the current session maintenance seam, but the external real-behavior proof is mock-only and the diff has a small lint-blocking test issue before it is merge-ready.

## What This Changes

Adds a session-maintenance preserve-key provider, registers active/pending subagent child session keys, threads those keys through load/save pruning and capping, and adds focused pruning and registry tests.

## Best Possible Solution

Land a narrow lifecycle-aware preservation fix after replacing the lint-blocking assertion and adding real runtime proof; keep archive fallback, extra instrumentation, and disk-budget coverage as follow-ups unless maintainers broaden the scope.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: with `session.maintenance.mode: "enforce"` and a saturated `maxEntries`, current main preserves only the active/static keys, so a pending subagent child row can be capped before announce/result capture reads it; I did not run a live Slack reproduction in this read-only review.

## Solution Assessment

Yes, for the reported `maxEntries` race, feeding active and pending-delivery subagent child keys into the existing `preserveKeys` seam is the narrow maintainable fix. The PR still needs the lint cleanup and real runtime proof before merge.
