## Summary

Keep open: this is a focused bug-fix PR with sufficient real-behavior proof, but the latest head still leaves cleanup dry-run preview out of sync with the new runtime preserve-key behavior.

## What This Changes

The PR adds a session-maintenance preserve-key provider, registers active and pending subagent child sessions, threads preserve keys through save/load/disk-budget maintenance, and adds regression coverage.

## Best Possible Solution

Land the lifecycle-aware preserve-key approach after dry-run and apply maintenance paths use the same preserve set; leave archive fallback and extra instrumentation for separate follow-ups if maintainers want them.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main preserves only the active writer key while subagent result capture still depends on the child session row under an enforced maxEntries cap.

## Solution Assessment

Yes, mostly. The provider-based preserve-key seam is the narrow maintainable fix, but cleanup dry-run preview should collect and pass the same preserve set before prune and cap.
