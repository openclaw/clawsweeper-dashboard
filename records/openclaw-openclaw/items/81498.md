## What This Changes

The PR adds runtime session-maintenance preserve-key providers, registers active or pending subagent child sessions, threads those keys through save/load/disk-budget maintenance, and adds regression tests.

## Best Possible Solution

Land the lifecycle-aware preserve-key approach after preview and apply behavior use the same preserve set; leave archive fallback and extra instrumentation for separate follow-ups if maintainers still want them.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence current-main path: enforced entry maintenance preserves only the active writer key, while subagent announce/result capture can still need a separate child session row.

## Solution Assessment

Yes, mostly. The provider-based preserve-key seam is the narrow maintainable fix, but cleanup dry-run preview should collect and pass the same preserve set before pruning and capping.
