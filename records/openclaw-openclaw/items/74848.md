## Summary

Keep open. Current main still only auto-repairs stale macOS Gateway TLS pins for loopback and Tailscale Serve hosts, while this report is about a public `wss://` host; open PR #75228 references this issue as the intended fix and should be reviewed before cleanup.

## What This Changes

The issue reports the macOS App remote node repeatedly disconnecting with cancelled WSS gateway errors on a public TLS endpoint while `openclaw node run --tls` works in the same environment.

## Best Possible Solution

Review and land PR #75228 or an equivalent macOS TLS pin repair that permits system-trusted public WSS certificate rotations while preserving fail-closed behavior for untrusted, private-CA, unavailable-certificate, and store-key-mismatched cases.

## Reproduction Assessment

Yes for source-level reproduction: current main has a test proving `wss://gateway.example.com` with `systemTrustOk: true` is not auto-repaired, and the TLS delegate cancels mismatched stored pins. Live reproduction needs the reporter’s stale stored app pin plus a rotated public certificate.

## Solution Assessment

Unclear until maintainers approve the TLS trust boundary. The narrow maintainable solution is a macOS stale-pin policy change with regression tests and changelog/docs alignment, not a broad bypass of required TLS pinning.
