## Summary

Keep open. Current main and the latest release still only auto-repair stale macOS gateway TLS pins for loopback and `.ts.net` hosts, while this issue reports the same reconnect loop on a public `wss://` host; the linked open PR at https://github.com/openclaw/openclaw/pull/75228 is the active fix candidate and should be reviewed rather than closing the issue early.

## What This Changes

The issue reports that the macOS app's remote node repeatedly disconnects with gateway TLS `cancelled` errors on a public `wss://` host while the CLI node works in the same environment.

## Best Possible Solution

Keep this issue open until https://github.com/openclaw/openclaw/pull/75228, or an equivalent security-reviewed macOS stale-pin repair policy with live certificate-rotation proof, is landed or explicitly rejected.

## Reproduction Assessment

Yes at source level. Current main has a direct path where a stored-pin mismatch on a non-loopback, non-`.ts.net` public `wss://` host records a TLS pin mismatch and refuses auto-repair; I did not run a live macOS certificate-rotation repro in this read-only review.

## Solution Assessment

Unclear until the linked PR receives maintainer security/product review. A default-off opt-in or another explicit operator-controlled pin refresh path is safer than silently broadening the default auto-repair policy for every system-trusted public host.
