## Summary

Keep open: current main still excludes system-trusted public WSS hosts from stale TLS-pin auto-repair, and open PR #75228 references this issue with closing syntax. The remaining fix changes the macOS app TLS pinning trust boundary, so cleanup should wait for maintainer/security review of that PR or an equivalent fix.

## What This Changes

The issue reports that the macOS app remote node repeatedly cancels WSS gateway connections while a CLI node connects successfully to the same public TLS gateway.

## Best Possible Solution

Land PR #75228 or an equivalent macOS fix that repairs stale pins for maintainer-approved system-trusted public WSS rotations while preserving fail-closed behavior for untrusted, private-CA, unavailable-certificate, and store-key-mismatched cases.

## Reproduction Assessment

Yes for source-level reproduction: current main has a unit test where trusted `wss://gateway.example.com` pin mismatches are not auto-repaired, and the TLS delegate cancels mismatched stored pins. A live reproduction would need the reporter's stale stored macOS app pin plus a rotated public certificate.

## Solution Assessment

Unclear until maintainers approve the TLS trust boundary. The narrow maintainable solution is a macOS stale-pin policy change with regression tests and explicit fail-closed coverage, not a broad bypass of required TLS pinning.
