## Summary

Keep open: current main still excludes non-loopback, non-`.ts.net` public WSS hosts from stale TLS pin auto-repair, and open PR #75228 references this issue with closing syntax, so cleanup should wait for that reviewed fix or an equivalent maintainer-approved change.

## What This Changes

The issue reports that the macOS app node repeatedly disconnects with TLS/WebSocket cancellation on a remote public WSS gateway while the CLI node connects successfully.

## Best Possible Solution

Review and land PR #75228 or an equivalent macOS fix that repairs stale pins for maintainer-approved system-trusted public WSS rotations while preserving fail-closed behavior for untrusted, private-CA, unavailable-certificate, and store-key-mismatched cases.

## Reproduction Assessment

Yes for source-level reproduction: current main has a unit test where trusted `wss://gateway.example.com` pin mismatches are not auto-repaired, and the TLS delegate cancels mismatched stored pins. A live repro would require the reporter's stale stored macOS app pin plus a rotated public certificate.

## Solution Assessment

Unclear until maintainers approve the TLS trust boundary. The narrow maintainable solution is a macOS stale-pin policy change with regression tests and explicit fail-closed coverage, not a broad bypass of required TLS pinning.
