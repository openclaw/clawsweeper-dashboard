## Summary

Keep open. Current main still excludes system-trusted public WSS hosts from stale TLS pin auto-repair, and the linked closing PR #75228 is open and unmerged, so this should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue reports the macOS app remote-mode node repeatedly disconnecting with `cancelled` on a public WSS gateway while `openclaw node run --tls` connects to the same gateway.

## Best Possible Solution

Review and either land PR #75228 or an equivalent macOS fix that repairs stale pins for system-trusted public WSS hosts while keeping self-signed, private-CA, unavailable, and otherwise untrusted certificate paths fail-closed, with focused Swift coverage and a user-facing changelog entry.

## Reproduction Assessment

Yes, for source-level reproduction: current main asserts a trusted `wss://gateway.example.com` pin mismatch is not auto-repaired, and the TLS delegate cancels mismatched stored pins. A live reproduction would need a stale Keychain pin state or a controlled public certificate rotation.

## Solution Assessment

Unclear until maintainers review the TLS trust boundary. If system trust is the intended repair gate, the linked PR’s direction is narrow; otherwise the safer outcome is to keep the current fail-closed policy and document manual pin reset or Remote over SSH.
