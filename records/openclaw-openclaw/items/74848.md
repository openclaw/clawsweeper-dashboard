## Summary

Keep open: current main and the latest release still reject stale-pin auto-repair for trusted public non-`*.ts.net` WSS hosts, while the linked closing PR remains open and still needs maintainer review on the macOS TLS trust boundary.

## What This Changes

The issue reports that the macOS companion app node repeatedly disconnects from a remote direct WSS gateway with `cancelled` while `openclaw node run --tls` works against the same gateway.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/75228 after its config-contract gaps are fixed, or replace it with an equivalent macOS fix that preserves strict defaults while giving operators an explicit recovery path for trusted public certificate rotations.

## Reproduction Assessment

Yes for source-level reproduction: current main has a test where trusted `wss://gateway.example.com` pin mismatches are not auto-repaired, and the TLS delegate cancels mismatched stored pins. A live reproduction would need the reporter's stale macOS keychain pin plus a rotated public certificate.

## Solution Assessment

Unclear until maintainers accept the TLS trust boundary. The narrow maintainable path is a default-strict macOS stale-pin repair policy with an explicit, documented opt-in and regression coverage, not a broad default bypass of pinning.
