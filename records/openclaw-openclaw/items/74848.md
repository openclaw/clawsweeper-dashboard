## Summary

Keep open: current main still excludes trusted public non-`.ts.net` WSS hosts from stale TLS pin auto-repair, and open PR #75228 references this issue with closing syntax. The remaining fix changes macOS TLS trust/config policy, so cleanup should wait for that PR or an equivalent maintainer-approved change to land.

## What This Changes

The issue asks the macOS app remote node to recover for public WSS gateways after certificate rotation instead of looping with `cancelled` while the CLI node connects successfully.

## Best Possible Solution

Review and land PR #75228 after registering the opt-in in the shared config schema/docs, or choose an equivalent maintainer-approved macOS TLS pin repair policy that preserves fail-closed behavior for untrusted and private-CA cases.

## Reproduction Assessment

Yes for source-level reproduction: current main has a test where a system-trusted `wss://gateway.example.com` pin mismatch is not auto-repaired, and the TLS delegate cancels stored-pin mismatches. A live reproduction would need the reporter's stale stored macOS app pin plus a rotated public certificate.

## Solution Assessment

Unclear until maintainers approve the TLS trust boundary. The narrow maintainable direction appears to be a default-off, schema-valid opt-in with regression tests and explicit fail-closed coverage rather than broad default repair for all system-trusted hosts.
