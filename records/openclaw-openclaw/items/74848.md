## Summary

Keep open. Current main still excludes trusted public `wss://` hosts from stale TLS pin auto-repair, and open PR #75228 references this issue with closing syntax, so cleanup should wait for that PR or an equivalent reviewed fix to land.

## What This Changes

The macOS app remote node should recover from stale TLS certificate pins on public WSS gateways instead of looping with `cancelled` while the CLI node connects.

## Best Possible Solution

Review and land PR #75228 or an equivalent macOS fix that repairs stale pins for system-trusted public WSS certificate rotations while preserving fail-closed behavior for untrusted, private-CA, unavailable, and store-key-mismatched cases.

## Reproduction Assessment

Yes for source-level reproduction: current main has a unit test showing trusted public `wss://gateway.example.com` pin mismatches are not auto-repaired, and the TLS delegate cancels mismatched stored pins. A live reproduction needs a stale stored macOS app pin plus a rotated public certificate on the reporter’s gateway.

## Solution Assessment

Unclear until maintainers approve the TLS trust boundary. The narrow maintainable path is a macOS stale-pin policy change with regression tests and changelog/docs alignment, not a broad bypass of required TLS pinning.
