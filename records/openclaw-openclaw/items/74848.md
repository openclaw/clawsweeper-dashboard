## Summary

Keep open: current main still excludes non-Tailscale public WSS hosts from stale TLS pin auto-repair, while open PR #75228 is the linked implementation candidate and changes a TLS trust boundary that needs maintainer/security review before this issue can close.

## What This Changes

The issue asks for macOS app direct remote WSS node connections to recover from stale TLS pins on public system-trusted hosts instead of reconnecting with `cancelled` while the CLI node works.

## Best Possible Solution

Land PR #75228 or an equivalent macOS fix that repairs stale pins for maintainer-approved system-trusted public WSS rotations while preserving fail-closed behavior for untrusted, private-CA, unavailable-certificate, and store-key-mismatched cases.

## Reproduction Assessment

Yes for source-level reproduction: current main rejects auto-repair for a trusted public WSS pin mismatch and cancels mismatched stored pins. A live reproduction would need the reporter's stale stored macOS app pin plus a rotated public certificate.

## Solution Assessment

Unclear until maintainers approve the TLS trust boundary. The narrow maintainable path is the linked macOS stale-pin policy change with regression tests and explicit fail-closed coverage, not bypassing required TLS pinning broadly.
