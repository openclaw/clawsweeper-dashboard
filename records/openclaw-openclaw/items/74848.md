## Summary

Keep open: current main still excludes public non-`*.ts.net` WSS hosts from stale TLS pin auto-repair, and open PR #75228 references this issue with closing syntax. The remaining change is a macOS TLS trust-boundary decision, so closure should wait for maintainer/security review and merge or rejection of that PR.

## What This Changes

The report asks the macOS app remote node to stop reconnecting with `cancelled` on a system-trusted public WSS gateway when the CLI node works.

## Best Possible Solution

Land PR #75228 or an equivalent macOS stale-pin repair if maintainers accept system-trusted public-host auto-repair, while preserving fail-closed handling for untrusted, private-CA, unavailable-certificate, and store-key-mismatched cases.

## Reproduction Assessment

Yes for source-level reproduction: a stored pin mismatch on `wss://gateway.example.com` with `systemTrustOk: true` is cancelled by the TLS delegate, and current tests assert that public host is not auto-repaired. A live reproduction would need a stale stored macOS app pin and a rotated public certificate.

## Solution Assessment

Unclear until maintainers approve the TLS trust boundary. The narrow maintainable solution is a macOS stale-pin policy change with regression tests and explicit fail-closed coverage, not a broad bypass of TLS pinning.
