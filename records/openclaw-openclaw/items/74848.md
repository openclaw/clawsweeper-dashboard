## Summary

Keep open: current main still refuses stale TLS pin auto-repair for trusted public non-Tailscale WSS hosts, and open PR #75228 references this issue with closing syntax, so cleanup should wait for that reviewed fix or an equivalent maintainer-approved change.

## What This Changes

The issue asks the macOS app node in direct remote WSS mode to recover from repeated `cancelled` disconnects when a CLI node on the same gateway works.

## Best Possible Solution

Land PR #75228 or an equivalent macOS fix that repairs stale pins for maintainer-approved system-trusted public WSS rotations while preserving fail-closed behavior for untrusted, private-CA, unavailable-certificate, and store-key-mismatched cases.

## Reproduction Assessment

Yes for source-level reproduction: current main has a test where trusted `wss://gateway.example.com` pin mismatches are not auto-repaired, and the TLS delegate cancels stored-pin mismatches. A live reproduction would need the reporter's stale stored macOS app pin plus a rotated public certificate.

## Solution Assessment

Unclear until maintainers approve the TLS trust boundary. The narrow maintainable path is PR #75228 or equivalent, with regression tests and fail-closed coverage rather than bypassing required TLS pinning.
