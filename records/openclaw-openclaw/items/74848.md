## Summary

Keep open. Current main still auto-repairs stale macOS Gateway TLS pins only for loopback and Tailscale Serve names, while the reporter’s public WSS host remains in the fail-closed pin-mismatch path; open PR #75228 already references this issue with closing syntax, so cleanup should wait for that review or an equivalent merged fix.

## What This Changes

The issue reports the macOS app remote node repeatedly disconnecting with `cancelled` against a public WSS gateway while `openclaw node run --tls` works from the same environment.

## Best Possible Solution

Review and land PR #75228 or an equivalent macOS fix that repairs stale pins for system-trusted public WSS certificate rotations while preserving fail-closed behavior for untrusted, private-CA, unavailable-certificate, and store-key-mismatched cases.

## Reproduction Assessment

Yes. Current main has a source-level reproduction: `wss://gateway.example.com` with `systemTrustOk: true` is expected not to auto-repair, and the TLS delegate cancels stored-pin mismatches.

## Solution Assessment

Unclear until maintainers approve the TLS trust boundary. The narrow maintainable direction is a macOS stale-pin policy change with regression coverage, not a broad bypass of required pinning.
