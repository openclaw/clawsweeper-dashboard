## Summary

Keep open. Current main still excludes arbitrary public WSS hosts from stale TLS pin auto-repair, and open PR #75228 is an unmerged closing PR for the same macOS-app-vs-CLI stale-pin behavior.

## What This Changes

The issue requests that the macOS app node stay connected to a remote public WSS gateway after a trusted certificate rotation instead of repeatedly disconnecting with a cancelled TLS connection while the CLI node works.

## Best Possible Solution

Review and either land PR #75228 or an equivalent macOS fix that repairs stale pins for system-trusted public WSS hosts, keeps self-signed/private-CA/untrusted mismatches fail-closed, and preserves focused Swift coverage for scheme, store-key, mismatch-kind, and trust-policy guards.

## Reproduction Assessment

Yes, for source-level reproduction: a trusted public wss://gateway.example.com pin mismatch is currently asserted not to auto-repair, and the TLS delegate cancels mismatched stored pins. A live reproduction would need the reporter's stale Keychain pin state or a controlled public certificate rotation.

## Solution Assessment

Unclear until maintainers review the TLS trust boundary in #75228. The proposed direction is narrow if systemTrustOk is accepted as the repair gate; otherwise the safer solution is to keep the current fail-closed policy and document manual pin reset or SSH fallback.
