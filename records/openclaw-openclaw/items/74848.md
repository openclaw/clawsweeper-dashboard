## Summary

Keep open: current main still only auto-repairs stale macOS gateway TLS pins for loopback and `ts.net` hosts, and it explicitly tests that a system-trusted public WSS host is not repaired. The open linked fix at https://github.com/openclaw/openclaw/pull/75228 still targets this issue, while https://github.com/openclaw/openclaw/pull/81609 says this remains a separate stale-pin repair-policy decision.

## What This Changes

The issue asks the macOS app node to recover from stale TLS pins on a public WSS gateway so it does not loop with `cancelled` while the CLI node connects.

## Best Possible Solution

Review and either land the linked opt-in fix after config-contract, documentation, security, and real-proof gaps are resolved, or make an explicit maintainer policy that public WSS hosts must recover through configured fingerprints or Remote over SSH.

## Reproduction Assessment

Yes for source-level reproduction: current main has a trusted public WSS pin-mismatch test that is not auto-repaired, and the TLS delegate cancels stored-pin mismatches. A live repro needs a macOS install with a stale stored pin and a rotated public certificate.

## Solution Assessment

Unclear until maintainers accept the TLS trust boundary. A default-strict, explicit opt-in repair path is safer than broad default public-host self-heal, but it must be registered in the config contract and documented before merge.
