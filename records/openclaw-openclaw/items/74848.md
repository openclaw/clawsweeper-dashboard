## Summary

Keep open. Current main still excludes system-trusted public WSS hosts from stale macOS app-node TLS pin auto-repair, and open PR #75228 is an unmerged closing PR for this exact behavior, so the issue should remain open until that PR or an equivalent security-reviewed fix lands or is rejected.

## What This Changes

The issue reports that the macOS app node repeatedly disconnects with a gateway WSS connection cancelled error while a CLI node works against the same remote gateway.

## Best Possible Solution

Review and either land PR #75228 or an equivalent macOS fix that repairs stale pins for system-trusted WSS hosts, keeps self-signed/private-CA/untrusted certificate mismatches fail-closed, preserves focused Swift coverage, and updates the remote macOS troubleshooting docs if the supported behavior changes.

## Reproduction Assessment

Yes for the source-level behavior: current main has a test fixture where a system-trusted public WSS host still does not auto-repair, while the pinning session cancels mismatched pins. A full live repro still requires the reporter’s stale Keychain pin state or a controlled certificate rotation on a public trusted gateway.

## Solution Assessment

Unclear until the linked PR receives maintainer/security review. The proposed direction is narrow and matches the app-vs-CLI difference if systemTrustOk is accepted as the repair gate; otherwise the safer end state is to keep the current fail-closed policy and document manual pin reset or SSH fallback.
