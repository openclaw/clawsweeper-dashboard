## Summary

Keep open: current main still refuses stale TLS pin auto-repair for system-trusted public non-Tailscale WSS hosts, and open PR #75228 references this issue with closing syntax, so closure should wait for that reviewed fix or an equivalent maintainer-approved change.

## What This Changes

The issue asks for macOS app remote direct WSS node connections to recover from stale TLS pin or certificate rotation instead of repeatedly disconnecting with "cancelled" while the CLI node connects.

## Best Possible Solution

Land PR #75228 or an equivalent macOS fix that repairs stale pins for maintainer-approved system-trusted public WSS rotations while preserving fail-closed behavior for untrusted, private-CA, unavailable-certificate, and store-key-mismatched cases.

## Reproduction Assessment

Yes for source-level reproduction: current main has a `wss://gateway.example.com` + `systemTrustOk: true` unit case that asserts no auto-repair, and the TLS delegate cancels stored-pin mismatches. A live reproduction would need the reporter's stale stored macOS app pin plus a rotated public certificate.

## Solution Assessment

Unclear until maintainers approve the TLS trust boundary. The narrow maintainable solution is a macOS stale-pin policy change with regression tests and explicit fail-closed coverage, not a broad bypass of required TLS pinning.
