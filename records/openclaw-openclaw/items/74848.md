## Summary

Keep open. Current main still excludes system-trusted public WSS hosts from stale TLS pin auto-repair, and the linked closing PR #75228 is open and unmerged, so this issue should stay open until that PR or an equivalent fix lands.

## What This Changes

Fix the macOS companion app remote direct WSS node reconnect loop when a stored gateway TLS leaf pin is stale while the CLI node connects successfully to the same system-trusted gateway.

## Best Possible Solution

Review and either land PR #75228 or an equivalent macOS fix that repairs stale pins for system-trusted public WSS hosts, keeps self-signed, private-CA, unavailable, and otherwise untrusted certificate paths fail-closed, and preserves focused Swift coverage for scheme, store-key, mismatch-kind, trust-policy, and one-shot replacement guards.

## Reproduction Assessment

Yes, for source-level reproduction: a trusted public `wss://gateway.example.com` pin mismatch is currently asserted not to auto-repair, and the TLS delegate cancels mismatched stored pins. A live reproduction would need the reporter's stale Keychain pin state or a controlled public certificate rotation.

## Solution Assessment

Unclear until maintainers review the TLS trust boundary in #75228. If accepting system trust as the repair gate is intended, that PR's shape is narrow; otherwise the safer solution is to keep the current fail-closed policy and document manual pin reset or Remote over SSH.
