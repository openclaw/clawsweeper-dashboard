## Summary

Keep open. Checked-out main still only auto-repairs stale macOS gateway TLS pins for loopback and `*.ts.net` hosts, while open PR #75228 explicitly references this issue with closing syntax and proposes the remaining public-host path.

## What This Changes

The issue reports the macOS companion app repeatedly disconnecting with TLS `cancelled` errors against a public `wss://` gateway while `openclaw node run --tls` works in the same environment.

## Best Possible Solution

Review and land PR #75228 or an equivalent opt-in macOS fix that permits stale-pin repair for system-trusted public `wss://` certificate rotations while keeping default pinning strict and untrusted/self-signed failures closed.

## Reproduction Assessment

Yes. The source-level reproduction is a stored macOS app TLS pin mismatch for `wss://gateway.example.com` with `systemTrustOk: true`, which current tests assert is not auto-repaired; a live reproduction needs the reporter's gateway or any public gateway with a stale stored app pin after certificate rotation.

## Solution Assessment

Unclear until the TLS trust boundary is approved, but the narrow opt-in config direction in PR #75228 is safer than silently repairing every system-trusted public-host pin mismatch by default.
