## Summary

Keep open: current main still excludes public non-`*.ts.net` WSS hosts from stale TLS pin auto-repair, and open PR #75228 references this issue with closing syntax. The remaining path changes the macOS app TLS trust boundary, so closure should wait for maintainer/security review of that PR or an equivalent fix.

## What This Changes

The issue reports that the macOS app remote node loops on TLS `cancelled` disconnects for a public WSS gateway while the CLI node connects successfully.

## Best Possible Solution

Review and land PR #75228 or an equivalent macOS stale-pin policy change if maintainers accept system-trusted public-host repair, while preserving strict fail-closed coverage.

## Reproduction Assessment

Yes for source-level reproduction: current main has an explicit trusted `wss://gateway.example.com` pin mismatch test expecting no auto-repair, and the TLS delegate cancels stored-pin mismatches. I did not live-reproduce because that requires the reporter's stale stored macOS pin and a public certificate rotation.

## Solution Assessment

Unclear until maintainers approve the TLS trust boundary. The narrow maintainable solution is a macOS stale-pin policy change with regression tests and explicit fail-closed coverage, not a broad bypass of required TLS pinning.
