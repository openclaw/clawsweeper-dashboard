## Summary

Keep open. Current main still excludes system-trusted public WSS hosts from stale TLS pin auto-repair, and the linked closing PR #75228 is open and unmerged in the provided GitHub context.

## What This Changes

The issue requests fixing macOS app remote direct mode so a stale TLS pin on a public WSS gateway does not leave the native Mac node reconnecting with `cancelled` while the CLI node works.

## Best Possible Solution

Review and land PR #75228 or an equivalent macOS fix that repairs stale pins for system-trusted public WSS hosts while keeping self-signed, private-CA, unavailable, and otherwise untrusted certificate paths fail-closed with focused Swift coverage for scheme, store-key, mismatch-kind, trust-policy, and one-shot replacement guards.

## Reproduction Assessment

Yes, for source-level reproduction: current tests assert that a trusted public `wss://gateway.example.com` pin mismatch is not auto-repaired, and the TLS delegate cancels mismatched stored pins. A live reproduction would require the reporter's stale pin state or a controlled public certificate rotation.

## Solution Assessment

Unclear until maintainers approve the TLS trust boundary. If accepting system trust as the repair gate is intended, PR #75228 is the narrowest visible fix; otherwise the safer outcome is to keep the current fail-closed policy and document manual pin reset or Remote over SSH.
