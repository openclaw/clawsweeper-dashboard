## Summary

Keep open. Current main still only auto-repairs stale macOS gateway TLS pins for loopback and `*.ts.net` hosts, while the report is about a system-trusted public `wss://` host. The linked closing PR #75228 is open and unmerged, so this issue should remain open until that PR or an equivalent maintainer-approved fix lands.

## What This Changes

The issue reports that the macOS app node repeatedly disconnects with a gateway WebSocket `cancelled` error on a remote public WSS endpoint while `openclaw node run --tls` works in the same environment.

## Best Possible Solution

Review and either land PR #75228 or an equivalent macOS fix that repairs stale pins for system-trusted public WSS certificate rotation while preserving fail-closed behavior for self-signed, private-CA, unavailable, non-WSS, store-key-mismatched, and repeated-repair cases.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: current main asserts that a `wss://gateway.example.com` pin mismatch with `systemTrustOk: true` is not auto-repaired, and the TLS delegate cancels mismatched stored pins. A live reproduction would require the reporter's stale stored pin or a controlled public certificate rotation.

## Solution Assessment

Unclear until maintainer/security review of PR #75228. If system trust is the intended boundary for public certificate rotation, the linked PR is the narrowest apparent fix; otherwise the safer product path is to keep the current fail-closed policy and document manual pin reset or Remote over SSH.
