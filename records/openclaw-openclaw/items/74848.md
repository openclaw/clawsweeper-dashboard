## Summary

Keep open. Current main still leaves trusted public `wss://` hosts outside stale TLS pin auto-repair, and open PR #75228 references this issue with closing syntax, so the issue should remain open until that or an equivalent reviewed fix lands.

## What This Changes

Fix the macOS app remote-direct node so a stale stored TLS pin for a rotated public WSS gateway certificate does not leave the Mac node reconnecting with `cancelled` while the CLI node works.

## Best Possible Solution

Review and land PR #75228 or an equivalent macOS fix that permits stale-pin repair for system-trusted public WSS certificate rotations while preserving fail-closed handling for self-signed, private-CA, untrusted, unavailable, and store-key-mismatched cases.

## Reproduction Assessment

Yes for source-level reproduction: current tests assert that a trusted public `wss://gateway.example.com` pin mismatch is not auto-repaired, and the TLS delegate cancels mismatched stored pins. A live reproduction needs a stale stored app pin plus a rotated public certificate or the reporter's original gateway environment.

## Solution Assessment

Unclear until maintainers approve the TLS trust boundary. The narrow maintainable shape is in the macOS coordinator's stale-pin policy with regression tests and docs/changelog alignment, not a broad bypass of required TLS pinning.

## Security Review

Status: needs_attention

Summary: The report and linked fix path affect TLS pin auto-repair for public WSS hosts, so maintainer/security review is required before widening the policy.

Concerns:

- **[medium] Confirm public-host stale-pin repair policy:** `apps/macos/Sources/OpenClaw/NodeMode/MacNodeModeCoordinator.swift:204`
  - body: Current main only repairs stale pins for loopback and Tailscale Serve hosts. Extending repair to any system-trusted public host changes the pinning threat model, especially on devices with user, admin, MDM, or interception roots.
