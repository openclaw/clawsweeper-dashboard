## Summary

Keep open: current main still excludes public non-`*.ts.net` WSS hosts from stale TLS pin auto-repair, and open PR #75228 references this issue with closing syntax. The remaining path changes the macOS TLS trust boundary, so closure should wait for maintainer/security review of that PR or an equivalent fix.

## What This Changes

The issue requests that the macOS app remote node stay connected to a public WSS gateway after certificate rotation instead of failing pin validation with "cancelled" while the CLI node works.

## Best Possible Solution

Review and land PR #75228 or an equivalent macOS fix if maintainers accept system-trusted public-host stale-pin repair, while preserving strict rejection for untrusted, private-CA, unavailable-certificate, non-WSS, and store-key-mismatched cases.

## Reproduction Assessment

Yes for source-level reproduction: current main has a test where a trusted `wss://gateway.example.com` pin mismatch is not auto-repaired, and the TLS delegate cancels mismatched stored pins. A live reproduction would need a stale stored macOS app pin plus a rotated public certificate.

## Solution Assessment

Unclear until maintainers approve the TLS trust boundary. The narrow maintainable solution is a macOS stale-pin policy change with regression tests and explicit fail-closed coverage, not a broad bypass of required TLS pinning.
