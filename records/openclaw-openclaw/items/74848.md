## Summary

Keep open. Current main still excludes non-Tailscale public `wss://` hosts from macOS stale TLS pin auto-repair, and open PR #75228 references this issue with closing syntax, so the issue should remain open until that PR is reviewed, merged, replaced, or rejected.

## What This Changes

The report asks for the macOS app node in remote direct WSS mode to stop looping on `cancelled` after a gateway certificate change when a CLI node can connect to the same host.

## Best Possible Solution

Review and land PR #75228 or an equivalent macOS fix that auto-repairs stale pins for system-trusted public WSS certificate rotations while preserving fail-closed behavior for untrusted, self-signed, private-CA, wrong-scheme, wrong-store-key, and repeated-repair cases.

## Reproduction Assessment

Yes for the source-level failure path: a stored pin mismatch on a system-trusted public `wss://gateway.example.com` still returns false in the current auto-repair helper, and the TLS session cancels mismatched challenges. A full live repro would require the reporter's stale Keychain pin state or a controlled public certificate rotation.

## Solution Assessment

Unclear until maintainers review the TLS boundary in #75228. The proposed direction is narrow and matches the app-vs-CLI difference, but the safer fallback is to keep the current fail-closed behavior and document manual pin reset or SSH use for non-Tailscale public hosts.
