## Summary

Keep open: current main still excludes public non-`*.ts.net` WSS hosts from stale TLS pin auto-repair, and open PR #75228 references this issue with closing syntax. The remaining fix path changes the macOS app TLS trust boundary, so cleanup should wait for maintainer/security review of that PR or an equivalent change.

## What This Changes

The report asks the macOS app remote-mode node to stay connected to a valid public WSS gateway after certificate rotation or stale pin mismatch, matching the CLI node path that currently works.

## Best Possible Solution

Review and land PR #75228 or an equivalent macOS fix if maintainers accept system-trusted public-host stale-pin repair, while preserving fail-closed behavior for untrusted, private-CA, unavailable-certificate, and store-key-mismatched cases.

## Reproduction Assessment

Yes for source-level reproduction: current main has a test where trusted `wss://gateway.example.com` pin mismatches are not auto-repaired, and the TLS delegate cancels mismatched stored pins. A live reproduction would need the reporter's stale stored macOS app pin plus a rotated public certificate.

## Solution Assessment

Unclear until maintainers approve the TLS trust boundary. The narrow maintainable solution is a macOS stale-pin policy change with regression tests and explicit fail-closed coverage, not a broad bypass of required TLS pinning.
