## Summary

Keep open. Current main still matches the reported macOS companion failure mode: required WSS TLS pinning cancels mismatched stored pins, and stale-pin auto-repair remains limited to loopback and Tailscale Serve hosts while a trusted public WSS host is explicitly covered as not repaired. PR #75228 is open, unmerged, and references this issue with closing syntax, so the issue should remain open until that or an equivalent fix lands.

## What This Changes

The reporter asks for the macOS app remote direct WSS node connection to stop reconnect-looping with `cancelled` when the CLI node can connect to the same gateway.

## Best Possible Solution

Review and land PR #75228 or an equivalent macOS fix that gives public ACME-style WSS gateways an explicit, documented stale-pin repair path while keeping self-signed, private-CA, unavailable, store-key-mismatched, and untrusted certificate paths fail-closed. If the opt-in config flag is retained, align it with the repo's config schema/help/docs contract before release.

## Reproduction Assessment

Yes for source-level reproduction: current tests assert that a trusted public `wss://gateway.example.com` pin mismatch is not auto-repaired, and the TLS delegate cancels mismatched stored pins. A live reproduction needs a stale stored app pin plus a rotated public certificate or the reporter's original environment.

## Solution Assessment

Unclear until maintainers approve the TLS trust boundary. PR #75228's current opt-in system-trusted fallback is a safer direction than silent broad repair, but it still needs maintainer/security review and config/docs alignment before this issue should close.
