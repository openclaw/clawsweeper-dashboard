## Summary

Keep open: current main still excludes public non-`*.ts.net` WSS hosts from stale TLS pin auto-repair, and open PR #75228 references this issue with closing syntax. The remaining path changes the macOS TLS trust boundary, so cleanup should wait for maintainer/security review of that PR or an equivalent fix.

## What This Changes

The issue reports the macOS app remote direct WSS node reconnecting with `cancelled` after a stale TLS pin/certificate rotation while the CLI node connects to the same gateway.

## Best Possible Solution

Review and land PR #75228 or an equivalent macOS fix if maintainers accept system-trusted public-host stale-pin repair, while preserving fail-closed behavior for untrusted, private-CA, unavailable-certificate, and store-key-mismatched cases.

## Reproduction Assessment

Yes for source-level reproduction: current main still rejects auto-repair for a trusted `wss://gateway.example.com` pin mismatch while the TLS delegate cancels stored-pin mismatches. A live repro would need the reporter's stale stored macOS app pin plus a rotated public certificate.

## Solution Assessment

Unclear until maintainers approve the TLS trust boundary. The narrow maintainable solution is a macOS stale-pin policy change with regression tests and explicit fail-closed coverage, not disabling TLS pinning or bypassing certificate validation.
