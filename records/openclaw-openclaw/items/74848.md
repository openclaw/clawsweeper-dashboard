## Summary

Keep open: current main still rejects stale TLS-pin auto-repair for trusted public non-`.ts.net` WSS hosts, and the open closing PR remains the active implementation path rather than a merged fix.

## What This Changes

The issue asks the macOS App remote node to recover from a stale public WSS gateway TLS pin instead of repeatedly disconnecting with `cancelled` while the CLI node connects.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/75228 after the config-contract gaps and TLS policy concerns are resolved, or replace it with an equivalent default-strict macOS recovery path.

## Reproduction Assessment

Yes for source-level reproduction: current main rejects a trusted public `wss://gateway.example.com` pin mismatch in the focused macOS test, and the TLS delegate cancels stored-pin mismatches. A live repro would require the reporter's stale macOS Keychain pin plus a rotated public certificate.

## Solution Assessment

Unclear until maintainers accept the TLS trust boundary. The narrow maintainable direction is a default-strict macOS stale-pin recovery path with an explicit documented opt-in and regression coverage, not broad default repair for every system-trusted public host.
