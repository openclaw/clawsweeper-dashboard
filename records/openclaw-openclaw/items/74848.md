## Summary

Keep open. Current main still rejects stale TLS pin auto-repair for trusted non-loopback, non-`.ts.net` public `wss://` hosts, and open PR #75228 references this issue with closing syntax, so cleanup should wait for that PR or an equivalent reviewed fix to land.

## What This Changes

The issue requests that the macOS app node stay connected to a direct remote `wss://` gateway after a stale TLS pin or certificate rotation instead of looping with `cancelled` while the CLI node works.

## Best Possible Solution

Review and land PR #75228 or an equivalent macOS fix that keeps default pinning strict, adds any arbitrary-host repair as an explicit schema-valid opt-in, and preserves fail-closed handling for untrusted chains and store-key mismatches.

## Reproduction Assessment

Yes for source-level reproduction: current main has a unit test proving trusted `wss://gateway.example.com` pin mismatches are not auto-repaired, and the TLS delegate cancels mismatched stored pins. A live reproduction needs a stale stored macOS app pin plus a rotated system-trusted certificate on a non-`.ts.net` public gateway.

## Solution Assessment

Unclear until the linked PR is finished and reviewed. The maintainable direction is a narrow macOS stale-pin policy change with config-schema/docs/tests and security review, not a broad default bypass of TLS pinning.
