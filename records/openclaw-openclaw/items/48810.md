## Summary

Keep this open. Current main still treats transcripts as a tree and active readers follow the latest leaf branch, but I did not find a compaction-retry reconciliation path that supersedes, deterministically replaces, or marks duplicate same-parent compaction entries; the remaining question is a transcript contract/product decision, not a safe cleanup close.

## What This Changes

The issue requests preventing compaction retries from creating duplicate same-parent compaction entries that leave one raw JSONL branch orphaned during parentId reconstruction.

## Best Possible Solution

Maintainers should decide the transcript contract explicitly, then either implement compaction-retry idempotency/supersession at the Pi/OpenClaw boundary or document and expose active-branch readers so downstream tools do not treat raw parentId as a linear list.

## Reproduction Assessment

Unclear. The issue gives concrete raw JSONL showing the duplicate same-parent compaction shape, but the reporter could not reproduce it and I did not establish a current-main live retry path in this read-only review.

## Solution Assessment

Unclear. Superseding or deterministic replacement may be the right fix if raw JSONL must remain linear, but current main's documented tree model and active-branch readers make this a maintainer contract decision first.
