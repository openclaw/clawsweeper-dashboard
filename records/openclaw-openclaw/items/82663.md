## Summary

The PR appears to address a real current-main bug in `loadSessionLogs` without introducing a code-level regression, but it should stay open because the external contributor only provided a unit-test command and the required after-fix real behavior proof is missing.

## What This Changes

The PR normalizes explicit `loadSessionLogs` limits so zero, negative, and non-finite values return no session log entries, with regression coverage for those cases.

## Best Possible Solution

Land this focused normalization after the contributor adds real after-fix evidence showing `loadSessionLogs` returns no entries for zero or invalid limits in a real run.

## Reproduction Assessment

Yes from source inspection: current main passes `limit: 0` through to `sortedLogs.slice(-limit)`, and JavaScript treats `slice(-0)` as `slice(0)`, so a transcript with messages would return all logs instead of none. I did not run the test because this review is read-only and local test commands can write artifacts in this checkout.

## Solution Assessment

Yes, the proposed code is the narrow maintainable fix: normalize explicit limits at the `loadSessionLogs` boundary, preserve the default for `undefined`, and return early for zero-result limits before reading and sorting the log file.
