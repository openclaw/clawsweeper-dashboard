The PR changes exec approval prompt copy so unavailable Allow Always actions can explain one-shot/non-persistable commands instead of always blaming ask=always policy.

## Best Possible Solution

Carry the unavailable reason or ask value through every approval renderer path so one-shot prompts and ask=always prompts remain distinguishable across core and Telegram surfaces.

## Reproduction Assessment

Yes from source: current and patched renderer paths can be traced from omitted ask plus precomputed decisions to the unavailable reason text. I did not run a live gateway or Telegram repro in this read-only review.

## Solution Assessment

No; deriving the reason only from ask is too narrow because some existing callers provide allowed decisions without ask. The maintainable fix should carry the reason explicitly or pass ask through all renderer callers.
