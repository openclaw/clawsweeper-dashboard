## Summary

Keep open: current main intentionally preserves the origin/external route when a webchat/control UI turn views an existing channel session, but the session tools still do not expose a clear active-vs-origin channel distinction for agents.

## What This Changes

The issue asks OpenClaw to make cross-channel sessions unambiguous by documenting or exposing origin-channel and active-channel metadata in `session_status` and `sessions_list`.

## Best Possible Solution

Keep stable origin-based session keys and external route preservation, but add explicit additive metadata and docs that separate origin/stored delivery channel from the active inbound channel.

## Reproduction Assessment

Yes, at source level: current main preserves the external/stored route for webchat views while `session_status` and `sessions_list` do not expose a distinct active/current channel field. I did not run a live Discord-to-webchat E2E.

## Solution Assessment

Yes: an additive tool-output and docs change is the maintainable path; changing session-key semantics or overwriting preserved routes would regress existing delivery behavior.

## Repair Work Prompt

Fix the session-origin/current-channel ambiguity reported in https://github.com/openclaw/openclaw/issues/84544. Add additive metadata to `session_status` and `sessions_list` so agents can distinguish the stable origin/stored delivery channel from the current inbound channel when a Discord/Telegram/etc. session is continued through webchat/control UI. Do not change session key generation, do not overwrite preserved external `lastChannel`/`deliveryContext`, and do not regress the route-preservation behavior from #47745/#58013.
