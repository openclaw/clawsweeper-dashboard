## What This Changes

## Best Possible Solution

Split or explicitly repair the two surfaces: land preflight compaction under a defined cancellation contract with real runtime proof, and merge shared MCP runtime only after the session-key lifecycle bug is fixed and maintainers accept the opt-in isolation boundary.

- **[P2] Preserve explicit preflight cancellation:** `src/auto-reply/reply/agent-runner-memory.ts:981`
  - body: By dropping `abortSignal` entirely, required preflight compaction no longer observes explicit user aborts or restarts from `ReplyOperation`; a canceled/restarted turn can keep compacting until `compaction.timeoutSeconds` and hold the session path. Use a preflight signal that excludes the reply lifecycle timeout but still preserves deliberate cancellation, or document and test the new cancellation policy.
  - confidence: 0.84
- **[P2] Rebind session keys after runtime retarget:** `src/agents/agent-bundle-mcp-runtime.ts:1080-1085`
  - body: `sessionIdBySessionKey` is set before detaching the old runtime key, but `detachSessionFromRuntimeKey` calls `forgetSessionKeysForSessionId`, deleting that just-written mapping. When the same session is recreated after an MCP config/workspace/scope change, `retireSessionMcpRuntimeForSessionKey` can no longer resolve the runtime, so run-end or subagent cleanup can miss the MCP process until fallback cleanup or idle sweep.
  - confidence: 0.88
