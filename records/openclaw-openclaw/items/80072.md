## Summary

Keep open: current main still has the route gap in source. The root-mounted handler special-cases nested `/assets/` requests, but root-level public files under the same prefix resolve to non-existent nested files and then hit the static-asset 404 path, matching the report.

## What This Changes

The issue asks for Control UI favicon, touch icon, and manifest files to load under the prefixed Control UI path instead of returning 404.

## Best Possible Solution

Add a narrow gateway-side mapping for the known shipped root Control UI public assets under SPA route prefixes while keeping unknown static paths and unsafe files rejected.

## Reproduction Assessment

Yes, source-level. With the default empty base path, `/__openclaw__/favicon.svg` maps to `__openclaw__/favicon.svg` and hits the static 404 branch, while `/__openclaw__/assets/...` is explicitly remapped to `assets/...`.

## Solution Assessment

Yes, a focused routing fix in the Control UI static handler is the narrowest maintainable path. Avoid changing the public base-path config model or weakening the SPA/static safety checks.
