## Summary

This issue should stay open: current main still has a source-level path where root-mounted Control UI requests for `/__openclaw__/favicon.svg` and similar top-level public assets are resolved as nested files, while `/__openclaw__/assets/...` gets special handling.

## What This Changes

Serve root-level Control UI public assets correctly when the SPA is reached under `/__openclaw__/` so favicons and the web manifest do not 404.

## Best Possible Solution

Keep the Control UI static-serving contract aligned with its built asset URLs by making top-level public assets resolve under nested SPA prefixes without weakening the existing path-safety checks.

## Reproduction Assessment

Yes, source-reproducible. With the default root-mounted Control UI, `/__openclaw__/favicon.svg` is served by the SPA handler, resolves to `__openclaw__/favicon.svg` under the asset root, and returns 404 even when `favicon.svg` exists at the root.

## Solution Assessment

Yes. The narrow fix belongs in the Control UI static asset resolver or built asset URL contract, with regression coverage for root-level public assets under nested SPA paths while preserving the existing symlink/hardlink safety behavior.
