## Summary

Keep open: the force-push fixes the earlier null-classification bug, but the new helper can still retry over non-text visible payloads, and the external PR only provides unit-test proof rather than real exhausted-CLI fallback proof.

## What This Changes

The PR changes the embedded-agent fallback classifier and tests so a sole generic external runner failure text payload returns a `format` / `external_runner_failure` fallback classification.

## Best Possible Solution

Tighten the helper so every non-error, non-reasoning visible payload is exactly the generic failure text, then require contributor-owned real fallback proof before merge.

## Reproduction Assessment

Yes, at source level: current main wraps CLI output text as a normal payload and the fallback classifier returns `null` on visible non-error payloads before the fallback engine can advance. I did not run a live exhausted Claude Code subscription.

## Solution Assessment

No, not as-is. The classifier boundary is the right place, but the helper should match the full visible-payload contract instead of checking only text-bearing visible payloads.
