## Summary

Keep this PR open. Current main still performs the eager `sessions.list` reload, so the PR is not obsolete, but the submitted patch needs changes before merge because it defers `loadSessions` without updating the existing synchronous test and introduces a timeout that is not cleared during Control UI teardown.

## What This Changes

The PR adds Control UI timer state and changes `sessions.changed` handling so the incremental session update remains immediate while the full `sessions.list` reconciliation is delayed behind a 5-second debounce.

## Best Possible Solution

Request changes on this PR: keep the incremental `sessions.changed` update immediate, debounce only the full `sessions.list` reconciliation, clear and null the pending timer during Control UI teardown, add fake-timer coverage for delayed and coalesced reloads, initialize the test fixture state, and add the active-version changelog entry if this remains classified as a user-facing Control UI performance fix.

## Reproduction Assessment

Yes. Current main can be verified from source: `handleGatewayEventUnsafe` immediately calls `loadSessions` on `sessions.changed`, and the existing unit test asserts that synchronous behavior. The PR-side regression is also reproducible by inspection because the call moves behind `setTimeout` while that test remains unchanged.

## Solution Assessment

No, not as submitted. The debounce direction is a narrow maintainable fix for the reported request fan-out, but the patch needs lifecycle cleanup, fake-timer tests, and changelog handling before it is the best implementation.
