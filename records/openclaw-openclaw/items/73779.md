## Summary

Keep this PR open. Current main still reloads `sessions.list` immediately for every `sessions.changed` event, and the PR targets the right Control UI path, but the submitted patch is not merge-ready because it leaves the existing synchronous test stale, does not clear the new timeout during teardown, and lacks the required user-facing changelog entry.

## What This Changes

The PR adds Control UI host timer state and changes `sessions.changed` handling to debounce the full `sessions.list` reload for five seconds after the last push event.

## Best Possible Solution

Repair this PR by keeping incremental session updates immediate, debouncing only the full reconciliation reload, clearing pending timers during teardown, adding fake-timer coverage for delayed and coalesced reloads, and adding the active changelog entry.

## Reproduction Assessment

Yes. Dispatching a `sessions.changed` event into `handleGatewayEvent` on current main reaches `applySessionsChangedEvent` and then calls `loadSessions` synchronously, and the existing unit test asserts that immediate reload behavior.

## Solution Assessment

No, not as submitted. The debounce direction is a narrow maintainable fix for the fanout, but the patch needs lifecycle cleanup, updated tests, fixture state, and changelog handling before merge.
