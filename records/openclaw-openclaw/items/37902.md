## Summary

Keep open: current main and the latest release still let an explicit native `sessions_spawn.runTimeoutSeconds` undercut `agents.defaults.subagents.runTimeoutSeconds`, and all directly related minimum-floor PRs are closed unmerged.

## What This Changes

The issue requests an opt-in `sessions_spawn` timeout floor so model-supplied `runTimeoutSeconds` values cannot undercut operator config.

## Best Possible Solution

Add an opt-in `agents.defaults.subagents.minRunTimeoutSeconds` floor across resolver, config type/schema, docs, tests, and changelog while preserving current behavior when the new key is unset.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: configure `agents.defaults.subagents.runTimeoutSeconds: 120` and pass explicit `runTimeoutSeconds: 2`; the current resolver returns `2`, and a focused current test asserts that behavior.

## Solution Assessment

Yes. An additive optional minimum-floor config is the narrowest maintainable direction; a lock mode or changing the existing default key into a floor would be broader and more compatibility-sensitive.
