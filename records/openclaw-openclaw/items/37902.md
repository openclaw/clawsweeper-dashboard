## Summary

Keep open. Current main and the latest release still implement `agents.defaults.subagents.runTimeoutSeconds` as a fallback default only; an explicit `sessions_spawn.runTimeoutSeconds` still wins, and no `minRunTimeoutSeconds` or lock-style config has landed.

## What This Changes

Add an operator-configured minimum subagent run timeout so model-provided `sessions_spawn.runTimeoutSeconds` values cannot undercut the configured floor.

## Best Possible Solution

Add a backward-compatible optional `agents.defaults.subagents.minRunTimeoutSeconds` floor for native subagent spawns, preserve existing behavior when the option is unset, and document/test the zero-timeout semantics.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: configure `agents.defaults.subagents.runTimeoutSeconds: 120` and pass explicit `runTimeoutSeconds: 2`; the current resolver returns `2`, and the focused unit test asserts that behavior. I did not execute a live model retry loop.

## Solution Assessment

Yes, the optional minimum floor is the narrowest maintainable direction for the reported timeout erosion because it preserves released fallback semantics when unset. A full lock mode would be broader and should remain out of scope unless maintainers explicitly want authoritative config behavior.
