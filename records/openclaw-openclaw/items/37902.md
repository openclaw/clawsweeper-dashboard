## Summary

Keep open. Current main still resolves `sessions_spawn.runTimeoutSeconds` as an explicit model override before the configured default, the public config type/schema/docs expose only `runTimeoutSeconds`, and all direct `minRunTimeoutSeconds` fix PRs are closed unmerged.

## What This Changes

Request an operator-configurable minimum or lock for subagent `runTimeoutSeconds` so model-supplied shorter values cannot undercut the configured timeout.

## Best Possible Solution

Add a backward-compatible optional `agents.defaults.subagents.minRunTimeoutSeconds` floor for native subagent spawns, with schema/types/docs/changelog alignment and regression coverage for omitted, explicit lower, explicit higher, and zero-timeout cases.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: with configured default `120` and explicit `runTimeoutSeconds: 2`, the current resolver returns `2`, and the current unit test asserts that behavior. I did not execute the test because this review is read-only.

## Solution Assessment

Yes. A new optional minimum floor is the narrowest maintainable fix; making the existing default authoritative or adding a lock mode would change released semantics more broadly.
