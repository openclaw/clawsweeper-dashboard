## Summary

Keep open: current main and v2026.5.12 still let an explicit `sessions_spawn.runTimeoutSeconds` undercut the configured default, and the requested minimum/lock config has not landed. The related fix PRs were closed unmerged, while the earlier shipped work only added fallback-default behavior.

## What This Changes

The issue requests a subagent timeout floor or lock so model-supplied `sessions_spawn.runTimeoutSeconds` cannot undercut configured defaults.

## Best Possible Solution

Add a backward-compatible optional `agents.defaults.subagents.minRunTimeoutSeconds` floor for native `sessions_spawn`, preserving existing fallback semantics when unset and documenting the chosen zero-timeout behavior.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: configure `agents.defaults.subagents.runTimeoutSeconds: 120` and pass explicit `runTimeoutSeconds: 2`; the current resolver returns `2`, and the focused test asserts that behavior. I did not execute a live model retry loop.

## Solution Assessment

Yes, an optional minimum floor is the narrowest maintainable direction because it preserves released fallback behavior when unset. A lock mode or changing `runTimeoutSeconds` itself into a floor would be broader compatibility-sensitive behavior.
