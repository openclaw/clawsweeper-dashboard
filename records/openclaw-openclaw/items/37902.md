## Summary

Keep open. Current main still gives an explicit `sessions_spawn.runTimeoutSeconds` precedence over the configured subagent default, the public config/schema/docs do not expose a floor or lock option, and the two direct `minRunTimeoutSeconds` implementation attempts are closed unmerged.

## What This Changes

The issue requests a subagent timeout floor or lock so model-provided `sessions_spawn.runTimeoutSeconds` values cannot undercut operator configuration.

## Best Possible Solution

Add a backward-compatible optional `agents.defaults.subagents.minRunTimeoutSeconds` floor for native subagent spawns, with explicit `0` semantics, schema/type/docs/changelog updates, and regression coverage for explicit model values below the floor.

## Reproduction Assessment

Yes. Source inspection on current main shows the resolver returns an explicit model-provided timeout before the configured default, and current tests assert that precedence.

## Solution Assessment

Yes, with one contract decision. An optional minimum floor is narrower and more backward-compatible than making the existing default key authoritative, but the implementation must define whether explicit `0` is clamped, rejected, or documented as an escape hatch.
