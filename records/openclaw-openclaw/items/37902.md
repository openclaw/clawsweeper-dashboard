## Summary

Keep open: current main and the latest tagged release still let an explicit `sessions_spawn.runTimeoutSeconds` undercut `agents.defaults.subagents.runTimeoutSeconds`, there is no `minRunTimeoutSeconds` or lock-style config surface, and the linked repair PRs were all closed unmerged.

## What This Changes

The issue requests a configurable subagent timeout floor or lock so model-supplied `sessions_spawn.runTimeoutSeconds` values cannot undercut operator defaults.

## Best Possible Solution

Add a backward-compatible optional `agents.defaults.subagents.minRunTimeoutSeconds` floor for native `sessions_spawn`, preserve current behavior when unset, and document/test zero-timeout semantics.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: configure `agents.defaults.subagents.runTimeoutSeconds: 120` and pass explicit `runTimeoutSeconds: 2`; the current resolver returns `2`, and a focused current test asserts that behavior. I did not execute a live retry loop.

## Solution Assessment

Yes, an additive optional minimum-floor config is the narrowest maintainable direction because it preserves released fallback semantics when unset. A lock mode or changing `runTimeoutSeconds` itself into a floor would be broader and more compatibility-sensitive.
