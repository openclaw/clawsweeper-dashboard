## Summary

Keep open: current main and latest release still treat `agents.defaults.subagents.runTimeoutSeconds` as a fallback default only, no `minRunTimeoutSeconds` or lock config exists, and all linked fix PRs were closed unmerged.

## What This Changes

Requests a config-level minimum or lock for `sessions_spawn.runTimeoutSeconds` so model-provided values cannot reduce subagent run timeouts below operator intent.

## Best Possible Solution

Add a backward-compatible optional `agents.defaults.subagents.minRunTimeoutSeconds` floor for native `sessions_spawn`, preserve current behavior when unset, and document/test the chosen zero-timeout semantics.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: configure `agents.defaults.subagents.runTimeoutSeconds: 120` and pass explicit `runTimeoutSeconds: 2`; the current resolver returns `2`, and the focused test asserts that behavior. I did not execute a live retry loop.

## Solution Assessment

Yes, an additive optional minimum-floor config is the narrowest maintainable direction because it preserves released fallback semantics when unset. A lock mode or changing `runTimeoutSeconds` itself into a floor would be broader and more compatibility-sensitive.
