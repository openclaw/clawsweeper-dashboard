## Summary

Keep open: current main and the latest tagged release still treat `agents.defaults.subagents.runTimeoutSeconds` as a fallback default only, explicit `sessions_spawn.runTimeoutSeconds` still wins, and all directly related minimum-floor PRs are closed unmerged.

## What This Changes

The issue requests a config-enforced minimum or lock for native `sessions_spawn.runTimeoutSeconds` so model-supplied lower timeout values cannot undercut operator defaults.

## Best Possible Solution

Add a backward-compatible optional `agents.defaults.subagents.minRunTimeoutSeconds` floor for native `sessions_spawn`, preserve current behavior when unset, and document/test zero-timeout semantics.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: configure `agents.defaults.subagents.runTimeoutSeconds: 120` and pass explicit `runTimeoutSeconds: 2`; the current resolver returns `2`, and a focused current test asserts that behavior. I did not execute a live retry loop in this read-only review.

## Solution Assessment

Yes, an additive optional minimum-floor config is the narrowest maintainable direction because it preserves released fallback semantics when unset. A lock mode or changing `runTimeoutSeconds` itself into a floor would be broader and more compatibility-sensitive.
