## Summary

Keep open: current main and v2026.5.12 still treat configured subagent `runTimeoutSeconds` as a fallback default only, and the requested minimum/lock config has not landed.

## What This Changes

Request an operator-enforced minimum or lock for `sessions_spawn.runTimeoutSeconds` so model-supplied retry values cannot undercut configured subagent timeouts.

## Best Possible Solution

Add a backward-compatible optional `agents.defaults.subagents.minRunTimeoutSeconds` floor for native subagent spawns, keeping existing fallback semantics when unset and documenting the zero-timeout behavior.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: configure `agents.defaults.subagents.runTimeoutSeconds: 120` and pass explicit `runTimeoutSeconds: 2`; the current resolver returns `2` and the focused test asserts that behavior. I did not execute a live model retry loop.

## Solution Assessment

Yes, an optional minimum floor is the narrowest maintainable direction because it preserves released fallback behavior when unset. A full lock mode would be broader and should remain out of scope unless maintainers explicitly want authoritative config behavior.
