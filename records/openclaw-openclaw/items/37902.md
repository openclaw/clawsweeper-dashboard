## Summary

Keep open. Current main still treats `agents.defaults.subagents.runTimeoutSeconds` as a fallback-only default, the model-provided `sessions_spawn.runTimeoutSeconds` still wins, and the direct `minRunTimeoutSeconds` fix attempts are closed unmerged.

## What This Changes

Request adding a minimum or locked subagent run-timeout config so model-provided `sessions_spawn.runTimeoutSeconds` cannot undercut operator policy.

## Best Possible Solution

Add a backward-compatible optional `agents.defaults.subagents.minRunTimeoutSeconds` floor for native subagent spawns, with aligned config schema/types/docs/changelog and regression coverage.

## Reproduction Assessment

Yes. Source inspection on current main shows configured `120` plus explicit `runTimeoutSeconds: 2` resolves to `2`, and the current unit test asserts that precedence.

## Solution Assessment

Yes. An optional minimum floor is narrower and more backward-compatible than changing the existing default key to be authoritative; the implementation should intentionally clamp or document explicit `0` behavior.
