## Summary

Keep open. Current main still treats `agents.defaults.subagents.runTimeoutSeconds` as a fallback only, lacks a `minRunTimeoutSeconds` or lock option, and the related fix PRs are closed unmerged.

## What This Changes

Request to add a subagent timeout floor or lock so model-supplied `sessions_spawn` timeouts cannot undercut configured policy.

## Best Possible Solution

Add a backward-compatible optional `agents.defaults.subagents.minRunTimeoutSeconds` floor in the native subagent timeout resolver, with config schema/docs and regression coverage for omitted, below-floor, above-floor, and zero/unset cases.

## Reproduction Assessment

Yes. Source-level reproduction is high-confidence: with config default `120` and explicit `runTimeoutSeconds: 2`, the current resolver returns `2`, and the test suite asserts that precedence.

## Solution Assessment

Yes. An optional minimum floor is the narrowest maintainable direction shown by the report and prior PRs; making the configured default fully authoritative would be more disruptive.

## Repair Work Prompt

Fix this issue by adding a backward-compatible optional `agents.defaults.subagents.minRunTimeoutSeconds` config key for native `sessions_spawn` subagents. Apply the floor in `resolveConfiguredSubagentRunTimeoutSeconds` or the immediately adjacent native subagent timeout planning path so model-provided positive timeouts below the configured minimum are clamped upward, while existing behavior is preserved when the key is unset or zero. Preserve `runTimeoutSeconds: 0` as no timeout unless the codebase already establishes different semantics. Update config types, zod schema, generated config schema/help as required, docs for subagents/config tools, focused regression tests, and `CHANGELOG.md`. Do not make `runTimeoutSeconds` fully authoritative or change ACP timeout behavior unless the existing code path requires shared helper coverage.
