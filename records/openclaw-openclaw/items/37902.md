## Summary

Keep open. Current main still treats a finite `sessions_spawn.runTimeoutSeconds` as authoritative over `agents.defaults.subagents.runTimeoutSeconds`, the requested minimum/lock config surface is absent, and the related fix attempts are closed unmerged.

## What This Changes

The issue requests a config-enforced floor or lock so model-supplied `sessions_spawn.runTimeoutSeconds` values cannot undercut configured subagent timeout policy.

## Best Possible Solution

Add a backward-compatible optional `agents.defaults.subagents.minRunTimeoutSeconds` floor for native subagent spawns, preserve current behavior when unset, document the zero-timeout rule, and cover the resolver/config/docs path with focused tests.

## Reproduction Assessment

Yes. By source inspection, configure `agents.defaults.subagents.runTimeoutSeconds: 120` and pass explicit `runTimeoutSeconds: 2`; the current resolver returns 2, and the focused unit test asserts that behavior.

## Solution Assessment

Yes. An optional minimum floor is the narrowest maintainable direction for the reported timeout erosion; making the default authoritative or adding a lock mode would change released semantics more broadly.
