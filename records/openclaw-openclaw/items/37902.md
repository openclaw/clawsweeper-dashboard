## Summary

Keep open. Current main and the latest release still let explicit model-supplied `sessions_spawn.runTimeoutSeconds` override the configured default, and the two direct minimum-timeout fix PRs are closed unmerged.

## What This Changes

The issue requests a subagent timeout floor or lock so model-supplied `sessions_spawn.runTimeoutSeconds` cannot undercut configured operator defaults.

## Best Possible Solution

Add a backward-compatible optional `agents.defaults.subagents.minRunTimeoutSeconds` floor for native subagent spawns, with schema/types/docs/changelog alignment and regression coverage for explicit values, omitted values, and `0` semantics.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence current-main path: with configured default `120` and explicit `runTimeoutSeconds: 2`, the resolver returns `2`, and the current unit test asserts that precedence.

## Solution Assessment

Yes. An optional minimum floor is the narrowest backward-compatible solution; making the existing default authoritative would change released semantics more broadly.

## Repair Work Prompt

Implement `agents.defaults.subagents.minRunTimeoutSeconds` as an optional floor for native `sessions_spawn` subagent runs. Keep existing behavior unchanged when the new key is unset or `0`; when it is positive, clamp the effective run timeout upward so model-supplied or defaulted values below the floor cannot undercut it, and explicitly cover/document the `runTimeoutSeconds: 0` case. Update the exported config type, strict Zod schema, subagent timeout resolver, native spawn path, public docs, config docs, focused regression tests, and changelog. Do not make `runTimeoutSeconds` itself authoritative, do not change ACP behavior unless the existing native/ACP contract requires it, and preserve existing `timeoutSeconds` alias handling.
