## Summary

Keep open. Current main still treats a finite model-supplied `sessions_spawn.runTimeoutSeconds` as authoritative over the configured default, and the requested minimum/lock config surface is absent; the linked fix PRs are closed unmerged and the related closed issue covers hard timeout termination, not this floor/lock request.

## What This Changes

Request an optional subagent timeout floor or lock so model-supplied `sessions_spawn.runTimeoutSeconds` cannot undercut configured subagent timeout policy.

## Best Possible Solution

Add a backward-compatible optional `agents.defaults.subagents.minRunTimeoutSeconds` floor for native subagent spawns, align type/schema/docs/changelog, and cover explicit lower, higher, omitted, unset, and zero-timeout cases with focused tests.

## Reproduction Assessment

Yes, by source inspection: configure default `120`, pass explicit `runTimeoutSeconds: 2`, and the current resolver returns `2`; the current unit test asserts that behavior. I did not execute tests because this review is read-only.

## Solution Assessment

Yes. An optional minimum floor is the narrowest maintainable direction; making the existing default authoritative or adding a lock mode would change released semantics more broadly.
