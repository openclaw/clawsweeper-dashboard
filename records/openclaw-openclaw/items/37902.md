## Summary

Keep open. Current main still implements and documents `agents.defaults.subagents.runTimeoutSeconds` as a fallback only, accepts shorter explicit tool values, and has no `minRunTimeoutSeconds` or lock-style config key; the related fix PRs in the provided context are closed unmerged.

## What This Changes

Add a config-enforced minimum or lock for native `sessions_spawn.runTimeoutSeconds` so model-provided shorter timeouts cannot undercut configured subagent defaults across retries.

## Best Possible Solution

Add a backward-compatible optional `agents.defaults.subagents.minRunTimeoutSeconds` floor in the native subagent timeout resolver, leaving existing behavior unchanged when the floor is unset or zero.

## Reproduction Assessment

Yes. Source-level reproduction is high-confidence: with config default `120` and explicit `runTimeoutSeconds: 2`, the current resolver returns `2`, and the test suite asserts that precedence.

## Solution Assessment

Yes. An optional minimum floor is the narrowest maintainable direction shown by the report and prior PRs; making the configured default fully authoritative would be more disruptive.

Add an optional `agents.defaults.subagents.minRunTimeoutSeconds` config key that clamps positive effective native `sessions_spawn.runTimeoutSeconds` values upward when the model passes a smaller timeout. Preserve current behavior when the floor is unset or zero, preserve `runTimeoutSeconds: 0` as no timeout, and do not make the configured default fully authoritative. Update the config types, strict zod schema, generated config/schema artifacts as required by repo tooling, docs for `agents.defaults.subagents` and `/tools/subagents`, focused regression tests for omitted/explicit/below-floor/above-floor/zero cases, and a user-facing changelog entry.
