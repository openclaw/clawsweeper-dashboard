## Summary

Keep open. Current main still documents and implements `agents.defaults.subagents.runTimeoutSeconds` as a fallback-only default, explicit `sessions_spawn.runTimeoutSeconds` still wins, and the two direct fix PRs are closed unmerged.

## What This Changes

The issue requests a subagent timeout floor or lock so model-supplied `sessions_spawn.runTimeoutSeconds` values cannot undercut configured operator defaults.

## Best Possible Solution

Add a backward-compatible optional `agents.defaults.subagents.minRunTimeoutSeconds` floor for native subagent spawns, with explicit `0` semantics, config/schema/docs/changelog alignment, and regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence current-main path: with configured default `120` and explicit `runTimeoutSeconds: 2`, the resolver returns `2`, and the current test suite asserts that precedence.

## Solution Assessment

Yes, with one caveat. An optional minimum floor is narrower than making the existing default authoritative, but the implementation should deliberately decide whether a configured minimum clamps explicit `0` no-timeout requests.
