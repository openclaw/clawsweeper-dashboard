## Summary

Keep open. Current main and the latest release still only support default fallback semantics for subagent timeouts: an explicit `sessions_spawn.runTimeoutSeconds` wins over `agents.defaults.subagents.runTimeoutSeconds`, there is no minimum or lock config, and the related `minRunTimeoutSeconds` fix PRs are closed unmerged.

## What This Changes

Request to add a subagent timeout minimum or lock config so model-supplied `sessions_spawn.runTimeoutSeconds` cannot undercut configured timeout policy across retries.

## Best Possible Solution

Add a backward-compatible optional `agents.defaults.subagents.minRunTimeoutSeconds` floor for native subagent spawns, preserve existing behavior when it is unset, document the zero-timeout rule, and cover resolver/config/docs behavior with focused tests.

## Reproduction Assessment

Yes. By source inspection, configure `agents.defaults.subagents.runTimeoutSeconds: 120` and pass explicit `runTimeoutSeconds: 2`; the current resolver returns `2`, and the focused unit test asserts that behavior.

## Solution Assessment

Yes. An optional minimum floor is the narrowest maintainable direction for the reported timeout erosion; making the default fully authoritative or adding a lock mode would change released semantics more broadly.
