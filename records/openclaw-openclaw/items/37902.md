## Summary

Keep open: current main and v2026.5.19 still implement `runTimeoutSeconds` as a fallback default only, the requested minimum/lock config surface is absent, and the linked fix PRs were closed unmerged.

## What This Changes

The issue requests a subagent timeout floor or lock so model-supplied `sessions_spawn.runTimeoutSeconds` values cannot undercut configured subagent timeout policy.

## Best Possible Solution

Add a backward-compatible optional `agents.defaults.subagents.minRunTimeoutSeconds` floor for native `sessions_spawn`, preserving current fallback behavior when unset and documenting zero-timeout behavior under the floor.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: configure `agents.defaults.subagents.runTimeoutSeconds: 120` and pass explicit `runTimeoutSeconds: 2`; the current resolver returns `2`, and the focused test asserts that behavior. I did not execute a live model retry loop.

## Solution Assessment

Yes, the optional minimum-floor config is the narrowest maintainable direction because it preserves released fallback semantics when unset. A lock mode or changing `runTimeoutSeconds` itself into a floor would be broader and more compatibility-sensitive.

## Repair Work Prompt

Implement an optional `agents.defaults.subagents.minRunTimeoutSeconds` for native `sessions_spawn`. Preserve current behavior when the new key is unset; when it is a positive integer, clamp model/tool-provided finite `runTimeoutSeconds` below the floor up to the floor, including `0` so a model cannot disable an operator-configured minimum. Update the public config type, strict Zod schema, docs for subagents and gateway config, and changelog. Add focused tests for omitted timeout using the default, explicit timeout below the floor, explicit timeout above the floor, unset/zero minimum preserving current behavior, and explicit `runTimeoutSeconds: 0` under a positive floor. Do not change ACP timeout handling, channel send timeouts, model-fallback hard-stop behavior, or existing `runTimeoutSeconds` fallback semantics when `minRunTimeoutSeconds` is unset.
