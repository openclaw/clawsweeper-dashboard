## Summary

Keep open. Current main still treats `agents.defaults.subagents.runTimeoutSeconds` as a fallback only: an explicit `sessions_spawn.runTimeoutSeconds` wins even when it is lower, and the config type/schema/docs/tests still have no minimum or lock option. Prior fix PRs are closed unmerged, and no precise open PR currently tracks `minRunTimeoutSeconds`, so this remains a narrow actionable bug/feature request.

## What This Changes

The issue requests a configured subagent timeout floor or lock so model-supplied `sessions_spawn.runTimeoutSeconds` values cannot undercut the configured subagent timeout.

## Best Possible Solution

Add a backward-compatible optional subagent timeout constraint, preferably `agents.defaults.subagents.minRunTimeoutSeconds`, applied in the native subagent timeout resolver before run registration and gateway dispatch. The config type, zod schema, generated schema/help/docs, and regression tests should cover omitted values, explicit values below and above the floor, and zero/unset behavior.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on current main: with config default `120` and explicit `runTimeoutSeconds: 2`, the resolver returns `2`, and the current test suite explicitly asserts that precedence.

## Solution Assessment

Yes, an optional minimum floor is the narrowest maintainable fix shown by the issue and prior PRs; making the default timeout fully authoritative would be more disruptive unless maintainers explicitly want lock semantics.
