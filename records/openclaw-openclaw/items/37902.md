## Summary

Keep open. Current main still treats `agents.defaults.subagents.runTimeoutSeconds` as a fallback only, has no `minRunTimeoutSeconds` or lock config, and the direct fix PRs are closed unmerged.

## What This Changes

The issue requests a configurable minimum or lock for `sessions_spawn.runTimeoutSeconds` so model-supplied values cannot undercut subagent timeout defaults.

## Best Possible Solution

Add a backward-compatible optional `agents.defaults.subagents.minRunTimeoutSeconds` floor for native subagent spawns, with schema/types/docs/changelog alignment and regression coverage for explicit values, omitted values, and `0` semantics.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence current-main path: with configured default `120` and explicit `runTimeoutSeconds: 2`, the resolver returns `2`, and the current unit test asserts that precedence.

## Solution Assessment

Yes. An optional minimum floor is the narrowest maintainable path; making the existing default authoritative or adding a lock mode would change released semantics more broadly.
