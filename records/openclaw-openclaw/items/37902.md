## Summary

Keep open: current main and v2026.5.20 still let an explicit `sessions_spawn.runTimeoutSeconds` undercut the configured default, the public config schema/docs expose no minimum or lock key, and the related fix PRs were closed unmerged.

## What This Changes

The issue requests a minimum or locked timeout config so model-supplied `sessions_spawn.runTimeoutSeconds` cannot undercut subagent defaults.

## Best Possible Solution

Add a backward-compatible optional `agents.defaults.subagents.minRunTimeoutSeconds` floor for native `sessions_spawn`, preserve current fallback behavior when unset, and document/test the chosen zero-timeout semantics.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: configure `agents.defaults.subagents.runTimeoutSeconds: 120` and pass explicit `runTimeoutSeconds: 2`; the current resolver returns `2`, and a focused test asserts that behavior.

## Solution Assessment

Yes. An optional minimum-floor config is the narrowest maintainable direction because it avoids changing released `runTimeoutSeconds` fallback semantics for users who do not opt in.
