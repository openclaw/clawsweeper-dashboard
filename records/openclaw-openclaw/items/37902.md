## Summary

Keep open. Current main and the latest tagged release still treat a finite model-supplied `sessions_spawn.runTimeoutSeconds` as authoritative over the configured default, and the requested `minRunTimeoutSeconds`/lock-style config surface is not present in source, schema, docs, or tests.

## What This Changes

Request a config-enforced minimum or locked run timeout for native `sessions_spawn` so model-supplied low timeout values cannot undercut subagent timeout policy.

## Best Possible Solution

Add a backward-compatible optional `agents.defaults.subagents.minRunTimeoutSeconds` floor for native subagent spawns, align config type/schema/docs/changelog, and cover lower, higher, omitted, unset, and zero-timeout cases with focused tests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure default `120`, pass explicit `runTimeoutSeconds: 2`, and the current resolver returns `2`; the current unit test asserts that behavior. I did not execute tests because this review is read-only.

## Solution Assessment

Yes. An optional minimum floor is the narrowest maintainable direction; making the existing default authoritative or adding a lock mode would change released semantics more broadly.
