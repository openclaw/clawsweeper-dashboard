## Summary

Keep open. Current main still lets a finite model-supplied `sessions_spawn.runTimeoutSeconds` override the configured default, and the requested minimum/lock config surface is still absent; all directly related fix PRs found in search are closed unmerged.

## What This Changes

The issue asks for a config-level minimum or lock so model-supplied `sessions_spawn.runTimeoutSeconds` cannot undercut the configured subagent timeout.

## Best Possible Solution

Add a backward-compatible optional `agents.defaults.subagents.minRunTimeoutSeconds` floor for native subagent spawns, align config type/schema/docs/changelog, and cover lower, higher, omitted, unset, and zero-timeout cases with focused tests.

## Reproduction Assessment

Yes. By source inspection, configure `agents.defaults.subagents.runTimeoutSeconds: 120`, pass explicit `runTimeoutSeconds: 2`, and current main resolves to `2`; the current unit test asserts that same behavior.

## Solution Assessment

Yes. An optional minimum floor is the narrowest maintainable direction; making the existing default authoritative or adding a lock mode would change released semantics more broadly.
