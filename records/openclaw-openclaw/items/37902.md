## Summary

Keep open. Current main still lets an explicit `sessions_spawn.runTimeoutSeconds` override the configured subagent default, the requested minimum/lock config surface is absent, and both related fix PRs are closed unmerged.

## What This Changes

The issue requests a config-controlled minimum for native subagent spawn timeouts so model-supplied `runTimeoutSeconds` values cannot undercut operator defaults.

## Best Possible Solution

Add a backward-compatible optional `agents.defaults.subagents.minRunTimeoutSeconds` floor for native subagent spawns, preserve current behavior when unset, document the zero-timeout rule, and cover resolver/config/docs behavior with focused tests.

## Reproduction Assessment

Yes. By source inspection, configure `agents.defaults.subagents.runTimeoutSeconds: 120` and pass explicit `runTimeoutSeconds: 2`; the current resolver returns 2, and the focused unit test asserts that behavior.

## Solution Assessment

Yes. An optional minimum floor is the narrowest maintainable direction for the reported timeout erosion; making the default authoritative or adding a lock mode would change released semantics more broadly.
