## Summary

Keep open. Current main and the latest release still treat `agents.defaults.subagents.runTimeoutSeconds` as a fallback default only, explicit `sessions_spawn.runTimeoutSeconds` still wins, and the requested minimum/lock config has not landed.

## What This Changes

Request to add a subagent timeout floor or lock so model-supplied `sessions_spawn.runTimeoutSeconds` cannot undercut operator config.

## Best Possible Solution

Add a backward-compatible optional `agents.defaults.subagents.minRunTimeoutSeconds` floor for native subagent spawns, preserve current behavior when unset, and document/test the chosen zero-timeout semantics.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: configure `agents.defaults.subagents.runTimeoutSeconds: 120` and pass explicit `runTimeoutSeconds: 2`; the current resolver returns `2`, and the focused test asserts that behavior.

## Solution Assessment

Yes. An optional minimum floor is the narrowest maintainable fix because it preserves released default-only semantics when unset; a full lock mode is broader and should stay out of scope unless maintainers explicitly request it.
