## Summary

Keep open: current main and the latest release still let an explicit model-supplied `runTimeoutSeconds` undercut the configured subagent default, no `minRunTimeoutSeconds` or lock config exists, and the linked fix attempts are closed unmerged.

## What This Changes

Request an additive config floor or lock so native `sessions_spawn` cannot undercut operator subagent timeout defaults with a lower model-supplied `runTimeoutSeconds` value.

## Best Possible Solution

Add a backward-compatible optional `agents.defaults.subagents.minRunTimeoutSeconds` floor for native `sessions_spawn`, preserve current behavior when unset, and document/test the chosen zero-timeout semantics.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: configure `agents.defaults.subagents.runTimeoutSeconds: 120` and pass explicit `runTimeoutSeconds: 2`; the current resolver returns `2`, and current focused tests assert that behavior.

## Solution Assessment

Yes. An additive optional minimum-floor config is the narrowest maintainable direction because it preserves released fallback semantics when unset; a lock mode or changing `runTimeoutSeconds` itself into a floor would be broader and more compatibility-sensitive.

Preserve current behavior when the new config is unset, keep `agents.defaults.subagents.runTimeoutSeconds` as the default used when the tool call omits `runTimeoutSeconds`, and make model-supplied positive timeouts below the configured minimum clamp upward.
