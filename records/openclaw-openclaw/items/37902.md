## Summary

Keep open. Current main still implements and documents `agents.defaults.subagents.runTimeoutSeconds` as a fallback-only default, explicit `sessions_spawn.runTimeoutSeconds` still wins, and the direct `minRunTimeoutSeconds` fix PRs are closed unmerged.

## What This Changes

The issue requests a configurable minimum or locked timeout for `sessions_spawn` so model-supplied `runTimeoutSeconds` values cannot undercut operator defaults.

## Best Possible Solution

Add a backward-compatible optional `agents.defaults.subagents.minRunTimeoutSeconds` floor for native subagent spawns, with deliberate `0` semantics, config/schema/docs/changelog alignment, and regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence current-main path: with configured default `120` and explicit `runTimeoutSeconds: 2`, the resolver returns `2`, and the current test suite asserts that precedence.

## Solution Assessment

Yes. An optional minimum floor is narrower and more backward-compatible than making the existing default authoritative, as long as the implementation explicitly documents whether a configured minimum clamps explicit `0` no-timeout requests.
