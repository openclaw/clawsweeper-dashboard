## Summary

Keep open. Current main and the latest release still let a finite model-supplied `sessions_spawn.runTimeoutSeconds` override the configured subagent default, and the requested minimum/lock config surface is still absent; the related fix PRs are closed unmerged.

## What This Changes

Add config-enforced minimum or locked timeout behavior for `sessions_spawn` so model-supplied `runTimeoutSeconds` values cannot undercut operator policy.

## Best Possible Solution

Add a backward-compatible optional `agents.defaults.subagents.minRunTimeoutSeconds` floor for native subagent spawns, align type/schema/docs/changelog, and cover lower, higher, omitted, unset, and zero-timeout cases with focused tests.

## Reproduction Assessment

Yes. By source inspection, configure `agents.defaults.subagents.runTimeoutSeconds: 120`, pass explicit `runTimeoutSeconds: 2`, and the current resolver returns `2`; the current unit test asserts that behavior.

## Solution Assessment

Yes. An optional minimum floor is the narrowest maintainable direction; making the existing default authoritative or adding a lock mode would change released semantics more broadly.
