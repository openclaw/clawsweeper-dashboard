## Summary

Keep open. Current main still treats a model-supplied `sessions_spawn.runTimeoutSeconds` as the effective value before the configured default, no minimum/lock config surface exists, and the direct fix PRs are closed unmerged.

## What This Changes

Request a config-enforced minimum or lock for `sessions_spawn.runTimeoutSeconds` so model-supplied lower values cannot undercut subagent timeout policy.

## Best Possible Solution

Add a backward-compatible optional `agents.defaults.subagents.minRunTimeoutSeconds` floor for native subagent spawns, with schema/types/docs/changelog alignment and regression coverage for omitted, lower explicit, higher explicit, and zero-timeout cases.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: with configured default `120` and explicit `sessions_spawn.runTimeoutSeconds: 2`, the resolver returns `2`, and the current unit test asserts that behavior.

## Solution Assessment

Yes. An optional minimum floor is narrower and more compatible than making the existing default authoritative or adding a broader lock mode; the implementation should define how the floor treats explicit zero.
