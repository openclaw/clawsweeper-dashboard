## Summary

Keep open. Current main has partial resilience around invalid plugin config, plugin register failures, hook-level timeouts, memory recall timeout, typing cleanup, and restart-aborted session recovery, but it still does not implement the requested plugin-scoped circuit breaker, a generic hard timeout/cancellation wrapper around plugin-owned tool execution, or typing-TTL-driven session recovery.

## What This Changes

The issue requests plugin-runtime resilience so repeated plugin failures do not freeze a session, including a plugin circuit breaker, typing-TTL recovery, and generic plugin tool-call timeouts.

## Best Possible Solution

Keep this issue open unless maintainers explicitly consolidate it into #41899 and preserve the missing tool-timeout and typing-TTL recovery asks. The end state should be a core plugin-runtime resilience design with plugin-scoped rolling failure budgets, timeout and AbortSignal semantics for plugin-owned tools, persisted open/half-open/forced-open state, operator status/reset controls, and a deliberate session recovery policy for typing TTL or stuck-session diagnostics.

## Reproduction Assessment

No, there is not a high-confidence live reproduction path from the issue alone: it names the environment and failure chain, but not enough config, logs, or data-rebuild steps to reproduce memory-lancedb-pro locally. Source inspection does give a high-confidence gap check because the requested breaker, generic plugin tool timeout, and typing-TTL recovery paths are still absent or only partially covered.

## Solution Assessment

Unclear; the issue is valid but the best implementation needs maintainer design rather than a narrow one-file patch. The current partial mitigations are useful, but a safer solution needs a generic plugin-runtime policy with explicit cancellation, breaker state, and operator recovery semantics instead of treating typing TTL alone as an automatic restart trigger.
