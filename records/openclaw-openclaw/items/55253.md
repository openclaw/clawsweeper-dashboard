## Summary

Keep open. Current main has partial safeguards for plugin hook timeouts, memory recall latency, Active Memory timeout cooldowns, and diagnostic no-progress recovery, but source inspection still shows no framework-level plugin circuit breaker, no generic hard timeout/cancellation around plugin-owned tool execution, and no typing-TTL-triggered session restart policy; the open circuit-breaker issue overlaps only part of this request.

## What This Changes

The issue requests plugin crash-loop resilience through plugin-scoped circuit breakers, hard plugin call timeouts, and typing-TTL-triggered session recovery.

## Best Possible Solution

Design one core plugin-runtime resilience feature that combines plugin-scoped breaker state, abortable hook/tool timeout contracts, operator status/reset controls, and explicit no-progress session recovery policy while preserving plugin-local safeguards where they are safer.

## Reproduction Assessment

No, there is not a high-confidence live reproduction path for the reported memory-lancedb-pro crash loop from the issue alone. Source inspection does give a clear current-main gap check for the missing generic plugin breaker, generic plugin tool timeout, and typing-TTL-specific recovery behavior.

## Solution Assessment

Unclear; the requested direction is reasonable, but a one-file timeout or typing restart patch would not be the best maintainable solution. The best path needs maintainer design across plugin runtime policy, cancellation semantics, persistence, operator UX, and session recovery.
