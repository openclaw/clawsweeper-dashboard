## Summary

Keep open. Current main has useful session diagnostics and partial hook timeout handling, but it still lacks the requested generic plugin circuit breaker, persisted breaker/status/reset policy, and hard framework timeout around plugin-owned tool execution; the newer macOS/Telegram comment keeps the frozen-session symptom active.

## What This Changes

The issue requests plugin-scoped circuit breaking, plugin call timeouts, and stuck-session recovery so repeated plugin failures do not leave Telegram sessions alive but unresponsive.

## Best Possible Solution

Design a staged runtime resilience feature that adds plugin-scoped failure accounting, abortable hook/tool timeout contracts, persisted breaker state/status/reset controls, and diagnostics-driven no-progress recovery without changing existing users to fail closed by surprise.

## Reproduction Assessment

No, there is not a high-confidence live reproduction path for the original memory-lancedb-pro crash loop from the issue alone. Source inspection does give a clear current-main gap check for the missing generic breaker, hard plugin-tool timeout, and direct typing-TTL recovery behavior.

## Solution Assessment

Unclear; the requested direction is reasonable, but a one-off timeout or typing-TTL restart patch would not be the best maintainable fix. The safer path is a designed runtime policy that composes with current diagnostics and the open circuit-breaker design thread.
