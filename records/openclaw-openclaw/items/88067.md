## Summary

Keep open: the fix is not on current main, and the PR addresses a source-reproducible Responses replay/session bug, but this external PR still lacks real behavior proof for the Azure/OpenAI Responses fallback recovery path.

## What This Changes

The PR strips paired Responses assistant message ids when replayable reasoning is dropped, makes synthetic fallback message ids unique across both Responses conversion paths, and adds regression coverage.

## Best Possible Solution

Land the focused sanitizer and conversion fix only after redacted real Azure/OpenAI Responses fallback proof shows the session can switch away, return to Responses, and continue without the orphaned msg_*/rs_* rejection.

## Reproduction Assessment

Yes from source, but not from a live run in this review. Current main drops replayable reasoning on model switches while the Responses conversion path still replays msg_* from textSignature; the linked report supplies the Azure 400 symptom.

## Solution Assessment

Yes at the code level: dropping the paired message id when its reasoning item is dropped is the narrow fix, and updating both conversion sites avoids duplicate synthetic ids. The remaining gap is real provider proof, especially for phase-bearing multi-text replay.
