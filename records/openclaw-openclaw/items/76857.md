## Summary

Keep open: the report matches current main behavior, and an open closing PR already targets the same IPv6-egress failure. Current guarded-fetch dispatchers still create per-request undici agents without inheriting the global `autoSelectFamily` policy, so the issue should stay open until that PR or an equivalent fix lands.

## What This Changes

The issue reports an OpenAI Codex provider regression where guarded requests to `chatgpt.com` fail on IPv4-only networks that still receive AAAA DNS answers.

## Best Possible Solution

Land #76870 or an equivalent generic transport fix that propagates the system `autoSelectFamily` decision into guarded per-request HTTP/1 dispatchers, with dispatcher regression coverage and a user-facing changelog entry.

## Reproduction Assessment

Yes from source inspection and reporter evidence, but not by a live run here. Current main still builds pinned guarded-fetch dispatchers without `autoSelectFamily`, and the issue provides repeated logs plus an IPv4 `/etc/hosts` workaround that isolates the failure mode.

## Solution Assessment

Yes, the linked fix direction is the narrow maintainable solution: repair the shared guarded-fetch dispatcher layer rather than adding OpenAI Codex-specific hosts, `/etc/hosts` guidance, or a new config knob.
