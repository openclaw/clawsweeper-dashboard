## Summary

Keep open. Current main still leaves guarded-fetch per-request dispatchers without the global autoSelectFamily/Happy Eyeballs setting, and #76870 is an open closing PR for this exact issue, so this should remain open until that PR merges or is superseded.

## What This Changes

Ensure openai-codex agent runs on IPv4-only/no-IPv6-egress networks fall back to IPv4 instead of failing on chatgpt.com IPv6 connection attempts.

## Best Possible Solution

Review and land or replace #76870 with a generic guarded-fetch undici-runtime fix that propagates Happy Eyeballs policy to per-request agents, with regression coverage for pinned direct, env-proxy, and explicit-proxy dispatchers.

## Reproduction Assessment

Yes. A high-confidence reproduction path is a Linux host where chatgpt.com returns AAAA records but outbound IPv6 is unavailable; current main shows guarded fetch builds pinned per-request undici agents that do not inherit autoSelectFamily.

## Solution Assessment

Yes. The narrow maintainable direction is the linked PR's generic transport fix in src/infra/net/undici-runtime.ts rather than an openai-codex special case or an /etc/hosts workaround.
