## Summary

Current main already fixes the regression by applying the shared Undici autoSelectFamily policy to guarded HTTP/1 dispatchers used by openai-codex Responses fetches; the proof commit is on main but not in v2026.5.2, so #76870 is superseded.

## What This Changes

The issue asks OpenClaw to make openai-codex agent requests work on IPv4-only networks where DNS still returns AAAA records.

## Best Possible Solution

Keep the generic guarded-fetch family fallback on main, close or supersede #76870, and ship the fix in the next release.

## Reproduction Assessment

Yes. The issue gives a concrete v2026.5.2 IPv4-only/AAAA reproduction path, and source inspection shows the affected provider requests use guarded fetch dispatchers that lacked the shared family policy before the current-main fix.

## Solution Assessment

Yes. The current-main fix is the narrowest maintainable direction because it repairs the generic guarded-fetch dispatcher path used by openai-codex and other provider transports instead of adding provider-specific DNS handling.
