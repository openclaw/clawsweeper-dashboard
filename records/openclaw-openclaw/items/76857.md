## Summary

Keep this open: current main still routes openai-codex model requests through guarded per-request Undici dispatchers that do not inherit the global `autoSelectFamily` setting, and open PR #76870 already references this issue with closing syntax.

## What This Changes

The issue reports openai-codex LLM requests failing with `fetch failed` on Linux networks where DNS returns IPv6 addresses but outbound IPv6 egress is unavailable.

## Best Possible Solution

Review and land PR #76870 or an equivalent generic guarded-fetch fix that propagates Node's family-selection behavior to per-request dispatchers, with regression coverage for direct, env-proxy, and explicit-proxy dispatchers.

## Reproduction Assessment

Yes, from source inspection and the reporter's concrete logs, but not from a live IPv6-disabled host in this read-only review. The current-main openai-codex provider fetch path uses per-request guarded Undici dispatchers, and those dispatchers currently omit the global `autoSelectFamily` behavior unless a caller manually supplies it.

## Solution Assessment

Yes, the linked PR's stated direction is the right boundary: fix the generic guarded-fetch dispatcher construction rather than adding an openai-codex-only workaround or a new config knob. The issue should remain open until that PR or an equivalent patch lands.
