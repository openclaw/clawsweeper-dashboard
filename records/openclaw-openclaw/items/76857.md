## Summary

Keep open: the report has strong regression evidence, and current main still routes openai-codex streaming through the shared OpenAI SDK/provider fetch path without an openai-codex-specific IPv4-egress regression test or verified fix.

## What This Changes

The issue reports that openai-codex agent streaming fails on IPv4-only egress networks when chatgpt.com DNS returns AAAA records, regressing from 2026.4.26 to 2026.5.2.

## Best Possible Solution

Fix the native openai-codex provider transport so chatgpt.com requests reliably use reachable IPv4/family fallback on IPv4-only egress hosts, with focused regression coverage and no broad user-facing config unless maintainers choose one.

## Reproduction Assessment

Yes, with high confidence from the issue evidence and source path: 2026.5.2 openai-codex runs fail after prep, direct IPv4 succeeds, direct IPv6 fails, and an IPv4 hosts pin fixes the next run. I did not live-run an IPv6-broken host here, so the structured status is source_reproducible rather than reproduced.

## Solution Assessment

No current fix is proven on main. The narrowest maintainable solution is in the openai-codex/provider transport path, with tests that simulate A+AAAA DNS and failed IPv6 egress without weakening the shared SSRF/proxy guard.
