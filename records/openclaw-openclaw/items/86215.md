## Summary

Keep open: current main has useful mitigations, but the central Codex/OpenAI auth incident is still not solved because Codex app-server external-auth callbacks still have a 10s upstream deadline, OpenClaw still forces OAuth refresh on that callback, fallback suppression is default-off and limited, and operator-visible incident/session handling remains incomplete.

## What This Changes

The issue reports Codex/OpenAI OAuth refresh and profile-selection failures that can leave agents, cron, or channels wedged without clear operator-visible incident handling.

## Best Possible Solution

Define a default Codex/OpenAI auth-profile incident model that reuses valid tokens where safe, distinguishes transient refresh slowness from hard auth failure, quarantines or rotates broken profiles, releases channel/cron session state cleanly, and exposes redacted operator status.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: current main still routes Codex app-server external-auth refresh requests through an OpenClaw forced-refresh path while upstream waits only 10 seconds for the callback.

## Solution Assessment

No automated fix path is best yet. The bug is real, but the durable solution needs maintainer decisions about token reuse, quarantine defaults, redacted status surfacing, session cleanup, and cost-bearing API-key fallback policy.
