- `P1`: The issue has repeated live reports of Codex/OpenAI auth failures wedging channel, cron, and agent workflows and causing missed user-visible outputs.

## Summary

Keep open: current main has partial mitigations, including a longer OAuth refresh timeout and auth-profile cooldown state, but it still does not fully solve the default Codex/OpenAI failure family because repeat primary auth failures are not durably quarantined, fallback suppression is opt-in/session-local, and API-key-backed Codex routes remain possible.

## What This Changes

The issue requests Codex/OpenAI OAuth failure handling that quarantines or rotates bad profiles, surfaces a clear operator incident, and prevents wedged agent/channel/cron runs.

## Best Possible Solution

Define a single Codex/OpenAI auth incident model that durably contains permanent or repeated refresh failures, rotates profiles safely, prevents silent API-key fallback unless explicit and visible, releases wedged sessions, and reports redacted status/doctor evidence.

## Reproduction Assessment

Yes from source for the remaining default gap, but not as a live current-main repro: current code and docs show opt-in-only fallback suppression, primary retries, and API-key-compatible Codex auth while issue comments provide multiple shipped-version live reproductions.

## Solution Assessment

No current complete solution exists on main; the timeout increase and cooldown state are useful partial mitigations, but the best fix needs a maintainer-approved Codex/OpenAI auth incident and fallback policy rather than a timeout-only repair.
