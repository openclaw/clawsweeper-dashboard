- `P1`: The issue has repeated live reports of valid-looking Codex/OpenAI auth causing cron, channel, and direct agent workflows to fail or wedge before producing replies.

## Summary

Keep open: current main has useful mitigations, but the central Codex/OpenAI auth incident remains because Codex app-server refresh callbacks still force OAuth refresh under upstream's 10s callback deadline, fallback suppression is default-off and limited, and operator-visible incident/session handling is still incomplete.

## What This Changes

The issue asks OpenClaw to classify repeated Codex OAuth refresh failures as an auth-profile incident, rotate or quarantine failing profiles more aggressively, release wedged channel/session state, and surface clear operator alerts.

## Best Possible Solution

Define a default Codex/OpenAI auth-profile incident model that reuses valid tokens where safe, distinguishes transient refresh slowness from hard auth failure, quarantines or rotates broken profiles, releases channel/cron session state cleanly, and exposes redacted operator status.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main still forces Codex app-server OAuth refresh through upstream's 10s external-auth callback, while fallback/session/status behavior remains too limited for the reported incident family.

## Solution Assessment

No automated fix path is best yet; the bug is real, but the durable solution requires maintainer decisions about token reuse, quarantine defaults, redaction, status ownership, and cost-bearing fallback policy.
