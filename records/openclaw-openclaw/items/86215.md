## Summary

Keep open: current main has useful partial mitigations for the Codex/OpenAI auth failure family, but the core problem remains. The issue now has live reports through OpenClaw 2026.6.x showing missed cron/channel output, repeat dead-profile retries, stale fallback paths, and cost-bearing API-key drift; current source still leaves repeat auth suppression opt-in/session-local, never skips primary candidates by default, and relies on policy-sensitive Codex auth-mode behavior that needs maintainer direction.

## What This Changes

The issue asks OpenClaw to classify repeated Codex/OpenAI OAuth/app-server failures as auth-profile incidents, alert operators, quarantine or rotate bad profiles/providers, release wedged sessions, and prevent silent API-key fallback surprises.

## Best Possible Solution

Define one default-on Codex/OpenAI auth incident path that reliably marks unhealthy profiles across app-server/profile/store boundaries, rotates to viable providers without wedging sessions, releases affected sessions, exposes redacted status/probe evidence, and makes API-key fallback explicit and cost-visible.

## Reproduction Assessment

Yes at source/log level, but not from a fresh live current-main repro in this read-only review; the issue has multiple shipped-version live reports and current source still shows the default remaining gaps.

## Solution Assessment

Current main is a partial mitigation, not the best complete solution; the best fix needs a maintainer-approved auth incident and fallback policy rather than timeout-only, opt-in skip-cache, or log-copy improvements.
