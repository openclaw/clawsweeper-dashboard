## Summary

Keep open: current main has useful mitigations, but it still does not fully solve the Codex/OpenAI auth incident family because Codex app-server refresh callbacks still force OAuth refresh under upstream’s 10s timeout, Codex transport timeouts do not poison auth-profile cooldowns, fallback suppression remains opt-in/session-scoped/non-primary, and status surfaces still split auth diagnostics away from a clear operator incident.

## What This Changes

The issue requests that repeated Codex OAuth refresh timeouts or invalidated-token responses trigger clearer operator alerts, faster auth-profile quarantine/rotation, clean session release, and visible cost-bearing auth-route reporting.

## Best Possible Solution

Define a default auth-profile incident model for Codex/OpenAI routes that distinguishes transient refresh slowness from hard auth failure, reuses still-valid tokens where safe, quarantines or rotates broken profiles without retry storms, releases channel/cron session state cleanly, and exposes a redacted operator-visible incident/status item.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main still forces Codex app-server OAuth refresh through an upstream 10s external auth callback, and Codex transport timeouts do not mark auth profiles failed. I did not perform a live OAuth outage reproduction in this read-only review.

## Solution Assessment

No for an automated fix path: the bug is real, but the best solution requires maintainer choices about cached-token reuse, default quarantine behavior, redacted incident/status UX, and cost-bearing API-key fallback policy.
