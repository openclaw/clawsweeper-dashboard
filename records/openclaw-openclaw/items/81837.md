## Summary

Current main still has the reported source path: the Dashboard sends only a `model` patch, `sessions.patch` applies that model without any auth-profile information, and the shared helper deletes the existing auth profile when no profile is supplied. This is recent and actionable, so it should stay open for a focused fix rather than cleanup closure.

## What This Changes

The issue reports that switching a session model through the Dashboard clears `authProfileOverride`, which can break provider authentication and overwrite the user's selected model.

## Best Possible Solution

Keep the issue open and fix the Gateway `sessions.patch` model path so Dashboard model switches retain or resolve compatible auth-profile state while preserving strict user-selected model semantics.

## Reproduction Assessment

Yes, there is a high-confidence source reproduction: a model-only `sessions.patch` reaches `applyModelOverrideToSessionEntry` without profile data, and the existing focused test asserts that the auth override is cleared. I did not run the test or a live Dashboard session because this review was read-only.

## Solution Assessment

Yes, the right fix belongs in the server-side `sessions.patch` model handling because both the Dashboard picker and SDK clients depend on that RPC. The narrow maintainable direction is to carry or preserve a compatible auth profile there, not to add a Dashboard-only workaround or new config knob.
