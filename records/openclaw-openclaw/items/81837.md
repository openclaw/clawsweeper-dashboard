## Summary

This should stay open: current main still has the reported Dashboard path, and the source shows a model patch can clear session auth-profile override state before the next API call.

## What This Changes

The issue reports that Dashboard model switching clears a session auth profile override because the sessions patch path does not pass profile override data into the model override helper.

## Best Possible Solution

Make `sessions.patch` keep or resolve the auth identity for explicit user model selections so the Control UI picker matches `/model` strict-selection semantics without preserving incompatible provider credentials.

## Reproduction Assessment

Yes at source level: a `sessions.patch` model update against a stored entry with `authProfileOverride` reaches `applyModelOverrideToSessionEntry` without profile data, which deletes the auth fields. I did not run a live Dashboard/API reproduction in this read-only review.

## Solution Assessment

No, the current implementation is not the best fix because it drops auth-profile state for Dashboard model changes while `/model` can carry profile resolution. The narrow maintainable fix belongs in the Gateway model patch path, with UI coverage only if the client payload or response contract changes.
