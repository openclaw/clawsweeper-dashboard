## Summary

Keep open. Current main and the latest shipped release still lack a Feishu outbound pacing/minimum-interval config, and the related broad rate-limit and Feishu retry/backoff issues do not supersede this proactive Feishu delivery pacing request.

## What This Changes

The issue requests configurable Feishu message rate limiting, ideally with control over both minimum interval between outbound messages and response segmentation.

## Best Possible Solution

Add a Feishu-owned per-account outbound pacing layer, configurable at top level and per account, and apply it consistently to all Feishu delivery surfaces that can burst API calls while keeping broad core rate limiting and retry/backoff on their separate trackers.

## Reproduction Assessment

Yes, at source level for the missing capability. Current main has no Feishu pacing config in schema/docs/manifest, and the reply, comment, outbound, send, media, and streaming-card paths call Feishu APIs without a shared minimum-interval gate; I did not run a live Feishu rate-limit scenario.

## Solution Assessment

Yes. A Feishu-plugin-owned outbound pacing helper is narrower and more maintainable than waiting for broad core rate limiting, while reactive retry/backoff should remain tracked separately at https://github.com/openclaw/openclaw/issues/70879.
