## Summary

Keep open. Current main and v2026.6.8 still lack the requested proactive Feishu outbound pacing or minimum-interval configuration; the merged retry work improves recovery after Feishu rate-limit responses, but it does not prevent bursty sends before Feishu sees them. The broader tracker https://github.com/openclaw/openclaw/issues/13615 and the closed retry issue https://github.com/openclaw/openclaw/issues/70879 overlap but do not supersede this Feishu-specific request.

## What This Changes

The issue requests configurable Feishu outbound message pacing/minimum intervals so bursty chunked responses do not trigger Feishu API limits.

## Best Possible Solution

Add a Feishu-owned outbound scheduler around message create/reply style sends, configurable at top level and per account with safe defaults, and document/test it across chunked replies, comment replies, media/card sends, message-tool outbound, and streaming-card initial sends while leaving broader cross-provider limits to https://github.com/openclaw/openclaw/issues/13615.

## Reproduction Assessment

Yes, at source level: current main has Feishu chunked/comment/outbound send paths and a reactive retry helper, but no pre-send minimum interval or scheduler. I did not run a live Feishu burst test in this read-only review.

## Solution Assessment

Yes, keeping this narrow Feishu plugin request open is the best path. The merged retry PR solves post-failure recovery, while a broad core rate-limit framework would be too large for this channel-specific pacing gap.
