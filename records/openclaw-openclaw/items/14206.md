## Summary

Keep open: current main and the latest release still lack a Feishu outbound pacing or minimum-interval setting, and the inspected reply, comment, message-tool, card, media, and streaming-card send paths still call Feishu delivery APIs directly. The related broad limiter and Feishu retry/backoff issues are useful context, but neither supersedes this proactive Feishu-specific pacing request.

## What This Changes

Request a configurable Feishu message minimum interval or outbound pacing layer so bursty chunked LLM replies do not trigger Feishu API limits.

## Best Possible Solution

Add a Feishu-owned per-account outbound pacing layer, expose it in schema/manifest/docs, and apply it consistently to burst-prone delivery paths while keeping broad cross-provider limits and reactive retry/backoff in their separate trackers.

## Reproduction Assessment

Not applicable as a bug reproduction because this is an enhancement/config request. Source inspection on current main and v2026.5.12 gives high confidence that the requested Feishu pacing config and shared outbound pacing helper are absent.

## Solution Assessment

Yes directionally: a Feishu-owned per-account pacing helper is narrower and cleaner than a broad core limiter for this request. The exact public config name, defaults, inheritance behavior, and delivery-path budget coverage need maintainer scoping before implementation.
