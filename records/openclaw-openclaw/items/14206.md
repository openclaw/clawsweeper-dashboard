## Summary

Keep open. Current main and v2026.6.5 have Feishu retry-after-rate-limit handling, but they still lack the requested proactive outbound minimum-interval/pacing configuration across Feishu send paths.

## What This Changes

The issue requests a configurable Feishu message pacing or minimum-interval setting so bursty chunked replies do not trigger Feishu API rate limits.

## Best Possible Solution

Add a Feishu-owned per-account outbound pacing helper with conservative defaults and route every burst-capable Feishu delivery surface through it, while leaving broader cross-provider limits on the separate tracker.

## Reproduction Assessment

Yes for source-level reproduction: current Feishu config, docs, reply, comment, outbound, media, and streaming-card paths show retry-after-error behavior but no proactive minimum-interval pacing. I did not run a live Feishu burst test in this read-only review.

## Solution Assessment

Yes. A Feishu-plugin pacing layer is the narrowest maintainable direction because the problem is channel-specific and the merged retry helper only reacts after rate-limit responses; a core-wide limiter would be broader than this issue needs.
