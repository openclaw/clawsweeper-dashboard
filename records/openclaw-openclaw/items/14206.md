## Summary

Keep open: current main adds reactive retries after Feishu rate-limit responses, but it still lacks the requested proactive outbound pacing or minimum-interval control, and no viable related item supersedes this narrower channel-specific work.

## What This Changes

Add configurable proactive pacing for bursty Feishu outbound messages, including chunked replies.

## Best Possible Solution

Add one Feishu-owned, per-account pacing coordinator at the canonical message-create/reply boundary, apply it consistently to normal replies, comments, message-tool output, media, and initial streaming-card delivery, and keep existing reactive retry and streaming-update coalescing as separate mechanisms.

## Reproduction Assessment

Yes at source level: current main contains burst-capable Feishu send paths and only reactive retries, while its schema and docs expose no proactive minimum interval. This read-only review did not run a new live Feishu stress test.

## Solution Assessment

Yes, with a refined boundary: the narrowest maintainable approach is one Feishu-owned per-account coordinator beneath all message-create/reply callers, rather than separate delays in each chunking or delivery path.
