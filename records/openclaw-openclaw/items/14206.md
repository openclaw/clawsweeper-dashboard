## Summary

Keep open. Current main still lacks a Feishu outbound pacing/minimum-interval config, and the related broad rate-limit tracker, Feishu retry tracker, and streaming-throttle PRs do not cover consistent proactive pacing across Feishu message delivery surfaces.

## What This Changes

The issue requests a configurable minimum outbound message interval for the Feishu channel to avoid Feishu API flow-control bans when OpenClaw sends bursty or chunked responses.

## Best Possible Solution

Add a Feishu-owned, per-account outbound pacing layer configurable at top level and per account, wire it through every Feishu delivery surface that can burst API calls, and document it in the Feishu channel docs, manifest, schema, and focused tests.

## Reproduction Assessment

Yes, at source level. Current main has no Feishu pacing config in schema/docs/manifest, and the reply, comment, outbound, send, media, and streaming paths call Feishu APIs without a shared minimum-interval gate; I did not run a live Feishu rate-limit scenario in this read-only review.

## Solution Assessment

Yes. A Feishu-plugin-owned outbound pacing helper is narrower and more maintainable than waiting for broad core rate limiting, while reactive retry/backoff should remain tracked separately at https://github.com/openclaw/openclaw/issues/70879.
