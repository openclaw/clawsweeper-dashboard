## Summary

Current main and the latest release still lack the requested proactive Feishu outbound pacing/minimum-interval configuration; the merged Feishu retry work improves recovery after rate-limit responses but does not prevent bursty sends before Feishu sees them.

## What This Changes

The issue requests Feishu-owned message rate limiting, preferably with a configurable minimum interval between sends and behavior that also reduces bursty response chunking.

## Best Possible Solution

Add a Feishu-owned outbound scheduler around message create/reply style sends, configurable at top level and per account with safe defaults, and document/test it across chunked replies, comment replies, media/card sends, message-tool outbound, and streaming-card initial sends while leaving broader cross-provider limits to https://github.com/openclaw/openclaw/issues/13615.

## Reproduction Assessment

Yes, at source level: current main has Feishu chunked/comment/outbound send paths and a reactive retry helper, but no pre-send minimum interval or scheduler. I did not run a live Feishu burst test in this read-only review.

## Solution Assessment

Yes, keeping this open is the right outcome: the merged retry PR solves post-failure recovery, not proactive pacing. The maintainable fix belongs in the Feishu plugin at the message API boundary, not as a broad core rate-limit framework for this narrow issue.
