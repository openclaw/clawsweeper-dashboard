## Summary

Keep open. Current main still lacks a Feishu outbound minimum-interval configuration or shared per-account send gate, and the related open trackers cover broader global limits and reactive retry/backoff rather than this proactive Feishu pacing request.

## What This Changes

The issue requests configurable outbound message pacing for the Feishu channel so bursty chunked LLM responses do not trigger Feishu API flow-control limits.

## Best Possible Solution

Add a Feishu-plugin-owned per-account outbound pacing helper, expose it in schema, manifest, and docs at top level and per account, and route burst-capable text, card, comment, media, and streaming-card delivery paths through it while leaving global limits and retry/backoff to their separate trackers.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature/config request. Source inspection gives high confidence that current main has no Feishu outbound pacing config or shared send gate.

## Solution Assessment

Yes. A Feishu-owned pacing layer is the narrowest maintainable direction for this channel-specific frequency problem; broader global limits belong in https://github.com/openclaw/openclaw/issues/13615 and retry/backoff belongs in https://github.com/openclaw/openclaw/issues/70879.
