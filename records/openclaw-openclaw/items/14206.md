## Summary

Keep open: current main still lacks a Feishu outbound pacing or minimum-interval setting, and the inspected reply, comment, message-tool, card, and media paths still deliver sequential chunks directly without a shared pacing gate. The related broad rate-limit and Feishu retry/backoff issues are useful context, but they do not supersede this proactive Feishu-specific pacing request.

## What This Changes

The issue requests configurable Feishu outbound message pacing, including a minimum interval between messages and possible coordination with reply chunking.

## Best Possible Solution

Add a Feishu-owned per-account outbound pacing layer, expose it in schema/manifest/docs, and apply it consistently to burst-prone delivery paths while leaving broad cross-provider limits and reactive retry/backoff to their separate trackers.

## Reproduction Assessment

Not applicable as a bug reproduction because this is an enhancement/config request. Source inspection on current main gives high confidence that the requested Feishu pacing config and shared outbound pacing helper are absent.

## Solution Assessment

Yes directionally: a Feishu-owned per-account pacing helper is narrower than a broad core limiter for this request. The exact public config name, defaults, inheritance behavior, and whether card updates/media/comment APIs share the same budget need maintainer scoping.
