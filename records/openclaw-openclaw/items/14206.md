## Summary

## What This Changes

Add configurable proactive pacing for Feishu outbound messages to avoid API rate limits during bursty, chunked responses.

## Best Possible Solution

Keep the shared bounded retry implementation as the canonical default and revisit proactive pacing only with post-fix failure evidence that cannot be solved by tuning the existing recovery behavior.

## Reproduction Assessment

No on current main: the original burst-induced delivery failure was reproduced by the related fix PR before the change, while its live after-fix runs recovered 20/20 and 100/100 concurrent sends through the current retry path.

## Solution Assessment

No, a new minimum-interval configuration is not currently the best solution because the merged shared retry layer fixes the observable delivery failure without adding another compatibility-sensitive setting; reopen the design only if failures persist after that fix.
