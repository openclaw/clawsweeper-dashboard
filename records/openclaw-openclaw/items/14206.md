## Label Justifications

- `impact:message-loss`: Bursting Feishu sends can trigger service limits and cause user-visible replies or chunks to be delayed or lost when delivery fails.

## Summary

Keep open. Current main has reactive Feishu retry/backoff for rate-limit responses, but it still lacks the requested proactive Feishu outbound pacing/minimum-interval config and the chunked reply, comment reply, outbound, media, and card paths can still burst sends without a shared per-account limiter.

## What This Changes

The issue requests configurable Feishu channel message pacing, including a minimum interval between outbound messages and behavior that accounts for chunked LLM replies.

## Best Possible Solution

Add a Feishu-owned per-account outbound pacing layer, configurable at top level and per account, and apply it consistently across Feishu text, card, media, comment, and streaming-card send surfaces while keeping broad core rate limiting tracked separately.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.8 lack a Feishu pacing/minimum-interval config, and the chunked reply, comment reply, and outbound paths send chunks or payloads directly. I did not run a live Feishu ban scenario in this read-only review.

## Solution Assessment

Yes. A narrow Feishu-plugin outbound pacing layer is the best current direction; the existing reactive retry helper is useful adjacent work but does not prevent bursts, and the broad core rate-limit issue should stay separate.
