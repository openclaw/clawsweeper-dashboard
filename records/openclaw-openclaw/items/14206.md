## Summary

Keep open. Current main and shipped v2026.5.20 still lack a Feishu outbound minimum-interval/pacing config, while chunked replies, comment replies, message-tool outbound, media, and initial streaming-card sends can call Feishu delivery APIs without a shared pacing gate; related broad rate-limit and reactive retry issues do not supersede this proactive Feishu-specific request.

## What This Changes

The issue requests a configurable minimum interval or pacing layer for Feishu outbound message sends so bursty chunked replies do not trigger Feishu flow control.

## Best Possible Solution

Add a Feishu-plugin-owned per-account outbound pacing helper, expose it in schema, manifest, docs, and changelog, and apply it consistently to burstable Feishu delivery surfaces while leaving broad core rate limiting and reactive retry/backoff on their own trackers.

## Reproduction Assessment

## Solution Assessment

Yes. A Feishu-owned per-account outbound pacing helper is the narrow maintainable direction; the broader rate-limit and retry/backoff trackers should remain separate unless maintainers intentionally merge the scopes.
