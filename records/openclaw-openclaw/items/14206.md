## Summary

Keep open: current main still lacks a Feishu outbound pacing/minimum-interval setting, and Feishu reply, comment, message-tool outbound, media, and streaming-card delivery paths still send through direct API calls without a shared per-account pacing helper. The broader rate-limit tracker and the Feishu retry/backoff tracker are useful related context, but neither supersedes this proactive Feishu-specific pacing request.

## What This Changes

The issue requests configurable Feishu channel message rate limiting, including a minimum interval between outbound messages and possible interaction with response chunking.

## Best Possible Solution

Add a Feishu-owned per-account outbound pacing layer, expose it in schema/manifest/docs, and apply it consistently to burst-prone delivery paths while leaving broad provider limits and reactive retry/backoff to their own trackers.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high confidence that current main lacks the requested Feishu pacing config and sends chunked/outbound messages without a shared minimum-interval helper.

## Solution Assessment

Yes. The narrow maintainable solution is a Feishu plugin pacing helper with top-level and per-account config, not a generic core limiter or a retry/backoff-only change.
