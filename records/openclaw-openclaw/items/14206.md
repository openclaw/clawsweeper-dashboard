## Summary

Keep this issue open. Current main now recovers many Feishu rate-limit rejections through bounded retry, but it still lacks the requested proactive outbound pacing configuration, and no canonical issue or viable PR supersedes that narrower channel-specific work.

## What This Changes

Add a configurable minimum interval between Feishu outbound messages to reduce bursty chunk delivery.

## Best Possible Solution

Implement one compatibility-preserving Feishu-owned scheduler shared by burst-capable outbound paths, keyed by account and the relevant destination, while retaining current reactive retry for residual rate-limit responses.

## Reproduction Assessment

Yes at source level: chunked delivery can issue consecutive Feishu calls, proactive pacing is absent, and current main only backs off after a rejection. The merged retry PR also provides live burst evidence, although this review did not repeat that live test.

## Solution Assessment

Yes, provided pacing is centralized at the Feishu account/API boundary with destination isolation and a backward-compatible default. Changing global LLM segmentation or inserting separate sleeps in each dispatcher would be broader and more prone to drift.
