## Summary

Keep this issue open. Current main has reactive Feishu retry handling, but successful chunked and other outbound sends still lack proactive pacing, so the central request remains distinct and necessary.

## What This Changes

Add configurable per-account Feishu outbound message pacing across chunked replies and other burst-capable delivery paths.

## Best Possible Solution

Retain bounded retry for residual throttling and add one Feishu-owned, per-account outbound scheduler at the lowest common send boundary, with top-level/per-account configuration, manifest and docs coverage, and tests proving all burst-capable callers share it.

## Reproduction Assessment

Yes at source level. Current main permits immediate sequential successful sends, while its retry delay starts only after Feishu reports throttling; this read-only review did not perform a new live stress run.

## Solution Assessment

Yes, with a narrower boundary than the original suggestion. Central Feishu API pacing is maintainable, while changing core LLM response segmentation is unnecessary and should remain out of scope.
