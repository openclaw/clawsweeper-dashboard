## Summary

Keep open. Current main still has no Feishu outbound minimum-interval or pacing setting, and the central Feishu reply, comment, message-tool, media, and streaming-card paths can still make consecutive Feishu API calls without a shared configurable per-account gate. The related open trackers #13615 and #70879 cover broader external API limiting and retry/backoff after rate-limit failures, but they do not supersede this narrower proactive Feishu send-spacing request.

## What This Changes

The issue requests configurable outbound message pacing for the Feishu channel so bursty chunked LLM responses do not trigger Feishu server flow control.

## Best Possible Solution

Add a Feishu-owned, default-off per-account outbound pacing layer resolved from top-level and account config, then wire it through every Feishu delivery path that can burst API calls. Document and expose the setting in the Feishu schema, plugin manifest, and public channel docs; leave broad cross-provider quotas to #13615 and retry-after-rate-limit recovery to #70879.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists: use a small Feishu textChunkLimit or a long/message-tool/media response that splits into multiple sends, then inspect the current loops to see consecutive Feishu API calls with no configured minimum interval.

## Solution Assessment

Yes. A Feishu plugin-owned pacing gate is the narrowest maintainable fix because the reported failure is Feishu outbound delivery frequency; a core-wide limiter is broader than this issue, while retry-only handling would not prevent the initial burst.
