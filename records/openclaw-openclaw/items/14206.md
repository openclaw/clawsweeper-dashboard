## Summary

Keep open. Current main and v2026.5.22 still lack a Feishu outbound minimum-interval/pacing config, and the related broad rate-limit tracker https://github.com/openclaw/openclaw/issues/13615 plus reactive retry tracker https://github.com/openclaw/openclaw/issues/70879 do not supersede this Feishu-specific proactive pacing request.

## What This Changes

The issue requests configurable Feishu message pacing to avoid Feishu API flow-control bans when OpenClaw sends many chunked or sequential outbound messages.

## Best Possible Solution

Add a Feishu-plugin-owned per-account outbound pacing helper, expose it in schema, manifest, docs, and changelog, and apply it consistently to burstable Feishu delivery surfaces while leaving broad core rate limiting and reactive retry/backoff on separate trackers.

## Reproduction Assessment

Yes, at source level for the missing capability. Current main has no Feishu pacing config in schema/docs/manifest, and reply, comment, outbound, media, and initial streaming-card paths can call Feishu APIs without a shared minimum-interval gate.

## Solution Assessment

Yes. A Feishu-owned per-account outbound pacing helper is the narrow maintainable direction; the broader rate-limit and retry/backoff trackers should remain separate unless maintainers intentionally merge those scopes.
