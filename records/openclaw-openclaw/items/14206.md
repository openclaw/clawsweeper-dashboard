## Summary

Keep open. Current main and v2026.5.20 still lack a Feishu-owned outbound minimum-interval/pacing config, and the related broader rate-limit, retry/backoff, and streaming-throttle items do not supersede this proactive Feishu delivery request.

## What This Changes

The issue requests configurable proactive message pacing for Feishu outbound sends to avoid Feishu flow-control bans when OpenClaw emits many short messages.

## Best Possible Solution

Add a Feishu-plugin-owned per-account outbound pacing helper, expose it in schema, manifest, and docs, and apply it consistently to burstable Feishu delivery surfaces while leaving broad core rate limiting and reactive retry/backoff on their own trackers.

## Reproduction Assessment

Yes, at source level for the missing capability. Current main has no Feishu pacing config in schema/docs/manifest, and the reply, comment, outbound, send, media, and streaming-card paths can call Feishu APIs without a shared minimum-interval gate.

## Solution Assessment

Yes. A Feishu-owned outbound pacing helper is the narrow maintainable direction; the broader core rate-limit tracker and Feishu retry/backoff tracker should remain separate unless maintainers intentionally merge the scopes.
