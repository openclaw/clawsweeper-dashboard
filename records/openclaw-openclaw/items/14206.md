## Summary

Keep open. Current main and the latest shipped release still lack a Feishu outbound minimum-interval/pacing setting, and the related open trackers cover broader cross-provider rate limits and reactive retry/backoff rather than this proactive Feishu message pacing request.

## What This Changes

The issue requests configurable outbound pacing for Feishu messages so bursty chunked replies do not trip Feishu API frequency limits.

## Best Possible Solution

Add a Feishu-plugin-owned per-account outbound pacing helper, expose it in schema/manifest/docs at top level and per account, and route every burst-capable Feishu delivery path through it while leaving broader rate-limit and retry policy to the linked trackers.

## Reproduction Assessment

Not applicable as a bug reproduction. This is an enhancement request, and source inspection on current main gives high confidence that the requested Feishu outbound pacing config and delivery gate are absent.

## Solution Assessment

Yes. A Feishu-owned pacing layer is the narrowest maintainable direction for this channel-specific frequency problem; a core global limiter belongs in the broader tracker instead.
