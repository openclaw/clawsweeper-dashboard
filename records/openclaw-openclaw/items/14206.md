## Summary

Keep this issue open: current main now retries recognized Feishu rate-limit responses, but it still does not proactively pace bursty message creation against Feishu’s documented recipient and group limits, and no other issue or PR owns that narrower work.

## What This Changes

Add configurable proactive pacing for outbound Feishu messages to avoid server rate-limit rejection during bursty sends.

## Best Possible Solution

Add one Feishu-owned scheduler around message create/reply requests, keyed by account and recipient or chat, exposed through the shared top-level/per-account configuration and composed with existing bounded retry; do not couple delivery pacing to LLM response segmentation.

## Reproduction Assessment

Yes. Feishu’s official contract documents a 5 QPS recipient/group limit, while current source permits chunked and concurrent message sends without proactive spacing; this establishes the gap without claiming a live ban reproduction.

## Solution Assessment

Yes, with a narrower boundary than the issue’s segmentation suggestion: recipient-scoped pacing at the shared message API layer is maintainable and covers all callers, while changing LLM segmentation would mix formatting policy with transport reliability.
