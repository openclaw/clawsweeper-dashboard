## Summary

Keep open: current main still lacks a Feishu outbound pacing/minimum-interval configuration, and the Feishu reply, comment, outbound, media, and streaming-card paths still send directly through Feishu APIs without a shared proactive pacing helper. The broader rate-limit and retry/backoff trackers are related context but do not supersede this Feishu-specific outbound pacing request.

## What This Changes

The issue requests a configurable minimum interval between Feishu outbound messages, ideally also influencing how long LLM responses are chunked before sending.

## Best Possible Solution

Add a Feishu-owned per-account outbound pacing layer, expose it in schema/manifest/docs, and apply it consistently to burst-prone delivery paths while leaving broader provider limits and retry/backoff to their own open trackers.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high confidence that current main lacks the requested Feishu pacing config and that chunked/outbound paths send without a shared minimum-interval helper.

## Solution Assessment

Yes. The narrow maintainable path is a Feishu plugin pacing helper with top-level and per-account config, not a generic core limiter or a retry/backoff-only change.
