## Summary

Keep open: current main and v2026.6.6 only retry after Feishu send rate-limit responses; neither has a proactive outbound pacing/minimum-interval config or scheduler, and no searched PR supersedes this Feishu-specific request.

## What This Changes

Request a Feishu channel message-rate-limit/minimum-interval configuration that reduces bursty outbound sends from chunked LLM replies.

## Best Possible Solution

Add one Feishu-owned outbound scheduler around message create/reply requests, keyed by account and recipient/chat, exposed through shared top-level and per-account config, and compose it with existing bounded retry without coupling delivery pacing to LLM segmentation.

## Reproduction Assessment

Yes. The source-level path is high-confidence: Feishu documents same-user/group send QPS limits, while current Feishu code permits chunked and concurrent message sends without proactive spacing.

## Solution Assessment

Yes, with a narrower boundary than changing LLM segmentation. Recipient/chat-scoped pacing at the Feishu message API layer covers the burst source cleanly and preserves formatting/chunking policy as a separate concern.
