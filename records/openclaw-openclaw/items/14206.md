## Summary

Keep open. Current main and v2026.5.19 still lack a Feishu outbound minimum-interval or pacing configuration, and the related broad rate-limit and Feishu retry/backoff threads do not supersede this Feishu-specific proactive pacing request.

## What This Changes

Request a configurable Feishu channel pacing layer that enforces a minimum interval between outbound message sends and can influence chunked response delivery.

## Best Possible Solution

Add a Feishu-owned per-account outbound pacing layer, configurable at top level and per account, and apply it consistently to burstable Feishu delivery surfaces while keeping broad cross-provider rate limiting and reactive retry/backoff on their separate trackers.

## Reproduction Assessment

Yes, at source level for the missing capability. Current main has no Feishu pacing config in schema/docs/manifest, and the reply, comment, outbound, send, media, and streaming-card paths can call Feishu APIs without a shared minimum-interval gate; I did not run a live Feishu rate-limit scenario.

## Solution Assessment

Yes. A Feishu-plugin-owned outbound pacing helper is narrower and more maintainable than broad core rate limiting for this issue, while retry/backoff after Feishu rate-limit errors should remain tracked separately at https://github.com/openclaw/openclaw/issues/70879.
