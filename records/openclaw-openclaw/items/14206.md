## Summary

Keep open. Current main and the latest shipped release still lack a Feishu-owned outbound minimum-interval/pacing configuration, and the related broad rate-limit, retry/backoff, and streaming-throttle threads do not supersede this proactive Feishu delivery request.

## What This Changes

The issue requests configurable Feishu outbound message pacing/minimum intervals to avoid service-side flow control when OpenClaw emits many short replies.

## Best Possible Solution

Add a Feishu-plugin-owned per-account outbound pacing helper, expose it in schema/manifest/docs, and apply it consistently to burstable Feishu delivery surfaces while leaving broad core rate limiting and reactive retry/backoff on their own trackers.

## Reproduction Assessment

Yes, at source level for the missing capability. Current main has no Feishu pacing config in schema/docs/manifest, and the reply, comment, outbound, send, media, and streaming-card paths can call Feishu APIs without a shared minimum-interval gate; I did not run a live Feishu rate-limit scenario.

## Solution Assessment

Yes. A Feishu-plugin-owned outbound pacing helper is the narrow maintainable direction, while broad cross-provider limits and Feishu retry/backoff after rate-limit errors should remain tracked separately.
