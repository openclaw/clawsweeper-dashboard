## Summary

Keep open: current main still lacks a Feishu outbound pacing or minimum-interval config/helper, and the inspected reply, comment, message-tool, card, media, and streaming-card paths can still send directly to Feishu APIs. The broader rate-limit tracker at https://github.com/openclaw/openclaw/issues/13615 and the Feishu retry/backoff tracker at https://github.com/openclaw/openclaw/issues/70879 are related, but they do not supersede this proactive Feishu-specific pacing request.

## What This Changes

Request a configurable Feishu outbound minimum-interval/pacing layer to avoid Feishu API bans when OpenClaw splits LLM responses into multiple messages.

## Best Possible Solution

Add a Feishu-owned per-account outbound pacing layer, expose it in schema/manifest/docs, and apply it consistently to burst-prone delivery paths while leaving broad core limits and reactive retry/backoff to their separate trackers.

## Reproduction Assessment

Not applicable as a bug reproduction because this is an enhancement/config request. Source inspection on current main gives high confidence that the requested Feishu pacing config and shared outbound pacing helper are absent.

## Solution Assessment

Yes directionally: a Feishu-owned per-account pacing helper is narrower than a broad core limiter for this request. The public config name, defaults, inheritance, and endpoint coverage still need maintainer scoping before implementation.
