## Summary

Current source shows the phase-report path and the Dream Diary write path are separate: phase completion can succeed while the best-effort diary narrative is skipped, fails, or runs detached after cron completion. The report is plausible and source-localized, but it is not fixed on `main` and was not reproduced end to end, so it should stay open for maintainer investigation.

## What This Changes

The issue reports that active dreaming cron runs still write phase outputs, but `DREAMS.md` stopped appending Dream Diary entries and is not recreated after deletion.

## Best Possible Solution

Keep the issue open and investigate the cron-triggered detached Dream Diary path with real gateway logs, then either make the diary failure visible/actionable or adjust cron handling so successful phase reports cannot leave the UI stale without diagnostics.

## Reproduction Assessment

No, not as a high-confidence end-to-end failure: current tests show `DREAMS.md` creation succeeds when the narrative subagent returns text. Source inspection does show a plausible path where phase outputs and cron success happen independently from best-effort detached diary generation.

## Solution Assessment

Unclear. A narrow code fix may be possible, but the best behavior needs maintainer judgment on whether cron should await diary writes, surface detached narrative failures in status/UI, or preserve best-effort semantics and improve diagnostics.
