## Summary

## What This Changes

## Best Possible Solution

Rework the node agent.request path to persist the media-bearing user turn through the existing user-turn transcript helpers, preserve imageOrder, and keep the linked issue open until that durable fix lands.

## Reproduction Assessment

Yes, source-level. Current main's agent.request path receives parsed attachment media but does not carry offloadedRefs into the durable transcript media/user-turn recorder path, while chat.send shows the expected persistence contract.

## Solution Assessment

No, not as implemented. The maintainable fix should write the media-bearing user turn through the existing transcript append/recorder path and preserve parsed image ordering, rather than only emitting an in-memory transcript update.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Persist the media-bearing turn, not only an update:** `src/gateway/server-node-events.ts:405`
  - body: `emitSessionTranscriptUpdate()` only notifies listeners; it does not append or rewrite the JSONL transcript. Because this handler never supplies a media-bearing user message to the existing transcript append/recorder path, the persisted history can still contain the plain text turn without `MediaPath`/`MediaPaths` after reload, which is the bug this PR is meant to fix.
  - confidence: 0.92
- **[P2] Preserve parsed image order when building MediaPaths:** `src/gateway/server-node-events.ts:358`
  - body: `parseMessageWithAttachments()` returns `imageOrder` so inline and offloaded images keep the user's original order, but this helper saves all inline images first and then appends every offloaded ref. A request whose first image is offloaded and second is inline will record reversed `MediaPaths`/`MediaTypes`; interleave saved entries using `imageOrder` like the `chat.send` path.
  - confidence: 0.88
