## Summary

Keep open: the PR targets a real doctor false warning from https://github.com/openclaw/openclaw/issues/82250, but the new regex is too broad because launchd KeepAlive dict keys are value-sensitive, and external real behavior proof is still missing.

## What This Changes

The PR broadens `src/daemon/service-audit.ts` so launchd service audit treats a `KeepAlive` dictionary containing `SuccessfulExit` or `Crashed` as present in addition to `<true/>`.

## Best Possible Solution

Accept only value-correct crash-recovery dict forms, such as `SuccessfulExit=false` and `Crashed=true`, alongside `<true/>`, and add focused service-audit regression tests.

## Reproduction Assessment

Yes. Current main only matches boolean KeepAlive, and the PR regex source path also shows a false-positive reproduction for value-inverted dicts such as `SuccessfulExit=true`.

## Solution Assessment

No, not yet. Accepting the dict form is the right layer for this PR, but the matcher needs to check key/value polarity and add focused audit tests before it is the best fix.

- **[P2] Require the KeepAlive dict boolean value:** `src/daemon/service-audit.ts:236`
  - body: The launchd KeepAlive dict keys are polarity-sensitive: `SuccessfulExit=true` restarts after clean exits, `SuccessfulExit=false` restarts after non-zero exits, and `Crashed=false` inverts the crash condition. ([keith.github.io](https://keith.github.io/xcode-man-pages/launchd.plist.5.html)) This regex only checks for the key name, so `doctor` will accept inverse policies like `SuccessfulExit=true` or `Crashed=false` and stop warning even though they do not provide the crash-recovery policy this PR is trying to bless. Please match the key/value pairs and add accepted/rejected service-audit cases.
