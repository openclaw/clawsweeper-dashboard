## Summary

Keep open: the PR targets a real doctor false positive for launchd `KeepAlive` dict plists, but the patch currently treats condition-key presence as sufficient and can pass policies with the wrong restart semantics; it also lacks external after-fix real behavior proof.

## What This Changes

The PR broadens the macOS launchd plist audit so `KeepAlive` passes when represented as a dict containing `SuccessfulExit` or `Crashed`, not only as a bare `<true/>`.

## Best Possible Solution

Parse or tightly match the accepted `KeepAlive` dict key/value pairs, add service-audit coverage for accepted and rejected launchd policies, then require redacted real doctor/status proof before merge.

## Reproduction Assessment

Yes, source-backed: current main reads the LaunchAgent plist and only matches `<key>KeepAlive</key><true/>`, so a dict-form plist like the related issue workaround will produce `launchd-keep-alive`. I did not run a live macOS doctor smoke in this read-only review.

## Solution Assessment

No. The narrow direction is useful, but the implementation should validate the dict condition values and cover accepted/rejected policies before it is the best fix.

Full review comments:

- **[P2] Check the KeepAlive condition value:** `src/daemon/service-audit.ts:236`
  - body: The new regex treats any `SuccessfulExit` or `Crashed` key inside the `KeepAlive` dict as healthy, but launchd gives different semantics to the boolean value: `SuccessfulExit=true` restarts only after exit status zero, and `Crashed=false` restarts in the inverse of crash. This can make doctor pass a LaunchAgent that does not implement the crash-restart policy described in the PR. Match the expected boolean values and cover inverse cases in `service-audit.test.ts`. ([keith.github.io](https://keith.github.io/xcode-man-pages/launchd.plist.5.html))
