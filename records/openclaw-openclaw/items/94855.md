## Summary

Keep open: this PR targets a valid cron delivery bug, but the current patch is not merge-ready because its new warning check can clear fatality from only the last error payload, proof is still unit-test-only, and maintainers need to choose one canonical fix among overlapping PRs.

## What This Changes

The PR changes `resolveCronPayloadOutcome` so recovered cron tool-warning text can be treated as non-fatal and final assistant text can replace warning payloads even when the channel does not prefer final assistant text.

## Best Possible Solution

Land one canonical narrow cron resolver fix that uses the stricter recovered-warning predicate, proves preference-false final-text delivery, and preserves fatal structured-error suppression for genuine errors.

## Reproduction Assessment

Yes at source level: current main can return before dispatch when a recovered cron tool warning remains a fatal structured payload, and the linked issue provides a real reduced scheduled Feishu repro. I did not run the live Feishu scheduled setup in this read-only review.

## Solution Assessment

No: the PR is a plausible partial fix, but using only the last error text is less safe than reusing the stricter recovered-warning predicate that requires all error payloads to match the cron tool-warning pattern. The best fix also needs focused regressions and real delivery proof.
