## Summary

Keep open: the Codex/MCP projection approach matches the current app-server path, but the new scope check currently treats an empty or invalid `codex.agents` value as unscoped, which can broaden a server to every Codex-backed agent.

## What This Changes

The PR adds Codex-specific MCP server config under `mcp.servers.<name>.codex`, filters Codex app-server user MCP projections by agent id, maps Codex approval-mode metadata, and updates docs/schema baselines.

## Best Possible Solution

Keep the Codex-owned projection seam, but make scoped MCP server lists fail closed and add regression coverage for empty, whitespace-only, and missing-agent-id cases before merge.

## Reproduction Assessment

Yes. Current main source shows all user `mcp.servers` are projected without agent filtering, and the PR head source shows the new empty/invalid `codex.agents` fail-open path.

## Solution Assessment

No, not yet. The selected seam is maintainable, but the scope filter must distinguish an omitted `agents` field from a present-but-empty or invalid one so scoping cannot broaden access.
