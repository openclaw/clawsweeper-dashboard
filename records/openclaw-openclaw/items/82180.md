## Summary

Keep open: this is a plausible Codex app-server MCP scoping fix, but the branch currently fails open for empty or unresolved `codex.agents` scopes and lets trusted-user approval overrides flow through the shared bundle-MCP normalizer.

## What This Changes

The PR adds `mcp.servers.<name>.codex` config for Codex MCP approval metadata and agent-specific projection into Codex app-server thread config.

## Best Possible Solution

Keep the new scoping fail-closed, distinguish absent scope from empty scope, and apply approval overrides only at an explicitly reviewed owner boundary with regression tests and real app-server proof.

## Reproduction Assessment

Yes, at source level: on the PR branch, `codex.agents: []` normalizes to no usable list and is treated as globally allowed. Source inspection also shows the approval-mode override is read in the shared MCP normalizer used by bundle MCP config.

## Solution Assessment

No. The direction is useful, but the current implementation should fail closed for scoped servers and keep trusted-user approval policy separate from plugin-bundled MCP normalization unless maintainers explicitly approve that broader contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Fail closed for empty or unresolved agent scopes:** `src/agents/cli-runner/bundle-mcp-codex.ts:46-47`
  - body: This allows a scoped MCP server whenever `codex.agents` normalizes to no entries or the caller lacks an agent id. Since the added schema accepts `agents: []`, a config that appears to expose the server to no agents becomes global instead, which defeats the isolation this PR is adding. Include the server only when the `agents` key is absent or the current agent is listed, and add coverage for empty lists and missing agent ids.
  - confidence: 0.91
- **[P2] Keep user approval overrides out of bundle MCP manifests:** `src/agents/codex-mcp-config.ts:70-72`
  - body: `normalizeCodexMcpServerConfig` is also used for enabled bundle MCP config, not only `cfg.mcp.servers`, so reading `server.codex` here lets a plugin manifest emit Codex `default_tools_approval_mode` for its MCP server even though the new schema/docs describe a trusted user-config knob. Apply this override in the user projection path or add an explicit plugin policy contract before allowing bundle manifests to change native approvals.
  - confidence: 0.85
