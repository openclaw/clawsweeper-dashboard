## Summary

Keep open. Current main still has the stale HTTP-route registry path described in the report, and open PR #73877 already targets the same bug with closing syntax, so this issue should remain open until that PR or an equivalent fix lands.

## What This Changes

Workspace plugin HTTP routes registered with `api.registerHttpRoute(...)` should dispatch to their handlers instead of returning 404 after plugin startup or deferred reload.

## Best Possible Solution

Update the gateway/plugin runtime HTTP-route surface so it resolves the registry that owns workspace and deferred plugin routes, with regression coverage for the reported dispatch path and no weakening of the separate channel registry pinning behavior.

## Reproduction Assessment

Yes. The issue gives concrete Docker/workspace-plugin curl steps, and current-main source inspection shows the same stale pinned-registry path that can leave the dispatcher with an empty route surface.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the gateway/plugin runtime route-registry surface rather than in docs or plugin workarounds; #73877 is the active implementation candidate for that direction.
