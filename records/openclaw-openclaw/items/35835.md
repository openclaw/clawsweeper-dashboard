## Summary

Keep open: current main and v2026.6.5 still route non-image files through UTF-8 text decoding, while the shared model/session contract only represents text and images. The broader native-media work in https://github.com/openclaw/openclaw/pull/78797 is open, conflicted, and its discussion confirms the same unresolved content-transport boundary, so it is neither an implemented fix nor a safe canonical replacement. This remains a valid feature, but maintainers must choose the typed audio contract, provider capability/fallback behavior, and local-file security limits before implementation.

## What This Changes

Allow the read tool to return native audio attachments to audio-capable models instead of decoding audio file bytes as text.

## Best Possible Solution

Define one typed native-audio content block and provider transport contract, with MIME sniffing, bounded payloads, explicit model capability checks, and a clear unsupported-model result; then reuse that contract from both the read tool and inbound media flows without provider-specific parsing in the tool.

## Reproduction Assessment

Yes, from source with high confidence: current main recognizes images only and decodes every other readable file buffer as UTF-8 text, while its result type cannot carry audio.

## Solution Assessment

No, adding audio extensions beside image detection is not the best solution; the narrow maintainable path starts with a shared typed audio and provider transport contract, then integrates read through that boundary.
