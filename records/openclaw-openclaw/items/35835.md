## Summary

Keep open: current `main` and latest release still have an image-only read/tool-result attachment path, and the related audio-native work is either closed unmerged or still open as broader product design. This remains valid, but it needs maintainer decisions on the typed audio content contract, provider capability gating, fallback behavior, and raw-audio security policy before a fix PR is safe.

## What This Changes

Request native audio-file handling in the read tool so supported models can receive audio as multimodal attachments instead of raw binary text.

## Best Possible Solution

Keep the read-tool acceptance criteria open while maintainers converge it with the broader native media design: one typed audio content block contract, provider support matrix, MIME/size policy, workspace guards, and explicit transcript fallback semantics.

## Reproduction Assessment

Yes, at source level: the read tool and shared LLM content contracts only create and transport text/image blocks, while audio remains represented elsewhere as transcription output. I did not run a live audio read session because this review is read-only and the source path is already explicit.

## Solution Assessment

No, the issue's simple extension-list proposal is not the best complete fix. The maintainable solution needs a typed audio media contract across read/tool results and provider serializers, with capability gating, size/MIME validation, workspace guards, and explicit fallback behavior.
