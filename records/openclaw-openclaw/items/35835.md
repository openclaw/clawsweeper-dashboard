## Summary

Keep open. Current main still has image-only read/prompt attachment plumbing; existing audio support is transcription/media-understanding or adjacent media handling, not native read-tool audio attachment parity.

## What This Changes

The issue requests native audio-file handling in the read tool so audio files are passed as model attachments instead of binary text.

## Best Possible Solution

Add a first-class typed audio attachment path shared by read results and prompt submission, with provider capability checks, MIME and size validation, workspace/sandbox egress controls, and a clear unsupported-model fallback.

## Reproduction Assessment

Yes, at source level: current main only normalizes and submits image attachments, while audio files have no typed read result or prompt submission path. I did not run a live provider session because the missing path is visible in source.

## Solution Assessment

No, not as a simple extension-list patch to read. The maintainable fix needs a typed audio attachment contract plus provider capability and local-file egress policy rather than reusing image-specific plumbing directly.
