## Summary

Keep open: current main and v2026.6.8 still support only text/image read-tool content, and audio files still fall into the text decode path; the broader native-audio tracker overlaps but does not remove this read-tool-specific API/security decision.

## What This Changes

Request native audio-file handling in the read tool so supported models receive audio bytes as multimodal attachments instead of raw text.

## Best Possible Solution

Define a shared typed audio content contract with capability checks, provider serialization, payload/egress policy, unsupported-model behavior, and read-tool regression coverage before wiring audio into read results.

## Reproduction Assessment

Yes. Source inspection shows the read tool only detects image MIME types and otherwise decodes file bytes as text, so an audio file reaches the text branch.

## Solution Assessment

No for a read-tool-only extension-list patch. The maintainable solution needs a shared audio content/provider contract and security/session policy before read can return audio blocks.
