## Summary

Keep open: current main and the latest release still have an image-only native read/prompt media boundary, while audio media understanding remains transcript-oriented. The related Google/Gemini audio PR was closed unmerged and only covered a narrower headless @file flow, so it does not resolve read-tool audio parity.

## What This Changes

Request native audio file support in the read tool by detecting audio extensions and forwarding audio as model multimodal attachments instead of raw binary text.

## Best Possible Solution

Define one typed native media attachment contract for read results and prompt refs, gated by provider capability and raw-media egress policy, with transcript fallback behavior, docs, focused tests, and live provider proof.

## Reproduction Assessment

Yes, at source level: current main's read wrapper and prompt submission paths only normalize and pass image attachments, while audio media understanding produces audio.transcription text. I did not run a live provider because this review is read-only and the missing contract is visible in source.

## Solution Assessment

No, not as a simple extension-list patch to read. The maintainable path needs a typed native media contract, provider gates, raw-audio egress policy, payload limits, docs, and real provider proof.
