## Summary

Keep open. Current main still only handles images in the read and prompt attachment path; existing audio support is transcription/media-understanding, and the related open Google headless PR is narrower than read-tool native audio parity.

## What This Changes

Request first-class audio-file handling in the agent read tool so audio files are sent to capable models as multimodal attachments instead of binary text.

## Best Possible Solution

Add a first-class typed audio attachment path shared by read results and prompt submission, with provider capability gating, MIME and size validation, workspace/sandbox egress controls, and a clear unsupported-model fallback.

## Reproduction Assessment

Yes, for the current-main absence of the requested capability: source inspection shows audio extensions cannot enter the image-only read normalization, prompt detection, or `activeSession.prompt(..., { images })` path. I did not run a live provider session because the missing path is visible in source.

## Solution Assessment

No, not as a simple extension-list patch on `read`. The maintainable solution needs a typed audio content contract plus provider capability checks and local-file egress policy rather than reusing image-specific plumbing directly.
