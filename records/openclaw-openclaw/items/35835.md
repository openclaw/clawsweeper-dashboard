## Summary

Keep this open: current main still decodes non-image audio files as UTF-8 text, and the closed related PR neither merged nor addressed the generic read-tool path. The request is valid, but implementation requires a maintainer-approved cross-provider audio content contract rather than a read-tool-only extension list.

## What This Changes

Allow the read tool to return supported audio files as native multimodal model input instead of decoding them as text.

## Best Possible Solution

Define a generic bounded AudioContent block plus model capability and provider serialization contracts, then have the read tool emit it for verified audio MIME types and return a deterministic unsupported-model result elsewhere.

## Reproduction Assessment

Yes, from source with high confidence. On current main, any audio file not detected as an image follows the UTF-8 text branch in `src/agents/sessions/tools/read.ts`, reproducing the reported binary-text behavior.

## Solution Assessment

No, extension detection and read-tool changes alone are not the best solution. The narrow maintainable design must first add a shared audio content type, explicit model capability checks, provider serializers, and bounded unsupported-model behavior.
