## Summary

Keep open. Current main still has image-only read and prompt attachment plumbing; existing audio support is transcription/media-understanding or delivery-adjacent, while open implementation PRs need product, provider-contract, and raw-media egress decisions before this can be safely merged.

## What This Changes

Request native audio-file support in the read tool so audio files are forwarded as multimodal model attachments instead of raw binary text.

## Best Possible Solution

Define a first-class audio attachment/read-result contract with capability checks, MIME and size validation, schema/docs/policy coverage, and workspace or sandbox egress controls, then converge the open PRs on that design.

## Reproduction Assessment

Yes, at source level: current main has image-only read normalization and prompt submission, while audio media handling produces transcripts rather than native audio prompt parts. I did not run a live audio-capable provider session because the missing path is visible in source.

## Solution Assessment

No, not as a simple extension-list patch to read. The maintainable fix needs a typed audio media contract plus provider capability gates and local-file egress policy.
