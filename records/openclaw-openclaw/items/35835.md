## Summary

Keep open: current main and the latest release still make the read tool text/image-only, and the related native-audio work is overlapping but not a landed or safe canonical fix. This needs maintainer product/security review because the fix crosses read-tool output, shared LLM content types, provider capability routing, raw-media egress, and session/audit semantics.

## What This Changes

The issue requests audio-file support in the read tool so audio files are passed to capable models as multimodal audio attachments instead of decoded as raw text.

## Best Possible Solution

Define a shared typed audio content contract with capability checks, provider serialization, explicit raw-media egress policy, unsupported-model behavior, and session/audit semantics before wiring audio into read results.

## Reproduction Assessment

Yes, source-level. Current main only classifies images before falling back to text decoding, and the normalized tool-result contract has no audio block.

## Solution Assessment

No for a read-tool-only extension-list patch. The maintainable solution needs shared audio content/provider contracts plus security, fallback, and session/audit policy first.
