## Summary

Keep open. Current main still has image-only read-tool media normalization and text/image-only provider replay, so audio-file read parity is not implemented and needs a small cross-provider contract decision rather than cleanup closure.

## What This Changes

Add native audio-file support to the read tool so common audio files are sent to capable models as multimodal attachments instead of raw binary text.

## Best Possible Solution

Define a small audio content-block contract for read/tool results, add audio MIME and size handling in the read path, gate forwarding by explicit model/provider audio support, return a clear unsupported-model result, and cover the read plus provider conversion paths with focused tests.

## Reproduction Assessment

Yes. Source inspection is enough: current read handling only normalizes image blocks, and the inspected provider replay paths only serialize text/image tool-result blocks, so current main cannot deliver a read audio file as a native model attachment.

## Solution Assessment

Unclear for implementation. The requested direction is valid, but the best fix needs maintainer agreement on the audio tool-result contract and provider capability semantics; the narrower headless Gemini @file work does not solve this issue.
