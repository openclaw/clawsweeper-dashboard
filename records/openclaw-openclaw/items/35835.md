## Summary

Keep open: current main still has image-only read/prompt attachment plumbing, while audio handling remains transcription-oriented and the related implementation threads still need provider-contract, config, and raw-media egress decisions.

## What This Changes

Request native audio-file handling in the read tool so audio files are sent as multimodal model attachments instead of raw binary text.

## Best Possible Solution

Define a first-class audio prompt/read attachment contract with MIME and size validation, provider capability gates, explicit raw-media egress policy, docs, and regression plus real provider proof.

## Reproduction Assessment

Yes, at source level: current main has image-only read and prompt submission paths, while audio media handling produces transcripts rather than native audio prompt parts. A live audio-capable provider run was not needed to see the missing contract.

## Solution Assessment

No, not as a simple extension-list patch to read. The maintainable solution needs a typed audio media contract, provider capability gates, workspace/sandbox egress limits, and docs before raw audio bytes are forwarded.
