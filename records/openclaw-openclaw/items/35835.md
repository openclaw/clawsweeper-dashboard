## Summary

Keep open: current main and v2026.6.10 still make the read tool text/image-only, and related native-audio work remains open or unmerged with unresolved provider, security, and content-contract questions.

## What This Changes

Request that read detect audio files and pass them as multimodal model attachments instead of decoding them as text.

## Best Possible Solution

Define shared typed audio content and provider serialization contracts with explicit raw-media egress policy, then wire read-tool audio through those contracts with regression coverage.

## Reproduction Assessment

Yes, source-level. Current main only classifies image files before falling back to text decoding, and the normalized LLM tool-result contract has no audio content block.

## Solution Assessment

No for a read-tool-only extension-list patch. The maintainable solution needs shared audio content/provider contracts plus security, fallback, and session/audit policy before read can return audio attachments.
