## Summary

Keep open: current main and the latest release still only support text and image content in the read tool, so audio files fall through to UTF-8 text output. The broader native-audio work at https://github.com/openclaw/openclaw/issues/51667 and https://github.com/openclaw/openclaw/pull/78797 overlaps, but it does not supersede this read-tool-specific request.

## What This Changes

The issue requests audio-file support in the read tool so audio-capable models can receive local audio as multimodal attachments instead of raw binary text.

## Best Possible Solution

Define a bounded shared AudioContent contract with provider capability checks, serialization, unsupported-model behavior, payload limits, and read-tool regression coverage before adding audio results to the read tool.

## Reproduction Assessment

Yes, from source inspection: audio files are not detected as images, so they enter the text branch in `src/agents/sessions/tools/read.ts` and are decoded as UTF-8. I did not run a live model/provider audio session in this read-only review.

## Solution Assessment

No, adding audio extensions to the read tool alone is not the best solution. The maintainable path needs a shared audio content type, provider serializers, model capability gates, and security/resource policy first.
