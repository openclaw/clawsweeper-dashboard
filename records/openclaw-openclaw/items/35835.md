## Summary

Keep open: current main still has image-only read/prompt attachment plumbing and transcription-oriented audio handling. The newer native-audio issues and PRs overlap on provider media contracts, but they do not fully replace this read-tool parity request.

## What This Changes

Add native audio-file handling to the read tool so audio files can be sent as model audio attachments instead of decoded as text.

## Best Possible Solution

Define a first-class typed audio prompt/read attachment contract with MIME sniffing, size limits, model/provider capability gates, explicit raw-media egress opt-in, docs, regression tests, and real provider proof, coordinated with the native-audio tracker and open native-media PR.

## Reproduction Assessment

Yes at source level: current main has image-only read and prompt attachment paths, while audio media handling is transcript-oriented. I did not run a live audio-capable provider flow because the missing contract is visible in source.

## Solution Assessment

No, not as a simple extension-list patch to read. The maintainable fix needs typed audio media blocks, provider capability gates, request-size and workspace/sandbox egress limits, config/policy alignment, and docs before raw audio bytes are forwarded.
