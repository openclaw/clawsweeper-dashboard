## Summary

Keep open: current main still has image-only read and prompt attachment plumbing, while audio handling remains transcription-oriented. The linked Gemini inline-audio PR is useful but narrower than read-tool multimodal audio parity and is still open.

## What This Changes

Request native audio-file support in the read tool so audio files are forwarded as multimodal model attachments instead of decoded as raw text.

## Best Possible Solution

Define a first-class typed audio prompt/read attachment contract with MIME sniffing, payload limits, provider/model capability gates, explicit raw-media egress policy, docs, regression tests, and real provider proof.

## Reproduction Assessment

Yes, at source level: current main and the pinned upstream read tool expose image-only multimodal read/prompt attachment paths, while audio is routed through transcription-oriented media understanding. I did not run a live audio-capable provider flow because the missing contract is visible in source.

## Solution Assessment

No, not as a simple extension-list patch to read. The maintainable path needs typed audio media blocks, provider capability gates, request-size and workspace/sandbox egress limits, opt-in policy, docs, and real provider proof.
