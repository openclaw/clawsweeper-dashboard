## Summary

Keep open: current main and v2026.6.5 still expose image-only native read and prompt attachments, and no merged canonical work provides audio parity. The feature is valid, but it needs maintainer product, provider-contract, and security design rather than a narrow read-wrapper patch.

## What This Changes

Add native audio-file handling to the agent read tool, analogous to image attachments, with clear unsupported-model behavior.

## Best Possible Solution

Define one typed native-audio attachment contract across agent command/session and provider surfaces, enforce existing workspace guards plus MIME sniffing and payload limits, and retain transcription as a separate explicit fallback rather than an implicit substitute.

## Reproduction Assessment

Yes, at source level. Current main's read operations, result normalization, command options, and model submission path expose image attachments but no native audio attachment path.

## Solution Assessment

No, adding audio extensions to the read wrapper alone is not the best solution. The maintainable fix requires a coordinated typed media contract, capability gating, MIME validation, payload policy, and provider serialization.
