## Summary

Keep open. Current main still has image-only read and prompt attachment plumbing; audio support exists through transcription/media-understanding, and the related open PR at https://github.com/openclaw/openclaw/pull/46607 is a narrower Google/Gemini headless `@file` path that does not solve read-tool audio parity.

## What This Changes

The issue requests native audio-file attachment support in the `read` tool so supported models can receive audio bytes instead of raw binary text.

## Best Possible Solution

Add a first-class typed audio attachment path shared by read results and prompt submission, with provider capability gating, MIME/size validation, workspace or sandbox egress controls, and a clear unsupported-model fallback.

## Reproduction Assessment

Yes, by source inspection: audio extensions cannot enter the image-only read normalization, prompt attachment detection, or `activeSession.prompt(..., { images })` path. I did not run a live provider session because the missing current-main path is visible in source.

## Solution Assessment

No, not as a simple extension-list patch on `read`. The maintainable solution needs a first-class audio content contract plus provider gating and local-file egress policy rather than reusing image-specific plumbing directly.
