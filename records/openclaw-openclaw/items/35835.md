## Summary

Keep open. Current main still has image-only read-tool normalization and text/image-only provider tool-result replay, so the requested audio-file read parity is not implemented. Related Google headless audio @file work is narrower and does not satisfy read-tool audio attachments in the active session contract.

## What This Changes

The issue requests native audio-file attachment support in the read tool, matching existing image attachment behavior for capable multimodal providers.

## Best Possible Solution

Define a small audio content-block contract for read/tool results, add audio MIME and size handling in the read path, forward audio only through providers and models that explicitly support audio input, return a clear unsupported-model result otherwise, and add focused read plus provider conversion regression tests.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: current read code only normalizes image blocks, and the provider replay paths only serialize text/image tool-result blocks, so reading an audio file cannot produce a native audio model attachment on current main.

## Solution Assessment

Unclear for implementation. The requested direction is valid, but the best fix needs maintainer API/product review because it crosses the core tool-result contract, read MIME handling, provider capability gates, and multiple transport schemas; the narrower Google @file approach is not sufficient for this issue.
