## Summary

Keep open: current main still has image-shaped read/prompt attachment plumbing and transcription-oriented audio handling; the related open items are narrower or overlapping and do not settle read-tool audio parity.

## What This Changes

Add native audio-file support to the read tool so local audio is sent as a multimodal model attachment instead of raw binary text, with a clear fallback when unsupported.

## Best Possible Solution

Define a typed audio prompt/read attachment contract shared by read, prompt refs, gateway attachments, and media-understanding, with MIME sniffing, size/workspace guards, provider capability gates, opt-in raw-media egress policy, docs, tests, and live provider proof.

Yes, at source level: current main's read wrapper and prompt submission paths only normalize/pass image attachments, while audio media understanding outputs audio.transcription text. I did not run a live provider because the missing contract is visible in source.

## Solution Assessment

No, not as a simple extension-list patch to read. The maintainable solution needs a typed audio media contract, provider gates, raw-media egress policy, payload limits, docs, and real provider proof.
