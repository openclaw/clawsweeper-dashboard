## Summary

Keep open. Current main still has image-only read-tool normalization and text/image-only tool-result provider transports, so audio files are not returned as native multimodal read attachments to the active model turn. Existing audio support covers adjacent STT, TTS, inbound media understanding, realtime voice, and narrower Google headless @file work, but not this read-tool parity request.

## What This Changes

Add read tool support for audio files as native multimodal attachments, with clear unsupported-model handling.

## Best Possible Solution

Define a small, stable audio content-block contract for read/tool results, add audio MIME and size handling in the read path, forward audio through providers that explicitly support audio input, return a clear unsupported-model result otherwise, and cover the read wrapper plus OpenAI/Google/Anthropic transport conversions with focused regression tests.
