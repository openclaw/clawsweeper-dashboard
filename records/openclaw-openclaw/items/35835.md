## Summary

Keep open. Current main still routes `read` through a text/image-only result contract, while audio support exists in separate media-understanding/transcription surfaces rather than native `read` attachments.

## What This Changes

Request native audio-file handling in the `read` tool so audio files are sent as multimodal model attachments instead of UTF-8 text.

## Best Possible Solution

Define a first-class audio content path for tool results and model capability checks, then extend `read` to sniff supported audio files and either attach them for audio-capable runtimes or return a clear transcription fallback.

## Reproduction Assessment

Yes, by source inspection: audio files cannot enter a native attachment path because current `read` handling and the pinned PI contract only support text and image result blocks. I did not run a live audio read invocation, but the source and dependency types establish the limitation.

## Solution Assessment

No for an immediate automated fix. Adding audio extensions to the image-specific wrapper would not be enough; the maintainable solution needs an audio content/runtime capability contract first.
