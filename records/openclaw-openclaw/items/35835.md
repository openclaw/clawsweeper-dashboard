## Summary

Keep open. Current main still treats `read` as a text/image-only tool: OpenClaw only normalizes image tool-result blocks, the pinned PI packages expose only text/image content for the read path, and the docs route audio files to transcription/media-understanding surfaces rather than native `read` attachments. The related #46601 is narrower and closed, but it does not cover this broader read-tool parity request.

## What This Changes

Request adding native audio-file handling to the `read` tool so audio files become multimodal model attachments instead of raw text.

## Best Possible Solution

Define a first-class audio content path for agent tool results and model capability checks, then extend `read` to sniff supported audio files and either attach them to audio-capable runtimes or return a clear fallback to the existing audio transcription/media-understanding path.

## Reproduction Assessment

Yes, by source inspection: audio files do not enter an audio attachment path because current `read` handling only detects image MIME types and otherwise treats file bytes as UTF-8 text. I did not run a live tool invocation, but the current code and pinned upstream package contract make the limitation clear.

## Solution Assessment

No for an immediate automated fix. The requested behavior is plausible, but the maintainable solution needs a defined audio content/runtime capability contract rather than only adding audio extensions to the existing image-specific read wrapper.
