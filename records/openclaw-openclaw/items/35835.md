## Summary

Keep open: current main and the latest release still have image-only read/prompt attachment plumbing, while audio remains a transcription-oriented media-understanding path. The related open native-audio work overlaps, but it does not yet provide a merged read-tool audio attachment contract.

## What This Changes

Request audio files read through the read tool be sent as native multimodal attachments, with clear unsupported-model handling, instead of returning binary text.

## Best Possible Solution

Define a shared typed audio prompt/read attachment contract across read, prompt refs, gateway attachments, and media-understanding, with MIME sniffing, workspace/sandbox and size guards, provider capability gates, opt-in raw-media egress, docs/tests, and live provider proof.

No, not as a simple extension-list patch to read. The maintainable path needs a typed audio media contract, provider gates, raw-media egress policy, payload limits, docs, and real provider proof.
