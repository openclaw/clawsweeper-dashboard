## Summary

Keep open. Current main still has a text/image-only `read` tool and prompt-media path; audio support exists in separate transcription/media-understanding surfaces, while the related native-audio PRs are adjacent, open, and not a shipped `read`-tool implementation.

## What This Changes

The issue requests native audio-file attachment support in the `read` tool, matching the existing image multimodal path.

## Best Possible Solution

Define a first-class audio content/tool-result and prompt-media contract with provider capability gating and fallback behavior, then extend `read` to sniff supported audio files and attach them for audio-capable runtimes or return a clear unsupported/transcription path.

## Reproduction Assessment

Yes, by source inspection. Current main can only produce text/image `read` results and only submits image prompt media, so an audio file cannot enter a native audio attachment path through `read`.

## Solution Assessment

No for a simple automated patch. The maintainable solution needs an audio content/runtime contract first; adding audio extensions to the current image-only MIME hook would still leave the tool result and provider prompt boundary unsupported.
