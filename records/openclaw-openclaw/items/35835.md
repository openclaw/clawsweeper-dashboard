## Summary

Keep open. Current main still has a text/image-only read and prompt-media path; audio support exists in transcription/media-understanding surfaces, while adjacent native-audio PRs remain open and do not ship read-tool parity.

## What This Changes

Requests native audio-file handling in the read tool so audio files are attached to audio-capable models instead of returned as raw binary text.

## Best Possible Solution

Define first-class audio content and prompt-media handling with capability gating and fallback behavior, then extend read to sniff supported audio files and attach them only for audio-capable runtimes.

## Reproduction Assessment

Yes, by source inspection. Current main normalizes read results only as text/images and submits only image prompt media, so an audio file read cannot become a native audio attachment.

## Solution Assessment

No for a simple extension-list patch. The maintainable solution needs a typed audio prompt-media contract and provider capability/fallback policy rather than reusing the current image-only MIME hook.
