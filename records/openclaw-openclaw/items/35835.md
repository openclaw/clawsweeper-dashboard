## Summary

Keep open. Current main still has image-only `read` and prompt attachment plumbing, while audio support is routed through transcription/media-understanding surfaces; the related open PR at https://github.com/openclaw/openclaw/pull/46607 covers a narrower Google/Gemini headless `@file` path and does not close broader read-tool audio parity.

## What This Changes

The issue requests native audio-file handling in the `read` tool so supported audio files are sent to capable models as multimodal attachments instead of raw binary text.

## Best Possible Solution

Add a typed audio attachment path shared by `read` tool results and prompt submission, with provider capability gating, workspace/sandbox egress controls, size/MIME validation, and a clear unsupported-model fallback.

## Reproduction Assessment

Yes, by source inspection: audio extensions cannot enter the image-only read and prompt attachment path, and the embedded runner only submits image attachments. I did not run a live provider session because the missing path is clear in current main.

## Solution Assessment

No, not as a simple extension-list patch on `read`. The maintainable solution needs a first-class audio content contract and provider gating rather than reusing image-specific plumbing directly.
