## Summary

Keep open. Current main still has image-only read and prompt attachment plumbing, while the existing audio path is transcription/media-understanding rather than native read-tool audio attachment parity; the related Google/Gemini PR is narrower and remains open.

## What This Changes

The issue requests first-class audio-file handling in the read tool so audio files are sent as model attachments instead of raw binary text.

## Best Possible Solution

Add a first-class audio attachment contract for read results and prompt submission, wired through model capability checks, MIME/size validation, and workspace/sandbox egress controls.

## Reproduction Assessment

Yes, at source level: current main has image-only read normalization, image-only prompt attachment detection, and no AgentCommandOpts audio attachment option. I did not run a live provider session because the missing contract is visible in source.

## Solution Assessment

No, not as a simple extension-list patch to read. The maintainable solution needs a typed audio attachment path plus provider capability checks, MIME/size validation, and local-file egress policy.
