## Summary

Keep open. Current main still has a text/image-only `read` and prompt-media path; audio exists in transcription/media-understanding surfaces, while related native-audio PRs are open and do not ship `read`-tool audio parity.

## What This Changes

The issue requests native audio-file support in the `read` tool so audio files are attached to multimodal-capable models instead of returned as raw text.

## Best Possible Solution

Define a typed audio prompt-media/tool-result contract with provider capability gating and egress policy, then extend `read` to attach supported audio files through that contract or fail clearly for unsupported models.

## Reproduction Assessment

Yes, by source inspection: the current `read` wrapper, model input filtering, and prompt submission path are text/image-only, so audio cannot enter a native attachment path through `read`. I did not run a live audio session because this read-only review could not create files or run mutating tests.

## Solution Assessment

No, not as a simple automated patch. The maintainable solution needs a first-class audio content/runtime contract before adding audio extensions to the image-specific read hook.
