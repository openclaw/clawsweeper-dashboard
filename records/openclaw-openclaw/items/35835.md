## Summary

Keep open. Current main still has a text/image-only `read` and prompt-media path; audio support is limited to transcription/media-understanding surfaces, and the adjacent native-audio PRs are open rather than shipped read-tool parity.

## What This Changes

Request native audio attachment support in the `read` tool so audio files are sent to audio-capable models instead of returned as raw binary text.

## Best Possible Solution

Define typed audio tool-result and prompt-media support with provider capability/fallback rules, then extend `read` to sniff supported audio files and either attach them or return a clear unsupported/transcription path.

## Reproduction Assessment

Yes, by source inspection. Current main can only turn `read` results and prompt attachments into text/image payloads, so an audio file cannot enter a native audio attachment path through `read`.

## Solution Assessment

No for a simple extension-list patch. The maintainable solution needs a typed audio prompt-media contract, provider gating, fallback behavior, and raw-media egress policy before wiring audio into `read`.
