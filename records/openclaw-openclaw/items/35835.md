## Summary

Keep open: current main still has image-only read/prompt attachment plumbing and transcription-oriented audio handling, while related native-audio work is open or closed as superseded and does not fully implement read-tool audio parity.

## What This Changes

Request that `read()` handle audio files as native multimodal attachments instead of returning binary text.

## Best Possible Solution

Define a first-class typed audio prompt/read attachment contract with MIME sniffing, payload limits, model/provider capability gates, explicit raw-media egress opt-in, docs, regression tests, and real provider proof, coordinated with the native-audio tracker.

## Reproduction Assessment

Yes at source level: current main has image-only read and prompt attachment paths, while audio media handling is transcription-oriented. I did not run a live audio-capable provider flow because the missing contract is visible in source.

## Solution Assessment

No, not as a simple extension-list patch to `read`. The maintainable fix needs typed audio media blocks, provider capability gates, request-size and workspace/sandbox egress limits, config/policy alignment, and docs before raw audio bytes are forwarded.
