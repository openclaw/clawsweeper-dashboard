## Summary

Keep open. Current main still clears trusted-proxy Control UI operator scopes for unpaired browser/device identity, and an existing regression test asserts the scope-empty behavior that matches the reported `missing operator.read` failures despite the trusted-proxy access docs.

## What This Changes

The issue asks for trusted-proxy Control UI WebSocket sessions to receive usable operator scopes when nginx/Authelia forwards trusted identity and `X-OpenClaw-Scopes` headers.

## Best Possible Solution

Preserve trusted-proxy operator scopes for authenticated Control UI WebSocket sessions, preferably by deriving or bounding them from the verified trusted-proxy/header contract, while keeping untrusted and non-Control UI clients unable to self-assign scopes.

## Reproduction Assessment

Yes, source-reproducible on current main: the server path and an existing test both show trusted-proxy Control UI sessions ending with empty scopes, which explains `missing operator.read` for read RPCs. I did not run the full reverse-proxy setup live.

## Solution Assessment

Unclear until a patch is proposed. The narrow maintainable direction is to honor trusted-proxy-scoped Control UI access only after verified proxy auth and origin checks, with tests proving non-proxy clients still cannot self-declare scopes.
