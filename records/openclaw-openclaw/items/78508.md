## Summary

Keep open. Current main still clears trusted-proxy Control UI WebSocket operator scopes for unpaired or unapproved browser device identity, matching the reported `missing operator.read` failures; a previous ClawSweeper review reached the same source-backed conclusion, and the remaining question is the intended auth-scope boundary.

## What This Changes

Restore or clarify trusted-proxy Control UI WebSocket operator scopes so nginx/Authelia sessions can load normal Control UI data instead of failing with `missing operator.read`.

## Best Possible Solution

Define the trusted-proxy Control UI contract explicitly, then either preserve bounded operator scopes for verified proxy-authenticated Control UI WebSocket sessions or document and provide a usable pairing/password fallback for that deployment mode.

## Reproduction Assessment

Yes. The current-main source path and an existing regression test both show trusted-proxy Control UI sessions ending with empty scopes, which explains the reported `missing operator.read`; I did not run a live nginx/Authelia deployment because this review is read-only.

## Solution Assessment

Unclear until maintainers decide the auth boundary. A code fix should only honor Control UI WebSocket scopes after verified trusted-proxy auth and origin checks, with regression coverage proving non-proxy clients still cannot self-assign scopes; documenting a required pairing/password fallback is the safer alternative if that trust boundary is not acceptable.
