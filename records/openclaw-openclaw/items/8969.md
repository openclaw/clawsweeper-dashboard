## Summary

Keep open. Current main covers the first requested quick win with `openclaw skills check`, but the two operator-facing surfaces called out in the issue are still missing: status does not expose manifest-registry diagnostics, and the non-tool skill command path rewrites the prompt without an operator-visible warning. The reporter's local scripts are useful implementation context, and the prior maintainer-context review in the discussion reached the same keep-open conclusion.

## What This Changes

Request to add three developer-experience diagnostics: skill readiness validation, plugin/skill loading diagnostics in status, and a warning when skill slash commands rewrite the prompt.

## Best Possible Solution

Preserve `openclaw skills check`, then add an additive manifest-registry diagnostics surface to `openclaw status`, `openclaw status --json`, and `openclaw status --all`; add a one-line operator warning/log only for non-tool skill command prompt rewrites; cover both with focused tests and update the status/skills docs.

## Reproduction Assessment

Not applicable as a feature request. The current-main verification path is high-confidence source inspection: the skills check command exists, while the status scan/result/rendering paths omit registry diagnostics and the skill rewrite branch mutates the request without logging.

## Solution Assessment

Yes, the requested direction is a maintainable additive fix if scoped to existing owner modules and status payloads. A safer review path would split the two remaining quick wins if maintainers want smaller PRs, but no new core architecture appears necessary.
