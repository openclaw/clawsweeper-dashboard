## What This Changes

The issue requests three developer/operator experience improvements: a skills validation CLI, plugin registry diagnostics in `openclaw status`, and a warning when skill commands rewrite the prompt for model-directed execution.

## Best Possible Solution

Keep the shipped `openclaw skills check` behavior, then either split or implement the two remaining operator surfaces with focused status text/JSON/all coverage, prompt-rewrite branch coverage, and docs updates after the warning UX is chosen.

## Reproduction Assessment

Yes, for the remaining source-level gaps: current main has `skills check`, while status paths do not expose manifest-registry diagnostics and the non-tool skill command branch rewrites the prompt without a warning. This is an enhancement rather than a failing existing contract.

## Solution Assessment

Yes, with a split-design caveat. The maintainable path is to keep the shipped skills command and handle the remaining status and prompt-rewrite visibility requests as focused operator-facing changes after maintainers choose the warning surface.
