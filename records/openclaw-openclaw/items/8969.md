## Summary

Keep open: current main implements the skills validation/debugging surface as `openclaw skills check`, but status still does not expose manifest-registry diagnostics and non-tool skill commands still rewrite the model request without an operator-visible diagnostic; the prior maintainer-context review remains consistent with current source.

## What This Changes

The issue requests three developer-experience improvements: a skills validation command, plugin/skill registry diagnostics in status, and a warning when skill command execution rewrites the prompt.

## Best Possible Solution

Keep the shipped `openclaw skills check` behavior, add sanitized manifest diagnostics through the canonical status text/JSON/all result path, and add a deliberately scoped diagnostic for non-tool skill prompt rewriting, preferably as two focused maintainer-reviewed changes.

## Reproduction Assessment

Yes, at source level: current main exposes no general manifest-registry diagnostics field in status, and the non-tool skill branch rewrites the agent prompt without an operator-visible signal. This is an enhancement gap rather than a failing existing contract.

## Solution Assessment

Partly: extending the canonical status result and payload is the right shape for registry diagnostics, but the prompt-rewrite warning needs a maintainer decision on whether it should be an inline notice, debug log, trace event, or another diagnostic surface.
