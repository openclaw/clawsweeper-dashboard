## Summary

Keep open: current main still blocks the TUI send path before Gateway queue handling, the documented non-steer queue modes require the Gateway to receive those messages, and the linked open PR is the active implementation candidate.

## What This Changes

The issue asks for the TUI to respect the effective Gateway queue mode and forward non-steer mid-run messages to the Gateway.

## Best Possible Solution

Review the linked PR and either land it as the fix or close it with a replacement plan; after a fix merges, close this issue with implementation proof and consolidate the older duplicate report if maintainers want one canonical thread.

## Reproduction Assessment

Yes. Source inspection shows `sendMessage` and `canSubmitTuiChatMessage` reject active Gateway sends before `client.sendChat`, and neither path currently receives a queue-mode value.

## Solution Assessment

Yes for the direction, but no merged implementation exists yet. The maintainable fix should pass the effective Gateway queue mode into TUI state and keep client-side blocking only for steer-mode or in-flight optimistic sends.
