## Summary

Keep open: current main and v2026.6.8 still reject gateway-mode active-run sends before Gateway queue handling, and an open closing PR is the active fix path.

## What This Changes

The issue asks the TUI to respect effective Gateway queue mode so followup, collect, and interrupt messages are forwarded instead of stopped by the busy guard.

## Best Possible Solution

Review, repair, or replace https://github.com/openclaw/openclaw/pull/89078 so the TUI receives effective queue mode and forwards allowed non-steer active-run sends to Gateway queue handling, then close this issue and consolidate https://github.com/openclaw/openclaw/issues/89059.

## Reproduction Assessment

Yes. Source inspection on current main shows both the typed-submit helper and sendMessage reject gateway-mode active-run sends before client.sendChat, and current tests still assert that blocking behavior.

## Solution Assessment

Yes for the requested direction, but no merged fix exists yet. The maintainable solution is to propagate effective Gateway queue mode into TUI session state and block locally only for steer-mode or duplicate in-flight sends.
