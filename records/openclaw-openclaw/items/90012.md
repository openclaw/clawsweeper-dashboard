## Summary

Keep open: current main and the latest release still block gateway-mode TUI active-run sends before Gateway queue handling, and the open linked PR https://github.com/openclaw/openclaw/pull/89078 is the active implementation candidate rather than a reason to close this issue early.

## What This Changes

The issue reports that TUI sends are rejected during active runs even when existing Gateway queue modes should allow followup, collect, or interrupt handling.

## Best Possible Solution

Review, repair, or replace the linked PR https://github.com/openclaw/openclaw/pull/89078 so the TUI receives effective queue mode and forwards allowed non-steer active-run sends to Gateway queue handling, then close this issue and consolidate the older related report.

## Reproduction Assessment

Yes. Source inspection on current main shows both the typed-submit helper and `sendMessage` reject gateway-mode active-run sends before `client.sendChat`, and current tests still assert that blocking behavior.

## Solution Assessment

Yes for the requested direction, but no merged fix exists yet. The maintainable solution is to propagate the effective Gateway queue mode into TUI session state and block locally only for steer-mode or duplicate in-flight sends.
