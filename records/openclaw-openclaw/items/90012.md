## Summary

Keep open: current main and v2026.6.8 still block gateway-mode active-run TUI sends before Gateway queue handling, and https://github.com/openclaw/openclaw/pull/89078 is the open linked fix candidate.

## What This Changes

The issue reports that TUI active-run submit guards reject messages before Gateway queue modes can handle followup, collect, or interrupt.

## Best Possible Solution

Review, repair, or replace https://github.com/openclaw/openclaw/pull/89078 so the TUI receives effective queue mode and forwards allowed non-steer active-run sends to Gateway queue handling.

## Reproduction Assessment

Yes. Source inspection on current main shows both canSubmitTuiChatMessage and sendMessage reject gateway active-run messages before client.sendChat, and current tests assert that behavior.

## Solution Assessment

Yes for the requested direction, but no merged fix exists yet. The maintainable solution is to propagate effective Gateway queue mode into TUI session state and block locally only for steer-mode or duplicate in-flight sends.
