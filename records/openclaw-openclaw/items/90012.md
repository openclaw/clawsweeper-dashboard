## Label Justifications

- `impact:message-loss`: The TUI can suppress followup, collect, or interrupt messages before they reach Gateway queue handling.

## Summary

Keep open: current main and v2026.6.8 still reject gateway-mode TUI sends during active runs before Gateway queue handling, and the linked closing PR is open rather than merged.

## What This Changes

The issue asks for TUI sends to honor non-steer Gateway queue modes instead of rejecting active-run messages with the busy guard.

## Best Possible Solution

Review, repair, or replace https://github.com/openclaw/openclaw/pull/89078 so the TUI receives effective queue mode and forwards allowed non-steer active-run sends to Gateway queue handling.

## Reproduction Assessment

Yes. Source inspection on current main shows both the typed-submit helper and `sendMessage` reject gateway active-run messages before `client.sendChat`, and current tests assert that blocking behavior.

## Solution Assessment

Yes for the requested direction, but no merged fix exists yet. The maintainable solution is to propagate effective Gateway queue mode into TUI session state and locally block only steer-mode or duplicate in-flight sends.
