## Summary

Keep open: current main and the latest release still reject gateway-mode TUI sends during active runs before Gateway queue handling can apply non-steer modes, and the open closing PR is the right implementation path to review before closing this issue.

## What This Changes

The issue reports that TUI chat submission rejects active-run messages with the busy guard even when Gateway queue mode is configured as followup, collect, or interrupt.

## Best Possible Solution

Review, repair, or replace https://github.com/openclaw/openclaw/pull/89078 so the TUI receives effective queue mode and forwards allowed non-steer active-run sends to Gateway queue handling.

## Reproduction Assessment

Yes. Source inspection on current main shows both the typed-submit helper and `sendMessage` reject gateway active-run messages before `client.sendChat`, and current tests assert that blocking behavior.

## Solution Assessment

Yes for the requested direction, but no merged fix exists yet. The maintainable fix is to propagate effective Gateway queue mode into TUI session state and locally block only steer-mode or duplicate in-flight sends.
