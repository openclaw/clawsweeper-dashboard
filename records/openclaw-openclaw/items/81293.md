The PR adds client-side request timeout handling to the Control UI gateway browser client and focused tests for timeout rejection, timer cleanup, and longer request timeout budgets.

## Best Possible Solution

Land the focused Control UI timeout implementation after normal maintainer review and gates, keeping the browser client aligned with the existing core gateway-client timeout behavior.

## Reproduction Assessment

Yes. Current main stores unanswered Control UI RPCs in `pending` without a per-request timer, so an open WebSocket that never sends a response can leave the request unresolved; this review established the path from source rather than running a failing repro.

## Solution Assessment
