The PR adds client-side request timeouts, timeout timing, timer cleanup, and focused tests to the Control UI gateway browser client.

## Best Possible Solution

Land the Control UI timeout fix after preserving known long-running RPC budgets, especially ClawHub skill installation, while keeping the timer cleanup and no-response proof coverage.

## Reproduction Assessment

Yes, from source inspection. Current main lacks a per-request timer in GatewayBrowserClient.requestOnSocket, and the remaining review finding is source-reproducible because installFromClawHub sends no timeoutMs while the new default caps it at 30s.

## Solution Assessment

No, not yet. The per-request timer approach matches core gateway prior art, but the branch should carry explicit longer budgets for existing long-running Control UI requests instead of letting them hit the new default.
