## What This Changes

The PR adds client-side timeout handling to Control UI `GatewayBrowserClient` requests, including timeout errors, timer cleanup, timeout-aware timing, and focused node tests.

## Best Possible Solution

## Reproduction Assessment

Yes. Source inspection shows current main stores unanswered Control UI RPCs in `pending` with no per-request timer, so an open WebSocket that never receives a response can leave the request unresolved; this read-only sweep did not run the repro test.

## Solution Assessment

Yes. Adding per-request timers to `GatewayBrowserClient`, clearing them on response/flush, and honoring longer request-declared timeouts is the narrowest maintainable fix and matches existing core gateway-client behavior.
