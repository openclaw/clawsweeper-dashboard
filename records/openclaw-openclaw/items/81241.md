## Summary

Keep open: the report matches a current-main source path where Telegram direct metadata is built as current-turn context, prepended into `promptForModel`, and then submitted as the session prompt, so the metadata can persist inside the user message body instead of being transcript-only hidden context.

## What This Changes

The issue reports that Telegram inbound messages persist `Conversation info` and `Sender` runtime-context metadata inside the user message text while also expecting that metadata to be delivered out of band.

## Best Possible Solution

Route Telegram current-turn metadata through the established runtime-context/system-context path and persist only the user-authored Telegram body as the user message text.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: Telegram metadata becomes `currentTurnContext`, `buildCurrentTurnPrompt` prepends it to `promptForModel`, and `promptActiveSession(promptForModel)` submits that combined text as the user turn. I did not run a live Telegram bot session in this read-only review.

## Solution Assessment

Yes, the narrowest maintainable solution is to extend the existing runtime-context separation from `11e6928b3edc4be7a78d6397df99c61799b22807` to current-turn channel metadata. The open compact-metadata PR is related but not sufficient because it leaves metadata inline and optional.
