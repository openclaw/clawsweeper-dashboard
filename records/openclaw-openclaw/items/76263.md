## Label Justifications

- `P1`: The report describes a real WhatsApp session-isolation bug that can mix distinct contacts' transcripts and context in a live channel workflow.
- `impact:session-state`: The affected behavior is shared session keys, JSONL transcripts, and model context across distinct WhatsApp contacts.
- `impact:security`: The shared context can expose one customer's conversational context to another customer through model replies.

## Summary

Keep open: current main and the latest release still let WhatsApp direct inbound routing inherit the global `session.dmScope` default, which collapses contacts into `agent:<agentId>:main`; the open account-aware fix PR is not merged and is currently conflicting.

## What This Changes

Request that WhatsApp inbound direct chats use per-account/per-contact session keys instead of sharing the agent main session.

## Best Possible Solution

Land a narrow account-aware WhatsApp direct-session isolation fix, with upgrade behavior and last-route metadata covered, then close this issue after the fix merges.

## Reproduction Assessment

Yes, this is source-reproducible: WhatsApp direct inbound routing passes the direct peer into `resolveAgentRoute`, which defaults `dmScope` to `main`, and the current test contract expects `agent:main:main`. I did not run a live WhatsApp account in this read-only review.

## Solution Assessment

Yes, account-aware per-contact WhatsApp DM isolation is the right fix shape; using a channel-owned override is narrower than changing the global DM default. The remaining decision is upgrade/session compatibility, not whether the bug is still present.
