## Label Justifications

- `P1`: The issue describes a real WhatsApp channel workflow where distinct contacts can share one model context through the default main DM session.
- `impact:session-state`: The affected behavior merges separate WhatsApp contacts into one session key and transcript context.
- `impact:security`: Shared WhatsApp direct-message context can expose one contact's history to another contact in multi-customer deployments.

## Summary

Keep open: current main and v2026.6.8 still let WhatsApp direct messages inherit the global `session.dmScope` default, and the open closing PR that would isolate direct sessions has not merged.

## What This Changes

The issue asks for WhatsApp inbound direct messages to use account/contact-scoped session keys instead of sharing the agent main lane.

## Best Possible Solution

Land or replace the account-aware WhatsApp direct-message isolation fix so inbound DMs use per-account/per-contact session keys, with focused routing tests and an explicit upgrade/session-continuity decision.

## Reproduction Assessment

Yes for source reproduction, though I did not run a live WhatsApp scenario. Current main routes direct WhatsApp messages into `resolveAgentRoute` without a DM-scope override, and the resolver/helper can return the agent main session when `session.dmScope` is unset.

## Solution Assessment

No current-main solution exists yet. The account-aware linked PR is the best visible path, but maintainers still need to resolve the conflicting branch and the upgrade/session-continuity tradeoff before this issue can close.
