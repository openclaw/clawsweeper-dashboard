## Summary

Keep open: this is a plausible Codex app-server bug fix with real behavior proof for a constrained Unix host, but the patch hard-codes the Unix requirements path and misses Codex's Windows system requirements path on a supported local stdio platform.

## What This Changes

This PR changes Codex app-server config, tests, docs, and changelog so implicit local stdio defaults switch from YOLO to guardian when Codex requirements disallow `danger-full-access`.

## Best Possible Solution

Keep the fix in the Codex plugin app-server boundary, but resolve the same platform-specific requirements file path Codex uses and cover Unix plus Windows path selection in focused tests before merge.

## Reproduction Assessment

Yes from source inspection: current main defaults local stdio Codex app-server runtime to `danger-full-access`, while upstream Codex can reject that mode through system requirements. I did not run a failing live Windows host in this read-only review.

## Solution Assessment

No: the direction is narrow and maintainable, but the implementation should use Codex's platform-specific requirements path instead of a Unix-only constant. The safer fix is a small testable path resolver or delegated upstream helper, preserving explicit overrides and websocket behavior.
