## Summary

## What This Changes

The issue asks OpenClaw to release or reconcile a stuck session lane after diagnostic recovery successfully aborts and drains an embedded run while session-level queued inbound work remains.

## Best Possible Solution

Add a focused recovery fix that treats aborted-and-drained active work with nonzero diagnostic session queue depth as needing lane/session reconciliation, while preserving the active-owner and unregistered-active-lane safeguards.

## Reproduction Assessment

Yes: source inspection shows `recoverStuckDiagnosticSession` can return `status=aborted` with `released=0` after `aborted=true` and `drained=true` when command-lane `queuedCount` is zero, despite nonzero diagnostic `queueDepth`. ## Solution Assessment

Yes: the issue's suggested direction is the right layer because recovery already receives both command-lane state and diagnostic session queue depth. The fix should stay in core diagnostic/session recovery rather than adding a channel-specific workaround.
