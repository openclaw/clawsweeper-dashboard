## Summary

Keep open: this PR has the protected `maintainer` label and needs explicit maintainer handling, and review found a blocking gap where the new permissions hardening is not wired through the real gateway request paths. The PR also lacks after-fix real behavior proof from a real setup.

## What This Changes

The PR changes node pairing reconciliation so metadata-only refreshes reuse a pending request while changed caps, commands, or permissions create a replacement request and broadcast superseded requests as resolved.

## Best Possible Solution

Keep the replacement-request behavior, but wire approval-relevant permissions through the real gateway request surfaces, update the matching protocol/docs coverage, and land only after maintainer review plus real behavior proof.

## Reproduction Assessment

Yes from source inspection: current main refreshes any single pending node request because `canRefreshSingle` always returns true. I did not run tests because this review was required to keep the checkout read-only.

## Solution Assessment

No: replacing changed approval surfaces is the right direction, but this patch does not make permission changes reach the node pairing comparison through the actual WebSocket or RPC gateway paths.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Thread permissions through the gateway pairing paths:** `src/infra/node-pairing.ts:176`
  - body: The new comparison treats `permissions` as approval-relevant, but real gateway requests still do not supply that field to `requestNodePairing`: WS connect drops `connectParams.permissions`, and the `node.pair.request` schema rejects a `permissions` property. A node that only changes permissions will not exercise the new replacement logic in the runtime paths, so add permissions to the gateway input/schema and cover that path.
  - confidence: 0.88
