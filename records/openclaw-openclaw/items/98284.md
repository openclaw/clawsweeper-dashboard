## Label Justifications

- `P1`: The linked bug breaks managed-browser login persistence across restarts, blocking real browser-SSO automation workflows.

## Summary

Keep open: this is a focused candidate for a real managed-browser session-state bug, but it is not merge-ready because successful empty cookie snapshots leave stale cookies on disk, after-fix real behavior proof is missing, and the default-on plaintext cookie sidecar still needs maintainer security/product approval.

## What This Changes

The PR adds CDP-based managed Chrome cookie save/restore, a plaintext per-profile cookie sidecar, periodic and stop-time snapshots, focused tests, and CLI browser docs.

## Best Possible Solution

Fix successful empty-cookie snapshots so they clear or replace the sidecar, then require maintainer security/product approval plus redacted real browser and gateway restart proof before merge.

## Reproduction Assessment

Yes for the PR-introduced defect: the source shows a successful empty Storage.getCookies result returns without clearing the existing sidecar, and restoreManagedChromeCookies later reads that stale file. The underlying browser-session bug also has a linked current-main live repro, but I did not rerun Chrome locally in this read-only review.

## Solution Assessment

No. CDP Storage.getCookies and Storage.setCookies are a plausible owner-boundary repair, but this implementation is not the best fix until it distinguishes command failure from an empty successful jar and maintainer-approved sensitive storage is settled.
