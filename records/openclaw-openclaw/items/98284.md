## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Delete the cookie sidecar when resetting profiles:** `extensions/browser/src/browser/cookie-persistence.ts:35`
  - body: `resolveCookieStorePath` puts `cookies.json` beside `user-data`, but `reset-profile` only moves the `userDataDir`. After a user resets the managed profile, the next launch can restore cookies from the untouched sidecar, so logins and sensitive cookies survive a reset. Put the sidecar under the reset path or update reset-profile to delete/move it with coverage.
  - confidence: 0.93
- **[P1] Normalize session cookies before restoring them:** `extensions/browser/src/browser/cookie-persistence.ts:87`
  - body: `Storage.getCookies` returns `Network.Cookie` snapshots, where session cookies use `expires: -1`; `Storage.setCookies` expects `Network.CookieParam`, where session cookies omit `expires`. Passing the raw snapshot through can restore session-only SSO cookies as expired or rejected, and the proof only covers a future-expiring cookie. Normalize to `CookieParam` and add `expires: -1` coverage.
  - confidence: 0.9

- Session-only cookies are not normalized or proven: CDP returns them as `Network.Cookie` objects with `expires: -1`, while restore should send `Network.CookieParam` values with `expires` omitted for session cookies.
- The branch intentionally adds a default plaintext JSON store for live browser login cookies; mode 0600 and docs reduce exposure, but maintainers still need to approve that sensitive-storage model.
