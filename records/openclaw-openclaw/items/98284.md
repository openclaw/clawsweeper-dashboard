## Summary

Keep open: the PR targets a real managed-browser session-loss bug, but it is not merge-ready because a successful empty cookie snapshot preserves and later restores stale live cookies, and the plaintext/default-on cookie sidecar still needs maintainer security/product approval.

## What This Changes

The PR adds managed Chrome cookie save/restore through CDP, periodic and stop-time snapshots, focused tests, and CLI docs for a new cookie sidecar.

## Best Possible Solution

Fix successful empty-cookie snapshots so they clear or delete the sidecar, then require maintainer security/product approval plus redacted real browser restart proof before merge.

## Reproduction Assessment

Yes for the PR-introduced defect: the source shows a successful empty Storage.getCookies result returns without clearing the existing sidecar, and restoreManagedChromeCookies later reads that stale file. The original bug also has a linked current-main live repro, but I did not rerun Chrome locally in this read-only review.

## Solution Assessment

No. CDP Storage.getCookies/Storage.setCookies is a plausible owner-boundary repair, but this implementation is not the best fix until it distinguishes command failure from an empty successful jar and maintainer-approved sensitive storage is settled.
