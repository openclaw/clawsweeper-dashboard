## Summary

Keep this PR open with a blocking correctness finding: the intended deadline-exceeded direction is narrow and well-targeted, but the patch still constructs the provider completion promise before the exceeded-deadline branch can stop it.

## What This Changes

The PR changes image-description timeout handling so exhausted per-call budgets report a deadline-exceeded error instead of scheduling a 1ms timeout, with a regression test and changelog entry.

## Best Possible Solution

Check the exceeded deadline before constructing provider work, or make the timeout wrapper accept a lazy task factory, so exhausted budgets fail clearly without starting a background completion.

## Reproduction Assessment

Yes at source level: current main deterministically coerces negative remaining image-description budget to 1ms. I did not live-reproduce the full multi-hour production retry loop in this read-only review.

## Solution Assessment

No for the current patch as written. The typed-deadline approach is the right narrow direction, but the implementation must stop constructing `complete()` before the exceeded-deadline branch runs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Avoid starting completion after the deadline is exceeded:** `src/media-understanding/image.ts:306-308`
  - body: When `deadline.kind === "exceeded"`, this branch throws before awaiting or otherwise handling `params.task`, but the call site has already evaluated `complete(model, context, ...)`. `pi-ai`'s `complete()` synchronously calls `stream(...)` before returning a promise, so an exhausted budget can still start provider work and leave a later rejection unhandled. Check the deadline before constructing `complete()`, or pass a lazy task callback into the timeout helper.
  - confidence: 0.9
