## Summary

Keep open. Current main still has the reported 1ms exhausted-budget clamp and this PR is a focused fix with supplied proof, but the patch still constructs `complete(...)` before the exceeded-deadline branch runs, which can start provider completion work after the budget is already exhausted.

## What This Changes

The PR changes image-description timeout resolution to return a typed deadline, throw an exceeded-deadline error for exhausted budgets, pass positive remaining budgets to `complete`, add a regression test, and add a changelog entry.

## Best Possible Solution

Handle exhausted image-description deadlines before starting any provider completion work, preserve the positive-budget timeout behavior, and keep focused regression coverage for the negative-budget path.

## Reproduction Assessment

Yes at source level: current main deterministically coerces an exhausted image-description budget to a 1ms timeout. I did not live-reproduce the full multi-hour provider retry loop.

## Solution Assessment

No as currently written. The typed-deadline direction is right, but the exceeded-deadline branch should run before `complete()` is constructed, or the wrapper should accept a lazy task factory.
