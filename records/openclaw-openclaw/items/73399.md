## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Require exact member matching for DM sender names:** `extensions/feishu/src/bot-sender-name.ts:141-143`
  - body: `resolveFeishuDirectMemberName` falls back to `members[0]` whenever the API returns one member but no `member_id` matches the inbound sender. If a p2p/private response is partial, paged differently, or permission-scoped, this puts the wrong human name into agent context and transcripts; only accept an exact member-id match or fall back to contact lookup/live-proven behavior.
  - confidence: 0.93
- **[P1] Bound the direct-member sender-name cache:** `extensions/feishu/src/bot-sender-name.ts:31`
  - body: The new process-wide `directMemberNameCache` is keyed by account, chat, and sender ids from Feishu traffic, but it has no max size and expired entries for one-off keys are never pruned. A busy or hostile long-running gateway can grow this map indefinitely, so mirror the bounded/pruned TTL pattern used by nearby Feishu caches.
  - confidence: 0.9
- **[P2] Validate grant URLs before agent guidance:** `extensions/feishu/src/bot-sender-name.ts:52`
  - body: The PR now treats raw Feishu API response objects as permission errors and propagates DM permission guidance, but the parsed `grantUrl` still accepts any `https://.../app/...` host before agent-facing system text displays it. Host-allowlist Feishu/Lark grant URLs or omit the URL when it is not trusted.
  - confidence: 0.82
