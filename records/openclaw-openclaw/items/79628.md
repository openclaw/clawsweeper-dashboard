## Summary

Keep open: the gateway fix targets a source-backed bug, but this external PR lacks after-fix real behavior proof and includes an unrelated skills/frontmatter change with a concrete config-gating regression risk.

## What This Changes

The PR adds a `server-chat.ts` tool-result hook to broadcast `sessions.changed` after `session_status` changes a model, adds a changelog entry, and also changes shared skill manifest config-requirement parsing.

## Best Possible Solution

Land a focused gateway-only fix that emits `sessions.changed` from the `session_status` changed-model result path with regression coverage and redacted real Control UI or WebSocket proof; split or drop the skills parser change.

## Reproduction Assessment

Yes, from source inspection: `session_status(model=...)` updates the store and marks `changedModel`, while current main has no `sessions.changed` emission on that tool-result path and the UI model display is driven by `sessionsResult`. I did not run a live browser reproduction because this review is read-only.

## Solution Assessment

No, not as submitted. Emitting `sessions.changed` in the gateway tool-result path is the right narrow direction, but this PR should be scoped to that fix and must preserve skill config gating or remove the unrelated skills change.
