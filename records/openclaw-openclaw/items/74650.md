## Summary

Keep open. Current main still starts the ACP bridge as a CLI-mode GatewayClient without suppressing the persisted device identity, so a lower-scoped paired CLI identity can keep hitting the non-silent scope-upgrade pairing path before ACP initialization completes; #74472 is adjacent but only covers the native approval client path.

## What This Changes

The issue asks for ACP runtime and direct ACPX launches to initialize after a single approved ACP scope request instead of looping on scope-upgrade approval after OpenClaw 2026.4.26.

## Best Possible Solution

Add a constrained ACP bridge identity-selection path for trusted loopback token/password launches, with regression coverage that keeps remote and unauthenticated pairing behavior explicit.

## Reproduction Assessment

Yes, source-backed. Seed a lower-scoped paired CLI identity, run openclaw acp or acpx openclaw against a loopback token/password Gateway, and current main sends that persisted CLI identity with default admin scope before ACP initialization can complete.

## Solution Assessment

Yes, the narrow maintainable direction is to mirror the existing local shared-auth identity omission for the ACP bridge only. Broadly changing GatewayClient defaults or silently auto-approving all scope upgrades would be riskier.
