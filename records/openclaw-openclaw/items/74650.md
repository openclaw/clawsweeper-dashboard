## Summary

Keep open. Current main still routes the ACP bridge through a CLI-mode GatewayClient that loads persisted device identity and requests admin scope, while adjacent stale-pairing fixes cover backend callGateway/native-approval clients but not `openclaw acp` or `acpx openclaw`.

## What This Changes

The issue asks for ACP runtime and direct ACPX `openclaw` launches to stop looping on scope-upgrade approval after the 2026.4.26 update and initialize after approval.

## Best Possible Solution

Add a focused ACP bridge gateway-auth fix that uses the existing trusted loopback shared-auth pattern for `openclaw acp` without widening remote, browser, unauthenticated, or device-token-only trust boundaries, and cover it with ACP startup plus gateway stale-pairing regression tests.

## Reproduction Assessment

Unclear. The report has strong symptoms but only one setup step; the source-backed reproduction is to seed a lower-scope paired default device identity, run `openclaw acp` or `acpx openclaw` against a loopback token/password Gateway, and verify it requests a scope upgrade before initialize.

## Solution Assessment

Yes, if the fix stays narrow. Reusing the existing loopback shared-auth backend pattern for the ACP bridge is more maintainable than broad pairing-policy changes here; broader same-device recovery semantics should remain with #74484.
