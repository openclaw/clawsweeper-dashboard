## Summary

Keep open. Current main still constructs the ACP bridge GatewayClient as a CLI client without a device-identity or scope override, so it can reuse a stale persisted CLI pairing and request the default admin scope. The adjacent merged fix only covers backend/native approval clients, not `openclaw acp` or the documented `acpx openclaw` bridge path.

## What This Changes

The issue asks OpenClaw to stop ACP runtime and direct `acpx openclaw` launches from looping on scope-upgrade approval after updating to 2026.4.26.

## Best Possible Solution

Add an ACP bridge-specific loopback shared-auth GatewayClient setup that avoids sending stale persisted CLI device identity for `openclaw acp` and documented `acpx openclaw`, with regression coverage proving ACP initialization no longer creates a pending scope-upgrade request while normal pairing boundaries remain intact.

## Reproduction Assessment

Yes. A high-confidence source-backed reproduction is to seed a read-scoped paired CLI identity, start the ACP bridge against a loopback token/password Gateway, and verify the bridge currently sends that persisted CLI identity with default admin scope before ACP initialize completes.

## Solution Assessment

Yes. Mirroring the existing loopback shared-auth device-identity omission for the ACP bridge is the narrow maintainable fix; changing global Gateway pairing policy or broadly auto-approving scope upgrades would be riskier and belongs with the broader #74484 discussion.
