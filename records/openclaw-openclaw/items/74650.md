## Summary

Keep open. Current main still builds the ACP bridge as a CLI-mode GatewayClient without an ACP-specific device-identity bypass, while the existing local shared-auth bypasses cover backend calls and native approval clients only.

## What This Changes

Fix ACP runtime and direct ACPX launches getting stuck in repeated Gateway scope-approval loops after approving the request.

## Best Possible Solution

Add an ACP-specific identity-omission path for trusted loopback token/password Gateway auth, with tests proving remote and unauthenticated ACP still use normal pairing.

## Reproduction Assessment

Yes. Seed a read-scoped paired CLI identity, run `openclaw acp` or `acpx openclaw` against a loopback token/password Gateway, and current source shows ACP sends the persisted CLI identity with default `operator.admin` before initialize can complete.

## Solution Assessment

Yes. Mirroring the native approval/backend shared-auth pattern at the ACP entry point is the narrow maintainable fix; changing global scope-upgrade approval policy would be broader and riskier.

## Repair Work Prompt

Fix openclaw/openclaw#74650. ACP bridge launches (`openclaw acp` and `acpx openclaw`) can get stuck in repeated Gateway scope-upgrade approval loops when a read-scoped persisted CLI identity exists and the bridge connects to a loopback Gateway with token/password auth. Implement a constrained helper in `src/acp/server.ts` or a small shared helper if that is cleaner, passing `deviceIdentity: null` only for trusted loopback token/password ACP bridge connections. Preserve normal device identity and pairing for remote URLs, no shared auth, explicit device-token/device identity paths, browser/control UI paths, and unrelated CLI calls. Add regression coverage in `src/acp/server.startup.test.ts` and/or `src/gateway/server.silent-scope-upgrade-reconnect.poc.test.ts` proving local ACP shared-auth avoids pending scope upgrades while remote/unauthenticated ACP keeps normal identity behavior. Add one single-line user-facing `CHANGELOG.md` fix entry. Do not broaden silent scope-upgrade auto-approval or change global Gateway pairing policy.
