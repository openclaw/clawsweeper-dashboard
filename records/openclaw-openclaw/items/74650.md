## Summary

Keep open; current main still attaches the persisted CLI device identity to the ACP bridge, so a lower-scoped paired CLI identity can continue producing scope-upgrade approval loops. The adjacent #74472 fix covers native approval clients and the direct-local backend helper, not `openclaw acp`.

## What This Changes

The issue reports that ACP runtime launches and direct `acpx openclaw` fail initialization after 2026.4.26 by repeatedly requesting scope-upgrade approval.

## Best Possible Solution

Add a constrained ACP bridge identity-selection path for trusted loopback token/password launches, with regression coverage that preserves remote, unauthenticated, and explicit device-token pairing behavior.

## Reproduction Assessment

Yes, source-backed. Seed a lower-scoped paired CLI identity, run `openclaw acp` or `acpx openclaw` against a loopback token/password Gateway, and current main will attach that persisted CLI identity while requesting the default admin scope before ACP initialization completes.

## Solution Assessment

Yes. Mirroring the native approval client's local shared-auth identity omission for the ACP bridge is the narrow maintainable direction; changing `GatewayClient` defaults or silently auto-approving scope upgrades would be broader and riskier.
