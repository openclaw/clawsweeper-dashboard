## Summary

Keep open. Current main still routes the ACP bridge through the normal CLI GatewayClient identity path, and the open #77229 fix has not merged yet, so this issue should remain open until that auth-boundary change lands.

## What This Changes

ACP runtime and direct ACPX launches get stuck in repeated gateway scope-approval prompts after an update instead of initializing after one approval.

## Best Possible Solution

Review and land the narrow ACP bridge identity-selection fix, preserving persisted identities for remote or unauthenticated ACP launches and covering all three cases in regression tests.

## Reproduction Assessment

Yes, source-backed: seed a read-scoped paired CLI identity, run openclaw acp or acpx openclaw against a loopback token/password Gateway, and current serveAcpGateway sends the persisted CLI identity with default admin scope. I did not run live tests in this read-only pass.

## Solution Assessment

Yes. Sharing the loopback shared-auth identity-omission rule with the ACP bridge, as #77229 proposes, is narrower than changing GatewayClient defaults or silently auto-approving scope upgrades.
