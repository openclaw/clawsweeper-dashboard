## Summary

Keep open. Current main still routes the standalone ACP bridge through the CLI GatewayClient identity path, so a stale lower-scope paired CLI identity can plausibly keep ACP and `acpx openclaw` in the reported scope-upgrade approval loop even after adjacent backend/native approval paths were fixed.

## What This Changes

The issue reports ACP runtime and direct ACPX `openclaw` launches getting stuck in repeated scope-approval requests after updating to OpenClaw 2026.4.26.

## Best Possible Solution

Add an ACP bridge-specific trusted-loopback shared-auth GatewayClient setup path that avoids stale persisted lower-scope CLI identities for `openclaw acp` and the ACPX `openclaw` target, with regression coverage proving initialization no longer creates a pending scope-upgrade request while remote and explicit-device pairing rules stay unchanged.

## Reproduction Assessment

Yes. A high-confidence synthetic repro is available from source: seed a persisted CLI device approved only for `operator.read`, run the ACP bridge against a loopback token/password Gateway, and verify the current bridge asks for default admin scope through that stale identity before ACP initialize can complete.

## Solution Assessment

Yes. Mirroring the existing loopback shared-auth bypass for this bridge is narrower than changing global Gateway pairing policy or weakening scope-upgrade checks, and it keeps the broader same-device recovery problem with #74484.
