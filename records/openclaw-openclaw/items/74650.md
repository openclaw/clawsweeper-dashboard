## Summary

Keep open. Current main still sends the OpenClaw ACP/acpx bridge through `serveAcpGateway` as a CLI-mode GatewayClient with persisted device identity, so a lower-scoped paired CLI identity can still hit the non-silent scope-upgrade path before ACP initialization; #74472 only covered native approval clients and #75820 is still open without touching the ACP bridge.

## What This Changes

The issue asks for ACP runtime and direct ACPX launches to initialize after one approved scope request instead of looping on repeated scope-upgrade approvals.

## Best Possible Solution

Add a constrained ACP bridge identity helper that omits persisted device identity only for trusted loopback token/password OpenClaw ACP bridge sessions, with regression coverage for ACP, remote, and no-shared-auth cases.

## Reproduction Assessment

Yes, source-backed. Seed a lower-scope paired CLI identity, run `openclaw acp` or `acpx openclaw` against a loopback token/password Gateway, and the ACP bridge sends the persisted CLI identity while requesting default admin scope before initialization completes.

## Solution Assessment

Yes. The narrow maintainable fix is to mirror the native/backend local shared-auth identity omission pattern only for the trusted ACP bridge path; changing GatewayClient defaults or auto-approving scope upgrades globally would be broader and riskier.

## Repair Work Prompt

Fix openclaw/openclaw#74650: OpenClaw ACP/acpx bridge launches can loop on `scope upgrade pending approval` when a persisted CLI device identity has lower scopes. In current main, `serveAcpGateway` creates a CLI-mode `GatewayClient` without suppressing persisted device identity, while acpx maps `openclaw` to `openclaw acp`. Add a narrow ACP bridge helper in `src/acp/server.ts` or a small shared Gateway helper so the bridge passes `deviceIdentity: null` only for loopback token/password shared-auth ACP bridge connections. Do not change `GatewayClient` defaults, broad CLI `callGateway` behavior, remote shared-auth behavior, explicit device-token behavior, or auto-approve scope upgrades globally. Add regression coverage proving local ACP bridge avoids the stale CLI baseline and remote/no-shared-auth paths still require normal identity/pairing. Add a single-line user-facing `CHANGELOG.md` fix entry.
