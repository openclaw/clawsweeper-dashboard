## Summary

Keep open: current main still routes the ACP bridge through a CLI-mode GatewayClient that can attach the persisted CLI device identity and request default admin scope before ACP initialize completes, while the merged native-approval fix does not cover this bridge path.

## What This Changes

The issue reports ACP runtime and direct ACPX/OpenClaw bridge launches getting stuck in repeated Gateway scope-approval requests after updating to OpenClaw 2026.4.26.

## Best Possible Solution

Add a narrowly constrained ACP bridge identity policy that omits persisted device identity only for trusted loopback token/password ACP bridge connections, with regression coverage for remote and no-shared-auth behavior.

## Reproduction Assessment

Yes, source-backed. Seed a persisted CLI device approved only for operator.read, run openclaw acp or acpx openclaw against a loopback token/password Gateway, and the bridge sends that CLI identity with operator.admin before ACP initialize can complete.

## Solution Assessment

Yes. Mirroring the native approval/local backend identity-omission pattern inside the ACP bridge is the narrow maintainable fix; changing GatewayClient defaults or globally auto-approving scope upgrades would be broader and riskier.

## Repair Work Prompt

Fix the ACP bridge scope-approval loop reported at https://github.com/openclaw/openclaw/issues/74650. Current main creates the ACP bridge GatewayClient in src/acp/server.ts as CLI/CLI with no explicit deviceIdentity policy, so loopback token/password ACP bridge launches can attach a stale persisted CLI identity and request operator.admin before initialize completes. Add a constrained helper for serveAcpGateway that passes deviceIdentity: null only when the resolved Gateway URL is loopback and the bridge is using shared token/password auth. Do not change global GatewayClient defaults, do not broaden callGateway identity omission, and preserve normal device identity behavior for remote URLs, no shared auth, explicit device-token paths, browser/node clients, and ordinary CLI commands. Add regression coverage in src/acp/server.startup.test.ts plus any focused Gateway auth tests needed to prove the remote/no-shared-auth negative cases. Add a one-line CHANGELOG.md fix entry if the PR changes user-facing ACP startup behavior.
