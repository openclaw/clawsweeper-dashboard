## Summary

Keep open. Current main still builds the ACP bridge GatewayClient without suppressing persisted CLI device identity, while an open focused PR (#77229) is already tracking the ACP-specific repair path.

## What This Changes

Approving an ACP scope request after updating to 2026.4.26 should let `openclaw acp` and `acpx openclaw` initialize instead of repeatedly requesting scope approval.

## Best Possible Solution

Review and land #77229 or an equivalent narrow ACP bridge change that shares the native approval client's loopback shared-auth identity omission and adds remote/no-shared-auth regression coverage.

## Reproduction Assessment

Yes, source-backed. Seed a read-scoped paired CLI identity, launch `openclaw acp` or `acpx openclaw` against a loopback token/password Gateway, and current main sends that persisted identity with default admin scope before ACP initialization completes.

## Solution Assessment

Yes. Reusing the native approval client's loopback shared-auth identity omission for the ACP bridge is narrower than changing GatewayClient defaults or silently approving scope upgrades.
