## Summary

Keep open. Current main still builds the ACP bridge as a CLI-mode GatewayClient without opting out of persisted device identity for trusted loopback shared-auth, while #74472 and the open #75820 cover adjacent native-approval and CLI fallback paths rather than the ACP bridge.

## What This Changes

The issue reports ACP and direct ACPX launches entering a repeated Gateway scope-approval loop after upgrading to OpenClaw 2026.4.26.

## Best Possible Solution

Add an ACP bridge helper that omits the persisted device identity only for trusted loopback token/password ACP connections, with regression coverage proving remote and no-shared-auth ACP still require normal pairing.

## Reproduction Assessment

Yes. A high-confidence source-backed reproduction is to seed a lower-scope paired CLI identity, run openclaw acp or acpx openclaw against a loopback token/password Gateway, and observe the bridge sending that persisted identity with operator.admin before initialize completes.

## Solution Assessment

Yes. Mirroring the existing backend/native approval pattern for only the ACP bridge is the narrowest maintainable direction; global auto-approval or broad pairing-policy changes would be riskier.
