## Summary

Keep open. Current main still routes the ACP bridge through the normal CLI GatewayClient identity path, so a stale lower-scoped paired CLI identity can still hit the interactive scope-upgrade path before ACP initialization; the adjacent native-approval and backend-RPC fixes do not cover this ACP bridge path.

## What This Changes

The issue reports that ACP runtime launches and direct `acpx openclaw` launches loop on scope-upgrade approval after updating to OpenClaw 2026.4.26.

## Best Possible Solution

Add a constrained ACP bridge identity policy for trusted loopback shared-token/password launches, with regression coverage that preserves remote, browser-origin, device-token, and explicit identity scope-upgrade enforcement.

## Reproduction Assessment

Yes, source-backed. Seed a paired CLI identity with only `operator.read`, launch `openclaw acp` or `acpx openclaw` against a loopback shared-token/password Gateway, and current main will send the persisted CLI identity with an admin-scoped connect before ACP initialization completes.

## Solution Assessment

Unclear until a maintainer confirms the exact auth boundary. The narrow maintainable direction is an ACP-specific trusted-loopback identity policy or backend-style bridge path, not changing GatewayClient defaults or silently approving scope upgrades.
