## Summary

Keep open. Current main still leaves the ACP bridge on the normal CLI GatewayClient identity path, so a stale lower-scoped paired CLI identity can still trigger a scope-upgrade approval loop before ACP initialization completes; #74472 fixed the adjacent native approval-client path only.

## What This Changes

Fix ACP runtime and direct ACPX launches that remain stuck in repeated Gateway scope-upgrade approval after upgrading to OpenClaw 2026.4.26.

## Best Possible Solution

Add a constrained ACP bridge identity-selection path for trusted loopback token/password launches, with regression coverage proving remote, unauthenticated, and explicit-device paths still require normal pairing approval.

## Reproduction Assessment

Yes, source-backed. Seed a lower-scoped paired CLI identity, then run openclaw acp or acpx openclaw against a loopback token/password Gateway; current main sends that persisted CLI identity with admin scope before ACP initialize can complete.

## Solution Assessment

Yes. Mirroring the existing local shared-auth identity omission for the ACP bridge is the narrow maintainable direction; changing GatewayClient defaults or silently auto-approving scope upgrades would be broader and riskier.
