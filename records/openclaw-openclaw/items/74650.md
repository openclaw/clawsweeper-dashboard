## Summary

Keep open. Current main still routes the ACP bridge through a CLI-mode GatewayClient with persisted device identity, while the existing stale-pairing bypass is limited to backend/native approval clients, so the ACP approval loop remains source-backed and unresolved by #74472, #74484, or the open #75820 CLI fallback PR.

## What This Changes

The issue asks for ACP and direct ACPX launches to initialize after one scope approval instead of looping on pending scope-upgrade approvals.

## Best Possible Solution

Keep ACP bridge initialization independent of stale persisted CLI pairing only for trusted local shared-auth connections, with regression coverage and normal remote pairing preserved.

## Reproduction Assessment

Yes. A high-confidence source-backed reproduction is to seed a lower-scope paired CLI identity, run `openclaw acp` or `acpx openclaw` against a loopback token/password Gateway, and observe the ACP bridge sending the persisted device identity with default admin scope before initialize completes.

## Solution Assessment

Yes. Mirroring the existing local shared-auth stale-pairing bypass for the ACP bridge is the narrow maintainable direction; global scope auto-approval or broad pairing-policy changes would be riskier.
