## Summary

Keep open. Current main still has the ACP bridge using the normal CLI GatewayClient identity path, while an open PR specifically addresses this ACP stale-pairing approval loop; this issue should close only after that fix or an equivalent maintainer-approved change lands.

## What This Changes

The issue reports ACP and direct ACPX launches looping on Gateway scope approval after updating to OpenClaw 2026.4.26.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/77229 or an equivalent constrained ACP bridge fix with targeted Gateway/ACP tests, then close this issue as implemented.

## Reproduction Assessment

Yes, source-backed. Seed a lower-scoped paired CLI identity, run openclaw acp or acpx openclaw against a loopback token/password Gateway, and current main can send that persisted identity with an admin-scope request before ACP initialization completes.

## Solution Assessment

Yes. The narrow maintainable direction is to mirror the trusted loopback shared-auth identity omission used by native approvals for ACP, while preserving normal remote and unauthenticated pairing behavior.
