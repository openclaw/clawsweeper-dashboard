## Summary

Keep open. Current main still leaves the ACP bridge on the stale-pairing-prone CLI GatewayClient path, while the adjacent merged auth work only covers backend/native-approval clients, so the reported ACP approval loop remains a valid narrow bug.

## What This Changes

The issue requests that ACP runtime launches and direct `acpx openclaw` initialize after a single ACP scope approval instead of repeatedly creating or reusing scope-upgrade approval requests.

## Best Possible Solution

Add a focused ACP bridge gateway-auth fix that keeps trusted loopback token/password-backed ACP bridge connections from reusing stale lower-scope paired identities, while preserving remote, browser-origin, unauthenticated, device-token-only, Control UI, and general CLI pairing boundaries.

## Reproduction Assessment

Yes, source-backed. Seed a persisted CLI device identity approved only for `operator.read`, then start `openclaw acp` or `acpx openclaw` against a loopback token/password Gateway; current main should request `operator.admin` through that stale identity and hit a scope-upgrade pending approval before ACP initialize.

## Solution Assessment

Yes, the narrow fix belongs at the ACP bridge auth setup or an ACP-specific shared helper, mirroring the existing backend/native-approval pattern. A broad Gateway pairing-policy change would be riskier and should stay with the broader #74484 recovery discussion.
