## Summary

Keep open. Current main still routes the ACP bridge through a CLI-mode GatewayClient with persisted device identity, while the existing stale-pairing bypasses are limited to backend gateway calls and operator approval clients; #74472 and #75820 do not implement the ACP bridge fix.

## What This Changes

The issue requests that approving an ACP scope request once should allow ACP and ACPX OpenClaw clients to initialize after upgrading to 2026.4.26.

## Best Possible Solution

Add a narrow ACP bridge identity-selection path for trusted loopback shared-auth connections, with tests proving remote and no-shared-auth ACP still use normal device identity handling.

## Reproduction Assessment

Yes. Seed a lower-scope persisted CLI identity, run `openclaw acp` or `acpx openclaw` against a loopback shared-auth Gateway, and current main will construct the ACP GatewayClient with the persisted CLI identity before initialize completes.

## Solution Assessment

Yes, the maintainable direction is to mirror the existing local shared-auth stale-pairing bypass for the ACP bridge rather than changing global pairing approval or scope policy.
