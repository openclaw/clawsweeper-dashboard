## Summary

Keep open. Current main still routes the ACP bridge through the persisted CLI Gateway identity path, so the source-backed approval loop remains possible; #77229 is the active targeted fix candidate and should be reviewed instead of closing this issue before merge.

## What This Changes

ACP runtime and direct ACPX launches should initialize after one approved scope request instead of looping on scope-upgrade approval after the 2026.4.26 update.

## Best Possible Solution

Review and land #77229 or an equivalent narrow ACP bridge change that omits persisted device identity only for trusted loopback shared-auth launches, with regression coverage for loopback token/password, remote, and unauthenticated cases.

## Reproduction Assessment

Yes, source-backed. Seed a lower-scoped paired CLI identity, launch ACP against a loopback token/password Gateway, and current main sends that persisted identity with default admin scope before initialization can complete.

## Solution Assessment

Yes. Reusing the native approval client's loopback shared-auth identity omission for the ACP bridge is the narrow maintainable direction; changing GatewayClient defaults or silently approving scope upgrades would be broader and riskier.
