## Summary

Keep open. Current main still builds the ACP bridge as a CLI-mode GatewayClient without suppressing persisted device identity, while GatewayClient loads a persisted identity and defaults to operator.admin scopes; a lower-scoped paired identity can therefore hit the same interactive scope-upgrade loop before ACP initialization. The related #74484 is broader, but this ACP bridge identity-selection path remains a distinct fix boundary.

## What This Changes

The issue reports ACP runtime and direct `acpx openclaw` launches getting stuck in repeated Gateway scope-approval prompts after updating to OpenClaw 2026.4.26.

## Best Possible Solution

Add a constrained ACP bridge identity-selection path for trusted loopback token/password launches, with tests that preserve remote and no-shared-auth pairing behavior.

## Reproduction Assessment

Yes, source-backed. Seed a lower-scoped paired CLI identity, run `openclaw acp` or `acpx openclaw` against a loopback token/password Gateway, and current main will send that persisted identity with default admin scope before ACP initialization completes.

## Solution Assessment

Yes. Mirroring the native approval client’s loopback shared-auth identity omission for the ACP bridge is narrower than changing GatewayClient defaults or silently auto-approving scope upgrades.
