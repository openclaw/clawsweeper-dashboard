## Summary

Keep open. Current main still routes the ACP bridge through the normal CLI GatewayClient identity path, so a lower-scoped paired CLI identity can request the default admin scope before ACP initialization completes; the merged native-approval fix is adjacent but does not cover ACP.

## What This Changes

The issue reports ACP/acpx launches getting stuck in repeated Gateway scope-approval requests after updating to OpenClaw 2026.4.26.

## Best Possible Solution

Add constrained ACP bridge identity selection for trusted loopback shared-auth launches, plus regression coverage for remote and no-shared-auth ACP paths.

## Reproduction Assessment

Yes, source-backed. Seed a paired CLI device with only operator.read, run openclaw acp or acpx openclaw against a loopback token/password Gateway, and current main sends the persisted CLI identity with default operator.admin scope before ACP initialization completes.

## Solution Assessment

Yes. The narrow maintainable direction is to mirror the native approval client's local shared-auth identity omission only for the ACP bridge; changing GatewayClient defaults or silently approving scope upgrades would be broader and riskier.
