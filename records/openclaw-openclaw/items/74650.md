## Summary

Keep open. Current main still routes the ACP bridge through the CLI GatewayClient path with persisted device identity, while the adjacent loopback shared-auth bypass only covers backend/native approval clients, so the reported ACP scope-upgrade loop remains source-backed and not implemented by the merged adjacent fix.

## What This Changes

The issue reports that `sessions_spawn(runtime="acp", agentId="openclaw", mode="run")` and direct `acpx openclaw` fail after 2026.4.26 by repeatedly requesting ACP scope approval that does not stick.

## Best Possible Solution

Add an ACP bridge-specific loopback shared-auth GatewayClient setup that avoids stale persisted lower-scope device identities for `openclaw acp` and `acpx openclaw`, with regression coverage proving ACP initialization no longer creates a pending scope-upgrade request while existing pairing boundaries remain intact.

## Reproduction Assessment

Yes. A high-confidence source-backed reproduction is to seed a lower-scope paired CLI identity, run the ACP bridge against a loopback token/password Gateway, and verify the bridge currently sends that persisted identity with default admin scope before ACP initialize completes.

## Solution Assessment

Yes. Mirroring the existing loopback shared-auth bypass for the ACP bridge is the narrow maintainable fix; changing global Gateway pairing policy or broadly auto-approving scope upgrades would be riskier and should stay with the broader #74484 discussion.
