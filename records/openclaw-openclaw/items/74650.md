## Summary

Keep open. Current main still routes the ACP bridge through a CLI-mode GatewayClient with persisted device identity, while the adjacent stale-pairing bypass only covers backend/native approval clients, so the reported ACP scope-upgrade loop remains source-backed and unresolved by #74472.

## What This Changes

ACP runtime and direct ACPX launches can get stuck in repeated Gateway scope-approval loops after updating to OpenClaw 2026.4.26.

## Best Possible Solution

Add an ACP bridge-specific loopback shared-auth device-identity omission or shared helper, with regression coverage proving `openclaw acp` and `acpx openclaw` initialize without a stale lower-scope CLI identity while normal remote pairing still applies.

## Reproduction Assessment

Yes. A high-confidence source-backed reproduction is to seed a lower-scope paired CLI identity, run the ACP bridge against a loopback token/password Gateway, and verify the bridge currently sends the persisted identity with default admin scope before ACP initialize completes.

## Solution Assessment

Yes. Mirroring the existing backend/native approval shared-auth bypass for the ACP bridge is the narrow maintainable direction; global scope auto-approval or broad pairing policy changes would be riskier.

## Repair Work Prompt

Fix the ACP bridge scope-approval loop reported in https://github.com/openclaw/openclaw/issues/74650. Current main constructs the ACP GatewayClient in `src/acp/server.ts` as CLI mode without `deviceIdentity: null`, so loopback shared-token/password ACP launches can reuse a stale lower-scope CLI identity and request `operator.admin` before initialize. Add a narrow ACP bridge-specific loopback shared-auth omission, preferably by sharing the existing loopback URL/shared-auth predicate used by backend/native approval paths instead of broadening global pairing policy. Do not disable device identity for remote ACP URLs, unauthenticated loopback ACP, normal CLI calls, or generic GatewayClient construction. Add/update focused tests in `src/acp/server.startup.test.ts` and any gateway helper tests needed to prove loopback shared-auth ACP passes `deviceIdentity: null` and remote/unauthenticated ACP keeps pairing behavior. Include a user-facing changelog fix entry. Validate with the targeted ACP/gateway tests and formatter; use Testbox for broader changed gates if promoted by a maintainer.
