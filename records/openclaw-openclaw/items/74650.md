## Summary

Keep open. Current main still routes the ACP bridge through a CLI-mode GatewayClient without suppressing persisted device identity, so the reported ACP/direct ACPX approval loop remains source-reproducible; the merged #74472 fix is adjacent but limited to native approval clients.

## What This Changes

The issue reports ACP runtime and direct `acpx openclaw` launches getting stuck in a repeated Gateway scope-upgrade approval loop after updating to OpenClaw 2026.4.26.

## Best Possible Solution

Add a constrained ACP bridge identity-selection path that omits stale persisted device identity only for trusted loopback token/password launches, with regression coverage for remote and no-shared-auth cases.

## Reproduction Assessment

Yes, source-backed. Seed a persisted CLI identity with only `operator.read`, launch `openclaw acp` or `acpx openclaw` against a loopback token/password Gateway, and the ACP bridge will send that persisted CLI identity with default admin scope before initialize completes.

## Solution Assessment

Yes. The narrow maintainable fix is to mirror the native approvals client's local shared-auth identity omission for the ACP bridge; changing GatewayClient defaults or auto-approving scope upgrades would be broader and riskier.
