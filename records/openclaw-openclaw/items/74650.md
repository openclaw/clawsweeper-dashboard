## Summary

Keep this open. Current main has adjacent stale-pairing fixes for backend RPC and native approval clients, but the ACP bridge still constructs its GatewayClient as a CLI client without a device-identity opt-out, so the reported ACP/ACPX scope-upgrade loop remains plausible and untested.

## What This Changes

The issue reports ACP runtime and direct ACPX OpenClaw launches looping on gateway scope-upgrade approval after updating to OpenClaw 2026.4.26.

## Best Possible Solution

Add a narrow ACP bridge gateway-auth fix that mirrors the existing trusted loopback shared-auth pattern for `openclaw acp`, with regression coverage proving it avoids stale paired CLI scope baselines without weakening remote or device-token pairing behavior.

## Reproduction Assessment

Unclear. The report lacks logs and full setup details, but current main has a concrete source-backed path: seed a lower-scope paired default device identity, start `openclaw acp` or `acpx openclaw` against a loopback token/password gateway, and the bridge can request `operator.admin` as that stale CLI identity before ACP initialize completes.

## Solution Assessment

No patch is proposed here. The current main fix is a good adjacent pattern but not sufficient for this issue because it covers backend/native approval clients and leaves the ACP bridge GatewayClient on the default CLI device-identity path.
