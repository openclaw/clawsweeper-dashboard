## Summary

Keep open. Current main still routes `openclaw acp` through a CLI-mode GatewayClient that loads the persisted device identity and defaults to admin scope, while the existing stale-pairing bypasses only cover direct-local backend calls and native approval clients.

## What This Changes

The issue requests that ACP launches stop looping on scope approval after a local Gateway approval, for both `sessions_spawn(runtime="acp")` and direct `acpx openclaw` paths.

## Best Possible Solution

Add a narrowly scoped ACP bridge path for trusted loopback shared-token/password connections that avoids stale persisted CLI pairing while preserving required ACP Gateway scopes and keeping normal remote/device approval behavior intact.

## Reproduction Assessment

Yes. A high-confidence source-backed reproduction is to pair the persisted CLI identity with only `operator.read`, then launch `openclaw acp` or `acpx openclaw` against a local token/password Gateway and observe the bridge requesting admin scope with that stale device identity before ACP initialize completes.

## Solution Assessment

Yes, with a constraint. The maintainable fix is an ACP-specific local shared-auth seam; a broad pairing-policy bypass or plain device-identity omission would be riskier and may not preserve ACP method scopes.

## Repair Work Prompt

Fix the ACP approval loop reported in this issue. Current main creates the ACP bridge GatewayClient as CLI mode without a device-identity override, so a stale lower-scope persisted CLI pairing can block `openclaw acp` and `acpx openclaw` with repeated `scope-upgrade` approvals. Implement a narrow ACP bridge path for trusted loopback shared-token/password auth that does not depend on the stale CLI pairing baseline and still preserves the Gateway scopes needed by ACP requests. Do not add global scope auto-approval, do not weaken remote Gateway pairing, do not change browser/node pairing semantics, and do not treat unauthenticated ACP as trusted. Likely files are `src/acp/server.ts`, ACP startup tests, and possibly the Gateway ws-connection policy/helper tests if a generic seam is needed. Add a CHANGELOG fix entry because this is user-facing.
