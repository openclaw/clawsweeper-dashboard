## Summary

Keep open. Current main still creates the ACP bridge GatewayClient on the CLI identity path without an ACP-specific device-identity bypass, while the existing stale-pairing bypasses only cover backend/native approval clients.

## What This Changes

Fix ACP runtime and `acpx openclaw` launches getting stuck in repeated scope-upgrade approval after updating to 2026.4.26.

## Best Possible Solution

Add a narrow ACP bridge loopback shared-auth device-identity omission or shared helper, while preserving normal remote pairing and device-token behavior.

## Reproduction Assessment

Yes. Seed a persisted CLI device identity approved only for `operator.read`, start `openclaw acp` or `acpx openclaw` against loopback token/password auth, and current main will send that identity with default admin scope before ACP initialization completes.

## Solution Assessment

Yes. Mirroring the existing backend/native approval shared-auth bypass for the ACP bridge is the narrow maintainable fix; global auto-approval or wider pairing policy changes would be riskier.
