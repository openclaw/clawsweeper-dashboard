## Summary

Keep open. Current main still leaves the ACP bridge on the persisted CLI device-identity path, so the adjacent #74472 native-approval fix does not cover this ACP/direct ACPX approval loop; #74484 is related but broader and does not replace the ACP-specific repair boundary.

## What This Changes

The issue asks for ACP and direct `acpx openclaw` launches to initialize after one approved Gateway scope request instead of repeatedly requesting scope upgrades.

## Best Possible Solution

Add a constrained ACP bridge identity-selection path for trusted loopback token/password launches, with regression coverage that preserves remote and no-shared-auth pairing behavior.

## Reproduction Assessment

Yes, source-backed. Seed a lower-scoped paired CLI identity, launch `openclaw acp` or `acpx openclaw` against a loopback token/password Gateway, and current main will send the persisted CLI identity with default admin scope before ACP initialization completes.

## Solution Assessment

Yes. The narrow maintainable direction is to mirror the native approval client's local shared-auth identity omission for the ACP bridge, while keeping GatewayClient defaults and scope-upgrade approval policy intact.
