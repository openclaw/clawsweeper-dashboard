## Summary

Keep open. Current main still routes the ACP bridge through a CLI-mode GatewayClient without suppressing the persisted device identity, while GatewayClient loads that identity by default and requests admin scope when no scopes are supplied; the adjacent native-approval fix does not cover ACP.

## What This Changes

The issue requests fixing ACP/acpx Gateway launches that repeatedly require scope approval instead of completing initialization after approval.

## Best Possible Solution

Add a constrained ACP bridge identity-selection path for trusted loopback shared-auth launches, with regression coverage for stale paired CLI identity, remote ACP, and no-shared-auth behavior.

## Reproduction Assessment

Yes, source-backed. Seed a paired CLI identity with only operator.read, then start openclaw acp or acpx openclaw against a loopback token/password Gateway; current main sends that persisted CLI identity with an admin scope request before ACP initialization completes.

## Solution Assessment

Yes. The narrow maintainable fix is ACP-specific identity suppression for trusted loopback shared-auth, mirroring the native approvals and backend callGateway patterns without changing GatewayClient defaults or silently auto-approving scope upgrades.
