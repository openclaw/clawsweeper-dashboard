## Summary

Keep open. Current main still constructs the standalone ACP bridge as a CLI-mode GatewayClient without the loopback shared-auth device-identity bypass that was added for adjacent backend/native approval paths, so the reported stale lower-scope pairing loop remains plausible and source-backed.

## What This Changes

The issue reports that `openclaw acp` and `acpx openclaw` on OpenClaw 2026.4.26 repeatedly hit `scope upgrade pending approval` even after approving the ACP scope request.

## Best Possible Solution

Add an ACP bridge-specific loopback shared-auth gateway-client setup path that avoids stale persisted lower-scope device identities for `openclaw acp`/`acpx openclaw`, with focused regression coverage proving ACP initialization no longer creates a scope-upgrade pending request while existing pairing security boundaries remain unchanged.

The reporter's one-step reproduction is incomplete, but the code path and adjacent tests make the focused repro clear.

## Solution Assessment

Yes. Mirroring the existing loopback shared-auth bypass pattern for the ACP bridge is narrower and safer than changing global Gateway pairing policy; the broader same-device recovery behavior should stay with #74484.

## Repair Work Prompt

Fix the ACP bridge scope-approval loop reported in https://github.com/openclaw/openclaw/issues/74650. Current main creates the standalone ACP bridge GatewayClient in `src/acp/server.ts` as CLI mode without overriding device identity, so trusted loopback token/password ACP bridge connections can reuse a stale lower-scope paired CLI identity and get stuck on `scope upgrade pending approval`. Reuse or mirror the existing pattern from `src/gateway/operator-approvals-client.ts` / `src/gateway/call.ts`: omit device identity only for trusted loopback Gateway URLs when resolved shared auth has a token or password. Keep remote URLs, no-auth/device-token flows, browser clients, general CLI pairing, and broad Gateway pairing policy unchanged. Add focused regression coverage proving `openclaw acp`/`acpx openclaw` startup no longer creates a scope-upgrade pending request when a lower-scope paired CLI identity exists, and proving remote or unauthenticated ACP bridge clients still keep the normal device-identity behavior. Include a user-facing `CHANGELOG.md` fix entry credited to @vharris817; reference #74484 only as broader context, not as the fix target.
