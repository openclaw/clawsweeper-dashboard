## Summary

Keep open. Current main still routes the ACP bridge through the normal CLI GatewayClient identity path, so a lower-scoped persisted CLI device can keep triggering a non-silent admin scope-upgrade request before ACP initialization completes.

## What This Changes

Report asks for ACP runtime and direct ACPX launches to stop looping on Gateway scope approval after updating to OpenClaw 2026.4.26.

## Best Possible Solution

Add a constrained ACP bridge identity-selection path for trusted loopback token/password launches, with regression coverage that preserves remote and no-shared-auth pairing behavior.

## Reproduction Assessment

Yes, source-backed. Seed a lower-scoped paired CLI identity, run openclaw acp or acpx openclaw against a loopback token/password Gateway, and current main sends that persisted CLI identity with the default admin scope before ACP initialization can complete.

## Solution Assessment

Yes. Mirroring the native approval client's local shared-auth identity omission for the ACP bridge is the narrow maintainable direction; changing GatewayClient defaults or auto-approving scope upgrades would be broader and riskier.
