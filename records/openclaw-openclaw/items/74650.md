## Summary

Keep open. Current main still builds the ACP bridge as a CLI-mode GatewayClient that keeps the persisted device identity and default admin scope, while the existing stale-pairing bypasses cover backend RPCs and native approval clients, not ACP startup.

## What This Changes

The issue reports ACP and direct ACPX launches looping on Gateway scope-upgrade approval after updating to OpenClaw 2026.4.26 instead of initializing after approval.

## Best Possible Solution

Add an ACP-specific helper that omits persisted device identity only for trusted loopback token/password ACP bridge connections, with tests preserving normal remote and no-shared-auth pairing behavior.

## Reproduction Assessment

Yes. A high-confidence source-backed reproduction is to seed a lower-scope paired CLI identity, launch openclaw acp or acpx openclaw against a loopback token/password Gateway, and observe the ACP bridge sending the persisted CLI identity with default operator.admin scope before initialization completes.

## Solution Assessment

Yes. Mirroring the native approval/local backend identity-omission pattern for the ACP bridge is the narrow maintainable direction; global silent scope-upgrade approval would be broader and riskier.
