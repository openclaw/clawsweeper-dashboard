## Summary

Keep open. Current main still routes the reported `sessions_spawn(runtime="acp", agentId="openclaw")` and direct `acpx openclaw` surfaces through the standalone `openclaw acp` bridge, and that bridge still constructs a CLI-mode `GatewayClient` without the loopback shared-auth device-identity bypass already added for adjacent backend/native approval paths. The stale lower-scope approval loop remains source-backed and narrow enough for a focused repair.

## What This Changes

Fix ACP runtime and direct ACPX/OpenClaw bridge launches so one approved ACP scope request persists and ACP initialization can complete after updating to 2026.4.26.

## Best Possible Solution

The ACP bridge should use the same trusted local loopback shared-auth treatment as the backend/native approval clients, scoped specifically to local token/password bridge connections while preserving remote ACP pairing and global scope-upgrade protections.

## Reproduction Assessment

Yes. Seed a persisted CLI device identity with only `operator.read`, start a token/password loopback Gateway, and launch the OpenClaw ACP bridge; current main should still attach that identity and request the default admin scope before initialize.

## Solution Assessment

Yes. A bridge-specific local shared-auth device-identity bypass or shared helper is the narrow maintainable fix; changing global Gateway scope-upgrade policy would be broader and risk weakening pairing boundaries.
