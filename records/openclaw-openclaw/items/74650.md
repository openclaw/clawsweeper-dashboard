## Summary

Keep open. Current main still leaves the ACP bridge on the persisted CLI device-identity path, while the adjacent stale-pairing bypass covers backend/native approval clients only, so the reported ACP scope-approval loop remains source-backed and unresolved.

## What This Changes

The issue requests that ACP runtime launches and direct `acpx openclaw` stop getting stuck in repeated scope-upgrade approval loops after updating to OpenClaw 2026.4.26.

## Best Possible Solution

Add an ACP bridge-specific or shared loopback shared-auth identity-omission helper, with regression coverage proving `openclaw acp` and `acpx openclaw` initialize despite a stale lower-scope CLI identity while normal remote pairing still applies.

## Reproduction Assessment

Yes. Seed a paired CLI identity with only `operator.read`, start the ACP bridge against a loopback token/password Gateway, and current main will construct the ACP GatewayClient without `deviceIdentity: null` while requesting default admin scope before ACP initialize completes.

## Solution Assessment

Yes. Mirroring the existing local shared-auth stale-identity bypass for the ACP bridge is the narrow maintainable direction; global scope auto-approval or broad pairing-policy changes would be riskier.
