## Summary

Keep open. Current main still routes the ACP bridge through a CLI-mode GatewayClient with persisted device identity and default admin scope, while the shipped #74472 fix only covers native approval clients and the open #75820 does not touch the ACP bridge path.

## What This Changes

The issue reports ACP runtime and direct ACPX launches getting stuck in repeated Gateway scope-upgrade approval after updating to 2026.4.26.

## Best Possible Solution

Add a constrained ACP bridge helper that omits persisted device identity only for trusted loopback token/password ACP connections, with regression coverage preserving remote and no-shared-auth behavior.

## Reproduction Assessment

Yes, source-backed. Seed a lower-scope paired CLI identity, run `openclaw acp` or `acpx openclaw` against a loopback token/password Gateway, and the ACP bridge sends the persisted CLI identity with default admin scope before initialization can complete.

## Solution Assessment

Yes. Mirroring the native approval/local backend identity-omission pattern for the ACP bridge is the narrow maintainable fix; global auto-approval or GatewayClient default changes would be broader and riskier.
