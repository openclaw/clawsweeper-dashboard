## Summary

Keep open. Current main still routes the ACP bridge through a CLI-mode GatewayClient without suppressing persisted device identity, so a lower-scoped paired CLI identity can keep requesting an admin scope upgrade before ACP initialization completes; the merged #74472 fix is adjacent but limited to native approval clients.

## What This Changes

The issue asks OpenClaw to stop ACP runtime and direct ACPX launches from looping on repeated Gateway scope-approval requests after an update.

## Best Possible Solution

Add a constrained ACP bridge identity-selection path for trusted loopback token/password launches, with regression coverage that preserves remote and no-shared-auth pairing behavior.

## Reproduction Assessment

Yes, source-backed. Seed a lower-scoped paired CLI identity, run openclaw acp or acpx openclaw against a loopback token/password Gateway, and current main sends that persisted CLI identity with default admin scope before ACP initialization can complete.

## Solution Assessment

Yes. Mirroring the native approval client's local shared-auth identity omission for the ACP bridge is the narrow maintainable direction; changing GatewayClient defaults or silently auto-approving scope upgrades would be broader and riskier.
