## Summary

Keep open. Current main still builds the ACP bridge as a CLI-mode GatewayClient with no ACP-specific device identity handling, so a lower-scoped persisted CLI identity can continue asking for the default admin scope before ACP initialization completes. The merged native-approval fix is adjacent but does not cover this ACP bridge path.

## What This Changes

Approving one ACP scope request should let `openclaw acp` and direct `acpx openclaw` initialize instead of looping on scope-upgrade approval after 2026.4.26.

## Best Possible Solution

Add a constrained ACP bridge connection path for trusted loopback token/password launches that avoids the stale lower-scoped CLI pairing baseline while preserving remote, browser-origin, and unauthenticated scope-upgrade protections.

## Reproduction Assessment

Yes, source-backed. Seed a lower-scoped paired CLI identity, run `openclaw acp` or `acpx openclaw` against a loopback token/password Gateway, and current main will send that persisted CLI identity with a default admin scope before ACP initialization can complete.

## Solution Assessment

Yes, the narrow maintainable direction is ACP-bridge-specific identity and scope handling for trusted local shared-auth launches. Changing GatewayClient defaults or silently approving scope upgrades would be broader and riskier.
