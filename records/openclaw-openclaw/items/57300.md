## Summary

Keep this PR open, but do not merge it as-is. Current main still has the linked `agent --deliver` problem, while this PR's low-level bypass would also catch non-CLI delivery workflows that intentionally use an agent turn before sending a user-facing reply.

## What This Changes

The PR adds an early `--deliver` path in `agentCommandInternal` that sends `opts.message` directly when no `internalEvents` are present, updates CLI help text, and adds/updates tests for direct delivery and gateway routing.

## Best Possible Solution

Keep the linked bug and this implementation path alive, but make direct delivery an explicit CLI notification mode separate from the generic `deliver` flag used by gateway/internal workflows. The final patch should preserve model synthesis for exec approval followups, internal-event announcements, ACP turns, and other non-CLI agent delivery paths.

## Reproduction Assessment

Yes. The linked issue gives a concrete CLI reproduction for the current bug, and current main source confirms `--deliver` still runs an ACP or embedded agent turn before delivery rather than sending the original message directly.

## Solution Assessment

No. The PR solves the direct CLI notification case, but the low-level guard is still too broad because `deliver: true` is also used by internal workflows that need the model to turn instruction/context prompts into safe user-facing replies.
