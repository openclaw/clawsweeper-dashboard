## Summary

Keep open. The linked user problem is real enough to keep reviewing, but this patch is not safe because it repurposes the shared `deliver` flag and would send exec-approval followup prompts directly to external channels instead of letting the agent synthesize a user-facing reply.

## What This Changes

This PR adds a direct `--deliver` bypass in `agentCommandInternal`, updates CLI help text, and adds CLI tests for direct delivery and gateway routing.

## Best Possible Solution

Introduce an explicit direct-delivery contract for the CLI notification case while keeping generic `deliver` as agent-reply delivery for followups, subagent announcements, ACP, and other internal workflows; update protocol schema, CLI/docs, tests, and changelog together.

## Reproduction Assessment

Yes. Source inspection is enough: current main still runs an agent turn before `--deliver` delivery, and the PR regression follows from exec approval followups calling gateway `agent` with `deliver: true`, no `internalEvents`, and instruction prompt text.

## Solution Assessment

No. Direct delivery is a useful direction, but the current patch hangs it on the shared `deliver` flag; the safer approach is an explicit direct-delivery mode limited to callers that truly want raw message delivery.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9
