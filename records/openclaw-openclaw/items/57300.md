## Summary

Keep open. Current main still does not implement direct delivery for the reported CLI notification path, but this PR is not merge-ready because the new broad `deliver` bypass would also intercept non-CLI delivery workflows that intentionally need an agent turn, especially exec approval followups.

## What This Changes

The PR changes `agentCommandInternal` to deliver `opts.message` directly when `deliver` is true and no `internalEvents` are present, updates CLI help text, and adds agent/gateway tests for direct delivery and gateway routing.

## Best Possible Solution

Revise the implementation so direct delivery is an explicit CLI notification mode, preserved through the gateway when needed, while internal-event announcements and exec approval followups stay on the model-synthesis path. Update the CLI docs and add regression coverage for direct CLI delivery, gateway routing, internal-event announcements, and exec approval followups.

## Reproduction Assessment

Yes. Code inspection gives a high-confidence path: current main runs `runAgentAttempt` before delivery for `--deliver`, and the PR regression can be reproduced with the existing exec approval followup path that sends `deliver: true` with no `internalEvents`.

## Solution Assessment

No. The PR fixes the direct CLI notification symptom too broadly; the safer solution is an explicit direct-delivery contract or flag that only the CLI notification path sets, leaving other `deliver: true` workflows on the agent turn.

## Review Findings
