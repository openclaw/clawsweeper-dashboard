## Summary

Keep open: the feature and proof are useful, but PR head still blocks the documented Codex ChatGPT Responses path before the Codex native-search wrapper can apply request-scoped options, and the stricter unsupported-backend 400 behavior needs maintainer acceptance.

## What This Changes

This PR adds `web_search_options` support to the OpenAI-compatible chat-completions Gateway, routes request-scoped native web-search options through agent/provider wrappers, maps them into OpenAI/Codex native search payloads, normalizes incompatibility errors, and updates docs/tests.

## Best Possible Solution

Keep the request-scoped forwarding design, route Codex-eligible models to the Codex native-search wrapper before any direct OpenAI rejection, and make the unsupported-backend 400 contract an explicit maintainer-approved API behavior.

## Reproduction Assessment

Yes. Source inspection of PR head shows that an `openai-chatgpt-responses` model with `nativeWebSearch` enters the outer direct OpenAI wrapper, fails `shouldEnableOpenAINativeWebSearch`, and throws before the Codex wrapper can apply the options.

## Solution Assessment

No. The owner-boundary shape is mostly right, but the current wrapper order is not the best solution until Codex-capable models can fall through to the Codex native-search wrapper and maintainers choose the unsupported-backend compatibility contract.
