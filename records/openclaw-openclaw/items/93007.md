## Summary

Keep open: current main does not implement `web_search_options`, and this PR is useful but not merge-ready because the current head blocks the claimed Codex ChatGPT Responses path before the Codex native-search wrapper can handle the request-scoped options.

## What This Changes

The PR adds `web_search_options` parsing to the OpenAI-compatible chat-completions endpoint, routes request-scoped native search options through agent extra params, maps them into OpenAI/Codex native `web_search` payloads, normalizes errors, and documents/tests the behavior.

## Best Possible Solution

Repair the OpenAI/Codex wrapper composition, keep the request-scoped forwarding shape, and merge only after maintainers accept or revise the stricter unsupported-backend 400 contract.

## Reproduction Assessment

Yes, source-reproducible for the PR defect: PR head composes the direct OpenAI native-search wrapper outside the Codex wrapper, and the new rejection branch throws for `nativeWebSearch` when `model.api` is `openai-chatgpt-responses`. I did not run a live repro because this review is read-only.

## Solution Assessment

No: the owner-boundary shape is mostly right, but the current wrapper composition is not the best solution because it blocks Codex models before the Codex native-search wrapper can handle request-scoped options. Fix the fallthrough/order, then decide the unsupported-backend compatibility contract.
