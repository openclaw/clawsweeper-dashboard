Keep open. Current main still has a source-visible pre-run provider auth cooldown gate before the later plugin-owned harness auth bypass, and the linked fix PR is still open, so this issue should not close until that PR lands or is closed.

## What This Changes

The issue asks OpenClaw to stop stale provider auth profile or auth-state data from blocking external CLI plugin harnesses that own their own credentials.

## Best Possible Solution

Land a focused fix that makes the outer model-fallback auth cooldown gate respect selected plugin-owned transport for explicit and auto-selected non-Codex harnesses while preserving PI fallback and Codex bootstrap behavior.

## Reproduction Assessment

Yes. Source inspection shows runWithModelFallback checks provider auth profile cooldowns before the embedded plugin harness path can apply its plugin-owned transport auth bypass.

## Solution Assessment

Yes. The narrow maintainable solution is to align the outer cooldown precheck with the existing plugin-owned transport contract; new harness auth metadata can be a later API discussion if maintainers want explicit per-harness policy.
