## Summary

Keep open. Current main still has a source-visible model-fallback auth cooldown gate that can skip or suspend an external CLI/plugin harness before the later plugin-owned transport path bypasses OpenClaw auth bootstrap.

## What This Changes

The issue asks OpenClaw to let external CLI/plugin harnesses that own their own auth start even when stale OpenClaw auth profiles or auth-state entries exist for the same provider.

## Best Possible Solution

Make the model-fallback auth cooldown precheck honor selected non-PI plugin harness transport ownership so stale provider auth state cannot block external CLI harness startup, while preserving PI provider-auth gating and Codex bootstrap behavior.

## Reproduction Assessment

Yes from source inspection, but not from a live CLI run. With stored provider profiles all in cooldown or disabled, runWithModelFallback can skip or suspend the candidate before a plugin-owned external harness reaches its empty-auth-store path.

## Solution Assessment

Yes. The narrow maintainable fix is to align the fallback precheck with the existing plugin-owned transport rule instead of adding a new public metadata surface first.
