## Summary

Keep open. Current main still has a source-visible pre-run model-fallback auth cooldown gate that can skip or suspend an external CLI/plugin harness before the later plugin-owned transport path gets to bypass OpenClaw auth bootstrap.

## What This Changes

The issue asks OpenClaw to stop blocking external CLI plugin harness startup on stale OpenClaw auth profile or auth-state data when the harness owns authentication itself.

## Best Possible Solution

Make the model-fallback auth cooldown precheck honor selected non-PI plugin harness transport ownership so stale provider auth state cannot prevent external CLI harness startup, while preserving Codex bootstrap behavior and PI provider-auth gating.

## Reproduction Assessment

Yes from source inspection: `runWithModelFallback` checks provider auth profiles for cooldown before the embedded plugin harness run, while the later runner path would skip generic OpenClaw auth for non-PI plugin harnesses. I did not run a live claude-tmux scenario.

## Solution Assessment

Yes: the narrow maintainable fix is to align the fallback precheck with the existing plugin-owned transport rule, rather than forcing external CLI harnesses through stale OpenClaw token state. Explicit auth metadata can be considered later if maintainers need per-harness overrides.
