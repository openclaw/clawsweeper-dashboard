## Summary

Keep open: the PR addresses a real ACP image-routing gap, but the current patch opens the ACP attachment path without enforcing the existing `sessions_spawn` attachment policy, leaves public docs contradictory, and still lacks real after-fix Codex ACP image proof.

## What This Changes

The PR removes the ACP `sessions_spawn` attachment rejection, forwards attachments through the Gateway `agent` call into ACP `runTurn`, adds focused tests, adjusts two OpenAI test casts, and adds a changelog entry.

## Best Possible Solution

Apply the existing sessions_spawn attachment opt-in and limits before ACP dispatch, route validated image attachments through the ACP turn attachment contract, update public docs, and require redacted live Codex/acpx image proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence current-main path: `sessions_spawn` with `runtime: "acp"` and attachments returns the unsupported-attachments error before `spawnAcpDirect`, and the ACP dispatch call currently sends no attachments.

## Solution Assessment

No, not yet. Reusing Gateway and ACP attachment plumbing is the right direction, but this patch needs to preserve the attachment policy, align docs, cover the image conversion boundary, and provide live after-fix proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Honor the sessions_spawn attachment policy:** `src/agents/tools/sessions-spawn-tool.ts:355`
  - body: Forwarding `attachments` directly into `spawnAcpDirect` bypasses `tools.sessions_spawn.attachments.enabled` plus the configured max file/count/byte checks.   - confidence: 0.9

  - confidence: 0.78
- **[P3] Update the ACP attachment docs:** `CHANGELOG.md:595`
  - body: The changelog announces ACP image support, but `docs/gateway/config-tools.md` still says ACP runtime rejects sessions_spawn attachments. Update the public docs to describe the image-only ACP boundary and the validation limits that apply.
  - confidence: 0.84
