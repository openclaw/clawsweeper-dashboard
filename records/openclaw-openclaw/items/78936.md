## Summary

Keep open: the PR is still not merge-ready because ACP spawn images are forwarded as `{ mediaType, data }`, while the gateway agent normalizer only accepts `{ mimeType, content }` or the existing `source` shape, so the new path drops the images before Codex can see them; the supplied proof is still unit/typecheck output rather than a live ACP/Codex image run.

## What This Changes

The branch adds opt-in ACP image attachment handling for `sessions_spawn`, forwards those attachments through `spawnAcpDirect`, updates direct ACP agent image passing, docs, changelog, and focused tests.

## Best Possible Solution

Route validated opt-in image attachments through the existing gateway attachment contract into ACP `runTurn`, keep docs/tests aligned with that contract, and merge only with redacted live ACP/Codex image proof.

## Reproduction Assessment

Yes by source inspection: current main rejects ACP `sessions_spawn` attachments, and the PR head still has a high-confidence failure path because gateway normalization drops the new `{ mediaType, data }` payload. I did not run live ACP/Codex in this read-only review.

## Solution Assessment

No: the direction is reasonable, but this implementation must use or extend the existing gateway attachment contract instead of passing an ACP-only shape through a gateway that ignores it. The live-proof gap also remains before merge.
