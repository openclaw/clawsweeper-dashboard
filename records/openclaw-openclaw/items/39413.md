## Summary

Keep open for maintainer review: the requested shell override is a real feature gap, but current main still lacks `tools.exec.shell` and the PR does not carry the selected interpreter through PTY and approval follow-up execution paths.

## What This Changes

The PR adds an optional global/per-agent `tools.exec.shell` config field and threads it into exec defaults, schema validation, and host shell resolution.

## Best Possible Solution

Ship a maintainer-approved exec shell contract that carries one resolved shell context through direct host runs, PTY, gateway/node approvals, sandbox/node boundaries where applicable, docs, changelog, schema, and targeted tests, using #49931 as the design anchor if this branch is not updated.

## Reproduction Assessment

Yes for the PR defects. Static tracing shows direct non-PTY execution would receive `defaults?.shell`, while PTY supervisor execution and gateway approval follow-up runs still resolve or execute without that shell value; the requested feature itself is not present on current main.

## Solution Assessment

No. The patch is a useful start, but the narrow pass-through is not the best solution for an interpreter-setting contract; the safer path is a shared, typed shell-resolution context with explicit approval/security semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Propagate the shell override to PTY runs:** `src/agents/bash-tools.exec-runtime.ts:383`
  - body: When `usePty` is true, this only places the configured shell in the child fallback argv. The real PTY path passes `ptyCommand` to the supervisor, which calls `getShellConfig()` again without the override, so interactive exec runs silently ignore `tools.exec.shell`.
  - confidence: 0.88
- **[P2] Pass the shell override through approval follow-up runs:** `src/agents/bash-tools.exec.ts:485`
  - body: This forwards `defaults?.shell` only to the immediate exec call. Approval-gated gateway commands are later executed from the follow-up path without any shell value, so approved runs can fall back to the default shell instead of the configured one.
  - confidence: 0.84
- **[P3] Trim the configured shell before using it:** `src/agents/shell-utils.ts:43`
  - body: `if (configuredShell)` accepts whitespace-only strings and attempts to spawn that value as the shell. Trimming first would let blank config fall back to existing default behavior instead of creating a confusing spawn failure.
  - confidence: 0.72
- **[P3] Document the new exec shell config:** `src/config/types.tools.ts:273`
  - body: This adds a user-facing config key but does not update the exec/config docs or changelog, so users and generated config references would not have the new contract documented before release.
  - confidence: 0.7
