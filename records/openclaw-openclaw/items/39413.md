## Summary

Keep this PR open for maintainer review: current main still lacks the requested config, but the patch does not carry the shell override through PTY or approval follow-up execution, and the related design issue calls out security-sensitive WSL/PATH/CWD semantics.

## What This Changes

The PR adds optional global and per-agent `tools.exec.shell` config/schema/default plumbing and passes it into direct host exec shell resolution.

## Best Possible Solution

Ship this only as a maintainer-approved exec contract change with a shared shell context carried through direct, PTY, gateway approval, node, and sandbox paths plus config docs and focused policy tests.

## Reproduction Assessment

Yes for the PR defects. Source tracing shows the shell value is added only to the immediate direct exec call, while PTY supervisor execution and gateway approval follow-up runs still resolve shells without that override.

## Solution Assessment

No. The patch is a useful starting point, but the narrow pass-through is not the best fix for a command-interpreter setting; the safer solution is a shared exec shell context with explicit approval/security semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Propagate the shell override to PTY runs:** `src/agents/bash-tools.exec-runtime.ts:383`
  - body: When `usePty` is true, this change passes the override only into the fallback child argv. The real PTY path sends `ptyCommand` to the supervisor, which resolves `getShellConfig()` again with no override, so interactive exec commands ignore `tools.exec.shell` unless PTY startup fails.
  - confidence: 0.88
- **[P2] Pass the shell override through approval follow-up runs:** `src/agents/bash-tools.exec.ts:485`
  - body: The immediate exec call receives `defaults?.shell`, but gateway approval follow-up execution is launched later without a shell field. Commands that require approval can therefore run under the default shell after the operator approves a command that was configured for another interpreter.
  - confidence: 0.84
- **[P3] Trim blank shell config before spawning:** `src/agents/shell-utils.ts:43`
  - body: The new truthy check accepts strings like `"   "` and will try to spawn whitespace as the executable. Trimming before selection would let blank config fall back to the existing default shell behavior.
  - confidence: 0.72

## Security Review

Status: needs_attention

Summary: The diff changes the command-interpreter boundary without preserving the selected shell through approval follow-up and PTY execution or defining the related policy semantics.

Concerns:
