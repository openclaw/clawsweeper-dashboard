## Summary

Keep open: current main still does not implement `tools.exec.shell`, and this PR remains a useful but incomplete feature patch with blocking gaps in PTY and approval follow-up execution paths. The broader design is already being tracked in #49931, which is the right place to settle the security, PATH, WSL, sandbox, and node semantics before merge.

## What This Changes

The PR adds an optional global/per-agent `tools.exec.shell` config field and threads it into direct host exec shell resolution.

## Best Possible Solution

Ship this only as a maintainer-approved exec contract change with typed config/schema/help/docs plus focused tests for precedence, fallback, Windows argument selection, PTY, gateway approvals, node routing, sandbox behavior, and policy boundaries.

## Reproduction Assessment

Yes. Static tracing of the PR diff against current main gives a high-confidence path: direct non-PTY execution would receive `defaults?.shell`, while PTY supervisor execution and gateway approval follow-up runs still resolve or execute without that shell value.

## Solution Assessment

No. The patch is a useful starting point, but the narrow pass-through is not the best fix for a command-interpreter setting; the safer solution is a shared shell-resolution context carried through every exec path with explicit approval/security semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Propagate the shell override to PTY runs:** `src/agents/bash-tools.exec-runtime.ts:383`
  - body: When `usePty` is true, this only places the configured shell in the child fallback argv. The real PTY path passes `ptyCommand` to the supervisor, which calls `getShellConfig()` again without the override, so interactive exec runs silently ignore `tools.exec.shell`.
  - confidence: 0.88
- **[P2] Pass the shell override through approval follow-up runs:** `src/agents/bash-tools.exec.ts:485`
  - body: This forwards `defaults?.shell` only to the immediate exec call. Approval-gated gateway commands are later executed from the follow-up path without any shell value, so approved runs can fall back to the default shell instead of the configured one.
  - confidence: 0.84
- **[P3] Trim the configured shell before using it:** `src/agents/shell-utils.ts:43`
  - body: `if (configuredShell)` accepts whitespace-only strings and attempts to spawn that value as the shell. Trimming first would let blank config fall back to existing default behavior instead of creating a confusing spawn failure.
  - confidence: 0.72

## Security Review

Status: needs_attention

Summary: Security review needs attention because the diff changes the command-execution interpreter boundary without preserving the selected shell through approval follow-up execution.

Concerns:

- **[medium] Bind shell overrides into approval semantics:** `src/agents/bash-tools.exec.ts:485`
  - body: A configured shell changes how command text is interpreted, but the PR does not carry that shell into gateway approval follow-up execution and does not add policy coverage for allowlists, safeBins, PATH, WSL, sandbox, or node routing. That can make the execution context differ from what approval policy or the operator expected.
  - confidence: 0.78
