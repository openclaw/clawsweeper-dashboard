## Summary

Keep open. Current main still lacks `tools.exec.shell`, and this PR is not merge-ready because the proposed override reaches only the direct host exec path while PTY and gateway approval follow-up execution would still use the default shell; the open #49931 discussion remains the right design anchor for the approval, PATH, WSL, and validation semantics.

## What This Changes

The PR adds an optional global/per-agent `tools.exec.shell` config field, threads it through exec defaults, and teaches the direct host shell resolver to use the configured shell path before falling back to existing defaults.

## Best Possible Solution

Ship the feature only after the shell override is treated as a complete exec contract: typed config/schema/help/docs, one shared shell-resolution context, consistent direct/PTY/gateway/node/sandbox behavior, and focused tests for precedence, fallback, Windows argument selection, approval follow-ups, and policy boundaries, ideally coordinated through #49931.

## Reproduction Assessment

Yes. Static tracing of the PR diff and current main gives a high-confidence reproduction path: a configured shell would reach direct non-PTY `runExecProcess`, while `pty: true` and gateway approval follow-up execution still resolve or run without that shell value.

## Solution Assessment

No. The proposed patch is a useful starting point, but it is too narrow for a command-interpretation setting; the safer solution is to pass a shared shell-resolution context through every exec path and settle the approval/security semantics before documenting the config.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Propagate the shell override to PTY runs:** `src/agents/bash-tools.exec-runtime.ts:383`
  - body: When `usePty` is true, this change only puts the configured shell in `childFallbackArgv`. The real PTY path passes `ptyCommand` to the supervisor, which calls `getShellConfig()` again without the override, so interactive exec runs silently ignore `tools.exec.shell`.
  - confidence: 0.88
- **[P2] Pass the shell override through approval follow-up runs:** `src/agents/bash-tools.exec.ts:485`
  - body: This forwards `defaults?.shell` only to the immediate exec call. Approval-gated gateway commands are later executed from the gateway follow-up path without any shell value, so approved runs can fall back to the default shell instead of the configured one.
  - confidence: 0.84
- **[P3] Trim the configured shell before using it:** `src/agents/shell-utils.ts:43`
  - body: `if (configuredShell)` accepts whitespace-only strings and attempts to spawn that value as the shell. Trimming first would let blank config fall back to the existing default behavior instead of creating a confusing runtime spawn failure.
  - confidence: 0.72

## Security Review
