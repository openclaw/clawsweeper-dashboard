## Summary

Keep open: the feature is not implemented on current main, and this PR is not merge-ready because it only partially threads the shell override through exec paths that have approval and PTY semantics. The open #49931 issue is the better place to settle the remaining product/security contract before implementation.

## What This Changes

The PR adds optional global/per-agent `tools.exec.shell` config plumbing and passes that value into host exec shell resolution.

## Best Possible Solution

Ship a maintainer-approved exec shell-selection contract that normalizes the configured shell once and carries it consistently through direct, PTY, approval follow-up, gateway/node, and sandbox paths with docs, generated schema/help, changelog, and focused tests.

## Reproduction Assessment

Yes. Static tracing of the PR diff against current main shows direct non-PTY execution would receive `defaults?.shell`, while PTY supervisor execution and approval follow-up runs still execute without that shell context.

## Solution Assessment

No. The patch is a useful sketch, but the maintainable fix needs a single normalized shell context with explicit approval/security semantics across all exec execution paths.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Propagate shell overrides into PTY supervisor runs:** `src/agents/bash-tools.exec-runtime.ts:383`
  - body: When `usePty` is true, this only places the configured shell in the child fallback argv. The real PTY path passes `ptyCommand` to the supervisor, which calls `getShellConfig()` again without the override, so interactive exec runs silently ignore `tools.exec.shell`.
  - confidence: 0.88
- **[P2] Carry shell overrides into approval follow-up execution:** `src/agents/bash-tools.exec.ts:485`
  - body: This forwards `defaults?.shell` only to the immediate exec call. Approval-gated gateway commands are later executed from the follow-up path without any shell value, so approved runs can fall back to the default shell instead of the configured one.
  - confidence: 0.84
- **[P3] Trim blank shell overrides before resolving:** `src/agents/shell-utils.ts:43`
  - body: `if (configuredShell)` accepts whitespace-only strings and attempts to spawn that value as the shell. Trimming first would let blank config fall back to existing default behavior instead of creating a confusing spawn failure.
  - confidence: 0.72
- **[P3] Add the public config docs and generated schema surfaces:** `src/config/types.tools.ts:271-272`
  - body: This introduces a public `tools.exec.shell` config key, but the PR does not update the generated config/help/docs surfaces or changelog. That leaves the option undiscoverable and misses the repo policy for user-facing feature changes.
  - confidence: 0.76

## Security Review

Status: needs_attention

Summary: The diff changes command interpreter selection without preserving that execution context through approval-backed runs.

Concerns:

- **[medium] Bind shell overrides into approval semantics:** `src/agents/bash-tools.exec.ts:485`
  - body: A configured shell changes how command text is interpreted, but the PR does not carry that shell into gateway approval follow-up execution or define policy coverage for allowlists, safe bins, PATH, WSL, node, and sandbox behavior. That can make the approved execution context differ from what policy or the operator expected.
