## Summary

Close as duplicate/superseded: the remaining configurable exec shell work is already tracked by the canonical open issue, while this PR is an incomplete, conflicting implementation of the same security-sensitive exec contract.

## What This Changes

The PR adds optional global/per-agent `tools.exec.shell` config plumbing and passes that value into host exec shell resolution.

## Best Possible Solution

Keep the canonical issue open for a maintainer-approved shell-selection contract, and close this stale branch rather than splitting design review across duplicate implementations.

## Reproduction Assessment

Yes. Static tracing shows the PR would pass `defaults?.shell` only to direct host exec, while PTY supervisor and approval follow-up paths still resolve shell context independently; current main also lacks the requested config key.

## Solution Assessment

No. A simple string passthrough is not the narrow maintainable solution for this command-execution boundary; the contract needs consistent PTY, approval, node/sandbox, policy, docs, and test coverage and is already tracked at https://github.com/openclaw/openclaw/issues/49931.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Propagate shell overrides into PTY supervisor runs:** `src/agents/bash-tools.exec-runtime.ts:383`
  - body: When `usePty` is true, this only places the configured shell in the child fallback argv. The real PTY path passes `ptyCommand` to the supervisor, which calls `getShellConfig()` again without the override, so interactive exec runs silently ignore `tools.exec.shell`.
  - confidence: 0.88
- **[P2] Carry shell overrides into approval follow-up execution:** `src/agents/bash-tools.exec.ts:485`
  - body: This forwards `defaults?.shell` only to the immediate exec call. Approval-gated gateway commands are later executed from the follow-up path without any shell value, so approved runs can fall back to the default shell instead of the configured one.
  - confidence: 0.84
- **[P3] Trim blank shell overrides before resolving:** `src/agents/shell-utils.ts:43`
  - body: `if (configuredShell)` accepts whitespace-only strings and attempts to spawn that value as the shell. Trimming first would let blank config fall back to existing default behavior instead of creating a confusing spawn failure.
  - confidence: 0.72

## Security Review

Status: needs_attention

Summary: The patch changes command interpreter selection without preserving that execution context through approval-backed and PTY runs.

Concerns:

- **[medium] Bind shell overrides into approval semantics:** `src/agents/bash-tools.exec.ts:485`
  - body: A configured shell changes how command text is interpreted, but the PR does not carry that shell into gateway approval follow-up execution or define policy coverage for allowlists, safe bins, PATH, WSL, node, and sandbox behavior.
  - confidence: 0.78
