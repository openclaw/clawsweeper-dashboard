## Summary

Keep this PR open. Current main still lacks the proposed `tools.exec.shell` config contract, and the PR diff is still incomplete for PTY execution and gateway approval follow-up runs; the related open design issue #49931 remains the better anchor for the remaining exec/security decisions.

## What This Changes

The PR adds an optional `tools.exec.shell` field to exec config/types/schema, resolves global/per-agent defaults, and passes the configured shell into the direct host exec shell resolver.

## Best Possible Solution

Consolidate the feature under #49931 or update this PR into a complete exec shell override contract: typed global and per-agent config, strict schema/help/docs, one shared shell-resolution path, consistent direct/PTY/gateway/node/sandbox behavior, and focused tests for precedence, fallback, Windows arguments, approval follow-ups, and security policy boundaries.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: with the PR applied, a configured shell reaches the direct non-PTY call, but PTY execution and gateway approval follow-up still resolve the shell independently or re-run without the override.

## Solution Assessment

No. The PR is a useful start, but the narrow pass-through is not the best maintainable solution for a command-execution setting because it leaves major execution paths inconsistent and does not settle the approval/security contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Propagate the shell override to PTY runs:** `src/agents/bash-tools.exec-runtime.ts:383`
  - body: When `usePty` is true, this change only puts the configured shell in `childFallbackArgv`. The actual PTY path passes `ptyCommand` to the supervisor, which calls `getShellConfig()` again without the override, so interactive commands silently ignore `tools.exec.shell`.
  - confidence: 0.88
- **[P2] Pass the shell override through approval follow-up runs:** `src/agents/bash-tools.exec.ts:485`
  - body: This forwards `defaults?.shell` only to the immediate exec call. Approval-gated gateway commands are later executed from the gateway follow-up path without any shell value, so the same command can be approved or resumed under the default shell instead of the configured one.
  - confidence: 0.84
- **[P3] Trim the configured shell before using it:** `src/agents/shell-utils.ts:43`
  - body: `if (configuredShell)` accepts whitespace-only config values and tries to spawn that string as the shell. Trimming before resolution would make empty/blank config fall back to the existing default behavior instead of producing a confusing runtime spawn error.
  - confidence: 0.72
