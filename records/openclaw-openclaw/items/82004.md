## Summary

Keep open: the response_format forwarding fix is pointed at the right gateway/runtime boundary, but the PR introduces a native Codex Responses regression by deleting the whole text payload after wrapper hooks, which can drop existing text.verbosity behavior.

## What This Changes

The PR threads chat-completions response_format through gateway stream params, embedded-runner extra params, and OpenAI Chat/Responses transport builders with validation and regression tests.

## Best Possible Solution

Keep the response_format forwarding, but narrow native Codex sanitization to remove only unsupported structured-output text.format while preserving existing text keys such as verbosity, with targeted regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: native Codex Responses can get text.verbosity from the existing wrapper, and the PR then deletes text during post-hook sanitization.

## Solution Assessment

No, not quite as written. The gateway forwarding and Chat/Responses mapping are the right shape, but the native Codex sanitizer is too broad and should preserve supported text fields while stripping only the unsupported structured-output format.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Preserve native Codex text verbosity:** `src/agents/openai-transport-stream.ts:1274`
  - body: Adding `text` to `OPENAI_CODEX_RESPONSES_UNSUPPORTED_PARAMS` deletes the entire `text` object after `onPayload` wrappers run. The existing OpenAI text verbosity wrapper still injects `text.verbosity` for `openai-codex-responses`, and current tests document that contract, so native Codex Responses requests with configured verbosity will silently lose it. Strip only `text.format` or otherwise preserve the existing `text` keys.
