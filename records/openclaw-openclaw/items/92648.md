Full review comments:

- **[P1] Do not time out every waiting TaskFlow:** `src/tasks/task-flow-registry.maintenance.ts:62`
  - body: This makes `waiting` flows expire solely from `updatedAt` after five minutes. The managed-flow API and docs expose durable waits with no deadline, so an existing approval or webhook workflow that legitimately waits longer will be marked `lost` the next time maintenance runs. Please make abandonment explicit in the controller/API contract, or only target a proven orphan signal, before changing the default lifecycle.
  - confidence: 0.86
- **[P2] Do not leave recovery behind the CLI sweep:** `src/tasks/task-flow-registry.maintenance.ts:214-218`
  - body: The new expiry only runs inside `runTaskFlowRegistryMaintenance`, whose production caller is `openclaw tasks maintenance`; I found no gateway startup or runtime caller. That means unattended sessions described in the linked issue can remain stuck until an operator manually runs the CLI, so this cannot close the heartbeat deadlock as-is.
  - confidence: 0.84
