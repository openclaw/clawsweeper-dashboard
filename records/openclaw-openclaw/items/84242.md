## Summary

Keep open: current main still has a plausible policy/materialization gap for the reported Codex dynamic tool surface. The memory-lancedb plugin declares and registers the tools, but Codex builds its dynamic tools through the normal OpenClaw tool policy path, where the common coding profile only grants core memory tools unless plugin tools are explicitly allowed.

## What This Changes

The issue asks OpenClaw/Codex to expose memory-lancedb's registered memory tools, especially memory_store, or explain why policy excluded them.

## Best Possible Solution

Make the selected memory plugin's agent tools either intentionally visible through a documented allow path or clearly diagnosed when current tool policy hides them.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection gives a credible reproduction path: memory-lancedb selected under a restrictive coding-style tool profile can be registered yet omitted from the Codex dynamic tool surface unless explicitly allowed.

## Solution Assessment

Unclear. Exposing memory_store by default may be the product fix, but because it is write-capable, maintainers should decide whether to auto-allow selected memory plugin tools or add a clear diagnostic and docs for the explicit allow path.
