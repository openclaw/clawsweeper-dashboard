## Summary

Current main does not clearly satisfy the report: the LanceDB memory plugin declares and registers the reported tools, but the default coding/OpenClaw tool materialization path still appears to expose only the core `memory_search`/`memory_get` memory tools unless an explicit plugin/tool allowlist admits the LanceDB tools. Keep this open for maintainer policy on whether write/delete memory plugin tools should be visible by default or require explicit opt-in.

## What This Changes

The issue asks for `@openclaw/memory-lancedb` tools such as `memory_store`, `memory_recall`, and `memory_forget` to appear in Codex/OpenClaw dynamic tool surfaces when the plugin is enabled.

## Best Possible Solution

Maintainers should choose the memory-tool exposure policy, then either expose selected memory plugin tools through a generic memory-slot/default-tool contract with tests and diagnostics or document explicit opt-in for write/delete tools.

## Reproduction Assessment

Yes from source for the default path, but not as a live run here: the plugin registers the tools while the default profile/allowlist path only admits the core memory names unless the LanceDB tools are explicitly allowed.

## Solution Assessment

Unclear until maintainers choose policy; the likely best fix is a generic memory-slot exposure or diagnostic path rather than hardcoding LanceDB tool names, with explicit opt-in docs if write/delete tools should stay disabled by default.
