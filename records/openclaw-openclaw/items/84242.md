## Summary

Keep open: current main still shows a real Codex/tool-policy boundary for this report. The memory-lancedb plugin declares and registers the tools, docs describe them as agent tools, but the common coding profile and Codex dynamic-tool path do not automatically grant or diagnose selected memory-plugin tools; the latest issue follow-up also reports a negative beta canary.

## What This Changes

The issue asks for memory-lancedb's registered `memory_store`, `memory_recall`, and `memory_forget` tools to be exposed to eligible Codex/OpenClaw agent turns or clearly diagnosed when policy hides them.

## Best Possible Solution

Define the selected memory-plugin tool contract for agent turns: either expose its declared tools through a documented allow path by default or emit a clear diagnostic and docs when current tool policy hides them, with focused Codex dynamic-surface coverage.

## Reproduction Assessment

No current-main live reproduction was run in this read-only review. The source path plus the May 20 negative canary give a credible reproduction: selected memory-lancedb can be registered while a coding-style tool profile omits its plugin-owned tools from the Codex dynamic surface.

## Solution Assessment

Unclear. Exposing memory_store by default may be the right bug fix, but because it is write-capable, maintainers should choose between automatic selected-memory grants and a clearer explicit allow/diagnostic path.
