## Summary

Keep open: current main still has the reported policy/materialization gap. The memory-lancedb plugin declares and registers the tools and the docs say agents get them, but the Codex dynamic tool path still flows through the coding tool policy where only core memory_search/memory_get are in the profile unless memory-lancedb tools are explicitly allowed.

## What This Changes

The issue reports that memory-lancedb's registered memory_store, memory_recall, and memory_forget tools are absent from Codex/OpenClaw agent dynamic tool surfaces, preventing agent-initiated LanceDB writes.

## Best Possible Solution

Make selected memory plugin tools intentionally visible to eligible agent tool surfaces through a documented policy path, or emit a clear diagnostic when current tool policy excludes write-capable memory tools.

## Reproduction Assessment

No high-confidence current-main live reproduction was run. The issue includes a negative 2026.5.19-beta.2 canary and current source still shows Codex dynamic tools flowing through a policy path that omits memory_store unless maintainers choose an allow path.

## Solution Assessment

Unclear. The narrow technical fix is to carry selected memory plugin tool contracts into Codex/embedded construction or diagnostics, but maintainers need decide the safe default for exposing the write-capable memory_store tool.
