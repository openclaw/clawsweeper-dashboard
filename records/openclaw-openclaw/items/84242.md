## Summary

Keep open: current main still supports the reported gap. The plugin declares and registers the LanceDB memory tools, but the common coding tool policy still grants only the core memory tools unless the LanceDB tool names are explicitly allowed, and the 5.19-beta.2 follow-up reports the same absence in a staged gateway canary.

## What This Changes

The issue asks for memory-lancedb's registered memory_store, memory_recall, and memory_forget tools to be materialized or clearly diagnosed in the Codex/OpenClaw agent dynamic tool surface.

## Best Possible Solution

Define the selected memory plugin tool policy explicitly: either make memory-lancedb's active-slot tools visible through a documented allow path, or emit a clear diagnostic when the current profile hides them.

## Reproduction Assessment

No high-confidence current-main live reproduction was run. Source inspection plus the 5.19-beta.2 negative canary give a credible reproduction path: memory-lancedb can be loaded and registered while coding-profile policy omits memory_store, memory_recall, and memory_forget from the dynamic tool surface.

## Solution Assessment

Unclear. Exposing the active memory plugin tools by default may be the right fix, but memory_store is write-capable, so maintainers should choose between default exposure, explicit allow guidance, and a diagnostic path.
