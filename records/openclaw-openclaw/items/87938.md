## Summary

Keep open: current main and v2026.6.6 still leave the reported Feishu DM session-state problem. The documented `per-account-channel-peer` setting is a useful multi-account workaround, but direct DM rows remain maintenance-removable and Feishu direct peer IDs are still accepted without Feishu-specific malformed `open_id` validation or repair.

## What This Changes

The issue reports Feishu direct-message sessions being rebuilt after gateway restarts due to duplicate Feishu session keys and session-store maintenance pruning.

## Best Possible Solution

Keep this issue open and choose a maintainer-approved direct-DM retention policy plus Feishu-boundary `open_id` validation/doctor repair, while keeping `per-account-channel-peer` as the documented multi-account workaround.

## Reproduction Assessment

Yes at source level for the maintenance and multi-account key paths: current main prunes/caps unprotected direct sessions and `per-channel-peer` omits `accountId`. The malformed near-duplicate `open_id` subclaim still needs a live Feishu repro or sanitized store fixture.

## Solution Assessment

No current implementation is not the final fix. `per-account-channel-peer` is the right documented workaround for multi-app collisions, but the best code path still needs a retention-policy decision and Feishu identity validation or repair.
