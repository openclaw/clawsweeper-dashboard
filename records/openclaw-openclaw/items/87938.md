## Summary

Keep open: current main still leaves the central session-state risk. ## What This Changes

Issue reports Feishu DM session rebuilding after gateway restart from direct-session maintenance pruning plus Feishu multi-app or malformed `open_id` session-key collisions.

## Best Possible Solution

Split the fix into a maintainer-approved direct-DM retention policy plus Feishu-boundary validation/repair for malformed `open_id` keys, while keeping `per-account-channel-peer` as the documented multi-account setup.

## Reproduction Assessment

Yes at source level for the maintenance and multi-account key paths: current main prunes/caps unprotected direct sessions and `per-channel-peer` omits `accountId`. The malformed near-duplicate `open_id` subclaim still needs a live Feishu repro or sanitized store fixture.

## Solution Assessment

Unclear as a single fix: the existing `per-account-channel-peer` path is the right workaround for multi-app collisions, but a durable code fix needs maintainer direction on direct-DM retention and malformed Feishu identity repair.
