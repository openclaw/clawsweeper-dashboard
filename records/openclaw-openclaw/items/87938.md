## Summary

Keep open: current main and the latest release still leave the Feishu direct-message session-state problem only partially mitigated. High-water batching reduces the original cap-at-every-write cascade, but direct DM rows remain maintenance-removable, `per-channel-peer` still omits account identity, and Feishu direct peer IDs are still accepted without Feishu-specific malformed `open_id` validation or repair.

## What This Changes

The issue reports Feishu direct-message sessions being rebuilt after gateway restart because direct-session keys can collide across accounts and maintenance can prune direct-session rows.

## Best Possible Solution

Keep the issue open and choose a maintainer-approved direct-DM retention policy plus Feishu-boundary `open_id` validation or doctor repair, while preserving `per-account-channel-peer` as the documented multi-account workaround.

## Reproduction Assessment

Partly: the source path is clear for direct DM rows being unprotected and for `per-channel-peer` omitting `accountId`. The malformed near-duplicate Feishu `open_id` subclaim still needs a live repro or sanitized state fixture for high-confidence end-to-end reproduction.

## Solution Assessment

No, current main is not the final fix. `per-account-channel-peer` is the right documented workaround for multi-app collisions, but the best code path still needs a maintainer-approved retention policy and Feishu identity validation or repair.
