## Label Justifications

- `impact:session-state`: The report is about direct-message session keys being rebuilt or split, causing session identity/state drift.
- `impact:data-loss`: Session-store maintenance can remove direct-session index entries and archive/delete associated session artifacts from the active store.

## Summary

Keep open: current main still has the central session-state risk. Feishu direct-session keys are derived from the inbound sender identity, and direct sessions are still removable by session-store maintenance while only the active write key is preserved.

## What This Changes

The issue reports Feishu direct-message sessions being rebuilt after gateway restart because direct-session entries can be pruned and duplicate Feishu open_id key variants can split one user's session state.

## Best Possible Solution

Preserve valid Feishu DM session bindings through maintenance and add an account-aware identity/repair path for duplicate direct keys, with focused regression coverage for both cleanup and multi-account Feishu routing.

## Reproduction Assessment

Yes at source level for the main mechanics: direct sessions are not protected by maintenance and Feishu DMs key off senderOpenId. I did not establish a live Feishu replay for the reported near-duplicate open_id variants.

## Solution Assessment

Unclear as a direct implementation path: increasing maxEntries or protecting every direct session would reduce symptoms but may weaken retention semantics. The better fix likely needs maintainer choice around Feishu identity normalization, account scoping, and doctor-style repair for existing duplicate entries.
