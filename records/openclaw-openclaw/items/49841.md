## Summary

Keep open: current main still has only file-backed gateway config loading, and this broad Nacos config-source PR still has correctness, security-contract, proof, conflict, and CI blockers before it can merge.

## What This Changes

The PR adds an opt-in Nacos-backed gateway config source with env selection, file/Nacos adapters, long-poll reload subscription, CLI/doctor/startup wiring, docs, changelog, and tests.

## Best Possible Solution

If maintainers want remote config in core, land a generic pre-start config-source seam plus a Nacos adapter that preserves write/reload semantics, documents the security and persistence model, and is proven against a real Nacos server.

## Reproduction Assessment

Yes for the blocking PR findings: source inspection of the PR head shows Nacos in-process writes do not enter the reload notification path, and the docs contradict standalone CLI write behavior. No for the full runtime feature path because the PR lacks real Nacos-backed gateway proof.

## Solution Assessment

No. The source abstraction is plausible, but this PR is not merge-ready until Nacos writes preserve or explicitly reject reload semantics, docs match behavior, and maintainers settle the remote-config security and persistence contract.
