Full review comments:

- **[P1] Route startup metadata reads through the active source:** `src/gateway/server-startup-config.ts:202`
  - body: After resolving `OPENCLAW_CONFIG_SOURCE=nacos`, startup still calls `readConfigFileSnapshotWithPluginMetadata()`, and that helper creates a file-backed `ConfigIO` directly. A Nacos-only gateway can therefore boot from the local file/default config while later code believes the active source is Nacos; make the metadata snapshot path consume the resolved source or build plugin metadata around `source.readSnapshot()`.
  - confidence: 0.91
- **[P2] Notify the reloader for Nacos in-process writes:** `src/config/io.ts:2466-2501`
  - body: In Nacos mode this branch refreshes the runtime snapshot and returns without emitting a config write notification, while subscribe-mode reload only listens for external Nacos changes. Successful `config.apply` or `config.patch` calls can update `config.get` but leave hooks, cron, channels, or auth-adjacent reload state stale until a remote edit or restart; route these writes through the same reload planner as file-backed writes.
  - confidence: 0.88
- **[P3] Correct standalone CLI write behavior in the docs:** `docs/gateway/configuration.md:744`
  - body: The docs say CLI `config set` updates the in-memory snapshot in Nacos mode, but a standalone CLI process has no gateway runtime snapshot or refresh handler, so the PR's Nacos write path throws instead of updating anything useful. Document standalone CLI writes as unsupported for Nacos, or implement an explicit gateway/RPC-backed write flow.
  - confidence: 0.83
