## Summary

Keep open. This is a plausible but broad core gateway/config feature PR, and the current branch still has source-reproducible correctness issues, missing real Nacos behavior proof, and unresolved remote-config security/product questions.

## What This Changes

The branch adds an opt-in Nacos-backed gateway config source with env selection, file/Nacos adapters, long-poll reload subscription, CLI/doctor/startup wiring, docs, changelog, and tests.

## Best Possible Solution

If maintainers want remote config in core, land a generic source seam where Nacos is an explicit validated adapter used consistently by startup, CLI/RPC reads and writes, reload planning, docs, tests, and security guidance; otherwise define the needed pre-start config-source plugin API and move Nacos outside core.

## Reproduction Assessment

Yes for the blocking PR findings: source inspection of PR head 191251c8e028ed9d47522906d31514bbadf7bd4f shows the startup read bypass and the Nacos write/reload notification gap. Real Nacos runtime behavior is not proven by the contributor yet.

## Solution Assessment

No. The source abstraction is plausible, but this patch is not the best mergeable form until startup actually consumes the selected source, Nacos writes preserve or clearly reject reload semantics, standalone CLI behavior is documented accurately, and remote-config security expectations are settled.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Use the selected config source for startup snapshots:** `src/gateway/server-startup-config.ts:202`
  - body: After setting the source, `loadGatewayStartupConfigSnapshot()` still reads through `readConfigFileSnapshotWithPluginMetadata()`, whose top-level wrapper constructs `createConfigIO()` and bypasses `getConfigSource()`. In Nacos-only deployments the gateway can boot from the local/default file while the reloader later subscribes to Nacos.
  - confidence: 0.9
- **[P2] Send Nacos writes through the reload planner:** `src/config/io.ts:2481`
  - body: The Nacos branch updates runtime state and returns without a write notification, and subscribe-mode reloaders never register `subscribeToWrites`. RPC `config.set`/`config.patch` can therefore report success while hooks, cron, channel, or plugin state keep running on the old config until a remote edit or restart.
  - confidence: 0.88
- **[P3] Document standalone CLI writes as unsupported:** `docs/gateway/configuration.md:744`
  - body: The docs say CLI `config set` updates the in-memory snapshot in Nacos mode, but standalone CLI processes now throw when no running gateway snapshot or refresh handler exists. Users would expect a persistent or live update that this code path cannot perform.
  - confidence: 0.83
