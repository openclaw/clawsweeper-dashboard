## Summary

Keep open: current main still has no Nacos config source, and the PR is a broad remote-config feature with a blocking reload semantics bug, missing real Nacos proof, and unresolved security/product guardrails.

## What This Changes

The PR adds an optional Nacos-backed gateway config source with file/Nacos source resolution, long-poll subscriptions, CLI/doctor/startup wiring, docs, changelog, and tests.

## Best Possible Solution

If maintainers want remote config in core, land a generic source seam with Nacos preserving existing reload/write/security semantics; otherwise define the needed pre-start plugin API and keep the Nacos adapter outside core.

## Reproduction Assessment

Yes for the blocking PR findings: source inspection at head 191251c8e028ed9d47522906d31514bbadf7bd4f shows Nacos writes return without a reload notification and subscribe mode bypasses subscribeToWrites. No live Nacos end-to-end reproduction was provided.

## Solution Assessment

No. The abstraction direction is plausible, but this is not the best mergeable form until Nacos mode preserves current config semantics, has a settled remote-config security contract, and is proven against a real Nacos server.
