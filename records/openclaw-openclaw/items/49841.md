## Summary

Keep open: current main does not implement a Nacos config source, and the PR still has a blocking Nacos write/reload semantics issue plus missing real behavior proof and an unsettled remote-config security contract.

## What This Changes

This PR adds an optional env-selected file/Nacos config source abstraction, a Nacos Open API long-poll client, gateway reload/startup/CLI/doctor integration, docs, changelog, and tests.

## Best Possible Solution

If maintainers want remote config in core, land a generic source seam with file as default and a Nacos adapter that preserves write/reload semantics, documents persistence limits, defines auth/error behavior, and has real Nacos validation; otherwise define a pre-start config-source plugin API before moving Nacos out of core.

## Reproduction Assessment

Yes for the blocking PR finding by source inspection: the Nacos write branch returns without emitting a runtime write notification, while subscribe mode bypasses subscribeToWrites. I did not run a live Nacos end-to-end reproduction, and the PR provides no after-fix real behavior proof.

## Solution Assessment

No. The abstraction direction is plausible, but this is not the best mergeable form until Nacos mode preserves current config semantics, has a settled remote-config security contract, and is proven against a real Nacos server.
