## Summary

Keep this PR open: current main still has no Nacos config source, and the proposed remote-config feature remains merge-blocked by reload/write semantics, missing real Nacos behavior proof, and unresolved security/product policy.

## What This Changes

The PR adds an opt-in Nacos-backed gateway config source with env selection, file/Nacos adapters, long-poll reload subscription, CLI/doctor/startup wiring, docs, changelog, and tests.

## Best Possible Solution

If maintainers want remote config in core, land a generic file-default source seam plus a validated Nacos adapter that preserves reload/write semantics, documents security and persistence limits, and is proven against a real Nacos server.

## Reproduction Assessment

Yes for the blocking PR findings: source inspection of the PR diff shows Nacos-mode in-process writes do not enter the reload notification path, and the docs contradict standalone CLI write behavior. No for the full runtime feature path because there is no real Nacos-backed gateway proof.

## Solution Assessment

No. The source abstraction is plausible, but this PR is not the best merge-ready solution until Nacos writes preserve or explicitly reject reload semantics, docs match behavior, and maintainers settle the remote-config security/persistence contract.
