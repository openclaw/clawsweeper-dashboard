## Summary

Keep open: current main does not implement a Nacos config source, and the PR remains a broad remote-config feature with a source-reproducible write/reload regression plus unsettled security guardrails.

## What This Changes

The branch adds an optional Nacos-backed gateway config source with env resolution, file/Nacos adapters, long-poll reload subscription, CLI/doctor/startup wiring, docs, changelog, and tests.

## Best Possible Solution

If maintainers want this in core, land a validated config-source seam where Nacos preserves existing config read/write/reload semantics, documents unsupported persistence clearly, and ships with explicit remote-config security guardrails; otherwise define a pre-start config-source plugin API and move Nacos outside core.

## Reproduction Assessment

Yes, by source inspection of the PR diff and current main contract. In Nacos mode, a gateway config write updates runtime state while the subscribe-mode reloader does not consume the existing in-process write notification path, so hot-reloadable services can stay on the old config until a remote Nacos edit or restart.

## Solution Assessment

No. The source abstraction is plausible, but the current patch is not the best mergeable form until Nacos writes either feed the same reload planner as file writes or are explicitly unsupported, and the remote-config security model is settled.
