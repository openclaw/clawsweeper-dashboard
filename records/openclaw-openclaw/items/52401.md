## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Do not delay terminal lifecycle behind agent_error:** `src/agents/pi-embedded-subscribe.handlers.lifecycle.ts:94-95`
  - body: `handleAgentEnd` now awaits plugin error replacement before the lifecycle `error` event is emitted. Because the command layer emits a synthetic `end` when it has not observed a terminal lifecycle event after the run result returns, any slow hook can still race that fallback even with the 2 s cap; keep terminal state coordinated before awaiting plugin work or otherwise block the fallback from emitting a false success.
  - confidence: 0.88
- **[P2] Gate agent_error behind conversation access:** `src/plugins/hook-types.ts:68`
  - body: Adding `agent_error` to the public hook names without adding it to `CONVERSATION_HOOK_NAMES` lets non-bundled plugins register for provider lifecycle error text without `plugins.entries.<id>.hooks.allowConversationAccess=true`, unlike adjacent conversation hooks. Add it to the guard and cover blocked/allowed registration.
  - confidence: 0.93
- **[P2] Honor per-handler hook timeouts:** `src/plugins/hooks.ts:690-695`
  - body: `runAgentError` directly awaits each handler, so `api.on(..., { timeoutMs })`, operator `hooks.timeouts.agent_error`, and the standard modifying-hook timeout path are skipped. A slow first plugin can consume the lifecycle-level timeout and prevent later handlers from running.
  - confidence: 0.9
- **[P3] Document the public agent_error hook:** `src/plugins/hook-types.ts:253-264`
  - body: This exposes a new typed public hook but leaves `docs/plugins/hooks.md` without its event shape, replacement semantics, access requirement, and timeout behavior. Plugin authors need that contract before relying on it.
  - confidence: 0.82
