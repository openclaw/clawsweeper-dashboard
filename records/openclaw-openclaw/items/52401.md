## Summary

Keep open: the feature is not already on current main, but the PR is not merge-ready because it adds a public provider-error replacement hook without the existing conversation-access guard, delays terminal lifecycle handling on a plugin hook, bypasses standard per-handler timeout semantics, and still lacks real behavior proof.

## What This Changes

The PR adds a public `agent_error` plugin hook, runs it from embedded Pi error finalization so plugins can replace the lifecycle error message, and adds unit tests plus a changelog entry.

## Best Possible Solution

Expose provider-error replacement only through a documented, conversation-access-gated plugin API that preserves terminal lifecycle semantics and uses the standard hook timeout machinery.

## Reproduction Assessment

Yes for the review blockers: PR-head source shows `agent_error` omitted from `CONVERSATION_HOOK_NAMES`, awaited before terminal lifecycle emission, and implemented outside the standard per-handler timeout path. The intended provider-error replacement behavior has not been shown in a real setup.

## Solution Assessment

No. The idea may be useful, but the current implementation exposes a sensitive lifecycle hook without the established guard and puts plugin latency on the terminal lifecycle path; the safer path is a documented guarded API wired through existing timeout and lifecycle contracts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Gate agent_error behind conversation access:** `src/plugins/hook-types.ts:68`
  - body: `agent_error` receives the error text before the lifecycle sanitizer builds the broadcast-safe preview, but adding it only to `PluginHookName` leaves it outside `CONVERSATION_HOOK_NAMES`. That lets non-bundled plugins register the hook without `plugins.entries.<id>.hooks.allowConversationAccess=true`, bypassing the existing guard for conversation/provider lifecycle text.
  - confidence: 0.91
- **[P1] Do not block terminal lifecycle on agent_error:** `src/agents/pi-embedded-subscribe.handlers.lifecycle.ts:94-95`
  - body: This await still holds terminal error emission for up to two seconds before `finalizeAgentEnd()` and `emitLifecycleTerminalOnce()` run. Current command code emits a fallback lifecycle `end` when no terminal lifecycle callback has set `lifecycleEnded`, so a slow plugin can make clients see success before the real error or delay run-completion bookkeeping.
  - confidence: 0.84
- **[P2] Honor per-handler timeouts for agent_error:** `src/plugins/hooks.ts:690-695`
  - body: The new runner awaits each handler directly, so it ignores the `timeoutMs` resolved for typed hooks and the existing modifying-hook timeout path. One slow high-priority handler can consume the outer lifecycle budget and prevent later handlers from running even if maintainers configured a shorter per-handler timeout.
  - confidence: 0.88
