## Summary

Keep this PR open: the requested hook is not implemented on current main, but the current branch still has merge-blocking API/security issues and lacks real behavior proof.

## What This Changes

The branch adds a public `agent_error` plugin hook, invokes it from embedded Pi agent-error lifecycle handling before broadcasting the error, adds hook runner types/tests, and updates the changelog.

## Best Possible Solution

Refresh the PR so `agent_error` is a documented, conversation-access-gated hook that uses the standard hook timeout path and has redacted live provider/plugin proof before merge.

## Reproduction Assessment

Yes for the PR review findings by source inspection: PR head adds `agent_error` while omitting it from `CONVERSATION_HOOK_NAMES` and directly awaits handlers in a custom runner. The requested feature itself is absent on current main, so the PR is not obsolete.

## Solution Assessment

No. A dedicated hook is a plausible seam, but this implementation is not the best landing shape until it matches existing access-control, timeout, documentation, and real-proof expectations for public plugin APIs.

  - body: Adding `agent_error` to the public hook names without adding it to `CONVERSATION_HOOK_NAMES` lets non-bundled plugins register a provider-error lifecycle hook without `plugins.entries.<id>.hooks.allowConversationAccess=true`, unlike adjacent conversation hooks.   - confidence: 0.9
