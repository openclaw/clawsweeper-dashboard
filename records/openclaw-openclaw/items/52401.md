## Summary

Keep open: current main does not have an `agent_error` hook, but this PR still has merge-blocking public plugin API issues around conversation-access policy and hook timeout semantics.

## What This Changes

The PR adds a public `agent_error` plugin hook, calls it from embedded Pi agent-error lifecycle handling before broadcast with a 2s outer timeout, adds lifecycle tests, and records a changelog entry.

## Best Possible Solution

Land the hook only after the branch updates the public API contract: guard `agent_error` like adjacent conversation hooks, preserve hook timeout semantics, document the hook and access policy, and then validate the touched plugin/lifecycle tests.

## Reproduction Assessment

Yes. A high-confidence static reproduction is to register a non-bundled plugin with `api.on("agent_error", ...)` and no `allowConversationAccess`; the PR adds the hook name but not the guarded-name list, so registration would not hit the existing conversation-access block. Another source check is a slow handler with `timeoutMs`, which `runAgentError` awaits directly instead of applying the per-handler timeout helper.

## Solution Assessment

No. A dedicated hook is a plausible seam, but this implementation is not the best way to land it until it follows the existing conversation-access, timeout, docs, and changelog contracts for public plugin APIs.

  - body: Adding `agent_error` to the public hook names without adding it to `CONVERSATION_HOOK_NAMES` lets non-bundled plugins observe provider lifecycle error text without `plugins.entries.<id>.hooks.allowConversationAccess=true`.
