## Summary

Keep open. Current main still lacks the requested hook, but this PR should not merge as-is because the provider-error hook is not guarded as conversation-access data, its custom runner bypasses per-handler timeout handling, and the external PR lacks after-fix real behavior proof.

## What This Changes

The PR adds a public `agent_error` plugin hook, invokes it from embedded Pi lifecycle error handling before broadcasting agent error events, adds lifecycle hook tests, and updates the changelog.

## Best Possible Solution

Land a refreshed hook only after it follows existing plugin contracts: guard provider-error text behind conversation access, preserve per-handler timeout behavior, document the new public hook, and show redacted real behavior proof.

## Reproduction Assessment

Yes for the PR review findings by source inspection: PR head adds `agent_error` as a public hook while omitting it from `CONVERSATION_HOOK_NAMES`, and `runAgentError` awaits hook handlers directly instead of the timeout-aware modifying-hook path. The requested hook itself is not present on current main.

## Solution Assessment

No. A dedicated hook is a plausible seam, but this implementation is not the best landing shape until it matches the existing access-control, timeout, documentation, and real-proof expectations for public plugin APIs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Gate agent_error behind conversation access:** `src/plugins/hook-types.ts:68`
  - body: Adding `agent_error` to the public hook names without adding it to `CONVERSATION_HOOK_NAMES` lets non-bundled plugins observe provider lifecycle error text without `plugins.entries.<id>.hooks.allowConversationAccess=true`. Please guard it like adjacent conversation hooks such as `agent_end`, `llm_input`, and `llm_output`.
  - confidence: 0.93

  - body: This custom loop awaits each `agent_error` handler directly, bypassing the `timeoutMs` handling used by other modifying hooks. A slow handler with its own timeout can block later handlers until the outer lifecycle race drops all replacements, violating the documented per-hook timeout behavior.
  - confidence: 0.9

  - body: The PR exposes a new plugin hook but does not document the event shape, replacement semantics, or required conversation-access policy. Public plugin seams should be documented when introduced so external plugin authors can use them safely.
  - confidence: 0.82
