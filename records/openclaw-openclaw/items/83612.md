## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep the fs-safe Python helper opt-in:** `src/infra/fs-safe-defaults.ts:7`
  - body: The base contract and security docs say OpenClaw keeps fs-safe's optional Python helper off unless the operator sets `FS_SAFE_PYTHON_MODE` or `OPENCLAW_FS_SAFE_PYTHON_MODE`. Flipping the default to `auto` changes existing hosts during upgrade and can start the helper without an explicit operator choice, so this belongs in a dedicated security/infra PR with docs and upgrade proof.
  - confidence: 0.93
- **[P1] Avoid process-global fs-safe mode flips:** `src/infra/archive.ts:43-47`
  - body: This wrapper changes the global fs-safe Python mode to `off` while one archive extraction runs, but the queue only serializes archive extractions. Any concurrent fs-safe operation in the same process can observe the downgraded mode even when the operator configured `auto`, so archive fallback needs an operation-local path or an upstream fs-safe fix.
  - confidence: 0.88
- **[P1] Use upstream ACL classification instead of a local shim:** `src/infra/permissions.ts:35-49`
  - body: Current base tests already cover localized SYSTEM account names through the existing `summarizeWindowsAcl` path, including `NT Authority\\Systéme`. Adding a one-off OpenClaw wrapper creates a competing security classifier with narrower behavior; keep the upstream helper here or fix `@openclaw/fs-safe` if that contract regressed.
  - confidence: 0.86
