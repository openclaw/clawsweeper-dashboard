## Summary

Keep open: the PR is a focused fix for a real DeepSeek/Foundry tool-call failure, but it is not merge-ready because recovered DSML calls can still finish as a normal stop turn, DSML string parameters can be coerced to other types, and the supplied proof is unit-test-only.

## What This Changes

The PR captures stripped DeepSeek DSML blocks, parses `invoke`/`parameter` markup into synthetic `toolCall` blocks, wires that recovery into the OpenAI completions stream, and adds parser/filter tests.

## Best Possible Solution

Land one corrected DSML recovery implementation that marks recovered calls as `toolUse`, preserves DSML-declared string parameter types, validates malformed and prompt-echo cases, and includes redacted live DeepSeek/Foundry proof from a patched build.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: the linked issue has live CLI evidence, and current main only promotes native `choiceDelta.tool_calls` while the DeepSeek filter strips DSML text. I did not run live provider credentials in this read-only review.

## Solution Assessment

Is this the best way to solve the issue? Not yet. Capturing DSML in the filter is plausible and improves on the older competing PR, but the implementation must set tool-use stop reason, preserve DSML string typing, and add live patched-runtime proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Mark recovered DSML calls as tool use:** `src/agents/openai-transport-stream.ts:2740-2742`
  - body: The observed fallback is DSML emitted as assistant text rather than native `tool_calls`, so providers can finish that chunk with `finish_reason: "stop"`. This patch appends synthetic `toolCall` blocks but never changes a stopped completion to `toolUse`; the existing Responses path does that when tool-call content is present. Please set `output.stopReason = "toolUse"` when recovered calls are injected and add a stream test with DSML content plus `finish_reason: "stop"`, otherwise the recovered call can still be treated as a normal terminal answer instead of a tool turn.
  - confidence: 0.9
- **[P2] Honor DSML string parameters:** `src/agents/deepseek-text-filter.ts:138`
  - body: DSML parameters can carry `string="true"`, including the reported `sessionKey` example, but the parser ignores the attribute and runs `JSON.parse` on every parameter body. Values like `00123`, `42`, `false`, or `null` would be passed to tools with the wrong type even though the provider marked them as strings. Capture the parameter attributes and skip JSON coercion for string parameters, with a regression test for a numeric-looking string.
  - confidence: 0.86
