## What This Changes

Adds a 256 KB-named cap to DeepSeek DSML tool-call recovery and a regression test for an oversized DSML block.

## Best Possible Solution

Land the bounded DSML recovery fix after enforcing the cap with UTF-8 byte accounting, adding multibyte regression coverage, and supplying redacted runtime or loopback-provider proof for the oversized unclosed DSML path.

## Reproduction Assessment

Yes, source-reproducible: current main appends streamed chunks to the DSML recovery `buffer` and returns without clearing it when an open token has no close token. I did not run a live oversized stream in this read-only review.

## Solution Assessment

No, not yet: the recoverer is the right owner path and a cap matching sibling stream limits is the right shape, but the implementation should compare UTF-8 bytes rather than string length and needs real runtime or loopback proof.
