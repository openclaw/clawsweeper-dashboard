## Label Justifications

- `P1`: The reported failure mode is unbounded agent-process memory growth during provider streaming, which can break active agent workflows.
- `merge-risk: 🚨 message-delivery`: The new cap intentionally suppresses recovered DSML tool calls whose buffered payload exceeds 256,000 UTF-8 bytes.

## Summary

Keep open: current main and v2026.6.9 still have the uncapped DSML recovery buffer, and this PR is a focused candidate fix with sufficient terminal proof and no blocking findings. Maintainers should still accept the intentional 256,000-byte cutoff semantics before merge because very large recovered DSML tool calls will be suppressed.

## What This Changes

The PR adds a byte-counted 256,000-byte cap for DeepSeek DSML recovery buffering and regression tests for oversized ASCII and multibyte DSML blocks.

## Best Possible Solution

Land the focused cap after maintainers accept the 256,000-byte recovery cutoff because it matches the sibling stream-buffer safety limits.

## Reproduction Assessment

Yes. Current-main source inspection shows the DSML recoverer retains an opened block across chunks and only clears it on close-token recovery or final flush, with no size cap in the no-close branch.

## Solution Assessment

Yes. The proposed cap is in the helper that owns the retained DSML recovery state, and downstream recovered text/tool-call routing stays on the existing parser pipeline.
