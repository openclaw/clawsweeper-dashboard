## Label Justifications

- `P1`: The PR targets agent-runtime memory growth from provider text streams, which can affect live agent availability and tool-call execution.
- `merge-risk: 🚨 availability`: The proposed cap still misses complete oversized DSML blocks, leaving a provider-controlled memory-pressure path in the changed safety surface.
- `merge-risk: 🚨 message-delivery`: When the cap does fire, oversized DSML tool-call markup is suppressed instead of recovered, so tool delivery behavior changes above the threshold.

## Summary

Keep open: the narrowed PR is now aimed at the active OpenAI completions DSML recoverer, but the new cap is still chunking-dependent and can be bypassed by an oversized complete DSML block that arrives with its close token in the same stream delta; the contributor proof is also focused test output rather than after-fix OpenClaw runtime proof.

## What This Changes

The PR adds a 256 KB DeepSeek DSML recovery-buffer limit in `src/agents/openai-transport-stream.ts` and a regression test for oversized DSML recovery input.

## Best Possible Solution

Apply one UTF-8 byte cap to the retained DSML block before both incomplete buffering and complete-block parsing, keep normal DSML recovery unchanged below the limit, and add redacted runtime proof for the oversized path.

## Reproduction Assessment

Yes. Source inspection shows current main retains DSML content until a close token or final flush, and the PR's synthetic stream test demonstrates the split-close oversized path, though I did not run tests in this read-only review.

## Solution Assessment

No. The PR is the right layer, but the current condition only caps no-close buffering; the best fix also caps complete DSML bodies before parsing and uses byte measurement consistent with sibling limits.
