## Summary

Keep open: this targets a real DeepSeek/Foundry tool-call loss, but the patch still promotes unclosed DSML spans and injects recovered DSML even when native tool calls already arrived. The PR also still lacks after-fix proof from a patched real runtime/provider path.

## What This Changes

The branch captures DeepSeek DSML spans in the text filter, parses `invoke`/`parameter` markup into synthetic `toolCall` blocks at stream end, and adds parser/filter unit coverage.

## Best Possible Solution

Keep recovery scoped to complete DSML blocks, preserve native `tool_calls` as authoritative, validate or owner-approve the recovered-name boundary, then add stream-level regression tests and redacted real runtime proof.

## Reproduction Assessment

Yes, source-reproducible: current main strips DSML text and only emits executable tool calls from native `choiceDelta.tool_calls`, while the linked issue provides live Foundry/DeepSeek output showing DSML returned as visible text. I did not run live provider credentials in this read-only review.

## Solution Assessment

No. The recovery direction is plausible, but this patch needs closed-wrapper gating, native-tool-call precedence, and real provider/runtime proof before it is the safest maintainable fix.
