## Label Justifications

- `merge-risk: 🚨 message-delivery`: The changed threshold controls whether oversized DSML text is recovered as a tool call or discarded from the recovered tool-call path.

## Summary

No blocking findings: this remains a useful active fix candidate because current main and v2026.6.9 still lack a DSML recovery-buffer cap, and the PR adds a byte-accurate 256 KB cap with focused regression coverage. The main maintainer choice is accepting the intentional behavior that oversized DSML blocks stop being recovered as tool calls.

## What This Changes

Adds a 256,000-byte UTF-8 cap to the DeepSeek DSML tool-call recovery buffer and tests oversized ASCII and multibyte DSML payloads.

## Best Possible Solution

Land this PR or an equivalent byte-counted recoverer cap after maintainers accept the 256 KB oversized-DSML behavior.

## Reproduction Assessment

Yes. Source inspection on current main shows the DSML recoverer retains an opened block across chunks and only clears it on close-token discovery or final flush, with no size cap before this PR.

## Solution Assessment

Yes. The recoverer is the narrow owner of the unbounded buffer, and a UTF-8 byte cap matching the sibling stream caps is the cleanest fix; adding only a downstream text-filter cap would leave the recoverer itself unbounded.
