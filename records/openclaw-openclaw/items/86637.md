## Summary

Keep this PR open, but it is not merge-ready: current main still lacks a DSML recovery cap, yet this patch only guards the no-close path and leaves oversized complete DSML blocks able to bypass the intended availability bound before parsing.

## Best Possible Solution

Apply one byte-measured DSML size limit before parsing or retaining any recovered DSML block, cover unclosed and same-delta/late-close oversized cases, and require redacted runtime proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Cap complete DSML blocks before parsing:** `src/agents/openai-transport-stream.ts:3125-3127`
  - body: This guard only fires when no close token is found. If the provider sends an oversized DSML body and the close token in the same delta, or crosses the limit on the delta that also includes the close token, this branch is skipped and the huge body is still passed to `parseDeepSeekDsmlToolCallBlock()` and can become a synthetic tool call. Apply the cap before parsing any complete DSML block and cover that close-present case in the regression test.
  - confidence: 0.9
- **[P2] Measure the new DSML cap in bytes:** `src/agents/openai-transport-stream.ts:3125`
  - body: The new constant is named and described as a byte cap, but `buffer.length` counts UTF-16 code units while the existing stream caps use UTF-8 byte measurement. Multibyte DSML content can exceed the intended 256 KB bound before this check fires; use the same byte measurement helper or equivalent here.
  - confidence: 0.86
