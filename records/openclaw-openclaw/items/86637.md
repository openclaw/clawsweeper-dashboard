## Label Justifications

## Summary

Keep open: current main still drops DeepSeek DSML text without recovering executable tool calls, so the linked bug remains real, but this PR is not merge-ready because the proof is mock/source-only and the patch adds security-sensitive text-to-tool execution with concrete scoping and buffering defects.

## What This Changes

The branch captures complete DeepSeek DSML invoke blocks, parses them into synthetic OpenAI completions toolCall blocks, discards truncated captures, skips recovery when native tool calls already exist, and adds parser/filter tests.

## Best Possible Solution

Land one canonical guarded recovery path that only promotes intended DSML tool-call wrappers, caps recovered/captured payloads like native tool calls, and includes redacted patched-runtime proof for the DeepSeek/Foundry tool-use path.

Full review comments:

- **[P1] Keep error DSML blocks non-executable:** `src/agents/deepseek-text-filter.ts:97-100`
  - body: `recoveredToolCalls()` parses every captured DSML block, but the filter still captures all DSML kinds, including `tool_use_error`. That means an error wrapper with invoke-shaped markup can be promoted into a `toolCall` even though the linked bug only needs recovery from tool-call wrappers; track the opened kind and recover only intended tool-call wrappers before promoting anything to execution.
  - confidence: 0.83
- **[P2] Bound DSML capture before retaining hidden text:** `src/agents/deepseek-text-filter.ts:51`
  - body: This appends every byte inside a DSML block into `dsmlCapture`, while the native tool-call path below caps streamed arguments at `MAX_TOOL_CALL_ARGUMENT_BUFFER_BYTES`. A malformed or very large provider DSML block that used to be dropped is now retained, parsed, and JSON-stringified outside that guard, so add a capture/recovered-argument limit and discard over-limit blocks.
  - confidence: 0.8
