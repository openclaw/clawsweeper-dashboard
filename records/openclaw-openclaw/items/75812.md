## Summary

Keep open. Current main still lacks an agent-id-scoped restart lifecycle surface, and the related restart/follow-up work is adjacent rather than a terminal implementation; this is a feature that needs maintainer lifecycle design.

## What This Changes

Request an `openclaw agents restart <agent-id>` or equivalent API that restarts one agent's session/provider lifecycle without restarting the whole Gateway.

## Best Possible Solution

Design an additive admin-scoped Gateway lifecycle API plus CLI command that restarts only the named agent and explicitly defines soft/hard provider, session, lock, channel, process cleanup, and health semantics.

## Reproduction Assessment

Yes for the capability gap: current source and docs show no agent-id-scoped restart command, Gateway method, protocol schema, handler, or scope entry. The runaway-provider recovery scenario is a feature use case rather than a deterministic current-main bug reproduction.

## Solution Assessment

Unclear because there is no patch yet. The requested CLI/API direction is plausible and belongs in core Gateway lifecycle control, but the soft/hard semantics need maintainer design before implementation.
