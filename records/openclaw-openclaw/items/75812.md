## Summary

Keep open. Current main still lacks the requested per-agent restart surface, and the existing session reset/abort paths are scoped to a session key or run id rather than an agent-wide provider/runtime lifecycle. The linked restart and session-lock reports add useful recovery context, but they do not supersede this narrower core feature request.

## What This Changes

Request to add an `openclaw agents restart <agent-id>` or equivalent Gateway-backed command for restarting one agent's session/provider lifecycle without restarting the whole Gateway.

## Best Possible Solution

Add an admin-scoped Gateway lifecycle method and `openclaw agents restart <agent-id>` CLI wrapper for one configured agent. The soft path should reuse existing session abort/reset cleanup where it applies, while any hard provider/runtime/channel reinitialization should be designed as an explicit, documented contract with targeted Gateway/session/runtime tests and a changelog entry.

## Reproduction Assessment

Yes for the missing capability: source and docs inspection show no `agents restart` CLI command, no `agents.restart` Gateway method, and only session-key/run-id scoped reset/abort machinery. The production runaway-agent scenario is a feature use case rather than a deterministic failing test.

## Solution Assessment

Unclear because there is no proposed patch. The requested direction belongs in core/Gateway rather than ClawHub because it mutates live agent/session/provider lifecycle state, but maintainers should settle the soft/hard semantics and authorization model before implementation.
