## Summary

This is a focused fix for a source-reproducible memory-search regression, but the branch introduces a native-OpenAI URL classifier that is stricter than the existing shared hostname-based rule, so valid native URL spellings can receive a router-qualified model and fail. The linked issue provides credible real-environment context, while the PR’s formal proof remains test-only and should be replaced with redacted output from the patched head.

## What This Changes

The PR restores the `openai/` model prefix for non-native OpenAI embedding base URLs and normalizes qualified model names for token-limit lookup.

## Best Possible Solution

Use the same parsed, case-insensitive native OpenAI hostname semantic already established by the shared embedding client, normalize the configured model once, and preserve qualification only for the intended custom-router path with native and Requesty regression coverage plus redacted live endpoint/index-reuse proof.

## Reproduction Assessment

Yes. The linked issue gives an exact current-release configuration and failures, and current source shows the qualified model is unconditionally stripped before both request and index use; this review did not independently call Requesty.

## Solution Assessment

No, not exactly as written. Conditional preservation at the OpenAI plugin boundary is appropriate, but native endpoint detection should reuse the repository’s parsed hostname semantic rather than introduce stricter full-URL equality.
