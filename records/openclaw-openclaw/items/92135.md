## Summary

Keep this PR open: the code change is a plausible focused fix for the linked Requesty/OpenAI embedding prefix bug, but it still needs after-fix real behavior proof and maintainer awareness that the embedding model identity changes for non-native OpenAI base URLs.

## What This Changes

The PR changes the OpenAI memory embedding provider to preserve an `openai/` model prefix for non-native OpenAI-compatible base URLs while keeping native OpenAI requests bare, with focused unit coverage.

## Best Possible Solution

Land a provider-local fix after real non-native-router proof is added and maintainers explicitly accept or document the narrow memory-index identity transition.

## Reproduction Assessment

Yes at source level: current main strips `openai/`, then sends `client.model` in embedding requests and compares it in index identity. I did not establish a live Requesty reproduction with credentials.

## Solution Assessment

Yes with proof and upgrade caveats: the OpenAI plugin owns this provider-specific model normalization, and the base URL is only known after the remote client resolves. The documented `openai-compatible` provider is an alternative for new configs, but it does not by itself preserve this shipped `provider: "openai"` setup.
