## Label Justifications

- `P1`: The report describes a provider request blocker that can stop all agent turns for users with Feishu bitable tools enabled on strict Bedrock-backed Anthropic routes.
- `impact:auth-provider`: The failure is in provider-facing tool-schema payload compatibility, causing the configured model provider to reject requests before agent work can proceed.

## Summary

Keep open: current main and shipped v2026.6.8 still expose the reported Feishu bitable write schemas with empty nested `patternProperties` value schemas, and the Bedrock provider path forwards tool parameter schemas into `inputSchema.json` without a matching Anthropic/Bedrock normalizer.

## What This Changes

Report that Feishu bitable write tools emit flexible field/property maps using `Type.Record(Type.String(), Type.Any())`, producing empty nested JSON Schema sub-schemas that strict Bedrock validation rejects.

## Best Possible Solution

Repair the Feishu bitable write schemas with an explicit non-empty flexible value schema and regression coverage; do not use `Type.Unknown()` alone because TypeBox 1.1.39 serializes it to the same empty schema.

## Reproduction Assessment

Yes for the schema emission path: current and shipped Feishu code still emits `Type.Record(Type.String(), Type.Any())`, and TypeBox 1.1.39 serializes that record value to an empty sub-schema. A live Bedrock call was not run in this read-only review.

## Solution Assessment

Yes, the narrowest maintainable fix is to change the affected Feishu bitable parameter schemas and add provider-facing schema regression coverage. A broader provider sanitizer is a possible follow-up, but it should not be required to unblock this specific plugin bug.
