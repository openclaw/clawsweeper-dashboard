## Summary

Keep open: current main still has a source-reproducible path where a classified Google-style rate-limit signal can be overwritten by the timeout path before auth-profile cooldown marking, matching the reporter's repeated-primary-profile symptom.

## What This Changes

The issue reports Google provider 429/quota failures being recorded as timeouts, preventing auth-profile cooldown and causing repeated retries of the exhausted Google profile.

## Best Possible Solution

Preserve concrete auth-profile failure reasons such as `rate_limit` over generic timeout when both signals are present, while continuing to avoid cooldown marking for pure timeout attempts.

## Reproduction Assessment

Yes, source-level: call `handleAssistantFailover` with a rotate-profile decision, `timedOut: true`, and `assistantProfileFailureReason: "rate_limit"`; current main converts the persisted failure reason to `timeout` and skips cooldown marking. I did not run a live Google quota-exhaustion flow.

## Solution Assessment

Yes, the narrow maintainable fix is to adjust embedded assistant failover reason precedence so a concrete classified profile reason wins over a generic timeout for cooldown marking. A broad Google-specific workaround or regex-only change would miss the shared race path.
