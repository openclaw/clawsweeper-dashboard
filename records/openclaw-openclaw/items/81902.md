## Summary

Keep open: current main still overwrites a classified `rate_limit` profile failure with `timeout` in the rotate-profile path, and the linked open fix PR targets this source path but has not merged.

## What This Changes

The issue reports that Google 429 quota errors can be classified as timeouts, preventing auth-profile cooldown and causing repeated retries against the same Google profile.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/81927 or an equivalent narrow fix after real behavior proof shows the Google multi-profile cooldown path working on a real setup.

## Reproduction Assessment

Yes, source-level: call `handleAssistantFailover` with `timedOut` or `idleTimedOut` true and `assistantProfileFailureReason: "rate_limit"`; current main resolves the persisted failure reason to `timeout` and skips cooldown marking. I did not run a live Google quota-exhaustion flow.

## Solution Assessment

Yes for the code direction: preserving an already-classified actionable profile failure reason over the generic timeout is the narrowest maintainable fix, while pure timeouts should remain unmarked. A parser/header change can be a follow-up only if runtime proof shows the rate-limit signal never reaches this decision point.
