## Summary

Keep open: current main still has a source-reproducible path where a classified `rate_limit` signal is overwritten by `timeout`, and the linked open PR already carries the implementation path.

## What This Changes

The issue reports Google rate-limit errors being converted to timeout during failover, preventing auth-profile cooldown and retrying the same exhausted Google profile.

## Best Possible Solution

Land a narrow failover reason-precedence fix with regression coverage so an already-classified `rate_limit` wins over generic timeout, while keeping broader bucket-aware quota work in https://github.com/openclaw/openclaw/issues/28847.

## Reproduction Assessment

Yes, source-level: pass `timedOut: true` or `idleTimedOut: true` with `assistantProfileFailureReason: "rate_limit"` into `handleAssistantFailover`; current main sets `failureReason` to `timeout` and returns before cooldown marking. I did not perform a live Google quota exhaustion run.

## Solution Assessment

Yes, preserving the already-classified profile failure reason over generic timeout in `handleAssistantFailover` is the narrowest maintainable fix. Header-level capture can remain a follow-up only if live proof shows no classified error reaches this path.
