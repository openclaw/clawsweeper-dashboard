## Summary

Keep open: current main still drops an actionable `rate_limit` profile failure reason when timeout flags are also set, and an open linked fix PR already uses closing syntax for this issue.

## What This Changes

The issue reports Google 429 rate-limit responses being classified as timeouts, preventing auth-profile cooldown and causing repeated retries of the exhausted Google profile.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/81927, or an equivalent narrow reason-preservation fix with proof, then close this issue after the fix merges.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: call `handleAssistantFailover` with rotate-profile, `timedOut` or `idleTimedOut`, and `assistantProfileFailureReason: "rate_limit"`; current main resolves the persisted reason to `timeout` and skips cooldown marking.

## Solution Assessment

Yes. Preserving an already-classified actionable profile failure reason over the generic timeout at the failover decision boundary is the narrowest maintainable fix; response-header or regex work should be follow-up only if proof shows classification is missing before this point.
