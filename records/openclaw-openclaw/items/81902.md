## Summary

Keep open: current main still has a source-level path where an already-classified `rate_limit` auth-profile failure is replaced by `timeout`, so cooldown marking is skipped. The open closing PR is the right implementation path, but it has not merged and is still waiting on real behavior proof.

## What This Changes

Report that Google 429/quota exhaustion can be treated as a timeout in assistant failover, preventing auth-profile cooldown and causing repeated retries of the same Google profile.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/81927, or an equivalent narrow reason-preservation fix with real Google multi-profile proof, then close this issue; keep broader bucket-aware key-pool behavior in https://github.com/openclaw/openclaw/issues/28847.

## Reproduction Assessment

Yes, at source level: pass `timedOut` or `idleTimedOut` with `assistantProfileFailureReason: "rate_limit"` and a failed profile id into `handleAssistantFailover`; current main resolves the persisted reason to `timeout` and skips marking. I did not establish a live Google quota-exhaustion reproduction in this read-only review.

## Solution Assessment

Yes: preserving an already-classified actionable failure reason over the generic timeout is the narrowest maintainable fix for the confirmed path. Header or parser changes should be follow-up work only if real behavior proof shows the classifier never receives the Google rate-limit signal.
