## Summary

Keep open: current main still has the source-level path where a classified `rate_limit` profile failure reason is overwritten as `timeout`, which prevents auth-profile cooldown marking. An open linked fix PR already targets the issue and should remain the implementation path until it has real behavior proof and merges.

## What This Changes

The issue asks OpenClaw to preserve Google 429/rate-limit classification through the idle-timeout failover path so rate-limited Google auth profiles enter cooldown instead of being retried every request.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/81927 or an equivalent narrow reason-preservation fix with focused regression coverage and real Google multi-profile proof, then close this issue as implemented.

## Reproduction Assessment

Yes, source-level: pass `timedOut` or `idleTimedOut` with `assistantProfileFailureReason: "rate_limit"` into `handleAssistantFailover`; current main resolves the persisted profile failure reason to `timeout` and skips cooldown marking. I did not run a live Google quota-exhaustion flow.

## Solution Assessment

Yes, the narrowest maintainable fix is to preserve an already-classified actionable profile failure reason over generic timeout while leaving pure timeouts unmarked. Header or parser capture should be a follow-up only if real behavior proof shows the classifier never receives a rate-limit signal.
