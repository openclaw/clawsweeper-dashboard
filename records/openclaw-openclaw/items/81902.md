## Summary

Keep open: current main still has the source-level path where a classified `rate_limit` profile failure is overwritten as `timeout`, and the open linked fix PR should remain the implementation path until reviewed and merged.

## What This Changes

The issue reports Google 429 rate-limit failures being classified as timeouts, preventing auth-profile cooldown and causing repeated retries of the same provider.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/81927 or an equivalent narrow reason-preservation fix, with proof that Google multi-profile cooldown works; keep the broader key-pool/backoff product work separate at https://github.com/openclaw/openclaw/issues/28847.

## Reproduction Assessment

Yes, source-level: current main still resolves rotate-profile failures to `timeout` whenever timeout flags are set, even if the caller already supplied `assistantProfileFailureReason: "rate_limit"`. I did not run a live Google quota-exhaustion flow.

## Solution Assessment

Yes, preserving an already-classified actionable profile failure reason over the generic timeout is the narrowest maintainable fix; response-header parsing should be a follow-up only if real proof shows the classifier never receives the rate-limit signal.
