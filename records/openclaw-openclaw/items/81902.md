## Summary

Keep open: current `main` still has a source-reproducible path where a concrete `rate_limit` signal is replaced by `timeout` before auth-profile cooldown marking, matching the reporter's repeated-Google-profile symptom.

## What This Changes

The issue asks OpenClaw to preserve Google 429/rate-limit classification through embedded failover so exhausted Google auth profiles enter cooldown and later requests use the backup profile.

## Best Possible Solution

Preserve concrete auth-profile failure reasons such as `rate_limit` over generic timeout when both signals are present, while keeping pure timeout attempts out of auth-profile cooldown.

## Reproduction Assessment

Yes, source-level: pass `timedOut: true` with `assistantProfileFailureReason: "rate_limit"` into `handleAssistantFailover`, and the current code converts the persisted failure reason to `timeout` and skips cooldown marking. I did not run a live Google quota exhaustion flow.

## Solution Assessment

Yes, the narrow maintainable fix is in embedded assistant failover reason precedence, not broad Google-specific policy; current shared classification already recognizes `429` and `RESOURCE_EXHAUSTED`-style messages.
