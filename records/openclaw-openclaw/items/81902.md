## Summary

Keep open: current main still has the source-reproducible failover reason-precedence bug, and the open linked PR is the implementation candidate rather than a reason to close before merge.

## What This Changes

The issue asks OpenClaw to preserve Google 429 rate-limit classification so auth-profile cooldown/rotation happens instead of retrying the same profile after a timeout.

## Best Possible Solution

Land a narrow fix that preserves concrete auth-profile failure reasons such as `rate_limit` over generic timeout when both signals are present, while keeping pure timeouts out of auth-profile cooldown.

## Reproduction Assessment

Yes, source-level: pass `timedOut: true` or `idleTimedOut: true` with `assistantProfileFailureReason: "rate_limit"` into `handleAssistantFailover`; current main converts the persisted failure reason to `timeout` and skips cooldown marking. I did not run a live Google quota-exhaustion flow.

## Solution Assessment

Yes, preserving an already-classified actionable reason in `handleAssistantFailover` is the narrowest maintainable fix; https://github.com/openclaw/openclaw/pull/81927 appears to target that path, so this issue should stay open until that PR or an equivalent fix merges.
