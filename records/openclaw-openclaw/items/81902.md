## Summary

Keep open: current main still has a source-reproducible path where a classified rate-limit profile failure is overwritten by a generic timeout before cooldown marking, and an open linked PR is already the active implementation candidate.

## What This Changes

The issue reports Google provider 429 quota responses being converted into timeout failover reasons, preventing auth-profile cooldown and causing repeated retries of the same rate-limited profile.

## Best Possible Solution

Review and either land or replace https://github.com/openclaw/openclaw/pull/81927, keeping this issue open until the failover reason-precedence fix is merged.

## Reproduction Assessment

Yes, source-level: current main can pass `timedOut` or `idleTimedOut` together with `assistantProfileFailureReason: "rate_limit"`, and `handleAssistantFailover` converts the persisted profile failure reason to `timeout`. I did not run a live Google quota-exhaustion flow.

## Solution Assessment

Yes for the known source path: preserving an already-classified actionable reason is the narrow maintainable fix while keeping pure timeouts unmarked. Header-level 429 capture may be a follow-up if live proof shows response inspection is skipped before classification.
