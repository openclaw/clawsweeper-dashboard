## Summary

Keep open: current main still has the source-level failover path where a classified rate-limit signal is overwritten as a timeout before profile cooldown marking, and the linked open fix PR should resolve this only after it is reviewed and proved.

## What This Changes

The issue requests that Google 429 or quota-exhausted responses trigger auth-profile cooldown and provider/profile rotation instead of being repeatedly treated as timeouts.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/81927 or an equivalent narrow fix after real behavior proof shows the Google multi-profile cooldown path working on a real setup.

## Reproduction Assessment

Yes, source-level: call `handleAssistantFailover` with `timedOut` or `idleTimedOut` true and `assistantProfileFailureReason: "rate_limit"`; current main resolves the persisted failure reason to `timeout` and skips profile cooldown marking. I did not run a live Google quota-exhaustion flow.

## Solution Assessment

Yes, preserving an already-classified actionable profile failure reason over the generic timeout in the rotate-profile branch is the narrowest maintainable fix. Header or regex parsing can remain a follow-up only if real proof shows the classifier never receives a rate-limit signal.
