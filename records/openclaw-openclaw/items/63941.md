## Summary

Keep open. The underlying Firecrawl self-hosting bug is real on current main and the discussion added useful confirmation of the downstream SSRF wall, but this PR is not ready because it selects the trusted SSRF path from a static hostname/IP suffix predicate instead of DNS-backed private-target validation or explicit operator trust. The safer path is to choose one canonical Firecrawl self-hosting PR/policy across #59666, #74013, and this PR before merging.

## What This Changes

The PR relaxes Firecrawl baseUrl validation to allow custom HTTPS hosts and HTTP private/local hosts, routes private/local Firecrawl API calls through the trusted web-tools guard, and expands endpoint and SecretRef tests.

## Best Possible Solution

Land one canonical Firecrawl self-hosting fix that keeps public endpoints on strict HTTPS/default SSRF protection, allows cleartext HTTP only for DNS-validated private or loopback targets or an explicit operator opt-in, uses the trusted guard only for validated trusted endpoints, preserves SecretRef coverage, updates Firecrawl docs and changelog, and then closes #63877.

## Reproduction Assessment

Yes. Current main can be checked directly through `resolveEndpoint`: HTTP private Firecrawl base URLs throw the HTTPS error, and custom HTTPS self-hosted hosts throw the allowlist error; the PR's remaining failure follows from private DNS names that resolve to RFC1918 addresses but do not match its static suffix predicate.

## Solution Assessment

No. The direction fixes a valid user-facing bug, but selecting the trusted SSRF path from `isPrivateOrLocalEndpoint(params.url)` is not the narrowest maintainable solution; use DNS-backed private-network validation such as `assertHttpUrlTargetsPrivateNetwork` or an explicit `allowSelfHosted`/trusted-host policy instead.
