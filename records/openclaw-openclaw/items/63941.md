## Summary

Keep open. Current main still blocks the reported self-hosted Firecrawl path, but this PR should not merge as-is because the latest patch selects trusted versus strict SSRF handling from a static hostname predicate and overlaps multiple open Firecrawl self-hosting/SSRF PRs that need one canonical maintainer-approved shape.

## What This Changes

The PR relaxes Firecrawl baseUrl validation to allow custom HTTPS hosts and HTTP private/local hosts, routes private/local Firecrawl API calls through the trusted web-tools guard, and expands Firecrawl endpoint and SecretRef tests.

## Best Possible Solution

Land one canonical Firecrawl self-hosting fix that keeps public endpoints on HTTPS and strict SSRF protection, allows HTTP only after DNS-backed private/loopback validation or an explicit operator opt-in, uses the trusted guard only for validated trusted endpoints, preserves SecretRef coverage, updates Firecrawl/web_fetch docs and changelog, and then closes #63877.

## Reproduction Assessment

Yes. Current main can be checked directly through `resolveEndpoint`: `http://host.openshell.internal:3002/v1` throws `Firecrawl baseUrl must use https`, and custom HTTPS hosts throw the allowlist error; the PR's remaining private-DNS gap follows from its static predicate plus the strict guard's resolved-address blocking.

## Solution Assessment

No. The direction fixes a real user problem, but selecting the trusted SSRF path from a hostname suffix/literal check is not the narrowest maintainable solution; a DNS-backed private-network classifier or explicit `allowSelfHosted`/trusted-host policy is safer.
