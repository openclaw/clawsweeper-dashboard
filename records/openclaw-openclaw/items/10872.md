## Summary

Keep open: current main still forwards iMessage reply context only when resolved `reply_to_*` fields are already present, while GUID/chat.db inline-reply metadata is not resolved before dispatch; open PR #42683 remains the active implementation path. ([github.com](https://github.com/openclaw/openclaw/pull/42683))

## What This Changes

The issue requests that inbound iMessage inline replies include the original replied-to message context in the agent-visible payload.

## Best Possible Solution

Land or replace #42683 so the iMessage plugin resolves reliable inline-reply origin metadata into `ReplyTo*` fields before dispatch, using the existing `dbPath` surface and focused regression coverage.

## Reproduction Assessment

Yes, at source level. An imsg notification with only GUID/chat.db reply metadata and no resolved `reply_to_text` is parsed and enqueued without enrichment, then `describeReplyContext` returns null because no reply body exists.

## Solution Assessment

Yes for the direction, with implementation details still needing review. The narrow maintainable fix belongs in the iMessage plugin before dispatch, but #42683’s lookup assumptions should be reviewed rather than closing this issue or queuing a duplicate repair.
