## Summary

Keep open. Current main still only forwards iMessage reply context when resolved `reply_to_*` fields are already present; GUID/chat.db inline reply metadata is not resolved before dispatch, and open PR #42683 remains the active implementation path.

## What This Changes

Request that incoming iMessage inline replies include the original replied-to message context for the agent.

## Best Possible Solution

Land or replace #42683 so the bundled iMessage plugin resolves reliable inline-reply origin metadata into `ReplyToId`, `ReplyToBody`, and `ReplyToSender` before dispatch, using the existing `dbPath` surface and focused regression coverage.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is an imsg RPC notification with GUID/chat.db reply metadata and no resolved `reply_to_text`; current main parses and enqueues it without enrichment, then `describeReplyContext` returns null because no reply body exists.

## Solution Assessment

Yes for triage. Keeping this issue open for review, porting, or replacement of #42683 is the narrowest maintainable path because a separate automated repair would duplicate active work and skip the remaining lookup assumptions that need maintainer judgment.
