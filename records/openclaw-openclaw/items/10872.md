## Summary

Keep open: current main still only forwards iMessage reply context when resolved `reply_to_*` fields already arrive, and the open implementation path still needs maintainer review or replacement before the user-visible gap is fixed.

## What This Changes

Request to include iMessage inline reply/thread context in inbound messages so agents can see the message being replied to.

## Best Possible Solution

Land or replace the active iMessage plugin fix so reliable inline-reply origin metadata is resolved into `ReplyToId`, `ReplyToBody`, and `ReplyToSender` under `extensions/imessage/src/monitor/*` with focused regression coverage.

## Reproduction Assessment

Yes, at source level. A notification with only GUID/`chat.db` reply metadata and no resolved `reply_to_text` is parsed and enqueued without enrichment, and `describeReplyContext` returns null because no reply body exists.

## Solution Assessment

Yes for the fix direction, but not as a cleanup close. The narrow maintainable path is an iMessage-plugin monitor change using the reliable origin metadata and existing `dbPath` surface, while the current open PR needs maintainer review or porting.
