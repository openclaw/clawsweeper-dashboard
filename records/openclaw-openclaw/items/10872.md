## Summary

Keep open: current main still only forwards iMessage reply context when resolved `reply_to_*` fields are already present, while GUID/chat.db inline-reply metadata is not resolved before dispatch.

## What This Changes

Request that the iMessage plugin include inline reply/thread context in incoming messages by resolving the replied-to message into `ReplyToId`, `ReplyToBody`, and `ReplyToSender`.

## Best Possible Solution

Land or replace #42683 so the iMessage plugin resolves reliable inline-reply origin metadata into `ReplyTo*` fields before dispatch, using the existing `dbPath` surface and focused regression coverage.

## Reproduction Assessment

Yes, at source level. An imsg RPC notification with reply metadata available only as GUID/chat.db state and no resolved `reply_to_text` is parsed and enqueued without enrichment, then `describeReplyContext` returns null because no reply body exists.

## Solution Assessment

Yes for triage, with implementation details still needing review. The narrow maintainable direction is iMessage-local enrichment using existing `dbPath`, but the active PR’s lookup assumptions should be reviewed rather than closing this issue or queuing a duplicate fix.
