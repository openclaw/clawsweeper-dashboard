## Summary

Keep open: current main still only forwards iMessage reply context when resolved `reply_to_*` fields are already present, while GUID-only or `chat.db` reply metadata is not preserved or enriched before dispatch. The same remaining gap is already covered by the open implementation path at https://github.com/openclaw/openclaw/pull/42683, so this should not be closed or split into a separate repair lane.

## What This Changes

Request to add inbound iMessage inline-reply/thread context so agents receive the replied-to message text, id, and sender.

## Best Possible Solution

Land or replace the active iMessage plugin fix so reliable inline-reply origin metadata is resolved into `ReplyToId`, `ReplyToBody`, and `ReplyToSender` under `extensions/imessage/src/monitor/*` with focused regression coverage.

## Reproduction Assessment

Yes, at source level. A notification with GUID or `chat.db` reply metadata but no resolved `reply_to_text` is parsed and enqueued without enrichment, and `describeReplyContext` returns null because no reply body exists.

## Solution Assessment

Yes for the fix direction, but not as a cleanup close. The narrow maintainable path is an active iMessage plugin change using reliable origin metadata and the existing `dbPath` surface, while https://github.com/openclaw/openclaw/pull/42683 needs review, porting, or replacement.
