## Summary

Keep open: current main still drops iMessage inline-reply context unless the payload already contains resolved `reply_to_*` fields, and the active path is maintainer review or replacement of the existing iMessage PR rather than cleanup closure.

## What This Changes

This issue requests that incoming iMessage inline replies include the quoted message id, body, and sender in the agent-visible context.

## Best Possible Solution

Land or replace the active iMessage plugin fix so reliable inline-reply origin metadata is resolved into `ReplyToId`, `ReplyToBody`, and `ReplyToSender` before dispatch, with focused tests for GUID-only replies and failure modes.

## Reproduction Assessment

Yes, at source level. A notification with only GUID or `chat.db` reply metadata and no resolved `reply_to_text` is parsed and enqueued without enrichment, and `describeReplyContext` then returns null because no reply body exists.

## Solution Assessment

Yes for the direction, but not as a separate duplicate repair while the active PR exists. The narrow maintainable fix belongs in `extensions/imessage/src/monitor/*`, using the existing `dbPath` surface and the reliable origin GUID path.
