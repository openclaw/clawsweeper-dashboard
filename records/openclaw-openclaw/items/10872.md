## Summary

Keep open: current main and the latest release still only forward already-resolved iMessage `reply_to_*` fields and do not resolve inline-reply GUID metadata from `chat.db`. The previously linked implementation was closed unmerged because it targeted a removed path, so a focused replacement against the active plugin is still useful.

## What This Changes

Request inbound iMessage inline replies to include resolved quote/thread context so agents can see which prior message the user referenced.

## Best Possible Solution

Implement active iMessage plugin enrichment that maps reliable origin metadata through the existing `dbPath` surface into `ReplyToId`, `ReplyToBody`, and `ReplyToSender` before dispatch, without adding core hooks or new config keys.

## Reproduction Assessment

Yes, at source level. Current main accepts and forwards only resolved `reply_to_*` fields, `describeReplyContext` returns null without `reply_to_text`, and I did not run a live macOS iMessage flow in this read-only review.

## Solution Assessment

Yes for the implementation direction, but not by reopening the stale branch as-is. The maintainable fix is a focused replacement in `extensions/imessage/src/monitor/*` that resolves reliable origin metadata before enqueue or dispatch and proves the live imsg/chat.db path.
