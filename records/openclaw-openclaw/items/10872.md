## Summary

Keep open. Current main still only forwards iMessage reply context when resolved `reply_to_*` fields are already present, while GUID-only or chat.db-only inline reply metadata is not resolved before dispatch; open PR #42683 remains the active implementation path.

## What This Changes

Request that the bundled iMessage plugin include inbound inline reply/thread context by resolving imsg/chat.db reply metadata into the existing `ReplyTo*` fields.

## Best Possible Solution

Land or replace the iMessage-focused fix so the bundled iMessage plugin resolves reliable inline-reply origin metadata into `ReplyToId`, `ReplyToBody`, and `ReplyToSender` before dispatch, using the existing `dbPath` surface and focused regression coverage.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is an imsg RPC notification with only GUID/chat.db reply metadata and no resolved `reply_to_text`; current main parses/enqueues it without enrichment, and `describeReplyContext` returns null because no reply body exists.

## Solution Assessment

Yes for triage. Keeping this open for review, porting, or superseding of #42683 is the narrowest maintainable path because a separate automated repair would duplicate active work and skip the remaining lookup assumptions that need maintainer judgment.
