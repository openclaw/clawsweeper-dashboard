## Summary

Keep open: current main still only forwards iMessage reply context when resolved `reply_to_*` fields are already present, and the related open PR is the active path for review, porting, or replacement rather than cleanup closure.

## What This Changes

The issue requests inbound iMessage inline-reply context to be resolved into agent-visible reply metadata before dispatch.

## Best Possible Solution

Land or replace the active iMessage plugin fix so reliable inline-reply origin metadata is resolved into `ReplyToId`, `ReplyToBody`, and `ReplyToSender` in `extensions/imessage/src/monitor/*`, with focused tests for GUID-only replies and fallback failures.

## Reproduction Assessment

Yes, at source level. A notification with only GUID or `chat.db` reply metadata and no resolved `reply_to_text` is parsed and enqueued without enrichment, and `describeReplyContext` returns null because no reply body exists.

## Solution Assessment

Yes for the fix direction, but not as a separate duplicate repair while the active related PR exists. The narrow maintainable path is an iMessage-plugin monitor change using the existing `dbPath` surface and reliable origin GUID data.
