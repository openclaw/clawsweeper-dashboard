## Summary

Keep open: current main still only converts already-resolved iMessage `reply_to_*` payload fields into agent-visible reply context, while the upstream imsg contract now exposes `thread_originator_guid` rather than resolved quoted text/sender. The related open PR remains the active implementation path, but it still needs maintainer review/porting before this issue can close.

## What This Changes

Request that incoming iMessage inline replies include the replied-to message id, body, and sender in agent context.

## Best Possible Solution

Land or replace the active iMessage plugin fix so reliable inline-reply origin metadata is resolved into `ReplyToId`, `ReplyToBody`, and `ReplyToSender` in `extensions/imessage/src/monitor/*`, using the existing `dbPath` surface and focused regression tests.

## Reproduction Assessment

Yes, at source level. A notification carrying only `thread_originator_guid`/`chat.db` reply metadata and no resolved `reply_to_text` is parsed and enqueued without enrichment, and `describeReplyContext` returns null because no reply body exists.

## Solution Assessment

Yes for the fix direction, but not as a duplicate cleanup close. The narrow maintainable path is an iMessage-plugin monitor change using upstream `thread_originator_guid` plus the existing `dbPath` surface, not a broad core channel hook.
