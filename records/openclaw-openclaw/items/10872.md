## Summary

Keep open. Current main still forwards iMessage reply context only when resolved `reply_to_*` fields are already present; it does not resolve GUID-only inline reply metadata from imsg/chat.db, and open PR #42683 remains the active implementation path with maintainer-review context.

## What This Changes

Request to include iMessage inline reply/thread context in incoming messages so agents can see the message being replied to.

## Best Possible Solution

Land or replace the iMessage-focused fix so the bundled iMessage plugin resolves reliable inline-reply origin metadata into `ReplyToId`, `ReplyToBody`, and `ReplyToSender` before dispatch.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is an imsg RPC notification with GUID-only reply metadata and no resolved `reply_to_text`; current main parses/enqueues it without enrichment, and `describeReplyContext` returns null because no reply body exists.

## Solution Assessment

Yes for triage. Keeping this open for review, porting, or superseding of #42683 is the narrowest maintainable path; a separate automated repair PR would duplicate the active implementation and skip the remaining lookup assumptions that need maintainer judgment.
