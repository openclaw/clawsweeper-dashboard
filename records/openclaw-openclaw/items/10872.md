## Summary

Keep open: current main still only forwards iMessage reply context when the incoming payload already has resolved `reply_to_*` fields, and the active monitor still does not resolve GUID-only inline reply metadata from `chat.db`. The prior implementation PR was closed unmerged because it targeted the removed `src/imessage` path, so a focused replacement against the active plugin remains useful.

## What This Changes

Request iMessage inbound inline replies to include the replied-to message id, body, and sender in agent-visible incoming context.

## Best Possible Solution

Implement active iMessage plugin enrichment that maps reliable inline-reply origin metadata through the existing `dbPath` surface into `ReplyToId`, `ReplyToBody`, and `ReplyToSender` before dispatch, without adding core hooks or new config keys.

## Reproduction Assessment

Yes, at source level. Current main accepts and forwards only resolved `reply_to_*` fields, `describeReplyContext` returns null without `reply_to_text`, and the active monitor enqueues parsed notifications without GUID enrichment; I did not run a live macOS iMessage flow.

## Solution Assessment

Yes for the direction, but not by reopening the stale branch as-is. The maintainable fix is a focused replacement in `extensions/imessage/src/monitor/*` that follows imsg’s `thread_originator_guid` contract, uses the existing `dbPath` surface, fails closed, and includes real macOS proof.
