## Summary

Keep this issue open. Current main still forwards iMessage reply context only when resolved `reply_to_*` fields are already present; it does not model or resolve GUID-only inline reply metadata from imsg/chat.db, and open PR #42683 remains the active implementation path.

## What This Changes

Add iMessage inbound reply/thread context so inline replies include the replied-to message id, body, and sender for the agent, resolving GUID-only metadata from imsg/chat.db when needed.

## Best Possible Solution

Land or replace the existing iMessage-focused fix so the bundled iMessage plugin resolves reliable inline-reply origin metadata into `ReplyToId`, `ReplyToBody`, and `ReplyToSender` before dispatch, using the existing `dbPath` surface for any read-only fallback and preserving the existing context-visibility filtering.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is an imsg RPC notification with GUID-only reply metadata and no resolved `reply_to_text`; current main parses/enqueues it without enrichment, and `describeReplyContext` returns null because no reply body exists.

## Solution Assessment

Yes for triage. Keeping this open for the existing PR path is the narrowest maintainable handling now; a separate automated repair PR would duplicate #42683 while the lookup-key and chat.db assumptions still need maintainer review.
