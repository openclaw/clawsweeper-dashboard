## Summary

Keep open: current main still ignores GUID-only iMessage inline-reply metadata and has no `chat.db` enrichment before dispatch. The related implementation path at https://github.com/openclaw/openclaw/pull/42683 is open but conflicting and targets a removed path, so this needs maintainer review or replacement rather than cleanup closure.

## What This Changes

Request iMessage incoming messages include inline reply/thread context by resolving the replied-to message into `ReplyToId`, `ReplyToBody`, and `ReplyToSender`.

## Best Possible Solution

Land or replace the active iMessage plugin fix under `extensions/imessage/src/monitor/*` so reliable inline-reply origin metadata is resolved through the existing `dbPath` surface into `ReplyToId`, `ReplyToBody`, and `ReplyToSender`, with focused failure-mode coverage.

## Reproduction Assessment

Yes, at source level. A payload with GUID-only reply metadata and no resolved `reply_to_text` reaches the queue without enrichment, and `describeReplyContext` returns null because no reply body exists; I did not run a live macOS iMessage flow in this read-only review.

## Solution Assessment

Yes for the implementation direction, but not as a cleanup close. The narrow maintainable path is to review, port, or replace the existing iMessage-focused fix so the active plugin resolves reliable origin metadata before dispatch.
