## Summary

Keep open: current main still only forwards iMessage reply context when resolved `reply_to_*` fields are already present, and the active implementation path still needs maintainer review, porting, or replacement before the user-visible gap is fixed.

## What This Changes

The issue requests inbound iMessage inline replies to include the original replied-to message context before dispatching to the agent.

## Best Possible Solution

Land or replace the active iMessage plugin fix so reliable inline-reply origin metadata is resolved into `ReplyToId`, `ReplyToBody`, and `ReplyToSender` under `extensions/imessage/src/monitor/*` with focused regression coverage.

## Reproduction Assessment

Yes, at source level. A notification with only GUID or `chat.db` reply metadata and no resolved `reply_to_text` is parsed and enqueued without enrichment, and `describeReplyContext` returns null because no reply body exists.

## Solution Assessment

Yes for the fix direction, but not as a cleanup close. The narrow maintainable path is an iMessage-plugin monitor change using reliable origin metadata and the existing `dbPath` surface, while the current open PR needs maintainer review, porting, or replacement.
