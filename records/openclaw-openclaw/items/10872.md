## Summary

Keep open: current main still only forwards iMessage reply context when resolved `reply_to_*` fields are already present, and the active implementation path is the open iMessage PR at https://github.com/openclaw/openclaw/pull/42683 rather than a duplicate cleanup close or new repair lane.

## What This Changes

Request that incoming iMessage inline replies include the original replied-to message id, body, and sender in agent context.

## Best Possible Solution

Land or replace the active iMessage plugin fix so reliable inline-reply origin metadata is resolved into `ReplyToId`, `ReplyToBody`, and `ReplyToSender` before dispatch, using the existing `dbPath` surface with focused failure-mode coverage.

## Reproduction Assessment

Yes, at source level. A notification with only GUID/chat.db reply metadata and no resolved `reply_to_text` is parsed and dispatched without enrichment, then `describeReplyContext` returns null because no reply body exists.

## Solution Assessment

Yes for the direction, but the exact implementation still needs review. The narrow maintainable fix belongs in `extensions/imessage/src/monitor/*` before dispatch, with care around the reliable GUID field and remote/local `dbPath` assumptions.
