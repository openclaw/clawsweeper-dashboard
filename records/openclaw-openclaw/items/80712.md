## Summary

Keep open. Current main still rejects `#` in exec SecretRef ids before the exec provider can run, and an open linked fix PR explicitly targets this issue, so the issue should remain open until that PR lands or is closed.

## What This Changes

The issue asks OpenClaw to align exec SecretRef id validation with AWS Secrets Manager resolver ids that use `secret#json_key`.

## Best Possible Solution

Review and land or close https://github.com/openclaw/openclaw/pull/80731; if it lands, this issue can close as implemented by that merged fix.

## Reproduction Assessment

Yes, at source level. A hash-containing active exec SecretRef reaches `resolveSecretRefValues`, fails `isValidExecSecretRefId`, and throws before the configured exec provider can receive the id.

## Solution Assessment

Yes. The linked PR’s direction is the narrow maintainable fix: update the shared exec SecretRef contract and shared vectors/docs rather than adding a one-off gateway workaround.
