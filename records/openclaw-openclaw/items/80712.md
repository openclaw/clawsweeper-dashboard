## Summary

Keep open. Current main and the latest release still reject `#` in exec SecretRef ids before the exec provider can run, and the open linked fix PR should be reviewed or landed before this issue closes.

## What This Changes

The issue asks OpenClaw to align exec SecretRef id validation with the documented AWS-style `secret#json_key` resolver convention.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/80731, or close that PR with an explicit alternate fix, then let this issue close only after the shared contract is fixed on main.

## Reproduction Assessment

Yes, at source level. A hash-containing exec SecretRef id fails the shared regex in `isValidExecSecretRefId`, and `resolveSecretRefValues` throws before the exec provider receives the id.

## Solution Assessment

Yes. The linked PR’s direction of updating the shared exec id contract plus docs and shared test vectors is the narrow maintainable fix, but this issue should remain open until that fix merges.
