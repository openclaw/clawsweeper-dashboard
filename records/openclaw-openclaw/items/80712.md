## Summary

Keep open. Current main and the latest release still reject hash-containing exec SecretRef ids before the exec resolver can run, and an open linked fix PR with closing syntax should be reviewed or landed before this issue is closed.

## What This Changes

The issue requests aligning exec SecretRef id validation with the documented AWS-style `secret#json_key` resolver convention so gateway startup does not fail on hash-containing ids.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/80731 or an equivalent shared SecretRef contract fix, then close this issue from the merged implementation.

## Reproduction Assessment

Yes, at source level. A hash-containing exec SecretRef id fails the shared validator in `resolveSecretRefValues` before the configured exec provider can parse it.

## Solution Assessment

Yes. The narrow maintainable fix is to update the shared exec SecretRef id contract, parity vectors, and docs together; the open linked PR follows that direction.
