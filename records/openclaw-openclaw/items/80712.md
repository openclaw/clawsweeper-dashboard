## Summary

Keep open. Current main still rejects hash-containing exec SecretRef ids before the exec provider runs, and the linked closing PR is open and unmerged, so this issue should close only after that PR or an equivalent main fix lands.

## What This Changes

Allow documented exec SecretRef ids such as `secret#json_key` to pass shared validation, gateway schema validation, and SecretRef tooling.

## Best Possible Solution

Land the existing linked SecretRef contract PR, preserving traversal rejection and synced docs/tests, then close this issue from the merged fix.

## Reproduction Assessment

Yes, at source level. A hash-containing exec SecretRef id fails `isValidExecSecretRefId` and is rejected in `resolveSecretRefValues` before the exec resolver can receive it.

## Solution Assessment

Yes. Updating the shared SecretRef id contract and shared test vector, as the linked PR does, is the narrow maintainable fix; keeping this issue open until that PR lands is the correct tracking state.
