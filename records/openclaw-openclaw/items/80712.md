## Summary

Keep open. Current `main` and the latest release still reject `#` in exec SecretRef ids before the exec provider can run, and the issue is now paired with an open fixing PR that uses closing syntax, so the issue should close only after that PR or an equivalent fix lands.

## What This Changes

The issue requests aligning exec SecretRef id validation with the AWS-style `secret#json_key` resolver convention so gateway startup no longer fails before the resolver runs.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/80731, or an equivalent shared SecretRef contract/docs/test-vector fix, then close this issue from the merged commit.

## Reproduction Assessment

Yes, at source level. A hash-containing exec SecretRef id fails `isValidExecSecretRefId`, and `resolveSecretRefValues` throws before the exec provider path can parse the id; gateway startup then wraps that as required secrets unavailable.

## Solution Assessment

Yes. The linked PR's shared regex, docs, and vector-backed parity approach is the narrowest maintainable fix; this issue should remain open until that PR or an equivalent patch merges.
