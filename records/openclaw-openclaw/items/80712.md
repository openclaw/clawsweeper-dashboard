## Summary

Keep open. Current main and the shipped v2026.5.7 tag still reject `#` in exec SecretRef ids before the exec resolver can run, but the reported AWS resolver template contract is not present in this checkout, so maintainers need to align the public contract before cleanup closure.

## What This Changes

The issue asks OpenClaw to reconcile exec SecretRef id validation with an AWS Secrets Manager resolver convention that uses `secret-id#json-key`.

## Best Possible Solution

Align the exec SecretRef id contract end to end: either deliberately allow `#` everywhere with regression coverage and docs, or update the AWS resolver/template path so it emits ids that current validators accept.

## Reproduction Assessment

Yes for the core failure path by source inspection: current regex validation rejects the reported `secret#key` id, and runtime resolution throws before exec provider dispatch. I did not run a full gateway startup because the checkout is read-only and local package tooling is unavailable.

## Solution Assessment

Unclear until maintainers choose the public contract. Allowing `#` is likely the lowest-impact code change, but updating the resolver/template contract may be safer if the restricted id grammar is intentional.
