## Summary

Keep open. Current main and v2026.5.12 still reject `#` in exec SecretRef ids before the exec provider runs, and the open fix path is https://github.com/openclaw/openclaw/pull/80731; this issue should close only after that PR or an equivalent fix lands.

## What This Changes

The issue asks OpenClaw to align exec SecretRef id validation with the documented AWS Secrets Manager `secret#json_key` resolver convention so gateway startup does not fail before the resolver runs.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/80731 or an equivalent shared-contract fix, then close this issue from the merged commit or release.

## Reproduction Assessment

Yes, at source level. An active exec SecretRef id containing `#` fails `isValidExecSecretRefId`, and `resolveSecretRefValues` throws before refs are grouped or the configured exec provider can receive the id.

## Solution Assessment

Yes. The narrow maintainable fix is to update the shared exec SecretRef id contract plus the schema/docs/vector-backed parity tests; the linked open PR is already attempting that path.
