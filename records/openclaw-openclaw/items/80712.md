## Summary

Keep open. Current main still rejects `#` in exec SecretRef ids before invoking the exec provider, and an open closing PR already carries the narrow fix, so this issue should wait for that PR or an equivalent merged change.

## What This Changes

Exec SecretRef ids containing `#` should not fail gateway startup before the exec resolver runs, because the documented AWS-style resolver convention uses `secret#json_key` ids.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/80731, or an equivalent shared SecretRef contract fix, then close this issue from the merged change.

## Reproduction Assessment

Yes, at source level. Current main rejects a hash-containing exec SecretRef id in `isValidExecSecretRefId` inside `resolveSecretRefValues` before the configured exec provider can receive it.

## Solution Assessment

Yes. Updating the shared regex/schema/docs/test vector is the narrowest maintainable fix; the issue should remain open until the linked PR or an equivalent fix merges.
