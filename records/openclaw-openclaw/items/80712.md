## Summary

Keep open. Current main still rejects `#` in exec SecretRef ids before the exec resolver runs, which matches the reported startup failure; I found no merged fix or narrower duplicate that already owns this exact contract mismatch.

## What This Changes

The issue asks OpenClaw to align exec SecretRef id validation with an AWS Secrets Manager resolver convention that uses `secret-id#json-key`.

## Best Possible Solution

Align the exec SecretRef id contract in the shared validator, gateway schema, plugin SDK/config surfaces, tests, and docs; if the AWS resolver `#` separator is canonical, allow `#` narrowly while preserving traversal rejection.

## Reproduction Assessment

Yes, at source level. An active exec SecretRef id containing `#` reaches `resolveSecretRefValues`, fails `isValidExecSecretRefId`, and throws before the configured exec provider can receive the id.

## Solution Assessment

Yes with one caveat. The narrow maintainable fix is to make the single shared exec id contract accept the intended separator everywhere, but the repair should first confirm the AWS resolver/template source so docs and generated examples move in the same direction.
