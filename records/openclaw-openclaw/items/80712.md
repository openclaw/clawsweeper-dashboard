## Summary

Keep open. Current main and the latest shipped tag still reject `#` in exec SecretRef ids before the exec provider runs, and the issue is already paired with an open closing PR that should be reviewed and landed rather than closing the issue early.

## What This Changes

The issue reports that exec SecretRef ids using an AWS-style `secret#json_key` selector are rejected before the resolver runs.

## Best Possible Solution

Review and land the linked shared SecretRef contract fix, keeping the validator, gateway schema, plugin SDK/config schemas, tests, and public docs in sync.

## Reproduction Assessment

Yes, at source level. A hash-containing exec SecretRef id fails the shared validator in `src/secrets/ref-contract.ts` and `resolveSecretRefValues` throws before the configured exec provider can receive the id.

## Solution Assessment

Yes. The narrow maintainable solution is the open PR's shared contract update plus parity coverage and docs sync, after normal maintainer review and checks.
