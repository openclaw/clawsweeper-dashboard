## Summary

Keep open. Current main and v2026.5.7 both reject `#` in exec SecretRef ids before provider execution, so the reported startup failure is plausible; the claimed AWS resolver template was not found in the checked-out source, so the fix needs maintainer/security choice on whether `#` becomes part of the exec id contract or the resolver/docs change.

## What This Changes

The issue asks OpenClaw to align exec SecretRef id validation with an AWS Secrets Manager resolver convention that uses `secret-id#json-key`.

## Best Possible Solution

Align the official exec SecretRef id contract across runtime validation, gateway protocol schema, configure/apply flows, tests, docs, and any shipped AWS resolver template; if `#` is intended, allow it deliberately, otherwise update the resolver/template and migration guidance to emit valid ids.

## Reproduction Assessment

Yes for the validation failure: the current exec id regex and JSON-schema pattern reject `gravity/shared/slack-gateway/credentials#bot_token`, and `resolveSecretRefValues` enforces that before provider execution. A full gateway launch was not needed to prove this specific failing gate.

## Solution Assessment

Unclear as an implementation choice. Allowing `#` is plausibly the smallest fix if the AWS resolver convention is official, but current repo docs forbid it and the resolver template was not found in source, so maintainers should choose the public contract first.
