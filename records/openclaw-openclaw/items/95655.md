## Summary

Keep open: this is a focused bug-fix PR for the canonical Mattermost private-channel session split, but the revised patch is still not merge-ready because it can set `chatType: "group"` while leaving the route peer and session key under `channel`, adds an unscoped process-global channel-kind cache, and lacks real Mattermost proof.

## What This Changes

The PR adds a Mattermost `inferTargetChatType` hook, moves shared target inference to consult plugin hooks before the generic `channel:` fallback, and uses a monitor-populated Mattermost channel-kind cache during session-route resolution.

## Best Possible Solution

Preserve authoritative Mattermost private-channel type through peer kind, chat type, and session key using a lifecycle-scoped/account-scoped source, then add focused cached-private, cached-public, cold-cache tests plus redacted live Mattermost proof.

## Reproduction Assessment

Yes for source-level reproduction: current main maps Mattermost private-channel type `P` to `group`, while outbound target/session routing can still derive or store the same `channel:<id>` destination as `channel`. I did not run a live Mattermost workspace.

## Solution Assessment

No as submitted. The plugin boundary is the right layer, but the route must align `peer.kind`, `chatType`, and session key from authoritative channel type instead of changing only `chatType` through an unscoped global cache.
