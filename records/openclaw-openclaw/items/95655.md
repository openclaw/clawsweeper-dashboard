## Summary

Keep open: the updated branch fixes the prior peer/session-key shape, but the final head now writes Mattermost channel-kind cache entries with account scope while the new hook reads them without account scope, so the core target-inference path can still fall back to `channel` for private channels.

## What This Changes

The PR adds a Mattermost target chat-type hook, reorders one outbound target inference helper, and adds a TTL/account-scoped Mattermost channel-kind cache used by session routing.

## Best Possible Solution

Repair this branch or a sibling so authoritative Mattermost channel type is available to every target/session inference reader with matching account scope, then add focused cached-private, cached-public, cold-cache tests and redacted real Mattermost proof.

## Reproduction Assessment

Yes for source-level reproduction: current main and v2026.6.9 still classify Mattermost private-channel delivery/session paths from `channel:<id>`, and the PR head's new hook misses account-scoped cache entries. I did not run a live Mattermost workspace.

## Solution Assessment

No. The plugin boundary is the right layer, but the implementation must keep cache writer/reader scope symmetric or carry account context through the resolver before it can be the best fix.
