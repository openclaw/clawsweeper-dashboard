## Summary

Keep open: current `main` still maps successful no-summary/NONE Active Memory runs and zero-hit terminal memory-search paths to user-visible `status=empty`, and tests still assert that behavior.

## What This Changes

The issue asks Active Memory to stop surfacing normal no-useful-memory recall outcomes as `status=empty` and to reserve empty/failure-like statuses for genuinely unavailable or failed recall paths.

## Best Possible Solution

Keep the hidden no-injection behavior for NONE, but classify successful no-relevant-memory runs separately from unavailable/error/timeout paths, avoid caching empty no-summary results, and update active-memory docs/tests/changelog accordingly.

## Reproduction Assessment

Yes, from source inspection: make the embedded recall subagent return `NONE` or write a zero-hit `memory_search` transcript and inspect the persisted Active Memory status line. Current tests already assert `status=empty` for those paths, although I did not run tests in this read-only review.

## Solution Assessment

No, the current implementation is not the best way to solve the issue because it uses `empty` for normal no-summary recall, zero-hit search, and unavailable/error-like paths. The narrow fix belongs in `extensions/active-memory/index.ts` status classification/cache eligibility with adjacent regression tests.
