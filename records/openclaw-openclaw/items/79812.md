## Summary

Current main still has the reported Active Memory status conflation: normal `NONE`/no-summary recall results, terminal zero-hit searches, and some unavailable paths all surface as `status=empty`. The issue is narrow, source-backed, and still needs a plugin fix rather than cleanup closure.

## What This Changes

The issue asks Active Memory to stop showing `status=empty` for successful no-useful-memory recall outcomes and to separate that from zero-hit, unavailable, timeout, and successful-recall states.

## Best Possible Solution

Keep the hidden no-injection behavior unchanged, but emit a distinct successful no-relevant-memory status for `NONE`, reserve unavailable/timeout/failure statuses for real failures, avoid sticky caching of empty no-context outcomes, and update the active-memory tests and docs.

## Reproduction Assessment

Yes, from source inspection: make the embedded recall subagent return `NONE` and inspect the persisted Active Memory debug line, or follow the existing zero-hit terminal-search unit path. Current tests already assert `status=empty` for those paths, so the behavior is high-confidence even without running tests here.

## Solution Assessment

No, the current implementation is not the best solution because it uses `empty` for normal no-summary recall, terminal zero-hit search, missing tools, and unavailable/error-like paths. The narrow maintainable fix belongs in `extensions/active-memory/index.ts` status classification, cache eligibility, and adjacent regression tests.
