## Summary

Keep open: current main still conflates normal no-useful-memory Active Memory results with `status=empty`, and there are open PRs with closing references that should be reviewed before this issue is closed.

## What This Changes

Request Active Memory to use a distinct successful no-relevant-memory status instead of surfacing `status=empty` for ordinary no-useful-memory recall.

## Best Possible Solution

Land one canonical Active Memory fix that preserves hidden no-injection behavior while assigning no-useful-memory, zero-hit, unavailable, and failure paths distinct statuses with updated tests and docs.

## Reproduction Assessment

Yes, from source inspection: a focused unit path can make the embedded recall subagent return `NONE` and then inspect the persisted Active Memory status line. Current tests also assert `status=empty` for cached empty recall and terminal zero-hit search paths.

## Solution Assessment

No, current main is not the best solution because it uses `empty` for normal no-summary recall, zero-hit search, and unavailable-like paths. The narrow maintainable solution is to update `extensions/active-memory/index.ts` classification/cache behavior and the adjacent tests, preferably through one reviewed closing PR.
