## Summary

Current main still conflates normal no-useful-memory Active Memory runs with `status=empty`, and two open PRs reference this issue with closing syntax, so this issue should stay open until a reviewed fix lands or the competing PRs are resolved.

## What This Changes

Separate Active Memory status classification so normal `NONE` or no-useful-memory recall does not surface as `status=empty`.

## Best Possible Solution

Land one covered Active Memory fix that distinguishes normal no-relevant-memory from empty, unavailable, and timeout states, and update the cache behavior, terminal watcher tests, docs, and changelog together.

## Reproduction Assessment

Yes. A focused unit path can make the embedded recall subagent return `NONE` and inspect the persisted Active Memory status line; current tests already assert `status=empty` for that path and for related cache/zero-hit behavior.

## Solution Assessment

No, current main is not the best solution because it uses `empty` for normal no-summary recall and other terminal paths. The maintainable fix is a narrow Active Memory status contract update with aligned tests and docs, likely through one reviewed linked PR.
