## Summary

Current main still conflates successful no-useful-memory Active Memory runs with `status=empty`; the source maps `NONE`/empty summaries and zero-hit terminal search results to `empty`, and tests still assert that behavior. The issue should remain open while maintainers review or consolidate the linked fix PRs.

## What This Changes

The issue requests clearer Active Memory status classification so normal no-useful-memory recall is not surfaced to users as `status=empty`.

## Best Possible Solution

Land one canonical Active Memory fix that keeps hidden no-injection behavior unchanged, emits a distinct successful no-relevant-memory status, avoids caching empty no-match results, and updates focused regression coverage and docs.

## Reproduction Assessment

Yes from source inspection: a focused unit path can make the embedded recall subagent return `NONE` and inspect the persisted Active Memory status line. Current tests also assert the problematic `status=empty` behavior on main.

## Solution Assessment

No for current main: it still uses `empty` for normal no-summary recall, zero-hit search, and unavailable-like paths. The narrow maintainable fix belongs in `extensions/active-memory/index.ts` status classification/cache eligibility plus adjacent tests, with a maintainer choice for the final status spelling.
