## Summary

Keep open: current main still maps normal Active Memory no-useful-memory results, zero-hit terminal search results, and generic failure paths to `status=empty`, and it caches empty results. The closed related PR attempted a narrow fix but was not merged; the open related PR only covers the warning-only subset.

## What This Changes

The issue asks Active Memory to distinguish successful no-relevant-memory recall from real failures, zero-hit retries, unavailable memory, and successful recalled context instead of surfacing all of them as `status=empty`.

## Best Possible Solution

Keep the hidden no-injection behavior for `NONE`, but emit a distinct successful no-relevant-memory status, stop caching no-summary empty results, reserve unavailable/failed statuses for hard errors, and update the Active Memory tests and changelog.

## Reproduction Assessment

Yes. A focused source reproduction can make the embedded recall subagent return `NONE` and inspect the persisted Active Memory status line; current tests already assert `status=empty` for that path and for zero-hit terminal search.

## Solution Assessment

Yes. The narrow maintainable fix is inside Active Memory status classification, terminal-search handling, and cache eligibility, with tests updated to preserve the existing hidden prompt behavior while changing the user-facing status.
