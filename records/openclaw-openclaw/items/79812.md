## Summary

Current main still conflates successful no-useful-memory Active Memory runs with `status=empty`; the source maps `NONE`/empty summaries and zero-hit terminal search results to `empty`, and tests currently assert that behavior.

## What This Changes

Active Memory should stop surfacing ordinary `NONE` or no-useful-memory recall outcomes as `status=empty` and separate them from zero-hit, unavailable, timeout, and successful recall states.

## Best Possible Solution

Keep the hidden no-injection behavior unchanged, but emit a distinct successful no-relevant-memory status for `NONE`, reserve failure-style statuses for unavailable/timeout/error paths, and update the active-memory regression tests and docs accordingly.

## Reproduction Assessment

Yes from source inspection: a focused unit path can make the embedded recall subagent return `NONE` and then inspect Active Memory plugin debug lines. Current tests already assert `status=empty` for that path, so the reported behavior is high-confidence even though I did not run tests in this read-only review.

## Solution Assessment

No, the current implementation is not the best way to solve the issue because it uses `empty` for normal no-summary recall, zero-hit search, and unavailable/error-like paths. The narrow maintainable fix is inside `extensions/active-memory/index.ts` status classification, cache eligibility, and adjacent tests.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/79812 by separating Active Memory no-useful-memory status from failure/empty states. In `extensions/active-memory/index.ts`, keep the existing behavior of not injecting hidden prompt context when the subagent returns `NONE`, but do not persist or surface ordinary no-summary recall as `status=empty`; use a distinct successful status such as `no_relevant_memory` or a clearly documented existing success status. Do not cache empty/no-summary recall results unless the result has a real summary. Stop treating a single zero-hit `memory_search` result as a terminal empty abort; reserve terminal short-circuiting for true unavailable/error/disabled conditions and surface those as `unavailable` or another failure-like status instead of normal no-memory. Update `extensions/active-memory/index.test.ts` to cover `NONE`, zero-hit retry/no-abort, unavailable/error, cache eligibility, and visible plugin debug lines. Update `docs/concepts/active-memory.md` and add a user-facing `CHANGELOG.md` fix entry if behavior/docs change. Do not change memory-core/QMD search semantics, provider defaults, or add a new config option.
