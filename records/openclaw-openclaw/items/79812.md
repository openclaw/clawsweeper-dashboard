## Summary

Keep open: current main still conflates successful no-useful-memory Active Memory runs with `status=empty`, and there are already open closing PRs attempting to fix the same issue.

## What This Changes

The issue requests that Active Memory stop surfacing `status=empty` for normal `NONE` or no-useful-memory recall outcomes and separate them from zero-hit, unavailable, timeout, and successful recall states.

## Best Possible Solution

Land one consolidated Active Memory fix that preserves no-injection behavior while reporting a non-failure status for normal `NONE` results, avoids sticky caching of empty no-summary runs, and updates the focused tests and docs.

## Reproduction Assessment

Yes. Source inspection shows the `NONE`/empty-summary path and zero-hit terminal watcher still produce `status=empty`, and current tests assert that user-visible status.

## Solution Assessment

No. Current main is not the best implementation because it uses `empty` for normal no-useful-memory, zero-hit, and unavailable/error-like outcomes; the narrow fix belongs in Active Memory status classification, cache eligibility, and adjacent regression tests.
