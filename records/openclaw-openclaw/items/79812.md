## Summary

Current main still maps normal Active Memory no-summary recall and zero-hit terminal memory-search results to `status=empty`, and the issue has open closing PRs, so it should remain open until one fix lands.

## What This Changes

Distinguish benign Active Memory no-useful-memory outcomes from failure or genuinely empty states so the visible status line no longer reports `status=empty` for normal recall misses.

## Best Possible Solution

Land one focused Active Memory fix that classifies benign no-relevant-memory separately, preserves failure-oriented statuses, updates cache and terminal-watcher coverage, and aligns docs/changelog with the chosen label.

## Reproduction Assessment

Yes. From source inspection, making the embedded recall subagent return `NONE` reaches the empty-summary branch and existing tests assert the persisted `status=empty` line; no live run was attempted in this read-only review.

## Solution Assessment

No. The current implementation is not the best solution because it conflates benign no-memory recall, zero-hit terminal search, cache behavior, and unavailable/error-like states under `empty`; the safer fix is a narrow status-classification change with regression tests.
