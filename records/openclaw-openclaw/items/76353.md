## Summary

Keep open: current main still unconditionally warns for compile-cache prune failures, and open PR #76362 is the active closing fix candidate rather than a reason to close this issue before merge.

## What This Changes

Suppress misleading EACCES/EPERM warnings when postinstall cannot prune another user's shared Node compile-cache directory.

## Best Possible Solution

Land PR #76362 or an equivalent narrow patch that suppresses only EACCES/EPERM for compile-cache prune/read failures while retaining warnings for unexpected errors.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: `EACCES` or `EPERM` from `readdirSync` or `rmSync` in `pruneOpenClawCompileCache` reaches unconditional `log.warn`; I did not run the root/non-root global install sequence because this review is read-only.

## Solution Assessment

Yes. A narrow errno guard in the compile-cache prune warning path is the maintainable fix boundary; changing the shared cache location or ownership policy would be broader than this cosmetic postinstall bug requires.
