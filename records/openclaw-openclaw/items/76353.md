## Summary

Keep open: current main still warns for any compile-cache prune failure, so the reported EACCES/EPERM shared-cache case is not handled. The related closed #72042 covered packaged-update and stale-cache failures, but not this permission-warning path.

## What This Changes

Suppress misleading permission-denied warnings when postinstall tries to prune a shared Node compile cache it cannot write.

## Best Possible Solution

Update the postinstall compile-cache prune helper to silently skip EACCES/EPERM on shared cache directories while preserving warnings for unexpected prune failures.

## Reproduction Assessment

Yes. Source inspection and the existing mocked failure test show that any rmSync/readdir failure in pruneOpenClawCompileCache emits the reported warning; I did not run the root/non-root global install steps.

## Solution Assessment

Yes. A narrow errno guard for EACCES/EPERM in the compile-cache prune catch path is the maintainable fix; changing cache path ownership/scoping would be a broader behavior change.
