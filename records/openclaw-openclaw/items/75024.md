## Summary

Keep open: current main still lacks the Feishu root fallback, and this PR is the narrow parser-side fix, but it is not merge-ready until the required user-facing changelog entry is added. ## What This Changes

The PR changes Feishu message parsing so `ctx.rootId` falls back from `root_id` to `reply_target_message_id`, `parent_id`, and `upper_message_id`, adds `upper_message_id` to the event type, and adds parser tests for the fallback order.

## Best Possible Solution

Merge the narrow Feishu parser fix after adding one active-version changelog Fixes bullet and checking merge overlap with #73958. Keep the fallback in `parseFeishuMessageEvent`, because downstream session resolution and reply dispatch already consume normalized `rootId`.

## Reproduction Assessment

Yes. Build a Feishu group or topic reply event with no `root_id` or `thread_id` but with `reply_target_message_id` or `parent_id`; current main leaves `ctx.rootId` unset, so the topic/session resolver cannot anchor the turn to that reply ancestor.

## Solution Assessment

Yes, after the changelog entry. Parser normalization is the narrowest maintainable fix because the session resolver, inbound context, and reply target selection already use `ctx.rootId`; broader routing changes belong in the separate #73958 review.
