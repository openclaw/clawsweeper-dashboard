## Summary

Keep open: current main still leaves Feishu reply events without `root_id` unable to supply a stable thread root, and this PR remains a narrow implementation path. It is not merge-ready because the user-visible Feishu routing fix still lacks the required Unreleased changelog entry.

## What This Changes

The PR updates Feishu message parsing to derive `rootId` from fallback reply/ancestor fields, adds `upper_message_id` to the event type, and adds parser fallback tests.

## Best Possible Solution

Land this Feishu plugin fix or an equivalent narrow parser normalization after adding the required Unreleased changelog entry; a dispatch-level regression for topic-session reuse would improve confidence but is not required to understand the intended boundary.

## Reproduction Assessment

Yes. A focused reproduction is a Feishu group/topic reply event with no `root_id` or `thread_id` but with `reply_target_message_id` or `parent_id`; current main leaves `ctx.rootId` unset, so routing cannot use that ancestor as the session root.

## Solution Assessment

Yes, with the changelog gap fixed. Normalizing the fallback in `parseFeishuMessageEvent` is the narrowest maintainable solution because downstream session routing and reply dispatch already consume `ctx.rootId`.
