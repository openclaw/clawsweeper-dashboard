## Summary

Keep open for normal PR review. Current main still leaves Feishu `ctx.rootId` unset when a reply event omits `root_id` but carries ancestor IDs, and this PR is a narrow parser/type/test/changelog fix for that remaining plugin-local gap.

## What This Changes

The PR updates Feishu message parsing so `rootId` falls back from `root_id` to `reply_target_message_id`, `parent_id`, and `upper_message_id`, with event typing, parser coverage, and a changelog entry.

## Best Possible Solution

Land this narrow Feishu parser, event type, test, and changelog fix after normal CI, with merge-order awareness for #73958.

## Reproduction Assessment

Yes. Construct a Feishu group or topic reply event with no `root_id` or `thread_id` but with `reply_target_message_id` or `parent_id`; current main leaves `ctx.rootId` unset, so topic-session routing cannot anchor to that ancestor.

## Solution Assessment

Yes. Normalizing the fallback in `parseFeishuMessageEvent` is the narrowest maintainable boundary because the resolver and reply dispatcher already consume `rootId`; the broader #73958 routing semantics can stay separate.
