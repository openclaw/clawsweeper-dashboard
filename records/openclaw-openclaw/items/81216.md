## Summary

Keep open: current main still relies on static/provider-owned catalog data and has no scheduled refresh workflow. The related picker regression was fixed by https://github.com/openclaw/openclaw/pull/81239, but this newer CI-refresh proposal remains a feature and maintainer policy/security decision.

## What This Changes

The issue requests a scheduled CI script and workflow that use maintainer provider keys to refresh claude-cli, codex-cli, and google-gemini-cli model catalogs and open reviewable PRs.

## Best Possible Solution

Maintainers should first choose the catalog ownership and security model, then implement any refresh as provider/plugin-owned generation with explicit filters, reviewable diffs, and tests for each affected runtime.

Unclear. CI-side refresh is plausible, but the best design needs maintainer agreement on secrets, workflow permissions, per-provider filtering, and which plugin-owned files are the source of truth before implementation.
