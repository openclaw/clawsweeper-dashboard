## Summary

Keep open: current main does not implement a CLI-backend live model enumeration hook, and `/models` still builds its provider/model data from the existing catalog and runtime-choice metadata. This is a feature and ownership/API decision, not conservative cleanup.

## What This Changes

Request dynamic live model discovery for CLI runtime providers instead of relying on bundled or cached static catalog rows in `/models`.

## Best Possible Solution

Define a plugin-owned live catalog or CLI-backend enumeration seam that preserves canonical provider refs plus runtime choices, has explicit timeout/error behavior, and avoids hardcoding upstream model snapshots in core.

## Reproduction Assessment

Not applicable as a feature request. Source inspection confirms the requested live CLI enumeration path is absent: `/models` reads the model catalog, while CLI backend contracts only expose execution config and related hooks.

## Solution Assessment

Unclear until maintainers choose the ownership boundary. The direction is plausible, but the narrowest maintainable implementation should probably reuse provider-owned catalog/plugin seams or add a typed CLI-backend catalog hook rather than embedding provider API policy in the picker.
