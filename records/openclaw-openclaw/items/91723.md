## Label Justifications

- `P1`: This is a released Microsoft Teams channel regression that visibly duplicates long assistant replies and forces affected deployments to disable streaming.
- `impact:message-loss`: The affected problem class is channel message delivery correctness: already-streamed assistant text can be delivered again as a separate Teams block message.

## Summary

Keep open: current main and the latest release still carry the SDK streaming path that can return the full final payload to block delivery after a streamed prefix is already visible, so the reported long-reply duplicate is not already fixed or safely superseded.

## What This Changes

The issue requests restoring Microsoft Teams partial-stream fallback behavior so replies over about 4000 characters do not duplicate already-streamed text after the SDK rebase.

## Best Possible Solution

Restore streamed-prefix trim or suppression in the Teams SDK stream controller, preferably using the SDK-owned stream state/clearText seam where appropriate, with regression coverage and live Teams DM proof for long replies.

## Reproduction Assessment

Yes. The source path is high-confidence: in a personal Teams chat with partial streaming, emitted text is retained in the SDK stream, and OpenClaw can return the full final payload to block delivery after finalize fallback; I did not run a live Teams client repro.

## Solution Assessment

Yes, the narrow fix belongs in the Teams plugin streaming/finalization boundary, not a new config option. The best repair should preserve current cancel behavior while restoring the old streamed-prefix trim/suppression semantics for long-reply fallback.
