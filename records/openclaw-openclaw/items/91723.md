## Summary

Keep open: the report describes a valid Teams long-reply streaming regression on current main, and the directly linked fix candidate remains open/draft rather than merged. Current msteams source still has the SDK-era finalize fallback path that can return the full pending final payload after a streamed close failure, so this issue should stay open until https://github.com/openclaw/openclaw/pull/91729 or a replacement fix lands with real Teams proof.

## What This Changes

The issue requests restoring MS Teams long-reply streamed-prefix trimming so fallback block delivery does not repost already streamed text after replies exceed the Teams 4000-character streaming limit.

## Best Possible Solution

Land or replace the focused msteams fix that trims or suppresses fallback text based on the already streamed prefix, with regression tests and real Teams long-reply proof before closing this issue.

## Reproduction Assessment

Yes from source, but not from a live Teams run in this review. Current main still has a fallback path that can return the full pending final payload after SDK stream close failure without trimming emittedTextLength, matching the reported >4000-character duplicate path.

## Solution Assessment

Yes, the best fix is a narrow msteams repair that restores the pre-rebase streamed-prefix trim/suppression contract in the SDK controller rather than changing provider behavior or disabling streaming globally.
