## Summary

Keep open. Current main still does not expose the selected backend provider/model on the OpenAI-compatible chat-completions response, and the request is narrow enough for a focused Gateway API repair.

## What This Changes

Expose the actual backend model used by `/v1/chat/completions` responses so API clients can display model attribution without probing the agent.

## Best Possible Solution

Add backend model attribution to the Gateway OpenAI-compatible chat-completions surface while preserving agent-target request routing. The end state should report fallback-selected provider/model accurately for both non-streaming and streaming responses, with docs and tests covering the response contract.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to call `/v1/chat/completions` with `model: "openclaw"` and a backend override/default model; current main writes `openclaw` into non-streaming and SSE response `model` fields and has no selected-model response header.

## Solution Assessment

Yes, the narrow maintainable direction is to thread actual selected provider/model metadata from the agent run into the Gateway response surface. Replacing the response `model` field may be acceptable, but an additive response attribution mechanism is lower risk unless maintainers intentionally change compatibility semantics.

- Changing the JSON `model` field directly may conflict with the documented agent-target request contract, so an additive attribution surface or carefully documented response-only semantic change is safer.
- Streaming responses currently flush SSE headers before the agent run completes, so a response-header-only implementation could be stale or unavailable when fallback changes the selected model.
- The selected provider/model is tracked near the runtime boundary but is not currently threaded through `agentCommandFromIngress` as a public result value for the Gateway handler.
