## Summary

Keep open. Current main still echoes the request-side OpenAI `model` value in `/v1/chat/completions` JSON and SSE output, while docs only define `x-openclaw-model` as a request override and no response attribution surface is implemented.

## What This Changes

Expose the actual backend provider/model used by `/v1/chat/completions` responses so external clients can display model attribution without probing the agent.

## Best Possible Solution

Add backend model attribution to the Gateway OpenAI-compatible chat-completions surface while preserving agent-target request routing. The maintainable end state is an additive, documented, and tested attribution mechanism for non-streaming and streaming responses that reports override and fallback-selected provider/model accurately.

## Reproduction Assessment

Yes. The current source path is enough to reproduce: submit a request with `model: "openclaw"` and the handler will return that same request-side value in the JSON response and SSE chunks, independent of the backend model selected by the agent runtime.

## Solution Assessment

Yes for keeping this open and routing it as a narrow repair. The safest fix is to thread selected-model metadata from the agent runtime into the Gateway response surface without repurposing the request `model` field away from agent targeting unless maintainers choose that compatibility break.

## Repair Work Prompt

Fix the Gateway OpenAI-compatible `/v1/chat/completions` response attribution gap from https://github.com/openclaw/openclaw/issues/55450. Current behavior returns the request-side agent target (`openclaw`, `openclaw/default`, or `openclaw/<agentId>`) as `model` in JSON and SSE chunks, while clients need the actual backend provider/model used after request override and fallback. Keep request routing semantics intact: do not repurpose the request `model` field away from agent targeting unless maintainers explicitly choose that product change. Prefer an additive response attribution surface such as `X-OpenClaw-Model` and/or a documented stream-safe metadata strategy that can accurately report fallback-selected provider/model. Thread actual selected model data from the agent runtime boundary, likely via an `onModelSelected` callback or result metadata, into `src/gateway/openai-http.ts`. Cover non-streaming responses, streaming chunks or metadata, request `x-openclaw-model` overrides, default agent model selection, and fallback-selected model attribution. Update `docs/gateway/openai-http-api.md` and add a user-facing changelog entry with reporter credit (`Thanks @k9spy`). Do not change provider-specific routing, model allowlist enforcement, or the `/v1/models` agent-target contract.
