## Summary

Keep open: current main and the latest release still accept and advertise inline macOS CLI gateway passwords, while the core Gateway CLI already has a safer file/env pattern that has not been carried over.

## What This Changes

The issue requests a safe non-argv way to provide macOS CLI gateway passwords instead of passing plaintext through `--password`.

## Best Possible Solution

Add a macOS CLI safe-secret input path aligned with the existing Gateway CLI file/env contract, keep inline flags backward-compatible with warnings, and cover connect, configure-remote, wizard, password, and token behavior consistently in tests and docs.

## Reproduction Assessment

Yes. Source inspection shows `openclaw-mac connect --password <secret>` and `configure-remote --password <secret>` store the secret in parsed argv options, which process-listing tools can observe while the process is running.

## Solution Assessment

Partly. A non-argv credential path is the right direction, but the best fix should align with the existing `--password-file`/env Gateway CLI pattern and decide whether token and wizard coverage ship in the same change.
