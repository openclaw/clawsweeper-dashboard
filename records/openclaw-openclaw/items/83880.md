- `impact:security`: The report concerns gateway credentials being exposed through process arguments and shell history.
- `impact:auth-provider`: The affected surface is gateway authentication credential handling for token/password-based access.

## Summary

Keep open: current main and latest release still accept macOS gateway credentials through argv with no macOS CLI safe-secret alternative, while the core Gateway CLI already has a file/env-backed pattern to align with.

## What This Changes

The issue requests a safer macOS CLI credential input path so gateway passwords are not exposed through plaintext command-line arguments.

## Best Possible Solution

Add a macOS CLI safe-secret path aligned with the existing Gateway CLI file/env pattern, keep inline flags compatible with explicit warnings, and cover connect, configure-remote, wizard, password, and token behavior with focused Swift tests and docs.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes; source inspection shows `--password` values are read from `CommandLine.arguments` and passed into gateway auth, so a long-running invocation exposes that secret through normal argv/process-listing behavior. I did not execute a live macOS process-list check during this read-only review.

## Solution Assessment

Is this the best way to solve the issue? The requested direction is right, but `--password-stdin` alone may be too narrow; the better fit is to reuse the existing file/env-backed Gateway CLI contract and decide token plus wizard coverage explicitly.
