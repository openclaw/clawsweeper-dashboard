## Summary

Keep open: current main and latest release still accept macOS CLI gateway passwords through argv, and the only linked fix PR closed unmerged while still incomplete.

## What This Changes

The issue requests a safer macOS CLI credential input path so gateway passwords do not need to be supplied as plaintext command-line arguments.

## Best Possible Solution

Add an additive macOS CLI safe-secret input contract aligned with the core Gateway CLI pattern, keep legacy inline flags compatible with warnings, and cover connect/configure-remote/wizard scope according to the maintainer security decision.

## Reproduction Assessment

Yes by source inspection: current main still maps `--password` from `CommandLine.arguments` into Gateway auth for `connect`, `configure-remote`, and `wizard`. I did not run a live macOS process-list reproduction in this read-only review.

## Solution Assessment

Unclear as a final design: a non-argv credential path is the right direction, but maintainers should choose the exact stdin/file/env contract and whether token plus wizard coverage belongs in the same change.
