## Label Justifications

- `P2`: This is a concrete credential-exposure hardening issue with limited local-machine blast radius rather than a core runtime outage.
- `impact:security`: The issue is about gateway credentials being exposed through process arguments and shell history.
- `impact:auth-provider`: The affected surface is gateway authentication credential handling for token/password-based access.

## Summary

Keep open: current main and v2026.5.19 still accept macOS gateway secrets through argv with no safe file/stdin/env alternative or inline warning, while the core Gateway CLI already has a safer pattern this surface has not adopted.

## What This Changes

The issue requests a safe macOS CLI credential input path so `--password` does not expose gateway secrets in process listings or shell history.

## Best Possible Solution

Add a macOS CLI safe-secret input contract aligned with the existing Gateway CLI file/env warning pattern, keep inline flags backward compatible with warnings, and cover connect, configure-remote, wizard, password, and token behavior consistently in tests/docs.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: pass `--password` to `openclaw-mac connect`, `configure-remote`, or `wizard`, and the secret is part of the process argv before being resolved into Gateway auth.

## Solution Assessment

Unclear. A non-argv credential path is the right direction, but the exact contract should be chosen against the existing `--password-file`/env Gateway CLI pattern and should include a maintainer decision on token and wizard coverage.
