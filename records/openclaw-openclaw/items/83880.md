## Summary

Keep open: current main and the latest release still expose macOS CLI gateway credentials through inline argv flags, and the remaining fix needs a security/product decision on the supported safe-secret contract.

## What This Changes

The issue asks for a safe non-argv credential input path for macOS CLI password handling while retaining compatibility and warning on inline secrets.

## Best Possible Solution

Add a macOS CLI safe-secret input path aligned with the core Gateway CLI file/env pattern, keep inline flags compatible with explicit warnings, and cover connect, configure-remote, wizard, password, and token behavior consistently.

## Reproduction Assessment

Yes. Source inspection shows the macOS CLI copies `--password` values from command-line arguments into credential fields, so a long-running invocation exposes that value through argv/process listings. I did not execute the macOS binary during this read-only review.

## Solution Assessment

The requested direction is partly right: a non-argv credential path is the right security fix, but maintainers should align it with the existing `--password-file`/env Gateway CLI pattern and decide token plus wizard coverage before implementation.
