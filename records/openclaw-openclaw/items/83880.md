## Label Justifications

- `P2`: This is a real credential-exposure hardening issue with local-machine scope rather than an outage or active remote bypass.
- `impact:security`: The issue concerns gateway credentials exposed through process arguments and shell history.
- `impact:auth-provider`: The affected surface is gateway token/password authentication input for macOS CLI connections.

## Summary

Keep open: current `main` and the latest release still accept macOS CLI gateway passwords through argv, while the core Gateway CLI already has a safer file/env pattern that this surface has not adopted.

## What This Changes

The issue asks for a safer non-argv way to provide macOS CLI gateway passwords instead of passing secrets with `--password`.

## Best Possible Solution

Add a macOS CLI safe-secret input path aligned with the existing Gateway CLI file/env pattern, keep inline flags compatible with explicit warnings, and cover `connect`, `configure-remote`, `wizard`, password, and token behavior consistently in docs and Swift tests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: pass `--password` to a long-running `openclaw-mac` command and the secret is part of `CommandLine.arguments`, which process-listing tools can observe.

## Solution Assessment

Partly. A non-argv credential path is the right direction, but the best implementation should follow the existing Gateway CLI `--password-file`/env warning pattern and settle token and wizard coverage before coding.
