## Label Justifications

- `P2`: This is a real credential-exposure hardening issue with local-machine blast radius, not a core runtime outage.
- `impact:security`: The report concerns gateway credentials being exposed through command-line arguments and shell history.
- `impact:auth-provider`: The affected surface is gateway authentication credential handling for token/password-based access.

## Summary

Keep open: current main and the latest release still accept macOS CLI gateway secrets through argv, while the core Gateway CLI already has a safer file/env pattern that the macOS CLI has not adopted.

## What This Changes

The issue requests a safer macOS CLI credential input path so gateway passwords are not exposed through process listings or shell history.

## Best Possible Solution

Add a macOS CLI safe-secret input path aligned with the existing Gateway CLI file/env pattern, keep inline flags compatible with clear warnings, and cover password/token behavior across connect, configure-remote, and wizard with focused Swift tests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: invoke a long-running `openclaw-mac` command with `--password secret`, and that secret is part of `CommandLine.arguments`; I did not execute the macOS binary in this read-only review.

## Solution Assessment

Unclear. A non-argv credential path is the right direction, but the best contract should reuse or intentionally diverge from the existing `--password-file` / `OPENCLAW_GATEWAY_PASSWORD` pattern and settle token/wizard coverage.
