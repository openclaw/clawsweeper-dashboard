- `impact:security`: The issue is about gateway credentials being exposed through process arguments and shell history.
- `impact:auth-provider`: The affected surface is gateway authentication credential handling for token/password-based access.

## Summary

Keep open: current main and v2026.5.18 still accept macOS CLI gateway passwords through argv, while the existing core Gateway CLI already has a safer file/env pattern this surface has not adopted.

## What This Changes

Add a safer non-argv credential input path for `openclaw-mac` password auth while keeping inline `--password` compatibility with warnings.

## Best Possible Solution

Add a macOS CLI safe-secret input path aligned with the existing Gateway CLI file/env pattern, keep inline flags compatible with explicit warnings, and cover password, token, connect, configure-remote, and wizard behavior consistently in docs and tests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: passing `--password` to a long-running `openclaw-mac` command places the secret in `CommandLine.arguments`, which process-listing tools can observe.

## Solution Assessment

Partly. A non-argv credential path is the right direction, but the best fix should align with the existing `--password-file`/env Gateway CLI contract and decide whether token and wizard coverage ship together.
