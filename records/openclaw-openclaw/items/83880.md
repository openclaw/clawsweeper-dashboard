## Label Justifications

- `impact:security`: The issue is about Gateway credentials being exposed through process arguments and shell history.
- `impact:auth-provider`: The affected surface is Gateway authentication credential handling for token/password-based access.

## Summary

Keep open: current main and the latest release still expose macOS Gateway secrets through argv-only `--password`/`--token` paths, and the existing safe-secret pattern has not been applied to the macOS CLI.

## What This Changes

The issue asks OpenClaw to stop requiring plaintext macOS CLI Gateway passwords in process-visible command arguments and shell history.

## Best Possible Solution

Add a macOS CLI safe-secret input path aligned with the existing `--password-file`/env Gateway CLI pattern, keep inline flags compatible with explicit warnings, and cover `connect`, `configure-remote`, `wizard`, password, and token behavior in tests/docs.

## Reproduction Assessment

Yes. Source inspection shows the macOS CLI stores `--password` directly from `CommandLine.arguments`, which is the process-listing and shell-history exposure path described by the report.

## Solution Assessment

Partly. A non-argv credential path is the right direction, but the best fix should align with the existing `--password-file`/env Gateway CLI pattern and settle token/wizard coverage before implementation.
