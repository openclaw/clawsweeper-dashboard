- `impact:security`: The issue is about gateway credentials being exposed through process arguments and shell history.
- `impact:auth-provider`: The affected surface is gateway authentication credential handling for token/password-based access.

## Summary

Keep open: the reported credential exposure is still present on current main and in v2026.5.18, while the core Gateway CLI already has a safer file/env pattern that the macOS CLI has not adopted.

## What This Changes

The issue requests a non-argv way to pass macOS CLI gateway credentials instead of exposing `--password` values in process listings and shell history.

## Best Possible Solution

Add a macOS CLI safe-secret input path aligned with the existing Gateway CLI file/env pattern, keep inline flags compatible with explicit warnings, and cover connect, configure-remote, wizard, password, and token behavior consistently in docs and tests.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: pass `--password` to the long-running macOS CLI command and the secret is part of `CommandLine.arguments`, which process-listing tools can observe. I did not execute the macOS process in this read-only review.

## Solution Assessment

The requested direction is partly right: a non-argv credential path is the right fix, but the best implementation should align with the existing `--password-file`/env Gateway CLI pattern and decide whether token and wizard coverage ship in the same change.
