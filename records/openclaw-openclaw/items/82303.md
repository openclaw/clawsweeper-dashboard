## Summary

This PR should stay open because it adds a new opt-in Telegram streaming UX, but the branch still has a concrete draft-stream lifecycle bug and lacks live Telegram delivery proof for the visible chat behavior.

## What This Changes

The PR adds `streaming.progress.assistantPreview`, routes Telegram progress-mode assistant partials to a separate transient lane, updates config schema/metadata, and adds focused tests.

## Best Possible Solution

Keep the opt-in config shape only if maintainers accept the UX, reset or recreate the assistant draft stream after clear, document the config surface, and require redacted live Telegram proof of the two-lane behavior before merge.

## Reproduction Assessment

Yes for the PR regression by source inspection: `clear()` stops the assistant draft stream and the added boundary path does not recreate or reset it before later `update()` calls. The requested assistant preview lane is a new feature rather than a current-main bug reproduction.

## Solution Assessment

No. The feature direction is narrow and additive, but this implementation must reset the stopped assistant draft stream and needs live Telegram proof before it is a maintainable merge candidate.
