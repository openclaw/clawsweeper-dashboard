## Summary

Keep open: the Telegram two-lane preview feature is plausible, but the current patch still has a draft-stream lifecycle bug and does not show after-fix real Telegram delivery. The maintainer follow-up and the Mantis run both leave the same blockers unresolved.

## What This Changes

The PR adds an opt-in `streaming.progress.assistantPreview` Telegram config that streams assistant partials on a separate transient lane while preserving the progress/tool draft lane, with schema/metadata and tests.

## Best Possible Solution

Revise the branch to reset or recreate the assistant draft stream after clears, keep the config/docs surface aligned, and land only after redacted live Telegram proof shows both progress and assistant preview lanes.

## Reproduction Assessment

Yes for the PR-introduced lifecycle bug by source inspection: PR head clears the assistant stream at assistant-message boundaries, while current draft-stream controls ignore updates after a clear stops the stream. The requested assistant preview lane itself is a feature, not a current-main bug reproduction.

## Solution Assessment

No. The additive opt-in config direction may be maintainable, but this implementation must honor the draft-stream reset contract and include live Telegram proof before it is the best solution.
