## Summary

Keep open: current main and v2026.6.6 still lack a native web-tool domain allow/deny egress firewall, and the related open PRs are partial SSRF plumbing work rather than a proof-positive canonical fix for this security/config feature.

## What This Changes

The issue requests configurable `allowedDomains`/`denyDomains` network access control for web tools such as `web_fetch` and `web_search`.

## Best Possible Solution

Define and implement a native web-tool egress policy with explicit schema, deny-before-allow semantics, redirect/cache/provider/native-search coverage, docs, audit behavior, and security review before exposing it as a trusted firewall.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a new security/config capability request. Source, tests, docs, and v2026.6.6 inspection confirm the requested web-tool domain allow/deny policy is absent today.

## Solution Assessment

No, the issue is not ready as a direct implementation plan. Reusing the SSRF guard is likely part of the right boundary, but maintainers still need to decide global versus per-agent semantics, deny-before-allow behavior, provider/native-search coverage, cache keys, proxy interactions, and docs/audit guarantees.
