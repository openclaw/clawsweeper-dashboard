## Summary

Keep open. Current main and v2026.6.5 have SSRF primitives and narrow web_fetch fake-IP opt-ins, but they still lack the requested operator-defined allowed/denied domain egress policy across web_fetch and web_search; the remaining work is a security-sensitive product/config decision, not a stale or already-implemented item.

## What This Changes

The issue requests a new operator-configurable egress policy with allowed and denied domains for OpenClaw web tools, covering web_fetch and web_search.

## Best Possible Solution

Define a documented egress-policy contract with an explicit threat model, then enforce it consistently at every OpenClaw-owned web/network chokepoint included in that contract while documenting excluded surfaces honestly.

## Reproduction Assessment

Not applicable; this is a new feature/config request, not a report of broken documented behavior. Source and release checks show current main and v2026.6.5 lack the requested allowed/denied-domain policy surface.

## Solution Assessment

No, the proposed web_fetch chokepoint alone is too narrow for an egress firewall. The best solution needs a maintained security contract first, then enforcement across the included web_fetch, web_search, provider-native, redirect, and cache paths.
