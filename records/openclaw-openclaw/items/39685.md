## Summary

Keep open. Current main still lacks a shared allowedDomains/denyDomains egress firewall for OpenClaw-managed web tools; existing code only exposes narrower web_fetch SSRF opt-ins and Codex-native search filters, while the related security-profile issue and draft SSRF PR leave the public config and enforcement semantics unresolved.

## What This Changes

The issue requests a global and per-agent web-tool network egress policy with allowed and denied domains enforced through the existing SSRF/web fetch guard path.

## Best Possible Solution

Define a documented first-class web-tool egress policy with typed global and per-agent config, deny-before-allow matching, explicit web_fetch and native/managed web_search coverage, redirect/cache/audit tests, and alignment with the broader security-profile work.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a bug reproduction: this is a missing security/config feature, but source inspection clearly shows the requested shared allow/deny-domain policy is absent from current main.

## Solution Assessment

Is this the best way to solve the issue? Not as a one-shot automated fix; reusing SsrFPolicy hostname primitives is plausible, but maintainers need to decide public config shape, policy precedence, provider coverage, redirects, cache boundaries, and audit behavior first.
