## Summary

Keep open: current main and the latest release still lack the requested generic `allowedDomains`/`denyDomains` egress firewall for OpenClaw web tools. The new open hostname-allowlist PR is relevant, but it is unmerged and currently covers only part of the requested `web_fetch` allowlist surface, not deny rules or managed/native `web_search` coverage.

## What This Changes

Request a config-level domain allow/deny egress policy for `web_fetch` and `web_search`.

## Best Possible Solution

Define one maintainer-approved web-tool egress contract, land the `web_fetch` allowlist path if accepted, and explicitly decide the denylist plus managed/native `web_search` boundaries before closing this tracker.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a requested security/config capability. Source and release inspection show current main and `v2026.6.10` still lack a generic web-tool domain allow/deny surface.

## Solution Assessment

Unclear as proposed. Reusing `SsrFPolicy.hostnameAllowlist` is plausible for direct `web_fetch`, but the best solution needs a maintainer-approved contract across fetch, search, redirects, cache isolation, provider fallback, and hosted/native search behavior.
