## Summary

Keep open: current main and v2026.6.6 still lack a generic allowedDomains/denyDomains egress firewall for web_fetch and web_search, and the related open work is either narrower or a broad draft security-boundary refactor that needs maintainer/security direction.

## What This Changes

The issue requests configurable domain allow and deny policy for OpenClaw web-tool egress.

## Best Possible Solution

Define one canonical network-policy contract for web-tool egress, then implement it with schema/help/docs, audit output, cache-key isolation, redirect coverage, provider-fallback handling, and focused SSRF/web-search tests.

## Reproduction Assessment

Yes at source level. Current main and v2026.6.6 expose only narrow web_fetch SSRF compatibility flags and native Codex search allowedDomains, not a generic allowedDomains/denyDomains firewall for web_fetch and managed web_search.

## Solution Assessment

Unclear as proposed. Mapping allowedDomains to SsrFPolicy.hostnameAllowlist is plausible for direct web_fetch, but maintainers still need to decide deny precedence, cache behavior, provider fallbacks, web_search, browser, plugin HTTP, and proxy-boundary semantics.
