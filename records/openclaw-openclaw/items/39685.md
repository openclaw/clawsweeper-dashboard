## Summary

Keep open. Current main has useful SSRF primitives and a narrow Codex-native `web_search` domain filter, but it still lacks the requested shared global/per-agent `allowedDomains` plus `denyDomains` egress policy for OpenClaw-managed `web_fetch` and `web_search`; the related draft PR and broader security-profile issue are adjacent, not replacements.

## What This Changes

The issue requests a first-class OpenClaw web-tool egress firewall with `allowedDomains` and `denyDomains` config for `web_fetch` and `web_search`.

## Best Possible Solution

Define a documented first-class web-tool egress policy with typed global and per-agent config, deny-before-allow matching, explicit managed/native `web_search` and `web_fetch` coverage, redirect/cache/provider-fallback/audit tests, and coordination with the broader security-profile work.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a missing security/config feature, not broken existing documented behavior. Source inspection gives high confidence that the shared allow/deny-domain config and enforcement hook are absent on current main.

## Solution Assessment

No for a one-shot automated fix. Reusing existing `SsrFPolicy` hostname primitives is plausible, but the best solution needs maintainer/security decisions for policy semantics, provider coverage, cache boundaries, redirects, and audit output.
