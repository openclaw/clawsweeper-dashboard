## Summary

Keep open: current main and v2026.6.6 still lack a generic `allowedDomains`/`denyDomains` egress firewall for `web_fetch` and managed `web_search`; existing domain filtering is narrow/provider-specific, and the related open PR is not a safe canonical replacement.

## What This Changes

The issue requests configurable domain allow/deny network egress policy for OpenClaw web tools, especially `web_fetch` and `web_search`, using the existing SSRF guard infrastructure.

## Best Possible Solution

Define one canonical web-tool egress policy contract, then implement schema/help/docs/audit output plus cache, redirect, fallback, managed-search, and native-search tests around that contract.

## Reproduction Assessment

Not applicable as a runtime bug, but yes at source level: current main and v2026.6.6 expose only Codex-native `allowedDomains` and narrow `web_fetch` fake-IP SSRF flags, not a generic `allowedDomains`/`denyDomains` firewall.

## Solution Assessment

Unclear as proposed. Reusing `SsrFPolicy.hostnameAllowlist` is plausible for direct `web_fetch`, but the best solution needs a maintainer-approved contract across managed/native search, fetch fallback, cache, redirects, and provider/plugin boundaries.
