## What This Changes

The PR attempts to pass Telegram proxy-aware fetch and dispatcher fallback options into outbound remote media loading for direct sends and bot replies, with regression coverage.

## Best Possible Solution

Land a current-main patch that carries TelegramTransport sourceFetch, dispatcherAttempts, explicit-proxy DNS trust, and retry handling through direct sends and all reply delivery paths into shared web-media fetching, with focused tests and a changelog entry.

Yes. Source inspection on current main shows remote Telegram send and reply media still call loadWebMedia without the Telegram transport, while inbound Telegram media already has the proxy-aware fetch contract.

## Solution Assessment

No. The requested direction is right, but this PR is not the best mergeable implementation until it is ported from stale pinned-dispatcher fields to the current dispatcherAttempts, trustExplicitProxyDns, and shouldRetryTelegramTransportFallback API.
