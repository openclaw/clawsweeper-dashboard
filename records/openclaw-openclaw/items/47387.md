## Summary

Keep open. The Telegram outbound media proxy gap is still present on current main, but this PR should not merge as-is because it targets older transport option names instead of the current dispatcher-attempt media contract.

## What This Changes

The PR threads Telegram transport options through outbound remote media loading for direct sends and bot reply delivery, extends shared media option plumbing, and adds regression tests.

## Best Possible Solution

Land a current-main patch that carries TelegramTransport.sourceFetch, dispatcherAttempts, explicit-proxy DNS trust, and shouldRetryTelegramTransportFallback through direct sends and reply delivery into shared web-media fetching, with focused regression coverage and a changelog entry.

## Reproduction Assessment

Yes. Configure Telegram in a proxy-required environment and send or reply with a remote media URL; current-main source inspection shows the outbound prefetch paths call loadWebMedia without Telegram transport options while inbound media already has the proxy-aware contract.

## Solution Assessment

No. The requested direction is correct, but this PR is not the best mergeable implementation until it is ported from stale pinned-dispatcher fields to the current dispatcherAttempts, trustExplicitProxyDns, and shouldRetryTelegramTransportFallback API.
