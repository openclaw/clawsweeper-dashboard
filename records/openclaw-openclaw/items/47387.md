## Summary

Keep open. The underlying Telegram outbound media proxy gap is still visible on current main, but this PR should not merge as-is because it targets older transport option names and does not line up with the current dispatcher-attempt media API.

## What This Changes

The PR threads Telegram transport options through outbound remote media loading for direct sends and bot reply delivery, extends shared media load option plumbing, and adds regression tests.

## Best Possible Solution

Land a current-main port that carries TelegramTransport.sourceFetch, dispatcherAttempts, explicit-proxy DNS trust, and shouldRetryTelegramTransportFallback through direct sends, regular replies, and native-command replies into the shared web-media/fetchRemoteMedia path, with focused regression coverage and a user-facing changelog entry.

## Reproduction Assessment

Yes. Configure Telegram in a proxy-required environment and send or reply with a remote media URL; current-main source inspection shows those outbound paths call loadWebMedia without Telegram transport options, while inbound media already has the proxy-aware contract.

## Solution Assessment

No. The requested direction is correct, but this PR is not the best mergeable implementation because it targets older pinned-dispatcher names and old media option plumbing instead of current main's dispatcherAttempts, trustExplicitProxyDns, and shouldRetryTelegramTransportFallback shape.
