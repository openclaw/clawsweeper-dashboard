## Summary

Keep open. Current main still leaves Telegram outbound remote-media prefetch without proxy-aware transport options, but this branch is stale against the newer transport/media-loader contract and lacks live Telegram/proxy proof, so it is not mergeable as-is.

## What This Changes

The PR threads Telegram transport options through outbound remote media loading for direct sends and reply delivery, updates shared media-load plumbing, and adds regression tests.

## Best Possible Solution

Port the transport threading to current main's dispatcherAttempts, shouldRetryTelegramTransportFallback, and trustExplicitProxyDns media contract for both direct sends and reply delivery, while preserving SSRF and size-limit behavior and adding redacted live Telegram/proxy proof.

## Reproduction Assessment

Yes for source-level reproduction. On current main, the direct-send and final-reply remote media paths reach loadWebMedia without Telegram fetch or dispatcher transport options, so a proxy-required setup can still bypass the configured proxy before media delivery.

## Solution Assessment

No. The intended direction is correct, but this branch targets removed pinnedDispatcherPolicy/fallbackPinnedDispatcherPolicy/shouldRetryTelegramIpv4Fallback fields instead of the current dispatcherAttempts/trustExplicitProxyDns contract.
