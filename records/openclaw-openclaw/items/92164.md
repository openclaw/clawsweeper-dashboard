## Summary

Current main still has the reported CJK/trigram keyword gap and the PR is a focused repair, but it should stay open for maintainer review because external real behavior proof is not sufficient yet and the expanded LIKE fallback has large-index latency risk that CI will not settle.

## What This Changes

The PR changes memory-core keyword planning/search so CJK-bearing trigram queries use per-character LIKE terms and empty MATCH results retry the LIKE fallback, with regression coverage in `manager-search.test.ts`.

## Best Possible Solution

Land a memory-core fix at the keyword planner/search boundary after maintainers either accept or bound the CJK LIKE-scan latency risk and the PR includes real OpenClaw behavior proof or an explicit proof override.

## Reproduction Assessment

Yes at the source/dependency layer: current main sends multi-character CJK trigram tokens to MATCH, vector-only hybrid rows keep `textScore: 0`, and a direct `node:sqlite` FTS5 trigram probe reproduces the empty MATCH versus successful LIKE behavior. I did not establish a full live OpenClaw `memory_search` reproduction with an embedding provider.

## Solution Assessment

Mostly yes: fixing `planKeywordSearch` and `searchKeyword` is the right boundary, and the tests cover the central regression. The remaining question is whether the expanded synchronous LIKE fallback needs bounded latency proof or a maintainer-approved tradeoff before merge.
