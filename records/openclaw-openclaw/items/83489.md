## Summary

Keep open: the PR now has sufficient Pi/systemd proof and covers the exit-78 path, but it should not merge until the new pre-lock service-mode health check proves it is talking to OpenClaw rather than any local `/healthz` endpoint.

## What This Changes

The PR changes gateway service-mode startup so a duplicate service probes an existing listener before stale-PID cleanup, exits early when the listener is healthy, and adds focused regression coverage.

## Best Possible Solution

Keep the early service-mode exit, but validate that the listener is an OpenClaw gateway before skipping stale cleanup and startup, with a regression for a non-OpenClaw `/healthz` listener.

## Reproduction Assessment

Yes, with high confidence from source inspection and the supplied incident logs: current main performs service-mode stale cleanup before the later supervised health arbitration, and the PR body shows duplicate systemd services killing each other on port 18789. I did not run the live Pi/systemd scenario in this read-only review.

## Solution Assessment

No, not as written: moving the cleanup behind an early health check is the right direction, but the new pre-lock exit must verify an OpenClaw listener instead of trusting any generic `/healthz` response.
