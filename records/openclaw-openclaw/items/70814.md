## Summary

Keep open. The PR targets a real cron announce delivery gap and adds useful regression coverage, but the patch changes malformed `delivery.channel` values into explicit targets, which skips the existing implicit-mode stored `allowFrom` recovery path. The same target-confusion concern was already raised in the PR discussion, and current `main` has not implemented the requested misplaced-channel recovery safely.

## What This Changes

The PR changes isolated cron announce delivery resolution to infer the sole configured announce channel for non-channel `delivery.channel` values, moves that raw value into `delivery.to`, adds numeric-target resolver tests, and includes small unrelated macOS SwiftFormat hunks.

## Best Possible Solution

Land a narrowed cron-delivery fix that preserves target authorization semantics: keep the normal numeric `--to` Telegram path covered, and either repair legacy misplaced `delivery.channel` values through doctor/migration or carry an inferred-target marker through resolution so stored/configured `allowFrom` and ambiguity policy are applied before delivery.

## Reproduction Assessment

Yes. Resolve a cron delivery payload with one configured announce channel, `channel: "834732674"`, and no `to`; current `main` treats the numeric value as the requested channel/missing target, while the PR's new test demonstrates the intended recovery path. The security regression can be checked by adding a stored `allowFrom` entry and observing that the inferred explicit target bypasses the implicit reroute path.

## Solution Assessment

No, not as written. The PR is narrow and the regression intent is useful, but promoting a raw non-channel field to `explicitTo` changes delivery mode; a safer fix preserves implicit policy or keeps legacy repair outside runtime delivery.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Preserve allowFrom for inferred targets:** `src/cron/isolated-agent/delivery-target.ts:104`
  - body: This assignment makes a raw `delivery.channel` value become `explicitTo`, so `resolveSessionDeliveryTarget` marks the delivery as explicit. `resolveDeliveryTarget` only reads the stored `allowFrom` entries and reroutes blocked targets in implicit mode, so an inferred legacy value can skip that policy and be accepted differently than normal implicit delivery. Track inferred targets separately and apply the same policy, or keep this repair out of runtime delivery.
  - confidence: 0.86
