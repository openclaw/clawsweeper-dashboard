## Summary

Close as superseded: current main and the shipped cron validation path already cover numeric Telegram IDs via `delivery.to`, while this PR's unique fallback promotes malformed `delivery.channel` text into an explicit target and conflicts with target-policy semantics.

## What This Changes

The PR changes cron delivery resolution to infer the sole configured channel and promote an invalid `delivery.channel` string into `delivery.to`, adds resolver tests, and includes unrelated macOS SwiftFormat brace churn.

## Best Possible Solution

Keep the shipped behavior: pass Telegram chat IDs through `delivery.to` or `--to`, reject malformed `delivery.channel` at validation, and handle any legacy stored-job cleanup through a policy-preserving doctor or migration path.

## Reproduction Assessment

Yes for the PR finding: source inspection shows the branch assigns malformed `delivery.channel` text to `explicitTo`, which makes downstream delivery explicit and skips implicit `allowFrom` rerouting. The original user-facing `--to` numeric ID problem is already covered on current main.

## Solution Assessment

No. Runtime reinterpretation of `delivery.channel` is broader than the safe fix; the narrow maintainable solution is explicit `delivery.to` support plus validation rejection of malformed channel fields.
