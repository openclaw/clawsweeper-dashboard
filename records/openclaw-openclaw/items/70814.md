## Summary

Close: current main already handles the linked numeric Telegram cron announce problem through the documented delivery.to path and shipped validation for invalid delivery.channel values; the PR's unique runtime inference path should not merge because it weakens the existing implicit allowFrom policy.

## What This Changes

The PR changes cron delivery target resolution to treat a non-channel delivery.channel value as delivery.to when exactly one announce channel is configured, adds numeric-target resolver tests, and includes two macOS SwiftFormat formatting hunks.

## Best Possible Solution

Keep the shipped main behavior: numeric Telegram announce destinations belong in delivery.to, while invalid delivery.channel values should be rejected or repaired outside runtime delivery rather than silently reinterpreted.

## Reproduction Assessment

Yes. Source inspection shows current main resolves explicit delivery.to through channel fallback, while the PR's malformed delivery.channel case would otherwise flow as a channel value; the PR body also includes harness output for both resolver paths.

## Solution Assessment

No, the PR branch is not the best solution. Current main already solves the user-facing numeric target path and validates bad channel fields, while the PR's runtime inference changes authorization semantics by converting a channel field into an explicit recipient.
