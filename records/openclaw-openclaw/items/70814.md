## Summary

Keep open. Current main still has the misplaced `delivery.channel` failure path, but the PR should not merge as written because it promotes an invalid channel value into an explicit target and skips the existing implicit delivery allowFrom reroute/store policy for inferred recipients.

## What This Changes

The PR changes cron delivery resolution to infer the sole configured announce channel when `delivery.channel` is not a real channel, moves that raw value into `delivery.to`, adds resolver regressions for numeric targets, and includes unrelated macOS SwiftFormat-only hunks.

## Best Possible Solution

Land a narrower cron-delivery fix that keeps numeric Telegram IDs working through the documented `delivery.to` path and handles legacy/misfiled `delivery.channel` values without weakening target policy. A safe end state would either repair bad stored jobs through doctor/migration or mark inferred targets separately and apply the same allowFrom/read-store protections and ambiguity checks as implicit delivery.

## Reproduction Assessment

Yes. On current main, resolving one configured announce channel with `channel: "834732674"` and no `to` leaves the numeric value as the requested channel, which then reaches outbound resolution as an unsupported channel; the PR's added test exercises that exact recovery path.

## Solution Assessment

No, not as written. The PR is focused on the reported symptom, but promoting a raw non-channel value into `explicitTo` is broader than the existing implicit fallback semantics; the safer alternative is to preserve or reapply allowFrom policy for inferred targets, or repair legacy config outside runtime delivery.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Preserve allowFrom for inferred targets:** `src/cron/isolated-agent/delivery-target.ts:104`
  - body: This assignment makes a raw `delivery.channel` value become `explicitTo`, which causes `resolveSessionDeliveryTarget` to return explicit mode. `resolveDeliveryTarget` only reads the store-backed allowFrom override and reroutes blocked recipients in implicit mode, so a malformed stored job can deliver to a recipient that implicit delivery would have rerouted or blocked. Track the inference separately and apply the same allowFrom policy, or keep this recovery out of runtime delivery.
  - confidence: 0.86
