## Summary

Close as superseded: the linked cron Telegram report is already closed as implemented on current main through the explicit `delivery.to` path and cron validation, while this PR's unique runtime reinterpretation of `delivery.channel` conflicts with current target-policy semantics.

## What This Changes

The PR changes cron delivery resolution so a non-channel `delivery.channel` value with no `delivery.to` is treated as `to` through the sole configured announce channel, adds numeric-target resolver tests, and includes SwiftFormat-only macOS hunks.

## Best Possible Solution

Keep current main's safer behavior: pass Telegram chat IDs through `delivery.to` or `--to`, reject malformed `delivery.channel` values at validation, and only revisit legacy stored-job repair through a policy-preserving doctor or migration path.

## Reproduction Assessment

Yes for the PR finding. Source inspection shows the branch turns a raw non-channel `delivery.channel` value into `explicitTo`, which changes the downstream mode to explicit and skips the implicit allowFrom reroute path.

## Solution Assessment

No. Runtime reinterpretation of `delivery.channel` is broader than the safe documented fix; the maintainable path is explicit `delivery.to` plus gateway validation, with any legacy cleanup handled outside hot delivery resolution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve allowFrom for inferred targets:** `src/cron/isolated-agent/delivery-target.ts:104`
  - body: This assignment makes a raw `delivery.channel` value become `explicitTo`, which makes `resolveSessionDeliveryTarget` return explicit mode. Cron delivery only applies the configured allowFrom reroute in implicit mode, so a malformed stored job can send to a recipient that implicit delivery would have rerouted or blocked. Track the inference separately and preserve the same policy, or keep this recovery out of runtime delivery.
  - confidence: 0.88
