## Summary

Keep open. Current main still displays cache counters in `/status`, but the cost estimate only forwards input/output tokens, while https://github.com/openclaw/openclaw/pull/65381 is the active narrow fix candidate.

## What This Changes

The issue asks for `/status` cost estimates to include cache read/write token costs when cache counters are present.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/65381 or an equivalent narrow fix that forwards `cacheRead` and `cacheWrite` into the existing status cost estimator with regression coverage for stored-session and transcript-derived cache usage.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: cache counters reach the visible `/status` cache line, but the cost calculation passes only input/output tokens into the cache-aware estimator.

## Solution Assessment

Yes. Reusing the existing `estimateUsageCost` contract from the `/status` caller is the narrowest maintainable fix; adding a separate pricing path would duplicate existing logic.
