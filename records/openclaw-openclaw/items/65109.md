## Summary

Keep open. Current main still displays cache counters in `/status`, but the status cost estimate passes only input/output tokens to the cache-aware estimator; open PR #65381 remains the focused implementation candidate.

## What This Changes

The issue reports that `/status` includes cache read/write token counts in the display but omits cache-read pricing from the total cost estimate.

## Best Possible Solution

Land PR #65381 or an equivalent narrow fix that forwards `cacheRead` and `cacheWrite` into the existing `/status` cost estimate and adds regression coverage for entry-derived and transcript-derived cache usage.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: nonzero cache counters reach the status cache line, but the cost calculation passes only input/output tokens into the estimator.

## Solution Assessment

Yes. Reusing the existing cache-aware `estimateUsageCost` contract from the `/status` caller is the narrowest maintainable solution; a separate pricing path would duplicate existing logic.
