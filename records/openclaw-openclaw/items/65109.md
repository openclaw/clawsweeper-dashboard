## Summary

Keep open. Current main still displays cache counters in `/status`, but the cost estimate call still builds usage from only input/output tokens; the existing shared estimator and docs already support cache pricing, and open PR #65381 remains the focused implementation candidate.

## What This Changes

The issue asks for `/status` cost estimates to include billable cache read and cache write tokens when model pricing defines cache rates.

## Best Possible Solution

Land PR #65381 or an equivalent narrow fix that forwards `cacheRead` and `cacheWrite` into the existing `/status` cost estimate, treats cache-only usage as billable usage, and adds regression coverage for both session-entry and transcript-derived cache usage.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: nonzero cache counters reach the status cache line, but the cost calculation passes only input/output tokens into the estimator.

## Solution Assessment

Yes. Reusing the existing cache-aware `estimateUsageCost` contract from the `/status` caller is the narrowest maintainable fix; a separate pricing path would add unnecessary duplication.
