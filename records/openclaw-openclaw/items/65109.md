## Summary

Keep open. Current main still displays cache counters in `/status`, but the cost estimate only passes input/output tokens to the cache-aware estimator; https://github.com/openclaw/openclaw/pull/65381 is the active narrow fix candidate but has not merged.

## What This Changes

The issue asks for `/status` cost estimation to include displayed `cacheRead` and `cacheWrite` token counts, especially Anthropic cache-read billing.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/65381 or an equivalent narrow main change that forwards cache counters into `estimateUsageCost` and preserves regression coverage.

## Reproduction Assessment

Yes. Source inspection on current main shows cache counters are read and rendered in `/status`, while `estimateUsageCost` receives only input/output tokens, so a cache-heavy session undercounts by the configured cache token cost.

## Solution Assessment

Yes. Reusing the existing cache-aware `estimateUsageCost` contract from the status caller is the narrowest maintainable fix; a separate pricing path would duplicate existing logic.
