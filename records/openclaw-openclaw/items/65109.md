## Summary

Keep open. Current `origin/main` still displays cache counters in `/status` but prices only input/output tokens, while https://github.com/openclaw/openclaw/pull/65381 is the active narrow fix path and is still open/conflicting.

## What This Changes

The issue reports that `/status` should include `cacheRead` and `cacheWrite` token pricing in the displayed cost estimate when cache tokens are shown.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/65381 after rebase/proof/validation, or land an equivalent narrow fix that forwards `cacheRead` and `cacheWrite` into the existing status cost estimator with regression coverage.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists on current main: cache counters reach the visible `/status` cache line, but the cost calculation passes only input/output tokens into the cache-aware estimator.

## Solution Assessment

Yes. Reusing the existing `estimateUsageCost` contract from the status caller is the narrow maintainable fix; adding a separate status-only pricing path would duplicate existing logic.
