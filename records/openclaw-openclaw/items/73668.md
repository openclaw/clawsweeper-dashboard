## Summary

Keep open. Current main still does not implement the proposed profile archive import/export surface, and the PR remains non-mergeable because the latest sanitizer still allows path-like config keys to enter a privacy-safe archive. Earlier review context was useful: the latest head appears to have addressed prior plugin-index, dictionary-collision, and privateKey/encryptKey gaps, but one privacy contract issue plus changelog/docs discoverability gaps remain.

## What This Changes

The PR adds `openclaw profile export/import`, profile archive creation/import infrastructure, docs, CLI/command/infra tests, and adjacent gateway/plugin test fixture updates.

## Best Possible Solution

Land this feature only after the exporter consistently strips secrets and host-local path material from all portable config projections, regression tests cover path-like sanitizer misses, plugin install records remain a portable projection, the CLI index and changelog are updated, and normal PR validation passes.

## Reproduction Assessment

Yes. A unit-level `buildSafeProfileConfig` input with `plugins.entries.demo.config.sourcePath`, `mcp.servers.demo.localPath`, `workspacePath`, or `configPath` will pass through the current key filter because those normalized keys are neither exact local-path keys nor secret suffixes.

## Solution Assessment

No. The feature direction is plausible, but the current PR is not the best mergeable solution until the sanitizer uses broader local-path semantics or the shared sensitive/path filtering model, with focused regression coverage.
