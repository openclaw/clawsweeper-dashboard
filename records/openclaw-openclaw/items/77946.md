## Summary

Keep open: source review confirms the enabled-only channel startup signal is a real gap, but this patch is incomplete because plugin auto-enable still uses a separate configured-state helper that ignores `{ enabled: true }`, and the external PR lacks after-fix real behavior proof.

## What This Changes

The PR changes channel config-presence to count `{ enabled: true }` as meaningful, adds focused config-presence tests, and adds a changelog entry.

## Best Possible Solution

Converge this with https://github.com/openclaw/openclaw/issues/81323 and https://github.com/openclaw/openclaw/pull/81331 into one generic channel-presence fix that updates both configured-channel semantics and adds startup/auto-enable regression coverage.

## Reproduction Assessment

Yes, at source level: current main drops `{ enabled: true }` in `src/channels/config-presence.ts`, while Gateway startup derives configured channel owners from that helper. I did not live-run a packaged Gateway/channel setup in this read-only review.

## Solution Assessment

No, not as submitted: the PR fixes one configured-channel signal path but leaves the separate `hasMeaningfulChannelConfigShallow` auto-enable path unchanged. The safer fix is to align both helpers or centralize the contract, then cover enabled-only external channel startup.
