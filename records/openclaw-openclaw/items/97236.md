## Summary

Keep open: current main has an assistant-identity path that should resolve `agent:<id>:subagent:<uuid>` to the owning agent, but the reported literal `subagent` label is not fully isolated and overlaps with active session-name/title work.

## What This Changes

The issue reports that WebChat renders subagent sessions as `subagent` instead of the configured agent `identity.name`.

## Best Possible Solution

Add a focused current-main WebChat regression that opens a subagent session with only `identity.name` set and verifies the chat header/bubble sender uses that identity; then fix only the failing projection or UI path, aligned with the canonical session-name contract.

## Reproduction Assessment

Unclear. Source inspection shows the active chat sender name comes from `senderLabel` or `assistantName`, and `assistantName` is loaded via `agent.identity.get`; however, the current source path that would produce the literal `subagent` label was not fully isolated in this read-only review.

## Solution Assessment

Unclear. Duplicating Control UI session-list resolution directly in the chat view may be the wrong layer because the backend identity handler already resolves the owning agent for subagent keys; the safer fix should follow a focused failing WebChat/gateway regression and the session-name contract work.
