## Summary

Keep this issue open: the report is a concrete WebChat identity display bug from the latest release, and current source shows the affected UI path depends on `assistantName`, but source inspection did not establish a high-confidence current-main reproduction or an already-merged fix for the specific subagent-session case.

## What This Changes

WebChat should show the configured agent `identity.name` instead of a raw subagent label for subagent sessions and chat bubble sender names.

## Best Possible Solution

Add a focused WebChat regression that opens or renders `agent:<agentId>:subagent:<uuid>` with only `identity.name` configured, then fix the narrow owner path so the chat header, composer, avatar, and bubble sender consistently use that configured identity without duplicating session-list parsing unnecessarily.

## Reproduction Assessment

Unclear for a full current-main browser repro. The issue body gives concrete release steps and the source path shows WebChat will display a bad `assistantName`, but this read-only review did not run a gateway/browser scenario and current backend source appears intended to resolve canonical subagent keys correctly.

## Solution Assessment

Unclear until a focused repro pins the failing layer. The most maintainable fix is to prove whether `agent.identity.get` or the WebChat state/render path is wrong, then repair that canonical path rather than adding a second display-name resolver unless the backend contract is insufficient.
