## Summary

Keep open: current main has the active `agent.identity.get` path needed to resolve a subagent session to its owning agent identity, but adjacent WebChat session-label and sender-label paths still have generic subagent fallbacks, and the open session-title PR only partially overlaps the remaining naming contract.

## What This Changes

WebChat should show the configured agent identity name for subagent sessions instead of generic `subagent` labels in the chat UI and sender names.

## Best Possible Solution

Define one WebChat subagent display-name precedence that reuses server-projected session title/displayName or agent `identity.name`, then add focused regression coverage for the active chat header, session picker, and assistant sender label.

## Reproduction Assessment

Yes at source level for the generic subagent-label fallback: current main can still render a subagent session option from `subagent:<uuid>` or `Subagent:` when no row label/displayName is present. No high-confidence browser repro was established for the exact chat-bubble sender string because the active `assistantName` path appears to resolve the owning agent identity.

## Solution Assessment

Unclear. Duplicating Control UI parsing directly in the chat view may be too narrow; the best fix should align the WebChat session picker, assistant identity, and message-level sender-label paths with the canonical session title/display-name contract.
