## Label Justifications

- `P3`: The issue is a low-risk WebChat display/ergonomics problem that makes subagent identity hard to distinguish but does not break message delivery or runtime availability.
- `impact:session-state`: The symptom depends on session keys and agent/session identity metadata used to distinguish subagent sessions in WebChat.
- `maturity:stable`: The affected WebChat hosted surface maps to the M4 Stable `gateway-runtime` Hosted Web Surface category in the maturity scorecard.

## Summary

Keep open: current main still has a WebChat/session-display path that falls back to `Subagent:` when a subagent row lacks `label` or `displayName`, while the active chat sender identity path appears to resolve through `agent.identity.get`; the remaining issue is the product contract for display-name precedence, not something already implemented on main.

## What This Changes

WebChat should show the configured agent `identity.name` instead of a generic subagent label for subagent chat sessions.

## Best Possible Solution

Define one shared display-name precedence for WebChat subagent session controls, chat header, and assistant groups, add a focused regression for an `agent:<id>:subagent:<uuid>` session with only `identity.name`, and land it through the session-title work or a narrow follow-up.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Partly: source inspection shows the WebChat session selector can still display the generic `Subagent:` fallback for subagent rows without label/displayName, but I did not establish the active chat bubble path producing literal `subagent` on current main.

## Solution Assessment

Is this the best way to solve the issue? Unclear: duplicating Control UI parsing in the chat view may be the wrong layer because the backend assistant identity path already resolves agent identity from session keys; the safer fix is a shared session-display/name contract with focused WebChat regression coverage.
