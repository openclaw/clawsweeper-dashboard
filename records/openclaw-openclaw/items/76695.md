## Summary

Keep open. Current main still has the reported Control UI Skills modal path: the Skills detail dialogs call `showModal()` from a Lit ref helper without an `isConnected` guard or exception fallback.

## What This Changes

The issue reports that clicking an installed skill in the Control UI Skills page throws because `showModal()` runs before the dialog is connected.

## Best Possible Solution

Fix the Skills dialog open helper in source with a connection-aware defer or fallback, add regression coverage, and leave distributed Control UI assets to the normal build pipeline.

## Reproduction Assessment

Yes, source-reproducible: setting `detailKey` renders a Skills detail `<dialog>` with `ref(showDialogWhenClosed)`, and that helper calls `showModal()` without a connection guard. I did not mark it physically reproduced because no live browser run was allowed in this read-only sweep.

## Solution Assessment

Yes, the narrow maintainable fix is in `ui/src/ui/views/skills.ts`: defer or guard `showModal()` until the dialog is connected, then cover the lifecycle in `ui/src/ui/views/skills.test.ts`.
