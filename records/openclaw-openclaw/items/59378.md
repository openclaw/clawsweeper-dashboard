## Summary

Keep open. Current main still lacks the scoped working-memory lane and `/context` memory-layer reporting, so the PR is not obsolete, but the branch is not mergeable as-is because it mixes scoped working-memory files into bootstrap accounting and leaves public config/changelog surfaces out of sync.

## What This Changes

The PR adds an explicit `.openclaw/working-memory/...` lane for isolated cron and heartbeat runs, threads it through runner, CLI, config, and protocol surfaces, updates `/context` memory-layer reporting, and adds tests and docs.

## Best Possible Solution

Land a corrected, explicitly opt-in core implementation only if maintainers want this as core behavior: scoped files should stay workspace-contained, show up only in their own memory lane, leave durable memory/search/recall boundaries clear, and keep config schema, docs, tests, and changelog aligned.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: apply the PR, use a bootstrap file whose basename matches `.openclaw/working-memory/.../<same-name>.md`, and `buildBootstrapInjectionStats` can report the bootstrap file as injected from the scoped file because of basename fallback.

## Solution Assessment

No, not as-is. The direction is plausible, but the safer implementation is to pass only bootstrap context files into bootstrap stats/reporting, keep scoped memory in the dedicated `memory.working` report path, and update the generated config/docs/changelog surfaces before merge.
