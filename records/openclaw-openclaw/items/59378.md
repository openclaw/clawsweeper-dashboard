## Summary

Keep open. Current main still lacks scoped working-memory lanes, and this PR is not mergeable as-is because scoped memory is mixed into bootstrap accounting while generated config/schema and changelog surfaces are not aligned.

## What This Changes

The PR adds explicit `.openclaw/working-memory/...` support for isolated cron and heartbeat runs, reports memory lanes in `/context`, filters durable memory/recall tools, and updates related docs, protocol, config, and tests.

## Best Possible Solution

Land this only after scoped files are accounted separately from bootstrap files, generated config/docs/changelog surfaces are updated, and maintainers confirm this belongs in core rather than a plugin workflow.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: apply the PR, use a bootstrap file such as `notes.md` and scoped `.openclaw/working-memory/cron/notes.md`, and the basename fallback in `buildBootstrapInjectionStats` can attribute scoped content to the bootstrap file.

## Solution Assessment

No, not as-is. The safer fix is to pass only bootstrap context files into bootstrap stats/reporting, keep scoped working memory in the dedicated `memory.working` lane, and regenerate the public config surfaces plus changelog before merge.
