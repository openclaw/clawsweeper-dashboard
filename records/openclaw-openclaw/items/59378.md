## Summary

Keep open: current main still does not implement scoped working-memory lanes, and this PR contains unique feature work, but the patch should not merge as-is because scoped memory is fed into bootstrap accounting, the new cron payload field is not preserved by the Control UI, and the external PR lacks real after-fix behavior proof.

## What This Changes

The PR adds `.openclaw/working-memory/...` scoped memory injection for cron/heartbeat runs, memory lane reporting in `/context`, tool filtering, docs, config/protocol plumbing, CLI flags, and regression tests.

## Best Possible Solution

Land the feature only after scoped files are accounted outside bootstrap stats, all cron/UI/config surfaces preserve the new field, real cron or heartbeat proof is added, and maintainers accept the core product direction.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: a bootstrap `notes.md` plus `.openclaw/working-memory/cron/notes.md` can be misattributed through the basename fallback, and the Control UI drop is source-reproducible from its payload type and replacement payload builder.

## Solution Assessment

No, not as-is. The safer path is to keep scoped working memory in its own lane, preserve the new cron payload field across UI/protocol surfaces, and require real behavior proof plus maintainer product acceptance before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Keep scoped memory out of bootstrap stats:** `src/agents/pi-embedded-runner/run/attempt.ts:409-412`
  - body: The PR appends the scoped working-memory file to `contextFiles` and then passes that combined array into `buildBootstrapInjectionStats`. That helper falls back by basename, so `.openclaw/working-memory/.../notes.md` can make an unrelated bootstrap `notes.md` look injected or truncated from scoped content; pass only bootstrap context files to bootstrap stats and keep scoped memory in its own lane.
  - confidence: 0.92
- **[P2] Preserve working memory in Control UI edits:** `src/cron/types.ts:102`
  - body: Adding `workingMemoryPath` as a persisted cron payload field without updating the Control UI means the UI still builds replacement `agentTurn` payloads that have no working-memory property. Editing a scoped-memory cron job in the web UI will silently clear the path, so wire or preserve the field there before exposing it.
  - confidence: 0.86
