## What This Changes

The PR adds a Telegram reaction path that starts a synthetic heartbeat session after reaction system events are enqueued, adds a focused unit test, and edits CHANGELOG.md.

## Best Possible Solution

Keep reaction notifications queue-only by default, add or align with a maintainer-approved opt-in wake policy, cover upgrade behavior, and require live Telegram proof before merge.

## Reproduction Assessment

Yes, source inspection reproduces the current gap: accepted Telegram reactions enqueue system events and return without waking a session. The PR's added test is mocked and does not reproduce the live Telegram transport path.

## Solution Assessment

No. The useful gap is real, but unconditional wake tied to reactionNotifications is not the narrowest maintainable solution because current docs describe notifications as enqueue-only and related work points toward an opt-in wake policy.
