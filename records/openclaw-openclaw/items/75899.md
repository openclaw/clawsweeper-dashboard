## Summary

Keep open. The PR targets a real Telegram reaction wake-up gap, but the proposed synthetic turn reuses the same inbound message id for every reaction on a Telegram message, so later reactions on that message can be skipped by inbound dedupe before their queued system events drain.

## What This Changes

The PR imports `HEARTBEAT_PROMPT`, calls `processMessage` after Telegram reaction system events are enqueued, and adds a Telegram bot unit test for the synthetic session trigger.

## Best Possible Solution

Keep the reaction wake-up direction, but key each synthetic turn by a per-update or per-reaction discriminator and lock it with focused two-reaction coverage plus a changelog entry.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is two authorized Telegram reactions on the same chat/message: the first synthetic turn commits the `reaction:<chatId>:<messageId>` dedupe key, and the second is skipped before its queued event drains.

## Solution Assessment

No. Triggering a synthetic session after enqueueing is a reasonable narrow approach, but the proposed `messageIdOverride` is not unique enough for OpenClaw's inbound dedupe contract.
