## Summary

Keep open. The PR targets a real Telegram reaction wake-up gap, but the synthetic turn uses a non-unique message id per reacted Telegram message, so subsequent reactions on that same message can be dropped by inbound dedupe before their queued system events drain.

## What This Changes

The PR adds a Telegram reaction follow-up that runs `processMessage` with `HEARTBEAT_PROMPT` after enqueueing reaction system events, plus a unit test asserting the synthetic session is triggered.

## Best Possible Solution

Keep the narrow reaction wake-up direction, but make each synthetic wake id unique per reaction/update and cover two reactions on the same message with a regression test and changelog entry.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is two authorized Telegram reactions on the same chat/message: the first synthetic turn commits the `reaction:<chatId>:<messageId>` dedupe key, and the second is skipped before its queued event drains.

## Solution Assessment

No. Triggering a synthetic session after enqueueing is a reasonable narrow approach, but the proposed `messageIdOverride` is not unique enough for OpenClaw's inbound dedupe contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Use a unique synthetic message id for each reaction wake:** `extensions/telegram/src/bot-handlers.runtime.ts:1017`
  - body: The synthetic turn always uses `reaction:${chatId}:${messageId}` as `MessageSid`. Inbound dedupe keys include `MessageSid`, so after one reaction to a message is processed, any later reaction on that same message is treated as a duplicate and returns before draining the newly queued system event. Include the update id, reacting user, reaction key, or another per-event discriminator and add a two-reaction regression.
