## Summary

Keep open. The PR targets a valid Telegram reaction wake-up bug, but the branch is not merge-ready because its fixed synthetic MessageSid can cause later reactions on the same Telegram message to be skipped by inbound dedupe before their queued system events drain.

## What This Changes

The PR adds a Telegram reaction follow-up that invokes processMessage with HEARTBEAT_PROMPT after enqueuing reaction system events and adds a unit test for the wake-up.

## Best Possible Solution

Keep the synthetic heartbeat wake-up direction, but make the synthetic inbound id unique per reaction/update and add regression coverage plus a changelog entry.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is two authorized message_reaction updates for the same chat/message: the first run commits the fixed reaction MessageSid, and the second is skipped as duplicate before its newly queued event drains.

## Solution Assessment

No. Triggering a synthetic session after enqueueing is a narrow maintainable approach, but the current MessageSid override is not unique enough for OpenClaw's inbound dedupe contract.
