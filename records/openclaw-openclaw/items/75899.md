## Summary

Keep open: the PR targets a valid Telegram reaction wake-up gap, but its fixed synthetic `MessageSid` would cause later reactions on the same Telegram message to be skipped by inbound dedupe before the queued event drains.

## What This Changes

The PR imports `HEARTBEAT_PROMPT`, runs `processMessage` with a synthetic heartbeat after Telegram reaction events are enqueued, and adds a Telegram bot test for that wake-up.

## Best Possible Solution

Telegram reaction events should wake a synthetic heartbeat turn immediately, with a per-update or per-reaction inbound id that cannot collide with later reactions on the same Telegram message.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is two authorized reactions on the same chat/message: the first synthetic turn commits `reaction:${chatId}:${messageId}`, and the second wake is treated as duplicate before processing its newly queued event.

## Solution Assessment

No. The synthetic heartbeat direction is narrow and maintainable, but the proposed id must include a per-event discriminator such as Telegram `update_id`, reacting user, or reaction key, plus a two-reaction regression test.
