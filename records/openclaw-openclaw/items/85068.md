## Summary

Keep open: current main and v2026.6.6 still lack a foreground partial-WIP/action-debt finalizer or TaskFlow state. Related foreground-fence work covers one delivery-race subproblem, but not the broader timeout/supersession-after-writes incident report.

## What This Changes

Request a foreground direct-channel lifecycle guard that reports partial WIP when a Telegram/direct turn times out or is superseded after file or tool writes.

## Best Possible Solution

Add a generic, privacy-safe foreground-run incident mechanism that records write/tool activity, detects timeout or supersession without a terminal report, and emits a best-effort partial-WIP notice plus bounded state metadata.

## Reproduction Assessment

No live high-confidence reproduction was run. Source inspection shows current main lacks the requested partial-WIP/action-debt states and still has foreground delivery suppression paths, but the private Telegram WIP scenario needs a redacted reproducer or focused integration test.

## Solution Assessment

Unclear. A finalizer/supervisor rule is plausible, but maintainers should choose the durable owner boundary before implementation because the fix crosses foreground reply delivery, task/flow state, dirty-worktree metadata, and detached-controller policy.
