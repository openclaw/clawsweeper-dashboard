## Summary

Keep open: current main has foreground freshness fencing, Telegram reply fencing, lane timeouts, and detached task tracking, but it still lacks a foreground partial-WIP incident/finalizer or TaskFlow status that would reliably notify the user after timeout or supersession with unreported work.

## What This Changes

The issue asks OpenClaw to detect foreground direct-channel turns that time out or are superseded after write/tool activity and report partial, unverified WIP to the user and task/status surfaces.

## Best Possible Solution

Add a generic foreground-run lifecycle incident hook that records write/tool activity, detects timeout or supersession without a terminal report, emits a redacted partial-WIP notice, and stores minimal state without changing detached task semantics.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection shows the relevant foreground suppression and missing partial-WIP state surfaces, but the private Telegram/release scenario still needs a redacted repro or focused integration test.

## Solution Assessment

Unclear. The requested direction is plausible, but maintainers should first decide whether the durable state belongs in foreground reply finalization, TaskFlow/Taskdash incident state, or detached-controller guidance.
