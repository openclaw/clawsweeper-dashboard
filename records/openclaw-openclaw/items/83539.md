## What This Changes

The PR makes memory-wiki lint ignore wikilink-shaped text in Markdown non-link regions, resolve Obsidian links by page title and aliases, and adds focused regression tests.

## Best Possible Solution

Keep the title/alias resolver direction, but replace the ad hoc Markdown masking with delimiter-aware parsing or helpers that handle matching fence and code-span lengths, then prove the CLI against a sample or redacted vault.

## Solution Assessment

No, not as currently implemented. The fix direction is right, but the Markdown stripping needs delimiter-aware handling or a parser-backed helper before this is the maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Handle matching-length fenced code blocks:** `extensions/memory-wiki/src/markdown.ts:110`
  - body: Markdown allows longer fences, such as four backticks around content that contains triple backticks. This regex matches the first three backticks of a four-backtick opener but cannot match the four-backtick closer, so it removes the rest of the document and hides real wikilinks after the block from lint.
  - confidence: 0.92
- **[P2] Handle multi-backtick inline code spans:** `extensions/memory-wiki/src/markdown.ts:111`
  - body: Valid inline code can use two or more backticks. The new pattern only strips single-backtick spans, so ``[[literal]]`` leaves `[[literal]]` searchable and still reports a false broken-wikilink warning for code text.
  - confidence: 0.9
