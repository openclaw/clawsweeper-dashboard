## Summary

Keep open: current main still has no zero-LLM cron exec payload, and the live discussion plus the closed unmerged implementation show this is an active core feature/security design rather than cleanup-ready stale work.

## What This Changes

Requests a direct cron execution mode that runs scheduled shell/script commands, captures output, and optionally delivers it to a channel without starting an agentTurn LLM session.

## Best Possible Solution

Keep this as the canonical core tracker and design a hardened additive cron exec contract across protocol/schema, validation, runtime dispatch, run history, delivery, CLI, Control UI, docs, tests, and host-command security policy.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a new cron capability request. Current-main source inspection gives high confidence that the direct exec path is absent: payloads are limited to systemEvent/agentTurn and isolated jobs still route through the agent runner.

## Solution Assessment

Unclear until maintainers settle the API and security policy. A native cron exec payload is a plausible seam, but the maintainable solution needs coordinated Gateway, CLI/UI, delivery, observability, and secops follow-through rather than a narrow spawn branch alone.

## Security Review

Status: needs_attention

Summary: The requested feature is security-sensitive because it would persist unattended host-command execution outside a live agent turn.

Concerns:

- **[medium] Define persisted cron exec policy:** `docs/tools/exec-approvals.md:55`
  - body: Stored cron commands need explicit rules for approvals, sandboxing, cwd/env binding, shell choice, output retention, cancellation, timeout, retry, and delivery before implementation.
  - confidence: 0.92
- **[medium] Keep cron validation under secops review:** `.github/CODEOWNERS:38`
  - body: Cron job validation is codeowned by openclaw/openclaw-secops, and allowing a new payload to execute host commands changes that validation boundary.
  - confidence: 0.9
- **[medium] Account for exec approval bypass risk:** `docs/security/THREAT-MODEL-ATLAS.md:252`
  - body: The security threat model calls out exec approval bypass and command allowlist manipulation as high residual risks, so a zero-LLM scheduled command path needs hardened policy rather than an unchecked spawn path.
