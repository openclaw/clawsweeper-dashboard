## Summary

Keep open. Current main still has no zero-LLM cron exec payload: the cron type/schema contract only accepts `systemEvent` and `agentTurn`, isolated/current/session validation still requires `agentTurn`, detached runtime still calls the isolated agent runner, and the CLI/UI/docs expose only the agent-mediated path. The issue discussion also shows this is the canonical core tracker after duplicate requests and a closed unmerged implementation PR, with several recent production datapoints plus an external shim workaround that does not replace OpenClaw cron observability and delivery.

## What This Changes

Add a direct cron exec payload or CLI mode that runs scheduled shell commands without creating an LLM agent turn and can deliver captured output through OpenClaw cron delivery.

## Best Possible Solution

Keep this as the canonical core cron tracker and land an additive, owner-reviewed command payload that updates cron types, Gateway protocol validation, job normalization/validation, scheduler dispatch, run history/accounting, delivery, CLI, Control UI, docs, and focused tests, with a documented host-command security model before merge.

## Reproduction Assessment

Yes. A high-confidence reproduction path is to validate or submit a cron job with `payload.kind="exec"` on current main; the protocol schema rejects it, job validation requires `agentTurn` for isolated-like targets, and detached execution has no direct exec branch.

## Solution Assessment

Yes for the requested direction, but not as an unsupervised quick patch. A core cron payload is the maintainable solution because ClawHub plugins or the posted shim cannot own OpenClaw's persisted scheduler, run history, retry accounting, and delivery contract; the exact policy needs maintainer and secops review.
