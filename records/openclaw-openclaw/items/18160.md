## Summary

Keep open: current main still has no zero-LLM cron exec payload, and recent discussion plus duplicate closures treat this as the canonical tracker for the remaining core/product decision.

## What This Changes

Request to add a direct cron `exec` payload that runs scheduled commands without an agent/LLM turn while preserving cron observability and optional channel delivery.

## Best Possible Solution

Keep this as the canonical design tracker; if maintainers choose core support, land an additive owner/secops-approved cron exec/script payload with protocol, validation, runtime, delivery, CLI, UI, docs, tests, and host-command policy together.

## Reproduction Assessment

Not applicable for a feature request. Source inspection on current main gives high confidence that cron only supports `systemEvent` and `agentTurn`, so the requested direct exec path is absent rather than a broken existing contract.

## Solution Assessment

Unclear until maintainers settle the core-vs-plugin and security policy decision. A native cron exec payload is a plausible maintainable seam for preserving OpenClaw observability and delivery, but it should not be automated without owner/secops design.
