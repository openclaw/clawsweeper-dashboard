## Summary

Keep open: current main still lacks a zero-LLM cron exec payload, this issue remains the canonical tracker for direct scheduled shell execution, and the requested persisted host-command mode needs product and security design before implementation.

## What This Changes

Requested an additive cron payload mode that runs a scheduled command directly, captures stdout/stderr, and optionally delivers output without starting an LLM-backed agent turn.

## Best Possible Solution

Keep this as the canonical core tracker and design an additive direct-exec cron contract covering protocol/schema, validation, runtime dispatch, run history, delivery, CLI, Control UI, docs, tests, and explicit host-command security policy.

## Reproduction Assessment

Not applicable for a feature request. Source inspection on current main gives high confidence that cron only supports systemEvent and agentTurn, so the requested direct exec path is absent rather than a broken existing contract.

## Solution Assessment

Unclear until maintainers settle the API and security policy. A native cron exec payload is a plausible maintainable seam, but it needs Gateway, UI/CLI, delivery, observability, and secops follow-through rather than a narrow child_process branch alone.
