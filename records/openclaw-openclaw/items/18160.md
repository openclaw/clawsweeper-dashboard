## Summary

Keep open: current main and v2026.5.12 still have no zero-LLM cron exec payload, and the newer direct tool-invoke work is an adjacent operator surface rather than scheduled cron delivery. The prior cron exec implementation was closed unmerged with explicit maintainer/security concerns, so this remains a product and secops design item rather than cleanup-close material.

## What This Changes

Add a direct cron exec payload and related API/CLI/UI support so scheduled shell commands can run without an LLM turn and optionally deliver output through OpenClaw channels.

## Best Possible Solution

Keep this as the canonical cron tracker and design a hardened additive cron exec contract that reuses the approved direct-tool/exec policy path while updating cron types, Gateway schema, validation, runtime dispatch, run history, retry/failure accounting, delivery, CLI, Control UI, docs, and tests together.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source inspection gives high confidence that current main and v2026.5.12 do not accept or execute a direct cron exec payload.

## Solution Assessment

Unclear until maintainers settle the API and security policy. A native cron exec payload is plausible, but it should reuse the approved direct-tool/exec boundary rather than adding an independent process-spawn path.
