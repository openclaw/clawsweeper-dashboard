## Label Justifications

- `impact:security`: The requested mode would add persisted unattended host-command execution to scheduled cron jobs.

## Summary

Keep open: current main and v2026.5.22 still do not provide a zero-LLM cron payload that runs a command, records cron state/history, and delivers stdout/stderr through the cron delivery path. The request remains a high-signal feature and security/product decision, with the closed cron exec PR unmerged and related direct-tool PRs still covering a different underlying surface.

## What This Changes

Add a cron payload mode, such as payload.kind="exec", for scheduled command execution without an agentTurn or LLM call, including stdout/stderr capture and optional channel delivery.

## Best Possible Solution

Keep this as the canonical cron direct-exec tracker and design a hardened additive cron payload that reuses the shared tool invocation and exec-approval boundary while updating cron protocol, runtime, delivery, CLI, UI, docs, and focused tests together.

## Reproduction Assessment

Yes, as source-level reproduction of a missing feature rather than a strict bug repro. Current main and v2026.5.22 still constrain cron payloads, schema, validation, runtime, CLI, UI, and docs to systemEvent or agentTurn paths.

## Solution Assessment

Unclear until maintainers settle the cron API and security policy. A native additive payload is plausible, but the maintainable solution should reuse the shared tool invocation and exec-approval boundary rather than adding an independent spawn path.
