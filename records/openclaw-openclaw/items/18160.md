## Summary

Keep open: current main and the latest release still only support agent-mediated cron execution, and the prior exec-payload implementation was closed unmerged pending an owner-reviewed security and behavior design.

## What This Changes

Request an additive cron payload mode that runs scheduled shell commands directly and delivers their output without invoking an LLM agent turn.

## Best Possible Solution

Keep this as the canonical core tracker and design a hardened additive cron exec contract across types, Gateway schema, validation, runtime dispatch, run history, delivery, CLI, Control UI, docs, tests, and host-command security policy.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source inspection on current main gives high confidence that the requested direct exec cron path is absent.

## Solution Assessment

Unclear until maintainers settle the API and security policy. A native cron exec payload is a plausible seam, but the maintainable solution needs coordinated protocol, CLI/UI, delivery, observability, and secops follow-through rather than a narrow process-spawn branch alone.

## Security Review

Status: needs_attention

Summary: Direct cron exec would persist unattended host-command execution outside a live agent turn, so it needs explicit security policy before implementation.
