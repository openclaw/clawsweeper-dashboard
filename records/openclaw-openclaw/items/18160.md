## Label Justifications

- `P2`: This is a normal-priority feature with strong production pain, but it requires product and security review rather than emergency handling.
- `impact:security`: The requested mode would add persisted unattended host-command execution to scheduled cron jobs.
- `impact:message-loss`: Reported cron timeouts prevent deterministic jobs from producing their expected scheduled channel output.

## Summary

Keep open: current main still lacks a cron payload, Gateway schema, runtime branch, CLI/UI surface, or docs path for running scheduled commands without an agent/LLM turn; the newer direct tool-invoke work is adjacent but does not persist or dispatch cron jobs.

## What This Changes

The issue requests a direct cron exec payload and related cron surfaces so deterministic scripts can run on schedule and deliver output without an LLM-mediated agent turn.

## Best Possible Solution

Keep this as the canonical cron direct-exec tracker and design a hardened additive cron payload that reuses the shared tool invocation and exec-approval boundary while updating protocol, runtime, delivery, CLI, UI, docs, and tests together.

## Reproduction Assessment

Yes for source-level capability absence, not as a bug reproduction: current main has no cron payload shape, schema, validation path, runtime branch, CLI flag, UI state, or docs for direct exec payloads.

## Solution Assessment

Unclear until maintainers settle the cron API and unattended-exec security policy. A native payload is plausible, but the maintainable direction should reuse the shared tool invocation and exec approval boundary instead of adding an independent spawn path.
