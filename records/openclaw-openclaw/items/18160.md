## Summary

Keep open: current main and v2026.5.12 still restrict cron payloads to `systemEvent` and `agentTurn`, while this issue is the canonical, actively discussed core feature/security design for zero-LLM scheduled commands.

## What This Changes

Add a direct cron exec/script payload that runs a scheduled command without an LLM turn, captures stdout/stderr, and delivers output through OpenClaw cron channels.

## Best Possible Solution

Keep this as the canonical core tracker and design a hardened additive cron exec contract across protocol/schema, validation, runtime dispatch, run history, delivery, CLI, Control UI, docs, tests, and host-command security policy.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a new capability request. Source inspection on current main and v2026.5.12 gives high confidence that the direct exec cron path is absent, not broken existing behavior.

## Solution Assessment

Unclear until maintainers settle the API and security policy. A native cron exec payload is a plausible seam, but the maintainable solution needs coordinated Gateway, UI/CLI, delivery, observability, and secops follow-through rather than a narrow child-process branch alone.

Status: needs_attention
