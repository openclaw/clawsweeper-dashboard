## Label Justifications

- `impact:message-loss`: The original reports described agent-mediated cron timeouts preventing deterministic scheduled outputs from reaching channels.

## Summary

Current main already implements the central zero-LLM cron command-job request as `payload.kind: "command"`, with CLI/API/docs, runtime execution, delivery, diagnostics, and migration coverage; the exact payload name differs from the proposed `exec`, but the user problem is solved and shipped.

## What This Changes

Request for a direct cron execution mode that runs deterministic scripts on schedule, captures stdout/stderr, and delivers output without starting an LLM-backed agent turn.

## Best Possible Solution

Close this issue and keep the shipped `payload.kind: "command"` cron implementation as the native zero-LLM path; any remaining refinements such as richer UI authoring or policy tuning should be tracked in narrower follow-ups.

## Reproduction Assessment

Not applicable as a bug reproduction: this was a feature request, not a broken documented contract. Source and history show the formerly missing zero-LLM cron command path now exists on current main and in release provenance.

## Solution Assessment

Yes. The merged `payload.kind: "command"` path is the maintainable core solution because it preserves cron scheduling/history/delivery while avoiding an agent/model turn; the naming differs from the requested `exec` but covers the central behavior.
