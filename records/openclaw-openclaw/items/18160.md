## Summary

Keep open: current main still lacks a zero-LLM cron exec payload, this issue remains the canonical tracker after duplicate reports were consolidated here, and the requested persisted host-command mode needs owner/secops design before implementation.

## What This Changes

Add a direct cron payload mode that runs a configured shell command on schedule, captures output, updates cron run state, and optionally delivers stdout without starting an agent turn or LLM call.

## Best Possible Solution

Keep this as the canonical core tracker and design an additive cron exec contract with protocol/schema, validation, runtime dispatch, run history, delivery, CLI, Control UI, docs, tests, and an explicit host-command security policy.

## Reproduction Assessment

Not applicable for a feature request. Source inspection on current main gives high confidence that cron only supports systemEvent and agentTurn, so the requested direct exec path is absent rather than a broken existing contract.

## Solution Assessment

Unclear until maintainers settle the API and security policy. A native cron exec payload is a plausible maintainable seam, but a simple child_process branch from the closed PR is not enough without Gateway, UI/CLI, delivery, observability, and secops follow-through.
