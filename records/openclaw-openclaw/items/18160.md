## Label Justifications

- `P2`: This is a high-signal feature request with repeated production pain, but it is not an emergency and requires product/security review before implementation.
- `impact:security`: The requested mode would add persisted unattended host-command execution to scheduled cron jobs.
- `impact:message-loss`: Reported cron timeouts and agent-mediated failures prevent deterministic scheduled jobs from producing expected channel output.

## Summary

Keep open: current main and the latest release still do not provide a zero-LLM cron payload for direct command execution, and the active /tools/invoke work is a possible lower-level mechanism rather than the cron scheduler, run-history, retry, and delivery surface requested here.

## What This Changes

Request to add a direct exec cron payload that runs scheduled commands without an LLM turn and routes captured output through OpenClaw cron delivery and run history.

## Best Possible Solution

Keep this as the canonical cron direct-exec tracker and design a hardened additive payload that reuses the shared tools.invoke/exec-approval boundary while updating cron types, protocol schemas, validation, scheduler dispatch, run history, retry/failure accounting, delivery, CLI, Control UI, docs, and focused tests together.

## Reproduction Assessment

Yes for source-level capability absence, not as a strict bug reproduction. Current main has no cron payload type, Gateway schema branch, validation path, runtime branch, CLI option, UI option, or agent cron-tool schema for direct exec payloads.

## Solution Assessment

Unclear until maintainers settle the cron API and host-command security policy. A native payload is plausible, but the maintainable solution should reuse the shared tools.invoke and exec-approval boundary instead of adding an independent spawn path.
