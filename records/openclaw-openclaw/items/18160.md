## Summary

Keep open: current main and the latest shipped release still lack a zero-LLM cron exec payload, and the active discussion plus the closed unmerged implementation show this remains a core product/security design item rather than cleanup-ready stale work.

## What This Changes

Request to add a cron payload that runs scheduled shell commands directly, captures output, and delivers it without an LLM agentTurn.

## Best Possible Solution

Keep this as the canonical cron exec tracker and design a hardened additive contract across types, Gateway schema, validation, runtime dispatch, run history, delivery, CLI, Control UI, docs, tests, and host-command security policy.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source inspection on current main gives high confidence that the requested direct exec path is absent.

## Solution Assessment

Unclear until maintainers settle the API and security policy. A native cron exec payload is a plausible seam, but the maintainable solution needs coordinated Gateway, CLI/UI, delivery, observability, and secops follow-through rather than a narrow child_process branch alone.

## Security Review

Status: needs_attention

Summary: Direct cron exec would persist unattended host-command execution outside a live agent turn, so it needs explicit security policy before implementation.

Concerns:

- **[medium] Define unattended cron exec policy:** `docs/tools/exec-approvals.md:55`
  - body: Stored cron commands need explicit rules for approvals, sandboxing, host routing, cwd/env binding, shell choice, output retention, cancellation, timeout, retry, and delivery before implementation.
  - confidence: 0.92
- **[medium] Preserve exec approval and sandbox boundaries:** `docs/security/THREAT-MODEL-ATLAS.md:252`
  - body: The threat model already tracks exec approval bypass and unauthorized command execution as high-risk areas, so a zero-LLM scheduled command path must not bypass the existing approval, audit, and sandbox expectations.
  - confidence: 0.9

  - body: CODEOWNERS routes cron validation-adjacent code to the security owner group, and adding a payload that executes host commands changes that validation boundary.
