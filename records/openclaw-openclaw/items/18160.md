## Summary

Keep open: current main and the latest release still lack a zero-LLM cron exec payload, while related direct-invoke work is an adjacent operator surface and the prior cron implementation was closed unmerged for design, validation, and security reasons.

## What This Changes

Request an additive cron payload mode that runs a command directly on schedule, captures output, and optionally delivers it to a channel without starting an agent turn.

## Best Possible Solution

Keep this as the canonical cron tracker and design a hardened additive cron exec contract that reuses the approved direct-tool/exec policy path while updating protocol, runtime, delivery, CLI, UI, docs, and tests together.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a bug reproduction because this is a feature request, but source inspection gives high confidence that current main and v2026.5.12 cannot accept or execute a direct cron exec payload.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers settle the API and security policy. A native cron exec payload is plausible, but it should reuse the direct-tool/exec boundary rather than adding an independent process-spawn path.

## Security Review

Status: needs_attention

Summary: Direct cron exec would persist unattended host-command execution outside a live agent turn, so it needs explicit security review before implementation.

Concerns:

- **[medium] Define unattended cron exec policy:** `docs/tools/exec-approvals.md:55`
  - body: Stored cron commands need explicit rules for approvals, sandboxing, host routing, cwd/env binding, shell choice, output retention, cancellation, timeout, retry, audit, and channel delivery before implementation.
  - confidence: 0.92
- **[medium] Preserve dangerous-tool boundaries:** `src/security/dangerous-tools.ts:9`
  - body: The current Gateway HTTP dangerous-tool list treats exec as default-denied RCE; a cron exec payload must preserve explicit opt-in, owner semantics, auditability, and fail-closed behavior.
  - confidence: 0.9
- **[medium] Keep cron validation under security review:** `.github/CODEOWNERS:38`
  - body: CODEOWNERS routes cron service validation to the security owner group, and adding a payload that launches host commands changes that validation boundary.
  - confidence: 0.86
- **[medium] Handle headless approval failures explicitly:** `src/agents/bash-tools.exec-host-shared.ts:394`
  - body: Current exec policy already treats cron as unable to wait for interactive approval; a cron exec payload must specify fail-closed behavior for approval-required commands instead of silently widening execution.
  - confidence: 0.88
