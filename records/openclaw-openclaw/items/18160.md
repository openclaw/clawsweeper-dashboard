## Summary

Keep open: current main still does not provide a zero-LLM cron exec payload, and the closed implementation PR did not land. The existing cron contract, Gateway validator, runtime dispatch, CLI, Control UI, and docs all still expose only systemEvent or agentTurn cron execution, while the discussion adds multiple production datapoints and an interim external shim rather than resolving the core request.

## What This Changes

Add a cron payload mode that runs configured shell commands directly on schedule, captures stdout/stderr, and delivers results without starting an LLM agent turn.

## Best Possible Solution

Keep this as the canonical core tracker and land an additive, owner-reviewed cron exec payload with protocol/schema, runtime, run-history, delivery, CLI, Control UI, docs, tests, and explicit secops policy alignment.

## Reproduction Assessment

Not applicable for a feature request. Source inspection gives high confidence that current main only supports systemEvent and agentTurn cron payloads, so the requested exec path is absent rather than a failing existing contract.

## Solution Assessment

Yes in direction, but not as an automated cleanup patch yet. A native exec payload is the maintainable place to preserve cron observability and channel delivery, but it needs an explicit host-command security/product design before implementation.
