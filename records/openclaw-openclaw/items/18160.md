## Label Justifications

- `P2`: This was a normal-priority product enhancement with real reliability and cost impact, now addressed by the shipped command cron path.
- `impact:security`: The issue centers on persisted scheduled host command execution, which is security-sensitive even when implemented as admin-authored Gateway automation.
- `impact:message-loss`: Reporter and comments describe scheduled jobs failing or timing out before their intended channel delivery.

## Summary

Current main and the v2026.6.5 release now provide first-class zero-LLM cron command jobs, including schema support, CLI/API authoring, direct Gateway execution, stdout/stderr capture, delivery handling, timeout/output bounds, and regression coverage.

## What This Changes

The issue requests a direct cron execution payload for scheduled shell/script jobs that should run without an agent/model turn and deliver bounded output through OpenClaw channels.

## Best Possible Solution

Keep the shipped command cron implementation as the canonical zero-LLM scheduled command path, and leave broader direct tool invocation or elevated exec scoping to their separate trackers.

## Reproduction Assessment

No. The old source-level limitation no longer reproduces on current main because cron payloads, Gateway schema, CLI, runtime dispatch, and docs now support command jobs without an agent/model turn.

## Solution Assessment

Yes. The shipped kind="command" argv-based design is a narrower and safer form of the requested direct exec mode than a raw string exec payload, while preserving cron history, delivery, output bounds, and timeout semantics.
