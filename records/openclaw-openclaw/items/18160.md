## Label Justifications

- `P2`: The request addresses real scheduled automation reliability and cost pain but needs design and security review before implementation.
- `impact:security`: The feature would add persisted unattended host-command execution to scheduled jobs.
- `impact:message-loss`: The reported failures include scheduled jobs timing out before their expected channel output can be delivered.

## Summary

Keep open: current main and v2026.5.12 still lack a zero-LLM cron exec payload, the prior cron exec implementation was closed unmerged for design/security/validation reasons, and the open direct tool-invocation work is adjacent rather than a scheduled cron delivery solution.

## What This Changes

Request an additive cron payload mode that runs scheduled shell commands directly and can deliver stdout without creating an agent turn.

## Best Possible Solution

Keep this as the canonical cron direct-exec tracker and design a hardened additive cron payload contract that reuses the approved exec/direct-tool boundary while updating protocol, scheduler runtime, delivery, CLI, UI, docs, and tests together.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for source-level capability absence: current main and v2026.5.12 have no cron payload shape, schema, validation path, runtime branch, CLI flag, or UI state for a direct exec payload.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers settle the cron API and security policy; a native payload is plausible, but it should reuse the direct-tool/exec boundary instead of adding an independent spawn path.
