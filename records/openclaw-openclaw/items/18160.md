## Label Justifications

- `P2`: This is a normal-priority feature with strong production pain but it needs product and security review before implementation.
- `impact:security`: The requested cron mode would add persisted unattended host-command execution to scheduled jobs.
- `impact:message-loss`: Reported cron failures include deterministic jobs timing out before their expected channel output can be delivered.

## Summary

Keep open: current main still lacks a zero-LLM cron exec payload, the earlier implementation was closed unmerged for design/security/validation reasons, and the merged direct-tool RPC foundation is adjacent rather than a scheduled cron delivery solution.

## What This Changes

The request is to add a cron payload mode that runs a scheduled command directly, captures stdout/stderr, and optionally delivers the result without creating an LLM agent turn.

## Best Possible Solution

Keep this as the canonical cron direct-exec tracker and design a hardened additive cron payload contract that reuses the Gateway tools.invoke/exec approval boundary while updating protocol, runtime, delivery, CLI, UI, docs, and tests together.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a bug repro, but yes for source-level capability absence: current main has no cron payload shape, schema, validation path, runtime branch, CLI flag, UI state, or docs for a direct exec payload.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers settle the cron API and security policy; a native payload is plausible, but it should reuse the shared tools.invoke/exec boundary instead of adding an independent spawn path.

## Risks / Open Questions
