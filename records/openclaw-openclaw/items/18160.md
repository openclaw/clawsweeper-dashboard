## Summary

Keep open: current main and v2026.5.12 still lack a zero-LLM cron exec payload, the prior cron implementation PR was closed unmerged, and the newer direct tool-invoke work is an adjacent operator surface rather than scheduled cron delivery.

## What This Changes

Request adds a direct cron exec payload or CLI mode that runs scheduled shell commands without starting an agent turn and keeps cron tracking and delivery.

## Best Possible Solution

Keep this as the canonical cron tracker and design a hardened additive cron exec contract that reuses the approved exec policy/direct-tool boundary while updating protocol, runtime, delivery, CLI, UI, docs, and tests together.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a bug repro, but source inspection gives high confidence that current main and v2026.5.12 cannot accept or execute a direct cron exec payload.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers settle the cron API and security policy; a native payload is plausible, but it should reuse the direct-tool/exec boundary instead of adding an independent spawn path.

Summary: Direct cron exec would persist unattended host-command execution outside an interactive agent turn, so it needs explicit security review before implementation.
