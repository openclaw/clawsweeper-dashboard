## Label Justifications

- `P2`: This is a high-signal feature request with real production pain, but it is not an emergency and needs product/security review before implementation.
- `impact:security`: The requested mode would add persisted unattended host-command execution to scheduled cron jobs.
- `impact:message-loss`: Reported cron timeouts prevent deterministic scheduled jobs from producing their expected channel output.

## Summary

Keep open: current main and the latest shipped release still lack a zero-LLM cron payload that executes a command and routes stdout/stderr through cron state, run history, retries, and delivery. The issue remains the canonical tracker; adjacent `/tools/invoke` work may provide an underlying mechanism, but it does not add the requested persisted cron payload surface.

## What This Changes

The issue requests a direct cron exec payload that can run scheduled commands without an agent turn or LLM call and then deliver the result through OpenClaw cron delivery.

## Best Possible Solution

Keep this as the canonical cron direct-exec tracker and design a hardened additive payload that reuses the shared tool invocation and exec-approval boundary while updating protocol, validation, runtime, delivery, CLI, UI, docs, and tests together.

## Reproduction Assessment

Yes, as source-level reproduction of a missing capability rather than a strict bug repro. Current main and v2026.5.22 still constrain cron payloads, schema, validation, runtime, CLI, UI, and docs to systemEvent or agentTurn paths.

## Solution Assessment

Unclear until maintainers settle the cron API and security policy. A native additive payload is plausible, but the maintainable implementation should reuse the shared tool invocation and exec-approval boundary rather than adding an independent spawn path.
