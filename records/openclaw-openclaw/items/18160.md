- `P2`: This is a normal-priority feature with strong production pain, but it requires product and security review rather than emergency handling.
- `impact:security`: The requested mode would add persisted unattended host-command execution to scheduled cron jobs.

## Summary

Keep open: current main still does not provide a cron payload that executes commands without an agent/LLM turn, and recent discussion points to this as the canonical cron-specific tracker rather than a superseded request.

## What This Changes

The issue requests a direct scheduled command-execution cron payload that bypasses agent/LLM turns while preserving OpenClaw cron delivery and observability.

## Best Possible Solution

Keep this as the canonical cron direct-exec tracker and design a hardened additive cron payload that reuses the shared tools.invoke and exec-approval boundary while updating protocol, runtime, delivery, CLI, UI, docs, and tests together.

## Reproduction Assessment

Yes for source-level capability absence, not as a bug reproduction: current main has no cron payload shape, schema, validation path, runtime branch, CLI flag, UI state, or docs for direct exec payloads.

## Solution Assessment

Unclear until maintainers settle the cron API and security policy. A native payload is plausible, but it should reuse the shared tools.invoke and exec approval boundary instead of adding an independent spawn path.
