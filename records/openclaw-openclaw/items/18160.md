## Summary

Keep open: current main still lacks a cron payload or CLI/UI path that runs a command without an agent turn, and live discussion plus the closed unmerged implementation PR show this remains the canonical cron-specific tracker rather than cleanup-ready work.

## What This Changes

The issue requests a zero-LLM cron execution mode that runs a configured command on schedule, captures stdout/stderr, and optionally delivers the result through OpenClaw channels.

## Best Possible Solution

Keep this as the canonical cron direct-exec tracker and design a hardened additive payload that reuses the shared tools.invoke and exec-approval boundary while updating protocol, validation, runtime, delivery, CLI, UI, docs, and tests together.

## Reproduction Assessment

Yes for source-level capability absence, not as a strict runtime bug. Current main has no cron payload type, schema, validation path, runtime branch, CLI option, UI type, or docs path for direct exec payloads.

## Solution Assessment

Unclear until maintainers settle the cron API and security policy. A native payload is plausible, but the maintainable implementation should reuse shared tools.invoke and exec-approval boundaries instead of adding an independent spawn path.
