## Label Justifications

- `P2`: This is a high-signal feature request with real production pain, but it is not an emergency and needs product/security review before implementation.
- `impact:security`: The requested mode would add persisted unattended host-command execution to scheduled cron jobs.
- `impact:message-loss`: Reported cron timeouts prevent deterministic scheduled jobs from producing their expected channel output.

## Summary

Keep open: current main still lacks a zero-LLM cron payload that executes commands and delivers output through the cron/run-history path, and the discussion shows this remains the canonical cron-specific tracker rather than a superseded request.

## What This Changes

Request to add a direct cron exec payload that runs scheduled scripts without an agent/LLM turn and can deliver stdout/stderr through OpenClaw channels.

## Best Possible Solution

Keep this as the canonical cron direct-exec tracker and design a hardened additive payload that reuses the shared tools.invoke and exec-approval boundary while updating protocol, validation, runtime, delivery, CLI, UI, docs, and tests together.

## Reproduction Assessment

Yes for source-level capability absence, not as a strict bug reproduction. Current main has no cron payload type, schema, validation path, runtime branch, CLI option, UI type, or docs path for direct exec payloads.

## Solution Assessment

Unclear until maintainers settle the cron API and security policy. A native payload is plausible, but the maintainable implementation should reuse the shared tools.invoke and exec-approval boundary instead of adding an independent spawn path.

## Risks / Open Questions

- Direct cron exec would persist unattended host-command execution, so approval, sandbox, host routing, cwd/env binding, shell selection, output limits, cancellation, timeout, retry, audit, and delivery policy need maintainer/security review.
- The existing /tools/invoke work may be the right lower-level mechanism, but it does not currently provide cron payload persistence, scheduler dispatch, run history, retries, or channel delivery.
- A child_process-only cron runner would repeat unresolved problems from the closed branch, especially process management, Windows shell behavior, and exec-approval/security drift.
