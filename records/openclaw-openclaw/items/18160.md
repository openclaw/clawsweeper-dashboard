## Label Justifications

- `P2`: This is a high-signal feature request with repeated production pain, but it is not an emergency and needs product/security review before implementation.
- `impact:security`: The requested mode would add persisted unattended host-command execution to scheduled cron jobs.
- `impact:message-loss`: Reported agentTurn timeouts prevent deterministic scheduled jobs from producing their expected channel output.

## Summary

Keep open: current main and v2026.5.22 still do not provide a zero-LLM cron payload that runs a command, records cron state/history, and delivers output through cron delivery. The related cron exec PR is closed unmerged, while the open direct-tool work is a useful underlying surface but does not implement the requested cron contract.

## What This Changes

Request an additive direct cron execution payload, such as `payload.kind="exec"`, for scheduled scripts that should bypass LLM agent turns and deliver their output through OpenClaw.

## Best Possible Solution

Keep this as the canonical cron direct-exec tracker and design a hardened additive cron payload that reuses the shared tool invocation and exec-approval boundary while updating protocol, runtime, delivery, CLI, UI, docs, and tests together.

## Reproduction Assessment

Yes, as source-level reproduction of a missing feature rather than a strict bug repro. Current main and v2026.5.22 still constrain cron payloads, schema, validation, runtime, CLI, UI, and docs to systemEvent or agentTurn paths.

## Solution Assessment

Unclear until maintainers settle the cron API and security policy. A native additive payload is plausible, but the maintainable solution should reuse the shared tool invocation and exec-approval boundary rather than adding an independent spawn path.
