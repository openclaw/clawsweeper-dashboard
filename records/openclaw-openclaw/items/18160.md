## Label Justifications

- `P2`: This is a normal-priority feature with strong production pain, but it needs product and security review before implementation.
- `impact:security`: The requested cron mode would add persisted unattended host-command execution to scheduled jobs.
- `impact:message-loss`: Reported cron failures include deterministic jobs timing out before their expected channel output can be delivered.

## Summary

Keep open: current main still has only `systemEvent` and `agentTurn` cron payloads, and the type, schema, validation, runtime, CLI, UI, and docs surfaces do not provide the requested zero-LLM scheduled exec path.

## What This Changes

The issue requests a cron payload mode that runs scheduled shell commands directly, captures stdout/stderr, and optionally delivers output without creating an LLM agent turn.

## Best Possible Solution

Keep this as the canonical cron direct-exec tracker and design a hardened additive cron payload contract that reuses the shared `tools.invoke` and exec-approval boundary while updating protocol, runtime, delivery, CLI, UI, docs, and tests together.

## Reproduction Assessment

Yes for source-level capability absence, not as a bug reproduction: current main has no cron payload shape, schema, validation path, runtime branch, CLI flag, UI state, or docs for direct exec payloads.

## Solution Assessment

Unclear until maintainers settle the cron API and security policy. A native payload is plausible, but it should reuse the shared `tools.invoke` and exec approval boundary instead of adding an independent spawn path.
