## Label Justifications

- `P2`: The request addresses real automation reliability and cost pain but needs design and security review before implementation.
- `impact:security`: The feature would add persisted unattended host-command execution to scheduled jobs.
- `impact:message-loss`: The reported failures include scheduled jobs timing out before their expected channel output can be delivered.

## Summary

Keep open: current main and v2026.5.12 still lack a zero-LLM cron exec payload, the prior implementation PR was closed unmerged for design/security/validation reasons, and the open direct `/tools/invoke` PRs are adjacent operator APIs rather than scheduled cron delivery.

## What This Changes

The issue requests a new cron payload mode that runs a host command directly on schedule and delivers its output without an agent or LLM turn.

## Best Possible Solution

Keep this as the canonical cron direct-exec tracker and design a hardened additive cron payload contract that reuses the approved exec/direct-tool boundary while updating protocol, scheduler runtime, delivery, CLI, UI, docs, and tests together.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for source-level capability absence: current main and v2026.5.12 have no cron payload shape, schema, validation path, runtime branch, CLI flag, or UI state for a direct exec payload.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers settle the cron API and security policy; a native payload is plausible, but it should reuse the direct-tool/exec boundary instead of adding an independent spawn path.

## Security Review

Status: needs_attention

Summary: Direct cron exec is security-sensitive because it would schedule unattended host-command execution outside an interactive agent turn.

Concerns:

- **[medium] Define unattended cron exec policy:** `docs/tools/exec-approvals.md:55`
  - body: Stored cron commands need explicit rules for approvals, sandboxing, host routing, cwd/env binding, shell choice, output retention, cancellation, timeout, retry, audit, and channel delivery before implementation.
  - confidence: 0.92
- **[medium] Preserve dangerous-tool boundaries:** `src/security/dangerous-tools.ts:9`
  - body: The current Gateway HTTP dangerous-tool list treats exec as default-denied RCE, so a cron exec payload should preserve explicit opt-in, owner semantics, auditability, and fail-closed behavior.
  - confidence: 0.9
- **[medium] Keep cron validation under security review:** `.github/CODEOWNERS:38`
  - body: CODEOWNERS routes cron service validation to the security owner group, and adding a payload that launches host commands changes that validation boundary.
  - confidence: 0.86
- **[medium] Handle headless approval failures explicitly:** `src/agents/bash-tools.exec-host-shared.ts:394`
  - body: Current exec policy treats cron as unable to wait for interactive approval; a cron exec payload must specify fail-closed behavior for approval-required commands instead of silently widening execution.
  - confidence: 0.88
