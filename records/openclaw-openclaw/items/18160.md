## Summary

Keep open: current main still lacks a zero-LLM cron exec payload, and the discussion shows this is the canonical tracker after duplicate shell/script cron requests were consolidated here. The request adds a public cron execution mode for persisted host commands, so it needs maintainer/secops design rather than cleanup closure.

## What This Changes

Add a cron payload mode, such as `payload.kind="exec"`, that runs a command directly on schedule, captures output, and optionally delivers it without creating an LLM-backed agent turn.

## Best Possible Solution

Keep this as the canonical core tracker and design a hardened additive cron command payload that updates protocol, validation, runtime dispatch, run logs, delivery, CLI, Control UI, docs, and tests with explicit host-command security rules.

## Reproduction Assessment

Not applicable for a bug reproduction. Source inspection on current main gives high confidence that cron only supports `systemEvent` and `agentTurn`, so the requested direct exec path is absent rather than a broken existing contract.

## Solution Assessment

Unclear until maintainers settle the API and security policy. A native cron exec payload is a plausible maintainable seam, but a simple child-process branch is not enough without Gateway, UI/CLI, delivery, observability, and secops follow-through.
