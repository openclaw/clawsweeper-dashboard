## Summary

Keep open: current main still lacks a cron `exec` payload, CLI/UI creation path, scheduler dispatch, and delivery/run-history path for zero-LLM scheduled shell work. The newer direct `/tools/invoke` work is useful lower-level plumbing, but the open follow-ups do not replace the cron-specific product and security design.

## What This Changes

Request a direct zero-LLM cron mode that runs a command on schedule, records stdout/stderr and exit status, and optionally delivers output to a channel.

## Best Possible Solution

Keep this as the canonical core cron tracker and design a hardened additive cron exec contract across types, Gateway schema, validation, runtime dispatch, run history, delivery, CLI, Control UI, docs, tests, and exec security policy.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source inspection on current main gives high confidence that the requested cron exec payload and runtime path are absent.

## Solution Assessment

Unclear until maintainers settle the API and security policy. A native cron exec payload is a plausible seam, but the durable solution needs coordinated protocol, CLI/UI, delivery, observability, and secops follow-through rather than a local spawn branch.
