## Summary

Current main contains the same-session follow-up drain fix that covers the recursive approved-exec completion loop: approved exec completions re-enter through the Gateway agent/follow-up path, and the reply-run lifecycle now clears the active guard before draining queued follow-ups.

## What This Changes

The issue asks OpenClaw to stop Control UI-approved exec completion events from being reprocessed recursively after approval.

## Best Possible Solution

Keep the current main lifecycle ordering so approved exec completion follow-ups cannot drain until the active reply operation is cleared, and ship it in the next release.

## Reproduction Assessment

Unclear as a live browser reproduction: the report gives concrete approval steps and source inspection maps the approved-exec completion into the same-session follow-up path, but I did not execute the Control UI flow. The fixed path is covered by source-level regression tests on current main.

## Solution Assessment

Yes. Clearing the active reply-run guard before draining queued same-session follow-ups is the narrow maintainable fix for this re-entry class; display filtering or another UI-only dedupe would not address the underlying recursive processing path.
