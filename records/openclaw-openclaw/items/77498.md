## Summary

Keep open. Current main still has the approved-exec completion path that sends an internal followup agent turn back into the same webchat session, and the existing coverage proves the single followup path but not the reported recursive consumption/dedup behavior; the prior related items address display leakage, not this loop variant.

## What This Changes

The issue asks OpenClaw to stop Control UI approved-exec completion events from being repeatedly reprocessed after an approval decision.

## Best Possible Solution

Add server-side consumption or deduplication for approved exec completion followups per approval id/session, with a regression test that repeated completion events cannot create another followup loop; keep UI filtering as defense-in-depth only.

## Reproduction Assessment

No high-confidence live reproduction was established. The source path is clear enough to investigate: approved exec completion in webchat-only mode sends an internal followup agent turn to the same session, but current tests only prove the first followup and not recursive reprocessing.

## Solution Assessment

Yes, keeping this open is the best current outcome. The narrow maintainable fix belongs in exec approval followup/session-event handling so each completion is consumed once, rather than relying on Control UI display filtering.
