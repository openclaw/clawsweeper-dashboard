## Summary

Keep open: current main still mitigates untrusted content with wrapping, special-token sanitization, tool/sandbox guidance, and manual reader-agent guidance, but it does not provide the requested automatic worker/manager LLM privilege boundary or strict validated JSON handoff for cron jobs. The ClawMoat plugin linked in the discussion is useful defense in depth, and the open CaMeL PR overlaps this area, but neither supersedes this cron-specific security architecture request on current main.

## What This Changes

Request automatic privilege separation for cron-related untrusted content by routing external reads through a read-only worker model and validated tagged handoff before a privileged manager model can act.

## Best Possible Solution

Define a maintainer-approved cron/untrusted-content isolation design that reuses existing wrapping, sandboxing, tool policy, provenance, and CaMeL/plugin defenses, then add a versioned handoff contract only where current APIs cannot express the required boundary.

## Reproduction Assessment

Not applicable. This is a security architecture feature request rather than a broken current contract, and source inspection shows the requested automatic worker/manager handoff is absent on current main.

## Solution Assessment

Unclear. Dual-LLM, CaMeL-style tainting, or provenance-based separation may be valid directions, but the cron trust boundary, validation semantics, config surface, failure mode, and compatibility story need maintainer security/product design before implementation.
