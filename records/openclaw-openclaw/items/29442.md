## Summary

Keep open: current main has external-content wrapping, suspicious-pattern logging, tool/sandbox policy, manual reader-agent guidance, and related memory-ingestion hardening, but it does not implement automatic worker/manager LLM privilege separation or a strict JSON middleware handoff for cron jobs. The ClawMoat plugin linked in the discussion is useful defense-in-depth, but it does not settle the core cron architecture request.

## What This Changes

Add automatic worker/manager LLM privilege separation and structured middleware tagging for untrusted data before cron jobs can act with privileged tools.

## Best Possible Solution

Define a scoped cron and untrusted-content isolation design that reuses existing sandbox, tool-policy, subagent, and provenance primitives where possible, adding a versioned core handoff contract only for gaps current APIs cannot express.

## Reproduction Assessment

No. This is a broad hardening architecture request rather than a reproducible bug report; source inspection confirms the requested automatic worker/manager boundary is not present, but the thread does not provide a current-main boundary-bypass PoC.

## Solution Assessment

Unclear. Dual-LLM separation is a plausible mitigation, but the issue does not define enough model-selection, sandbox, policy, JSON validation, failure-mode, or compatibility semantics to treat it as the best maintainable implementation.
