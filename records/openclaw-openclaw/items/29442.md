## Summary

Keep open: current main has external-content wrapping, tool-policy and sandbox controls, manual sub-agent isolation, and adjacent memory-ingestion hardening, but it does not implement automatic worker/manager LLM privilege separation or a strict JSON middleware handoff for cron jobs. The ClawMoat link in the discussion is useful defense-in-depth, but the core cron architecture request still needs maintainer security/product design.

## What This Changes

Request automatic dual-LLM privilege separation for cron jobs, with a read-only worker processing untrusted input and a structured/tagged handoff to a privileged manager.

## Best Possible Solution

Define a scoped cron and untrusted-content isolation design that reuses existing sandbox, tool-policy, and sub-agent primitives where possible, adding a versioned core handoff contract only for gaps that plugins or current APIs cannot express.

## Reproduction Assessment

No. This is a broad hardening feature request rather than a reproducible bug report; source inspection confirms the requested architecture is not present, but the thread does not provide a current-main boundary-bypass PoC.

## Solution Assessment

Unclear. Dual-LLM separation is a plausible mitigation, but the issue does not yet define enough model-selection, sandbox, policy, JSON validation, failure-mode, or compatibility semantics to be the best maintainable implementation.
