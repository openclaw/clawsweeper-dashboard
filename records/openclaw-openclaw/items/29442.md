## Summary

Keep open. Current main still uses external-content wrapping and operator mitigations for cron/hook input; the ClawMoat link in the discussion is useful defense-in-depth, but it does not replace a core privilege-separated or schema-validated ingestion boundary.

## What This Changes

The issue requests a privilege-separated, dual-model or equivalent untrusted-content ingestion boundary for cron jobs so hostile external data cannot flow into privileged agent execution as ordinary prompt text.

## Best Possible Solution

Add an explicit untrusted-content ingestion boundary for cron and hooks, preserving plugin extensibility while preventing raw external content from reaching privileged cron execution or memory capture as ordinary prompt text.

## Reproduction Assessment

Yes for the current behavior gap: source and tests show external hook payloads are wrapped and passed to the normal cron agent prompt path. No live exploit reproduction was provided or run.

## Solution Assessment

Unclear. The requested dual-LLM split is plausible, but the narrowest maintainable OpenClaw solution needs maintainer security/product choice between a default reader-worker, structured sanitizer, taint tracking, or plugin API hook.
