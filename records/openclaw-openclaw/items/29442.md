## Summary

Keep open: current main has external-content wrapping, suspicious-pattern logging, special-token sanitization, sandbox/tool-policy guidance, and manual reader-agent guidance, but it still does not implement automatic worker/manager LLM privilege separation or a strict JSON middleware handoff for cron jobs. The ClawMoat plugin linked in the discussion is useful defense-in-depth, and related open issues track narrower tainting, scanning, memory, and sandbox slices, but none supersede this cron trust-boundary request.

## What This Changes

The issue requests cron prompt-injection hardening through a read-only worker LLM, structured provenance tagging, validation middleware, and an isolated privileged manager LLM.

## Best Possible Solution

Define a scoped cron and untrusted-content isolation design that reuses existing wrappers, tool policy, sandboxing, subagent/reader-agent guidance, and provenance primitives where possible, adding a versioned core handoff contract only for gaps current APIs cannot express.

## Reproduction Assessment

Not applicable. This is a security architecture feature/hardening request rather than a bug report with a current-main PoC; source inspection shows the requested automatic worker/manager boundary is absent.

## Solution Assessment

Unclear. Dual-LLM separation is a plausible mitigation, but the proposal does not yet settle model selection, sandbox scope, JSON validation, policy enforcement, failure modes, or compatibility semantics enough to treat it as the best maintainable implementation.
