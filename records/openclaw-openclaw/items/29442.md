## Summary

Keep open. Current main has meaningful external-content wrapping and docs-level guidance, but it still sends wrapped untrusted hook/cron content into the ordinary cron agent prompt path rather than implementing the requested read-only worker, schema-validated handoff, or privileged manager continuation boundary.

## What This Changes

The issue requests a dual-LLM privilege-separation design for cron and external-content ingestion, with untrusted data processed by a read-only worker and only structured validated output passed to a privileged manager agent.

## Best Possible Solution

Add an explicit untrusted-content ingestion boundary for cron and hooks: either a first-class read-only/tool-disabled reader-worker flow with schema-validated output before any privileged continuation, or a narrower generic sanitizer/taint hook that can block or replace wrapped content before it reaches the manager prompt. The design should preserve plugin extensibility and include regression tests showing untrusted external content cannot silently reach privileged cron execution or memory capture as ordinary prompt text.

## Reproduction Assessment

Yes for the current behavior gap: the source and tests show external hook content is wrapped and then passed to runEmbeddedPiAgent as prompt text. No live exploit reproduction was run, and the issue does not provide one.

## Solution Assessment

Unclear. The requested dual-LLM split is a plausible hardening direction, but the narrowest maintainable OpenClaw solution needs maintainer security/product choice between a default reader-worker, structured llm-task-style sanitizer, taint tracking, or plugin API hook.
