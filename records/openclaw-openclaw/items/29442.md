## Summary

Keep open: current main has wrapping, suspicious-pattern logging, special-token sanitization, sandbox/tool-policy guidance, and manual reader-agent guidance, but it still lacks the requested automatic worker/manager LLM privilege boundary and strict JSON middleware handoff for cron jobs. The ClawMoat plugin linked in the discussion is useful defense-in-depth, and the open CaMeL PR overlaps this space, but neither supersedes this cron-specific trust-boundary request on current main.

## What This Changes

Request automatic cron privilege separation where a read-only worker model handles untrusted content and passes validated tagged JSON to a privileged manager model.

## Best Possible Solution

Define a maintainer-approved cron and untrusted-content isolation design, reusing existing wrappers, tool policy, sandboxing, provenance, and CaMeL work where appropriate, then add a versioned handoff contract only for gaps current APIs cannot express.

## Reproduction Assessment

Not applicable. This is a security architecture feature/hardening request rather than a broken current contract; source inspection shows the requested automatic worker/manager handoff is absent on current main.

## Solution Assessment

Unclear. Dual-LLM or CaMeL-style separation is plausible, but the cron trust boundary, model routing, validation semantics, config, failure modes, and compatibility story need maintainer security/product design before implementation.
