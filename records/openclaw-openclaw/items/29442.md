## Summary

Keep open: current main has defense-in-depth wrappers, sandbox/tool-policy controls, and docs recommending reader agents, but it does not implement the requested automatic worker/manager LLM boundary or structured middleware handoff for cron jobs.

## What This Changes

Request automatic dual-LLM privilege separation for cron jobs so untrusted external data is read by a read-only worker and only sanitized structured output reaches a privileged manager.

## Best Possible Solution

Define a scoped security design for cron and untrusted-content isolation, reusing existing sandbox/tool-policy primitives where possible and adding core APIs only where automatic reader/manager routing cannot be expressed today.

## Reproduction Assessment

No. This is a broad hardening feature request, and the thread does not provide a current-main proof of concept that crosses a documented OpenClaw boundary; source inspection only confirms the requested architecture is not present.

## Solution Assessment

Unclear. The dual-LLM split is a plausible direction and current docs already recommend manual reader-agent separation, but automatic core routing needs maintainer design around tool policy, sandboxing, model choice, and structured handoff contracts.
