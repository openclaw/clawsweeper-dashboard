## Summary

Keep open: current main still uses marker-based external-content wrapping and a single cron agent run, not automatic read-only worker/manager LLM privilege separation or a strict JSON middleware handoff. The ClawMoat plugin and related sandbox/memory issues are useful adjacent defenses, but they do not supersede the core cron trust-boundary request.

## What This Changes

Request automatic dual-LLM privilege separation for cron and untrusted-content flows, with a read-only worker, strict tagged/JSON middleware handoff, and optional validator before privileged manager execution.

## Best Possible Solution

Define a scoped secops-approved cron and untrusted-content isolation design that reuses existing wrappers, sandbox/tool policy, subagent/reader-agent, and provenance primitives, adding a versioned core handoff contract only where current APIs cannot express the boundary.

## Reproduction Assessment

Not applicable. This is a security architecture feature request rather than a bug report with a current-main PoC; source inspection shows the requested automatic worker/manager boundary is absent.

## Solution Assessment

Unclear. Dual-LLM separation is a plausible mitigation, but the proposal needs maintainer/secops decisions on model routing, sandbox/tool policy, schema validation, failure modes, and configuration before it can be treated as the best maintainable fix.
