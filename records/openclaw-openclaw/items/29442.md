## Summary

Keep open: current main still relies on wrapping, sanitization, logging, tool policy, sandbox guidance, and manual reader-agent guidance, but it does not implement the requested automatic worker/manager LLM privilege boundary or strict validated handoff for cron jobs. The ClawMoat plugin linked in the discussion is useful defense in depth, and related memory/sandbox issues overlap, but they do not supersede this cron-specific security architecture request.

## What This Changes

Add automatic worker/manager LLM privilege separation and strict validated handoff for cron jobs so untrusted external data is processed without privileged tool access.

## Best Possible Solution

Define a maintainer-approved cron and untrusted-content isolation design that reuses existing wrapping, sandboxing, tool policy, provenance, and plugin defenses, then add a versioned handoff/API contract only where current surfaces cannot express the required boundary.

## Reproduction Assessment

Not applicable. This is a security architecture feature request rather than a broken current contract; source inspection shows current cron still routes one prepared prompt into one agent runner instead of an automatic worker/manager handoff.

## Solution Assessment

Unclear. Dual-LLM separation is a plausible direction, but CaMeL-style tainting, provenance-aware middleware, model routing, validation semantics, config shape, and failure modes need maintainer security/product design before implementation.
