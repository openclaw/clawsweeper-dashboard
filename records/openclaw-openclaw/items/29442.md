## Summary

Keep open: current main has external-content wrapping, suspicious-pattern logging, special-token sanitization, tool/sandbox guidance, and manual reader-agent guidance, but it does not implement automatic worker/manager LLM privilege separation or a strict JSON middleware handoff for cron jobs. The ClawMoat plugin linked in the discussion is useful defense-in-depth, and related open issues cover narrower memory/sandbox slices, but none supersede this core cron trust-boundary request.

## What This Changes

The issue requests automatic dual-LLM privilege separation for cron and untrusted-data work, with a read-only worker, strict JSON middleware tagging, and optional validation before privileged manager execution.

## Best Possible Solution

Define a scoped cron and untrusted-content isolation design that reuses existing external-content wrappers, tool policy, sandboxing, subagent/reader-agent, and provenance primitives where possible, adding a versioned core handoff contract only for gaps current APIs cannot express.

## Reproduction Assessment

Not applicable. This is a security architecture feature/hardening request rather than a bug report with a current-main PoC; source inspection does show that the requested automatic worker/manager boundary is absent.

## Solution Assessment

Unclear. Dual-LLM separation is a plausible mitigation, but the proposal does not yet define enough model-selection, sandbox, policy, JSON validation, failure-mode, or compatibility semantics to treat it as the best maintainable implementation.
