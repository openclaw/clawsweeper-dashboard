## Summary

Keep open: current main has isolated cron sessions, external-content wrapping, tool allow-lists, and sandbox backends, but it does not implement the requested dual-LLM worker/manager privilege separation or validator boundary. The related ClawMoat skill and the open issues at https://github.com/openclaw/openclaw/issues/28913 and https://github.com/openclaw/openclaw/issues/78965 are useful adjacent defenses, but they do not supersede this broader security architecture request.

## What This Changes

The issue requests dual-LLM privilege separation, structured provenance tagging, and optional validation for cron/external-content runs to reduce prompt-injection privilege escalation.

## Best Possible Solution

Keep this as a maintainer-owned security architecture track, then split accepted pieces into narrower canonical implementation issues for provenance, tool authority, sandboxing, and cron runtime behavior.

## Reproduction Assessment

Not applicable as a strict bug reproduction: this is a security architecture feature request. Source inspection does confirm the current cron path wraps external-hook content into a single agent prompt and has no dual-worker/manager implementation.

## Solution Assessment

Unclear. Dual-LLM isolation is a plausible design, but maintainers should decide whether the best core solution is model separation, framework-level provenance/taint checks, stricter tool policy, sandboxing, or a combination; the ClawMoat skill is useful defense-in-depth, not a replacement for that decision.
