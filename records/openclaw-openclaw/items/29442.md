## Summary

Keep open: current main has meaningful prompt-injection defenses for cron and external content, but it does not implement the requested automatic dual-LLM privilege separation or taint-aware middleware. The related discussion points to useful partial tracks, including ClawMoat as defense-in-depth and open follow-ups for sandboxing and memory filtering, but none fully supersedes this broader security architecture request.

## What This Changes

Request automatic dual-LLM and taint-tagged cron routing so untrusted external data is handled by read-only worker models before any privileged agent can act.

## Best Possible Solution

Treat this as a security architecture design item: define a generic taint/provenance contract and cron/subagent routing policy, then split the chosen approach into narrower implementation issues tied to existing external-content wrapping, sandboxing, and tool allowlist surfaces.

## Reproduction Assessment

Not applicable as a feature/security architecture request. No single exploit repro was supplied, but source and docs confirm the relevant current behavior is prompt wrapping plus policy/sandbox mitigations rather than automatic dual-LLM separation.

## Solution Assessment

Unclear whether the proposed dual-LLM design is the best final implementation. Maintainers should choose between dual-model isolation, taint tracking, validator hooks, and narrower sandbox/tool-policy work before a code change is attempted.
