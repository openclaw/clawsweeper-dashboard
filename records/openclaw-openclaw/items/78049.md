## Summary

Keep open: current main and the latest release still document and implement the macOS local Gateway as an external launchd-managed CLI/Node service, while the app-brokered `system.run` path and the open bundled-node PR are partial mitigations rather than a complete TCC-safe lifecycle for default local file and CLI workflows.

## What This Changes

The issue requests a reliable macOS Gateway startup or execution path for TCC-protected local file and CLI workflows instead of relying on a launchd-managed bare Node process.

## Best Possible Solution

Keep this issue open as the broader macOS TCC lifecycle work: pick one supported model for protected-folder local CLI/file operations, then align launchd, app broker routing, docs, and live macOS proof around it.

## Reproduction Assessment

Yes at source level: current docs and code still expose an external launchd Gateway path while the issue gives concrete macOS TCC failure steps. I did not run a live macOS TCC reproduction in this read-only review.

## Solution Assessment

No current implementation is the full best fix yet. The app-brokered `system.run` path and bundled-node PR are useful mitigations, but maintainers still need to choose the supported TCC-safe lifecycle for default local file and CLI workflows.
