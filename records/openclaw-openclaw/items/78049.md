## Summary

Keep this open: current main still defaults macOS local Gateway management to a launchd-backed Node service, and the TCC-safe macOS app `system.run` path is an opt-in node execution route rather than the default local CLI/file workflow path.

## What This Changes

The issue asks for an official macOS Gateway startup or execution path that preserves access to TCC-protected user folders for local CLI/file workflows.

## Best Possible Solution

Define and document a first-class macOS path for TCC-sensitive local exec that does not depend on a bare launchd Node process, likely by routing the relevant local CLI/file workflow through the signed macOS app node host or by supporting an explicit Terminal-backed startup mode.

## Reproduction Assessment

No, not at high confidence from this environment: the provided steps are concrete, and current source/docs match the reported launchd Gateway and default Gateway exec path, but I did not live-test macOS TCC behavior on a matching Mac.

## Solution Assessment

Unclear: the requested directions are plausible, but the best fix needs maintainer product judgment between defaulting Mac local exec to the signed app/node path, adding an explicit startup mode, or documenting the current wrapper/manual workaround more strongly.
