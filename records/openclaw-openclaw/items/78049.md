## Summary

Keep open: current main and the latest release still use a launchd-managed external Node Gateway for macOS local mode, while default exec still routes to the Gateway unless the user explicitly chooses a node path. The related TCC issues overlap, but none fully owns this issue's broader local CLI/file workflow case for protected folders like Documents.

## What This Changes

The issue asks OpenClaw to provide a supported macOS Gateway startup or local exec path for TCC-protected folders instead of relying on a bare launchd-managed Node process.

## Best Possible Solution

Choose and document a first-class macOS TCC-sensitive local exec path, likely a signed app/node broker or explicit Terminal/manual mode, while preserving launchd fallback and explicit user consent.

## Reproduction Assessment

No live high-confidence reproduction was run in this environment. The issue is source-reproducible in the sense that current main still uses launchd-managed Node for local Gateway startup and default exec routes to Gateway, but the actual TCC failure requires a real macOS protected-folder smoke test.

## Solution Assessment

Unclear. The requested directions are plausible, but maintainers need to choose between a signed app/node broker, dedicated runtime identity, Terminal-backed mode, or stronger documentation before a maintainable fix can be scoped.
