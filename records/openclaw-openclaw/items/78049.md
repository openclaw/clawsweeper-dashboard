## Summary

Keep open: current main still uses a launchd-managed Node Gateway for the default macOS local path, while the requested TCC-safe protected-folder workflow is not implemented or clearly documented as supported. The related #51299 is relevant evidence for the same launchd/TCC root cause, but this issue adds broader non-iMessage CLI/file workflows and needs maintainer product direction.

## What This Changes

Request an official macOS Gateway startup or execution path that preserves TCC access for local CLI/file workflows against Documents, Downloads, and Desktop.

## Best Possible Solution

Define one supported macOS local-exec model for TCC-protected folders, preferably by routing protected local CLI work through the signed macOS app/node `system.run` path or by adding an explicit supported Terminal/native-helper startup mode with docs and live macOS proof.

## Reproduction Assessment

Yes, there is a high-confidence source-level reproduction path: use the default macOS LaunchAgent Node Gateway, run a CLI/file listing against a protected Documents path, and compare with the same Gateway command started from Terminal.app. I did not live-run it because this review environment is Linux, so the status is source-reproducible rather than directly reproduced.

## Solution Assessment

Unclear; the issue's suggested directions are plausible but not yet the single best fix. Maintainers need to choose the supported macOS permission owner and execution route before a narrow implementation should be attempted.
