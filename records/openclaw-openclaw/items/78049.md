## Summary

Keep open: current main still uses a launchd-managed external Node Gateway for macOS local mode, and implicit local exec still resolves to the Gateway when no sandbox is active. The related TCC issues overlap, but this issue preserves the broader protected-folder CLI/file workflow case beyond iMessage or shared-Node scoping alone.

## What This Changes

Request that macOS local Gateway support TCC-protected folder CLI/file workflows through a signed/app-owned or Terminal-backed execution path instead of a bare launchd Node process.

## Best Possible Solution

Define and document a first-class macOS TCC-sensitive local command path, likely a signed app/runtime broker or explicit Terminal/manual mode, while preserving launchd fallback and user-visible consent.

## Reproduction Assessment

No live high-confidence reproduction was run. The source path is reproducible because current main still starts macOS local Gateway through launchd/Node and default exec routes to that Gateway, but protected-folder denial needs a real macOS TCC smoke test.

## Solution Assessment

Unclear. The requested directions are plausible, but maintainers need to choose between a signed app/runtime broker, dedicated runtime identity, Terminal-backed mode, or routing selected commands through app-owned `system.run`.
