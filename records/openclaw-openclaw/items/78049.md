## Summary

Keep open: current main and the latest stable release still use the macOS launchd plus Node Gateway path, and implicit local exec still runs in the Gateway context when no sandbox is active. Related issues either point back here for the broader macOS TCC owner problem or track narrower iMessage/runtime symptoms, so this remains the canonical local CLI/file workflow tracker.

## What This Changes

The issue reports that the macOS launchd-managed Gateway cannot reliably run local CLI/file workflows against TCC-protected folders and asks for a supported signed-app/helper, app-brokered, or Terminal-backed startup path.

## Best Possible Solution

Define and document a first-class macOS TCC-sensitive local command path, likely a signed app/runtime broker or explicit Terminal/manual mode, while preserving launchd fallback and user-visible consent.

## Reproduction Assessment

No, not as a live high-confidence reproduction in this review. Source inspection confirms the launchd/Node Gateway path and default Gateway exec routing still exist, but the protected-folder denial needs a real macOS TCC smoke test.

## Solution Assessment

Unclear. The requested directions are plausible, but maintainers need to choose between a signed app/runtime broker, dedicated runtime identity, Terminal-backed mode, or routing selected commands through app-owned `system.run`.
