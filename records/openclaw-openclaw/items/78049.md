## Summary

Keep open: current main and v2026.6.10 still expose the macOS local Gateway as a launchd-managed external Node service, while app-brokered system.run and bundled-Node work are only partial mitigations for the protected-folder TCC workflow.

## What This Changes

The issue requests a supported macOS Gateway startup path that preserves access to TCC-protected user folders for local CLI/file workflows instead of relying on a bare launchd-managed Node process.

## Best Possible Solution

Choose one supported macOS TCC owner and Gateway/local-exec lifecycle, then align launchd, native app or capability host, docs, upgrade behavior, and live protected-folder proof around that model.

## Reproduction Assessment

Yes at source level: current main still exposes the LaunchAgent-managed external Node Gateway path, and the report gives concrete macOS TCC steps. No live macOS protected-folder reproduction was run in this read-only review.

## Solution Assessment

No current implementation is the full best fix. App-brokered system.run and bundled-Node selection are useful partial mitigations, but the supported default TCC owner and startup/security model still need a maintainer decision.
