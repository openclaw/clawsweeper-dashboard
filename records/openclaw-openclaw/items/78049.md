## Summary

Keep open: current main and v2026.6.8 still implement and document the macOS local Gateway as a launchd-managed Node service, while app-brokered `system.run` and the bundled-node PR are only partial mitigations for the protected-folder TCC workflow.

## What This Changes

The issue asks OpenClaw to provide a supported macOS Gateway lifecycle that can reliably run local CLI/file workflows against TCC-protected folders such as Documents, Downloads, and Desktop.

## Best Possible Solution

Choose one supported macOS TCC owner and Gateway lifecycle for protected-folder local CLI/file operations, then align launchd, app-broker, Terminal-backed docs, and live macOS proof around it.

## Reproduction Assessment

Yes at source level: current main still exposes the LaunchAgent-managed Node Gateway path and the report gives concrete macOS TCC steps, but I did not run live macOS TCC proof.

## Solution Assessment

No current implementation is the full best fix; app node `system.run` and bundled-node runtime preference are useful mitigations but do not settle the default TCC-safe Gateway lifecycle.
