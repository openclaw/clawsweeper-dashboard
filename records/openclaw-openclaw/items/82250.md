## Summary

Keep open: current main and the latest release still combine unconditional macOS `KeepAlive=true` with a launchd duplicate-gateway path that returns cleanly after finding a healthy existing gateway. The reporter's live follow-up strengthens the workaround evidence, but the final fix needs maintainer lifecycle-policy coordination because the related open crash-loop PR uses clean-exit relaunches and does not by itself solve this clean duplicate-exit loop.

## What This Changes

The issue requests a macOS LaunchAgent lifecycle change so a launchd-started gateway that exits cleanly after finding an existing healthy foreground gateway is not relaunched forever.

## Best Possible Solution

Add a focused launchd lifecycle fix that prevents healthy duplicate-gateway exits from respawning indefinitely while preserving intentional update/restart handoffs and aligning service audit or doctor repair with the chosen canonical policy.

## Reproduction Assessment

Yes, source-backed rather than live: current main emits boolean `KeepAlive=true`, the launchd duplicate-healthy-gateway path returns cleanly, and current tests pin that clean launchd handoff. I did not run a live macOS LaunchAgent reproduction in this read-only review.

## Solution Assessment

Unclear as proposed: `SuccessfulExit=false` matches the reporter's live workaround for this loop, but it conflicts with clean-exit update/restart handoffs and with the related `SuccessfulExit=true` crash-loop direction. The best fix likely needs explicit duplicate-gateway exit semantics coordinated with the canonical launchd KeepAlive policy and audit behavior.
