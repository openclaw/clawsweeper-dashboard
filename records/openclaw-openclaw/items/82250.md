## Label Justifications

- `P2`: This is a real macOS LaunchAgent lifecycle bug with bounded platform scope and an unresolved policy interaction rather than an emergency outage.
- `impact:crash-loop`: The issue is about launchd repeatedly respawning a gateway process after a clean duplicate-gateway exit.

## Summary

Keep open: current main and v2026.5.18 still generate unconditional macOS `KeepAlive=true`, while the launchd-supervised healthy-duplicate path returns cleanly, so the reported respawn loop remains source-backed. The related PR attempts are closed or cover the different startup-failure policy, not this clean duplicate-gateway exit case.

## What This Changes

The issue requests macOS LaunchAgent behavior that avoids respawning after a launchd-owned starter exits cleanly because another healthy gateway already owns the port.

## Best Possible Solution

Adopt a coordinated launchd lifecycle policy that stops healthy duplicate LaunchAgent attempts without breaking intentional supervised restarts and updates, with service-audit/doctor behavior and live macOS smoke covering duplicate-gateway, startup-failure, explicit stop, crash recovery, and restart/update handoff paths.

## Reproduction Assessment

Yes, source-backed but not live-run here: start a foreground gateway, load the macOS LaunchAgent on the same port, and current code returns normally after the healthy duplicate check while boolean `KeepAlive=true` relaunches it. The reporter also supplied live macOS output showing the workaround stops the loop.

## Solution Assessment

Unclear as proposed: `SuccessfulExit=false` addresses the clean duplicate-exit churn, but current launchd restart/update handoffs also rely on clean exits being relaunched. A maintainable fix needs distinct duplicate-exit handling or supervisor control rather than a plist-only swap.
