## Summary

Keep open: current main and v2026.5.12 still render boolean macOS `KeepAlive=true` while the launchd duplicate-gateway path returns cleanly after finding a healthy gateway, so the reported respawn churn remains source-backed. The reporter's follow-up strengthens the `SuccessfulExit=false` workaround evidence, but the final upstream fix still needs launchd lifecycle-policy review because existing restart/update handoffs also rely on clean exits.

## What This Changes

The issue asks OpenClaw to stop macOS launchd from repeatedly respawning a LaunchAgent process that exits cleanly after detecting an already healthy foreground gateway.

## Best Possible Solution

Implement a focused launchd lifecycle path that prevents healthy duplicate-gateway exits from respawning forever while preserving intentional restart/update handoffs and teaching service audit/doctor the canonical policy.

## Reproduction Assessment

Yes, source-backed but not locally live-reproduced: run a foreground gateway, load the macOS LaunchAgent on the same port, and current code returns success after the health probe while boolean `KeepAlive=true` respawns it. The reporter also provided OpenClaw 2026.5.12 macOS output showing the `SuccessfulExit=false` workaround stops the loop and survives reboot.

## Solution Assessment

Unclear as proposed: `SuccessfulExit=false` is a credible operator workaround for this clean-exit loop, but applying it globally can conflict with existing clean-exit launchd restart/update handoffs. A safer upstream fix likely needs distinct duplicate-gateway semantics or start-time handoff handling plus audit updates.
