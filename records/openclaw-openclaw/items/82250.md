## Label Justifications

- `P1`: A latest-stable macOS report shows a launchd-supervised gateway can enter a repeated restart loop after update and fail to recover without operator intervention.
- `impact:crash-loop`: The issue is about repeated launchd respawns and gateway process churn around clean exits.

## Summary

Keep open: current main and the latest stable release still generate unconditional macOS `KeepAlive=true`, while the launchd duplicate-gateway path returns success after finding a healthy existing gateway. Two independent issue comments now provide live macOS evidence, including OpenClaw 2026.5.20 after an update, and the open related PR only targets the opposite nonzero-failure policy.

## What This Changes

The issue requests a macOS LaunchAgent lifecycle fix so clean exits caused by detecting an already-running healthy gateway do not create repeated launchd respawns.

## Best Possible Solution

Handle healthy duplicate-gateway exits as a distinct launchd lifecycle case while preserving intentional update/restart clean-exit handoffs, then align service audit/doctor, regression tests, and live macOS LaunchAgent proof with that chosen policy.

## Reproduction Assessment

Yes, source-backed with live reporter evidence: start or refresh a launchd-supervised gateway while another healthy gateway owns the port, and current code exits 0 while boolean `KeepAlive=true` respawns it. I did not run a live macOS LaunchAgent repro in this read-only review.

## Solution Assessment

No, not as the simple `SuccessfulExit=false` plist-only workaround. That workaround is validated for the reported loop, but the maintainable upstream fix must also preserve clean-exit update and restart handoffs that currently rely on launchd relaunching the gateway.
