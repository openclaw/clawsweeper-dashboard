## Label Justifications

- `P1`: The latest-stable follow-up reports a launchd-supervised update path entering a repeated gateway restart loop on macOS.
- `impact:crash-loop`: The issue is about launchd repeatedly respawning or killing gateway processes after clean exits and handoff races.

## Summary

Keep open: current main and the latest stable tag still use boolean macOS LaunchAgent `KeepAlive=true`, while the healthy duplicate-gateway path can exit cleanly under launchd and be respawned again. The later OpenClaw 2026.5.20 update-path report makes this a current, actionable launchd lifecycle bug rather than a cleanup-close candidate.

## What This Changes

The issue asks OpenClaw to prevent macOS launchd from repeatedly restarting the gateway after a clean duplicate-gateway exit.

## Best Possible Solution

Define a launchd lifecycle contract where healthy duplicate-gateway exits do not request relaunch, intentional restart/update handoffs still relaunch explicitly or via a compatible exit policy, and service audit/doctor recognizes and repairs the chosen plist shape.

## Reproduction Assessment

Yes, source-backed. Current main generates boolean `KeepAlive=true`; the launchd healthy-duplicate path returns successfully after seeing an existing `/healthz` responder; launchd's documented boolean policy then keeps the job alive and can respawn it.

## Solution Assessment

No, not as a one-line plist-only fix. `SuccessfulExit=false` stops the reported clean-exit loop, but the final fix must also preserve intentional launchd update/restart handoffs and update service audit/doctor behavior.
