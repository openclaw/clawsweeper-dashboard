## Summary

Keep open: current main and v2026.5.12 still combine boolean macOS `KeepAlive=true` with a launchd duplicate-gateway path that returns success after detecting a healthy existing gateway. The active related PRs cover overlapping launchd lifecycle work, but they have not shipped a fix for this clean duplicate-exit respawn case.

## What This Changes

The issue asks OpenClaw to stop macOS launchd from repeatedly respawning a clean duplicate-gateway exit when another healthy gateway already owns the port.

## Best Possible Solution

Implement one coherent launchd lifecycle policy that prevents healthy duplicate-gateway exits from respawning forever, preserves intentional restart/update handoffs, and updates service audit/doctor plus tests to recognize or repair the chosen plist shape.

## Reproduction Assessment

Yes, source-reproducible: start a foreground gateway, then start the macOS LaunchAgent on the same port; current launchd duplicate handling returns successfully after a healthy probe while boolean `KeepAlive=true` causes launchd to respawn it. I did not live-run macOS launchd in this read-only review.

## Solution Assessment

Unclear as proposed: `SuccessfulExit=false` matches the reporter's live workaround, but it changes clean-exit relaunch behavior used by restart/update handoffs. A safer fix likely needs canonical KeepAlive semantics plus a distinct exit/status path for the healthy duplicate-gateway case.
