## Summary

Keep open: current main and the latest stable release still generate boolean macOS `KeepAlive=true`, while the launchd duplicate-gateway path returns successfully after finding a healthy existing gateway. The newer v2026.5.20 update-path report confirms the same clean-exit respawn loop is still user-visible, and the open adjacent fix for https://github.com/openclaw/openclaw/issues/73673 does not clearly solve this clean-exit case.

## What This Changes

The issue asks OpenClaw to stop macOS LaunchAgents from relaunching forever after a clean gateway exit caused by a healthy duplicate gateway or update handoff.

## Best Possible Solution

Implement a launchd lifecycle policy that distinguishes healthy-duplicate or obsolete clean exits from intentional update/restart handoffs, updates service audit/doctor to the chosen canonical policy, and proves the main-gateway and update paths on macOS.

## Reproduction Assessment

Yes, source-backed and externally live-reported: current main writes `KeepAlive=true` and returns successfully for a healthy launchd duplicate, while v2026.5.20 comments show launchd runs climbing after clean exit. I did not run a live macOS LaunchAgent smoke in this read-only pass.

## Solution Assessment

Unclear as proposed. `SuccessfulExit=false` stops the reported clean-exit loop in user tests, but current restart/update paths also use clean exits for intentional launchd handoff, so the best fix must change the lifecycle contract rather than only flipping the plist value.
