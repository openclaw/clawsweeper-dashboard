## Label Justifications

- `P1`: The latest follow-up shows a current stable macOS LaunchAgent restart loop after `openclaw update`, affecting a real supervised gateway workflow now.
- `impact:crash-loop`: The issue is about launchd repeatedly respawning or replacing gateway processes after clean exits.
- `impact:message-loss`: The reported launchd loop restarts the gateway and kills channel-adjacent processes, which can interrupt active message delivery while the service churns.

## Summary

Keep open: current main still generates boolean macOS LaunchAgent `KeepAlive=true`, the launchd duplicate-gateway path can exit with status 0 after detecting a healthy existing gateway, and the latest user follow-up reports the loop on v2026.5.20 after `openclaw update`. The issue is source-backed and has independent macOS operator confirmations.

## What This Changes

The issue requests changing macOS LaunchAgent restart behavior so a clean already-running gateway exit does not cause launchd to respawn the job forever.

## Best Possible Solution

Define a launchd lifecycle policy that distinguishes intentional clean-exit handoffs from healthy-duplicate exits, then update plist generation, duplicate-gateway exit semantics, service audit, and tests together.

## Reproduction Assessment

Yes, source-backed: current main emits `KeepAlive=true`, the launchd duplicate-gateway path returns cleanly after a healthy existing gateway check, and commenters reproduced the resulting respawn loop on macOS 2026.5.12 and 2026.5.20.

## Solution Assessment

Unclear as proposed: `SuccessfulExit=false` fixes the observed loop, but the maintainable fix should also preserve intentional update/restart handoffs and update service-audit expectations.
