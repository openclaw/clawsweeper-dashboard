## Summary

## What This Changes

The branch adds a known-errno fallback in `writeUsageCostCacheLockAtomically` to create the lock with `flag: "wx"` when hard-link publishing fails, plus a regression test that mocks `ENOTSUP`.

## Best Possible Solution

Refresh or replace this branch on current main, then land one canonical fallback with the existing lock semantics plus redacted real unsupported-hardlink filesystem proof or an explicit maintainer proof override.

## Reproduction Assessment

Yes at source level: force `fs.promises.link` to reject with `ENOTSUP` and current main rethrows it because only `EEXIST` is handled. I did not establish a live SMB/NFS/virtiofs mount reproduction in this read-only review.

## Solution Assessment

Yes for the code shape: preserving the hard-link path and falling back to exclusive create at the lock writer matches the right owner boundary and repository precedent. It is not merge-ready until the branch is refreshed and the real-environment proof gap is resolved or explicitly overridden by maintainers.
