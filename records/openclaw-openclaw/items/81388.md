## Summary

Keep open: current main and v2026.6.9 still have the hard-link-only usage-cost lock path, and this PR remains a useful candidate fix. ## What This Changes

The PR adds an unsupported-hardlink fallback from `fs.promises.link` to exclusive-create `writeFile(..., { flag: "wx" })` for the session usage-cost cache lock, plus an ENOTSUP regression test.

## Best Possible Solution

Refresh or replace this branch on current main, then land one canonical unsupported-hardlink fallback that preserves existing `EEXIST` and stale-lock semantics with real mount proof or an explicit maintainer proof override.

## Reproduction Assessment

Yes, at source level: current main writes the usage-cost cache lock by hard-linking a temp file and rethrows any non-`EEXIST` error, so an `ENOTSUP` from `link` follows the reported failure path. I did not establish a live SMB/NFS/virtiofs mount reproduction in this read-only review.

## Solution Assessment

Yes for the code shape: the fallback is centralized at `writeUsageCostCacheLockAtomically`, preserves existing contention and stale-lock handling, and targets the reported failure. It is not merge-ready until the branch conflict and proof gap are resolved or explicitly overridden.
