## What This Changes

The PR adds a fallback from hard-link lock publication to exclusive lock-file creation when session usage-cost cache lock creation fails on filesystems without hard-link support.

## Best Possible Solution

Land the narrow fallback after the contributor adds real-environment proof that session cost cache refresh works on an unsupported-hardlink filesystem while preserving existing lock contention and cleanup behavior.

## Reproduction Assessment

Yes. Current main has a clear source-level path where `fs.promises.link` errors other than `EEXIST` escape `refreshCostUsageCache`, and the linked crash report gives an `ENOTSUP` lock error under `.usage-cost-cache.json.lock`.

## Solution Assessment

Yes, the proposed code shape is the narrow maintainable fix: keep the existing hard-link path and fall back to `flag: "wx"` only for known unsupported-hardlink errno codes. Merge should still wait for real behavior proof from an affected filesystem class.
