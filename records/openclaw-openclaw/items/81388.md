## What This Changes

Adds a fallback in `writeUsageCostCacheLockAtomically` from `fs.promises.link` to exclusive `writeFile(..., { flag: "wx" })` for known unsupported-hardlink errno codes, plus an `ENOTSUP` regression test.

## Best Possible Solution

Refresh or replace this branch on current main, then land one canonical lock-writer fallback that preserves existing `EEXIST` and stale-lock semantics with real unsupported-filesystem proof or an explicit maintainer proof override.

## Reproduction Assessment

Yes at source level: current main publishes the lock with `fs.promises.link` and rethrows non-`EEXIST` errors, so forcing `link` to reject with `ENOTSUP` reproduces the failure path. I did not establish a live SMB/NFS/virtiofs mount reproduction in this read-only review.

## Solution Assessment

Yes for the code shape: the fallback belongs at `writeUsageCostCacheLockAtomically` and preserves the existing lock contract. It is not merge-ready until the branch is refreshed and the proof gap is resolved or explicitly overridden by maintainers.
