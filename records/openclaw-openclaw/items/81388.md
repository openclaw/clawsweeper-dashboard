This PR is a focused and plausible fix for the current session usage-cost cache lock crash, and I did not find a concrete correctness or security bug in the diff. ## What This Changes

The PR adds an unsupported-hardlink fallback in `src/infra/session-cost-usage.ts` and a regression test that forces `fs.link` to reject with `ENOTSUP`.

## Best Possible Solution

Land the narrow fallback once real-environment proof shows session usage-cost cache refresh works on an unsupported-hardlink filesystem while preserving existing lock contention and cleanup behavior.

## Reproduction Assessment

Yes, by source inspection and the linked crash report. Current main calls `fs.promises.link` during `refreshCostUsageCache`, and only `EEXIST` is handled, so an `ENOTSUP` from the reported mount class escapes the refresh path.

## Solution Assessment

Yes, the proposed code shape is the narrow maintainable fix: keep the existing hard-link path and fall back to exclusive create only for known unsupported-hardlink error codes. Merge should still wait for real behavior proof from an affected filesystem class.
