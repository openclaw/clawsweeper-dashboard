## Summary

Keep open: current main and the latest release still have the hard-link-only usage-cost cache lock path, and this PR remains the only open linked candidate for the canonical bug. The code shape is a focused fix, but the branch is currently dirty against main and the supplied terminal proof stops at a mocked `fs.promises.link` failure rather than patched OpenClaw on an affected filesystem.

## What This Changes

The PR adds a known-error fallback from `fs.promises.link` to exclusive-create `writeFile(..., { flag: "wx" })` for session usage-cost cache refresh locks, plus an `ENOTSUP` regression test.

## Best Possible Solution

Refresh or replace this branch on current main, then land one canonical unsupported-hardlink fallback that preserves existing `EEXIST` and stale-lock semantics with real mount proof or an explicit maintainer proof override.

## Reproduction Assessment

Yes at source level: current main publishes the usage-cost cache lock with `fs.promises.link` and rethrows any non-`EEXIST` error, so an unsupported-link error follows the reported failure path. I did not reproduce the failure on a live SMB/NFS/virtiofs mount in this read-only review.

## Solution Assessment

Yes for the code shape: the fallback is centralized at `writeUsageCostCacheLockAtomically`, preserves the existing busy/stale-lock caller path through `EEXIST`, and avoids changing unrelated cache logic. It is not merge-ready until the dirty branch and proof gap are resolved or explicitly overridden by maintainers.
