## Summary

Keep open: current main and the latest release still scope `OPENCLAW_NO_AUTO_UPDATE` to configured startup/background auto-update applies, while supervised Gateway `update.run` can still create a managed handoff and schedule an `update.run` restart. The remaining question is a maintainer product contract decision, not stale cleanup.

## What This Changes

The issue asks OpenClaw to document and honor a reliable kill-switch for macOS launchd Gateway package/background update restarts before `gateway-update` creates an `update.run` handoff.

## Best Possible Solution

Maintainers should decide the kill-switch scope, then either gate Gateway-originated managed `update.run` handoffs before restart scheduling and update the public docs, or document explicit `update.run` as outside the kill-switch and track launchd handoff reliability under the broader update-run issue.

## Reproduction Assessment

Yes for source-level reproduction: current main still reads `OPENCLAW_NO_AUTO_UPDATE` in startup auto-update code, while the supervised `update.run` managed handoff path has no matching guard. I did not run a live macOS launchd repro in this read-only review.

## Solution Assessment

Unclear: using the existing env kill-switch is a narrow implementation option, but it broadens that variable from configured background auto-updates to explicit control-plane update requests. The best next step is maintainer confirmation of that contract before implementation.
