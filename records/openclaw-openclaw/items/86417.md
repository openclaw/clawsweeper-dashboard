## Summary

Keep open: current main and v2026.6.8 still scope `OPENCLAW_NO_AUTO_UPDATE` to configured startup auto-update applies, while explicit Gateway `update.run` on supervised/global installs can still create the reported `gateway-update` launchd restart handoff without consulting that kill-switch.

## What This Changes

Issue requests a documented, reliable kill-switch for launchd-managed Gateway `gateway-update` restart handoffs, preferably honoring `OPENCLAW_NO_AUTO_UPDATE=1`.

## Best Possible Solution

Define the kill-switch contract, then either make Gateway-originated package update handoffs honor that policy before spawning helpers or scheduling restarts and document it, or explicitly document that `update.run` is outside the auto-update kill-switch.

## Reproduction Assessment

Yes. Source inspection on current main shows a supervised global `update.run` request still enters the managed-service handoff branch and schedules an `update.run` restart, while `OPENCLAW_NO_AUTO_UPDATE` is only consulted by the startup auto-update check.

## Solution Assessment

Unclear. Reusing `OPENCLAW_NO_AUTO_UPDATE` is the narrowest code path, but it would extend a background auto-update kill-switch to explicit control-plane update requests, which needs maintainer product confirmation.
