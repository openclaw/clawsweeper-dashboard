## Summary

Keep open: current main and v2026.6.8 still scope `OPENCLAW_NO_AUTO_UPDATE` to configured startup auto-update applies, while explicit supervised Gateway `update.run` can still create a managed-service handoff and schedule the reported launchd `gateway-update` restart.

## What This Changes

The issue asks OpenClaw to document and honor an auto-update kill switch for macOS launchd Gateway package-update restart handoffs.

## Best Possible Solution

Define the updater kill-switch contract, then either make Gateway-originated package update handoffs honor it before spawning helpers or explicitly document that `update.run` remains outside the auto-update kill switch and leave launchd handoff reliability on the canonical update-run tracker.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: current main reads `OPENCLAW_NO_AUTO_UPDATE` only in startup auto-update checks, while explicit supervised `update.run` still starts managed handoff and schedules `reason: "update.run"`. No fresh live macOS launchd repro was run in this read-only review.

## Solution Assessment

Unclear: reusing the existing kill switch is a small implementation shape, but it changes the contract for explicit control-plane update requests. Maintainers should first decide whether the kill switch blocks only configured automatic applies or all Gateway-originated package update restarts.
