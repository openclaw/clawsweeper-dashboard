## Summary

Keep open: current main still has a source-visible gap between the documented background auto-update kill-switch and the Gateway control-plane `update.run` managed-service handoff path that creates the reported launchd restart handoff.

## What This Changes

The issue requests a documented and reliable operator kill-switch for macOS launchd Gateway update restarts, especially `gateway-update` handoffs created by `update.run`.

## Best Possible Solution

Define the operator policy, then make Gateway-originated package update handoffs honor the chosen kill-switch before spawning helpers or scheduling restarts, and document how launchd-managed services should set it.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: on a global install with a detected supervisor, `update.run` starts a managed-service handoff and schedules an `update.run` restart without checking `OPENCLAW_NO_AUTO_UPDATE`. I did not run a live macOS launchd repro.

## Solution Assessment

Unclear until maintainers decide semantics: reusing `OPENCLAW_NO_AUTO_UPDATE` for Gateway-originated `update.run` handoffs is a narrow implementation, but it changes how an explicit operator/admin update request behaves.
