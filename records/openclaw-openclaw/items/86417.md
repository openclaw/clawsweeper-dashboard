## Label Justifications

- `P2`: The issue describes a bounded but real Gateway availability and operator-control problem under supervised macOS update restarts.
- `impact:crash-loop`: The reported behavior creates unwanted launchd-managed Gateway restarts and intermittent process availability.

## Summary

Keep open: current main and the latest release still scope `OPENCLAW_NO_AUTO_UPDATE` to configured startup auto-update applies, while explicit supervised Gateway `update.run` can still start the managed-service handoff and create the reported `gateway-update` launchd restart handoff.

## What This Changes

Request a documented and reliable way to suppress macOS launchd Gateway package/update restarts, preferably by honoring `OPENCLAW_NO_AUTO_UPDATE=1` before `gateway-update` restart handoffs are created.

## Best Possible Solution

Define the updater kill-switch contract, then either make Gateway-originated package update handoffs honor it before spawning helpers or scheduling restarts and document that behavior, or explicitly document that `update.run` remains outside the auto-update kill-switch and route launchd handoff reliability to the canonical update-run tracker.

## Reproduction Assessment

Yes for source-level reproduction: current main still lets supervised global/git `update.run` start a managed handoff and schedule an `update.run` restart without checking `OPENCLAW_NO_AUTO_UPDATE`. No fresh live macOS launchd repro was run in this read-only review.

## Solution Assessment

Unclear: reusing `OPENCLAW_NO_AUTO_UPDATE` is a small implementation shape, but it changes the contract for explicit control-plane update requests. Maintainers should first decide whether the kill-switch blocks only configured automatic applies or all Gateway-originated package update restarts.
