## Summary

Keep open: current main and v2026.6.6 still scope `OPENCLAW_NO_AUTO_UPDATE` to configured background auto-updates, while the Gateway `update.run` managed-service handoff can still create the reported launchd `gateway-update` restart path without consulting that kill-switch.

## What This Changes

The issue asks OpenClaw to document and honor an auto-update kill-switch for macOS launchd Gateway update restarts.

## Best Possible Solution

Define the operator policy, then make Gateway-originated package update handoffs honor the chosen kill-switch before spawning helpers or scheduling restarts, and document how launchd-managed services should set it.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: on a supervised global install, `update.run` starts a managed-service handoff and schedules an `update.run` restart without checking `OPENCLAW_NO_AUTO_UPDATE`; I did not run a live macOS launchd repro in this read-only review.

## Solution Assessment

Unclear. Reusing `OPENCLAW_NO_AUTO_UPDATE` is a narrow implementation path, but it changes semantics for explicit Gateway control-plane `update.run` requests rather than only configured background auto-updates.
