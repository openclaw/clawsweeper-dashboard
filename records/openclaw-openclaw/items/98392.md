## What This Changes

The issue requests that Control UI ignore stale approval resolve failures unless the same approval is still active on the same gateway client.

## Best Possible Solution

Land a focused fix that captures the resolve client and verifies the same approval is still active before setting `execApprovalError`, with regression tests for newer-approval and reconnect races.

## Reproduction Assessment

Yes, at source level: current main can write `execApprovalError` after a stale resolve failure when the old approval id still exists somewhere in the queue, while the view renders that global error on the active prompt. I did not execute the browser flow in this read-only pass.

## Solution Assessment

Yes, the narrow fix direction is to guard the error write in `handleExecApprovalDecision` by both captured gateway client and active approval id. Moving this into the view would only hide the stale state rather than preventing the wrong state write.
