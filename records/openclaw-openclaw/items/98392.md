- `impact:other`: The issue can mislead operators by showing an approval resolve error on the wrong active Control UI prompt, which is meaningful workflow impact outside the narrower owned labels.

## Summary

Keep open: current main still allows a stale approval resolve failure to write the shared Control UI approval error after the active approval or gateway client changes, and the linked closing PR is open and draft rather than merged.

## What This Changes

Issue reports a Control UI race where a failed approval resolve can attach its stale error to a newer active approval prompt or a replaced gateway client.

## Best Possible Solution

Review and land or close https://github.com/openclaw/openclaw/pull/98394; after a fix merges, close this issue as implemented with the merge commit.

## Reproduction Assessment

Yes, at source level. On current main, if a failed resolve returns after the old approval is no longer the queue head but its id is still present, or after the client is swapped, the handler can still write the shared `execApprovalError`.

## Solution Assessment

Yes. The narrow maintainable fix is to guard stale error writes by both the request client and the active approval id, with regression tests for newer-active approval and reconnect cases.
