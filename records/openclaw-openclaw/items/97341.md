## Summary

Keep open. Current main already gives Slack channel threads separate `:thread:<thread_ts>` sessions and bounded history support, but the requested per-thread customization/resolver semantics are not implemented and need maintainer product/API design rather than cleanup closure.

## What This Changes

Request per-Slack-thread context customization and isolation, including thread-aware routing, history/metadata injection, and per-thread prompts, skills, persona, and tools.

## Best Possible Solution

Define a maintainer-approved Slack plugin/SDK design for exact per-thread matching and context override semantics, reusing current `:thread` session keys and runtime conversation bindings, then implement only the missing prompt/skills/routing gap with docs and focused tests.

## Reproduction Assessment

Not applicable as a feature/product request. Source inspection shows current main already has basic Slack thread session isolation; the remaining request is for new per-thread customization semantics rather than a reproduced current-main bug.

## Solution Assessment

Unclear as proposed. A dedicated resolver may be plausible, but maintainers first need to decide whether per-thread matching belongs in Slack plugin config, runtime conversation bindings, ACP bindings, or a generic Plugin SDK seam.
