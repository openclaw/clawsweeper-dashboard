## Summary

Current main now addresses the reported startup stall shape: `chat.history` no longer waits on model-catalog discovery, regression coverage asserts that contract, and Control UI chat readiness is not gated on the model picker finishing. The earlier keep-open review was valid for older main, but the later gateway/startup changes satisfy the central user-visible problem.

## What This Changes

The issue reports very slow Control UI/WebChat startup where `chat.history` and `models.list` took minutes on low-powered Windows/WSL hosts, leaving chat unusable.

## Best Possible Solution

Keep the current server-side history path that returns stored chat history without full catalog discovery, and treat any remaining cold `models.list` performance concern as a narrower model-catalog issue rather than keeping this startup-blocking report open.

## Reproduction Assessment

Yes. The original report and WSL follow-up provide concrete slow-RPC evidence, and current main has a focused deferred-catalog test that reproduces the old coupling at unit level and proves it no longer occurs.

## Solution Assessment

Yes. Removing the server-side `chat.history` dependency on full model-catalog discovery is the narrowest maintainable fix for chat startup responsiveness; the separate frontend timeout PR can still be reviewed for generic timeout UX, but it is not the canonical fix for this issue.
