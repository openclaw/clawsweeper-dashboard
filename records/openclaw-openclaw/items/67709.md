## Summary

Current main now fixes the reported catalog-coupled startup stall: `chat.history` returns without model-catalog discovery, default/configured `models.list` falls back after a short catalog wait, and both paths have focused regression coverage. The related open frontend timeout PR remains useful for generic never-settling RPC UX, but it is not the canonical fix for this server-side startup coupling.

## What This Changes

The issue reports very slow Control UI/WebChat startup where `chat.history` and `models.list` could take minutes and leave chat unusable.

## Best Possible Solution

Keep the current server-side decoupling and bounded model-list fallback as the resolution for this startup-blocking report, and handle generic UI request-timeout behavior separately through the existing focused frontend timeout work.

## Reproduction Assessment

Yes. The original Windows report and WSL follow-up provide concrete slow-RPC evidence, and current tests reproduce the old failure mode with deferred model-catalog promises for `chat.history` and `models.list`.

## Solution Assessment

Yes. Fixing the server-side catalog coupling is the narrowest maintainable solution for this issue; a client-side timeout alone would mask the symptom without making history and configured model lists return promptly.
