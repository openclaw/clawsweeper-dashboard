## Summary

Keep open. Current main still leaves canonical inbound media refs outside the WebUI assistant-media proxy, and https://github.com/openclaw/openclaw/pull/90172 is an open draft closing PR for the same bug, so this issue should remain open until that fix lands or is replaced.

## What This Changes

The issue asks the WebUI to render inbound channel `media://inbound/<id>` image attachments inline instead of showing an attachment placeholder or unloadable image source.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/90172 or an equivalent narrow renderer fix that recognizes validated `media://inbound/<id>` refs and reuses the existing authenticated assistant-media route, with real Control UI proof before merge.

## Reproduction Assessment

Yes, at source level: current main extracts image `MediaPath` values but only proxies local filesystem-style sources, so a `media://inbound/<id>` image remains an unloadable browser source. I did not run a live Telegram/iMessage reproduction.

## Solution Assessment

Yes: the best fix is to update the WebUI renderer to recognize canonical inbound refs and reuse the existing authenticated, ticketed assistant-media route. Adding a new unauthenticated `/media/inbound` serving path would be a worse security boundary.
