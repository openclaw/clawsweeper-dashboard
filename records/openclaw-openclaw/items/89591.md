## Summary

Keep open. Current main and the latest release still leave `media://inbound/<id>` image refs outside the WebUI assistant-media proxy, and https://github.com/openclaw/openclaw/pull/90172 is an open draft closing PR for this bug.

## What This Changes

The issue requests inline WebUI previews for channel-saved inbound image attachments referenced as `media://inbound/<id>`.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/90172, or an equivalent narrow renderer fix that recognizes validated `media://inbound/<id>` refs and reuses the existing authenticated assistant-media route.

## Reproduction Assessment

Yes, at source level: construct a transcript message with `MediaPath: "media://inbound/<id>"` and image `MediaType`. Current main appends that value as an image URL, then does not classify the media scheme as proxyable through assistant-media.

## Solution Assessment

Yes: the narrow maintainable fix is in the WebUI renderer proxy decision, reusing the existing authenticated, ticketed assistant-media route. Adding a new public `/media/inbound` serving path would be the wrong security boundary.
