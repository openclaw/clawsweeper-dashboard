## Summary

Keep open. Current main and v2026.6.9 still extract `media://inbound/<id>` image refs into the WebUI image path without classifying them as proxyable through assistant-media, and the linked draft PR is still the active candidate fix.

## What This Changes

The issue requests inline WebUI previews for inbound channel image attachments stored as `media://inbound` refs.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/90172, or an equivalent narrow renderer fix that recognizes validated `media://inbound/<id>` refs and reuses the existing authenticated assistant-media route.

## Reproduction Assessment

Yes, at source level: construct a user transcript message with `MediaPath: "media://inbound/<id>.png"` and an image `MediaType`. Current main appends it as an image URL but does not route that scheme through assistant-media.

## Solution Assessment

Yes: the best fix is in the WebUI renderer proxy decision, reusing the existing authenticated, ticketed assistant-media route. Adding a new public `/media/inbound` serving path would be a weaker security boundary.
