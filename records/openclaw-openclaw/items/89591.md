## What This Changes

The issue requests inline WebUI previews for inbound image attachments stored as `media://inbound/...` instead of showing only the attachment placeholder.

## Best Possible Solution

Land a narrow renderer fix that recognizes canonical inbound media refs and routes them through the existing authenticated assistant-media ticket route, with WebUI/Gateway behavior proof.

## Reproduction Assessment

Yes, with high-confidence source inspection: a message with image `MediaPath: "media://inbound/photo.png"` is extracted as an image, but the renderer only proxies file-like local paths. I did not run a live Telegram or iMessage repro in this read-only review.

## Solution Assessment

Yes, the best direction is the narrow UI renderer fix: the gateway already has an authenticated, ticketed assistant-media route for canonical inbound refs, so adding a new unauthenticated media route would be the wrong boundary.
