## Summary

Keep open. Current main still leaves canonical inbound media refs outside the WebUI assistant-media proxy, and https://github.com/openclaw/openclaw/pull/90172 is an open draft closing PR for the same bug, so this issue should remain open until that fix lands or is replaced.

## What This Changes

The issue requests inline WebUI previews for inbound channel image attachments stored as canonical media://inbound refs instead of a text-only attached-image placeholder.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/90172 or an equivalent narrow renderer fix that recognizes validated media://inbound/<id> refs and reuses the existing authenticated assistant-media route.

## Reproduction Assessment

Yes, at source level: construct a transcript message with MediaPath set to media://inbound/<id> and MediaType image/png. Current main appends that path as an image URL, then does not classify the media scheme as proxyable through assistant-media.

## Solution Assessment

Yes: the best fix is to update the WebUI renderer to recognize canonical inbound refs and reuse the existing authenticated, ticketed assistant-media route. Adding a new unauthenticated /media/inbound serving path would be a worse security boundary.
