## Summary

Keep open. Current main still has the Telegram native-quote gate that prevents the answer preview stream from being created, while the Telegram and streaming docs still present tool-progress preview as default-on without the reply-to caveat, and doctor warnings still do not report this config conflict. Open closing PRs #73495 and #73508 already provide candidate fixes, so the issue should stay open until maintainers choose and land one.

## What This Changes

The issue asks OpenClaw to surface Telegram's `replyToMode` and tool-progress preview incompatibility in docs and/or doctor warnings because current runtime silently drops preview tool-progress when native quote replies block answer preview streaming.

## Best Possible Solution

Land the preferred Telegram-scoped clarification, ideally including both user-facing docs and an `openclaw doctor` warning for configs where `replyToMode` can block preview tool-progress. Treat a persistent-message fallback as a separate product/runtime decision if maintainers want tool visibility even with native quote replies enabled.

## Reproduction Assessment

Yes. Configure Telegram preview streaming with `streaming.preview.toolProgress` enabled and `replyToMode` set to a quote-reply mode, then trigger tool use on a quotable inbound message; current source shows no answer-lane stream is created, and the issue discussion confirms this on released builds.

## Solution Assessment

Yes for the near-term direction. A Telegram docs plus doctor-warning fix is the narrowest maintainable solution to the silent doc/runtime mismatch; changing runtime fallback delivery would be larger and should not block documenting and warning about the current limitation.
