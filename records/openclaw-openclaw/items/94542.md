## Label Justifications

- `P2`: This is a credible privacy and agent-correctness improvement with source evidence, but it needs design review rather than urgent regression handling.
- `impact:data-loss`: Per-file channel/chat provenance for passively received media is not durably retained, so source attribution cannot be reconstructed later.
- `impact:security`: The issue concerns sensitive DM and group media being pooled without durable source attribution or separation.
- `impact:session-state`: Flat media storage can let an agent associate the wrong inbound file with the current channel, chat, or user turn.

## Summary

Keep open: current main and v2026.6.8 still use a shared inbound media bucket without durable channel/account/chat/sender provenance, and the broader media-isolation tracker does not fully cover this same-agent DM/group attribution request.

## What This Changes

Request channel/account/chat namespacing or durable provenance for inbound media so DM and group/channel files are separable and auditable.

## Best Possible Solution

Define a maintainer-approved scoped inbound-media contract in core that persists channel/account/chat/sender provenance, preserves existing `media://` references, and updates staging, cleanup, docs, and regression coverage together.

## Reproduction Assessment

Yes at source level: current main saves representative inbound media into the shared inbound bucket, rejects nested managed refs, and has no durable per-file channel/chat/sender provenance table. I did not run a live WhatsApp multi-group setup in this read-only review.

## Solution Assessment

Unclear as proposed: channel-namespaced paths are plausible, but an opt-in toggle and save-path-only change would leave resolver, staging, cleanup, SDK, and privacy semantics incomplete. The best fix is a scoped media/provenance contract across the whole pipeline.
