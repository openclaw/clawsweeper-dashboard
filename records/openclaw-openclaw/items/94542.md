## Summary

Keep open: current main and v2026.6.8 still save representative inbound channel media into a shared `inbound` bucket, reject or collapse nested managed-media refs, and lack a durable channel/account/chat media provenance contract.

## What This Changes

Requested channel/account/chat provenance or namespacing for inbound media so passively received media from DMs and groups is separable on disk or through a persisted index.

## Best Possible Solution

Define one canonical media provenance/index contract before changing storage layout, then update channel save callers, resolvers, staging, docs, and regression tests together.

## Reproduction Assessment

Yes for the source-level behavior: current main still has flat `inbound` save callers and canonical media refs that reject or erase nested paths. I did not run a live multi-channel ingest scenario in this read-only review.

## Solution Assessment

Unclear as requested: a channel-namespaced path or persisted index is plausible, but the best fix needs maintainer direction on identity fields, config/default behavior, migration, and whether this belongs in core media storage or channel/plugin contracts.
