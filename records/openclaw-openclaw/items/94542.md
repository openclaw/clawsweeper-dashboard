## Label Justifications

- `P2`: This is a credible privacy and agent-correctness improvement with source evidence, but it needs design work rather than urgent regression handling.
- `impact:data-loss`: Per-file channel/chat provenance for passively received media is not durably retained, so source attribution cannot be reconstructed later.
- `impact:security`: The issue concerns sensitive DM and group media being pooled without durable source attribution or separation.
- `impact:session-state`: Flat media storage can let an agent associate the wrong inbound file with the current channel, chat, or user turn.

## Summary

Keep open: current main and v2026.6.8 still use shared inbound media storage and do not persist channel/account/chat provenance for stored media. The related isolation and logging reports only partially cover this request, so the remaining work needs maintainer product and security design rather than cleanup closure.

## What This Changes

Request to namespace or durably index inbound media by channel/account/chat so passively received DM and group files remain separable while preserving media:// references.

## Best Possible Solution

Design a durable scoped inbound-media contract, preferably SQLite-backed provenance plus resolver and staging updates, that preserves existing media:// refs while defining channel/account/chat identity and migration semantics.

## Reproduction Assessment

Yes at source level: current main saves representative channel media into the shared inbound bucket, rejects scoped media:// refs, and lacks durable channel/chat/sender columns for media files. I did not run a live multi-channel WhatsApp setup in this read-only review.

## Solution Assessment

Unclear as proposed: channel-namespaced paths are plausible, but an opt-in path-only toggle would leave resolver, staging, cleanup, SDK, and provenance semantics incomplete. The best fix is a maintainer-approved scoped media/provenance contract across the pipeline.
