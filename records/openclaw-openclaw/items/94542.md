## Summary

Keep open: current main and v2026.6.8 still use flat first-level inbound media references and representative bundled channel saves still write into the shared inbound bucket, while the broad media-isolation tracker does not fully cover this issue's channel/chat provenance request.

## What This Changes

The issue requests channel/account/chat provenance or namespacing for inbound media so passively received DM and group media is not stored as one unattributed flat pool.

## Best Possible Solution

Fold this into a maintainer-approved scoped inbound-media design that provides durable SQLite-backed channel/account/chat provenance while preserving existing media:// refs and sandbox staging behavior.

## Reproduction Assessment

Yes, at source level: current main saves representative inbound channel media into the shared inbound bucket, rejects scoped media:// refs, stages sandbox media flat, and has no durable media provenance columns for channel/chat/sender. I did not run a live WhatsApp multi-group setup in this read-only review.

## Solution Assessment

Unclear as proposed: channel-namespaced paths are plausible, but an opt-in toggle and save-path-only change would leave resolver, staging, cleanup, SDK, and privacy semantics incomplete. The best fix is a maintainer-approved scoped media/provenance contract across the pipeline.
