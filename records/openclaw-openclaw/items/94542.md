## Summary

Keep open: current main and v2026.6.9 still use flat inbound media storage and first-level managed media refs without a durable channel/account/chat provenance contract. The open broad tracker https://github.com/openclaw/openclaw/issues/19330 and Feishu slice https://github.com/openclaw/openclaw/issues/60154 overlap, but neither fully replaces this issue's channel/chat attribution and media-index request.

## What This Changes

The issue requests channel/account/chat-namespaced or indexed inbound media storage so passive media files retain durable provenance instead of landing in one flat inbound directory.

## Best Possible Solution

Define one canonical scoped media provenance contract, preferably SQLite-backed/indexed, that preserves existing media:// refs and updates save, resolve, staging, cleanup, docs, and channel callers together.

## Reproduction Assessment

Yes at source level. Current main and v2026.6.9 save representative inbound media into the shared inbound bucket, reject nested managed refs, and lack durable media provenance fields; I did not run a live multi-channel ingest scenario.

## Solution Assessment

Unclear as proposed. Channel-namespaced paths and a persisted index are both plausible, but the best solution needs a maintainer-chosen schema, privacy/retention rules, URI compatibility, and migration/default posture.
