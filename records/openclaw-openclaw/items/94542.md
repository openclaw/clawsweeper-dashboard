## Label Justifications

- `P2`: This is a credible privacy and agent-correctness improvement with source evidence, but it needs maintainer design rather than urgent regression handling.
- `impact:data-loss`: Per-file channel/chat provenance for passively received media is not durably retained, so source attribution cannot be reconstructed later.
- `impact:security`: The issue concerns sensitive DM and group media being pooled without durable source attribution or separation.
- `impact:session-state`: Flat media storage can let an agent associate the wrong inbound file with the current channel, chat, or user turn.

## Summary

Keep open: current main and v2026.6.8 still save representative inbound channel media into the shared inbound media bucket, and the remaining work needs maintainer product/security design rather than cleanup closure.

## What This Changes

The issue requests channel/account/chat provenance or namespacing for passively received inbound media so DM and group media can be separated on disk or through a durable index.

## Best Possible Solution

Design a durable scoped media/provenance contract across the media store, resolver, sandbox staging, SDK, and channel callers, preferably SQLite-backed and compatible with existing media:// refs.

## Reproduction Assessment

Yes, at source level: current main saves representative channel media into the shared inbound bucket, rejects nested media:// inbound IDs, and lacks durable per-file channel/chat/sender provenance columns. I did not run a live multi-group WhatsApp setup in this read-only review.

## Solution Assessment

Unclear as proposed: channel-namespaced paths are plausible, but an opt-in toggle or save-path-only change would leave resolver, staging, cleanup, SDK, and privacy semantics incomplete. The maintainable solution is a maintainer-approved scoped media/provenance contract.
