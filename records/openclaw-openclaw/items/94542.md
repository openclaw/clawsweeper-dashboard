## Label Justifications

- `P2`: This is a credible privacy and agent-correctness improvement with source evidence, but it needs bounded maintainer design rather than urgent regression handling.
- `impact:data-loss`: Per-file channel/chat provenance for passively received media is not durably retained, so source attribution cannot be reconstructed later.
- `impact:security`: The issue concerns sensitive DM and group media being pooled without durable source attribution or separation.
- `impact:session-state`: Flat media storage can let an agent associate the wrong inbound file with the current channel, chat, or user turn.

## Summary

Keep open: current main and the latest release still use a flat shared inbound media path with no durable channel/chat/sender provenance, while the broader media-isolation tracker does not fully cover this issue's same-agent DM/group attribution request.

## What This Changes

The issue requests channel/account/chat namespacing or a durable provenance index for passively received inbound media.

## Best Possible Solution

Define a maintainer-approved scoped inbound-media contract that stores durable channel/account/chat provenance, preserves existing media:// references through a resolver or index, and updates staging, cleanup, access checks, docs, and regression coverage together.

## Reproduction Assessment

Yes at source level: current main saves representative inbound channel media into the shared inbound bucket, rejects scoped media:// refs, and has no durable media provenance columns for channel/chat/sender. I did not run a live WhatsApp multi-group setup in this read-only review.

## Solution Assessment

Unclear as proposed: channel-namespaced paths are plausible, but an opt-in toggle and save-path-only change would leave resolver, staging, cleanup, SDK, and privacy semantics incomplete. The best fix is a maintainer-approved scoped media/provenance contract across the pipeline.
