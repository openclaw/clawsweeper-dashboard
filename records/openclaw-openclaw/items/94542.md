## Summary

Keep open: current main and v2026.6.9 still use a flat first-level inbound media store and do not provide durable channel/account/chat provenance for passively received media.

## What This Changes

The issue requests channel/account/chat provenance or namespacing for inbound media so files from DMs and group channels are not pooled in one flat inbound media store.

## Best Possible Solution

Define one canonical media provenance/index contract first, then update storage, resolver/staging, channel save callers, docs, and regression tests together with an explicit privacy and migration decision.

## Reproduction Assessment

Yes for the source-level behavior: current main still has flat `inbound` save callers, rejects nested managed-media refs, and lacks a durable channel/account/chat media provenance table. I did not run a live multi-channel ingest scenario in this read-only review.

## Solution Assessment

Unclear as requested: channel-namespaced paths or a persisted index are plausible, but the best fix needs maintainer direction on identity fields, resolver compatibility, storage migration, retention, and whether the proposed config knob should exist.
