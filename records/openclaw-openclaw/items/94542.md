## Summary

Keep open: current main and v2026.6.10 still use flat inbound media storage, first-level `media://inbound/<id>` references, and no durable channel/account/chat/sender provenance contract. The gap is source-backed, but the fix needs maintainer product/security direction because it changes durable media storage, resolver compatibility, and privacy-sensitive metadata.

## What This Changes

The issue requests durable channel/account/chat provenance or namespacing for inbound media files so passively received media from DMs and allow-listed groups is not stored as one flat pool.

## Best Possible Solution

Define one canonical SQLite-backed media provenance/index contract with explicit privacy and retention rules, then update media storage, resolvers, channel callers, sandbox staging, cleanup, docs, and focused regression tests together.

## Reproduction Assessment

Yes. Source inspection shows current main and v2026.6.10 still save and resolve inbound media through a flat `inbound` bucket without durable channel/account/chat provenance; I did not run a live multi-channel ingest scenario.

## Solution Assessment

Unclear. Channel-namespaced paths and a persisted media index are both plausible, but maintainers need to choose the durable schema, privacy rules, resolver compatibility, migration/default posture, and whether a new config toggle should exist.
