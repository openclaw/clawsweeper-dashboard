## Label Justifications

- `P2`: The report describes a normal-priority media isolation and provenance improvement with limited blast radius but real privacy and attribution impact.
- `impact:data-loss`: Passive inbound files lack durable channel/chat provenance, so source attribution can be lost and cannot be reconstructed later.
- `impact:security`: The shared inbound media pool can mix sensitive DM and group media from different trust contexts.
- `impact:session-state`: Flat media storage can let an agent associate the wrong inbound file with the current channel, chat, or user turn.

## Summary

Keep open: current main and the latest release still use a flat shared inbound media path and do not persist channel/chat/sender provenance for passive media. The broader media-isolation tracker overlaps, but it does not fully cover this issue's same-agent DM/group channel-provenance request, so this needs maintainer product and security design rather than cleanup closure.

## What This Changes

The issue requests channel/account/chat namespacing or persisted provenance for passively received inbound media so DM and group files are separable on disk or via an index.

## Best Possible Solution

Design a scoped inbound-media/provenance contract that records channel/account/chat identity durably, preserves existing `media://` references and flat-file compatibility through migration or resolver support, and updates sandbox staging, cleanup, docs, SDK behavior, and representative channel callers together.

## Reproduction Assessment

Yes, at source level: current main saves representative inbound channel media into the shared `inbound` bucket, rejects scoped `media://inbound/<channel>/<chat>/<id>` references, and has no durable channel/chat/sender media index. I did not run a live multi-channel WhatsApp reproduction in this read-only review.

## Solution Assessment

Unclear as proposed: channel-namespaced paths or a media index are plausible, but the durable identity, migration, resolver, cleanup, sandbox staging, and SDK contract need maintainer design first. A save-path-only toggle would not be the best fix.
