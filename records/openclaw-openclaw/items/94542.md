## Summary

Keep open: current main and v2026.6.9 still use shared flat inbound media storage, first-level `media://inbound/<id>` references, and a `media_blobs` schema without durable channel/account/chat/sender provenance.

## What This Changes

Requested channel-namespaced or durably indexed inbound media storage so media can be separated by channel/account/chat instead of pooling in a flat inbound directory.

## Best Possible Solution

Define one canonical media provenance/index contract, then update storage, resolvers, channel callers, sandbox staging, docs, and regression tests together.

## Reproduction Assessment

Yes. Source inspection shows current main and v2026.6.9 still save and resolve inbound media through a flat `inbound` bucket without a durable channel/account/chat/sender provenance table; I did not run a live multi-channel ingest scenario.

## Solution Assessment

Unclear as proposed. Channel-namespaced paths or a persisted media index are plausible, but maintainers need to choose the durable schema, privacy rules, compatibility path, and config/default posture first.
