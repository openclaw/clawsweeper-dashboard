## Summary

Keep open: current main and v2026.6.9 still use flat inbound media storage and first-level `media://inbound/<id>` references without a durable channel/account/chat provenance contract.

## What This Changes

The issue requests channel/account/chat provenance or namespaced storage for inbound media so DM and group media are separable and attributable after ingest.

## Best Possible Solution

Define one canonical media provenance/index contract, then update storage, resolvers, channel callers, sandbox staging, docs, and regression tests together.

## Reproduction Assessment

Yes. Source inspection shows current main and v2026.6.9 still save and resolve inbound media through a flat `inbound` bucket without a durable provenance table; I did not run a live multi-channel ingest scenario.

## Solution Assessment

Unclear as proposed. Channel-namespaced paths or a persisted media index are plausible, but maintainers need to choose the durable schema, privacy rules, compatibility path, and config/default posture first.
