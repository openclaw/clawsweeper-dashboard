## Summary

Keep open: current main and v2026.6.9 still use a shared flat inbound media bucket and first-level `media://inbound/<id>` contract, with no durable channel/account/chat/sender provenance for passive inbound media.

## What This Changes

The issue requests channel/account/chat-scoped inbound media storage or a persisted media provenance index so DM and group/channel attachments can be separated and attributed later.

## Best Possible Solution

Define one canonical SQLite-backed media provenance/index contract, then update channel ingest callers, media resolvers, sandbox staging, cleanup/retention, docs, and regression tests around that contract.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main and v2026.6.9 save and resolve inbound media through a shared first-level `inbound` bucket without durable channel/account/chat provenance. I did not run a live multi-channel ingest scenario.

## Solution Assessment

Unclear as proposed. Channel-namespaced paths and a persisted index are both plausible, but maintainers need to choose the durable schema, privacy/retention rules, default/config posture, and `media://` compatibility contract first.
