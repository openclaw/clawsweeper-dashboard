## Label Justifications

- `P2`: This is a credible privacy and agent-correctness improvement with source evidence, but it needs bounded maintainer design rather than urgent regression handling.
- `impact:security`: The issue concerns sensitive DM and group media being pooled without durable source attribution or separation.
- `impact:session-state`: Flat media storage can let an agent associate the wrong inbound file with the current channel, chat, or user turn.
- `impact:data-loss`: Per-file channel/chat provenance for passively received media is not durably retained, so source attribution cannot be reconstructed later.

## Summary

Keep open: current main and v2026.6.8 still have a shared inbound media pipeline, and the existing broad media-isolation tracker does not fully cover this issue's channel/chat provenance request.

## What This Changes

Request channel/account/chat-scoped inbound media storage or persisted provenance so passive DM and group media are separable and resolvable by source.

## Best Possible Solution

Fold this into the scoped inbound-media design as channel/account/chat provenance, with a durable SQLite-backed index or scoped media contract that preserves existing media:// refs and staging behavior.

## Reproduction Assessment

Yes at source level: current main saves representative inbound channel media into the shared inbound bucket, rejects scoped media:// refs, and has no durable media provenance columns for channel/chat/sender. I did not run a live WhatsApp multi-group setup in this read-only review.

## Solution Assessment

Unclear as proposed: channel-namespaced paths are a plausible direction, but an opt-in toggle and save-path-only change would leave resolver, staging, cleanup, SDK, and privacy semantics incomplete. The best fix is a maintainer-approved scoped media/provenance contract across the pipeline.
