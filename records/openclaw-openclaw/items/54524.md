## Summary

Keep open. Current main and the latest shipped tag still allow the reported overlap: session transcript indexing can run beside workspace memory indexing, the bundled hook still writes session-derived Markdown into that memory directory, and the public docs still do not warn users about duplicate session-derived results.

## What This Changes

Clarify or otherwise avoid duplicate indexing when the session-memory hook writes summaries into workspace memory while memorySearch also indexes raw session transcripts.

## Best Possible Solution

Add concise public docs guidance in the memory config and session-memory hook docs recommending either memory-file summaries or raw session transcript indexing unless the user intentionally wants both; leave runtime dedupe and output-path design to separate tracked work.

## Reproduction Assessment

Yes, at source level. Enabling `experimental.sessionMemory` with `sources: ["memory", "sessions"]` while enabling the `session-memory` hook leaves two current-main indexing paths for the same conversation-derived content.

## Solution Assessment

Yes for the immediate issue. A narrow docs repair is the lowest-risk solution; provenance-aware dedupe or configurable hook output paths are larger follow-ups already represented by related work.
