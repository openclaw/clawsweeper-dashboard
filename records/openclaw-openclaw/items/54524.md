## Summary

Keep open. Current main and the latest release still allow the reported overlap: `memorySearch` can index both workspace memory files and raw session transcripts, the bundled `session-memory` hook still writes session-derived Markdown into workspace memory, and public docs still describe the features separately without warning users about duplicate session-derived content.

## What This Changes

The issue requests documentation or design guidance to avoid indexing the same session-derived content through both raw session transcripts and session-memory hook output.

## Best Possible Solution

Add concise public docs guidance in the memory search/config and session-memory hook docs that these mechanisms overlap, recommending either memory-file summaries or raw session transcript indexing for the same conversation unless the user explicitly wants both.

## Reproduction Assessment

Yes, at source level. Enabling `experimental.sessionMemory` with `sources: ["memory", "sessions"]` and enabling the `session-memory` hook lets two independent current-main indexing paths ingest session-derived content.

## Solution Assessment

Yes for the immediate issue. A narrow docs repair is the lowest-risk fix; runtime provenance dedupe or configurable hook output paths can stay separate follow-up work.
