## Summary

Keep open. Current main still permits the reported overlap: session transcript indexing and the bundled session-memory hook can both feed the memory index, and the docs still do not warn users to choose one path or separate the outputs.

## What This Changes

The issue asks for documentation or design guidance to avoid duplicate memory indexing when `memorySearch.sources` includes `sessions` while the session-memory hook writes session summaries into workspace memory.

## Best Possible Solution

Add public guidance in the memory config/search docs and session-memory hook docs explaining the overlap, recommending `sources: ["memory"]` when relying on the hook or disabling/redirecting the hook when indexing raw session transcripts. A later runtime change can add provenance-aware dedupe or configurable hook output without changing current defaults.

## Repair Work Prompt

Explain that `memorySearch.experimental.sessionMemory` with `sources: ["memory", "sessions"]` indexes raw session transcripts, while the bundled `session-memory` hook writes session-derived markdown under `<workspace>/memory`, which is indexed by the normal memory source. Coordinate wording with related PR #40618, but do not implement runtime dedupe, change defaults, add config schema, or change hook output behavior in this repair.
