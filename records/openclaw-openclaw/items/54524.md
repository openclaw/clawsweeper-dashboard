## Summary

Keep open. Current main and v2026.5.18 still allow the reported overlap, and the public docs still do not warn users that session-memory hook output and raw session transcript indexing can duplicate conversation-derived memory-search results.

## What This Changes

The issue requests docs/design guidance or deduplication for combining the session-memory hook with `memorySearch` session transcript indexing.

## Best Possible Solution

Add concise public guidance in the memory config and session-memory hook docs recommending either memory-file summaries or raw session transcript indexing unless users intentionally want both.

## Reproduction Assessment

Yes, at source level. Enabling `experimental.sessionMemory` with `sources: ["memory", "sessions"]` while enabling the session-memory hook leaves two current-main indexing paths for the same conversation-derived content.

## Solution Assessment

Yes for the immediate issue. A narrow docs repair is the lowest-risk fix; provenance-aware query dedupe or configurable hook output paths are larger follow-ups.
