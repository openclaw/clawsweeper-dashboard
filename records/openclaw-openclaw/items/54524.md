## Summary

Keep open. Current main still permits the reported overlap, the latest shipped tag has the same behavior, and the public docs still do not warn users that session-memory hook output and raw session transcript indexing can duplicate conversation-derived memory-search results.

## What This Changes

The issue requests docs/design guidance for avoiding duplicate memory-search indexing when the session-memory hook and experimental session transcript indexing are enabled together.

## Best Possible Solution

Add concise public docs guidance in the memory-search/config docs and session-memory hook docs recommending either memory-file summaries or raw session transcript indexing unless users intentionally want both.

## Reproduction Assessment

Yes, at source level. Enabling `experimental.sessionMemory` with `sources: ["memory", "sessions"]` while enabling the session-memory hook leaves two current-main indexing paths for the same conversation-derived content.

## Solution Assessment

Yes for the immediate issue. A narrow docs repair is the lowest-risk fix; provenance-aware query dedupe or configurable hook output paths are larger follow-ups already represented by related work.
