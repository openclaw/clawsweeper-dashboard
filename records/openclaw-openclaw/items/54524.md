## Summary

Keep open. Current main still permits the reported overlap between session transcript indexing and session-memory hook Markdown indexing, and the public docs still do not warn users that enabling both paths can duplicate session-derived memory-search results and embeddings.

## What This Changes

The issue requests docs or design guidance for avoiding duplicate memory-search indexing when `sources` includes both `memory` and `sessions` while the bundled `session-memory` hook writes summaries into workspace memory.

## Best Possible Solution

Add concise public guidance to the memory config and session-memory hook docs explaining that raw session transcript indexing and session-memory hook summaries overlap, with recommended configurations for choosing one path unless users intentionally want both.

## Reproduction Assessment

Yes, at source level. Current main still permits `sources: ["memory", "sessions"]`, the bundled hook still writes session summaries under workspace `memory/`, and sync/search code treats those paths as independent indexed sources.

## Solution Assessment

Yes for the immediate issue. A narrow docs repair is the lowest-risk maintainable fix; provenance-aware query dedupe or configurable hook output belongs as a separate product/design follow-up.
