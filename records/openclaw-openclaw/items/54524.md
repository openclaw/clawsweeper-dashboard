## Summary

Keep open. Current main still allows the reported overlap between raw session transcript indexing and session-memory hook Markdown indexing, and the public docs still do not warn users that enabling both paths can duplicate session-derived memory-search results and embedding work.

## What This Changes

The issue requests documentation or design guidance for avoiding duplicate memory-search indexing when both raw session indexing and the session-memory hook are enabled.

## Best Possible Solution

Add concise public guidance to the memory config and session-memory hook docs explaining the overlap and recommending one path unless users intentionally want both.

## Reproduction Assessment

Yes, at source level. Current main still permits memory and sessions sources together, writes session-memory summaries into workspace memory, and indexes those directories as independent sources.

## Solution Assessment

Yes for the immediate issue. A narrow docs change is the lowest-risk maintainable fix; provenance-aware query dedupe or configurable hook output can remain a separate product/design follow-up.

## Repair Work Prompt

Add a narrow docs-only fix for https://github.com/openclaw/openclaw/issues/54524. Document that `memorySearch.sources` containing both `memory` and `sessions` with `memorySearch.experimental.sessionMemory: true` overlaps with the bundled `session-memory` hook because the hook writes session-derived Markdown under workspace `memory/`, which is also indexed by the `memory` source. Add concise guidance to the memory config docs and session-memory hook docs recommending `sources: ["memory"]` when relying on the hook, or disabling/redirecting the hook when indexing raw sessions, unless users intentionally want both.
