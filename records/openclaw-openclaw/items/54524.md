## Summary

Keep open. Current main still permits the reported overlap: transcript indexing and workspace memory indexing can both be enabled, the session-memory hook writes transcript-derived Markdown into the workspace memory tree, and the docs still do not warn users that combining the two paths can duplicate session-derived content.

## What This Changes

Document that `memorySearch` session transcript indexing and the bundled `session-memory` hook overlap, and guide users to use one mechanism or an explicit dedupe/design path.

## Best Possible Solution

Add public docs guidance in the memory search/config docs and session-memory hook docs explaining the overlap and recommending either `sources: ["memory"]` with the hook or transcript indexing without the hook-generated memory summary path.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: enable `experimental.sessionMemory` with both `memory` and `sessions` sources, enable the bundled session-memory hook, and the two independent index paths can ingest session-derived content separately.

## Solution Assessment

Yes for the immediate report. A narrow docs repair is the lowest-risk maintainable fix; runtime provenance-aware dedupe or an output-directory option can remain separate follow-up work.

Fix the docs gap for https://github.com/openclaw/openclaw/issues/54524. Add guidance that `memorySearch.experimental.sessionMemory` with `sources: ["memory", "sessions"]` overlaps with the bundled `session-memory` hook because the hook writes session-derived Markdown under `<workspace>/memory/`, which the memory source indexes separately from raw session transcripts. Update the memory search/config docs and session-memory hook docs so users are told to use `sources: ["memory"]` when relying on the hook, or disable/redirect the hook when they want raw transcript indexing.
