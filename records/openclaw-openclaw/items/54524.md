## Summary

Keep open. Current main still allows the reported overlap: memory search can index both workspace memory files and session transcripts, the bundled session-memory hook still writes session-derived Markdown into workspace memory, and the docs do not warn users not to combine those mechanisms.

## What This Changes

The issue asks OpenClaw to document or prevent duplicate indexing when session transcript indexing and the bundled session-memory hook both cover the same conversation.

## Best Possible Solution

Add public guidance in the memory search/config docs and session-memory hook docs explaining the overlap: use `sources: ["memory"]` when relying on the hook, or disable/redirect the hook when indexing raw session transcripts. A later runtime change can add provenance-aware dedupe or configurable output without changing current defaults.

## Reproduction Assessment

Yes. The source path is high-confidence without a live run: enable `memorySearch.experimental.sessionMemory` with `sources: ["memory", "sessions"]`, enable the bundled session-memory hook, and current sync/index code will treat the hook artifact and transcript as separate source entries.

## Solution Assessment

Yes for the first repair step. A narrow docs fix is the safest maintainable fix for the reported documentation/design gap; #40618's output-directory mitigation is useful but partial, while runtime dedupe needs a separate provenance design.
