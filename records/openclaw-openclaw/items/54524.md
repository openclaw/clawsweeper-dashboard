## Summary

Keep open. Current main and the latest shipped tag still allow the reported overlap: raw session transcript indexing can run alongside workspace memory-file indexing, the session-memory hook still writes session-derived Markdown into that memory tree, and the public docs still do not warn users that enabling both can duplicate results and embedding work.

## What This Changes

The issue asks for docs/design guidance to prevent duplicate memory indexing when raw session transcript indexing is enabled alongside the session-memory hook.

## Best Possible Solution

Add concise public docs guidance in the memory config/search docs and session-memory hook docs recommending either raw session transcript indexing or hook-written memory summaries unless the user intentionally wants both.

## Reproduction Assessment

Yes, at source level. Enabling `experimental.sessionMemory` with `sources: ["memory", "sessions"]` while the `session-memory` hook writes into `<workspace>/memory` leaves two independent indexing paths for the same conversation-derived content.

## Solution Assessment

Yes for the immediate issue. A narrow docs repair is the lowest-risk solution; provenance-aware runtime dedupe or configurable hook output paths are larger follow-ups already represented by related work.
