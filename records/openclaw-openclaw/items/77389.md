## Summary

Keep open: current main still keeps Telegram draft message IDs only in per-process stream state, so a restart can lose the ID needed to delete a stale progress or tool-progress preview. Current main covers same-process cleanup, and the related closed PR at https://github.com/openclaw/openclaw/pull/81486 addressed final-delivery cleanup, but neither adds persisted pending-draft cleanup across process restart.

## What This Changes

The issue asks Telegram progress and tool-progress draft previews to be cleared or replaced correctly after an OpenClaw restart instead of persisting and recurring during later command steps.

## Best Possible Solution

Telegram should persist only unfinished progress/tool-progress preview message IDs with their chat/thread context and clear those stale pending drafts on startup or the next dispatch, while removing the record as soon as the draft is finalized or deleted.

## Reproduction Assessment

Yes, at source level: create a Telegram draft preview, restart before cleanup, and the new process has no streamMessageId for clear() to delete. I did not run a live Telegram bot restart in this read-only review.

## Solution Assessment

Yes: the narrow maintainable fix is Telegram-owned pending-draft persistence and stale cleanup with restart coverage. Deleting arbitrary old Telegram messages without a stored pending ID would be riskier.
