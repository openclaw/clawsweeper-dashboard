## Summary

Keep open: current main still keeps Telegram draft preview IDs in process-local state, so a restart or lost-ID path can leave a visible progress/tool-status draft without a proven cleanup or final replacement despite recent adjacent cleanup.

## What This Changes

The issue asks Telegram preview/progress drafts to update one stable message, finalize into the answer, and avoid persisting or stacking after an OpenClaw restart.

## Best Possible Solution

Harden the Telegram plugin preview lifecycle so lost or stale draft IDs after restart cannot suppress final delivery, and clean up any known active or archived preview IDs with focused regression coverage.

## Reproduction Assessment

No live high-confidence reproduction was run. Source inspection on current main shows a credible restart/lost-ID path because Telegram draft IDs are in-memory only and cleanup/finalization depends on knowing a preview message ID.

## Solution Assessment

No current-main solution is proven sufficient. The narrow maintainable fix is in the Telegram plugin draft/finalization lifecycle, with tests for restart-like ID loss and fallback final delivery rather than a new config workaround.
