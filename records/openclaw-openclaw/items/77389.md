## Summary

Keep open: current main still has Telegram draft cleanup/finalization paths that depend on process-local preview IDs, while the documented behavior requires a single draft to become the final answer or fall back to normal final delivery. No later code or regression coverage proves the restart/lost-ID phantom draft path is fixed.

## What This Changes

The issue asks for Telegram streaming/progress drafts to clear or finalize correctly after an OpenClaw restart instead of leaving a persistent `Sifting...` preview through later tool steps.

## Best Possible Solution

Harden the Telegram plugin draft lifecycle so known previews are cleaned up and lost-ID or restart-like paths always produce a final answer without stacking progress drafts.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection provides a credible lost-ID path because Telegram preview IDs are process-local and finalization can retain a missing-ID preview instead of sending a fresh final answer.

## Solution Assessment

No current-main fix is proven sufficient. The narrow maintainable solution is a Telegram plugin lifecycle repair with restart/lost-ID regression tests, not a new configuration workaround.
