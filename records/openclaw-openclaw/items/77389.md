## Summary

Keep open: current main has a same-process Telegram progress-draft cleanup path, but the restart/lost-ID part of the report remains source-visible because Telegram draft message IDs are still held in per-process stream state. The closed related PR at https://github.com/openclaw/openclaw/pull/81486 addressed active/retained final-delivery cleanup, not persisted cleanup across process restart.

## What This Changes

Telegram progress drafts should clear or finalize correctly across process restart instead of leaving a stale “Sifting...” draft and creating repeated progress bubbles on later turns.

## Best Possible Solution

Telegram should track unfinished progress draft message IDs across process lifetime and clear only stale unfinished drafts on startup or the next dispatch, while preserving finalized replies.

## Reproduction Assessment

Yes, at source level: send a Telegram progress draft, restart before cleanup, and the new process has no streamMessageId for clear() to delete. I did not run a live Telegram bot restart in this read-only review.

## Solution Assessment

Yes, the narrow maintainable direction is a Telegram-owned pending-draft ID cleanup path with restart tests. Fetching or deleting arbitrary old Telegram messages without stored IDs would be less reliable and riskier.
