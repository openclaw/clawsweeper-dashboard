## Summary

Keep open: current main still keeps Telegram preview message IDs in process-local turn state, so a restart or lost-ID path can leave a visible progress/tool-status draft outside the cleanup and finalization paths. Recent cleanup covers tool-only assistant-boundary previews, but not recovery after process restart.

## What This Changes

The issue requests that Telegram progress/tool-status drafts created during partial/progress streaming be finalized or cleared correctly after an OpenClaw restart.

## Best Possible Solution

Harden the Telegram plugin draft lifecycle so known active or archived preview IDs survive the relevant restart/lost-ID boundary, then ensure final delivery edits, deletes, or replaces that preview with focused regression coverage.

## Reproduction Assessment

No live high-confidence Telegram restart reproduction was run. Source inspection gives a credible current-main path because preview IDs and archived previews are process-local while cleanup can only delete IDs known to the current stream instance.

## Solution Assessment

Unclear whether persistence, restart recovery, or a narrower lost-ID cleanup is the best implementation detail. The maintainable direction is still a Telegram-plugin lifecycle repair with regression tests, not a config workaround or core migration.
