## Summary

Keep open: current main has same-process Telegram progress/tool-progress draft cleanup, but the restart case remains source-visible because active draft message IDs are still held in per-process stream state and are lost across process restart. The related closed PR addressed final-delivery cleanup, not persisted cleanup of an unfinished draft after restart.

## What This Changes

The issue requests that Telegram progress/tool-progress drafts not survive an OpenClaw restart or stack on later turns, and instead be cleared or replaced when final delivery completes.

## Best Possible Solution

Telegram should persist only unfinished progress/tool-progress preview message IDs per account/chat/thread and clear those stale drafts on startup or the next dispatch, while removing records once the draft is finalized or cleared.

## Reproduction Assessment

Yes, at source level: send a Telegram progress/tool-progress draft, restart before clear/final cleanup, and the new process has no streamMessageId for clear() to delete the old Telegram message. I did not run a live Telegram restart in this read-only pass.

## Solution Assessment

Yes, the narrow maintainable fix is a Telegram-owned pending-draft ID cleanup path with restart coverage. Deleting arbitrary old Telegram messages by content or time would be riskier than clearing only persisted unfinished draft IDs.
