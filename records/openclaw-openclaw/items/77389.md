## Summary

Keep open: current main fixes same-process Telegram progress draft cleanup before the final reply, but the reported restart leak is still source-reproducible because pending draft message IDs remain process-local and are not persisted for cleanup after a restart.

## What This Changes

The issue asks OpenClaw to prevent Telegram progress drafts such as "Sifting..." from persisting or stacking after the bot process restarts.

## Best Possible Solution

Add Telegram-owned persistence for unfinished progress/tool-progress draft message IDs with chat/thread context, clear those stale pending drafts on startup or the next dispatch, and remove the record immediately when the draft is finalized or deleted.

## Reproduction Assessment

Yes, at source level: send a Telegram progress draft, restart before cleanup, and the new process has no streamMessageId to delete. I did not run a live Telegram bot restart in this read-only review.

## Solution Assessment

No, current main is only a partial solution: it clears active same-process progress drafts before the final reply, but it does not persist pending draft IDs for restart cleanup. The narrow maintainable fix is a Telegram-local pending-draft store and stale cleanup path.
