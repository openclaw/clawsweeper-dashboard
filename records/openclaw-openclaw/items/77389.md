## Summary

Keep open: current main still keeps Telegram draft preview IDs in process-local state, and cleanup/finalization still depends on knowing a preview message ID, so the reported restart/lost-ID phantom draft path remains credible.

## What This Changes

The issue reports that Telegram progress/tool-progress drafts such as `Sifting...` persist and repeat after restarting OpenClaw instead of finalizing or clearing.

## Best Possible Solution

Harden the Telegram plugin so lost or stale draft IDs after restart cannot suppress final delivery, and clean up any known active or archived preview IDs with focused regression coverage.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection gives a credible current-main reproduction path because Telegram preview IDs are in-memory only and final cleanup/fallback behavior still depends on knowing or safely classifying the preview ID.

## Solution Assessment

No current-main solution is proven sufficient. The narrow maintainable solution is a Telegram-plugin draft/finalization lifecycle fix with restart-like lost-ID tests, not a new config workaround.
