## Summary

This should stay open: current main still has a plausible Telegram-plugin lifecycle bug where progress drafts are tracked only in memory, so a process restart can leave a previously sent progress message outside the cleanup/finalization path while later progress-mode turns create new editable drafts.

## What This Changes

The issue asks for Telegram `streaming.mode: "progress"` drafts to be cleared or finalized correctly after an OpenClaw restart instead of persisting or stacking during later commands.

## Best Possible Solution

Fix the Telegram plugin’s draft lifecycle so restart or lost-preview-id cases do not leave stale progress messages visible, with a focused regression test around progress mode, lost in-memory draft state, and final delivery.

## Reproduction Assessment

Yes from source inspection, but not from a live end-to-end run. The current Telegram draft stream loses `streamMessageId` on restart, making cleanup/finalization unable to target a pre-restart draft while later progress-mode work can still create new drafts.

## Solution Assessment

Yes, the narrow maintainable fix belongs in the Telegram plugin’s draft/progress lifecycle rather than in core config migration or a new user setting. A repair should preserve progress mode and make stale pre-restart drafts harmless or recoverable.
