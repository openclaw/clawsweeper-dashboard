## Summary

Keep open: current `main` has recent Telegram preview cleanup, but the restart/lost-message-id path is still not proven fixed and source inspection shows credible ways for a visible draft to survive or be treated as delivered without a final replacement.

## What This Changes

The issue reports that Telegram partial-streaming progress drafts persist after an OpenClaw restart and reappear during later execution steps instead of being finalized or cleared.

## Best Possible Solution

Keep the issue open and make Telegram draft finalization robust when preview ids are missing or stale, falling back to a fresh final response while deleting any known archived previews.

## Reproduction Assessment

No live high-confidence reproduction was run. Source inspection does show a credible current-main path where process-local draft ids are lost and the finalizer can retain a visible draft instead of delivering a fresh final response.

## Solution Assessment

No: treating a missing preview id as delivered is not the safest solution for this report. The narrower maintainable fix is to fall back to final delivery when the active Telegram preview cannot be identified, while preserving the existing duplicate-avoidance behavior for known ambiguous sends.
