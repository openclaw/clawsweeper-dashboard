## Summary

Keep this open. Current main still skips unchanged daily memory files once `memory/.dreams/daily-ingestion.json` has the same mtime/size fingerprint, so daily dreaming signals do not accrue on later sweeps and default deep promotion can remain stuck at `Promoted 0`.

## What This Changes

The issue reports that memory-core dreaming promotion still produces repeated `Promoted 0` results on v2026.4.29 because recall/daily signals do not accumulate enough to pass promotion thresholds.

## Best Possible Solution

Make daily dreaming re-ingest eligible unchanged daily memory files once per dreaming day while preserving same-day dedupe, then cover the normal persisted-checkpoint path with a regression test.

## Reproduction Assessment

Yes. Create a daily memory note, run a light/dreaming ingestion sweep, keep `memory/.dreams/daily-ingestion.json`, then run the next day's sweep without editing the note; current main skips the unchanged file, so the daily signal cannot increase.

## Solution Assessment

No for the issue's normalization-reset theory; current normalization preserves counters. The narrow maintainable fix is daily-ingestion checkpoint semantics plus coverage that does not delete the checkpoint between sweeps.

## Repair Work Prompt

Fix the memory-core dreaming promotion regression reported in this issue. The source-level blocker is that `collectDailyIngestionBatches` skips unchanged `memory/YYYY-MM-DD.md` files solely by mtime/size in `memory/.dreams/daily-ingestion.json`, so later daily sweeps do not add daily signals and deep promotion can remain below the default signal gates. Update the daily ingestion checkpoint semantics so eligible unchanged daily files are re-ingested at most once per dreaming day, while preserving same-day dedupe and not lowering promotion thresholds or changing scoring policy. Add a regression test in `extensions/memory-core/src/dreaming-phases.test.ts` that runs two sweeps on consecutive dreaming days without deleting `memory/.dreams/daily-ingestion.json` and verifies `dailyCount`/candidate eligibility grows. Keep the fix inside memory-core; do not add core owner-specific migrations. Add a user-facing `CHANGELOG.md` fix entry because this affects shipped dreaming promotion behavior. Related context: #64068 and #67061.
