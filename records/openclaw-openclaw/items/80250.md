## Summary

Keep open: the PR is a focused fix with credible real CLI proof, but the new doctor target collection misses default-agent fallback routes whenever any explicit route binding exists, so part of the linked bug remains uncovered.

## What This Changes

This PR adds policy guidance to the agent system prompt, a doctor preview warning for channel-bound agents without the message tool, regression tests, and a changelog entry.

## Best Possible Solution

Keep the prompt guidance and warning direction, but make the doctor target collection mirror routing fallback so default agents are warned for enabled configured channels that can still reach them without a matching binding.

## Reproduction Assessment

Yes. The linked bug is source-reproducible from current main’s tool-policy filtering and doctor warning gap, and the PR-specific missed case is source-reproducible from routing’s default fallback plus the new `routeBindings.length === 0` guard.

## Solution Assessment

No. The proposed direction is maintainable, but this implementation is incomplete for partial route configurations; the safer fix is to include default-agent fallback targets for enabled configured channels even when other bindings exist.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Include default-route channels when bindings are partial:** `src/commands/doctor/shared/preview-warnings.ts:242-247`
  - body: This only adds the default agent's configured channels when there are zero route bindings. Routing still falls back to the default agent when no binding matches, so a config with Discord explicitly routed to another agent and Telegram configured without a binding will route Telegram to the default agent but emit no warning if that default agent lacks `message`. Please include fallback-default channel targets for mixed binding configs and cover that case in the new tests.
