## Summary

Keep open: the direction and supplied proof are good, but the current head still misses default-agent fallback channels when any explicit route binding exists, so `doctor` can remain silent for mixed route configurations.

## What This Changes

This PR adds agent prompt guidance, a doctor preview warning for channel-bound agents missing the `message` tool, regression tests, and a changelog entry.

## Best Possible Solution

Keep the prompt guidance and warning, but make doctor target collection mirror routing fallback by warning for enabled configured channels that can still reach the default agent, with regression coverage for mixed explicit/default routes.

## Reproduction Assessment

Yes. The linked bug and the PR blocker are source-reproducible from tool-policy filtering, current main’s missing route-bound warning, and routing’s default fallback path; I did not run a live Discord/Ollama flow in this read-only review.

## Solution Assessment

No. The proposed direction is maintainable, but this implementation is incomplete for mixed binding configurations; the safer fix is to include default-agent fallback channel targets when configured channels are not covered by explicit bindings.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Include default-route channels when bindings are partial:** `src/commands/doctor/shared/preview-warnings.ts:242-247`
  - body: This only adds the default agent's configured channels when there are zero route bindings. Routing still falls back to the default agent when no binding matches, so a config with Discord explicitly routed to another agent and Telegram configured without a binding will route Telegram to the default agent but emit no warning if that default agent lacks `message`. Please include fallback-default channel targets for mixed binding configs and cover that case in the new tests.
