## Summary

Keep open: the PR is aimed at the right bug and includes sufficient real CLI proof, but the latest head still treats a route with no `accountId` as covering every account. Current routing treats that route as default-account scoped, so doctor can still miss the default-agent warning for other accounts on the same channel.

## What This Changes

The PR adds prompt guidance for policy-filtered tools, doctor preview warnings for routed/default channel agents missing the `message` tool, regression tests, and a changelog entry.

## Best Possible Solution

Keep the prompt guidance and doctor warning, but mirror routing semantics: only `accountId: "*"` should fully cover all accounts, while default-account-only bindings must leave uncovered channel traffic eligible for the default-agent warning.

## Reproduction Assessment

Yes. Source inspection shows a route with no `accountId` is default-account scoped, and the PR’s doctor collection treats that same route as fully covering the channel.

## Solution Assessment

No. The direction is maintainable, but the implementation must distinguish wildcard account routes from default-account bindings before it is the narrowest correct fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Treat missing accountId as default-account scoped:** `src/commands/doctor/shared/preview-warnings.ts:221-223`
  - body: A route with `match.channel` and no `accountId` only matches the default account; non-default accounts fall through to the default agent. This helper marks that route as fully covering the channel, so a multi-account channel can suppress the default-agent `message` warning even though that default agent still receives uncovered traffic. Only `accountId: "*"`, or a proven single-account case, should suppress the fallback warning.
