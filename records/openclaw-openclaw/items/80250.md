## Summary

Keep this PR open: the doctor-only diagnostic is a good narrow direction and the supplied proof is enough, but the latest head still has a route-coverage false positive for channels whose effective account set comes from account-scoped bindings rather than `channels.<channel>.accounts`.

## What This Changes

The PR adds a `doctor` preview warning for channel-routed or default-routed agents whose resolved tool policy does not include the `message` tool, with regression tests and a changelog entry.

## Best Possible Solution

Keep the doctor-only warning, but make fallback coverage use the same effective account universe that runtime channel startup and account selection use before claiming uncovered traffic falls to the default agent.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: on the PR head, a Telegram config with no `channels.telegram.accounts` map and an account-scoped route for `accountId: "work"` can be runtime-covered while doctor still adds a default-agent warning because it only checks `DEFAULT_ACCOUNT_ID` when configured accounts are empty.

## Solution Assessment

No as-is. The doctor-only solution is the right narrow fix, but the coverage helper should include route-derived/default effective account ids before warning about default-agent fallback.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Honor route-derived account coverage:** `src/commands/doctor/shared/preview-warnings.ts:329-331`
  - body: When a channel has no `accounts` map, this branch only treats `DEFAULT_ACCOUNT_ID` as covered before adding the default-agent warning. Telegram can derive its effective account list from account-scoped route bindings, so a config with only `match.accountId: "work"` can route all runtime traffic to a messaging-capable agent while `doctor` still warns that `main` handles the channel. Include route-derived effective account ids, or add a regression for this case, before warning.
