## Summary

Keep open: the latest head fixed the prior scoped-binding gap, but it still treats a binding with no `accountId` as covering the whole channel even though routing treats it as default-account-only, so `doctor` can miss default-agent warnings for other configured accounts.

## What This Changes

The PR adds system prompt guidance, doctor preview warnings for routed/default channel agents missing the `message` tool, regression tests, and a changelog entry.

## Best Possible Solution

Mirror route account semantics in doctor target collection: only wildcard account bindings fully cover a channel, and default-agent warnings should remain for uncovered configured accounts.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a configured channel with another account, a route using only `match.channel` to a messaging-capable agent, and a default agent without `message`; routing sends the other account to default while this PR suppresses the warning.

## Solution Assessment

No. The direction is maintainable, but target collection must distinguish default-account bindings from wildcard account bindings before this is the narrowest correct fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Include default warnings for default-account bindings:** `src/commands/doctor/shared/preview-warnings.ts:221-223`
  - body: A route with `match.channel` and no `accountId` is not an any-account route: routing indexes it under the default account, and non-default accounts fall through to the default agent. Because this treats the missing account as unscoped, a multi-account channel can suppress the default-agent `message` warning even though traffic for another account still reaches that default agent. Treat only `accountId: "*"` or a proven single-account channel as fully covered, and add a regression for the multi-account fallback.
  - confidence: 0.9
