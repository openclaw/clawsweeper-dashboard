## What This Changes

## Best Possible Solution

Keep the memory-core fix, but gate or move the EPERM fallback so the old target is preserved before any copy overwrite, then add focused rollback tests and real Windows CLI proof.

No, not as submitted. The target helper is plausible, but the fallback must not short-circuit the rollback-safe publish path that protects the old memory index before overwrite.

Full review comments:

- **[P1] Keep EPERM fallback inside the rollback-safe swap path:** `extensions/memory-core/src/memory/manager-atomic-reindex.ts:76-80`
  - body: When `swapMemoryIndexFiles` calls `renameWithRetry(tempPath, targetPath)`, EPERM is supposed to throw into the catch below so the old live index is backed up before publishing. Returning here after `copyFile`/`unlink` can overwrite the live index without that backup/rollback protection, so a copy or unlink failure can leave memory state mixed or unrestorable. Scope this fallback to moves where the old target is already protected, or pass an explicit mode and add coverage for EPERM on the first publish.
