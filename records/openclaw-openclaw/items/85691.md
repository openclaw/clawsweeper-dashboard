## Summary

This is a useful fix for a real report, but the current branch is not merge-ready because the repair still persists a fully hydrated session store and bypasses the existing session-store writer/cache path. The supplied proof is also still a standalone mirror-script run rather than patched `openclaw doctor --fix` against copied affected data with prompt blobs.

## What This Changes

The PR adds `doctor --fix` repair logic for stale bundled skill paths in session snapshots and wires the session snapshot health check to receive the repair flag.

## Best Possible Solution

Keep the scanner layering, but apply repairs through the session-store writer with entry-scoped persistence, add focused regression coverage for promptRef-backed and unrelated sessions, and require redacted product-path proof before merge.

## Reproduction Assessment

Yes, from source: current main scans stale bundled skill snapshot paths and reports them, while `runSessionSnapshotsHealth` never passes a repair flag. The linked issue and PR body also provide live pre-fix output showing stale cached snapshot paths after a pnpm runtime-root change.

## Solution Assessment

No, not yet. Layering repair on the existing scanner is the right direction, but the persistence path should be writer-owned and entry-scoped before this is the safest maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P1] Keep repaired persistence entry-scoped:** `src/commands/doctor-session-snapshots.ts:480`
  - body: This projects and later writes the entire store after `loadSessionStoreForSnapshotScan()` has hydrated every promptRef. In current main, that hydration deletes `skillsSnapshot` when a prompt blob cannot be read, so a store with one repairable session and one unrelated missing/corrupt prompt blob would have the unrelated session's snapshot persisted as deleted during `doctor --fix`. Apply only the diagnosed entry changes through the session-store writer, or otherwise preserve raw untouched entries.
  - confidence: 0.86
- **[P1] Route repair writes through the session-store writer:** `src/commands/doctor-session-snapshots.ts:490`
  - body: Writing `sessions.json` directly bypasses the `updateSessionStore` path that serializes session mutations and refreshes the serialized/object caches after a save. If this health check runs in a process that has already touched the session store, later same-process reads or writes can continue from stale cached state and lose either the repair or other session changes. Use the existing writer/save path for this mutation.
  - confidence: 0.8
