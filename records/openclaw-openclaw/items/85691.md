## Summary

Keep this PR open, but it is not merge-ready: current main still only reports stale session snapshot paths, while the proposed repair still misses encoded/path cases and writes `sessions.json` outside the normal session-store writer. Adds `shouldRepair` handling to session snapshot doctor health, passes the doctor repair flag into it, rewrites stale snapshot paths in `sessions.json`, and adds a root-level standalone repro script.

## Best Possible Solution

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve XML escaping when rewriting prompt paths:** `src/commands/doctor-session-snapshots.ts:331-338`
  - body: `skillsSnapshot.prompt` locations are generated with XML escaping, and the scanner decodes those locations before returning `finding.cachedPath`. Replacing only `JSON.stringify(finding.cachedPath)` or the raw decoded string will not match a stored prompt path containing entities such as `&amp;`, so `doctor --fix` will keep reporting those sessions instead of repairing them.
  - confidence: 0.88
- **[P2] Handle Windows baseDir suffixes:** `src/commands/doctor-session-snapshots.ts:350-352`
  - body: On Windows, the baseDir scan path is formed with `\SKILL.md`, but this repair branch only runs when the synthetic finding ends with `/SKILL.md`. For a persisted Windows `resolvedSkills.baseDir`, the raw JSON contains only the directory path, so neither the full `cachedPath` replacement nor this baseDir branch updates it.
  - confidence: 0.84
- **[P2] Route repairs through the session-store writer:** `src/commands/doctor-session-snapshots.ts:372`
  - body: This writes `sessions.json` directly after reading raw text. Current session-store saves go through the exclusive writer, normalization, atomic write helper, and cache updates; bypassing that path risks stale in-process reads or lost session-store updates when doctor repair runs alongside other session activity.
  - confidence: 0.78
