## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Preserve unrelated promptRef snapshots during repair:** `src/commands/doctor-session-snapshots.ts:462`
  - body: `loadSessionStoreForSnapshotScan()` hydrates every entry, and current `hydrateSessionStoreSkillPromptRefs()` deletes `skillsSnapshot` when a referenced prompt blob cannot be read. Because this repair later serializes the whole hydrated store, fixing one stale path can persist that deletion for an unrelated session in the same `sessions.json`; merge only the repaired fields back into the raw store or otherwise preserve entries whose `promptRef` could not be hydrated.
  - confidence: 0.9
- **[P2] Keep repair writes on the normalized session-store contract:** `src/commands/doctor-session-snapshots.ts:489-490`
  - body: The new write path serializes `projectSessionStoreForPersistence()` directly, bypassing `normalizeSessionStore()`. Current session-store code intentionally strips `skillsSnapshot.resolvedSkills` on every canonical load/save, so a stale legacy store that this scanner can find may be rewritten while keeping that runtime-only cache on disk; normalize or use the canonical writer while preserving the intended repair fields.
  - confidence: 0.84
- **[P2] Move the mirror scenarios into production-path tests:** `repro-doctor-session-snapshot-repair.mjs:1`
  - body: The root repro script duplicates the scanner and repair logic instead of importing the production doctor path, so it can pass while the real `noteSessionSnapshotHealth({ shouldRepair: true })` path breaks. Convert those scenarios into focused Vitest coverage in the existing doctor snapshot test file and remove the one-off root script.
  - confidence: 0.78
