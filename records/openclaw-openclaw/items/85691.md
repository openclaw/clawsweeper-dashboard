## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve XML escaping when replacing prompt paths:** `src/commands/doctor-session-snapshots.ts:331-332`
  - body: `extractSkillLocations()` decodes XML entities before the finding reaches this repair loop, but the file still contains the original XML text inside JSON. For a cached prompt like `<location>...foo&amp;bar/SKILL.md</location>`, neither `jsonEscaped` nor `finding.cachedPath` matches the raw store, so `doctor --fix` can report the stale path and then leave it unchanged. Repair the parsed field or preserve/re-escape the original XML text before replacing.
  - confidence: 0.87
- **[P2] Handle Windows baseDir suffixes:** `src/commands/doctor-session-snapshots.ts:350`
  - body: This baseDir repair only runs when the synthesized cached path ends in `/SKILL.md`. On Windows, `collectResolvedSkillPaths()` builds that path with `path.join`, so the suffix is `\SKILL.md`; the `filePath` can be repaired while `resolvedSkills.baseDir` remains pointed at the stale root. Use `path.dirname`/platform-aware suffix handling or repair the parsed `baseDir` field directly.
  - confidence: 0.9
- **[P2] Use the session store writer for repairs:** `src/commands/doctor-session-snapshots.ts:372`
  - body: This writes the earlier raw `sessions.json` snapshot straight back with `fs.writeFileSync`, outside the session writer queue and atomic save path used by `updateSessionStore`. If a live gateway or another command updates the same store between the scan/read and this write, that newer session metadata can be silently lost; the repair should patch through the store writer or an explicitly offline atomic repair path.
  - confidence: 0.84
