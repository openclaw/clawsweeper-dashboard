## Summary

This PR should stay open. The requested doctor repair is valid and current main is still report-only, but the latest patch still persists a whole hydrated scanner store and bypasses the session-store writer/cache contract, so it can lose unrelated session metadata or race session saves; the supplied proof is also a standalone mirror-script run rather than patched `openclaw doctor --fix` proof.

## What This Changes

The PR wires `doctor --fix` into session snapshot health, adds stale bundled-skill snapshot-path replacement with prompt-blob persistence and backup writes, and adds a standalone reproduction script.

## Best Possible Solution

Land an entry-scoped session-store repair through `updateSessionStore` or an equivalent writer-owned path that preserves unrelated promptRef-backed sessions, updates caches, and is covered by regression tests plus redacted patched `doctor --fix` proof.

## Reproduction Assessment

Yes. Current main's snapshot health function has no repair mode and the doctor flow calls it without repair, while the linked issue includes live stale-path output showing the report-only behavior.

## Solution Assessment

No. Adding a `doctor --fix` repair is the right product direction, but this implementation should use the existing session-store writer contract and entry-scoped persistence before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Avoid persisting the hydrated scanner store:** `src/commands/doctor-session-snapshots.ts:480-482`
  - body: This serializes the whole store after `loadSessionStoreForSnapshotScan()` has hydrated it. That hydrator deletes `skillsSnapshot` when any unrelated session has a missing or invalid `promptRef` blob, so repairing one stale path can silently persist metadata loss for other sessions; keep the repair entry-scoped or preserve raw entries that were not diagnosed.
  - confidence: 0.9
- **[P1] Use the session-store writer for repair writes:** `src/commands/doctor-session-snapshots.ts:490`
  - body: The repair writes `sessions.json` directly after reading an earlier snapshot, bypassing `updateSessionStore` and its per-store writer queue/cache updates. A concurrent agent/channel save can be overwritten, and in-process readers may keep stale cached session state after doctor reports success.
  - confidence: 0.88
