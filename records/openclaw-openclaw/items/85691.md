## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Use the session-store writer for repair writes:** `src/commands/doctor-session-snapshots.ts:401-405`
  - body: `repairSessionSnapshotStore` rewrites `sessions.json` directly after loading and mutating a scanner copy. The normal writer serializes session updates through an update queue and refreshes the in-process cache; bypassing it means a concurrent save can overwrite the doctor's repair or the doctor can clobber a newer save, and a running process can keep serving stale cached session state after `doctor --fix`. Route this through `updateSessionStore`/the writer-owned path, or an equivalent entry-scoped API that preserves the same queue/cache guarantees.
  - confidence: 0.89
- **[P1] Avoid persisting the whole hydrated scanner store:** `src/commands/doctor-session-snapshots.ts:391-400`
  - body: The repair loads the scanner store with promptRef hydration and then projects/writes the entire `repairedStore`. If any unrelated session in the same store has a missing, invalid, or concurrently changed prompt blob, this repair can rewrite or drop that session's promptRef-backed metadata even though the stale finding was for another session. Persist only entries/fields that had replacements, and add a regression test where an unrelated promptRef-backed session remains byte-for-byte outside the repair set.
  - confidence: 0.84
- **[P1] Add product-path proof for copied affected prompt blobs:** `repro-doctor-session-snapshot-repair.mjs:1`
  - body: The PR body still proves a standalone mirror script and includes pre-fix live output, but it explicitly does not show patched `openclaw doctor --fix` on copied affected data with promptRef blobs. This change is a session-store repair path, so before merge it needs redacted terminal output, logs, or an artifact from the actual command showing prompt-blob repair, backup/write behavior, rescan leftovers, and unrelated session preservation.
  - confidence: 0.78
