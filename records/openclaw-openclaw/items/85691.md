## What This Changes

The PR wires `doctor --fix` repair intent into the session snapshot health check and adds logic plus a standalone reproduction script to rewrite stale bundled-skill paths in persisted session snapshot metadata.

## Best Possible Solution

Keep the layered scanner approach, but persist only explicitly repaired snapshot fields through the canonical session-store write/projection boundary, add colocated repair regression tests, and then provide redacted patched-command output from copied affected data.

## Solution Assessment

No. Adding repair support is the right direction, but serializing the scan-hydrated full store is not the safest implementation because it can persist unrelated promptRef cleanup and bypass normal session-store normalization.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Avoid persisting scan-hydrated session cleanup:** `src/commands/doctor-session-snapshots.ts:480-490`
  - body: `loadSessionStoreForSnapshotScan()` hydrates every promptRef and deletes `skillsSnapshot` when a blob/ref is missing. Because this block serializes the entire hydrated store after any replacement, `doctor --fix` can permanently strip unrelated session snapshots whenever the same store also has one repairable stale path. Keep scan hydration read-only, or persist only the explicitly repaired fields through the canonical writer.
  - confidence: 0.88
- **[P2] Preserve the resolvedSkills persistence contract:** `src/commands/doctor-session-snapshots.ts:489-490`
  - body: The normal session-store writer calls `normalizeSessionStore()` before persistence, and current tests require `resolvedSkills` to be stripped from disk. This direct `JSON.stringify(persisted)` path can rewrite legacy `resolvedSkills` entries that the scanner read and keep their full skill payloads in `sessions.json`; route the repair through normalization or an equivalent scoped serializer.
  - confidence: 0.78
