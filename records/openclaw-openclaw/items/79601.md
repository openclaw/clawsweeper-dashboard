## Summary

Keep open: current main still lacks the requested pre-connect client identity label for gateway WebSocket handshake timeouts. Existing `connect.client` identity is available only after a connect frame arrives, and the related phase-logging work does not settle the unverified client-hint/header decision.

## What This Changes

The issue requests gateway WebSocket logs include a client identity label, especially for handshake-timeout connections where no validated connect frame arrives.

## Best Possible Solution

Define a small sanitized unverified pre-connect client hint for Gateway WS upgrade handling, then carry validated `connect.client` metadata after connect with focused log tests and docs.

## Reproduction Assessment

Yes, at source level: opening an idle WebSocket that never sends `connect` reaches the timeout path that logs only `conn`, `peer`, and `remote`. I did not run a live Windows gateway log capture in this read-only review.

## Solution Assessment

Unclear until maintainers choose the unverified identity contract. Reusing validated `connect.client` after connect is narrow, but the critical timeout case needs an explicit sanitized header or User-Agent convention.
