## Summary

Keep open. Current main and the latest release still lack a canonical client identity hint on the primary Gateway WebSocket preauth timeout log, and the remaining fix needs a maintainer decision on the unverified pre-auth client-hint contract.

## What This Changes

The issue asks for gateway WebSocket logs to include a client identity label, especially for pre-connect handshake timeouts where only conn and peer are currently visible.

## Best Possible Solution

Define a gateway-owned, sanitized, explicitly unverified client hint for upgrade-time failure logs and keep it separate from validated connect.client identity.

## Reproduction Assessment

Yes, as a source-level check: current main’s timeout path logs only conn, peer, and remote, while validated connect.client metadata is unavailable until after the first connect frame.

## Solution Assessment

Unclear. A sanitized unverified hint is plausible and narrow, but maintainers need to settle the trust model and browser-compatible carrier before adding a new pre-auth logging contract.
