## Summary

Keep open: current main still lacks a pre-auth client label on handshake-timeout logs, and the reporter clarified that this timeout path is the core diagnostic gap.

## What This Changes

Requested gateway WebSocket timeout and close diagnostics should identify the initiating client or subsystem instead of only opaque connection IDs and socket peers.

## Best Possible Solution

Add an explicitly unverified, sanitized pre-auth client hint supported by the built-in clients that can be logged on timeout/close, with validated post-connect identity reused separately if desired.

## Reproduction Assessment

Yes, at source level: open a gateway WebSocket, receive the challenge, and never send connect; the timeout log path emits conn/peer/remote without any client label. A live gateway log run was not performed because this was a read-only review.

## Solution Assessment

Unclear until maintainers choose the pre-auth identity transport. Adding a client-id field to ConnectParams alone cannot solve the timeout case, and a custom header alone does not cover browser clients.
