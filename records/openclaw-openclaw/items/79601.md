## Summary

Keep open: current main still cannot attribute the critical pre-connect Gateway WebSocket timeout to a client label; solving that requires a small logging/protocol decision for unverified upgrade-request hints.

## What This Changes

Request adding sanitized client identity labels to Gateway WebSocket lifecycle logs, especially pre-connect handshake timeouts.

## Best Possible Solution

Define a sanitized unverified upgrade-request hint for timeout diagnostics, have built-in clients send the best available label before connect where possible, and add focused Gateway log coverage for timeout and close-before-connect paths.

## Reproduction Assessment

Yes, at source level: the current timeout path logs conn/peer/remote without any client/header label, and built-in client identity is sent only later in connect params. I did not run a live gateway log session because this review is read-only.

## Solution Assessment

Unclear until maintainers choose the pre-connect identity contract. Reusing validated connect metadata after connect is narrow, but the critical timeout case should use an explicitly unverified sanitized header/User-Agent hint rather than pretending ConnectParams exists before the first frame.
