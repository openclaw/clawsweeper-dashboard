## Summary

Keep open: current main still lacks a distinct client/subsystem label for pre-connect WebSocket timeouts, and the reporter clarified that this pre-connect timeout attribution is the critical gap. The already-validated `connect.client` metadata exists only after a connect frame arrives, while an earlier hint would be unauthenticated and needs a small protocol/logging decision before implementation.

## What This Changes

The issue asks the gateway to tag WebSocket timeout and close logs with client or subsystem identity, especially before a connect frame is validated.

## Best Possible Solution

Define a documented, sanitized pre-connect client hint that works for Node/native and browser clients, log it as unverified on timeout/close paths, and reuse validated `connect.client` metadata after authentication.

## Reproduction Assessment

Yes, at source level: current main’s handshake-timeout warning formats only `conn`, `peer`, and `remote`, and validated `connect.client` metadata is unavailable unless a connect frame arrives. I did not run a live gateway log session in this read-only review.

## Solution Assessment

Unclear until maintainers choose the pre-connect identity model. Reusing validated `connect.client` post-connect is straightforward, but timeout attribution needs an explicitly untrusted hint contract that also works for browser WebSocket clients.

- A custom `X-OpenClaw-Client`-style header would not work for browser WebSocket clients, so the pre-connect hint contract needs a browser-compatible path or a scoped fallback.

- The adjacent phase-logging issue may improve timeout diagnosis, but it does not identify which subsystem opened the timed-out socket.
