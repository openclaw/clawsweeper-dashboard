## Summary

Keep open: current main still logs pre-connect Gateway WebSocket timeout and close paths without a client identity label, and I found no duplicate issue or open PR that already tracks this exact observability gap.

## What This Changes

The issue requests an additive client identity label for Gateway WebSocket connection logs so timeout and early-close records identify the initiating subsystem/client.

## Best Possible Solution

Add a sanitized, non-secret claimed client label to Gateway WebSocket upgrade/preauth diagnostics and propagate it from owned clients, while keeping authenticated identity based on the existing connect frame.

## Reproduction Assessment

Yes from source inspection: the current timeout and closed-before-connect log strings omit any client label, and owned clients only send client.id in the later connect frame. I did not run a live gateway log capture during this read-only review.

## Solution Assessment

Yes, an additive diagnostic label is the narrowest maintainable direction if it is sanitized and treated as spoofable. The safer implementation is to carry a claimed preauth label only for logging, then continue using connectParams.client.id for authenticated client identity after the connect frame validates.

Implement https://github.com/openclaw/openclaw/issues/79601 as a focused Gateway WebSocket diagnostics change. Current main logs preauth `handshake timeout` and `closed before connect` records with only conn/peer/remote/header context; add a sanitized, non-secret claimed client identity label that owned clients can provide before the first connect frame.
