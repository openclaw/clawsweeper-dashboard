## Summary

Keep open: current main already has a `connect.client.id` protocol field and logs it for successful connect/auth-failure paths, but the reported lifecycle gaps still exist for timeout, close, and post-connect RPC log lines.

## What This Changes

The issue requests client identity labels on gateway WebSocket handshakes and gateway/ws connection lifecycle logs.

## Best Possible Solution

Define a small gateway logging contract that reuses validated `connect.client` metadata after connect and, if needed, adds an explicitly unauthenticated sanitized pre-connect hint for timeout diagnostics.

## Reproduction Assessment

Yes, at source level: current main’s pre-auth timeout and close-before-connect log paths omit client identity, and post-connect request/response logs do not carry the already validated client metadata. A live gateway log run was not performed because this review is read-only.

## Solution Assessment

Unclear until maintainers decide the pre-connect identity model. Reusing validated `connect.client` metadata for post-connect logs is narrow, but timeout attribution needs an explicit decision on whether to log an unauthenticated client hint.
