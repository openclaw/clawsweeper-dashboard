## Summary

Keep open: current main still lacks a client/subsystem label on the critical pre-connect WebSocket timeout path, and the reporter clarified that this timeout attribution is the actual diagnostic gap.

## What This Changes

Request gateway WebSocket lifecycle logs to include a client identity label, especially on pre-connect handshake timeouts, so startup race diagnostics can identify the initiating client or subsystem.

## Best Possible Solution

Define a small gateway pre-auth diagnostic label contract that built-in clients can send through a browser-compatible channel, sanitize it, log it as unverified on handshake-timeout and closed-before-connect lines, and optionally propagate validated connect.client metadata on later post-connect WS logs.

## Reproduction Assessment

Yes, at source level: opening an idle WebSocket and never sending the connect frame reaches the current handshake-timeout path, whose log line contains only conn, peer, and remote. I did not run a live gateway log session because this review is read-only.

## Solution Assessment

Unclear as requested: reusing validated connect.client after connect is straightforward, but the critical timeout case needs a new unauthenticated pre-connect hint and a maintainer decision about the client transport and log wording.
