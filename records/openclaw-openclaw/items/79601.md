## Summary

Keep open: current main still cannot attribute a pre-connect Gateway WebSocket timeout to a client or subsystem. The related phase-logging work is useful context, but it answers where the handshake stalled, not which client initiated the socket.

## What This Changes

Request to add sanitized client identity labels to Gateway WebSocket connect, timeout, and close logs, especially before a connect frame is received.

## Best Possible Solution

Define a small sanitized, explicitly unverified pre-connect client-hint contract that works for relevant clients and include it in handshake-timeout and closed-before-connect Gateway WS logs without treating it as authentication.

## Reproduction Assessment

Yes, at source level: an idle pre-connect WebSocket reaches the timeout path that logs conn/peer/remote only, and close-before-connect lacks a normalized client label. A live repro would start the gateway, open a socket without sending connect, and inspect gateway/ws logs.

## Solution Assessment

Yes for the clarified diagnostic direction if it is scoped as an unverified hint; validated connect.client cannot cover the critical timeout because that frame never arrives. The exact hint transport and wording still need maintainer judgment.
