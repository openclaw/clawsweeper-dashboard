## Summary

Keep open. Current main still has a pre-connect attribution gap: validated `connect.client` metadata exists only after the first frame, while the handshake-timeout warning still logs only connection and peer data. The broader client-type discussion at https://github.com/openclaw/openclaw/issues/39597 is related, but it does not replace this diagnostics-only timeout logging request.

## What This Changes

Request to add client identity or client-hint labels to Gateway WebSocket timeout and close logs, especially before the connect frame is received.

## Best Possible Solution

Define a gateway-owned diagnostics-only unverified client hint for upgrade-time failures, sanitize it, log it on timeout/pre-connect close paths, and keep it separate from validated `connect.client` identity.

## Reproduction Assessment

Yes, as a source-level check: current main’s timeout path logs only `conn`, `peer`, and `remote`, and validated `connect.client` metadata is not available until after the first connect frame.

## Solution Assessment

Unclear. A sanitized unverified hint is plausible, but maintainers should first settle the trust model and browser-compatible carrier before adding a new pre-auth logging contract.
