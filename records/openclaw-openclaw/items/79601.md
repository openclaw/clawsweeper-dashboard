## Summary

Keep open: current main still lacks the critical pre-connect client attribution for gateway WebSocket handshake timeouts. The useful fix is a diagnostic logging/protocol enhancement rather than an already-shipped implementation.

## What This Changes

Request gateway WebSocket logs to carry client identity labels, especially for pre-connect handshake timeouts, using a client-id field or best-effort upgrade-header hint.

## Best Possible Solution

Define a small gateway WS logging contract that logs a sanitized unverified client hint from upgrade headers for handshake-timeout and closed-before-connect paths, then reuse validated ConnectParams client metadata for connected lifecycle and RPC logs.

## Reproduction Assessment

Yes, at source level: open a gateway WebSocket and do not send the connect frame until the pre-auth timeout. Current main’s timeout log formats only conn, peer, and remote, so the reporter’s diagnostic gap remains source-reproducible.

## Solution Assessment

Unclear until maintainers choose the pre-connect identity model. A narrow additive `X-Openclaw-Client` or similar sanitized unverified header, with a user-agent fallback, looks maintainable if it is documented as diagnostic rather than trusted identity.
