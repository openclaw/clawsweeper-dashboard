## Summary

Keep open: current main still cannot attribute a pre-connect Gateway WebSocket timeout to a client or subsystem. The adjacent phase-logging issue and open PR cover where the handshake stalled, but they do not solve the clarified client-identity pain point.

## What This Changes

Request Gateway WebSocket timeout and close diagnostics to include a client or subsystem identity label, especially before a connect frame is received.

## Best Possible Solution

Define a small unverified pre-connect client-hint contract and carry a sanitized hint into handshake-timeout and closed-before-connect logs without treating it as authentication.

## Reproduction Assessment

Yes, at source level: an idle pre-connect WebSocket reaches the timeout path that logs conn/peer/remote only. A live repro would start the gateway, open a socket without sending connect, and inspect gateway/ws logs.

## Solution Assessment

Yes for the clarified direction if it is scoped as unverified diagnostics; using validated connect.client is not sufficient because the critical timeout happens before that frame exists.
