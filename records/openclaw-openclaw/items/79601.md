## Summary

Keep open: current main still omits a client identity label on the pre-connect handshake-timeout path, and the reporter clarified that this timeout attribution gap is the central diagnostic problem. The remaining work needs a maintainer decision because the useful fix likely logs an unauthenticated, sanitized upgrade-request hint before any validated connect frame exists.

## What This Changes

The issue requests recognizable client identity labels in gateway WebSocket logs so timeout and close diagnostics can identify the initiating subsystem.

## Best Possible Solution

Define a small gateway logging contract that adds a sanitized unverified client hint for upgrade/pre-connect lifecycle logs and, if desired, reuses validated connect.client metadata on later WS request/response lines.

## Reproduction Assessment

Yes, at source level: current main’s handshake-timeout warning logs only conn, peer, and remote before any connect frame is processed. The existing preauth timeout test covers closure behavior, but not client-label logging, and no live gateway log run was performed in this read-only review.

## Solution Assessment

Unclear until maintainers choose the pre-connect identity model. Reusing validated connect.client metadata after connect is narrow, but the critical timeout path needs an explicit decision on an unauthenticated header or User-Agent-derived hint.
