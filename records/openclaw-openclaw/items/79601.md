## Summary

Keep open. Current main and v2026.6.8 still log the primary handshake-timeout warning with only connection and peer data; the later pre-connect close log includes coarse headers, but not a canonical unverified client hint, so the reporter’s diagnostics gap remains.

## What This Changes

Issue requests gateway WebSocket diagnostics to attach client identity labels, especially to pre-connect handshake-timeout logs.

## Best Possible Solution

Define a gateway-owned diagnostics-only unverified client hint for upgrade-time failures, sanitize it, log it on timeout/pre-connect close paths, and keep it separate from validated connect.client identity.

## Reproduction Assessment

Yes, as a source-level check: current main’s timeout path logs only conn, peer, and remote, while validated connect.client metadata is unavailable until after the first connect frame.

## Solution Assessment

Unclear. A sanitized unverified hint is plausible, but maintainers should first settle the trust model and browser-compatible carrier before adding a new pre-auth logging contract.
