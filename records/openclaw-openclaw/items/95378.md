## Label Justifications

- `P1`: The issue reports a user-facing shared-session context bug that can cause a stale Telegram proposal to drive a state-mutating tool action.
- `impact:session-state`: The model-visible context can drift from the shared session transcript when the newest turn arrived through a different ingress.
- `impact:message-loss`: The cross-ingress turn is effectively omitted from the Telegram-selected prompt context before the agent interprets the reply.

## Summary

Keep open. Current main still builds Telegram's model-visible recent context from a Telegram account/chat cache while Gateway/iOS chat.send turns persist through the shared session transcript path, so the reported cross-ingress stale-context bug is not implemented or superseded.

## What This Changes

The issue reports that Telegram replies in a shared session use Telegram-only recent context and can bind an ambiguous reply to a stale proposal instead of the newest turn from another ingress.

## Best Possible Solution

Add a bounded session-scoped recent-context source for shared direct sessions from the canonical transcript/state, while preserving Telegram reply-target, group, topic, and privacy boundaries.

## Reproduction Assessment

Yes, at source level. Current main still assembles Telegram chat_window context from a Telegram account/chat cache while Gateway/iOS turns are persisted through the shared session transcript path; a live stale-tool replay was not run.

## Solution Assessment

Yes in direction. The maintainable fix is session-scoped bounded recency from canonical transcript/state; adding a config knob or merging raw per-ingress caches would be less safe.
