## Label Justifications

- `P1`: The issue reports a user-facing shared-session context bug that can bind an ambiguous Telegram reply to stale history and execute the wrong state-mutating action.
- `impact:session-state`: The model-visible context can diverge from the shared session transcript when the newest turn arrived through another ingress.
- `impact:message-loss`: The cross-ingress turn is effectively omitted from the Telegram-selected prompt context before the agent interprets the reply.

## Summary

Keep open. Current main and the latest release still build Telegram prompt recency from a Telegram-only message cache while Gateway/iOS turns persist through the shared session transcript path; https://github.com/openclaw/openclaw/pull/95390 is an open candidate fix, so this issue should remain open until a fix merges or is replaced.

## What This Changes

The issue requests session-scoped recent conversation context so a Telegram reply in a shared session can see fresher turns from another ingress instead of binding to stale Telegram-only history.

## Best Possible Solution

Review and land a bounded fix like https://github.com/openclaw/openclaw/pull/95390 that reads direct-session Telegram recency from the canonical session transcript while preserving group/topic/reply boundaries, then close this issue after merge.

## Reproduction Assessment

Yes, at source level. Current main still assembles Telegram chat_window context from the Telegram account/chat cache while Gateway/iOS chat.send turns persist through the shared session transcript path; I did not run a live stale-tool replay.

## Solution Assessment

Yes in direction, but maintainer review is still needed. The maintainable fix is bounded session-scoped recency from the canonical transcript for direct sessions; raw cache merging or a new config knob would be less safe unless maintainers explicitly choose that policy.
