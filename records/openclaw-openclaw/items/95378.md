## Summary

Keep open. Current main and the latest release still assemble Telegram direct prompt recency from the Telegram account/chat message cache, while the shared Gateway/iOS session transcript path exists outside that cache; the linked closing pull request is still open and should be reviewed rather than closing this issue early.

## What This Changes

The issue reports that a shared-session Telegram reply can omit fresher Gateway/iOS turns from model-visible context and bind an ambiguous reply to stale Telegram-only history.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/95390 or a narrow equivalent that reads direct-session Telegram recency from the canonical session transcript while preserving Telegram privacy boundaries.

## Reproduction Assessment

## Solution Assessment

Yes in direction, but not yet complete. The narrow maintainable fix is direct-session recency from the canonical transcript rather than raw cache merging or a new config knob, with maintainer approval for the SDK/session boundary.
