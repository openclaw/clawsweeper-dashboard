## Label Justifications

- `P1`: The issue reports a high-impact cross-channel context bug that can bind an ambiguous reply to stale state and trigger the wrong action.
- `impact:session-state`: The model-visible context can diverge from the shared session transcript when the latest turn came from another ingress.
- `impact:message-loss`: The cross-ingress turn is effectively omitted from Telegram's selected prompt context before reply interpretation.

## Summary

Keep open. Current main and v2026.6.8 still build Telegram's selected prompt context from the Telegram account/chat cache, while Gateway `chat.send` turns persist through the shared session transcript path; https://github.com/openclaw/openclaw/pull/95390 is the open candidate fix but has not merged.

## What This Changes

The issue reports that a reply sent through Telegram in a shared session can miss newer turns from another ingress and bind the reply to stale Telegram-only context.

## Best Possible Solution

Resolve the open candidate PR or replace it with an owner-approved fix that uses bounded session-scoped recency for shared direct sessions while preserving Telegram group/topic and privacy boundaries.

## Reproduction Assessment

Yes, at source level. Current main still assembles Telegram recent prompt context from a Telegram account/chat cache while Gateway `chat.send` turns persist through the shared session transcript path; I did not run a live iOS/Gateway-to-Telegram replay.

## Solution Assessment

Unclear for the open PR. The right direction is bounded session-scoped recency for direct shared sessions, but maintainers still need to settle topic/privacy isolation and SDK contract shape before a fix should land.
