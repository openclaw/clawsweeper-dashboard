## Summary

Keep open: current main and v2026.6.8 still build Telegram's fresh Conversation context from a Telegram-scoped cache and inject it as current-turn prompt context, so a shared-session turn from another ingress can be absent from the recency block. Broader shared-session trackers partially overlap but do not replace this concrete cross-ingress stale-context bug.

## What This Changes

The issue requests interpreting replies in a shared agent session against shared-session recent context across iOS/gateway and Telegram, instead of Telegram-only recent cache context.

## Best Possible Solution

Design a session-scoped recent-context source for the active shared session, preserving Telegram reply-target/topic/mention filters and prompt-cache behavior, then add focused cross-ingress regression coverage.

## Reproduction Assessment

Yes, at source level. Current main still builds Telegram's current-turn Conversation context from a Telegram-scoped cache and injects it into the run prompt, so a fresh shared-session turn from iOS/gateway can be absent from that recency block; no live Telegram+iOS repro was run.

## Solution Assessment

Unclear as an implementation path. A session-scoped recent-context source is the right direction, but maintainers need to choose the trust-boundary and filtering semantics before a narrow autonomous patch is safe.
