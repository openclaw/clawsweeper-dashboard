## Summary

This PR should stay open: the reply-to `/approve` direction is useful, but the current patch can truncate valid approval IDs and needs the required changelog entry before merge.

## What This Changes

The PR adds reply-to-message ID extraction for single-token `/approve <decision>` commands, updates approval hint text, and adds approval parser/handler tests.

## Best Possible Solution

Keep the reply-to shortcut, but capture and trim the full `ID:` line, cover whitespace-containing IDs, and add a single-line Unreleased changelog entry.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: create an approval with an ID like `task 1`, reply to `ID: task 1` with `/approve allow-once`, and the PR extractor submits only `task`.

## Solution Assessment

No. The reply-to shortcut is a good direction, but the implementation should preserve the full approval ID line and satisfy the changelog policy before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Capture the full replied approval ID line:** `src/auto-reply/reply/commands-approve.ts:41`
  - body: The new extractor uses `\S+`, so replying to a request with a valid ID containing whitespace, such as `ID: task 1`, submits only `task`. Gateway approval IDs are any non-empty trimmed string, and lookup falls back to prefix matching, so this can fail as ambiguous or approve a different pending approval; capture the whole `ID:` line and trim it instead.
  - confidence: 0.92
- **[P3] Add the required changelog entry:** `CHANGELOG.md:11`
  - body: This is a user-facing `feat` for chat approvals, but the PR diff does not update `CHANGELOG.md`. Repo policy requires user-facing feature changes to include an Unreleased changelog entry before merge.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The reply-to extractor can truncate a valid approval ID and route the approval decision to a different pending request.

Concerns:

- **[medium] Truncated reply ID can target another approval:** `src/auto-reply/reply/commands-approve.ts:41`
  - body: Because the regex captures only the first non-whitespace token from `ID:`, a reply to `ID: task 1` submits `task`; with exact or unique prefix matching, the gateway can resolve a different pending approval than the message being replied to.
  - confidence: 0.9
