## Summary

Keep this PR open: current main already has ID prefix matching, but reply-to approval extraction is still missing and the PR addresses a real chat ergonomics gap. The proposed extractor needs a blocking fix because it trusts the first `ID:` line in replied text, which can be injected ahead of plugin approval IDs.

## What This Changes

The PR adds reply-to-message ID extraction for `/approve <decision>`, updates approval prompt hints, and adds parser/extractor and forwarding tests while relying on current main's existing prefix lookup.

## Best Possible Solution

Land the reply-to shortcut after making replied-message ID extraction unambiguous for OpenClaw-rendered approval messages, covering plugin messages with multiline fields and full-line approval IDs.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: a plugin approval title/description containing a newline `ID: <other-id>` appears before the generated ID line, and the PR extractor takes that first match from the replied body.

## Solution Assessment

No. The reply-to direction is useful, but the extractor should identify the generated approval ID line safely and capture the full trimmed line; the PR also needs the required changelog entry.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Parse only the trusted full approval ID line:** `src/auto-reply/reply/commands-approve.ts:40`
  - body: The new reply extractor scans the whole replied body and captures the first non-space token after any line starting `ID:`. Plugin approval messages render plugin-controlled title/description/tool fields before the real generated ID, and those schema fields can contain newlines, so an injected `ID: <other approval>` line can make `/approve allow-once` resolve the wrong request; it also truncates valid custom IDs containing spaces. Extract the generated ID line unambiguously and trim the full line.
  - confidence: 0.9
- **[P3] Add the required changelog entry:** `src/auto-reply/reply/commands-approve.ts:159`
  - body: This is a user-facing `feat` that changes chat approval behavior and approval prompt copy, but the PR does not include `CHANGELOG.md`. OpenClaw policy requires user-facing feature changes to add a changelog entry before merge.
  - confidence: 0.86
