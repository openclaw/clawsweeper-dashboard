## Summary

Keep open: current main already has approval ID prefix lookup, but it does not implement the reply-to `/approve <decision>` shortcut. The PR still has blocking correctness issues in reply ID extraction and plugin approval prompt text, and the required real Telegram proof is missing.

## What This Changes

The PR adds single-token `/approve <decision>` reply-body ID extraction plus approval prompt hints and tests for chat approval shortcuts.

## Best Possible Solution

Keep the reply-to shortcut, parse the full generated ID line, render plugin prompt decisions from each request's allowed list, and attach redacted live Telegram proof before merge.

## Reproduction Assessment

Yes, for the review findings by source inspection: the PR regex truncates whitespace-containing IDs, and the plugin prompt hard-codes decisions that current main validates per request. I did not run tests because this is a read-only review.

## Solution Assessment

No. The feature direction is narrow and useful, but the patch should preserve full generated IDs, avoid misleading plugin approval text, and provide real Telegram proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Parse the full replied approval ID line:** `src/auto-reply/reply/commands-approve.ts:40`
  - body: `APPROVAL_ID_RE` captures only `\S+`, so a replied `ID: task 1` line submits `task` even though approval IDs are generic non-empty strings and the decision-first parser already preserves spaces. Capture and trim the full generated ID line before resolving it.
  - confidence: 0.93
- **[P2] Keep plugin approval tips request-scoped:** `src/infra/plugin-approvals.ts:70-72`
  - body: The added plugin approval tip always advertises `allow-always`, but plugin approvals can restrict `allowedDecisions` and the resolver rejects unavailable decisions. Build this reply-to tip from the request's allowed decision list so unavailable actions are not shown.
  - confidence: 0.95
