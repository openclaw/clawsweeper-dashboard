## Summary

Keep open: current main already has prefix approval lookup, but it does not implement reply-to `/approve`; the PR’s remaining reply extractor can select spoofed or truncated IDs and needs revision before merge.

## What This Changes

The PR adds reply-only `/approve <decision>` handling from `ReplyToBody`, exports parser helpers/tests, and updates exec/plugin approval prompt tips while relying on main’s existing prefix lookup.

## Best Possible Solution

Update the branch to extract the canonical approval-message ID as a full trimmed line, cover embedded `ID:` and whitespace-ID cases, and add a single-line Unreleased changelog entry.

## Reproduction Assessment

Yes. A source-level reproduction is a plugin approval whose title or description contains a newline followed by `ID: plugin:other` before the canonical ID, then replying with `/approve allow-once`; the PR regex captures the earlier line. A custom exec approval ID containing whitespace also reproduces truncation.

## Solution Assessment

No. Reply-to-message approval is the right UX direction, but first-`ID:` plus `\S+` extraction is not the safest implementation for an approval path; parsing the known canonical full ID line is narrower and safer.
