## Summary

Keep open: the reply-to approval feature is still useful, but this PR is not mergeable as-is because reply-body ID extraction truncates valid IDs, plugin approval hint text conflicts with request-scoped decisions, and external Telegram proof is missing.

## What This Changes

The PR adds reply-to `/approve <decision>` parsing from `ReplyToBody`, updates exec/plugin approval hint text, and adds approval parser/handler/forwarder tests.

## Best Possible Solution

Keep the reply-to shortcut, but parse and trim the full `ID:` line, keep approval prompt text based on each request's allowed decisions, rebase on main, and attach redacted Telegram proof before merge.

## Reproduction Assessment

Yes for the review findings by source inspection: the PR captures only `\S+` from replied `ID:` lines and hard-codes plugin decisions while current main accepts generic IDs and has request-scoped plugin approval tests. I did not run tests because this review is read-only.

## Solution Assessment

No. The reply-to approval direction is narrow and useful, but the current implementation should capture the full replied ID line, preserve request-scoped plugin decisions in the tip, and provide real Telegram proof before merge.
