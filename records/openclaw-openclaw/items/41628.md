## Summary

Keep open: the rebased PR is still useful, and current main already covers the prefix-matching part, but the new reply-to ID extractor can truncate valid approval IDs and misroute an approval before merge.

## What This Changes

The PR adds reply-to `/approve <decision>` handling from `ReplyToBody`, updates exec/plugin approval prompt hints, and adds approval parser/forwarder tests.

## Best Possible Solution

Keep the current-main gateway prefix lookup, then land the reply-to UX after parsing the full `ID:` line, adding regression coverage, and adding the required changelog entry.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: create an exec approval with an explicit ID containing internal whitespace, reply to the forwarded prompt with `/approve allow-once`, and the PR extractor submits only the first token even though the schema accepts the full ID.

## Solution Assessment

No. The feature direction is good, but the PR should capture and trim the entire `ID:` line and preserve current gateway prefix matching rather than parsing only `\S+`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Capture the full replied approval ID line:** `src/auto-reply/reply/commands-approve.ts:40`
  - body: The new reply-to path extracts `ID:` with `\S+`, but gateway approval request IDs are generic `NonEmptyString` values and the forwarder prints the full ID line. For a valid ID like `task 1`, replying with `/approve allow-once` submits only `task`, which can fail or resolve a different pending prefix. Capture the whole `ID:` line and trim it.
