## Summary

Keep open. Current main already has the prefix lookup part, but it still lacks reply-only `/approve <decision>` parsing, and the current PR head has a concrete approval-routing bug in reply-body ID extraction that should be fixed before merge.

## What This Changes

The PR adds reply-only `/approve <decision>` handling by extracting approval IDs from `ReplyToBody`, exports parser/extractor helpers for tests, and updates exec/plugin approval prompts to mention reply and prefix shortcuts.

## Best Possible Solution

Keep the PR open and ask for a branch update that parses the canonical rendered approval ID as a full trimmed line from the known approval-message structure, ignores freeform title/description `ID:` lines, and adds regression coverage for embedded `ID:` lines plus custom exec IDs containing whitespace. Prefix handling should continue to rely on main’s existing `lookupPendingId` behavior.

## Reproduction Assessment

Yes. A source-level reproduction is a plugin approval whose title or description contains a newline followed by `ID: plugin:other` before the canonical ID line, then replying with `/approve allow-once`; the PR regex captures the earlier line.

## Solution Assessment

No. The UX direction is useful, but the PR’s first-`ID:` and first-token extraction is not the safest implementation for an approval path; parsing the canonical full ID line from the rendered approval message is the narrower maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Capture the canonical full ID line:** `src/auto-reply/reply/commands-approve.ts:40`
  - body: `APPROVAL_ID_RE` matches the first `ID:` line and captures only `\S+`. Parse the canonical rendered ID as a full trimmed line and cover embedded `ID:` lines plus whitespace-containing custom IDs.
