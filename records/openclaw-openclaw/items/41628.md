## Summary

Keep open; the PR is still useful because current main has prefix lookup but not reply-to approval extraction, yet the diff truncates valid whitespace-bearing approval IDs and is missing the required user-facing changelog entry.

## What This Changes

The PR adds reply-to-message approval ID extraction for single-token `/approve <decision>` commands, updates exec/plugin approval hint text, and adds parser, handler, and message tests.

## Best Possible Solution

Land the narrowed reply-to approval shortcut after preserving full ID lines, keeping main’s existing prefix lookup, and adding focused regression tests plus changelog coverage.

## Reproduction Assessment

Yes. Source inspection shows current main rejects the single-token reply form, and the PR diff’s `APPROVAL_ID_RE` source-proves the whitespace-ID truncation path.

## Solution Assessment

No. The direction is maintainable, but the implementation should capture and trim the complete `ID:` line and add the required changelog entry before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Capture the full replied ID line:** `src/auto-reply/reply/commands-approve.ts:40`
  - body: The reply extractor captures only `\S+`, but exec approval IDs are only constrained to `NonEmptyString` and the manager preserves internal spaces. Replying to a prompt for `ID: task 1` submits `task`; if a pending `task` exists it can approve the wrong request, and otherwise it depends on prefix matching instead of the ID printed in the prompt. Capture the whole line and trim it.
  - confidence: 0.87
- **[P3] Add the required changelog entry:** `src/auto-reply/reply/commands-approve.ts:37`
  - body: This is a user-facing `feat` change to chat approval behavior and prompt copy, but the diff does not update `CHANGELOG.md`. Add a single-line Unreleased entry so the new `/approve` reply shortcut ships with release notes.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The diff adds no dependency or CI changes, but the new approval shortcut can misroute approvals when it truncates valid whitespace-bearing IDs.

Concerns:

- **[medium] Approval reply extraction can misroute custom IDs:** `src/auto-reply/reply/commands-approve.ts:40`
  - body: Because the new extractor keeps only the first non-space token from the replied `ID:` line, an authorized user replying to one approval can submit a different exact ID or ambiguous prefix when custom exec IDs contain spaces. Approval resolution is security-sensitive, so the full line should be preserved.
  - confidence: 0.86
