## Summary

Keep open: the reply-to `/approve` shortcut is useful and is not on current main, but this PR is not merge-ready because it still has two concrete P2 correctness issues and lacks after-fix real Telegram proof.

## What This Changes

The PR adds single-token `/approve <decision>` parsing from replied message bodies, updates exec/plugin approval prompt hints, and adds approval parser and forwarder tests.

## Best Possible Solution

Rebase the PR, keep the reply-to shortcut, parse the exact generated ID line, render plugin tips from request-scoped allowed decisions, and attach redacted live Telegram proof.

## Reproduction Assessment

Yes for the review findings by source inspection: the PR truncates whitespace-containing replied IDs and hard-codes unavailable plugin decisions, while current main accepts generic IDs and validates request-scoped decisions. I did not run tests because this review is read-only.

## Solution Assessment

No. The feature direction is narrow and useful, but the current patch should preserve full generated IDs, avoid misleading plugin approval text, and provide real Telegram proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Parse the full replied approval ID line:** `src/auto-reply/reply/commands-approve.ts:40`
  - body: `APPROVAL_ID_RE` captures only `\S+`, so a replied `ID: task 1` line submits `task` even though approval IDs are generic non-empty strings and explicit decision-first parsing preserves spaces. Capture and trim the full generated ID line before resolving it.
  - confidence: 0.93
- **[P2] Keep plugin approval tips request-scoped:** `src/infra/plugin-approvals.ts:70-72`
  - body: The added plugin approval tip always advertises `allow-always`, but plugin approvals can restrict `allowedDecisions` and the resolver rejects unavailable decisions. Build the reply-to tip from the request's allowed decision list so unavailable actions are not shown.
  - confidence: 0.95

## Security Review

Status: needs_attention

Summary: Reply ID extraction can submit an unintended approval ID; no dependency, workflow, or supply-chain changes were found.

Concerns:

- **[medium] Reply ID extraction can target another approval:** `src/auto-reply/reply/commands-approve.ts:40`
  - body: A replied `ID: task 1` line is submitted as `task`; because approval resolution accepts exact IDs or unique prefixes, this can fail the intended approval or resolve a different pending approval.
  - confidence: 0.9
