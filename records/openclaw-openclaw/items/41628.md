## Summary

Keep this PR open: current main already has approval ID prefix lookup, but it still lacks the reply-to `/approve <decision>` shortcut. The PR is still useful, but it needs contributor revision for two source-inspected approval-flow bugs plus redacted real Telegram proof before merge.

## What This Changes

The PR adds reply-to approval ID extraction for single-token `/approve <decision>`, adds prompt tips for reply/prefix approval shortcuts, and extends approval command/forwarder tests.

## Best Possible Solution

Keep the reply-to shortcut direction, but parse the full generated ID line, keep plugin prompt hints request-scoped, and attach redacted live Telegram proof before merge.

## Reproduction Assessment

Yes, for the review findings by source inspection: the PR regex truncates whitespace-containing IDs, and the plugin tip hard-codes decisions that current main validates per request. I did not run tests because this is a read-only review.

## Solution Assessment

No. The feature direction is narrow and useful, but this patch should preserve full generated IDs, avoid misleading plugin approval text, and provide real Telegram proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Parse the full replied approval ID line:** `src/auto-reply/reply/commands-approve.ts:40`
  - body: `APPROVAL_ID_RE` captures only `\S+`, so a replied `ID: task 1` line submits `task` even though approval IDs are generic non-empty strings and the gateway accepts unique prefixes. Capture and trim the full generated ID line before resolving it.
  - confidence: 0.93
- **[P2] Build plugin approval tips from allowed decisions:** `src/infra/plugin-approvals.ts:99`
  - body: The added plugin approval tip always advertises `allow-always`, but plugin approvals can restrict `allowedDecisions` and the resolver rejects unavailable decisions. Build this reply-to tip from the request's allowed decision list, matching the existing `Reply with` line.
  - confidence: 0.95

## Security Review

Status: needs_attention

Summary: Needs attention because reply ID extraction can submit a truncated or prefix ID into the approval resolver; no dependency, workflow, or supply-chain changes were found.

  - body: A replied `ID: task 1` line is submitted as `task`; because approval resolution accepts exact IDs or unique prefixes, this can fail the intended approval or resolve a different pending approval.
