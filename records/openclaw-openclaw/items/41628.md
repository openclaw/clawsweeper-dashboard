## Summary

Keep open: the reply-to shortcut is useful and not yet on main, but the current patch can truncate valid approval IDs and is missing the required user-facing changelog entry.

## What This Changes

The PR changes chat `/approve` handling to extract an approval ID from a replied-to message, updates approval prompt hints for reply/prefix shortcuts, and adds parser, handler, and forwarder tests.

## Best Possible Solution

Capture and trim the full `ID:` line for reply-to approvals, add regression coverage for whitespace-containing IDs and conflicting prefixes, and add a single-line Unreleased changelog entry.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: create an exec approval with a valid ID such as `task 1`, reply to the generated `ID: task 1` message with `/approve allow-once`, and the PR extractor submits only `task`.

## Solution Assessment

No. The reply-to shortcut is a reasonable direction, but the implementation should preserve the full approval ID and satisfy the changelog policy before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Capture the full replied approval ID line:** `src/auto-reply/reply/commands-approve.ts:41`
  - body: The new extractor uses `\S+`, so replying to a request with a valid ID containing whitespace, such as `ID: task 1`, submits only `task`. Gateway approval IDs are any non-empty trimmed string, and lookup prefers exact matches before prefix matching, so this can fail as ambiguous or approve a different pending `task`; capture the whole `ID:` line and trim it instead.
  - confidence: 0.92
- **[P3] Add the required changelog entry:** `CHANGELOG.md:11`
  - body: This is a user-facing `feat` for chat approvals, but the PR diff does not update `CHANGELOG.md`. Repo policy requires user-facing feature changes to include an Unreleased changelog entry before merge.
  - confidence: 0.86
