## Summary

Keep open. Current main already has the ID-prefix lookup half, but the remaining reply-to-message change is still useful and the branch currently has a concrete approval-routing bug before merge.

## What This Changes

The PR adds reply-only `/approve <decision>` handling from `ReplyToBody`, exports parser/extractor helpers for tests, and updates exec/plugin approval prompts to mention reply and prefix shortcuts.

## Best Possible Solution

Keep the reply-to-message UX, but parse the canonical full approval request ID line safely, preserve full trimmed IDs, cover spoofed/freeform `ID:` lines in tests, and add the required changelog entry before merge.

## Reproduction Assessment

Yes. A source-level reproduction is a plugin approval whose title or description contains a newline followed by `ID: plugin:other` before the canonical ID, then replying with `/approve allow-once`; the PR regex captures the earlier line. A custom exec approval ID containing whitespace also reproduces truncation.

## Solution Assessment

No. Reply-to-message approval is the right direction, but first-`ID:` plus `\S+` extraction is not the safest implementation for an approval path; parsing the canonical full rendered ID line is narrower and safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Capture the canonical full approval ID line:** `src/auto-reply/reply/commands-approve.ts:40`
  - body: `APPROVAL_ID_RE` matches the first line that starts with `ID:` and captures only the first non-space token. Plugin approval messages render freeform title and description before the canonical ID, and those fields can contain newlines, so reply-only approvals can route to a spoofed or truncated ID. Parse the canonical full ID line and cover embedded `ID:` lines plus whitespace-containing custom IDs.
  - confidence: 0.9

  - body: This PR adds user-facing approval behavior and updates approval prompt copy, but the diff does not add a `CHANGELOG.md` entry. Add a single-line Unreleased entry for the reply-to approval shortcut before merge.

## Security Review

Status: needs_attention

Summary: The diff has no CI, dependency, or supply-chain changes, but it changes approval resolution handling and can misroute reply-extracted approval IDs.

Concerns:

- **[medium] Reply extraction can misroute approvals:** `src/auto-reply/reply/commands-approve.ts:40`
  - body: The new extractor uses the first non-space token from the first `ID:` line in replied text. Because plugin approval text can contain earlier freeform `ID:` lines before the canonical ID, a reply-only approval can be applied to the wrong pending request.
