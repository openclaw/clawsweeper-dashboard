## Summary

Keep open: the reply-to approval shortcut is useful, but the current branch still has two blocking correctness issues and lacks real Telegram behavior proof. Prefix matching is already present on current main, but the reply-to change itself is not ready to merge.

## What This Changes

The PR adds reply-to `/approve <decision>` parsing from `ctx.ReplyToBody`, exports parser helpers for tests, and updates exec/plugin approval request text to mention reply-to and prefix shortcuts.

## Best Possible Solution

Keep the reply-to shortcut, parse the full generated ID line, make plugin tips reflect request-scoped allowed decisions, rebase, and attach redacted Telegram proof.

## Reproduction Assessment

Yes for the review findings by source inspection: the PR truncates whitespace-containing replied IDs and advertises unavailable plugin decisions while current main accepts generic IDs and validates request-scoped decisions. I did not run tests because this review is read-only.

## Solution Assessment

No. The direction is narrow and useful, but this patch should preserve full generated IDs, avoid misleading plugin approval text, and provide real Telegram proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Parse the exact generated approval ID line:** `src/auto-reply/reply/commands-approve.ts:40`
  - body: `APPROVAL_ID_RE` captures only `\S+`, so a replied `ID: task 1` line submits `task` even though approval IDs are generic non-empty strings and explicit `/approve allow-once <id>` preserves spaces. Capture and trim the full generated ID line before sending it to the prefix-capable resolver.
  - confidence: 0.93
- **[P2] Keep plugin approval tips request-scoped:** `src/infra/plugin-approvals.ts:71-72`
  - body: The added plugin approval tip always advertises `allow-always`, but plugin approvals can restrict `allowedDecisions` and the resolver rejects unavailable decisions. Build the reply-to tip from `resolvePluginApprovalRequestAllowedDecisions(request.request)` so unavailable actions are not shown.
  - confidence: 0.95
