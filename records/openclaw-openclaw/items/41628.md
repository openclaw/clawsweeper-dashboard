## Summary

Keep open: the PR’s reply-to approval shortcut is useful and not implemented on current main, but the current head still has a correctness/security bug in reply-body ID extraction and lacks required real Telegram proof.

## What This Changes

The PR adds reply-to `/approve <decision>` ID extraction from `ReplyToBody` and updates exec/plugin approval request hint text and tests to advertise reply-to and prefix shortcuts.

## Best Possible Solution

Preserve the reply-to shortcut, but capture and trim the full `ID:` line, add whitespace/custom-ID regression coverage, and attach redacted real Telegram proof before merge.

## Reproduction Assessment

Yes for the review finding by source inspection: the PR head captures only `\S+` from `ID:` lines, while current main accepts generic non-empty IDs and preserves multi-token explicit IDs. I did not run tests because this review is read-only.

## Solution Assessment

No. The reply-to shortcut is a reasonable narrow feature, but this implementation should preserve the full approval ID line and include real Telegram proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Capture the full replied approval ID line:** `src/auto-reply/reply/commands-approve.ts:40`
  - body: `APPROVAL_ID_RE` captures only `\S+`, so a reply to an approval message containing `ID: task 1` submits `task`. Approval IDs are generic non-empty strings and the gateway accepts exact IDs or unique prefixes, so the truncated value can fail the intended approval or resolve another pending approval; capture the whole `ID:` line and trim it.
