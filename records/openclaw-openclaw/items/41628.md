## Summary

Keep open: the reply-to `/approve` shortcut is useful and not yet on current main, but this PR still has two blocking correctness issues, lacks required real Telegram proof, and is currently conflicting with main.

## What This Changes

The PR adds reply-to `/approve <decision>` parsing from `ReplyToBody`, approval ID extraction tests, and hint text advertising reply-to approvals plus unique ID prefixes.

## Best Possible Solution

Keep the reply-to shortcut, but parse and trim the full `ID:` line, build plugin tip text from each request's allowed decisions, rebase on main, and attach redacted Telegram proof before merge.

## Reproduction Assessment

Yes for the review findings by source inspection: the PR truncates whitespace-containing replied IDs and hard-codes plugin decisions while current main supports request-scoped approval decisions. I did not run tests because this review is read-only.

## Solution Assessment

No. The direction is narrow and useful, but the current patch should preserve full replied IDs, keep plugin approval text request-scoped, and provide real Telegram proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Capture the full replied approval ID line:** `src/auto-reply/reply/commands-approve.ts:40`
  - body: `APPROVAL_ID_RE` captures only `\S+`, so replying to a message with `ID: task 1` submits `task`. Approval IDs are generic non-empty strings and current explicit `/approve allow-once <id>` preserves spaces, so this can fail or resolve the wrong prefix-capable approval.
  - confidence: 0.93
- **[P2] Keep plugin approval tips request-scoped:** `src/infra/plugin-approvals.ts:71-72`
  - body: The added tip always advertises `allow-always`, but plugin approvals can restrict `allowedDecisions`. Current text/tests and the resolver reject unavailable decisions, so requests allowing only `allow-once|deny` would display an action that fails.
  - confidence: 0.95
