## Summary

Keep this PR open: the reply-to `/approve <decision>` UX is not implemented on current main, but the current branch still has a concrete approval-ID extraction bug and is missing the required changelog entry before it should merge.

## What This Changes

The PR adds reply-only `/approve <decision>` handling from `ReplyToBody`, exports parser/extractor tests, and updates exec/plugin approval prompt hints while relying on main’s existing prefix lookup.

## Best Possible Solution

Update the branch to parse the canonical approval-message ID as a full trimmed line, add regression coverage for embedded `ID:` and whitespace-ID cases, and include one single-line Unreleased changelog entry.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a plugin approval title or description containing a newline followed by `ID: plugin:other` appears before the canonical ID, and a reply with `/approve allow-once` captures that earlier ID; an exec approval ID with internal whitespace is also truncated.

## Solution Assessment

No. Reply-to-message approval is the right UX direction, but first-`ID:` plus `\S+` extraction is not the narrowest safe implementation for an approval path; parsing the known canonical full ID line is safer.
