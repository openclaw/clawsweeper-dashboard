## Summary

Keep open: the scoped-down PR still adds useful reply-to-message approval UX, but the current diff has a concrete approval-ID extraction bug in a security-sensitive approval path and is missing the required changelog entry.

## What This Changes

The PR adds reply-to-message `/approve <decision>` ID extraction from `ReplyToBody`, updates exec/plugin approval prompt hints, and adds parser/extractor plus approval-flow tests while relying on mainâ€™s existing prefix lookup.

## Best Possible Solution

Update the branch to parse the canonical approval-message ID as a full trimmed line, add regression coverage for embedded `ID:` and whitespace-ID cases, and include one single-line Unreleased changelog entry.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a plugin approval title or description can contain a newline followed by `ID:` before the canonical ID, and the PR extractor captures that earlier non-space token; valid exec approval IDs with internal whitespace are also truncated.

## Solution Assessment

No. Reply-to-message approval is the right UX direction, but first-`ID:` plus `\S+` extraction is not the safest maintainable implementation for an approval path; parse the known canonical full ID line and keep gateway prefix lookup unchanged.
