## Summary

Keep open: this is a useful chat approval improvement, but the current PR still has two blocking correctness issues and lacks the required real Telegram behavior proof.

## What This Changes

The PR adds reply-to `/approve <decision>` parsing from replied approval messages, updates exec/plugin approval hint text, and adds parser/forwarder tests.

## Best Possible Solution

Keep the reply-to shortcut, parse the full generated ID line, render plugin tips from request-scoped decisions, and require redacted live Telegram proof before merge.

## Reproduction Assessment

Yes, for the review findings by source inspection: the PR regex truncates whitespace-containing IDs, and the plugin tip hard-codes decisions that current main validates per request. I did not run tests because this is a read-only review.

## Solution Assessment

No. The feature direction is narrow and useful, but the current patch should preserve full generated IDs, avoid misleading plugin approval text, and provide real Telegram proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Parse the full replied approval ID line:** `src/auto-reply/reply/commands-approve.ts:40`
  - body: `APPROVAL_ID_RE` captures only `\S+`, so a replied `ID: task 1` line submits `task` even though approval IDs are generic non-empty strings and the existing decision-first parser preserves spaces. Capture and trim the full generated ID line before resolving it.
  - confidence: 0.93
- **[P2] Keep plugin approval tips request-scoped:** `src/infra/plugin-approvals.ts:70-72`
  - body: The added plugin approval tip always advertises `allow-always`, but plugin approvals can restrict `allowedDecisions` and the resolver rejects unavailable decisions. Build this reply-to tip from the request's allowed decision list so unavailable actions are not shown.
  - confidence: 0.95
