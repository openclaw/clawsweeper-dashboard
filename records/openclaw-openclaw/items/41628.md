## Summary

Keep this PR open: the reply-to approval shortcut is useful and current main already has the ID-prefix backend, but the current diff has a concrete ID parsing bug and still lacks after-fix real Telegram proof.

## What This Changes

The PR adds reply-to approval ID extraction for single-token `/approve <decision>` commands and updates exec/plugin approval prompt tips to advertise reply-to and prefix shortcuts.

## Best Possible Solution

Keep the reply-to shortcut, capture the full `ID:` line and trim it, add a whitespace-ID regression test, then require redacted live Telegram proof before merge.

## Reproduction Assessment

Yes for the blocking review finding from source inspection: current schema and parser behavior allow space-containing IDs, while the new reply extractor truncates them at the first whitespace. I did not run tests because this review is read-only.

## Solution Assessment

No. Reply-to extraction is the right direction, but the implementation should preserve the full approval ID line and prove the real Telegram reply flow before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Capture the full approval ID line from replies:** `src/auto-reply/reply/commands-approve.ts:40`
  - body: `ExecApprovalRequestParams.id` is only a non-empty string, and the existing decision-first `/approve allow-once <id>` parser preserves spaces. The new reply extractor captures only `\S+`, so replying to an approval whose `ID:` line is `task 1` submits `task`; with prefix lookup that can fail or approve a different pending request. Capture the full line, trim it, and add a regression for a space-containing exec approval ID.
  - confidence: 0.9
