## Summary

Keep open. The reply-to approval UX is useful, but the current branch is not merge-ready because its reply-body parser can select a spoofed or truncated approval ID on an approval path; prefix lookup itself is already present on main.

## What This Changes

This branch adds reply-only `/approve <decision>` handling by extracting an approval ID from `ReplyToBody`, plus prompt text and tests for reply and prefix approval usage.

## Best Possible Solution

Update the branch to extract the canonical approval-message ID as a full trimmed line, cover embedded/freeform `ID:` lines and whitespace-containing IDs, and add a single-line Unreleased changelog entry.

## Reproduction Assessment

Yes. A source-level reproduction is a plugin approval whose title or description contains a newline followed by `ID: plugin:other` before the canonical ID, then replying with `/approve allow-once`; the PR regex captures the earlier line. An exec approval ID containing whitespace also reproduces truncation.

## Solution Assessment

No. Reply-to-message approval is the right UX direction, but first-`ID:` and `\S+` extraction is not the safest implementation for an approval path; parsing the known canonical full ID line is narrower and safer.
