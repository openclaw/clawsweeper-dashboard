## Summary

Keep open: current main already has prefix lookup, but it still lacks reply-to approvals, and the PR still has two blocking correctness issues plus missing real Telegram proof.

## What This Changes

The PR branch adds single-token `/approve <decision>` parsing from replied approval messages and updates exec/plugin approval hint text.

## Best Possible Solution

Keep the reply-to shortcut, parse the exact generated ID line, render approval tips from request-scoped decisions, and require redacted live Telegram proof before merge.

## Reproduction Assessment

Yes, for the review findings by source inspection: the PR regex truncates whitespace-containing IDs, and the plugin tip hard-codes decisions that current main validates per request. I did not run tests because this is a read-only review.

## Solution Assessment

No. The feature direction is narrow and useful, but the current patch should preserve full generated IDs, avoid misleading plugin approval text, and provide real Telegram proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Parse the full replied approval ID line:** `src/auto-reply/reply/commands-approve.ts:40`
  - body: `APPROVAL_ID_RE` captures only `\S+`, so a replied `ID: task 1` line submits `task` even though approval IDs are generic non-empty strings and the existing decision-first parser preserves spaces. Capture and trim the full generated ID line before resolving it.
  - confidence: 0.93
- **[P2] Keep plugin approval tips request-scoped:** `src/infra/plugin-approvals.ts:70-72`
  - body: The added plugin approval tip always advertises `allow-always`, but plugin approvals can restrict `allowedDecisions` and the resolver rejects unavailable decisions. Build this reply-to tip from the request's allowed decision list so unavailable actions are not shown.
  - confidence: 0.95

## Security Review

Status: needs_attention

Summary: Reply ID extraction can submit an unintended approval ID; no dependency, workflow, or supply-chain changes were found.

Concerns:

- **[medium] Reply ID extraction can target another approval:** `src/auto-reply/reply/commands-approve.ts:40`
  - body: A replied `ID: task 1` line is submitted as `task`; because approval resolution accepts exact IDs or unique prefixes, this can fail the intended approval or resolve a different pending approval.
  - confidence: 0.9
