## Summary

Keep open. Current main already has the prefix approval lookup portion, but this PR still adds reply-to approval extraction and prompt copy; the extractor can select a spoofed or truncated approval ID, and the user-facing change lacks the required changelog entry.

## What This Changes

This PR adds reply-only `/approve <decision>` handling by extracting an approval ID from replied-to message text, updates approval prompt hints, and adds parser/handler/forwarder tests.

## Best Possible Solution

Update the branch to parse the canonical approval-message ID as a full trimmed line from the known request shape, add spoof/truncation coverage, and add a single-line Unreleased changelog entry while leaving main's prefix lookup intact.

## Reproduction Assessment

Yes. A source-level reproduction is a plugin approval whose title, description, or other rendered freeform field contains a newline followed by `ID: plugin:<other pending id>` before the canonical ID, then replying with `/approve allow-once`; a whitespace-containing exec approval ID also reproduces truncation.

## Solution Assessment

No. Reply-to-message approval is the right UX direction, but first-`ID:` and `\S+` parsing is not the safest implementation for an approval path; parsing the canonical full ID line or using structured reply metadata is narrower and safer.
