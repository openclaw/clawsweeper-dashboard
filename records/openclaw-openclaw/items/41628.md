- **[P1] Keep the current approval resolver path:** `src/auto-reply/reply/commands-approve.ts:235-237`
  - body: Current main routes `/approve` through `resolveApprovalOverGateway` because generic `callGateway` submissions can miss approval-runtime visibility and return `unknown or expired` for live exec/plugin approvals. Rebase this reply-to change on top of that resolver path instead of submitting through `callGateway`.
  - confidence: 0.93
- **[P1] Keep the current gateway scope gate:** `src/auto-reply/reply/commands-approve.ts:224-228`
  - body: This branch still calls `requireGatewayClientScopeForInternalChannel`, while current main enforces `operator.approvals` or `operator.admin` whenever a gateway scope context is present. A conflict resolution that keeps the branch helper would let scoped gateway callers bypass the newer `/approve` gate when route metadata names an external channel.
  - confidence: 0.91
- **[P2] Capture the full replied approval ID line:** `src/auto-reply/reply/commands-approve.ts:40`
  - body: `APPROVAL_ID_RE` captures only `\S+`, but approval IDs are generic non-empty strings. A replied message with `ID: task 1` would submit `task`, which can fail, become ambiguous, or hit a different pending approval once prefix matching is in play; capture and trim the whole line instead.
  - confidence: 0.88
- **[P2] Use request-scoped plugin approval decisions:** `src/infra/plugin-approvals.ts:72`
  - body: Current main formats plugin approval text from `resolvePluginApprovalRequestAllowedDecisions`, but this branch adds a hard-coded reply-to tip that always advertises `allow-always`. Restricted plugin approvals can then show a command the server rejects, so the tip needs to use the request's allowed decisions too.
  - confidence: 0.86
