## Summary

Keep open. The reply-to approval UX is useful, and current main already covers prefix matching, but this PR can still mis-extract approval IDs from replied messages on a security-sensitive approval path.

## What This Changes

The branch adds reply-only `/approve <decision>` handling from `ReplyToBody`, exports parser/extractor helpers for tests, and updates exec/plugin approval prompts to mention reply and prefix shortcuts.

## Best Possible Solution

Keep the PR open for an author update that extracts the canonical full `ID:` line from the known approval-message structure, covers spoofed/freeform `ID:` lines and whitespace-containing IDs, and adds the required changelog entry.

## Reproduction Assessment

Yes. A source-level reproduction is a plugin approval whose title or description contains a newline followed by `ID: plugin:other` before the canonical ID, then replying with `/approve allow-once`; the PR regex captures the earlier line. A custom exec ID containing whitespace also reproduces truncation.

## Solution Assessment

No. Reply-to-message approval is the right UX direction, but first-`ID:` and `\S+` extraction is not the safest implementation for an approval path; parsing the canonical full ID line is narrower and safer.
