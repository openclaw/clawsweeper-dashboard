## Summary

Keep open: the feature direction is useful, but this PR is not merge-ready because it lacks real WhatsApp proof, is currently merge-conflicting, and introduces a batch-context defect where forwarded metadata can describe only the last message in a combined inbound batch.

## What This Changes

The PR adds WhatsApp Baileys forwarding metadata extraction, optional inbound message fields, forwarded-context prompt rendering for `forwarding_score`, and focused contract tests.

## Best Possible Solution

Land a rebased WhatsApp-owned implementation that extracts forwarding metadata once, defines batch-safe semantics, coordinates prompt and hook metadata with the open hook issue, and includes redacted live WhatsApp proof.

## Reproduction Assessment

Yes, source-level for the review finding: current main debounces multiple WhatsApp messages into one combined message copied from the last entry, while the PR attaches forwarding fields before that copy. Real WhatsApp proof for the feature itself is still missing.

## Solution Assessment

No, not as-is. The additive WhatsApp metadata direction is reasonable, but the implementation needs batch-safe semantics, a rebase, real WhatsApp proof, and maintainer scope alignment with https://github.com/openclaw/openclaw/issues/69374.
