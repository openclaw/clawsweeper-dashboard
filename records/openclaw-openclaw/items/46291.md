## Summary

Keep open: current main still does not surface WhatsApp forwarding metadata, but this PR is not merge-ready because debounced batches can misattribute the new metadata and live WhatsApp proof is missing.

## What This Changes

The PR extracts Baileys WhatsApp forwarding metadata, adds optional inbound message fields, renders forwarding metadata in the agent prompt, and adds contract coverage.

## Best Possible Solution

Land a rebased WhatsApp-owned implementation that extracts forwarding metadata once, handles single and batched messages without misattribution, coordinates the prompt/plugin hook surface with https://github.com/openclaw/openclaw/issues/69374, and includes redacted live proof.

## Reproduction Assessment

Yes, source-level. Current main has Baileys context extraction and prompt/hook carriers without `isForwarded` or `forwardingScore`, and the PR's batch defect follows from the debouncer building combined messages from `...last` while replacing the body with all entries.

## Solution Assessment

No, not merge-ready. The additive direction fits the WhatsApp plugin boundary, but the implementation needs batch-safe metadata semantics and contributor live WhatsApp proof before maintainers can merge it.

Overall correctness: patch is incorrect

Overall confidence: 0.78

- **[P2] Preserve forwarding metadata across debounced batches:** `extensions/whatsapp/src/inbound/monitor.ts:385-386`
