## Summary

Keep open: current main still drops WhatsApp forwarding metadata, but this PR is not merge-ready because it lacks real WhatsApp proof and can misattribute forwarding metadata in debounced batches.

## What This Changes

The PR adds Baileys WhatsApp forwarding metadata extraction and threads `isForwarded`/`forwardingScore` through inbound messages, agent prompt context rendering, and contract tests.

## Best Possible Solution

Land a rebased WhatsApp-owned implementation that extracts forwarding metadata once, handles single and batched inbound messages without misattribution, exposes the agreed agent/plugin context, and includes redacted live proof.

## Reproduction Assessment

Yes, source-level: current main has raw `contextInfo` extraction but no forwarding carrier or prompt/hook field, and the PR's batch issue follows from the debouncer's `...last` combined message. Live WhatsApp forwarding was not reproduced because the PR body explicitly says no end-to-end account test was run.

## Solution Assessment

No, not merge-ready. The additive direction fits the WhatsApp plugin boundary, but the patch needs batch-safe metadata handling, live proof, and maintainer scope judgment with https://github.com/openclaw/openclaw/issues/69374.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.78

Full review comments:

- **[P2] Preserve forwarding metadata across debounced batches:** `extensions/whatsapp/src/inbound/monitor.ts:385-386`
  - body: When debounce combines several WhatsApp messages, the combined payload is copied from the last entry while the body contains all entries. Copying `isForwarded`/`forwardingScore` from only that entry can drop an earlier forwarded message or mark direct text as forwarded when only the last item was forwarded; carry per-entry forwarding facts or omit them for mixed batches.
  - confidence: 0.78
