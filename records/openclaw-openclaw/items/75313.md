## Summary

Keep this issue open. Current main still destructively drops blank-only user messages during session-file repair, and the linked closing PR #75368 is open and unmerged, so the issue should remain open until that implementation or an equivalent fix lands.

## What This Changes

The issue requests changing session-file repair so blank-only user-role messages are rewritten to a non-empty placeholder instead of being removed from persisted history.

## Best Possible Solution

Land PR #75368 or an equivalent narrow fix that rewrites blank-only persisted user entries to a neutral non-empty placeholder, preserves media-block cleanup behavior, updates the existing destructive-drop regression test, and adds coverage for a repaired transcript that would otherwise contain only system/assistant history.

## Reproduction Assessment

Yes. The current helper and its regression test provide a high-confidence source reproduction: a blank-only user session entry is removed during repair, and the repair pass runs before session replay in both normal attempts and compaction.

## Solution Assessment

Yes. The narrowest maintainable solution belongs in session-file repair with focused tests; provider-specific transport guards should only be considered if review proves session repair does not cover the failing path.
