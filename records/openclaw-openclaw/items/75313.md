## Summary

Keep this issue open. Current main still destructively drops blank-only user-role session entries during session-file repair, and the linked implementation PR is open and unmerged, so the issue should stay open until that PR or an equivalent fix lands.

## What This Changes

The issue asks for session-file repair to preserve blank-only user-role messages by rewriting them to a non-empty placeholder instead of removing the user turn from persisted JSONL history.

## Best Possible Solution

Land PR #75368 or an equivalent narrow change that rewrites blank-only persisted user entries to a neutral non-empty placeholder, preserves mixed media-block cleanup, updates the repair counters/log wording, and replaces the destructive-drop regression test with coverage for assistant-only post-repair history.

## Reproduction Assessment

Yes. The current helper plus existing unit test provide a high-confidence source reproduction: a blank persisted user entry is removed from the repaired file, and the repair hook runs before session replay in both normal attempts and compaction.

## Solution Assessment

Yes. The best fix is a focused change in `src/agents/session-file-repair.ts` with colocated regression tests; provider-specific OpenAI-compatible transport guards should only be added if later evidence shows this file-repair path does not cover a separate failure.
