## Summary

Keep open. Current main still destructively removes blank-only user-role session entries during session-file repair, and the existing regression test explicitly expects that drop, so the reported strict-provider failure remains actionable.

## What This Changes

The issue asks OpenClaw to rewrite blank-only persisted user messages to a non-empty neutral placeholder instead of dropping them during session-file repair.

## Best Possible Solution

Centralize blank user-message cleanup so persisted and in-memory replay paths preserve strict-provider-valid turn shape: remove blank text blocks, keep media blocks, and rewrite blank-only user turns to a neutral non-empty placeholder without adding provider-specific Qwen or MLX special cases.

## Reproduction Assessment

Yes. A focused session JSONL fixture with one blank user entry followed by assistant entries exercises `repairSessionFileIfNeeded` and currently proves the user entry is removed; the reported Qwen/mlx-vlm 500 is downstream confirmation, not required to reproduce the source bug.

## Solution Assessment

Yes, with one scope adjustment. Rewriting instead of dropping is the right direction, but the safest fix should use or share a neutral existing placeholder and audit the in-memory replay drop path rather than adding only a transport-level provider guard.
