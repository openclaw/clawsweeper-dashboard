## Summary

Keep open: the Feishu invite path is useful and the PR supplies live log proof, but the patch should not merge as-is because it records a synthetic `vc-meeting:` value as the Feishu last delivery route for future session sends.

## What This Changes

The PR registers Feishu `vc.bot.meeting_invited_v1` events and converts them into synthetic no-reply inbound turns that ask the agent to join by meeting number with `call_id` context.

## Best Possible Solution

Keep the Feishu-owned invite handler, but stop writing the synthetic meeting target into deliverable session route state before merge.

## Reproduction Assessment

Yes for the review finding: the PR source writes `vc-meeting:<meetingNo>` into `record.updateLastRoute`, and current session delivery code later inherits stored `lastTo` for implicit sends. I did not run the PR branch in this read-only review.

## Solution Assessment

No; the plugin-local event handler is the right boundary, but a no-reply synthetic turn should not persist a fake Feishu delivery target. The safer fix is to omit the last-route update or store a real inviter DM target only if maintainers want future replies to DM the inviter.
