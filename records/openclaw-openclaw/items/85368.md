## Summary

Keep open. The direction matches the plugin boundary guidance, but the patch currently narrows the audio provider listing in ways that can hide a bundled provider and diverge from disabled-plugin behavior, and the external PR still needs real CLI proof.

## What This Changes

The branch switches `openclaw capability audio providers` to media-understanding manifest metadata, adds control-plane filtering, and narrows cold media provider loading to configured media providers.

## Best Possible Solution

Use manifest metadata for the control-plane listing, but first make it behaviorally equivalent to the current provider set by covering missing bundled metadata, honoring disabled-plugin config, and validating the real CLI output.

## Reproduction Assessment

Source-reproducible: current main's audio provider listing calls the runtime media registry, which flows through plugin capability runtime loading. I did not run a live installed cold-start command because this was a read-only review.

## Solution Assessment

No, not yet. Manifest metadata is the right owner-boundary direction for a control-plane listing, but the patch needs metadata parity and disabled-plugin semantics before it is the safest fix.

  - body: Switching `audio providers` to `buildMediaUnderstandingManifestMetadataRegistry` drops runtime audio providers that do not have manifest metadata. This can make `openclaw capability audio providers` report disabled bundled providers as available, so please keep the disabled case empty or explicitly mark it unavailable with coverage.
