## Summary

Keep open: the follow-up fixes address the prior SenseAudio/global-disable gaps and the PR now has real CLI proof, but the manifest-based listing still does not preserve documented plugin allow/disable semantics and may change the external plugin listing contract.

## What This Changes

This PR changes `capability audio providers` to read media provider metadata from plugin manifests, adds SenseAudio metadata, filters metadata by availability, and narrows fallback runtime media provider loading.

## Best Possible Solution

Use manifest metadata for the cold listing only after it shares the same plugin activation and allow/deny/entry-disabled policy as runtime loading, and preserve or explicitly migrate the external plugin metadata contract.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: apply this PR and run `capability audio providers` with a bundled audio provider disabled by `plugins.entries.<id>.enabled:false` or excluded by `plugins.allow`; the patched metadata helper still treats bundled plugins as available. A metadata-less external audio provider is also source-reproducible as omitted from the new metadata-only listing path.

## Solution Assessment

No, not yet. Manifest metadata is the right direction for avoiding eager runtime startup, but the current patch needs plugin availability parity and a clear compatibility path for optional metadata before it is the narrow maintainable fix.
