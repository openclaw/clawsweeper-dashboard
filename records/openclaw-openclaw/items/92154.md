## Summary

This protected maintainer PR addresses a real QQBot group-safety gap, but it should not merge as written: the new policy duplicates the canonical command registry, the implicit `safety` default changes existing group behavior, and the PR provides tests rather than real QQBot proof.

## What This Changes

The PR adds per-group QQBot `all`, `safety`, and `strict` command levels, classifies core slash commands in the QQBot plugin, and replies that blocked commands are private-chat-only.

## Best Possible Solution

Keep command enforcement and user-facing responses in QQBot, but source exposure metadata from the canonical command owner through a narrow SDK projection, preserve existing behavior by default unless maintainers explicitly approve the break, and add live upgrade/fresh-config proof.

## Reproduction Assessment

Yes from source: an existing group without `commandLevel` resolves to `safety`, and the new gate rejects selected commands before normal routing. Real transport behavior has not been demonstrated.

## Solution Assessment

No. QQBot-local enforcement is the correct owner boundary, but copying the evolving core registry and changing the default is less maintainable and less upgrade-safe than consuming canonical metadata with an opt-in strict policy.
