## Summary

This PR should stay open: its per-group policy is useful and the focused checks are green, but `strict` mode is not actually strict because unclassified slash commands bypass the new gate, and the compatibility-sensitive default still needs explicit owner acceptance plus live QQBot proof.

## What This Changes

Adds per-group QQBot command levels (`all`, `safety`, and `strict`) and rejects group commands classified as private-only with a private-chat guidance reply.

## Best Possible Solution

Keep the group policy in QQBot, but make strict mode default-deny every slash command except `/new` and `/reset`, and derive safety classification from a canonical plugin-safe command metadata contract or enforce exhaustive drift coverage.

## Reproduction Assessment

## Solution Assessment

No. The product direction is plausible, but the implementation must default-deny strict mode and avoid an incomplete second inventory of canonical core commands.
