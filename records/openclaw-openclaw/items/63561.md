## Summary

Keep open: current main still has the reported user-systemd false-unavailable path, and the open closing implementation at https://github.com/openclaw/openclaw/pull/73888 has not merged.

## What This Changes

The issue reports that gateway status/install misclassify a working Linux user-systemd service as unavailable when spawned `systemctl --user` calls fail to connect to the user bus.

## Best Possible Solution

Land a narrow daemon/systemd fix, likely via the existing linked PR after repair, that recovers missing DBus/XDG env for user `systemctl` spawns while preserving target-user scope and service-secret boundaries.

## Reproduction Assessment

Yes, source-level. The issue provides concrete Ubuntu commands/logs, and current main still has a mocked `Failed to connect to bus: Permission denied` path that resolves user systemd as unavailable without env recovery.

## Solution Assessment

Yes, the daemon/systemd env-recovery direction is the narrow maintainable fix. Status-only wording or hints would still leave `gateway install` unable to manage a working user service, and the linked PR should be repaired rather than duplicated.
