## Summary

Current main and the published v2026.5.19 release now recover the Linux user systemd bus environment for shared `systemctl --user` gateway calls, so the reported false-unavailable status/install path is implemented.

## What This Changes

The issue reports that Linux gateway status/install misclassifies a working user systemd service as unavailable when spawned `systemctl --user` calls fail to connect to the user bus.

## Best Possible Solution

Keep the shipped shared systemd env-recovery implementation and handle the now-stale linked PR separately rather than leaving this fixed user report open.

## Reproduction Assessment

Yes, source-level. The reporter provided concrete Ubuntu commands/logs, and the current regression test covers the missing DBus/XDG environment case that previously caused `systemctl --user` to be classified as unavailable.

## Solution Assessment

Yes. Centralizing DBus/XDG recovery in the daemon/systemd `systemctl` helper is the narrow maintainable fix because it covers gateway status and install without command-specific warning-only behavior.
