## Summary

The LM Studio delivery bug and live fix are convincing, but the PR should not merge while it exposes a helper whose current contract explicitly marks it deprecated and unsuitable for third-party plugins. The runtime changes can remain useful after maintainers choose a stable plugin-SDK boundary.

## What This Changes

The PR makes thinking off rewrite or remove OpenAI-completions reasoning_effort for LM Studio and other wrapped compatible models, adds none/off map-key aliases, exports the thinking wrapper through the plugin SDK, and updates tests and documentation.

## Best Possible Solution

Retain the canonical disabled-effort resolver and LM Studio owner-boundary composition, but expose a stable family-level or purpose-built provider-stream seam instead of the deprecated wrapper; alternatively, explicitly redesign the wrapper as supported public API with aligned annotations, documentation, and compatibility tests.

## Reproduction Assessment

Yes. Current main's LM Studio hook lacks the thinking-level payload wrapper while the completions path can supply an enabled default, so thinking off can reach LM Studio as enabled reasoning; the supplied live before/after output matches that path.

## Solution Assessment

No, not in its current API shape. The payload rewrite is a narrow and well-proven fix, but exporting a helper explicitly marked deprecated and prohibited for third-party plugins is inferior to a stable provider-family or purpose-built SDK seam.
