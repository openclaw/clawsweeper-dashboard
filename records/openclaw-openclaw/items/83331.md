## Summary

Keep open: current main only preserves numeric-looking object keys for the patch path, while path-based `config set` still passes through the array-index heuristic. The related fix PR at https://github.com/openclaw/openclaw/pull/83341 was closed unmerged because the naive global option flip would break bracket-index array paths.

## What This Changes

The issue asks `openclaw config patch` and `openclaw config set` to preserve numeric-looking string keys for Record/object config paths such as Discord guild IDs instead of treating them as array indices.

## Best Possible Solution

Make config path parsing or missing-container creation distinguish bracket array indices from dot numeric Record keys, or use schema-aware container selection, then add regression coverage for both Discord snowflake object keys and real array paths.

## Reproduction Assessment

Yes, at source level: `config set` still leaves `numericObjectKeys` false, and `setAtPath` then creates array parents for all-digit next segments even though `channels.discord.guilds` is a string-keyed record. I did not execute the CLI because this review is read-only.

## Solution Assessment

Yes, the best fix is a narrow config CLI path-construction repair plus tests. The already-closed option-flip PR is not the best implementation because it can break bracket array paths.
