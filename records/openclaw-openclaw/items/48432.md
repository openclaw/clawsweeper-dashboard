## Summary

Keep open. Current main still rewrites long unprotected TUI tokens by inserting literal spaces, hyphen-only package names still miss the copy-sensitive guard, and the issue is paired with open unmerged PR #69340, so maintainer review should decide whether that narrow PR or a broader code/table-aware fix lands.

## What This Changes

The issue asks for TUI code blocks with long lines to render without inserted spaces at wrap positions, preserving copyable code and table content.

## Best Possible Solution

Preserve copy-sensitive fenced code and table-like TUI content without injecting literal display spaces, then land or replace the open PR with focused regression tests for the hyphenated package-name example and the code/table rendering boundary.

## Reproduction Assessment

Yes. The code-level reproduction against current main shows the sanitizer turns a 33-character hyphenated package token into a token with an inserted literal space, and assistant TUI content still routes through that sanitizer before rendering.

## Solution Assessment

Unclear. The open PR is a plausible narrow fix for the package-name example, but the safer end state should validate fenced code blocks and box-drawn tables rather than only broadening the file-like token guard.
