## Summary

Keep open. Current main still has the TUI formatter path that can insert literal spaces into long non-space tokens, hyphen-only package names are not protected, assistant messages still pass through that formatter, and the linked closing PR #69340 is still open and unmerged in the provided GitHub context.

## What This Changes

The issue reports that the TUI inserts visible spaces into long code-block content at wrap positions, corrupting copied code, tables, and shell continuations.

## Best Possible Solution

Preserve copy-sensitive fenced code, table, and shell continuation content in the TUI without injecting literal display spaces, then land or replace #69340 with focused regression coverage for hyphenated package names, fenced code blocks, box-drawn tables, and backslash line continuations.

## Reproduction Assessment

Yes. The current formatter code path deterministically turns long unprotected hyphenated tokens into 32-character chunks joined by literal spaces, and assistant TUI rendering still routes message text through that sanitizer before display.

## Solution Assessment

Unclear. The linked PR is a plausible narrow fix for the package-name example, but the safest maintainable fix should validate the whole code-block/table/copy boundary rather than only adding hyphens to the existing token guard.
