Keep open: the report describes a concrete Telegram topic-delivery failure, but current main already has source paths and regression tests that should carry `threadId` into Telegram `message_thread_id`, so this needs focused live/runtime triage rather than cleanup closure.

The issue reports that Telegram forum-topic replies preserve thread state internally but outbound `sendMessage` calls omit Telegram's `message_thread_id`, causing replies to land in the supergroup main chat instead of the topic.

## Best Possible Solution

Identify the exact runtime path that produced a threadless Telegram topic send, preserve the existing plugin-owned thread pass-through, and add a regression test for any uncovered bypass.
