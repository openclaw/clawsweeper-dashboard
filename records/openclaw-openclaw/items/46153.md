## Summary

Keep open. Current main still lacks a central stdin binding for generated macOS Gateway LaunchAgent plists, and fresh install, staging, and restart-repair all reuse that same renderer, so the reported launchd-only hang is not implemented or superseded.

## What This Changes

The issue asks OpenClaw to prevent macOS launchd-started Gateway processes from hanging silently at startup, with the focused suspected fix being `StandardInPath=/dev/null` in generated LaunchAgent plists.

## Best Possible Solution

Add `StandardInPath` pointing at `/dev/null` in the shared macOS LaunchAgent plist generation path, cover fresh install/staging and restart-rewrite behavior in launchd tests, and add a user-facing changelog fix entry without mixing in the separate external-volume log-path issue.

## Reproduction Assessment

Yes. The issue includes concrete `launchctl` reproduction steps and current-main source inspection confirms the generated LaunchAgent still lacks the stdin binding implicated by the report; live macOS execution was not run in this read-only pass.

## Solution Assessment

Yes. The narrowest maintainable fix is in the shared LaunchAgent plist renderer because install, staging, and restart-repair all reuse it; a manual plist edit or docs-only workaround would be overwritten.
