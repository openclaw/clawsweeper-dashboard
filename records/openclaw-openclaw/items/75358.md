## Summary

Keep open. Current main still has the reported text-only degradation path when PDF fallback image rendering needs `@napi-rs/canvas`, there is no Poppler/pdftoppm fallback in the plugin/tool/docs surface, and open PR #75370 already targets this issue with closing syntax, so the issue should remain open until that PR is reviewed and either landed or rejected.

## What This Changes

Requests a `document-extract` PDF page-image fallback to Poppler/pdftoppm when `@napi-rs/canvas` is unavailable, instead of returning text-only extraction with no images.

## Best Possible Solution

Review PR #75370 or a bounded equivalent in the bundled `document-extract` plugin, with regression coverage for missing canvas, documented Poppler dependency behavior, and a clear security boundary for external PDF rendering.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: use a low-text PDF with a non-native PDF/vision fallback path and no `@napi-rs/canvas`; current main reaches `loadCanvasModule()`, catches the missing optional dependency, and returns `images: []`.

## Solution Assessment

Unclear. A pdftoppm fallback is a plausible narrow owner-module fix, but it should only ship after maintainer/security review of the external renderer boundary and user-facing dependency behavior.
