## Summary

Keep open. Current main still has the reported canvas-only PDF image rendering path, and open PR #75370 already uses closing syntax for the same fix, so this issue should wait for that PR's review rather than be closed or queued separately.

## What This Changes

The issue asks for a Poppler/pdftoppm PDF page-image fallback in the bundled `document-extract` plugin when optional `@napi-rs/canvas` is unavailable.

## Best Possible Solution

Review and land a bounded `document-extract` Poppler fallback in #75370, or explicitly surface and document canvas-missing image degradation if maintainers reject that fallback.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: a low-text PDF reaches image rendering, a missing `@napi-rs/canvas` import is caught, and the extractor returns no images while the agent `pdf` tool omits the warning callback.

## Solution Assessment

Unclear. A `pdftoppm` fallback in the owner plugin is a plausible narrow fix, but the external renderer boundary needs bounds, cleanup, binary-discovery, and failure-reporting review before it is clearly the best accepted solution.
