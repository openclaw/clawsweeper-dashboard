## Summary

Keep open. Current main still has the reported canvas-missing path: PDF image rendering depends on `@napi-rs/canvas`, catches the missing optional import, and returns `images: []`; the agent `pdf` tool path still does not surface the warning callback. An open linked PR (#75370) already targets this issue with closing syntax, so this should stay open for review of that implementation rather than be closed or queued as duplicate repair work.

## What This Changes

Add graceful PDF page-image rendering fallback, such as `pdftoppm`, when `@napi-rs/canvas` is unavailable so extraction fallback does not silently degrade to text-only output.

## Best Possible Solution

Review PR #75370 and either land a bounded, documented Poppler fallback with regression coverage or choose a clearer canvas-missing degradation/error policy in the bundled `document-extract` plugin and PDF docs.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: a low/no-text PDF in extraction fallback mode reaches image rendering, a failed `@napi-rs/canvas` import is caught, and the extractor returns `{ text, images: [] }`; the agent `pdf` tool path does not pass the callback that would surface the warning.

## Solution Assessment

Unclear. A `pdftoppm` fallback is a plausible owner-module fix, but it executes an external renderer on user-supplied PDFs, so the linked PR needs security and maintainer review before that path is accepted as the best solution.
