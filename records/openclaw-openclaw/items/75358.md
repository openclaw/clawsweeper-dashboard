## Summary

Keep open. Current main still returns no PDF page images when `@napi-rs/canvas` is unavailable, and open PR #75370 already targets this issue with closing syntax, so this should wait for that PR’s functional and security review instead of being closed or queued separately.

## What This Changes

The issue requests a `document-extract` fallback PDF page renderer, such as `pdftoppm`, when `@napi-rs/canvas` is unavailable.

## Best Possible Solution

Review PR #75370 and land a bounded `document-extract` plugin fallback, or explicitly document and surface canvas-missing degradation if Poppler is rejected.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: a low-text PDF reaches image rendering, a missing `@napi-rs/canvas` import is caught, and the extractor returns no images; the agent `pdf` tool path also omits the warning callback.

## Solution Assessment

Unclear. A `pdftoppm` fallback is a plausible owner-module fix, but the external renderer boundary needs maintainer/security review before it is the best accepted solution.
