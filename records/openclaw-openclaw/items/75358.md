## Summary

Keep open. Current main still degrades to text-only PDF extraction with `images: []` when `@napi-rs/canvas` cannot be imported, there is no Poppler/pdftoppm fallback in the current source or docs, and open PR #75370 already targets this issue with closing syntax.

## What This Changes

Request a `pdftoppm`/Poppler page-image rendering fallback in the bundled `document-extract` plugin when the optional `@napi-rs/canvas` dependency is unavailable.

## Best Possible Solution

Review PR #75370 and either land a bounded, documented Poppler fallback in `extensions/document-extract` with regression coverage, or intentionally document a clearer canvas-missing degradation/error policy for the PDF extraction fallback.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: a low-text PDF reaches image rendering, missing `@napi-rs/canvas` makes `loadCanvasModule()` throw, and the catch path returns text with `images: []`.

## Solution Assessment

Unclear. A `pdftoppm` fallback is a plausible narrow plugin-owned fix, but it changes the external-process boundary for untrusted PDFs, so the linked PR needs focused security and maintainer review before it is the best accepted solution.
