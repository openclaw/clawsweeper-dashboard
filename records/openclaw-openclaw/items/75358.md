## Summary

Keep open. Current main and the shipped v2026.5.2 release still use the canvas-only PDF image-rendering path, and open PR #75370 uses closing syntax for this issue, so the issue should wait for that PR to be reviewed, merged, or rejected.

## What This Changes

The issue asks for the bundled document-extract PDF image renderer to fall back to Poppler `pdftoppm` when optional `@napi-rs/canvas` is unavailable.

## Best Possible Solution

Review and either land a bounded `document-extract` fallback in PR #75370 or explicitly surface and document canvas-missing image degradation if Poppler is rejected.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: a low-text PDF reaches image rendering, the missing `@napi-rs/canvas` import is caught, and the extractor returns no images; the agent `pdf` tool path does not pass the warning callback.

## Solution Assessment

Unclear. A `pdftoppm` fallback is a plausible plugin-owned fix, but the external renderer boundary needs maintainer/security review before it is the accepted best solution.
