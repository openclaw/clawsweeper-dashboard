## Summary

Keep open. Current main still has the reported canvas-missing path: the bundled PDF extractor catches a missing `@napi-rs/canvas` import and returns `images: []`, while the agent `pdf` tool does not pass the warning callback. Open PR #75370 already targets this with closing syntax, so this issue should remain open for review of that implementation rather than be closed or queued as a separate repair.

## What This Changes

Request a graceful PDF page image-rendering fallback, such as `pdftoppm`, when optional `@napi-rs/canvas` is unavailable on headless Linux installs.

## Best Possible Solution

Review PR #75370 and either land a bounded, documented Poppler fallback with regression coverage or choose a clearer canvas-missing degradation/error policy in the bundled `document-extract` plugin and PDF docs.

## Reproduction Assessment

Yes. Current-main static reproduction is high-confidence: a low/no-text PDF reaches image rendering, a failed `@napi-rs/canvas` import is caught, and the extractor returns `{ text, images: [] }`; the agent `pdf` tool does not pass the callback that would surface the warning.

## Solution Assessment

Unclear. A `pdftoppm` fallback is a plausible owner-module fix, but it expands the external process boundary for untrusted PDF input, so the linked PR needs security and maintainer review before treating that path as the best solution.
