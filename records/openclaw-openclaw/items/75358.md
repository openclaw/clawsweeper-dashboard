## Summary

Keep open. Current main still has canvas-only PDF rasterization that returns empty images when `@napi-rs/canvas` is unavailable, and #75370 is an open unmerged PR with closing syntax, so this issue should wait for that PR review rather than be closed or queued separately.

## What This Changes

The issue requests a Poppler/pdftoppm fallback for PDF page image rendering when `@napi-rs/canvas` is unavailable.

## Best Possible Solution

Review #75370 and land a bounded `document-extract` Poppler fallback, or explicitly document and surface canvas-missing degradation if Poppler is rejected.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: a low-text PDF reaches image rendering, missing `@napi-rs/canvas` makes `loadCanvasModule()` throw, and the extractor returns no images while the agent `pdf` tool does not pass the warning callback.

## Solution Assessment

Unclear. A `pdftoppm` fallback is a plausible owner-module fix, but the external renderer boundary needs maintainer/security review before it is the best accepted solution.
