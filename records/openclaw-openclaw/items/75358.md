## Summary

Keep open. Current main still has the reported canvas-only PDF image-rendering path, and open PR #75370 already uses closing syntax for this issue, so the issue should wait for that PR to be reviewed, merged, or rejected.

## What This Changes

Request to add graceful PDF page-image rendering fallback, such as Poppler `pdftoppm`, when optional `@napi-rs/canvas` is unavailable in the bundled `document-extract` plugin.

## Best Possible Solution

Review and land a bounded `document-extract` plugin fallback in #75370, or explicitly surface and document canvas-missing image degradation if Poppler is rejected.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: a low-text PDF reaches image rendering, the missing `@napi-rs/canvas` import is caught, and the extractor returns no images; the agent `pdf` tool path does not pass the warning callback.

## Solution Assessment

Unclear. A `pdftoppm` fallback is a plausible owner-module fix, but the external renderer boundary needs maintainer/security review before it is the best accepted solution.
