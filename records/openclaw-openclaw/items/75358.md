## Summary

Keep open. Current main still catches the missing `@napi-rs/canvas` import and returns text-only PDF extraction with `images: []`, no Poppler fallback exists in the relevant plugin/tool/docs surface, and an open linked PR already targets this issue with closing syntax, so the issue should stay open until that PR is reviewed and either landed or rejected.

## What This Changes

The issue requests a Poppler/`pdftoppm` fallback for PDF page image extraction when the optional `@napi-rs/canvas` package is unavailable.

## Best Possible Solution

Review PR #75370 and either land a bounded, documented Poppler fallback with regression coverage or choose a clearer canvas-missing degradation/error policy in the bundled `document-extract` plugin and PDF docs.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: a low-text PDF reaches image rendering, `loadCanvasModule()` throws when `@napi-rs/canvas` is unavailable, and the catch path returns `{ text, images: [] }`.

## Solution Assessment

Unclear. A `pdftoppm` fallback is a plausible narrow owner-module fix, but it changes the external-process and untrusted-PDF processing contract, so the linked PR needs security and maintainer review before it is considered the best solution.
