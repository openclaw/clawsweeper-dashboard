## Summary

Keep open. Current main still has the reported canvas-missing PDF image path: the bundled document-extract plugin catches a failed @napi-rs/canvas import and returns no images, while the agent pdf tool does not pass the callback that would surface that warning. Open PR #75370 already targets this issue with closing syntax, so the right cleanup path is review of that implementation rather than closing or queueing a duplicate repair.

## What This Changes

The issue requests a pdftoppm/Poppler fallback for PDF page image rendering when @napi-rs/canvas is unavailable in the bundled document-extract plugin.

## Best Possible Solution

Review PR #75370 and either land a bounded, documented Poppler fallback with regression coverage or choose a clearer canvas-missing degradation/error policy in the bundled document-extract plugin, PDF tool, docs, and changelog.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: a low/no-text PDF in extraction fallback mode reaches image rendering, a failed @napi-rs/canvas import is caught, and current main returns { text, images: [] }; the agent pdf tool path does not pass the callback that would surface the warning.

## Solution Assessment

Unclear. A pdftoppm fallback is a plausible owner-module fix, but it changes the trusted execution boundary by invoking an external PDF renderer, so the linked PR needs security and maintainer review before that is accepted as the best solution.
