## Summary

Keep open. Current main still has the reported canvas-missing path: the bundled PDF extractor catches the missing optional `@napi-rs/canvas` import and returns text-only extraction with `images: []`, while the agent `pdf` tool call path does not pass the warning callback. The supplied GitHub context shows open PR #75370 already uses closing syntax for this issue, so the right cleanup is to review that PR instead of closing this issue or queueing duplicate repair work.

## What This Changes

Request a `document-extract` PDF image-rendering fallback to Poppler/`pdftoppm` when optional `@napi-rs/canvas` is unavailable, instead of silently degrading to text-only extraction.

## Best Possible Solution

Review PR #75370 and either land a bounded, documented Poppler fallback with regression coverage, or choose a clearer canvas-missing degradation/error policy in the bundled `document-extract` plugin and PDF docs.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: a low/no-text PDF in extraction fallback mode reaches image rendering, a failed `@napi-rs/canvas` import is caught, and the extractor returns `{ text, images: [] }`; the agent `pdf` tool path does not pass the callback that would surface the warning.

## Solution Assessment

Unclear. A `pdftoppm` fallback is a plausible owner-module fix, but it changes the external-process and untrusted-PDF processing boundary, so the linked PR needs security and maintainer review before it is the accepted solution.
