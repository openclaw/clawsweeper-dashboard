## Summary

Keep open. Current main still degrades low-text PDFs to text-only when `@napi-rs/canvas` cannot be imported, there is no Poppler/`pdftoppm` fallback in the bundled `document-extract` plugin, and the linked implementation candidate is still an open draft PR with closing syntax rather than a merged fix.

## What This Changes

The issue requests a PDF page image fallback renderer, such as Poppler `pdftoppm`, when `@napi-rs/canvas` is unavailable so vision-model PDF analysis does not silently degrade to text-only.

## Best Possible Solution

Keep the issue open and resolve it in the bundled `document-extract` plugin boundary: either add a bounded, documented Poppler fallback or surface a clear degradation/error path, then update `docs/tools/pdf.md` and focused regression tests for the chosen dependency contract.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: a low-text PDF reaches page rendering, `loadCanvasModule()` throws when `@napi-rs/canvas` is unavailable, and the extractor returns an empty image list; the agent PDF tool path also omits the warning callback.

## Solution Assessment

Unclear. The requested `pdftoppm` fallback is plausible and localized, but it changes the external-process and optional-runtime dependency contract, so PR #75370 should be reviewed for security and product policy before this is considered solved.
