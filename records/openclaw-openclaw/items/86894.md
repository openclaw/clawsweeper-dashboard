## Summary

Keep open. Current main and v2026.6.9 still render assistant MEDIA audio as manual native controls with no autoplay attribute, opt-in playback setting, or autoplay directive; related threads cover inline rendering, delivery, or manual Read aloud rather than this autoplay product decision.

## What This Changes

Request an opt-in Control UI/WebChat mechanism to auto-play assistant MEDIA audio attachments, with possible config, directive, or permission-gate designs.

## Best Possible Solution

Add a default-off Control UI assistant-audio autoplay design that covers MEDIA and TTS attachments, reuses the assistant-media ticket route, avoids retired WebChat config unless explicitly approved, and documents browser fallback behavior.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection gives a high-confidence current-main check: MEDIA audio is rendered, but only as manual controls with metadata preload and no opt-in autoplay state.

## Solution Assessment

Unclear until maintainers choose the product surface. A browser-local consent setting or explicit directive appears safer than reviving `channels.webchat` config or changing playback by default.
