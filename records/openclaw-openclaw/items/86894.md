## Summary

Keep open. Current main and v2026.6.10 still render assistant MEDIA audio as manual native controls with no autoplay attribute, opt-in playback setting, or autoplay directive; related items cover inline rendering, delivery, provider-backed voice, or manual Read aloud rather than this autoplay product decision.

## What This Changes

The issue requests an opt-in mechanism for Control UI/WebChat to auto-play assistant audio attachments delivered by `MEDIA:` directives.

## Best Possible Solution

Add a default-off Control UI assistant-audio autoplay design that covers MEDIA and TTS attachments, reuses the assistant-media ticket route, avoids retired WebChat config unless explicitly approved, and documents browser fallback behavior.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request; yes as a source-level verification. Current main renders MEDIA audio as manual controls and has no opt-in autoplay setting or directive.

## Solution Assessment

Unclear until maintainers choose the product surface. A browser-local consent setting or explicit per-message directive appears safer than reviving retired `channels.webchat` config or changing playback by default.
