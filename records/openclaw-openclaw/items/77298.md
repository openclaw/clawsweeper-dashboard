## Summary

Keep open: current main and v2026.6.8 still mark gateway-restart-interrupted cron runs as ordinary errors and increment `consecutiveErrors`, and the recent production update confirms this still affects operator monitoring. The closed retry PR is not a viable canonical fix, while the overlapping open alert-fatigue issue needs a coherent maintainer decision rather than a separate counter-only patch.

## What This Changes

Request that gateway-restart-interrupted cron runs stop inflating the ordinary `consecutiveErrors` agent-failure counter and become distinguishable in cron state.

## Best Possible Solution

Define one canonical cron restart-interruption outcome that preserves real agent/model failures in `consecutiveErrors` while aligning retry backoff, failure alerts, run history, and list/filter state with the overlapping notification work.

## Reproduction Assessment

Yes. A current-main cron job with a persisted `runningAtMs` marker will go through `start()` and `markInterruptedStartupRun`, which sets `error` status and increments `consecutiveErrors`; I did not run a live gateway restart in this read-only review.

## Solution Assessment

Unclear. Distinguishing restart interruptions from real agent failures is the right direction, but a reset flag alone would not fix automatic backoff or alert semantics, and a new status/counter needs maintainer-approved persisted-state behavior.
