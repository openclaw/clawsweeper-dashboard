## Summary

Keep open because current main still counts gateway-restart interruptions as cron execution errors, and the remaining fix needs maintainer choice on public cron interruption/status semantics.

## What This Changes

The issue requests that gateway-restart interruptions stay visible without inflating cron consecutive execution-error counters, backoff, or agent-failure monitoring.

## Best Possible Solution

Preserve the visible restart-interruption record while separating operator restarts from execution-failure counters/backoff and updating tests/docs for the chosen cron state contract.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: a persisted cron job with `state.runningAtMs` is marked as a gateway-restart error during `start()`, and `markInterruptedStartupRun()` increments `consecutiveErrors`.

## Solution Assessment

Unclear. The previous fix correctly stopped silent success/replay, but the best follow-up depends on whether maintainers want restart interruptions to remain `error`, become `skipped`, or gain a new public `interrupted` state/counter.
