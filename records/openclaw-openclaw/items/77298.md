## Summary

Keep open: current main still records gateway-restart-interrupted cron runs as errors and increments `consecutiveErrors`, so the reported monitoring/backoff inflation remains source-reproducible. The related retry PR is open but unmerged and only covers isolated agent-turn restart retries.

## What This Changes

The issue asks cron restart interruptions to stop inflating `consecutiveErrors` and to be distinguishable from real agent execution failures.

## Best Possible Solution

Keep the visible restart-interruption record, but prevent normal Gateway restart or hot-reload interruptions from inflating execution-failure counters/backoff, and document the chosen cron state semantics.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: persist a cron job with `state.runningAtMs`, start the cron service, and `markInterruptedStartupRun()` records a gateway-restart error while incrementing `consecutiveErrors`.

## Solution Assessment

Unclear. A counter-only fix could be narrow, but the current public status surface has only `ok`, `error`, and `skipped`, and the open retry PR chooses a different isolated-agent path, so maintainers should settle the state contract first.
