## Summary

Keep open. Current main still records gateway-restart-interrupted cron jobs as `error` and increments `consecutiveErrors`, matching the report and the existing ClawSweeper comment; the remaining decision is the intended public contract for restart interruptions versus execution failures.

## What This Changes

The issue requests that gateway-restart-interrupted cron runs stop inflating the execution-error counter, either through a distinct interruption status/reason or a separate/resettable counter.

## Best Possible Solution

Preserve the visible restart-interruption record while separating operator restarts from execution-failure counters/backoff, then document and test the chosen cron state semantics.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: a persisted cron job with `state.runningAtMs` is recovered during `start()`, marked as a gateway-restart error, and has `consecutiveErrors` incremented.

## Solution Assessment

Unclear. Excluding restart interruptions from execution-failure counters is likely the narrow fix, but maintainers need to choose whether the public status remains `error`, reuses `skipped`, or grows a new `interrupted` state.
