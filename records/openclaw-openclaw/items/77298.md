## Summary

The issue still represents a valid remaining bug: current main still records restart-interrupted cron executions as `status: "error"` with `lastError`/`lastDeliveryError`, and the error-status path increments `state.consecutiveErrors`, so restart churn can still inflate the agent-failure counter. The open related pull request https://github.com/openclaw/openclaw/pull/75510 is a plausible implementation path for part of the restart-interruption problem, but it is not merged and does not justify closing this issue yet.

## What This Changes

The issue asks for gateway-restart cron interruptions to be tracked separately from real agent failures so `consecutiveErrors` remains a useful failure-health signal.

## Best Possible Solution

Land a cron-state fix that records gateway-restart interruptions as a distinct interruption/retry outcome or otherwise excludes them from `consecutiveErrors`, with regression coverage for both restart-interrupted runs and real agent-turn failures.

## Reproduction Assessment

Yes from source inspection: restart catch-up can mark a run as `error`, and the shared completion path increments `consecutiveErrors` for every error. I did not run a live gateway restart, so this is source-reproducible rather than locally reproduced.

## Solution Assessment

Unclear whether the reporter's exact `interrupted` status is the best API, because the open related PR retries some interrupted isolated jobs instead. The maintainable fix should make restart interruptions distinct from agent failures in the canonical cron state path without adding an operator reset flag as the primary remedy.
