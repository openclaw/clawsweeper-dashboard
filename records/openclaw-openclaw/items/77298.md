## Summary

This should stay open: current main and the latest release still mark gateway-restart-interrupted cron runs as ordinary errors and increment `consecutiveErrors`, so the reported monitoring drift remains source-reproducible. The related retry PR is still open and has its own proof/merge-readiness blockers, so it is not a safe reason to close this issue.

## What This Changes

The issue asks for gateway-restart cron interruptions to be distinguished from real agent failures and excluded from the `consecutiveErrors` agent-failure counter.

## Best Possible Solution

Define a canonical restart-interruption outcome for cron state that keeps real agent/model failures in `consecutiveErrors` while recording restart interruptions separately or retrying them without polluting the failure-rate counter.

## Reproduction Assessment

Yes, from source inspection: startup recovery writes restart-interrupted runs as `error` and increments `consecutiveErrors`. I did not run a live gateway restart, so this is source-reproducible rather than locally reproduced.

## Solution Assessment

Unclear: excluding restart interruptions from `consecutiveErrors` is the right product direction, but the exact durable state shape needs maintainer choice because current code also ties those errors to retries, alerts, run logs, and the related retry PR.
