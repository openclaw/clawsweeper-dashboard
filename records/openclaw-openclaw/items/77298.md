## Label Justifications

- `P2`: This is a normal-priority cron state correctness issue with real operator monitoring impact but no evidence of data loss, security bypass, crash loop, or unusable core runtime.
- `impact:session-state`: The issue is about persisted cron run status and counters drifting from the actual agent-failure history after restart interruptions.

## Summary

Keep open: current main and v2026.6.10 still treat gateway-restart-interrupted cron runs as ordinary `error` outcomes and increment `consecutiveErrors`; the related notification issue overlaps but does not replace this counter/state-specific report.

## What This Changes

The issue asks for gateway-restart-interrupted cron runs to be distinguished from real agent failures so `consecutiveErrors` remains a useful health signal.

## Best Possible Solution

Define one canonical cron restart-interruption outcome, then align persisted state, counters, retry/backoff, run logs, Gateway schema, CLI filters, failure notifications, and docs to that policy.

## Reproduction Assessment

Yes, at source level. A cron job restored with `state.runningAtMs` on current main goes through `start()` and `markInterruptedStartupRun`, which marks it `error` and increments `consecutiveErrors`; I did not run a live gateway restart.

## Solution Assessment

Unclear until maintainers choose the cron semantics. Distinguishing restart interruptions from real agent failures is the right direction, but a reset flag alone would not fix backoff, alerting, run-history, or protocol consistency.
