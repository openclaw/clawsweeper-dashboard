## Summary

Keep open: current main and v2026.6.8 still classify persisted in-flight gateway-restart interruptions as ordinary cron errors and increment `consecutiveErrors`. The closed retry PR was not merged, and the broader open alert issue does not replace this counter-specific report.

## What This Changes

The issue requests cron gateway-restart interruptions be distinguished from real cron failures so `consecutiveErrors` remains useful for agent-failure monitoring.

## Best Possible Solution

Define one canonical cron restart-interruption outcome that preserves real agent/model failures in `consecutiveErrors` while aligning retry backoff, failure alerts, run history, and list/filter state with the overlapping notification work.

## Reproduction Assessment

Yes. A current-main cron job with a persisted `runningAtMs` marker goes through `start()` and `markInterruptedStartupRun`, which sets `error` status and increments `consecutiveErrors`; I did not run a live gateway restart because this review is read-only.

## Solution Assessment

Unclear. Distinguishing restart interruptions from real agent failures is the right direction, but a reset flag alone would not fix automatic backoff or alert semantics, and a new status/counter needs maintainer-approved persisted-state behavior.
