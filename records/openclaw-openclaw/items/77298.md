## Summary

Keep open because current main still increments cron execution-error/backoff state for persisted runs interrupted by gateway restart; the remaining decision is the public cron-state semantics for operator interruptions versus agent failures.

## What This Changes

The issue asks cron restart interruptions to be separated from agent execution failures so routine gateway restarts do not inflate `consecutiveErrors`.

## Best Possible Solution

Preserve durable visible restart-interruption records while separating operator restarts from agent-failure counters, backoff, and alerts, then update cron protocol/docs/tests for the chosen semantics.

## Reproduction Assessment

Yes. Source inspection on current main shows `start()` calls `markInterruptedStartupRun()` for persisted `state.runningAtMs`, and that helper writes `lastRunStatus: "error"` plus `consecutiveErrors = previousErrors + 1`.

## Solution Assessment

Unclear. The previous fix correctly stopped silent success/replay, but the narrow follow-up depends on whether maintainers want restart interruptions to remain `error`, become `skipped`, or gain a public `interrupted` state or reason.
