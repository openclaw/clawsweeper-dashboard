## Summary

Keep open: current main still marks persisted cron runs interrupted by gateway restart as `error` and increments `consecutiveErrors`, so the reported metric inflation remains source-reproducible. An open related PR addresses the isolated agent-job retry path, but it is not merged and still needs maintainer/proof follow-through.

## What This Changes

The issue reports that restart-induced cron interruptions are counted as consecutive execution errors after the earlier restart fix, obscuring real agent-failure metrics.

## Best Possible Solution

Keep the visible restart-interruption record, but route operator restarts through a retry or interruption path that does not inflate execution-failure counters/backoff, and document the chosen cron state semantics.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: persist a cron job with `state.runningAtMs`, start the cron service, and `markInterruptedStartupRun()` records a gateway-restart error while incrementing `consecutiveErrors`.

## Solution Assessment

Unclear. The open isolated-agent retry PR is a plausible narrow direction for the reported job shape, but maintainers still need to decide whether restart interruptions should remain `error`, become `skipped`, be retried, or gain a new public status.
