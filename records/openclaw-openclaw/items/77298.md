## Summary

Keep open because current `main` still increments cron execution-error state for gateway-restart interruptions, and the remaining fix requires a maintained public semantics choice for interruption versus agent failure.

## What This Changes

The issue asks for gateway-restart-interrupted cron runs to stop inflating `consecutiveErrors` and to be distinguishable from real agent-turn failures.

## Best Possible Solution

Preserve visible restart-interruption records while separating operator restart interruptions from agent execution-failure counters, backoff, and alerts, then document the chosen cron state semantics.

## Reproduction Assessment

Yes. Source inspection on current `main` gives a high-confidence path: a persisted cron job with `state.runningAtMs` is marked as a gateway-restart error and has `consecutiveErrors` incremented during `start()`.

## Solution Assessment

Unclear. The complaint is source-backed, but the narrowest maintainable fix depends on maintainer choice between preserving `error` with counter/backoff changes, reusing `skipped`, or adding a new public interruption state.
