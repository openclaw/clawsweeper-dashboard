## Summary

Keep open: current main and the latest release still classify gateway-restart-interrupted cron runs as ordinary `error` outcomes and increment `consecutiveErrors`, while the production update confirms this remains operator-visible. The related alert-fatigue issue overlaps but does not replace this counter/state-specific report.

## What This Changes

The issue asks for cron gateway-restart interruptions to be distinguished from real agent failures so `consecutiveErrors` remains useful for monitoring.

## Best Possible Solution

Define one canonical cron restart-interruption outcome, then align persisted state, counters, retry/backoff, run logs, failure notifications, Gateway schema, CLI filters, and docs to that policy.

## Reproduction Assessment

Yes, at source level. A cron job restored with `state.runningAtMs` on current main goes through `start()` and `markInterruptedStartupRun`, which marks it `error` and increments `consecutiveErrors`; I did not run a live gateway restart.

## Solution Assessment

Unclear until maintainers choose the cron semantics. Distinguishing restart interruptions from real agent failures is the right direction, but a reset flag alone would not fix backoff, alerting, run-history, or protocol consistency.
