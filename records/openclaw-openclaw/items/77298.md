## Summary

Keep open: current main still treats startup-detected cron gateway interruptions as execution errors and increments `consecutiveErrors`, but the durable fix needs a maintainer decision on public cron status/counter semantics.

## What This Changes

Request to separate gateway-restart cron interruptions from agent execution failures so routine restarts do not inflate `consecutiveErrors` or related backoff/alert metrics.

## Best Possible Solution

Preserve a visible record for restart-interrupted runs while separating operator restart interruptions from agent execution-failure counters, backoff, and alerting, with docs/protocol updates for whichever public semantics maintainers choose.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: a persisted cron job with `state.runningAtMs` is processed during `start()`, marked as a gateway-restart error, and has `consecutiveErrors` incremented.

## Solution Assessment

Unclear. A narrow code change could exclude restart interruptions from `consecutiveErrors`, but adding an `interrupted` status or separate interruption counter changes public cron semantics and should be decided first.
