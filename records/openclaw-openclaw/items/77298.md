## Summary

Keep open because current main still increments cron execution-error state for gateway-restart interruptions, and the remaining question is the intended public semantics for interruption vs agent failure counters.

## What This Changes

The issue requests that cron jobs interrupted by gateway restarts be distinguished from agent-turn failures and stop inflating `consecutiveErrors`.

## Best Possible Solution

Preserve the visible restart-interruption record while separating operator restarts from agent execution-failure counters/backoff and documenting the chosen cron state semantics.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: a persisted cron job with `state.runningAtMs` is marked as a gateway-restart error and has `consecutiveErrors` incremented during `start()`.

## Solution Assessment

Unclear. The previous fix correctly stopped silent success/replay, but the best follow-up depends on whether maintainers want restart interruptions to remain `error`, become `skipped`, or gain a new public `interrupted` state.
