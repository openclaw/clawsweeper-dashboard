## Summary

Keep open: current main still has the reported accounting path, where startup recovery marks gateway-restart interruptions as errors and increments `consecutiveErrors` even though the documented counter is for execution-error backoff.

## What This Changes

The issue asks cron startup recovery to distinguish gateway-restart interruptions from real agent failures so they do not inflate `consecutiveErrors`.

## Best Possible Solution

Keep the visible failed/run-log state for restart interruptions, but stop treating that startup-recovery marker as an agent execution failure for `consecutiveErrors` and backoff; add focused regression coverage around a persisted `runningAtMs` job with an existing counter.

## Reproduction Assessment

Yes, by source inspection: seed a persisted cron job with `state.runningAtMs` and an existing `consecutiveErrors`, then `CronService.start()` reaches `markInterruptedStartupRun` and increments the counter. I did not run a live gateway reproduction because this review is read-only.

## Solution Assessment

Yes, the narrowest maintainable fix is to adjust startup-interruption accounting while preserving #50414's visible failure/run-log behavior. A new status or reset flag can be considered later if maintainers want a broader API change.
