## Summary

Keep open because current main still increments cron execution-error state for gateway-restart interruptions, while the type/docs contract treats `consecutiveErrors` as execution-error/backoff state rather than an operator-restart counter.

## What This Changes

The issue requests separating gateway-restart cron interruptions from real agent execution failures so `consecutiveErrors` remains useful for monitoring and retry behavior.

## Best Possible Solution

Preserve a visible record that the run was interrupted by gateway restart while keeping operator restarts out of agent execution-failure counters/backoff, then document the chosen cron state semantics.

## Reproduction Assessment

Yes. Source inspection on current main shows `start()` calls `markInterruptedStartupRun()` for persisted `state.runningAtMs`, and that helper marks the run as a gateway-restart error while incrementing `consecutiveErrors`.

## Solution Assessment

Unclear. The fix likely belongs in cron startup recovery, but maintainers should first choose whether restart interruptions remain `error`, map to `skipped`, or get a new public interruption status/reason.
