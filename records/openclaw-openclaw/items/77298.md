## Summary

Keep open because current main still records gateway-restart-interrupted cron runs as `error` and increments the execution-error counter used for backoff, while the remaining fix depends on the intended public semantics for restart interruptions.

## What This Changes

The issue asks OpenClaw to distinguish cron gateway-restart interruptions from real agent execution failures so `consecutiveErrors` does not misrepresent agent health.

## Best Possible Solution

Define a restart-interruption outcome that preserves visible failure history but keeps execution-failure counters and backoff reserved for real run failures, then update cron state/protocol/docs/tests accordingly.

## Reproduction Assessment

Yes. A persisted cron job with `state.runningAtMs` on gateway startup reaches `start()` -> `markInterruptedStartupRun()`, which marks the run as a gateway-restart error and increments `consecutiveErrors`.

## Solution Assessment

Unclear. The post-#50414 behavior correctly stopped silent success/replay, but the narrow follow-up depends on whether restart interruptions should remain `error`, become `skipped`, or gain a new public `interrupted` state.
