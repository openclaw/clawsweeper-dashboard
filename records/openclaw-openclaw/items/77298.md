## Summary

Keep open: current main still routes persisted running cron jobs through the gateway-restart interruption path and increments `consecutiveErrors`, while the related open PR covers a narrower isolated-agent retry path and is not merged.

## What This Changes

The issue requests gateway-restart-interrupted cron runs to be tracked separately from agent execution failures so they do not inflate `consecutiveErrors`.

## Best Possible Solution

Define restart-interruption semantics once, then keep interruption visibility while preventing normal gateway restart or hot-reload interruptions from inflating execution-failure counters/backoff; update cron protocol, docs, and tests if a new status or counter is chosen.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: persist a cron job with `state.runningAtMs`, start the cron service, and `markInterruptedStartupRun()` records the gateway-restart error while incrementing `consecutiveErrors`.

## Solution Assessment

Unclear. A counter-only fix could be narrow, but adding a distinct interruption status or counter would touch the public cron contract, so maintainers should settle the state semantics before repair automation.
