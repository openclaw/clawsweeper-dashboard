## Summary

Keep open: current main still increments the cron execution-error counter for startup-detected gateway-restart interruptions, and the remaining fix depends on the intended public cron state semantics.

## What This Changes

The issue asks for gateway-restart-interrupted cron runs to stop polluting real agent-failure counters or to gain a distinct interruption state/counter.

## Best Possible Solution

Preserve the visible restart-interruption record while separating operator restarts from execution-error counters/backoff/failure-alert thresholds, then document the chosen cron state semantics.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a persisted cron job with `state.runningAtMs` is marked as a gateway-restart error during `start()` and has `consecutiveErrors` incremented.

## Solution Assessment

Unclear. Avoiding the counter increment is narrow, but the public status and alert/backoff semantics need maintainer choice before changing the cron state contract.
