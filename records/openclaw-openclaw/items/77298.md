## Summary

Keep open. Current `main` still marks gateway-restart-interrupted cron jobs as `error` and increments `consecutiveErrors`, while the remaining decision is how OpenClaw should publicly represent restart interruptions without polluting execution-failure counters/backoff after #50414.

## What This Changes

The issue asks for gateway-restart cron interruptions to be distinguishable from agent failures and not inflate `consecutiveErrors`.

## Best Possible Solution

Define cron restart-interruption semantics, then update state, counters/backoff, tests, and docs so operator restarts remain visible without inflating execution-failure metrics.

## Reproduction Assessment

Yes. Source inspection on current `main` gives a high-confidence path: a persisted cron job with `state.runningAtMs` is processed during `start()`, marked as a gateway-restart error, and has `consecutiveErrors` incremented.

## Solution Assessment

Unclear. Excluding restart interruptions from execution-error counters appears necessary, but the maintainable fix depends on whether maintainers want to keep `error`, reuse `skipped`, or introduce an `interrupted` status/reason.
