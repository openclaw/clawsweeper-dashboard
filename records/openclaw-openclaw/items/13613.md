## Summary

Keep open: current main still implements cron as Gateway-local, file-backed scheduling with process-local locking and status, while this asks for HA coordination across separate Gateway processes. The broader horizontal-scaling proposal at https://github.com/openclaw/openclaw/issues/25883 is still open architecture input, not a merged replacement.

## What This Changes

Request distributed cron job coordination across multiple OpenClaw gateways, with leader election or distributed locks plus new coordination config and CLI/status surfaces.

## Best Possible Solution

Define a supported distributed cron ownership model, then implement it with a versioned config or plugin API boundary, health/status visibility, CLI inspection, docs, and tests that simulate separate Gateway processes sharing scheduled jobs.

## Reproduction Assessment

Not applicable; this is an enhancement request rather than a reproducible defect. Source and docs inspection gives high confidence that current main lacks distributed cron ownership across Gateway processes.

## Solution Assessment

Unclear until maintainers choose the architecture. The requested Redis/etcd/DynamoDB directions are plausible, but the narrow maintainable path should first settle ownership semantics, failure behavior, config/API boundaries, and how this relates to the broader horizontal-scaling work.
