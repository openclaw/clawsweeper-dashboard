## Summary

Keep open. Current main still implements cron as a Gateway-local, file-backed scheduler with process-local locking and no accepted coordination config, leader/lease status, shared backend, or multi-process ownership contract; the related horizontal-scaling proposal remains open and does not supersede this cron-specific request.

## What This Changes

Request to add distributed coordination or leader election for cron jobs so multiple Gateway instances do not execute the same scheduled job concurrently.

## Best Possible Solution

Define a supported distributed cron ownership model, likely through a generic core or plugin lease boundary, then add config/schema/docs/status/CLI support and multi-process regression coverage before recommending HA cron deployments.

## Reproduction Assessment

Not applicable; this is a feature request rather than a broken existing behavior report. Source and docs inspection gives high confidence that current main lacks distributed cron coordination today.

## Solution Assessment

Unclear until maintainers choose the architecture. The Redis/leader-election examples are plausible, but the maintainable path should define a generic distributed ownership contract before committing to one backend shape.
