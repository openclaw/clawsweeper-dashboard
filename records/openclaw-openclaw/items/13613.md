## Summary

Keep open. Current main and the latest v2026.5.19 tag still implement cron as Gateway-local scheduling with local state, process-local locking, and local status/CLI surfaces; related BullMQ/minions attempts were closed or routed to plugin/seam work rather than merged as distributed cron coordination.

## What This Changes

The issue requests distributed coordination for cron jobs across multiple OpenClaw Gateway instances so HA deployments do not execute the same scheduled work more than once.

## Best Possible Solution

Define a supported distributed cron ownership model, likely through a generic core or plugin lease boundary, then add config/schema/docs/status/CLI support and multi-process regression coverage before recommending HA cron deployments.

## Reproduction Assessment

Not applicable; this is a feature request rather than broken behavior against an existing documented contract. Source, docs, release-tag, and related-item inspection give high confidence that current OpenClaw lacks distributed cron coordination.

## Solution Assessment

Unclear until maintainers choose the architecture. The Redis and leader-election examples are plausible, but the maintainable path should define a generic distributed ownership contract, failure semantics, and health/CLI surface before committing to one backend shape.
