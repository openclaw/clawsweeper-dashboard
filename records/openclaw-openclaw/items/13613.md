## Summary

Keep open. Current main still implements cron as a Gateway-local, file-backed scheduler with process-local locking and no distributed lease, leader election, shared coordination backend, HA health/status, or cron-leader CLI surface.

## What This Changes

The issue requests distributed coordination for cron jobs so multiple Gateway instances in HA deployments do not execute the same scheduled jobs independently.

## Best Possible Solution

Define a supported distributed cron ownership model, then implement it behind a versioned config or plugin/API boundary with health/status visibility, CLI inspection, docs, and cross-process tests.

## Reproduction Assessment

Yes for the observable HA gap. Source inspection shows independent Gateway processes have no shared lease or leader check, while the only duplicate-run guard is process-local and the current duplicate test stays inside one Node process.

## Solution Assessment

Unclear. Distributed locks or leader election are plausible, but maintainers first need to choose the supported ownership model, backend/API surface, failure behavior, health reporting, CLI surface, and relationship to https://github.com/openclaw/openclaw/issues/25883.
