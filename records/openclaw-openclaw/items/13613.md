## Summary

Keep open. Current main and v2026.5.22 still expose cron as Gateway-local scheduling with local files/state, local CLI/status, and process-local locking; the requested multi-Gateway cron ownership model remains missing and needs maintainer architecture/product direction.

## What This Changes

The issue requests distributed coordination for cron jobs so multiple Gateway instances in HA deployments do not execute the same scheduled work concurrently.

## Best Possible Solution

Define a supported distributed cron ownership model behind a generic or plugin-capable lease boundary, then add config/schema/docs/status/CLI support and multi-process Gateway regression coverage.

## Reproduction Assessment

Not applicable: this is a feature and architecture request rather than a failure against an existing documented contract. Source, docs, release-tag, and live GitHub checks give high confidence that OpenClaw lacks distributed cron coordination.

## Solution Assessment

Unclear until maintainers choose the architecture. Redis-style locks and leader election are plausible examples, but the maintainable path should first settle ownership semantics, backend/API boundaries, failure behavior, health/status visibility, and security posture.
