## Summary

Keep open. Current main and v2026.5.20 still implement cron as Gateway-local scheduling with local files, process-local locking, and local status/CLI surfaces; https://github.com/openclaw/openclaw/issues/25883 remains related architecture context rather than a merged cron HA replacement.

## What This Changes

Request distributed locking or leader election for cron jobs so multiple HA Gateway instances do not run the same scheduled job concurrently.

## Best Possible Solution

Define a supported distributed cron ownership model, likely through a generic core or plugin-capable lease boundary, then add config/schema/docs/status/CLI support and multi-process regression coverage before recommending HA cron deployments.

## Reproduction Assessment

Not applicable; this is a feature and architecture request rather than broken behavior against an existing documented contract. Source, docs, latest-release, and related-item inspection give high confidence that OpenClaw lacks distributed cron coordination.

## Solution Assessment

Unclear until maintainers choose the architecture. Redis locks and leader election are plausible examples, but the maintainable path should define a generic ownership contract, failure semantics, config boundary, and health/CLI surface before committing to one backend shape.
