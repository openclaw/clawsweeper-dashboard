## Summary

Keep open. Current main still implements cron as Gateway-local scheduling with local files, process-local locking, local status/CLI surfaces, and no distributed lease, leader election, or shared coordination backend; the broader horizontal-scaling proposal remains open architecture input rather than a shipped replacement.

## What This Changes

Requested distributed coordination for cron jobs across multiple OpenClaw Gateway instances in high-availability deployments.

## Best Possible Solution

Define a supported distributed cron ownership model, then implement it with versioned config or plugin API boundaries, health/status visibility, CLI inspection, docs, and multi-process regression coverage.

## Reproduction Assessment

Not applicable; this is a feature request rather than a broken existing behavior report. Source and docs inspection give high confidence that current main lacks distributed cron coordination today.

## Solution Assessment

Unclear until maintainers choose the architecture. The issue's Redis/leader-election examples are plausible, but the maintainable path should define a generic distributed ownership contract instead of hardcoding one backend shape prematurely.
