## Label Justifications

- `P2`: Duplicate scheduled execution in HA deployments is a production reliability gap, but it is a broad enhancement requiring architecture decisions rather than an urgent confirmed regression.
- `impact:message-loss`: The issue describes duplicate Slack, Discord, and other channel deliveries when multiple Gateways execute the same scheduled slot.

## Summary

Keep open: current main and v2026.6.10 still lack cross-Gateway cron ownership, leader election, distributed leases, coordination config, or HA cron status, so the request remains a valid architecture/product follow-up rather than cleanup-close material.

## What This Changes

The issue requests distributed coordination for cron jobs so multiple OpenClaw gateway instances do not run the same scheduled job independently.

## Best Possible Solution

Define a supported distributed cron ownership contract first, then implement backend-neutral fenced claiming or leader leasing with upgrade-safe config or plugin API, health/CLI visibility, docs, and separate-process tests.

## Reproduction Assessment

Yes, from source inspection with high confidence: independent Gateway processes still have no shared cron lease, leader check, or coordination config/API. I did not run a live multi-host reproduction in this read-only review.

## Solution Assessment

No, not as the backend-specific proposal alone. Redis, etcd, and DynamoDB are plausible backends, but maintainers first need to choose the supported ownership boundary, failure semantics, and whether this belongs in core or a plugin/API seam.
