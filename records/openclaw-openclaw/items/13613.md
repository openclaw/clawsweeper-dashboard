## Summary

Keep open. Current main still implements cron as Gateway-local scheduling with local files, process-local locking, local status, and no distributed lease, leader election, or shared coordination backend; the related horizontal-scaling proposal remains open architecture input, not a shipped replacement.

## What This Changes

The issue requests distributed coordination or locking for cron jobs across multiple Gateway instances in HA deployments.

## Best Possible Solution

Define a supported distributed cron ownership model, then implement it with versioned config or plugin API boundaries, health/status visibility, CLI inspection, docs, and multi-process regression coverage.

## Reproduction Assessment

Not applicable; this is a feature request rather than a broken existing behavior report. Source and docs inspection give high confidence that current main lacks distributed cron coordination today.

## Solution Assessment

Unclear until maintainers choose the architecture. The maintainable path is likely a generic distributed scheduler ownership contract, possibly plugin-capable, rather than committing directly to one hardcoded Redis-only shape from the issue body.
