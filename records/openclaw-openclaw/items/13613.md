## Label Justifications

- `P2`: Duplicate scheduled execution in HA deployments is a real production reliability gap, but it is a broad feature requiring architecture decisions rather than an urgent confirmed regression.
- `impact:message-loss`: The issue describes duplicate Slack, Discord, and other channel deliveries when multiple Gateways execute the same scheduled slot.

## Summary

Keep open: current main and the latest release still lack cross-Gateway cron ownership, leader election, distributed leases, or a coordination backend. The broader horizontal-scaling and distributed-runtime issues overlap, but they are still open architecture discussions rather than merged replacements for this narrower cron HA request.

## What This Changes

Request adding distributed cron ownership or locking so multiple Gateway instances do not execute the same scheduled job in HA deployments.

## Best Possible Solution

Define a supported cron ownership contract first, then implement backend-neutral fenced job claiming or leader leasing with upgrade-safe config or plugin API, health/CLI visibility, docs, and separate-process tests.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: independent Gateway processes have no shared cron lease, leader check, or coordination backend, and current locks are explicitly process-local. I did not run a live multi-host HA setup in this read-only review.

## Solution Assessment

No, not as the backend-specific proposal alone. Redis, etcd, and DynamoDB are plausible building blocks, but maintainers first need to choose the supported ownership boundary, failure semantics, security posture, and whether this belongs in core or a plugin/API seam.
