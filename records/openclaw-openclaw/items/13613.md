## Summary

Keep open. Current main still implements cron as Gateway-local scheduling with local config/state files, process-local locking, and local status/CLI surfaces; it has no distributed lease, leader election, shared coordination backend, or HA ownership visibility.

## What This Changes

The issue requests distributed coordination for cron jobs so multiple Gateway instances in HA deployments do not execute the same scheduled work concurrently.

## Best Possible Solution

Define a supported distributed cron ownership model, then implement it behind a versioned config or plugin/API boundary with health/status visibility, CLI inspection, docs, and cross-process tests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: run two independent Gateway instances configured with the same due cron job and both can execute it because current ownership is local to each Gateway process and there is no shared lease or leader check.

## Solution Assessment

Unclear. The issue’s Redis/etcd/DynamoDB approaches are plausible, but the narrow maintainable solution needs maintainer decisions on backend support, API shape, failure behavior, observability, and security first.
