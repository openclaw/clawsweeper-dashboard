## Label Justifications

- `P2`: Duplicate scheduled execution is a production reliability gap with meaningful but multi-Gateway-specific blast radius.
- `impact:message-loss`: Independent Gateways can duplicate Slack, Discord, or other channel deliveries for the same scheduled slot.

## Summary

Keep open: current main and v2026.6.6 still lack cross-Gateway cron ownership, and the live discussion/labels show this remains a maintainer architecture decision rather than cleanup that can be closed.

## What This Changes

The issue requests distributed cron coordination, such as leader election or per-job leases, so multiple Gateway instances do not execute the same scheduled job simultaneously.

## Best Possible Solution

Define a supported cron ownership contract first, then add an atomic fenced claim or leader lease at the scheduler boundary with health/status visibility, operator inspection, upgrade-safe configuration, and separate-process tests.

## Reproduction Assessment

Yes, from source with high confidence: independent Gateway processes have no shared lease or leader check, and the current lock is explicitly process-local. No live multi-host reproduction was run in this read-only review.

## Solution Assessment

No, not as directly proposed yet. Redis, etcd, and DynamoDB are plausible backends, but maintainers first need to choose the supported topology and whether coordination belongs in core SQLite, a plugin API, or the broader horizontal-scaling design.
