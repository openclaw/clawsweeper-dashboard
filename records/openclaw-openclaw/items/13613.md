## Summary

Keep open. Current main and v2026.5.20 still treat cron as Gateway-local scheduling with local files, process-local serialization, and local status/CLI surfaces; distributed leases, leader election, shared coordination backends, and HA cron status remain unimplemented and need maintainer architecture direction.

## What This Changes

The issue requests distributed coordination for cron jobs across multiple Gateway instances in HA, load-balanced, blue/green, or multi-region deployments.

## Best Possible Solution

Define a supported distributed cron ownership model, likely behind a generic or plugin-capable lease boundary, then add config/schema/docs/status/CLI support and multi-process Gateway regression coverage.

## Reproduction Assessment

Not applicable; this is a feature and architecture request rather than a failure against an existing documented contract. Source, docs, latest-release, and related-item inspection give high confidence that OpenClaw lacks distributed cron coordination.

## Solution Assessment

Unclear until maintainers choose the architecture. Redis locks and leader election are plausible examples, but the maintainable path should first define ownership semantics, backend/API boundaries, failure behavior, health/status visibility, and security posture.
