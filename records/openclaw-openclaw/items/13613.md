## Label Justifications

- `P2`: Duplicate cron execution in HA deployments is a real reliability gap with meaningful but topology-specific production impact.
- `impact:message-loss`: The issue describes duplicate Slack, Discord, or other channel deliveries when multiple Gateways execute the same scheduled slot.

## Summary

Keep open: current main still lacks cross-Gateway cron ownership, leader election, distributed leases, or a coordination backend, and the live related work remains architecture input rather than a merged replacement.

## What This Changes

The issue requests distributed coordination for cron jobs so HA or multi-Gateway deployments do not execute the same scheduled job multiple times.

## Best Possible Solution

Define a supported cron ownership contract first, then implement atomic fenced job claiming or leader leasing at the scheduler boundary with health/status visibility, upgrade-safe config or plugin API, docs, and separate-process tests.

## Reproduction Assessment

Yes, from source with high confidence: independent Gateway processes have no shared cron lease or leader check, and current cron locking is explicitly process-local. I did not run a live multi-host reproduction in this read-only review.

## Solution Assessment

No, not as the backend-specific proposal alone. Redis, etcd, and DynamoDB are plausible backends, but maintainers first need to choose the supported ownership boundary and failure semantics.
