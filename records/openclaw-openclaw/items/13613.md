## Label Justifications

- `P2`: This is a material HA reliability gap for multi-Gateway operators, but the standard single-Gateway scheduler remains usable.
- `impact:message-loss`: Independent Gateways can execute the same scheduled slot and duplicate Slack, Discord, or other channel deliveries.

## Summary

This remains a valid, distinct feature gap: both current main and v2026.6.5 coordinate cron work only within one Gateway process, while the broader horizontal-scaling proposal remains open and does not yet own a settled implementation. Keep this issue open for maintainer architecture and product direction.

## What This Changes

Add cross-Gateway cron ownership so high-availability deployments execute each scheduled job only once.

## Best Possible Solution

Define a backend-neutral atomic scheduled-slot claim at the canonical persistence boundary, including fencing or equivalent stale-owner protection, crash recovery, status/health visibility, and separate-process tests; add an external backend only if the supported multi-host topology cannot use that contract.

## Reproduction Assessment

Yes—source inspection gives a high-confidence path: run separate Gateway processes against the same logical cron store, where each has an independent timer and the only serialization helper is explicitly process-local. A real multi-process execution was not run during this read-only review.

## Solution Assessment

No—the proposed provider-specific coordination configuration is not yet the best maintainable solution. Maintainers should first select a backend-neutral ownership, fencing, and failure contract that fits OpenClaw's SQLite-first architecture and supported HA topology.

## Risks / Open Questions

- An incomplete lease design could exchange duplicate external side effects for missed jobs after crashes, pauses, or network partitions.
- Lease expiry, fencing, clock behavior, takeover, and idempotency must be designed together; a runningAtMs timestamp alone is not a safe distributed ownership token.
- Adding Redis-, etcd-, or DynamoDB-specific settings directly to core would expand compatibility-sensitive configuration before the supported HA deployment boundary is settled.
- The design must be coordinated with https://github.com/openclaw/openclaw/issues/25883 so cron does not create a separate competing orchestration architecture.
