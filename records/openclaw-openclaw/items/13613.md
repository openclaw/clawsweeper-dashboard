## Summary

Keep open. Current main and the latest release still implement cron as a Gateway-local scheduler with local state, process-local locking, local status/CLI surfaces, and no distributed leader/lease/backend contract; related horizontal-scaling work remains open rather than a merged replacement.

## What This Changes

The issue requests distributed coordination for OpenClaw cron jobs so multi-gateway HA deployments do not run the same scheduled job multiple times.

## Best Possible Solution

Define a supported distributed cron ownership model, likely through a generic core or plugin lease boundary, then add config/schema/docs/status/CLI support and multi-process regression coverage before recommending HA cron deployments.

## Reproduction Assessment

Not applicable; this is a feature request rather than a broken existing behavior report. Source, docs, release-tag, and related-issue inspection give high confidence that current OpenClaw lacks distributed cron coordination today.

## Solution Assessment

Unclear until maintainers choose the architecture. The Redis and leader-election examples are plausible, but the maintainable path should define a generic distributed ownership contract and failure semantics before committing to one backend shape.
