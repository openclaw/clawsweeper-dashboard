## Summary

Keep open. Current main still implements cron as a Gateway-local, file-backed scheduler with process-local locking and local status/CLI surfaces; it has no distributed lease, leader election, shared coordination backend, or HA ownership visibility. The broader horizontal-scaling proposal in #25883 remains open and does not supersede this cron-specific gap.

## What This Changes

The issue requests distributed coordination for cron jobs across multiple OpenClaw Gateway instances in HA deployments.

## Best Possible Solution

Define a supported distributed cron ownership model for HA deployments, then implement it behind a versioned config or plugin/API boundary with health/status visibility, CLI inspection, docs, and tests that simulate separate Gateway processes sharing scheduled jobs.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: two separate Gateway processes with the same due cron job can both attempt execution because the only current lock is process-local and timer execution has no external lease before marking jobs running.

## Solution Assessment

Unclear. The issue's Redis/etcd/DynamoDB approaches are plausible, but the best maintainable fix needs maintainer decisions on backend support, failure behavior, security model, and the relationship to #25883 before implementation.
