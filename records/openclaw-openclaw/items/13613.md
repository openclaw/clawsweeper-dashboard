## Label Justifications

- `P2`: Duplicate scheduled execution is a production reliability gap with meaningful but multi-Gateway-specific blast radius.
- `impact:message-loss`: The issue describes duplicate Slack, Discord, or other channel deliveries when multiple Gateways execute the same scheduled slot.

## Summary

Keep open: current main and v2026.6.6 still lack cross-Gateway cron ownership, and the live discussion/labels still point to a maintainer architecture decision rather than closeable cleanup.

## What This Changes

Request distributed locking or leader election so cron jobs execute once across multiple OpenClaw Gateways in HA deployments.

## Best Possible Solution

Define a supported cron ownership contract first, then add an atomic fenced claim or leader lease at the scheduler boundary with health/status visibility, operator inspection, upgrade-safe config or plugin API, docs, and separate-process tests.

## Reproduction Assessment

Yes, from source with high confidence: independent Gateway processes have no shared lease or leader check, and the current lock is explicitly process-local. I did not run a live multi-host reproduction in this read-only review.

## Solution Assessment

No, not as directly proposed yet. Redis, etcd, and DynamoDB are plausible backends, but maintainers first need to choose the supported topology and whether coordination belongs in core SQLite, a plugin API, or the broader horizontal-scaling design.
