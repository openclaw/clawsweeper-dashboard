## Summary

Keep open. Current main and the latest v2026.5.19 release still implement cron as Gateway-local scheduling with local files, process-local locking, and local status/CLI surfaces; the related horizontal-scaling proposal remains open architecture context rather than a merged replacement.

## What This Changes

The issue requests distributed coordination for cron jobs so multiple Gateway instances do not independently execute the same scheduled work.

## Best Possible Solution

Define a supported distributed cron ownership model, likely through a generic core or plugin lease boundary, then add config/schema/docs/status/CLI support and multi-process regression coverage before recommending HA cron deployments.

## Reproduction Assessment

Not applicable; this is a feature request rather than broken behavior against an existing documented contract. Source, docs, latest-release, and related-item inspection give high confidence that OpenClaw lacks distributed cron coordination.

## Solution Assessment

Unclear until maintainers choose the architecture. The Redis and leader-election examples are plausible, but the maintainable path should define a generic distributed ownership contract, failure semantics, and health/CLI surface before committing to one backend shape.
