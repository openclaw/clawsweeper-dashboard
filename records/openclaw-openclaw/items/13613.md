## Summary

Keep open. Current main and v2026.5.20 still implement cron as Gateway-local scheduling with local files, process-local locking, and local status/CLI surfaces; the related horizontal-scaling proposal remains open architecture context rather than a merged cron HA replacement.

## What This Changes

Request distributed cron ownership across multiple Gateway instances, such as leader election or per-job distributed locks, with config, health, and CLI visibility.

## Best Possible Solution

Define a supported distributed cron ownership model, likely through a generic or plugin-capable lease boundary, then add config/schema/docs/status/CLI support and multi-process regression coverage before recommending HA cron deployments.

## Reproduction Assessment

Not applicable; this is a feature and architecture request rather than broken behavior against an existing documented contract. Source, docs, latest-release, and related-item inspection give high confidence that OpenClaw lacks distributed cron coordination.

## Solution Assessment

Unclear until maintainers choose the architecture. Redis locks and leader election are plausible examples, but the maintainable path should define ownership semantics, failure behavior, config/API boundaries, health/status visibility, and security posture first.
