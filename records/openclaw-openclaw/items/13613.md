## Label Justifications

- `impact:message-loss`: Multiple active Gateway schedulers can independently send the same scheduled Slack, Discord, or other channel delivery.

## Summary

Keep this issue open. Current main still has only process-local cron serialization and Gateway-local scheduler ownership, while the broader horizontal-scaling proposal remains open and unimplemented; the gap is valid but requires maintainer product and architecture decisions before implementation.

## What This Changes

Add distributed coordination so cron jobs execute once across multiple Gateway instances.

## Best Possible Solution

Approve one canonical distributed cron ownership model aligned with the horizontal-scaling direction, then implement the minimum lease/fencing, health and status, CLI inspection, config and migration, docs, and separate-process integration coverage needed to make its failure behavior explicit.

## Reproduction Assessment

Yes, at source level. `src/cron/service/locked.ts` explicitly limits serialization to one process and each Gateway owns a local CronService, so separate Gateway processes have no shared owner check; this read-only review did not run a live HA deployment.

## Solution Assessment

No, not as currently specified. Once-only distributed execution is the right outcome, but a backend matrix and warning-and-continue policy are premature until maintainers approve the owner boundary, lease and fencing semantics, outage behavior, and relationship to the broader scaling architecture.

## Risks / Open Questions

- The broader horizontal-scaling proposal is still open and substantially wider, so closing this issue as a duplicate would lose the concrete cron once-only execution requirement.
- Adding Redis, etcd, or DynamoDB configuration now would establish a compatibility-sensitive core contract before maintainers choose core versus plugin ownership and supported backend scope.
- Failing open when coordination is unavailable can recreate duplicate execution and delivery, while failing closed can silently stop scheduled automation; that availability policy needs an explicit product decision.
- Lease expiry, long-running jobs, clock behavior, takeover, fencing, and regional partitions can still permit concurrent execution unless the ownership contract and job idempotency expectations are defined together.
