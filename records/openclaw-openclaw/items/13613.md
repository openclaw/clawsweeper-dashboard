## Label Justifications

- `impact:message-loss`: Separate active gateways can independently send the same scheduled Slack, Discord, or other channel delivery.

## Summary

Current main still has only process-local cron serialization, and neither the open horizontal-scaling proposal nor the closed unmerged dedup PR provides a canonical replacement. The request remains necessary, but it requires maintainer product and architecture decisions before implementation.

## What This Changes

Add distributed coordination so cron jobs execute once across multiple gateway instances in high-availability deployments.

## Best Possible Solution

Define one canonical fenced lease/claim contract at the cron scheduler boundary, including backend ownership, takeover semantics, failure policy, health/status and CLI visibility, then validate it with separate gateway processes sharing scheduled work.

## Reproduction Assessment

Yes. Current main explicitly uses process-local serialization and has no shared lease or fencing token in the due-job claim path, so distinct gateway processes are not prevented from executing the same schedule.

## Solution Assessment

No final implementation direction is established. The issue identifies the correct gap, but selecting backend-specific locking before defining ownership, fencing, outage behavior, and the core-versus-plugin boundary would not be the narrowest maintainable solution.

## Risks / Open Questions

- A direct Redis, etcd, or DynamoDB integration would add credentials, availability dependencies, configuration, health behavior, and upgrade obligations without an agreed owner boundary.
- A lock without fencing can allow a stale gateway to finish a job or deliver messages after another gateway has taken ownership.
- Fail-open degradation would knowingly permit duplicate jobs during coordination outages, while fail-closed behavior would stop existing scheduled automation; this requires an explicit product decision.
- The relationship between this cron-specific feature and the broader open horizontal-scaling proposal is unresolved.
