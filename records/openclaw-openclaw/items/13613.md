## Summary

Keep open: current main still implements cron as Gateway-local, file-backed scheduling with process-local locking only, and the requested HA cron coordination remains an unimplemented feature that needs maintainer architecture direction.

## What This Changes

Request distributed coordination for cron jobs across multiple Gateway instances, including leader election or per-job locks plus config, health/status, and CLI visibility.

## Best Possible Solution

Define a supported distributed cron ownership model, then implement it behind a versioned config or plugin/API boundary with health/status visibility, CLI inspection, docs, and cross-process tests.

## Reproduction Assessment

Not applicable as a defect reproduction; this is a feature and architecture request. Source and docs inspection give high confidence that current main lacks distributed cron ownership: cron is Gateway-local, file-backed, and process-locked only.

## Solution Assessment

Unclear until maintainers choose the architecture. Distributed locks or leader election are plausible, but the maintainable path needs a selected ownership model, backend/API surface, failure behavior, health reporting, CLI surface, and coordination with https://github.com/openclaw/openclaw/issues/25883.
