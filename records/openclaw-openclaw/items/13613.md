- `impact:message-loss`: The issue describes duplicate Slack, Discord, or other channel deliveries when multiple Gateways execute the same scheduled slot.

## Summary

Keep open: current main and v2026.6.8 still lack cross-Gateway cron ownership, and the live issue state still points to maintainer architecture/product review rather than closeable cleanup.

## What This Changes

The issue requests distributed coordination for cron execution across multiple OpenClaw Gateway instances in high-availability deployments.

## Best Possible Solution

Define a supported cron ownership contract first, then add an atomic fenced claim or leader lease at the scheduler boundary with health/status visibility, operator inspection, upgrade-safe config or plugin API, docs, and separate-process tests.

## Reproduction Assessment

Yes, from source with high confidence: independent Gateway processes have no shared lease or leader check, and the current lock/active-job tracking is explicitly process-local. I did not run a live multi-host reproduction in this read-only review.

## Solution Assessment

No, not as the backend-specific proposal alone. Redis, etcd, and DynamoDB are plausible backends, but maintainers first need to choose the supported topology and whether coordination belongs in core SQLite, a plugin API, or the broader horizontal-scaling design.
