## Summary

Keep open: current main still has Gateway-local cron scheduling with process-local locking and no distributed ownership, shared coordination backend, leader election, or HA status/CLI surface, so this remains a valid feature gap needing maintainer architecture direction.

## What This Changes

The issue requests distributed coordination for cron jobs across multiple OpenClaw gateway instances in HA deployments.

## Best Possible Solution

Define a supported distributed cron ownership model, then implement it behind a versioned config or plugin/API boundary with health/status visibility, CLI inspection, docs, and cross-process tests.

## Reproduction Assessment

Not applicable as a defect reproduction; this is a feature and architecture request. Source and docs inspection give high confidence that current main lacks distributed cron ownership because cron remains Gateway-local, file-backed, and process-locked only.

## Solution Assessment

Unclear until maintainers choose the architecture. Distributed locks or leader election are plausible, but the maintainable path needs a selected ownership model, backend/API surface, failure behavior, health reporting, CLI surface, and coordination with https://github.com/openclaw/openclaw/issues/25883.
