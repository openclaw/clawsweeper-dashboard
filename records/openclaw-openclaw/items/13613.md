## Label Justifications

- `P2`: This is a real production HA reliability gap with significant but topology-specific impact rather than an urgent regression in the default single-Gateway workflow.
- `impact:message-loss`: Independent Gateway schedulers can duplicate scheduled Slack, Discord, or other channel deliveries.

## Summary

Current main and the latest release still coordinate cron work only inside one Gateway process. Shared SQLite persistence does not atomically claim due jobs across processes, and the broader horizontal-scaling proposal remains open, so this valid HA feature gap needs maintainer architecture direction rather than cleanup closure.

## What This Changes

Add cross-gateway ownership or locking for cron jobs so HA deployments do not execute and deliver the same schedule multiple times.

## Best Possible Solution

Define a narrow atomic cron claim or lease contract around due-job selection, using canonical SQLite where the supported topology permits and a generic backend seam where it does not; include expiry and failover semantics, ownership in status/health output, and separate-process regression coverage without making a full queue platform the default.

## Reproduction Assessment

Yes. Source provides a high-confidence reproduction model: start separate Gateway processes against the same cron store and allow both to select the same due row before either state-only write completes; the only scheduler lock is process-local.

## Solution Assessment

The requested outcome is valid, but the backend-specific configuration in the issue is not yet the best settled design. The maintainable path is a narrow ownership contract aligned with the supported HA topology and the broader horizontal-scaling decision.
