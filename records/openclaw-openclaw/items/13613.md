## Label Justifications

- `P2`: Duplicate scheduled execution is a production HA reliability gap, but it is a broad architecture enhancement rather than an urgent confirmed regression in the default single-Gateway path.
- `impact:message-loss`: The issue describes duplicate Slack, Discord, or other channel deliveries when multiple Gateways execute the same scheduled job.

## Summary

Keep open: current main and v2026.6.8 still run cron as Gateway-local scheduling with process-local serialization and no distributed lease, leader, fencing, or shared coordination backend. The request is valid but needs maintainer architecture/product direction before implementation.

## What This Changes

Request distributed cron ownership across multiple Gateway instances so HA deployments do not execute the same scheduled job more than once.

## Best Possible Solution

Define a supported cron ownership contract first, then add an atomic fenced claim or leader lease at the scheduler boundary with health/status visibility, operator inspection, upgrade-safe config or plugin API, docs, and separate-process tests.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: independent Gateway processes have no shared lease, leader check, or distributed ownership contract, and the current guards are process-local. I did not run a live multi-host reproduction in this read-only review.

## Solution Assessment

No, not as the backend-specific config proposal alone. Redis, etcd, and DynamoDB are plausible backends, but maintainers first need to choose the supported topology and whether coordination belongs in core, a plugin API, or the broader horizontal-scaling design.
