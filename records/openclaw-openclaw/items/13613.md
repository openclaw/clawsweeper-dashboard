## Summary

Keep open. Current main still runs cron as a Gateway-local, file-backed scheduler with process-local locking/status/CLI surfaces, and the broader horizontal-scaling proposal remains open rather than a merged replacement.

## What This Changes

The issue requests distributed cron ownership for multi-Gateway HA deployments, including leader election or per-job locks plus coordination config, health/status, and CLI visibility.

## Best Possible Solution

Define a supported distributed cron ownership model, then implement it with a versioned config or plugin API boundary, health/status visibility, CLI inspection, docs, and multi-process regression coverage.

## Reproduction Assessment

Not applicable; this is a feature request rather than a broken existing behavior report. Source and docs inspection give high confidence that current main lacks distributed cron coordination today.

## Solution Assessment

Unclear until maintainers choose the architecture. The maintainable path is likely a generic distributed scheduler/ownership contract, possibly plugin-capable, rather than committing directly to one hardcoded Redis-only shape from the issue body.
