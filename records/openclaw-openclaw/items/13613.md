## Summary

Keep open. Current main still implements cron as Gateway-local scheduling with local state, process-local locking, and local status/CLI surfaces; the prior issue discussion also kept this open because no distributed lease, leader election, shared backend, or HA ownership model exists yet.

## What This Changes

The issue requests distributed cron coordination across multiple OpenClaw Gateways so HA deployments do not execute the same scheduled jobs more than once.

## Best Possible Solution

Define a supported distributed cron ownership model, then implement it behind a versioned config or plugin/API boundary with health/status visibility, CLI inspection, docs, and cross-process tests.

## Reproduction Assessment

Yes for the observable HA gap. Source inspection shows two independent Gateway processes have no shared lease or leader check, while the only duplicate-run guard is process-local and the existing test covers two services in one process.

## Solution Assessment

Unclear. The requested Redis/etcd/DynamoDB approaches are plausible, but maintainers first need to choose the supported ownership model, backend surface, failure semantics, observability, and alignment with #25883.
