## Summary

Keep open. Current main and the latest release still implement cron as Gateway-local scheduling with process-local locking and no distributed cron ownership, while the broader horizontal-scaling proposal remains open rather than an implemented replacement.

## What This Changes

The issue requests distributed coordination for cron jobs across multiple Gateway processes in HA deployments.

## Best Possible Solution

Define a supported distributed cron ownership model before adding config or backend dependencies, then implement it with health/status visibility, upgrade-safe config/docs, and separate-process tests.

## Reproduction Assessment

Yes, for the missing capability from source inspection: current cron scheduling is owned by Gateway-local CronService state and process-local locks, with no distributed coordination config or leader surface. This is not a strict bug reproduction because the request is for a new HA behavior rather than a broken documented single-Gateway contract.

## Solution Assessment

Unclear; Redis-style locks and leader election are plausible examples, but the best OpenClaw solution needs maintainer architecture review first. The durable end state should settle ownership semantics, backend/API boundaries, failure behavior, health/status visibility, and security posture before implementation.
