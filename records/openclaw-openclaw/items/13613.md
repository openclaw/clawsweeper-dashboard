## Label Justifications

- `P2`: Duplicate scheduled execution is a production reliability gap with meaningful but multi-Gateway-specific blast radius.
- `impact:message-loss`: Independent Gateways can duplicate Slack, Discord, or other channel deliveries for the same scheduled slot.

## Summary

Current main still lacks cross-Gateway cron ownership: each process independently loads due jobs and only later writes its running marker, while the broader horizontal-scaling proposal remains open and does not replace this cron-specific request.

## What This Changes

Add high-availability coordination so multiple Gateway instances do not execute and deliver the same scheduled cron job concurrently.

## Best Possible Solution

Define a supported cron ownership contract first, then add an atomic fenced claim or leader lease at the canonical scheduler boundary, with health/status visibility, operator inspection, upgrade-safe configuration, and separate-process tests covering takeover and backend failure.

## Reproduction Assessment

Yes, from source with high confidence: independent Gateway processes can load the same due job before either persists its running marker, and the existing duplicate test covers only two services sharing one Node process. No live multi-host reproduction was run.

## Solution Assessment

No, the issue's specific Redis, etcd, and DynamoDB proposal is not yet proven to be the best permanent boundary. Maintainers should first choose the supported topology and whether coordination belongs in the core SQLite scheduler, a generic plugin API, or a broader horizontal-scaling design.
