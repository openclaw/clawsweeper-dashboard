## Label Justifications

- `P2`: Duplicate scheduled execution in HA deployments is a meaningful production reliability gap, but this is a broad enhancement requiring architecture decisions rather than an urgent confirmed regression.
- `impact:message-loss`: The issue describes duplicated Slack, Discord, and other channel deliveries when multiple Gateways execute the same scheduled slot.

## Summary

Keep open: current main and v2026.6.10 still do not provide cross-Gateway cron ownership, leader election, distributed leases, or a coordination backend, and the broader horizontal-scaling issue remains open architecture input rather than a merged replacement.

## What This Changes

The issue requests distributed coordination for cron jobs so multiple OpenClaw Gateway instances do not execute the same scheduled work concurrently.

## Best Possible Solution

Define a supported distributed cron ownership contract first, then implement backend-neutral fenced job claiming or leader leasing with upgrade-safe config or plugin API, health/CLI visibility, docs, and separate-process tests.

## Reproduction Assessment

Yes. Source inspection shows independent Gateway processes have no shared cron lease or leader check, while current cron locking and active-job tracking are process-local; I did not run a live multi-host reproduction in this read-only review.

## Solution Assessment

No, not as the backend-specific proposal alone. Redis, etcd, and DynamoDB are plausible backends, but maintainers first need to choose the supported ownership boundary, failure semantics, and whether this belongs in core or a plugin/API seam.
