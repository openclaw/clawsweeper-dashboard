- `P2`: The issue describes meaningful HA cron reliability impact, but it is a broad enhancement requiring architecture decisions rather than an urgent confirmed regression.
- `impact:message-loss`: Multiple Gateways can duplicate scheduled channel deliveries when there is no shared cron ownership.

## Summary

Keep open: current main and v2026.6.10 still do not provide cross-Gateway cron ownership, leader election, distributed leases, or a coordination backend. This remains a valid cron HA feature gap, but it needs maintainer architecture/product direction before implementation.

## What This Changes

The issue requests distributed cron ownership across multiple Gateway instances, likely via leader election or per-job distributed locks plus cron coordination configuration.

## Best Possible Solution

Define a supported cron ownership contract first, then implement backend-neutral job claiming or leader leasing with upgrade-safe config/API, health/CLI visibility, docs, and separate-process tests.

## Reproduction Assessment

Yes, from source inspection: cron scheduling still uses process-local locks and active-job state, and no live multi-Gateway repro was run in this read-only review.

## Solution Assessment

No, not as the backend-specific proposal alone; Redis, etcd, DynamoDB, and current SQLite lease primitives are possible ingredients, but maintainers first need to choose the supported ownership boundary and failure semantics.
