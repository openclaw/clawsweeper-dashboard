## Summary

Keep open. Current main and the latest release still expose cron as Gateway-local scheduling with process-local execution guards and no supported multi-Gateway lease, leader, shared lock backend, or cron leader/status surface; the related horizontal-scaling proposal remains open rather than a merged replacement.

## What This Changes

Request distributed coordination for cron jobs across multiple Gateway instances so HA deployments do not execute the same scheduled work more than once.

## Best Possible Solution

Define a maintainer-approved distributed cron ownership contract, then implement lease or leader semantics with config/schema/docs/status/CLI support and multi-process Gateway regression coverage.

## Reproduction Assessment

Not applicable: this is a feature and architecture request rather than a failure against an existing documented contract. Source, docs, release-tag, and live GitHub checks give high confidence that OpenClaw lacks distributed cron coordination.

## Solution Assessment

Unclear until maintainers choose the architecture. Redis-style locks, leader election, or a plugin-capable lease boundary are plausible, but the maintainable path must first settle ownership semantics, backend/API boundaries, failure behavior, health/status visibility, and security posture.
