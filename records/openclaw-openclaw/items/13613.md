## Summary

Keep open. Current main still implements cron as Gateway-local scheduling with local files, process-local locking, local status/CLI surfaces, and no distributed lease, leader election, shared coordination backend, or HA cron ownership model; the related horizontal-scaling proposal remains open and the withdrawn slot-dedup PR was explicitly process-local.

## What This Changes

Request distributed cron coordination so multiple Gateway instances in HA deployments do not execute the same scheduled job concurrently.

## Best Possible Solution

Define a supported distributed cron ownership model, then implement it behind a versioned config or plugin/API boundary with health/status visibility, CLI inspection, docs, and cross-process tests.

## Reproduction Assessment

Yes for the observable HA gap. Source inspection shows independent Gateway processes have no shared lease or leader check, while the only duplicate-run guard is process-local and the current duplicate test stays inside one Node process.

## Solution Assessment

Unclear until maintainers choose the architecture. Distributed locks or leader election are plausible, but the maintainable path needs a chosen ownership model, backend/API surface, failure behavior, health reporting, CLI surface, and coordination with https://github.com/openclaw/openclaw/issues/25883.
