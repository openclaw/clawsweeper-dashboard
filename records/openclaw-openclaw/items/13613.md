## Summary

Keep open: current main still has Gateway-local, file-backed cron with process-local locking and no distributed lease, leader, or coordination backend; the remaining work is an architecture/product decision, not conservative cleanup.

## What This Changes

The issue requests distributed coordination for cron jobs so multiple Gateway instances do not execute the same scheduled work independently.

## Best Possible Solution

Define a supported distributed cron ownership model, then implement it behind a versioned config or plugin/API boundary with health/status visibility, CLI inspection, docs, and cross-process tests.

## Reproduction Assessment

Not applicable as a defect reproduction; this is a feature/architecture request. Source and docs inspection give high confidence that current main lacks distributed cron ownership: the scheduler is Gateway-local, file-backed, and process-locked only.

## Solution Assessment

Unclear until maintainers choose the architecture. Distributed locks or leader election are plausible, but the maintainable path needs a chosen ownership model, backend/API surface, failure behavior, health reporting, CLI surface, and coordination with https://github.com/openclaw/openclaw/issues/25883.
