## Summary

Current main still lacks cross-process cron ownership, and the broader scaling proposal remains open; this needs a maintainer architecture decision rather than closure or automated implementation.

## What This Changes

Requests distributed coordination so cron jobs execute once across multiple Gateway instances.

## Best Possible Solution

Define a backend-neutral, fenced cron ownership lease with explicit slot identity, renewal, failover, outage policy, health/status, and CLI observability, then place backend implementations at the approved plugin or core boundary and prove behavior with separate Gateway processes.

## Reproduction Assessment

Yes. Current source explicitly limits cron serialization to one process and exposes no external ownership check, so separate gateways configured with the same jobs can independently execute the same scheduled slot.

## Solution Assessment

No, the backend-specific configuration in the issue is only a plausible implementation. The maintainable solution first needs a generic ownership contract, fencing and outage semantics, and an explicit core-versus-plugin decision.

## Risks / Open Questions

- Fail-open behavior during coordination-backend outages can duplicate API calls and channel deliveries, while fail-closed behavior can suppress scheduled work.
- Bundling Redis, etcd, or DynamoDB clients would add credentials, configuration, startup, health, and dependency surface to core without a settled owner boundary.
- A correct design needs fencing or equivalent atomic ownership around scheduled slots; a TTL-only lock can permit overlapping execution after expiry or process pauses.
