## Label Justifications

- `P2`: Duplicate scheduled execution in HA deployments is a real production reliability gap, but it is a broad enhancement requiring architecture decisions rather than an urgent confirmed regression.
- `impact:message-loss`: The issue describes duplicate Slack, Discord, and other channel deliveries when multiple Gateways execute the same scheduled slot.

## Summary

Keep open: current main and v2026.6.8 still do not provide cross-Gateway cron ownership, leader election, distributed leases, or a coordination backend; the broader horizontal-scaling issue remains open architecture input rather than a merged replacement.

## What This Changes

Request distributed cron coordination across multiple Gateway instances, including leader election or locks, backend configuration, health/status visibility, and CLI inspection.

## Best Possible Solution

Define a supported cron ownership contract first, then implement backend-neutral fenced job claiming or leader leasing with upgrade-safe config or plugin API, health/CLI visibility, docs, and separate-process tests.

## Reproduction Assessment

Yes, from source inspection with high confidence: independent Gateway processes have no shared cron lease or leader check, and current cron locking is explicitly process-local. I did not run a live multi-host reproduction in this read-only review.

## Solution Assessment

No, not as the backend-specific proposal alone. Redis, etcd, and DynamoDB are plausible backends, but maintainers first need to choose the supported ownership boundary, failure semantics, and whether this belongs in core or a plugin/API seam.
