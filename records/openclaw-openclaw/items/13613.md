## Label Justifications

- `impact:message-loss`: The issue describes duplicate Slack, Discord, or other channel deliveries when multiple Gateways execute the same scheduled slot.

## Summary

Keep open: current main and latest release still lack cross-Gateway cron ownership, and the live related horizontal-scaling item remains open rather than a merged replacement for this cron-specific HA request.

## What This Changes

Add distributed coordination for cron jobs so multiple Gateway instances do not independently execute the same scheduled work in high-availability deployments.

## Best Possible Solution

Define a supported cron ownership contract first, then add an atomic fenced claim or leader lease at the scheduler boundary with health/status visibility, upgrade-safe config or plugin API, docs, and separate-process tests.

## Reproduction Assessment

Yes, from source with high confidence: independent Gateway processes have no shared lease or leader check, and the current lock is explicitly process-local. I did not run a live multi-host reproduction in this read-only review.

## Solution Assessment

No, not as directly proposed yet. Redis, etcd, and DynamoDB are plausible backends, but maintainers first need to choose the supported topology and whether coordination belongs in core, a plugin API, or the broader horizontal-scaling design.
