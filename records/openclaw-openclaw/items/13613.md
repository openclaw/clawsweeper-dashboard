## Summary

Keep open. Current main and v2026.5.20 still implement cron as Gateway-local scheduling with local files, process-local locking, and local CLI/status surfaces; distributed cron ownership remains a real feature gap that needs maintainer architecture direction.

## What This Changes

The issue requests distributed coordination for cron jobs across multiple Gateway instances so HA deployments do not execute the same scheduled work more than once.

## Best Possible Solution

Define a supported distributed cron ownership model, likely behind a generic or plugin-capable lease boundary, then add config/schema/docs/status/CLI support and multi-process Gateway regression coverage.

## Reproduction Assessment

Not applicable: this is a feature and architecture request rather than a failure against an existing documented contract. Source, docs, release-tag, and live issue checks give high confidence that OpenClaw lacks distributed cron coordination.

## Solution Assessment

Unclear until maintainers choose the architecture. Redis locks and leader election are plausible examples, but the maintainable path should first settle ownership semantics, backend/API boundaries, failure behavior, health/status visibility, and security posture.
