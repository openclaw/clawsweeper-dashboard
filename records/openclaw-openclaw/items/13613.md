## Label Justifications

- `P2`: This is a normal-priority production HA cron feature gap with real but bounded scheduler and deployment impact.
- `impact:message-loss`: Uncoordinated scheduled jobs across Gateways can duplicate or misroute scheduled Slack, Discord, or other channel deliveries.

## Summary

Keep open. Current main and v2026.5.22 still expose cron as Gateway-local scheduling with local config, local status/CLI, and process-local locking; the requested HA cron ownership model remains missing and needs maintainer architecture/product direction.

## What This Changes

Request distributed cron coordination, such as leader election or distributed locks, so multiple Gateway instances do not execute the same scheduled jobs.

## Best Possible Solution

Define a supported distributed cron ownership model behind a generic or plugin-capable lease boundary, then add config/schema/docs/status/CLI support and multi-process Gateway regression coverage.

## Reproduction Assessment

Not applicable: this is a feature and architecture request rather than a failure against an existing documented contract. Source, docs, release-tag, and live issue checks give high confidence that OpenClaw lacks distributed cron coordination.

## Solution Assessment

Unclear until maintainers choose the architecture. Redis locks and leader election are plausible examples, but the maintainable path should first settle ownership semantics, backend/API boundaries, failure behavior, health/status visibility, and security posture.
