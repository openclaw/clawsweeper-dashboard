## Summary

Keep open: current main still has Gateway-local, file-backed cron with only process-local locking and no distributed lease, leader election, shared coordination backend, HA status, or `cron leader` CLI surface. This remains a valid enhancement, but it needs maintainer architecture/product direction rather than conservative cleanup.

## What This Changes

Add distributed coordination for cron jobs across multiple OpenClaw Gateway instances so HA deployments do not execute the same scheduled work more than once.

## Best Possible Solution

Define a supported distributed cron ownership model, then implement it behind a versioned config or plugin/API boundary with health/status visibility, CLI inspection, docs, and cross-process tests.

## Reproduction Assessment

Not applicable as a defect reproduction; this is a feature and architecture request. Source and docs inspection give high confidence that current main lacks distributed cron ownership: scheduling is Gateway-local, file-backed, and process-locked only.

## Solution Assessment

Unclear until maintainers choose the architecture. Distributed locks or leader election are plausible, but the maintainable path needs a selected ownership model, backend/API surface, failure behavior, health reporting, CLI surface, and coordination with https://github.com/openclaw/openclaw/issues/25883.
