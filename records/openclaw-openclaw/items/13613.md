## Summary

Keep open. Current main still lacks distributed cron ownership: cron remains Gateway-local and process/file-backed, the accepted config/schema/status/CLI surfaces have no coordination contract, and the broader horizontal-scaling proposal at https://github.com/openclaw/openclaw/issues/25883 is still open rather than a merged replacement.

## What This Changes

Request distributed locking or leader election so cron jobs execute once across multiple Gateway instances in HA deployments.

## Best Possible Solution

Define a supported distributed cron ownership model, likely through a generic core or plugin lease boundary, then add config/schema/docs/status/CLI support and multi-process regression coverage before recommending HA cron deployments.

## Reproduction Assessment

Not applicable; this is a feature request rather than a broken existing behavior report. Source and docs inspection give high confidence that current main lacks distributed cron coordination today.

## Solution Assessment

Unclear until maintainers choose the architecture. The Redis/leader-election examples are plausible, but the maintainable path should define a generic distributed ownership contract before committing to one backend shape.
