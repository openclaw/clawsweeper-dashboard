## Label Justifications

- `impact:message-loss`: The reported symptom was a generated reply that did not reach Telegram.
- `impact:auth-provider`: The failure depended on insufficient paired-device scopes carried through Gateway authorization.

## Summary

The reported false-success path is now covered by the shipped durable-delivery implementation: Telegram uses the shared final-reply sender, Gateway scopes are carried into outbound execution, failed or partial sends remain failures, and Telegram marks delivery only after a visible handled result.

## What This Changes

Request reliable Telegram failure reporting when insufficient local CLI device scopes prevent a generated reply from reaching the chat.

## Best Possible Solution

Keep the shipped scope-aware durable-delivery path as the canonical behavior and use the broader cross-channel contract issue for any remaining observability or conformance work.

## Reproduction Assessment

Yes at source level: the reported under-scoped boundary maps to GatewayClientScopes in durable outbound delivery, and the shipped tests prove failed or partial durable sends remain terminal failures rather than successful delivery.

## Solution Assessment

Yes. Propagating the authenticated device scopes through the shared durable-delivery contract and marking Telegram delivered only after a visible handled result is the narrowest maintainable fix.
