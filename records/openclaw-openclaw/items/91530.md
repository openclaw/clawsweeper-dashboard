## Summary

Keep open. Current main already has explicit Gateway scope-upgrade diagnostics, but Telegram delivery/fallback accounting still treats delivery misses as generic failed/skipped reply state, and the related open Telegram delivery issues do not cover this device-scope cause.

## What This Changes

The issue asks OpenClaw to surface a clearer operator-facing delivery error when Telegram reply generation succeeds locally but delivery fails because the active device lacks required scopes.

## Best Possible Solution

Bridge existing Gateway scope-upgrade and missing-scope details into Telegram delivery/status diagnostics so operators can see that generation succeeded, delivery failed, and a specific scope approval is required.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection does show the relevant current-main mechanics: scope errors are explicit in Gateway diagnostics, while Telegram delivery fallback currently has only generic delivered/skipped/failed accounting.

## Solution Assessment

No, current behavior is not the best end state because it leaves the operator-facing scope cause disconnected from Telegram delivery failure reporting. The maintainable fix is to reuse existing Gateway scope-error details in the delivery/status path without adding config or weakening authorization.
