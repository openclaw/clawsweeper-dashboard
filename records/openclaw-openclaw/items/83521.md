## Label Justifications

- `P1`: The report describes a plausible Slack production workflow where duplicate turns or thread-to-channel fallback can affect real users and expose thread-scoped content.
- `impact:message-loss`: The issue is about duplicated Slack inbound turns and replies routed to the wrong Slack surface.
- `impact:session-state`: Duplicate inbound events can create extra agent turns and wrong thread context can attach a reply to the wrong conversation state.
- `impact:security`: The reported fail-open path can expose Slack thread-scoped content to a broader channel audience.

## Summary

Keep open. Current main still has source-level evidence for both reported Slack risks: duplicate inbound suppression is a 60s in-memory cache, and same-channel Slack sends can proceed without `thread_ts` when thread context cannot be resolved.

## What This Changes

The issue requests durable Slack inbound idempotency and fail-closed thread routing for Slack thread-originated message-tool sends.

## Best Possible Solution

Add Slack-owned durable inbound replay/idempotency for account/channel/timestamp/event ids, keep paired `message` and `app_mention` converged beyond 60 seconds, and enforce an explicit `topLevel: true` or `threadId: null` opt-out before any thread-originated same-channel send may post at channel root.

## Reproduction Assessment

Yes, at source level: current main uses a 60s in-memory Slack dedupe key and returns no thread id when same-channel thread context is missing. I did not run a live Slack replay, so this is not a full end-to-end reproduction.

## Solution Assessment

Unclear until maintainers confirm the fail-closed compatibility policy. The durable claimable-dedupe plus Slack-specific thread guard direction is likely maintainable, but the exact TTL/storage namespace and error behavior should be owner-reviewed.
