## Label Justifications

- `P1`: The report describes duplicate Slack turns and possible thread-to-channel leakage in a real channel workflow with privacy impact.
- `impact:message-loss`: The affected Slack behavior can duplicate inbound turns or route replies to the wrong Slack surface.
- `impact:session-state`: The failure depends on losing or failing to resolve Slack thread context for a thread-originated turn.
- `impact:security`: Thread-scoped Slack content may be posted to a broader parent channel when context is lost.

## Summary

Keep open: current main still has the reported Slack source paths, and the related open threading item is a separate feature request rather than a duplicate of this idempotency and thread-leakage safety bug.

## What This Changes

The issue asks OpenClaw to make Slack inbound handling idempotent beyond the current short in-memory window and prevent thread-originated message-tool sends from falling back to channel-root posts without an explicit opt-out.

## Best Possible Solution

Keep this open and fix Slack with account-scoped durable or longer-lived inbound idempotency plus a same-channel threaded-send guard that errors unless a thread is resolved or `topLevel`/`threadId: null` is explicit.

## Reproduction Assessment

Yes, at source level: replaying the same Slack channel/timestamp after the 60s in-memory window reaches the handler again, and a same-channel send with missing `currentThreadTs`/`currentChannelId` posts without `thread_ts`. I did not run a live Slack replay in this read-only review.

## Solution Assessment

Unclear without maintainer compatibility input: durable account-scoped dedupe is the right direction, but fail-closed same-channel routing should be accepted explicitly because it changes contextless send behavior.
