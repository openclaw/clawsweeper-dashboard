## Summary

Keep open: current main still lacks mobile transport-level proxy-auth metadata and pre-upgrade header forwarding for Android/iOS clients, and the only directly linked implementation PR is closed unmerged rather than a viable fix path.

## What This Changes

Request transport-level proxy authentication support in mobile setup/manual-connect flows so Android and iOS clients can pass bearer/basic/header/cookie/OIDC credentials through an authenticating reverse proxy before the Gateway WebSocket upgrade.

## Best Possible Solution

Define a shared Android/iOS setup-code and manual-connect transport-auth contract, including credential storage, clearing, redaction, TLS, and OIDC browser-return rules, then land a new proof-backed implementation slice or explicitly split the work.

## Reproduction Assessment

Not applicable as a bug: this is a missing feature request, not broken documented behavior. Source inspection confirms current setup-code, Android, and iOS WebSocket paths do not carry proxy-layer auth fields before the HTTP upgrade.

## Solution Assessment

Unclear as proposed: static bearer/basic support is a plausible first slice, but the best maintainable fix needs shared mobile credential and security direction for Android, iOS, fixed headers, cookies, and OIDC before implementation proceeds.
