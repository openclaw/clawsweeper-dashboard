## Summary

Keep this issue open. Current main still lacks a way for mobile clients to provide reverse-proxy credentials before the WebSocket upgrade, and open PR #73168 is an unmerged implementation candidate that uses closing syntax for this issue.

## What This Changes

Request to add mobile transport-level authentication support for reverse proxies, including setup-code or manual configuration that lets Android and iOS send bearer/basic/custom-header/cookie/OIDC credentials before reaching the Gateway.

## Best Possible Solution

Review PR #73168 as the current candidate, or replace it with an additive transport-auth design that extends setup-code/manual configuration and both mobile WebSocket clients without weakening OpenClaw pairing auth.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: current main's setup payload and mobile config models have no proxy-auth fields, and Android/iOS create the WebSocket upgrade from a URL without pre-upgrade auth headers.

## Solution Assessment

Unclear. The requested direction is valid, but maintainers still need to decide whether the first accepted fix is Android-only bearer/basic support or a cross-platform schema that also covers custom headers, cookies, and OIDC.
