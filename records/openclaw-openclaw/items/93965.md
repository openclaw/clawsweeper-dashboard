## Summary

Keep open: the provider-scoped, abortable boundary matches maintainer feedback, but this patch currently starts its idle timer before any upstream stream event, adding a new 120s opencode-go cron time-to-first-event cap. The supplied proof is fake-timer unit coverage only, so the external PR still needs real after-fix provider proof before merge.

## What This Changes

Adds an opencode-go stream-termination wrapper around `wrapStreamFn` plus tests, injecting an AbortSignal and emitting an aborted error when the wrapped stream stays idle.

## Best Possible Solution

Keep the fix provider-owned, but start the stall timer only after the first forwarded upstream event and add real opencode-go proof plus delayed-first-event coverage before merge.

## Reproduction Assessment

No high-confidence live reproduction was established in this review. The source path and reporter logs make the stuck-stream shape plausible, and the PR adds mock boundary tests, but neither the report nor the PR proves the after-fix behavior in a real opencode-go run.

## Solution Assessment

No, not as submitted. The provider-owned abortable boundary is the right direction, but the timer lifecycle is too broad because it starts before first stream progress and the PR lacks real behavior proof.
