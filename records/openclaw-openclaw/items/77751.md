## Summary

Keep open: the patch still has a concrete no-input stdin regression and lacks required real behavior proof. Current main does not already contain the requested mitigation, so the PR remains useful but is not mergeable as-is.

## What This Changes

The PR changes child supervisor spawning to default omitted stdin to a pipe in non-TTY runtimes and adds an all-pipe EBADF fallback.

## Best Possible Solution

Revise the mitigation to preserve EOF for no-input service spawns, add focused regression coverage, attach redacted launchd/gateway proof, and explicitly cover or split the MCP stdio startup path.

## Reproduction Assessment

Unclear for the original high-FD macOS launchd EBADF failure because this review did not recreate that setup. The PR-introduced stdin regression is source-reproducible and backed by a direct Node probe showing pipe stdin keeps a no-input reader alive.

## Solution Assessment

No. Retrying with safer stdio is plausible, but this patch should preserve no-input EOF semantics and should not close the linked MCP-startup scope without covering or splitting that direct spawn path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Close no-input pipes after non-TTY fallback:** `src/process/supervisor/adapters/child.ts:39`
  - body: When `process.stdin` is not a TTY and no input is provided, this selects an open pipe but `createChildAdapter` only ends stdin for explicit input or `pipe-closed`. Commands that read stdin, such as `cat`, can now hang until timeout instead of seeing EOF from inherited `/dev/null`; use `pipe-closed`/`ignore` semantics or close the fallback stdin after spawn.
  - confidence: 0.92
