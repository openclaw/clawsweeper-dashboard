## Summary

Keep open. The target code path is valid for the related service-mode spawn report, but the proposed patch can leave no-input child processes waiting forever on an open stdin pipe and still needs concrete after-fix runtime proof before merge.

## What This Changes

The PR changes `createChildAdapter` to avoid inherited stdin for non-TTY runs and always append a pipe-stdio spawn fallback for EBADF retries.

## Best Possible Solution

Keep the service-mode mitigation, but implement no-input non-TTY and EBADF fallback paths with closed or ignored stdin semantics, add regression coverage and a changelog entry, and attach redacted real gateway proof.

## Reproduction Assessment

Unclear for the original EBADF report: the source path and related issue logs line up, but I did not recreate the high-fd macOS service failure. The introduced stdin hang is high-confidence from the adapter source and a direct Node spawn probe.

## Solution Assessment

No. Retrying with a pipe is a reasonable direction, but leaving stdin open changes the previous `/dev/null` EOF behavior; the narrower fix is to close or ignore stdin for no-input non-TTY children and for the fallback.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Close stdin for no-input non-TTY children:** `src/process/supervisor/adapters/child.ts:41`
  - body: When `process.stdin.isTTY` is false and no input is provided, this now selects an open pipe. Commands that read stdin used to see EOF from inherited `/dev/null`; with an open pipe they can hang until timeout. Use `pipe-closed` semantics, or explicitly close the fallback stdin after spawn, so no-input children still see EOF.
