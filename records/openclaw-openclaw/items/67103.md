Full review comments:

- **[P1] Restore the markdown helper definitions:** `ui/src/ui/markdown.ts:84-91`
  - body: This replacement block deletes `installHooks`, `getCachedMarkdown`, and `setCachedMarkdown`, but `toSanitizedMarkdownHtml` still calls all three later in the same file. The PR head will fail type checking and the shared renderer loses the existing cache and DOMPurify link-hardening path.
  - confidence: 0.99
- **[P1] Update the lockfile with KaTeX:** `ui/package.json:17`
  - body: The PR adds `katex` to `ui/package.json` without a matching `pnpm-lock.yaml` change. OpenClaw CI and packaging use frozen pnpm installs, so the dependency addition is not installable/reviewable as submitted.
  - confidence: 0.93
- **[P1] Respect dollar delimiter boundaries:** `ui/src/ui/markdown.ts:151-159`
  - body: The inline scanner treats any non-space `$` as math and pairs it with the next `$`, so prose like `between $5 and $10` is converted into KaTeX instead of remaining normal chat text. That is a user-visible compatibility regression for the shared markdown renderer.
  - confidence: 0.9
- **[P2] Skip complete code spans before scanning math:** `ui/src/ui/markdown.ts:127-133`
  - body: The code-span bypass only skips to the next single backtick. Markdown spans or fences delimited with multiple backticks can expose `$...$` content to the math extractor, so literal code examples can be rendered as KaTeX.
  - confidence: 0.86
- **[P2] Scope inline styles to KaTeX output:** `ui/src/ui/markdown.ts:60`
  - body: Adding `style` to the global DOMPurify allowlist permits inline styles on every allowed tag, not just KaTeX markup. Keep the existing sanitizer boundary narrow by allowing KaTeX-required style only for verified KaTeX output or by avoiding inline style allowance where possible.
  - confidence: 0.8
