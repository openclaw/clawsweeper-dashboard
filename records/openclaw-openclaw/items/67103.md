## Summary

Keep open. Current main still lacks Control UI math rendering, and this active implementation PR is not safe to merge because it breaks the shared markdown renderer, weakens sanitizer boundaries, omits lockfile proof, and lacks after-fix real behavior proof.

## What This Changes

This PR adds a KaTeX dependency and changes the Control UI shared markdown renderer and tests to render inline and display LaTeX math in chat markdown.

## Best Possible Solution

Rework this PR or land a successor in the shared Control UI markdown renderer that preserves cache and DOMPurify link hardening, uses robust delimiter rules, scopes KaTeX-specific sanitizer allowances, includes lockfile/dependency review, and proves the rendered UI behavior.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not as a bug repro, but yes as source-level current behavior: Control UI chat flows through toSanitizedMarkdownHtml, and current main has no math-rendering stage or math-rendering dependency.

## Solution Assessment

Is this the best way to solve the issue? No. The shared markdown renderer is the right surface, but this patch is not the best implementation because it breaks existing helpers, weakens sanitizer boundaries, omits lockfile proof, lacks real behavior proof, and uses fragile delimiter parsing.

  - body: This replacement block removes installHooks, getCachedMarkdown, and setCachedMarkdown, but toSanitizedMarkdownHtml still calls all three. The branch will fail before rendering chat HTML and also drops the existing cache plus DOMPurify link-hardening hook.

  - confidence: 0.9

  - body: The PR adds katex to ui/package.json but does not update pnpm-lock.yaml, so frozen-lockfile installs do not have a reproducible resolution for the new dependency.
  - confidence: 0.86
