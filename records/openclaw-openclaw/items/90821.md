## Summary

Keep open: the PR is a useful partial direction for OpenClaw-owned/context-engine compaction, but it is not merge-ready because native Codex app-server compaction still continues after `thread/compact/start` and the PR clears the active-run handle without a real server-side interrupt path; the supplied proof is still focused Vitest output, not real runtime cancellation proof.

## What This Changes

Registers manual `/compact` as an active embedded run with a local `AbortController` and passes that signal into `compactEmbeddedAgentSession`.

## Best Possible Solution

Extend cancellation so every supported `/compact` backend remains represented by an active handle and aborts through its real cancel primitive, then add redacted live runtime proof before closing the linked bug.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: current main invokes `/compact` without active-run registration or `abortSignal`, while Codex native compaction returns pending after `thread/compact/start`. I did not run a live gateway scenario in this read-only review.

## Solution Assessment

No. The active-run handle plus AbortSignal is a good partial owner-boundary fix for OpenClaw-owned compaction, but it is not the best complete fix while native Codex compaction ignores that signal and is not interrupted through Codex's server-side turn API.
