## Summary

Keep this PR open, but not merge-ready: it is aimed at a real cancelability bug from https://github.com/openclaw/openclaw/issues/66535, yet the patch adds a parallel compaction registry instead of making manual compaction visible through the canonical active-run surfaces. That leaves session-key lookup, wait/drain, restart handling, diagnostics, and gateway `chat.abort` able to disagree with the new direct-abort path.

## What This Changes

The PR adds a compaction abort-controller registry and passes its signal into manual `/compact` so `abortEmbeddedAgentRun` can abort some in-progress compactions.

## Best Possible Solution

Register manual compaction through the canonical active-run lifecycle, or extend that canonical state in one place, then prove `/stop`, direct session-id abort, gateway/control abort, wait/drain cleanup, and restart handling all observe the same compaction state.

## Reproduction Assessment

Yes from source: current main starts manual `/compact` without an `abortSignal`, while the existing active-run and chat abort surfaces only see embedded/reply/chat run registries. I did not run a live long compaction in this read-only review.

## Solution Assessment

No. Passing an abort signal is necessary, but a side registry is not the best fix because it leaves the canonical active-run, session-key, drain, diagnostics, and gateway abort surfaces unaware.
