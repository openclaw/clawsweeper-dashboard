Keep open: current main and the latest release already cover the originally reported `content[0].text` case, but this PR still contains unique array-`text`/string-fragment support and still needs bounds plus real behavior proof before merge.

## What This Changes

The PR adds Control UI tool-card extraction for array-shaped tool result `text` and `content` payloads, node coverage, and a changelog entry.

## Best Possible Solution

Merge bounded array-shape support only after adding fragment/character caps, focused regression coverage, and redacted real Control UI/WebSocket proof.

## Reproduction Assessment

Yes for the PR blocker: source inspection of the PR diff shows array fragments are collected and joined without explicit bounds before preview parsing/rendering. The original linked issue is not a current-main reproduction because main and v2026.5.12 already extract `content[0].text`.

## Solution Assessment

No, not as currently proposed. Reading array-shaped tool output belongs in the tool-card path, but this patch should bound collection/truncation and include real Control UI/WebSocket proof before merge.
