## Summary

Keep open: current main already fixes the originally reported `content[0].text` shape, but this PR still contains broader array-shaped `text`/`content` extraction work and had a recent maintainer-side repair. It is not merge-ready because the new array collection is unbounded and the external PR still lacks real Control UI/WebSocket behavior proof.

## What This Changes

The PR updates Control UI tool-card extraction to join array-shaped tool result `text` and `content` fragments, adds node coverage, and adds a changelog entry.

## Best Possible Solution

Bound fragment count and total output characters in the array extraction path, preserve a visible truncation marker, keep focused regression coverage, and require redacted real Control UI/WebSocket proof before merge.

## Reproduction Assessment

Yes for the blocking PR defect: source inspection shows array-shaped fragments are collected and joined before preview/sidebar parsing without explicit limits. The original `content[0].text` report is not a current-main repro because main already handles content text blocks.

## Solution Assessment

No, not yet. Reading array-shaped tool output belongs in the tool-card path, but the implementation should bound collection and include real behavior proof before it is merge-ready.
