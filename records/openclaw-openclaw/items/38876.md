## Summary

Keep open: the linked original WebUI `content[0].text` bug is already handled on current `main`, but this branch still carries broader tool-result array parsing work that is not merge-ready because it joins provider-influenced fragments without bounds and lacks real behavior proof.

## What This Changes

The branch adds Control UI extraction, node tests, and a changelog entry for joining array-shaped tool-result `text` and `content` fragments.

## Best Possible Solution

Bound fragment count and total extracted characters with a visible truncation marker, keep focused regression coverage, and require redacted real Control UI/WebSocket proof before merge.

## Reproduction Assessment

Yes for the blocking PR defect: source inspection shows array-shaped fragments are collected and joined before preview parsing and rendering without explicit limits. The original `content[0].text` report is not a current-main repro because that shape is already covered.

## Solution Assessment

No, not yet. Reading array-shaped tool output belongs in the tool-card path, but the implementation should bound collection and include real behavior proof before it is merge-ready.

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Bound array tool-output collection:** `ui/src/ui/chat/tool-cards.ts:49-60`
