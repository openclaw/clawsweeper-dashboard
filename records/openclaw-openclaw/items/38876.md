## Summary

Keep open: current main already covers the original `content[0].text` bug, but this PR still proposes broader Control UI extraction for array-shaped `text` and `content` payloads. It remains blocked by missing real behavior proof and an unbounded client-side extraction path introduced by the branch.

The branch adds Control UI tool-card extraction for array-shaped tool result `text` and `content` payloads, node regression tests, and a changelog entry.

## Best Possible Solution

Bound fragment count and total extracted characters with a visible truncation marker, keep focused regression coverage for array text/content fragments, and require redacted real Control UI/WebSocket proof before merge.

## Reproduction Assessment

Yes for the merge blocker: the PR diff shows array branches collect fragments and return `parts.join("\n")` before preview parsing. The original `content[0].text` report is not a current-main repro because current main has content-array extraction and a node test.

## Solution Assessment

No, not yet. Reading array-shaped tool output belongs in this tool-card path, but the implementation should bound collection and include real UI/WebSocket proof before merge.

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Bound array tool-output collection:** `ui/src/ui/chat/tool-cards.ts:49-60`
  - body: The new array handling collects every string fragment and joins the full output before preview parsing and sidebar rendering. ## Security Review

Status: needs_attention
