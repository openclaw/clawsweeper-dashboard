## Summary

Keep open. Current main still does not extract array-shaped tool-result text, so the PR addresses a real Control UI bug, but the current head still joins every array fragment without a size bound before preview extraction, sidebar formatting, and rendering; that is a concrete client-side resource-exhaustion blocker before merge.

## What This Changes

This PR updates Control UI tool-card extraction to join readable fragments from array-shaped `text` and `content` tool-result payloads, adds node coverage for array and empty-array cases, and adds a changelog entry for the linked tool-output display fix.

## Best Possible Solution

Amend the PR so array-shaped tool-result extraction is bounded by fragment count and total collected characters before joining, preview parsing, sidebar formatting, or Lit rendering; keep the regression tests and changelog credit, then land after targeted UI tests and changed-surface validation.

## Reproduction Assessment

Yes. Pass an assistant message with a `tool_result` item whose `content` is `[{ text: "ok" }]`; current main returns `undefined` from `extractToolText` and falls back to the visible `No output` path. The PR blocker is also source-reproducible with a very large array because the new branch joins every collected fragment.

## Solution Assessment

No, not as currently proposed. Normalizing array-shaped tool-result text is the right narrow fix, but the implementation should cap collection before `join`, `extractCanvasFromText`, `formatToolOutputForSidebar`, and rendering.
