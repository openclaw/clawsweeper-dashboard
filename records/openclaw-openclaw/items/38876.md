## Summary

Keep open: current main already covers the original `content[0].text` failure, but this PR remains the canonical broader array-fragment patch and is not merge-ready because it lacks real behavior proof and still joins provider-influenced fragments without bounds.

## What This Changes

The branch adds Control UI tool-card extraction for array-shaped `text` and `content` payloads, with node tests and a changelog entry.

## Best Possible Solution

Bound fragment count and total extracted characters with a visible truncation marker, keep focused regression coverage for array text/content fragments, and require redacted real Control UI/WebSocket proof before merge.

## Reproduction Assessment

Yes for the blocking PR defect: source inspection shows array fragments are collected and joined without explicit limits before preview parsing and rendering. The original `content[0].text` issue is not a current-main repro because that shape is already extracted and covered by a node test.

## Solution Assessment

No, not yet. Reading array-shaped tool output belongs in this tool-card path, but the implementation should bound collection and include real behavior proof before it is merge-ready.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Bound array tool-output collection:** `ui/src/ui/chat/tool-cards.ts:49-60`
  - body: The new array extraction path collects every string fragment and joins the full output before preview parsing and sidebar rendering. A provider-influenced payload with many fragments or huge strings can allocate a large browser string and drive expensive JSON parsing/rendering work, so cap fragment count and total characters and preserve a visible truncation marker before returning `outputText`.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The PR introduces a client-side resource-exhaustion risk by joining provider-influenced array-shaped tool output without explicit bounds.

Concerns:

- **[medium] Unbounded tool-result fragment joining:** `ui/src/ui/chat/tool-cards.ts:49`
  - body: `collectTextParts()` walks every array entry and callers join all collected strings into one `outputText`, which is then parsed for previews and rendered. Very large arrays or fragments can force large allocations and expensive browser work, so explicit limits are needed.
  - confidence: 0.86
