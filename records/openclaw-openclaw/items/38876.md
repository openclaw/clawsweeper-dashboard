## Summary

Keep open. Current main now covers the original `content[0].text` display path, but this PR still proposes broader array extraction and its head still joins provider/tool-influenced fragments without bounds, matching the existing security review concern.

## What This Changes

The PR adds array-shaped tool-result `text`/`content` extraction in Control UI tool cards, node coverage for those payloads, and an unreleased changelog entry.

## Best Possible Solution

Bound collected tool-output fragments by count and total characters, mark truncation, keep regression coverage, and then decide whether the broader `item.text` and string-fragment support should land on top of current main.

## Reproduction Assessment

Yes for the blocking review finding: source inspection of PR head shows `collectTextParts` walks every array entry and joins all collected strings. The original `content[0].text` bug is no longer a failing current-main path because current main added object text-block extraction.

## Solution Assessment

No, not yet. Reading array-shaped tool output is the right narrow owner-path fix, but the PR should bound collection before preview parsing, sidebar formatting, and Lit rendering.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Bound array tool-output collection:** `ui/src/ui/chat/tool-cards.ts:49-60`
  - body: The new array path collects every string fragment and joins the full result before preview extraction, sidebar formatting, and rendering. Tool results can be provider-influenced, so a huge array or huge fragments can hang the Control UI or crash the tab; cap fragment count and total characters, and mark truncation, before joining.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The PR still has a concrete client-side resource-exhaustion risk from joining unbounded array-shaped tool-result fragments.

Concerns:

- **[medium] Unbounded tool-result fragment joining:** `ui/src/ui/chat/tool-cards.ts:49`
  - body: `collectTextParts` walks every array entry and callers join every collected string into one `outputText`; that full string is then parsed for previews, formatted for the sidebar, and rendered. A very large array or large fragments can force large allocations and expensive browser work, so explicit size limits are needed before merge.
  - confidence: 0.86
