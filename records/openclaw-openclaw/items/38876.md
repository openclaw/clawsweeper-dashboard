## Summary

Keep open. The PR targets a real Control UI regression, but the current head still concatenates every array fragment into one browser-side string before preview parsing, sidebar formatting, and rendering, leaving a concrete client-side resource-exhaustion blocker.

## What This Changes

This PR updates Control UI tool-card extraction to read array-shaped tool-result `text` and `content` fragments, adds node tests for array and empty-array cases, and adds a changelog credit for the linked display fix.

## Best Possible Solution

Keep the contributor PR as the canonical fix, but amend it so array-shaped tool-result extraction enforces explicit maximum fragment count and total collected characters before joining, carries a truncation marker, and covers the bounded path in the existing node tests before running the targeted UI test and changed-surface gate.

## Reproduction Assessment

Yes. On current `main`, an assistant message containing a `tool_result` item with `content: [{ text: "ok" }]` reaches `extractToolText` and returns `undefined`, which drives the visible `No output` fallback. The PR blocker is also source-reproducible: a very large array makes `collectTextParts` walk every entry and join the full result before downstream parse/render paths.

## Solution Assessment

No, not yet. Reading array-shaped `text` and `content` is the right narrow solution, but the implementation should bound collection before `join`, `extractCanvasFromText`, `formatToolOutputForSidebar`, and Lit rendering.
