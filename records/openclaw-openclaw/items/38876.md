## Summary

Keep open: current main and the latest release already cover the original `content[0].text` report, but this PR still proposes broader array-shape extraction and is not merge-ready because real behavior proof is missing and the new collection path needs bounds.

## What This Changes

The PR expands Control UI tool-card extraction for array-shaped tool result `text` and `content` payloads, adds node regression coverage, and adds a changelog entry.

## Best Possible Solution

Keep the shipped `content[0].text` fix, and merge broader array-shape support only if it is bounded, covered by focused tests, and backed by redacted real Control UI/WebSocket proof.

## Reproduction Assessment

Yes for the PR blocker: source inspection of PR head shows array fragments are joined without bounds before preview parsing and rendering. The original `content[0].text` bug is not a current-main repro because current main and `v2026.5.12` already extract content-array text.

## Solution Assessment

No, not as currently proposed. Reading array-shaped tool output belongs in the tool-card path, but this patch should bound collection and include real Control UI/WebSocket proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Bound new array tool-output extraction:** `ui/src/ui/chat/tool-cards.ts:48-60`
  - body: The PR expands extraction to array-shaped `text`/`content` fragments and joins every collected string before preview parsing and sidebar rendering. A provider-influenced payload can allocate very large strings and force expensive parsing, so cap fragment count/total characters and return a visible truncation marker.
