## Summary

Keep this PR open because it still carries useful broader array-shaped tool-result handling that is not fully on current main, but the current diff should not merge until the new unbounded join is capped and real Control UI/WebSocket proof is added.

## What This Changes

The PR updates Control UI tool-card extraction to join text fragments from array-shaped tool result `text` and `content` payloads, adds node coverage, and adds a changelog entry.

## Best Possible Solution

Merge bounded array-shape support only after fragment/character caps, focused regression coverage, and redacted real Control UI/WebSocket proof; keep the shipped `content[0].text` behavior as-is meanwhile.

## Reproduction Assessment

Yes for the PR blocker: source inspection of PR head shows accepted array fragments are joined without explicit bounds before preview parsing and rendering. The original `content[0].text` bug is not a current-main reproduction because current main and `v2026.5.12` already extract content-array text.

## Solution Assessment

No, not as currently proposed. Reading array-shaped tool output belongs in the tool-card path, but this patch should bound collection/truncation and include real Control UI/WebSocket proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

- **[P2] Bound new array tool-output extraction:** `ui/src/ui/chat/tool-cards.ts:48-60`
  - body: The new array branches collect every accepted `text`/`content` fragment and join the full result before preview parsing and rendering.
