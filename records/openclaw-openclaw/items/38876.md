## Summary

Keep this PR open: it still carries useful Control UI array-tool-result work beyond current main, but it is not merge-ready because the new array collection is unbounded and the thread still lacks real after-fix UI/WebSocket proof.

## What This Changes

The PR changes Control UI tool-card extraction to join text fragments from array-shaped tool-result `text` and `content` payloads, adds node tests, and adds a changelog entry.

## Best Possible Solution

Keep the useful array extraction direction, but cap fragment count and total output characters before joining, mark truncation, retain the focused regression tests, and add redacted real Control UI/WebSocket proof before merge.

## Reproduction Assessment

Yes for the blocking review finding: PR head source shows the new array paths collect all matching fragments and join them before preview parsing and rendering. The original `content[0].text` report is not a high-confidence current-main failure because current main already extracts object text blocks from `content` arrays.

## Solution Assessment

No, not yet. Reading array-shaped tool output is the right owner-path fix, but the PR needs bounded collection and real behavior proof before it is the narrowest maintainable solution.
