## Summary

Keep open: current main covers the linked `content[0].text` bug, but this PR still proposes broader array-fragment extraction and has a concrete unbounded-join security finding plus missing real behavior proof.

## What This Changes

The PR extends Control UI tool-card extraction to join array-shaped tool-result `text` and `content` fragments, adds node tests, and adds a changelog entry.

## Best Possible Solution

Bound fragment count and total output characters in the array extraction path, preserve truncation in tool-card output, keep the focused tests, and require redacted real Control UI/WebSocket proof before merge.

## Reproduction Assessment

Yes for the blocking PR defect: source inspection shows array-shaped fragments are collected and joined before preview/sidebar parsing without limits. The original `content[0].text` report is not a current-main repro because main already handles `content` text blocks.

## Solution Assessment

No, not yet. Reading array-shaped tool output belongs in the tool-card path, but the implementation should bound collection and include real behavior proof before it is merge-ready.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Bound array tool-output collection:** `ui/src/ui/chat/tool-cards.ts:49-60`
  - body: The new array extraction path collects every string fragment and joins the full output before `extractToolPreview()` and sidebar formatting. A provider-influenced payload with many fragments or huge strings can allocate a very large browser string and drive expensive JSON parsing/rendering; cap fragment count and total characters, and preserve a visible truncation marker, before returning `outputText`.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The PR introduces a client-side resource-exhaustion risk by joining provider-influenced array-shaped tool output without explicit bounds.

Concerns:

- **[medium] Unbounded tool-result fragment joining:** `ui/src/ui/chat/tool-cards.ts:49`
  - body: `collectTextParts()` walks every array entry and callers join all collected strings into one `outputText`, which is then parsed for previews and rendered. Very large arrays or fragments can force large allocations and expensive browser work, so explicit limits are needed.
  - confidence: 0.86
