## Summary

Keep open. Current main still has the reported Control UI gap: embedded `tool_result` blocks only extract string-valued `text` or `content`, while the PR adds the missing array-shaped extraction and tests. The branch is still useful, but it should not merge as-is because the new array path joins all fragments without a fragment-count or total-character bound.

## What This Changes

This PR updates `ui/src/ui/chat/tool-cards.ts`, its node tests, and the changelog so tool cards join readable string fragments from array-shaped `text` and `content` tool-result payloads.

## Best Possible Solution

Keep this PR open as the canonical implementation candidate, amend the array extraction to cap fragment count and total collected characters with clear truncation behavior, keep the regression tests and changelog credit, then land after normal maintainer/security review and changed-surface validation.

## Reproduction Assessment

Yes. A source-level reproduction is clear: pass an assistant message with a `tool_result` item whose `content` is an array containing `{ text: "..." }`; current main calls `extractToolText`, returns `undefined`, and the sidebar falls back to `No output`.

## Solution Assessment

No, not as currently proposed. Supporting array-shaped tool-result text is the right direction, but the safest maintainable fix should bound collection before joining and before downstream preview/JSON parsing.

## Review Findings

Overall correctness: patch is incorrect

- **[P2] Bound array tool-output collection:** `ui/src/ui/chat/tool-cards.ts:51`
  - body: The new array path walks every entry and joins every string into one `outputText` before preview extraction and rendering.
