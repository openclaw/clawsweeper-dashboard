## Summary

Keep open: the underlying Control UI display bug still exists on current main, but this PR head still has an unbounded client-side collection path that prior security review also flagged.

## What This Changes

The PR changes Control UI tool cards to read and join array-shaped tool-result `text`/`content`, adds node coverage for array and empty-array cases, and adds a changelog entry.

## Best Possible Solution

Bound fragment count and total output characters with truncation before preview/sidebar/rendering, then merge the array-output display fix with regression coverage and changelog credit intact.

## Reproduction Assessment

Yes, from source. Current main returns only string-valued `text`/`content`, so `content: [{ text: "ok" }]` produces no output text and the existing UI fallback says `No output`.

## Solution Assessment

No, not yet. Reading array-shaped `text` and `content` is the right narrow fix, but the PR should bound collection before joining and downstream client processing.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Bound array tool-output collection:** `ui/src/ui/chat/tool-cards.ts:51-60`
  - body: The new array path collects every string fragment and joins the full result before preview extraction, sidebar formatting, and rendering. Tool results can be provider-influenced, so a huge array or huge fragments can hang the Control UI or crash the tab; cap fragment count and total characters, and mark truncation, before joining.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The PR still has a concrete client-side resource-exhaustion risk from joining unbounded array-shaped tool-result fragments.

Concerns:

- **[medium] Unbounded tool-result fragment joining:** `ui/src/ui/chat/tool-cards.ts:51`
  - body: `collectTextParts` walks every array entry and the caller joins every collected string into one `outputText`; that full string is then parsed for previews, formatted for the sidebar, and rendered. Very large arrays or fragments can force large allocations and expensive browser work, so explicit size limits are needed before merge.
  - confidence: 0.84
