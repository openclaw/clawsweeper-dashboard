## Summary

Keep this PR open, but do not merge it as-is: the branch still adds unique Control UI extraction behavior beyond current main, while the current head joins provider-influenced array fragments without size bounds and lacks after-fix real behavior proof.

## What This Changes

The PR adds Control UI extraction for array-shaped tool-result `text` and `content` payloads, focused node coverage, and a changelog entry.

## Best Possible Solution

Keep the useful array extraction direction, but bound fragment count and total output characters, mark truncation, retain focused regression tests, and add real after-fix UI or runtime proof before merge.

The original `content[0].text` report is not a high-confidence current-main failing path because current main already extracts object text blocks.

## Solution Assessment

No, not yet. Reading array-shaped tool output is the right owner-path fix, but the PR should bound collection and provide real behavior proof before it is the narrowest maintainable solution.
