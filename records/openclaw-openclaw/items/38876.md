## Summary

Keep open: current main still drops array-shaped tool-result output, and this PR targets the right Control UI path, but the latest diff still needs bounded extraction before merge.

## What This Changes

This PR updates Control UI tool cards to join array-shaped `text` and `content` tool-result fragments, adds node coverage for array and empty-array cases, and adds a changelog entry for the linked display fix.

## Best Possible Solution

Amend the PR to cap fragment count and total collected characters before joining, mark truncation, and cover the bounded behavior in the existing tool-card node tests.

## Reproduction Assessment

Yes from source, though not executed live in this read-only review. Current `main` returns only string-valued `item.text` or `item.content`, so `content: [{ text: "ok" }]` produces `undefined` and the UI uses its `No output` fallback.

## Solution Assessment

No, not yet. Reading array-shaped `text` and `content` is the right narrow owner-path fix, but the implementation should bound collection before preview parsing, sidebar formatting, and Lit rendering.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Bound array tool-output collection:** `ui/src/ui/chat/tool-cards.ts:51-60`
  - body: The new array path collects every string fragment and joins the full result before preview extraction, sidebar formatting, and rendering. Tool results can be provider-influenced, so a huge array or huge fragments can hang the Control UI or crash the tab; cap fragment count and total characters, and mark truncation, before joining.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The PR still has a concrete client-side resource-exhaustion risk from joining unbounded array-shaped tool-result fragments.

Concerns:

- **[medium] Unbounded tool-result fragment joining:** `ui/src/ui/chat/tool-cards.ts:51`
  - body: `collectTextParts` walks every array entry and the caller joins every collected string into one `outputText`; that full string is then parsed for canvas previews, formatted for the sidebar, and rendered. A very large array or large fragments can force large allocations and expensive browser work, so this needs explicit size limits before merge.
  - confidence: 0.84
