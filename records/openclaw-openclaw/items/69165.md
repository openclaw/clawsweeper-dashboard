## Summary

Keep open. Current main has adjacent Telegram protections, including token-scoped throttling and retry_after-aware per-call retry, but it still lacks the requested outbound per-chat/topic pacing queue, queue depth/drop policy, non-blocking submit semantics, and documented outboundQueue configuration.

## What This Changes

The issue requests a Telegram outbound send queue that serializes sends per chat/topic, honors retry_after, adds exponential backoff, and exposes configurable queue depth/drop behavior.

## Best Possible Solution

Define a Telegram plugin-owned outbound pacing queue keyed by account plus chat/topic, route all Telegram outbound send paths through it, and document/test retry_after, backoff, depth/drop, and delivery-result semantics.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request. Source inspection gives high confidence that the requested queue/config surface is still absent while adjacent retry and throttling pieces exist.

## Solution Assessment

Unclear. Per-chat/topic pacing is plausible, but non-blocking submit, queue depth/drop policy, and batching interaction need a maintainer-approved delivery contract before implementation.
