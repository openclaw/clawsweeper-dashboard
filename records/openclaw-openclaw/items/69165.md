## Summary

Keep open. Current main has adjacent Telegram protections, including retry_after-aware per-call retry, a shared durable delivery queue, and bot-token-scoped grammY throttling, but it still lacks the requested outbound per-chat/topic pacing queue, max-depth/drop policy, non-blocking submit semantics, and documented outboundQueue configuration.

## What This Changes

Request an outbound Telegram send queue that serializes per chat/topic, respects retry_after with backoff, exposes queue/rate configuration, and supports best-effort queue depth behavior.

## Best Possible Solution

Define a Telegram plugin-owned outbound pacing queue keyed by account plus chat/topic, route all Telegram outbound send paths through it, and document/test retry_after, backoff, depth/drop, and delivery-result semantics.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request. Source inspection gives high confidence that current main lacks the requested queue/config surface while adjacent retry and token-throttling pieces exist.

## Solution Assessment

Unclear. Per-chat/topic pacing is plausible, but non-blocking submit, drop policy, queue depth, priority, and batching interaction need a maintainer-approved delivery contract before implementation.
