## Summary

Keep open. Current main has meaningful partial coverage for Telegram bursts, including shared token-scoped grammY throttling, retry_after-aware per-call retries, and a merged fair group-topic queue, but it still lacks the requested non-blocking outboundQueue contract, queue-depth/drop policy, operator-facing config, and cron delivery-result semantics.

## What This Changes

The issue requests a Telegram outbound send queue that serializes sends per chat/topic, respects retry_after with backoff, exposes queue-depth/rate config, and avoids marking completed cron work failed solely because outbound delivery was rate-limited.

## Best Possible Solution

Keep the existing token-scoped grammY throttler and fair group-topic queue as the base, then design a Telegram-owned outboundQueue contract for non-blocking drain, configurable depth/drop, retry_after/backoff, priority, and cron outcomes, coordinated with https://github.com/openclaw/openclaw/issues/42246.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature request. Source inspection and the issue discussion give high confidence that current main has partial throttling/backoff but not the requested outbound queue contract.

## Solution Assessment

Unclear. The existing grammY throttler, retry_after-aware retry, and fair queue are the right foundation, but the requested non-blocking submit, drop policy, configurable limits, and cron result semantics need maintainer design before implementation.
