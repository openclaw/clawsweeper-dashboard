## Summary

Keep open. Current main has adjacent Telegram protections, including per-call retry_after handling, durable delivery replay, and bot-token-scoped grammY throttling, but it still lacks the requested outbound per-chat/topic pacing queue, queue depth/drop policy, non-blocking submit contract, and documented outboundQueue configuration.

## What This Changes

Request an outbound Telegram send queue that serializes sends per chat/topic, honors Telegram 429 retry_after backoff, adds tunable rate/depth/drop config, and avoids cron delivery bursts failing completed work.

## Best Possible Solution

Define a Telegram plugin-owned outbound pacing queue keyed by account plus chat/topic, route all Telegram outbound send paths through it, and document/test retry_after, backoff, depth/drop, priority, and delivery-result semantics.
