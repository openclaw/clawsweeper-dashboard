## Summary

Keep open. Current main has useful partial protections for Telegram sends, including shared token-scoped grammY throttling and retry_after-aware per-call retry, but it still lacks the requested outbound per-chat/topic pacing queue with queue depth/drop policy, non-blocking submit semantics, and documented outboundQueue configuration.

## What This Changes

The issue requests a Telegram outbound transport queue that serializes sends per chat/topic, respects retry_after on 429s, adds exponential backoff, and exposes queue/rate/depth configuration.

## Best Possible Solution

Define a Telegram plugin-owned outbound pacing queue keyed by account plus chat/topic, route all Telegram outbound send paths through it, and document/test retry_after, backoff, depth/drop, and delivery-result semantics before implementation.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request. Current-main source inspection gives high confidence that the requested queue/config surface is still absent while adjacent retry/throttling pieces exist.

## Solution Assessment

Unclear. Per-chat/topic pacing is a plausible direction, but the requested non-blocking submit, queue depth/drop policy, and batching interaction need a maintainer-approved delivery contract before implementation.
