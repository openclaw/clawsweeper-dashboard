## Summary

Keep open. Current main has adjacent Telegram retry_after handling, token-scoped grammY throttling, and a recent per-group/topic fairness queue, but it still does not provide the requested configurable, non-blocking per-chat outbound queue with depth/drop policy and delivery-result semantics.

## What This Changes

The issue requests a Telegram outbound send queue with per-chat/topic pacing, retry_after-aware backoff, configurable rate/depth limits, and non-blocking best-effort delivery behavior.

## Best Possible Solution

Define a Telegram-owned outbound pacing contract keyed by account plus chat/topic that composes with the existing token throttler and durable delivery queue, then add config schema, docs, focused tests, and live Telegram burst proof.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature/config request. Source inspection gives high confidence that the requested queue/depth/drop configuration and non-blocking delivery semantics are still absent on current main.

## Solution Assessment

Unclear. Per-chat/topic pacing is plausible, but the remaining non-blocking submit, drop policy, priority, durability, and batching interactions need a maintainer-approved delivery contract before implementation.
