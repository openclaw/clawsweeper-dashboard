## Summary

Keep open. Current main has retry_after-aware per-call Telegram retries and a crash-recovery write-ahead delivery queue, but it still does not implement the requested per-chat/topic outbound pacing queue, queue-depth/drop policy, non-blocking submit contract, or Telegram outboundQueue config surface.

## What This Changes

The issue requests a Telegram outbound send queue with per-chat throttling, retry_after-aware backoff, configurable queue limits, and non-blocking delivery semantics.

## Best Possible Solution

Design a Telegram plugin-owned outbound pacing layer keyed by account plus chat/topic, route all Telegram outbound sends through it, and document/test the accepted config, depth/drop, retry_after, and delivery-result semantics alongside #74508 and #42246.

## Reproduction Assessment

Yes for the missing behavior: current main statically shows direct awaited Telegram sends, per-call retry_after retry, and a recovery-only queue with no outbound pacing config. A live Telegram burst reproduction was not run in this read-only review.

## Solution Assessment

Unclear. Per-chat/topic pacing is the right direction, but the exact queue contract, non-idempotent retry behavior, drop policy, and composition with #74508/#42246 need maintainer design before a narrow implementation is safe.
