## Summary

Keep open. Current main has meaningful partial coverage through token-scoped grammY throttling, retry_after-aware per-call retries, and a merged forum-topic fair queue, but it still lacks the requested Telegram outboundQueue contract for non-blocking submission, queue depth/drop policy, operator-tunable per-chat limits, and cron delivery-result semantics.

## What This Changes

Add a Telegram outbound send queue with per-chat rate limiting, retry_after-aware backoff, configurable queue limits, and best-effort delivery behavior for bursty cron/agent notifications.

## Best Possible Solution

Keep the current token-scoped grammY throttling and forum-topic fair queue as the base, then design a Telegram-owned outboundQueue contract for non-blocking drain, configurable depth/drop, retry_after/backoff, priority, and cron delivery outcomes coordinated with https://github.com/openclaw/openclaw/issues/42246.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature request. Source inspection confirms current main has partial throttling/backoff but not the requested outboundQueue config, non-blocking drain, or depth/drop contract.

## Solution Assessment

Unclear. The existing grammY throttler, retry_after-aware retry, and fair group-topic queue are the right foundation, but non-blocking submit, drop policy, configurable limits, and cron result semantics need maintainer design before implementation.
