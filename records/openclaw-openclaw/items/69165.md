## Summary

Keep open. Current main has useful Telegram throttling and retry coverage, but the requested non-blocking outbound queue contract, configurable queue depth/drop policy, and cron delivery-result semantics are still unresolved product work.

## What This Changes

Request an outbound Telegram per-chat send queue that rate-limits sends, honors retry_after/backoff, and exposes queue-depth and best-effort delivery configuration.

## Best Possible Solution

Keep the current token-scoped throttling and fair queue as the base, then design a Telegram-owned outboundQueue contract for non-blocking drain, configurable depth/drop, retry_after/backoff, priority, and cron delivery outcomes coordinated with https://github.com/openclaw/openclaw/issues/42246.
