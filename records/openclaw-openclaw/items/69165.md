## Summary

Keep open. Current main has adjacent Telegram protections, including bot-token-scoped grammY throttling and retry_after-aware per-call retry, but it still lacks the requested outbound per-chat/topic pacing queue, queue depth/drop policy, non-blocking submit semantics, and documented outboundQueue configuration.

## What This Changes

The issue requests a Telegram outbound send queue with per-chat rate limiting, retry_after-aware backoff, configurable queue depth, and non-blocking delivery semantics.

## Best Possible Solution

Define a Telegram plugin-owned outbound pacing queue keyed by account plus chat/topic, route all Telegram outbound send paths through it, and document/test retry_after, backoff, depth/drop, and delivery-result semantics.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request. Source inspection gives high confidence that current main lacks the requested queue/config surface while adjacent retry and token-throttling pieces exist.

## Solution Assessment

Unclear. Per-chat/topic pacing is plausible, but non-blocking submit, queue depth/drop policy, and batching interaction need a maintainer-approved delivery contract before implementation.
