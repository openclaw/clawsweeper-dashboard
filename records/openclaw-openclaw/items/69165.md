## Summary

Keep open. Current main has per-call retry_after-aware Telegram retries and inbound per-chat/per-topic processing lanes, but it still lacks the requested outbound per-chat/topic queue, configurable pacing/depth/drop policy, and non-blocking delivery semantics; the related PR mentioned in discussion is explicitly partial and still open.

## What This Changes

Add a Telegram outbound send queue that serializes sends per chat/topic, respects retry_after with backoff, exposes queue/rate/depth config, and avoids marking completed cron work failed solely because delivery is delayed.

## Best Possible Solution

Design and implement a Telegram plugin-owned outbound pacing layer keyed by account plus chat/topic, integrate it with the existing retry_after-aware retry runner, and document/test the queue, depth, drop, and delivery-result semantics alongside #42246 and #74508.

## Reproduction Assessment

Yes for the missing behavior: current main statically shows direct awaited Telegram sends, per-call retry_after handling, and no outbound queue config. This is a feature gap rather than a broken existing contract, so no live failure reproduction was required for the keep-open decision.

## Solution Assessment

Unclear. Per-chat/topic outbound pacing is the maintainable direction, but the exact non-blocking, drop, queue-depth, and cron-result contract needs maintainer design before implementation.
