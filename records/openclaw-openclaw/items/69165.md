## Summary

Keep open. Current main has Telegram retry_after-aware retries for individual API calls and a write-ahead recovery queue, but it still lacks the requested per-chat/topic outbound pacing queue, queue depth/drop policy, configurable outbound queue settings, and non-blocking submit semantics.

## What This Changes

Add a Telegram outbound send queue that serializes sends per chat/topic, honors retry_after on 429s, backs off repeated rate limits, and exposes queue tuning config.

## Best Possible Solution

Implement a Telegram plugin-owned outbound pacing layer keyed by account plus chat/topic, wrap all Telegram outbound send paths through it, reuse the existing retry_after-aware retry runner, and document/test config for limits, depth, dropping, and delivery-result semantics.

## Reproduction Assessment

Yes for the missing behavior: current main statically shows direct awaited Telegram sends, per-call retry_after retry, and recovery-only queueing with no outbound pacing config. A live Telegram burst reproduction was not run in this read-only review.

## Solution Assessment

Unclear. Per-chat/topic pacing is the right direction, but the requested non-blocking/drop semantics and interaction with #42246 need a maintainer-approved contract before implementation.
