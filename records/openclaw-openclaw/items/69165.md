## Summary

Keep open. Current main has retry_after-aware retry for individual Telegram API calls and a shared write-ahead recovery queue, but it still lacks the requested per-chat/topic outbound pacing queue, queue depth/drop config, and non-blocking queue-submit contract. The related #42246 batching request is complementary rather than a canonical duplicate.

## What This Changes

Add a Telegram outbound send queue keyed by chat/topic with per-chat rate limiting, retry_after-aware backoff, configurable queue depth/drop behavior, and best-effort non-blocking delivery semantics.

## Best Possible Solution

Implement a Telegram plugin-owned outbound pacing layer keyed by account/chat/topic that wraps all relevant Telegram outbound paths, reuses the existing retry_after-aware runner, adds documented config and tests for depth/drop behavior, and defines duplicate-safe retry and delivery-result semantics before exposing the feature.

## Reproduction Assessment

Not applicable as a pure live bug reproduction, but there is a high-confidence static verification path. Current main shows direct awaited Telegram sends, per-call retry_after retry, and recovery-only queueing with no per-chat outbound pacing config or queue implementation.

## Solution Assessment

Unclear. Per-chat pacing is the right product direction for this failure mode, but the requested non-blocking/drop semantics and interaction with non-idempotent sends and #42246 need maintainer design before a safe implementation PR.

## Risks / Open Questions

- A naive retry layer for non-idempotent Telegram sends can duplicate visible messages when Telegram received the original request but the client saw an ambiguous failure.
- Non-blocking submit plus max-depth dropping needs a clear delivery-result contract so important cron reports are not silently treated as delivered when still queued or dropped.
- The design should compose with #42246 batching so OpenClaw does not add two competing Telegram outbound buffering layers.
