## Summary

Keep open. Current main has important partial coverage: token-scoped grammY throttling, retry_after-aware per-call retries, and a merged forum-topic fair queue, but it still lacks the requested Telegram outboundQueue contract for non-blocking drain, queue depth/drop policy, operator-tunable settings, and cron delivery-result semantics.

## What This Changes

The issue requests a Telegram-owned outbound queue that rate-limits sends per chat/topic, honors retry_after backoff, and exposes config/depth/drop behavior for high-volume cron and agent bursts.

## Best Possible Solution

Keep the current token-scoped grammY throttling and fair group-topic queue as the base, then design a Telegram-owned outboundQueue contract for non-blocking drain, configurable depth/drop, retry_after/backoff, priority, and cron delivery outcomes, coordinated with https://github.com/openclaw/openclaw/issues/42246.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature request. Source inspection and discussion give high confidence that current main has partial throttling/backoff but not the requested outbound queue contract.

## Solution Assessment

Unclear. The existing grammY throttler and fair queue are the right foundation, but the requested non-blocking submit, drop policy, configurable limits, and cron result semantics need maintainer design before implementation.
