## Summary

Keep open: current main still separates SecretRef-tolerant reachability probes from strict authenticated gateway RPC calls, matching the reported failures, and no merged or viable open PR currently owns the remaining repair.

**Likely related people**
- `joshavant` — high confidence; introduced gateway SecretRef support and fail-closed local-auth behavior.
- `CodeForgeNet` — high confidence; implemented the SecretRef-aware gateway status probe path.

## What This Changes

The issue asks gateway health, channel status, browser doctor, and doctor commands to handle SecretRef-backed gateway credentials consistently instead of reporting a healthy gateway as unavailable.

## Best Possible Solution

Define one shared read-only CLI authentication policy that resolves permitted SecretRefs, restricts unauthenticated degradation to reachability-only probes, and returns a clean actionable diagnostic when an RPC genuinely requires credentials.

## Reproduction Assessment

Yes, at source level with high confidence: current main retains the distinct safe-probe and strict-RPC credential paths that match the reported command split. A live Bitwarden Secrets Manager reproduction was not independently run.

## Solution Assessment

The requested consistency is necessary, but copying the unauthenticated status-probe fallback into every affected command is not yet the best proven fix because authenticated RPC methods have a different security contract.
