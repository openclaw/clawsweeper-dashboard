## Summary

Keep open: current main still has the reported split between safe `gateway status --deep` probe auth and strict Gateway RPC credential resolution for `gateway health`, `channels status`, and browser doctor paths. The bug is source-reproducible, but the right fix needs maintainer choice because it touches SecretRef execution policy and fail-closed Gateway auth behavior.

## What This Changes

The issue reports that SecretRef-backed `gateway.auth.password` makes several read-only CLI diagnostics fail with `GatewaySecretRefUnavailableError` even though `gateway status --deep` can probe the healthy gateway.

## Best Possible Solution

Keep this issue open and align the shared read-only Gateway CLI diagnostic paths so they use SecretRef-aware active-snapshot/probe behavior or a deliberately chosen degraded diagnostic mode without weakening runtime auth.

## Reproduction Assessment

Yes, at source level: the reported commands still flow into strict `callGateway` credential resolution, while `gateway status --deep` uses the safe probe-auth helper. I did not run a live Bitwarden-backed SecretRef environment.

## Solution Assessment

Unclear until maintainers pick the command contract. The safest direction is a shared read-only Gateway credential/diagnostic boundary that preserves fail-closed runtime auth rather than per-command plaintext or silent fallback behavior.
