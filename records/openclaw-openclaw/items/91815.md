## Summary

Keep open: current main still has the reported split between safe probe auth for gateway status and strict gateway RPC credential resolution for the affected read-only CLI paths, with no open or merged PR specifically owning this issue.

## What This Changes

The issue reports that SecretRef-backed gateway credentials make selected gateway-dependent CLI commands fail while `gateway status --deep` succeeds.

## Best Possible Solution

Define one gateway SecretRef credential policy for read-only diagnostics, then apply it consistently to gateway health, channels status, and browser doctor with targeted regression coverage.

## Reproduction Assessment

Yes, at source level: current main shows `gateway status --deep` using the safe SecretRef probe resolver while `gateway health`, `channels status`, and `browser doctor` still route through strict gateway RPC credential resolution. I did not run a live Bitwarden-backed setup in this read-only review.

## Solution Assessment

Unclear; making every command silently probe without auth may not be the best fix because gateway auth and exec SecretRefs are security-sensitive. The maintainable direction is a maintainer-approved read-only diagnostic contract plus focused tests for strict and degraded paths.
