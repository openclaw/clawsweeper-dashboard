## Summary

Keep open: current main still has strict Gateway RPC command paths that can surface `GatewaySecretRefUnavailableError` for SecretRef-backed gateway auth while the status/deep probe path deliberately uses a safe degraded resolver.

## What This Changes

SecretRef-backed gateway auth should behave consistently across gateway-dependent CLI diagnostics so a healthy local gateway does not appear unhealthy.

## Best Possible Solution

Define one maintainer-approved contract for read-only gateway diagnostic commands, then update gateway health, channels status, browser doctor, and doctor paths to use it without weakening strict send/auth RPC behavior.

## Reproduction Assessment

Yes, source-reproducible: an unresolved active `gateway.auth.password` SecretRef reaches strict `callGateway` in the reported CLI paths, while status/deep uses a safe probe resolver. I did not run a live Bitwarden reproduction.

## Solution Assessment

Unclear as a direct fix because the issue discussion lists several valid behaviors with different security tradeoffs. The best fix is a maintainer-approved read-only command contract plus focused regression coverage for each affected command path.
