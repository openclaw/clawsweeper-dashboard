## Summary

Keep open: current main still has strict gateway RPC paths that can surface `GatewaySecretRefUnavailableError` for SecretRef-backed gateway auth, while `gateway status --deep` uses the safe probe-auth path and can report the gateway healthy.

## What This Changes

The issue reports inconsistent SecretRef-backed gateway auth handling between `gateway status --deep` and other gateway-dependent CLI diagnostics such as gateway health, browser doctor, and channels status.

## Best Possible Solution

Maintainers should define and implement one read-only gateway diagnostic contract that keeps runtime auth fail-closed, preserves SecretRef safety, and makes health/channel/browser diagnostics consistent with `gateway status --deep`.

## Reproduction Assessment

Yes, at source level: the reported commands still flow through strict `callGateway` credential resolution that can throw on active `gateway.auth.password` SecretRefs, while `gateway status --deep` uses the safe probe-auth helper. I did not run a live Bitwarden SecretRef environment in this read-only review.

## Solution Assessment

Unclear until maintainers choose the intended SecretRef contract for these read-only diagnostics. The best direction is not plaintext fallback, but a narrow shared diagnostic path that preserves fail-closed runtime auth while degrading consistently for status-style checks.
