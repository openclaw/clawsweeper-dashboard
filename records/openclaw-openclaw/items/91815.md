## Summary

Keep open: current main still has gateway-dependent CLI paths that can throw or surface `GatewaySecretRefUnavailableError` for SecretRef-backed `gateway.auth.password`, while the `gateway status --deep` path uses a safe probe-auth resolver that can continue diagnostics.

## What This Changes

This issue reports that `gateway health`, `browser doctor`, and `channels status` handle SecretRef-backed gateway password auth less gracefully than `gateway status --deep`.

## Best Possible Solution

Keep the issue open and align the shared CLI gateway credential path so these diagnostics either use the same SecretRef-aware active-snapshot/probe behavior as `gateway status --deep` or degrade read-only output without misreporting a healthy gateway as unreachable.

## Reproduction Assessment

Yes, at source level: the reported commands flow into `callGateway` credential resolution that can throw on active `gateway.auth.password` SecretRefs, while `gateway status --deep` uses the safe probe-auth helper. I did not run a local Bitwarden SecretRef environment.

## Solution Assessment

The best direction is to fix the shared Gateway CLI credential/diagnostic boundary rather than ask users to switch to plaintext or per-command environment overrides. The exact implementation should preserve fail-closed runtime auth while making read-only diagnostics degrade consistently.
