## Summary

Keep open: current main still has the reported split between safe gateway status probing and strict gateway RPC credential resolution, and the later discussion correctly identifies this as a contract decision for read-only diagnostics rather than a completed fix.

## What This Changes

The issue requests consistent handling of SecretRef-backed `gateway.auth.password` for `gateway health`, `channels status`, and `browser doctor` when `gateway status --deep` can still report the gateway healthy.

## Best Possible Solution

Align the documented read-only diagnostic command behavior across gateway health, channel status, and browser doctor while preserving strict fail-closed SecretRef handling for startup, reload, send, and privileged runtime paths.

## Reproduction Assessment

Yes, at source level: configure an unavailable active `gateway.auth.password` SecretRef and the reported commands still flow through strict `callGateway` credential resolution, while `gateway status --deep` uses the safe probe-auth resolver. I did not run a live Bitwarden-backed environment in this read-only review.

## Solution Assessment

Unclear; no patch is proposed yet, and the best implementation depends on a maintainer decision about degraded read-only probes versus strict errors or active-snapshot credential resolution. A shared gateway CLI/read-only credential boundary is preferable to one-off per-command suppression.
