## Summary

Keep open: the PR targets a real gateway TLS SAN bug that is still present on current main and the latest release, but the current patch can still replace explicit operator-provided cert/key files when they use `CN=openclaw-gateway` and lack SANs.

## What This Changes

The PR adds subjectAltName entries to auto-generated gateway TLS certificates and adds runtime regeneration for older CN-only generated certificates, with regression tests.

## Best Possible Solution

Ship SAN-bearing generated certs, but limit automatic old-cert repair to OpenClaw-owned default generated files or move that repair into a guarded doctor migration.

## Reproduction Assessment

Yes, at source level: current main and `v2026.6.10` still invoke OpenSSL with CN only, and Node's TLS identity check expects IP SANs for IP hostnames. I did not run a full gateway handshake in this read-only review.

## Solution Assessment

No, not yet. Adding SANs in the generator is the right layer, but the automatic upgrade path is too broad for explicit operator certificate paths and the final SAN identity set still needs maintainer confirmation.
