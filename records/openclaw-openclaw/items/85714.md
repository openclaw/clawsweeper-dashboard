## Summary

Keep open: current main and the latest release still suppress ordinary final Codex direct-chat replies when the effective delivery mode is message-tool-only. The remaining question is a maintainer policy decision, because fallback delivery would change intentional tool-only visibility semantics and overlaps the open Codex default PR.

## What This Changes

The issue requests a fallback that delivers a final Codex agent message when the model forgets to call the configured message delivery tool.

## Best Possible Solution

Have maintainers choose the Codex/source visible-reply contract first, then either land the automatic-default path or implement a shared, scoped fallback/warning policy with duplicate-delivery and privacy tests.

## Reproduction Assessment

Yes, at source level: current main defaults Codex direct/source replies to message-tool-only delivery and suppresses ordinary final payloads when no message-tool send happened. I did not run a live Telegram/Codex reproduction in this read-only review.

## Solution Assessment

Unclear: the requested fallback would heal the observed silence, but it changes documented message-tool-only privacy semantics. The maintainable path is a maintainer-approved contract, not an automatic repair in one branch.
