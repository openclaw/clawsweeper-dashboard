## Label Justifications

- `impact:message-loss`: The reported final agent response is generated and recorded internally but not delivered to the originating channel.

## Summary

Keep open: current main and the latest release still make Codex direct/source replies message-tool-only by default, and ordinary final assistant text is suppressed when no message-tool send occurs. The report is source-reproducible, but the requested fallback changes documented message-tool-only visibility semantics and overlaps with active maintainer/product decisions in https://github.com/openclaw/openclaw/pull/83814 and https://github.com/openclaw/openclaw/issues/85384.

## What This Changes

The issue requests a fallback that delivers a non-sentinel final agent message to the originating channel when an agent configured for message-tool delivery completes without calling the delivery tool.

## Best Possible Solution

Maintainers should choose the visibility contract first: either land the automatic Codex direct-chat default from https://github.com/openclaw/openclaw/pull/83814, or design one shared fallback that excludes NO_REPLY, room events, duplicate message-tool sends, and privacy-sensitive tool-only configurations.

## Reproduction Assessment

Yes, at source level: current main defaults Codex direct/source chats to message-tool delivery, preserves final agentMessage text as assistant text, and suppresses ordinary final payload delivery under message_tool_only. I did not run a live Telegram/Codex reproduction in this read-only review.

## Solution Assessment

Unclear: the requested fallback would heal the observed silent reply, but it changes documented message-tool-only privacy semantics. The safer path is a maintainer decision between the open automatic-default PR and a deliberately scoped fallback policy.
