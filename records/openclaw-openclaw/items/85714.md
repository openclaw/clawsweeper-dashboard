## Summary

Keep open: current main and v2026.5.22 still have a source-reproducible path where Codex direct/source chats default to message-tool-only visible delivery, preserve ordinary final assistant text, and suppress that final text when no message-tool send happened. The remaining fix overlaps with the open Codex default PR at https://github.com/openclaw/openclaw/pull/83814 and the broader message-tool-only fallback policy in https://github.com/openclaw/openclaw/issues/85384, so this needs maintainer product review rather than closure or a new automated repair.

## What This Changes

The issue requests a fallback that delivers Codex final agent text or `task_complete.last_agent_message` to the originating channel when the configured message tool was not called and the final text is not the silent sentinel.

## Best Possible Solution

Maintainers should decide the visible-reply contract first: either land the automatic Codex default from https://github.com/openclaw/openclaw/pull/83814 or design one shared fallback/warning policy that excludes silent tokens, room events, duplicate message-tool sends, and privacy-sensitive tool-only configurations.

## Reproduction Assessment

Yes, at source level: current main defaults Codex direct/source chats to message-tool delivery, preserves final assistant text and message-tool telemetry, and suppresses ordinary final payloads under `message_tool_only`. I did not run a live Telegram/Codex reproduction in this read-only review.

## Solution Assessment

Unclear: the requested fallback would heal the observed silence, but it changes the documented privacy semantics of message-tool-only delivery. The narrower maintainable path is a maintainer decision between the open automatic-default PR and a deliberately scoped fallback/warning policy.
