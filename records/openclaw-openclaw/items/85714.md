## Summary

Keep open: current main still has a source-reproducible path where Codex direct/source chats default to message-tool-only visible delivery and an ordinary final assistant message is suppressed when no message-tool send happened. The requested fallback overlaps with the open direct-chat default PR at https://github.com/openclaw/openclaw/pull/83814 and the message-tool-only policy discussion in https://github.com/openclaw/openclaw/issues/85384, so this needs maintainer product review rather than a new automated repair.

## What This Changes

The issue requests fallback delivery of a non-silent final Codex `agent_message` when the agent completes without invoking the configured `message(action="send")` tool.

## Best Possible Solution

Decide the Codex direct-chat contract first: either land the automatic-default approach from https://github.com/openclaw/openclaw/pull/83814 or design a narrowly scoped fallback that excludes silent tokens, room events, duplicate message-tool sends, and privacy-sensitive tool-only configurations.

## Reproduction Assessment

Yes, at source level: current main defaults Codex direct/source chats to message-tool delivery, preserves final `agentMessage` text, and then suppresses ordinary final payloads under `message_tool_only` unless a bypass marker is present. I did not run a live Telegram/Codex reproduction in this read-only review.

## Solution Assessment

Unclear: the requested fallback would heal the observed silence, but it changes documented message-tool-only privacy semantics. The safer maintainable path is a maintainer decision between the open automatic Codex-default PR and a deliberately scoped fallback policy.
