- **[P1] Do not touch Matrix bindings during route lookup:** `extensions/matrix/src/matrix/monitor/route.ts:116-119`
  - body: `resolveMatrixInboundRoute()` runs before the no-mention room-message drop path, and the shared resolver touches a matched binding immediately. Current Matrix code intentionally waits until the accepted handler path before touching `_runtimeBindingId`, with coverage that dropped room messages do not refresh bindings; this change would keep idle Matrix thread bindings alive from ignored chatter. Keep a no-touch lookup here or defer the touch until the message is accepted.
  - confidence: 0.92
- **[P1] Preserve runtime-bound Matrix thread sessions:** `extensions/matrix/src/matrix/monitor/route.ts:145`
  - body: After `resolveRuntimeConversationBindingRoute()` rewrites `route` to the persisted binding target, this branch still runs for any `threadId` because it only checks configured bindings. A bound Matrix thread that currently records `agent:bound:session-1` would be rewritten to a suffixed thread key, so replies miss the bound session; skip thread isolation when a runtime route has a bound session key.
  - confidence: 0.9
- **[P2] Cover omitted-account room bindings:** `extensions/matrix/src/matrix/monitor/route.test.ts:34`
  - body: The new room-binding coverage goes through `matrixBinding()`, which hard-codes `match.accountId: "ops"`; the linked report's config omits `match.accountId`, and current routing treats omission as the default account rather than the named Matrix account. This proves an explicit-account path while leaving the reported per-room binding miss unresolved, so add Matrix-specific omitted-account coverage and the corresponding fix or narrow the PR scope.
