## Summary

Keep open: current `main` has better source-config preservation and config-audit plumbing, but the restart half of the report is still present in source and the full background-write path has not been proven fixed or superseded.

## What This Changes

The issue asks OpenClaw to stop background gateway logic from writing `memory.qmd.update` defaults into user config and to stop auto-restarting the gateway after such config changes without explicit user authorization.

## Best Possible Solution

Keep authored config mutations explicit and operator-visible, keep QMD defaults runtime-only or doctor-recommended, and replace reload-triggered hard restarts with a clear pending-restart/opt-in path for sensitive config changes.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not for the full background-write report on current `main`; source inspection does show that a `memory.qmd.update.*` config diff still follows the restart-required path.

## Solution Assessment

Is this the best way to solve the issue? Unclear pending maintainer decision: the direction is aligned, but maintainers should choose between no autonomous writes, a pending-restart surface, QMD-specific reload rules, and any explicit opt-in config before implementation.
