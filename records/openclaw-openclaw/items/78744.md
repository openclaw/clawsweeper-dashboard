## Summary

Keep open: the latest PR head has blocking ACPX lifecycle regressions despite useful related issue context and sufficient real behavior proof.

## What This Changes

The PR adds ACPX process leases, safer wrapper/reaper cleanup, requester-owned ACP session visibility, isolated Codex trust config handling, docs, changelog entries, and tests.

## Best Possible Solution

Keep the lease-backed cleanup design, but keep lease identity out of acpx’s stable persisted command identity, cancel forced-kill timers safely, and run both lease and live orphan startup cleanup inside the acpx plugin boundary.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: acpx 0.6.1 compares persisted agentCommand exactly, this PR injects a fresh lease command per ensureSession, the wrapper leaves forceKillTimer pending after orphan child exit, and service.start never calls the new orphan reaper.

## Solution Assessment

No. The overall lease-backed cleanup direction is maintainable, but this implementation needs the command identity, forced-kill timer, and startup orphan-reap gaps fixed before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Keep lease metadata out of acpx command identity:** `extensions/acpx/src/runtime.ts:616-620`
  - body: `commandWithLaunchLease` appends a fresh lease id to the command that acpx resolves during every `ensureSession`. acpx 0.6.1 reuses persistent sessions only when the stored `agentCommand` exactly equals the current command, so a second ensure for the same persistent session will create a new backend session instead of reusing the existing one. Keep the lease identity out of the command acpx persists or normalize it before acpx compares reuse keys.
  - confidence: 0.94
- **[P1] Clear forced kills when orphaned children exit:** `extensions/acpx/src/codex-auth-bridge.ts:291-292`
  - body: When orphan cleanup has started, the child `exit` handler returns before clearing `forceKillTimer`. If the child exits after SIGTERM, the timer can still fire 1.5 seconds later and signal `-child.pid`; at that point the process group id may have been reused, so cleanup can target an unrelated process group. Clear the timer and exit the wrapper once the orphaned child is gone.
  - confidence: 0.9
- **[P2] Run the orphan reaper during startup:** `extensions/acpx/src/service.ts:286-290`
  - body: Startup currently calls only `reapOpenAcpxProcessLeases`, so wrappers that are already orphaned but do not have one of the new lease records are never considered. That leaves the `reapStaleOpenClawOwnedAcpxOrphans` implementation and its test unused, and it undercuts the PR’s startup-reap behavior for existing stale ACPX processes. Wire the live orphan scan into service startup or narrow the documented/changelog claim.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The diff introduces a concrete process-safety concern: orphan cleanup can still send a delayed SIGKILL to a reused process group.

Concerns:

- **[medium] Delayed process-group SIGKILL can target a reused PGID:** `extensions/acpx/src/codex-auth-bridge.ts:291`
  - body: The generated wrapper keeps the forced SIGKILL timer active after the child exits during orphan cleanup. Because the callback later signals `-child.pid`, a fast PID/PGID reuse can make OpenClaw terminate an unrelated process group, which is an availability and host-safety regression in this cleanup hardening path.
  - confidence: 0.9
