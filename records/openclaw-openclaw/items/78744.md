## Summary

Keep open: the latest head still has three blocking lifecycle defects in ACPX command identity, wrapper cleanup, and startup orphan reaping. The PR has useful related work and sufficient real-behavior proof, but the patch is not ready to merge.

## What This Changes

The PR adds ACPX lease-backed process cleanup, generated wrapper reaping, Codex trust config propagation, requester-owned ACP session visibility changes, docs, changelog entries, and tests.

## Best Possible Solution

Keep the lease-backed cleanup direction, but revise it so acpx sees a stable command identity, wrapper forced-kill timers are cancelled on child exit, and startup cleanup covers both lease records and legacy OpenClaw-owned orphans.

## Reproduction Assessment

Yes. Source inspection against latest head and acpx 0.6.1 gives high-confidence paths for broken persistent reuse, a lingering forced-kill timer, and a startup orphan reaper that is never called.

## Solution Assessment

No. The lease-backed design is a maintainable direction, but this implementation needs the command identity, timer cleanup, and startup reaping gaps fixed before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Keep lease metadata out of acpx command identity:** `extensions/acpx/src/runtime.ts:616-620`
  - body: `commandWithLaunchLease` appends a fresh lease id and gateway id to the command passed through acpx on each ensure. acpx 0.6.1 reuses persistent sessions only when the stored `agentCommand` exactly equals the current command, so the next ensure for the same persistent session will create a new backend session instead of reusing the existing one.
  - confidence: 0.94
- **[P1] Clear forced kills when orphaned children exit:** `extensions/acpx/src/codex-auth-bridge.ts:291-292`
  - body: When orphan cleanup has started, the child `exit` handler returns before clearing `forceKillTimer`. If the child exits after SIGTERM, the timer can still fire later and signal `-child.pid`, which may target a reused process group.
  - confidence: 0.9
- **[P2] Run orphan reaping during startup:** `extensions/acpx/src/service.ts:286-290`
  - body: Startup only calls `reapOpenAcpxProcessLeases`, so wrappers that are already orphaned but do not yet have lease records are never scanned. That leaves the new live orphan reaper unused and makes the startup cleanup docs/changelog claim incomplete for existing stale ACPX processes.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The diff still has a process-safety regression: orphan cleanup can leave a delayed SIGKILL that may target a reused process group.

Concerns:

- **[medium] Delayed process-group SIGKILL can target a reused PGID:** `extensions/acpx/src/codex-auth-bridge.ts:291`
  - body: The generated wrapper keeps the forced SIGKILL timer active after the child exits during orphan cleanup. Because the callback later signals `-child.pid`, PID or process-group reuse can make OpenClaw terminate an unrelated process group.
  - confidence: 0.9
