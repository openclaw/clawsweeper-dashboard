## Summary

Keep open: current main still has a source-level path where pre-rendered Telegram HTML can re-enter the generic lazy sender without formatting metadata, matching the reported double-render and raw-anchor fallback behavior.

## What This Changes

Telegram cron announce delivery should keep Markdown links clickable in Telegram and avoid exposing literal HTML anchors after parse fallback.

## Best Possible Solution

Carry formatting/textMode and a true plainText fallback through the generic lazy sender boundary, then cover cron Telegram announce and Telegram HTML parse fallback with focused regressions.

## Reproduction Assessment

Yes, at source level: the durable pipeline marks Telegram chunks as HTML, but the lazy runtime sender drops the formatting before calling the Telegram adapter again. I did not run a live Telegram repro in this read-only review.

## Solution Assessment

Yes, the narrow maintainable fix is to preserve the existing formatting contract through src/cli/send-runtime/channel-outbound-send.ts and supply safe plain-text fallback text for HTML-mode sends.
