## Summary

Keep open: current main still has a source-level path where cron direct announce delivery can route Telegram HTML chunks through the CLI lazy runtime sender, which drops the formatting/plain fallback metadata before re-entering the Telegram adapter.

## What This Changes

The issue asks OpenClaw to preserve Telegram formatting for cron announce Markdown links and avoid raw HTML anchor text on parse fallback.

## Best Possible Solution

Carry generic formatting and plain fallback metadata through the lazy channel sender, keep the fix plugin-agnostic, and add regressions for cron Telegram announce links plus HTML parse fallback plain text.

## Reproduction Assessment

Yes, at source level: current main shows cron direct delivery can hand Telegram HTML-formatted chunks to a lazy runtime sender that omits formatting/textMode/plainText before the Telegram adapter handles the text again. I did not run a live Telegram job in this read-only review.

## Solution Assessment

Yes, the issue's suggested direction is the narrow maintainable fix if implemented generically: preserve formatting/plain fallback state through the lazy runtime path and avoid raw rendered HTML on plain retries. Hard-coding Telegram behavior in cron would be a worse boundary.
