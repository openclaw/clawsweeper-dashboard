## Summary

Keep open. Current main and the latest tagged release still intentionally block workspace `OPENCLAW_CONFIG_PATH`; resolving this would change the documented workspace dotenv trust boundary, and the target-repo implementation PR was closed while only an external fork PR remains open.

## What This Changes

The issue requests loading `OPENCLAW_CONFIG_PATH` from a workspace-local `.env` during CLI bootstrap so project-specific config files are used.

## Best Possible Solution

Keep the fail-closed workspace dotenv policy unless owners approve and document a narrow trusted exception; otherwise steer project-specific config to trusted process or global runtime env.

## Reproduction Assessment

Yes, at source level. Put `OPENCLAW_CONFIG_PATH` in a workspace `.env`; `loadWorkspaceDotEnvFile` filters it before `resolveCanonicalConfigPath` can observe the override, so config resolution falls back to the default path.

## Solution Assessment

Unclear. Directly unblocking this key is not clearly the best fix because it relaxes a documented security boundary; the maintainable path is a maintainer-approved exception with matching docs/tests, or continued use of trusted env surfaces.
