## Summary

Keep open. Current `main` still blocks the reported project-local `.env` overrides, the fail-closed behavior is documented and covered by tests, and open PR #57416 remains the active implementation candidate with closing syntax for this issue.

## What This Changes

The issue requests that CLI bootstrap load project-local `.env` values for `OPENCLAW_CONFIG_PATH` and `OPENCLAW_STATE_DIR` so commands resolve the intended project config instead of the default OpenClaw config path.

## Best Possible Solution

Keep this issue open and review #57416 as the implementation candidate. The desired end state should be either a security-reviewed, documented, narrow project-local config selection mechanism with tests, or a clear maintainer answer that workspace `.env` cannot set OpenClaw runtime-control variables and users should use trusted environment surfaces instead.

## Reproduction Assessment

Yes. Current source and regression tests provide a high-confidence reproduction path: put `OPENCLAW_CONFIG_PATH` and `OPENCLAW_STATE_DIR` in a workspace `.env`, load through `loadWorkspaceDotEnvFile` or `loadCliDotEnv`, and those keys are filtered before config path resolution.

## Solution Assessment

Unclear. Allowing both reported keys from workspace `.env` is not clearly the best solution because `OPENCLAW_STATE_DIR` can redirect trusted global dotenv loading; a safer candidate would keep state-dir blocked and only consider a documented, security-reviewed config-path opt-in if maintainers approve the trust-boundary change.
