## Summary

Keep this issue open. Current main and the latest release still block `OPENCLAW_CONFIG_PATH` from workspace `.env`, the current docs/tests treat that as a fail-closed security boundary, and the provided GitHub context lists an open candidate PR for the same requested exception.

## What This Changes

The issue asks OpenClaw to load project-local `.env` early enough for `OPENCLAW_CONFIG_PATH` to select a project config instead of falling back to the default user config.

## Best Possible Solution

Maintainers should decide whether a narrow workspace `OPENCLAW_CONFIG_PATH` exception is acceptable; if yes, refine and land the linked candidate with tests/docs, otherwise document the trusted process or global `.env` path as the supported alternative.

## Reproduction Assessment

Yes from source inspection: workspace `.env` is loaded through `loadWorkspaceDotEnvFile`, and that path currently filters out `OPENCLAW_CONFIG_PATH` before `config file` resolves the active config path. I did not run a live CLI repro because this cleanup review is read-only.

## Solution Assessment

Unclear. Allowing the key would restore the reporter's workflow, but the current tests, docs, and AGENTS.md policy make workspace runtime controls fail-closed, so the best fix needs maintainer security direction before implementation.
