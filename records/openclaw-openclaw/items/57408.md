## Summary

Keep open. Current main and the latest shipped release still intentionally block workspace `OPENCLAW_CONFIG_PATH`, while the open linked fork PR proposes a security-sensitive exception that needs maintainer approval.

## What This Changes

The issue asks OpenClaw to load project-local `.env` entries for `OPENCLAW_CONFIG_PATH` so CLI and gateway commands use a project config instead of the default state config.

## Best Possible Solution

Keep the fail-closed workspace dotenv policy unless maintainers approve a narrow, tested, documented exception or opt-in for project config selection.

## Reproduction Assessment

Yes, at source level. Put `OPENCLAW_CONFIG_PATH` in a workspace `.env`; `loadCliDotEnv` routes it through `loadWorkspaceDotEnvFile`, which blocks the key before `resolveConfigPath` can read it.

## Solution Assessment

Unclear. Allowing the key from workspace `.env` would fix the reported workflow, but it conflicts with current docs, tests, and the fail-closed security boundary unless maintainers approve that exception.
