## Summary

Keep open. Current main still blocks the reported project-local `.env` overrides, and an open PR already references this issue as the implementation candidate; accepting that direction needs maintainer/security judgment because it changes the documented workspace dotenv trust boundary.

## What This Changes

The issue requests that project-local `.env` values for `OPENCLAW_CONFIG_PATH` be respected during CLI bootstrap instead of falling back to the default user config path.

## Best Possible Solution

Keep the issue open until maintainers decide the trust boundary; if project-local config selection is approved, land #57416 or a revised narrow PR with matching tests and docs, otherwise document the supported trusted env paths more clearly.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: put `OPENCLAW_CONFIG_PATH` in a workspace `.env`, run the CLI dotenv path, and the key is filtered before config resolution; I did not execute tests because this review is read-only.

## Solution Assessment

Unclear. The open PR is a plausible narrow implementation for the requested behavior, but it is not clearly the best solution until maintainers approve an exception to the fail-closed `OPENCLAW_*` workspace dotenv policy.
