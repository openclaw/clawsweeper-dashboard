## Summary

Keep open. Current main and the latest shipped release still intentionally block workspace `OPENCLAW_CONFIG_PATH` under the documented fail-closed workspace dotenv boundary, while the linked fork PR proposes a security-sensitive exception that needs maintainer approval.

## What This Changes

The issue asks OpenClaw to load project-local `.env` values for `OPENCLAW_CONFIG_PATH` so CLI and gateway commands use a project config instead of the default user config.

## Best Possible Solution

Keep the fail-closed workspace dotenv policy unless maintainers approve a narrow, tested, documented exception or opt-in; otherwise guide users to trusted process env or global runtime `.env` for this workflow.

## Reproduction Assessment

Yes, at source level. Put `OPENCLAW_CONFIG_PATH` in a workspace `.env`; `loadCliDotEnv` routes it through `loadWorkspaceDotEnvFile`, which blocks `OPENCLAW_*` before `resolveConfigPath` can read the override.

## Solution Assessment

Unclear. Allowing `OPENCLAW_CONFIG_PATH` from workspace `.env` would fix the reported workflow, but it conflicts with current docs/tests and the fail-closed security boundary unless maintainers approve that exception with updated coverage and docs.
