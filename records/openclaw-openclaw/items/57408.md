## Summary

Keep open. Current main and the latest shipped release still intentionally block workspace `OPENCLAW_CONFIG_PATH` under the documented fail-closed workspace dotenv boundary, and an open fork PR still references the issue, so cleanup closure would be premature.

## What This Changes

The issue requests restoring project-local `.env` support for `OPENCLAW_CONFIG_PATH` so CLI and gateway startup use a project-specific config file instead of defaulting to `~/.openclaw/openclaw.json`.

## Best Possible Solution

Keep the issue open until maintainers decide whether to preserve the fail-closed workspace dotenv policy or approve a narrow, tested, documented exception for `OPENCLAW_CONFIG_PATH` only.

## Reproduction Assessment

Yes, at source level. Put `OPENCLAW_CONFIG_PATH` in a workspace `.env`; `loadCliDotEnv` routes it through `loadWorkspaceDotEnvFile`, which blocks it before `resolveConfigPath` can observe the override.

## Solution Assessment

Unclear. Allowing `OPENCLAW_CONFIG_PATH` from workspace `.env` would fix the reported workflow, but it conflicts with current docs/tests unless maintainers approve that security-boundary exception and update coverage and docs together.
