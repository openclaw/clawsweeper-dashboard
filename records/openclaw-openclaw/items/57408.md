Keep open. Current main and the latest shipped release still intentionally block workspace `OPENCLAW_CONFIG_PATH` under the fail-closed workspace dotenv boundary, while the linked open fork PR proposes a security-sensitive exception that needs maintainer approval.

## What This Changes

The issue asks OpenClaw to honor project-local `.env` `OPENCLAW_CONFIG_PATH` during CLI bootstrap instead of falling back to the default config path after workspace dotenv hardening.

## Best Possible Solution

Keep fail-closed behavior by default unless maintainers approve a narrow, documented, tested exception or opt-in for workspace `OPENCLAW_CONFIG_PATH`.

## Reproduction Assessment

Yes, at source level. Put `OPENCLAW_CONFIG_PATH` in a workspace `.env`; `loadCliDotEnv` routes it through `loadWorkspaceDotEnvFile`, which blocks `OPENCLAW_*` before config resolution can read the override.

## Solution Assessment

Unclear. Allowing `OPENCLAW_CONFIG_PATH` from workspace `.env` would fix the reported workflow, but it conflicts with the current fail-closed security boundary unless maintainers approve that exception with coverage and docs.
