## Summary

Keep this issue open. Current `main` and the latest release still block `OPENCLAW_CONFIG_PATH` from workspace `.env`, current docs/tests treat that as a security boundary, and an open candidate PR still references this issue, so a cleanup close would be premature.

## What This Changes

The issue asks CLI bootstrap to honor project-local `.env` entries for `OPENCLAW_CONFIG_PATH` so config resolution and `gateway.mode=local` use the project config instead of `~/.openclaw/openclaw.json`.

## Best Possible Solution

Keep this as the canonical product/security decision; either land a maintainer-approved narrow exception with tests/docs/proof, or confirm the documented trusted env surfaces are the intended replacement.

## Reproduction Assessment

Yes from source inspection: CLI startup loads CWD `.env` through the filtered workspace loader, and current tests assert `OPENCLAW_CONFIG_PATH` from that file remains unset. I did not run the CLI because this review is read-only and tests would create artifacts.

## Solution Assessment

Unclear until maintainers choose the security policy. If approved, the narrowest plausible fix is an absolute-path-only `OPENCLAW_CONFIG_PATH` exception while preserving blocks for `OPENCLAW_STATE_DIR`, relative paths, credentials, gateway controls, and broad runtime-control keys.
