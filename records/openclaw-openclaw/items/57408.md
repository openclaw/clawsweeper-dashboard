## Summary

Keep open. Current main still filters the reported key by design, the target-repo PR that implemented the exception was closed as superseded, and the remaining decision is a maintainer/security policy call rather than conservative cleanup.

## What This Changes

The issue asks OpenClaw to load a project-local `.env` during CLI bootstrap so `OPENCLAW_CONFIG_PATH` selects the project config instead of the default user config.

## Best Possible Solution

Keep the fail-closed workspace dotenv policy unless maintainers explicitly approve and document a narrow trusted exception; today the supported path is trusted process environment or global `~/.openclaw/.env`.

## Reproduction Assessment

Yes, at source level. Put `OPENCLAW_CONFIG_PATH` in a workspace `.env`; `loadCliDotEnv` calls `loadWorkspaceDotEnvFile`, which filters the key before config path resolution reads `process.env`.

## Solution Assessment

Unclear. Directly allowing this key from workspace `.env` conflicts with current docs and tests; a trusted global env surface or maintainer-approved opt-in is the narrower maintainable direction.
