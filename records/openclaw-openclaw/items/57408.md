## Best Possible Solution

Maintainers should decide the trusted project-config mechanism, then either land a narrow security-reviewed exception with tests/docs or keep the documented process/global-env path as the supported replacement.

## Reproduction Assessment

Yes, source-level: the CLI loads the project `.env` through `loadWorkspaceDotEnvFile`, and that loader blocks `OPENCLAW_CONFIG_PATH`; adjacent tests assert the variable remains unset. I did not run the CLI because this review is read-only.

## Solution Assessment

Unclear. Allowing project-local config paths is plausible for setup reliability, but the best solution must first settle the security contract for workspace `.env` rather than simply removing the current block.
