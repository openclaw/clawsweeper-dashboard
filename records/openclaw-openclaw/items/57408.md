## Summary

Keep open. Current main still blocks `OPENCLAW_CONFIG_PATH` from workspace `.env`, and https://github.com/openclaw/openclaw/pull/57416 is still the open implementation candidate; closing this issue before that security/product decision lands would be premature.

## What This Changes

The issue asks OpenClaw to let a project-local `.env` set `OPENCLAW_CONFIG_PATH` so CLI and gateway commands use a project config instead of the default user config.

## Best Possible Solution

Keep the fail-closed workspace dotenv policy unless maintainers explicitly approve and document a narrow trusted exception; if approved, revise or land https://github.com/openclaw/openclaw/pull/57416 with matching tests, docs, and real runtime proof.

## Reproduction Assessment

Yes, at source level. Put `OPENCLAW_CONFIG_PATH` in a workspace `.env`; `loadWorkspaceDotEnvFile` filters it before `resolveCanonicalConfigPath` can see it, so the CLI falls back to the default config path.

## Solution Assessment

Unclear. Allowing this one runtime-control key is a plausible compatibility fix, but it is not clearly the best solution until maintainers decide whether workspace `.env` may select runtime config at all.
