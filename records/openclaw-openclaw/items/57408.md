## Summary

Keep open. Current main and the latest shipped release still block workspace `OPENCLAW_CONFIG_PATH` under the documented fail-closed workspace dotenv boundary, while the remaining fix direction needs explicit maintainer/security policy approval.

## What This Changes

The issue requests restoring project-local `.env` support for `OPENCLAW_CONFIG_PATH` during CLI bootstrap so project-specific config files are used instead of the default config path.

## Best Possible Solution

Keep the fail-closed workspace dotenv policy unless maintainers approve a narrow, tested, documented exception or opt-in for `OPENCLAW_CONFIG_PATH`; use trusted process or global runtime env for this workflow in the meantime.

## Reproduction Assessment

Yes, at source level. Put `OPENCLAW_CONFIG_PATH` in a workspace `.env`; `loadCliDotEnv` routes it through `loadWorkspaceDotEnvFile`, which blocks it before `resolveConfigPath` can observe the override.

## Solution Assessment

Unclear. Allowing `OPENCLAW_CONFIG_PATH` from workspace `.env` would fix the reported workflow, but it conflicts with current docs/tests unless maintainers approve that security-boundary exception and update coverage and docs together.
