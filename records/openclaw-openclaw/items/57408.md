## Summary

Keep open. Current main and v2026.5.12 still intentionally block workspace `OPENCLAW_CONFIG_PATH` as part of the documented workspace dotenv trust boundary, so resolving this needs maintainer/security judgment rather than conservative cleanup.

## What This Changes

Allow project-local `.env` files to set `OPENCLAW_CONFIG_PATH` so CLI config resolution uses a project-specific config file instead of the default state-dir config.

## Best Possible Solution

Keep the fail-closed workspace dotenv policy unless owners approve and document a narrow trusted exception; otherwise direct project-specific config to trusted shell or global runtime env.

## Reproduction Assessment

Yes, at source level. Put `OPENCLAW_CONFIG_PATH` in a workspace `.env`; `loadCliDotEnv` calls `loadWorkspaceDotEnvFile`, which filters `OPENCLAW_` keys before config path resolution can observe the override.

## Solution Assessment

Unclear. Directly unblocking `OPENCLAW_CONFIG_PATH` from workspace `.env` is not clearly the best fix because it changes a documented security boundary; the maintainable path is an owner-approved exception with docs/tests or continued use of trusted env surfaces.
