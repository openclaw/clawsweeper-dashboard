## Summary

Keep this issue open. Current main and the latest release still intentionally block `OPENCLAW_CONFIG_PATH` from workspace `.env`, so the reported behavior remains a live config/security policy decision rather than an implemented cleanup item.

## What This Changes

The issue asks OpenClaw to honor `OPENCLAW_CONFIG_PATH` from a project-local `.env` during CLI bootstrap instead of resolving the default config file.

## Best Possible Solution

Keep this as the canonical product/security decision; if maintainers approve the workflow, land a narrow absolute-path-only `OPENCLAW_CONFIG_PATH` exception with tests that keep `OPENCLAW_STATE_DIR`, credentials, gateway controls, relative paths, and broad `OPENCLAW_*` runtime controls blocked.

## Reproduction Assessment

Yes from source inspection: `loadCliDotEnv` reads the CWD `.env` through `loadWorkspaceDotEnvFile`, and current tests prove `OPENCLAW_CONFIG_PATH` from that file is ignored. I did not run the CLI because this review is read-only and tests would write artifacts.

## Solution Assessment

Unclear until maintainers choose the security policy. The narrowest plausible fix is to allow only absolute `OPENCLAW_CONFIG_PATH` from workspace `.env` while preserving fail-closed blocking for `OPENCLAW_STATE_DIR`, relative paths, credentials, gateway controls, and other `OPENCLAW_*` keys.
