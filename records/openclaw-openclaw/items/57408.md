## Summary

Keep open. Current main and v2026.5.19 still intentionally block workspace `OPENCLAW_CONFIG_PATH` under the documented fail-closed workspace dotenv boundary, and the open linked fork PR carries the proposed exception that needs maintainer/security approval rather than cleanup closure.

## What This Changes

The issue reports a regression where project-local `.env` no longer sets `OPENCLAW_CONFIG_PATH`, causing CLI config resolution to fall back to the user config instead of a project runtime config.

## Best Possible Solution

Keep the fail-closed workspace dotenv policy unless maintainers approve a narrow, tested, documented exception or opt-in for `OPENCLAW_CONFIG_PATH`; otherwise guide users to trusted process or global runtime env for this workflow.

## Reproduction Assessment

Yes, at source level. Put `OPENCLAW_CONFIG_PATH` in a workspace `.env`; `loadCliDotEnv` routes it through `loadWorkspaceDotEnvFile`, which blocks `OPENCLAW_*` before `resolveConfigPath` can read the override.

## Solution Assessment

Unclear. Allowing `OPENCLAW_CONFIG_PATH` from workspace `.env` would fix the reported workflow, but it conflicts with current docs/tests and the fail-closed security boundary unless maintainers approve that exception with updated coverage and docs.
