## Summary

Keep open. Current main and the latest shipped v2026.5.18 release still intentionally block workspace `OPENCLAW_CONFIG_PATH` under the documented workspace dotenv trust boundary, and an unmerged fork PR still references this issue, so cleanup closure would be premature.

## What This Changes

The issue asks OpenClaw to load project-local `.env` values for `OPENCLAW_CONFIG_PATH` so CLI config resolution does not fall back to the default user config file.

## Best Possible Solution

Keep the issue open until owners decide whether to preserve the fail-closed workspace dotenv policy or approve a narrow, tested, documented exception for `OPENCLAW_CONFIG_PATH` only.

## Reproduction Assessment

Yes, at source level. Put `OPENCLAW_CONFIG_PATH` in a workspace `.env`; `loadCliDotEnv` routes it through `loadWorkspaceDotEnvFile`, which blocks it before `resolveConfigPath` can observe the override.

## Solution Assessment

Unclear. Directly unblocking `OPENCLAW_CONFIG_PATH` from workspace `.env` would solve the reported workflow but changes a documented security boundary, so the maintainable fix needs explicit owner approval plus docs and tests.
