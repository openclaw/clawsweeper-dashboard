## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Restore manifest-backed env auth lookup:** `src/llm/env-api-keys.ts:131-163`
  - body: `getApiKeyEnvVars` now relies on this fixed map, so providers declared only by plugin metadata no longer work with `findEnvKeys`/`getEnvApiKey`. For example, the Arcee manifest still declares `ARCEEAI_API_KEY`, and Hugging Face declares `HUGGINGFACE_HUB_TOKEN`, but this map misses those cases; `AuthStorage.hasAuth` and `getAuthStatus` call this path, so env-only installs can be reported unauthenticated or lose fallback credentials. Keep the manifest/setup-provider lookup as the source of truth instead of hardcoding every provider here.
  - confidence: 0.93
- **[P1] Do not trust nested official plugin paths:** `src/plugins/manifest-registry.ts:785`
  - body: Adding `isPathInside(candidatePath, trackedPath)` makes any candidate rooted inside the recorded official install directory match that install record. A config-scoped plugin can point at a nested subtree and still become `trustedOfficialInstall`, which weakens the official-plugin trust boundary used downstream for official-only behavior. Keep the match to the recorded package root or a candidate that contains the tracked install, not the reverse.
  - confidence: 0.9
- **[P1] Map legacy middleware runtime to OpenClaw:** `src/plugins/agent-tool-result-middleware.ts:17-19`
  - body: Existing plugin configs and typed constants can still use `runtimes: ["pi"]` or deprecated `harnesses: ["pi"]` for the built-in runtime. This normalizer only maps `codex-app-server`, so `pi` normalizes to nothing and middleware stops running on built-in turns; the public type also drops the old token. Keep `pi` as a deprecated alias of `openclaw` through the migration window.
  - confidence: 0.88
