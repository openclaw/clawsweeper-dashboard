  - body: This lowers provider IDs but drops the canonical mappings that current code relies on, including bedrock/aws-bedrock -> amazon-bedrock. - **[P1] Use manifest env vars for provider API-key lookup:** `src/llm/env-api-keys.ts:131`
  - body: This fixed map misses provider-owned env contracts that are still declared in plugin manifests, such as google accepting GOOGLE_API_KEY, cloudflare-ai-gateway requiring CLOUDFLARE_AI_GATEWAY_API_KEY, and moonshot using provider id moonshot.   - confidence: 0.94
- **[P1] Keep providerDiscoveryEntry as a legacy manifest input:** `src/plugins/manifest-registry.ts:469-475`
  - body: The manifest record now ignores providerDiscoveryEntry unless providerCatalogEntry is present, but current docs and main still treat providerDiscoveryEntry as the legacy spelling that existing plugins may use. Those plugins would lose provider discovery/catalog loading after upgrade until their manifests are manually renamed.
  - confidence: 0.91
- **[P1] Do not trust nested plugin paths as official installs:** `src/plugins/manifest-registry.ts:785`
  - body: Adding isPathInside(candidatePath, trackedPath) means a configured plugin path inside a recorded official install directory can be marked trustedOfficialInstall even though it is not the tracked package root.   - confidence: 0.88
- **[P1] Restore plugins.bundledDiscovery config compatibility:** `src/config/zod-schema.ts:1181-1183`
  - body: Removing plugins.bundledDiscovery from the strict plugins schema makes configs written by the earlier doctor migration invalid, and this branch also removes the visible repair path. Keep the key accepted or migrate it before making strict validation reject existing installs.
  - confidence: 0.9
