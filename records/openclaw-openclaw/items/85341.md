Full review comments:

- **[P1] Preserve providerDiscoveryEntry manifests:** `src/plugins/manifest-registry.ts:470-475`
  - body: Current main still accepts the deprecated `providerDiscoveryEntry` manifest key and the docs/tests describe it as working for existing plugins. This branch only builds `providerSourceEntry` from `providerCatalogEntry`, so upgraded external plugins that have not renamed the field lose provider discovery/auth catalog loading until operators edit manifests manually. Keep the legacy key as a deprecated fallback while preferring `providerCatalogEntry`.
  - confidence: 0.94
- **[P1] Use plugin env candidates for API-key lookup:** `src/llm/env-api-keys.ts:121-163`
  - body: This new resolver hard-codes one env var per provider instead of using the plugin manifest/provider env contract, so documented aliases such as `GOOGLE_API_KEY`, `GH_TOKEN`, `GITHUB_TOKEN`, `KIMICODE_API_KEY`, and `CLOUDFLARE_AI_GATEWAY_API_KEY` are invisible to `findEnvKeys`/`getEnvApiKey`. Env-only setups that follow the plugin docs can be reported unauthenticated or lose credential fallback; source this list from the manifest-backed provider env candidates or keep it fully synchronized.
  - confidence: 0.92
- **[P2] Keep the deprecated pi SDK runtime token:** `packages/sdk/src/types.ts:31`
  - body: `RuntimeSelection` is a public SDK type, and current main still accepts `{ type: "embedded", id: "pi" }`. Removing `pi` from this type makes existing SDK clients fail to type-check during the migration even if runtime code can normalize the legacy token elsewhere; keep it as a deprecated alias until the public removal window.
  - confidence: 0.88
