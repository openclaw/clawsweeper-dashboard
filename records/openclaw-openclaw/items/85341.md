Full review comments:

- **[P1] Restore manifest-driven env auth lookup:** `src/llm/env-api-keys.ts:121-164`
  - body: `getApiKeyEnvVars` now resolves credentials from a fixed hard-coded map and returns `undefined` for any provider id not listed, so manifest-declared providers are no longer discoverable unless core is edited for each one. `AuthStorage.hasAuth/getAuthStatus` and OpenAI-compatible request fallbacks depend on `findEnvKeys/getEnvApiKey`, which means env-only setups for providers such as `arcee` or documented secondary env vars like `GOOGLE_API_KEY` can be reported as unauthenticated or run without the expected credential. Keep a manifest/setup-provider fallback path instead of hardcoding provider ids.
  - confidence: 0.93
- **[P1] Keep the legacy provider-usage SDK export:** `src/plugin-sdk/provider-usage.ts:16`
  - body: This public Plugin SDK subpath no longer re-exports `resolveLegacyPiAgentAccessToken`, which current main still exposes for existing plugins. Removing that shipped symbol without a deprecated alias makes third-party plugins that import `openclaw/plugin-sdk/provider-usage` fail to type-check or build during upgrade, so keep the old export pointing at the new implementation until a documented removal window passes.
  - confidence: 0.9
- **[P1] Restore cross-provider Mistral compat contribution:** `extensions/mistral/index.ts:46`
  - body: The Mistral plugin now only normalizes models it directly owns, while current main also contributes Mistral compat flags for OpenAI-compatible routes that are Mistral behind another provider such as OpenRouter or a custom base URL. Without that contribution, those routes can fall back to default OpenAI-compatible payload behavior, including the wrong max-token field or store support, and send invalid requests to Mistral-compatible backends.
  - confidence: 0.87
