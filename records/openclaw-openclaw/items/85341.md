Full review comments:

- **[P1] Preserve the deprecated pi middleware runtime:** `src/plugins/agent-tool-result-middleware-types.ts:5-7`
  - body: Existing plugins can still compile or declare `runtimes: ["pi"]` / legacy harnesses from the shipped API. This change narrows the public type to `"openclaw" | "codex"` and the normalizer only maps `codex-app-server`, so those plugins either stop type-checking or normalize to no built-in runtime. Keep `"pi"` as a deprecated alias that maps to `"openclaw"`.
  - confidence: 0.93
- **[P1] Keep the legacy Pi access-token SDK export:** `src/plugin-sdk/provider-usage.ts:16`
  - body: `openclaw/plugin-sdk/provider-usage` currently exposes `resolveLegacyPiAgentAccessToken`; removing it from the public barrel makes existing provider plugins fail to build even though this PR is supposed to keep deprecated SDK compatibility at named boundaries. Re-export the old symbol as a deprecated alias to the new helper until a documented removal window passes.
  - confidence: 0.91
- **[P1] Derive env auth keys from provider contracts:** `src/llm/env-api-keys.ts:131-160`
  - body: This hard-coded env map disagrees with the provider manifests and docs: for example Google still accepts `GOOGLE_API_KEY`, Hugging Face accepts `HUGGINGFACE_HUB_TOKEN`, GitHub Copilot accepts `GH_TOKEN`/`GITHUB_TOKEN`, and Qwen/Alibaba expose Qwen/DashScope keys. Because auth status and OpenAI-compatible requests call `getEnvApiKey(model.provider)`, env-only users for those providers can be treated as unauthenticated or run without the configured key.
  - confidence: 0.9
