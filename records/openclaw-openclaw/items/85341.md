- **[P1] Derive API-key env lookup from provider metadata:** `src/llm/env-api-keys.ts:131`
  - body: The new hard-coded envMap misses provider contracts declared in this same PR, such as microsoft-foundry/AZURE_OPENAI_API_KEY and several OpenAI-compatible provider env vars.   - confidence: 0.93
- **[P1] Normalize legacy pi runtime ids during migration:** `src/agents/agent-runtime-id.ts:20`
  - body: normalizeEmbeddedAgentRuntime returns legacy "pi" unchanged, so persisted configs and sessions that still carry agentRuntime.id: "pi" can be classified as custom plugin runtimes instead of the built-in OpenClaw runtime. Map pi to openclaw at this central boundary while doctor/migration removes old values.
  - confidence: 0.9
- **[P1] Restore Groq resolved-model compat registration:** `extensions/groq/index.ts:15-16`
  - body: The PR leaves Groq's reasoning compat helper detached from provider registration, so Groq models that need reasoning-effort normalization can receive unsupported OpenAI-compatible request parameters.   - confidence: 0.86

  - body: resolveCacheRetention now reads OPENCLAW_CACHE_RETENTION only, so installs still using the shipped PI_CACHE_RETENTION=long fallback silently switch to short prompt-cache retention on OpenAI-compatible completions. Preserve the deprecated env alias through the migration window.
  - confidence: 0.82
- **[P2] Keep legacy token discovery in usage auth gating:** `src/infra/provider-usage.auth.ts:421-423`
  - body: The provider-usage auth gate no longer considers legacy Pi token discovery before calling plugin auth, so migrated users with credentials only in the legacy auth file can get false unauthenticated usage/auth results.
