Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Honor OpenClaw runtime pins before CLI inference:** `src/auto-reply/reply/agent-runner-execution.ts:1678-1682`
  - body: `resolveSessionRuntimeOverrideForProvider` now returns `"openclaw"` for explicit built-in runtime pins, including legacy `pi` values, but this path only short-circuits CLI provider IDs before calling `resolveCliRuntimeExecutionProvider`. That lets CLI auth-profile inference reroute a session that was explicitly pinned to the built-in runtime, so keep `openclaw` on the base provider before CLI inference.
  - confidence: 0.9
- **[P1] Keep follow-up runs on explicit OpenClaw runtime pins:** `src/auto-reply/reply/followup-runner.ts:579-583`
  - body: The follow-up runner has the same regression: a persisted `openclaw` runtime override falls through to CLI provider inference because only CLI runtime IDs are honored here. Queued follow-up turns can therefore run on a CLI harness even though the session was pinned to the built-in runtime.
  - confidence: 0.9
- **[P2] Preserve the legacy QA runtime alias:** `src/agents/model-runtime-policy.ts:185`
  - body: Changing this gate to accept only `openclaw`/`codex` drops the shipped `OPENCLAW_QA_FORCE_RUNTIME=pi` value, while other compatibility paths still treat `pi` as the built-in runtime alias. Normalize `pi` to `openclaw` here and in the QA live auth gate so existing private QA automation does not silently change runtime or auth behavior.
  - confidence: 0.88
- **[P2] Use graceful process-tree termination:** `packages/agent-core/src/harness/env/nodejs.ts:241`
  - body: This helper kills the process group with immediate `SIGKILL`, bypassing the shared `src/process/kill-tree.ts` behavior that sends `SIGTERM`, waits a bounded grace period, and only then escalates. Reuse or extract the shared graceful helper so agent-core command timeouts do not skip cleanup handlers or widen process-boundary risk.
  - confidence: 0.87
