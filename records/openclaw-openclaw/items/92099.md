## Summary

Keep open. The branch is a plausible fix candidate for the linked Active Memory bug, but it is not merge-ready because it reads the recall query from historical `event.messages` instead of the current prompt, lacks real large-envelope behavior proof, and adds a compatibility-sensitive config surface that still needs maintainer direction.

## What This Changes

The PR adds an optional Active Memory `messageMaxChars` config/schema/docs field and truncates message-mode query input while changing the latest-user-message source from `event.prompt` to `event.messages`.

## Best Possible Solution

Keep Active Memory on a current-turn input source, either by capping `event.prompt` or by adding a maintainer-approved slim-intent hook field, then prove the large-envelope path with redacted real behavior logs before landing.

## Reproduction Assessment

Yes, from source. Current main passes `event.prompt` into Active Memory and `message` mode returns it unchanged; the PR regression is also source-reproducible because `event.messages` is historical while `event.prompt` carries the latest turn.

## Solution Assessment

No. The truncation helper is plausible, but selecting the latest message from `event.messages` is not the best fix for the current-turn contract; the safer path is to cap `event.prompt` or add an explicit slim-intent hook field.
