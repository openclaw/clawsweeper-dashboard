## Summary

The linked report identifies a valid source-level problem, but this patch is not merge-ready: it caps generic assembled prompt text rather than reliably selecting the latest user turn, its new truncation tests copy the implementation instead of exercising production code, and its diagnostic can falsely report truncation outside message mode. Real after-fix behavior proof is also still required.

## What This Changes

Adds an Active Memory `messageMaxChars` configuration field, message-mode truncation and diagnostics, schema coverage, and documentation.

## Best Possible Solution

Build message mode from the latest normalized user turn in the prepared message list, apply any configured cap to that canonical value, test the registered production hook, and verify a real large-envelope run still performs useful recall.

## Reproduction Assessment

Yes at source level: Active Memory message mode forwards the generic hook prompt unchanged, and the hook contract does not guarantee that prompt is a slim latest-user-message value. The exact long-session runtime path was not rerun during this review.

## Solution Assessment

No. Bounding the prefix of generic assembled prompt text does not restore the documented latest-user-message contract; selecting the prepared latest user turn first is narrower and preserves intent more reliably.

Full review comments:

- **[P1] Use the actual latest user turn before truncating:** `extensions/active-memory/index.ts:2126-2133`
  - body: `event.prompt` is generic prompt text and may be the assembled envelope described by the linked issue. Slicing its first `messageMaxChars` characters only bounds that envelope and can omit the current user intent, preventing relevant recall. Build message mode from the latest normalized user turn in `event.messages` or an explicitly typed slim-intent field, then apply the cap to that value.
  - confidence: 0.93
- **[P2] Only log truncation in message mode:** `extensions/active-memory/index.ts:3094-3101`
  - body: This condition can fire when `queryMode` is `recent` or `full`, although `buildQuery` does not apply `messageMaxChars` in those modes. Operators would therefore receive a false "query truncated" diagnostic; include the message-mode condition and base the lengths on the actual selected input.
  - confidence: 0.99
- **[P2] Exercise the production `buildQuery` path:** `extensions/active-memory/buildQuery.test.ts:5-25`
  - body: This test defines a second local `buildQuery`, so every assertion remains green if the implementation in `index.ts` is removed or changes. Move these cases into the existing registered-hook tests or export a narrow production helper so the regression coverage executes the changed code.
