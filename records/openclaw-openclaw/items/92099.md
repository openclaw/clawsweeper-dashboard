## Summary

Keep open. The branch adds a useful cap idea, but it changes Active Memory to treat historical hook messages as the current turn, so message-mode recall can query a stale prior user message or nothing; it also still has mock-only proof and a conflicting head.

## What This Changes

The PR adds an optional Active Memory `messageMaxChars` plugin config field, documents and validates it, and changes message-mode query building to extract and truncate a user message from hook `event.messages`.

## Best Possible Solution

Keep Active Memory on a current-turn input source, either `event.prompt` with a maintainer-approved cap or a new slim-intent hook contract, then prove the large-envelope path with redacted real behavior logs before landing.

## Reproduction Assessment

Yes from source. Current main passes `event.prompt` into message mode unchanged, and the PR regression is also source-reproducible because hook tests document `messages` as historical while `prompt` is the latest turn.

## Solution Assessment

No. The truncation helper is a plausible part of the fix, but selecting the latest message from `event.messages` is the wrong layer for the current-turn contract; the safer fix is to cap `event.prompt` or first add an explicit slim-intent hook field.
