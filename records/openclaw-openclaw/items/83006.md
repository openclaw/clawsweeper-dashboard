## What This Changes

The PR adds a typed `defineToolPlugin` SDK helper, `openclaw plugins init/build/validate` authoring commands, tests, docs, SDK export metadata, and a changelog entry for simple tool plugins.

## Best Possible Solution

Land a repaired version that preserves manifest-first discovery, enforces plugin-root entry boundaries cross-platform, uses the same JSON5-compatible manifest parsing for freshness checks, and includes real CLI authoring proof.

## Reproduction Assessment

Not applicable for the feature itself; this is a new SDK/CLI capability rather than a bug report. The blocking review issues are source-reproducible from the PR head by inspecting the new path normalization and manifest re-read code.

## Solution Assessment

No, not as currently written. The overall manifest-first authoring direction is reasonable, but the command implementation should share the manifest parser contract and harden root-boundary path checks before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Reject cross-drive entries outside the plugin root:** `src/cli/plugins-authoring-command.ts:52-55`
  - body: `normalizeRelativePath` only rejects `../` escapes. On Windows, `path.relative()` returns a drive-qualified path for a different drive, so a cross-drive `--entry` can be rewritten as a relative-looking value and then imported and persisted outside the plugin root.
  - confidence: 0.9
- **[P2] Parse plugin manifests with the JSON5-compatible loader:** `src/cli/plugins-authoring-command.ts:251`
  - body: `plugins validate` accepts the manifest through `loadPluginManifest`, but then re-reads the same file with strict `JSON.parse`. A manifest with comments or trailing commas passes the normal loader and then crashes this new validation/freshness path.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The new local authoring command intentionally imports plugin code, but the current Windows path guard can let an entry outside the plugin root be imported and written into package metadata.

  - body: Because `plugins build/validate` imports the selected entry and persists it to `openclaw.extensions`, cross-drive paths that bypass the intended plugin-root guard can create an out-of-package runtime reference on Windows.
