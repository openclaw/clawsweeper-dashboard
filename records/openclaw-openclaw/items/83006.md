## Summary

Keep open. Current main does not contain the requested SDK/CLI feature, and the active protected PR still has source-reproducible P2 blockers plus mock-only real behavior proof.

## What This Changes

The PR adds a `defineToolPlugin` SDK subpath, tool-plugin authoring CLI commands, docs, tests, package exports, and an `llm-task` migration to the helper.

## Best Possible Solution

Land a repaired version that keeps manifest-first discovery, hardens authoring path boundaries, validates exact generated package metadata, scaffolds publishable package names, and includes redacted real CLI authoring proof.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a current-main bug, but yes for the review blockers: the latest PR head shows the failing source paths, and the Windows path guard issue is reproduced with a read-only `path.win32.relative` probe.

## Solution Assessment

Is this the best way to solve the issue? No, not as currently written. The SDK/CLI direction is plausible, but the command should share the existing manifest parser contract, harden root-boundary checks, validate the single generated runtime entry, and generate publishable scaffold metadata.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Reject cross-drive entries outside the plugin root:** `src/cli/plugins-authoring-command.ts:49-56`
  - body: `normalizeRelativePath` only rejects `..` escapes. On Windows, an entry on another drive normalizes to a drive-qualified relative path like `D:/outside/index.js`, passes this guard, and can then be imported and persisted as if it lived under the plugin root.
  - confidence: 0.9
- **[P2] Parse plugin manifests with the JSON5-compatible loader:** `src/cli/plugins-authoring-command.ts:37-38`
  - body: `readJsonFile` uses strict `JSON.parse` for `openclaw.plugin.json` in build/check/validate paths, but the normal manifest loader accepts JSON5 features such as comments and trailing commas. Supported manifests will fail this new authoring path.
  - confidence: 0.9
- **[P2] Generate publishable names for scoped plugin ids:** `src/cli/plugins-authoring-command.ts:330-332`
  - body: `plugins init` interpolates the raw plugin id into `package.json.name`, so a supported scoped id such as `@acme/quotes` becomes `openclaw-plugin-@acme/quotes`, which is not a valid npm package name for publishing.
  - confidence: 0.86
- **[P2] Reject stale extra runtime entries during validation:** `src/cli/plugins-authoring-command.ts:236-238`
  - body: `validateToolPluginProject` only checks whether `openclaw.extensions` includes the selected entry. A package with `['./dist/index.js', './dist/old.js']` passes validation even though build would rewrite it to the selected single runtime entry, allowing unintended stale runtime entrypoints to ship.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The patch introduces authoring-time runtime imports and package metadata generation, with concrete path-boundary and stale-entry concerns.

Concerns:

- **[low] Harden authoring entry root checks:** `src/cli/plugins-authoring-command.ts:49`
  - body: Because `plugins build/validate` imports the selected entry and writes it into package metadata, cross-drive paths that bypass the intended root guard can create an out-of-package runtime reference on Windows.
  - confidence: 0.82
- **[low] Reject stale package runtime entries:** `src/cli/plugins-authoring-command.ts:236`
  - body: Validation can pass while extra `openclaw.extensions` entries remain, so a package can ship unintended plugin runtime entrypoints even though the generated metadata contract says there is one selected entry.
  - confidence: 0.78
