## Label Justifications

- `P2`: This is a bounded but real status/usage bug that misleads Anthropic API-key users without breaking inference.
- `impact:auth-provider`: The issue is about provider credential type handling and misleading auth status for a valid Anthropic API-key profile.

## Summary

Keep open: current main still lets `openclaw status --usage` resolve an Anthropic API-key credential through the generic usage fallback and then send it to Anthropic's OAuth usage fetcher, so the misleading bearer-token error remains actionable even though chat `/status` has a narrower OAuth-only guard.

## What This Changes

The issue asks OpenClaw to avoid reporting valid Anthropic pay-per-use API keys as invalid bearer tokens during provider usage/status checks.

## Best Possible Solution

Make Anthropic usage auth truly OAuth/admin-only for usage snapshots: API-key-only Anthropic profiles should not hit the OAuth usage endpoint, while API-key usage for providers that actually support usage endpoints must remain intact.

## Reproduction Assessment

Yes, at source level: `openclaw status --usage` can resolve an Anthropic API key through the generic usage fallback, then `fetchClaudeUsage` sends that token as an OAuth bearer token. I did not run a live Anthropic-key network repro in this read-only review.

## Solution Assessment

Yes, the best repair belongs in provider-usage/auth resolution rather than CLI string post-processing. Anthropic API-key credentials should be omitted or represented as no usage data before the OAuth usage fetcher is called.
