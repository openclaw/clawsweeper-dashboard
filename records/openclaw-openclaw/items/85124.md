## Summary

Keep open: current main and the latest release still have a source-level path where an Anthropic API-key-only profile can be selected for provider usage and then sent to Anthropic's OAuth usage endpoint, producing the misleading bearer-token failure described here.

## What This Changes

The issue asks OpenClaw to avoid reporting Anthropic pay-per-use API keys as invalid bearer tokens when status usage/quota data is unavailable for that auth type.

## Best Possible Solution

Make provider usage auth respect provider-owned credential types: Anthropic API-key-only profiles should produce no usage data or an explicit no-quota message, while Anthropic OAuth/setup-token usage and other providers' API-key usage remain intact.

## Reproduction Assessment

Yes, at source level: an Anthropic API-key-only profile can fall through to generic usage auth, then fetchClaudeUsage sends that token to the OAuth bearer usage endpoint. I did not use a live Anthropic key in this read-only review.

## Solution Assessment

Yes, the requested skip/no-data behavior is the right direction, but it should be implemented at the provider-usage auth boundary rather than by string-matching CLI output. The safest fix preserves plugin-owned API-key usage for providers that support it.

## Repair Work Prompt

On current main, Anthropic API-key-only profiles can be returned by the generic provider-usage fallback and then sent to fetchClaudeUsage, which calls Anthropic's OAuth usage endpoint with Authorization: Bearer and surfaces a misleading invalid bearer token error. Keep the fix in the provider-usage/auth boundary, not CLI output formatting. Add regression coverage for Anthropic API-key-only profiles resolving to no usage auth/no status usage error, Anthropic OAuth/token profiles still using fetchClaudeUsage, and status/gateway usage surfaces avoiding the misleading error.
