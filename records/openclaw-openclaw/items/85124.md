- `impact:auth-provider`: The report concerns provider credential type handling and misleading auth status for valid Anthropic API-key profiles.

## Summary

Keep open: current main still has a source-level path where an Anthropic API-key-only profile can be selected for provider usage and then passed to Anthropic's OAuth-only usage fetcher, producing a misleading auth error instead of omitting usage or showing a non-actionable-data message.

## What This Changes

The issue asks OpenClaw to stop reporting valid Anthropic API-key auth as an invalid bearer token in status usage/quota output.

## Best Possible Solution

Make Anthropic usage auth truly OAuth/admin-only: API-key-only profiles should not hit the OAuth usage endpoint, while OAuth/setup-token behavior and API-key usage for providers that support it remain intact.

## Reproduction Assessment

Yes, at source level: an Anthropic API-key-only profile can be resolved by the generic usage-auth fallback, then sent to the Anthropic OAuth usage endpoint with a bearer header. I did not run a live Anthropic-key repro in this read-only review.

## Solution Assessment

Yes, the best fix is to keep this in the provider-usage/auth boundary rather than adding broad CLI string handling. Anthropic API-key profiles should be omitted from OAuth usage fetches or represented as explicit no-data without changing inference auth.
