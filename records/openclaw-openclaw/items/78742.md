- **[P1] Keep restart audit state in SQLite:** `src/infra/restart.ts:207-219`
  - body: This branch adds audit data to the old `gateway-restart-intent.json` path, but current main persists restart intent and sentinel state in SQLite. Resolve the conflict by extending the current SQLite-backed payload/rows instead of restoring file-backed runtime state.
  - confidence: 0.9
- **[P1] Pass the restart session key through scheduling:** `src/auto-reply/reply/commands-session.ts:710-713`
  - body: The modified `/restart` scheduling call adds `triggerAudit` but omits current main's `sessionKey: sentinelPayload?.sessionKey` guard. Without that guard, coalesced restarts from another session can overwrite or drop the original restart continuation.
  - confidence: 0.88
- **[P2] Preserve attribution for supervisor SIGTERMs:** `src/cli/gateway-cli/run-loop.ts:466-472`
  - body: The SIGTERM audit line reads `restartIntent?.audit`, but the slash-command supervisor path only passes audit to `triggerOpenClawRestart` and the sentinel. A real launchctl/systemd SIGTERM can still log `source=external`; carry the same audit into the signal-consumed state or read it from the sentinel.
  - confidence: 0.9
- **[P2] Use distinct keys for trigger and lifecycle actions:** `src/cli/gateway-cli/run-loop.ts:466-472`
  - body: When `audit.actionLabel` is present, `appendGatewayRestartAuditLine` already writes `action=/restart`; adding `extra: { action: "restart" }` appends a second `action=` key. Parsers may keep the last key and lose the trigger label.
  - confidence: 0.86
- **[P1] Redact restart audit identifiers before logging:** `src/infra/restart.ts:671-682`
  - body: The new log writer can emit raw `sender_id`, `session_key`, `delivery_context`, and `ws_conn_id` values. Those can identify users, sessions, or local routing context in support logs, so hash/remove them or get explicit maintainer approval for the local-log contract.
  - confidence: 0.82
