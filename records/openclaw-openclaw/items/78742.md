- **[P1] Keep restart audit state in SQLite:** `src/infra/restart.ts:206-225`
  - body: Current main writes restart intent through the shared `gateway_restart_intent` SQLite table, but this branch adds audit data to the old JSON intent-file writer. Rebase this onto the current SQLite state path rather than reintroducing file-backed runtime restart state.
  - confidence: 0.9

  - body: Current main passes `sessionKey: sentinelPayload?.sessionKey` into `scheduleGatewaySigusr1Restart` so coalesced restarts cannot replace another session's continuation.   - confidence: 0.88
- **[P1] Redact restart audit identifiers before logging:** `src/infra/restart.ts:671-686`
  - body: The audit writer appends raw sender IDs, session keys, delivery context, and WebSocket IDs to `gateway-restart.log`. Those values can identify users or routing context in shared support logs, so redact/hash them or get explicit maintainer approval for that log contract.
  - confidence: 0.86
- **[P2] Carry supervisor audit into the signal receiver:** `src/cli/gateway-cli/run-loop.ts:466-470`
  - body: The supervisor path passes audit to dispatch/sentinel state, but the SIGTERM receiver only reads `restartIntent?.audit`. A real launchctl/systemd SIGTERM can still log `source=external`, leaving the middle line of the proposed three-line chain unattributed.
  - confidence: 0.86
- **[P2] Avoid duplicate action keys in restart audit lines:** `src/cli/gateway-cli/run-loop.ts:466-471`
  - body: When `audit.actionLabel` is set, `appendGatewayRestartAuditLine` already writes `action=/restart`; this signal path also passes `extra: { action: ... }`, producing duplicate `action=` fields that parsers may interpret inconsistently.
  - confidence: 0.82
