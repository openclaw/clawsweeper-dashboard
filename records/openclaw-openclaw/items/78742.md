Full review comments:

- **[P1] Keep restart audit state in SQLite:** `src/infra/restart.ts:207-224`
  - body: This adds audit data to the old file-backed restart-intent writer. Current main persists restart intent in the `gateway_restart_intent` SQLite table, so a rebase must extend the SQLite state model instead of reviving sidecar runtime state.
  - confidence: 0.91
- **[P1] Preserve the restart session guard:** `src/auto-reply/reply/commands-session.ts:710-713`
  - body: The changed `/restart` scheduling call adds `triggerAudit` but omits current main's `sessionKey: sentinelPayload?.sessionKey` guard. Without that guard, coalesced restarts from another session can replace or drop the original restart continuation.
  - confidence: 0.89
- **[P1] Redact restart audit identifiers before logging:** `src/infra/restart.ts:671-682`
  - body: The audit writer emits raw sender, session, delivery, and WebSocket identifiers into `gateway-restart.log`. Those values can identify users or routing context in shared diagnostics, so redact them or get explicit maintainer approval for the local-log contract.
  - confidence: 0.86
- **[P2] Carry audit into the SIGTERM receiver path:** `src/cli/gateway-cli/run-loop.ts:466-470`
  - body: The SIGTERM audit line reads only `restartIntent?.audit`, but the slash-command supervisor path does not write a restart intent before `launchctl` or `systemctl` kills the process. A real supervisor SIGTERM can therefore still log `source=external`; persist or read the same audit context for the receiver side.
  - confidence: 0.88
- **[P2] Avoid duplicate action keys in audit lines:** `src/cli/gateway-cli/run-loop.ts:466-471`
  - body: When `audit.actionLabel` is present, `appendGatewayRestartAuditLine` already writes `action=/restart`; adding `extra: { action: "restart" }` appends a second `action=` key, so parsers may keep the lifecycle value and lose the trigger label.
  - confidence: 0.82
