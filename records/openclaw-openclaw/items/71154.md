## Summary

Keep open. Current main still has the old backslash-newline allowlist rejection, so the PR is not obsolete, but the proposed denial split leaves unparseable allowlist analysis overrideable by durable approval on both gateway and node-host paths; that is a security-sensitive policy regression that should block merge until maintainers decide and the branch is updated.

## What This Changes

The PR normalizes POSIX backslash-newline continuations before exec allowlist analysis, adds an unsupported-shell-syntax denial signal through gateway and node-host policy, updates user messaging/tests, and adds a changelog entry.

## Best Possible Solution

Land an adjusted version that preserves POSIX backslash-newline splicing for analyzable allowlisted commands, reserves allowlist-miss for successfully analyzed non-allowlisted commands, and hard-denies or explicitly maintainer-documents unparseable syntax before durable approval reuse across gateway and node-host paths, with matching tests and changelog credit.

## Reproduction Assessment

Yes. Current main is source-reproducible: any command containing backslash-newline reaches hasShellLineContinuation() and returns analysisOk=false, and the existing test asserts that behavior. The PR blocker is also source-reproducible by setting analysisOk=false with durableApprovalSatisfied=true in the gateway or node-host policy paths.

## Solution Assessment

No, not as currently proposed. The normalization and distinct denial direction are the right narrow shape, but unsupported shell syntax must be denied before durable trust is considered unless maintainers deliberately approve and document a different policy.
