## Summary

Keep open: current main still has the reported exec allowlist behavior, and this PR is the canonical implementation path, but the PR head still has a blocking security issue where durable approvals can bypass the new unsupported-syntax denial path.

## What This Changes

The PR normalizes POSIX backslash-newline continuations before exec allowlist analysis and adds unsupported-shell-syntax denial reasons across gateway, node-host, and user-message paths.

## Best Possible Solution

Land an adjusted PR that keeps POSIX continuation normalization, preserves true allowlist-miss semantics, hard-denies unsupported syntax before durable approvals unless maintainers explicitly document an exception, and includes focused tests plus real behavior proof.

## Reproduction Assessment

Yes. Source inspection shows current main returns analysisFailure for backslash-newline before allowlist analysis, and the existing test asserts that old behavior; the PR blocker is also source-reproducible with analysisOk=false and durableApprovalSatisfied=true in the gateway or node policy path.

## Solution Assessment

No, not as currently proposed. The normalization direction is narrow and useful, but unsupported shell syntax should be denied before durable trust is considered unless maintainers deliberately approve and document a different policy.
