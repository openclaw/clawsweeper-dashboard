## Summary

## What This Changes

The PR normalizes POSIX backslash-newline continuations before exec allowlist analysis, adds unsupported-shell-syntax denial signaling, and updates gateway/node-host tests plus the changelog.

## Best Possible Solution

Land an adjusted implementation that either fits the current parser or the newer authorization planner, hard-denies unsupported syntax before durable approval reuse unless maintainers explicitly choose otherwise, and includes focused tests plus redacted real behavior proof.

## Reproduction Assessment

Yes, at source level. Current main still hard-fails backslash-newline before allowlist analysis, and the PR blocker is visible from the PR-head control flow where analysisOk=false is checked after durable approval paths.

## Solution Assessment

No, not as currently proposed. The normalization direction is useful, but unsupported shell syntax should not be bypassable by durable trust unless maintainers make and document that security policy decision.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Deny unparseable gateway commands before durable trust:** `src/agents/bash-tools.exec-host-gateway.ts:101-103`
  - body: `resolveGatewayDeniedReason()` delegates to `hasGatewayAllowlistMiss()`, which returns false when `durableApprovalSatisfied` is true. That lets exact-command durable trust skip the new unsupported-shell-syntax denial for `analysisOk=false` content and continue through gateway execution.
  - confidence: 0.86
- **[P1] Deny unparseable node commands before durable trust:** `src/node-host/exec-policy.ts:136`
  - body: The new unsupported-shell-syntax branch runs after the durable approval allow branch. In allowlist mode, persisted durable trust can still allow `analysisOk=false` commands instead of returning `SYSTEM_RUN_DENIED: unsupported shell syntax`.
  - confidence: 0.84
