## Summary

Keep open. The PR addresses a real current-main exec allowlist bug, but the patch still lets durable approvals bypass the new unparseable-syntax denial and the external contributor has not supplied after-fix real behavior proof.

## What This Changes

The PR splices POSIX backslash-newline continuations before exec allowlist analysis, adds unsupported-shell-syntax denial plumbing across gateway/node-host paths, updates tests, and adds a changelog entry.

## Best Possible Solution

Land an adjusted implementation that keeps the line-continuation normalization, fail-closes `analysisOk=false` before durable approval reuse unless maintainers explicitly approve an exception, rebases against the planner direction, and includes focused tests plus redacted real behavior proof.

## Reproduction Assessment

Yes. Current main source still hard-fails `\<newline>` before allowlist analysis, and the PR blocker is source-reproducible with `analysisOk=false` plus `durableApprovalSatisfied=true` in gateway or node policy.

## Solution Assessment

No, not as currently proposed. The normalization approach is useful, but unsupported shell syntax should be denied before durable trust in both gateway and node-host paths unless maintainers explicitly document a policy exception.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Deny unparseable gateway commands before durable trust:** `src/agents/bash-tools.exec-host-gateway.ts:101-103`
  - body: `resolveGatewayDeniedReason()` first calls `hasGatewayAllowlistMiss()`, and that helper returns false when `durableApprovalSatisfied` is true. With `analysisOk=false`, durable exact-command trust can skip the new unsupported-shell-syntax denial and continue toward execution.
  - confidence: 0.88
- **[P1] Deny unparseable node commands before durable trust:** `src/node-host/exec-policy.ts:136-140`
  - body: Move this unsupported-syntax branch ahead of durable approval reuse. As written, `durableApprovalSatisfied` returns `allowed: true` before the new branch runs, so unparseable allowlist analysis can be executed instead of denied.
  - confidence: 0.86
