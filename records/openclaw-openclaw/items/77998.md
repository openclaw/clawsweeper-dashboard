## Summary

Keep open. The PR targets a real skills watcher lifecycle leak, but the current patch can suppress refresh after a cleared workspace version falls back to an older global version.

## What This Changes

The PR adds per-workspace skills snapshot version clearing on watcher teardown, opportunistic idle eviction for skills watchers, regression tests, and changelog entries.

## Best Possible Solution

Keep the lifecycle cleanup, but make teardown and idle eviction leave a monotonic or explicit invalidation state so the next session refreshes, with regression coverage for the global fallback case.

## Reproduction Assessment

Yes. Current main has a source-reproducible leak path: watcher events write workspace-scoped versions, while watcher teardown closes only the watcher and leaves the workspace version key behind; the PR body also includes terminal proof for the Map cleanup behavior.

## Solution Assessment

No, not yet. The cleanup boundary is right, but deleting the workspace version can lower the observable version to an older global value and let a stale cached skills snapshot survive after idle eviction.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve refresh invalidation when clearing workspace versions:** `src/agents/skills/refresh-state.ts:75`
  - body: Deleting the workspace entry can make `getSkillsSnapshotVersion` fall back to an older nonzero `globalVersion`. Because session setup reads that version before re-ensuring the watcher and `shouldRefreshSnapshotForVersion(cached, next)` returns false when `cached` is higher, a stale skills snapshot can survive after idle eviction until a later watched event. Make clear/eviction force a refresh or preserve a monotonic invalidation value, and cover this case in tests.
  - confidence: 0.86
