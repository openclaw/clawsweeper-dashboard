## Summary

Keep open: the cleanup direction is appropriate, but the current patch can make a cleared workspace version fall back to an older nonzero global version and skip a needed skills snapshot rebuild.

## What This Changes

The PR adds a per-workspace skills snapshot version clear helper, wires watcher teardown through it, evicts idle skills watchers after one hour, adds regression tests, and updates the changelog.

## Best Possible Solution

Keep the lifecycle cleanup, but make workspace-version clearing preserve a forced refresh/invalidation invariant and add a regression test for the nonzero-global fallback before merge.

## Reproduction Assessment

Yes, source-reproducible: current main keeps workspace-local skill versions and watcher state without a production session-end cleanup path. The PR body also includes terminal proof of the leak shape and patched cleanup, though I did not run a live chokidar gateway in this read-only review.

## Solution Assessment

No, not yet: the cleanup boundary is narrow, but deleting the workspace version can make the observable version fall back to an older nonzero global value and reuse a stale snapshot. A safer fix should force refresh after clear/eviction and cover that case in tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve refresh invalidation when clearing workspace versions:** `src/agents/skills/refresh-state.ts:75`
  - body: Deleting the workspace entry can make the observable version go backwards when `globalVersion` is nonzero but older than the cached workspace-local version. After idle eviction, a skills file can change while no watcher is attached; on the next turn, `shouldRefreshSnapshotForVersion(cachedLocal, olderGlobal)` returns false, so the stale skills snapshot can persist until another watched event. Make the clear path or comparison force a refresh for cleared workspaces and cover the nonzero-global case.
  - confidence: 0.86
- **[P3] Credit the contributor in the changelog:** `CHANGELOG.md:175-176`
  - body: Repo policy asks contributor-facing changelog entries to include a human `Thanks @...` credit. These user-facing fix entries omit the PR author's credit, so release notes would drop attribution.
  - confidence: 0.94
