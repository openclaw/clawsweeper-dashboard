## Summary

Keep open. Current `main` still lacks a modifying plugin seam between final reply text creation and TTS directive parsing, so the per-message voice-routing use case remains unsolved; the adjacent hook reports cover different phases and do not supersede this request.

## What This Changes

Add a plugin hook that can modify final reply text, or skip TTS, after LLM output and before TTS directive parsing/audio generation.

## Best Possible Solution

Add a documented, additive modifying hook at the final-reply/pre-TTS boundary so plugins can inject `[[tts:...]]` directives or suppress TTS without owning full reply dispatch.

## Reproduction Assessment

Not applicable as a runtime bug reproduction because this is a feature request. The missing capability is high-confidence from source inspection: current `main` has no hook type, runner method, dispatch integration, or docs entry for a pre-TTS text transform.

## Solution Assessment

Yes, the requested direction is the narrowest maintainable fix. A plugin alone cannot affect TTS directives before audio generation with today’s hooks, while a generic additive hook keeps the voice-routing policy outside core.

## Repair Work Prompt

Implement a narrow additive plugin hook for https://github.com/openclaw/openclaw/issues/69307. The observable gap is that final reply text goes directly into `maybeApplyTtsToReplyPayload()` before `parseTtsDirectives()`, while `llm_output` is observation-only and `message_sending` runs after TTS audio is already generated. Add a generic pre-TTS modifying hook, using the existing hook-runner style, that runs only after final reply text exists and before TTS directive parsing/audio generation. Let handlers modify text and provide a maintainable way to skip TTS for that message if the hook contract includes that result. Keep the hook generic: do not add bundled plugin ids, chat-specific voice policy, provider-specific defaults, or core migrations. Decide and document conversation-content gating for non-bundled plugins consistently with existing hook policy. Update hook types/runner tests, final reply dispatch tests, public plugin docs, TTS docs if useful, SDK API generation expectations, and `CHANGELOG.md` because this is user-facing plugin API behavior.
