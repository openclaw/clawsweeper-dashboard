## Summary

Keep open. Current main still has no plugin-modifying hook between final assistant text and TTS directive parsing, and the requested seam needs maintainer API and conversation-access policy review rather than conservative cleanup.

## What This Changes

The issue requests a new plugin hook that can modify final assistant reply text before TTS directive parsing so plugins can route voices per message.

## Best Possible Solution

Add a documented, backward-compatible pre-TTS modifying hook only after maintainer API/security approval, guarded by the appropriate conversation-access policy and covered by hook-runner plus reply-dispatch/TTS tests.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection gives high confidence that current main lacks a before_tts/reply_transform-style modifying seam before TTS processing.

## Solution Assessment

Unclear for the exact API name and result shape. The maintainable direction is a generic documented hook with explicit access gating, not forcing plugins to own reply_dispatch or rely on model prompt instructions.
