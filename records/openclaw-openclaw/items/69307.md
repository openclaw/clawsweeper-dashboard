## Summary

Keep open. Current `main` still has no modifying plugin seam between final reply text and TTS directive parsing, so the per-message voice-routing use case remains unsolved; the adjacent hook-related reports are not duplicates because they cover pre-model or pre-tool routing rather than this pre-TTS transform point.

## What This Changes

Add a modifying plugin hook that can rewrite or skip assistant reply text after model output and before TTS directive parsing for per-message voice routing.

## Best Possible Solution

Add a documented, additive modifying plugin hook, likely `before_tts` or a maintainer-approved equivalent, that runs after final reply text exists and before TTS directive parsing. A ClawHub plugin can then implement per-chat voice routing, but core first needs to expose this missing runtime seam.

## Reproduction Assessment

Not applicable as a runtime bug reproduction because this is a feature request. The missing capability is high-confidence from source inspection: current `main` has no registered hook, runner method, dispatch integration, or docs entry for a pre-TTS text transform.

## Solution Assessment

Yes, the requested direction is the narrowest maintainable solution. A plugin alone cannot inject TTS directives before audio generation with today’s hooks, while a small additive hook matches the project’s plugin-first boundary.

## Repair Work Prompt

Implement the missing pre-TTS modifying plugin hook for https://github.com/openclaw/openclaw/issues/69307. Add an additive hook, likely named `before_tts` unless an existing naming pattern strongly argues for a better name, that runs after final assistant reply text exists and before `maybeApplyTtsToReplyPayload` parses TTS directives. The hook should let plugins rewrite reply text to inject `[[tts:voice=...]]` directives and optionally skip TTS for that message while still delivering the normal text payload. Keep `reply_dispatch`, static TTS config precedence, directive parsing/provider policy, downstream `message_sending`, and existing delivery behavior intact. Update hook types, runner wiring, plugin SDK public surface as needed, docs, focused tests, and `CHANGELOG.md` with a single-line user-facing feature entry crediting `Thanks @IGI30`. Do not move TTS provider policy into core or add bundled voice-routing behavior; this PR should only expose the generic plugin seam.
