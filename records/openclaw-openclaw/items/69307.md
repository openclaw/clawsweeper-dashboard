## Summary

Keep open. Current main still lacks a plugin-modifying seam between final reply payload creation and TTS directive parsing, and the prior ClawSweeper comment correctly treated this as a real plugin API gap rather than a ClawHub-only extension request.

## What This Changes

Request a new modifying plugin hook between assistant reply generation and TTS processing so plugins can inject or suppress per-message TTS directives.

## Best Possible Solution

Add a documented, backward-compatible pre-TTS modifying hook in the plugin SDK, with focused dispatch/TTS tests for directive injection and TTS suppression while keeping voice-routing policy in plugins.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection gives high confidence that the requested pre-TTS modifying seam is absent on current main.

## Solution Assessment

Yes on the direction: a generic pre-TTS modifying hook is the narrowest maintainable way to let plugins inject `[[tts:...]]` directives without owning full reply dispatch. The exact public SDK contract and access gating still need maintainer approval.
