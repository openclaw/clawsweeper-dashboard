## Summary

Keep open. Current main and the latest release still lack a typed modifying hook between final assistant text and TTS directive parsing, and the requested plugin API needs maintainer product and conversation-access security review before implementation.

## What This Changes

Add a plugin hook between final LLM reply creation and TTS directive parsing so plugins can modify reply text or skip TTS for per-message voice routing.

## Best Possible Solution

Add a documented, backward-compatible pre-TTS transform hook only after maintainers approve the hook contract, access policy, and exact TTS pipeline scope.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature/API request. Source inspection gives high confidence that current main lacks a before_tts or reply_transform-style modifying seam before TTS processing.

## Solution Assessment

Unclear for the exact API name, result shape, and skip semantics. The maintainable direction is a generic typed hook with explicit conversation-access gating, not forcing plugins to own reply_dispatch or rely on model prompt instructions.
