## Summary

Keep open: current main and the latest release still reject the bare grammY network envelope in send context, and the open closing PR at https://github.com/openclaw/openclaw/pull/80529 is the active fix path rather than a reason to close this issue early.

## What This Changes

The issue asks for Telegram send-context retry classification to treat bare grammY `Network request for 'X' failed!` errors as recoverable so transient pre-connect failures retry instead of dropping outbound sends.

## Best Possible Solution

Land a focused Telegram retry fix through the open PR or a narrow replacement that classifies dependency-proven pre-connect grammY failures safely, updates send-context tests, and preserves duplicate-delivery protection for ambiguous post-connect sends.

## Reproduction Assessment

Yes for the classifier path: current source and tests show the bare `Network request for 'sendMessage' failed!` message returns false in send context. I did not run a live low-resource Telegram reproduction in this read-only review.

## Solution Assessment

Unclear that the exact optional-regex suggestion is the safest final shape. The maintainable fix should distinguish dependency-proven pre-connect failures from ambiguous non-idempotent send failures and include focused tests for both.
