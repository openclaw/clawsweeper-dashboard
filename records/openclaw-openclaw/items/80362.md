## Summary

Keep open: current main and v2026.6.8 still reject the reported bare grammY network envelope in send-context classification, and the open linked PR is a candidate fix rather than a merged implementation. The remaining blocker is maintainer judgment on Telegram send retry safety, because prior merged work deliberately avoided retrying ambiguous plain send envelopes without nested pre-connect evidence.

## What This Changes

The issue requests that Telegram send retries classify bare grammY `Network request for '<method>' failed!` errors as recoverable so transient network failures do not drop outbound replies.

## Best Possible Solution

Keep this issue as the canonical report while maintainers either land a safe, dependency-backed Telegram retry fix or replace the linked PR with a narrower implementation that proves pre-connect failure without retrying ambiguous sends.

## Reproduction Assessment

Yes for the source-level path: current main explicitly rejects the bare grammY network envelope in send context, and the retry callers depend on these predicates. I did not run a live low-resource Telegram reproduction because this cleanup review is read-only.

## Solution Assessment

No for the broad regex suggestion as-is: it is plausible for the reported symptom but conflicts with the existing strict send-retry safety boundary. The better path is maintainer-approved retry semantics based on grammY’s nested error details or other concrete pre-connect evidence, plus focused tests and live Telegram proof on the linked PR.
