## What This Changes

The issue asks for Telegram send-context retry classification to treat bare grammY `Network request for 'X' failed!` errors as recoverable so transient outbound failures are retried instead of dropped.

## Best Possible Solution

Keep this issue paired with https://github.com/openclaw/openclaw/pull/80529 or a replacement focused Telegram fix that proves nested pre-connect evidence while preserving non-idempotent send safeguards.

## Reproduction Assessment

No high-confidence live reproduction was established for the low-resource Telegram dropped-message behavior. Source inspection does reproduce the helper-level mismatch: current main returns false for the bare send-context grammY envelope while polling context remains true.

## Solution Assessment

No, not as a broad optional-`after` regex alone. The safer solution is to tie send retry eligibility to nested pre-connect evidence, update focused regression coverage, and provide redacted live Telegram transport proof before merge.
