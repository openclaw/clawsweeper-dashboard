## Summary

Keep open: current main still has the source-level bare grammY send-context classifier gap, and the open closing PR is an implementation candidate with proof and retry-safety blockers rather than a merged fix.

## What This Changes

The issue requests Telegram retry classification for bare grammY `Network request for 'X' failed!` errors in send context without weakening duplicate-delivery safeguards.

## Best Possible Solution

Keep this issue open until the linked PR is narrowed and landed, or replace it with a focused fix that uses code-backed pre-connect evidence and live Telegram proof.

## Reproduction Assessment

Partially: source inspection gives a high-confidence reproduction of the classifier mismatch because current tests assert the bare send-context envelope is false. There is not a high-confidence live reproduction of the reporter's low-resource Telegram failure in this review.

## Solution Assessment

No, the broad optional-`after` regex is not clearly the safest fix. The maintainable path should preserve the existing non-idempotent send boundary by requiring nested pre-connect evidence or otherwise proving the retry cannot duplicate visible messages.
