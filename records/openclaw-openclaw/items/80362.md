## Summary

Keep open: current main and the latest release still reject the reported bare grammY send-context envelope, and the linked open fix PR is a candidate path rather than a merged implementation.

## What This Changes

The issue asks for bare grammY `Network request for 'X' failed!` send errors to be classified as recoverable so outbound Telegram messages are retried instead of dropped.

## Best Possible Solution

Resolve the Telegram retry-safety decision, then land or replace the linked PR with a focused fix that preserves duplicate-delivery guardrails and proves the real Telegram transport path.

## Reproduction Assessment

Yes for the source-level classifier path: current main explicitly expects the bare `Network request for 'sendMessage' failed!` envelope to be false in send context. I did not run a live low-resource Telegram reproduction in this read-only review.

## Solution Assessment

No for the optional-regex fix as-is; making the outer envelope retryable is broader than the safest current boundary. The better fix is a maintainer-approved retry rule based on grammY nested `HttpError.error` evidence or an explicit decision to accept bare-envelope send retries, with regression tests and live Telegram proof.
