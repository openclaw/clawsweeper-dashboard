## Summary

Keep open: current main still has the reported source-level Telegram classifier mismatch, and an open closing PR already owns the implementation path but has review blockers and missing real Telegram transport proof.

## What This Changes

The issue requests that Telegram classify bare grammY `Network request for 'X' failed!` errors as recoverable for send-context retry without weakening non-idempotent send safeguards.

## Best Possible Solution

Keep the issue open until https://github.com/openclaw/openclaw/pull/80529 is narrowed and landed, or replace it with a focused retry fix that requires nested pre-connect evidence and includes redacted real Telegram transport proof.

## Reproduction Assessment

Yes for the narrow helper mismatch by source inspection: current main expects a bare `Network request for 'sendMessage' failed!` to be false for send context and true for polling context. No high-confidence live reproduction was established for the low-resource Telegram drop itself.

## Solution Assessment

No, the broad optional-`after` regex in the linked PR is not clearly the safest fix because grammY can wrap non-transport transformer failures in the same bare `HttpError` envelope. A safer solution should require nested pre-connect evidence or otherwise prove the live transport retry path.
