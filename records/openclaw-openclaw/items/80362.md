## What This Changes

The issue asks Telegram retry classification to treat bare grammY `Network request for ... failed!` send-context errors as recoverable so transient outbound sends retry instead of dropping.

## Best Possible Solution

Keep this issue open until https://github.com/openclaw/openclaw/pull/80367 is narrowed and landed, or replace it with a focused retry fix that proves pre-connect safety while preserving non-idempotent send safeguards.

## Reproduction Assessment

No high-confidence live reproduction was established for the low-resource Telegram failure. Source inspection does reproduce the narrow classifier mismatch because current main keeps the bare send-context envelope false in the helper test contract.

## Solution Assessment

No, the broad optional-`after` regex is not clearly the safest fix because grammY can wrap transformer errors in the same bare `HttpError` envelope. A safer solution should require nested pre-connect evidence or provide live proof for the broader match.
