## Summary

Keep open: current main still has the reported send-context classifier behavior, and https://github.com/openclaw/openclaw/pull/80529 is an open closing PR with review/proof blockers rather than a merged fix. Prior cleanup on https://github.com/openclaw/openclaw/issues/74203 also established that a plain grammY envelope alone should not be retried for non-idempotent sends without nested pre-connect evidence.

## What This Changes

Request to treat bare grammY `Network request for ... failed!` Telegram errors as recoverable for send context so Telegram outbound operations retry instead of dropping.

## Best Possible Solution

Keep this issue open until https://github.com/openclaw/openclaw/pull/80529 is fixed and landed, or replace it with a focused Telegram retry patch that proves pre-connect safety while preserving non-idempotent duplicate-delivery safeguards.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection does reproduce the narrow classifier mismatch on current main: the bare grammY envelope is false for send context, while polling and `failed after` cases still pass.

## Solution Assessment

No, not as the broad optional-`after` regex suggested in the issue. The safer solution should require nested pre-connect evidence or adjust the strict safe-send path directly, with real Telegram transport proof before merge.
