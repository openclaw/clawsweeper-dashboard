## Summary

Keep open. The performance direction is still potentially useful, and the author’s follow-up commits addressed the earlier manifest/comment/message-hint review items, but the latest PR head still conflicts with current main’s prompt trust boundary by appending group context as final trusted system text while that branch still includes channel-sourced group names and participant labels there.

## What This Changes

The PR reorders agent system-prompt sections for better KV-cache prefix stability, adds a dynamic runtime tail, moves tool/skills/memory/group context sections, sanitizes context-file prompt headers, gates message hints, and updates prompt tests.

## Best Possible Solution

Rebase the cache-ordering work over current main, preserve `SYSTEM_PROMPT_CACHE_BOUNDARY`, keep channel/user-sourced group subject/member labels in the user-role untrusted context path, and rerun focused prompt/cache-boundary tests plus the changed gate before merge.

## Reproduction Assessment

Yes for the review finding: inspect or render the PR head for a group-chat prompt and observe that `extraSystemPrompt` is appended last while PR-head `buildGroupChatContext` interpolates `GroupSubject` and `GroupMembers`. The performance claim is reproducible separately by rendering paired prompts and measuring the first differing byte after changing runtime/context inputs.

## Solution Assessment

No, not as-is. The volatility-ordering idea is plausible, but the current PR branch should not solve cache stability by making untrusted group metadata the final trusted system section; a rebase should preserve current main’s untrusted metadata split or add equivalent trusted delimiters/guardrails.
