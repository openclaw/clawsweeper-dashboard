## Summary

Current main already fixes this update-restart failure mode: live package updates hard-respawn the Gateway into a fresh process, while shell package updates stop, refresh, restart, and version-check the managed service. The proof commit is on main only and is not contained in v2026.5.2.

## What This Changes

The issue asks for update semantics that fully restart or re-exec a running Gateway after package files are replaced so stale hashed dist chunks cannot be imported.

## Best Possible Solution

Keep the current-main hard process respawn and managed-service restart/version verification, then ship it in the next release for users affected on v2026.5.2.

## Reproduction Assessment

Yes, from source inspection rather than a live package-update run. The reported stale-chunk failure matches the old in-memory Gateway/package-swap hazard now covered by update-specific hard respawn code and tests on current main.

## Solution Assessment

Yes. The current main approach is the narrow maintainable fix: shell updates manage the service around the package swap, and live Gateway updates force a fresh OS process instead of relying on in-process reload.
