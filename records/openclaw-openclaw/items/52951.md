## Summary

Keep this PR open: the `tools.fs.roots` feature is still unique and not present on current main, but the branch currently changes an existing restrictive-profile filesystem policy in a way that needs correction before merge.

## What This Changes

The PR adds a new `tools.fs.roots` config surface and threads root-scoped host filesystem/media access through FS tools, `apply_patch`, outbound delivery, channel adapters, docs, generated baselines, and tests.

## Best Possible Solution

Keep the `tools.fs.roots` contract only if maintainers want this core security surface, preserve the existing explicit-`alsoAllow` policy for restrictive profiles, and require real behavior proof plus exact-head validation before landing.

## Reproduction Assessment

Not applicable as a feature PR rather than a bug report. Source inspection confirms current main lacks `tools.fs.roots`, and the blocking concern is source-reproducible by comparing current policy tests/docs with the PR head.

## Solution Assessment

No, not as-is. The root-scoped design is coherent, but the PR should not make an explicit `tools.fs` block bypass the current restrictive-profile `tools.alsoAllow` contract.
