## Summary

Keep this PR open, but it is not merge-ready: the branch still reintroduces implicit filesystem grants that current main removed for restrictive profiles, and the external PR still lacks real behavior proof for the new host-root/media behavior.

## What This Changes

The branch adds `tools.fs.roots` with per-root kind/access modes and threads it through host filesystem tools, `apply_patch`, media loading, outbound delivery, generated config docs, and multiple channel adapters.

## Best Possible Solution

Keep `tools.fs.roots` as a constraint on already-granted filesystem/media access, require explicit profile or `alsoAllow` grants, and rebase with real allowed/denied host-root media proof before merge.

## Reproduction Assessment

Yes for the blocking PR regression: source inspection of the PR head shows `tools.fs` presence adds filesystem tools back into the effective allow list. I did not run tests because this review is read-only, but the conflicting current-main contract is clear in source and history.

## Solution Assessment

No: the proposed roots policy should not grant filesystem tools by config-section presence. The maintainable path is to preserve the current-main explicit-grant contract and make roots a boundary on already-authorized access.
