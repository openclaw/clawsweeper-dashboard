## Summary

Keep open: current main does not yet provide granular per-agent filesystem roots, but this PR is not merge-ready because the current head reintroduces implicit filesystem grants for restrictive profiles and still lacks real behavior proof for the new host-root/media paths.

## What This Changes

This PR adds `tools.fs.roots` config with file/dir roots and `ro`/`rw` access, then threads that policy through host FS tools, `apply_patch`, local media, outbound delivery, prompt-image loading, config schema/docs, and channel adapters.

## Best Possible Solution

Keep the granular roots feature open for maintainer review, but require the branch to preserve the explicit profile/`alsoAllow` grant contract, rebase cleanly, and provide redacted real host-root plus Telegram/media proof before merge.

## Reproduction Assessment

Yes, from source inspection: the PR head makes `tools.fs` presence grant effective filesystem access under a messaging profile, while current-main code and tests require that to remain false without explicit `alsoAllow`. I did not run tests because this review is read-only.

## Solution Assessment

No, not as currently implemented. The right shape is to add roots as a constraint on already-granted filesystem access, not as a grant, then prove the resulting allowed and denied paths in a real setup.
