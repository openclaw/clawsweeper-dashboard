## Summary

Keep open: this PR is still useful but not merge-ready because the current head can widen an explicit empty `tools.fs.roots` deny-all policy back to workspace media access, the broad filesystem/security API direction still needs maintainer acceptance, and the supplied real behavior proof does not cover the current head's later outbound and Telegram-visible paths.

## What This Changes

The PR adds `tools.fs.roots` with per-root `dir`/`file` and `ro`/`rw` modes, threads it through host filesystem tools, `apply_patch`, local media and channel reply delivery, updates config metadata, and exposes a plugin SDK media helper.

## Best Possible Solution

Preserve explicit deny-all roots through every outbound media path, add current-head proof for host roots plus representative outbound/sandbox/Telegram media behavior, and have maintainers choose whether this broad roots contract should land alongside or instead of the narrower workspace-alias and filesystem-policy paths.

## Reproduction Assessment

Yes. Source inspection on the PR head shows `tools.fs.roots: []` can produce `baseLocalRoots: []`, and then `appendWorkspaceDirToLocalRoots` turns that into a workspace allowlist when a workspace is resolved.

## Solution Assessment

No as-is. A broad roots allowlist is plausible, but this implementation must preserve deny-all semantics and needs maintainer security/API approval against the narrower workspace-alias and filesystem-policy alternatives before it is the right merge path.
