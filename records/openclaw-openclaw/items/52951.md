## Summary

Keep this PR open, but it is not merge-ready: the branch adds useful `tools.fs.roots` work, yet it reintroduces an implicit filesystem access grant that current main deliberately removed, and the external contributor still has not supplied inspectable real behavior proof.

## What This Changes

Adds `tools.fs.roots` with per-root `dir/file` and `ro/rw` access and threads root-scoped host filesystem/media access through agent tools, outbound delivery, channel plugins, generated config/docs, and tests.

## Best Possible Solution

Land a maintainer-approved roots design that treats roots as containment, preserves existing explicit tool-grant requirements, documents compatibility/security expectations, and includes real host, sandbox, and channel proof.

## Reproduction Assessment

Yes, source-reproducible: current main's tool-fs-policy tests assert `tools.fs` alone must not enable root expansion, while PR head changes that policy and wires it into host media reads. I did not run tests because this review is read-only.

## Solution Assessment

No; root-scoped containment is a plausible direction, but making any `tools.fs` block grant read/write/edit is not the narrowest maintainable fix. The safer path is to keep access grants in the existing profile/alsoAllow/tool-policy layer and let roots only constrain already-granted access.
