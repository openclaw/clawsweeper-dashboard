## Summary

Keep this PR open, but it is not merge-ready: the branch still has a concrete security-boundary regression around explicit empty filesystem roots, and the supplied real behavior proof is mostly older host-tool proof rather than current-head proof for the later outbound/channel changes.

## What This Changes

The PR adds `tools.fs.roots`, a global/per-agent host filesystem allowlist with `dir`/`file` and `ro`/`rw` modes, and threads it through file tools, `apply_patch`, media loading, outbound delivery, config docs, and the plugin SDK media runtime.

## Best Possible Solution

Preserve explicit deny-all roots through every outbound media path, add a regression with `tools.fs.roots: []` plus an agent workspace, then get maintainer security/API approval for whether the broad roots contract should land instead of or alongside the narrower workspace-alias design.

## Reproduction Assessment

Yes, source-level. On the PR head, `tools.fs.roots: []` produces `baseLocalRoots: []`, but `resolveAgentScopedOutboundMediaAccess` then appends `resolvedWorkspaceDir`, so a path intended to be deny-all can regain workspace local-media access when an agent workspace is present.

## Solution Assessment

No. A broad roots allowlist is a plausible direction, but this implementation must first preserve deny-all semantics and then get maintainer approval against the narrower workspace-alias alternative.
