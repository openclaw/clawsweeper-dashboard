## Summary

Keep open: the broad filesystem-root feature is not merge-ready because source inspection still shows root-policy gaps in outbound media and sibling media-generation tools, plus the config/security direction needs maintainer approval.

## What This Changes

The PR adds a new `tools.fs.roots` config surface with per-root file/dir and read/write modes, and threads it through host file tools, media loading, outbound delivery, prompt-image loading, generated config/API baselines, docs/help text, and tests.

## Best Possible Solution

Fix the concrete root-enforcement gaps, add proof for representative media and Telegram-visible delivery paths, then have maintainers make an explicit product/security call on this broad config surface versus the narrower alias work.

## Reproduction Assessment

Yes. Source inspection is enough to reproduce the two blocker paths: `tools.fs.roots: []` is converted back into an agent workspace local root in outbound media access, and video/music reference loaders never pass configured roots into the updated media-root helper.

## Solution Assessment

No, not as submitted. The concept may be useful, but the current implementation is not the narrowest safe solution until sibling media callers obey the same policy, deny-all remains deny-all, and maintainers choose the config/security direction.
