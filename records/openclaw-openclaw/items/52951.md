## Summary

Keep open: the feature is still unique work, but the latest head reintroduces implicit filesystem grants from `tools.fs`, conflicting with current main's security contract, and the external PR still lacks real behavior proof for the new host filesystem/media path.

## What This Changes

Adds a `tools.fs.roots` config surface with dir/file and ro/rw modes, then threads root-scoped host filesystem and media access through agents, outbound delivery, prompt images, docs, and tests.

## Best Possible Solution

Keep the root-scoped host filesystem design as a limiter only: `tools.fs.roots` should constrain filesystem and media reads after the active profile or `tools.alsoAllow` has already granted the relevant tools, then the branch needs redacted host-root/media and Telegram-visible proof before merge.

## Reproduction Assessment

Yes for the blocking PR regression: the PR-head source and tests show `tools.fs` adding filesystem grants under `messaging`, while current main tests require explicit `alsoAllow`. Not applicable as a current-main bug because this is a new feature PR.

## Solution Assessment

No, not as-is. The `tools.fs.roots` direction may be maintainable, but it must preserve the current explicit filesystem grant contract and provide real behavior proof before it is safe to merge.
