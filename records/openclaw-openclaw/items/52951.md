## Summary

## What This Changes

The PR adds global and per-agent `tools.fs.roots` entries with `dir`/`file` and `ro`/`rw` access modes, then threads that policy through host filesystem tools, media loading, outbound delivery, generated config baselines, and bundled channel adapters.

## Best Possible Solution

Land a maintainer-sponsored filesystem/media root contract only after preserving `roots: []` deny-all semantics with `workspaceDir`, refreshing the branch cleanly, and documenting that shell `exec` remains outside this boundary.

## Reproduction Assessment

Yes, source-level for the blocking PR bug: with `tools.fs.roots: []` and a `workspaceDir`, the PR head's resolver builds `localRoots` from an empty root set and then appends the workspace, while the existing deny-all test omits `workspaceDir`. I did not run tests because this review is read-only.

## Solution Assessment

No, not as-is: the roots design is a plausible candidate, but it must first preserve deny-all semantics, refresh against current main, and receive maintainer approval for the new config and SDK security contract. It is also only a partial answer to the broader allowed/deny plus exec filesystem policy request.
