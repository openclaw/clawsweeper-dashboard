Keep this PR open: the requested `tools.fs.roots` capability is not on current main and is useful, but the patch currently reopens current main's explicit filesystem opt-in boundary under restrictive profiles and still lacks after-fix real behavior proof.

The branch adds `tools.fs.roots` with per-root file/dir and read/write access, then threads that policy through host filesystem tools, media loading, outbound delivery, prompt images, docs, generated baselines, and tests.

## Best Possible Solution

Preserve the root-scoped filesystem design only after rebasing onto current main, keeping explicit `alsoAllow` required for restrictive profiles, and adding redacted real host-root/media plus Telegram-visible proof.

## Reproduction Assessment

Not applicable as a user bug reproduction, but the blocking PR regression is source-reproducible. Current main's tests require explicit filesystem opt-in, while the PR head adds filesystem grants from `tools.fs` presence.

## Solution Assessment

No, not as-is. The feature direction may be maintainable, but the implementation must preserve current main's explicit filesystem opt-in contract before it is a safe solution.
