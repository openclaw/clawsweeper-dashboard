## Summary

Keep open. Current main does not implement `tools.fs.roots`, but this PR is not merge-ready because it reintroduces an implicit filesystem permission grant that current main and the merged security fix for the same policy class explicitly removed.

The PR adds `tools.fs.roots` with per-root `dir/file` and `ro/rw` modes, then threads that policy through host filesystem tools, `apply_patch`, local media loading, outbound reply delivery, prompt-image loading, generated config/docs, SDK exports, and channel tests.

## Reproduction Assessment

Yes for the patch defect: source inspection shows `tools.fs` now implicitly adds `read`, `write`, and `edit` to `profileAlsoAllow`, while current main has tests preventing `tools.fs` alone from granting filesystem reach. The overall PR is a new feature, not a bug reproduction.

## Solution Assessment

No as currently written: the roots model is plausible, but it must preserve the existing tool-permission gate and needs maintainer approval for the new config/security boundary before it is the best path.
