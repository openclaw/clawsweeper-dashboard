## Summary

Keep open: this is a meaningful filesystem-policy feature, but the branch currently conflates a path allowlist with permission grants and reverses current main's explicit #47487 guard that `tools.fs` presence alone must not grant read access under restricted tool profiles.

## What This Changes

The PR adds `tools.fs.roots` as per-agent/global filesystem roots with read-only/read-write access modes and threads that policy through file tools, media loading, outbound delivery, channel adapters, docs, and Plugin SDK surfaces.

## Best Possible Solution

Land only after roots remain a path allowlist behind explicit read/write/edit permission, or maintainers deliberately approve the new permission semantics with docs, tests, and upgrade guidance, plus a refreshed branch and real host-agent/media proof.

## Reproduction Assessment

Yes from source: current main denies root expansion for restricted profiles with `tools.fs` alone, while PR head explicitly flips that behavior; no live configured-agent run was performed in this read-only review.

## Solution Assessment

No: a typed roots API is plausible, but this implementation is not the best fix while it changes the permission model and adds public config/SDK surface without a settled maintainer contract.
