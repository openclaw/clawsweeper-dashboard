## Summary

Keep open: current main still accepts `threadName` in the message-tool schema but Discord `send` paths do not propagate, consume, rename, or warn on that parameter, while existing Discord channel-edit code already has a name PATCH path.

## What This Changes

Request that Discord `message` sends to an existing thread stop silently ignoring `threadName`, preferably by renaming the thread or warning when it is unsupported.

## Best Possible Solution

Keep this open and fix the Discord-owned send path so `threadName` on a send to an existing Discord thread is honored through the existing rename helper or reported as not applied, with focused regression coverage.

## Reproduction Assessment

Yes, there is a high-confidence source-level reproduction: invoke `message` with `action:"send"`, `channel:"discord"`, an existing thread target, and `threadName`; current main sends the message without reading or forwarding `threadName`. I did not run a live Discord rename because this review is read-only.

## Solution Assessment

Yes, the maintainable fix belongs in the Discord plugin/action send pipeline, not in core policy: use the existing Discord channel-edit/name path when the resolved target is a thread, or return a structured warning when the parameter cannot be applied.
