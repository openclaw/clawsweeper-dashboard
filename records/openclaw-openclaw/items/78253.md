## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve the legacy import root:** `extensions/qqbot/src/engine/utils/platform.ts:86`
  - body: `getQQBotDataPath()` is not only the live data directory display path: current main uses it to import legacy credential backups, sessions, known users, and ref-index files from the old OS-home tree. Redirecting it to `QQBOT_DATA_DIR` means a user who adds the env var after already running QQBot can skip those legacy imports, including credential recovery data. Add a separate validated current data-dir helper or explicit legacy candidates instead of changing this shared legacy helper in place.
  - confidence: 0.9
- **[P1] Reject blank data-dir overrides:** `extensions/qqbot/src/engine/utils/platform.ts:92-93`
  - body: A whitespace-only `QQBOT_DATA_DIR` is truthy here, then `normalizePath()` trims it to an empty string. Calls like `getQQBotDataPath("data")` then resolve under the process cwd, so QQBot storage and credential lookup can move outside the intended data root. Treat blank/sentinel values as unset or fail closed before returning the override.
  - confidence: 0.88
- **[P2] Finish the diagnostic path fix:** `extensions/qqbot/src/engine/utils/platform.ts:86`
  - body: The linked report is about the Gateway diagnostic still showing or warning about the old Chinese home path after `QQBOT_DATA_DIR` is set. Changing this helper updates the displayed data dir, but diagnostics still unconditionally warns on the OS home path and tells the user to set the same env var even when the override is valid. The fix needs diagnostic logic and proof for the reported Windows flow.
  - confidence: 0.8
