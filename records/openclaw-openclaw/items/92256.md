## Summary

Keep open: current main and latest release still authorize `sessions.send` through broad `operator.write`, and the device-token path filters operator-role grants to `operator.*`, so a `sessions.send`-only device token is not implemented. The request is valid security/auth-provider product work, but the exact restricted-plugin shape should not be automated before maintainers choose the least-privilege token model.

## What This Changes

Request adding `sessions.send`-scoped device pairing plus a restricted pairing mode for least-privilege A2A gateway messaging.

## Best Possible Solution

Add one canonical least-privilege gateway capability for session message delivery, then integrate issuance, revocation, and persistence through the existing core device-token lifecycle rather than a parallel restricted plugin path.

## Reproduction Assessment

Yes for the capability gap: current source maps `sessions.send` to `operator.write`, and device-pairing approval rejects `sessions.send` as an operator-role token scope. This is a missing feature rather than broken documented behavior.

## Solution Assessment

No for the exact restricted-plugin shape; device pairing and token lifecycle are already core gateway functionality, so the least-privilege goal should be designed as one canonical core authorization capability.
