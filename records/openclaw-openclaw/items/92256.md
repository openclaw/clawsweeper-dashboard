## Summary

Keep open: current main and v2026.6.8 still have only broad `operator.write` authorization for `sessions.send`, and no send-only device credential path; the request is useful but needs maintainer product/security design before implementation.

## What This Changes

Request a least-privilege device pairing/token path that can issue `sessions.send`-only credentials and a restricted pairing mode for fleet A2A messaging.

## Best Possible Solution

Define one Gateway-owned least-privilege session-send capability and issuance path, integrate it with device-token lifecycle/revocation and cross-gateway sender-auth decisions, then add CLI/docs only after the trust model is settled.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.8 classify `sessions.send` under `operator.write` and expose no `sessions.send`-only operator scope or device-token path. I did not run a live fleet pairing reproduction.

## Solution Assessment

No, not as the exact restricted-plugin shape. The durable fix should extend the Gateway authorization and device-token contract rather than add a parallel plugin-owned pairing policy.
