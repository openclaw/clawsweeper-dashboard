## Summary

This remains a valid least-privilege security feature: both current main and v2026.6.5 authorize `sessions.send` through broad `operator.write` credentials, with no send-only device scope. The requested restricted-plugin design does not match current core ownership, and no canonical issue or merged PR supersedes the remaining credential work.

## What This Changes

The issue requests a device credential restricted to `sessions.send` and a restricted pairing mode for fleet gateway-to-gateway messaging.

## Best Possible Solution

Add one canonical least-privilege core gateway capability or narrow role for session message delivery, integrating issuance, storage, rotation, revocation, protocol documentation, and upgrade-safe authorization tests with the existing device-pairing lifecycle rather than introducing a parallel plugin-owned pairing path.

## Reproduction Assessment

Yes for the capability gap: current source and v2026.6.5 map `sessions.send` to `operator.write`, while the accepted scope union has no send-only credential. This is a missing feature, not a failing documented contract.

## Solution Assessment

No, not exactly as proposed: the send-only capability is well motivated, but device pairing and token authorization are currently core-owned, so a restricted plugin mode would duplicate the security boundary rather than extend its canonical path.
