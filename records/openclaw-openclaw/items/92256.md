## Summary

Keep open: current main and v2026.6.8 support `sessions.send`, but only through broad `operator.write` authorization and existing device tokens still use operator-scope contracts rather than a send-only credential.

## What This Changes

Request to add a least-privilege `sessions.send` credential path for device-paired A2A messaging, plus a restricted pairing mode separate from broad device management.

## Best Possible Solution

Define one Gateway-owned least-privilege credential model for session message delivery, then wire it through device-token issuance or the scoped-principal design with docs, tests, revocation, and migration/doctor behavior.

## Reproduction Assessment

Yes for the capability gap by source inspection: current main and v2026.6.8 authorize `sessions.send` with `operator.write` and do not expose a send-only device-token scope. This is not a broken documented behavior reproduction; it is missing least-privilege capability proof.

## Solution Assessment

No for the exact restricted-plugin proposal. The maintainable fix should extend the core gateway/device-token or scoped-principal authorization contract rather than add a parallel plugin-owned pairing mode.
