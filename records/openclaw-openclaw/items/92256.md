## Summary

Keep open: current main and the latest release still authorize `sessions.send` through broad `operator.write` credentials and do not provide a send-only device or gateway token. The transport half is handled by the closed related issue, but the least-privilege credential and trust-boundary design remains real security/product work.

## What This Changes

Request a restricted device-pairing or token-issuance path that can mint credentials limited to `sessions.send` for authenticated cross-gateway A2A messaging.

## Best Possible Solution

Add one core-owned least-privilege gateway capability for session message delivery, integrate it with device-token issuance/rotation/revocation, and coordinate it with the open scoped-token and sender-authentication designs.

## Reproduction Assessment

Yes for the capability gap. Source inspection shows `sessions.send` is still an `operator.write` method and device operator tokens accept only the existing `operator.*` scope set.

## Solution Assessment

No for the exact restricted-plugin shape. The least-privilege goal is valid, but the maintainable path is a core gateway/device-token authorization contract rather than a second plugin-owned pairing mode.
