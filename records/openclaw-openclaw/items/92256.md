## Summary

This remains a valid least-privilege feature request: current main grants `sessions.send` through the broad `operator.write` scope and has no send-only device credential. The proposed restricted-plugin shape does not match current core ownership, and the security model must be coordinated with the same-author open transport issue https://github.com/openclaw/openclaw/issues/92255.

## What This Changes

Request a least-privilege device token that can invoke only `sessions.send`, plus a restricted pairing enablement mode for fleet A2A messaging.

## Best Possible Solution

Add one canonical least-privilege gateway capability or narrowly scoped role for session message delivery, integrate issuance and revocation with the existing core device pairing lifecycle, and design it jointly with the remote `sessions.send` transport path.

## Reproduction Assessment

Yes for the stated capability gap: current source and the v2026.6.5 release classify `sessions.send` under broad `operator.write` authorization and expose no send-only credential path. This is a feature gap, not broken documented behavior.

## Solution Assessment

No, the exact restricted-plugin proposal is not the best architectural fit. The least-privilege goal should extend the existing core gateway authorization and device-token contracts rather than introduce a second plugin-owned pairing path.
