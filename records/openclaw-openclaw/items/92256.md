## Summary

Keep open: current main and v2026.6.9 still authorize `sessions.send` through broad `operator.write` and current device tokens only use the existing `operator.*` scope model, so the requested send-only credential/restricted pairing mode remains unimplemented and needs maintainer security/product direction.

## What This Changes

Request to add a restricted device-pairing path that can issue `sessions.send`-only tokens for fleet A2A messaging without broad operator credentials.

## Best Possible Solution

Define one Gateway-owned scoped credential model that covers `sessions.send` and related token/principal needs, then wire device pairing to it only if maintainers approve that issuance path.

## Reproduction Assessment

Yes, for the source-level capability gap: current main authorizes `sessions.send` through `operator.write` and device tokens accept only the current `operator.*` scope vocabulary. This is not a broken documented behavior reproduction; it is proof that the least-privilege `sessions.send` token capability is absent.

## Solution Assessment

No for the exact restricted-plugin mode as the default fix; it risks a parallel auth owner path. The stronger solution is a single Gateway-owned scoped principal/token model that device pairing can use if maintainers approve that issuance path.
