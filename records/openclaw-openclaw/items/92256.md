## Summary

Keep open. Current main and v2026.6.6 still authorize `sessions.send` through broad `operator.write`, and device-token scopes are still role/operator scoped rather than a send-only credential; the related open auth/provenance issues are useful context but not safe replacements for this device-token request.

## What This Changes

The issue requests scoped device pairing that can mint a `sessions.send`-only token for gateway-to-gateway messaging, plus a restricted pairing mode that avoids exposing broader device management.

## Best Possible Solution

Define one gateway-owned least-privilege session-delivery capability or scoped-token primitive and wire it through the existing device pairing, rotation, revocation, docs, and tests instead of adding a parallel device plugin path.

## Reproduction Assessment

Yes for the capability gap by source inspection. Current main and v2026.6.6 require `operator.write` for `sessions.send`, and the device-token contract has no `sessions.send`-only scope.

## Solution Assessment

No for the exact restricted-plugin proposal. The least-privilege goal is valid, but the maintainable fix should extend the core gateway authorization and device-token lifecycle rather than create a second plugin-owned pairing path.
