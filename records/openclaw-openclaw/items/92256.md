## Summary

Keep open: current main and v2026.6.6 still authorize `sessions.send` through broad `operator.write`, with no send-only device credential. The capability is valid, but its permanent authorization and pairing contract requires explicit security and product judgment.

## What This Changes

Add a device credential restricted to `sessions.send` and a security-conscious pairing path for fleet gateway-to-gateway messaging.

## Best Possible Solution

Define one canonical least-privilege gateway capability or narrowly scoped role for session message delivery, then integrate its issuance, persistence, rotation, revocation, and method checks into the existing core device lifecycle rather than creating a parallel restricted plugin mode.

## Reproduction Assessment

Yes at the source level: current main only accepts broad operator scopes, and `sessions.send` requires `operator.write`. This clearly establishes the least-privilege capability gap without needing a failing runtime test.

## Solution Assessment

No, the proposed separate restricted-plugin mode is not the best permanent boundary. The narrowest maintainable solution is a canonical core authorization capability reused by the existing pairing and token lifecycle, with the CLI flag added only after that contract is approved.
