## Label Justifications

- `P1`: The report describes a recent regression that can make every OpenRouter-backed agent turn fail for affected users.
- `impact:auth-provider`: The failure is tied to provider/model metadata and OpenRouter request routing, specifically the resolved output-token cap sent to the provider.

## Summary

Current main already fixes the reported OpenRouter context-overflow shape by clamping OpenAI-compatible completions output tokens to the remaining context budget after payload shaping; the fix is main-only and not in v2026.5.22.

## What This Changes

The issue requests that OpenRouter/default model output-token limits stop using the full context window when prompt and tool tokens already consume part of that window.

## Best Possible Solution

Keep the current main payload-builder clamp and ensure the next release carries it for OpenRouter users.

## Reproduction Assessment

Yes. The reporter's OpenRouter log and the v2026.5.22 source show the full model max token value being sent as the output cap; current main source and regression tests show that path is now clamped.

## Solution Assessment

Yes. Fixing the canonical OpenAI-compatible completions payload builder is the narrow maintainable solution because it covers OpenRouter and other proxy-like routes without requiring users to configure every model by hand.
