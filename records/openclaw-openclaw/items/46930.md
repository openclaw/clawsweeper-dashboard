## Summary

Keep open. Current main still sends the best-effort multi-gateway advisory through the CLI error channel, while the docs describe intentional isolated multi-gateway setups and classify related probe output as warnings. I did not find an exact open implementation PR or merged fix for this severity downgrade.

## What This Changes

Downgrade the human-facing multi-gateway detection advisory from error-level output to warn/info-level output.

## Best Possible Solution

Downgrade only the non-empty extraServices advisory branch in printDaemonStatus to warning/info-style human output, preserving error output for real service, port, RPC, probe, and config failures. Add a focused unit test for the non-error channel and a user-facing changelog fix crediting the reporter.

Yes. A deterministic source-level reproduction is to call printDaemonStatus with json: false and a non-empty extraServices array; current main routes that advisory through defaultRuntime.error(...).

## Solution Assessment

Yes. The narrowest maintainable fix is to change only this advisory output path, likely using defaultRuntime.log with warnText, without changing gateway detection, JSON status shape, doctor remediation, or unrelated diagnostics.
