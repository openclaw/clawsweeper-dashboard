## Summary

Current main still leaves the Telegram empty-response/failure fallback reachable for `message_tool_only` turns when the normal delivery lane records a non-silent skipped or failed payload, so this issue remains valid and should stay open for a narrow Telegram dispatch fix.

## What This Changes

The issue requests that Telegram suppress the generic empty-response fallback when a group-visible reply was intentionally delivered through the message tool and the final assistant text is `NO_REPLY`.

## Best Possible Solution

Add the existing `message_tool_only` suppression to the Telegram failure/empty-response fallback branch and cover the skipped/failed non-silent delivery-summary shape with a focused regression test.

## Reproduction Assessment

Yes for source reproduction: current main shows the failure fallback branch can be reached independently of the existing `message_tool_only` silent-fallback suppression. I did not run a live Telegram replay in this read-only review.

## Solution Assessment

Yes, the suggested fix is the narrowest maintainable path: reuse the existing `message_tool_only` suppression guard in the sibling empty-response fallback branch and add a focused regression for the non-silent skipped/failed delivery-lane case.
