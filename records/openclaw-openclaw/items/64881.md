## Summary

Keep open: current main still treats missing `fullAccessAvailable` as available when elevated exec is enabled and allowed, and the auto-reply and command prompt paths still feed the resolver config-only state rather than a provider/runtime capability signal.

## What This Changes

The issue requests real provider/runtime-backed `fullAccessAvailable` signals, fail-closed handling for ambiguous elevated exec state, and regression coverage across prompt paths.

## Best Possible Solution

Define a generic provider/runtime full-access capability contract, thread it into auto-reply and command prompt construction, then update resolver and prompt tests so ambiguous input fails closed under an agreed upgrade policy.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: call `resolveEmbeddedFullAccessState` with enabled and allowed elevation but no `fullAccessAvailable`, and current main returns available; the existing resolver test asserts that behavior.

## Solution Assessment

Unclear. The requested fail-closed direction is sound, but the best maintainable fix needs maintainer agreement on the provider/runtime capability contract and compatibility behavior before changing defaults.
