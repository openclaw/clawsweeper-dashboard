## Summary

Keep open: current `main` still treats missing `fullAccessAvailable` as available whenever elevated exec is enabled and allowed, so the reported false `/elevated full` guidance gap remains valid.

## What This Changes

Request `fullAccessAvailable` to come from a real provider/runtime capability signal, fail closed when ambiguous, and add regression coverage.

## Best Possible Solution

Define a generic provider/runtime full-access capability contract, thread it into auto-reply and command prompt construction, then update resolver and prompt tests so ambiguous input fails closed.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: call `resolveEmbeddedFullAccessState` with `{ enabled: true, allowed: true, defaultLevel: "full" }` and no `fullAccessAvailable`; current `main` returns available and the unit test asserts that behavior.

## Solution Assessment

No, current `main` is not the best solution because it equates configured permission with runtime capability. The requested fail-closed direction is sound, but the maintainable fix should start with a generic provider/runtime capability seam.
