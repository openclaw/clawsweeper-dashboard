## Summary

Keep open: current main still derives `fullAccessAvailable` from enabled-and-allowed elevated config rather than a real provider/runtime capability signal, and the focused resolver test still locks in that optimistic fallback.

## What This Changes

Request that embedded-agent `/elevated full` guidance be backed by a real runtime capability signal, fail closed when ambiguous, and gain provider/runtime regression coverage.

## Best Possible Solution

Add a generic elevated full-access runtime capability source, thread it through auto-reply and command prompt construction, make ambiguous resolver input unavailable with an explicit blocked reason, and cover resolver, prompt, and provider/runtime cases with focused tests.

## Reproduction Assessment

Yes. A source-level reproduction is to call `resolveEmbeddedFullAccessState` with `{ enabled: true, allowed: true, defaultLevel: "full" }` and no `fullAccessAvailable`; current main returns available, and the existing unit test asserts that behavior.

## Solution Assessment

No, current main is not the best solution because it treats configuration permission as capability. The requested fail-closed direction is sound, but maintainers should choose the generic provider/runtime capability seam before changing behavior broadly.
