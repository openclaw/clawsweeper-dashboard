## Summary

Keep open: current main still derives full-access availability from elevated exec config when no explicit provider/runtime signal is present, so the reported false `/elevated full` guidance path remains source-reproducible.

## What This Changes

Thread `fullAccessAvailable` from a real provider/runtime capability signal, fail closed on ambiguous input, and add regression coverage for resolver, prompt, and provider/runtime cases.

## Best Possible Solution

Add a generic provider/runtime full-access capability source, thread it through auto-reply and command prompt construction, make ambiguous resolver input unavailable, and cover the resolver, prompt, and representative provider/runtime cases with focused tests.

## Reproduction Assessment

Yes. A source-level reproduction is to call `resolveEmbeddedFullAccessState` with `{ enabled: true, allowed: true, defaultLevel: "full" }` and no `fullAccessAvailable`; current main returns available, and the existing unit test asserts that behavior.

## Solution Assessment

No. Current main is not the best solution because it treats configuration permission as runtime capability; the maintainable fix is to thread a real capability signal and fail closed when that signal is absent.
