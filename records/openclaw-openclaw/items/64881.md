## Summary

Keep open. Current main still has the reported source-level gap: ambiguous enabled-and-allowed elevated config resolves as full-access available, both prompt construction paths still pass config-only state into the resolver, and the focused resolver test still asserts the optimistic fallback.

## What This Changes

Request that elevated full-access prompt guidance be backed by an explicit provider/runtime capability signal, with fail-closed ambiguous handling and regression coverage.

## Best Possible Solution

Add a generic runtime capability source for elevated full-access availability, thread it through both auto-reply and command-system-prompt construction, make ambiguous resolver input unavailable with an explicit blocked reason, and cover the resolver, system prompt, and provider/runtime matrix with focused tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to call `resolveEmbeddedFullAccessState` with `{ enabled: true, allowed: true, defaultLevel: "full" }` and no `fullAccessAvailable`; current main returns available, and the existing focused test asserts that behavior.

## Solution Assessment

No, the current implementation is not the best solution because it treats configuration permission as runtime capability. The maintainable fix should introduce or reuse a generic provider/runtime capability signal before changing fallback semantics broadly.
