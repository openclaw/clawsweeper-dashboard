## Summary

Keep open: current `main` still derives `fullAccessAvailable` from enabled-and-allowed elevated config instead of a provider/runtime capability signal, and the focused resolver test still asserts that optimistic fallback.

## What This Changes

Require elevated full-access availability to come from a real provider/runtime signal, fail closed on ambiguous input, and add regression coverage.

## Best Possible Solution

Define a generic elevated full-access capability source at the runtime/provider boundary, thread it through auto-reply and command prompt construction, and update resolver, prompt, and provider/runtime tests so ambiguous input fails closed.

## Reproduction Assessment

Yes. A source-level reproduction is to call `resolveEmbeddedFullAccessState` with `{ enabled: true, allowed: true, defaultLevel: "full" }` and no `fullAccessAvailable`; current `main` returns available, and the existing unit test asserts that behavior.

## Solution Assessment

No. Current `main` is not the best solution because it treats configuration permission as runtime capability; the requested fail-closed direction is sound, but maintainers should choose the generic provider/runtime capability seam before changing behavior broadly.
