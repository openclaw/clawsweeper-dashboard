## Summary

Keep open: current main still treats missing `fullAccessAvailable` as available when elevated exec is enabled and allowed, so the false `/elevated full` guidance gap remains valid.

## What This Changes

The issue asks OpenClaw to derive embedded full-access availability from a real provider/runtime signal, fail closed when that signal is absent, and add regression coverage.

## Best Possible Solution

Define a generic provider/runtime full-access capability contract, thread it into auto-reply and command prompt construction, then update resolver and prompt tests so ambiguous input fails closed under agreed compatibility semantics.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: call `resolveEmbeddedFullAccessState` with `{ enabled: true, allowed: true, defaultLevel: "full" }` and no `fullAccessAvailable`; current main returns available, and the focused unit test asserts that behavior.

## Solution Assessment

No. The requested fail-closed direction is sound, but the maintainable fix should start with a provider/runtime capability seam and an explicit compatibility decision for direct host runs.
