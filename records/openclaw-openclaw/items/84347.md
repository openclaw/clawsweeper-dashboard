## Summary

Keep open: this is a focused documentation-only PR for a real iMessage operator footgun. Current main documents SSH `cliPath` wrappers but does not warn that wrappers must stream newline-framed JSON-RPC stdio incrementally, while the iMessage runtime source confirms the long-lived piped stdio contract and timeout symptom.

## What This Changes

The PR adds a Warning block to the iMessage Remote Mac over SSH docs explaining that `cliPath` wrappers must transparently stream JSON-RPC stdin/stdout without buffering.

## Best Possible Solution

Merge the focused warning in the Remote Mac over SSH section after normal docs CI, then let the linked issue at https://github.com/openclaw/openclaw/issues/84330 close through the PR.

## Reproduction Assessment

No live iMessage wrapper repro was run, but the failure mode is source-reproducible: current main writes newline-framed JSON-RPC requests to a long-lived child stdin, reads stdout line-by-line, and times out when buffered wrappers do not forward small frames.

## Solution Assessment

Yes. Documenting the streaming requirement exactly where operators configure the SSH wrapper is the narrowest maintainable fix; no runtime behavior change is needed for the reported docs gap.
