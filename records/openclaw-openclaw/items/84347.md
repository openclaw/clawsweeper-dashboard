## Summary

Keep open: the focused docs fix is useful, but the added safe example contradicts the warning by using socat's `stderr` option, which redirects subprocess stderr into stdout and can corrupt the JSON-RPC stream.

## What This Changes

The PR adds a Warning block to the iMessage Remote Mac over SSH docs documenting streaming stdio requirements for `cliPath` wrappers.

## Best Possible Solution

Keep the warning but correct or remove the socat example so every recommended wrapper preserves stdout as JSON-RPC only, then merge after docs CI.

## Reproduction Assessment

Yes, by source and manual inspection rather than live iMessage: current main uses long-lived newline-framed JSON-RPC over child stdio, and the PR's socat example uses an option documented to redirect stderr into stdout.

## Solution Assessment

No, not quite as written. The warning belongs in this docs section, but the safe example must not recommend a stderr-merging socat invocation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Keep stderr out of the socat example:** `docs/channels/imessage.md:130`
  - body: This warning tells operators to keep stderr separate, but the same sentence labels `socat - EXEC:"ssh host imsg",pty,stderr` as safe. Socat's `stderr` option redirects subprocess stderr into the output channel, so any SSH or imsg diagnostics can enter the JSON-RPC stdout stream and cause parse failures or the same timeout symptoms this warning is trying to prevent. Please drop `,stderr` or replace the example with one that keeps diagnostics off stdout. ([man7.org](https://man7.org/linux/man-pages/man1/socat.1.html))
  - confidence: 0.93
