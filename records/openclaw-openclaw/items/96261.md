## Summary

Keep open: the unbounded success-body read is real, but this patch is not merge-ready because it bypasses the shared provider JSON helper, hard-codes the generic 16 MiB cap on a large image-response surface, and lacks real provider-path proof.

## What This Changes

The branch replaces the OpenAI-compatible image provider's raw success-body `response.json()` call with a direct `readResponseWithLimit` plus `JSON.parse` path capped at 16 MiB.

## Best Possible Solution

Use the shared `readProviderJsonResponse` helper with a maintained image-specific byte budget, or an explicit documented 16 MiB product limit, plus provider-path proof for valid-large and hostile-overflow responses.

## Reproduction Assessment

Yes at source level: current main calls `response.json()` on a provider-controlled image success body in the shared OpenAI-compatible image provider. I did not run a live image endpoint in this read-only review.

## Solution Assessment

No as written: bounding the read is the right direction, but the best fix should reuse `readProviderJsonResponse` and settle the image response byte budget with tests and provider-path proof.
