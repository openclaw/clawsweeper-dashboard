## Summary

Keep open. This is a real bounded-response hardening target, but the branch reimplements the existing provider JSON reader and hard-codes a 16 MiB image response cap without proving supported large image envelopes remain accepted; the related broader PR is not a safe superseding close target while it still has compatibility/proof blockers.

## What This Changes

The PR replaces the OpenAI-compatible image provider's raw JSON response read with a bounded response read before parsing image generation and edit responses.

## Best Possible Solution

Use the shared `readProviderJsonResponse` helper with a maintained image-specific byte budget, or an explicit documented 16 MiB product limit, plus provider-path tests/proof for valid-large and hostile-overflow responses.

## Reproduction Assessment

Yes at source level: current main calls `response.json()` on a provider-controlled image success body in the shared OpenAI-compatible image provider. I did not execute a live image endpoint in this read-only review.

## Solution Assessment

No as written: bounding the read is the right direction, but the best fix should reuse the provider JSON helper and settle the image response byte budget with tests/proof instead of adding a local one-off parser.
