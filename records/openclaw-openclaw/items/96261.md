## Summary

Keep open: the unbounded image-provider read still exists on current main and the latest release, but this PR is not merge-ready because it duplicates the shared provider JSON reader, hard-codes a compatibility-sensitive 16 MiB image JSON cap, and still lacks provider-path proof.

## What This Changes

The PR replaces the OpenAI-compatible image provider success-body `response.json()` parse with a direct `readResponseWithLimit` bounded read and manual `JSON.parse`.

## Best Possible Solution

Use `readProviderJsonResponse` with a maintained image-specific byte budget, or an explicit documented 16 MiB product limit, plus provider-path tests/proof for valid-large and hostile-overflow responses.

## Reproduction Assessment

Yes, at source level: current main and `v2026.6.10` both call `response.json()` on a provider-controlled image success body in the shared OpenAI-compatible image provider. I did not run a live image endpoint in this read-only review.

## Solution Assessment

No as written: bounding the read is the right direction, but the best fix should reuse the provider JSON helper and settle the image response byte budget with tests/proof instead of adding a local parser and constant.
