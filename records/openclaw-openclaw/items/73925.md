## Summary

Keep open: current main still does not add the two-phase connect-auth watchdog, but this PR head should not merge because repeated valid preauth `connect` frames can reset the new watchdog while async auth is pending.

## What This Changes

The branch moves Gateway WebSocket preauth timer clearing to a valid `connect` frame, adds a connect-auth watchdog and cleanup hooks, and updates Gateway docs, changelog, and focused tests.

## Best Possible Solution

Repair the branch with a one-shot connect-auth deadline or explicit connecting state before the first auth await, add duplicate-connect regression coverage, and preserve the original contributor credit when landing.

## Reproduction Assessment

Yes. Source inspection of PR head shows a valid connect whose auth stalls can be followed by another valid preauth connect before timeout; because no client is registered yet, the second path re-arms the watchdog.

## Solution Assessment

No. The two-phase timer direction is the right fix boundary, but this implementation is not mergeable until duplicate preauth connects cannot extend the auth deadline.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Make the connect-auth watchdog one-shot:** `src/gateway/server/ws-connection.ts:408`
  - body: `armConnectAuthTimer()` clears and recreates the auth watchdog while auth is still pending. Because the socket has no registered client until after async auth completes, a second valid preauth connect can re-enter this path and extend the deadline indefinitely; preserve the first deadline or reject duplicate connects before rearming.
  - confidence: 0.93
