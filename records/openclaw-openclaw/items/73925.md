## Summary

Keep open. The PR is a useful repair of the contributor work from #49751, but the current head still has a blocking Gateway security/correctness issue: duplicate valid preauth `connect` frames can rearm the new auth watchdog while auth is still awaiting.

## What This Changes

The PR branch adds a two-phase Gateway WebSocket preauth/connect-auth timeout, updates protocol docs and the changelog, and adds gateway/client tests for stalled connect auth and timeout resolver behavior.

## Best Possible Solution

Repair the existing PR branch with a one-shot connect-auth watchdog or explicit preauth connecting state set before the first auth await, reject or close duplicate preauth connect frames deterministically, preserve cleanup on close and successful registration, and add regression coverage for repeated valid connect frames during stalled auth.

## Reproduction Assessment

Yes. On the PR head, send a valid connect frame whose auth path stalls, then send another valid preauth connect before the watchdog fires; because getClient remains null, the second frame re-enters the handshake path and rearms the watchdog.

## Solution Assessment

No. The two-phase timeout direction is the right narrow design, but this implementation is not the best fix while the auth-phase deadline is resettable; a one-shot watchdog or explicit connecting state is safer.
