## Summary

Keep open. Current main still has the single preauth timer behavior this PR is trying to repair, but PR head `9a16c5b` is not merge-ready because its new connect-auth watchdog can be reset by repeated valid preauth `connect` frames.

## What This Changes

The PR adds a two-phase Gateway WebSocket handshake timeout, protocol/changelog docs, and gateway tests so valid connects clear the preauth timer while async auth remains bounded.

## Best Possible Solution

Repair the existing branch with either a one-shot connect-auth watchdog or an explicit connecting state set before the first auth await, reject or close duplicate preauth connects deterministically, add a repeated-connect regression test, and preserve the original contributor credit from #49751.

## Reproduction Assessment

Yes. Send a valid `connect` frame whose auth resolution stalls, then send another valid preauth `connect` before the watchdog fires; because `getClient()` is still null, the PR re-enters the connect path and rearms the watchdog.

## Solution Assessment

No. The two-phase timer direction is the narrow maintainable approach, but this implementation is not the best fix while the auth-phase deadline can be reset by duplicate preauth connects.
