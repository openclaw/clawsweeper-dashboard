## Summary

Keep open. Current main still has the underlying Gateway WebSocket auth-timer gap, and this PR is a useful repair of the source contributor work, but the proposed connect-auth watchdog is resettable by repeated valid preauth `connect` frames while async auth is still pending.

## What This Changes

The PR branch adds a Gateway WebSocket connect-auth watchdog after valid `connect` parsing, clears the new timer on close/registration, updates Gateway protocol docs and the changelog, and adds gateway timeout/auth tests.

## Best Possible Solution

Repair the existing PR branch with a one-shot connect-auth watchdog or an explicit `connecting` state set before the first auth await, deterministically reject or close duplicate preauth `connect` frames, keep timer cleanup on close and successful registration, and add regression coverage for repeated valid `connect` frames during stalled auth.

## Reproduction Assessment

Yes. Send one valid `connect` frame that stalls inside `resolveConnectAuthState` or `resolveConnectAuthDecision`, then send another valid preauth `connect` before the watchdog fires; because no client/in-flight state exists yet, the second frame can re-enter the PR’s connect path and rearm the watchdog.

## Solution Assessment

No. The two-phase timer design is the right narrow direction, but this implementation is not the best fix while the auth-phase deadline can be reset; the safer path is a one-shot watchdog or explicit connecting state plus duplicate-connect regression tests.
