## Summary

Keep open: the PR targets a real gateway WebSocket auth timing gap that current main still does not implement, but this head is not merge-ready because the new watchdog can be reset by duplicate valid `connect` frames and the live PR is currently dirty.

## What This Changes

The PR adds a second WebSocket connect-auth timeout after a valid gateway `connect` frame, updates gateway protocol docs, and adds focused gateway tests.

## Best Possible Solution

Land an equivalent rebased gateway fix that starts the connect-auth deadline once per syntactically valid connect, rejects or closes duplicate pre-client connects while auth is in progress, clears the watchdog on registration/close, and adds duplicate-connect stalled-auth coverage.

## Reproduction Assessment

Yes, from source inspection. A duplicate valid `connect` frame can arrive while the first `resolveConnectAuthState` await is pending, `getClient()` is still null, and the PR calls `armConnectAuthTimer()` again.

## Solution Assessment

No, not as currently proposed. The two-phase timeout is the right boundary, but the implementation must make connect auth one-shot or mark the handshake as connecting before the first await.
