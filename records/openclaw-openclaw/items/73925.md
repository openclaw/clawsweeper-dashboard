## Summary

Keep open: the PR addresses a real Gateway WebSocket handshake/auth race, but the proposed connect-auth watchdog can be reset by repeated valid preauth connect frames while auth is still pending, so the security property it adds is not reliable yet.

## What This Changes

The PR moves Gateway WebSocket preauth timer clearing to a valid connect frame, adds a connect-auth watchdog, updates Gateway protocol/changelog docs, and adds Gateway/client tests.

## Best Possible Solution

Repair the branch with a one-shot connect-auth deadline or explicit connecting state before the first auth await, reject duplicate preauth connects deterministically, and add regression coverage for repeated valid connect frames while auth is pending.

## Reproduction Assessment

Yes, source-reproducible. Send a valid connect frame whose auth resolution stalls, then send another valid preauth connect before the watchdog fires; PR head re-enters because no client is registered and resets the watchdog at src/gateway/server/ws-connection.ts:408.

## Solution Assessment

No. The two-phase timer direction is the right fix boundary, but this implementation is not the best solution until duplicate preauth connects cannot extend the auth deadline.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Make the auth watchdog one-shot:** `src/gateway/server/ws-connection.ts:408`
  - body: `armConnectAuthTimer()` clears and recreates the connect-auth timer while auth is still pending. Because the socket has no registered client until after async auth completes, a second valid preauth connect can re-enter this path and extend the deadline indefinitely; keep the original auth deadline or reject duplicate connects before rearming.
  - confidence: 0.92
