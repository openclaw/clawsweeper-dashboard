## Summary

Keep open. Current main still accepts a thread-shaped target, resolves it to a canonical session key, and dispatches an internal agent run without any refusal or warning; the existing thread helpers and docs make the requested guard feasible and narrowly scoped.

## What This Changes

The issue requests that `sessions_send` reject or warn when an inter-agent send targets a thread-shaped session key.

## Best Possible Solution

Add a focused guard in `createSessionsSendTool` after canonical target resolution and visibility/access checks, before baseline history reads or `agent` dispatch. The guard should use shared session-thread parsing to reject canonical thread sessions with a clear error directing agents to the parent channel session key, while preserving existing thread-bound announce delivery behavior.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to call `createSessionsSendTool` with a visible target such as `agent:main:slack:channel:C123:thread:1710000000.000100`; current main reaches the mocked gateway `agent` call with that same resolved session key because no guard exists.

## Solution Assessment

Yes. A transport-layer refusal after canonical resolution is the narrowest maintainable fix because it covers raw keys, labels, and sessionId lookups; warning-only would leave the human-visible failure mode in place.
