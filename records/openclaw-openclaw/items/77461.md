## Summary

Keep open: the report matches a plausible packaged runtime dependency-boundary bug, with generated/core runtime code importing `json5` while the plugin dependency collection path is package-manifest driven. I did not find a current-main shipped fix or duplicate tracker in the provided context, and the exact packaged sandbox behavior still needs validation.

## What This Changes

The issue reports that `memory_search` can fail in a generated plugin runtime-deps sandbox because generated runtime chunks import `json5` but the sandbox manifest omits it.

## Best Possible Solution

Make the packaged/runtime dependency manifest account for every package imported by generated runtime chunks, or remove those core-owned imports from isolated plugin-runtime chunks, then add package-level regression coverage for `json5` resolution.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No: the reporter's released-version log and current source make the dependency miss credible, but I did not establish a failing packaged current-main sandbox in this read-only review.

## Solution Assessment

Is this the best way to solve the issue? No: manually installing `json5` in `~/.openclaw/plugin-runtime-deps` is only a workaround; the maintainable fix belongs in generated package/runtime dependency closure or the generated chunk graph.
