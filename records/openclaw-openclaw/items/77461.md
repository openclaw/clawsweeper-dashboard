## Summary

This issue should stay open because the report matches a plausible current dependency-boundary gap: generated runtime code can import `json5`, while the plugin/runtime dependency collection path is manifest-driven and the memory plugin manifest does not declare `json5`. I did not find a current-main shipped fix or a duplicate tracker in the provided context.

## What This Changes

The issue reports that the generated plugin runtime dependency sandbox omits `json5`, causing `memory_search` to fail when generated runtime files import it.

## Best Possible Solution

Make the packaged/plugin runtime dependency manifest account for packages imported by generated runtime chunks, then add regression coverage that a `memory_search` runtime sandbox can resolve `json5`.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source evidence and the reporter's exact released-version log make the dependency miss credible, but I did not run a packaged current-main sandbox reproduction in this read-only review.

## Solution Assessment

Is this the best way to solve the issue? No: manually installing `json5` in `~/.openclaw/plugin-runtime-deps` is only a local workaround; the maintainable fix is to make the generated runtime dependency set include `json5` or remove that runtime import from isolated chunks.
