## Summary

Keep open. Current main still has a credible auto-update failure path for Homebrew/package-managed gateways, and an open closing PR already tracks a proposed resolution; closing before that PR lands would orphan a valid report.

## What This Changes

The reporter requests that Homebrew-installed gateways honor `update.auto.enabled` by auto-installing a newer stable version after the configured rollout delay and restarting.

## Best Possible Solution

Resolve the Homebrew auto-update contract explicitly: either implement a safe Homebrew-aware apply flow outside the live gateway process, or land the skip-and-hint approach from PR #74870 with aligned docs, tests, and changelog coverage.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is a package-managed gateway service with auto-update due: the auto-update child inherits gateway service markers, then `openclaw update` refuses package replacement inside that service process.

## Solution Assessment

Unclear. PR #74870 is a plausible narrow fix if maintainers choose skip-and-hint for Homebrew installs, but it does not implement the originally expected auto-install behavior.
