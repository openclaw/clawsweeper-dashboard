## Summary

Keep open. Current main still routes due package auto-updates through the generic package updater without Homebrew-specific handling, and the linked open PR with closing syntax remains the active fix path rather than a reason to close this issue early.

## What This Changes

The issue requests that Homebrew-installed gateways actually apply detected stable auto-updates after the configured rollout delay.

## Best Possible Solution

Land or replace the linked PR with a Homebrew-aware updater contract that is consistent across runtime logs, docs, tests, and real Homebrew proof, then close the issue after merge.

## Reproduction Assessment

Yes, from source-level inspection on current main. A Homebrew/package-managed gateway with a due update still reaches the generic package auto-update path and can inherit gateway service markers that the update CLI blocks; I did not run a live Homebrew gateway.

## Solution Assessment

Unclear. The linked skip-and-hint direction may be maintainable, but it changes the user-facing Homebrew auto-update contract unless docs and runtime guidance explicitly align with that decision.
