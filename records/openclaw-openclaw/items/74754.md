## Summary

Keep open. Current main still has the credible Homebrew/package-managed auto-update failure path, and open PR #74870 already references this issue with closing syntax, so the issue should remain open until that PR lands or maintainers choose a different update contract.

## What This Changes

Homebrew-installed gateways with auto-update enabled should install a detected stable update after the configured delay and restart, instead of remaining on the old version.

## Best Possible Solution

Resolve the Homebrew auto-update contract explicitly: either implement a safe Homebrew-aware apply flow outside the live gateway process, or land PR #74870's skip-and-hint approach with aligned docs, tests, and changelog coverage.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is a package-managed gateway service with auto-update due: the runner launches `openclaw update`, the child inherits gateway service markers, and the update command refuses package replacement inside that service process.

## Solution Assessment

Unclear. PR #74870 is a plausible narrow fix if maintainers choose manual Homebrew upgrade hints, but it intentionally does not implement the originally expected automatic install behavior.
