## Summary

Keep open. Current main still has the Homebrew/package-managed auto-update failure path: the gateway auto-update runner shells out to the generic update CLI, the child inherits gateway service markers, and the update CLI refuses package replacement from inside the gateway service. PR #74870 is an open, unmerged implementation candidate with closing syntax, so this issue should remain open until that PR lands or maintainers choose a different Homebrew update contract.

## What This Changes

The issue reports that a Homebrew-installed gateway detects a newer stable version after `update.auto.*` is enabled but never auto-installs or restarts, requiring a manual `brew upgrade openclaw`.

## Best Possible Solution

Resolve the Homebrew auto-update contract in one place: either land a safe Homebrew-aware apply flow that runs outside the live gateway service process, or land PR #74870's manual-upgrade hint behavior with aligned docs, tests, and changelog coverage.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is a package-managed gateway service with a due auto-update: the startup runner launches the update CLI, the child keeps gateway service markers, and the update command refuses package replacement inside that service process.

## Solution Assessment

Unclear. PR #74870 is a narrow and maintainable fix if the intended contract is to skip Homebrew auto-install with a clear `brew upgrade openclaw` hint, but it does not satisfy the originally reported expectation that the gateway auto-installs the newer Homebrew version.
