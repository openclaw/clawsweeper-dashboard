## Summary

Keep open. Current main still has the credible package-managed auto-update failure path, and open PR #74870 already references this issue with closing syntax, so the issue should remain open until that PR lands or maintainers choose a different Homebrew update contract.

## What This Changes

The issue requests that Gateway auto-update install a newer Homebrew-managed stable version after the configured rollout delay and restart itself instead of requiring a manual `brew upgrade`.

## Best Possible Solution

Resolve the Homebrew auto-update contract explicitly: either implement a safe Homebrew-aware apply flow outside the live gateway process, or land PR #74870's skip-and-hint approach with aligned docs, tests, and changelog coverage.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is a package-managed gateway service with auto-update due: the runner invokes the update CLI, the child inherits gateway service markers, and the update command refuses package replacement inside that service process.

## Solution Assessment

Unclear. PR #74870 is a plausible narrow fix if maintainers choose manual Homebrew upgrade hints, but it intentionally does not implement the originally expected automatic install behavior.
