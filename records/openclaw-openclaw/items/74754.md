## Summary

Keep open. Current main still routes due gateway auto-updates through the normal package update command, which refuses package replacement from inside a gateway service process, and there is no Homebrew-specific apply or skip path in main. Open PR #74870 references this issue with closing syntax, so cleanup should wait for that PR to merge, be replaced, or be explicitly rejected.

## What This Changes

The issue reports that a Homebrew-installed gateway detects a newer OpenClaw version but does not perform the configured automatic stable update after the delay.

## Best Possible Solution

Resolve the Homebrew auto-update contract explicitly: either implement a safe Homebrew-aware apply flow outside the live gateway service process, or land the skip-and-hint approach from #74870 with aligned docs, tests, and changelog so Homebrew users are told to run `brew upgrade openclaw`.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is a package-managed gateway service with auto-update due: the auto-update child inherits gateway service markers, then `openclaw update` refuses package replacement inside that service process.

## Solution Assessment

Unclear. The current open PR is a plausible narrow mitigation if maintainers want Homebrew installs to be manual, but it does not implement the originally requested automatic install-and-restart behavior.
