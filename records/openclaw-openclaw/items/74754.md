## Summary

Keep open. Current main still has the Homebrew/package-managed auto-update failure path, and open PR #74870 uses closing syntax for this issue but remains unmerged, so the issue should stay open until that implementation is reviewed, merged, or replaced.

## What This Changes

Homebrew-installed gateways should apply a detected stable update after `update.auto.stableDelayHours` instead of staying on the old version indefinitely.

## Best Possible Solution

Define the Homebrew update contract and land the linked PR or a replacement: either skip with one-shot `brew upgrade openclaw` guidance, or implement package-managed auto-apply outside the running gateway service.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is a Homebrew/package-managed gateway service with auto-update due: startup spawns the update CLI from the service environment, and the update command refuses package replacement inside that service process.

## Solution Assessment

Unclear. The linked PR's skip-and-hint approach is narrow and maintainable if maintainers choose manual Homebrew upgrades as the contract; true auto-install would need a safer updater flow outside the live gateway service.
