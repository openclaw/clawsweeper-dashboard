## Summary

Keep open. Current main still has the Homebrew/package-managed auto-update failure path: the gateway auto-updater launches the CLI update path from the service environment, while the CLI refuses package updates from inside the gateway service; open PR #74870 already uses closing syntax for this issue and should be reviewed before any cleanup close.

## What This Changes

Make package auto-update install a newer stable Homebrew-managed OpenClaw after `stableDelayHours` instead of only recording availability.

## Best Possible Solution

Define the Homebrew update contract explicitly: either land the Homebrew-aware skip-and-hint behavior with aligned docs/tests/changelog, or implement a safe updater flow that runs outside the live gateway service before replacing a package-managed install.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is a package-managed gateway service with auto-update due: the runner launches the update CLI from the service-marked environment, and the update command refuses package replacement inside that service process.

## Solution Assessment

Unclear. Current main does not solve the report; PR #74870 is a plausible narrow fix if maintainers choose manual Homebrew upgrade hints, but it intentionally does not implement the originally expected automatic install behavior.
