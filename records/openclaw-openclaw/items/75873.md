## Summary

Keep open. Current main still has the timeout/fallback gap, and the PR is a useful narrow fix with the prior compaction/payload and changelog concerns addressed, but the latest diff makes a new harness result field required on the exported plugin SDK type.

## What This Changes

The PR adds a `timedOutDuringToolExecution` flag, active-tool counting, failover and timeout-branch exemptions, trajectory metadata plumbing, regression coverage, and a changelog entry.

## Best Possible Solution

Land the tool-execution timeout exemption with a backward-compatible harness result contract that defaults absent `timedOutDuringToolExecution` to `false` for existing plugin harnesses.

## Reproduction Assessment

Yes. The linked issue gives concrete long `process(poll)` steps and logs, and current main source still treats non-compaction timeouts as fallback-eligible; the review finding is reproducible by compiling an existing external harness against the new required SDK result field.

## Solution Assessment

No. The flag-and-policy approach is the right narrow direction, but the field should be optional or normalized at the runner boundary so public harness plugins do not have to implement a core-owned signal immediately.
