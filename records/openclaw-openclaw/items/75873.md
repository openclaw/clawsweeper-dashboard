## Summary

Keep open. The PR has the right core direction for #52147, but it only exempts assistant failover; the outer runner still treats the same tool-execution timeout as an LLM-style timeout for compaction recovery and no-response payload synthesis.

## What This Changes

The PR adds a `timedOutDuringToolExecution` attempt flag and threads it through failover, trajectory, and tests so active-tool run timeouts skip assistant profile rotation and model fallback.

## Best Possible Solution

Use the new timeout-cause flag consistently anywhere the runner applies LLM-timeout-only recovery or messaging, then add the required changelog entry and targeted regression coverage for the full run path.

## Reproduction Assessment

Yes. The linked issue gives a concrete path: configure a primary model with fallbacks, run a long background exec/process(poll) loop past the default 600s run timeout, and observe the current `timedOut && !timedOutDuringCompaction` failover classification.

## Solution Assessment

No. The flag-and-policy approach is maintainable, but the current patch stops at fallback policy and leaves other LLM-timeout-only branches active for the same tool-execution timeout.
