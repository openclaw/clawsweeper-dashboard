## Summary

Keep open. The PR targets a valid #52147 bug and the failover-policy change is directionally right, but it only exempts the assistant failover decision; earlier LLM-timeout recovery branches still ignore the new flag, and the required changelog entry is missing.

## What This Changes

The PR adds `timedOutDuringToolExecution` plumbing from active tool tracking into embedded-run attempt results, trajectory metadata, and assistant failover decisions to avoid model fallback after tool-execution run timeouts.

## Best Possible Solution

Finish the branch by applying the new timeout cause consistently across all LLM-timeout-only paths, keep LLM-phase timeout behavior unchanged, add focused regression coverage, and include a user-facing changelog entry before landing it to close #52147.

## Reproduction Assessment

Yes. #52147 gives concrete steps and logs for a long `exec`/`process(poll)` run exceeding the agent timeout, and current main source shows that such a timeout flows into the assistant fallback condition without any tool-execution cause.

## Solution Assessment

No. The flag-and-policy direction is maintainable, but the proposed patch is incomplete because timeout-triggered compaction and generic timeout payload synthesis still key only on `timedOut && !timedOutDuringCompaction`.
