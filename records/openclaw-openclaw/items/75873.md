## Summary

Keep open. The linked bug is source-backed on current main, and this draft PR is directionally useful, but the patch is incomplete: it exempts tool-execution timeouts from assistant failover while leaving the later empty-payload timeout synthesis and public harness result contract inconsistent with the new flag.

## What This Changes

The PR adds a `timedOutDuringToolExecution` attempt-result flag, detects active tool executions from the embedded tool-start map, threads the flag through assistant failover and trajectory artifacts, and adds failover-policy regression tests.

## Best Possible Solution

Keep the PR open and finish the same approach: classify active-tool run timeouts, skip assistant profile/model fallback for that cause, avoid the misleading no-payload timeout error for the same cause, preserve the public harness SDK contract, and cover the full terminal path with focused tests plus a changelog entry.

## Reproduction Assessment

Yes. The linked issue gives a concrete `exec` plus repeated `process(poll)` scenario, and current main source confirms non-compaction timeouts still enter assistant timeout failover without checking whether a tool was active.

## Solution Assessment

No. The proposed flag-and-policy direction is the right narrow fix, but the patch stops at failover and leaves the user-visible timeout synthesis and public harness result typing inconsistent with the new classification.

Make the field optional at the plugin boundary or normalize missing values to `false` without requiring third-party implementations to change in lockstep.
  - confidence: 0.79
