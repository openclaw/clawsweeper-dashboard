## Summary

Keep open for normal PR review: current main still misclassifies non-compaction run timeouts, and the latest patch matches the existing compaction-timeout boundary with no blocking findings.

## What This Changes

Adds a `timedOutDuringToolExecution` attempt flag, derives it from active tool lifecycle state, exempts those timeouts from compaction/payload/failover paths, and records the flag in trajectory metadata with tests and changelog coverage.

## Best Possible Solution

Merge this PR after the targeted runner/failover/harness/trajectory tests and changed gate pass, preserving the focused active-tool timeout exemption without broad timeout-policy refactoring.

## Reproduction Assessment

Yes. Use a short `agents.defaults.timeoutSeconds`, a long `process(poll)`, browser, or `exec` tool call, and a configured fallback chain; current main can also be source-verified because non-compaction `timedOut` still feeds assistant rotation/fallback.

## Solution Assessment

Yes. The active-tool flag mirrors the established compaction exemption, keeps LLM-phase timeouts on the existing fallback path, and makes the new attempt-result field optional for public harness compatibility.
