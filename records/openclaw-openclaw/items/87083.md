## Summary

Keep this PR open, but do not merge it as-is. The fix targets a real crash-loop/session-growth problem, but the new cap counts bootstrap markers that are already invalidated by a later compaction, which can prevent a fresh post-compaction marker from ever being written and leave normal sessions repeatedly injecting full bootstrap context.

## What This Changes

Adds a hardcoded recent-tail cap for persisted full-bootstrap completion markers, wires it into the embedded attempt append path, and adds helper-level Vitest coverage.

## Best Possible Solution

Land a rebased cap that preserves the existing compaction reset contract, adds a regression for historical markers before compaction, and includes redacted real gateway or preloaded degraded-session proof before merge.

## Reproduction Assessment

Yes at source level: current main has an uncapped full-bootstrap marker append path, and the linked field reports describe repeated marker growth during crash restarts. I did not run the live gateway OOM loop in this read-only review.

## Solution Assessment

No, not yet: a recent-entry circuit breaker is a plausible narrow mitigation, but it must preserve the existing compaction boundary before it is the best fix path.
