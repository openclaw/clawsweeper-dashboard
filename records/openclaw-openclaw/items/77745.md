## Summary

Current main already removes both reported EPERM paths: Feishu prefers the SDK readable stream path before writeFile, and saved media writes go through fs-safe rather than the old read-only handle.sync implementation.

## What This Changes

Feishu media downloads on Windows should save inbound image/media files without failing on EPERM from SDK writeFile/fsync or writeSavedMediaBuffer handle.sync.

## Best Possible Solution

Keep the current fs-safe media-store path and Feishu readable-stream-first save path; users on affected releases should pick up the next release containing the current main fix rather than patching dist files manually.

## Reproduction Assessment

No high-confidence current-main reproduction remains: the old compiled read-only fsync path described by the report is not present in current source, and Feishu now saves readable streams directly. The old-release failure is well supported by the issue logs and follow-up analysis.

## Solution Assessment

Yes. Moving media writes through fs-safe and using Feishu's readable stream path before SDK writeFile is narrower and more maintainable than patching generated dist files or node_modules.
