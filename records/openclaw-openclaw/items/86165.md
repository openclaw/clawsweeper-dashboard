## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P1] Implement or defer the inbound webhook contract:** `docs/channels/channel-broker.md:17-18`
  - body: The new docs make signed inbound HTTP webhooks part of V1, and the config exposes `signingSecret`, but the plugin only wires outbound sending plus receive metadata. A provider following this contract has no OpenClaw endpoint that verifies signatures and injects `BrokerInboundEventV1` into the receive pipeline, so either implement and test that path now or make V1 explicitly outbound-only.
  - confidence: 0.86
- **[P2] Validate broker receipts before committing sends:** `extensions/channel-broker/src/runtime.ts:46-50`
  - body: `parseBrokerReceipt` accepts any object from the provider HTTP response as a receipt, and the send path commits message ids after only checking `status === "sent"`. Validate the receipt schema and match `version`, `requestId`, `providerId`, `platform`, status, and message ids against the outbound request before returning it to durable send commit.
  - confidence: 0.84
- **[P2] Preserve typed targets when resolving sessions:** `extensions/channel-broker/src/channel.ts:44-48`
  - body: Direct/group type is encoded into canonical targets as part of `conversationId`, but `resolveBrokerSessionConversation` checks `parsed.conversationType`, which the base parser never sets. Canonical `discord:direct%3A...` or `slack:direct%3A...` targets can therefore be treated as channel conversations during session conversation resolution, so parse the embedded type or carry it through the target helper and cover DM routes.
  - confidence: 0.78
