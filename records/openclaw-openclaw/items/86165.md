Overall confidence: 0.88

Full review comments:

- **[P1] Implement the documented inbound webhook path:** `docs/channels/channel-broker.md:17-18`
  - body: The new docs say V1 includes signed inbound HTTP webhooks, and the PR adds `signingSecret` SecretRef surfaces, but the diff only implements outbound `/v1/outbound`; there is no channel-broker HTTP route that verifies the signature and accepts `BrokerInboundEventV1`. Providers following this contract cannot deliver inbound messages, so either implement and test the webhook path or mark it future-only before merging.
  - confidence: 0.9
- **[P1] Preserve typed broker targets:** `src/plugin-sdk/channel-broker.ts:249-255`
  - body: `BrokerConversationTarget` exposes `conversationType`, and the channel plugin depends on that type for direct/group/thread routing, but `buildBrokerConversationTarget()` serializes only the platform and conversation id. An SDK caller that builds a direct target round-trips through `parseBrokerConversationTarget()` with the type lost, so it defaults to channel routing; encode the type consistently or remove it from this helper contract and document the explicit typed-id form.
  - confidence: 0.86
- **[P2] Validate broker receipts before committing them:** `extensions/channel-broker/src/runtime.ts:46-50`
  - body: `parseBrokerReceipt()` accepts any object as `BrokerReceiptV1`, so a malformed or mismatched provider response can crash later on `messageIds[0]` or commit receipt state for the wrong request/provider/platform. Validate the HTTP response against the V1 receipt shape and match it to the outbound request before returning it from the transport.
  - confidence: 0.88
