## Summary

Keep this issue open: current main and the latest release still emit warn-level loop-detection logs for repeat detections, and the open linked PR is still only a candidate fix rather than merged behavior.

## What This Changes

The issue requests downgrading bounded isolated-cron loop-detection warnings from warn to info so legitimate retry-heavy cron runs stop filling the gateway error log.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/90257 or a narrow replacement that centralizes isolated-cron loop-warning severity while preserving critical and non-cron warning behavior.

## Reproduction Assessment

Yes, at source level: current main still routes generic loop warnings through warn-level detector logging and warning diagnostic events, while isolated cron runs use fresh run-scoped session identity. I did not run the one-hour live cron log-count scenario in this read-only review.

## Solution Assessment

Yes, the requested severity downgrade is the right class of fix, but the best implementation should make the detector/log emission boundary context-aware instead of relying on local log filtering. The linked PR is the current candidate but remains unmerged.
