## Summary

Keep open. Current main still emits generic repeated-tool loop events as warning-level logs/diagnostics, and isolated cron runs still do not pass session target, schedule interval, or bounded lifetime context into the loop-warning decision; the related generic no-progress fix did not cover this log-level demotion.

## What This Changes

Request to demote non-critical loop-detection warnings to info for isolated cron sessions whose run lifetime is bounded by the cron interval.

## Best Possible Solution

Thread typed isolated-cron schedule/run metadata into the tool-loop path, demote only non-critical bounded isolated-cron diagnostics to info, and preserve all critical blocking behavior unchanged.

## Reproduction Assessment

Yes, source-backed. An isolated cron run that makes repeated `exec` calls in one run reaches the generic-repeat warning path, while the current hook has only run-id scoping and logs the result as warning-level.

## Solution Assessment

Yes, if implemented with typed cron metadata and a conservative schedule bound. The hardcoded five-minute proxy in the issue body is not the best direction because it could demote real warnings without proving the run is bounded by its schedule.
