## Summary

Keep open. Current main and the latest shipped release still emit generic repeat loop diagnostics at warning level, and the cron execution path does not yet carry the bounded isolated-cron metadata needed for a conservative info-level demotion.

## What This Changes

The issue asks to reduce false-positive loop-detection warning noise for bounded isolated cron sessions by demoting those events from warn to info.

## Best Possible Solution

Thread typed isolated-cron context and schedule/run boundedness into tool-loop diagnostics, demote only bounded isolated-cron warning events to info, and keep critical or non-isolated behavior unchanged.

## Reproduction Assessment

Yes, source-backed: repeated same-tool/same-argument calls still reach generic_repeat warning detection, then the hook logs warning-level diagnostics. I did not run a live cron job in this read-only review.

## Solution Assessment

Yes for a typed metadata-based demotion; no for the proposed hardcoded five-minute proxy. The maintainable fix should prove the run is isolated and bounded by actual schedule/run timing before lowering severity.
