## Summary

Keep open. Current main still emits generic repeated tool-call detections as warning-level logs and diagnostics, and the isolated cron path still does not pass session target, schedule interval, or elapsed lifetime context into the loop-warning decision.

## What This Changes

The issue asks OpenClaw to demote loop-detection warnings to info for bounded isolated cron sessions whose lifetime does not exceed the cron interval.

## Best Possible Solution

Pass typed bounded isolated-cron execution context from cron scheduling into the tool-loop warning path and demote only non-critical loop diagnostics for runs proven to be isolated and within their schedule bound.

## Reproduction Assessment

Yes. The issue gives a concrete isolated-cron repeated-exec reproduction, and current source confirms the repeated-exec path still emits warning-level logs without bounded cron metadata.

## Solution Assessment

Yes for the requested direction, but the maintainable fix should use job schedule metadata and run timing rather than a hardcoded interval proxy.

Fix issue https://github.com/openclaw/openclaw/issues/69408. Current main emits warning-level tool-loop logs/diagnostics for bounded isolated cron runs. Add typed cron execution context from the cron run path into the before-tool-call/loop-detection warning decision so non-critical loop results are demoted to info only when the run is sessionTarget="isolated" and the elapsed run lifetime is within the job's schedule interval/bound. Do not demote critical loop blocks, non-cron runs, persistent/current/session-target cron runs, or unbounded/unknown schedule cases.
