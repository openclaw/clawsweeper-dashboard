## Summary

Keep open. Current main still emits non-critical tool-loop detections through warning-level detector, hook, and diagnostic paths, while isolated cron execution does not pass typed schedule boundedness or run lifetime metadata to the warning decision point.

## What This Changes

Request to demote loop-detection warning logs to info for bounded isolated cron sessions while preserving audit visibility and warning behavior for unbounded or interactive sessions.

## Best Possible Solution

Add explicit bounded isolated-cron execution context from cron scheduling into the tool-loop warning path, compute the applicable schedule bound from job metadata, and emit an info-level diagnostic/log only for non-critical loop warnings whose isolated run remains within that bound.

## Reproduction Assessment

Yes. The issue gives a concrete isolated-cron repeated-exec reproduction, and current source confirms the same path must still emit warning-level logs and diagnostics because the hook lacks bounded cron metadata.

## Solution Assessment

Yes for the requested direction, but the maintainable implementation should avoid a hardcoded 5-minute proxy. The safer fix is to pass typed cron schedule and run-lifetime context through the existing trigger/jobId path and demote only the intended warning-level false positives.
