## Summary

Keep open. Current main still treats repeated generic tool calls as warning-level loop diagnostics, and the isolated cron path still does not pass session target, schedule interval, or bounded run-lifetime metadata into that decision; no merged fix or canonical duplicate was found in the available related context/search.

## What This Changes

The issue requests demoting non-critical loop-detection warning logs to info for isolated cron sessions whose run lifetime is bounded by the cron interval.

## Best Possible Solution

Thread typed isolated-cron run metadata into the tool-loop warning path, demote only bounded non-critical isolated-cron diagnostics to info, and leave critical blocking semantics unchanged.

## Reproduction Assessment

Yes, source-backed. A cron-triggered isolated embedded run that repeats `exec` reaches the generic-repeat warning path, while the current hook only scopes by run id and always emits non-critical loop events as warning-level diagnostics.

## Solution Assessment

Yes, if the fix uses typed cron schedule/session-target metadata and a conservative lifetime bound. The hardcoded five-minute proxy from the issue body is not the best path because it could demote warnings without proving the run is bounded by its schedule.
