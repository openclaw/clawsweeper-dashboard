## Label Justifications

- `P1`: The routing regression can silently send live traffic through a provider the operator removed and can affect multiple active sessions.
- `impact:session-state`: Persisted per-session provider/model fields survive restarts and override the current configured route.
- `impact:auth-provider`: The stale session state selects a provider absent from the current provider configuration.

## Summary

Current main still returns persisted provider/model overrides before validating them against current provider visibility. A clean open PR specifically closes this issue, so the issue remains necessary until that fix is merged or rejected.

## What This Changes

Invalidate persisted session provider/model routes when their provider has been removed, fall back safely to the configured default, and improve recovery diagnostics.

## Best Possible Solution

Land the focused resolver/chat guard after maintainer acceptance of fallback-to-default semantics, then track persistent cleanup, doctor warnings, and reset commands as narrower follow-up work.

## Reproduction Assessment

Yes at source level: current main returns the stored override or runtime provider/model before any current-config or catalog visibility check, matching the reported stale-route path. This read-only review did not run paid-provider traffic.

## Solution Assessment

Yes for the central bug: validating the persisted route in the shared resolver and supplying catalog visibility from chat routing is narrower and safer than deleting the entire session store. Doctor diagnostics and explicit reset commands are separable enhancements.
