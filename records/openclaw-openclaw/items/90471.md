## Label Justifications

- `P1`: A persisted session route can continue steering real messages to an unintended provider across WebChat and Telegram sessions, which is an urgent broken workflow with billing impact.
- `impact:session-state`: The issue centers on stale persisted session model/provider override state surviving restarts and normal reset attempts.
- `impact:auth-provider`: The affected state changes effective provider/model routing after provider config is removed.

## Summary

Keep open: current main still resolves persisted session model/provider overrides ahead of the configured default without validating that the provider remains configured, and the linked closing PR at https://github.com/openclaw/openclaw/pull/90472 is open but unrelated to this session-routing bug. The related report at https://github.com/openclaw/openclaw/issues/90462 covers similar fallback/session pinning symptoms, but this issue adds the deleted-provider and multi-session route-state case.

## What This Changes

The issue asks OpenClaw to invalidate, flag, or make recoverable persisted session model/provider overrides when the referenced provider has been removed from config.

## Best Possible Solution

Define the invalid session-route policy, then update gateway resolution and doctor/reset coverage so absent providers cannot silently remain the effective route after config changes.

## Reproduction Assessment

Yes, source-level: current main resolves stored providerOverride/modelOverride and modelProvider/model before the configured default, and chat send consumes that resolved provider. I did not run a live Windows gateway repro in this read-only review.

## Solution Assessment

Unclear: the reported fall-through-to-default behavior conflicts with current exact user-selection semantics, so maintainers should first choose whether invalid deleted-provider overrides should fail visibly, clear to default, or be doctor-repaired.
