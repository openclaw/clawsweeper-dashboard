## Summary

Current main still accepts persisted session provider/model fields before validating them against current configuration or catalog visibility, so the reported routing regression remains necessary to fix. The clean open closing PR at https://github.com/openclaw/openclaw/pull/90500 already owns the focused implementation and must be reviewed, merged, or rejected before this issue closes. Likely related people: @vincentkoc for recent chat/catalog work, @steipete for session utility work, and @TurboTheTurtle for the active fix, with confidence noted below.

## What This Changes

Invalidate persisted session provider/model routes when their provider is no longer present in current configuration or model catalog visibility.

## Best Possible Solution

Land a catalog-aware resolver guard that falls back from removed-provider session state while preserving valid configured and catalog-backed routes; track persistent cleanup, doctor diagnostics, and reset commands as separate follow-up scope.

## Reproduction Assessment

Yes at source level: persist provider/model override or runtime fields, remove that provider from configuration, and current chat.send still resolves and uses those fields without visibility validation. This review did not repeat the reporter's paid-provider traffic test because the routing defect is directly established by current source.

## Solution Assessment

Yes for the central bug: resolver-time validation is the narrowest maintainable fix and avoids destructively deleting unrelated session data. Doctor warnings and user-facing reset commands are useful but are not required to correct the live routing path.
