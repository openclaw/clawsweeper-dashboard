## Summary

Keep open. Current main and v2026.5.22 still lack a Google Chat receive-all space-message mode, while threaded replies are only partially handled on main and still overlap active open PR work.

## What This Changes

The issue requests Google Chat threaded outbound replies and an optional way to receive non-mention messages from spaces.

## Best Possible Solution

Land or replace the focused thread-reply work, then design a Google Chat plugin-owned opt-in receive-all mode using user OAuth or Workspace Events while preserving safe service-account defaults, credential-aware docs, tests, and live Workspace proof.

## Reproduction Assessment

Yes, at source level. Current main and v2026.5.22 accept only service-account Google Chat credentials, hardcode chat.bot auth, document @mention-gated spaces, and lack userOAuth or receive-all event subscription handling; a May 22 user comment also says the behavior still happens.

## Solution Assessment

Unclear as one combined change. The maintainable path is to split thread placement/session work from the receive-all auth/event-delivery feature and make the latter opt-in with explicit security and default-behavior decisions.
