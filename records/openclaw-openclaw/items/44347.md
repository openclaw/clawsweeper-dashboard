## Summary

Keep open. Current main has useful Google Chat thread-send plumbing, and active PRs cover parts of thread placement/session behavior, but the opt-in receive-all space-message request still lacks auth, config, docs, and live Workspace proof.

## What This Changes

Requests Google Chat replies stay in incoming threads and an opt-in mode to receive non-mention messages in spaces.

## Best Possible Solution

Land or replace the focused thread-routing work, then design a Google Chat plugin-owned opt-in receive-all mode using user OAuth or Workspace Events with safe defaults, secret-aware credentials, docs, and live Workspace proof coordinated with https://github.com/openclaw/openclaw/issues/9764.

## Reproduction Assessment

Yes, at source level. Current main hardcodes service-account chat.bot auth, has no userOAuth/receive-all/Workspace Events config surface, and defaults Google Chat group ingress to mention gating; no live Workspace repro was run in this read-only review.

## Solution Assessment

Unclear as one combined change. The maintainable path is to handle thread placement in focused PRs and treat receive-all delivery as a deliberate auth/event-delivery feature with explicit opt-in defaults.
