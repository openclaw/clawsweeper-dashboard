## Summary

Keep open. Current main covers threaded replies for delivered Google Chat messages, but the non-mention/all-space-message delivery request is still missing from auth, config/schema, webhook setup, and docs; #9764 is related OAuth groundwork but does not fully cover this delivery behavior.

## What This Changes

Request Google Chat replies in the triggering thread and an optional mode to receive all space messages without @mentions.

## Best Possible Solution

Keep the remaining work in the Google Chat plugin: design an opt-in user-OAuth or Workspace Events delivery mode, preserve service-account fail-closed defaults, and align secrets, config, setup, docs, and tests in coordination with #9764.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: current main can be inspected to show threaded replies are wired while auth/config/webhook/docs expose no user-OAuth or all-message receive mode; no live Google Chat run was possible because no relevant live-test env/profile entries were present.

## Solution Assessment

Is this the best way to solve the issue? Not yet; closing or adding a core workaround would be the wrong boundary. The maintainable path is a plugin-owned opt-in auth and delivery design with explicit security review.
