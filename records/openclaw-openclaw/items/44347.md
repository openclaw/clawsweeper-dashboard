## Summary

Keep open. Current main has solved the threaded-reply half for delivered Google Chat messages, but the all-message space delivery half is still absent from the plugin auth, config, setup, docs, and webhook surfaces.

## What This Changes

The issue requests Google Chat replies to stay in inbound threads and optionally receive non-mention messages from spaces.

## Best Possible Solution

Keep the issue open or split it after the threaded-reply half. The remaining implementation should stay in the Google Chat plugin: add an optional user-OAuth or Workspace Events delivery mode for non-mention space messages, keep service-account webhook behavior fail-closed, and update secret-aware config/schema/setup/docs/tests in coordination with #9764.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the source-level gap: current main can be inspected to show threaded replies are implemented while no user-OAuth/chat.messages/Workspace Events/all-message receive mode exists. Live end-to-end non-mention delivery was not run because no Google Chat live-test credentials were present in env or ~/.profile.

## Solution Assessment

Is this the best way to solve the issue? No current complete solution exists. Threaded replies are handled well on main, but the non-mention delivery half needs a deliberate plugin-owned auth/delivery design rather than a core workaround or a service-account behavior change.
