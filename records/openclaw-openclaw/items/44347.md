## Summary

Keep open. Current main has partial Google Chat thread-send plumbing, but the issue still has unsolved receive-all space-message delivery that requires new Google Chat auth/event-delivery design, and the focused thread-routing work is still covered by open PRs rather than shipped on main.

## What This Changes

The issue requests Google Chat threaded bot replies plus an opt-in way for the channel to receive all space messages without requiring an @mention.

## Best Possible Solution

Land or replace the focused thread-routing work, then design a Google Chat plugin-owned opt-in receive-all mode using user OAuth or Workspace Events with safe defaults, secret-aware credentials, docs, and live Workspace proof coordinated with https://github.com/openclaw/openclaw/issues/9764.

## Reproduction Assessment

Yes, at source level. Current main still hardcodes service-account `chat.bot` auth, has no userOAuth/receive-all/Workspace Events config surface, and keeps Google Chat group ingress on mention gating by default; no live Workspace repro was run in this read-only review.

## Solution Assessment

Unclear as one combined change. The maintainable path is to handle thread placement in focused PRs and treat receive-all delivery as a deliberate auth/event-delivery feature with explicit opt-in defaults.
