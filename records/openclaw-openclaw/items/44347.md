## Summary

Keep open. Current main has useful Google Chat thread-reply plumbing, but the issue still has live remaining work: the receive-all space-message mode is absent from config/auth/docs, and the focused thread/session follow-up is still an open PR.

## What This Changes

The issue asks Google Chat to keep bot replies in the inbound thread and optionally receive non-mention messages in spaces.

## Best Possible Solution

Land or replace the focused thread-routing work, then design a Google Chat plugin-owned opt-in receive-all mode using user OAuth or Workspace Events with safe defaults, credential handling, docs, and live Workspace proof; coordinate the auth surface with https://github.com/openclaw/openclaw/issues/9764.

## Reproduction Assessment

Yes, at source level. Current main has service-account-only chat.bot auth, strict config has no userOAuth/all-message/Workspace Events surface, and Google Chat group processing still defaults to mention gating.

## Solution Assessment

Unclear as one combined change. The maintainable path is to finish thread placement in a focused PR and treat receive-all as a deliberate Google Chat auth/event-delivery feature with safe opt-in defaults.
