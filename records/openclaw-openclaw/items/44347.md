## Summary

Keep this open: current main and v2026.6.6 cover the outbound thread-reply primitive, but the separate request to receive non-mention Google Chat space messages still needs a delegated-auth or Workspace Events design with security/product approval.

## What This Changes

The issue requests Google Chat replies to stay in the triggering thread and asks for an optional way to receive all messages in Google Chat spaces, not only @mentions.

## Best Possible Solution

Keep this issue open as the non-mention Google Chat space-message tracker, preserve the shipped thread-reply primitive, and have maintainers choose a plugin-owned OAuth or Workspace Events subscription design before implementation.

## Reproduction Assessment

Yes at source level, but not live: current main has no user-OAuth or Workspace Events credential/subscription path and still uses service-account chat.bot auth only. A real Google Chat space/live credential check would still be needed before landing a fix.

## Solution Assessment

No current implementation fully solves it. The maintainable solution is not another mention-gate toggle; it is a plugin-owned delegated-auth or Workspace Events design with explicit scope, identity, fallback, and docs decisions.
