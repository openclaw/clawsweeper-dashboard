## Summary

Keep open: current main has a source-reproducible TUI abort gap, and this PR’s TUI-local pending runId fix looks correct with no blocking review findings. The protected `maintainer` label means this should stay open for normal maintainer handling rather than cleanup closure.

## What This Changes

The PR adds TUI `pendingChatRunId` state, uses it as an abort fallback before the first chat event binds `activeChatRunId`, clears it across send/session/event paths, and adds regression tests plus a changelog entry.

## Best Possible Solution

Land this TUI-local state fix after maintainer review and CI, preserving the existing strict `chat.abort { sessionKey, runId }` protocol and regression coverage.

## Reproduction Assessment

Yes, source-reproducible. On current main, press Esc after `sendChat` resolves but before the first matching chat event sets `activeChatRunId`; `abortActive` then takes the `no active run` branch even though the gateway has registered the client runId.

## Solution Assessment

Yes. Tracking the sent runId in TUI state is the narrowest maintainable fix for the waiting gap and avoids broadening the gateway abort protocol.
