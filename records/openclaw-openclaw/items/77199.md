## What This Changes

The PR adds TUI `pendingChatRunId` state so Esc can abort a sent chat run before the first chat event binds `activeChatRunId`, with regression tests and a changelog entry.

## Best Possible Solution

Land a mergeable version of this TUI-local state fix after maintainer review and CI, preserving the existing gateway abort protocol and regression coverage.

## Reproduction Assessment

Yes. On current main, send a TUI message with a delayed first chat event and press Esc after `sendChat` resolves but before `handleChatEvent` sets `activeChatRunId`; `abortActive` then follows the `no active run` branch.

## Solution Assessment

Yes. Tracking the registered client runId in TUI state is a narrow client-side fix that preserves the existing strict `chat.abort { sessionKey, runId }` contract instead of adding a broader protocol fallback.
