## Summary

Keep open. Current main still resolves a trusted-proxy user at the auth boundary but does not carry that server-owned identity into WebChat connection state or the agent message context.

## What This Changes

Request is to surface `X-Forwarded-User` trusted-proxy identity from WebChat auth into agent message context.

## Best Possible Solution

Persist the accepted trusted-proxy subject from `authResult.user` on server-owned WebSocket state, expose it through a documented WebChat identity contract, and add focused negative tests for spoofed metadata and non-trusted auth paths.

## Reproduction Assessment

Yes. The report gives concrete proxy/config steps, and source inspection on current main reproduces the gap without needing a live proxy: auth returns a user, WebSocket state drops it, and `chat.send` reads sender fields from connect payload metadata.

## Solution Assessment

Unclear until maintainers choose the public identity contract. Server-side propagation from `authResult.user` is the maintainable boundary, but the field semantics need explicit product/security approval.

  - confidence: 0.88

  - body: The WebSocket registration currently omits `authResult.user`; any new identity field should be populated only from successful trusted-proxy auth and kept separate from token/password/device paths unless maintainers explicitly choose otherwise.
