## Summary

Keep open. Current main still authenticates and scopes trusted-proxy WebChat users, but the accepted proxy subject is not persisted on WebSocket client state or attached to WebChat agent message context; exposing that server-owned identity needs an explicit gateway/WebChat identity contract.

## What This Changes

The issue requests carrying the trusted-proxy `X-Forwarded-User` subject into WebChat agent message context so agents can identify the authenticated human.

## Best Possible Solution

Persist the accepted trusted-proxy subject from successful server-side auth on WebSocket connection state, expose it through a documented WebChat identity contract, and add spoofing plus non-trusted-auth regression coverage.

## Reproduction Assessment

Yes, at source level. Current main returns a trusted-proxy `user`, drops it during WebSocket client registration, and builds WebChat `MsgContext` without any server-owned proxy identity available.

## Solution Assessment

Unclear until maintainers choose the public identity contract. Server-side propagation from `authResult.user` is the right boundary, but the target field semantics and always-on versus opt-in behavior need product/security approval.
