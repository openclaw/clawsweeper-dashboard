## Summary

Keep open. Current main still extracts and authorizes the trusted-proxy user, but does not persist that subject on WebSocket client state or attach it to WebChat `MsgContext`; the remaining work is valid and security-sensitive because it changes which human identity agents can see.

## What This Changes

Expose trusted-proxy WebChat identity from `X-Forwarded-User` to the agent-visible message context.

## Best Possible Solution

Persist the accepted trusted-proxy subject on server-owned WebSocket connection state, expose it through a documented WebChat identity contract, and add regression coverage for spoofing and non-trusted auth paths.

## Reproduction Assessment

Yes, at source level. Current main returns a trusted-proxy `user`, drops it during WebSocket registration, and builds WebChat `MsgContext` without any server-owned proxy identity available.

## Solution Assessment

Unclear whether the exact requested `SenderId`/`SenderName` mapping is the best contract. Server-side propagation from `authResult.user` is the right boundary, but maintainers need to choose field semantics and always-on versus opt-in behavior.
