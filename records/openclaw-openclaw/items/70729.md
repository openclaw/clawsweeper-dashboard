- `impact:security`: The request changes which proxy-authenticated human identity becomes visible to agents and therefore touches the gateway trust boundary.

- `impact:auth-provider`: The affected surface is trusted-proxy gateway auth identity propagation and its interaction with Control UI/WebChat connections.

## Summary

Keep open. Current main still authenticates the trusted-proxy user but does not preserve that server-owned subject on WebSocket client state or attach it to WebChat message context, and the remaining fix needs maintainer product/security agreement on agent-visible identity semantics.

## What This Changes

The issue asks WebChat to surface the trusted-proxy authenticated user from X-Forwarded-User into agent-visible message context.

## Best Possible Solution

Persist the accepted trusted-proxy subject on server-owned WebSocket connection state, expose it through an explicit documented WebChat identity contract, and cover trusted, spoofed, and non-trusted auth paths with regression tests.

## Reproduction Assessment

Yes, at source level. Current main returns a trusted-proxy user from auth, then drops it before WebSocket client registration and builds WebChat MsgContext without any server-owned proxy identity available.

## Solution Assessment

Unclear whether mapping the proxy subject directly to SenderId/SenderName is the best contract. Server-side propagation from authResult.user is the right boundary, but maintainers should choose field semantics and always-on versus opt-in behavior before implementation.
