## Summary

Keep open. Current main still authenticates the trusted-proxy user, but does not carry that server-owned identity into WebChat connection state or the agent message context; the remaining decision is the security/product contract for exposing it.

## What This Changes

Expose the trusted-proxy user from `X-Forwarded-User` to WebChat-originated agent message context.

## Best Possible Solution

Persist the accepted trusted-proxy subject from `authResult.user` on server-owned WebSocket state, expose it through a documented WebChat identity contract, and add spoofing plus non-trusted-auth regression tests.

## Reproduction Assessment

Yes, at source level. Current main returns a trusted-proxy `user` from auth, drops it when creating `GatewayWsClient`, and `chat.send` has no server-owned proxy identity available when building `MsgContext`.

## Solution Assessment

Unclear until maintainers choose the public identity contract. Server-side propagation from `authResult.user` is the maintainable boundary, but the field semantics and any always-on versus opt-in behavior need security/product approval.
