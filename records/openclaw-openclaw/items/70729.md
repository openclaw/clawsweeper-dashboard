## Summary

Keep open. Current main still has the reported gap: trusted-proxy auth resolves a server-authenticated user, but WebSocket client state does not retain that identity and WebChat `chat.send` still builds sender fields from client-declared Control UI metadata. The issue is actionable, but the sender/authenticated-user contract crosses a trusted-proxy security boundary and should be chosen by maintainers before repair automation.

## What This Changes

Surface the trusted-proxy `X-Forwarded-User` identity from WebChat gateway auth into the agent message context, with optional session metadata attribution.

## Best Possible Solution

Persist the accepted trusted-proxy subject from `authResult.user` on server-owned WebSocket connection state, propagate it into WebChat message context through a maintainer-approved identity contract, and document the user-visible behavior. Add negative tests covering spoofed client metadata and non-trusted auth paths.

## Reproduction Assessment

Yes. The report gives concrete proxy/config steps, and current-main source inspection reproduces the gap: trusted-proxy auth returns a user, the WebSocket client does not store it, and `chat.send` uses connect-payload client metadata for sender fields.

## Solution Assessment

Unclear until maintainer review chooses the public identity contract. Server-side propagation from `authResult.user` is the right implementation boundary, but reusing sender fields versus adding a distinct authenticated identity field is a product/security decision.
