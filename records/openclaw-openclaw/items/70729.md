## Summary

Keep open. Current main still authenticates and scopes trusted-proxy WebChat users, but the accepted proxy subject is not stored on server-owned WebSocket state or attached to the WebChat agent message context, and exposing that identity needs an explicit gateway/WebChat security contract.

## What This Changes

Expose the trusted-proxy `X-Forwarded-User` identity from WebChat sessions to agent message context.

## Best Possible Solution

Persist the accepted trusted-proxy subject from successful server-side auth on WebSocket connection state, expose it through a documented WebChat identity contract, and add spoofing plus non-trusted-auth regression coverage.

## Reproduction Assessment

Yes, at source level. Current main returns a trusted-proxy `user`, drops it during WebSocket registration, and builds WebChat `MsgContext` without any server-owned proxy identity available.

## Solution Assessment

Unclear until maintainers choose the public identity contract. Server-side propagation from `authResult.user` is the right boundary, but the target field semantics and always-on versus opt-in behavior need product/security approval.
