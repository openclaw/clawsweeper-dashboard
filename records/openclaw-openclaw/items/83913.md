Keep open: current main still assigns directly to Axios interceptor `handlers`, the Feishu tests do not exercise that interceptor surface, and upstream Axios provides a public `clear()` interceptor API, so this remains a narrow maintainability cleanup rather than a close candidate.

## What This Changes

The issue asks OpenClaw to replace Feishu's direct Axios `interceptors.request.handlers` mutation with a public-contract User-Agent override and regression coverage.

## Best Possible Solution

Refactor Feishu HTTP User-Agent handling onto public Lark/Axios request contracts and add focused coverage that exercises the actual interceptor or custom `HttpInstance` path.

## Reproduction Assessment

No high-confidence current-main runtime failure path is proven. Source inspection does reproduce the direct `handlers = []` mutation and missing coverage, but the user-visible failure depends on a dependency representation change or upgrade scenario.

## Solution Assessment

Yes, the requested direction is the maintainable fix. The narrowest path is to stop assigning `handlers` directly, preserve the OpenClaw Feishu User-Agent through a public request wrapper or interceptor API, and cover that behavior in `client.test.ts`.
