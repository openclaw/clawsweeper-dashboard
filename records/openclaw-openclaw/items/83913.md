## Summary

Keep open: current main and the latest stable release still reset Feishu's Axios request interceptor by assigning to `handlers`, and dependency inspection shows supported seams exist that can preserve the custom User-Agent without reaching into that internal array.

## What This Changes

The issue requests replacing Feishu's User-Agent interceptor reset so it no longer depends on Axios interceptor internals.

## Best Possible Solution

Move the final Feishu User-Agent injection to a documented SDK/Axios seam, preferably the existing timeout-aware `httpInstance` wrapper, and add a focused regression test that proves outgoing SDK requests carry `getFeishuUserAgent()` without touching `handlers`.

## Reproduction Assessment

Yes for the source-level concern: current main and v2026.5.18 still assign to `inst.interceptors.request.handlers`. A live failure is not reproduced against the current pinned dependency because the reported symptom depends on an incompatible future SDK/Axios change.

## Solution Assessment

No: the current approach is not the best solution because it depends on Axios interceptor storage internals even though the Lark SDK already accepts a custom `httpInstance` and Axios exposes public interceptor methods. A narrow wrapper-level header injection plus regression coverage is more maintainable.
