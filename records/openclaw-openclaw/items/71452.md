## Summary

Keep open. Current main still has the observable message CLI mismatch: provider paths can request or fetch more than 25 message-like rows, but the shared text formatter still truncates rendered read, list-pins, and search output at 25 and does not surface a reusable pagination token contract.

## What This Changes

The issue asks for message read/list-pins/search and related chat-listing paths to honor caller limits and support pagination instead of hardcoded 25-row output caps.

## Best Possible Solution

Implement an additive message-action pagination contract: thread caller display limits into formatMessageCliText/renderMessageList while preserving the default 25-row display when no explicit limit is supplied, and return provider pagination metadata consistently where stable upstream cursors already exist.

## Reproduction Assessment

Yes. A focused unit test can construct read, list-pins, or Discord-style search payloads with more than 25 message-like rows and call formatMessageCliText; current main truncates through messages.slice(0, 25).

## Solution Assessment

Yes. The narrow maintainable direction is a shared formatter/runner contract fix plus provider-specific cursor metadata where supported, rather than one-off provider-only changes that leave the CLI renderer capped.
