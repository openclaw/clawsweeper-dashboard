## Summary

Keep open. Current main still hard-caps shared CLI message rendering at 25 rows while the CLI and several providers can request or fetch more, so the reported pagination/limit mismatch remains valid.

## What This Changes

The issue requests pagination and limit-aware rendering/fetching for message read, pins, search, and related Teams list actions instead of a fixed 25-row display cap.

## Best Possible Solution

Implement an additive message-action pagination/display-limit contract that preserves the default 25-row display when no explicit limit is requested, honors caller limits for read/pins/search, and surfaces provider next-page metadata consistently where upstream APIs expose it.

## Reproduction Assessment

Yes. A focused unit test can build read, list-pins, or Discord-style search payloads with more than 25 rows and call formatMessageCliText; current main truncates through renderMessageList's messages.slice(0, 25).

## Solution Assessment

Yes. The narrow maintainable fix is to thread caller limits through the shared formatter/runner boundary and add provider cursor metadata where available, rather than patching one provider while leaving the shared CLI renderer capped.
