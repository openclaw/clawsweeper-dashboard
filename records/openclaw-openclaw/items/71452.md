## Summary

Keep open: current main and the latest release still hard-cap shared CLI message rendering at 25 rows, and the linked closing pull request is open but not merged.

## What This Changes

The issue asks for message read, pinned-message listing, search, and related Teams listing paths to honor requested limits and support pagination instead of truncating CLI output at 25 rows.

## Best Possible Solution

Review and land or replace https://github.com/openclaw/openclaw/pull/79593 for the shared formatter limit and pagination-hint fix, while tracking provider-specific pagination gaps separately, including https://github.com/openclaw/openclaw/issues/74729.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence repro: pass more than 25 message-like rows through formatMessageCliText for read, list-pins, or search and renderMessageList truncates them with messages.slice(0, 25).

## Solution Assessment

Yes. The narrow maintainable solution is to thread the caller display limit into the shared formatter, preserve the default 25-row behavior when no limit is passed, and surface existing pagination metadata without broad provider rewrites in the first step.
