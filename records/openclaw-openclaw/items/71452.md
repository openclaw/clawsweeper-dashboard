## Summary

Keep open: current main and v2026.5.18 still hard-cap rendered message rows at 25, and the linked fix PR at https://github.com/openclaw/openclaw/pull/79593 is open, unmerged, and currently conflict-marked.

## What This Changes

Requests the message CLI to honor caller-supplied limits and support pagination metadata for message, pin, search, and chat listing output instead of a fixed 25-row renderer cap.

## Best Possible Solution

Review, land, replace, or close https://github.com/openclaw/openclaw/pull/79593 for the shared formatter limit fix, then keep provider-specific pagination work in narrower follow-up issues.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to pass more than 25 message-like rows through `formatMessageCliText` for `read`, `list-pins`, or Discord `search`; current main truncates through `messages.slice(0, 25)`.

## Solution Assessment

Yes for the central bug. The narrow maintainable fix is to thread the caller display limit into the shared formatter while preserving the default 25-row behavior; broader provider cursor pagination can follow separately.
