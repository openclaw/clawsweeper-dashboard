## Summary

Keep open. Current main still has the shared CLI renderer hard-capped at 25 rows, and the open issue-fixing pull request at https://github.com/openclaw/openclaw/pull/79593 references this issue with closing syntax but has not merged yet.

## What This Changes

Request to honor message action limits and add pagination/cursor support for message and chat listing instead of an unconditional 25-row renderer cap.

## Best Possible Solution

Review and land or replace https://github.com/openclaw/openclaw/pull/79593 for the shared formatter limit/hint fix, then keep provider-specific pagination gaps tracked by narrower follow-ups such as https://github.com/openclaw/openclaw/issues/74729.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence unit repro: build a `read`, `list-pins`, or Discord `search` payload with more than 25 message-like rows and call `formatMessageCliText`; current main truncates through `messages.slice(0, 25)`.

## Solution Assessment

Yes. The best direction is a narrow additive formatter contract that preserves the default 25-row display, honors explicit caller limits, and surfaces existing provider pagination metadata without broad provider rewrites in the first step.
