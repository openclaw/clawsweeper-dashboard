## Summary

Keep this issue open: current main and the latest release still hard-cap rendered message rows at 25, while the only linked fixing PR is still open and conflicting rather than merged.

## What This Changes

The issue asks for message read, pins, search, and related chat/message listing paths to honor caller limits and expose pagination instead of truncating rendered output at 25 rows.

## Best Possible Solution

Review and repair https://github.com/openclaw/openclaw/pull/79593 or replace it with a narrow PR that threads caller limits through shared CLI rendering, preserves the default cap, exposes pagination metadata where providers support it, and adds formatter coverage for read, pins, and search.

No, not fully yet. The linked PR is aimed at the right shared formatter boundary, but it is still open, conflicting, and needs maintainer review for the remaining pagination contract details.
