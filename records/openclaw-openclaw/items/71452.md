## Summary

Keep open. Current main still has the reported user-visible mismatch: the CLI exposes limits for message reads, pins, and search, but the shared text renderer hard-caps message-like tables at 25 rows, and provider pagination metadata is still uneven across the affected surfaces.

## What This Changes

The issue asks for message read, pins/search output, and related chat/channel listing paths to honor caller limits and support pagination instead of fixed one-page or 25-row behavior.

## Best Possible Solution

Implement an additive message action contract that threads caller limits into CLI rendering, preserves the default 25-row display when no limit is supplied, and exposes next-page metadata consistently where providers already support it; keep this issue as the canonical shared tracking item and treat the Feishu report as adjacent provider evidence.

## Reproduction Assessment

Yes. A focused unit check can pass read, list-pins, or Discord-style search payloads with more than 25 entries through formatMessageCliText and observe truncation from messages.slice(0, 25); this review stayed read-only, so tests were not executed.

## Solution Assessment

Yes. The narrow maintainable fix is to thread existing caller limits and pagination metadata across the action/formatter boundary, then add provider-specific cursor support only where the provider API already exposes it.
