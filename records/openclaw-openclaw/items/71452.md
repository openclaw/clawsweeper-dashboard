## Summary

Keep open. Current main still hard-caps shared message-like CLI rendering at 25 rows, while CLI and provider paths can request or return larger or paginated result sets, so the reported limit/pagination mismatch remains valid.

## What This Changes

The issue requests pagination and caller-controlled limits for chat/message list output instead of a fixed 25-row renderer cap.

## Best Possible Solution

Implement an additive message-action pagination contract that preserves the default 25-row display when no explicit limit is requested, honors caller limits in read/pins/search rendering, and surfaces provider pagination metadata consistently.

## Reproduction Assessment

Yes. A focused unit test can build `read`, `list-pins`, or Discord-style `search` payloads with more than 25 rows and call `formatMessageCliText`; current main truncates through `messages.slice(0, 25)`.

## Solution Assessment

Yes. The narrow maintainable fix is to thread caller limits through the shared formatter/runner boundary and add cursor metadata where providers already expose it, rather than fixing only one provider while the renderer stays capped.
