## Summary

Keep open. Current main still hard-caps shared message CLI text output at 25 rows, so read, list-pins, and search can still hide rows even when CLI/provider paths request or fetch more.

## What This Changes

The issue requests pagination and caller-controlled limits for message read, list-pins, search, and chat-list style results instead of a hardcoded 25-row output cap.

## Best Possible Solution

Implement an additive message-action pagination/display contract that honors explicit caller limits, keeps the default 25-row display when no limit is provided, and exposes stable provider cursor metadata without breaking existing JSON payloads.

## Reproduction Assessment

Yes. A focused unit test can build read, list-pins, or Discord-style search payloads with more than 25 message-like rows and call formatMessageCliText; current main truncates through renderMessageList. I did not run tests because this was a read-only review and pnpm is unavailable in the environment.

## Solution Assessment

Yes. The narrow maintainable solution is to thread caller limits and pagination metadata through the shared formatter/action boundary, then add provider cursor fields only where upstream APIs already expose them.
