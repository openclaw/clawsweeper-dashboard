## What This Changes

The issue asks for message listing output to honor caller limits and expose pagination instead of truncating rendered message, pin, and search rows at 25.

## Best Possible Solution

Review and repair `https://github.com/openclaw/openclaw/pull/79593`, or replace it with a narrow PR that threads caller limits through shared CLI rendering, preserves default behavior, exposes pagination hints where providers support them, and adds formatter coverage for read, pins, and search.

Yes. At source level, current main exposes `--limit` for affected message actions, but `renderMessageList` still slices rendered rows to 25 and `messageCommand` does not pass the caller limit into the formatter.
