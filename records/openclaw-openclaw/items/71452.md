## Summary

Keep open. Current main still has the observable CLI mismatch: message-like table output is capped at 25 rows, while limit-capable actions and partial provider pagination cannot consistently expose larger or cursor-based result sets.

## What This Changes

The issue requests pagination and caller-controlled limits for message read, list-pins, search, and chat/channel listing paths instead of a fixed 25-row display cap.

## Best Possible Solution

Implement an additive message-action pagination contract: thread caller display limits into the CLI formatter, preserve the default 25-row display when no explicit limit is supplied, and return provider pagination metadata consistently where provider APIs already expose it.

## Reproduction Assessment

Yes. A focused unit test can construct `read`, `list-pins`, or Discord-style `search` payloads with more than 25 message-like rows and call `formatMessageCliText`; current main truncates through `messages.slice(0, 25)`.

## Solution Assessment

Yes. The narrow maintainable solution is to carry existing `limit`, `pageSize`, and `pageToken`-style parameters through the runner/formatter boundary and add provider-specific pagination only where the provider already has a stable upstream concept.
