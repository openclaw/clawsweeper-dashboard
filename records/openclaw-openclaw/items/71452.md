## Summary

Keep open. Current main still truncates message-like CLI text output at 25 rows while CLI and provider paths can request or fetch more, so the observable pagination/limit mismatch remains valid.

## What This Changes

The issue requests pagination and caller-controlled limits for message read, pins, search, and related chat listing output instead of a hardcoded 25-row cap.

## Best Possible Solution

Implement an additive message-action pagination/display-limit contract: keep the default 25-row display when no explicit limit is passed, honor caller limits in read/pins/search rendering, and surface stable provider pagination metadata without breaking existing JSON payloads.

## Reproduction Assessment

Yes. A focused unit test can build read, list-pins, or Discord-style search payloads with more than 25 message-like rows and call `formatMessageCliText`; current main truncates through `messages.slice(0, 25)`.

## Solution Assessment

Yes. The narrow maintainable fix is to thread caller limits through the shared formatter/runner boundary and add provider cursor metadata where upstream APIs already expose it, rather than fixing one provider while leaving the CLI renderer capped.
