## Summary

Keep open. Current main still has the observable mismatch: message-like CLI output is capped at 25 rows, while limit-capable actions and partial provider pagination remain unable to expose larger or cursor-based result sets consistently.

## What This Changes

The issue asks the message CLI/action contract to honor caller limits and expose pagination for read, list-pins, search, and related chat-list actions instead of truncating rendered results at 25 rows.

## Best Possible Solution

Implement an additive message action contract that threads caller limits into the CLI formatter, preserves the default 25-row display when no limit is supplied, and exposes provider pagination metadata where existing provider APIs already return it.

## Reproduction Assessment

Yes. A focused unit check can pass read, list-pins, or Discord-style search payloads with more than 25 message-like entries into formatMessageCliText and observe truncation from messages.slice(0, 25); tests were not run because this review was read-only.

## Solution Assessment

Yes. The narrow maintainable path is to carry the existing limit and next-page metadata through the runner/formatter boundary, with provider-specific pagination support added only where the provider already has a stable cursor or nextLink concept.

Keep the change narrow and additive: thread the caller's requested limit through message action results into formatMessageCliText/renderMessageList, preserve the existing 25-row default when no limit is supplied, and remove/configure the hardcoded renderer cap when a higher explicit limit is present. Add focused regression tests for read, list-pins, and search rendering more than 25 rows when an explicit limit is requested, and defaulting to at most 25 rows when omitted.
