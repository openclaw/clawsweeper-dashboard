## Summary

Keep open. Current main still hard-caps rendered message rows at 25, and the open fix pull request at https://github.com/openclaw/openclaw/pull/79593 references this issue with closing syntax but has not merged.

## What This Changes

The issue requests that message read, list-pins, search, and related chat listing paths honor caller limits and expose pagination instead of truncating rendered results at 25 rows.

## Best Possible Solution

Review and land or replace https://github.com/openclaw/openclaw/pull/79593 for the shared formatter limit and pagination-hint fix, while tracking provider-specific pagination work separately in issues such as https://github.com/openclaw/openclaw/issues/74729.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence unit repro: pass a read, list-pins, or Discord search payload with more than 25 message-like rows to formatMessageCliText and current main truncates through messages.slice(0, 25).

## Solution Assessment

Yes. The narrow maintainable direction is to thread the caller display limit into the shared formatter, preserve the default 25-row display, and surface existing pagination metadata without broad provider rewrites in the first step.
