## Summary

Keep open. Current main still hard-caps rendered message rows at 25, and the open fix pull request at https://github.com/openclaw/openclaw/pull/79593 references this issue with closing syntax but has not merged.

## What This Changes

The issue requests that message read, pins, search, and chat listing honor caller limits and support pagination instead of a shared 25-row text-rendering cap.

## Best Possible Solution

Review and land or replace https://github.com/openclaw/openclaw/pull/79593 for the shared formatter limit and pagination-hint fix, while tracking provider-specific pagination separately through focused follow-ups such as https://github.com/openclaw/openclaw/issues/74729.

## Reproduction Assessment

Yes. A high-confidence source-level unit repro is to pass more than 25 message-like rows through `formatMessageCliText` for `read`, `list-pins`, or Discord `search`; current main truncates through `messages.slice(0, 25)`.

## Solution Assessment

Yes for the central bug. The narrow maintainable fix is to thread the caller display limit into the shared formatter, preserve the default 25-row behavior when no limit is supplied, and surface existing pagination metadata without broad provider rewrites in the first step.
