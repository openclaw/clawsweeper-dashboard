## Summary

Keep open: current main and v2026.5.18 still cap CLI-rendered message rows at 25, and the linked fix PR at https://github.com/openclaw/openclaw/pull/79593 remains open and unmerged.

## What This Changes

The issue asks for message read, pins, search, and related chat/message listing paths to honor caller limits and expose pagination instead of a fixed 25-row cap.

## Best Possible Solution

Review, land, replace, or close https://github.com/openclaw/openclaw/pull/79593 for the shared formatter limit fix, then track provider-specific pagination gaps through focused follow-ups such as https://github.com/openclaw/openclaw/issues/74729.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to pass more than 25 message-like rows through `formatMessageCliText` for `read`, `list-pins`, or Discord `search`; current main truncates via `messages.slice(0, 25)`.

## Solution Assessment

Yes for the central bug. The narrow maintainable fix is to thread the caller display limit into the shared formatter while preserving the default 25-row behavior; broader provider cursor pagination can follow separately.
