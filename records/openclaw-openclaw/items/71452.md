## Summary

Keep open. Current main still has the observable mismatch reported here: the shared CLI formatter hard-caps rendered message-like rows at 25, while the CLI and tool schema expose larger limits and provider-side pagination is only partial.

## What This Changes

Request pagination and caller-controlled limits for message read, list-pins, search, and chat/channel listing paths instead of a fixed 25-row display cap.

## Best Possible Solution

Implement an additive message-action pagination contract: pass the caller's requested display limit into CLI text rendering for `read`, `list-pins`, and search results, keep the current 25-row default when no explicit limit is supplied, and surface provider pagination metadata consistently where existing provider APIs already return it.

## Reproduction Assessment

Yes. A focused unit test can construct `read`, `list-pins`, or Discord-style `search` payloads with more than 25 message-like rows and call `formatMessageCliText`; current main truncates through `messages.slice(0, 25)`.

## Solution Assessment

Yes. The narrow maintainable path is to thread existing `limit`, `pageSize`, and `pageToken`-style parameters through the runner/formatter boundary and add provider-specific pagination only where the provider already supports it.
