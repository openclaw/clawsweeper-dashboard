## Summary

Keep open. Current main still hard-caps rendered message output at 25 rows, so caller limits and provider-side pagination cannot be observed in the CLI text paths.

## What This Changes

The issue requests limit-aware and cursor-aware pagination for message read, pinned-message listing, search, and possibly Microsoft Teams chat listing.

## Best Possible Solution

Add an additive message-action display-limit and pagination metadata contract that preserves the default 25-row display when no explicit limit is passed, honors requested limits in read/pins/search rendering, and exposes provider cursors consistently where available.

## Reproduction Assessment

Yes. A focused unit test can construct read, list-pins, or Discord search payloads with more than 25 message-like rows and call `formatMessageCliText`; current main truncates through `messages.slice(0, 25)`.

## Solution Assessment

Yes. The narrow maintainable direction is to thread caller limits through the shared runner/formatter boundary and normalize existing provider pagination hints, rather than patching one provider while leaving the shared CLI renderer capped.

## Repair Work Prompt

Fix the message CLI pagination/display-limit mismatch tracked by https://github.com/openclaw/openclaw/issues/71452. Keep the change narrow and additive: preserve the existing default 25-row display when no explicit limit is provided, thread the caller's normalized `limit` from the message action invocation into `formatMessageCliText`/`renderMessageList`, and make read, list-pins, and search text output honor limits greater than 25. Where provider payloads already expose pagination hints (`hasMore`, `nextBatch`/`prevBatch`, Teams `@odata.nextLink`), surface stable additive metadata without renaming/removing existing fields. If touching Microsoft Teams pins, pass the caller limit through the adapter and keep existing `@odata.nextLink` behavior. Do not claim to resolve Feishu-specific issue #74729 unless Feishu read-list pagination is actually implemented. Add focused regression tests showing more than 25 rendered rows for read, pins, and search when an explicit limit is present, plus default behavior when it is absent. Add a user-facing changelog entry if behavior changes are user visible.
