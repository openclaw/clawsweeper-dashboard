## Summary

Keep open. Current main still has the user-visible mismatch: the shared message text formatter truncates read, pin, and search tables at 25 rows even when the CLI and some providers accept larger limits or expose pagination metadata.

## What This Changes

The issue requests that message read, list-pins, search, and related chat-list paths respect caller limits and support pagination instead of relying on a hardcoded 25-row rendering cap.

## Best Possible Solution

Implement an additive message-action pagination contract: thread the explicit caller display limit into `formatMessageCliText` and `renderMessageList`, keep the existing 25-row default when no limit is provided, and surface stable next-page metadata where providers already expose cursors. Update focused tests, CLI docs, and the changelog alongside the code.

## Reproduction Assessment

Yes. A focused unit test can construct read, list-pins, or Discord-style search payloads with more than 25 message-like rows and call `formatMessageCliText`; current main truncates through `messages.slice(0, 25)`.

## Solution Assessment

Yes. The narrowest maintainable fix is a shared formatter/runner contract change plus additive provider cursor metadata, rather than provider-only patches that still leave the CLI renderer capped.
