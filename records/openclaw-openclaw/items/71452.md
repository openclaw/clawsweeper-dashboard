## Summary

Keep open. Current main still has the reported mismatch: text rendering for read, list-pins, and search goes through a shared formatter that hard-caps rows at 25 even though the CLI and several providers accept larger limits or expose partial pagination metadata. The follow-up plan in the discussion is directionally sound and this remains a narrow repair candidate.

## What This Changes

The issue asks message listing actions to honor caller limits and expose pagination metadata instead of hard-capping rendered results at 25 rows.

## Best Possible Solution

Implement an additive message action contract that threads the caller's requested display limit into formatMessageCliText/renderMessageList, keeps the existing 25-row default when no limit is supplied, and surfaces provider pagination metadata only where the provider already has a stable cursor, nextLink, or batch token.

## Reproduction Assessment

Yes. A focused unit check can pass more than 25 read, list-pins, or Discord-style search entries into the current formatter path and observe truncation from messages.slice(0, 25); this review did not run tests because the task is read-only.

## Solution Assessment

Yes. The narrow maintainable fix is to carry existing limit and continuation metadata through the runner/formatter boundary, with provider-specific additions only where current provider APIs already support them.
