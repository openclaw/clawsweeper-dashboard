## Summary

Keep open. Current main still has the observable CLI mismatch: message-like text output is capped at 25 rows, while read/search/pin paths can request or fetch more results and do not expose a consistent pagination contract.

## What This Changes

The issue asks for message list/read/pin/search flows to honor requested limits and expose pagination instead of truncating output at 25 rows.

## Best Possible Solution

Implement an additive message-action pagination contract: thread caller display limits into formatMessageCliText/renderMessageList, preserve the default 25-row display when no explicit limit is supplied, and return provider pagination metadata consistently where each provider already exposes stable upstream cursors.

## Reproduction Assessment

Yes. A focused unit test can construct read, list-pins, or Discord-style search payloads with more than 25 message-like rows and call formatMessageCliText; current main truncates through messages.slice(0, 25).

## Solution Assessment

Yes. The narrow maintainable solution is to carry existing limit and cursor-like parameters through the runner/formatter boundary, then add provider-specific pagination metadata only where the provider already has a stable concept.
