## Summary

Keep open: current main and v2026.6.8 still make user-sourced model overrides strict, while https://github.com/openclaw/openclaw/pull/84867 is an open closing PR and https://github.com/openclaw/openclaw/pull/80181 proposes a competing opt-in policy shape. The remaining work is a maintainer decision about exact user model selection versus resilient fallback behavior.

## What This Changes

The issue asks OpenClaw to avoid session stalls when a user-selected model becomes unavailable by giving that selection a recovery fallback path.

## Best Possible Solution

Maintainers should decide the user-selected model fallback contract, then land one aligned path with source, docs, tests, and runtime proof: strict visible failure if exact selection remains the contract, or an approved resilient or opt-in fallback policy if availability wins.

## Reproduction Assessment

Yes for the source-level behavior: current main returns `[]` for user-sourced session model overrides, and the current command test asserts one failed selected-model attempt with no fallback. The full live provider-outage deadlock was not reproduced in this read-only review.

## Solution Assessment

Unclear: enabling fallback for every user-selected model is a plausible availability fix, but it changes the documented exact-selection contract. A strict visible-failure repair or a maintainer-approved opt-in resilient policy may be safer depending on product direction.
