## Summary

Keep open: current main still treats user-sourced model overrides as strict, and the linked implementation pull request remains open. The remaining decision is whether to change or opt into user-pin fallback semantics, or instead fix the stall while preserving strict selection.

## What This Changes

The issue reports that a user-selected session model disables fallback candidates, leaving provider outages able to stall the session instead of recovering.

## Best Possible Solution

Make the user-pin policy explicit: either preserve strict user selections and ensure provider failures fail visibly without stalls, or land an opt-in resilient fallback policy with aligned docs, regression tests, and live proof.

## Reproduction Assessment

Yes for the source-level fallback path: current main returns an empty fallback override for user-sourced model selections and tests assert only the selected model is attempted. The full provider-outage deadlock was not run end-to-end in this read-only review.

## Solution Assessment

Unclear. The proposed unconditional fallback change may fix availability, but it conflicts with documented strict user-selection semantics; an opt-in policy or a visible-failure stall fix is safer unless maintainers choose to change the default.
