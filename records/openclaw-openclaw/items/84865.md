- `impact:session-state`: The bug centers on persisted session model override state and whether fallback selections are allowed or persisted.
- `impact:auth-provider`: The affected behavior is model/provider routing and fallback selection during provider errors.

## Summary

Keep open: current main still disables configured fallbacks for user-sourced model overrides, and an open linked implementation PR is already the right place to review whether to change that documented policy.

## What This Changes

The issue requests a recovery path for sessions pinned by `/model` so a provider outage does not leave the session stuck on the unavailable model.

## Best Possible Solution

Keep exact user model selection semantics explicit: either preserve strict selection and make provider failures fail visibly without stalls, or intentionally change the documented policy with focused regression tests and docs.

## Reproduction Assessment

Partly: source and tests show user-sourced model overrides resolve to an empty fallback list on current main. The full provider-outage deadlock was not run end-to-end in this read-only review.

## Solution Assessment

Unclear: allowing fallbacks for user-selected models may fix the incident, but it conflicts with current docs that treat `/model` as an exact strict selection. Maintainers should decide the policy before landing the linked PR or an alternative visible-failure fix.
