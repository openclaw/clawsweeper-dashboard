## Summary

Keep open: current main still disables configured fallbacks for user-sourced model overrides, and the linked open PR is the right place to decide whether this should change or whether strict model pins should fail visibly instead of stalling.

## What This Changes

The issue requests allowing user-switched session models to use the configured model fallback chain during provider outages without persisting the fallback choice.

## Best Possible Solution

Preserve exact `/model` selection by default and make provider failures fail visibly or recover without stalls; if maintainers want user-pinned fallback, land a policy-approved change with aligned docs, tests, and live proof.

## Reproduction Assessment

No high-confidence end-to-end reproduction was run in this review. Source and tests do show the no-fallback portion on current main: user-sourced model overrides resolve to an empty fallback list and only the selected model is attempted.

## Solution Assessment

Unclear: enabling fallback for user-selected models would address availability, but it conflicts with the documented exact-selection policy. Maintainers should decide whether to keep strict pins and fix the stall path or approve fallback-eligible user pins with docs and tests.
