## Summary

Keep open: current main and v2026.6.8 still keep user-selected models strict, while the directly linked implementation PR is open and changes a compatibility-sensitive provider/model routing contract. The remaining decision is whether explicit `/model` selections should stay exact, fail visibly without stalling, or opt into resilient fallback behavior.

## What This Changes

The issue requests that user-switched model sessions retain a configured fallback chain during provider outages instead of stalling with `fallbackConfigured=false`.

## Best Possible Solution

Maintainers should decide the user-selection fallback contract, then align code, docs, tests, and runtime proof around that choice: strict visible failure if exact pins remain the contract, or an approved resilient or opt-in fallback policy if availability wins.

## Reproduction Assessment

Yes for the source-level behavior: current main returns `[]` for user-sourced session model overrides, and tests assert one failed attempt with no fallback. The full live provider-outage deadlock was not reproduced in this read-only review.

## Solution Assessment

Unclear. Allowing fallback for every user-selected model is a plausible availability fix, but it conflicts with the documented exact-selection contract; an explicit opt-in fallback policy or a visible-failure/stall repair may be safer depending on maintainer direction.
