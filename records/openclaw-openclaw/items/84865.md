## Summary

Keep open: current main still disables configured fallbacks for user-sourced model overrides, and https://github.com/openclaw/openclaw/pull/84867 is the open linked implementation candidate. The remaining blocker is maintainer policy review of exact user model pins versus fallback recovery, plus proof for the linked PR, not cleanup closure.

## What This Changes

The issue reports that `/model` user session overrides receive an empty fallback chain, so provider timeouts can leave the session stalled instead of recovering.

## Best Possible Solution

Keep exact user model selection semantics explicit: either preserve strict pins and convert provider outages into a visible recoverable failure path, or intentionally add fallback-eligible user pins with aligned docs, tests, and real outage proof.

## Reproduction Assessment

Partly: source and tests show user-sourced model overrides resolve to an empty fallback list on current main, matching the `fallbackConfigured=false` incident evidence. I did not run a live provider-outage reproduction of the full stalled-session behavior.

## Solution Assessment

Unclear: allowing configured fallbacks for user-selected models may address the outage stall, but it conflicts with current exact-selection docs. Maintainers should choose between strict visible failure recovery and an intentional fallback policy before landing the linked PR.
