## Summary

Keep open: current main still gives user-sourced model overrides an empty fallback list, and the linked implementation PR is open but needs maintainer policy review because current docs define `/model` as an exact selection.

## What This Changes

The issue requests fallback recovery for user-selected `/model` overrides during provider outages without persisting the fallback selection.

## Best Possible Solution

Decide the user-selection policy first, then either keep strict pins and fix stalled runs to fail visibly, or deliberately allow fallback for user selections with aligned docs, tests, and redacted live failover proof.

## Reproduction Assessment

Partly: source inspection gives a high-confidence path for the disabled fallback chain because current main returns `[]` for `modelOverrideSource: "user"`. I did not reproduce the full live provider-outage stall end to end.

## Solution Assessment

Unclear: allowing fallbacks for user-selected models may fix the incident, but it changes the documented exact-selection contract. Maintainers should choose between strict visible failure and explicit fallback-eligible user pins before landing the linked PR.
