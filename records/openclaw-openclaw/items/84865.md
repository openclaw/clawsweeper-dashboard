## Summary

Keep open: current main and the latest shipped release still implement strict user model overrides, and the open linked PR is the right place to decide whether to change that policy or instead fix the stall-to-visible-error path.

## What This Changes

The issue requests changing user-selected model overrides so configured model fallbacks can recover from provider outages instead of leaving the session stalled.

## Best Possible Solution

Keep user model selection semantics explicit: either preserve strict selections and make provider failures fail visibly without stalls, or intentionally add fallback eligibility with matching docs, tests, and live proof.

## Reproduction Assessment

No high-confidence end-to-end reproduction was run in this read-only review. Source inspection confirms the disabled fallback chain for `modelOverrideSource: "user"`, and the issue provides incident logs for the provider-outage stall.

## Solution Assessment

Unclear: the proposed fallback-chain fix may address the incident, but it conflicts with the documented exact-selection contract. Maintainers should decide whether to change `/model` policy or keep it strict and repair the visible-failure path.
