## Summary

Keep open: current main archives a prior transcript on the normal `/new` path, but archive failures are still swallowed as best-effort, so the silent-success data-loss path remains plausible and is not fully fixed.

## What This Changes

`/stop` followed by `/new` should not report success unless the prior session transcript is durably archived or an explicit archive error is surfaced.

## Best Possible Solution

Keep `/stop` abort-only, make `/new` and reset archive failures explicit before reporting success, preserve archive update notifications, and add regression coverage for `/stop` then `/new` with non-empty transcript content.

## Reproduction Assessment

No, not with high confidence from a live setup. Source inspection shows `/new` can silently continue after archive rename failure because the archive helper catches errors, while current tests only prove the happy path.

## Solution Assessment

No, the literal `/stop`-archives-before-returning direction conflicts with the documented abort-only `/stop` contract. The narrower maintainable fix is to make the reset/new archive operation durable or fail-loud and cover the `/stop` followed by `/new` sequence.
