## Label Justifications

- `P1`: A real session rollover workflow can silently hide transcript archive failure and affect downstream ingestion for active users.
- `impact:data-loss`: The issue is about session transcript content not being durably archived or discoverable after rollover.
- `impact:session-state`: The failure sits in reset/new transcript lifecycle state, where user-visible session success can diverge from archive state.

## Summary

Keep open: current main and the latest release still swallow transcript archive rename failures during reset/new rollover, and the linked fix at https://github.com/openclaw/openclaw/pull/82081 is still open rather than merged.

## What This Changes

The issue requests durable archiving and visible failure reporting when `/stop` is followed by `/new` and the prior session transcript cannot be archived.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/82081 or an equivalent narrow fix that surfaces reset/new archive failures while preserving the documented `/stop` abort-only behavior.

## Reproduction Assessment

Yes at source level: if `archiveFileOnDisk` throws during `/new` rollover, `archiveSessionTranscriptsDetailed` catches and ignores it, so the new session can proceed without surfacing the failed archive. I did not run a live gateway or Telegram reproduction in this read-only review.

## Solution Assessment

Yes in direction: surfacing reset/new archive failures is the narrow maintainable fix, while changing `/stop` into a transcript finalizer would conflict with the documented command contract. The open linked PR is the active candidate and should be reviewed or replaced before this issue closes.
