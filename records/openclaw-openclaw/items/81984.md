## Summary

Keep open: current main has the normal `/new` archive path, but archive rename failures are still swallowed as best-effort, so the silent-success transcript-loss path remains plausible and no merged fix or canonical duplicate covers this exact durability bug.

## What This Changes

The issue reports that `/stop` followed immediately by `/new` can return success while the stopped session transcript is not durably archived.

## Best Possible Solution

Keep `/stop` abort-only, make `/new`/reset transcript archiving durable or fail-loud before reporting success, preserve archive update notifications, and add regression coverage for `/stop` then `/new` plus archive failure handling.

## Reproduction Assessment

No, not with a live setup. Source inspection shows a current-main path where `/new` calls `archiveSessionTranscriptsDetailed`, and that helper catches archive rename failures and lets the reset continue successfully.

## Solution Assessment

Yes for keeping this open and fixing reset/archive durability; no for making `/stop` itself the archive boundary because docs define `/stop` as abort-only. The narrower maintainable fix is to make `/new`/reset fail loudly or otherwise wait for a durable archive result.
