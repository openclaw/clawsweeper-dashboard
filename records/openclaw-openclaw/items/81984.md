## Summary

Keep open: current main has a normal `/new` archive path, but archive rename failures are still swallowed as best-effort, so the reported silent-success transcript loss remains plausible and narrowly actionable.

## What This Changes

/stop followed by immediate /new is reported to return success while failing to persist the stopped session transcript archive.

## Best Possible Solution

Keep `/stop` abort-only, make explicit `/new` and reset rollover surface archive failures before reporting success, and add focused regression coverage for the `/stop` then `/new` archive path.

## Reproduction Assessment

No live high-confidence reproduction was established, but source inspection gives a high-confidence failure path: archive rename errors are caught and ignored while reset/new callers can still return success.

## Solution Assessment

No for the literal request to make `/stop` archive before returning, because docs define `/stop` as abort-only. The maintainable fix is to make the reset/new archive operation durable or fail-loud while preserving the current command boundary.
