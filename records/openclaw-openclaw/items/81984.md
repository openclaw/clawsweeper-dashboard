## Summary

Keep open: current main still catches and ignores transcript archive rename failures during session rollover, and https://github.com/openclaw/openclaw/pull/82081 is open and unmerged, so this should close only after that fix or an equivalent lands.

## What This Changes

The issue reports that `/stop` followed by `/new` can return success while the previous session transcript is not durably archived for downstream ingestion.

## Best Possible Solution

Land the linked fix or an equivalent patch that makes previous-transcript archive failures observable during session rollover and adds regression coverage for the rename failure path.

## Reproduction Assessment

Yes at source level: if `archiveFileOnDisk` throws during `/new` rollover, `archiveSessionTranscriptsDetailed` catches and ignores it, so the new session can continue without surfacing the failed archive. I did not run a live gateway reproduction in this read-only review.

## Solution Assessment

Yes in direction: surfacing reset/new archive failures is the narrow maintainable fix, while changing `/stop` into a transcript finalizer would conflict with the documented command contract. The linked PR is the active candidate and should be reviewed, landed, or replaced before this issue closes.
