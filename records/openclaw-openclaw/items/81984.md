## Summary

Keep open: current main and the latest release still swallow transcript archive rename failures during `/new` rollover, and the linked closing PR is open rather than merged.

## What This Changes

The issue reports that `/stop` followed by `/new` can silently leave the previous session transcript unarchived and invisible to downstream ingestion.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/82081 or an equivalent narrow fix that surfaces reset/new archive failures while preserving `/stop` as abort-only.

## Reproduction Assessment

Yes at source level, but not as a live gateway run: if `archiveFileOnDisk` throws during `/new` rollover, `archiveSessionTranscriptsDetailed` catches and ignores it, so the new session can continue without surfacing the failed archive.

## Solution Assessment

Yes in direction: surfacing reset/new archive failures is the narrow maintainable fix, while changing `/stop` into a transcript finalizer would conflict with the documented command contract.
