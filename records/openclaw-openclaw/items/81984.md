## What This Changes

The issue reports that `/stop` followed by `/new` can silently lose the previous session transcript when rollover archiving fails.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/82081 or an equivalent narrow fix that surfaces reset/new archive failures while preserving `/stop` as an abort command.

## Reproduction Assessment

Yes at source level: force `archiveFileOnDisk`/`fs.renameSync` to fail during `/new` rollover and current main catches the error as best-effort while continuing. I did not run a live Telegram/gateway reproduction in this read-only review.

## Solution Assessment

Yes in direction: the narrow maintainable fix is to surface or fail the reset/new archive error at the rollover boundary. Changing `/stop` into a transcript finalization command is not the best fit for the documented command contract.
