## Label Justifications

- `P1`: A real session workflow can silently lose or hide transcript archive content during `/new` rollover, affecting downstream users now.
- `impact:data-loss`: The report and source path concern transcript content not being durably archived or visible to ingestion after session rollover.
- `impact:session-state`: The failure sits in session rollover and transcript lifecycle state, where prior-session archive state can drift from the user-visible session result.

## Summary

Keep open: current main still catches and ignores transcript archive rename failures during `/new` rollover, and the linked fix PR is open rather than merged, so this issue should stay open until that path lands or is replaced.

## What This Changes

The issue requests that `/stop` followed by `/new` preserve the prior transcript archive or visibly surface archive failure instead of silently losing the stopped session's content.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/82081 or an equivalent narrow fix that surfaces reset/new archive failures while preserving the documented `/stop` abort-only behavior.

## Reproduction Assessment

Yes at source level, but not as a live gateway run: if `archiveFileOnDisk` throws during `/new` rollover, `archiveSessionTranscriptsDetailed` catches and ignores it, so the new session can continue without surfacing the failed archive.

## Solution Assessment

Yes in direction: surfacing reset/new archive failures is the narrow maintainable fix, while changing `/stop` into a transcript finalizer would fight the documented command contract. The open linked PR is the active candidate but still needs normal review/proof before this issue closes.
