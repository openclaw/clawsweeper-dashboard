## Summary

Keep open: current main still catches and ignores transcript archive rename failures during `/new` rollover, and the linked fix PR is open rather than merged.

## What This Changes

The issue reports that `/stop` followed by `/new` can silently fail to archive the previous session transcript when the archive/rename operation fails.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/82081 or an equivalent narrow fix that surfaces `/new` rollover archive failures, adds failure-path coverage, and preserves the documented `/stop` abort-only contract.

## Reproduction Assessment

Yes at source level, but not as a live gateway run: if `archiveFileOnDisk` throws during `/new` rollover, `archiveSessionTranscriptsDetailed` catches and ignores it so the new session can continue without surfacing the failed archive.

## Solution Assessment

Yes in direction: surfacing reset/new archive failures is the narrow maintainable fix, while changing `/stop` into a transcript finalizer would fight the documented command contract. The open linked PR is the active candidate, with a maintainer decision still needed on warn-versus-block behavior.
