## Label Justifications

- `P1`: A real session rollover workflow can hide transcript archive failure and affect downstream users before they discover missing content.
- `impact:data-loss`: The report and source path concern transcript content not being durably archived or visible to ingestion after session rollover.
- `impact:session-state`: The failure sits in session rollover and transcript lifecycle state, where prior-session archive state can diverge from the successful `/new` result.

## Summary

Keep open: current main and the latest release still suppress transcript archive failures during `/new` rollover, and the linked fixing PR remains open rather than merged.

## What This Changes

The issue requests durable transcript archival or visible archive failure when a user runs `/stop` followed by `/new`.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/82081 or an equivalent narrow fix that surfaces `/new` archive failures while preserving the documented `/stop` abort contract.

## Reproduction Assessment

Yes at source level: a failing `archiveFileOnDisk` call during `/new` rollover is swallowed by `archiveSessionTranscriptsDetailed`, so the new session can continue without surfacing the archive failure. I did not run a live gateway/systemd reproduction in this read-only review.

## Solution Assessment

Yes in direction, with maintainer judgment still needed on warning versus fail-loud behavior. The narrow maintainable solution is to surface reset/new archive failures and add regression coverage without changing `/stop` into a transcript finalizer.
