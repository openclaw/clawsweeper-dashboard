## Summary

Keep open: current main and the latest release archive the prior transcript on the normal `/new` path, but archive rename failures are still swallowed as best-effort, so a silent-success data-loss path remains unresolved. No open PR currently references this issue with closing syntax.

## What This Changes

The issue asks for `/stop` followed by `/new` to durably archive the prior transcript and surface archive failures instead of silently succeeding.

## Best Possible Solution

Keep `/stop` abort-only; make explicit `/new` and reset archive failures loud before reporting success, preserve archive update notifications, and add focused regression coverage for `/stop` then `/new` plus archive failure handling.

## Reproduction Assessment

No high-confidence live reproduction is available. Source inspection does show a current-main failure path where `/new` can continue successfully after `archiveFileOnDisk` throws, because `archiveSessionTranscriptsDetailed` catches and ignores the error.

## Solution Assessment

Yes for a narrow repair, but not the literal `/stop` archives-before-returning direction. The maintainable fix is to keep `/stop` as abort-only and make the reset/new archive operation durable or fail-loud.
