## What This Changes

The issue requests that `/stop` followed by `/new` durably archives the prior transcript or surfaces archive failure instead of silently losing session content.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/82081 or an equivalent narrow fix that surfaces previous-transcript archive failures during `/new` rollover while preserving `/stop` as abort-only.

Yes at source level: if `archiveFileOnDisk` throws during `/new` rollover, current main catches and ignores the failure in `archiveSessionTranscriptsDetailed`. I did not run a live gateway or Telegram reproduction in this read-only review.

## Solution Assessment

Yes in direction: surfacing reset/new archive failures is the narrow maintainable fix, while changing `/stop` into a transcript finalizer would conflict with the documented command contract. The active linked PR is the current fix path, so this issue should close only after that or an equivalent patch lands.
