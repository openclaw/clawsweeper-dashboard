## Summary

Keep open: current main still swallows previous-transcript archive failures during `/new` rollover, and the linked open PR is the active fix path rather than a reason to close this issue before merge.

## What This Changes

The issue requests that `/stop` followed by `/new` reliably persist or loudly surface failures when archiving the prior session transcript.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/82081, or an equivalent narrow fix that surfaces `/new` archive failures while keeping `/stop` abort-only.

## Reproduction Assessment

No live end-to-end reproduction was established, but source inspection gives a high-confidence path: `fs.renameSync` failures are swallowed and `/new` uses that helper during prior-session rollover.

## Solution Assessment

Yes in direction: surfacing archive failures from the reset/new archive operation is the narrow maintainable fix. Changing `/stop` into a transcript-finalization command or adding a new journal feature is not required for the core silent-loss bug.
