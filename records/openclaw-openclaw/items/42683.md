## Summary

Keep open: the underlying iMessage reply-context gap is still valid, but this PR is not merge-ready because it patches the removed `src/imessage` tree and resolves quote context from the less reliable `reply_to_guid` path.

## What This Changes

The PR adds a read-only `chat.db` lookup in the old iMessage monitor to populate missing inbound `reply_to_*` fields, plus focused unit coverage.

## Best Possible Solution

Land a narrow active-plugin implementation that enriches missing inbound iMessage reply context before enqueue, uses the reliable reply-origin field, fails closed on DB/schema errors, skips remote SSH-wrapper local DB reads, and includes regression tests plus a changelog entry.

## Reproduction Assessment

Yes. Source inspection shows the active monitor parses and enqueues inbound messages without fallback enrichment, while the downstream context builder only emits reply context when resolved `reply_to_text` is already present.

## Solution Assessment

No. The direction is useful, but this patch is not the best landing shape because it modifies a removed path and bases lookup on `reply_to_guid` instead of the reliable reply-origin field identified in related iMessage analysis.

## Review Findings
