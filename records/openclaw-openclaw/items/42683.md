## Summary

Keep open. The bug is real and source-reproducible, but this PR is stale against current main: it changes removed `src/imessage` files, uses the less reliable `reply_to_guid`, and lacks the required changelog entry, so it should be ported or replaced rather than merged as-is.

## What This Changes

Adds a read-only iMessage `chat.db` reply-context lookup to the old monitor path and unit tests for missing, valid, and incompatible reply rows.

## Best Possible Solution

Land a narrow active iMessage plugin change that enriches missing inbound reply context from the configured local `chat.db` using `thread_originator_guid`, skips remote SSH-wrapper local reads, fails closed on DB/schema errors, and documents the user-visible fix.

## Reproduction Assessment

Yes. Source inspection of current main shows the active iMessage monitor parses and enqueues RPC messages with no fallback, while reply context is only built when `reply_to_text` is already present; the PR body also gives concrete macOS reproduction steps.

## Solution Assessment

No. The direction is useful, but this patch is not the best landing shape because it modifies a removed path and resolves quote targets through `reply_to_guid` instead of the more reliable `thread_originator_guid`.
