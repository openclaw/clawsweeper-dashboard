## Summary

Keep open, but this PR needs changes before merge: the reported iMessage reply-context gap is real, yet the branch edits the removed `src/imessage` path instead of the active bundled plugin and resolves quote targets through the less reliable GUID field. It also lacks after-fix proof from a real macOS iMessage setup.

## What This Changes

The PR adds a read-only `chat.db` reply-context lookup and unit tests under the old `src/imessage` monitor path to populate `reply_to_*` before enqueue.

## Best Possible Solution

Land or replace this with an active `extensions/imessage/src/monitor/*` change that resolves reliable inline-reply origin metadata into `ReplyToId`, `ReplyToBody`, and `ReplyToSender` via the existing `dbPath` surface, fails closed, and covers failure modes.

## Reproduction Assessment

Yes, at source level. A GUID-only iMessage reply payload with no resolved `reply_to_text` reaches the active queue without enrichment, and `describeReplyContext` returns null because no reply body exists; I did not run a live macOS iMessage flow in this read-only review.

## Solution Assessment

No. The direction is useful, but this patch is not the best landing shape because it modifies a removed path and looks up `reply_to_guid` instead of using the more reliable `thread_originator_guid` contract.
