## Summary

Keep open. Current main still lacks the missing iMessage reply-context fallback, and this PR is useful source work, but it is not mergeable as-is because it targets the removed `src/imessage` tree and resolves quotes from the less reliable `reply_to_guid` field.

## What This Changes

The PR adds a chat.db lookup that populates missing inbound iMessage `reply_to_id`, `reply_to_text`, and `reply_to_sender` fields before dispatch, with unit tests for missing, old-schema, non-reply, and valid-reply databases.

## Best Possible Solution

Land a narrow replacement or port in `extensions/imessage/src/monitor` that enriches missing inbound reply context before enqueue/dispatch, prefers already-resolved upstream fields, uses `thread_originator_guid` as the reply-origin source, skips local DB reads for remote SSH-wrapper sessions, keeps chat.db access read-only and fail-closed, and adds focused regression coverage plus a changelog entry.

## Reproduction Assessment

Yes. Configure iMessage with `dbPath`, send an inline reply, and inspect the current active monitor path: when `imsg` omits resolved `reply_to_*`, `handleMessage` parses and enqueues the payload without enrichment, while the consumer only emits reply context when `reply_to_text` is already present.

## Solution Assessment

No. The direction is useful, but this patch is not the best landing shape because it modifies a removed path and resolves from `reply_to_guid`; the maintainable fix belongs in the active plugin and should use the reliable reply-origin field.
