## Summary

Keep open: the iMessage reply-context gap still exists on current main, but this PR is not mergeable as-is because it targets the removed `src/imessage` path, resolves through the less reliable `reply_to_guid`, and lacks real macOS behavior proof.

## What This Changes

The PR adds a read-only `chat.db` lookup intended to populate inbound iMessage `reply_to_id`, `reply_to_text`, and `reply_to_sender` when imsg omits resolved reply fields.

## Best Possible Solution

Land or replace this with an active `extensions/imessage/src/monitor/*` change that resolves reliable inline-reply origin metadata into `ReplyToId`, `ReplyToBody`, and `ReplyToSender` through the existing `dbPath` surface, fails closed, and includes failure-mode tests plus real macOS proof.

## Reproduction Assessment

Yes, at source level. A GUID-only iMessage reply payload with no resolved `reply_to_text` reaches the active queue without enrichment, and `describeReplyContext` returns null because no reply body exists; I did not run a live macOS iMessage flow in this read-only review.

## Solution Assessment

No. The direction is useful, but this branch is not the right landing shape because it modifies a removed path and resolves through `reply_to_guid` instead of the upstream-supported `thread_originator_guid` contract.
