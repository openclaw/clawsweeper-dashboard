## Summary

Keep open. The iMessage reply-context gap is real and the PR is useful source work, but it should not merge as-is because current main ships the iMessage monitor from `extensions/imessage`, while this branch changes the removed `src/imessage` path, and the lookup keys off `reply_to_guid` despite upstream and tracker evidence that `thread_originator_guid` is the reliable inline-reply target.

## What This Changes

The PR adds a synchronous read-only `chat.db` lookup under the old iMessage monitor path to populate missing `reply_to_*` fields before inbound dispatch, with unit tests for missing databases, non-replies, old schemas, deleted quotes, and valid replies.

## Best Possible Solution

Supersede or port this into `extensions/imessage/src/monitor/*`, enriching missing inbound reply context before dispatch while preserving existing context-visibility filtering. The fix should prefer already-resolved `reply_to_*` fields, use `thread_originator_guid` as the reliable origin when only GUID metadata is available, keep any `chat.db` access read-only and failure-tolerant, skip local database reads for remote SSH-wrapper sessions, and add regression coverage around valid replies, old/missing/locked database cases, and remote-runtime behavior.

## Reproduction Assessment

Yes. Configure iMessage with `dbPath`, send an inline reply, and observe that current main enqueues the parsed imsg payload without enriching `reply_to_*`; the static path confirms `describeReplyContext` returns null unless `reply_to_text` is already present.

## Solution Assessment

No, not as-is. The concept is a narrow maintainable direction, but the implementation should land in the active bundled plugin path and use the reliable reply-origin GUID rather than `reply_to_guid` alone.
