## Summary

No blocking findings. This PR is a narrow, well-evidenced fix for a queued Telegram topic delivery bug, but it should stay open for exact-head automerge and maintainer acceptance of the queued user-request cancellation contract.

## What This Changes

The PR stops queued `user_request` followups from inheriting source abort signals, preserves abort metadata for queued `room_event` followups, and adds regression tests plus a changelog entry.

## Best Possible Solution

Land the narrow queued-abort contract if maintainers agree accepted user requests are durable after queue admission, while keeping room-event cancellation and the new regression coverage.

## Reproduction Assessment

Yes, source-level. Current main stores `opts.abortSignal` on every queued followup and the queued runner falls back to it, and the PR reports before-fix regression failures against those exact expectations.

## Solution Assessment

Yes. The patch makes queued item metadata the cancellation source, preserving `room_event` cancellation while preventing accepted `user_request` turns from being canceled by later source-fence supersession.
