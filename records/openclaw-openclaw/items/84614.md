## Summary

Keep open: current main still has a distinct secondary simple-completion path that can bypass Copilot's provider wrapper, while docs promise Copilot headers on compaction and follow-up turns; the related closed reports do not own this exact cron/isolated failure.

## What This Changes

The issue reports that GitHub Copilot GPT models fail in isolated or cron sessions when secondary LLM calls resolve the provider transport.

## Best Possible Solution

Route Copilot secondary LLM calls through the same provider-owned auth, header, and payload-shaping path as primary agent calls, with focused regression coverage and live Copilot cron proof when credentials are available.

## Reproduction Assessment

Yes for a source-reproducible path: configure github-copilot/gpt-4o-mini or gpt-5-mini in an isolated cron job that triggers a secondary LLM call. I did not execute the live Copilot Enterprise flow, but current source inspection shows the distinct secondary completion path and missing Copilot-specific registration/wrapping coverage.

## Solution Assessment

Yes, the best fix is at the Copilot provider/simple-completion boundary rather than a cron-specific workaround. The repair should keep provider policy in the plugin/runtime seam and add regression coverage for GPT secondary calls plus a separate probe for the Haiku schema rejection if it still reproduces.
