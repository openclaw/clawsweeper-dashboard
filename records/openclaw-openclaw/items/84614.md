## Summary

Keep open: the report remains plausible against current main because the Copilot plugin is still a stream wrapper while the secondary/simple-completion path does not clearly force Copilot GPT calls through that provider-compatible stream boundary. The related closed reports either were the same report re-filed under the correct account or covered narrower replay/schema failures, not this isolated/cron secondary-call path.

## What This Changes

The issue reports that GitHub Copilot GPT models fail in isolated or cron sessions when secondary LLM calls cannot resolve the `github-copilot` provider transport.

## Best Possible Solution

Route Copilot GPT secondary calls through the same provider-compatible auth, headers, payload shaping, and stream wrapper semantics used by primary agent calls, with focused regression coverage for cron/isolated compaction and a separate Haiku schema probe if that still reproduces.

## Reproduction Assessment

Yes, there is a high-confidence reproduction shape: configure a GitHub Copilot GPT model in an isolated cron session and trigger a secondary LLM call such as compaction. I did not execute the live Copilot Enterprise path, but current source still shows separate primary and secondary transport paths that make the report credible.

## Solution Assessment

Yes, the best fix is at the Copilot provider/simple-completion boundary rather than a cron-only workaround. The repair should reuse the provider-compatible stream/auth/header/tool-shaping path and add regression coverage for isolated secondary calls.
