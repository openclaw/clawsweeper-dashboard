## Summary

Keep open. Current main still builds the CLI JSON result from raw runner output and then delivers those payloads unchanged; the available `before_message_write` finalization path is not threaded back into `agent --json`, and the related open PR only covers the pi embedded `--expect-final` surface.

## What This Changes

Make `openclaw agent --json` return the same post-`before_message_write` assistant text that is finalized for the session transcript.

## Best Possible Solution

Thread one finalized assistant message through CLI runner persistence and delivery so `payloads` and `meta.finalAssistantVisibleText` use the post-hook value, with regression coverage for rewrite and block cases.

## Reproduction Assessment

Yes, from source inspection. Register a synchronous `before_message_write` hook that rewrites or blocks the final assistant message, run `openclaw agent --json` through the CLI runner path, and compare `payloads[].text`/`meta.finalAssistantVisibleText` with the finalized transcript value.

## Solution Assessment

Yes. The narrow maintainable fix is to share one finalized assistant value across CLI transcript persistence and JSON delivery; the related pi embedded runner PR is useful adjacent work but does not solve this CLI surface.
