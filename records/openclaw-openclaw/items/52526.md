## Summary

Keep open. Current main still persists CLI assistant text through the `before_message_write` hook path but builds `agent --json` payloads from the original CLI output, and the linked fix PR is open/conflicting against removed runner files.

## What This Changes

The issue reports that `agent --json` can return pre-hook assistant text while the persisted transcript contains the `before_message_write`-finalized assistant message.

## Best Possible Solution

Retarget the fix to current `src/agents/cli-runner.ts` so CLI JSON consumes the finalized assistant message accepted by the transcript write-hook path, with an explicit blocked-hook policy.

## Reproduction Assessment

Yes at source level: current main persists CLI assistant text through `beforeMessageWrite`, then builds returned JSON payloads from the original CLI output text. I did not run a live plugin/CLI reproduction because this review is read-only.

## Solution Assessment

Unclear. Returning finalized text is the right end state for hook rewrites, but maintainers should first define safe JSON semantics for hook-blocked replies so raw text is not leaked after a block.
