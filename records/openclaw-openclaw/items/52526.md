## Label Justifications

- `P2`: This is a valid result-surface bug with limited blast radius but real downstream automation and plugin-policy impact.
- `impact:session-state`: The CLI JSON result and persisted session transcript can disagree about the final assistant message for the same run.
- `impact:security`: Plugin-enforced warning or redaction text can be present in the transcript while downstream JSON automation still receives pre-hook assistant text.

## Summary

Keep open: current main still assembles `agent --json` payloads from the raw CLI output path while `before_message_write` only affects transcript persistence, and existing tests codify that blocked persistence can still return the raw payload. The report is source-reproducible, but changing this needs maintainer product/security direction because it changes whether a persistence hook also owns CLI result content.

## What This Changes

Make `agent --json` return the same post-`before_message_write` assistant message that is persisted in the session transcript.

## Best Possible Solution

Define the CLI JSON contract for mutated and blocked `before_message_write` outcomes, then update `src/agents/cli-runner.ts` and focused regression tests so payloads, metadata, and transcript persistence agree without broadening unrelated plugin hook semantics.

## Reproduction Assessment

Yes at source level: current `main` still builds the JSON result from CLI output text while the write hook only mutates the transcript append path. I did not run a live CLI/plugin scenario in this read-only review.

## Solution Assessment

No, not yet as an implementation path. The desired behavior is plausible, but maintainers first need to decide whether `before_message_write` is persistence-only or must also control `agent --json` result content, especially for blocked writes.
