## Summary

Keep open. Current main still builds the CLI JSON result from the raw CLI runner output and then serializes those supplied payloads, while the `before_message_write` hook remains wired to the transcript write path rather than to the `agent --json` result surface.

## What This Changes

The issue asks for `openclaw agent --json` to return the same post-`before_message_write` finalized assistant text that is persisted in the session transcript.

## Best Possible Solution

Thread or retrieve one finalized CLI assistant message after `before_message_write` semantics are applied, then use that same value for JSON payloads, result metadata, and transcript persistence with focused regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: register a synchronous `before_message_write` hook that rewrites or blocks the final assistant message, run `openclaw agent --json`, and compare `payloads[].text` / `meta.finalAssistantVisibleText` with the finalized transcript behavior.

## Solution Assessment

Yes. The narrow maintainable fix is to compute or read the finalized assistant value once in the CLI path and reuse it across JSON result and persistence; the related pi-runner PR does not cover this CLI surface.
