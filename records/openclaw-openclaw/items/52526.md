## Label Justifications

- `P2`: This is a source-backed CLI result-surface bug with safety/audit implications, but it is limited to a specific runner output path rather than a core outage.
- `impact:security`: Plugins can use the hook for safety rewrites, and the JSON surface can expose pre-hook text to downstream automation.
- `impact:session-state`: The reported behavior creates divergence between the persisted transcript and the JSON result metadata/payloads for the same assistant turn.

## Summary

Keep open. Current main and v2026.5.18 still build CLI JSON payloads and `meta.finalAssistantVisibleText` from raw CLI runner output while `before_message_write` finalizes only the transcript append path, so the reported mismatch remains source-backed; the related open pull request at https://github.com/openclaw/openclaw/pull/56031 targets the pi embedded runner and does not close this CLI runner surface.

## What This Changes

The issue asks for `agent --json` to return the finalized assistant text after `before_message_write` hooks instead of raw pre-hook CLI runner output.

## Best Possible Solution

Finalize the assistant message once for the CLI runner result path, then reuse that value for JSON payloads, `meta.finalAssistantVisibleText`, and transcript mirroring with focused regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: register a synchronous `before_message_write` hook that rewrites the final assistant message, run `openclaw agent --json`, and compare `payloads[].text` plus `meta.finalAssistantVisibleText` with the finalized transcript.

## Solution Assessment

Yes. The narrow maintainable fix is to thread or retrieve the finalized assistant value in the CLI runner path and use it consistently, rather than changing hook policy or adding a new option.
