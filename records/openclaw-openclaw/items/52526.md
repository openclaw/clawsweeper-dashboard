## Summary

Keep open. Current main still has the source-level mismatch: CLI assistant text is persisted through the `before_message_write` hook path, but `agent --json` payloads are assembled from the original CLI output text, and the linked PR is still open/conflicting against removed runner paths.

## What This Changes

The issue requests that `agent --json` return the finalized assistant message after `before_message_write`, matching the persisted session transcript.

## Best Possible Solution

Make CLI JSON output consume the finalized assistant message accepted by the transcript write-hook path, with an explicit policy for blocked hook results.

## Reproduction Assessment

Yes at source level: current main persists CLI assistant text through `beforeMessageWrite`, then builds returned JSON payloads from the original CLI output text. I did not run a live plugin/CLI reproduction because this review is read-only.

## Solution Assessment

Unclear. A targeted CLI result fix is plausible, but maintainers should first decide the safe JSON semantics for hook-blocked replies so raw text is not leaked after a block.
