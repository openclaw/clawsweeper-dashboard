## Label Justifications

- `P2`: This is a source-backed CLI result-surface bug with safety and audit implications, but it is limited to a specific runner output path rather than a core outage.
- `impact:security`: Plugins can use `before_message_write` for safety rewrites, and the JSON surface can expose pre-hook text to downstream automation.
- `impact:session-state`: The report creates divergence between the persisted transcript and returned JSON payloads/metadata for the same assistant turn.

## Summary

Keep open. Current main and v2026.5.19 still build the CLI JSON payload and `meta.finalAssistantVisibleText` from raw CLI runner output, while `before_message_write` finalizes only the transcript append path; the related open PR targets the pi embedded runner, not this CLI runner surface.

## What This Changes

The issue reports that `agent --json` can return pre-hook assistant text instead of the `before_message_write`-finalized assistant content persisted to the session transcript.

## Best Possible Solution

Finalize or retrieve the assistant message once for the CLI runner result path, then reuse that finalized value for JSON payloads, `meta.finalAssistantVisibleText`, and transcript mirroring with focused regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: register a synchronous `before_message_write` hook that rewrites the final assistant message, run `openclaw agent --json`, and compare `payloads[].text` plus `meta.finalAssistantVisibleText` with the finalized transcript.

## Solution Assessment

Yes. The narrow maintainable fix is to thread or retrieve the finalized assistant value in the CLI runner path and use it consistently, rather than changing hook policy or adding a new option.
