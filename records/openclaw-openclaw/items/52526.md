## Summary

Keep open. Current main still builds CLI `agent --json` payloads from the raw CLI runner output and delivery serializes those supplied payloads without re-reading or sharing a post-`before_message_write` assistant value; the related #56031 PR is open and scoped to the pi embedded runner `--expect-final` path, not this CLI surface.

## What This Changes

The issue requests that `openclaw agent --json` return the same finalized assistant text that `before_message_write` hooks persist to the session transcript.

## Best Possible Solution

Thread one finalized CLI assistant message through the runner result so `payloads`, `meta.finalAssistantVisibleText`, and transcript persistence all reflect `before_message_write` replacement or block semantics exactly once.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: register a synchronous `before_message_write` hook that rewrites or blocks the final assistant message, run the CLI runner through `openclaw agent --json`, then compare the JSON `payloads[].text`/`meta.finalAssistantVisibleText` with the finalized transcript behavior. I did not execute the repro because this review was read-only and `pnpm` is unavailable in the checkout.

## Solution Assessment

Yes. The narrow maintainable fix is to compute or retrieve the finalized assistant value once in the CLI path and use that same value for result JSON and persistence; #56031 is useful related work but only addresses the pi embedded runner `--expect-final` surface.
