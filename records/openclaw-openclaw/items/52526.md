## Summary

Keep open: current main and the latest release still build the CLI JSON result from raw CLI output while transcript persistence runs `before_message_write`, so the reported result-surface mismatch remains. The linked PR is open and scoped to an older sibling Pi runner path, not the current `agent --json` CLI path.

## What This Changes

The issue reports that `agent --json` can return pre-hook assistant text instead of the finalized assistant message produced by `before_message_write` and persisted to the transcript.

## Best Possible Solution

Make the CLI JSON result consume the same finalized assistant-write outcome as transcript persistence, including explicit behavior for rewritten and blocked messages, then add focused regression coverage and align any shared runner invariant with related hook-surface work.

## Reproduction Assessment

Yes, at source level: current main persists the assistant message through `before_message_write` but then returns `payloads` from raw CLI output text. I did not run a live CLI/plugin repro in this read-only review.

## Solution Assessment

No, current behavior is not the best solution because it lets JSON output diverge from the finalized transcript; the safer path is to thread or read the finalized assistant write outcome before constructing `payloads`.
