## Summary

Keep open. Current main still returns CLI JSON payload text from the original CLI output path after separately persisting the assistant through the `before_message_write` hook path, and the linked fix PR is stale/conflicting against removed runner files.

## What This Changes

The issue asks for `agent --json` payloads to match the finalized assistant message after `before_message_write` hooks mutate or block transcript persistence.

## Best Possible Solution

Make the current CLI JSON result path consume the finalized assistant message accepted by the transcript write-hook path, with explicit non-leaking behavior for hook-blocked replies.

## Reproduction Assessment

Yes at source level. Current main persists CLI assistant text through `beforeMessageWrite`, then builds returned JSON payloads from the original CLI output text; I did not run a live plugin/CLI reproduction in this read-only review.

## Solution Assessment

Unclear. A targeted current-main CLI result fix is the right boundary, but maintainers should first decide blocked-hook JSON semantics so raw text is not leaked after a hook block.
