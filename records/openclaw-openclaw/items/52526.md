## Summary

Keep open: current main still has a source-proven mismatch where CLI JSON payloads are built from raw CLI output after transcript persistence runs through `before_message_write`, and the linked fix PR is still open, conflicting, and aimed at the older Pi runner path rather than the current `cli-runner` surface.

## What This Changes

The issue asks for `agent --json` to return the same finalized assistant message produced after `before_message_write` hooks as the message persisted to the session transcript.

## Best Possible Solution

Define the CLI JSON finalization contract, then update the current CLI and embedded result paths plus regression tests so automation receives the same finalized assistant content that maintainers intend plugins to enforce.

## Reproduction Assessment

Yes, at source level: current `runPreparedCliAgent` persists through `before_message_write` but returns JSON payloads from raw `output.text`, and current tests show a blocked hook can leave the transcript empty while returning the raw payload. I did not run a live CLI/plugin repro in this read-only review.

## Solution Assessment

Unclear as currently proposed: the linked PR was aimed at the older Pi runner path and does not cover the current CLI result builder or the test that codifies this behavior. The safer fix is a maintainer-approved contract update across the current CLI/embedded payload path and tests.
