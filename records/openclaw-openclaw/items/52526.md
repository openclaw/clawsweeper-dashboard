## Summary

Keep open. Current main still builds CLI `payloads` and `meta.finalAssistantVisibleText` from raw CLI output, and JSON delivery serializes those supplied payloads without using a `before_message_write`-finalized assistant value; the related #56031 is open and scoped to the pi embedded runner, not this CLI surface.

## What This Changes

The issue asks for `agent --json` to return the same `before_message_write`-finalized assistant text that is persisted to the session transcript.

## Best Possible Solution

Thread one finalized CLI assistant message through transcript persistence, `payloads`, delivery, and `meta.finalAssistantVisibleText`, preserving existing synchronous replace and block semantics from `before_message_write`.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: register a synchronous `before_message_write` hook that rewrites or blocks the final assistant message, run `openclaw agent --json` on a CLI-backed runtime, then compare the JSON payload/meta text against the finalized assistant transcript behavior.

## Solution Assessment

Yes. The narrow maintainable fix is to reuse or thread the existing hook-finalized assistant value through the CLI result path; #56031 is useful related pi-runner work but does not solve this CLI runner bug.
