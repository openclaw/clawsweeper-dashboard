## Label Justifications

- `P2`: This is a source-backed CLI result-surface bug with safety and audit implications, but it is limited to one runner output path rather than a core outage.
- `impact:security`: Plugins can use the hook for safety rewrites, and the JSON surface can expose pre-hook text to downstream automation.
- `impact:session-state`: The same assistant turn can diverge between persisted transcript state and the JSON result payloads and metadata.

## Summary

Keep open. Current main and the latest release v2026.5.19 still build CLI JSON payloads and `meta.finalAssistantVisibleText` from raw CLI runner output, while `before_message_write` can replace or block messages before transcript persistence; the related open pull request at https://github.com/openclaw/openclaw/pull/56031 targets the pi embedded runner and does not resolve this CLI runner surface.

## What This Changes

Make `agent --json` return the same finalized assistant text that `before_message_write` hooks leave in the persisted transcript.

## Best Possible Solution

Finalize the assistant message once for the CLI runner result path, then reuse that value for JSON payloads, `meta.finalAssistantVisibleText`, and transcript mirroring with focused regression coverage for rewritten and blocked `before_message_write` outcomes.

## Reproduction Assessment

Yes, at source level. Register a synchronous `before_message_write` hook that rewrites the final assistant message, run `openclaw agent --json`, and compare `payloads[].text` plus `meta.finalAssistantVisibleText` with the finalized transcript.

## Solution Assessment

Yes. The narrow maintainable fix is to thread or retrieve the finalized assistant value in the CLI runner result path and use it consistently, rather than changing hook policy or adding a new option.
