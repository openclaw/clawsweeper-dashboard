## Summary

Keep open. Current main still has a source-level mismatch between the hook-mutated transcript write path and the `agent --json` payload path, and the paired open PR is conflicting and targets removed pi-runner files rather than a landed current-main fix.

## What This Changes

This issue requests that `agent --json` return the same finalized assistant message that `before_message_write` hooks allow into the persisted session transcript.

## Best Possible Solution

Make the CLI JSON result consume the finalized assistant message accepted by the write-hook path, with explicit behavior for hook rewrites and hook blocks before implementation proceeds.

## Reproduction Assessment

Yes at source level: current main persists CLI assistant text through `appendExactAssistantMessageToSessionTranscript` with `beforeMessageWrite`, then builds `payloads` from `output.text`. I did not run a live plugin/CLI repro because this review is read-only.

## Solution Assessment

Unclear. A targeted CLI result fix is plausible, but maintainers should first decide whether blocked hooks produce an empty/error JSON result, omit payloads, or preserve a separate diagnostic without leaking raw text.
