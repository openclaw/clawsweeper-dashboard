## Label Justifications

- `impact:session-state`: The issue is about divergence between persisted session transcript state and the JSON result returned to callers.
- `impact:security`: Plugins that enforce safety or warning text through hooks can have that finalized text omitted from downstream CLI JSON automation.

## Summary

Keep open: current main still persists CLI assistant output through `before_message_write` while `agent --json` payloads are assembled from raw CLI output, so the JSON result can diverge from the finalized transcript. The related PR is open but conflicting and targets an older/narrower runner path, so it is not a safe canonical resolution yet.

## What This Changes

Make `agent --json` return the finalized assistant message after `before_message_write` hooks rather than the pre-hook CLI output.

## Best Possible Solution

Define and implement CLI result finalization so `payloads` and `meta.finalAssistantVisibleText` reflect the same post-hook assistant message persisted to the session, including explicit block semantics and focused regression coverage.

## Reproduction Assessment

Yes at source level. Current main persists CLI assistant output through `before_message_write` but builds `agent --json` payloads from `output.text`; I did not run a live plugin repro in this read-only review.

## Solution Assessment

No, current main is not yet the best solution. The best fix should land in the current CLI/transcript result seam, while the open related PR needs maintainer review because it is conflicting and targets an older/narrower runner path.
