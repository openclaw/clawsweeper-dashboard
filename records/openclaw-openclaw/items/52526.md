## Summary

Keep open. Current main still does not enforce the requested invariant: CLI JSON payloads are built from raw CLI output and later delivered unchanged, while message-write finalization is a separate persistence concern. The related #56031 work is scoped to the pi embedded runner `--expect-final` path, so it does not close this CLI `agent --json` surface.

## What This Changes

The issue requests that `agent --json` return the same finalized assistant text produced after `before_message_write` hooks as the session transcript persists.

## Best Possible Solution

Implement a narrow CLI result-path fix that computes the final assistant message after the CLI model output is available, applies the synchronous `before_message_write` semantics once, and reuses that same finalized result for `payloads`, `meta.finalAssistantVisibleText`, and CLI transcript persistence. Add regression coverage for replaced and blocked assistant messages so JSON output and the stored transcript cannot diverge.

## Reproduction Assessment

Yes. The issue provides concrete reproduction steps and current main has a source-level path to reproduce the stale JSON surface: register a `before_message_write` hook that rewrites or blocks the final assistant, run `openclaw agent --json`, then compare returned `payloads[].text` with the intended finalized assistant/transcript result.

## Solution Assessment

Yes, the requested direction is the narrow maintainable fix, but the safest implementation is to thread or share a single finalized assistant value rather than re-reading or independently reapplying hooks in multiple places. #56031 is not the best solution for this issue because it targets the pi embedded runner only.
