## Summary

Keep open. Current main still builds the CLI JSON result from raw CLI runner output while `before_message_write` is applied on the transcript append path, so the reported result-surface mismatch remains source-backed.

## What This Changes

The issue asks for `agent --json` to return the post-`before_message_write` finalized assistant text that matches the persisted transcript.

## Best Possible Solution

Finalize the assistant message once for the CLI runner result path, then reuse that value for JSON payloads, result metadata, and transcript persistence with focused regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: register a synchronous `before_message_write` hook that rewrites or blocks the final assistant message, run `openclaw agent --json`, and compare `payloads[].text` / `meta.finalAssistantVisibleText` with the finalized transcript behavior.

## Solution Assessment

Yes. The narrow maintainable fix is to thread or retrieve the finalized assistant value in the CLI path and use it consistently, rather than adding a new config option or changing plugin hook policy.
