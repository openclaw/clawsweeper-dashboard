## Label Justifications

- `P2`: This is a confirmed result-surface correctness bug with security implications, but it is scoped to CLI JSON consumers rather than a core runtime outage.
- `impact:session-state`: The issue is specifically about divergence between persisted session transcript state and the JSON result returned to callers.
- `impact:security`: Plugins using write hooks to enforce safety or warning text can have that finalized text omitted from the CLI JSON response.

## Summary

Keep open: current main and v2026.6.8 still build the `agent --json` payload from raw CLI output while the transcript append path applies `before_message_write`, so the reported result/transcript mismatch remains. The fix needs maintainer direction for rewritten versus blocked message semantics before automation should change the result surface.

## What This Changes

The issue asks for `agent --json` payloads to return the finalized post-`before_message_write` assistant message instead of the pre-hook CLI text.

## Best Possible Solution

Define CLI JSON semantics for rewritten and blocked write-hook results, then make `runPreparedCliAgent` return the same finalized assistant text that transcript persistence committed, with focused regression coverage.

## Reproduction Assessment

Yes at source level. Current main still applies `before_message_write` on CLI transcript persistence while `agent --json` payloads are assembled from raw `output.text`; I did not run a live plugin repro in this read-only review.

## Solution Assessment

Unclear as an implementation choice. Reading or threading the post-write assistant message is the maintainable direction for rewritten messages, but maintainers should first decide what `agent --json` returns when the hook blocks persistence entirely.
