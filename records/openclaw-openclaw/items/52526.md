## Label Justifications

- `P2`: This is a source-backed CLI result-surface bug with safety and audit implications, but it is limited to a specific runner output path rather than a core outage.
- `impact:session-state`: The reported behavior creates divergence between the persisted session transcript and returned JSON payloads/metadata for the same assistant turn.
- `impact:security`: Plugins can use `before_message_write` for safety rewrites, and the JSON surface can expose pre-hook text to downstream automation.

## Summary

Keep open: current main and latest release still build the CLI JSON payload and `meta.finalAssistantVisibleText` from raw CLI runner output, while `before_message_write` mutations run on the transcript append path; the related open PR only targets the pi embedded runner `--expect-final` surface, not CLI `agent --json`.

## What This Changes

The issue asks for `agent --json` to return the same post-`before_message_write` assistant text that is persisted to the session transcript.

## Best Possible Solution

Finalize or retrieve the post-hook assistant message once in the CLI runner result path, then reuse that finalized value for JSON payloads, `meta.finalAssistantVisibleText`, and delivery normalization with focused regression coverage.

## Reproduction Assessment

Yes, as a source-level reproduction. Register a synchronous `before_message_write` hook that rewrites the final assistant message, run `openclaw agent --json`, and compare `payloads[].text` plus `meta.finalAssistantVisibleText` with the finalized transcript.

## Solution Assessment

Yes. The narrow maintainable fix is to thread or retrieve the finalized assistant value in the CLI runner result path, rather than changing hook policy or adding a new option.
