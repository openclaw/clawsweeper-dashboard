## Summary

Keep open: current main intentionally uses Telegram message-edit previews and has no Telegram sendMessageDraft transport/config, while the discussion has fresh contributor and maintainer follow-up. This is a Telegram plugin feature/product decision, not cleanup closure.

## What This Changes

The issue requests a Telegram native sendMessageDraft streaming transport and configuration path for typewriter-style private-chat previews.

## Best Possible Solution

Have Telegram channel owners decide whether to reintroduce sendMessageDraft as an opt-in plugin-owned transport, preserving message-edit fallback, finalization/dedupe guarantees, docs, and regression coverage.

## Reproduction Assessment

Not applicable: this is a feature request, not a broken documented behavior. Source, docs, and tests do verify the current-main baseline that Telegram previews use sendMessage plus editMessageText and lack a sendMessageDraft transport.

## Solution Assessment

Unclear: sendMessageDraft is technically plausible, but it reopens a recently removed Telegram transport and needs owner judgment on UX, fallback scope, finalization semantics, and config shape.
