- `impact:security`: The issue is about a missing authorization check across a Discord guild-admin security boundary.

## Summary

Keep open. Current main and the latest release still show the Discord guild-admin requester-authorization gap, and the linked closing PR remains open and conflicting rather than merged.

## What This Changes

The issue asks OpenClaw to require requester identity and Discord permission checks before bot-privileged Discord guild-admin message actions mutate guild state.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/68716 with a narrow Discord plugin/runtime fix that requires trusted requester identity for Discord-origin guild-admin actions, checks matching Discord permissions before mutation, and preserves trusted owner/manual execution paths explicitly.

## Reproduction Assessment

Yes, by source inspection. On current main, a Discord guild-admin action such as channel-delete can flow through the dispatcher, Discord adapter, guild-admin handler, and guild runtime without a requester permission check before deleteChannelDiscord.

## Solution Assessment

Yes for the issue direction, but the final patch needs maintainer/security judgment on the owner-vs-requester trust boundary. The fix should live in the Discord plugin/runtime authorization path rather than as a broad core policy change.
