## Summary

Keep open. Current main still has the reported Discord guild-admin requester authorization gap, and open PR #68716 uses closing syntax for this issue but has not merged, so maintainer cleanup should leave the issue open until that fix or a replacement lands.

## What This Changes

The issue reports that Discord guild-admin message actions can execute bot-privileged guild mutations without verifying the requester's Discord guild permissions.

## Best Possible Solution

Land a narrow Discord-owned authorization fix that marks guild-admin mutation actions as trusted-requester-only, forwards the trusted Discord requester id into guild-admin runtime calls, verifies the matching Discord permission bit before bot-credentialed mutations, and adds regression tests for channel, role, event, emoji/sticker, and direct runtime permission paths.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists on current main: the shared dispatcher depends on a plugin trusted-requester hook, Discord does not define that hook for guild-admin actions, the guild-admin handler does not forward requesterSenderId, and runtime.guild.ts performs privileged mutations after feature gates only.

## Solution Assessment

Yes, the requested direction is the narrowest maintainable fix. The authorization belongs in the Discord plugin boundary and runtime permission checks, analogous to moderation, rather than hard-coding Discord-specific policy in shared core dispatch.
