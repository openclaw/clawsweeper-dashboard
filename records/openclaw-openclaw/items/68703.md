## Summary

Keep open. Current `main` still has the Discord guild-admin requester-authorization gap, and the open closing PR is the right tracking path until a corrected fix lands.

## What This Changes

The issue requests Discord guild-admin mutation actions to require trusted requester identity and matching Discord permissions before bot-credentialed mutations run.

## Best Possible Solution

Land or replace the open Discord plugin fix with a narrow patch that marks privileged guild-admin actions as trusted-requester-only, forwards the Discord requester id, checks the matching Discord permission before mutation, and preserves trusted owner/manual flows with regression coverage.

## Reproduction Assessment

Yes. Source inspection on current `main` gives a high-confidence path: dispatch a Discord guild-admin action such as `channel-delete` without requester authorization and it reaches the bot-credentialed guild runtime with only feature-gate checks.

## Solution Assessment

Yes for the issue direction. The maintainable fix belongs in the Discord plugin adapter and guild runtime because core already has the trusted-requester seam and the Discord moderation runtime already demonstrates the permission-check pattern.
