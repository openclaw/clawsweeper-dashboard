## Summary

Keep open. Current main and the latest release still show the Discord guild-admin requester-authorization gap, and the linked closing PR remains open/conflicting with prior trust-boundary review concerns.

## What This Changes

The issue reports that Discord guild-admin message actions can execute bot-privileged mutations without verifying the requesting guild member's Discord permissions.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/68716 with a narrow Discord plugin fix that requires trusted requester identity for Discord-origin guild-admin actions, checks matching Discord permissions before mutation, and preserves trusted owner/manual execution paths.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a Discord guild-admin action such as channel-delete can flow from the message action dispatcher into bot-credentialed guild mutation code without requester authorization.

## Solution Assessment

Yes for the issue direction, but the exact fix needs maintainer/security judgment on the owner-vs-requester trust boundary. The safer path is to correct the Discord plugin/runtime authorization layer rather than add a broad core policy change.
