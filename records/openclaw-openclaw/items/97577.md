## Summary

Keep open: the linked bug is valid, but this PR still only counts adjacent identical result hashes, so stable cross-tool no-progress loops with different per-tool outputs can evade the intended session-global breaker. The PR also still needs contributor-owned real behavior proof beyond unit/format output.

## What This Changes

This PR adds a session-wide result-hash streak helper for `globalCircuitBreakerThreshold`, rewires the global breaker to use it, and adds detector tests for cross-tool same-result blocking.

## Best Possible Solution

Fix the detector to aggregate stable no-progress outcomes per tool/args pattern across the run-scoped session tail, keep changing outcomes as progress, and add detector plus before-tool-call/runtime proof.

## Reproduction Assessment

Yes, source-level. Current main uses a per-tool/args no-progress streak for the global breaker, and the PR helper still breaks on different adjacent result hashes; I did not run a live Gateway replay in this read-only review.

## Solution Assessment

No. The detector is the right layer, but the best fix needs stable per-pattern aggregation across the session tail instead of requiring every adjacent call to share one result hash.
