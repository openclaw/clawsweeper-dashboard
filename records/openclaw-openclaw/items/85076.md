## Summary

Keep open: current main still pre-creates only the Docker state directory, while the bundled Compose file and Docker docs expose the workspace and auth-profile secret directories as separate mount targets. The reporter's shipped-image output matches the source gap, and no open PR was found that references this issue, so this is a narrow valid Docker first-run bug rather than cleanup-close material.

## What This Changes

The issue requests node-owned precreation of the documented Docker workspace and auth-profile secret mount targets so fresh named-volume deployments do not fail with EACCES.

## Best Possible Solution

Pre-create every documented Docker named-volume target in the runtime image before `USER node`, keep bind-mount behavior unchanged, and cover the invariant with Dockerfile/Compose structural tests plus a fresh named-volume smoke.

## Reproduction Assessment

Yes. The release report gives a concrete fresh named-volume Docker command and current source still has the same missing pre-created paths, but I did not execute a current-main image smoke in this read-only review.

## Solution Assessment

Yes. Extending the existing Dockerfile `install -d`/`stat` pattern to the two remaining documented mount targets is the narrowest maintainable fix and avoids runtime chown migrations or bind-mount behavior changes.
