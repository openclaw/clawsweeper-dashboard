## Label Justifications

- `P2`: The bug blocks first-run Docker named-volume deployments, but the blast radius is limited to that install layout and has a manual chown workaround.
- `impact:session-state`: The root-owned workspace mount prevents bootstrap workspace state such as `AGENTS.md` from being created.
- `impact:auth-provider`: One affected mount is the auth-profile secret directory used for OAuth-backed auth profile key material.

## Summary

Keep open: current main still only pre-creates `/home/node/.openclaw`, while Compose and docs expose `/home/node/.openclaw/workspace` and `/home/node/.config/openclaw` as separate Docker mount targets. Two open PRs now reference and attempt to fix this issue, so the issue should stay open until one lands.

## What This Changes

Pre-create the Docker image workspace and auth-profile secret mount targets with `node:node` ownership before `USER node` so fresh named volumes are writable on first run.

## Best Possible Solution

Review and land one of the linked Dockerfile/test PRs, then close this issue as implemented once the fix is on main and included in a published image.

## Reproduction Assessment

Yes. The issue provides a concrete fresh named-volume Docker command and current source still shows the missing pre-created mount points, though I did not run a current-main Docker image smoke in this read-only review.

## Solution Assessment

Yes. Extending the existing `install -d`/`stat` Dockerfile pattern to every documented mount target before `USER node` is the narrowest maintainable fix and matches Docker's empty-volume copy-up behavior.
