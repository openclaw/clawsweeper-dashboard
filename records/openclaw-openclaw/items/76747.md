## Summary

Keep open: the latest head adds useful Mantis Discord coverage, but source inspection still shows P2 oracle and evidence defects that can let the live scenario pass without proving the intended lifecycle.

## What This Changes

The PR adds an opt-in Discord Mantis status-reaction live QA scenario, Discord API helper reuse, docs/package-boundary wiring, and a manual before/after GitHub workflow with artifacts.

## Best Possible Solution

Harden the scenario so pass requires ordered, SUT-authored, stable lifecycle evidence and required PNG output, then run the default-branch Mantis workflow for baseline/candidate proof.

## Reproduction Assessment

Yes. Source inspection of the latest PR head shows membership-only sequence matching, aggregate reaction snapshots, warning-only screenshot failures, and a config path that can make the terminal reaction transient.

## Solution Assessment

No. The Mantis workflow direction is useful, but this patch is not ready until the oracle proves ordered SUT-authored lifecycle evidence and fails when required visual evidence is missing.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Hold the terminal reaction long enough to observe it:** `extensions/qa-lab/src/live-transports/discord/discord-live.runtime.ts:354-371`
  - body: The scenario expects the poller to observe 👀 -> 🤔 -> 👍, but the injected config leaves messages.removeAckAfterReply at its default. Current Discord processing calls setDone() and then restores the initial reaction when that option is false, so 👍 can disappear before the 250 ms REST poll catches it.
  - confidence: 0.86
- **[P2] Require ordered reaction evidence before passing:** `extensions/qa-lab/src/live-transports/discord/discord-live.runtime.ts:1092-1098`
  - body: The pass condition only checks that every expected emoji appeared at least once. A timeline such as 👀 -> 👍 -> 🤔 leaves missing empty and is marked pass even though the documented lifecycle order is wrong.
  - confidence: 0.9
- **[P2] Filter reaction evidence to the SUT bot:** `extensions/qa-lab/src/live-transports/discord/discord-live.runtime.ts:493-497`
  - body: The REST snapshot records aggregate emoji state, and the sequence builder accepts any matching emoji on the message. In a shared QA channel, another actor can add 🤔 or 👍 and satisfy the oracle even if the SUT never emitted that lifecycle reaction.
  - confidence: 0.82
- **[P2] Fail when PNG evidence cannot be captured:** `extensions/qa-lab/src/live-transports/discord/discord-live.runtime.ts:609-610`
  - body: writeDiscordStatusReactionEvidence catches Playwright launch or screenshot errors and returns only screenshotWarning, while the scenario can still pass from emoji presence alone. The workflow promises PNG before/after evidence, so missing screenshot capture should fail the scenario or comparison.
  - confidence: 0.8
