Full review comments:

- **[P2] Hold the terminal reaction long enough to observe it:** `extensions/qa-lab/src/live-transports/discord/discord-live.runtime.ts:354-371`
  - body: This scenario expects the REST poller to observe `👀 -> 🤔 -> 👍`, but the injected config leaves `messages.removeAckAfterReply` at its default. Current Discord processing calls `setDone()` and then restores the initial reaction when that option is false, so `👍` can disappear before the 250 ms poll catches it.
  - confidence: 0.84
- **[P2] Require ordered reaction evidence before passing:** `extensions/qa-lab/src/live-transports/discord/discord-live.runtime.ts:1092-1098`
  - body: The pass condition only checks that each expected emoji appeared at least once. A timeline such as `👀 -> 👍 -> 🤔` leaves `missing` empty and is marked pass even though the documented lifecycle order is wrong.
  - confidence: 0.88
- **[P2] Filter reaction evidence to the SUT bot:** `extensions/qa-lab/src/live-transports/discord/discord-live.runtime.ts:494-497`
  - body: The REST snapshot records only aggregate emoji/count/current-token state, and the sequence builder accepts any matching emoji on the message. In a shared QA channel, another actor can add `🤔` or `👍` and satisfy the oracle even if the SUT never emitted that lifecycle reaction.
  - confidence: 0.78
- **[P2] Fail when PNG evidence cannot be captured:** `extensions/qa-lab/src/live-transports/discord/discord-live.runtime.ts:609-610`
  - body: `writeDiscordStatusReactionEvidence` catches Playwright launch or screenshot errors and returns only `screenshotWarning`, while the scenario can still pass from emoji presence alone. The new workflow promises PNG before/after evidence, so missing screenshot capture should fail the scenario or comparison.
  - confidence: 0.8
