## Summary

Keep open: current main and the latest release still expose `memorySearch.extraPaths` as an inclusion-only surface, builtin indexing recursively scans configured roots without a public exclusion contract, and diagnostics only cover missing/inaccessible extra paths rather than workspace-intent drift. Related open items cover narrower slices, but none fully replaces this broader docs/config/diagnostics product decision.

## What This Changes

The issue requests memory-search docs plus config/runtime support for path hygiene, benchmark-file exclusion, and stronger diagnostics around `extraPaths` drift.

## Best Possible Solution

Define a backend-consistent memory indexing exclusion and workspace-intent diagnostics contract, then apply it across config types/schema/help, builtin scanning/read/sync/status, QMD collection behavior, docs, and focused regression coverage.

## Reproduction Assessment

No high-confidence end-to-end reproduction was established in this read-only review. Source inspection does confirm the current missing exclusion, docs, and workspace-intent warning surfaces that make the reported hygiene problem plausible.

## Solution Assessment

Unclear until maintainers choose the public contract. Exclusion/glob support and docs are plausible, but the maintainable fix needs agreed key names, matcher semantics, defaults, and builtin/QMD parity before implementation.
