Overall confidence: 0.88

Full review comments:

- **[P2] Require a session key before marking cards running:** `ui/src/ui/controllers/workboard.ts:752-761`
  - body: `sessions.create` can return without a usable `key`, and the existing Sessions controller treats that as an error. This path still updates the Workboard card to `running` without a linked session and then returns `null`, leaving a running card with no navigable handoff target.
  - confidence: 0.9
- **[P2] Treat running sessions as live in card actions:** `ui/src/ui/views/workboard.ts:557`
  - body: `renderCard` only checks `hasActiveRun === true`, but current session lifecycle helpers treat `status: "running"` as active too. Rows that report running status without `hasActiveRun` will hide the live badge and Stop action even though the session is active.
  - confidence: 0.87
- **[P2] Offer restart when a linked session is missing:** `ui/src/ui/views/workboard.ts:619-646`
  - body: The action branch keys only on `card.sessionKey`, so a card whose linked session was deleted or expired still shows only Open session for a dead key. Since lifecycle already marks that state as missing, the UI should allow starting or relinking the card instead of trapping it.
  - confidence: 0.85
- **[P2] Avoid stale runId-only stops after relinking:** `ui/src/ui/controllers/workboard.ts:785-788`
  - body: Cards can be edited to point at a different `sessionKey` while retaining the old `runId`, and Stop then forwards that stale run id to `chat.abort`. When the run id is no longer active, the gateway returns `aborted: false`, so Workboard cannot stop the live linked session.
  - confidence: 0.84
- **[P2] Preserve explicit zero card positions:** `extensions/workboard/src/store.ts:172-175`
  - body: Using `normalizePosition(input.position, 0) || ...` discards a valid explicit `position: 0` and appends the card to the end of the column. Importers or callers that pin a card at the top cannot persist that ordering.
  - confidence: 0.88
