## Label Justifications

- `P1`: The report describes a current-release Control UI Talk workflow showing a stale provider error despite successful relay startup, which can break real voice-session diagnosis and retry behavior.
- `impact:session-state`: The bug is about stale Talk/UI error state persisting across session attempts and mismatching the current relay session state.

## Summary

This should stay open: current main still has a source-backed stale-error path in Control UI Talk, and the relay startup path is still split between client-owned and Gateway-owned session creation instead of clearly following the configured relay path.

## What This Changes

The issue reports stale Control UI Chat Talk error text after gateway-relay Talk session creation succeeds, plus a shutdown append-audio race after the relay session closes.

## Best Possible Solution

Make Talk error display state belong to the active Talk attempt: clear stale `lastError` on a fresh start, route explicit Gateway relay sessions through `talk.session.create`, and add focused tests for retry/startup and post-close append-audio behavior.

## Reproduction Assessment

No live browser repro was run, but source inspection gives a clear current-main reproduction path for the stale error: set `lastError`, start Talk successfully, and the Chat error callout can still render the old string.

## Solution Assessment

Yes, a narrow bug fix is the right direction; the repair should update the existing Talk startup/error lifecycle and tests rather than adding a new mode or config knob.
