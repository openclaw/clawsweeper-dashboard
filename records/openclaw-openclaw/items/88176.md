## Summary

Keep this open: current main and the latest release still have a source-backed stale Chat error path for Control UI Talk, and adjacent realtime auth/Azure work does not close this UI session-state bug.

## What This Changes

The issue reports that Control UI Chat Talk can keep showing an old `gpt-realtime` access error even after the configured gateway-relay `talk.session.create` path succeeds.

## Best Possible Solution

Make Chat-visible Talk errors belong to the active Talk attempt: clear or scope stale `lastError` on a fresh start, preserve the documented client-owned versus Gateway-owned Talk boundary, and add focused regression coverage for retry and relay-close behavior.

## Reproduction Assessment

Yes by source inspection for the stale-error path: seed `lastError`, start Talk successfully, and Chat still receives `state.lastError` while the new Talk detail is cleared. I did not run a live browser/Gateway repro in this read-only sweep.

## Solution Assessment

Yes, a narrow lifecycle repair is the best path. The existing Talk state and relay transport code should be fixed directly instead of adding a new mode, config knob, or broad provider-auth redesign.
