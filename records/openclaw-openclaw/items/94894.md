## Summary

Keep open: current main and v2026.6.10 still have source-backed gaps for Feishu DM image delivery. Direct reply media is sent without local-root context even though the Feishu media loader requires it for local paths, and pathless `read` image results still do not promote to outbound media; related PRs cover partial slices but are open/conflicting or closed unmerged.

## What This Changes

The issue reports that Feishu DM outbound media from `MEDIA:` directives and `read` tool image results can be lost or downgraded to text when streaming-card replacement discards the image.

## Best Possible Solution

Consolidate a maintainer-owned Feishu media delivery fix that preserves trusted read-tool media, passes local-root context into direct reply media sends, and verifies live Feishu DM streaming/final media behavior before closing.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current reply media sends local paths without the required root context, and current read-image extraction still returns no media when image content lacks `details.path`. I did not run a live Feishu DM replay in this read-only review.

## Solution Assessment

Unclear as one patch: the best fix likely needs both the shared read-image media bridge and the Feishu direct reply-media local-root handoff, with the separate multipart upload PR handled on its own track. A streaming-card-only fallback would be too narrow.
