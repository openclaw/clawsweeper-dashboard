## Summary

Keep open: current main has the documented Control UI live tool-card machinery, but the Codex harness path does not currently project Codex dynamic/app-server tool lifecycle into the standard live `tool` event shape that WebChat consumes.

## What This Changes

The issue asks for Codex harness tool-call lifecycle events to appear as live Control UI/WebChat tool cards with progress and completion state.

## Best Possible Solution

Project Codex dynamic/app-server tool lifecycle into the existing runtime-neutral live tool-card contract with start, update/output, result/failure, stable `toolCallId`, compact sanitized args, and sanitized output. Keep generic `item` events for broader progress, but make Control UI/WebChat receive the same live card stream used by PI tools.

## Reproduction Assessment

Yes. A code-level reproduction path is clear: simulate a Codex dynamic tool call through the bridge/projector and observe that Control UI’s live-card handler ignores the resulting non-`tool` events, leaving `chatToolMessages` empty.

## Solution Assessment

Yes. The narrowest maintainable fix is to add a Codex-to-standard-tool projection in the Codex-owned harness path or a small shared adapter that feeds the existing WebChat card pipeline, rather than inventing a parallel Codex-only UI renderer.
