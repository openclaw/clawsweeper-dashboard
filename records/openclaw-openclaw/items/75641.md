## Summary

Keep open. Current main still leaves Codex dynamic tool lifecycle on `item`/`codex_app_server.*` or internal trajectory paths while Control UI/WebChat live cards consume only the standard `stream: "tool"` contract.

## What This Changes

The issue requests live Control UI/WebChat tool-call cards and progress for Codex harness dynamic tool calls.

## Best Possible Solution

Add Codex-owned projection from OpenClaw-owned dynamic tool start/update/result/failure into the existing runtime-neutral `stream: "tool"` contract while preserving existing debug/item events.

## Reproduction Assessment

Yes at source level. Route a Codex `item/tool/call`/`dynamicToolCall` through the app-server bridge/projector and the emitted events are `item` or `codex_app_server.*`; `handleAgentEvent` ignores them because they are not `stream: "tool"` with a `toolCallId`.

## Solution Assessment

Yes. The narrow maintainable fix is to project Codex dynamic tool lifecycle into the existing Gateway/Control UI tool-event contract, not add a Codex-only renderer or config flag.
