## Summary

Keep open. Current main still has a Codex harness projection gap: Codex dynamic/app-server tool activity is emitted as item/Codex-specific events or returned as tool results, while Control UI/WebChat live tool cards only consume the standard `stream: "tool"` event contract.

## What This Changes

Project Codex harness live dynamic tool lifecycle events into Control UI/WebChat tool cards with progress and completion state.

## Best Possible Solution

Implement a Codex harness projection layer that maps stable dynamic/app-server tool lifecycle data onto the existing runtime-neutral `stream: "tool"` contract with stable IDs, redacted compact args/output, and terminal success/failure phases.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to pass a Codex dynamic app-server tool lifecycle through the current projector/bridge and observe that WebChat receives `item` or `codex_app_server.*` events, while `handleAgentEvent` ignores them for tool cards because they are not `stream: "tool"` with `toolCallId`.

## Solution Assessment

Yes. The narrow maintainable fix is to emit the existing Gateway/Control UI tool event contract from the Codex harness path rather than adding a parallel Codex-only renderer in Control UI.
