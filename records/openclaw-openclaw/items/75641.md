## Summary

Keep open. Current main still has a Codex harness projection gap: Codex dynamic/app-server tool activity is emitted as `item` or `codex_app_server.*` events, while Control UI/WebChat live tool cards only consume the standard `stream: "tool"` contract with a `toolCallId`.

## What This Changes

Request projection of Codex harness tool lifecycle events into Control UI/WebChat live tool-call cards and progress.

## Best Possible Solution

Emit runtime-neutral `stream: "tool"` start/update/result events from the Codex harness for OpenClaw-owned dynamic tools, using stable IDs, sanitized compact args/results, and tests across the projector/bridge, Gateway routing, and WebChat handler.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to pass a Codex dynamic app-server tool lifecycle through current main and observe that the projector/bridge produces `item` or `codex_app_server.*` data while `handleAgentEvent` ignores it because it is not `stream: "tool"` with `toolCallId`.

## Solution Assessment

Yes. The maintainable fix is to project Codex harness tool lifecycle data into the existing Gateway/UI `stream: "tool"` contract rather than adding a parallel Codex-only renderer in WebChat.
