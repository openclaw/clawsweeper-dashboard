## Summary

Keep open: current main still has the documented live tool-card pipeline, but the Codex harness path does not project Codex/OpenClaw dynamic tool lifecycle into the standard `tool` agent events that Control UI/WebChat renders. The earlier ClawSweeper comment matches the current code shape, so this remains a focused implementation issue rather than a stale or duplicate cleanup item.

## What This Changes

Request that Codex harness live tool-call lifecycle be projected into the existing Control UI/WebChat tool-card stream, including start, progress/output, completion, and failure states.

## Best Possible Solution

Add a Codex-owned adapter that emits the existing runtime-neutral `tool` agent event lifecycle for OpenClaw dynamic tools, using stable `toolCallId`, compact sanitized args, bounded/sanitized output, and the current gateway/UI live-card pipeline. Avoid adding a parallel Codex-specific renderer in Control UI.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is available: simulate a Codex dynamic tool call through the bridge/projector and observe that WebChat only creates cards for `stream: "tool"` events with `toolCallId`, leaving Codex `item`/`codex_app_server.item` projections out of `chatToolMessages`.

## Solution Assessment

Yes. The narrowest maintainable solution is to project Codex dynamic tool lifecycle into the existing standard tool-event contract, rather than teaching Control UI to special-case Codex app-server item streams.
