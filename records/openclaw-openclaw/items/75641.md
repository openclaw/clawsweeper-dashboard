## Summary

Keep open. Current main still has the reported projection gap: Codex app-server tool activity is surfaced as `item`/`codex_app_server.*` events or text tool summaries, while Control UI live tool cards only consume the standard `stream: "tool"` contract with a `toolCallId`.

## What This Changes

Request Codex harness tool execution to appear in Control UI/WebChat as live tool-call cards with start, output/progress, and completion/failure state.

## Best Possible Solution

Project Codex app-server dynamic/OpenClaw-owned tool lifecycle data onto the existing runtime-neutral `stream: "tool"` event contract instead of adding a Codex-only renderer in Control UI.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to pass a Codex dynamic `item/tool/call` through the current bridge/projector and observe that Control UI receives `item` or `codex_app_server.*` events, while `handleAgentEvent` ignores them for live cards because they are not `stream: "tool"` with `toolCallId`.

## Solution Assessment

Yes. The narrow maintainable fix is to emit the existing Gateway/Control UI tool-event contract from the Codex harness path, preserving current assistant text segmentation rather than creating a parallel UI event model.
