## Summary

Keep open. Current main still routes Control UI/WebChat live tool cards only from the runtime-neutral `stream: "tool"` contract, while the Codex harness path emits Codex-specific `item`/`codex_app_server.*` events and returns dynamic tool results to Codex without projecting a standard live tool lifecycle.

## What This Changes

The issue asks OpenClaw to show Codex harness live tool execution in Control UI/WebChat as the same tool cards/progress used for native agent tool events.

## Best Possible Solution

Add a Codex-owned projection layer that maps stable OpenClaw-owned Codex dynamic/app-server tool lifecycle data into the existing runtime-neutral `stream: "tool"` event contract, preserving existing assistant text segmentation and Codex debug events.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to pass a Codex dynamic tool call or app-server item lifecycle through current main and observe that Control UI receives `item`/`codex_app_server.*` events while `handleAgentEvent` only builds tool cards from `stream: "tool"` events with `data.toolCallId`.

## Solution Assessment

Yes. The narrow maintainable fix is to emit the existing standard live tool-card contract from the Codex harness path instead of adding a parallel Codex-only renderer in Control UI.
