## Summary

Keep open. Current main still has a source-confirmed Codex harness projection gap: dynamic app-server tool lifecycle is projected as `item`/`codex_app_server.*` or returned to Codex, while Control UI/WebChat live cards consume only the standard `stream: "tool"` contract.

## What This Changes

Request live Codex harness tool-call progress in Control UI/WebChat using the existing OpenClaw tool-card event path.

## Best Possible Solution

Add a Codex-owned projection that maps OpenClaw-owned dynamic tool start/update/result/failure onto the runtime-neutral `stream: "tool"` event contract while preserving existing `item` and `codex_app_server.*` debug events.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to send a Codex `dynamicToolCall` through the current projector/bridge and observe `item` or `codex_app_server.*` events, then pass those events to `handleAgentEvent`, which ignores them because they are not `stream: "tool"` with `toolCallId`.

## Solution Assessment

Yes. The narrow maintainable fix is to project Codex dynamic tool lifecycle into the existing Gateway/Control UI tool-event contract, rather than adding a Codex-only renderer or changing the generic WebChat card model.
