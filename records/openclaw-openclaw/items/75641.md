## Summary

Keep open. Current main still leaves Codex harness dynamic tool lifecycle on Codex item/debug/trajectory paths, while Control UI/WebChat live tool cards consume only the standard `stream: "tool"` agent-event contract.

## What This Changes

The issue asks for Codex harness tool execution lifecycle events to appear as live Control UI/WebChat tool cards with progress and completion state.

## Best Possible Solution

Add Codex-owned projection from OpenClaw-owned dynamic tool start/update/result/failure into the existing `stream: "tool"` contract while preserving existing Codex item/debug events.

## Reproduction Assessment

Yes, at source level. Route a Codex `item/tool/call` or `dynamicToolCall` through the app-server bridge/projector and current main keeps it on `item`, `codex_app_server.*`, or trajectory paths rather than emitting `stream: "tool"` with a `toolCallId`.

## Solution Assessment

Yes. The narrow maintainable solution is to project Codex dynamic tool lifecycle events into the existing Gateway/Control UI tool-card contract, not add a Codex-only renderer or config flag.
