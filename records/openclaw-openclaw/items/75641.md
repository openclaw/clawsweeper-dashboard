## Summary

Keep open. Current main still has the reported Codex harness projection gap: Codex app-server tool activity is emitted as `item`/`codex_app_server.*` events or returned as tool results, while Control UI/WebChat live tool cards only consume the standard `stream: "tool"` contract.

## What This Changes

The issue asks for Codex harness live dynamic tool lifecycle events to appear as Control UI/WebChat tool cards with progress and completion state.

## Best Possible Solution

Implement a Codex-owned projection layer that maps stable app-server dynamic tool lifecycle data onto the existing runtime-neutral `stream: "tool"` event contract, preserving current `item` events and transcript delivery.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to pass a Codex dynamic app-server tool lifecycle through the current projector/bridge and observe that WebChat receives `item` or `codex_app_server.*` events, while `handleAgentEvent` ignores them because they are not `stream: "tool"` with `toolCallId`.

## Solution Assessment

Yes. The narrow maintainable fix is to emit the existing Gateway/Control UI tool-event contract from the Codex harness path rather than adding a parallel Codex-only renderer in Control UI.
