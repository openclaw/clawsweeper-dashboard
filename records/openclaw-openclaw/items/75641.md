## Summary

Keep open. Current main still has the Control UI/Gateway live-card path keyed to standard `stream: "tool"` events, while the Codex harness projects app-server tool activity as generic `item` / `codex_app_server.*` events and dynamic tool calls return Codex content without emitting that live-card contract. The related reports are broader or adjacent and do not supersede this focused parity gap.

## What This Changes

The issue asks for Codex harness tool-call lifecycle events to appear in Control UI/WebChat as live tool cards with progress, output, and completion state.

## Best Possible Solution

Project Codex dynamic/app-server tool lifecycle into the existing runtime-neutral live tool-card event contract with stable `toolCallId`, start/update/result or failure phases, compact sanitized args, and sanitized output. Keep generic `item` events for broader progress, but make Control UI/WebChat receive the same live card stream used by PI tools.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to simulate a Codex dynamic tool/app-server item through the bridge/projector and observe that the UI receives non-`tool` events, while `handleAgentEvent` ignores those events and leaves `chatToolMessages` empty.

## Solution Assessment

Yes. The narrow maintainable solution is a Codex-owned projection or small shared adapter that emits the standard `stream: "tool"` lifecycle, rather than adding a parallel Codex-only renderer in Control UI.
