## Summary

Keep open. Current main still has a clear gap between the Codex harness event projection and the live tool-card contract consumed by Control UI/WebChat: Codex app-server activity is projected as generic item/Codex-specific events, while the UI and Gateway live-card path are keyed to standard tool-stream events. The related reports are broader, adjacent, or already closed for different symptoms, so none supersede this focused parity issue.

## What This Changes

Project Codex harness dynamic/app-server tool lifecycle events into Control UI/WebChat live tool-call cards with start, progress/output, completion, and failure states while preserving assistant text around tools.

## Best Possible Solution

Add a Codex-owned projection layer, or a small shared adapter used by the Codex harness, that maps stable Codex dynamic/app-server tool lifecycle data into the existing runtime-neutral stream "tool" contract with stable toolCallId, sanitized compact args, updates/output, and terminal success/failure phases. Keep generic item/Codex debug events for diagnostics, but do not require a parallel Codex-only renderer in Control UI.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to project a Codex dynamic tool/app-server item through the current bridge and observe that the UI receives item or codex_app_server.* events; handleAgentEvent ignores those non-tool streams and leaves chatToolMessages empty.

## Solution Assessment

Yes. The narrow maintainable fix is to emit the standard live tool-card event contract from the Codex harness path rather than teaching Control UI a separate Codex-specific event model.
