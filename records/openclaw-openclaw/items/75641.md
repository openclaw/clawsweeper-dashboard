## Summary

Keep open. Current main still has the documented live tool-card machinery, but Codex app-server dynamic tool execution is projected as generic Codex/item events and verbose tool-result messages, not the standard live `tool` stream that Control UI/WebChat consumes; the related issues are adjacent but do not supersede this focused parity gap.

## What This Changes

Request that Codex harness runs surface live tool start, output/progress, completion, and failure as Control UI/WebChat tool cards without duplicating surrounding assistant text.

## Best Possible Solution

Project Codex OpenClaw-owned dynamic tool lifecycle into the existing runtime-neutral live `tool` event contract with stable `toolCallId`, sanitized compact args, progress/output updates, and result/failure status, while keeping the generic `item`/`codex_app_server.*` events for diagnostics and broader progress surfaces.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to simulate a Codex dynamic tool call through the bridge/projector and pass the resulting event stream to `handleAgentEvent`; current main produces or forwards non-`tool` events for that path, and WebChat ignores them for live cards. No live UI session was run because this review was read-only.

## Solution Assessment

Yes. The narrowest maintainable fix is to add Codex-to-standard-tool projection in the Codex harness adapter or a tiny shared projection helper, rather than building a Codex-only renderer in WebChat.
