## Summary

Keep open. Current main still projects Codex tool-like lifecycle as `stream: "item"` or `codex_app_server.*` events, or returns dynamic-tool results directly to Codex, while Control UI/WebChat live cards consume only the standard `stream: "tool"` event shape with a `toolCallId`.

## What This Changes

Request Codex harness dynamic tool calls to appear in Control UI/WebChat as live tool-call cards with start, progress/output, completion, and failure state.

## Best Possible Solution

Add a Codex-owned projection that maps OpenClaw-owned dynamic tool start, update/output, result, and failure onto the runtime-neutral `stream: "tool"` event contract while preserving existing Codex debug events.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to send a Codex `dynamicToolCall` or `item/tool/call` through the current projector/bridge and observe `item`/`codex_app_server.*` events or a direct Codex response; passing those events to `handleAgentEvent` produces no live card because they are not `stream: "tool"` with `toolCallId`.

## Solution Assessment

Yes. The narrow maintainable fix is to project Codex dynamic tool lifecycle into the existing Gateway/Control UI tool-event contract, rather than adding a Codex-specific WebChat renderer or changing generic card handling.
