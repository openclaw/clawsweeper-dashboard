## Summary

Keep open. Current main still has a Codex harness projection gap: Codex app-server tool lifecycle is exposed as item/Codex-specific events or returned tool results, while Control UI/WebChat live tool cards are driven by the standard `stream: "tool"` event contract.

## What This Changes

Request that Codex harness runs surface live tool-call lifecycle and progress as Control UI/WebChat tool cards instead of only final assistant text.

## Best Possible Solution

Emit a small Codex-owned projection from stable Codex dynamic/app-server tool lifecycle data into the existing runtime-neutral `stream: "tool"` contract with stable `toolCallId`, sanitized args, progress/output updates, and terminal success/failure, while keeping Codex-specific item/debug streams for diagnostics.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to feed a Codex dynamic tool/app-server item through the current projector: it emits `item` or `codex_app_server.item`, while Gateway and Control UI only create live cards for `stream: "tool"` events with `toolCallId`.

## Solution Assessment

Yes. The narrow maintainable fix is to project Codex tool lifecycle into the existing tool-stream contract rather than adding a parallel Codex-only renderer in Control UI/WebChat.
