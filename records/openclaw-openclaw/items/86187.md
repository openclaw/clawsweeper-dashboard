## Summary

Keep open: the patch targets the right Control UI session-loading helper, but source review shows the picker search and pagination RPC path still builds `sessions.list` params without `agentId`, so the same picker surface can still return other agents' sessions during search.

## What This Changes

The PR changes Control UI chat session loading to derive an agent id from the current chat session key and include it in session-list overrides.

## Best Possible Solution

Apply the selected-agent filter consistently across normal refresh, picker search, and picker pagination, then show redacted after-fix Control UI proof for the multi-agent session picker.

## Reproduction Assessment

Yes, from source inspection: current main and `v2026.5.22` build chat picker search params without `agentId`, while the backend-facing sessions controller already supports forwarding `overrides.agentId`. I did not run a browser repro because this is a read-only review.

## Solution Assessment

No. The proposed helper change is the right direction for normal session refreshes, but the maintainable fix also needs the picker search/page request builder to carry the same selected-agent filter.
