## Summary

Keep open: current main still has the Mattermost media-alias gap, but this PR is not merge-ready because blank `media` values still prevent valid `filePath` or `path` fallbacks from being used.

## What This Changes

The PR changes Mattermost message-tool send handling to derive `mediaUrl` from `params.media`, then `params.filePath`, then `params.path`.

## Best Possible Solution

Normalize each media candidate independently, preserve the intended precedence, add focused Mattermost handleAction tests for both aliases and blank fallthrough, and add the required changelog fix entry.

## Reproduction Assessment

Yes. A focused unit test can call Mattermost `handleAction` with `action: "send"`, blank or whitespace `media`, and a valid `filePath`; the PR head would forward `mediaUrl: undefined`, while current main ignores `filePath` and `path` entirely.

## Solution Assessment

No, not as currently patched. The direction is right, but the fallback should use normalized optional-string semantics for each candidate before falling through, with regression coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Skip blank media before falling back to aliases:** `extensions/mattermost/src/channel.ts:216-218`
  - body: The fallback chain only reaches `filePath` or `path` when `params.media` is not a string. If a tool call sends `media: ""` or whitespace alongside a valid local path, the trim returns `undefined` and the alias upload is still skipped, so normalize each candidate before falling through.
  - confidence: 0.91
