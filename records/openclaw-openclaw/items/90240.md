## Summary

Keep open: current main has recovery machinery for stale active-run bookkeeping, but this report reproduces against the latest shipped release and the post-release diff does not show a clear fix for the pinned active-run/reply-run state described here.

## What This Changes

Reports a 2026.6.1 Slack DM stall where active embedded-run bookkeeping stayed nonzero after completed large and normal turns, leaving later inbound messages queued until gateway restart.

## Best Possible Solution

Investigate this as a core gateway/session-lane bug, identify which active-run owner survives after terminal completion, and add regression coverage that proves queued inbound messages drain without a manual restart after oversized and back-to-back completed turns.

## Reproduction Assessment

No, not with high confidence. The report has concrete production logs and the current source maps to the active embedded-run/reply-run bookkeeping path, but I did not establish a deterministic current-main replay for the single oversized Opus turn.

## Solution Assessment

Unclear. Periodic reconciliation may be useful, but the best fix should first prove whether the stale owner is the embedded registry, reply registry, diagnostic activity, or queue handoff and then repair that owner directly.
