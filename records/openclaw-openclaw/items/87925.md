## Summary

Keep open. Current main and the latest release still persist the thinkingLevel remap in both user-initiated model-switch paths, while the linked PR explicitly leaves this as follow-up work; because docs/tests also encode stored-level remapping, this needs a maintainer compatibility decision rather than cleanup closure.

## What This Changes

The issue asks whether model switching should preserve an explicit session thinkingLevel preference instead of permanently persisting a provider-support downgrade.

## Best Possible Solution

Make the model-switch thinking contract explicit: either preserve the stored preference and downgrade only the effective turn value, or keep persistent remapping but document and notify it consistently across sessions.patch and /model.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: switch a session with an unsupported stored thinkingLevel to a model without sending an explicit thinkingLevel, and the current code writes the remapped value back to the store. I did not run a live gateway repro in this read-only review.

## Solution Assessment

Unclear: the mechanical fix is bounded, but the correct end state depends on whether maintainers want model switches to preserve user intent or normalize stored state for the selected model. The safest path is to decide that contract first, then update both model-switch paths and tests together.
