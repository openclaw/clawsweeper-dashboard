## Summary

Keep open. Current main and the latest release still persist the model-switch thinkingLevel remap, and current docs/tests encode that behavior, so this needs a maintainer product/compatibility decision rather than cleanup closure.

## What This Changes

The issue asks whether model-switch paths should preserve a stored thinkingLevel preference instead of persisting a support-driven downgrade when the new model does not support the inherited level.

## Best Possible Solution

Make the model-switch thinking contract explicit, then update sessions.patch, /model persistence, user-facing notification behavior, docs, and focused regression tests together to match that contract.

## Reproduction Assessment

Yes. Store a thinkingLevel unsupported by the next selected model, switch the session model without an explicit thinkingLevel or /think directive, and current code writes the remapped level back to persistent session state.

## Solution Assessment

Unclear. The mechanical change is bounded, but the best solution depends on the maintainer choice between preserving the stored preference and normalizing stored state for the active model.
