## Summary

Keep open: current main still has a source-reproducible Slack interaction wake gap, and https://github.com/openclaw/openclaw/pull/79836 is an open closing PR for the same bug, so this issue should wait for that PR to merge or be rejected.

## What This Changes

Slack interactive reply buttons/selects should route received click events back into the current Slack session/thread and visibly continue the turn.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/79836 if its thread-aware wake fix remains correct, then close this issue as implemented; keep broader Slack thread-routing follow-up in https://github.com/openclaw/openclaw/issues/61502 if needed.

## Reproduction Assessment

Yes by source inspection, though not by live Slack proof: current main parses and queues Slack block-action interactions but does not request a wake or pass `threadTs` into session resolution. The issue log and related Socket Mode report confirm Slack delivery succeeds before continuation fails.

## Solution Assessment

Yes, the best direction is the narrow Slack-owned fix already proposed in https://github.com/openclaw/openclaw/pull/79836: resolve the same thread/session as inbound Slack messages and wake the queued interaction once. Closing this issue before that PR lands would be premature.
