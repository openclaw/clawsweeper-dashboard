## Summary

Keep open: current main still acknowledges Slack reply-button interactions and enqueues them as system events, but I did not find a wake or continuation step that would make the click visibly continue the active Slack session.

## What This Changes

Slack interactive reply buttons/selects should continue the current Slack session after OpenClaw receives the interaction event.

## Best Possible Solution

Fix the Slack-owned interaction path so queued reply-button/select events immediately wake or continue the same resolved Slack session while preserving existing auth, approval, plugin-dispatch, and threading behavior.

## Reproduction Assessment

Yes by source inspection, but not by live Slack proof: the block-action handler enqueues the interaction into a resolved session and then updates the Slack message without any visible wake/continuation call on that branch.

## Solution Assessment

Yes, the best direction is a narrow Slack interaction fix that wakes the resolved session after enqueueing the event; changing `replyToMode`, requiring threads, or treating this as setup support would not address the received-but-silent click path.
