## Summary

This PR should stay open because it fixes a real plugin-runtime bug, but the current head is not merge-ready: the merge ref still fails `check-test-types`, and the catalog metadata change can still miss a pinned tool-only registry when the active registry has channel entries.

## What This Changes

The PR prevents tool-discovery plugin registry loads from replacing the active provider registry, tracks workspaceDir on pinned surfaces, and lets `tools.catalog` read metadata and declared tools from active and channel registries.

## Best Possible Solution

Keep the provider-preservation guard, but repair the catalog/tool selector so pinned tool-discovery metadata is read directly and land only after type and focused runtime/catalog checks pass.

## Reproduction Assessment

Yes. Source inspection shows current main promotes `activate:false` tool-discovery registries to active, and the PR body includes before/after terminal output showing providers stay at 1 after the guard; I did not run the repro in this read-only review.

## Solution Assessment

No, not yet. The active-registry guard is the right layer for provider preservation, but the current PR still needs a type-clean test and a direct pinned tool-registry metadata path that is not hidden by active-channel selection.
