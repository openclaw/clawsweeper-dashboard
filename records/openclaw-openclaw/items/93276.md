## Summary

Keep this PR open: it fixes a real provider-clobbering path on current main, but the current branch still has a blocking selector gap where a pinned tool-only registry can be hidden behind the active channel registry, leaving catalog metadata and declared tools incomplete.

## What This Changes

The PR guards tool-discovery standalone registry loads from replacing the active plugin registry, tracks workspaceDir per runtime surface, and extends tools.catalog to read metadata and declared tools from a channel registry.

## Best Possible Solution

Keep the toolDiscovery active-registry guard, add a raw pinned tool-discovery registry accessor or selector mode for catalog/tool resolution with active-metadata precedence, and cover the active-with-channel plus pinned-tool-only case before merge.

## Reproduction Assessment

Yes. Current main unconditionally promotes activate:false tool-discovery registries in installStandaloneRegistry, and the PR body provides before/after terminal output showing providers drop before the guard and remain preserved after it; I did not run the repro in this read-only review.

## Solution Assessment

No, not yet. The active-registry guard is the right fix location for provider preservation, but the catalog/tool metadata path still needs a direct pinned tool-registry read that is not hidden by active-channel selection.
