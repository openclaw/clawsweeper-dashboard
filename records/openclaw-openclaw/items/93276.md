## Summary

Keep open: this is a plausible core runtime bug fix, but the current head still has a test type error and the catalog metadata fix can miss tool-only pinned registries when the active registry already has channels.

## What This Changes

The PR prevents `activate:false` tool-discovery registry loads from replacing the active plugin registry, tracks workspaceDir per pinned surface, and extends `tools.catalog` metadata projection to the channel registry.

## Best Possible Solution

Keep the active-provider preservation guard, but make the pinned tool-discovery registry directly reachable for tool catalog/tool resolution even when the active registry has channels, then land with type-clean regression coverage.

## Reproduction Assessment

Yes. Source inspection shows current main promotes `activate:false` tool-discovery registries to active, and the PR body includes before/after terminal output showing providers stay at 1 after the guard; I did not run the repro in this read-only review.

## Solution Assessment

No. The active-registry guard is the right layer for provider preservation, but the best fix also needs type-clean tests and a selector/raw-registry path that does not hide pinned tool-only registries behind an active channel registry.
