## Summary

Keep open: the provider-preservation direction is right and the supplied proof supports the loader bug, but the submitted fix leaves `tools.catalog` reading plugin tool metadata from the active registry after tool-discovery registries stop becoming active, so plugin tool display/risk metadata can be dropped.

## What This Changes

The PR makes standalone tool-discovery registry installs skip `setActivePluginRegistry` while adding loader tests for active-registry preservation, channel pinning, and the non-tool-discovery migration-provider path.

## Best Possible Solution

Preserve active providers while making the tool catalog/tool resolution path carry or read metadata from the same pinned tool-discovery registry, with regression coverage for both provider preservation and catalog metadata.

## Reproduction Assessment

Yes by source inspection and contributor terminal proof: current main promotes `activate:false` tool-discovery registries to active, and the PR body shows before/after provider counts through the real loader path. I did not run the repro because this review is read-only.

## Solution Assessment

No as submitted: the guard is the right layer for preserving active providers, but the best fix must also move `tools.catalog` metadata projection off the active-registry side effect.
