## Summary

Keep open: the guard fixes the active-provider clobber path, but as submitted it leaves `tools.catalog` reading plugin tool metadata from the active registry after tool-discovery registries are no longer promoted there.

## What This Changes

The PR changes standalone plugin registry installation so `toolDiscovery` loads skip active-registry promotion while preserving channel/http-route pinning, with loader regression tests.

## Best Possible Solution

Keep the `toolDiscovery` active-registry guard, but make the tool catalog read metadata from the same loaded or pinned tool registry and cover both provider preservation and catalog metadata in regression tests.

## Reproduction Assessment

Yes. Source inspection shows current main promotes `activate:false` tool-discovery registries to active, and the PR body includes before/after terminal output from the real loader path showing providers drop before the patch and remain after it.

## Solution Assessment

No as submitted. The active-registry guard is the right layer for preserving providers, but the best fix must also move `tools.catalog` metadata projection off the active-registry side effect.
