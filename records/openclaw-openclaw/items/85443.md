## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Attach capture before reporting same-channel start:** `extensions/discord/src/voice/meeting-notes-source.ts:55`
  - body: `manager.join` returns success when the bot is already in the same channel, but that early return does not attach the new `meetingNotes` callback. This reports a started notes session while no transcript utterances can be captured.
  - confidence: 0.91
- **[P1] Finalize notes when provider stop is non-critical:** `extensions/meeting-notes/src/tool.ts:182-183`
  - body: Throwing on any provider stop error prevents `stoppedAt`, active-session cleanup, and summary generation. A Discord session that is already disconnected should still be finalized so captured transcript data is not left in an open session.
  - confidence: 0.89
- **[P2] Reject ambiguous Discord manager selection:** `extensions/discord/src/voice/meeting-notes-source.ts:20-25`
  - body: When `accountId` is omitted, this chooses the first registered manager by insertion order. Multi-account gateways can start, stop, or report status for the wrong Discord account, so ambiguous requests should fail unless there is exactly one manager.
  - confidence: 0.88
- **[P2] Stop late auto-start sessions during shutdown:** `extensions/meeting-notes/src/tool.ts:358-362`
  - body: If shutdown happens while `startMeetingNotes` is still in flight, this callback can add a session id after `stop()` already ran. That leaves a capture session outside the service teardown path.
  - confidence: 0.84
- **[P2] Merge meeting-notes providers when active:** `src/plugins/capability-provider-runtime.ts:66`
  - body: Adding `meetingNotesSourceProviders` without including it in the active-plus-manifest merge path means `listMeetingNotesSourceProviders()` can return only active providers once the meeting-notes plugin is loaded, hiding manifest-owned providers such as `discord-voice` from status output.
  - confidence: 0.82
