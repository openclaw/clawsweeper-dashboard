## Summary

Keep open: current main still applies `responsePrefix` on the direct reply path, but visible `message(action="send")` text is built and forwarded without the same prefix cascade. The latest human follow-up confirms the same WhatsApp provenance workflow, and the related per-channel prefix request is already closed as a different scope.

## What This Changes

Apply configured `messages.responsePrefix`/channel/account prefix handling to visible outbound sends made through the message tool, matching normal direct replies.

## Best Possible Solution

Normalize visible message-tool send payloads through the existing reply-prefix context and `normalizeReplyPayload` before channel dispatch, without adding a per-send opt-out unless maintainers decide that separately.

## Reproduction Assessment

Yes. Source inspection on current main shows direct replies call `normalizeReplyPayload` with resolved prefix context, while `message(action="send")` builds raw `payload.text` and forwards it through `executeSendAction`/`sendMessage` unchanged.

## Solution Assessment

Yes. The narrow maintainable fix is to reuse the existing prefix cascade/normalizer in the shared visible send path before plugin/core/gateway delivery; a new suppression flag should remain a separate product decision.
