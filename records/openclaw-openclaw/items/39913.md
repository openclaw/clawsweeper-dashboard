## Summary

Current main still has a real gap: direct reply normalization applies `responsePrefix`, but `message(action="send")` routes text through outbound send without that normalization. The request is valid, but changing all tool-initiated sends needs maintainer policy on scope and opt-out behavior.

## What This Changes

Apply configured outbound `responsePrefix` to message-tool `send` calls so tool-initiated visible messages match direct replies.

## Best Possible Solution

Define the prefix scope for message-tool sends, then add shared send normalization plus regression coverage for core and plugin delivery paths.

## Reproduction Assessment

Yes. Source inspection shows direct replies use `normalizeReplyPayload` with `responsePrefix`, while the message-tool send path forwards the parsed message to outbound delivery without that normalization.

## Solution Assessment

Unclear. Applying prefix normalization in the shared message-action send path looks like the narrow technical fix, but maintainers should first decide the intended scope and any suppress-prefix semantics.
