## Summary

Current main still leaves the `message(action="send")` path outside the existing `responsePrefix` normalization, while docs describe `messages.responsePrefix` as an outbound prefix cascade and group visible replies as message-tool-driven. This remains a valid, narrow bug to fix.

## What This Changes

Apply the configured `messages.responsePrefix` cascade to outbound sends made through the shared `message` tool.

## Best Possible Solution

Resolve the effective response prefix for message-tool send actions using the same global/channel/account cascade as normal replies, apply it once before plugin/gateway/core dispatch, and add focused regression coverage for global and per-channel/account prefixes.

## Reproduction Assessment

Yes, from source inspection: configure `messages.responsePrefix`, invoke `message(action="send", target=..., message="Hello")`, and the send path forwards raw `params.message` without calling the existing prefix normalization. I did not run a live WhatsApp path, but the current-main code path is clear.

## Solution Assessment

The issue direction is right, but applying this at every gateway egress point is broader than needed. The safer fix is a shared send-path normalization step in the message-action path before plugin/gateway/core dispatch, using the established prefix cascade and no-double-prefix behavior.
