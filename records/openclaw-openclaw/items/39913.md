## Summary

Keep open: current main still applies the configured response-prefix cascade only on the normal reply normalization path, while visible `message(action="send")` builds and dispatches raw text. The May 9 follow-up adds concrete WhatsApp provenance/model-attribution impact, and the earlier same-direction pull request was closed unmerged rather than superseding this issue.

## What This Changes

Apply the existing `messages.responsePrefix` behavior to visible outbound sends initiated through the message tool.

## Best Possible Solution

Normalize visible message-tool send payloads through the existing prefix cascade and template resolver before gateway, plugin, or core dispatch, while preserving no-prefix behavior and double-prefix idempotency.

## Reproduction Assessment

Yes, source-reproducible. Direct replies pass through `createReplyPrefixContext` and `normalizeReplyPayload`, while `message(action="send")` builds raw `message` and `payload.text` without any prefix normalizer; no live WhatsApp send was run in this read-only review.

## Solution Assessment

Yes. The maintainable fix is to reuse the existing prefix cascade and reply normalizer at the shared visible message-tool send boundary; any per-send opt-out should be a separate product decision.
