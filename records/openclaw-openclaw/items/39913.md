- `P2`: This is a normal-priority visible message delivery consistency bug affecting users who rely on configured prefixes for provenance.

## Summary

Keep this open: current main and the latest release still build message-tool sends without applying `messages.responsePrefix`, and https://github.com/openclaw/openclaw/pull/92299 is an open closing PR for the exact gap. The follow-up discussion adds useful template-variable acceptance details that should be checked during PR review before the issue closes.

## What This Changes

The issue asks for configured response prefixes to be applied consistently to visible `message(action="send")` outbound sends, not only normal direct replies.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/92299, or an equivalent core outbound fix, with regression coverage for prefix precedence, idempotency, and template-variable parity.

## Reproduction Assessment

Yes, at source level. Current main applies `responsePrefix` on the normal reply path, while `buildSendPayloadParts` constructs message-tool send text without resolving or prepending that prefix.

## Solution Assessment

Yes, the best fix belongs in the shared core outbound message-tool send builder before channel delivery. The issue should remain open until the linked PR or an equivalent implementation lands.
