## Summary

Keep this open: current `main` and latest release `v2026.6.6` still construct message-tool send payloads without applying `messages.responsePrefix`, and https://github.com/openclaw/openclaw/pull/92299 is the open closing PR for this exact gap.

## What This Changes

The issue requests applying `messages.responsePrefix` to visible outbound `message(action="send")` sends so they match normal direct replies.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/92299 or an equivalent core outbound fix with regression coverage for prefix precedence, idempotency, and template-variable parity.

## Reproduction Assessment

Yes, at source level. Current `main` applies `responsePrefix` on the normal reply path, while `buildSendPayloadParts` constructs message-tool send text without resolving or prepending that prefix.

## Solution Assessment

Yes, the best direction is a narrow fix in the shared core outbound message-tool send builder before channel delivery. The issue should remain open until the linked PR or an equivalent implementation lands.
