## Summary

Keep open: current main still has a source-reproducible inconsistency where direct replies apply the configured outbound prefix cascade, while visible `message(action="send")` delivery builds and dispatches raw message text without the same prefix normalizer. The recent WhatsApp group follow-up adds concrete user-visible provenance impact, and the earlier same-direction pull request is closed unmerged rather than a superseding fix.

## What This Changes

Apply `messages.responsePrefix` consistently to visible outbound sends produced by the `message` tool.

## Best Possible Solution

Reuse the existing reply-prefix context and `normalizeReplyPayload` at the shared visible message-tool send boundary before gateway, plugin, or core dispatch, with regression coverage for prefix cascade, template values, and double-prefix avoidance.

## Reproduction Assessment

Yes. Source inspection on current main shows direct replies pass through `createReplyPrefixContext` and `normalizeReplyPayload`, while `message(action="send")` builds and dispatches raw `message`/`payload.text`; I did not perform a live WhatsApp send.

## Solution Assessment

Yes. The narrow maintainable fix is to reuse the existing prefix cascade and reply normalizer for visible message-tool sends; any per-send suppression flag should be a separate product decision.
