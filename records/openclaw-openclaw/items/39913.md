## Summary

Keep open: current main still applies the response-prefix cascade on normal reply delivery, but visible `message(action="send")` sends build raw `message` and `payload.text` before plugin, gateway, or core dispatch. The May 9 WhatsApp follow-up and the closed unmerged prior attempt at https://github.com/openclaw/openclaw/pull/9103 support that the attribution gap remains.

## What This Changes

The issue asks OpenClaw to prepend the configured `messages.responsePrefix` to visible message-tool sends the same way it does for normal direct replies.

## Best Possible Solution

Normalize visible message-tool send payloads through the existing reply-prefix context and template resolver before plugin, gateway, or core dispatch, preserving dry-run behavior, no-prefix compatibility, and double-prefix idempotency.

## Reproduction Assessment

Yes, source-reproducible. Normal replies call `createReplyPrefixContext` and `normalizeReplyPayload`, while `message(action="send")` builds raw `message` and `payload.text` without that prefix normalizer; no live WhatsApp send was run.

## Solution Assessment

Yes. The narrow maintainable fix is to reuse the existing prefix cascade and reply normalizer at the shared visible message-tool send boundary; any per-send opt-out should be separate product work.
