## Summary

Keep open: current main and the latest release still have a source-reproducible mismatch where normal reply delivery applies the configured response-prefix cascade, while visible `message(action="send")` builds and dispatches raw text without the same normalization. The May 9 follow-up adds concrete WhatsApp provenance/model-attribution impact, and the earlier same-direction pull request was closed unmerged rather than superseding this issue.

## What This Changes

Apply configured `messages.responsePrefix` and channel/account response-prefix overrides to visible outbound sends made through the `message` tool.

## Best Possible Solution

Reuse the existing reply-prefix context and `normalizeReplyPayload` at the shared visible message-tool send boundary before gateway, plugin, or core dispatch, with regression coverage for prefix cascade, template values, media/caption sends, and double-prefix avoidance.

## Reproduction Assessment

Yes, source-reproducible. Current main shows direct replies passing through `createReplyPrefixContext` and `normalizeReplyPayload`, while `message(action="send")` builds and sends raw `message`/`payload.text`; no live WhatsApp send was run in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is to reuse the existing prefix cascade and reply normalizer for visible message-tool sends; any per-send suppression flag should be a separate product decision.
