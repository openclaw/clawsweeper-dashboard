## Summary

Keep open: current main still applies the response-prefix cascade on normal reply delivery, but `message(action="send")` constructs raw `message` and `payload.text` before plugin, gateway, or core dispatch. The May 9 WhatsApp follow-up and the earlier closed-unmerged message-tool prefix PR confirm the same user-visible attribution gap remains.

## What This Changes

Apply the existing `messages.responsePrefix` cascade to visible outbound sends made through the `message` tool.

## Best Possible Solution

Normalize visible message-tool send payloads through the existing prefix cascade and template resolver before plugin, gateway, or core dispatch, preserving dry-run behavior, no-prefix compatibility, and double-prefix idempotency.

## Reproduction Assessment

Yes, source-reproducible. Direct replies call `createReplyPrefixContext` and `normalizeReplyPayload`, while `message(action="send")` builds raw `message` and `payload.text` without the prefix normalizer; no live WhatsApp send was run.

## Solution Assessment

Yes. The maintainable fix is to reuse the existing prefix cascade and reply normalizer at the shared visible message-tool send boundary; any per-send opt-out should remain separate follow-up product work.
