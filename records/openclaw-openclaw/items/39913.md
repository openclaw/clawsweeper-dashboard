## Summary

Keep this open: current `main` and the latest release still build message-tool sends without applying `messages.responsePrefix`, while https://github.com/openclaw/openclaw/pull/92299 is an open closing PR for the same gap. The issue should close after that PR, or an equivalent core outbound fix, lands.

## What This Changes

Requests that configured `messages.responsePrefix` be applied to visible outbound sends made through `message(action=send)`, matching normal reply delivery.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/92299, or an equivalent core outbound fix, with regression coverage for prefix precedence, idempotency, and template-variable parity.

## Reproduction Assessment

Yes, at source level. Current `main` applies `responsePrefix` on the normal reply path, while `buildSendPayloadParts` constructs message-tool send text without resolving or prepending that prefix.

## Solution Assessment

Yes, the best fix belongs in the shared core outbound message-tool send builder before channel delivery. The issue should remain open until the linked PR or an equivalent implementation lands.
