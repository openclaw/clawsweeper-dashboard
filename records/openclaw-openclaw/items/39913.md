## Summary

Keep open: current main still applies the outbound prefix cascade on direct reply delivery, but visible `message(action="send")` sends build and dispatch raw message text without the same normalizer. The WhatsApp group follow-up reinforces the same user-visible provenance problem, and the related prior PRs are closed stale rather than merged fixes.

## What This Changes

Apply configured `messages.responsePrefix` and the channel/account prefix cascade to visible outbound sends made through the `message` tool.

## Best Possible Solution

Reuse the existing reply-prefix context and `normalizeReplyPayload` in the shared visible message-tool send path before channel dispatch, with focused regression coverage; leave any per-send suppression flag for a separate product decision.

## Reproduction Assessment

Yes. Source inspection on current main shows direct replies pass through the prefix normalizer, while `message(action="send")` builds and dispatches raw `payload.text`; I did not run a live WhatsApp send in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is to reuse the existing prefix cascade/normalizer for visible message-tool sends before gateway/plugin/core delivery, without adding a new opt-out or config surface in this issue.
