## Summary

Keep open: current main still forwards visible message-tool send text without the responsePrefix cascade, while direct reply delivery resolves and applies that prefix. The recent follow-up comment describes the same operational gap, and targeted GitHub searches did not find a linked implementation PR or narrower canonical issue.

## What This Changes

Apply configured responsePrefix behavior to visible message-tool outbound sends so they match direct reply delivery.

## Best Possible Solution

Normalize visible message-tool send payloads through the same effective responsePrefix cascade before plugin/core/gateway delivery, with focused regression coverage and no new opt-out flag unless maintainers make that product decision separately.
