## Summary

Keep open: current main still applies the responsePrefix cascade on normal reply delivery but not on visible `message(action="send")` delivery, and the latest issue discussion confirms this remains an active WhatsApp/group workflow gap.

## What This Changes

Apply the configured `messages.responsePrefix` cascade to visible outbound sends created through the `message` tool.

## Best Possible Solution

Normalize visible, non-dry-run message-tool send payloads through the existing responsePrefix cascade and template resolver before plugin preparation/core delivery, with regression coverage for global, channel, account, template, and idempotency cases.

## Reproduction Assessment

Yes, source-reproducible on current main: the normal reply path calls `createReplyPrefixContext` and `normalizeReplyPayload`, while the message-tool send path forwards raw message text through `executeSendAction` and `sendMessage`. I did not run a live WhatsApp scenario because this review is read-only.

## Solution Assessment

Yes, applying the existing prefix normalization in the shared message-action send path is the narrow maintainable fix. A per-send opt-out should stay out of the repair unless maintainers make a separate product decision.
