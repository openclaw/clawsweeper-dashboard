## Summary

Keep open: current main still sends message-tool outbound text through a raw send path without applying the responsePrefix cascade, while direct replies already apply it. The related per-channel prefix work addressed #2316, but it does not cover this message tool path.

## What This Changes

Apply configured messages.responsePrefix to outbound sends made through the shared message tool, not only normal direct replies.

## Best Possible Solution

Normalize message-tool send text through the same responsePrefix cascade before any core, plugin, or gateway delivery, with regression coverage for global, channel, and account prefixes and no new suppression flag unless maintainers explicitly choose that product option.

## Reproduction Assessment

Yes, source-reproducible: message(action=send) reaches handleSendAction, executeSendAction, and sendMessage with raw text, while the direct reply path calls normalizeReplyPayload with responsePrefix. I did not run a live WhatsApp repro because this review was read-only.

## Solution Assessment

Yes, the requested direction is the narrow maintainable fix if implemented in the shared message-action send path rather than only in sendMessage. A separate suppress-prefix flag should stay out of the first fix unless maintainers make a product decision for it.
