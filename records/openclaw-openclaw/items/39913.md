## Summary

Keep open: current main still applies the response-prefix cascade on normal reply normalization, but visible `message(action="send")` payload construction still carries raw `message`/`payload.text` into plugin or core dispatch. The May 9 follow-up adds concrete WhatsApp provenance/model-attribution impact, and the related message-tool prefix PR was closed unmerged rather than superseding this issue.

## What This Changes

Apply the configured `messages.responsePrefix` cascade to visible `message(action="send")` outbound sends so message-tool delivery matches normal replies.

## Best Possible Solution

Normalize visible message-tool send payloads through the existing prefix cascade and template resolver before plugin, gateway, or core dispatch, while preserving no-prefix behavior, dry-run semantics, and double-prefix idempotency.

## Reproduction Assessment

Yes, source-reproducible. Direct replies pass through `createReplyPrefixContext` and `normalizeReplyPayload`, while `message(action="send")` builds raw `message` and `payload.text` without any prefix normalizer; no live WhatsApp send was run in this read-only review.

## Solution Assessment

Yes. The maintainable fix is to reuse the existing prefix cascade and reply normalizer at the shared visible message-tool send boundary; any per-send opt-out should remain a separate product decision.
