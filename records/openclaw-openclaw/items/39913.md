## Summary

Keep open: current main still applies the configured response prefix on the direct reply path, but the message-tool send path builds and dispatches raw message text without running the same prefix normalizer.

## What This Changes

Request that visible `message(action="send")` outbound sends respect the existing `messages.responsePrefix` cascade and template behavior.

## Best Possible Solution

Normalize visible message-tool send payloads through the existing reply-prefix context and template resolver at the shared outbound send boundary, while preserving dry-run behavior, no-prefix compatibility, and double-prefix idempotency.

## Reproduction Assessment

Yes, source-reproducible. Normal replies call `createReplyPrefixContext` and `normalizeReplyPayload`, while `message(action="send")` builds raw `message` and `payload.text` without that prefix normalizer; no live WhatsApp send was run.

## Solution Assessment

Yes. The narrow maintainable fix is to reuse the existing prefix cascade and reply normalizer at the shared visible message-tool send boundary; per-send opt-out should remain separate product work.
