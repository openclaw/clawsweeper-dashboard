## Summary

Keep open: current main still has separate direct-reply and message-tool send paths, and the message-tool path does not apply the configured outbound response prefix even though docs/tests define that prefix cascade for visible outbound replies.

## What This Changes

The issue requests automatic application of configured `messages.responsePrefix` to visible outbound sends made through the agent `message` tool.

## Best Possible Solution

Reuse the canonical effective-prefix cascade and template resolver for assistant-originated visible `message(action="send")` text after channel/account resolution, with double-prefix protection and no new config option.

## Reproduction Assessment

Yes. Current-main source shows direct replies pass through prefix-aware normalization, while `message(action="send")` builds and dispatches send text without any responsePrefix reference; no live WhatsApp run was performed in this read-only review.

## Solution Assessment

No to the suggested generic gateway-egress placement because it is broader than the documented assistant visible-reply contract. The maintainable fix is a focused message-tool send integration that reuses existing prefix cascade and template semantics.
