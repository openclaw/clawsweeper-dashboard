## Summary

Keep open: current main still has the reported inconsistency. Direct replies resolve and apply the effective response prefix cascade, but the message-tool send path still forwards the normalized tool text to gateway plugin dispatch, plugin fallback dispatch, and core send without a response-prefix step.

## What This Changes

The issue requests that `messages.responsePrefix` be applied automatically to outbound `message(action=send, ...)` sends, matching normal direct replies.

## Best Possible Solution

Add a shared message-tool send normalization step that resolves the effective prefix with agent, channel, and account context, applies it idempotently to textual sends before plugin/core delivery and mirroring, and leaves channel-specific policy out of core.

## Reproduction Assessment

Yes. A focused unit reproduction can call `runMessageAction` for `send` with `messages.responsePrefix` configured and mocked plugin/core delivery, then assert the delivered `params.message` or `sendMessage` content remains unprefixed on current main.

## Solution Assessment

Yes. Applying the prefix once in the shared `handleSendAction` boundary before gateway plugin dispatch, plugin fallback, core fallback, and mirroring is the narrowest maintainable fix; a WhatsApp-only patch would leave the shared outbound contract inconsistent.
