## Summary

Keep open: current main still applies `messages.responsePrefix` through the direct reply normalizer, while `message(action="send")` builds `message` and `payload.text` from raw tool input before plugin/core dispatch. The May 9 WhatsApp workflow comment and the closed, unmerged prior attempt at https://github.com/openclaw/openclaw/pull/9103 both support that the same visible attribution gap remains.

## What This Changes

The issue requests that `messages.responsePrefix` also be applied to user-visible outbound sends made through the `message` tool.

## Best Possible Solution

Normalize visible message-tool send payloads through the existing reply-prefix context and template resolver at the shared outbound send boundary, preserving dry-run behavior, no-prefix compatibility, and double-prefix idempotency.

## Reproduction Assessment

Yes, source-reproducible. Normal replies call `createReplyPrefixContext` and `normalizeReplyPayload`, while `message(action="send")` builds raw `message` and `payload.text` without that prefix normalizer; no live WhatsApp send was run.

## Solution Assessment

Yes. The narrow maintainable fix is to reuse the existing prefix cascade and reply normalizer at the shared visible message-tool send boundary; per-send opt-out should remain separate product work.
