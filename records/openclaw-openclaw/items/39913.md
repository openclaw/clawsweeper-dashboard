## Summary

Keep open: current main still sends message-tool outbound text through raw send paths while direct replies already apply the responsePrefix cascade. The related per-channel prefix work addressed #2316, but it does not cover this message-tool send path.

## What This Changes

Apply configured messages.responsePrefix to outbound sends initiated by the message tool so they match direct replies.

## Best Possible Solution

Normalize message-tool send text through the existing responsePrefix cascade before delivery, with regression coverage for global, channel, and account prefixes and no new suppression option unless maintainers choose that separately.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: message(action=send) reaches handleSendAction, executeSendAction, and sendMessage with raw text, while direct replies call normalizeReplyPayload with responsePrefix. I did not run a live WhatsApp send because this review is read-only.

## Solution Assessment

Yes, the narrow maintainable fix is to reuse the existing prefix cascade in the shared message-action send path rather than only in sendMessage. A suppress-prefix flag would be a separate product/config decision and should stay out of this bug fix.
