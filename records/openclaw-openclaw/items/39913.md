## Label Justifications

- `P2`: This is a source-proven user-visible messaging consistency bug with a clear but limited outbound delivery surface.
- `impact:message-loss`: The configured visible provenance prefix is silently omitted from one user-visible send path.

## Summary

Keep open: current main and the latest release still apply the configured response prefix on the normal reply path, but `message(action="send")` builds and dispatches raw visible text without the same prefix normalizer. A prior matching PR was closed unmerged, and no open linked PR currently owns the fix.

## What This Changes

The issue requests that configured response prefixes apply to visible outbound sends made through the `message` tool.

## Best Possible Solution

Normalize non-dry-run visible message-tool send payloads through the existing reply-prefix cascade and template resolver, while preserving no-prefix behavior, dry-run behavior, channel/account precedence, and double-prefix idempotency.

## Reproduction Assessment

Yes, source-reproducible. Normal replies call the reply-prefix context and `normalizeReplyPayload`, while message-tool sends build and dispatch raw `message`/`payload.text` without that normalizer; no live WhatsApp send was run in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is to reuse the existing prefix cascade and reply normalizer at the shared visible message-tool send boundary; a per-send opt-out can remain separate product work.
