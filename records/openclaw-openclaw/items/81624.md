## Summary

Close as duplicate of the open Feishu group `replies=0` tracking issue: the non-working accounts in this report reach dispatch but show no outbound send attempt, which matches the documented group visible-reply policy and the live confirmations already collected in the canonical issue rather than proving a separate account-specific transport failure.

## What This Changes

The issue reports that in Feishu WebSocket multi-account group chats, only one configured account sends group replies while other accounts dispatch but finish with `replies=0`.

## Best Possible Solution

Keep the remaining Feishu group `replies=0` behavior centralized in https://github.com/openclaw/openclaw/issues/77666, and use the existing `messages.groupChat.visibleReplies: "automatic"` workaround or message-tool delivery while maintainers decide the compatibility path.

## Reproduction Assessment

Yes for the central `replies=0`/no outbound-action symptom by source and test inspection: default group turns suppress ordinary final delivery unless the message tool is used or `messages.groupChat.visibleReplies` is `automatic`. I did not run a live five-account Feishu setup, so a distinct account-specific send failure is not reproduced.

## Solution Assessment

Yes: closing this as a duplicate is the narrowest cleanup path because the canonical open issue already tracks the remaining Feishu group visible-reply behavior and verified workaround. A separate fix path should require logs showing an actual per-account send attempt failing after `handleAction` or `deliverMessage`.
