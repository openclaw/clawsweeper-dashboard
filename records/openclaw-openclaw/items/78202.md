## Summary

Current main confirms that local Codex app-server launches intentionally isolate `CODEX_HOME` and `HOME`, and docs now mention that boundary, but the requested operator-home mitigation is not fully implemented and needs maintainer product/security judgment before changing the harness contract.

## What This Changes

Request to preserve operator-home semantics or add explicit documentation, warnings, and an operator-home escape hatch for Codex harness sessions.

## Best Possible Solution

Keep isolation as the default unless maintainers approve a safer operator-home affordance, then pair that implementation with explicit docs and runbook guidance about `~` inside Codex harness sessions.

## Reproduction Assessment

Yes, from source inspection: current main deterministically sets `HOME` to the per-agent Codex native home for local stdio starts, and tests assert that contract. I did not perform a live macOS harness run during this read-only review.

## Solution Assessment

Unclear as proposed: preserving operator `HOME` would conflict with the documented isolation boundary. The safer direction is maintainer-approved explicit operator-home exposure or stronger docs/warnings, not an automatic change to `HOME`.
