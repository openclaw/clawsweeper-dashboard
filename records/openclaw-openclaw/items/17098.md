## Summary

Keep open. Current main still lacks xCloud hosting documentation, but the PR head is not merge-ready because it still mislabels Claude Code as an AI provider and leaves the hosted Control UI auth/origin/device-pairing path unspecified for a provider-managed admin surface.

## What This Changes

This PR adds an xCloud managed-VPS hosting guide, xCloud redirects/navigation under docs install hosting, and an xCloud card on the VPS provider page.

## Best Possible Solution

Publish an xCloud guide only after a maintainer or xCloud/provider owner verifies the managed deployment against current Gateway and Control UI contracts, then revise the guide to document the exact auth mode, allowed origins, device-pairing or trusted-proxy/Tailscale path, Telegram allowlist behavior, credential handling, and any root-user tradeoffs.

## Reproduction Assessment

Not applicable as a docs feature PR rather than a runtime bug. The high-confidence verification path is docs/source inspection: current main lacks xCloud coverage, while the PR head adds the page and exposes the provider wording and hosted-Control-UI gaps.

## Solution Assessment

No, this is not the best way to solve the docs gap as-is. The docs surface is plausible, but official docs should not endorse the hosted xCloud flow until the provider-specific auth, origin, device-pairing, and credential behavior is verified and documented.
