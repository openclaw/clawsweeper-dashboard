## Summary

Keep this PR open. Current main still lacks xCloud hosting coverage, but the PR head is not merge-ready because it still mislabels Claude Code as an AI provider and directs users to a hosted Control UI Basic Auth flow without documenting how xCloud satisfies current Gateway auth, allowed origins, and device-pairing or trusted-proxy requirements.

## What This Changes

The PR adds a new xCloud managed-VPS hosting guide, xCloud redirects and navigation under the install hosting docs, and an xCloud card on the VPS provider page.

## Best Possible Solution

Publish an xCloud guide only after a maintainer or xCloud/provider owner verifies the managed deployment against current OpenClaw Gateway and Control UI contracts, then revise the page to document the exact auth mode, allowed origins, device-pairing or trusted-proxy/Tailscale path, Telegram allowlist behavior, credential handling, and root-user tradeoffs.

## Reproduction Assessment

Not applicable: this is a docs feature PR rather than a runtime bug. The high-confidence verification path is source/docs inspection: current main has no xCloud docs, while the PR head's added page exposes the provider wording and hosted-Control-UI gaps.

## Solution Assessment

No, the PR is not the best solution as-is. A provider-specific hosting page is plausible, but official docs should not endorse the hosted xCloud flow until its auth, origin, device-pairing, credential, and allowlist behavior is verified and documented.
