## Summary

Keep open. Current main still lacks xCloud hosting coverage, so this PR is not obsolete, but the current PR head is not merge-ready because it still mislabels Claude Code as a provider and does not verify or document the hosted Control UI auth, origin, and device-pairing path.

## What This Changes

The PR adds an xCloud managed-VPS hosting guide, xCloud docs navigation and redirects, and an xCloud card on the VPS provider page.

## Best Possible Solution

Publish an xCloud guide only after a maintainer or xCloud/provider owner verifies the managed deployment against current OpenClaw Gateway and Control UI contracts, then revise the page to document the exact auth mode, allowed origins, device-pairing or trusted-proxy/Tailscale path, Telegram allowlist behavior, credential handling, and root-user tradeoffs.

## Reproduction Assessment

Not applicable as a docs feature PR rather than a runtime bug. The high-confidence verification path is source/docs inspection: current main lacks xCloud coverage, while the PR head adds the page and exposes the provider wording and hosted-Control-UI gaps.

## Solution Assessment

No, the PR is not the best solution as-is. Adding a provider-specific hosting page is plausible, but official docs should not endorse the hosted xCloud flow until the auth, origin, device-pairing, credential, and allowlist behavior is verified and documented.
