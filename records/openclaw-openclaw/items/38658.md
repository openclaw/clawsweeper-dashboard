## Summary

Keep open. The requested Slack context-retention behavior is still not implemented on current main, but this PR needs maintainer judgment because it changes the documented pending-only group-history contract, is stale against the Slack plugin move, and still lacks after-fix real Slack proof.

## What This Changes

The branch threads `requireMention` through Slack prepared messages, records processed `requireMention=false` room messages into the history window in reply and no-reply paths, updates Slack history tests, and adds a changelog entry.

## Best Possible Solution

If maintainers want always-on Slack channels to carry recent processed channel context, implement it in `extensions/slack` with ingress-time or timestamp-ordered retention, update docs/tests/changelog together, and require redacted live Slack proof.

## Reproduction Assessment

Yes, via source inspection: current main clears Slack room history after processed turns and the existing Slack tool-result test asserts a second `requireMention=false` channel prompt excludes the first message. I did not run a live Slack workspace repro.

## Solution Assessment

No. The direction may be useful, but the proposed implementation writes retained history after async dispatch and conflicts with the current pending-only group-history docs; ingress-time retention or timestamp-sorted formatting after maintainer approval is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Preserve ingress order for retained Slack history:** `src/slack/monitor/message-handler/dispatch.ts:550`
  - body: The PR appends `requireMention=false` messages only after dispatch finishes. Top-level Slack channel messages are debounced by their own timestamp, so overlapping turns can complete out of order; `buildPendingHistoryContextFromMap` then emits insertion order and can feed the next prompt reversed context. Record at ingress or sort by Slack timestamp before formatting.
  - confidence: 0.82
