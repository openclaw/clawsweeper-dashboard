## Summary

Keep open: the PR targets a real current-main Slack behavior, but it changes the documented pending-only group-history contract, the branch is stale/conflicting after Slack moved into `extensions/slack`, it still needs real Slack proof, and the proposed post-dispatch retention can reorder retained context under overlapping channel turns.

## What This Changes

The PR threads `requireMention` through prepared Slack messages and records processed `requireMention=false` channel messages into the history window instead of clearing it after dispatch.

## Best Possible Solution

If maintainers want always-on Slack channels to carry processed channel context, implement it in `extensions/slack` with ingress-time or timestamp-ordered retention, update the group-history docs/tests/changelog together, and require redacted live Slack proof.

## Reproduction Assessment

Yes, via source inspection: current main clears Slack room history after processed room turns, and the existing Slack tool-result test asserts a second `requireMention=false` channel prompt excludes the first message. I did not run a live Slack workspace reproduction.

## Solution Assessment

No. The direction may be useful, but this implementation records retained history after async dispatch and conflicts with current pending-only group-history docs; ingress-time retention or timestamp-sorted formatting after maintainer approval is safer.

## Review Findings

- **[P2] Preserve ingress order for retained Slack history:** `src/slack/monitor/message-handler/dispatch.ts:550`
