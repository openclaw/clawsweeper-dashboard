## Summary

Keep open: current main still has the Slack context-loss behavior this PR targets, and the contributor addressed the earlier no-reply gap, but the branch should not merge as-is because it changes the documented pending-only group-history contract and records retained messages only after dispatch, which can miss or reorder context under overlapping channel turns.

## What This Changes

The PR threads `requireMention` into Slack prepared messages, records processed `requireMention=false` room messages into sliding history after no-reply and reply paths, updates Slack history tests, and adds a changelog entry.

## Best Possible Solution

Decide the group-history product contract first; if always-on Slack channels should retain recent context, rebase the change into `extensions/slack`, record accepted `requireMention=false` room messages before async dispatch or timestamp-sort retained history, and update docs, tests, and changelog together.

## Reproduction Assessment

Yes. Source inspection plus `extensions/slack/src/monitor.tool-result.test.ts` gives a high-confidence current-main path: two `requireMention=false` Slack channel messages are processed, and the second prompt is expected not to include the first message as retained history.

## Solution Assessment

No. The PR targets a real context gap, but post-dispatch recording is not the safest implementation and the docs still define history as pending-only; ingress-time recording or timestamp-ordered formatting after a maintainer product decision would be narrower and more maintainable.

## Review Findings
