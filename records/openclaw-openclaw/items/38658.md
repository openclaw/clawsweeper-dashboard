## Summary

Keep open: the PR addresses a real Slack context gap, but it changes the documented pending-only group-history contract, is stale against the Slack extension migration, lacks real Slack workspace proof, and records retained history after async dispatch where overlapping turns can be replayed out of order.

## What This Changes

The PR threads `requireMention` through Slack prepared messages, records processed `requireMention=false` room messages into the history window in both reply and no-reply dispatch paths, updates one Slack tool-result expectation, adds dispatch-history tests, and adds a changelog entry.

## Best Possible Solution

Decide the group-history contract first; if approved, rebase the change into `extensions/slack`, preserve chronological order by recording at ingress or sorting by Slack timestamp, update docs/tests/changelog together, and verify it in a real Slack workspace.

## Reproduction Assessment

Yes. Source inspection on current main shows accepted `requireMention=false` Slack channel messages are not retained for the next prompt, and the existing Slack tool-result test asserts that no processed room history is injected.

## Solution Assessment

No. The direction may be useful, but the proposed implementation writes history after async dispatch and leaves the pending-only docs contract unresolved; ingress-time retention or timestamp-ordered formatting after maintainer approval is safer.
