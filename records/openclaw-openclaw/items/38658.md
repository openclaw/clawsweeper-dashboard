## Summary

Keep open: current main still has the reported Slack history behavior, but this PR changes the documented pending-only group-history contract, has a concrete ordering concern, is stale against the Slack plugin move, and lacks real Slack workspace proof.

## What This Changes

The PR threads `requireMention` through Slack prepared messages and changes `requireMention=false` channel turns to record processed messages into the Slack history window instead of clearing it, with tests and a changelog entry.

## Best Possible Solution

Decide whether always-on Slack channels should retain processed channel context; if yes, rebase into `extensions/slack` with ingress-time or timestamp-ordered retention, docs/tests/changelog updates, and redacted Slack workspace proof.

## Reproduction Assessment

Yes for the current behavior: source inspection shows current main clears Slack room history after processed room turns, and the existing Slack test asserts a second `requireMention:false` channel prompt excludes the first message. I did not run a live Slack workspace repro.

## Solution Assessment

No. The direction may be useful, but this implementation writes retained history after async dispatch and conflicts with current pending-only group-history docs; ingress-time retention or timestamp-sorted formatting after maintainer approval is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Preserve ingress order for retained Slack history:** `src/slack/monitor/message-handler/dispatch.ts:539`
  - body: The PR appends `requireMention=false` messages only after dispatch finishes. Top-level Slack channel messages are keyed by their own timestamp, so overlapping turns can complete out of order; `buildPendingHistoryContextFromMap` then emits insertion order and can feed the next prompt reversed context. Record at ingress or sort by Slack timestamp before formatting.
