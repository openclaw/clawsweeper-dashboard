## Summary

Keep this PR open for maintainer review. Current main still has the reported Slack context-loss behavior, but this PR is not merge-ready because it changes the documented pending-only group history contract, needs rebasing into the current `extensions/slack` plugin layout, and records retained history after dispatch where concurrent Slack channel turns can be appended out of ingress order.

## What This Changes

The PR threads `requireMention` into Slack prepared messages, records processed `requireMention=false` room messages into the sliding channel history window instead of clearing it, updates Slack history tests, and adds a changelog entry.

## Best Possible Solution

If maintainers want always-on Slack channels to preserve recent conversation context, implement it in the current Slack plugin paths by recording processed room messages at ingress/prepare time or by timestamp-sorting retained history before prompt formatting, then update Slack/groups docs and tests to make the new non-pending history contract explicit. If maintainers prefer the current pending-only model, close this PR with that product rationale.

## Reproduction Assessment

Yes. Current main has an executable regression path in `extensions/slack/src/monitor.tool-result.test.ts`: two `requireMention=false` channel messages are processed, and the second prompt is expected not to contain the first message or history marker.

## Solution Assessment

No. The PR targets a real behavior gap, but recording processed messages only after dispatch is not the narrowest maintainable fix because it can miss the next prompt or reorder retained context under concurrent Slack channel turns; the safer approach is ingress-time recording or timestamp-ordered formatting in the current plugin layout.
