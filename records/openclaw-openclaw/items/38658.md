## Summary

Keep open: current main still has the observable Slack context behavior, but this PR changes documented pending-only group history semantics and has a completion-order history bug, so it needs maintainer product review and revision rather than cleanup closure.

## What This Changes

The branch threads `requireMention` into Slack prepared messages and records processed `requireMention=false` room messages into sliding history after no-reply and reply dispatch paths, with Slack tests and a changelog entry.

## Best Possible Solution

Decide the Slack/group-history contract first; if accepted, implement it in `extensions/slack` with ingress-ordered or timestamp-sorted retention and update docs, tests, and changelog together.

## Reproduction Assessment

Yes for the observable behavior: source inspection and the current Slack monitor test show that two `requireMention=false` channel messages do not carry the first message into the second prompt. That reproduces the context gap, but current docs/tests define it as pending-only behavior rather than an established bug contract.

## Solution Assessment

No. The PR targets useful behavior, but post-dispatch recording can store history in completion order and the docs still describe pending-only group history; ingress-time retention or timestamp-sorted formatting after maintainer approval is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Preserve ingress order when retaining Slack history:** `src/slack/monitor/message-handler/dispatch.ts:550`
  - body: Recording the message only after dispatch completes can store concurrent top-level Slack turns in completion order instead of arrival order. Since history formatting replays insertion order, a slower earlier turn can appear after a later one; record before async dispatch or sort retained entries by Slack timestamp before building prompt context.
  - confidence: 0.82
