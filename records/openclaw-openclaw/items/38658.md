## Summary

Keep open. Current main still does not implement the PR's intended always-on Slack channel history retention: Slack room history is still pending-only, dispatch clears room history after processed turns, and the current regression test still asserts that a second `requireMention=false` channel message does not see the first as history. The PR remains useful source work, but it is stale against the current `extensions/slack` plugin layout and still needs maintainer product review around ordering and prompt-history semantics.

## What This Changes

The PR modifies the old Slack monitor message-handler paths to carry `requireMention`, record processed room messages in history when mentions are disabled, adjust Slack history tests, and add a changelog note.

## Best Possible Solution

Keep this PR open for maintainer review. If the desired product behavior is retained context for always-on Slack channels, rebase the change into `extensions/slack/src/monitor/message-handler/*`, record processed `requireMention=false` room messages at ingress/prepare time or otherwise preserve Slack timestamp order, cover reply and no-reply outcomes in current Slack tests, and update the Groups/Slack docs away from the pending-only model. If maintainers prefer pending-only semantics, close the PR with that explicit rationale.

## Reproduction Assessment

Yes. Current main has an executable regression path in `extensions/slack/src/monitor.tool-result.test.ts`: two `requireMention=false` channel messages are processed, and the second prompt is expected not to contain the first message or history marker.

## Solution Assessment

No. The PR points in a plausible direction for the reported sequential context loss, but it is not the best current solution because it records after dispatch, can miss or reorder concurrent channel turns, and has not been rebased into the current plugin layout or reconciled with the documented pending-only history contract.

- **[P2] Preserve ingress order before recording Slack history:** `src/slack/monitor/message-handler/dispatch.ts:550`
