## Summary

Keep open: current main still has the reported Slack behavior, but this PR is not merge-ready because it changes the documented pending-only group-history contract, is stale against the Slack plugin migration, lacks real Slack workspace proof, and has a concrete ordering bug in the proposed retention path.

## What This Changes

The PR threads `requireMention` through Slack prepared messages, changes `requireMention=false` room dispatch to retain processed messages in group history, updates Slack history tests, and adds a changelog entry.

## Best Possible Solution

Decide the group-history contract first; if approved, rebase into `extensions/slack`, retain messages in chronological order, update docs/tests/changelog together, and verify in a real Slack workspace.

## Reproduction Assessment

Yes. Source inspection on current main shows accepted `requireMention=false` Slack channel messages are not retained for the next prompt, and the existing Slack tool-result test asserts that processed room history is not injected.

## Solution Assessment

No. The direction may be useful, but the proposed implementation writes retained history after async dispatch and leaves the pending-only docs contract unresolved; ingress-time retention or timestamp-ordered formatting after maintainer approval is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Preserve ingress order for retained Slack history:** `src/slack/monitor/message-handler/dispatch.ts:550`
  - body: The PR appends `requireMention=false` messages to `channelHistories` only after dispatch completes. Top-level Slack messages use timestamp-specific debounce keys and independent flushes, while history formatting replays insertion order, so a slower earlier turn can be stored after a faster later turn and give the next prompt reversed context. Record at ingress or sort retained entries by Slack timestamp before formatting.
  - confidence: 0.82
