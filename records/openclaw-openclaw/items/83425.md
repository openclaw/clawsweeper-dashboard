## Summary

Close as implemented: current main now offers a provider-owned xAI device-code OAuth path that avoids the localhost redirect URI entirely, is documented for VPS/SSH setups, has focused tests, and shipped in v2026.5.19-beta.2. The browser loopback OAuth path still exists, but the reported user goal of authenticating SuperGrok without XAI_API_KEY is covered by the shipped redirect-free OAuth flow.

## What This Changes

The issue reports xAI browser OAuth failing with a redirect_uri mismatch while trying to use a SuperGrok subscription instead of XAI_API_KEY.

## Best Possible Solution

Keep the provider-owned device-code OAuth path as the supported redirect-free xAI login option, and treat any remaining browser callback registration work as separate xAI provider or OAuth app administration follow-up.

## Reproduction Assessment

Yes: the issue body and independent comment include the full generated authorize URL and live xAI error, and source inspection shows the browser OAuth path still builds that localhost redirect. I did not rerun the live browser failure because current main now provides a redirect-free OAuth path for the affected setup.

## Solution Assessment

Yes: the merged device-code method is the narrow maintainable OpenClaw-side solution for remote/VPS users because it stays in the xAI plugin, uses the existing generic --device-code CLI seam, and avoids depending on localhost redirect registration.
