## Label Justifications

- `P1`: The documented xAI OAuth path is reported broken for SuperGrok users and blocks provider authentication, even though API-key auth remains available.
- `impact:auth-provider`: The issue is specifically about xAI OAuth redirect registration and provider auth flow behavior.

## Summary

Keep open: current main and the latest release still generate and document the exact xAI OAuth localhost redirect URI reported by users, while the related token-exchange fix does not address the earlier authorization-step redirect mismatch.

## What This Changes

The issue asks OpenClaw to make the documented xAI OAuth login path work when xAI rejects the fixed localhost callback redirect URI.

## Best Possible Solution

Make xAI OAuth use a redirect URI registered for OpenClaw's public client or move the xAI auth method to xAI's advertised device-code flow, then update docs/tests and verify with a real SuperGrok login.

## Reproduction Assessment

Partially: current main deterministically generates the exact authorize URL and localhost redirect from the reports, and two users report the same provider-side error. I did not complete a live xAI account authorization in this read-only review, so the provider rejection itself remains source-and-discussion backed rather than locally reproduced.

## Solution Assessment

No: the current localhost-callback implementation and the already-landed PKCE token-body fix do not solve an authorization-step redirect mismatch. A maintainer needs to choose between external OAuth app registration repair and a device-code flow before a code PR is the right next step.
