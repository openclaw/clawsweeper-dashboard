## Summary

Keep open: current main still emits the exact xAI OAuth localhost redirect URI the reporter pasted, and the provider docs plus plugin metadata still advertise this OAuth path. Related PRs either introduced/fixed adjacent xAI OAuth behavior or were closed without changing this redirect, so a maintainer/provider-owner decision is still needed.

## What This Changes

The issue reports that xAI OAuth login for SuperGrok on OpenClaw 2026.5.16-beta.6 generates a localhost callback redirect URI that xAI rejects as unregistered.

## Best Possible Solution

Make xAI OAuth use a redirect URI registered for OpenClaw's xAI client or move the xAI auth method to a supported device-code flow, then update docs/tests and verify with a real eligible SuperGrok login.

## Reproduction Assessment

No high-confidence live reproduction yet: source and tests reproduce the exact authorize URL and redirect URI, but I did not complete an xAI browser-login flow that reaches the provider error. The reporter's pasted URL is consistent with current main.

## Solution Assessment

No: the current localhost-callback-only implementation is not enough if xAI rejects that callback for the shared client. The maintainable fix needs a maintainer choice between correcting the external OAuth app registration and changing the provider-owned auth flow.
