## Summary

Keep open. The PR addresses a real Control UI attachment gap that current main still has, but the proposed patch is not ready because it still rewrites external or signed attachment URLs while adding download handling, and it lacks the required user-facing changelog entry.

## What This Changes

The PR normalizes assistant `mediaUrl`/`mediaUrls` payload fields into Control UI attachment cards, adds download links for non-image assistant attachments, and sets attachment `Content-Disposition` for local assistant-media downloads with focused Gateway and UI tests.

## Best Possible Solution

Land a narrow revision that keeps external, provider-hosted, signed, and data attachment URLs byte-for-byte unchanged, adds `download=1` only for OpenClaw-controlled media routes that can honor it server-side, keeps the authenticated markdown attachment behavior, adds regression coverage for signed external URLs, and includes the required changelog credit.

## Reproduction Assessment

Yes. A code-level reproduction is an assistant document attachment with an external signed URL such as `MEDIA:https://example.com/files/report.pdf?sig=1`; the PR routes the resulting href through the download URL builder and changes it to include `download=1`.

## Solution Assessment

No. The overall direction is right for local assistant-media, but the current patch applies URL mutation too broadly; the maintainable fix is to use `download` without changing external URLs and reserve server-recognized query parameters for OpenClaw-owned routes.
