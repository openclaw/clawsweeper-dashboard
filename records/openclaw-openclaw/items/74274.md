## Summary

Keep this PR open. Current main still lacks the requested assistant attachment download behavior, but the proposed patch is not merge-ready because it rewrites external non-image attachment URLs and still lacks the required user-facing changelog entry.

## What This Changes

The PR renders assistant `mediaUrl`/`mediaUrls` payloads as Control UI attachment cards, adds download links for non-image assistant attachments, and sets attachment disposition for local assistant-media downloads.

## Best Possible Solution

Land a revised patch that keeps local assistant-media download handling, only adds download parameters to OpenClaw-controlled media routes, preserves external/data/provider URLs exactly, adds a signed external URL regression test, and records the fix in the active changelog.

## Reproduction Assessment

Yes. A code-level reproduction is an assistant document attachment such as `MEDIA:https://example.com/files/report.pdf?sig=1`: current main preserves that href, while this PR builds the non-image download href by appending `download=1`.

## Solution Assessment

No. The local assistant-media download direction is maintainable, but the safer fix is to scope server-recognized download parameters to OpenClaw-owned media URLs and leave external hrefs unchanged.
