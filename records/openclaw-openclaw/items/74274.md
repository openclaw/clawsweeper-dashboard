## Summary

Keep this PR open. Current main does not include the intended assistant markdown/download behavior, and the latest head appears to address the earlier bot comments, but the patch is still not merge-ready because it mutates external non-image attachment URLs and lacks the required changelog entry.

## What This Changes

The branch normalizes assistant `mediaUrl`/`mediaUrls` into Control UI attachment cards, adds download links for non-image assistant attachments, and sets attachment `Content-Disposition` for local assistant-media downloads.

## Best Possible Solution

Land a narrow revision that uses server-honored `download=1` only for OpenClaw-owned assistant-media routes, leaves external/provider-hosted/signed/data hrefs byte-for-byte unchanged, keeps the authenticated markdown attachment behavior, adds regression coverage for external document URLs, and includes the required changelog credit.

## Reproduction Assessment

Yes. A code-level reproduction is an assistant document attachment such as `MEDIA:https://example.com/files/report.pdf?sig=1`; the PR routes the rendered href through the download URL builder and would append `download=1`.

## Solution Assessment

No. The local assistant-media direction is maintainable, but applying query mutation to every non-image attachment is too broad; the safer solution is to rely on the browser `download` attribute for external URLs and only add query parameters to routes OpenClaw controls.
