## Summary

Keep open. Current main still has the Control UI attachment gap this PR targets, but the proposed patch is not ready because it mutates external or signed non-image attachment URLs while adding download handling, and it still lacks the required user-facing changelog entry.

## What This Changes

The PR normalizes assistant `mediaUrl`/`mediaUrls` payload fields into Control UI attachment cards, adds non-image attachment download links, and sets attachment `Content-Disposition` for local assistant-media downloads with focused Gateway and UI tests.

## Best Possible Solution

Land a narrow revision that leaves external, provider-hosted, signed, and data attachment URLs byte-for-byte unchanged, reserves `download=1` for OpenClaw-controlled routes that can honor it server-side, keeps authenticated markdown attachment behavior, adds signed-external-URL regression coverage, and includes the required changelog credit.

## Reproduction Assessment

Yes. A code-level reproduction is an assistant document attachment such as `MEDIA:https://example.com/files/report.pdf?sig=1`; the PR routes the rendered href through the download URL builder and changes it by appending `download=1`.

## Solution Assessment

No. The direction is right for local assistant-media, but the current patch applies URL mutation too broadly; the safer fix is to rely on the browser `download` attribute for external URLs and only add server-recognized query parameters to OpenClaw-owned media routes.
