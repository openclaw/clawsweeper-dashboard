## Summary

Keep this PR open. It addresses a real Control UI attachment-download gap and the latest patch appears to have addressed the earlier bot comments about assistant-only normalization and server-side `download=1`, but it is not merge-ready because it still mutates external or signed attachment URLs and lacks the required user-facing changelog entry.

## What This Changes

The PR renders assistant `mediaUrl`/`mediaUrls` fields as Control UI attachment cards, adds download links for non-image assistant attachments, and sets attachment `Content-Disposition` for local assistant-media downloads with focused Gateway and UI tests.

## Best Possible Solution

Land a revised patch that keeps the local assistant-media download behavior, preserves provider-hosted, signed, external, and data URLs byte-for-byte, adds regression coverage for signed external document URLs, and records the user-facing Control UI fix in the active changelog.

## Reproduction Assessment

Yes. A code-level reproduction is an assistant document attachment such as `MEDIA:https://example.com/files/report.pdf?sig=1`; current main treats it as a valid attachment and preserves the URL, while the PR routes the href through a download builder that appends `download=1`.

## Solution Assessment

No. The local assistant-media direction is good, but the PR applies URL mutation too broadly; the safer solution is to add server-recognized query parameters only to OpenClaw-owned media routes and rely on the browser `download` attribute without changing external hrefs.
