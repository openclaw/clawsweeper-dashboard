Keep this PR open. The latest patch addresses the earlier assistant-only normalization and server-side `download=1` comments, but it is still not merge-ready because it mutates external/data attachment URLs and lacks the required changelog entry.

## What This Changes

The PR adds assistant `mediaUrl`/`mediaUrls` normalization, download links for non-image assistant attachments, `Content-Disposition` handling for assistant-media downloads, and focused UI/Gateway tests.

## Best Possible Solution

Land a revised patch that scopes `download=1` to OpenClaw-owned media routes, preserves external and data URLs byte-for-byte, adds signed/external URL regression coverage, and records the user-facing fix in `CHANGELOG.md`.

## Reproduction Assessment

Yes. Render an assistant document attachment such as `MEDIA:https://example.com/files/report.pdf?sig=1`; current main preserves that href, while this PR appends `download=1` to the non-image download link.

## Solution Assessment

No. The local assistant-media `Content-Disposition` direction is appropriate, but mutating every non-image URL is broader than needed; the safer fix is to only add query parameters to same-origin OpenClaw-controlled media routes.
