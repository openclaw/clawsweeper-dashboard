## Summary

Keep open. The PR is close, but the latest patch still mutates external/provider-hosted non-image attachment URLs by appending `download=1`, and it is missing the required user-facing changelog entry.

## What This Changes

The PR renders assistant `mediaUrl`/`mediaUrls` as Control UI attachment cards and changes local assistant-media responses/links so markdown and other non-image files download instead of opening inline.

## Best Possible Solution

Land a revised patch that keeps local assistant-media download handling, only adds download parameters to OpenClaw-owned media routes, preserves external/data/provider URLs exactly, adds a signed external URL regression test, and records the fix in the active changelog.

## Reproduction Assessment

Yes. A code-level reproduction is an assistant document attachment such as `MEDIA:https://example.com/files/report.pdf?sig=1`: current main preserves that href, while this PR builds the non-image download href by appending `download=1`.

## Solution Assessment

No. The local assistant-media download direction is maintainable, but the safer fix is to scope server-recognized download parameters to OpenClaw-controlled media URLs and leave external hrefs unchanged while relying on the browser `download` attribute where applicable.
