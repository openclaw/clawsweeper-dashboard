## Summary

This PR is a plausible security hardening direction, but it is not merge-ready: the proposed hostname checks leave a mapped-IPv6 metadata bypass, reject documented private gateway hostnames, lack real CLI behavior proof, and the branch is currently dirty against main.

## What This Changes

The PR adds WebSocket target hostname/IP filtering to `gateway-status` URL normalization and adds a new security test file for blocked and allowed gateway probe URLs.

## Best Possible Solution

Use gateway-status-specific validation backed by the shared link-local/cloud-metadata/special-use IP classifiers, preserve documented private gateway URLs unless the compatibility contract is intentionally changed, then add mapped-IPv6 coverage and redacted CLI proof.

## Solution Assessment

No. The hardening direction is valid, but the implementation should use the existing shared IP classifiers and make the gateway hostname compatibility policy explicit instead of applying the HTTP SSRF hostname deny-list to gateway probe names.
