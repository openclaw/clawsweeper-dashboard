## Summary

Keep this PR open, but do not merge it as-is. The hardening targets a real current-main gap in gateway-status probing, and the discussion already identified blocking security and compatibility issues that remain in the proposed patch.

## What This Changes

The PR branch adds hostname/IP validation to gateway-status WebSocket target normalization and a new security test file covering blocked and allowed probe URLs.

## Best Possible Solution

Keep the hardening direction, but implement a gateway-specific target policy that uses shared IP/SSRF parsing for literal addresses, rejects metadata/link-local/wildcard targets including mapped IPv6 forms, and preserves documented `.local` and private-LAN gateway semantics.

## Reproduction Assessment

Yes. The bypass is reproducible with Node URL canonicalization, and current main shows normalized explicit/configured targets flowing into the gateway probe path.

## Solution Assessment

No. String-prefix hostname checks plus the generic HTTP SSRF hostname block are not the narrowest maintainable gateway-status policy; shared IP classification with gateway-specific route rules is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Reject IPv4-mapped metadata addresses:** `src/commands/gateway-status/helpers.ts:103-106`
  - body: The new guard only checks dotted `169.254.` strings and `0.0.0.0`. Node canonicalizes `ws://[::ffff:169.254.169.254]:80` to an IPv6 hostname like `::ffff:a9fe:a9fe`, so the metadata endpoint bypasses this SSRF mitigation. Use the shared IP parser/classifier and add IPv4-mapped and hex-form tests.
  - confidence: 0.94
- **[P2] Preserve documented `.local` gateway targets:** `src/commands/gateway-status/helpers.ts:110-112`
  - body: `isBlockedHostname()` rejects `.local` for both `ws://` and `wss://`, but OpenClaw documents `.local` gateway routes for private LAN/Bonjour flows with TLS or explicit private-network opt-in. This would drop legitimate targets such as `wss://gateway.local:18789` instead of probing them.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The diff has no dependency, CI, or supply-chain changes, but the SSRF mitigation is incomplete for IPv4-mapped IPv6 metadata targets.

Concerns:

- **[high] IPv4-mapped IPv6 metadata bypass:** `src/commands/gateway-status/helpers.ts:103`
  - body: The PR blocks link-local metadata only through a dotted `169.254.` hostname prefix. Parsed IPv4-mapped IPv6 forms canonicalize to hexadecimal IPv6 hostnames and can represent the same metadata endpoint, so the CWE-918 mitigation remains bypassable.
  - confidence: 0.94
