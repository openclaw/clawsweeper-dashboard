## Summary

Keep open: current main still lacks gateway-status target hardening, but this PR is not mergeable as-is because the proposed checks leave a mapped-IPv6 metadata bypass, reject documented gateway hostnames, and lack contributor real behavior proof.

## What This Changes

The PR adds gateway-status WebSocket URL hostname filtering plus a new security test file for metadata, wildcard, blocked-hostname, localhost, LAN, and public-host cases.

## Best Possible Solution

Implement gateway-specific URL validation that reuses the shared IP-literal classifier for metadata, link-local, wildcard, and malformed IPs while preserving documented gateway hostnames.

## Reproduction Assessment

Yes. Source inspection shows current main accepts normalized ws(s) explicit/config URLs and passes them to probeGateway(), and a focused Node URL check reproduces the mapped-IPv6 bypass in the PR's string policy.

## Solution Assessment

No. The PR targets the right gateway-status surface, but the implementation should be replaced with a gateway-specific policy built on the shared IP classifier instead of broad hostname suffix denial.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Reject mapped IPv6 metadata addresses:** `src/commands/gateway-status/helpers.ts:103-106`
  - body: The new guard only blocks 0.0.0.0 and hostnames starting with 169.254., so an explicit URL like `ws://[::ffff:169.254.169.254]:80` canonicalizes to an IPv6 hostname that bypasses this branch. That leaves the cloud metadata endpoint reachable despite the SSRF mitigation.
  - confidence: 0.94
- **[P2] Preserve documented gateway hostnames:** `src/commands/gateway-status/helpers.ts:110-112`
  - body: Calling `isBlockedHostname()` rejects `.local` and `.internal`, but OpenClaw documents Bonjour `.local` gateway hosts and wide-area domains such as `openclaw.internal.`. This would silently drop legitimate explicit or configured gateway-status probes.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The SSRF patch is incomplete because mapped IPv6 literals can still represent the metadata endpoint without matching the new string checks.

Concerns:

- **[high] IPv4-mapped IPv6 metadata bypass:** `src/commands/gateway-status/helpers.ts:103`
  - body: The PR blocks link-local metadata only through a dotted `169.254.` hostname prefix. Canonical IPv4-mapped IPv6 forms can represent the same metadata endpoint without matching that prefix, leaving the CWE-918 mitigation incomplete.
