## Summary

Keep open: this PR targets a real gateway-status SSRF surface, but the proposed patch is incomplete because mapped IPv6 metadata addresses bypass the new string checks, the `.local`/`.internal` block conflicts with documented gateway routes, and the contributor proof is still test output only.

## What This Changes

The PR adds gateway-status WebSocket URL hostname filtering plus a new helper security test file for metadata, wildcard, blocked-hostname, localhost, LAN, and public targets.

## Best Possible Solution

Use a gateway-specific URL policy built on the shared special-use IP classifier, blocking metadata/link-local/wildcard/malformed IPs while preserving documented LAN, Bonjour, and tailnet gateway hostnames.

## Reproduction Assessment

Yes for source-level review: current main accepts explicit/configured `ws(s)` gateway-status URLs and passes them to `probeGateway()`, and a focused check shows the PR's string policy still allows mapped IPv6 metadata literals. I did not run a live `openclaw gateway-status` probe.

## Solution Assessment

No. The PR touches the right helper, but ad hoc hostname suffix checks should be replaced with a gateway-specific policy that reuses shared IP-literal classification and adds compatibility coverage for documented gateway hostnames.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Reject mapped IPv6 metadata addresses:** `src/commands/gateway-status/helpers.ts:103-106`
  - body: The new guard only rejects `0.0.0.0` and hostnames that start with `169.254.`; Node canonicalizes `ws://[::ffff:169.254.169.254]:80` to `::ffff:a9fe:a9fe`, so the metadata endpoint bypasses the mitigation.
  - confidence: 0.94
- **[P2] Preserve documented gateway hostnames:** `src/commands/gateway-status/helpers.ts:110-112`
  - body: Calling `isBlockedHostname()` drops `.local` and `.internal` targets, but the gateway docs describe Bonjour `<hostname>.local` and wide-area domains such as `openclaw.internal.` as valid discovery/direct routes, so legitimate probes would be silently omitted.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff improves some plain hostname/IP cases, but its metadata SSRF mitigation is incomplete for IPv4-mapped IPv6 literals.
