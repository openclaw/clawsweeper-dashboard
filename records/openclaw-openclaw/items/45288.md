## Summary

Keep open: this is a plausible gateway-status security hardening PR, but the current patch is incomplete and would reject documented gateway discovery hosts.

## What This Changes

The PR adds gateway-status WebSocket hostname/IP filtering and a new security test file for metadata, internal/local, wildcard, and allowed targets.

## Best Possible Solution

Implement gateway-specific URL validation that blocks metadata/link-local/wildcard IP literals, including mapped IPv6, while preserving documented `.local`, private-LAN, and wide-area discovery hosts.

## Reproduction Assessment

Yes. Source inspection plus Node URL canonicalization shows the PR's mapped IPv6 metadata bypass, and current main shows accepted targets flow directly into the gateway probe path.

## Solution Assessment

No. The PR's string-prefix and shared-hostname-block approach is incomplete and over-broad; the safer fix should use shared IP-literal classification plus gateway-specific hostname policy.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Reject IPv4-mapped metadata addresses:** `src/commands/gateway-status/helpers.ts:103-106`
  - body: The new guard only blocks `0.0.0.0` and hostnames starting with `169.254.`. A target like `ws://[::ffff:169.254.169.254]:80` canonicalizes to `[::ffff:a9fe:a9fe]`, so the metadata endpoint still bypasses this mitigation.
  - confidence: 0.94
- **[P2] Preserve documented gateway discovery hosts:** `src/commands/gateway-status/helpers.ts:110-112`
  - body: Calling `isBlockedHostname()` rejects `.local` and `.internal` targets, but OpenClaw documents `.local` LAN gateways and wide-area DNS-SD domains such as `openclaw.internal.`. This would drop legitimate configured or explicit gateway probes.
  - confidence: 0.86
