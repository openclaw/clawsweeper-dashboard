## Summary

Keep open: this is plausible gateway-status SSRF hardening, but the proposed patch is incomplete, regresses documented gateway hostnames, and still lacks required real behavior proof.

## What This Changes

The PR adds gateway-status WebSocket hostname/IP filtering and a new security test file for metadata, wildcard, blocked-hostname, localhost, LAN, public, and non-WS URL cases.

## Best Possible Solution

Use a gateway-status-specific target policy that blocks metadata, link-local, wildcard, and malformed IP literals via shared IP classification while preserving documented localhost, LAN, Bonjour .local, and configured wide-area DNS-SD gateway names.

## Reproduction Assessment

Yes for source-level review: current main accepts explicit/configured ws(s) targets into the probe path, and a focused Node check against the PR's string policy shows mapped IPv6 metadata literals still pass. I did not run a live gateway-status probe.

## Solution Assessment

No. The PR touches the right helper, but the fix should reuse shared IP/special-use classification with a gateway-specific hostname policy instead of broad isBlockedHostname() suffix blocking.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Reject mapped IPv6 metadata addresses:** `src/commands/gateway-status/helpers.ts:103-106`
  - body: The new guard only rejects hostnames equal to 0.0.0.0 or beginning with 169.254.; Node canonicalizes mapped metadata literals like ws://[::ffff:169.254.169.254]:80 to an IPv6 hostname, so they bypass the stated SSRF mitigation.
  - confidence: 0.94
- **[P2] Preserve documented gateway hostnames:** `src/commands/gateway-status/helpers.ts:110-112`
  - body: Calling isBlockedHostname() rejects .local and .internal names, but OpenClaw documents Bonjour <hostname>.local and wide-area DNS-SD domains such as openclaw.internal. as valid gateway routes, so gateway-status would omit legitimate probes.
  - confidence: 0.86
