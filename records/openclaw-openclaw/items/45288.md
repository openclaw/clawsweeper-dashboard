## Summary

Keep open: this is plausible security hardening, but the current PR is not ready because it leaves an IPv4-mapped IPv6 metadata bypass, blocks documented gateway discovery hosts, and lacks real behavior proof.

## What This Changes

The PR adds gateway-status WebSocket target filtering and security tests for cloud metadata, link-local, wildcard, internal/local, and allowed gateway targets.

## Best Possible Solution

Implement gateway-specific URL validation that blocks metadata, link-local, wildcard, and malformed IP literals including mapped IPv6, while preserving documented `.local`, private-LAN, and wide-area discovery hosts.

## Reproduction Assessment

Yes. Source inspection shows current main passes accepted targets directly to the gateway probe path, and Node URL canonicalization shows the PR's mapped IPv6 metadata bypass without needing a full live gateway run.

## Solution Assessment

No. The string-prefix and shared-hostname-block approach is incomplete and over-broad; the narrower maintainable fix should reuse shared IP-literal classification with a gateway-specific hostname policy.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Reject IPv4-mapped metadata addresses:** `src/commands/gateway-status/helpers.ts:103-106`
  - body: The new guard only blocks `0.0.0.0` and dotted hosts starting with `169.254.`. Node canonicalizes `ws://[::ffff:169.254.169.254]:80` to `[::ffff:a9fe:a9fe]`, so this metadata endpoint still bypasses the mitigation.
  - confidence: 0.94
- **[P2] Preserve documented gateway discovery hosts:** `src/commands/gateway-status/helpers.ts:110-112`
  - body: Calling `isBlockedHostname()` here rejects `.local` and `.internal` targets, but OpenClaw documents `.local` LAN gateways and wide-area DNS-SD domains such as `openclaw.internal.`; this would drop legitimate explicit or configured gateway probes.
  - confidence: 0.87
