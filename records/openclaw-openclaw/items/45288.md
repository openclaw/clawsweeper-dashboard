## Summary

Keep open: the PR targets a real security-sensitive gateway-status gap, but the proposed mitigation is incomplete and needs real behavior proof plus maintainer security review before merge.

## What This Changes

The PR adds gateway-status WebSocket hostname validation and a new security test file for metadata, wildcard, blocked-hostname, and allowed-gateway cases.

## Best Possible Solution

Implement gateway-specific URL validation that reuses the shared IP-literal classifier for metadata/link-local/wildcard/malformed IPs while preserving documented gateway hostnames.

## Reproduction Assessment

Yes. Source inspection shows resolveTargets() feeds accepted ws(s) targets into probeGateway(), and a focused Node URL check shows mapped-IPv6 metadata URLs canonicalize to a hostname that bypasses the PR's checks.

## Solution Assessment

No. The PR's string-prefix plus shared hostname-denylist approach misses mapped IPv6 metadata addresses and overblocks documented gateway host suffixes; a gateway-specific policy on top of the shared IP classifier is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Reject mapped IPv6 metadata addresses:** `src/commands/gateway-status/helpers.ts:103-106`
  - body: The new guard only rejects 0.0.0.0 and dotted hosts starting with 169.254. Node canonicalizes ws://[::ffff:169.254.169.254]:80 to [::ffff:a9fe:a9fe], so the same metadata endpoint bypasses this SSRF fix.
  - confidence: 0.94
- **[P2] Preserve documented gateway hostnames:** `src/commands/gateway-status/helpers.ts:110-112`
  - body: Calling isBlockedHostname() here rejects .local and .internal, but OpenClaw documents LAN local. discovery and wide-area domains such as openclaw.internal. This would drop legitimate explicit or configured gateway-status probes.
  - confidence: 0.86
