## Summary

Keep this PR open, but do not merge it as-is. Current main still only scheme-checks gateway-status WebSocket URLs, while the PR’s string-based SSRF filter has a confirmed IPv4-mapped IPv6 metadata bypass and would also reject legitimate `.local` gateway targets documented and tested elsewhere.

## What This Changes

The PR adds hostname/IP filtering to gateway-status WebSocket URL normalization and a security test file for metadata/internal/wildcard rejection plus allowed gateway targets.

## Best Possible Solution

Keep the hardening direction, but replace the string-prefix checks with a gateway-specific policy that uses shared IP classification for literals, blocks metadata/link-local/wildcard targets including mapped IPv6 forms, preserves supported `.local` and private-LAN gateway routes where intended, and adds focused regression coverage plus a changelog entry if user-facing behavior changes.

## Reproduction Assessment

Yes. The bypass is reproducible with Node URL canonicalization, and source inspection shows the PR’s proposed checks would miss the canonical mapped IPv6 hostname while rejecting `.local` through the shared hostname block.

## Solution Assessment

No. The proposed generic hostname block plus dotted-prefix IP checks are not the narrowest maintainable fix; shared literal-IP classification with gateway-specific host rules is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Reject IPv4-mapped metadata addresses:** `src/commands/gateway-status/helpers.ts:103-106`
  - body: The new guard only checks `0.0.0.0` and dotted hostnames starting with `169.254.`. `ws://[::ffff:169.254.169.254]:80` canonicalizes to `[::ffff:a9fe:a9fe]`, so the metadata endpoint still bypasses this mitigation; use the shared IP classifier and add mapped-form coverage.
  - confidence: 0.94
- **[P2] Preserve `.local` gateway targets:** `src/commands/gateway-status/helpers.ts:110-112`
  - body: `isBlockedHostname()` rejects `.local` for both `ws://` and `wss://`, but OpenClaw documents and tests `.local` gateway routes for LAN/Bonjour flows. This would drop legitimate targets such as `wss://gateway.local:18789` instead of probing them.
  - confidence: 0.84
