## Summary

The PR targets a real security-sensitive gateway-status probe path, but it should stay open because the proposed fix leaves mapped-IPv6 metadata targets reachable and rejects gateway hostnames that current docs and tests treat as valid private/local targets.

## What This Changes

The PR adds hostname/IP filtering to `normalizeWsUrl()` in gateway-status helpers and adds security tests for blocked metadata, internal, local, wildcard, and allowed gateway WebSocket URLs.

## Best Possible Solution

Use gateway-status-specific validation backed by the shared special-use IP classifiers for metadata and link-local addresses, while preserving documented private gateway DNS targets unless maintainers explicitly decide to tighten that contract.

## Reproduction Assessment

Yes for source-level review: current main accepts explicit WebSocket probe targets, and the PR's proposed string-prefix logic can be bypassed by mapped-IPv6 metadata forms. I did not run a live gateway probe in this read-only pass.

## Solution Assessment

No. The direction is valid, but the implementation should reuse the existing IP classification logic and make any gateway hostname policy change explicit instead of broadly applying the HTTP SSRF hostname deny-list to gateway probe names.
