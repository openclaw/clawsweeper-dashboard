## Summary

Keep open: this is a plausible security bug fix for gateway-status, but the proposed patch is incomplete for the stated SSRF threat model and the external PR still lacks real behavior proof.

## What This Changes

The PR adds hostname filtering and security tests for explicit `gateway status` WebSocket targets, blocking selected metadata, wildcard, `.internal`, and `.local` hosts.

## Best Possible Solution

Reuse parsed host/IP SSRF classification for gateway-status explicit URLs so metadata/link-local forms are blocked across IPv4 and IPv6 while intentional local gateway targets remain allowed, then add focused regression tests and real CLI proof.

## Reproduction Assessment

Yes, by source inspection. Current main accepts explicit `ws://`/`wss://` gateway-status URLs by scheme only, and the PR's proposed string checks miss IPv4-mapped IPv6 metadata literals after URL normalization.

## Solution Assessment

No. The patch narrows some SSRF cases, but string prefix checks are not the best final mitigation; parsed IP/link-local detection using the existing SSRF helper layer is the safer maintainable path.
