## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Preserve guarded fetch redirect and abort semantics:** `src/agents/model-worker.ts:43-47`
  - body: The worker reconstructs the request with only method, headers, and body, dropping `redirect: "manual"`, dispatcher, and signal values supplied by the guarded model fetch path. That lets the worker fetch follow redirects and lose cancellation outside the current guard loop.
  - confidence: 0.92
- **[P1] Stream chunks instead of buffering worker responses:** `src/agents/model-worker.ts:50`
  - body: This waits for `response.arrayBuffer()` before posting one body back to the main thread, so SSE/model output is delivered only after the upstream request finishes. The worker path needs chunk forwarding or streaming calls should stay on the current transport.
  - confidence: 0.97
- **[P1] Gate force-drain behind explicit config:** `src/infra/restart.ts:521-523`
  - body: `opts.forceDrain` is optional, but the PR normalizes `undefined` into default 10/20/30/40s phases and wires hard kill to `process.exit(0)`. A normal active-work restart can now hard-exit before the existing restart deferral budget completes.
  - confidence: 0.95
- **[P1] Cancel the worker request before falling back:** `src/agents/model-worker-pool.ts:300-303`
  - body: On timeout this deletes the pending entry and rejects, then the wrapper falls back to main-thread fetch while the worker fetch can still be running. That can send the same model request twice and leave worker state stale.
  - confidence: 0.9
- **[P2] Queue requests instead of reusing a busy worker:** `src/agents/model-worker-pool.ts:184-191`
  - body: When every worker is busy, this selects a busy worker and posts another request to it rather than queueing. `maxWorkers` therefore does not bound concurrent model fetches and the single busy flag can be cleared while another request is still in flight.
  - confidence: 0.86
