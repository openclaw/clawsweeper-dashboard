## Summary

Keep open. Current main still does not implement Vapi as a bundled Voice Call provider, and the latest issue discussion explicitly kept this as a product/ownership decision. The documented Clawdia Vapi bridge is useful related context, but it is not `provider: "vapi"` support in `@openclaw/voice-call`, and the current plugin SDK does not expose a public telephony-provider registration seam equivalent to the realtime voice/transcription provider seams.

## What This Changes

Request to add Vapi as a Voice Call plugin provider, including provider code, config schema, runtime registration, tests, and docs.

## Best Possible Solution

The desired end state is an explicit ownership decision, followed by either a first-class bundled Vapi provider wired through Voice Call end to end, or a documented external/ClawHub bridge path backed by any missing public telephony-provider API.

## Reproduction Assessment

Not applicable as a runtime bug report. The missing feature is high-confidence by source inspection: `provider: "vapi"` is not accepted by schemas, no Vapi provider module exists, and runtime provider resolution has no Vapi branch.

## Solution Assessment

Unclear until maintainers choose the product boundary. If Vapi belongs in the bundled Voice Call plugin, the narrowest maintainable implementation is provider/config/runtime/manifest/docs/tests together; otherwise the safer path is an external bridge/plugin plus any needed public seam.
