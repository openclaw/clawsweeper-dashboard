## Summary

The requested Vapi workflow is useful, but it should remain an externally maintained integration: current project direction excludes bundled non-model commercial services, the broader Vapi request reached the same scope decision, and the repository already documents an active Vapi-to-OpenClaw bridge that provides phone-call access without adding Vapi policy to the bundled Voice Call plugin.

## What This Changes

Add Vapi as a bundled Voice Call telephony provider, including provider code, credentials/config schema, runtime selection, tests, and documentation.

## Best Possible Solution

Keep the Vapi bridge or a dedicated Vapi code plugin in its own repository and publish it through ClawHub; propose a separate generic Voice Call provider-registration API only if an external implementation demonstrates a concrete capability that public Gateway/plugin surfaces cannot support.

## Reproduction Assessment

Not applicable: this is a request for a new bundled provider, not a report of broken existing behavior; current main intentionally supports only its defined provider set.

## Solution Assessment

No. Adding Vapi to the bundled provider switch would create a new core-owned commercial integration, while the documented external bridge already delivers the central phone-call workflow with a cleaner ownership boundary.
