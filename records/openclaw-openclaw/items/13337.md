## Summary

Keep open. Current main still lacks bundled Vapi support in the Voice Call plugin, and the discussion plus plugin-boundary docs make this a maintainer product/ownership decision rather than stale cleanup or a ClawHub close.

## What This Changes

The issue requests adding Vapi as a first-class provider to the bundled Voice Call plugin, including provider code, config, runtime registration, tests, and docs.

## Best Possible Solution

Maintainers should choose the ownership path first, then either implement bundled Vapi support end to end or document the external bridge/ClawHub path and add a public telephony-provider seam if that is the intended extension model.

## Reproduction Assessment

Not applicable as a runtime bug report. Source inspection gives a high-confidence absence check: `vapi` is not accepted by config/types, no provider module exists, runtime has no Vapi branch, and docs/manifest omit it.

## Solution Assessment

Unclear until maintainers choose the product boundary. If bundled support is approved, the maintainable solution is provider, config/env/schema, runtime, manifest, tests, docs, and changelog together; otherwise the safer solution is an external bridge/plugin path plus any missing public seam.
