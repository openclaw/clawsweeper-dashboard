## Summary

Keep open. Current main still has the reported contract gap: heartbeat preflight reads base-session system events, isolated heartbeat execution switches to the sibling `:heartbeat` key, and active-session consumers only peek the isolated key. Recent code and tests also make this a trust and delivery-context boundary, so this needs maintainer direction rather than conservative closure.

## What This Changes

Expose or transfer pending system-event entries queued on the base heartbeat session to the isolated `:heartbeat` session used when `heartbeat.isolatedSession` is enabled.

## Best Possible Solution

Define a narrow, documented transfer or alias rule for pre-isolation system-event entries at the heartbeat isolation boundary, with tests that preserve single-consumption order, delivery-context routing, duplicate handling, and explicit handling for untrusted base-session events.

## Reproduction Assessment

Yes. A focused regression can enqueue a system event on the base heartbeat session, run `runHeartbeatOnce` with `heartbeat.isolatedSession: true`, and assert that active-session inspection of the resulting `:heartbeat` key does not see that base entry on current main.

## Solution Assessment

Unclear. A scoped transfer or alias is plausible, but current tests encode different trust behavior for base-only events under isolation, so maintainers should first decide the intended trust and delivery semantics.
