## Summary

Keep open. Current main still leaves pending base-session system-event entries invisible to consumers that call `peekSystemEventEntries` on the isolated `:heartbeat` run key, so the report remains a real latent contract gap rather than an obsolete issue. Existing code consumes/inlines some base events for prompt construction, and memory-core has a local workaround for one cron case, but there is no general transfer or alias across the isolation boundary.

## What This Changes

The issue requests transferring or aliasing pending system-event entries from a base heartbeat session to its isolated `:heartbeat` session when `heartbeat.isolatedSession` is enabled.

## Best Possible Solution

Define a narrow, documented event-visibility contract at the heartbeat isolation boundary: pre-isolation events should either be transferred or aliased so isolated-session consumers can observe them exactly once, while preserving base-queue ordering, consumption, delivery-context routing, duplicate handling, and trust-boundary behavior.

## Reproduction Assessment

Yes. A focused test can enqueue a system event on the base heartbeat session, run `runHeartbeatOnce` with `heartbeat.isolatedSession: true`, and assert that `peekSystemEventEntries` on the resulting `:heartbeat` key is empty on current main; source inspection confirms that path.

## Solution Assessment

Unclear. A scoped transfer-or-alias helper is plausible, but maintainers should first decide whether base-session untrusted and wake-triggered events are intended to affect active isolated-session inspection and owner-downgrade routing.
