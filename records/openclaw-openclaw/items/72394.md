## Summary

Keep open. Current main still has the reported contract gap: heartbeat preflight reads system events from the base session, isolated execution switches to a `:heartbeat` session, and active-session consumers then peek the isolated key where those base-session entries are not present. The related #72814 is open and adjacent, but it does not supersede this issue and appears to preserve the current empty-isolated-queue behavior.

## What This Changes

The issue requests transferring or aliasing pending base-session system-event entries when heartbeat execution crosses into an isolated `:heartbeat` session.

## Best Possible Solution

Add a narrow, documented transfer or alias for pre-isolation system-event entries at the heartbeat isolation boundary so isolated-session consumers can observe them exactly once, while preserving base-queue consumption order, delivery-context routing, duplicate handling, and explicit trust-boundary behavior.

## Reproduction Assessment

Yes. A focused test can enqueue a system event on the base heartbeat session, run `runHeartbeatOnce` with `heartbeat.isolatedSession: true`, and assert that `peekSystemEventEntries` on the resulting `:heartbeat` key remains empty on current main; I did not run it because this review is read-only.

## Solution Assessment

Unclear. A scoped transfer-or-alias is the likely code direction, but maintainers should first decide whether untrusted and wake-triggered base events are meant to affect active isolated-session inspection and owner-downgrade routing.
