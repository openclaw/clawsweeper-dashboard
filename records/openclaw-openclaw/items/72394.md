## Summary

Keep open. Current main still leaves base-session system-event entries invisible to isolated `:heartbeat` session-key consumers, so the report describes a real contract gap rather than an obsolete issue. The gap is latent because prompt construction consumes/inlines the base events today, but active-session inspection and plugin consumers can still see an empty isolated queue.

## What This Changes

The issue requests a heartbeat-boundary transfer or equivalent read path so pending system events queued on the base session are visible to isolated heartbeat runs.

## Best Possible Solution

Add a narrow, documented transfer or alias for pre-isolation system-event entries at the heartbeat isolation boundary so isolated-session consumers can observe them exactly once, while preserving base-queue consumption order, delivery-context routing, duplicate handling, and explicit trust-boundary behavior.

## Reproduction Assessment

Yes. A focused test can enqueue a system event on the base heartbeat session, run `runHeartbeatOnce` with `heartbeat.isolatedSession: true`, and assert that `peekSystemEventEntries` on the resulting `:heartbeat` key remains empty on current main; I did not run it because this review is read-only.

## Solution Assessment

Unclear. A scoped transfer-or-alias looks like the narrowest likely code direction, but maintainers should first decide whether untrusted and wake-triggered base events are meant to affect active isolated-session inspection and owner-downgrade routing.
