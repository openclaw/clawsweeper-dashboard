## Summary

Keep open. Current main still has no generic transfer, alias, or base-key read contract for pending base-session system-event entries once an isolated heartbeat switches to the `:heartbeat` session; existing prompt paths cover selected consumers, but active-key consumers can still observe an empty isolated queue.

## What This Changes

Request that isolated heartbeat runs preserve visibility of pending base-session system-event entries across the `:heartbeat` session boundary.

## Best Possible Solution

Define the isolated-heartbeat event visibility contract, document it, then implement either selective transfer or explicit base-key reads with regression coverage for exec, cron, untrusted, delivery-context, duplicate, and single-consumption behavior.

## Reproduction Assessment

Yes. A focused regression can enqueue a system event on the base heartbeat session, run or step through an isolated heartbeat, and assert that `peekSystemEventEntries` on the `:heartbeat` key remains empty while active-key consumers inspect that key.

## Solution Assessment

Unclear. Mirroring the cron session-entry transfer is plausible, but the current trust and delivery tests show that the safest implementation depends on a maintainer decision about which event classes should cross the boundary.
