## Summary

Keep open. Current main still lacks a generic transfer or alias for base-session system-event entries into the isolated `:heartbeat` queue; recent code covers some prompt/consumer cases, but the requested boundary contract is still unresolved.

## What This Changes

The issue requests a mechanism or policy so system-event entries queued on a base heartbeat session remain visible to active isolated `:heartbeat` session consumers.

## Best Possible Solution

Define a documented pre-isolation event visibility rule, then cover transfer or explicit base-key reads with tests for ordering, single consumption, duplicates, delivery context, and untrusted-event handling.

## Reproduction Assessment

Yes. A focused regression can enqueue an event on the base heartbeat session, run or step through an isolated heartbeat, and assert that `peekSystemEventEntries` on the `:heartbeat` key remains empty while active consumers read that key.

## Solution Assessment

Unclear. Mirroring the session-entry transfer is plausible, but current code and tests intentionally keep some base-session events out of active isolated classification, so maintainers should choose the trust and delivery semantics first.
