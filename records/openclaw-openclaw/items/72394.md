## Label Justifications

- `P2`: This is a normal-priority heartbeat/session-state correctness gap with limited current blast radius and no confirmed live regression.
- `impact:session-state`: The issue is about pending event state being visible on one session key but not on the active isolated heartbeat run key.
- `impact:message-loss`: Future consumers that read only the isolated active session key could miss queued system-event content.

## Summary

Keep open. Current main and v2026.6.8 still peek and consume selected pending events on the base heartbeat session while the model run executes on the isolated `:heartbeat` key, so the requested isolated-session event visibility contract remains unresolved.

## What This Changes

The issue asks OpenClaw to transfer or otherwise expose pending base-session system event entries across the isolated heartbeat session boundary.

## Best Possible Solution

Define one explicit heartbeat system-event visibility and ownership contract, then update the runner and regression tests so base and isolated session keys behave predictably without stale routing context.

## Reproduction Assessment

No live reproduction was established, but source inspection gives a high-confidence reproduction of the latent invariant: base-session events are peeked before the run switches to `:heartbeat`, and system-event queues are exact-key scoped.

## Solution Assessment

Unclear as an implementation choice. Clone/move and shared-read options are both plausible, so maintainers should choose the durable session/event ownership contract before implementation.
