## Summary

Keep open. Current main handles some base-session events through preflight-specific prompt and consumption paths, and memory-core has a narrow base-key fallback, but there is still no general contract or transfer path that makes base-session system-event entries visible to consumers reading the active isolated `:heartbeat` session key.

## What This Changes

Request a general isolated-heartbeat event-visibility contract so pending system events queued on the base session are available across the `:heartbeat` execution boundary.

## Best Possible Solution

Define the isolated-heartbeat event visibility contract first, then implement selective transfer or explicit base-key reads with regression coverage that preserves owner-downgrade, routing, and single-consumption semantics.

## Reproduction Assessment

Yes, by source inspection. Enqueue a system event on the base heartbeat session, run with `heartbeat.isolatedSession: true`, and a consumer that calls `peekSystemEventEntries` on the active `:heartbeat` session key reads that exact-keyed isolated queue rather than the base queue unless it has a consumer-specific fallback.

## Solution Assessment

Unclear. Mirroring the cron session-entry transfer is plausible, but current tests intentionally protect some base-session hook/wake and owner-downgrade events from affecting isolated runs, so maintainers should decide the event-class contract before changing behavior.
