## Summary

Keep open. Current main still reads pending events from the base heartbeat session during preflight, then switches active isolated consumers to the `:heartbeat` key without transferring or aliasing the event queue; current tests also make this a contract decision because some base-session events intentionally do not affect isolated-run trust/classification.

## What This Changes

Request to transfer or explicitly read pending system-event entries across the `heartbeat.isolatedSession` boundary.

## Best Possible Solution

Define the isolated-heartbeat event visibility contract, then implement selective transfer or explicit base-key reads with regression coverage for trust, delivery context, and single-consumption semantics.

## Reproduction Assessment

Yes, by source inspection. Enqueue a system event on the base heartbeat session, run with `heartbeat.isolatedSession: true`, and `peekSystemEventEntries` on the active `:heartbeat` key remains empty because queues are key-scoped and no transfer helper is called.

## Solution Assessment

Unclear. Mirroring the cron session-entry transfer is plausible, but current isolated heartbeat tests intentionally keep some base-session hook and untrusted events from affecting active isolated classification, so maintainers should decide which event classes cross the boundary first.
