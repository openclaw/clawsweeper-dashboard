## Label Justifications

- `P2`: This is a normal-priority core heartbeat/session-state correctness gap with limited current blast radius and no confirmed live regression.
- `impact:session-state`: Pending event state is associated with the base session while the active isolated heartbeat run uses a different session key.
- `impact:message-loss`: Future consumers that read only the isolated run key could miss queued system-event content.

## Summary

Keep open. Current main and v2026.6.10 still peek and consume pending system events on the resolved base heartbeat session while running the model turn on the isolated `:heartbeat` session, so the underlying ownership gap remains.

## What This Changes

Request to make pending system-event entries visible or transferable across the base-to-isolated heartbeat session boundary.

## Best Possible Solution

Define one explicit event-ownership contract for isolated heartbeat runs, then update the runner and regression tests so base and isolated session keys expose, consume, and route pending events predictably.

## Reproduction Assessment

Yes for source-level reproduction: current main peeks pending events from the resolved base heartbeat session while the model turn runs on the isolated `:heartbeat` key. I did not run a live user-flow reproduction in this read-only review.

## Solution Assessment

Unclear as an implementation choice. Transferring entries is plausible, but clone, move, and shared-read semantics affect ownership and consumption differently, so maintainers should choose the contract first.
