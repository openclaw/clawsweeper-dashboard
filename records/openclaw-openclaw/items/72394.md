## Label Justifications

- `P2`: This is a normal-priority heartbeat/session-state correctness gap with limited current blast radius and no confirmed live regression.
- `impact:session-state`: The issue is about pending event state being visible on the base session key but not on the active isolated heartbeat run key.
- `impact:message-loss`: Future consumers that read only the active isolated session key could miss queued system-event content.

## Summary

Keep open. Current main and v2026.6.8 still peek pending events from the resolved base heartbeat session, switch the model run to the isolated `:heartbeat` key, and consume selected events from the base key without making those entries visible on the isolated key.

## What This Changes

The issue requests a clear transfer or shared-read contract for pending system event entries when `heartbeat.isolatedSession: true` moves a heartbeat run from the base session to its isolated `:heartbeat` session.

## Best Possible Solution

Define one explicit heartbeat system-event ownership contract, then update the runner and regression tests so base and isolated session keys expose, consume, and route pending events predictably.

## Reproduction Assessment

Yes for source-level reproduction: current main peeks and consumes the base-session queue while the model run uses the isolated `:heartbeat` key. I did not run a live user-flow reproduction in this read-only review.

## Solution Assessment

Unclear as an implementation choice. Transferring entries is plausible, but clone, move, and shared-read semantics affect ownership and consumption differently, so maintainers should choose the contract first.
