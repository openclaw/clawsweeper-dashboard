## Label Justifications

- `P2`: This is a normal-priority heartbeat/session-state improvement with limited current blast radius but real future contract risk.
- `impact:session-state`: The issue concerns system-event state being scoped to the base session while the active heartbeat run uses an isolated sibling session.
- `impact:message-loss`: Consumers that read system events from the active isolated session can miss queued event content unless they implement a base-key fallback.

## Summary

Keep open. Current main still lacks a general transfer or shared visibility contract for base-session system-event entries when heartbeat switches into the isolated `:heartbeat` run key; current prompt paths and memory-core handle selected cases, but an active-session consumer can still read an empty isolated queue.

## What This Changes

The issue asks OpenClaw to make pending system-event entries queued on the base heartbeat session visible across the isolated `:heartbeat` session boundary.

## Best Possible Solution

Define the isolated-heartbeat event visibility contract, then implement selective transfer or explicit base-key reads with regression coverage for cron, exec, wake, owner-downgrade, routing, and single-consumption behavior.

## Reproduction Assessment

Yes, by source inspection. Enqueue a system event under the base heartbeat session, run with `heartbeat.isolatedSession: true`, and an active-run consumer calling `peekSystemEventEntries` on the `:heartbeat` session key reads that exact isolated queue rather than the base queue.

## Solution Assessment

Unclear. Mirroring the cron session-entry transfer is plausible, but current code intentionally avoids base-session turnSource routing for isolated runs and memory-core uses a targeted fallback, so maintainers should choose the event-class contract before changing behavior.
