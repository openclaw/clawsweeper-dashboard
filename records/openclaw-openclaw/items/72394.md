## Summary

Keep open. Current main still lacks a generic transfer, alias, or documented base-key read contract for pending base-session system-event entries after an isolated heartbeat switches to the `:heartbeat` key; selected prompt and memory consumers handle specific cases, but active-key consumers can still observe an empty isolated queue.

## What This Changes

Request to make pending base-session system-event entries visible across the isolated heartbeat boundary, either by transferring them to the `:heartbeat` session key or by explicitly reading from the base key.

## Best Possible Solution

Define the isolated-heartbeat event visibility contract, then implement selective transfer or explicit base-key reads with regression coverage for cron, exec, wake, untrusted, delivery-context, duplicate, and single-consumption behavior.

## Reproduction Assessment

Yes. A focused regression can enqueue a system event on the base heartbeat session, run an isolated heartbeat, and assert that `peekSystemEventEntries` on the `:heartbeat` key remains empty while active-key consumers inspect that key.

## Solution Assessment

Unclear. Mirroring the cron session-entry transfer is plausible, but current trust and delivery tests show the safest fix depends on deciding which event classes should cross the boundary.
