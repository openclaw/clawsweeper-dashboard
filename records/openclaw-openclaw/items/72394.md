## Summary

Keep open. Current main still has the isolated-heartbeat system-event visibility gap described in the issue: preflight reads events from the base session, the run switches to a sibling `:heartbeat` session, and active-session event consumers then peek that isolated key without a transfer or alias. The existing ClawSweeper comment and related PR #72814 are relevant context, but #72814 remains open and does not itself implement this event-entry transfer policy.

## What This Changes

The issue asks OpenClaw to transfer or alias pending base-session system event entries when `heartbeat.isolatedSession: true` moves a heartbeat run onto the sibling `:heartbeat` session key.

## Best Possible Solution

Define a narrow, documented rule for pre-isolation system events: either transfer a bounded snapshot to the isolated key or make isolated consumers explicitly read the base key, with regression tests covering ordering, single consumption, duplicate handling, delivery context, and untrusted-event policy.

## Reproduction Assessment

Yes. A focused regression can enqueue a system event on the base heartbeat session, run `runHeartbeatOnce` with `heartbeat.isolatedSession: true`, and assert that `peekSystemEventEntries` on the resulting `:heartbeat` key does not see the base entry on current main.

## Solution Assessment

Unclear. Mirroring the session-entry transfer is plausible, but current tests intentionally distinguish base-session events from active isolated-session events, so maintainers should choose the trust and delivery semantics before implementation.
