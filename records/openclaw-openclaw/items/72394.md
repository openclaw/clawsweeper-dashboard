## Label Justifications

- `P2`: This is a normal-priority heartbeat/session-state correctness gap with limited current blast radius and no confirmed urgent live regression.
- `impact:session-state`: The issue is about pending event state being visible under the base session key but not under the active isolated heartbeat run key.
- `impact:message-loss`: Future consumers that read only the isolated run key could miss queued system-event content that was placed on the base session.

## Summary

Keep open. Current main and v2026.6.8 still peek and consume pending system events on the base heartbeat session while the model run uses the isolated `:heartbeat` key, so the requested event ownership contract remains unresolved.

## What This Changes

The issue requests consistent transfer or exposure of pending base-session system event entries when `heartbeat.isolatedSession` moves the run to an isolated `:heartbeat` session.

## Best Possible Solution

Define one explicit heartbeat system-event ownership contract, then update the runner and regression tests so base and isolated session keys expose, consume, and route pending events predictably.

## Reproduction Assessment

Yes for source-level reproduction: configure isolated heartbeat, enqueue a pending event under the base session, and current main peeks/consumes that base queue while the model run uses the isolated key. I did not run a live user flow in this read-only review.

## Solution Assessment

Unclear as an implementation choice. Transferring entries is plausible, but clone, move, and shared-read semantics affect ownership and consumption differently, so maintainers should choose the contract first.
