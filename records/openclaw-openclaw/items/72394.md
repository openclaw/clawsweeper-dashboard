- `impact:message-loss`: System-event content can be invisible to consumers that read only the active isolated run session key.

## Summary

Keep open. Current main and the latest release still have session-key-scoped system-event queues, preflight still peeks the base heartbeat session before switching the actual run to the isolated `:heartbeat` key, and there is no general transfer/shared-read contract for future isolated-session consumers.

## What This Changes

The issue requests transferring or otherwise sharing pending base-session system-event entries when heartbeat runs cross into an isolated `:heartbeat` session.

## Best Possible Solution

Define one explicit heartbeat system-event visibility contract, then update the runner and regression coverage so base and isolated session keys behave predictably without duplicating stale routing state.

## Reproduction Assessment

Source-reproducible, but not live-reproduced: system events are keyed queues, preflight peeks the base key, and the model run later uses the isolated key. There is no high-confidence current user-facing failure path because existing prompt paths inline selected events before the isolated run.

## Solution Assessment

Unclear as an implementation choice: the issue's clone/move and shared-read options are both plausible, but maintainers should choose the durable session/event ownership contract before a fix PR.
