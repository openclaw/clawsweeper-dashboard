## Summary

Keep open. Current main still keeps system-event queues session-key scoped: isolated heartbeat setup creates/updates the `:heartbeat` session entry, but it does not transfer or alias base-session system-event entries, so active-key consumers can still see an empty isolated queue.

## What This Changes

Define how pending system-event entries queued on a base heartbeat session should remain visible after `heartbeat.isolatedSession` switches the run to the `:heartbeat` session.

## Best Possible Solution

Document the pre-isolation event visibility contract, then implement either selective transfer/aliasing or explicit base-key reads with regression tests for ordering, single consumption, delivery context, and trust classification.

## Reproduction Assessment

Yes. A focused regression can enqueue a system event on the base session, run an isolated heartbeat, and assert that `peekSystemEventEntries` on the `:heartbeat` key remains empty while selected prompt paths still consume base-session content.

## Solution Assessment

Unclear. Mirroring the cron session-entry transfer is plausible, but current trust and active-session tests show the correct fix depends on deciding which base events should become visible to isolated active-key consumers.
