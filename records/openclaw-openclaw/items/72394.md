## Summary

Keep open. Current main still keeps system-event queues scoped to explicit session keys, and isolated heartbeat runs create/use a `:heartbeat` session without transferring base-session entries; existing tests show a blanket transfer could change hook, routing, and trust behavior, so this needs a maintainer contract decision before implementation.

## What This Changes

The issue requests transferring or otherwise exposing pending base-session system event entries to isolated heartbeat runs that execute on the `:heartbeat` session key.

## Best Possible Solution

Define and document the isolated-heartbeat event visibility contract, then implement selective transfer or explicit base-key reads with regression coverage that preserves trust, delivery context, and single-consumption semantics.

## Reproduction Assessment

Yes, by source inspection. Enqueue a system event on the base heartbeat session, enable `heartbeat.isolatedSession: true`, and the active `:heartbeat` key still peeks a separate empty queue because no transfer helper is called.

## Solution Assessment

Unclear. Mirroring cron session-entry transfer is plausible, but current tests intentionally keep some base-session hook and untrusted events from active isolated classification, so maintainers should decide which event classes cross first.
