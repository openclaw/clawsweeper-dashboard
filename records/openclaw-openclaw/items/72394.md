## Summary

Keep open. Current main still lacks a general contract or transfer path for base-session system-event entries into the isolated heartbeat `:heartbeat` queue; a newer memory-core workaround covers one managed-dreaming case, but the broader event-visibility decision remains unresolved.

## What This Changes

Request to define and implement how pending system-event entries queued on a base heartbeat session should be visible to an isolated heartbeat run.

## Best Possible Solution

Define the isolated-heartbeat event visibility contract, then implement selective transfer or explicit base-key reads with regression coverage that preserves trust, routing, and single-consumption semantics.

## Reproduction Assessment

Yes, by source inspection. Enqueue a system event on the base heartbeat session, run with `heartbeat.isolatedSession: true`, and any consumer that calls `peekSystemEventEntries` on the active `:heartbeat` key will see the isolated queue rather than the base queue unless it has a consumer-specific fallback like memory-core now does.

## Solution Assessment

Unclear. Mirroring the cron session-entry transfer is plausible, but current tests intentionally keep some base-session hook/wake and untrusted events from affecting isolated active-session classification, so maintainers should decide the event-class contract first.
