## Label Justifications

- `impact:session-state`: The issue is about pending event state being visible on one session key but not the isolated heartbeat run key.
- `impact:message-loss`: Future consumers that read only the active isolated session key could miss queued system-event content.

## Summary

Keep open. Current main and the latest release still consume selected pending events from the base session while the isolated heartbeat model run executes on the `:heartbeat` key, so the requested isolated-session event visibility contract remains unresolved.

## What This Changes

The issue asks OpenClaw to transfer or otherwise expose pending system-event entries across the isolated heartbeat session boundary.

## Best Possible Solution

Define one explicit heartbeat system-event visibility contract, then update the runner and regression tests so base and isolated session keys behave predictably without preserving stale routing context.

## Reproduction Assessment

No high-confidence live reproduction is established. Source inspection reproduces the latent invariant: base-session events are peeked before the run switches to the isolated `:heartbeat` key, and `peekSystemEventEntries` is exact-key scoped.

## Solution Assessment

Unclear as an implementation choice. Clone/move and shared-read options are both plausible, but maintainers should choose the durable session/event ownership contract before implementation.
