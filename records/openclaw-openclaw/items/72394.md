## Summary

Keep open. Current main still peeks base-session events for heartbeat prompt construction, then switches active isolated consumers to the `:heartbeat` key without transferring the queue; the boundary gap remains and needs a contract decision because trust/routing tests currently rely on some base-session events not crossing.

## What This Changes

The issue asks for pending system event entries queued on a base heartbeat session to be transferred or read when the heartbeat runs in its isolated `:heartbeat` session.

## Best Possible Solution

Define and document the isolated-heartbeat event visibility contract, then implement either selective transfer or explicit base-key reads with regression coverage that preserves trust, delivery-context, and single-consumption semantics.

## Reproduction Assessment

Yes, by source inspection. Enqueue a system event on the base heartbeat session, run with `heartbeat.isolatedSession: true`, and `peekSystemEventEntries` on the active `:heartbeat` key remains empty because queues are key-scoped and no transfer helper is called.

## Solution Assessment

Unclear. Mirroring the cron session-entry transfer is plausible, but current tests intentionally keep some base-session hook and untrusted events from active isolated classification, so maintainers should first decide which event classes cross the boundary.
