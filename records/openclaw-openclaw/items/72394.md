## Summary

Keep open. Current main still lacks a generic transfer, alias, or base-key read contract for system-event queues after an isolated heartbeat switches to the `:heartbeat` session; selected prompt and memory-core paths compensate, but active-key consumers can still observe an empty isolated queue.

## What This Changes

The issue requests that isolated heartbeat runs preserve visibility of pending base-session system-event entries across the `:heartbeat` session boundary.

## Best Possible Solution

Define the isolated-heartbeat event visibility contract, document it, then implement either selective transfer or explicit base-key reads with regression coverage for exec, cron, untrusted, delivery-context, duplicate, and single-consumption behavior.

## Reproduction Assessment

Yes. A focused regression can enqueue a system event on the base heartbeat session, run `runHeartbeatOnce` with `heartbeat.isolatedSession: true`, and assert that `peekSystemEventEntries` on the `:heartbeat` key remains empty while active-key consumers inspect that key.

## Solution Assessment

Unclear. Mirroring the cron session-entry transfer is plausible, but the safest implementation depends on explicit rules for which event classes cross the boundary and how they are consumed.
