## Summary

Keep open. Current main still has a real isolated-heartbeat event-visibility gap: base-session events are used for prompt construction, but active isolated consumers read the `:heartbeat` key without any queue transfer, and existing tests show trust/routing semantics need a maintainer contract decision before changing that behavior.

## What This Changes

Request to make pending system-event entries visible across the `heartbeat.isolatedSession` base-session to `:heartbeat` boundary.

## Best Possible Solution

Define and document the isolated-heartbeat event visibility contract, then implement selective transfer or explicit base-key reads with regression coverage that preserves trust, delivery routing, and single-consumption semantics.

## Reproduction Assessment

Yes, by source inspection. Enqueue a system event on the base heartbeat session, run with `heartbeat.isolatedSession: true`, and active consumers that call `peekSystemEventEntries` on the `:heartbeat` key will see an empty queue because queues are key-scoped and no transfer helper is called.

## Solution Assessment

Unclear. Mirroring the cron session-entry transfer is plausible, but current tests intentionally keep some base-session hook and untrusted events from active isolated classification, so maintainers should decide which event classes cross the boundary first.
