## Summary

Keep open. Current main still has a core contract gap: preflight peeks pending system-event entries on the base heartbeat session, but the isolated run switches to the `:heartbeat` key and active-session consumers only peek that isolated key. The prompt path drains the base events today, and memory-core has a targeted base-key fallback, but there is still no general transfer, alias, or documented visibility contract for base-session events across the isolated heartbeat boundary.

## What This Changes

The issue asks for pending system-event entries queued on the base heartbeat session to remain visible or available when a heartbeat run switches to the isolated `:heartbeat` session key.

## Best Possible Solution

Define a narrow core contract at the heartbeat isolation boundary so pre-isolation base-session system events are visible to isolated-session consumers exactly once, with focused tests for base-to-isolated peek/drain behavior, untrusted owner downgrade, delivery-context routing, duplicate handling, and memory-core dreaming triggers.

## Reproduction Assessment

Yes. A focused test can enqueue a system event on the base heartbeat session, run `runHeartbeatOnce` with `heartbeat.isolatedSession: true`, and assert that `peekSystemEventEntries` on the resulting `:heartbeat` key remains empty on current main; I did not run it because this review is read-only.

## Solution Assessment

Unclear. A scoped transfer or alias at the isolation boundary looks like the likely maintainable direction, but maintainers should decide the exact trust, delivery-context, and consumption semantics before changing core behavior.
