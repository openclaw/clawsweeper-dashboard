## Summary

Keep open. Current main still uses key-scoped system-event queues and inspects the isolated `:heartbeat` run key after crossing the session boundary without transferring pending base-session entries; prompt-specific handling exists, but the generic boundary contract remains unresolved and overlaps open PR #72814.

## What This Changes

The issue asks OpenClaw to make pending system-event entries queued on a base heartbeat session visible across the isolated `:heartbeat` heartbeat session boundary.

## Best Possible Solution

Define the isolated-heartbeat event visibility contract, then implement one generic transfer, alias, or explicit base-key read path with regression coverage for cron, exec, trust classification, delivery context, duplicate handling, and single consumption.

## Reproduction Assessment

Yes. A focused regression can enqueue a system event on the base heartbeat session, run or step through an isolated heartbeat, and verify active-key consumers read `peekSystemEventEntries` on the `:heartbeat` key while the base entry was only seen by preflight. No tests were run because this was a read-only review.

## Solution Assessment

Unclear. Mirroring the cron session-entry transfer is plausible, but current tests intentionally keep some base-session events out of isolated active classification, so the narrowest maintainable fix needs a maintainer decision about trust and delivery semantics.
