## Summary

Keep open. Current main still has the reported isolated-heartbeat system-event visibility gap: preflight peeks the base session, isolation creates and runs the sibling `:heartbeat` session, and active-session consumers then peek the isolated key without any transfer or alias of base-session event entries. The issue is valid but should stay with maintainers because the fix needs an explicit trust and delivery-context policy decision and overlaps the still-open isolated-routing PR #72814.

## What This Changes

Request to transfer or alias pending base-session system-event entries when `heartbeat.isolatedSession: true` moves a heartbeat run onto the sibling `:heartbeat` session key.

## Best Possible Solution

Define a narrow, documented rule for pre-isolation system events: either transfer a bounded snapshot to the isolated key or make isolated consumers explicitly read the base key, with tests for ordering, single consumption, duplicate handling, delivery context, and untrusted-event policy.

## Reproduction Assessment

Yes. A focused regression can enqueue a system event on the base heartbeat session, run `runHeartbeatOnce` with `heartbeat.isolatedSession: true`, and assert that `peekSystemEventEntries` on the resulting `:heartbeat` key does not see the base entry on current main.

## Solution Assessment

Unclear. Mirroring the session-entry transfer is plausible, but current tests intentionally distinguish base-session events from active isolated-session events, so maintainers should choose the trust and delivery semantics before implementation.
