## Summary

Keep open. Current main still has the base-session-to-isolated-session system-event visibility gap: the heartbeat preflight inspects the base queue, the isolated run creates only the `:heartbeat` session entry, and later active-session consumers inspect the isolated key. The memory-core workaround and open #72814 are useful related context, but they do not establish a generic transfer/alias contract on main.

## What This Changes

The issue requests carrying pending system-event entries from the base heartbeat session to the isolated `:heartbeat` session, or otherwise making isolated heartbeat consumers see those entries safely.

## Best Possible Solution

Add a narrow, documented heartbeat-boundary transfer or lookup contract for pre-isolation system-event entries, with tests that pin exact-once consumption, trust downgrade behavior, delivery-context routing, and memory-core dreaming behavior. Coordinate it with #72814 rather than landing a separate semantic change blindly.

## Reproduction Assessment

Yes. A focused test can enqueue a system event on the base session, run `runHeartbeatOnce` with `heartbeat.isolatedSession: true`, and assert that `peekSystemEventEntries` on the isolated `:heartbeat` key remains empty on current main; this review did not run tests because the task is read-only.

## Solution Assessment

Unclear. The issue's transfer option is plausible and narrow, but a direct move/copy is not automatically the best fix until maintainers decide how base queued events should affect isolated-session trust flags, routing context, and one-time consumption.
