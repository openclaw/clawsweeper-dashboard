## Summary

Keep open. Current main still has no generic transfer, alias, or documented base-key fallback for pending base-session system-event entries after an isolated heartbeat switches to the `:heartbeat` run key; selected consumers handle the base queue, but active-key consumers can still observe an empty queue.

## What This Changes

Request that isolated heartbeat runs make base-session pending system-event entries visible across the `:heartbeat` session boundary.

## Best Possible Solution

Define the isolated-heartbeat event visibility contract, then implement either selective transfer or explicit base-key fallback reads with regression coverage for cron, exec, untrusted, delivery-context, duplicate, and single-consumption behavior.

## Reproduction Assessment

Yes. A focused regression can enqueue a system event on the base heartbeat session, run or step through an isolated heartbeat, and assert that `peekSystemEventEntries` on the `:heartbeat` key remains empty while active-key consumers inspect that key.

## Solution Assessment

Unclear. Mirroring the cron session-entry transfer is plausible, but the current trust and delivery tests show the safer solution depends on a maintainer decision about which event classes should cross the boundary.
