## Summary

Keep open. Current main still has the isolated-heartbeat system-event visibility gap: base-session events are used for prompt construction and selected consumption, but active isolated-session consumers still read the `:heartbeat` queue without any queue transfer; the related routing PR was closed unmerged and no open canonical replacement was found.

## What This Changes

The issue requests transferring or otherwise exposing pending base-session system event entries across the isolated heartbeat session boundary.

## Best Possible Solution

Define and document the isolated-heartbeat event visibility contract, then implement selective transfer or explicit base-key reads with regression coverage that preserves trust, delivery routing, and single-consumption semantics.

## Reproduction Assessment

Yes, by source inspection. Enqueue a system event on the base heartbeat session, run with `heartbeat.isolatedSession: true`, and any active consumer that calls `peekSystemEventEntries` on the `:heartbeat` key will see an empty queue because queues are session-key scoped and no transfer helper is called.

## Solution Assessment

Unclear. Mirroring the cron session-entry pattern is plausible, but current tests intentionally keep some base-session hook and untrusted events out of isolated active-session classification, so maintainers should decide which event classes cross the boundary first.
