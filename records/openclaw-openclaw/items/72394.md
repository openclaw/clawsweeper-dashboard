## Summary

Keep open. Current main still keeps system-event queues session-key scoped, peeks the base heartbeat session during preflight, then switches active isolated consumers to the `:heartbeat` session without transferring or aliasing the pending entries; existing tests also show some base-session events intentionally do not affect isolated trust/routing behavior.

## What This Changes

Request to make pending base-session system-event entries visible across the `heartbeat.isolatedSession: true` boundary.

## Best Possible Solution

Define the isolated-heartbeat event visibility contract, then implement selective transfer or explicit base-key reads with regression coverage for trust, delivery context, and single-consumption semantics.

## Reproduction Assessment

Yes, by source inspection. Enqueue a system event on the base heartbeat session, run with `heartbeat.isolatedSession: true`, and `peekSystemEventEntries` on the active `:heartbeat` key remains empty because queues are exact-key scoped and no transfer helper is called.

## Solution Assessment

Unclear. Mirroring the cron session-entry transfer is plausible, but current trust and routing tests intentionally keep some base-session events from affecting isolated runs, so maintainers should decide which event classes cross first.
