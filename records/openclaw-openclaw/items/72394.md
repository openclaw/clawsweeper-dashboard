## Label Justifications

- `P2`: This is a bounded heartbeat/session-state correctness gap with limited current blast radius but real future-maintainer impact.
- `impact:session-state`: The issue concerns pending event state being visible on one session key but not the isolated heartbeat run key.
- `impact:message-loss`: Future consumers that read only the active isolated session key could miss queued system-event content.

## Summary

Keep open. Current main and the latest release still keep system-event queues scoped by session key: heartbeat preflight peeks the base session, isolated runs execute on the `:heartbeat` key, and selected entries are consumed from the base queue after the run rather than being mirrored or shared for general isolated-session consumers.

## What This Changes

Request an explicit system-event visibility contract for pending entries when heartbeat runs cross from a base session to an isolated `:heartbeat` session.

## Best Possible Solution

Define one explicit heartbeat system-event visibility contract, then update the runner and regression tests so base and isolated session keys behave predictably without preserving stale routing context.

## Reproduction Assessment

No high-confidence live reproduction is established. Source inspection does reproduce the latent invariant: preflight peeks base-session events, the model run switches to `:heartbeat`, and general isolated-session queue readers would see an empty queue.

## Solution Assessment

Unclear as an implementation choice. The issue's clone/move and shared-read options are both plausible, but maintainers should choose the durable session/event ownership contract before implementation.
