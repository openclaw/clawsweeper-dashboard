## Summary

Keep this PR open for maintainer follow-up. Current main still does not implement the PR's node-role loopback exemption or auth.mode="none" backend self-pairing change, but the proposed patch broadens gateway device-identity bypass behavior and still does not clearly match the named cron, sessions_spawn, and ACP caller paths.

## What This Changes

The PR adds a loopback node-role missing-device bypass in the Gateway WebSocket connect policy, plus tests, and adds an auth.mode="none" backend self-pairing skip in handshake helpers, plus tests.

## Best Possible Solution

A rebased fix should target the confirmed runtime rejection path in current main: implement any auth.mode="none" backend exception in the locality-aware gateway-client/backend self-pairing path with direct-loopback and no-browser-origin guards, and add a node-role exemption only if a real node-role internal caller is proven, scoped away from Control UI with regression tests and docs updates.

## Reproduction Assessment

Yes at the policy-test level: current main's evaluator and default-token auth suite still reject device-less node-role clients with valid shared auth. Full runtime reproduction for the reported cron/sessions_spawn/ACP failures is unclear because the current named call paths use operator/backend gateway-client identities and the PR author later said the compiled rejection branch needed re-validation.

## Solution Assessment

No: the PR is not the best fix as-is because it mixes a broad node-role device-identity bypass with a separate backend self-pairing change, while the observed internal callers appear to need the backend path. The safer path is a narrower, rebased change against current main's locality-aware backend helper, with any node exception separately justified and scoped.
