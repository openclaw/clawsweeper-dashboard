## Summary

Keep open because the PR still relaxes Gateway device-identity and pairing policy in a way that is broader than the confirmed current-main backend trust path. Current main reserves device-less loopback behavior for same-process backend `gateway-client` calls authenticated with shared token/password, while this PR also allows authenticated loopback `node` role sessions without device identity and the author later noted the runtime rejection path still needed re-validation.

## What This Changes

The PR adds an authenticated loopback `node` role missing-device bypass in the Gateway WebSocket connect policy, adds tests for that matrix, and adds an `auth.mode="none"` backend self-pairing skip with tests.

## Best Possible Solution

A maintainer-reviewed replacement or rebase should target the confirmed current-main rejection path: keep the documented backend trust boundary on direct loopback `gateway-client` backend calls, consider an `auth.mode="none"` backend pairing exception only with direct-local and no-browser-origin guards, and add a node-role exception only if a real node-role internal caller is proven and scoped away from Control UI.

## Reproduction Assessment

Yes at the focused policy level: current main's evaluator and default-token auth suite still reject device-less node clients with valid shared auth. Full runtime reproduction for the reported cron/sessions_spawn/ACP failure is unclear because the named current-main paths use operator/backend gateway-client connections, and the PR author later said the compiled rejection branch needed re-validation.

## Solution Assessment

No. The PR is not the best fix as-is because it combines a broad node-role missing-device bypass with a separate backend self-pairing change; the safer solution is a narrower current-main backend helper change, with any node-role exception separately justified and tested.
