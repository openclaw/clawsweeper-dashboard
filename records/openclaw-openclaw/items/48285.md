## Summary

Keep this PR open but do not merge it as-is: the backend loopback direction is plausible, but the diff broadens the device-less node-role allow path into Control UI handshakes and the current PR discussion already notes the cron/bootstrap failure may not be fully explained by this patch.

## What This Changes

The PR adds a loopback node-role device-identity bypass plus an auth.mode=none local backend self-pairing bypass, with unit coverage for those paths.

## Best Possible Solution

Keep the backend loopback/no-auth fix only if it is validated against current handshake behavior, and scope any node-role device-identity exemption to non-Control UI clients with regression coverage.

## Reproduction Assessment

Yes for the PR-introduced regression path from source inspection: a local Control UI client can self-declare `role: "node"` and hit the new allow branch without device identity. I did not live-run the original cron/bootstrap failure path.

## Solution Assessment

No. The proposed fix needs at least a Control UI exclusion on the node-role bypass and fresh validation that the auth.mode=none backend self-pairing change is still the narrowest fix on current main.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Restrict the node loopback bypass to non-Control UI:** `src/gateway/server/ws-connection/connect-policy.ts:124-126`
