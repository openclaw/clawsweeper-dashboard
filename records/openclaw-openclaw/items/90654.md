## What This Changes

The issue reports that CLI message write commands should connect and send normally instead of closing the gateway WebSocket during the `secrets.resolve` handshake with a `.trim()` TypeError.

## Best Possible Solution

Normalize or guard all untrusted connect-time role/scope list entries before trimming, return a structured handshake error instead of throwing, and add coverage for a CLI `secrets.resolve`/message-write connection using malformed or legacy pairing state.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source-level yes, live-run no: the reported `message send` path maps to `messageCommand` -> `resolveCommandConfigWithSecrets` -> `secrets.resolve`, and current/release code still has an unsafe connect-time trim path, but I did not run a gateway repro in this read-only review.

## Solution Assessment

Is this the best way to solve the issue? Yes in direction, but the durable fix should normalize the handshake/pairing inputs at the gateway boundary rather than patching only the generated `dist` line or one observed field.
