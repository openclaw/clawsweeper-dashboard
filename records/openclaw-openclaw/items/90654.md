## Label Justifications

- `P1`: The report describes a reproducible CLI write-path regression that blocks send, broadcast, and probe workflows for affected users.
- `impact:message-loss`: The affected CLI write commands fail before the requested outbound message can be delivered.

## Summary

Current main and the latest release still contain the unguarded gateway handshake/pairing paths that can call `.trim()` on malformed role or scope entries, while the linked closing PR remains open and unmerged. This issue should stay open until that PR lands or maintainers choose a replacement fix.

## What This Changes

The issue reports that CLI write commands fail before connecting because the gateway WebSocket handshake throws a `TypeError` while trimming malformed or undefined role/scope data.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/92178, or replace it with an equivalent gateway/device-pairing guard that turns malformed persisted role/scope data into a structured handshake failure instead of a WebSocket parse crash.

## Reproduction Assessment

Yes. Source inspection shows malformed non-string role/scope entries can reach unguarded `.trim()` calls during WebSocket connect; I did not run the live gateway because this review is read-only.

## Solution Assessment

Yes. A narrow guard at the gateway audit formatter and pairing merge boundary is the maintainable fix; changing only CLI secret resolution would leave other gateway connect clients exposed.
