## Label Justifications

- `P1`: The PR targets a duplicate-reply restart bug in an active iMessage channel workflow, and the proposed state change can also affect message delivery.
- `merge-risk: 🚨 message-delivery`: Merging as-is could skip unqueried iMessage rows after partial catchup or replay later rows from coalesced live buckets after restart.
- `merge-risk: 🚨 session-state`: The patch writes persisted iMessage catchup cursor state, so incorrect advancement changes behavior across gateway sessions.

## Summary

Keep open: the bug-fix direction is valid, but the patch can skip rows after partial catchup, can still replay coalesced live rows, and still relies on mocked imsg proof rather than a live macOS restart path.

## What This Changes

The PR adds an iMessage live-path catchup cursor writer after startup catchup, bypasses that writer during catchup replay, expands cursor/monitor tests, and documents restart cursor behavior.

## Best Possible Solution

Keep the live cursor advancement direction, but enable it only after all catchup chats complete, persist a true high-water row for coalesced live buckets, and add redacted live macOS restart proof.

## Reproduction Assessment

Yes, source-level: current main persists the iMessage catchup cursor only in the startup catchup pass while live notifications go through handleMessageNow and process-local dedupe. I did not run a live macOS Messages restart repro in this read-only review.

## Solution Assessment

No as written: advancing the cursor from live handling is the right fix boundary, but enabling it after partial catchup and reading only the merged payload id are unsafe.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Gate live cursor advances on full catchup success:** `extensions/imessage/src/monitor/monitor-provider.ts:958`
  - body: `querySucceeded` only proves the top-level catchup fetch resolved; the bridge still logs and continues past individual `messages.history` failures. After such a partial pass, a later live row can move the persisted cursor past rows from the failed chat so they are never replayed; surface an all-chats-complete flag and keep live advancement disabled if any per-chat fetch failed.
  - confidence: 0.91
- **[P2] Advance coalesced live buckets to their high-water row:** `extensions/imessage/src/monitor/monitor-provider.ts:359-369`
  - body: `combineIMessagePayloads` preserves the first payload's `id` while taking the latest timestamp, so this cursor writer records the first rowid for a multi-row debounce flush. With `coalesceSameSenderDms`, rows after the first can still be replayed after restart; track the max source rowid/date or advance each source row after the combined turn succeeds.
  - confidence: 0.84
