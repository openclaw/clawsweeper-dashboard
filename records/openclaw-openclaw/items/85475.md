## Summary

Keep open: this is a focused fix for a real iMessage catchup bug, but the branch is not merge-ready because the new cursor advance can skip rows after partial catchup and the PR proof is mock-only.

## What This Changes

The PR adds iMessage live-path catchup cursor advancement after startup catchup, bypasses that writer during catchup replay, updates cursor/monitor tests, and documents the restart behavior.

## Best Possible Solution

Keep the live cursor advancement direction, but gate it on a fully completed catchup pass, preserve the high-water row for coalesced live buckets, and require redacted live restart proof before merge.

## Reproduction Assessment

Yes, source-level: current main has startup catchup cursor persistence but no live-path cursor writer, matching the linked restart replay report. I did not run a live macOS Messages reproduction in this read-only review.

## Solution Assessment

No as currently written: advancing the cursor from live handling is the right fix boundary, but enabling it after partial catchup and reading only the merged payload id are unsafe.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Gate live cursor advances on full catchup success:** `extensions/imessage/src/monitor/monitor-provider.ts:958`
  - body: `querySucceeded` only proves the top-level catchup query resolved; the bridge still logs and continues past individual `messages.history` failures. After such a partial pass, the next live message can advance the persisted cursor past rows from the failed chat, so those offline messages are never retried on restart. Surface a stricter all-chats-complete flag, or keep live advancement disabled when any per-chat catchup fetch fails.
  - confidence: 0.91
- **[P2] Advance coalesced live buckets to their high-water row:** `extensions/imessage/src/monitor/monitor-provider.ts:359-369`
  - body: `combineIMessagePayloads` preserves the first payload's `id` while taking the latest timestamp, so this cursor writer records the first rowid for a multi-row debounce flush. With `coalesceSameSenderDms`, rows after the first can still be replayed after restart; track the max source rowid/date for the bucket or advance each source row after the combined turn succeeds.
  - confidence: 0.84
