## Summary

Keep open: current main and the latest release still omit the explicit MiniMax TTS `stream` and `output_format` fields, the adjacent MiniMax envelope-error PR does not supersede this request-shape fix, and the PR is a narrow plugin-local patch with sufficient live-output proof.

## What This Changes

The PR adds `stream: false` and `output_format: "hex"` to MiniMax TTS requests and updates the speech-provider test to assert that request shape.

## Best Possible Solution

Land the narrow MiniMax plugin request-shape fix after maintainer review and proof-check refresh; no core speech API or broader fallback change is needed.

## Reproduction Assessment

Yes from source and upstream contract, but not from an independently replayed live current-main failure. Current main decodes MiniMax `data.audio` as hex while omitting the explicit output-shape fields, and the contributor supplied live after-fix output showing MiniMax accepts those fields.

## Solution Assessment

Yes. The best fix is to make the MiniMax plugin request the non-streaming hex response it already decodes, rather than moving vendor-specific request behavior into core or duplicating the separate envelope-error fix from https://github.com/openclaw/openclaw/pull/93688.
