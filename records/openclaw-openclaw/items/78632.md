## Summary

The report matches current `main`: `/status` intentionally skips external auth overlays, has a Codex CLI file-backed OAuth fallback, but has no equivalent Claude CLI fallback, so a clean agent with only Claude CLI OAuth credentials can display `unknown`.

## What This Changes

The issue asks for `/status` to display Claude CLI OAuth auth for working `claude-cli` runtime sessions instead of `unknown`.

## Best Possible Solution

Add a non-prompting Claude CLI OAuth label path for status, cover both legacy `claude-cli/*` refs and Anthropic models running on the Claude CLI runtime, and keep token material out of display output.

## Reproduction Assessment

Yes, source-reproducible: `/status` bypasses external CLI overlays, `resolveModelAuthLabel` has only a Codex CLI direct credential fallback, and the Claude CLI reader exists but is unused in that fallback path. I did not run a live credential-backed session in this read-only review.

## Solution Assessment

Yes, the narrow maintainable fix is to mirror the existing non-prompting Codex CLI status-label fallback for Claude CLI and add regression coverage. For the canonical Anthropic model plus Claude CLI runtime case, route the status auth-provider lookup to `claude-cli` only when the effective runtime is Claude CLI.
