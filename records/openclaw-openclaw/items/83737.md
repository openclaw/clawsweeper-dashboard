## Summary

Keep open: current main and the latest release still build a Codex app-server `workspaceWrite` policy with only the effective workspace in `writableRoots`, while OpenClaw docs and bind parsing preserve independent `:rw` bind semantics. The issue is source-reproducible and already assigned, but the repair needs sandbox/security review before widening the writable filesystem boundary.

## What This Changes

Honor Docker `:rw` bind mount targets in Codex app-server writable roots when an OpenClaw sandbox uses read-only workspace access.

## Best Possible Solution

Patch the Codex app-server sandbox policy to derive writable roots from already-validated writable Docker binds while preserving `:ro` failures, external-source validation, blocked-root checks, and host/container path safety, then add focused unit coverage plus a redacted Docker sandbox smoke.

## Reproduction Assessment

Yes at source level: with `workspaceAccess: "ro"`, current main passes only the sandbox workspace cwd into Codex `writableRoots`, so an additional `/data:rw` Docker bind is absent from the app-server policy. I did not run a live Docker/Codex repro in this read-only review.

## Solution Assessment

The issue direction is correct, but the exact suggested fix should not blindly add container targets until Codex app-server path semantics are confirmed. The maintainable fix should reuse the validated bind parser and prove both `:rw` and `:ro` behavior.
