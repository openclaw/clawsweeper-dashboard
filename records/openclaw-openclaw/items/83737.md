## Summary

Keep open: current main and the latest release still build the Codex app-server sandbox policy with only the effective workspace in `writableRoots`, so a validated writable Docker bind is not represented when `workspaceAccess` is read-only. The report is source-reproducible, but the repair touches a sandbox filesystem boundary and needs owner/security review before changing writable-root semantics.

## What This Changes

The issue reports that Docker bind mounts declared `:rw` are not writable through Codex app-server code-mode turns when OpenClaw sandboxing uses `workspaceAccess: "ro"`.

## Best Possible Solution

Patch the Codex app-server sandbox policy to derive writable roots from already-validated writable Docker binds using the correct host/container path semantics, with tests proving `:rw` binds are writable, `:ro` binds stay read-only, and blocked bind validation is unchanged.

## Reproduction Assessment

Yes, at source level: with `workspaceAccess: "ro"`, current main passes only the scratch sandbox workspace into Codex `writableRoots`, so an additional `/data:rw` bind is absent from the app-server policy. I did not run the live Docker write flow because this review had to keep the checkout read-only.

## Solution Assessment

Yes on direction, with a safety condition: the policy should honor validated writable binds, but the implementation should derive roots from the existing bind parser and prove Codex app-server path semantics before adding anything to `writableRoots`.
