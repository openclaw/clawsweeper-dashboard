## Label Justifications

- `P1`: This is a fresh regression in sandboxed agent shell writes where the only reported workaround disables sandboxing entirely.
- `impact:security`: The affected behavior is the sandbox filesystem boundary and Docker bind-mount permission model.

## Summary

Keep open: current main still sends the Codex app-server a sandbox policy with only the effective workspace in `writableRoots`, while the sandbox docs and bind parsing treat Docker `:rw` bind mounts as independent from `workspaceAccess`. The report is source-reproducible, but the fix needs sandbox-boundary review because blindly adding container targets could widen host writes.

## What This Changes

Requests that Docker bind mounts declared `:rw` remain writable when an agent uses `sandbox.mode: "all"` with `workspaceAccess: "ro"`.

## Best Possible Solution

Patch the Codex app-server sandbox policy to honor already-validated writable Docker bind mounts under read-only workspace access, with regression coverage proving `:rw` binds are writable, `:ro` binds stay read-only, and blocked bind-source validation is not weakened.

## Reproduction Assessment

Yes at source level: with `workspaceAccess: "ro"`, current main passes only the sandbox workspace cwd into Codex `writableRoots`, so an additional `/data:rw` bind is absent from the app-server policy. I did not run a live Docker repro because this review had to keep the checkout read-only.

## Solution Assessment

The issue direction is correct, but the exact suggested fix should not blindly add container targets until Codex app-server host/container path semantics are confirmed. The maintainable fix should derive roots from the existing validated bind parser and test both `:rw` and `:ro` cases.
