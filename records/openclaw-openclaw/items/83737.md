## Summary

Keep open: current main still sends Codex app-server turns a `workspaceWrite` policy whose `writableRoots` contain only the effective workspace, while Docker bind parsing and public docs treat `:rw` bind mounts as independent of `workspaceAccess`. The report is source-reproducible and security-sensitive, so it needs maintainer/secops review before changing the sandbox write boundary.

## What This Changes

Issue reports that Docker bind mounts declared `:rw` are not included in Codex app-server `writableRoots` when the OpenClaw sandbox uses `workspaceAccess: "ro"`.

## Best Possible Solution

Patch the Codex app-server sandbox policy to honor already-validated writable Docker bind mounts under read-only workspace access, while preserving `:ro` bind failures, blocked-source validation, and explicit tests for host/container path semantics.

## Reproduction Assessment

Yes at source level: with `workspaceAccess: "ro"`, current main passes only the sandbox workspace cwd into Codex `writableRoots`, so an additional `/data:rw` bind is absent from the app-server policy. I did not run a live Docker repro because this review had to keep the checkout read-only.

## Solution Assessment

The issue direction is correct, but the suggested target-path-only fix is not safe enough until Codex app-server host/container path semantics are confirmed. The maintainable repair should derive writable roots from the existing validated bind parser and test both `:rw` and `:ro` cases.
