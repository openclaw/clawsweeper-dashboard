## Summary

Keep open: current main still sends Codex app-server turns a `workspaceWrite` policy whose `writableRoots` contain only the effective workspace, while the documented Docker bind contract treats `:rw` bind mounts as independent from `workspaceAccess`. The report is source-reproducible, but the repair crosses a sandbox write boundary and should stay with maintainer/security review.

## What This Changes

The issue requests that Docker `:rw` bind mounts remain writable in Codex app-server turns when OpenClaw sandboxing uses read-only workspace access.

## Best Possible Solution

Patch the Codex app-server sandbox policy to honor already-validated writable Docker bind mounts under read-only workspace access, with coverage proving `:rw` binds are writable, `:ro` binds stay read-only, and blocked bind-source validation is not weakened.

## Reproduction Assessment

Yes, at source level: with `workspaceAccess: "ro"`, current main passes only the sandbox workspace cwd into Codex `writableRoots`, so an additional `/data:rw` bind is absent from the app-server policy. I did not run the live Docker path in this read-only review.

## Solution Assessment

The issue direction is correct, but the exact suggested fix should not blindly add container targets until Codex app-server path semantics are confirmed. The maintainable fix should derive roots from existing validated bind parsing and test both `:rw` and `:ro` cases.
