## Summary

Keep open: current main still builds the Codex app-server sandbox policy with only the effective workspace in `writableRoots`, while Docker bind docs and bind parsing treat `:rw` bind mounts as independent writable surfaces. This is source-reproducible and security-sensitive, so a human owner should review the sandbox-boundary semantics before a fix changes writable roots.

## What This Changes

The issue reports that writable Docker bind mounts are not included in Codex app-server `writableRoots` when `workspaceAccess` is `ro`.

## Best Possible Solution

Patch the Codex app-server sandbox policy to honor already-validated writable Docker binds under read-only workspace access, with regression coverage for `:rw`, `:ro`, blocked bind-source validation, and host/container path semantics.

## Reproduction Assessment

Yes, at source level: with `workspaceAccess: "ro"`, current main passes only the sandbox workspace cwd into Codex `writableRoots`, so an additional `/data:rw` bind is absent from the app-server policy. I did not run a live Docker repro in this read-only review.

## Solution Assessment

The issue direction is correct, but the suggested fix should not blindly add bind targets until Codex app-server host/container path semantics are confirmed. The maintainable repair should derive writable roots from the existing validated bind parser and preserve blocked-source checks.
