## Summary

Keep open: current main and v2026.5.18 still send Codex app-server a `workspaceWrite` policy with only the effective workspace in `writableRoots`, while docs and bind parsing preserve `:rw` bind modes independently of `workspaceAccess`. This remains a source-reproducible sandbox regression and needs security-aware maintainer review before changing the writable-root boundary.

## What This Changes

The issue asks OpenClaw to include explicitly writable Docker bind mounts in Codex app-server `writableRoots` when `workspaceAccess` is `ro`, so shell writes to those mounts work without disabling sandboxing.

## Best Possible Solution

Update the Codex app-server sandbox policy to derive writable roots from already validated Docker bind metadata, honoring only `:rw` binds while preserving `:ro`, blocked-source, and reserved-target protections with focused regression and Docker smoke coverage.

## Reproduction Assessment

Yes at source level: with `workspaceAccess: "ro"`, current main uses `sandbox.workspaceDir` as the effective workspace and sends only that path in `writableRoots`, so an additional `/data:rw` bind is absent before `turn/start`. I did not run a live Docker repro because this review was read-only.

## Solution Assessment

Yes in direction, but the implementation must be security-aware. The narrow maintainable fix is to reuse validated bind parsing and add only proven writable bind roots after confirming Codex app-server host/container path semantics.
