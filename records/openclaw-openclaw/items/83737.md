- `impact:security`: The affected behavior is the sandbox filesystem boundary and Docker bind-mount permission model.

## Summary

Keep open: current main still builds the Codex app-server sandbox policy with only the effective workspace in `writableRoots`, while docs and sandbox bind parsing treat Docker `:rw` bind mounts as independent from `workspaceAccess`. The report is source-reproducible, but the repair is sandbox-boundary-sensitive because Codex app-server runs on the Gateway host and a naive target-path addition could broaden host writes.

## What This Changes

The issue asks OpenClaw to include operator-declared Docker `:rw` bind-mount targets in Codex app-server `writableRoots` when `workspaceAccess` is read-only.

## Best Possible Solution

Patch the Codex app-server sandbox policy only after owner/security review confirms the host-versus-container path contract, reusing the validated bind parser so `:rw` binds become writable, `:ro` binds remain read-only, and blocked-source validation is not weakened.

## Reproduction Assessment

Yes, at source level: with `workspaceAccess: "ro"`, current main passes only the sandbox workspace cwd into Codex `writableRoots`, so an additional `/data:rw` bind is absent from the app-server policy. I did not run a live Docker repro in this read-only review.

## Solution Assessment

Unclear until sandbox owners decide the path semantics. The issue direction is correct, but the fix should derive writable roots from existing validated bind parsing and prove host/container behavior rather than blindly adding container targets.
