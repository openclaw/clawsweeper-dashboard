## Summary

Keep this PR open. The latest head addresses several earlier review comments, but the sanitizer still strips only the fixed legacy sentinel list while the detector now rejects broader inbound metadata labels, so messages with Location or custom structured metadata can still lose legitimate auto-capture instead of storing clean user text.

## What This Changes

The PR adds memory-lancedb envelope metadata filtering and sanitization around auto-capture and recall, updates prompt escaping, expands extension tests, and adds a changelog entry.

## Best Possible Solution

Keep the plugin-level filter, but make `sanitizeForMemoryCapture` remove the same generic fenced inbound metadata block shape that `buildInboundUserContextPrefix` emits, with Location and structured-context auto-capture regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a `Location (untrusted metadata):` or custom structured metadata block prepended to `I always prefer dark mode` survives `sanitizeForMemoryCapture`, then `shouldCapture` rejects the sanitized candidate because the generic label remains.

## Solution Assessment

No. The plugin-level boundary is the right place to fix this, but the latest head broadens detection without broadening sanitization, so the narrow maintainable fix is to strip generic inbound metadata blocks before the capture gate.
