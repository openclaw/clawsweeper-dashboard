## Summary

Keep open: this is a useful, focused bug-fix PR for a real memory-lancedb contamination path, but the current head still misses parts of the current inbound metadata contract and lacks required real behavior proof.

## What This Changes

The PR adds envelope metadata detection and sanitization to memory-lancedb capture/recall, filters recalled memories, strips media annotations during prompt escaping, adds tests, and updates the changelog.

## Best Possible Solution

Keep the plugin-level fix, but align both detection and sanitization with the full current buildInboundUserContextPrefix contract, add regression coverage for reply-chain and generic metadata blocks, then require redacted real capture/recall proof.

## Reproduction Assessment

Yes, source-reproducible. Current main emits reply-chain and generic metadata blocks that the PR head does not fully strip or detect before memory capture and recall filtering.

## Solution Assessment

No. memory-lancedb is the right boundary for this hardening, but this branch needs to follow the current inbound metadata contract rather than relying on a partial fixed-list sanitizer and narrower detector.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Cover reply-chain inbound headers:** `extensions/memory-lancedb/index.ts:619-620`
  - body: Current main can emit `Reply chain of current user message (untrusted, nearest first):`, but this regex only accepts `(untrusted metadata)` or `(untrusted, for context)`. Body-only reply-chain JSON can also miss the narrow JSON-key fallback, so quoted reply context can still be captured and later recalled as memory sludge.
  - confidence: 0.91

  - confidence: 0.88
