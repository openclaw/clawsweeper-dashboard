## Summary

Keep this PR open. The memory-lancedb boundary is the right place for this fix and the branch supersedes the noisy closed #67475, but the current patch still misses current-main inbound metadata shapes and whitespace-tolerant sentinel handling, leaving concrete capture/recall bypasses before merge.

## What This Changes

The PR adds memory-lancedb envelope metadata detection and sanitization for auto-capture and recall, strips recalled media annotations, overfetches then filters recall results, and expands extension tests.

## Best Possible Solution

Revise the extension-local filter so it mirrors the current inbound metadata contract: tolerate leading whitespace, cover Location and arbitrary structured '(untrusted metadata):' JSON blocks, add focused regression tests for those cases, rebase onto current main, then run the memory-lancedb tests and changed gate.

## Reproduction Assessment

Yes. Feed the PR head functions an indented `Sender (untrusted metadata):` block or a current-main `Location (untrusted metadata):` / structured metadata block followed by a trigger like `I always prefer dark mode`; source inspection shows those forms can survive filtering and reach capture or recall.

## Solution Assessment

No, not as-is. Keeping the fix in memory-lancedb is the right boundary, but the detector and sanitizer need to fully track current core-generated inbound metadata forms before this is the narrowest maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Accept indented metadata sentinels:** `extensions/memory-lancedb/index.ts:533`
  - body: The detector only matches sentinels at column zero, while the canonical inbound metadata stripper treats a trimmed line as the sentinel. An indented `Sender (untrusted metadata):` block can bypass this guard and still be embedded or recalled if trigger text remains, so allow optional leading whitespace in the detector and sanitizer and add regression tests.
  - confidence: 0.9
- **[P2] Reject every current inbound metadata label:** `extensions/memory-lancedb/index.ts:498-505`
  - body: The copied sentinel list omits current `Location (untrusted metadata):` blocks and arbitrary structured labels such as `WhatsApp contact (untrusted metadata):`. Those payloads use fields the JSON fallback does not catch, so metadata followed by a memory trigger can still be persisted; use a generic line-anchored untrusted-metadata block rule or sync the full producer contract with tests.
  - confidence: 0.86
