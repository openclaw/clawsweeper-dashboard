## Summary

Keep open: the PR is a focused fix for a real memory/session poisoning path, but the current diff still misses current inbound metadata shapes and a documented envelope configuration, so it can leave the same class of sludge in auto-capture.

## What This Changes

The PR adds memory-lancedb envelope-sludge detection and sanitization, filters auto-capture and auto-recall, strips recalled media annotations, expands tests, and adds a changelog entry.

## Best Possible Solution

Align the memory-lancedb sanitizer/detector with the current inbound-meta and envelope contracts, or expose clean user content to hooks, then add regression tests for reply-chain/target blocks and markerless envelopes before merge.

## Reproduction Assessment

Yes from source inspection, though I did not execute a local repro in this read-only review. Current main stores raw user-role text, and the PR still misses source-visible current envelope labels and markerless envelope shapes that can carry MEMORY_TRIGGER text into memory capture.

## Solution Assessment

No. Sanitizing at memory-lancedb's capture and recall boundaries is a useful short-term fix, but this implementation needs to cover the current core metadata labels and documented envelope settings; the longer-term clean solution is a hook payload with raw user intent separate from envelope scaffolding.

## Review Findings

Full review comments:

- **[P1] Cover the current reply-chain metadata label:** `extensions/memory-lancedb/index.ts:619-620`
  - body: Current core emits `Reply chain of current user message (untrusted, nearest first):`, but this regex only recognizes `(untrusted metadata)` and `(untrusted, for context)`, and the sanitizer list still has the older `Replied message` name. A reply-chain JSON block with a quoted `always`/`prefer` body can therefore survive sanitize/shouldCapture and be stored as a contaminated memory. Add the current labels/patterns and regression tests from `buildInboundUserContextPrefix`.
  - confidence: 0.91
- **[P2] Handle markerless envelope headers:** `extensions/memory-lancedb/index.ts:659-660`
  - body: OpenClaw supports `envelopeTimestamp: "off"` and `envelopeElapsed: "off"`, and current tests show that produces headers like `[Telegram Alice] follow-up message`. This detector only catches `+elapsed` or weekday/date markers, so low-token envelope setups can still store `[channel sender]` and group sender prefixes with the user's memory. Cover that supported shape or move to a structural clean-content hook before relying on this filter.
  - confidence: 0.87
