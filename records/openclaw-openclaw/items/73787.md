Keep the plugin-level fix, but align detection and sanitization with the full current buildInboundUserContextPrefix contract and require redacted real capture/recall proof before merge.

No. The memory-lancedb boundary is the right place to filter stored/recalled memories, but this branch needs a sanitizer/detector that follows the current inbound metadata contract rather than a partial fixed list plus a narrower regex.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Cover current reply-chain metadata headers:** `extensions/memory-lancedb/index.ts:619-620`
  - body: Current core emits `Reply chain of current user message (untrusted, nearest first):`, but this detector only accepts `(untrusted metadata)` and `(untrusted, for context)` headers. Because that block is a JSON array, the envelope-JSON fallback will not reliably catch it either, so reply-chain sludge can still be captured and later recalled. Include the current reply-chain header shape in the detector/filter path.
  - confidence: 0.91
- **[P2] Strip generic metadata blocks before capture:** `extensions/memory-lancedb/index.ts:699-707`
  - body: `sanitizeForMemoryCapture` still strips only the explicit sentinel list, while current core emits `Location (untrusted metadata):` and arbitrary structured labels. Those blocks remain in `sanitized`, then `shouldCapture` rejects the whole message via the generic sludge detector, so a valid user preference accompanied by a location or structured-context block is skipped instead of stored cleanly. Strip the generic `formatUntrustedJsonBlock` shapes before calling `shouldCapture`.
  - confidence: 0.89
