## Summary

Keep open. The PR is a useful clean reimplementation of https://github.com/openclaw/openclaw/pull/67475, but the latest head still has source-proven gaps in the inbound-envelope and inbound-metadata coverage, and it lacks after-fix real behavior proof for memory-lancedb capture/recall.

## What This Changes

This PR adds envelope-sludge detection and sanitization to memory-lancedb auto-capture and auto-recall, filters recalled memories, strips media annotations before prompt escaping, and adds tests plus a changelog entry.

## Best Possible Solution

Keep the plugin-level hardening, but align detection and sanitization with the full current inbound metadata/envelope contract before requiring redacted real capture/recall proof.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: markerless formatInboundEnvelope output and compact chat-window context can pass the PR's sanitizer/detector in ways that either persist metadata or drop valid user memories.

## Solution Assessment

No. memory-lancedb is the right owner boundary for this hardening, but the current patch is still partial against the current inbound metadata contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Handle markerless inbound envelopes:** `extensions/memory-lancedb/index.ts:659-660`
  - body: `formatInboundEnvelope` can emit prefixes with no elapsed or timestamp marker, for example when timestamps are absent or disabled. This regex only detects brackets containing `+5m`-style elapsed text or a weekday/date, so auto-capture can still store `[Channel room] Sender: I prefer...` as polluted memory.
  - confidence: 0.86
- **[P2] Cover compact chat-window context:** `extensions/memory-lancedb/index.ts:619-620`
  - body: Current main can emit chat-window structured context as `<label> (untrusted, chronological, before current message):` followed by transcript lines, but this detector only recognizes `(untrusted metadata)` and exact `(untrusted, for context)` headers. Because the sanitizer also has no chat-window stripping path, trigger words in quoted chat history can still be captured and later recalled as memory.
  - confidence: 0.9
- **[P2] Strip generic metadata blocks before capture gating:** `extensions/memory-lancedb/index.ts:757-765`
  - body: `sanitizeForMemoryCapture` still strips only the fixed sentinel list, while current core emits `Location (untrusted metadata):` and arbitrary structured-context labels. Those blocks remain in `sanitized`, then `shouldCapture` rejects the whole message via the generic detector, so a valid user preference accompanied by location or structured context is skipped instead of stored cleanly.
  - confidence: 0.88
