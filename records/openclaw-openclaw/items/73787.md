## Summary

Keep open. The latest head addresses the changelog and earlier whitespace-formatting feedback, but the new filter still misses current metadata forms: indented metadata headers and generic `Location` / structured untrusted metadata blocks emitted by current main.

## What This Changes

The PR adds memory-lancedb envelope metadata detection and sanitization for auto-capture and recall, filters recalled memories, strips recalled media annotations, expands tests, and adds a changelog entry.

## Best Possible Solution

Keep the fix extension-local, but make the detector and sanitizer mirror the current inbound metadata producer contract with optional leading whitespace and generic `(untrusted metadata):` JSON blocks, then add focused regression tests.

## Reproduction Assessment

Yes. Feed PR-head `shouldCapture`/recall filtering an indented plain-text `Sender (untrusted metadata):` block or a `Location (untrusted metadata):` / `WhatsApp contact (untrusted metadata):` JSON block containing a trigger such as `always`; source inspection shows those forms can bypass the new filters.

## Solution Assessment

No, not as-is. The extension boundary is right, but the current hardcoded labels and column-zero regexes are narrower than the core metadata contract; a generic line-anchored rule with optional leading whitespace is the safer maintainable path.
