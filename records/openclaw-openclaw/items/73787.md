## Summary

Keep this PR open. The latest head is a focused plugin-side fix for a real memory-lancedb contamination bug, but it still has a narrow sanitizer gap: generic inbound metadata blocks are detected as sludge but not stripped before auto-capture, so valid user facts behind Location or structured-context metadata are dropped.

## What This Changes

This PR adds memory-lancedb envelope metadata detection, capture sanitization, contaminated recall filtering, media-annotation stripping, regression tests, and a changelog entry.

## Best Possible Solution

Keep the plugin-level filtering approach, but make sanitizeForMemoryCapture strip the same generic fenced inbound metadata block shape that buildInboundUserContextPrefix emits, then cover Location and arbitrary structured-context auto-capture cases.

## Reproduction Assessment

Yes. From source inspection, a Location or arbitrary structured untrusted metadata JSON block prepended before `I always prefer dark mode` survives sanitizeForMemoryCapture and then causes shouldCapture to reject the whole sanitized candidate.

## Solution Assessment

No. The plugin boundary is the right place to solve this, but the current PR broadens detection without matching sanitizer coverage for current generic inbound metadata labels.
