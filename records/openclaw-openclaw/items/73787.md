## Summary

Keep open: current head fixes the earlier detection and formatting review comments, but the sanitizer still only removes the six explicit sentinel blocks, so current Location/structured metadata prefixes can make otherwise valid user memories get dropped instead of sanitized and captured.

## What This Changes

The PR adds memory-lancedb envelope metadata detection/sanitization, filters contaminated recall results, strips media annotations from recalled memories, expands tests, and adds a changelog entry.

## Best Possible Solution

Broaden sanitizeForMemoryCapture to remove generic line-start <label> (untrusted metadata): and (untrusted, for context): fenced blocks emitted by buildInboundUserContextPrefix, then add Location and structured-context auto-capture regression tests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a Location or structured untrusted metadata block prepended to 'I always prefer dark mode' survives sanitization, then shouldCapture rejects the candidate because the generic label is still present.

## Solution Assessment

No. The plugin-level filtering boundary is right, but the sanitizer should share the current generic inbound-metadata block shape instead of relying on a stale fixed sentinel list.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Strip generic metadata blocks before capture:** `extensions/memory-lancedb/index.ts:556-563`
  - body: The sanitizer still only removes the six fixed sentinels listed here, but current core also prefixes user text with `Location (untrusted metadata):` and arbitrary structured `<label> (untrusted metadata):` JSON blocks. Please strip the generic fenced inbound-metadata block shape before the capture gate and cover Location/structured cases.
