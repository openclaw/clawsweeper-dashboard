## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Cover current reply-chain metadata headers:** `extensions/memory-lancedb/index.ts:619-620`
  - body: Current main emits `Reply chain of current user message (untrusted, nearest first):`, but this regex only accepts `(untrusted metadata)` and `(untrusted, for context)` headers. The JSON fallback is not enough for body-only reply-chain payloads, where `buildReplyChainPayload` can omit all selected identifier keys, so quoted trigger text can still look capturable. Include the current reply-chain header shape in the detector and sanitizer path.
  - confidence: 0.9
- **[P2] Strip generic metadata blocks before capture gating:** `extensions/memory-lancedb/index.ts:757-765`
  - body: `sanitizeForMemoryCapture` still iterates only the fixed sentinel list, while current core emits `Location (untrusted metadata):` and arbitrary structured-context labels. Those blocks remain in `sanitized`, then `shouldCapture` rejects the whole message via the generic detector, so a valid user preference accompanied by location or structured context is skipped instead of stored cleanly.
  - confidence: 0.91
- **[P3] Preserve direct-message text before colon labels:** `extensions/memory-lancedb/index.ts:749-753`
  - body: After any envelope prefix is removed, the sanitizer unconditionally strips the first `Name: `-shaped prefix. That is correct for group envelopes where OpenClaw injects the sender label, but direct-message envelopes preserve the raw user body, so a direct message like `Project X: I prefer dark mode` would be stored without its subject. Only remove the sender prefix when the envelope shape proves it was injected, or preserve the body text.
  - confidence: 0.78
