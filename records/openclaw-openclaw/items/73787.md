## Summary

Keep this PR open. The change targets a real gap in current main, and the closed #67475 is explicitly superseded by this cleaner branch, but the PR still leaves concrete memory privacy/security bypasses in the filter semantics before it is safe to merge.

## What This Changes

The branch adds memory-lancedb envelope metadata detection and sanitization for auto-capture, filters contaminated recall results, strips recalled media annotations, and expands the extension tests.

## Best Possible Solution

Revise the extension-local filter so it mirrors the current inbound metadata contract: accept leading whitespace, cover Location and arbitrary structured untrusted metadata labels, add focused regression tests for those cases, rebase onto current main, and then run the memory-lancedb tests plus the changed gate.

## Reproduction Assessment

Yes. Source inspection gives a direct reproduction: apply the PR head behavior to an indented `Sender (untrusted metadata):` block or a `Location (untrusted metadata):`/structured contact block followed by a memory trigger such as 'I always prefer dark mode'; the metadata can survive into capture or recall filtering.

## Solution Assessment

No, not as-is. Keeping the fix inside memory-lancedb is the right boundary, but the detector and sanitizer need to fully track current core-generated inbound metadata forms before this is the narrowest maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Accept indented metadata sentinels:** `extensions/memory-lancedb/index.ts:533`
  - body: The new detector and plaintext sanitizer match sentinels with `^${escapedSentinel}`, but the canonical inbound metadata stripper treats `line.trim()` as the sentinel. An indented `Sender (untrusted metadata):` block can therefore bypass the new guard and still be embedded/stored if trigger text remains. Allow optional leading whitespace and add indented-sentinel tests.
  - confidence: 0.9
- **[P2] Reject every current inbound metadata label:** `extensions/memory-lancedb/index.ts:498-505`
  - body: The copied sentinel list omits current `Location (untrusted metadata):` blocks and arbitrary structured labels such as `WhatsApp contact (untrusted metadata):`. Those payloads use fields like `latitude`, `longitude`, `source`, `type`, and `payload`, which the JSON-key fallback does not catch, so metadata followed by a memory trigger can still be persisted. Add generic line-anchored untrusted-metadata handling or sync the full producer contract with tests.
  - confidence: 0.86

## Security Review
