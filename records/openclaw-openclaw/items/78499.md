## Summary

Current main preserves the documented doctor migration to `openai/*` plus `agentRuntime.id: "codex"`, but now forwards an `openai-codex` auth profile into the Codex harness, so the reported auth-profile break is fixed after `v2026.5.5`.

## What This Changes

The issue reports that `openclaw doctor --fix` rewrites Codex runtime model references to `openai/*`, causing Codex runtime turns to select `openai:default` instead of an `openai-codex` auth profile.

## Best Possible Solution

Keep the canonical doctor migration and ship the current-main Codex harness auth forwarding so `v2026.5.5` users no longer need the `codex/*` workaround.

## Reproduction Assessment

Yes for the released `v2026.5.5` behavior: the issue body provides a concrete doctor-to-Telegram path and logs showing the Codex harness rejected `openai:default`. A failing current-main path is not present in source inspection because current main now selects and forwards `openai-codex` auth for the same route.

## Solution Assessment

Yes. Fixing auth forwarding for `openai/*` plus `agentRuntime.id: "codex"` is narrower and more consistent with the documented canonical model refs than reverting doctor back to persistent `codex/*` refs.
