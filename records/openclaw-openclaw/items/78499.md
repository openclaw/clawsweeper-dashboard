## Summary

This looks like a valid current-main bug, not cleanup-closeable: the canonical `openai/*` plus Codex runtime route is documented and tested, but stale or preserved public OpenAI auth-profile overrides can still be forwarded into the Codex app-server and trigger the reported provider mismatch.

## What This Changes

The issue asks `doctor --fix` and subsequent Telegram Codex-runtime turns to avoid selecting `openai:default` when the agent should use an `openai-codex` auth profile.

## Best Possible Solution

Keep the canonical `openai/*` plus `agentRuntime.id: "codex"` route, but make doctor/session preparation clear or replace incompatible `openai:*` auth overrides when the Codex harness is selected, with regression coverage for Telegram-style persisted sessions.

## Reproduction Assessment

Yes, source-reproducible: a persisted `openai:default` auth override on an `openai/gpt-5.5` session with Codex runtime is allowed through session preparation and then rejected by the Codex app-server auth bridge. I did not perform a live Telegram run in this read-only pass.

## Solution Assessment

No, the current doctor/session repair is incomplete for this path. The narrow fix is to treat `openai:*` auth overrides as incompatible with the Codex harness and select or fall back to an `openai-codex` profile instead of reverting to legacy `codex/*` model refs.
