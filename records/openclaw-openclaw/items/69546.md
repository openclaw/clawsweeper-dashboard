## Summary

Keep open. Current main still feeds QQBot inbound routing from the gateway startup config snapshot while bindings remain classified as no-restart config, and PR #73567 is the open closing implementation path.

## What This Changes

The issue asks QQBot routing to apply peer-specific binding edits on the next inbound message without requiring a gateway restart.

## Best Possible Solution

Review and land PR #73567, or close it and apply an equivalent narrow QQBot-side fix that fetches active runtime config for each inbound routing decision.

## Reproduction Assessment

Yes. The reporter gave concrete QQBot C2C steps and before/after routing logs, and current main still passes the startup `ctx.cfg` into the route decision. I did not run a live QQBot account in this read-only review.

## Solution Assessment

Yes. Resolving the active runtime config at the QQBot inbound boundary is the narrowest maintainable fix; making binding edits restart-required or changing generic routing cache policy would be broader than necessary.
