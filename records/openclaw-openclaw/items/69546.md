## Summary

Keep open. Current main still routes QQBot inbound messages through the gateway startup config snapshot while `bindings` is classified as no-restart config, and the linked closing PR #73567 is open and unmerged, so closing this issue before that PR lands would be premature.

## What This Changes

QQBot should apply newly added peer-specific routing bindings to the next inbound message without requiring a gateway restart.

## Best Possible Solution

Review and land PR #73567, or close it and apply an equivalent narrow QQBot-side fix that resolves the active runtime config for each inbound routing decision and covers account-to-peer binding changes without a gateway restart.

## Reproduction Assessment

Yes. The reporter provided concrete steps and before/after routing logs, and current main's source still routes QQBot inbound messages from `ctx.cfg`; I did not run a live QQBot account in this read-only review.

## Solution Assessment

Yes. Fetching the active runtime config at the QQBot gateway boundary is the narrowest maintainable fix; changing generic route caching or making binding edits restart-required would be broader and would conflict with current reload semantics.
