## Summary

Keep open: current main still batches memory embeddings from the per-file indexing path, and this PR is the active implementation for the reported source-wide OpenAI-compatible batch behavior. The branch has useful real CLI/provider proof, but it still needs maintainer acceptance of the plugin-facing runtime opt-in and fixed JSONL byte-cap policy before merge.

## What This Changes

The PR batches dirty memory/session embedding chunks source-wide for opted-in providers, enables that path for OpenAI-compatible embeddings, adds JSONL byte-cap splitting and polling backoff/progress, and updates docs/tests/status handling.

## Best Possible Solution

Land the source-wide batch path only after maintainers explicitly accept the runtime opt-in contract and provider byte-cap policy, keeping the opt-in documented and covered by focused contract/runtime tests.

## Reproduction Assessment

Yes. Source inspection of current main shows batch embedding is invoked from each file's indexFile path, and the linked report plus PR discussion include logs showing one provider batch per file-boundary sequence before the branch behavior.

## Solution Assessment

Mostly yes. The explicit sourceWideBatchEmbed opt-in is the safer maintainable shape than changing all batch providers by default, but the final merge decision is whether maintainers want this new runtime contract and fixed provider byte-cap policy.
