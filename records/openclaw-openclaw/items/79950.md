## Summary

Keep open. The current `sessions_send` A2A path still matches the reported source-level failure mode: async sends start the reply-back/announce flow, the reply loop can inject repeated inter-session wrapper turns into the requester, and current tests intentionally preserve ping-pong/target-announce behavior, so a fix needs a maintainer contract decision rather than cleanup closure.

## What This Changes

The issue requests clean one-time delivery of async `sessions_send` results back to a Telegram-bound requester without leaking inter-session wrappers, duplicating completions, or alternating routing provenance.

## Best Possible Solution

Define the intended async A2A result contract, then update `sessions_send` so Telegram-bound requester sessions receive one clean completion or explicitly opt into the existing conversation loop, with regression coverage for wrapper suppression and route preservation.

## Reproduction Assessment

Yes for the source-level A2A duplicate/wrapper path: current main starts the A2A flow for `timeoutSeconds: 0`, and tests expect repeated inter-session agent steps. The full Telegram external delivery symptom still needs live channel proof before a risky behavior change.

## Solution Assessment

Unclear. Disabling the A2A flow globally is not the best fix because current docs and tests preserve conversation semantics; the safer path is an explicit maintainer decision about async dispatch versus requester delivery semantics.
