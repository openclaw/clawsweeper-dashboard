## Summary

Keep open. Current main still starts the A2A reply-back and announce flow for `timeoutSeconds: 0`, and that flow can inject inter-session wrapper turns into the requester; the Telegram delivery symptom is plausible from source but still needs live channel proof before changing the contract.

## What This Changes

The issue requests clean single delivery of async `sessions_send` results back to a Telegram-bound requester without duplicate internal wrappers or route drift.

## Best Possible Solution

Define the intended async A2A result contract, then update `sessions_send` so Telegram-bound requester sessions either receive one clean external completion or explicitly opt into the existing conversation loop, with regression coverage for wrapper suppression and route preservation.

## Reproduction Assessment

Yes for the source-level duplicate/wrapper path: current main starts A2A follow-up after `timeoutSeconds: 0` and annotates injected turns as inter-session messages. The full Telegram delivery symptom still needs a live channel reproduction.

## Solution Assessment

Unclear. Disabling the A2A flow globally would not be the best fix because docs and tests preserve conversation semantics; maintainers should first decide async dispatch versus requester delivery semantics.
