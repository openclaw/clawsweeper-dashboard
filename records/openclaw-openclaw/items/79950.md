## Summary

Keep open. The report matches a real current-main tension in `sessions_send`: async sends still start the A2A reply-back/announce flow, which can inject inter-session wrapper turns into the requester and is already related to open discussions about ping-pong duplicates and clean handoff semantics.

## What This Changes

The issue asks for async `sessions_send` results from a Telegram-bound requester to be delivered once, without internal wrapper leakage, duplicate ping-pong, or `webchat`/`telegram` routing drift.

## Best Possible Solution

Define the intended async A2A result contract, then update `sessions_send` so Telegram-bound requester sessions either receive one clean external completion or explicitly opt into the existing conversation loop, with regression coverage for wrapper suppression and route preservation.

## Reproduction Assessment

Yes for the source-level A2A duplicate/wrapper path: current main starts the reply-back flow for `timeoutSeconds: 0` and tests expect multiple inter-session agent steps. The full Telegram delivery symptom still needs a live channel reproduction before making a risky behavior change.

## Solution Assessment

Unclear. Disabling the A2A flow globally would not be the best fix because current docs/tests preserve conversation semantics; a safer solution needs an explicit maintainer decision about async dispatch versus requester delivery semantics.
