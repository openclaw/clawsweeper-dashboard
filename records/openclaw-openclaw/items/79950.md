## Summary

Keep open. Current main still starts the async `sessions_send` A2A follow-up path for `timeoutSeconds: 0`, can re-inject inter-session wrapper turns, and still routes the final announce around the target session rather than proving one clean Telegram requester delivery.

## What This Changes

The issue requests one clean Telegram-delivered async `sessions_send` completion for the requester session, without inter-session wrapper leakage, duplicate ping-pong, or `sourceChannel` flips.

## Best Possible Solution

Define the intended async A2A result contract, then update `sessions_send` so Telegram-bound requester sessions get one clean external completion or explicitly opt into the current conversation loop.

## Reproduction Assessment

## Solution Assessment

Unclear. Disabling the A2A flow globally is not the best fix because current docs preserve peer conversation semantics; maintainers should first choose the requester-delivery versus dispatch/conversation contract.
