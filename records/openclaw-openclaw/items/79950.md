## Summary

Keep open. The report matches current-main `sessions_send` behavior: async sends still enter the A2A reply-back/announce flow, which can inject inter-session wrapper turns, while the Telegram-specific external delivery result still needs maintainer contract work rather than cleanup closure.

## What This Changes

The issue asks for async `sessions_send(timeoutSeconds: 0)` from a Telegram-bound requester to deliver one clean result without duplicate inter-session wrappers or routing flips.

## Best Possible Solution

Define the intended async A2A result contract, then update `sessions_send` so Telegram-bound requester sessions either receive one clean external completion or explicitly opt into the existing conversation loop, with regression coverage for wrapper suppression and route preservation.

## Reproduction Assessment

Yes for the source-level A2A duplicate/wrapper path: current main starts the reply-back flow for `timeoutSeconds: 0` and tests expect inter-session agent steps. The full Telegram delivery symptom still needs a live channel run.

## Solution Assessment

Unclear. Disabling A2A globally is not the best fix because current docs/tests preserve conversation semantics; the maintainable path needs a maintainer decision about async dispatch versus requester delivery semantics.
