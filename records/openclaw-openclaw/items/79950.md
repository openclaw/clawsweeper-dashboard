## Summary

Keep open. Current `main` still runs the async `sessions_send` path through the A2A reply-back/announce flow, and the reported duplicate wrapper/source-channel behavior overlaps with but is not fully superseded by the open A2A duplicate and dispatch-mode threads.

## What This Changes

The issue reports that `sessions_send` with `timeoutSeconds: 0` from a Telegram-bound requester can duplicate inter-session wrapper payloads and fail to deliver one clean result back to Telegram.

## Best Possible Solution

Define the intended async A2A result contract, then make Telegram-bound requester sessions receive one route-preserving clean completion or explicitly opt into the existing conversation loop, with regression coverage for wrapper suppression and provenance preservation.

## Reproduction Assessment

Yes for the source-level A2A duplicate/wrapper path: current main starts the follow-up A2A flow for `timeoutSeconds: 0`, and tests expect multiple inter-session agent steps. The full Telegram chat delivery symptom still needs live channel proof before a risky routing change.

## Solution Assessment

Unclear. Disabling A2A globally is not the best fix because docs and tests preserve conversation semantics; the maintainable path needs a deliberate async delivery contract and targeted Telegram/requester routing coverage.
