## Summary

Keep open. ## What This Changes

The issue requests that async `sessions_send` from a Telegram-bound requester deliver one clean target result without wrapper leakage, duplicate ping-pong, or channel provenance flips.

## Best Possible Solution

Keep conversation mode explicit, then add regression-covered async Telegram delivery handling so one result is routed cleanly without internal wrapper text or duplicate A2A turns.

## Reproduction Assessment

Partly. Source inspection gives high confidence for the duplicate/wrapper A2A path because async sends still start the reply/announce loop, but the exact Telegram delivery failure still needs a live channel reproduction before changing routing behavior.

## Solution Assessment

Unclear. A global A2A disable would be risky because docs and tests preserve conversation semantics; maintainers should choose the intended async result contract before implementation.
