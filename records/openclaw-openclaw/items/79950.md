Keep open. ## What This Changes

The issue requests clean one-time Telegram delivery of async `sessions_send` results without inter-session wrapper leakage, duplicate ping-pong, or source-channel route churn.

## Best Possible Solution

Define the async A2A result semantics, then make Telegram-bound requester sessions receive at most one clean completion or an explicit conversation-loop event with wrapper suppression and route preservation.

## Reproduction Assessment

Yes for the source-level duplicate/wrapper path: current main still starts A2A follow-up handling for `timeoutSeconds: 0`, and the loop can reinject inter-session wrapper turns. The full Telegram delivery symptom still needs live Telegram proof before a risky behavior change.

## Solution Assessment

Unclear. Globally disabling the A2A flow is not the best fix because docs and tests preserve conversation semantics; the safer direction is a maintainer-approved async delivery contract with targeted route and wrapper regression coverage.
