## Summary

Keep open. Current main still runs the async `sessions_send` A2A follow-up path for `timeoutSeconds: 0`, and that path can inject inter-session wrapper turns and announce from the target route; the Telegram-specific external delivery failure still needs live proof before changing the contract.

## What This Changes

The issue asks for async `sessions_send` results from a Telegram-bound requester to be delivered once as a clean Telegram message without wrapper leakage, duplicate ping-pong copies, or route drift.

## Best Possible Solution

Define the intended async A2A result contract, then update `sessions_send` so Telegram-bound requester sessions get one clean external completion or explicitly opt into the existing conversation loop, with wrapper suppression, route preservation, and live Telegram proof.

## Reproduction Assessment

Yes for the source-level duplicate/wrapper path: current main still starts A2A follow-up for `timeoutSeconds: 0`, serializes inter-session wrappers, and tests preserve ping-pong behavior. No live Telegram run was performed here, so the exact external delivery miss remains unproven in this review.

## Solution Assessment

Unclear. Disabling the A2A flow globally is not the best fix because current docs/tests preserve peer conversation semantics; maintainers should first choose the requester-delivery versus dispatch/conversation contract.
