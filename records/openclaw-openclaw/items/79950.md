## Summary

Keep open. Current main still schedules async `sessions_send` into the A2A reply-back and announce flow, which can inject inter-session wrapper turns and preserve the reported provenance confusion; the exact Telegram delivery failure still needs live verification before changing the contract.

## What This Changes

Ensure async `sessions_send` results from Telegram-bound requester sessions are delivered once cleanly without leaking inter-session wrappers, duplicating replies, or losing Telegram routing.

## Best Possible Solution

Define the intended async A2A result contract, then update `sessions_send` so Telegram-bound requester sessions receive one clean external completion or explicitly remain in conversation mode, with regression coverage for wrapper suppression and route preservation.

## Reproduction Assessment

Yes for the source-level duplicate/wrapper path: current main schedules A2A follow-up for `timeoutSeconds: 0` and tests expect nested inter-session steps. No live high-confidence Telegram external-delivery reproduction was run here.

## Solution Assessment

Unclear. A global A2A skip is not the best fix because current docs and tests preserve conversation semantics; maintainers should decide the async requester-delivery contract before implementation.
