## Summary

Keep open. The report is source-backed on current main: async `sessions_send` still enters the A2A reply-back/announce flow, which can inject inter-session wrapper turns and overlaps active unresolved A2A ping-pong and Telegram routing work.

## What This Changes

The issue reports that async `sessions_send` from a Telegram-bound requester can duplicate inter-session wrapper results and fail to deliver one clean Telegram-bound completion.

## Best Possible Solution

Define the intended async A2A delivery contract, then update `sessions_send` so Telegram-bound requester sessions receive one clean external completion or explicitly stay in conversation-mode A2A with wrapper suppression and route-preservation coverage.

## Reproduction Assessment

Yes for the source-level A2A duplicate/wrapper path: current main starts reply-back/announce handling for `timeoutSeconds: 0`, and tests expect multiple inter-session turns. The full Telegram delivery symptom still needs a live channel reproduction before making a risky behavior change.

## Solution Assessment

Unclear. Globally disabling A2A follow-up is not the best fix because current docs and tests preserve conversation semantics; maintainers should first choose the async A2A contract and then scope route/wrapper suppression accordingly.
