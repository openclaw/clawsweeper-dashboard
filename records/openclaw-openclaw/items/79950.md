## Summary

Keep open. Current main and the latest shipped release still preserve the async `sessions_send` A2A follow-up path that can reinject inter-session wrapper turns and alternate requester/target provenance; the exact Telegram delivery miss still needs live proof before changing the contract.

## What This Changes

The issue asks for async `sessions_send` results from a Telegram-bound requester to be delivered once as a clean Telegram message without wrapper leakage or route-provenance flips.

## Best Possible Solution

Define the async A2A result contract, then make `sessions_send` deliver one clean Telegram-bound requester result or require explicit reply-back opt-in, with wrapper suppression, route preservation, tests, and live Telegram proof.

## Reproduction Assessment

Yes for the source-level wrapper and ping-pong path: current main still starts A2A follow-up for `timeoutSeconds: 0`, serializes inter-session wrappers, and tests preserve reply-back behavior. The exact Telegram outbound delivery miss still needs a live Telegram proof run.

## Solution Assessment

Unclear. Disabling A2A globally is not the best fix because current docs and tests preserve peer conversation semantics; maintainers should first choose the requester-delivery versus reply-back contract.
