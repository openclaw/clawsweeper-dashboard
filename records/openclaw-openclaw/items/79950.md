## Summary

Keep open. Current main still has the source-level async `sessions_send` A2A path that can inject wrapper turns and alternate requester/target provenance, while the exact Telegram delivery contract overlaps open A2A product decisions rather than a safe cleanup close.

## What This Changes

The issue requests clean once-only delivery of async `sessions_send` results back to a Telegram-bound requester session without inter-session wrapper leakage or duplicate ping-pong turns.

## Best Possible Solution

Define the intended async A2A result contract, then update `sessions_send` to preserve Telegram requester routing, suppress user-visible internal wrappers, and avoid duplicate ping-pong without breaking existing conversation-mode A2A semantics.

## Reproduction Assessment

Yes for the source-level A2A duplicate/wrapper path: current main still starts the reply-back flow for `timeoutSeconds: 0` and tests preserve multiple inter-session steps. The exact Telegram outbound delivery failure still needs a live channel reproduction before changing the contract.

## Solution Assessment

Unclear. Disabling the A2A flow globally is not the best fix because current docs and tests preserve conversation semantics; maintainers should choose whether async means dispatch, conversation, requester announce, or a new explicit mode.
