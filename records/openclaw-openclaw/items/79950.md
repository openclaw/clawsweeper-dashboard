## Summary

Keep open. Current main still starts the A2A reply-back and announce flow for async `sessions_send`, which can re-inject wrapper-prefixed inter-session turns into the requester; the exact Telegram delivery symptom still needs live proof and a maintainer decision alongside https://github.com/openclaw/openclaw/issues/44309 and https://github.com/openclaw/openclaw/issues/39476.

## What This Changes

The issue asks for async `sessions_send(timeoutSeconds: 0)` results from a Telegram-bound requester to arrive once as a clean Telegram message without inter-session wrappers, duplicate ping-pong, or route drift.

## Best Possible Solution

Define the intended async A2A result contract, then update `sessions_send` so Telegram-bound requester sessions either receive one clean external completion or explicitly opt into the existing conversation loop, with regression coverage for wrapper suppression and route preservation.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Partly: the duplicate/wrapper path is source-reproducible on current main because async sends still enter A2A reply-back and announce flow, but the full Telegram external-delivery symptom was not live-reproduced here.

## Solution Assessment

Is this the best way to solve the issue? Unclear: disabling A2A globally is not safe because current docs and tests preserve conversation semantics; maintainers need to choose the async requester-delivery contract before implementation.
