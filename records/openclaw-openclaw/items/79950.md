## Summary

Keep open. Current main still has a source-backed async sessions_send path that schedules A2A reply-back/announce handling for timeoutSeconds: 0, can reinject inter-session wrapper turns, and overlaps but is not fully replaced by the open A2A duplicate and dispatch-mode issues.

## What This Changes

The issue reports that async sessions_send results from a Telegram-bound requester can duplicate internal inter-session wrappers and fail clean Telegram delivery.

## Best Possible Solution

Define the intended async A2A result contract, then implement a route-preserving sessions_send path that delivers one clean requester-visible result or explicitly opts into the existing conversation loop.

## Reproduction Assessment

Yes for the source-level wrapper and duplicate path: current main starts A2A follow-up for timeoutSeconds: 0 and nested A2A steps annotate inter-session messages. The full Telegram delivery symptom was not live-reproduced in this read-only review.

## Solution Assessment

Unclear. Disabling the A2A flow globally is not the best fix because current docs and tests preserve conversation semantics; maintainers should first choose the async requester-delivery versus conversation contract.
