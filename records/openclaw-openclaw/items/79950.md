Keep open. The report is concrete, current-main source still routes async `sessions_send` through the A2A reply-back and target-announce flow, and the remaining behavior is not fully covered by the adjacent closed Telegram delivery fixes.

## What This Changes

The issue requests clean once-only delivery of async `sessions_send` results back to a Telegram-bound requester session without leaking inter-session wrappers or flipping route provenance.

## Best Possible Solution

Define the async A2A delivery contract, then update `sessions_send` so Telegram-bound requester sessions get one clean external completion or explicitly opt into the existing conversation loop, with regression coverage for wrapper suppression and route preservation.

Yes for the source-level duplicate/wrapper path: current main still starts A2A flow for `timeoutSeconds: 0`, wraps inter-session messages, and keeps nested steps internal. The full Telegram delivery symptom still needs live channel proof.

## Solution Assessment

Unclear. Disabling the A2A flow globally would conflict with current docs/tests; the safer solution needs an explicit maintainer decision on async dispatch versus reply-back and announce semantics.
