## Summary

Keep open. Current main still sends async `sessions_send` through the A2A reply-back/announce path, which can inject inter-session wrapper turns and alternating provenance into the requester session; the exact Telegram delivery symptom still needs live channel proof before changing the contract.

## What This Changes

Ensure async `sessions_send` from a Telegram-bound requester delivers the target result once cleanly without inter-session wrapper leakage, duplicate ping-pong, or route provenance flips.

## Best Possible Solution

Define the async A2A delivery contract, then update `sessions_send` so Telegram-bound requester sessions either get one clean completion or explicitly opt into the existing conversation loop, with route-preservation and wrapper-suppression coverage.

## Reproduction Assessment

Partly. There is a high-confidence source path for the duplicate/wrapper behavior because `timeoutSeconds: 0` still schedules A2A reply-back/announce handling; the full Telegram delivery symptom still needs a live Telegram reproduction.

## Solution Assessment

Unclear. Disabling the A2A loop globally is not the best fix because current docs and tests preserve conversation semantics; maintainers should first decide the async A2A result and announce contract.
