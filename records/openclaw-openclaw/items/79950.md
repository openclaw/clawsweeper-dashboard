## Summary

Keep open. Current main still starts the A2A reply-back/announce path for async `sessions_send`, can inject `sessions_send` inter-session wrapper turns into the requester session, and the documented/tested conversation semantics make this a maintainer contract decision rather than a cleanup-close candidate.

## What This Changes

Requests that async `sessions_send` from a Telegram-bound requester deliver one clean result without leaked inter-session wrappers, duplicate ping-pong, or Telegram route flips.

## Best Possible Solution

Define the async A2A delivery contract, then update `sessions_send`, visible-history projection, and routing tests so Telegram-bound requester sessions either receive one clean completion or explicitly opt into the existing conversation loop.

## Reproduction Assessment

Yes for the source-level A2A duplicate/wrapper path: current main starts the reply-back flow for `timeoutSeconds: 0`, and tests expect multiple inter-session agent steps. The full Telegram delivery symptom still needs live channel reproduction before a risky behavior change.

## Solution Assessment

Unclear. Disabling A2A globally is not the best fix because current docs and tests preserve conversation semantics; maintainers need to decide whether async sends mean dispatch, requester delivery, target announce, or the current reply-back loop.
