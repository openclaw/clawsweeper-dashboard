## Summary

Keep open. Current main still routes async `sessions_send` through the A2A reply-back and announce flow, so the reported duplicate/wrapper path is source-reproducible and overlaps unresolved A2A delivery semantics rather than a cleanup-close case.

## What This Changes

The issue reports that `sessions_send(..., timeoutSeconds: 0)` from a Telegram-bound requester can produce duplicate inter-session wrapper payloads and fail to deliver one clean result back to the requester chat.

## Best Possible Solution

Define the intended async A2A result contract, then update `sessions_send` so Telegram-bound requesters receive one clean completion or explicitly opt into the existing conversation loop, with route-preservation and wrapper-suppression coverage.

## Reproduction Assessment

Yes for the source-level duplicate/wrapper path: current main starts A2A follow-up for `timeoutSeconds: 0`, and tests expect multiple inter-session steps. The exact Telegram outbound delivery failure still needs live channel proof.

## Solution Assessment

Unclear. A global A2A disable is not the best fix because current docs/tests preserve conversation behavior; the safer path is an explicit contract decision that coordinates this issue with https://github.com/openclaw/openclaw/issues/39476 and https://github.com/openclaw/openclaw/issues/44309.
