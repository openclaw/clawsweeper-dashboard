## Summary

Keep open. Current main has a useful partial fix for sessions_send handoffs preserving external delivery routes, but the async A2A path still runs the bounded ping-pong/announce flow and can inject inter-session wrapper turns before announcing from the target route, so the full Telegram-bound requester delivery problem is not proven resolved.

## What This Changes

The issue reports that async `sessions_send` results from a Telegram-bound requester session can be duplicated, leak inter-session wrapper text, and fail clean Telegram delivery.

## Best Possible Solution

Decide the intended async A2A delivery contract, then make requester-bound completion delivery single-owner, route-preserving, and wrapper-safe with focused regression coverage plus live Telegram proof.

## Reproduction Assessment

Partially yes: source inspection shows the current async `sessions_send` path still runs A2A wrapper/ping-pong and target announce logic. The exact Telegram-visible missing-delivery symptom still needs a live Telegram reproduction on current main.

## Solution Assessment

Unclear: the current route-preservation fix is a good partial repair, but it does not settle whether async peer A2A should deliver through the requester route, target route, or a different single-owner completion path.
