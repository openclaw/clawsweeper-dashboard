## Summary

Keep open. Current main still starts the async `sessions_send` A2A follow-up path for `timeoutSeconds: 0`, still injects inter-session wrapper turns, and still resolves final announce delivery from the target side; related PRs and issues are active but unresolved.

## What This Changes

The issue asks for async `sessions_send` results from a Telegram-bound requester to be delivered once as a clean requester-routed Telegram message without wrapper leakage or channel ping-pong.

## Best Possible Solution

Define the async A2A delivery contract, then update `sessions_send` so a Telegram-bound requester either receives one clean requester-routed completion or explicitly opts into conversation-mode follow-up, with wrapper suppression and route-preservation coverage.

## Reproduction Assessment

Yes for the source-level path: current main still starts A2A for `timeoutSeconds: 0`, wraps inter-session turns, and can alternate requester/target provenance. The full Telegram delivery symptom still needs live channel proof before a risky behavior change ships.

## Solution Assessment

Unclear. Disabling the A2A flow globally is too broad because docs and tests preserve conversation semantics; the safer path is a maintainer-approved contract plus a narrow requester-routing/wrapper-suppression fix.
