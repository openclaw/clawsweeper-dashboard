## Summary

Keep open. Current main and the latest release still require a nonempty `turnSourceTo` for trusted backend chat approval replay, while the webchat path can create approval records with that field absent/null, so the reported `APPROVAL_CLIENT_MISMATCH` path is source-proven and not covered by the existing Telegram/WeCom replay tests.

## What This Changes

The issue requests fixing webchat node exec approval replay so a null `turnSourceTo` does not cause `APPROVAL_CLIENT_MISMATCH` after approval.

## Best Possible Solution

Make only the `turnSourceTo` comparison optional in `canBridgeNoDeviceChatApprovalFromBackend`, then add a webchat/null-target regression that still rejects changed non-null chat targets and other binding mismatches.

## Reproduction Assessment

Yes, source-reproducible: current main stores missing webchat `turnSourceTo` as null, then the backend chat replay guard requires that field to be nonempty and returns the `APPROVAL_CLIENT_MISMATCH` path. The reporter also provided live gateway logs and a local one-line fix proof, but I did not live-run the Docker/node scenario.

## Solution Assessment

Yes, the requested direction is the narrowest maintainable fix: `turnSourceTo` is already schema-optional/null and peer fields use `matchesOptionalString`. The implementation should avoid broadening the backend bridge beyond this nullable field and should add focused regression coverage.
