## Summary

Keep open: the patch is a focused, source-aligned fix for the linked QMD memory_search stall, but it still needs after-fix real QMD behavior proof before merge and maintainers should explicitly accept the QMD stale-index tradeoff from removing the search-time forced sync fallback.

## What This Changes

The PR skips the zero-hit forced sync-and-retry path only when memory_search is backed by QMD and adds a regression test for that QMD empty-result case.

## Best Possible Solution

Land the backend-aware guard after redacted real QMD proof shows zero-hit memory_search returns promptly without qmd update, while keeping the non-QMD retry behavior intact and documenting the accepted QMD freshness tradeoff in the PR body or merge note.

## Reproduction Assessment

Yes at source level: current main and v2026.6.1 synchronously await a forced manager sync after zero memory hits, and QMD sync awaits qmd update. I did not run a live QMD timing reproduction in this read-only review.

## Solution Assessment

Yes, this is the narrowest maintainable fix shape for the reported stall: keep the existing non-QMD zero-hit retry and skip QMD's synchronous maintenance in the interactive search path. A broader async/debounced refresh design could be a follow-up if maintainers want to preserve QMD freshness repair without blocking turns.
