## Summary

Keep open: the patch is a focused and plausible fix, but this branch is conflict-marked and still has only test-level proof. Current main and the latest release still have the synchronous QMD zero-hit sync path, and the other proof-positive closing PR is also conflict-marked, so this PR is not safely obsolete or superseded yet.

## What This Changes

The PR changes memory-core so QMD zero-hit `memory_search` skips the synchronous forced sync-and-retry path while preserving the retry for other backends, with a regression test.

## Best Possible Solution

Land a clean backend-aware guard after redacted real QMD proof shows zero-hit `memory_search` returns promptly without `qmd update`, while maintainers explicitly accept or replace the QMD freshness tradeoff.

## Reproduction Assessment

Yes at source level: current main and v2026.6.6 still await forced sync after zero memory hits, and QMD sync awaits update work with a longer timeout budget. I did not run an independent live QMD timing reproduction in this read-only review.

## Solution Assessment

Yes with maintainer acceptance: the guard is the narrow owner-local fix for keeping QMD update work out of the interactive search path while preserving non-QMD retry behavior. If immediate QMD freshness is contractual, an async or debounced refresh path is safer than the current synchronous fallback.
