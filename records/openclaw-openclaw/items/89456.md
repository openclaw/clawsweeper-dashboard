## Summary

This PR is still necessary: current main only applies setup trust filtering to workspace-origin channel plugins and does not expose trusted `plugins.load.paths` channels to catalog resolution. The branch consistently extends that boundary to config/global origins with strong real package proof, but its deliberate fail-closed upgrade behavior and local code-execution policy require explicit maintainer acceptance before merge.

## What This Changes

The branch threads configured plugin load paths into channel catalog discovery, applies origin-aware fallback filtering, blocks untrusted non-bundled setup-only imports, and adds focused unit, CLI, and Docker proof coverage.

## Best Possible Solution

Adopt one documented explicit-trust rule for every non-bundled setup-only channel import, with catalog and loader coverage for trusted, untrusted, denylisted, disabled, fallback, fresh-install, and upgrade behavior.

## Reproduction Assessment

Yes. Current main omits configured load paths from trusted channel catalog discovery, and the supplied marker-based blackbox path shows a trusted configured plugin cannot reach setup while the branch can.

## Solution Assessment

Yes structurally: the patch uses shared manifest-owner policy helpers at both selection and execution layers and avoids suppressing safe same-id fallbacks. Explicit maintainer acceptance is still required for the deliberate compatibility policy.
