## Summary

Keep open. The PR now supplies real terminal proof, but it is not merge-ready because the CJK fallback still treats multi-character CJK as whole-substring LIKE terms, the empty-MATCH fallback broadens multi-token recall with OR semantics, and the overlap merge can bypass the existing keyword-preservation fallback.

## What This Changes

The PR changes memory-core keyword planning/search fallback and hybrid result merging so CJK or short-token keyword hits can contribute `textScore` when FTS MATCH returns no rows or chunk IDs differ.

## Best Possible Solution

Repair this branch or a narrow replacement with an owner-approved CJK term decomposition strategy, a merge fallback that preserves keyword hits, focused regression tests, and then land it to close the linked memory search issues.

## Solution Assessment

## Review Findings

Overall confidence: 0.87

Full review comments:

- **[P2] Split CJK fallback terms before exact LIKE matching:** `extensions/memory-core/src/memory/manager-search.ts:118`
  - body: The patch routes every CJK token to `substringTerms`, but still pushes a multi-character query such as `飞书插件配置` as one exact substring. The PR body itself says that query still gets `textScore: 0`, and the related open issue reports this can match issue-discussion files instead of the relevant partial phrases, so this should use an owner-approved split-term plan with coverage before closing the CJK bug.
  - confidence: 0.9
- **[P2] Avoid broad OR fallback for empty MATCH queries:** `extensions/memory-core/src/memory/manager-search.ts:371`
  - body: The existing exception fallback and tests require each query token to match the row, but this new empty-MATCH branch switches to `OR`, giving `textScore: 1` to rows that contain only one token from a multi-token query. That can inject unrelated memory chunks whenever MATCH returns no rows; share the existing fallback semantics or add a deliberately tested ranking strategy.
  - confidence: 0.82
- **[P2] Preserve keyword fallback after overlap merges:** `extensions/memory-core/src/memory/hybrid.ts:143-149`
  - body: When a keyword hit overlaps a vector hit, this code merges it into the vector entry and does not keep a keyword-only entry. If the combined score is still below `minScore`, `MemoryManager.search`'s relaxed fallback filters by the original keyword source/path/line key, so the merged vector-range entry can be dropped even though the keyword hit should be preserved.
  - confidence: 0.84
- **[P3] Add braces around range updates:** `extensions/memory-core/src/memory/hybrid.ts:144-145`
  - body: The new single-line range-update `if` statements are in the current diff and the live branch still has a failing lint check. Use braced blocks so the extension lint gate can pass.
  - confidence: 0.8
