## Summary

This should stay open because the report is recent, concrete, and the current code still has a split between gateway startup SecretRef resolution and pre-auth CLI/TUI credential resolution. Source inspection shows file-backed gateway auth SecretRefs can resolve locally, but the exact reported /etc/service-context case is not covered and changing fallback to the active daemon context would touch auth policy.

## What This Changes

The issue reports that user-level CLI and TUI commands fail to resolve a file-backed gateway.auth.token SecretRef stored under /etc/openclaw/secrets.json even though the gateway daemon can start with the same token.

## Best Possible Solution

Keep the fail-closed auth boundary unless maintainers approve a secure daemon-mediated credential path; otherwise improve local file-provider diagnostics and add regression coverage for the accepted /etc provider setup.

## Reproduction Assessment

Yes from source for the unresolved-error path: TUI and CLI gateway calls must resolve gateway.auth.token before connecting, and failures in file SecretRef resolution produce the reported generic message. No live /etc reproduction was established for the reporter's exact permissions and service context.

## Solution Assessment

Unclear: resolving the gateway authentication token through the already-running gateway would be circular unless a separate trusted broker/device-auth policy is designed. The safer first step is maintainer review of the auth policy, then either clearer local file-provider failure diagnostics or a deliberate secure daemon-mediated path.
