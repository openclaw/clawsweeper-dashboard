## Summary

This should stay open: the report is recent and concrete, and current main still resolves TUI and devices CLI gateway credentials locally before opening the gateway connection, so an already-running daemon cannot currently satisfy a service-context `/etc` file SecretRef for `gateway.auth.token`. The requested daemon-context behavior crosses the pre-auth gateway auth boundary, so it needs maintainer auth-policy review rather than conservative cleanup.

## What This Changes

The issue requests that TUI and CLI gateway-management commands resolve `/etc` file-backed `gateway.auth.token` SecretRefs using the active gateway/session context instead of failing in the user process.

## Best Possible Solution

Define the secure local-management auth boundary for service-owned SecretRefs; if daemon-mediated gateway-auth resolution is not approved, keep fail-closed local resolution but improve provider-level diagnostics and document `/etc` ownership/permission expectations.

## Reproduction Assessment

Yes for the source-level unresolved path: TUI and devices CLI resolve `gateway.auth.token` locally before connecting, and an unresolved file SecretRef returns the reported error style. No live `/etc` service-user reproduction was established in this read-only review.

## Solution Assessment

Unclear: using the already-running gateway to obtain the token needed to authenticate to that gateway is circular unless maintainers approve a separate trusted local credential path. A safer first step is auth-policy review, then either a deliberate daemon-mediated mechanism or clearer local file-provider diagnostics.
