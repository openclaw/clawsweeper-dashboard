## Summary

Keep open: current main still has a real pre-auth SecretRef failure path for TUI and CLI gateway calls, and the requested daemon-context fallback is an auth/security policy decision rather than a conservative cleanup close.

## What This Changes

Request that TUI and CLI gateway management commands resolve `/etc` file-provider `gateway.auth.token` SecretRefs, or use the active daemon’s authenticated context, instead of failing with unresolved SecretRef.

## Best Possible Solution

Preserve the fail-closed SecretRef and gateway-auth boundary until maintainers approve a secure local-service or daemon-mediated credential design, then add diagnostics and regression coverage for the accepted `/etc` provider setup.

## Reproduction Assessment

Partially. Source inspection gives a high-confidence path to the unresolved error before TUI/CLI connect, but I did not establish a live `/etc` service-user reproduction for the reporter’s exact permissions and provider config.

## Solution Assessment

Unclear. Letting unauthenticated CLI/TUI clients recover the gateway token through an already-running gateway is an auth-policy change; the safer path is maintainer review, then either clearer local file-provider diagnostics or an explicit local-service credential design.
