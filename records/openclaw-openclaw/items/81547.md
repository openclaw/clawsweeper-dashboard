## Summary

Keep open. Current main still resolves gateway-auth SecretRefs in the CLI/TUI process before opening the gateway connection; using the daemon’s already-active runtime context for this auth bootstrap would cross an auth/security boundary and needs maintainer design rather than cleanup closure.

## What This Changes

The issue asks CLI and TUI gateway management commands to resolve file-backed `gateway.auth.token` SecretRefs using the active daemon context instead of failing in the user shell.

## Best Possible Solution

Define a supported bootstrap contract for gateway-auth SecretRefs in separate CLI/TUI processes: either require caller-context provider resolution with clearer diagnostics, or add a secured local IPC/session mechanism that does not leak the gateway token.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection does show the likely path: TUI and devices CLI resolve `gateway.auth.token` in the caller process before connecting, so a locally unavailable file provider can fail before any gateway RPC runs.

## Solution Assessment

No, not as an automatic fix. Resolving the daemon’s active SecretRef snapshot for the credential needed to authenticate to that daemon is a circular, security-sensitive design change; maintainer input should choose the safe contract first.
