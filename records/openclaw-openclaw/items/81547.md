## Label Justifications

- `P1`: The report blocks TUI and management CLI auth for users whose gateway token is configured through a supported file-backed SecretRef setup.
- `impact:auth-provider`: The failure is in resolving `gateway.auth.token` SecretRefs for gateway authentication.
- `impact:security`: The affected value is the gateway shared secret, and the likely fix changes file-permission policy for credential reads.
- `impact:crash-loop`: The reported user-visible behavior is CLI/TUI process exit with an unresolved SecretRef error.

## Summary

This should stay open. Current main and the latest release still use local SecretRef resolution for CLI/TUI gateway auth before the client can authenticate to the daemon, and the file provider read path does not expose the narrower group-readable `/etc` permission policy proposed in the closed unmerged fix PR at https://github.com/openclaw/openclaw/pull/82649.

## What This Changes

The issue reports that CLI and TUI commands cannot resolve file-backed `gateway.auth.token` SecretRefs from `/etc/openclaw/secrets.json` even though the gateway daemon starts with the same secret.

## Best Possible Solution

Keep the fail-closed default, add a documented and tested system-level file-secret path only if maintainers approve the permission model, and cover daemon startup plus CLI/TUI management commands against the same gateway auth SecretRef setup.

## Reproduction Assessment

No live repro was run, but source inspection gives a medium-confidence reproduction path: configure `gateway.auth.token` with a file provider backed by a group-readable or root/system-owned `/etc` secret file, then run a local TUI or `openclaw devices list` path that must resolve auth locally before gateway connection.

## Solution Assessment

No, this should not be closed as solved. The safest fix needs maintainer choice around a narrow `allowReadableByOthers`-style provider policy, service/user ownership expectations, or documentation of the supported workaround before changing credential-read behavior.
