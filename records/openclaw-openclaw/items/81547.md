## Summary

Keep this open. Current main and the latest release still resolve TUI and gateway CLI auth SecretRefs in the local client process before an authenticated gateway call can be made, and the file provider still uses the default secure-file policy that rejects group/world-readable or different-uid credential files.

## What This Changes

The issue reports that CLI and TUI clients cannot resolve gateway.auth.token SecretRefs backed by a file provider under /etc/openclaw/secrets.json while the gateway daemon itself is authenticated.

## Best Possible Solution

Define an explicit gateway-auth system-secret policy, then cover both TUI connection and CLI gateway-call auth with regression tests for fresh install and upgrade behavior.

## Reproduction Assessment

Yes, source-reproducible but not live-reproduced: TUI and gateway CLI calls resolve gateway.auth.token locally, and the current file provider delegates to readSecureFile defaults that reject broader /etc-style credential permissions.

## Solution Assessment

Unclear; passing allowReadableByOthers alone is probably not the whole fix because fs-safe still rejects different POSIX owners, so maintainers need to choose the intended system-secret contract before implementation.
