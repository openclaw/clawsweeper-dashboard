## Summary

Keep open: current main still lacks lock-loss holder notification and post-lock-loss write protection, so the central bug is not already implemented. The PR is a plausible fix direction, but it is conflicting/stale against current main, lacks real behavior proof, and has merge-blocking current-main compatibility issues.

## What This Changes

The PR adds session write-lock force-release callbacks, runtime lock ownership checks, a SessionManager write guard, and tests to prevent transcript writes after session lock loss.

## Best Possible Solution

Refresh or replace the PR with a current-main implementation that notifies lock holders, checks ownership through SESSION_LOCKS, routes lock loss through abortRun and cleanup, uses the current Pi package, and includes redacted concurrent gateway proof.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection shows current main can force-release a held session lock while runEmbeddedAttempt owns the SessionManager write path, but the concurrent gateway corruption scenario was not executed.

## Solution Assessment

No, not as currently proposed. The direction is useful, but the branch must be rebased onto current SESSION_LOCKS, current Pi package naming, and abortRun/abort-settle cleanup before it is the narrowest maintainable fix.

  - body: The new guard imports `@mariozechner/pi-coding-agent`, but current main and `package.json` use `@earendil-works/pi-coding-agent`. Rebasing as written leaves the new file resolving a package OpenClaw no longer depends on, so the patch will not build.

  - confidence: 0.9
- **[P2] Route lock loss through the current abort lifecycle:** `src/agents/pi-embedded-runner/run/attempt.ts:1715-1717`
  - body: The callbacks abort `runAbortController` and the session directly. Current main derives aborted cleanup state and abort-settle behavior from `abortRun`, so bypassing that helper can leave lock-loss cleanup classified as non-aborted after a rebase.
  - confidence: 0.86
- **[P2] Handle guarded cleanup after lock loss explicitly:** `src/agents/pi-embedded-runner/run/attempt.ts:2829-2831`
  - body: The write guard remains installed through the final pending-tool-result flush. If lock loss already happened, that cleanup write can be rejected and swallowed by current cleanup, leaving the orphaned tool-result state this PR is trying to avoid; the refreshed patch needs an explicit cleanup strategy for that path.
  - confidence: 0.76
