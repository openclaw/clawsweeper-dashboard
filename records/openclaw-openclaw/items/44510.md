## Summary

Keep open. The underlying session-corruption bug is plausible and not implemented on current main, but this PR still lacks real behavior proof and needs a current-main refresh for the lock manager, abort cleanup, and Pi package changes before it can merge.

## What This Changes

The PR adds session lock force-release callbacks, lock ownership checks, a SessionManager write guard with fsync, and tests across the embedded attempt and session-lock paths.

## Best Possible Solution

Refresh or replace the PR with a current-main implementation that notifies lock holders, rejects post-loss writes using SESSION_LOCKS, routes lock loss through the current abortRun cleanup lifecycle, and includes redacted concurrent gateway proof.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection shows current main can force-release a held session lock while runEmbeddedAttempt owns the SessionManager write path, but the concurrent gateway corruption scenario was not executed.

## Solution Assessment

No, not as currently proposed. The direction is useful, but the branch must be rebased onto current SESSION_LOCKS, current Pi package naming, and abortRun/abort-settle cleanup before it is the narrowest maintainable fix.
