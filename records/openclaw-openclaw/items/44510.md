## Summary

Keep open: the PR targets a real session-lock corruption risk, but it is not mergeable against current main because it bypasses the current abort-settle lifecycle, still targets old lock/Pi APIs, and lacks after-fix real behavior proof.

## What This Changes

The PR adds session-lock force-release callbacks, a lock-held helper, a SessionManager write guard with fsync, and attempt wiring/tests to abort and reject writes after lock loss.

## Best Possible Solution

Rebase or replace this with a current-main fix that adds lock-loss notification and write guarding through `SESSION_LOCKS` and the `abortRun` cleanup lifecycle, with focused regression tests and redacted runtime proof.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection shows current main can force-release a held session lock while an embedded attempt owns the SessionManager write path, but the concurrent gateway corruption scenario was not executed.

## Solution Assessment

No, not as currently proposed. The direction is useful, but the branch must be rebased onto the current Pi package name, `SESSION_LOCKS`, and the `abortRun`/abort-settle lifecycle before it is the narrowest maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Full review comments:

- **[P2] Route lock loss through abortRun:** `src/agents/pi-embedded-runner/run/attempt.ts:1714-1716`
