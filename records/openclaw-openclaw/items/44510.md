## Summary

Keep open: the lock-loss corruption surface is still plausible on current main, but this PR is stale/conflicting, lacks real behavior proof, and has current-main correctness blockers before it can merge.

## What This Changes

The PR adds session lock force-release callbacks, a SessionManager write guard, lock ownership checks, and tests to stop transcript writes after a watchdog releases a held lock.

## Best Possible Solution

Rebase or replace the PR with a current-main implementation that notifies lock holders, rejects post-loss writes without stale imports, routes lock loss through the current abort/cleanup lifecycle, adds focused regression tests, and includes redacted real concurrent gateway proof.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection shows current main can force-release a held session lock while runEmbeddedAttempt owns the SessionManager write path, but the concurrent gateway corruption scenario was not executed.

## Solution Assessment

No, not as currently proposed. The direction is useful, but the branch must be rebased onto current SESSION_LOCKS, current Pi package naming, and abortRun/abort-settle cleanup before it is the narrowest maintainable fix.
