## Summary

Keep open: this is a plausible fix for a real session-lock write race, but this PR is not merge-ready because the teardown guard order can still break cleanup, the branch conflicts with current main’s refactored lock manager/cleanup helper, and the contributor has not provided after-fix real behavior proof.

## What This Changes

The PR adds session-lock force-release callbacks and a SessionManager write guard that aborts the active run/session and rejects session JSONL writes after lock loss.

## Best Possible Solution

Rebase the fix onto current main by adding a lock-loss notification/ownership check to the SESSION_LOCKS path, integrate the write guard with cleanupEmbeddedAttemptResources so teardown remains best-effort and always releases locks, and require redacted real gateway concurrency proof before merge.

## Reproduction Assessment

No, not at high confidence from live behavior. Source inspection shows a plausible current-main path where the watchdog force-releases a held session lock while SessionManager can still persist transcript writes, but I did not run a concurrent gateway/heartbeat reproduction and the PR body leaves that proof unchecked.

## Solution Assessment

No, not as currently submitted. The direction is maintainable, but the submitted branch keeps the guard active through teardown flushes and is stale against current main; the safer path is a rebased patch that disarms or bypasses the guard before best-effort cleanup flushes while still blocking normal writes after lock loss.
