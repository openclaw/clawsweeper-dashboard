## Summary

Keep open: the PR targets a real session-state/data-loss class, but current main has not implemented its lock-loss notification/write guard and the branch is stale against the current lock manager, Pi package, and abort lifecycle. It also still lacks after-fix real behavior proof for the concurrent gateway scenario.

## What This Changes

The PR adds a session lock force-release callback, a SessionManager write guard with fsync/lock checks, embedded-runner abort wiring, and lock/guard tests.

## Best Possible Solution

Refresh or replace the PR with a current-main implementation that notifies lock holders, checks ownership through SESSION_LOCKS, routes lock loss through abortRun and cleanupEmbeddedAttemptResources, uses the current Pi package, and includes redacted concurrent gateway proof.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection shows current main can force-release a held session lock without notifying the holder, but the concurrent gateway corruption scenario was not executed.

## Solution Assessment

No, not as currently proposed. The direction is useful, but the branch must be rebased onto current SESSION_LOCKS, package naming, abortRun, and cleanup-settle behavior before it is the narrowest maintainable fix.

- **[P2] Use the current Pi coding-agent package:** `src/agents/session-write-guard.ts:2`
  - body: The new guard imports `@mariozechner/pi-coding-agent`, but current main and `package.json` use `@earendil-works/pi-coding-agent`. Rebasing as written leaves this file resolving a package OpenClaw no longer depends on, so the patch will not build.
  - confidence: 0.92
- **[P2] Rebase lock ownership checks onto SESSION_LOCKS:** `src/agents/session-write-lock.ts:469`
  - body: `isSessionLockHeld()` checks `HELD_LOCKS`, but current main stores held locks in `SESSION_LOCKS` from `createFileLockManager`. After resolving conflicts, this either will not compile or will check the wrong ownership store, making the write guard unreliable.
  - confidence: 0.9
- **[P2] Route lock loss through the current abort lifecycle:** `src/agents/pi-embedded-runner/run/attempt.ts:1713-1715`
  - body: The callbacks abort `runAbortController` and the session directly. Current main derives aborted cleanup state and abort-settle behavior from `abortRun`, so bypassing that helper can leave lock-loss cleanup classified as non-aborted after a rebase.
  - confidence: 0.86
- **[P2] Dispose the write guard without blocking teardown:** `src/agents/pi-embedded-runner/run/attempt.ts:2829-2831`
  - body: The write guard is removed only after the final pending-tool-result flush. If lock loss already happened, that guarded flush can throw on this older cleanup path and skip later teardown, including session disposal and session lock release.
  - confidence: 0.78
