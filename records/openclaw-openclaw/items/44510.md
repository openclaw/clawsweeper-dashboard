## Summary

Keep open. The lock-loss corruption path is plausible and not implemented on current main, but this PR is stale against the current lock manager, Pi package, and abort cleanup lifecycle, and it still lacks real after-fix behavior proof.

## What This Changes

The PR adds session-lock force-release callbacks, a runtime lock-held helper, a SessionManager write guard with fsync, and tests/wiring to reject transcript writes after lock loss.

## Best Possible Solution

Add lock-loss notification and write guarding against the current SESSION_LOCKS and abortRun cleanup lifecycle, with focused regression tests and redacted real concurrent gateway proof.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection shows current main can force-release a held session lock while runEmbeddedAttempt owns the SessionManager write path, but the concurrent gateway corruption scenario was not executed.

## Solution Assessment

No, not as currently proposed. The direction is useful, but the branch must be rebased onto the current lock manager, current Pi package, and abortRun/abort-settle lifecycle before it is the narrowest maintainable fix.
