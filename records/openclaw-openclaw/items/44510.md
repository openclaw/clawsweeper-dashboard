## Summary

Keep open: current main still has a meaningful lock-loss/write-after-release gap, but this external PR is not merge-ready because it lacks real behavior proof and needs to be reconciled with the newer session-lock and abort-cleanup code on main.

## What This Changes

The PR adds session-lock loss callbacks, runtime lock ownership checks, and a SessionManager write guard to abort and reject writes after a session write lock is force-released or lost.

## Best Possible Solution

Keep the bug open and either have the contributor rebase this PR onto current session-lock/abort-settle code with runtime proof, or replace it with a narrower main-based fix that adds lock-loss notification, write guarding, and focused regression coverage without bypassing the existing abort lifecycle.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection shows current main can force-release a held session lock while `SessionManager` still owns private append/rewrite paths, but the concurrent gateway corruption scenario was not executed.

## Solution Assessment

No, not as currently proposed. The direction is useful, but the implementation needs to be rebased onto the current file-lock-manager state and routed through the current abort/settle lifecycle before it is the narrowest maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.8

Full review comments:

- **[P2] Route lock loss through the attempt abort path:** `src/agents/pi-embedded-runner/run/attempt.ts:1712-1716`
  - body: The lock-loss callbacks call `runAbortController.abort()` and `session?.abort()` directly, bypassing the existing `abortRun` path that sets `aborted`, aborts compaction, and tracks active-session abort settlement before cleanup. On current main this can make a lock-loss abort look like an ordinary prompt error and skip the settle barrier added for session-lock safety, so wire lock loss into the same abort lifecycle instead of duplicating only part of it.
  - confidence: 0.83
- **[P2] Rebase the lock-held check onto SESSION_LOCKS:** `src/agents/session-write-lock.ts:452-466`
  - body: Current main no longer has the old `HELD_LOCKS` map this helper was written for; session locks are managed through `SESSION_LOCKS = createFileLockManager(...)` and exposed as held entries. As-is, the branch's ownership check and test export do not match the current implementation, so the guard must inspect the current file-lock manager state after rebase.
  - confidence: 0.79
