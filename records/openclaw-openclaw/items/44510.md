## Summary

Keep open: current main still has the lock-loss gap this PR targets, but the branch is not mergeable as-is because it predates the current SESSION_LOCKS implementation, Pi package name, and abort-settle cleanup lifecycle, and it lacks after-fix real behavior proof.

## What This Changes

The PR adds session-lock force-release callbacks, a lock-held helper, a SessionManager write guard with fsync, and tests/wiring to reject writes after session-lock loss.

## Best Possible Solution

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection shows current main can force-release a held session lock while runEmbeddedAttempt owns the SessionManager write path, but the concurrent gateway corruption scenario was not executed.

## Solution Assessment

No, not as currently proposed. The direction is useful, but the branch must be rebased onto the current lock manager, current Pi package, and abortRun/abort-settle lifecycle before it is the narrowest maintainable fix.
