## Summary

Keep open: current main still has the unsafe lock-loss path, and this PR is useful but not mergeable as-is because its guard lifecycle can break lock-loss cleanup and needs to be rebased onto the current cleanup helper.

## What This Changes

The PR adds a session write-lock force-release callback, a lock-held probe, a SessionManager write guard with fsync, embedded-attempt lock-loss abort wiring, and unit coverage.

## Best Possible Solution

Land a rebased core fix that notifies active attempts on forced lock release, aborts the run and active session, blocks later transcript writes without lock ownership, and makes cleanup skip or clear synthetic transcript flushes after lock loss while still releasing resources.

## Reproduction Assessment

Yes, at source level. Acquire a session write lock with a tiny maxHoldMs, run the watchdog, then continue through a SessionManager persistence path after another writer can acquire the lock; current main has no holder callback or write guard on that path.

## Solution Assessment

No. The callback plus write-guard direction is reasonable, but this branch is not the safest current-main solution until it is rebased onto the cleanup helper and avoids cleanup transcript writes after lock loss.
