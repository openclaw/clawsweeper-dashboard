## Summary

Keep open: this is a plausible session-state/data-loss fix, but the branch is not merge-ready because it targets pre-refactor lock/package/lifecycle APIs and still lacks required after-fix real behavior proof.

## What This Changes

The PR adds lock-loss callbacks, runtime lock ownership checks, and a SessionManager write guard intended to stop transcript writes after a session write lock is force-released.

## Best Possible Solution

Refresh or replace the PR with a current-main implementation that notifies lock holders, checks ownership through SESSION_LOCKS, routes lock loss through abortRun/cleanup, uses the current Pi package, and includes redacted concurrent gateway proof.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection shows current main can force-release a held session lock while runEmbeddedAttempt owns the SessionManager write path, but the concurrent gateway corruption scenario was not executed.

## Solution Assessment

No, not as currently proposed. The direction is useful, but the branch must be rebased onto current SESSION_LOCKS, package naming, and abortRun/abort-settle cleanup before it is the narrowest maintainable fix.

## Review Findings

Overall confidence: 0.88

Full review comments:

- **[P2] Use the current Pi coding-agent package:** `src/agents/session-write-guard.ts:2`
  - body: The new guard imports `@mariozechner/pi-coding-agent`, but current main and `package.json` use `@earendil-works/pi-coding-agent`. Rebasing as written leaves this file resolving a package OpenClaw no longer depends on, so the patch will not build.
  - confidence: 0.92
- **[P2] Rebase lock ownership checks onto SESSION_LOCKS:** `src/agents/session-write-lock.ts:469`
  - body: `isSessionLockHeld()` checks `HELD_LOCKS`, but current main stores held locks in `SESSION_LOCKS` from `createFileLockManager`. After resolving the conflict, this code either will not compile or will check the wrong ownership store, making the write guard unreliable.
  - confidence: 0.9
- **[P2] Route lock loss through the current abort lifecycle:** `src/agents/pi-embedded-runner/run/attempt.ts:1715-1717`
  - body: The callbacks abort `runAbortController` and the session directly. Current main derives aborted cleanup state and abort-settle behavior from `abortRun`, so bypassing that helper can leave lock-loss cleanup classified as non-aborted after a rebase.
  - confidence: 0.86
- **[P2] Handle guarded cleanup after lock loss explicitly:** `src/agents/pi-embedded-runner/run/attempt.ts:2829-2831`
  - body: The write guard is removed only after the final pending-tool-result flush. If lock loss already happened, that cleanup write can be rejected and swallowed by current cleanup, leaving the orphaned tool-result state this PR is trying to avoid.
  - confidence: 0.76
