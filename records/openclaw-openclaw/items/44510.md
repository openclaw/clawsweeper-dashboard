## Summary

Keep open. The lock-loss protection is still a plausible bug fix, but this branch is not mergeable against current main: it needs real behavior proof and a rebase onto the current Pi package, file-lock manager, and abort-settle cleanup lifecycle.

## What This Changes

The PR adds session-write-lock force-release callbacks, a lock-held helper, a SessionManager write guard, and attempt wiring/tests to abort and reject session writes after lock loss.

## Best Possible Solution

Rebase or replace with a main-based fix that adds lock-loss notification and write guarding through the current file-lock manager and abort lifecycle, with focused regression tests and redacted runtime proof.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection shows current main can force-release a held session lock while an embedded attempt owns `SessionManager` write paths, but the concurrent gateway corruption scenario was not executed.

## Solution Assessment

No, not as currently proposed. The direction is useful, but the branch must be rebased onto the current Pi package name, `SESSION_LOCKS`, and the `abortRun`/abort-settle cleanup lifecycle before it is the narrowest maintainable fix.
