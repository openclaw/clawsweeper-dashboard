## Summary

Keep this PR open, but it is not ready to merge: the lock-loss direction is plausible, the earlier abort-controller review comments were addressed, and current main still lacks equivalent write-after-lock-loss protection, but the branch needs real runtime proof, a rebase onto the current lock/cleanup code, and a cleanup-order fix.

## What This Changes

The PR adds session lock force-release callbacks, lock-ownership checks, and a guarded SessionManager write path intended to stop writes after a session lock is lost.

## Best Possible Solution

Port the lock-loss guard onto current main's `SESSION_LOCKS` and `cleanupEmbeddedAttemptResources` flow: notify/abort on force release, reject post-loss writes, preserve best-effort cleanup/release, add focused regression coverage, and attach redacted real gateway proof.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection shows current main can force-release a held session lock without notifying the writer, and upstream SessionManager writes directly to JSONL, but the gateway/concurrent-write scenario was not run.

## Solution Assessment

No. The general approach is reasonable, but this branch should be rebased onto the current lock manager and cleanup helper, and the write guard must not cause cleanup to fail after lock loss.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.78

Full review comments:

- **[P2] Dispose the write guard before cleanup flushes:** `src/agents/pi-embedded-runner/run/attempt.ts:2829`
  - body: The guard stays installed while `flushPendingToolResultsAfterIdle` runs above this line. In the lock-loss path added by this PR, that flush can try to persist synthetic tool results after `isSessionLockHeld()` is already false, so the guard throws before cleanup reaches `session.dispose()`, `releaseWsSession()`, or the lock release. Move guard disposal before the best-effort flush, or wire it into the current cleanup helper so cleanup still releases resources after lock loss.
