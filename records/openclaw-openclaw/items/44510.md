## Summary

Keep this PR open: current main still releases over-held session locks without notifying the active attempt, so the underlying corruption-prevention problem is real and not implemented on main. The proposed direction is useful, and the later PR commits address the earlier orphaned AbortController and duplicate callback comments, but the patch should not merge as-is because its write guard remains active during final pending-tool-result flush after lock loss and the branch needs to be rebased onto the current cleanup helper shape.

## What This Changes

The PR adds a session-lock force-release callback, a process-local lock-held probe, a SessionManager write guard that rejects writes after lock loss and fsyncs writes, embedded-attempt abort wiring, and lock/write-guard tests.

## Best Possible Solution

Land a rebased core fix that notifies the active embedded attempt when its session lock is force-released, aborts both the run controller and active session, prevents further transcript writes without a valid lock, and makes teardown best-effort without writing synthetic cleanup entries after lock loss. Keep the backup-related comments as useful recovery context, but treat them as complementary rather than a substitute for core session write safety.

## Reproduction Assessment

Yes. A focused reproduction is to acquire a session write lock with a tiny `maxHoldMs`, run the watchdog check, and then attempt SessionManager persistence after another writer can acquire the same session file; current main has no callback or write guard on that path. The PR adds unit coverage for the callback/probe/guard pieces, but still needs an integration-style cleanup-path proof for the concurrent gateway scenario.

## Solution Assessment

No. The callback plus write-guard direction is a reasonable owner-boundary fix, but the submitted patch leaves the guard active during final pending-tool-result flush after lock loss and does not match current main's cleanup helper. A safer version should explicitly track lock loss and skip/clear cleanup writes that would occur without the lock, then dispose resources and release handles deterministically.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.78

Full review comments:

- **[P2] Avoid flushing pending results through a lost lock:** `src/agents/pi-embedded-runner/run/attempt.ts:2829`
  - body: The write guard is removed only after `flushPendingToolResultsAfterIdle`, but lock loss is exactly when `isSessionLockHeld()` is false. If cleanup synthesizes pending tool results on that path, it will call the guarded SessionManager write path and throw before the old inline cleanup reaches session disposal, WebSocket release, and lock release. Track lock loss and skip/clear that flush, or make the cleanup path dispose resources without attempting transcript writes after the lock is gone.
  - confidence: 0.82
