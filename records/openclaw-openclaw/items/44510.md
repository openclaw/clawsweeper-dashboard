## Summary

Keep this PR open: current main still lacks holder notification or a write guard after watchdog force-release, so the underlying data-corruption bug is not implemented elsewhere. The PR is useful but not merge-ready because its cleanup path can still try to flush transcript writes after lock loss, and the branch is stale against the current cleanup helper.

## What This Changes

The PR adds a session-lock force-release callback, a process-local lock-held probe, a SessionManager write guard that rejects writes after lock loss and fsyncs writes, embedded-attempt abort wiring, and lock/write-guard tests.

## Best Possible Solution

Land a rebased core fix that notifies the active embedded attempt when its session lock is force-released, aborts the run controller and active session, prevents further transcript writes without a valid lock, and makes teardown skip or clear synthetic transcript flushes after lock loss while still disposing sessions and releasing resources deterministically. The keepmyclaw backup comments are useful recovery context, but they are complementary rather than a substitute for core session write safety.

## Reproduction Assessment

Yes. A focused source-level reproduction is to acquire a session write lock with a tiny `maxHoldMs`, run the watchdog, then continue through a SessionManager persistence path after another writer can acquire the lock; current main has no holder callback or write guard on that path.

## Solution Assessment

No. The callback plus write-guard direction is reasonable, but the submitted patch is not the safest current-main solution until it is rebased onto the cleanup helper and explicitly avoids cleanup transcript writes after lock loss.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.78

Full review comments:

- **[P2] Skip cleanup transcript writes after lock loss:** `src/agents/pi-embedded-runner/run/attempt.ts:2826-2828`
  - body: The guard stays installed while `flushPendingToolResultsAfterIdle` runs, but lock loss is exactly when `isSessionLockHeld()` is false. If cleanup synthesizes pending tool results on that path, the guarded write throws before the branch's inline teardown reaches session disposal, WebSocket release, and lock release. Track lock loss and skip or clear this flush, or move the guard lifecycle into the current cleanup helper so resource teardown remains best-effort.
