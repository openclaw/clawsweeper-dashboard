## Summary

Keep open: the session lock-loss corruption path is still plausible on current main, but this PR is not merge-ready because it is conflicting, lacks real behavior proof, and still has lock-loss cleanup correctness issues.

## What This Changes

The PR adds watchdog lock-loss callbacks plus a SessionManager write guard to reject session JSONL writes after the session write lock is force-released.

## Best Possible Solution

Rework the fix against current main by adding lock-loss notification to the current fs-safe-backed session lock path, routing lock loss through the existing attempt abort and cleanup state, avoiding any post-lock-loss transcript writes, and proving it with a real concurrent gateway run.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection shows current main can force-release a held lock without notifying the holder, and pi-coding-agent persists messages through appendFileSync, so the race is plausible from code but not runtime-proven here.

## Solution Assessment

No. The direction is right, but this branch is stale against current main and its lock-loss abort/cleanup path still bypasses the attempt's aborted state and can flush guarded writes during teardown.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Route lock loss through the attempt abort state:** `src/agents/pi-embedded-runner/run/attempt.ts:1715`
  - body: Calling `runAbortController.abort()` directly bypasses the `abortRun` path that sets `aborted` and drives abort-aware cleanup. A lock-loss abort can therefore be reported and cleaned up like a normal error, which leaves teardown free to wait and flush pending tool results instead of taking the no-write aborted cleanup path.
  - confidence: 0.82
- **[P2] Avoid flushing guarded writes after lock loss:** `src/agents/pi-embedded-runner/run/attempt.ts:2831`
  - body: The write guard is removed only after the final pending-tool-result flush. If the watchdog already released the lock, that flush can call into the guarded SessionManager write path and throw before `session.dispose()`, `releaseWsSession()`, and `sessionLock.release()` run, so the lock-loss path can still leak cleanup.
  - confidence: 0.84
