## Summary

Keep open. Current main and the latest release still expose independent `sessions_send` selectors and still reject mixed `sessionKey` plus `label` payloads before resolution, while open implementation PRs remain unmerged.

## What This Changes

The issue asks OpenClaw to prevent `sessions_send` from encouraging or hard-failing redundant `sessionKey` plus `label` calls through schema guidance and/or executor normalization.

## Best Possible Solution

Land one canonical fix that treats explicit `sessionKey` or `sessionId` input as authoritative for redundant label-style fields, preserves A2A visibility and thread safeguards, updates tool/schema/docs guidance, and adds focused regression tests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: call `createSessionsSendTool().execute` with non-empty `sessionKey` and `label`, and current main returns the reported mutual-exclusion error before resolution or delivery.

## Solution Assessment

Yes. Executor-side normalization plus schema/tool/docs guidance is the narrow maintainable fix; guidance-only PRs reduce confusion but cannot protect callers that already submit redundant selectors.
