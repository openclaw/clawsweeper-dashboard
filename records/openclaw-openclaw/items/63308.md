## Summary

Keep open. Current main still exposes `sessionKey`, `label`, and `agentId` as independent optional `sessions_send` parameters and still rejects a mixed `sessionKey` plus `label` call before send resolution; overlapping open PRs cover the likely fix path but have not landed.

## What This Changes

The issue asks for `sessions_send` to guide or normalize target selection so models do not submit both `sessionKey` and `label` and trigger the mutual-exclusion error.

## Best Possible Solution

Land one canonical fix that makes `sessionKey` authoritative when present, ignores redundant `label` and `agentId` on that path, documents both valid addressing modes, and adds focused regression coverage.

## Reproduction Assessment

Yes, from source inspection. A `createSessionsSendTool().execute` call with both `sessionKey` and `label` reaches the guard in `src/agents/tools/sessions-send-tool.ts` and returns the reported error before gateway resolution.

## Solution Assessment

Yes. Runtime `sessionKey` precedence plus schema/tool/docs guidance is the narrow maintainable fix; schema text alone is weaker because real reports describe redundant labels appearing in the submitted payload.
