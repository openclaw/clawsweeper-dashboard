## Summary

Keep open. Current main still exposes `sessionKey`, `label`, and `agentId` as independent optional `sessions_send` fields and still returns the reported mixed-selector error before target resolution; the latest release tag shows the same behavior. Multiple open PRs already cover the runtime and schema/docs paths, so this should stay open for maintainer consolidation rather than being closed or queued again.

## What This Changes

Improve `sessions_send` so model-facing schema/docs discourage invalid `sessionKey` plus `label` calls and the runtime preferably treats `sessionKey` as authoritative when redundant `label` or `agentId` values are present.

## Best Possible Solution

Land one canonical fix that makes `sessionKey` authoritative when present, ignores redundant `label` and `agentId` on that path, documents the two valid addressing modes in schema/tool/docs text, and adds focused regression coverage for `sessionKey`-only, `label`-only, and mixed-selector calls.

## Reproduction Assessment

Yes. A focused source-level check can call `createSessionsSendTool().execute` with both `sessionKey` and `label`; current main reaches the exact reported error before any gateway send resolution.

## Solution Assessment

Yes. The narrow maintainable direction is runtime `sessionKey` precedence plus schema/docs guidance; the open schema-only path is helpful but incomplete unless the runtime also tolerates redundant selectors from real caller paths.
