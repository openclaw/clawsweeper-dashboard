## Summary

Keep open. Current main and the latest release still expose `sessionKey`, `label`, and `agentId` as independent optional `sessions_send` fields and still reject mixed `sessionKey` plus `label` inputs with the reported error, while open PRs already cover the runtime and schema/docs directions.

## What This Changes

The issue requests clearer `sessions_send` tool guidance and/or executor normalization so mixed `sessionKey` plus `label` calls do not fail when `sessionKey` is the authoritative target.

## Best Possible Solution

Land one canonical fix that makes `sessionKey` authoritative when present, ignores redundant `label` and `agentId` on that path, documents both valid addressing modes in the tool metadata/docs, and adds focused regression coverage for `sessionKey`-only, `label`-only, and mixed-selector calls.

## Reproduction Assessment

Yes. A focused check can call `createSessionsSendTool().execute` with both `sessionKey` and `label`; current main returns the exact reported mixed-selector error before resolving or sending to the target session.

## Solution Assessment

Yes. The narrow maintainable fix is runtime `sessionKey` precedence plus schema/docs guidance; schema text alone is helpful but incomplete for the real-world injected-label path.
