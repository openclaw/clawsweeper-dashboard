## Summary

Keep open. Current main and the latest release still expose `sessions_send` selectors as independent optionals and still reject mixed `sessionKey` plus `label` calls before resolution; related PRs are open but unmerged.

## What This Changes

The issue requests clearer `sessions_send` schema/tool guidance and/or executor normalization so redundant `sessionKey` plus `label` calls do not fail unnecessarily.

## Best Possible Solution

Land one canonical fix that treats explicit `sessionKey` or session-id input as authoritative, preserves visibility/A2A/thread safeguards, updates tool/schema/docs guidance, and adds focused regression tests for valid and mixed selector calls.

Yes. Source inspection gives a high-confidence path: call `createSessionsSendTool().execute` with non-empty `sessionKey` and `label`, and current main returns the reported mutual-exclusion error before gateway resolution.

## Solution Assessment

Yes. Executor-side normalization plus schema/tool/docs guidance is the narrow maintainable fix; docs-only guidance would not protect callers that already submit redundant selectors.
