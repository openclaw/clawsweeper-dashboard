## Summary

Keep open. Current main and the latest release still expose the selectors as independent optionals and still reject mixed `sessionKey` plus `label` before send resolution; active PRs cover the likely fix but have not landed.

## What This Changes

The issue requests clearer `sessions_send` selector guidance and/or executor normalization so mixed `sessionKey` plus `label` calls do not fail unnecessarily.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/74009 or an equivalent current-main repair that makes `sessionKey` or session-id input authoritative, preserves visibility/A2A confidentiality and thread guards, updates schema/tool/docs guidance, and adds focused regression tests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: call `createSessionsSendTool().execute` with both non-empty `sessionKey` and `label`, and current main returns the reported error before gateway resolution.

## Solution Assessment

Yes. Executor-side normalization plus schema/tool/docs guidance is the narrow maintainable fix; docs-only guidance is insufficient because real reports show redundant labels already reaching the executor.
