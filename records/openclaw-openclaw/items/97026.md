## Summary

Keep open: current main and v2026.6.10 still expose GitHub workflow auth through `GH_TOKEN`, one skill `apiKey`, plain `gh` auth, and a `GH_CONFIG_DIR` diagnostic, but not named purpose-scoped profiles, redacted purpose audit metadata, or GitHub token capability failures. The least-privilege goal fits OpenClaw security priorities, but the credential ownership and scope-check contract needs maintainer product/security direction before implementation.

## What This Changes

Request purpose-scoped GitHub auth profiles so private tracking and public contribution workflows can use separate least-privilege tokens with audit metadata and clear missing-scope failures.

## Best Possible Solution

Design a maintainer-approved GitHub credential-purpose contract that keeps defaults unchanged, uses SecretRef-compatible least-privilege token purposes, records only redacted purpose metadata, and defines authoritative scope checks before wiring skills or plugins to it.

## Reproduction Assessment

Not applicable: this is a feature request rather than a broken existing contract. Source inspection gives high confidence that current main lacks a first-class GitHub purpose selector.

## Solution Assessment

Unclear until maintainers choose the ownership boundary. The safest direction is a narrow GitHub skill/plugin credential-purpose contract with explicit security semantics, not a broad generic auth expansion by default.
