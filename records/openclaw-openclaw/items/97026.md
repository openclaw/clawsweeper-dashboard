## Label Justifications

- `P2`: This is a bounded least-privilege GitHub auth improvement with real user value but no confirmed active outage or emergency.
- `impact:security`: The request is about reducing token over-privilege and exposing only redacted token-purpose metadata in audit output.
- `impact:auth-provider`: The issue concerns auth profile selection, credential scope, SecretRef-compatible token handling, and missing-scope failures for GitHub workflows.

## Summary

Keep open: current main and v2026.6.10 still expose GitHub skill auth as a single `GH_TOKEN`/`skills.entries.<skill>.apiKey` path or plain `gh auth`, while the requested purpose-selected token profiles, redacted audit purpose, and missing-scope diagnostics are not implemented.

## What This Changes

Request purpose-scoped GitHub auth profiles so private tracking and public contribution workflows can use separate least-privilege tokens.

## Best Possible Solution

Design a narrow GitHub skill/plugin-owned credential-purpose contract with SecretRef-backed token purposes, explicit capability checks, redacted profile-purpose audit metadata, unchanged default behavior, focused tests, and docs that recommend least-privilege separation.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high confidence that current behavior has only one `GH_TOKEN`/single skill `apiKey` path for the bundled GitHub issue workflow, not purpose-scoped profiles.

## Solution Assessment

Unclear until maintainers choose the ownership boundary. The safest direction is a narrow GitHub skill/plugin purpose selector with explicit security semantics, rather than a broad core auth-profile expansion by default.
