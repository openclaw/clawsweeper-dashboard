## Summary

Keep open: current main and v2026.6.10 still expose GitHub workflow auth through `GH_TOKEN`, plain `gh auth`, or one per-skill `apiKey`, so the requested named purpose profiles, redacted purpose audit metadata, and missing-scope diagnostics remain unimplemented.

## What This Changes

Request purpose-scoped GitHub auth profiles so private tracking and public contribution workflows can use separate least-privilege tokens.

## Best Possible Solution

Design a maintainer-approved GitHub credential-purpose contract with SecretRef-backed token purposes, explicit capability checks, redacted purpose audit metadata, unchanged defaults, focused upgrade tests, and least-privilege docs.

## Reproduction Assessment

Not applicable: this is a feature request rather than a broken existing contract. Source and release inspection give high confidence that current OpenClaw lacks a first-class GitHub purpose selector.

## Solution Assessment

Unclear until maintainers choose the ownership boundary. The safest direction is a narrow credential-purpose contract for GitHub workflows, not a broad ad hoc expansion of generic auth profiles.
