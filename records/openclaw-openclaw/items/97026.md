## Summary

Keep open: current main still exposes GitHub workflow auth through plain `gh auth`, `GH_TOKEN`, or one skill `apiKey`/env path, while the requested purpose-scoped private-tracker versus public-contribution profiles, redacted purpose audit metadata, and missing-scope diagnostics are not implemented.

## What This Changes

The issue requests purpose-scoped GitHub authentication profiles so private tracking and public contribution workflows can use different least-privilege tokens.

## Best Possible Solution

Design a maintainer-approved GitHub credential-purpose contract with SecretRef-backed token purposes, redacted audit metadata, explicit capability checks, unchanged defaults, focused tests, and least-privilege docs.

## Reproduction Assessment

Not applicable; this is a feature and security-design request, not a broken existing contract. Source inspection gives high confidence that current main lacks a first-class GitHub purpose selector.

## Solution Assessment

Unclear until maintainers choose the credential ownership boundary. The narrowest maintainable path appears to be a GitHub skill/plugin purpose contract with explicit security semantics, not a broad generic model auth-profile expansion by default.
