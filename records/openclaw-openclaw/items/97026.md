## Summary

Keep open: current main and v2026.6.10 still expose GitHub workflow auth through a single `GH_TOKEN` / single skill `apiKey` path or plain `gh auth`, so the requested purpose-scoped profiles, redacted purpose audit metadata, and missing-scope diagnostics are not implemented. The request is aligned with OpenClaw's security goals, but it needs maintainer security/product direction before code should be queued.

## What This Changes

The issue requests named GitHub auth purposes so private tracking and public contribution workflows can use different least-privilege tokens with redacted audit metadata and actionable missing-scope errors.

## Best Possible Solution

Design a maintainer-approved GitHub skill/plugin credential-purpose contract with SecretRef-backed token purposes, explicit capability checks, redacted profile-purpose audit metadata, unchanged default behavior, focused tests, and least-privilege docs.

## Reproduction Assessment

Not applicable: this is a feature request rather than a broken existing contract. Source inspection gives high confidence that current main lacks the requested first-class GitHub purpose selector.

## Solution Assessment

Unclear until maintainers choose the ownership boundary. The safest direction is a narrow GitHub skill/plugin credential-purpose contract with explicit security semantics, not a broad generic auth-profile expansion by default.
