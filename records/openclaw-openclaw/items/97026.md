## Summary

Keep open: current main and the latest release still route GitHub workflow auth through `GH_TOKEN`, plain `gh auth`, or one skill `apiKey`, so the requested purpose-scoped token profiles, redacted purpose audit metadata, and capability-specific failures remain unimplemented. The request has real least-privilege value, but it needs maintainer security/product direction before implementation.

## What This Changes

Request purpose-scoped GitHub auth profiles so private tracking and public contribution workflows can use different least-privilege tokens with auditable purpose metadata.

## Best Possible Solution

Design a maintainer-approved GitHub credential-purpose contract with SecretRef-backed tokens, explicit capability checks, redacted purpose audit metadata, unchanged default behavior, focused tests, and least-privilege docs.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable: this is a feature request rather than a broken existing contract, and source inspection gives high confidence that current main lacks a first-class GitHub purpose selector.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers choose the ownership boundary; the safest direction is a narrow GitHub skill/plugin credential-purpose contract with explicit security semantics rather than a broad generic auth-profile expansion by default.
