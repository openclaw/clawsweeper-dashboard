## Summary

Keep open: current main and v2026.6.10 still route GitHub workflow auth through `GH_TOKEN`, plain `gh auth`, or one skill `apiKey`; provider auth profiles do not provide purpose-scoped GitHub workflow tokens, scope checks, or redacted purpose audit metadata. This is a valid least-privilege security request, but it needs maintainer product/security direction before implementation.

## What This Changes

The issue requests purpose-scoped GitHub auth profiles so private tracking and public contribution workflows can use separate least-privilege tokens.

## Best Possible Solution

Design a maintainer-approved GitHub workflow credential-purpose contract with SecretRef-compatible token selection, explicit capability checks, redacted purpose audit metadata, unchanged defaults, focused tests, and least-privilege docs.

## Reproduction Assessment

Not applicable: this is a feature request, not a broken existing behavior report. Source and release inspection give high confidence that current OpenClaw lacks a first-class GitHub purpose selector.

## Solution Assessment

Unclear until maintainers choose the ownership boundary. The safest direction is a narrow GitHub skill/plugin credential-purpose contract with explicit security semantics, not a broad generic auth-profile expansion by default.
