## Summary

Keep open: current main and the latest release still expose GitHub workflow auth through a single `GH_TOKEN`/`gh auth` or single skill `apiKey` path, while this issue asks for named purpose-scoped profiles, audit-safe purpose metadata, and capability-specific scope failures. The direction fits OpenClaw's least-privilege goals, but it is a security- and auth-provider-sensitive feature that needs maintainer product/security design before implementation.

## What This Changes

The issue requests named GitHub auth purposes/profiles so private tracking and public contribution workflows can use separate least-privilege tokens with redacted audit metadata and clear missing-scope failures.

## Best Possible Solution

Design a maintainer-approved GitHub credential-purpose contract that preserves current defaults, supports SecretRef-backed narrow tokens, emits redacted purpose metadata, and adds focused docs/tests for scope failures before any code is queued.

## Reproduction Assessment

Not applicable: this is a feature request rather than a broken existing contract. Source inspection gives high confidence that current main lacks the requested first-class GitHub purpose selector.

## Solution Assessment

Unclear until maintainers choose the ownership boundary. The safest direction is a narrow GitHub skill/plugin credential-purpose contract with explicit security semantics, not a broad generic auth-profile expansion by default.
