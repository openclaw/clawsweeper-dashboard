## Summary

Keep open: current main and the latest release still expose GitHub workflow auth through a single `GH_TOKEN` / single skill `apiKey` path or ambient `gh auth`, while the requested named purpose profiles, redacted purpose audit metadata, and missing-scope diagnostics remain unimplemented. The request fits OpenClaw's security and safe-default goals, but it needs maintainer security/product direction before implementation.

## What This Changes

The issue requests named, purpose-scoped GitHub auth profiles so private tracking and public contribution workflows can use separate least-privilege tokens with redacted audit metadata and clear missing-scope failures.

## Best Possible Solution

Define a maintainer-approved GitHub skill/plugin credential-purpose contract with SecretRef-backed token purposes, explicit capability checks, redacted profile-purpose audit metadata, unchanged defaults, focused tests, and least-privilege docs.

## Reproduction Assessment

Not applicable: this is a feature request rather than a broken existing contract. Source and release inspection give high confidence that current main lacks the requested first-class GitHub purpose selector.

## Solution Assessment

Unclear until maintainers choose the ownership boundary. The safest direction is a narrow GitHub skill/plugin credential-purpose contract with explicit security semantics, not a broad generic auth-profile expansion by default.
