## Summary

Keep open: current main and v2026.6.10 still expose GitHub workflow auth through one `GH_TOKEN` or one skill `apiKey` path, plus plain GitHub CLI auth/`GH_CONFIG_DIR` diagnostics. The requested named purpose profiles, least-privilege capability checks, redacted purpose audit metadata, and missing-scope workflow errors remain product/security design work.

## What This Changes

Purpose-scoped GitHub auth profiles are requested so private tracking and public contribution workflows can use separate least-privilege tokens.

## Best Possible Solution

Design a maintainer-approved GitHub-owned credential-purpose contract with SecretRef-backed token purposes, explicit capability checks, redacted purpose audit metadata, unchanged defaults, focused tests, and least-privilege docs.

## Reproduction Assessment

Not applicable: this is a feature request rather than a broken existing contract. Source and docs inspection give high confidence that current main lacks a first-class GitHub purpose selector.

## Solution Assessment

Unclear until maintainers choose the ownership boundary. The safest direction is a narrow GitHub skill/plugin or extension credential-purpose contract with explicit security semantics, not a broad generic auth-profile expansion by default.
