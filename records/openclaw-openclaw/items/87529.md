## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Do not persist webhook auth headers in completion state:** `extensions/webhooks/src/completion.ts:117`
  - body: `storeCompletionDispatch` copies the full inbound header map into the completion context that is registered in plugin state. Those headers include `Authorization` or route-specific secret headers, so completion delivery can persist webhook credentials until TTL; store a redacted allowlist instead.
  - confidence: 0.94
- **[P1] Key completion delivery to the scheduled run:** `extensions/webhooks/src/completion.ts:119-120`
  - body: Pending completion context is stored under `sessionKey` before the scheduler returns a job handle. Two webhooks targeting the same session can overwrite each other before lifecycle start, causing the wrong completion text to be delivered to an external target; key this to the accepted scheduled run or another unique run token.
  - confidence: 0.92
- **[P1] Gate exec delivery before exposing it to webhooks:** `extensions/webhooks/src/delivery.ts:58-61`
  - body: A route with `deliver.mode: "exec"` can spawn a local process in the Gateway with webhook-rendered args/env/stdin. That is a new remote-triggered command execution path and needs explicit maintainer-approved gating or removal before generic webhooks ship.
  - confidence: 0.9
- **[P1] Keep scheduled turns on the documented authority boundary:** `src/plugins/host-hook-scheduled-turns.ts:209-210`
  - body: This changes scheduled turns from bundled-only to bundled-or-trusted-official installs, but the public SDK docs still describe the surface as bundled-only. Either keep the old boundary or update the SDK docs/contracts with an explicit approval trail for installed official plugins scheduling session turns.
  - confidence: 0.86
- **[P2] Keep subscription state out of the request path:** `extensions/webhooks/index.ts:63-65`
  - body: Every webhook request calls `loadTargets()`, which stats and may reread the subscription file even for static routes. Gateway plugin metadata is process-stable in this repo, so dynamic subscription updates should update a lifecycle-owned in-memory snapshot instead of adding filesystem freshness checks to the hot path.
  - confidence: 0.86
- **[P2] Do not publish unverified live E2E claims:** `docs/plugins/webhooks-enterprise-validation.md:36-39`
  - body: This new docs page says a live GitHub end-to-end run verified Gateway-managed webhook scheduling, while the PR body says fresh real E2E reports for the current Gateway-managed provider flows are still not completed. Attach the proof and align the PR body, or downgrade this section to the tests that actually ran.
  - confidence: 0.82
