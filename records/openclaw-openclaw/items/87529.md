## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Store completion context only after scheduling succeeds:** `extensions/webhooks/src/dispatch.ts:55-60`
  - body: `onAgentCompletionDispatch` persists the webhook completion context before the scheduler returns a handle. If scheduling throws or is rejected, the response can be a failure while stale completion state remains keyed to the session and can attach to a later run; move this after a successful schedule or roll it back on failure.
  - confidence: 0.9
- **[P1] Do not persist webhook auth headers in completion state:** `extensions/webhooks/src/completion.ts:117`
  - body: The completion context stores the full request headers and later exposes them to delivery templating. For bearer/header/HMAC routes this can persist Authorization or signature headers in the keyed store and make them available to channel or exec delivery; strip sensitive headers before storing completion context.
  - confidence: 0.92
- **[P1] Correlate completion delivery to the scheduled run:** `extensions/webhooks/src/completion.ts:119-120`
  - body: Pending completion context is registered under `sessionKey`, so two webhook-triggered runs for the same session can overwrite each other before lifecycle start. The first run can then consume the second delivery context; key this by a scheduler/run correlation or queue entries in a way that cannot mis-associate completions.
  - confidence: 0.88
- **[P1] Gate exec delivery before exposing it to webhooks:** `extensions/webhooks/src/delivery.ts:58-60`
  - body: `mode: "exec"` runs a configured local command with templated args/env from the webhook context. Even with route auth, this is a new remote-to-local process execution surface and should be removed, hidden behind an explicit maintainer-owned opt-in policy, or otherwise gated before the generic webhook feature ships.
  - confidence: 0.9
- **[P1] Keep scheduled turns on the documented authority boundary:** `src/plugins/host-hook-scheduled-turns.ts:209`
  - body: This changes scheduled-turn authorization from bundled-only to also allow trusted official installed plugins, while the public SDK docs still describe `scheduleSessionTurn` as bundled-only. Either keep the existing boundary or update the contract/docs/tests with explicit maintainer approval for the new plugin authority.
  - confidence: 0.86
- **[P2] Keep subscription reloads out of the request path:** `extensions/webhooks/index.ts:63-65`
  - body: Every webhook request can call `subscriptionStore.loadTargets()`, which stats/reads/parses the subscription file and rebuilds targets. Gateway plugin metadata is process-stable in this repo, so dynamic subscription changes need a lifecycle-owned snapshot/update path rather than filesystem freshness checks on inbound delivery.
  - confidence: 0.84
