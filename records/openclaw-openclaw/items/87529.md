## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Correlate completion delivery to the accepted run:** `extensions/webhooks/src/dispatch.ts:54-60`
  - body: `onAgentCompletionDispatch` runs before the scheduler accepts the turn, while completion lookup later matches only by `sessionKey`. If scheduling throws/returns undefined, or two webhooks schedule the same session before the first run ends, a later lifecycle event can pick up the wrong pending context and deliver another run's output to this route's completion target. Register the context after acceptance and match it to the accepted job/run instead of only the session.
  - confidence: 0.9
- **[P2] Keep the shipped Gmail webhook commands documented:** `docs/cli/webhooks.md:24-38`
  - body: The PR still registers the built-in `openclaw webhooks gmail setup|run` commands before loading plugin CLI commands, and other docs still point to that Gmail flow, but this page now lists only the new dynamic subscription commands. Please keep the Gmail setup/run reference alongside the new Webhooks plugin commands so a shipped CLI surface does not disappear from docs.
  - confidence: 0.87
- **[P2] Avoid polling subscription files from webhook delivery:** `extensions/webhooks/src/subscriptions.ts:239-254`
  - body: `loadTargets()` stats and may reread the subscription JSON file, and `index.ts` calls it through `resolveTargetsByPath` for each webhook delivery. Gateway/plugin runtime hot paths should reuse lifecycle-owned snapshots; subscribe/remove can update the in-memory targets and startup can load persisted subscriptions without a filesystem freshness check on every inbound webhook.
  - confidence: 0.82
