## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Store completion context only after scheduling succeeds:** `extensions/webhooks/src/dispatch.ts:54-60`
  - body: `onAgentCompletionDispatch` runs before `scheduler(scheduleRequest)`, so a thrown or rejected schedule leaves pending completion context for a run that never exists; a later run with the same session key can consume that stale context and deliver the wrong completion.
  - confidence: 0.94
- **[P1] Do not persist webhook auth headers in completion state:** `extensions/webhooks/src/completion.ts:117`
  - body: The stored completion context includes the full inbound header map, which can contain `Authorization` or webhook signature headers. Persist only the sanitized fields needed for templating or completion delivery.
  - confidence: 0.93
- **[P1] Correlate completion delivery to the scheduled run:** `extensions/webhooks/src/completion.ts:119-120`
  - body: Pending completion context is keyed by `sessionKey` and later adopted on lifecycle start by that same session key, so concurrent webhook deliveries targeting one session can overwrite or misattach completion delivery. Store and consume by the scheduled job/run identity instead.
  - confidence: 0.9
- **[P1] Gate exec delivery before exposing it to webhooks:** `extensions/webhooks/src/delivery.ts:121-130`
  - body: This branch lets a configured webhook route execute a local command through `spawn` when an authenticated external POST arrives. That is a new security boundary and should be removed or explicitly gated/approved before merge.
  - confidence: 0.91
- **[P1] Keep scheduled turns on the documented authority boundary:** `src/plugins/host-hook-scheduled-turns.ts:209-210`
  - body: Changing scheduled turns from bundled-only to `trustedOfficialInstall` widens a Plugin SDK authority surface while the public SDK docs still call it bundled-only. Either keep the current boundary or update the contract/docs with maintainer approval.
  - confidence: 0.88
- **[P2] Keep subscription reloads out of the request path:** `extensions/webhooks/index.ts:64-67`
  - body: Every webhook POST calls `resolveTargetsByPath`, which loads dynamic targets from the subscription store and may stat/read the JSON state file. Gateway plugin metadata is process-stable, so dynamic subscription changes should update a lifecycle-owned in-memory snapshot instead of adding filesystem freshness checks to inbound delivery.
  - confidence: 0.87
