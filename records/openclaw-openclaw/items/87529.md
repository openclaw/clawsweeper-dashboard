## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Store completion context only after scheduling succeeds:** `extensions/webhooks/src/dispatch.ts:54-60`
  - body: `executeAgentDispatch` stores completion context before `scheduler(scheduleRequest)` has accepted the turn. If scheduling throws or returns no handle, the session-keyed pending context remains; if multiple webhook deliveries share a session, the next lifecycle start can pick up stale or wrong webhook payload/headers. Store after a successful handle and correlate to the scheduled job/run instead of the shared session key.
  - confidence: 0.91
- **[P1] Gate exec delivery before exposing it to webhook routes:** `extensions/webhooks/src/delivery.ts:58-62`
  - body: `runExecDelivery` lets an authenticated webhook route spawn a configured local command with templated payload/env. A provider webhook secret is not enough review authority for remote-triggered local process execution; remove this mode or put it behind an explicit maintainer-approved operator gate with docs and tests.
  - confidence: 0.9
- **[P1] Keep scheduled-turn authority on the approved boundary:** `src/plugins/host-hook-scheduled-turns.ts:209-210`
  - body: This widens scheduled-turn access from bundled plugins to trusted official installed plugins. That is a plugin API/security-boundary change, not just Webhooks plumbing, and it needs explicit maintainer approval plus public contract/docs coverage or should remain bundled-only in this PR.
  - confidence: 0.86
- **[P2] Keep dynamic subscription reads out of webhook POST handling:** `extensions/webhooks/index.ts:76-79`
  - body: Every webhook request calls `subscriptionStore.loadTargets()`, which stats/reads/parses the dynamic subscription file before handling delivery. This puts filesystem freshness work on the inbound message path; keep an in-memory snapshot updated by subscribe/remove or add an explicit reload path instead.
  - confidence: 0.84
- **[P3] Document the still-registered Gmail subcommands:** `docs/cli/webhooks.md:24-31`
  - body: The new CLI page lists only `subscribe`, `list`, `remove`, and `test`, but the core registrar still installs `openclaw webhooks gmail setup/run`. Either keep those shipped subcommands in the reference or intentionally remove/redirect them so the docs match the CLI.
  - confidence: 0.8
