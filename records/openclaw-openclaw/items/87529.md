## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Correlate completion delivery to the accepted run:** `extensions/webhooks/src/dispatch.ts:54-60`
  - body: `onAgentCompletionDispatch` runs before the scheduler accepts the turn, and completion lookup later matches by `sessionKey`. If scheduling fails or two webhooks schedule the same session, a later lifecycle event can pick up the wrong pending context and deliver another run's output to this route's completion target; register after acceptance and key it to the accepted job/run.
  - confidence: 0.9
- **[P1] Persist dynamic subscriptions through plugin state:** `extensions/webhooks/src/subscriptions.ts:106`
  - body: Dynamic subscriptions include HMAC secrets, but this stores them in `webhooks/subscriptions.json`. OpenClaw runtime state and plugin scratch data should use SQLite/plugin KV rather than JSON sidecars, so this needs to move to the approved state path before merge.
  - confidence: 0.84
- **[P2] Keep the shipped Gmail webhook commands documented:** `docs/cli/webhooks.md:24-38`
  - body: The PR still loads the core `openclaw webhooks gmail setup|run` commands before plugin CLI commands, but this page now lists only the dynamic subscription commands. Keep the Gmail setup/run reference alongside the new commands so a shipped CLI surface does not disappear from docs.
  - confidence: 0.87
- **[P2] Avoid polling subscription files from webhook delivery:** `extensions/webhooks/index.ts:76-79`
  - body: The request handler calls `resolveTargetsByPath()` for each inbound webhook, and that path reloads dynamic targets from subscription storage. Gateway/plugin hot paths should use a lifecycle-owned target snapshot updated by subscribe/remove/startup instead of stat/read/rebuild work on every delivery.
  - confidence: 0.82

## Risks / Open Questions
