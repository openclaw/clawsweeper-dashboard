## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Store completion context only after scheduling succeeds:** `extensions/webhooks/src/dispatch.ts:55-60`
  - body: `onAgentCompletionDispatch` persists the completion delivery context before `scheduler(scheduleRequest)` runs. If scheduling fails or another run starts for the same sessionKey, the later lifecycle event can consume stale webhook body/header context and deliver the wrong completion; move this storage after a successful handle and correlate it to the scheduled run rather than only the session.
  - confidence: 0.91
- **[P1] Do not persist webhook auth headers for completions:** `extensions/webhooks/src/completion.ts:135`
  - body: The completion context stores the full inbound header map, which can include `Authorization`, HMAC signatures, or custom secret headers, into the keyed store for up to the pending TTL. Redact or omit authentication headers before persistence and before exposing them to delivery templates.
  - confidence: 0.88
- **[P1] Gate exec delivery before exposing it to webhook routes:** `extensions/webhooks/src/config.ts:100-108`
  - body: Adding `mode: "exec"` makes an authenticated webhook route able to spawn a local command, and the implementation inherits the Gateway process environment. This needs to be removed or put behind an explicit maintainer-approved/local-only gate so a leaked webhook secret cannot become remote local code execution.
  - confidence: 0.87
- **[P1] Keep scheduled-turn authority on the approved boundary:** `src/plugins/host-hook-scheduled-turns.ts:209-210`
  - body: This changes the scheduler guard so trusted official installed plugins can schedule and unschedule session turns, but current SDK docs still describe this API as bundled-only. Preserve the existing boundary or add the maintainer-approved SDK/docs/contract path for expanding it.
  - confidence: 0.84
- **[P2] Keep dynamic subscription reads out of webhook delivery:** `extensions/webhooks/index.ts:76-78`
  - body: Every webhook POST now calls `resolveTargetsByPath`, which reloads dynamic subscriptions through a filesystem-backed JSON store. That adds stat/read/parse latency and failure modes to inbound delivery; keep route state in a lifecycle-owned in-memory snapshot updated by subscribe/remove instead.
  - confidence: 0.82
