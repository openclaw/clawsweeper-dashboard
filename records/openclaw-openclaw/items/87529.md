## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Do not persist webhook auth headers in completion state:** `extensions/webhooks/src/completion.ts:117-120`
  - body: `storeCompletionDispatch` copies every inbound webhook header into the completion context and then persists it via `openKeyedStore`. Bearer/header-auth routes can put the route secret in `authorization` or a custom auth header, so completion delivery should store a redacted header set or only the specific non-secret fields needed for templates.
  - confidence: 0.89
- **[P1] Key completion delivery to the scheduled run, not the session:** `extensions/webhooks/src/completion.ts:119-120`
  - body: Pending completion state is keyed only by `sessionKey`, so two webhook events queued to the same session before the first run starts will overwrite each other and the first completion can be delivered with the second event's route/body/delivery target. The completion context needs to be associated with the scheduled job/run or another unique dispatch token instead of the shared session.
  - confidence: 0.86
- **[P1] Gate exec delivery before exposing it to webhooks:** `extensions/webhooks/src/delivery.ts:58-61`
  - body: This path lets a configured webhook route render external payload data into process args/env and spawn a local command inside the Gateway runtime. That is a new remote-to-local execution boundary and should either be removed for this PR or guarded by an explicit maintainer-approved operator setting/allowlist with docs and tests.
  - confidence: 0.84
- **[P1] Keep scheduled turns on the documented authority boundary:** `src/plugins/host-hook-scheduled-turns.ts:209-210`
  - body: The PR changes scheduled session turns from bundled-only to also allow trusted official installed plugins, but the public SDK docs still define this as bundled-only authority. Either keep the runtime guard unchanged for this feature or make the plugin API contract change explicit with maintainer approval, docs, and compatibility tests.
  - confidence: 0.82
- **[P2] Keep subscription state out of the webhook request path:** `extensions/webhooks/index.ts:63-65`
  - body: Every webhook request now calls `resolveTargetsByPath`, which loads dynamic targets by stat/read of the subscription file before routing. Gateway/plugin metadata is process-stable in this repo, so dynamic subscription changes should update an owned in-memory snapshot on subscribe/remove rather than polling the file on each static or dynamic delivery.
  - confidence: 0.82
- **[P2] Align the documented agent dispatch response:** `docs/plugins/webhooks.md:101-110`
  - body: The docs promise a `202` response with `jobId`, but the implementation fast-ACKs before `scheduler(...)` resolves and returns only `accepted: true`. Please update the docs/example or change the implementation to return the actual scheduled job id consistently.
  - confidence: 0.88

## Risks / Open Questions

- Completion delivery currently persists raw webhook headers, which can include Authorization or custom webhook secret headers, in the plugin state store.
- Completion delivery is keyed by sessionKey instead of the scheduled run/job, so two webhooks targeting the same session can overwrite each other and deliver the wrong completion to an external target.
- The exec delivery mode lets an authenticated webhook route trigger a configured local process in the Gateway process with webhook-rendered args/env, which needs explicit maintainer security acceptance or a narrower gate.
- The PR broadens scheduled-turn authority from bundled plugins to trusted official installed plugins while public SDK docs still describe the surface as bundled-only.
- Dynamic subscription loading adds subscription-file stat/read behavior to every webhook request path, including static routes, which conflicts with the process-stable metadata guidance for hot paths.
- The PR body still says fresh real E2E proof is missing for the current Gateway-managed provider flows, so reviewer confidence should not rely on prior direct-ingress proof alone.
