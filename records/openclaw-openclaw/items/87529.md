## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Key completion context by run, not session:** `extensions/webhooks/src/completion.ts:118-120`
  - body: This stores pending completion delivery under only `sessionKey`. If two webhook events schedule agent runs into the same session before the first run starts, the second context overwrites the first; whichever run starts can then deliver its completion to the wrong external target. Completion delivery needs a run/job-specific correlation key or another collision-safe queue before this can be safe.
  - confidence: 0.9
- **[P2] Keep validation claims aligned with supplied proof:** `docs/plugins/webhooks-enterprise-validation.md:36-39`
  - body: The new validation page says a live GitHub webhook E2E delivered `pull_request` and `pull_request_review` events, but the PR body says a live GitHub webhook delivery was not tested and only Codebase E2E was run. Please change this to the actual Codebase proof or remove the live GitHub claim.
  - confidence: 0.94
- **[P2] Mirror auth header requirements in the manifest schema:** `extensions/webhooks/openclaw.plugin.json:40`
  - body: The runtime schema rejects `header` and `hmac-sha256` auth without `header`, but the manifest schema only requires `mode` and `secret`. That lets manifest-based config validation accept a route that later fails at plugin startup; add the same conditional requirement to `openclaw.plugin.json` or narrow the supported manifest surface.
  - confidence: 0.83
