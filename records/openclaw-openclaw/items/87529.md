## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Gate exec delivery behind an explicit security boundary:** `extensions/webhooks/src/delivery.ts:58-60`
  - body: This spawns a configured local command with templated args/env/stdin from the authenticated webhook payload while inheriting the Gateway environment. Without an explicit operator opt-in, warning/docs, and accepted command policy, exposing a route to a SaaS sender turns a leaked route secret or misconfiguration into remote command execution on the Gateway host.
  - confidence: 0.91
- **[P1] Keep scheduled turns on the documented authority boundary:** `src/plugins/host-hook-scheduled-turns.ts:209-210`
  - body: This changes the guard so trusted official external installs can schedule and unschedule Cron-backed session turns, but the current public Plugin SDK docs still describe this surface as bundled-only. Either keep the bundled-only boundary, or land a maintainer-approved SDK contract/docs/test update that makes the authority expansion explicit.
  - confidence: 0.88
- **[P1] Key completion delivery by run instead of session:** `extensions/webhooks/src/completion.ts:119-122`
  - body: Pending completion context is stored under `context.sessionKey`, then lifecycle start looks it up by session key. Two accepted webhooks for the same session can overwrite each other before the first run starts, and a failed schedule can leave stale context for a later unrelated run; key this by the scheduled job/run correlation instead.
  - confidence: 0.9
- **[P2] Keep dynamic subscription lookup out of the request hot path:** `extensions/webhooks/index.ts:63-65`
  - body: Every webhook request calls `loadTargets()`, which goes through the subscription store's file stat/read path. Dynamic subscribe/remove can update a lifecycle-owned in-memory snapshot; request-time filesystem freshness checks add avoidable latency and conflict with the repo's process-stable metadata rule.
  - confidence: 0.82
- **[P2] Fix the documented agent dispatch response:** `docs/plugins/webhooks.md:101-110`
  - body: The docs say accepted agent dispatch returns a `jobId`, but the implementation fast-acks before `scheduleSessionTurn` resolves and only returns `accepted`, `sessionKey`, and optional `completionDelivery`. Operators following this page cannot get the documented job id from the endpoint.
  - confidence: 0.93
- **[P2] Remove unproven live GitHub validation claims:** `docs/plugins/webhooks-enterprise-validation.md:36-39`
  - body: This page claims a live GitHub end-to-end run, while the PR body says fresh real E2E reports for GitHub PR, Codebase MR, and Meego events are still not completed. Either attach and reference the real proof, or describe these scenarios as unvalidated/planned.
  - confidence: 0.86
