## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Key completion delivery by the scheduled run:** `extensions/webhooks/src/completion.ts:119-120`
  - body: `storeCompletionDispatch` writes pending completion context under `sessionKey`, and `executeAgentDispatch` stores it before the scheduler has accepted the run. If two webhook-triggered runs share a session, or scheduling fails and a later run starts in that session, the completion can be delivered to the wrong external channel or exec target. Correlate the context to the accepted scheduled run/run context and clear it on scheduling failure.
  - confidence: 0.9
- **[P1] Gate exec delivery behind an explicit security boundary:** `extensions/webhooks/src/delivery.ts:58-61`
  - body: This new path spawns a local command with templated webhook data as args/env/stdin under Gateway privileges. A route secret leak or overbroad route config becomes remote command execution against the operator machine, but the PR does not add an explicit operator gate, warning, or accepted security model for that boundary. Remove exec delivery or gate and document it before merge.
  - confidence: 0.86
- **[P1] Keep scheduled turns bundled-only until the SDK contract changes:** `src/plugins/host-hook-scheduled-turns.ts:209-210`
  - body: The PR passes `trustedOfficialInstall` into `schedulePluginSessionTurn` and changes the guard so official installed plugins can schedule session turns, but the SDK docs still advertise this capability as bundled-only and there is no visible maintainer approval for the plugin-contract expansion. Keep the existing bundled-only boundary, or update the SDK contract and get explicit approval with compatibility/security proof.
  - confidence: 0.84
- **[P2] Remove unproven GitHub live-validation claims:** `docs/plugins/webhooks-enterprise-validation.md:36-39`
  - body: The validation page says a live GitHub webhook delivered PR events through a public tunnel, while the PR body supplies Codebase direct-HTTP proof and says the stricter webhook acceptance reports are not complete. Keep the docs to the proof that exists or add the actual redacted GitHub/WebSocket relay evidence before merging.
