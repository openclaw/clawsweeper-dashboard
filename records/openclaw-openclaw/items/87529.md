## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Add relay to the manifest config schema:** `extensions/webhooks/src/config.ts:667`
  - body: `resolveWebhooksPluginRuntimeConfig` now reads `parsed.relay`, but the bundled plugin manifest still has `additionalProperties: false` and only exposes `routes` at the top level. User config containing `plugins.entries.webhooks.config.relay` will be rejected by manifest validation before this runtime code can start the connector, so add the relay schema to `openclaw.plugin.json` and cover manifest validation.
  - confidence: 0.93
- **[P1] Key completion delivery by the scheduled run:** `extensions/webhooks/src/dispatch.ts:54-60`
  - body: The completion context is stored under `sessionKey` before the scheduler returns a handle, and the event subscriber later binds the first lifecycle start for that session to that context. Two webhook events targeting the same session, or a scheduling failure after the context is stored, can deliver the wrong completion or a stale one; store after accepted scheduling with run/job identity and add same-session/failure coverage.
  - confidence: 0.9
- **[P1] Gate exec delivery behind an explicit security boundary:** `extensions/webhooks/src/delivery.ts:58-61`
  - body: Any authenticated route with `deliver.mode: "exec"` now spawns a configured command under Gateway privileges with webhook-rendered args/env/stdin. That is a remote-to-local command execution path, so this needs an explicit opt-in/security decision, operator warnings, and boundary proof before being exposed in a bundled plugin.
  - confidence: 0.88
- **[P1] Keep scheduled turns bundled-only until the SDK contract changes:** `src/plugins/host-hook-scheduled-turns.ts:209-210`
  - body: This changes the host hook from `origin === "bundled"` to also allow `trustedOfficialInstall`, while the public Plugin SDK docs still call `scheduleSessionTurn` and tag cleanup bundled-only. Either remove this widening for the bundled Webhooks use case or land the SDK/security contract and docs update together.
  - confidence: 0.86
- **[P2] Remove unproven GitHub validation claims:** `docs/plugins/webhooks-enterprise-validation.md:36-39`
  - body: The PR body says relay-backed GitHub, Codebase, and Meego real reports are not completed and only describes Codebase direct-ingress proof, but this page claims a live GitHub end-to-end run was verified. Replace the claim with the supplied evidence or attach redacted live proof before publishing the validation page.
  - confidence: 0.88
- **[P2] Fix the documented agent dispatch response:** `docs/plugins/webhooks.md:101-112`
  - body: The docs promise a scheduled `jobId` in the `agent_dispatch` response, but the implementation returns a fast `202` with `accepted: true` and does not wait for the scheduler handle. Update the response example or change the implementation so users do not build against a field that is never returned.
  - confidence: 0.84
