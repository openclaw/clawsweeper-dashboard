## Summary

Keep open: the index-first direction is valid, but current main already superseded the update-command half with a shared resolver while the auto-update fallback still has the entry-first gap. The branch is conflicting, duplicates entrypoint-selection policy, and still lacks after-fix real runtime proof.

## What This Changes

The PR reorders update and auto-update gateway entrypoint candidate lists to prefer `dist/index.*` before `dist/entry.*` and adds focused Vitest coverage.

## Best Possible Solution

Rebase to current main, change only the auto-update fallback to use the shared index-first gateway entrypoint resolver, keep focused regression coverage, and add redacted real upgrade/runtime proof.

## Reproduction Assessment

Yes, source-level. On current main, when the auto-update fallback runs under Node/Bun without `process.argv[1]` and both `dist/entry.js` and `dist/index.js` exist, `runAutoUpdateCommand` checks `entry.js` first.

## Solution Assessment

No. The index-first direction is correct, but the maintainable fix is to reuse the existing shared resolver for auto-update and drop the obsolete update-command hunk.
