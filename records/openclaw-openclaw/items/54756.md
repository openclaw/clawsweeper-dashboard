## Summary

Keep open: current main already fixed the update service-refresh side through the shared index-first gateway entrypoint resolver, but the auto-update root fallback still prefers legacy `dist/entry.js` before `dist/index.js`. The PR is useful source work, but it needs rebasing or a narrow replacement because the `update-command.ts` part is now obsolete on main.

## What This Changes

The PR reorders gateway entrypoint candidate lists in update refresh and auto-update fallback paths so `dist/index.js`/`dist/index.mjs` are preferred before legacy `dist/entry.js`/`dist/entry.mjs`, with focused tests.

## Best Possible Solution

Land a rebased or replacement patch that updates only the remaining auto-update root fallback to share the current gateway entrypoint resolver, preserves the existing `process.argv[1]` behavior, adds the focused regression test, and includes the required changelog entry.

## Reproduction Assessment

Yes, from source inspection: when `runAutoUpdateCommand()` has a Node/Bun `execPath`, no `process.argv[1]`, and a root containing both `dist/entry.js` and `dist/index.js`, the current fallback loop chooses `entry.js` first. I did not execute the regression test because `pnpm` is unavailable here.

## Solution Assessment

Partially. The index-first direction is correct and already matches the shared resolver, but the PR should be rebased or replaced so it removes obsolete `update-command.ts` changes and fixes the remaining `update-startup.ts` fallback against current main.
