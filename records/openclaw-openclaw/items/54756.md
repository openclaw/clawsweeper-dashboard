## Summary

## What This Changes

The PR reorders gateway update and auto-update entrypoint candidates to prefer `dist/index.*` over `dist/entry.*` and adds focused Vitest coverage.

## Best Possible Solution

Rebase this into a narrow change that makes `update-startup` reuse `resolveGatewayInstallEntrypoint`, keeps update-command on the current shared resolver, adds focused regression coverage, and includes redacted real runtime proof before merge.

Yes, source-reproducible: current main will choose `dist/entry.js` first in the auto-update root fallback when `process.argv[1]` is absent and both built entrypoints exist. I did not run the flow in this read-only review.

## Solution Assessment

No. The direction is right, but the best current fix is to call the shared gateway entrypoint resolver from auto-update and drop the obsolete update-command resolver hunk.
