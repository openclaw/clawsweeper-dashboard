## What This Changes

The PR reorders gateway update and auto-update entrypoint selection to prefer `dist/index.*` over `dist/entry.*` and adds focused Vitest coverage.

## Best Possible Solution

Rebase into a narrow fix that makes auto-update reuse the shared gateway entrypoint resolver, leaves update-command on the current shared resolver, keeps focused regression coverage, and includes redacted runtime proof before merge.

## Reproduction Assessment

Yes, source-reproducible: current main will choose `dist/entry.js` first in the auto-update root fallback when `process.argv[1]` is absent and both built entrypoints exist. I did not run the live update flow in this read-only review.

## Solution Assessment

No. The direction is right, but current main already moved update-command to the shared resolver, and the remaining auto-update fallback should call that helper instead of copying another hardcoded list.
