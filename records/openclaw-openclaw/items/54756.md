## Summary

Keep open. Prior comments are right that index-first resolution is the correct direction, but current main already superseded the update-command half via the shared resolver, while the auto-update fallback still has a source-visible entry-first gap and the PR lacks real runtime proof.

## What This Changes

The PR reorders gateway update and auto-update entrypoint fallback candidates to prefer `dist/index.*` before `dist/entry.*` and adds focused regression tests.

## Best Possible Solution

Rebase into a narrow fix that makes auto-update reuse the shared gateway entrypoint resolver, drops the superseded update-command hunk, keeps focused regression coverage, and includes redacted runtime proof before merge.

## Reproduction Assessment

Yes, source-level. On current main, when `process.argv[1]` is absent, the runtime is Node/Bun, and both `dist/entry.js` and `dist/index.js` exist under the package root, `runAutoUpdateCommand` checks `entry.js` first; I did not run the live update flow.

## Solution Assessment

No. The direction is correct, but current main already has a shared index-first resolver, so the remaining auto-update fallback should call that helper instead of copying another hardcoded list.
