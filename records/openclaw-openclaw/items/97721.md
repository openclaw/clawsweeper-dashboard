## Summary

Keep open: the bounded-read change is useful and the loopback proof is credible, but the patch currently breaks the typed Discord gateway fetch seam by casting a narrow response object to a full `Response`. Existing gateway proxy tests and any current custom `DiscordGatewayFetch` implementation that satisfies the exported `ok/status/text` contract can now fail before metadata is parsed.

## What This Changes

The PR caps Discord gateway metadata response materialization and text reads at 16 MiB, then adds a loopback HTTP bounded-read regression test.

## Best Possible Solution

Keep the 16 MiB cap, but make the fetch response contract honest: either require a full `Response` and update all Discord monitor mocks/callers, or preserve the narrow seam with a bounded reader that does not rely on an unsafe cast.

## Reproduction Assessment

Yes, source-reproducible. The patched line passes a `Pick<Response, "ok" | "status" | "text">` into a helper that needs `.body` or `.arrayBuffer()`, and existing Discord proxy tests already construct such narrow response objects.

## Solution Assessment

No, not as written. The bounded read belongs in this path, but the maintainable fix is to align the response type and tests with full `Response` objects or implement a bounded helper that explicitly preserves the narrow contract.
