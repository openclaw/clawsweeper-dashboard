## Summary

Keep open. Current main still rejects `tools.web.search.maxResults` in the agent-facing gateway tool before `config.patch` reaches Gateway, while the field is documented/schema-backed; the remaining decision is whether this web-search tuning leaf belongs in the fail-closed config-write allowlist and whether SearXNG should honor the same default.

## What This Changes

Allow `tools.web.search.maxResults` to be changed through `gateway config.patch` so native `web_search` defaults are not fixed at 5 results.

## Best Possible Solution

Approve a narrow policy for `tools.web.search.maxResults`, then implement exact allowlist and guard coverage plus any provider-default fix needed for SearXNG; otherwise document that this setting must remain operator-edited outside the agent gateway tool.

## Reproduction Assessment

Yes, source-reproducible. A patch changing `tools.web.search.maxResults` produces a changed path absent from `ALLOWED_GATEWAY_CONFIG_PATHS`, and the agent-facing gateway tool throws before forwarding `config.patch`.

## Solution Assessment

Unclear. Adding one allowlist entry is the smallest fix for the protected-path error, but maintainers should first confirm this web-search result-count leaf is safe to make agent-tunable and whether SearXNG should also honor it as a global default.
