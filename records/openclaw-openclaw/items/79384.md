## Summary

Keep open: current main and the latest release still reject this agent-facing `gateway config.patch` path, and the linked fix PR was closed unmerged. The request is plausible, but widening this allowlist changes a documented model-to-operator trust boundary, so it needs maintainer policy review rather than cleanup closure.

## What This Changes

The issue asks to make `tools.web.search.maxResults` configurable through `gateway config.patch` instead of being rejected as a protected config path.

## Best Possible Solution

Make an explicit maintainer decision on whether web-search result limits belong in the agent-tunable gateway allowlist; if yes, add the narrow allowlist entry with regression coverage and make provider defaults consistently honor `tools.web.search.maxResults`.

## Reproduction Assessment

Yes, from source inspection. Current main lacks `tools.web.search.maxResults` in the gateway tool allowlist, and the guard rejects any changed config path outside that list before forwarding `config.patch`.

## Solution Assessment

Unclear. Adding one allowlist path is the smallest patch for the reported rejection, but the allowlist is intentionally fail-closed security policy and SearXNG may also need provider-default wiring for the user-visible result-count problem.
