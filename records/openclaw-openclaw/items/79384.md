## Summary

Keep open: current main and `v2026.6.6` still reject agent-facing `gateway config.patch` changes to `tools.web.search.maxResults` because the gateway tool config allowlist intentionally fails closed. The direct fix PR closed unmerged, and broader protected-config policy work remains open, so this needs maintainer/security direction rather than cleanup closure.

## What This Changes

The issue asks for the native `web_search` default result limit to be changeable through `tools.web.search.maxResults` when using the agent-facing gateway config patch path.

## Best Possible Solution

Decide whether web-search tuning belongs in the built-in low-risk agent config allowlist or should wait for the broader protected-config approval policy, then update the allowlist, guard tests, and docs consistently.

## Reproduction Assessment

Yes. Source inspection shows an agent `gateway` tool `config.patch` changing `tools.web.search.maxResults` will be rejected because that path is absent from `ALLOWED_GATEWAY_CONFIG_PATHS`; tests were not run because this review is read-only.

## Solution Assessment

Unclear. Adding the path to the allowlist is a small plausible fix, but the best solution needs maintainer/security approval because the allowlist is an intentional model-to-operator trust boundary.
