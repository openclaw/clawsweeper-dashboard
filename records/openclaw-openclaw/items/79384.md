## Summary

Keep this open: current main still blocks the documented web-search maxResults config path, and an allow-list-only attempted fix was closed unmerged. The source also shows SearXNG still falls back to the shared default count unless args.count is supplied, so the best fix should cover both config.patch and the runtime default behavior.

## What This Changes

The issue requests making tools.web.search.maxResults configurable through gateway config.patch instead of being rejected as a protected path.

## Best Possible Solution

Allow the existing maxResults config key through the gateway mutation guard and ensure managed web_search uses it as the default count when the tool call omits count, especially for SearXNG.

## Reproduction Assessment

Yes, by source inspection: a config.patch changing tools.web.search.maxResults is collected as a changed leaf path and rejected because it is absent from ALLOWED_GATEWAY_CONFIG_PATHS. The linked closed PR also supplied a focused gateway-tool test proving the same path would be forwarded after adding the allow-list entry.

## Solution Assessment

No for the linked allow-list-only fix as a complete solution. It is necessary, but SearXNG currently reads only args.count and otherwise falls back to DEFAULT_SEARCH_COUNT, so the safer full fix should also make the configured maxResults feed provider execution when count is omitted.
