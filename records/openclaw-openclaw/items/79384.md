## Summary

Current main still blocks this path: `tools.web.search.maxResults` is documented and schema-backed, but the agent-facing gateway config mutation allowlist does not include it, so a `config.patch` changing only that field remains rejected. The linked fix PR addressed the same path but was closed unmerged, and expanding this allowlist needs maintainer policy review because it is an explicit model-to-operator trust boundary.

## What This Changes

Request to make `tools.web.search.maxResults` patchable through the agent-facing gateway `config.patch` flow instead of being blocked as a protected config path.

## Best Possible Solution

Keep the fail-closed mutation model, and if maintainers approve this as a bounded tuning knob, add only `tools.web.search.maxResults` to the allowlist with a focused gateway-tool regression test while leaving sensitive web-search fields protected.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: patch current config from `tools.web.search.maxResults: 5` to `10` through the agent-facing gateway tool, and the changed path is not allowlisted, so `config.patch` throws before forwarding.

## Solution Assessment

Unclear pending maintainer policy review. The closed linked PR’s single-path allowlist change is the narrowest code shape if approved, while removing broader protected paths such as timeout or API-key-related fields would be riskier.
