## Summary

Keep open. Current main still rejects `tools.web.search.maxResults` through the agent gateway config mutation guard, and the SearXNG default path still falls back to 5 without reading the documented global `maxResults`; the related fix PR closed unmerged. The allowlist is an explicit model-to-operator trust boundary, so maintainer scope approval is needed before changing it.

## What This Changes

Make `tools.web.search.maxResults` control native `web_search` result count and allow it to be changed through `gateway config.patch` instead of staying effectively fixed at 5 for SearXNG-heavy use.

## Best Possible Solution

Maintain the existing config contract, make affected web-search providers honor `tools.web.search.maxResults`, and have maintainers decide whether the bounded web-search tuning fields should be added to the agent gateway mutation allowlist.

## Reproduction Assessment

Yes, by source inspection. A gateway-tool `config.patch` that changes `tools.web.search.maxResults` reaches `assertGatewayConfigMutationAllowed`, collects that leaf path, and rejects it because the path is absent from `ALLOWED_GATEWAY_CONFIG_PATHS`; the SearXNG no-count path also resolves to the shared default of 5.

## Solution Assessment

Unclear until maintainer scope is set. Adding `tools.web.search.maxResults` to the allowlist is the narrow runtime-patch fix, but the security docs frame that allowlist as a deliberate trust-boundary decision; provider-level config honoring can be fixed separately without broadening the allowlist.
