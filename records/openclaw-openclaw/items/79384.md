## Summary

Keep this open: current main still blocks `tools.web.search.maxResults` through the agent-facing gateway config mutation guard, and the related fix PR was closed unmerged. Source also shows SearXNG still falls back to the shared default of 5 unless the tool call passes an explicit count, so the reported self-hosted search case remains real.

## What This Changes

The issue requests making `tools.web.search.maxResults` effective and runtime-configurable for native `web_search`, especially for SearXNG-backed searches.

## Best Possible Solution

Keep the documented `tools.web.search.maxResults` contract, make native providers such as SearXNG honor it consistently, and only add the gateway `config.patch` allowlist entry if maintainers agree this bounded tuning field is safe for agent-facing runtime edits.

## Reproduction Assessment

Yes from source: the current guard rejects any changed path outside `ALLOWED_GATEWAY_CONFIG_PATHS`, and that list does not include `tools.web.search.maxResults`. The SearXNG path also source-reproduces the 5-result fallback when no explicit count is passed.

## Solution Assessment

Unclear until maintainers approve the allowlist policy; the narrow code fix is likely to add `tools.web.search.maxResults` regression coverage and make SearXNG read `ctx.searchConfig?.maxResults`, but runtime patch permission crosses a trust-boundary decision.
