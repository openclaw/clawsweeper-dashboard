## Summary

Keep open: current main still blocks agent-facing gateway config.patch changes to tools.web.search.maxResults, and the linked fix PR was closed unmerged. The fix needs maintainer judgment because this allowlist is an explicit model-to-operator trust boundary and the SearXNG case likely needs provider handling too.

## What This Changes

Request to make the native web_search result-count setting configurable through tools.web.search.maxResults via gateway config.patch.

## Best Possible Solution

Define a bounded web-search tuning policy for gateway config.patch, then implement the approved maxResults behavior with regression coverage and provider consistency, including SearXNG if intended.

## Reproduction Assessment

Yes, source-reproducible: a config.patch that changes tools.web.search.maxResults is collected as that leaf path and rejected because the path is absent from ALLOWED_GATEWAY_CONFIG_PATHS. I did not run a live gateway command because this review is read-only.

## Solution Assessment

Unclear: the closed PR's allowlist-only approach is a plausible start for maxResults, but it does not address timeoutSeconds policy or the SearXNG provider ignoring global searchConfig.maxResults. A maintainer should first approve the safe mutable fields and provider scope.
