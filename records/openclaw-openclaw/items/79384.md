## Summary

Keep open. Current main still blocks agent-driven `gateway config.patch` edits to `tools.web.search.maxResults`, while the web_search providers already read that config when it is present; the related fix PR at https://github.com/openclaw/openclaw/pull/79391 closed unmerged, so nothing has landed for this issue.

## What This Changes

The issue asks for `gateway config.patch` to allow changing the documented `tools.web.search.maxResults` setting for native `web_search` result count.

## Best Possible Solution

Decide the allowlist policy explicitly; if approved, add the chosen `tools.web.search.*` tuning paths to the gateway config mutation allowlist with focused regression coverage, otherwise document that these knobs must be changed through direct config/CLI flows rather than the agent-facing gateway tool.

## Reproduction Assessment

Yes, source-level: a `config.patch` that changes `tools.web.search.maxResults` becomes a changed path outside `ALLOWED_GATEWAY_CONFIG_PATHS`, so the current guard rejects it before forwarding. I did not run a live gateway command because this review was read-only.

## Solution Assessment

Unclear until policy is confirmed. Adding the allowlist entry is mechanically narrow, but the file and security docs frame this list as a deliberate trust-boundary control, so the safest solution is maintainer approval plus targeted tests.
