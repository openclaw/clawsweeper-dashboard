## Summary

Keep open: current main still maps an explicit device-token `scope-mismatch` to the generic `device_token_mismatch`, and the active fix path is the open linked PR at https://github.com/openclaw/openclaw/pull/79296 rather than closing this issue before merge.

## What This Changes

The issue asks for operator device-token scope mismatches to be surfaced distinctly instead of appearing as a generic device token mismatch.

## Best Possible Solution

Review and land or revise https://github.com/openclaw/openclaw/pull/79296, then close this issue after merge and split any remaining client-scope alignment into a separate focused follow-up if needed.

## Reproduction Assessment

Yes, by source inspection: the clients request scopes outside the bootstrap handoff allowlist, `verifyDeviceToken` can return `scope-mismatch`, and current auth-context collapses explicit device-token failures to `device_token_mismatch`. A focused unit repro would stub `verifyDeviceToken` to return `scope-mismatch` in `resolveConnectAuthDecision`.

## Solution Assessment

Unclear as a complete fix: the linked PR is the narrowest server-side diagnostic repair, but it does not change the client/server scope gap. Maintainers should decide during that PR review whether diagnostic propagation alone is sufficient or client default scopes also need a follow-up.
