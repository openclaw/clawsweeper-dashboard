## Summary

Keep open: current main still has the source-level request-time reload path, and an open closing PR at https://github.com/openclaw/openclaw/pull/80691 is already paired with this issue but has not merged and has compatibility/security review feedback to resolve.

## What This Changes

The issue reports that first fresh inbound dispatch can reload the standalone runtime plugin registry even after gateway startup populated a warm active registry.

## Best Possible Solution

Fix and land the open linked PR with a compatibility-aware warm active-registry reuse path plus stale-config/workspace/plugin-scope regression coverage.

## Reproduction Assessment

Yes. The issue gives concrete gateway boot and fresh-DM steps with profiling data, and current main source traces dispatch through ensureRuntimePluginsLoaded into a strict compatibility miss followed by loadOpenClawPlugins; I did not run the live heap profile.

## Solution Assessment

Unclear as currently proposed in the linked PR. Reusing the warm registry is the right direction, but the fix should recognize the startup-rich versus dispatch-lean compatibility case without dropping cache-shaping loadOptions safeguards.
