## What This Changes

The issue reports a redundant standalone runtime plugin registry reload on the first dispatch after gateway startup when startup and dispatch load-option cache keys differ.

## Best Possible Solution

Land a compatibility-aware warm-registry reuse fix that preserves workspace, plugin-id, config, activation, trust, load-path, and runtime-mode reload boundaries, then close this issue from the merged PR.

## Reproduction Assessment

Yes, source-level: current main dispatch calls ensureRuntimePluginsLoaded, the active lookup can return undefined before workspace fallback after a strict loadOptions miss, and the standalone loader then calls loadOpenClawPlugins. I did not rerun the live heap benchmark in this read-only review.

## Solution Assessment

Yes in direction, but not by closing this issue now: the best path is to review and land the linked compatibility-aware fix or an equivalent corrected patch, then close once it is on main.
