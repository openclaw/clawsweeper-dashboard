## Summary

Keep open: current main still has a source-traceable first-dispatch reload path, and the linked open fix PR is the right place to finish the compatibility-safe repair rather than closing this issue early.

## What This Changes

The issue asks OpenClaw to avoid a redundant standalone runtime plugin registry reload on first inbound dispatch when gateway startup has already loaded a compatible active registry.

## Best Possible Solution

Land a compatibility-aware warm registry reuse fix with regression coverage for gateway-startup rich options versus dispatch-time lean options, preserving workspace, plugin-scope, config, trust, activation, and runtime-mode reload boundaries.

## Reproduction Assessment

Yes, at source level. Current main still routes inbound dispatch through lean ensureRuntimePluginsLoaded options into a strict active-registry compatibility check that falls through to loadOpenClawPlugins after a cache-key miss; the reporter also supplied real-host profiling evidence, but I did not rerun that profile.

## Solution Assessment

Yes in direction, but not by closing this issue now. Reusing the warm active registry is the right fix, provided the implementation remains compatibility-aware and the open linked PR is reviewed or repaired before merge.
