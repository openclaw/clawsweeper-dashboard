## Summary

Keep open: current main still has a source-traceable request-time reload path after a strict runtime-registry compatibility miss, and https://github.com/openclaw/openclaw/pull/80691 is an open closing PR for this issue. The issue should stay open until that PR is repaired/landed or explicitly rejected.

## What This Changes

Report that standalone runtime plugin registry loading can run on the first inbound dispatch even after gateway startup has warmed an active registry, adding dispatch latency and heap churn.

## Best Possible Solution

Review and repair or land the linked fast path while preserving workspace, plugin scope, config, trust, activation, load-path, and runtime-mode cache guards; close this issue only after that PR is merged or intentionally rejected.

## Reproduction Assessment

Yes, at source level. The issue gives concrete gateway/fresh-dispatch steps, and current main still traces from dispatchReplyFromConfig through ensureRuntimePluginsLoaded into a strict getLoadedRuntimePluginRegistry miss and loadOpenClawPlugins fallback; I did not run the live heap bench.

## Solution Assessment

Unclear as currently proposed. Reusing the warm registry is the right direction, but https://github.com/openclaw/openclaw/pull/80691 needs to preserve the loader compatibility contract rather than broadly retrying without loadOptions.
