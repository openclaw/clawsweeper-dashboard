## What This Changes

The issue requests avoiding a redundant full plugin registry reload on the first inbound dispatch by reusing a compatible warm gateway-startup registry.

## Best Possible Solution

Land or revise the linked PR into a compatibility-aware warm-registry reuse fix with regression coverage that preserves workspace, plugin-scope, and config-change reload boundaries.

Yes on direction, not yet on the exact open patch. Reusing the warm registry is the right fix, but the implementation should remain compatibility-aware rather than ignoring load-option inputs that intentionally force reloads.
