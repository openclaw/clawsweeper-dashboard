## Summary

Keep open. Current main still has the dispatch hot-path failure shape described here: the loaded-registry helper returns `undefined` on a strict load-options miss before it can apply the active registry workspace and plugin-ID fallback, so the prior fix in https://github.com/openclaw/openclaw/pull/76004 does not fully cover this narrower startup-vs-dispatch option mismatch.

## What This Changes

The issue requests a performance fix so dispatch-time standalone runtime plugin loading reuses a compatible gateway-startup registry instead of reloading plugins on first inbound dispatch.

## Best Possible Solution

Add a compatibility-aware fallback in `getLoadedRuntimePluginRegistry` for the explicit standalone required-plugin-ID startup path, while preserving workspace/config/env safeguards and the existing full-load-options mismatch behavior.

## Reproduction Assessment

Yes, at source level: current main routes dispatch through `ensureRuntimePluginsLoaded` into `ensureStandaloneRuntimePluginRegistryLoaded`, and the strict active-registry branch can return `undefined` before the workspace/plugin-ID fallback. I did not run the full hosted gateway timing reproduction in this read-only review.

## Solution Assessment

The proposed direction is right, but the safest fix is not an unconditional fallthrough for every load-options miss. It should target the explicit standalone required-plugin-ID case and keep existing tests that reject genuinely incompatible load options.
