## Summary

Keep this PR open: the requested cron list filtering is a valid feature gap and the PR wires the right surfaces, but the current head still compares the requested agent ID against stored job IDs without canonicalizing it, so mixed-case or otherwise sanitizable IDs will silently miss jobs.

## What This Changes

The PR adds an optional `agentId` filter to `cron.list` across the Gateway protocol, cron service pagination, agent cron tool, CLI, and generated Swift protocol models.

## Best Possible Solution

Land the feature after the list filter uses the same agent-ID canonicalization as create/edit paths and the PR adds focused regression tests, docs, changelog, and regenerated protocol artifacts as needed.

## Reproduction Assessment

Yes. Source inspection shows current main has no `agentId` parameter for `cron.list`, and the PR defect is reproducible from code because created/edited jobs store canonicalized agent IDs while the new filter compares the raw requested value.

## Solution Assessment

No, not yet. The API direction is narrow and maintainable, but the filter value should be normalized at the service or boundary before comparison, with tests proving CLI/tool/gateway behavior and documentation coverage.
