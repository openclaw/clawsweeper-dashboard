## Summary

Keep this PR open: the feature is useful for the linked multi-agent cron issue, but the current head still compares the requested filter value verbatim, so case/sanitization mismatches can hide matching jobs. The user-facing CLI/tool/protocol change also still needs docs, changelog, and focused regression coverage before automerge.

## What This Changes

The PR adds an optional `agentId` filter for `cron.list` through the protocol, cron service, gateway handler, agent cron tool, CLI `cron list --agent`, and generated Swift protocol models.

## Best Possible Solution

Land the focused cron list filter once the service normalizes agent IDs consistently and the CLI/tool/protocol behavior is covered in tests and docs.

## Reproduction Assessment

Yes. Source inspection shows current main has no `agentId` field in `cron.list`, and the PR defect is visible because stored cron job agent IDs are normalized while the new list filter compares the raw requested value.

## Solution Assessment

No, not yet. The API shape is reasonable, but the maintainable implementation should normalize the requested agent ID before filtering and ship with regression tests, docs, and changelog coverage.
