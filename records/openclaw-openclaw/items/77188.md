## Summary

Keep open: the feature gap is real on current main and the PR is useful, but it should not merge yet because the new filter compares raw requested agent IDs against normalized stored IDs and the user-facing change lacks required docs/changelog/tests.

## What This Changes

The PR adds optional `agentId` filtering for `cron.list` across the protocol schema, cron service, gateway handler, agent cron tool, CLI `cron list --agent`, and generated Swift protocol models.

## Best Possible Solution

Land the feature after normalizing the requested agent ID at the service boundary and shipping the protocol/CLI/tool/client updates with focused regression tests, cron docs, and a changelog entry.

## Reproduction Assessment

Yes. Source inspection shows current main has no `agentId` field in `cron.list`, and the PR defect is source-reproducible because stored cron job agent IDs are normalized while the new filter compares the raw requested value.

## Solution Assessment

No, not yet. The API shape is reasonable, but the maintainable implementation should normalize `agentId` before filtering and include regression coverage plus docs/changelog for the new user-facing CLI/tool/protocol behavior.
