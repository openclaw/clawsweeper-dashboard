## Summary

Keep open: the PR addresses a valid multi-agent cron-list feature request, but the new filter compares agent IDs verbatim while cron creation canonicalizes them, so common CLI/tool inputs can miss existing jobs; it also needs the required docs/changelog follow-through.

## What This Changes

The PR adds optional agentId filtering to cron.list across the gateway protocol/service, auto-fills it in the cron tool, and adds --agent to openclaw cron list.

## Best Possible Solution

Land this as a focused cron feature after normalizing the filter value consistently for all list callers and adding regression coverage plus user-facing docs and changelog entries.

## Reproduction Assessment

Yes for the PR defect by source inspection: cron add/edit canonicalize agent IDs, while the proposed list filter compares the raw requested value to stored job.agentId. Current main also source-reproduces the linked feature gap because cron.list has no agentId parameter.

## Solution Assessment

No, not yet: the API shape is reasonable, but the maintainable fix should normalize the requested agentId at the gateway/service boundary and include tests, docs, and changelog before merge.
