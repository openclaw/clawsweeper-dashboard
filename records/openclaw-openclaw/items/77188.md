## Summary

Keep open: the requested cron filtering gap is real on current main, but this PR still has a correctness blocker because the new filter compares the requested agent ID verbatim against stored cron job IDs that are normalized on create/update. The user-facing CLI/tool/protocol feature also needs docs, changelog, and focused regression coverage before automerge.

## What This Changes

The PR adds an optional `agentId` filter to `cron.list` across the gateway protocol, cron service options, gateway handler, agent cron tool, CLI, and generated Swift protocol models.

## Best Possible Solution

Land this as a focused cron-list filtering feature after normalizing the incoming filter consistently and adding regression tests plus public docs and changelog coverage.

## Reproduction Assessment

Yes. Source inspection shows current main has no `agentId` path for `cron.list`, and the PR defect is source-reproducible because create/update normalize stored cron `agentId` values while the new list filter compares the raw requested value.

## Solution Assessment

No, not yet. The API shape is reasonable, but the maintainable solution should normalize the filter value before comparison and ship with focused service/protocol/gateway/tool/CLI coverage plus docs and changelog.
