## Summary

Keep open: current main and latest release still lack a durable managed-service extra-CA setting, still filter edited `NODE_EXTRA_CA_CERTS` service-env values as dangerous overrides, and the linked config-key PR was closed unmerged with the author saying this issue was not resolved.

## What This Changes

The issue reports that `openclaw gateway install --force` regenerates the managed gateway service environment and drops a custom `NODE_EXTRA_CA_CERTS` value needed for macOS self-signed TLS deployments.

## Best Possible Solution

Land a maintainer-approved trust-boundary fix that uses a durable, explicit gateway service configuration or a narrower internal TLS trust mechanism instead of preserving arbitrary generated env edits.

## Reproduction Assessment

Yes from source: the forced install path rebuilds the service environment and filters `NODE_EXTRA_CA_CERTS` as a dangerous override, while macOS service env generation falls back to `/etc/ssl/cert.pem` when no installer env value is supplied. I did not run a live macOS LaunchAgent reinstall in this read-only review.

## Solution Assessment

Unclear until maintainers choose the trust boundary. Preserving arbitrary generated env edits is not the best fit for the current host-env security policy; a config-owned extra-CA path or narrower fingerprint-based trust path needs maintainer/security approval.
