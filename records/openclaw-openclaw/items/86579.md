- `P1`: The issue describes an update-time managed gateway regression that breaks real macOS self-signed/private-root TLS deployments.
- `impact:security`: The affected behavior controls TLS trust roots for the managed gateway process.

## Summary

Keep open: current main and v2026.6.9 still lack a durable managed-service extra-CA setting, still filter edited `NODE_EXTRA_CA_CERTS` service-env values as dangerous overrides, and the direct config-key PR was closed unmerged. The adjacent TLS-fingerprint PR may address one self-signed client path, but it explicitly does not change config schema or service env behavior.

## What This Changes

The issue requests a reinstall-safe way for macOS managed gateway installs to preserve or configure `NODE_EXTRA_CA_CERTS` instead of resetting it to the default CA bundle.

## Best Possible Solution

Land a maintainer/security-approved durable trust path, likely an explicit config-owned extra CA setting or narrower gateway TLS fingerprint trust mechanism, instead of preserving arbitrary generated env edits.

## Reproduction Assessment

Yes from source: the forced install path rebuilds the service environment and filters `NODE_EXTRA_CA_CERTS` as a dangerous override, while macOS service env generation falls back to `/etc/ssl/cert.pem` when no installer env value is supplied. I did not run a live macOS LaunchAgent reinstall in this read-only review.

## Solution Assessment

Unclear until maintainers choose the trust boundary. Preserving arbitrary generated env edits is not the best fit for the current host-env security policy; a config-owned extra-CA path or narrower fingerprint-based trust path needs maintainer/security approval.
