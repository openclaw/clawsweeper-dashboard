## Label Justifications

- `P1`: The issue describes an update-time managed gateway regression that can break real macOS self-signed or private-root TLS deployments.
- `impact:security`: The affected behavior controls Node TLS trust roots for the managed gateway process.

## Summary

Keep open: current main still filters edited `NODE_EXTRA_CA_CERTS` service-env values as dangerous overrides, still regenerates macOS managed service env with the `/etc/ssl/cert.pem` fallback when no installer env override is supplied, and still lacks a durable `gateway.tls.extraCaCerts`-style setting. The linked candidate PR was closed unmerged and explicitly did not resolve this issue.

## What This Changes

The issue reports that `openclaw gateway install --force` regenerates managed gateway service env files and drops a custom `NODE_EXTRA_CA_CERTS` value needed for macOS self-signed/private-root TLS trust.

## Best Possible Solution

Land a maintainer-approved trust-boundary fix that uses durable gateway service configuration or another explicit trust mechanism instead of relying on generated env-file edits.

## Reproduction Assessment

Yes from source: the install plan preserves only safe existing service-env values, while `NODE_EXTRA_CA_CERTS` is classified as a dangerous override and macOS generated service env falls back to `/etc/ssl/cert.pem` when no installer env override is present. I did not run a live macOS LaunchAgent reinstall in this read-only review.

## Solution Assessment

Unclear until maintainers choose the trust boundary. Preserving generated env edits conflicts with the current dangerous-override policy; the cleaner path is an explicit config-owned extra-CA trust setting or a narrower TLS trust mechanism with security review.
