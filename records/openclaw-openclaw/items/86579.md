## Label Justifications

- `P1`: The issue describes an update-time managed gateway regression that breaks real macOS self-signed/private-root TLS deployments and floods approval logs after reinstall.
- `impact:security`: The affected behavior controls TLS trust roots and self-signed gateway certificate handling, which is a security boundary even though the symptom is availability-facing.

## Summary

Keep open: current main and v2026.6.6 still do not provide a durable managed-service extra-CA setting, still filter existing `NODE_EXTRA_CA_CERTS` service-env edits as dangerous overrides, and the only linked config-based fix PR was closed unmerged with the author explicitly saying this issue was not resolved.

## What This Changes

The issue asks for `openclaw gateway install --force` on macOS to stop dropping custom `NODE_EXTRA_CA_CERTS` trust needed by self-signed/private-root TLS gateway deployments.

## Best Possible Solution

Land or adapt the shared TLS fingerprint fix for internal approval clients, then make a separate maintainer/security decision on whether OpenClaw should expose a config-owned extra-CA path for managed gateway services instead of preserving arbitrary generated env edits.

## Reproduction Assessment

Yes, source-level: forced install rebuilds the managed service environment, filters an existing `NODE_EXTRA_CA_CERTS` value as a dangerous override, and falls back to the macOS default when no installer env value is present. I did not run a live macOS LaunchAgent update cycle in this read-only review.

## Solution Assessment

Unclear until maintainers choose the trust boundary. Preserving arbitrary edits in generated env files is not the best fix under current security policy; a safer path is shared local TLS fingerprint propagation plus an explicit config-owned extra-CA surface only if process-wide trust is accepted.
