## Summary

Keep open: current main still has a source-proven path where Telegram asks shared dispatch to suppress default tool progress, but shared dispatch lets session verbose mode override that suppression and send visible tool/status messages.

## What This Changes

The issue asks OpenClaw to prevent Telegram from delivering internal tool/status cards when streaming is off or tool progress is disabled.

## Best Possible Solution

Add a hard channel-owned suppression path for Telegram final-only and `toolProgress: false` turns that shared dispatch honors while still allowing media, errors, and approval prompts that are intentionally user-facing.

## Reproduction Assessment

Yes. The current source and existing direct-Telegram regression test show that `suppressDefaultToolProgressMessages` is bypassed when verbose progress is on, which matches the reported leak path; a live Telegram run still remains for end-to-end proof.

## Solution Assessment

Yes. The narrowest maintainable fix is to distinguish hard channel suppression from ordinary preview suppression in the shared reply options, then set it from Telegram when streaming is off or configured tool progress is false.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/79804. In current main, Telegram passes `suppressDefaultToolProgressMessages` for final-only/streamed turns, but `dispatch-from-config` lets `/verbose` override it and send default tool/status cards. Add a source-level fix, not dist edits, so Telegram hard-suppresses default tool/progress cards when `channels.telegram.streaming.mode` is `off` or when the relevant `streaming.preview.toolProgress` / `streaming.progress.toolProgress` setting is `false`. Preserve intentional user-facing delivery for final replies, media, errors, and approval prompts, and preserve existing verbose behavior for other channels or Telegram turns where tool progress is explicitly enabled. Likely approach: add a typed reply option such as `forceSuppressDefaultToolProgressMessages` or equivalent shared-dispatch policy, have Telegram set it from stream mode/toolProgress config, and apply it consistently to `onToolResult`, plan updates, approval working statuses, command output, and patch summaries. Add focused regressions in Telegram dispatch and shared dispatch tests, update docs only if the public contract text needs clarification, and add a one-line changelog fix entry because this is user-visible chat behavior.
