## Summary

Keep open: current main still has a source-level path where Telegram requests default tool-progress suppression, but shared dispatch disables that suppression when session verbose progress is on. The docs say Telegram `streaming.mode: "off"` is final-only, while current tests intentionally expect the verbose override, so this needs maintainer precedence judgment rather than cleanup closure.

## What This Changes

The issue reports Telegram direct chats receiving internal tool/status progress cards despite Telegram streaming and tool-progress settings being disabled.

## Best Possible Solution

Define the precedence between session `/verbose` and channel-level `streaming.mode: "off"` or `toolProgress: false`, then align shared dispatch, Telegram wiring, docs, and regression tests with that policy while preserving approvals, media, and errors.

## Reproduction Assessment

Yes. Source-level reproduction is high-confidence: set a direct Telegram turn with `suppressDefaultToolProgressMessages: true` and session `verboseLevel: "on"`; current dispatch then sends text-only tool/status payloads despite the suppression flag.

## Solution Assessment

Unclear until maintainers decide precedence. A force-suppression path is plausible, but it would intentionally change the current tested behavior where `/verbose on` overrides default progress suppression.
