## Summary

Keep open: current main and the latest release still allow session verbose mode to bypass Telegram default progress suppression, and current tests explicitly codify that leak while the docs and reply-option contract promise suppression for final-only/off mode. The narrow fix direction was proposed in closed, unmerged PRs, and the still-open adjacent PR proposes the opposite policy, so this issue remains the canonical bug to fix.

## What This Changes

The issue asks OpenClaw to stop sending internal tool/status progress cards into Telegram chats when Telegram streaming/tool-progress settings are disabled.

## Best Possible Solution

Make `suppressDefaultToolProgressMessages` authoritative for default text-only tool/progress messages even when session verbose is on, keep media, exec approvals, and errors deliverable, and attach live Telegram proof before landing.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: current main combines a Telegram direct context, `verboseLevel: "on"`, and `suppressDefaultToolProgressMessages: true`, then current tests expect default tool/status output to be sent. I did not run a live Telegram Bot API scenario in this read-only review.

## Solution Assessment

Yes, the narrowest maintainable fix is to make the shared suppression predicate honor the suppression flag regardless of session verbose, while preserving the existing exceptions for media, exec approvals, and errors. The opposite direction in https://github.com/openclaw/openclaw/pull/77211 would contradict the current public Telegram off-mode contract.
