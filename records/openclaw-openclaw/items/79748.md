## Summary

Keep open. Current source and docs explain one likely cause: Feishu group/channel turns default to message-tool-only visible replies, but the reported post-browser “blocked session” is still plausible and cannot be classified without Feishu logs or a live run.

## What This Changes

Reports Feishu group chats not returning visible replies and appearing blocked after browser operations, while private chats work.

## Best Possible Solution

Verify the flow with Feishu logs and trajectory data; then either guide/configure automatic group replies when the model skips `message(action=send)`, or fix a proven Feishu/browser queue stall with focused regression coverage.

## Reproduction Assessment

No. The issue has concrete user steps and plausible source paths, but there is no high-confidence current-main reproduction without Feishu runtime logs showing whether the agent emitted private final text, skipped the message tool, or stayed stuck in the Feishu/browser dispatch queue.

## Solution Assessment

Unclear. Forcing group final replies to match DMs is not the narrowest maintainable fix because current docs define a safer message-tool-only group default; the next step is to prove whether this is config/model behavior or a Feishu/browser queue defect.
