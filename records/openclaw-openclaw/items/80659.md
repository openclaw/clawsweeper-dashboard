## Summary

Keep open: the reported timeout bug is plausible from source, but this PR should not merge as written because the patch removes existing delivery safeguards, can replay incomplete assistant text, and still carries unrelated high-risk workflow churn.

## What This Changes

The PR changes embedded Pi runner prompt-timeout handling to always append an idle-timeout error payload while preserving existing payloads, and the branch also carries many unrelated release, workflow, docs, app, plugin, and UI changes.

## Best Possible Solution

Rebase and narrow the PR to the runner plus focused regression tests, append the timeout error after successful non-messaging payloads, and preserve the existing messaging-delivery and partial-fragment safeguards.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main can skip timeout-error injection when prompt timeout occurs with existing non-messaging payloads. I did not run a live provider timeout reproduction.

## Solution Assessment

No, this patch is not the best fix as written because it makes prompt timeout handling unconditional, bypassing delivery and partial-fragment protections. A safer fix should target the missing non-messaging payload case only and add regression coverage.
