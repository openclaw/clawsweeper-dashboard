## Summary

Keep open: current main still rejects `nlink > 1` workspace bootstrap reads, and https://github.com/openclaw/openclaw/pull/79215 is an open closing PR for the same behavior. The reporter also added substantive hardlink threat-model context, so this should stay open until maintainers review and land or reject the linked implementation.

## What This Changes

Allow hardlinked workspace bootstrap files such as `AGENTS.md`, `SOUL.md`, and `USER.md` to load instead of being reported as `[MISSING]`.

## Best Possible Solution

Resolve the bootstrap hardlink trust policy, then land or close https://github.com/openclaw/openclaw/pull/79215 with matching regression coverage, changelog, and preservation of stricter credential/write-path guards.

## Reproduction Assessment

Yes, source-reproducible: current main routes bootstrap reads through `openRootFile` with default hardlink rejection, and the existing POSIX test expects hardlinked `AGENTS.md` to be missing. I did not run a live gateway session in this read-only review.

## Solution Assessment

Unclear as a policy matter: `rejectHardlinks: false` on the fixed bootstrap reader is the narrow implementation seam, but it changes a deliberate hardlink guard. Maintainers should decide that trust boundary before the linked PR closes this issue.
