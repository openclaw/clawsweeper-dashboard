## Summary

Keep open: current main still sends workspace bootstrap reads through the default hardlink-rejecting safe-open path, then renders any guarded-read failure as `[MISSING]`. The report is source-reproducible, but relaxing `nlink > 1` behavior touches an intentional filesystem safety guard and needs a scoped maintainer decision rather than cleanup closure.

## What This Changes

The issue requests that readable hardlinked workspace bootstrap files such as `AGENTS.md`, `SOUL.md`, and `USER.md` load normally instead of being reported as missing.

## Best Possible Solution

Make a maintainer-approved, narrowly scoped bootstrap-reader change if the trust model allows it: explicit fixed workspace bootstrap files may allow hardlinked regular files while preserving root boundary, type, size, cache identity, symlink, write-path, and non-bootstrap protections with regression tests.

## Reproduction Assessment

Yes, at source level: a bootstrap file with `nlink > 1` reaches `openRootFile` with default hardlink rejection, the failed open is converted to `missing: true`, and the prompt renderer emits `[MISSING] Expected at: ...`. I did not run a live gateway fixture during this read-only review.

## Solution Assessment

Unclear until maintainers decide the hardlink trust model. A scoped `rejectHardlinks: false` only for explicit workspace bootstrap reads is the narrowest plausible fix; removing the guard globally is not the best solution.
