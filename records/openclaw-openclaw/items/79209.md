## Summary

Keep open: current main still rejects nlink>1 bootstrap reads and has tests asserting that hardlinked bootstrap aliases become missing, but changing that behavior relaxes a deliberate workspace boundary policy and needs maintainer/security judgment.

## What This Changes

The issue asks for workspace bootstrap files such as AGENTS.md and SOUL.md to load when they are POSIX hardlinks instead of being reported as missing.

## Best Possible Solution

Define a narrow workspace-bootstrap sharing policy in the reader layer, with tests that either explicitly allow fixed bootstrap-file hardlinks or keep rejection but surface a diagnostic instead of a misleading missing-file marker.

## Reproduction Assessment

Yes, source-reproducible: current main calls openRootFile with hardlink rejection enabled, and the existing POSIX hardlink test expects the bootstrap file to be missing. I did not run a live gateway session.

## Solution Assessment

No clear best fix yet: passing rejectHardlinks: false in the bootstrap reader is the obvious narrow hook, but it changes the hardlink boundary policy that fs-safe and current tests deliberately enforce. Maintainers should decide the trust model before implementation.
