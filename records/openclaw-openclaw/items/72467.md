## Summary

Keep open: current main still has the Discord audit false-positive bug, but this PR is not merge-ready because the latest patch reads merged account config instead of the effective resolver and still lacks real behavior proof.

## What This Changes

The PR adds top-level Discord `allowFrom` to the security-audit `ownerAllowFromConfigured` calculation, adds a regression test, and updates `CHANGELOG.md`.

## Best Possible Solution

Make Discord security audit compute the owner allowlist through the same effective resolver used by runtime, plus credential-store entries, with regression coverage for top-level `allowFrom` and named-account precedence.

## Reproduction Assessment

Yes, at source level. On current main, native Discord commands with a configured guild, no guild/channel users allowlist, no store or legacy allowlist, and only top-level `channels.discord.allowFrom` leave `ownerAllowFromConfigured` false and still reach `channels.discord.commands.native.no_allowlists`.

## Solution Assessment

No. The direction is right, but the PR should reuse `resolveDiscordAccountAllowFrom({ cfg, accountId })` plus credential-store entries so audit behavior matches runtime account/root precedence.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Use the effective Discord allowFrom resolver:** `extensions/discord/src/security-audit.ts:173-176`
  - body: This counts `discordCfg.allowFrom` from the merged audit account.   - confidence: 0.9
