## Summary

Keep open: this PR targets a real source-reproducible Discord audit false positive, but the latest diff still reads the merged account config directly and can suppress the warning for named accounts whose effective runtime allowlist is empty. It also still lacks contributor real behavior proof.

## What This Changes

The branch adds top-level Discord `allowFrom` to the security-audit `ownerAllowFromConfigured` check, adds a regression test, and updates `CHANGELOG.md`.

## Best Possible Solution

Make the audit compute the owner allowlist through `resolveDiscordAccountAllowFrom({ cfg, accountId })` plus credential-store entries, with regression coverage for top-level `allowFrom` and named-account precedence.

## Reproduction Assessment

Yes, at source level. On current main, native Discord commands with a configured guild, no guild/channel users allowlist, no store or legacy allowlist, and only top-level `channels.discord.allowFrom` leave `ownerAllowFromConfigured` false and still reach `channels.discord.commands.native.no_allowlists`.

## Solution Assessment

No. The direction is right, but the PR should use the existing effective Discord allowFrom resolver so audit behavior matches runtime account/root precedence instead of adding another direct merged-config read.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

- **[P2] Use the effective Discord allowFrom resolver:** `extensions/discord/src/security-audit.ts:173-176`
