## Summary

Keep open. The PR fixes a real Discord security-audit false positive, and the Greptile cast nit appears addressed, but the latest diff still bypasses the runtime allowlist resolver and can leave a named-account precedence false negative in the audit.

## What This Changes

The PR adds top-level Discord `allowFrom` to the security-audit owner allowlist check, adds a regression test, and updates the changelog.

## Best Possible Solution

Compute the audit owner allowlist through the same effective Discord account resolver used by runtime authorization, then add the missing account-precedence regression.

## Reproduction Assessment

Yes. Current main reproduces the original bug because the audit omits top-level `channels.discord.allowFrom`; the PR blocker is reproducible by a named Discord account that inherits root `allowFrom` but sets account legacy `dm.allowFrom` to an empty list, leaving runtime with an empty effective owner allowlist while the PR audit path still counts the inherited root list.

## Solution Assessment

No. Counting `discordCfg.allowFrom` fixes the default top-level case, but the narrow maintainable fix is to call `resolveDiscordAccountAllowFrom({ cfg, accountId })` so audit and runtime share the same account/root precedence contract.

## Review Findings
