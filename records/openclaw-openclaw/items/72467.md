## Summary

Keep this PR open: it targets a real Discord security-audit false positive, but the current patch still reads from the merged account config instead of the effective allowFrom resolver and the external contributor has not provided real after-fix audit output.

## What This Changes

The PR adds top-level Discord `allowFrom` to the Discord security-audit owner allowlist calculation, adds a focused regression test, and updates `CHANGELOG.md`.

## Best Possible Solution

Compute the audit owner allowlist from `resolveDiscordAccountAllowFrom({ cfg, accountId })` plus the credential-store allowlist, add top-level and account-precedence regression coverage, then merge after redacted real audit output is provided.

## Reproduction Assessment

Yes, at source level. Current main leaves `ownerAllowFromConfigured` false when only top-level `channels.discord.allowFrom` is configured, and the PR blocker follows from the existing named-account precedence resolver contract.

## Solution Assessment

No. The direction is right, but the PR should use the existing effective Discord allowFrom resolver rather than adding another direct merged-config read.
