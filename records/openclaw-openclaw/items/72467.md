## Summary

Keep open. The PR targets a real current-main Discord audit false positive and the Greptile cast nit appears addressed, but the direct merged-config read still leaves the security audit out of sync with the effective runtime allowlist resolver for named accounts.

## What This Changes

The PR adds top-level Discord `allowFrom` to the security-audit owner allowlist check, adds a regression test, and updates the changelog.

## Best Possible Solution

Compute the audit’s owner allowlist from `resolveDiscordAccountAllowFrom({ cfg, accountId })` plus the credential-store allowlist, preserving the same account inheritance and legacy precedence as runtime authorization.

## Reproduction Assessment

Yes. Current main reproduces the original bug with native Discord commands, a configured guild, and only top-level `channels.discord.allowFrom`; the PR blocker is reproducible by a named account where root `allowFrom` is inherited into the merged config but account legacy `dm.allowFrom` overrides the effective runtime list.

## Solution Assessment

No. The direction is right, but the proposed direct merged-config read is not the best fix because the existing resolver is already the runtime contract and avoids account-precedence drift.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Use the effective Discord allowFrom resolver:** `extensions/discord/src/security-audit.ts:173-176`
  - body: This reads `discordCfg.allowFrom` from the merged audit account. For a named account with inherited root `allowFrom` and an account-level legacy `dm.allowFrom` override, runtime resolves the account field before the parent, but this audit path still counts the inherited root value and can suppress `no_allowlists` when the effective account allowlist is empty. Build this from `resolveDiscordAccountAllowFrom({ cfg, accountId })` plus the store allowlist and add that precedence regression.
  - confidence: 0.88
