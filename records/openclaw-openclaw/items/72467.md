## Summary

Keep open: this PR targets a real Discord security-audit bug, but the current diff reads the merged account `allowFrom` directly and can suppress `no_allowlists` where the effective runtime resolver would not. It also still needs real after-fix behavior proof before merge.

## What This Changes

The PR adds top-level Discord `allowFrom` to the `no_allowlists` audit calculation, adds a focused regression test, and updates `CHANGELOG.md`.

## Best Possible Solution

Compute the audit owner allowlist from `resolveDiscordAccountAllowFrom({ cfg, accountId })` plus the credential-store allowlist, then cover both top-level `allowFrom` and account-precedence behavior.

## Reproduction Assessment

Yes, at source level. Current main still emits the warning when only top-level `channels.discord.allowFrom` is configured, and the PR blocker follows from a named-account precedence case already covered by existing resolver tests.

## Solution Assessment

No. The direction is right, but the direct merged-config read is not the best fix because the existing resolver is the runtime contract and preserves account/root precedence.
