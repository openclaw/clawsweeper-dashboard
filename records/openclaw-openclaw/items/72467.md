## Summary

Keep open: the underlying Discord audit bug is valid, but this patch should not merge as-is because it reads `discordCfg.allowFrom` from merged account config instead of the effective runtime resolver, which can create a named-account precedence false negative. The external PR also still needs real after-fix behavior proof before merge.

## What This Changes

The PR adds top-level Discord `allowFrom` to the `no_allowlists` security-audit check, adds a focused regression test, and updates `CHANGELOG.md`.

## Best Possible Solution

Compute the audit owner allowlist from `resolveDiscordAccountAllowFrom({ cfg, accountId })` plus the credential-store allowlist, then cover both top-level `allowFrom` and account-precedence override cases.

## Reproduction Assessment

Yes, at source level. Current main still emits the warning when only top-level `channels.discord.allowFrom` is configured, and the PR blocker follows from a named account where root `allowFrom` is inherited into merged config while account legacy `dm.allowFrom` overrides the effective runtime list.

## Solution Assessment

No. The direction is right, but the direct merged-config read is not the best fix because the existing resolver is the runtime contract and preserves account/root precedence.
