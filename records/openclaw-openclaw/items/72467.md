## Summary

Keep open. The reported Discord audit/runtime mismatch is real and the PR is close, but the current patch should not merge as-is because it reads top-level `allowFrom` from the merged audit account config and can suppress `no_allowlists` for a named account whose effective allowlist is empty under the documented account/legacy precedence.

## What This Changes

The PR updates the Discord security audit to count top-level `allowFrom` in `ownerAllowFromConfigured`, adds a regression test, and adds a changelog entry.

## Best Possible Solution

Make the Discord audit use the same effective allowlist resolver as the runtime owner/native-command paths, so simple top-level `allowFrom` suppresses the warning while named-account overrides and legacy precedence remain consistent.

## Reproduction Assessment

Yes. Current main reproduces the original bug by enabling Discord native commands with a configured guild and only top-level `channels.discord.allowFrom`; the audit still reaches `no_allowlists` because that field is ignored. The PR blocker is also reproducible from code with a named account that has inherited root `allowFrom` plus explicit empty legacy `dm.allowFrom`.

## Solution Assessment

No. The intended direction is correct, but this patch is not the best implementation because it reads from the merged account config instead of the effective allowlist resolver; the safer alternative is to build `ownerAllowFromConfigured` from `resolveDiscordAccountAllowFrom({ cfg, accountId })` plus the credential-store allowlist.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Use the effective Discord allowFrom resolver:** `extensions/discord/src/security-audit.ts:173`
  - body: This reads `discordCfg.allowFrom` from the merged audit account. For a named account with root `channels.discord.allowFrom` and an explicit legacy `accounts.<id>.dm.allowFrom: []`, runtime `resolveChannelDmAllowFrom` treats the account legacy list as overriding the inherited root list, but this PR still counts the inherited root value and suppresses `no_allowlists`. Build this check from `resolveDiscordAccountAllowFrom({ cfg, accountId })` plus the store allowlist, and add that account-precedence regression.
  - confidence: 0.87
