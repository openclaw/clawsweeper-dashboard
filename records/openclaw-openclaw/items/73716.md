## Summary

Keep open. Current main still sends the raw Control UI session key, canonicalizes it in the gateway, and then drops the resulting live chat event in the UI before the active run-id guard can match it. The focused implementation PR linked from the issue is closed unmerged, so the bug still needs a narrow fix.

## What This Changes

Ensure Control UI live chat updates render when `chat.send` used a raw session key but gateway live events use the canonical session key for the same in-flight run.

## Best Possible Solution

Add a narrow Control UI event-acceptance fix so live `chat` events are accepted when either the event session key matches the UI session key or the event run id matches the active client run, with regression coverage for delta and final events and no broad gateway canonicalization change.

## Reproduction Assessment

Yes. A source-level reproduction is a `ChatState` with `sessionKey: "main"` and `chatRunId: "run-1"`, then a `chat` payload with `sessionKey: "agent:main:main"` and `runId: "run-1"`; current main returns `null` at the strict session-key check.

## Solution Assessment

Yes. The narrowest maintainable solution is UI-side acceptance or equivalent UI-side canonicalization, because gateway canonicalization is expected and the live payload already carries the client run id needed to prove ownership.
