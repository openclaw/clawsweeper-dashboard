## Summary

Keep open. Current main still has the raw/canonical session-key mismatch in the Control UI event path, and the only directly linked fix PR is closed without merge, so the remaining work is valid, narrow, and actionable.

## What This Changes

The issue reports that Control UI live chat can drop live gateway `chat` events when the UI sends `chat.send` with a raw session key like `main` but the gateway emits events for the canonical key `agent:main:main`.

## Best Possible Solution

Add a narrow Control UI fix that accepts live `chat` events when either the session key matches or the payload `runId` matches the active client run, with regression coverage for both `delta` and `final` events and preserved rejection for unrelated sessions/runs.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is `handleChatEvent` with `state.sessionKey = "main"`, `state.chatRunId = "run-1"`, and a payload using `sessionKey = "agent:main:main"` plus `runId = "run-1"`; current main returns `null` at the strict session-key check.

## Solution Assessment

Yes. The narrowest maintainable fix is in Control UI event acceptance or equivalent UI-side session-key normalization; provider/model handling and gateway canonicalization are not the failing layer.
