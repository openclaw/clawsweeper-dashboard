## Summary

Keep open. Current main still sends the raw Control UI session key, canonicalizes it in the gateway, emits live chat events under the canonical key, and then drops those events in the UI before the active run-id guard can match them; the linked implementation PR is closed unmerged.

## What This Changes

The issue asks for Control UI live chat updates to render when `chat.send` uses a raw session key and gateway `chat` events use the canonical session key for the same in-flight run.

## Best Possible Solution

Add a focused Control UI fix so live `chat` events are accepted when either the event session key matches the UI session key or the payload `runId` matches the active client run, while preserving the existing same-session handling for other-run finals and rejecting events with neither match.

## Reproduction Assessment

Yes. A source-level reproduction is `state.sessionKey = "main"`, `state.chatRunId = "run-1"`, and a `chat` payload with `sessionKey = "agent:main:main"` plus `runId = "run-1"`; current main returns `null` at the strict session-key check before the run-id guard.

## Solution Assessment

Yes. The narrowest maintainable fix is in Control UI event acceptance or equivalent UI-side canonicalization, because the gateway already canonicalizes the raw key and emits the same client run id for the live response.
