## Summary

Keep open. Current main still has the raw/canonical session-key mismatch in the Control UI live chat path: the UI sends the raw active `state.sessionKey`, the gateway canonicalizes and emits `chat` events under the canonical key, and `handleChatEvent` drops the event before considering the matching active run id. The previously linked implementation PR is closed unmerged, so this remains a narrow actionable bug.

## What This Changes

The issue requests that Control UI live chat updates render without refresh when `chat.send` used a raw session key but gateway `chat` events carry the canonical session key for the same in-flight run.

## Best Possible Solution

Add a focused Control UI fix so live `chat` events for the active client run are accepted across raw/canonical session-key aliases, or normalize the UI to the canonical session key before filtering, while keeping events with neither a matching session key nor active run id filtered out.

## Reproduction Assessment

Yes. A source-level reproduction is `state.sessionKey = "main"`, `state.chatRunId = "run-1"`, and a `chat` payload with `sessionKey = "agent:main:main"` and `runId = "run-1"`; current main returns `null` at the strict session-key check before run-id matching.

## Solution Assessment

Yes. The narrowest maintainable solution is in the Control UI event acceptance or UI-side session-key normalization, because the gateway already canonicalizes the raw key and emits the same client run id for the live response.
