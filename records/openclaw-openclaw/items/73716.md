## Summary

Keep open. Current main still has the reported raw-vs-canonical session-key gap: the UI sends the current raw `state.sessionKey`, gateway resolves `chat.send` to the canonical session key used in emitted live events, and `handleChatEvent` returns before the active `runId` check can match the in-flight response. The related implementation PR #73721 is closed and unmerged, so the bug remains actionable.

## What This Changes

Control UI live chat should display assistant responses for an in-flight send when the UI used a raw session key such as `main` but gateway live events use the canonical key such as `agent:main:main`.

## Best Possible Solution

Add a narrow Control UI fix so live `chat` events are accepted when either the event session key matches the UI session key or the event `runId` matches the active client run, with regression coverage for raw/canonical alias deltas and finals. Keep gateway canonicalization unchanged unless UI-side canonicalization is chosen instead, and add a user-facing changelog entry.

## Reproduction Assessment

Yes. A source-level reproduction is a `ChatState` with `sessionKey: "main"` and `chatRunId: "run-1"`, followed by a `chat` payload with `sessionKey: "agent:main:main"`, `runId: "run-1"`, and `state: "delta"` or `"final"`; current main returns `null` at the strict session-key check.

## Solution Assessment

Yes. The narrowest maintainable fix is in Control UI event acceptance or equivalent UI-side canonicalization, because gateway canonicalization is expected and the live payload retains the client run id needed to prove the event belongs to the active send.
