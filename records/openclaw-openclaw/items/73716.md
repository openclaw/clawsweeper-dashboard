## Summary

Keep open. Current main still has the reported raw/canonical session-key mismatch: the Control UI sends the active raw session key, drops live chat events before checking the matching active run id, and the Gateway emits chat events under the canonical session key. The linked implementation PR #73721 is closed unmerged, so the issue remains a narrow actionable bug.

## What This Changes

The issue requests that Control UI live chat updates render without refresh when `chat.send` uses a raw session key such as `main` but Gateway chat events use the canonical key such as `agent:main:main`.

## Best Possible Solution

Add a focused Control UI fix so live `chat` events for the active client `runId` are accepted across raw/canonical session-key aliases, or normalize the UI to the canonical key returned by the Gateway, while keeping unrelated-session and unrelated-run events filtered.

## Reproduction Assessment

Yes. A source-level reproduction is `state.sessionKey = "main"`, `state.chatRunId = "run-1"`, and a `chat` payload with `sessionKey = "agent:main:main"` plus `runId = "run-1"`; current main returns `null` at the session-key check before run-id matching.

## Solution Assessment

Yes. The narrow maintainable fix belongs in Control UI event acceptance or UI-side session-key normalization because the Gateway already canonicalizes raw keys and keeps the client run id on emitted live events.
