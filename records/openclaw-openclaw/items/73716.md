## Summary

Keep open. Current main still has the reported raw-vs-canonical Control UI gap: the browser sends the raw state key, the gateway registers and emits the run under the canonical key, and the UI drops the event before the active run-id guard can match it.

## What This Changes

Control UI should render live assistant updates when `chat.send` uses a raw session key and gateway chat events use the canonical session key for the same run.

## Best Possible Solution

Add a narrow Control UI event-acceptance fix so chat events are delivered when either the session key matches or the event run id matches the active client run, with regression coverage for delta and final payloads.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a `ChatState` with `sessionKey: "main"` and `chatRunId: "run-1"`, followed by a `chat` payload with `sessionKey: "agent:main:main"` and `runId: "run-1"`; current main returns `null` at the strict session-key check.

## Solution Assessment

Yes. The narrowest maintainable solution is UI-side event acceptance or equivalent UI-side canonicalization, because gateway canonicalization is expected and the live payload already carries the client run id needed to prove ownership.

Fix the Control UI live-chat raw/canonical session-key mismatch from https://github.com/openclaw/openclaw/issues/73716. In `ui/src/ui/controllers/chat.ts`, make `handleChatEvent` accept chat events when either `payload.sessionKey === state.sessionKey` or the UI has an active `state.chatRunId` matching `payload.runId`; preserve the existing protection against unrelated sub-agent/fleet announcements and do not change gateway session-key canonicalization. Add regression tests in `ui/src/ui/controllers/chat.test.ts` for a raw `main` UI state receiving canonical `agent:main:main` `delta` and `final` events with the active run id, plus a negative case where neither session key nor run id matches.
