## Summary

Keep open. Current main still has the reported raw-vs-canonical session-key gap: the UI sends the raw state key, the gateway resolves it to a canonical key for live events, and the UI drops the event before the active run-id guard can match it. The linked implementation PR is closed unmerged, so the issue still needs a focused fix.

## What This Changes

The issue requests that Control UI live chat updates keep rendering when `chat.send` uses a raw session key such as `main` but gateway `chat` events arrive under the canonical key such as `agent:main:main`.

## Best Possible Solution

Add a narrow Control UI event-acceptance fix so `chat` events are delivered when either the event session key matches the UI session key or the event `runId` matches the active client run, with regression coverage for both `delta` and `final` and no broad gateway behavior change.

## Reproduction Assessment

Yes. A source-level reproduction is a `ChatState` with `sessionKey: "main"` and `chatRunId: "run-1"`, then a `chat` payload with `sessionKey: "agent:main:main"` and `runId: "run-1"`; current main returns `null` at the strict session-key check.

## Solution Assessment

Yes. The narrowest maintainable repair is in Control UI event acceptance or equivalent UI-side canonicalization, because gateway canonicalization is already expected and the live payload preserves the client run id needed to prove ownership.
