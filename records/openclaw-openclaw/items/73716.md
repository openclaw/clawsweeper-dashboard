## Summary

Keep open. Current main still has the reported raw/canonical session-key mismatch in Control UI live chat handling, and the only linked implementation PR is closed unmerged.

## What This Changes

The issue asks for Control UI live chat updates to keep rendering when `chat.send` uses a raw session key but gateway chat events for the same active run use the canonical session key.

## Best Possible Solution

Add a focused Control UI fix that accepts live `chat` events when either the event session key matches the current UI session key or the payload `runId` matches the active client run, with regression tests covering both the raw/canonical match and unrelated-event rejection.

## Reproduction Assessment

Yes. A source-level reproduction is `state.sessionKey = "main"`, `state.chatRunId = "run-1"`, and a `chat` payload with `sessionKey = "agent:main:main"` plus `runId = "run-1"`; current main returns `null` at the strict session-key check before the run-id guard.

## Solution Assessment

Yes. The narrowest maintainable path is a UI-side event-acceptance change or equivalent UI-side canonicalization, because the gateway already canonicalizes the raw key and preserves the same client run id in live events.

Fix the Control UI live chat raw/canonical session-key mismatch reported in https://github.com/openclaw/openclaw/issues/73716. Current main sends `state.sessionKey` directly in `chat.send`, gateway canonicalizes raw aliases such as `main` to keys such as `agent:main:main`, and `handleChatEvent` drops the resulting live `chat` payload before the active run-id guard can match it. Add/adjust colocated tests in `ui/src/ui/controllers/chat.test.ts` for raw `main` UI state receiving canonical `agent:main:main` payload with matching active `runId`, and for a payload where neither session key nor run id matches still being dropped.
