## Summary

Keep open: this is a focused candidate fix for the linked WebUI inbound-media preview bug, and current main plus v2026.6.9 still do not proxy `media://inbound/<id>` image refs through the Control UI media route. The patch shape looks correct, but contributor proof is still jsdom/mock-backed and needs real browser or live Gateway Control UI proof before merge.

## What This Changes

The PR updates the Control UI grouped renderer to route canonical single-file `media://inbound/<id>` image refs through the authenticated `__openclaw__/assistant-media` route and adds a regression test.

## Best Possible Solution

Land this or an equivalent narrow renderer fix after real browser or live Control UI proof shows a canonical inbound image rendering inline through the existing ticketed assistant-media route.

## Reproduction Assessment

Yes, at source level: current main extracts image `MediaPath` values as raw image URLs, then does not classify `media://inbound/<id>` as proxyable through assistant-media. That leaves the browser with an unloadable custom scheme instead of an inline preview.

## Solution Assessment

Yes for the implementation shape: the gateway already owns validated, authenticated serving of canonical inbound refs, so the maintainable fix is the renderer proxy decision. The remaining blocker is proof from a real browser or live Control UI path, not a different code layer.
