## Summary

Keep this issue open. Current main still forwards the CLI override as `connect.client.instanceId`, but gateway registration, paired-node lookup, reconciliation, and catalog resolution still key the node from `device.id` or `client.id`, so the custom ID remains unresolvable by `node.list`, `node.describe`, and node invocation paths.

## What This Changes

The issue requests that `openclaw node run --node-id <id>` make the provided ID the node identity usable by node listing, describing, and `tools.exec.node` targeting.

## Best Possible Solution

Centralize gateway node-id resolution so a trimmed non-empty `connect.client.instanceId` wins over device/client fallbacks, then use that helper consistently for live registration, paired lookup, reconciliation, catalog-facing behavior, invocation, and pending-work drain.

## Reproduction Assessment

Yes, at source level. `--node-id my-mac-node` reaches the gateway as `connect.client.instanceId`, but current main still registers and reconciles the node under `connect.device?.id ?? connect.client.id`, so `node.describe` by the custom id has no matching catalog entry.

## Solution Assessment

Yes. The narrow maintainable fix is a shared gateway identity helper used at each node boundary; changing only `NodeRegistry.register()` is incomplete because pairing reconciliation and paired-node lookup still use the old identity rule.
