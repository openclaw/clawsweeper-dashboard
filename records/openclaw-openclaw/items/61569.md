## Summary

Keep this issue open. Current main still forwards the CLI override as `connect.client.instanceId`, but gateway node registration, paired-node lookup/reconciliation, pending-work drain, and invoke-result validation still key the node from `device.id` or `client.id`, so the custom id remains unresolvable by `node.list`, `node.describe`, and node invocation paths.

## What This Changes

The issue requests that `openclaw node run --node-id <id>` make that custom id the usable gateway node id, including for `tools.exec.node`.

## Best Possible Solution

Centralize gateway node-id resolution so a trimmed non-empty `connect.client.instanceId` wins over device/client fallbacks, then use that helper consistently at node registration, paired lookup, reconciliation, catalog, pending-work, and invoke-result boundaries.

## Reproduction Assessment

Yes, at source level. `--node-id my-mac-node` reaches the gateway as `connect.client.instanceId`, but current main still registers and reconciles the node under `connect.device?.id ?? connect.client.id`, so `node.describe` by the custom id has no matching catalog entry.

## Solution Assessment

Yes. The narrow maintainable fix is a shared gateway identity helper used at each node boundary; the prior closed PRs that only changed `NodeRegistry.register()` were useful but incomplete because related pairing and result paths would still disagree.
