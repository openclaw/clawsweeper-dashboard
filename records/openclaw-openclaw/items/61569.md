## Summary

Keep open. Current main still accepts and forwards the CLI override as `client.instanceId`, but the gateway registry, pairing reconciliation, paired-node lookup, and catalog-facing node resolution continue deriving the effective node ID from device/client IDs, so the custom ID remains unresolved by list, describe, and exec-node binding paths.

## What This Changes

Honor a user-supplied node ID from `openclaw node run --node-id` as the node identity usable by `nodes list/describe` and exec node binding.

## Best Possible Solution

Use one gateway node-identity resolver that prefers trimmed `connect.client.instanceId` for node sessions, preserves existing device/client fallbacks, and applies consistently across pairing, registry, catalog, describe, pending work, and invocation paths.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: `--node-id` reaches `GatewayClient.instanceId`, but current main still registers and reconciles the node under device/client IDs instead of that instance ID.

## Solution Assessment

Yes. The narrow maintainable fix is a shared gateway identity rule with regression coverage; the prior registry-only PR shape is too small because pairing and lookup paths still disagree.

## Repair Work Prompt

Fix the custom node ID behavior reported in https://github.com/openclaw/openclaw/issues/61569. Current main forwards `openclaw node run --node-id <id>` to `GatewayClient.instanceId`, but gateway node identity still uses `connect.device?.id ?? connect.client.id` in registry, pairing reconciliation, paired-node lookup, pending action pull/ack, and catalog-facing resolution. Implement a shared gateway node identity helper that prefers a non-empty trimmed `connect.client.instanceId` for node identity while preserving existing device/client fallbacks. Apply it consistently to fresh pairing, reconnect/paired-node lookup, live `NodeRegistry.register()`, node list/describe, pending work pull/ack, and node invocation paths as needed. Do not change operator/client device auth semantics, token formats, release/version metadata, or unrelated node command allowlist behavior. Treat closed PRs #61587 and #61719 as useful context but avoid a registry-only fix. Add focused regression tests proving a node connected with `instanceId: "my-mac-node"` is listed/described/invoked by that ID after pairing, and preserve fallback behavior for device ID and client ID. Add a user-facing `CHANGELOG.md` fix entry because this changes CLI/runtime behavior.
