## Label Justifications

- `P2`: This is a real behavior bug in node identity and exec-node selection, but the affected surface is bounded to gateway/node routing and has no crash, data-loss, or security-bypass evidence.

## Summary

Keep this issue open. Current `main` still sends the CLI override as `client.instanceId`, but the gateway registry and catalog paths still key live nodes under `device.id ?? client.id`, so lookup by the custom node id remains source-reproducible.

## What This Changes

The issue requests that `openclaw node run --node-id <id>` be honored as the node id usable by `nodes describe`, node lists, and `tools.exec.node` selection.

## Best Possible Solution

Add one shared gateway node-id resolver where trimmed non-empty `connect.client.instanceId` wins, then use it consistently in registration, pairing reconciliation, catalog/list/describe, pending work, node events, and invoke-result validation.

## Reproduction Assessment

Yes, at source level: current `main` sends the override as `client.instanceId`, while registry/list/describe still key the live node by `device.id ?? client.id`. I did not execute the full live pairing flow in this read-only review.

## Solution Assessment

Yes. A shared resolver is the narrow maintainable fix; changing only `NodeRegistry.register()` would leave pairing, pending work, and invoke-result paths at risk of using a different node key.
