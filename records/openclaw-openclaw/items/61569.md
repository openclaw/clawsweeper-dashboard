## Summary

Keep this issue open. Current main still forwards the CLI override as `connect.client.instanceId`, but the gateway continues to key live registration, pairing reconciliation, pending-node work, and invoke-result validation from `device.id` or `client.id`, so the custom id remains unreliable for node lookup and execution.

## What This Changes

The issue requests that `openclaw node run --node-id <id>` register and resolve a node under the custom id for `nodes describe`, `nodes list`, and `tools.exec.node`.

## Best Possible Solution

Introduce a shared gateway node-id resolver where a trimmed non-empty `connect.client.instanceId` wins over device/client fallbacks, then use it consistently across registration, pairing reconciliation, catalog/list/describe, pending work, and invoke-result validation.

## Reproduction Assessment

Yes, at source level. The CLI sends `--node-id` as `connect.client.instanceId`, while current main registers and reconciles the node under `connect.device?.id ?? connect.client.id`, so a lookup by the custom id cannot resolve through the catalog or invocation paths.

## Solution Assessment

Yes. The best fix is a narrow shared identity helper used at every gateway node boundary; the prior closed PRs that only changed `NodeRegistry.register()` would leave pairing, pending work, and invoke-result checks inconsistent.
