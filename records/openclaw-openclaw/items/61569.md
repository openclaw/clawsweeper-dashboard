## Summary

Keep this issue open. Current main still drops the custom node identity at the gateway: the CLI and node host pass `--node-id` as `connect.client.instanceId`, but node registration and node-pairing reconciliation still choose `connect.device?.id ?? connect.client.id`, and `node.list`/`node.describe` are built from those IDs. The two linked fix PRs correctly identified the same root cause, but both were closed unmerged and no open replacement PR is currently tracking it.

## What This Changes

The issue asks for `openclaw node run --node-id <id>` to make that custom ID visible and usable in `openclaw nodes describe --node <id>` and `tools.exec.node` binding.

## Best Possible Solution

Update the gateway node identity contract so a non-empty trimmed `connect.client.instanceId` is the preferred node ID consistently across node pairing reconciliation, live registry registration, metadata updates, catalog/list/describe behavior, and exec-node binding, while preserving device/client fallback for existing nodes.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: `--node-id` reaches `GatewayClient.instanceId`, but current main still keys pairing and live node registration from the device/client ID, so the custom ID never becomes the catalog ID that `nodes describe` resolves.

## Solution Assessment

No. Current main is not the best solution because it accepts and documents a custom node ID but ignores it at the gateway identity boundary; the safest fix is a narrow gateway/node identity change with regression coverage for fresh pairing, reconnect, `node.list`, and `node.describe`.
