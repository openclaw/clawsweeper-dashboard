## Summary

Keep this issue open. Current main still forwards the CLI override as `connect.client.instanceId`, but the gateway continues to key node pairing, live registration, and catalog lookup from the device/client ID, so the custom ID is not the node ID that `node.list`, `node.describe`, or exec-node binding can resolve.

## What This Changes

Honor `openclaw node run --node-id <id>` as the node ID shown and resolved by node list/describe and `tools.exec.node`.

## Best Possible Solution

Make node identity resolution consistent at the gateway boundary: prefer a non-empty trimmed `connect.client.instanceId` for node IDs, preserve existing device/client fallbacks, and cover fresh pairing, reconnect, catalog, describe, and invocation paths.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: `--node-id` reaches `GatewayClient.instanceId`, but current main still registers and reconciles the node under `connect.device?.id ?? connect.client.id`.

## Solution Assessment

Yes. A focused gateway identity helper used by registration, reconciliation, paired lookup, and catalog-facing tests is the narrowest maintainable fix; changing only `NodeRegistry.register()` is not enough.
