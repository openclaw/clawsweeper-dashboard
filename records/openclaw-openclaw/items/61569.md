## Summary

Keep this issue open. Current main and the latest shipped tag still forward the CLI override as `client.instanceId` but resolve gateway node identity from `device.id` or `client.id`, so the custom node id is still not usable for node lookup or exec-node routing.

## What This Changes

Honor the `openclaw node run --node-id` override as the usable node id for nodes list/describe and `tools.exec.node` routing.

## Best Possible Solution

Add one shared gateway node-id resolver where a trimmed non-empty `connect.client.instanceId` wins, then use it consistently across registration, pairing reconciliation, catalog/list/describe, pending work, and invoke-result validation.

## Reproduction Assessment

Yes, at source level. Current main forwards `--node-id` as `connect.client.instanceId`, while registry, pairing reconciliation, pending drain, and invoke-result validation still resolve node ids from `device.id` or `client.id`.

## Solution Assessment

Yes. The maintainable fix is a narrow shared identity helper used by all gateway node identity boundaries; the closed PRs that only changed `NodeRegistry.register()` would leave other paths inconsistent.
