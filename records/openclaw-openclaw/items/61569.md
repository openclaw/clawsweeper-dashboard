## Summary

Keep this issue open. Current `main` still advertises and forwards `--node-id` as `connect.client.instanceId`, but the gateway still registers, reconciles, catalogs, and drains node work under `device.id` or `client.id`, so the custom ID remains unusable for node lookup and exec routing.

## What This Changes

Honor a custom node ID supplied through `openclaw node run --node-id` so it appears in node listing/describe and can be used by `tools.exec.node`.

## Best Possible Solution

Add a shared gateway node-id resolver where trimmed non-empty `connect.client.instanceId` wins, then use it consistently for registration, pairing reconciliation, catalog/list/describe, pending work, and invoke-result validation.

## Reproduction Assessment

Yes, at source level. Current `main` forwards `--node-id` as `connect.client.instanceId`, while registry, pairing reconciliation, and pending work still resolve node ids from `device.id ?? client.id`, so lookup by the custom id cannot be reliable.

## Solution Assessment

Yes. The maintainable fix is a narrow shared identity helper used across all gateway node boundaries; the prior closed PRs that only changed `NodeRegistry.register()` would leave paired metadata and pending work on a different key.
