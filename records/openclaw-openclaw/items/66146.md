## Summary

Keep open. Current main still omits the model max-token fallback in both OpenAI-compatible builders, but this PR is not merge-ready because it mixes the valid agents transport fix with stale, unrelated TUI changes that regress current optimistic-run and error-status handling.

## What This Changes

The PR makes OpenAI Responses and Completions request builders fall back to `model.maxTokens` when `options.maxTokens` is absent, and also refactors TUI event handling/tests for orphaned streaming status.

## Best Possible Solution

Land a narrow agents-only fix that forwards a positive effective max-token value from explicit stream options first and model metadata second, preserves existing compat field selection and Codex Responses sanitization, and leaves TUI orphan-streaming work to current main or a separate focused PR.

## Reproduction Assessment

Yes. A source-level reproduction is to call `buildOpenAIResponsesParams` or `buildOpenAICompletionsParams` with `model.maxTokens` set and `options.maxTokens` omitted; current main omits the API max-token field in both paths.

## Solution Assessment

No. The transport direction is correct, but this PR is not the best merge vehicle because it bundles unrelated stale TUI changes; the safer fix is a narrow agents-only patch with omitted-option regression tests and a changelog entry.
