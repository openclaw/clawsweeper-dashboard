## Summary

Keep open. Current main still omits the OpenAI-compatible max-token field when `options.maxTokens` is absent, so the core agents bug this PR targets remains real; however, the PR branch is not mergeable as-is because it mixes that useful transport fix with stale TUI event-handler changes that break the handler contract and regress local/error status handling.

## What This Changes

The PR makes OpenAI Responses and Completions request builders fall back from missing `options.maxTokens` to `model.maxTokens`, and also refactors TUI event handling/tests for orphaned streaming status.

## Best Possible Solution

Land a narrow agents-only fix that resolves a positive effective max-token value from explicit stream options first and model metadata second, forwards it through the existing Responses/Completions compat fields, preserves current Codex Responses sanitization, and adds focused omitted-option regression tests; keep the TUI orphan-streaming work separate because current main has already evolved that area.

## Reproduction Assessment

Yes. A focused source-level reproduction is to call `buildOpenAIResponsesParams` or `buildOpenAICompletionsParams` with `model.maxTokens` set and `options.maxTokens` omitted; current main omits the API max-token field, matching the reported OpenRouter/provider-default truncation path.

## Solution Assessment

No. The transport direction is the right fix, but this PR is not the best merge vehicle because it includes unrelated stale TUI changes; a narrow replacement or repaired branch should carry only the agents request-building change and tests.
