Keep open: the PR still contains substantive TUI behavior work that is not implemented on current main, but the branch changes a documented shortcut contract and is currently incorrect against newer TUI abort/shutdown behavior.

## What This Changes

The PR changes TUI Ctrl+C handling to abort active runs before clear/exit behavior, track pending aborts, preserve partial assistant text on aborted runs, and add focused unit tests.

## Best Possible Solution

Decide the intended Ctrl+C policy, then rebase a narrow TUI patch that preserves current pending-run and disconnect-exit behavior, aligns docs/help with the chosen shortcut contract, keeps regression tests, and includes redacted live TUI proof.

## Reproduction Assessment

Yes, source-level: current main's Ctrl+C path does not consult activeChatRunId or call abortActive, and the aborted event handler terminates without finalizing partial streamed text. I did not run a live TUI session in this read-only review.

## Solution Assessment

No as written: the branch is a plausible implementation if maintainers want Ctrl+C to abort active runs, but it must align the shortcut contract, preserve current pending-run and disconnect-exit behavior, and add real TUI proof before merge.
