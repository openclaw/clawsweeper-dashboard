## Summary

Keep open. The PR addresses a real TUI interaction surface, but it is not merge-ready: current main still documents and implements Ctrl+C as clear/exit while Esc and /abort own run aborts, the branch is stale against the newer pending-run abort path from https://github.com/openclaw/openclaw/pull/77199, and the external PR only provides unit-test proof rather than a real TUI run.

## What This Changes

The PR changes TUI Ctrl+C handling so active runs take precedence over draft clearing and exit confirmation, adds pending-abort state and abort failure handling, preserves partial assistant text on aborted events, and adds focused unit tests.

## Best Possible Solution

If maintainers want Ctrl+C to abort active TUI runs, land a rebased version that preserves current pendingChatRunId abort behavior, updates the shortcut docs/help, keeps the focused regression tests, and includes redacted real TUI proof; otherwise keep the documented Esc and /abort flow as the abort mechanism.

## Reproduction Assessment

Yes, source-level only: current main's Ctrl+C path clears/warns/exits and does not call abortActive, while Esc and /abort do. I did not run an interactive TUI session, so the live UX still needs contributor proof.

## Solution Assessment

Unclear. The implementation direction is plausible, but it must be reconciled with current pending-run abort handling and the documented Ctrl+C contract before it is the narrowest maintainable fix.
