## Summary

Keep open, but this PR is not ready as-is. Current `main` documents Ctrl+C as clear/exit and Esc or `/abort` as the abort path, so changing Ctrl+C precedence needs maintainer product/docs approval; the branch also needs to be rebased against newer TUI pending-run and disconnect Ctrl+C handling and still lacks real interactive TUI proof.

## What This Changes

The PR changes TUI Ctrl+C handling to abort an active run before draft clearing or exit confirmation, reset abort-pending state, and preserve partial assistant text on abort.

## Best Possible Solution

Decide whether Ctrl+C should become an abort shortcut; if yes, rebase onto current TUI state handling, update the shortcut docs/help, preserve pending-run and disconnect semantics, and add real terminal proof.

## Reproduction Assessment

No high-confidence bug reproduction under the current documented contract. Source confirms Ctrl+C currently follows the documented clear/warn/exit path, while Esc and `/abort` are the documented abort paths.

## Solution Assessment

No, not as submitted. If maintainers want this shortcut behavior, the implementation needs a rebase-aware update for `pendingChatRunId`/disconnect handling, docs/help alignment, and real interactive proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Keep Ctrl+C aligned with the documented shortcut contract:** `src/tui/tui.ts:302-307`
  - body: This branch changes Ctrl+C to abort before the existing clear-input path whenever a run is active. Current docs say Esc aborts active runs while Ctrl+C clears input and exits on a second press, so this silently changes a documented shortcut and the expected draft-clearing action. Please either keep Ctrl+C as documented, or get the shortcut contract approved and update docs/help/tests with that intentional behavior.
  - confidence: 0.88
- **[P2] Carry the pending-run abort state through Ctrl+C:** `src/tui/tui.ts:904-905`
  - body: Current main tracks `pendingChatRunId` so Esc and `/abort` work between `chat.send` resolving and the first event. This Ctrl+C path only checks `activeChatRunId`, so after a rebase Ctrl+C would still fall back to clear/exit in that in-flight window and miss the run users are waiting on. Use the same `activeChatRunId ?? pendingChatRunId` target as the current abort path.
  - confidence: 0.82
