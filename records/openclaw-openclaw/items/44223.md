## Summary

Keep open: current main still follows the documented Esc/`/abort` cancellation contract and does not implement Ctrl+C abort, while this PR needs a maintainer shortcut-policy choice, a rebase over newer TUI abort work, docs alignment, and real behavior proof before merge.

## What This Changes

The PR changes TUI Ctrl+C handling so active runs abort before draft clearing or exit confirmation, adds pending/stale abort handling, preserves partial aborted assistant text, and adds TUI unit coverage.

## Best Possible Solution

Choose one TUI shortcut contract; if Ctrl+C should abort active runs, rebase the branch over current main, preserve the pending-run abort fix, update the public TUI docs, and add redacted interactive TUI proof before merge.

## Reproduction Assessment

Yes, source-level: current main routes Ctrl+C through clear/warn/exit handling without reading active run state, while Esc and `/abort` use `abortActive`. I did not run a live interactive TUI session because this review was read-only.

## Solution Assessment

Unclear: the implementation is narrow if maintainers want abort-first Ctrl+C, but it is not the best solution until the project chooses between preserving Ctrl+C as clear/exit with better active-run messaging and changing Ctrl+C into an abort shortcut.
