## Summary

Keep open: the PR still contains potentially useful TUI abort-state work, but it changes the documented Ctrl+C keybinding policy, needs reconciliation with newer pending-run abort handling on main, and lacks after-fix real TUI proof.

## What This Changes

This PR changes TUI Ctrl+C handling to abort active runs before clearing draft input or exiting, resets abort/exit state around stale aborts, preserves partial aborted assistant text, and adds focused unit tests.

## Best Possible Solution

Choose the intended Ctrl+C policy, then rebase the TUI patch onto current main, preserve pendingChatRunId abort behavior, update the user-facing shortcut contract, and add real interactive proof.

## Reproduction Assessment

Yes, source-level only: current main's Ctrl+C path never reads activeChatRunId, while Esc and /abort route through abortActive. That proves the proposed behavior is not current behavior, but it does not prove that changing Ctrl+C policy is the right product choice.

## Solution Assessment

No as submitted. The PR needs to preserve current pendingChatRunId abort fallback and align docs/help with the keybinding policy before it is a maintainable fix direction.

## Review Findings

  - body: This branch makes Ctrl+C return `abort` whenever a run is active, but the TUI docs still tell users that Esc aborts and Ctrl+C clears input or exits. Either keep Ctrl+C on the documented clear/exit path or update the user-facing shortcut/help contract with the new abort behavior.
  - confidence: 0.86

  - confidence: 0.78
