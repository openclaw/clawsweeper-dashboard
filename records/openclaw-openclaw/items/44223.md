## Summary

Keep open for maintainer review. Current main has not implemented the abort-first Ctrl+C behavior, and the remaining question is a TUI product contract choice because related open items propose preserving the existing clear/exit binding and adding an active-run hint instead. The patch is useful but not merge-ready until docs and changelog policy are handled if maintainers accept the keybinding change.

## What This Changes

The PR changes TUI Ctrl+C handling to abort active runs before clear/exit, track pending or stale abort requests, preserve partial assistant text on aborted runs, and add focused regression tests.

## Best Possible Solution

Maintain one explicit TUI shortcut contract. Either land this PR or an equivalent abort-first patch with docs, changelog, and focused TUI tests, or keep Ctrl+C as clear/exit and resolve the user confusion through the active-run hint path tracked by #38501/#38502.

## Reproduction Assessment

Yes. The current-main source gives a deterministic reproduction path: with an active run and non-empty draft, Ctrl+C still resolves from editor input only and follows the clear-input branch instead of aborting the active run.

## Solution Assessment

Unclear. The implementation is a plausible narrow fix if maintainers want Ctrl+C to abort active runs, but it is not clearly the best product solution until maintainers choose between that behavior and the hint-only contract in #38501/#38502.
