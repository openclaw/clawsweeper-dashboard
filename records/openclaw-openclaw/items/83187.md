## Summary

Keep this PR open. It addresses a real heartbeat pending-final cleanup bug, but the current patch can clear older unsatisfied pending-final recovery state when a heartbeat response-tool/message_tool path sends a visible notice without refreshing pendingFinalDelivery for that run.

## What This Changes

The PR adds a shared pendingFinalDelivery cleanup field set to the heartbeat runner, clears it after heartbeat send-success and duplicate-suppression paths, and adds regression tests for those paths.

## Best Possible Solution

Keep the heartbeat clear-on-success fix, but gate both clear sites so pending-final state is cleared only when the current heartbeat delivery actually refreshed or satisfied that pending state.

## Reproduction Assessment

Yes. Current main and v2026.6.9 can record visible heartbeat send success without clearing pendingFinalDelivery, and the PR blocker is source-reproducible because message_tool_only suppresses pending capture while the PR clears existing pending state anyway.

## Solution Assessment

No, not yet. Clearing in heartbeat-runner is the right owner-boundary fix, but the latest patch must preserve older pending-final state when the current heartbeat delivery did not create or satisfy it.
