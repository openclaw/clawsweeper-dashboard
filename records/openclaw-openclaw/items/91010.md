## Summary

Keep open: this PR is still the focused implementation candidate because current main still builds status-only index identity from `memorySearch.model` or adapter fallback, while local indexing records `local.modelPath` as the effective model. The patch aligns that comparison for direct and alias-to-local providers, adds targeted regression coverage, and I found no blocking review findings.

## What This Changes

The PR changes memory-core status-only index identity resolution to use `local.modelPath` for providers resolving to the local adapter and expands regression coverage for direct and alias local providers.

## Best Possible Solution

Merge this focused memory-core identity fix after maintainer approval; keep status-only identity aligned with the provider model contract without changing local embedding runtime or config shape.

## Reproduction Assessment

Yes, by source inspection: local indexing persists `provider.model` from `local.modelPath`, while current-main status-only identity can expect `memorySearch.model` or the adapter fallback before provider initialization. The linked issue also supplies user-visible `Dirty: yes` output after rebuild.

## Solution Assessment

Yes. Updating the status-only expected identity to mirror the local provider's effective model is narrower than initializing the local embedding provider during plain status, changing metadata format, or adding a new compatibility/config path.
