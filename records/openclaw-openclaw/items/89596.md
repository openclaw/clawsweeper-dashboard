## Summary

Keep open: the PR is a focused and useful fix for the declared-tool false-warning path, but the current patch still over-filters declared MCP servers when a denylist names one concrete server tool. That would leave a valid `server__*` allowlist warning as unknown even though only one tool under that server is denied, so the PR needs a small repair before merge.

## What This Changes

The PR adds declared plugin and MCP allowlist context to the tool-policy analyzer and wires that context through agent, embedded, gateway, and skill tool resolution.

## Best Possible Solution

Keep the metadata-only declared context approach, but only remove an MCP server declaration for `bundle-mcp` or server-wide namespace denies, then add a regression for exact concrete MCP tool denies.

## Reproduction Assessment

Yes. Current main is source-reproducible because allowlist analysis only sees materialized tools, and the PR defect is source-reproducible with `mcp.servers.paperless`, `tools.allow: ["paperless__*"]`, and `tools.deny: ["paperless__delete"]`.

## Solution Assessment

No as written. The declared-context layer is the right narrow fix, but MCP deny matching needs to preserve the difference between exact tool denies and server-wide namespace denies.
