## What This Changes

The PR adds declared plugin and MCP allowlist context to tool-policy diagnostics across agent, embedded, gateway, and skill tool resolution without changing public config or plugin API surfaces.

## Best Possible Solution

Keep the declared-context approach, fix MCP deny matching so only namespace-wide denials remove a server declaration, add a regression for exact per-tool denies, then provide redacted real runtime diagnostic proof before maintainer merge review.

## Reproduction Assessment

Yes at source level: current main classifies allowlist entries from materialized tool groups only, and the PR's new MCP deny check can be traced to a false warning for exact per-tool denies. I did not run tests or a live gateway in this read-only review.

## Solution Assessment

No as submitted: the internal declared-context design is the right layer, but the current deny check treats single-tool MCP denies as whole-server denies. A small matcher fix plus regression coverage is a safer solution.

- **[P2] Do not drop an MCP server for single-tool denies:** `src/agents/tool-policy-declared-context.ts:36-39`
