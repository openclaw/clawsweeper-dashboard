## Summary

Keep open: the latest branch fixes the prior `group:plugins` gap, but it still treats any exact server-prefixed MCP deny as removing the whole declared server, which can create false unknown-allowlist warnings for valid configs. The proof currently shown is focused test/lint output rather than a real OpenClaw doctor, gateway, or agent run.

## What This Changes

The PR adds declared plugin/MCP allowlist context to the tool-policy pipeline so configured but not-yet-materialized plugin and MCP allowlist entries do not warn as unknown.

## Best Possible Solution

Keep the declared-context approach, but distinguish exact MCP tool denies from server-wide MCP namespace denies and add both a focused regression and real CLI/runtime proof before maintainer merge review.

## Reproduction Assessment

Yes, source-level reproduction is clear: configure `mcp.servers.paperless`, allow `paperless__*`, and deny an exact tool such as `paperless__delete`. The PR head drops the entire `paperless` declaration because the deny entry starts with the server prefix, even though runtime deny matching only blocks the exact denied tool.

## Solution Assessment

No, not yet. The declared-context layer is the right boundary for this diagnostic, but it needs to preserve server declarations for exact MCP tool denies instead of treating every server-prefixed deny as server-wide.
