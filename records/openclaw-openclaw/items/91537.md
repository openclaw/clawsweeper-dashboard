## Summary

Keep open: current main and the latest release retain a credible Codex lifecycle gap—normal conversation and run paths release shared-client leases without necessarily closing the stdio child, while binding files intentionally persist for thread resume. The closed proposed fix targeted approval resolution instead of the lifecycle owner and was withdrawn after review showed it would break successful bound conversations.

## What This Changes

Fix Codex yolo-mode lifecycle cleanup so ended sessions do not leave app-server processes or stale binding sidecars behind.

## Best Possible Solution

Define one plugin-owned lifecycle for zero-lease stdio clients: retain bindings only while threads are intentionally resumable, schedule bounded idle retirement through the existing close-and-wait path, and add a safe migration or doctor cleanup for provably stale historical sidecars.

## Reproduction Assessment

Yes at source level: Codex stdio shutdown depends on connection closure, while ordinary OpenClaw paths can release leases without closing the shared client. A live current-main reproduction showing process IDs and sidecar counts before and after session/app shutdown is still needed for high-confidence runtime proof.

## Solution Assessment

No, clearing bindings from the approval-resolution hook is not the best solution because successful bound turns require that sidecar and the hook does not own the child process. Cleanup should be implemented at the shared-client/session disposal boundary with explicit idle and resume semantics.
