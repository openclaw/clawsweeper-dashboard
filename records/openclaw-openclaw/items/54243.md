## Summary

Keep open. Current main still makes `--no-include-workspace` affect only backup planning; archive creation still tars the broad state directory and only filters installed plugin dependency folders, so default workspaces under `~/.openclaw` can still be included.

## What This Changes

The issue requests that `openclaw backup create --no-include-workspace` materially exclude agent workspace contents even when default workspaces are nested under `~/.openclaw`, or else provide clearer semantics or a first-class exclude path.

## Best Possible Solution

Make `--no-include-workspace` archive-real by composing canonical workspace exclusions into state-asset archive writes when workspace inclusion is disabled, while preserving required OpenClaw state and keeping generic exclude-pattern work separate unless maintainers deliberately merge the two efforts.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on current main: the default workspace is under `~/.openclaw`, the flag removes workspace candidates from the plan, the state asset remains included, and the only tar filter skips plugin `node_modules`, not nested workspaces.

## Solution Assessment

Yes. The narrow maintainable fix is to add a workspace-aware archive filter for state entries when `includeWorkspace` is false; a docs-only clarification would preserve surprising archive scope despite the CLI and docs promising workspace exclusion.
