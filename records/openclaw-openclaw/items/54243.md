## Summary

Keep open. Current main still drops workspace candidates from the plan when workspace inclusion is disabled, but the broad state asset remains archived and the tar filter only removes installed plugin dependency trees, so default workspaces under `~/.openclaw` can still enter the archive.

## What This Changes

Make `openclaw backup create --no-include-workspace` materially exclude agent workspace contents even when those workspaces live under `~/.openclaw`.

## Best Possible Solution

Make the existing flag archive-real by applying canonical workspace exclusions during state-asset tar creation when `includeWorkspace` is false, while preserving state backup and keeping generic exclude-pattern UX separate unless maintainers choose to land it together.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence reproduction path: default workspace paths are inside `~/.openclaw`, `--no-include-workspace` removes only separate workspace candidates, and the archive still writes the state tree with no workspace filter.

## Solution Assessment

Yes. The narrow maintainable fix is to add a workspace-aware archive filter for nested state contents when workspace inclusion is disabled; a docs-only clarification would leave behavior contrary to current CLI help and docs.
