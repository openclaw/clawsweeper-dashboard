## Label Justifications

- `P2`: This is a normal-priority correctness bug in an existing backup flag with a clear source path and limited CLI/archive blast radius.
- `impact:security`: The flag can unexpectedly include workspace/project contents in backup archives despite asking to exclude them, which can expose sensitive workspace data.

## Summary

Keep open: current main and v2026.5.20 still include the broad state asset and do not filter nested workspace paths during archive traversal, so the existing CLI/docs contract remains broken.

## What This Changes

The issue asks for `--no-include-workspace` to materially exclude workspace contents when default workspaces live under `~/.openclaw`, or for the command contract to stop promising that exclusion.

## Best Possible Solution

Make the existing flag archive-real by carrying workspace exclusion roots into archive creation and filtering those roots from state-asset traversal when `includeWorkspace` is false, while leaving generic exclude-pattern UX to the broader backup-exclude work.

## Reproduction Assessment

Yes, by source inspection rather than a live archive run. Current main still includes the state root when `includeWorkspace` is false, default workspaces live under that root, and the tar filter has no workspace exclusion branch.

## Solution Assessment

Yes. The narrow maintainable fix is a workspace-aware archive filter keyed off the existing `includeWorkspace` flag; a docs-only clarification would leave behavior contrary to current CLI help and backup docs.
