## Label Justifications

- `P2`: This is a normal-priority correctness bug in an existing backup flag with a clear source path and limited CLI/archive blast radius.
- `impact:security`: The flag can unexpectedly include workspace/project contents in backup archives despite asking to exclude them, which can expose sensitive workspace data.

## Summary

Keep open. Current main and the latest release still plan the broad state directory and the archive filter has no workspace-root exclusion, so `--no-include-workspace` remains misleading for default `~/.openclaw/workspace` layouts.

## What This Changes

Make `openclaw backup create --no-include-workspace` materially exclude workspace contents even when default workspaces are nested under the state directory.

## Best Possible Solution

Make the existing flag archive-real by excluding canonical workspace roots from the state-asset tar traversal when `includeWorkspace` is false, while leaving generic exclude-pattern UX to the separate product/security path unless maintainers intentionally fold it in.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: default workspaces live under `~/.openclaw`, the planner still includes the broad state asset, and the tar filter has no workspace-root exclusion when `includeWorkspace` is false. I did not create a live archive because this review was required to keep the checkout read-only.

## Solution Assessment

Yes. The narrow maintainable fix is to apply a workspace-aware archive filter keyed off the existing `includeWorkspace` flag; a docs-only clarification would leave behavior contrary to current CLI help and backup docs.
