- `P2`: This is a normal-priority correctness bug in an existing backup flag with a clear source path and limited CLI/archive blast radius.
- `impact:security`: The flag can unexpectedly include workspace/project contents in backup archives despite asking to exclude them, which can expose sensitive workspace data.

## Summary

Keep open. Current main and the latest release still include the broad state asset and only drop separate workspace candidates from the plan, so nested `~/.openclaw/workspace*` contents are still archived despite the CLI/docs promise.

## What This Changes

The issue requests that `backup create --no-include-workspace` materially exclude workspace contents even when default workspaces live under `~/.openclaw`.

## Best Possible Solution

Make the existing flag archive-real by carrying workspace exclusion roots into archive creation and filtering those roots from the state-asset traversal when `includeWorkspace` is false, leaving generic exclude-pattern UX to the separate feature path.

## Reproduction Assessment

Yes, by source inspection. I did not create a live archive because this review is read-only, but current main still has the exact planner/filter path needed to reproduce the reported nested-workspace inclusion.

## Solution Assessment

Yes. The narrow maintainable fix is to apply a workspace-aware archive filter keyed off the existing `includeWorkspace` flag; a docs-only clarification would leave behavior contrary to current CLI help and backup docs.
