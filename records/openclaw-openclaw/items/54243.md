## Summary

Keep open. Current main still removes workspace paths only as separate backup candidates while archiving the broad state tree, and the archive filter has no workspace exclusion branch, so default workspaces nested under `~/.openclaw` can still be included.

## What This Changes

The issue asks for `openclaw backup create --no-include-workspace` to materially exclude workspace contents even when those workspaces live under `~/.openclaw`.

## Best Possible Solution

Make the existing flag archive-real by excluding canonical workspace roots during state-asset tar creation when workspace inclusion is disabled, while keeping generic exclude-pattern UX tracked separately unless maintainers land it together.

## Reproduction Assessment

Yes, via source inspection rather than a live command: the default workspace is under the default state directory, the flag drops only workspace candidates, and the tar filter still archives state entries without a workspace check.

## Solution Assessment

Yes. The narrow maintainable fix is a workspace-aware state archive filter keyed off the existing `includeWorkspace` flag; a docs-only clarification would leave the CLI behavior contrary to current help and backup guidance.
