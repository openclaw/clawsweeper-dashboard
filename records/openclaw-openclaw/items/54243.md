## Summary

Keep open. Current main still plans `--no-include-workspace` by removing workspace assets from the plan, but it still archives the broad state directory, and the archive filter only excludes installed plugin `node_modules` under state extensions. That means a default workspace under the state tree can still be included in the real archive despite the CLI/docs promise.

## What This Changes

The issue requests that `backup create --no-include-workspace` materially exclude agent workspace contents from backup archives, or otherwise clarify/replace the flag with real nested excludes.

## Best Possible Solution

Make the flag archive-real by composing canonical workspace exclusions into state-asset tar creation when `includeWorkspace` is false, while preserving required OpenClaw state. Add an archive regression test proving a file under the default nested workspace is absent, and keep generic user-supplied exclude-pattern work tracked separately unless it intentionally subsumes this default behavior.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on current main: default workspaces live under `~/.openclaw`, the flag removes only workspace candidates, the state asset remains included, and the state tar filter has no workspace exclusion branch.

## Solution Assessment

Yes. The narrow maintainable repair is to exclude discovered workspace directories during state-asset archive writes when workspace inclusion is disabled; a docs-only clarification would leave surprising archive scope, and the broad exclude-pattern PR is not merged.
