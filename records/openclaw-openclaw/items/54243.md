## Summary

Keep open. Current main still removes workspace candidates from the backup plan when workspace inclusion is disabled, but it still archives the broad state directory and the only state-tree tar filter skips installed plugin dependency folders, not nested workspace directories.

## What This Changes

The issue asks for `backup create --no-include-workspace` to materially exclude default workspace contents nested under `~/.openclaw`, or for the CLI/docs contract to stop implying that it does.

## Best Possible Solution

Make `--no-include-workspace` archive-real by carrying canonical workspace exclusions into archive creation when the state asset contains those directories, while preserving the existing state backup for required OpenClaw state. Add a real archive regression test proving a file under `~/.openclaw/workspace` is absent when workspace inclusion is disabled, and keep generic exclude-pattern work tracked separately unless it explicitly satisfies this default nested-workspace contract.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on current main: default workspaces live under `~/.openclaw`, the flag removes workspace candidates from planning, the state asset remains included, and the tar filter only removes plugin `node_modules`, so workspace files under state would still be traversed into the archive.

## Solution Assessment

Yes. The narrow maintainable fix is to compose a workspace-exclusion filter for state-asset archive writes when `includeWorkspace` is false; a docs-only clarification would preserve surprising backup scope despite current docs/help promising workspace exclusion.
