## Label Justifications

- `P2`: This is a normal-priority correctness bug in an existing backup flag with a clear source path and limited CLI/archive blast radius.
- `impact:security`: The flag can unexpectedly include workspace/project contents in backup archives despite asking to exclude them, which can expose sensitive workspace data.

## Summary

Keep open. Current main and v2026.5.18 still drop workspace candidates from the plan while archiving the broad state root, and the archive filter does not remove nested workspace paths, so the existing flag contract remains broken.

## What This Changes

The issue asks for `backup create --no-include-workspace` to materially exclude nested agent workspaces from backup archives, or to expose a clearer exclusion path.

## Best Possible Solution

Make the existing flag archive-real by carrying workspace exclusion roots into archive creation and filtering those roots from state-asset traversal when `includeWorkspace` is false, while leaving generic exclude-pattern UX to the broader backup-exclude work.

## Reproduction Assessment

Yes, by source inspection. I did not create a live archive in this read-only review, but current main still has the exact planner/filter path needed to reproduce nested-workspace inclusion.

## Solution Assessment

Yes. The narrow maintainable fix is to apply a workspace-aware archive filter keyed off the existing `includeWorkspace` flag; a docs-only clarification would leave behavior contrary to current CLI help and backup docs.
