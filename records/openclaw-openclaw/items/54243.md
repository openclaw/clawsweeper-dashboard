## Label Justifications

- `P2`: This is a normal-priority correctness bug in an existing backup flag with bounded CLI/archive blast radius.
- `impact:security`: The flag can unexpectedly include workspace/project contents in backup archives despite asking to exclude them, which can expose sensitive workspace data.

## Summary

Keep open: current main still plans the broad state directory and the archive filter does not remove nested workspace roots, so `--no-include-workspace` can still archive default workspaces under `~/.openclaw`. The broader exclude-pattern tracker and implementation candidate are useful related work, but they remain open and do not supersede this narrower flag-contract bug.

## What This Changes

Make `backup create --no-include-workspace` actually exclude nested agent workspaces from backup archives instead of only omitting them as separate top-level assets.

## Best Possible Solution

Make the existing flag archive-real by filtering known workspace roots out of state-asset traversal when `includeWorkspace` is false, while leaving generic exclude-pattern UX to the broader backup-exclude track.

## Reproduction Assessment

Yes, by source inspection. I did not create a live archive in this read-only review, but current main still includes the state source root and has no workspace-root filter in the archive traversal path.

## Solution Assessment

Yes. The narrow maintainable fix is a workspace-aware archive filter keyed off the existing `includeWorkspace` flag; a docs-only clarification would leave behavior contrary to current CLI help and backup docs.
