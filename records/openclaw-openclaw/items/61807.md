## Summary

Keep open. The PR targets a real Feishu media gap that is not implemented on current main, but it still lacks real behavior proof and has two merge-blocking implementation issues: extension-boundary imports and unbounded ffmpeg output buffering.

## What This Changes

The PR adds Feishu video cover extraction with ffmpeg and includes the uploaded `image_key` in native media message payloads.

## Best Possible Solution

Implement thumbnail extraction through the plugin SDK media runtime or a focused SDK helper, bound ffmpeg output, add success/fallback coverage, then merge only after redacted live Feishu proof shows a native video cover.

## Reproduction Assessment

Yes, at source level: current main routes MP4/video to `msg_type: "media"` but serializes only `{ file_key }` in the message content. I did not run a live Feishu tenant send, so this is source-reproducible rather than fully reproduced.

## Solution Assessment

No as-is. Owner-local thumbnail generation is the right direction, but this implementation should stay inside the plugin SDK boundary and use bounded ffmpeg output before live proof is accepted.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88
