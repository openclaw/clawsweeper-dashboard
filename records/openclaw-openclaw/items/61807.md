## Summary

Keep open: current main still lacks Feishu video `image_key` support and the PR targets the right bug, but it cannot merge as-is because after-fix real Feishu proof is missing and the patch introduces a bare, unbounded `ffmpeg` process instead of using the shared trusted media runtime.

## What This Changes

The branch changes `extensions/feishu/src/media.ts` to extract a first-frame JPEG with ffmpeg, upload it to Feishu, and include the returned `image_key` when sending `msg_type: "media"`.

## Best Possible Solution

Rework thumbnail extraction through a trusted, timeout-bounded ffmpeg path, preserve the best-effort fallback to sending the video, add focused Feishu media tests, and require redacted real Feishu proof before merge.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: send an MP4 through `sendMediaFeishu`, observe routing to `msg_type: "media"`, and then `sendFileFeishu` builds content with only `file_key`. I did not run a live Feishu account flow in this read-only review.

## Solution Assessment

No as-is. The owner-local thumbnail direction is narrow, but the implementation should use or extend the shared trusted, timeout-bounded ffmpeg runtime and preserve current Feishu media behavior after rebase.

## Review Findings

Overall correctness: patch is incorrect
