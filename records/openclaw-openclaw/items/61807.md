## Summary

Keep open for maintainer review: current main already fixed the linked MP4-as-file regression, but it still does not generate or send a Feishu video cover `image_key`, so the PR is not obsolete. The branch cannot merge as-is because it crosses the plugin boundary, buffers ffmpeg output without a cap, lacks after-fix Feishu proof, and is currently conflicting.

## What This Changes

The PR adds Feishu video thumbnail extraction with ffmpeg, uploads the extracted JPEG as an image, and sends its `image_key` with native media messages plus a fallback test.

## Best Possible Solution

Implement Feishu video cover generation inside the plugin SDK boundary with bounded ffmpeg output, keep the no-ffmpeg fallback, add success/fallback coverage, then merge only after redacted live Feishu proof shows a native video cover.

## Reproduction Assessment

Yes at source level: current main routes MP4/video sends to `msg_type: "media"` but serializes only `{ file_key }`, while Feishu's message-content docs describe `image_key` as the optional video cover key. I did not run a live Feishu tenant send.

## Solution Assessment

No as-is. Owner-local thumbnail generation is a reasonable direction, but this implementation should use the plugin SDK boundary, bound ffmpeg output, and provide real Feishu proof before merge.
