## Summary

Keep this PR open. Current main still sends Feishu `media` messages with only `file_key`, so the requested native video-cover behavior is not implemented; however, this branch is dirty against current main and the new thumbnail extraction path should not merge as-is because it adds unbounded PATH-resolved `ffmpeg` execution instead of using the repository's trusted media runtime safeguards.

## What This Changes

The PR modifies `extensions/feishu/src/media.ts` to extract a JPEG cover from outbound videos with `ffmpeg`, upload it as a Feishu image, and include the resulting `image_key` in `media` message content.

## Best Possible Solution

Land a Feishu-owned current-main fix that preserves the PR's useful behavior, extracts and uploads a video cover best-effort, includes `image_key` only when extraction/upload succeeds, reuses or extends the shared trusted ffmpeg helper with timeout handling, and adds focused Feishu media tests plus the required changelog entry.

## Reproduction Assessment

Yes. A static current-main reproduction is enough: send an mp4 through `sendMediaFeishu`, observe routing to `msgType: "media"`, then `sendFileFeishu` builds content with only `file_key`; the Feishu docs show `image_key` is the cover field.

## Solution Assessment

No as-is. The owner-local direction is right, but the implementation should be rebased onto current main, use trusted timeout-bounded ffmpeg execution, and cover the new payload/fallback behavior in tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Use bounded trusted ffmpeg for thumbnails:** `extensions/feishu/src/media.ts:28-40`
  - body: `extractVideoCover` starts `ffmpeg` directly from PATH and waits for `close` with no timeout or kill path. If ffmpeg hangs on a malformed or truncated video, `sendMediaFeishu` waits here before uploading the original file, so the intended soft fallback can stall the whole Feishu send; it also bypasses the trusted system-bin resolution used by the shared media runtime. Reuse or extend the shared ffmpeg helper, or add equivalent timeout and trusted executable handling.
  - confidence: 0.87

  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The diff introduces a new child-process execution path for `ffmpeg` that resolves through PATH and has no timeout.

Concerns:

- **[medium] Untrusted unbounded ffmpeg execution:** `extensions/feishu/src/media.ts:28`
  - body: The added `spawn("ffmpeg", ...)` path can execute whichever binary appears first on PATH and can wait indefinitely if ffmpeg wedges, while current media runtime code resolves trusted system binaries and applies timeout/maxBuffer limits. This should be routed through the shared media runtime or given equivalent controls before merge.
  - confidence: 0.87
