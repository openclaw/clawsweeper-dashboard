## Summary

Keep open for maintainer review. Current main does not implement the PR’s explicit retryability metadata, and the patch is too narrow for the linked malformed-input surface because it only tags one required-parameter helper while shared `ToolInputError` and `web_fetch` invalid-URL paths remain plain errors; the external PR also lacks after-fix real behavior proof.

## What This Changes

The PR marks Pi required-parameter validation errors as non-retryable and propagates a `retryable` flag through the Pi tool-definition adapter’s serialized error result.

## Best Possible Solution

Use a typed malformed-tool-input/error-result contract shared by required read/write/edit params, `ToolInputError`, and `web_fetch` validation, with focused adapter and parameter tests and coordination with the open chat-surface suppression work.

## Reproduction Assessment

Yes by source inspection. On current main, required-param validation, shared `ToolInputError`, and invalid `web_fetch` URLs serialize without `retryable: false`, and the PR only covers one of those malformed-input paths.

## Solution Assessment

No. The PR points in a plausible direction, but tagging only `parameterValidationError()` with an ad-hoc property is not the narrowest maintainable solution for all malformed tool-input errors.
