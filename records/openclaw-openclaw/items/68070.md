## Summary

Keep this PR open, but do not merge it as-is. Current main still lacks non-retryable metadata for tool parameter validation failures, and the proposed patch is incomplete because the linked bug also names `web_fetch` invalid URL validation, which still throws a plain `Error` outside the helper changed by this PR.

## What This Changes

The PR marks `pi-tools.params` validation errors as `retryable: false` and propagates that optional flag through the Pi tool-definition adapter error payload.

## Best Possible Solution

Land a tested validation-error contract that marks OpenClaw-owned malformed tool inputs non-retryable across the shared helper paths, including read/write/edit required params and `web_fetch` URL validation, while preserving ordinary tool-error behavior and coordinating with the parameter-example work in #56697.

## Reproduction Assessment

Yes. Current-main source gives a high-confidence reproduction path: call read/write/edit with missing required params or `web_fetch` with an invalid URL, then observe that the adapter serializes an error result without `retryable: false`.

## Solution Assessment

No. The direction is useful, but this implementation is too narrow; a shared typed input-error or validation helper is the safer maintainable solution, with regression coverage for both required-param failures and `web_fetch` invalid URLs.
