## Summary

Current main still strips final tags as plain markup and only collapses exact adjacent blocks separated by blank lines, so the reported glued and same-line duplicate gap remains. The issue should stay open for a maintainer choice on the conservative heuristic and canonical owner boundary: the prior narrow PRs closed unmerged, while the only open related PR is a broad draft with compatibility and message-delivery risk rather than a safe superseding landing path.

## What This Changes

Improve shared assistant-output sanitization for glued final-tag repeats, same-line duplicates, and conservative partial-duplicate handling.

## Best Possible Solution

Use one canonical final-delivery normalization path and add only structurally strong duplicate recovery—such as the concrete glued final-tag boundary—with regression tests preserving intentional repetition, ordinary single-newline formatting, quoted targets, indentation, and fenced code.

## Reproduction Assessment

Yes at source level. The reported glued final-tag string deterministically passes through tag stripping and then bypasses the blank-line-only duplicate collapse on current main; no live WhatsApp reproduction was performed.

## Solution Assessment

No, the broad substring and near-duplicate suggestions are not safe as written. The best approach is a narrowly structured rule at the canonical final-delivery boundary, with explicit preservation cases and maintainer agreement on supported duplicate shapes.
