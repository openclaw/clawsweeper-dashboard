## Summary

Keep open: current main and v2026.6.8 still strip final tags textually and only collapse exact duplicate blocks separated by blank lines, so the reported same-line/tag-boundary duplicate case remains. The narrow prior fix PRs are closed unmerged, while the open normalizer PR is a broad draft/conflicting partial overlap rather than a landed canonical fix.

## What This Changes

The issue requests stronger user-facing output sanitization for glued final-tag repeats, same-line or single-newline duplicates, and bounded partial duplicate cleanup.

## Best Possible Solution

Land a conservative sanitizer repair for the concrete glued `</final>` repeat and any explicitly supported adjacent same-line or single-newline duplicate form, with regression tests that preserve intentional repetition, formatting, and code-block indentation.

## Reproduction Assessment

Yes, at source level. Current main removes `<final>` tags before a paragraph-only duplicate collapse that returns unchanged for same-line or single-newline duplicates, so the reported glued duplicate shape remains unhandled without a live WhatsApp run.

## Solution Assessment

No, the current implementation is not the best end state. The narrow maintainable fix is a tested sanitizer rule for concrete adjacent/tag-boundary duplicates, while broad near-duplicate cleanup needs maintainer product bounds first.
