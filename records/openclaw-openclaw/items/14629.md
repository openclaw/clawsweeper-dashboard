## Summary

Keep open: current main and v2026.6.5 still strip final tags before a paragraph-only duplicate-collapse pass, so the reported same-line/tag-boundary duplicate remains source-reproducible; the broader visible-output PR is still open draft/conflicting and does not close this issue.

## What This Changes

The issue asks OpenClaw to improve user-facing output sanitization so same-line, tag-boundary, and narrowly supported partial duplicate assistant text is not delivered after final-tag stripping.

## Best Possible Solution

Define a conservative sanitizer contract for glued final-tag repeats and explicitly supported adjacent single-newline duplicates, or land that behavior through the broader visible-output normalizer, with regression tests that preserve intentional repetition and formatting.

## Reproduction Assessment

Yes. Source inspection shows final tags are removed before dedupe, the dedupe pass only splits on blank lines, and outbound replies call this sanitizer before delivery; I did not run tests because this review is read-only.

## Solution Assessment

Unclear for the full requested heuristic. A narrow adjacent/tag-boundary sanitizer pass is likely the best fix for the concrete leak, while partial or near-duplicate detection needs a maintainer product boundary to avoid deleting intentional repetition.
