## Summary

Keep open: current main and the latest release still strip final tags as plain text and only collapse exact blank-line-separated duplicate blocks, so the glued same-line final-tag duplicate remains source-reproducible. The broader partial-duplicate behavior still needs maintainer bounds, and the closest active normalizer work is an open draft rather than shipped code.

## What This Changes

Request to improve `sanitizeUserFacingText()` so same-line, single-newline, tag-boundary, and partial duplicate assistant output is not delivered to users.

## Best Possible Solution

Define conservative delivery-sanitizer behavior for glued final-tag repeats and exact adjacent same-line or single-newline duplicates, while leaving fuzzy partial-duplicate cleanup to an approved visible-output normalizer design.

## Reproduction Assessment

Yes, at source level. The reported string loses the `</final>` marker through textual final-tag stripping and then remains a single block, so the current duplicate collapse returns it unchanged.

## Solution Assessment

Partly. A narrow exact duplicate or final-tag-boundary pass is maintainable, but broad substring and near-duplicate cleanup needs maintainer product bounds to avoid dropping legitimate repeated text.
