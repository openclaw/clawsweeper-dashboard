## Summary

Current main and v2026.6.5 still reproduce the concrete sanitizer gap from source: final tags are removed before a deduplication pass that only recognizes exact adjacent blocks separated by blank lines. The open overlapping normalizer PR is draft and conflicted, so it is not yet a safe canonical replacement; maintainers should keep this issue open while choosing a conservative rule that does not erase intentional repetition.

## What This Changes

Improve user-facing output sanitization to recognize glued same-line duplicates and carefully scoped partial duplicates after model-control tags are removed.

## Best Possible Solution

Define one canonical delivery-normalization boundary and add an artifact-specific exact adjacent or tag-boundary recovery rule, with preservation tests for explicitly requested repetition, ordinary single-newline prose, trailing whitespace, lists, and indented code.

## Reproduction Assessment

Yes—there is a high-confidence source reproduction: stripping the embedded `</final>` produces one glued line, and the current duplicate collapse immediately returns unchanged because it finds no blank-line-separated blocks. A live current-main WhatsApp reproduction was not performed.

## Solution Assessment

No, the full proposal is not yet the best solution because arbitrary partial-duplicate and substring detection is too destructive for a delivery sanitizer. A narrowly specified exact artifact rule in the canonical visible-output path is safer.
