## Summary

Keep open: current main and the latest shipped tag still strip final tags textually before paragraph-only duplicate collapse, so the reported same-line/glued final-tag duplicate path remains. ## What This Changes

The issue requests stronger user-facing output sanitization for same-line, tag-boundary, and partial duplicate assistant replies.

## Best Possible Solution

Define a conservative delivery-sanitizer rule for glued final-tag repeats and exact adjacent same-line or single-newline duplicates, while leaving fuzzy partial dedupe to an approved visible-output normalizer design.

## Reproduction Assessment

Yes, at source level. The reported string loses `</final>` through textual final-tag stripping and then remains one block without `\n\n`, so current duplicate collapse returns it unchanged; no live WhatsApp replay was run.

## Solution Assessment

Partly. A narrow exact duplicate or final-tag-boundary pass is maintainable, but broad substring and near-duplicate cleanup needs maintainer product bounds to avoid dropping legitimate repeated text.
