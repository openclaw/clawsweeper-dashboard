## Summary

Keep open: current main and the latest release still strip final tags textually and only collapse exact duplicate blocks separated by blank lines. The same-line/tag-boundary duplicate case remains, prior narrow fix PRs are closed unmerged, and the open broader normalizer PR is only a partial, draft overlap rather than a landed fix.

## What This Changes

The issue requests a safer `sanitizeUserFacingText()` duplicate pass for glued same-line, single-newline, and partial duplicates left after final-tag stripping.

## Best Possible Solution

Land a conservative sanitizer repair for the concrete glued `</final>` repeat and any explicitly supported adjacent single-line/single-newline duplicate form, with tests that preserve intentional repetition, formatting, and code-block indentation.

## Reproduction Assessment

Yes, at source level. Current main removes `<final>` tags before a paragraph-only duplicate collapse that returns unchanged for same-line or single-newline duplicates, so the reported glued duplicate shape remains unhandled without needing a live WhatsApp run.

## Solution Assessment

No, the current implementation is not the best end state. The maintainable fix is a narrow, tested sanitizer rule for concrete adjacent/tag-boundary duplicates, while broad near-duplicate cleanup needs maintainer product bounds first.
