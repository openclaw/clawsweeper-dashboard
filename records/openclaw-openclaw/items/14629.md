## Summary

Keep open: current main still strips final tags as plain text and only collapses blank-line-separated duplicate blocks, so the reported glued same-line duplicate remains source-reproducible. The prior same-line fix PRs are closed unmerged, and the open visible-output PR is a broader draft branch rather than a landed terminal fix for this issue.

## What This Changes

Request to improve `sanitizeUserFacingText()` so user-facing output can catch glued same-line duplicates, single-newline duplicates, and carefully scoped partial/tag-boundary duplicate patterns.

## Best Possible Solution

Add a conservative regression and sanitizer pass for glued `</final>` repeats and deliberately supported adjacent same-line/single-newline duplicate shapes, while leaving broad fuzzy partial-dedupe behavior for a separate maintainer-approved design.

## Reproduction Assessment

Yes, at source level. The reported string reaches final-tag stripping, becomes a glued same-line repeat, and then bypasses duplicate collapse because the helper only splits on blank-line separators.

## Solution Assessment

Partly. A narrow tag-boundary or adjacent-duplicate sanitizer repair is maintainable; broad partial/near-duplicate detection is not the best path without explicit product approval because it risks suppressing intentional repeats.
