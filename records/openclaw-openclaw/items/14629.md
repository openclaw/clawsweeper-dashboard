## Summary

Current main still has the reported sanitizer gap: final tags are stripped as plain tag removal, duplicate collapse remains blank-line paragraph-only, and no merged related PR now owns the same-line/tag-boundary duplicate fix.

## What This Changes

The issue asks for `sanitizeUserFacingText()` to catch same-line, tag-boundary, and conservative partial duplicate assistant output that the current paragraph-only duplicate collapse misses.

## Best Possible Solution

Keep this issue open as the canonical sanitizer gap until a current-main fix lands, preferably a conservative `sanitizeUserFacingText()` pass with focused preservation tests or a ported, review-clean version of the broader visible-output normalizer work.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: final tags are stripped without boundary semantics, then duplicate collapse returns unchanged text for single-line or single-newline repeats because it only splits on blank-line-separated blocks.

## Solution Assessment

Unclear pending maintainer product judgment. A narrow sanitizer repair is the likely best shape, but maintainers still need to choose how aggressive partial-duplicate detection may be and whether to land it via a smaller current-main fix or the broader normalizer PR.
