## Summary

Keep open: current main and v2026.6.10 still strip final tags as plain markers and only collapse exact duplicate blocks separated by blank lines, so the reported same-line/tag-boundary duplicate output remains. The direct fix attempts are closed unmerged, and the open broad normalizer PR is a draft/conflicting partial overlap rather than a landed canonical fix.

## What This Changes

Requests improved user-facing output sanitization for same-line, tag-boundary, and partial duplicate assistant text after final-tag stripping.

## Best Possible Solution

Land a conservative, tested sanitizer rule for adjacent glued </final> repeats and any explicitly supported single-newline duplicate pattern, while leaving broad semantic near-duplicate cleanup to maintainer-approved sanitizer policy.

## Reproduction Assessment

Yes, at source level: current main removes <final> tags before a paragraph-only duplicate collapse that returns unchanged for same-line or single-newline duplicates. I did not run a live WhatsApp reproduction or a test command in this read-only review.

## Solution Assessment

No, the current implementation is not the best end state because it misses the reported glued/tag-boundary duplicate form. The best fix is a narrow sanitizer rule with regression tests; broad near-duplicate cleanup needs maintainer product bounds first.
