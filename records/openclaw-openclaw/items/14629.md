## Summary

Keep open: current main and v2026.6.10 still strip final tags textually and only collapse exact duplicate blocks separated by blank lines, so the reported same-line/tag-boundary duplicate cleanup is not implemented. The direct fix PRs are closed unmerged, and https://github.com/openclaw/openclaw/pull/68986 is an open draft/conflicting partial overlap rather than a landed canonical fix.

## What This Changes

The issue asks to extend `sanitizeUserFacingText()` so same-line, single-newline, tag-boundary, or partial duplicate model output is collapsed before delivery.

## Best Possible Solution

Land a conservative sanitizer repair for the concrete glued `</final>` repeat and any explicitly approved adjacent same-line or single-newline duplicate form, with regressions that preserve intentional repetition and formatting.

## Reproduction Assessment

Yes, at source level. Current main removes `<final>` tags before a paragraph-only duplicate collapse that returns unchanged for same-line or single-newline duplicates, so the reported glued duplicate shape remains unhandled without a live WhatsApp run.

## Solution Assessment

No, current main is not the best end state. The narrow maintainable path is a tested sanitizer rule for concrete adjacent/tag-boundary duplicates; broader near-duplicate cleanup needs maintainer product bounds first.
