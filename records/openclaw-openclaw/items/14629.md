## Summary

Keep open: current main still has the reported sanitizer gap. Final tags are removed as plain text, duplicate collapse remains blank-line block-only, and outbound replies still pass through this sanitizer, so the glued same-line repeat can still reach users.

## What This Changes

Improve `sanitizeUserFacingText()` so glued or same-line duplicates left by final-tag stripping are removed without damaging normal reply formatting.

## Best Possible Solution

Add a targeted sanitizer pass for the concrete tag-boundary/glued duplicate pattern and explicitly supported adjacent same-line or single-newline duplicates, with preservation tests for normal formatting.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: after `</final>` replacement the reported example becomes one continuous block, and the current duplicate collapse returns one-block text unchanged because it only splits on blank lines.

## Solution Assessment

Yes. A narrow repair in the existing `sanitizeUserFacingText()` path is the maintainable direction; broad near-duplicate or whole-response coherence heuristics should remain out of scope unless separately designed.

Current main strips `<final>`/`</final>` tags as plain text and then only collapses exact adjacent duplicate blocks separated by blank lines, so glued same-line repeats such as `Atlas here...?</final>Atlas here...?Fair point.` can reach outbound replies. Implement a conservative pass in `src/agents/pi-embedded-helpers/sanitize-user-facing-text.ts` for the concrete tag-boundary/glued duplicate case and any explicitly supported adjacent same-line or single-newline exact duplicate form. Do not add broad partial/near-duplicate or whole-response coherence heuristics. Add regression tests in `src/agents/pi-embedded-helpers.sanitizeuserfacingtext.test.ts` proving the reported shape is cleaned and ordinary repeated wording, normal single-newline formatting, trailing whitespace, and code-block indentation are preserved.
