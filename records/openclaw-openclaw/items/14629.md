## Summary

Keep open: current main and the latest release still remove `<final>` tags as text and only collapse duplicate blocks separated by blank lines, so the reported same-line/tag-boundary duplicate gap remains. The direct fix PRs are closed unmerged, and the open broad normalizer PR is draft/conflicting partial overlap rather than a landed canonical fix.

## What This Changes

The issue requests stricter user-facing sanitizer duplicate detection for same-line, single-newline, partial, and final-tag-boundary repeated output.

## Best Possible Solution

Land a conservative sanitizer rule for concrete adjacent/tag-boundary duplicates, with regression tests that preserve intentional repetition, ordinary single-newline formatting, trailing whitespace, and code indentation.

## Reproduction Assessment

Yes, at source level. Current main removes `<final>` tags before a paragraph-only duplicate collapse that returns unchanged for same-line or single-newline duplicates, so the reported glued duplicate shape remains unhandled without a live WhatsApp run.

## Solution Assessment

No, the current implementation is not the best end state. The narrow maintainable fix is a tested sanitizer rule for concrete adjacent/tag-boundary duplicates, while broad near-duplicate cleanup needs maintainer product bounds first.
