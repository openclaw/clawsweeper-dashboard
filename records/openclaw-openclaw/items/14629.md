## Summary

Current main and the latest release still strip final tags as plain markers and only collapse duplicate blocks separated by blank lines. The direct same-line duplicate PRs are closed unmerged, and the open visible-output normalizer PR is broad, draft, and not a safe canonical fix, so this issue remains necessary.

## What This Changes

The issue asks OpenClaw to improve user-facing output sanitization so glued same-line duplicates, single-newline duplicates, and tag-boundary duplicate text do not reach channels.

## Best Possible Solution

Define a conservative sanitizer boundary, then land a focused repair for glued final-tag repeats and deliberately supported adjacent duplicate forms with regression tests that preserve intentional repeats, normal single-newline formatting, and code-block whitespace.

## Reproduction Assessment

Yes, at source level: current main strips final tags as plain markers and only collapses duplicate blocks split by blank lines, so the reported glued same-line duplicate remains possible on the outbound reply path. I did not run a live WhatsApp reproduction in this read-only review.

## Solution Assessment

No, not as the full broad heuristic request. The maintainable path is a conservative sanitizer repair for concrete adjacent, glued, or tag-boundary patterns, not a general partial-duplicate remover without false-positive guardrails.
