## Summary

Keep open: current main still has the reported user-facing sanitizer gap. The existing sanitizer strips `<final>` tags as plain text, then only collapses exact adjacent blocks separated by blank lines, so glued same-line repeats after `</final>` are still not removed; newer assistant-visible and WhatsApp delivery sanitizers strip scaffolding/tags but do not add duplicate collapse for this pattern.

## What This Changes

The issue requests improved output sanitization for same-line, single-newline, partial, and `</final>` boundary duplicate text left in assistant replies after final-tag stripping.

## Best Possible Solution

Add a conservative sanitizer pass for the concrete glued `</final>` repeat and explicitly supported adjacent same-line or single-newline exact duplicate forms, with regression tests proving ordinary repeated wording, normal single-newline formatting, trailing whitespace, and code-block indentation are preserved.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: after `</final>` replacement the reported example becomes one glued block, and the current duplicate collapse returns single-block text unchanged because it only splits on blank lines.

## Solution Assessment

Yes. A narrow sanitizer repair in the existing user-facing sanitizer path is the maintainable direction; broad near-duplicate or whole-response coherence heuristics should be avoided unless tightly constrained to observed leak patterns and covered by tests.

## Repair Work Prompt

Fix the user-facing sanitizer gap from https://github.com/openclaw/openclaw/issues/14629. On current main, `sanitizeUserFacingText()` strips `</final>` as plain text and `collapseConsecutiveDuplicateBlocks()` only handles exact adjacent blank-line paragraph duplicates, so glued same-line repeats such as `Atlas here...?</final>Atlas here...?Fair point.` can still reach users. Add a conservative sanitizer pass in the existing sanitizer path for the concrete glued `</final>` repeat and any explicitly supported adjacent same-line or single-newline exact duplicate forms. Keep the heuristic narrow: do not add broad fuzzy/near-duplicate removal that could strip intentional repeated wording. Add focused tests in `src/agents/pi-embedded-helpers.sanitizeuserfacingtext.test.ts` for the reported glued final-tag pattern, supported same-line/single-newline exact duplicates, ordinary repeated wording that must be preserved, normal single-newline formatting, trailing whitespace, and code-block indentation. Before opening a PR, re-check whether draft PR #68986 has landed; if it has, adapt to current main instead of duplicating it. Changelog: include a user-facing fix entry only if the repo expects sanitizer behavior fixes in the active changelog, crediting the reporter/source contributors and not bot accounts.
