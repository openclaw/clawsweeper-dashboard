## Summary

Keep open: current main and the latest release still strip final tags as plain tag removal and only collapse duplicate blocks separated by blank lines, so the reported same-line/tag-boundary duplicate remains; the open related PR is a broad draft/conflicting candidate, not a merged fix.

## What This Changes

The issue asks OpenClaw to make the user-facing output sanitizer catch same-line, single-newline, partial, and tag-boundary duplicate text left after final-tag stripping.

## Best Possible Solution

Land a narrow core sanitizer repair for the concrete glued final-tag and adjacent same-line duplicate cases, with tests that preserve intentional repetition and ordinary single-newline formatting; use https://github.com/openclaw/openclaw/pull/68986 only if it is ported and narrowed enough for the active runtime.

## Reproduction Assessment

Yes, at source level: after final-tag stripping, a same-line duplicate has no blank-line split, so `collapseConsecutiveDuplicateBlocks()` returns the text unchanged. I did not run tests in this read-only review.

## Solution Assessment

The best direction is a bounded sanitizer repair, not a broad near-duplicate heuristic. Maintainers should define the conservative duplicate boundary so intentional repeated phrasing is not removed.
