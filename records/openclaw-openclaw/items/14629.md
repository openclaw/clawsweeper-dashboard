## Summary

Keep open: current main and the latest release still remove final tags as plain token deletion and only collapse exact blank-line duplicate blocks, so the reported glued same-line duplicate remains source-reproducible. The closest open implementation path is a draft, conflicting normalizer PR, so it is not a safe superseding item yet.

## What This Changes

The issue asks OpenClaw to improve user-facing output sanitization so same-line, single-newline, and tag-boundary duplicate text does not leak to channels.

## Best Possible Solution

Define conservative delivery-sanitizer behavior for glued final-tag repeats and exact adjacent same-line or single-newline duplicates, while leaving fuzzy partial-duplicate cleanup to an approved visible-output normalizer design.

## Reproduction Assessment

Yes, at source level. The reported string loses the closing final tag through textual final-tag stripping and then remains one block, so current duplicate collapse returns it unchanged.

## Solution Assessment

Unclear. A narrow exact duplicate or final-tag-boundary pass is maintainable, but broad substring and near-duplicate cleanup needs maintainer product bounds to avoid dropping legitimate repeated text.
