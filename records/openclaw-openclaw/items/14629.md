## Summary

Keep open: current main still strips `<final>` tags as plain text and only collapses exact duplicate blocks separated by blank lines, so the reported glued same-line duplicate remains on the outbound sanitizer path. The earlier same-line duplicate PRs were closed unmerged, and the open adjacent PRs are either narrower indentation work or broader draft sanitizer work that has not landed.

## What This Changes

Improve the user-facing output sanitizer so tag-stripped same-line, single-newline, or partial duplicate assistant text does not reach users.

## Best Possible Solution

Add a conservative sanitizer pass for the concrete glued `</final>` repeat and explicitly supported adjacent same-line or single-newline exact duplicate forms, with regression tests preserving normal repeated wording, single-newline formatting, trailing whitespace, and code-block indentation.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: after `</final>` replacement the reported example becomes one glued block, and current duplicate collapse returns single-block text unchanged because it only splits on blank lines.

## Solution Assessment

Yes. The maintainable fix belongs in the existing user-facing sanitizer path with narrow exact/tag-boundary duplicate handling; broad near-duplicate or whole-response coherence heuristics should stay out of this repair unless separately designed.
