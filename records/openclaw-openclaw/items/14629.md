## Summary

Keep open: current main still strips final tags as plain text and only collapses exact adjacent blocks separated by blank lines, so the reported same-line/final-tag-boundary duplicate gap remains. The focused same-line PRs are closed unmerged, while the broader visible-output normalizer is still an open draft needing changes rather than a shipped superseding fix.

## What This Changes

The issue asks to expand user-facing output sanitization to collapse same-line, final-tag-boundary, and possibly partial duplicates that current paragraph-only collapse misses.

## Best Possible Solution

Define a conservative delivery-sanitizer rule for glued `</final>` repeats and exact adjacent same-line or single-newline duplicates, then leave broad fuzzy partial-dedupe behavior to a maintainer-approved visible-output normalizer design.

## Reproduction Assessment

Yes, at source level. The reported string loses `</final>` through final-tag stripping, then remains a single block without `\n\n`, so the duplicate collapse path returns it unchanged; no live WhatsApp replay was run.

## Solution Assessment

Partly. A narrow exact duplicate or final-tag-boundary pass is maintainable, but broad substring and near-duplicate cleanup needs product bounds to avoid dropping legitimate repeated text.
