## Summary

Keep open: current main and the latest shipped tag still have the reported sanitizer gap. The active helper strips `<final>` tags textually, then only collapses exact adjacent blocks separated by blank lines, so the reported glued same-line duplicate remains on the outbound reply path.

## What This Changes

The issue asks for `sanitizeUserFacingText()` to catch same-line, single-newline, and tag-boundary duplicate assistant output that survives final-tag stripping.

## Best Possible Solution

Add a conservative sanitizer pass in the active helper for the concrete glued `</final>` repeat and any explicitly supported adjacent single-newline duplicate form, with regression coverage that preserves normal repetition, single-newline formatting, trailing whitespace, and code indentation.

## Reproduction Assessment

Yes, from source inspection. The reported input reaches `stripFinalTagsFromText()`, becomes a glued same-line repeat, then bypasses `collapseConsecutiveDuplicateBlocks()` because there is no blank-line separator.

## Solution Assessment

Yes, if the fix stays conservative. A narrow helper-level sanitizer repair is the maintainable path; broad fuzzy partial-duplicate removal should be avoided unless maintainers explicitly choose that product behavior.
