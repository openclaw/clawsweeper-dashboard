## Summary

Keep open: current main and the latest release still strip final tags as plain tag removal and only collapse duplicate blocks separated by blank lines, so the same-line/final-tag duplicate gap remains. The open related visible-output PR overlaps, but it is draft, conflicting, broad, and not a merged current-main fix.

## What This Changes

The issue asks for safer duplicate-output sanitization for same-line, single-newline, partial, and final-tag-boundary duplicate assistant replies.

## Best Possible Solution

Land a narrow core sanitizer repair for the concrete glued final-tag and adjacent same-line duplicate cases, with regression tests that preserve intentional repetition, normal single-newline formatting, and code-block indentation.

## Reproduction Assessment

Yes, at source level: after final-tag stripping, a same-line duplicate has no blank-line split, so collapseConsecutiveDuplicateBlocks returns the text unchanged. I did not run tests in this read-only review.

## Solution Assessment

No, the broad partial/near-duplicate direction is not yet the best bounded fix. The maintainable solution is a conservative sanitizer pass for concrete glued/tag-boundary repeats, with tests proving ordinary repetition is preserved.
