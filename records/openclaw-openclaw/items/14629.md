## Summary

Keep open: current main and v2026.6.6 still strip final tags as plain markers and only collapse exact adjacent blocks separated by blank lines, so the reported same-line/tag-boundary duplicate output is not implemented. The direct same-line duplicate PRs are closed unmerged, and the broader visible-output PR is open but draft, conflicting, and only a partial overlap.

## What This Changes

The issue requests stronger user-facing output sanitization for same-line, single-newline, tag-boundary, and partial duplicate model output.

## Best Possible Solution

Define and implement a conservative core sanitizer rule for the concrete glued final-tag duplicate and any deliberately supported adjacent duplicate form, with regression coverage that preserves ordinary repeated wording, single-newline formatting, trailing whitespace, and code indentation.

## Reproduction Assessment

Yes, at source level: after `</final>` is stripped, the current duplicate collapse returns unchanged text unless there are blank-line-separated blocks. I did not run a live WhatsApp/model reproduction in this read-only review.

## Solution Assessment

Unclear as requested because broad partial-duplicate removal needs maintainer judgment; the best maintainable path is a narrow sanitizer rule for concrete glued or adjacent duplicates, not an open-ended structural sanity heuristic.
