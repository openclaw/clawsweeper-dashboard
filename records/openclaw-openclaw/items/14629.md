## Summary

Keep open: current main and the latest release still strip `<final>` tags textually and only collapse exact duplicate blocks separated by blank lines, while the broader related normalizer PR is open, draft, and not a safe canonical fix yet.

## What This Changes

The issue asks the user-facing output sanitizer to catch same-line, single-newline, partial, and tag-boundary duplicates left after final-tag stripping.

## Best Possible Solution

Sponsor a conservative sanitizer repair for the concrete glued `</final>` repeat and any explicitly supported adjacent single-newline duplicate form, with regression tests that preserve intentional repeats, normal single-newline formatting, trailing whitespace, and code indentation.

## Reproduction Assessment

Yes, at source level: stripping `</final>` leaves adjacent text together, and the current duplicate collapse returns unchanged text unless duplicates are separated by blank lines. I did not run tests because this review is read-only and the source path is sufficient to prove the gap.

## Solution Assessment

Unclear as requested in full: the concrete glued final-tag duplicate is a narrow bug fix, but broad partial/near-duplicate removal needs maintainer-chosen boundaries to avoid deleting intentional text.
