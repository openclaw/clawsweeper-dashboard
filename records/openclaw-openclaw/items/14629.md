## Summary

Keep open: current main still strips final tags as plain text before duplicate comparison and only collapses exact adjacent blocks separated by blank lines, so the reported glued final-tag and same-line duplicate path remains. The broader visible-output normalizer is still an open draft, not shipped main behavior.

## What This Changes

The issue requests more conservative duplicate detection in the user-facing output sanitizer for same-line, single-newline, partial, and final-tag-boundary duplicate assistant text.

## Best Possible Solution

Define a conservative delivery-sanitizer rule for glued final-tag repeats and exact adjacent same-line or single-newline duplicates, while leaving fuzzy partial dedupe to a maintainer-approved visible-output normalizer design.

## Reproduction Assessment

Yes, at source level. The reported string loses `</final>` through textual final-tag stripping and then remains one block without `\n\n`, so current duplicate collapse returns it unchanged; no live WhatsApp replay was run.

## Solution Assessment

Partly. A narrow exact duplicate or final-tag-boundary pass is maintainable, but broad substring and near-duplicate cleanup needs maintainer product bounds to avoid dropping legitimate repeated text.
