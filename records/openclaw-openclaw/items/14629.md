## Summary

Keep open: current main still strips final tags textually and only collapses exact duplicate blocks separated by blank lines, so the reported glued `</final>` and same-line duplicate sanitizer gap remains; the broader visible-output normalizer is still an open draft, not a shipped superseding fix.

## What This Changes

The issue requests stronger user-facing output sanitization for same-line duplicates, partial repeats, and final-tag boundary repeats in assistant replies.

## Best Possible Solution

Define and land a conservative delivery-sanitizer rule for glued `</final>` repeats and exact adjacent same-line or single-newline duplicates, while leaving fuzzy partial-dedupe behavior to a maintainer-approved visible-output normalizer design.

## Reproduction Assessment

Yes, at source level. The reported string loses `</final>` through final-tag stripping, then remains one block without `\n\n`, so the current duplicate collapse path returns it unchanged; no live WhatsApp replay was run.

## Solution Assessment

Partly. A narrow exact duplicate or final-tag-boundary pass is maintainable, but broad substring and near-duplicate cleanup needs maintainer product bounds to avoid dropping legitimate repeated text.
