## Summary

Keep open: current main still strips `<final>` tags textually and only collapses exact duplicate blocks separated by blank lines, so the reported glued same-line duplicate can still reach the user-facing reply path.

## What This Changes

The issue asks for `sanitizeUserFacingText()` to catch same-line, single-newline, partial, and tag-boundary duplicate output that is not removed by the current paragraph-only duplicate collapse.

## Best Possible Solution

Land a conservative sanitizer repair for the glued final-tag and adjacent same-line exact duplicate cases, with regression tests and changelog, while leaving broad fuzzy near-duplicate policy to a separate maintainer decision.

## Reproduction Assessment

Yes for the concrete glued final-tag case from source inspection: stripping `</final>` leaves adjacent text, and the duplicate collapse returns unchanged unless it sees a blank-line separator. I did not run a live sanitizer test because this was a read-only review.

## Solution Assessment

Yes, a narrow sanitizer repair is the best path for the demonstrated failure. Broad near-duplicate or whole-response coherence heuristics should be avoided unless maintainers explicitly choose that product behavior.
