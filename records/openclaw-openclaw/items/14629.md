## Summary

Keep open: current main and the latest release still strip final tags as plain text and only collapse exact duplicate blocks separated by blank lines, so the reported glued same-line repeat remains a valid sanitizer gap. The closed same-line duplicate PRs were not merged, and the open adjacent PRs do not fully supersede this issue.

## What This Changes

The issue asks OpenClaw to improve `sanitizeUserFacingText()` duplicate handling for glued same-line repeats, single-newline repeats, and final-tag boundary cleanup.

## Best Possible Solution

Land a conservative sanitizer repair for the concrete glued final-tag repeated-prefix case and any deliberately supported adjacent exact same-line or single-newline duplicate form, with regressions that preserve ordinary formatting and defer fuzzy near-duplicate policy.

## Reproduction Assessment

Yes, source-level. Current main strips `</final>` before duplicate collapse, then the collapse helper returns unchanged unless it sees a blank-line separator, matching the reported glued same-line repeat path.

## Solution Assessment

Yes for the demonstrated failure. A narrow sanitizer pass with regression coverage is maintainable; broad partial-duplicate or whole-response coherence heuristics should stay out unless maintainers explicitly choose that policy.
