## Summary

Keep open: current main still removes final tags as text and only collapses exact adjacent blocks separated by blank lines, so the reported glued `</final>` and same-line duplicate path remains in the outbound sanitizer. The broader visible-output normalizer is still an open draft rather than a shipped superseding fix.

## What This Changes

The issue asks for the user-facing output sanitizer to catch same-line duplicates, partial duplicates, and final-tag-boundary repeats that survive the existing paragraph-level duplicate collapse.

## Best Possible Solution

Define a conservative delivery-sanitizer rule for glued `</final>` repeats and exact adjacent same-line or single-newline duplicates, while leaving fuzzy partial-dedupe behavior to a maintainer-approved visible-output normalizer design.

## Reproduction Assessment

Yes, at source level. The reported string loses `</final>` through final-tag stripping, then remains a single block without `\n\n`, so the current duplicate collapse path returns it unchanged; no live WhatsApp replay was run.

## Solution Assessment

Partly. A narrow exact duplicate or final-tag-boundary pass is maintainable, but broad substring and near-duplicate cleanup needs maintainer product bounds to avoid dropping legitimate repeated text.
