## Summary

Keep open: current main and the latest release still strip `<final>` tags as plain markers and only collapse exact adjacent duplicate blocks separated by blank lines, so the reported glued same-line duplicate sanitizer gap remains. The prior narrow fix PRs are closed unmerged, and the open broad visible-output normalizer is a draft with compatibility and message-delivery risk rather than a merged superseding fix.

## What This Changes

The issue asks OpenClaw to improve user-facing output sanitization so glued same-line duplicates, partial adjacent duplicates, and duplicate content around stripped final tags are removed without damaging normal output.

## Best Possible Solution

Choose the canonical normalization path, then add a conservative sanitizer repair for the glued final-tag boundary and any explicitly supported adjacent duplicate form with preservation tests for intentional repetition and formatting.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: stripFinalTags() removes `</final>` while preserving both sides, then collapseConsecutiveDuplicateBlocks() returns same-line text unchanged because it only splits on blank lines. I did not run a live WhatsApp/model reproduction in this read-only review.

## Solution Assessment

Unclear as an implementation direction until maintainers choose the false-positive boundary. A narrow tag-boundary or adjacent-duplicate repair is safer than a broad semantic deduper, but it should be coordinated with the open visible-output normalizer work.
