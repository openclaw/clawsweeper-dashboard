## Summary

Keep open: current main and the latest release still drop explicit reasoning block replies in Discord, even though the shared agent layer and docs define `/reasoning on` as a separate visible Thinking message.

## What This Changes

The issue reports that Discord does not render model `type:"thinking"` content as persistent messages when `/reasoning on` is enabled.

## Best Possible Solution

Make Discord honor the existing `/reasoning on` contract by rendering explicit `isReasoning` block replies as persistent Thinking messages, while keeping `/reasoning off` and internal/non-opted-in reasoning suppressed.

## Reproduction Assessment

Yes, source-level: shared reasoning emission produces an `isReasoning` block reply for `/reasoning on`, and Discord immediately drops `payload.isReasoning`. I did not perform a live Discord provider run in this read-only review.

## Solution Assessment

Yes, the best fix is a narrow Discord delivery/formatting repair because the provider extraction and shared agent reasoning emission path already produce the right payload. A new config option or provider-specific DeepSeek change would be the wrong layer.
