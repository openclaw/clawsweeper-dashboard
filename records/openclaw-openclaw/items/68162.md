## Summary

Keep open. The report is recent, corroborated by a second install, and current main still does not prove that Control UI reconnect/page refresh preserves the same backing gateway session without firing a reset/new-session rollover. Source inspection shows reconnect/history recovery paths, but `chat.send` still sends only `sessionKey` and the gateway can allocate a new `sessionId` when its stored entry is missing or fails freshness, so closure would be premature without a targeted regression.

## What This Changes

The issue requests that Control UI/WebChat resume the existing active session after WebSocket reconnects, browser refreshes, or tab lifecycle interruptions instead of silently creating a fresh session and losing visible context.

## Best Possible Solution

Add a focused reconnect/page-refresh regression and make Control UI session continuity explicit: after reconnect or remount with the same selected session, history reload and follow-up send should use the same backing `sessionId` unless the user intentionally starts a new session or the configured reset policy legitimately expires it.

## Reproduction Assessment

Unclear for current main. The issue and follow-up comment provide a concrete release-era scenario and log evidence, but there is no current-main automated or live reproduction that forces reconnect/page refresh and asserts the same backing `sessionId` after the next `chat.send`.

## Solution Assessment

Yes, the narrowest maintainable path is to add the missing regression first, then fix the specific UI/gateway resume or reset-trigger boundary it exposes. A broad session-management redesign should stay out of this fix except where needed to avoid duplicating #73725.
