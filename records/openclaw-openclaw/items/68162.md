## Summary

Keep open. The report is recent, corroborated by another install, and current main still has the same unresolved continuity boundary: Control UI reconnects and refreshes state, but follow-up sends are keyed rather than explicitly bound to the previous backing session, and the gateway can still allocate a new session id when the stored entry is missing or considered stale.

## What This Changes

The issue asks for Control UI/WebChat WebSocket reconnects or page refreshes to resume the existing session instead of creating a new backing session and losing visible context.

## Best Possible Solution

Make Control UI session continuity explicit: after reconnect or remount with the same selected session, history reload and the next `chat.send` should continue against the same backing `sessionId` unless the user intentionally creates/resets a session or the configured reset policy legitimately expires it.

## Reproduction Assessment

Unclear for current main. The issue and follow-up comment provide concrete release-era logs, and current source still shows a plausible rollover boundary, but there is no focused current-main test or live reproduction that forces reconnect/page refresh and asserts the backing `sessionId` remains stable.

## Solution Assessment

Yes for the proposed repair direction: add the missing regression first, then fix the narrow UI/gateway resume boundary it exposes. A broad session-management redesign should stay out of this fix, and #73725's New Chat/session-create work should remain separate unless the regression proves the same root cause.
