## Summary

Keep open: the hardening is still needed because current main and the latest release still have unbounded image-generation success JSON reads, but this PR is not merge-ready because it applies the shared 16 MiB JSON cap to base64 image-response providers where valid multi-image payloads can exceed that aggregate size.

## What This Changes

The PR replaces six bundled image-generation success `response.json()` reads with `readProviderJsonResponse` and updates OpenAI/OpenRouter provider-http test mocks.

## Best Possible Solution

Keep the bounded-reader approach, but use an image-response-sized JSON budget for base64-returning providers and focused valid-large plus overflow coverage before merge.

## Reproduction Assessment

Yes at source level: current main and `v2026.6.10` still use unbounded `response.json()` reads on provider-controlled image-generation success bodies. I did not rerun the contributor's Node HTTP harness in this read-only review.

## Solution Assessment

No, not as merge-ready. Reusing `readProviderJsonResponse` is the right layer, but base64 image JSON paths need an image-sized cap rather than the generic 16 MiB default.
