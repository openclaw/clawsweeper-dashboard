## Summary

Keep this PR open for maintainer review: the patch appears to be the right hardening layer and has sufficient real loopback proof, but the new successful-response caps are a compatibility policy tradeoff and overlap with sibling provider hardening PRs.

## What This Changes

The branch routes image-generation success JSON reads through bounded provider readers, adds an inline image JSON cap helper exported through the Plugin SDK, and updates provider tests plus SDK surface baselines.

## Best Possible Solution

Land this branch or a maintainer-equivalent after accepting the image JSON cap policy, with overlapping fal and MiniMax PRs rebased or narrowed around the canonical image-generation change.

## Reproduction Assessment

Yes, at source and proof level. Current main uses raw response.json() on provider-controlled image-generation success bodies, and the PR includes loopback proof showing bounded reads stop oversized streams while the old path buffers the full body.

## Solution Assessment

Yes, this is the best implementation layer. Reusing readProviderJsonResponse avoids a custom reader path, and the follow-up image-sized cap addresses the main inline base64 compatibility concern better than a flat 16 MiB cap.
