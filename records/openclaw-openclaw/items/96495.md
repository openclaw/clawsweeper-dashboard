## What This Changes

The PR routes image-generation provider success JSON responses through bounded provider JSON readers and adds image-sized inline response caps for base64 image payloads.

## Best Possible Solution

Land one coherent image-generation response hardening path with image-sized caps for inline payloads, generic caps for metadata/job responses, and separate handling for broader fal or MiniMax non-image media work.

## Reproduction Assessment

Yes, at source level. Current main uses raw response.json() on provider-controlled image-generation success bodies, and the PR body includes loopback proof showing bounded reads stop oversized streams while the old path buffers the full body.

## Solution Assessment

Yes, this is the right implementation layer. Reusing readProviderJsonResponse avoids a parallel reader path, and the follow-up image-sized cap addresses the main inline base64 compatibility concern while leaving a maintainer policy choice on the cap.
