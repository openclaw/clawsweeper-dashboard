## Summary

Keep open. Current main still treats cron `delivery.accountId` as a static explicit route, so the reported stale-account breakage for `openclaw-weixin` is not implemented, obsolete, or superseded.

## What This Changes

Cron delivery is requested to resolve a currently active channel account when a stored account ID becomes stale for plugins such as `openclaw-weixin`.

## Best Possible Solution

Add a channel-agnostic delivery-time account resolution policy that preserves valid explicit accounts, recovers only when exactly one safe account exists, and fails with an actionable diagnostic when recovery is ambiguous.

## Reproduction Assessment

Yes, at source level. A stale explicit `delivery.accountId` is normalized, given highest precedence, and passed into outbound delivery without validation or dynamic replacement on current main.

## Solution Assessment

Unclear until maintainers choose the account recovery policy. The safest direction is a generic delivery-time fallback, not startup mutation of cron jobs or a WeChat-specific branch.
