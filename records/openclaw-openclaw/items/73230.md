## Summary

Keep open. Current main still persists and forwards cron delivery `accountId` as a static explicit value, so the reported stale-account failure remains valid and narrow enough for a generic cron delivery fix.

## What This Changes

Request cron announce delivery to resolve stale or omitted `delivery.accountId` dynamically for channels such as `openclaw-weixin` whose active account id can rotate after re-login or restart.

## Best Possible Solution

Add channel-agnostic delivery-time stale-account recovery: preserve valid explicit accounts, recover only when exactly one eligible account exists for the same channel, and fail with an actionable diagnostic when recovery is unsafe.

## Reproduction Assessment

Yes. A source-level reproduction exists: configure an isolated cron announce job with a stale `delivery.accountId`; current main threads that string through delivery planning, target resolution, and outbound delivery without validating or replacing it.

## Solution Assessment

Yes. A narrow generic recovery path in cron delivery target resolution is the maintainable fix; mutating persisted jobs on Gateway startup or adding WeChat-specific logic would be riskier.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/73230 with a narrow, channel-agnostic cron delivery account recovery path. In `resolveDeliveryTarget`, preserve valid explicit `delivery.accountId` values, but when the explicit account is stale/unconfigured/disabled, or when no explicit/session/binding account is available, recover only if exactly one eligible configured and enabled account exists for the same loaded channel. If zero or multiple eligible accounts exist, fail with an actionable diagnostic rather than silently choosing. Do not hardcode `openclaw-weixin`, do not mutate persisted cron jobs at Gateway startup, and do not weaken existing explicit-account precedence for valid accounts. Add focused tests for stale explicit account recovery, valid explicit account preservation, ambiguous multi-account failure, omitted-account single-active fallback if implemented, and disabled/unconfigured account handling. Add a user-facing `CHANGELOG.md` fix entry. Validate with the targeted cron/channel tests plus changed gate in Blacksmith Testbox before handoff.
