## Summary

Keep open. Current main still treats cron `delivery.accountId` as a static explicit account identity, with no `auto`/`active` selector, stale-account validation, single-active-account recovery, or startup diagnostic for rotating channel plugin account IDs.

## What This Changes

Allow cron announce delivery to resolve the current channel account dynamically when a persisted `delivery.accountId` becomes stale for channels whose account tokens rotate.

## Best Possible Solution

Add channel-agnostic delivery-time account recovery: preserve valid explicit accounts, recover only when the stored account is invalid and exactly one eligible active/configured account exists for the same channel, and otherwise fail with an actionable diagnostic.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: configure an isolated cron announce job with a stale `delivery.accountId`; current main threads that value through `resolveCronDeliveryPlan`, `resolveDeliveryTarget`, and `deliverOutboundPayloads` without validating or recovering it.

## Solution Assessment

Yes. The narrow maintainable direction is generic delivery-time stale-account recovery with strict ambiguity handling; mutating cron job configs on Gateway startup or hard-coding a WeChat-specific branch would be less safe.

## Repair Work Prompt

Fix cron announce delivery for stale explicit `delivery.accountId` values caused by rotating channel plugin account IDs. Implement this as a generic core cron/channel-routing repair, not an `openclaw-weixin` special case. Preserve valid explicit accounts. If the explicit account no longer resolves for the selected channel and exactly one eligible active/configured account exists for that same channel, recover to that account at delivery time and surface the resolved account in diagnostics; if zero or multiple candidates exist, fail with an actionable error instead of silently rerouting. Do not mutate stored cron job configs on Gateway startup. Update focused tests for stale explicit-account recovery, ambiguity, valid explicit-account preservation, and failure messaging; update user-facing docs/changelog if behavior changes.
