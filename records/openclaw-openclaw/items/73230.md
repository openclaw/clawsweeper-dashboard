## Summary

Keep open. Current main still preserves an explicit cron delivery `accountId` as a static route and passes it to outbound delivery without stale-account validation or dynamic single-account recovery, so the reported cron/channel routing gap remains.

## What This Changes

Request cron announce delivery to resolve stale saved account IDs dynamically for channels whose login/session account ID changes, such as `openclaw-weixin`.

## Best Possible Solution

Define a channel-agnostic stale-account recovery policy for cron announce delivery: keep valid explicit accounts, recover only when exactly one configured/enabled account for the channel is safe, and otherwise fail with actionable diagnostics.

## Reproduction Assessment

Yes. Source inspection on current main shows a cron job's `delivery.accountId` is normalized into the delivery plan, given highest precedence in target resolution, and sent unchanged to outbound delivery with no dynamic or unknown-account recovery path.

## Solution Assessment

Unclear until maintainers choose the policy. The narrowest maintainable direction appears to be generic delivery-time stale-account validation/recovery, not WeChat-specific branches or mutating persisted jobs on Gateway startup.
