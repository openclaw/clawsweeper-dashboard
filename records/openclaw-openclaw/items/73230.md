## Summary

Keep open. Current main still treats cron delivery `accountId` as a static explicit identity, so a channel plugin account that rotates after re-login can leave saved cron jobs pointing at a stale account with no generic recovery path.

## What This Changes

Request cron announce delivery to resolve or recover stale `delivery.accountId` dynamically for channel plugins whose account IDs rotate, such as `openclaw-weixin`.

## Best Possible Solution

Add channel-agnostic delivery-time stale-account recovery: preserve valid explicit accounts, recover only when exactly one eligible configured/enabled account exists for the same channel, and fail with an actionable diagnostic when recovery is unsafe.

## Reproduction Assessment

Yes. A source-level reproduction exists: configure an isolated cron announce job with a stale `delivery.accountId`; current main threads that string through delivery planning, target resolution, and outbound delivery without validating or replacing it.

## Solution Assessment

Yes. A narrow generic recovery path in cron delivery target resolution is the maintainable fix; mutating persisted jobs on Gateway startup or adding a WeChat-specific branch would be riskier.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/73230 by adding channel-agnostic cron announce delivery-time stale `delivery.accountId` recovery. In the cron delivery target path, keep an explicit account when it still matches an eligible configured/enabled account for the resolved channel. When the explicit account is missing or stale, recover only if exactly one eligible configured/enabled account exists for that same channel; otherwise return a clear resolution error naming the channel/account and the remediation. Prefer existing plugin config APIs such as `listAccountIds`, `resolveAccount`, `isConfigured`, and `isEnabled`, adding only a narrow helper if needed.
