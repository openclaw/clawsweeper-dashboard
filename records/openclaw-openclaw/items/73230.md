## Summary

Keep open. Current main still treats cron delivery `accountId` as an explicit static override; it only falls back to session or binding data when the field is absent, and there is no stale-account recovery despite generic channel account APIs being available.

## What This Changes

The issue asks cron announce delivery to recover from stale static `delivery.accountId` values by resolving the currently usable account for channels such as `openclaw-weixin` at delivery time.

## Best Possible Solution

Add channel-agnostic delivery-time stale-account recovery: preserve valid explicit accounts, recover only when exactly one eligible configured/enabled account exists for the same channel, and fail with an actionable diagnostic when recovery is unsafe.

## Reproduction Assessment

Yes. A source-level reproduction exists: configure an isolated cron announce job with a stale `delivery.accountId`; current main threads that string through delivery planning, target resolution, and outbound delivery without validating or replacing it.

## Solution Assessment

Yes. The narrow maintainable solution is generic recovery in cron delivery target resolution; mutating persisted jobs on Gateway startup or hardcoding `openclaw-weixin` would be less safe.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/73230. Cron announce delivery with `delivery.accountId` pointing at a stale channel account should not break when the selected channel has exactly one currently eligible configured/enabled account. Implement this in the generic cron delivery target path, likely around `src/cron/isolated-agent/delivery-target.ts` and its runtime helpers. Preserve valid explicit accounts. If an explicit account is missing, unknown, unconfigured, or disabled, recover only when exactly one eligible account exists for the same channel; otherwise fail with an actionable diagnostic. Use existing channel plugin account APIs such as `listAccountIds`, `resolveAccount`, `defaultAccountId`, `isEnabled`, and `isConfigured`; do not hardcode `openclaw-weixin`, change the external plugin, or mutate persisted cron jobs on Gateway startup. Add focused regression tests for stale explicit account recovery, valid explicit account preservation, multi-account ambiguity, and missing/unconfigured account diagnostics. Update user-facing docs/changelog if behavior changes.
