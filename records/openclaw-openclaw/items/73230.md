## Summary

Keep open. Current main still preserves cron `delivery.accountId` as a static explicit route, and the related open Weixin cron issues cover different failure modes rather than superseding this dynamic-account request.

## What This Changes

Cron delivery is requested to resolve or recover the account id dynamically when a saved channel `delivery.accountId` becomes stale after session or token rotation.

## Best Possible Solution

Add channel-agnostic delivery-time account recovery that keeps valid explicit accounts, falls back only when exactly one eligible same-channel account is available, and otherwise fails with an actionable diagnostic.

## Reproduction Assessment

Yes, at source level: configure an isolated cron announce job with a stale explicit `delivery.accountId`, and current main threads that string through planning, target resolution, and outbound delivery without validating or replacing it. I did not establish a live Weixin credential reproduction in this read-only pass.

## Solution Assessment

Unclear until maintainers choose the generic account freshness policy. A narrow resolver-level recovery path is preferable to a Weixin-specific branch or startup mutation of persisted cron jobs.
