## Summary

Keep open. Current main still treats cron delivery `accountId` as a static explicit route, and the reported stale-account failure remains a real cron/channel routing gap rather than something already implemented or superseded.

## What This Changes

Cron delivery should resolve or recover from stale `delivery.accountId` values for channel plugins whose active account identity changes after re-login or Gateway restart.

## Best Possible Solution

Add generic delivery-time stale-account recovery in the cron target resolver: keep valid explicit accounts, recover only to exactly one eligible same-channel account, and otherwise fail with an actionable diagnostic.

## Reproduction Assessment

Yes, at source level: configure an isolated cron announce job with a stale explicit `delivery.accountId`, and current main threads that string through planning, target resolution, and outbound delivery without validating or replacing it. I did not run a live Weixin credential flow in this read-only review.

## Solution Assessment

Yes. A narrow generic recovery path in cron delivery target resolution is the maintainable direction; adding a Weixin-specific branch, mutating persisted jobs on Gateway startup, or requiring a new placeholder value would be less conservative.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/73230 with a channel-agnostic cron delivery account recovery path. In `resolveDeliveryTarget`, preserve an explicit `delivery.accountId` when it is still valid for the resolved channel. When the explicit account is stale or unknown, recover only if the channel plugin/account contracts show exactly one eligible same-channel account; otherwise fail the cron delivery with an actionable diagnostic that names the stale account and how to update or log in. Do not hard-code `openclaw-weixin`, do not mutate persisted cron jobs on Gateway startup, and do not add a new `auto`/`active` config value unless the existing plugin account/status contracts are insufficient. Add regression tests for stale explicit account recovery, multi-account ambiguity, valid explicit-account preservation, and existing binding/session precedence. Add a changelog entry if the final patch changes user-visible cron delivery behavior.
