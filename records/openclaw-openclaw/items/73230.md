## Summary

Keep open. Current main still preserves cron delivery account IDs as static explicit routing data, and I found no implemented `auto`/`active` selector, single-active-account fallback, or stale-account warning/recovery path for rotating channel accounts.

## What This Changes

The issue requests dynamic cron delivery account resolution so channels with rotating account IDs can keep delivering after relogin or Gateway restart.

## Best Possible Solution

Implement generic delivery-time stale-account recovery in cron: preserve explicit account IDs when they still resolve, recover only when the stored ID is stale and exactly one eligible configured/active account exists for the same channel, and otherwise fail with a clear diagnostic. Keep WeChat session-token policy in the external plugin and document the core cron behavior.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure an isolated announce cron job with a stale `delivery.accountId`; current main normalizes that value, gives it highest precedence, and forwards it to outbound delivery without validating or recovering it.

## Solution Assessment

Yes, if the fix is generic and conservative. Delivery-time stale-account recovery with strict ambiguity handling is narrower and safer than mutating job config on Gateway startup or adding a WeChat-specific branch in core.

## Repair Work Prompt

Fix issue https://github.com/openclaw/openclaw/issues/73230. Current main treats cron `delivery.accountId` as a static explicit account identity, so channels with rotating account IDs such as `openclaw-weixin` break after relogin or Gateway restart. Implement a generic, channel-agnostic delivery-time recovery path for isolated cron announce delivery. Preserve explicit account IDs when they still resolve for the selected channel. If the stored account ID is stale or absent and exactly one eligible enabled/configured/active account exists for the same resolved channel, use that account for delivery. If there are zero or multiple eligible accounts, fail closed with an actionable error mentioning the stale account and the fix path. Do not add WeChat-specific core logic, do not mutate cron job config on startup, and do not weaken pairing/allow-from/account-boundary checks or the account-spoof guard from #69940. Update focused tests for valid explicit ID preservation, stale explicit single-account recovery, ambiguous multi-account failure, no eligible account failure, and existing session/binding precedence. Update user-facing cron docs/CLI help or diagnostics as needed, and add a CHANGELOG.md fix entry with reporter credit if policy requires it.
