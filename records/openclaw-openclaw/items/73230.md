## Summary

Keep open. Current main still treats cron delivery `accountId` as a static explicit identity and has no `auto`/`active` selector, single-active-account recovery, or stale-account diagnostic for rotating channel plugin accounts.

## What This Changes

Cron announce delivery should resolve or recover stale `delivery.accountId` values at delivery time for channels whose account identifiers rotate, such as the external WeChat plugin.

## Best Possible Solution

Add a channel-agnostic delivery-time stale-account recovery path: keep valid explicit accounts, recover only when exactly one eligible configured active account exists for the same channel, and otherwise fail with an actionable diagnostic.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: configure an isolated cron announce job with a stale `delivery.accountId`; current main threads that string through `resolveCronDeliveryPlan`, `resolveDeliveryTarget`, and `deliverOutboundPayloads` without validating or recovering it.

## Solution Assessment

Yes. The narrow maintainable solution is generic stale-account recovery with strict ambiguity handling; mutating job configs on Gateway startup or adding a WeChat-specific core branch would be less safe.
