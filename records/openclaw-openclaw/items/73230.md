## Summary

Keep open. Current main still treats cron delivery `accountId` as a static explicit identity and has no `auto`/`active` selector, stale-account validation, single-active-account recovery, or startup diagnostic for rotating channel plugin accounts.

## What This Changes

The issue requests dynamic or recoverable cron `delivery.accountId` resolution for channel plugins whose account IDs can change after login or Gateway restart, such as `openclaw-weixin`.

## Best Possible Solution

Add a conservative, channel-agnostic delivery-time account recovery path: keep valid explicit accounts, recover only when the stored account no longer resolves and exactly one eligible active/configured account exists for the same channel, and otherwise fail with an actionable diagnostic.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: configure an isolated cron announce job with a stale `delivery.accountId`; current main threads that string through `resolveCronDeliveryPlan`, `resolveDeliveryTarget`, and `deliverOutboundPayloads` without validating or recovering it.

## Solution Assessment

Yes. The narrow maintainable fix is generic delivery-time stale-account recovery with strict ambiguity handling; mutating job configs on Gateway startup or adding a WeChat-specific core branch would be less safe.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/73230. Current cron announce delivery persists an explicit `delivery.accountId` and passes it unchanged; rotating channel plugins such as `openclaw-weixin` can generate a new account ID after login or Gateway restart, leaving cron jobs pointed at a stale account. Implement a generic, channel-agnostic delivery-time recovery path. Preserve valid explicit account IDs and existing account-boundary protections. Recover only when the stored account no longer resolves/is not configured for the resolved channel and exactly one eligible enabled/configured account exists for that channel or another already-safe generic source identifies the same account. Ambiguous or missing recovery must fail closed with an actionable error. Do not add WeChat-specific core code, do not mutate cron job configs on startup, and do not weaken the account-spoof guard from https://github.com/openclaw/openclaw/pull/69940. Add focused tests for stale explicit account recovery, no recovery when multiple accounts exist, and unchanged explicit-valid behavior. Update cron docs and CHANGELOG.md if the repair changes user-facing behavior.
