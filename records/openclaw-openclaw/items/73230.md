## Summary

Keep open. Current main still persists and forwards explicit cron account ids without a delivery-time freshness policy, while related open work covers static validation or external-channel lookup rather than dynamic active-account selection.

## What This Changes

Request adding dynamic account-id resolution for cron announce delivery so channels with ephemeral account ids can keep delivering after relogin or Gateway restart.

## Best Possible Solution

Add a channel-agnostic, preferably opt-in delivery-time account resolver that keeps valid explicit routes, recovers only when exactly one eligible same-channel account is available, and otherwise fails with an actionable diagnostic.

## Reproduction Assessment

Yes, at source level: configure an isolated cron announce job with a stale explicit delivery.accountId, and current main carries that value through planning, target resolution, and outbound send without validating or replacing it. I did not run a live Weixin credential relogin path.

## Solution Assessment

Unclear until maintainers choose the account freshness semantics. A generic resolver-level recovery path is preferable to Weixin-specific code or startup mutation of persisted cron jobs, but multiple-account behavior needs an explicit product/API decision.
