## Summary

Keep open. Current main still treats cron `delivery.accountId` as a static explicit account override, and the affected cron/channel files have not changed since the prior review, so rotating plugin account IDs can still leave saved cron jobs pointing at a stale account.

## What This Changes

Cron delivery should support resolving or recovering stale channel account IDs at delivery time for plugins whose account IDs rotate after re-login or Gateway restart.

## Best Possible Solution

Add channel-agnostic delivery-time stale-account recovery: preserve valid explicit accounts, recover only when exactly one eligible configured/enabled account exists for the same channel, and fail with an actionable diagnostic when recovery is unsafe.

## Reproduction Assessment

Yes. A source-level reproduction exists: configure an isolated cron announce job with a stale `delivery.accountId`; current main threads that string through delivery planning, target resolution, and outbound delivery without validating or replacing it.

## Solution Assessment

Yes. A narrow generic recovery path in cron delivery target resolution is the maintainable fix; mutating persisted jobs on Gateway startup or adding a WeChat-specific branch would be riskier.
