## Summary

Keep open. Current main still preserves cron `delivery.accountId` as a static explicit route, so the stale-account failure remains unresolved and is narrow enough to fix in the cron delivery target path.

## What This Changes

Cron announce delivery should resolve stale channel account IDs dynamically for channels whose account IDs can change after re-login or Gateway restart.

## Best Possible Solution

Add channel-agnostic delivery-time stale-account recovery: keep valid explicit accounts, recover only when exactly one eligible account exists for the same channel, and otherwise fail with an actionable diagnostic.

## Reproduction Assessment

Yes. Source inspection on current main shows a stale `delivery.accountId` is normalized, selected with highest precedence, and passed to outbound delivery without validation or replacement.

## Solution Assessment

Yes. The narrow maintainable direction is generic recovery in cron delivery target resolution; mutating persisted jobs on Gateway startup or adding a WeChat-specific core branch would be less safe.

## Repair Work Prompt

Fix cron announce delivery account resolution for stale explicit account IDs. In current main, `delivery.accountId` is treated as a static override; when that account no longer exists for the selected channel, cron delivery should recover only when exactly one eligible configured/enabled account exists for the same channel, then use that account for delivery and trace output. Preserve valid explicit account IDs, preserve session-derived and binding-derived precedence when no explicit account is set, and do not add a WeChat-specific branch or mutate persisted jobs on Gateway startup. If multiple accounts or no eligible account exists, fail with an actionable diagnostic that names the stale account and points the user to edit the cron job or re-login/configure the channel. Use existing plugin/channel account APIs where possible; stop for maintainer review if a new public SDK seam is required. Add regression tests for stale explicit account recovery, ambiguous multi-account refusal, valid explicit account preservation, and existing binding/session fallback behavior. Add a CHANGELOG entry because this is user-facing cron delivery behavior.
