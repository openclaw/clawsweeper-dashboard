## Summary

Keep open. Current main still treats cron delivery accountId as a static explicit account identity, and the code/test/docs surfaces do not implement an `auto`/`active` selector, single-active-account fallback, or stale-account warning/recovery path for rotating channel account IDs.

## What This Changes

Cron delivery should support resolving or recovering stale channel account IDs at delivery time for channels such as `openclaw-weixin` whose session-derived account IDs can rotate after login or Gateway restart.

## Best Possible Solution

Implement generic delivery-time stale-account recovery in cron: keep explicit account IDs when they still resolve, recover only when the stored account is stale and exactly one eligible configured/active account exists for the same channel, and otherwise fail with a clear actionable error. Document the behavior and keep WeChat-specific login/session policy inside the plugin.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: configure an isolated cron announce with a stale `delivery.accountId`; current main threads that value through `resolveCronDeliveryPlan`, `resolveDeliveryTarget`, and `deliverOutboundPayloads` without validating or recovering it.

## Solution Assessment

Yes, if implemented generically and conservatively. Delivery-time stale-account recovery with strict ambiguity handling is narrower and safer than mutating job configs on startup or adding a WeChat-specific branch in core.
