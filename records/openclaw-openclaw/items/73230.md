## Summary

Keep open. Current main still treats cron delivery accountId as a static explicit channel account identity; it does not implement an auto/active selector, single-account recovery for stale IDs, or a stale-account warning path. The related merged account-spoof guard is useful adjacent context, but it preserves account-boundary checks rather than resolving rotating account IDs.

## What This Changes

Cron delivery should resolve or recover channel account IDs dynamically for channels whose configured account identity changes after login, restart, or token refresh.

## Best Possible Solution

Implement generic delivery-time stale-account recovery in cron: preserve explicit account IDs while they still resolve, recover only when the stored account is stale and exactly one eligible configured or active account exists for the same channel, and otherwise fail with a clear actionable delivery-target error. Add focused tests and cron docs so the behavior is predictable for all channel plugins.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: configure isolated cron announce with a stale delivery.accountId; current main threads that value through resolveCronDeliveryPlan, resolveDeliveryTarget, dispatchCronDelivery, and deliverOutboundPayloads without validating or recovering it.

## Solution Assessment

Yes, if solved generically and conservatively. Delivery-time stale-account recovery with strict ambiguity handling is narrower and safer than mutating job configs on startup or adding an openclaw-weixin-specific branch in core.
