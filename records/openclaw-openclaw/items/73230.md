## Summary

Keep open. Current main still treats cron delivery `accountId` as a static persisted identity, and explicit job delivery account IDs take precedence over session- and binding-derived fallbacks. The related merged account-spoof guard and the closed WeChat delivery symptom are useful context, but neither implements stale-account recovery for rotating channel account IDs.

## What This Changes

The issue requests dynamic or fallback `accountId` resolution for cron announce delivery when a channel plugin rotates account IDs, instead of leaving jobs pinned to stale static values.

## Best Possible Solution

Implement conservative generic delivery-time account recovery for cron announce delivery: keep valid explicit account IDs, recover only when the stored account cannot be resolved and exactly one eligible enabled/configured account exists for the same channel, and otherwise fail loudly with repair guidance. Cover the behavior in cron delivery target tests and update user-facing cron docs if the visible contract changes.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: create an isolated cron announce job with a stale `delivery.accountId`; current main threads that value through delivery planning, target resolution, and outbound delivery without validating or recovering it.

## Solution Assessment

Yes, a narrow generic delivery-time recovery path is the best direction. It is safer than mutating cron job configs on Gateway startup and avoids a WeChat-specific branch while preserving explicit-account and spoof-guard semantics.
