## Summary

Keep open. Current main still treats cron `delivery.accountId` as a static explicit route, and the related account-validation PR plus Weixin cron issues do not implement dynamic active-account resolution after relogin or Gateway restart.

## What This Changes

Request dynamic cron delivery account resolution or diagnostics so jobs targeting channels with rotating account IDs keep delivering after relogin or Gateway restart.

## Best Possible Solution

Add a channel-agnostic, preferably opt-in delivery-time account resolver that keeps valid explicit routes, falls back only when exactly one eligible same-channel account is available, and otherwise fails with an actionable diagnostic.

## Reproduction Assessment

Yes, at source level: configure an isolated cron announce job with a stale explicit `delivery.accountId`, and current main threads that string through planning, target resolution, and outbound delivery without validating or replacing it. I did not establish a live Weixin credential reproduction in this read-only pass.

## Solution Assessment

Unclear until maintainers choose the generic account freshness policy. A resolver-level recovery path is preferable to a Weixin-specific branch or startup mutation of persisted cron jobs, but it needs explicit semantics for multiple accounts.
