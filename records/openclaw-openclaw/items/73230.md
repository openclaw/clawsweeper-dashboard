## Summary

Keep open. Current main still treats cron delivery `accountId` as a static explicit identity: it preserves the configured value through delivery-plan resolution, target resolution, and outbound sending, and there is no `auto`/`active` selector, stale-account recovery, or startup warning for rotated channel accounts. The related merged spoof guard remains useful context but does not solve this dynamic resolution bug.

## What This Changes

Support dynamic or stale-safe cron delivery account resolution for channel plugins whose account IDs can rotate after restart or re-login, such as `openclaw-weixin`.

## Best Possible Solution

Implement generic delivery-time stale-account recovery in cron: preserve explicit account IDs when they still resolve, recover only under a same-channel single-eligible-account condition, and fail with an actionable diagnostic when recovery would be ambiguous. Cover the resolver, CLI/API schema, preview/update behavior, docs, and the existing spoof-guard path.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: configure isolated cron announce delivery with a stale `delivery.accountId`; current main threads that value through `resolveCronDeliveryPlan`, `resolveDeliveryTarget`, and `deliverOutboundPayloads` without validating or recovering it.

## Solution Assessment

Yes, if solved generically and conservatively. Delivery-time stale-account recovery with strict ambiguity handling is narrower and safer than mutating job configs on Gateway startup or adding a WeChat-specific branch in core.
