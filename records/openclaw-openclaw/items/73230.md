## Summary

Keep open. Current main still persists and forwards cron delivery `accountId` as a static account identity, while the issue asks for dynamic recovery when a channel plugin rotates account IDs. The related WeChat delivery symptom and merged account-spoof guard are useful context, but neither implements stale-account recovery for cron delivery.

## What This Changes

Cron delivery should resolve or recover the current account for channels whose account IDs can change after relogin or Gateway restart, instead of requiring users to manually edit every job.

## Best Possible Solution

Implement conservative generic delivery-time stale-account recovery for cron announce delivery: keep valid explicit account IDs, recover only when the stored account no longer resolves and exactly one eligible active or configured account exists for the same channel, and otherwise fail with an actionable error. Add focused cron tests and docs for the predictable multi-account behavior.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: configure an isolated cron announce with a stale delivery.accountId; current main threads that value through resolveCronDeliveryPlan, resolveDeliveryTarget, and deliverOutboundPayloads without validating or recovering it.

## Solution Assessment

Yes, if implemented generically and narrowly. Delivery-time stale-account recovery with strict ambiguity handling is safer than mutating job configs on Gateway startup or adding a WeChat-specific branch in core.
