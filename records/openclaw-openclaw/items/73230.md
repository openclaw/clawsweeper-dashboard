## Summary

Keep open. Current main still treats cron delivery `accountId` as a static explicit account identity and does not implement an `auto`/`active` selector, single-active-account recovery, or stale-account warning path. The related merged account-spoof guard is useful adjacent context, and the closed WeChat delivery-symptom issue correctly leaves this dynamic account-resolution work as the remaining core-side item.

## What This Changes

Support dynamic or stale-safe cron delivery account resolution for channel plugins whose account IDs can rotate after restart or re-login, such as `openclaw-weixin`.

## Best Possible Solution

Implement generic delivery-time stale-account recovery in cron: keep explicit account IDs when they still resolve, recover only when the stored account is stale and exactly one eligible configured or active account exists for the same channel, and otherwise fail with a clear actionable error. Add focused tests and docs so the behavior is predictable for all channel plugins.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: configure an isolated cron announce with a stale `delivery.accountId`; current main threads that value through `resolveCronDeliveryPlan`, `resolveDeliveryTarget`, and `deliverOutboundPayloads` without validating or recovering it.

## Solution Assessment

Yes, if solved generically and conservatively. Delivery-time stale-account recovery with strict ambiguity handling is narrower and safer than mutating job configs on startup or adding a WeChat-specific branch in core.
