Keep open: current main still matches the reported safeguard loop surface, and the latest release tag carries the same empty-preparation behavior. The issue is valid and narrow enough for a focused repair PR.

## What This Changes

The issue requests a fix for safeguard-mode compaction sessions that repeatedly retrigger `session_before_compact` after an empty preparation writes a boundary marker.

## Best Possible Solution

Add scoped idempotent suppression for repeated empty safeguard preparations, or make the trigger honor the prior boundary, with regression coverage for repeated empty preparations before any new real entries.

## Reproduction Assessment

No high-confidence live reproduction was established. The report provides deterministic after-state logs, and current source plus tests still expose the same repeated empty-preparation path, so this is source-reproducible but not fully exercised end to end here.

## Solution Assessment

Yes, the requested direction is the right class of fix. The narrowest maintainable path is to suppress repeated evaluation of the same empty boundary or make boundary state authoritative, not to change the default compaction mode or add an operator watchdog.
