## Summary

Keep open: current `main` still has the safeguard empty-preparation path that writes a boundary result without an OpenClaw-owned idempotency guard for the same empty prefix, and current coverage still allows repeated empty preparations to write repeated boundaries. The source path matches the reporter's logs closely, but I did not establish the corrupted session state end to end.

## What This Changes

The issue reports that safeguard compaction can re-trigger indefinitely for broken sessions with no real conversation messages, blocking affected session keys until manual quarantine.

## Best Possible Solution

Add a scoped idempotency/suppression point for repeated empty safeguard preparations so the same session boundary is not re-evaluated until new real conversation entries arrive, while preserving normal compaction for real user, assistant, or tool-result content.

## Reproduction Assessment

No, not as a live end-to-end reproduction. The current source and tests expose the same repeated empty-preparation behavior described in the logs, but the corrupted/broken journal precondition was not constructed locally.

## Solution Assessment

Yes, a narrow idempotency guard in the compaction safeguard or embedded-runner compaction boundary path is the right direction. Changing the default compaction mode or documenting the quarantine workaround would leave the existing runtime loop possible.
