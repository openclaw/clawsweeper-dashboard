## Summary

Keep open: current main and v2026.6.6 have fixed the built-in agent runtime model/tool content path, but the remaining Codex and Claude CLI harness tool spans still emit without private tool content, and that parity needs an explicit maintainer/security decision.

## What This Changes

The issue asks OpenClaw to make `diagnostics.otel.captureContent.*` populate OTel span content, or clearly document/deprecate the no-op behavior.

## Best Possible Solution

Keep the trusted private-data implementation for the built-in runtime and decide whether Codex and Claude CLI should gain the same gated private-content path or remain documented as unsupported.

## Reproduction Assessment

Yes. Source inspection shows the built-in runtime path now emits private tool content, while Codex and Claude CLI tool diagnostics still emit terminal events without `privateData.toolContent`; I did not run a live OTLP export.

## Solution Assessment

No, not as a close. The current implementation is the right shape for the built-in runtime, but the remaining external-harness parity needs a maintainer/security decision on support and privacy boundaries.
