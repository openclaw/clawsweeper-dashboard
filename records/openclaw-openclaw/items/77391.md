## Summary

Keep open. Current main still exposes the documented/exporter side of OTEL content capture, but the runtime model and tool diagnostic events do not populate the content fields, and an open linked PR is already using closing syntax for this issue.

## What This Changes

The issue reports that documented `diagnostics.otel.captureContent.*` options do not populate model/tool content attributes because runtime diagnostic events only carry metadata or sanitized summaries.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/79691 for model content capture, then implement or explicitly split the remaining tool input/output capture before closing this issue.

## Reproduction Assessment

Yes. It is source-reproducible on current main: docs and the OTEL exporter expect content fields, but the model/tool diagnostic event contracts and emitters do not populate them.

## Solution Assessment

Yes for the active direction. Policy-gated runtime population at the diagnostic event seam is the narrow maintainable fix, but the broader tool-content portion should be handled before this issue is closed.
