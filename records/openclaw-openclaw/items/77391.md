## Summary

Keep open: current main still has the documented `diagnostics.otel.captureContent.*` contract, but production model/tool diagnostic events do not carry the content fields that `diagnostics-otel` reads, so the reported no-op is valid from source inspection.

## What This Changes

The issue asks for documented `diagnostics.otel.captureContent` flags to emit bounded, redacted model/tool content into OTel spans or be documented as unsupported.

## Best Possible Solution

Make the documented opt-in content capture work through a privacy-preserving internal path with regression coverage, or deliberately retire/deprecate the config/docs if maintainers decide OpenClaw should not export this content.

## Reproduction Assessment

Yes, source-level: docs and config expose the flags, but current main's production model/tool emitters never provide the content fields consumed by `diagnostics-otel`. I did not run a live OTLP collector in this read-only sweep.

## Solution Assessment

Unclear as an implementation direction: wiring content directly onto all diagnostic events is not necessarily the safest fix. An owner should choose the narrowest content path that satisfies the documented OTel contract without broadening raw-content access.
