## Summary

Keep open: current main still documents and parses `captureContent`, and `diagnostics-otel` can attach content attributes when synthetic fields are present, but the production model/tool diagnostic event types and emitters do not carry those fields.

## What This Changes

The issue asks for the documented `diagnostics.otel.captureContent.*` settings to emit bounded raw model and tool content into OpenTelemetry spans instead of only sanitized summaries.

## Best Possible Solution

Make the documented opt-in capture path real without broadening default telemetry, ideally by feeding `diagnostics-otel` from content-bearing runtime hooks or gated event fields and adding regression coverage for the production emitters.

## Reproduction Assessment

Yes by source inspection: current main's production model/tool diagnostic events do not include the content fields that `diagnostics-otel` reads, so enabling the flags cannot populate those attributes. I did not run a live collector reproduction in this read-only environment.

## Solution Assessment

Not applicable as a patch because this is an issue, but the narrow maintainable fix is to honor the existing documented flags while keeping sanitized telemetry as the default. Deprecating the config would contradict the current docs and tests unless maintainers intentionally reverse that contract.
