## Summary

Keep open. Current main still has the documented/exporter side of OTEL content capture, but the runtime diagnostic events do not populate the model or tool content fields, and an active linked implementation PR is using closing syntax for this issue.

## What This Changes

The issue requests that `diagnostics.otel.captureContent.*` actually populate raw model/tool content attributes in OTEL spans, or that the documented knob be deprecated as a no-op.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/79691 for policy-gated model input/output capture, then either implement or split the remaining tool content-capture gap before closing this issue.

## Reproduction Assessment

Yes, source-reproducible on current main: docs and the OTEL exporter expect content fields, but the model/tool diagnostic event contracts and emitters do not populate them. I did not run a live OTLP backend for this read-only review.

## Solution Assessment

Yes for the model input/output portion: the linked PR's policy-gated runtime population is the narrow maintainable path. The tool input/output remainder should be added there only if intended for that PR, otherwise split to a follow-up before closing this issue.
