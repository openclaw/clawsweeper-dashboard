## Summary

Keep open: current `main` has a real docs-backed behavior gap. The OTel plugin can add `openclaw.content.*` only when diagnostic events already carry content fields, but the production model/tool diagnostic event types and emitters currently send size/timing metadata or `paramsSummary` rather than raw input/output/tool content.

## What This Changes

The issue requests making `diagnostics.otel.captureContent.*` populate documented raw-content OTel span attributes, or documenting the flags as deprecated/no-op.

## Best Possible Solution

Honor the documented config with a trusted, opt-in, bounded/redacted content path and regression coverage, or explicitly deprecate/remove the content-capture promise if maintainers do not want bundled OTel to export content.

## Reproduction Assessment

Yes from source inspection: the production model/tool diagnostic emitters do not populate the fields that `diagnostics-otel` reads for content attributes. I did not run a live OTLP collector, so this is source-reproducible rather than end-to-end reproduced.

## Solution Assessment

Unclear for automation: wiring the documented flags through is the narrow behavior fix, but maintainers should choose the privacy boundary before raw prompt/tool data is added to internal diagnostics. A docs-only deprecation is the safer alternative if content export is no longer desired.
