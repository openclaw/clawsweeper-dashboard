## Summary

Keep open: current main still has the documented config and the diagnostics-otel mapper, but the trusted runtime events do not carry the raw content fields the mapper needs. This is a source-reproducible bug, not a duplicate of #77390, which tracks the separate external-install capability gate.

## What This Changes

The issue asks for documented `diagnostics.otel.captureContent` settings to populate bounded, redacted raw model/tool content attributes in `diagnostics-otel` spans.

## Best Possible Solution

Wire the existing opt-in config to real trusted model/tool diagnostic payloads with bounded redaction and regression coverage that exercises runtime emission, while leaving the external-install trust issue tracked separately in #77390.

## Reproduction Assessment

Yes: source inspection gives a high-confidence path. The plugin maps `openclaw.content.*` only from raw event fields, but current trusted model/tool emitters and diagnostic event types do not include those fields.

## Solution Assessment

Yes, option (a) is the maintainable direction because docs and config already define the behavior. The safer shape is to keep capture disabled by default, add bounded/redacted fields only on trusted opt-in diagnostics, and test real runtime emissions rather than synthetic plugin-only events.
