## Summary

Keep open: current main still accepts and documents the `diagnostics.otel.captureContent.*` knobs, and `diagnostics-otel` looks for raw content fields, but the runtime diagnostic events emitted by model and tool paths do not carry those fields.

## What This Changes

The issue asks OpenClaw to make `diagnostics.otel.captureContent.*` populate raw model/tool content attributes or document the knobs as deprecated/no-op.

## Best Possible Solution

Preserve the documented opt-in feature by wiring the existing capture policy to safe model/tool content emission and regression coverage, while keeping #77390 as the separate trust-boundary follow-up for external installs.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: enable capture, run a model/tool turn, and the runtime emits diagnostic events without the raw fields that `diagnostics-otel` reads for `openclaw.content.*` attributes.

## Solution Assessment

Yes, implementing the documented opt-in content path is the best fix; documenting the knobs as no-op would contradict the existing config contract and tests unless maintainers intentionally retire the feature.
