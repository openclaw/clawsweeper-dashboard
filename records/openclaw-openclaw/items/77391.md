## Summary

Keep open. Current main still documents and schemas `diagnostics.otel.captureContent.*`, and the OTel plugin can attach `openclaw.content.*` only if event payloads contain raw content, but the model/tool diagnostic event contracts and emitters on the runtime path carry sizes or `paramsSummary` instead. The linked tutorclaw tracing workaround is consistent with the public hook docs, but it does not make the bundled OTel knob functional.

## What This Changes

The issue requests that documented `diagnostics.otel.captureContent.*` settings actually emit bounded, redacted model and tool content on OTel spans, or be explicitly deprecated as unsupported.

## Best Possible Solution

Implement the documented opt-in by feeding bounded, redacted content into diagnostics-otel only when `diagnostics.otel.captureContent.*` is enabled, preferably through a narrow internal path that does not turn public diagnostic events into raw-content broadcasts.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current docs/config promise content attributes, but the runtime diagnostic event types and emitters do not carry the fields that diagnostics-otel reads. I did not run a live OTLP export in this read-only pass.

## Solution Assessment

Yes, fixing the existing capture path is the best direction because the config and docs already define the behavior. The safer implementation is a gated, redacted OTel/internal capture path, not unconditional raw-content fields on every diagnostic event.
