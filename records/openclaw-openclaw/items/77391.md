## Summary

Keep open. Current main still documents and exports OTEL content attributes when event fields exist, but the model/tool runtime diagnostic events do not populate those content fields; the linked closing PR is still open and only covers the model input/output portion.

## What This Changes

The issue reports that documented `diagnostics.otel.captureContent.*` flags do not populate raw model/tool content attributes because runtime diagnostic events emit only sanitized summaries.

## Best Possible Solution

Review and land or supersede https://github.com/openclaw/openclaw/pull/79691 for model content capture, then implement tool input/output capture or split that remainder to a narrower canonical follow-up before closing this issue.

## Reproduction Assessment

Yes. Source inspection on current main shows the docs and OTEL exporter expect content fields, while the model and tool runtime diagnostic emitters do not populate them.

## Solution Assessment

Yes for the active model-content direction. Runtime population at the diagnostic event seam behind the existing capture-content policy is the narrow maintainable fix, but the tool-content portion should be handled before this issue is closed.
