## Summary

Keep open: current main still has a source-level mismatch where `diagnostics.otel.captureContent.*` is documented and consumed by `diagnostics-otel`, but the production model/tool diagnostic events do not carry the raw content fields needed to populate the span attributes.

## What This Changes

The issue asks for documented `diagnostics.otel.captureContent.*` flags to emit bounded redacted model/tool content attributes, or for the docs to stop promising that behavior.

## Best Possible Solution

Make the docs and runtime agree: either implement a gated, redacted content-capture path used only when `diagnostics.otel.captureContent.*` is enabled, or explicitly deprecate that OTel content promise and point operators to the raw-content hook pattern.

## Reproduction Assessment

Yes from source inspection, but not from a live collector run: the documented capture flags are consumed downstream, while current production diagnostic events do not carry the content fields required to produce those attributes.

## Solution Assessment

Unclear: the report identifies a real contract mismatch, but the maintainable fix needs a maintainer decision between a tightly gated diagnostics content path and a docs/API correction that keeps diagnostics metadata-only.
