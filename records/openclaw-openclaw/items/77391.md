## Summary

Keep open: current main fixes the built-in model and core tool OTel content path through gated private diagnostic payloads, but Codex and Claude CLI tool spans still emit without content payloads, so the remaining issue is a maintainer/security decision rather than conservative cleanup.

## What This Changes

The issue asks for `diagnostics.otel.captureContent.*` to populate OTel model/tool content attributes instead of exporting only sanitized summaries.

## Best Possible Solution

Keep the built-in private gated content path and decide whether to extend the same private-data design to Codex/Claude tool events or keep the documented limitation.

## Reproduction Assessment

Yes for the remaining current-main gap: source shows Codex and Claude CLI tool diagnostics emit without `privateData.toolContent`, while the OTel exporter only exports tool content from private data.

## Solution Assessment

No close yet: the current built-in implementation is the right privacy shape, but external harness parity requires an explicit maintainer/security decision before implementation or closure.
