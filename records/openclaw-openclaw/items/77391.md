## Summary

This should stay open: current main documents and parses `diagnostics.otel.captureContent`, and the OTel exporter can consume content fields, but the runtime diagnostic events that feed it do not carry the raw model or tool content the exporter expects.

## What This Changes

The issue asks for documented `diagnostics.otel.captureContent.*` settings to populate raw model and tool content span attributes, or for the no-op behavior to be documented/deprecated.

## Best Possible Solution

Honor the documented contract by threading opt-in, redacted, bounded content into trusted runtime diagnostic events and the OTel exporter, with regression coverage for model input/output and tool input/output.

## Reproduction Assessment

Yes, source-reproducible: current main's runtime diagnostic events do not carry the raw fields that `diagnostics-otel` reads for content attributes. I did not mark it live-reproduced because no gateway/collector run was performed in this read-only review.

## Solution Assessment

Yes for keeping this open: the public-hook workaround is useful evidence, but the maintainable fix is to make the existing bundled plugin config work rather than requiring every operator to ship a separate tracing plugin.
