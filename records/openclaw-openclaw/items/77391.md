## Summary

Current main still has a source-level gap: the docs and diagnostics-otel exporter support captureContent, but the real model/tool diagnostic events do not carry the raw content fields the exporter reads. The issue should stay open; the related external-install no-spans problem is separate from this content-emission bug.

## What This Changes

The issue asks for documented diagnostics.otel.captureContent.* settings to populate bounded, redacted model/tool content span attributes, or for the no-op behavior to be documented instead.

## Best Possible Solution

Wire opt-in content through the runtime diagnostic event types and model/tool emitters, then add regression coverage proving disabled capture stays content-free and enabled capture reaches diagnostics-otel redacted and bounded.

## Reproduction Assessment

Yes: source inspection gives a high-confidence reproduction path because current runtime emitters never include the content fields that diagnostics-otel reads. I did not run a live OTLP trace in this read-only review.

## Solution Assessment

Yes: fixing the runtime diagnostic event emission is the narrowest maintainable solution for the documented contract. Deprecating the config would leave an existing user-facing capability non-functional.
