## Summary

Keep open: current main and the latest release have a model-content path for `inputMessages`/`outputMessages`, but the inspected tool execution producers still emit only `paramsSummary`, so the documented `toolInputs`/`toolOutputs` part of the report remains real. The open related PR only covers model-call content and explicitly leaves tool content out of scope.

## What This Changes

The issue reports that documented `diagnostics.otel.captureContent.*` settings do not populate raw model and tool content in exported OTel spans.

## Best Possible Solution

Keep this issue open as the canonical tracking item for the remaining tool input/output capture gap, and implement it with the same explicit opt-in, bounded, redacted, trusted/private-data boundary used for model content.

## Reproduction Assessment

Yes, source-reproducible: current main’s core tool wrapper emits `paramsSummary` on tool execution events and the diagnostics-otel plugin reads `event.toolInput`/`event.toolOutput`, so enabled `toolInputs`/`toolOutputs` still have no producer. I did not run a live OTLP export in this read-only review.

## Solution Assessment

No, the issue is not fully solved by current main or the open related PR because both leave tool content capture unresolved. The best fix is not to add raw fields to public events, but to extend the trusted/private diagnostic content path for tool inputs and outputs under the existing opt-in policy.
