## Summary

Keep open: current main still has a source-reproducible gap between the documented OTel captureContent contract and the production diagnostic events that diagnostics-otel consumes.

## What This Changes

The issue asks OpenClaw to make diagnostics.otel.captureContent input/output/tool flags populate raw-content OTel span attributes, or document those flags as a no-op.

## Best Possible Solution

Keep the issue open and either wire a tightly scoped, redacted content-capture path into diagnostics-otel or deliberately deprecate the documented captureContent contract.

## Reproduction Assessment

Yes. Source inspection on current main shows the exporter reads raw-content fields that production diagnostic event types and emitters never supply.

## Solution Assessment

Unclear. The documented behavior should be honored, but maintainers should choose the safest boundary before moving raw content through internal diagnostics or plugin hooks.
