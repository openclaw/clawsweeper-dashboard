## Summary

Keep open: current main still has only a partial implementation. The docs/config promise opt-in content attributes, and diagnostics-otel can attach them if synthetic event fields exist, but the real model/tool diagnostic emitters do not populate those fields. The linked tutorclaw tracing plugin is useful evidence that public hooks can observe the missing content, but it does not make the bundled plugin's captureContent path work.

## What This Changes

The issue reports that documented diagnostics.otel.captureContent flags do not produce raw OTel content attributes because runtime diagnostic events only carry sanitized summaries.

## Best Possible Solution

Keep the documented opt-in behavior, but wire diagnostics-otel to a deliberately gated raw-content source, likely via plugin hook observations or explicit content-bearing diagnostic events, and add regression coverage that exercises real runtime emitters rather than synthetic event casts.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: enable captureContent, then the real model/tool diagnostic events still lack the fields diagnostics-otel reads for openclaw.content.* attributes. I did not need a live collector to see the broken data path.

## Solution Assessment

No, the current implementation is not sufficient because it only attaches content when tests inject non-contract fields into diagnostic events. The maintainable fix should preserve the sanitized default stream and introduce a narrow opt-in content path for the bundled exporter.
