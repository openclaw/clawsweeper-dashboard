## Summary

Keep open: current main still has a real contract gap where the documented OTel content-capture flags exist, but production diagnostic events do not carry the raw fields that the diagnostics-otel plugin tries to export.

## What This Changes

The issue asks OpenClaw to make documented `diagnostics.otel.captureContent.*` settings emit bounded raw model and tool content attributes, or to document the settings as deprecated/no-op.

## Best Possible Solution

Keep the issue open and fix the existing documented contract by adding an opt-in, bounded, redacted content path from production model/tool emitters into diagnostics-otel, with tests proving the real emitter path supplies the fields.

## Reproduction Assessment

Yes from source inspection: current main's production model/tool diagnostic emitters do not include the raw fields that diagnostics-otel reads, so enabled captureContent flags have no content to attach. I did not run a live OTLP collector in this read-only review.

## Solution Assessment

No patch is present to assess. Wiring the existing documented flags is the narrowest user-facing fix, but maintainers should review the raw-content diagnostics boundary before implementation.
