## Summary

Keep open. Current main still has a source-reproducible gap between the documented OTEL content-capture contract and the runtime diagnostic events, and the linked implementation PR remains open with closing syntax for this issue.

## What This Changes

The issue asks for documented `diagnostics.otel.captureContent.*` flags to populate raw model/tool content on OTEL spans, or for the no-op behavior to be documented.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/79691 for model content capture, then implement or explicitly split the remaining tool input/output capture before closing this issue.

## Reproduction Assessment

Yes. It is source-reproducible on current main: docs, config, and the OTEL exporter expect content fields, while the model and tool runtime diagnostic events do not populate them.

## Solution Assessment

Yes for the active direction. Policy-gated population at the diagnostic event seam is the narrow maintainable fix, but the issue should not close until the tool-content portion is fixed or deliberately split.
