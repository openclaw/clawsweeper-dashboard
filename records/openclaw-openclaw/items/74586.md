## Summary

Keep open. Current main has an adjacent Active Memory memory-tool allowlist fix, but the reported embedded-run timeout handoff, completed-`NONE` timeout classification, and persisted `openclaw:prompt-error` leakage still match current source behavior, and the issue has strong persisted-transcript evidence plus an independent confirmation.

## What This Changes

The issue requests a fix for Active Memory embedded recall runs where embedded `memory_search` calls are aborted, completed empty recall is still reported as a timeout, and prior prompt-error metadata leaks into later recall context.

## Best Possible Solution

Keep Active Memory bounded by a hard outer deadline, but make the embedded recall runner and tool dispatch share the intended effective budget, classify a completed or recovered `NONE` as empty rather than timeout, and prevent `openclaw:prompt-error` diagnostic entries from entering later LLM-boundary context.

## Reproduction Assessment

Yes. The issue provides a high-confidence reproduction path using Active Memory with persisted transcripts and a direct `memory_search` control, and a second user independently confirmed the same pattern on a later release.

## Solution Assessment

Unclear because there is no patch on this issue. The narrow maintainable direction is a focused Active Memory plus embedded-runner lifecycle fix; model tuning or a generic external timeout-doctor workaround would not address the source-localized abort and prompt-error paths.
