## Summary

Keep open. Current main still has source paths matching the reported Active Memory embedded-run abort, timeout classification, and prompt-error carryover behavior, and the discussion includes both a persisted-transcript report and an independent confirmation. Open PR #74480 is related, but it covers only part of the timeout-budget handoff rather than the full abort, completed-`NONE`, and prompt-error leakage report.

## What This Changes

The issue asks OpenClaw to fix Active Memory embedded recall so `memory_search` tools are not aborted prematurely, completed empty/`NONE` runs are classified accurately, and stale prompt-error events do not leak into later recall context.

## Best Possible Solution

Keep Active Memory bounded by a hard outer deadline, but separate deadline enforcement from ordinary memory tool dispatch until the deadline actually wins; classify completed empty or `NONE` recall as `empty`; filter prior `openclaw:prompt-error` custom entries out of future LLM-boundary context; and cover those paths with focused Active Memory plus embedded-runner tests.

## Reproduction Assessment

Yes. The issue provides a high-confidence reproduction path using Active Memory with persisted transcripts plus a direct `memory_search` control, and a second user independently confirmed the same behavior on v2026.4.27.

## Solution Assessment

Unclear because there is no proposed patch on this issue. The narrow maintainable direction is in Active Memory embedded recall and the embedded-runner abort/prompt lifecycle, not in model tuning or a generic timeout-doctor workaround.
