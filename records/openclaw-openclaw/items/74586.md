## Summary

Keep open: current main has adjacent Active Memory fixes, but the full reported failure is still not resolved. The source still passes the Active Memory watchdog abort signal into embedded memory-tool execution, still maps empty recovered output such as `NONE` on the timeout path to `status: "timeout"`, and still preserves `openclaw:prompt-error` custom entries outside the current LLM-boundary filter; #74480 only covers the setup-grace handoff and is still open.

## What This Changes

Active Memory embedded recall should stop aborting `memory_search` calls, avoid reporting successful empty embedded completions as timeouts, and prevent prior prompt-error diagnostics from leaking into later recall context.

## Best Possible Solution

Keep this issue as the canonical Active Memory embedded recall abort/classification bug and land one focused repair that preserves the hard deadline while separating successful empty recall from true timeout and filtering stale prompt-error diagnostics.

## Reproduction Assessment

Yes. The issue has persisted transcripts plus an independent confirmation, and current main still exposes the matching abort propagation, timeout classification, and prompt-error persistence paths, though I did not run a live provider reproduction in this read-only pass.

## Solution Assessment

No patch is attached, and #74480 is not the full solution because it only handles setup-grace timeout budgeting. The narrow maintainable fix belongs in `extensions/active-memory` plus the embedded-runner abort/prompt-error lifecycle where the shared behavior is generic.
