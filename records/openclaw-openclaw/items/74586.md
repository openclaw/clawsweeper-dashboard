## Summary

Keep open: current main still has the Active Memory embedded recall paths that match the reported abort, timeout classification, and prompt-error carryover symptoms. The discussion includes a strong persisted-transcript reproduction and an independent confirmation, while related PR #74480 is open and intentionally narrower than the remaining tool-abort and prompt-error lifecycle work.

## What This Changes

Active Memory should stop aborting embedded `memory_search` calls, classify completed `NONE` recalls as empty instead of timeout, and prevent prior `openclaw:prompt-error` events from leaking into later embedded recall context.

## Best Possible Solution

Keep Active Memory bounded by one effective outer deadline, but ensure ordinary embedded memory tools are not handed an already-fired abort signal before that deadline wins; when the embedded recall returns `NONE` or empty output before the deadline, persist an empty result rather than a timeout; and filter or expire `openclaw:prompt-error` custom entries before they can enter later model context.

## Reproduction Assessment

Yes. The issue provides a high-confidence persisted-transcript reproduction plus a direct `memory_search` control on the same host, and a second user independently confirmed the same direct-search-versus-Active-Memory failure pattern.

## Solution Assessment

Unclear because this issue has no proposed patch. The narrow maintainable solution belongs in the Active Memory embedded recall wrapper plus the embedded-runner abort/custom-entry lifecycle, with PR #74480 treated as only the setup-grace slice if it lands first.
