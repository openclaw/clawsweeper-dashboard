## Summary

Keep open. Current main still contains the Active Memory embedded-run timeout and abort propagation shape described in the report, and the issue has strong transcript evidence plus an independent confirmation; the related open PR only covers part of the timeout-budget handoff.

## What This Changes

The issue requests fixing Active Memory embedded recall so `memory_search` calls are not prematurely aborted, completed `NONE` runs are not classified as timeouts, and prior `prompt-error` entries do not leak into later embedded-run context.

## Best Possible Solution

Keep Active Memory bounded by a hard outer deadline, but separate deadline enforcement from ordinary memory tool dispatch until the deadline actually wins; completed empty or `NONE` recall should classify as `empty`, and `openclaw:prompt-error` custom entries should be filtered from future LLM-boundary context with focused regression tests.

## Reproduction Assessment

Yes. The issue provides a high-confidence reproduction path using Active Memory persisted transcripts plus a direct `memory_search` control, and a second user independently confirmed the same embedded recall failure pattern on v2026.4.27.

## Solution Assessment

Unclear because there is no proposed patch on this issue. The maintainable fix should stay in Active Memory and the embedded-runner abort/prompt lifecycle rather than treating this as model tuning or a generic provider timeout problem.

Current main forwards the Active Memory watchdog `AbortSignal` into `runEmbeddedPiAgent` and memory tool wrappers, then maps recovered `NONE` through the timeout path to `status=timeout`; prompt-error custom entries can also re-enter later LLM-boundary context.
