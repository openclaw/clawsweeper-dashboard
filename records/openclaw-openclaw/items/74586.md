## Summary

Keep open: current main has a related memory-tool materialization fix, but the Active Memory embedded recall timeout and abort lifecycle described here is not fully fixed. The source still passes the raw configured timeout into the embedded runner while the outer watchdog uses `timeoutMs + setupGraceTimeoutMs`, the timeout recovery path still classifies empty or `NONE` output as `timeout`, and the prompt-error custom-entry path still exists without a visible Active Memory-specific lifecycle cleanup.

## What This Changes

The issue requests fixing Active Memory embedded recall so `memory_search` calls are not prematurely aborted, completed `NONE` recall is not classified as timeout, and prior prompt-error metadata does not leak into later embedded runs.

## Best Possible Solution

Keep this open until Active Memory uses one effective embedded recall deadline, memory tool dispatch cannot inherit stale or premature abort state before that deadline, completed empty or `NONE` recall is classified as `empty` rather than timeout, and prior `openclaw:prompt-error` metadata is prevented from entering later Active Memory LLM input. Preserve the already-merged #74592 memory-tool materialization behavior and treat #74480 as the canonical setup-grace slice if it lands first.

## Reproduction Assessment

Yes. The issue body gives a high-confidence reproduction path with persisted Active Memory transcripts plus a direct `memory_search` control, and a second user independently confirmed the same failure shape on OpenClaw 2026.4.27.

## Solution Assessment

Unclear because this issue has no proposed patch. The narrow maintainable direction is Active Memory-owned deadline/classification fixes plus a generic embedded-runner or tool-abort seam only if needed; #74480 is useful but only a partial solution.
