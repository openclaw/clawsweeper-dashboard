## Summary

Keep open: current main still contains the Active Memory embedded recall abort, timeout classification, and prompt-error carryover paths described in the report, and a second user independently confirmed the same failure pattern. Open PR #74480 is related but only covers the setup-grace timeout handoff, so it does not supersede the remaining `memory_search` abort and prompt-error lifecycle work.

## What This Changes

The issue reports that Active Memory embedded recall aborts `memory_search` tool calls, records timeout with zero summary even when the embedded model returns `NONE`, and carries prior prompt-error entries into later runs.

## Best Possible Solution

Keep Active Memory bounded by a hard outer deadline, but separate deadline enforcement from ordinary memory tool dispatch until the deadline actually wins; treat completed empty or `NONE` recall as `empty`; prevent prior `openclaw:prompt-error` custom entries from re-entering future LLM-boundary context; and add focused regression coverage for those paths.

## Reproduction Assessment

Yes. The report provides a high-confidence reproduction path with persisted Active Memory transcripts and a direct `memory_search` control, and an independent user confirmed the same behavior on v2026.4.27.

## Solution Assessment

Unclear because this issue has no proposed patch. The narrow maintainable solution is in Active Memory embedded recall plus the embedded-runner abort and prompt-error lifecycle; PR #74480 is useful but only solves part of the timeout-budget handoff.
