## Summary

Keep open. Current main still rejects Gemini `web_search` freshness/date filters before building a Gemini request, still sends an empty Google Search grounding tool, and still documents those filters as unsupported; the related native-passthrough issue is broader and does not supersede this narrow provider fix.

## What This Changes

Request Gemini-backed `web_search` to honor OpenClaw freshness/date filters instead of returning unsupported-filter errors.

## Best Possible Solution

Add a provider-scoped Gemini change that maps existing OpenClaw freshness/date inputs into Google Search grounding `timeRangeFilter`, updates cache-key normalization, revises Gemini/web-search docs, and adds focused tests for the supported behavior.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to invoke the Gemini `web_search` execution path with `freshness`, `date_after`, or `date_before`; current main returns the shared unsupported-filter response before any Gemini request is made.

## Solution Assessment

Yes. The narrowest maintainable fix is to map existing OpenClaw freshness/date fields to Gemini's time-range filter in the Google plugin, while leaving unrelated filters such as country, language, and domain out of scope.
