## Summary

Keep open. Current main still rejects Gemini `web_search` freshness/date filters before constructing a Gemini request, still sends an empty Google Search grounding tool, and the local Gemini docs still describe freshness/date filters as unsupported. The request is narrow and actionable because Google's current API reference documents `GoogleSearch.timeRangeFilter` with RFC3339 `startTime` and `endTime`. ([ai.google.dev](https://ai.google.dev/api/caching))

## What This Changes

The issue requests making Gemini-backed `web_search` respect freshness/date parameters by passing them to Google Search grounding instead of returning unsupported-filter errors.

## Best Possible Solution

Keep this issue open until the bundled Google plugin converts supported OpenClaw freshness/date inputs into Gemini Google Search grounding `timeRangeFilter` intervals, includes the normalized time range in the Gemini cache key, and updates focused tests plus Gemini/web-search docs. The fix should stay scoped to Gemini freshness/date handling rather than expanding into the broader native-search passthrough work tracked by #17925.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to call the Gemini `web_search` execution path with a non-empty `freshness`, `date_after`, or `date_before` value; current main returns the shared unsupported-filter response before credentials or any Gemini request are built.

## Solution Assessment

Yes. The narrowest maintainable solution is to update the Google plugin/runtime to map existing shared freshness/date inputs to Gemini's `GoogleSearch.timeRangeFilter`, while leaving unrelated filters like country, language, and domain unsupported unless a separate provider API supports them.
