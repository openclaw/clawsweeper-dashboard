## Label Justifications

- `P2`: This is a normal-priority CLI/session bug with a clear workaround but real impact for users with configured session stores.
- `impact:session-state`: The export command can miss stored session trajectory data when the session is visible through the configured session store but export looks elsewhere.

## Summary

Keep open: current main and v2026.6.9 still have the `sessions export-trajectory` store-resolution mismatch, and the linked open PR is the active candidate fix rather than a reason to close this issue early.

## What This Changes

The issue asks for `openclaw sessions export-trajectory` to use the shared session-store resolver for both implicit and explicit store paths.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/95570 or an equivalent focused fix, keeping this issue canonical until the resolver alignment reaches main.

## Reproduction Assessment

Yes. Source inspection on current main shows export without `--store` bypasses configured `session.store` while sibling session commands use the shared resolver; I did not run a live CLI repro because this review is read-only.

## Solution Assessment

Yes. The narrow maintainable solution is to reuse the existing runtime config and `resolveStorePath` contract in the export command while preserving explicit `--store` precedence.
