## Label Justifications

- `P2`: This is a normal-priority CLI/session bug with a clear workaround, limited blast radius, and real impact for configured session-store users.
- `impact:session-state`: The command can miss stored session trajectory data when session rows live in a configured session store rather than the default agent store.

## Summary

Keep open: current main and v2026.6.9 still resolve `sessions export-trajectory` without `--store` from the default agent session store, while the sibling sessions commands honor configured `session.store`; the open linked PR is the right candidate fix path.

## What This Changes

The issue requests that `openclaw sessions export-trajectory` use configured `session.store`, including `{agentId}` templating, when `--store` is omitted.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/95570 or an equivalent narrow fix that reuses the existing configured-store resolver while preserving explicit `--store` precedence.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: configure `session.store`, list a session through the sibling sessions path, then run export without `--store`; current main reads the default agent store instead.

## Solution Assessment

Yes. The best fix is narrow: resolve export's implicit store through the same config-backed `resolveStorePath` contract used by sibling session commands, while preserving explicit `--store` as the override.
