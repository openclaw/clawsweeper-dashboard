## Label Justifications

- `P2`: This is a normal CLI/session bug with a clear workaround and limited blast radius, but it affects real configured session-store users.
- `impact:session-state`: The command can miss stored session trajectory data when session rows live in the configured session store.

## Summary

Keep open: current main and the latest release still resolve `sessions export-trajectory`'s implicit store from the default agent sessions directory instead of configured `session.store`, and the linked fix is an open draft PR rather than a merged implementation.

## What This Changes

Request that `openclaw sessions export-trajectory` resolve its implicit store from configured `session.store`, including `{agentId}` templating, when `--store` is omitted.

## Best Possible Solution

Land or replace the linked PR with the narrow CLI change that makes export use configured `session.store`, preserves explicit `--store`, and updates help plus regression tests.

## Reproduction Assessment

Yes. Source inspection on current main shows export without `--store` uses the default store while listing uses configured `session.store`; the reporter also described a temporary configured-store repro. I did not rerun a live CLI command because this review is read-only.

## Solution Assessment

Yes. Resolving the implicit export store through the existing `resolveStorePath(getRuntimeConfig().session?.store, { agentId })` path is the narrow maintainable fix and keeps explicit `--store` as the override.
