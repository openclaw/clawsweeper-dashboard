## Label Justifications

- `P2`: This is a focused CLI/session-store bug with a workaround, but it can block exporting existing session trajectory data from configured stores.
- `impact:session-state`: The command can look in the wrong persisted session store when trajectory export should use configured session state.

## Summary

Keep open: current `main` and the latest release still resolve `sessions export-trajectory` stores differently from the rest of the sessions CLI, and https://github.com/openclaw/openclaw/pull/95570 is the open candidate fix that should land or be rejected before this issue closes.

## What This Changes

The issue requests that `openclaw sessions export-trajectory` use the shared session-store resolver for both omitted and explicit `--store` values.

## Best Possible Solution

Review and land or explicitly reject https://github.com/openclaw/openclaw/pull/95570, then close this issue once the resolver alignment is merged.

## Reproduction Assessment

Yes. Source inspection on current `main` shows `sessions export-trajectory` bypasses configured `session.store` and the shared resolver, while sibling sessions commands route through `resolveSessionStoreTargetsOrExit` and `resolveStorePath`.

## Solution Assessment

Yes. The narrow shared-resolver alignment in the linked PR is the best fix shape because it preserves explicit `--store` precedence while making implicit and templated paths match the existing sessions CLI contract.
