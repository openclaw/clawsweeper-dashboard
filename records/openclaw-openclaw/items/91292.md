## Summary

Keep this PR open. The patch targets a real provider-catalog planning bug: blank explicit provider base URLs can override bundled defaults before generated model registry filtering, and the proposed normalization is at the right boundary, but the config/routing semantic change is upgrade-sensitive and would benefit from focused committed regression coverage.

## What This Changes

The PR strips blank `models.providers.*.baseUrl` values before provider discovery and catalog merging so partial provider config does not clobber bundled provider defaults.

## Best Possible Solution

Treat blank provider `baseUrl` as omitted before catalog discovery and merge, preserve nonblank explicit overrides, and add focused regression coverage for generated Google catalog resolution plus a non-bundled blank-baseUrl provider that should not become writable.

## Reproduction Assessment

Yes. The current source path shows blank explicit `baseUrl` can flow into discovery/merge and then fail `filterWritableProviders`; the PR body also includes before/after output from the production `ensureOpenClawModelsJson` plus `resolveModelAsync` path.

## Solution Assessment

Yes, this is likely the best fix. Normalizing blank overlays before both provider discovery and explicit/implicit merge preserves the generated catalog at the config-planning boundary; a resolver-side fallback would leave `models.json` poisoned.
