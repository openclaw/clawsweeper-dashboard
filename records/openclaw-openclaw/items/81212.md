## Summary

Keep open: current main still hides legacy CLI runtime providers from the shared provider list, and Telegram/Mattermost do not expose an equivalent runtime-picker path. The fix direction is not a conservative cleanup close because runtime selection UX and authorization policy need maintainer judgment.

## What This Changes

Restore a discoverable click-through path in `/models` for CLI runtimes such as `claude-cli`, `codex-cli`, and `google-gemini-cli`.

## Best Possible Solution

Add a vetted runtime-selection step to channel model pickers using the existing `runtimeChoicesByProvider`, with authorization/availability validation preserved and legacy alias support kept as compatibility rather than fake provider policy.

## Reproduction Assessment

Yes, at source level: current main filters every legacy runtime provider before building `/models` providers, and Telegram renders that filtered list directly. I did not run a live Telegram bot, so this is source-reproducible rather than fully reproduced.

## Solution Assessment

Unclear: reverting the visibility filter is not obviously the best fix because the merged runtime-selection work intentionally split provider/model selection from runtime selection. A runtime selector is likely safer, but maintainers should decide the UX and authorization boundary before implementation.
