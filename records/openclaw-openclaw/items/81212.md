## Summary

Keep open: current main still filters legacy CLI runtime providers out of shared `/models` provider data, and Telegram/Mattermost render that filtered provider list without an equivalent runtime chooser. The related open PR fixes a different alias leak and explicitly defers the canonical runtime-selector UX, so this needs maintainer product/security direction rather than cleanup closure.

## What This Changes

Request to restore discoverable `/models` inline-button selection for CLI runtime-backed providers or add an equivalent runtime-selection path.

## Best Possible Solution

Add a vetted runtime-selection step to channel model pickers using the existing `runtimeChoicesByProvider`, while preserving runtime policy validation and treating legacy provider aliases as compatibility rather than first-class provider IDs.

## Reproduction Assessment

Yes, at source level: current main filters every legacy runtime provider before building `/models` providers, and Telegram/Mattermost render that filtered list directly. I did not run a live Telegram bot, so this is source-reproducible rather than fully reproduced.

## Solution Assessment

Unclear: reverting the visibility filter would restore the old click path but conflicts with the merged provider/runtime split and current runtime validation boundary. A channel runtime selector is likely the safer maintainable direction, but maintainers need to choose the UX and policy.
