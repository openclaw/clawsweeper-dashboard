## Summary

Keep open: current main and v2026.6.10 still route Google/Gemini text generation through generateContent/streamGenerateContent, while the pinned Google SDK and current Google documentation expose Interactions as a separate API surface. The request fits provider roadmap work, but it needs maintainer API/product direction before implementation because it changes transport selection, config/defaulting, streaming/tool handling, and server-side interaction state semantics.

## What This Changes

Request support for Google's Gemini Interactions API as an opt-in transport alongside the existing generateContent path.

## Best Possible Solution

Design a provider-local Google transport that can call Interactions for supported Gemini models and agents while preserving generateContent as the default until maintainers approve rollout, config shape, and state semantics.

## Reproduction Assessment

Yes, as a source-level missing-capability check: current main only registers generateContent/streamGenerateContent paths for Google text generation, and a repository search found no Interactions implementation. This is not a runtime bug reproduction for an existing documented command.

## Solution Assessment

Unclear as a complete implementation plan: a provider-local Google transport is the right owner boundary, but maintainers still need to choose the config/defaulting model and interaction-state contract before code should be queued.
