## Summary

Keep open: current main and the latest release still route Gemini text generation through generateContent/streamGenerateContent, while the pinned Google SDK and current Google docs expose Interactions as a separate API surface. The request is vision-aligned provider work, but it needs maintainer API/product direction before an automated fix because it changes transport selection, defaulting, and multi-turn state semantics.

## What This Changes

Request to add Google's Interactions API as a Gemini provider transport alongside the existing generateContent path.

## Best Possible Solution

Design a provider-local Google transport that can call Interactions for supported Gemini models/agents while preserving the current generateContent path until maintainers explicitly approve rollout and state-handling semantics.

## Reproduction Assessment

Yes, as a source-level missing-capability check: current Google provider registration and transport code route text generation through generateContent/streamGenerateContent, while the pinned SDK exposes Interactions APIs. This is not a runtime bug reproduction for an existing documented command.

## Solution Assessment

Unclear as a complete implementation plan: adding an opt-in Google-provider transport is the right boundary, but maintainers need to decide defaulting, previous-interaction state persistence, streaming/tool-call mapping, and Vertex/Developer API parity.
