## Summary

Keep open: current main still routes Gemini text generation through generateContent/streamGenerateContent, while Google's current docs and the pinned JS SDK expose Interactions as a separate API surface. This is valid provider roadmap work, but it needs maintainer product/API direction because it affects transport selection, opt-in configuration, streaming/tool event mapping, and server-side interaction state.

## What This Changes

The issue requests an opt-in Gemini Interactions API transport alongside the existing generateContent path.

## Best Possible Solution

Design a provider-local Google transport path that can call Interactions where appropriate while keeping existing generateContent behavior stable until maintainers approve rollout semantics.

## Reproduction Assessment

Yes, as a source-level missing-capability check: current Google provider paths register and call generateContent/streamGenerateContent, while current Google docs and the pinned SDK expose Interactions as a separate create/stream/state API. This is not a runtime bug reproduction for an existing documented command.

## Solution Assessment
