## Summary

Keep this PR open: it targets a real ACP task-maintenance bug that current main still has, but the current patch can falsely mark live ACP tasks lost because it checks raw transcript paths instead of the existing session-file resolver contract, and the external PR still lacks real behavior proof.

## What This Changes

This PR adds ACP zombie-session detection in task-registry maintenance plus regression tests for stale running ACP entries with missing or dangling transcript files.

## Best Possible Solution

Resolve ACP transcript liveness through the existing session-file path helpers, add focused coverage for derived and relative transcript paths, then land after redacted real behavior proof and clean exact-head checks.

## Reproduction Assessment

Yes, at source level. Current main keeps ACP tasks alive whenever readAcpSessionEntry returns an entry, while the documented lost contract says ACP tasks should lose active status when authoritative backing metadata is gone after the grace period.

## Solution Assessment

No, not yet. The fix direction is narrow, but the implementation should resolve transcript liveness through resolveSessionFilePath rather than treating a missing or raw sessionFile value as definitive.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.83

Full review comments:

- **[P2] Resolve ACP transcript paths before declaring them missing:** `src/tasks/task-registry.maintenance.ts:526-530`
  - body: resolveSessionFilePath() derives the canonical transcript path when sessionFile is absent and resolves relative or legacy values. This change returns lost before resolving and then checks fs.existsSync() on the raw stored string, so a stale-but-live ACP session with a valid derived or relative transcript can be marked lost during maintenance. Resolve through the existing session path helpers and cover that case.
  - confidence: 0.84
