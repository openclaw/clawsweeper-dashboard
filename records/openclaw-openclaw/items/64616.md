## Summary

Keep open. Current main still accepts any ACP session-store entry as backing state, and the related stale-row issue was closed in favor of this PR, but the patch still needs resolver-based transcript liveness and contributor real behavior proof before it can merge.

## What This Changes

The PR changes task-registry maintenance to mark stale running ACP session metadata as lost when its transcript file is absent or missing, with regression tests for zombie and live ACP cases.

## Best Possible Solution

Keep this PR as the canonical fix path, resolve ACP transcript liveness through resolveSessionFilePathOptions/resolveSessionFilePath using ACP entry store context, add derived/relative path coverage, and merge after redacted real behavior proof is added.

## Reproduction Assessment

Yes, at source level. Current main's ACP branch returns Boolean(acpEntry.entry), so a stale ACP session-store row can keep an active task from becoming lost after the documented grace period.

## Solution Assessment

No, not yet. The fix direction is narrow, but the implementation should use the existing session-file resolver instead of treating missing or raw sessionFile strings as definitive.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

- **[P2] Resolve ACP transcript paths before marking sessions lost:** `src/tasks/task-registry.maintenance.ts:526-531`
