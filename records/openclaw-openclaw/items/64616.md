## Summary

Keep open. Current main still treats any readable ACP session-store row as ACP backing state, while this PR remains the focused fix for stale running ACP metadata with missing transcript backing; I found no discrete functional code finding, but the external PR still needs real behavior proof and green exact-head validation before merge.

## What This Changes

The PR changes task-registry maintenance to mark stale running ACP tasks lost when ACP metadata remains but the transcript file is missing or dangling, and adds regression tests for zombie and live ACP cases.

## Best Possible Solution

Land this narrow task-maintenance fix after contributor real behavior proof is added and exact-head CI is green, while keeping the broader CLI zombie-task work on https://github.com/openclaw/openclaw/issues/73121.

## Reproduction Assessment

Yes, at source level. On current main, an old active `runtime: "acp"` task whose ACP session-store row still exists with `acp.state: "running"` and no or dangling transcript file remains backed because maintenance only checks `Boolean(acpEntry.entry)`.

## Solution Assessment

Yes. The PR uses the existing task-maintenance `lost` path, preserves fail-open behavior for unreadable ACP stores, and adds focused regression coverage; the remaining blockers are proof and validation rather than a different code design.
