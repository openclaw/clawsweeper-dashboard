## Summary

Keep this PR open, but it is not merge-ready: the core bug is still present on current main and the PR is the canonical fix path, but the latest head still bypasses the session-file resolver and lacks contributor-supplied real behavior proof.

## What This Changes

The PR adds ACP task-maintenance logic and tests to mark stale running ACP session-store rows lost when their transcript metadata is absent or dangling.

## Best Possible Solution

Resolve ACP transcript liveness through resolveSessionFilePathOptions and resolveSessionFilePath using the ACP store/session context, then land with focused resolver-backed regression coverage and redacted real behavior proof.

## Reproduction Assessment

Yes, at source level. Current main returns Boolean(acpEntry.entry) for ACP backing state, and the PR-introduced failure can be exercised with a stale running ACP entry whose transcript exists only at the resolver-derived path.

## Solution Assessment

No, not yet. The direction is narrow and appropriate, but the implementation should use the existing session-file resolver instead of checking missing or raw sessionFile strings directly.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Resolve ACP transcript paths before checking liveness:** `src/tasks/task-registry.maintenance.ts:526-530`
  - body: This checks `fs.existsSync` on the raw stored `sessionFile` and treats a missing field as a zombie, but the session resolver can derive or normalize the live transcript path from the ACP store/session context. A stale running ACP task with a derived, relative, or legacy transcript path can be marked lost even though its transcript still exists; use `resolveSessionFilePathOptions`/`resolveSessionFilePath` and cover that path.
  - confidence: 0.86
