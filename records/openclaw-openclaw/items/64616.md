## Summary

Keep open. Current main still has the ACP stale-row backing-session bug and this PR is the focused fix path, but the patch needs a session-path resolution fix plus external real behavior proof before merge.

## What This Changes

The PR changes task-registry maintenance to mark stale running ACP task records lost when ACP metadata has no transcript file or a dangling transcript path, and adds zombie/live ACP regression tests.

## Best Possible Solution

Resolve ACP transcript liveness through the existing session-file path helpers, add focused coverage for derived or relative transcript paths, then land after redacted real behavior proof and exact-head checks.

## Reproduction Assessment

Yes, at source level. On current main, `hasBackingSession()` returns `Boolean(acpEntry.entry)` for ACP tasks after a readable ACP store row, so a stale running ACP row still backs the task despite the documented lost-state contract.

## Solution Assessment

No, not yet. The direction is narrow, but the implementation checks missing/raw `sessionFile` values instead of resolving through the existing session transcript path helpers, which can mark valid derived or relative transcripts lost.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.8

Full review comments:

- **[P2] Resolve ACP transcript paths before declaring them missing:** `src/tasks/task-registry.maintenance.ts:526-530`
  - body: `resolveSessionFilePath()` treats a missing `sessionFile` as a derivable transcript path and resolves relative or legacy paths against the session store. This new check returns `true` before resolving and then calls `existsSync` on the raw string, so a stale-but-live ACP entry with a valid derived or relative transcript can be marked `lost`. Resolve the entry through the existing session path helpers and check that path instead.
  - confidence: 0.84
