## Summary

Keep open: current main still accepts any readable ACP session-store row as backing state, and this PR is the focused fix path, but the current head can falsely mark live ACP sessions lost when transcript paths need the existing resolver and it still lacks after-fix real behavior proof.

## What This Changes

The PR adds ACP zombie-session detection in task-registry maintenance and regression tests for stale and live ACP transcript cases.

## Best Possible Solution

Keep this PR as the canonical fix path, route ACP transcript liveness through the existing session-file resolver with store-path and agent context, add derived/relative path coverage, then merge after redacted real behavior proof is added.

## Reproduction Assessment

Yes, at source level. Current main keeps ACP tasks active whenever readAcpSessionEntry returns an entry, while the task contract says active tasks should become lost when authoritative backing state is gone after the grace period.

## Solution Assessment

No, not yet. The fix direction is narrow, but the implementation should use resolveSessionFilePath/resolveSessionFilePathOptions instead of treating missing or raw sessionFile strings as definitive.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Resolve ACP transcript paths before marking sessions missing:** `src/tasks/task-registry.maintenance.ts:526-530`

This code returns `lost` when `sessionFile` is absent and checks `fs.existsSync` on the raw stored string, so a live ACP task can be marked lost when its transcript is reachable only through the resolver; use the existing path helpers and cover derived/relative paths.
