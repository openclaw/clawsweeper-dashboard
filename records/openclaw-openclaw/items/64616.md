## Summary

Keep open. Current main still trusts ACP session-store row existence for active ACP tasks, so the PR targets a real task-maintenance gap, but the patch only treats an empty `sessionFile` value as zombie state. A retained `sessionFile` path whose transcript was deleted still keeps the stale ACP task running, which leaves the stated user problem unresolved.

## What This Changes

The PR adds ACP zombie-session detection in task-registry maintenance for old running ACP metadata with no transcript file, plus regression tests for zombie and live ACP cases.

## Best Possible Solution

Land a narrow task-maintenance fix that treats a stale running ACP row as backing state only when the stored ACP metadata is non-terminal and the resolved transcript backing file still exists or is readable, while preserving current fail-open behavior for unreadable session stores and leaving cron, subagent, CLI, and broader ACP lifecycle policy unchanged.

## Reproduction Assessment

Yes. A focused regression can create an ACP task older than the maintenance grace period with `acp.state: "running"` and a `sessionFile` value pointing to a nonexistent transcript; the PR head keeps it `running` because it only checks path truthiness.

## Solution Assessment

No. The PR is directionally useful for missing `sessionFile` metadata, but the maintainable fix needs to validate the resolved transcript backing file rather than treating any non-empty `sessionFile` string as proof of a live ACP backing session.
