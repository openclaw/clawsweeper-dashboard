## Summary

Keep open. Current main still uses the strict SDK `listTools()` path for bundled MCP cataloging, so the proposed bug fix is not obsolete; however the PR should not merge yet because the raw `client.request()` replacement bypasses SDK tool metadata caching, and the contributor proof is only tests/checks rather than a real MCP/provider run.

## What This Changes

The PR changes bundled MCP cataloging to parse `tools/list` with a lenient schema, backfill missing root `inputSchema.type`, skip explicit non-object root schemas, update regression tests, and add a changelog entry.

## Best Possible Solution

Keep the lenient `tools/list` parser and malformed-root filtering, but preserve the SDK metadata side effects or equivalent outputSchema/task-support behavior and require redacted live MCP/provider proof before merge.

## Reproduction Assessment

Yes, source-reproducible. Current main calls SDK `client.listTools()` for bundled MCP cataloging, the pinned SDK requires root `inputSchema.type` to be literal `object`, and the linked report names Playwright and Puppeteer MCP servers that return typeless schemas.

## Solution Assessment

No. The lenient listing and invalid-root filtering are the right direction, but replacing `client.listTools()` with raw `client.request()` is not the narrowest safe fix unless it preserves SDK tool metadata behavior.
