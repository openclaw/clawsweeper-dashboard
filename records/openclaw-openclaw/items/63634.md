## Summary

The underlying client-side MCP catalog bug remains present on current main, and the supplied live proof is convincing. This PR should stay open, but the current two-request fallback has a blocking validation-to-use gap: it classifies the first server response, then catalogs a separately fetched response under broader validation.

## What This Changes

The PR adds a narrowly gated lenient MCP tool-list fallback, backfills missing root object types, filters explicit non-object schemas, preserves SDK metadata caching, and expands runtime regression coverage.

## Best Possible Solution

Validate and catalog each `tools/list` page from one response using a schema that differs from the pinned SDK ToolSchema only by allowing a missing root `inputSchema.type`, then normalize accepted tools and populate the SDK-equivalent metadata cache without a re-list window.

## Reproduction Assessment

Yes. Current main calls the pinned SDK's strict `listTools()` parser, whose ToolSchema requires root `inputSchema.type` to be `object`; the supplied real stdio output independently demonstrates that this drops the whole server catalog.

## Solution Assessment

No, not in its current form. The compatibility boundary is appropriate, but validating one response and consuming a second response is weaker than parsing the original page once with only the intended schema relaxation.
