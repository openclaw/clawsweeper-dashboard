## Summary

Keep this PR open. Current main still has the strict MCP SDK `listTools()` path, so the linked typeless `inputSchema` failure remains unfixed on main, but the PR should not merge as-is because its raw `client.request()` replacement bypasses MCP SDK 1.29.0 tool metadata caching used later by `callTool()`.

## What This Changes

The PR switches bundled MCP tool discovery from `Client.listTools()` to a lenient `tools/list` request, normalizes or skips root input schemas, adds regression tests, and adds a changelog entry for the MCP loading failure in #63602.

## Best Possible Solution

Land a reworked version that keeps lenient `tools/list` parsing, missing root-type backfill, and malformed-root isolation, while preserving or locally reimplementing the SDK `listTools()` metadata semantics for output-schema validation and required task-based tool calls.

## Reproduction Assessment

Yes. A mock `tools/list` response with `inputSchema` missing root `type` reaches current main's `client.listTools()` call and is rejected by the pinned SDK `ToolSchema`; the linked #63602 Playwright MCP setup is a concrete user-level reproduction path.

## Solution Assessment

No. The PR targets the right bug and its latest schema filtering is useful, but replacing `listTools()` with raw `request()` without preserving metadata caching is not the narrowest maintainable fix.
