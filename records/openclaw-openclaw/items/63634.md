## Summary

Keep open: the PR is aimed at a real bundle-MCP cataloging failure, but the current branch should not merge because the raw `client.request()` listing path bypasses SDK `listTools()` metadata caching and the supplied proof is only local tests/checks.

## What This Changes

The PR changes bundle MCP tool listing to tolerate typeless `inputSchema` values, normalize accepted schemas, filter explicit non-object root schemas, add regression tests, and add a changelog entry.

## Best Possible Solution

Keep the lenient typeless-schema catalog fix and malformed-root filtering, but preserve SDK-equivalent metadata caching and require redacted live or Testbox MCP/provider proof before merge.

## Reproduction Assessment

Yes, from source and dependency contracts: current main uses SDK `Client.listTools()`, and SDK 1.29.0 rejects listed tools whose root `inputSchema.type` is missing. I did not run a live MCP server in this read-only review, so this is source-reproducible rather than locally reproduced.

## Solution Assessment

No, not as-is. The lenient parser and malformed-root filtering are the right narrow direction, but the raw request path must preserve SDK `listTools()` metadata behavior before this is the safest fix.
