## Summary

Keep this PR open, but it is not merge-ready: the bug fix direction is valid and source-reproducible, while the current diff still bypasses MCP SDK tool metadata caching and lacks real MCP/provider behavior proof.

## What This Changes

This PR changes bundled MCP tool cataloging to parse `tools/list` with a lenient schema, backfill missing root `inputSchema.type`, skip explicit non-object root schemas, add regression tests, and add a changelog entry.

## Best Possible Solution

Keep the lenient typeless-schema catalog fix and malformed-root filtering, but preserve the SDK `listTools()` metadata behavior before merging with redacted real MCP/provider proof.

## Reproduction Assessment

Yes. Current main uses SDK `Client.listTools()` for bundled MCP cataloging, and the pinned SDK requires root `inputSchema.type` to be literal `object`, matching the linked Playwright/Puppeteer MCP failure mode.

## Solution Assessment

No, not as-is. The lenient parser and filtering are the right narrow direction, but the raw request path must preserve SDK tool metadata caching before this is the maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Preserve SDK tool metadata when listing leniently:** `src/agents/pi-bundle-mcp-runtime.ts:205-208`
  - body: Using raw `client.request()` here avoids the strict schema, but it also bypasses SDK v1.29.0 `Client.listTools()`, which caches `outputSchema` validators and required task-tool metadata that `callTool()` consults. A server with `outputSchema` or `execution.taskSupport` would lose that validation/routing after this patch; keep the lenient parse while preserving the SDK metadata cache.
