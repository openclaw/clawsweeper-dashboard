## Summary

Keep open. The PR addresses a source-reproducible bundled MCP cataloging bug and resolves the earlier malformed-root filtering comments, but it still bypasses the MCP SDK `Client.listTools()` metadata cache used by `callTool()`, and the contributor proof is only local test/check output.

## What This Changes

The PR changes bundled MCP tool cataloging to parse `tools/list` leniently, backfill missing `inputSchema.type`, skip explicit non-object root schemas, add regression tests, and add a changelog entry.

## Best Possible Solution

Keep the lenient typeless-schema catalog fix and malformed-root filtering, but preserve the SDK `listTools()` metadata behavior and require redacted real MCP/provider proof before merge.

## Reproduction Assessment

Yes, source-reproducible. Current main uses SDK `Client.listTools()` for bundled MCP cataloging, the pinned SDK requires root `inputSchema.type` to be literal `object`, and the linked report names Playwright/Puppeteer MCP servers that omit it.

## Solution Assessment

No, not as-is. The lenient parser and malformed-root filtering are the right direction, but the raw request path must preserve SDK tool metadata caching before this is the narrow maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Preserve listTools metadata when listing leniently:** `src/agents/pi-bundle-mcp-runtime.ts:205-208`
  - body: Using raw `client.request()` here bypasses MCP SDK v1.29.0 `Client.listTools()`, which calls `cacheToolMetadata(result.tools)`. Later `client.callTool()` checks those caches for required task tools and `outputSchema` validators, so tools with `execution.taskSupport` or `outputSchema` would lose current runtime validation after this patch. Preserve that metadata population when listing leniently.
  - confidence: 0.96
