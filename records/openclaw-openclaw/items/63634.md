## Label Justifications

- `P1`: A valid third-party MCP schema can currently remove the server's entire tool catalog from active agent sessions.
- `merge-risk: 🚨 compatibility`: The patch changes accepted third-party MCP schemas and adds direct coupling to an SDK-private metadata method.
- `merge-risk: 🚨 security-boundary`: The second `tools/list` request uses weaker validation and can accept unrelated malformed metadata if its response differs from the rejected strict response.

## Summary

The underlying MCP catalog bug remains present on current main, and the PR has convincing real-runtime proof, but the fallback performs a second, substantially less validated `tools/list` request whose response can differ from the strictly rejected response. That validation gap should be repaired before merge.

## What This Changes

The PR adds a strict-first MCP tool-listing path with a lenient retry for omitted root `inputSchema.type`, filters explicit non-object schemas, restores SDK metadata caching, and adds regression coverage.

## Best Possible Solution

Parse one server response through a schema that preserves the SDK's complete tool-result validation while relaxing only the missing root `inputSchema.type` requirement, then cache exactly that validated response and retain the explicit non-object catalog gate.

## Reproduction Assessment

Yes. Current-main source shows the strict-only catalog path, and the supplied real stdio comparison demonstrates the complete catalog failure and the repaired behavior on this branch.

## Solution Assessment

No, not yet. Strict-first recovery is appropriate, but retrying the server with a much weaker result schema is less safe than validating a single response while relaxing only the one known incompatibility.

- **[P2] Validate the same tool-list response before accepting it:** `src/agents/agent-bundle-mcp-runtime.ts:358-361`
  - body: The strict failure is checked against the first `tools/list` response, but this line fetches a second response using `LenientListToolsResultSchema`, which does not validate SDK fields such as `outputSchema` and task metadata.
