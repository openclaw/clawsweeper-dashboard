## Summary

Keep this PR open: the lenient bundle MCP cataloging direction matches a real source-reproducible bug, but the branch still bypasses SDK `listTools()` metadata caching and only supplies unit/check output instead of real MCP/provider behavior proof.

## What This Changes

This PR replaces bundle MCP `Client.listTools()` cataloging with a lenient `tools/list` request, normalizes typeless input schemas, filters explicit non-object schemas, adds tests, and updates the changelog.

## Best Possible Solution

Keep the lenient cataloging and malformed-tool filtering approach, but preserve SDK-equivalent metadata behavior before `callTool()` and require redacted real MCP/provider proof before merge.

## Reproduction Assessment

Yes, source-reproducible: current main calls SDK `Client.listTools()`, and SDK v1.29.0 requires root `inputSchema.type` to be literal `object`. I did not run a live MCP server in this read-only review.

## Solution Assessment

No, not as written. The lenient schema and malformed-tool filtering are the right direction, but the raw request path must preserve SDK metadata behavior before this is the best fix.
