## Summary

Keep open. The PR is pointed at a real bundle MCP regression and the latest branch addresses the earlier non-object-schema filtering/test comments, but it still bypasses the MCP SDK metadata cache that `callTool()` depends on, and the supplied proof is test-only rather than real MCP/provider behavior proof.

## What This Changes

The PR changes bundle MCP tool listing to use a lenient `tools/list` schema, backfill missing root `inputSchema.type`, skip explicit non-object root schemas, and add focused tests plus a changelog entry.

## Best Possible Solution

Keep the lenient cataloging and malformed-tool isolation approach, restore SDK-equivalent metadata caching for normalized listed tools, then require redacted live or Testbox MCP/provider proof before merge.

## Reproduction Assessment

Yes, source-reproducible: current `main` calls SDK `Client.listTools()`, and SDK 1.29.0 requires root `inputSchema.type` to be literal `object`. I did not run a live MCP server in this read-only review.

## Solution Assessment

No, not yet. The lenient schema and bad-tool filtering are the right narrow direction, but the implementation must preserve `Client.listTools()` metadata semantics and be proven in a real MCP/provider path.
