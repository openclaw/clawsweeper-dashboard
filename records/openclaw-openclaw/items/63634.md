## Summary

Keep open: this PR addresses a real, source-reproducible bundled MCP cataloging bug, but the latest diff still bypasses MCP SDK listTools metadata side effects and the contributor proof is only tests/checks rather than a real MCP/provider run.

## What This Changes

The branch changes bundled MCP cataloging to parse tools/list with a lenient schema, backfill missing root inputSchema.type, skip explicit non-object root schemas, update regression tests, and add a changelog entry.

## Best Possible Solution

Keep the lenient tools/list parsing and invalid-root filtering, but preserve the SDK listTools metadata behavior and require redacted live MCP/provider proof before merge.

## Reproduction Assessment

Yes, source-reproducible. Current main calls SDK Client.listTools() for bundled MCP cataloging, and the pinned MCP SDK requires root inputSchema.type to be literal object while the linked report names Playwright/Puppeteer MCP servers that omit it.

## Solution Assessment

No, not as-is. The lenient parser and malformed-root filtering are the right direction, but replacing Client.listTools() with raw request() is not safe unless it preserves the SDK metadata side effects used by callTool().

## Review Findings
