## Summary

Keep this PR open: it targets a real bundle MCP client-side catalog regression that is still present on current main and the latest release, but the current patch is not merge-ready because paginated tool lists can leave earlier-page tools cataloged after their SDK call-time metadata guards have been cleared.

## What This Changes

The branch changes bundle MCP client tool listing to use an SDK-derived relaxed tools/list schema for typeless root inputSchema values, skip explicit non-object roots, call SDK metadata caching, and add regression coverage.

## Best Possible Solution

Land one reconciled bundle MCP catalog fix that accepts typeless object roots while rebuilding SDK metadata once for the final accepted paginated tool set.

## Reproduction Assessment

Yes. Current main and v2026.6.10 still call the pinned SDK's strict client.listTools() on the bundle MCP consuming path, and the SDK requires root inputSchema.type to be object; I did not run a live repro in this read-only review.

## Solution Assessment

No, not yet. The consuming-path fix is the right layer, but metadata caching must cover the final accepted paginated tool set and maintainers need to accept the SDK compatibility exception before this is the safest implementation.
