## Summary

Keep open: the MCP catalog regression is real and the branch is directionally useful, but it is not merge-ready because the raw lenient `tools/list` path bypasses SDK tool metadata caching and the supplied proof is still test-output only.

## What This Changes

The PR changes bundle MCP tool listing to use a lenient `tools/list` schema, backfill missing root `inputSchema.type`, skip explicit non-object root schemas, update focused tests, and add a changelog entry.

## Best Possible Solution

Keep the lenient cataloging and malformed-tool filtering approach, but restore SDK-equivalent metadata caching after lenient parsing and require redacted real MCP/provider proof before merge.

## Reproduction Assessment

Yes, source-reproducible: current main and `v2026.5.18` call SDK `Client.listTools()`, and SDK 1.29.0 requires root `inputSchema.type` to be literal `object`. I did not run a live MCP server in this read-only review.

## Solution Assessment

No, not as written. The lenient schema and bad-tool filtering are the right narrow direction, but the raw request path must preserve `Client.listTools()` metadata caching before this is the best fix.
