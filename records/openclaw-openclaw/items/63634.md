## Summary

This PR is still necessary for the bundle MCP consuming path, but it should not merge yet: the current patch caches SDK metadata once per tools/list page even though the pinned SDK clears the previous cache on each call, so earlier paginated tools can remain cataloged without output-schema validators or required-task guards.

## What This Changes

The branch changes bundled MCP client tool cataloging to use an SDK-derived relaxed tools/list schema, backfill missing root inputSchema.type, skip explicit non-object root schemas, call SDK metadata caching, and add regression tests.

## Best Possible Solution

Keep the typeless-root compatibility fix, rebuild SDK metadata once for the final accepted paginated tool set, then get bundle MCP owner review or reconcile the exact fix with https://github.com/openclaw/openclaw/pull/89686.

## Reproduction Assessment

Yes, at source level: current main still calls the SDK's strict client.listTools() on the bundle MCP consuming path, and the pinned SDK requires root inputSchema.type to be object. The PR body also supplies real stdio-server main-versus-branch terminal proof for the central failure.

## Solution Assessment

No, not yet. The consuming-path fix is the right layer, but metadata caching must cover the final accepted paginated tool set and bundle MCP owners need to accept the SDK compatibility exception before this is the safest implementation.
