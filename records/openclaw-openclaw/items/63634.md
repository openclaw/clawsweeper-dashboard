## Summary

Keep open: this is a valid source-reproducible MCP cataloging bug fix, but the latest PR head still bypasses SDK tool metadata caching and only provides test/check output instead of real MCP/provider behavior proof.

## What This Changes

This PR changes bundled MCP tool cataloging to parse `tools/list` with a lenient schema, normalize typeless input schemas, skip explicit non-object roots, add tests, and add a changelog entry.

## Best Possible Solution

Keep the lenient MCP catalog fix, but preserve SDK `listTools()` metadata behavior and require redacted live MCP/provider proof before merge.

## Reproduction Assessment

Yes, source-reproducible. Current main calls SDK `Client.listTools()` for bundled MCP cataloging, the pinned SDK rejects typeless root `inputSchema` objects, and the linked report provides concrete Playwright/Puppeteer server examples.

## Solution Assessment

No, not as-is. The lenient parser and malformed-root filtering target the right code path, but raw `client.request()` must preserve SDK tool metadata caching before this is the narrow maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Preserve SDK tool metadata when listing leniently:** `src/agents/pi-bundle-mcp-runtime.ts:205-208`
  - body: Using raw `client.request()` here skips MCP SDK v1.29.0 `Client.listTools()`, which caches output-schema validators and task-support metadata for later `callTool()` calls. Tools with `outputSchema` or `execution.taskSupport` would still be cataloged, but they lose the current validation/task guard after this patch; preserve that metadata population when using the lenient result schema.
