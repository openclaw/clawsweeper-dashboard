## Summary

Keep open: the bug is source-reproducible and the fix direction is useful, but PR head still bypasses SDK `listTools()` metadata caching and the contributor proof is limited to tests/checks rather than the real MCP/provider path.

## What This Changes

The PR changes bundled MCP tool listing to parse typeless MCP input schemas leniently, normalize accepted schemas, filter explicit non-object roots, update regression tests, and add a changelog entry.

## Best Possible Solution

Keep the lenient `tools/list` parser and malformed-root filtering, but preserve the SDK metadata side effects or equivalent outputSchema/task-support cache before requiring redacted live Playwright or Puppeteer MCP provider proof.

## Reproduction Assessment

Yes, source-reproducible. Current main calls SDK `client.listTools()` for bundled MCP cataloging, the pinned SDK requires root `inputSchema.type` to be literal `object`, and the linked report names Playwright/Puppeteer MCP servers that return typeless schemas.

## Solution Assessment

No. The lenient listing and invalid-root filtering are the right shape, but replacing `client.listTools()` with raw `client.request()` without preserving the SDK metadata cache is not the narrowest safe fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Preserve SDK tool metadata when listing leniently:** `src/agents/pi-bundle-mcp-runtime.ts:205-208`
  - body: The SDK's `Client.listTools()` does more than request `tools/list`: in v1.29.0 it calls `cacheToolMetadata(result.tools)`, and `callTool()` later uses that cache to reject required task tools and validate `outputSchema`. This raw request path never repopulates the cache, so accepted MCP tools with `outputSchema` or `execution.taskSupport` are called without behavior that current main gets after listing.
  - confidence: 0.96
