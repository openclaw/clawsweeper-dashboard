## Summary

Keep open: the PR targets a valid bundled MCP bug that still exists on current main, but the patch bypasses SDK `listTools()` metadata caching and the real behavior proof is still only local tests/checks.

## What This Changes

This PR changes bundled MCP tool cataloging to use a lenient `tools/list` schema, backfill missing root `inputSchema.type`, skip explicit non-object schemas, add regression tests, and update the changelog.

## Best Possible Solution

Land a narrow bundled MCP runtime fix that accepts typeless object schemas, filters malformed root schemas, preserves SDK output-schema/task metadata for `callTool()`, and includes regression plus real MCP/provider proof.

## Reproduction Assessment

Yes. Current main calls SDK `Client.listTools()` in bundled MCP cataloging, and the pinned SDK v1.29.0 rejects tools whose root `inputSchema.type` is missing rather than literal `object`.

## Solution Assessment

No, not as-is. The lenient parser and malformed-root filtering are the right direction, but the raw request path must preserve SDK tool metadata caching before this is a maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Preserve SDK tool metadata when listing leniently:** `src/agents/pi-bundle-mcp-runtime.ts:205-208`
  - body: Using `client.request()` here avoids the strict schema, but it also bypasses SDK v1.29.0 `Client.listTools()`, which calls `cacheToolMetadata(result.tools)`. `callTool()` uses that cache to reject required task tools and validate `outputSchema`, so servers that define `execution.taskSupport` or `outputSchema` would lose existing runtime behavior after cataloging; keep the lenient parse while still populating the SDK metadata cache.
  - confidence: 0.96
