## Summary

Keep open: the lint blocker is gone, but the PR still leaves the passwordless stale-SSSS forced-reset path able to recreate SSSS and retry `setupNewCrossSigning`, so the double-reset state-destruction path remains for that case; the supplied proof is still mocked Matrix SDK unit coverage only.

## What This Changes

The PR changes Matrix crypto bootstrap sequencing and tests so password-backed `--force-reset-cross-signing` bootstraps SSSS before the destructive reset, removes the post-SSSS second cross-signing call, and preserves passwordless deferral.

## Best Possible Solution

Keep the fix in `MatrixCryptoBootstrapper`, but disable the forced-reset repair retry after SSSS recreation for passwordless stale-SSSS as well, add focused regression coverage, and require redacted live Matrix E2EE reset/recovery proof before merge.

## Solution Assessment

No, not yet: the owner-boundary fix location is right, and the password-backed path is directionally good, but the passwordless stale-SSSS path still keeps the double-reset behavior the PR claims to remove. The safer repair is to block the repair retry for all forced resets, or add an equivalent guard plus tests proving no second destructive reset.
