## Summary

Keep open: the Matrix bootstrapper owner boundary is right and the password-backed path is directionally useful, but this PR head still permits the repair retry for passwordless forced resets and only provides mocked Matrix SDK test proof.

## What This Changes

The branch changes Matrix crypto bootstrap sequencing and tests so password-backed forced reset bootstraps SSSS before cross-signing, removes the post-SSSS second cross-signing call, and preserves passwordless deferral.

## Best Possible Solution

Keep the fix in `MatrixCryptoBootstrapper`, but prevent the repair-block second reset for every forced-reset stale-SSSS path, retain the passwordless UIA guard, add focused regression coverage, and require redacted live Matrix proof.

## Reproduction Assessment

I did not run a live Matrix homeserver reproduction in this read-only review.

## Solution Assessment

No, not yet: the owner-boundary fix location is right, and the password-backed path is directionally good, but the passwordless stale-SSSS path still keeps the double-reset behavior this PR claims to remove. The safer repair is to block the repair retry for all forced resets, or add an equivalent guard plus tests proving no second destructive reset.
