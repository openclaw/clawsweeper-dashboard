## Summary

Keep open: this is a focused fix for a real Matrix E2EE double-reset bug, but the patch currently removes the passwordless-bot SSSS guard and can recreate secret storage before the UIA-protected reset upload is known to succeed. The supplied proof is structured, but it is still mock-only unit-test proof rather than a live Matrix homeserver recovery run.

## What This Changes

The PR changes Matrix crypto bootstrap sequencing so forced cross-signing reset bootstraps secret storage before cross-signing, disables the forced-reset repair retry, and updates adjacent unit tests.

## Best Possible Solution

Keep the one-reset forced-repair behavior, but preserve the passwordless/no-UIA SSSS mutation guard and prove the final recovery-key, cross-signing, and backup state in a live Matrix environment.

## Reproduction Assessment

Yes, at source level: current main still skips initial SSSS bootstrap for forced reset, allows a repair-block SSSS recreation, and can call setupNewCrossSigning again. I did not run a live Matrix homeserver reproduction in this read-only review.

## Solution Assessment

No, this PR is not the best fix yet because it removes the prior passwordless-bot SSSS mutation guard while solving the double-reset path. The safer fix keeps one forced reset and aborts or defers SSSS recreation when upload auth cannot be satisfied.

- **[P1] Preserve the passwordless forced-reset guard:** `extensions/matrix/src/matrix/sdk/crypto-bootstrap.ts:64`
  - body: This now recreates or repairs SSSS before attempting the UIA-protected `setupNewCrossSigning` upload. The old forced-reset path deliberately deferred that mutation so passwordless bots on homeservers that later reject password UIA would not partially rewrite SSSS; upstream matrix-js-sdk also warns resetCrossSigning can fail in a bad state when password or 4S material is unavailable. Keep SSSS recreation behind a successful upload/auth path, or add a no-password failure path that proves no server SSSS mutation happened.
