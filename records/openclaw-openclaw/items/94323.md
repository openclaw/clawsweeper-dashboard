## Summary

Keep open for maintainer review. Current main still has the cron add/remove recompute shape that can advance a due recurring job without running it, and this PR applies the established maintenance recompute boundary with focused regression coverage and sufficient real-runtime proof.

## What This Changes

This PR changes cron add/remove loading and recompute behavior so unrelated mutations do not drop another recurring job's due pending run, and adds a colocated regression test for the add path.

## Best Possible Solution

Land the focused cron mutation fix with its regression test, preserving due slots without adding scheduler API or configuration surface.

## Reproduction Assessment

Yes. Current main's `add()` path calls full recompute after appending a job, and full recompute advances any sibling whose `nextRunAtMs` is already due; I did not execute tests because this review is read-only.

## Solution Assessment

Yes. The patch changes the cron mutation entry points that cause the sibling recompute and matches the existing maintenance helper used by read, timer, and manual-run paths; a broader scheduler rewrite is not needed.
