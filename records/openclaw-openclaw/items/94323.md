- `P2`: The PR fixes a bounded cron scheduling bug that can silently skip a due recurring run without adding public API, config, or migration surface.

## Summary

This PR is still necessary and appears correct: current main can still let `cron.add()` and cold-load `cron.remove()` run full schedule recompute over a present due slot, while the branch moves those mutation paths onto the established maintenance recompute invariant and adds focused regression coverage.

## What This Changes

The branch changes cron add/remove operations to skip load-time full recompute, run `recomputeNextRunsForMaintenance()` before persistence, and add regression tests for due recurring jobs.

## Best Possible Solution

Land this focused cron ops fix after normal maintainer review, while keeping the separate startup catch-up deferral work in https://github.com/openclaw/openclaw/pull/94022 independent.

## Reproduction Assessment

Yes. Source inspection shows current main add() and cold-load remove() can reach full recompute, and full recompute advances present past-due nextRunAtMs values; the PR body also supplies before/after live CronService output against the SQLite store.

## Solution Assessment

Yes. Moving add/remove to `skipRecompute` plus maintenance recompute is the narrow owner-boundary fix because it matches the existing read/startup/timer invariant instead of adding a second scheduling path.
