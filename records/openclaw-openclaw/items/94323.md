## Summary

No blocking findings. The PR is still necessary because current main still lets add() and cold-load remove() reach the full recompute path, while the branch moves those mutation paths onto the established maintenance-recompute invariant and has focused proof.

## What This Changes

The PR changes cron add/remove mutation paths to skip load-time full recompute, run maintenance recompute before persisting, and add regression tests for due recurring jobs.

## Best Possible Solution

Land this focused cron ops fix after normal maintainer review, while tracking the separate startup deferral persistence work in https://github.com/openclaw/openclaw/pull/94022 independently.

## Reproduction Assessment

Yes. Source inspection shows current main add() and cold-load remove() can reach full recompute, and full recompute advances present past-due nextRunAtMs values; the PR body also supplies before/after live CronService output against the SQLite store.

## Solution Assessment

Yes. Moving add/remove to skipRecompute plus maintenance recompute is the narrow owner-boundary fix because it matches the existing read/startup/timer invariant instead of adding a second scheduling model or keeping unsafe full recompute.
