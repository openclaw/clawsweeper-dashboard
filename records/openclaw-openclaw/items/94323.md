## What This Changes

The PR changes cron add/remove mutation paths to avoid full next-run recomputation and adds a regression test for an unrelated add preserving a due recurring job's pending run.

## Best Possible Solution

Land the small cron mutation fix after the contributor supplies redacted real runtime proof, or after a maintainer explicitly accepts the proof gap.

## Reproduction Assessment

Yes, source-reproducible: current `add()` calls full recompute after appending a job, and full recompute advances any sibling whose `nextRunAtMs` is already due. I did not execute the test because this review is read-only.

## Solution Assessment

Yes for the reproduced add path: `ops.ts` owns cron mutation entry points and existing read/timer siblings already use maintenance recompute to preserve due slots. The remove change is a weaker but reasonable consistency guard rather than a separately proven user-visible reproduction.
