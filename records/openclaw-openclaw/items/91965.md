## Label Justifications

- `P1`: The reported released-version regression disables semantic memory retrieval and can return a repaired index to a broken state.
- `impact:session-state`: The issue concerns agent memory retrieval and consistency between persisted index metadata and gateway runtime state.

## Summary

Current main no longer trusts a permanently cached identity result: both gateway search and sync re-read the SQLite metadata before validating identity or starting a reindex, so a valid CLI rebuild is recognized without restarting the gateway manager.

## What This Changes

Fix a memory-search regression where gateway-side index identity state could remain stale and repeatedly remove valid index metadata after an interrupted rebuild.

## Best Possible Solution

Keep the current single SQLite-backed identity path and close this report; if the same sequence still fails on a release containing commit 7cb2571a992b91274b84f2b473c348c90af29763, open a new issue with fresh logs and the post-search meta row.

## Reproduction Assessment

No on current main: search and sync independently re-read the canonical meta row before identity validation, so the reported permanently stale constructor-time state is no longer the current code path.

## Solution Assessment

Yes. Refreshing identity through the existing SQLite-backed validation path is narrower and safer than adding cache invalidation signals, sidecar state, or another compatibility path.
