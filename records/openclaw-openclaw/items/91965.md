## Label Justifications

- `P2`: The bug disabled semantic memory recall for affected memory-core users, but the blast radius is limited to memory index recovery rather than the whole runtime.
- `impact:session-state`: The issue is about indexed memory state becoming unusable for agent recall.
- `impact:auth-provider`: The reported failure depends on embedding provider/model identity and runtime provider availability matching the stored index metadata.

## Summary

Current main already fixes the reported memory_search missing-metadata loop: missing metadata with chunks now routes through safe rebuild when appropriate, live SQLite swaps no longer expose an empty-database window, and timed-out memory_search calls abort orphaned embedding work.

## What This Changes

The issue reports that gateway memory_search remains disabled with missing index metadata after a CLI rebuild/fix and then removes metadata again during tool-triggered recovery.

## Best Possible Solution

Keep the current-main memory index swap, missing-metadata recovery, and timeout-cancellation fixes; ask for a new report only if the same Ollama metadata loss reproduces on a build containing 865fdab075 and 8d72cb9401.

## Reproduction Assessment

Yes, at source level for the released path: v2026.6.5 still had an absent target-file swap window and read-write DatabaseSync open that could create an empty no-meta DB. I did not run the exact macOS Ollama qwen3-embedding live reproduction against current main because this is a read-only review.

## Solution Assessment

Yes. The current-main fix addresses the storage/reindex and timeout layers directly instead of adding a cache shim around indexIdentityState, and regression tests cover the missing-meta rebuild, no-absent swap, and no-create reopen behavior.
