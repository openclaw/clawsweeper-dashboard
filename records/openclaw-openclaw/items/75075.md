## Summary

Keep open. Current main still lacks the requested built-in tool-call exposure, and this PR is the active implementation for #75074 after the parallel #75107 was closed unmerged. The patch looks directionally sound with no blocking correctness finding from this read-only pass, but it changes a public Gateway response shape and should get maintainer API-semantics review plus normal exact-head checks before merge.

## What This Changes

The PR adds a default-off `gateway.http.endpoints.responses.exposeBuiltInToolCalls` flag that surfaces run-scoped built-in tool-start events as audit-only `/v1/responses` `function_call` output items in JSON and SSE responses, with config schema, docs, changelog, and e2e tests.

## Best Possible Solution

Land a maintainer-approved version that preserves the default-off config, run-scoped sanitized tool-start capture, JSON/SSE ordering contract, client-tool dedupe, docs/schema/changelog alignment, and keeps tool result capture or built-in delegation out of scope.

## Reproduction Assessment

Yes. Emit a run-scoped `stream: "tool"`, `phase: "start"` event during a `/v1/responses` turn; current main ignores that stream in both JSON and SSE output, while the PR tests cover the default-off, JSON, SSE, and dedupe paths.

## Solution Assessment

Mostly yes. Reusing the existing sanitized agent-event stream behind an opt-in flag is the narrowest maintainable implementation path; the remaining decision is maintainer approval of the exact response semantics and merge-time validation.
