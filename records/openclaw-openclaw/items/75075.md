## Summary

Keep open for maintainer review. Current main still lacks any `/v1/responses` path that surfaces built-in tool-start events, and this PR appears to be a narrow, default-off implementation with schema, docs, changelog, and focused JSON/SSE coverage; I found no blocking correctness or security finding in the diff. ## What This Changes

The PR adds a default-off `gateway.http.endpoints.responses.exposeBuiltInToolCalls` flag, captures run-scoped built-in tool-start events into `/v1/responses` JSON/SSE `function_call` output items, and updates config schema, docs, changelog, and OpenResponses tests.

## Best Possible Solution

Land a maintainer-approved version that keeps the behavior opt-in, consumes the existing sanitized run-scoped tool-start event stream, preserves client-tool delegation semantics, and ships with aligned schema, docs, changelog, and JSON/SSE ordering coverage.

## Reproduction Assessment

Yes. Emit a run-scoped `stream: "tool"`, `phase: "start"` event during a `/v1/responses` turn; current main has no JSON or SSE branch that forwards it into response output, while the PR tests model that path.

## Solution Assessment

Mostly yes. The PR is additive and default-off, uses the existing sanitized event stream instead of creating a new runtime channel, and avoids changing caller-provided client-tool handling; the only unresolved point is maintainer approval of the public semantics.
