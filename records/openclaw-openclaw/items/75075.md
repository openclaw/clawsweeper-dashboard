## Summary

Keep open: current main still lacks this audit surface, and the PR appears to implement it with a default-off config path and JSON/SSE coverage. It is a new Gateway API/config feature that needs maintainer contract review, with one low-priority docs/API-name mismatch around `bash` versus normalized `exec`.

## What This Changes

This PR adds a default-off `gateway.http.endpoints.responses.exposeBuiltInToolCalls` flag, captures run-scoped built-in tool-start events as `/v1/responses` JSON/SSE `function_call` output items, and updates config schema, docs, changelog, and regression tests.

## Best Possible Solution

Land this PR or an equivalent change after maintainer approval of the new config/API contract and after aligning the public shell tool-name documentation/tests with the normalized event name.

## Reproduction Assessment

Yes for source-level behavior: current main has no `exposeBuiltInToolCalls` config and the `/v1/responses` JSON/SSE handlers ignore `tool` stream start events, while the agent layer already emits run-scoped tool-start events. I did not live-run the endpoint in this read-only review.

## Solution Assessment

Yes for the implementation shape, pending API review: the default-off, run-scoped, sanitized, start-only forwarding is narrow and avoids changing existing clients. The safer adjustment before merge is to align the `bash` versus `exec` public naming contract.
