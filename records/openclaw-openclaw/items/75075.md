## Summary

Keep open: the PR addresses a real source-level gap and I found no blocking correctness or security issue, but it adds a new config/API contract and has a low-priority shell tool-name documentation mismatch to resolve before merge.

## What This Changes

The PR adds a default-off Gateway `/v1/responses` option to expose built-in agent tool-start events as audit-only `function_call` output items in JSON and SSE responses.

## Best Possible Solution

Land this PR or an equivalent change after maintainer review confirms the new config/API contract and aligns the documented shell tool name with actual emitted events.

## Reproduction Assessment

Yes for the source-level gap: current main has no config flag and both JSON and SSE `/v1/responses` paths ignore run-scoped built-in `tool/start` events. I did not live-run the endpoint because this review was constrained to read-only inspection.

## Solution Assessment

Yes for the implementation shape, with maintainer approval for the new surface. The default-off, run-scoped, sanitized, start-only forwarding is narrow; the remaining question is whether shell calls should be exposed as normalized `exec` or documented as `bash`.

## Review Findings

Overall correctness: patch is correct

Overall confidence: 0.86
