## Summary

Keep open: current main still lacks this opt-in `/v1/responses` audit surface, and the PR is a narrow default-off implementation with tests/docs/changelog. It still needs maintainer API review, plus one small documentation/API naming alignment around shell calls being emitted as normalized `exec` rather than documented `bash`.

## What This Changes

The PR adds `gateway.http.endpoints.responses.exposeBuiltInToolCalls` and captures run-scoped built-in tool-start events as `/v1/responses` `function_call` output items in JSON and SSE responses, with config schema, tests, docs, and changelog updates.

## Best Possible Solution

Land this PR or an equivalent default-off implementation after maintainer review confirms the public API shape and the documented shell tool name matches the emitted runtime value.

## Reproduction Assessment

Yes for the source-level gap: current main has no config flag and `/v1/responses` does not forward run-scoped built-in `tool/start` events in either JSON or SSE paths. I did not live-run the endpoint in this read-only review.

## Solution Assessment

Yes, with maintainer approval for the new API surface. The implementation is additive, default-off, run-scoped, start-only, and backed by focused tests; the safer remaining adjustment is to align docs/tests with the normalized shell tool name or explicitly map it back.
