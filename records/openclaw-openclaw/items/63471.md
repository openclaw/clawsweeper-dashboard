## Label Justifications

- `P2`: This is a normal-priority Docker onboarding improvement with clear source-level behavior and existing setup workarounds, but it requires product/security review before implementation.
- `impact:security`: The requested startup relaxation changes when an unconfigured Docker gateway may serve before durable auth, bind, and Control UI origin policy are established.
- `impact:crash-loop`: The reported raw Compose flow exits the gateway container before serving, which Docker managers can present as a restart loop.

## Summary

Keep open: current main and v2026.6.10 still require setup or persisted gateway config before a raw Dockge-style Compose startup from an empty config, and changing that safely needs a maintainer product/security decision.

## What This Changes

Request a Dockge/Portainer-friendly Docker Compose startup path that can launch from pasted Compose/.env without extra setup commands while preserving safe gateway defaults.

## Best Possible Solution

Add a maintainer-approved Docker-manager bootstrap that persists durable auth, `gateway.mode=local`, bind policy, and Control UI origins before the gateway serves any published port.

## Reproduction Assessment

Yes, source-level: a raw Compose gateway start with an empty mounted config reaches the missing-config guard and exits before `startGatewayServer`. I did not run Docker in this read-only review.

## Solution Assessment

No, not as proposed: letting the container start unconfigured would weaken the fail-closed gateway boundary. The safer direction is a defined Docker bootstrap contract that establishes auth, mode, bind, and origins before serving.
