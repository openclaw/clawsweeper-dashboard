## Label Justifications

- `P2`: This is a normal-priority Docker onboarding improvement with clear source-level behavior but it needs product/security review before implementation.
- `impact:security`: The requested relaxation changes when an unconfigured Docker gateway may serve before durable auth and Control UI origin policy are established.
- `impact:crash-loop`: The reported raw Compose flow exits the gateway container before serving, which can become a restart loop under Docker managers.

## Summary

Keep open: current main and the latest release still require setup or manual config before raw Docker Compose gateway startup from an empty config, and making Dockge-style paste-and-start safe needs a maintainer security/product decision rather than cleanup closure.

## What This Changes

Request Docker Compose/Dockge deployments to start successfully from a pasted Compose YAML and .env without an extra OpenClaw setup/onboarding step.

## Best Possible Solution

Add a maintainer-approved Docker first-run bootstrap that works in manager UIs, persists safe gateway config and durable auth, seeds Control UI origins for the published port, and never serves an exposed gateway before those facts exist.

## Reproduction Assessment

Yes, source-level. Starting the bundled Compose gateway with an empty mounted config directory and no pre-start setup reaches the missing-config guard and exits before server startup.

## Solution Assessment

No. Letting the container start by bypassing the guard is not the best solution; the safer direction is an explicit Docker bootstrap contract that establishes auth, mode, bind, and origins before serving.
