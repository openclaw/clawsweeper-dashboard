## Label Justifications

- `P2`: This is a normal-priority Docker onboarding improvement with clear source-level behavior, existing setup workarounds, and a required product/security decision.
- `impact:security`: The requested startup relaxation changes when an unconfigured Docker gateway may serve before durable auth, bind, and Control UI origin policy are established.
- `impact:crash-loop`: The reported raw Compose flow exits the gateway container before serving, which Docker managers can turn into a restart loop.

## Summary

Keep open: current main still requires setup or explicit gateway config before a raw Dockge-style Compose startup from an empty config, and making that flow safe remains a maintainer product/security decision rather than cleanup closure.

## What This Changes

The issue requests a Dockge/Portainer-friendly Docker Compose first-run flow that can start from pasted Compose plus environment values without running an extra OpenClaw setup/config command.

## Best Possible Solution

Add a maintainer-approved Docker/Compose first-run bootstrap that persists durable auth, `gateway.mode=local`, bind policy, and Control UI origins before the gateway serves any published port.

## Reproduction Assessment

Yes, source-level: a raw Compose gateway start with an empty mounted config reaches the missing-config guard and exits before `startGatewayServer`. I did not run Docker because this review is read-only.

## Solution Assessment

No, not as proposed: letting the container start unconfigured would weaken the fail-closed gateway boundary. The safer direction is a defined Docker bootstrap contract that establishes auth, mode, bind, and origins before serving.
