## Summary

Keeping this open: current main and the latest release still require Docker users to run the setup or manual pre-start config path before the gateway serves from an empty config, and changing that for Dockge-style paste-and-start crosses gateway auth, bind, and Control UI origin policy.

## What This Changes

The issue requests a Docker/Dockge paste-and-start Compose path that does not exit on an empty config before setup commands run.

## Best Possible Solution

Add a maintainer-approved Docker first-run bootstrap that persists local gateway config, establishes durable auth, seeds safe Control UI origins for the published port, and documents the raw Compose/Dockge workflow.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: start the bundled Compose gateway with an empty mounted config directory and no `--allow-unconfigured`, and the missing-config guard exits before the server starts.

## Solution Assessment

No. Letting the container start by bypassing the guard is not the best standalone fix; the maintainable solution needs an explicit Docker bootstrap/auth/origin contract before serving.
