## Summary

Keeping this open: current main still does not provide a Dockge-style raw Compose paste-and-start path from an empty config, and the supported Docker path still relies on setup/manual pre-start config writes. Relaxing the startup guard touches gateway auth, bind mode, and Control UI origin policy, so it needs maintainer product/security direction before implementation.

## What This Changes

The issue requests that Docker Compose gateway startup work in Dockge-like paste-and-run environments from Compose YAML and `.env` without running `openclaw setup` or other pre-start CLI commands.

## Best Possible Solution

Add a maintainer-approved Docker/Compose first-run bootstrap that writes persisted local gateway config, provisions durable gateway auth, seeds safe Control UI origins for the published port, and documents the Dockge/raw Compose workflow.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: start the bundled Compose gateway with an empty mounted config directory and no `--allow-unconfigured`, and the command reaches the missing-config guard before the server starts.

## Solution Assessment

No. Letting the container start or changing bind defaults alone is not the best fix; the maintainable solution needs an explicit Docker bootstrap/auth/origin contract before serving.
