## Summary

Keep open: current main still fails closed for raw Dockge-style Compose startup from an empty config, and the remaining fix needs a maintainer product/security decision because recent Docker hardening intentionally removed unconfigured exposed startup.

## What This Changes

The issue asks for Docker Compose/Dockge startup to work from a pasted Compose file and .env without running setup or adding extra gateway CLI flags.

## Best Possible Solution

Add a maintainer-approved Docker/Compose first-run bootstrap that works in manager UIs, persists local gateway config and durable auth, seeds safe Control UI origins for the published port, and does not serve an exposed gateway before those facts exist.

## Reproduction Assessment

Yes, source-level. Starting the bundled Compose gateway with an empty mounted config directory and no `--allow-unconfigured` reaches the missing-config guard and exits before server startup.

## Solution Assessment

No. Letting the container start by bypassing the guard is not the best solution; the safer direction is an explicit Docker bootstrap contract that establishes auth, mode, bind, and origins before serving.
