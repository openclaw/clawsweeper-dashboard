## Summary

Keep open: current main and v2026.6.9 have partial fixes for adjacent reload issues, but the central request remains because Control UI API/RPC responses are still intentionally uncached and sessions-view state is still volatile across a browser reload.

## What This Changes

The issue asks for Control UI reloads to avoid slow full refetches and state loss by caching selected API/RPC responses, persisting lightweight UI state, and keeping sessions list loading bounded.

## Best Possible Solution

Keep this as a security-aware Control UI reload design item: persist only harmless per-view UI state first, and change authenticated API caching only after maintainers approve scoped storage, invalidation, and auth-boundary rules.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a full document reload rebuilds <openclaw-app>, the service worker skips /api and /rpc, and active-tab refresh paths call Gateway loaders again.

## Solution Assessment

Unclear as requested. Persisting harmless UI state is likely maintainable, but stale-while-revalidate caching for authenticated Control UI API data needs a maintainer-approved security and invalidation policy first.
