## Summary

Keep open: current main and v2026.6.9 have already mitigated the sessions-list limit and related navigation-auth side effect, but the central reload request remains because API/RPC responses are still deliberately uncached and sessions-view state is still reconstructed from defaults after a full browser reload.

## What This Changes

The issue requests faster and state-preserving Control UI full-page reloads by caching selected API/RPC responses, persisting lightweight UI state, and avoiding unbounded session reloads.

## Best Possible Solution

Keep this as a maintainer design issue: preserve the no-cache authenticated API boundary until scoped caching is approved, persist only harmless per-view UI state, and retain the shipped bounded sessions-list behavior.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a full document reload rebuilds `<openclaw-app>`, the service worker skips `/api` and `/rpc`, and active-tab refresh paths call Gateway loaders again.

## Solution Assessment

Unclear as requested. Persisting harmless UI state is likely maintainable, but stale-while-revalidate caching for authenticated Control UI API data needs a maintainer-approved security and invalidation policy first.
