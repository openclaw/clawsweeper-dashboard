## Summary

Keep open: current main and v2026.6.9 have fixed the related navigation-auth side effect and bounded the sessions-list defaults, but the central request remains. API/RPC responses are still intentionally uncached, and sessions view controls such as filters, selected rows, and expanded checkpoint state remain volatile across a browser reload.

## What This Changes

The issue requests faster Control UI full-page reloads by caching selected API/RPC data and persisting lightweight per-view UI state, with related sessions-list pagination and limit improvements.

## Best Possible Solution

Keep this as a security-aware Control UI reload design item: preserve the no-cache authenticated API boundary until scoped caching is approved, persist only harmless per-view UI state, and keep the shipped bounded sessions-list behavior.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a full document reload rebuilds `<openclaw-app>`, the service worker skips `/api` and `/rpc`, and active-tab refresh paths call Gateway loaders again.

## Solution Assessment

Unclear as requested. Persisting harmless UI state is likely maintainable, but stale-while-revalidate caching for authenticated Control UI API data needs a maintainer-approved security and invalidation policy first.
