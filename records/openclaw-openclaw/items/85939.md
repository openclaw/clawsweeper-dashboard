## Label Justifications

- `P2`: This is a real Control UI performance and reload-state improvement request with limited blast radius, but it is not an urgent regression or core runtime outage.
- `impact:session-state`: The issue is about losing Control UI tab, filter, selection, and expanded-session state across a browser reload.
- `impact:security`: The main proposed fix would cache authenticated chat/session/status responses, which is a sensitive data and authorization-scope surface.

## Summary

Keep this issue open: current main still bypasses service-worker caching for Control UI API/RPC routes and reconstructs volatile UI state after a browser reload, while the sessions-list pagination concern is already partly addressed.

## What This Changes

Request faster Control UI full-page reloads by caching selected API responses, persisting lightweight UI state, and avoiding unbounded sessions.list reloads.

## Best Possible Solution

Design a security-aware reload strategy that persists harmless per-view state, keeps route/session URL restoration intact, and only caches allowlisted authenticated responses with explicit scoping and invalidation while preserving the existing bounded sessions.list behavior.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: reload the Control UI and the service worker will not cache /api or /rpc responses, while the app reconstructs volatile view state from defaults. I did not run a live large-store timing repro.

## Solution Assessment

Unclear as written: sessionStorage for harmless UI state is likely maintainable, but stale-while-revalidate for authenticated chat/session API data needs a security and invalidation design first. The sessions-list limit part is already partly solved on current main and v2026.5.22.
