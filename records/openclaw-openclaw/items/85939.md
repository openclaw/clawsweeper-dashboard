## Summary

Keep open: current main and v2026.6.6 fixed the related top-level navigation/auth side effect and bounded the sessions list, but API/RPC responses are still intentionally skipped by the service worker and per-view sessions UI state remains volatile. The remaining reload-cache/state request needs a security-aware product decision.

## What This Changes

Request to improve Control UI full-page reloads by caching selected API data, persisting lightweight UI state, and bounding sessions.list loads.

## Best Possible Solution

Keep this as a security-aware Control UI reload design item: preserve the current bounded sessions.list behavior, persist only harmless per-view state, and add API caching only for allowlisted responses with explicit auth/gateway scoping and invalidation.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: a full document reload rebuilds Lit state, the service worker skips /api and /rpc responses, and active tab refresh paths call gateway loaders again. I did not run a live large-store timing repro.

## Solution Assessment

Unclear as requested: sessionStorage for harmless UI state is likely maintainable, but stale-while-revalidate caching for authenticated chat/session/status API data needs a maintainer-approved security and invalidation policy first. The sessions-list limit portion is already mitigated.
