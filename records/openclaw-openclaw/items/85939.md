## Summary

Keep open: current main and the latest release still deliberately bypass Control UI API/RPC/plugin responses in the service worker and still keep the named per-view session controls in volatile Lit state. The sessions-list performance concern is partly mitigated by bounded defaults, but the remaining reload-cache/state request needs a security-aware product decision.

## What This Changes

The issue requests faster, state-preserving Control UI full-page reloads through selected API/RPC response caching, lightweight UI state persistence, and bounded session loading.

## Best Possible Solution

Keep this as a security-aware Control UI reload design item: persist only harmless per-view state first, keep the bounded sessions-list defaults, and add API caching only with maintainer-approved allowlists, auth scoping, and invalidation.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a document reload reconstructs Lit component state, the service worker bypasses /api and /rpc, and active-tab refresh calls the gateway loaders again; no live large-store timing repro was run.

## Solution Assessment

No, not as a blanket stale-while-revalidate change. Lightweight sessionStorage for harmless UI state is likely the narrowest first step, while authenticated API caching needs an explicit security and invalidation design before implementation.
