## Summary

Keep this issue open: current main still excludes Control UI API/RPC/plugin routes from service-worker caching and rebuilds volatile view state after a full browser reload. The sessions-list pagination concern is partly addressed, but caching authenticated API data needs a maintainer security/product decision first.

## What This Changes

Request to improve Control UI full-page reloads by caching selected API/RPC responses, restoring lightweight UI state, and keeping sessions list loads bounded.

## Best Possible Solution

Design a security-aware reload strategy that persists harmless per-view UI state, coordinates with WebChat refresh-state work, and only caches allowlisted authenticated responses with explicit scoping and invalidation.

## Reproduction Assessment

Yes, at source level: reloading the Control UI creates a fresh app instance, the service worker skips /api and /rpc responses, and the active tab refetches from the gateway. I did not run a live browser timing repro on a large session store.

## Solution Assessment

Unclear as written: persisting harmless UI controls in sessionStorage is likely maintainable, but stale-while-revalidate caching for authenticated API data needs a security and invalidation design first.
