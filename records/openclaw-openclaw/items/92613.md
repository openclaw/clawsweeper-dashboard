## Summary

Keep open for maintainer review: the PR is a focused boundary-layer fix for the linked long-session tool-result replay bug, but it intentionally changes model-visible session replay by pruning old tool results and preserving the latest assistant tool-call batch even when that batch exceeds 24 results.

## What This Changes

The PR adds `toolResult` count pruning in the embedded agent LLM-boundary normalizer, preserves matching assistant tool-call/result pairs, and adds regression coverage for 30 historical pairs plus an oversized latest batch.

## Best Possible Solution

Land the boundary guard if maintainers accept latest-batch preservation over a hard global cap, ideally with one focused provider-payload conversion check for a pruned long history before merge.

## Reproduction Assessment

Yes at source level: current main has no `toolResult` count cap in `normalizeMessagesForLlmBoundary`, and the PR tests exercise the 30-pair boundary behavior. I did not reproduce the original live provider rejection.

## Solution Assessment

Yes as a boundary-layer mitigation if maintainers accept the latest-batch exception: it runs immediately before provider conversion and keeps persisted history intact. A stricter provider-aware cap or clear error would be safer if the provider rejects any 25+ result payload, including the active batch.
