## Summary

Keep open, but do not merge as-is: the ACP fallback-prefix fix is still useful, while the branch lacks real behavior proof and includes an unsafe/superseded task-maintenance hunk plus stale docs.

## What This Changes

The PR changes ACP fallback sessions to use `acp-bridge:<uuid>`, adds raw/canonical pending-prompt matching, and changes task maintenance to treat terminal child sessions as missing for cron/cli/subagent tasks.

## Best Possible Solution

Land a rebased, narrower ACP fix that separates bridge fallback keys from ACP runtime keys, preserves current run-id reconciliation, updates docs/tests, and leaves task liveness on the shipped cron/run-context authority.

## Reproduction Assessment

Yes, source-level rather than live: current main still creates metadata-less ACP bridge sessions with `acp:${sessionId}`, and the session-key utility still reserves raw and agent-scoped `acp:` keys for ACP runtime semantics. I did not establish a live failing `acpx ... openclaw exec` path in this read-only review.

## Solution Assessment

No. The ACP prefix direction is maintainable, but the branch should be rebased and narrowed because current main already has run-id canonical reconciliation, issue 73121 fixed task liveness separately, the cron child-session hunk is unsafe, and docs need to move with the runtime change.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep active cron tasks off child-session liveness:** `src/tasks/task-registry.maintenance.ts:159-170`
  - body: Including `cron` in this child-session branch makes an active cron job depend on a stored child session row after `isCronJobActive(jobId)` already returned true. A terminal or missing row can then mark a live cron task lost, so keep cron liveness owned by the cron runtime path.
  - confidence: 0.9

  - body: Changing the runtime fallback to `acp-bridge:<uuid>` leaves `docs/cli/acp.md` documenting the default isolated key as `acp:<uuid>`. Update the public ACP docs with the new prefix so users do not rely on the old contract.
  - confidence: 0.84
