## Summary

Keep open: current main and v2026.5.18 still have the ACP bridge `acp:` fallback, so the central bug is not implemented on main, but this PR is not merge-ready because it is conflicting, lacks real behavior proof, and its task-maintenance hunk can mark active cron tasks lost.

## What This Changes

The branch changes ACP bridge fallback session keys from `acp:` to `acp-bridge:`, adds raw/canonical pending-prompt matching, and treats terminal child sessions as missing for cron/cli/subagent task maintenance.

## Best Possible Solution

Land a rebased, narrower ACP fix that separates bridge fallback keys from ACP runtime keys, preserves current run-id reconciliation, updates ACP docs/tests, and leaves active cron liveness owned by the cron runtime.

## Reproduction Assessment

Yes, source-level: current main's metadata-free ACP client path still falls back to `acp:<uuid>`, and the ACP manager treats ACP-shaped keys without persisted metadata as stale. I did not run a live `openclaw acp client` or `acpx` session in this read-only review.

## Solution Assessment

No for the current PR as written. The bridge prefix split is the right direction, but the task-maintenance hunk is unsafe against current cron liveness, the branch is conflicted, and docs plus real behavior proof are still missing.
