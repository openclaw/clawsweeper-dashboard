## Summary

Keep open for maintainer handling: the ACP bug remains valid and the bridge-key direction is plausible, but this branch still has blocking session-key aliasing and cron-liveness defects, is conflicting, and lacks real behavior proof.

## What This Changes

The PR changes ACP fallback session keys to `acp-bridge:*`, broadens pending-prompt session-key matching, and treats terminal child sessions as missing backing state for cron/cli/subagent task maintenance.

## Best Possible Solution

Land a rebased, focused ACP bridge-key fix that keeps metadata-free bridge sessions out of ACP runtime key space, preserves current run-id reconciliation or scopes aliasing to `acp-bridge:*`, updates docs/tests, and leaves cron liveness owned by cron runtime state.

## Reproduction Assessment

Yes, at source level: current main creates metadata-free ACP client sessions without `_meta`, falls back to `acp:<uuid>`, and ACP runtime resolution treats ACP-shaped keys without stored ACP metadata as stale. I did not run a live ACP/acpx session in this read-only review.

## Solution Assessment

No, not as written. The bridge-key direction is good, but the implementation should be rebased onto current run-id reconciliation, scoped to ACP bridge raw/canonical keys, documented, and separated from the cron child-session liveness change.
