## Summary

Keep open: the ACP fallback-prefix fix is still useful, but the proposed task-maintenance hunk can regress active cron jobs, and the external PR only provides test-command validation rather than real after-fix behavior proof.

## What This Changes

The PR changes ACP fallback session keys and pending-prompt session-key matching, adds targeted tests, and treats terminal child sessions as missing during task registry maintenance.

## Best Possible Solution

Land a rebased or replacement fix that keeps ACP bridge fallback sessions out of the ACP runtime namespace, updates ACP docs/tests, and preserves cron runtime authority while leaving broader CLI zombie-task cleanup to the canonical task issue.

## Reproduction Assessment

Yes, source-level. Current main still creates fallback ACP bridge sessions as `acp:<uuid>`, and the PR regression follows from the patch path where an active cron task with a terminal child session would bypass the authoritative `isCronJobActive(jobId)` result.

## Solution Assessment

No. The ACP fallback-prefix direction is maintainable, but the task-maintenance hunk is not the best fix because child-session state should not override active cron runtime ownership.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Preserve active cron ownership:** `src/tasks/task-registry.maintenance.ts:159-170`
  - body: Including `cron` in this child-session lookup path makes an active cron task depend on the stored child session row. If that row is missing or terminal, maintenance can mark the task lost even though `isCronJobActive(jobId)` is still true, so keep the active-cron short-circuit or scope terminal-child handling away from cron ownership.
  - confidence: 0.9
- **[P3] Update the ACP session-key docs:** `src/acp/translator.ts:540`
  - body: Changing the default bridge fallback to `acp-bridge:<uuid>` leaves `docs/cli/acp.md` saying ACP defaults to an `acp:` prefix. Update the user docs so the documented session-key semantics match the runtime change.
  - confidence: 0.83
