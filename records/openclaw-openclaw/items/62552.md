## Summary

Keep open: the ACP bridge-key fix is still useful and not on current main, but this PR is not merge-ready because it is conflicting, lacks real behavior proof, and includes a task-maintenance change that can misclassify active cron work.

## What This Changes

The PR changes metadata-free ACP bridge fallback sessions from `acp:*` to `acp-bridge:*`, adds raw/canonical pending-prompt matching, and extends task maintenance to treat terminal child sessions as missing for cron/cli/subagent tasks.

## Best Possible Solution

Land a rebased, narrower ACP fix that separates bridge fallback keys from ACP runtime keys, preserves current run-id reconciliation, updates ACP docs/tests, and leaves active cron liveness owned by the cron runtime.

## Reproduction Assessment

Yes, at source level: current main still creates metadata-free ACP bridge sessions with `acp:<uuid>`, and ACP runtime resolution treats ACP-shaped keys without stored ACP metadata as stale. I did not run a live `openclaw acp client` or `acpx` session in this read-only review.

## Solution Assessment

No for this PR as written. The bridge prefix split is the right direction, but the cron liveness hunk is unsafe against current main, the branch is conflicted, and docs plus real behavior proof are still missing.
