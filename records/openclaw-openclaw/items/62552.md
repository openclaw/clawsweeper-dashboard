## Summary

Keep open. The ACP prefix change addresses a source-level bug that is not fixed on current main, but this PR is not mergeable as-is because it lacks real behavior proof, carries an unsafe and now-superseded cron maintenance hunk, and misses the public ACP docs update for the new session-key prefix.

## What This Changes

The PR switches metadata-less ACP bridge sessions from `acp:` to `acp-bridge:`, adds raw/canonical pending-prompt matching, and treats terminal child session records as missing for cron/cli/subagent task maintenance.

## Best Possible Solution

Land a rebased, narrower ACP fix that separates bridge fallback keys from ACP runtime keys, preserves current run-id reconciliation, updates ACP docs/tests, and leaves task liveness on the shipped cron/run-context authority.

## Reproduction Assessment

Yes, source-level: current main still creates metadata-less ACP bridge sessions with `acp:${sessionId}`, while the session-key utility reserves raw and agent-scoped `acp:` for ACP runtime semantics. I did not establish a live failing `acpx ... openclaw exec` path in this read-only review.

## Solution Assessment

No. The ACP prefix split is maintainable, but the current PR should be rebased and narrowed because current main already has run-id reconciliation, the task-maintenance part is unsafe and superseded, and docs need to move with the runtime contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep active cron tasks off child-session liveness:** `src/tasks/task-registry.maintenance.ts:159-170`
  - body: Adding `cron` to this child-session branch makes an active cron task depend on a stored child session row after `isCronJobActive(jobId)` has already returned true. A terminal or missing row can then mark a live cron task `lost`, so keep cron liveness owned by the cron runtime path.
  - confidence: 0.9

  - confidence: 0.84
