## Summary

Keep open: the ACP bridge key bug still exists on current main and the latest release, but this PR is not mergeable as-is because the task-maintenance hunk can break active cron ownership and the external contributor proof gate is still missing.

## What This Changes

The PR changes ACP fallback sessions to use `acp-bridge:<uuid>`, adds raw/canonical pending-prompt matching, and extends task maintenance to treat terminal child sessions as missing for cron, CLI, and subagent tasks.

## Best Possible Solution

Land a rebased replacement that limits this work to ACP bridge key isolation and canonical matching, updates ACP docs/tests, and leaves zombie-task cleanup on the shipped task-maintenance path.

## Reproduction Assessment

Yes, source-level. Current main and `v2026.5.7` still create metadata-less ACP bridge sessions with `acp:` and have no `acp-bridge` implementation; the cron regression risk follows from the PR diff routing active cron tasks into child-session terminal-state checks.

## Solution Assessment

No. The ACP prefix and raw/canonical matching direction is maintainable, but the task-maintenance hunk should not override cron runtime authority and the docs need to be updated with the new prefix.
