## Summary

Keep open; the ACP fix is still relevant on current main, but this PR cannot merge as-is because the cron maintenance hunk conflicts with the current runtime-owned cron liveness contract and external real behavior proof is missing.

## What This Changes

The branch changes ACP fallback sessions to use `acp-bridge:<uuid>`, adds raw/canonical ACP pending-prompt matching, and extends task maintenance to treat terminal child sessions as missing.

## Best Possible Solution

Land a rebased, narrower change that keeps the ACP bridge key isolation and raw/canonical matching, updates ACP docs/tests, and leaves cron/task liveness on the shipped maintenance contract.

## Reproduction Assessment

Yes, source-level. Current main still uses `acp:` fallback keys and exact pending-prompt session-key matching; the cron regression is visible from the PR diff adding active cron tasks to child-session terminal-state checks.

## Solution Assessment

No. The ACP prefix and canonical matching direction is maintainable, but the task-maintenance hunk should be removed or replaced and the ACP docs should move with the runtime change.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Keep cron liveness on the cron runtime:** `src/tasks/task-registry.maintenance.ts:159-170`
  - body: Including `cron` in the child-session branch makes an active cron task depend on a stored child session row. If that row is terminal or missing, maintenance can mark the task lost even while `isCronJobActive(jobId)` is true, so keep cron liveness owned by the cron runtime path.
  - confidence: 0.9
- **[P3] Update the ACP session-key docs:** `src/acp/translator.ts:540`
  - body: Changing the fallback to `acp-bridge:<uuid>` leaves `docs/cli/acp.md` documenting default isolated ACP keys as `acp:<uuid>` / `acp:`. Update the public ACP docs with the runtime change so users do not rely on the old prefix.
  - confidence: 0.84
