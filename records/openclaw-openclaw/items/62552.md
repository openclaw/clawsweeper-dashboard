## Summary

Keep open: the ACP bridge-key bug is still valid on current main, but this PR is not merge-ready because it conflicts, lacks real behavior proof, and includes a task-maintenance change that can misclassify active cron task state.

## What This Changes

The PR changes ACP bridge fallback session keys to `acp-bridge:*`, adds raw/canonical pending-prompt matching, and treats terminal child session rows as missing backing sessions during task maintenance.

## Best Possible Solution

Land a rebased, narrower ACP fix that separates bridge fallback keys from ACP runtime keys, preserves current run-id reconciliation, updates ACP docs/tests, and leaves active cron liveness owned by the cron runtime.

## Reproduction Assessment

Yes, at source level: current main still creates metadata-free ACP bridge sessions with `acp:<uuid>`, and ACP runtime resolution treats ACP-shaped keys without stored `acp` metadata as stale. I did not run a live `openclaw acp client` or `acpx` session in this read-only review.

## Solution Assessment

No for the PR as written. The bridge prefix split is the right direction, but the task-maintenance hunk is unsafe against current cron liveness, the branch is conflicted, and docs plus real behavior proof are still missing.
