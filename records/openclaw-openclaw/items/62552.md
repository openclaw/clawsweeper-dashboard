## Summary

Keep open. The ACP bridge fallback fix addresses a real source-level bug still present on current main, but this branch should not merge as-is because the task-maintenance hunk can mark active cron tasks lost and the PR lacks docs plus real behavior proof.

## What This Changes

The PR changes ACP bridge fallback sessions from `acp:<uuid>` to `acp-bridge:<uuid>`, adds raw/canonical pending-prompt matching, and treats terminal child sessions as missing backing sessions during task maintenance.

## Best Possible Solution

Land a revised PR or replacement that separates ACP bridge fallback keys from ACP runtime-session keys, matches raw/canonical Gateway keys, updates ACP docs, and preserves cron runtime authority while leaving broader CLI zombie cleanup to https://github.com/openclaw/openclaw/issues/73121.

## Reproduction Assessment

Yes, source-level. Current main still maps metadata-free ACP bridge sessions to `acp:<uuid>` and exact-matches pending prompt keys before run-id reconciliation; I did not run a live ACP client flow in this read-only review.

## Solution Assessment

No. The ACP fallback-prefix and canonical-key direction is narrow and maintainable, but the branch is not the best complete solution until cron ownership is preserved, docs are updated, and real behavior proof is provided.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve active cron ownership:** `src/tasks/task-registry.maintenance.ts:159-170`
  - body: Including `cron` in this child-session branch makes active cron tasks depend on a stored child session row. If that row is terminal or missing, maintenance can mark the task lost even while `isCronJobActive(jobId)` is true, so keep the cron runtime short-circuit or scope terminal-session handling away from cron ownership.
  - confidence: 0.9

  - confidence: 0.84
