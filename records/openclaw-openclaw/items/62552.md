## Summary

Keep this PR open because current main still has the ACP bridge fallback-key and exact pending-session-key behavior the PR is trying to fix, but the proposed patch is not merge-ready: its cron maintenance change can mark an active cron run lost based on a terminal or missing child session, and the branch is dirty against current main.

## What This Changes

The PR changes ACP bridge fallback sessions to `acp-bridge:<uuid>`, matches pending ACP prompts across raw and Gateway-canonical session keys, and changes task maintenance to treat terminal child sessions as missing for cron, CLI, and subagent tasks with targeted regression tests.

## Best Possible Solution

Land a rebased or replacement patch that keeps ACP bridge fallback sessions on a non-ACP-runtime prefix and matches Gateway-canonical keys, while preserving current main's active-cron liveness contract. The final change should update ACP docs, add the required changelog entry, and cover both ACP key matching and active-cron-with-terminal-child regression cases.

## Reproduction Assessment

Yes. The blocking PR regression can be reproduced conceptually with a stale active cron task whose `sourceId` is still in `activeCronJobIds` and whose `childSessionKey` points to a missing or terminal session entry; the PR's `hasBackingSession` then returns false and maintenance marks it lost.

## Solution Assessment

No. The ACP bridge key direction is plausible, but the cron maintenance hunk is not the narrowest safe fix because it lets child-session state override active cron runtime ownership; preserving the active-cron short-circuit or scoping terminal-child handling away from active cron is safer.
