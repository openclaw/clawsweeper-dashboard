## Summary

Keep open. Current main still has the reported failure shape: completed auto-compaction is tracked and the prompt can be retried, but a later mixed timeout plus unavailable-fallback `FallbackSummaryError` with verbose details off still formats through the generic external-run failure path. The related reports do not supersede this issue: #58957 is broader model-switch/context UX, closed #20910 covered all-model timeout reset before this post-compaction evidence, and #74073 is a separate subagent-output race.

## What This Changes

The issue requests a compaction- and timeout-aware user-facing recovery message when an embedded run successfully auto-compacts, retries, then hits an LLM idle timeout and exhausts unavailable fallbacks.

## Best Possible Solution

Carry structured completed-compaction state into the external-run failure formatting path and add narrowly scoped recovery copy for post-compaction retry timeouts with exhausted or unavailable fallbacks, while leaving broader model-switch UX and unrelated subagent-output races to their own reports.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is available by making `runAgentTurnWithFallback` observe completed compaction state, then reject with a mixed `FallbackSummaryError` containing a timeout attempt plus unavailable or billing fallback attempts while `resolvedVerboseLevel` is off; the current mixed-cause test already proves the final output is generic.

## Solution Assessment

Yes. The best fix is a narrow formatter/state-plumbing change in the embedded reply-runner path with focused regression coverage; auto-resetting sessions or changing global fallback policy would be broader than the reported messaging bug requires.
