## Summary

Keep open. Current main still preserves the reported failure shape: post-compaction timeout/fallback exhaustion can flow into the generic external-run failure formatter when verbose details are off, and the existing mixed-cause fallback test still pins that generic `/new` text.

## What This Changes

Report that a successful embedded-agent auto-compaction followed by a retried LLM idle timeout and exhausted/unavailable fallbacks is shown to users as a generic `/new` failure.

## Best Possible Solution

Preserve completed-compaction state and fallback-attempt causes in embedded reply failure formatting, with focused regression coverage for post-compaction timeout plus unavailable or billing fallback exhaustion.

## Reproduction Assessment

Yes, source-level. Mock `runWithModelFallback` to throw a mixed `FallbackSummaryError` after a completed compaction has been observed, with `resolvedVerboseLevel: "off"`; the current mixed-cause test already proves that this class of summary formats to the generic `/new` text.

## Solution Assessment

Yes. The narrow maintainable fix is to plumb completed-compaction state into the existing failure formatter and use `FallbackSummaryError.attempts` for a specific recovery message; auto-resetting sessions or changing global fallback policy would be broader than this bug requires.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/67750 on current main. When an embedded agent has completed auto-compaction and the retried run later fails with timeout plus exhausted or unavailable fallback attempts while verbose details are off, the final user-facing reply must preserve the useful cause chain: compaction was attempted and completed, the retried model call timed out, fallbacks were unavailable/billing/rate-limited as applicable, and the channel/gateway transport is not implied broken. Keep the change scoped to `src/auto-reply/reply/agent-runner-execution.ts` formatter/state plumbing and focused tests. Use `FallbackSummaryError.attempts` rather than string-only parsing where possible. Do not add a config option, auto-reset sessions, or change global fallback/model-switch policy. Add or update a user-facing changelog entry if the repair changes reply text.
