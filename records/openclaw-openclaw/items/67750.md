## Label Justifications

- `P2`: This is a normal-priority user-facing bug in long-session agent failure reporting with a clear, limited implementation surface.
- `impact:session-state`: The failure depends on accumulated session context, completed compaction, and continued use of the same long-running session.
- `impact:auth-provider`: The terminal failure combines model timeout with unavailable fallback providers, so provider routing and fallback attempt classification are central to the bug.

## Summary

Keep open. Current main still tracks completed compaction, but the terminal embedded-run failure formatter does not receive that state, so mixed timeout plus unavailable-fallback exhaustion can still collapse to the generic `/new` failure text when verbose details are off.

## What This Changes

After successful auto-compaction, post-compaction timeout plus exhausted fallbacks should produce a compaction/timeout-aware recovery message instead of the generic `/new` fallback.

## Best Possible Solution

Thread completed-compaction state into the existing failure reply decision and build sanitized timeout/fallback-aware copy from structured fallback attempts, without changing reset policy, fallback selection, or global compaction behavior.

## Reproduction Assessment

Yes, at source level. A focused test can emit a completed compaction event, reject the fallback runner with timeout plus unavailable-fallback `FallbackSummaryError` under verbose-off settings, and observe the generic failure text path.

## Solution Assessment

Yes for the narrow issue. The maintainable fix is to thread completed-compaction state into existing failure formatting and consume structured fallback attempt reasons; auto-resetting sessions or changing global fallback policy is broader than this report requires.
