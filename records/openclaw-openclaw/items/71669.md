## Summary

Keep open. Current main still enforces and documents the Control UI secure-context/device-identity rejection, but the gateway startup logger only emits the generic dangerous-config warning and lacks the requested targeted warning; open PR #73163 already uses closing syntax for the same change and should be reviewed, landed, replaced, or closed first.

## What This Changes

Request a gateway startup warning when Control UI is enabled on non-secure, non-localhost access with device auth still required, so operators understand why remote plain-HTTP connections are rejected with device-identity errors.

## Best Possible Solution

Land PR #73163 or an equivalent narrow gateway startup-log change with regression tests. The warning should use the existing startup inputs to identify enabled Control UI over non-loopback plain HTTP with device auth still required, and it should not alter the WebSocket handshake or device-identity enforcement.

## Reproduction Assessment

Yes. Source and test inspection give a high-confidence reproduction path: Control UI skips device identity when crypto.subtle is unavailable, the gateway rejects that device-less Control UI connect with CONTROL_UI_DEVICE_IDENTITY_REQUIRED, and current startup logging has no targeted warning for the configuration.

## Solution Assessment

Yes. A startup warning is the narrowest maintainable fix for the reported operator confusion because it adds server-side guidance without relaxing authentication; broader remote-HTTP support belongs in a separate product/security discussion such as #53274.
