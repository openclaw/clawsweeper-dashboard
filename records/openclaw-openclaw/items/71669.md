## Summary

Keep open. Current main still enforces and documents the non-secure remote Control UI rejection, but the gateway startup logger does not emit the requested targeted warning; open PR #73163 already uses closing syntax for this issue and should be reviewed, landed, replaced, or closed before the issue is resolved.

## What This Changes

Request a gateway startup warning when Control UI is enabled and remote plain-HTTP/non-secure connections will be rejected because device identity remains required.

## Best Possible Solution

Add a narrow startup-log warning in the gateway startup path using the existing config, bind, and TLS inputs, with regression tests for warning and quiet cases. The implementation should preserve current WebSocket/device-identity enforcement and treat any change to remote HTTP support as separate product/security work.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a non-secure remote browser skips device identity, the gateway rejects the Control UI connection with CONTROL_UI_DEVICE_IDENTITY_REQUIRED, and the current startup logger has no targeted warning for that configuration.

## Solution Assessment

Yes. A startup warning is the narrowest maintainable fix for the reported operator confusion because it adds server-side guidance without relaxing authentication; PR #73163 is the active implementation path to review or replace.
