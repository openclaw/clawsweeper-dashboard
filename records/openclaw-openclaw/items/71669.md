## Summary

Keep open. Current main still enforces and documents the non-secure remote Control UI rejection, but the gateway startup logger does not emit the requested targeted warning, and the linked implementation PR has not merged.

## What This Changes

Request to add a gateway startup warning when enabled Control UI over non-secure non-loopback HTTP will reject device-less connections unless the break-glass bypass is set.

## Best Possible Solution

Land or replace PR #73163 with a narrow startup-log warning and regression tests while preserving the existing Control UI device-identity enforcement.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: non-secure remote Control UI skips browser device identity, the gateway rejects that connection with CONTROL_UI_DEVICE_IDENTITY_REQUIRED, and startup logging has no targeted guidance for that configuration.

## Solution Assessment

Yes. A log-only startup warning is the narrowest maintainable fix for the reported operator confusion because it adds guidance without weakening authentication or WebSocket policy.
