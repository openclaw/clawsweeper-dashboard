## Summary

Keep open. Current main still enforces and documents the remote HTTP Control UI device-identity rejection, but the gateway startup logger does not emit the requested targeted warning; the open closing PR #73163 should be reviewed, landed, replaced, or closed before this issue is closed.

## What This Changes

Request a gateway startup warning when Control UI is enabled in a configuration where remote non-secure HTTP browsers will be rejected for missing device identity.

## Best Possible Solution

Land or replace #73163 with a narrow startup-log warning and regression tests while preserving the existing Control UI device-identity enforcement unchanged.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a non-secure remote browser skips device identity, the gateway rejects the Control UI connection with CONTROL_UI_DEVICE_IDENTITY_REQUIRED, and current startup logging has no targeted warning for that configuration.

## Solution Assessment

Yes. A log-only startup warning is the narrowest maintainable solution for the reported operator confusion because it adds server-side guidance without relaxing authentication or changing the WebSocket policy.
