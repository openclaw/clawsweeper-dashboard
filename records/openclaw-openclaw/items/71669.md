## Summary

Keep open: current main and the latest release still preserve the remote HTTP secure-context/device-identity rejection without the requested startup warning, and the direct fix path is the still-open, conflicting pull request at https://github.com/openclaw/openclaw/pull/73163.

## What This Changes

Request a Gateway startup warning when Control UI is enabled in a mode where remote non-localhost HTTP clients will be rejected for missing secure-context device identity.

## Best Possible Solution

Land or replace a narrow Gateway startup warning that reflects the resolved bind/TLS/auth state, points users to HTTPS, localhost, trusted-proxy, or break-glass docs, and preserves the secure-context/device-identity boundary.

## Reproduction Assessment

Yes, by source inspection: current main skips device identity when `crypto.subtle` is unavailable, rejects remote device-less Control UI, and startup logging lacks a matching warning. I did not run a live Docker/LAN browser repro in this read-only review.

## Solution Assessment

Mostly yes: a startup warning is the safest narrow mitigation for this DX gap, but exact conditions and wording should be settled in the linked PR to avoid weakening the secure-context boundary.
