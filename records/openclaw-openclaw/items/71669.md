## Summary

Keep open. Current main still documents and enforces the insecure Control UI/device-identity rejection, but the gateway startup logger does not yet emit the requested operator warning; open PR #73163 already targets this issue with closing syntax and should be reviewed, landed, replaced, or closed first.

## What This Changes

The issue requests a gateway startup warning when Control UI is enabled in a configuration that will reject non-localhost plain HTTP clients because device identity is required.

## Best Possible Solution

Land PR #73163 or an equivalent narrow gateway startup-log change with regression tests. The warning should use existing startup inputs to identify enabled Control UI over non-loopback plain HTTP with device auth still required, while leaving WebSocket authentication and device-identity enforcement unchanged.

## Reproduction Assessment

Yes. Source and test inspection give a high-confidence path: a non-secure browser skips device identity, the gateway rejects a device-less remote Control UI connect with CONTROL_UI_DEVICE_IDENTITY_REQUIRED, and current startup logging has no targeted warning for that configuration.

## Solution Assessment

Yes. A startup warning is the narrowest maintainable fix for the reported operator confusion because it adds server-side guidance without weakening authentication; broader remote-HTTP behavior belongs in separate product/security work such as #53274.
