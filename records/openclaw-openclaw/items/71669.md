## Summary

Keep open. Current main still enforces and documents the insecure remote Control UI/device-identity rejection, but the gateway startup logger does not emit the requested targeted warning; open PR #73163 already uses closing syntax for this issue and should be reviewed, landed, replaced, or closed before this issue is closed.

## What This Changes

Request a gateway startup warning when Control UI is enabled for non-loopback plain HTTP without the device-auth break-glass flag, because affected remote browsers are rejected without server-side guidance.

## Best Possible Solution

Add a narrow gateway startup-log warning using the existing startup config, bind, and TLS inputs, with regression tests for warning and quiet cases. The warning should guide users toward HTTPS, localhost or SSH tunnel, trusted proxy where appropriate, and the break-glass flag, while preserving current WebSocket/device-identity enforcement unchanged.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a non-secure browser skips device identity, the gateway rejects a remote Control UI connect with CONTROL_UI_DEVICE_IDENTITY_REQUIRED, and current startup logging has no targeted warning for that configuration.

## Solution Assessment

Yes. A startup warning is the narrowest maintainable fix for the reported operator confusion because it adds server-side guidance without relaxing authentication; broader changes to remote HTTP/device identity belong in separate product and security work such as #53274.
