## Summary

Keep open. Current main still rejects device-less remote plain-HTTP Control UI connections and documents the secure-context requirement, but the gateway startup log does not yet emit the requested targeted warning. Open PR #73163 already uses closing syntax for this issue, so the issue should remain open until that PR is reviewed and landed, replaced, or closed.

## What This Changes

The issue requests a gateway startup warning when Control UI is enabled but remote plain-HTTP browser connections will be rejected because the browser cannot provide device identity in a secure context.

## Best Possible Solution

Add a focused gateway startup warning in the existing startup logging path, using the already-available config, bind host(s), and TLS state. Cover warn and quiet cases in startup-log tests, and preserve the current WebSocket/device-identity enforcement unchanged.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: a remote plain-HTTP browser skips device identity, the gateway rejects the Control UI handshake with CONTROL_UI_DEVICE_IDENTITY_REQUIRED, and current startup logging has no targeted warning for that configuration.

## Solution Assessment

Yes. A startup warning is the narrowest maintainable solution for the reported operator confusion because it adds server-side guidance without relaxing authentication; the already-linked PR #73163 is the active implementation path to review.
