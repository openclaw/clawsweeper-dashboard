## Summary

Keep open. Current main still rejects the reported non-secure Control UI path and documents the behavior, but gateway startup logging only emits generic startup and dangerous-config warnings, not the requested warning. The issue is also paired with open, unmerged PR #73163 using closing syntax, so it should remain open until that PR lands, is replaced, or is closed.

## What This Changes

Request a gateway startup warning when Control UI is enabled for insecure remote access with device auth required, so operators understand that non-localhost plain-HTTP clients will be rejected.

## Best Possible Solution

Review and land PR #73163 or an equivalent narrow gateway startup-log change with focused regression tests, while preserving the current Control UI device-identity rejection policy. Broader changes to allow device identity over non-secure HTTP should stay with #53274 or a dedicated security/product thread.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: open the Control UI from a non-localhost plain-HTTP browser so crypto.subtle is unavailable, the UI sends no device identity, and the gateway rejects the connection with CONTROL_UI_DEVICE_IDENTITY_REQUIRED while startup logging has no targeted warning.

## Solution Assessment

Yes. A startup warning is the narrowest maintainable fix for this issue because it improves operator guidance without weakening the current auth policy; the existing open PR appears aligned pending normal review.
