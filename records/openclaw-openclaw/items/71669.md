## Summary

Keep open. Current main still documents and enforces the remote HTTP Control UI device-identity rejection, but the gateway startup logger does not emit the requested targeted warning; open PR #73163 already uses closing syntax for this issue and should be reviewed, landed, replaced, or closed before this issue is closed.

## What This Changes

Request a gateway startup warning when enabled Control UI over non-loopback plain HTTP will reject device-less browser connections unless device auth is explicitly disabled.

## Best Possible Solution

Land or replace #73163 with a narrow startup-log warning and regression tests while preserving current Control UI device-identity enforcement unchanged.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a non-secure remote browser skips device identity, the gateway rejects the Control UI connection with CONTROL_UI_DEVICE_IDENTITY_REQUIRED, and current startup logging has no targeted warning for that configuration.

## Solution Assessment

Yes. A log-only startup warning is the narrowest maintainable solution for the reported operator confusion because it adds server-side guidance without relaxing authentication or changing the WebSocket policy.
