## Summary

Keep open. Current main still has the Control UI secure-context/device-identity rejection path and docs, but the gateway startup logger does not emit the requested targeted warning; an open, unmerged PR already references this issue with closing syntax, so this should stay open until that PR lands, is replaced, or is closed.

## What This Changes

The issue requests a gateway startup warning when Control UI is enabled over non-loopback HTTP while device identity remains required.

## Best Possible Solution

Land PR #73163 or an equivalent narrow gateway startup-log change with regression tests, while keeping the existing Control UI device-identity enforcement intact.

## Reproduction Assessment

Yes. Source and test inspection give a high-confidence path: a non-localhost plain-HTTP Control UI lacks crypto.subtle, sends no device identity, and the gateway rejects it with CONTROL_UI_DEVICE_IDENTITY_REQUIRED while startup logging has no targeted warning.

## Solution Assessment

Yes. A startup warning is the narrowest maintainable solution for this issue because it improves operator guidance without weakening authentication; broader remote-HTTP support should remain separate.
