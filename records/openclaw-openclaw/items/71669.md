## Summary

Keep open. Current `main` and latest release still enforce/document the remote plain-HTTP Control UI rejection but do not emit the requested startup warning, and the issue is paired with the open closing PR https://github.com/openclaw/openclaw/pull/73163.

## What This Changes

The issue requests a gateway startup warning when Control UI is enabled in a configuration where remote plain-HTTP browser connections will be rejected for lacking device identity.

## Best Possible Solution

Land or replace the linked PR with a narrow startup-log warning and focused regression tests while preserving the current Control UI device-identity enforcement.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: remote plain HTTP lacks `crypto.subtle`, the UI skips device identity, and the gateway rejects the device-less Control UI connection with `CONTROL_UI_DEVICE_IDENTITY_REQUIRED` while startup logging has no targeted warning.

## Solution Assessment

Yes for the narrow diagnostic direction. A startup warning is the maintainable fix for operator confusion because it adds guidance without relaxing authentication; the existing linked PR is the right place to finish or replace that work.
