## Summary

Keep open. Current main still enforces and documents the remote HTTP Control UI rejection path, but gateway startup logging only emits the normal ready/log-file/Nix/dangerous-config messages; the linked #73163 uses closing syntax and should be reviewed, landed, replaced, or closed before this issue closes.

## What This Changes

Request to emit a gateway startup warning when Control UI is enabled in a configuration where non-localhost HTTP browsers will be rejected for missing device identity.

## Best Possible Solution

Land or replace #73163 with a log-only startup warning and regression tests while preserving the existing Control UI device-identity enforcement unchanged.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: non-secure remote HTTP skips browser device identity, the gateway rejects the Control UI connection with CONTROL_UI_DEVICE_IDENTITY_REQUIRED, and startup logging has no targeted warning for that configuration.

## Solution Assessment

Yes. A startup warning is the narrowest maintainable fix for the operator confusion because it adds server-side guidance without relaxing WebSocket authentication or device-identity policy.
