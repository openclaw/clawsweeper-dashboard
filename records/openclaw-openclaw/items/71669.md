## Summary

Keep open. Current main still rejects non-localhost plain-HTTP Control UI connections without device identity and documents that behavior, but the gateway startup logger does not yet emit the requested targeted warning. An open, unmerged PR already references this issue with closing syntax, so the issue should remain open until that PR lands, is replaced, or is closed.

## What This Changes

The issue requests a gateway startup warning when Control UI is enabled in a non-secure remote HTTP configuration that will later reject device-identity-required connections.

## Best Possible Solution

Review and land PR #73163 or an equivalent narrow gateway startup-log change with regression tests, while keeping the existing Control UI device-identity enforcement intact and leaving broader remote-HTTP/device-identity redesign to separate issues such as #53274.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a non-localhost plain-HTTP Control UI lacks crypto.subtle, sends no device identity, and the gateway rejects it with CONTROL_UI_DEVICE_IDENTITY_REQUIRED while startup logging lacks the requested targeted warning.

## Solution Assessment

Yes. A startup warning is the narrowest maintainable fix for this issue because it improves operator guidance without weakening authentication; the existing logGatewayStartup call already receives the needed bind and TLS context.
