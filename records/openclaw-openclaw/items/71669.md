## Summary

Keep open. Current main still has the Control UI secure-context/device-identity rejection behavior and user-facing docs, but the gateway startup logger does not emit the requested targeted warning. An open, unmerged PR already references this issue with closing syntax, so this issue should remain open until that PR lands, is replaced, or is closed.

## What This Changes

The issue requests a gateway startup warning when Control UI is enabled, device auth remains required, and non-localhost plain-HTTP access will be rejected because the browser cannot provide device identity.

## Best Possible Solution

Land PR #73163 or an equivalent narrow gateway startup-log change with regression tests and a user-facing changelog entry, while preserving the current secure-context/device-identity rejection policy.

## Reproduction Assessment

Yes. Source and test inspection provide a high-confidence path: a non-localhost plain-HTTP Control UI can skip device identity because `crypto.subtle` is unavailable, and the gateway rejects that connection with `CONTROL_UI_DEVICE_IDENTITY_REQUIRED`; the startup logger has no matching targeted warning.

## Solution Assessment

Yes. A startup warning is the narrowest maintainable fix for this issue because it improves operator guidance without changing authentication semantics; broader remote-HTTP/device-identity changes belong in separate issues such as #53274.
