## Summary

Keep open: current main still has the reported Codex app-server bridge gap. Declined native tools are recognized as blocked tool events, but the projector result still does not populate `lastToolError`, so the existing terminal payload error path can be bypassed for an aborted empty turn.

## What This Changes

The issue requests that declined or blocked Codex app-server native tool executions surface a visible terminal error reply instead of silently producing no external-channel response.

## Best Possible Solution

Add a narrow Codex app-server adapter fix that records blocked or failed native tool completions as attempt-level `lastToolError`, then prove the existing payload layer emits a visible error reply.

## Reproduction Assessment

Yes, as a source-level reproduction: a declined `commandExecution` becomes a blocked native tool event, but current `buildResult` does not carry that non-success state into `lastToolError`. I did not run a live external-channel reproduction in this read-only pass.

## Solution Assessment

Yes. The narrowest maintainable fix is to bridge blocked/failed native Codex tool items into the existing `lastToolError` payload path, without adding Telegram-specific fallback policy or a new config option.
