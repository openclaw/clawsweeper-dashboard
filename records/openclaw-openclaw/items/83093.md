## What This Changes

The issue requests that declined or blocked Codex app-server native tool completions surface a visible external-channel error instead of ending an empty turn silently.

## Best Possible Solution

Review the existing linked fix PRs and land one narrow implementation that records non-success Codex native tool completions as `lastToolError` with regression coverage.

## Reproduction Assessment

Yes, from source inspection rather than a live run: a declined `commandExecution` becomes a blocked `bash` tool event, but current `buildResult` does not carry it into `lastToolError`. The downstream payload path only emits the visible tool-error reply when that field is present.

## Solution Assessment

Yes. The narrowest maintainable fix is to reuse the existing `ToolErrorSummary` and `lastToolError` payload path rather than adding Telegram-specific or channel-specific fallback behavior.
