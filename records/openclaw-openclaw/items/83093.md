## Summary

Keep open: current main still has the reported gap. Declined Codex native tools are projected as blocked events, but the app-server attempt result does not carry that non-success native tool into `lastToolError`, so the existing payload error path can be bypassed for an aborted, empty turn.

## What This Changes

The issue requests that declined or blocked Codex app-server native tool executions produce a visible external-channel failure reply instead of ending silently.

## Best Possible Solution

Keep this open for a narrow Codex app-server fix that converts blocked or failed native tool completions into attempt-level `lastToolError` and proves the visible payload behavior with regression tests.

## Reproduction Assessment

Yes, from source inspection rather than a live run: a declined `commandExecution` becomes a blocked native tool event, but `buildResult` does not set `lastToolError`, and the aborted empty terminal path can return no payload.

## Solution Assessment

Yes. The narrowest maintainable fix is to record non-success native Codex tool completions in the projector and pass them through the existing `lastToolError` payload path, without adding channel-specific Telegram behavior or new policy.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/83093. Current main maps declined Codex app-server native `commandExecution` items to blocked tool events, but `CodexAppServerEventProjector.buildResult` does not populate `EmbeddedRunAttemptResult.lastToolError`, so an aborted turn with no assistant text can produce no external-channel reply. Implement the smallest owner-boundary fix in the Codex app-server projector/result plumbing: record the latest non-success native tool completion (`blocked` or `failed`) as a `ToolErrorSummary` and include it in the attempt result, with declined `commandExecution` represented as a `bash` error like `codex native tool blocked` and mutating action metadata when appropriate. Reuse the existing `buildEmbeddedRunPayloads` error path; do not add Telegram-specific behavior, new config, core discovery, permission-policy changes, or broad retries. Add regression coverage for declined native command item -> `lastToolError` and aborted/no-assistant-text + native tool error -> visible error payload. Add a changelog entry because this changes user-visible external-channel failure behavior.
