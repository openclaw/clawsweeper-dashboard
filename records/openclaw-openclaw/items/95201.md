## Summary

Keep open: the latest head no longer raw-reseeds across auth boundaries, but it now resumes a stored CLI session whenever the auth profile id is unchanged, even though current auth-epoch code intentionally changes on same-profile token/account changes. That needs maintainer/security rework before merge.

## What This Changes

The PR changes CLI session reuse so an auth-epoch mismatch with the same auth profile still returns the stored CLI session id, and updates the matching unit expectation.

## Best Possible Solution

Land a maintainer-approved auth-identity fix that preserves Claude CLI session reuse only when unchanged identity is proven, while keeping raw transcript replay blocked for real or unproven auth changes.

## Reproduction Assessment

Yes, at source level. Current main can invalidate a CLI session on auth-profile/auth-epoch mismatch and keep raw reseed blocked for those reasons; I did not run a live claude-cli auth invalidation in this read-only review.

## Solution Assessment

No. Avoiding raw transcript replay is the right boundary, but returning a session id for every same-profile auth-epoch mismatch is too broad because the auth epoch is the credential identity guard.
