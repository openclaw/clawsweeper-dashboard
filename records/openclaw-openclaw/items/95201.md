- `P1`: The PR targets a real Claude CLI continuity regression but currently risks breaking the auth/session boundary for user CLI runs.
- `merge-risk: 🚨 auth-provider`: The diff can reuse a CLI session after same-profile credential, token, or OAuth identity changes that the auth epoch was meant to detect.
- `merge-risk: 🚨 session-state`: The diff can keep prior CLI session state attached to a run whose auth epoch no longer matches current state.
- `merge-risk: 🚨 security-boundary`: Resuming a stored CLI session after an unproven auth identity change can carry conversation context across a credential boundary.

## Summary

Keep open, but do not merge as-is: the patch treats a same auth profile id as proof of the same credential identity, while current auth-epoch code and docs use epoch mismatches to detect same-profile static key, token, or OAuth identity changes. Returning the old CLI session id on that mismatch would resume stale CLI sessions across an auth boundary.

## What This Changes

This PR changes `resolveCliSessionReuse` so an auth-epoch mismatch with the same auth profile returns the stored CLI `sessionId`, and updates the unit test expectation.

## Best Possible Solution

Keep auth-epoch mismatches fail-closed unless the auth epoch layer or a backend-specific mode proves identity equivalence, then land that canonical fix with focused tests and real Claude CLI proof.

## Reproduction Assessment

Yes, at source level. The PR returns `sessionId` for an auth-epoch mismatch, and current prepare/execute code uses that session id to resume the CLI run; I did not run a live Claude CLI auth invalidation in this read-only review.

## Solution Assessment

No. The patch is too broad because same profile id is not proof of same credential identity; the safer fix belongs in auth-epoch identity equivalence or a backend-specific host-only contract.
