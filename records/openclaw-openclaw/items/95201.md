## Summary

Keep open: the current patch is not safe to merge because it treats a same-profile auth-epoch mismatch as reusable, even though the epoch is the credential/identity boundary that should force a reset unless unchanged identity is proven upstream.

## What This Changes

The PR changes CLI session reuse so an auth-epoch mismatch keeps the stored session id when the auth profile id is unchanged, with a matching test expectation update.

## Best Possible Solution

Keep auth-epoch mismatches fail-closed in session reuse, and solve cosmetic Claude CLI rotations by making the auth-epoch calculation stable only when the host CLI identity is proven unchanged.

## Reproduction Assessment

Yes, at source level. With the patch, a same-profile authEpoch mismatch returns a sessionId and prepareCliRunContext then resumes that CLI session instead of treating the epoch change as a reset; I did not run a live Claude CLI auth invalidation.

## Solution Assessment

No. The patch is a plausible symptom fix but not the best fix because the reuse layer cannot prove that a changed auth epoch is the same identity; the safer fix belongs in auth-epoch identity calculation or the canonical auth-identity PR path.

  - body: `authEpoch` is the credential-derived boundary for reusable CLI sessions.
