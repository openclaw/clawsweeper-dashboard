## Summary

The PR should stay open: it targets a real claude-cli continuity bug, but the patch changes auth-epoch session reuse semantics and still lacks real patched claude-cli auth-boundary proof.

## What This Changes

The branch changes `resolveCliSessionReuse` so an auth-epoch mismatch returns the existing `sessionId` with `invalidatedReason: "auth-epoch"`, and updates the helper test expectation.

## Best Possible Solution

Keep raw transcript replay blocked across auth reasons, and land the maintainer-approved auth identity fix with redacted live proof that cosmetic rotation/token refresh preserves continuity while real credential changes still invalidate.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source inspection shows the current-main path that drops `sessionId` on auth-epoch invalidation and the PR path that would resume it, but this review did not reproduce a live claude-cli auth invalidation end to end.

## Solution Assessment

Is this the best way to solve the issue? No; it is a plausible symptom fix, but the reuse layer cannot prove that an auth-epoch change is the same identity, so the safer fix likely belongs in auth-epoch identity calculation or a maintainer-approved sibling path.
