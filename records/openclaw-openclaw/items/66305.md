## Summary

Keep open: the source-level performance problem is still plausible for non-read-only catalog loads, but this branch is not mergeable as-is because it predates current read-only and full-catalog retry contracts and lacks real runtime proof.

## What This Changes

The PR adds a 60-second negative cache for empty `loadModelCatalog()` results plus regression tests for reuse, bypass, stale concurrent loads, and retry after expiry.

## Best Possible Solution

If maintainers still want bounded core negative caching, port it narrowly onto current `main` behind the non-read-only path, preserve full-catalog retry semantics, add focused coverage for both contracts, and require redacted real runtime proof.

## Reproduction Assessment

Yes, at source level. Mock `ModelRegistry.getAll()` to return `[]` and call non-read-only `loadModelCatalog()` twice; current `main` clears `modelCatalogPromise` after the first empty result, so the second call can re-enter discovery.

## Solution Assessment

No, not as-is. The bounded negative cache may be useful, but this branch predates current read-only and gateway full-catalog retry contracts and needs maintainer scope judgment plus a careful rebase before it is the right fix.
