## Summary

Keep open: the PR targets a real source-level cache issue, but current main has newer read-only and gateway catalog-cache contracts that this stale branch can break, and the contributor still has not provided real runtime proof.

## What This Changes

The PR adds a 60-second negative cache for empty `loadModelCatalog()` results and regression tests for cache reuse, bypass, stale concurrent loads, and retry after expiry.

## Best Possible Solution

Keep the shipped normal-turn fast path for the linked user issue; if maintainers still want bounded empty-catalog caching, rebase it narrowly onto current main without changing read-only `models.list` or full-catalog retry semantics.

## Reproduction Assessment

Yes, at source level. Mock `ModelRegistry.getAll()` to return `[]` and call non-read-only `loadModelCatalog()` twice; current main clears `modelCatalogPromise` after the first empty result, so the second call can re-enter discovery.

## Solution Assessment

No, not as-is. The bounded cache may be useful, but this branch predates current read-only and gateway full-catalog retry contracts and needs a scoped rebase plus real runtime proof before it is the right fix.
