## Summary

Keep this PR open: the empty-catalog cache bug is still present on current main, but the branch is dirty against newer read-only model-catalog semantics and should not be merged as-is.

## What This Changes

The PR adds a bounded negative cache for successful empty `loadModelCatalog()` results and regression tests for reuse, bypass, stale concurrent loads, and retry after the window.

## Best Possible Solution

Port the negative-cache behavior onto current main behind the existing non-read-only cache gates, preserving explicit cache resets, error retryability, stale-load guards, read-only side-effect-free catalog loading, and focused regression coverage.

## Reproduction Assessment

Yes. A source-level reproduction is to mock `ModelRegistry.getAll()` to return `[]` and call non-read-only `loadModelCatalog()` twice; current main clears `modelCatalogPromise` after the first empty result, so the second call re-enters discovery.

## Solution Assessment

No, not as-is. The negative-cache approach is the right narrow fix, but this branch must be rebased or replaced so the newer read-only model-catalog contract remains intact.
