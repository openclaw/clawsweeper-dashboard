## Summary

Keep this PR open. Current main still immediately clears successful empty `loadModelCatalog()` results, so the branch targets a real remaining cache behavior, but the patch must be ported onto the newer read-only catalog contract and needs the required changelog entry before merge.

## What This Changes

The PR adds a 60-second negative cache for successful empty `loadModelCatalog()` results plus focused regression tests for reuse, `useCache: false`, stale concurrent loads, and retry after the window.

## Best Possible Solution

Port the negative-cache behavior onto current main with a bounded retry window for ordinary cached loads while preserving read-only non-mutating behavior, explicit cache resets, error retryability, stale-load guards, configured-model merging, and focused regression coverage.

## Reproduction Assessment

Yes. A unit-level reproduction is to mock `ModelRegistry.getAll()` to return `[]` and call `loadModelCatalog()` twice with cache enabled; current main clears the cache after the first empty result, so the second call re-enters discovery.

## Solution Assessment

No, not as-is. The negative-cache idea is the right narrow fix, but this branch predates current read-only catalog semantics and should be rebased or replaced with those guards preserved.
