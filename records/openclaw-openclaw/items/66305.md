## Summary

Keep open for maintainer/contributor follow-up, not cleanup close. The linked user issue is already closed by a shipped MiniMax/model-selection fast path, but current `main` still clears empty non-read-only `loadModelCatalog()` results; this PR’s core idea is therefore not fully implemented, while the branch is now conflicting, lacks real runtime proof, and needs a current-main policy decision because newer gateway tests intentionally keep full-catalog empty results retryable.

## What This Changes

The PR adds a 60-second negative cache for empty `loadModelCatalog()` results plus regression tests for reuse, bypass, stale concurrent loads, and retry after expiry.

## Best Possible Solution

Have a maintainer decide whether a bounded core negative cache is still wanted after the shipped MiniMax fast path and gateway retry fix; if yes, port it onto current main behind the non-read-only path with coverage that preserves read-only and full-catalog retry semantics.

## Reproduction Assessment

Yes, at source level. Mock `ModelRegistry.getAll()` to return `[]` and call non-read-only `loadModelCatalog()` twice; current main clears `modelCatalogPromise` after the first empty result, so the second call re-enters discovery.

## Solution Assessment

No, not as-is. The bounded negative cache may be a useful narrow fix, but this branch must be rebased and scoped against the newer read-only and gateway full-catalog retry contracts before it is mergeable.
