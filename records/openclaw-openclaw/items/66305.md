## Summary

Keep open: current main still clears successful empty non-read-only catalog loads, so the negative-cache fix remains useful, but this branch predates current read-only model-catalog semantics and should not merge as-is.

## What This Changes

The PR adds a 60-second negative cache for successful empty `loadModelCatalog()` results and regression tests for reuse, `useCache: false`, stale concurrent loads, and retry after the window.

## Best Possible Solution

Port the negative-cache behavior onto current main only for ordinary cached non-read-only loads, preserving read-only fast paths, explicit cache bypass, error retryability, stale-load guards, configured-model merging, and focused regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence unit reproduction: mock `ModelRegistry.getAll()` to return `[]`, call `loadModelCatalog()` twice with cache enabled, and current main clears `modelCatalogPromise` after the first empty non-read-only result.

## Solution Assessment

No, not as-is. The negative-cache direction is the narrow maintainable fix, but this branch must be rebased or replaced so the new cache state stays behind current `!readOnly` guards.
