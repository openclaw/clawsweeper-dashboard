## Summary

Keep open. Current main mitigates the original 600s default by raising the default agent runtime timeout to 48 hours and adding model idle-timeout handling, but the embedded run still uses a single wall-clock abort timer when a run timeout is configured, and the remaining idle-vs-max timeout contract needs maintainer product judgment.

## What This Changes

The issue requests activity-aware gateway run timeout behavior so active multi-tool runs are not killed only because they exceed a fixed wall-clock timeout.

## Best Possible Solution

Define an explicit run-timeout contract, likely separate idle and absolute max budgets, then update the agent config schema, docs, embedded-runner timer logic, and focused tests while keeping the Codex dynamic-tool watchdog repair separate.

## Reproduction Assessment

Yes, source-reproducible rather than live reproduced: configure a finite run timeout and keep an embedded run active past it, and current main schedules a single abort timer without activity resets. The specific 600s default path is outdated because unset timeouts now default to 48 hours.

## Solution Assessment

Unclear. The issue discussion’s idle-timeout plus absolute-cap direction looks safer than unlimited runs, but it changes the public timeout contract and should be chosen by maintainers before implementation.
