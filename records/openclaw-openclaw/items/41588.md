## Summary

Keep open: current main and the latest release mitigate the original 600s default with a 48h default and clearer timeout reporting, but the core gateway and embedded-runner paths still use fixed run deadlines rather than an activity-aware idle-plus-hard-cap timeout contract. The closed liveness umbrella explicitly points back here as the canonical core tracker, and the open Telegram PR is a channel-specific mitigation rather than the core fix.

## What This Changes

Request activity-aware agent run timeout semantics so active multi-tool turns are not killed solely by a flat `agents.defaults.timeoutSeconds` wall-clock budget.

## Best Possible Solution

Keep this as the canonical core design thread, decide the idle-timeout plus hard-cap semantics, then implement them across gateway registration, maintenance, embedded-runner progress hooks, config/schema/docs, and focused timeout tests.

## Reproduction Assessment

Yes. Source-reproducible: configure a short `agents.defaults.timeoutSeconds` and run a progressing multi-tool turn longer than that budget; current main still has one gateway/runner deadline and no progress reset, though I did not run a live channel reproduction in this read-only review.

## Solution Assessment

No implementation is selected yet. The idle-timeout plus absolute-cap direction is plausible, but the exact semantics and config contract need maintainer decision before a fix PR.
