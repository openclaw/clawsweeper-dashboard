## Summary

Keep open. Current main has mitigations for the original 600s failure mode, but it still does not expose a run-level idle timeout plus absolute cap that is refreshed by model/tool progress.

## What This Changes

Request an activity-aware agent run timeout that distinguishes idle progress gaps from an absolute wall-clock cap for long multi-tool runs.

## Best Possible Solution

Add a run-level idle timeout refreshed by model and tool progress, keep a separate absolute runtime cap, and align schema, help text, docs, and focused runner tests around those semantics.

## Reproduction Assessment

Yes, source-level: configure a short positive agents.defaults.timeoutSeconds or per-run timeout and keep an embedded turn active beyond it; runEmbeddedAttempt arms a single deadline from run start. I did not run a live multi-tool scenario because this review was read-only.

## Solution Assessment

Yes on direction: the best maintainable design is the discussion's separate idle timeout plus absolute maximum runtime. The current 48-hour default, model idle watchdog, and diagnostics liveness recovery are useful mitigations, not the requested run-level contract.
