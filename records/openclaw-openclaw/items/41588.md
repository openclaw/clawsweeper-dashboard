## Summary

Keep open: current main and v2026.6.8 still use a single run deadline with no activity reset; the 48h default and per-run no-timeout override mitigate the original 600s pain, but the requested idle-timeout plus hard-cap behavior remains unresolved.

## What This Changes

Requests activity-aware agent run timeout semantics so active multi-tool runs are not killed solely by a flat wall-clock timeout.

## Best Possible Solution

Keep this as the canonical core timeout design thread, decide idle-timeout plus hard-cap semantics, then implement the contract across gateway abort tracking, embedded-runner progress hooks, config/schema/docs, and focused timeout tests.

## Reproduction Assessment

Yes. Source inspection shows a short configured agents.defaults.timeoutSeconds still produces one run deadline with no reset on tool or model progress; no live channel reproduction was run in this read-only review.

## Solution Assessment

Unclear until maintainers choose the timeout contract. The idle-timeout plus absolute hard-cap direction is the strongest candidate, but the exact reset signals and config behavior need product review before implementation.
