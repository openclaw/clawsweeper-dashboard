## Summary

Keep open. Current main and the latest release still use fixed run deadlines rather than an activity-aware idle timeout with a separate hard cap; the 48-hour default and improved timeout messaging mitigate parts of the original report but do not implement the requested timeout contract.

## What This Changes

The issue requests activity-aware timeout semantics so active multi-tool agent runs are not killed solely by a fixed elapsed-time budget.

## Best Possible Solution

Add a documented idle timeout refreshed by model/tool progress, keep a separate absolute run cap, and route timeout reasons through the normal run/channel reporting path with focused gateway and embedded-runner coverage.

## Reproduction Assessment

Yes, at source level. Configure a positive run timeout and keep an embedded turn active beyond that elapsed budget; current main keeps fixed gateway and embedded-runner deadlines that are not refreshed by ordinary tool or model progress.

## Solution Assessment

Unclear as an implementation because no patch is proposed, but the direction is sound. The safest shape is the discussion's idle-timeout plus absolute-cap design, with maintainer approval for names, defaults, reset events, compatibility, and timeout reporting.
