## Summary

Keep open. Current main has mitigations for the original 600s failure mode, but it still lacks a run-level idle timeout plus separate absolute cap that refreshes on model/tool progress.

## What This Changes

The issue requests activity-aware gateway run timeout semantics so active multi-tool runs are not killed by a flat wall-clock deadline.

## Best Possible Solution

Add a documented run-level idle timeout refreshed by model and tool progress while keeping a separate absolute runtime cap, with schema, help text, docs, and focused runner tests aligned.

## Reproduction Assessment

Yes, at source level: configure a short positive agents.defaults.timeoutSeconds or per-run timeout and keep an embedded run active past it; runEmbeddedAttempt arms a single deadline from run start. I did not run a live multi-tool scenario because this review is read-only.

## Solution Assessment

Yes on direction: the separate idle timeout plus absolute maximum runtime proposed in discussion is the narrowest maintainable contract. The current 48-hour default, model idle watchdog, and diagnostics recovery are useful mitigations, not the requested run-level behavior.
