## Summary

Keep open: current main and v2026.6.10 still implement a single whole-run timeout, while the open related PRs only address narrower symptoms such as timeout notices, Telegram progress, or subagent LLM-idle watchdogs.

## What This Changes

Request activity-aware agent run timeout semantics so active multi-tool turns are not killed solely by one wall-clock deadline.

## Best Possible Solution

Keep this as the canonical core timeout design thread and implement an idle-timeout plus absolute hard-cap contract across gateway abort tracking, embedded-runner progress hooks, config/schema/docs, and focused timeout tests after maintainer approval.

## Reproduction Assessment

Yes, at source level. Current main still turns configured/per-run timeoutSeconds into one gateway expiresAtMs and one embedded-runner abort timer, with no reset on tool or model progress; no live multi-tool run was executed.

## Solution Assessment

Unclear until maintainers choose the timeout contract. Idle timeout plus an absolute hard cap is the strongest current direction, but reset signals, default compatibility, and config shape need product review before implementation.
