## Summary

Keep open: current main and v2026.6.8 still use a single run deadline with no activity reset. The 48h default and explicit per-run no-timeout override mitigate the original 600s failure mode, but they do not implement the requested idle-timeout plus hard-cap semantics.

## What This Changes

The issue requests activity-aware agent run timeout behavior so active multi-tool runs are not killed by a flat wall-clock deadline.

## Best Possible Solution

Keep this as the canonical core timeout design thread, choose an idle-timeout plus absolute hard-cap contract, then implement it consistently across gateway abort tracking, embedded-runner progress hooks, config/schema/docs, and focused timeout tests.

## Reproduction Assessment

Yes. Source inspection shows a short configured run timeout still becomes one run deadline with no reset on tool or model progress; I did not run a live 10+ tool workflow in this read-only review.

## Solution Assessment

Unclear until maintainers choose the timeout contract. Idle-timeout plus absolute hard cap is the strongest current direction, but reset signals, default compatibility, and config shape need product review before implementation.
