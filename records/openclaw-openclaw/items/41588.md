## Summary

Keep open: current main mitigates the old 600s default with a 48h runtime default and explicit timeout payloads, but the core gateway/embedded-runner path still uses a fixed run deadline rather than an activity-aware idle/max timeout contract. The closed liveness umbrella explicitly points back to this issue as the canonical tracker, and the open Telegram PR is channel-specific rather than a core fix.

## What This Changes

The issue requests activity-aware or split idle/max timeouts so active multi-tool agent runs are not terminated by a flat wall-clock deadline.

## Best Possible Solution

Keep this issue as the canonical core design thread, decide the idle/max timeout contract first, then implement it across gateway registration, maintenance, embedded-runner progress hooks, schema/docs, and focused timeout tests.

## Reproduction Assessment

Yes, source-reproducible: configure a short `agents.defaults.timeoutSeconds` and run a progressing multi-tool turn longer than that budget; current main still has one `expiresAtMs`/abort timer and no progress reset. I did not run a live multi-tool channel reproduction.

## Solution Assessment

No implementation is selected yet. The issue’s idle-timeout plus absolute-cap direction is plausible and matches the main discussion, but the exact semantics and config contract need maintainer decision before a fix PR.
