## Summary

Keep open. Current main and v2026.5.20 mitigate the original 600s default by raising the default run timeout to 48h, but the requested idle/max timeout contract is still missing: gateway and embedded-runner deadlines remain fixed wall-clock timers that are not refreshed by tool or model progress.

## What This Changes

Request an activity-aware run timeout for long active multi-tool agent runs, preferably with an idle timeout, an absolute cap, and clearer timeout reporting.

## Best Possible Solution

Add a documented run-level idle timeout refreshed by model/tool progress, retain a separate absolute runtime cap, and surface timeout reason text through the normal run/channel reporting path with focused gateway and embedded-runner tests.

## Reproduction Assessment

Yes, at source level. Configure a positive `agents.defaults.timeoutSeconds` or per-run timeout and keep an embedded turn active past that wall-clock budget; current main keeps fixed gateway and embedded-runner deadlines that are not refreshed by normal tool or model progress.

## Solution Assessment

Unclear as an implementation because no patch is proposed, but the direction is sound. The best maintainable shape is the discussion's idle-timeout plus absolute-cap design, with maintainer approval for names, defaults, reset events, compatibility, and timeout reporting.
