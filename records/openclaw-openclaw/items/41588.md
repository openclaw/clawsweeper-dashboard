## Summary

Current main still uses fixed total-run deadlines, so the central request remains necessary. The original 600-second default was mitigated to 48 hours, and a separate open PR targets silent timeout notices, but neither adds activity-aware semantics; the broader duplicate was explicitly closed in favor of this issue as the canonical design thread.

## What This Changes

Request liveness-aware run timeout semantics so active multi-tool turns are not terminated solely because their total wall-clock duration exceeds one fixed budget.

## Best Possible Solution

Keep this as the canonical core design issue and define one progress-sensitive inactivity deadline plus an intentional absolute safety ceiling, while landing user-visible timeout delivery independently and avoiding new config unless existing timeout semantics cannot express the chosen policy.

## Reproduction Assessment

Yes at source level: current main still resolves and enforces a fixed total-run deadline with no ordinary tool-completion or model-output reset. A live long-duration turn was not run in this read-only review.

## Solution Assessment

The activity-aware direction is appropriate, but no single proposed variant is yet the best confirmed fix; maintainers should pair inactivity detection with an absolute safety ceiling and avoid multiplying public config unless the chosen contract requires it.
