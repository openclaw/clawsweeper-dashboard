## Summary

Keep open. Current main and the latest release mitigate the original 600s default with a 48-hour default, model-stream idle watchdogs, diagnostics, and clearer timeout classification, but they still do not provide the requested run-level idle timeout refreshed by model/tool progress plus a separate absolute cap.

## What This Changes

Request adding activity-aware agent run timeout behavior so active multi-tool runs are not killed solely by a flat wall-clock deadline.

## Best Possible Solution

Add a documented run-level idle timeout refreshed by model and tool progress while retaining a separate absolute runtime cap, with schema, help text, docs, and focused runner tests aligned.

## Reproduction Assessment

Yes, at source level: configure a short positive `agents.defaults.timeoutSeconds` or per-run timeout and keep an embedded run active beyond it; `runEmbeddedAttempt` arms a deadline from run start and does not refresh it on model or tool progress. I did not run a live multi-tool scenario because this review is read-only.

## Solution Assessment

Yes on direction: the issue discussion’s idle timeout plus absolute maximum cap is the narrow maintainable contract. The current 48-hour default, model idle watchdog, diagnostics, and clearer timeout paths are useful mitigations, not a complete solution.
