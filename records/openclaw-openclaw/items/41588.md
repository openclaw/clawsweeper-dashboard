## Summary

Keep open. Current main mitigates the original 600s default with a 48-hour default, no-timeout overrides, model-stream idle watchdogs, and diagnostics, but it still lacks the requested run-level idle timeout refreshed by model/tool progress plus a separate absolute cap.

## What This Changes

The issue requests activity-aware run timeout behavior so active multi-tool runs are not killed solely by elapsed wall-clock time.

## Best Possible Solution

Add a documented run-level idle timeout refreshed by model and tool progress while retaining a separate absolute runtime cap, with schema/help/docs and focused embedded-runner tests aligned.

## Reproduction Assessment

Yes, at source level: configure a short positive `agents.defaults.timeoutSeconds` or per-run timeout and keep an embedded run active past it. `runEmbeddedAttempt` schedules a single run deadline from `params.timeoutMs` and does not refresh it on model or tool progress.

## Solution Assessment

Yes on direction: the issue discussion's idle timeout plus absolute maximum cap is the narrow maintainable contract. The current 48-hour default, model idle watchdog, diagnostics, and clearer timeout classification are useful mitigations, not a complete solution.
