## Summary

Keep open: current main and v2026.6.6 mitigate the original 600s pain with a 48h default and per-run no-timeout overrides, but the core run timeout is still a single deadline rather than an activity-aware idle-timeout plus hard-cap contract.

## What This Changes

Request activity-aware agent run timeouts so active multi-tool runs are not killed by a flat wall-clock `agents.defaults.timeoutSeconds` deadline.

## Best Possible Solution

Keep this as the canonical core timeout design thread, decide idle-timeout plus hard-cap semantics, then implement the contract across gateway abort tracking, embedded-runner progress hooks, config/schema/docs, and focused timeout tests.

## Reproduction Assessment

Yes. Source inspection shows a short configured `agents.defaults.timeoutSeconds` still produces one run deadline with no reset on tool or model progress; I did not run a live channel reproduction in this read-only review.

## Solution Assessment

Unclear until maintainers choose the timeout contract. The idle-timeout plus absolute hard-cap direction is the strongest candidate, but the exact reset signals and config behavior need product review before implementation.
