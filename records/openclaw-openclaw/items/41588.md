- `impact:message-loss`: The reported failure mode is completed tool work followed by no final assistant response after the run timeout aborts.

## Summary

Keep open. Current main and the latest shipped release still use a single resolved agent run timeout with fixed gateway and embedded-runner deadlines; the closed liveness umbrella was explicitly superseded by this issue, and the open cron timeout issue is adjacent rather than a replacement.

## What This Changes

Request an activity-aware agent run timeout that uses idle/liveness tracking for active multi-tool runs instead of only a single wall-clock `agents.defaults.timeoutSeconds` deadline.

## Best Possible Solution

Add a documented run-level idle timeout refreshed by model and tool progress while retaining a separate absolute runtime cap and clear user-visible timeout reporting.

## Reproduction Assessment

Yes, at source level: configure a short positive `agents.defaults.timeoutSeconds` or per-run timeout and keep an embedded turn active past it. Current main keeps fixed gateway and embedded-runner deadlines that are not refreshed by normal model or tool progress; no live multi-tool turn was run.

## Solution Assessment

Yes on direction, but not as an automation-ready patch. The idle-timeout plus absolute-cap design is the maintainable shape, while reset events, defaults, config names, and timeout reporting need maintainer approval first.
