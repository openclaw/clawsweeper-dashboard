## Summary

Keep open. Current main and v2026.5.20 mitigate the original 600s default by using a 48h default and per-run no-timeout sentinel, but the requested activity-aware idle timeout plus hard cap is still missing from the gateway and embedded-runner timeout path.

## What This Changes

Add activity-aware run timeout semantics so active model/tool progress does not lose an agent turn under a fixed wall-clock timeout.

## Best Possible Solution

Add a documented idle timeout refreshed by model/tool progress, retain a separate absolute runtime cap, and surface timeout reason text through the normal run/channel reporting path with focused gateway and embedded-runner coverage.

## Reproduction Assessment

Yes, at source level. Configure a positive `agents.defaults.timeoutSeconds` or per-run timeout and keep an embedded turn active past that wall-clock budget; current main keeps fixed gateway and embedded-runner deadlines that are not refreshed by normal tool or model progress.

## Solution Assessment

Unclear as an implementation because no patch is proposed, but the direction is sound. The maintainable shape is the discussion's idle-timeout plus absolute-cap design, with maintainer approval for names, defaults, reset events, compatibility, and timeout reporting.
