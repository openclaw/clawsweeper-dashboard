## Summary

Keep open: current main and v2026.6.8 still enforce a single run deadline, so the requested activity-aware idle timeout plus hard cap is not implemented. The 48h default and per-run no-timeout override mitigate the original 600s pain, but they do not solve the core timeout semantics.

## What This Changes

Request activity-aware run timeout behavior so active multi-tool agent runs are not killed only because a flat wall-clock deadline expires.

## Best Possible Solution

Keep this as the canonical core timeout design thread, choose idle-timeout plus hard-cap semantics, then implement the contract across gateway abort tracking, embedded-runner progress hooks, config/schema/docs, and focused timeout tests.

## Reproduction Assessment

Yes, at source level. A short configured run timeout still becomes one gateway expiresAtMs and one embedded-runner timer with no reset on tool or model progress; I did not run a live channel reproduction.

## Solution Assessment

Unclear until maintainers choose the timeout contract. Idle timeout plus an absolute hard cap is the strongest direction from the discussion, while raising the default or landing a Telegram-only preview path is only a mitigation.
