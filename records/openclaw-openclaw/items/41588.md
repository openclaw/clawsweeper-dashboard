## Summary

Keep open. Current main mitigates the original 600s default with a 48-hour runtime default, model-idle watchdogs, and clearer timeout errors, but it still lacks the requested run-level idle timeout plus separate absolute cap refreshed by model/tool progress.

## What This Changes

Add activity-aware or idle/max run timeout semantics so active multi-tool runs are not killed solely by the wall-clock `agents.defaults.timeoutSeconds` deadline.

## Best Possible Solution

Add a documented run-level idle timeout refreshed by model and tool progress while keeping a separate absolute runtime cap, with schema, help text, docs, and focused runner tests aligned.

## Reproduction Assessment

Yes, at source level: configure a short positive `agents.defaults.timeoutSeconds` or per-run timeout and keep an embedded run active past it; `runEmbeddedAttempt` arms a run-start deadline that is not refreshed by model/tool progress. I did not run a live multi-tool scenario because this review is read-only.

## Solution Assessment

Yes on direction: separate idle timeout plus an absolute maximum runtime is the narrow maintainable contract. The current 48-hour default, model idle watchdog, and clearer timeout payloads are useful mitigations, not the requested run-level behavior.
