## Summary

Keep open. Current main still has a single run deadline plus a narrower model-stream idle watchdog, not the requested run-level activity clock with a separate absolute cap; the newer related thread was closed back to this issue as the canonical tracker.

## What This Changes

The issue requests activity-aware agent run timeout semantics so active multi-tool runs are not killed by one wall-clock deadline.

## Best Possible Solution

Add a documented run-level idle timeout refreshed by model/tool progress while retaining a separate absolute runtime cap, with schema/help/docs and focused runner/gateway tests aligned.

## Reproduction Assessment

Yes, at source level: configure a short positive `agents.defaults.timeoutSeconds` or per-run timeout and keep an embedded turn active past it. Current main keeps fixed gateway and embedded-runner deadlines that are not refreshed by model or tool progress.

## Solution Assessment

Yes on direction, but it is not automation-ready. The idle-timeout plus absolute-cap design is the maintainable shape, while reset events, defaults, config names, and abort/reporting behavior need maintainer approval first.
