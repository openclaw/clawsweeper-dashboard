## Summary

Keep open: current main and v2026.6.6 mitigate the original 600s pain with a 48h default and per-run no-timeout overrides, but the core run timeout is still a fixed deadline rather than an activity-aware idle-timeout plus hard-cap contract. This issue remains the canonical core design thread; related PRs/issues are duplicate, channel-specific, or timeout-message-only partial work.

## What This Changes

Request activity-aware run timeout behavior so productive multi-tool runs are not aborted by a flat wall-clock deadline.

## Best Possible Solution

Keep this as the canonical core timeout design thread and decide idle-timeout plus hard-cap semantics before implementing gateway, runner, config, docs, and regression coverage together.

## Reproduction Assessment

Yes. Source inspection shows a short configured `agents.defaults.timeoutSeconds` still produces one run deadline with no reset on tool or model progress; I did not run a live channel reproduction in this read-only review.

## Solution Assessment

Unclear until maintainers choose the timeout contract. The idle-timeout plus absolute hard-cap direction is the strongest candidate, but reset signals and config defaults need product review before implementation.
