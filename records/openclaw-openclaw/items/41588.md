## Summary

Keep open: current main and the latest release mitigate the original 600s default with a 48h default and some queue-level progress tracking, but the core gateway and embedded-runner abort path still uses fixed run deadlines rather than an activity-aware idle/max timeout contract. The related closed liveness umbrella explicitly points back to this issue as the canonical tracker, and the open Telegram PR is channel-specific rather than a core fix.

## What This Changes

The issue requests activity-aware or idle/max timeout semantics for long agent runs so active multi-tool workflows are not killed by a fixed wall-clock deadline.

## Best Possible Solution

Keep this issue as the canonical core design thread, decide the idle/max timeout contract first, then implement it across gateway registration, maintenance, embedded-runner progress hooks, schema/docs, and focused timeout tests.

## Reproduction Assessment

Yes, source-reproducible: configure a short agents.defaults.timeoutSeconds or run-specific timeout and let an active multi-tool turn exceed it; current main still has fixed gateway and embedded-runner deadlines without progress reset. I did not run a live multi-tool channel reproduction.

## Solution Assessment

No implementation is selected yet. The idle-timeout plus absolute-cap direction is plausible and matches the discussion, but maintainers still need to decide the activity boundary, config shape, and user-facing timeout behavior.
