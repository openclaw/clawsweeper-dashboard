## Summary

Keep open. Current main mitigates the original 600s failure mode with a 48-hour default, model-stream idle watchdogs, session liveness diagnostics, and explicit timeout error text, but it still lacks the requested activity-aware run-level idle/max timeout contract.

## What This Changes

The issue requests activity-aware gateway run timeout behavior so active multi-tool sessions are not killed solely by elapsed wall-clock time.

## Best Possible Solution

Add a run-level idle timeout refreshed by assistant stream activity and tool lifecycle completion, keep a separate absolute runtime cap, expose both through config schema/help/docs, and add focused tests for active multi-tool runs, idle aborts, and over-cap aborts.

## Reproduction Assessment

Yes, source-level: configure a short positive agents.defaults.timeoutSeconds or per-run timeout and keep a turn active beyond it; runEmbeddedAttempt arms a single run-start timer and does not refresh that deadline on tool or assistant activity. I did not run a live multi-tool scenario because this review was read-only.

## Solution Assessment

Not applicable to a PR, but the best direction is the discussion's option 2: a separate idle timeout plus absolute maximum runtime. The current 48-hour default and model idle watchdog are useful mitigations, not the requested run-level activity-aware contract.
