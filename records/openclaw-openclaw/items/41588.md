## Summary

Keep open: current main and v2026.6.10 still use a single wall-clock run deadline, while the related open PRs cover narrower notice, Telegram, or subagent idle-watchdog slices rather than the core activity-aware timeout contract.

## What This Changes

The issue requests activity-aware agent run timeout semantics so active multi-tool runs are not killed solely by elapsed wall-clock time.

## Best Possible Solution

Keep this as the canonical core timeout design thread and implement an idle-timeout plus absolute hard-cap contract across gateway abort tracking, embedded-runner progress hooks, config/schema/docs, and focused timeout tests after maintainer approval.

## Reproduction Assessment

Yes, at source level: current main still turns timeoutSeconds into one gateway expiresAtMs and one embedded-runner abort timer, with no reset on tool or model progress. I did not run a live multi-tool workflow.

## Solution Assessment

Unclear until maintainers choose the timeout contract. Idle timeout plus an absolute hard cap is the strongest current direction, but reset signals, defaults, and config shape need product review before implementation.
