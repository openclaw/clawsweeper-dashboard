## Summary

Keep open: current main and v2026.6.10 still use a single run deadline, while the related open PRs cover only partial symptoms such as Telegram progress, subagent idle watchdogs, or timeout notices.

## What This Changes

Requests activity-aware agent run timeout semantics so productive multi-tool runs are not killed solely by a flat wall-clock deadline.

## Best Possible Solution

Keep this as the canonical core timeout design thread, then implement an idle-timeout plus absolute hard-cap contract across gateway abort tracking, embedded-runner progress hooks, config/schema/docs, and focused timeout tests after maintainer approval.

## Reproduction Assessment

Yes. Source inspection shows a short configured run timeout still becomes one run deadline with no reset on tool or model progress; no live 10+ tool workflow was run in this read-only review.

## Solution Assessment

Unclear until maintainers choose the timeout contract. Idle timeout plus an absolute hard cap is the strongest current direction, but reset signals, default compatibility, and config shape need product review before implementation.
