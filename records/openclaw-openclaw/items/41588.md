## Summary

Keep open. Current main still has one agent runtime timeout and one gateway/runner deadline; the 48-hour default, per-run no-timeout sentinel, and model-stream idle watchdog mitigate pieces but do not implement a run-level idle timeout plus separate absolute cap. The newer liveness/progress umbrella was closed back to this issue as the canonical tracker.

## What This Changes

Request to make agent run timeouts activity-aware so active multi-tool runs are not killed solely by a flat wall-clock budget.

## Best Possible Solution

Add a documented run-level idle timeout refreshed by model/tool progress while retaining a separate absolute runtime cap, with schema/help/docs and gateway/runner tests aligned.

## Reproduction Assessment

Yes, at source level: configure a short positive agents.defaults.timeoutSeconds or per-run timeout and keep an embedded turn active past it. Current main keeps fixed gateway and embedded-runner deadlines that are not refreshed by normal model or tool progress; no live multi-tool turn was run.

## Solution Assessment

Yes on direction, but not as an automation-ready patch. The idle-timeout plus absolute-cap design is the maintainable shape, while reset events, defaults, config names, and timeout reporting need maintainer approval first.
