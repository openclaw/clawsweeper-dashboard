## Summary

Keep open. Current main mitigates the original 600s failure mode with a much longer default runtime, model-stream idle watchdogs, compaction timeout handling, and clearer timeout text, but it still does not expose or enforce an activity-aware run-level idle/max timeout contract refreshed by assistant or tool activity.

## What This Changes

Request activity-aware gateway run timeout behavior so active multi-tool agent runs are not killed by a flat wall-clock `agents.defaults.timeoutSeconds` deadline.

## Best Possible Solution

Add a run-level idle timeout that refreshes on assistant stream activity and tool lifecycle completion, keep a separate absolute maximum runtime cap, expose both through config schema/help/docs, and add focused tests proving active multi-tool runs can exceed the old flat deadline while idle or over-cap runs still abort clearly.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure a short positive `agents.defaults.timeoutSeconds` or per-run timeout and run an active multi-tool turn that lasts beyond it; current `runEmbeddedAttempt` schedules the abort once from `params.timeoutMs`, so ordinary assistant/tool activity does not refresh the deadline.

## Solution Assessment

Not applicable to a PR. The best direction is the issue discussion's option 2: separate idle and absolute run limits; the current 48-hour default and model idle watchdog reduce common failures but do not provide the requested activity-aware runtime contract.
