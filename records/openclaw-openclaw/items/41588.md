## Label Justifications

- `P2`: This is a normal-priority behavior improvement for long agent runs with limited blast radius but public timeout semantics to settle.
- `impact:message-loss`: The reported failure mode is completed tool work followed by no final assistant response after the run timeout aborts.

## Summary

Keep open. Current main and the latest release still use a single run timeout plus fixed gateway and embedded-runner deadlines; the existing model-stream idle watchdog is narrower and does not provide a run-level idle timeout with a separate absolute cap.

## What This Changes

The issue requests activity-aware agent run timeout semantics so active multi-tool runs are not killed solely by a flat wall-clock deadline.

## Best Possible Solution

Add a documented run-level idle timeout refreshed by model/tool progress while retaining a separate absolute runtime cap, with schema/help/docs and gateway/runner tests aligned.

## Reproduction Assessment

Yes, at source level: configure a short positive agents.defaults.timeoutSeconds or per-run timeout and keep an embedded turn active past it. Current main keeps fixed gateway and embedded-runner deadlines that are not refreshed by normal model or tool progress; no live multi-tool turn was run.

## Solution Assessment

Yes on direction, but not as an automation-ready patch. The idle-timeout plus absolute-cap design is the maintainable shape, while reset events, defaults, config names, and timeout reporting need maintainer approval first.
