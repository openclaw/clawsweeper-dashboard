## Summary

Keep open. Current main mitigates the original 600s failure mode with a 48-hour default, explicit zero mapping, timeout diagnostics, and model-stream idle watchdogs, but it still lacks the requested run-level activity timeout refreshed by model/tool progress with a separate absolute cap.

## What This Changes

Request an activity-aware gateway run timeout so productive multi-tool runs are not killed solely by the flat `agents.defaults.timeoutSeconds` wall-clock timer.

## Best Possible Solution

Add a documented run-level idle timeout refreshed by model and tool progress while retaining a separate absolute runtime cap, with schema/help/docs and focused embedded-runner regression coverage aligned.

## Reproduction Assessment

Yes, at source level: configure a short positive `agents.defaults.timeoutSeconds` or per-run timeout and keep an embedded run active beyond it; the run-level deadline is scheduled from run start and is not refreshed by model or tool progress. I did not run a live multi-tool scenario in this read-only review.

## Solution Assessment

Yes on direction: the separate idle timeout plus absolute maximum cap described in the discussion is the narrow maintainable contract. The current 48-hour default, zero sentinel, diagnostics, and model-stream idle watchdog are useful mitigations but not the requested run-level activity timeout.
