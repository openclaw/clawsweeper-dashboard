## Summary

Keep open. Current main mitigates the original 600s default and now surfaces a timeout error in some runner paths, but the requested activity-aware idle timeout plus separate hard cap is still not implemented in the gateway or embedded runner.

## What This Changes

Request activity-aware agent run timeout semantics so active multi-tool runs are not killed solely by one wall-clock budget.

## Best Possible Solution

Add a documented idle timeout refreshed by model/tool progress, keep a separate absolute run cap, and route timeout reasons through the normal run/channel reporting path with focused gateway and embedded-runner coverage.

## Reproduction Assessment

Yes, at source level. Configure a positive run timeout and keep an embedded turn active beyond that wall-clock budget; current main keeps fixed gateway and embedded-runner deadlines that are not refreshed by ordinary tool/model progress.

## Solution Assessment

Unclear as an implementation because no patch is proposed, but the direction is sound. The safest shape is the discussion's idle-timeout plus absolute-cap design, with maintainer approval for names, defaults, reset events, compatibility, and timeout reporting.
