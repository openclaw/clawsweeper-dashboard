## Summary

Keep open: current main and the latest release still use a single run deadline with no activity reset, so the central idle-timeout plus hard-cap request remains unresolved. The 48h default and per-run no-timeout override mitigate the original 600s failure mode but do not implement activity-aware timeout semantics.

## What This Changes

Request activity-aware agent run timeouts so active multi-tool turns are not killed by a single wall-clock deadline, likely by separating idle and hard-cap timeout semantics.

## Best Possible Solution

Keep this as the canonical core timeout design thread, decide idle-timeout plus hard-cap semantics, then implement the contract across gateway abort tracking, embedded-runner progress hooks, config/schema/docs, and focused timeout tests.

## Reproduction Assessment

Yes. Source inspection shows a short configured agents.defaults.timeoutSeconds still produces one run deadline with no reset on tool or model progress; no live multi-tool channel reproduction was run in this read-only review.

## Solution Assessment

Unclear until maintainers choose the timeout contract. The idle-timeout plus absolute hard-cap direction is the strongest candidate, but reset signals and config behavior need product review before implementation.
