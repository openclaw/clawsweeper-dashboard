## Summary

Keep open: current main and v2026.5.20 still collapse inbound WhatsApp GIF-picker video messages to `<media:video>` and only forward contact structured context, while Baileys exposes the requested GIF/ad metadata fields and the companion implementation PR is closed unmerged.

## What This Changes

Request WhatsApp inbound media extraction to classify GIF-picker videos and expose `externalAdReply` title/body/source URL metadata to the agent.

## Best Possible Solution

Add WhatsApp-owned media metadata extraction that classifies `gifPlayback` videos as GIFs and forwards allowlisted `externalAdReply` fields through `untrustedStructuredContext`, with focused tests, docs, and changelog coverage.

## Reproduction Assessment

Yes. A Baileys `videoMessage` with `gifPlayback` and `contextInfo.externalAdReply` reaches `extractMediaPlaceholder`, where current main returns only `<media:video>` and drops the metadata.

## Solution Assessment

No, not exactly as proposed. Inline pseudo-XML attributes would mix provider text into the message body; the narrower maintainable path is to carry allowlisted metadata through existing untrusted structured context.
