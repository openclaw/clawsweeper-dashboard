## Summary

Keep open: current main and the latest release still collapse inbound WhatsApp GIF-picker video messages to `<media:video>` and do not surface `gifPlayback` or `externalAdReply` metadata, while the pinned Baileys contract exposes those fields and the linked implementation PR was closed unmerged.

## What This Changes

Expose WhatsApp GIF-picker source URL, title, and description metadata from inbound media so agents can distinguish GIFs from ordinary videos and understand the media context.

## Best Possible Solution

Add a focused WhatsApp plugin repair that classifies `videoMessage.gifPlayback` media as GIF and forwards allowlisted `externalAdReply` fields as untrusted structured media context, with extractor/monitor/dispatch/prompt tests and WhatsApp docs/changelog coverage.

## Reproduction Assessment

Yes, source-level: a Baileys `videoMessage` with `gifPlayback` and `contextInfo.externalAdReply` reaches `extractMediaPlaceholder`, where current main returns only `<media:video>` and drops the metadata.

## Solution Assessment

Yes, with adjustment: the narrow maintainable fix is in the WhatsApp plugin, but provider strings should travel through the existing untrusted structured context path rather than inline placeholder attributes.
