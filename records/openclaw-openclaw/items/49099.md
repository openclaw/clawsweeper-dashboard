## Summary

Keep open. Current main still normalizes inbound WhatsApp videos to a fixed `<media:video>` placeholder and does not inspect the GIF flag or external ad metadata; the companion implementation PR was closed unmerged, and Baileys exposes the requested fields.

## What This Changes

The issue requests extracting WhatsApp GIF/media metadata such as `gifPlayback` and `externalAdReply` title, body, and source URL from inbound media.

## Best Possible Solution

Implement this inside the WhatsApp plugin by classifying GIF playback and carrying provider-supplied media metadata through the existing fenced untrusted structured context path, with focused docs and regression coverage.

## Reproduction Assessment

Yes, at source level: a Baileys `videoMessage` with `gifPlayback: true` and `contextInfo.externalAdReply` still flows through `extractMediaPlaceholder` as only `<media:video>`. I did not perform a live WhatsApp GIF-picker send.

## Solution Assessment

No, not exactly as suggested. The requested inline placeholder attributes would expose untrusted provider text in prompt markup; the narrower maintainable solution is to keep placeholders terse and forward the metadata as fenced untrusted structured context.
