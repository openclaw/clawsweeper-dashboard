## Summary

Keep open. Current main and the latest release still reduce inbound WhatsApp GIF-picker videos to plain media placeholders without reading `gifPlayback` or `contextInfo.externalAdReply`; the companion implementation PR is closed unmerged, and the only related closed media-metadata issue fixed a different plugin-hook payload path.

## What This Changes

Request to enrich inbound WhatsApp media by distinguishing GIF-picker videos and surfacing `externalAdReply` source, title, and body metadata to the agent.

## Best Possible Solution

Implement this in the WhatsApp plugin by classifying `videoMessage.gifPlayback` as GIF and carrying `externalAdReply` fields through explicit untrusted structured media context rather than inline placeholder attributes.

## Reproduction Assessment

Yes, at source level. `extractMediaPlaceholder` currently returns `<media:video>` for any `videoMessage`, inbound enrichment uses that placeholder for media-only messages, and Baileys `7.0.0-rc13` exposes the requested `gifPlayback` and `externalAdReply` fields.

## Solution Assessment

No for the exact inline-attribute proposal. The maintainable fix is a focused WhatsApp plugin change that keeps placeholders compact and forwards provider metadata through the existing untrusted structured-context path with prompt-rendering tests.
