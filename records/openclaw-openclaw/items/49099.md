## Summary

Keep open. Current main still returns a fixed WhatsApp video placeholder for inbound GIF-picker videos and does not forward `externalAdReply` metadata; the linked implementation attempt at https://github.com/openclaw/openclaw/pull/49103 is closed and unmerged.

## What This Changes

The issue requests extracting WhatsApp GIF/media metadata from inbound Baileys messages so agents can understand picker GIFs without video processing.

## Best Possible Solution

Add a WhatsApp plugin-owned extractor that classifies `videoMessage.gifPlayback` as GIF and forwards bounded `externalAdReply` fields through `untrustedStructuredContext`, with focused prompt, docs, and regression coverage.

## Reproduction Assessment

Yes, source-level. A synthetic Baileys `videoMessage` with `gifPlayback` and `contextInfo.externalAdReply` reaches the current extractor, but target main still returns `<media:video>` and drops that metadata.

## Solution Assessment

Yes for the keep-open path. The narrow maintainable fix is additive WhatsApp plugin extraction into the existing untrusted structured metadata carrier, not a core channel change or inline pseudo-XML metadata expansion.
