## Summary

Keep open. Current main still drops the requested inbound WhatsApp GIF/ad metadata: GIF videos are normalized as plain `<media:video>`, `contextInfo.externalAdReply` is not extracted, and the only structured context path wired by the monitor is contact metadata. The closed companion PR did not merge, so there is no implemented fix on main.

## What This Changes

Request extraction of WhatsApp inbound GIF/media metadata from Baileys `videoMessage.gifPlayback` and `contextInfo.externalAdReply` so agents can distinguish GIFs and understand provider title/body/source URL without video processing.

## Best Possible Solution

Implement the fix inside the WhatsApp plugin: classify `videoMessage.gifPlayback` as GIF for media-only placeholders, extract bounded `externalAdReply` title/body/source URL for relevant inbound media, and pass those provider-controlled values through existing untrusted structured context with focused tests, docs, and changelog credit.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to pass a Baileys-shaped `proto.IMessage` with `videoMessage.gifPlayback: true` and `contextInfo.externalAdReply` into `extractMediaPlaceholder`; current main returns `<media:video>` and the monitor has no media metadata context path.

## Solution Assessment

No for the issue's inline pseudo-XML attribute sketch, because provider title/body/source values are untrusted prompt input. Yes for a narrow WhatsApp-owned repair that keeps the visible GIF/video distinction simple and carries metadata via fenced untrusted structured context.
