## Summary

Keep open. Current `origin/main` still reduces inbound WhatsApp GIF-picker videos to fixed media placeholders and never reads `videoMessage.gifPlayback` or `contextInfo.externalAdReply`; the companion implementation at https://github.com/openclaw/openclaw/pull/49103 was closed unmerged, and the pinned Baileys types expose the requested fields.

## What This Changes

Request inbound WhatsApp media handling to distinguish GIF-picker videos and surface `externalAdReply` title, body, and source URL metadata to the agent.

## Best Possible Solution

Add a WhatsApp plugin-owned extractor that emits a GIF placeholder for `videoMessage.gifPlayback` and forwards `externalAdReply` title/body/source URL through the existing fenced untrusted structured context path.

## Reproduction Assessment

Yes, source-level. A `videoMessage` with `gifPlayback` and `contextInfo.externalAdReply` currently takes the fixed video branch in `extractMediaPlaceholder`, so the agent would see `<media:video>` and no metadata without needing ffmpeg.

## Solution Assessment

Yes, a focused WhatsApp plugin repair is the narrow maintainable direction. The issue's inline attribute sketch should be adapted to the existing untrusted structured-context path to avoid mixing provider-fetched strings into pseudo-XML prompt text.
