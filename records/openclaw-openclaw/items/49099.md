## Summary

Keep open. Current main and the latest release still map inbound WhatsApp video messages to a plain `<media:video>` placeholder, never read `gifPlayback` or `contextInfo.externalAdReply`, and both related implementation PRs are closed unmerged.

## What This Changes

The issue requests extracting WhatsApp GIF picker metadata from inbound media so agents can distinguish GIF playback and see source/title/description context.

## Best Possible Solution

Classify GIF media in the WhatsApp plugin, keep the media body placeholder compact, and pass externalAdReply title/body/source URL through the existing untrusted structured-context path with focused tests and docs alignment.

## Reproduction Assessment

Yes, source-reproducible: a WhatsApp `videoMessage` with `gifPlayback` or `externalAdReply` still takes the current `<media:video>` branch and drops the metadata. I did not run a live WhatsApp GIF-picker reproduction in this read-only review.

## Solution Assessment

Partly: the WhatsApp plugin is the right owner and GIF classification is narrow, but the issue's inline pseudo-XML attribute sketch is not the best final shape for untrusted provider strings. The better fix is compact media body text plus fenced `untrustedStructuredContext` metadata.
