## Summary

Keep open. Current main still drops the requested WhatsApp GIF picker metadata: inbound video messages are normalized to a fixed `<media:video>` placeholder, the monitor only forwards contact structured context, and the related implementation PR #49103 is closed and unmerged. This is a narrow WhatsApp plugin repair, but the metadata should be carried as untrusted structured context rather than inline pseudo-XML attributes.

## What This Changes

Request WhatsApp inbound media handling to distinguish GIF picker videos and expose `externalAdReply` title/body/source URL metadata to the agent.

## Best Possible Solution

Implement a focused WhatsApp plugin fix: classify `videoMessage.gifPlayback` media-only messages as GIFs, extract bounded `contextInfo.externalAdReply` fields for relevant inbound media, pass them through `untrustedStructuredContext`, and add targeted extractor, monitor/dispatch, prompt-rendering, docs, and changelog coverage.

## Reproduction Assessment

Yes. A source-level reproduction is to pass a Baileys-shaped `proto.IMessage` with `videoMessage.gifPlayback: true` and `contextInfo.externalAdReply` into `extractMediaPlaceholder`; current main returns `<media:video>` and the monitor has no path for that metadata.

## Solution Assessment

No, the issue's inline attribute sketch is not the safest final shape. The maintainable fix should stay in the WhatsApp plugin, use `<media:gif>` only for classification, and carry provider metadata via the existing untrusted structured context path.
