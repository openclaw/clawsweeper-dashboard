## Summary

Keep open. Current main and the latest release still normalize inbound WhatsApp GIF-picker media to generic media placeholders without reading `gifPlayback` or `externalAdReply`, and the linked implementation PR is closed and unmerged.

## What This Changes

The issue asks the WhatsApp inbound path to distinguish GIF picker videos from normal videos and expose provider title/source/body metadata from Baileys `contextInfo.externalAdReply`.

## Best Possible Solution

Implement this in the WhatsApp plugin by classifying `gifPlayback` media and forwarding provider metadata through the existing untrusted structured context path with focused extractor, dispatch, prompt, docs, and changelog coverage.

## Reproduction Assessment

Yes, source-level. A synthetic Baileys `videoMessage` with `gifPlayback` and `contextInfo.externalAdReply` reaches `extractMediaPlaceholder`, where current main still returns `<media:video>` and drops the metadata.

## Solution Assessment

No, not exactly as proposed. The maintainable route is to extract the data in the WhatsApp plugin and carry it as untrusted structured context rather than inline attributes in the message body.
