Keep open: current main and the latest release still reduce inbound WhatsApp GIF-picker media to generic media placeholders and do not forward `externalAdReply` metadata, while the only linked implementation PR is closed and unmerged.

## What This Changes

Expose inbound WhatsApp GIF-picker metadata from `videoMessage.gifPlayback` and `contextInfo.externalAdReply` so agents can understand GIF source/title without video processing.

## Best Possible Solution

Add a WhatsApp-owned extractor for allowlisted GIF/media metadata and forward it as untrusted structured media context, with `<media:gif>` classification, focused regression coverage, docs, and changelog updates.

## Reproduction Assessment

Yes, source-level. A synthetic Baileys `videoMessage` with `gifPlayback` and `contextInfo.externalAdReply` reaches `extractMediaPlaceholder`, where current main still returns `<media:video>` and drops the metadata.

## Solution Assessment

No, not exactly as proposed. The safer maintainable route is to extract allowlisted data in the WhatsApp plugin and carry it as untrusted structured context rather than embedding provider strings in inline placeholder attributes.
