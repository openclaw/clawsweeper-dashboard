## Summary

Keep open. Current main still normalizes inbound WhatsApp video messages to a fixed `<media:video>` placeholder and only forwards contact structured context, while the pinned Baileys contract exposes the GIF/ad-reply fields requested here and the companion implementation at https://github.com/openclaw/openclaw/pull/49103 is closed and unmerged.

## What This Changes

Request extraction of WhatsApp GIF-picker `videoMessage.gifPlayback` and `contextInfo.externalAdReply` metadata so agents can see GIF title, source URL, and description.

## Best Possible Solution

Add a focused WhatsApp plugin extractor for GIF/ad-reply media metadata and carry it through the existing untrusted structured-context path, with narrow tests, docs, and changelog coverage.

## Reproduction Assessment

Yes, source-level. A synthetic Baileys `videoMessage` with `gifPlayback` and `contextInfo.externalAdReply` reaches `extractMediaPlaceholder`, where current main returns `<media:video>` and drops the metadata.

## Solution Assessment

No, not exactly as proposed. Inline pseudo-XML attributes would expose untrusted provider text in the message body; the maintainable fix is plugin-local extraction into `untrustedStructuredContext`.
