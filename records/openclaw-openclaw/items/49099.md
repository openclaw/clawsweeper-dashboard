## Summary

Keep open. Current main still normalizes inbound WhatsApp video messages to a fixed `<media:video>` placeholder, does not read `videoMessage.gifPlayback` or `contextInfo.externalAdReply`, and the companion implementation at https://github.com/openclaw/openclaw/pull/49103 was closed unmerged.

## What This Changes

The issue requests WhatsApp inbound GIF-picker media to be classified as GIFs and to expose `externalAdReply` title/source/body metadata to the agent.

## Best Possible Solution

Implement this in the WhatsApp plugin by classifying `gifPlayback` media-only messages as `<media:gif>` and forwarding bounded `externalAdReply` fields through `untrustedStructuredContext`.

## Reproduction Assessment

Yes. A source-level reproduction is to pass a Baileys-shaped inbound `videoMessage` with `gifPlayback: true` and `contextInfo.externalAdReply` into `extractMediaPlaceholder`; current main returns `<media:video>` and the monitor has no media metadata context path.

## Solution Assessment

Yes for the safer implementation direction. The issue's inline attribute sketch should be replaced with WhatsApp-owned extraction into existing untrusted structured context, plus GIF classification in the placeholder.
