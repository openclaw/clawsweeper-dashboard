## Summary

Keep open: current main still maps every inbound WhatsApp video message to `<media:video>` and does not read GIF playback or external ad metadata; the two linked implementation PRs were closed unmerged, and the latest attempt raised a safer prompt-boundary shape to preserve.

## What This Changes

The issue requests extracting WhatsApp GIF-picker metadata from inbound media so agents can distinguish GIFs and use source/title/description context instead of seeing only `<media:video>`.

## Best Possible Solution

Classify WhatsApp `gifPlayback` media in the plugin, keep the body placeholder compact, and route external ad metadata through untrusted structured context with focused extractor, dispatch, prompt-rendering, and regression coverage.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: a WhatsApp `videoMessage` with `gifPlayback: true` and `contextInfo.externalAdReply` still follows the current plain video branch and becomes `<media:video>`. I did not run tests because this review is read-only.

## Solution Assessment

Yes for keeping this in the WhatsApp inbound path, but no for the inline pseudo-XML attribute shape from the old PRs. The maintainable fix is to classify GIFs in the placeholder and pass provider metadata through the existing untrusted structured-context route.
