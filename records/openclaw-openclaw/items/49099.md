## Summary

Keep open: current main and v2026.5.20 still collapse inbound WhatsApp GIF-picker video messages to `<media:video>` and do not surface `gifPlayback` or `externalAdReply` metadata, while the dependency exposes the requested fields and the linked implementation PR is closed unmerged.

## What This Changes

The issue requests extracting WhatsApp GIF-picker media metadata, including source URL and title, from inbound messages so agents can understand GIF context without video processing.

## Best Possible Solution

Add a focused WhatsApp plugin extractor that classifies `videoMessage.gifPlayback` media as GIF and forwards allowlisted `externalAdReply` fields through untrusted structured media context with tests and docs.

## Reproduction Assessment

Yes, source-level: a Baileys `videoMessage` with `gifPlayback` and `contextInfo.externalAdReply` reaches `extractMediaPlaceholder`, where current main returns only `<media:video>` and drops the metadata.

## Solution Assessment

No, not exactly as proposed; inline placeholder attributes would expose untrusted provider strings in a weaker form than the existing fenced untrusted structured context path.
