## Summary

Keep open. Current main still drops the requested WhatsApp GIF picker metadata: inbound video messages normalize to a fixed `<media:video>` placeholder, `externalAdReply` is not read anywhere in the WhatsApp inbound path, and the related implementation PR #49103 is closed and unmerged.

## What This Changes

The issue asks OpenClaw's WhatsApp plugin to distinguish inbound GIF-picker videos and expose provider metadata such as source URL, title, and description.

## Best Possible Solution

Implement a focused WhatsApp plugin fix: classify `videoMessage.gifPlayback` media-only messages as GIFs, extract bounded `contextInfo.externalAdReply` fields for relevant inbound media, and forward them as untrusted structured media context with tests, docs, and changelog coverage.

## Reproduction Assessment

Yes. A source-level reproduction is to pass a Baileys-shaped `proto.IMessage` with `videoMessage.gifPlayback: true` and `contextInfo.externalAdReply` into `extractMediaPlaceholder`; current main returns `<media:video>`, and the monitor has no media metadata context path.

## Solution Assessment

No. The inline-attribute sketch is not the best final shape because provider metadata is untrusted; the maintainable fix should use `<media:gif>` for classification and carry title/body/source through `untrustedStructuredContext`.
