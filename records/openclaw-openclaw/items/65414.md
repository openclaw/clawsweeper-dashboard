## Summary

Keep open. Current main still parses promptless Android assistant launches, then routes every assistant launch through the Chat destination; the Voice tab has only manual mic/Talk permission continuations and no one-shot assistant-origin mic activation state.

## What This Changes

Route promptless Android ACTION_ASSIST launches to the Voice tab with one-shot mic activation while keeping prompt-bearing App Actions in Chat.

## Best Possible Solution

Implement the Android-side routing at the assistant launch boundary: promptless source="assist" requests HomeDestination.Voice and sets a one-shot pending mic activation flag, while prompt-bearing app-action launches continue to Chat. VoiceTabScreen should consume the flag once, request RECORD_AUDIO when needed, enable the mic only after grant, and update focused Android tests plus user-facing docs/changelog.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: ACTION_ASSIST parses to source="assist" with no prompt, handleAssistantLaunch currently selects HomeDestination.Chat, and VoiceTabScreen has no assistant-origin auto-activation branch.

## Solution Assessment

Yes. The requested direction is the narrowest maintainable fix: keep parsing, manifest, App Actions, gateway config, and schema stable while adding a promptless-assist special case and one-shot Voice-tab mic activation.
