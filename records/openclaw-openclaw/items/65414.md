## Summary

Keep open. Current main still routes every Android assistant launch to Chat, while promptless `ACTION_ASSIST` already parses distinctly and the Voice tab has no one-shot assistant mic activation path.

## What This Changes

Route promptless Android `ACTION_ASSIST` launches to the Voice tab with one-shot mic activation while keeping prompt-bearing `ASK_OPENCLAW` App Actions in Chat.

## Best Possible Solution

Implement the Android-side promptless-assist branch and one-shot Voice-tab mic activation, keep prompt-bearing App Actions on Chat, and update focused Android tests plus user-facing docs/changelog.

## Repair Work Prompt

Fix openclaw/openclaw#65414. In Android, promptless `Intent.ACTION_ASSIST` launches should open the Voice tab and auto-enable the mic exactly once; prompt-bearing `ASK_OPENCLAW` App Actions must keep routing to Chat and must not auto-send. Keep the boundary narrow: no manifest, gateway config, schema, or App Actions metadata changes. Likely implementation: add a one-shot `pendingVoiceAutoActivate` StateFlow plus consume method in `MainViewModel`; in `handleAssistantLaunch`, when `request.source == "assist" && request.prompt == null`, request `HomeDestination.Voice`, set the one-shot flag, clear stale chat draft/auto-send state, and return. In `VoiceTabScreen`, collect and consume the flag from a `LaunchedEffect`; if `RECORD_AUDIO` is already granted, call `setMicEnabled(true)`, otherwise launch the existing permission request with an assistant-mic pending action and enable only after grant. Add focused Android tests for parser/route behavior and any feasible permission/consume helper coverage. Update `docs/platforms/android.md` and `CHANGELOG.md` because this is user-facing. Do not change unrelated Android assistant/security behavior from #70714.
