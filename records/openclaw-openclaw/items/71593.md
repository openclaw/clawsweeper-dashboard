## Summary

Keep open: current main and the latest shipped release still expose only env/file/exec SecretRef providers, while the plugin-provider seam and broader field coverage remain open, security-sensitive API work with active implementation candidates.

## What This Changes

The issue requests a plugin-based SecretRef provider mechanism so users can add external secret backends and cover sensitive OpenClaw config fields without repeated core schema changes.

## Best Possible Solution

Choose one security-reviewed core/plugin contract for external SecretRef providers, then handle remaining field widening through focused follow-ups for env vars, plugin config paths, and channel credentials.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/API request. Source and docs inspection give a high-confidence current-main and v2026.5.18 status check that SecretRef providers remain env/file/exec-only while selected manifest-declared plugin config SecretInput paths already exist.

## Solution Assessment

Unclear until maintainers choose the contract. A plugin or manifest seam fits OpenClaw's plugin architecture, but secret resolution through plugins is security-sensitive and should be separated from broader field-coverage expansion.
