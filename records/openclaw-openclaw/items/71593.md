## Summary

Keep open. Current main and v2026.6.8 shipped the pluginIntegration exec-provider seam, but the requested broad schema coverage remains unresolved: `SecretRefSource` is still `env`/`file`/`exec`, `skills.entries.*.env` is still string-only, and generic `plugins.entries.*.config` is still an unknown record rather than a SecretInput surface.

## What This Changes

Request support for plugin-owned SecretRef providers and broader SecretRef coverage across currently plain core config surfaces such as skill env values and plugin config.

## Best Possible Solution

Keep the shipped pluginIntegration provider seam, then define a bounded maintainer-approved SecretRef field-coverage contract instead of making every plugin or skill config value implicitly resolvable.

## Reproduction Assessment

Not applicable as a bug reproduction, but yes as a source-level capability check. Current source shows the provider-integration seam exists while the cited skill env and generic plugin config fields still do not accept SecretInput values.

## Solution Assessment

Unclear until maintainers choose the contract. The shipped manifest-level pluginIntegration path is the right partial provider seam, but arbitrary all-field SecretRef expansion is not the safest final shape without a bounded field allowlist and trust model.
