## Summary

Keep open. Current main still has a closed SecretRef provider model limited to env/file/exec, and the issue is already paired with open closing implementation PR #72548, so the right next step is maintainer review of that PR rather than closing this issue early.

## What This Changes

The issue requests a plugin-defined SecretRef provider mechanism so users can resolve secret references across core schema fields such as channel credentials, skill env values, and plugin config without repeated core schema PRs.

## Best Possible Solution

Land or replace PR #72548 with a versioned, documented plugin SecretRef provider contract that preserves strict built-in env/file/exec behavior, honors plugin activation and origin boundaries, and tracks any remaining SecretRef field-coverage gaps separately.

## Reproduction Assessment

Not applicable as a feature/API request. The gap is high-confidence by static inspection: current main rejects non-env/file/exec SecretRef sources in config, SDK typing, gateway wire schemas, and resolver dispatch.

## Solution Assessment

Unclear until maintainer review of PR #72548. A plugin-defined provider seam matches OpenClaw's plugin-first architecture, but the current implementation candidate defers broader schema field coverage, so maintainers need to decide that scope explicitly.
