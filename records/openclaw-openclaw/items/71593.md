## Summary

Keep open: current main still has built-in-only SecretRef provider sources, and the existing manifest `secretInputs` support covers selected plugin config fields rather than a pluggable provider contract across core schemas. The linked implementation PRs remain open, so this issue should stay as the API/security decision point.

## What This Changes

The issue requests a plugin-based SecretRef provider contract that can cover sensitive config fields without repeated core schema edits.

## Best Possible Solution

Keep this issue open until maintainers choose and land a security-reviewed provider contract, then split any remaining field-widening into focused follow-ups for skill env values, plugin config paths, and channel credentials.

Not applicable as a bug reproduction; this is a feature/API request. Source inspection gives a high-confidence current-main status: SecretRef provider sources remain env/file/exec-only, while selected manifest-declared plugin config SecretInput paths already exist.

## Solution Assessment

Unclear until maintainers choose the contract. A plugin or manifest seam fits OpenClaw's architecture, but secret resolution through plugins is security-sensitive and should be separated from broader field-coverage expansion.
