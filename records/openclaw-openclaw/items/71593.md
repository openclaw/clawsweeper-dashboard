## Summary

Keep open. Current main still has a closed `env`/`file`/`exec` SecretRef provider model and only path-declared plugin config SecretInput coverage, while open PR #72548 is the explicit closing implementation candidate. The maximus-dss comment adds a concrete plugin-config use case for engram, so this should stay open until #72548 is reviewed, landed, replaced, or closed.

## What This Changes

Allow plugins to implement SecretRef providers and make SecretRef objects usable across core credential/schema fields without repeated core schema patches.

## Best Possible Solution

Review and either land or replace #72548 with a versioned, documented plugin SecretRef provider contract that keeps built-in env/file/exec validation strict, then explicitly include or split out the broader field-coverage work for plugin config, skill env, and channel credential surfaces.

## Reproduction Assessment

Not applicable as a feature/API request. The gap is high-confidence by static inspection of current main: types, zod schemas, gateway wire schemas, docs, and resolver dispatch remain closed to env/file/exec, and the engram comment gives a concrete user-facing configuration example.

## Solution Assessment

Unclear until #72548 is reviewed. A plugin-defined provider seam matches OpenClaw's plugin-first direction, but the final shape needs maintainer security/API review and a deliberate decision on whether broad config field coverage is part of the same change.
