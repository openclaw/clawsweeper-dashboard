## Summary

Keep open: current main and v2026.6.10 include the pluginIntegration provider seam and manifest-declared plugin config SecretInput paths, but the broader schema coverage request remains unresolved and security-sensitive.

## What This Changes

Request plugin-based SecretRef providers plus broader OpenClaw config schema coverage so plugin, skill, and channel secrets can use SecretRef objects without repeated core edits.

## Best Possible Solution

Keep the shipped providerIntegration seam and define a bounded, maintainer-approved SecretInput coverage contract for specific sensitive skill, plugin, and channel paths with audit, redaction, and activation rules.

## Reproduction Assessment

Yes, as a source-level capability check: current main still keeps `skills.entries.*.env` and `env.vars` string-only while plugin config SecretRefs are limited to manifest-declared paths.

## Solution Assessment

Unclear until maintainers choose the remaining trust boundary; the shipped providerIntegration plus manifest `secretInputs` path is the right partial seam, but arbitrary all-field SecretRef expansion would be too broad without an allowlisted security model.
