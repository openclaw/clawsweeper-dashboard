## Summary

Keep open: current main still has an env/file/exec-only SecretRef source, config schema, gateway wire schema, and resolver dispatch. The linked implementation work is still open rather than merged, and the broader field-coverage part remains a maintainer API/security/product decision.

## What This Changes

The issue requests a plugin-owned SecretRef provider mechanism so new secret backends and broader config fields can use SecretRef objects without repeated core schema changes.

## Best Possible Solution

Review and choose between https://github.com/openclaw/openclaw/pull/72548 and https://github.com/openclaw/openclaw/pull/82326, land a documented security-reviewed contract, and keep broader field widening tracked through focused follow-ups such as https://github.com/openclaw/openclaw/issues/72390.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/API request. Source inspection gives a high-confidence current-main check that SecretRef sources, schemas, docs, gateway wire schema, and resolver dispatch are still env/file/exec-only.

## Solution Assessment

Unclear until maintainers choose the contract. A plugin or manifest seam fits the architecture, but secret resolution through plugins needs explicit security boundaries and a separate decision on broad config-field coverage.
