## Summary

Keep open. Current main still has a closed SecretRef provider model and no plugin-owned SecretRef source seam; the open closing implementation PR #72548 is the right place for maintainer API/security/product review before this issue can close.

## What This Changes

The issue requests a plugin-based SecretRef provider mechanism so users can add secret backends and cover sensitive config fields without repeated core schema edits.

## Best Possible Solution

Review, land, or replace PR #72548 with a versioned and documented plugin SecretRef provider contract, then decide whether the broader field-coverage gaps for skills env values, plugin config, and channel credentials are included there or tracked as narrower follow-ups.

## Reproduction Assessment

Not applicable as a feature/API request. The gap is high-confidence by static inspection: current main rejects non-env/file/exec SecretRef sources in config, SDK-adjacent helpers, Gateway protocol schemas, and resolver dispatch.

## Solution Assessment

Unclear until maintainer review of PR #72548. A plugin-defined provider seam matches the plugin-first architecture, but the open implementation defers part of the issue's requested field coverage and touches secret-resolution security boundaries.
