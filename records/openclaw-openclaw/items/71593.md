## Summary

Keep open. Current main still has a closed SecretRef provider model limited to env/file/exec, and an open closing implementation PR (#72548) is already paired with this issue, so the right outcome is maintainer review of that PR or a maintainer-directed replacement rather than closing this issue early.

## What This Changes

Request adding a plugin-owned SecretRef provider mechanism so users can define external secret backends and avoid repeated core schema changes for channel credentials, skill env values, and plugin config secrets.

## Best Possible Solution

Review and land PR #72548 or a replacement that adds a versioned, documented plugin SecretRef provider contract while preserving strict built-in env/file/exec behavior, plugin activation/origin boundaries, and Gateway/SDK schema parity; track any remaining broad field-coverage gaps separately if they are intentionally out of scope.

## Reproduction Assessment

Not applicable as a feature/API request. The gap is high-confidence by static inspection: current main rejects non-env/file/exec SecretRef sources in config types, Zod schemas, Gateway wire schemas, and resolver dispatch.

## Solution Assessment

Unclear until maintainer review of PR #72548. A plugin-defined provider seam matches OpenClaw's plugin-first architecture, but the current candidate defers broader schema field coverage and needs API/security review before it can be considered the best fix.
