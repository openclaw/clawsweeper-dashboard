## Summary

Keep open. Current main still exposes SecretRef providers as a closed env/file/exec contract across config types, zod validation, Gateway wire schema, and resolver dispatch; plugin config SecretInput support is path-declared metadata, not a plugin-defined provider registration seam. The issue is also already paired with open PR #72548, so cleanup should wait for that PR to be reviewed, merged, or replaced.

## What This Changes

The issue requests a plugin-defined SecretRef provider API and broader SecretRef acceptance across core config schemas so new secret backends or plugin config secret paths do not require repeated core schema changes.

## Best Possible Solution

Land or replace PR #72548 with a documented, versioned plugin SecretRef provider contract that preserves strict built-in env/file/exec behavior, honors plugin activation/origin boundaries, and explicitly decides which broader SecretRef field-coverage gaps remain in this issue versus narrower follow-ups.

## Reproduction Assessment

Not applicable as a feature/API request. The gap is high-confidence by static inspection: current main rejects non-env/file/exec SecretRef sources in the core type/schema/wire/resolver path and lacks a public secret-provider plugin SDK subpath.

## Solution Assessment

Unclear until maintainer review of PR #72548. A plugin-defined provider seam matches OpenClaw's plugin-first architecture, but the implementation and broader schema coverage need API/security/product review before this issue can close.
