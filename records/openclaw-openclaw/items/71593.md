## Summary

Keep open: current main has shipped the pluginIntegration exec-provider seam, but the broader requested SecretRef coverage for skill env values and arbitrary plugin config remains unresolved and security-sensitive.

## What This Changes

The issue asks for plugin-defined SecretRef providers plus schema/runtime coverage so plugin, skill, and channel credential fields can accept SecretRef objects without repeated core schema edits.

## Best Possible Solution

Keep the shipped pluginIntegration provider seam, then define a bounded maintainer-approved SecretRef field-coverage contract instead of making every plugin or skill config value implicitly resolvable.

## Reproduction Assessment

Not applicable as a bug reproduction, but yes as a source-level capability check: current schemas still make `skills.entries.*.env` string-only and generic `plugins.entries.*.config` unknown-only while allowing only specific SecretInput paths.

## Solution Assessment

Unclear until maintainers choose the contract; the shipped manifest-level providerIntegration path is the right partial seam, but arbitrary all-field SecretRef expansion is not the safest final shape without an allowlisted trust model.
