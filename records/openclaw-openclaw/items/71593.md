## Summary

Keep open. Current main still has a closed SecretRef provider model and narrower field coverage than requested, and the issue is paired with open closing PR #72548, so the right path is maintainer review of that PR rather than early cleanup or a separate repair job.

## What This Changes

Allow plugins to define SecretRef provider sources and broaden SecretRef support across OpenClaw config fields so users can mask plugin, skill, and channel secrets without repeated core schema changes.

## Best Possible Solution

Land or replace #72548 with a versioned, documented plugin SecretRef provider contract that preserves strict built-in env/file/exec behavior, honors plugin activation and origin boundaries, and separately tracks any remaining field-coverage gaps such as generic plugin config and skill env values.

## Reproduction Assessment

Not applicable as a feature/API request. The gap is high-confidence by static inspection: current main rejects non-env/file/exec sources in config typing, zod schemas, gateway wire schemas, SDK SecretInput schema, and resolver dispatch.

## Solution Assessment

Unclear until maintainers review #72548. A plugin-defined provider seam matches the plugin-first direction, but this is security-sensitive and the current implementation candidate intentionally defers the broader schema coverage requested here.
