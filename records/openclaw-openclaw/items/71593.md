## Summary

Keep open. Current main still exposes a closed SecretRef provider model and narrower SecretRef field coverage than requested, and the issue is paired with open closing PR #72548, so this should stay open until that PR is reviewed, landed, replaced, or closed.

## What This Changes

Request to let plugins define SecretRef providers and make SecretRef objects usable across core credential schemas such as channel credentials, skill env values, and plugin config.

## Best Possible Solution

Land or replace #72548 with a versioned, documented plugin SecretRef provider contract that preserves strict built-in env/file/exec behavior, honors plugin activation and origin boundaries, and tracks any remaining field-coverage gaps as explicit follow-up work.

## Reproduction Assessment

Not applicable as a feature/API request. The gap is high-confidence by static inspection of current main: config types, zod schemas, gateway wire schemas, and resolver dispatch are still closed to env/file/exec provider sources.

## Solution Assessment

Unclear until maintainers review #72548. A plugin-defined provider seam matches the plugin-first direction, but the final solution needs explicit security/API review and a separate decision on broad SecretRef field coverage.
