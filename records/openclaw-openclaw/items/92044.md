## Summary

Keep this issue open: current main and the latest release still contain the Workboard CLI scope override, while the Workboard gateway method and docs say dispatch should work with `operator.write`. The open linked fix PR is the active implementation candidate and should be reviewed or landed before this issue closes.

## What This Changes

The issue reports that `openclaw workboard dispatch` is rejected as needing `operator.admin` even though the Workboard dispatch method is documented and registered as `operator.write`.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/92066 or an equivalent narrow fix, then close this issue after release or main proof shows CLI dispatch no longer forces the under-scoped call.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: current main still forces `operator.write`/`operator.read` for Workboard CLI dispatch while the generic CLI fallback would request broader scopes for unclassified plugin methods. A live Gateway command was not run in this read-only review.

## Solution Assessment

Yes for the immediate issue: the narrowest maintainable fix is to stop the Workboard CLI from overriding shared gateway scope selection. A broader registry-descriptor fix may still be worth tracking separately for related reports, but it is not required before keeping this issue open for the linked PR.
