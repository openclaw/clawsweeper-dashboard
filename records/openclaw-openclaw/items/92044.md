## Label Justifications

- `P1`: This blocks the documented headless Workboard dispatch workflow for users running the bundled plugin through the CLI.
- `impact:auth-provider`: The failure is caused by Gateway operator-scope resolution and rejects a documented `operator.write` call as needing `operator.admin`.

## Summary

Keep open: current main and the latest release still contain the reported hardcoded Workboard CLI scope path, and the open linked PR is the active implementation candidate rather than a reason to close the issue before merge.

## What This Changes

`openclaw workboard dispatch` should be able to call `workboard.cards.dispatch` with the documented `operator.write` authorization instead of failing as `missing scope: operator.admin`.

## Best Possible Solution

Review, revise if needed, and land a narrow fix that lets Workboard CLI dispatch succeed while preserving the intended Gateway method scope contract, then close this issue after the fix merges.

## Reproduction Assessment

Yes. Source inspection shows current main still sends only `operator.write`/`operator.read` from the Workboard CLI, while unclassified gateway methods authorize as `operator.admin`, and the docs define dispatch as a write-scope operation.

## Solution Assessment

Unclear until the linked PR is reviewed. Removing the explicit CLI scopes is a narrow plausible repair, but maintainers should confirm it is better than fixing the plugin descriptor snapshot used by gateway scope classification.
