## Summary

Keep this issue open: current main still matches the reported Workboard CLI scope path, the latest release still contains the same hard-coded CLI scopes, and an open linked fix PR is already the active implementation candidate.

## What This Changes

The issue reports that `openclaw workboard dispatch` is rejected as `missing scope: operator.admin` even though the Workboard dispatch Gateway method is documented and registered as `operator.write`.

## Best Possible Solution

Keep the issue open until the linked PR or an equivalent general registry-scope fix lands with proof that Workboard CLI dispatch succeeds under the documented `operator.write` path.

## Reproduction Assessment

Yes, source-reproducible with the reported descriptor miss: current CLI dispatch presents only `operator.write`/`operator.read`, while Gateway authorization falls back to `operator.admin` for unclassified methods. I did not run a live Gateway in this read-only sweep.

## Solution Assessment

Unclear: the linked PR is a small plausible fix by letting the CLI use the broader default for unclassified plugin methods, but maintainers should verify whether the better permanent fix is to keep plugin Gateway descriptors visible to the scope resolver.
