## Summary

Keep open: the bug is real and the PR moves in the right direction, but it flips a shared process-kill default without migrating existing detached callers, which would regress process-group cleanup.

## What This Changes

The PR changes Unix process-tree termination to direct-PID kills by default, requires `detached: true` for process-group kills, updates one exec timeout caller, and adjusts kill-tree tests.

## Best Possible Solution

Make non-detached kills safe while preserving group cleanup for every actually detached process-group leader, either by migrating all such callers to pass `detached: true` or by adding a helper-side group-leader guard.

## Reproduction Assessment

Yes at source level: current main defaults omitted detachment to group kill, and the PR flips that default while detached production callers still omit the option. I did not run a live macOS gateway or real MCP process-tree reproduction in this read-only review.

## Solution Assessment

No as submitted: changing the shared default is a plausible core fix for the gateway SIGTERM bug, but it is too broad unless existing detached process-group callers are migrated or the helper verifies group leadership.

- **[P1] Preserve group kills for detached callers:** `packages/agent-core/src/harness/env/kill-tree.ts:37`
  - body: This default flip changes every omitted-option caller to direct-PID kills, but some omitted callers are intentionally detached. `OpenClawStdioClientTransport.start()` spawns MCP servers detached on non-Windows, and `close()` currently relies on `killProcessTree(pid)` / `signalProcessTree(pid, "SIGKILL")` to reap the process group. After this patch those calls only signal the parent PID, so MCP descendants can survive transport shutdown. Audit the detached callers and pass `detached: true`, or make the helper detect group leaders, before changing the shared default.
