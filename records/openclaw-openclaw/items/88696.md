## Summary

Keep open: this is a focused memory-core bug fix, but the new DELETE fallback can still leave `vector.available` true while `chunks_vec` is absent, and the PR proof is only targeted tests rather than real unsafe-reindex behavior with sqlite-vec enabled.

## What This Changes

The PR changes memory-core unsafe reindex reset logic to delete rows from `chunks_vec` instead of dropping the vector table when sqlite-vec is available.

## Best Possible Solution

Keep the row-delete approach, but make the fallback leave vector readiness and table existence consistent before sync prepares vector deletes, then add focused regression coverage and real unsafe-reindex proof.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main can keep `vector.available` true after dropping `chunks_vec`, then prepare `DELETE FROM chunks_vec` before the write path can recreate the table. I did not run a live Windows/sqlite-vec unsafe reindex in this read-only review.

## Solution Assessment

No, not yet: the direction is right, but the added fallback can preserve the same availability/table mismatch. The narrow fix is to either keep/create the vector table or reset vector readiness/availability before sync setup can prepare vector-table statements.
