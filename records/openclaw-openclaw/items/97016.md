- `impact:crash-loop`: The reported path can exhaust CUDA VRAM during LM Studio model preload and make semantic memory search unavailable for affected local setups.

## Summary

Keep open. Current main and v2026.6.10 still omit `requestedContextLength` on the LM Studio memory embedding preload, while the shared loader falls back to the default load context capped by LM Studio's advertised maximum; that makes the reported 8k-to-32k load path credible from source.

## What This Changes

LM Studio `memory_search` embedding preload should honor the configured `contextTokens`/`contextWindow` instead of loading the embedding model at the default or advertised maximum context.

## Best Possible Solution

Resolve the LM Studio embedding model's configured `contextTokens`/`contextWindow` in the plugin and pass it to `ensureLmstudioModelLoaded`, sharing the inference-side invariant and adding a focused regression test.

## Reproduction Assessment

Yes from source, but not from a live CUDA run: an LM Studio memory embedding preload with configured `contextWindow: 8192` currently calls the loader without `requestedContextLength`, so a model advertising 32768 can be loaded at 32768.

## Solution Assessment

Yes. The best fix is to pass the existing configured context cap into the existing LM Studio loader, preferably reusing the inference resolver that already prefers `contextTokens` over `contextWindow`.
