## Summary

Keep open: current main still has adjacent prompt, tool, history, and session controls, but no user-facing per-session chat versus agent mode. This is an additive core runtime and UX feature that needs maintainer design before implementation.

## What This Changes

Add an opt-in session-level `/mode chat|agent` switch that persists per session and drives lighter prompt assembly, shorter history, and reduced tool exposure for low-cost conversations.

## Best Possible Solution

Define a core session-mode contract that consistently drives command UX, durable session state, prompt assembly, history limiting, tool policy, docs, and regression tests across TUI, WebChat, and messaging channels.

## Reproduction Assessment

Not applicable. This is an additive feature request, and source review confirms there is no current-main `/mode chat|agent` behavior to reproduce as a bug.

## Solution Assessment

Unclear until maintainer design. The requested direction is plausible, but the maintainable solution should be a core session/runtime contract rather than only a slash-command alias because it affects prompt assembly, history, and tool policy.
