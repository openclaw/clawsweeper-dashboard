## Summary

Keep open: current main has adjacent internal prompt modes, config-level tool profiles, history limits, and per-session controls, but no user-facing session-level chat/agent mode. Implementing this would change core session/runtime semantics and needs maintainer product and security design first.

## What This Changes

Add a session-persistent `/mode chat|agent` switch that runs lightweight chat sessions with reduced prompt, history, and tool exposure while preserving full agent behavior in agent mode.

## Best Possible Solution

Define a durable core session-mode contract that consistently drives command UX, persisted session state, prompt assembly, history limits, tool policy, docs, and tests across TUI, WebChat, and messaging channels.

## Reproduction Assessment

Not applicable. This is an additive feature request, and source review confirms there is no current-main `/mode chat|agent` behavior or persisted chat/agent session mode to reproduce as a bug.

## Solution Assessment

Unclear until maintainer design. The requested direction is plausible, but the maintainable solution should be a core session/runtime contract rather than only a slash-command alias because it affects prompt assembly, history, and tool policy.
