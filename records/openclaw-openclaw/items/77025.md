## Summary

Keep open: current main has related building blocks, including internal prompt modes, tool profiles, and history limits, but no user-facing per-session chat/agent mode. The request changes core session/runtime behavior and needs maintainer product design before implementation.

## What This Changes

Request an opt-in per-session `/mode chat|agent` switch that makes lightweight conversations use a shorter prompt/history and reduced tool exposure while preserving full agent behavior.

## Best Possible Solution

Define a core session mode contract and implement it as durable per-session state that consistently drives command UX, prompt assembly, history limiting, tool policy, docs, and tests across TUI, WebChat, and messaging channels.

## Reproduction Assessment

Not applicable. This is an additive feature request, and source review confirms there is no current-main `/mode chat|agent` behavior to reproduce as a bug.

## Solution Assessment

Unclear until maintainer design. The requested direction is plausible, but the maintainable solution should be a core session/runtime contract rather than only a slash-command alias because it affects prompt assembly, history, and tool policy.
