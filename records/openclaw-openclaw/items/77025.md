## Summary

Keep open: current main and v2026.6.10 still lack a user-facing, session-persistent chat/agent mode, and the remaining work is a cross-surface product/runtime design rather than conservative cleanup.

## What This Changes

Add a session-level chat/agent mode switch that uses lighter prompt, history, and tool exposure for low-cost conversational sessions while preserving full agent behavior separately.

## Best Possible Solution

Define a maintainer-approved session-mode contract first, then wire durable session state, command/UI surfaces, prompt/history/tool policy, docs, and focused tests without changing the default full-agent behavior.

## Reproduction Assessment

Not applicable: this is an additive feature request, not a broken existing behavior. Source and release inspection show adjacent session controls but no requested `/mode chat` or `/mode agent` behavior on current main or v2026.6.10.

## Solution Assessment

Unclear until maintainer design: the proposed slash-command UX is plausible, but the best implementation needs one core session-mode contract rather than exposing today’s lower-level promptMode, bootstrapContextMode, and tool-allowlist primitives directly.
