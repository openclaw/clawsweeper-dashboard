## Summary

Keep open: current main has adjacent internal prompt modes, config-level tool profiles, and history limits, but no user-facing per-session chat/agent mode. Implementing this would change core session/runtime behavior and needs maintainer product and security design before a focused implementation is safe.

## What This Changes

Add an opt-in session-level chat/agent mode switch with slash-command UX, persisted session state, lighter prompt/history behavior, and reduced tool exposure for lightweight conversations.

## Best Possible Solution

Define a core session mode contract and implement it consistently across command UX, persisted session state, prompt assembly, history limiting, tool policy, docs, and tests for TUI, WebChat, and messaging channels.

## Reproduction Assessment

Not applicable. This is an additive feature request, and source review confirms current main does not expose `/mode chat|agent` behavior to reproduce as a bug.

## Solution Assessment

Unclear until maintainer design. The requested direction is plausible, but the durable solution should be a core session/runtime contract rather than only a slash-command alias because it affects prompt assembly, history, and tool policy.
