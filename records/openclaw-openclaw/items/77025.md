## Summary

Keep open: current main has related primitives, but it does not implement the requested user-facing per-session chat/agent mode, and the request changes core runtime semantics that need maintainer product design.

## What This Changes

Request an opt-in session-level `/mode chat|agent` switch that persists per session and changes prompt weight, history window, and tool exposure.

## Best Possible Solution

Define a durable core session mode contract that drives slash command/status UX, persisted session state, prompt assembly, history limiting, tool policy, docs, and tests across TUI, WebChat, and messaging channels.

## Reproduction Assessment

Not applicable. This is an additive feature request, and source review shows ordinary sessions still resolve to the full agent path with no current `/mode chat|agent` behavior.

## Solution Assessment

Unclear until maintainer design. The requested direction is plausible, but the maintainable solution should be a coherent core session/runtime contract rather than a UI-only alias or one-off tool-profile shortcut.
