## Summary

Keep open: current main has related building blocks, including internal prompt modes, tool profiles, history limits, and lean-mode config, but no user-facing per-session `/mode chat|agent` switch or persisted mode that drives prompt, history, and tool policy. This is a core runtime/product design feature, not a cleanup close.

## What This Changes

Request to add an opt-in per-session chat/agent mode switch that makes chat sessions use lighter prompt, shorter history, and reduced tool exposure.

## Best Possible Solution

Define a durable core session mode contract that consistently drives command UX, persisted session state, prompt assembly, history limiting, tool policy, docs, and tests across TUI, WebChat, and messaging channels.

## Reproduction Assessment

Not applicable. This is an additive feature request, and source review confirms there is no current-main `/mode chat|agent` behavior to reproduce as a bug.

## Solution Assessment

Unclear until maintainer design. The requested direction is plausible, but the maintainable solution should be a core session/runtime contract rather than only a slash-command alias because it affects prompt assembly, history, and tool policy.
