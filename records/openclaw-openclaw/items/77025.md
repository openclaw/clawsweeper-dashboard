## Summary

Keep open: current main has adjacent building blocks for prompt modes, tool profiles, history limits, and session controls, but it does not provide a user-facing per-session `chat`/`agent` mode. This is an additive core runtime and UX feature that needs maintainer product/security design before implementation.

## What This Changes

The issue requests a session-level `/mode chat|agent` switch that persists per session and uses lighter prompt, history, and tool exposure for low-cost conversations.

## Best Possible Solution

Define a core session-mode contract and implement it as durable per-session state that consistently drives command UX, prompt assembly, history limiting, tool policy, docs, and tests across TUI, WebChat, and messaging channels.

## Reproduction Assessment

Not applicable. This is an additive feature request, and source review gives a high-confidence current-main check that `/mode chat|agent` and a persisted chat/agent session mode do not exist.

## Solution Assessment

Unclear until maintainer design. The requested direction is plausible, but the maintainable solution should be a core session/runtime contract rather than only a slash-command alias because it affects prompt assembly, history policy, tool exposure, and multiple clients.
