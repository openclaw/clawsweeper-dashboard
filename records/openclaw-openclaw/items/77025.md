## Summary

Keep open: current main has related building blocks, but no user-facing per-session chat/agent mode. The request changes core session state, prompt assembly, history trimming, and tool policy, so it needs maintainer product design before implementation.

## What This Changes

Add an opt-in session-level chat/agent mode switch that uses a lighter prompt, shorter history, and reduced tools for lightweight conversations while preserving full agent behavior.

## Best Possible Solution

Define a core session mode contract that consistently drives command UX, durable session state, prompt assembly, history limiting, tool policy, docs, and regression tests across TUI, WebChat, and messaging channels.

## Reproduction Assessment

Not applicable. This is an additive feature request, and source/docs review confirms current main has no `/mode chat|agent` behavior to reproduce as a bug.

## Solution Assessment

Unclear until maintainer design. The requested direction is plausible, but the maintainable solution should be a core session/runtime contract rather than only a slash-command alias because it affects prompt assembly, history, and tool policy.
