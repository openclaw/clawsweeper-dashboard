## Summary

Keep open: the PR is aimed at the right media-reference gap, but it is not mergeable because the touched file does not typecheck and the resolver is wired only into the optional workspace guard rather than the read tool path that fails by default.

## What This Changes

The branch imports `resolveMediaReferenceLocalPath` and attempts to substitute resolved `media://` paths inside `wrapToolWorkspaceRootGuardWithOptions` before sandbox checks.

## Best Possible Solution

Resolve `media://` references in the read execution path for all filesystem policies, preserve sandbox boundaries with an explicit managed-inbound-media allowance, and add focused read-tool regression coverage.

## Reproduction Assessment

Yes. Source inspection shows current main can place `media://inbound/...` in prompt notes while the default read tool path still sends that string to the upstream filesystem resolver; CI also gives a high-confidence reproduction of this PR's type failure.

## Solution Assessment

No. Reusing the existing media resolver is the right direction, but placing it only inside the optional workspace guard misses the default read path and still needs a guarded managed-media allowance.

## Review Findings

Overall correctness: patch is incorrect

Full review comments:

- **[P1] Keep the normalized path typed as string:** `src/agents/pi-tools.read.ts:686`
  - body: Changing `filePath` to `let filePath = record?.[key]` leaves it inferred as `unknown` after the resolver block, and CI now fails `tsgo` with `TS2322` at later `filePath` uses in this function.   - body: This resolver block only runs when `wrapToolWorkspaceRootGuardWithOptions` is applied, but current main skips that wrapper for the default `tools.fs.workspaceOnly` unset/false read tool. In that common configuration, `Read({ path: "media://inbound/..." })` still reaches the upstream read tool unchanged, so the linked issue remains unfixed.
  - confidence: 0.94
- **[P2] Allow managed inbound media through the guard:** `src/agents/pi-tools.read.ts:703-705`
  - body: For hardened `workspaceOnly` reads, this resolves `media://inbound/...` to an absolute managed-media path and then immediately checks it against only the workspace/additional roots, so the guard still rejects it as outside the sandbox. Add a narrow inbound-media allowance or equivalent validated path before calling the base read tool.
  - confidence: 0.9
