## Summary

Keep open: the PR uses the right existing media helper, but it wires it into the workspace-only guard instead of the actual read path, so the default `read` flow still misses `media://inbound/...` and workspace-only reads still fail the root guard. The supplied proof is terminal output, but it uses a stand-in resolver rather than exercising the patched OpenClaw read tool.

## What This Changes

The PR imports `resolveMediaReferenceLocalPath` and resolves `media://` path arguments inside `wrapToolWorkspaceRootGuardWithOptions` before sandbox validation.

## Best Possible Solution

Normalize supported `media://inbound/...` references in the actual OpenClaw read execution path, preserve the managed-inbound media safety checks, and add focused regression coverage for default and workspace-only reads.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: current main can produce `media://inbound/...` notes, while the default read tool bypasses the new guard-only resolver path. I did not run a live Signal or Telegram attachment flow in this read-only review.

## Solution Assessment

No, this patch is not the best fix because it runs only in the workspace-only guard and still lacks the inbound media allowance there. The narrower maintainable solution is to resolve supported media references inside the read path itself and test both guarded and unguarded modes.
