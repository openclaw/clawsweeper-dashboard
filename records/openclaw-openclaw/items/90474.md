## Summary

Keep this PR open: it targets a real repeated legacy install-index warning, but the new supersession guard can archive conflicted legacy metadata when one matching resolved field masks disagreeing spec/version fields.

## What This Changes

The PR adds a legacy install-index migration guard that archives older npm JSON records when current SQLite metadata has matching package identity, an install path, and a newer version, with regression tests.

## Best Possible Solution

Tighten the guard so every present npm package/version identity field agrees before archiving, then land only if maintainers accept current SQLite as authoritative for that bounded upgrade state.

## Reproduction Assessment

Yes. Source inspection shows current main leaves conflicting legacy/current install records unarchived, so the same legacy file can be read and warned on future doctor or migration paths.

## Solution Assessment

No, not yet. The fix is in the right migration owner, but the new helper should reject internally inconsistent package/version metadata instead of trusting the first matching field.
