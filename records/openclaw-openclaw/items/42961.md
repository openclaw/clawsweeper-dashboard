## Summary

Keep this PR open. Current main still injects short OpenRouter Anthropic cache markers without reading `cacheRetention`, so the PR addresses a real remaining cost/caching bug. The contributor's live OpenRouter observations, Greptile's review, and current source all point to the same gap, though the branch needs a careful refresh against the newer provider-wrapper and endpoint-gating code.

## What This Changes

The PR makes OpenRouter Anthropic models resolve `cacheRetention`, passes that value into the OpenRouter system cache wrapper, suppresses `cache_control` for `none`, emits `ttl: "1h"` for `long`, and adds regression tests for those cases.

## Best Possible Solution

Land a refreshed implementation that threads the resolved OpenRouter Anthropic retention setting into the current verified-route payload wrapper/helper, keeps default short behavior, suppresses markers for `none`, emits `{ type: "ephemeral", ttl: "1h" }` for `long`, preserves endpoint gating and thinking-block cleanup, and updates regression tests for the full retention matrix.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure `cacheRetention: "none"` or `"long"` for an `openrouter/anthropic/*` model and inspect the payload path: current main has no retention input to the OpenRouter marker wrapper and always writes the short marker form. The PR also provides live OpenRouter log observations for the end-to-end cost symptom.

## Solution Assessment

Yes in direction, but the branch needs refresh against current main. The safest implementation is the PR's retention-aware behavior applied to the current wrapper/helper shape rather than reverting to the older, provider-only wrapper composition.
