## Summary

Keep open: the feature direction is plausible, but this draft still has P1 privacy/security blockers and insufficient real behavior proof. Current main does not implement the requested Claude CLI rolling marker/tool-event behavior, and the overlapping open bridge work at https://github.com/openclaw/openclaw/pull/80046 needs maintainer sequencing rather than cleanup closure.

## What This Changes

This PR adds Claude stream-json tool-use tracking, inline assistant tool markers with a rolling timer, and structured tool events from the Claude CLI runner paths.

## Best Possible Solution

Keep the feature behind the existing session-aware verbose and redaction contracts, add focused parser/runner/server-chat coverage, and reconcile the approach with the overlapping CLI tool-event bridge before merge.

## Reproduction Assessment

Yes for the review findings by source inspection: current main has session-aware verbose resolution and a sanitizer contract, while the PR uses a run-context-only inline gate and raw Claude args. No current-main bug reproduction is required because this is a feature PR.

## Solution Assessment

No. The concept is useful, but the patch should reuse the existing session-aware verbose and tool-argument redaction contracts instead of creating a weaker parallel path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Use the session-aware verbose gate for inline markers:** `src/agents/cli-runner/execute.ts:467-472`
  - body: The inline marker resolver only checks `getAgentRunContext` plus the config default. Current `server-chat` also reloads the session entry when it changed after the run started, and `sessions.patch` persists verbose changes without updating this run context, so a mid-run verbose-off change can stop structured tool delivery while these assistant-delta markers still reveal command/path/query details.
  - confidence: 0.9
- **[P1] Redact inline tool details before streaming them:** `src/agents/cli-output.ts:471-475`
  - body: The marker text is built directly from raw Claude args such as `command`, `file_path`, `query`, and `url` before any redaction helper runs. A command or URL containing a token can therefore be sent as normal assistant text to Telegram/chat surfaces, bypassing the existing tool-event sanitizer contract.
  - confidence: 0.88
- **[P1] Redact Claude tool args before publishing events:** `src/agents/cli-runner/execute.ts:506`
  - body: The new tool-event emitters forward `evt.args` raw into `stream: "tool"` payloads. The embedded runtime sanitizes tool args before publishing because these payloads can include headers, API keys, or token-bearing commands, and this path should preserve that same contract at both emit sites.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The diff introduces concrete privacy regressions around inline tool markers and raw structured tool args.

Concerns:

- **[medium] Inline markers can bypass newer verbose-off session state:** `src/agents/cli-runner/execute.ts:467`
  - body: The marker decision is re-evaluated per tool, but it only reads the run context and config default; current channel delivery also honors a newer persisted session entry. A user toggling verbose off mid-run can still receive raw assistant marker text.
  - confidence: 0.9
- **[medium] Raw Claude tool args skip the existing redaction contract:** `src/agents/cli-runner/execute.ts:506`
  - body: The PR forwards Claude tool args and formats inline details before applying `sanitizeToolArgs` or equivalent redaction, so secret-bearing commands, headers, URLs, or fields can reach live consumers.
  - confidence: 0.88
