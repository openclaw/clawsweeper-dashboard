## Summary

Keep open. This is a recent draft feature PR with useful direction, but the latest head still has blocking privacy/security regressions around inline marker gating and raw tool args, and the supplied Telegram proof only covers part of the changed behavior.

## What This Changes

This PR adds Claude stream-json tool-use tracking, inline assistant tool markers with a rolling timer, and `stream: "tool"` events from the Claude CLI runner paths.

## Best Possible Solution

Keep the feature behind the existing session-aware verbose and redaction contracts, add focused parser/runner/server-chat tests, and refresh real Telegram plus structured-event proof before choosing this path or the related bridge PR.

## Reproduction Assessment

Yes for the review findings by source inspection: current main has a session-aware verbose resolver and sanitizer contract, while the PR uses a run-context-only inline gate and raw Claude args. No current-main bug reproduction is required because this is a feature PR.

## Solution Assessment

No. The concept is plausible, but the implementation should reuse the existing session-aware verbose and tool-argument redaction contracts instead of creating a weaker parallel path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Use the session-aware verbose gate for inline markers:** `src/agents/cli-runner/execute.ts:469-473`
  - body: The inline marker resolver only checks `getAgentRunContext` plus the config default. Current `server-chat` also reloads the session entry when it changed after the run started, and `sessions.patch` persists verbose changes without updating this run context. If a user turns verbose off during a long Claude run, structured tool events stop reaching chat surfaces but these assistant-delta markers can still send command/path/query details.
  - confidence: 0.9
- **[P1] Redact Claude tool args before publishing them:** `src/agents/cli-runner/execute.ts:506`
  - body: The existing embedded runtime sanitizes tool args before publishing `stream: "tool"` because args can include Authorization headers, API keys, or commands with tokens. This PR emits `evt.args` raw at both new Claude emit sites, and the inline marker formats raw command/file/query values too, so verbose consumers can receive secrets the current tool-event path would mask.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The patch introduces concrete secret-exposure risks in both inline assistant markers and structured Claude tool events.

Concerns:

- **[medium] Inline markers can bypass a later verbose-off session change:** `src/agents/cli-runner/execute.ts:473`
  - body: The marker decision is re-evaluated per tool, but it only reads the run context and default; current channel delivery also honors a newer session entry. A mid-run verbose-off session change can therefore stop structured tool events while inline assistant markers continue to reveal tool details.
  - confidence: 0.9
- **[medium] Raw Claude tool args skip the existing redaction contract:** `src/agents/cli-runner/execute.ts:506`
  - body: The PR forwards Claude tool args and formats inline details before applying the existing `sanitizeToolArgs` or tool-detail redaction helpers, so token-bearing commands, headers, URLs, or secret fields can reach live consumers.
  - confidence: 0.88
