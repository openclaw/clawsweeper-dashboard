## Summary

Keep open. The Telegram rendering idea is useful, but the latest head still changes documented `/reasoning stream` cleanup, double-formats existing reasoning splitter output, and can skip retry delivery after transient Telegram send failures; it also still depends on https://github.com/openclaw/openclaw/pull/80046 for the canonical CLI tool-event source.

## What This Changes

The PR changes Telegram draft rendering to combine reasoning text, CLI tool-progress lines, and a rolling timer in one reasoning preview, and changes draft streams to continue after transient Telegram send errors.

## Best Possible Solution

Keep the Telegram renderer behind the existing stream semantics: consume the canonical CLI bridge after it lands, preserve `/reasoning stream` cleanup unless maintainers explicitly choose a new persistent mode, avoid rewrapping formatted splitter output, and add focused Telegram dispatch/draft-stream coverage plus latest-head live proof.

## Reproduction Assessment

Yes for the PR defects: source inspection against current docs, splitter tests, and the shared draft-loop contract shows the cleanup, formatting, and retry-cache paths. This is a feature PR rather than a current-main bug report.

## Solution Assessment

No. The rendering goal is reasonable, but the current implementation should preserve existing reasoning-stream lifecycle, reuse splitter output without reformatting it, fix transient retry cache handling, and wait for the canonical CLI event bridge.
