## Summary

Keep open. The Telegram UX goal is useful, but the current branch still has blocking stream-lifecycle, formatting, and retry-cache defects, depends on the open canonical CLI bridge, and the supplied proof does not cover the latest rescoped head.

## What This Changes

The branch changes Telegram dispatch to render reasoning and CLI tool-progress with a rolling timer in one reasoning draft message, and changes Telegram draft streams to continue after transient send errors.

## Best Possible Solution

Keep the Telegram renderer behind the canonical CLI bridge, preserve `/reasoning stream` cleanup unless maintainers explicitly choose a persistent mode, reuse splitter output without rewrapping it, and add focused tests plus latest-head Telegram proof.

## Reproduction Assessment

Yes for the PR defects: source inspection against current docs, splitter tests, and the draft-stream send cache shows the cleanup, formatting, and retry paths. This is a feature PR rather than a current-main bug report.

## Solution Assessment

No. The rendering goal is reasonable, but the current implementation should preserve existing reasoning-stream lifecycle, avoid reformatting already formatted splitter output, fix transient retry cache handling, and wait for the canonical CLI event bridge.

  - body: `/reasoning stream` is documented as a live preview that is deleted after final delivery, with `/reasoning on` reserved for persistent reasoning.   - confidence: 0.92

  - body: `splitTelegramReasoningText(..., true)` already returns a formatted `Thinking\n\n_..._` message. Wrapping every returned line in underscores and then prepending another `Thinking` label produces duplicate labels and double italics in the reasoning draft.
  - confidence: 0.9
- **[P2] Reset the sent cache after transient failures:** `extensions/telegram/src/draft-stream.ts:287`
  - body: `lastSentText` is updated before Telegram accepts the send/edit. Returning `false` for transient errors without clearing it means the next identical pending preview hits the equality fast path and is treated as delivered even though Telegram never accepted it.
  - confidence: 0.86
