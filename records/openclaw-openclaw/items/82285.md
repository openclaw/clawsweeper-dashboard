## What This Changes

The PR modifies Telegram draft streaming to interleave reasoning and tool-progress lines with a rolling timer in the reasoning lane, and changes draft streams to continue after transient Telegram send errors.

## Best Possible Solution

Land the canonical CLI tool-event producer first, then merge a Telegram renderer that preserves `/reasoning stream` as transient, formats reasoning through the existing splitter contract once, fixes transient retry state, covers those paths with focused tests, and includes updated live Telegram proof.

## Reproduction Assessment

Yes for the PR defects: source inspection shows the PR marks the reasoning lane finalized against the current transient docs, rewraps already formatted splitter output, and returns from transient send failures after advancing `lastSentText`. I did not run tests because this review is read-only.

## Solution Assessment

No. The Telegram-side direction should preserve the existing transient reasoning contract, reuse the splitter output correctly, and fix retry state before adding the rolling interleave UX.

Overall confidence: 0.9

Full review comments:

- **[P1] Keep streamed reasoning transient:** `extensions/telegram/src/bot-message-dispatch.ts:1700`
  - body: Current docs say Telegram `/reasoning stream` is a transient draft deleted after final delivery, with `/reasoning on` as the persistent mode. This assignment makes cleanup keep the reasoning lane, so reasoning and tool details remain in chat after the final answer. Please keep the stream lane clearable by default, or make persistence an explicit maintainer-approved product/docs change.
  - confidence: 0.92
- **[P2] Do not rewrap formatted reasoning output:** `extensions/telegram/src/bot-message-dispatch.ts:1688-1691`
  - body: `splitTelegramReasoningText(payload.text, true)` already returns `formatReasoningMessage(...)` output such as `Thinking\n\n_Thinking_`. The new code then prepends another `Thinking` header and wraps every non-empty line in underscores, so tagged reasoning renders with duplicate labels and double italics. Use the formatted splitter result directly or extract raw reasoning before adding interleaved markup.
  - confidence: 0.9
- **[P2] Reset sent cache on retryable preview failures:** `extensions/telegram/src/draft-stream.ts:283-287`
  - body: `lastSentText` and `lastSentParseMode` are set before the Telegram send/edit completes. When this new transient-error branch returns `false`, the next flush with the same pending text hits the equality fast path and can be treated as delivered without another API call; ambiguous first-send 429/5xx failures also still risk duplicate preview sends. Roll the cache back on retryable failures or only update it after confirmed delivery while preserving `sendMayHaveLanded` semantics.
  - confidence: 0.86
