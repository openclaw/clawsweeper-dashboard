## Summary

This PR should stay open: the Telegram interleaving idea is useful, but the current patch still changes documented reasoning-stream cleanup semantics, corrupts/duplicates formatted reasoning during cumulative updates, can skip retry delivery after transient Telegram errors, and depends on a separate open CLI tool-event bridge.

## What This Changes

This PR changes Telegram streaming to render reasoning, CLI tool-progress lines, and a rolling timer in one reasoning draft message, and changes draft streams to continue after transient Telegram send errors.

## Best Possible Solution

Consume the canonical CLI tool-event bridge after it lands, keep /reasoning stream transient unless maintainers explicitly change that policy, format reasoning once from raw cumulative data, repair the transient retry cache, and add focused tests plus latest-head live Telegram proof.

## Reproduction Assessment

Yes for the PR defects: current docs, splitter behavior, cumulative reasoning tests, and the draft-loop retry contract make the failure paths source-reproducible. This is a feature PR rather than a current-main bug report.

## Solution Assessment

No. The rendering goal is reasonable, but the patch should preserve existing reasoning-stream lifecycle, avoid formatting/checkpointing already formatted text, fix transient retry cache handling, and wait for the canonical CLI event bridge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Track raw reasoning before formatting:** `extensions/telegram/src/bot-message-dispatch.ts:1685-1693`
  - body: `splitTelegramReasoningText(..., true)` already returns formatted `Thinking\n\n_..._` text, while the CLI reasoning bridge sends cumulative text. Checkpointing that formatted wrapper and then wrapping every line in underscores both double-formats the first update and corrupts later cumulative updates because new content is inserted before the trailing underscore, so the next slice can miss the actual added reasoning.
  - confidence: 0.94
- **[P1] Preserve stream-only reasoning cleanup:** `extensions/telegram/src/bot-message-dispatch.ts:1698`
  - body: `/reasoning stream` is documented as a transient Telegram preview that is deleted after final delivery, with `/reasoning on` reserved for persistent reasoning. Marking the reasoning lane finalized here makes cleanup stop the stream instead of clearing it, so reasoning and interleaved tool details remain in chat history for users who selected the temporary mode.
  - confidence: 0.93
- **[P2] Restore the sent cache after transient preview failures:** `extensions/telegram/src/draft-stream.ts:288`
  - body: `lastSentText` and `lastSentParseMode` are assigned before the Telegram transport call. In this new transient-error path, returning `false` requeues the same pending text, but the next send sees it as already sent and returns `true` without calling Telegram, so the retry can be silently skipped.
  - confidence: 0.9
