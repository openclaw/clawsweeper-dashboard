## Summary

Keep this draft PR open. The feature is plausible, but the current patch is not merge-ready because it sends raw Claude tool arguments into visible/structured streams and uses `itemId` instead of the established `toolCallId` field that current tool consumers require.

## What This Changes

The branch adds Claude stream-json tool-use tracking, assistant-text tool markers with a rolling timer, and optional `stream: "tool"` event emission from CLI runner paths.

## Best Possible Solution

Keep the draft open until it reuses the established sanitized `toolCallId` tool-event contract, redacts visible detail text, adds focused parser coverage, and provides live proof for the marker/timer plus structured event path.

## Reproduction Assessment

Yes for the review findings from source inspection: current consumers require `data.toolCallId` and existing tool producers sanitize args, while this PR emits `itemId` and raw args. I did not run the PR branch because this was a read-only review.

## Solution Assessment

No. The user-facing tool marker can be useful, but the maintainable path is to reuse the existing sanitized tool event contract instead of adding a parallel raw payload shape.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Redact Claude tool args before streaming them:** `src/agents/cli-output.ts:493`
  - body: `emitTool` forwards the raw Claude args and also derives the visible marker detail from raw `command`/`file_path`/`url`-style fields. The existing tool event path runs args through `sanitizeToolArgs` before emitting, so this can expose tokens, private URLs, or sensitive command text into Telegram/control UI streams. Redact before choosing the display detail and before calling `onToolEvent`.
  - confidence: 0.87
- **[P2] Emit the established toolCallId field:** `src/agents/cli-runner/execute.ts:492`
  - body: The new `stream: "tool"` payload only includes `itemId`, but current consumers such as the Control UI read `data.toolCallId` and return early when it is missing. Existing producers emit `toolCallId`; as written these structured CLI tool events will not render in tool tiles. Send `toolCallId` at both new emitters and align the parser event type with that contract.
  - confidence: 0.92
