## Summary

Keep open: current main still has the recursive clone pattern, and this PR is a useful narrow performance fix, but it should not merge until the wrapper preserves caller immutability and the contributor adds real benchmark or runtime proof.

## What This Changes

The PR refactors `src/config/schema.hints.ts` so `mapSensitivePaths` uses a private in-place accumulator instead of shallow-copying the hints map during every recursive Zod schema visit.

## Best Possible Solution

Keep the O(N) accumulator approach, but clone once in the public wrapper, mutate only the private accumulator, and add focused proof for immutability plus full-schema timing.

## Reproduction Assessment

No high-confidence wall-clock reproduction is present for the reported 30s stall. Source inspection does confirm the current recursive helper clones the growing hint map on each schema visit, so the algorithmic issue is source-reproducible.

## Solution Assessment

No, not as written. The private mutating accumulator is the right fix shape, but the public wrapper should clone once before recursion so the traversal becomes O(N) without changing the helper's caller-visible mutation behavior.
