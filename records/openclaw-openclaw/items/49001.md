## Summary

Keep this PR open, but it is not ready to merge as-is. Current main still lacks the durable post-reset startup marker and queued-message recovery steer, while the PR has a real transient-failure clearing bug and is stale against newer steering semantics on main.

## What This Changes

The PR adds a Discord thread post-reset startup window, wraps queued owner steer prompts with startup-recovery instructions, clears the window after runs, and adds focused session/agent-runner regression tests.

## Best Possible Solution

Land a rebased, combined-safe session-layer guard that persists a short post-reset startup marker, wraps queued Discord owner steers with recovery instructions, preserves current queue steering behavior, clears the marker only after successful recovery/completion, and includes focused regression coverage for thread and channel resets or clearly scopes channel work to #53524.

## Reproduction Assessment

Yes. The code-level reproduction is a Discord thread `/new` or `/reset` followed by an owner message queued while the startup run is still streaming; the PR adds a focused unit test for that steer-wrapping path, and current main lacks the marker/wrapper being asserted.

## Solution Assessment

No, not as currently patched. The durable startup-window direction matches the reported race, but the marker should not be cleared after a failed run and the wrapper must be rebased onto current main's queueEmbeddedPiMessage options instead of reverting to the older two-argument call.
