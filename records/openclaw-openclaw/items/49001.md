## Summary

Keep open: the PR addresses a real Discord post-reset startup race, but the current branch is not ready because it clears the recovery marker on failed runs and is stale against current main's active steering call shape.

## What This Changes

The PR adds a Discord thread post-reset startup window, wraps queued owner steer prompts with startup-recovery instructions, clears the window after runs, and adds focused session and agent-runner regression tests.

## Best Possible Solution

Land one consolidated Discord reset-window implementation that covers the intended thread scope, preserves current active-steering behavior, clears only after successful recovery consumption, and either handles channel resets or deliberately coordinates that work with #53524.

## Reproduction Assessment

Yes. The source-level reproduction is a Discord thread `/new` or `/reset` followed by an owner message queued while startup preload reads are still running; current main lacks the marker/wrapper, and the PR adds a focused test for that path.

## Solution Assessment

No. The durable startup-window direction is appropriate, but this patch should be rebased onto current main's `queueEmbeddedPiMessage` call and should not clear the marker after failed runs.
