## What This Changes

The PR adds a Discord thread post-reset startup window, wraps queued owner steer prompts with startup-recovery instructions, clears the window after runs, and adds focused regression tests.

## Best Possible Solution

Land one rebased Discord reset-window implementation that preserves current steering routing/options, clears the marker only after successful recovery consumption, includes regression tests and changelog, and coordinates channel coverage with #53524.

## Reproduction Assessment

Yes. The source-level reproduction is a Discord thread `/new` or `/reset` followed by an owner message queued while startup preload reads are still running; current main lacks a durable pending-startup marker and the PR adds focused coverage for that path.

## Solution Assessment

No. The durable startup-window direction is appropriate, but this branch should be rebased onto current main's steering call, preserve retry recovery on failed runs, and resolve the overlap with #53524 before merge.
