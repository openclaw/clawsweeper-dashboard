## Summary

This PR should close as superseded: the Discord reset race is real, but this branch is stale, thread-only, and has blocking regressions against current main's steering path, while https://github.com/openclaw/openclaw/pull/53524 is the active consolidated channel-and-thread implementation.

## What This Changes

This PR adds a Discord thread-only post-reset startup window, wraps queued owner steers with startup-recovery instructions, clears the marker after runs, and adds focused session/reply tests.

## Best Possible Solution

Review and land the consolidated current-main implementation in https://github.com/openclaw/openclaw/pull/53524, then keep this older branch closed to avoid competing reset-window implementations.

## Reproduction Assessment

Yes, at source level. The path is a Discord thread /new or /reset followed by an owner message queued while startup preload reads are still running; current main has no durable pending-startup marker and queues the raw steer prompt.

## Solution Assessment

No. The durable marker direction is appropriate, but this branch is not the best merge path because it drops current steering semantics and is superseded by the broader current-main implementation in https://github.com/openclaw/openclaw/pull/53524.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve current steering routing when wrapping prompts:** `src/auto-reply/reply/agent-runner.ts:262`
  - body: The wrapped prompt is queued with followupRun.run.sessionId and no options. Current main resolves the active run id through sessionKey and forwards steeringMode/debounceMs, so this can target a stale post-reset session and ignore queue policy.
  - confidence: 0.88
- **[P2] Clear the startup window only after successful recovery:** `src/auto-reply/reply/agent-runner.ts:413`
  - body: startedRun is set before runAgentTurnWithFallback resolves, then finally clears the window for any started run. A transient provider or network failure can therefore remove the recovery marker before a successful retry consumes it.
  - confidence: 0.86
