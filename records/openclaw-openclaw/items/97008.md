Full review comments:

- **[P1] Mark no-op writes as terminal tool errors:** `src/agents/sessions/tools/write.ts:405`
  - body: This new no-op branch returns `details: undefined`, and the edit branch does the same. The runtime only classifies returned tool results as errors when structured details include a failure status, `ok: false`, `success: false`, `error`, etc.; otherwise embedded observers emit a completed tool result and may clear `lastToolError`. The linked bug asks for terminal no-op/failure semantics, so return a structured no-op status such as `invalid`/`blocked` or `ok: false` instead of a normal success payload.
  - confidence: 0.91
- **[P2] Do not drop valid edits from mixed edit batches:** `src/agents/sessions/tools/edit.ts:391`
  - body: Using `edits.some(...)` stops the whole edit call when any one entry has identical old/new text, even if other entries would change the file. The edit tool contract supports multiple disjoint edits in one call, so this can skip real requested edits while reporting `No changes`. Treat an all-no-op batch as terminal, or reject the offending entry with a structured error without silently discarding valid sibling edits.
  - confidence: 0.84
