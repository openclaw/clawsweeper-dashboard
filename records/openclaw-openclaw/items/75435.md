## Summary

Keep open for normal maintainer review and merge. Current main still directly calls `.search` on any truthy shared memory manager, while this PR adds the narrow fallback, regression coverage, changelog entry, and now has green GitHub check runs for the reviewed head SHA.

## What This Changes

The PR adds a guarded shared-memory search helper in memory-wiki, query/tool regression tests for malformed or throwing shared managers, and a Memory/wiki changelog fix entry for the linked bug.

## Best Possible Solution

Land this PR after maintainer review so the linked `wiki_search` bug closes through the PR linkage and memory-wiki treats optional shared recall failures as empty memory results while preserving local wiki results.

## Reproduction Assessment

Yes. Configure memory-wiki with `search.backend="shared"` and `corpus="all"` or `"memory"`, then have `getActiveMemorySearchManager` return a truthy manager without callable `.search` or with a rejecting `.search`; current main reaches the direct `.search` call.

## Solution Assessment

Yes. Guarding and catching at the memory-wiki consumer boundary is the narrowest maintainable fix because active memory managers may come from bundled or third-party runtimes, and the public config/tool contract does not need to change.
