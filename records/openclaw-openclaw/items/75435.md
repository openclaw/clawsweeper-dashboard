## Summary

Keep open: current main still has the unguarded shared-memory `.search` call, and this PR is a narrow, tested consumer-boundary fallback with no blocking functional or security findings.

## What This Changes

The PR adds guarded memory-wiki shared-search fallback logic, query/tool regressions for malformed or throwing shared managers, and a changelog fix note for #75312.

## Best Possible Solution

Land this PR after maintainer review so #75312 closes through the PR linkage and shared-memory unavailability becomes an empty optional corpus instead of a `wiki_search` failure.

## Reproduction Assessment

Yes. Configure memory-wiki with `search.backend="shared"` and `corpus="all"` or `"memory"`, then have `getActiveMemorySearchManager` return a truthy manager without callable `.search` or one whose `.search` rejects; current main reaches the unguarded call.

## Solution Assessment

Yes. Guarding and catching at the memory-wiki consumer boundary is the narrowest maintainable fix because the public config/tool contract stays unchanged and external memory runtimes can still return unavailable or malformed values.
