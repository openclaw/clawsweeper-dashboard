## Summary

Keep open. Current main still has the reported unguarded shared-memory search call, while this active contributor PR makes a narrow memory-wiki boundary fix with focused regression coverage and no definite review findings.

## What This Changes

The PR adds a guarded shared-memory search helper in memory-wiki, query/tool regressions for malformed or throwing shared managers, and a changelog fix entry for the linked `wiki_search` bug.

## Best Possible Solution

Land this PR after maintainer review so memory-wiki treats optional shared recall failures as empty memory results while preserving local wiki results, and let the linked issue close through the PR linkage.

## Reproduction Assessment

Yes. Configure memory-wiki with `search.backend="shared"` and call `wiki_search` with `corpus="all"` or `"memory"` while the active memory runtime returns a truthy manager without callable `.search` or a throwing `.search`; current main reaches the direct `.search` call.

## Solution Assessment

Yes. Guarding and catching at the memory-wiki consumer boundary is the narrowest maintainable fix because active memory managers can come from bundled or third-party runtimes, and the public config/tool contract does not need to change.
