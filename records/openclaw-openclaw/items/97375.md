## Summary

Keep open: the PR addresses a real config-cleanup problem, but the patch is not mergeable because it treats the custom `models.providers` map as the complete provider registry and can rewrite valid model refs into invalid synthetic refs.

## What This Changes

Adds a model-catalog helper and core doctor check that detect configured model refs whose provider is absent from `models.providers` and repair them by pruning allowlist/fallback entries or rewriting refs.

## Best Possible Solution

Keep the PR open and revise the repair to validate against runtime-known provider/model refs, only rewrite to concrete configured or default model refs, and include redacted real doctor before/after proof.

Full review comments:

- **[P1] Use the runtime provider registry for orphan checks:** `src/flows/doctor-core-checks.ts:822`
  - body: `models.providers` is only the custom provider map, so this treats valid built-in refs as orphaned. For example, a normal config can use `openai/gpt-5.5` without `models.providers.openai`, but this check would warn and `doctor --fix` would rewrite or remove that valid selection. Build the known-provider set from the same runtime/provider catalog contract used by model selection, not just custom config keys.
  - confidence: 0.94
- **[P1] Stop inventing provider/default model refs:** `packages/model-catalog-core/src/configured-model-refs.ts:179`
  - body: When every configured primary is orphaned, this synthesizes `${firstProvider}/default`, but custom providers are only valid for concrete registered model ids. That can turn a stale ref into `myproxy/default` or `ollama/default`, which current runtime will still report as an unknown model; choose an existing concrete configured/default model ref or leave/delete the stale optional field instead.
  - confidence: 0.9
