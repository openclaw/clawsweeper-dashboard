Keep open: current main and v2026.6.10 still have the ACP-bound first-turn gap, and the direct fixing pull request is open but unmerged. The issue should stay open as the canonical tracker until that PR or an equivalent fix lands.

## What This Changes

The issue reports that Telegram topic bindings configured as `type: "acp"` create ACP sessions and spawn the backend process but drop the first turn before `systemSent`, transcript creation, or a visible reply.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/97649, or an equivalent direct ACP fallback, with redacted Telegram/acpx proof that the first bound-topic turn reaches ACP and produces a visible reply.

## Reproduction Assessment

Yes for a source-reproducible path: current main only dispatches ACP-bound first turns through `reply_dispatch` hooks, and if they decline it falls through rather than calling the existing ACP runtime owner. I did not run a live Telegram/acpx/opencode reproduction in this read-only review.

## Solution Assessment

Yes: repairing the existing configured ACP binding dispatch path is the narrowest maintainable solution. The linked PR follows that direction by using the existing ACP runtime dispatch owner instead of adding a new binding type or making `type: "route"` the workaround.
