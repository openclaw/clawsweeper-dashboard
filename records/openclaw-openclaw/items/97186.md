## Summary

Keep open: Telegram forum-topic `type:"acp"` bindings are documented and wired as supported behavior, the report is against latest release v2026.6.10, and current main does not show a matching fix for first-prompt delivery into acpx/opencode.

## What This Changes

Telegram `type:"acp"` bound topic messages should deliver the first prompt into the configured ACP session instead of creating an idle ACP process with no `systemSent` or transcript.

## Best Possible Solution

Repair the existing Telegram configured ACP binding dispatch so a matched topic both initializes the ACP session and sends the inbound prompt through `reply_dispatch`/`acpManager.runTurn`, surfacing a visible error if dispatch cannot start.

## Reproduction Assessment

No live Telegram/acpx reproduction was run, but docs and current source show this is a supported path and the observed idle ACP session matches the gap between configured binding initialization and first-turn dispatch.

## Solution Assessment

Yes: the best fix is to preserve the existing `type:"acp"` binding contract and repair first-prompt dispatch, not add a new binding type or make `type:"route"` the workaround.
