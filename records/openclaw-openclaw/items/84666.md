## Summary

Keep open: the PR targets a real diagnostic recovery edge, but the current patch suppresses active-abort recovery using raw `queueDepth`, which also covers the currently processing queued turn and can leave ordinary queued work stuck. External real behavior proof is still only targeted test evidence, so the PR is not merge-ready.

## What This Changes

The PR passes diagnostic session state and queue depth into active-abort eligibility, suppresses stalled embedded-run active aborts for processing sessions with queued work, and updates the queued embedded-work regression expectation.

## Best Possible Solution

Make active-abort eligibility use real queued backlog rather than raw `queueDepth`, preserving recovery for the active queued turn while suppressing abort only when legitimate backlog is still waiting behind active work.

## Reproduction Assessment

Yes: source inspection of current main shows the existing queued terminal embedded-work test reaches active-abort recovery with `queueDepth: 1`, and the state model shows that value can include the active queued turn. I did not run tests because this was a read-only checkout review.

## Solution Assessment

No: the patch is aimed at the right recovery boundary, but raw `queueDepth > 0` is too broad for the stated backlog case. The maintainable fix is to gate on actual queued backlog, not the active queued turn.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Gate on real backlog before suppressing abort:** `src/logging/diagnostic.ts:490`
  - body: In `processing` state, `queueDepth` includes the currently active message when that turn came from the queue; current liveness code tracks this with `activeQueuedTurn` and subtracts it before counting backlog. Please pass or compute the active-turn-adjusted backlog and cover both the single active queued turn and real backlog cases.
