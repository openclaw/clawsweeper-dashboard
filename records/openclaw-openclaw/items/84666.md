## What This Changes

The PR adds session state and queue-depth inputs to diagnostic active-abort eligibility and changes the queued terminal-progress regression test to expect no recovery call.

## Best Possible Solution

Narrow the suppression to the queued terminal-progress false-positive shape, or add a later stale-progress fallback, then require redacted real setup proof before merge.

## Reproduction Assessment

Yes, at source level: current main classifies queued terminal embedded progress as stalled and the focused diagnostic test expects active-abort recovery. I did not run a live gateway reproduction in this read-only review.

## Solution Assessment

No as written: the fix is in the right scheduler path, but the guard is too broad because it suppresses non-terminal stale embedded-run recovery when queued work remains. A safer fix narrows the guard to queued terminal-progress false positives or adds an explicit stale-progress fallback.
