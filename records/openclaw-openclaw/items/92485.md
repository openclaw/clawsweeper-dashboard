## Summary

The Windows ENOENT report is credible, but this branch is not merge-ready: it resolves npm shims to `.cmd`/`.bat` paths while the supervisor still calls `spawn` without a shell, and its hardcoded extension order conflicts with the repository’s PATHEXT-aware executable resolver. The PR also lacks inspectable after-fix proof from the actual Windows supervisor path.

## What This Changes

This PR adds a Windows-only PATH walk for non-allowlisted bare commands in `resolveWindowsCommandShim` and adds filesystem-backed resolver tests.

## Best Possible Solution

Use one canonical PATH/PATHEXT-aware lookup policy and update the supervisor so a resolved batch shim is launched through the existing safe shell decision, with focused regression tests and redacted live Windows supervisor proof.

## Reproduction Assessment

Yes at source level: latest release and current main leave non-allowlisted bare Windows commands unresolved before supervisor spawn. The branch’s remaining `.cmd` launch failure is also directly traceable because the supervisor uses plain `spawn` without the shell decision used by `src/process/exec.ts`.

## Solution Assessment

No. Resolving the pathname without fixing the supervisor spawn contract cannot solve the full failure, and the new lookup duplicates established PATHEXT-aware behavior.
