## Summary

Keep this PR open: the Cron zh-CN localization is useful and the supplied screenshot is sufficient real behavior proof, but the current head still routes every `zh*` locale through Simplified Chinese relative-time strings, which regresses the supported `zh-TW` locale.

## What This Changes

The PR extracts Cron Control UI strings into i18n keys, adds zh-CN translations, syncs locale metadata, and passes the active UI locale into Cron schedule and relative-time formatting.

## Best Possible Solution

Keep the Cron i18n extraction, but make relative-time localization respect `zh-CN` and `zh-TW` separately or use locale-backed translation keys with focused regression tests.

## Reproduction Assessment

Yes. Source inspection shows the current-main Cron zh-CN hardcoded-copy gap, and PR-head source shows that selecting `zh-TW` would now flow into the broad `startsWith("zh")` Simplified Chinese relative-time branch.

## Solution Assessment

No, not yet. Extracting the strings is the right direction, but the formatter should preserve the supported locale boundary instead of treating all Chinese locales as Simplified Chinese.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve the zh-TW relative-time locale:** `src/infra/format-time/format-relative.ts:84`
  - body: This treats every locale beginning with `zh` as Simplified Chinese, and the UI wrapper now passes the active Control UI locale into this formatter. With `zh-TW` selected, relative timestamps will render Simplified strings like `刚刚` and `分钟前` even though `zh-CN` and `zh-TW` are separate supported bundles. Gate this on `zh-CN` or add a Traditional Chinese branch before merging.
