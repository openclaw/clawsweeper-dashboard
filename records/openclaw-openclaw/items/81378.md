## Summary

Keep open: the PR addresses real Cron-page i18n gaps, but it introduces a locale correctness regression for Traditional Chinese relative timestamps and still needs inspectable real behavior proof before merge.

## What This Changes

The branch extracts Cron-page UI copy and schedule/relative-time labels into Control UI i18n keys, adds zh-CN translations, regenerates locale metadata, and passes the active UI locale into relative-time formatting.

## Best Possible Solution

Keep the Cron i18n extraction, but preserve the zh-CN/zh-TW split for relative times and add focused locale coverage plus inspectable proof from a real Cron tab run.

## Reproduction Assessment

Yes, from source inspection: current main still renders several Cron-page labels and action strings directly in English while zh-CN is a supported Control UI locale. I did not run a browser reproduction in this read-only review.

## Solution Assessment

No, not as written: the i18n extraction is the right direction, but `startsWith("zh")` collapses zh-TW into Simplified Chinese relative-time output. The safer fix is to branch on zh-CN only or use locale-aware relative-time formatting with separate zh-TW output.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Limit Simplified Chinese relative times to zh-CN:** `src/infra/format-time/format-relative.ts:84`
  - body: This branch treats every locale starting with `zh` as Simplified Chinese. The Control UI supports both `zh-CN` and `zh-TW`, so Traditional Chinese users would see Simplified strings such as `分钟` and `刚刚` for relative timestamps after this change. Gate this branch on `zh-CN`, or add a separate zh-TW mapping/`Intl.RelativeTimeFormat`, before merging.
  - confidence: 0.88
