## Summary

Keep open for maintainer review: the patch is small, source-backed, and has sufficient real behavior proof, but it intentionally changes the gateway approval-runtime authentication boundary by trusting the shared exec-approvals socket token for loopback/shared-state clients.

## What This Changes

The PR changes gateway approval clients to use an existing `exec-approvals.json` socket token as the approval-runtime token when available, while retaining a process-local fallback and omitting the token for non-loopback gateway URLs.

## Best Possible Solution

Land the narrowed shared-token fix once maintainers confirm the exec-approvals socket token is the intended approval-runtime credential and the split with https://github.com/openclaw/openclaw/pull/86771 remains coherent.

## Reproduction Assessment

Yes, from source inspection: current main stores the approval-runtime token in process memory and drops it for explicit `gatewayUrl`, so a separate loopback approval client sharing only `OPENCLAW_HOME` cannot present the runtime marker. I did not run a failing current-main smoke.

## Solution Assessment

Yes, mostly: reusing the existing exec-approvals socket token keeps this out of config and preserves remote URL omission. The remaining question is maintainer acceptance of that token as the approval-runtime credential, not a mechanical code defect.
