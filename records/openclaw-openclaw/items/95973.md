## Summary

Keep open: the branch addresses the linked Telegram feedback bug, but the current helper can still show exec-approval setup text for plugin approval failures whose route is controlled by plugin forwarding, and the external-contributor proof remains mock/test-only.

## What This Changes

The PR appends native exec approval setup guidance to agent-side plugin approval no-route/timeout failures and adds focused before_tool_call coverage for Telegram-originated approvals.

## Best Possible Solution

Keep the focused agent-side feedback path, but append setup guidance only when the same native/plugin delivery gate explains the plugin approval failure, add split-route regression coverage, and require redacted real Telegram or equivalent live terminal/log proof before merge.

## Reproduction Assessment

Yes, source-reproducible. Current main returns generic plugin approval no-route/timeout text in `requestPluginToolApproval`, and https://github.com/openclaw/openclaw/issues/95800 includes v2026.6.9 timeout logs; I did not run a live Telegram gateway in this read-only review.

## Solution Assessment

No. The PR is in the right agent failure path, but using exec-surface disabled as the only condition for plugin approval failures is too broad for channels where plugin and exec forwarding are separate.
