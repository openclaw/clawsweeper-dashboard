## Summary

Keep open: the PR addresses a real Telegram/plugin approval feedback gap, but the latest head introduces a blocking approval-auth regression by making plugin approval availability depend on native exec delivery.

## What This Changes

PR #95973 appends channel-specific setup guidance to plugin approval no-route and timeout failures, adds a `describePluginApprovalSetup` channel capability hook, and updates SDK docs/tests around plugin approval availability.

## Best Possible Solution

Keep the failure-guidance path, but split plugin initiating-surface availability from same-chat approval auth availability so disabling native delivery does not revoke configured approver authorization.

## Reproduction Assessment

Yes, source inspection is enough: current main returns generic plugin no-route/timeout reasons, and the linked issue includes repeated 120-second `plugin.approval.waitDecision` timeout logs; I did not run a live Telegram repro in this read-only review.

## Solution Assessment

No, not yet; the agent-side failure path and plugin setup hook are plausible, but the current implementation overloads auth availability with initiating-surface availability and can break valid plugin approval commands.
