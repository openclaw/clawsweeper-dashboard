## Summary

Keep open: current main still routes the reported `openclaw cron list`, `add`, and `rm` paths through Gateway WebSocket RPC. The wait is bounded by the CLI timeout now, but the operator command still depends on a responsive Gateway, and the related cron reports do not cover this remaining control-plane dependency.

## What This Changes

Request that `openclaw cron list`, `add`, and `rm` avoid blocking on a degraded Gateway by using the local cron store directly where safe.

## Best Possible Solution

Add a conservative local cron-store path for inspection first, then keep mutations coordinated: prefer Gateway RPC when healthy, fall back to direct store writes only for local unreachable/timed-out Gateway or an explicit local mode, and make the running Gateway reload or re-arm from the changed store deterministically.

## Reproduction Assessment

Yes for the underlying dependency: stall or mock `callGatewayFromCli` for `cron.list`, `cron.add`, or `cron.remove`, and the CLI waits on Gateway RPC rather than reading the local store. The exact wall-clock duration from the report needs a live degraded Gateway scenario because current main also has a 30s default CLI timeout.

## Solution Assessment

Unclear for the full proposed mutation behavior. Direct local reads are the narrowest maintainable fix for `list`, but `add` and `rm` should use a guarded fallback or explicit local mode with reload/rearm handling rather than blind writes behind a live Gateway.
