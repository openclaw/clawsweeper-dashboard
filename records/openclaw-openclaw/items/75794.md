## Summary

Keep open: current main still routes the cron CLI through Gateway RPC for the reported list/add/remove surfaces, while the local cron store APIs and docs exist but are not used by those CLI commands. The default RPC path has a timeout, so the literal indefinite hang may be bounded in current code, but the degraded-gateway dependency reported here remains real.

## What This Changes

Request that `openclaw cron list`, `add`, and `rm` avoid blocking on a busy or degraded Gateway by reading or mutating the local cron store directly or falling back locally.

## Best Possible Solution

Add a conservative local cron-store fallback or fast path for operator inspection, then handle mutations only with clear coordination semantics: prefer Gateway RPC when healthy, fall back to `jobs.json`/`jobs-state.json` only when Gateway is unreachable or explicitly requested, and make the running Gateway reload/rearm behavior deterministic.

## Reproduction Assessment

Yes for the underlying dependency: the current CLI code can be reproduced by stalling or timing out `cron.*` Gateway RPC and observing `list`, `add`, and `rm` wait on `callGatewayFromCli`. The exact >60s duration needs a live degraded-Gateway scenario because current main also has a 30s default CLI timeout.

## Solution Assessment

Unclear for the full proposed mutation behavior. Direct local reads are the narrowest maintainable fix for `list`/inspection, but `add` and `rm` should use a guarded fallback or explicit local mode with reload/rearm handling rather than blindly writing behind a live Gateway.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/75794. Current `openclaw cron list/add/rm` all go through `callGatewayFromCli` and block/fail when the Gateway is busy or degraded. Implement a conservative local-store path using existing cron store/config helpers (`resolveCronStorePath`, `loadCronStore`, `saveCronStore`, normalization/create/patch helpers where appropriate). Keep Gateway RPC as the healthy/default path for behavior that needs live runtime state, but make operator inspection (`cron list`, and any `show` reuse if touched) able to return from `jobs.json`/`jobs-state.json` when Gateway RPC times out or is unreachable and no explicit remote `--url` was requested. For `add`/`rm`, either implement a clearly logged local fallback only on Gateway transport timeout/closed/unreachable with safe store writes and deterministic Gateway reload/rearm behavior, or leave mutations on RPC and document why if a safe fallback is not possible in one PR. Do not change `cron run` execution semantics. Add focused CLI tests that mock Gateway timeout/closed failures and assert the local fallback output/write behavior, plus store/service tests if mutation reload semantics change. Add a user-facing `CHANGELOG.md` fix entry crediting @JosephMatsiko and update cron CLI docs if the fallback or flag surface changes.
