## Summary

Keep open: current main still routes the reported `openclaw cron list`, `add`, and `rm` paths through Gateway RPC. The literal hang is bounded by the CLI timeout in current code, but the operator still depends on a responsive Gateway for the affected cron control-plane commands.

## What This Changes

Request `openclaw cron list/add/rm` to remain usable during Gateway degradation by reading, and carefully mutating, the local cron store instead of relying only on WebSocket Gateway RPC.

## Best Possible Solution

Add a conservative local cron-store fast path for inspection, then handle mutations with deterministic coordination: prefer Gateway RPC when healthy, fall back to direct store writes only on unreachable/timed-out local Gateway or an explicit local mode, and ensure the running Gateway reloads or re-arms from the changed store safely.

## Reproduction Assessment

Yes for the underlying dependency: stall or mock `callGatewayFromCli` for `cron.list`, `cron.add`, or `cron.remove` and the CLI waits on Gateway RPC rather than reading the local store. The exact duration from the report needs a live degraded Gateway scenario because current main also has a 30s default CLI timeout.

## Solution Assessment

Unclear for the full proposed mutation behavior. Direct local reads are the narrowest maintainable fix for `list`, but `add` and `rm` should use a guarded fallback or explicit local mode with reload/rearm handling rather than blindly writing behind a live Gateway.
