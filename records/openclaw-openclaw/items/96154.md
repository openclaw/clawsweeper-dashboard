## Summary

Keep open: the patch changes a real cron heartbeat delivery gate, but the `every()` predicate would reintroduce the previously fixed quiet-narration-plus-`HEARTBEAT_OK` spam case and conflicts with the documented `ackMaxChars` contract.

## What This Changes

The PR changes cron heartbeat suppression from skipping when any text-only payload is a heartbeat acknowledgement to skipping only when every payload is one, with focused test updates.

## Best Possible Solution

Preserve the existing `ackMaxChars` contract across the whole text-only payload batch: suppress short no-attention narration plus final `HEARTBEAT_OK`, deliver substantive content beyond the threshold, and keep non-text payloads as delivery-forcing.

## Reproduction Assessment

Yes. Source and history show the current `some()` behavior suppresses short narration plus final `HEARTBEAT_OK`, and the prior closed report documents that as expected; I did not run tests because this review is read-only.

## Solution Assessment

No. The PR is too broad because `every()` delivers short no-attention narration followed by `HEARTBEAT_OK`; a better fix would apply the existing ack threshold to the combined text-only heartbeat output.
