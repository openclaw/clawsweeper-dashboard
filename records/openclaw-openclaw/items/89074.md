## Summary

Keep open for normal PR review: the patch appears to move the active-run decision to the Gateway, which matches the queue contract, and I found no blocking code finding. The remaining merge gate is proof: the PR only reports focused tests, not real TUI + Gateway behavior for this user-visible message-delivery path.

## What This Changes

The PR relaxes TUI gateway-mode active-run submit guards so prompts are sent to `chat.send` and handled by Gateway queue policy while pending-registration and stop/abort guards remain local.

## Best Possible Solution

Land after redacted real TUI + Gateway proof shows an active-run prompt reaches Gateway queue policy while pending-submit and `/stop` behavior remain guarded.

## Reproduction Assessment

Yes, source-level: current main blocks non-local active-run TUI sends before `client.sendChat()`, while queue docs say active-run messages should be governed by Gateway queue modes. I did not run a live TUI/Gateway reproduction in this read-only review.

## Solution Assessment

Yes, this is the best code direction: the TUI should keep only connection, pending-registration, and stop/abort guards while the Gateway owns queue-mode policy. Teaching the TUI to mirror queue modes would duplicate backend policy and risk drift.
