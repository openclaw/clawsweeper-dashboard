## Summary

Keep open: current main still returns an approval-pending tool result before approved WebChat gateway exec output can reach the original foreground tool call, and this PR is a focused draft fix with tests and supplied live proof. It should proceed through owner review and final validation, not cleanup closure.

## What This Changes

The PR changes gateway exec approval handling so native WebChat approvals can be awaited inline, returning approved output or denial as the foreground exec result, with focused regression tests.

## Best Possible Solution

Land the narrow inline-wait behavior for native WebChat foreground exec approvals after undrafting, coordination with the related WebChat approval work, and focused validation on the final head.

## Reproduction Assessment

Yes, by source inspection: current main returns approval-pending from the gateway approval path before approved command output can reach the original WebChat exec tool call. A live repro would be a WebChat command such as whoami that requires native approval, but I did not run that flow in this read-only review.

## Solution Assessment

Yes. The PR uses the existing approval registration and decision path, limits inline waiting to non-direct WebChat/internal turns, returns denial as a foreground failed result, and preserves direct diagnostics and non-WebChat follow-ups on the async path.
