## Summary

Keep open: current `main` and the latest stable release still let a partially stale explicit auth order resolve to only surviving configured profiles, and the fix needs a maintainer decision because runtime append behavior would change explicit-order whitelist semantics.

## What This Changes

The issue asks OpenClaw to handle partially stale explicit `auth.order` profile lists so profile rotation is not silently narrowed after an update.

## Best Possible Solution

Add a stale-`auth.order` diagnostic and likely `doctor --fix` repair for missing ordered profile IDs, while preserving explicit-order whitelist semantics unless maintainers explicitly approve runtime append behavior.

## Reproduction Assessment

Yes, source-level: current `resolveAuthProfileOrder` filters a partially stale explicit order down to surviving configured entries, and current tests assert that behavior. I did not run tests because this review is read-only.

## Solution Assessment

Unclear as reported: appending store profiles would solve the symptom but changes explicit-order semantics. A diagnostic plus `doctor --fix` repair is the safer maintainable path unless maintainers intentionally approve runtime append behavior.
