## Summary

Keep open: current `main` still has the partial-stale explicit-order behavior, and the safest fix needs maintainer choice because `auth.order` is also documented as an explicit exclusion list.

## What This Changes

The issue requests safer handling or diagnostics when stale IDs in explicit `auth.order` collapse provider profile rotation after an update.

## Best Possible Solution

Add a stale-`auth.order` diagnostic and likely `doctor --fix` repair for missing ordered profile IDs, preserving explicit-order whitelist behavior unless maintainers deliberately approve runtime append semantics.

## Reproduction Assessment

Yes. Source inspection and existing tests show that an explicit order with one missing profile ID and one valid profile ID resolves to only the surviving configured profile on current `main` and `v2026.5.12`.

## Solution Assessment

Unclear as requested. Runtime append would fix the symptom but changes documented explicit-order filtering, so a doctor/probe/config diagnostic or repair is the safer maintainable path until maintainers choose the semantics.
