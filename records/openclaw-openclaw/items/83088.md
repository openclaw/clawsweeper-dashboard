## Summary

Keep open: current main still has the reported partial-stale explicit auth-order behavior, and existing diagnostics document omitted stored profiles but do not warn about or repair missing configured profile IDs. A fix needs maintainer choice because appending store profiles at runtime could weaken explicit auth-order exclusion semantics.

## What This Changes

The issue asks OpenClaw to handle stale IDs in explicit `auth.order[provider]` without collapsing provider profile rotation to the surviving configured profiles.

## Best Possible Solution

Add a stale-`auth.order` diagnostic and likely `doctor --fix` repair for missing ordered profile IDs, while preserving explicit-order whitelist behavior unless maintainers deliberately approve runtime append semantics.

## Reproduction Assessment

Yes. Source inspection plus the existing auth-order test show that a partially stale explicit order resolves to only the surviving configured profile; I did not run tests because this review was required to keep the checkout read-only.

## Solution Assessment

Unclear as requested. Appending remaining store profiles would fix the symptom but conflicts with documented explicit-order exclusion semantics, so diagnostics or doctor repair are the safer maintainable path unless maintainers approve a resolver behavior change.
