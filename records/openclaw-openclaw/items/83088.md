## Summary

Keep open: current `main` and the reported `v2026.5.12` release still have the partial-stale explicit-order behavior, and the existing diagnostics document excluded stored profiles but do not prevent or repair missing configured profile IDs.

## What This Changes

The issue asks OpenClaw to warn, repair, or safely merge store profiles when stale explicit `auth.order[provider]` entries shrink auth-profile rotation after an update.

## Best Possible Solution

Add a clear stale-`auth.order` diagnostic and likely `doctor --fix` repair for missing ordered profile IDs, while preserving explicit-order whitelist behavior unless maintainers explicitly approve runtime append semantics.

## Reproduction Assessment

Yes, by source inspection and existing tests: a config order containing missing IDs plus one valid profile resolves to only the valid configured profile. I did not run tests because this review was required to keep the checkout read-only.

## Solution Assessment

Unclear as written: appending remaining store profiles would fix the symptom but changes explicit-order semantics. A safer maintainable path is a doctor/config/probe warning or repair for missing IDs, with any runtime semantic change reviewed deliberately.
