## Summary

Keep open: current main and v2026.6.10 still treat configured `auth.order` as a hard constraint, and tests still assert that stale explicit entries are dropped rather than merged with remaining stored profiles. The remaining work is a maintainer product choice between preserving whitelist semantics with clearer diagnostics/doctor repair or changing runtime ordering semantics.

## What This Changes

The issue asks for stale explicit `auth.order` entries to be diagnosed, repaired, or merged with valid stored profiles so profile rotation does not collapse after updates.

## Best Possible Solution

Keep strict explicit-order semantics unless maintainers approve a runtime rule change, and add clear stale-`auth.order` diagnostics plus likely `openclaw doctor --fix` repair for missing ordered profile IDs.

## Reproduction Assessment

Yes, source-level: current resolver code and tests still encode the reported collapsed-rotation shape. I did not run a live provider-auth repro in this read-only review.

## Solution Assessment

Unclear as a runtime append fix: appending stored profiles would solve the symptom but changes explicit-order whitelist semantics. Diagnostics plus doctor repair is the safer maintainable path unless maintainers intentionally approve a new runtime rule.
