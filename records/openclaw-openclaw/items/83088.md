## Label Justifications

- `P2`: The issue is a normal-priority auth routing reliability problem with limited blast radius and a clear but policy-sensitive affected path.
- `impact:auth-provider`: The reported behavior changes which auth profiles are available for provider/model failover after stale profile IDs remain in `auth.order`.

## Summary

Keep open: current main still resolves a partially stale explicit `auth.order` to only the surviving configured profiles, and the repair needs a maintainer auth-routing decision because appending store profiles at runtime would change explicit-order whitelist semantics.

## What This Changes

The issue asks OpenClaw to handle partially stale explicit `auth.order` entries so valid stored profiles are not silently skipped during auth-profile rotation.

## Best Possible Solution

Add a clear stale-`auth.order` diagnostic and likely `doctor --fix` repair for missing ordered profile IDs, while preserving explicit-order whitelist behavior unless maintainers explicitly approve runtime append semantics.

## Reproduction Assessment

Yes. Source inspection on current main shows explicit order becomes the base order, missing IDs are filtered, and store-profile fallback only runs for the all-missing case; the existing test also expects a partially stale order to keep only the surviving configured profile.

## Solution Assessment

Unclear as submitted. Appending remaining store profiles fixes the symptom but changes explicit-order semantics, so a warning or `doctor --fix` repair is likely safer unless maintainers decide to change runtime ordering behavior.
