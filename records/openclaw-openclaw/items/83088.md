## Summary

Keep open: current `main` and the reported `v2026.5.12` release still allow a partially stale explicit auth order to resolve to only the surviving configured profiles, and the safest fix shape needs maintainer policy review.

## What This Changes

The issue requests safer handling or diagnostics for `auth.order[provider]` entries that reference missing profiles so provider profile rotation does not silently shrink after updates.

## Best Possible Solution

Add a stale-`auth.order` diagnostic and likely `doctor --fix` repair for missing ordered profile IDs, while preserving explicit-order whitelist semantics unless maintainers explicitly choose runtime append behavior.

## Reproduction Assessment

Yes, source-level: current `resolveAuthProfileOrder` filters a partially stale explicit order down to surviving configured entries, and existing tests assert that behavior. I did not run tests because this review was required to keep the checkout read-only.

## Solution Assessment

Unclear as written: appending store profiles would solve the symptom but changes explicit-order semantics. A diagnostic plus `doctor --fix` repair is the safer maintainable path unless maintainers intentionally approve runtime append behavior.
