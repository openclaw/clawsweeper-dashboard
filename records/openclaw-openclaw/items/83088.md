## Summary

Keep open: current main and v2026.6.8 still treat bare configured `auth.order` as a hard constraint, so a partially stale order can collapse rotation to the surviving configured profiles. The remaining decision is whether to preserve strict whitelist semantics with stale-order diagnostics/doctor repair or intentionally append compatible store profiles at runtime.

## What This Changes

The issue requests safer handling, validation, warning, or repair when explicit `auth.order` entries reference profiles that no longer exist in the active auth store.

## Best Possible Solution

Add stale-`auth.order` diagnostics and likely `openclaw doctor --fix` repair for missing ordered profile IDs, preserving hard whitelist semantics unless maintainers explicitly approve runtime append behavior.

## Reproduction Assessment

Yes, source-level: current resolver code and tests still encode the reported behavior. I did not run a live repro because this review is read-only.

## Solution Assessment

Unclear as a runtime append fix: appending stored profiles would solve the symptom but changes explicit-order whitelist semantics. Diagnostics plus doctor repair is the safer maintainable path unless maintainers intentionally approve a new runtime rule.
