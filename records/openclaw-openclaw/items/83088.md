## Summary

Keep open: current main and v2026.6.8 still treat configured `auth.order` as a hard constraint, so partially stale configured IDs can reduce rotation to the surviving configured profiles. The remaining work is a maintainer decision between strict whitelist semantics with diagnostics/doctor repair and runtime append/repair behavior.

## What This Changes

The issue asks OpenClaw to detect, warn about, or repair stale explicit `auth.order` profile IDs so provider profile rotation does not silently collapse after updates.

## Best Possible Solution

Make a maintainer decision on explicit-order semantics, then implement either focused doctor/config diagnostics or a tested runtime repair path that preserves operator intent.

## Reproduction Assessment

Yes at source level: `resolveAuthProfileOrder` uses configured `auth.order` as the base list and filters missing entries, so a partially stale configured order can collapse to the surviving configured profile. I did not run live tests because this review is read-only.

## Solution Assessment

No for the suggested append behavior as an automatic fix: current docs and tests preserve explicit-order exclusion semantics. The safer next step is a maintainer choice between strict whitelist semantics with doctor/config diagnostics and runtime repair.
