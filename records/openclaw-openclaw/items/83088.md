## Summary

Keep open: current `main` and the latest release still allow a partially stale explicit auth order to resolve only to surviving configured profiles, and the remaining fix requires maintainer policy choice between strict explicit-order semantics and automatic repair/append behavior.

## What This Changes

The issue asks OpenClaw to detect, repair, or warn about stale explicit `auth.order` profile IDs that silently shrink provider profile rotation.

## Best Possible Solution

Add a stale-`auth.order` diagnostic and likely `doctor --fix` repair for missing ordered profile IDs, while preserving explicit-order whitelist semantics unless maintainers explicitly approve runtime append behavior.

## Reproduction Assessment

Yes, source-level: current `resolveAuthProfileOrder` filters a partially stale explicit order down to surviving configured entries, and current tests assert that behavior. I did not run tests because this review was required to keep the checkout read-only.

## Solution Assessment

Unclear as a runtime append fix: appending store profiles would solve the symptom but changes explicit-order semantics. A diagnostic plus `doctor --fix` repair is the safer maintainable path unless maintainers intentionally approve runtime append behavior.
