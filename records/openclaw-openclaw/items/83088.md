## Summary

Keep open: current main and v2026.6.8 still treat configured auth.order as a hard constraint, so partially stale configured entries can reduce rotation to the surviving configured profile. The remaining work needs maintainer direction between preserving whitelist semantics with stronger doctor/probe diagnostics and changing runtime resolution to append compatible stored profiles.

## What This Changes

The issue requests safer handling or diagnostics when explicit auth.order entries reference stale provider profile IDs and reduce provider profile rotation.

## Best Possible Solution

Add stale-auth.order diagnostics and likely doctor repair for missing ordered profile IDs, preserving hard whitelist semantics unless maintainers explicitly approve runtime append behavior.

## Reproduction Assessment

Yes, source-level: current resolveAuthProfileOrder filters configured explicit order and current tests assert no fallback past explicit configured order. I did not run tests because this review is read-only.

## Solution Assessment

Unclear as a runtime append fix: appending store profiles would solve the symptom but changes explicit-order semantics. A diagnostic plus doctor --fix repair is the safer maintainable path unless maintainers intentionally approve runtime append behavior.
