## Summary

Keep open: current main and v2026.5.12 still have the reported source behavior, but silently appending omitted store profiles would conflict with the documented explicit auth-order exclusion contract, so this needs a maintainer decision on the diagnostic or repair path.

## What This Changes

Report that stale IDs in `auth.order[provider]` can reduce provider profile rotation to only the surviving configured profiles after an update.

## Best Possible Solution

Preserve explicit-order exclusion semantics, then add a clear doctor or `models status --probe` warning/repair path for missing stale `auth.order` IDs so users can clear or update the stale order intentionally.

## Reproduction Assessment

Yes, by source inspection. With explicit order `[old-1, profile-2, old-3]` and store profiles `[profile-1, profile-2, profile-3, profile-4]`, the current resolver filters the explicit order and returns only `profile-2`.

## Solution Assessment

No, silently appending all store profiles is not the best default because current docs define explicit `auth.order` as an exclusion boundary. A safer solution is diagnostic visibility or an explicit doctor repair that preserves operator intent.
