## Summary

Keep open: current main and the reported v2026.5.12 release still filter a partially stale explicit auth order down to the surviving configured profiles, and the remaining fix needs a maintainer policy choice between diagnostic repair and changing runtime selection semantics.

## What This Changes

The issue asks OpenClaw to detect or repair stale `auth.order` profile IDs so provider profile rotation does not collapse after stored profile IDs change.

## Best Possible Solution

Add a stale-`auth.order` diagnostic and likely `doctor --fix` repair for missing ordered profile IDs, while preserving explicit-order whitelist semantics unless maintainers explicitly approve runtime append behavior.

## Reproduction Assessment

Yes, source-level: current `resolveAuthProfileOrder` filters a partially stale explicit order down to surviving configured entries, and current tests assert that behavior. I did not run tests because this review was required to keep the checkout read-only.

## Solution Assessment

Unclear as filed: appending store profiles would solve the symptom but changes explicit-order semantics. A diagnostic plus `doctor --fix` repair is the safer maintainable path unless maintainers intentionally approve runtime append behavior.
