## Summary

Keep open: current main and v2026.6.8 still treat explicit config `auth.order` as a hard profile constraint, so partially stale entries can reduce rotation to the surviving configured profiles; the remaining choice is whether maintainers want strict-order diagnostics/doctor repair or a runtime append rule.

## What This Changes

The issue asks OpenClaw to handle stale `auth.order[provider]` entries so auth-profile rotation is not silently reduced after profile-store changes.

## Best Possible Solution

Keep strict explicit-order semantics unless maintainers choose otherwise, and add clear stale-`auth.order` diagnostics plus likely `openclaw doctor --fix` repair for missing ordered profile IDs.

## Reproduction Assessment

Yes, source-level: current resolver code and tests still encode the reported behavior. I did not run a live repro because this review is read-only.

## Solution Assessment

Unclear as a runtime append fix: appending stored profiles would solve the symptom but changes explicit-order whitelist semantics. Diagnostics plus doctor repair is the safer maintainable path unless maintainers intentionally approve a new runtime rule.
