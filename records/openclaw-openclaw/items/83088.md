## Summary

Keep open: current `main` still resolves a partially stale explicit auth order to only the surviving configured profile, and existing diagnostics explain omitted stored profiles but do not warn about missing configured IDs. The fix needs a behavior decision because runtime appending store profiles could weaken explicit-order whitelist semantics.

## What This Changes

Detect, warn about, or repair stale `auth.order[provider]` entries so profile rotation does not silently collapse after stored profiles change.

## Best Possible Solution

Add a stale-`auth.order` diagnostic and likely `doctor --fix` repair for missing ordered profile IDs, while preserving explicit-order whitelist behavior unless maintainers deliberately approve runtime append semantics.

## Reproduction Assessment

Yes. Source inspection plus existing tests show that an explicit order with missing IDs and one valid ID resolves to only the valid configured ID; I did not run tests because this review was required to keep the checkout read-only.

## Solution Assessment

Unclear as written. Appending store profiles would fix the symptom, but a warning or doctor repair is safer unless maintainers explicitly choose to relax explicit-order whitelist semantics.
