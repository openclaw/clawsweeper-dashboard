## Summary

Keep open: the Logs localization is still useful on current main, but this PR is not ready because it updates only English and pt-BR instead of the generated locale/.i18n surface and still lacks after-fix real behavior proof.

## What This Changes

The branch adds Logs translation keys in English and pt-BR and rewires the Logs view through `t(...)` while preserving the stable export filename suffix.

## Best Possible Solution

Refresh the PR against current main, localize the Logs labels through the existing Control UI i18n pipeline, keep file-safe export suffixes as `filtered`/`visible`, and include redacted after-fix UI proof.

## Reproduction Assessment

Yes. Source inspection on current main shows `ui/src/ui/views/logs.ts` still hardcodes the Logs title, filter, export, file, truncation, and empty-state text, so selecting a non-English locale leaves mixed-language UI.

## Solution Assessment

No. The view wiring and export suffix split are the right direction, but the branch needs to be refreshed through the Control UI i18n sync pipeline so all shipped locale bundles and `.i18n` metadata match English.
