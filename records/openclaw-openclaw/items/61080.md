## Summary

Keep open. The Logs localization is still useful because current main still renders several Logs-screen labels directly in English, but this PR is not merge-ready against the current Control UI i18n pipeline until the generated locale outputs and release note are refreshed.

## What This Changes

The PR adds Logs-specific locale keys to English and pt-BR and updates the Control UI Logs view to render title, subtitle, loading/export/filter/file/truncation/empty-state copy through `t(...)` while keeping export filename suffixes stable.

## Best Possible Solution

Refresh the Logs localization against current main by keeping the view-only localization boundary, preserving the `filtered`/`visible` filename suffix contract, regenerating every shipped locale and `.i18n` artifact through the Control UI sync pipeline, and adding the required user-facing changelog entry.

## Reproduction Assessment

Yes. Source inspection on current main is enough: the Logs view still contains the hardcoded English labels, and selecting a non-English locale would leave those strings untranslated.

## Solution Assessment

No. The implementation direction and export-suffix split are right, but the PR is incomplete for the current generated-locale workflow because it hand-updates only English and pt-BR and omits generated locale metadata and changelog coverage.
