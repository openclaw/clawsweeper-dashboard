## Summary

Current main already implements the ACP cleanup this PR was opened for via the `consumeAcpTurnStream` helper, and that implementation shipped in `v2026.5.2`; the PR branch is now obsolete and its ACP hunk would regress failure bookkeeping order.

## What This Changes

The PR refactors ACP turn error handling and also changes OAuth auth-profile annotations from `OAuthProvider` to `OAuthProviderId`.

## Best Possible Solution

Keep the shipped ACP helper split and, if the OAuth deprecation cleanup is still desired, handle it as a focused type-only PR without touching ACP turn failure control flow.

## Reproduction Assessment

Yes. For the PR finding, source inspection gives a high-confidence path: a turn failure followed by a rejection from post-turn cleanup exits before the PR's post-finally `turnError` block can persist terminal/error state; current main no longer has the local caught-throw shape.

## Solution Assessment

No. The PR is not the best solution now because current main already avoids the warning with a helper split while preserving catch-time failure bookkeeping; the OAuth type cleanup should be split if maintainers want it.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Persist turn failure state before cleanup:** `src/acp/control-plane/manager.core.ts:938-960`
  - body: The new `turnError` block runs only after `finally`. If post-turn cleanup rejects, `runTurn` exits before `recordTurnCompletion`, background-task terminal marking, and `setSessionState(..., state: "error")`, masking the ACP failure and leaving the failed turn non-terminal. Keep failure bookkeeping in the catch path or make cleanup unable to preempt it before moving this block.
  - confidence: 0.9
