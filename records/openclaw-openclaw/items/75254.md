## Summary

Keep this PR open, but do not merge it as-is. The terminal backstop direction is useful, yet the proposed fixed 30-second WebChat fallback conflicts with current `chat.send` timeout semantics for omitted `timeoutMs` and explicit `timeoutMs: 0`, and the user-facing fix still needs a changelog entry.

## What This Changes

The PR adds a WebChat-only `chat.send` terminal-event timeout backstop, initializes missing transcript files before eager user transcript updates, and adds focused gateway regression coverage.

## Best Possible Solution

Keep the terminal backstop and transcript initialization, but derive the WebChat terminal deadline from the same resolved `chat.send` timeout semantics as the registered abort controller, preserve explicit `timeoutMs: 0` as no-timeout, cover omitted/zero/custom timeout cases, and add the required single-line changelog fix entry before merge.

## Reproduction Assessment

Yes. A focused gateway test can simulate a WebChat `chat.send` dispatch that starts but never emits a final event; omitted `timeoutMs` and `timeoutMs: 0` should follow `resolveAgentTimeoutMs`, while the PR's fixed fallback would abort those paths after 30 seconds.

## Solution Assessment

No. The transcript-file guard and terminal backstop are reasonable, but the fixed fallback is not the narrowest maintainable solution because it bypasses the existing gateway timeout resolver used by `chat.send`.
