## Summary

Keep open: this PR targets a plausible WebChat `chat.send` terminal-timeout bug, but the current patch adds a WebChat-only 30s fallback that conflicts with the existing `chat.send` timeout contract and lacks real behavior proof.

## What This Changes

The PR adds a WebChat `chat.send` terminal-event timeout backstop, creates missing transcript files before eager user transcript updates, and adds gateway regression coverage.

## Best Possible Solution

Keep the terminal-event backstop, but derive it from the existing `chat.send` timeout contract, preserve explicit zero as no-timeout, and prove the behavior with focused gateway coverage plus real WebChat or gateway runtime evidence.

## Reproduction Assessment

Yes. A source-level reproduction is clear: WebChat `chat.send` with omitted `timeoutMs` or explicit `timeoutMs: 0` would schedule the PR's 30s terminal timer even though current main resolves omitted values through config/defaults and treats zero as no-timeout.

## Solution Assessment

No. The transcript-file guard and terminal backstop are reasonable, but the timeout backstop should reuse the existing resolved timeout semantics instead of adding a WebChat-only 30s policy.
