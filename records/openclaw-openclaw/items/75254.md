## Summary

Keep this PR open: it targets a real WebChat/gateway terminal-event gap that is not implemented on current main, but the proposed timeout resolver changes the existing `chat.send` timeout contract and still lacks real behavior proof.

## What This Changes

The PR adds a WebChat `chat.send` terminal-event timeout backstop, creates missing transcript files before eager user transcript updates, and adds focused gateway regression coverage.

## Best Possible Solution

Keep the terminal-event backstop idea, but derive it from the existing resolved `chat.send` timeout contract, preserve explicit zero as no-timeout, and prove it with focused gateway coverage plus real WebChat or gateway runtime evidence.

## Reproduction Assessment

Yes, source-level. The Control UI omits `timeoutMs`, current main treats omitted values through defaults and explicit zero as no-timeout, and the PR resolver would cap those WebChat sends at 30 seconds.

## Solution Assessment

No. The transcript-file guard and terminal backstop are reasonable, but the timeout backstop should reuse the existing resolved timeout semantics instead of adding a WebChat-only 30-second policy.
