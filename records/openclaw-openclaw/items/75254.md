## Summary

Keep open for review rather than cleanup-close: the WebChat timeout problem is plausible and related recovery work remains open, but this PR is not merge-ready because it would cap omitted-timeout WebChat sends at 30s, omit selected-agent routing on its new timeout error, and lacks real behavior proof.

## What This Changes

The branch adds a WebChat-only `chat.send` terminal timeout backstop in the gateway, creates missing transcript files before user transcript updates, and adds a focused gateway regression test.

## Best Possible Solution

Preserve the configured/requested agent timeout for any terminal backstop, pass the selected agent through timeout broadcasts, rebase the branch, and provide redacted real WebChat/Gateway proof; otherwise follow the narrower active recovery and ACK-status PRs.

## Reproduction Assessment

Yes for the patch defects via source inspection: current Control UI sends `chat.send` without `timeoutMs` and current agent timeout defaults to 48 hours, while the PR caps omitted WebChat sends at 30s. I did not run a live WebChat timeout scenario in this read-only review.

## Solution Assessment

No. A server terminal backstop may be a plausible mitigation, but hardcoding a shorter WebChat default and omitting selected-agent routing is not the best fix; the safer path reuses existing timeout semantics and coordinates with the active UI watchdog and terminal ACK work.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Preserve the configured timeout for WebChat sends:** `src/gateway/server-methods/chat.ts:1657-1660`
  - body: For WebChat clients that omit `timeoutMs`, current Control UI sends no timeout value and `resolveAgentTimeoutMs` falls back to the configured agent timeout, which defaults to 48 hours. This `Math.min(..., 30_000)` changes those ordinary sends into a 30s terminal error, so long-running WebChat tasks will be aborted even though existing config allows them to continue. Use the resolved agent/request timeout for the backstop instead of adding a shorter WebChat default.
  - confidence: 0.94
- **[P2] Route timeout errors to the selected agent:** `src/gateway/server-methods/chat.ts:2167-2173`
  - body: The new timeout branch calls `broadcastChatError` without `agentId`, while the existing final and error paths pass `agentId` so selected global-agent sessions route to `agent:<id>:global` instead of the default global session. Include the resolved `agentId` here or timeout errors for non-default global WebChat sessions can be delivered to the wrong session target.
  - confidence: 0.86
