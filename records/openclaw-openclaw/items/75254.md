## Summary

Keep open because the bug-fix direction is plausible, but the current patch regresses the existing `chat.send` timeout contract and lacks after-fix real behavior proof.

## What This Changes

The PR adds a WebChat `chat.send` terminal-event timeout backstop, initializes missing transcript files before eager user transcript updates, and adds gateway regression coverage.

## Best Possible Solution

Keep the terminal backstop and transcript-file guard only if the timeout derives from the existing resolved `chat.send` contract, explicit zero remains no-timeout, and a real WebChat or Gateway run proves the terminal error behavior.

## Reproduction Assessment

Yes. A source-level reproduction is clear: on the PR head, WebChat `chat.send` with omitted `timeoutMs` or `timeoutMs: 0` reaches the new 30s fallback, while current main resolves omitted values through config/defaults and treats zero as no-timeout.

## Solution Assessment

No. The terminal backstop direction is reasonable, but the timeout backstop should reuse the existing resolved timeout semantics instead of adding a WebChat-only 30s cap.
