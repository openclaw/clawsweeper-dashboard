## Summary

Keep open. The PR addresses the earlier trust concern and handles prompt/routed reply paths, but normal Telegram same-channel silent-reply planning still lacks the new trusted-thread context, so a threaded Telegram direct/topic turn can still rewrite `NO_REPLY` into filler text.

## What This Changes

The PR adds a trusted structured `:thread:` session-key predicate, threads that policy context through several auto-reply/outbound silent-reply call sites, updates outbound session context, adds regression tests, and records a changelog entry.

## Best Possible Solution

Complete this narrow PR by propagating trusted thread context into all same-channel Telegram silent-reply planning paths, including fallback synthesis and `deliverReplies` normalization, with regression coverage for trusted direct threads, trusted group/channel threads, and malformed or mismatched suffixes.

## Reproduction Assessment

Yes. Current main can be reproduced by classifying `agent:main:telegram:direct:435427284:thread:435427284:300118` as direct and planning a `NO_REPLY` payload, which takes the default direct rewrite path; the latest PR still leaves the Telegram same-channel planner without the new trust signal.

## Solution Assessment

No. The trusted-thread predicate is the right direction, but the PR is not yet the best complete fix because it only reaches some auto-reply paths and misses Telegram delivery paths that still run the shared silent-reply planner from session key alone.
