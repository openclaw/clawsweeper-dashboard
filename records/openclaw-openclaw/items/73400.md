## Summary

Keep open. The PR targets a real silent-reply bug and its latest commits address the earlier Telegram fallback and delivery-normalization gaps, but the patch still misses the pre-delivery final-reply dispatcher path for Telegram DM-topic sessions, so exact final `NO_REPLY` can still be rewritten to filler text before Telegram delivery sees the trusted thread context.

## What This Changes

The PR adds trusted structured thread-session classification for silent-reply policy, propagates that trust through auto-reply, outbound, and Telegram delivery paths, adds regression tests, and records a changelog entry.

## Best Possible Solution

Complete this PR by carrying Telegram-verified thread trust into the dispatcher and prompt silent-reply context before any exact final `NO_REPLY` rewrite can happen, while preserving the strict malformed or mismatched suffix behavior.

## Reproduction Assessment

Yes. A source-level reproduction is a Telegram DM-topic turn with `SessionKey` like `agent:main:telegram:direct:123:thread:123:42`, `MessageThreadId` `42`, and an exact final `NO_REPLY`; at PR head the generic dispatcher trust check remains false and the direct rewrite path runs before Telegram delivery.

## Solution Assessment

No. The trusted structured-thread model and the new delivery/fallback propagation are the right direction, but the implementation is incomplete for Telegram chat-scoped thread suffixes; passing the Telegram-verified trust into the dispatcher context is safer than returning to broad substring classification.
