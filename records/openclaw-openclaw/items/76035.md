## Summary

Keep open. Current main still reproduces the reported hook-content gap, and open PR #76036 is already linked with closing syntax as the implementation candidate, so this issue should stay open until that PR lands or is replaced.

## What This Changes

Slack bot alert messages with blank command text should still expose attachment-derived alert text as plugin and internal message-received hook content.

## Best Possible Solution

Review and land or rework #76036 so shared hook content prefers nonblank command text, then falls back to raw/body content, with mapper and Slack regression coverage.

## Reproduction Assessment

Yes. The source-level path is high confidence: Slack bot attachment text reaches `RawBody`, blank top-level Slack text leaves `BodyForCommands` empty, and the current mapper returns that empty string as hook content.

## Solution Assessment

Yes. The linked PR's described direction is the narrowest maintainable fix because the bug is in the shared hook-content precedence rule, with Slack coverage proving the channel-specific input shape.
