## Label Justifications

- `P2`: This is a clear Twitch plugin message-delivery bug with limited blast radius and a narrow fix path.
- `impact:message-loss`: A stale cleanup can remove the active inbound Twitch handler, causing subsequent chat messages for that account to be dropped.

## Summary

Keep open: current `main` still removes the Twitch account handler by key without checking that the cleanup belongs to the handler being removed, so the reported stale-cleanup message-loss path remains valid.

## What This Changes

The issue asks for `TwitchClientManager.onMessage` cleanup to remove only the handler registered by that call, so an older cleanup cannot delete a newer active Twitch handler.

## Best Possible Solution

Keep the one-handler-per-account model, but make each cleanup idempotently remove only its own handler and add a regression test for stale cleanup after replacement.

## Reproduction Assessment

Yes, from source inspection. Registering handler B for the same account key after handler A, then calling A's cleanup, deletes the shared key and removes B because cleanup does not compare handler identity.

## Solution Assessment

Yes. The narrow maintainable fix is a referential guard in the cleanup closure, plus regression coverage that proves stale cleanup does not remove the newer handler.

Fix the Twitch handler cleanup bug reported at https://github.com/openclaw/openclaw/issues/83888. In `extensions/twitch/src/twitch-client.ts`, update `TwitchClientManager.onMessage` so the returned cleanup removes the account key only when `messageHandlers.get(key) === handler`; do not change the one-active-handler-per-account behavior. Add a focused regression in `extensions/twitch/src/twitch-client.test.ts` that registers handler A, registers handler B for the same account, calls A's cleanup, and verifies B remains active and receives a simulated inbound message.
