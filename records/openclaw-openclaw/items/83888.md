## Summary

Keep open: current `main` still deletes the Twitch handler map entry unconditionally, and the linked fixing PR is open rather than merged, so this issue should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue asks for Twitch `onMessage` cleanup to remove only the handler registered by that cleanup, not a newer handler for the same account.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/84195, or an equivalent identity-guard fix with regression tests, then close this issue as fixed.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: register handler A, register handler B for the same account key, then call cleanup A and current `main` deletes B from the handler map.

## Solution Assessment

Yes. A referential guard in the cleanup closure is the narrow maintainable fix because it preserves single-handler cleanup while preventing stale cleanup from deleting a newer registration.
