## Summary

Keep open. Current main supports explicit Slack Block Kit payloads, shared presentation rendering, and interactive controls, but the broader Slack rich-output request still has unresolved product/API scope around automatic table/rich_text rendering and whether provider-native Slack blocks should be a blessed producer surface.

## What This Changes

Allow Slack agents to compose and send richer Block Kit messages from agent replies, including structured sections, buttons/selects, context, and table or rich formatting beyond plain text.

## Best Possible Solution

Maintainers should choose the Slack plugin rich-output boundary, keeping shared `presentation` canonical while either blessing provider-native Slack blocks with fallback rules or designing a portable report/rich-text extension.

## Reproduction Assessment

Yes at the source level: explicit Slack blocks work, but ordinary Markdown replies still resolve table mode without block-renderer capability and flow through text chunk rendering. No live Slack failure was reproduced because this is a feature/product-scope gap, not a broken current contract.

## Solution Assessment

No, the original broad tool/markup/hybrid proposal is not yet the best single implementation shape. The maintainable path is a maintainer-approved Slack plugin rendering or producer API boundary, with portable `presentation` kept as the shared default.
