## Summary

Keep open: current main now supports explicit Slack Block Kit payloads and interactive/presentation controls, but it still does not satisfy the broader rich-output umbrella because ordinary markdown/table replies are rendered through text chunks and `markdown.tables: "block"` is still coerced back to code. The remaining table/rich_text direction is product-scoped and overlaps the open RFC at https://github.com/openclaw/openclaw/issues/57708 rather than being cleanup-closeable.

## What This Changes

Request to let Slack agent messages use richer Block Kit output, including authored blocks, tables, buttons/selects, context, and higher-level rendering beyond plain text.

## Best Possible Solution

Keep this umbrella open until maintainers either land the Slack table/rich-output send path or explicitly move the remaining markdown-to-rich_text direction to https://github.com/openclaw/openclaw/issues/57708, while keeping button-click and modal lifecycle gaps on their dedicated issues.

## Reproduction Assessment

Yes at source level, though this is a feature gap rather than a bug reproduction: current main's Slack send and slash reply paths still render ordinary markdown through `markdownToSlackMrkdwnChunks`, and table `block` mode is coerced back to `code`. I did not run a live Slack workspace in this read-only review.

## Solution Assessment

No, closing this as implemented is not the best solution: current main implements explicit raw blocks and interactive/presentation controls, but the ordinary markdown/table-rich output path remains unresolved product-scoped work. The maintainable next step is a Slack-plugin-owned table/rich_text design decision rather than a new broad automated repair PR.
