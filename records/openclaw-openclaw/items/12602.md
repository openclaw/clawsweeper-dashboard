## Summary

Keep open: current main and v2026.5.22 support explicit Slack Block Kit payloads, presentation controls, and interactive reply blocks, but ordinary Slack replies still render through the mrkdwn/text path and native table/rich_text defaulting remains an unresolved product decision tracked by narrower related work.

## What This Changes

The issue requests richer Slack agent messages using Slack Block Kit, including authored block payloads, tables, buttons, selects, and metadata blocks alongside plain-text fallback.

## Best Possible Solution

Keep this as the Slack rich-output umbrella until maintainers either land or explicitly defer native table/rich_text conversion in https://github.com/openclaw/openclaw/issues/57708, while keeping Slack interaction wake behavior separate under https://github.com/openclaw/openclaw/issues/61502.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature umbrella. Source inspection gives a high-confidence current boundary: explicit blocks and presentation controls work, while ordinary Slack replies still render through mrkdwn/text chunks and block table mode is coerced to code.

## Solution Assessment

No, closing or auto-implementing this now is not the best path. The maintainable route is maintainer product review for native table/rich_text defaults, streaming/chunking behavior, fallback text, and explicit-block precedence in the narrower related work.
