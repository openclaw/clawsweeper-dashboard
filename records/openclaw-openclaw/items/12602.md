## Summary

Keep open: current main and the latest release support explicit Slack Block Kit payloads, shared presentation buttons/selects, and interactive delivery, but ordinary Slack markdown/table-rich output is still routed through the text/mrkdwn path and the narrower rich_text RFC remains open.

## What This Changes

The issue requests richer Slack agent messages using Block Kit structures such as sections, fields, dividers, buttons, selects, context metadata, and table-like output.

## Best Possible Solution

Keep this umbrella open for maintainer follow-up while resolving the remaining automatic Slack rich_text/table conversion direction in the Slack plugin, preferably through the narrower open RFC or a focused replacement PR with live Slack proof.

## Reproduction Assessment

Not applicable as a bug repro, but source inspection gives a high-confidence remaining gap: `markdown.tables: "block"` is coerced to `code`, and non-block Slack replies still render through text chunks when no explicit blocks are provided.

## Solution Assessment

No, closing as implemented would be premature. The direct Block Kit slice is shipped, but the best remaining path is a maintainer-approved Slack plugin design for native table/rich_text conversion, likely coordinated with the open rich_text RFC.
