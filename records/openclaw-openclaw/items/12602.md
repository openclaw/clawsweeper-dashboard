## Summary

Keep open: current main has shipped explicit Slack Block Kit payload support and interactive controls, but the broader rich-output umbrella remains unresolved because ordinary Slack markdown replies still render through text chunks and the current table-mode contract still falls back to code unless a block-aware renderer is added.

## What This Changes

Request Slack agent messages to support richer Block Kit output such as structured sections, fields, buttons, selects, context blocks, and native table rendering instead of only plain text or markdown.

## Best Possible Solution

Keep this as the canonical Slack rich-output umbrella, preserve the shipped explicit blocks and typed presentation-action paths, and narrow any remaining automatic renderer to a Slack-owned contract with deterministic text/code fallback.

## Reproduction Assessment

Yes, at the source level: current main validates explicit Slack blocks but ordinary markdown replies still flow through text chunk rendering, and the current table-mode resolver/test suite coerces unsupported block mode to code. No live Slack failure was reproduced because this is a feature gap rather than a broken existing contract.

## Solution Assessment

No, the original broad alternatives are not the best single implementation shape. The maintainable solution is to keep explicit Slack payload support and typed presentation actions, then separately approve any automatic table or rich_text renderer with Slack-specific limits and fallback behavior.
