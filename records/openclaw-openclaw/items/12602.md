## Summary

Keep open: explicit Slack Block Kit payloads and presentation controls are supported, but the broader rich-output request is still incomplete because ordinary Slack agent replies remain on the mrkdwn/text path, `markdown.tables: "block"` is still coerced to code rendering, and the narrower rich_text RFC at https://github.com/openclaw/openclaw/issues/57708 remains open.

## What This Changes

Request richer Slack agent messages via Block Kit output, including authored blocks, tables, buttons/selects, context, and automatic rich rendering alongside text fallback.

## Best Possible Solution

Keep this as the Slack rich-output umbrella until maintainers either land a narrow table-send replacement or explicitly defer tables, while https://github.com/openclaw/openclaw/issues/57708 carries the richer rich_text/defaulting decision.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature umbrella. Source inspection gives a high-confidence current boundary: explicit blocks work, while ordinary replies still render as mrkdwn/text and block table mode is coerced to code.

## Solution Assessment

No, closing this as fully implemented is not the best path. The maintainable route is to keep the umbrella open, let https://github.com/openclaw/openclaw/issues/57708 settle rich_text/defaulting semantics, and separately decide whether a narrow Slack table-send replacement should land.
