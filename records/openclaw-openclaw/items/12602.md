## Summary

Keep open: current main supports explicit Slack Block Kit payloads, message-tool blocks, and opt-in interactive directives, but ordinary Slack markdown replies still render through mrkdwn/text chunks, `markdown.tables: "block"` is still coerced back to code, and the remaining table/rich-output work is active in focused follow-ups.

## What This Changes

The issue requests richer Slack agent responses using Block Kit blocks, including structured sections, tables, context blocks, buttons, selects, and possible tool or markup formats for agents to author those payloads.

## Best Possible Solution

Keep this umbrella open until the table send path lands or is intentionally superseded, then let the focused Slack rich_text RFC carry the remaining automatic rich-output decision with docs, regression coverage, and real Slack proof.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature umbrella. Current-main source inspection gives a high-confidence verification path: explicit blocks and interactive directives work, while native table delivery and automatic rich_text conversion are not complete on main.

## Solution Assessment

No, closing this as fully implemented is not the best current action. The maintainable path is to review, rebase, or replace https://github.com/openclaw/openclaw/pull/57717 for table delivery and resolve https://github.com/openclaw/openclaw/issues/57708 for automatic rich output.
