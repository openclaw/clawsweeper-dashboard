## Summary

The requested umbrella is still necessary, but it is now narrower: current main supports validated authored Block Kit payloads and interactive presentation controls, while ordinary Slack markdown replies still use text rendering and explicitly downgrade `markdown.tables: "block"` to code. The remaining automatic rendering scope needs maintainer product judgment rather than another broad repair PR.

## What This Changes

Allow agents to compose richer Slack messages with Block Kit layouts, interactive controls, and native structured rendering instead of plain markdown-only responses.

## Best Possible Solution

Keep the shipped validated `blocks` and typed presentation-action paths, then narrow the remaining work to a Slack-owned renderer for explicitly selected structures such as markdown tables, with deterministic text/code fallback and no model-facing transport-string inference.

## Reproduction Assessment

Yes, at the source level: current main deterministically coerces Slack block table mode to code, and a focused test asserts that behavior. No runtime failure is alleged because this is a feature gap rather than a broken existing contract.

## Solution Assessment

No, the original broad alternatives should not be implemented as one generic markup feature. The narrow maintainable path is typed portable actions plus validated Slack-specific payloads, with separately approved automatic renderers for selected content.
