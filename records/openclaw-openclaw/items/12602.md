## Summary

Keep open. Current main implements explicit Slack Block Kit payload delivery, shared presentation blocks, and opt-in interactive controls, but it still does not complete the ordinary markdown-table/rich-output path raised in the discussion: `markdown.tables: "block"` is coerced back to `code`, and normal Slack replies without explicit blocks still go through mrkdwn text chunking. The remaining work is already clustered around #57717 and #57708, so this needs maintainer follow-up rather than cleanup closure.

## What This Changes

Request to let Slack agent messages use Block Kit for richer responses, including structured sections, dividers, buttons, select menus, context metadata, and better table rendering.

## Best Possible Solution

Keep this as the Slack rich-output umbrella until maintainers land or explicitly replace the native markdown-table Block Kit send path, then track broader automatic Markdown-to-rich_text conversion under #57708 with conservative text/code fallbacks and Slack docs updates.

## Reproduction Assessment

Yes. Configure `channels.slack.markdown.tables: "block"` and send an ordinary Slack markdown table reply without explicit blocks; current source and tests show the resolver returns `code`, and the send path posts mrkdwn text chunks instead of native Block Kit table blocks.

## Solution Assessment

No, closing this as implemented is not the best current action. Raw Block Kit and shared presentation support are implemented, but the remaining native table/rich-output work should be resolved by reviewing #57717 and scoping #57708 rather than creating duplicate cleanup work.
