## Summary

Keep open. Current main covers explicit Slack Block Kit payloads, shared presentation rendering, and interactive controls, but the remaining automatic table/rich_text rendering and producer-facing API boundary still need maintainer product direction.

## What This Changes

The issue requests richer Slack Block Kit support for general Slack agent messages beyond plain text, including structured sections, fields, buttons, selects, context blocks, and better report/table rendering.

## Best Possible Solution

Maintainers should choose the Slack plugin rich-output boundary, keep shared presentation as the portable default, and then either bless provider-native Slack block producers with fallback rules or design a portable report/rich-text extension.

## Reproduction Assessment

Yes at the source level: explicit Slack blocks work, but ordinary Markdown replies still route through text chunking and block table mode is capability-gated back to code. No live Slack failure was reproduced because the remaining gap is feature/product scope, not a broken current contract.

## Solution Assessment

No, the original broad tool/markup/hybrid proposal is not yet the best single implementation shape. The maintainable path is a maintainer-approved Slack plugin rendering or producer API boundary, with shared presentation kept as the portable default.
