## Summary

Keep open. This is a valid gateway bug fix on the right surface, but the current PR still passes through malformed or unknown typed content blocks and lacks after-fix real HTTP MCP behavior proof, so it is not merge-ready.

## What This Changes

This PR broadens the MCP HTTP tool-result normalizer to preserve text, image, audio, embedded resource, and resource_link blocks, with helper-level tests for those shapes and fallback cases.

## Best Possible Solution

Update the normalizer to preserve only SDK-valid MCP content block variants, fall back for unknown or malformed entries, add HTTP tools/call regression coverage, and include redacted real-run proof before merge.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: a loopback tool returning image/audio/resource content through HTTP MCP tools/call reaches normalizeToolCallContent, which current main collapses to type/text only; I did not execute a live request because this review is read-only.

## Solution Assessment

No. The PR targets the right boundary, but the current implementation should validate known SDK content-block shapes and required fields before pass-through, and it needs endpoint-level regression coverage plus real behavior proof.
