## Summary

Keep open: this PR targets a real current HTTP MCP gateway data-loss bug, but the submitted patch is not merge-ready because it passes through malformed or unknown typed blocks, the branch is currently conflicting, and the external PR still lacks after-fix real behavior proof.

## What This Changes

The PR widens MCP HTTP tool-result normalization from text-only output to text, image, audio, resource, and resource_link blocks and adds helper-level regression tests for those shapes.

## Best Possible Solution

Revise this PR or a narrow replacement so the gateway preserves only SDK-valid MCP content blocks, falls back to text for malformed or unknown entries, covers the HTTP tools/call path, rebases cleanly, and includes redacted real MCP behavior proof before merge.

## Reproduction Assessment

Yes. Source inspection shows current main returns `normalizeToolCallContent(result)` for HTTP MCP tools/call, and that helper drops all non-type/text fields from array content blocks; a live HTTP MCP client was not run because this review is read-only.

## Solution Assessment

No, not as submitted. Preserving valid MCP content at the gateway boundary is the right fix location, but the implementation should validate known SDK block shapes and required fields before passthrough rather than casting any object with a string type.
