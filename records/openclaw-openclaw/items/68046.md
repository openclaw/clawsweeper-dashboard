## Summary

Keep open. Current main still has the lossy MCP HTTP content normalization, and this focused PR addresses the right bug, but the patch is not merge-ready because it passes through malformed or unknown content blocks and lacks inspectable real behavior proof.

## What This Changes

The PR changes the MCP HTTP tools/call content normalizer to pass through text, image, audio, resource, and resource_link blocks and adds helper-level tests for those variants.

## Best Possible Solution

Revise this PR or a narrow replacement so the gateway preserves only SDK-valid MCP content blocks, falls back to text for malformed or unknown entries, adds HTTP tools/call regression coverage, and includes redacted runtime proof before merge.

## Reproduction Assessment

Yes. A high-confidence source path exists: a loopback tool returning content with an image block reaches normalizeToolCallContent through HTTP MCP tools/call on current main, where data and mimeType are dropped.

## Solution Assessment

No. The PR fixes the right boundary for valid multimodal blocks, but the maintainable fix should validate known MCP block shapes and required fields before passthrough and prove the endpoint path, not only the helper.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Validate MCP block shapes before passthrough:** `src/gateway/mcp-http.handlers.ts:25-30`
