## Summary

Keep open. Current main still has the lossy MCP HTTP `tools/call` normalization bug, and this PR targets the right gateway boundary, but the patch is not merge-ready because it passes through malformed or unknown typed blocks and lacks real after-fix behavior proof.

## What This Changes

This PR changes the MCP HTTP gateway content normalizer to return broader MCP content blocks and adds helper-level tests for text, image, audio, resource, resource_link, and fallback cases.

## Best Possible Solution

Revise this PR or a narrow replacement so the HTTP MCP response path preserves only valid MCP text, image, audio, embedded resource, and resource_link blocks, converts malformed or unknown blocks to text fallback output, and adds an endpoint regression for `tools/call`.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a loopback tool returning `{ content: [{ type: "image", data, mimeType }] }` through HTTP MCP `tools/call`; current main maps it down to only `type` and `text` before returning the JSON-RPC result.

## Solution Assessment

No. The PR fixes the right boundary for valid multimodal blocks, but the maintainable version should validate known MCP shapes and required fields before passthrough and cover the endpoint path, not only the helper.
