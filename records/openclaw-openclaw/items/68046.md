## Summary

Keep open. This PR targets a real MCP HTTP gateway data-loss bug and remains the focused implementation candidate, but the current patch accepts malformed or unknown typed blocks as valid MCP output and still lacks inspectable after-fix real behavior proof.

## What This Changes

The PR broadens `normalizeToolCallContent` to return MCP content blocks, passes through typed blocks, and adds helper-level tests for text, image, audio, resource, resource_link, and fallback cases.

## Best Possible Solution

Revise this PR, or a narrow replacement, so the gateway validates known MCP content block shapes, preserves only valid text/image/audio/resource/resource_link blocks, falls back to text for malformed or unknown entries, adds an HTTP `tools/call` regression, and includes redacted real behavior proof.

## Reproduction Assessment

Yes. A high-confidence source path exists: a loopback tool returning `{ content: [{ type: "image", data, mimeType }] }` through HTTP MCP `tools/call` reaches `normalizeToolCallContent`, which currently drops `data` and `mimeType`; I did not run a live request in this read-only review.

## Solution Assessment

No. The PR fixes the right boundary for valid multimodal blocks, but the maintainable version should validate SDK content block shapes and required fields before passthrough and cover the endpoint path, not only the helper.
