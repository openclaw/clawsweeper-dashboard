## Summary

Keep open: this is a valid gateway bug fix, but the branch still passes through any object with a string `type` and lacks inspectable real HTTP MCP proof, so it should not merge as-is.

## What This Changes

This PR changes the MCP HTTP `tools/call` content normalizer to return MCP text, image, audio, resource, and resource_link blocks and adds helper-level tests for those variants.

## Best Possible Solution

Revise this PR, or replace it narrowly, so the gateway preserves only SDK-valid MCP content blocks, falls back to text for malformed or unknown entries, proves the HTTP `tools/call` path, and then closes https://github.com/openclaw/openclaw/issues/68037 after merge.

## Reproduction Assessment

Yes, source-level. A loopback tool returning `{ content: [{ type: "image", data, mimeType }] }` reaches `normalizeToolCallContent` through HTTP MCP `tools/call` on current main, where the payload fields are dropped.

## Solution Assessment

No. The affected boundary is correct, but the maintainable fix should validate SDK-defined content-block shapes and required fields before passthrough and add endpoint-path proof before merge.
