## Summary

Keep this PR open: it targets a real MCP HTTP gateway data-loss bug and is linked to an open fixing issue, but the patch still needs stricter MCP content-block validation and contributor-provided real behavior proof before merge.

## What This Changes

This PR changes MCP HTTP tool-result normalization to pass through text, image, audio, resource, and resource_link blocks, and adds direct helper tests for those content types and fallback cases.

## Best Possible Solution

Preserve valid MCP content blocks according to the SDK contract, fall back to text for unknown or malformed blocks, add an HTTP tools/call regression for a non-text block, and keep the broader resource-to-model issue separate.

## Reproduction Assessment

Yes, from source inspection. Current main maps an image/audio/resource content block through normalizeToolCallContent into a shape that lacks the SDK-required non-text fields, and tools/call returns that normalized content directly.

## Solution Assessment

No. Passing through valid MCP blocks is the right direction, but this implementation should validate the SDK content-block shapes and prove the actual HTTP tools/call path rather than trusting any object with a string type.
