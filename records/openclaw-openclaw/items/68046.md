## Summary

Keep open. Current main still strips non-text MCP content fields on the HTTP `tools/call` path, and this PR is a focused fix candidate for the linked bug, but the proposed passthrough guard is still too broad and can emit unknown or malformed typed blocks as successful content instead of validating the SDK shapes first.

## What This Changes

This PR broadens the MCP HTTP tool-result content normalizer to pass through typed content blocks and adds direct helper tests for text, image, audio, resource, resource_link, and fallback cases.

## Best Possible Solution

Revise this PR, or a narrow replacement, so the gateway preserves only valid MCP text, image, audio, embedded resource, and resource_link blocks according to the SDK contract; strings and malformed entries should become text fallback blocks. Add an HTTP `tools/call` regression that returns an image block from a loopback tool and verifies `data` and `mimeType` survive before maintainer review and merge for #68037.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: a loopback tool returning `{ content: [{ type: "image", data, mimeType }] }` through HTTP MCP `tools/call` is currently mapped down to only `type` and `text`; the browser screenshot path naturally returns that image content shape.

## Solution Assessment

No. The PR targets the right boundary and preserves the important fields for valid blocks, but the final fix should validate known MCP block shapes and required fields before passthrough, then cover the HTTP route instead of only the helper.
