## Summary

Keep open. This PR targets a real current-main MCP HTTP gateway bug, but the proposed passthrough accepts any object with a string `type`, including malformed known blocks and unknown block types, and the external contributor has not provided inspectable real behavior proof.

## What This Changes

The PR broadens the MCP HTTP `tools/call` content normalizer to preserve text, image, audio, resource, and resource_link blocks, and adds helper-level tests for those variants and fallback cases.

## Best Possible Solution

Revise this PR, or a narrow replacement, so the gateway preserves only SDK-valid MCP content blocks, falls back to text for malformed or unknown entries, adds an HTTP `tools/call` regression, and includes redacted runtime proof.

## Reproduction Assessment

Yes. A high-confidence source path exists: a loopback tool returning `{ content: [{ type: "image", data, mimeType }] }` through HTTP MCP `tools/call` reaches `normalizeToolCallContent`, which currently drops `data` and `mimeType`; I did not run a live request in this read-only review.

## Solution Assessment

No. The PR fixes the right boundary for valid multimodal blocks, but the maintainable version should validate known MCP content block shapes and required fields before passthrough and cover the endpoint path, not only the helper.
