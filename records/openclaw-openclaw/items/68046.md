## Summary

Keep open. The underlying MCP HTTP gateway bug is valid on current main and in the latest release, but this PR still has a blocking shape-validation defect and lacks inspectable real HTTP MCP proof.

## What This Changes

The PR changes the MCP HTTP gateway content normalizer to pass through typed MCP text, image, audio, resource, and resource_link blocks and adds helper-level tests for those variants.

## Best Possible Solution

Revise this PR or replace it with a narrow gateway fix that preserves only SDK-valid MCP content blocks, falls back to text for malformed or unknown entries, adds an HTTP `tools/call` regression, and includes redacted real-call proof before merge.

## Reproduction Assessment

Yes. Source inspection shows a loopback tool returning an image/audio/resource block reaches `normalizeToolCallContent` through HTTP MCP `tools/call`, and current main drops all non-text payload fields there.

## Solution Assessment

No. Fixing `normalizeToolCallContent` is the right boundary, but this implementation should validate known SDK block shapes and required fields before passthrough, prove the endpoint path, and include real behavior proof.
