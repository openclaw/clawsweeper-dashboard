## Summary

Keep open. Current main still strips non-text MCP content fields on the HTTP `tools/call` path, and this PR targets the right gateway boundary, but the proposed guard accepts malformed or unknown typed blocks as valid MCP content and needs a narrower validation pass before merge.

## What This Changes

This PR broadens the MCP HTTP tool-result normalizer to return multimodal MCP content blocks and adds helper-level tests for text, image, audio, resource, resource_link, and fallback cases.

## Best Possible Solution

Preserve valid MCP text, image, audio, embedded resource, and resource_link blocks according to the SDK contract in the HTTP MCP `tools/call` response path, while converting malformed or unknown entries to safe text fallback output and covering the endpoint path.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a loopback tool returning `{ content: [{ type: "image", data, mimeType }] }` through HTTP MCP `tools/call`; current main maps it down to only `type` and `text` before returning the JSON-RPC result.

## Solution Assessment

No. The PR fixes the right boundary and preserves valid multimodal fields in the happy path, but the best version validates known MCP block shapes and required fields before passthrough and adds an HTTP route regression.
