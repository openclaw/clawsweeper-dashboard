## Summary

Keep open. Current main and the latest stable release still use the lossy HTTP MCP normalizer, and this PR is the active fix path, but it needs real behavior proof plus stricter SDK content-block validation before merge.

## What This Changes

The branch changes the MCP HTTP `tools/call` content normalizer to pass through text, image, audio, resource, and resource_link blocks, and adds direct helper tests for those variants.

## Best Possible Solution

Revise this PR, or replace it narrowly, so HTTP MCP preserves only SDK-valid text, image, audio, embedded resource, and resource_link blocks, falls back to text for malformed entries, and proves the real HTTP `tools/call` path before closing the linked issue.

## Reproduction Assessment

Yes, source-level. A loopback tool returning `{ content: [{ type: "image", data, mimeType }] }` reaches `normalizeToolCallContent` through HTTP MCP `tools/call` on current main, where only `type` and `text` are emitted.

## Solution Assessment

No. The boundary is correct, but the maintainable fix should validate SDK-defined block shapes and required fields before passthrough and should add endpoint-path proof before merge.
