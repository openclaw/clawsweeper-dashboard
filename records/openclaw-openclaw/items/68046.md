Keep open. The PR targets a real MCP HTTP gateway data-loss bug, but the branch still passes through typed blocks without validating the MCP SDK-required shape and lacks inspectable real HTTP MCP after-fix proof.

## What This Changes

The PR changes the MCP HTTP tool-result normalizer to preserve non-text content blocks and adds direct helper tests for text, image, audio, resource, resource_link, and fallback cases.

## Best Possible Solution

Refresh the PR so the gateway preserves SDK-valid MCP content blocks, falls back for malformed entries, keeps current handler hooks/signals intact, adds HTTP tools/call regression coverage, and includes redacted runtime proof.

## Reproduction Assessment

Yes. A high-confidence source path exists: an HTTP MCP loopback tool returning an image block reaches normalizeToolCallContent on current main, where data and mimeType are dropped before the JSON-RPC result is returned.

## Solution Assessment

No. Fixing normalizeToolCallContent is the right boundary, but this PR should validate known MCP block shapes and required fields, add endpoint-path coverage, and provide real after-fix proof before merge.
