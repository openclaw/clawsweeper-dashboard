## Summary

Keep open. Current `main` still relists the full task registry inside the per-task maintenance loop, so the O(n^2) CPU path described in the issue remains present. The same-author PR #75709 is open, unmerged, and explicitly targets this issue, so the right cleanup action is to leave the issue open until that PR or an equivalent fix lands.

## What This Changes

The issue asks for task registry maintenance to reuse the existing sweep snapshot for terminal ACP cleanup instead of calling the full registry listing once per task.

## Best Possible Solution

Review and land PR #75709, or an equivalent narrow patch, so terminal ACP cleanup reuses the initial maintenance snapshot while retaining the final fresh snapshot for orphaned parent-owned ACP sessions and adding the list-count regression test.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to run `runTaskRegistryMaintenance()` with many task records and a counting `listTaskRecords` runtime; current main calls the full registry listing once for the initial sweep, once per task at terminal ACP cleanup, and once for final orphan cleanup.

## Solution Assessment

Yes. Reusing the already captured `tasks` array for `cleanupTerminalAcpSession(current, tasks)` is the narrowest maintainable fix because that helper only needs the supplied task array for active child-session checks.
