## Summary

Keep open. Current main still relists the whole task registry inside the per-task maintenance loop, and PR #75709 is open, unmerged, and explicitly targets this issue, so the issue should stay open until that PR or an equivalent fix lands.

## What This Changes

The issue asks task registry maintenance to avoid O(n^2) CPU work by not cloning, sorting, and mapping the full task registry once per task during ACP session cleanup.

## Best Possible Solution

Update and land PR #75709, or an equivalent narrow patch, so terminal ACP cleanup avoids the per-task full registry relist while preserving current active-child-session safety, adds focused regression coverage, and includes the required changelog entry.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to inject a counting `listTaskRecords`, seed many terminal ACP tasks, and run `runTaskRegistryMaintenance()`; current main calls the full listing once for the initial sweep, once per task that reaches terminal ACP cleanup, and once for orphan cleanup.

## Solution Assessment

No, not as a simple stale-snapshot swap. The maintainable fix should remove the full per-task listing with a current indexed same-session lookup or equivalent live view, rather than making the cleanup guard blind to concurrent task creation.
