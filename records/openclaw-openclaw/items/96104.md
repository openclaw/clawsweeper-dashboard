## Summary

Keep open: the PR targets a useful exec observability feature, but the patch is incorrect because it attaches `OPENCLAW_SESSION_ID` to the built-in `bash` tool instance that OpenClaw discards, while the real subprocess environment is built later by the `exec` runtime.

## What This Changes

The PR attempts to pass `OPENCLAW_SESSION_ID` into spawned subprocesses by adding a `BashSpawnHook` option when `createOpenClawCodingTools` creates base coding tools.

## Best Possible Solution

Align `OPENCLAW_SESSION_ID` with the canonical exec env-marker contract, then inject it at the actual exec child-env boundary with focused tests and real behavior proof.

## Reproduction Assessment

Yes for the review finding: source inspection shows `createOpenClawCodingTools` skips the modified `bash` tool and constructs the real `exec` tool separately, so the PR cannot affect spawned exec subprocesses. The requested feature itself is not a bug reproduction; current main and `v2026.6.9` still lack the marker.

## Solution Assessment

No. Injecting a session marker at the child-process env boundary is plausible, but this PR changes the wrong layer and still needs maintainer-set trust and override semantics before it can be the best fix.
