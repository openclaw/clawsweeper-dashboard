## Summary

This should stay open: the report describes a real shipped post-update Control UI auth recovery failure, and current source only proves the individual stale-device-token and missing-token states, not a complete service-repair recovery path.

## What This Changes

The issue asks OpenClaw to recover or clearly guide users when the Control UI loads after an update/service repair but cannot authenticate to the Gateway because browser/device token state is stale or missing.

## Best Possible Solution

Define a SecretRef-safe post-update recovery path that either relaunches with a fresh tokenized URL when safe or surfaces a non-looping auth recovery state after stale device-token rejection, with focused Control UI, dashboard-command, and service-repair coverage.

## Reproduction Assessment

No full live macOS update reproduction was established here. The report has detailed logs, and current source/tests clearly cover the token_missing and device_token_mismatch branches that explain the observed failed connection states.

## Solution Assessment

Unclear. The current implementation handles pieces of the failure, but the best fix needs maintainer direction on the safe boundary between automatic token handoff, stale browser auth invalidation, and SecretRef-safe manual recovery.
