## Summary

Keep open for human review. Current main and the documented config contract still define post-compaction section extraction as H2/H3, while this PR broadens runtime parsing to H1-H4 without the matching docs/config/help updates, without real after-fix behavior proof, and with an unanswered member review question about the real H1 case and H4 scope.

## What This Changes

This PR widens `extractSections` to match H1-H4 Markdown headings and adds H1/H4 post-compaction context regression tests.

## Best Possible Solution

End with a maintainer-approved heading-level contract where parser behavior, regression tests, config comments, schema help, and public docs agree; if H1 is accepted, narrow the supported levels deliberately, otherwise retain the documented H2/H3 behavior.

## Reproduction Assessment

Yes for the source-level parser behavior: current main only matches H2/H3 headings, so H1/H4 headings are skipped. No for a high-confidence bug against the public contract, because docs, typed config help, and the shipped template define H2/H3 extraction.

## Solution Assessment

No as proposed. The diff broadens shared runtime behavior to H1-H4 without settling the heading-level policy or updating all contract surfaces, so the safer path is maintainer approval followed by aligned code, docs, and tests.
