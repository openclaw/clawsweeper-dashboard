## Summary

Keep open: current main and the latest release still cap active non-loopback gateway status/probe targets at 1500ms, and this PR’s narrow resolver reorder matches the documented timeout contract and the linked bug. I found no blocking patch or security findings; the remaining uncertainty is only that the original slow remote gateway topology was not exercised live.

## What This Changes

This PR reorders gateway status probe budget resolution so active configured or explicit remote targets receive the caller timeout while inactive remote/local and SSH tunnel probes keep short caps, with focused regression tests.

## Best Possible Solution

Land the narrow resolver reorder with its regression tests after normal CI, preserving short caps only for inactive/background and SSH tunnel probes.

## Reproduction Assessment

Yes from source inspection: current main and `v2026.5.28` cap any non-loopback target at 1500ms before checking whether it is the active target, so an active remote probe cannot honor a larger `--timeout`. I did not run a live slow remote gateway.

## Solution Assessment

Yes: `resolveProbeBudgetMs()` is the narrow owner boundary consumed directly by `runGatewayStatusProbePass()`, and reordering active-target handling before the non-loopback cap fixes active configured and explicit remotes while preserving SSH and inactive-target caps.
