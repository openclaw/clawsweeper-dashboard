## What This Changes

Reorders gateway-status probe budget resolution so active configured or explicit remote targets use the caller timeout, with helper and command-level regression tests.

## Best Possible Solution

Land the narrow resolver reorder with its regression tests after normal CI and maintainer review, preserving short caps for inactive/background and SSH tunnel probes.

## Reproduction Assessment

Yes, from source inspection: current main and `v2026.6.6` cap active non-loopback `configRemote` and explicit remote targets at 1500 ms before `target.active` can allow the caller timeout. I did not run the reporter's private slow remote gateway.

## Solution Assessment

Yes: `resolveProbeBudgetMs()` is the narrow owner boundary consumed directly by `runGatewayStatusProbePass()`, and the PR fixes active remotes while preserving SSH and inactive-target caps. A broader CLI or probe-client rewrite is not needed for this bug.
