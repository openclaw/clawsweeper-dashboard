## Summary

Keep open: current main and v2026.6.6 still cap active non-loopback gateway probes at 1500ms, and this PR is the focused implementation candidate with no blocking code findings from this pass.

## What This Changes

This PR reorders gateway status probe budget resolution so active configured or explicit remote targets honor the caller timeout while inactive remote/local and SSH tunnel targets keep short caps, with regression tests.

## Best Possible Solution

Land the narrow resolver reorder with its regression tests if maintainers accept the active-remote latency tradeoff, keeping inactive/background and SSH tunnel probes bounded.

## Reproduction Assessment

Yes, from source inspection: current main and v2026.6.6 cap active non-loopback configRemote and explicit remote targets at 1500ms before active-target handling can use the caller timeout. I did not run the reporter's private slow remote gateway.

## Solution Assessment

Yes: resolveProbeBudgetMs is the narrow owner boundary consumed directly by runGatewayStatusProbePass, and the PR fixes active remote targets while preserving inactive/background and SSH tunnel caps. Raising the cap globally or only changing output would be broader or less faithful to the requested timeout.
