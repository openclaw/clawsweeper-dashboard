## Summary

Keep open: this is the viable linked fix candidate for the remote gateway probe timeout bug. Current main and v2026.6.9 still cap active non-loopback probes at 1500ms, while this PR has the narrow source change, focused tests, and after-fix live output; the remaining step is maintainer acceptance of the diagnostic latency tradeoff.

## What This Changes

The PR reorders gateway status probe budget resolution so active configured or explicit remote targets receive the caller timeout while inactive remote/local and SSH tunnel probes keep short caps, with regression tests.

## Best Possible Solution

Land the focused resolver change if maintainers accept the active-remote timeout tradeoff, then let it close https://github.com/openclaw/openclaw/issues/65355.

## Reproduction Assessment

Yes. Current main and v2026.6.9 cap active non-loopback gateway-status targets at 1500ms before active targets can receive the caller budget, and the linked issue plus PR proof show slow reachable remotes crossing that cap.

## Solution Assessment

Yes. resolveProbeBudgetMs is the narrow owner boundary consumed by runGatewayStatusProbePass, and the PR changes active remote behavior while preserving inactive remote/local and SSH short caps.
