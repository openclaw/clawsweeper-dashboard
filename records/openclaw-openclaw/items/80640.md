## Summary

Close as superseded: the default-on request was already covered by https://github.com/openclaw/openclaw/issues/77474 and https://github.com/openclaw/openclaw/pull/77491, while current main already runs the post-compaction guard unless the loop-detection switch is explicitly false.

## What This Changes

Request to make tool-loop detection default-on, or warn operators when long-running agents leave it disabled, because default-off leaves rolling loop detection unused.

## Best Possible Solution

Keep the default-off policy, enable `tools.loopDetection.enabled: true` for affected deployments, keep the merged post-compaction guard, and handle disabled-mode diagnostics through https://github.com/openclaw/openclaw/pull/77607.

## Reproduction Assessment

Not applicable as a strict bug reproduction. Source inspection confirms the rolling detector default is false, but current main and docs refute the post-compaction guard dormancy claim because the guard runs unless the switch is explicitly false.

## Solution Assessment

Yes for cleanup. Closing this as superseded is the best route because the default flip was explicitly declined, the post-compaction guard is already on current main, and the warning/diagnostic alternative is already tracked by an open PR.
