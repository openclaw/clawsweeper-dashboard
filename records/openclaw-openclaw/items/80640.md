## Summary

Keep open: current main already prevents the post-compaction guard from sitting dormant by default, but the remaining request to enable rolling-history loop detection by default or add a doctor warning is still a product/config-policy decision.

## What This Changes

Request to enable tool-loop detection by default for new installs or warn when long-running daemon agents leave it disabled.

## Best Possible Solution

Set an explicit maintainer policy for rolling-history detector defaults: either enable them for new installs with docs and regression coverage, or add a targeted doctor warning for daemon/long-lived agents while preserving explicit false opt-out.

## Reproduction Assessment

Yes for the current default behavior from source: no config leaves rolling-history detectors disabled, and detectToolCallLoop returns no-op when enabled is false. I did not reproduce the reported 13,508-call production run, and current main does not reproduce the post-compaction guard being dormant unless enabled:false is explicit.

## Solution Assessment

Unclear: flipping DEFAULT_LOOP_DETECTION_CONFIG.enabled to true is one possible fix, but current docs intentionally split default-off rolling detectors from default-on post-compaction protection. Maintainers should choose between a default flip and a doctor warning before implementation.
