Close as superseded: the post-compaction guard is already default-on unless explicitly disabled, the default-on rolling detector policy was already rejected in the earlier canonical discussion, and the remaining disabled-mode visibility work is covered by an open diagnostic PR.

## What This Changes

Request to enable rolling tool-loop detection by default, or warn operators when long-running agents leave it disabled, because repeated tool calls can burn through long-lived sessions.

## Best Possible Solution

Keep rolling loop detection opt-in per the prior policy, preserve the shipped default-on post-compaction guard, and handle any disabled-mode visibility through the existing diagnostic PR or a new maintainer-approved policy item.

Yes for the source-level default behavior: no loopDetection config leaves rolling-history detectors disabled, while current main keeps the post-compaction guard active unless enabled is explicitly false. I did not reproduce the reported 13,508-call production run.
