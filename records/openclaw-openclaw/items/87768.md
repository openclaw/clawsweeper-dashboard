- `P2`: The issue is a normal-priority bug affecting a real macOS push-to-talk workflow for an Ollama model, with a narrow and traceable surface.
- `impact:message-loss`: The reported failure causes the dictated push-to-talk message not to appear in WebChat after the gateway rejects the run.
- `impact:auth-provider`: The failure depends on provider/model thinking capability resolution for Ollama, where the selected model only supports `off`.

## Summary

Keep open: current main and the latest release still hard-code macOS Voice Wake/push-to-talk forwarding to `thinking: "low"`, and the gateway treats that as an explicit thinking override that can fail for Ollama models whose resolved profile only allows `off`.

## What This Changes

The issue reports that macOS companion push-to-talk sends Ollama requests with an unsupported hard-coded thinking level of `low`, resulting in no webchat message.

## Best Possible Solution

Make macOS voice forwarding inherit the session/provider-supported thinking policy instead of forcing `low`, then add focused macOS forwarding tests and verify with an Ollama off-only model path.

## Reproduction Assessment

Yes, from source: macOS PTT calls `VoiceWakeForwarder`, which sends `thinking: "low"`, and the gateway rejects explicit unsupported thinking for provider/model profiles that only allow `off`. I did not run a live macOS/Ollama PTT session in this read-only review.

## Solution Assessment

Yes, the narrow maintainable fix is to stop hard-coding `low` in the native voice forwarding path and let the gateway/session/provider policy choose or validate the supported thinking level. This should not change Ollama provider policy globally.
