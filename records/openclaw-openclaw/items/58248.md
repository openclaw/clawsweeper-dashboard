- **[P1] Delete the duplicate browser origin helper:** `src/gateway/http-utils.ts:46-57`
  - body: This PR declares `resolveHttpBrowserOriginPolicy` twice in the same module. The duplicate top-level export makes `http-utils` invalid, so importing gateway HTTP auth paths will fail before the new feature can run.
  - confidence: 0.96
- **[P1] Preserve the configured auth mode:** `src/commands/auth-rotate.ts:43`
  - body: Running `auth rotate` on a config using `password`, `none`, or `trusted-proxy` reaches this write and persists `mode: "token"`, silently changing the gateway auth strategy. Reject non-token modes or require an explicit mode change before writing a token.
  - confidence: 0.95
- **[P1] Reject env-managed tokens before writing config:** `src/commands/auth-rotate.ts:23-27`
  - body: This path checks only the token visible in the config snapshot, while current main's `sourceConfig` is already after env substitution and the gateway also supports `OPENCLAW_GATEWAY_TOKEN` with no config token. Rotation can persist a plaintext config token that takes precedence over the env-managed source.
  - confidence: 0.91
- **[P2] Emit expiry warnings from shared HTTP auth:** `src/gateway/http-auth-helpers.ts:35-40`
  - body: The warning runs only in `authorizeGatewayBearerRequestOrReply`, but normal HTTP routes use the shared HTTP auth utility path on current main. Those common routes will not evaluate `tokenExpiryHours`, so the new config is ineffective for ordinary gateway usage.
  - confidence: 0.87
- **[P2] Make rotated token activation explicit:** `src/commands/auth-rotate.ts:53-55`
  - body: This updates only CLI process memory and prints the new token immediately, but current gateway auth config changes are restart-backed. A running gateway can still require the old token until restart or reload completes, so users can switch too early and see unauthorized failures.
  - confidence: 0.86
