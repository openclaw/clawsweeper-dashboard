## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Delete the duplicate browser-origin helper:** `src/gateway/http-utils.ts:46-57`
  - body: This patch adds a second top-level `resolveHttpBrowserOriginPolicy` declaration while the same function already exists later in the file, so importing this module fails with a duplicate identifier before HTTP auth can run.
  - confidence: 0.96
- **[P1] Preserve the configured auth mode:** `src/commands/auth-rotate.ts:43`
  - body: `auth rotate` writes `mode: "token"` for every valid config. A gateway using password, none, or trusted-proxy auth would be silently switched to token auth on disk, changing access behavior on reload or restart.
  - confidence: 0.95
- **[P1] Reject env-managed tokens before writing config:** `src/commands/auth-rotate.ts:23-27`
  - body: The command only checks the config token value, so deployments that provide `OPENCLAW_GATEWAY_TOKEN` or an env-template token can get a new plaintext token persisted to config. Because current auth resolution is config-first, that changes the secret source of truth.
  - confidence: 0.92
- **[P2] Warn that the printed token is restart-only:** `src/commands/auth-rotate.ts:54-55`
  - body: The command prints the new token immediately after writing config, but gateway auth changes are restart-backed and can be deferred while a running gateway still accepts the old token. Users or scripts can switch to the printed token too early and get unauthorized failures.
  - confidence: 0.86
