## Summary

This PR should stay open: the idea is plausible, but the current branch is not merge-ready because PR-head source inspection shows a duplicate exported helper that breaks the module plus blocking gateway credential-mode and secret-source regressions. The PR body/comments still provide no redacted real gateway proof for either `openclaw auth rotate` or the expiry warning path.

## What This Changes

The PR adds a `gateway.auth.tokenExpiryHours` config field, process-local gateway token-expiry warning state, an `openclaw auth rotate` CLI, generated config metadata, and focused unit tests.

## Best Possible Solution

Rebase onto current main and rework the feature around the shared HTTP auth/config-write contracts: constrain rotation to active plaintext token-mode configs unless maintainers explicitly choose a broader policy, reject env/SecretRef-managed sources, make restart activation explicit, and add redacted real gateway proof.

## Reproduction Assessment

Yes. Source inspection of PR head reproduces the duplicate helper declaration, unconditional mode overwrite, env/SecretRef source risk, narrow warning hook, and restart-activation ambiguity without needing a live gateway run.

## Solution Assessment

No. The current implementation is not the best way to solve it because it bypasses current main's shared auth/config contracts; a safer solution would preserve existing modes and credential ownership by default and make activation semantics explicit.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Delete the duplicate browser-origin helper:** `src/gateway/http-utils.ts:46-57`
  - body: This patch adds a second top-level `resolveHttpBrowserOriginPolicy` declaration in `http-utils.ts`. Importing the module will fail with a duplicate identifier before the HTTP auth routes can run.
  - confidence: 0.98
- **[P1] Preserve the configured auth mode:** `src/commands/auth-rotate.ts:43`
  - body: `auth rotate` unconditionally writes `mode: "token"`, so running it on password, none, or trusted-proxy gateways silently changes the auth strategy on disk. Reject non-token modes or update only an already-token-mode config.
  - confidence: 0.96
- **[P1] Reject env-managed tokens before writing config:** `src/commands/auth-rotate.ts:23-27`
  - body: The command only checks the config token value and then persists a new plaintext token. For `OPENCLAW_GATEWAY_TOKEN` or env-template deployments, this changes the credential source of truth and can put a token on disk.
  - confidence: 0.91
- **[P2] Emit expiry warnings from the shared HTTP auth path:** `src/gateway/http-auth-helpers.ts:35-39`
  - body: The warning is only attached to `authorizeGatewayBearerRequestOrReply`, but normal gateway endpoints authenticate through the shared HTTP request auth utility. As written, `tokenExpiryHours` can be set without warning on common authenticated traffic.
  - confidence: 0.86
- **[P2] Warn or wait for restart before printing a usable token:** `src/commands/auth-rotate.ts:53-55`
  - body: Gateway auth token/mode changes are restart-backed, but the command prints the new token immediately after writing config. Users or scripts that switch to it right away can get unauthorized responses until restart completes.
  - confidence: 0.84
