## Summary

Keep this PR open, but it is not merge-ready: current main does not implement the requested feature, while the branch is conflicting, lacks real behavior proof, and still has concrete gateway auth/security defects.

## What This Changes

The PR adds a `gateway.auth.tokenExpiryHours` config field, token-expiry warning state/tests, an `openclaw auth rotate` CLI, generated config metadata, and a small Matrix import guardrail fix.

## Best Possible Solution

Rework the feature on current main so token rotation is limited to an already-active plaintext token-mode config, external secret sources stay external, expiry warnings run through shared HTTP auth, and restart activation is explicit and tested.

## Reproduction Assessment

Yes for the blocking PR findings by source inspection: the diff writes token mode unconditionally, can override env-managed token setups, and adds warning logic outside current main's shared HTTP auth path. I did not run the command because this is a read-only review.

## Solution Assessment

No. The feature direction may be useful, but this implementation changes auth mode/source semantics and does not match the current gateway auth module layout.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Preserve non-token auth modes during rotation:** `src/commands/auth-rotate.ts:44`
  - body: Running the new rotate command on a config using `password`, `none`, or `trusted-proxy` reaches this write and persists `mode: "token"`, silently changing the gateway auth strategy. Reject non-token modes or require an explicit mode change before writing a token.
  - confidence: 0.94
- **[P1] Reject externally managed tokens before writing config:** `src/commands/auth-rotate.ts:29-33`
  - body: This check only looks at the config token field. If the deployment uses an env template or `OPENCLAW_GATEWAY_TOKEN` with no config token, the command can write a new plaintext config token; current auth resolution prefers config credentials on startup, so this changes the source of truth.
  - confidence: 0.9
- **[P2] Run expiry warnings from the shared HTTP auth path:** `src/gateway/http-auth-helpers.ts:32-42`
  - body: The warning is emitted only from the branch's bearer-helper path, but current main's common HTTP endpoints authenticate through `checkGatewayHttpRequestAuth`. Token-authenticated requests on those routes would not evaluate `tokenExpiryHours`, so the config field would appear ineffective.
  - confidence: 0.86
- **[P2] Warn that the printed token is restart-gated:** `src/commands/auth-rotate.ts:53-55`
  - body: `gateway.auth.token` and `gateway.auth.mode` are restart-class config changes, yet the command logs success and prints the new token immediately after writing the file. A running gateway can still require the old token until restart, so scripts may switch too early and receive unauthorized responses.
  - confidence: 0.84
