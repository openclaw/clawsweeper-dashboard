## Summary

Keep open. Current main does not already implement the requested command/config, but this PR head is not merge-ready because the rotate flow can silently change auth mode and credential source, and the expiry warning is wired outside current main’s shared HTTP auth path.

## What This Changes

The PR adds a gateway token age warning config/state path, generated config metadata/tests, and an `openclaw auth rotate` CLI that persists a new gateway token.

## Best Possible Solution

Rework the feature against current main so rotation only affects an already-active plaintext config token, external secret sources remain external, expiry warnings run from shared auth code, and the CLI clearly handles or reports restart activation.

## Reproduction Assessment

Yes. From the PR diff, configure `gateway.auth.mode` as `password`, `none`, or `trusted-proxy` and run the proposed rotate command; it writes `mode: "token"`. A second path is using `OPENCLAW_GATEWAY_TOKEN` with no config token, where the command still persists a new plaintext config token.

## Solution Assessment

No. The feature direction is useful, but this patch is not the narrowest maintainable solution because it bypasses current credential-source semantics, misses the shared HTTP auth path, and treats restart-only auth changes as immediately active.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Preserve non-token auth modes when rotating:** `src/commands/auth-rotate.ts:44`
  - body: Running `auth rotate` on `password`, `none`, or `trusted-proxy` configs reaches this write and persists `mode: "token"`, silently changing the gateway auth strategy. Reject non-token modes or require an explicit mode change instead.
  - confidence: 0.94
- **[P1] Reject env-managed tokens before writing config:** `src/commands/auth-rotate.ts:21-31`
  - body: The command only checks the config value. If the deployment relies on `OPENCLAW_GATEWAY_TOKEN` with no config token, this path writes a plaintext config token, and current auth resolution gives config credentials precedence on startup.
  - confidence: 0.9
- **[P2] Move expiry warnings into the shared HTTP auth path:** `src/gateway/http-auth-helpers.ts:32-42`
  - body: Current main authenticates most HTTP routes through `checkGatewayHttpRequestAuth`, not this bearer helper. With the warning here, common token-authenticated endpoints never evaluate `tokenExpiryHours`; emit the warning from the shared auth path.
  - confidence: 0.86
- **[P2] Account for restart before printing the rotated token:** `src/commands/auth-rotate.ts:53-55`
  - body: `gateway.auth.token` changes are restart-class changes, but this command prints success and the new token immediately after writing config. A running gateway can keep accepting only the old token until restart, so scripts that switch immediately can lock themselves out.
  - confidence: 0.84
- **[P3] Add the required changelog entry:** `src/cli/auth-cli.ts:18`
  - body: This adds a user-facing CLI command and config field without a `CHANGELOG.md` entry. Add a single-line entry under the active release section so the new auth workflow is discoverable.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The patch changes gateway credential persistence and auth mode behavior, with concrete mode-switch and secret-source risks.

Concerns:

- **[high] Token rotation can silently change auth mode:** `src/commands/auth-rotate.ts:44`
  - body: Persisting `mode: "token"` during rotation can replace password, none, or trusted-proxy auth without explicit operator consent, changing the gateway security posture and potentially breaking access.
  - confidence: 0.94
- **[medium] Env-managed tokens can become plaintext config secrets:** `src/commands/auth-rotate.ts:21`
  - body: When `OPENCLAW_GATEWAY_TOKEN` is the source of truth and config has no token, the PR writes a new plaintext token into config, changing credential precedence and persisting a secret unexpectedly.
  - confidence: 0.9
