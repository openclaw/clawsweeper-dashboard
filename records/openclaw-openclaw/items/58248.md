## Summary

This PR should stay open but is not mergeable in its current form: the head still has source-reproducible auth/security defects, an invalid duplicate export, failing checks, and no real behavior proof for the new gateway credential flow.

## What This Changes

This PR adds a gateway token-expiry warning config, an `openclaw auth rotate` CLI that writes a new gateway token, generated config metadata/tests, and a small Matrix import-barrel cleanup.

## Best Possible Solution

Rework the feature against current main so rotation only updates active plaintext token-mode configs, refuses external secret sources, makes restart/activation semantics explicit, emits warnings from the shared HTTP auth path, and proves the flow in a real gateway setup.

## Reproduction Assessment

Yes. Source inspection of the PR head reproduces the blocking defects, and current main confirms the env-substitution, config-first auth, shared HTTP auth, and restart-gated token behavior they depend on.

## Solution Assessment

No. The feature direction may be useful, but this implementation changes credential and auth-mode semantics, misses the current shared auth layout, and needs real behavior proof before it is mergeable.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Remove the duplicate browser origin helper:** `src/gateway/http-utils.ts:46-57`
  - body: This branch declares `resolveHttpBrowserOriginPolicy` twice in the same module. TypeScript/ESM will reject the duplicate top-level export, so importing the HTTP auth utility module can fail before the new auth behavior runs.
  - confidence: 0.95
- **[P1] Preserve the configured auth mode:** `src/commands/auth-rotate.ts:43`
  - body: Running `auth rotate` on a config using `password`, `none`, or `trusted-proxy` reaches this write and persists `mode: "token"`, silently changing the gateway auth strategy. Reject non-token modes or require an explicit mode change before writing a token.
  - confidence: 0.94
- **[P1] Reject env-managed tokens before writing config:** `src/commands/auth-rotate.ts:23-27`
  - body: This check only rejects SecretRef values already visible in the snapshot. If the deployment uses `OPENCLAW_GATEWAY_TOKEN` or an env-template token that has already been substituted, the command can persist a plaintext config token and change the credential source of truth.
  - confidence: 0.9
- **[P2] Emit expiry warnings from the shared HTTP auth path:** `src/gateway/http-auth-helpers.ts:35-40`
  - body: The warning is emitted only from `authorizeGatewayBearerRequestOrReply`, but current normal HTTP routes authenticate through the shared HTTP auth utility path. Those routes will not evaluate `tokenExpiryHours`, making the config ineffective for common gateway usage.
  - confidence: 0.87
- **[P2] Clarify restart-gated token activation:** `src/commands/auth-rotate.ts:53-55`
  - body: The command prints the new token immediately after writing config and updates only process-local CLI memory. A running gateway can still require the old in-memory token until restart completes, so users or scripts may switch too early and get unauthorized responses.
  - confidence: 0.84
