## Summary

Keep open. Current main does not implement the token-expiry warning or `openclaw auth rotate`, but the latest PR head still has blocking auth/token persistence bugs, security-sensitive behavior changes, failing CI, and no after-fix real behavior proof.

## What This Changes

The branch adds `gateway.auth.tokenExpiryHours`, in-process expiry warning state/tests, generated config metadata, and an `openclaw auth rotate` CLI that writes a generated gateway token.

## Best Possible Solution

Rework the feature against current main so rotation only affects an already-active plaintext token-mode config, external secret sources remain external, expiry warnings run from shared auth, and restart behavior is explicit and tested.

## Reproduction Assessment

Yes for the blocking PR findings by source inspection: the latest diff writes `mode: "token"`, writes a generated token without detecting env-managed sources, and places warnings outside the shared HTTP auth path. I did not run the command because this is a read-only review.

## Solution Assessment

No. The feature direction may be useful, but this patch is not the best implementation because it changes auth mode/source semantics, misses common HTTP auth routes, and reports a rotated token before restart activation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Preserve non-token auth modes during rotation:** `src/commands/auth-rotate.ts:44`
  - body: Running the new rotate command on a config using `password`, `none`, or `trusted-proxy` reaches this write and persists `mode: "token"`, silently changing the gateway auth strategy. Reject non-token modes or require an explicit mode change before writing a token.
  - confidence: 0.94
- **[P1] Reject externally managed tokens before writing config:** `src/commands/auth-rotate.ts:21-31`
  - body: This check only looks at the config token value. If the deployment uses an env-template or `OPENCLAW_GATEWAY_TOKEN` with no config token, the command can write a new plaintext config token; current auth resolution prefers config credentials on startup, so this changes the source of truth.
  - confidence: 0.9
- **[P2] Run expiry warnings from the shared HTTP auth path:** `src/gateway/http-auth-helpers.ts:32-42`
  - body: The warning is emitted only from `authorizeGatewayBearerRequestOrReply`, but current main's common HTTP endpoints authenticate through `checkGatewayHttpRequestAuth`. Token-authenticated requests on those routes would never evaluate `tokenExpiryHours`, so the config field would appear ineffective.
  - confidence: 0.86
- **[P2] Warn that the printed token is restart-gated:** `src/commands/auth-rotate.ts:53-55`
  - body: `gateway.auth.token` and `gateway.auth.mode` are restart-class config changes, yet the command logs success and prints the new token immediately after writing the file. A running gateway can still require the old token until restart, so scripts may switch too early and receive unauthorized responses.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The patch changes gateway credential persistence and auth mode behavior, and the latest diff still has concrete mode-switch and secret-source risks.

Concerns:

- **[high] Token rotation can silently change auth mode:** `src/commands/auth-rotate.ts:44`
  - body: Persisting `mode: "token"` during rotation can replace password, none, or trusted-proxy auth without explicit operator consent, changing the gateway security posture and potentially breaking access.
  - confidence: 0.94
- **[medium] Env-managed tokens can become plaintext config secrets:** `src/commands/auth-rotate.ts:21`
  - body: When an env-template or `OPENCLAW_GATEWAY_TOKEN` is the source of truth, the command can persist a new plaintext token into config, changing credential precedence and leaving a secret on disk unexpectedly.
