## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Remove the duplicate browser-origin helper:** `src/gateway/http-utils.ts:65`
  - body: PR head declares `resolveHttpBrowserOriginPolicy` twice in this module, so importing the gateway HTTP utilities can fail with an identifier redeclaration before the new auth behavior runs.
  - confidence: 0.97
- **[P1] Preserve the configured auth mode:** `src/commands/auth-rotate.ts:43`
  - body: `auth rotate` always writes `mode: "token"`, so running it against password, none, or trusted-proxy configs silently changes the gateway auth strategy. Reject non-token modes or preserve the existing mode instead of rewriting it.
  - confidence: 0.97
- **[P1] Reject env-managed gateway tokens:** `src/commands/auth-rotate.ts:23-27`
  - body: The command only checks `gateway.auth.token` in config, so a deployment using `OPENCLAW_GATEWAY_TOKEN` with no config token can get a new plaintext token written to config. Since config tokens take precedence later, future startups can ignore the intended env-managed credential.
  - confidence: 0.94
- **[P2] Move expiry warnings into shared HTTP auth:** `src/gateway/http-auth-helpers.ts:35-39`
  - body: The warning runs only in `authorizeGatewayBearerRequestOrReply`, while current main's normal HTTP auth is centralized in `checkGatewayHttpRequestAuth`. As written, the new policy can miss ordinary authenticated gateway traffic.
  - confidence: 0.88
- **[P2] Do not reset gateway expiry state in the CLI process:** `src/commands/auth-rotate.ts:53`
  - body: `setGatewayTokenIssuedAtNow()` updates module-local state in the short-lived CLI process, not the running gateway process that evaluates expiry warnings. Keeping this call here makes the rotation behavior look more complete than it is.
  - confidence: 0.86
