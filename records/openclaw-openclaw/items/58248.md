## Summary

Keep open. Current main does not already contain the requested token-expiry setting or `auth rotate` CLI, so the PR is not obsolete, but the proposed patch is not merge-ready because it can silently change gateway auth mode, can convert externally managed tokens into plaintext config, and wires the warning into a narrower auth path than current main uses.

## What This Changes

The PR adds `gateway.auth.tokenExpiryHours`, in-process token expiry warning state, generated config metadata, tests, and an `openclaw auth rotate` CLI that writes a fresh gateway token to config.

## Best Possible Solution

Rework the feature so rotation only updates an already-active plaintext config token, rejects or explicitly opts into mode/source changes, leaves env and SecretRef-managed credentials at their source, emits expiry warnings from the shared HTTP auth path, and clearly handles the gateway restart boundary before telling users a new token is active.

## Reproduction Assessment

Yes. The main blockers can be reproduced from the PR diff by configuring `gateway.auth.mode` as `password`, `none`, or `trusted-proxy` and running the proposed rotate command, or by setting `OPENCLAW_GATEWAY_TOKEN` with no config token and observing that the PR persists a new plaintext config token.

## Solution Assessment

No. The feature direction is useful, but this patch is not the best way to solve it because it bypasses current credential-source semantics, misses the shared HTTP auth path on current main, and reports rotation success before the running gateway necessarily loads the new token.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve non-token auth modes when rotating:** `src/commands/auth-rotate.ts:44`
  - body: `auth rotate` writes `mode: "token"` unconditionally. On a gateway configured for `password`, `none`, or `trusted-proxy`, running the helper silently changes the auth strategy on disk; reject non-token modes or preserve the existing mode instead.
  - confidence: 0.94
- **[P1] Reject env-managed tokens before writing config:** `src/commands/auth-rotate.ts:21-31`
  - body: The command only checks `gateway.auth.token` in the loaded config. If the deployment uses an env template or `OPENCLAW_GATEWAY_TOKEN`, rotation persists a new plaintext config token and changes the credential source away from the external secret.
  - confidence: 0.9
- **[P2] Move expiry warnings into the shared HTTP auth path:** `src/gateway/http-auth-helpers.ts:32-42`
  - body: The warning runs only in this bearer helper, but current main routes normal HTTP endpoints through `checkGatewayHttpRequestAuth` / `authorizeGatewayHttpRequestOrReply`. As written, many authenticated gateway requests would never evaluate `tokenExpiryHours`.
  - confidence: 0.86
- **[P2] Account for restart before printing the rotated token:** `src/commands/auth-rotate.ts:53-55`
  - body: The command writes `gateway.auth.token`, resets only the short-lived CLI process state, and immediately prints success. Gateway auth config changes are restart-class changes, so users can switch clients to a token the running gateway has not loaded yet.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The PR changes gateway auth and token storage behavior and introduces concrete auth-mode and credential-source risks.

Concerns:

- **[high] Token rotation can silently change auth mode:** `src/commands/auth-rotate.ts:44`
  - body: Writing `mode: "token"` during rotation can replace password, none, or trusted-proxy auth without explicit operator consent, changing the gateway security posture and potentially breaking access.
  - confidence: 0.94
- **[medium] Externally managed tokens can become plaintext config:** `src/commands/auth-rotate.ts:21`
  - body: When the token is supplied by an env template or `OPENCLAW_GATEWAY_TOKEN`, the PR can write a new literal token to config, changing credential precedence and persisting a secret unexpectedly.
  - confidence: 0.9
