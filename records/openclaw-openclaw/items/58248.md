## Summary

Keep open: this is a plausible auth/gateway feature PR, but it is not mergeable as-is because source inspection reproduces blocking auth-mode, env-secret, duplicate-export, warning-coverage, and activation-semantics defects, and the PR still lacks real behavior proof.

## What This Changes

Adds a gateway token-expiry warning config, an `openclaw auth rotate` CLI, generated config metadata/tests, and a small Matrix import-barrel cleanup.

## Best Possible Solution

Rework the feature so rotation only updates active plaintext token-mode configs, rejects env/SecretRef-managed sources, records activation semantics in the gateway process or persisted metadata, emits warnings from the shared HTTP auth path, and includes redacted real gateway proof.

## Reproduction Assessment

Yes. Source inspection of the PR head reproduces the blocking defects, and current main confirms the env-substituted `sourceConfig`, config-first credential precedence, shared HTTP auth path, and restart-gated config behavior they depend on.

## Solution Assessment

No. The feature direction may be useful, but this patch is not the narrowest maintainable solution because it changes auth mode/secret-source semantics and does not prove the live gateway behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Delete the duplicate browser origin helper:** `src/gateway/http-utils.ts:46-57`
  - body: This PR head declares `resolveHttpBrowserOriginPolicy` twice in the same module. The duplicate top-level export makes the module invalid, so gateway HTTP auth can fail before the new token behavior runs.
  - confidence: 0.95
- **[P1] Preserve the configured auth mode:** `src/commands/auth-rotate.ts:43`
  - body: Running `auth rotate` on a config using `password`, `none`, or `trusted-proxy` reaches this write and persists `mode: "token"`, silently changing the gateway auth strategy. Reject non-token modes or require an explicit mode change before writing a token.
  - confidence: 0.94
- **[P1] Reject env-managed tokens before writing config:** `src/commands/auth-rotate.ts:23-27`
  - body: This check only rejects SecretRef values visible in `sourceConfig`, which current main defines as after env substitution. Env-template or `OPENCLAW_GATEWAY_TOKEN` deployments can therefore get a new plaintext config token that becomes the credential source of truth.
  - confidence: 0.9
- **[P2] Emit expiry warnings from the shared HTTP auth path:** `src/gateway/http-auth-helpers.ts:35-40`
  - body: The warning runs only in `authorizeGatewayBearerRequestOrReply`, while normal HTTP routes use the shared HTTP auth utility path. Those common routes will not evaluate `tokenExpiryHours`, making the new config ineffective for ordinary gateway usage.
  - confidence: 0.87
- **[P2] Make rotated token activation explicit:** `src/commands/auth-rotate.ts:53-55`
  - body: This updates only CLI process memory and then prints the new token. A running gateway can still require the old in-memory token until restart, and a restarted gateway with a configured token does not get the CLI process timestamp, so users can switch too early and the warning clock can be wrong.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff changes gateway credential persistence and auth mode handling and has concrete mode-switch and secret-source risks.

Concerns:

- **[high] Token rotation can silently change auth mode:** `src/commands/auth-rotate.ts:43`
  - body: Persisting `mode: "token"` during rotation can replace password, none, or trusted-proxy auth without explicit operator consent, changing the gateway security posture and potentially breaking access.
  - confidence: 0.94

  - body: When `OPENCLAW_GATEWAY_TOKEN` or a resolved env-template token is the source of truth, the command can persist a new plaintext token into config, changing credential precedence and leaving a secret on disk unexpectedly.
