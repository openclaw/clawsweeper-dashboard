## Review Findings

Overall correctness: patch is incorrect

Full review comments:

- **[P1] Delete the duplicate browser-origin helper:** `src/gateway/http-utils.ts:65`
  - body: This file declares `resolveHttpBrowserOriginPolicy` twice. Importing the module will fail with a duplicate identifier error, breaking HTTP auth paths that load `http-utils`.
  - confidence: 0.97
- **[P1] Preserve the configured auth mode:** `src/commands/auth-rotate.ts:43`
  - body: `auth rotate` always writes `mode: "token"`, so running it against password, none, or trusted-proxy configs silently changes the gateway auth strategy. Reject non-token modes or preserve the existing mode instead of rewriting it.
  - confidence: 0.97
- **[P1] Reject env-managed gateway tokens:** `src/commands/auth-rotate.ts:24-27`
  - body: The command only checks the config token, so a deployment using `OPENCLAW_GATEWAY_TOKEN` with no config token can get a new plaintext token written to config. Since config tokens take precedence, future startups can ignore the intended env-managed credential.
  - confidence: 0.94
- **[P2] Move the expiry warning to shared HTTP auth:** `src/gateway/http-auth-helpers.ts:35-40`
  - body: The warning is emitted only from the older bearer helper, while current main routes normal HTTP auth through `checkGatewayHttpRequestAuth` in `src/gateway/http-auth-utils.ts`. Put the warning on the shared path so `tokenExpiryHours` applies to normal gateway traffic.
  - confidence: 0.88
- **[P2] Do not reset gateway expiry state in the CLI process:** `src/commands/auth-rotate.ts:53`
  - body: `setGatewayTokenIssuedAtNow()` updates memory in the short-lived CLI process, not the running gateway process that checks expiry. Remove this misleading reset or move the reset/activation into a gateway-owned reload path.
  - confidence: 0.9
