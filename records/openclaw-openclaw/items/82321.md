## Summary

This issue should stay open: current main and the latest release still do not run daily session resets on an independent wall-clock scheduler, and the open linked fix PR is the correct path to resolve it before GitHub closes the issue on merge.

## What This Changes

The issue asks for daily session reset to fire at the configured 4am wall-clock boundary even when no inbound message arrives after that time.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/84009, or an equivalent gateway-owned scheduler that reuses the existing reset service, skips protected active/provider-owned cases, and covers the no-inbound-message case.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction for the central bug: daily freshness is evaluated only when a session-use path runs, and current gateway maintenance has no independent daily reset sweep; I did not run a multi-hour live gateway reproduction in this read-only review.

## Solution Assessment

Yes, the linked PR is the best current fix direction because the reset belongs in gateway maintenance and should reuse the existing reset service rather than duplicating reset semantics. The issue should remain open until that PR, or an equivalent replacement, merges.
