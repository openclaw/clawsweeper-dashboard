## Summary

Keep open: current main and the latest release still only perform daily expiry when a non-system turn initializes the session; unattended wall-clock rollover at 4am is not implemented.

## What This Changes

The issue requests that `session.reset.mode: "daily"` with `atHour: 4` reset sessions at the configured wall-clock boundary even when no message arrives after that time.

## Best Possible Solution

Add a gateway-owned daily reset sweep that archives and rolls over eligible configured sessions at the wall-clock boundary while preserving lazy rollover on the next real interaction as a fallback.

## Reproduction Assessment

Yes, from source inspection: create a daily-reset session before 4am, let only heartbeat/cron/exec system events occur after 4am, and current main keeps the existing session because those events bypass expiry and no scheduler runs. I did not run tests because this review is read-only.

## Solution Assessment

Yes: the narrow maintainable fix is a dedicated session lifecycle sweep, not changing heartbeat or cron turns to reset sessions directly. That satisfies the documented daily reset contract while keeping the current system-event safety behavior.
