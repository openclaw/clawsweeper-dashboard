## Summary

Keep open: current main has source-level support for lazy daily expiry on the next non-system turn, but it still intentionally preserves stale daily sessions during heartbeat/cron/exec system events and has no gateway-owned wall-clock reset sweep that would fire at 4am without an incoming message.

## What This Changes

The issue asks for daily session reset to fire at the configured wall-clock hour even when no user message arrives after that time.

## Best Possible Solution

Add a gateway-owned daily reset sweep that archives and rolls over eligible configured sessions at the wall-clock boundary, while keeping heartbeat/cron/exec turns non-resetting and preserving the existing lazy reset fallback on the next real interaction.

## Reproduction Assessment

Yes from source inspection: create a daily-reset session before 4am, let a heartbeat/cron/system event run after 4am, and current main keeps the same sessionId because system events bypass expiry. I did not run Vitest because this review is read-only and tests may create artifacts.

## Solution Assessment

Yes: the maintainable fix is a dedicated lifecycle sweep, not changing heartbeat or cron message handling to reset sessions. That keeps the existing system-event safety boundary while satisfying the documented daily reset contract.
