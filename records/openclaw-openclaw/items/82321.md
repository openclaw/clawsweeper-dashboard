## Summary

Keep open: current main and v2026.5.12 still implement daily reset as lazy freshness on the next non-system turn, while heartbeat/cron/exec system events intentionally preserve the stale session and no gateway-owned wall-clock reset sweep exists.

## What This Changes

Ensure daily session reset fires at the configured wall-clock hour even when no user message arrives afterward.

## Best Possible Solution

Add a gateway-owned daily reset sweep that archives and rolls over eligible configured sessions at the wall-clock boundary, while retaining the existing lazy fallback for the next real interaction.

## Reproduction Assessment

Yes, from source inspection: create a daily-reset session before 4am and let only no event or a heartbeat/cron/exec system event occur afterward; current main has no timer path and system events preserve the stale sessionId. I did not run tests because this review is read-only.

## Solution Assessment

Yes. The narrow maintainable fix is a gateway lifecycle sweep for eligible sessions, not changing heartbeat or cron message handling to reset sessions.
