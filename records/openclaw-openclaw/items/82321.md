## Summary

Keep open: current main still appears to implement daily expiry as a lazy rollover during a later session turn, while the documented contract says daily reset happens at the configured local `atHour`. I found no gateway-owned wall-clock reset path that would rotate an idle Telegram session when no message arrives after 4am.

## What This Changes

The issue requests that `session.reset.mode: "daily"` reset sessions at the configured wall-clock hour even when no inbound message arrives.

## Best Possible Solution

Add a gateway-owned scheduled daily-expiry path that rotates expired session entries at `atHour` using the existing lifecycle, archive, and hook machinery, without adding new config or Telegram-specific behavior.

## Reproduction Assessment

Yes, at source level: docs promise `atHour` rollover, but current reset evaluation runs only during session initialization/gateway agent send paths and no background reset timer was found. I did not run a live wall-clock Telegram repro.

## Solution Assessment

No patch is proposed here. The best fix is a narrow core gateway session lifecycle service or maintenance timer, not a Telegram plugin workaround and not a change to the existing reset config contract.
