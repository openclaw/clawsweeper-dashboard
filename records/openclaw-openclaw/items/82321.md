## Summary

Keep open: current main supports lazy daily rollover on the next real interaction, but it does not contain an unconditional wall-clock reset sweep, and the requested behavior conflicts with docs that describe next-message rollover.

## What This Changes

Request daily session reset to occur at the configured wall-clock hour even when no inbound user message arrives, instead of only lazily on the next real interaction.

## Best Possible Solution

Clarify the intended reset contract, then either document lazy next-message rollover more explicitly or add a gateway-owned daily reset sweep that reuses the existing reset lifecycle safely.

## Reproduction Assessment

Yes for the no-message/system-event gap: source inspection shows only session initialization evaluates freshness, system events preserve the existing session, and no gateway reset scheduler is present. No high-confidence reproduction exists yet for the separate claim that a later real user message still keeps the old session, because current tests cover that as a lazy rollover.

## Solution Assessment

Unclear: a gateway-owned sweep is the likely implementation if maintainers want unconditional wall-clock resets, but it is not safe to solve this by making heartbeat or cron events trigger session expiry.
