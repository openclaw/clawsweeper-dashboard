## Summary

Keep open: current main supports lazy daily expiry on the next real turn, but it does not have a gateway-owned wall-clock reset sweep, and heartbeat/cron/exec events intentionally keep stale sessions unchanged.

## What This Changes

The issue asks for `session.reset.mode: "daily"` with `atHour: 4` to reset sessions at that wall-clock time even when no user message arrives after the boundary.

## Best Possible Solution

Add a gateway-owned daily reset sweep that rolls over eligible configured sessions at the wall-clock boundary using the existing reset policy and reset lifecycle helpers, while preserving the current lazy fallback on the next real interaction.

## Reproduction Assessment

Yes, from source inspection: create a daily-reset session before `atHour`, let only heartbeat/cron/internal events run after the boundary, and current main keeps the same `sessionId` because system events force reuse. I did not run Vitest because this was a read-only review.

## Solution Assessment

Yes, the best fix is a dedicated gateway lifecycle sweep rather than changing heartbeat or cron message handling to reset sessions. That preserves the established system-event safety boundary while satisfying the daily reset contract.
