## Summary

Keep open: current main still has a source-backed path where a tool-loop overflow after successful compaction can re-enter overflow compaction, and the related open issues cover adjacent failure modes rather than this successful-but-insufficient compaction loop.

## What This Changes

The issue reports repeated successful auto-compactions followed by new tool-loop context overflows in a Telegram browser-heavy session on OpenClaw v2026.5.5.

## Best Possible Solution

Handle successful-but-insufficient tool-loop compactions in the embedded runner by escalating to persisted tool-result truncation or a bounded clear failure before another silent compaction cycle, without adding new user config.

## Reproduction Assessment

Yes, at source level: current main has the tool-loop overflow throw path and a runner test that allows three successful compaction retries before surfacing overflow. A live Telegram/browser reproduction was not run in this read-only review.

## Solution Assessment

Yes, a narrow runner/guard fix is the best next step: escalate repeated post-compaction tool-loop overflow to persisted truncation or an explicit bounded diagnostic. Reworking provider fallback or adding config would be broader than this report requires.

Current main can respond to a tool-loop context overflow by successfully compacting, retrying, then immediately hitting the same tool-loop overflow path again; `run.overflow-compaction.loop.test.ts` even models three successful compactions before final overflow. Keep the fix narrow in `src/agents/pi-embedded-runner`: when a tool-loop/precheck overflow recurs after a successful compaction in the same run/session, escalate to persisted tool-result truncation or an existing compact-then-truncate recovery before another compaction retry, and after a bounded failure surface a clear context-overflow/loop diagnostic.
