## Summary

Keep open: current main still has a source-backed path where tool-loop overflow can compact successfully, retry, and overflow again; the related truncation-first PR is open but not merged or proven to cover this broader successful-compaction loop.

## What This Changes

The issue asks OpenClaw to avoid repeated successful auto-compaction/retry cycles when tool-heavy sessions keep overflowing context.

## Best Possible Solution

Land or adapt the narrow truncation-first recovery change, then verify whether successful post-compaction overflows still need a bounded escalation or clearer diagnostic.

## Reproduction Assessment

Yes, at source level. Current main has the tool-loop overflow throw path, a three-attempt overflow compaction loop, and regression coverage that allows repeated successful compactions before final overflow; I did not run a live Telegram/browser reproduction.

## Solution Assessment

Yes for the next code direction: reuse deterministic tool-result reduction for tool-heavy overflow before repeated compaction, coordinated with the open truncation-first PR. It is not yet proven as a complete fix for this broader successful-compaction loop.
