## Summary

Keep open: current main still has source-backed repeated overflow recovery paths, and recent Slack/vLLM follow-up shows the symptom is still provider-independent rather than fixed by the latest release.

## What This Changes

The issue asks OpenClaw to prevent repeated context-overflow auto-compaction loops after successful compaction in tool-heavy sessions.

## Best Possible Solution

Unify overflow recovery by rejecting impossible usage snapshots, using current-prompt or last-call usage for overflow decisions, truncating dominant tool output before repeated compaction, adding a no-real-conversation terminal breaker, and persisting compaction trigger metadata.

## Reproduction Assessment

Yes, source-reproducible but not live-reproduced here: the issue and related comments include concrete runtime logs, and current main still has the overflow classifier, retry, compaction, and fallback ordering paths that explain the symptom.

## Solution Assessment

Unclear as a single narrow patch: the requested end state is right, but the best fix should cover usage sanity, recovery ordering, no-real-conversation circuit breaking, and compaction trigger metadata together.
