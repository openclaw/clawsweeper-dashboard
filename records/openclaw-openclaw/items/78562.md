Keep open: current main has useful partial mitigations, but the reported repeated-compaction family is still not fully resolved, and the June 2026 follow-up data points to a remaining source-level path through inflated usage/context-overflow decisions.

## What This Changes

The issue asks OpenClaw to stop repeated auto-compaction loops after successful compaction in tool-heavy sessions and to provide clearer recovery or diagnostics.

## Best Possible Solution

Coordinate the overflow cluster so parent-turn usage snapshots or overflow classification cannot compact on impossible token data, then land the accepted truncation-first/recovery-order change with focused regression and live or packaged-runtime proof.

## Reproduction Assessment

No deterministic current-main reproduction is established. The live logs plus current source make the repeated-compaction path source-reproducible, especially through usage.input overflow classification and compaction-before-truncation ordering.

## Solution Assessment

Unclear as a single fix. The best path likely combines an impossible-usage guard or last-call usage repair with the open truncation-first recovery work, rather than treating any one related PR as the complete solution.
