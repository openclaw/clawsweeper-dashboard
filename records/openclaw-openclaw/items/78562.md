## Summary

Keep open: current main still has a source-backed path where tool-loop context overflow can compact successfully, retry, and then hit overflow again; the related truncation-first PR is open but not merged and does not yet prove this broader successful-compaction loop is fixed.

## What This Changes

The issue asks OpenClaw to avoid repeated auto-compaction loops after successful compaction in tool-heavy Telegram/browser sessions.

## Best Possible Solution

Land or adapt the narrow truncation-first recovery change, then verify whether successful post-compaction overflows still need a bounded escalation or clearer diagnostic.

## Reproduction Assessment

Yes, at source level. Current main has the tool-loop overflow throw path, a three-attempt overflow compaction loop, and regression coverage that allows repeated successful compactions before final overflow; I did not run a live Telegram/browser reproduction.

## Solution Assessment

Yes, the likely best direction is a narrow embedded-runner recovery change that tries deterministic tool-result reduction for tool-heavy overflow before or alongside repeated compaction, coordinated with the open truncation-first PR rather than adding new config first.
