## Summary

Keep open: current main has partial hardening, including the merged post-compaction identical-tool-result guard and optional mid-turn precheck, but the reported browser/tool-output overflow path can still run repeated successful overflow compactions before truncation or a terminal error.

## What This Changes

The issue asks OpenClaw to avoid repeated successful auto-compaction cycles after tool-heavy Telegram/browser sessions hit tool-loop context overflow.

## Best Possible Solution

Use the existing overflow-recovery and tool-result truncation seams to escalate repeated tool-loop overflows after a successful compaction into truncation or compact_then_truncate before attempting another compaction.

## Reproduction Assessment

Unclear for a full live reproduction: the v2026.5.5 logs are concrete, and current source/tests show the repeated successful-compaction path, but I did not run a live Telegram/browser replay against current main.

## Solution Assessment

No, current main is only a partial solution. The narrow maintainable fix is to reuse existing tool-result truncation/compact_then_truncate recovery when a tool-loop overflow persists after successful compaction, without changing global loop-detection defaults.
