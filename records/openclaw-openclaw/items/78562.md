## Summary

Keep open: current main still has a source-backed path where a tool-loop overflow can trigger repeated overflow compactions after prior compaction succeeds. The newer open issue and PR around truncating tool results before compaction are highly relevant, but they are not merged and do not fully close this successful-but-insufficient compaction loop yet.

## What This Changes

The issue reports repeated successful auto-compactions after tool-loop context overflows in tool-heavy Telegram sessions.

## Best Possible Solution

Land or adapt the narrow tool-result-heavy overflow recovery fix, then verify whether successful post-compaction overflows still need an explicit bounded escalation or diagnostic separate from stale preflight usage work.

## Reproduction Assessment

Yes at source level: current main has the tool-loop overflow throw path, a three-attempt overflow compaction loop, and regression coverage that allows repeated successful compactions before final overflow. I did not run a live Telegram/browser reproduction.

## Solution Assessment

Yes for the likely direction: a narrow embedded-runner/tool-result recovery change is the maintainable fix, preferably coordinated with the open truncation-before-compaction PR and related stale-usage issue rather than adding new config first.
