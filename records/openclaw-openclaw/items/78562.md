## Summary

Keep open: current main still has a source-backed path where a tool-loop context overflow after successful compaction can re-enter overflow recovery, and the related reports cover adjacent failure modes rather than replacing this issue.

## What This Changes

The issue asks OpenClaw to stop repeated post-compaction tool-loop context overflows from causing successive auto-compactions in the same live task.

## Best Possible Solution

Add a focused embedded-runner/tool-result guard fix that escalates repeated post-compaction tool-loop overflow into persisted tool-result truncation or a bounded explicit failure, covered by regression tests.

Yes, at source level: current main still has the tool-loop overflow throw path and a regression test that allows three successful compaction retries before surfacing overflow. I did not run a live Telegram/browser replay in this read-only review.

## Solution Assessment
