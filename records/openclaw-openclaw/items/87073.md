## Summary

Keep open for maintainer judgment: current main has install/update-time package dist verification and plugin payload checks, but I did not find the requested root `node_modules` dependency sentinel or `openai`/`typebox` entrypoint validation. The exact ExecStartPre reinstall pattern would change startup/package-manager behavior and needs an explicit product and ops decision.

## What This Changes

Requests a built-in or documented dependency guard that validates npm-installed `node_modules` with a version sentinel and wipes/reinstalls on detected corruption before Gateway start.

## Best Possible Solution

Make a maintainer decision on a narrow install/update/doctor-time dependency integrity check or public runbook, and avoid deleting or reinstalling `node_modules` during Gateway startup unless that policy is explicitly approved.

## Reproduction Assessment

No. The reporter gives plausible real-world evidence from a Raspberry Pi npm install, but I did not find a deterministic current-main reproduction or an existing test harness that produces the missing dependency entrypoints.

## Solution Assessment

No for the exact proposed default. A versioned sentinel may be useful, but running npm cleanup/reinstall from Gateway startup conflicts with current verify-only startup behavior; a safer maintainer-approved path is update/doctor validation or documented operator recovery.
