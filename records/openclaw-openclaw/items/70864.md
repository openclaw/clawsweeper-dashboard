## Summary

Keep open: current main does not contain the scoped mention-pattern policy, and the PR still has a source-reproducible regression where pattern-only channels can stop enforcing `requireMention` when regex patterns are policy-denied.

## What This Changes

The PR adds scoped mention-pattern policy config/helpers and SDK exports, applies allow/deny scoping across bundled channel runtimes, updates docs/generated metadata/changelog/tests, and includes post-rebase test/SSRF/loader maintenance edits.

## Best Possible Solution

Keep the shared resolver, but separate detection capability from policy-filtered regex evidence in pattern-only handlers and cover Signal, iMessage, and BlueBubbles before merge.

## Reproduction Assessment

Yes. Source-reproducible on the PR head: configure a Signal group with `requireMention`, configured `mentionPatterns`, and a deny policy; the PR empties `mentionRegexes`, which makes `canDetectMention` false and prevents the no-mention drop for unmentioned text.

## Solution Assessment

No, not yet. The resolver direction is maintainable, but callers that use configured regexes as their only detection signal need a separate configured-detection flag while denied regexes stay out of `wasMentioned`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve mention gating when regex patterns are denied:** `extensions/signal/src/monitor/event-handler.ts:686-694`

Keep denied regexes out of `wasMentioned`, but derive detection capability from whether mention detection is configured; audit the same pattern-only shape in iMessage and BlueBubbles.
