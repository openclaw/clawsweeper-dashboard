Keep this PR open: the latest head is not obsolete on main, but it still leaves a Nextcloud Talk require-mention room unreachable when the new policy disables configured regex mentions. Earlier review follow-ups fixed several cross-channel issues, but this remaining source-level regression should block merge.

## What This Changes

The branch adds scoped mention-pattern policy controls across config schemas, SDK/runtime helpers, bundled channel runtimes, generated metadata, docs, changelog, and focused tests.

## Best Possible Solution

Keep the shared scoped policy direction, but make Nextcloud Talk use the shared mention decision helper or pass an explicit canDetectMention signal so disabling configured patterns does not make requireMention rooms impossible to use.

## Reproduction Assessment

Yes for the PR finding. Source inspection of the latest head shows a Nextcloud Talk group with requireMention enabled and provider/global policy denying configured patterns gets mentionRegexes=[], wasMentioned=false, and then the custom gate skips every non-command message.

## Solution Assessment

No, not yet. The resolver and cross-channel policy shape are reasonable, but Nextcloud Talk needs the same detection-capability semantics as the shared mention gate before this is the maintainable fix.
