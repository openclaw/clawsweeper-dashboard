## Summary

Keep open. Current main does not have the scoped mention-pattern policy, and this PR is active, but the latest head still has a source-level policy gap plus red CI, so it is not a conservative cleanup close or merge candidate.

## What This Changes

The branch adds global, agent, provider, and conversation-scoped controls for configured regex mention matching across channel runtimes, SDK exports, docs, generated metadata, and tests.

## Best Possible Solution

Complete policy wiring for every configured-pattern consumer or explicitly document exclusions, clear CI, then let maintainers decide whether to accept the public config and SDK contract.

## Reproduction Assessment

Not applicable as an issue reproduction: this is a feature/config PR. For the review finding, source inspection is high-confidence because qa-channel still computes WasMentioned from configured regexes without the new policy gate.

## Solution Assessment

No: the shared resolver and SDK direction are plausible, but the current patch should also apply the policy to qa-channel configured mention matching and clear failing checks before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Gate qa-channel through the new mention policy:** `src/config/types.messages.ts:16`
  - body: Adding `messages.groupChat.mentionPatternsMode` makes configured regex mentions policy-controlled, but qa-channel inbound still builds regexes unconditionally and sets `WasMentioned` from them. A qa-channel group message can therefore count as mentioned even when the global mode is `deny`, so the new policy is not applied across this bundled channel runtime used by qa-lab. Please route qa-channel through `resolveMentionPatternsEnabled` or explicitly exclude it with docs/tests.
