## Summary

Keep this PR open: current main does not already include the scoped mention-pattern policy, and the latest PR head still has a source-level required-mention regression when configured patterns are denied.

## What This Changes

The PR adds global, agent, provider, and conversation-scoped mention-pattern policy controls, wires them through bundled channel runtimes, updates SDK/docs/generated config metadata, and includes post-rebase cleanup/test adjustments.

## Best Possible Solution

Keep the shared scoped policy shape, but separate configured-pattern contribution from mention-detection capability in channels without native mention metadata and add denied-policy plus requireMention regression coverage.

## Reproduction Assessment

Yes, from source inspection: configure a Signal group with requireMention, configured mention patterns, and a global/provider policy denial; the PR empties mentionRegexes, canDetectMention becomes false, and the missing-mention skip path cannot fire.

## Solution Assessment

No, not yet; the resolver and docs direction are maintainable, but the caller integration must not let disabled configured patterns turn off required-mention enforcement for channels with no native mention metadata.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Preserve required mention gating when patterns are denied:** `extensions/signal/src/monitor/event-handler.ts:689-697`
  - body: When policy denies configured patterns, this code sets `mentionRegexes` to `[]`, and the later gate derives `canDetectMention` from `mentionRegexes.length`. In Signal groups with `requireMention` and configured mention patterns, unmentioned messages now pass whenever the policy denies patterns; iMessage and BlueBubbles have the same call shape. Keep denied patterns from contributing to `wasMentioned`, but do not make the required-mention gate think detection is unavailable.
  - confidence: 0.87
