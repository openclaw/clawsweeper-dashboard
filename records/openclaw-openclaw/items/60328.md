## Summary

Keep this PR open: the Feishu native exec approval feature is still absent from current main, but the current branch has two source-reproducible approval handling blockers and still lacks inspectable real Feishu behavior proof.

## What This Changes

Adds a Feishu native exec approval handler, interactive card templates and callbacks, config/routing hooks, SDK/export adjustments, and Feishu approval tests.

## Best Possible Solution

Keep the feature in the Feishu plugin on the shared native approval capability, validate parsed approver IDs after parsing, either scope this handler to exec approvals or render plugin approvals correctly, then require redacted real Feishu proof before merge.

## Reproduction Assessment

Yes. The merge blockers reproduce by source inspection of PR head 7692637b4ca1 against current-main Feishu ID parsing, card callback authorization, and native approval event dispatch.

## Solution Assessment

No. The Feishu plugin and shared handler pattern are the right direction, but this branch needs parsed-ID validation, correct plugin-approval handling, and real Feishu proof before it is a maintainable merge candidate.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Validate parsed Feishu approver IDs:** `extensions/feishu/src/exec-approvals.ts:32-34`
  - body: `parseFeishuDirectConversationId()` can return any `user:`/`dm:`/`open_id:` suffix, but this function returns that parsed value before verifying it is an `ou_` open_id. A config like `approvers: ["user:foo"]` enables the native client with an approver that can never match `event.operator.open_id`, leaving card approvals unresolvable. Validate and normalize the parsed value before returning it, and add a regression test.
  - confidence: 0.93
- **[P2] Keep plugin approvals out of the exec card handler:** `extensions/feishu/src/exec-approvals-handler.ts:59`
  - body: The handler subscribes to plugin approval events, but resolved plugin approvals are always rendered through `createExecApprovalResolvedCard`. A plugin approval delivered here is replaced with a command-execution approval card and loses its plugin title/description. Either limit this handler to exec events or render plugin pending/resolved cards explicitly.
  - confidence: 0.86
