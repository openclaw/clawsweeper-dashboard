## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P1] Validate parsed Feishu approver IDs:** `extensions/feishu/src/exec-approvals.ts:32-34`
  - body: `parseFeishuDirectConversationId()` returns the suffix of `user:`, `dm:`, and `open_id:` forms before checking for an `ou_` ID. Returning that value directly means config like `user:foo` or `user:OU_123` counts as an approver and can enable the native client, but it will never match `event.operator.open_id` during button authorization. Re-check the parsed value for a valid `ou_` open_id before returning it and add coverage for invalid prefixed forms.
  - confidence: 0.91
- **[P2] Keep plugin approvals out of exec-card finalizers:** `extensions/feishu/src/exec-approvals-handler.ts:59`
  - body: The handler subscribes to `plugin` approval events, but plugin requests are delivered as a text fallback card and every resolution is rendered with `createExecApprovalResolvedCard()`. That can overwrite a plugin approval card as a command-execution approval and lose the plugin context; keep this runtime exec-only unless plugin-specific Feishu cards and finalizers are implemented.
  - confidence: 0.86
- **[P2] Use the approval resolver instead of raw gateway RPC:** `src/plugin-sdk/infra-runtime.ts:52-53`
  - body: This exposes `callGateway` and gateway client constants through the broad `infra-runtime` SDK barrel so the Feishu plugin can call `exec.approval.resolve` directly. Current main already has `openclaw/plugin-sdk/approval-gateway-runtime` for approval resolution, including plugin-id routing and the operator-approval client lifecycle; use that narrower seam instead of widening generic gateway RPC access.
  - confidence: 0.82
