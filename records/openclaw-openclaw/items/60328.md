Keep this PR open, but it is not ready to merge: current main still lacks Feishu native exec approval cards, and the current PR head still has source-reproducible approval handling blockers plus missing real Feishu behavior proof.

## What This Changes

The PR adds Feishu native exec approval interactive cards, callback RPC handling, handler runtime wiring, target routing config, SDK/export adjustments, and Feishu approval tests.

## Best Possible Solution

Keep the feature in the Feishu plugin on the shared native approval capability, validate parsed approver IDs after parsing, handle plugin approvals correctly or scope this handler to exec approvals, and require redacted real Feishu proof before merge.

## Reproduction Assessment

Yes. The merge blockers are source-reproducible on PR head 7692637b4ca1 by following parsed Feishu approver IDs into open_id callback authorization and by following plugin approval events into the Feishu handler's exec-card resolution path.

## Solution Assessment

No. The Handler pattern and plugin-owned implementation direction are appropriate, but the branch needs parsed-ID validation, correct plugin-approval behavior, and real Feishu proof before it is a maintainable merge candidate.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Validate parsed Feishu approver IDs:** `extensions/feishu/src/exec-approvals.ts:32-34`
  - body: `parseFeishuDirectConversationId()` can return any `user:`/`dm:`/`open_id:` suffix, but this function returns that parsed value before checking it is an `ou_` open_id. A config such as `approvers: ["user:foo"]` enables the native client with an approver that can never match `event.operator.open_id`, leaving card approvals unresolvable; validate the parsed value before returning it and add a regression test.
  - confidence: 0.93
- **[P2] Keep plugin approvals out of the exec card handler:** `extensions/feishu/src/exec-approvals-handler.ts:59`
  - body: The handler subscribes to `eventKinds: ["exec", "plugin"]`, but resolved plugin approvals are always rendered through `createExecApprovalResolvedCard`. A plugin approval delivered here can be replaced with a command-execution approval card and lose its plugin title/description; either limit this handler to exec events or render plugin pending/resolved cards explicitly.
  - confidence: 0.86
