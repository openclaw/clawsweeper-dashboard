## Summary

Keep open: this is the canonical Feishu native exec approval PR and the feature is still not on current main, but PR head still has source-proven approval-flow blockers and lacks required real behavior proof.

## What This Changes

The branch adds Feishu native exec approval interactive cards, card-action RPC resolution, handler/runtime wiring, target-routing config, SDK exports, and Feishu approval tests.

## Best Possible Solution

Keep the implementation in the Feishu plugin on the shared native approval capability, validate parsed approver IDs as ou_ open_ids after parsing, scope or render plugin approvals correctly, and require redacted real Feishu proof before merge.

## Reproduction Assessment

Yes. The blockers are source-reproducible at PR head by tracing parsed Feishu approver IDs into open_id callback authorization and tracing plugin approval events into exec-card resolution.

## Solution Assessment

No. The Handler pattern and plugin-owned direction are appropriate, but the branch needs parsed-ID validation, correct plugin-approval behavior, and real Feishu proof before it is a maintainable merge candidate.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Validate parsed Feishu approver IDs:** `extensions/feishu/src/exec-approvals.ts:32-34`
  - body: `parseFeishuDirectConversationId()` can return any `user:`/`dm:`/`open_id:` suffix, and it also accepts bare `on_` IDs, but this function returns that parsed value before checking that it is an `ou_` open_id. A config such as `approvers: ["user:foo"]` or `approvers: ["on_789"]` can enable the native client with an approver that can never match `event.operator.open_id`, leaving card approvals unresolvable; validate the parsed value before returning it and add regression coverage for prefixed invalid IDs.
  - confidence: 0.94
- **[P2] Keep plugin approvals out of the exec card handler:** `extensions/feishu/src/exec-approvals-handler.ts:59`
  - body: The handler subscribes to `eventKinds: ["exec", "plugin"]`, but resolved plugin approvals are always rendered through `createExecApprovalResolvedCard`. A plugin approval delivered here can be replaced with a command-execution approval card and lose its plugin title/description; either limit this handler to exec events or render plugin pending/resolved cards explicitly.
  - confidence: 0.87
