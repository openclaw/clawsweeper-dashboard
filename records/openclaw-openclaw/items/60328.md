## Summary

Keep open. The feature is useful and not implemented on current `main`, but the PR head still has narrow correctness blockers around Feishu approver normalization and plugin-approval rendering, plus the required user-facing changelog entry is missing.

## What This Changes

The PR adds Feishu native exec approval cards, handler-based delivery and card update paths, approval config, card action resolution, SDK/export support, and related tests.

## Best Possible Solution

Keep the feature in the Feishu plugin on the generic native approval seams, validate only resolvable Feishu `ou_` open IDs as approvers, keep plugin approvals on plugin-specific rendering or out of this handler, and document the user-facing config/UX in the changelog.

## Reproduction Assessment

Yes. The remaining defects reproduce by static inspection of PR head `7692637b`: `user:foo` is accepted as an approver while callbacks authorize `event.operator.open_id`, and plugin approval events flow into an exec resolved-card renderer.

## Solution Assessment

No. The Handler pattern is the right direction, but the current patch is not the best mergeable solution until the normalization, plugin-event rendering, and changelog gaps are fixed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Validate parsed Feishu approver IDs:** `extensions/feishu/src/exec-approvals.ts:32-34`
  - body: `normalizeFeishuDirectApproverId` returns the parsed `user:`/`dm:`/`open_id:` suffix before checking that it is an `ou_` open_id. A config such as `approvers: ["user:foo"]` therefore enables the native client with an approver that can never match `event.operator.open_id`, leaving card approvals unresolvable. Validate the parsed value before returning it and add a regression test.
  - confidence: 0.92
- **[P2] Keep plugin approvals out of the exec card handler:** `extensions/feishu/src/exec-approvals-handler.ts:59`
  - body: The handler subscribes to plugin approvals, but `finalizeResolved` always builds a Feishu exec approval resolved card. A plugin approval delivered through this path is replaced with a command-execution approval card and loses its plugin title/description. Either limit this handler to exec events or render plugin-specific pending/resolved cards.
  - confidence: 0.84
- **[P3] Add the required changelog entry:** `extensions/feishu/src/config-schema.ts:141-149`
  - body: This adds user-facing Feishu approval buttons and a new `channels.feishu.execApprovals` config surface, but the PR does not update `CHANGELOG.md`. Repo policy requires a user-facing `feat` change to add an active-version changelog entry with appropriate contributor credit.
  - confidence: 0.94
