## Best Possible Solution

Land Feishu-owned native approval cards on the current shared approval capability/runtime seam with strict `ou_` approver normalization, correct exec/plugin rendering, docs/config alignment, focused regression coverage, and redacted real Feishu proof.

## Reproduction Assessment

## Solution Assessment

No. The Feishu-plugin direction is appropriate, but this branch should fix approver validation, handle or exclude plugin approval events correctly, rebase onto the current approval-runtime seams, and include real Feishu proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Validate parsed Feishu approver IDs:** `extensions/feishu/src/exec-approvals.ts:32-34`
  - body: `parseFeishuDirectConversationId()` can return prefixed suffixes like `user:foo` and bare `on_...`, but this branch returns that parsed value before enforcing the `ou_` open_id check used by Feishu card callbacks. Invalid entries can enable the native client with approvers that can never match `event.operator.open_id`, leaving approvals unresolvable; validate the parsed value before returning it and add coverage for prefixed invalid IDs.
  - confidence: 0.94
- **[P2] Keep plugin approvals out of exec cards:** `extensions/feishu/src/exec-approvals-handler.ts:59`
  - body: Subscribing this handler to `plugin.approval.*` means plugin approval prompts are delivered here, but the finalizers always call `createExecApprovalResolvedCard()`. Resolving or expiring a plugin approval can replace the plugin prompt with an exec approval result and drop the plugin title/description; either subscribe only to exec events or render plugin resolved/expired cards explicitly.
  - confidence: 0.87
