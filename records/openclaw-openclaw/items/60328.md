Full review comments:

- **[P1] Validate parsed approver IDs before enabling Feishu approvals:** `extensions/feishu/src/exec-approvals.ts:33-34`
  - body: `parseFeishuDirectConversationId` returns any suffix for `user:`, `dm:`, or `open_id:` before the `ou_` check here. That means `user:foo` can count as a configured approver and enable the native handler, but Feishu callbacks compare against `event.operator.open_id`, so no real approver can resolve the card.
  - confidence: 0.92
- **[P2] Do not finalize plugin approvals with exec cards:** `extensions/feishu/src/exec-approvals-handler.ts:59`
  - body: Subscribing this handler to plugin approvals makes the runtime deliver plugin requests too, but `finalizeResolved` always builds a command-exec resolved card. A plugin approval can therefore be replaced in-place with the wrong approval type and without the plugin title/description; either render plugin-specific cards or subscribe only to exec events.
  - confidence: 0.88
- **[P2] Use the focused approval gateway runtime:** `src/plugin-sdk/infra-runtime.ts:52-53`
  - body: This exposes raw `callGateway` plus backend client constants through the deprecated `infra-runtime` SDK subpath for one approval button flow. Existing channel buttons use `resolveApprovalOverGateway` from `approval-gateway-runtime`, which keeps the public plugin seam scoped to approval resolution instead of broad gateway RPC access.
  - confidence: 0.86
