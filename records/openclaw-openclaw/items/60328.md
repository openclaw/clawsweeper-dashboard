Full review comments:

- **[P1] Validate parsed approver IDs before accepting them:** `extensions/feishu/src/exec-approvals.ts:33-34`
  - body: `parseFeishuDirectConversationId` returns arbitrary `user:`/`dm:`/`open_id:` suffixes before any `ou_` validation, so values like `user:foo` or `user:OU_123` become configured approvers. That can enable the native approval client with ids that will never equal Feishu `event.operator.open_id`, leaving card approvals unresolvable for those configs.
  - confidence: 0.93
- **[P1] Use the focused approval gateway resolver:** `src/plugin-sdk/infra-runtime.ts:52-53`
  - body: Publishing `callGateway` and gateway client constants through `openclaw/plugin-sdk/infra-runtime` expands a deprecated broad SDK barrel for this feature. Current main already exposes `openclaw/plugin-sdk/approval-gateway-runtime` and Discord/Telegram approval callbacks use `resolveApprovalOverGateway`, which preserves the narrow approval-only gateway contract.
  - confidence: 0.88
- **[P2] Render plugin approvals with plugin-specific state:** `extensions/feishu/src/exec-approvals-handler.ts:59-60`
  - body: The handler subscribes to `plugin` approval events, but plugin pending requests are delivered as a generic text card and all resolutions are finalized through `createExecApprovalResolvedCard`. Non-exec approvals can therefore be overwritten with an exec approval title and misleading result labels instead of preserving plugin context.
  - confidence: 0.82
