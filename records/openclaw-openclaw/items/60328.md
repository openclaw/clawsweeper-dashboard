## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Validate parsed Feishu approver IDs before returning them:** `extensions/feishu/src/exec-approvals.ts:33-34`
  - body: `parseFeishuDirectConversationId` accepts any `user:`/`dm:`/`open_id:` suffix, so this early return lets values like `user:foo` count as configured approvers. That can enable the native approval client and send cards that no Feishu `event.operator.open_id` can ever authorize; validate the parsed value is an `ou_` open_id before returning it and add a regression case for prefixed invalid IDs.
  - confidence: 0.93
- **[P1] Use the focused approval gateway resolver:** `src/plugin-sdk/infra-runtime.ts:52-53`
  - body: The PR exports raw `callGateway` plus backend gateway constants from the broad `infra-runtime` SDK barrel, but current main already has `openclaw/plugin-sdk/approval-gateway-runtime` with `resolveApprovalOverGateway` for exactly this approval-only path. Keeping the broad export expands the plugin-facing gateway RPC surface and duplicates the approval resolution contract; import the focused resolver instead.
  - confidence: 0.89
- **[P2] Align Feishu exec approval enabled mode with native clients:** `extensions/feishu/src/config-schema.ts:145`
  - body: The new Feishu schema only accepts boolean `enabled`, while the shared native approval helpers and sibling channel docs/tests support `enabled: "auto"` for native approval clients. Users following the generic native approval docs would get config rejection for Feishu, so either support the same `"auto"` mode here or document and test an intentional Feishu-only exception.
  - confidence: 0.82
