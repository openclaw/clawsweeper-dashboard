## Summary

Keep this PR open, but it is not ready to merge. The conformance-harness direction is useful, yet the branch is stale against current `main`: the new test imports old Pi/typebox package names, and the client-tool abort check can reject after the client has already been notified.

## What This Changes

The PR adds a Pi tool-adapter conformance matrix and tests, documents the harness, and adds abort-signal fast-fail checks around Pi tool execution and client delegation.

## Best Possible Solution

Rebase the harness onto current Pi packages and recorder APIs, keep abort pre-checks before agent execution/client delegation, and avoid checking abort after client notification.

## Reproduction Assessment

Yes for the patch review: current `main` dependency declarations and adjacent tests prove the new test imports stale packages, and the client-notification abort edge is visible directly in the diff. I did not run the PR branch because this was a read-only review.

## Solution Assessment

No. The harness direction is maintainable, but the current patch should update stale imports and adjust the client-tool abort ordering before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Use current Pi and typebox imports:** `src/agents/pi-tool-definition-adapter.conformance.test.ts:1-2`
  - body: Current `main` depends on `@earendil-works/pi-agent-core` and `typebox`, and the adjacent Pi tests import those packages. This new test imports `@mariozechner/pi-agent-core` and `@sinclair/typebox`, which are not current dependencies, so the conformance suite will fail to resolve after merge. Update the imports to the current package names.
  - confidence: 0.96
- **[P2] Do not abort after notifying client tools:** `src/agents/pi-tool-definition-adapter.ts:242-244`
  - body: This re-check runs after `onClientToolCall` has synchronously notified the client. If that callback aborts its controller as a side effect, the adapter rejects with `AbortError` even though the client already received the delegation, so Pi can observe a failed tool call while the client is processing it. Drop this post-notification check or move the last check before notification.
  - confidence: 0.84
