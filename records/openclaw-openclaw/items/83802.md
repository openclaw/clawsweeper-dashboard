## What This Changes

This PR changes Telegram text-fragment and media-group buffering from process-wide flush chains to per chat/topic buffer-key queues and adds handler e2e coverage for independent forum-topic flushes.

## Best Possible Solution

Land the per-key queueing fix after redacted live Telegram topic proof and normal CI confirm buffered text fragments and same-id media groups in separate forum topics do not block or merge.

## Reproduction Assessment

Source-reproducible: current main uses global media/text buffer promise chains, and the PR reports the new topic-parallel tests fail on unpatched main with only one reply start. I did not run tests because this review is read-only.

## Solution Assessment

Yes, the implementation is the right narrow boundary: queue by the existing buffer identity so same-buffer ordering is preserved while independent topics can flush concurrently. The remaining gap is proof, not a different code direction.
