## Summary

Keep this PR open: it is addressing a real gateway session/provider routing bug with useful proof, but the current patch still misses provider-qualified `modelOverride` entries that do not also have `providerOverride`, so one persisted stale-provider route shape can survive the new catalog validation.

## What This Changes

The PR adds catalog/config-aware validation to gateway session model resolution and wires `chat.send` to load catalog visibility only for risky persisted session routes.

## Best Possible Solution

Before merge, cover provider-qualified modelOverride-only persisted routes in the catalog-load predicate and resolver tests, then keep the current fallback-to-default behavior as the intentional stale-provider policy.
