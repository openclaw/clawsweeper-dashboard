## Label Justifications

- `P1`: The PR addresses provider/auth routing for live chat traffic where stale persisted session state can continue using a removed provider.
- `merge-risk: 🚨 compatibility`: The patch changes upgrade behavior for existing sessions whose persisted provider/model no longer appears in current config/catalog.
- `merge-risk: 🚨 auth-provider`: Fallback-to-default semantics can route requests through a different provider/auth profile than the previously persisted session route.
- `merge-risk: 🚨 availability`: The new pre-registration catalog await can make stale-provider sends unabortable or unobservable while catalog loading is slow.

## Summary

Keep open: this PR addresses the linked stale-provider routing bug with strong after-fix proof, but the current `chat.send` integration awaits catalog validation before in-flight/abort registration, so a slow stale-provider catalog path can leave the send unobservable and uncancellable before ACK.

## What This Changes

The PR adds catalog/config-aware validation for persisted gateway session provider/model references in `resolveSessionModelRef`, wires `chat.send` to load catalog visibility when needed, and adds gateway/session resolver regressions.

## Best Possible Solution

Keep the resolver-level stale-route guard, but preserve `chat.send` pre-ACK cancellation/in-flight semantics and leave doctor/reset cleanup as a separate maintainer policy decision.

## Reproduction Assessment

Yes. Current-main source shows `resolveSessionModelRef` returns persisted provider/model fields before visibility checks, and `chat.send` uses that resolved provider for live routing.

## Solution Assessment

No, not as currently ordered. The resolver guard is the right layer, but the `chat.send` wiring should preserve existing pre-ACK cancel and duplicate-send semantics before awaiting catalog validation.
