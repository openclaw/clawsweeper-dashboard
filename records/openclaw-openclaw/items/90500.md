## Summary

Keep open: the stale-provider resolver direction is useful and has focused proof, but this PR still moves `/stop` ahead of session send-policy enforcement, which would let denied sessions abort runs instead of returning the configured policy denial.

## What This Changes

The branch adds catalog-aware validation to persisted session model/provider resolution and wires `chat.send` to load the model catalog only when persisted routes need visibility checks.

## Best Possible Solution

Keep the stale-provider invalidation approach, but enforce send policy before `/stop`, leave catalog loading after `/stop`, and make the upgrade/fail-open routing behavior an explicit maintainer-accepted choice.

## Reproduction Assessment

Yes. Source inspection shows the PR head handles `/stop` before `resolveSendPolicy`, while current main enforces send policy before stop handling; a denied-session `/stop` regression would reproduce the failure path.

## Solution Assessment

No. The resolver guard is a plausible and useful fix for stale deleted-provider routes, but the `chat.send` ordering is not the best final shape because policy denial should run before `/stop` while catalog loading can still stay after `/stop`.
