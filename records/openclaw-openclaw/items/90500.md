## Label Justifications

- `P1`: The PR addresses a provider/auth routing bug where stale persisted session state can send live chat traffic through a removed provider.
- `merge-risk: 🚨 compatibility`: The patch changes upgrade-visible behavior for existing sessions whose persisted provider/model no longer appears in current config/catalog.
- `merge-risk: 🚨 auth-provider`: Fallback-to-default semantics can route requests through a different provider/auth profile than the previously persisted session route.
- `merge-risk: 🚨 security-boundary`: Provider routing determines where prompt content and credentials-backed traffic are sent, so stale-route fallback policy is security-sensitive even without secret-handling code changes.

## Summary

Keep open: current main still resolves persisted provider/model state before checking current catalog or config visibility, and this PR is the live focused fix path with no blocking code findings. The remaining question is maintainer acceptance of the upgrade-visible provider fallback policy.

## What This Changes

The PR adds catalog/config-aware validation for persisted gateway session provider/model routes and wires `chat.send` to load catalog visibility only when stale persisted routes need validation.

## Best Possible Solution

Land the focused runtime guard if maintainers accept fallback-to-default semantics, and track doctor/reset-session cleanup separately.

## Reproduction Assessment

Yes. Source inspection on current main shows persisted provider/model state wins before visibility checks in `resolveSessionModelRef`, and `chat.send` consumes that resolved provider for live routing.

## Solution Assessment

Yes, if maintainers accept the policy tradeoff. Resolver-level validation plus conditional `chat.send` catalog visibility is the narrow runtime fix; doctor/reset cleanup is a separate product scope.
