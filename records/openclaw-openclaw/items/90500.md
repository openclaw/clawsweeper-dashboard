## Label Justifications

- `P1`: The underlying bug can silently route real chat traffic through a removed provider, affecting billing/auth routing for active users.
- `merge-risk: 🚨 compatibility`: Changing how persisted session overrides are honored can alter behavior for existing upgraded sessions and stale session state.

- `merge-risk: 🚨 session-state`: The behavior depends on persisted session override fields and can leave stored session state diverging from the effective runtime route.

## Summary

Keep open: the patch still validates the session model in the gateway path without proving that the actual auto-reply model execution uses the same validated provider/model, and the supplied proof is resolver/test output rather than a real gateway turn.

## What This Changes

The PR adds catalog-aware validation of persisted session provider/model state in `resolveSessionModelRef`, conditionally loads catalog visibility from `chat.send`, and adds gateway/session regression tests.

## Best Possible Solution

Use one canonical validated model-selection decision for both gateway metadata and actual auto-reply provider dispatch, add a regression that observes the selected provider/model in the real reply path after provider deletion, and document or explicitly decide the runtime-versus-doctor behavior.

## Reproduction Assessment

Yes at source level: current main returns persisted overrides before current config/catalog validation in the gateway helper, and the auto-reply model-selection path can reapply stored overrides when visibility is allow-any. I did not run tests or live commands because this was a read-only review.

## Solution Assessment

No: validating `resolveSessionModelRef` inside `chat.send` is only a plausible partial fix because the actual model run is selected later by the auto-reply path. The safer fix is to feed the validated decision into reply execution or validate stored overrides where they are consumed.
