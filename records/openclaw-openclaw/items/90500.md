## Label Justifications

- `P1`: The PR addresses a provider/auth routing regression where stale persisted session state can send live chat traffic through a removed provider.
- `merge-risk: 🚨 compatibility`: The patch changes upgrade-visible behavior for existing sessions whose persisted provider/model no longer appears in current config/catalog.
- `merge-risk: 🚨 auth-provider`: Fallback-to-default semantics can route requests through a different provider/auth profile than the previously persisted session route.
- `merge-risk: 🚨 security-boundary`: Provider routing determines where prompt content and credentials-backed traffic are sent, so the stale-route fallback policy is security-sensitive.

## Summary

Keep open: this is the active linked fix for stale deleted-provider session routing, and the patch appears correct and well-proven, but it intentionally changes provider-routing behavior for existing sessions and needs maintainer acceptance before merge.

## What This Changes

The PR adds catalog/config-aware session model-route validation in the gateway, wires `chat.send` to load catalog visibility only when persisted routes need validation, moves `/stop` ahead of that load, and adds resolver/chat regressions.

## Best Possible Solution

Land this focused runtime guard if maintainers accept fallback-to-default semantics, and track doctor/reset-session cleanup as separate product scope.

## Reproduction Assessment

Yes. Source inspection on current main shows persisted session provider/model state wins before visibility checks, and `chat.send` consumes that route for provider/auth selection.

## Solution Assessment

Yes, if maintainers accept the policy tradeoff. Resolver-level validation plus conditional `chat.send` catalog visibility is the narrow runtime fix; doctor/reset cleanup is separate product scope.
