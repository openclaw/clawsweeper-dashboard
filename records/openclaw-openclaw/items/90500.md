## Summary

This PR should stay open because the central stale-provider routing bug is still not fixed in the actual reply-run model selection path, and one persisted model-only shape still bypasses catalog validation.

## What This Changes

The PR adds catalog-aware checks to gateway session model resolution, wires `chat.send` to load catalog visibility for some persisted session routes, and adds gateway/session regression tests.

## Best Possible Solution

Apply stale-provider validation in the shared reply model-selection path that actually chooses the run provider/model, cover provider-qualified model-only persisted state, and keep valid configured or catalog-backed selections with regression proof.

## Reproduction Assessment

Yes. Source inspection shows current main honors persisted session provider/model fields before catalog validation, and the PR still lets the reply-run path read those fields directly.

## Solution Assessment

No. The pre-dispatch resolver guard is a plausible mitigation, but the best fix needs to validate or clear stale persisted route state at the shared model-selection point that actually starts the agent run.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Route the validated model into the reply run:** `src/gateway/server-methods/chat.ts:3030-3032`
  - body: This computes a catalog-validated `resolvedSessionModel`, but the value is only used for gateway metadata, prompt cache, abort bookkeeping, and attachment checks. The actual agent dispatch still calls `dispatchInboundMessage` with the original session entry, and `getReply`/model-selection continues to honor `providerOverride`/`modelOverride` from that entry, so a session with the deleted provider can still run on the stale provider while the gateway thinks it selected the default.
  - confidence: 0.92
- **[P1] Validate provider-qualified model-only refs:** `src/gateway/session-utils.ts:1714-1715`
  - body: The catalog-load predicate only checks override refs when both `providerOverride` and `modelOverride` are present, and runtime refs only when both `modelProvider` and `model` are present. A persisted `modelOverride` or `model` like `ollama-cloud/deepseek-v4-pro:cloud` without the separate provider field is later parsed by `resolveSessionModelRef`, but this helper returns false, so chat.send skips catalog validation and the deleted-provider route survives.
  - confidence: 0.9
