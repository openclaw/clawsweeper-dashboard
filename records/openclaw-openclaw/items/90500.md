## Summary

Keep this PR open, but do not merge it yet: it addresses a real stale-provider session bug, but the current patch still validates only the `chat.send` pre-dispatch metadata path while the actual reply execution path can select the persisted stale override.

## What This Changes

Adds catalog-aware gateway session model resolution, conditionally loads model catalog visibility in `chat.send`, moves `/stop` ahead of model validation, and adds gateway/session regression tests.

## Best Possible Solution

Move deleted-provider invalidation into the shared runtime model-selection path used before every agent run, derive `chat.send` metadata from that same selected ref, and handle required catalog-load failure explicitly.

## Reproduction Assessment

Yes, source-reproducible: current main returns persisted overrides before validating config/catalog visibility, and the auto-reply path applies stored overrides during execution. I did not run a live turn because this review is read-only.

## Solution Assessment

No. The current PR is a plausible helper-level mitigation, but the best fix is to validate stale providers in the shared execution model-selection path and then keep gateway metadata aligned with that result.
