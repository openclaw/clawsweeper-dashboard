## Summary

Keep this PR open, but do not merge yet: the new catalog-aware resolver can return the safe default, but the actual reply execution path still re-reads the stale persisted session override, so the central deleted-provider routing bug can survive.

## What This Changes

The PR adds catalog-aware validation around persisted session provider/model state in `resolveSessionModelRef`, conditionally loads model catalog visibility from `chat.send`, and adds gateway/session regression tests.

## Best Possible Solution

Make the reply model-selection boundary consume the same validated provider/model decision as `chat.send`, preserve valid configured and catalog-backed providers, and prove the real agent dispatch selects the fallback default after provider deletion.

## Reproduction Assessment

Yes, from source inspection: a session entry with stale `providerOverride`/`modelOverride` can still be read by `getReplyFromConfig` even after `chat.send` computes a fallback default. I did not run a live gateway turn in this read-only review.

## Solution Assessment

No. Validating in `chat.send` is a plausible part of the fix, but it is not the best complete fix until the validated decision reaches the actual reply execution boundary.
