## Summary

Keep open: the PR is pointed at a real P1 session/provider routing bug, but the current patch validates a model in `chat.send` metadata while the actual reply dispatch path can still read and use the stale persisted session override. It also keeps the new catalog lookup ahead of idempotent retry fast paths for stale sessions.

## What This Changes

The PR adds catalog-aware validation to `resolveSessionModelRef`, wires `chat.send` to load catalog visibility for risky persisted session routes, and adds gateway/session regression coverage for deleted-provider session state.

## Best Possible Solution

Put the stale-provider visibility check in the canonical reply/session model-selection path used by `chat.send` and channel turns, preserve configured or catalog-backed valid routes, keep retry fast paths ahead of catalog discovery, and require redacted proof from an actual gateway turn.

## Reproduction Assessment

Yes. Source inspection shows the remaining patch bug: `chat.send` computes a validated model, but the dispatched reply still resolves model selection from persisted session overrides.

## Solution Assessment

No. This is only a partial mitigation; the best fix is to share the catalog/config visibility guard with the actual reply model-selection path and prove the selected provider after dispatch.
