## Summary

Keep open: the PR addresses the right stale-provider bug area, but the validated route is not yet applied to the actual auto-reply run selector, and required catalog-validation failures still fail open.

## What This Changes

The PR adds catalog-aware stale session provider validation in `resolveSessionModelRef`, wires `chat.send` to load catalog visibility when needed, moves `/stop` before that validation, and adds gateway/session regression tests.

## Best Possible Solution

Make stale-provider invalidation canonical in the auto-reply model-selection path, define the catalog-load failure policy, and prove a real `chat.send` or agent-run path selects the fallback after provider deletion.

## Reproduction Assessment

Yes. Current main source shows `resolveSessionModelRef` returns persisted overrides before visibility validation, and the PR source still leaves the real auto-reply run selector on its independent persisted-override path.

## Solution Assessment

No. This is a plausible partial fix, but the best fix is to make the actual run selector and gateway metadata share the same validated provider/model decision with an explicit catalog-load failure policy.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Feed the validated session model into the run selector:** `src/gateway/server-methods/chat.ts:3030-3032`
  - body: `resolvedSessionModel` is computed here, but the real run still enters `dispatchInboundMessage` with only `ctx` and `cfg`; `getReplyFromConfig` then calls `createModelSelectionState`, which rereads `sessionEntry.providerOverride`/`modelOverride` and can allow it when no model allowlist is configured. That means the agent run can still use the deleted provider while gateway metadata/prompt-cache/auth tracking reports the fallback/default provider. Please make the auto-reply selector consume the same validated route, or move this stale-route validation into that canonical selector, and assert the selected run provider/model.
  - confidence: 0.93
- **[P2] Do not fail open when required catalog validation fails:** `src/gateway/server-methods/chat.ts:3023`
  - body: When `shouldLoadModelCatalogForSessionModelResolution` says catalog visibility is needed, this `catch(() => undefined)` turns a catalog-load failure into the same state as no catalog. In merge mode, `resolveSessionModelRef` then allows non-configured providers, so a deleted-provider session can still route through the stale provider exactly when validation failed. Please make this policy explicit and tested, for example fallback to the configured default or return a clear send error when required validation cannot load.
  - confidence: 0.88
