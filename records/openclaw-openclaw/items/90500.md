## Summary

Keep open: this is a useful gateway/session routing fix, but it is not merge-ready because the new catalog-load predicate still misses provider-qualified `modelOverride` entries that have no `providerOverride`, leaving one supported stale-session shape able to route through a removed provider.

## What This Changes

The PR adds catalog-aware validation to gateway session model resolution, conditionally loads the model catalog during `chat.send`, moves `/stop` ahead of model resolution, and adds gateway/session regression tests.

## Best Possible Solution

Land a catalog-aware chat-routing fix after the predicate covers every supported persisted selected-model shape and maintainers explicitly accept the removed-provider fallback semantics.

Yes, at source level: current main resolves provider-qualified or explicit persisted session overrides before checking current provider visibility, and the PR body includes focused terminal proof for the removed-provider path. I did not run tests in this read-only review.

## Solution Assessment

No, not yet: the resolver is the right layer for the chat-routing mitigation, but the catalog-load predicate misses the existing provider-qualified `modelOverride`-only shape. The safer fix is to feed every supported persisted selected-model ref into the predicate while preserving the no-catalog fast paths.
