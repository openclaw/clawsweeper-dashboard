## Summary

This PR is still necessary for the linked provider-routing bug, but the current patch is not merge-ready: the sanitized route is not fed into the actual reply/model-selection path, and required catalog validation still fails open when catalog loading rejects.

## What This Changes

The PR adds catalog-aware validation to session model resolution, wires `chat.send` to call it for risky persisted routes, and adds resolver/chat-send regression tests for stale provider overrides.

## Best Possible Solution

Use one canonical route-validation path that either repairs or sanitizes invalid session model overrides before dispatch, or passes an immutable validated selection into the reply/agent run selector, and make required catalog-load failure visibly fail or fallback to a safe default instead of trusting stale state.

## Reproduction Assessment

Yes, this is source-reproducible: current and PR-head dispatch still let the reply/model-selection pipeline read the original session entry, while the no-allowlist visibility policy can allow stored provider overrides.

## Solution Assessment

No, not yet; the resolver guard is a plausible part of the fix, but it is not the best complete fix until the validated route controls the actual run selector and required catalog-load failure has non-stale behavior.
