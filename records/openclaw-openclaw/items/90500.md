## Summary

Keep open: the direction is useful, but the patch still does not feed the catalog-validated provider/model into the auto-reply execution path, and the supplied proof does not show a real gateway turn selecting the post-fix provider.

## What This Changes

The branch adds catalog-aware validation around persisted session provider/model routes in gateway chat.send and adds resolver/chat regression coverage for removed-provider fallbacks.

## Best Possible Solution

Move removed-provider invalidation into the actual getReplyFromConfig/model-selection path or pass an explicit validated selection through chat.send, keep idempotent/stop fast paths ahead of catalog loading, and get maintainer signoff on runtime-only handling of stale sessions.json.

## Solution Assessment

No. The resolver guard is a plausible helper, but the best fix must reach the auto-reply model-selection path that actually selects the provider for the agent run.
