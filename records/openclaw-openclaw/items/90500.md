Full review comments:

- **[P1] Apply stale-provider validation to the run model:** `src/gateway/server-methods/chat.ts:3030-3032`
  - body: This computes a sanitized `resolvedSessionModel` for `chat.send`, but the dispatched context still carries the original session key and the auto-reply path rebuilds the actual run model from `sessionEntry.providerOverride/modelOverride`. With default allow-any visibility, that persisted override remains allowed, so the agent run can still use the deleted provider while the surrounding metadata says it fell back. Feed the sanitized selection into the canonical run model-selection path or clear/ignore the stale entry before dispatch, and cover the actual run provider in a regression.
  - confidence: 0.92
- **[P2] Validate provider-qualified model-only overrides:** `src/gateway/session-utils.ts:1703-1712`
  - body: The catalog-load predicate only considers persisted overrides when both `providerOverride` and `modelOverride` are present. A legacy/store entry with only `modelOverride: "deleted-provider/model"` skips catalog validation, then `resolveSessionModelRef` falls back to merge-mode allow-any behavior and can still reuse the removed provider. Parse provider-qualified model-only overrides before deciding whether validation is needed.
  - confidence: 0.88
- **[P2] Do not fail open after required catalog validation fails:** `src/gateway/server-methods/chat.ts:3021-3023`
  - body: When `shouldLoadModelCatalogForSessionModelResolution` returns true, the catalog is required to prove whether the persisted provider still exists. Catching `loadGatewayModelCatalog` to `undefined` makes `currentConfigAllowsPersistedModelRef` treat the catalog as unavailable and preserve the stale route in default merge mode. Return a distinct failed-required-validation state and choose a tested fallback or error policy.
  - confidence: 0.86
