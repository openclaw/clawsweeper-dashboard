## Summary

Keep open: the PR has the right direction, but the current patch is not merge-ready because the catalog-aware fallback is computed in `chat.send` bookkeeping and does not reach the downstream reply/run model-selection path that can still apply the stale persisted override.

## What This Changes

This PR adds catalog-aware validation to gateway session model resolution, conditionally loads catalog visibility from `chat.send`, moves `/stop` before catalog validation, and adds resolver/chat regression coverage.

## Best Possible Solution

Move the stale-provider validation into the actual reply model-selection path, or pass a validated selection through that path, then prove an end-to-end chat send falls back to the current default before closing the linked bug.

## Reproduction Assessment

Yes for the review finding from source: `chat.send` computes a validated model ref, but `dispatchInboundMessage` later follows `get-reply` and model-selection code that can reapply the stored override. I did not run a live gateway turn.

## Solution Assessment

No: validating only `chat.send`'s pre-dispatch resolver is too shallow because the actual reply run owns final provider/model selection. The safer fix is to validate stale stored refs where reply model selection applies them, with `chat.send` using that same resolved state.
