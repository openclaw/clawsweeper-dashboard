## Summary

Keep open: this is a useful gateway/provider-routing bug fix, but the current predicate still skips catalog validation when the deleted provider was the last custom provider and the current config has no `models.providers` rows, so stale session routing can survive.

## What This Changes

The PR changes gateway chat session model resolution to validate persisted session provider/model fields against current config/catalog visibility and adds resolver regression coverage.

## Best Possible Solution

Before merge, make persisted session validation catalog-aware whenever a stored session provider is non-default and absent from current config, while preserving valid built-in and catalog-backed merge-mode providers.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: current main returns persisted override/runtime provider fields before checking whether that provider is still configured or catalog-visible. I did not run tests because this review is read-only.

## Solution Assessment

No, not yet: validating persisted session routes against current config/catalog is the right layer, but the current catalog-load predicate misses the deletion case where no provider rows remain. The safer fix is to trigger catalog-aware validation based on the stored session route, not only `models.providers` shape.
