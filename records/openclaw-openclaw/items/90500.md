## Summary

This PR is still useful and targets a real current-main routing bug, but it has one blocking gap: the new catalog-load predicate misses supported persisted entries where `modelOverride` is already a qualified `provider/model` ref and `providerOverride` is absent, so that stale provider route can still bypass validation.

## What This Changes

The PR changes gateway session model resolution and `chat.send` so persisted session provider/model routes are checked against current config/catalog visibility before reuse.

## Best Possible Solution

Fix the qualified `modelOverride`-only gap, keep the catalog-aware runtime guard, and have maintainers explicitly accept the remaining compatibility tradeoff before merge.
