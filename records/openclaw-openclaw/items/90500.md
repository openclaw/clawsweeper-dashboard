## Label Justifications

- `P1`: The PR addresses a provider/auth routing bug where stale persisted session state can send live chat traffic through a removed provider.
- `merge-risk: 🚨 compatibility`: The patch changes upgrade-visible behavior for existing sessions whose persisted provider/model no longer appears in current config/catalog.

- `merge-risk: 🚨 security-boundary`: Provider routing determines where prompt content and credentials-backed traffic are sent, so stale-route fallback policy is security-sensitive even without secret-handling code changes.

## Summary

Keep open for maintainer review: the patch addresses a real source-reproducible gateway routing bug and I found no blocking code defect, but it intentionally changes provider/auth routing for existing persisted sessions and needs maintainer acceptance before merge.

## What This Changes

The PR adds catalog/config-aware validation to gateway session model resolution, wires `chat.send` to load catalog visibility only when needed, moves `/stop` before that load, and adds focused resolver/chat regressions.

## Best Possible Solution

Land the focused runtime guard if maintainers accept fallback-to-default semantics, and track doctor/reset-session cleanup separately.

## Reproduction Assessment

Yes. Source inspection on current main shows persisted provider/model state wins before visibility checks in `resolveSessionModelRef`, and `chat.send` consumes that resolved provider for live routing.

## Solution Assessment

Yes, if maintainers accept the policy tradeoff. Resolver-level validation plus conditional `chat.send` catalog visibility is the narrow runtime fix; doctor/reset cleanup is separate product scope.
