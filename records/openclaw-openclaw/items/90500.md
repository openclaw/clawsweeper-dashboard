- `P1`: The PR addresses a provider/auth routing regression where stale persisted session state can send live chat traffic through a removed provider.

## Summary

Keep open for maintainer review: the patch appears correct and well-proven, but it intentionally changes upgrade-visible provider routing for existing sessions whose persisted route no longer exists in current config/catalog.

## What This Changes

The PR adds catalog/config-aware validation to gateway session model resolution, wires `chat.send` to pass catalog visibility when needed, moves `/stop` ahead of that validation, and adds resolver/chat regressions.

## Best Possible Solution

Land this focused runtime guard if maintainers accept fallback-to-default semantics, and track doctor/reset-session cleanup as separate product scope.

## Reproduction Assessment

Yes. Source inspection on current main shows persisted provider/model state wins before visibility checks in `resolveSessionModelRef`, and `chat.send` consumes that provider for routing.

## Solution Assessment

Yes, if maintainers accept the policy tradeoff. Resolver-level validation plus conditional `chat.send` catalog visibility is the narrow runtime fix; doctor/reset cleanup is separate product scope.
