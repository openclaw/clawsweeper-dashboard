## Label Justifications

- `P1`: This PR addresses a live provider/session routing regression where persisted state can silently select the wrong provider for real users.
- `merge-risk: 🚨 compatibility`: The diff changes upgrade behavior for existing persisted session provider/model overrides.
- `merge-risk: 🚨 auth-provider`: Changing or missing the effective provider route can change which provider credentials or auth profile are used.
- `merge-risk: 🚨 session-state`: The patch changes runtime interpretation of stored session model/provider fields and can leave Gateway metadata out of sync with the run selector.

## Summary

Keep open: the PR targets the right provider/session bug, but the validated fallback/default model is not fed into the actual reply/run selector, so stale persisted provider state can still drive the agent run.

## What This Changes

The PR adds catalog-aware validation for persisted session provider/model routes and wires `chat.send` to load catalog visibility before resolving the session model.

## Best Possible Solution

Use one validated session model decision for both Gateway metadata and the actual reply/run selection path, with explicit tested behavior when required catalog validation cannot load the catalog.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: create a session with stale persisted provider/model fields, let `chat.send` resolve a fallback/default model, and the downstream reply/run selector can still read the raw stale override from the session entry.

## Solution Assessment

No. The patch is a plausible resolver fix, but not the best complete fix until the validated provider/model is used by the actual run selector and required catalog-load failure behavior is defined.
