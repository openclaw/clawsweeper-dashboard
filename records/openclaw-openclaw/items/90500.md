## Summary

Keep open: the PR is aimed at the right gateway routing layer and has useful terminal proof, but it misses provider-qualified model-only persisted session refs, so a removed provider can still survive in a supported session shape.

## What This Changes

The PR adds catalog-aware validation to gateway session model resolution and wires `chat.send` to fall stale persisted provider/model routes back to the configured default, with resolver and chat regression tests.

## Best Possible Solution

Keep the catalog-aware `chat.send` approach, but validate every persisted selected-model shape already supported by `resolveSessionModelRef` before merge and preserve the compatibility tests for configured providers and catalog-backed providers.

## Reproduction Assessment

Yes. Source inspection shows the current resolver supports provider-qualified `modelOverride`/`model` without provider companion fields, while the PR predicate would not load catalog validation for those stale-provider shapes.

## Solution Assessment

No. The resolver-level guard plus `chat.send` wiring is the right direction, but it needs to validate all persisted selected-model shapes that the resolver can currently honor.
