## Summary

Keep open: the patch is aimed at the right provider-routing bug, but it can still treat persisted models.json rows as evidence that a deleted provider is visible, so the central stale-route problem is not fully fixed.

## What This Changes

The PR adds catalog-aware validation to gateway session model resolution, wires chat.send to pass catalog visibility, and expands resolver regression tests plus raw-fetch allowlist offsets.

## Best Possible Solution

Validate persisted session routes against current configured providers plus live/manifest plugin visibility that excludes stale persisted models.json rows for deleted config providers, then prove the chat.send path with a preserved-models.json regression.

## Reproduction Assessment

Yes, source-reproducible: current main returns persisted overrides before validating provider visibility, and the PR still accepts a stale provider if the read-only models.json catalog contains it. I did not run a live gateway reproduction in this read-only review.

## Solution Assessment

No: catalog-aware validation is the right layer, but the proposed source of truth is too broad because it treats persisted models.json rows as current visibility. A safer fix needs a catalog source or filter that separates current config/live plugin availability from stale generated state.
