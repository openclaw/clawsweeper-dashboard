## Summary

Keep open: the reported provider/session-state bug is real and still not fixed on current main or v2026.6.1, but this PR is not merge-ready because the validated model is not passed into the actual reply execution path and contributor proof still does not show a real gateway turn.

## What This Changes

The PR adds catalog-aware validation for persisted session model/provider routes in gateway `chat.send` and session resolver tests so deleted providers fall back to the configured default.

## Best Possible Solution

Route one canonical validated provider/model decision into the actual reply resolver, prove `onModelSelected` or the provider dispatch path uses the configured default after provider deletion, and make the runtime-vs-migration behavior explicit before merge.

## Reproduction Assessment

Yes at source level: current main returns persisted overrides before current config validation in `resolveSessionModelRef`, and the real reply resolver still applies `resolveStoredModelOverride` from session state. I did not run a live gateway turn because this was a read-only review.

## Solution Assessment

No: validating `resolveSessionModelRef` inside `chat.send` is only a plausible partial fix because the actual model run is selected later by `getReplyFromConfig`. The maintainable fix is to make the same validated decision feed the reply resolver or centralize validation where stored overrides are consumed.
