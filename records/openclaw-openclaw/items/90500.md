- `P1`: This PR targets an active provider-routing/session-state regression with billing and auth-provider impact, but still has merge-blocking route-selection gaps.
- `merge-risk: 🚨 compatibility`: Merging changes how existing persisted sessions route after provider/config changes, which can alter established upgrade behavior.

- `merge-risk: 🚨 session-state`: The change interprets persisted session provider/model fields differently and can leave stale state effective unless the downstream selector is fixed.

## Summary

Keep this PR open: the bug and fix direction are valid, but the current patch still validates stale session routes only in the gateway wrapper while the downstream auto-reply model selector can keep honoring the persisted deleted-provider override. The catalog-load failure path also still fails open, so the provider/auth/session-state safety behavior needs repair or an explicit maintainer policy before merge.

## What This Changes

The PR adds catalog-aware stale provider validation to `resolveSessionModelRef`, wires `chat.send` to load catalog visibility when needed, moves `/stop` before model resolution, and adds gateway/session tests.

## Best Possible Solution

Make one canonical validated session route feed both gateway metadata/auth tracking and auto-reply run selection, with tested behavior for catalog-load failure and preservation of configured or catalog-visible routes.

## Reproduction Assessment

Yes, at source level: current main returns persisted provider/model overrides before config/catalog visibility checks, and the downstream auto-reply selector can still honor stored overrides when no explicit model allowlist exists. I did not run a live paid-provider reproduction.

## Solution Assessment

No, this is a plausible fix but not the best complete fix yet. The validation needs to be applied at the actual run model-selection boundary, or the validated route must be passed into that boundary, with an explicit catalog-load failure policy.
