## Summary

Keep this PR open: the gateway-owned stale-provider fix is useful and well covered, but the new catalog lookup now runs before existing idempotency and active-run short-circuits, so stale-session retries can block on catalog loading before returning an already-known cached or in-flight result.

## What This Changes

The PR adds catalog-aware gateway session model resolution and wires `chat.send` so persisted routes to removed providers fall back to the configured default while configured or catalog-backed routes remain usable.

## Best Possible Solution

Land the gateway-owned runtime guard with the existing preservation semantics, but keep idempotency and active-run fast paths ahead of catalog loading and get explicit maintainer acceptance of the persisted-session upgrade policy.

Yes, source-level reproduction is high-confidence: current main returns stored provider/model override fields before checking current config/catalog visibility, matching the stale deleted-provider route described by the linked issue.

## Solution Assessment

No, not merge-ready as-is. The resolver guard is the right owner boundary for the stale-provider bug, but the catalog load should not precede existing cached, aborted, and active-run short-circuits in `chat.send`.
