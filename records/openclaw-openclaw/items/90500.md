## Summary

Keep open: the patch addresses a real provider/session routing bug and I did not find a discrete correctness blocker in the current diff. The remaining gate is maintainer acceptance of the compatibility-sensitive runtime policy: stale persisted session provider/model state can now be reinterpreted at dispatch time instead of honored exactly as stored.

## What This Changes

Adds catalog-aware validation to gateway session model resolution and wires `chat.send` plus regression tests so persisted routes to removed providers fall back to the configured default while configured or catalog-backed routes remain usable.

## Best Possible Solution

Land the runtime guard after maintainer acceptance of the session/provider routing policy, and keep broader reset or doctor diagnostics tracked outside this PR if they are still desired.

## Solution Assessment

Yes, this is a bounded mitigation for the central deleted-provider dispatch bug: resolve the effective route against current visibility before chat dispatch. A stronger cleanup design would add doctor/reset migration work, but that is broader than this PR's focused runtime guard.
