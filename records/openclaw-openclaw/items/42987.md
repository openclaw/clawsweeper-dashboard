- `P2`: This is a normal-priority cron/session-state feature PR with limited blast radius because it is opt-in, but it has merge-blocking correctness and security issues.
- `merge-risk: 🚨 security-boundary`: The PR can persist raw cron error text containing sensitive operational data to disk when the new env-gated feature is enabled.

## Summary

Keep open. The proposed cron memory feature is not on current main, but the PR is still not merge-ready: it records from an inactive cron wrapper instead of the active finalization paths, drops recovery classifier metadata, persists raw cron errors, lacks real behavior proof, and is currently merge-conflicting.

## What This Changes

The PR adds an opt-in cron procedural playbook memory layer with cron/gateway recording hooks, focused tests, and a draft autonomy roadmap docs page.

## Best Possible Solution

Rework the feature around canonical cron finalization, persisted classifier metadata, bounded redacted storage, documented opt-in retention, focused tests for scheduled/startup/manual paths, and real failure/recovery proof before merge.

## Reproduction Assessment

Yes for the review findings by source inspection: scheduled ticks, startup catch-up, and manual cron runs use executeJobCoreWithTimeout plus applyOutcomeToStoredJob or applyJobResult, while the PR records only from executeJob. No real-environment proof exists for the proposed feature behavior itself.

## Solution Assessment

No. The direction may be useful, but the current implementation should move to canonical cron finalization points, persist prior classifier metadata, and redact or bound stored error text before adding persistent cron procedural memory.

  - body: The memory layer persists signal.error verbatim as lastError. Cron errors can include provider URLs, identifiers, stack traces, user content, or tokens, so the value needs redaction and a length bound before it is written to persistent JSON state.
  - confidence: 0.84
