## Summary

Keep this PR open but do not merge it as-is: current main does not implement the requested cron playbook memory, and the branch still has source-proven runtime gaps, a data-at-rest concern, and no real behavior proof.

## What This Changes

The PR adds an opt-in cron procedural playbook memory layer, cron service/gateway signal wiring, focused tests, and a draft autonomy roadmap doc.

## Best Possible Solution

Rework this as a narrower opt-in cron memory implementation that records from canonical outcome finalizers, preserves classifier state for recoveries, redacts and bounds persisted error text, and separates the broad autonomy roadmap for owner review.

## Reproduction Assessment

Yes. Source inspection shows the branch records only in executeJob while current scheduled, startup catch-up, and manual cron run flows use executeJobCoreWithTimeout plus finalizers; I did not run a live cron setup.

## Solution Assessment

No. The branch hooks too narrow a helper path, loses classifier state on recovery, and stores raw errors; the maintainable solution is to record from shared cron finalization with bounded, redacted persistence.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Record playbook signals in real cron finalizers:** `src/cron/service/timer.ts:1071-1092`
  - body: The new hook is added only in executeJob, but scheduled ticks, startup catch-up, and manual cron runs execute through executeJobCoreWithTimeout plus finalizers instead. In production those failures and recoveries will not populate the playbook memory, so the feature mostly updates only in the helper path and tests.
  - confidence: 0.93
- **[P2] Preserve errorKind for recovery signals:** `src/cron/service/timer.ts:1082-1090`
  - body: The recovery signal rebuilds the failure signature from previousError only because CronJobState does not persist the prior errorKind. When classification depended on errorKind and the message does not match the regex heuristics, the success signal targets a different signature and the recovery count is silently dropped.
  - confidence: 0.87
- **[P2] Redact stored playbook errors before saving:** `src/cron/procedural-playbook-memory-v0.ts:236`
  - body: The memory layer stores signal.error verbatim as lastError in a persistent JSON file. Cron errors can include provider URLs, identifiers, stack traces, user content, or tokens, so the stored text should be redacted and length-bounded before persistence.
  - confidence: 0.82
