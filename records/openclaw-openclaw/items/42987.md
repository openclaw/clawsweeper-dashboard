## Summary

Keep open: current main has no equivalent implementation, and this branch still has source-reproducible correctness blockers plus no real behavior proof from an actual cron run.

## What This Changes

The PR adds cron procedural playbook memory, records failure/recovery signals from cron execution, wires the gateway dependency, adds tests, and adds an autonomy roadmap docs page.

## Best Possible Solution

Record playbook signals from the shared cron outcome-application path used by scheduler, startup catch-up, and manual runs; persist the prior failure classifier needed for recovery signatures; then prove the behavior with a real cron run plus focused tests.

## Reproduction Assessment

Yes for the PR defects by source inspection: the real scheduler and manual cron run paths bypass the patched wrapper, and the persisted cron state lacks the classifier needed to rebuild recovery signatures. I did not run tests because this was a read-only review.

## Solution Assessment

No. The current implementation wires the feature in a wrapper path that production cron flows bypass; the narrower maintainable solution is to record from shared outcome handling and carry the classifier through state.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Record playbook signals from real cron run paths:** `src/cron/service/timer.ts:1075`
  - body: The hook is only added to `executeJob`, but scheduled jobs and manual `openclaw cron run` execute through `executeJobCoreWithTimeout` and then `applyOutcomeToStoredJob`/`applyJobResult`. That leaves production cron failures and recoveries unrecorded, so the new memory layer is effectively inactive outside this wrapper and its tests.
  - confidence: 0.91
- **[P2] Persist the failure classifier for recovery signals:** `src/cron/service/timer.ts:1092`
  - body: Recovery records rebuild the signature from `previousError` without the original `errorKind`, while `CronJobState` only persists `lastError`. A failure classified only by `errorKind` can be recorded under one signature and then recover under `unknown`, silently dropping the recovery count.
  - confidence: 0.86
