## Summary

Keep open. Current main does not implement cron procedural playbook memory, but this PR should not merge as-is because the hook misses the real cron execution paths, recovery classification can silently miss, raw error text is persisted, and the contributor has not provided real behavior proof.

## What This Changes

The PR adds an opt-in cron procedural playbook memory store, cron service/gateway signal wiring, focused tests, and a draft autonomy roadmap doc.

## Best Possible Solution

Move signal emission into the shared cron outcome/finalization path, persist enough classifier state for recoveries, redact and bound stored error text, and prove the opt-in feature with a real cron failure/recovery run.

## Reproduction Assessment

Yes, for the blocking review findings from source inspection: scheduled, startup catch-up, and manual cron paths use executeJobCoreWithTimeout plus finalizers, while the PR hook is only in executeJob. I did not run a live cron setup.

## Solution Assessment

No. The maintainable solution is to record from the canonical cron outcome/finalization flow, carry the prior failure classifier through recovery, and avoid persisting raw unbounded errors.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Record playbook signals in real cron finalizers:** `src/cron/service/timer.ts:1071-1092`
  - body: The new hook is added only in executeJob, but scheduled ticks, startup catch-up, and manual cron runs execute through executeJobCoreWithTimeout plus finalizers instead. In production those failures and recoveries will not populate the playbook memory, so the feature mostly updates only in tests.
  - confidence: 0.93
- **[P2] Preserve errorKind for recovery signals:** `src/cron/service/timer.ts:1082-1090`
  - body: The recovery signal rebuilds the failure signature from previousError only because CronJobState does not persist the prior errorKind. When classification depended on errorKind and the message does not match the regex heuristics, the success signal targets a different signature and the recovery count is silently dropped.
  - confidence: 0.87
- **[P2] Redact stored playbook errors before saving:** `src/cron/procedural-playbook-memory-v0.ts:236`
  - body: The memory layer stores signal.error verbatim as lastError in a persistent JSON file. Cron errors can include provider URLs, identifiers, stack traces, user content, or tokens, so the stored text should be redacted and length-bounded before persistence.
  - confidence: 0.82
