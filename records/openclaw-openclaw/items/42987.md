## Summary

Keep open: the feature is not on current main, and this PR currently misses real scheduled/manual cron runs because the new signal hook is attached to a helper path rather than the canonical cron outcome paths. Recovery tracking also loses `errorKind`, and the contributor proof is tests/build output only.

## What This Changes

The PR adds an opt-in cron procedural playbook memory JSON store, wires a cron/gateway signal hook, adds targeted tests, and adds a draft autonomy roadmap page.

## Best Possible Solution

Move playbook signal emission into the canonical cron outcome finalization paths for scheduler, startup catch-up, and manual runs; preserve the failure classifier through recovery; cover those paths with regression tests; then add redacted real cron proof before merge.

## Reproduction Assessment

Yes, from source inspection. Scheduler, startup catch-up, and manual cron runs call `executeJobCoreWithTimeout` plus outcome finalizers, while the PR's new hook is only in `executeJob`; I did not run a live cron proof.

## Solution Assessment

No. The maintainable fix is to attach recording to the shared cron outcome/finalization flow and carry classifier state there, rather than adding side effects to an execution helper bypassed by real run paths.

## Review Findings

  - body: The new signal hook only runs inside `executeJob`, but scheduled ticks, startup catch-up, and manual `openclaw cron run` execute through `executeJobCoreWithTimeout` and finalizers instead.   - confidence: 0.87
