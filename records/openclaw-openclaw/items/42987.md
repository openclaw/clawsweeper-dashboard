## Summary

Keep this PR open for maintainer follow-up. Current main still does not implement cron procedural playbook memory, so the PR is not obsolete, but the proposed patch is not merge-ready because the recorder is wired to a non-primary cron path, recovery classification can be lost, persistence drops some valid cron session targets, and the durable error-memory design needs privacy, retention, and concurrency handling.

## What This Changes

The PR adds an opt-in cron procedural playbook memory module, gateway recorder wiring, targeted cron memory tests, and a draft autonomy roadmap docs page.

## Best Possible Solution

The best end state is a maintainer-directed split: keep roadmap docs separate, then either implement cron playbook memory in a plugin using an additive classified-error cron hook, or implement it in core from the shared cron outcome path with lastErrorKind propagation, serialized writes, redaction/retention policy, and coverage for scheduled, startup catch-up, manual, and custom-session runs.

## Reproduction Assessment

Yes. Static inspection gives a high-confidence reproduction path: scheduled, startup catch-up, and manual cron runs use executeJobCoreWithTimeout plus applyOutcomeToStoredJob/applyJobResult, while the PR records only from executeJob; the PR parser also demonstrably drops saved entries for current and session:<id> targets on reload.

## Solution Assessment

No. The proposed patch is not the best way to solve the issue because it records from a secondary wrapper instead of the shared outcome path and adds durable raw error memory before the classified-error, concurrency, redaction, and retention seams are settled.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Record signals from shared cron outcomes:** `src/cron/service/timer.ts:1071-1082`
  - body: The recorder is only called from executeJob, but scheduled ticks, startup catch-up, and manual runs execute via executeJobCoreWithTimeout and then apply outcomes without calling this wrapper. That makes the new memory layer miss the real production cron failures and recoveries; move emission into the shared outcome-application path.
  - confidence: 0.96
- **[P2] Carry the prior error kind into recovery signals:** `src/cron/service/timer.ts:1083-1090`
  - body: Recovery signals only pass the previous error string. If the original failure was classified by errorKind and the message does not match the regex fallback, the recovery is written under a different signature and the success count is silently dropped. Persist a lastErrorKind and pass it with the recovery signal.
  - confidence: 0.92
- **[P2] Load entries for every cron session target:** `src/cron/procedural-playbook-memory-v0.ts:543-545`
  - body: CronSessionTarget includes current and session:<id>, and recordSignal can save those values, but toEntry rejects everything except main and isolated. After a restart, memory for current/custom-session cron jobs disappears from the loaded state.
  - confidence: 0.9
- **[P2] Serialize file-backed memory writes:** `src/cron/procedural-playbook-memory-v0.ts:446-450`
  - body: Each signal creates a fresh file-backed layer, loads the JSON file, mutates it, and saves it. With cron maxConcurrentRuns above 1, concurrent signals can load the same old state and the later save can overwrite the earlier update, losing failure or recovery counts.
  - confidence: 0.84
- **[P2] Redact or expire persisted cron error memory:** `src/cron/procedural-playbook-memory-v0.ts:236-238`
  - body: The new state stores raw error text, job names, and job ids without a retention limit or redaction policy. Cron errors can include delivery/account/config details, so core should not persist them indefinitely without documented redaction and pruning behavior.
  - confidence: 0.82
