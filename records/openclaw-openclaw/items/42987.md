## Summary

Keep open: current main does not already implement cron procedural playbook memory, and the PR still has blocking correctness, proof, and security issues before it can merge.

## What This Changes

The PR adds an opt-in cron procedural playbook memory layer, cron/gateway recording hooks, targeted tests, and a draft autonomy roadmap docs page.

## Best Possible Solution

Rework this as a smaller cron memory feature that records at canonical finalization points, preserves classifier metadata for recovery, redacts and bounds persisted errors, documents the opt-in storage contract, and includes real failure/recovery proof.

## Reproduction Assessment

Yes for the review findings: source inspection shows scheduled ticks, startup catch-up, and manual cron runs finalize through executeJobCoreWithTimeout plus applyOutcomeToStoredJob/applyJobResult while the PR records only from executeJob. Real behavior proof for the proposed feature itself is still absent.

## Solution Assessment

No. The direction may be useful, but the implementation should hook canonical cron finalization, persist prior classifier metadata, and redact or bound stored errors before adding persistent cron procedural memory.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Record playbook signals in cron finalizers:** `src/cron/service/timer.ts:1071-1092`
  - body: The new hook is only in executeJob, but current scheduled ticks, startup catch-up, and manual cron runs execute through executeJobCoreWithTimeout and then finalize through applyOutcomeToStoredJob/applyJobResult. Real runtime failures and recoveries will not populate the playbook memory from those paths.
  - confidence: 0.93
- **[P2] Preserve errorKind for recovery signals:** `src/cron/service/timer.ts:1082-1090`
  - body: The recovery branch rebuilds the failure signature from previousError only because CronJobState does not store the prior errorKind. If the original failure was classified by errorKind and the text does not match a regex, recovery tracking targets a different signature and is silently dropped.
  - confidence: 0.87
- **[P2] Redact stored playbook errors before saving:** `src/cron/procedural-playbook-memory-v0.ts:236`
  - body: The memory layer persists signal.error verbatim as lastError. Cron errors can include provider URLs, identifiers, stack traces, user content, or tokens, so the value needs redaction and a length bound before it is written to persistent JSON state.
  - confidence: 0.84
