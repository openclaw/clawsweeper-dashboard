## Summary

Keep open, but the patch is not merge-ready: it adds a product- and security-sensitive durable cron memory feature, while the current diff misses real cron execution paths and persists raw error context without retention or redaction policy.

## What This Changes

The PR adds opt-in cron procedural playbook memory, wires a recorder into Gateway cron service execution, adds tests, and adds a draft autonomy roadmap page to the docs nav.

## Best Possible Solution

Split the draft roadmap from the runtime feature, then redesign cron playbook memory around the shared cron outcome path with classified signals, serialized persistence, target-compatible reloads, redaction, pruning, and coverage for scheduled, startup catch-up, manual, main-session, isolated, current-session, and custom-session runs.

## Reproduction Assessment

Yes. Static inspection gives a high-confidence reproduction path for the PR defects: production scheduled, startup catch-up, and manual runs use executeJobCoreWithTimeout plus shared outcome application, while the patch records only from executeJob.

## Solution Assessment

No. The proposed patch is not the best way to solve the issue because durable memory is wired into a secondary wrapper and lands before classifier propagation, concurrency, redaction, retention, and session-target contracts are settled.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Record signals from the shared cron outcome path:** `src/cron/service/timer.ts:1071-1082`
  - body: The recorder is called only from executeJob, but scheduled ticks, startup catch-up, and manual cron.run flows execute through executeJobCoreWithTimeout and persist through applyOutcomeToStoredJob/applyJobResult. Real cron failures and recoveries would bypass the new memory layer, so emit from the shared outcome-application path instead.
  - confidence: 0.96
- **[P2] Preserve the previous error kind for recoveries:** `src/cron/service/timer.ts:1083-1090`
  - body: Recovery signals only include the previous error string. If the failure was classified by errorKind and the message does not match the regex fallback, the recovery is recorded under a different signature or dropped; persist a prior error kind in job state and pass it with the recovery signal.
  - confidence: 0.92
- **[P2] Accept every persisted cron session target:** `src/cron/procedural-playbook-memory-v0.ts:543-545`
  - body: CronSessionTarget includes current and session:<id>, and recordSignal can save those values, but toEntry rejects everything except main and isolated. After restart, playbook memory for current/custom-session cron jobs disappears from loaded state.
  - confidence: 0.9
- **[P2] Serialize file-backed playbook updates:** `src/cron/procedural-playbook-memory-v0.ts:446-450`
  - body: Each signal creates a fresh file-backed layer, loads the JSON file, mutates it, and saves it. With concurrent cron runs, two signals can load the same old state and the later save can overwrite the earlier update, losing failure or recovery counts.
  - confidence: 0.84
- **[P2] Redact or expire stored cron error context:** `src/cron/procedural-playbook-memory-v0.ts:236-238`
  - body: The new state stores raw error text, job names, and job ids without a retention limit or redaction policy. Cron errors can include delivery targets, account/config details, or provider text, so core should not persist them indefinitely without documented redaction and pruning behavior.
  - confidence: 0.82
