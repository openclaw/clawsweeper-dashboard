## Summary

Keep open. Current main does not implement cron procedural playbook memory, and this PR still has blocking runtime wiring and recovery-tracking defects plus no real cron-run proof; the broader autonomy/memory direction also needs maintainer product judgment.

## What This Changes

The branch adds an opt-in cron procedural playbook memory layer, cron service and gateway wiring, focused tests, and an autonomy roadmap docs page.

## Best Possible Solution

Record playbook signals from the shared cron outcome-application path used by scheduler, startup catch-up, and manual runs; persist the failure classifier needed for recovery signatures; then decide the cron memory and roadmap scope with maintainer review after real cron-run proof.

## Reproduction Assessment

Yes for the PR defects by source inspection: scheduled, startup catch-up, and manual cron paths call executeJobCoreWithTimeout plus outcome application rather than the patched executeJob wrapper, and CronJobState lacks lastErrorKind. I did not run tests because this was a read-only review.

## Solution Assessment

No. Hooking only executeJob is not the narrow maintainable integration point; the safer fix is to record from shared outcome handling and carry the classifier through persisted cron state.
