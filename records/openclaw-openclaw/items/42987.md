## Summary

Keep open: current main does not implement this cron playbook memory feature, and the PR still has blocking runtime placement, persistence, recovery-classification, proof, and security/product-scope issues before it can be merged.

## What This Changes

The PR adds an opt-in cron procedural playbook memory layer, gateway recording hook, targeted tests, and a draft autonomy roadmap doc.

## Best Possible Solution

Split the roadmap from the runtime feature, then redesign cron playbook memory around shared outcome application, classified recoveries, serialized redacted persistence, full CronSessionTarget reload compatibility, pruning, and real scheduled/manual/startup coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path for the PR defects: scheduled ticks, startup catch-up, and manual cron.run call executeJobCoreWithTimeout plus shared outcome application, while the PR records only inside executeJob.

## Solution Assessment

No. The proposed approach adds durable memory before the shared hook point, recovery classifier propagation, target compatibility, concurrency, and retention/redaction contracts are settled.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Record playbook signals from the shared outcome path:** `src/cron/service/timer.ts:1071-1082`
  - body: The hook is only in executeJob, while scheduled ticks, startup catch-up, and manual cron.run execute through executeJobCoreWithTimeout and shared outcome application. Real cron failures and recoveries would bypass the memory layer, so emit from the shared result-application path.
  - confidence: 0.96
- **[P2] Preserve the previous error kind for recoveries:** `src/cron/service/timer.ts:1082-1090`
  - body: Recovery signals include only the previous error string. If the original failure was classified by errorKind and the message does not match the fallback regex, the recovery is recorded under a different signature or silently dropped.
  - confidence: 0.92
- **[P2] Accept every persisted cron session target:** `src/cron/procedural-playbook-memory-v0.ts:543-545`
  - body: CronSessionTarget includes current and session:<id>, and recordSignal can save those values, but toEntry rejects everything except main and isolated. After restart, memory for current/custom-session cron jobs disappears.
  - confidence: 0.9
- **[P2] Serialize file-backed playbook updates:** `src/cron/procedural-playbook-memory-v0.ts:446-450`
  - body: Each signal creates a fresh file-backed layer, loads the JSON file, mutates it, and saves it. Concurrent cron runs can load the same old state and overwrite each other's counts.
  - confidence: 0.84
- **[P2] Redact or expire stored cron error context:** `src/cron/procedural-playbook-memory-v0.ts:236-238`
  - body: The new state stores raw error text, job names, and job ids without a retention limit or redaction policy. Cron errors can include delivery targets, account/config details, or provider text.
  - confidence: 0.82
