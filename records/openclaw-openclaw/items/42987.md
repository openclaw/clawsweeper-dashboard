## What This Changes

The PR adds an opt-in cron procedural playbook memory module, cron/gateway recording hooks, targeted tests, and a draft autonomy roadmap docs page.

## Best Possible Solution

Land a smaller version only after shared finalizer coverage, classifier persistence, redacted bounded storage, and real runtime proof are in place.

## Reproduction Assessment

Yes for the review findings: source inspection shows scheduled ticks, startup catch-up, and manual cron runs finalize through shared outcome paths while the PR records only from executeJob. Real behavior proof for the proposed feature itself is still missing.

## Solution Assessment

No. The direction may be useful, but the implementation should hook canonical finalization, preserve prior classifier metadata for recovery, and redact or bound persisted errors before adding cron procedural memory.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Record playbook signals in cron finalizers:** `src/cron/service/timer.ts:1071-1092`
  - body: The new hook is only in executeJob, but scheduled ticks, startup catch-up, and manual cron run flows execute through executeJobCoreWithTimeout and then finalize via applyOutcomeToStoredJob/applyJobResult. Real production failures and recoveries will not populate the playbook memory from those paths.
  - confidence: 0.93
- **[P2] Preserve errorKind for recovery signals:** `src/cron/service/timer.ts:1082-1090`
  - body: The recovery signal rebuilds the failure signature from previousError only because CronJobState does not store the prior errorKind. If the original classification depended on errorKind and the text does not match a regex, the success targets a different signature and recovery tracking is silently dropped.
  - confidence: 0.87
- **[P2] Redact stored playbook errors before saving:** `src/cron/procedural-playbook-memory-v0.ts:236`
  - body: The memory layer persists signal.error verbatim as lastError. Cron errors can include provider URLs, identifiers, stack traces, user content, or tokens, so this should be redacted and length-bounded before writing persistent JSON state.
  - confidence: 0.82
