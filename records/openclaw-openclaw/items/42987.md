## Summary

Keep open: current main does not implement the proposed cron playbook memory and I found no merged replacement, but this PR is not mergeable as-is because source inspection confirms the hook misses real cron paths, recovery classification can be lost, raw errors would be persisted, and contributor real-behavior proof is still missing.

## What This Changes

The PR adds an opt-in cron procedural playbook memory store, cron service/gateway signal wiring, focused tests, and a draft autonomy roadmap doc.

## Best Possible Solution

Replace or rebase this with a narrower opt-in cron memory implementation that records from canonical outcome finalizers, carries classifier state through recovery, redacts and bounds persisted error text, adds route-level tests, and gets owner review for the roadmap.

## Reproduction Assessment

Yes, for the blocking PR findings from source inspection: scheduled, startup catch-up, and manual cron paths use executeJobCoreWithTimeout plus finalizers, while the PR hook is only in executeJob. I did not run a live cron setup.

## Solution Assessment

No. The maintainable solution is to record from the shared cron outcome/finalization flow, persist enough classifier state for recoveries, and avoid storing raw unbounded error text.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Record playbook signals in real cron finalizers:** `src/cron/service/timer.ts:1071-1092`
  - body: The new hook is added only in executeJob, but scheduled ticks, startup catch-up, and manual cron runs execute through executeJobCoreWithTimeout plus finalizers instead. In production those failures and recoveries will not populate the playbook memory, so the feature mostly updates only in the helper path and tests.
  - confidence: 0.93
- **[P2] Preserve errorKind for recovery signals:** `src/cron/service/timer.ts:1082-1090`
  - body: The recovery signal rebuilds the failure signature from previousError only because CronJobState does not persist the prior errorKind. When classification depended on errorKind and the message does not match the regex heuristics, the success signal targets a different signature and the recovery count is silently dropped.
  - confidence: 0.87
- **[P2] Redact stored playbook errors before saving:** `src/cron/procedural-playbook-memory-v0.ts:236`
  - body: The memory layer stores signal.error verbatim as lastError in a persistent JSON file. Cron errors can include provider URLs, identifiers, stack traces, user content, or tokens, so the stored text should be redacted and length-bounded before persistence.
  - confidence: 0.82
