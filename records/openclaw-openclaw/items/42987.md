## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Move playbook signals into cron finalizers:** `src/cron/service/timer.ts:1071-1092`
  - body: The hook is wired only inside executeJob, but scheduled ticks, startup catch-up, and manual cron runs execute through executeJobCoreWithTimeout and finalize through applyOutcomeToStoredJob or applyJobResult. Real runtime failures and recoveries will not populate playbook memory from those paths, making the feature effectively inactive outside the new tests.
  - confidence: 0.93
- **[P2] Carry errorKind into recovery signals:** `src/cron/service/timer.ts:1082-1090`
  - body: The recovery branch rebuilds the failure signature from previousError because CronJobState does not store the prior errorKind. If the failed run was classified by errorKind and the text does not match a regex, recovery tracking targets a different signature and is silently dropped.
  - confidence: 0.87
- **[P2] Redact stored cron errors before saving:** `src/cron/procedural-playbook-memory-v0.ts:236`
  - body: The memory layer persists signal.error verbatim as lastError. Cron errors can include provider URLs, identifiers, stack traces, user content, or tokens, so the value needs redaction and a length bound before it is written to persistent JSON state.
  - confidence: 0.84
