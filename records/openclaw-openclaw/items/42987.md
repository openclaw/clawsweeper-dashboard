## Summary

Keep open: current main does not implement this feature, and the PR still has blocking runtime, recovery-classification, persistence-security, and proof gaps that need contributor and owner follow-up.

## What This Changes

The branch adds a cron procedural playbook memory layer, cron/gateway service wiring, focused tests, and a draft autonomy roadmap docs page.

## Best Possible Solution

Rework this as a narrower opt-in cron memory feature recorded from canonical outcome finalizers, with preserved classifier state, bounded redacted persistence, focused real behavior proof, and separate owner review for the autonomy roadmap.

## Reproduction Assessment

Yes. Source inspection shows scheduled, startup catch-up, and manual cron run flows finish through executeJobCoreWithTimeout plus applyOutcomeToStoredJob/applyJobResult, while the PR records only inside executeJob.

## Solution Assessment

No. The PR hooks too narrow a helper path, loses classifier state on recovery, and stores raw error text; recording from shared finalization with redacted bounded state is the maintainable path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Record playbook signals in real cron finalizers:** `src/cron/service/timer.ts:1071-1092`
  - body: The new hook is only in executeJob, but current scheduled ticks, startup catch-up, and manual cron runs execute through executeJobCoreWithTimeout and then applyOutcomeToStoredJob/applyJobResult. Real production failures and recoveries will not populate the playbook memory from those paths.
  - confidence: 0.93
- **[P2] Preserve errorKind for recovery signals:** `src/cron/service/timer.ts:1082-1090`
  - body: The recovery signal rebuilds the failure signature from previousError only because CronJobState does not store the prior errorKind. If the original classification depended on errorKind and the text does not match a regex, the success targets a different signature and recovery tracking is silently dropped.
  - confidence: 0.87
- **[P2] Redact stored playbook errors before saving:** `src/cron/procedural-playbook-memory-v0.ts:236`
  - body: The memory layer persists signal.error verbatim as lastError. Cron errors can include provider URLs, identifiers, stack traces, user content, or tokens, so this should be redacted and length-bounded before writing the JSON state.
  - confidence: 0.82
