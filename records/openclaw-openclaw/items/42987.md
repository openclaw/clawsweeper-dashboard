## Summary

Keep this PR open, but it is not mergeable as-is: source review still shows the new cron memory hook misses real cron execution paths, recovery tracking can lose the original classifier, raw errors are persisted, and the external PR still lacks real behavior proof.

## What This Changes

The PR adds an opt-in cron procedural playbook memory layer, gateway service wiring, focused tests, and a draft autonomy roadmap docs page.

## Best Possible Solution

Rework this as a narrower opt-in cron memory feature recorded from canonical outcome finalizers, with preserved classifier state, bounded redacted persistence, real behavior proof, and separate owner review for the autonomy roadmap.

## Reproduction Assessment

Yes. Source inspection against current main shows scheduled timer, startup catch-up, and manual cron runs execute through executeJobCoreWithTimeout plus applyOutcomeToStoredJob/applyJobResult, while the PR records only inside executeJob.

## Solution Assessment

No. The maintainable direction is to record from shared cron outcome finalization, preserve the original classifier state for recovery, and redact/bound persisted errors before exposing this as an opt-in feature.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Record playbook signals in cron finalizers:** `src/cron/service/timer.ts:1071-1092`
  - body: The new hook is only in executeJob, but scheduled ticks, startup catch-up, and manual openclaw cron run flows execute through executeJobCoreWithTimeout and then finalize via applyOutcomeToStoredJob/applyJobResult. Real production failures and recoveries will not populate the playbook memory from those paths.
  - confidence: 0.93
- **[P2] Preserve errorKind for recovery signals:** `src/cron/service/timer.ts:1082-1090`
  - body: The recovery signal rebuilds the failure signature from previousError only because CronJobState does not store the prior errorKind. If the original classification depended on errorKind and the text does not match a regex, the success targets a different signature and recovery tracking is silently dropped.
  - confidence: 0.87
- **[P2] Redact stored playbook errors before saving:** `src/cron/procedural-playbook-memory-v0.ts:236`
  - body: The memory layer persists signal.error verbatim as lastError. Cron errors can include provider URLs, identifiers, stack traces, user content, or tokens, so this should be redacted and length-bounded before writing persistent JSON state.
  - confidence: 0.82
