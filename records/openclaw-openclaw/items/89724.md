## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Gate Teams transcripts on recording status:** `extensions/voice-call/src/providers/msteams.ts:279-287`
  - body: `handleTranscript` immediately turns every STT result into `call.speech`, which the manager persists in the call transcript and then uses for response generation. For Teams media bots, Microsoft requires a successful `updateRecordingStatus` before recording or persisting media-derived data, so the external worker needs to send a confirmed recording-active signal and OpenClaw must hold or drop transcripts until that state is true. ([learn.microsoft.com](https://learn.microsoft.com/en-us/graph/api/call-updaterecordingstatus?view=graph-rest-1.0&utm_source=openai))
  - confidence: 0.92
- **[P1] Resolve the provider boundary before bundling msteams:** `extensions/voice-call/openclaw.plugin.json:260-263`
  - body: Adding `msteams` directly to the bundled voice-call provider enum makes this a core-distributed provider even though the Windows Graph/Skype media worker is a separate service-specific component. Repository policy routes optional integrations to plugins/ClawHub when current seams suffice, so this should either move to an optional plugin or land only after explicit maintainer approval that Teams voice belongs in the bundled voice-call plugin.
  - confidence: 0.84
- **[P2] Fail fast when Teams cannot transcribe calls:** `extensions/voice-call/src/runtime.ts:556-564`
  - body: When `streaming.enabled` is false, the runtime only warns and then still starts the Teams WebSocket server. That lets the external worker connect real calls even though `handleSessionStart` has no transcription dependency and will close the session as `not-ready`, so `provider: "msteams"` should fail validation/startup unless streaming and a configured STT provider are available.
  - confidence: 0.89
- **[P2] Preserve audio while the STT session connects:** `extensions/voice-call/src/providers/msteams.ts:262-266`
  - body: The Teams provider drops inbound frames whenever `sttSession.isConnected()` is false. The shared realtime-transcription WebSocket session already queues pre-ready audio within a bounded byte limit, and the Twilio stream path sends audio into the session without this extra drop, so early caller speech can be lost on Teams during provider connect latency.
  - confidence: 0.88
- **[P2] Document the new Teams voice-call setup before exposing it:** `extensions/voice-call/openclaw.plugin.json:260-263`
  - body: The PR exposes `msteams` as a selectable voice-call provider in plugin metadata, but the public voice-call docs still list only Twilio, Telnyx, Plivo, and mock and do not explain the external Windows worker, HMAC headers, bind address, streaming/TTS requirements, inbound-only behavior, or recording-status obligation. Operators need that setup/security path before this config surface is advertised.
  - confidence: 0.86
