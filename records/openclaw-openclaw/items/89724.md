## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Gate Teams transcripts on recording status:** `extensions/voice-call/src/providers/msteams.ts:279-287`
  - body: `handleTranscript` immediately records Teams STT output through `call.speech`, and current manager handling persists final speech into the call record. Before OpenClaw stores or responds from Teams media, this provider needs an explicit contract that the external worker successfully set the Teams recording status for the call, or a maintainer-approved reason this path is exempt; otherwise an operator can collect and persist Teams call transcripts without the Teams recording/compliance state OpenClaw can audit.
  - confidence: 0.87
- **[P1] Enforce connection caps before accepting Teams sockets:** `extensions/voice-call/src/msteams-media-stream.ts:145-148`
  - body: The existing Twilio media stream path bounds pending sockets, per-IP pending sockets, active sessions, and pre-start idle time, but this new `WebSocketServer` only sets `maxPayload` and then stores every authenticated session in a map. A valid shared secret or buggy bridge can hold unbounded Teams sockets and exhaust gateway resources; carry the existing `streaming.maxConnections`, `maxPendingConnections`, `maxPendingConnectionsPerIp`, and timeout behavior into this transport or add equivalent msteams-scoped limits and tests.
  - confidence: 0.92
- **[P2] Reject session.start call ids that differ from the URL:** `extensions/voice-call/src/msteams-media-stream.ts:302-308`
  - body: The HMAC is verified over the URL `callId`, but `session.start` forwards `parsed.callId` to the provider while send/audio/end continue using the URL id. If those differ, the provider creates a call under one id, drops audio under the other, and never ends the registered call; reject mismatched `session.start.callId` or remove that duplicate field from the worker message.
  - confidence: 0.9
- **[P2] Preserve audio while the STT session connects:** `extensions/voice-call/src/providers/msteams.ts:262-267`
  - body: `handleSessionStart` starts `sttSession.connect()` asynchronously, but `handleAudioFrame` drops every frame until `isConnected()` becomes true. Current voice-call streaming docs and Twilio behavior queue media while transcription connects; this Teams path can lose the caller's first words on every call, so buffer bounded audio until connect succeeds or do not accept frames until the provider is ready.
  - confidence: 0.88
- **[P2] Reject msteams configs that cannot transcribe calls:** `extensions/voice-call/src/runtime.ts:545-559`
  - body: When `provider` is `msteams`, the runtime logs if `streaming.enabled` is false or if no transcription provider resolves, but it still starts the Teams WebSocket listener. That lets operators accept Teams calls that immediately close as `not-ready` or can never produce transcripts; validation should fail unless the selected Teams config can resolve an STT provider, with focused config/runtime tests for disabled streaming and missing provider cases.
  - confidence: 0.84
- **[P2] Add Teams voice-call setup and security docs:** `extensions/voice-call/openclaw.plugin.json:260-263`
  - body: The branch exposes `msteams` as a public provider and SecretRef surface, but the user-facing voice-call docs still list only Twilio/Telnyx/Plivo/mock and explain `fromNumber` plus public webhooks as required setup. Operators need docs for the external Windows worker, Teams app calling support, HMAC headers, loopback/default bind posture, required config, recording-status expectations, inbound-only behavior, and unsupported outbound dialing before this can be a supported provider.
  - confidence: 0.82
