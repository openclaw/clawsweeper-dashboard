## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Gate STT audio before recording is active:** `extensions/voice-call/src/providers/msteams.ts:334-335`
  - body: With `requireRecordingStatus` defaulting true, this still forwards caller PCM to the STT session before `recordingActive` is checked; only the later transcript callback is dropped. That means an external STT provider can process call audio before the Teams recording-status gate that the PR says protects media-derived processing. Buffer or drop raw frames, or delay STT connection, until recording is active.
  - confidence: 0.92
- **[P1] Close Teams calls when STT connect rejects:** `extensions/voice-call/src/providers/msteams.ts:311-315`
  - body: If `sttSession.connect()` rejects, the catch only logs after the call has already been registered and answered. The WebSocket stays open with no transcription path, leaving the caller in silence while frames buffer or are ignored; tear down the session/call and add a regression test for connect failure.
  - confidence: 0.9
- **[P2] Fail startup when streaming has no STT provider:** `extensions/voice-call/src/msteams.runtime.ts:69-72`
  - body: When `resolveConfiguredCapabilityProvider` fails, this logs a warning and returns, but `runtime.ts` still sets `handlerWired = true` for `streaming.enabled` and binds the Teams listener. A misconfigured runtime will accept calls it cannot transcribe; throw or return a failure before `msteamsProvider.start()`.
  - confidence: 0.88
- **[P2] Add requireRecordingStatus to the manifest schema:** `extensions/voice-call/openclaw.plugin.json:318-320`
  - body: The runtime config and README use `msteams.requireRecordingStatus`, but the manifest schema for `msteams` is `additionalProperties: false` and omits that property. Metadata/setup/schema consumers can reject the documented opt-out config or omit it from generated surfaces until this field is added with coverage.
  - confidence: 0.9
