## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Bind the Teams media socket to an explicit safe host:** `extensions/voice-call/src/msteams-media-stream.ts:149`
  - body: `server.listen(this.config.port)` omits the host, and Node documents that this accepts connections on unspecified IPv6/IPv4 addresses when available. That means enabling the provider can expose the HMAC bridge on all interfaces, while the PR body says operators should bind it to a trusted segment. Add a bind setting with a safe default, wire it through config/manifest/docs, and cover the resulting address behavior. ([nodejs.org](https://nodejs.org/download/release/v18.9.0/docs/api/net.html?utm_source=openai))
  - confidence: 0.9
- **[P1] Make Teams media startup part of the runtime lifecycle:** `extensions/voice-call/src/providers/msteams.ts:125-127`
  - body: The provider starts the WebSocket server in the constructor and catches bind failures only in the logger, and `VoiceCallRuntime.stop()` never calls `MsteamsProvider.stop()`. A port conflict can therefore leave Gateway startup looking successful with no Teams listener, and a normal runtime stop can leave the Teams port bound for the next start. Move start/stop into the awaited runtime lifecycle and add a restart/bind-failure test.
  - confidence: 0.89
- **[P2] Require the Teams transport config and secret contract:** `extensions/voice-call/src/config.ts:68-72`
  - body: With `port`, `path`, and `sharedSecret` all optional and no msteams validation block, `provider: "msteams"` can pass validation while the constructor only logs that no server was started. The HMAC secret is also a plain string and is not listed in the plugin `secretInputs` contract, so operators cannot use the established SecretRef path for this new credential. Make the active provider fail validation until its required transport settings and secret input are configured.
  - confidence: 0.86
