## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Gate Teams transcript persistence on recording status:** `extensions/voice-call/src/providers/msteams.ts:279-287`
  - body: `handleTranscript` persists Teams-derived STT text through `call.speech` without any protocol field or runtime check showing the external Teams worker successfully called Microsoft Graph `updateRecordingStatus` first. Microsoft documents that Media Access API media or derived data must not be persisted before that API succeeds, so the bridge contract needs to carry and enforce recording-status state before OpenClaw stores transcripts or generates responses from them. ([learn.microsoft.com](https://learn.microsoft.com/en-us/graph/api/call-updaterecordingstatus?view=graph-rest-1.0))
  - confidence: 0.91
- **[P1] Enforce connection caps before accepting Teams sockets:** `extensions/voice-call/src/msteams-media-stream.ts:253-255`
  - body: The Teams listener authenticates the upgrade and then immediately hands it to `ws`, but it has no equivalent to the existing Twilio media-stream `maxConnections`, `maxPendingConnections`, or per-IP pending caps. A valid shared secret, leaked worker, or misbehaving bridge can keep opening call sockets until the process runs out of file descriptors or memory, so the new listener should reject above bounded totals before accepting the socket.
  - confidence: 0.88
- **[P2] Reject msteams configs that cannot transcribe calls:** `extensions/voice-call/src/runtime.ts:543-565`
  - body: When `provider.name === "msteams"` but `streaming.enabled` is false, runtime only logs a warning and still starts the WebSocket server; incoming calls then close as `not-ready` because no transcription dependency was wired. Since msteams is entirely driven by realtime transcription, `validateProviderConfig` or startup should fail before binding the listener unless streaming and a usable STT provider are configured.
  - confidence: 0.86
- **[P2] Preserve audio while the STT session connects:** `extensions/voice-call/src/providers/msteams.ts:262-267`
  - body: `handleAudioFrame` drops every inbound PCM frame until `sttSession.isConnected()` flips true. The worker can start sending caller audio immediately after `session.start`, and the existing Twilio stream path is documented to queue/forward media while the transcription provider connects, so early words can be silently lost on real calls unless this path buffers or gates audio until STT is ready.
  - confidence: 0.84
- **[P2] Validate session.start matches the authenticated call id:** `extensions/voice-call/src/msteams-media-stream.ts:300-308`
  - body: The HMAC authenticates the call id in the URL, but `session.start` trusts `parsed.callId` and passes it to the provider while the session send/close closures still target the URL call id. If those differ, OpenClaw can create a call record under one id while later audio/end frames are keyed under another, leaking an active call and losing audio; reject mismatched `session.start.callId` before invoking `onSessionStart`.
  - confidence: 0.82
- **[P2] Add Teams voice-call setup and security docs:** `extensions/voice-call/openclaw.plugin.json:260-263`
  - body: Adding `msteams` to the public provider enum makes this a user-facing provider, but the voice-call docs and README still list only Twilio, Telnyx, Plivo, and mock and do not describe the external Windows worker, HMAC headers, bindAddress posture, required `streaming`/TTS config, recording-status obligations, or inbound-only limitation. Users cannot safely configure this provider from the shipped docs yet.
  - confidence: 0.9
