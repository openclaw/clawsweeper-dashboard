## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Add the Teams recording-status contract before persisting transcripts:** `extensions/voice-call/src/providers/msteams.ts:279-287`
  - body: This path turns every Teams STT result into `call.speech`; current manager handling appends final transcripts and persists the call record. Microsoft requires `updateRecordingStatus` before persisting Teams media content or data derived from it, and this PR leaves that proof entirely out-of-repo, so add an explicit bridge signal/config/docs+test before storing Teams-derived transcripts. ([learn.microsoft.com](https://learn.microsoft.com/en-us/graph/api/call-updaterecordingstatus?view=graph-rest-1.0&utm_source=openai))
  - confidence: 0.88
- **[P1] Enforce connection caps before accepting Teams sockets:** `extensions/voice-call/src/msteams-media-stream.ts:253-254`
  - body: Every valid-HMAC upgrade reaches `handleUpgrade` and then `sessions.set()` with no total, pending, or per-IP cap. The existing Twilio media path has those bounds, and without them a bridge bug, leaked secret, or trusted-network client can hold unlimited sockets and memory despite the PR body claiming these caps exist.
  - confidence: 0.9
- **[P2] Reject msteams configs that cannot transcribe calls:** `extensions/voice-call/src/runtime.ts:545-559`
  - body: When `streaming.enabled` is false, this branch only logs a warning but still starts the Teams listener, so a config that passed validation can accept a call and then close it as `not-ready` because STT/TTS dependencies were never wired. Require or default the streaming path for `msteams`, or fail validation/startup instead of binding a nonfunctional provider.
  - confidence: 0.86
- **[P2] Do not drop Teams audio while STT is connecting:** `extensions/voice-call/src/providers/msteams.ts:264-267`
  - body: Teams audio frames are discarded until `isConnected()` flips true. The existing Twilio stream path intentionally forwards early media before readiness and has a regression test for it, so this can silently lose the caller's first words during normal STT startup latency.
  - confidence: 0.84
