## Summary

Keep this issue open. Current main still uses `abortedLastRun` for both explicit user aborts and restart-interrupted work, and main-session restart recovery still resumes any running session with that marker when the transcript tail is resumable.

## What This Changes

The issue requests that explicit stop/no-action instructions stop active work and prevent later internal continuation or retry messages.

## Best Possible Solution

Persist explicit user abort intent separately from restart-interruption recovery, and make synthetic resume eligible only for restart-marked interrupted work. Explicit `/stop`, `chat.abort`, and recognized no-action abort phrases should leave the session terminal or recovery-suppressed across Gateway restarts.
