## Summary

Keep this issue open. Current main still lets explicit `/stop` and recognized no-action abort phrases persist the same `abortedLastRun` marker that main-session restart recovery uses to inject a synthetic resume turn, so the reported user-intent violation remains possible after a restart when the transcript tail is resumable.

## What This Changes

The issue reports that explicit user stop/no-action instructions must suppress all later OpenClaw continuation behavior, including internal synthetic resume messages.

## Best Possible Solution

Persist explicit user-abort intent separately from restart-interruption recovery and make synthetic main-session resume eligible only for restart-marked interrupted work, with regression coverage for `/stop`, natural-language abort triggers, and restart recovery.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: persist a running main session with `abortedLastRun: true` from an explicit `/stop` or abort trigger, give it a user/tool-result transcript tail, and `recoverRestartAbortedMainSessions` will call `agent` with the synthetic resume message.

## Solution Assessment

Yes. The narrowest maintainable fix is to split durable restart-recovery state from explicit user-abort state rather than weakening abort detection or removing restart recovery entirely.
