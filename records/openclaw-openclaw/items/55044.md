## Label Justifications

- `P1`: An explicit stop/no-action instruction can be overridden by hidden recovery, allowing an agent workflow to continue after the user asked it to stop.
- `impact:session-state`: The root cause is persisted session-state drift: explicit abort intent and restart-recovery eligibility share `abortedLastRun`.

## Summary

Keep open: current main and the latest release still let explicit stop/no-action aborts set the same `abortedLastRun` marker that restart recovery later treats as permission to inject a hidden continuation turn.

## What This Changes

The issue reports that OpenClaw can ignore an explicit stop/no-action instruction and later inject an internal continuation message.

## Best Possible Solution

Persist restart-recovery eligibility separately from explicit user-abort intent, then skip synthetic continuation for sessions stopped by `/stop` or no-action abort phrases while preserving restart recovery for real gateway interruptions.

## Reproduction Assessment

Yes at source level: explicit stop/no-action paths set `abortedLastRun: true`, and restart recovery later resumes any running main session with that marker when the transcript tail is resumable. I did not run a live provider reproduction in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is to split restart-interruption eligibility from explicit user-abort state and add regression coverage proving explicit stop/no-action sessions are skipped by synthetic recovery.
