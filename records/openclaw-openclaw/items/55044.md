## Label Justifications

- `P1`: An explicit stop/no-action instruction can be overridden by hidden recovery, allowing an active agent workflow to continue after the user asked it to stop.
- `impact:session-state`: The root cause is persisted session-state drift: explicit abort intent and restart-recovery eligibility share `abortedLastRun`.

## Summary

Keep open: current main and the latest release still use `abortedLastRun` for both explicit stop/no-action aborts and restart-recovery eligibility, so a running session with a resumable transcript tail can still receive a hidden synthetic recovery turn after the user asked it to stop.

## What This Changes

The issue asks OpenClaw to respect explicit stop/no-action instructions and never inject an internal continuation turn after that user intent.

## Best Possible Solution

Persist restart-recovery eligibility separately from explicit user-abort intent, and only synthesize continuation for sessions marked by restart cleanup while keeping explicit stop/no-action state terminal.

## Reproduction Assessment

Yes, at source level: a running session marked `abortedLastRun: true` by /stop or a no-action abort phrase can later satisfy restart-recovery selection when the transcript tail is resumable. I did not execute a live provider reproduction in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is to split restart-interruption eligibility from explicit user-abort state, then add regression coverage proving explicit stop/no-action sessions are skipped by synthetic recovery.
