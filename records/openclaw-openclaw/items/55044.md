## Label Justifications

- `P1`: Explicit stop/no-action controls can be overridden by hidden recovery, so a running agent workflow may continue after the user asks it to stop.
- `impact:session-state`: The bug is caused by conflating persisted explicit abort state with restart-recovery session state.

## Summary

Keep open: current main and latest release still use `abortedLastRun` as both the explicit stop/no-action abort marker and the restart-recovery eligibility marker, so a running session with a resumable transcript tail can still receive the hidden recovery continuation path. The adjacent open PRs are useful context, but they do not specifically close the explicit user-abort versus restart-recovery state split.

## What This Changes

The issue reports that OpenClaw can ignore explicit stop or no-action user intent by later injecting an internal continuation message.

## Best Possible Solution

Persist restart-recovery eligibility separately from explicit user-abort intent, and allow the synthetic resume path only for sessions marked by restart cleanup while preserving automatic recovery for true restart interruptions.

## Reproduction Assessment

Yes, at source level: persist a running session entry with `abortedLastRun: true` from `/stop` or a no-action abort phrase, give the transcript a user/tool-result tail, and `recoverRestartAbortedMainSessions` can call the synthetic recovery path. I did not run a live WSL/OpenAI reproduction in this read-only review.

## Solution Assessment

Yes. The narrowest maintainable fix is to split restart-interruption state from explicit user-abort state so explicit stop remains durable while restart recovery only resumes sessions marked by restart cleanup.
