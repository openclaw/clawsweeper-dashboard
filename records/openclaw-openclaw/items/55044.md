## Label Justifications

- `P1`: An explicit stop/no-action control can be overridden by hidden recovery, so an active agent workflow may continue after the user asked it to stop.
- `impact:session-state`: The root cause is conflated persisted session state: explicit abort state and restart-recovery eligibility share `abortedLastRun`.

## Summary

Keep open: current main and latest release still use `abortedLastRun` for both explicit stop/no-action aborts and restart-recovery eligibility, so a running session with a resumable transcript tail can still receive a hidden synthetic recovery turn after the user asked it to stop.

## What This Changes

The issue asks OpenClaw to honor explicit stop or no-action instructions by preventing internal continuation/recovery prompts from resuming the session.

## Best Possible Solution

Persist restart-recovery eligibility separately from explicit user-abort intent, and allow synthetic resume only for sessions marked by restart cleanup while preserving automatic recovery for true restart interruptions.

## Reproduction Assessment

Yes, at source level: persist a running session entry with `abortedLastRun: true` from `/stop` or a no-action abort phrase, give the transcript a user/tool-result tail, and `recoverRestartAbortedMainSessions` can call the synthetic recovery path. I did not run a live WSL/OpenAI reproduction in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is to split restart-interruption eligibility from explicit user-abort state so user stop remains durable and restart recovery only resumes sessions marked by restart cleanup.
