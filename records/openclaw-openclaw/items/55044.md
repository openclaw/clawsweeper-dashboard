## Label Justifications

- `P1`: Explicit stop/no-action controls can be overridden by hidden recovery, so a running agent workflow may continue after the user asks it to stop.
- `impact:session-state`: The bug is caused by conflating persisted explicit abort state with restart-recovery session state.

## Summary

Keep open: current main still lets explicit stop/no-action aborts and restart-owned recovery share `abortedLastRun`, so a running session with a resumable transcript tail can still receive the hidden restart continuation path. The nearby open PRs are useful context, but they target channel-backed replay or normal channel-session state cleanup rather than the explicit user-abort/restart-recovery state split.

## What This Changes

The issue asks OpenClaw to respect explicit stop/no-action instructions and avoid injecting internal continuation messages after the user asked it to stop.

## Best Possible Solution

Persist restart-recovery eligibility separately from explicit user-abort intent, and gate synthetic resume only on the restart-owned marker while preserving automatic recovery for true restart interruptions.

## Reproduction Assessment

Yes, at source level: persist a running session entry with `abortedLastRun: true` from `/stop` or a no-action abort phrase, give the transcript a user/tool-result tail, and `recoverRestartAbortedMainSessions` can call the synthetic recovery path. I did not run a live WSL/OpenAI reproduction in this read-only review.

## Solution Assessment

Yes. The narrowest maintainable fix is to split restart-interruption state from explicit user-abort state so explicit stop remains durable while restart recovery only resumes sessions marked by restart cleanup.
