## Summary

Keep open: current main still uses the same durable `abortedLastRun` marker for explicit stop/no-action aborts and restart-interruption recovery, so a stopped running main session with a resumable transcript tail can still receive the synthetic restart-resume turn. The related open PR only covers channel-backed restart recovery and does not resolve explicit user-abort intent.

## What This Changes

The issue asks OpenClaw to honor explicit `/stop` or no-action abort instructions by preventing later synthetic restart-continuation turns.

## Best Possible Solution

Persist restart-recovery eligibility separately from explicit user-abort intent and gate synthetic resume on the restart-owned marker while preserving automatic recovery for true restart interruptions.

## Reproduction Assessment

Yes, at source level: persist a running main-session entry with `abortedLastRun: true` from `/stop` or an abort phrase, give the transcript a user/tool-result tail, and current `recoverRestartAbortedMainSessions` will call the synthetic `agent` recovery path. I did not run tests because this review was required to keep the checkout read-only.

## Solution Assessment

Yes. The narrowest maintainable fix is to split restart-interruption state from explicit user-abort state, so stop/abort remains durable while restart recovery only resumes sessions marked by restart cleanup.
