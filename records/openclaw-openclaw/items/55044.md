## Summary

Keep open: current main still conflates explicit stop/no-action aborts and restart-interruption recovery through `abortedLastRun`, so a stopped running main session with a resumable transcript tail can still receive the hidden restart-recovery turn. The related open PR https://github.com/openclaw/openclaw/pull/74687 is scoped to channel-backed restart replay and does not split explicit user-abort intent from restart-owned recovery eligibility.

## What This Changes

The issue reports that OpenClaw ignored an explicit stop/no-action instruction and later injected an internal continuation turn.

## Best Possible Solution

Persist restart-recovery eligibility separately from explicit user-abort intent, and gate synthetic resume only on the restart-owned marker while preserving automatic recovery for true restart interruptions.

## Reproduction Assessment

Yes, at source level: persist a running main-session entry with `abortedLastRun: true` from `/stop` or an abort phrase, give the transcript a user/tool-result tail, and current `recoverRestartAbortedMainSessions` will call the synthetic recovery path. I did not run a live WSL/OpenAI reproduction because this review had to keep the checkout read-only.

## Solution Assessment

Yes. The narrowest maintainable fix is to split restart-interruption state from explicit user-abort state, so explicit stop remains durable while restart recovery only resumes sessions marked by restart cleanup.
