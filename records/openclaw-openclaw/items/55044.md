## Summary

Keep open: current main still uses the same `abortedLastRun` marker for explicit user stop/no-action aborts and restart-owned recovery, so a stopped running main session with a resumable transcript tail can still receive the synthetic restart continuation turn. The related open PR https://github.com/openclaw/openclaw/pull/74687 is channel-recovery scoped and does not resolve the explicit user-abort/restart-recovery split.

## What This Changes

The issue reports that an explicit stop/no-action request was followed by internal continuation recovery instead of remaining stopped.

## Best Possible Solution

Persist restart-recovery eligibility separately from explicit user-abort intent, then gate synthetic resume only on the restart-owned marker while preserving explicit stop as a durable no-action state.

## Reproduction Assessment

Yes, at source level: persist a running main-session entry with `abortedLastRun: true` from `/stop` or an abort phrase, give the transcript a user/tool-result tail, and `recoverRestartAbortedMainSessions` will call the synthetic recovery path. I did not run a live WSL/OpenAI reproduction because this review had to keep the checkout read-only.

## Solution Assessment

Yes. The narrowest maintainable fix is to split restart-interruption state from explicit user-abort state, so explicit stop remains durable while restart recovery only resumes sessions marked by restart cleanup.
