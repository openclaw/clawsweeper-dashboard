## Summary

Keep open. Current main still has a plausible no-ack path for Discord application-command interactions whose remote slash command exists but whose local plugin command wrapper was not present at startup; loaded wrappers defer promptly, but plugin command specs still come from the runtime registry rather than manifest-level ownership metadata.

## What This Changes

The issue reports plugin-owned Discord slash commands timing out with `Unknown interaction` because plugin command registration appears to happen lazily during first interaction instead of being ready to acknowledge within Discord's deadline.

## Best Possible Solution

Define an explicit Discord-native readiness contract for plugin-owned slash commands: any deployed command must have a local wrapper or placeholder that acknowledges immediately, then deliberately activates the owning plugin before dispatch. Prefer a manifest-backed descriptor or targeted activation plan over broad eager loading, and keep `/vc` on the dedicated Discord voice command path.

## Reproduction Assessment

Unclear: the report has concrete logs and current source explains a high-probability no-ack path, but this review did not run a live Discord interaction. A high-confidence reproduction should enable Discord native commands, invoke a remotely registered but locally absent startup-lazy plugin command such as `/codex`, and capture received-to-defer plus plugin registration timing.

## Solution Assessment

Unclear: there is no patch to assess, and the suggested blanket eager registration is not obviously the narrowest maintainable fix. A manifest-backed placeholder/readiness contract that defers first and activates the owner plugin deliberately looks safer than loading every plugin at gateway startup.
