## Summary

Keep open. Current main still has a Discord application-command path that returns without acknowledging when Discord routes a command that has no local wrapper, and plugin command specs are still sourced from runtime registration before background reconciliation completes, so the plugin-command readiness gap is not definitely fixed.

## What This Changes

The issue reports Discord plugin-registered slash commands missing Discord's 3-second initial-response deadline and failing with "Unknown interaction" or duplicate-ack errors.

## Best Possible Solution

Define a Discord native plugin-command readiness and reconciliation contract so every Discord-routable plugin command has a local defer-first wrapper before interactions arrive, while tracking saturated event-loop recovery in the related session/load issues.

## Reproduction Assessment

Yes for a high-confidence source-level path, but not for a live end-to-end repro in this review. Current main can receive a Discord application command with no matching local wrapper and return without an initial acknowledgement.

## Solution Assessment

No, the issue's blanket eager-loading suggestion is not the best owner-boundary fix. A narrower readiness and stale-command reconciliation contract is safer than forcing eager plugin loading across the plugin architecture.
