## Summary

Keep open. Current main protects Discord native command objects that already exist locally, but plugin-owned native commands are still sourced from the runtime plugin command registry; manifest-only runtime-slash aliases such as Codex are not converted into Discord-ready command wrappers before the gateway starts, and the interaction dispatcher has no fallback acknowledgement for a remotely registered command missing from the local command list.

## What This Changes

The issue asks OpenClaw to make Discord plugin-registered slash commands acknowledge within Discord's 3-second deadline instead of timing out with Unknown interaction or double-ack errors.

## Best Possible Solution

Define an explicit Discord-native readiness contract for plugin-owned slash commands: either predeploy manifest-declared commands with a lightweight immediate-defer dispatcher that activates the owner plugin, or only deploy commands whose handlers are loaded before gateway readiness. Keep Discord voice `/vc` handling on its own native subcommand path.

## Reproduction Assessment

Unclear: the issue has concrete logs and the current code explains a plausible no-ack path, but this review did not run a live Discord bot. A high-confidence reproduction should enable `commands.native`, invoke a startup-lazy plugin alias such as `/codex` or a remotely registered but locally absent plugin command, and capture received-to-defer plus plugin registration timing.

## Solution Assessment

Unclear: there is no proposed patch, and blindly eager-loading every plugin would fight the manifest-first laziness contract. The safer solution is a narrow Discord command readiness/placeholder contract that acknowledges immediately without broad startup imports.
