## Summary

Keep open. Current main still has a plausible Discord-native no-ack path when Discord sends an application-command interaction for a remotely registered plugin command that is missing from the local `client.commands` set; loaded local wrappers defer promptly, but plugin command specs still come from the runtime registry rather than manifest alias metadata.

## What This Changes

The issue reports plugin-registered Discord slash commands missing Discord's 3-second acknowledgment deadline and failing with `Unknown interaction` or already-acknowledged errors.

## Best Possible Solution

Define an explicit Discord-native readiness contract for plugin-owned slash commands: only deploy commands whose local handlers or placeholders can acknowledge immediately, and use manifest metadata or a targeted activation plan to load the owning plugin after the ack. Keep `/vc` on the dedicated Discord voice native path and avoid broad eager loading unless maintainers accept that startup tradeoff.

## Reproduction Assessment

Unclear: the issue has concrete logs and current source explains a high-probability no-ack path, but this review did not run a live Discord interaction. A high-confidence reproduction should enable Discord native commands, invoke a remotely registered but locally absent startup-lazy plugin command such as `/codex`, and capture received-to-defer plus plugin registration timing.

## Solution Assessment

Unclear: there is no patch to assess, and the proposed blanket eager registration is not obviously the narrowest maintainable fix. A readiness/placeholder contract that acknowledges first and activates the owner plugin deliberately looks safer than loading every plugin at gateway startup.
