## Summary

Keep open. Current main still acknowledges Discord native plugin commands only after a matching local command wrapper exists, while plugin native command specs are derived from the loaded runtime registry rather than manifest command-alias metadata. That leaves the reported remotely registered-but-locally-missing command path plausible, and the issue includes concrete timing logs showing plugin command registration during interaction handling.

## What This Changes

The issue reports plugin-registered Discord slash commands missing Discord's 3-second acknowledgment deadline and failing with unknown/already-acknowledged interaction errors.

## Best Possible Solution

Define an explicit Discord-native readiness contract for plugin-owned slash commands: either deploy only commands whose handlers are loaded before Discord readiness, or deploy manifest-declared placeholders that immediately acknowledge and then activate the owner plugin. Keep `/vc` on the dedicated Discord voice native path and avoid loading every plugin at startup unless maintainers choose that product tradeoff.

## Reproduction Assessment

Unclear: the issue has concrete logs and current source explains a high-probability no-ack path, but this review did not run a live Discord interaction. A high-confidence reproduction should enable native Discord commands, invoke a remotely registered but locally absent plugin command such as `/codex`, and capture received-to-defer plus plugin registration timing.

## Solution Assessment

Unclear: there is no patch to assess, and the suggested eager startup registration is not obviously the narrowest maintainable fix. A readiness/placeholder contract would be safer than loading every plugin at gateway startup by default.
