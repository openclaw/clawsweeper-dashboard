## Summary

Keep open. Current main still has a Discord application-command path that returns without acknowledging when Discord routes a command that has no local wrapper, and plugin native command specs are still assembled from runtime-registered command state rather than a manifest-backed readiness contract.

## What This Changes

The issue asks for Discord plugin-registered slash commands such as `/vc`, `/voice`, and `/codex` to be acknowledged within Discord's 3-second interaction window instead of failing with `Unknown interaction`.

## Best Possible Solution

Define a Discord native plugin-command readiness contract where every command deployed to Discord has a local defer-first wrapper before interactions can arrive, and stale or unavailable plugin commands are reconciled during startup without breaking manifest-first plugin loading.

## Reproduction Assessment

Unclear. The issue has concrete live logs from 2026.4.26 and current source shows a stale remote-command/no-local-wrapper path that would miss acknowledgment, but I did not run a live Discord smoke against current main.

## Solution Assessment

No. The issue's blanket eager-loading suggestion is not clearly the best fit for the plugin architecture; a readiness/deploy reconciliation contract that preserves selective plugin activation is safer.
