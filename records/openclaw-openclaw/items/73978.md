## Summary

Keep open. Current main still has a source-level no-ack path for Discord application-command interactions when a remotely deployed plugin command has no local wrapper, and plugin command specs still come from runtime registration rather than manifest aliases.

## What This Changes

The report asks OpenClaw to prevent Discord plugin slash commands such as `/vc`, `/codex`, and `/voice` from timing out by acknowledging interactions before plugin registration or handler work.

## Best Possible Solution

Define and implement a Discord-native plugin-command readiness contract: deploy only commands with local immediate-ack wrappers, or install manifest-backed placeholders that defer first and then activate or dispatch the owning plugin deliberately.

## Reproduction Assessment

Unclear. The issue has concrete logs and current source shows a missing-wrapper no-ack path, but I did not run a live Discord interaction against current main.

## Solution Assessment

No. Blanket eager loading is not clearly the best fix because it conflicts with the manifest-first and selective activation direction; a defer-first wrapper/readiness contract is safer.
