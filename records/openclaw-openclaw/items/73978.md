## Summary

Keep open. Current main still only guarantees an immediate defer after a Discord native command object already exists, while plugin-owned native command specs are built from the runtime plugin command registry rather than manifest command aliases. Startup-lazy plugins such as Codex can still remain metadata-only until runtime activation, so this is not implemented on main and needs maintainer design around Discord command readiness and plugin laziness.

## What This Changes

The issue reports that Discord slash commands registered by plugins can miss Discord's 3-second acknowledgment window when plugin command registration is lazy.

## Best Possible Solution

Define one Discord-native readiness contract for plugin-owned slash commands: either deploy manifest-declared commands with a lightweight immediate-defer dispatcher that can activate the owner plugin, or only deploy commands whose handlers are loaded before gateway readiness. Keep /vc voice behavior on its own deferred path and validate it separately.

## Reproduction Assessment

Unclear: the issue has concrete logs and plausible steps, but this read-only review did not run a live Discord bot. A high-confidence reproduction should enable Discord native commands with a startup-lazy plugin such as Codex and capture interaction received-to-defer timing plus plugin registration timing.

## Solution Assessment

Unclear because there is no proposed patch, and current main does not solve startup-lazy plugin-owned native commands. The safer direction is an explicit Discord command readiness contract rather than broadly eager-loading every plugin at startup.
