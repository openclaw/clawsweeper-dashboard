## Summary

Keep open. Current main still protects already-created Discord native command objects with an up-front defer, but plugin-owned command deployment is still sourced from the runtime plugin command registry; manifest-only startup-lazy commands such as Codex are not enough evidence for a fixed plugin-command readiness path.

## What This Changes

The issue reports that Discord plugin-registered slash commands can miss Discord's 3-second interaction acknowledgement deadline when plugin command handlers are registered lazily.

## Best Possible Solution

Define one Discord-native readiness contract for plugin-owned slash commands: either predeploy manifest-declared command wrappers that can immediately defer and then activate the owner plugin, or deploy only commands whose handlers are loaded before gateway readiness. Keep /vc voice handling separate from generic plugin command dispatch.

## Reproduction Assessment

Unclear: the issue has concrete logs and plausible steps, but this read-only review did not run a live Discord bot. A high-confidence reproduction should enable Discord native commands with a startup-lazy plugin command such as /codex and capture startup command-spec registration plus received-to-defer timing.

## Solution Assessment

Unclear: there is no proposed patch, and current main only proves immediate defer for already-created native command objects. The safer direction is an explicit Discord-visible plugin command readiness contract rather than blindly eager-loading every plugin at startup.
