## Summary

Keep open. Current main still has a Discord native-command path where Discord-routed application commands only get acknowledged if a matching local command wrapper exists, while plugin command specs are still derived from runtime plugin command registration state.

## What This Changes

Ensure Discord plugin-registered slash commands have a local defer-first handler before Discord can route them, avoiding 3-second acknowledgement deadline misses.

## Best Possible Solution

Define a Discord-native plugin-command readiness contract so any command Discord can route has a local defer-first wrapper before interaction delivery, or is removed/reconciled during startup when the owning plugin is unavailable.

## Reproduction Assessment

Unclear. The issue includes concrete live logs and current main still shows a source-level no-ack path for missing local wrappers, but I did not run a live Discord smoke against current main.

## Solution Assessment

Unclear. The suggested eager registration may fix this symptom, but a wrapper/readiness or startup reconciliation contract is safer than broadly changing plugin activation semantics.
