## Summary

Keep open. Current main defers Discord native commands once a local command object exists, including registry-backed plugin commands, but it still has a plausible no-ack path for Discord commands that are registered remotely while missing from the local `client.commands` list. The issue discussion also provides concrete logs showing plugin command registration during interaction handling, so this is not stale cleanup.

## What This Changes

The issue reports that Discord native slash commands owned by plugins can miss Discord's three-second acknowledgement deadline when plugin command registration happens lazily during invocation.

## Best Possible Solution

Define an explicit Discord-native readiness contract for plugin-owned slash commands: either deploy only commands with handlers loaded before Discord readiness, or deploy manifest-declared placeholders that immediately acknowledge and then activate the owner plugin. Keep `/vc` on the dedicated Discord voice native path.

## Reproduction Assessment

Unclear: the issue has concrete logs and current source explains a high-probability no-ack path, but this review did not run a live Discord interaction. A high-confidence reproduction should enable native Discord commands, invoke a remotely registered but locally absent plugin command such as `/codex`, and capture received-to-defer plus plugin registration timing.

## Solution Assessment

Unclear: there is no patch to assess, and the suggested eager startup registration is not obviously the narrowest maintainable fix. A readiness/placeholder contract would be safer than loading every plugin at gateway startup.
