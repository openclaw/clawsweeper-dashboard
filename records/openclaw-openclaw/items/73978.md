## Summary

Keep open. Current main still acknowledges Discord native slash interactions only after a local command object exists, while plugin-owned command specs are still derived from the runtime plugin command registry rather than manifest-only aliases or a placeholder readiness contract. The prior ClawSweeper conclusion remains applicable on current main; no landed change in the central dispatch/spec files closes the plugin-command gap.

## What This Changes

Ensure Discord plugin-owned slash commands are registered or acknowledged before Discord's 3-second deadline so invocations do not produce Unknown interaction or double-ack errors.

## Best Possible Solution

Define an explicit Discord native-command readiness contract for plugin-owned slash commands: either deploy lightweight manifest-backed placeholders that acknowledge immediately and then activate/dispatch the owner plugin, or only deploy commands whose handlers are loaded before gateway readiness. Also reconcile stale remote commands so locally missing commands receive a controlled response rather than timing out.

## Reproduction Assessment

Unclear. The issue has concrete logs and current main shows a plausible no-ack path for remotely registered or manifest-only plugin commands missing from `client.commands`, but this review did not run a live Discord bot to capture interaction received-to-defer timing.

## Solution Assessment

Unclear because there is no proposed patch. The suggested eager startup registration is only safe if it preserves plugin laziness; the narrower maintainable direction is a Discord command readiness or placeholder contract with immediate acknowledgement.
