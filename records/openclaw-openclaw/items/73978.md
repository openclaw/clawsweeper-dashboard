## Summary

Keep open. Current main still has an immediate defer only after a Discord native command object already exists, while plugin-owned native command specs are still sourced from the runtime plugin command registry; startup-lazy manifest aliases such as Codex remain metadata until the plugin runtime registers handlers. The post-report/current-main changes I inspected do not provide a strong implemented-on-main close path.

## What This Changes

The issue reports that Discord slash commands registered by plugins can miss Discord's 3-second interaction acknowledgement deadline because plugin command registration appears to happen lazily during interaction handling.

## Best Possible Solution

Define one Discord-native readiness contract for plugin-owned slash commands: either predeploy manifest-declared commands with a lightweight immediate-defer dispatcher that activates the owner plugin, or only deploy commands whose handlers are loaded before gateway readiness. Keep voice /vc behavior separate from generic plugin command dispatch.

## Reproduction Assessment

Unclear: the issue has concrete logs and plausible steps, but this read-only review did not run a live Discord bot. A high-confidence reproduction should enable Discord native commands with a startup-lazy runtime alias such as /codex and capture command registration plus received-to-defer timing.

## Solution Assessment

Unclear because there is no proposed patch, and current main is not sufficient evidence for startup-lazy plugin-owned native commands. The safer direction is an explicit Discord-visible command readiness contract rather than blindly eager-loading every plugin at startup.
