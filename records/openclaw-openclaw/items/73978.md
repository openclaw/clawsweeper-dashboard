## Summary

Keep open. Current main immediately defers Discord native command objects once they exist, but the Discord startup path still builds plugin slash command specs from the runtime plugin command registry, not from manifest command aliases. Startup-lazy plugins such as Codex can still advertise runtime-slash aliases without being loaded into that registry before Discord interactions arrive, so there is no strong implemented-on-main close path for the plugin-registered command regression described here.

## What This Changes

The issue reports that Discord native slash commands provided by plugins can miss Discord's 3-second interaction acknowledgment deadline when plugin command registration happens during first interaction instead of before gateway readiness.

## Best Possible Solution

Define one Discord-native readiness contract for plugin-owned slash commands: either predeploy manifest-declared runtime-slash aliases with a lightweight immediate-defer dispatcher that activates the owner plugin before execution, or deploy only plugin commands whose handlers are loaded before gateway readiness. Keep Discord voice /vc behavior and generic plugin command dispatch covered by separate focused tests.

## Reproduction Assessment

Unclear: the issue has concrete logs and plausible steps, but this review did not run a live Discord bot. A high-confidence reproduction should enable Discord native commands with a startup-lazy runtime alias such as /codex, invoke it from Discord, and capture interaction received-to-defer timing plus plugin command registration timing.

## Solution Assessment

No: current main's immediate defer is the right shape for existing Discord command wrappers, but it is not sufficient for startup-lazy plugin-owned native commands. The safer solution is a manifest-backed readiness/activation contract, not simply eager-loading every plugin at startup.
