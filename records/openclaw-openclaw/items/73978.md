## Summary

Keep open. Current main still has a source-backed Discord application-command path that returns without acknowledging when Discord routes a command without a local wrapper, and the plugin native-command list still depends on runtime-registered command state; recent comments also add session-lock/event-loop saturation as a related root-cause path, so closing this as implemented or duplicate would be premature.

## What This Changes

The issue reports plugin-registered Discord slash commands such as `/vc`, `/voice`, and `/codex` missing Discord's acknowledgement window and failing with `Unknown interaction` or duplicate-ack errors.

## Best Possible Solution

Define a Discord native plugin-command readiness and reconciliation contract so every Discord-routable command has a local defer-first wrapper before interactions arrive, while keeping broader session-lock/event-loop fixes tracked in their own open issues.

## Reproduction Assessment

Unclear for a live end-to-end repro: the issue includes concrete 2026.4.26 logs and current source shows a no-local-wrapper path that misses acknowledgment, but I did not run a live Discord smoke against current main.

## Solution Assessment

No. The issue's blanket eager-loading suggestion is not clearly the best fit for the plugin architecture; a readiness/deploy reconciliation contract that preserves manifest-first loading and selective activation is safer.
