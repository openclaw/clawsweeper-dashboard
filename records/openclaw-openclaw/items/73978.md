## Summary

Keep open. Current main has defer-first handling when a Discord native command has a local wrapper, but it still has an unacknowledged application-command path when Discord routes a command with no local wrapper, and plugin native command specs still depend on runtime-registered plugin command state rather than a manifest-backed readiness contract.

## What This Changes

The issue requests that Discord plugin-registered slash commands be ready and deferred within Discord's 3-second interaction deadline instead of failing with Unknown interaction or already-acknowledged errors.

## Best Possible Solution

Define a Discord native plugin-command readiness and reconciliation contract so every command deployed to Discord has a local defer-first wrapper before interactions can arrive, while stale remote commands are cleaned up without broadly eager-loading unrelated plugins.

## Reproduction Assessment

No live high-confidence reproduction was established against current main. Source inspection does show a current no-local-wrapper application-command path that returns without acknowledgment, and the issue provides concrete live 2026.4.26 logs for the same Discord failure mode.

## Solution Assessment

No. The issue's suggested blanket eager registration is not clearly the safest solution for the plugin architecture; a narrower readiness/deploy reconciliation contract would preserve selective plugin activation while fixing the Discord interaction deadline.
