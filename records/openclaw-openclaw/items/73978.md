## Summary

Keep open. Current main still has a Discord application-command path that silently returns when the remotely invoked command has no local wrapper, while Discord plugin command specs are still derived from runtime `registerCommand()` state rather than manifest-backed startup placeholders.

## What This Changes

The issue reports that Discord plugin-registered slash commands miss Discord’s 3-second interaction acknowledgement deadline and fail with expired or double-ack interaction errors.

## Best Possible Solution

Define a Discord-native plugin-command readiness contract so any command Discord can route has a local defer-first wrapper before interaction delivery, or is removed/updated during startup when the owning plugin is unavailable.

## Reproduction Assessment

Unclear. The report has concrete live logs and current source shows a source-level no-ack path for missing local wrappers, but I did not run a live Discord smoke against current main.

## Solution Assessment

No. Blanket eager plugin loading is not clearly the best fix because it conflicts with manifest-first and selective activation; a defer-first wrapper/readiness contract is the safer direction.
