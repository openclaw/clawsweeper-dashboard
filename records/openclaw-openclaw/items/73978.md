## Summary

Keep open. Current main still has an unclosed readiness gap for Discord-native plugin slash commands: loaded Discord native wrappers defer promptly, but plugin command specs are still derived from the loaded runtime registry, while startup activation can leave command-owning plugins lazy. That means a remotely registered command can still arrive without a local `client.commands` entry to acknowledge it.

## What This Changes

The issue requests fixing Discord native slash commands registered by plugins so they acknowledge within Discord's 3-second interaction deadline instead of failing with `Unknown interaction` or double-ack errors.

## Best Possible Solution

Define an explicit Discord-native readiness contract for plugin-owned slash commands: only deploy or keep commands whose local handler or placeholder can acknowledge immediately, then activate the owning plugin after the ack using manifest ownership or a targeted activation plan.

## Reproduction Assessment

Unclear: the issue includes concrete logs and current source explains a high-probability no-ack path, but this review did not run a live Discord smoke. A high-confidence reproduction should enable Discord native commands, invoke a remotely registered but locally absent startup-lazy plugin command such as `/codex`, and capture received-to-defer plus plugin registration timing.

## Solution Assessment

Unclear: there is no patch to assess, and blanket eager plugin registration is not clearly the narrowest maintainable solution. A manifest-backed readiness or placeholder contract that acknowledges first and activates the owner deliberately looks safer.
