## Summary

Keep open. Current main defers quickly once a Discord-native local wrapper exists, but the interaction dispatcher still returns without any acknowledgement when Discord sends an application command that has no matching local `client.commands` entry, and plugin command specs still depend on activated runtime registry state.

## What This Changes

The issue reports that Discord plugin-registered slash commands can miss Discord's 3-second acknowledgement deadline because their local wrappers may be absent or registered too late.

## Best Possible Solution

Define a Discord-native plugin-command readiness contract that either deploys only commands backed by local immediate-ack wrappers or creates manifest-backed placeholders that defer first before targeted plugin activation and dispatch.

## Reproduction Assessment

Unclear: the report has concrete logs and current source shows a no-ack path for remote commands without local wrappers, but this review did not run live Discord. A high-confidence reproduction should enable Discord native commands, invoke a plugin command whose wrapper is absent at startup, and capture received-to-defer timing plus plugin registration timing.

## Solution Assessment

No: blanket eager registration is not clearly the best fix because it conflicts with the manifest-first, selective plugin activation direction. A narrower deployment/readiness or defer-first placeholder contract is the safer maintainable path.
