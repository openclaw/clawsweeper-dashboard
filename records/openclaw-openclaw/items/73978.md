## Summary

Keep open. Current main still has a plausible Discord-native plugin command readiness gap: loaded local wrappers defer immediately, but Discord startup builds plugin command wrappers only from the already-populated runtime plugin-command registry, while manifest `commandAliases` remain metadata and some plugins such as Codex are intentionally not startup-loaded.

## What This Changes

The issue asks OpenClaw to prevent plugin-registered Discord slash commands from missing Discord's 3-second interaction acknowledgement deadline.

## Best Possible Solution

Define a Discord-native plugin-command readiness contract: deploy only commands backed by local defer-first handlers, or install manifest-backed placeholders that immediately defer and then activate or dispatch the owning plugin deliberately.

## Reproduction Assessment

Unclear. The issue has concrete logs and current source shows a no-ack path when Discord invokes a command absent from local `client.commands`, but high-confidence reproduction still needs a live Discord native-command smoke with a startup-lazy plugin such as `/codex` and received-to-defer timing.

## Solution Assessment

Unclear. Blanket eager loading of every command-alias plugin is probably not the narrowest maintainable fix because it conflicts with manifest-first lazy activation; a deploy/readiness gate or defer-first placeholder contract looks safer.
