## Summary

Keep open. Current main still has a Discord-native readiness gap: loaded command wrappers defer promptly, but plugin-owned slash command specs still come from the loaded runtime registry, so a remotely registered plugin command that lacks a local wrapper can return without any interaction acknowledgement.

## What This Changes

The issue reports plugin-owned Discord slash commands missing Discord’s 3-second acknowledgement deadline and failing with expired or already-acknowledged interactions.

## Best Possible Solution

Define a Discord-native plugin-command readiness contract: deploy only commands with local immediate-ack handlers, or create manifest-backed placeholders that defer first and then deliberately activate/dispatch the owning plugin.

## Reproduction Assessment

Unclear: the issue has concrete logs and current source explains a high-probability no-ack path, but this review did not run a live Discord client interaction. A high-confidence reproduction should enable Discord native commands, invoke a remotely registered plugin command absent from local `client.commands`, and capture received-to-defer timing plus plugin registration timing.

## Solution Assessment

No: the suggested blanket eager registration is not clearly the narrowest maintainable fix. A placeholder/readiness contract that acknowledges first and then loads the owning plugin is safer than broad startup activation.
