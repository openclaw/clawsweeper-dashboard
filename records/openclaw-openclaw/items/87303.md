## Label Justifications

## Summary

Keep open: current main and the latest release still derive identity mention patterns with ASCII word boundaries, and the open linked PR remains the canonical fix path.

## What This Changes

The issue reports that single-character CJK nicknames in group chats, such as `@包`, are treated as plain text instead of resolving as mentions.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/89864 so the shared identity-derived mention helper treats single-character CJK names as standalone mention targets without matching substrings inside longer CJK words.

## Reproduction Assessment

Yes. Current main derives `\b@?包\b`, and a focused JavaScript regex check shows that pattern does not match `@包 你好`; a live DingTalk group reproduction was not run in this read-only review.

## Solution Assessment

Yes. The linked PR's shared-helper fix is the right owner boundary because identity-derived mention matching is exported through the plugin SDK and consumed by multiple channels.
