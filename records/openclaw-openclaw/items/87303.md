## Label Justifications

- `P2`: The report describes a concrete group-channel mention detection regression with a narrow shared helper fix path and limited blast radius.
- `impact:message-loss`: Single-character CJK mentions can be missed as routing signals, causing intended group mentions not to reach the agent.

## Summary

Keep open: current main and the latest release still derive identity mention patterns with ASCII-style word boundaries, while the linked implementation PR that closes this issue is open and unmerged.

## What This Changes

The issue reports that DingTalk group messages mentioning an agent with a one-character Chinese nickname, such as `@包`, are treated as plain text while multi-character names work.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/89864 so shared identity-derived mention matching treats single-character CJK names as standalone mention targets without matching substrings inside longer CJK words.

## Reproduction Assessment

Yes from source inspection: current main still wraps derived identity names with `\b@?...\b`, which is a known mismatch for single CJK characters in JavaScript regex. I did not run a live DingTalk reproduction in this read-only review.

## Solution Assessment

Yes: fixing the shared `deriveMentionPatterns` helper is the right owner boundary because multiple channel plugins consume `buildMentionRegexes`; the issue should stay open until the linked PR lands or maintainers choose a replacement.
