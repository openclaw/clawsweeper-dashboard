## Summary

Keep open: current main and the latest release still use ASCII-style word boundaries for identity-derived mention names, and the focused implementation PR that closes this issue is open but unmerged.

## What This Changes

The issue reports that DingTalk group-chat mentions fail for one-character Chinese nicknames while multi-character nicknames work.

## Best Possible Solution

Land or replace the linked shared mention-helper fix so identity-derived CJK names match standalone group mentions without matching substrings inside longer CJK text.

## Reproduction Assessment

Yes from source, but not with a live DingTalk room in this read-only pass. Current main emits `\b@?包\b`, and a direct JavaScript regex check shows that pattern does not match `@包 你好`.

## Solution Assessment

Yes: fixing the shared `deriveMentionPatterns` helper is the right owner boundary because identity-derived mentions are documented and multiple channel/plugin surfaces consume `buildMentionRegexes`. The issue should stay open until the linked PR lands or maintainers choose a replacement.
