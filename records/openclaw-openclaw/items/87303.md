## Summary

Keep open: current main and the latest release still derive identity mention names with ASCII word boundaries that fail standalone Han mention text, and the open linked PR is the candidate fix that should resolve this issue when merged.

## What This Changes

The issue reports DingTalk group-chat mentions failing when the target nickname is a single CJK character, so `@包` is treated as plain text while multi-character names work.

## Best Possible Solution

Keep this issue open and review or land https://github.com/openclaw/openclaw/pull/89864 so the shared mention helper handles standalone Han identity names without matching substrings inside longer Han text.

## Reproduction Assessment

Yes. Current main derives `\b@?包\b` for a one-character Han identity name, and a focused Node regex check shows that regex does not match standalone `@包 你好`.

## Solution Assessment

Yes. Fixing `deriveMentionPatterns` is the right owner boundary because `buildMentionRegexes` is shared through the runtime and plugin SDK; a DingTalk-only parser fix would leave sibling consumers with the same boundary bug.
