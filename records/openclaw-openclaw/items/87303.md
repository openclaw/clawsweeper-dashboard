## Summary

Keep open: current main and the latest release still derive identity-based mention regexes with ASCII word boundaries, and the open linked PR that closes this issue has not merged yet.

## What This Changes

The issue reports that DingTalk group mentions fail when an agent or user nickname is a single CJK character, while multi-character nicknames still match.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/89864 so the shared identity-derived mention matcher handles standalone single-character CJK names without matching inside longer CJK words.

## Reproduction Assessment

Yes, from source inspection: documented identity-derived mention patterns still compile a single CJK name through `\b@?...\b`, which does not provide the intended CJK boundary behavior. I did not run a live DingTalk group reproduction.

## Solution Assessment

Yes, the linked PR's shared-helper approach is the right layer because identity-derived mention patterns are core behavior and the helper is exported to channel/plugin consumers. A DingTalk-only parser change would be too narrow for the shared invariant.
