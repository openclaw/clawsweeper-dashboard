Current main and the latest release still use ASCII word-boundary-derived identity mention patterns, so the CJK mention bug remains open; the linked open PR is the canonical fix path and should be reviewed or replaced before this issue closes.

## What This Changes

The issue reports that DingTalk group chat mentions fail for single-character CJK nicknames such as `@包`, while multi-character nicknames work.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/89864, or replace it with an equivalent shared-helper fix that preserves false-positive protection for longer CJK text.

## Reproduction Assessment

Yes at source level: current main still derives `\b@?包\b` for an identity name, and a focused JavaScript check shows that regex does not match `@包 你好`. I did not run a live DingTalk group-chat reproduction.

## Solution Assessment

Yes, the linked PR is the right solution shape because it fixes the shared identity-derived mention helper used by channel plugins instead of adding a DingTalk-only workaround. The issue should stay open until that PR or an equivalent replacement merges.
