## Summary

Keep open: current main does not implement this PR’s polling-cadence change, and maintainer triage explicitly says it is related but not superseded by #75986. The PR still has a blocking Logs live-tail regression and a missing changelog entry.

## What This Changes

The PR extracts Control UI polling interval constants and changes Nodes/Logs/Debug polling from 5s/2s/3s to 30s/5s/10s in `ui/src/ui/app-polling.ts`.

## Best Possible Solution

Ship a narrow Control UI churn reduction that preserves loss-resistant Logs live tailing, tunes only the low-risk polling paths with evidence, and records the user-facing fix in the changelog while leaving broader churn/cache work to #69454.

## Reproduction Assessment

Yes for the PR-introduced regression: let the gateway log grow by more than the active `maxBytes` or line tail budget between cursor polls, and `readLogSlice` resets to the newest window while `loadLogs` replaces entries. The original progressive-sluggishness report is plausible but still rests on reported runtime evidence rather than a local reproduction here.

## Solution Assessment

No, this is not the best fix as written. The safer path is to keep Logs at the existing 2-second cadence or make log-tail cursor recovery lossless before increasing that interval, while handling Nodes/Debug cadence separately.
