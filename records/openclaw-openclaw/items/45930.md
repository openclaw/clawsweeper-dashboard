## Summary

This PR should stay open as needs-changes: it targets a real Control UI churn problem, but slowing active Logs polling from 2s to 5s can lose log lines with the current bounded cursor recovery, and maintainer overlap triage says #75986 does not supersede this narrower polling-cadence PR.

## What This Changes

The PR extracts Control UI polling intervals into named constants and changes nodes/logs/debug polling from 5s/2s/3s to 30s/5s/10s.

## Best Possible Solution

Keep the useful nodes/debug churn reduction only if live log-tail semantics are preserved, preferably by leaving active Logs polling at 2s until bounded tail recovery is made lossless.

## Reproduction Assessment

Yes for the PR-introduced regression: a log file that grows by more than the current 250 KB or 500-line tail budget between cursor requests takes the server reset path and loses older lines. The original sluggishness report itself remains based on reporter runtime observation, not a local reproduction here.

## Solution Assessment

No, this is not the best fix as written. The narrow maintainable path is to preserve the shorter Logs cadence or first make log recovery lossless, then reduce nodes/debug traffic without folding in the broader #69454/#75986 work.
