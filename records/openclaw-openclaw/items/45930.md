Keep this PR open. It is a useful Control UI polling-churn reduction, but the current diff still regresses active Logs live-tail reliability and lacks the required changelog entry; the maintainer comment also explicitly says #75986 is related but not a duplicate or superseding close for this PR.

## What This Changes

The PR adds named Control UI polling interval constants and changes Nodes/Logs/Debug timers from 5s/2s/3s to 30s/5s/10s in `ui/src/ui/app-polling.ts`.

## Best Possible Solution

Ship a narrow Control UI churn reduction that keeps named constants, reduces low-risk Nodes/Debug polling if accepted, preserves active Logs live-tail semantics, and records the user-facing fix in the changelog.

## Reproduction Assessment

Yes for the PR-introduced regression: let a busy gateway write more than the active 250 KB/500-line tail budget between cursor requests and `readLogSlice` resets to the newest window. The original dashboard sluggishness report remains runtime-observed rather than locally reproduced here.

## Solution Assessment

No, the PR is not the best solution as written. The safer path is to keep Logs at the current 2-second cadence or first make log-tail cursor recovery lossless, while handling Nodes/Debug churn separately.
