## Summary

Keep open: the churn-reduction goal is plausible, but the current diff introduces a Logs tab live-tail regression while recovery is still bounded; current main has not implemented this interval/constant change, and #69454 is still open rather than a shipped replacement.

## What This Changes

The branch extracts Control UI polling interval constants and changes nodes/logs/debug polling from 5s/2s/3s to 30s/5s/10s.

## Best Possible Solution

Ship a Control UI churn reduction that preserves live-tail semantics, preferably by keeping Logs polling at the current cadence until tail recovery is lossless and handling nodes/debug churn through tab-scoped or coordinated refresh logic.

## Reproduction Assessment

Yes for the blocking regression introduced by this patch: a log file that grows by more than the current 250 KB or 500-line tail budget between cursor requests takes the server reset path and loses older lines. The original sluggishness report remains externally observed rather than reproduced here.

## Solution Assessment

No, this PR is not the best solution as written. The safer narrow path is to keep Logs at 2s or first make `logs.tail` recovery lossless, then reduce nodes/debug traffic without expanding #69454's broader scope accidentally.
