## Summary

Keep this PR open, but it still needs changes before merge: the current diff slows active Logs polling without making log cursor catch-up loss-resistant, and the external PR does not include observable after-fix behavior proof. Maintainer overlap triage also explicitly says this PR is related to, but not superseded by, the merged Control UI responsiveness work.

## What This Changes

The PR extracts Control UI polling interval constants and changes Nodes/Logs/Debug polling cadences from 5s/2s/3s to 30s/5s/10s in `ui/src/ui/app-polling.ts`.

## Best Possible Solution

Keep the useful constant extraction and any validated low-risk Nodes/Debug churn reduction, but preserve the 2s active Logs cadence or make cursor catch-up lossless before changing Logs polling.

## Reproduction Assessment

Yes for the PR-introduced Logs regression by source inspection: active polling can fall outside the 500-line/250KB cursor window and `readLogSlice` then resets to the newest window. The original dashboard sluggishness improvement was not independently reproduced here and remains based on the PR author's runtime claim.

## Solution Assessment

No, not as written. The narrow maintainable fix is to preserve live-tail semantics first, either by keeping Logs at 2s or by adding lossless cursor catch-up before increasing that interval.
