## Summary

Keep this PR open, but it still needs changes before merge. The polling-cadence work is related to #45698 and explicitly not superseded by #69454 or #75986, but the current patch slows active Logs polling in a way that can skip log output under the existing bounded cursor recovery and it lacks the required changelog entry.

## What This Changes

This PR extracts Control UI polling interval constants and changes Nodes/Logs/Debug polling from 5s/2s/3s to 30s/5s/10s in `ui/src/ui/app-polling.ts`.

## Best Possible Solution

Ship a narrow Control UI churn reduction that keeps named polling constants, reduces low-risk Nodes/Debug churn where appropriate, preserves active Logs live-tail data semantics, and records the user-facing fix in the changelog.

## Reproduction Assessment

Yes for the PR-introduced regression: let a log grow by more than the active `maxBytes`/line tail budget between cursor requests and `readLogSlice` resets to the newest window, losing older bytes. The original dashboard sluggishness claim was not locally reproduced here and remains based on reported runtime observation.

## Solution Assessment

No, this is not the best solution as written. The safer path is to keep Logs at the current 2-second cadence or make log-tail cursor recovery lossless before increasing that interval, while handling Nodes/Debug churn separately.
