## Summary

Keep open: the PR targets a real Control UI churn problem, but it should not merge as-is because the slower Logs tab poll can lose log lines under the current bounded tail contract. The broader churn work in #69454 is relevant, while this branch needs at least a narrow Logs cadence fix and release-note coverage.

## What This Changes

The PR extracts Control UI polling interval literals in `ui/src/ui/app-polling.ts` and changes nodes/logs/debug polling from 5s/2s/3s to 30s/5s/10s.

## Best Possible Solution

Reduce unnecessary Control UI gateway churn without weakening live-tail semantics: keep Logs tab polling at the current short cadence unless the tail API/UI becomes lossless across larger cursor gaps, handle nodes/debug load through tab-scoped or coordinated churn-reduction work, and document any user-facing performance fix in the changelog.

## Reproduction Assessment

Yes for the introduced Logs tab regression: with the PR's 5s cadence, any gateway that writes more than the current 250 KB or 500-line tail budget between polls can trigger the server reset path and skip older lines. I did not run live UI/performance tests because this review is read-only.

## Solution Assessment

No, the PR is not the best form as written. A safer narrow revision is to keep the Logs tab at 2s, or first make `logs.tail` recovery lossless for larger gaps; broader gateway churn should be coordinated with #69454's tab-scoped/debounced approach.
