## What This Changes

The PR extracts Control UI polling intervals into named constants and lengthens Nodes, Logs, and Debug polling in `ui/src/ui/app-polling.ts`.

## Best Possible Solution

Ship a narrow Control UI polling cleanup that keeps named constants, reduces safe Nodes/Debug churn, preserves active Logs live-tail semantics or makes cursor catch-up lossless, and records the user-facing fix in the changelog.

## Reproduction Assessment

Yes for the PR-introduced regression by source inspection: a 5s Logs cadence can fall beyond the 250KB/500-line cursor budget, after which `readLogSlice` resets to the newest window. The original progressive dashboard sluggishness was not independently reproduced here.

## Solution Assessment

No, not as written. The safer repair is to keep Logs at the existing 2s cadence or implement lossless catch-up before increasing it, while applying the lower-risk Nodes/Debug polling cleanup separately.
