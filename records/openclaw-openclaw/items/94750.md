## Label Justifications

- `P1`: A current Discord channel workflow can silently lose conversational continuity after reset or restart, which is highly visible for active channel users.
- `impact:session-state`: The issue is about absent same-channel conversation context in a fresh session despite a stable Discord channel session key.
- `impact:security`: The requested recovery changes what guild-channel history may be injected into model context, so privacy and trust boundaries are part of the issue.

## Summary

Keep open: current main and the latest release still build Discord channel continuity from a provider-local pending-history map, so fresh idle/reset/gateway-restart sessions can start without durable same-channel recovery context. The remaining work is product- and privacy-sensitive because it changes what guild-channel content may be injected into a new model turn.

## What This Changes

Requests privacy-aware Discord channel/session recovery so fresh sessions after reset include bounded recent same-channel context.

## Best Possible Solution

Define a privacy-aware same-channel recovery contract for Discord fresh sessions, using a bounded durable source with sender attribution, first-turn/reset gating, diagnostics, regression tests, and redacted live proof.

## Reproduction Assessment

Yes, at source level. Current main creates Discord prompt history from a provider-local pending-history map, and the latest release has the same pending-only path, so reset/restart recovery cannot pull prior same-channel context when that map is empty.

## Solution Assessment

Unclear as a final product path. Bounded recovery is plausible, but maintainers need to choose the durable source, privacy default, config shape, and debug/status surface before implementation.
