## Label Justifications

- `P1`: A normal Discord guild-channel workflow can silently lose conversational continuity after reset or restart, which is highly visible for active channel users.
- `impact:session-state`: The issue is about absent same-channel context in a fresh session despite a stable Discord channel session key.
- `impact:security`: Any fix changes what guild-channel history is injected into model context, so privacy and sensitive-context boundaries are part of the issue.

## Summary

Keep open: current main and v2026.6.10 still build Discord guild context from a provider-local pending-history map, so a fresh monitor/session after reset or restart has no durable same-channel recovery window. The remaining work is privacy- and product-sensitive because it changes what guild-channel messages may be injected into a new model turn.

## What This Changes

The issue requests Discord same-channel session recovery so a fresh session after idle, manual reset, or gateway restart receives bounded recent channel context for ambiguous follow-ups.

## Best Possible Solution

Define a privacy-aware same-channel recovery contract for fresh Discord guild sessions, reusing existing transcript/history primitives where possible, with bounded attribution-preserving context, diagnostics, and tests before changing defaults.

## Reproduction Assessment

Yes, at source level: current main constructs Discord prompt history from a provider-local map, while docs define that history as pending-only, so an empty map after reset/restart cannot recover prior same-channel context. I did not run a live Discord guild repro in this read-only review.

## Solution Assessment

Unclear as a final product path: bounded recovery is plausible, but maintainers need to choose the durable source, privacy default, config shape, and debug/status surface before implementation.
