## Label Justifications

- `P1`: A current Discord channel workflow can silently lose conversational continuity after reset or restart, which is highly visible for active channel users.
- `impact:session-state`: The issue is about absent channel conversation context in a fresh session despite a stable session key.

## Summary

Keep open: current main still builds Discord channel continuity from a provider-local pending-history map, and the docs define that history as pending-only rather than durable reset/restart recovery. The older matching report is closed stale, while the broader shared-session and hook PRs do not own this same-channel recovery behavior.

## What This Changes

The issue asks for Discord guild/channel sessions to recover bounded recent same-channel context after idle, manual, or gateway reset so ambiguous follow-ups retain continuity.

## Best Possible Solution

Define privacy-aware Discord session recovery for fresh, reset, or restarted channel sessions, preferably reusing current history/transcript primitives before adding config, with diagnostics and regression/live proof.

## Reproduction Assessment

Yes, at source level. Discord history is recorded and read from a provider-local pending map, so a fresh provider/session cannot recover prior same-channel context through the existing history window; I did not rerun live Discord.

## Solution Assessment

Unclear as a final product path. Bounded recovery is plausible, but maintainers need to choose the source of truth, privacy default, config shape, and status/debug surface before implementation.
