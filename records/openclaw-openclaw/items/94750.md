## Label Justifications

- `P1`: A current Discord channel workflow can silently lose conversational continuity after reset or restart, which is highly visible for active channel users.
- `impact:session-state`: The issue is about absent same-channel conversation context in a fresh session despite a stable channel session key.
- `impact:security`: The requested recovery changes what guild-channel history may be injected into model context, so privacy and trust boundaries are part of the issue.

## Summary

Keep open: current main and the latest release still use Discord's process-local pending-history window, so a fresh Discord guild-channel session after reset/restart can start without durable same-channel recovery context. The remaining work is privacy- and product-sensitive, so it should not be auto-closed or sent straight to a fix PR.

## What This Changes

The issue requests bounded same-channel Discord history recovery after a session reset so ambiguous follow-ups retain recent conversation continuity.

## Best Possible Solution

Define a maintainer-approved, privacy-aware same-channel recovery contract for fresh Discord channel sessions, reusing existing transcript or channel-history primitives where possible and adding diagnostics plus regression/live proof.

## Reproduction Assessment

Yes, at source level. Current main creates Discord prompt history from a provider-local map and the latest release has the same pending-only path, so reset/restart recovery cannot pull prior same-channel context when that map is empty.

## Solution Assessment

Unclear as a final product path. Bounded same-channel recovery is plausible, but maintainers need to choose the durable source, privacy default, config shape, and debug/status surface before implementation.
