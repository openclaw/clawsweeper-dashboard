## Label Justifications

- `P1`: A current Discord channel workflow can silently lose conversational continuity after idle/reset/restart, which is highly visible for active channel users.
- `impact:session-state`: The issue is about absent same-channel conversation context in a fresh session despite stable Discord channel session keys.
- `impact:security`: The requested recovery changes what guild-channel history may be injected into model context, so privacy and trust boundaries are part of the issue.

## Summary

Keep open: current main and the latest release still build Discord guild-channel prompt history from a provider-local pending-history map, so a fresh idle/reset/gateway restart can still start without durable same-channel recovery context. The older stale report is superseded by this fresh repro, and https://github.com/openclaw/openclaw/pull/49914 only partially overlaps because it is about chat session switching rather than Discord recovery after reset.

## What This Changes

The issue requests privacy-aware Discord same-channel session recovery so fresh sessions can answer ambiguous follow-ups using bounded recent channel history.

## Best Possible Solution

Define a privacy-aware same-channel recovery contract for fresh Discord channel sessions, preferably reusing existing transcript or history primitives before adding new config, with visible diagnostics and regression/live proof.

## Reproduction Assessment

Yes, at source level. Current main builds Discord history from a provider-local pending map, and both current main and v2026.6.9 lack a durable same-channel recovery path when that map is empty after reset or restart.

## Solution Assessment

Unclear as a final product path. Bounded recovery is plausible, but maintainers need to choose the durable source, privacy default, config shape, and debug/status surface before implementation.
