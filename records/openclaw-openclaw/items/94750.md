## Label Justifications

- `P1`: A current Discord channel workflow can silently lose conversational continuity after reset or restart, which is highly visible for active channel users.
- `impact:session-state`: The issue is about absent same-channel conversation context in a fresh session despite a stable Discord channel session key.

## Summary

Keep open: current main and the latest release still build Discord guild/channel continuity from a provider-local pending-history map, so a fresh idle/reset/gateway restart can start without same-channel recovery context. The remaining fix is product- and privacy-sensitive because it changes what guild-channel history may be injected into a new model turn.

## What This Changes

Request bounded, privacy-aware Discord same-channel history recovery for fresh channel sessions after reset or restart so ambiguous follow-ups retain continuity.

## Best Possible Solution

Define a privacy-aware same-channel recovery contract in the Discord channel/session boundary, reusing existing transcript or history primitives where possible, with diagnostics and regression/live proof before adding new config.

## Reproduction Assessment

Yes, at source level. I did not run a live Discord replay, but current main and `v2026.6.9` only recover Discord guild/channel prompt history from a provider-local map, so a restart or fresh handler with an empty map cannot inject prior same-channel context.

## Solution Assessment

Unclear as a final product path. Bounded recovery is plausible, but maintainers need to choose the durable source, privacy default, config shape, and debug/status surface before implementation.
