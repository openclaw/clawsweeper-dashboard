## Label Justifications

- `P1`: A current Discord channel workflow can silently lose conversational continuity after reset or restart, which is highly visible for active channel users.
- `impact:session-state`: The issue is about absent channel conversation context in a fresh session despite a stable same-channel session key.

## Summary

Keep open: current main still only uses Discord's provider-local pending-history map, so a fresh idle/reset/restart session has no durable same-channel history recovery. The request remains product- and privacy-sensitive because enabling recovery could change what prior guild-channel context is injected by default.

## What This Changes

Request same-Discord-channel reset recovery that injects bounded recent channel context into fresh sessions so short follow-ups can continue prior discussion.

## Best Possible Solution

Define privacy-aware same-channel recovery for fresh Discord channel sessions, preferably reusing existing history or transcript primitives before adding new config, with diagnostics and regression/live proof.

## Reproduction Assessment

Yes, at source level. Current main creates Discord history from a provider-local map and session rollover mints a new session id, so reset/restart recovery cannot pull prior same-channel context from the existing history window.

## Solution Assessment

Unclear as a final product path. Bounded recovery is plausible, but maintainers need to choose the source of truth, privacy default, config shape, and debug/status surface before implementation.
