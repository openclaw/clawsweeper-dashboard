## Label Justifications

- `P1`: A current Discord channel workflow can silently lose conversational continuity after reset or restart, which is highly visible for active channel users.
- `impact:session-state`: The issue is about absent same-channel conversation context in a fresh session despite a stable channel session key.
- `impact:security`: The requested recovery changes what guild-channel history may be injected into model context, so privacy and trust boundaries are part of the issue.

## Summary

Keep open: current main and the latest release still use Discord's provider-local pending-history map, so a fresh idle/reset/gateway restart can start without durable same-channel recovery context. The remaining work is product- and privacy-sensitive because it changes what guild-channel history is injected into a new model turn.

## What This Changes

The issue requests privacy-aware Discord same-channel history recovery after idle, manual, or gateway session reset so short follow-ups keep recent conversation continuity.

## Best Possible Solution

Define a privacy-aware same-channel recovery contract for fresh Discord channel sessions, preferably reusing existing transcript or history primitives before adding new config, with diagnostics and regression/live proof.

## Reproduction Assessment

Yes, at source level. Current main creates Discord prompt history from a provider-local map and the latest release has the same pending-only path, so reset/restart recovery cannot pull prior same-channel context when that map is empty.

## Solution Assessment

Unclear as a final product path. Bounded recovery is plausible, but maintainers need to choose the durable source, privacy default, config shape, and debug/status surface before implementation.
