## Label Justifications

- `P1`: A current Discord channel workflow can silently lose conversational continuity after reset or restart, which is highly visible for active channel users.
- `impact:session-state`: The issue is about missing same-channel conversation context in a fresh session despite a stable session key.

## Summary

Keep open: current main and the latest release still only provide pending, in-memory Discord channel history, not durable same-channel recovery after reset or restart. The older matching report is closed as stale, and the broader session-history work does not replace this narrower Discord continuity decision.

## What This Changes

Requests Discord channel session recovery that injects bounded recent same-channel history after reset or restart so ambiguous follow-ups keep context.

## Best Possible Solution

Define a privacy-aware Discord session-recovery contract, then implement bounded same-channel rehydration for fresh/reset/restarted sessions with diagnostics and focused regression coverage.

## Reproduction Assessment

Yes at source level: Discord history is stored in a provider-local pending map and prompt assembly reads only that map. I did not run a live Discord repro in this read-only review, but the reporter supplied a fresh live observation from June 18-19, 2026 UTC.

## Solution Assessment

Unclear as a final design: bounded recovery is plausible, but maintainers need to choose whether the source is session transcripts, Discord API history, a new setting, or an existing config path, plus the guild-channel privacy default.
