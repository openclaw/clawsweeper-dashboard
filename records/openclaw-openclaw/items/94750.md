## Label Justifications

- `P1`: A current Discord channel workflow can silently lose conversational continuity after reset or restart, which is highly visible for active channel users.
- `impact:session-state`: The issue is about missing same-channel conversation context in a fresh session despite a stable channel session key.

## Summary

Keep open: current main and v2026.6.8 still build Discord channel context from provider-local pending history, so a fresh reset/restart session has no durable same-channel recovery path. The remaining work is real but needs maintainer product/privacy direction before an automated fix because it changes what guild-channel context enters a fresh model turn.

## What This Changes

Request to add privacy-aware Discord same-channel recent-history recovery after session reset or restart so vague follow-ups retain conversation continuity.

## Best Possible Solution

Define privacy-aware same-channel recovery for fresh Discord channel sessions, reusing existing transcript or channel-history primitives where possible, with agreed default/opt-in semantics, diagnostics, regression coverage, and live Discord proof.

## Reproduction Assessment

Yes at source level. Current main and v2026.6.8 use a provider-local pending history map for Discord prompt context, and the reporter also supplied a fresh live v2026.6.8 guild-channel reproduction; I did not run a separate live Discord transport repro in this read-only review.

## Solution Assessment

Unclear as a final product path. Bounded same-channel recovery is plausible, but maintainers should choose the source of truth, privacy default, config shape, and debug/status surface before implementation.
