## Label Justifications

- `P1`: A current Discord channel workflow can silently lose conversational continuity after reset or restart, which is highly visible for active channel users.
- `impact:session-state`: The issue is about absent channel conversation context in a fresh session despite a stable session key.

## Summary

Keep open: current main has an in-memory Discord group history window, but it does not provide durable same-channel history recovery after a fresh session, reset, or gateway restart. The older matching report was closed stale and no open PR appears to own this narrower behavior, so this issue should stay as the canonical maintainer product/design item.

## What This Changes

Request automatic, bounded Discord same-channel history recovery when a channel session starts fresh after reset, idle expiry, or gateway restart.

## Best Possible Solution

Define a privacy-aware Discord session-recovery design that rehydrates bounded same-channel context only for fresh/reset/restarted sessions, then implement it with diagnostics and focused regression coverage.

## Reproduction Assessment

Yes for the current behavior, from source inspection: Discord history is recorded in a provider-local map and prompt assembly reads only that map, so a fresh provider/session cannot recover prior channel context through the existing history window. I did not run a live Discord repro in this read-only review.

## Solution Assessment

Unclear: bounded recovery is a plausible direction, but the best fix needs maintainer judgment on whether to use session transcripts, Discord API history, or a new explicit setting, and on the privacy default for guild channels.
