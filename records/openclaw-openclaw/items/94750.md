## Label Justifications

- `P1`: A current Discord guild-channel workflow can silently lose conversational continuity after reset or restart, which is highly visible for active channel users.
- `impact:session-state`: The issue is about absent same-channel conversation context in a fresh session despite a stable Discord channel session key.
- `impact:security`: The requested recovery changes what guild-channel history may be injected into model context, so privacy and trust boundaries are part of the issue.

## Summary

Keep open: current main and v2026.6.9 still build Discord guild/channel continuity from a provider-local pending-history map, not durable same-channel recovery after reset or restart. The remaining fix is product- and privacy-sensitive because it changes what prior guild-channel content may enter a fresh model turn.

## What This Changes

Request asks OpenClaw to recover bounded recent same-Discord-channel history when a guild/channel session starts fresh after idle, manual reset, or gateway restart.

## Best Possible Solution

Define a privacy-aware same-channel recovery contract for fresh Discord channel sessions, preferably reusing existing transcript or history primitives before adding new config, with diagnostics and regression/live proof.

## Reproduction Assessment

Yes, at source level. Current main creates Discord prompt history from a provider-local map and v2026.6.9 has the same pending-only path, so reset or restart recovery cannot pull prior same-channel context when that map is empty.

## Solution Assessment

Unclear as a final product path. Bounded recovery is plausible, but maintainers need to choose the durable source, privacy default, config shape, and debug/status surface before implementation.
