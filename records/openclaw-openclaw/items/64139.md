## Summary

Keep open. Current main appears to have improved same-host CLI pairing through shared-secret loopback/local pairing paths, but it still does not prove or document the reported macOS LaunchAgent `/usr/bin/crontab` exec failure. The open generic SIGKILL diagnostics issue is related context, but it does not cover the crontab-specific behavior or the cron administration trust-boundary question.

## What This Changes

The issue requests fixing or clearly explaining macOS `crontab` commands being timed out or SIGKILLed through exec, plus reducing or documenting Gateway cron CLI pairing friction for non-local chat-driven workflows.

## Best Possible Solution

Keep this open until a maintainer reproduces `crontab -l`, stdin `crontab -`, and `crontab -e` through current exec on a macOS LaunchAgent install, then either fix the exec/PTY/stdin behavior or document a precise limitation. Treat Gateway cron CLI auth as a separate security/product decision that should preserve the existing admin-scope boundary.
