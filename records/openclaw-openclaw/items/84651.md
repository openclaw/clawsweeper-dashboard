## Label Justifications

- `P2`: This is a concrete WebChat UX problem with inaccessible assistant/tool content and limited blast radius across Control UI and Gateway chat history.
- `impact:message-loss`: The issue is about users seeing truncated assistant/tool content with no reliable UI path to recover omitted text.

## Summary

Keep open. Current main still bounds `chat.history` and the existing assistant/tool sidebar paths reuse the already-loaded text, while the linked implementation PR is open and uses closing syntax for this issue.

## What This Changes

The issue requests an on-demand WebChat reader that can show full assistant and tool content when the displayed history preview is long or truncated.

## Best Possible Solution

Review and either land or reject https://github.com/openclaw/openclaw/pull/84670; the desired end state is an additive bounded full-message fetch path wired to the sidebar without increasing default history payloads.

## Reproduction Assessment

Yes, at source level. Current main requests bounded `chat.history`, projection appends `...(truncated)...`, and the assistant/tool sidebars open the same loaded markdown or output text rather than fetching a full transcript entry.

## Solution Assessment

Yes as a product direction, but it is not implemented on current main. An on-demand bounded reader is the maintainable shape because it preserves lightweight history payloads while making truncation actionable; the exact Gateway/API contract still needs maintainer review through the linked PR.
