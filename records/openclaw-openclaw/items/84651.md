## Label Justifications

- `P2`: This is a concrete WebChat UX and retrieval improvement with real user impact but limited blast radius.
- `impact:message-loss`: Users can see assistant or tool content ending with a truncation marker and have no current UI path to recover the omitted text.

## Summary

Keep open. Current main still caps WebChat history and the assistant/tool side reader receives the already-loaded capped text; the open implementation candidate at https://github.com/openclaw/openclaw/pull/84670 closes this issue, so the issue should stay open until that PR lands or is replaced.

## What This Changes

The issue requests an explicit WebChat reader that can show full assistant and tool content when the main chat preview is long or truncated.

## Best Possible Solution

Land or deliberately replace https://github.com/openclaw/openclaw/pull/84670 with an additive bounded full-message reader that preserves default history caps and shows an explicit unavailable state when full content cannot be returned.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: load a long assistant history row, Control UI requests `chat.history` with `maxChars: 4000`, Gateway appends `...(truncated)...`, and the sidebar opens that same markdown string. I did not run a live browser repro in this read-only review.

## Solution Assessment

Yes as a direction: an on-demand full-message reader is the maintainable shape because it keeps default history payloads bounded. The exact additive Gateway method and UI unavailable-state contract still need maintainer review, currently via https://github.com/openclaw/openclaw/pull/84670.
