## Summary

Keep open. Current main still returns bounded WebChat history and the existing assistant/tool side panels only receive the already-loaded text, while the related open issues cover cap policy or tool-summary UX rather than the requested on-demand full-message reader.

## What This Changes

Request to add a WebChat full-message reader for truncated or long assistant messages and tool outputs without increasing the default chat-history payload.

## Best Possible Solution

Add explicit truncation/long-content metadata plus an additive bounded full-content retrieval path, then have the WebChat sidebar fetch assistant and tool text on demand.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: load a long assistant message through WebChat history and the client requests a 4,000-character cap while gateway projection can append `...(truncated)...`; the sidebar then opens that same loaded markdown/output. I did not run a live browser session because this review is read-only.

## Solution Assessment

Yes as a direction: an on-demand full-message sidebar preserves bounded history payloads while giving users an explicit recovery path. The exact Gateway route or history-query extension still needs maintainer API/UX decision.
