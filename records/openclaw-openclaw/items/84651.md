## Label Justifications

- `P2`: This is a concrete WebChat UX and content-access problem with real user impact but limited blast radius to Gateway history and Control UI reader behavior.
- `impact:message-loss`: Users can see assistant/tool content replaced by a truncation marker with no current UI path to recover the omitted text.

## Summary

Keep open. Current main still serves bounded WebChat history and the assistant/tool side panels reuse the already-loaded text, while the linked full-message-reader implementation is an open closing PR rather than merged code.

## What This Changes

Request an explicit WebChat full-message reader/on-demand fetch path for long or truncated assistant and tool messages without increasing the default history payload.

## Best Possible Solution

Review and land, revise, or reject https://github.com/openclaw/openclaw/pull/84670; the issue should close only after an additive bounded full-message reader contract is merged.

## Reproduction Assessment

Yes. Source-level reproduction is clear: load a long assistant/tool history row, `chat.history` can cap it at 4,000/8,000 characters with `...(truncated)...`, and the current sidebar opens the same already-loaded text.

## Solution Assessment

Yes as a direction, but not complete on main. A bounded on-demand reader is more maintainable than inflating every history payload; the exact Gateway method and unavailable-state semantics still need maintainer acceptance through the linked PR.
