## Summary

Keep open. Current main still caps WebChat history and the sidebar opens the already-loaded assistant/tool text, while https://github.com/openclaw/openclaw/pull/84670 is an open closing PR for the requested full-message reader.

## What This Changes

Add a WebChat side-panel reader with bounded on-demand full-content fetch for truncated assistant and tool messages without increasing default history payloads.

## Best Possible Solution

Review and either land or close https://github.com/openclaw/openclaw/pull/84670; the desired end state is an additive bounded full-message retrieval contract wired into the WebChat reader for assistant and tool content.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: create a long assistant or tool message, load WebChat history, and the client requests a capped `chat.history` payload that the sidebar then reopens as-is. I did not run a live browser session because this review is read-only.

## Solution Assessment

Yes as a direction: a bounded on-demand reader is safer than raising default history caps or expanding huge text inline. The exact Gateway method and unavailable-state contract still need maintainer review through the linked PR.
