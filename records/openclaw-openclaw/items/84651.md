## What This Changes

Request an on-demand WebChat full-message reader for truncated assistant and tool output without substantially increasing the default history payload.

## Best Possible Solution

Review and either land or reject the open full-message reader PR; if it lands, close this issue as implemented, otherwise keep this issue as the canonical Gateway/UI reader design target.

## Solution Assessment

Yes as a direction: an on-demand side reader is preferable to expanding huge content inline or raising default payload caps. The exact additive Gateway method and unavailable-state contract still need maintainer review through the linked PR.
