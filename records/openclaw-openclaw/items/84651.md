## Summary

Keep open. Current main still serves bounded WebChat history and the assistant/tool side-panel actions reuse the already-loaded text, while the linked open implementation PR is the right place to review the new full-message fetch contract before closing this issue.

## What This Changes

Request an explicit WebChat full-message reader that can open full assistant and tool content when the displayed history preview is truncated.

## Best Possible Solution

Land a reviewed additive Gateway/UI implementation that fetches one bounded full transcript entry on demand and shows an honest unavailable state when full content cannot be returned.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: load a long stored assistant/tool message through WebChat history, receive a capped preview with `...(truncated)...`, then open the existing side panel and see the same capped text.

## Solution Assessment

Yes as a direction, but not yet implemented on main. The maintainable solution is an additive on-demand full-message read path plus sidebar plumbing, with maintainers deciding the exact Gateway protocol and identity contract.
