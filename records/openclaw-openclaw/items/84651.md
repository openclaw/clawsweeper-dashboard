## Summary

Keep open. Current main still has the reported WebChat limitation, and the open linked PR is the right implementation candidate rather than a reason to close this issue before merge.

## What This Changes

Request an explicit WebChat full-message reader that can open truncated assistant messages and tool outputs without increasing the default history payload.

## Best Possible Solution

Review and land or reject the linked bounded full-message reader PR, then close this issue only after the accepted implementation reaches main.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: load a long assistant or tool message through WebChat history, receive a capped `chat.history` projection with `...(truncated)...`, then open the existing sidebar and see the same truncated string.

## Solution Assessment

Yes as a direction. A bounded on-demand reader keyed by stable transcript identity is more maintainable than raising default history caps or storing full hidden text in the DOM, but the final Gateway/API contract needs maintainer acceptance.
