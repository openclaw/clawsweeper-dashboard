## Label Justifications

- `P2`: This is a concrete WebChat UX/data-access problem with limited blast radius but real user impact for long assistant and tool outputs.
- `impact:message-loss`: Users can see truncated assistant/tool content with no current UI path to recover the omitted text.

## Summary

Keep open: current main still caps WebChat history and the sidebar only opens already-loaded assistant/tool text, while the linked fix PR is still open rather than merged.

## What This Changes

The issue requests a WebChat full-message reader that can fetch long or truncated assistant/tool content on demand instead of repeating capped history text.

## Best Possible Solution

Land an additive bounded full-message retrieval path, or an equivalent history-query extension, and wire the WebChat sidebar to fetch assistant and tool content on demand while keeping default history payloads bounded.

## Reproduction Assessment

Yes, at source level: create a long assistant or tool message, load WebChat history, and the client requests a capped `chat.history` payload while the sidebar opens the same truncated markdown/output text. I did not run a live browser reproduction in this read-only review.

## Solution Assessment

Yes as a direction: an on-demand reader is the maintainable fix because it preserves bounded history payloads while giving users a deliberate full-content path. The exact Gateway method and unavailable-state contract still need maintainer API/product acceptance.
