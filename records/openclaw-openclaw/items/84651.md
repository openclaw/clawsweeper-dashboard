- `impact:message-loss`: The issue is about users seeing truncated assistant/tool content with no UI path to recover the omitted text.

## Summary

Keep open. Current main still bounds WebChat history and the existing sidebar only receives the already-loaded assistant/tool text, while the related issues cover truncation limits or tool-summary UX rather than an on-demand full-message reader.

## What This Changes

The issue requests a WebChat full-message reader that can fetch and display complete assistant or tool content when the main transcript preview is truncated.

## Best Possible Solution

Add an explicit truncated/long-content contract plus an additive bounded full-message retrieval path, then wire the WebChat sidebar to fetch assistant and tool text on demand without increasing the default history payload.

## Reproduction Assessment

Yes, source-level reproduction is clear: create a long assistant message, load WebChat history, and the client requests a 4,000-char cap while projection appends `...(truncated)...`; the sidebar then opens that same truncated markdown. I did not run a live browser session because this review is read-only.

## Solution Assessment

Yes as a direction: an on-demand full-message sidebar is the maintainable shape because it preserves bounded history payloads while giving users a deliberate escape hatch. The exact Gateway method or history-query extension still needs maintainer API/UX decision before implementation.
