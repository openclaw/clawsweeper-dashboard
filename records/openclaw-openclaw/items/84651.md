## Summary

Keep open. Current main still caps WebChat history and passes already-projected assistant/tool strings to the side panel; related open items address truncation limits or tool-summary noise, not the requested on-demand full-message reader.

## What This Changes

Request to add a WebChat full-message reader for long or truncated assistant/tool messages without expanding huge text directly in the chat thread.

## Best Possible Solution

Add explicit truncation metadata plus an additive bounded full-message retrieval path, then wire the WebChat sidebar to fetch assistant and tool content on demand without increasing default history payloads.

## Reproduction Assessment

Yes. Source-level reproduction is clear: load a long assistant or tool message through WebChat history, the client requests maxChars 4000 and Gateway projection appends the truncation sentinel, then the sidebar opens the same already-truncated string.

## Solution Assessment

Yes as a direction. An on-demand full-message sidebar is the maintainable shape because it keeps default history bounded; the exact Gateway method, message identity, and unavailable-state semantics still need maintainer API/UX judgment.
