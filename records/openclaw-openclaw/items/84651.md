## Summary

Keep open: current main still truncates bounded WebChat history and reuses the already-loaded assistant/tool text in the side reader, while the open linked PR is the active implementation path and has not merged.

## What This Changes

The issue requests an explicit WebChat full-message reader that can fetch truncated assistant and tool content on demand without increasing the default chat history payload.

## Best Possible Solution

Review and either land or explicitly reject the linked PR’s additive bounded per-message reader, keeping default history payloads bounded while giving WebChat a real full-content escape hatch.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: create a long assistant/tool history entry, load WebChat history with the 4,000-character client cap, and the sidebar opens the same truncated markdown or output text. I did not run a live browser session because this review is read-only.

## Solution Assessment

Yes as a direction: an on-demand full-message sidebar is narrower than raising history caps or storing full hidden text in the DOM. The exact additive Gateway contract remains a maintainer API/product decision, and https://github.com/openclaw/openclaw/pull/84670 is the active candidate.
