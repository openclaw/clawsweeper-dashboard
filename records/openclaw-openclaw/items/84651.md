- `impact:message-loss`: Users can see truncated assistant or tool content with no current UI path to recover the omitted text.

## Summary

Keep open. Current main still bounds WebChat history and the sidebar opens only the already-loaded assistant/tool text, and the linked fix PR is open with closing syntax, so this issue should stay open until that PR lands or is replaced.

## What This Changes

Request an explicit WebChat side-reader flow for long or truncated assistant and tool messages, with bounded on-demand full-content retrieval instead of expanding huge text inline.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/84670 or an equivalent additive bounded reader after maintainers confirm the protocol/UI shape and proof, while keeping default `chat.history` lightweight.

## Reproduction Assessment

Yes. Source-level reproduction is clear: load a long assistant/tool history entry, the client asks for a 4,000-character `chat.history` cap, projection can append `...(truncated)...`, and the sidebar opens the same loaded markdown or output text.

## Solution Assessment

Yes as a direction. An additive on-demand full-message reader is the maintainable shape because it preserves bounded history payloads; the exact Gateway method and unavailable-state contract still need maintainer review.
