## Label Justifications

- `P2`: This is a concrete WebChat usability and content-access issue with real impact but limited blast radius.
- `impact:message-loss`: Users can see assistant/tool content end in a truncation marker with no current UI path to recover the omitted text.

## Summary

Keep open. Current main still uses bounded WebChat history plus a sidebar that only receives the already-loaded assistant/tool text, and the open linked pull request is the active implementation path rather than a reason to close this issue before merge.

## What This Changes

The issue requests an explicit WebChat reader that can open full assistant and tool content when the main chat preview is long or truncated.

## Best Possible Solution

Review and either land or supersede https://github.com/openclaw/openclaw/pull/84670 with an additive bounded full-message retrieval contract and sidebar fetch path while keeping default history payloads lightweight.

## Reproduction Assessment

Yes. Source inspection shows Control UI requests chat.history with maxChars 4000, Gateway appends ...(truncated)... past the cap, and the current assistant/tool sidebars open the same loaded text rather than fetching full content.

## Solution Assessment

Yes as a product direction, but the exact implementation still needs maintainer review. An on-demand full-message reader is narrower than raising default history payloads and avoids expanding huge content inline.
