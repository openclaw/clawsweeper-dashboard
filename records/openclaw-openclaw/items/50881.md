## Summary

Keep this PR open as useful source work, but do not merge it as-is: current main has already shipped the `already_reacted` add-side fix, while the remove-side `no_reaction` behavior still needs a small current-main adaptation and this branch still drops opts on the remove-own listing path.

## What This Changes

The PR changes Slack reaction helpers to treat duplicate adds and missing removals as idempotent no-ops, and routes remove-own cleanup through the removal helper.

## Best Possible Solution

Land a narrow current-main Slack patch that keeps the shipped add-side behavior, treats `no_reaction` as idempotent success in `removeSlackReaction`, routes remove-own cleanup through that helper without dropping options, and adds focused tests plus a changelog entry.

## Reproduction Assessment

Yes. Mock a Slack `WebClient` whose `reactions.remove` rejects with `{ data: { error: "no_reaction" } }`, then call `removeSlackReaction` or `removeOwnSlackReactions`; current main will reject on the remove path, while the add-side `already_reacted` case is already covered by `actions.reactions.test.ts`.

## Solution Assessment

No as this PR currently stands. The helper-level idempotency direction is right, but the branch is stale against current config/client contracts and still drops opts when listing reactions; a narrow current-main adaptation is the safer fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Forward opts when listing own Slack reactions:** `extensions/slack/src/actions.ts:142`
  - body: In this branch, `getClient` resolves a token before returning `opts.client`. `removeOwnSlackReactions` calls `listSlackReactions` with only `{ client }`, so callers whose valid `token` or `accountId` lived only in `opts` can throw before the injected client is used. Forward `{ ...opts, client }` here or make `getClient` prefer `opts.client` first.
