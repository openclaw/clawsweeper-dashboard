## Summary

Keep open: current main still lacks a Gateway/Control UI hard-reset surface, and the existing reset and live-session paths do not reliably abandon a persisted Claude CLI resume binding for this recovery case.

## What This Changes

The issue requests an explicit hard-recovery control that abandons a Claude CLI resume token when live-session restart keeps reusing stuck session state.

## Best Possible Solution

Expose a deliberate documented hard-recovery control that reuses the existing CLI-binding clear semantics, preserves OpenClaw-visible history, and updates the Gateway protocol plus Control UI consistently.

## Reproduction Assessment

No, there is not a high-confidence live reproduction of the corrupted Claude transcript loop. Source inspection does show that current main can restart a Claude live subprocess while a persisted Claude CLI resume binding remains available, and the requested hard-reset surface does not exist.

## Solution Assessment

Unclear: a deliberate hard-reset control is plausible, but maintainers should decide whether it should be a new `sessions.hardReset` method, a `sessions.reset` option, or a Control UI action backed by the existing `/reset soft` semantics.
