## Label Justifications

- `P1`: The report describes a real stuck session workflow where restart reused corrupted Claude CLI session state and user retries could not recover without a full reset.
- `impact:session-state`: The affected state is the persisted CLI resume binding and transcript/session association that can make a fresh turn re-enter stale context.

## Summary

Close: current main implements the hard-recovery path by making Gateway `sessions.reset` clear CLI provider bindings for non-subagent sessions, with regression coverage and the Control UI already calling that RPC. The fix is on current main only; it is not present in `v2026.5.18`.

## What This Changes

The issue asks for a hard recovery/reset path that discards a stuck Claude CLI resume binding after restarts while preserving OpenClaw-visible chat history.

## Best Possible Solution

Keep the current `sessions.reset` behavior as the hard recovery surface, ship it in the next release, and track any distinct automatic-restart corruption only if it has a fresh reproduction after this reset fix.

## Reproduction Assessment

No live reproduction of the original corrupted Claude JSONL loop was rerun. The stale-binding failure is high-confidence from source inspection, the merged PR’s before/after evidence, and current regression tests that exercise the binding-clearing path.

## Solution Assessment

Yes: fixing existing `sessions.reset` is narrower than adding a parallel `sessions.hardReset` RPC and now gives the Gateway and Control UI a deliberate recovery path while preserving the spawned-subagent continuity contract.
