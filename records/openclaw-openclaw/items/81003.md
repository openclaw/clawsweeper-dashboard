## Summary

Keep open: current main still lacks a Gateway or Control UI hard-recovery surface, while the existing reset path preserves Claude CLI resume bindings and the CLI runner will resume when a stored binding is available. The report is actionable as a recovery/API design issue, but it needs maintainer product direction before it is a safe automated fix.

## What This Changes

The issue requests a deliberate hard-recovery action that abandons a Claude CLI resume token after restart while preserving OpenClaw-visible chat history.

## Best Possible Solution

Expose a deliberate, documented hard-recovery control that reuses the existing CLI-binding clear semantics, preserves OpenClaw-visible history, and updates the Gateway protocol plus Control UI consistently.

## Reproduction Assessment

No. I do not have the corrupted Claude CLI transcript/log artifact to replay the synthetic-output loop, but source inspection on current main shows `sessions.reset` preserves the binding and CLI dispatch will resume when a stored binding is available.

## Solution Assessment

Unclear. A deliberate hard-recovery control is plausible, but maintainers should choose whether this should be a new `sessions.hardReset` method, a `sessions.reset` option, or a Control UI action backed by the existing `/reset soft` semantics.
