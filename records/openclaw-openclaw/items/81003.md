## Summary

Keep open: current main has a shipped slash-command recovery path that clears CLI resume bindings, but it does not provide the requested Gateway/Control UI hard-reset surface for this failure mode.

## What This Changes

Request an explicit hard-recovery session control that closes a stuck Claude live session, discards the next Claude CLI resume token use, and preserves OpenClaw chat history.

## Best Possible Solution

Expose a deliberate documented hard-recovery control that reuses the existing CLI-binding clear semantics, preserves OpenClaw-visible history as intended, and updates the Gateway protocol plus Control UI consistently.

## Reproduction Assessment

No, there is not a high-confidence live reproduction of the corrupted Claude transcript loop. Source inspection does show current main can restart a Claude live process while still reusing the persisted Claude CLI resume binding, and no hard-reset RPC/UI surface exists.

## Solution Assessment

Unclear: an explicit hard-reset control is a plausible direction, but maintainers should decide whether this should be a new `sessions.hardReset` method, a `sessions.reset` option, or a Control UI wrapper around the existing `/reset soft` semantics.
