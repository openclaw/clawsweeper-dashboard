## Summary

Keep open: current main still lacks a Gateway/Control UI hard-reset surface, and the existing `sessions.reset` path intentionally preserves CLI resume bindings. The useful existing recovery path is `/reset soft`, but exposing that semantics through the Gateway/UI is a new API/UX decision rather than a conservative cleanup close.

## What This Changes

The issue requests a deliberate hard-recovery session control that discards the Claude CLI resume token after a restart-stuck state.

## Best Possible Solution

Add a deliberate, documented hard-recovery path that clears provider-owned CLI session bindings while preserving OpenClaw chat history, with protocol, Gateway, Control UI, docs, and tests aligned.

## Reproduction Assessment

No, there is not a high-confidence live reproduction of the corrupted Claude transcript loop. Source inspection does show that current main can preserve and reuse a stored Claude CLI resume binding through Gateway reset/restart paths, and no hard-reset RPC/UI surface exists.

## Solution Assessment

Unclear: a deliberate hard-reset control is plausible, but maintainers should choose whether it belongs as a new Gateway method, an option on `sessions.reset`, or a UI path that reuses `/reset soft` semantics.
