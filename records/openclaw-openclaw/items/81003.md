## Summary

Keep open: current main still lacks the requested Gateway/Control UI hard-recovery surface, and the existing restart path can continue using a stored Claude CLI resume binding. This is useful, but it needs maintainer API/UX direction before implementation.

## What This Changes

Request a deliberate hard-recovery control that closes a stuck Claude live session and forces the next CLI run to start without the stored resume token while preserving OpenClaw chat history.

## Best Possible Solution

Expose a deliberate, documented hard-recovery control that clears stored CLI resume bindings for recovery while preserving OpenClaw-visible chat history, with Gateway protocol, Control UI, docs, and tests aligned.

## Reproduction Assessment

No, not for the full incident: there is no high-confidence live reproduction of the corrupted Claude transcript loop. Source inspection does confirm the missing hard-reset surface and the restart/resume path that can keep reusing a stored CLI session binding.

## Solution Assessment

Unclear: a hard-recovery control is a plausible fix, but maintainers should choose the public API shape before implementation. The narrowest maintainable solution likely reuses the existing clearAllCliSessions semantics instead of inventing unrelated recovery state.
