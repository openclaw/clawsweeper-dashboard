## Summary

Keep open: current main still has no Gateway or Control UI hard-recovery surface, and the current reset/resume path can preserve a Claude CLI resume binding. The existing `/reset soft` clear path is useful evidence, but deciding whether to expose this as `sessions.hardReset`, a reset option, or a UI action needs maintainer product/API judgment.

## What This Changes

The issue requests a deliberate hard-recovery/reset path that abandons a Claude CLI resume token when restart-based recovery keeps reusing a stuck session.

## Best Possible Solution

Expose a deliberate documented hard-recovery control that reuses the existing CLI-binding clear semantics, preserves OpenClaw-visible history, and updates the Gateway protocol plus Control UI consistently.

## Reproduction Assessment

No, there is not a high-confidence live reproduction of the corrupted Claude transcript loop. Source inspection does show the missing hard-reset surface and the resume-binding path that can keep `useResume=true` when a Claude CLI binding remains stored.

## Solution Assessment

Unclear: a deliberate hard-recovery control is plausible, but maintainers should choose whether it belongs as `sessions.hardReset`, a `sessions.reset` option, or a Control UI action backed by existing `/reset soft` clear semantics.
