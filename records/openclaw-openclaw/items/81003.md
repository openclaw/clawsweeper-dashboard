## Label Justifications

- `P1`: The reported failure left a user-facing agent session stuck in a repeated stale-response loop where normal restart recovery could not clear provider-side session state.
- `impact:session-state`: The issue is about stale Claude CLI resume bindings causing OpenClaw session state to drift from the intended reset/recovery state.

## Summary

Close as implemented on current main: the merged reset fix now clears stored CLI resume bindings for non-subagent Gateway resets, so the existing reset path provides the hard recovery this issue needed without adding a new protocol method. This is not in the latest tagged release I checked, so the close is based on current main.

## What This Changes

The issue asks for a hard recovery path that abandons a stuck Claude CLI resume token instead of restarting the same provider-side session.

## Best Possible Solution

Keep the current `sessions.reset` semantics as the deliberate hard-recovery path for non-subagent sessions, preserve spawned-subagent continuity, and let the next release carry the fix to users.

## Reproduction Assessment

No current-main failing reproduction remains: source and regression tests show non-subagent `sessions.reset` now clears the stored CLI resume fields. I did not live-reproduce the original corrupted Claude JSONL loop, but the reset binding path it depended on is fixed.

## Solution Assessment

Yes: using the existing `sessions.reset` RPC and Control UI reset path avoids a new protocol surface while solving the stale-resume recovery failure. The implementation keeps the intentional spawned-subagent exception tested.
