## Summary

Keep open: current main still lacks a Gateway/Control UI hard-reset surface, while `sessions.reset` preserves CLI resume bindings and CLI dispatch will resume when a binding is present. `/reset soft` already clears those bindings, but it is not exposed as the requested deliberate UI/RPC recovery control.

## What This Changes

The issue requests a hard-recovery reset path that abandons Claude CLI resume state while preserving OpenClaw-visible chat history.

## Best Possible Solution

Expose a deliberate documented Gateway/Control UI recovery control that clears CLI resume bindings with the existing clear-session semantics, preserves chat history, and keeps the operator/admin contract explicit.

## Reproduction Assessment

No, there is not a high-confidence live reproduction of the corrupted Claude transcript loop. Source inspection does show the relevant current-main path: Gateway reset preserves CLI bindings, and CLI execution resumes when those bindings remain available.

## Solution Assessment

Unclear: a deliberate hard-recovery control is plausible, but maintainers should choose whether the maintainable API is a new `sessions.hardReset`, a `sessions.reset` option, or a Control UI action that reuses `/reset soft` semantics.
