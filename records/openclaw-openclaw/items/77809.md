## Summary

Keep open: the patch improves a thrown invalid-channel diagnostic, but source and test evidence show the reported unconfigured-login path can complete with `started: false`, so the new catch-only hint can still miss the user-visible problem.

## What This Changes

The PR changes gateway `channels.start`/`channels.stop` invalid-channel messages to list available channel plugins and appends a config-enable hint to the CLI post-login restart warning when restart throws.

## Best Possible Solution

Surface the unconfigured or non-started start result as actionable CLI guidance, add focused regression coverage and a changelog entry, and leave any auto-config behavior to explicit maintainer product judgment.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: `channels.start` can return a successful `{ started: false }` response after the manager records an unconfigured account as stopped. I did not run a live gateway scenario in this read-only review.

## Solution Assessment

No. The PR improves one thrown error branch, but the safer fix is to handle the successful non-started result or make the gateway return an explicit unconfigured error, then verify the actual CLI output.
