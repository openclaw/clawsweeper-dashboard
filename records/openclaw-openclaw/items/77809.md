## Summary

Keep open: the PR improves one thrown invalid-channel diagnostic, but it still misses the successful `{ started: false }` path used when a configured channel plugin remains stopped or unconfigured, and it lacks after-fix login proof plus the required changelog entry.

## What This Changes

The PR changes gateway `channels.start`/`channels.stop` invalid-channel errors to list channel plugins and adds a CLI configuration hint when the post-login gateway restart throws.

## Best Possible Solution

Surface unconfigured or non-started post-login startup results as actionable CLI guidance, add focused regression coverage and a changelog entry, and verify the actual `channels login` output from a modified build.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: `channels.start` can return `{ started: false }` after `startChannel` records an unconfigured runtime instead of throwing; I did not run a live gateway scenario in this read-only review.

## Solution Assessment

No. The catch-only hint improves a thrown error branch, but the narrower maintainable fix is to inspect the successful non-started result or make the gateway return an explicit unconfigured error, then cover that behavior in tests.
