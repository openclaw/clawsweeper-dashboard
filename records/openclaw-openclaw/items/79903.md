## Summary

Keep open: current main has internal usage-family metadata, alias-aware sessionId selection, usage rollups, and ACP subagent lineage metadata, but it still does not expose the requested public durable session-family/current-continuation discovery contract; the issue is also paired with open implementation PRs.

## What This Changes

Expose a public helper or projection for resolving a sessionId to canonical session ownership and known family lineage across rotations.

## Best Possible Solution

Settle and land one narrow, versioned Gateway/SDK contract that returns resolved/ambiguous/not_found plus canonical key, stable family identity, known member sessionIds, and live continuation without exposing raw row blobs.

## Reproduction Assessment

Not applicable: this is a feature/API seam request, not a reproducible bug. Source inspection gives high confidence that current main has internal lineage data but lacks the requested public discovery contract.

## Solution Assessment

Yes directionally, but not complete on current main. The maintainable solution is to review the linked implementation PRs and choose one small public Gateway/SDK contract rather than having consumers parse session rows independently.
