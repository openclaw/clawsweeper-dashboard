## Summary

Keep open: current `main` and the latest release still provide internal alias-aware session-id resolution plus usage-family rollup fields, but they do not expose the requested public family/lineage discovery seam for rotated session ids. The closest implementation path is still the open, conflicting https://github.com/openclaw/openclaw/pull/79962, while the earlier helper branch at https://github.com/openclaw/openclaw/pull/79970 closed unmerged.

## What This Changes

The issue requests a public Gateway/SDK seam to resolve `{agentId, sessionId}` to canonical session ownership and expose durable rotated-session family membership, predecessor/successor identity, and ambiguity policy.

## Best Possible Solution

Keep this issue as the focused tracker until maintainers land a narrow Gateway/SDK lineage-discovery contract or explicitly make https://github.com/openclaw/openclaw/issues/79902 plus a repaired implementation PR the canonical remaining path.

## Reproduction Assessment

Not applicable; this is a feature/API seam request, not a broken existing behavior report. Source inspection confirms current `main` and `v2026.6.10` still lack the requested public lineage/family discovery output on `sessions.resolve` or the Plugin SDK session-store subpath.

## Solution Assessment

Unclear as a final API design. A narrow public Gateway/SDK seam is plausible, but maintainers still need to choose the stable contract for ambiguity, predecessor/successor ids, agent scoping, and how it relates to the existing usage-family rollup.
