## Summary

Keep open: the patch is a narrow and likely-correct fix for the agent web_search/runtime-provider split, but it still needs human merge handling because the supplied proof is source-level and provider-routing precedence needs owner signoff before landing.

## What This Changes

The PR keeps agent-side `web_search` runtime-provider discovery enabled for configured manifest-owned search providers while leaving `web_fetch` bundled-owner gating in place.

## Best Possible Solution

Land this only after redacted live provider-path proof and owner confirmation that search should remain runtime-first while fetch keeps its bundled-owner gate.

## Reproduction Assessment

No full live reproduction was established here. Source inspection shows current main can pass `preferRuntimeProviders: false` for a manifest-owned configured search provider before `runWebSearch` reaches the loaded runtime registry.

## Solution Assessment

Likely yes, but not fully proven for merge: moving search ownership to downstream `runWebSearch` load scoping is narrower than duplicating manifest-owner policy at the agent gate. The remaining question is owner confirmation that no supported search configuration intentionally needs static provider resolution over the loaded runtime registry.
