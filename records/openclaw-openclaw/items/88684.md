## Summary

Keep open: the patch looks like the right narrow fix for the linked agent web_search provider-resolution regression, but it still needs real provider-path proof and maintainer confirmation of the search-versus-fetch compatibility split before merge.

## What This Changes

The PR makes agent-side web_search stay runtime-provider-first while preserving the bundled-owner provider gate for web_fetch.

## Best Possible Solution

Land this narrow runtime-first search fix after provider/runtime owner review, sufficient redacted real behavior proof, and a non-conflicting plan with the broader explicit-provider routing PR; keep web_fetch's bundled-owner gate intact.

## Reproduction Assessment

Yes, at source level: current main can set `preferRuntimeProviders` false for a manifest-owned configured search provider before createWebSearchTool forwards the flag into runWebSearch. I did not live-reproduce the multi-hour gateway uptime failure.

## Solution Assessment

Yes, this is the narrowest maintainable fix shape I found: keep agent web_search runtime-first and leave runWebSearch responsible for manifest-owner scoping, while preserving web_fetch's bundled-owner path. The remaining blocker is proof and owner confirmation, not an obvious code defect.
