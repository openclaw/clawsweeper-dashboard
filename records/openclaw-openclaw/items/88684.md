## Summary

Keep this PR open: it is a plausible narrow fix for the linked agent-side web_search provider-registry bug, but it still needs real provider-path proof and provider/runtime owner confirmation before merge.

## What This Changes

The PR makes agent web_search always prefer runtime provider discovery while preserving the bundled-owner manifest gate for web_fetch, and updates focused tests for that search/fetch split.

## Best Possible Solution

Land this narrow runtime-first search fix after redacted provider-path proof and provider/runtime owner confirmation, or explicitly supersede it with a merged broader routing contract.

## Reproduction Assessment

Yes, at source level: current main can set preferRuntimeProviders to false for a manifest-owned configured search provider, and createWebSearchTool forwards that into runWebSearch. I did not live-reproduce the multi-hour gateway uptime failure.

## Solution Assessment

Likely yes, but not fully merge-proven. Keeping agent web_search runtime-first while preserving the web_fetch bundled-owner gate is the narrowest fix shape found; the remaining gaps are live provider-path proof and owner confirmation of the routing contract.
