## Summary

Keep open. Current main still leaves the macOS companion with coupled automatic Sparkle checks/downloads and an audit-only Swift config-clobber path; the reporter also confirmed the suggested recovery workaround did not help. Gateway-side TypeScript config recovery has improved, but that does not cover the macOS helper behavior shown by the current Swift test.

## What This Changes

The issue asks OpenClaw to prevent the macOS companion app from automatically applying incompatible beta updates and to preserve or recover the user's OpenClaw configuration after companion app reinstall/update flows.

## Best Possible Solution

Keep this open until the macOS companion separates Sparkle check/download/install policy, defaults automatic downloads conservatively, requires explicit beta-channel opt-in, and makes the Swift config helper actively preserve or restore a last-known-good config for update-channel-only clobbers.

## Reproduction Assessment

Yes. A focused current-main reproduction exists in `OpenClawConfigFileTests`: write a full config with `gateway.mode`, replace it with an update-channel-only root, and `loadDict()` returns a config with no gateway while only audit artifacts are produced.

## Solution Assessment

No. The current implementation is not the best fix because it treats suspicious Swift config loss as audit-only and uses one app setting to force both Sparkle checks and downloads; the safer path is conservative updater defaults plus active last-known-good recovery or merge.
