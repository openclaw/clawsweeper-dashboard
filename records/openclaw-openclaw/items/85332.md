## Summary

Keep open: current main and the shipped Docker docs support extra APT packages during local Docker/Podman image builds, while this issue asks for a pulled-image startup or first-boot installer. The related report at https://github.com/openclaw/openclaw/issues/58078 is useful context, but it is narrower and does not supersede this generic product/security decision.

## What This Changes

The issue requests an opt-in slim Docker image/startup mode that installs deployer-specified APT packages for remotely pulled OpenClaw images.

## Best Possible Solution

If maintainers want this, add a clearly named opt-in runtime bootstrap contract that preserves the existing build-time path and documents security limits, privilege requirements, caching, failure behavior, and unsupported locked-down runtimes.

## Reproduction Assessment

Not applicable; this is a feature request rather than a bug report with failing current behavior. Source inspection confirms current main supports build-time package args, but there is no documented startup-time package-install contract to reproduce as broken.

## Solution Assessment

Unclear; the proposed startup APT installer addresses the remote-image pain point, but it is not automatically the best solution because it changes container startup, privilege, and supply-chain behavior. Maintainers should choose between keeping the documented build-time path, adding a separate bootstrap image/mode, or implementing a guarded runtime bootstrap.

## Risks / Open Questions

- A runtime APT bootstrap would need root or equivalent privileges, network egress, and writable package-manager state even though the current official image switches to the non-root node user for runtime.
- Reusing OPENCLAW_IMAGE_APT_PACKAGES for startup-time behavior would change the semantics of a documented build-time variable and could confuse existing Docker and Podman setup flows.
- The related Synology/macvlan report says some target hosts block apt-get inside containers, so a startup installer may not solve the locked-down-host case without a separate policy decision.
