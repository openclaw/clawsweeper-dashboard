## Summary

Keep open: current main and v2026.5.20 only support extra APT packages as a build-time image argument, while this issue asks for a startup or first-boot path for remote-pulled images. The follow-up discussion reinforces that this is a Docker packaging product/security decision, not conservative cleanup.

## What This Changes

Request an opt-in slim-image startup or first-boot mode that installs deployer-specified Debian packages from an environment variable when using remote OpenClaw Docker images.

## Best Possible Solution

If maintainers want this, add a clearly named, opt-in Docker bootstrap contract with documented security limits, predictable caching/failure behavior, and proof for both remote-pulled startup and the existing build-time customization path.

## Reproduction Assessment

Not applicable; this is a feature request rather than a bug with failing current behavior. Source, docs, tests, and v2026.5.20 confirm the current contract is build-time package installation only.

## Solution Assessment

Unclear; the proposed runtime installer targets the reported remote-image pain point, but maintainers need to choose whether official images should perform root-level package-manager work at startup or keep package additions in custom builds.

## Risks / Open Questions

- A startup APT bootstrap would need root or elevated package-manager access, network access, writable apt state, clear failure behavior, and a trusted package-name policy inside the official image startup path.
- Reusing OPENCLAW_IMAGE_APT_PACKAGES for runtime behavior would change the semantics of a documented build-time variable unless maintainers define a compatibility story.
- The related Synology/macvlan report says some hosts block apt-get inside containers, so runtime installation may not solve every remote-pull deployment without documenting host limitations.
