## Summary

Keep open: current main already supports extra APT packages only during local Docker/Podman image builds, while this issue asks for a remote-pulled image startup or first-boot path. That remaining behavior is a new Docker packaging feature with security and product tradeoffs, especially given the related locked-down-host report at https://github.com/openclaw/openclaw/issues/58078.

## What This Changes

The issue requests an opt-in Docker startup/image mode that installs deployer-specified APT packages from an environment variable while keeping the default image slim.

## Best Possible Solution

If maintainers want this, add a clearly named, opt-in runtime bootstrap contract with documented security limits, predictable failure/caching behavior, and tests for both pulled-image startup and existing build-time customization.

## Reproduction Assessment

Not applicable; this is a feature request rather than a bug report with failing current behavior. Source inspection confirms current main supports build-time package args, but there is no documented startup-time package-install contract to reproduce as broken.

## Solution Assessment

Unclear; the proposed startup APT installer addresses the remote-image pain point, but it is not automatically the best solution because it changes container startup, privilege, and supply-chain behavior. Maintainers should choose between keeping the documented build-time path, adding a separate bootstrap image/mode, or implementing a guarded runtime bootstrap.

## Risks / Open Questions

- Reusing OPENCLAW_IMAGE_APT_PACKAGES for startup-time behavior would change the semantics of a documented build-time variable and could confuse existing Docker/Podman setup flows.
- A runtime APT bootstrap would need root privileges, network access, writable package-manager state, and clear failure behavior; the related Synology/macvlan report says some hosts block apt-get inside containers.
- Installing deployer-selected packages during official image startup is supply-chain sensitive and needs an explicit maintainer security decision before implementation.
