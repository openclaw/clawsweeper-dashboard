## Summary

Keep open: current main and the latest release only support extra APT packages during local image builds, while this issue asks for a pulled official image to install deployer-selected packages at startup or first boot. That is a new Docker packaging contract with product and security tradeoffs, not conservative cleanup.

## What This Changes

Add an opt-in Docker startup or slim-image mode that reads configured APT package names and installs them for remote-pulled OpenClaw images.

## Best Possible Solution

If maintainers want this, add a clearly named opt-in runtime bootstrap contract with documented security limits, deterministic failure and caching behavior, and tests covering both pulled-image startup and the existing build-time customization path.

## Reproduction Assessment

Not applicable; this is a feature request rather than a bug report with failing current behavior. Source inspection confirms the current contract is build-time package installation only, with no documented startup-time APT installer.

## Solution Assessment

Unclear; a startup APT installer addresses the remote-image pain point, but it changes container startup, privilege, and supply-chain behavior. Maintainers should choose between keeping the build-time path, adding a separate bootstrap image or mode, or implementing a guarded runtime bootstrap.
