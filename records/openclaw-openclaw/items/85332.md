## Summary

Keep open: current main and v2026.5.22 only support extra apt packages during image build, while this issue asks for a remote-image bootstrap/prebuild path. The latest discussion also narrows the desired shape away from gateway-startup APT work, so this remains a product and security decision.

## What This Changes

The issue asks for a slim official Docker image path that can install deployer-specified APT packages for remote-pulled deployments without manually rebuilding after each update.

## Best Possible Solution

If maintainers want this, add an explicit pre-gateway derived-image/bootstrap flow with documented security limits, caching/failure behavior, and tests, while preserving the current build-time custom-image path.

## Reproduction Assessment

No; this is a feature request rather than a failing existing contract. Source inspection confirms the current supported path is build-time package installation, not remote-image startup/bootstrap installation.

## Solution Assessment

Unclear; the requested capability addresses a real deployment ergonomics gap, but entrypoint-time APT installation has downtime and security drawbacks. A maintainer-approved prebuild/bootstrap flow looks safer than silently adding package-manager work to gateway startup.

- Runtime or entrypoint-time APT installation would need root privileges, network access, writable package-manager state, and explicit failure behavior in the official container flow.
- Reusing OPENCLAW_IMAGE_APT_PACKAGES for startup behavior could change the semantics of a documented build-time variable.
- The related Synology/macvlan report says some hosts block apt-get inside containers, so a runtime-only installer may not solve every motivating deployment.
