## Summary

Keep open: current main and v2026.5.20 support extra APT packages only as build-time image arguments, while this issue asks for a remotely pulled image to install packages during startup or first boot. The follow-up comment gives a plausible bootstrap shape, but the remaining decision changes container startup, privilege, and supply-chain behavior, and the related Synology/iproute2 report remains open at https://github.com/openclaw/openclaw/issues/58078.

## What This Changes

The issue requests an opt-in slim Docker image/startup mode that installs deployer-specified APT packages from an environment variable when using a remotely pulled OpenClaw image.

## Best Possible Solution

If maintainers want this, add a clearly named opt-in bootstrap contract with documented security limits, package-list hash/marker behavior, explicit failure semantics, Docker/Podman docs, and tests that preserve the existing build-time path.

## Reproduction Assessment

Not applicable; this is a feature request rather than a broken existing contract. Source and release inspection show the current supported path is build-time package installation, with no startup package-install contract to reproduce as failing.

## Solution Assessment

Unclear; the proposed startup installer addresses the remote-image pain point, but it is not automatically the best solution because it changes container privilege, network, and supply-chain behavior. Maintainers should choose between a guarded runtime bootstrap, a separate image/mode, or keeping the documented custom-image path.
