## Summary

Keep open: current main and the latest release still support extra APT packages only during Docker/Podman image builds, while this issue asks for a remote-pulled image bootstrap or pre-restart path. The follow-up discussion also narrowed the desired production shape toward building a derived image before gateway downtime, so this remains a Docker packaging product/security decision rather than cleanup to close.

## What This Changes

The issue requests an opt-in slim Docker image or bootstrap path that installs deployer-specified APT packages for remote-pulled container deployments.

## Best Possible Solution

Make an explicit Docker image policy call, then implement either a documented derived-image prebuild/bootstrap command or a guarded dev-only startup fallback with security limits, failure behavior, caching, and Docker setup tests.

## Risks / Open Questions

- A runtime APT bootstrap would need root privileges, network access, writable package-manager state, and clear failure behavior inside the official container flow.
- Reusing `OPENCLAW_IMAGE_APT_PACKAGES` for startup-time behavior would change the semantics of a documented build-time variable unless maintainers define a migration or separate knob.
- The latest reporter comment argues package installation should happen before gateway downtime, so an entrypoint-only implementation could solve the ergonomics problem while creating long cold-start outages.
