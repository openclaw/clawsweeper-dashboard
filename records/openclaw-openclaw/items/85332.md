## Summary

Keep open: current main and the latest release still support extra APT packages only as image-build arguments, while this issue asks for a remote-image bootstrap/startup or derived-image workflow. The later discussion usefully narrows the likely production-safe direction to prebuilding an extended image before gateway downtime, but that is still a Docker packaging feature with product and security tradeoffs.

## What This Changes

Requests an opt-in Docker image mode or bootstrap path that can install deployer-selected APT packages for remote-pulled OpenClaw images while keeping the default image slim.

## Best Possible Solution

Make an explicit Docker packaging decision: keep the existing build-time contract, document a Dockerfile.extend pattern, or add a guarded prebuild/derived-image command with clear security limits, failure behavior, docs, and focused Docker setup tests.

## Reproduction Assessment

Not applicable; this is a feature request rather than a bug report. Source inspection confirms current main and v2026.5.22 only support build-time package installation, so there is no existing startup-time contract to reproduce as broken.

## Solution Assessment

Unclear; the remote-image pain point is real, but entrypoint-time APT installation is not automatically the best solution. A prebuild/derived-image flow may better match the latest discussion by avoiding gateway downtime and keeping root package-manager work out of normal startup.

## Risks / Open Questions

- Runtime APT installation would need root privileges, network access, writable package-manager state, and clearly defined failure behavior inside the official container flow.
- Reusing OPENCLAW_IMAGE_APT_PACKAGES for startup behavior would change the semantics of a documented build-time variable and could surprise existing Docker/Podman users.
- The reporter's production-safe follow-up points toward a derived-image prebuild flow, so implementing entrypoint-time installs first could increase gateway downtime on slow mirrors.
