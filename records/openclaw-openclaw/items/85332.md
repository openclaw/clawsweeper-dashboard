## Summary

Keep open: current main and the latest release still only support `OPENCLAW_IMAGE_APT_PACKAGES` during local image builds, while this issue asks for an opt-in remote-image startup or prebuild path with clear security and downtime tradeoffs.

## What This Changes

The issue requests an opt-in slim Docker image mode that installs deployer-specified APT packages for remote-pulled images without baking them into the default image.

## Best Possible Solution

Maintainers should choose an explicit Docker packaging contract, likely a prebuild/derived-image path for production plus clear docs/tests, while keeping the default image non-root and security-first.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request. Source inspection gives high confidence that current main and `v2026.6.6` only support APT packages during image build, not pulled-image startup or prebuild automation.

## Solution Assessment

Unclear. The requested capability addresses a real deployment ergonomics gap, but entrypoint-time APT installation is probably not the safest production default; maintainers should choose between a derived-image prebuild command, documented Dockerfile pattern, or a guarded dev-only bootstrap.

## Risks / Open Questions

- A startup-time APT installer would need root or elevated package-manager privileges even though the current runtime image switches to the non-root `node` user.
- Installing deployer-selected packages from APT during official image startup is supply-chain and availability sensitive because mirrors, package names, and install failures would affect gateway startup.
- Reusing `OPENCLAW_IMAGE_APT_PACKAGES` for both build-time and startup-time behavior could change a documented env-var contract and confuse current Docker/Podman setup flows.
- The related Synology/macvlan report at https://github.com/openclaw/openclaw/issues/58078 says some hosts block `apt-get` inside containers, so runtime installation may not solve all affected deployments.
