## Best Possible Solution

Maintainers should choose an explicit Docker packaging contract, likely a prebuild or derived-image path for production plus docs/tests, while preserving the slim non-root default image and avoiding gateway downtime during package installation.

## Risks / Open Questions

- A startup-time APT installer would require root-level package-manager execution even though the current runtime image is designed around a non-root gateway process.
- Installing deployer-selected packages from APT in the official image flow is supply-chain and availability sensitive because mirrors, package names, and install failures could affect gateway startup.
- Reusing `OPENCLAW_IMAGE_APT_PACKAGES` for both build-time and startup/prebuild behavior could change the documented Docker and Podman env-var contract.
- The linked Synology/macvlan context says some hosts block `apt-get` inside containers, so runtime installation may not solve all affected deployments.
