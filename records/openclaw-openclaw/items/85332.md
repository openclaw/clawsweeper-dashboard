## Summary

Keep open: current main and the latest release support extra APT packages only as a local image-build customization, while this issue asks for a remote-image startup/prebuild mode. The follow-up comments usefully narrowed the production concern toward building an extended image before gateway downtime, so the remaining work is a Docker packaging feature that needs maintainer product and security review.

## What This Changes

The issue requests an opt-in slim Docker image/bootstrap mode that can install deployer-selected APT packages for remote-pulled OpenClaw Docker deployments.

## Best Possible Solution

Maintainers should choose an explicit Docker image policy, likely a prebuild/derived-image command or documented extension pattern with security limits, and only add entrypoint-time installation as a deliberate dev/convenience fallback if they accept the downtime and hardening tradeoffs.

## Reproduction Assessment

Not applicable; this is a feature request rather than a bug report. Source and shipped-release inspection confirm the current contract is build-time package installation only, so there is no broken startup-time contract to reproduce.

## Solution Assessment

Unclear; automatic startup APT installation solves part of the remote-image pain point but is not obviously the safest production shape. A prebuild or derived-image workflow before gateway restart appears safer, but that needs maintainer product and security approval.

## Risks / Open Questions

- A runtime APT bootstrap would need root/package-manager permissions, network access, writable APT state, and clear failure/caching behavior inside official Docker workflows.
- Reusing OPENCLAW_IMAGE_APT_PACKAGES for startup-time behavior would change the semantics of a documented build-time variable unless maintainers intentionally split or rename the runtime contract.
- Entrypoint-time APT work could keep the gateway offline during cold starts or recreates; the newer issue comment argues for a prebuild/derived-image step before restarting the gateway.
- The related Synology/macvlan report says some hosts block apt-get inside containers, so a runtime installer may not solve every target deployment even if implemented.
