## Summary

Keep open: current main still contains the source-level policy that turns any `room_event` into `message_tool_only` before requested automatic delivery can apply, so the reported reply-loss regression is plausible. The exact Feishu/Lark reproduction is not yet high-confidence because the bundled Feishu path does not set `InboundEventKind` itself, which suggests the remaining decision is how this strict room-event policy should interact with Feishu/Lark or external plugin-provided room events.

## What This Changes

The issue reports that the 2026.5.18 room-event source-reply policy forces Feishu/Lark inbound replies into `message_tool_only`, preventing visible replies from landing.

## Best Possible Solution

Keep ambient room events strict for channels that intentionally support that mode, but add a narrow contract for Feishu/Lark or plugin-requested automatic delivery so normal visible replies can still land, with regression coverage for both strict ambient rooms and automatic Feishu/Lark-style room events.

## Reproduction Assessment

No live high-confidence Feishu/Lark reproduction is available from the issue text. Source inspection does show that any inbound context carrying `InboundEventKind: "room_event"` is forced to `message_tool_only`, while the bundled Feishu path itself does not set that field.

## Solution Assessment

Unclear as a direct code change: broadly removing strict room-event delivery would conflict with the documented ambient-room behavior. The safer solution is a scoped policy or contract that preserves strict ambient rooms while honoring Feishu/Lark or plugin-requested automatic delivery where appropriate.
