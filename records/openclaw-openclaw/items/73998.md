## Summary

Keep open. The PR is aimed at the right Feishu-owned client/monitor boundary, but it is not merge-ready against current main: malformed WS endpoint responses can still stay inside the Lark SDK's default infinite reconnect loop, and the branch needs to preserve the current-main retry-exhaustion identity handling from #73739.

## What This Changes

The PR replaces Feishu WSClient constructor-level `wsConfig` handling with a guarded HTTP instance and SDK lifecycle hooks, rewrites monitor retry handling around those callbacks, adds focused Feishu tests, and adds a changelog entry.

## Best Possible Solution

Rebase on current main, keep the Feishu-owned SDK lifecycle/httpInstance composition, preserve #73739 monitor recovery semantics, and make missing `ClientConfig` plus non-OK WS endpoint responses deterministically reach a finite monitor retry path with focused tests.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to mock the WS endpoint response with `code: 0` and missing `ClientConfig`, or a system-busy non-OK response, then verify the SDK returns false with default `reconnectCount: -1` and the PR's monitor `onError` recovery is not reached. A second focused monitor test can trigger SDK `onError` and assert `botOpenIds`/`botNames` survive the retry cycle.

## Solution Assessment

No. The PR uses the right owner boundary and avoids global prototype mutation, but the current patch is incomplete; the safer solution is an explicit malformed-endpoint policy plus current-main identity-preserving retry cleanup, backed by tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Handle missing WS ClientConfig:** `extensions/feishu/src/client.ts:126-127`
  - body: The sanitizer returns the endpoint response unchanged when `data.ClientConfig` is absent. In the pinned Lark SDK, that leaves `reconnectCount` at its default `-1`; `pullConnectConfig` catches the later `ClientConfig.PingInterval` failure and returns false, so `onError` never fires and the monitor retry/backoff path can stall inside the SDK loop. Route missing `ClientConfig` and non-OK endpoint responses into a finite Feishu recovery path and cover it in tests.
  - confidence: 0.88
- **[P2] Preserve bot identity across retry cycles:** `extensions/feishu/src/monitor.transport.ts:227`
  - body: After `waitForFeishuWsStop` returns an SDK error, this cleanup call deletes the Feishu bot identity maps before the retry creates a replacement WSClient. Current main deliberately keeps those maps during supervisor recovery so requireMention, reactions, and card actions do not lose bot identity while reconnecting; carry that `clearIdentity` behavior forward and test it.
  - confidence: 0.79
