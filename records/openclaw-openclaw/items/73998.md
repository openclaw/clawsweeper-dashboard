## Summary

Keep this PR open and request changes before merge. The branch is pointed at the right Feishu-owned surface, but it still leaves malformed WebSocket endpoint responses in the SDK's infinite reconnect path and drops current-main bot identity preservation during retry cleanup.

## What This Changes

The PR replaces Feishu WSClient constructor-level `wsConfig` handling with a guarded HTTP instance and SDK lifecycle callbacks, rewrites monitor retry handling around those callbacks, adds Feishu tests, and updates the changelog.

## Best Possible Solution

Rebase onto current main, keep the Feishu-owned SDK lifecycle and HTTP-instance composition, preserve #73739 identity semantics, and make missing `ClientConfig` plus non-OK endpoint responses deterministically reach a finite monitor retry path with focused tests.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to mock the WS endpoint response as `code: 0` with missing `ClientConfig`, or a system-busy response without endpoint data, then verify the pinned SDK returns false without invoking monitor `onError`; a second focused monitor test can assert bot identity maps survive SDK-error retry cycles.

## Solution Assessment

No. The direction is maintainable because it stays inside the Feishu plugin and avoids process-wide SDK mutation, but the current patch is incomplete; the safer alternative is an explicit malformed-endpoint policy plus current-main identity-preserving cleanup carried through the PR branch.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Route malformed WS endpoint responses into retry:** `extensions/feishu/src/client.ts:122-127`
  - body: The sanitizer returns responses unchanged when `data` or `ClientConfig` is missing. In the pinned Lark SDK that means the later `ClientConfig.PingInterval` access is caught as `false`, while `reconnectCount` remains the default `-1`; `onError` never fires and the monitor can wait forever in the SDK loop. Convert these malformed/non-OK endpoint responses into a finite Feishu recovery path and cover it in tests.
  - confidence: 0.88
- **[P2] Preserve bot identity across retry cleanup:** `extensions/feishu/src/monitor.transport.ts:227`
  - body: This cleanup runs after an SDK error before the replacement WSClient is created, but `cleanupFeishuWsClient` always deletes `botOpenIds` and `botNames`. Current main deliberately keeps those maps during supervisor recovery, so this branch can make mention gating, reactions, and card actions lose bot identity while reconnecting. Carry the `clearIdentity: false` behavior forward and test it.
  - confidence: 0.79
