## Summary

Keep this PR open. It moves Feishu WebSocket recovery in the right direction by using the Lark SDK lifecycle hooks instead of global prototype mutation, but the patch still leaves malformed endpoint responses on the SDK's infinite reconnect path, so the monitor may never receive the onError signal needed to apply OpenClaw-owned backoff.

## What This Changes

This PR updates Feishu WebSocket construction to use a guarded Lark SDK httpInstance and lifecycle callbacks, rewrites monitor retry handling around SDK ready/error/reconnect events, adds focused Feishu tests, and adds a changelog entry.

## Best Possible Solution

Keep the SDK lifecycle/httpInstance composition approach, but make malformed endpoint responses deterministic: either synthesize a safe finite ClientConfig or route missing/non-object ClientConfig and non-OK startup responses into Feishu's monitor recovery path with focused tests. Do not reintroduce process-wide Lark.WSClient prototype mutation.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to stub the WS endpoint response with system-busy or missing data.ClientConfig and follow the Lark SDK 1.62.0 path: pullConnectConfig returns false or throws, then the SDK uses an infinite reconnect count and never calls monitor onError.

## Solution Assessment

No. The proposed boundary is the right one, but this patch is not yet the best fix because the malformed endpoint path is still allowed to stay in SDK-owned infinite reconnect instead of reaching the Feishu monitor's finite recovery policy.

## Review Findings

- **[P2] Route malformed endpoint responses to monitor retry:** `extensions/feishu/src/client.ts:126-127`
