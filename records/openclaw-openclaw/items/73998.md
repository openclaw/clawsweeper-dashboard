## Summary

Keep this PR open. The Feishu-owned SDK lifecycle direction is still useful and current main does not already implement it, but the patch leaves the reported malformed endpoint path able to stay inside the SDK's infinite reconnect loop instead of reaching the new monitor recovery path.

## What This Changes

The PR replaces Feishu WebSocket `wsConfig` casting with a guarded SDK `httpInstance` plus lifecycle hooks, rewrites monitor retry handling around SDK callbacks, adds focused Feishu tests, and adds a changelog entry.

## Best Possible Solution

Keep the SDK `httpInstance` and lifecycle composition, but add an explicit Feishu malformed-endpoint policy so missing `ClientConfig` and non-OK system-busy responses deterministically reach finite monitor recovery/backoff without restoring global prototype mutation or unsupported `wsConfig` casts.

## Reproduction Assessment

Yes. Mock the WS endpoint response with missing `data.ClientConfig` or a non-OK system-busy body; the SDK returns false and continues with default `reconnectCount: -1`, so the monitor waits on `onError` instead of applying its retry path.

## Solution Assessment

No. The PR uses the right owner boundary and SDK seams, but the current sanitizer is incomplete; the safer fix is to define and test the malformed endpoint recovery policy rather than reintroducing the old prototype mutation or untyped heartbeat cast.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Handle missing WS ClientConfig:** `extensions/feishu/src/client.ts:126-127`
  - body: The sanitizer returns unchanged when `data.ClientConfig` is absent. In the Lark SDK, that path is caught as a failed connect and leaves the client on the default `reconnectCount: -1` loop, so the new monitor `onError` recovery path may never run for the reported system-busy/PingInterval failures. Please route malformed endpoint responses into a finite Feishu-owned recovery path and cover it in tests.
