## Summary

Keep open. The PR is aimed at the right Feishu-owned boundary, but the proposed patch still leaves the reported malformed WebSocket endpoint path able to stay inside the Lark SDK's default infinite reconnect loop instead of reaching the new monitor recovery path.

## What This Changes

The PR replaces Feishu WebSocket constructor-level `wsConfig` use with a guarded SDK `httpInstance` plus lifecycle hooks, rewrites monitor retry handling around SDK reconnect callbacks, adds focused Feishu tests, and adds a changelog entry.

## Best Possible Solution

Land a narrow Feishu-owned repair that keeps SDK lifecycle/httpInstance composition, avoids global `Lark.WSClient` prototype mutation, and explicitly routes malformed endpoint responses into a finite monitor retry path while leaving normal SDK reconnect supervision alone.

## Reproduction Assessment

Yes. A focused reproduction is to mock the WS endpoint response with missing `ClientConfig` or a system-busy non-OK response and observe that the SDK returns `false` and continues with default `reconnectCount: -1`, so the monitor `onError` recovery promise is not resolved.

## Solution Assessment

No. The PR uses the right owner boundary and SDK hooks, but the sanitizer/recovery policy is incomplete; the safer fix is an explicit malformed-endpoint policy with tests rather than restoring the unsupported constructor `wsConfig` cast or relying on infinite SDK retries.

## Review Findings

Please route missing `ClientConfig` and non-OK endpoint responses into a finite Feishu-owned retry path and cover it in tests.
