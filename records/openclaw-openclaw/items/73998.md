## Summary

Keep open: current main still lacks the intended Feishu SDK lifecycle recovery path, and this PR is a plausible replacement direction but has a blocking gap where malformed or non-OK WebSocket endpoint responses can still stay inside the SDK's default infinite reconnect loop instead of reaching the monitor retry path.

## What This Changes

This PR replaces Feishu WebSocket constructor-level heartbeat config with a guarded SDK httpInstance and lifecycle hooks, rewrites monitor retry handling around SDK ready/error/reconnect callbacks, adds focused Feishu tests, and adds a changelog entry.

## Best Possible Solution

Land a narrow Feishu-owned repair that keeps the SDK httpInstance/lifecycle composition, avoids process-wide Lark prototype mutation, and makes missing ClientConfig plus non-OK startup responses deterministically reach a finite monitor retry path without stacking a second backoff while the SDK reconnect loop is active.

## Reproduction Assessment

Yes. A focused code-level reproduction is to mock the WS endpoint response with missing ClientConfig or a system-busy non-OK response and observe that the SDK returns false and keeps reconnecting with its default reconnectCount -1 instead of invoking the monitor onError recovery path.

## Solution Assessment

No. The PR is the right boundary and API direction, but the current sanitizer is incomplete; the safer fix is an explicit malformed-endpoint policy backed by tests, not restoring the old wsConfig/PingTimeout cast that SDK 1.62.0 does not type or destructure.
