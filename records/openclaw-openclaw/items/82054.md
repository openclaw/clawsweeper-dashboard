## Summary

Keep open: this is a valid core architecture feature request for caller-controlled context lifecycle, not something current main already implements or a stale cleanup target. OpenClaw has compaction, session pruning, tool-result truncation, and context-engine maintenance rewrites, but those are internal or policy-driven paths rather than a public stable allocate/free API for tool, skill, or plugin outputs.

## What This Changes

The issue requests stable IDs for tool, skill, or plugin outputs and a caller-visible way to explicitly remove stale outputs from prompt context.

## Best Possible Solution

Design an additive context-lifecycle contract that can mark prompt-visible outputs stale or replace them with placeholders while preserving transcript validity, provider tool-call/result pairing, and plugin SDK compatibility.

## Reproduction Assessment

Not applicable: this is a feature request, not broken behavior under an existing OpenClaw contract. Source review confirms current main lacks the requested caller-visible allocate/free API.

## Solution Assessment

Unclear: the malloc/free model captures the user need, but maintainers still need to choose the API boundary around stable IDs, prompt-only versus transcript rewrites, provider replay, and SDK exposure.
