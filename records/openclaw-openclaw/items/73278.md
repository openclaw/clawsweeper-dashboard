## What This Changes

The issue requests clearer Discord/Codex reply guidance so autoThread channel mentions produce one normal conversational reply in the created thread instead of an extra parent-channel message-tool reply.

## Best Possible Solution

Land one narrow prompt/skill clarification that says normal current-conversation replies should use the final assistant response when automatic delivery is active, while keeping the message tool available for explicit sideband actions and for configured message-tool-only visible output.

## Reproduction Assessment

Yes. The issue gives concrete Discord autoThread steps, and current main still has the broad prompt sources plus the final-delivery path that routes created-thread replies; the default group/channel mode now reduces the default case, but automatic final delivery remains supported and documented.

## Solution Assessment

Yes as a direction, but the fix should be consolidated through one open PR. The narrow maintainable path is prompt/skill wording, not a Discord routing rewrite, because the thread delivery code already targets the created thread.
