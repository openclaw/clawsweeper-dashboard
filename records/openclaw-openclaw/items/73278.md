## Summary

Keep open. Current main still has broad Codex runtime and Discord skill wording that can steer normal Discord conversation replies through the message tool, while the autoThread final-response path still routes to the created thread; the open closing PRs are implementation candidates, not a pre-merge close reason.

## What This Changes

The issue asks OpenClaw to avoid duplicate Discord replies when autoThread mentions trigger both a parent-channel message-tool send and final-response delivery in the created thread.

## Best Possible Solution

Land one canonical prompt and skill clarification that reserves `message.send` for explicit sideband Discord operations and leaves normal current-conversation replies to final-response delivery.

## Reproduction Assessment

Yes. The source-level reproduction path is Discord `autoThread: true` plus automatic visible replies: broad prompt guidance can elicit `message.send`, while normal final delivery is routed to the created thread.

## Solution Assessment

Yes. The narrow maintainable fix is prompt and skill wording, because current autoThread delivery mechanics already route final responses to the created thread and the duplicate comes from agent guidance.
