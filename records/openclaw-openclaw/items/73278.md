## Summary

Keep open. Current main still contains the broad Codex runtime and Discord skill messaging guidance that can prompt a manual Discord send while autoThread final delivery targets the created thread; the related open PRs are the right consolidation path rather than closing this issue now.

## What This Changes

The issue reports that Discord `autoThread: true` mentions can produce both a manual parent-channel `message.send` reply and the normal final assistant reply in the created thread.

## Best Possible Solution

Review and land one canonical prompt/skill clarification that distinguishes normal current-conversation final replies from explicit Discord message-tool actions, while preserving the current tool-only default for group/channel output and leaving autoThread routing unchanged.

## Reproduction Assessment

Yes. The original report gives concrete Discord `autoThread: true` steps, and current main still has the code path needed to reproduce the remaining automatic-mode risk: broad prompt guidance can encourage `message.send` while final delivery is routed to the created thread.

## Solution Assessment

Yes as a direction, but it should be consolidated through one of the open PRs. The maintainable fix is narrow prompt/skill wording that preserves explicit `message` operations and avoids manual sends for the same conversational reply when automatic final delivery is active.
