## Summary

Keep open. Current main still contains the broad Codex runtime and Discord skill wording that can steer a normal Discord channel reply through the message tool, while the autoThread final-delivery path can still route the normal final response to the created thread when automatic visible replies are configured. Open closing PRs already target this same report, so the issue should stay open until one canonical fix lands.

## What This Changes

The issue requests that Discord autoThread mentions produce only one conversational reply by reserving `message.send` for explicit sideband messaging and using the final assistant response for the current conversation.

## Best Possible Solution

Land one canonical prompt/skill clarification that tells agents to use the final assistant response for normal current-conversation replies and reserves Discord message-tool sends for explicit sideband actions, while leaving the existing autoThread routing mechanics intact.

## Reproduction Assessment

Yes. The issue provides concrete Discord `autoThread: true` steps, and current source inspection confirms the same prompt wording plus automatic final delivery to the created thread can still coexist.

## Solution Assessment

Yes. The narrow maintainable solution is to clarify the Codex runtime and Discord skill guidance; changing thread creation or final-response delivery would be broader than needed because that path already targets the created thread.
