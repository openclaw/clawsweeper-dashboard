## Summary

Keep open. Current main still has the conflicting Codex runtime and bundled Discord skill guidance that can steer a normal Discord reply through `message.send`, while the Discord autoThread final-response path can still target the created thread; the three open closing PRs are implementation candidates, not proof that this is fixed.

## What This Changes

Discord `autoThread: true` channel mentions should avoid producing both a parent-channel message-tool reply and a final response in the created thread.

## Best Possible Solution

Land one canonical delivery-mode-aware prompt and skill clarification that reserves `message.send` for explicit or tool-only visible Discord operations while leaving automatic current-conversation replies to final-response delivery.

## Reproduction Assessment

Yes. The source-level path is high-confidence: Discord `autoThread: true` can retarget final-response delivery to the created thread while current Codex and Discord prompts can still elicit an extra `message.send` for the same conversational reply.

## Solution Assessment

Yes, with a constraint. A narrow prompt/skill wording fix is the maintainable path, but it should be delivery-mode-aware so default Discord channel tool-only replies remain visible.
