## Summary

Keep open. Current main reduces the default duplicate-reply path by making group/channel final replies message-tool-only, but the broad Codex runtime instruction and Discord skill wording that caused this report are still present, and documented/tested automatic visible replies can still route a normal final response into the auto-created thread while the agent is also prompted toward `message.send`. The three linked open PRs are implementation candidates, so this issue should remain open until one canonical fix is reviewed and merged.

## What This Changes

Discord autoThread mentions should avoid producing both a manual parent-channel message-tool reply and a normal final reply in the created thread.

## Best Possible Solution

Land one canonical prompt and skill clarification that says normal current-conversation replies should use the final assistant response when automatic delivery is active, while reserving `message.send` for explicit sideband sends, reactions, reads, notifications, and other non-default Discord actions.

## Reproduction Assessment

Yes for the remaining current-main risk: configure a Discord guild channel with `autoThread: true` and `messages.groupChat.visibleReplies: "automatic"`, then mention OpenClaw so the agent sees broad messaging-tool guidance while final delivery targets the created thread. The exact live duplicate is model-dependent, but the source-level prompt and routing conditions are present.

## Solution Assessment

Yes as a direction, but not yet implemented on main. A narrow prompt/skill wording fix is the maintainable solution; changing autoThread routing would risk breaking the final delivery path that already points at the created thread.
