## Summary

Keep open. Current main still contains the broad Codex runtime and Discord skill wording that can steer an agent to use the message tool for a normal Discord channel reply, while the Discord autoThread final-delivery path still routes normal final replies to the created thread. The default group/channel visible-reply mode now mitigates the default path, but configured automatic replies remain supported, and open PRs #73279, #73369, and #73394 already target this same report.

## What This Changes

The issue asks OpenClaw to prevent Discord `autoThread: true` channel mentions from producing both a parent-channel `message.send` reply and a normal final response in the created thread.

## Best Possible Solution

Land one canonical prompt/skill clarification that tells agents to use the final assistant response for the normal current-conversation reply and reserves `message.send` and related Discord actions for explicit sideband sends, reactions, reads, edits, notifications, components, or different targets. Keep autoThread routing mechanics unchanged and close the overlapping PRs after the chosen fix lands.

## Reproduction Assessment

Yes. The issue gives concrete Discord `autoThread: true` steps, and source inspection verifies the same prompt conditions plus automatic final delivery to the created thread remain possible on current main.

## Solution Assessment

Yes. The narrow maintainable fix is to clarify the Codex runtime and Discord skill guidance rather than changing thread creation or final-response delivery, because the delivery path already routes normal final replies to the created thread.
