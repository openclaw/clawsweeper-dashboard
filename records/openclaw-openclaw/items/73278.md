## Summary

Keep open. Current main still has the broad Codex runtime and Discord skill guidance that can steer a model to use the message tool for a normal channel reply, while the autoThread final-delivery path still targets the created thread when automatic final replies are enabled. The newer group/channel default mitigates the default path by using message-tool-only visible replies, but automatic final replies remain supported and three open PRs already target this report, including a paired PR from the reporter.

## What This Changes

The issue reports that Discord parent-channel mentions with `autoThread: true` can produce both a manual parent-channel `message.send` reply and a normal final assistant reply in the created thread.

## Best Possible Solution

Land one canonical prompt/skill clarification that tells agents to use the final assistant response for the normal current-conversation reply and reserves `message.send` and related Discord actions for explicit sideband sends, reactions, reads, edits, notifications, or different targets. Keep the existing autoThread routing mechanics intact and close the other overlapping PRs after the chosen fix lands.

## Reproduction Assessment

Yes. The report gives concrete Discord `autoThread: true` steps, and source inspection verifies the same prompt conditions plus automatic final delivery to the created thread remain possible on current main.

## Solution Assessment

Yes. The narrow maintainable solution is to clarify the Codex runtime and Discord skill guidance rather than changing thread creation or final-response delivery, because the delivery path already routes normal final replies to the created thread.
