## Summary

Keep open. Current main still contains the broad Codex runtime and Discord skill messaging guidance described in the report, while autoThread final delivery still routes normal replies to the created thread. The current group/channel default reduces the default-path exposure by suppressing automatic final replies, but configured automatic final replies remain supported and three open closing PRs are already targeting this issue, so this should stay open until one canonical fix is reviewed and merged.

## What This Changes

The issue requests a Discord/Codex prompt guidance fix so autoThread channel mentions do not produce both a parent-channel `message.send` reply and a final response in the created thread.

## Best Possible Solution

Land one canonical prompt/skill clarification that tells agents to use the final assistant response for the normal current-conversation reply while reserving `message.send` and related Discord actions for explicit sideband sends, reactions, reads, edits, notifications, or a different target. Keep the autoThread routing behavior unchanged and include a focused regression check for the Codex developer instruction or bundled Discord skill text.

## Reproduction Assessment

Yes. The issue gives concrete Discord `autoThread: true` steps, and current main still has the two source conditions needed for the prompt-driven duplicate path: broad messaging-tool guidance and automatic final delivery that targets the created thread when enabled.

## Solution Assessment

Yes, the proposed direction is the narrow maintainable fix. Runtime autoThread delivery already routes final replies to the thread; the safer change is to clarify prompt/skill wording rather than changing Discord thread creation or delivery mechanics.
