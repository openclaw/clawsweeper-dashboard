## Summary

Keep open. Current main still contains the broad Codex runtime and Discord skill wording that can steer a normal Discord current-conversation reply through the message tool, while autoThread final-response delivery still targets the created thread; the three linked open PRs are implementation candidates rather than a reason to close this issue.

## What This Changes

Discord `autoThread: true` channel mentions can produce duplicate visible replies when prompt guidance causes a manual `message.send` reply before normal final-response delivery posts to the created thread.

## Best Possible Solution

Land one canonical prompt/skill clarification that reserves `message.send` for explicit sideband Discord actions and keeps normal current-conversation replies on final-response delivery.

## Reproduction Assessment

Yes. The issue gives concrete `autoThread: true` Discord steps, and current source inspection verifies the same broad prompt guidance plus configured automatic final delivery to the created thread remain possible.

## Solution Assessment

Yes. The narrow maintainable solution is to clarify the Codex runtime and Discord skill guidance, because the autoThread delivery mechanics already route normal final replies to the created thread.
