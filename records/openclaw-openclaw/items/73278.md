## Summary

Keep open. Current main still has the conflicting prompt path: Codex developer instructions and the bundled Discord skill can steer a normal Discord reply through `message.send`, while automatic visible replies can still deliver the final response to the created autoThread; the open closing PRs are implementation candidates, not a reason to close before merge.

## What This Changes

Discord autoThread mentions should avoid an extra parent-channel `message.send` when the normal final response is routed to the created thread.

## Best Possible Solution

Land one canonical prompt and skill clarification that reserves `message.send` for explicit sideband Discord operations and leaves normal current-conversation replies to final response delivery.

## Reproduction Assessment

Yes. The source-level path is clear: Discord `autoThread: true` plus automatic visible group replies can route the final response to the created thread while broad Codex/Discord prompt wording can elicit an extra `message.send`.

## Solution Assessment

Yes. The narrow maintainable fix is prompt and skill wording, because current autoThread delivery mechanics already route final responses to the created thread.
