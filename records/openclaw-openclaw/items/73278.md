## Summary

Keep open. Current main still has the conflicting Codex developer instruction and broad Discord skill guidance, while Discord autoThread final delivery can route the final response to the created thread; the three linked closing PRs are open implementation candidates, not a reason to close the issue before one lands.

## What This Changes

Discord autoThread channel mentions should avoid duplicate conversational replies caused by prompt guidance that can steer the agent to call `message.send` in addition to emitting a final response.

## Best Possible Solution

Land one canonical delivery-mode-aware Codex prompt and Discord skill update that uses final assistant output for current-conversation replies when OpenClaw will deliver them automatically, while reserving `message.send` for explicit sideband sends.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: automatic Discord autoThread delivery targets the created thread, while the Codex and Discord prompts can still steer the agent to also call `message.send`.

## Solution Assessment

Yes. The narrow maintainable fix is prompt and skill guidance that is aware of source reply delivery mode, not a routing rewrite or unconditional final-response wording that would break default tool-only group/channel behavior.
