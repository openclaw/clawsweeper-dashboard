## Summary

Keep open. Current main still has broad Codex runtime and Discord skill wording that can steer a normal Discord conversation reply through `message.send`, while the autoThread final-delivery path can still target the created thread when automatic visible replies are configured; the linked open PRs are implementation candidates, not a pre-merge close reason.

## What This Changes

Prevent Discord `autoThread` mentions from producing both a parent-channel message-tool reply and a final response in the created thread.

## Best Possible Solution

Land one canonical prompt and skill clarification that reserves `message.send` for explicit sideband Discord operations and leaves normal current-conversation replies to final response delivery.

## Reproduction Assessment

Yes. The source-level path is clear with Discord `autoThread: true` plus automatic visible group replies: broad prompt guidance can elicit a `message.send`, while normal final delivery is routed to the created thread.

## Solution Assessment

Yes. The narrow maintainable fix is prompt/skill wording, because the autoThread delivery mechanics already route final responses to the created thread and the reported duplicate comes from agent guidance.
