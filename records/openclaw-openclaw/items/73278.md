## Summary

Keep open. Current main still contains the broad Codex runtime and Discord skill wording that can steer a model to use the message tool for a normal current-conversation reply, while the Discord autoThread final-delivery path can still route an automatic final reply to the created thread. The newer message-tool-only group/channel default mitigates the default path, but automatic final replies remain configurable and three open closing PRs already target this issue.

## What This Changes

The issue requests preventing Discord autoThread channel mentions from producing both a manual parent-channel message-tool reply and a normal final reply in the created thread.

## Best Possible Solution

Land one canonical prompt/skill clarification that tells agents to use the final assistant response for the normal current conversation reply and reserves `message.send` and related Discord actions for explicit sideband operations, then close the overlapping fix PRs that are not chosen.

## Reproduction Assessment

Yes. The report gives concrete Discord `autoThread: true` steps from 2026.4.26, and current-main source inspection shows the same broad prompt guidance plus automatic final delivery to the created thread can still be selected through config, although the live model choosing `message.send` is probabilistic.

## Solution Assessment

Yes. The narrow maintainable fix is to clarify the Codex runtime and Discord skill guidance rather than changing thread creation or final-response delivery, because the final delivery path already routes normal automatic replies to the created thread.
