## Summary

Keep open. Current main still has the conflicting Codex/Discord prompt path that can steer a normal Discord reply through `message.send` while the autoThread final-response path can deliver to the created thread; the open closing PRs are implementation candidates, not proof that the issue is fixed.

## What This Changes

Discord `autoThread: true` channel mentions should produce one conversational reply in the created thread instead of both a parent-channel `message.send` reply and a thread final response.

## Best Possible Solution

Land one canonical prompt and skill clarification that reserves `message.send` for explicit sideband Discord operations while leaving normal current-conversation replies to final-response delivery.

## Reproduction Assessment

Yes. The source-level path is high-confidence: Discord `autoThread: true` can route final responses to the created thread while current Codex and Discord prompts can still elicit an extra `message.send` for the same conversational reply.

## Solution Assessment

Yes. A narrow prompt and skill wording fix is the maintainable solution because the current autoThread delivery mechanics already target the created thread for final responses.
