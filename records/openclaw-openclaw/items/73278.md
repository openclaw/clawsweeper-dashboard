## Summary

Keep open. Current main still has the broad Codex and Discord skill prompt sources that can steer a Discord autoThread turn toward a manual same-conversation message send while automatic final delivery targets the created thread; the three open closing PRs are the right place to consolidate the fix before this issue closes.

## What This Changes

The issue asks OpenClaw to prevent Discord autoThread mentions from producing both a manual parent-channel message-tool reply and an automatic final response in the created thread.

## Best Possible Solution

Land or close one canonical PR that clarifies the Codex runtime and Discord skill guidance: normal current-conversation replies should use final assistant delivery when automatic delivery is active, while message.send remains the visible-output path for message_tool_only Discord turns and explicit sideband actions.

## Reproduction Assessment

Yes. The original report gives concrete Discord autoThread steps against 2026.4.26, and current main still shows the same automatic-mode mechanics: broad prompt guidance can encourage message.send while final delivery is rewritten to the created thread.

## Solution Assessment

Yes as a direction, with one important boundary. A prompt/skill clarification is the narrow maintainable fix, but it must preserve current message_tool_only guidance for default guild/channel visibility rather than blanket-discouraging the message tool.
