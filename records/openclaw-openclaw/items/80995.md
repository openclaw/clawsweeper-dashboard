## Summary

Keep open: current main still lacks the Google Chat message-tool threading adapter, so `[[reply_to_current]]` can still resolve to the Google Chat message resource instead of the inbound thread resource. The open fixing PR at https://github.com/openclaw/openclaw/pull/80996 explicitly closes this issue, so the issue should stay open until that PR or an equivalent fix lands.

## What This Changes

The issue requests that Google Chat `message(action=send)` replies with `[[reply_to_current]]` use the inbound `spaces/.../threads/...` resource instead of posting top-level messages.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/80996 or an equivalent Google Chat-owned threading adapter from https://github.com/openclaw/openclaw/pull/74235 with regression coverage, then close this issue after the fix reaches main.

## Reproduction Assessment

Yes, at source level. Current main carries the inbound Google Chat thread as `ReplyToId`, but without a Google Chat `buildToolContext`, the message tool falls back to the inbound message resource as `currentMessageId`.

## Solution Assessment

Yes. A Google Chat-owned threading adapter is the narrow maintainable fix because it maps the existing inbound thread resource into the existing message-tool reply context without changing shared core semantics or adding configuration.
