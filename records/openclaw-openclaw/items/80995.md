## Summary

Keep open: current main still carries the Google Chat thread resource into the turn context, but Google Chat has no channel-specific message-tool threading adapter, so `[[reply_to_current]]` can still inherit the message resource instead of the thread resource. The linked open PR is the right implementation path and explicitly closes this issue, so the issue should stay open until that PR or an equivalent fix lands.

## What This Changes

Ensure Google Chat `message(action=send)` replies using `[[reply_to_current]]` stay in the inbound Google Chat thread when `message.thread.name` and `replyToMode` are present.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/80996 or an equivalent Google Chat `buildToolContext` adapter with regression coverage, then close this issue after the fix is on main.

## Reproduction Assessment

Yes, at source level. Current main carries `message.thread.name` as `ReplyToId`, but without a Google Chat `buildToolContext` the message tool inherits `MessageSidFull`/`MessageSid` as `currentMessageId`, which can make `replyToMode: "all"` target the message resource instead of `spaces/.../threads/...`.

## Solution Assessment

Yes. The linked PR's channel-owned adapter is the narrow maintainable fix because it maps the existing Google Chat thread resource into the existing message-tool reply context without changing shared core semantics or adding configuration.
