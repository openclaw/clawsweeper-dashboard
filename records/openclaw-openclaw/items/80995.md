## Label Justifications

- `P1`: Google Chat replies can post into the wrong conversation context, breaking an active channel workflow for real users.
- `impact:message-loss`: The report is about channel messages being misrouted outside the intended Google Chat thread.

## Summary

Keep open: current main and the latest release still expose the Google Chat thread resource, but the message-tool threading context can still fall back to the inbound message resource. An open focused PR explicitly closes this issue, so cleanup should wait for that PR or an equivalent fix to merge.

## What This Changes

Google Chat `message(action=send)` replies using `[[reply_to_current]]` should target the inbound `spaces/.../threads/...` resource instead of posting as top-level space messages.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/80996 or an equivalent Google Chat `buildToolContext` adapter with regression coverage, then close this issue after the fix reaches main.

## Reproduction Assessment

Yes. Source inspection shows inbound `message.thread.name` reaches `ReplyToId`, but the current message-tool fallback still uses `MessageSidFull` or `MessageSid` as `currentMessageId`; the linked PR adds live before/after proof.

## Solution Assessment

Yes. A Google Chat-owned `buildToolContext` adapter is the narrow maintainable fix because it keeps Google Chat resource-shape translation inside the plugin boundary without changing shared core semantics.
