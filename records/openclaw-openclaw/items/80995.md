## Summary

Keep open: current main still has the Google Chat thread resource, but the Google Chat threading adapter does not map it into the message tool context, so `[[reply_to_current]]` can still inherit the inbound message resource. An open PR explicitly closes this issue, so the issue should stay open until that PR or the broader related threading PR lands.

## What This Changes

Report that Google Chat `message(action=send)` replies using `[[reply_to_current]]` can post outside the inbound Google Chat thread despite `replyToMode: "all"` and an available `message.thread.name`.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/80996 or the broader https://github.com/openclaw/openclaw/pull/74235 after choosing one implementation, then close this issue once the fix is on main.

## Reproduction Assessment

Yes, at source level. Current main captures `message.thread.name` as `ReplyToId`, but without a Google Chat `buildToolContext`, the message tool can inherit `MessageSidFull` or `MessageSid` as `currentMessageId` and send that as the reply target.

## Solution Assessment

Yes. A Google Chat-owned threading adapter or equivalent plugin-boundary normalization is the narrow maintainable fix because it uses the existing channel threading seam without changing shared core semantics or adding configuration.
