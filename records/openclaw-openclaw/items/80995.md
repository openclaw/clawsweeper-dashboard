## Summary

Keep open: current main still has the reported Google Chat message-tool thread escape path, and an open linked PR explicitly targets this issue but has not merged. The issue is concrete, source-reproducible, and should remain open until that PR or an equivalent fix lands on main.

## What This Changes

Google Chat `message(action=send)` replies using `[[reply_to_current]]` should resolve to the inbound `message.thread.name` resource instead of posting as top-level space messages.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/80996 or the broader equivalent in https://github.com/openclaw/openclaw/pull/74235 with regression coverage, then close this issue once the fix is on main.

## Reproduction Assessment

Yes, at source level. Current main captures the Google Chat thread resource but the message-tool threading fallback can still inherit `MessageSidFull`/`MessageSid`, so `[[reply_to_current]]` can resolve to a message resource instead of `spaces/.../threads/...`.

## Solution Assessment

Yes. A Google Chat-owned threading tool-context adapter or equivalent channel-boundary normalization is the narrow maintainable fix because it uses existing inbound thread data without changing shared core semantics.
