## Summary

Keep open: current main still has the Google Chat message-tool threading gap, and open implementation PRs already cover the fix path, including a direct closing PR for this issue.

## What This Changes

Google Chat `message(action=send)` replies using `[[reply_to_current]]` should stay in the inbound Google Chat thread when the event includes `message.thread.name`.

## Best Possible Solution

Land one Google Chat-owned threading fix that maps the inbound thread resource into the message-tool reply context with regression coverage, then close this issue.

## Reproduction Assessment

Yes, at source level. A Google Chat event with `message.thread.name`, `replyToMode: "all"`, and a same-conversation `message(action=send)` using `[[reply_to_current]]` follows a clear current-main path from thread resource to message-resource fallback.

## Solution Assessment

Yes. A Google Chat `buildToolContext` adapter is the narrow maintainable fix because it translates the channel-specific thread resource at the plugin boundary without changing shared message-tool semantics.
