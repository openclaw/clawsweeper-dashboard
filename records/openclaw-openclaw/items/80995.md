## Summary

Keep this issue open. Current main still has the source-level Google Chat threading gap, and the open linked pull request at https://github.com/openclaw/openclaw/pull/80996 uses closing syntax for this issue, so the right cleanup is to review or land that PR rather than close the issue early.

## What This Changes

Make Google Chat `message(action=send)` replies with `[[reply_to_current]]` target the inbound `message.thread.name` thread resource when reply threading is enabled.

## Best Possible Solution

Review and land the linked Google Chat plugin adapter that maps inbound `ReplyToId` into the message-tool current reply target, then close this issue after the merge.

## Reproduction Assessment

Yes, at source level. Current main carries `message.thread.name` in `ReplyToId`, but the generic message-tool context still inherits `MessageSidFull` as `currentMessageId`, so `[[reply_to_current]]` can resolve to the Google Chat message resource instead of `spaces/.../threads/...`.

## Solution Assessment

Yes. The linked PR's channel-owned `buildToolContext` adapter is the narrowest maintainable fix because it translates Google Chat's thread resource into the existing message-tool reply contract without changing shared core semantics.
