## Label Justifications

- `P1`: This is a real Google Chat channel workflow break where visible assistant replies can be delivered outside the user's active thread.
- `impact:message-loss`: The report is about misrouted channel messages: replies land as top-level Google Chat space messages instead of the intended thread.

## Summary

Keep open: current main and v2026.5.18 still have the Google Chat thread resource available, but they do not map it into the message-tool reply context, and the explicit fix PRs are still open.

## What This Changes

Reports that Google Chat `message(action=send)` with `[[reply_to_current]]` can post outside the inbound thread despite `message.thread.name` and `replyToMode: "all"`.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/80996 or the broader https://github.com/openclaw/openclaw/pull/74235 with regression coverage, then close this issue after the fix reaches main; keep the typing-placeholder follow-up separate.

## Reproduction Assessment

Yes, at source level. Current main carries the Google Chat thread resource as `ReplyToId`, but the generic message-tool context can still inherit the message resource as `currentMessageId`, which is then copied into the outbound reply target.

## Solution Assessment

Yes. A Google Chat-owned `buildToolContext` adapter is the narrow maintainable fix because it maps the existing channel thread resource into the existing message-tool reply context without changing shared core semantics.
