## Summary

Keep open: current main still has the Google Chat threading gap, and an open PR explicitly closes this issue. The source path still lets Google Chat message-tool replies inherit the inbound message resource instead of the thread resource, so the issue should close only after the focused fix or the broader related Google Chat threading PR lands.

## What This Changes

Google Chat `message(action=send)` replies using `[[reply_to_current]]` should stay in the inbound Google Chat thread when `message.thread.name` is present.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/80996 or the broader https://github.com/openclaw/openclaw/pull/74235 equivalent, then close this issue once the Google Chat thread resource is preserved on main.

## Reproduction Assessment

Yes, at source level. Current main carries `message.thread.name` into `ReplyToId`, but Google Chat has no `buildToolContext`, so `[[reply_to_current]]` can inherit `MessageSidFull`/`MessageSid` instead of the `spaces/.../threads/...` resource.

## Solution Assessment

Yes, the channel-owned adapter path is the narrow maintainable fix because it maps Google Chat's existing thread resource into the existing message-tool reply context without changing shared core semantics.
