## Label Justifications

- `P1`: This is a source-reproducible Google Chat channel workflow bug where visible replies can be delivered outside the user’s original thread.
- `impact:message-loss`: The issue is about channel messages being misrouted from the intended Google Chat thread to the top-level space.

## Summary

Keep open: current `main` still lacks a Google Chat message-tool threading adapter, and the open closing PR has not merged. The thread resource is present on inbound turns, but `[[reply_to_current]]` can still fall back to the Google Chat message resource and post outside the original thread; the latest release tag has the same gap.

## What This Changes

Ensure Google Chat `message(action=send)` replies using `[[reply_to_current]]` resolve to the inbound `spaces/.../threads/...` resource instead of posting top-level.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/80996 or the broader https://github.com/openclaw/openclaw/pull/74235 after resolving overlap, then close this issue once the fix is on main.

## Reproduction Assessment

Yes, source-level. Current main carries `message.thread.name` into `ReplyToId`, but because Google Chat has no `buildToolContext`, the message tool inherits `MessageSidFull`/`MessageSid` as `currentMessageId` and can send a message resource where Google Chat expects a thread resource.

## Solution Assessment

Yes. A Google Chat-owned threading adapter with regression coverage is the narrow maintainable fix because it maps the existing inbound thread resource into the existing message-tool reply context without changing shared core semantics.
