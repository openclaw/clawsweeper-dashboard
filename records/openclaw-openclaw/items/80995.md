## Label Justifications

- `P1`: A real Google Chat channel workflow can misroute user-visible replies outside the source thread, which is an urgent channel-delivery regression for affected users.
- `impact:message-loss`: The issue is about message-tool replies being delivered to the wrong Google Chat conversation location rather than the original thread.

## Summary

Keep open: current `main` still captures the inbound Google Chat thread, but the message-tool threading context still falls back to the inbound message id because Google Chat has no `buildToolContext` adapter on `main`. The linked open PR https://github.com/openclaw/openclaw/pull/80996 explicitly closes this issue, so the issue should stay open until that PR or the broader thread-routing work in https://github.com/openclaw/openclaw/pull/74235 lands.

## What This Changes

Google Chat `message(action=send)` replies using `[[reply_to_current]]` should target the inbound `message.thread.name` when reply threading is enabled.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/80996 or consolidate its adapter and regression test into the broader Google Chat thread-routing work, then close this issue once the fix reaches `main`.

## Reproduction Assessment

Yes, source-level. Current `main` carries `message.thread.name` as `ReplyToId`, but without a Google Chat `buildToolContext`, `[[reply_to_current]]` inherits the inbound message resource as `currentMessageId`; the linked PR also provides redacted live before/after proof.

## Solution Assessment

Yes. A Google Chat-owned threading adapter is the narrow maintainable fix because it maps the existing channel thread resource into the existing shared message-tool context without changing core semantics or adding configuration.
