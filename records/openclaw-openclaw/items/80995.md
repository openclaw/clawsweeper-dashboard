## Label Justifications

- `P1`: This is a user-visible Google Chat channel workflow bug where assistant/tool replies can be routed outside the user's original thread.
- `impact:message-loss`: The issue is about misrouted Google Chat messages, with replies appearing as top-level space messages instead of in-thread replies.

## Summary

Keep open: current main and the latest release still lack the Google Chat message-tool threading adapter, and an open PR explicitly closes this issue, so the issue should stay open until that fix or an equivalent one lands.

## What This Changes

Google Chat `message(action=send)` replies using `[[reply_to_current]]` should target the inbound `message.thread.name` instead of escaping to top-level space messages.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/80996 or an equivalent Google Chat-owned threading adapter with regression coverage, then close this issue after the fix reaches main.

## Reproduction Assessment

Yes, at source level. Current main carries the inbound Google Chat thread in `ReplyToId`, but without a Google Chat `buildToolContext`, `[[reply_to_current]]` can inherit the message resource as `currentMessageId`.

## Solution Assessment

Yes. The narrow maintainable fix is the linked channel-owned adapter that maps the existing Google Chat thread resource into the existing message-tool reply context without changing shared core semantics.
