## Label Justifications

- `P1`: The report describes a real Google Chat channel workflow where agent-visible replies can land outside the user's active thread.
- `impact:message-loss`: The affected message is delivered to the wrong Google Chat location rather than the intended thread.

## Summary

Keep open: current main and the latest v2026.5.18 release still lack the Google Chat message-tool threading adapter, so `[[reply_to_current]]` can still inherit the inbound message resource instead of the Google Chat thread resource. The linked open PR is an implementation candidate that explicitly closes this issue, and the broader open Google Chat thread-placement PR also covers the same user-visible bug.

## What This Changes

The issue asks Google Chat `message(action=send)` replies using `[[reply_to_current]]` to stay in the inbound `message.thread.name` thread when `replyToMode` is `all`.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/80996 or the broader https://github.com/openclaw/openclaw/pull/74235 fix if maintainers prefer that shape, with regression coverage proving Google Chat tool replies use the inbound thread resource.

## Reproduction Assessment

Yes, at source level. Current main captures the Google Chat thread as `ReplyToId`, but absent a Google Chat `buildToolContext`, the message tool can use `MessageSidFull`/`MessageSid` as the current reply target.

## Solution Assessment

Yes. A channel-owned Google Chat threading adapter is the narrow maintainable fix because it maps an existing inbound thread resource into the existing message-tool reply context without changing shared core semantics or adding configuration.
