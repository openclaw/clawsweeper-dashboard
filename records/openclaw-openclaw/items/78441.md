## Summary

The feature is still necessary and the initial restricted-child behavior is convincingly demonstrated, but the persisted allowlist is implemented as an overridable default: a later Gateway agent call can supply a different `toolsAllow` and broaden the child beyond the restriction established at spawn. That is a blocking least-privilege and session-state defect, not a reason to close the PR.

## What This Changes

Adds `toolsAllow` to `sessions_spawn` and Gateway agent requests, persists it on native child sessions, and forwards it through embedded and CLI-backed execution paths.

## Best Possible Solution

Keep one canonical `toolsAllow` feature, but treat the persisted child-session list as an immutable upper bound: intersect it with any later request-level list, preserve omitted versus explicit-empty semantics, and prove both initial and resumed turns remain restricted across Gateway and embedded execution.

## Reproduction Assessment

Yes, from the current branch source: spawn persists `runtimeToolsAllow`, while `agentCommandInternal` uses it only when a later request omits `toolsAllow`; supplying another list replaces the durable restriction. A focused resumed-session test can demonstrate the broadening without external services.

## Solution Assessment

No, not yet. Reusing the existing embedded-runner allowlist is the right mechanism, but the durable session policy must be an upper bound rather than an overridable default; otherwise the advertised follow-up-turn restriction is not reliable.
