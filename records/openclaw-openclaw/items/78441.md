## Summary

Keep open: this is a useful scoped subagent permission feature with solid happy-path proof, but the patch still lets a later explicit Gateway `toolsAllow` widen a child session's stored tool policy, so it is not safe to merge yet.

## What This Changes

The PR adds `toolsAllow` to `sessions_spawn` and Gateway `agent.run`, persists it as `runtimeToolsAllow` on child sessions, forwards it into attempt execution, and updates protocol models, docs, tests, and prompt snapshots.

## Best Possible Solution

Merge only after stored `runtimeToolsAllow` is enforced as a non-widenable ceiling for later child-session turns, with focused tests for inherited-only, same, narrower, and attempted wider explicit allowlists.

## Reproduction Assessment

Yes from source inspection: create a child session with stored `runtimeToolsAllow: ["read"]`, then call the same child session through Gateway with explicit `toolsAllow: ["read", "exec"]`; the changed `effectiveOpts` path forwards the wider list.

## Solution Assessment

No. Persisting and forwarding the allowlist is the right layer, but the implementation should treat stored child-session policy as a ceiling by rejecting or intersecting wider per-run lists.
