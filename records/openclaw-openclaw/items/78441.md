## Summary

Keep open: this is useful feature work, but the PR still exposes an access-control allowlist that CLI-backed native runs can silently bypass, and its runtime proof is stale for the current head.

## What This Changes

The PR adds `sessions_spawn.toolsAllow` plumbing through the subagent tool, Gateway agent request, Swift protocol models, embedded-runner prompt handling, docs, changelog, prompt snapshots, and focused tests.

## Best Possible Solution

Land a maintainer-approved per-spawn allowlist only after every supported native backend either enforces or rejects it, the docs state the exact backend scope, and current-head Gateway proof demonstrates the enforced behavior.

## Reproduction Assessment

Not applicable for the feature request against current main; source inspection shows `toolsAllow` is absent today. The blocking PR defect is source-reproducible: a native subagent using a CLI provider returns through `runCliAgent` before embedded-runner filtering can apply.

## Solution Assessment

No. Reusing embedded-runner filtering is narrow for embedded runs, but the PR should fail closed or enforce equivalent behavior for CLI-backed native runs before presenting `toolsAllow` as a subagent access-control field.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Fail closed for CLI-backed toolsAllow runs:** `src/agents/command/attempt-execution.ts:614`
  - body: This forwards `toolsAllow` only to the embedded-runner branch. Please reject `toolsAllow` for CLI-backed native runs or enforce equivalent filtering there.
  - confidence: 0.86
