## Summary

## Best Possible Solution

Land the additive `toolsAllow` API only after stored child-session allowlists are authoritative for follow-up turns and covered by widening-attempt regression tests.

## Reproduction Assessment

Yes from source inspection: a child session stores `runtimeToolsAllow`, Gateway forwards request `toolsAllow`, and `agent-command.ts` only inherits the stored value when the request omits one. I did not run a live widening repro in this read-only review.

## Solution Assessment

No: the PR is close and uses the right embedded-runner boundary, but it is not the best merge shape until stored allowlists cannot be widened by later request-level values. The safer repair is to treat the stored list as authoritative, either by rejecting widening or intersecting with the stored policy.
