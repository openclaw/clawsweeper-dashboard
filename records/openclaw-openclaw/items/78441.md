## Summary

Keep open: the prior duplicate-field blockers are fixed and CI/proof look mostly healthy, but the allowlist is only applied to the initial spawned run, so persistent session-mode children can lose the intended restriction on follow-up turns.

## What This Changes

The branch adds `sessions_spawn.toolsAllow`, forwards it through native subagent, Gateway, embedded-runner, and Codex dynamic-tool paths, rejects unsupported ACP/CLI paths, and updates docs, changelog, protocol models, snapshots, and tests.

## Best Possible Solution

Keep the allowlist plumbing, but either persist the policy for session-mode children and reapply it on follow-up runs, or reject `toolsAllow` with `thread`/`mode: "session"` until that contract exists.

## Reproduction Assessment

Yes for the PR defect by source inspection: the PR passes `toolsAllow` only on the initial child `agent` request, while `sessions_send` follow-up turns build new `agent` params without it. I did not run local tests because this review is read-only and tests can write caches.

## Solution Assessment

No as submitted. Reusing existing embedded-runner allowlist machinery is a maintainable direction, but the current patch should not expose the option for persistent session spawns unless the restriction survives later child turns.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Persist or reject toolsAllow for session-mode spawns:** `src/agents/subagent-spawn.ts:1143`
  - body: This only attaches `toolsAllow` to the initial Gateway `agent` call for the spawned child. If the caller creates a thread-bound or `mode: "session"` child, later `sessions_send` turns start new `agent` runs without `toolsAllow`, so the child can regain its normal tool policy after the first turn. Persist the allowlist for the child session and reapply it on follow-up turns, or reject `toolsAllow` for session-mode spawns.
  - confidence: 0.88
