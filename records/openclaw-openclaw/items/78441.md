## Summary

This PR is useful but should not merge yet: the latest head fixes the earlier CLI-backed native bypass and provides real embedded-runner proof, but source review still finds an empty-allowlist enforcement gap in the Codex native harness path that the new forwarding would expose.

## What This Changes

The PR adds a `toolsAllow` array to `sessions_spawn`, forwards it through native subagent Gateway runs into embedded attempts, rejects unsupported ACP and CLI-backed native paths, and updates docs, changelog, generated protocol/schema snapshots, and focused tests.

## Best Possible Solution

Keep the feature direction, but make every native harness that accepts forwarded `toolsAllow` treat `undefined` as default behavior and any defined array, including `[]`, as an authoritative allowlist before merge.

## Reproduction Assessment

Yes for the remaining blocker by source inspection: a `sessions_spawn` native Codex app-server run with `toolsAllow: []` would reach a filter that only narrows non-empty arrays and otherwise keeps the full dynamic tool list. I did not run a live Codex app-server session during this read-only review.

## Solution Assessment

No, not yet: forwarding the field is the right narrow shape, and rejecting ACP/CLI is good, but the implementation must close the Codex native harness empty-array bypass or reject that harness path before claiming `[]` disables tools.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Honor empty allowlists in native harnesses:** `src/agents/command/attempt-execution.ts:616`
  - body: This forwards `toolsAllow` into native runs, but the Codex app-server filter only filters when the array is non-empty (`params.toolsAllow && params.toolsAllow.length > 0`). A spawn with `toolsAllow: []` therefore still receives the normal dynamic tools, contradicting the new schema/docs promise that `[]` disables tools and creating an enforcement gap. Please make native harness filtering use `toolsAllow !== undefined` semantics, or reject unsupported harnesses before dispatch.
  - confidence: 0.87
