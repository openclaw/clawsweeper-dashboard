## Summary

Keep this PR open: the feature is useful and has good current-head proof, but the patch still lets a later Gateway `agent` request widen a stored child-session `toolsAllow` policy. That is a concrete security/session-state blocker before merge.

## What This Changes

The PR adds `toolsAllow` to `sessions_spawn` and Gateway agent runs, persists it for native child sessions, and updates protocol/schema/docs/tests/prompt snapshots.

## Best Possible Solution

Make the persisted child-session allowlist the authoritative ceiling for follow-up turns, allowing later requests only to keep or narrow that policy, then let maintainers approve the additive API boundary.

## Reproduction Assessment

Yes, from source inspection: a child session can store `runtimeToolsAllow`, while `agentCommand` only uses that stored value when request `opts.toolsAllow` is absent. A later Gateway `agent` call for the same child session can therefore supply a wider list and bypass the stored policy.

## Solution Assessment

No. The PR is in the right owner area, but the best fix must make stored `runtimeToolsAllow` a non-widenable ceiling, with focused regression tests for later follow-up turns.
