## Summary

The PR adds a useful native subagent tool-allowlist surface, but it should not merge yet because follow-up runs can override and widen a persisted child-session allowlist. That turns a durable session capability boundary into a request-level preference, which is the wrong default for a stored subagent policy.

## What This Changes

The PR adds `toolsAllow` to `sessions_spawn`, forwards it through Gateway agent runs into embedded subagent attempts, persists it on child session entries, and updates focused tests, docs, generated protocol/schema output, and prompt snapshots.

## Best Possible Solution

Before merge, make persisted `SessionEntry.runtimeToolsAllow` an upper bound for later child-session runs or require an explicit maintainer-approved override contract with focused regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: create or load a child session with `runtimeToolsAllow:["read"]`, then call the Gateway agent method for that session with a broader `toolsAllow`; the PR code prefers the request value over the stored policy.

## Solution Assessment

No. The new surface is useful, but the best fix must settle stored-policy precedence before exposing it as durable session behavior.
