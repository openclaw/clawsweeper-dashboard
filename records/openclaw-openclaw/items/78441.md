## Summary

Keep open: this PR is a plausible implementation path for the open subagent tool-policy request, but it still has a blocking retained-session policy gap and the supplied real behavior proof targets an older head than the live PR.

## What This Changes

The PR adds `sessions_spawn.toolsAllow`, forwards it through native subagent and Gateway agent runs into embedded/Codex tool filtering, rejects unsupported runtimes, and updates docs, changelog, Swift protocol models, tests, and prompt snapshots.

## Best Possible Solution

Define one enforced subagent tool-policy contract: either make `toolsAllow` truly one-shot by requiring deletion after restricted spawns, or persist the policy on the child session and reapply it to all follow-up turns.

## Reproduction Assessment

Not applicable: this is an additive feature PR, not a bug report. Source inspection does provide a high-confidence review path for the retained-session policy gap.

## Solution Assessment

No: reusing the embedded runner allowlist is a good narrow building block, but the PR should close the retained-session follow-up gap before presenting this as enforced subagent tool isolation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Prevent unrestricted follow-up turns after toolsAllow spawns:** `src/agents/tools/sessions-spawn-tool.ts:367-370`
  - body: `toolsAllow` is rejected for thread/session spawns, but normal run-mode spawns still default to `cleanup: "keep"` and return a `childSessionKey`. A parent can then call `sessions_send` to that retained child; that path starts a new Gateway `agent` run without any `toolsAllow`, so a child advertised as restricted gets full tools on follow-up. Either require `cleanup: "delete"` when `toolsAllow` is set, or persist and reapply the allowlist for follow-up turns.
  - confidence: 0.87
