## Summary

Keep this PR open: the `toolsAllow` feature is useful and well-proven for the initial spawn, but the branch still lets a later explicit Gateway `toolsAllow` widen a persisted child-session allowlist, which breaks the durable session policy it adds.

## What This Changes

The PR adds `toolsAllow` plumbing for native `sessions_spawn` and Gateway `agent.run`, persists it on spawned subagent sessions, forwards it into embedded attempts, updates protocol/docs, and refreshes prompt snapshots.

## Best Possible Solution

Make stored `runtimeToolsAllow` the upper bound for every later child-session turn, add a regression test for attempted widening, then have maintainers accept the additive public API semantics before resolving the dirty merge.

## Reproduction Assessment

Yes. Source inspection of the PR head shows `agentCommandInternal` only applies stored `runtimeToolsAllow` when the incoming request omits `toolsAllow`, so a later explicit request can widen a persisted child session.

## Solution Assessment

No. The API direction is plausible, but the best implementation must treat the stored child-session allowlist as an upper bound rather than a default that later Gateway requests can replace.
