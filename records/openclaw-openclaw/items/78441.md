## Summary

Keep open: the supplied proof covers the initial Gateway-backed native spawn path, but the PR still lets a later explicit Gateway `toolsAllow` broaden a child session's persisted `runtimeToolsAllow`, which is a session-state and tool-boundary blocker before merge.

## What This Changes

The PR adds `toolsAllow` plumbing for `sessions_spawn` and Gateway `agent.run`, persists it on native child sessions, forwards it into embedded/CLI runtime boundaries, and updates protocol models, docs, tests, and prompt snapshots.

## Best Possible Solution

Make the stored child-session allowlist the ceiling for all follow-up turns, either by always applying it over explicit per-run values or by rejecting explicit values that are not a subset of the stored policy.

## Reproduction Assessment

Yes. Source inspection is enough to reproduce the review finding: create a child session with stored `runtimeToolsAllow:["read"]`, then run it later with explicit `toolsAllow:["read","exec"]`; the PR's `effectiveOpts` chooses the explicit value.

## Solution Assessment

No, not yet. The owner boundary is reasonable, but the safest solution must treat the stored child-session policy as authoritative or reject explicit widening before dispatch.
