## Summary

Keep open: the PR is useful and well-proven for the initial spawn path, but a later direct Gateway run can widen a stored child-session tool allowlist, so the security/session boundary needs a narrow fix before maintainer API acceptance.

## What This Changes

The PR adds optional `toolsAllow` plumbing for native `sessions_spawn` and Gateway agent runs, persists it on child sessions, forwards it into embedded/CLI run boundaries, and updates docs, generated protocol models, tests, and prompt snapshots.

## Best Possible Solution

Make stored `runtimeToolsAllow` a non-widening upper bound for every follow-up child-session turn, then have maintainers explicitly accept the public API and fail-closed semantics before merge.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: create a child session with stored `runtimeToolsAllow: ["read"]`, then send a later `agent.run` for that session with `toolsAllow: ["read", "exec"]`; the new `effectiveOpts` branch uses the caller-provided list instead of the stored bound.

## Solution Assessment

No. The plumbing is the right layer, but the durable allowlist should be intersected with or reject broader per-run values so follow-up requests cannot widen the stored child policy.
