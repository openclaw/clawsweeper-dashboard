## Summary

Keep open: the feature is useful and the supplied Gateway proof is sufficient, but the patch lets a later explicit Gateway `toolsAllow` replace a persisted child-session allowlist, which can broaden a subagent restriction that `sessions_spawn` saved for follow-up turns.

## What This Changes

The branch adds `toolsAllow` to `sessions_spawn` and Gateway `agent.run`, persists it on child sessions, forwards it through embedded/CLI runtime paths, and updates protocol models, docs, tests, and prompt snapshots.

## Best Possible Solution

Keep the additive plumbing, but make persisted `runtimeToolsAllow` an upper bound for every later run on that child session, rejecting or intersecting explicit per-run values that would broaden it, with focused regression coverage.

## Reproduction Assessment

Yes for the review finding: source inspection shows `spawnSubagentDirect` stores `runtimeToolsAllow`, while `agentCommandInternal` only uses it when incoming `opts.toolsAllow` is undefined. A later Gateway call with a different explicit list therefore follows the broader explicit list.

## Solution Assessment

No, not yet: the PR is in the right Gateway/session/embedded-runner area, but persisted child-session allowlists must remain authoritative or be intersected before the public field is safe to merge.
