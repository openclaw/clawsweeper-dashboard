## Summary

Keep open. The feature is useful and the current-head Gateway proof is strong, but the branch still lets a later `agent.run` request widen a persisted child-session tool allowlist, so it needs a narrow security fix plus maintainer approval of the new public API semantics before merge.

## What This Changes

The PR adds `toolsAllow` plumbing for `sessions_spawn` and Gateway `agent.run`, persists the allowlist on native child sessions, forwards it into embedded/CLI runner paths, updates protocol/docs/Swift models, and adds focused tests and prompt snapshots.

## Best Possible Solution

Make `runtimeToolsAllow` a durable upper bound for child sessions, keep unsupported ACP/CLI paths fail-closed, and land only after maintainer approval of the public `toolsAllow` API semantics.

Yes. Source inspection shows `spawnSubagentDirect` persists `runtimeToolsAllow`, Gateway forwards explicit `toolsAllow`, and `agentCommandInternal` lets explicit request options win over the stored policy.

## Solution Assessment

No. The plumbing layer is plausible, but the implementation is not the best fix until stored child-session policies are enforced as non-widening and the new public API/security semantics are maintainer-approved.
