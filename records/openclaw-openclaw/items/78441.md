## Summary

Keep open; the branch is close, but a continued child session can still widen its stored tool allowlist through a later Gateway run, so the new durable tool policy is not yet a reliable session/security boundary.

## What This Changes

The PR adds `toolsAllow` to `sessions_spawn` and Gateway `agent.run`, persists `runtimeToolsAllow` on native subagent sessions, forwards it into embedded and CLI attempt paths, and updates docs/protocol/generated snapshots plus focused tests.

## Best Possible Solution

Make stored child-session `runtimeToolsAllow` authoritative as a ceiling for later turns, or reject conflicting per-run allowlists, with regression coverage for the wider-follow-up case before merge.

## Reproduction Assessment

Yes from source inspection: a persisted child `runtimeToolsAllow` is only inherited when `opts.toolsAllow` is undefined, while Gateway can supply an explicit wider list. I did not run a live repro because the branch logic is directly visible in the PR head.

## Solution Assessment

No. The plumbing is the right shape, but the best fix must make durable child-session policy authoritative, or clearly reject/document conflicting per-turn policy, before exposing this as a public API.
