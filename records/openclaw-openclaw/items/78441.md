## Summary

Keep open: the feature plumbing is mostly focused, but the proposed access-control surface currently accepts `toolsAllow` for native subagent runs that can route to CLI backends where the allowlist is not enforced, and the supplied real behavior proof is stale against the current PR head.

## What This Changes

The PR adds `sessions_spawn.toolsAllow`, forwards it through Gateway `agent` execution into embedded-runner tool filtering, updates docs/changelog/Swift protocol snapshots, and adds focused tests plus minor cleanup.

## Best Possible Solution

Land a maintainer-approved per-spawn allowlist only after every supported native backend either enforces or rejects it, the backend scope is documented, and current-head Gateway proof is refreshed.

## Reproduction Assessment

Not applicable for the feature request against current `main`; source inspection shows the requested surface is absent. The blocking PR defect is source-reproducible: configure a native subagent to use a CLI backend and pass `toolsAllow`, and the code returns through `runCliAgent` before embedded-runner filtering.

## Solution Assessment

No. Reusing embedded-runner filtering is a good narrow path for Pi-backed runs, but the PR should fail closed or enforce equivalent behavior for CLI-backed native runs before presenting `toolsAllow` as a subagent access-control field.

## Review Findings

Overall correctness: patch is incorrect

- **[P1] Fail closed when toolsAllow reaches CLI-backed runs:** `src/agents/command/attempt-execution.ts:614`
