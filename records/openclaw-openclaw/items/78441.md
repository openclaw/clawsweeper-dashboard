## Summary

The PR is useful but not merge-ready because the durable child-session tool allowlist is not enforced as a ceiling on later runs; a later explicit Gateway toolsAllow can widen a stored restriction, which undercuts the permission boundary the feature is trying to add.

## What This Changes

The PR adds a toolsAllow parameter for sessions_spawn/native subagent runs, stores it on child sessions, forwards it through Gateway agent runs, updates protocol/schema state, and adds focused tests and prompt snapshots.

## Best Possible Solution

Enforce runtimeToolsAllow as the durable child-session ceiling on every follow-up run, with explicit broader requests rejected or narrowed by a documented rule, before accepting the new Gateway/session API.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: spawn a child with runtimeToolsAllow:["read"], then invoke the same child session through Gateway with explicit toolsAllow:["read","exec"] and agentCommand will prefer the explicit list over the stored ceiling.

## Solution Assessment

No, the proposed path is close but not the best fix until the durable session allowlist is enforced as a ceiling for later runs or maintainers explicitly choose override semantics.
