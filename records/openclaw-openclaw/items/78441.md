## Summary

Keep this PR open: the public plumbing is still useful and current main does not expose this `sessions_spawn.toolsAllow` path, but the patch still lets a later explicit Gateway `agent.run.toolsAllow` override a stored child-session allowlist.

## What This Changes

The PR adds and documents `toolsAllow` for `sessions_spawn` and Gateway `agent.run`, persists it on child sessions, forwards it into runner paths, and updates protocol models, prompt snapshots, and tests.

## Best Possible Solution

Land the feature only after stored child-session runtime allowlists remain authoritative on every later turn, with explicit Gateway allowlists intersected or rejected and regression coverage for the widening case.

## Reproduction Assessment

Yes, from source: seed a child session with `runtimeToolsAllow: ["read"]`, then invoke the same session with explicit `toolsAllow: ["read", "exec"]`; the PR code uses the explicit option because it only inherits stored policy when `opts.toolsAllow` is undefined. I did not run tests in this read-only review.

## Solution Assessment

No. The forwarding/persistence shape is plausible, but the best fix must make the stored child-session allowlist authoritative before dispatching to embedded or CLI runner paths.
