## Summary

This PR is still useful but not ready to merge: the new durable child-session `runtimeToolsAllow` policy can be widened on a later `agent.run` call because the stored allowlist is ignored whenever the caller supplies a per-run `toolsAllow`. The real behavior proof is strong for the initial spawn path, but the durable session/security semantics need one blocking repair and the dirty merge state still needs refresh.

## What This Changes

The PR adds `toolsAllow` plumbing for native `sessions_spawn` and Gateway agent runs, persists it on child sessions, forwards it to embedded/CLI runtime boundaries, and updates protocol models, docs, prompt snapshots, and tests.

## Best Possible Solution

Land only after stored child-session `runtimeToolsAllow` remains a non-widening upper bound across follow-up turns, with explicit tests for stored plus per-run policy composition and accepted public API semantics.

Yes for the blocking review finding: source inspection shows stored `runtimeToolsAllow` is applied only when no per-run `toolsAllow` is supplied, while Gateway now accepts and forwards per-run `toolsAllow`. I did not run a live repro because this is a read-only review.

## Solution Assessment

No, not yet: the PR is the right owner boundary for public `sessions_spawn`/Gateway plumbing, but stored child-session allowlists must be treated as an upper bound rather than replaceable per-run state.
