## Label Justifications

- `P1`: The PR changes agent tool permission semantics and currently leaves a path to widen a persisted subagent allowlist before merge.
- `merge-risk: 🚨 session-state`: Merging as-is would store `runtimeToolsAllow` on child sessions without enforcing it as the authoritative later-turn session policy.
- `merge-risk: 🚨 security-boundary`: A broadened later `toolsAllow` can re-expose tools such as `exec` that the stored child-session policy was meant to remove.

## Summary

Keep this PR open but do not merge it yet: the feature plumbing is still useful and current main does not expose `sessions_spawn.toolsAllow`, but the PR still lets a later turn widen a stored child-session allowlist.

## What This Changes

This PR adds `toolsAllow` to `sessions_spawn` and Gateway `agent.run`, persists it on native subagent sessions, forwards it into embedded/CLI attempt paths, and updates docs, generated protocol models, tests, and prompt snapshots.

## Best Possible Solution

Keep the additive `toolsAllow` plumbing, but enforce any stored `runtimeToolsAllow` as a durable ceiling on every later child-session turn before merge.

## Reproduction Assessment

Yes: source inspection gives a high-confidence reproduction path on the PR branch by seeding a child session with `runtimeToolsAllow: ["read"]` and invoking the same session with `toolsAllow: ["read", "exec"]`. The PR currently forwards the incoming broader list instead of enforcing the stored ceiling.

## Solution Assessment

No: this is the right owner boundary for the feature, but the current implementation is only a plausible fix because the durable session policy is not authoritative on later turns. The safer path is to intersect or reject broadened later-turn `toolsAllow` before dispatching to embedded/CLI attempt execution.
