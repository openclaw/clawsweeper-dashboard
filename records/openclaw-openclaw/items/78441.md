## Summary

Keep open: the feature and current-head proof are useful, but the branch still has a blocking tool-policy bug where a persisted child-session allowlist can be widened by a later explicit Gateway run allowlist.

## What This Changes

The PR adds `toolsAllow` plumbing for native `sessions_spawn` and Gateway `agent.run`, persists it on child sessions, forwards it into embedded/CLI runtime boundaries, and updates docs, generated protocol models, tests, and prompt snapshots.

## Best Possible Solution

Keep the feature, but make stored child-session `runtimeToolsAllow` authoritative over any later per-run `toolsAllow`, either by rejecting non-subset conflicts or intersecting before runner dispatch.

## Reproduction Assessment

Yes, from source inspection: create a child session with stored `runtimeToolsAllow: ["read"]`, then continue it through Gateway `agent.run` with `toolsAllow: ["read", "exec"]`; the PR head chooses the explicit list and forwards it to the embedded runner.

## Solution Assessment

No, not yet; the plumbing is in the right boundary, but the durable session policy must remain authoritative or reject conflicting per-run values before this is the safest implementation.
