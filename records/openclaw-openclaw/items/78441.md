## Summary

Keep open; this is useful least-privilege subagent plumbing, but it is not merge-ready because a later direct Gateway run can widen a stored child-session tool allowlist.

## What This Changes

The PR adds `toolsAllow` to native `sessions_spawn` and Gateway `agent.run`, persists it on child sessions, forwards it into embedded and CLI attempt paths, and updates docs, protocol models, tests, and prompt snapshots.

## Best Possible Solution

Make stored `runtimeToolsAllow` a non-widening cap for follow-up child-session turns, then have maintainers explicitly accept the public API and fail-closed semantics.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: create a child session with stored `runtimeToolsAllow: ["read"]`, then send a later Gateway `agent.run` for that session with `toolsAllow: ["read", "exec"]`; PR head uses the request value instead of bounding it by the stored value.

## Solution Assessment

No. The plumbing is in the right owner path, but stored child-session policy should intersect with or reject broader per-run values so later turns cannot widen a durable allowlist.
