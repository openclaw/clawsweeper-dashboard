## Summary

Keep open: the feature is useful and the supplied Gateway proof is credible, but the PR currently lets a later explicit `agent.run.toolsAllow` replace a stored child-session allowlist, which undermines the durable subagent tool boundary.

## What This Changes

The PR adds `toolsAllow` plumbing from `sessions_spawn` and Gateway `agent.run` through native embedded subagent sessions, docs, protocol models, and focused tests.

## Best Possible Solution

Preserve `runtimeToolsAllow` as the durable child-session ceiling, allowing later per-run values only to narrow that stored policy or rejecting attempted widening before the runner starts.

## Reproduction Assessment

Yes, source-reproducible: create a child session with stored `runtimeToolsAllow: ["read"]`, then call `agent.run` for that child with explicit `toolsAllow: ["exec"]`; the current PR chooses the explicit opts instead of the stored policy.

## Solution Assessment

No, not yet; forwarding and persisting the allowlist is the right shape, but the stored policy must be treated as a ceiling rather than a default that explicit per-run input can replace.

- **[P1] Do not let per-run toolsAllow widen stored child policy:** `src/agents/agent-command.ts:847-849`
