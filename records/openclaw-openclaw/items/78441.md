## Summary

Keep open: the feature is useful and the happy-path proof is strong, but the branch is not mergeable yet because a persisted `runtimeToolsAllow` can still be bypassed by an explicit wider `toolsAllow` on a later Gateway continuation.

## What This Changes

The branch adds `toolsAllow` plumbing for `sessions_spawn` and Gateway `agent.run`, persists native child-session allowlists, updates generated protocol/docs/prompt snapshots, and adds focused tests.

## Best Possible Solution

Persisted child-session allowlists should remain authoritative on follow-up turns, with explicit request allowlists allowed only when they narrow or intersect the stored policy.

## Reproduction Assessment

Yes. Source inspection shows `spawnSubagentDirect` persists `runtimeToolsAllow`, Gateway forwards explicit `toolsAllow`, and `agent-command.ts` uses the stored value only when the explicit request value is undefined.

## Solution Assessment

No, not yet. The plumbing is in the right owner path, but the stored child-session policy needs to be authoritative before this is the best maintainable fix.
