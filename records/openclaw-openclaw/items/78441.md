## Summary

Keep open: the PR is still useful because current main lacks the public `sessions_spawn.toolsAllow` plumbing, but current PR head reintroduces a fail-open Gateway `agent.run` boundary by forwarding raw `toolsAllow` values into the embedded tool-policy matcher.

## What This Changes

Adds an allow-only `toolsAllow` parameter to `sessions_spawn` and Gateway `agent.run`, persists it on child session metadata, passes it into embedded runner filtering, and updates docs, protocol models, prompt snapshots, and focused tests.

## Best Possible Solution

Land the additive per-spawn allowlist only after Gateway `agent.run` normalizes `toolsAllow` with the same fail-closed semantics as `sessions_spawn` and maintainers accept the public API/session-state contract.

## Reproduction Assessment

Yes. Source inspection shows `agent.run.toolsAllow:[" "]` passes the new Gateway schema, is forwarded raw, and reaches the embedded matcher path where blank allow patterns compile away and an empty compiled allow set allows every tool.

## Solution Assessment

Mostly yes on ownership: threading the policy from `sessions_spawn` through Gateway, session metadata, and embedded runner filtering is the right layer, but it is not the best merge-ready fix until the direct Gateway input path has matching normalization.
