## Summary

Keep open: this PR targets a real iMessage receive blocker at the right shared RPC client boundary, but the new stdout handler can corrupt multibyte UTF-8 across chunk boundaries and the external PR still lacks after-fix real behavior proof.

## What This Changes

The PR replaces iMessage RPC stdout readline framing with a manual LF-only buffer and adds unit coverage for U+2028/U+2029 JSON-RPC responses.

## Best Possible Solution

Use one corrected iMessage RPC client fix that splits only on LF while preserving streaming UTF-8, add a split-multibyte regression test, and land it with redacted real imsg proof for https://github.com/openclaw/openclaw/issues/89830.

## Reproduction Assessment

Yes for a source-level reproduction: current main uses `node:readline` and per-line `JSON.parse`, and a local Node probe shows raw U+2028 splits one valid JSON object into unparseable fragments. No live macOS/imsg reproduction was run in this read-only review.

## Solution Assessment

No as submitted: LF-only framing in `IMessageRpcClient` is the right fix boundary, but the implementation must preserve UTF-8 across Buffer chunks. A streaming decoder plus focused regression coverage is the safer maintainable version.

- **[P1] Decode stdout with a streaming UTF-8 decoder:** `extensions/imessage/src/client.ts:102`
  - body: This new handler decodes every Buffer chunk independently before appending to `stdoutBuffer`. If `imsg` splits a multibyte character across chunks, `Buffer.toString("utf-8")` replaces the partial bytes with U+FFFD, so inbound text containing U+2028, emoji, or other non-ASCII content can be silently corrupted before `JSON.parse`; use `StringDecoder` or stream encoding and add a split-multibyte chunk test.
  - confidence: 0.91
