## Summary

Keep open: the PR targets the right iMessage RPC framing bug, but the current branch still decodes arbitrary stdout Buffer chunks with `Buffer.toString()` before buffering, which can corrupt split UTF-8 code points, and it still lacks real macOS/imsg behavior proof.

## What This Changes

This PR replaces the iMessage RPC client's `node:readline` stdout framing with a manual LF-only buffer and adds client tests for U+2028/U+2029 JSON-RPC responses.

## Best Possible Solution

Use one corrected iMessage RPC client fix that splits only on LF while preserving streaming UTF-8, add split-multibyte regression coverage, and land it with redacted real imsg proof for https://github.com/openclaw/openclaw/issues/89830.

## Reproduction Assessment

Yes, for source/dependency reproduction: current main and the latest release use `node:readline` on `imsg rpc` stdout, and a Node v24.16.0 probe split a JSON object containing raw U+2028 into two lines. I did not run a live macOS/imsg reproduction in this read-only review.

## Solution Assessment

No as submitted: LF-only framing in `IMessageRpcClient` is the right fix boundary, but the implementation must preserve UTF-8 across Buffer chunks. A streaming decoder plus split-byte regression coverage is the safer maintainable solution.
