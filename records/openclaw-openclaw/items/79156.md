## Summary

Keep open: current main still has the existing Discord STT/TTS voice path, and the PR head has blocking realtime stream-lifecycle and speaker-auth-context bugs. The protected `maintainer` label also rules out cleanup closure.

## What This Changes

The PR adds Discord `voice.mode` config/schema/docs and runtime wiring for `stt-tts`, realtime `talk-buffer`, and bidi realtime consult modes, plus shared realtime consult policy helpers and tests.

## Best Possible Solution

Keep the shared realtime bridge direction, but pin speaker/authorization context to each queued consult and close unauthorized manual receive streams, with receive-recovery reset on successful realtime ingest, before this lands.

## Reproduction Assessment

Yes for the review blockers by source inspection: realtime capture creates a manual receive stream and returns early on failed ingress auth, and async consults read mutable session-level speaker context after it can be overwritten. I did not run live Discord voice because this review is read-only.

## Solution Assessment

No, not yet. Reusing the shared realtime voice bridge is a reasonable direction, but the current patch needs per-turn auth context, unauthorized stream cleanup, and recovery-state parity with the STT path before it is the maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Destroy unauthorized realtime capture streams:** `extensions/discord/src/voice/manager.ts:631`
  - body: In realtime modes, this branch returns after ingress auth fails, but the stream was created with `EndBehaviorType.Manual` and `finishVoiceCapture` then removes it from capture state without destroying it. Repeated unauthorized speech can leave Discord receive streams alive until session teardown.
  - confidence: 0.91
- **[P1] Keep realtime speaker context per turn:** `extensions/discord/src/voice/realtime.ts:164`
  - body: `setSpeakerContext` stores owner status and prompt context on the shared session, while talk-buffer debounced consults and bidi tool calls run later. A second speaker can overwrite this field before the first consult executes, so the wrong `senderIsOwner` and prompt context can be used.
  - confidence: 0.93
- **[P2] Reset decrypt-failure state after realtime ingest:** `extensions/discord/src/voice/manager.ts:570-571`
  - body: The realtime branch returns after `processRealtimeAudioCapture` and skips the `resetDecryptFailureState` call used by the STT path after successful decode. Intermittent DAVE decrypt errors can accumulate across successful realtime captures and trigger unnecessary rejoin recovery.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff keeps ingress authorization in the new helper, but still has cross-speaker authorization-context reuse and unauthorized manual stream leak risks.

Concerns:

- **[high] Cross-speaker owner context reuse:** `extensions/discord/src/voice/realtime.ts:164`
  - body: Realtime speaker context is stored on the session instead of the queued turn/tool call, so one Discord speaker can overwrite another speaker's owner status before the agent consult executes, mixing owner-only tool eligibility and prompt context.
