## Summary

Keep open: current main does not have Discord realtime voice modes, and the latest PR head still has source-backed blockers in realtime capture timing, receive recovery, teardown, and talk-buffer speaker metadata.

## What This Changes

The PR adds Discord `voice.mode` and `voice.realtime` configuration, realtime voice bridge integration, consult policy plumbing, docs, changelog, and focused tests.

## Best Possible Solution

Keep the shared realtime bridge direction, fix the P1/P2 blockers with focused regression tests, then land with maintainer approval and live Discord proof or an explicit proof override.

## Reproduction Assessment

Yes for the review blockers by source and dependency inspection: the PR delays `receiver.subscribe`, returns before receive-recovery reset, closes realtime only from `stop()`, and coalesces talkback metadata across fragments. I did not run a live Discord voice call.

## Solution Assessment

No, not yet. Reusing the shared realtime voice bridge is reasonable, but the PR should subscribe promptly while preserving authorization cleanup, reset receive recovery after successful realtime ingest, close the bridge on all teardown paths, and avoid merging text across different speaker contexts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Subscribe before realtime ingress awaits:** `extensions/discord/src/voice/manager.ts:551-553`
  - body: `handleSpeakingStart` waits for async guild/member/auth resolution before creating the receive stream. `@discordjs/voice` drops UDP packets when there is no subscription for that user, so normal lookup latency can clip realtime turn starts and delay barge-in; subscribe first, then destroy or ignore the stream if authorization fails.
  - confidence: 0.9
- **[P1] Avoid merging talkback text across speaker contexts:** `src/talk/agent-talkback-runtime.ts:142-143`
  - body: `appendPendingQuestion` concatenates the next transcript fragment but keeps only the current metadata. In Discord talk-buffer mode, that metadata carries speaker identity and owner status, so a later speaker's text can run under the first speaker's auth context. Split entries or only merge fragments with the same speaker metadata.
  - confidence: 0.93
- **[P2] Reset receive recovery after realtime audio:** `extensions/discord/src/voice/manager.ts:584-591`
  - body: The realtime branch returns immediately after `processRealtimeAudioCapture`, so successful realtime audio never clears the DAVE decrypt-failure counter like the STT path does. Intermittent decrypt errors can accumulate across otherwise successful realtime turns and trigger unnecessary rejoin recovery.
  - confidence: 0.86
- **[P2] Close realtime bridge on every teardown path:** `extensions/discord/src/voice/manager.ts:356`
  - body: This closes the realtime bridge only from `stop()`, but disconnect-recovery failure and destroyed handlers can clear/destroy the voice session without calling `stop()`. In realtime modes that can leave the provider bridge/socket running after Discord voice has gone away.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: Talk-buffer metadata coalescing can reuse the first speaker's owner context for later speaker text, which is security-sensitive for realtime consult tool policy.

Concerns:

- **[high] Cross-speaker talkback context reuse:** `src/talk/agent-talkback-runtime.ts:143`
  - body: `appendPendingQuestion` keeps `current.metadata` while appending `next.question`; in Discord talk-buffer mode, mixed-speaker fragments can be consulted with the wrong `senderIsOwner` and prompt context.
  - confidence: 0.93
