## Summary

This maintainer-labeled feature PR should stay open for normal maintainer handling; the direction is useful, but the current diff has blocking realtime voice correctness and security issues around stream cleanup and speaker authorization context.

## What This Changes

The PR adds Discord `voice.mode` config, realtime bridge-backed `talk-buffer`/`bidi` modes, shared consult policy guidance, tests, docs, and a changelog entry.

## Best Possible Solution

Keep the shared realtime bridge direction, but scope speaker authorization context to each transcript/tool-call turn and always close or drain unauthorized manual receive streams before this lands.

## Reproduction Assessment

Yes for the review blockers by source inspection: realtime mode creates manual receive streams and returns early on failed ingress auth, and async consults read mutable session-level speaker context after it can be overwritten. I did not run live Discord voice because this review is read-only.

## Solution Assessment

No, not yet. Reusing the shared realtime voice bridge is the right direction, but the current implementation needs per-turn auth context and unauthorized-stream cleanup before it is the narrow maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Destroy unauthorized realtime capture streams:** `extensions/discord/src/voice/manager.ts:630`
  - body: In realtime modes, unauthorized speakers return from this branch without draining or destroying the `receiver.subscribe(..., EndBehaviorType.Manual)` stream. `finishVoiceCapture` then removes the stream from capture state, so the later speaker-end finalizer has no handle to destroy, leaking receive streams for repeated unauthorized attempts.
  - confidence: 0.91
- **[P1] Keep realtime speaker context scoped to each turn:** `extensions/discord/src/voice/realtime.ts:164`
  - body: `setSpeakerContext` writes owner status and channel prompt context into a shared realtime session field, but talk-buffer debounced consults and bidi tool calls read it later in `runAgentTurn`. A second speaker can overwrite this before the first consult runs, causing the first turn to use the wrong `senderIsOwner` and prompt context.
  - confidence: 0.93

## Security Review

Status: needs_attention

Summary: The same-file missing-authorize bot finding appears mitigated by the new ingress helper, but the diff introduces concrete cross-speaker authorization-context and unauthorized-stream leak risks.

Concerns:

- **[high] Cross-speaker owner context reuse:** `extensions/discord/src/voice/realtime.ts:164`
  - body: Because realtime speaker context is stored on the session instead of the queued turn/tool call, one Discord speaker can overwrite another speaker's owner status before the agent consult executes, which can mix up owner-only tool eligibility and prompt context.
  - confidence: 0.92
- **[medium] Unauthorized receive stream leak:** `extensions/discord/src/voice/manager.ts:630`
  - body: Failed realtime ingress authorization returns without closing the manual Discord receive stream, creating a resource-exhaustion path for repeated unauthorized speaking attempts.
  - confidence: 0.9
