## Summary

Keep open. Current main does not include Discord realtime voice modes, and the latest PR head fixed the earlier unauthorized pre-subscribe receive-stream issue, but source inspection still finds blocking cross-speaker context reuse and realtime receive-recovery parity gaps.

## What This Changes

The PR adds Discord `voice.mode`/`voice.realtime` config, docs, runtime bridging to shared realtime voice sessions, consult policy helpers, and tests.

## Best Possible Solution

Keep the shared realtime bridge direction, but pin speaker/auth context to each queued transcript or tool call and reset receive recovery after successful realtime audio ingestion before merge.

## Reproduction Assessment

Yes for the review blockers by source inspection: a queued realtime consult reads mutable session-level speaker context after other speakers can overwrite it, and the realtime capture branch returns before receive-recovery reset. I did not run a live Discord voice call in this read-only review.

## Solution Assessment

No, not yet. Reusing the shared realtime voice bridge is a reasonable direction, but the PR needs per-turn auth context and receive-recovery parity before it is the maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Keep realtime speaker context per turn:** `extensions/discord/src/voice/realtime.ts:165-166`
  - body: `setSpeakerContext` stores owner status and prompt context on the shared realtime session, but talk-buffer debounced consults and bidi tool calls read it later. A second speaker can overwrite this field before the first consult executes, so the wrong `senderIsOwner`, prompt context, and owner-policy tool eligibility can be used.
  - confidence: 0.93
- **[P2] Reset decrypt-failure state after realtime ingest:** `extensions/discord/src/voice/manager.ts:584-587`
  - body: The realtime branch returns after `processRealtimeAudioCapture`, so it skips the `resetDecryptFailureState` call used by the STT path after successful decode. Intermittent DAVE decrypt errors can accumulate across successful realtime captures and trigger unnecessary rejoin recovery.
  - confidence: 0.84
