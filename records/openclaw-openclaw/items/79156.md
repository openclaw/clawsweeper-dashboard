## What This Changes

The PR adds Discord `voice.mode` and `voice.realtime` config, docs, shared realtime voice bridging, consult policy handling, and regression tests.

## Best Possible Solution

Keep the shared realtime bridge direction, but subscribe promptly while preserving unauthorized-speaker cleanup, pin speaker/auth context to each queued turn or tool call, reset receive recovery after successful realtime ingest, then land with focused tests and live Discord proof when feasible.

## Reproduction Assessment

Yes for the review blockers by source and dependency inspection: the PR delays `receiver.subscribe`, stores mutable session speaker context, and returns before receive-recovery reset. I did not run a live Discord voice call.

## Solution Assessment

No, not yet. Reusing the shared realtime voice bridge is a reasonable implementation direction, but the latest head needs the capture timing, per-turn auth context, and receive-recovery fixes before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Subscribe before awaiting realtime ingress checks:** `extensions/discord/src/voice/manager.ts:551-553`
  - body: `handleSpeakingStart` waits for async Discord ingress resolution before creating the receive stream. `@discordjs/voice` drops packets when there is no subscription for the user, so normal guild/member/auth latency can clip the start of realtime utterances and delay barge-in; subscribe promptly and destroy or ignore the stream if auth fails.
  - confidence: 0.9
- **[P1] Keep realtime speaker context per turn:** `extensions/discord/src/voice/realtime.ts:166-167`
  - body: `setSpeakerContext` writes the speaker identity and owner status into one mutable session field, but talk-buffer and bidi consults read it later in async code. A second speaker can overwrite the field before the first queued consult runs, mixing prompt context and owner-policy tool eligibility between users.
  - confidence: 0.93
- **[P2] Reset decrypt state after realtime ingest:** `extensions/discord/src/voice/manager.ts:584-587`
  - body: The realtime branch returns immediately after `processRealtimeAudioCapture`, so successful realtime audio does not clear the DAVE decrypt-failure counter like the STT path does. Intermittent decrypt errors can accumulate across successful realtime captures and trigger unnecessary rejoin recovery.
  - confidence: 0.84
