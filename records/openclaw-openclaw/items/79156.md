## What This Changes

The PR adds Discord `voice.mode`/`voice.realtime` config for STT/TTS, talk-buffer, and bidi modes, wires Discord PCM into the shared realtime voice bridge and agent consult path, and updates docs, changelog, and focused tests.

## Best Possible Solution

## Reproduction Assessment

Yes for the PR review blockers by source and dependency inspection: `@discordjs/voice` drops packets without a subscription, the realtime branch returns before recovery reset, and teardown paths bypass `stop()`. I did not run a live Discord voice call.

## Solution Assessment

No, not yet. Reusing the shared realtime bridge is a plausible direction, but the PR should fix subscription timing, receive recovery reset, and bridge cleanup before maintainer review or landing.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Subscribe before async ingress checks:** `extensions/discord/src/voice/manager.ts:551-553`
  - body: `handleSpeakingStart` awaits `resolveDiscordVoiceIngressContext` before `receiver.subscribe`. In `@discordjs/voice@0.19.2`, UDP packets are ignored when no subscription stream exists, so normal guild/member/auth latency can clip realtime turn starts and delay barge-in; subscribe first, then destroy or ignore the stream if authorization fails.
  - confidence: 0.91
- **[P2] Reset receive recovery after realtime audio:** `extensions/discord/src/voice/manager.ts:584-591`
  - body: The realtime branch returns after `processRealtimeAudioCapture`, so successful realtime audio never clears the DAVE decrypt-failure counter like the STT path does. Intermittent decrypt errors can accumulate across otherwise successful realtime turns and trigger unnecessary rejoin recovery.
  - confidence: 0.87
- **[P2] Close the realtime bridge from every teardown path:** `extensions/discord/src/voice/manager.ts:356`
  - body: `entry.realtime?.close()` only runs from `stop()`, while disconnect-recovery failure and destroyed handlers clear or destroy sessions without calling `stop()`. Realtime modes can leave the provider bridge or socket active after Discord voice is gone, so route all teardown paths through shared cleanup.
  - confidence: 0.84
