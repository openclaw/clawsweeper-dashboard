## Summary

Keep open. Current main still has no blocking plugin hook or documented lifecycle contract before Telegram preview streaming sends, edits, or finalizes visible content, and the related open items cover narrower final-hook bypasses or broader outbound-policy guarantees rather than fully replacing this streaming-edit API request.

## What This Changes

The issue requests a blocking hook or equivalent control so plugins can intercept, rewrite, or cancel Telegram streaming edit/final content before it becomes visible.

## Best Possible Solution

Define a shared, documented blocking pre-preview delivery contract that covers preview creation, partial updates, retained previews, and final edits while preserving Telegram's one-visible-preview behavior.

## Reproduction Assessment

Yes, at source level. Telegram preview streaming can call stream.update and finalize preview-finalized content without entering the normal hook-backed deliverReplies/runMessageSending path; no live bot run was performed.

## Solution Assessment

Unclear. A final-only message_edit_sending hook or narrow message_sending patch would not cover partial preview leakage; the safer direction is a shared pre-preview lifecycle or a carefully extended message_sending contract with streaming metadata and cleanup semantics.
