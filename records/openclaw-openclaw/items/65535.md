## Summary

Keep open. Current main still has no blocking hook or documented lifecycle contract before Telegram preview streaming sends/edits visible content, and the related open items cover narrower final-hook bypasses or broader outbound policy umbrellas rather than superseding this streaming-edit design gap.

## What This Changes

The issue requests a blocking plugin interception point for streamed Telegram outbound content, either through a streaming-edit hook, expanded `message_sending` semantics, or per-chat streaming overrides.

## Best Possible Solution

Define a shared, documented blocking pre-preview-delivery contract that covers preview creation, partial updates, retained previews, and final edits while preserving Telegram's intentional one-visible-preview behavior.

## Reproduction Assessment

Yes, at source level. Telegram preview streaming can call `stream.update(...)` and direct `editMessageText`/preview-finalization paths without the normal hook-backed `deliverReplies`/`runMessageSending` path.

## Solution Assessment

Unclear. A final-only `message_edit_sending` hook or narrow `message_sending` patch would not cover partial preview leakage; the safer direction is a shared pre-preview lifecycle or carefully extended `message_sending` contract with streaming metadata and cleanup semantics.
