## Summary

Keep open. Current main still lacks a blocking hook or documented lifecycle contract for Telegram streamed preview updates and final in-place preview edits, so the privacy-control gap described here remains real even though related shared reply-dispatch hook work has landed.

## What This Changes

Request to add a blocking interception point for streamed outbound message edits, especially Telegram partial and final preview delivery.

## Best Possible Solution

Define a shared, documented blocking pre-preview-delivery contract that covers preview creates, partial updates, retained previews, and final edits, then coordinate the final-message subset with https://github.com/openclaw/openclaw/issues/61550.

## Reproduction Assessment

Yes, at source level. With Telegram preview streaming enabled, partial updates can reach `answerLane.stream.update(...)` and final text can return `preview-finalized` without the normal hook-backed `deliverReplies`/`runMessageSending` path.

## Solution Assessment

Unclear. The requested directions are valid, but a final-only `message_edit_sending` hook would not cover partial preview leakage; the safer maintainable shape is an explicit shared pre-preview-delivery lifecycle or a carefully extended `message_sending` contract with streaming metadata.
