## Summary

Keep open. Current main still lacks a blocking plugin interception point for streamed preview updates and final in-place preview edits; normal outbound sends can run `message_sending`, but Telegram preview paths still update visible messages through draft-stream and preview-edit code directly. Related work such as #38340 and #62676 was useful but closed unmerged, and #74077 is an adjacent streaming-control request rather than a substitute for content filtering.

## What This Changes

The issue requests a blocking plugin hook or equivalent scoped streaming control so streamed Telegram message edits/final content can be inspected, rewritten, or canceled before sensitive content becomes visible.

## Best Possible Solution

Define a shared, documented blocking pre-delivery contract for all user-visible preview creates, preview updates, and final preview edits, with clear streaming metadata and cancellation/cleanup semantics. Wire Telegram first at the draft-stream update and preview-finalization seams, then extend the same contract through the generic draft-preview finalizer and other preview-capable channels; treat per-chat streaming disable as a separate mitigation feature.

## Reproduction Assessment

Yes. A high-confidence reproduction path is a Telegram plugin registering `message_sending`, `channels.telegram.streaming.mode="partial"`, and a streamed reply containing a sensitive token; current code routes partials through `answerLane.stream.update(...)` and final preview edits through `editMessageTelegram(...)`, not through `runMessageSending`.

## Solution Assessment

Unclear. The issue's proposed directions are valid, but a final-only `message_edit_sending` hook is not enough for partial-stream leakage; the safer maintainable solution is a shared pre-preview-delivery hook or extension of `message_sending` that covers every visible streaming edit with explicit metadata.

## Risks / Open Questions

- The public API shape is unresolved: cancellation cleanup, rewrite semantics, fail-open/fail-closed behavior, and metadata must be stable for third-party plugins.
- A final-only hook would not fully solve the sensitive-data use case because partial previews can expose content before finalization.
- The issue may span Telegram, Discord, Slack, and generic draft-preview finalizers, so a channel-only fix could leave the same bypass on another streaming surface.
