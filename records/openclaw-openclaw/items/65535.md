## Summary

Keep open. Current main still lacks a blocking pre-preview/streaming-edit hook for Telegram visible preview updates; https://github.com/openclaw/openclaw/issues/61550 covers the narrower final `message_sending` bypass, while this issue still owns the broader streaming-edit lifecycle and privacy-control design gap.

## What This Changes

The issue requests a blocking hook or equivalent mechanism so plugins can inspect, cancel, or rewrite Telegram streaming edit/final content before it becomes visible.

## Best Possible Solution

Define a shared, documented blocking pre-preview-delivery contract that covers preview creates, partial updates, retained previews, and final edits, while preserving Telegram’s one-visible-preview behavior and coordinating the narrow final subset with https://github.com/openclaw/openclaw/issues/61550.

## Reproduction Assessment

Yes, at source level. Telegram preview streaming can call `stream.update(...)` and direct `editMessageText`/finalization paths without the normal hook-backed `deliverReplies`/`runMessageSending` path.

## Solution Assessment

Unclear. A final-only `message_edit_sending` hook or narrow `message_sending` fix would not cover partial preview leakage; the safer design is a shared pre-preview lifecycle or a carefully extended `message_sending` contract with streaming metadata and cleanup semantics.
