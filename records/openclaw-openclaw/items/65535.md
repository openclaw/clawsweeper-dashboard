## Label Justifications

- `P2`: This is a real plugin lifecycle and Telegram streaming guardrail gap with bounded channel/API blast radius, but it needs design work rather than emergency runtime repair.
- `impact:security`: Outbound DLP, approval, PII filtering, and validation plugins can miss sensitive streamed Telegram content before they get a blocking decision point.

## Summary

Keep open. Current main and the latest release still lack a blocking hook or documented lifecycle contract before Telegram streaming preview sends/edits, and https://github.com/openclaw/openclaw/issues/61550 tracks only the narrower final `message_sending` bypass rather than this broader streaming-edit design gap.

## What This Changes

The issue requests a blocking lifecycle hook or per-chat streaming override so plugins can inspect, rewrite, or cancel Telegram streamed edits before visible content is shown.

## Best Possible Solution

Define a shared, documented blocking pre-preview delivery contract that covers preview creation, partial updates, retained previews, final edits, and cleanup while preserving Telegram's one-visible-preview behavior.

## Reproduction Assessment

Yes, at source level. Telegram preview streaming can call `stream.update(...)` and direct `editMessageText`/preview-finalization paths without the normal hook-backed `deliverReplies`/`runMessageSending` path.

## Solution Assessment

Unclear. The requested direction is valid, but a final-only hook would not cover partial preview leakage; the maintainable fix is a shared pre-preview lifecycle or carefully extended `message_sending` contract with streaming metadata and cleanup semantics.
