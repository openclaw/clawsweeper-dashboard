## Summary

Keep this open: current main still recreates a missing Codex app-server bound thread and immediately retries without recovery-level unsubscribe cleanup, while nearby app-server run paths already treat `thread/unsubscribe` as the cleanup contract.

## What This Changes

The issue asks OpenClaw to release the replacement Codex app-server thread if missing-thread recovery spawns a new bound thread and the retry rejects.

## Best Possible Solution

Add best-effort `thread/unsubscribe` cleanup for bound app-server retry failures, with focused regression coverage, while preserving the persistent conversation binding.

## Reproduction Assessment

Yes for source reproduction: a focused mock can make the first `turn/start` throw `thread not found`, `thread/start` return `thread-new`, and the retry reject. I did not establish a live production reproduction.

## Solution Assessment

Yes: reusing the existing best-effort `thread/unsubscribe` pattern in `conversation-binding.ts` is the narrowest maintainable fix. The cleanup should not clear the persisted binding except on already-defined invalid-binding paths.
