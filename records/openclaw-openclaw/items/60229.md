## Summary

Keep open: this PR targets a source-reproducible Telegram collect-mode bug, but it still lacks inspectable live Telegram proof and introduces correctness/test blockers in busy flush completion, text-fragment ordering, and sequential-key expectations.

## What This Changes

The PR adds an opt-in busy-coalescing mode to the shared inbound debouncer, enables it for Telegram, and changes ordinary Telegram sequentialization keys to include message IDs.

## Best Possible Solution

Revise the PR so busy-buffered enqueue preserves awaited processing semantics, Telegram only relaxes ordering after ordered preprocessing is safe, focused tests pass, and redacted live Telegram proof shows the coalesced queued turn.

## Reproduction Assessment

Yes, source-reproducible. Current main shows chat/topic sequentialization plus same-key debouncer chaining before active-run collect routing, matching the PR body's rapid Telegram burst scenario.

## Solution Assessment

No. The direction is plausible, but the current patch returns before the coalesced flush completes and removes same-chat ordering too broadly for text-fragment-capable messages.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Await the coalesced busy flush:** `src/auto-reply/inbound-debounce.ts:201`
  - body: In the coalesceWhileBusy branch, enqueue schedules the buffered flush with `void enqueueKeyTask(...)` and returns before `onFlush` runs. Telegram treats awaited enqueue completion as update handler completion, so a restart after the update is checkpointed can drop a message that was buffered but never processed.
  - confidence: 0.86
- **[P2] Keep fragment-capable messages ordered:** `extensions/telegram/src/sequential-key.ts:107-109`
  - body: Adding `:msg:${messageId}` to every ordinary sequential key lets same-chat text messages run concurrently, but text-fragment assembly uses shared per-chat/topic/sender state after awaited setup work. Adjacent long-paste fragments can race before the first message installs the buffer and then dispatch separately or out of order.
  - confidence: 0.82
- **[P2] Update the sequential-key expectations:** `extensions/telegram/src/sequential-key.ts:107-109`
  - body: The PR changes ordinary keys to include `:msg:1`, but `extensions/telegram/src/sequential-key.test.ts` still expects bare chat/topic keys. The Telegram extension CI already fails 11 sequential-key cases on the PR head until the tests are updated or the source change is narrowed.
  - confidence: 0.9
