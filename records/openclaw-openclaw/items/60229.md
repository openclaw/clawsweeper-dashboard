## Summary

Keep open: the Telegram collect-mode bug is real and the idea is useful, but this branch is not merge-ready because source review shows a fire-and-forget busy flush can acknowledge Telegram updates before delivery and the per-message sequential key can break ordered text-fragment assembly.

## What This Changes

The PR changes Telegram inbound processing so follow-up messages sent during an active agent run can be buffered and flushed together instead of being serialized one at a time.

## Best Possible Solution

Land a revised Telegram coalescing fix that keeps update acknowledgement tied to actual coalesced flush completion, preserves ordered lanes for shared Telegram buffers, updates focused tests, and includes redacted live Telegram proof.

## Reproduction Assessment

Yes by source inspection, not by a live run in this read-only pass: current main sequentializes ordinary Telegram updates by chat/topic and the inbound debouncer serializes same-key active flushes, which prevents collect-mode follow-ups from reaching queue collection while the agent is still active.

## Solution Assessment

No as submitted. The coalescing direction is plausible, but the best fix must decouple active agent runs from Telegram update serialization without returning before delivery completion or removing ordering from shared buffer paths.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Await busy coalesced flushes before acking updates:** `src/auto-reply/inbound-debounce.ts:207`
  - body: The new busy-buffer path schedules the follow-up flush with `void enqueueKeyTask(...)` and then returns from `enqueue`. Telegram treats `await inboundDebouncer.enqueue(...)` as the dispatch completion boundary, so the update can be finished and persisted before `runFlush(pending)` actually delivers it; a restart in that window skips the queued message. Return or await the queued task, or otherwise defer update completion until the coalesced flush settles.
  - confidence: 0.9
- **[P2] Keep fragment assembly on an ordered lane:** `extensions/telegram/src/sequential-key.ts:63`
  - body: Appending `:msg:${messageId}` makes ordinary same-chat/topic updates run concurrently, but the handler’s long-text fragment combiner mutates `textFragmentBuffer` and decides whether to append based on the previous message id. A later Telegram continuation can reach that block before the first part creates the buffer, causing split or out-of-order turns. Preserve an ordered chat/topic lane for these shared buffer paths, or add an explicit per-buffer queue before making regular messages concurrent.
  - confidence: 0.84
