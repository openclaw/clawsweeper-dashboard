## Summary

Keep open. The PR targets a real Telegram queue/coalescing gap on current main, but the proposed patch still changes enqueue completion semantics, relaxes Telegram ordering before fragment assembly is safe, leaves focused tests stale, and lacks the required changelog entry.

## What This Changes

The PR adds opt-in busy coalescing to `createInboundDebouncer`, enables it for Telegram inbound messages, and appends Telegram message IDs to ordinary Telegram sequentialization keys.

## Best Possible Solution

Revise this PR or a narrow replacement so busy coalescing preserves awaited enqueue semantics, Telegram only relaxes ordering after fragment preprocessing is protected, and regression tests plus changelog cover the behavior.

## Reproduction Assessment

Yes. Current main shows chat/topic sequentialization plus same-key debouncer chaining that can make rapid Telegram follow-ups reach queue policy only after the active run is gone; the PR body also gives a live DM burst reproduction path.

## Solution Assessment

No. The direction is maintainable, but this patch should not merge until buffered busy enqueues await their actual flush and fragment-capable Telegram messages remain ordered or move through an ordered assembly lane first.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Await the coalesced busy flush:** `src/auto-reply/inbound-debounce.ts:207`
  - body: In the `coalesceWhileBusy` branch, `enqueue` schedules the follow-up flush with `void enqueueKeyTask(...)` and returns before `runFlush(pending)` handles the buffered item. Telegram awaits `inboundDebouncer.enqueue(...)` as part of update handling, so the update can be treated as handled before the message reaches `onFlush`; a restart in that window can skip it.
  - confidence: 0.86
- **[P2] Keep fragment-capable messages ordered:** `extensions/telegram/src/sequential-key.ts:65`
  - body: Appending `:msg:${messageId}` to every ordinary Telegram sequential key lets same-chat text messages run concurrently, but text-fragment assembly mutates a shared per-chat/thread/sender buffer. Split long pastes can now race before the first part installs the buffer, so fragments may dispatch separately or out of order.
  - confidence: 0.82
- **[P2] Update the sequential-key regression expectations:** `extensions/telegram/src/sequential-key.ts:65`
  - body: The changed ordinary key now returns values like `telegram:123:msg:1`, while `extensions/telegram/src/sequential-key.test.ts` still expects bare chat/topic keys for the default mock message. CI will fail unless the expectations are updated or the per-message key change is narrowed.
  - confidence: 0.78
- **[P3] Add the required changelog entry:** `extensions/telegram/src/bot-handlers.runtime.ts:202`
  - body: This is a user-facing Telegram feature change, but the PR does not touch `CHANGELOG.md`. Repo policy requires a single-line active-version changelog entry for user-facing `feat`, `fix`, or `perf` behavior changes.
  - confidence: 0.9
