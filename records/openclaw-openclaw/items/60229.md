## Summary

Keep open. The PR targets a source-reproducible Telegram collect-mode gap, but the current branch still has blocking correctness/test issues and lacks inspectable live Telegram proof.

## What This Changes

The PR adds a `coalesceWhileBusy` inbound debounce option, enables it for Telegram, and changes ordinary Telegram sequentialization keys to include `message_id`.

## Best Possible Solution

Revise the PR to preserve enqueue await semantics, keep fragment-capable messages ordered or move the bypass after fragment preprocessing, update focused tests, and attach redacted live Telegram proof.

## Reproduction Assessment

Yes, source-reproducible from current main: Telegram awaits the inbound debouncer, current sequentialization is chat/topic scoped, and same-key debouncer work is chained before queue routing. I did not run a live Telegram session.

## Solution Assessment

No. The direction is plausible, but this patch returns before the coalesced flush completes and broadly removes ordering around shared Telegram text-fragment state; the safer fix must preserve ack semantics and narrow the concurrency bypass.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Await coalesced busy tasks before returning:** `src/auto-reply/inbound-debounce.ts:209`
  - body: In the `coalesceWhileBusy` branch, `enqueue` schedules the follow-up flush with `void enqueueKeyTask(...)` and returns before `onFlush` observes the item. Telegram marks updates complete after the handler finishes, so a restart after that completion can drop the buffered message.
  - confidence: 0.86
- **[P2] Keep fragment-capable messages ordered:** `extensions/telegram/src/sequential-key.ts:65`
  - body: Adding `:msg:${messageId}` to every ordinary Telegram key lets same-chat text messages enter the handler concurrently, but text-fragment assembly uses shared `textFragmentBuffer` keyed by chat/topic/sender. Adjacent long-paste fragments can race before the first entry exists and be emitted separately or out of order.
  - confidence: 0.82
- **[P2] Update the sequential-key expectations:** `extensions/telegram/src/sequential-key.ts:65`
  - body: The PR changes ordinary keys to include `:msg:1`, but existing tests still expect bare chat/topic keys. CI fails 11 cases in `sequential-key.test.ts` until the expectations or source behavior are updated.
  - confidence: 0.9
