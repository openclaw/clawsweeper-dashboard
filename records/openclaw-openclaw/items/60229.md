## Summary

Keep open: the Telegram collect-mode problem is real, and the PR direction has useful discussion support, but this branch still lacks inspectable real Telegram proof and has blocking correctness/test findings.

## What This Changes

The PR adds a `coalesceWhileBusy` option to the shared inbound debouncer, enables it for Telegram, and changes ordinary Telegram sequential keys to include `message_id`.

## Best Possible Solution

Land a revised Telegram fix that lets same-chat busy followups reach collect mode while preserving update checkpoint durability, text-fragment ordering, and current sequential-key test coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main serializes ordinary Telegram updates by chat/topic and then chains same-key debouncer flushes, so rapid same-chat followups can reach queue routing only after the active run is no longer active.

## Solution Assessment

No. The PR is aimed at the right bug, but the current patch returns before a coalesced busy flush completes, weakens text-fragment ordering, and leaves sequential-key tests failing.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Await the coalesced busy flush:** `src/auto-reply/inbound-debounce.ts:207-213`
  - body: In the `coalesceWhileBusy` branch, `enqueue` schedules the follow-up flush with `void enqueueKeyTask(...)` and returns before `runFlush(pending)` runs. Telegram awaits `inboundDebouncer.enqueue(...)` before update tracking finishes, so a crash or restart in that window can checkpoint a follow-up update before the buffered message is actually flushed.
  - confidence: 0.88
- **[P2] Keep fragment-capable messages ordered:** `extensions/telegram/src/sequential-key.ts:65`
  - body: Adding `:msg:${messageId}` to every ordinary Telegram sequential key lets same-chat/topic text messages enter `processInboundMessage` concurrently. The long-text fragment combiner mutates shared chat/thread/sender state and appends adjacent message IDs, so split Telegram long-paste parts can race and be emitted separately or out of order.
  - confidence: 0.84
- **[P2] Update the sequential-key expectations:** `extensions/telegram/src/sequential-key.ts:65`
  - body: The source change makes ordinary mock messages return keys with `:msg:1`, but `sequential-key.test.ts` still expects bare chat/topic keys and the PR CI log shows those failures. Update the expectations and add the missing-message-id fallback case, or narrow the source change so the current contract still holds.
  - confidence: 0.93
