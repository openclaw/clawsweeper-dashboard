## Summary

Keep open. The PR targets a real Telegram collect-mode gap, but it is not ready to merge because inspectable real Telegram proof is missing and the patch introduces checkpoint, ordering, and test regressions.

## What This Changes

The PR adds a `coalesceWhileBusy` inbound debouncer option, enables it for Telegram, and changes ordinary Telegram sequential keys to include `message_id`.

## Best Possible Solution

Land a rebased Telegram fix that lets busy same-chat follow-ups reach collect mode while preserving awaited update checkpointing, same-topic fragment ordering, tests, and redacted real Telegram proof.

## Reproduction Assessment

Yes, source-reproducible: current main serializes ordinary Telegram updates by chat/topic and awaits same-key debounce chains, so rapid same-chat follow-ups can reach queue routing only after the active run is no longer active. I did not run a live Telegram session in this read-only review.

## Solution Assessment

No. The PR aims at the right bug, but this patch returns before coalesced busy flush completion, weakens fragment-capable message ordering, and leaves the sequential-key test contract failing.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Await the coalesced busy flush:** `src/auto-reply/inbound-debounce.ts:207-213`
  - body: The new `coalesceWhileBusy` branch schedules `enqueueKeyTask` with `void` and returns immediately. Telegram marks an update complete after the handler returns, so a restart before the queued flush runs can checkpoint a follow-up update that never reached `onFlush`.
  - confidence: 0.88
- **[P2] Preserve fragment-capable message ordering:** `extensions/telegram/src/sequential-key.ts:65`
  - body: Appending `:msg:${messageId}` to ordinary sequential keys lets same-chat/topic messages enter the handler concurrently. The text-fragment combiner mutates a shared chat/thread/sender buffer and depends on adjacent message IDs, so split long-paste parts can race and be emitted separately or out of order.
  - confidence: 0.84
- **[P2] Update the sequential-key test contract:** `extensions/telegram/src/sequential-key.ts:65`
  - body: The source change makes ordinary mock messages return keys with `:msg:1`, but the existing sequential-key test matrix still expects bare chat/topic keys and CI fails 11 assertions. Either update the expectations and fallback coverage, or narrow the key change to preserve the current contract.
  - confidence: 0.93
