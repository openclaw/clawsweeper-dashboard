## Summary

Keep open: the PR targets a valid Telegram collect-mode bug, but the current patch introduces update-completion and same-chat ordering regressions and still lacks attached real Telegram proof.

## What This Changes

The PR adds a `coalesceWhileBusy` inbound debouncer option, enables it for Telegram, and changes ordinary Telegram sequentialization keys to include `message_id`.

## Best Possible Solution

Land a rebased Telegram fix that lets busy same-chat followups reach collect mode while awaiting coalesced flushes, preserving fragment-capable ordering, updating focused tests, and attaching redacted real Telegram proof.

## Reproduction Assessment

Yes, at source level: current main serializes ordinary Telegram updates by chat/topic and awaits active same-key debouncer chains, so rapid followups can reach active-run routing only after the prior run finishes. I did not run a live Telegram repro in this read-only review.

## Solution Assessment

No. The PR targets the right bug, but the current implementation can complete updates before busy-buffer flushes run and weakens ordering needed by Telegram text-fragment assembly.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Await busy-buffer flushes before completing updates:** `src/auto-reply/inbound-debounce.ts:207-213`
  - body: The new `coalesceWhileBusy` path schedules `enqueueKeyTask` with `void` and returns immediately. Telegram treats the awaited handler return as update completion, so a restart in that window can checkpoint a follow-up update before it has reached `onFlush`.
  - confidence: 0.88
- **[P2] Preserve chat ordering for fragment assembly:** `extensions/telegram/src/sequential-key.ts:65`
  - body: Appending `:msg:${messageId}` to every ordinary sequential key lets same-chat/topic messages enter the handler concurrently. The text-fragment buffer mutates shared chat/thread/sender state and depends on adjacent message IDs, so split long-paste parts can be emitted separately or out of order.
  - confidence: 0.84
- **[P2] Update sequential-key expectations:** `extensions/telegram/src/sequential-key.ts:65`
  - body: The source change makes ordinary mock messages return keys with `:msg:1`, but the existing sequential-key test matrix still expects bare chat/topic keys. Update the expectations and missing-message-id fallback coverage, or narrow the key change to preserve the current contract.
  - confidence: 0.93
