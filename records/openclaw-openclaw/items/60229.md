## Summary

Keep open: the PR targets a valid Telegram collect-mode bug, but the current branch still has merge-blocking correctness issues, failing Telegram sequential-key tests, and no inspectable real Telegram proof artifact.

## What This Changes

The PR adds an opt-in busy-coalescing mode to the inbound debouncer, enables it for Telegram, and changes ordinary Telegram sequentialization keys to include message IDs.

## Best Possible Solution

Revise this PR so busy-buffered enqueue preserves await semantics, Telegram relaxes ordering only after fragment preprocessing is safe, focused tests pass, and redacted live Telegram proof demonstrates the coalesced queued turn.

## Reproduction Assessment

Yes, source-reproducible. Current main shows chat/topic sequentialization plus same-key debouncer chaining before collect-mode active-run routing, and the PR body describes the live Telegram DM burst scenario.

## Solution Assessment

No. The direction is reasonable, but this patch returns before the buffered busy flush completes and broadly removes ordering for text-fragment-capable messages.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Await the coalesced busy flush:** `src/auto-reply/inbound-debounce.ts:201`
  - body: In the new `coalesceWhileBusy` branch, `enqueue` schedules the buffered flush with `void enqueueKeyTask(...)` and returns. Telegram treats awaited enqueue completion as update handler completion, so a restart after the update is checkpointed can drop a message that never reached `onFlush`.
  - confidence: 0.86
- **[P2] Keep fragment-capable messages ordered:** `extensions/telegram/src/sequential-key.ts:107`
  - body: Adding `:msg:${messageId}` to every ordinary key lets same-chat text messages run concurrently, but text-fragment assembly uses a shared per-chat/topic/sender buffer. Adjacent long-paste fragments can race before the first message installs the buffer and dispatch as separate or out-of-order turns.
  - confidence: 0.82
- **[P2] Update the sequential-key expectations:** `extensions/telegram/src/sequential-key.ts:107`
  - body: The PR changes ordinary keys to include `:msg:1`, but `extensions/telegram/src/sequential-key.test.ts` still expects bare chat/topic keys. The Telegram extension CI already fails 11 sequential-key cases on the PR head until the tests are updated or the source change is narrowed.
  - confidence: 0.9
