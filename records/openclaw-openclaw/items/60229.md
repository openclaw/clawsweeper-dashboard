## What This Changes

The PR adds a coalesceWhileBusy mode to the inbound debouncer, enables it for Telegram, and appends message_id to ordinary Telegram sequentialization keys.

## Best Possible Solution

## Reproduction Assessment

Yes, at source level: current main serializes ordinary Telegram updates by chat/topic and then awaits active same-key debouncer chains, so rapid follow-ups can reach active-run routing only after the prior run finishes. I did not run a live Telegram repro in this read-only review.

## Solution Assessment

No. The PR targets the right bug, but this implementation can complete updates before busy-buffer flushes run and weakens ordering needed by Telegram text-fragment assembly.
