## Summary

Keep open: current main and the latest release still use chat/topic Telegram lanes plus same-key inbound debounce chains, so the reported collect-mode busy-followup problem remains. The PR is not merge-ready because it can acknowledge buffered updates before delivery, can break Telegram text-fragment ordering, is conflicted with current main, and still lacks attached real Telegram proof.

## What This Changes

Adds a busy-run coalescing option to inbound debouncing, enables it for Telegram, and changes ordinary Telegram sequentialization keys to include message IDs.

## Best Possible Solution

## Solution Assessment

Overall confidence: 0.9

Full review comments:

- **[P1] Await coalesced busy flushes before completing enqueue:** `src/auto-reply/inbound-debounce.ts:207-213`
  - body: The new branch schedules the busy-buffer flush with `void enqueueKeyTask(...)` and returns immediately. Telegram awaits `inboundDebouncer.enqueue(...)` before the update tracker marks middleware complete, so this can persist update completion before `runFlush(pending)` reaches the agent; a restart in that window skips messages that were never processed.
  - confidence: 0.9
- **[P2] Preserve ordered lanes for Telegram text fragments:** `extensions/telegram/src/sequential-key.ts:63-65`
  - body: Adding `:msg:${messageId}` moves ordinary same-chat messages onto separate grammY lanes. The text-fragment buffer is shared by chat/thread/sender and assumes adjacent message IDs arrive in order, so concurrent long-paste parts can miss the buffer and be delivered as separate or out-of-order turns.
  - confidence: 0.86
- **[P2] Update sequential-key coverage for per-message keys:** `extensions/telegram/src/sequential-key.ts:63-65`
  - body: The existing sequential-key tests still expect ordinary keys such as `telegram:123`, while the PR now returns `telegram:123:msg:1` for the mock default. The focused Telegram CI lane failed, so update expectations and add the missing-message_id fallback case.
