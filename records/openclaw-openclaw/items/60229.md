## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Await busy coalesced flushes before acking updates:** `src/auto-reply/inbound-debounce.ts:207`
  - body: The new busy-buffer path schedules the follow-up flush with `void enqueueKeyTask(...)` and then returns from `enqueue`. Telegram treats `await inboundDebouncer.enqueue(...)` as the dispatch completion boundary, so the update can be finished and persisted before `runFlush(pending)` actually delivers it; a restart in that window skips the queued message. Return or await the queued task, or otherwise defer update completion until the coalesced flush settles.
  - confidence: 0.9
- **[P2] Keep fragment assembly on an ordered lane:** `extensions/telegram/src/sequential-key.ts:63`
  - body: Appending `:msg:${messageId}` makes ordinary same-chat/topic updates run concurrently, but the handler’s long-text fragment combiner mutates `textFragmentBuffer` and decides whether to append based on the previous message id. A later Telegram continuation can reach that block before the first part creates the buffer, causing split or out-of-order turns. Preserve an ordered chat/topic lane for these shared buffer paths, or add an explicit per-buffer queue before making regular messages concurrent.
  - confidence: 0.84
