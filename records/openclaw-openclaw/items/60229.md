## Summary

Keep open. The PR targets a real Telegram collect-mode gap that current main still does not fully solve, but the proposed patch is not merge-ready because the busy coalescing promise can complete before the buffered flush runs, and the broad per-message Telegram sequential key can break existing ordered text-fragment assembly.

## What This Changes

The PR adds an opt-in busy coalescing mode to the shared inbound debouncer, enables it for Telegram, and changes ordinary Telegram sequentialization keys to include the message ID.

## Best Possible Solution

Keep the PR open, preserve the useful busy-message coalescing direction, and revise it so the shared debouncer keeps await/completion semantics while Telegram only relaxes sequentialization for messages that do not need ordered shared-buffer preprocessing.

## Reproduction Assessment

Yes for review purposes: current main shows the chat/topic sequentializer plus same-key debouncer chain that prevents queue collect from seeing rapid Telegram follow-ups during an active run. The PR’s live Telegram DM notes also give a plausible manual reproduction path, but I did not run it because this task is read-only.

## Solution Assessment

No, the current PR is not the best safe solution as written. The maintainable path is to keep busy coalescing opt-in and additive, but make every coalesced enqueue await its actual flush and avoid removing same-chat ordering for fragment-capable Telegram messages.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Await the coalesced busy flush:** `src/auto-reply/inbound-debounce.ts:201`
  - body: In the coalesce-while-busy branch, enqueue schedules the follow-up flush and returns immediately. Telegram awaits enqueue before the middleware returns, so the update can be treated as handled while its buffered message has not reached onFlush yet; a restart or process exit in that window can lose the message. Store the scheduled busy flush promise and await it for every item coalesced into that buffer.
  - confidence: 0.86
- **[P2] Keep fragment candidates on an ordered Telegram lane:** `extensions/telegram/src/sequential-key.ts:107`
  - body: Appending message_id to every ordinary Telegram sequential key lets adjacent same-chat messages run concurrently. The text-fragment assembler uses a shared per-chat/topic/sender buffer after async authorization work, so split paste parts can reach that map out of order or before the first part installs the buffer and then dispatch separately. Keep fragment-capable messages serialized, or move fragment assembly behind its own ordered lane before using per-message keys.
  - confidence: 0.82
- **[P2] Update the sequential-key regression expectations:** `extensions/telegram/src/sequential-key.ts:107`
  - body: The changed default key returns values like telegram:123:msg:1, but the existing sequential-key tests still expect bare chat/topic keys. The PR will fail that test file unless the expectations are updated or the broad per-message key change is narrowed.
  - confidence: 0.78
