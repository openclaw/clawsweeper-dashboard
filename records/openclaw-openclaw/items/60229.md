## Summary

Keep open: the PR targets a real source-reproducible Telegram collect-mode bug, and the recent discussion supports the direction, but the branch still lacks inspectable real Telegram proof and has correctness/CI blockers in the changed scheduling and keying behavior.

## What This Changes

The PR adds a `coalesceWhileBusy` path to the shared inbound debouncer, enables it for Telegram, and appends message IDs to ordinary Telegram sequentialization keys.

## Best Possible Solution

Revise the patch to preserve enqueue completion semantics and text-fragment ordering while still enabling busy-time coalescing, update focused tests, and attach redacted live Telegram proof.

## Reproduction Assessment

Yes, source-reproducible. Current main documents collect-mode batching, but Telegram still uses chat/topic sequentialization plus same-key debouncer chaining before active-run queue routing; I did not run a live Telegram session.

## Solution Assessment

No. The direction is plausible, but this PR returns before the coalesced busy flush completes and relaxes ordering before fragment-safe preprocessing; the narrower maintainable fix should preserve those contracts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Await the coalesced busy flush:** `src/auto-reply/inbound-debounce.ts:213`
  - body: In the new `coalesceWhileBusy` branch, `enqueue` schedules `enqueueKeyTask` with `void` and returns before `onFlush` sees the buffered item. Telegram awaits enqueue before the update tracker finishes, so a restart after that checkpoint can drop a queued follow-up message.
  - confidence: 0.86
- **[P2] Keep fragment-capable messages ordered:** `extensions/telegram/src/sequential-key.ts:65`
  - body: Appending `:msg:${messageId}` to every ordinary key lets same-chat text messages run concurrently, but text-fragment assembly uses shared chat/topic/sender state. Adjacent long-paste fragments can race before the first message installs the buffer and then emit separate or out-of-order turns.
  - confidence: 0.82
- **[P2] Update the sequential-key expectations:** `extensions/telegram/src/sequential-key.ts:65`
  - body: The PR changes ordinary sequential keys to include `:msg:1`, but the existing sequential-key tests still expect bare chat/topic keys. CI on the PR head fails 11 cases until the tests are updated or the source change is narrowed.
  - confidence: 0.9
