## Summary

Keep open. Current main still has the reported Telegram same-chat serialization path, and this PR is a useful direction, but the current patch introduces blocking await/completion and Telegram fragment-ordering regressions and still needs focused tests plus the required changelog entry.

## What This Changes

The PR adds an opt-in busy coalescing mode to the shared inbound debouncer, enables it for Telegram, and appends Telegram message IDs to ordinary sequentialization keys.

## Best Possible Solution

Revise this PR or a narrow replacement so busy coalescing preserves enqueue await semantics, Telegram only relaxes ordering after fragment preprocessing is safe, and tests plus changelog cover the behavior.

## Reproduction Assessment

Yes. Current main shows the chat/topic sequentializer plus same-key debouncer chain that prevents rapid Telegram follow-ups from reaching collect mode during an active run; the PR body also gives a live DM reproduction path.

## Solution Assessment

No. The direction is maintainable, but the current patch should not merge until buffered busy enqueues await their actual flush and Telegram fragment-capable messages remain ordered or are moved behind their own ordered assembly lane.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Await the coalesced busy flush:** `src/auto-reply/inbound-debounce.ts:201`
  - body: In the coalesceWhileBusy branch, enqueue schedules the follow-up flush with void enqueueKeyTask and returns before runFlush handles the buffered item. Telegram treats awaited enqueue completion as handler completion, so a restart after the update is marked handled can skip a message that never reached onFlush.
  - confidence: 0.86
- **[P2] Keep fragment-capable messages ordered:** `extensions/telegram/src/sequential-key.ts:107`
  - body: Adding :msg:{messageId} to every ordinary Telegram sequential key lets same-chat text messages run concurrently, but the text-fragment combiner mutates a shared per-chat/topic/sender buffer. Split long pastes can now race before the first part installs the buffer and dispatch separately or out of order.
  - confidence: 0.82
- **[P2] Update the sequential-key regression expectations:** `extensions/telegram/src/sequential-key.ts:107`
  - body: The changed default key now returns values like telegram:123:msg:1, while the existing sequential-key tests still expect bare chat/topic keys. CI will fail unless the expectations are updated or the per-message key change is narrowed.
  - confidence: 0.78
- **[P3] Add the required changelog entry:** `extensions/telegram/src/bot-handlers.runtime.ts:200`
  - body: This is a user-facing Telegram feat change, but the PR does not touch CHANGELOG.md. Repo policy requires a single-line active-version changelog entry for user-facing feat/fix/perf behavior changes.
  - confidence: 0.9
