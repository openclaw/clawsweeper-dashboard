## Summary

Keep open: the reported Telegram collect-mode bug is source-reproducible and worth fixing, but this PR still has blocking correctness/test findings and lacks inspectable real Telegram proof.

## What This Changes

The branch adds an opt-in busy-buffer path to the shared inbound debouncer, enables it for Telegram, and changes ordinary Telegram sequentialization keys to include message IDs.

## Best Possible Solution

Land a revised Telegram fix that lets busy followups reach collect mode while preserving update checkpoint waiting and fragment-capable message ordering, with focused tests plus real Telegram proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main uses chat/topic Telegram sequentialization plus same-key debouncer chaining before active-run queue routing, so rapid same-chat followups cannot reach collect handling while the active run is still active; I did not run a live Telegram session in this read-only review.

## Solution Assessment

No. The direction is plausible, but this patch returns before the coalesced busy flush completes, weakens long-text fragment ordering, and leaves sequential-key expectations failing; the safer fix should await the busy flush and preserve a serialized lane for fragment-capable text or move that assembly before parallel dispatch.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Await the coalesced busy flush:** `src/auto-reply/inbound-debounce.ts:207-213`
  - body: In the `coalesceWhileBusy` branch, `enqueue` schedules the follow-up flush with `void enqueueKeyTask(...)` and returns immediately. Telegram awaits `enqueue` before update completion, so the update can be checkpointed before `runFlush(pending)` processes the message, leaving a crash/restart window where a queued follow-up is skipped.
  - confidence: 0.88
- **[P2] Keep fragment-capable messages ordered:** `extensions/telegram/src/sequential-key.ts:65`
  - body: Appending `:msg:${messageId}` to every ordinary Telegram sequential key lets same-chat text messages enter `processInboundMessage` concurrently. The long-text fragment combiner mutates a shared chat/thread/sender buffer, so split long-paste parts can race and be emitted separately or out of order.
  - confidence: 0.84
- **[P2] Update key expectations with the new format:** `extensions/telegram/src/sequential-key.ts:65`
  - body: The source change makes ordinary mock messages return keys with `:msg:1`, but `sequential-key.test.ts` still expects bare chat/topic keys and the PR check rollup shows Telegram check failures. Update the expectations and add a missing-message-id fallback case, or narrow the source change.
  - confidence: 0.93
