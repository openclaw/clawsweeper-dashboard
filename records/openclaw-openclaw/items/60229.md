## Summary

Keep open: current main still appears source-reproducible for the Telegram collect-mode bug, but this PR is not merge-ready because the patch can acknowledge updates before the buffered flush runs, weakens ordering needed by text-fragment assembly, leaves the sequential-key test contract stale, and lacks attached real Telegram proof.

## What This Changes

The PR adds an opt-in `coalesceWhileBusy` inbound debouncer path, enables it for Telegram, and changes ordinary Telegram sequential keys to include `message_id`.

## Best Possible Solution

Land a rebased Telegram fix that lets same-chat follow-ups reach collect mode while keeping update acknowledgment tied to completed flush work, preserving text-fragment ordering, and updating focused tests plus real Telegram proof.

## Reproduction Assessment

Yes, at source level: current main serializes ordinary Telegram updates by chat/topic and awaits same-key debouncer chains, so rapid follow-ups can reach queue routing only after the active run is no longer active. I did not run a live Telegram repro in this read-only review.

## Solution Assessment

No. The PR targets the right bug, but the current patch can return before the coalesced busy flush completes and broad per-message sequential keys weaken fragment ordering; a safer fix should preserve those contracts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Await the coalesced busy flush:** `src/auto-reply/inbound-debounce.ts:207-213`
  - body: The new `coalesceWhileBusy` branch schedules `enqueueKeyTask` with `void` and returns immediately. Telegram awaits `inboundDebouncer.enqueue(...)` before finishing update tracking, so a restart after handler return but before the queued flush runs can checkpoint a follow-up update that never reached `onFlush`.
  - confidence: 0.88
- **[P2] Preserve fragment-capable message ordering:** `extensions/telegram/src/sequential-key.ts:65`
  - body: Appending `:msg:${messageId}` to ordinary sequential keys lets same-chat/topic messages enter the handler concurrently. The text-fragment combiner mutates a shared chat/thread/sender buffer and depends on adjacent message IDs, so split long-paste parts can race and be emitted separately or out of order.
  - confidence: 0.84
- **[P2] Update the sequential-key expectations:** `extensions/telegram/src/sequential-key.ts:65`
  - body: The source change makes ordinary mock messages return keys with `:msg:1`, but the existing sequential-key test matrix still expects bare chat/topic keys. Update the expectations and fallback coverage, or narrow the key change to preserve the current contract.
  - confidence: 0.93
