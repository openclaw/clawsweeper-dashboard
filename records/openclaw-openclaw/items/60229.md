## Summary

Keep this PR open: it targets a real Telegram collect-mode bug, and a recent comment agrees the direction is useful, but the branch still needs real Telegram proof, a rebase, and fixes for durability, text-fragment ordering, and failing sequential-key expectations before merge.

## What This Changes

The PR adds a `coalesceWhileBusy` inbound debouncer option, enables it for Telegram, and changes ordinary Telegram sequential keys to include `message_id`.

## Best Possible Solution

Land a rebased Telegram fix that lets busy same-chat follow-ups reach collect mode while preserving update checkpoint durability, text-fragment ordering, current sequential-key coverage, and redacted real Telegram proof.

## Reproduction Assessment

Yes, at source level: current main still serializes ordinary Telegram updates by chat/topic and awaits same-key debounce chains, so rapid same-chat follow-ups can reach queue routing only after the active run is no longer active. I did not run a live Telegram repro in this read-only review.

## Solution Assessment

No. The PR aims at the right bug, but this patch returns before coalesced busy flush completion, weakens fragment-capable message ordering, and leaves the sequential-key test contract failing.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Await the coalesced busy flush:** `src/auto-reply/inbound-debounce.ts:207-213`
  - body: The new `coalesceWhileBusy` path schedules `enqueueKeyTask` with `void` and returns immediately. Telegram marks an update complete after the handler returns, so a restart before the queued flush runs can checkpoint a follow-up update that never reached `onFlush`.
  - confidence: 0.88
- **[P2] Preserve fragment-capable message ordering:** `extensions/telegram/src/sequential-key.ts:65`
  - body: Appending `:msg:${messageId}` to every ordinary sequential key lets same-chat/topic messages enter the handler concurrently. The text-fragment combiner mutates a shared chat/thread/sender buffer and depends on adjacent message IDs, so split long-paste parts can race and be emitted separately or out of order.
  - confidence: 0.84
- **[P2] Update the sequential-key expectations:** `extensions/telegram/src/sequential-key.ts:65`
  - body: The source change makes ordinary mock messages return keys with `:msg:1`, but the existing sequential-key test matrix still expects bare chat/topic keys and CI fails 11 tests. Either update the expectations and fallback coverage, or narrow the key change to preserve the current contract.
  - confidence: 0.93
