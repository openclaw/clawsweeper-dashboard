## Summary

Keep open: the PR targets a real Telegram collect-mode bug, and the later supportive comment is consistent with the source-level issue, but the current patch is still not mergeable because it has blocking correctness findings, failing Telegram key tests, no inspectable real Telegram proof artifact, and a conflicting branch.

## What This Changes

The PR adds a shared `coalesceWhileBusy` debounce option, enables it for Telegram inbound handling, and changes ordinary Telegram sequential keys to include `message_id`.

## Best Possible Solution

Land a rebased Telegram fix that lets busy same-chat follow-ups reach collect mode while preserving update checkpoint durability, text-fragment ordering, current sequential-key coverage, and real Telegram proof.

## Reproduction Assessment

Yes. Source inspection of current main shows ordinary Telegram updates are serialized by chat/topic and same-key debouncer flushes are awaited, so rapid same-chat follow-ups can reach queue routing only after the active run is no longer active; I did not run a live Telegram repro in this read-only review.

## Solution Assessment

No. The PR is aimed at the right bug, but the current patch returns before the coalesced flush completes, weakens fragment-capable message ordering, and leaves the sequential-key test contract failing.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Await the coalesced busy flush:** `src/auto-reply/inbound-debounce.ts:207-213`
  - body: The new `coalesceWhileBusy` path schedules `enqueueKeyTask` with `void` and returns immediately. Telegram awaits `inboundDebouncer.enqueue(...)` before the update tracker finishes the update, so a restart after handler return but before the queued flush runs can checkpoint a follow-up update that never reached `onFlush`.
  - confidence: 0.88
- **[P2] Preserve fragment-capable message ordering:** `extensions/telegram/src/sequential-key.ts:65`
  - body: Appending `:msg:${messageId}` to every ordinary sequential key lets same-chat/topic messages enter `processInboundMessage` concurrently. The text-fragment path mutates a shared per-chat/thread/sender buffer and depends on adjacent message IDs, so split long Telegram paste parts can race and be emitted separately or out of order.
  - confidence: 0.84
- **[P2] Update the sequential-key expectations:** `extensions/telegram/src/sequential-key.ts:65`
  - body: The source change makes ordinary mock messages return keys with `:msg:1`, but the existing test matrix still expects bare chat/topic keys and CI is failing in `sequential-key.test.ts`. Update the expectations and add the missing-message-id fallback, or narrow the key change to preserve the current contract.
  - confidence: 0.93
