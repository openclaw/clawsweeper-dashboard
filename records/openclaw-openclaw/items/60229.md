## Summary

Keep open: the Telegram collect-mode problem is source-reproducible, but this PR still has blocking correctness issues, stale sequential-key expectations, and no inspectable after-fix Telegram proof.

## What This Changes

The PR adds busy-time coalescing to the inbound debouncer, enables it for Telegram, and appends Telegram message IDs to ordinary sequentialization keys.

## Best Possible Solution

Revise the PR so busy coalescing preserves enqueue await semantics, relaxes Telegram ordering only after fragment-safe preprocessing, updates focused tests, and includes redacted live Telegram proof.

## Reproduction Assessment

Yes, source-reproducible. Current main has documented collect-mode batching, chat/topic Telegram sequentialization, and same-key debouncer chaining before active-run queue routing; I did not run a live Telegram session.

## Solution Assessment

No. The direction is plausible, but this patch returns before the buffered flush completes and broadly removes ordering from text-fragment-capable messages; the safer fix must preserve ack semantics and fragment ordering.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Await the coalesced busy flush:** `src/auto-reply/inbound-debounce.ts:213`
  - body: In the new `coalesceWhileBusy` branch, `enqueue` schedules `enqueueKeyTask` with `void` and returns before `onFlush` sees the buffered item. Telegram awaits enqueue before the update tracker finishes, so a restart after that checkpoint can drop a queued follow-up message.
  - confidence: 0.86
- **[P2] Keep fragment-capable messages ordered:** `extensions/telegram/src/sequential-key.ts:65`
  - body: Appending `:msg:${messageId}` to every ordinary key lets same-chat text messages run concurrently, but text-fragment assembly uses shared chat/topic/sender state. Adjacent long-paste fragments can race before the first message installs the buffer and then emit separate or out-of-order turns.
  - confidence: 0.82
- **[P2] Update the sequential-key expectations:** `extensions/telegram/src/sequential-key.ts:65`
  - body: The PR changes ordinary sequential keys to include `:msg:1`, but the existing sequential-key tests still expect bare chat/topic keys. CI on the PR head already fails 11 cases until the tests are updated or the source change is narrowed.
  - confidence: 0.9
