## Summary

Keep open: current main still has the Telegram busy-message coalescing gap, but this PR remains blocked by missing inspectable real Telegram proof plus concrete code/test issues in the proposed patch.

## What This Changes

The PR adds an opt-in busy-buffering mode to the shared inbound debouncer, enables it for Telegram, and changes ordinary Telegram sequentialization keys to include message IDs.

## Best Possible Solution

Revise the PR so busy-buffered enqueue preserves Telegram update completion semantics, text-fragment preprocessing remains ordered, focused tests pass, and redacted live Telegram proof demonstrates one coalesced queued turn.

## Reproduction Assessment

Yes, source-reproducible. Current main shows chat/topic sequentialization plus same-key inbound-debouncer chaining before active-run collect routing, and the PR body gives the live DM burst scenario.

## Solution Assessment

No. The overall direction is plausible, but this patch returns before the first busy-buffer flush completes, relaxes ordering across text-fragment-capable messages, and leaves sequential-key tests failing.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Await the first busy-buffer flush:** `src/auto-reply/inbound-debounce.ts:201`
  - body: In the coalesceWhileBusy branch, enqueue schedules the buffered flush with void enqueueKeyTask and returns immediately. Telegram treats awaited enqueue completion as update-handler completion, so the first busy update can be marked complete and persisted before onFlush has delivered it; a restart in that window can skip the message.
  - confidence: 0.88
- **[P2] Preserve ordered text-fragment preprocessing:** `extensions/telegram/src/sequential-key.ts:107`
  - body: Adding :msg:${messageId} to every ordinary key lets same-chat text messages run concurrently, but text-fragment assembly uses shared per-chat/topic/sender state. Adjacent long-paste fragments can race before the first message installs the buffer and then dispatch as separate or out-of-order turns.
  - confidence: 0.82
- **[P2] Update the sequential-key tests:** `extensions/telegram/src/sequential-key.ts:107`
  - body: The source now returns message-id-suffixed keys for ordinary messages, but the existing sequential-key tests still expect bare chat/topic keys. The Telegram extension CI already fails 11 cases until the expectations are updated or the source change is narrowed.
  - confidence: 0.92
