## Summary

Keep open: the underlying Telegram collect-mode bug is source-reproducible on current main, but this PR is not mergeable because the branch returns from coalesced busy enqueue before the flush completes, weakens ordering needed by Telegram text-fragment assembly, has failing sequential-key tests, and lacks inspectable real Telegram proof.

## What This Changes

The PR adds an opt-in busy-buffering mode to the inbound debouncer, enables it for Telegram, and changes ordinary Telegram sequential keys to include `message_id`.

## Best Possible Solution

Land a revised Telegram fix that lets same-chat busy followups reach collect mode while preserving update checkpoint durability, text-fragment ordering, and sequential-key coverage.

## Reproduction Assessment

Yes. Source inspection on current main shows ordinary Telegram updates are still keyed by chat/topic and Telegram awaits inbound enqueue before completing the update, so rapid same-chat followups can be serialized past the active run before collect mode sees them.

## Solution Assessment

No. The PR targets the right bug, but the current implementation changes the enqueue completion contract, removes ordering needed by text-fragment assembly, and leaves existing sequential-key tests failing.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Await the coalesced busy flush:** `src/auto-reply/inbound-debounce.ts:207-213`
  - body: In the `coalesceWhileBusy` branch, `enqueue` schedules the follow-up flush with `void enqueueKeyTask(...)` and returns before `runFlush(pending)` runs. Telegram awaits `inboundDebouncer.enqueue(...)` before update completion, so a restart in that window can checkpoint a follow-up update before the buffered message is actually flushed.
  - confidence: 0.88
- **[P2] Preserve ordering for text fragments:** `extensions/telegram/src/sequential-key.ts:65`
  - body: Appending `:msg:${messageId}` to every ordinary sequential key lets same-chat/topic messages race through the long-text fragment combiner. That buffer appends only positive adjacent message IDs, so a later split fragment that reaches the handler first can force the parts to be emitted separately or out of order.
  - confidence: 0.84
- **[P2] Update the sequential-key expectations:** `extensions/telegram/src/sequential-key.ts:65`
  - body: The source change makes ordinary mock messages return keys with `:msg:1`, but `sequential-key.test.ts` still expects bare chat/topic keys and the PR CI log shows those failures. Update the expectations and add the missing-message-id fallback case, or narrow the source change so the current contract still holds.
  - confidence: 0.93
