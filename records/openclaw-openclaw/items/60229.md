## Summary

Keep open: the underlying Telegram collect-mode bug is source-reproducible, but this PR is not merge-ready because it lacks attached real Telegram proof and introduces message-delivery risks around early update completion and same-chat ordering.

## What This Changes

The PR adds a `coalesceWhileBusy` inbound debouncer option, enables it for Telegram, and changes ordinary Telegram sequentialization keys to include `message_id`.

## Best Possible Solution

Land a rebased Telegram fix that lets busy follow-ups reach collect mode while awaiting coalesced flushes, preserving fragment-capable ordering, updating focused tests, and attaching redacted real Telegram proof.

## Reproduction Assessment

Yes, at source level: current main serializes ordinary Telegram updates by chat/topic and awaits active same-key debouncer chains, so rapid follow-ups can reach active-run routing only after the prior run finishes. I did not run a live Telegram repro in this read-only review.

## Solution Assessment

No. The PR addresses the right bug, but the implementation should await coalesced busy flushes and preserve chat/topic ordering for fragment assembly instead of broadly making ordinary messages per-message concurrent.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Await busy-buffer flushes before returning:** `src/auto-reply/inbound-debounce.ts:207-213`
  - body: The `coalesceWhileBusy` path schedules `enqueueKeyTask` with `void` and returns immediately. Telegram persists update completion after the awaited middleware returns, so a restart in that window can skip buffered follow-up messages that have not reached `onFlush`.
  - confidence: 0.88
- **[P2] Preserve ordering for text fragments:** `extensions/telegram/src/sequential-key.ts:65`
  - body: Appending `:msg:${messageId}` to every ordinary sequential key lets same-chat/topic messages run concurrently. The text-fragment buffer mutates shared chat/thread/sender state and relies on adjacent message IDs, so split long-paste parts can be emitted separately or out of order.
  - confidence: 0.84
- **[P2] Update sequential-key expectations:** `extensions/telegram/src/sequential-key.ts:65`
  - body: The source change makes ordinary mock messages return keys with `:msg:1`, but the existing sequential-key test matrix still expects bare chat/topic keys. Update the expectations and missing-message-id fallback coverage, or narrow the key change to preserve the current contract.
  - confidence: 0.93
