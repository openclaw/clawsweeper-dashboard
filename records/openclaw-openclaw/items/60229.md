## Summary

Keep open: this targets a real Telegram busy-run collect bug, but the current patch is not mergeable as-is because it can acknowledge updates before the coalesced flush runs, removes ordering needed by text-fragment assembly, and leaves existing sequential-key tests failing; real Telegram proof is also still missing.

## What This Changes

The PR adds a shared `coalesceWhileBusy` debounce option, enables it for Telegram inbound handling, and changes ordinary Telegram sequential keys to include `message_id`.

## Best Possible Solution

Land a rebased Telegram fix that lets busy same-chat follow-ups reach collect mode while preserving update checkpoint durability, text-fragment ordering, current sequential-key coverage, and redacted real Telegram proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main sequentializes ordinary Telegram updates by chat/topic and same-key debouncer flushes are awaited, so rapid same-chat follow-ups can reach queue routing only after the active run clears; I did not run a live Telegram repro in this read-only review.

## Solution Assessment

No. The PR is aimed at the right bug, but the current patch returns before the coalesced flush completes, weakens fragment-capable message ordering, and leaves the sequential-key test contract failing.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Await the coalesced busy flush:** `src/auto-reply/inbound-debounce.ts:207-213`
  - body: The new busy branch schedules `enqueueKeyTask` with `void` and returns before `runFlush` completes. Telegram finishes update tracking after the handler returns, so a restart in that window can persist the follow-up update as handled even though its buffered message never reached `onFlush`.
  - confidence: 0.88
- **[P2] Preserve fragment-capable message ordering:** `extensions/telegram/src/sequential-key.ts:65`
  - body: Appending `:msg:${messageId}` to every ordinary key lets same-chat/topic messages enter `processInboundMessage` concurrently. The text-fragment combiner mutates shared per-chat/thread/sender state and relies on adjacent message IDs, so split long Telegram pastes can be emitted separately or out of order.
  - confidence: 0.84
- **[P2] Keep sequential-key tests aligned:** `extensions/telegram/src/sequential-key.ts:65`
  - body: The changed key shape makes ordinary mock messages return `:msg:1` suffixed keys, but the existing sequential-key matrix still expects bare chat/topic keys and PR CI is failing with 11 assertions. Update the contract tests or narrow the key change so the current contract remains true.
  - confidence: 0.93
