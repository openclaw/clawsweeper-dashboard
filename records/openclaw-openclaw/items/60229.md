## Summary

Keep open: the underlying Telegram collect-mode bug is source-reproducible, and the recent “what’s pending” comment is answered by the still-open blockers: missing inspectable real Telegram proof plus concrete correctness/test issues in the patch.

## What This Changes

The PR adds a `coalesceWhileBusy` path to the shared inbound debouncer, enables it for Telegram, and changes ordinary Telegram sequential keys to include message IDs.

## Best Possible Solution

Revise the patch to coalesce busy followups without returning before flush completion or breaking text-fragment ordering, update focused tests, then attach redacted live Telegram proof.

## Reproduction Assessment

Yes, source-reproducible. Current main documents collect-mode coalescing while Telegram still uses chat/topic sequentialization and same-key debouncer chaining before active-run queue routing; I did not run a live Telegram session.

## Solution Assessment

No. The direction is plausible, but this patch returns from `enqueue` before the coalesced flush completes and removes ordering that text-fragment assembly currently relies on.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Await the coalesced busy flush:** `src/auto-reply/inbound-debounce.ts:213`
  - body: In the `coalesceWhileBusy` branch, `enqueue` schedules the next same-key flush with `void enqueueKeyTask(...)` and returns. Telegram treats awaited handler completion as the update checkpoint boundary, so a crash or restart after that return can mark a follow-up update processed before `onFlush` handles it.
  - confidence: 0.86
- **[P2] Keep fragment-capable messages ordered:** `extensions/telegram/src/sequential-key.ts:65`
  - body: Appending `:msg:${messageId}` to every ordinary Telegram key lets adjacent same-chat text messages enter `processInboundMessage` concurrently. The long-text fragment combiner mutates a shared chat/thread/sender buffer, so split Telegram long-paste parts can race and emit separate or out-of-order turns.
  - confidence: 0.82
- **[P2] Update the sequential-key expectations:** `extensions/telegram/src/sequential-key.ts:65`
  - body: The source change makes ordinary mock messages return keys with `:msg:1`, but `sequential-key.test.ts` still expects bare chat/topic keys. The Telegram test lane will fail until expectations and a missing-message-id fallback case are updated or the source change is narrowed.
  - confidence: 0.9
