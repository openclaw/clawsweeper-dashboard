## Summary

This PR should stay open for author or maintainer follow-up: the reported Telegram collect-mode problem is source-reproducible, but the current patch still has merge-blocking correctness issues and lacks inspectable real Telegram proof.

## What This Changes

The PR adds an opt-in busy-buffering mode to the shared inbound debouncer, enables it for Telegram, and changes ordinary Telegram sequentialization keys to include the Telegram message ID.

## Best Possible Solution

Land a revised Telegram fix that lets busy followups reach collect mode while preserving the awaited update-checkpoint contract and keeping fragment-capable message handling ordered or explicitly concurrency-safe.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: current main uses chat/topic Telegram sequentialization plus same-key debouncer chaining before queue collect routing, so rapid same-chat followups cannot reach collect handling while the active run is still marked active. I did not run a live Telegram session in this read-only review.

## Solution Assessment

No. The direction is plausible, but this patch returns from the coalesced busy enqueue path before the flush completes and removes ordering needed by text-fragment assembly; a safer fix should preserve both contracts while enabling busy coalescing.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Await the coalesced busy flush:** `src/auto-reply/inbound-debounce.ts:207-213`
  - body: In the coalesceWhileBusy branch, enqueue schedules the follow-up flush with void enqueueKeyTask(...) and returns immediately. Telegram awaits enqueue as part of update handling, so the update can be checkpointed before runFlush(pending) processes the message, leaving a crash/restart window where a queued follow-up is skipped.
  - confidence: 0.88
- **[P2] Keep fragment-capable messages ordered:** `extensions/telegram/src/sequential-key.ts:65`
  - body: Adding :msg:${messageId} to every ordinary Telegram sequential key lets same-chat text messages enter processInboundMessage concurrently. The long-text fragment combiner mutates a shared chat/thread/sender buffer, so split Telegram long-paste parts can race and be emitted separately or out of order.
  - confidence: 0.84
- **[P2] Update the sequential-key expectations:** `extensions/telegram/src/sequential-key.ts:65`
  - body: The source change makes ordinary mock messages return keys with :msg:1, but sequential-key.test.ts still expects bare chat/topic keys and the PR CI log shows those failures. The test lane will fail until the expectations and a missing-message-id fallback case are updated, or the source change is narrowed.
  - confidence: 0.93
