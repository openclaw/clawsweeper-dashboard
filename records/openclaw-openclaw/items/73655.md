## Summary

Keep this open. Current main has several useful mitigations around startup readiness, handshake timing, runtime-deps hot paths, and stale-session cleanup, but the reported root causes are not fully closed: session-store writes still persist under the store lock, plugin replacement cleanup is limited to registered lifecycle/host-hook cleanup, and the Manifest/clawrouter 2099/2100 listener implementation is not present in the checked-in plugin tree to verify as fixed. Follow-up comments also confirm the WebSocket starvation/session-lock shape on Linux without Manifest, so this remains an active gateway lifecycle/performance bug rather than a stale single-host report.

## What This Changes

Investigate and fix gateway restart leaks involving Manifest port rebinding, process signal-handler accumulation, and session persistence lock contention that can starve WebSocket handshakes.

## Best Possible Solution

Split the remaining work into maintainer-owned root-cause fixes: make the Manifest/clawrouter listener startup idempotent wherever that plugin is sourced, add or require a lifecycle cleanup/dedup surface for plugin-owned process resources, and redesign session persistence so slow durable writes cannot block latency-sensitive Gateway handshakes or contending session lanes.

## Reproduction Assessment

Unclear for the full triad. The report has concrete logs, probes, and independent multi-host confirmations, but there is no deterministic current-main reproduction; the session-lock/WS-starvation path is the most reproducible/actionable part.

## Solution Assessment

No, current main is not a complete solution. The existing readiness, timeout, startup, runtime-deps, and stuck-session mitigations reduce symptoms, but they do not remove the durable-write lock scope, arbitrary process-listener cleanup gap, or the unverified Manifest listener idempotence issue.
