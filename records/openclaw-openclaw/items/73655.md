## Summary

Keep this open. Current main has useful mitigations for readiness diagnostics, handshake timeouts, lightweight session listing, and stuck-session recovery, but source inspection still does not prove the reported root causes are fixed: durable session-store writes still run inside the per-store writer, plugin replacement cleanup remains registered-hook scoped, and the reported Manifest/clawrouter listener source is not present in this checkout.

## What This Changes

The issue reports a gateway resource/starvation triad on restart involving Manifest/clawrouter port rebinding, process signal-listener accumulation, and long session persistence lock holds that can starve WebSocket handshakes.

## Best Possible Solution

Split the umbrella into maintainer-owned fixes: locate and make the Manifest/clawrouter listener startup idempotent, add or enforce a generic cleanup/dedup pattern for plugin-owned process resources, and redesign session persistence so durable writes cannot starve latency-sensitive gateway work.

## Reproduction Assessment

Partial yes. Current source gives a high-confidence path for one contributor, because session-store updates still await durable atomic writes inside the per-store writer, while the full Manifest/signal/session triad remains non-deterministic on current main.

## Solution Assessment

No. Closing on the current mitigations would not be the best solution; the maintainable path is to split the root causes and make the resource lifecycle and persistence contracts explicit.
