## Summary

Keep this open. Current main still shows the session-store fsync path inside the queued session-store writer, plugin replacement cleanup remains registered-hook scoped, and the newest issue context points to open PR #76881 as a partial fix rather than a merged implementation.

## What This Changes

The issue reports gateway WebSocket starvation after restart from Manifest port rebinding, plugin signal-listener accumulation, and session-store persistence work holding session write locks under load.

## Best Possible Solution

Review and land a narrow session-store persistence fix if #76881 is acceptable, then split the remaining Manifest listener idempotency and plugin process-resource cleanup into owner-specific follow-ups instead of closing the umbrella on mitigations.

## Reproduction Assessment

Partial yes. The session-store contributor is source-reproducible on current main because updateSessionStore still awaits durable atomic writes, including fsync, inside the serialized writer; the full three-part restart triad remains non-deterministic and environment-dependent.

## Solution Assessment

No. The latest proposed direction is a good narrow candidate for leak #3, but it is not a complete solution for this issue because Manifest port rebinding, process signal-listener cleanup, and residual CPU starvation are still not proven fixed on main.
