## Summary

Keep this open. Current main has several mitigations around handshake timing, startup readiness, stuck-session recovery, and plugin runtime-deps churn, but the reported root causes are not fully closed: the Manifest/clawrouter port-binding implementation is not present in the checked-in plugin tree, session-store writes still hold the store lock through durable atomic fsync work, and plugin restart cleanup remains limited to registered host lifecycle hooks rather than centrally deduplicating arbitrary process listeners or sockets. The latest discussion also reports the same starvation shape reproducing on 2026.4.29, so this is active rather than stale or implemented.

## What This Changes

Requests fixes for gateway restart/load starvation involving Manifest/clawrouter port rebinding, leaked process signal handlers, and session JSON persistence lock holds that can block WebSocket handshakes.

## Best Possible Solution

Split the remaining work into maintainer-owned root-cause fixes: make the Manifest/clawrouter listener startup idempotent, add or enforce a generic plugin lifecycle cleanup/dedup path for process-level resources, and redesign session persistence so slow durable writes cannot block latency-sensitive Gateway handshakes or session lanes.

## Reproduction Assessment

Unclear for the complete triad. The issue has strong logs and multiple release-install confirmations, including a 2026.4.29 reproduction, but I did not find a deterministic current-main reproduction path or a checked-in Manifest listener implementation to exercise locally.

## Solution Assessment

No, the current main mitigations are not the complete solution. The narrowest maintainable path is targeted lifecycle and persistence work that removes the starvation sources rather than only extending timeouts, adding readiness probes, or recovering after stalls.
