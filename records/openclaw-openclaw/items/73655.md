## Summary

Keep this open. Current main has useful mitigations for gateway readiness, diagnostics, and session-store write serialization, but it still does not prove the reported triad is fixed: session-store persistence still awaits durable fsync work, plugin restart cleanup remains hook-scoped, and the reported Manifest/clawrouter listener is not present in the checked-in bundled plugin source. The latest discussion also points to open PR #76881 for only the fsync/session-store slice, so the broader issue should stay open for maintainer follow-up.

## What This Changes

The issue requests fixes for a gateway restart resource-leak cluster where plugin listener retries, process signal listeners, and session-store JSON persistence can starve WebSocket handshakes under load.

## Best Possible Solution

Review and land or revise #76881 for the session-store durability trade-off, then split the remaining Manifest listener idempotency and plugin-owned process-resource cleanup into maintainer-owned follow-ups with targeted startup/load validation.

## Reproduction Assessment

Partial: yes for the session-store fsync contributor, because current source still awaits durable atomic writes through the session-store writer. No high-confidence local reproduction exists for the full Manifest/signal/session triad on current main; the broader proof is reporter logs plus cross-platform confirmations in the issue discussion.

## Solution Assessment

No, current main is not a complete solution. PR #76881 appears to be a plausible narrow fix for the fsync slice, but the maintainable outcome is to land that only if the durability contract is accepted and separately fix the remaining plugin listener and process-resource lifecycle paths.
